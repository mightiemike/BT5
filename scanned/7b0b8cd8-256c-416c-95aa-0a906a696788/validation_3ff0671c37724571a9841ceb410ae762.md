### Title
`SwapAllowlistExtension` gates the router's address instead of the end user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the direct `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool sees the router as `msg.sender`, so the extension checks whether the **router** is allowlisted — not the end user. A pool admin who allowlists the router (the natural configuration for router-enabled pools) inadvertently opens the allowlist to every user, defeating the per-user gate entirely.

---

### Finding Description

`MetricOmmPool.swap()` captures `msg.sender` and forwards it as the `sender` argument to every `beforeSwap` extension hook: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value verbatim into the call to each registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool forwarded: [3](#0-2) 

**Direct swap path** (`user → pool.swap()`):
`sender` = user address → `allowedSwapper[pool][user]` — correct.

**Router-mediated path** (`user → MetricOmmSimpleRouter → pool.swap()`):
`msg.sender` to the pool = router → `sender` = router address → `allowedSwapper[pool][router]` — the end user's identity is never checked.

The pool admin faces an impossible choice:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Every router-mediated swap reverts — allowlisted users cannot use the router |
| Router **allowlisted** | Every user, allowlisted or not, can swap through the router — allowlist is fully bypassed |

There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

The `allowedSwapper` mapping is keyed by `(pool, swapper)`: [4](#0-3) 

The admin setter operates on individual swapper addresses: [5](#0-4) 

Nothing in the extension or the pool propagates the original end-user address through the router hop.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (e.g., KYC-verified counterparties, institutional participants) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. Once the router is allowlisted — the only way to make the router usable for legitimate users — the per-user gate is gone. Unauthorized users can execute swaps against the pool, causing adverse selection, unintended price impact, and potential extraction of LP value from a pool whose liquidity providers deposited under the assumption of a restricted trading environment. This is a direct admin-boundary break: the pool admin's configured access control is bypassed by a public, unprivileged path.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool that (a) deploys `SwapAllowlistExtension` and (b) wants its allowlisted users to be able to use the router must allowlist the router. This is the natural and expected operational configuration, making the bypass reachable in every realistic restricted-pool deployment.

---

### Recommendation

The extension must verify the **end user's** identity, not the intermediary's. Two viable approaches:

1. **Extension-data attestation**: Require the router to embed the original user address in `extensionData` (e.g., `abi.encode(userAddress)`). The extension decodes and checks `allowedSwapper[pool][decodedUser]`. The router must be trusted to supply the correct address, and the extension must reject calls where `extensionData` is absent or malformed.

2. **Dedicated router with identity forwarding**: Introduce a pool-level concept of a trusted forwarder (similar to ERC-2771 meta-transactions) so the pool itself can recover the original sender from a signed or encoded payload, and pass that recovered address as `sender` to extensions.

Until one of these is in place, pools that need per-user swap gating should instruct users to call `pool.swap()` directly and must **not** allowlist the router.

---

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension registered on beforeSwap.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to enable router-mediated swaps for allowlisted users.
3. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
   // bob is NOT added to the allowlist

Attack
──────
4. bob (non-allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(...)
   targeting the restricted pool.
5. Router calls pool.swap(recipient=bob, ...).
6. pool.swap captures msg.sender = router, calls _beforeSwap(sender=router, ...).
7. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] → true.
8. Swap executes successfully for bob despite bob never being allowlisted.

Expected: revert NotAllowedToSwap
Actual:   swap succeeds — allowlist bypassed
``` [3](#0-2) [6](#0-5)
