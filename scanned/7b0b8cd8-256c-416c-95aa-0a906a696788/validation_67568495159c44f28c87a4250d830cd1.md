### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass a Curated Pool's Allowlist via the Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The allowlist therefore gates the router's address rather than the actual swapper. If a pool admin allowlists the router (a natural action to enable router-mediated swaps on a curated pool), every user—including those the admin intended to exclude—can bypass the per-user allowlist by going through the router.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same is true for `exactOutputSingle`, `exactInput` (all hops), and `exactOutput` (all hops): [5](#0-4) [6](#0-5) 

In every router path the extension receives `sender = address(router)`, not the end user's address. The allowlist lookup `allowedSwapper[pool][router]` is therefore the only check that matters for all router-mediated swaps.

This is the direct analog of the MarmoStork bug: just as MarmoStork accepted a `bytes` argument and used its length to generate bytecode without validating that it was a valid 20-byte address, `SwapAllowlistExtension` accepts a `sender` argument that is structurally an `address` but semantically the wrong identity—the intermediary router rather than the economic actor the admin intended to gate.

### Impact Explanation

Two fund-impacting outcomes follow:

1. **Allowlist bypass (primary):** A pool admin who allowlists the router address (e.g., `setAllowedToSwap(pool, router, true)`) to enable router-mediated swaps inadvertently opens the pool to every user. Any address—including those the admin explicitly excluded—can call `exactInputSingle` or `exactInput` through the router and pass the allowlist check. The curated pool's access control is completely defeated.

2. **Broken guard for allowlisted users (secondary):** If the admin allowlists specific user addresses but not the router, those users cannot swap through the router at all (the extension sees `sender = router`, which is not allowlisted, and reverts). The only path available is a direct `pool.swap()` call, which requires the caller to implement `IMetricOmmSwapCallback`. This makes the router—the primary user-facing interface—unusable for any allowlisted pool, breaking core swap functionality.

### Likelihood Explanation

The router is the standard user-facing entry point documented and deployed alongside the core pool. A pool admin who wants to enable router-mediated swaps on a curated pool has no mechanism other than allowlisting the router address, which immediately triggers the bypass. The misconfiguration is not obvious from the extension's interface or documentation. Any curated pool that uses `SwapAllowlistExtension` and the router simultaneously is affected.

### Recommendation

The `sender` argument passed to `beforeSwap` should represent the end user, not the intermediary. Two complementary fixes:

1. **In the router:** Store the original `msg.sender` in transient storage at entry (alongside the callback context already stored there) and expose it so the pool can forward it as the true swapper identity. The pool would then pass this value—not its own `msg.sender`—as `sender` to extensions.

2. **In `SwapAllowlistExtension`:** Until the pool-level fix is in place, document clearly that `sender` is the direct caller of `pool.swap()` (i.e., the router, not the end user) and that per-user allowlisting is incompatible with router-mediated swaps. Alternatively, gate on `recipient` if the pool admin controls who receives output tokens, though this has its own limitations.

The root fix must ensure the identity checked by the allowlist is the same actor the economic action is attributed to—matching the invariant stated in the protocol's own audit targets.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension (beforeSwap order = extension 1).
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to allow router-mediated swaps while blocking direct calls from unknown addresses).
  - Pool admin does NOT allowlist attacker's EOA.

Attack:
  - Attacker calls router.exactInputSingle({pool: pool, ...}).
  - Router calls pool.swap(recipient, ...) → msg.sender of pool.swap() = router.
  - Pool calls _beforeSwap(msg.sender=router, ...).
  - SwapAllowlistExtension.beforeSwap(sender=router, ...) checks
      allowedSwapper[pool][router] → true  ✓
  - Swap executes. Attacker receives output tokens.

Expected: Attacker's swap should revert (NotAllowedToSwap).
Actual:   Swap succeeds because the router is allowlisted, not the attacker.
