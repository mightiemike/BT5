### Title
`SwapAllowlistExtension` gates the router address instead of the originating user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract, not the original user. This creates an irresolvable deadlock for pool admins: either the router is not allowlisted (allowlisted users cannot use it) or the router is allowlisted (any user bypasses the allowlist entirely). There is no configuration that enforces per-user allowlisting through the supported router path.

---

### Finding Description

**Call path:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient, ..., extensionData)   [msg.sender = router]
     → ExtensionCalling._beforeSwap(sender=router, recipient, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
     → allowedSwapper[pool][router]  ← checks router, not user
```

In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap`, the router is `msg.sender` of that call: [4](#0-3) 

The pool admin faces an irresolvable deadlock:

| Router allowlisted? | Effect |
|---|---|
| No | Allowlisted users cannot swap through the router |
| Yes | **Any** user bypasses the allowlist via the router |

There is no configuration that allows specific users to swap through the router while blocking others. The `DepositAllowlistExtension` avoids this problem by checking `owner` (the position owner, passed explicitly) rather than `sender` (the caller): [5](#0-4) 

The swap extension has no equivalent — it only receives `sender` (the router) and `recipient`, neither of which is the originating user.

---

### Impact Explanation

A pool admin who allowlists the router to enable standard periphery usage inadvertently opens the allowlist to every public caller. Any non-allowlisted user can execute swaps on a curated pool by routing through `MetricOmmSimpleRouter`, bypassing the intended access control. This breaks the core invariant: *a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it.* The bypass is direct and requires no special privileges — only the public router.

---

### Likelihood Explanation

Allowlisting the router is the natural and expected configuration for any pool that wants to support the standard periphery. The pool admin has no way to know that doing so opens the allowlist to all users, because the extension's documentation and interface give no indication that `sender` will be the router rather than the originating user. The attack path is fully permissionless once the router is allowlisted.

---

### Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor, not the intermediary. Two viable approaches:

1. **Check `recipient` instead of `sender`**: The recipient is the address that receives output tokens and is the economic beneficiary of the swap. This is already available as the second argument to `beforeSwap`.

2. **Require the router to embed the originating user in `extensionData`** and have the extension decode and verify that address. This requires a coordinated change to the router and extension.

Option 1 is simpler and consistent with how `DepositAllowlistExtension` gates by `owner` (the economic beneficiary of the deposit) rather than `sender` (the caller).

---

### Proof of Concept

1. Pool admin deploys pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin allowlists user A: `setAllowedToSwap(pool, userA, true)`.
3. Pool admin allowlists the router to enable periphery usage: `setAllowedToSwap(pool, router, true)`.
4. Non-allowlisted user B calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
5. Router calls `pool.swap(userB, ...)` — `msg.sender` of this call is the router.
6. Extension checks `allowedSwapper[pool][router]` → `true`.
7. User B's swap executes successfully despite never being allowlisted.

The allowlist is fully bypassed. User A's explicit allowlisting provides no meaningful restriction once the router is permitted.
