### Title
`SwapAllowlistExtension` Checks Router Address as Swapper Identity, Allowing Any User to Bypass the Allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` = `msg.sender` of the pool's `swap()` call. When a user routes through the public `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the actual end-user. The allowlist therefore gates the router address, not the individual trader. Any pool admin who allowlists the router (the only way to permit router-mediated swaps) simultaneously grants every user on the network the ability to bypass the per-user swap restriction.

---

### Finding Description

**Hook binding in `MetricOmmPool.swap`:**

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // ← always the router when called via MetricOmmSimpleRouter
  recipient,
  ...
);
``` [1](#0-0) 

**Router calls pool directly:**

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
``` [2](#0-1) 

**Guard checks the wrong identity:**

```solidity
// SwapAllowlistExtension.sol line 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the router. The effective check is `allowedSwapper[pool][router]`. The pool admin faces an impossible choice:

| Admin action | Result |
|---|---|
| Do **not** allowlist the router | All router-mediated swaps revert — router is unusable for this pool |
| **Allowlist the router** | Every user on the network can swap through the router — allowlist is nullified |

There is no configuration that simultaneously permits router-mediated swaps and enforces per-user restrictions.

The `ExtensionCalling._beforeSwap` dispatcher faithfully forwards `sender` as received from the pool, so the misbinding originates at the extension's identity check, not the dispatcher. [4](#0-3) 

---

### Impact Explanation

A pool deployer uses `SwapAllowlistExtension` to restrict trading to a curated set of counterparties — for example, to prevent adverse selection against oracle-anchored LP positions, or to comply with regulatory requirements. Once the router is allowlisted (the only way to support normal user flows), any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` and execute swaps against the pool. The oracle-anchored bid/ask spread is thin by design; unrestricted adversarial traders can systematically extract value from LP positions at oracle-fair prices, causing direct loss of LP principal. This matches the "Broken core pool functionality causing loss of funds" and "Admin-boundary break bypassed by an unprivileged path" impact categories.

---

### Likelihood Explanation

The trigger requires only a standard public call to `MetricOmmSimpleRouter.exactInputSingle` — no special role, no privileged access, no malicious setup. Any user who knows the pool address can execute it. The pool admin's only defense (not allowlisting the router) makes the pool incompatible with the standard periphery, so in any realistic deployment the router will be allowlisted and the bypass will be live.

---

### Recommendation

The extension must verify the actual end-user identity, not the intermediary. Two sound approaches:

1. **Pass original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool; the extension decodes and checks that address. This requires a convention between router and extension.

2. **Check `sender` only for direct pool calls; require explicit user identity in `extensionData` for router calls**: The extension can detect router-mediated calls (e.g., `sender == knownRouter`) and require a signed or encoded user identity in `extensionData`.

The `DepositAllowlistExtension` does not share this flaw because it checks `owner` (the position recipient, explicitly passed by the caller), not `sender`. [5](#0-4) 

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured; set `allowAllSwappers[pool] = false`.
2. Call `setAllowedToSwap(pool, router, true)` — necessary for any router-mediated swap to succeed.
3. From an address `alice` that has **never** been added to `allowedSwapper[pool]`, call:
   ```solidity
   router.exactInputSingle(ExactInputSingleParams({
     pool: restrictedPool,
     tokenIn: token0,
     tokenOut: token1,
     zeroForOne: true,
     amountIn: 1000,
     ...
   }));
   ```
4. The pool calls `beforeSwap(router, ...)`. The check `allowedSwapper[pool][router]` is `true`. The hook returns the success selector. Alice's swap executes against the restricted pool despite never being allowlisted.
5. Confirm: calling `pool.swap(...)` directly from `alice` (without the router) reverts with `NotAllowedToSwap`, proving the allowlist is only enforced for direct calls.
