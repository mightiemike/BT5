### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is supposed to gate who can add liquidity to a restricted pool. However, it silently discards the `sender` parameter (the actual caller of `addLiquidity`) and instead checks `owner` (the address that will receive the position). Because `owner` is a free caller-supplied argument, any unprivileged user can bypass the allowlist entirely by passing an authorized address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding the actual caller as `sender` and the caller-supplied position recipient as `owner`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both and dispatches them to every configured extension in order: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but **ignores `sender`** (unnamed first parameter) and checks `owner` instead: [3](#0-2) 

Because `owner` is a free argument supplied by the caller, any address can call `pool.addLiquidity(authorizedAddress, ...)`, pass the allowlist check (`allowedDepositor[pool][authorizedAddress] == true`), provide tokens via the swap callback, and create a position for `authorizedAddress` â€” all without being on the allowlist themselves.

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual caller): [4](#0-3) 

The asymmetry is the root cause: the swap guard checks the right actor; the deposit guard checks the wrong one.

---

### Impact Explanation

The deposit allowlist is a core access-control feature for restricted pools (KYC/AML gating, institutional-only pools, etc.). With this bug the guard is completely ineffective:

1. **Allowlist bypass**: Any unprivileged address can add liquidity to a pool that is supposed to be restricted, simply by setting `owner` to any address that is on the allowlist.
2. **Pool state manipulation**: An unauthorized depositor can shift bin balances, alter `curBinIdx`/`curPosInBin`, and interact with other extensions (e.g., `OracleValueStopLossExtension`) in ways that affect existing LPs' value-per-share metrics and stop-loss watermarks.
3. **Broken core pool functionality**: The entire purpose of the extension â€” restricting who may deposit â€” is rendered inoperative, matching the "broken core pool functionality" impact category.

---

### Likelihood Explanation

- No special privilege is required; any EOA or contract can call `addLiquidity`.
- The attacker only needs to know one authorized address (publicly readable from `allowedDepositor` mapping or from past events).
- The bypass is deterministic and requires no oracle manipulation, flash loans, or timing.

---

### Recommendation

Replace the `owner` check with a `sender` check, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`sender` is `msg.sender` of the originating `addLiquidity` call â€” the address that actually provides tokens via the callback â€” and is the correct entity to gate.
