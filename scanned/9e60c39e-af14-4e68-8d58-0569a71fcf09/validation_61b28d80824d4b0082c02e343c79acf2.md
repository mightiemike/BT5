### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Position `owner` Instead of Actual Depositor `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` **by depositor address**. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual caller) and instead validates the `owner` argument (the position recipient). Because `owner` is a free caller-supplied parameter with no restriction beyond `!= address(0)`, any non-allowlisted address can deposit into a curated pool by nominating any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct actors to the extension:

```solidity
// MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `sender` = `msg.sender` — the address that is actually calling the pool and paying tokens through the callback.
- `owner` = the caller-supplied position recipient — the address that will hold the resulting shares.

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (first parameter is unnamed `address,`) and gates only on `owner`:

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`_validateOwner` in `MetricOmmPoolLiquidityAdder` only rejects `address(0)`:

```solidity
// MetricOmmPoolLiquidityAdder.sol:247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

**Attack path (direct pool call):**

1. Pool is deployed with `DepositAllowlistExtension`; Alice (`0xAlice`) is allowlisted; Bob (`0xBob`) is not.
2. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(bob, alice, ...)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true`. Hook passes.
5. Bob's tokens are pulled through the callback; Alice receives the position shares.

**Attack path (via `MetricOmmPoolLiquidityAdder`):**

1. Bob calls `addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)`.
2. `_validateOwner(alice)` passes (non-zero).
3. LiquidityAdder calls `pool.addLiquidity(alice, ...)` with `payer = bob`.
4. Extension checks `owner = alice` → allowlisted → passes.
5. Bob's tokens are pulled; Alice receives the position.

In both paths the allowlist is completely bypassed: the actual depositor (Bob) is never checked.

---

### Impact Explanation

The `DepositAllowlistExtension` is the sole on-chain mechanism for pool admins to curate who may provide liquidity to a pool. With this bug the guard is rendered entirely ineffective: any address, regardless of allowlist status, can deposit tokens into a curated pool by supplying any allowlisted address as `owner`. The pool admin's curation policy — which may exist for regulatory, risk, or composition reasons — is bypassed by an unprivileged caller with no special access. This is a direct admin-boundary break on a live user flow.

---

### Likelihood Explanation

The bypass requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. No privileged access, flash loan, or oracle manipulation is needed. Any address that can observe the allowlist (via `allowedDepositor` public mapping or emitted `AllowedToDepositSet` events) can execute the bypass immediately. Likelihood is **High**.

---

### Recommendation

Gate on `sender` (the actual depositor/payer), not `owner` (the position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swapper) and ignores `recipient`.

---

### Proof of Concept

```solidity
// Pool has DepositAllowlistExtension configured.
// Alice is allowlisted; Bob is not.

// Admin setup:
extension.setAllowedToDeposit(address(pool), alice, true);
// Bob is NOT allowlisted.

// Bob's bypass — direct pool call:
vm.prank(bob);
// Bob specifies alice as owner; extension checks owner=alice → passes.
pool.addLiquidity(
    alice,          // owner: allowlisted → check passes
    0,              // salt
    deltas,
    callbackData,
    extensionData
);
// Bob's tokens are pulled in the callback; Alice receives shares.
// The allowlist did not block Bob.

// Verify: Bob's tokens entered the pool despite Bob not being allowlisted.
assertGt(pool.balance0(), 0); // pool received Bob's tokens
```

The `SwapAllowlistExtension` correctly checks `sender` at line 37:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```
