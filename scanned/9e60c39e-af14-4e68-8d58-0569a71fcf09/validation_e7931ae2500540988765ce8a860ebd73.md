Based on my analysis of the repository structure and the key contracts I was able to read, here is my assessment:

### Title
`DepositAllowlistExtension` Checks `sender` But LP Shares Are Minted to `owner`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol` / `metric-core/contracts/MetricOmmPool.sol`)

### Summary

The `addLiquidity` function in `MetricOmmPool` deliberately separates the caller (`msg.sender`) from the position recipient (`owner`). The `_beforeAddLiquidity` extension hook receives both values. If `DepositAllowlistExtension.beforeAddLiquidity` validates only `sender` (the caller) and not `owner` (the LP share recipient), a non-allowlisted address can receive LP shares by having an allowlisted address call `addLiquidity` on its behalf — directly analogous to the GeodeFi `stake()` missing `operatorId` check.

### Finding Description

In `MetricOmmPool.addLiquidity`:

```solidity
function addLiquidity(
    address owner,       // ← position recipient, caller-supplied, can differ from msg.sender
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData); // ← both passed
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, ...   // ← shares minted to owner
    );
    _afterAddLiquidity(msg.sender, owner, salt, deltas, amount0Added, amount1Added, extensionData);
}
``` [1](#0-0) 

The `_beforeAddLiquidity` hook in `ExtensionCalling` encodes both `sender` and `owner` and dispatches to configured extensions:

```solidity
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
``` [2](#0-1) 

If `DepositAllowlistExtension.beforeAddLiquidity` validates only `sender` (the msg.sender) and not `owner` (the actual LP share recipient), the allowlist is bypassed for the owner. The LP shares are permanently credited to `owner`, not `sender`.

Critically, `removeLiquidity` enforces `msg.sender == owner`:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [3](#0-2) 

This means the non-allowlisted `owner` can later call `removeLiquidity` themselves (no allowlist check on removal) and withdraw the underlying tokens.

### Impact Explanation

A non-allowlisted address receives LP shares in a deposit-restricted pool and can withdraw the underlying token0/token1 via `removeLiquidity`. This directly breaks the deposit allowlist invariant — the guard is applied to the wrong actor. The LP claim is real and redeemable, constituting a direct loss of the pool's ability to enforce access control over who holds LP positions and can drain liquidity.

### Likelihood Explanation

Any allowlisted address (e.g., a router, a whitelisted LP manager, or any address the pool admin has approved) can trivially call `addLiquidity(nonAllowlistedOwner, ...)`. No special privileges beyond being on the allowlist are required. The `owner` parameter is fully caller-controlled with no validation in the pool core.

### Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` must validate **both** `sender` and `owner` against the allowlist, not just `sender`. Since LP shares are minted to `owner`, the allowlist check must cover `owner` as the primary gating condition. Alternatively, the pool core should enforce `owner == msg.sender` in `addLiquidity` (as it does in `removeLiquidity`), but this would break legitimate delegated-deposit use cases.

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. `allowlistedSender` is on the deposit allowlist; `nonAllowlistedOwner` is not.
3. `allowlistedSender` calls `pool.addLiquidity(nonAllowlistedOwner, salt, deltas, callbackData, extensionData)`.
4. `_beforeAddLiquidity(allowlistedSender, nonAllowlistedOwner, ...)` fires — extension checks `sender` (allowlisted) → passes.
5. `LiquidityLib.addLiquidity(..., nonAllowlistedOwner, ...)` mints LP shares to `nonAllowlistedOwner`.
6. `nonAllowlistedOwner` calls `pool.removeLiquidity(nonAllowlistedOwner, salt, deltas, extensionData)` — `msg.sender == owner` check passes, no allowlist check on removal.
7. `nonAllowlistedOwner` receives token0 and token1, having fully bypassed the deposit allowlist.
