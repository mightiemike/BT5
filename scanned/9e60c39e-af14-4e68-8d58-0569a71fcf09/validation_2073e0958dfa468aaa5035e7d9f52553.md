### Title
`SwapAllowlistExtension` gates the router address instead of the end user, allowing any user to bypass per-user swap restrictions via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, `msg.sender` of `pool.swap()` is the **router contract**, not the end user. A pool admin who allowlists the router to enable standard periphery usage inadvertently grants every user unrestricted swap access, completely defeating the per-user allowlist. Conversely, if the router is not allowlisted, individually allowlisted users are locked out of the only public swap periphery.

---

### Finding Description

**Swap path — what `sender` the extension actually sees**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension:

```solidity
// ExtensionCalling.sol:162-165
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, ...)   // sender == msg.sender of pool.swap()
)
```

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, the pool's `msg.sender` is the **router contract address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**The two failure modes**

| Router allowlisted? | Result |
|---|---|
| Yes | Every user bypasses the per-user allowlist by routing through the router |
| No | Every individually allowlisted user is blocked from using the standard periphery |

**Contrast with `DepositAllowlistExtension` — the correct pattern**

`DepositAllowlistExtension.beforeAddLiquidity` deliberately ignores the `sender` (caller of `pool.addLiquidity`) and checks `owner` instead — the actual position owner that the pool passes as a separate, explicit parameter:

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

Because `pool.addLiquidity(owner, ...)` carries the real beneficiary as an explicit argument, the deposit extension can gate the correct identity regardless of who the intermediary caller is. The swap interface has no equivalent "actual user" parameter — only `sender` (the immediate caller) — so `SwapAllowlistExtension` has no on-chain way to recover the true end user when the router is in the call stack.

---

### Impact Explanation

**Bypass path (router allowlisted):** A pool admin who wants allowlisted users to use the standard periphery must add the router to `allowedSwapper`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every swap that passes through it, regardless of who called the router. Any unprivileged user can call `router.exactInputSingle(...)` and the extension check passes unconditionally. The per-user allowlist is rendered inoperative for all router-mediated swaps.

**Lockout path (router not allowlisted):** If the admin does not allowlist the router, individually allowlisted users who call `router.exactInputSingle(...)` receive `NotAllowedToSwap` even though they are on the allowlist. The only usable swap path requires the user to call `pool.swap()` directly and implement `metricOmmSwapCallback` themselves — the standard periphery is broken for every allowlisted pool.

Both outcomes are fund-impacting: the bypass allows unauthorized traders to execute swaps in pools designed to be restricted (adverse selection against LPs, regulatory breach); the lockout makes the swap flow unusable for legitimate users who have deposited liquidity expecting to be able to trade.

---

### Likelihood Explanation

The trigger requires no special privilege. Any user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The bypass materialises the moment a pool admin allowlists the router — a natural operational step when deploying a pool intended for use with the standard periphery. The lockout materialises for every allowlisted pool that does not also allowlist the router, which is the default state.

---

### Recommendation

1. **Preferred fix — pass actual user through `extensionData`:** Have the router encode `msg.sender` into `extensionData` before forwarding to the pool. `SwapAllowlistExtension` can then decode and check the real end user when `extensionData` is present, falling back to `sender` for direct pool calls.

2. **Alternative — add a dedicated `actualSender` field to the swap extension interface:** Extend `IMetricOmmExtensions.beforeSwap` with an `actualSender` parameter that the router populates with `msg.sender` before calling the pool (e.g., via a transient-storage context similar to the callback payer pattern already used in `MetricOmmSimpleRouter`).

3. **Minimum mitigation — documentation and factory guard:** Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and add a factory-level check that prevents deploying a pool with `SwapAllowlistExtension` in its extension order unless `allowAllSwappers` is set.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Pool admin calls swapExtension.setAllowedToSwap(pool, alice, true)
   — alice is the only allowlisted swapper.
3. Pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
   — necessary so alice can use the standard periphery.
4. Bob (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
   The router calls pool.swap(recipient, ...) with msg.sender = router.
   The pool calls _beforeSwap(router, ...).
   SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
   Bob's swap succeeds despite not being on the allowlist.

Alternatively (lockout path):
3'. Pool admin does NOT allowlist the router.
4'. Alice calls router.exactInputSingle(...).
    The pool calls _beforeSwap(router, ...).
    SwapAllowlistExtension checks allowedSwapper[pool][router] → false.
    Revert: NotAllowedToSwap — alice is locked out of the standard periphery.
```
