### Title
`SwapAllowlistExtension` gates the router address instead of the end user, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` to the pool. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the actual end user. If the pool admin allowlists the router to enable router-mediated swaps, every user — including those not individually allowlisted — can bypass the curated gate by routing through the router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // ← always the immediate caller of the pool
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap()` then checks that exact value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` inside the extension is the pool (correct), and `sender` is whoever called the pool. When a user calls `MetricOmmSimpleRouter.exactInputSingle()` or `exactInput()`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

`msg.sender` to the pool is the **router address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

This produces two broken states:

| Pool admin configuration | Outcome |
|---|---|
| Router is allowlisted (to support router users) | Every user bypasses the allowlist by routing through the router |
| Router is NOT allowlisted | Individually allowlisted users cannot use the router at all |

Neither state correctly enforces the intended per-user curation policy.

The same path exists for `exactInput()` (multi-hop) and `exactOutput()` / `exactOutputSingle()` — all four router entry points call `pool.swap()` with `msg.sender = router`.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` intends to restrict trading to a specific set of addresses. Any non-allowlisted user can bypass this restriction entirely by calling `MetricOmmSimpleRouter.exactInputSingle()` instead of `pool.swap()` directly, provided the router is allowlisted (which is the only way to make the router usable on that pool). The allowlist provides zero protection against router-mediated swaps, meaning unauthorized users can trade on a pool that was designed to exclude them. This is a direct policy bypass with fund-impacting consequences: unauthorized users drain pool liquidity at oracle prices the pool admin reserved for a curated counterparty set.

---

### Likelihood Explanation

The bypass requires only that the attacker call a publicly deployed periphery contract (`MetricOmmSimpleRouter`) instead of the pool directly. No privileged access, no special tokens, no flash loans, and no admin cooperation are needed. The router is a standard supported entrypoint documented in the protocol. Any user who reads the contract or the README can discover and exploit this in a single transaction.

---

### Recommendation

Pass the original end-user identity through the swap path so the extension can gate the economically relevant actor. Two approaches:

1. **Preferred — forward `tx.origin` or a user-supplied identity via `extensionData`**: The router encodes `msg.sender` (the actual user) into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap()` decodes and checks that value instead of (or in addition to) `sender`. This keeps the check trustless as long as the extension validates the encoding.

2. **Alternative — check both `sender` and `recipient`**: Gate on `recipient` when `sender` is a known router, or require the extension to check `allowedSwapper[pool][recipient]` as a fallback. This is fragile because `recipient` can be set to any address.

The cleanest fix is option 1: the router appends `abi.encode(msg.sender)` to `extensionData`, and `SwapAllowlistExtension` reads and checks that value when present, falling back to `sender` for direct pool calls.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so router-mediated swaps work.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ..., recipient: attacker})`.
5. The router calls `pool.swap(attacker, ...)` with `msg.sender = router`.
6. `_beforeSwap(router, attacker, ...)` is dispatched; `SwapAllowlistExtension` evaluates `allowedSwapper[pool][router]` → `true`.
7. The swap executes. The attacker receives output tokens despite never being individually allowlisted.
