### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. A pool admin who allowlists the router address to support standard periphery swaps inadvertently opens the pool to every user, completely defeating the allowlist.

---

### Finding Description

**Call chain when routing through MetricOmmSimpleRouter:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient, ...)          // msg.sender = router
     → _beforeSwap(msg.sender, ...)       // sender = router address
     → SwapAllowlistExtension.beforeSwap(sender = router_address, ...)
     → allowedSwapper[pool][router_address]  // checks router, not user
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:231
_beforeSwap(
    msg.sender,   // <-- router address when called via router
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the allowlist keyed by `msg.sender` (the pool):

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When the router calls the pool, `sender` = router address. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted.

This creates an irreconcilable dilemma for pool admins:

| Admin action | Result |
|---|---|
| Allowlist individual users only | Those users cannot swap through the router (router not allowlisted → revert) |
| Allowlist the router address | **Any** user can bypass the allowlist by routing through the router |

The second row is the critical bypass. A pool admin who wants to support the standard periphery path (the router) must allowlist the router, which opens the pool to all users.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly checks `owner` (the economic actor who receives the LP position), not `sender` (the caller). This design is immune to the same intermediary problem. The swap extension should analogously check the actual swapper identity, but the `beforeSwap` interface only exposes `sender` (= pool's `msg.sender`), which collapses to the router address on any router-mediated swap.

---

### Impact Explanation

**Severity: High**

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-controlled addresses) can have that restriction completely bypassed. Any unprivileged user routes through `MetricOmmSimpleRouter`, which is allowlisted by the admin to support normal periphery usage. The allowlist provides zero protection: the user receives the swap output, the pool's token balances change, and the curation policy is violated. This is a direct loss of the pool's intended access-control invariant with fund-impacting consequences (unauthorized parties trade against the pool's liquidity).

---

### Likelihood Explanation

**Likelihood: High**

`MetricOmmSimpleRouter` is the canonical, documented periphery entry point for swaps. Any pool admin who deploys a swap-allowlisted pool and also wants to support normal user tooling (wallets, aggregators, front-ends that use the router) will allowlist the router. The bypass requires no special knowledge or privileged access — any user calls `exactInputSingle` on the router with the target pool. The admin's mistake is structurally forced by the design: there is no mechanism to allowlist specific users *and* allow them to use the router simultaneously.

---

### Recommendation

1. **Short-term:** Change `SwapAllowlistExtension.beforeSwap` to check `recipient` instead of `sender` if the intent is to gate who receives the output, or add a dedicated `swapper` field to the extension payload that the router populates with `msg.sender` before calling the pool.

2. **Preferred fix:** Introduce a `swapper` parameter distinct from `sender` in the `beforeSwap` hook interface, where `swapper` is the original transaction initiator. The router would pass `msg.sender` (the user) as `swapper` when calling the pool, and the pool would forward it to the extension. This mirrors how `DepositAllowlistExtension` uses `owner` (the economic actor) rather than `sender` (the intermediary).

3. **Documentation guard (insufficient alone):** Document that `SwapAllowlistExtension` gates the direct pool caller, not the end user, and that allowlisting the router opens the pool to all users. This does not fix the bypass but prevents misconfiguration.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Admin also allowlists the router to support periphery swaps.
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // <-- admin enables router

// Attack: bannedUser (not on allowlist) swaps through the router.
vm.prank(bannedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1_000,
        amountOutMinimum: 0,
        recipient: bannedUser,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// bannedUser receives token1 output. Allowlist bypassed.
// Extension checked allowedSwapper[pool][router] == true → passes.
// bannedUser's identity was never consulted.
```
