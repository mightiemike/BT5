### Title
`SwapAllowlistExtension` gates on the router's address instead of the real swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router address**, not the user. A pool admin who allowlists the router to enable periphery access inadvertently opens the allowlist to every user on the internet.

---

### Finding Description

The call chain is:

```
User → MetricOmmSimpleRouter.exactInputSingle(...)
         └─ pool.swap(recipient, ...)          // msg.sender = router
               └─ _beforeSwap(msg.sender=router, ...)
                     └─ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                           └─ allowedSwapper[pool][router]  ← checked, NOT the user
```

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim:

```solidity
// ExtensionCalling.sol line 162-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)   // sender = router
)
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct), but `sender` is the router (wrong). The allowlist lookup is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Two broken outcomes result:**

1. **Bypass (high impact):** The pool admin allowlists the router so that allowlisted users can use the standard periphery. This makes `allowedSwapper[pool][router] = true`. Now *any* user — including non-allowlisted ones — can call `router.exactInputSingle(...)` and the extension passes, because it only sees `sender = router`.

2. **Broken functionality:** If the admin does *not* allowlist the router, allowlisted users cannot use `MetricOmmSimpleRouter` at all. The extension always sees `sender = router` (not allowlisted) and reverts, making the supported periphery unusable for curated pools.

Note that `DepositAllowlistExtension` does **not** share this bug — it correctly checks `owner` (the position owner, an explicit parameter), not `msg.sender` of the pool call.

---

### Impact Explanation

A pool using `SwapAllowlistExtension` is a curated pool: the admin intends to restrict trading to specific counterparties (e.g., KYC'd users, whitelisted market makers, or a specific strategy). If the router is allowlisted (the natural operational step to enable periphery access), the curation is completely defeated. Any user can trade on the pool by routing through `MetricOmmSimpleRouter`. LP principals are exposed to counterparties the pool was explicitly designed to exclude, causing direct LP loss if the pool's profitability depends on counterparty selection.

---

### Likelihood Explanation

The trigger is a single, natural admin action: `setAllowedToSwap(pool, router, true)`. Any pool admin who wants allowlisted users to use the router will take this step. The exploit requires no special privileges — any EOA can call `MetricOmmSimpleRouter.exactInputSingle`. The router is a production, in-scope periphery contract explicitly listed in scope.

---

### Recommendation

Pass the original transaction initiator (`tx.origin`) or, better, thread the real user identity through the swap path. The cleanest fix is to have the pool pass the `recipient` (or a dedicated `originator` field) as the identity the allowlist checks, or to require that `extensionData` carries a signed user identity. Alternatively, document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the factory level by rejecting pools that configure both a swap allowlist and a non-zero `beforeSwapOrder` when the router is a known caller.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, only alice is allowlisted
swapAllowlist.setAllowedToSwap(pool, alice, true);
// Admin allowlists the router so alice can use periphery
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attack: bob (not allowlisted) bypasses the guard via router
vm.prank(bob);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: token0,
        recipient: bob,
        zeroForOne: true,
        amountIn: 1000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// ✓ succeeds — extension saw sender=router (allowlisted), not bob (not allowlisted)
// bob traded on a curated pool he was explicitly excluded from
