### Title
`DepositAllowlistExtension` Validates `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist â€” (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument and checks the `owner` argument instead. Because `owner` is freely chosen by the caller of `addLiquidity`, any unprivileged address can bypass the allowlist by supplying an already-allowed address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` dispatches the before-hook with the actual caller as the first argument and the caller-supplied `owner` as the second:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, â€¦)` but discards `sender` (unnamed first parameter) and gates only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The contract's own NatSpec and `setAllowedToDeposit` name the gated entity a **depositor** â€” the party calling `addLiquidity` â€” not the position owner:

```solidity
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
``` [4](#0-3) 

Because `owner` is a free argument supplied by the caller, any address can pass the guard by setting `owner = allowedAddress`.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), not `recipient`:

```solidity

    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [5](#0-4) 

The asymmetry confirms the deposit check is wrong.

### Impact Explanation

A pool admin deploys a permissioned pool with `DepositAllowlistExtension` to restrict who may provide liquidity (e.g., a KYC-gated or institutional pool). Any unprivileged address bypasses this control entirely by calling:

```solidity
pool.addLiquidity(allowedAddress, salt, deltas, callbackData, extensionData);
```

The hook sees `owner = allowedAddress` (which is on the allowlist) and returns the correct selector. The pool proceeds to credit the position to `allowedAddress` while collecting tokens from the unauthorized caller via the swap callback. The unauthorized caller can:

- Add liquidity to arbitrary bins in a pool that is supposed to be permissioned, breaking the admin-configured access boundary.
- Manipulate bin balances and the pool's internal price curve (`curBinIdx`, `curPosInBin`) in ways the pool admin did not authorize.
- Force an allowed address to hold an unwanted LP position, which only that address can remove â€” a griefing vector against legitimate LPs.

This is a direct admin-boundary break: a pool-admin-configured guard is bypassed by an unprivileged path with no special role or token required.

---

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a publicly observable allowed address as `owner`. No flash loan, oracle manipulation, or privileged role is needed. Any address can trigger it at any time against any pool using `DepositAllowlistExtension`.

---

### Recommendation

Check `sender` (the actual caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`.

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as an extension; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Only `alice` is supposed to deposit.
3. Attacker (not on allowlist) calls:
   ```solidity
   pool.addLiquidity(
       alice,          // owner â€” alice IS on the allowlist
       salt,
       deltas,
       callbackData,
       extensionData
   );
   ```
4. `beforeAddLiquidity` is invoked with `sender = attacker`, `owner = alice`.
5. The hook checks `allowedDepositor[pool][alice]` â†’ `true`. Guard passes.
6. Pool credits the LP position to `alice`; attacker's tokens are pulled via callback.
7. Attacker has successfully deposited into a permissioned pool without being on the allowlist. The deposit allowlist is fully neutralized.
