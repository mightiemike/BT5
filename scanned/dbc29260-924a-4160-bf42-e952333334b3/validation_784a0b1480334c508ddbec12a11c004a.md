### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Allowlist Bypass and Breaking Legitimate Vault-Owner Deposits — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the deposit allowlist against the `owner` parameter (the LP-position recipient address) rather than the `sender` parameter (the actual caller who provides tokens via callback). Because `owner` is a free, caller-controlled argument with no on-chain restriction, the guard is misbound: an unauthorized caller can bypass it by nominating an allowlisted address as `owner`, and a legitimately authorized caller is blocked whenever they want to assign the LP position to an address (e.g., a vault or multisig) that is not itself in the allowlist.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a free `owner` parameter and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values faithfully to the extension: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first argument (`sender` — the actual caller) is silently discarded (unnamed `address`), and the allowlist lookup is performed on `owner` instead: [3](#0-2) 

The allowlist state is keyed `allowedDepositor[pool][depositor]` and is set via `setAllowedToDeposit`: [4](#0-3) 

There is no restriction in `addLiquidity` preventing a caller from supplying an arbitrary `owner` address: [1](#0-0) 

By contrast, `removeLiquidity` correctly enforces `msg.sender == owner`, so LP positions can only be withdrawn by the owner: [5](#0-4) 

The `SwapAllowlistExtension` does not share this flaw — it correctly checks `sender` (the actual caller): [6](#0-5) 

---

### Impact Explanation

**Path A — Allowlist bypass (unauthorized deposit):**
An address not in the allowlist calls `pool.addLiquidity(owner = allowedAddress, ...)`. The extension checks `allowedDepositor[pool][allowedAddress]` → `true` → passes. The unauthorized caller provides tokens via the swap callback; the LP position is credited to `allowedAddress`. The pool admin's intent to restrict who may inject liquidity is violated. Any regulatory, KYC, or risk-isolation purpose of the allowlist is defeated.

**Path B — Broken core functionality for authorized depositors:**
An authorized depositor (Alice, in the allowlist) calls `pool.addLiquidity(owner = vaultContract, ...)` where `vaultContract` is a legitimate LP-management contract not individually listed. The extension checks `allowedDepositor[pool][vaultContract]` → `false` → reverts with `NotAllowedToDeposit`. Alice cannot use any vault, multisig, or position-manager as the LP-position owner, making the vault-owner pattern entirely unusable for allowlisted pools. This is the direct analog of the Debita finding: a valid protocol actor (the vault) is not in the checked set, so a core liquidity flow is permanently broken.

---

### Likelihood Explanation

Any pool that deploys `DepositAllowlistExtension` with a non-trivial allowlist (i.e., `allowAllDepositors == false`) is immediately affected. The bypass requires only a standard `addLiquidity` call with a crafted `owner` argument — no special permissions, no flash loans, no oracle manipulation. Path B is triggered by any authorized depositor who attempts the common pattern of depositing into a vault-owned position.

---

### Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
// Before (wrong — checks LP-position recipient, not actual caller)
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}

// After (correct — checks actual caller)
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins who need to allow vault-owner patterns should allowlist the vault's caller address (the `sender`), not the vault contract itself.

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  allowedDepositor[pool][Bob]   = false  (Bob is unauthorized)
  allowedDepositor[pool][Vault] = false  (Vault is not listed)

--- Path A: Bypass ---
Bob calls: pool.addLiquidity(owner=Alice, salt=0, deltas=..., ...)
  Extension receives: beforeAddLiquidity(sender=Bob, owner=Alice, ...)
  Check: allowedDepositor[pool][Alice] == true  → PASSES
  Result: Bob's tokens enter the pool; Alice receives the LP shares.
  Bob has bypassed the allowlist.

--- Path B: Broken functionality ---
Alice calls: pool.addLiquidity(owner=Vault, salt=0, deltas=..., ...)
  Extension receives: beforeAddLiquidity(sender=Alice, owner=Vault, ...)
  Check: allowedDepositor[pool][Vault] == false → REVERTS NotAllowedToDeposit
  Result: Alice (an authorized depositor) cannot assign her LP position
          to a vault contract. Core liquidity flow is broken.
```
