### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which is always `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the end user. If the pool admin allowlists the router (the natural step to enable router-mediated swaps for their curated users), every unprivileged user can bypass the allowlist by routing through the router.

### Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)`. At this point `msg.sender` to the pool is the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`. [1](#0-0) 

4. `ExtensionCalling._beforeSwap` encodes `sender = router` and dispatches to the extension. [2](#0-1) 

5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][router]`. [3](#0-2) 

The pool admin intends to gate individual user addresses. They add KYC'd users: `allowedSwapper[pool][kyc_user] = true`. To also allow those users to swap through the standard periphery router, the admin must add the router: `allowedSwapper[pool][router] = true`. The moment the router is allowlisted, **every** user — including non-KYC'd ones — can call `router.exactInputSingle(pool, ...)` and the extension passes because it only sees the router address.

The router never passes the originating user's address as `sender`; it always calls `pool.swap()` directly, making itself `msg.sender`. [4](#0-3) [5](#0-4) 

### Impact Explanation
Any user can trade on a pool whose swap allowlist was configured to restrict access to a curated set of addresses. LP funds are exposed to unauthorized counterparties. The pool's curation policy — the only mechanism protecting LPs from unwanted flow — is silently voided. This is a direct broken-core-functionality impact: the allowlist guard fails open for every router-mediated swap once the router is allowlisted.

### Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery swap path documented and deployed by the protocol. Pool admins who want their allowlisted users to access the router (the normal UX) must add the router to the allowlist. This is the obvious operational step, and it is the step that triggers the bypass. Any non-allowlisted user who discovers the router is allowlisted can immediately exploit it with a single `exactInputSingle` call.

### Recommendation
The extension must gate on the **economic actor**, not the intermediary. Two sound approaches:

1. **Pass the originating user through the router.** Add a `swapper` parameter to the pool's `swap()` signature (separate from `msg.sender`) that the router populates with `msg.sender` before calling the pool. The extension then checks that field. This requires a core change.

2. **Check `msg.sender` of the pool call inside the extension, not the forwarded `sender`.** Since the extension already receives `msg.sender` as the pool (used for the allowlist key), the pool could store the originating caller in transient storage before dispatching to extensions, and expose it via a view. The extension reads that slot instead of the forwarded `sender` argument.

3. **Short-term mitigation:** Document that pools using `SwapAllowlistExtension` must not allowlist the router, and that allowlisted users must call `pool.swap()` directly. This breaks the standard UX but preserves the guard.
