### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` gates on the `owner` parameter (the position recipient) rather than the `sender` parameter (the actual caller paying tokens). Any unprivileged address can bypass the allowlist by naming an allowlisted address as `owner`, causing unauthorized liquidity to enter a restricted pool and breaking the admin-configured access-control invariant.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with both the real caller (`msg.sender` → `sender`) and the position recipient (`owner`):

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension hook receives both parameters but silently discards `sender` and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The first parameter (`sender`) is unnamed and unused. The check `allowedDepositor[msg.sender][owner]` passes whenever `owner` is on the allowlist, regardless of who `sender` is.

Compare with `SwapAllowlistExtension`, which correctly checks `sender`:

```solidity
// SwapAllowlistExtension.sol line 31-38
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [3](#0-2) 

The asymmetry confirms the deposit extension is checking the wrong actor.

After the hook passes, `LiquidityLib.addLiquidity` credits the position to `owner` and pulls tokens from `msg.sender` (the actual caller) via the callback:

```solidity
// LiquidityLib.sol lines 147-154
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
``` [4](#0-3) 

And `removeLiquidity` enforces `msg.sender == owner`, so only the named `owner` can withdraw:

```solidity
// MetricOmmPool.sol line 206
if (msg.sender != owner) revert NotPositionOwner();
``` [5](#0-4) 

---

### Impact Explanation

An unauthorized address (not on the allowlist) calls `addLiquidity(owner = allowlisted_address, ...)`. The extension checks `allowedDepositor[pool][allowlisted_address]` → `true` → hook passes. The unauthorized caller pays tokens via the callback; the position is credited to `allowlisted_address`. The allowlisted address can then call `removeLiquidity` and withdraw those tokens.

With collusion between the unauthorized depositor and any allowlisted address, the allowlist is completely neutralized: unauthorized parties can provide liquidity to restricted pools, breaking the pool admin's access-control invariant. Restricted pools (e.g., KYC-gated, institutional-only) receive unauthorized liquidity, diluting fee share among existing LPs and violating the pool's intended participation rules.

---

### Likelihood Explanation

The trigger requires only a valid allowlisted `owner` address (publicly readable from `allowedDepositor`) and a router or direct call to `addLiquidity`. No privileged role is needed. Any address can execute this with a single transaction. Likelihood is **high** once a restricted pool with at least one allowlisted depositor exists.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller) instead of `owner` (the position recipient), matching the intent stated in the NatSpec ("Gates `addLiquidity` by depositor address") and the pattern used in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` configured; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner=alice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
4. `_beforeAddLiquidity(sender=bob, owner=alice, ...)` is called.
5. `DepositAllowlistExtension.beforeAddLiquidity` checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. `LiquidityLib.addLiquidity` credits shares to `alice`, calls `metricOmmModifyLiquidityCallback` on Bob, Bob pays tokens.
7. Alice calls `removeLiquidity(owner=alice, ...)` and withdraws Bob's tokens.
8. Bob has lost tokens; Alice has gained them; the allowlist was never enforced against the actual depositor.
