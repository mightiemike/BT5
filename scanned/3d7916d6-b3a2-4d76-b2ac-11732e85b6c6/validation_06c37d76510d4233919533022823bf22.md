### Title
`SwapAllowlistExtension` Bypass via Router Intermediary — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which is the direct `msg.sender` of `pool.swap`. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router address, not the actual user. If the pool admin allowlists the router (required for any router-mediated swap to succeed), every user on the network can bypass the per-user allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool) and `sender` is the first argument forwarded from `MetricOmmPool.swap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` passes this value unchanged into the ABI-encoded call to the extension: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
``` [4](#0-3) 

At this point `msg.sender` of `pool.swap` is the **router contract**, not the actual user. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The same identity substitution occurs for `exactInput`, `exactOutputSingle`, and `exactOutput`, and also inside the recursive `_exactOutputIterateCallback` which calls `pool.swap` again with `msg.sender` = the pool (acting as callback recipient), further obscuring the real user. [5](#0-4) 

**The structural trap:** For any allowlisted user to swap via the router, the pool admin must add the router to the allowlist. The moment the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether that caller is individually allowlisted. The guard is structurally incapable of simultaneously (a) permitting allowlisted users to use the router and (b) blocking non-allowlisted users from using the router.

---

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., only specific market makers, only KYC'd addresses, or only protocol-owned bots) loses that restriction entirely once the router is added to the allowlist. Any unprivileged user can call `exactInputSingle` or `exactInput` through the public router and execute swaps against the pool. Depending on the pool's oracle-anchored pricing and LP composition, this can result in:

- Unauthorized extraction of LP assets at oracle-quoted prices by parties the pool admin explicitly intended to exclude.
- Circumvention of any rate-limiting or counterparty-risk controls the allowlist was meant to enforce.

---

### Likelihood Explanation

The router is a public, permissionless contract. Any user can call it. The bypass is triggered whenever the pool admin adds the router to the allowlist — a step they are forced to take if they want allowlisted users to be able to use the standard periphery. There is no on-chain signal distinguishing "router allowlisted to serve specific users" from "router allowlisted to serve everyone." The condition is reachable by any unprivileged address with no special setup beyond knowing the router address.

---

### Recommendation

The extension must gate on the **economic actor** (the human or contract that initiated the trade), not the immediate `msg.sender` of `pool.swap`. Two viable approaches:

1. **Extension-data forwarding:** The router encodes the real user's address into `extensionData`; the extension decodes and verifies it. The extension must also verify that `sender` is a trusted router before trusting the embedded address, otherwise any caller can forge it.

2. **Router-aware allowlist:** Add a separate `allowedRouter` mapping. When `sender` is a known router, decode the real user from `extensionData` and check that address instead. Reject calls where `sender` is a router but no valid user address is embedded.

The `DepositAllowlistExtension` avoids this problem by gating on `owner` (the position recipient) rather than `sender` (the operator/payer), which is the correct identity for deposit access control. [6](#0-5)

### Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension in beforeSwap slot
  admin calls setAllowedToSwap(pool, allowedMarketMaker, true)
  admin calls setAllowedToSwap(pool, address(router), true)   // required for MM to use router
  LP adds liquidity

Attack:
  unauthorizedUser calls router.exactInputSingle({
      pool: pool,
      recipient: unauthorizedUser,
      zeroForOne: true,
      amountIn: X,
      ...
  })

  Execution trace:
    router.exactInputSingle  (msg.sender = unauthorizedUser)
      pool.swap(...)         (msg.sender = router)
        _beforeSwap(sender=router, ...)
          extension.beforeSwap(sender=router, ...)
            allowedSwapper[pool][router] == true  → PASSES
        swap executes, tokens transferred to unauthorizedUser

Result: unauthorizedUser swaps successfully despite not being individually allowlisted.
```
