### Title
SwapAllowlistExtension Gates on Router Address Instead of Actual Swapper, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address, not the actual user. If the router is allowlisted for a pool (required for router-mediated swaps to function), every user — including those not individually allowlisted — can bypass the per-user swap allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then enforces the allowlist against that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter` (the public-facing router), the router calls `pool.swap(...)`. At that point `msg.sender` inside the pool is the **router**, so `sender` delivered to the extension is the router's address, not the originating user. The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

This creates an irreconcilable conflict:

| Scenario | `sender` seen by extension | Result |
|---|---|---|
| User swaps directly | user address | correctly gated |
| User swaps via router, router NOT allowlisted | router address | all router users blocked, even allowlisted ones |
| User swaps via router, router IS allowlisted | router address | **all users bypass per-user allowlist** |

For the router to be usable at all on an allowlisted pool, the pool admin must allowlist the router. Once the router is allowlisted, the per-user gate is completely inoperative for router-mediated swaps.

The `DepositAllowlistExtension` does not share this flaw because `addLiquidity` accepts an explicit `owner` parameter that the liquidity adder sets to the actual user, so the deposit hook always sees the real depositor regardless of who the `msg.sender` is. [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` for compliance, KYC, or access-control purposes has its entire swap gate nullified for any user who routes through `MetricOmmSimpleRouter`. Any non-allowlisted address can execute swaps against a pool that was explicitly configured to restrict them. This breaks a core pool-level access-control invariant and constitutes a broken core pool functionality with direct fund-flow consequences (unauthorized parties drain or manipulate restricted liquidity pools).

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public-facing swap entry point documented and deployed alongside the protocol. Any pool that enables `SwapAllowlistExtension` and also needs to support router-based swaps (the normal operational mode) must allowlist the router, triggering the bypass automatically. No special privileges, flash loans, or exotic tokens are required — a standard router call suffices.

---

### Recommendation

Pass the originating user's address through the router to the pool, and have the pool forward it to extensions as a verified `originSender`. One concrete approach: the router encodes the real user address into `extensionData`, and the extension reads it from there after verifying `msg.sender` is a trusted router. Alternatively, the pool can accept an explicit `swapper` parameter (analogous to `owner` in `addLiquidity`) that the router sets to `msg.sender` before calling the pool, and extensions gate on that field rather than the pool-level `msg.sender`.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension as the beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so that router-mediated swaps are not universally blocked.
3. Pool admin does NOT call setAllowedToSwap(pool, alice, true).
4. Alice (non-allowlisted) calls MetricOmmSimpleRouter.exactInput(...) targeting the pool.
5. Router calls pool.swap(recipient, ...) — msg.sender to pool = router.
6. Pool calls _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] = true.
7. Swap executes successfully despite Alice never being allowlisted.
8. Direct call: alice calls pool.swap(...) directly → extension checks allowedSwapper[pool][alice] = false → reverts NotAllowedToSwap.
```

Alice bypasses the allowlist exclusively by routing through the public router, mirroring the Pet.sol pattern where an alternate code path (Custodian/Minter transfer) circumvented the role restriction intended for the Cloner.
