### Title
`SwapAllowlistExtension` per-user allowlist is fully bypassed when swaps are routed through `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`. When `MetricOmmSimpleRouter` is the caller, `sender` is the router address, not the originating EOA. `SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]`, so it evaluates the router's allowlist status, not the actual user's. Any pool that allowlists the router (to permit router-mediated swaps) inadvertently grants swap access to every user of that router.

### Finding Description
`SwapAllowlistExtension` is documented as gating `swap` by swapper address, per pool. The extension receives `sender` from the pool, which is always `msg.sender` of the `swap()` call. `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly, making the router the `msg.sender`. The extension has no way to recover the originating EOA. A pool configured with `allowedSwapper[pool][router] = true` and no individual EOA entries will pass the allowlist check for every user who routes through the router.

### Impact Explanation
The `SwapAllowlistExtension`'s access-control invariant ŌĆö that only explicitly allowlisted addresses may swap ŌĆö is broken for any pool that permits router-mediated swaps. Pools intended to restrict trading to KYC'd, institutional, or otherwise vetted addresses are silently open to all router users. This is broken core pool functionality: the extension's primary purpose (gating swaps) does not function as intended when the standard periphery router is used.

### Likelihood Explanation
The router is the primary user-facing entry point for swaps. Any pool operator who deploys a `SwapAllowlistExtension` and allowlists the router (the natural way to "allow swaps via the official router") will unknowingly expose the pool to all router users. The configuration is non-obvious and the extension provides no warning.

### Recommendation
`SwapAllowlistExtension.beforeSwap()` should check the `recipient` or require the pool to pass `tx.origin` (with appropriate trust assumptions), or the router should accept a `swapper` parameter and forward it as `extensionData` for the extension to decode. Alternatively, the extension's documentation should explicitly warn that allowlisting a router address grants access to all users of that router, and the recommended pattern should be to allowlist individual EOAs only.
