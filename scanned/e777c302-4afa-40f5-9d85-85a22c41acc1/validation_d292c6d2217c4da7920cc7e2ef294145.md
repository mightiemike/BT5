### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, making the allowlist trivially bypassable or permanently broken for router-mediated swaps - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the end user. The extension therefore checks whether the **router** is allowlisted, not whether the actual swapper is allowlisted. This makes the allowlist either trivially bypassable (if the router is allowlisted, every user bypasses it) or permanently broken (if the router is not allowlisted, every individually-allowlisted user is blocked).

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [4](#0-3) 

At that point `msg.sender` to the pool is the **router address**, so `sender` delivered to the extension is the router, not the end user. The extension has no visibility into who originally called the router.

The same mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, and to the recursive hop inside `_exactOutputIterateCallback` where the pool is called again with `msg.sender` = router: [5](#0-4) 

---

### Impact Explanation

Two mutually exclusive failure modes, both fund-impacting:

**Mode A – Allowlist bypass (High):** The pool admin allowlists the router so that users can reach the pool through the supported periphery path. Because the extension checks `allowedSwapper[pool][router]`, every user who calls the router is treated as allowlisted regardless of their individual status. Any non-allowlisted address can swap on a curated pool by routing through `MetricOmmSimpleRouter`, draining liquidity that the pool admin intended to restrict.

**Mode B – Allowlisted users permanently blocked (Medium):** The pool admin does not allowlist the router (only individual users). Every individually-allowlisted user who calls the router is blocked because the extension sees the router address, which is not in the allowlist. The pool's swap functionality is broken for all router-mediated paths, which is the only supported periphery swap path.

Both outcomes break the core invariant that a curated pool enforces the same allowlist policy regardless of which supported public entrypoint reaches it.

---

### Likelihood Explanation

The router is the primary and documented user-facing swap entrypoint. Any pool that deploys `SwapAllowlistExtension` and expects users to swap through the router will immediately encounter one of the two failure modes. No special timing, privileged access, or unusual token behavior is required. A single public `exactInputSingle` call is sufficient to trigger the bug.

---

### Recommendation

The extension must check the **economically relevant actor** — the address that initiated the swap and will pay for it — not the intermediate contract that called the pool. Two options:

1. **Pass the original initiator through the router.** Have the router store `msg.sender` in transient storage (analogous to how it already stores the payer in `_setNextCallbackContext`) and expose it so the extension can read it. The pool would need to forward this value, or the extension would read it directly from the router's transient slot.

2. **Check `sender` only for direct pool calls; require the router to forward the real user.** Add a `recipient`-or-`payer` field to the extension data that the router populates with `msg.sender`, and have the extension validate that field when `sender` is a known router address.

The simplest safe fix is option 1: store the original `msg.sender` in transient storage at router entry and have the extension read it, so the checked identity is always the address that economically initiated the swap.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  admin does NOT allowlist bob
  admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so users can swap

Attack (Mode A – bypass):
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)         // msg.sender to pool = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap(router, ...)
    → checks allowedSwapper[pool][router] == true    ✓ passes
    → bob's swap executes despite not being allowlisted

Broken invariant (Mode B – alice blocked):
  admin does NOT allowlist router
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)         // msg.sender to pool = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap(router, ...)
    → checks allowedSwapper[pool][router] == false   ✗ reverts NotAllowedToSwap
    → alice is blocked even though she is individually allowlisted
```
