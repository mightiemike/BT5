### Title
SwapAllowlistExtension Checks Router Address as Swapper Instead of Actual User, Enabling Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the `sender` forwarded to the extension is the router address—not the originating user. If the pool admin allowlists the router (which is required for any allowlisted user to use the router), every unpermissioned address can bypass the curated allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool via `ExtensionCalling._callExtensionsInOrder`). `sender` is the first argument the pool passes when it calls `_beforeSwap`, which is the pool's own `msg.sender`—i.e., whoever called `pool.swap()`.

`ExtensionCalling._beforeSwap` encodes `sender` directly from the pool's caller:

```solidity
// metric-core/contracts/ExtensionCalling.sol L151-177
function _beforeSwap(
    address sender,   // ← pool's msg.sender
    address recipient,
    ...
) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
    );
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the router. Therefore the extension receives `sender = router`, and the check becomes:

```
allowedSwapper[pool][router]
```

not

```
allowedSwapper[pool][actual_user]
```

**The invariant break**: the allowlist is supposed to gate the economically relevant actor (the user who initiates and pays for the swap). Instead it gates the intermediary contract. A pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist. Once the router is allowlisted, every address on-chain can bypass the curated gate by calling `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` through the router.

The same structural flaw applies to multi-hop `exactInput` intermediate hops, where the router is also the direct caller of each intermediate pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L103-112
_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
    .swap(
        i == last ? params.recipient : address(this),
        zeroForOne,
        amount,
        ...
    );
```

For hops `i > 0`, the payer context is `address(this)` (the router itself), and the pool's `msg.sender` is still the router—so the extension sees `sender = router` for every hop.

---

### Impact Explanation

Any address can swap in a pool that is intended to be restricted to a curated set of users (e.g., KYC-gated, institutional-only, or whitelist-only pools). The bypass is achieved through the official, production-deployed `MetricOmmSimpleRouter`. The attacker does not need any special privilege—only the ability to call the router. Consequences include:

- Unauthorized users executing swaps against LP capital that was deposited under the assumption of a restricted counterparty set.
- LP providers suffering adverse selection or regulatory exposure from trades they did not consent to.
- Complete nullification of the `SwapAllowlistExtension` guard whenever the router is allowlisted.

This is a direct loss-of-pool-integrity / broken-core-functionality impact: the configured protection hook fails open for all router-mediated swaps.

---

### Likelihood Explanation

**High.** The `MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who deploys a swap-allowlisted pool and wants their allowlisted users to be able to use the router must add the router to the allowlist. This is the natural operational path. Once done, the bypass is unconditional and requires no further attacker action beyond calling the router.

---

### Recommendation

The extension must check the actual originating user, not the direct caller of the pool. Two complementary fixes:

1. **Extension-side**: The `beforeSwap` hook should check `sender` only when `sender` is not a known router/intermediary, or the pool should forward the true originator. A cleaner approach is to have the router pass the real user's address as part of `extensionData` and have the extension decode it—but this requires router cooperation and is fragile.

2. **Router-side (preferred)**: The router should pass the real user (`msg.sender` of the router call) as the `sender` argument to the pool's `swap` function, rather than having the pool use its own `msg.sender`. This requires a pool-level change to accept an explicit `sender` override from trusted periphery, or the extension must be redesigned to receive the true originator through `extensionData`.

3. **Immediate mitigation**: Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used with direct pool calls until the actor-binding is corrected.

### Proof of Concept

```
Setup:
  - Pool P is deployed with SwapAllowlistExtension E configured on beforeSwap.
  - Pool admin allowlists Alice: allowedSwapper[P][alice] = true.
  - Pool admin allowlists the router R so Alice can use it: allowedSwapper[P][R] = true.

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, recipient: bob, ...}).
  2. Router calls P.swap(bob, zeroForOne, amount, ...) — pool's msg.sender = R.
  3. Pool calls ExtensionCalling._beforeSwap(sender=R, recipient=bob, ...).
  4. Extension checks: allowedSwapper[P][R] == true → passes.
  5. Swap executes. Bob receives output tokens from the allowlist-gated pool.

Result: Bob, who is not on the allowlist, successfully swaps against LP capital
        in a pool that was configured to restrict swaps to Alice only.

      if (extensionIndex == 0) break;

      order >>= 3;
    }
  }
