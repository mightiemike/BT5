### Title
`DepositAllowlistExtension.beforeAddLiquidity` Guards on `owner` Instead of `sender`, Allowing Unauthorized Callers to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by the depositor's address. Its `beforeAddLiquidity` hook silently ignores the `sender` argument (the actual caller who provides tokens) and instead checks `owner` (the position beneficiary). Any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowlisted_address, ...)`, naming an already-approved address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook: [1](#0-0) 

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller who will be asked to supply tokens via the swap-callback; `owner` is an arbitrary address supplied by the caller that will receive the minted position shares.

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both: [2](#0-1) 

```
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first parameter but discards it (unnamed `address`), then gates on `owner`: [3](#0-2) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The admin-facing setter names its parameter `depositor`, confirming the intended subject of the check is the caller, not the beneficiary: [4](#0-3) 

The sister extension `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper) and ignores `recipient`, establishing the intended pattern: [5](#0-4) 

---

### Impact Explanation

An unprivileged address `Bob` calls:

```
pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)
```

where `alice` is in the allowlist and `Bob` is not. The hook checks `allowedDepositor[pool][alice]` → passes. The pool then calls `IMetricOmmSwapCallback(Bob).metricOmmSwapCallback(...)` to collect tokens from Bob. Bob's tokens enter the pool; Alice receives the position shares.

Consequences:
- The deposit allowlist — the pool admin's primary tool for restricting who may supply liquidity — is fully bypassed by any unprivileged caller.
- Tokens from non-allowlisted sources enter a pool that was explicitly configured to exclude them (compliance, KYC, or market-maker-only pools).
- Because `removeLiquidity` enforces `msg.sender == owner`, only Alice can withdraw; Bob's tokens are permanently locked unless Alice cooperates, creating a griefing vector against the caller and an unwanted position forced onto Alice. [6](#0-5) 

---

### Likelihood Explanation

The bypass requires no special privilege: any EOA or contract can call `addLiquidity` with an allowlisted address as `owner`. The allowlist is publicly readable (`allowedDepositor` is a public mapping), so an attacker can trivially identify a valid `owner` to use. The only cost is the tokens the attacker must supply via the callback.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the check targets the address that actually provides tokens, consistent with the `setAllowedToDeposit` / `depositor` naming and the `SwapAllowlistExtension` pattern.

---

### Proof of Concept

```solidity
// Pool has DepositAllowlistExtension configured.
// Pool admin has called: extension.setAllowedToDeposit(pool, alice, true);
// Bob is NOT in the allowlist.

// Bob calls addLiquidity naming alice as owner:
pool.addLiquidity(
    alice,          // owner — passes the allowlist check
    salt,
    deltas,
    callbackData,   // Bob's contract supplies tokens here
    extensionData
);
// Result: Bob's tokens enter the pool, alice receives the position.
// The allowlist guard never checked Bob.
```
