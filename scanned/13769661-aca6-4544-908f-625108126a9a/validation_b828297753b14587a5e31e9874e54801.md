### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, the pool's `swap` is called by the router, so `sender = router address`. If the pool admin allowlists the router (a natural step to enable router-mediated swaps for allowlisted users), every non-allowlisted user can bypass the guard by routing through the router.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

In `MetricOmmPool.swap`, the `_beforeSwap` dispatcher is called with `msg.sender` as the first argument:

```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
``` [1](#0-0) 

**Step 2 — Router calls `pool.swap` directly, so `msg.sender` of the pool is the router.**

`MetricOmmSimpleRouter.exactInputSingle` calls:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
``` [2](#0-1) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [3](#0-2) 

**Step 3 — The allowlist check is keyed on `sender`, which is the router, not the user.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [4](#0-3) 

`msg.sender` here is the pool (correct). `sender` is the router (wrong — it should be the actual user). The effective check is `allowedSwapper[pool][router]`.

**The admin faces an impossible choice:**

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all (broken supported path) |
| **Allowlist the router** | Every non-allowlisted user bypasses the guard by routing through the router |

The admin who wants allowlisted users to be able to use the router will naturally allowlist the router, unknowingly opening the pool to all users.

---

### Impact Explanation

Once the router is allowlisted, any non-allowlisted address calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting the curated pool. The `beforeSwap` hook sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and allows the swap. The curation mechanism is fully bypassed. For pools designed to restrict swaps to trusted market makers or KYC-verified counterparties, this exposes LP capital to adversarial or unintended swap flow — a direct loss-of-funds risk for LPs.

---

### Likelihood Explanation

Medium. The admin must explicitly allowlist the router, but this is the natural and expected action when deploying a curated pool that is intended to be used with the protocol's own periphery. The bypass requires no special privilege or unusual setup beyond that single admin call.

---

### Recommendation

The extension must check the **economically responsible actor** — the address that initiated the trade — not the intermediate caller. Two viable fixes:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and extension.

2. **Document incompatibility**: Clearly state in the extension's NatSpec that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and must only be used with direct pool calls. Enforce this at the factory level by rejecting pools that configure both the router as an allowed caller and the swap allowlist extension.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Admin: setAllowedToSwap(pool, user1, true)       // allowlist user1
3. Admin: setAllowedToSwap(pool, router, true)       // allowlist router so user1 can use it
4. user2 (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...); pool calls _beforeSwap(router, ...)
6. Extension checks allowedSwapper[pool][router] == true → passes
7. user2 swaps successfully — allowlist bypassed.
```

The root cause is that `sender` in `beforeSwap` is `msg.sender` of the pool's `swap` call, which is the router, not the originating user. [4](#0-3) [1](#0-0) [5](#0-4)
