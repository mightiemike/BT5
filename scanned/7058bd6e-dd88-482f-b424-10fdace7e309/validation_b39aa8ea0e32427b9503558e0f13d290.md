### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router contract**, not the actual user. If the pool admin allowlists the router (the only way to let users swap through it), every user — including those not individually allowlisted — can bypass the swap allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool's `_beforeSwap` hook, which passes `msg.sender` of `pool.swap()`:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), e.g. the router
    recipient,
    ...
);
```

The extension then checks:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) calls `pool.swap()`, `sender` is the **router address**, not the EOA user. The allowlist check therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates an irreconcilable dilemma for the pool admin:

| Admin action | Effect |
|---|---|
| Allowlist the router | Every user (including non-allowlisted ones) can swap through the router — allowlist bypassed |
| Do not allowlist the router | Individually allowlisted users cannot use the router — core swap path broken |

There is no configuration that achieves "only allowlisted users may swap through the router."

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position owner, the actual user), not `sender` (the intermediary caller):

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

The asymmetry between the two extensions confirms this is an unintended design flaw in `SwapAllowlistExtension`.

---

### Impact Explanation

**Medium-High.** A pool configured with `SwapAllowlistExtension` (e.g., a KYC-gated or access-restricted pool) is rendered either:

1. **Fully bypassable**: If the router is allowlisted, any unprivileged user can execute swaps against the restricted pool by calling `MetricOmmSimpleRouter`, draining liquidity or executing trades the pool admin intended to block.
2. **Broken for legitimate users**: If the router is not allowlisted, individually allowlisted users cannot use the standard periphery swap path, making the pool's primary user-facing interface unusable.

Both outcomes violate the core invariant that the allowlist restricts swaps to approved addresses.

---

### Likelihood Explanation

**Medium.** The `SwapAllowlistExtension` is a production periphery contract explicitly designed for deployment. Any pool that activates it and expects users to route through `MetricOmmSimpleRouter` (the standard swap interface) is affected. The bypass requires no special privileges — any user can call the router.

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to accept the actual user identity via `extensionData`, or restructure the hook to check `recipient` (if that maps to the user) or require the router to encode the originating user in `extensionData` and verify it. The simplest fix mirrors `DepositAllowlistExtension`: check a field that represents the actual user, not the intermediary caller.

Alternatively, document that `sender` in `beforeSwap` is always the direct pool caller and require integrators to allowlist routers rather than individual users — but then provide a separate mechanism (e.g., a router-level allowlist) to enforce per-user restrictions.

---

### Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` active on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow users to swap via `MetricOmmSimpleRouter`.
3. Pool admin does **not** allowlist `alice` (an unprivileged user).
4. `alice` calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. The router calls `pool.swap(recipient, ...)` — `msg.sender` of `pool.swap()` is the router.
6. `_beforeSwap(router, recipient, ...)` is called; `sender = router`.
7. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true`.
8. The swap succeeds. `alice` has bypassed the allowlist.
