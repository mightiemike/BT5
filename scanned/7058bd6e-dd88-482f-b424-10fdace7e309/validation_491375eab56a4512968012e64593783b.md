### Title
`DepositAllowlistExtension` gates position `owner` but not the actual depositor (`sender`), allowing any unprivileged user to bypass the deposit allowlist via `MetricOmmPoolLiquidityAdder` - (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` argument (the actual caller of `addLiquidity`) and only checks `owner` (the position recipient). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a fully caller-controlled `owner` address, any non-allowlisted user can bypass the deposit gate by supplying an allowlisted address as `owner`, paying the tokens themselves, and causing the pool to mint LP shares for that allowlisted address.

---

### Finding Description

**Step 1 – The allowlist checks the wrong identity.**

`DepositAllowlistExtension.beforeAddLiquidity` receives two identity arguments: `sender` (the `msg.sender` of the pool's `addLiquidity` call, i.e. the actual depositor) and `owner` (the position recipient). The implementation discards `sender` entirely and only gates on `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}

**Step 2 – The pool passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`.**

```solidity

**Step 3 – `MetricOmmPoolLiquidityAdder` accepts a fully caller-controlled `owner`.**

The overload that accepts an explicit `owner` has only a zero-address check; there is no requirement that `owner == msg.sender`:

```solidity
// MetricOmmPoolLiquidityAdder.sol L56-68
function addLiquidityExactShares(
    address pool,
    address owner,          // ← fully caller-controlled
    uint80 salt,
    LiquidityDelta calldata deltas,
    ...
) external payable override returns (...) {
    _validateOwner(owner);  // only checks owner != address(0)
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [4](#0-3) [5](#0-4) 

**Step 4 – The bypass path.**

When a non-allowlisted user calls `addLiquidityExactShares(pool, allowlisted_address, ...)`:

| Value | Who it is |
|---|---|
| `sender` (passed to hook) | `MetricOmmPoolLiquidityAdder` — **ignored by the hook** |
| `owner` (passed to hook) | Caller-supplied allowlisted address — **checked and passes** |
| Payer | Non-allowlisted `msg.sender` — **never checked** |

The allowlist check passes because `owner` is allowlisted. The non-allowlisted user pays the tokens; the allowlisted address receives the LP shares. The pool admin's intent to restrict who can interact with the pool is violated.

---

### Impact Explanation

- The deposit allowlist's invariant — that only allowlisted addresses can deposit into the pool — is broken by any unprivileged user.
- A non-allowlisted user can add liquidity to any bin, moving the pool cursor and altering bin balances. This directly affects the per-share metrics consumed by `OracleValueStopLossExtension._afterSwapOracleStopLoss`, potentially suppressing or triggering stop-loss watermark breaches.
- Non-allowlisted users can force LP positions onto allowlisted addresses without their consent (griefing).
- Regulatory/compliance pools (KYC-gated) are rendered ineffective: any actor can deposit tokens into the pool.

---

### Likelihood Explanation

- `MetricOmmPoolLiquidityAdder` is a public, permissionless contract with no factory verification.
- The bypass requires zero privileged access: any EOA can call `addLiquidityExactShares` with a known allowlisted address as `owner`.
- The attacker only needs to know one allowlisted address (e.g., the pool admin, which is publicly readable from the factory). [7](#0-6) 

---

### Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` should gate on `sender` (the actual caller of `addLiquidity`) rather than — or in addition to — `owner`. When the pool is called through the `MetricOmmPoolLiquidityAdder`, `sender` is the adder contract, so the extension should also check the adder's stored payer context, or the adder should forward the real payer identity through `extensionData` for the hook to decode and verify.

Alternatively, the pool's `addLiquidity` should enforce `msg.sender == owner` when no trusted intermediary is involved, or the adder should be updated to pass the real payer as `owner` only when `owner == msg.sender`.

### Proof of Concept

```solidity
// Setup: pool with DepositAllowlistExtension; only `allowedUser` is allowlisted.
// attacker is NOT allowlisted.

address allowedUser = ...; // known allowlisted address
address attacker    = ...; // not allowlisted

// Direct attempt — correctly blocked:
vm.prank(attacker);
pool.addLiquidity(attacker, salt, deltas, callbackData, extensionData);
// → reverts NotAllowedToDeposit ✓

// Bypass via MetricOmmPoolLiquidityAdder:
vm.prank(attacker);
adder.addLiquidityExactShares(
    address(pool),
    allowedUser,   // owner = allowlisted address → check passes
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    extensionData
);
// → succeeds; attacker paid tokens, allowedUser received LP shares,
