### Title
SwapAllowlistExtension Gates on Router Address Instead of End User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument—which is `msg.sender` of the pool's `swap` call—against the per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's address. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user on the network the ability to bypass the curated pool's access control.

### Finding Description

**Step 1 – Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` parameter to every configured extension: [2](#0-1) 

**Step 2 – Router is the immediate caller of `pool.swap`.**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly. The router is therefore `msg.sender` inside the pool, so `sender` delivered to the extension is the router address, not the end user: [3](#0-2) 

**Step 3 – Extension checks the wrong actor.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [4](#0-3) 

When the router calls the pool, the effective check becomes `allowedSwapper[pool][router]`.

**Step 4 – The dilemma.**

A pool admin who wants allowlisted users to be able to swap through the standard periphery must call `setAllowedToSwap(pool, router, true)`. The moment they do, the check `allowedSwapper[pool][router]` is `true` for every caller, so any non-allowlisted user can bypass the gate by routing through `MetricOmmSimpleRouter`. Conversely, if the admin never allowlists the router, even legitimately allowlisted users cannot use the router. [5](#0-4) 

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., institutional LPs, KYC'd users, or protocol-controlled addresses) loses that guarantee the moment the router is allowlisted. Any arbitrary address can call `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool and execute swaps as if they were allowlisted. This is a direct, unprivileged bypass of the pool's primary access-control invariant, with fund-impacting consequences: LPs who deposited under the assumption of restricted counterparties are exposed to the full public order flow they intended to exclude.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical periphery for swaps. Any pool admin who wants their allowlisted users to interact via the standard UX must allowlist the router. This is the expected operational path, making the vulnerable configuration the default production setup for any curated pool that supports router access.

### Recommendation

The extension must gate on the economically relevant actor—the end user—not the immediate caller of `pool.swap`. Two sound approaches:

1. **Extension-data forwarding**: Require the router to encode the original `msg.sender` in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address. The pool admin would allowlist end users, not the router.
2. **Dedicated router identity field**: Add an explicit `originator` field to the swap interface that the pool passes to extensions alongside `sender`, allowing extensions to distinguish the router from the end user.

Either approach must be enforced at the extension level; the pool's `sender` binding cannot be changed without breaking the core interface.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension; pool admin allowlists only `alice`:
       extension.setAllowedToSwap(pool, alice, true)

2. Pool admin also allowlists the router so alice can use the standard UI:
       extension.setAllowedToSwap(pool, router, true)

3. Non-allowlisted `bob` calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})

4. Router calls pool.swap(...) — msg.sender inside pool = router.

5. Pool calls extension.beforeSwap(router, ...).

6. Extension evaluates: allowedSwapper[pool][router] == true → passes.

7. Bob's swap executes on the curated pool despite never being allowlisted.
```

The corrupted invariant: `allowedSwapper[pool][bob]` is `false`, yet bob's swap settles successfully because the check evaluated `allowedSwapper[pool][router]` instead.
