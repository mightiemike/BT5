### Title
Pool Admin Can Reduce Timelock to Zero and Immediately Execute Drawdown Weakening — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` uses a per-pool timelock to give LPs reaction time before the pool admin can change drawdown, decay, or watermark parameters. However, there is no minimum-timelock validation, so the admin can propose `timelock = 0`, wait the current timelock once, execute it, and then immediately propose **and** execute a drawdown increase in the same block — giving LPs zero reaction time for the parameter change that actually weakens their protection.

---

### Finding Description

`_afterTimelock` computes the execution timestamp for any pending change: [1](#0-0) 

When `oracleStopLossConfig[pool_].timelock == 0`, this returns exactly `block.timestamp`.

`_requireElapsed` then checks: [2](#0-1) 

`block.timestamp < block.timestamp` is always `false`, so the revert never fires. Propose and execute can happen in the same transaction.

There is no `_validateTimelock` anywhere in the contract. `initialize` accepts any `uint32 timelock` including 0: [3](#0-2) 

`proposeOracleStopLossTimelock` similarly accepts any `uint32 newTimelock` with no lower-bound check: [4](#0-3) 

**Attack sequence:**

1. Admin calls `proposeOracleStopLossTimelock(pool, 0)` — `pendingTimelockExecuteAfter = block.timestamp + T` (current timelock T).
2. Admin waits T seconds (one-time cost).
3. Admin calls `executeOracleStopLossTimelock(pool)` — `oracleStopLossConfig[pool].timelock = 0`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, higherDrawdown)` — `pendingDrawdownExecuteAfter = block.timestamp + 0 = block.timestamp`.
5. Admin calls `executeOracleStopLossDrawdown(pool)` in the **same block** — `_requireElapsed(block.timestamp)` passes, drawdown is updated immediately. [5](#0-4) 

Steps 4 and 5 can be batched in a single transaction. LPs have zero seconds to react to the drawdown change.

---

### Impact Explanation

The NatDoc explicitly states the invariant: *"Drawdown and decay changes are timelocked so LPs can react."* [6](#0-5) 

Once the timelock is 0, the admin can raise `drawdownE6` to its maximum (`E6 = 1e6`, i.e., 100% drawdown) in a single atomic transaction, effectively disabling the stop-loss entirely. Any subsequent swap that would have been blocked by the guard now executes freely, allowing value to drain from the pool without LP recourse. This is a direct admin-boundary break: the pool admin exceeds the intended constraint by bypassing the timelock that was the sole LP protection mechanism.

---

### Likelihood Explanation

The pool admin is a real, non-factory role assigned per pool at creation. The attack requires only that the admin is malicious or compromised. The one-time cost (waiting the initial timelock) is a known, observable event, but LPs who do not monitor continuously will miss the subsequent same-block drawdown change. The `onlyPoolAdmin` modifier confirms this is a reachable path for the pool admin: [7](#0-6) 

---

### Recommendation

Add a minimum timelock floor in a `_validateTimelock` helper and call it from both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 hours; // or contest-appropriate value

function _validateTimelock(uint32 timelock) private pure {
    if (timelock < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(timelock);
}
```

Call it in `initialize`: [3](#0-2) 

And in `proposeOracleStopLossTimelock`: [4](#0-3) 

---

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_timelockZeroBypass() public {
    uint32 initialTimelock = 1 days;
    // Pool initialized with timelock = 1 day, drawdownE6 = 100_000 (10%)

    // Step 1: propose timelock = 0
    vm.prank(admin);
    extension.proposeOracleStopLossTimelock(pool, 0);

    // Step 2: wait the existing timelock
    vm.warp(block.timestamp + initialTimelock);

    // Step 3: execute timelock = 0
    vm.prank(admin);
    extension.executeOracleStopLossTimelock(pool);
    assertEq(extension.oracleStopLossConfig(pool).timelock, 0);

    // Step 4+5: propose and execute drawdown increase atomically (same block)
    vm.prank(admin);
    extension.proposeOracleStopLossDrawdown(pool, 900_000); // 90% drawdown
    vm.prank(admin);
    extension.executeOracleStopLossDrawdown(pool);          // no revert

    // Total elapsed since drawdown proposal: 0 seconds
    assertEq(extension.oracleStopLossConfig(pool).drawdownE6, 900_000);
    // LPs had 0 seconds to react to the drawdown weakening
}
```

The total elapsed time from drawdown proposal to execution is 0 seconds — less than the original 1-day timelock that LPs were relying on. [8](#0-7)
