### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating user, allowing any caller to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension` checks the `sender` argument passed by the pool, which equals `msg.sender` inside `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, that value is the router's address, not the end user's address. A pool admin who allowlists the router so that their permitted users can reach the pool through the router inadvertently opens the pool to every caller, because the router itself carries no per-user access control.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every configured extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that value against the per-pool allowlist: [2](#0-1) 

`msg.sender` inside the extension is the pool (correct), and `sender` is whatever the pool received as its own `msg.sender`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router becomes the pool's `msg.sender`: [3](#0-2) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

A pool admin who wants allowlisted users to reach the pool through the router has only two options:

1. **Do not allowlist the router** — allowlisted users cannot use the router at all.
2. **Allowlist the router** — every user on the network can now swap through the router, because the check resolves to `allowedSwapper[pool][router] == true` for every caller.

There is no configuration that simultaneously permits router-mediated swaps for allowlisted users while blocking non-allowlisted users. The actor the extension gates (the router) is not the actor the pool admin intends to gate (the individual user).

### Impact Explanation

Any user who is explicitly excluded from a curated pool's swap allowlist can bypass the restriction by routing through `MetricOmmSimpleRouter`. Once the router is allowlisted (a natural and expected admin action), the allowlist provides zero protection against router-mediated swaps. Unauthorized users can trade in pools that were designed to be restricted, breaking the admin-set access-control boundary and constituting an admin-boundary break via an unprivileged path.

### Likelihood Explanation

Pool admins who deploy a `SwapAllowlistExtension` almost certainly want their permitted users to be able to use the protocol's own router for better UX (slippage protection, multi-hop, etc.). Allowlisting the router is the obvious and expected step to enable that. The flaw is silent — the admin receives no warning that doing so opens the pool to everyone — making accidental exploitation highly likely in any production deployment that uses both the allowlist extension and the router.

### Recommendation

The `SwapAllowlistExtension` should gate the economically relevant actor — the originating user — rather than the immediate pool caller. Two complementary approaches:

1. **Pass the original user through the router.** `MetricOmmSimpleRouter` already stores the payer in transient storage; it could additionally forward the originating `msg.sender` inside `extensionData` so the extension can decode and check it.
2. **Check `sender` only when `sender` is not a known router.** The extension could maintain a registry of trusted routers and, when `sender` is a router, require the original user's address to be encoded in `extensionData` and verified there.

Either approach ensures the allowlist gates the user who initiated the trade, not the intermediary contract.

### Proof of Concept

```
Setup
─────
1. Factory deploys Pool with SwapAllowlistExtension as a before-swap hook.
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is KYC'd
3. Pool admin calls setAllowedToSwap(pool, router, true)  // so alice can use the router
   (bob is NOT allowlisted)

Attack
──────
4. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
   → inside pool.swap: msg.sender == router
6. Pool calls _beforeSwap(router, recipient, ...)
7. SwapAllowlistExtension.beforeSwap receives sender = router
8. Check: allowedSwapper[pool][router] == true  → passes
9. bob's swap executes successfully in the restricted pool.

Direct call (control)
─────────────────────
10. bob calls pool.swap(...) directly
    → sender = bob → allowedSwapper[pool][bob] == false → NotAllowedToSwap ✓
```

The bypass is reachable by any unprivileged user the moment the pool admin takes the natural step of allowlisting the router. [4](#0-3) [5](#0-4) [6](#0-5)

        "",

    amountOut = MetricOmmSwapInputs.int128ToUint128(out);

  }
