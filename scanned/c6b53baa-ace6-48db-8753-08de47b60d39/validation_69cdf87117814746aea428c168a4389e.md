### Title
SwapAllowlistExtension Checks Direct Pool Caller (`sender`) Instead of Actual User, Enabling Full Allowlist Bypass via Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of the `pool.swap()` call. When users trade through `MetricOmmSimpleRouter`, `sender` is the router address, not the actual user. A pool admin who allowlists the router to support legitimate users inadvertently opens the pool to every user, completely defeating the per-user curation the extension is designed to enforce.

### Finding Description

The pool's `swap()` function passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the per-pool allowlist: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [4](#0-3) 

So the extension receives `sender = router address`. The pool admin faces an impossible choice:

- **Do not allowlist the router** → allowlisted users cannot use the router at all (broken UX for the primary user-facing interface).
- **Allowlist the router** → `allowedSwapper[pool][router] == true`, so every user who routes through the router passes the check, regardless of whether they are individually allowlisted.

The `DepositAllowlistExtension` does not share this flaw because it gates by `owner` (the position owner explicitly passed through the call stack), not by `sender`: [5](#0-4) 

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise vetted addresses loses all enforcement the moment the router is allowlisted. Any unprivileged user can call `router.exactInputSingle` or `router.exactInput` and trade against the pool's liquidity. This constitutes a direct admin-boundary break: the pool admin's configured access policy is bypassed by an unprivileged path (the public router), with direct fund-flow consequences (unauthorized users extract value from LP positions in a pool that was supposed to be curated).

### Likelihood Explanation

The router is the primary user-facing swap interface. A pool admin who wants allowlisted users to be able to trade via the router will naturally allowlist the router address. The NatSpec of `SwapAllowlistExtension` says it "Gates `swap` by swapper address, per pool," giving no indication that allowlisting the router breaks per-user enforcement. The misconfiguration is foreseeable and likely in any production deployment that combines the allowlist extension with the router.

### Recommendation

Replace the `sender` check with a check on the `recipient` parameter, or add a dedicated `swapper` field to the extension interface that the pool populates from a transient-storage context set by the router (analogous to how the router already stores payer identity in `_setNextCallbackContext`). Alternatively, document explicitly that `SwapAllowlistExtension` is incompatible with the router and that pools using it must be accessed only via direct `pool.swap()` calls.

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls swapExtension.setAllowedToSwap(pool, alice, true)
   — alice is the only intended swapper.
3. Admin calls swapExtension.setAllowedToSwap(pool, router, true)
   — necessary so alice can use the router.
4. bob (not allowlisted) calls:
     router.exactInputSingle(ExactInputSingleParams{
       pool: pool, zeroForOne: true, amountIn: X, ...
     })
5. Router calls pool.swap(recipient, ...) → msg.sender in pool = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes against the curated pool's liquidity.
   The per-user allowlist is completely bypassed.
``` [3](#0-2) [1](#0-0) [4](#0-3)
