### Title
`SwapAllowlistExtension` checks the router address instead of the actual swapper, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` ‚Äî (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore checks the router's allowlist status, not the actual swapper's. Any pool that allowlists the router to support router-mediated swaps simultaneously grants every user on-chain access to that curated pool, defeating the allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap`: [4](#0-3) 

The same pattern holds for `exactInput` (first hop uses `msg.sender` as payer but the pool still sees the router as `msg.sender`), `exactOutputSingle`, and `exactOutput`. In every case the pool receives `msg.sender = router`, so the extension checks `allowedSwapper[pool][router]` ‚Äî not `allowedSwapper[pool][user]`.

---

### Impact Explanation

A pool deployer who wants to restrict swaps to a curated set of addresses and also wants those addresses to be able to use the standard router faces an impossible choice:

- **Do not allowlist the router**: allowlisted users cannot use the router at all ‚Äî broken core swap functionality.
- **Allowlist the router**: the allowlist is completely bypassed ‚Äî every address on-chain can swap through the router regardless of individual allowlist status.

In the second (natural) configuration, any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool and the `beforeSwap` hook passes, granting them full swap access. This is a direct loss of the curation guarantee the pool admin configured, and depending on the pool's purpose (e.g., institutional-only liquidity, KYC-gated pools), it constitutes a high-severity policy bypass with direct fund-impact consequences for LPs who deposited under the assumption that only approved counterparties could trade against them.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical, documented periphery entry point for swaps. Any pool operator who deploys `SwapAllowlistExtension` and also wants end users to be able to use the router will naturally allowlist the router address. The bypass requires no special privileges, no flash loans, and no unusual token behavior ‚Äî a single standard router call suffices.

---

### Recommendation

The `sender` argument delivered to `beforeSwap` must represent the economic actor, not the immediate `msg.sender` of `pool.swap`. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: store the originating `msg.sender` in transient storage alongside the callback context and expose it via a standardized field in `extensionData` or a dedicated transient slot that extensions can read. The pool or extension can then recover the true initiator.

2. **In `SwapAllowlistExtension`**: decode the true initiator from `extensionData` (if the router encodes it) and check that address, falling back to `sender` for direct pool calls.

Alternatively, document clearly that `SwapAllowlistExtension` gates the immediate pool caller, not the end user, and that allowlisting the router opens the pool to all users ‚Äî and provide a separate `OriginAllowlistExtension` that enforces per-user identity through a signed payload or transient-storage originator field.

---

### Proof of Concept

```
Setup:
  1. Deploy a pool with SwapAllowlistExtension configured as a beforeSwap hook.
  2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router so users can swap
  3. Pool admin does NOT call setAllowedToSwap(pool, alice, true)  // alice is not individually allowed

Attack:
  4. Alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
     ‚Üí router calls pool.swap(recipient, ...)
     ‚Üí pool calls _beforeSwap(msg.sender=router, ...)
     ‚Üí SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ‚úì
     ‚Üí swap executes successfully

Result:
  Alice, who is not on the allowlist, successfully swaps on the curated pool
  because the extension checked the router's allowlist status, not hers.
