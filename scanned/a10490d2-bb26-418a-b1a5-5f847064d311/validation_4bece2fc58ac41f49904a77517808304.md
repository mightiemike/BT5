### Title
`SwapAllowlistExtension` gates by router address instead of actual user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The allowlist therefore gates by router address rather than by individual user identity, breaking the intended access control in both directions.

---

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to every configured extension:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 230-240
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that argument against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol  lines 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the value the pool forwarded — i.e. whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol  lines 72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
```

The call chain is therefore:

```
user → router.exactInputSingle()
         → pool.swap()          [msg.sender = router]
             → extension.beforeSwap(sender = router, ...)
                 → allowedSwapper[pool][router]   ← checks router, not user
```

The extension never sees the actual user's address. The effective check is whether the **router contract** is on the allowlist, not whether the **user** is.

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, including the recursive callback hops in `_exactOutputIterateCallback` where `pool.swap(msg.sender=router, ...)` is called again.

---

### Impact Explanation

**Scenario A — router is allowlisted (bypass):** A pool admin who allowlists the router as a "trusted intermediary" inadvertently opens the pool to every user. Any address can call `router.exactInputSingle()` and the extension passes because `allowedSwapper[pool][router] == true`. The curated pool's access control is fully defeated.

**Scenario B — router is not allowlisted (broken functionality):** A legitimately allowlisted user who tries to swap through the router receives `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`. The only working path is a direct `pool.swap()` call, making the router — the protocol's primary swap interface — unusable for any curated pool.

Both outcomes represent a broken core invariant: the allowlist must enforce the same policy regardless of which supported public entrypoint reaches the pool.

---

### Likelihood Explanation

Every swap routed through `MetricOmmSimpleRouter` against a pool that has `SwapAllowlistExtension` configured in its `BEFORE_SWAP_ORDER` triggers this mismatch. The router is the documented primary swap path. Any curated pool that deploys this extension and expects users to use the router will be affected immediately.

---

### Recommendation

The pool must surface the original user's address to extensions. Two options:

1. **Pool-level fix (preferred):** Add an `originalSender` field to the extension call data, populated from a transient-storage slot that the router writes before calling `pool.swap()` (similar to how the router already stores callback context). The extension then checks `originalSender` instead of `sender`.

2. **Extension-level workaround:** Require callers to embed their address in `extensionData`; the extension verifies the embedded address against `msg.sender` of the outermost call via a signed payload. This is fragile and shifts burden to every integrator.

The root fix mirrors the Optimism finding: the intermediary (router / factory) must pass the real initiating address down to the component that enforces identity-based policy.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`.
2. Call `setAllowedToSwap(pool, router, true)` — allowlisting the router (Scenario A), or leave the router un-allowlisted (Scenario B).
3. **Scenario A:** Call `router.exactInputSingle(...)` from any EOA not in the allowlist. The swap succeeds because `allowedSwapper[pool][router] == true`.
4. **Scenario B:** Call `router.exactInputSingle(...)` from an EOA that IS in the allowlist. The swap reverts with `NotAllowedToSwap` because `allowedSwapper[pool][router] == false`, even though the user is individually permitted.

The corrupted value in both cases is the identity checked by the guard: `allowedSwapper[pool][router]` is evaluated where `allowedSwapper[pool][user]` is the invariant-correct check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
