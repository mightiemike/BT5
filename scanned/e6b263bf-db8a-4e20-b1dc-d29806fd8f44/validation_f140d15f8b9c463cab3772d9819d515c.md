### Title
Swap allowlist bypassed via router: `sender` is router address, not originating EOA — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap` call. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router contract, not the originating EOA. Any EOA can therefore bypass the allowlist by routing through an allowlisted router.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension. `SwapAllowlistExtension.beforeSwap` uses this value to look up `allowedSwapper[pool][sender]`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router function) calls `pool.swap`, the pool sees the router as `msg.sender`. If the pool admin has allowlisted the router (a natural configuration for a curated pool that still wants to support router-based UX), every EOA — including explicitly disallowed ones — can execute swaps by routing through the router. The original EOA identity is never checked.

### Impact Explanation
The swap allowlist extension's core invariant — that only explicitly permitted addresses may swap on a curated pool — is completely broken for any pool that allowlists the router. A disallowed EOA executes swaps that the pool admin intended to block. This is broken core functionality of a production extension.

### Likelihood Explanation
Allowlisting the router is the expected operational pattern for a curated pool that still wants to support standard periphery UX. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — only calling `exactInputSingle` from a disallowed EOA.

### Recommendation
Pass the originating caller through the call chain rather than `msg.sender`. One approach: add a `swapper` parameter to `IMetricOmmPoolActions.swap` that the pool passes to extensions, and have the router supply `msg.sender` explicitly. Alternatively, `SwapAllowlistExtension.beforeSwap` should reject calls where `sender` is a known router/contract unless the pool admin has explicitly opted into router-mediated allowlist semantics.

### Proof of Concept
```solidity

// 2. allowedSwapper[pool][router] = true   (router is allowlisted)
// 3. allowedSwapper[pool][eoa]    = false  (EOA is NOT allowlisted)
// 4. vm.prank(eoa);
//    router.exactInputSingle(ExactInputSingleParams({ pool: pool, ... }));
// 5. Assert: swap succeeds — eoa bypassed the allowlist via the router
```
