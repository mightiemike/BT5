### Title
`SwapAllowlistExtension.beforeSwap` checks the router's address instead of the actual end-user, allowing any non-allowlisted user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. The pool always passes `msg.sender` as `sender`, which is the **router contract** when a user routes through `MetricOmmSimpleRouter`. The allowlist check therefore validates the router's address, not the actual end-user's address. Any non-allowlisted user can bypass the swap allowlist on a curated pool by routing through the public `MetricOmmSimpleRouter`.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is whatever the pool passed — i.e., the pool's own `msg.sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInput()` or `exactOutputSingle()`, the router calls `pool.swap()` directly: [4](#0-3) 

At this point, the pool's `msg.sender` is the **router**, so `sender` forwarded to `beforeSwap` is the router's address. The allowlist check becomes:

```
allowedSwapper[pool][router]   // NOT allowedSwapper[pool][end_user]
```

This creates an irreconcilable dilemma for any pool admin who configures `SwapAllowlistExtension`:

- **If the router is NOT allowlisted**: all router-mediated swaps revert with `NotAllowedToSwap`, even for users the admin intended to allow. Allowlisted users cannot use the router at all.
- **If the router IS allowlisted** (the only way to make router swaps work): the check passes for every user who routes through the router, regardless of whether that user is on the allowlist. The allowlist is completely bypassed.

The `sender` parameter in the `IMetricOmmExtensions.beforeSwap` interface is documented as the swap initiator: [5](#0-4) 

But the pool only ever supplies `msg.sender` (the immediate caller), so the extension never sees the true end-user when an intermediary router is involved.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC-verified users, whitelisted counterparties, or protocol-internal actors) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user executes a real swap against the pool's liquidity, receiving output tokens and paying input tokens — a direct, fund-impacting bypass of the intended access control. LP funds are exposed to counterparties the pool admin explicitly excluded.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint documented and supported by the protocol. Any user aware of the allowlist restriction has a trivial, one-step bypass: call `exactInputSingle` or `exactInput` on the router instead of calling `pool.swap()` directly. No special privileges, flash loans, or multi-transaction setups are required. The router is a deployed, permissionless contract.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the **actual end-user**, not the immediate pool caller. Two complementary fixes:

1. **Pool-level**: Have `MetricOmmPool.swap()` accept an explicit `sender` parameter (the true originator) and pass it through to extensions, similar to how `addLiquidity` separates `msg.sender` (payer) from `owner` (position holder).

2. **Extension-level (short-term)**: `SwapAllowlistExtension.beforeSwap` should check both the `sender` argument **and** the `recipient` argument, or the pool admin should be documented that allowlisting the router address grants unrestricted access to all router users — and the extension should expose a separate "trusted forwarder" concept that re-checks the original initiator from callback context.

```diff
// In SwapAllowlistExtension.beforeSwap:
- if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+ // sender must be the true end-user, not an intermediate router
+ if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Until the pool passes the true originator, the extension cannot enforce per-user allowlists for router-mediated swaps.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for router to work

Attack:
  - Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  - Router calls pool.swap(recipient=bob, ...) → pool's msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  - Bob's swap executes successfully despite not being on the allowlist

Result:
  - Bob receives output tokens from the curated pool
  - The allowlist that was supposed to restrict trading to alice is fully bypassed
  - If the admin does NOT allowlist the router, alice also cannot use the router
```

The `sender` the extension sees is always the router address, never Bob's address, because `MetricOmmPool.swap()` passes `msg.sender` (the router) as `sender` to `_beforeSwap`, and `MetricOmmSimpleRouter` calls `pool.swap()` directly without any mechanism to forward the original caller's identity to the extension layer.
