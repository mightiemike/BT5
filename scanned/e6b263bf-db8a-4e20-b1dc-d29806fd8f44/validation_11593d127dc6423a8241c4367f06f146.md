### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the `msg.sender` of the `pool.swap()` call. When a user routes through the public `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the allowlist lookup becomes `allowedSwapper[pool][router]` â€” not `allowedSwapper[pool][user]`. If the pool admin allowlists the router address to enable router-mediated swaps for legitimate users, every unprivileged address can bypass the per-user gate by simply calling through the router.

### Finding Description

`SwapAllowlistExtension` is designed to restrict swaps to a configured set of addresses, per pool. The admin sets entries via `setAllowedToSwap(pool, swapper, true)`, keyed on `(pool, swapper)`. [1](#0-0)

Here `msg.sender` is the pool (the caller of the extension) and `sender` is the value the pool forwarded â€” which is the `msg.sender` of the original `pool.swap()` call.

`ExtensionCalling._beforeSwap` passes `sender` directly from the pool's execution context: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(recipient, ...)`. At that point `msg.sender` inside the pool is the **router**, so the pool forwards `sender = router` to the extension. The allowlist check resolves to:

```
allowedSwapper[pool][router]   // NOT allowedSwapper[pool][user]
```

This creates an irresolvable dilemma for the pool admin:

| Admin choice | Effect on legitimate users | Effect on non-allowlisted users |
|---|---|---|
| Do **not** allowlist the router | Allowlisted users **cannot** use the router | Non-allowlisted users also blocked via router |
| **Allowlist the router** | Allowlisted users can use the router | **Every user bypasses the allowlist via the router** |

The second row is the exploitable path: once the router is allowlisted (the only way to let legitimate users use the standard periphery), the per-user gate is completely open to any caller.

The analog to the external report is exact: the external bug keyed canonical assets on only `id` (dropping `domain`), collapsing two distinct identities into one lookup. Here the allowlist keys on only the **immediate caller** (dropping the **end user**), collapsing all router-mediated swaps into a single identity â€” the router â€” regardless of who initiated the transaction.

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties, protocol-internal actors, or whitelisted LPs) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The non-allowlisted user executes a real swap, receives output tokens, and the pool's intended access control is silently bypassed. This constitutes broken core pool functionality with direct fund-impacting consequences: the pool trades with actors it was configured to exclude.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless contract. No special privilege, flash loan, or multi-step setup is required. Any address can call `exactInputSingle` or `exactOutputSingle` targeting the restricted pool. The bypass is one transaction and is reachable on every pool that (a) uses `SwapAllowlistExtension` and (b) has the router allowlisted to support normal user flows.

---

### Recommendation

The extension must check the **economic actor** (the end user), not the **transport layer** (the router). Two sound approaches:

1. **Decode the real caller from `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. This requires the router to be trusted to supply the correct value, which is acceptable since the router is a known, audited periphery contract.

2. **Check `sender` only for direct pool calls; require `extensionData` attestation for router calls**: The extension can distinguish direct calls (where `sender` is the real user) from router calls by checking whether `sender` is a known router, and in that case require a signed or encoded real-caller field in `extensionData`.
