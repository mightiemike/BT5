### Title
SwapAllowlistExtension gates the router address instead of the end user, making the allowlist either universally bypassable or unusable through the router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's address. This creates an irreconcilable dilemma for any curated pool: if the router is allowlisted, every user bypasses the allowlist; if the router is not allowlisted, allowlisted users cannot use the router at all.

### Finding Description

`SwapAllowlistExtension` is designed to restrict swaps on a curated pool to a set of approved addresses. The guard is wired into the `beforeSwap` hook and receives `sender` — the address that called `pool.swap()` — as the identity to check:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmPool.swap()` sets `sender = msg.sender` (L231), which is the router when the user enters through `MetricOmmSimpleRouter`. The router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```

So the extension sees `sender = router`, never the end user. The pool admin faces two broken outcomes:

1. **Allowlist bypass**: Admin allowlists the router address (the only way to enable router-mediated swaps). Every user — including those not individually allowlisted — can now swap freely through the router. The allowlist is completely ineffective.

2. **Broken router path**: Admin allowlists individual user addresses instead of the router. Those users cannot swap through the router (router address fails the check). They must call `pool.swap()` directly, bypassing the standard periphery.

The analog to the external report is exact: the guard (`SwapAllowlistExtension`) requires a prerequisite binding (the correct actor identity) that is never established on the router path. The guard either silently passes for all users (bypass) or silently blocks all router users (broken functionality), depending on how the admin configures it.

### Impact Explanation

**High.** A curated pool deploying `SwapAllowlistExtension` to restrict trading to approved counterparties cannot enforce that restriction when the standard router is involved. Any unprivileged user can bypass the allowlist by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) on the restricted pool. The pool's token reserves are exposed to unrestricted swaps, directly contradicting the admin's configured access policy and potentially draining LP value through trades the pool was designed to exclude.

### Likelihood Explanation

**High.** `MetricOmmSimpleRouter` is the canonical swap entry point for end users. Any pool that deploys `SwapAllowlistExtension` and also wants users to be able to use the router must allowlist the router address — at which point the bypass is unconditional and requires no special knowledge or privilege from the attacker.

### Recommendation

The `beforeSwap` hook should receive and check the **economic actor** (the end user), not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a protocol-level convention.

2. **Check `sender` against a router registry and then verify the user from callback context**: The extension recognises the router as a trusted forwarder and reads the actual user from a standardised field in `extensionData`.

Either way, the extension must be updated so that `allowedSwapper[pool][actualUser]` is checked, not `allowedSwapper[pool][router]`.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `trustedUser` is allowlisted.
// Admin allowlists the router so that trustedUser can use it.
extension.setAllowedToSwap(pool, address(router), true);

// Attacker (not allowlisted) bypasses the guard via the router:
router.exactInputSingle(ExactInputSingleParams({
    pool:            pool,
    recipient:       attacker,
    tokenIn:         token0,
    zeroForOne:      true,
    amountIn:        1_000e18,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    deadline:        block.timestamp,
    extensionData:   ""
}));
// Swap succeeds; allowlist is bypassed.
// The extension received sender = address(router), which is allowlisted.
```
