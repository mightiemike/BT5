## Analysis

I need to trace the exact call chain from router → pool → extension to determine whether the `sender` identity is preserved.

**Step 1: Router calls pool**

In `MetricOmmSimpleRouter::exactInputSingle`, the router calls the pool's `swap` directly: [1](#0-0) 

At this point, `msg.sender` seen by the pool is the **router's address**, not the original user.

**Step 2: Pool passes `msg.sender` as `sender` to the hook**

In `MetricOmmPool::swap`, the pool calls `_beforeSwap(msg.sender, ...)`: [2](#0-1) 

And `ExtensionCalling::_beforeSwap` encodes that `sender` directly into the extension call: [3](#0-2) 

**Step 3: The hook checks `sender` against the allowlist**

In `SwapAllowlistExtension::beforeSwap`: [4](#0-3) 

- `msg.sender` = the pool (correct — used as the pool key)
- `sender` = whoever called `pool.swap()` = **the router**, not the original user

**The Identity Substitution**

| Flow | `sender` seen by hook |
|---|---|
| User → Pool directly | User's address |
| User → Router → Pool | Router's address |

**The Bypass**

A pool admin who wants to restrict swaps to specific addresses (e.g., KYC'd users `alice`, `bob`) sets:
- `allowedSwapper[pool][alice] = true`
- `allowedSwapper[pool][bob] = true`

For router-mediated swaps to work at all, the admin must also add:
- `allowedSwapper[pool][router] = true`

Once the router is allowlisted, **any user** — including `charlie` who is not allowlisted — can call `router.exactInputSingle(...)` and the hook will see `sender = router`, which passes the check. The per-user allowlist is completely bypassed.

Conversely, if the admin does NOT allowlist the router, then even `alice` and `bob` cannot use the router — they must call the pool directly. This makes the extension incompatible with the router for per-user gating.

**The "two transactions" framing** in the question does not map to a timed-threshold or stale-observation attack — there is no time-based state in `SwapAllowlistExtension`. The bypass is a single router call. The question's framing around "stale threshold state" is not supported by the code.

**Impact Assessment**

The bypass is real and structural: the allowlist cannot simultaneously (a) allow router-mediated swaps and (b) restrict swaps to specific users. However, the impact depends on whether a pool admin actually allowlists the router. This is not a direct fund-loss path — it allows unauthorized swaps, which could harm pool integrity if the allowlist was a loss-prevention control (e.g., restricting to specific market makers). Under the contest rules, this qualifies as broken core functionality of a security extension.

### Title
Router Identity Substitution Bypasses Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension::beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the hook checks the router's address rather than the original user's address. Allowlisting the router (required for router-mediated swaps) grants every user bypass access to the allowlist.

### Finding Description
`MetricOmmPool::swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [5](#0-4) 

When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router, not the original user. `ExtensionCalling::_beforeSwap` forwards this value unchanged to the extension: [6](#0-5) 

`SwapAllowlistExtension::beforeSwap` then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router: [7](#0-6) 

If the router is allowlisted (necessary for any router-mediated swap to succeed), the check passes for all users regardless of their individual allowlist status.

### Impact Explanation
Any user can bypass a per-user swap allowlist by routing through `MetricOmmSimpleRouter`. If the allowlist is used as a loss-prevention control (e.g., restricting swaps to KYC'd addresses or specific market makers), this bypass allows unauthorized parties to trade against the pool, breaking the intended access control.

### Likelihood Explanation
Pools using `SwapAllowlistExtension` with per-user restrictions that also need to support router-mediated swaps must allowlist the router, making the bypass trivially reachable by any public user.

### Recommendation
Pass the original user's address through the call chain. One approach: include the original `msg.sender` in `extensionData` and have the extension read it from there with pool-level authentication, or add a dedicated `originalSender` field to the hook signature. Alternatively, document that allowlisting the router grants universal access and that per-user gating requires direct pool interaction only.
