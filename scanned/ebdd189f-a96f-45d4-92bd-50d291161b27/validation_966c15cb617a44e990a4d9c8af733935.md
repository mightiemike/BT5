### Title
`uint32 lastDecayTs` Truncation in `OracleValueStopLossExtension` Permanently Disables Stop-Loss Guard After Year 2106 — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`BinHighWatermarks.lastDecayTs` is stored as `uint32`. After year 2106, `block.timestamp` exceeds `type(uint32).max`, causing `uint32(block.timestamp)` to wrap to a small value. The subsequent `dt` computation in `_checkAndUpdateWatermarks` and `currentHighWatermarks` produces a massively inflated elapsed-time value, which forces `_decayed` to return `0` for every watermark. With all watermarks zeroed, `_applyWatermark` never detects a breach, permanently disabling the stop-loss guard for every pool that uses this extension.

---

### Finding Description

**Storage truncation:** [1](#0-0) 

```solidity
hwmS.lastDecayTs = uint32(block.timestamp);   // truncated to 32 bits
```

**Elapsed-time computation:** [2](#0-1) 

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
```

`hwmS.lastDecayTs` is zero-extended to `uint256` before the subtraction. After year 2106, `block.timestamp ≈ 4_294_967_296 + X` while `uint32(block.timestamp) = X` (wrapped). The subtraction yields `dt ≈ 4_294_967_296`, a value that is orders of magnitude larger than any real elapsed time.

**Decay function short-circuits to zero:** [3](#0-2) 

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;          // ← always true with inflated dt
    return hwm - (hwm * factor) / E8;
}
```

`ratePerSecondE8` is validated to be at most `E8 = 1e8`. With `dt ≈ 4_294_967_296`, `factor = 1e8 × 4_294_967_296 ≈ 4.3 × 10^17 ≥ 1e8`, so `_decayed` always returns `0`.

**Watermark check never fires:** [4](#0-3) 

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // hwm == 0 → always true
    ...
}
```

With `hwm = 0`, `metric >= 0` is always satisfied, so `breached` is always `false`. Every swap silently updates the watermark to the current metric and no `OracleStopLossTriggered` revert is ever emitted.

**Secondary: `_afterTimelock` overflow also bypasses timelocks after year 2106:** [5](#0-4) 

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

After year 2106, `block.timestamp + timelock` always overflows `uint32`, wrapping `executeAfter` to a value smaller than `block.timestamp`. `_requireElapsed` then passes immediately for every pending proposal, bypassing all timelocks.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain guard that prevents LP value from draining below a configured drawdown floor. After year 2106:

1. All per-bin watermarks are silently zeroed on every swap.
2. No stop-loss breach is ever detected regardless of how far the oracle mid-price moves.
3. Swaps that should be reverted by `OracleStopLossTriggered` execute freely.
4. LP principal can be drained without bound through adversarial or oracle-driven price moves.
5. All pending admin changes (drawdown, decay, watermark resets) can be executed without waiting for the configured timelock.

This matches the "oracle guard path fails open" and "direct loss of LP principal" impact categories.

---

### Likelihood Explanation

Identical to the JalaPair finding: the event occurs with **100% probability** once `block.timestamp` crosses `type(uint32).max` (approximately year 2106, ~80 years from now). No attacker action is required; the protocol simply needs to still be running. The comment in the analogous UniswapV2 code ("overflow is desired") is absent here, confirming this is an unintentional design defect rather than a deliberate choice.

---

### Recommendation

Replace `uint32 lastDecayTs` with `uint64` in `BinHighWatermarks`. `uint64` does not overflow until approximately year 584 billion and costs no additional storage slot given the existing `uint104 + uint104 + uint32` packing (the struct already occupies a full 256-bit slot; widening to `uint48` or `uint64` requires a minor repack but eliminates the hazard entirely).

```solidity
// Before
struct BinHighWatermarks {
    uint104 token0;
    uint104 token1;
    uint32  lastDecayTs;   // overflows year 2106
}

// After
struct BinHighWatermarks {
    uint104 token0;
    uint104 token1;
    uint48  lastDecayTs;   // safe until year ~8 million
}
```

Apply the same fix to `_afterTimelock`: widen the intermediate arithmetic to `uint256` and only truncate the final stored value if a compact representation is still desired, or store `pendingXxxExecuteAfter` fields as `uint64`.

### Proof of Concept

```solidity
// Warp block.timestamp to just past uint32 max
vm.warp(uint256(type(uint32).max) + 100);

// lastDecayTs was set before the wrap, e.g. uint32(type(uint32).max - 10) = 4294967285
// After warp: dt = (4294967295 + 100) - 4294967285 = 110  ← still correct

// Now trigger one swap so lastDecayTs is updated to uint32(block.timestamp)
// uint32(4294967395) = 4294967395 - 4294967296 = 99  ← wrapped!
pool.swap(...);

// Warp forward 1 hour
vm.warp(block.timestamp + 3600);

// dt = (4294967395 + 3600) - 99 = 4294970896  ← ~136 years of elapsed time
// factor = ratePerSecondE8 * 4294970896 >= E8 → _decayed returns 0
// _applyWatermark(metric, 0, floor) → breached = false always
// Stop-loss is permanently disabled; swap below drawdown floor succeeds
pool.swap(...);  // should revert with OracleStopLossTriggered, but does not
```
