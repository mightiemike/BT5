### Title
`SwapAllowlistExtension.beforeSwap()` Checks `sender` (Router) Instead of the Actual User, Enabling Allowlist Bypass Through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` gates swaps by checking the `sender` argument passed by the pool, which equals `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the user. If the router is allowlisted, every user — including those not individually allowlisted — can bypass the per-user gate. This is the direct analog of the external bug: one "ledger" (the allowlist check) is applied to the wrong entity, while the parallel extension (`DepositAllowlistExtension`) correctly checks the economic actor (`owner`).

---

### Finding Description

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]`: [1](#0-0) 

`sender` is forwarded verbatim from `ExtensionCalling._beforeSwap()`, which passes `msg.sender` of the pool's `swap()` call: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [3](#0-2) 

So `msg.sender` of `pool.swap()` is the router, making `sender` = router in `beforeSwap`. The extension therefore checks `allowedSwapper[pool][router]`, not the actual user.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity()` ignores `sender` and checks `owner` — the economic actor who receives the LP position: [4](#0-3) 

This asymmetry is the dual-ledger mismatch: the deposit allowlist correctly gates the economic beneficiary; the swap allowlist gates the intermediary instead.

---

### Impact Explanation

**Allowlist bypass (High):** A pool admin who allowlists the router — a natural action when the router is the intended user-facing entry point — inadvertently opens the gate to every user of the router. Any address, regardless of individual allowlist status, can call `router.exactInputSingle()` and have the pool accept the swap because `allowedSwapper[pool][router] = true`. The curated-pool invariant ("only approved counterparties may trade") is silently violated, and LP assets are exposed to unauthorized swap flow.

**Broken allowlist for legitimate users (Medium):** If the admin allowlists individual users but not the router, those users cannot swap through the router at all — the check sees the router address and reverts — making the standard periphery path unusable for the very users the admin intended to serve.

---

### Likelihood Explanation

The router is the canonical user-facing swap entry point documented and deployed alongside the pool. A pool admin configuring a curated pool will naturally allowlist the router as a trusted intermediary, not realizing that doing so collapses the per-user gate entirely. The bypass requires no special privilege: any unprivileged address can call the public router.

---

### Recommendation

The swap allowlist must gate the actual initiating user, not the intermediary. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Check `recipient` instead of `sender`**: For most swap flows the recipient is the user-controlled address. This is imperfect (recipient can differ from initiator) but closer to the economic actor than `sender`.
3. **Document and enforce direct-call-only**: If the allowlist is only meaningful for direct `pool.swap()` calls, the extension and pool documentation must state this explicitly and the router must not be allowlisted on curated pools.

The `DepositAllowlistExtension` pattern — checking `owner`, not `sender` — should be the model.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension; allowAllSwappers = false.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to allow the standard periphery path.
3. Charlie (not individually allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: charlie, ...})
4. Router calls pool.swap(charlie, zeroForOne, amount, ...)
     → msg.sender of pool.swap() = router
5. Pool calls extension.beforeSwap(sender=router, ...)
     → allowedSwapper[pool][router] = true → check passes
6. Charlie's swap executes on the restricted pool.
   Any user of the router now bypasses the per-user allowlist.
```
