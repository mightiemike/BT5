The code path is fully traceable. Here is the complete analysis.

---

**Tracing the exact call chain:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)` where `msg.sender` is whoever called `pool.swap()`. [1](#0-0) 

`_beforeSwap` in `ExtensionCalling` forwards `sender` (= `msg.sender` of the pool call) to the extension. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks only `sender` — the second parameter `recipient` is explicitly unnamed and ignored. [3](#0-2) 

In `MetricOmmSimpleRouter.exactInput`, for **every** hop (including the last), the direct caller of `pool.swap()` is the router (`address(this)`). The payer context switches between `msg.sender` and `address(this)`, but the `msg.sender` seen by the pool is always the router. [4](#0-3) 

---

**The invariant break:**

The claim's framing is slightly imprecise — the bypass is not unique to intermediate hops. For **every** hop in `exactInput` (and for `exactInputSingle` too), `sender` seen by the extension is the router. If the router is allowlisted on a pool, any EOA can call the router and the allowlist check passes unconditionally, regardless of who the `recipient` is.

The `recipient` is available in `beforeSwap`'s signature but is silently discarded:

```solidity
// SwapAllowlistExtension.sol line 31
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
//                                   ^^^^^^^ recipient is unnamed and never checked
```

---

**Is this a real vulnerability?**

Yes. The `SwapAllowlistExtension` is documented as gating "swap by swapper address, per pool." A pool admin who allowlists the router intends to permit router-mediated swaps for their authorized users — not to open the pool to every EOA on the network. Because the extension never checks `recipient`, any un-allowlisted EOA can:

1. Call `exactInputSingle` or `exactInput` on the router targeting the restricted pool.
2. The router calls `pool.swap(recipient=attacker, ...)` — `sender` = router (allowlisted) → check passes.
3. The attacker receives output tokens from a pool that was supposed to reject them.

This is a direct access-control bypass on a production extension that ships with the protocol. The LP's swap-restriction invariant is broken: the pool admin believes only allowlisted addresses can extract value from the pool, but any address can do so via the router.

### Title
`SwapAllowlistExtension` only checks `sender` (direct `pool.swap()` caller), not `recipient`, allowing any EOA to bypass per-pool swap restrictions via the router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` ignores the `recipient` argument. When the router is allowlisted on a pool, any un-allowlisted EOA can route through the router and receive output tokens from the restricted pool, fully bypassing the intended access control.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` and the caller-supplied `recipient` to `_beforeSwap`, which forwards both to the extension. `SwapAllowlistExtension.beforeSwap` accepts `recipient` as its second parameter but leaves it unnamed and never reads it. The check `allowedSwapper[msg.sender][sender]` only validates the direct caller of `pool.swap()`.

When `MetricOmmSimpleRouter.exactInput` (or `exactInputSingle`) is used, `msg.sender` seen by the pool is always the router. If the pool admin has called `setAllowedToSwap(pool, router, true)` — a natural configuration for any pool that is meant to be accessible via the router — then the allowlist check passes for every swap routed through the router, regardless of who the actual `recipient` is.

### Impact Explanation
Any un-allowlisted EOA can extract tokens from a swap-restricted pool by calling the router. The pool admin's intent to restrict economic access is completely defeated. LP funds are exposed to actors the pool admin explicitly intended to block. This is a direct loss-of-access-control impact with a clear fund-extraction path.

### Likelihood Explanation
Likelihood is medium-high. Allowlisting the router is the expected operational step for any pool that wants to support router-mediated trading. The pool admin has no on-chain signal that doing so opens the pool to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — just a standard router call.

### Recommendation
`SwapAllowlistExtension.beforeSwap` should check `recipient` in addition to (or instead of) `sender`, depending on the intended semantics:

```solidity
function beforeSwap(address sender, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    address pool_ = msg.sender;
    // Gate on the economic beneficiary, not just the direct caller
    if (!allowAllSwappers[pool_] && !allowedSwapper[pool_][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, the router could be modified to pass the original `msg.sender` through `extensionData` so extensions can verify the initiating user, but changing the extension check is simpler and more robust.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as `beforeSwap` hook.
2. Call `setAllowedToSwap(pool, router, true)` — do NOT allowlist the attacker EOA.
3. From the attacker EOA, call `router.exactInputSingle(pool, ..., recipient=attacker)`.
4. `pool.swap` is called with `msg.sender=router` (allowlisted) → `beforeSwap` passes.
5. Assert attacker receives output tokens despite not being in the allowlist.

The same test works with `exactInput` for a multi-hop path where any hop's pool has the allowlist configured with the router allowlisted.
