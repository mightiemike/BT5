### Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which equals `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the originating EOA. If the pool admin allowlists the router address (a natural step to enable router-mediated swaps for their curated users), every user — including those explicitly not on the allowlist — can bypass the gate by calling any of the router's `exact*` entry points.

---

### Finding Description

**Actor binding in the extension:**

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (enforced by `onlyPool`). `sender` is whatever the pool passed as the first argument to `beforeSwap`. [1](#0-0) 

**What the pool passes as `sender`:**

`MetricOmmPool.swap` passes `msg.sender` — the immediate caller of the pool — as `sender` to every extension hook:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient,
  ...
);
``` [2](#0-1) 

**What the router passes as `msg.sender` to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The pool therefore sees `msg.sender = router`, not the originating EOA:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
``` [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**The bypass path:**

A pool admin who wants their allowlisted users to be able to use the router must call `setAllowedToSwap(pool, router, true)`. Once the router address is in `allowedSwapper[pool]`, the extension's check passes for every call that arrives through the router — regardless of who the originating EOA is. A disallowed user calls `exactInputSingle` on the router; the router calls `pool.swap`; the pool passes `sender = router`; the extension sees `allowedSwapper[pool][router] == true` and permits the swap.

The admin has no mechanism to express "only my specific users may use the router" because the extension has no visibility into the router's caller. The only options are:

| Admin config | Allowed users (direct) | Allowed users (router) | Disallowed users (router) |
|---|---|---|---|
| Allowlist specific EOAs only | ✓ | ✗ (router blocked) | ✗ |
| Allowlist specific EOAs + router | ✓ | ✓ | **✓ (bypass)** | [1](#0-0) 

---

### Impact Explanation

Any user can trade against a curated pool's liquidity without being on the allowlist, simply by routing through `MetricOmmSimpleRouter`. LPs who deposited into a curated pool expecting only specific counterparties (e.g., institutional traders, KYC'd addresses) are exposed to unrestricted swap flow. This constitutes a broken core pool functionality — the allowlist guard that pool admins configure to protect LP assets fails open for all router-mediated swaps once the router is allowlisted.

---

### Likelihood Explanation

The scenario is realistic and likely:

1. A pool admin deploys a curated pool with `SwapAllowlistExtension`.
2. They allowlist specific EOAs.
3. Those EOAs want to use the router for slippage protection or multi-hop paths.
4. The admin allowlists the router address — a natural, expected step.
5. From that point, any EOA can bypass the allowlist via the router.

The admin has no documentation warning that allowlisting the router is equivalent to disabling the allowlist for all router users. The `ISwapAllowlistExtension` interface and `SwapAllowlistExtension` NatDoc give no indication of this limitation. [5](#0-4) 

---

### Recommendation

`SwapAllowlistExtension` should not be the sole enforcement point when the router is in the picture. Two complementary fixes:

1. **Extension-level**: Accept an optional `bytes calldata extensionData` payload that the router populates with the originating EOA (signed or passed through), and check that identity instead of (or in addition to) `sender`. This requires a coordinated change in the router.

2. **Router-level**: `MetricOmmSimpleRouter` should forward the originating user's address in `extensionData` so that allowlist extensions can decode and check it. The extension would then verify both that the router is a trusted intermediary and that the embedded user identity is allowlisted.

3. **Documentation**: Until a code fix is in place, document explicitly that allowlisting the router address in `SwapAllowlistExtension` grants swap access to all router users, and that curated pools should not allowlist the router if they intend to restrict individual swappers. [1](#0-0) [6](#0-5)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension.
// 2. Admin allowlists allowedUser and the router:
//      swapExt.setAllowedToSwap(pool, allowedUser, true);
//      swapExt.setAllowedToSwap(pool, address(router), true);
// 3. disallowedUser is NOT on the allowlist.

// Attack:
// disallowedUser calls router directly:
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:            address(pool),
        tokenIn:         address(token1),
        recipient:       disallowedUser,
        zeroForOne:      false,
        amountIn:        1000,
        amountOutMinimum: 0,
        priceLimitX64:   type(uint128).max,
        deadline:        block.timestamp + 1,
        extensionData:   ""
    })
);
// pool.swap is called with msg.sender = router.
// _beforeSwap passes sender = router to the extension.
// allowedSwapper[pool][router] == true → check passes.
// disallowedUser's swap executes successfully despite not being allowlisted.
```

The direct pool call from `disallowedUser` would revert with `NotAllowedToSwap`; the router-mediated call succeeds, demonstrating the bypass. [1](#0-0) [2](#0-1) [6](#0-5)
