The call chain is fully traceable. Here is the exact identity mismatch:

**Step 1** — User calls `MetricOmmSimpleRouter.exactInputSingle`. The router stores `msg.sender` (the user) only in transient callback context for payment, then calls the pool directly: [1](#0-0) 

**Step 2** — `MetricOmmPool.swap` passes `msg.sender` (= the **router address**) as `sender` to `_beforeSwap`: [2](#0-1) 

**Step 3** — `ExtensionCalling._beforeSwap` encodes that same `sender` (router) and dispatches it to the extension: [3](#0-2) 

**Step 4** — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][router]`, never the actual end-user: [4](#0-3) 

The result is a binary trap for pool admins:

| Router allowlisted? | Effect |
|---|---|
| Yes | Every user, including non-allowlisted ones, can bypass the gate via the router |
| No | Every allowlisted user is silently blocked from using the router |

The "before-after inconsistency / paused pool / bin-threshold" framing in the question is noise. The `swap` function carries `whenNotPaused`, so a paused pool reverts before `beforeSwap` is ever reached, and bin positions are irrelevant to the allowlist check. The real and only issue is the sender identity substitution.

### Title
`SwapAllowlistExtension` gates the router address instead of the end-user, allowing any caller to bypass per-user swap restrictions — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap` function receives the router as `msg.sender` and forwards that address as `sender` to `beforeSwap`. The `SwapAllowlistExtension` then checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any user who is not individually allowlisted can bypass the restriction by calling the router, provided the router itself is allowlisted.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230
_beforeSwap(
    msg.sender,   // <-- router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` encodes that value and calls the extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert NotAllowedToSwap();
}
// msg.sender = pool, sender = router (not the end-user)
```

There is no mechanism in the router to propagate the original caller's identity to the hook. The user's address is stored only in transient callback context for payment settlement and is never visible to the extension.

### Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses can be bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The attacker executes a swap on a pool they are not authorized to trade on, potentially extracting value from a pool intended for a closed set of participants (e.g., a private OTC pool or a KYC-gated pool). This breaks the core access-control invariant the extension is designed to enforce.

### Likelihood Explanation
The router is a standard, publicly deployed periphery contract. Any user aware of the allowlist restriction can trivially route through it. The only precondition is that the router address is itself allowlisted on the pool, which is the expected operational state for any pool that wants to support router-mediated swaps at all.

### Recommendation
Pass the original caller's identity through the swap path so extensions can gate on the true end-user. One approach: add an `originator` field to the `extensionData` ABI encoding that the router populates with `msg.sender`, and have `SwapAllowlistExtension.beforeSwap` decode and check that field. A stricter approach is to have the pool accept an explicit `originator` parameter in `swap` and forward it to hooks, similar to how Uniswap v4 handles `hookData`.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured; do **not** add Alice to `allowedSwapper`.
2. Add the router address to `allowedSwapper[pool][router] = true`.
3. Alice calls `MetricOmmSimpleRouter.exactInputSingle` targeting that pool.
4. The pool calls `_beforeSwap(router, ...)`. The extension checks `allowedSwapper[pool][router]` → `true`. The swap succeeds.
5. Assert: Alice, who is not allowlisted, completed a swap on a restricted pool.

Direct call control: Alice calls `MetricOmmPool.swap` directly → `_beforeSwap(Alice, ...)` → `allowedSwapper[pool][Alice]` = `false` → `revert NotAllowedToSwap()`. The bypass is exclusive to the router path.
