### Title
`DepositAllowlistExtension#beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor** address. However, its `beforeAddLiquidity` hook validates the `owner` argument (the LP-share recipient) rather than the `sender` argument (the address that actually calls `addLiquidity` and provides tokens via the callback). Because `owner` is a free caller-supplied parameter, any address can pass the allowlist check by naming an allowlisted address as `owner`, while itself remaining the actual token provider.

### Finding Description

`MetricOmmPool.addLiquidity` passes both `msg.sender` (the actual caller) and the caller-supplied `owner` to the extension hook:

```solidity
// MetricOmmPool.sol – addLiquidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both values to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first (unnamed, discarded) parameter and `owner` as its second, but only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is entirely caller-controlled. A non-allowlisted address `Bob` can call `pool.addLiquidity(owner = Alice, ...)` where `Alice` is allowlisted. The hook reads `allowedDepositor[pool][Alice] == true` and passes. Bob's callback then transfers Bob's tokens into the pool; Alice receives the LP shares. The allowlist is fully bypassed with respect to the actual token provider.

Compare with `SwapAllowlistExtension`, which correctly checks `sender` (the actual swap initiator):

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

The asymmetry between the two allowlist extensions confirms the deposit extension is checking the wrong address.

### Impact Explanation

The deposit allowlist is a core access-control primitive. Pools that rely on it to restrict liquidity providers (e.g., permissioned pools, KYC-gated pools, or pools where LP composition is security-critical) receive no protection: any address can add liquidity by naming an allowlisted address as `owner`. Concretely:

1. **Allowlist fully bypassed**: A non-allowlisted address provides tokens and shifts the pool's bin state (current bin index, position within bin) without restriction.
2. **Pool-state manipulation enabling LP loss**: After adding liquidity to move the pool into a favorable bin position, the attacker swaps against the pool (no swap allowlist required) to extract value from existing LPs at an advantageous price.
3. **Colluding-LP path**: With a cooperating allowlisted address, the attacker recovers the deposited tokens via `removeLiquidity` (only `owner` can call it), completing a round-trip that effectively makes the attacker an LP in a pool that was supposed to exclude them.

### Likelihood Explanation

- Requires only knowledge of one allowlisted address (publicly readable from `allowedDepositor` mapping or on-chain events).
- No privileged role, no special token, no oracle manipulation needed.
- Triggerable on any pool that has deployed `DepositAllowlistExtension` without also deploying a complementary sender-level guard.
- The `owner` parameter is a standard, documented field of `addLiquidity`; any integrator or attacker reading the ABI can exploit this immediately.

### Recommendation

Check `sender` (the actual caller) instead of `owner` (the LP-share recipient), mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict LP-share ownership (not token provision), rename the extension and its setter to reflect that, and document the `owner`-check semantics explicitly so integrators are not misled.

### Proof of Concept

```
Setup:
  pool P with DepositAllowlistExtension E
  allowedDepositor[P][Alice] = true
  allowedDepositor[P][Bob]   = false   // Bob is NOT allowlisted

Attack:
  Bob calls pool.addLiquidity(
      owner        = Alice,   // allowlisted — passes the hook
      salt         = 0,
      deltas       = <large liquidity delta in bin 0>,
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    sender = Bob  (ignored — first param is unnamed/discarded)
    owner  = Alice → allowedDepositor[P][Alice] == true → hook passes

  LiquidityLib calls IMetricOmmModifyLiquidityCallback(Bob).metricOmmModifyLiquidityCallback(...)
    Bob transfers tokens to pool.

  Alice receives LP shares for bin 0.

Result:
  Bob has added liquidity to a pool that was supposed to exclude him.
  Pool's curBinIdx / curPosInBin are now shifted by Bob's deposit.
  Bob (or colluding Alice) can now swap against the pool or remove
  liquidity to recover tokens, completing the allowlist bypass.
