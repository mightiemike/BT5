### Title
`SwapAllowlistExtension` gates the router address instead of the originating user, allowing any caller to bypass per-user swap restrictions on allowlisted pools — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` = router address. If the pool admin allowlists the router address to enable router-mediated swaps, every user — including those the allowlist was meant to exclude — can bypass the per-user gate.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks the allowlist keyed by `[pool][sender]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` to the pool: [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) [6](#0-5) 

Because the router never forwards the originating user's address as `sender`, the extension cannot distinguish between different users going through the same router. The allowlist is keyed on the router contract address, not on the human actor.

This is structurally asymmetric with `DepositAllowlistExtension`, which correctly gates the `owner` field — the economic beneficiary — rather than `msg.sender` of `addLiquidity`: [7](#0-6) 

The pool documentation explicitly separates payer from owner for deposits, but no equivalent separation exists for swaps.

### Impact Explanation

Two concrete failure modes arise:

1. **Allowlist bypass (unauthorized access):** A pool admin allowlists the router address so that their allowlisted users can swap via the router. Because `allowedSwapper[pool][router] = true`, every user — including those the admin intended to exclude — can call `MetricOmmSimpleRouter` and pass the gate. Unauthorized traders can execute swaps on a pool designed to be restricted (e.g., institutional-only, KYC-gated), causing price impact that harms LPs and enabling arbitrage that drains pool value.

2. **Allowlist lockout (broken functionality):** A pool admin allowlists specific user addresses. Those users call the router. The extension checks `allowedSwapper[pool][router]`, which is `false`, so the swap reverts with `NotAllowedToSwap` even for legitimately allowlisted users. The allowlist becomes unusable with the router.

Both outcomes break the core invariant that the swap allowlist gates the economically relevant actor. The first outcome is the fund-impacting one: unauthorized swaps on a restricted pool directly harm LP principal through adverse price impact and arbitrage.

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary user-facing swap interface; most users will route through it rather than calling the pool directly.
- Any user can call the router — it is a public, permissionless contract.
- The bypass is triggered the moment a pool admin allowlists the router address, which is the natural action to take when allowlisted users report that router swaps are failing.
- No special privilege, flash loan, or unusual token behavior is required.

### Recommendation

Pass the originating user's address as `sender` through the router, analogous to how `owner` is passed for deposits. One approach: add a `swapper` parameter to the router's swap functions and forward it as the first argument to `pool.swap()`. The pool would then pass this explicit swapper address to `_beforeSwap` instead of `msg.sender`. Alternatively, document that `SwapAllowlistExtension` gates the direct caller of `pool.swap()` and that router-mediated swaps must allowlist the router address, making per-user gating impossible through the router.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on the `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — intending to enable router-mediated swaps for their allowlisted users.
3. Attacker (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. `_beforeSwap` passes `sender = router` to `SwapAllowlistExtension.beforeSwap`.
6. The check `allowedSwapper[pool][router]` returns `true` → swap proceeds.
7. Attacker successfully swaps on a pool they were never individually authorized to access. [8](#0-7) [4](#0-3)
