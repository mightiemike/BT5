### Title
Arithmetic Mid Price Used Instead of Geometric Mid Price in `OracleValueStopLossExtension` Metric Computation Allows Stop-Loss to Fail Open - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean `(bid + ask) / 2`, while the pool's core swap math always uses the geometric mean `sqrt(bid * ask)`. Because the arithmetic mean is strictly greater than the geometric mean whenever `bid ≠ ask`, the per-share `metricT1` value (token1-denominated) is systematically inflated relative to the value the pool actually settled at. This inflation can prevent the stop-loss from triggering when the pool is genuinely below the configured drawdown floor, allowing value-leaking swaps to proceed and LP funds to drain.

### Finding Description

The pool's canonical mid price is the geometric mean of the oracle bid and ask, computed in `SwapMath.midAndSpreadFeeX64FromBidAsk`:

```solidity
// SwapMath.sol line 70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
```

Every swap settles at prices derived from this geometric mid. The `OracleValueStopLossExtension` receives the same `bidPriceX64` and `askPriceX64` in its `afterSwap` hook, but computes a different mid price:

```solidity
// OracleValueStopLossExtension.sol line 218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
```

This arithmetic mean is then fed into `_metrics`, which computes the token1-denominated per-share value:

```solidity
// OracleValueStopLossExtension.sol line 255
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

By AM-GM inequality, `(bid + ask)/2 ≥ sqrt(bid * ask)` always, with equality only when `bid == ask`. The arithmetic mid is therefore always at least as large as the geometric mid. A larger `midPriceX64` inflates the `t0 * midPriceX64 / Q64` term in `metricT1`, making the computed metric appear higher than the true oracle-settled value.

The watermark breach check in `_applyWatermark` is:

```solidity
// OracleValueStopLossExtension.sol line 334
breached = metric < (hwm * floorMultiplier) / E6;
```

When `metricT1` is inflated above the floor by the arithmetic-mean error, `breach1` is `false` even though the true geometric-mid metric is below the floor. The guard then allows the `!zeroForOne` swap (token1 in, token0 out) to complete:

```solidity
// OracleValueStopLossExtension.sol line 276
if (breach1 && !zeroForOne) {
    revert OracleStopLossTriggered(...);
}
```

The swap is not reverted, LP token0 leaves the pool, and the watermark is ratcheted up to the inflated (incorrect) metric value, resetting the high-water mark to a level that is too high. Subsequent swaps continue to compare against this inflated watermark, compounding the failure.

The symmetric error on `metricT0` (arithmetic mid deflates the `t1/price` term) causes false-positive stop-loss triggers on `zeroForOne` swaps, blocking legitimate trades.

### Impact Explanation

**Direct loss of LP principal.** The stop-loss extension exists specifically to prevent per-share value from falling below the configured drawdown floor. When the arithmetic-mean error inflates `metricT1` above the floor, the guard fails open on `!zeroForOne` swaps. Each such swap drains token0 from the pool at a price the stop-loss was supposed to block. The magnitude of the error scales with the oracle spread: for a spread `s` (where `ask = mid*(1+s)`, `bid = mid*(1-s)`), the arithmetic mid exceeds the geometric mid by approximately `mid * s²`. For a 1% oracle spread this is a 0.01% mid-price inflation; for a 5% spread (common during volatile markets) it is a 0.25% inflation. Near the drawdown boundary, even a small inflation is enough to suppress the breach flag and allow the draining swap to proceed.

### Likelihood Explanation

Every public swap through a pool that has `OracleValueStopLossExtension` configured in its `afterSwap` order triggers this code path. No privileged action is required. The window where the error matters is whenever the true geometric-mid metric is below the floor but the arithmetic-mid metric is above it — a condition that is reachable by any trader who times a swap when the pool is near the configured drawdown boundary. The attacker does not need to manipulate the oracle; they only need to observe that the pool is near the boundary and submit a `!zeroForOne` swap.

### Recommendation

Replace the arithmetic mean with the same geometric-mean formula the pool uses for swap settlement:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct, consistent with SwapMath):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(bidPriceX64, askPriceX64);
```

This ensures the stop-loss metrics are computed at the same price the pool used for settlement, eliminating the systematic inflation of `metricT1` and the systematic deflation of `metricT0`.

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured in `afterSwap`, with `drawdownE6 = 50_000` (5% drawdown floor) and a non-zero `decayPerSecondE8`.
2. Set the oracle to return `bid = 0.99 * mid_X64` and `ask = 1.01 * mid_X64` (1% spread). The geometric mid is `mid_X64`; the arithmetic mid is `mid_X64 * (1 + 0.0001)` — 0.01% higher.
3. Seed the pool with token0 and token1 such that the true geometric-mid `metricT1` is exactly at `hwm * 0.95` (just at the floor).
4. Execute a `!zeroForOne` swap. The extension computes `metricT1` using the arithmetic mid, which is 0.01% higher than the geometric mid. The inflated `metricT1` sits above the floor. `breach1` is `false`. The swap completes, draining token0.
5. Repeat: each swap updates the watermark to the inflated value, and the floor is never enforced.
