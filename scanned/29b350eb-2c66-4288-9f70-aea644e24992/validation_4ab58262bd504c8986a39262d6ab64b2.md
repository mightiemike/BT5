### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor address**. However, its `beforeAddLiquidity` hook validates the `owner` parameter (the address that will own the resulting position) rather than the `sender` parameter (the actual `msg.sender` who calls the pool and pays the tokens). Because `owner` is a free caller-controlled argument in `MetricOmmPool.addLiquidity`, any unprivileged address can bypass the allowlist entirely by passing an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` argument and passes both `msg.sender` (as `sender`) and `owner` to the extension hook: [1](#0-0) 

The pool imposes **no constraint** that `msg.sender == owner` for `addLiquidity` (unlike `removeLiquidity`, which enforces `msg.sender != owner` â†’ revert): [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both `sender` and `owner` to every registered extension: [3](#0-2) 

`DepositAllowlistExtension.beforeAddLiquidity` then performs its allowlist check exclusively against `owner`, silently ignoring the `sender` argument entirely: [4](#0-3) 

The mapping and admin setter are both named around the concept of a **depositor** (the caller), not the position owner: [5](#0-4) 

The mismatch is exact: the guard is keyed on `owner` (a free parameter), but the entity that actually deposits tokens and triggers the hook is `sender` (`msg.sender`).

---

### Impact Explanation

The deposit allowlist is completely ineffective. Any address â€” regardless of allowlist status â€” can call `addLiquidity` on a restricted pool by supplying any allowlisted address as `owner`. The unauthorized caller pays the tokens (via the liquidity callback on `msg.sender`) and the position is credited to the allowlisted `owner`. The pool admin's configured access control is bypassed by an unprivileged path with no special permissions required. This fits the **admin-boundary break** impact category: a factory/pool admin-configured guard is bypassed by an unprivileged caller.

---

### Likelihood Explanation

Exploitation requires only knowing one allowlisted address for the target pool, which is publicly readable from `allowedDepositor` or observable from past transactions. No privileged role, flash loan, or special setup is needed. Any EOA or contract can trigger this in a single transaction.

---

### Recommendation

Change the allowlist check to validate `sender` (the actual depositor/caller) instead of `owner`:

```solidity
// Current (wrong): checks owner, not the actual caller
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// Fixed: check sender, the address that is actually depositing
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to restrict which addresses may *own* positions (rather than which addresses may *call* the function), the naming (`setAllowedToDeposit`, `allowedDepositor`) and NatSpec ("Gates `addLiquidity` by depositor address") must be updated to reflect that, and the pool's `addLiquidity` should enforce `msg.sender == owner` to prevent the same bypass via a different path.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` registered on `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` â€” only `alice` is meant to deposit.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. The pool calls `extension.beforeAddLiquidity(bob, alice, ...)`.
5. The extension checks `allowedDepositor[pool][alice]` â†’ `true` â†’ no revert.
6. `bob`'s callback pays the tokens; the position is recorded under `alice`.
7. `bob` has successfully deposited into a pool he is not authorized to access, bypassing the allowlist entirely. [4](#0-3) [6](#0-5)
