### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass Per-User Swap Allowlist — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which is `msg.sender` of the pool's `swap()` call. When users route through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the actual end user. A pool admin who allowlists the router to permit any router-mediated swap inadvertently opens the gate to every user, defeating the per-user curation the extension is designed to enforce.

---

### Finding Description

The pool's `swap()` function passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to the extension: [1](#0-0) 

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(params.recipient, ...)` directly, so the pool sees `msg.sender = router`: [2](#0-1) 

The extension then checks `allowedSwapper[pool][sender]`, where `sender` is the router address: [3](#0-2) 

The actual initiating user (`msg.sender` of `exactInputSingle`) is stored only in transient storage for the payment callback and is **never forwarded to the extension**. The extension has no mechanism to distinguish individual users arriving through the same router.

This creates an irreconcilable dilemma for any pool admin:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | All router-mediated swaps revert — legitimate allowlisted users cannot use the router at all |
| Allowlist the router | Every user on the network can swap through the router — the per-user allowlist is completely bypassed |

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (second argument), which `MetricOmmPoolLiquidityAdder` populates with the actual position owner, not the adder contract itself: [4](#0-3) 

The swap extension applies the wrong actor binding relative to its deposit-side counterpart.

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a whitelist of addresses (e.g., KYC'd counterparties, protocol-owned addresses, or specific market makers) loses all enforcement the moment the router is allowlisted. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and trade against the pool's LP assets at oracle-derived prices, draining value from LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct loss of LP principal and a broken core pool functionality (curation) above Sherlock high-severity thresholds.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint documented in the periphery. Any pool admin who configures `SwapAllowlistExtension` and then attempts to allow legitimate users to swap via the router will naturally allowlist the router address. The bypass is triggered by normal, expected usage of the supported periphery path — no special exploit setup is required.

---

### Recommendation

The `beforeSwap` hook should gate on the **economic initiator**, not the intermediary. Two complementary fixes:

1. **Short-term — check `recipient` as a proxy, or require the router to embed the real user in `extensionData`**: The extension can decode a user address from `extensionData` when `sender` is a known router. This is fragile because `extensionData` is user-supplied.

2. **Correct fix — align with `DepositAllowlistExtension`**: Change `SwapAllowlistExtension` to check the `recipient` field (which the router sets to the user-specified output receiver) instead of `sender`. While `recipient` is also user-controlled, it is the closest available proxy for the economic beneficiary in the current interface. Alternatively, extend the `beforeSwap` hook interface to carry a separate `initiator` field populated by the pool from a trusted source.

```solidity
// Current (wrong actor):
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// Proposed (check recipient — the economic beneficiary):
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice should be able to swap.
3. Admin also calls `setAllowedToSwap(pool, router, true)` so that Alice can use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The pool calls `extension.beforeSwap(router, bobRecipient, ...)`.
6. The extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Bob's swap executes against LP assets despite never being allowlisted.

The check at step 6 uses the router address, not Bob's address, so the per-user gate is silently open to all router users. [5](#0-4) [6](#0-5) [2](#0-1)
