### Title
`SwapAllowlistExtension` gates the router address instead of the real user, allowing any disallowed swapper to bypass the allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument it receives from the pool. The pool always passes its own `msg.sender` as `sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks whether the **router** is allowlisted — not the actual user. Any disallowed user can bypass the per-user swap allowlist by routing through the router if the router is allowlisted, or the allowlist silently blocks all router-mediated swaps for legitimate users if the router is not allowlisted.

---

### Finding Description

**Actor binding in `SwapAllowlistExtension`**

`SwapAllowlistExtension.beforeSwap` enforces the guard as:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the contract calling the extension), and `sender` is the first argument forwarded by the pool.

**What the pool passes as `sender`**

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

**What the router passes to the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
```

The router is `msg.sender` of the pool's `swap()` call. Therefore the pool passes the **router address** as `sender` to `_beforeSwap`, and the extension checks `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actualUser]`.

**The two broken states this creates:**

| Pool admin configuration | Direct swap by disallowed user | Router swap by disallowed user |
|---|---|---|
| Router NOT allowlisted | Blocked ✓ | Blocked — but so are all legitimate router users ✗ |
| Router allowlisted | Blocked ✓ | **Allowed — allowlist fully bypassed** ✗ |

In the second row, any disallowed user calls `exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) and the extension sees the allowlisted router, passes the check, and the swap executes. The individual-user allowlist is completely defeated for all router-mediated paths.

---

### Impact Explanation

**Direct loss / policy bypass on curated pools.** The `SwapAllowlistExtension` is the production mechanism for restricting which addresses may trade on a pool. If the pool admin allowlists the router to support normal user flows, every disallowed address gains unrestricted swap access through that same router. This is a complete curation failure: the pool cannot simultaneously support router-mediated swaps and enforce per-user restrictions. Any value-extracting or regulatory-restricted address that is explicitly blocked from direct swaps can trivially bypass the gate by routing through `MetricOmmSimpleRouter`.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool that deploys `SwapAllowlistExtension` and also expects users to swap through the router (the normal production flow) is immediately vulnerable. The attacker needs no special setup: they simply call the public router functions. The pool admin has no way to fix this without either removing the router from the allowlist (breaking all router users) or removing the allowlist entirely.

---

### Recommendation

The extension must check the **original user**, not the intermediary. Two complementary approaches:

1. **Pass the original user through the router.** The router already tracks the real payer in transient storage (`_getPayer()`). Encode the original `msg.sender` into `extensionData` and have the extension decode and check it. This requires a coordinated extension format.

2. **Check `recipient` instead of `sender` for swap allowlists.** For exact-input swaps the recipient is the intended beneficiary; however this does not work for exact-output where the recipient is set by the caller.

3. **Preferred: gate on `recipient` or add a dedicated "originator" field.** The cleanest fix is to add an `originator` parameter to the `beforeSwap` hook (the address that initiated the economic action) distinct from `sender` (the immediate caller). The pool would pass a caller-supplied originator, and the router would forward `msg.sender` as originator in `extensionData`.

Until the hook interface is extended, the minimum safe workaround is: **do not allowlist the router on pools that use `SwapAllowlistExtension`**, and require users to call the pool directly. Document this constraint explicitly.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook
  - Pool admin allowlists the router: setAllowedToSwap(pool, router, true)
  - Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] == false

Attack:
  1. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, zeroForOne, amount, ...)
     → pool.msg.sender = router
  3. Pool calls _beforeSwap(msg.sender=router, ...)
  4. SwapAllowlistExtension.beforeSwap receives sender=router
  5. Check: allowedSwapper[pool][router] == true  → passes
  6. Swap executes for attacker despite attacker being individually blocked

Result: attacker swaps on a curated pool that was supposed to block them.
```

The existing test `test_allowedSwapSucceeds` in `FullMetricExtension.t.sol` (line 68–74) allowlists `callers[0]` (a `TestCaller` contract) and calls `_swap` through that caller — this test passes because the caller IS the pool's `msg.sender`. No test covers the router-mediated path where the router is allowlisted but the originating user is not, which is exactly the gap that hides this bypass.
