### Title
`SwapAllowlistExtension.beforeSwap` checks the immediate pool caller (router) instead of the actual end-user swapper, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to gate swaps by swapper address per pool. However, its `beforeSwap` hook checks the `sender` parameter, which is `msg.sender` of the pool's `swap()` call — the router contract — not the actual end user. When the router is allowlisted (a necessary step for any router-mediated swap to work), every user can bypass the per-user restriction by routing through `MetricOmmSimpleRouter`.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its allowlist check against the `sender` argument: [1](#0-0) 

`sender` is the first parameter forwarded by the pool's `_beforeSwap` dispatcher, which is `msg.sender` of the `pool.swap()` call: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router is the entity that calls `pool.swap()`: [3](#0-2) 

So `sender` seen by the extension is the **router address**, not the actual user. The allowlist check `allowedSwapper[pool][router]` passes for every user who routes through the router, regardless of whether that user is individually allowlisted.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all of which call `pool.swap()` from the router's address.

---

### Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses (e.g., KYC-verified market makers, whitelisted counterparties). The guard is completely bypassed for any user who routes through `MetricOmmSimpleRouter`:

- Unauthorized users can execute swaps against the pool, draining LP-owned token reserves at oracle-quoted prices.
- The pool admin has no on-chain mechanism to enforce per-user restrictions while simultaneously supporting router-mediated swaps.
- LP principal is at direct risk: any unauthorized actor can extract token1 (or token0) from the pool at the oracle bid/ask, with no recourse once the swap settles.

---

### Likelihood Explanation

The precondition is that the router is allowlisted for the pool. This is a **necessary** configuration step: if the router is not allowlisted, no user can swap through the router at all, making the router unusable for that pool. Any pool admin who wants to support router-mediated swaps for even one allowlisted user must allowlist the router — at which point the bypass is open to all users. The trigger is an unprivileged call to any `exact*` function on `MetricOmmSimpleRouter`.

---

### Recommendation

The `beforeSwap` hook should check the **actual end user**, not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation and is trust-dependent.
2. **Check the first parameter as the caller, not the swapper**: Redesign the allowlist to gate on the pool-level `msg.sender` (the router) and require the router to attest the real user identity — or gate only on direct pool calls and document that router-mediated swaps are always open.

The cleanest fix is to have the router forward the originating user in `extensionData` and have the extension decode and verify that identity, so the allowlist check is always against the economic actor, not the intermediary.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — required for any router-mediated swap to work.
3. Pool admin calls `setAllowedToSwap(pool, unauthorizedUser, false)` — intending to block this user.
4. `unauthorizedUser` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)`.
6. The pool calls `extension.beforeSwap(routerAddress, recipient, ...)`.
7. The check `allowedSwapper[pool][routerAddress]` returns `true` — bypass succeeds.
8. The swap executes; `unauthorizedUser` receives tokens from the pool. [1](#0-0) [4](#0-3)
