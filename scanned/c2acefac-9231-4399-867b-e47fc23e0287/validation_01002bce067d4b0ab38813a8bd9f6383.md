### Title
SwapAllowlistExtension Checks Router Address as `sender` Instead of Actual User, Allowing Any User to Bypass the Per-User Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the original user. If the router is allowlisted (a natural admin action to enable router-based swaps), every user on the internet can bypass the per-user allowlist entirely.

### Finding Description

The pool's `swap` function passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // <-- router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
```

The router is a public, permissionless contract. Any user can call it. If the pool admin allowlists the router address (the natural step to enable router-based swaps for any user), the allowlist check passes for every caller regardless of whether they are individually permitted.

The same bypass applies to multi-hop `exactInput` and `exactOutput` paths.

### Impact Explanation

Two mutually exclusive failure modes exist, both fund-impacting:

1. **Allowlist bypass (primary)**: If the admin allowlists the router to enable router-based swaps, every unprivileged address can swap on a pool that was intended to be restricted. Restricted pools may hold concentrated LP positions whose value depends on controlled access (e.g., private market-making pools, KYC-gated pools). Unauthorized swaps drain LP value at oracle prices without restriction.

2. **Broken core swap flow**: If the admin allowlists only specific user addresses (not the router), those allowlisted users cannot use the router at all — their swaps revert with `NotAllowedToSwap` even though they are individually permitted. This breaks the primary swap UX path for legitimate participants.

Both outcomes violate the core invariant that the allowlist gates the exact actor the pool designer intended.

### Likelihood Explanation

- Trigger is fully unprivileged: any EOA calls `MetricOmmSimpleRouter.exactInputSingle`.
- The router is a public, deployed periphery contract with no access control.
- The admin allowlisting the router is the expected operational step to enable router-based swaps; the bypass is therefore the default outcome of normal configuration.
- No special timing, flash loan, or oracle manipulation is required.

### Recommendation

The extension must recover the original user identity rather than trusting the pool-forwarded `sender`. Two options:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a convention between router and extension.

2. **Check `sender` against the router and require the router to forward the real user**: Add a `getRealSender()` interface to the router that the extension can call back, or use ERC-2771-style meta-transaction forwarding.

The simplest correct fix is for the extension to check `tx.origin` as a fallback when `sender` is a known router, but the cleanest solution is to encode the real user in `extensionData` and have the extension verify it when `sender` is a recognized intermediary.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  alice (not individually allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})
      → pool.swap(msg.sender=router, ...)
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router)
            → allowedSwapper[pool][router] == true  → PASSES
  Alice's swap executes despite never being individually allowlisted.

Alternatively:
  admin calls setAllowedToSwap(pool, alice, true)    // allowlist alice directly
  alice calls router.exactInputSingle({pool: pool, ...})
    → pool.swap(msg.sender=router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router)
        → allowedSwapper[pool][router] == false → REVERTS NotAllowedToSwap
  Alice cannot use the router even though she is allowlisted.
```
