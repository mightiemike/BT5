### Title
`DepositAllowlistExtension` Guards `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist â€” (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently passes for any unauthorized `sender` who specifies an allowlisted `owner`. The guard checks the wrong address, so the allowlist never fires against the actual depositing party.

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook: `msg.sender` (the entity calling `addLiquidity`, i.e., the actual depositor) as `sender`, and the caller-supplied `owner` (the LP-position beneficiary). [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and evaluates only `owner`: [3](#0-2) 

Because `owner` is a free parameter supplied by the caller of `addLiquidity`, any address can pass the guard by setting `owner` to any address that the pool admin has previously allowlisted. The actual depositor (`sender`) is never checked.

### Impact Explanation

An unauthorized address calls `pool.addLiquidity(owner = knownAllowedAddress, ...)`. The `DepositAllowlistExtension` evaluates `allowedDepositor[pool][knownAllowedAddress]` â€” which is `true` â€” and returns the expected selector without reverting. The pool then executes the full liquidity-add path: [4](#0-3) 

The unauthorized caller's tokens are pulled via the modify-liquidity callback, bin balances and `binTotals` are updated, and the LP shares are credited to `owner` â€” not to the caller. The caller has no position and no recovery path (`removeLiquidity` enforces `msg.sender == owner`): [5](#0-4) 

Consequences:
1. **Allowlist bypass**: a pool configured as restricted (e.g., KYC-gated, institutional-only) accepts deposits from any address.
2. **Permanent principal loss**: the unauthorized depositor's tokens are locked in the pool under `owner`'s position, irrecoverable by the depositor.

### Likelihood Explanation

- Exploitability requires only knowing one allowlisted address (observable on-chain via `AllowedToDepositSet` events or direct mapping reads).
- No special role, flash loan, or oracle manipulation is needed â€” a plain `addLiquidity` call suffices.
- Any pool that deploys `DepositAllowlistExtension` with a non-empty allowlist is affected.

### Recommendation

Check `sender` (the first parameter) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swap caller): [6](#0-5) 

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT allowlisted

Attack:
  bob calls pool.addLiquidity(
      owner  = alice,   // allowlisted â€” guard passes
      salt   = 0,
      deltas = { binIdxs: [0], shares: [1e18] },
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    sender (bob) is silently discarded
    owner  (alice) passes the allowlist check

  LiquidityLib.addLiquidity executes:
    - bin balances updated
    - positionBinShares[keccak(alice, 0, 0)] += 1e18
    - callback fires, pulling bob's tokens into the pool

Result:
  - bob's tokens are in the pool
  - alice holds the LP position
  - bob cannot call removeLiquidity (NotPositionOwner)
  - pool's deposit restriction is silently bypassed
```
