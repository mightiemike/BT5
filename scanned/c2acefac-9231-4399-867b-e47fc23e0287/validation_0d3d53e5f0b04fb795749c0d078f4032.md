### Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension`, breaking per-user allowlist gating — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` from the pool's perspective. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The hook therefore checks whether the **router** is allowlisted, not the actual economic actor.

### Finding Description

In `MetricOmmPool.swap()`, the `sender` forwarded to the extension is always `msg.sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` passes this value unchanged as the first argument to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

In `MetricOmmSimpleRouter.exactInputSingle`, the pool's `swap` is called with the router as `msg.sender`: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

### Impact Explanation

Two concrete broken invariants result:

**Bypass path**: If the pool admin allowlists the router address (the natural configuration to enable router-mediated swaps for their allowlisted users), then **every user** — including those not individually allowlisted — can bypass the per-user gate by routing through `MetricOmmSimpleRouter`. The allowlist becomes entirely ineffective for router-mediated swaps.

**Denial path**: If the pool admin allowlists only individual user addresses (not the router), those allowlisted users **cannot** use the router at all — the hook sees the router address and reverts. The two configurations are mutually exclusive: there is no way to simultaneously allow specific users through the router while blocking others.

The `SwapAllowlistExtension` is designed to gate access to curated pools (e.g., KYC-gated, institutional, or restricted pools). Either outcome breaks the core purpose of the extension.

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and also wants to support the official `MetricOmmSimpleRouter` faces this dilemma immediately. Allowlisting the router is the obvious step to enable router support, and doing so silently opens the pool to all users. This is a predictable and likely misconfiguration path for any production pool using this extension.

### Recommendation

The `beforeSwap` hook should not rely on `sender` (the immediate caller) to identify the economic actor. Options:

1. Pass the actual payer/originator through `extensionData` and verify it in the hook, with the router encoding `msg.sender` (the user) into `extensionData` before calling the pool.
2. Require direct pool interaction for allowlisted pools (document that the router is incompatible with `SwapAllowlistExtension`).
3. Redesign the extension to check `recipient` or a signed attestation rather than `sender`.

### Proof of Concept

```
1. Deploy pool P with SwapAllowlistExtension; set allowedSwapper[P][router] = true
   (admin enables router support).
2. Non-allowlisted attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...}).
3. Router calls P.swap(recipient, ...) — msg.sender to pool = router.
4. Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...).
5. Hook checks allowedSwapper[P][router] → true → passes.
6. Attacker completes swap on a pool they were never individually allowlisted for.

Alternatively:
1. Deploy pool P with SwapAllowlistExtension; set allowedSwapper[P][userA] = true
   (router NOT allowlisted).
2. UserA calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...}).
3. Hook checks allowedSwapper[P][router] → false → reverts NotAllowedToSwap().
4. UserA is blocked from using the router despite being individually allowlisted.
```
