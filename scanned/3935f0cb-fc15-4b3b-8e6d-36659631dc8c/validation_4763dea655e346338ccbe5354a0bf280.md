### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is designed to gate who may call `addLiquidity` on a restricted pool. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the caller-supplied `owner` (the position beneficiary). Because `owner` is a free parameter that any caller can set to any address, any unprivileged address can bypass the allowlist entirely by nominating an already-allowlisted address as the position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
// MetricOmmPool.sol – addLiquidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension hook receives both arguments, but discards `sender` (unnamed `address,`) and validates only `owner`:

```solidity
// DepositAllowlistExtension.sol – beforeAddLiquidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The admin-facing setter names its parameter `depositor`, making the intent unambiguous — the guard is meant to restrict who provides tokens, i.e., the `sender`:

```solidity
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
``` [3](#0-2) 

Because `owner` is a free parameter with no constraint that it equals `msg.sender` (unlike `removeLiquidity`, which enforces `msg.sender == owner`), any caller can pass any allowlisted address as `owner` and the guard passes unconditionally. [4](#0-3) 

---

### Impact Explanation

**Broken core pool functionality / admin-boundary break.** The deposit allowlist is the pool admin's primary mechanism for restricting who may provide liquidity to a pool (e.g., KYC-gated pools, curated market-maker-only pools). With the guard checking the wrong actor:

1. **Full allowlist bypass**: Any unprivileged address calls `addLiquidity(owner = allowlisted_address, ...)`. The extension sees an allowlisted `owner`, passes, and the deposit proceeds.
2. **Unauthorized liquidity injection**: The attacker deposits tokens into arbitrary bins, disrupting the pool's intended liquidity profile, diluting existing LPs' fee earnings, and potentially shifting `curBinIdx`/`curPosInBin` in ways the pool admin did not sanction.
3. **Irrecoverable token loss for the caller**: Because `removeLiquidity` enforces `msg.sender == owner`, the attacker cannot reclaim the deposited tokens — they are permanently credited to the allowlisted `owner`. This makes the bypass a one-way, fund-consuming action, but the pool's invariant (restricted depositor set) is still broken.

---

### Likelihood Explanation

**High.** No special privilege is required. Any externally-owned account or contract can call `addLiquidity` with `owner` set to any address that appears in `allowedDepositor`. The allowlisted address is publicly readable on-chain via `allowedDepositor(pool, addr)` or `isAllowedToDeposit`. The bypass requires a single transaction with no preconditions.

---

### Recommendation

Replace the `owner` check with a `sender` check to match the stated intent of the allowlist:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
``` [2](#0-1) 

---

### Proof of Concept

```
Setup:
  - Pool P deployed with DepositAllowlistExtension E configured.
  - Admin calls E.setAllowedToDeposit(P, Alice, true).
  - Bob is NOT in the allowlist.

Attack:
  1. Bob calls P.addLiquidity(owner = Alice, salt, deltas, callbackData, extensionData).
  2. Pool calls E.beforeAddLiquidity(sender=Bob, owner=Alice, ...).
  3. Extension evaluates: allowedDepositor[P][Alice] == true → no revert.
  4. LiquidityLib.addLiquidity executes; Bob's tokens are pulled via callback.
  5. Position is minted under Alice's key (owner=Alice, salt).
  6. Bob's tokens are now locked in Alice's position; Bob cannot remove them
     (removeLiquidity enforces msg.sender == owner).

Result:
  - Bob (not allowlisted) successfully deposited into a restricted pool.
  - The deposit allowlist is completely ineffective against any caller who
    knows at least one allowlisted address.
```
