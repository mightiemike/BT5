### Title
`SwapAllowlistExtension.beforeSwap` gates the intermediary caller (`sender`) instead of the actual swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to restrict swaps on curated pools to a configured set of addresses. Its `beforeSwap` hook checks `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When swaps are routed through the public `MetricOmmSimpleRouter`, `sender` equals the router's address, not the end user's address. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently opens the pool to every user, because any unprivileged actor can call the router and have their swap pass the allowlist check under the router's identity.

---

### Finding Description

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then uses that `sender` as the key for its allowlist lookup:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When a user calls `MetricOmmSimpleRouter` (a public, permissionless contract), the router calls `pool.swap()`, so `msg.sender` inside the pool is the router address. The allowlist check therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This is the direct analog of the seeded bug: just as `isPairZeroFee(exOrderId)` used the wrong identifier for the guard lookup, `allowedSwapper[pool][sender]` uses the wrong actor — the intermediary router — instead of the actual economic actor (the end user).

Two concrete failure modes follow:

**Mode A — Bypass (High impact):** The pool admin sets `allowedSwapper[pool][router] = true` to allow router-mediated swaps for their permitted users. Because the router is a public contract, every unprivileged user can now route through it and pass the allowlist check. The restriction is completely nullified.

**Mode B — Broken flow (Medium impact):** The admin does not allowlist the router. Legitimately allowlisted users who attempt to swap through `MetricOmmSimpleRouter` receive `NotAllowedToSwap`, even though they are individually permitted. They are forced to call the pool directly, breaking the supported periphery flow.

---

### Impact Explanation

**Mode A** is a direct loss-of-curation impact: a pool designed to restrict trading to specific counterparties (e.g., whitelisted market makers or KYC'd participants) can be traded against by any unprivileged address. Unauthorized swaps drain LP value at oracle-derived prices without the LP's consent, constituting a direct loss of LP principal and a broken pool invariant ("only permitted swappers may trade").

**Mode B** breaks the core swap flow for legitimately allowlisted users through the supported periphery path, rendering the router unusable for curated pools.

---

### Likelihood Explanation

Allowlisting the router is the natural, expected configuration for any pool that wants to support both direct and router-mediated swaps for its permitted users. The admin has no way to achieve this intent correctly with the current code — allowlisting the router is the only mechanism available, and it triggers the bypass. The trigger requires no privileged malicious action; the admin acts in good faith, and any unprivileged user completes the exploit by calling the public router.

---

### Recommendation

The extension must gate the actual economic actor, not the intermediary. Two options:

1. **Pass the originating user through `extensionData`**: The router encodes the end user's address in `extensionData`; the extension decodes and checks it. This requires the router to be trusted to supply the correct value.

2. **Add an originator field to the hook interface**: Extend `beforeSwap` with an explicit `originator` parameter that the pool populates from a transient context set by the router before calling `pool.swap()`. This is the most robust fix and mirrors how Uniswap v4 handles hook actor binding.

Until fixed, pools relying on `SwapAllowlistExtension` should not allowlist the router and should document that router-mediated swaps are incompatible with the allowlist guard.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Admin calls: extension.setAllowedToSwap(pool, router, true)
   // Intended: allow router so permitted users can swap via router
3. Disallowed user (alice, NOT in allowlist) calls:
   MetricOmmSimpleRouter.exactInput(pool, ..., extensionData)
4. Router calls: pool.swap(recipient, zeroForOne, amount, priceLimit, callbackData, extensionData)
   // msg.sender inside pool = router address
5. Pool calls: _beforeSwap(msg.sender=router, ...)
6. Extension evaluates: allowedSwapper[pool][router] == true  → passes
7. Alice's swap executes successfully despite not being allowlisted.
```
