### Title
SwapAllowlistExtension Gates the Router Address Instead of the End-User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end-user. If the router address is allowlisted (the only way to let allowlisted users trade via the router), every non-allowlisted user can also bypass the guard by routing through the same public router.

### Finding Description

The pool's `swap` function passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value just described: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [4](#0-3) 

The extension therefore checks `allowedSwapper[pool][router]` — the router's allowlist entry — rather than the originating user's entry. There are two broken outcomes:

1. **Bypass path**: The pool admin allowlists the router address so that allowlisted users can trade via the router. Because the check is on the router, every user — including those explicitly excluded — can call `exactInputSingle` and pass the guard.
2. **Broken path**: The pool admin does not allowlist the router. Now even explicitly allowlisted users cannot trade via the router, breaking the intended UX.

Neither outcome matches the intended invariant. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the position owner), not `sender`, so the deposit guard is not affected: [5](#0-4) 

### Impact Explanation

A curated pool that relies on `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise approved addresses is fully bypassed. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`) against the pool and execute swaps that the allowlist was designed to block. The pool's token reserves are directly accessible to the bypassing user, constituting a broken core pool functionality and a direct loss of the curation guarantee that LP depositors relied upon when providing liquidity.

### Likelihood Explanation

The router is the standard, documented entry point for end-user swaps. Any pool that deploys `SwapAllowlistExtension` and also expects users to trade via the router must allowlist the router, which immediately opens the bypass to all users. The trigger requires only a standard `exactInputSingle` call — no special permissions, no flash loans, no multi-step setup.

### Recommendation

The extension must recover the true end-user identity. Two sound approaches:

1. **Pass the original caller through the router**: Add a `payer`/`originator` field to the swap path or extension data that the router populates with `msg.sender` before calling the pool, and have the extension verify that field against the allowlist. The pool's `sender` parameter already carries the router address and cannot be repurposed without a protocol-level change.

2. **Check `sender` against the allowlist only when `sender` is not a known router**: Maintain a registry of trusted routers in the extension; when `sender` is a trusted router, require the allowlist check to pass for the `recipient` or an explicit `originator` encoded in `extensionData`.

The deposit allowlist's pattern of checking `owner` (the economically relevant actor) rather than `sender` (the intermediary) is the correct model and should be mirrored in the swap allowlist.

### Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension as extension1,
         beforeSwap order = [1]
  allowedUser  = 0xAAAA  (explicitly allowlisted)
  blockedUser  = 0xBBBB  (not on allowlist)

Step 1 – admin allowlists the router so allowedUser can trade via router:
  SwapAllowlistExtension.setAllowedToSwap(pool, router, true)

Step 2 – blockedUser calls router directly:
  MetricOmmSimpleRouter.exactInputSingle({
    pool:        pool,
    zeroForOne:  true,
    amountIn:    1000,
    recipient:   blockedUser,
    ...
  })

Execution trace:
  router.exactInputSingle()
    → pool.swap(msg.sender = router, ...)
      → _beforeSwap(sender = router, ...)
        → SwapAllowlistExtension.beforeSwap(sender = router, ...)
          → allowedSwapper[pool][router] == true  ✓  (guard passes)
      → swap executes, blockedUser receives tokens

Result:
  blockedUser successfully swaps despite not being on the allowlist.
  The guard checked the router's allowlist entry (which is valid) instead
  of blockedUser's entry (which is absent), exactly mirroring the
  "session not invalidated" pattern: the guard was satisfied by a
  credential that belongs to a different identity than the one performing
  the action.
