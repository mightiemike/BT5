### Title
`SwapAllowlistExtension` gates the router address instead of the originating user, allowing any user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router**, not the user. If the pool admin allowlists the router (the only way to let any user swap through it), every unpermissioned user can bypass the allowlist by routing through the router. If the router is not allowlisted, no allowlisted user can use the router at all. Either outcome breaks the intended access control.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
              msg.sender = router
         → _beforeSwap(msg.sender=router, ...)
         → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              checks allowedSwapper[pool][router]
```

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = router. The check becomes `allowedSwapper[pool][router]`.

**Two broken outcomes:**

1. **Allowlist bypass (high impact):** Pool admin allowlists the router so that allowlisted users can swap through it. Because the check is on the router address, every user — including those not on the allowlist — can now swap by routing through `MetricOmmSimpleRouter`.

2. **Broken functionality:** Pool admin does not allowlist the router. Allowlisted users cannot use the router at all; they must call the pool directly. The periphery is effectively unusable for allowlisted pools.

The `DepositAllowlistExtension` does **not** share this flaw: it checks `owner` (the position owner), which is correctly forwarded as the actual depositor regardless of whether the call comes through `MetricOmmPoolLiquidityAdder` or directly.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict swapping to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers). The bypass allows any unpermissioned address to execute swaps on such a pool by routing through the public `MetricOmmSimpleRouter`. This directly violates the pool admin's access-control invariant and allows unauthorized parties to trade against the pool's liquidity, potentially draining LP value through arbitrage or front-running that the allowlist was designed to prevent.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any user who discovers the allowlist can trivially route through the router. The router is a public, permissionless contract. No privileged access or special setup is required beyond the pool admin having allowlisted the router (the only way to make the router usable for any user).

---

### Recommendation

The extension must gate the **economic actor**, not the immediate caller. Two options:

1. **Pass the original user through the router.** Modify `MetricOmmSimpleRouter` to encode the originating `msg.sender` in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address. This requires a convention between router and extension.

2. **Check `recipient` instead of `sender`.** If the pool's design intent is that the recipient is the economic beneficiary, check `allowedSwapper[pool][recipient]`. This is simpler but only correct if recipient always equals the user.

3. **Preferred: gate on `sender` but require the router to be non-allowlistable.** Document that pools using `SwapAllowlistExtension` must not use the router, and add a check in the extension or factory that rejects router addresses from the allowlist.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; alice is allowlisted, bob is not.
swapExtension.setAllowedToSwap(address(pool), alice, true);
// router is allowlisted so alice can use it
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Bob (not allowlisted) calls directly → reverts correctly
vm.prank(bob);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(bob, false, 1000, type(uint128).max, "", "");

// Bob routes through the router → succeeds, bypassing the allowlist
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: bob,
    tokenIn: token1,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// Bob successfully swaps despite not being on the allowlist.
```
