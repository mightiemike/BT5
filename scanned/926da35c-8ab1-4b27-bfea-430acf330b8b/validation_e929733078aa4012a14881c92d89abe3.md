### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Non-Allowlisted Users to Bypass the Swap Gate via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. However, `MetricOmmPool.swap` always passes `msg.sender` as `sender` — which is the **router contract**, not the originating user — when a swap is routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (a natural configuration), every user, including those not individually allowlisted, can bypass the per-user swap gate by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- always the direct caller of swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` (the direct caller) is allowlisted for the pool:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the **router**, so `sender` forwarded to `beforeSwap` is the router's address, not the originating user's address. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` directly, making the router the `msg.sender` in every case.

**Bypass scenario:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow "normal" users to swap through the standard router (a natural configuration — without this, no router user can swap at all).
3. A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. `beforeSwap` receives `sender = router`, checks `allowedSwapper[pool][router] == true`, and passes.
6. The non-allowlisted user's swap executes successfully, bypassing the per-user gate.

**Inverse problem:** If the admin does NOT allowlist the router, then individually allowlisted users also cannot use the router — they must call `pool.swap()` directly. This makes the allowlist incompatible with the standard periphery.

---

### Impact Explanation

The `SwapAllowlistExtension` is the primary mechanism for pools to restrict swap access to specific counterparties (e.g., KYC'd users, institutional traders, or whitelisted market participants). When the router is allowlisted — which is required for any router-mediated swap to work — the per-user allowlist is entirely bypassed. Any user can trade against the pool's liquidity at oracle-anchored prices, defeating the LP's intended access control. LPs who deployed a restricted pool to avoid toxic flow or unvetted counterparties are exposed to trades they explicitly intended to block, resulting in direct loss of LP assets through unintended swap settlement.

---

### Likelihood Explanation

The likelihood is medium-high. Any pool that uses `SwapAllowlistExtension` and also wants users to access the pool through the standard `MetricOmmSimpleRouter` must allowlist the router. This is the only way to make the router work with an allowlisted pool, so the misconfiguration is structurally forced. A pool admin who allowlists the router believing it preserves per-user restrictions will unknowingly open the pool to all users.

---

### Recommendation

Pass the **originating user** as `sender` rather than the intermediate router. The router already tracks the original `msg.sender` in transient storage as the payer (via `_setNextCallbackContext`). The pool should accept an explicit `sender` parameter from the caller, or the extension should read the originating user from a trusted forwarding mechanism. Alternatively, the `SwapAllowlistExtension` documentation must explicitly warn that allowlisting the router grants access to all users, and pool admins must allowlist individual users and require them to call the pool directly.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][router] = true   (admin allowlists router)
  - allowedSwapper[pool][alice] = true    (alice is individually approved)
  - allowedSwapper[pool][bob]   = false   (bob is NOT approved)

Direct call (bob → pool.swap):
  pool.swap() receives msg.sender = bob
  beforeSwap(sender=bob) → allowedSwapper[pool][bob] == false → REVERT ✓

Router call (bob → router.exactInputSingle → pool.swap):
  pool.swap() receives msg.sender = router
  beforeSwap(sender=router) → allowedSwapper[pool][router] == true → PASS ✗

Result: bob bypasses the swap allowlist by routing through MetricOmmSimpleRouter.
```
