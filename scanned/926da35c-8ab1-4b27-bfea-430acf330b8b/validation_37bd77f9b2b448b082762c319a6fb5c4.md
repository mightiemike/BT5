### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the end user, allowing any user to bypass a per-user swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` — the `MetricOmmSimpleRouter` — not the originating end user. When a pool admin allowlists the router (a natural configuration for "allow swaps through the official periphery"), every unprivileged user can swap against a pool that was designed to be access-controlled.

---

### Finding Description

The pool calls `_beforeSwap(msg.sender, ...)` where `msg.sender` is whoever called `pool.swap()`. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value as the `sender` argument to every configured extension. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` as `msg.sender`. The pool therefore passes `sender = address(router)` to the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. [4](#0-3) 

The same mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

Note the contrast with `DepositAllowlistExtension`, which correctly ignores `sender` and gates by `owner` (the position owner), making the deposit guard work correctly through the `MetricOmmPoolLiquidityAdder`. [6](#0-5) 

---

### Impact Explanation

Two fund-impacting outcomes arise from the same root cause:

**Bypass (primary impact):** A pool admin allowlists the router address — a natural configuration meaning "allow swaps through the official periphery." Because the extension checks the router, not the end user, every unprivileged user can swap against a pool that was designed to be restricted (e.g., KYC-gated, institutional-only, or whitelist-only). Unauthorized users can drain LP liquidity at oracle prices with no access control enforced.

**Broken allowlist (secondary impact):** A pool admin allowlists individual user addresses. Those users call the router. The router is not in the allowlist, so `NotAllowedToSwap` reverts for every allowlisted user going through the standard periphery. The pool's swap functionality is broken for its intended participants.

Both outcomes represent broken core pool functionality with direct LP fund impact.

---

### Likelihood Explanation

The trigger requires no privileged action by the attacker. The pool admin allowlisting the router is a standard, expected configuration for any pool that wants to accept swaps through the official periphery. Once the router is allowlisted, any user can call `exactInputSingle` or `exactInput` and the extension passes. The attacker needs only a token approval and a call to the public router.

---

### Recommendation

Pass the originating end user through the swap path so the extension can gate the correct identity. One approach: have the router store the end user in transient storage and expose it via a callback, then pass it as an additional argument to `beforeSwap`. A simpler approach: change `SwapAllowlistExtension.beforeSwap` to check `recipient` instead of `sender` when the pool is called through a router, or require the pool to pass the original `tx.origin`-equivalent through a dedicated transient slot. The cleanest fix is to add a `swapInitiator` field to the `beforeSwap` hook arguments that the pool populates from a transient "initiator" slot set by the router before calling `pool.swap()`.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: swapExtension.setAllowedToSwap(pool, address(router), true)
    // admin intends: "allow swaps through the official router"
    // actual effect: any user can swap

Attack:
  attacker (not in any per-user allowlist) calls:
    router.exactInputSingle({pool: pool, tokenIn: token0, ...})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, zeroForOne, amount, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (passes)
        → swap executes, LP funds transferred to attacker

Result:
  Attacker swaps against a pool intended to be access-controlled.
  LP funds are transferred at oracle prices to an unauthorized counterparty.
  The per-user allowlist is completely bypassed.
