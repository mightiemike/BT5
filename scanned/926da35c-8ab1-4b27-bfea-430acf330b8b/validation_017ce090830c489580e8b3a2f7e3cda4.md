### Title
Caller-Supplied Bid/Ask Prices in `simulateSwapAndRevert` Are Forwarded to Extension Hooks Without Oracle Validation, Enabling Persistent Corruption of Oracle Guard Extension State — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`simulateSwapAndRevert` accepts caller-controlled `bidPriceX64` / `askPriceX64` and forwards them verbatim to `_beforeSwap` and `_afterSwap` extension hooks. Because extension contracts are separate accounts, any state they write during those hook calls persists on-chain even though the pool itself reverts. The real `swap` path, by contrast, fetches bid/ask from the oracle and validates them before passing them to the same hooks. The divergence means any unprivileged caller can drive oracle-guard extensions (e.g., `PriceVelocityGuardExtension`, `OracleValueStopLossExtension`) with arbitrary prices, corrupting their recorded observations, and then execute a real swap that the now-poisoned guard no longer blocks.

---

### Finding Description

**Real `swap` path — oracle-validated prices reach hooks:**

In `swap`, bid/ask are fetched from the configured price provider and validated before any hook is called:

```solidity
(uint128 bidPriceX64, uint128 askPriceX64) = _getBidAndAskPriceX64();
_beforeSwap(msg.sender, recipient, zeroForOne, amountSpecified,
            priceLimitX64, packedSlot0Initial,
            bidPriceX64, askPriceX64, extensionData);
```

`_getBidAndAskPriceX64` enforces `bid < ask` and `bid != 0` against the live oracle:

```solidity
try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
    if (bid >= ask) revert BidGreaterThanAsk();
    if (bid == 0) revert BidIsZero();
    return (bid, ask);
}

**`simulateSwapAndRevert` path — caller-supplied prices reach the same hooks:**

`simulateSwapAndRevert` accepts `bidPriceX64` and `askPriceX64` as explicit parameters and applies only a trivial structural check (`bid < ask`, `bid != 0`) — no oracle comparison:

```solidity

    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,
    uint128 bidPriceX64,   // ← caller-controlled
    uint128 askPriceX64,   // ← caller-controlled
    bytes calldata extensionData

    ...
    if (bidPriceX64 >= askPriceX64) revert BidGreaterThanAsk();
    if (bidPriceX64 == 0) revert BidIsZero();
    ...
    _beforeSwap(msg.sender, recipient, zeroForOne, amountSpecified,
                priceLimitX64, packedSlot0Initial,
                bidPriceX64, askPriceX64, extensionData);   // ← arbitrary prices
    ...
    _afterSwap(..., bidPriceX64, askPriceX64, ...);         // ← arbitrary prices
    ...
    _nonReentrantAfter();
    revert SimulateSwap(amount0Delta, amount1Delta);
``` [3](#0-2) 

**Extension hook dispatch — external calls whose state is not rolled back:**

`_callExtensionsInOrder` issues a regular external `call` to each extension contract:

```solidity
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    ...
    CallExtension.callExtension(extension, data);
    ...
}
``` [4](#0-3) 

Because extensions are independent contracts, any storage they write during `beforeSwap` / `afterSwap` is committed to the blockchain immediately. The subsequent `revert SimulateSwap(...)` in the pool only rolls back the pool's own storage — it cannot undo writes made to external extension contracts.

**`simulateSwapAndRevert` also lacks `whenNotPaused`:**

The real `swap` carries `whenNotPaused`; `simulateSwapAndRevert` does not. Extension hooks can therefore be triggered with arbitrary prices even while the pool is administratively paused. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Oracle guard extensions such as `PriceVelocityGuardExtension` and `OracleValueStopLossExtension` record prior price observations in their own storage during hook calls so they can compare them against the current price on the next real swap. An attacker who poisons those observations via `simulateSwapAndRevert` with a manipulated `bidPriceX64`/`askPriceX64` can:

1. Make a velocity guard believe the price moved only a tiny amount (by setting the "last recorded price" close to the real current price), allowing a subsequent real swap at a price that would otherwise be blocked.
2. Make a stop-loss guard believe the pool is within safe bounds when it is not, allowing a real swap to execute at a price that should trigger a halt.

Either outcome allows a real `swap` to execute at a bad price, causing direct loss of LP principal through adverse swap settlement. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

- `simulateSwapAndRevert` is a public, permissionless function — no role or privilege is required.
- The only validation on the supplied prices is `bid < ask` and `bid != 0`, both trivially satisfiable with any two small positive integers.
- The attack requires two transactions (one `simulateSwapAndRevert` to poison, one `swap` to exploit) and no special setup.
- Any pool that has a `PriceVelocityGuardExtension` or `OracleValueStopLossExtension` configured is affected.

---

### Recommendation

1. **Do not forward caller-supplied prices to extension hooks in `simulateSwapAndRevert`.** Either fetch the oracle price inside `simulateSwapAndRevert` (as `swap` does) and use that, or omit the hook calls entirely for simulation paths.
2. **Add `whenNotPaused` to `simulateSwapAndRevert`** so extension hooks cannot be triggered while the pool is paused.
3. **In oracle guard extensions**, fetch the oracle price independently inside the hook rather than trusting the `bidPriceX64`/`askPriceX64` parameters passed by the pool, as a defense-in-depth measure.

---

### Proof of Concept

```
// Setup: pool has PriceVelocityGuardExtension configured.
// Oracle real price: bid=1000, ask=1001.
// Velocity guard threshold: revert if price moved > 5% since last observation.

// Step 1 — Attacker poisons the guard's last-observed price.
pool.simulateSwapAndRevert(
    attacker,
    true,           // zeroForOne
    1e18,           // amountSpecified
    0,              // priceLimitX64
    950,            // bidPriceX64  ← far from oracle, but passes bid<ask check
    951,            // askPriceX64
    ""
);
// Pool reverts → pool state unchanged.
// PriceVelocityGuardExtension.afterSwap() ran and stored lastPrice = 950.

// Step 2 — Attacker executes a real swap.
// Guard computes velocity: |1000 - 950| / 950 ≈ 5.26%.
// If threshold is 10%, guard passes. Real swap executes at oracle price 1000,
// but the guard was calibrated against a poisoned baseline of 950,
// allowing a future swap at 1050 (10.5% above 950) to also pass —
// a price that is actually 5% above the real oracle, harming LPs.
pool.swap(attacker, true, 1e18, 0, callbackData, "");

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
