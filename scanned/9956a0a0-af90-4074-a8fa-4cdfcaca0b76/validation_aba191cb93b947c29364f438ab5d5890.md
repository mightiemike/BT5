### Title
SwapAllowlistExtension Gates the Router Address Instead of the Economic Actor, Allowing Any User to Bypass a Curated Pool's Swap Allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the originating user. If the pool admin allowlists the router (the natural configuration for enabling standard periphery access), every user — including those not individually allowlisted — can bypass the curated pool's swap gate by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    zeroForOne,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly without forwarding the originating user's address:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

When this call reaches the pool, `msg.sender` is the **router address**. The pool passes the router address as `sender` to the extension. The extension therefore checks `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`.

**Bypass path**: A pool admin who wants to allow users to use the standard periphery router must allowlist the router address. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every swap routed through it, regardless of who the originating user is. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle(pool, ...)` and the extension will approve the swap because it sees the router — not the user — as the swapper.

The same issue applies to the multi-hop `exactInput` path (line 104) and both exact-output paths (lines 136, 165, 220).

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` is intended to restrict trading to a specific set of addresses (e.g., KYC'd users, whitelisted market makers, or protocol-controlled addresses). The bypass completely defeats this invariant: any address can trade on the pool by going through the router, gaining access to oracle-priced liquidity that the pool admin explicitly intended to restrict. This constitutes a broken core pool functionality and a direct policy bypass on curated pools, with potential for unauthorized extraction of LP assets at oracle-fair prices.

---

### Likelihood Explanation

The trigger is a standard, expected configuration: a pool admin who deploys a `SwapAllowlistExtension`-gated pool and also wants users to use the official `MetricOmmSimpleRouter` periphery must allowlist the router. This is not a misconfiguration — it is the only way to simultaneously use the allowlist extension and the standard router. The bypass is therefore reachable on any production pool that uses both components together, which is the intended deployment pattern described in the periphery documentation.

---

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the **economic actor** (the originating user), not the immediate caller of `pool.swap()`. Two complementary fixes should be applied:

1. **In `MetricOmmSimpleRouter`**: pass the originating user's address as part of `extensionData` or as a dedicated field so extensions can recover it. Alternatively, the pool interface could be extended with an explicit `originator` parameter.

2. **In `SwapAllowlistExtension.beforeSwap`**: gate on the originating user identity recovered from `extensionData` rather than the raw `sender` argument, when the immediate caller is a known router.

A simpler short-term fix is to document that allowlisting the router is equivalent to `allowAllSwappers = true`, and require pool admins to only allowlist individual EOAs/contracts — never the router — if they want meaningful curation. However, this breaks the usability of the standard periphery for curated pools and is not a real fix.
