### Title
SwapAllowlistExtension Gates on Router Address Instead of Actual User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension.beforeSwap` hook checks the `sender` argument it receives from the pool. In `MetricOmmPool.swap()`, `sender` is always `msg.sender` of the pool call — which is the `MetricOmmSimpleRouter` address when a user routes through the periphery. Because the allowlist is keyed on this immediate caller rather than on the actual end-user, any pool admin who allowlists the router (the only way to permit any router-mediated swap) simultaneously opens the pool to every user on the network, defeating the per-user access control the extension is designed to enforce.

---

### Finding Description

**Root cause — wrong identity checked by the allowlist hook**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and uses it as the identity to look up in the allowlist (confirmed by the unit test that passes `swapper` as the first positional argument and expects a revert when that address is not allowlisted): [3](#0-2) 

**Attack path through the router**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly: [4](#0-3) 

When this call reaches the pool, `msg.sender` is the router contract. The `sender` forwarded to `SwapAllowlistExtension.beforeSwap` is therefore the router address, not the end-user who called `exactInputSingle`.

**The forced all-or-nothing choice**

A pool admin who wants to allow *any* router-mediated swap must allowlist the router address. Once the router is allowlisted, the extension passes for every user who routes through it — the per-user restriction is completely bypassed. There is no mechanism to allowlist the router for specific users only, because the extension never sees the actual user's address in the router flow.

The same structural problem applies to multi-hop `exactInput` and `exactOutput` paths: [5](#0-4) 

The research pivot file explicitly identifies this as the critical validation gap: [6](#0-5) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` and a curated allowlist (e.g., a private institutional pool, a KYC-gated pool, or a pool restricted to a single market-maker) can be freely accessed by any unprivileged user simply by calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The attacker receives real token output from the pool's liquidity at oracle-anchored prices. The pool's LP providers suffer direct loss of principal because swaps they intended to restrict to trusted counterparties are executed by arbitrary actors.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point documented in the protocol. Any pool that deploys `SwapAllowlistExtension` and also wants to support router-mediated swaps (the normal production configuration) must allowlist the router, triggering the bypass. The attacker needs no special privileges, no flash loan, and no oracle manipulation — a single standard router call suffices.

---

### Recommendation

The `SwapAllowlistExtension` should gate on the economically relevant actor, not the immediate pool caller. Two sound approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted router convention.
2. **Check `recipient` instead of (or in addition to) `sender`**: For swap allowlists the recipient is the party receiving value; gating on recipient prevents value extraction by unauthorized parties regardless of routing path.
3. **Require direct pool interaction for allowlisted pools**: Document and enforce that pools using `SwapAllowlistExtension` must not allowlist the router; instead, allowlisted users must call the pool directly. This is operationally fragile but avoids contract changes.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured (allowAll = false).
  - Admin allowlists only address(0xALICE) as an approved swapper.
  - Admin also allowlists address(router) so that Alice can use the router.

Attack:
  1. Bob (address(0xBOB), not on the allowlist) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob})
  2. Router calls pool.swap(bob, ...) — msg.sender = router.
  3. Pool calls _beforeSwap(router, bob, ...).
  4. SwapAllowlistExtension.beforeSwap(sender=router, ...) — router IS allowlisted → passes.
  5. Swap executes; Bob receives token output from the restricted pool.

Result: Bob, an unprivileged address, extracts value from a pool that was intended to be
        restricted to Alice only.
```
