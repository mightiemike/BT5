Based on my investigation, here is the finding:

---

### Title
Payable Liquidity-Add and Swap Entry-Points Never Consume `msg.value` for Non-WETH Tokens, Permanently Locking Caller ETH — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

Every user-facing entry-point in `MetricOmmPoolLiquidityAdder` and `MetricOmmSimpleRouter` is declared `payable`, yet the internal `pay()` helper in `PeripheryPayments` only consumes native ETH when the token being paid is exactly `WETH`. When either pool token (or `tokenIn` for swaps) is not WETH, `msg.value` is silently ignored. Any ETH attached to such a call is stranded on the contract and is claimable by any subsequent caller of the permissionless `refundETH()` function.

---

### Finding Description

`MetricOmmPoolLiquidityAdder` exposes four `payable` entry-points: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`MetricOmmSimpleRouter` exposes four `payable` swap entry-points: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

None of these functions reference `msg.value` directly. Token settlement is delegated entirely to `PeripheryPayments.pay()`: [9](#0-8) 

The critical branch is:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // uses ALL contract ETH, not msg.value
    if (nativeBalance >= value) { ... }
    ...
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value);  // ETH completely ignored
}
```

When `token != WETH`, the `else` branch executes a plain ERC-20 pull and the native ETH attached to the call is never touched. The `receive()` hook reverts for any sender other than WETH: [10](#0-9) 

So ETH can only enter the contract through these `payable` calls, and once it is there it is not automatically returned.

The recovery function is: [11](#0-10) 

`refundETH()` is permissionless — it sends the **entire** contract ETH balance to `msg.sender`. A victim who sends ETH in transaction T1 and calls `refundETH()` in a separate transaction T2 can be front-run: an attacker calls `refundETH()` first and receives the victim's ETH.

---

### Impact Explanation

**Direct loss of user principal.** Any user who calls `addLiquidityExactShares`, `addLiquidityWeighted`, `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` with `msg.value > 0` on a pool whose relevant token is not WETH loses that ETH. The loss is permanent unless the user calls `refundETH()` in the same multicall batch. If they do not, any third party can drain the stranded ETH via `refundETH()`.

---

### Likelihood Explanation

- All six entry-points are `payable` with no NatSpec warning against sending ETH for non-WETH pools.
- The `multicall` wrapper is also `payable`, encouraging users to bundle ETH with batched calls.
- A user interacting with a non-WETH/non-WETH pool (e.g., USDC/DAI) who mistakenly attaches ETH loses it immediately.
- MEV bots routinely monitor for stranded ETH on router contracts and call `refundETH()` to extract it.

---

### Recommendation

1. **For non-WETH operations**: add a guard at the top of each entry-point:
   ```solidity
   if (msg.value > 0 && tokenIn != WETH) revert NativeNotAccepted();
   ```
   For liquidity adders, check both `token0` and `token1` of the pool before accepting ETH.

2. **Alternatively**: remove the `payable` modifier from functions that have no WETH-wrapping path (i.e., when the pool contains no WETH leg).

3. **Minimum mitigation**: append an automatic `refundETH()` at the end of every entry-point so leftover ETH is always returned to `msg.sender` in the same transaction, eliminating the front-run window.

---

### Proof of Concept

```
1. Deploy a MetricOmm pool with token0 = USDC, token1 = DAI (neither is WETH).
2. Victim calls addLiquidityExactShares{value: 1 ether}(pool, ...) directly (not via multicall).
3. The callback settles via safeTransferFrom for both USDC and DAI; msg.value is never touched.
4. 1 ETH now sits on MetricOmmPoolLiquidityAdder.
5. Attacker calls refundETH() → receives 1 ETH.
6. Victim's 1 ETH is permanently lost.
```

The same sequence applies to `exactInputSingle{value: 1 ether}(...)` on any non-WETH pool via `MetricOmmSimpleRouter`.

    _checkDeadline(params.deadline);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
