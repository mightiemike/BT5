### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual user when swaps route through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to gate swaps on a per-user basis. However, the hook receives `sender` as the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is the direct caller, so the extension checks the router's address against the allowlist instead of the actual user's address. This creates a wrong-actor binding: either allowlisted users cannot use the standard router at all, or — if the admin allowlists the router to enable router-based swaps — every user on the internet can bypass the per-user allowlist.

---

### Finding Description

**Actor binding in the pool:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` then encodes that value as the first positional argument to the extension call: [2](#0-1) 

**What the extension checks:**

`SwapAllowlistExtension.beforeSwap` reads the first positional argument as `sender` and checks it against the per-pool allowlist: [3](#0-2) 

**What the router actually sends:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. The router is therefore `msg.sender` inside the pool, so `sender` delivered to the extension is the router's address — not the user who called the router: [4](#0-3) 

The same mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

---

### Impact Explanation

Two concrete outcomes, both fund-impacting:

**Scenario A — Allowlist bypass (high impact):**
A pool admin allowlists the router address so that router-based swaps are possible (a natural operational step). Because the extension checks `allowedSwapper[pool][router]`, every user who calls the router passes the guard regardless of whether they are individually allowlisted. The curated pool's per-user access control is completely nullified for the router path.

**Scenario B — Allowlisted users locked out of the router (medium impact):**
If the admin does not allowlist the router, every call through `MetricOmmSimpleRouter` reverts with `NotAllowedToSwap`, even for users who are individually allowlisted. The standard periphery swap path is permanently broken for the pool, forcing users to implement raw pool callbacks themselves or not trade at all.

Both outcomes break the invariant stated in the Smart Audit Pivots: *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."*

---

### Likelihood Explanation

- The `SwapAllowlistExtension` is a production periphery contract explicitly designed for curated pools.
- `MetricOmmSimpleRouter` is the canonical swap entrypoint for end users.
- Any pool operator who deploys both together and allowlists the router (Scenario A) or any allowlisted user who tries to use the router (Scenario B) triggers the bug with no special privileges or malicious setup.
- No non-standard tokens, no admin collusion, and no off-chain components are required.

---

### Recommendation

The extension must identify the true economic actor. Two options:

1. **Pass the user through the router:** Have `MetricOmmSimpleRouter` encode the original `msg.sender` in `extensionData` and have the extension decode it. This requires a coordinated interface change.

2. **Check `recipient` instead of `sender`:** If the semantic intent is "who receives the output," use the second positional argument (`recipient`). However, this still does not identify the payer.

3. **Preferred — router-aware actor forwarding:** Add a standardized field (e.g., the first word of `extensionData`) that the router always populates with `msg.sender`, and have the extension read that field when `sender` is a known router. The extension should validate that the forwarded address is consistent with the callback payer.

### Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended allowlisted user
  admin calls setAllowedToSwap(pool, router, true)  // admin enables router-based swaps

Attack (Scenario A — bypass):
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  Execution trace:
    router → pool.swap(recipient=bob, ...)          // msg.sender = router
    pool   → _beforeSwap(sender=router, ...)
    pool   → extension.beforeSwap(sender=router, recipient=bob, ...)
    extension checks: allowedSwapper[pool][router]  // TRUE — router is allowlisted
    → swap proceeds; bob receives tokens despite never being allowlisted

Broken functionality (Scenario B):
  alice (allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: alice, ...})

  Execution trace:
    router → pool.swap(recipient=alice, ...)        // msg.sender = router
    pool   → extension.beforeSwap(sender=router, ...)
    extension checks: allowedSwapper[pool][router]  // FALSE — router not allowlisted
    → revert NotAllowedToSwap; alice cannot use the router
