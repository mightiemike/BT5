### Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing any user to bypass the per-user swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which is the direct `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` equals the **router address**, not the end user. A pool admin who allowlists the router (required for any router-based swap to work) inadvertently opens the gate to **all** users, completely defeating the per-user allowlist.

---

### Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding its own `msg.sender` as the `sender` argument to every registered extension. [1](#0-0) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap(params.recipient, ...)` directly, so the pool sees `msg.sender = router`. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

`sender` here is the **router**, not the end user. The allowlist therefore operates on the router address, not on individual swappers.

This creates a binary, all-or-nothing outcome for router users:

| Router allowlisted? | Effect |
|---|---|
| Yes | Every user can swap through the router — allowlist fully bypassed |
| No | Every allowlisted user is also blocked when using the router — DoS |

**Contrast with `DepositAllowlistExtension`**, which correctly ignores the first parameter (the caller of `addLiquidity`) and checks the `owner` argument instead — the actual LP owner:

```solidity
function beforeAddLiquidity(address, address owner, uint80, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

The swap extension applies the opposite (wrong) logic: it checks the intermediary caller rather than the economic actor.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) is silently rendered open to all users the moment the pool admin allowlists the router. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle()` and swap against the restricted pool. This is an **admin-boundary break**: the allowlist cap set by the pool admin is bypassed by an unprivileged path (the public router).

---

### Likelihood Explanation

The router is the primary user-facing entry point for swaps. A pool admin who wants router-based swaps to work for any allowlisted user **must** allowlist the router address. This is the natural, expected configuration. The bypass is therefore triggered by the standard operational setup, not by an exotic edge case.

---

### Recommendation

Mirror the `DepositAllowlistExtension` pattern: check the economic actor, not the intermediary caller. For swaps the economic actor is the address that initiated the swap. Two options:

1. **Pass end-user via `extensionData`**: Have the router ABI-encode `msg.sender` (the end user) into `extensionData`, and have `SwapAllowlistExtension` decode and check it. The extension must also verify `msg.sender` (the pool) is a known pool to prevent spoofing.

2. **Check `recipient` instead of `sender`**: If the pool's design guarantees `recipient` is the end user, check `recipient` (second parameter of `beforeSwap`). This is the direct analog of how `DepositAllowlistExtension` checks `owner`.

The simplest safe fix consistent with the existing deposit pattern:

```solidity
// beforeSwap signature: (address sender, address recipient, ...)
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` registered as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, routerAddress, true)` — required for any router swap to work.
3. Unprivileged address `attacker` (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. `attacker` successfully swaps against the restricted pool despite never being allowlisted. [3](#0-2) [5](#0-4)
