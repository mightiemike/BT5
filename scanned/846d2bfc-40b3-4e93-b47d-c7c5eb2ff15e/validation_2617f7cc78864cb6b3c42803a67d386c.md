### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual end-user, making the swap allowlist guard permanently broken for router-routed swaps ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the end-user. The allowlist therefore checks whether the router is allowlisted, not whether the actual user is allowlisted. This makes the guard permanently misapplied for all router-routed swaps: allowlisted users cannot swap through the router, and if the admin allowlists the router to compensate, every user bypasses the per-user restriction.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router is the entity that calls `pool.swap()`: [4](#0-3) 

The actual end-user (`msg.sender` of `exactInputSingle`) is stored only in transient storage as the payer and is never forwarded to the extension. The extension therefore evaluates `allowedSwapper[pool][router_address]` ŌĆö a check that is structurally wrong for per-user access control.

Note the asymmetry with `DepositAllowlistExtension`, which correctly checks the `owner` parameter (the position owner) rather than `sender` (the caller), making it immune to this class of bypass: [5](#0-4) 

---

### Impact Explanation

Two concrete broken states arise:

**State A ŌĆö Allowlisted users locked out of the router.** A pool admin allowlists specific user addresses (e.g., KYC'd counterparties). Those users call `exactInputSingle` through the router. The extension checks `allowedSwapper[pool][router]`, which is `false`. Every swap reverts with `NotAllowedToSwap`. The router ŌĆö the primary user-facing swap interface ŌĆö is permanently unusable for any pool that has a per-user `SwapAllowlistExtension` configured.

**State B ŌĆö Full allowlist bypass.** To unblock router users, the admin allowlists the router address: `setAllowedToSwap(pool, router, true)`. Now `allowedSwapper[pool][router] = true`, and the guard passes for every caller of `exactInputSingle`, regardless of who they are. Any unprivileged user can swap against a pool that was intended to be restricted to a specific set of counterparties.

State B directly satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact criterion. State A satisfies "Broken core pool functionality causing unusable swap flows."

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` with a `beforeSwap` order and expects users to interact via `MetricOmmSimpleRouter` hits this immediately. The extension and router are both first-party periphery contracts shipped together. The admin has no in-protocol mechanism to pass the real user identity through the router to the extension, so there is no correct configuration that avoids both broken states simultaneously.

---

### Recommendation

The `SwapAllowlistExtension` must receive the real initiating user, not the intermediary. Two options:

1. **Pass the real user via `extensionData`.** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes and checks that address. This requires a coordinated change to the router and the extension.

2. **Check `recipient` as a proxy.** If the protocol convention is that `recipient == initiating user`, the extension can check `recipient` instead of `sender`. This is weaker (recipient can be set to any address) but requires no router change.

The deposit allowlist's pattern ŌĆö checking the semantically meaningful actor (`owner`) rather than the structural caller (`sender`) ŌĆö is the correct model to follow.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as extension1, beforeSwap order = 1.
2. Admin calls swapExtension.setAllowedToSwap(pool, alice, true).
   ŌåÆ allowedSwapper[pool][alice] = true
3. Alice calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient: alice, ...}).
   ŌåÆ router calls pool.swap(alice, ...) with msg.sender = router
   ŌåÆ pool calls _beforeSwap(sender = router, ...)
   ŌåÆ SwapAllowlistExtension checks allowedSwapper[pool][router] ŌåÆ false
   ŌåÆ revert NotAllowedToSwap   ŌåÉ Alice is allowlisted but cannot swap via router.

4. Admin calls swapExtension.setAllowedToSwap(pool, router, true) to fix Alice.
   ŌåÆ allowedSwapper[pool][router] = true
5. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool, ...}).
   ŌåÆ router calls pool.swap(...) with msg.sender = router
   ŌåÆ SwapAllowlistExtension checks allowedSwapper[pool][router] ŌåÆ true
   ŌåÆ swap succeeds   ŌåÉ Bob bypasses the allowlist entirely.
```
