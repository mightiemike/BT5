### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid Price, Weakening Stop-Loss Protection for `zeroForOne` Swaps — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask, while every other component in the protocol — `SwapMath.midAndSpreadFeeX64FromBidAsk`, `PriceVelocityGuardExtension`, `MetricOmmPool.getSellAndBuyPrices`, and `MetricOmmPoolDataProvider` — uses the geometric mean (`sqrt(bid * ask)`). By the AM-GM inequality, arithmetic mean ≥ geometric mean, so the stop-loss always overestimates the mid price. This causes the `metricToken0` watermark to be initialized and maintained below its correct value, making the stop-loss guard for the `zeroForOne == true` direction (token1 outflow) systematically less protective than configured. The discrepancy grows with the bid/ask spread, which is widest precisely during the volatile conditions the guard is meant to defend against.

---

### Finding Description

In `_afterSwapOracleStopLoss`, line 218:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

Every other mid-price computation in the codebase uses the geometric mean:

```solidity
// SwapMath.midAndSpreadFeeX64FromBidAsk (used by pool swap, PriceVelocityGuardExtension, getSellAndBuyPrices)
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`PriceVelocityGuardExtension.beforeSwap` calls `SwapMath.midAndSpreadFeeX64FromBidAsk` with the same bid/ask pair: [3](#0-2) 

`MetricOmmPoolDataProvider` also uses `Math.sqrt(uint256(env.oracleBidX64) * uint256(env.oracleAskX64))`: [4](#0-3) 

The per-share value metrics computed from this mid price are:

```
metricToken0 = t0·SCALE/shares + (t1·Q64 / mid)·SCALE/shares
metricToken1 = (t0·mid / Q64)·SCALE/shares + t1·SCALE/shares
``` [5](#0-4) 

Because arithmetic mean > geometric mean, the stop-loss extension always uses a mid price that is too high. This has two consequences:

- **`metricToken0` is underestimated** (the `t1/mid` term shrinks). The high-watermark for token0 is ratcheted up to a value that is lower than the true value per share. The drawdown threshold `hwm0 * floorMultiplier / E6` is therefore also lower than intended, so the guard for `zeroForOne == true` (token1 outflow) fails to trigger until a larger real drawdown has occurred.
- **`metricToken1` is overestimated** (the `t0·mid` term grows). The watermark for token1 is set too high, causing the guard for `zeroForOne == false` to trigger prematurely (false positives that lock out legitimate swaps).

The breach check in `_applyWatermark`:

```solidity
breached = metric < (hwm * floorMultiplier) / E6;
``` [6](#0-5) 

compares the current (also arithmetic-mean-derived) metric against the (also arithmetic-mean-derived) watermark. The relative percentage check is therefore internally consistent, but the absolute value being tracked diverges from the protocol's actual pricing basis. When the spread widens — e.g., during oracle stress — the arithmetic mean diverges most from the geometric mean, and the effective drawdown budget silently expands beyond the configured `drawdownE6`.

---

### Impact Explanation

**Direct loss of LP principal.** The stop-loss extension is the primary on-chain mechanism preventing value leakage from a bin beyond the configured drawdown. When the bid/ask spread is `s` (as a fraction), the arithmetic mean exceeds the geometric mean by approximately `s²/8`. For a 10% spread, the mid price is overestimated by ~1.25%; for a 20% spread, by ~5%. This directly translates to the `metricToken0` watermark being set ~1.25–5% below its correct value, meaning the effective drawdown budget is silently expanded by that same margin. A swap sequence that should have been blocked by the configured drawdown threshold is allowed to proceed, draining token1 from the pool beyond what LPs consented to.

---

### Likelihood Explanation

This affects every pool that deploys `OracleValueStopLossExtension`. The trigger requires no privileged action: any ordinary swap through the pool causes `afterSwap` to be called, which invokes `_afterSwapOracleStopLoss` with the live bid/ask prices. The discrepancy is always present and grows monotonically with the spread. Pools using oracle feeds with wider confidence intervals (e.g., Pyth Lazer during high volatility) are most exposed, and those are exactly the conditions under which the stop-loss is most needed.

---

### Recommendation

Replace the arithmetic mean on line 218 with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This aligns the stop-loss's value assessment with the same mid-price basis used by the pool's swap execution, `PriceVelocityGuardExtension`, and all view/lens contracts.

### Proof of Concept

**Setup:** Pool with `drawdownE6 = 100_000` (10% drawdown allowed). Oracle reports `bid = 1.0 × 2^64`, `ask = 1.44 × 2^64` (20% spread). Bin holds `t0 = 1000`, `t1 = 1000`, `shares = 1000`.

**Arithmetic mean (current code):**
```
mid_arith = (1.0 + 1.44) / 2 = 1.22
metricToken0 = 1 + 1000/(1.22 × 1000) = 1 + 0.8197 = 1.8197 × SCALE
hwm0 = 1.8197 × SCALE
threshold = hwm0 × 0.9 = 1.6377 × SCALE
```

**Geometric mean (correct):**
```
mid_geo = sqrt(1.0 × 1.44) = 1.2
metricToken0 = 1 + 1000/(1.2 × 1000) = 1 + 0.8333 = 1.8333 × SCALE
hwm0 = 1.8333 × SCALE
threshold = hwm0 × 0.9 = 1.65 × SCALE
```

**Attack:** Attacker drains token1 (zeroForOne = true) until `metricToken0` falls to `1.638 × SCALE`.

- Under the **correct** geometric-mean guard: `1.638 < 1.65` → **stop-loss triggers, swap reverts**.
- Under the **current** arithmetic-mean guard: `1.638 > 1.6377` → **stop-loss does not trigger, swap succeeds**.

The attacker extracts token1 beyond the 10% drawdown limit that LPs configured, with the excess proportional to the spread-induced bias (~1.4% additional drain in this example, scaling to larger amounts for larger pools or wider spreads).
