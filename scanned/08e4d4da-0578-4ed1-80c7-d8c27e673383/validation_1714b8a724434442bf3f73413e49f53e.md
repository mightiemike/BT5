### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Trader, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument it receives from the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks whether the **router** is allowlisted — not the actual end-user. Any user who calls the router on an allowlisted pool bypasses the per-user curation gate entirely.

---

### Finding Description

`MetricOmmPool.swap` fires `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then enforces the allowlist against that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInput` (or any `exact*` entry point), the router is the direct caller of `pool.swap()`. The pool therefore passes the **router's address** as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

Two failure modes follow:

1. **Allowlist bypass (primary impact):** The pool admin must allowlist the router for any router-mediated swap to succeed. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every user who calls through it — the per-user gate is completely defeated. Any non-allowlisted user can swap on the curated pool simply by routing through `MetricOmmSimpleRouter`.

2. **Allowlisted users locked out of the router:** If the admin does *not* allowlist the router, every allowlisted user who tries to use the router is blocked, because the extension sees the router address and finds it absent from the allowlist.

The `allowedSwapper` mapping is keyed by `(pool, swapper)`: [4](#0-3) 

There is no mechanism in the extension or the pool to recover the original `tx.origin` or a forwarded user identity from the router, so the wrong-actor binding is structural.

---

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., KYC'd addresses, whitelisted market makers) loses that protection entirely for any user who routes through the public `MetricOmmSimpleRouter`. The attacker receives the same execution as an allowlisted user — full access to pool liquidity at oracle-derived bid/ask prices — with no principal at risk beyond normal swap fees. This constitutes a direct policy bypass with fund-impacting consequences: the pool's LP providers are exposed to trades from actors the pool admin explicitly intended to exclude.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public entrypoint documented for end-users. Any non-allowlisted user who discovers the pool is curated can trivially route through the router. No special privileges, flash loans, or multi-transaction setup are required. The trigger is a single public `exactInput` or `exactOutput` call.

---

### Recommendation

The extension must check the **economically relevant actor**, not the intermediary. Two sound approaches:

1. **Check `tx.origin` in the extension** (acceptable only if the router is the sole trusted intermediary and re-entrancy is already blocked by the pool's nonReentrant guard).

2. **Have the router forward the originating user address inside `extensionData`**, and have the extension decode and verify it — combined with a check that `msg.sender` (the pool) is a known pool so the payload cannot be spoofed by a direct caller.

3. **Require direct pool interaction** for allowlisted pools (document that `SwapAllowlistExtension` is incompatible with router-mediated flows and enforce this at the factory or extension configuration level).

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the only allowed trader
  allowedSwapper[pool][router] = false // router not explicitly listed

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInput(pool, zeroForOne=true, ...)

  Router calls:
    pool.swap(recipient=bob, ...)

  Pool calls _beforeSwap(sender=router, ...)

  Extension evaluates:
    allowAllSwappers[pool]          → false
    allowedSwapper[pool][router]    → false
    → revert NotAllowedToSwap()     ← bob is blocked here

  Admin, wanting router to work for alice, sets:
    allowedSwapper[pool][router] = true

  Now bob calls again through the router:
    allowedSwapper[pool][router]    → true
    → swap proceeds for bob         ← allowlist bypassed
```

The moment the admin enables the router for any allowlisted user, the gate opens for all users. The allowlist provides zero protection against router-mediated swaps.
