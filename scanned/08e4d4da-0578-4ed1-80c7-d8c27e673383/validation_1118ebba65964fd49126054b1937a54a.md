### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol` / `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. `SwapAllowlistExtension.beforeSwap` keys its allowlist lookup on `(pool, sender)`, so it evaluates the router's address — not the actual swapper's address. A pool admin who allowlists the router to support router-mediated swaps for their curated users inadvertently opens the gate to every unprivileged user, because any caller can reach the pool through the same router address.

---

### Finding Description

**Root cause — wrong actor bound in the hook dispatcher:**

`MetricOmmPool.swap()` captures `msg.sender` once and passes it as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
```

`ExtensionCalling._beforeSwap` encodes that value and dispatches it to every extension in `BEFORE_SWAP_ORDER`:

```solidity
// ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified,
         priceLimitX64, packedSlot0Initial, bidPriceX64, askPriceX64, extensionData))
);
```

`SwapAllowlistExtension.beforeSwap` performs an `allowAll / allowedSwapper` lookup keyed on `(pool, sender)`. When the user enters through `MetricOmmSimpleRouter.exactInputSingle()`:

```solidity
// MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(   // ← router is msg.sender of pool.swap()
    params.recipient, params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64, "", params.extensionData
);
```

The pool receives `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**The ordering / binding mismatch (analog to Alchemix):**

In Alchemix, `_distribute()` ran before `minter.updatePeriod()`, so the guard saw stale (zero) reward state and the real value arrived too late to be distributed. Here, the guard sees the wrong actor (router) instead of the real economic principal (end user). The guard's decision is made against a value that was never the intended gating criterion, so the protection is structurally misapplied on every router-mediated swap.

**Bypass path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension`, allowlisting a set of KYC'd / trusted users.
2. To let those users trade via the standard router, the admin also adds `allowedSwapper[pool][router] = true` (or sets `allowAll[pool] = true` for the router entry).
3. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`. The extension sees `sender = router`, finds it allowlisted, and returns success.
4. The disallowed user's swap executes against the curated pool's LP positions.

---

### Impact Explanation

A curated pool whose LP positions are sized and priced for a restricted counterparty set is now reachable by arbitrary swappers. Depending on pool configuration this can cause:

- **Direct LP principal loss**: adversarial swappers can drain mispriced or thin bins that were only safe under the assumption of trusted counterparties.
- **Protocol fee leakage**: fee tiers calibrated for a closed set of participants are exposed to the open market.
- **Curation failure**: the entire purpose of the allowlist extension — restricting who may trade — is nullified for any pool that must also support router-mediated access.

Severity: **High** — direct loss of LP assets and broken core pool access-control functionality.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the canonical, documented swap entrypoint for end users. Any pool admin who wants their allowlisted users to trade conveniently will allowlist the router.
- No special privilege or malicious setup is required from the attacker; any EOA can call `exactInputSingle`.
- The bypass is deterministic and repeatable on every block once the router is allowlisted.

---

### Recommendation

The extension must recover the original end-user identity rather than trusting the immediate `msg.sender` of `pool.swap()`. Two viable approaches:

1. **Pass the original payer through `extensionData`**: the router encodes `msg.sender` (the real user) into `extensionData`; the extension decodes and verifies it, trusting only calls from a known router address.
2. **Check `sender` against a router registry and fall back to a forwarded identity field**: if `sender` is a known trusted router, the extension reads the real swapper from a signed or transient-storage-backed field set by the router before calling the pool.
3. **Alternatively, remove router support from allowlisted pools** and require direct `pool.swap()` calls only — but this breaks UX for legitimate allowlisted users.

```text
  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
```
