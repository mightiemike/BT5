### Title
`SwapAllowlistExtension` checks router address instead of actual swapper, allowing allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks whether the **router** is allowlisted rather than the actual user. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users inadvertently opens the gate to every user on-chain.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol line 231
_beforeSwap(
    msg.sender,   // ÔćÉ whoever called pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that argument against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly without forwarding the original `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol lines 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

When this call reaches the pool, `msg.sender` = router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

---

### Impact Explanation

Two concrete fund-impacting outcomes:

1. **Allowlist bypass**: A pool admin who allowlists the router address (the natural fix to let legitimate users trade through the supported periphery) simultaneously grants every unpermissioned address the ability to swap, because the extension cannot distinguish which end-user initiated the router call. Unauthorized users drain or manipulate a pool that was designed to be curated.

2. **Broken core functionality for legitimate users**: If the admin does *not* allowlist the router, every allowlisted user who routes through `MetricOmmSimpleRouter` is blocked, making the only supported periphery path unusable for the pool's intended participants.

Both outcomes are direct consequences of the wrong-actor binding: the economic actor (the user) is never the address the guard evaluates.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary public swap entrypoint documented in the periphery.
- Any user can call it permissionlessly; no special role or setup is required.
- The pool admin has no on-chain mechanism to distinguish router-mediated calls from direct calls inside the extension, so the misconfiguration is structurally forced.
- The bypass requires only a standard router swap call ÔÇö no flash loans, callbacks, or multi-step sequences.

---

### Recommendation

Pass the original user address through the swap path so the extension can check the correct actor. Two complementary fixes:

1. **In the router**: store `msg.sender` in transient storage at entry (alongside the existing callback context) and expose it via a `swapInitiator()` getter that the pool can read.

2. **In `MetricOmmPool.swap`**: read the initiator from the router's transient slot when `msg.sender` is a known factory-registered router, and pass that address as `sender` to `_beforeSwap`. For direct calls, `msg.sender` is already correct.

Alternatively, redesign `SwapAllowlistExtension` to gate on `recipient` (which the user controls and which is forwarded correctly) or require the router to pass the user address explicitly in `extensionData` and verify it with a signature.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for legitimate users (or is forced to do so after discovering direct-user allowlisting blocks router paths).
3. Attacker (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool.
4. Inside `pool.swap`, `msg.sender` = router Ôćĺ extension evaluates `allowedSwapper[pool][router]` = `true` Ôćĺ swap proceeds.
5. Attacker receives output tokens from a pool that was supposed to be restricted to specific participants.
