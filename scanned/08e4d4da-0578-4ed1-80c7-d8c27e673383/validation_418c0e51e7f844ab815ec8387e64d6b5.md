### Title
`SwapAllowlistExtension` checks the router address as `sender`, not the actual end user â€” allowlist fully bypassed when the router is allowlisted (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual end user. If the pool admin allowlists the router address to enable router-mediated swaps for permitted users, every user â€” including non-allowlisted ones â€” can bypass the allowlist by routing through the router.

---

### Finding Description

**Step 1 â€” Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the `sender` argument forwarded to `_beforeSwap` is always `msg.sender` of the pool call: [1](#0-0) 

**Step 2 â€” Router calls the pool directly; the pool sees the router as `msg.sender`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` with no mechanism to forward the original caller's address: [2](#0-1) 

The same applies to `exactInput` (all hops) and `exactOutput` (all intermediate hops): [3](#0-2) 

**Step 3 â€” The extension checks the router address, not the end user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

**The invariant break:** The pool admin intends to gate individual swapper identities. The extension is keyed on `sender`, which is the router when the user enters through the supported periphery path. There is no way for the extension to recover the actual end user's address â€” the router stores the payer in transient storage for the callback but never passes it to the pool's `swap` call.

This creates two mutually exclusive failure modes:

| Admin configuration | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot swap through the router at all â€” broken functionality |
| Router **allowlisted** | Every user, including non-allowlisted ones, can bypass the allowlist â€” full bypass |

The second scenario is the direct fund-impacting path: a pool admin who wants to support router-mediated swaps for permitted users will naturally allowlist the router, inadvertently opening the gate to all users.

---

### Impact Explanation

A non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` targeting a curated pool. The pool's `_beforeSwap` passes `msg.sender = router` as `sender` to the extension. The extension checks `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes and the swap executes â€” the allowlist is completely bypassed. The user receives pool output tokens they were not permitted to receive, violating the pool's curation policy (KYC gate, sanctions screen, or restricted LP pool).

---

### Likelihood Explanation

The router is the canonical periphery entry point documented and expected to be used by end users. A pool admin who configures a `SwapAllowlistExtension` and also wants to support router-mediated swaps for their permitted users will allowlist the router address â€” this is the natural, expected configuration. The bypass is therefore reachable through normal, documented usage of the protocol.

---

### Recommendation

The `SwapAllowlistExtension` must gate the actual end user, not the intermediary. Two options:

1. **Pass the original caller through the router.** Add an optional `originator` field to the pool's `swap` signature (or a separate periphery-level forwarding mechanism) so the extension can check the real user.

2. **Gate on `recipient` instead of `sender` for router flows.** The `recipient` is the address that receives output tokens and is set to the actual user by the router. However, this changes the semantics of the allowlist (output recipient vs. input payer) and may not match the admin's intent.

3. **Document that the router must never be allowlisted and that allowlisted users must call the pool directly.** This is the least-code fix but breaks the usability of the router for curated pools.

The cleanest fix is option 1: the router should forward `msg.sender` as an `originator` parameter so the extension can check `allowedSwapper[pool][originator]` instead of `allowedSwapper[pool][sender]`.

### Proof of Concept

```

  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][router] = true   // admin enables router-mediated swaps
  allowedSwapper[pool][alice] = true    // alice is a permitted user
  allowedSwapper[pool][bob] = false     // bob is NOT permitted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  â†’ router calls pool.swap(recipient=bob, ...)

  â†’ check passes, swap executes
  â†’ bob receives output tokens from a pool he is not permitted to access

  The allowlist is bypassed. Bob swaps successfully despite not being allowlisted.
```
