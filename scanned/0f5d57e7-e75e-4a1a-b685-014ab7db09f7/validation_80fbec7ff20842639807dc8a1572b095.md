### Title
`SwapAllowlistExtension` Gates on Router Address Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router contract**, not the end user. Allowlisting the router — the only way to permit any router-mediated swap — opens the gate to every user, defeating the per-user access control the extension is designed to enforce.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// msg.sender = pool (caller of the extension)
// sender     = whoever called pool.swap (the router, when routing)
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards whatever `sender` the pool received:

```solidity
function _beforeSwap(address sender, ...) internal {
    _callExtensionsInOrder(
      BEFORE_SWAP_ORDER,
      abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
    );
}
``` [2](#0-1) 

The pool sets `sender = msg.sender` (confirmed by `simulateSwapAndRevert`, which uses the same `_beforeSwap` call): [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` inside the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

Consequently, the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. There are only two outcomes:

| Router allowlisted? | Result |
|---|---|
| **No** | Every allowlisted user is silently broken — they cannot use the router even though they are individually permitted. |
| **Yes** | Every non-allowlisted user can bypass the gate by routing through the router. |

There is no configuration that simultaneously (a) allows allowlisted users to use the router and (b) blocks non-allowlisted users from using the router.

---

### Impact Explanation

A pool deployer who configures `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC'd traders, institutional LPs, whitelist-only launch) loses that guarantee the moment any user routes through `MetricOmmSimpleRouter`. If the router is allowlisted — the only way to support router-mediated swaps — the allowlist is fully bypassed. Any unprivileged user can execute swaps in a pool that was intended to be restricted, accessing liquidity and oracle-anchored prices that the pool admin explicitly did not grant them. This satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact criterion.

---

### Likelihood Explanation

High. `MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who want to support normal UX must allowlist the router, which is the exact configuration that enables the bypass. The flaw is structural: the extension API receives the direct caller of `pool.swap`, not the originating user, so no amount of careful allowlist management can close the gap while the router is in use.

---

### Recommendation

The end-user identity must be threaded through the swap path so the extension can gate on it. Two viable approaches:

1. **`extensionData` forwarding**: The router encodes `msg.sender` (the end user) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. The pool already forwards `extensionData` verbatim to every extension hook.
2. **Dedicated `originator` field**: Add an `originator` parameter to `pool.swap` that the router populates with `msg.sender`. The pool passes it to `_beforeSwap` alongside `sender`. Extensions that need user-level gating check `originator`; extensions that need caller-level gating check `sender`.

Either approach must be paired with a trust model: the extension must only accept the originator claim from a trusted router, otherwise any caller can spoof it.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin allowlists the router:
       swapExtension.setAllowedToSwap(pool, address(router), true)
   (required so that any router-mediated swap can proceed)
3. Pool admin does NOT allowlist attacker:
       allowedSwapper[pool][attacker] == false
4. Attacker calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
5. Router calls pool.swap(...) — msg.sender inside pool = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates:
       allowedSwapper[pool][router] == true  →  passes
8. Attacker's swap executes in the restricted pool.
   The per-user allowlist is fully bypassed.
``` [1](#0-0) [5](#0-4) [2](#0-1)
