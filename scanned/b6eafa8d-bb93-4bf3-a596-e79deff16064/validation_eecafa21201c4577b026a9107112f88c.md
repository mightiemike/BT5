### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist ŌĆö (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` **by depositor address**. However, its `beforeAddLiquidity` hook silently drops the `sender` parameter and enforces the allowlist against `owner` (the LP-share recipient) instead. Any address not in the allowlist can bypass the restriction by naming an allowlisted address as `owner`, providing tokens themselves via the callback, and crediting shares to that address.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two address parameters: the first (`sender` ŌĆö the actual caller who will provide tokens via the callback) and the second (`owner` ŌĆö the address that receives LP shares). The hook silently discards `sender` and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol  L32-L42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The `SwapAllowlistExtension` ŌĆö the direct parallel ŌĆö correctly checks `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol  L31-L41
function beforeSwap(address sender, address, ...)
    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [2](#0-1) 

In `MetricOmmPool.addLiquidity`, `msg.sender` of the pool call becomes `sender` in the hook, while `owner` is a caller-supplied argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol  L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [3](#0-2) 

Because `sender` is the unnamed first parameter and is never read, the allowlist check is performed on the wrong identity. The actual token provider (`sender`) is never validated.

---

### Impact Explanation

The deposit allowlist is completely ineffective as an access control. Any unprivileged address can:

1. Call `pool.addLiquidity(allowlisted_address, salt, deltas, callbackData, extensionData)`.
2. The hook checks `allowedDepositor[pool][allowlisted_address]` ŌåÆ passes.
3. The caller provides tokens via the liquidity callback (tokens pulled from `msg.sender` of the pool, i.e., the unauthorized caller).
4. LP shares are credited to `allowlisted_address`.

The unauthorized caller has successfully deposited into a restricted pool, modifying bin balances and pool state. This breaks the admin-configured access boundary and allows arbitrary parties to manipulate bin liquidity distribution, affecting pricing and swap execution for all existing LPs.

---

### Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no complex setup. Any address can call `addLiquidity` on a pool with this extension configured, supplying any allowlisted address as `owner`. The bypass is unconditional and reachable on every `addLiquidity` call.

---

### Recommendation

Mirror the `SwapAllowlistExtension` pattern: name and check `sender`, not `owner`.

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the caller **and** the owner, both should be checked explicitly.

---

### Proof of Concept

**Setup:**
- Pool is deployed with `DepositAllowlistExtension`.
- Admin calls `setAllowedToDeposit(pool, alice, true)` ŌĆö only `alice` is allowlisted.
- `bob` is not allowlisted.

**Attack:**
```solidity
// bob calls addLiquidity with owner = alice
pool.addLiquidity(
    alice,          // owner ŌĆö allowlisted, check passes
    salt,
    deltas,
    callbackData,   // callback pulls tokens from bob (msg.sender of pool)
    extensionData
);
```

**Result:**
- `beforeAddLiquidity` receives `sender = bob`, `owner = alice`.
- Check: `allowedDepositor[pool][alice]` ŌåÆ `true` ŌåÆ no revert.
- Bob's tokens are transferred into the pool via the callback.
- Alice receives LP shares she never requested.
- Bob has deposited into a restricted pool, modifying bin state and pool pricing, bypassing the admin-configured allowlist entirely. [1](#0-0) [2](#0-1) [4](#0-3)
