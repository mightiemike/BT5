### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass a Curated Pool's Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` from the pool's perspective. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks whether the **router** is allowlisted, not the actual swapper. Any user can bypass a curated pool's swap allowlist by routing through the supported periphery router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct key), and `sender` is what the pool forwarded — the router address — when the swap originates from `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the router, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

For the pool admin to allow router-mediated swaps at all, the router must be added to the allowlist. Once the router is allowlisted, **every user** — including those the admin explicitly excluded — can call `exactInputSingle` or `exactInput` through the router and pass the check, because the extension sees only the router's address.

The same bypass applies to the multi-hop `exactInput` path (all hops after the first use `address(this)` as the pool caller) and to `exactOutput` / `exactOutputSingle`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter (the position owner explicitly passed by the caller), not `sender`, so the deposit allowlist is not affected.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers) is completely bypassed for any user who routes through `MetricOmmSimpleRouter`. The allowlist provides zero protection on the router path. Any non-allowlisted user can drain liquidity from the pool at oracle-quoted prices, defeating the pool admin's curation policy and causing direct loss of LP assets to unauthorized counterparties.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported periphery swap path. A pool admin who deploys a curated pool with `SwapAllowlistExtension` and also wants legitimate allowlisted users to use the router must allowlist the router address. This is the expected operational configuration. Once the router is allowlisted, the bypass is immediately available to any address with no special privileges or setup. The attacker only needs to call `exactInputSingle` on the public router.

---

### Recommendation

The extension must gate the **economically relevant actor** — the end user — not the intermediary contract. Two approaches:

1. **Pass the original initiator through the router**: Have `MetricOmmSimpleRouter` forward the original `msg.sender` as a verified field in `extensionData`, and have `SwapAllowlistExtension` decode and check that field when `sender` is a known router. This requires a trust relationship between the extension and the router.

2. **Check `sender` against the allowlist and require direct pool calls for curated pools**: Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the extension or factory level (e.g., reject pool configurations that combine a swap allowlist with a router-compatible setup).

The cleanest fix is option 1: the router stores the original caller in transient storage (it already does this for the payer via `_setNextCallbackContext`) and exposes it so extensions can verify the true initiator.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — router is allowed so Alice can use it.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool=pool, ...)`.
5. The router calls `pool.swap(recipient, ...)` — pool's `msg.sender` = router.
6. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully swaps on a pool he was explicitly excluded from.

```solidity
// Attacker (Bob, not allowlisted) bypasses the allowlist:
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(curatedPool),
    tokenIn:         address(token0),
    tokenOut:        address(token1),
    zeroForOne:      true,
    amountIn:        1_000_000,
    amountOutMinimum: 0,
    recipient:       bob,
    deadline:        block.timestamp + 1,
    priceLimitX64:   0,
    extensionData:   ""
}));
// Succeeds because extension sees sender=router (allowlisted), not bob (not allowlisted)
