### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the direct `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, the router is the direct caller of the pool, so the extension checks whether the **router** is allowlisted — not the actual user. A pool admin who allowlists the router to enable router-based swaps inadvertently opens the allowlist to every user on the internet.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces its gate as follows: [1](#0-0) 

`msg.sender` inside the extension is the pool (the caller of the hook), and `sender` (first argument) is whatever the pool received as its own `msg.sender` when `swap` was called. `ExtensionCalling._beforeSwap` forwards that value verbatim: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly, making the router the pool's `msg.sender`: [3](#0-2) 

The router never forwards the original `msg.sender` (the real user) to the pool. The pool therefore passes the router's address as `sender` to the extension. The extension then evaluates `allowedSwapper[pool][routerAddress]` — not `allowedSwapper[pool][realUser]`.

The integration test confirms the design intent: the allowlisted address must be the **direct pool caller**: [4](#0-3) 

`callers[0]` is a `TestCaller` that calls the pool directly; the test never exercises the router path. No test covers the case where an individually allowlisted user attempts to swap through the router, nor the case where the router is allowlisted and a non-allowlisted user exploits it.

---

### Impact Explanation

Two mutually exclusive failure modes exist for any pool that combines `SwapAllowlistExtension` with `MetricOmmSimpleRouter`:

| Pool admin choice | Consequence |
|---|---|
| Allowlist the router address | Every user on the internet can bypass the individual allowlist by routing through the public router |
| Do not allowlist the router | All router-based swaps revert with `NotAllowedToSwap`; the primary user-facing swap path is permanently broken for this pool |

In the bypass scenario, the allowlist — which is the sole access-control mechanism for curated pools — is completely defeated. Any non-allowlisted address can execute swaps against a pool that was explicitly configured to restrict trading to a specific set of actors (e.g., KYC-verified addresses, protocol-owned accounts, or whitelisted market makers).

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the canonical, documented swap entry point for end users.
- Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router will face this issue.
- No special privileges, flash loans, or unusual token behavior are required — a standard `exactInputSingle` call suffices.
- The bypass is reachable on every block once the router is allowlisted.

---

### Recommendation

The extension must resolve the real user identity rather than accepting the `sender` argument blindly. Two approaches:

1. **Pass the original caller through the pool**: Modify the pool's `swap` signature or the extension data to carry the original `msg.sender` from the router, and verify it inside the extension.

2. **Check `recipient` instead of `sender`** (partial mitigation only — recipient can be set to any address): Not recommended as a standalone fix.

3. **Preferred — router-aware identity forwarding**: Have `MetricOmmSimpleRouter` encode the real user address in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it, combined with a check that `msg.sender` (the pool) is a known factory-deployed pool so the data cannot be forged by a direct pool caller.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true          // alice is the intended gated user
  allowedSwapper[pool][router] = true         // admin must set this for router swaps to work

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, tokenIn: ..., ...})

  router calls:
    pool.swap(recipient=bob, ...)             // router is msg.sender of pool.swap

  pool calls extension:
    extension.beforeSwap(sender=router, ...)  // sender = router, not bob

  extension checks:
    allowedSwapper[pool][router] == true      // PASSES — bob bypasses the allowlist

Result:
  bob executes a swap on a pool that was supposed to block him.
  The SwapAllowlistExtension provides zero protection against router-mediated swaps.
```
