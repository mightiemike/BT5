### Title
Unbounded `timelock` in `OracleValueStopLossExtension` allows pool admin to bypass LP-protection timelock via `uint32` overflow — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary
`OracleValueStopLossExtension` stores a `uint32 timelock` that governs how long LPs have to react before any drawdown/decay/watermark change takes effect. Neither `initialize` nor `proposeOracleStopLossTimelock` validates an upper bound on this value. When `timelock` is large enough to overflow `uint32` arithmetic in `_afterTimelock`, the computed `executeAfter` wraps to a timestamp already in the past, letting the pool admin execute any pending change immediately — defeating the entire LP-protection mechanism.

---

### Finding Description

`initialize` decodes three fields from `data` but only validates two of them:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);   // ✓ checked
_validateDecay(decayPerSecondE8); // ✓ checked
// timelock — never checked
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
``` [1](#0-0) 

The setter has the same gap:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses current (possibly huge) timelock
    sched.pendingTimelock = newTimelock;            // newTimelock never validated
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
``` [2](#0-1) 

The overflow happens here:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [3](#0-2) 

`block.timestamp` is currently ≈ 1.75 × 10⁹. `type(uint32).max` ≈ 4.29 × 10⁹. Their sum ≈ 6.04 × 10⁹ overflows `uint32` and wraps to ≈ 1.75 × 10⁹ − 1, i.e., one second in the past. The guard then passes trivially:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;  // block.timestamp > wrapped past value → passes
}
``` [4](#0-3) 

Any `timelock > type(uint32).max − block.timestamp` (≈ 80 years, a plausible "set it and forget it" mistake or deliberate attack) triggers the overflow. Values below that threshold but still very large (e.g., 68 years) do not overflow but permanently brick all future configuration changes, because every `executeAfter` is set decades into the future.

---

### Impact Explanation

**Overflow path (timelock bypass):** After a legitimate pool is running with a normal timelock (e.g., 1 day), the pool admin proposes `newTimelock = type(uint32).max`, waits one day, executes it, then immediately proposes and executes `drawdownE6 = 999_999` (99.9999 %). The stop-loss floor collapses to effectively zero. Subsequent swaps can drain per-share value from every bin without the extension ever reverting. LPs who deposited relying on the stop-loss guarantee suffer direct principal loss with no reaction window.

**Brick path (no overflow):** A timelock of ~68 years makes `pendingDrawdownExecuteAfter`, `pendingDecayExecuteAfter`, and `pendingHighWatermarkExecuteAfter` all unreachable. The stop-loss config is frozen forever; the pool admin cannot tighten a dangerously loose drawdown even if they want to.

Both outcomes break the core LP-protection invariant the extension was deployed to enforce.

---

### Likelihood Explanation

Low. Requires a malicious or compromised pool admin. However, the timelock mechanism exists precisely because the pool admin is semi-trusted; bypassing it is the exact admin-boundary break the scope flags as in-scope. An honest admin could also trigger the brick path by accidentally supplying a very large value (e.g., `type(uint32).max` as a "no timelock" sentinel).

---

### Recommendation

Add a `_validateTimelock` helper and call it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint32 timelock_) private pure {
    if (timelock_ > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock_);
}
```

Call it alongside `_validateDrawdown` and `_validateDecay` in `initialize`, and at the top of `proposeOracleStopLossTimelock` before writing `sched.pendingTimelock`.

### Proof of Concept

```
// Assume pool already live with timelock = 1 day (legitimate setup).

// Step 1 – pool admin escalates timelock to overflow value
poolAdmin calls: proposeOracleStopLossTimelock(pool, type(uint32).max)
  → executeAfter = uint32(block.timestamp + 1 day)  // normal, must wait

// Step 2 – after 1 day elapses
poolAdmin calls: executeOracleStopLossTimelock(pool)
  → oracleStopLossConfig[pool].timelock = type(uint32).max  ✓

// Step 3 – propose near-total drawdown (stop-loss disabled)
poolAdmin calls: proposeOracleStopLossDrawdown(pool, 999_999)
  → executeAfter = uint32(block.timestamp + type(uint32).max)
  → uint32(~6.04e9) wraps to ~1.75e9 - 1  (past timestamp)

// Step 4 – execute immediately (no wait)
poolAdmin calls: executeOracleStopLossDrawdown(pool)
  → _requireElapsed(past_timestamp) passes
  → oracleStopLossConfig[pool].drawdownE6 = 999_999  ✓

// Step 5 – stop-loss floor = metric * (1e6 - 999_999) / 1e6 ≈ 0
// Any swap now passes _checkAndUpdateWatermarks without reverting.
// Pool admin (or colluding trader) drains LP value via swaps.
```
