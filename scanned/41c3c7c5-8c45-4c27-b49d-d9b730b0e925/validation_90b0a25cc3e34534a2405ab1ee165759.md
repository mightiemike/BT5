### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Allowing Any User to Bypass the Per-User Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the immediate `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for curated users, any unprivileged user can bypass the per-user allowlist by routing through the public router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as the first argument and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool). `sender` is whatever the pool passed as the first argument to `_beforeSwap`, which is `msg.sender` of the pool's `swap()` call:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap()
    recipient,
    ...
)
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` with `msg.sender = router`:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

So the extension receives `sender = router address`, not the actual user's address. The allowlist check becomes `allowedSwapper[pool][router]`.

**The inconsistency with `DepositAllowlistExtension`**: The deposit allowlist correctly gates the `owner` (the economically relevant actor), not the `sender`:

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The deposit allowlist gates `owner` regardless of whether the call comes through a liquidity adder or directly. The swap allowlist gates `sender`, which changes identity depending on the entry path. This is the direct analog to Velodrome Issue 5.3.9: the protection is applied inconsistently across entry paths.

**Concrete bypass scenario**:

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin allowlists `alice` and also allowlists the `MetricOmmSimpleRouter` address (to allow router-mediated swaps for curated users).
3. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle()` targeting the pool.
4. The router calls `pool.swap()` → pool calls `extension.beforeSwap(router, ...)`.
5. The check evaluates `allowedSwapper[pool][router]` = `true`.
6. `bob` successfully swaps on the curated pool, bypassing the per-user allowlist entirely.

The pool admin cannot simultaneously allow router-mediated swaps for curated users and block non-curated users from using the router, because the extension cannot distinguish the actual user behind a router call.

---

### Impact Explanation

A curated pool's swap allowlist is fully bypassed for any user who routes through the public `MetricOmmSimpleRouter`. If the router is allowlisted (a necessary step for any allowlisted user to use the router), the allowlist provides zero protection against non-allowlisted users. Depending on the pool's purpose — private pricing, restricted counterparties, or LP-protected liquidity — unauthorized swaps can drain LP assets or exploit favorable pricing that was intended only for specific counterparties. This is a direct loss of LP principal above Sherlock thresholds if the pool holds meaningful liquidity.

---

### Likelihood Explanation

The pool admin must allowlist the router for the bypass to be reachable. However, this is a natural and expected operational step: any pool admin who wants allowlisted users to be able to use the standard router will allowlist the router address. The admin has no on-chain signal that doing so opens the pool to all users. The bypass is then reachable by any unprivileged user with no special setup.

---

### Recommendation

Gate the actual user, not the immediate caller of `pool.swap()`. Two options:

1. **Mirror the deposit allowlist pattern**: Pass the economically relevant actor (the end user) through `extensionData` and have the router encode `msg.sender` there. The extension decodes and checks it. This requires a coordinated change in the router and extension.

2. **Check `sender` only when `sender != router`; require explicit per-user proof otherwise**: The extension can maintain a separate allowlist for trusted intermediaries (like the router) that must supply a signed user identity in `extensionData`.

At minimum, document clearly that allowlisting the router address opens the pool to all users, so pool admins do not inadvertently break their curation policy.

---
