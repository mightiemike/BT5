### Title
Swap Allowlist Checks Router Address Instead of End-User, Allowing Full Allowlist Bypass When Router Is Allowlisted - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of its own `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address, not the end-user's address. A pool admin who allowlists the router (a natural step to enable router-mediated swaps for their curated pool) inadvertently opens the allowlist to every user on the network, because the router is a public, permissionless contract.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

The pool therefore receives `msg.sender = router`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`. The end-user's identity is never consulted.

This creates two broken states:

1. **Bypass**: If the admin allowlists the router address (to let allowlisted users access multi-hop or router UX), every user on the network can bypass the allowlist by routing through the router.
2. **Lockout**: If the admin does not allowlist the router, allowlisted users cannot use the router at all, breaking the supported periphery path.

Note the asymmetry with `DepositAllowlistExtension`, which correctly checks `owner` (the economic beneficiary) rather than `sender` (the direct caller): [5](#0-4) 

The deposit extension gates the right actor; the swap extension does not.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and then allowlists the router (to enable router-mediated swaps for their approved users) inadvertently grants swap access to every address on the network. Non-allowlisted users can drain liquidity from the curated pool, bypass KYC/compliance controls, or execute arbitrage that the curation policy was designed to prevent. This is a direct loss of the pool's curation guarantee and can result in LP fund loss if the pool's pricing or liquidity assumptions depend on a restricted counterparty set.

---

### Likelihood Explanation

The trigger is a pool admin allowlisting the router — a natural and expected administrative action for any curated pool that wants to support the protocol's own periphery. The router is the primary user-facing swap interface. An admin who allowlists it to enable router-mediated swaps for their approved users will not realize they have opened the pool to everyone. No attacker privilege is required beyond calling the public router.

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to check the `recipient` or, better, require the pool to forward the original initiator's address. The cleanest fix is to mirror the deposit allowlist pattern: gate on the economically relevant actor. For swaps, this is the address that initiated the transaction. One approach is to have the router pass the original `msg.sender` as part of `extensionData`, and have the extension decode and check that address. Alternatively, the pool could expose a separate `initiator` field in the hook signature that the router populates with `msg.sender` before calling the pool.

---

### Proof of Concept

```
1. Admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)       // Alice is KYC'd
3. Admin calls setAllowedToSwap(pool, router, true)      // Enable router for Alice
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) — pool.msg.sender = router
6. Pool calls _beforeSwap(router, ...)
7. Extension checks allowedSwapper[pool][router] == true  ✓
8. Bob's swap executes successfully — allowlist bypassed.
```

The extension never sees Bob's address. The check at `SwapAllowlistExtension.sol:37` evaluates `allowedSwapper[pool][router]`, which the admin set to `true` in step 3, so the guard passes for any caller who routes through the router. [6](#0-5) [7](#0-6)
