### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (position recipient) instead of the actual token payer, allowing any unauthorized user to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor** address. However, its `beforeAddLiquidity` hook validates the `owner` parameter (the LP-position recipient), not the address that actually provides the tokens. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` with no requirement that `owner == msg.sender`, any unprivileged caller can route a deposit through an allowlisted address as `owner`, bypassing the guard entirely while the caller's own funds enter the pool.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` performs:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension caller) and `owner` is the position recipient supplied by the original caller. The actual token payer — stored in transient storage inside `MetricOmmPoolLiquidityAdder` — is never exposed to the extension.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the `owner`-explicit overload) validates `owner` only for the zero address:

```solidity
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [2](#0-1) 

The payer is always `msg.sender` of the adder call, stored separately in transient context:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [3](#0-2) 

The pool's `addLiquidity` passes `msg.sender` (the adder contract) as `sender` and the caller-supplied `owner` as the position owner to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [4](#0-3) 

The extension therefore has no access to the real payer. It can only see `sender` (the adder contract — identical for every user) or `owner` (the position recipient — caller-controlled). It checks `owner`.

**Attack path:**

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. Unauthorized `bob` calls `addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`.
3. `_validateOwner(alice)` passes (non-zero).
4. Pool calls `beforeAddLiquidity(adderContract, alice, ...)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Callback pulls tokens from `bob` (the payer stored in transient storage).
7. `alice` receives the LP position; `bob`'s tokens enter the restricted pool.

The allowlist is fully bypassed. `bob` is not allowlisted, yet his funds enter the pool and a position is minted under `alice`'s name.

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who can provide liquidity to a pool (e.g., for regulatory compliance, risk management, or KYC-gated pools). The bypass allows any unprivileged address to inject funds into a restricted pool. The pool receives unauthorized liquidity, altering its composition and potentially affecting existing LPs' value. The allowlisted address (`alice`) receives an unsolicited position; while she can remove it via `removeLiquidity` (which enforces `msg.sender == owner`), the pool's state has already been modified during the window before removal.

---

### Likelihood Explanation

The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point for EOAs. The `owner`-explicit overload is a documented, publicly callable function. Any actor who knows a single allowlisted address (observable on-chain via `AllowedToDepositSet` events) can execute the bypass without any special privilege. The only cost is the tokens deposited, which the attacker voluntarily provides.

---

### Recommendation

The extension should validate the **actual depositor** (the address providing tokens), not the position recipient. Two complementary fixes:

1. **In `DepositAllowlistExtension.beforeAddLiquidity`**: check `sender` (the direct pool caller) rather than `owner`, and require that the adder contract itself enforces `owner == msg.sender` for the allowlist to be meaningful at the user level.

2. **In `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (owner-explicit overload)**: enforce `owner == msg.sender` so the payer and position recipient are always the same address, making the extension's `owner` check equivalent to a payer check.

Alternatively, thread the real payer address through `extensionData` so the extension can validate it directly.

---

### Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension; only alice is allowlisted.
// bob is NOT allowlisted.

// bob approves the adder for his tokens
token0.approve(address(adder), type(uint256).max);
token1.approve(address(adder), type(uint256).max);

// bob calls with alice as owner — allowlist check passes on alice, bob pays
vm.prank(bob);
adder.addLiquidityExactShares(
    address(pool),
    alice,          // owner: allowlisted → check passes
    uint80(1),
    deltas,
    1_000e18,
    1_000e18,
    ""
);

// Result: bob's tokens are in the pool, alice holds the LP position
// The deposit allowlist did not block bob
assertGt(pool.positionBinShares(alice, 1, binIdx), 0);
```
