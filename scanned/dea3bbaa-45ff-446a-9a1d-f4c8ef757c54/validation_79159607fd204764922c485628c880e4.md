### Title
`SwapAllowlistExtension` Gates Router Address Instead of Actual User, Enabling Complete Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument forwarded by the pool, which is the pool's own `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is that direct caller. If the pool admin allowlists the router (the natural step to enable standard-periphery access for their curated pool), every unprivileged user can bypass the allowlist entirely by routing through the router.

---

### Finding Description

**Actor binding in `SwapAllowlistExtension`:** [1](#0-0) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` is the pool (enforced by `onlyPool`) and `sender` is the first argument the pool passes — which is the pool's own `msg.sender` at the time `swap()` was called.

**How the pool populates `sender`:** [2](#0-1) 

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ... {
    ...
    _beforeSwap(
      msg.sender,   // ← whoever called pool.swap()
      recipient, ...
    );
```

**How the router calls the pool:** [3](#0-2) 

```solidity
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    ...
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router calls `pool.swap()` directly, making itself `msg.sender` of that call. The pool then forwards `msg.sender` (= router) as `sender` to every `beforeSwap` hook. The extension therefore evaluates:

```
allowedSwapper[pool][router]
```

…not `allowedSwapper[pool][actualUser]`.

**The two broken states this creates:**

| Pool admin action | Consequence |
|---|---|
| Allowlists the router to support standard periphery | Every unprivileged user can swap by routing through the router — allowlist is completely void |
| Does not allowlist the router | Individually allowlisted users cannot use the router at all — standard periphery is broken for the curated pool |

Neither state matches the pool admin's intent: gate specific users while supporting the standard periphery.

---

### Impact Explanation

A curated pool (e.g., KYC-gated, institutional, or partner-only) that deploys `SwapAllowlistExtension` and allowlists `MetricOmmSimpleRouter` to support normal user flows inadvertently opens swap access to every address on-chain. Any actor — including sanctioned addresses, competitors, or bots — can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router and successfully execute swaps on the restricted pool. The allowlist provides zero protection once the router is allowlisted.

This is an admin-boundary break: an unprivileged path (`MetricOmmSimpleRouter`) bypasses the pool admin's configured access control, satisfying the contest impact gate.

---

### Likelihood Explanation

High. The `MetricOmmSimpleRouter` is the protocol's own production periphery. Any pool admin who wants their curated pool to be usable by end users through the standard interface must allowlist the router. The wrong-actor binding is invisible from the admin's perspective — `setAllowedToSwap(pool, router, true)` reads as "allow router-mediated swaps," but it silently means "allow all swaps." No special attacker capability is required beyond calling the public router.

---

### Recommendation

The extension must gate the **economic actor** (the end user), not the intermediary. Two viable approaches:

1. **Pass real user through `extensionData`**: Have the router encode `msg.sender` into `extensionData` and have the extension decode and verify it. This requires a coordinated convention between router and extension.

2. **Check `recipient` instead of `sender`**: For swap allowlists the recipient is often the economically relevant party; however this also has edge cases with multi-hop routing.

3. **Dedicated router wrapper**: Require that allowlisted pools be accessed only through a purpose-built router that appends a verified user identity to `extensionData`, rather than the generic `MetricOmmSimpleRouter`.

The cleanest fix is option 1 with a documented encoding standard so pool admins know which router versions are compatible with the allowlist extension.

### Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls: setAllowedToSwap(pool, address(router), true)
    (intending to enable standard-periphery access for their allowlisted users)
  - Attacker address is NOT individually allowlisted

Attack:
  1. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — router is msg.sender
  3. Pool calls _beforeSwap(msg.sender=router, ...)
  4. ExtensionCalling encodes sender=router and calls extension.beforeSwap(router, ...)
  5. Extension evaluates: allowedSwapper[pool][router] == true  ✓
  6. Swap executes — attacker trades on the curated pool with zero allowlist friction

Result: The allowlist is completely bypassed. Any user can swap on the
        restricted pool by routing through MetricOmmSimpleRouter.
```
