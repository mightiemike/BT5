### Title
`SwapAllowlistExtension.beforeSwap()` gates the router address instead of the end user, allowing any unprivileged user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to restrict which addresses may swap in a pool. Its `beforeSwap` hook checks the `sender` argument, which is always `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, not the end user. If the pool admin allowlists the router (a natural configuration so that allowlisted users can use the standard router), every user of the router bypasses the individual allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap()` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool â€” which is `msg.sender` of `pool.swap()`. [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

The pool records `msg.sender = router` and passes it as `sender` to the extension. The extension therefore checks `allowedSwapper[pool][router]`, not the identity of the end user who initiated the call.

**The bypass:** A pool admin who wants allowlisted users to be able to use the router must add the router to `allowedSwapper`. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every call that arrives through the router â€” regardless of who the actual end user is. Any unprivileged address can call `router.exactInputSingle()` and the extension will pass.

**Contrast with `DepositAllowlistExtension`:** The deposit allowlist correctly gates the position `owner` (the economic beneficiary), not `sender` (the intermediary). The `MetricOmmPoolLiquidityAdder` explicitly passes the real owner through to the pool, so the extension always sees the correct identity. No equivalent mechanism exists on the swap path. [4](#0-3) 

The project's own audit-target specification confirms this concern:

> *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."* [5](#0-4) 

---

### Impact Explanation

A pool admin who deploys a permissioned pool (e.g., for institutional LPs, KYC'd counterparties, or a closed market-making arrangement) and allowlists the router loses all swap-side access control. Any address on the network can swap against the pool's liquidity at oracle-derived prices, causing LP losses that the admin explicitly tried to prevent. This is a direct loss of LP principal above dust thresholds whenever the oracle price differs from the market price (i.e., always during normal operation).

---

### Likelihood Explanation

Allowlisting the router is the natural and expected configuration for any pool that wants allowlisted users to enjoy the router's slippage protection, deadline checks, and multi-hop routing. A pool admin who does not allowlist the router forces allowlisted users to call `pool.swap()` directly, losing all periphery conveniences. The bypass therefore activates under a common, well-motivated admin configuration.

---

### Recommendation

The `SwapAllowlistExtension` should gate the `recipient` (the address that receives output tokens and is the economic beneficiary of the swap) rather than `sender` (the intermediary). Alternatively, the router should encode the originating user's address in `extensionData`, and the extension should decode and verify it â€” though this requires the extension to trust the router's claim. The cleanest fix mirrors the deposit allowlist design: the pool interface should expose a dedicated "on-behalf-of" field for swaps so the extension always sees the real actor.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` attached to `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, userA, true)` and `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
3. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, zeroForOne, amount, ...)` â€” `msg.sender` at the pool is `router`.
5. Pool calls `_beforeSwap(sender=router, ...)` â†’ extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ no revert.
6. `userB`'s swap executes at oracle prices against the pool's LP liquidity, bypassing the allowlist entirely. [6](#0-5) [7](#0-6) [8](#0-7)

        "",

  }
