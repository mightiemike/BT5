### Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Allowlist Bypass via Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the actual user's address. If a pool admin allowlists the router (a natural configuration to enable router-mediated swaps for permitted users), every user â€” including explicitly disallowed ones â€” can bypass the allowlist by routing through the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument â€” the router's address when the router is the direct caller: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the original user's identity: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Attack path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists Alice and Bob as permitted swappers.
2. Admin also calls `setAllowedToSwap(pool, router, true)` so that Alice and Bob can use the router (a natural operational step).
3. Disallowed user Eve calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` â€” pool's `msg.sender` = router.
5. Extension checks `allowedSwapper[pool][router]` = `true` â†’ passes.
6. Eve's swap executes on the curated pool despite being explicitly disallowed.

The extension has no way to distinguish Alice-via-router from Eve-via-router; both present `sender` = router.

### Impact Explanation

Any user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter` whenever the router address is itself allowlisted. This completely defeats the curation guarantee of the `SwapAllowlistExtension`: disallowed counterparties gain full swap access to pools that were designed to be restricted (e.g., for compliance, KYC, or LP-protection reasons). The result is unauthorized trading and potential direct loss to LPs who relied on the allowlist to control pool exposure.

### Likelihood Explanation

The trigger condition â€” the router being allowlisted â€” is a natural and expected operational step for any pool admin who wants permitted users to be able to use the standard router. There is no documentation warning against it, and the admin has no alternative mechanism to allow router-mediated swaps for specific users without opening the pool to all users. The bypass is reachable by any unprivileged user with no special setup beyond calling the public router.

### Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor, not the intermediary. Two viable approaches:

1. **Router-forwarded identity**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a known router. This requires the extension to trust the router, so the factory or extension admin must maintain a router allowlist.

2. **Separate `swapOnBehalfOf` entrypoint**: Add a pool-level function that accepts an explicit `originator` address (verified via a trusted router registry), and pass `originator` as `sender` to extensions. This cleanly separates the payer/router identity from the gated identity.

Until fixed, pool admins using `SwapAllowlistExtension` must not allowlist the router address, which means permitted users cannot use the standard router â€” a broken UX that defeats the purpose of the periphery layer.
