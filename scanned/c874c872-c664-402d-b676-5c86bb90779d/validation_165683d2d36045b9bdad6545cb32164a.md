### Title
SwapAllowlistExtension Gates Router Address Instead of End User, Allowing Any User to Bypass Swap Allowlist via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and forwards `router` as `sender` to the extension. The extension therefore checks whether the **router contract** is allowlisted, not the actual end user. Any user who calls the public router can bypass a pool's swap allowlist as long as the router address itself is permitted.

---

### Finding Description

**Hook plumbing — identity mismatch between checked actor and economic actor.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`msg.sender` here is whoever called `pool.swap()`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` is the router contract address, not the end user.

`ExtensionCalling._beforeSwap` encodes this value as the `sender` argument and dispatches it to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct), and `sender` is the router address (wrong — should be the end user). The allowlist lookup `allowedSwapper[pool][router]` checks the router's permission, not the individual user's.

**Attack path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of KYC'd or privileged addresses.
2. Pool admin allowlists `MetricOmmSimpleRouter` so that permitted users can route through it (or the router is allowlisted by default for usability).
3. Any unpermitted user calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting this pool.
4. The router calls `pool.swap(recipient, ...)` — `msg.sender` in the pool is the router.
5. `_beforeSwap(router, ...)` is dispatched; the extension checks `allowedSwapper[pool][router]` → `true` → passes.
6. The unpermitted user's swap executes at oracle price against the pool's liquidity.

The analog to the BLS rogue key attack is exact: in BLS, the identity being cryptographically verified (the aggregated key) does not correspond to the actual authorizing party (the victim). Here, the identity being checked by the guard (the router address) does not correspond to the actual economic actor (the end user), so the guard passes without the intended party being verified.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to enforce access control (e.g., institutional-only pools, KYC-gated pools, or pools restricted to specific market makers) can be fully bypassed by any user routing through the public `MetricOmmSimpleRouter`. The attacker can execute swaps at oracle-anchored prices against liquidity that the pool admin intended to restrict, causing direct loss of LP assets and breaking the core access-control invariant of the extension system.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public-facing swap interface. Any pool that wants to support router-mediated swaps must either allowlist the router or open to all (`allowAllSwappers = true`). A pool admin who allowlists the router expecting it to act as a gated intermediary will inadvertently open the pool to all users. The bypass requires no special privileges — any user with a standard ERC-20 approval can exploit it.

---

### Recommendation

The `sender` forwarded to extensions must represent the **end user**, not the intermediary contract. Two complementary fixes:

1. **Router forwards the originating user**: `MetricOmmSimpleRouter` should pass `msg.sender` (the end user) as the `recipient`-equivalent identity in a way the pool can forward to extensions. The cleanest approach is for the router to pass the end user address in `callbackData` or a dedicated field, and for the pool to forward it as `sender` to extensions instead of `msg.sender`.

2. **Extension checks the correct field**: Alternatively, `SwapAllowlistExtension` could be redesigned to check the `recipient` field (which the router sets to the end user) rather than `sender`. However, `recipient` is the token destination, not necessarily the authorizing party, so this is a weaker fix.

The root fix is architectural: the pool's extension dispatch must carry the **originating user identity** through the router hop, analogous to how EIP-2771 meta-transactions forward the original signer rather than the relayer.

### Proof of Concept

```solidity
// Pool admin sets up allowlist pool, allowlists only the router
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Pool deployed with ext as beforeSwap extension
ext.setAllowedToSwap(pool, address(router), true); // router is allowlisted
// ext.setAllowedToSwap(pool, attacker, false);    // attacker is NOT allowlisted

// Attacker (not allowlisted) calls router directly
// router.exactInputSingle(...) -> pool.swap(msg.sender=router, ...)
// extension checks allowedSwapper[pool][router] = true -> PASSES
// Attacker's swap executes against the restricted pool
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: restrictedPool,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp,
        callbackData: "",
        extensionData: ""
    })
);
