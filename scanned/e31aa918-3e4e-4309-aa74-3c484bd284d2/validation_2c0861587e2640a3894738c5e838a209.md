### Title
SwapAllowlistExtension gates the router address instead of the actual user, enabling full allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the actual end user. The allowlist therefore gates the router address, not the individual user, making per-user swap restrictions unenforceable through the standard periphery path.

### Finding Description

**Pool → Extension call chain:**

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as `sender` to the extension dispatcher: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**Router call chain:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The actual end user (`msg.sender` of `exactInputSingle`) is stored only in the transient callback context for payment settlement and is never forwarded to the pool or to any extension.

**Result — two irreconcilable states:**

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | ❌ Blocked (broken UX) | ❌ Blocked (correct) |
| Yes | ✅ Allowed (correct) | ✅ Allowed (**bypass**) |

If the pool admin allowlists the router so that legitimate users can trade through the standard periphery, every non-allowlisted address can also bypass the restriction by routing through the same contract. If the admin does not allowlist the router, no user — including allowlisted ones — can use the standard swap path.

The same structural problem does **not** affect `DepositAllowlistExtension`, which checks the `owner` parameter (the position owner passed explicitly to `addLiquidity`), not `sender`. The LiquidityAdder preserves the correct `owner` through its call chain. [5](#0-4) 

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC'd addresses, whitelisted market makers) cannot enforce that restriction when the standard `MetricOmmSimpleRouter` is used. Any non-allowlisted address can execute swaps against the pool's LP positions by calling `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router, draining LP value at oracle-quoted prices without the pool admin's intended access control. This constitutes a direct loss of LP principal on pools whose security model depends on the allowlist.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical, documented swap entry point for end users. Any pool that configures `SwapAllowlistExtension` and also needs to support router-mediated swaps (the normal case) will be forced to allowlist the router, immediately opening the bypass to all callers. The trigger requires no special privilege — any EOA can call the router.

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the economically relevant actor, not the intermediary. Two viable fixes:

1. **Pass the actual user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; `SwapAllowlistExtension` decodes and checks it. This requires a coordinated change to the router and the extension.

2. **Check `recipient` as a proxy for the user**: For single-hop swaps the recipient is often the user, but this breaks for multi-hop paths where intermediate recipients are the router itself.

3. **Preferred — add an explicit `swapper` field to the pool's swap interface**: The pool accepts a caller-supplied `swapper` address (validated by the router as `msg.sender` before the pool call), and passes it as a distinct parameter to extensions, separate from the intermediary `sender`.

### Proof of Concept

```
1. Pool is deployed with SwapAllowlistExtension configured.
2. Admin calls setAllowedToSwap(pool, router, true)
   — necessary so that allowlisted users can trade via the router.
3. Non-allowlisted attacker calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: attacker,
       zeroForOne: true,
       amountIn: X,
       ...
     })
4. Router calls pool.swap(attacker, true, X, ...) with msg.sender = router.
5. Pool calls _beforeSwap(router, attacker, ...).
6. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → passes.
7. Swap executes; attacker receives output tokens.
   allowedSwapper[pool][attacker] was never checked.
``` [3](#0-2) [1](#0-0) [6](#0-5)
