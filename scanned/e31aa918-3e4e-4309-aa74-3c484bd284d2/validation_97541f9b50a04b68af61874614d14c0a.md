### Title
SwapAllowlistExtension checks router address as swapper identity, allowing any user to bypass the configured swap allowlist via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. The pool passes `msg.sender` of its own `swap()` call as `sender`. When `MetricOmmSimpleRouter` intermediates the swap, `msg.sender` inside the pool is the **router address**, not the actual user. If the pool admin allowlists the router (the only way to permit router-mediated swaps for legitimate users), every non-allowlisted user can bypass the restriction by routing through the same public router contract.

---

### Finding Description

**Root cause — wrong identity forwarded to the extension hook:**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the allowlist: [3](#0-2) 

**How the router breaks the identity:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The pool therefore sees `msg.sender = router`: [4](#0-3) 

The same is true for every other router entry point (`exactInput`, `exactOutputSingle`, `exactOutput`). In the multi-hop `exactInput` path the router is always the direct caller of each pool: [5](#0-4) 

**The forced dilemma for the pool admin:**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all — broken functionality |
| **Allowlist the router** | Every non-allowlisted user can bypass the gate by routing through the router |

There is no configuration that simultaneously allows legitimate users to use the router and blocks non-allowlisted users.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` (e.g., a KYC-gated or institutional-only pool) is completely open to any public user who calls `MetricOmmSimpleRouter`. The attacker pays no special cost and needs no privileged access. Unauthorized swaps can:

- Drain pool liquidity at oracle-derived prices the pool admin did not intend to expose to the public.
- Cause unexpected price impact and pool insolvency if the pool's risk parameters were calibrated for a restricted set of counterparties.
- Violate regulatory or contractual access controls the pool admin relied on.

This is a direct loss of user principal / LP assets and a broken core pool invariant (access-controlled swap).

---

### Likelihood Explanation

Likelihood is **high**:

1. The router is a public, permissionless contract — any EOA or contract can call it.
2. The bypass requires zero privileged access; the attacker only needs to know the pool address and use the standard router interface.
3. The only precondition is that the pool admin has allowlisted the router (the natural step to enable router-mediated swaps for legitimate users), which is the expected operational configuration.

---

### Recommendation

The extension must gate on the **economic actor** (the end user), not the intermediary. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and checks it. This requires a coordinated convention between router and extension.

2. **Check `tx.origin` as a fallback**: When `msg.sender` (the pool's direct caller) is a known router, the extension falls back to `tx.origin`. This is simpler but carries the standard `tx.origin` caveats (incompatible with smart-contract wallets).

3. **Preferred — router-aware identity field**: Add an explicit `swapper` field to the router's swap params that is forwarded in `extensionData`; the extension reads it only when `msg.sender` (the pool) reports a known router as `sender`. The pool admin allowlists individual users, not the router.

Until fixed, pool admins should **not** allowlist the router address in `SwapAllowlistExtension` and should document that allowlisted users must call the pool directly.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` registered as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice may swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — so Alice can use the router.
4. Mallory (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` — `msg.sender` inside the pool is the router.
6. `_beforeSwap` passes `sender = router` to `SwapAllowlistExtension.beforeSwap`.
7. The extension evaluates `allowedSwapper[pool][router]` → `true` → **no revert**.
8. Mallory's swap executes on the restricted pool, bypassing the allowlist entirely. [6](#0-5) [1](#0-0) [7](#0-6)
