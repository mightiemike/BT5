### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual calling address) and gates access on `owner` (the LP-position recipient, which is a free caller-supplied parameter). Because `owner` is not authenticated, any non-allowlisted address can add liquidity to an allowlist-protected pool by nominating any already-allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

which encodes `(sender=msg.sender, owner=owner, ...)` and dispatches to the extension.

`DepositAllowlistExtension.beforeAddLiquidity` receives both values but discards `sender` entirely:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The guard checks `allowedDepositor[pool][owner]`, not `allowedDepositor[pool][sender]`. `owner` is a free parameter supplied by the caller — `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts it directly from `msg.sender` with no allowlist validation:

```solidity
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1, bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

An attacker who is **not** on the allowlist sets `owner` to any address that **is** on the allowlist. The extension sees an allowlisted `owner`, the guard passes, and the attacker's tokens are pulled via the callback (`payer = msg.sender` of the LiquidityAdder call). The LP position is minted to the nominated allowlisted address.

The same bypass works when calling `pool.addLiquidity` directly (the caller implements `metricOmmModifyLiquidityCallback` to pay), because the pool passes `msg.sender` as `sender` and the attacker-supplied value as `owner`; the extension only inspects `owner`.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

The asymmetry confirms the deposit guard has the wrong field.

---

### Impact Explanation

The `DepositAllowlistExtension` is the sole on-chain mechanism by which a pool admin restricts who may add liquidity. Bypassing it allows any unprivileged address to:

1. Add liquidity to bins of their choice, altering the pool's liquidity distribution and effective depth at any price level.
2. Interact with pools that are intentionally restricted to vetted LPs (e.g., institutional or KYC-gated pools), violating the admin's access-control intent.
3. Pair the bypass with the `OracleValueStopLossExtension` or `PriceVelocityGuardExtension` to deposit into bins in a way that shifts watermarks or triggers guard conditions, potentially blocking legitimate swaps.

This is an admin-boundary break: an unprivileged path circumvents an admin-configured access control, matching the allowed-impact gate.

---

### Likelihood Explanation

The bypass requires no special permissions, no flash loan, and no privileged role. The attacker only needs to know one allowlisted address (trivially discoverable from on-chain events or the `allowedDepositor` mapping). The attack is a single transaction through the existing `MetricOmmPoolLiquidityAdder` router. Likelihood is **high**.

---

### Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`.

---

### Proof of Concept

```solidity
// Setup: pool has DepositAllowlistExtension; only `alice` is allowlisted.
// Attacker (bob) is NOT allowlisted.

// Bob calls the LiquidityAdder, nominating alice as owner.
liquidityAdder.addLiquidityExactShares(
    pool,
    alice,          // owner — allowlisted, passes the guard
    0,              // salt
    deltas,
    maxAmount0,
    maxAmount1,
    ""
);
// Extension checks allowedDepositor[pool][alice] == true → no revert.
// Bob's tokens are pulled via callback (payer = bob).
// LP position minted to alice.
// Bob has successfully added liquidity despite not being allowlisted.
