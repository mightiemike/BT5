### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` = `msg.sender` as seen by the pool. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. If the pool admin allowlists the router address (the natural action to enable router-mediated swaps on a permissioned pool), every user on the network bypasses the allowlist entirely.

---

### Finding Description

**Call chain for a direct swap:**

```
User → pool.swap() → _beforeSwap(msg.sender = User, ...) →
SwapAllowlistExtension.beforeSwap(sender = User) →
allowedSwapper[pool][User]  ✓ correct identity checked
```

**Call chain for a router-mediated swap:**

```
User → MetricOmmSimpleRouter.exactInputSingle() →
pool.swap() → _beforeSwap(msg.sender = Router, ...) →
SwapAllowlistExtension.beforeSwap(sender = Router) →
allowedSwapper[pool][Router]  ✗ wrong identity checked
```

In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

That `sender` value is forwarded verbatim to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when routing through `MetricOmmSimpleRouter`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no mechanism to forward the original `msg.sender` (the end user) into the pool's `sender` slot: [4](#0-3) 

The pool admin is therefore in an inescapable catch-22:

| Admin action | Result |
|---|---|
| Allowlist the router | Every user on the network bypasses the allowlist |
| Do not allowlist the router | Allowlisted users cannot swap through the router |

Neither option achieves the intended goal of "only specific users may swap through the router."

The `DepositAllowlistExtension` does not share this flaw because it gates on the explicit `owner` parameter (the position owner), not on `sender` (the caller): [5](#0-4) 

The existing integration test (`FullMetricExtension.t.sol`) only exercises direct pool calls via `TestCaller`, never a router-mediated path, so the bypass is untested: [6](#0-5) 

---

### Impact Explanation

A pool admin who deploys a permissioned pool with `SwapAllowlistExtension` and then allowlists the router (to let their approved users trade through the standard periphery) inadvertently opens the pool to every address on the network. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutput` and execute swaps that the allowlist was meant to block. This allows unauthorized parties to drain LP assets, execute swaps at oracle-driven prices the pool was not intended to serve them, and undermine any fee-tier or risk-management rationale behind the permissioned pool design.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a natural and expected configuration step for any permissioned pool that is meant to be accessible through the standard periphery. The admin has no on-chain signal that this action opens the gate to all users; the extension's `isAllowedToSwap` view function returns `true` for the router, giving no indication of the broader bypass. The router is a public, unprivileged contract callable by anyone.

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` must gate on the actual end user, not the intermediary caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires the extension to trust that the router is the caller (verify `msg.sender` is a known router).

2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the end user, but this breaks for multi-hop paths where intermediate recipients are the router itself.

3. **Dedicated router allowlist entry**: Document clearly that allowlisting the router grants access to all users, and provide a separate "router-aware" extension that decodes the real user from `extensionData` supplied by a trusted router.

The simplest safe fix is option 1: require the router to ABI-encode the original `msg.sender` into `extensionData`, and have the extension verify `msg.sender == trustedRouter` before trusting the decoded address.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   (alice is the only intended swapper)
  allowedSwapper[pool][router] = true  (admin adds router so alice can use it)

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      tokenIn: token1,
      zeroForOne: false,
      amountIn: 10_000,
      amountOutMinimum: 0,
      recipient: bob,
      ...
    })

  pool.swap() is called with msg.sender = router
  _beforeSwap(sender = router, ...)
  SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  Swap executes; bob receives token0 from the permissioned pool.

Result:
  bob, who was never allowlisted, successfully swaps on a pool
  the admin intended to restrict to alice only.
