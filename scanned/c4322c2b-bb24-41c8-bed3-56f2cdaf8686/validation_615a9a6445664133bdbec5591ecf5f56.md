### Title
`SwapAllowlistExtension` Gates on the Router Address Instead of the End User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the value forwarded by the pool as `msg.sender` at the time `swap()` was called. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The allowlist therefore gates on the router's address rather than the actual trader. A pool admin who allowlists the router (the only way to let any user reach the pool through the supported periphery) simultaneously opens the pool to every unprivileged user, defeating the entire purpose of the allowlist.

---

### Finding Description

**Actor binding in `SwapAllowlistExtension.beforeSwap`:** [1](#0-0) 

`sender` is the first argument forwarded by the pool. Inside `MetricOmmPool.swap`, that argument is always `msg.sender` — the immediate caller of the pool: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [3](#0-2) 

At that point `msg.sender` inside the pool is the **router address**, so `sender` forwarded to the extension is the router, not the end user. The allowlist lookup becomes `allowedSwapper[pool][router]`.

The pool admin faces an inescapable dilemma:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Every allowlisted user is blocked from using the supported periphery |
| Allowlist the router | Every unprivileged user can bypass the allowlist via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

The same structural problem exists in the multi-hop `exactInput` path: [4](#0-3) 

And in the recursive `exactOutput` callback path, where intermediate hops are called from inside `metricOmmSwapCallback` — `msg.sender` in the pool is again the router: [5](#0-4) 

The `ExtensionCalling._beforeSwap` dispatcher faithfully forwards whatever `msg.sender` the pool sees, so the misbinding is structural, not a dispatcher bug: [6](#0-5) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (KYC, whitelist, private market) is fully bypassable by any unprivileged user who routes through `MetricOmmSimpleRouter`. The bypass requires no special privilege: the attacker simply calls the public router. If the router is allowlisted (the only way to make the periphery usable for legitimate users), the allowlist provides zero protection. Unauthorized swaps drain pool liquidity at oracle prices, directly harming LPs who deposited under the assumption that only vetted counterparties could trade.

---

### Likelihood Explanation

`SwapAllowlistExtension` is a production extension shipped in the periphery. Any pool that deploys it and also wants users to access the pool through the standard router — a normal operational expectation — will allowlist the router. The bypass is then unconditionally reachable by any public caller with no preconditions beyond knowing the router address.

---

### Recommendation

The extension must gate on the **economic actor** (the end user), not the immediate pool caller. Two sound approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and checks that address. This requires a coordinated convention between router and extension.

2. **Check `sender` only for direct pool calls; require a signed or trusted forwarded identity for router calls**: The extension inspects whether `sender` is a known router and, if so, reads the real user from a standardized field in `extensionData`.

The simplest safe fix is option 1: the router always appends `abi.encode(msg.sender)` to `extensionData`, and the extension decodes the real swapper from there when `sender` is a recognized router address.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   (alice is the only intended swapper)
  allowedSwapper[pool][router] = true  (admin allowlists router so alice can use periphery)

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, recipient: bob, ...})

  router calls:
    pool.swap(bob, zeroForOne, amount, limit, "", extensionData)
    // msg.sender inside pool = router

  pool calls extension:
    extension.beforeSwap(router, bob, ...)
    // msg.sender in extension = pool
    // sender in extension = router

  Check: allowedSwapper[pool][router] == true  → PASSES

  bob's swap executes on the allowlisted pool despite never being allowlisted.
``` [7](#0-6) [3](#0-2)
