### Title
`DepositAllowlistExtension` guards LP position `owner` instead of calling `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary
The `DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor** address. However, its `beforeAddLiquidity` hook silently drops the `sender` parameter and checks only `owner` (the LP position recipient). Any unprivileged address can bypass the allowlist by calling `addLiquidity` with `owner` set to an already-allowed address, depositing tokens into a permissioned pool without authorization.

---

### Finding Description
`ExtensionCalling._beforeAddLiquidity` passes two distinct address arguments to every extension: `sender` (the address that called `addLiquidity` on the pool and is providing the tokens) and `owner` (the address that will own the resulting LP position). [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first positional argument but leaves it unnamed and ignores it entirely. The guard is applied only to `owner`: [2](#0-1) 

Because `owner` is a **free, caller-controlled argument** in `addLiquidity`, any address can satisfy the allowlist check by supplying an already-allowed address as `owner`, regardless of who `sender` is. The analog to the external report is exact: the guard is placed on the wrong entity — `owner` (the destination) rather than `sender` (the actor performing the action) — leaving the actual execution path unchecked.

---

### Impact Explanation
The deposit allowlist — an admin-configured access-control boundary — is bypassed by any unprivileged caller. The pool admin's intent to restrict who can deposit is violated: unauthorized liquidity enters the pool, and LP shares are credited to the supplied `owner` address rather than the actual depositor. This is a direct **admin-boundary break**: a factory/pool role check is bypassed by an unprivileged path, which is an explicitly listed allowed impact. [3](#0-2) 

---

### Likelihood Explanation
The bypass requires only knowledge of one allowed address, which is publicly readable from the `allowedDepositor` mapping or from on-chain `AllowedToDepositSet` events. No special privileges, flash loans, or oracle manipulation are needed. Any address can execute this in a single transaction. [4](#0-3) 

---

### Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`, consistent with the extension's stated purpose of gating by **depositor** address:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured; only Alice is added to the allowlist via `setAllowedToDeposit`.
2. Bob (not on the allowlist) reads Alice's address from the `AllowedToDepositSet` event or the `allowedDepositor` mapping.
3. Bob calls `pool.addLiquidity(owner = Alice, ...)` supplying his own tokens.
4. The pool calls `_beforeAddLiquidity(sender=Bob, owner=Alice, ...)`, which dispatches to the extension.
5. Inside `beforeAddLiquidity`, `sender` (Bob) is ignored; the check evaluates `allowedDepositor[pool][Alice]` — Alice is allowed, so no revert occurs.
6. Bob's tokens are deposited into the permissioned pool; Alice receives the LP shares.
7. The deposit allowlist is fully bypassed without any privileged access. [2](#0-1) [1](#0-0)
