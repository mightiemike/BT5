### Title
SwapAllowlistExtension gates the router address instead of the actual end-user, enabling full allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router contract, not the actual end-user. If the pool admin allowlists the router (which is required for any router-mediated swap to succeed), every user on-chain can bypass the allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured before-swap extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first positional argument of the `IMetricOmmExtensions.beforeSwap` ABI call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` value is on the per-pool allowlist, using `msg.sender` (the pool) as the mapping key: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router becomes `msg.sender` of `pool.swap()`: [4](#0-3) 

The call chain is therefore:

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → MetricOmmPool.swap()          [msg.sender = router]
     → _beforeSwap(sender = router, ...)
     → SwapAllowlistExtension.beforeSwap(sender = router, ...)
     → allowedSwapper[pool][router]  ← checked, NOT the actual user
```

For allowlisted users to be able to use the router at all, the pool admin must add the router to `allowedSwapper[pool][router]`. Once that entry exists, **every** address can call the router and have the allowlist check pass — the extension sees only the router, never the originating user.

The pool admin faces an impossible choice:
- **Do not allowlist the router** → allowlisted users are restricted to direct pool calls only; the router is unusable for this pool.
- **Allowlist the router** → the allowlist is completely bypassed for all users via the router.

There is no configuration that simultaneously permits router-mediated swaps for allowlisted users and blocks non-allowlisted users from using the router.

---

### Impact Explanation

The `SwapAllowlistExtension` is a production access-control guard. Its stated purpose is to restrict which addresses may swap against a pool. When the router is allowlisted (the only way to enable router-based swaps), any unprivileged user can bypass the guard by routing through `MetricOmmSimpleRouter`. This breaks the core pool access-control invariant: pools intended for restricted counterparties (e.g., institutional market makers, whitelisted integrators) become open to arbitrary swappers. Depending on pool pricing and liquidity configuration, this can result in unauthorized extraction of LP value or violation of regulatory/contractual access requirements.

---

### Likelihood Explanation

Any pool that (a) deploys `SwapAllowlistExtension` as a before-swap hook and (b) needs to support the public router must allowlist the router. Both conditions are expected in normal production use — the router is the primary user-facing entry point documented in the periphery. The bypass is therefore reachable by any user with knowledge of the router address, requiring no special privileges, no flash loans, and no oracle manipulation.

---

### Recommendation

The extension must gate the **actual end-user**, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Require the router to attest the real user via `extensionData`**: The router encodes `msg.sender` (the actual user) into `extensionData` before forwarding to the pool. The extension decodes and checks that address. This requires a trusted router; the extension should also verify `sender` is a known router when `extensionData` carries a delegated identity.

2. **Check `sender` AND require `sender == msg.sender` (no intermediary)**: Pools that need strict allowlisting should reject router-mediated calls entirely by reverting when `sender != tx.origin` or when `sender` is a known router contract. This is simpler but prevents router usage.

At minimum, the `SwapAllowlistExtension` documentation must warn that allowlisting the router grants unrestricted swap access to all users.

```
Setup:
  pool configured with SwapAllowlistExtension as before-swap extension
  pool admin calls setAllowedToSwap(pool, userA, true)       // allowlist userA
  pool admin calls setAllowedToSwap(pool, router, true)      // required for router usage

Attack:
  userC (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      amountIn: X,

        → swap executes, userC receives output tokens
