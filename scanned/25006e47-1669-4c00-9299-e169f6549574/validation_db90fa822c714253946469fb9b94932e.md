### Title
Single-slot `pendingHighWatermark` keyed only by pool silently overwrites a prior bin's proposal, permanently losing the first bin's stop-loss update — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` stores at most one pending watermark proposal per pool in `mapping(address pool => PendingHighWatermarks) public pendingHighWatermark`. The `PendingHighWatermarks` struct carries a `binIdx` field, but the mapping key is only `pool_`. A second call to `proposeOracleStopLossHighWatermarks` for a **different** `binIdx` silently overwrites the first proposal with no revert, no guard, and no event indicating the loss. When `executeOracleStopLossHighWatermarks` is later called, only the second bin's watermark is applied; the first bin's update is permanently discarded.

---

### Finding Description

`proposeOracleStopLossHighWatermarks` writes unconditionally to `pendingHighWatermark[pool_]`:

```solidity
pendingHighWatermark[pool_] =
  PendingHighWatermarks({token0: newHwmToken0, token1: newHwmToken1, binIdx: binIdx, executeAfter: executeAfter});
``` [1](#0-0) 

There is no check for an existing pending entry, no revert, and no event signalling the overwrite. The mapping is declared as a flat single-slot per pool:

```solidity
mapping(address pool => PendingHighWatermarks) public pendingHighWatermark;
``` [2](#0-1) 

`executeOracleStopLossHighWatermarks` reads the single slot and applies it to `highWatermarks[pool_][pending.binIdx]`:

```solidity
PendingHighWatermarks memory pending = pendingHighWatermark[pool_];
...
highWatermarks[pool_][pending.binIdx] =
  BinHighWatermarks({token0: pending.token0, token1: pending.token1, lastDecayTs: uint32(block.timestamp)});
``` [3](#0-2) 

The live watermarks are stored per-bin:

```solidity
mapping(address pool => mapping(int8 binIdx => BinHighWatermarks)) public highWatermarks;
``` [4](#0-3) 

The key dimension mismatch — `pendingHighWatermark` is indexed by `pool` only, while `highWatermarks` is indexed by `(pool, binIdx)` — is the direct analog of the external report's `txnHash` collision: a key that omits a distinguishing dimension causes one entry to overwrite another.

`cancelOracleStopLossHighWatermarks` compounds the confusion: it deletes the single slot without checking which `binIdx` it holds, so a cancel after an overwrite silently discards the second proposal while the admin believes they cancelled the first. [5](#0-4) 

---

### Impact Explanation

The stop-loss extension is the primary on-chain mechanism protecting LP principal from oracle-price-driven value drain. When a bin's watermark update is silently lost:

- If the admin was **tightening** the stop-loss (lowering the watermark), the bin retains a looser floor. A subsequent swap that would have been blocked by the intended tighter watermark proceeds, draining LP value beyond the configured threshold.
- The `_checkAndUpdateWatermarks` path that enforces the floor compares against the stale (un-updated) watermark, so the guard fires at the wrong level or not at all. [6](#0-5) 

This is a direct loss of LP principal above Sherlock thresholds: the stop-loss guard that LPs rely on to cap drawdown is silently misconfigured, and swaps that should be blocked are permitted.

---

### Likelihood Explanation

The trigger is the pool admin (semi-trusted, not adversarial) proposing watermarks for two different bins within the same timelock window — a routine multi-bin management operation. No external attacker is required; the overwrite happens automatically on the second legitimate admin call. The timelock window (which can be zero) makes back-to-back proposals for different bins a normal operational pattern.

---

### Recommendation

Change the pending watermark mapping to include `binIdx` as a second key dimension, matching the live `highWatermarks` layout:

```solidity
// Before
mapping(address pool => PendingHighWatermarks) public pendingHighWatermark;

// After
mapping(address pool => mapping(int8 binIdx => PendingHighWatermarks)) public pendingHighWatermark;
```

Update `proposeOracleStopLossHighWatermarks`, `executeOracleStopLossHighWatermarks`, and `cancelOracleStopLossHighWatermarks` to pass `binIdx` as the second key. Optionally add a guard in `propose` that reverts if a non-zero `executeAfter` already exists for that `(pool, binIdx)` pair, preventing silent overwrites.

---

### Proof of Concept

```
1. Pool admin calls proposeOracleStopLossHighWatermarks(pool, binIdx=0, hwm0_A, hwm1_A)
   → pendingHighWatermark[pool] = {binIdx:0, token0:hwm0_A, token1:hwm1_A, executeAfter:T+timelock}

2. Pool admin calls proposeOracleStopLossHighWatermarks(pool, binIdx=1, hwm0_B, hwm1_B)
   → pendingHighWatermark[pool] = {binIdx:1, token0:hwm0_B, token1:hwm1_B, executeAfter:T+timelock}
   (bin 0's proposal is silently overwritten; no revert, no warning event)

3. After timelock elapses, admin calls executeOracleStopLossHighWatermarks(pool)
   → highWatermarks[pool][1] updated to (hwm0_B, hwm1_B)
   → highWatermarks[pool][0] is NEVER updated

4. Bin 0 retains its old (stale/loose) watermark.
   A swap traversing bin 0 is evaluated against the stale watermark.
   If the intended tighter watermark would have blocked the swap, it proceeds,
   draining LP value beyond the configured stop-loss floor.
```
