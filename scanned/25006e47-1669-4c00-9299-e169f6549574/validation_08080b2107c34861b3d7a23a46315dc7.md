Looking at the `SwapAllowlistExtension` and how the pool calls it versus how `DepositAllowlistExtension` handles the same pattern, I can identify a concrete wrong-actor binding vulnerability.

**Key evidence gathered:**

1. Pool's `swap()` passes `msg.sender` as `sender` to `_beforeSwap` — when the router calls the pool, `msg.sender` = router address.
2. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`.
3. `DepositAllowlistExtension.beforeAddLiquidity` checks `allowedDepositor[msg.sender][owner]` = `allowedDepositor[pool][actual_LP_owner]` — correctly keyed to the real actor.
4. The liquidity adder passes the real `positionOwner` through to the pool's `addLiquidity`, so the deposit allowlist works correctly end-to-end.
5. The router stores the real payer only in transient storage; it is never forwarded to the pool or extension.

### Title
`SwapAllowlistExtension` gates by router address instead of actual swapper, enabling full allowlist bypass through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router, not the user. A pool admin who allowlists the router to enable router-based swaps for their curated users inadvertently grants every address on-chain the ability to bypass the per-user allowlist.

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives that value as `sender` and checks it against the per-pool allowlist: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, the pool's `msg.sender` is the router contract: [3](#0-2) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][real_user]`. The real user's address is stored only in the router's transient payment context and is never forwarded to the pool or extension.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter — the actual LP owner — which the liquidity adder passes through correctly: [4](#0-3) [5](#0-4) 

This asymmetry confirms that `SwapAllowlistExtension` binds authorization to the wrong actor.

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (the natural step to let their approved users trade via the standard periphery) simultaneously grants every address on-chain the ability to swap in that pool. The allowlist is completely nullified for all router-based swaps. Any unauthorized address can call `router.exactInputSingle(...)` targeting the curated pool and the extension will pass because `allowedSwapper[pool][router] == true`.

If the admin does not allowlist the router, the opposite problem occurs: allowlisted users cannot use the standard periphery path at all, breaking core swap functionality for the intended audience.

Either outcome — full allowlist bypass or unusable periphery path — constitutes a broken core pool invariant with direct policy impact on curated pools.

### Likelihood Explanation

The trigger requires only that the pool admin allowlists the router address, which is the expected operational step for any curated pool whose approved users are expected to interact via the standard router. No privileged attacker capability is needed; any EOA can exploit the bypass by calling the public router.

### Recommendation

Pass the real initiating user through the swap path so the extension can check it. One approach: add a `payer` or `initiator` field to the `beforeSwap` signature (analogous to how `owner` is passed in `beforeAddLiquidity`), and have the router forward `msg.sender` explicitly. Alternatively, document that `SwapAllowlistExtension` gates by immediate pool caller and rename/redesign it so pool admins are not misled into allowlisting the router.

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` attached.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — intending only Alice to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — intending Alice to use the router.
4. Attacker (Charlie, not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` with `msg.sender = router`.
6. Pool calls `extension.beforeSwap(router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Charlie's swap executes in the curated pool, bypassing the allowlist entirely. [6](#0-5)
