### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` instead. Because `owner` is a free caller-supplied parameter with no binding to `msg.sender`, any unprivileged address can pass the allowlist check by nominating an already-allowlisted address as the position owner.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` is declared with `sender` as an unnamed, ignored parameter:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The pool passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`_validateOwner` in the periphery adder only rejects `address(0)`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [3](#0-2) 

There is no restriction preventing an arbitrary caller from supplying any non-zero address as `owner`. The extension's own NatSpec states it "Gates `addLiquidity` by depositor address," but the depositor is `sender`, not `owner`. [4](#0-3) 

**Note on function name:** The question references `addLiquidityWithOwner`, which does not exist in the codebase. The actual entry points are `addLiquidityExactShares(pool, owner, ...)` and `addLiquidityWeighted(pool, owner, ...)`, both of which pass an attacker-controlled `owner` through to the pool unchanged. [5](#0-4) 

---

### Impact Explanation

The pool admin deploys a curated pool with `DepositAllowlistExtension` to restrict who may provide liquidity. The invariant is broken: any address can call `pool.addLiquidity(owner=allowlisted_address, ...)` directly, or route through `addLiquidityExactShares`/`addLiquidityWeighted` with `owner=allowlisted_address`. The extension passes because `allowedDepositor[pool][allowlisted_address]` is `true`. LP shares are minted to `allowlisted_address`; tokens are pulled from the actual caller via the callback.

Concrete consequences:
- **Pool curation invariant broken**: non-allowlisted parties inject liquidity into a pool the admin intended to be restricted.
- **Griefing of allowlisted addresses**: an attacker can force LP positions onto allowlisted addresses without their consent, at the attacker's own token cost.
- **Admin-boundary break**: the pool admin's access control is bypassed by an unprivileged path with no trusted-role assumption required.

The attacker does not profit directly (they pay tokens; shares go to the nominated owner), so this is medium severity — broken core access-control functionality rather than direct fund extraction.

---

### Likelihood Explanation

The bypass requires only a public call to `pool.addLiquidity` or the periphery adder with a known allowlisted address as `owner`. No privileged role, oracle manipulation, or special token behavior is needed. Any address that can observe the allowlist state (public mapping) can execute this. [6](#0-5) 

---

### Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the check with the stated invariant ("gates by depositor address") and with the `sender` value the pool already passes as the first argument. [1](#0-0)

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_depositAllowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only `allowlisted` is allowed
    address allowlisted = makeAddr("allowlisted");
    depositExtension.setAllowedToDeposit(address(pool), allowlisted, true);

    // Attacker is NOT allowlisted
    address attacker = makeAddr("attacker");
    _fundAndApprove(attacker, address(pool));

    // Attacker calls addLiquidity with owner=allowlisted
    vm.prank(attacker);
    // Does NOT revert — extension checks owner (allowlisted), not sender (attacker)
    pool.addLiquidity(allowlisted, salt, deltas, callbackData, extensionData);

    // LP shares minted to allowlisted, tokens pulled from attacker
    assertGt(positionShares(pool, allowlisted, salt, bin), 0);
}
```

The call chain is: `attacker → pool.addLiquidity(owner=allowlisted)` → `_beforeAddLiquidity(sender=attacker, owner=allowlisted)` → `extension.beforeAddLiquidity(_, owner=allowlisted)` → `allowedDepositor[pool][allowlisted] == true` → passes → shares minted. [7](#0-6) [8](#0-7)
