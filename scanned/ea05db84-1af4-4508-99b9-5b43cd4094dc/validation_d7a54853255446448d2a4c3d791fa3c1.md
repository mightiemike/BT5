### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Causing Stop-Loss to Fail Open Beyond the Configured Drawdown Floor — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the mid price as the **arithmetic mean** of bid and ask, while every other component in the protocol — including the pool's own swap execution — uses the **geometric mean** via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because the arithmetic mean is always ≥ the geometric mean (AM-GM inequality), the per-bin `metricT1` value is systematically overestimated. When the oracle spread is larger at the time of a swap than it was when the high-watermark was last set, the stop-loss floor for the `!zeroForOne` direction is effectively lower than the admin configured, allowing value leakage to continue past the drawdown threshold.

---

### Finding Description

**Wrong formula in the stop-loss extension:** [1](#0-0) 

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

**Correct formula used everywhere else in the protocol (pool swap execution and `PriceVelocityGuardExtension`):** [2](#0-1) 

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);  // geometric mean
``` [3](#0-2) 

The pool's `swap()` calls `SwapMath.midAndSpreadFeeX64FromBidAsk` to derive the geometric mid price that drives all bin-price arithmetic and settlement. [4](#0-3) 

`PriceVelocityGuardExtension.beforeSwap` also correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk`.

**How the wrong mid price corrupts the metrics:** [5](#0-4) 

```
metricT0 = t0ps + (t1 * Q64 / midPriceX64) * SCALE / shares   ← divides by mid
metricT1 = (t0 * midPriceX64 / Q64) * SCALE / shares + t1ps   ← multiplies by mid
```

By AM-GM: `(bid + ask)/2 ≥ sqrt(bid * ask)`, so the arithmetic mean is always ≥ the geometric mean. Substituting a larger mid price:

- `metricT0` is **underestimated** (divides by a larger number) — conservative, stop-loss triggers early for `zeroForOne`.
- `metricT1` is **overestimated** (multiplies by a larger number) — dangerous, stop-loss triggers **late** for `!zeroForOne`.

**The critical scenario — spread changes between watermark set and current check:**

The high-watermark is a ratchet: it is set to the maximum `metricT1` seen so far. [6](#0-5) 

If the oracle spread was **small** when the watermark was last written (overestimation ε_max ≈ 0, so `HWM ≈ T1_true_max`), and the spread is **large** at the time of the current swap (overestimation ε_current > 0), the breach check becomes:

```
T1_true_current + ε_current  <  T1_true_max × (1 − drawdown)
```

Rearranging:

```
T1_true_current  <  T1_true_max × (1 − drawdown) − ε_current
```

The stop-loss only fires when the true value has fallen by `drawdown + ε_current / T1_true_max` — **more than the configured drawdown**. The gap is:

| Oracle spread | ε_current (approx.) | Extra loss before stop-loss fires (1 % drawdown) |
|---|---|---|
| 1 % | 0.00125 % of mid | ~0.00125 % of watermark |
| 10 % | 0.125 % of mid | ~0.125 % of watermark |
| 50 % | 3.125 % of mid | ~3.125 % of watermark |

For pools with wide oracle spreads (common in volatile or illiquid markets), the stop-loss guard is materially weakened.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value leakage after a swap. When `metricT1` is overestimated, the stop-loss for the `!zeroForOne` direction (token0 outflow) fails to trigger at the configured drawdown floor. LPs suffer losses beyond the threshold the pool admin set. For pools with spreads of 10 %+, the effective drawdown floor is shifted by ~0.125 % or more of the watermark — a direct, quantifiable loss of LP principal that the guard was explicitly designed to prevent.

---

### Likelihood Explanation

Every swap on a pool with `OracleValueStopLossExtension` configured and `drawdownE6 > 0` reaches this code path via `afterSwap → _afterSwapOracleStopLoss`. Any unprivileged trader can execute a swap. The discrepancy is always present whenever `bid ≠ ask` (i.e., always in practice). The worst-case scenario — spread large at swap time, small when watermark was set — is reachable by any public trader who times their swap to a period of elevated oracle spread.

---

### Recommendation

Replace the arithmetic mean with the geometric mean, matching the formula used by the pool and `PriceVelocityGuardExtension`:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
import {SwapMath} from "@metric-core/libraries/SwapMath.sol";
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This ensures the stop-loss metrics are computed against the same mid price the pool used for swap settlement, making the drawdown floor exact.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 10_000` (1 %), and an oracle that can vary its spread.
2. Set oracle spread to 0.1 % (bid ≈ ask). Execute a swap. The watermark is set with ε_max ≈ 0, so `HWM ≈ T1_true`.
3. Change oracle spread to 50 % (bid/ask ratio = 1.5). Now ε_current ≈ 3.125 % of mid.
4. Execute swaps that drain token0 from the pool, reducing the true `metricT1` by exactly 1 % (the configured drawdown).
5. Observe: the stop-loss does **not** revert. The computed `metricT1` (using arithmetic mean) is still above the floor because the overestimation ε_current ≈ 3.125 % offsets the 1 % true drawdown.
6. Continue draining until the true `metricT1` has fallen by ~4.125 % before the stop-loss finally triggers — 3.125 % more than the admin intended.
