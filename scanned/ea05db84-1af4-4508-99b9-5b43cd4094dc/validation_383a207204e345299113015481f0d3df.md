### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any Caller to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` enforces its allowlist check against the `sender` argument forwarded by the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks whether the **router** is allowlisted rather than the **end user**. If the router is allowlisted for a pool (a natural operational choice so that normal users can trade), the allowlist gate is silently bypassed for every caller who routes through it.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct for per-pool namespacing) and `sender` is the first argument the pool passes when it calls the extension hook. The pool forwards its own `msg.sender` — the immediate caller of `pool.swap()` — as `sender`. When a user enters through `MetricOmmSimpleRouter`, the call chain is:

```
User → MetricOmmSimpleRouter.exactInput*() → pool.swap() → extension.beforeSwap(sender = router, ...)
```

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. A pool admin who allowlists the router (so that normal trading works) inadvertently opens the gate to every address that calls the router, including addresses the admin explicitly never allowlisted.

The analog to the external report is exact: just as the consensus check computed `minimum_requirement` as a fraction of the very value being compared — making the inequality structurally impossible to fail — the allowlist check here evaluates the identity of the intermediary rather than the identity of the economic actor, making the guard structurally impossible to enforce for router-mediated swaps. [2](#0-1) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers) provides **no effective restriction** once the router is allowlisted. Any unpermissioned address can call `MetricOmmSimpleRouter` and execute swaps against the pool, receiving output tokens and draining liquidity that the pool admin intended to reserve for allowlisted participants only. This is a direct loss of LP assets and a complete failure of the pool's curation policy.

---

### Likelihood Explanation

The router is the standard, documented entry point for swaps. A pool admin who deploys a `SwapAllowlistExtension` and then allowlists the router to enable normal trading will trigger the bypass for every non-allowlisted user who uses the router. The trigger requires no special privilege — any public caller of `MetricOmmSimpleRouter` reaches the vulnerable path. [3](#0-2) 

---

### Recommendation

The extension must resolve the true end-user identity rather than accepting the forwarded `sender` argument at face value. Two approaches:

1. **Pass the original caller through the extension payload**: the router encodes the real user address in `extensionData`, and the extension decodes and checks it — but this requires the router to be trusted to supply honest data.
2. **Check `sender` only for direct pool calls; require the router to forward the real user**: modify the pool or router so that the `sender` argument always reflects the economic actor, not the intermediary. The pool could accept an explicit `onBehalfOf` field that the router populates with `msg.sender` before calling `pool.swap()`.

The safest fix is to ensure the pool's `swap()` interface carries the originating user address as a distinct field that the extension can check, independent of which contract is the immediate caller.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` attached.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so normal trading works.
3. Pool admin does **not** allowlist `attacker`.
4. `attacker` calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The pool calls `extension.beforeSwap(sender = router, ...)`.
7. The extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. The attacker's swap executes successfully, bypassing the allowlist entirely. [1](#0-0)
