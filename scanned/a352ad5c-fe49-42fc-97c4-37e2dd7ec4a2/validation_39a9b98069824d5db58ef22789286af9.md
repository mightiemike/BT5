### Title
Swap Allowlist Guard Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router (the natural step to enable router-mediated swaps), every unprivileged user bypasses the allowlist entirely by calling through the router.

---

### Finding Description

**`SwapAllowlistExtension.beforeSwap` identity check:** [1](#0-0) 

The extension receives `sender` as its first argument and checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct) and `sender` is whoever called the pool's `swap` function.

**Pool passes `msg.sender` as `sender` to the extension:** [2](#0-1) 

**Router calls `pool.swap(...)` directly, making itself `msg.sender` to the pool:** [3](#0-2) 

The call chain is:

```
user ŌåÆ MetricOmmSimpleRouter.exactInputSingle(...)
         ŌåÆ pool.swap(recipient, ...)          [msg.sender = router]
             ŌåÆ _beforeSwap(msg.sender=router, ...)
                 ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                     ŌåÆ checks allowedSwapper[pool][router]   ŌåÉ WRONG ACTOR
```

The extension never sees the actual user's address. It sees only the router's address.

**Consequence:** To allow any router-mediated swap on an allowlisted pool, the admin must add the router to `allowedSwapper`. But the router is a public, permissionless contract ŌĆö allowlisting it grants every address on-chain the ability to bypass the allowlist by routing through it.

The `DepositAllowlistExtension` does **not** share this flaw because it gates on `owner` (the position owner explicitly passed by the caller), not on `sender`: [4](#0-3) 

---

### Impact Explanation

A pool deployer uses `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers). Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting that pool. The router is the pool's `msg.sender`, so the extension checks `allowedSwapper[pool][router]`. If the router is allowlisted (required for any router-mediated swap to work), the non-allowlisted user's swap succeeds. The curated pool's access control is completely nullified, allowing arbitrary users to trade against LP funds under oracle-derived prices ŌĆö a direct loss of curation policy and potential LP-fund exposure to unintended counterparties.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, deployed periphery contract with no access control of its own.
- Any user can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` targeting any pool.
- The bypass requires zero privileges: one transaction through the router suffices.
- Pool admins who want router-mediated swaps to work at all must allowlist the router, which is the natural operational step, making the bypass condition the default operational state.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the **economically relevant actor** ŌĆö the address that initiated the swap and will pay for it ŌĆö not on the intermediate contract that forwarded the call. Two sound approaches:

1. **Pass the original initiator through the router.** The router stores `msg.sender` in transient storage (already done for the payer in `_setNextCallbackContext`). The pool could read this value and pass it as `sender` to extensions, or the router could pass it as part of `extensionData` and the extension could decode it ŌĆö but this requires trust in the router, which is fragile.

2. **Gate on `sender` and require direct pool calls for allowlisted pools.** Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and revert if `sender` is a known router. This is the simplest safe fix.

3. **Preferred: use a wrapper extension that reads the true initiator from a trusted transient-storage slot set by the router.** The router already stores the payer in transient storage; extend this pattern to also store the originating user and have the extension read it via a trusted interface.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Admin calls setAllowedToSwap(pool, router, true)
    (required so that any router-mediated swap can pass the guard).
  - Alice (address not in allowedSwapper) wants to swap.

Attack:
  1. Alice calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: alice,
       zeroForOne: true,
       amountIn: X,
       ...
     });

  2. Router calls pool.swap(alice, true, X, limit, "", extensionData).
     pool.msg.sender = router.

  3. Pool calls _beforeSwap(sender=router, ...).

  4. SwapAllowlistExtension checks:
       allowAllSwappers[pool]          ŌåÆ false
       allowedSwapper[pool][router]    ŌåÆ true  ŌåÉ router is allowlisted
     ŌåÆ hook returns selector, swap proceeds.

  5. Alice receives token output. Allowlist bypassed.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds.
``` [5](#0-4) [6](#0-5) [3](#0-2)
