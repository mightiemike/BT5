### Title
`SwapAllowlistExtension` Gates on Direct Pool Caller (`sender`) Instead of Originating User — Router-Mediated Swaps Bypass Per-User Allowlist - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is the direct caller, so the pool passes `address(router)` as `sender` to the extension. A pool admin who allowlists the router address (to enable router-mediated swaps for legitimate users) inadvertently opens the allowlist to every user who routes through the same public router, bypassing the intended per-user curation.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool from its own `swap()` call — which is the direct caller of `pool.swap()`, not the originating EOA. When a user swaps through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the pool forwards `address(router)` as `sender` to the extension.

The extension's allowlist storage is keyed `allowedSwapper[pool][swapper]`:

```solidity
mapping(address pool => mapping(address swapper => bool)) public allowedSwapper;
``` [2](#0-1) 

The admin setter `setAllowedToSwap` takes an explicit `swapper` address:

```solidity
function setAllowedToSwap(address pool_, address swapper, bool allowed) external onlyPoolAdmin(pool_) {
    allowedSwapper[pool_][swapper] = allowed;
``` [3](#0-2) 

The integration test confirms the design: the allowlisted address is `address(callers[0])` — the intermediate `TestCaller` contract that directly calls the pool — not `users[0]` (the originating EOA):

```solidity
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
``` [4](#0-3) 

This means:

- **Bypass path**: A pool admin who allowlists `address(MetricOmmSimpleRouter)` (the natural step to let legitimate users swap via the router) simultaneously grants every public user access, because any EOA can call the router and the extension will see `sender = router`, which is allowlisted.
- **Broken path**: A pool admin who allowlists individual EOA addresses finds that those users cannot swap through the router at all (the router address is not allowlisted), making the router unusable on curated pools.

The analog to the external report is exact: just as `recoverSigner()` returning `address(0)` causes `require(signer == owner())` to pass for any caller when the owner is renounced, the extension checking the router address causes `allowedSwapper[pool][router]` to pass for any user when the router is allowlisted.

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC'd addresses, protocol-owned contracts, or whitelisted market makers) can be fully bypassed by any public user routing through `MetricOmmSimpleRouter`. The attacker can drain LP value through unrestricted swaps on a pool whose entire security model depends on the allowlist being enforced. This is a direct loss of LP principal and a complete failure of the pool's core access-control invariant.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported periphery entry point for swaps. Any pool admin who deploys a curated pool and then allowlists the router (the natural operational step) triggers the bypass immediately. The attacker needs no special setup: a single public `exactInput` or `exactOutput` call through the router suffices. The condition is reachable on any production pool that uses `SwapAllowlistExtension` with router support enabled. [5](#0-4) 

---

### Recommendation

The extension must gate on the **originating user**, not the direct pool caller. Two approaches:

1. **Pass the originating user through the call chain**: The pool's `swap()` function should accept an explicit `originSender` parameter (the EOA or contract that initiated the action), and the pool should forward this to the extension as `sender` rather than `msg.sender`. The router would populate this with the user's address.

2. **Check both router and user**: The extension can require that either the direct caller or an explicit `sender` field embedded in `extensionData` is allowlisted, with the pool or router attesting to the original user identity.

The `DepositAllowlistExtension` should be audited for the same issue on the `owner` vs. payer separation path through `MetricOmmPoolLiquidityAdder`. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension configured.
  2. Pool admin calls setAllowedToSwap(pool, address(MetricOmmSimpleRouter), true)
     — intended to let allowlisted users swap via the router.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. Attacker (any EOA) calls MetricOmmSimpleRouter.exactInput(pool, ...).
  5. Router calls pool.swap(sender=attacker_or_router, ...).
  6. Pool calls SwapAllowlistExtension.beforeSwap(sender=address(router), ...).
  7. Extension checks: allowedSwapper[pool][address(router)] == true → passes.
  8. Swap executes. Attacker trades on a pool they were never authorized to access.

Result:
  allowedSwapper[pool][attacker] == false, yet the swap succeeds because
  the extension sees sender = address(router), which is allowlisted.
  The per-user allowlist is completely bypassed for all router-mediated swaps.
``` [7](#0-6)
