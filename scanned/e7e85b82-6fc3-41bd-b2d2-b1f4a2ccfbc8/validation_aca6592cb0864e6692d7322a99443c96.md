### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end user. The allowlist therefore gates the router address, not the actual swapper. Any user can bypass a curated pool's swap allowlist by routing through the public router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router calls `pool.swap(recipient, ...)` directly: [4](#0-3) 

At that point `msg.sender` inside the pool is the **router**, so the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`. The actual end user's address is never visible to the extension.

This is the direct analog of the external bug: just as `contains()` returned the index *after* the target element (operating on the wrong array slot), `beforeSwap` receives the *caller of the pool* rather than the *economic actor* (operating on the wrong identity). The deposit allowlist avoids this by keying on `owner` (the position owner, explicitly passed and independent of the call chain), but the swap allowlist has no equivalent separation. [5](#0-4) 

### Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension` to restrict swaps to a curated set of addresses faces an inescapable dilemma:

- **Router not allowlisted**: allowlisted users cannot use `MetricOmmSimpleRouter` at all (broken core swap flow for the intended users).
- **Router allowlisted**: every user on the network can call the router and bypass the allowlist entirely, defeating the curation policy and allowing unauthorized swaps that drain pool liquidity at oracle-determined prices.

There is no configuration that simultaneously preserves router usability for allowlisted users and blocks non-allowlisted users. The allowlist guard silently fails open for any user who routes through the public router.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point documented and deployed by the protocol. Any user who discovers the bypass (or simply uses the router by default) can exploit it immediately without any privileged access, special tokens, or setup. The trigger is a normal `exactInputSingle` or `exactInput` call.

### Recommendation

Pass the original end-user address through the swap path so the extension can gate on it. Two options:

1. **Preferred**: Add an explicit `swapper` (or `originator`) field to the swap parameters that the router populates with `msg.sender` before calling the pool, and forward it to extension hooks as a separate argument distinct from `sender`.
2. **Alternative**: Document that `SwapAllowlistExtension` gates the *direct pool caller* (not the end user) and require curated pools to interact with the pool directly rather than through the router. This is a severe usability restriction and should be made explicit in the extension interface.

### Proof of Concept

```
Setup:
  pool P configured with SwapAllowlistExtension E
  pool admin allowlists only address Alice: allowedSwapper[P][Alice] = true
  router R is NOT allowlisted

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=Bob, ...)  ← msg.sender at pool = Router R
  3. Pool calls _beforeSwap(sender=R, ...)
  4. Extension checks allowedSwapper[P][R]  → false (router not allowlisted)
  5. Reverts — but now admin allowlists R for usability:

Setup (admin adds router):
  allowedSwapper[P][R] = true  (so Alice can use the router)

Attack (Bob, still not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=Bob, ...)  ← msg.sender at pool = Router R
  3. Pool calls _beforeSwap(sender=R, ...)
  4. Extension checks allowedSwapper[P][R]  → true  ✓ (bypass succeeds)
  5. Swap executes — Bob, who was never allowlisted, swaps freely
```

The allowlist is completely bypassed the moment the router is allowlisted for any legitimate user's benefit.
