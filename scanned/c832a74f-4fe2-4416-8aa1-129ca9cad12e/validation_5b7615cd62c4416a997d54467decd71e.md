### Title
SwapAllowlistExtension Gates Router Address Instead of End-User, Enabling Full Allowlist Bypass on Curated Pools — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender`. When users swap through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router address**, not the actual end-user. If the pool admin allowlists the router to enable router-mediated swaps for their curated users, any unprivileged user can bypass the allowlist entirely by routing through the router.

---

### Finding Description

**Root cause — wrong actor checked in the extension:**

`SwapAllowlistExtension.beforeSwap` (line 37) checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` = the pool (caller of the extension) and `sender` = the first parameter forwarded by the pool. [1](#0-0) 

**How `sender` is bound in the pool:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the pool's own `msg.sender` as `sender`. [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to the extension's `beforeSwap`. [3](#0-2) 

**How the router calls the pool:**

Every router entry-point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly. The pool's `msg.sender` is therefore the **router contract**, not the end-user. [4](#0-3) 

For multi-hop `exactInput`, intermediate hops also call the pool from the router: [5](#0-4) 

**The irreconcilable dilemma this creates:**

| Admin configuration | Consequence |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use `MetricOmmSimpleRouter` — the primary user-facing interface is broken for the curated pool |
| Router **allowlisted** (natural fix) | `allowedSwapper[pool][router] = true` → **every** user routing through the router passes the check, allowlist is fully bypassed |

The extension has no mechanism to distinguish which end-user is behind a router call. The check `allowedSwapper[pool][router]` is a single boolean that opens the gate for all users simultaneously.

---

### Impact Explanation

A curated pool's swap allowlist is completely neutralized. Any unprivileged user can execute swaps on a pool the admin intended to restrict to specific counterparties. Depending on the pool's purpose this enables:

- Unauthorized traders draining LP inventory at oracle prices on a pool designed for specific market makers only.
- Regulatory/KYC policy violations if the allowlist enforces compliance requirements.
- Circumvention of any economic protection the admin encoded via the allowlist (e.g., preventing adversarial flow from specific addresses).

The LP providers' principal is at risk because the pool's access-control invariant — "only allowlisted addresses may swap" — is broken for all router-mediated paths, which is the primary production swap path.

---

### Likelihood Explanation

**Medium.** The bypass requires the admin to allowlist the router address. This is a natural and expected configuration step because:

1. `MetricOmmSimpleRouter` is the primary user-facing swap interface documented by the protocol.
2. An admin setting up a curated pool would naturally want their allowlisted users to be able to use the router.
3. The allowlist setter `setAllowedToSwap(pool, router, true)` looks identical to allowlisting any other address — there is no warning or documentation that this collapses the entire allowlist.
4. The protocol's own test infrastructure (`SwapAllowlistExtension` tests) only exercises direct pool calls, never router-mediated calls, so the interaction is untested. [6](#0-5) 

---

### Recommendation

The extension must check the **actual end-user**, not the pool's `msg.sender`. Concrete options:

1. **Extension-data forwarding**: The router encodes the real user's address in `extensionData`; the extension decodes and checks it. The router already forwards `extensionData` verbatim to the pool.
2. **Separate router allowlist**: Maintain `allowedRouter[pool][router]` and `allowedSwapper[pool][swapper]`; require both to be set, with the router responsible for passing the real swapper identity.
3. **`sender` vs `origin` split**: Pass both `msg.sender` (pool's caller) and a user-supplied identity field through `extensionData`, and verify the caller is an approved router before trusting the identity field.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin: setAllowedToSwap(pool, alice, true)          // alice is the intended curated user
  admin: setAllowedToSwap(pool, router, true)          // admin enables router for alice's convenience

Attack (Bob, not allowlisted):
  Bob calls router.exactInputSingle({
      pool:      pool,
      recipient: bob,
      tokenIn:   token0,
      amountIn:  X,
      ...
  })

Execution trace:
  router.exactInputSingle(...)
    → pool.swap(bob, zeroForOne, X, ...)   // pool.msg.sender = router
        → _beforeSwap(router, bob, ...)
            → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                allowedSwapper[pool][router] == true  ✓  // check passes
        → swap executes, bob receives output tokens
```

Bob's swap succeeds on a pool that should have rejected him. The allowlist is fully bypassed. [1](#0-0) [7](#0-6) [8](#0-7)
