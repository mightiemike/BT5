### Title
`DepositAllowlistExtension.beforeAddLiquidity()` Gates the LP Recipient (`owner`) Instead of the Token Payer (`sender`), Allowing Any Unauthorized Party to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual token payer) and checks only the `owner` argument (the LP-share recipient). Because `owner` is a free caller-controlled parameter in `MetricOmmPool.addLiquidity`, any address not on the allowlist can bypass the guard entirely by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```solidity
// MetricOmmPool.sol
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the party that will pay tokens via the liquidity callback; `owner` is the address that will receive the minted LP shares. These are independent: any caller may supply any `owner`.

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

An unauthorized address `U` (not in `allowedDepositor[pool][U]`) calls:

```
pool.addLiquidity(authorizedOwner, salt, deltas, callbackData, extensionData)
```

The hook evaluates `allowedDepositor[pool][authorizedOwner]` → `true` and returns without reverting. `U` then pays the tokens via the callback, and `authorizedOwner` receives the LP shares. The guard never inspects `U` at all.

The analog to the external report is exact: just as `increaseStakeAndLock()` applies a lock-duration check unconditionally even when `lockIncrease == 0` (wrong condition), `beforeAddLiquidity` applies the allowlist check to the wrong actor (`owner` instead of `sender`), making the guard structurally ineffective against the party it is supposed to gate.

---

### Impact Explanation

Any address not on the deposit allowlist can add liquidity to a curated pool by specifying any allowlisted address as `owner`. Consequences:

1. **Allowlist guard is fully broken.** The deposit restriction — the only mechanism protecting a curated pool from unauthorized liquidity additions — is bypassed by a single parameter choice available to every caller.
2. **Pool-state manipulation.** The unauthorized depositor controls which bins receive liquidity and in what amounts. Targeted bin additions shift the pool's internal price curve (`curBinIdx`, `curPosInBin`, `curBinDistFromProvidedPriceE6`), directly affecting subsequent swap execution prices and LP fee accrual.
3. **LP dilution.** Existing LP positions in targeted bins are diluted by the unauthorized deposit; their share of future fees and token balances decreases proportionally.
4. **Forced LP exposure on the named owner.** The allowlisted `owner` receives LP shares they never requested, creating unwanted token exposure.

---

### Likelihood Explanation

Exploitation requires only knowing one allowlisted address (trivially discoverable on-chain via `AllowedToDepositSet` events or `allowedDepositor` view calls) and calling `addLiquidity` directly on the pool. No privileged access, flash loan, or multi-step setup is needed. Any pool deploying `DepositAllowlistExtension` as its sole deposit guard is immediately vulnerable.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    // Gate the token payer (sender), not the LP recipient (owner).
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both who pays and who holds LP shares, gate both `sender` and `owner`.

### Proof of Concept

```
Setup:
  pool deploys with DepositAllowlistExtension (extension1 = depositExt, beforeAddLiquidity order = 1)
  admin calls depositExt.setAllowedToDeposit(pool, alice, true)   // alice is authorized
  bob is NOT on the allowlist

Attack:
  bob calls pool.addLiquidity(
      owner        = alice,   // allowlisted → check passes
      salt         = 0,
      deltas       = { binIdxs: [0], shares: [10_000] },
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    allowedDepositor[pool][alice] == true  →  no revert

  bob's metricOmmSwapCallback fires; bob transfers tokens to pool.
  alice receives 10_000 LP shares in bin 0.
  bob has manipulated bin 0's balance without ever being on the allowlist.
