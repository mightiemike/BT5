### Title
`SwapAllowlistExtension.beforeSwap()` Checks Router Address Instead of Originating User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension` is designed to gate swaps on curated pools by individual swapper address. However, `beforeSwap()` checks the `sender` argument passed by the pool, which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` mediates the swap, `sender` is the router address, not the originating user. If the router is allowlisted (required for any router-mediated swap to work on the pool), every user — including those not individually allowlisted — can bypass the per-user gate by routing through the router.

### Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing its own `msg.sender` (the immediate caller) as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` (or any `exact*` variant) calls `pool.swap()`, the pool's `msg.sender` is the **router**, not the originating user: [4](#0-3) 

This creates an irreconcilable dilemma for the pool admin:

- **Router not allowlisted**: allowlisted users cannot use the router at all — the extension reverts `NotAllowedToSwap` for every router-mediated call.
- **Router allowlisted**: the per-user allowlist is completely bypassed — any user can call `router.exactInputSingle()` and the extension passes because `allowedSwapper[pool][router] == true`.

There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties, whitelisted market makers) cannot enforce that restriction when the public `MetricOmmSimpleRouter` is available. Any non-allowlisted user routes through the router, the extension sees the router as the swapper, and the trade executes. This is a direct allowlist bypass enabling unauthorized users to extract value from a pool that was explicitly configured to prevent it — a broken core pool functionality with direct loss of curation policy and potential LP fund impact.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard public entrypoint for swaps. Any user aware of the allowlist restriction can trivially route through it. No special privileges, flash loans, or multi-step setup are required — a single `exactInputSingle` call suffices. The bypass is reachable on every pool that has `SwapAllowlistExtension` configured and needs router support.

### Recommendation

Pass the **originating user** through the extension call chain rather than the immediate `msg.sender` of `pool.swap()`. One approach: the router stores the originating user in transient storage before calling `pool.swap()` and the pool reads it as an optional "on-behalf-of" hint for extensions. Alternatively, `SwapAllowlistExtension` should check `sender` only when `sender` is an EOA, and when `sender` is a known router, require the router to forward the originating user in `extensionData` and verify that identity instead.

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only `alice`.
2. Admin also allowlists the router address so that `alice` can use the router.
3. `bob` (not individually allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient=bob, ...)` — pool's `msg.sender` is the router.
5. `_beforeSwap(sender=router, ...)` is dispatched to `SwapAllowlistExtension`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. `bob` receives output tokens from the curated pool despite never being allowlisted.

If the admin does **not** allowlist the router to prevent step 6, then `alice` also cannot use the router (step 6 reverts for her too), breaking the intended user experience for legitimate participants. [3](#0-2) [1](#0-0) [5](#0-4)
