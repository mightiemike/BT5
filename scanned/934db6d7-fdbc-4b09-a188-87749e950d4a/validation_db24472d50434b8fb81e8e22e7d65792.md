Let me check the pool's `addLiquidity` to understand what `sender` is passed to the extension hook.

The call trace is fully traceable. Here is the complete analysis:

**Call path:**

1. Attacker (non-allowlisted) calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, allowlistedOwner, salt, deltas, max0, max1, extensionData)`
2. The adder stores `payer = msg.sender` (attacker) in transient storage, then calls `pool.addLiquidity(positionOwner=allowlistedOwner, ...)`
3. The pool calls `_beforeAddLiquidity(msg.sender=adder, owner=allowlistedOwner, ...)`
4. `ExtensionCalling` forwards `(sender=adder, owner=allowlistedOwner, ...)` to the extension
5. `DepositAllowlistExtension.beforeAddLiquidity` checks `allowedDepositor[msg.sender][owner]` = `allowedDepositor[pool][allowlistedOwner]` → **true → passes**
6. Pool mints position to `allowlistedOwner`, callback pulls tokens from attacker

The bypass is confirmed. The extension ignores the `sender` parameter entirely (unnamed `address,`) and only checks `owner`. The actual human payer is never visible to the extension — it lives in transient storage inside the adder.

### Title
DepositAllowlistExtension checks position `owner` instead of actual payer, allowing non-allowlisted addresses to deposit into curated pools — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` gates deposits by checking whether the `owner` (position recipient) is allowlisted, not the address that actually pays tokens. Because `MetricOmmPoolLiquidityAdder` separates `owner` from `payer`, any non-allowlisted address can bypass the allowlist by naming an allowlisted address as `owner`.

### Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` receives two address parameters: `sender` (the direct caller of `pool.addLiquidity`, i.e. the router) and `owner` (the position recipient). It silently discards `sender` and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  ...
}
``` [1](#0-0) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` explicitly allows the caller to supply any `owner` while always using `msg.sender` as the payer:

```solidity
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [2](#0-1) 

The actual payer is stored in transient storage and used only in the callback to pull tokens — it is never passed to the extension: [3](#0-2) 

Even if the extension checked `sender`, it would see the router address (`MetricOmmPoolLiquidityAdder`), not the human caller, because the pool passes `msg.sender` (the adder) as `sender` to `_beforeAddLiquidity`: [4](#0-3) 

The extension's own NatSpec states it "Gates `addLiquidity` by depositor address" — but the depositor (payer) is never checked. [5](#0-4) 

### Impact Explanation
Any non-allowlisted address can deposit tokens into a pool configured with `DepositAllowlistExtension` by calling `addLiquidityExactShares(pool, allowlistedOwner, ...)`. The allowlist — the pool admin's primary curation mechanism — is rendered completely ineffective. Non-allowlisted funds enter the curated pool, violating the `DEPOSITOR_IS_ALLOWLISTED` invariant. The position is minted to the named `allowlistedOwner`, but the tokens are pulled from the non-allowlisted payer. This breaks core pool functionality (deposit gating) and constitutes a broken-invariant finding with direct fund-flow impact.

### Likelihood Explanation
The attack requires only: (1) a pool configured with `DepositAllowlistExtension`, (2) knowledge of any one allowlisted address for that pool (publicly readable from `allowedDepositor`), and (3) token approval to the adder. No privileged access, no flash loan, no oracle manipulation. The `addLiquidityExactShares(pool, owner, ...)` overload is a standard public entrypoint explicitly designed to allow `owner != msg.sender`. [6](#0-5) 

### Recommendation
The extension must check the actual depositor, not the position owner. Since the actual human payer is not propagated through the extension call chain, the fix requires one of:

1. **Check `sender` instead of `owner`** — but this only works for direct pool calls (not router calls, where `sender` = router).
2. **Propagate the real payer** — pass the actual payer through `extensionData` or a dedicated field, and have the extension verify it.
3. **Enforce `owner == sender`** in the extension — require that the position owner and the direct caller of `pool.addLiquidity` are the same address, blocking router-mediated owner-splitting.

The most robust fix is option 3 combined with requiring the router to enforce `owner == msg.sender` when a deposit allowlist is active, or option 2 with a trusted payer field signed/verified by the router.
