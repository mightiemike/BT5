### Title
`DepositAllowlistExtension` Checks LP Position Owner Instead of Actual Caller, Allowing Full Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller of `addLiquidity`) and instead checks the `owner` parameter (the LP position recipient) against the allowlist. Any unauthorized address can bypass the deposit allowlist by supplying an allowlisted address as `owner`.

---

### Finding Description

In `MetricOmmPool.addLiquidity`, the pool passes `msg.sender` as `sender` and the user-supplied `owner` argument as `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first positional parameter (`sender`, the actual caller) is unnamed and discarded. The allowlist check is performed only on `owner`: [3](#0-2) 

The admin-facing setter is named `setAllowedToDeposit(address depositor, ...)` and the mapping is `allowedDepositor`, making the design intent unambiguous: the guard is supposed to gate the depositing caller, not the LP position recipient. [4](#0-3) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly names and checks `sender`: [5](#0-4) 

The two sibling extensions are structurally identical except that the deposit extension checks the wrong positional argument.

---

### Impact Explanation

The deposit allowlist is completely ineffective. Any unprivileged address can call `pool.addLiquidity(owner = allowlisted_address, ...)`. The extension receives `sender = attacker` (discarded) and `owner = allowlisted_address` (checked and passes). The attacker's tokens enter the pool via the callback, LP shares are minted to the allowlisted address, and the pool's private/restricted access control is fully bypassed.

Consequences:
- **Unauthorized liquidity injection**: An attacker can deposit into any bin of a restricted pool, manipulating the pool's liquidity distribution and affecting swap routing, bin-crossing behavior, and the `OracleValueStopLossExtension`'s per-bin value metrics.
- **Stop-loss manipulation**: By depositing into specific bins, an attacker can inflate or deflate per-bin `metricT0`/`metricT1` values, potentially suppressing a legitimate stop-loss trigger or causing a false one on the next swap.
- **Broken core invariant**: The pool admin's intent that only approved addresses can affect pool liquidity is violated. The `DepositAllowlistExtension` provides no actual access control over who deposits. [6](#0-5) 

---

### Likelihood Explanation

The bypass requires no special privileges, no flash loan, and no complex setup. Any address can execute it in a single transaction by passing any allowlisted address as `owner`. The only cost to the attacker is the deposited tokens, which are credited to the specified `owner` (not the attacker). The attacker cannot withdraw those shares (`removeLiquidity` enforces `msg.sender == owner`), so the attack is a griefing/manipulation vector rather than a direct theft, but the allowlist guard is unconditionally bypassed. [7](#0-6) 

---

### Recommendation

Name and check `sender` (the actual caller) instead of `owner` in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
``` [3](#0-2)

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowAllDepositors[pool] = false`; only `Alice` is allowlisted (`allowedDepositor[pool][Alice] = true`).
2. `Charlie` (not allowlisted) calls `pool.addLiquidity(owner=Alice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
3. Pool calls `extension.beforeAddLiquidity(sender=Charlie, owner=Alice, ...)`.
4. Extension evaluates `allowedDepositor[pool][Alice]` → `true` → no revert.
5. `Charlie` satisfies the token callback; LP shares are minted to `Alice`.
6. `Charlie` has deposited into a restricted pool, bypassing the allowlist entirely.

This is the direct analog of the external report's bug class: a configured guard (`DepositAllowlistExtension`) is misapplied to the wrong parameter (`owner` instead of `sender`), making the guard permanently ineffective for its intended purpose — exactly as `LimitPriceCOM.canBeFilled()` was permanently unsatisfiable due to a constraint applied to the wrong quantity.
