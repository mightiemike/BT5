### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the end user. If the router address is allowlisted (the natural configuration for a pool that wants to support router-mediated swaps), any unprivileged user can bypass the swap allowlist entirely by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument — i.e., whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly: [4](#0-3) 

At that point `msg.sender` inside `pool.swap` is the **router contract address**, not the end user. The extension therefore checks whether the **router** is allowlisted, not whether the actual trader is allowlisted.

The same identity substitution occurs for `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and wants to support router-mediated swaps must allowlist the router address. Once the router is allowlisted, **every user on the network** can call any `exact*` function on the router and the extension will see `sender = router`, which passes the check. The per-user allowlist is completely defeated. Any non-allowlisted address — including addresses the pool admin explicitly excluded — can trade on the restricted pool.

This is a direct loss of the allowlist protection that the pool admin configured, and it allows unauthorized parties to access pools that may carry special pricing, regulatory restrictions, or curated liquidity terms.

---

### Likelihood Explanation

The likelihood is **high**. The `MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Any pool admin who wants to support normal user flows through the router must allowlist it. The allowlist bypass is therefore reachable by any user on any pool that has both `SwapAllowlistExtension` active and the router allowlisted — a configuration that is both natural and expected in production.

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` should gate on the **end user**, not the intermediary. Two complementary fixes:

1. **Pass the original caller through the router.** The router should forward the original `msg.sender` (the end user) as part of `extensionData` or as a dedicated parameter, and the extension should decode and check that address.

2. **Alternatively, check `recipient` instead of `sender`.** The `recipient` argument is the address that actually receives the output tokens and is the economically relevant actor. The extension already receives `recipient` as its second argument (currently ignored with `address`). Checking `recipient` instead of `sender` would correctly gate the actual beneficiary of the swap regardless of which intermediary called the pool.

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the position owner, not the `sender`/caller): [8](#0-7) 

The swap allowlist should adopt the same pattern — gate on the economically relevant actor, not the call-chain intermediary.

---

### Proof of Concept

```
Setup:
  - Pool deployed with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)      // router allowlisted for normal UX
  - bob is NOT allowlisted

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient=bob, ...)
     → pool.swap: msg.sender = router
  3. Pool calls _beforeSwap(sender=router, recipient=bob, ...)
  4. SwapAllowlistExtension.beforeSwap:
       allowedSwapper[pool][router] == true  ✓  → no revert
  5. Swap executes. bob receives output tokens.

Result: bob, a non-allowlisted address, successfully swaps on a curated pool.
```

The root cause is that `sender` in `SwapAllowlistExtension.beforeSwap` is the pool's `msg.sender` (the router), not the end user, mirroring the BatchTrade analog where the `taker` receiving trade proceeds could differ from the `spender` subject to fee checks.
