### Title
`SwapAllowlistExtension` checks the router's address as `sender` instead of the originating user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` â€” (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool always sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool. The extension therefore checks whether the **router** is allowlisted, not the actual originating user. If the pool admin allowlists the router (a natural action to enable router-based swaps for their curated users), every unprivileged user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

**Call chain producing the wrong actor:**

`MetricOmmPool.swap()` always passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly â€” making the router `msg.sender` to the pool, so `sender = router address` in the extension: [4](#0-3) 

The same substitution occurs in every router entry point (`exactInput`, `exactOutputSingle`, `exactOutput`, and the recursive `_exactOutputIterateCallback`), because all of them call `pool.swap()` from within the router contract. [5](#0-4) 

**The broken invariant:**

The pool admin's intent when deploying a `SwapAllowlistExtension` is to restrict swaps to a named set of addresses (e.g., KYC-verified users). The extension's allowlist is keyed `allowedSwapper[pool][swapper]`. When a user swaps directly, `swapper = user` â€” the check is correct. When the same user routes through the router, `swapper = router` â€” the check is against the router's entry in the allowlist, not the user's.

This creates an irreconcilable dilemma for the pool admin:

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| Allowlist the router | **Every** user, including non-allowlisted ones, can bypass the guard via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position owner), not `sender` (the caller): [6](#0-5) 

The `owner` is passed through from the liquidity adder unchanged, so the economically relevant actor is always checked. `SwapAllowlistExtension` does not apply the same pattern â€” it checks the immediate caller of `pool.swap()` rather than the originating user.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of users (e.g., KYC-gated, institutional-only, or whitelist-only pools) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The bypassing user trades at the oracle-anchored price, extracting value from LP positions that were deposited under the assumption that only vetted counterparties would trade against them. This is a direct loss of LP principal and fee revenue to unauthorized counterparties, matching the "allowlist bypass" impact class.

---

### Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is a natural and expected administrative action: any pool admin who wants their allowlisted users to be able to use the standard periphery router must allowlist it. The router is a trusted, factory-deployed contract, so allowlisting it is not an obviously dangerous act. The probability that at least one production pool with a `SwapAllowlistExtension` will allowlist the router is high. Once the router is allowlisted, the bypass is trivially reachable by any user with no special privileges.

---

### Recommendation

Two complementary fixes:

1. **Pass the originating user through `extensionData`**: The router should encode `msg.sender` (the originating user) into `extensionData` and the extension should decode and check it. This requires a convention between the router and the extension.

2. **Check `recipient` or require a signed proof**: Alternatively, redesign `SwapAllowlistExtension` to check `recipient` (the address receiving swap output) rather than `sender`, since the recipient is the economically relevant actor for output-side gating. For input-side gating, require the router to pass the originating user in `extensionData` and verify it.

3. **Document the limitation**: At minimum, document clearly that `SwapAllowlistExtension` cannot gate router-mediated swaps by originating user, and that allowlisting the router opens the pool to all users.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension on beforeSwap order.
// 2. Admin allowlists `alice` (a legitimate user) and the router
//    (so alice can use the router).
// 3. `charlie` (not allowlisted) calls the router.

function test_swapAllowlistBypassViaRouter() public {
    // Admin allowlists alice and the router
    swapExtension.setAllowedToSwap(address(pool), alice, true);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // charlie is NOT allowlisted
    assertFalse(swapExtension.isAllowedToSwap(address(pool), charlie));

    // charlie routes through the router â€” extension sees sender=router, not charlie
    vm.prank(charlie);
    // This succeeds because allowedSwapper[pool][router] == true
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: charlie,
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        tokenIn: address(token1),
        extensionData: "",
        deadline: block.timestamp
    }));
    // charlie successfully swapped on a pool he was not allowlisted for
}
```

The root cause is in `SwapAllowlistExtension.beforeSwap` at line 37, which checks `allowedSwapper[msg.sender][sender]` where `sender` is the router address when the swap is router-mediated, rather than the originating user. [7](#0-6)
