### Title
`SwapAllowlistExtension` checks router address as `sender` instead of actual user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` = router address. If the pool admin allowlists the router (the only way to permit router-based swaps for any user), every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `pool.swap(params.recipient, ...)` — the pool's `msg.sender` is the **router**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, forwarding the **router address** as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. The extension checks `allowedSwapper[pool][router]` — not the actual user. [1](#0-0) [2](#0-1) [3](#0-2) 

The `sender` the extension receives is always the router when the user goes through the periphery, not the EOA or contract the pool admin intended to gate.

**Two broken states result:**

| Pool admin action | Effect |
|---|---|
| Does **not** allowlist the router | Allowlisted users cannot swap through the router at all — core functionality broken |
| **Allowlists the router** (to enable router-based swaps) | **Every user** can bypass the allowlist by routing through the router |

The second state is the direct-loss path: the pool admin allowlists the router believing it is a trusted contract, not realizing any EOA can call it.

---

### Impact Explanation

A pool with `SwapAllowlistExtension` is a curated pool — the admin intends to restrict trading to specific counterparties (e.g., KYC'd users, institutional traders, or whitelisted market makers). Once the router is allowlisted, any unprivileged user can:

- Execute swaps against the pool's oracle-anchored liquidity without authorization.
- Extract value from LP positions through arbitrage or directional trading that the allowlist was designed to prevent.
- Drain LP principal in pools where the allowlist was the sole access control.

This is a direct loss of LP principal and a broken core pool invariant.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery swap path. Any user who discovers the allowlist can trivially route through the router. No special privileges, flash loans, or complex setup are required — a single `exactInputSingle` call suffices. The pool admin's allowlist configuration provides zero protection once the router is allowlisted.

---

### Recommendation

The extension must check the **actual user**, not the intermediary. Two options:

1. **Pass the original `msg.sender` through the router.** The router should encode the real user in `extensionData` and the extension should decode and check it. This requires a trusted router binding (the extension must verify `msg.sender` is a known router before trusting the decoded identity).

2. **Check `recipient` instead of `sender`.** For swap allowlists, the economically relevant actor is the recipient of the output tokens. The pool already passes `recipient` as the second argument to `beforeSwap`. The extension should check `allowedSwapper[pool][recipient]` rather than `allowedSwapper[pool][sender]`.

Option 2 is simpler and does not require router trust, but the pool admin must understand that the allowlist gates the output recipient, not the payer.

---

### Proof of Concept

```
Setup:
  - Pool P has SwapAllowlistExtension configured.
  - Pool admin allowlists Alice (address A) and the router (address R)
    so that Alice can swap through the router.
  - Bob (address B) is NOT allowlisted.

Attack:
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, recipient: Bob, ...}).
  2. Router calls P.swap(recipient=Bob, ...) — pool sees msg.sender = router (R).
  3. Pool calls _beforeSwap(sender=R, recipient=Bob, ...).
  4. SwapAllowlistExtension checks allowedSwapper[P][R] → true (router is allowlisted).
  5. Swap executes. Bob receives output tokens. Allowlist bypassed.

Result:
  Bob, an unauthorized user, successfully swaps against the curated pool.
  LP funds are exposed to any unprivileged actor who routes through the router.
