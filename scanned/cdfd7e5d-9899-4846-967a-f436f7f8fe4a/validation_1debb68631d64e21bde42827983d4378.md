### Title
`SwapAllowlistExtension` checks the router address instead of the actual user on every router-mediated swap, making the per-user allowlist trivially bypassable — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the value `MetricOmmPool.swap` passes as `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the user. The extension therefore checks whether the router is allowlisted, not whether the individual user is allowlisted. A pool admin who allowlists the router to let users access it inadvertently opens the gate to every address on the network.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to every extension.** [1](#0-0) 

`_beforeSwap` is called with `msg.sender` as the first argument. When the router calls `pool.swap(...)`, `msg.sender` inside the pool is the router's address.

**Step 2 — `SwapAllowlistExtension` checks that `sender` value.** [2](#0-1) 

`msg.sender` here is the pool; `sender` is whoever called `pool.swap()`. For a direct EOA call that is the user. For any router call it is the router contract.

**Step 3 — The router always calls `pool.swap` as itself.** [3](#0-2) 

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `IMetricOmmPoolActions(pool).swap(...)` from the router's own execution context. The pool therefore always sees `msg.sender = router`.

**Step 4 — Contrast with `DepositAllowlistExtension`, which correctly uses `owner`.** [4](#0-3) 

The deposit extension ignores `sender` (the liquidity-adder contract) and checks `owner` — the economically relevant actor explicitly threaded through `addLiquidity`. No equivalent "real user" parameter exists on the swap path.

**The irreconcilable conflict.** A pool admin who wants to restrict swaps to a KYC'd set of addresses has two choices:

| Router allowlisted? | Effect |
|---|---|
| No | Allowlisted EOAs cannot use the router at all — broken UX |
| Yes | Every address on the network can bypass the per-user allowlist through the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` for institutional or KYC-restricted trading can be accessed by any arbitrary address simply by routing through `MetricOmmSimpleRouter`. The allowlist guard — the only on-chain mechanism preventing unauthorized swaps — is rendered ineffective for the entire router-mediated surface. Unauthorized traders can execute swaps against the pool's oracle-priced bins, extracting value from LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct admin-boundary break: an unprivileged path (the public router) bypasses a configured access-control guard with fund-impacting consequences for LP principal.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who deploys a restricted pool and also wants users to benefit from the router's slippage protection, multi-hop routing, and deadline enforcement will naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no unusual token behavior — a single call to `exactInputSingle` from any EOA suffices.

---

### Recommendation

Mirror the `DepositAllowlistExtension` pattern: gate on the **economically relevant actor**, not the immediate caller. Two viable approaches:

1. **Add a `swapper` parameter to `pool.swap`** (analogous to `owner` in `addLiquidity`). The router would pass `msg.sender` (the real user) as `swapper`; the pool would forward it to extensions. Extensions check `allowedSwapper[pool][swapper]`.

2. **Standardized `extensionData` forwarding.** Define a convention where the router prepends the real user address to `extensionData`. The extension decodes and checks that address. This requires the router to cooperate and the extension to trust the encoding, so approach 1 is cleaner.

Until fixed, pool admins should be warned that `SwapAllowlistExtension` provides no meaningful restriction for router-mediated swaps.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
   — intending to let allowlisted users access the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)    // allowlist alice
   — intending to restrict individual swappers.
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob, ...})
   The router calls pool.swap(bob, ...) with msg.sender = router.
   The pool calls extension.beforeSwap(router, bob, ...).
   Extension checks: allowedSwapper[pool][router] == true  → PASSES.
5. Bob successfully swaps in a pool he was never authorized to access.
   Alice's allowlist entry is irrelevant; the router entry is the only one checked.
```
