### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` keys its allowlist check on `sender`, which is `msg.sender` of the pool's `swap` call. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating user. A pool admin who allowlists the router to let legitimate users access it simultaneously opens the gate to every unprivileged caller, because the extension cannot distinguish between users behind the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, so `msg.sender` seen by the pool is the router, not the originating EOA: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The pool admin therefore faces an inescapable dilemma:

- **Do not allowlist the router** → every allowlisted EOA is blocked from using the supported periphery path, breaking the intended UX.
- **Allowlist the router** → every unprivileged address can bypass the allowlist by calling `router.exactInputSingle` (or any other router entry point), because the extension sees only the router address and approves it unconditionally.

This is structurally identical to the Dojo namespace inconsistency: the identity used for access control (`sender` = router) is decoupled from the identity that carries the economic action (the originating EOA), and no on-chain mechanism reconciles the two.

### Impact Explanation

Any pool that deploys `SwapAllowlistExtension` to enforce a curated or permissioned swap policy (KYC, institutional counterparty lists, regulatory restrictions) has that policy completely nullified the moment the router is allowlisted. An unprivileged attacker calls `router.exactInputSingle` with the restricted pool as the target; the extension approves the router address; the swap executes at oracle-derived prices against LP liquidity that was deposited under the assumption that only vetted counterparties would trade. LP providers suffer unauthorized extraction of their liquidity at oracle prices, and the pool's curation invariant is permanently broken for every block the router remains allowlisted.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported swap path for EOAs. Any pool admin who wants allowlisted users to have a normal UX will allowlist the router. The bypass requires no special privileges, no flash loans, and no multi-transaction setup — a single `exactInputSingle` call from any address suffices. The condition (router allowlisted on a curated pool) is the expected production configuration, not an edge case.

### Recommendation

The extension must receive the originating user identity, not the immediate caller. Two complementary fixes:

1. **Pass the original `msg.sender` through the router.** Add an optional `swapper` parameter to the pool's `swap` signature (or encode it in `extensionData`) so the router can forward the originating EOA. The extension then checks `allowedSwapper[pool][swapper]` instead of `allowedSwapper[pool][sender]`.

2. **Alternatively, enforce in the extension that direct-pool callers are EOAs** and reject contract callers that are not explicitly allowlisted as trusted forwarders, requiring forwarders to attest the originating address in `extensionData` with a verifiable binding.

Either approach closes the gap between the identity used for access control and the identity that carries the economic action.

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension (beforeSwap order = extension 1)
  allowedSwapper[pool][alice] = true          // alice is KYC'd
  allowedSwapper[pool][router] = true         // admin allowlists router so alice can use it
  allowedSwapper[pool][attacker] = false      // attacker is NOT allowlisted

Attack (single transaction, no special privileges):
  attacker calls router.exactInputSingle({
      pool:      pool,
      recipient: attacker,
      zeroForOne: true,
      amountIn:  X,
      ...
  })

  router calls pool.swap(recipient=attacker, ...) with msg.sender = router
  pool calls extension.beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router] → true → PASSES
  swap executes; attacker receives output tokens from LP liquidity

Result:
  attacker swapped on a pool they are explicitly blocked from accessing.
  LP providers' funds were traded against by an unauthorized counterparty.
```
