### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the pool's `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every disallowed user can bypass the allowlist by routing through the router — with no per-user restriction and no cap on the number of bypassed swaps.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool — which is the pool's own `msg.sender`. The pool calls `_beforeSwap(msg.sender, ...)`:

```solidity
// metric-core/contracts/ExtensionCalling.sol L88-99
function _beforeAddLiquidity(address sender, ...) internal {
    _callExtensionsInOrder(
        BEFORE_ADD_LIQUIDITY_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
    );
}
``` [2](#0-1) 

And `MetricOmmPool.addLiquidity` (and `swap`) passes `msg.sender` as `sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [3](#0-2) 

**Consequence of the actor mismatch:**

| Call path | `sender` seen by extension | Allowlist key checked |
|---|---|---|
| User → Pool directly | User EOA | `allowedSwapper[pool][user]` ✓ |
| User → Router → Pool | Router contract | `allowedSwapper[pool][router]` ✗ |

The extension cannot distinguish "router called by an allowlisted user" from "router called by a disallowed user." The only way for a pool admin to enable router-mediated swaps for legitimate users is to allowlist the router itself — which simultaneously grants every disallowed user the ability to bypass the allowlist by routing through the same router.

The `IMetricOmmPoolActions` documentation confirms the design intent is to gate `msg.sender` of the pool call:

```
/// @notice Swap allowlist rejected `msg.sender`.
error NotAllowedToSwap();
``` [4](#0-3) 

But the extension receives the router's address as `sender` when the router is the pool's `msg.sender`, breaking the invariant stated in the project's own security analysis: *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."*

---

### Impact Explanation

Once the router is allowlisted (a necessary step for any admin who wants allowlisted users to use the router), the bypass is unlimited:

- Any disallowed user can execute an arbitrary number of swaps on the curated pool.
- LP principal is exposed to unauthorized trading activity on a pool explicitly configured to restrict access.
- The pool's curation policy (e.g., KYC, institutional-only, compliance-gated) is silently voided with no on-chain signal.
- There is no cap on the number of bypassed swaps — the bypass persists as long as the router remains allowlisted.

This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact gate and the "broken core pool functionality" gate (the allowlist is a core configured protection).

---

### Likelihood Explanation

The trigger is a reasonable admin action: allowlisting the `MetricOmmSimpleRouter` so that allowlisted users can use the standard periphery. The admin has no on-chain signal that doing so opens the allowlist to all users. The bypass requires no special privileges, no flash loan, and no oracle manipulation — only a standard router call. Any user who discovers the allowlisted router can exploit it immediately and repeatedly.

---

### Recommendation

1. **Short-term — document the incompatibility**: Add an explicit NatSpec warning to `SwapAllowlistExtension` and `MetricOmmSimpleRouter` that the router cannot be allowlisted on a restricted pool without voiding the allowlist for all users.

2. **Long-term — propagate the original initiator**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` (or a dedicated transient slot readable by extensions), and update `SwapAllowlistExtension.beforeSwap` to extract and check that identity when the direct `sender` is a known router.

3. **Alternative**: Add an `onlyPool` guard that also validates the `sender` argument against a registry of trusted routers, and require routers to attest the real initiator via a signed or transient-storage mechanism.
