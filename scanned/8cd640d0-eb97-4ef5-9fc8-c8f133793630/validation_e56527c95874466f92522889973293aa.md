### Title
`SwapAllowlistExtension` Gates on Router Address Instead of End-User — Any User Bypasses the Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` parameter against the per-pool allowlist. The pool passes `msg.sender` of `pool.swap()` as `sender`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the **router**, not the end-user. If the pool admin allowlists the router (a natural action to "permit all router users"), every non-allowlisted address can bypass the swap gate by routing through the router.

---

### Finding Description

**Call chain when a user swaps directly:**

```
user → pool.swap()
  msg.sender = user
  _beforeSwap(sender=user, ...)
  extension.beforeSwap(sender=user)
  allowedSwapper[pool][user]  ← correct identity checked
```

**Call chain when a user swaps through the router:**

```
user → router.exactInputSingle(pool, ...)
  router → pool.swap(recipient=user, ...)
    msg.sender = router
    _beforeSwap(sender=router, ...)
    extension.beforeSwap(sender=router)
    allowedSwapper[pool][router]  ← router identity checked, not user
```

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap()` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — the router when going through periphery. The extension has no `onlyPool` guard in its override, so the lookup key is always the direct caller of the pool, not the economic actor.

This produces two broken states with no valid middle ground:

| Router allowlisted? | Effect |
|---|---|
| **Yes** | Any non-allowlisted user bypasses the gate by routing through the router |
| **No** | Every allowlisted user is blocked from using the router |

---

### Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension` to restrict swaps to KYC'd addresses, specific counterparties, or whitelisted market makers cannot achieve that goal if the router is allowlisted. Any address — including adversarial arbitrageurs — can call `router.exactInputSingle()` and the extension will see `sender = router`, which passes the check. LP providers in the restricted pool suffer adverse selection from actors the pool was explicitly designed to exclude. The pool's core access-control invariant is broken by a public, unprivileged path.

---

### Likelihood Explanation

Pool admins who want to allow "all users going through the official router" will naturally add the router to the allowlist. The router is a trusted, public periphery contract and allowlisting it is a reasonable operational decision. Nothing in the extension's interface, NatSpec, or error messages warns that allowlisting the router opens the pool to everyone. The misconfiguration is therefore likely in any deployment that combines `SwapAllowlistExtension` with `MetricOmmSimpleRouter`.

---

### Recommendation

Pass the **economic actor** — not the direct `msg.sender` — through the hook. Two options:

1. **Preferred**: Add a `swapper` field to the extension data that the router populates with `msg.sender` before calling the pool, and have the extension verify it against the allowlist (with the pool verifying the router is a trusted forwarder).

2. **Simpler**: Document that the router must never be allowlisted and add an on-chain check in `setAllowedToSwap` that rejects the router address. This prevents the bypass but still leaves allowlisted users unable to use the router.

---

### Proof of Concept

```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Pool admin allowlists the router so "all router users" can swap
ext.setAllowedToSwap(pool, address(router), true);
// alice is NOT on the allowlist
// bob IS on the allowlist

// Alice bypasses the allowlist via the router
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: alice,
    zeroForOne: true,
    amountIn: 1e18,
    ...
}));
// pool.swap() is called with msg.sender = router
// extension sees sender = router → allowedSwapper[pool][router] = true → passes
// Alice swaps successfully despite not being allowlisted

// Bob (allowlisted) tries to use the router — also works, but only because router is allowlisted
// If admin had NOT allowlisted the router, Bob's direct allowlist entry would be useless via router
```
