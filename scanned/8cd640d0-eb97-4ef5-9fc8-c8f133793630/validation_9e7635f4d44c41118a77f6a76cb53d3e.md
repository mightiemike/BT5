### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `sender`, which is the pool's `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the original user's address. A pool admin who wants allowlisted users to trade through the router must allowlist the router itself. Once the router is allowlisted, **any unprivileged user** can bypass the allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool — which is the pool's own `msg.sender` at the time `swap` was called.

The pool's `swap` function calls the extension dispatcher as:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
         packedSlot0Initial, bidPriceX64, askPriceX64, extensionData)
    )
);
```

where `sender` is `msg.sender` of the pool's `swap` call.

When a user calls `MetricOmmSimpleRouter.exactInputSingle(...)`, the router calls:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

`msg.sender` to the pool is the **router contract**, so `sender` in the extension is the router's address. The allowlist check becomes:

```
allowedSwapper[pool][router]
```

The original user's address is never passed to the extension and is not available in any parameter of `beforeSwap`. The extension has no way to distinguish a direct swap by an allowlisted user from a router-mediated swap by a non-allowlisted user.

This creates an inescapable dilemma for the pool admin:

| Router allowlisted? | Allowlisted users can use router? | Non-allowlisted users can bypass? |
|---|---|---|
| No | ❌ Blocked | ✅ Safe |
| Yes | ✅ Works | ❌ **Bypassed** |

If the admin allowlists the router (the only way to let allowlisted users trade through the router), every user — including non-allowlisted ones — can bypass the guard by routing through `MetricOmmSimpleRouter`.

---

### Impact Explanation

The `SwapAllowlistExtension` is the protocol's primary mechanism for restricting swap access to specific addresses (e.g., KYC'd users, institutional counterparties, or whitelisted market makers). When the router is allowlisted, the guard is fully bypassed: any address can execute swaps against a pool that was intended to be access-controlled. LPs who deposited under the assumption that only vetted counterparties would trade against them are exposed to unrestricted swap flow, which can drain their positions at oracle-derived prices without their consent. This constitutes broken core pool functionality and an admin-boundary break via an unprivileged path.

---

### Likelihood Explanation

The trigger condition — the router being allowlisted — is not a misconfiguration; it is the **only valid operational state** for a pool that uses `SwapAllowlistExtension` and also wants allowlisted users to trade through the standard periphery router. Any pool operator who deploys this extension and expects users to interact via `MetricOmmSimpleRouter` will inevitably reach this state. The bypass is then reachable by any unprivileged user with no special access, no admin role, and no malicious setup beyond calling the public router.

---

### Recommendation

The `beforeSwap` hook signature does not carry the original user's address. Two remediation paths exist:

1. **Protocol-level fix**: Extend the `beforeSwap` hook signature to include the original initiator address (e.g., passed by the router via `extensionData` and decoded by the extension). The router would encode `msg.sender` into `extensionData` before calling the pool, and the extension would decode and check it.

2. **Extension-level workaround**: Document that `SwapAllowlistExtension` is incompatible with any intermediary router and must only be used with direct pool calls. Enforce this by checking that `sender == tx.origin` (with the known limitations of that check) or by requiring the extension to be paired with a dedicated, non-public router that enforces its own identity checks before calling the pool.

The cleanest fix is option 1: pass `msg.sender` from the router into `extensionData` and have the extension decode and verify it, so the checked identity is always the economic initiator, not the intermediary.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
2. Admin allowlists user1: allowedSwapper[pool][user1] = true.
3. Admin allowlists the router so user1 can trade via MetricOmmSimpleRouter:
       allowedSwapper[pool][router] = true.
4. Attacker (user2, not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) — msg.sender to pool = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true → passes.
8. Swap executes. user2 has bypassed the allowlist with no privileged access.
