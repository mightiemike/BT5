### Title
`SwapAllowlistExtension` Gates on Router Address Instead of Actual User, Enabling Full Allowlist Bypass — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` (the immediate caller of `pool.swap`). When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router so that allowlisted users can reach the pool via the router, every unpermissioned user can bypass the allowlist by routing through the same public router contract.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument — the immediate caller of `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap` directly, making the router the pool's `msg.sender`. The original end-user address is stored only in transient storage for the payment callback and is never forwarded to the extension: [4](#0-3) 

Consequence: the extension can only see the router address as `sender`. A pool admin who wants allowlisted users to reach the pool via the router must add `allowedSwapper[pool][router] = true`. Once that entry exists, every public user — allowlisted or not — can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) and the extension will pass them through, because the check resolves to `allowedSwapper[pool][router]`, which is `true`.

The `DepositAllowlistExtension` does not share this flaw: it gates on the `owner` argument (the position owner), which the liquidity adder passes explicitly and which is independent of the intermediate caller. [5](#0-4) 

---

### Impact Explanation

Any user can trade on a pool that the admin intended to restrict to a curated set of swappers. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, or rate-limited), this allows:

- Unauthorized price impact on a pool whose LP positions were sized for a controlled flow.
- Extraction of LP value by actors the pool was explicitly designed to exclude.
- Violation of any regulatory or contractual restriction the allowlist was meant to enforce.

The loss is direct: LP funds are exposed to swap flows the pool admin explicitly prohibited.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a natural and expected action for any allowlisted pool that is meant to be usable through the standard periphery. The `MetricOmmSimpleRouter` is the canonical swap entry point documented in the periphery. A pool admin who does not allowlist the router blocks all router-mediated swaps for everyone, including allowlisted users, making the allowlist effectively unusable with the router. The admin is therefore forced into the vulnerable configuration to provide normal UX.

---

### Recommendation

Pass the original end-user address through the extension mechanism rather than the immediate pool caller. Two concrete options:

1. **Extension-data forwarding**: Require the router to encode the originating user address in `extensionData` and have `SwapAllowlistExtension` decode and verify it. The pool's `_beforeSwap` already forwards `extensionData` verbatim to extensions.

2. **Dedicated sender field**: Add an explicit `originator` field to the swap call signature (analogous to how `addLiquidity` separates `sender` from `owner`) so the pool can pass the true end-user to extensions independently of `msg.sender`.

Until fixed, pool admins should not allowlist the router on pools with `SwapAllowlistExtension`; they must require direct `pool.swap` calls from allowlisted EOAs only.

---

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension (beforeSwap order set).
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:          <restricted pool>,
           recipient:     bob,
           zeroForOne:    true,
           amountIn:      X,
           ...
       })

5. Router calls pool.swap(...) — pool's msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true  →  passes.
8. Swap executes; Bob receives output tokens from the restricted pool.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
``` [6](#0-5) [7](#0-6)
