### Title
OracleValueStopLossExtension Uses Arithmetic Mid-Price Instead of Geometric Mid-Price, Causing Stop-Loss to Fail for `!zeroForOne` Swaps — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes `midPriceX64` as the arithmetic mean `(bidPriceX64 + askPriceX64) / 2`, while the pool's swap execution and the `PriceVelocityGuardExtension` both use `SwapMath.midAndSpreadFeeX64FromBidAsk` (geometric mean). By AM-GM inequality the arithmetic mean is always ≥ the geometric mean, so the stop-loss inflates `metricT1` for every swap. This makes the stop-loss systematically less likely to trigger for `!zeroForOne` swaps (token1 in, token0 out), allowing token0 to drain from the pool past the configured drawdown floor.

---

### Finding Description

**Root cause — wrong mid-price formula in the guard:**

In `_afterSwapOracleStopLoss`:

```solidity
// OracleValueStopLossExtension.sol line 218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

The pool's swap path uses:

```solidity
// MetricOmmPool.sol lines 242-243
(uint256 midPriceX64, uint256 baseFeeX64) =
    SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));  // geometric mean
```

`PriceVelocityGuardExtension` also uses the geometric mean:

```solidity
// PriceVelocityGuardExtension.sol line 48
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

**How the inflated mid corrupts `metricT1`:**

The `_metrics` function computes:

```
metricT1 = (t0 * mid / Q64) * SCALE / shares  +  t1 * SCALE / shares
```

Because `mid_arithmetic ≥ mid_geometric`, the term `t0 * mid / Q64` is always overestimated. The stop-loss blocks `!zeroForOne` swaps only when `metricT1 < hwm1 * floorMultiplier / E6`. An inflated `metricT1` raises the apparent value per share, making the breach condition harder to satisfy. Swaps that should have been reverted by the stop-loss are allowed to execute.

**Magnitude of the discrepancy:**

| Bid-ask spread | Arithmetic mean | Geometric mean | Overestimation |
|---|---|---|---|
| 1 % | 1.0000 | 0.999987 | ~0.001 % |
| 10 % | 1.0000 | 0.99875 | ~0.125 % |
| 50 % | 1.0000 | 0.9683 | ~3.3 % |

For a pool configured with a 3 % drawdown floor and a 50 % oracle spread, the stop-loss would fail to trigger even when the actual per-share value has already fallen 6.3 % below the watermark.

**Analogy to the external bug class:**

The correct mid price (geometric mean, computable from the same `bidPriceX64`/`askPriceX64` inputs already passed to the hook) is available but not used. The guard instead applies a different, always-higher value — exactly the "calculated but not applied" pattern from the seed report.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism preventing per-share value from falling below the configured drawdown floor. When it fails to trigger for `!zeroForOne` swaps, token0 exits the pool without reversal. LP principal is permanently lost up to the magnitude of the overestimation times the watermark value. This is a direct loss of LP assets above Sherlock thresholds for pools with non-trivial oracle spreads.

---

### Likelihood Explanation

- Every pool that deploys `OracleValueStopLossExtension` is affected; no special attacker setup is required.
- The discrepancy is always present whenever `bid ≠ ask` (i.e., any real pool with a non-zero spread).
- The impact scales with the oracle spread, which is controlled by the price provider and can be large during volatile markets.
- Any ordinary `!zeroForOne` swap that crosses the drawdown boundary exploits the gap; no privileged access is needed.

---

### Recommendation

Replace the arithmetic mean with the same geometric-mean helper used by the pool and the velocity guard:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

This ensures the stop-loss evaluates per-share value at the same mid price the pool uses for swap execution, eliminating the systematic overestimation of `metricT1`.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured with `drawdownE6 = 30_000` (3 % floor).
2. Set the oracle to return `bid = 0.75 * ONE_X64`, `ask = 1.25 * ONE_X64` (50 % spread).
   - Arithmetic mid = `1.0 * ONE_X64`; geometric mid ≈ `0.9683 * ONE_X64`.
3. Seed the pool with token0 and token1 so that the initial watermarks are set at the arithmetic-mid metric values.
4. Execute a sequence of `!zeroForOne` swaps that drain token0 until the actual per-share value (at geometric mid) has fallen 6 % below the watermark — 3 % beyond the configured floor.
5. Observe: the stop-loss does not revert any of these swaps because `metricT1` computed with the inflated arithmetic mid still appears above the floor.
6. LP withdraws and receives ~3 % less token0 than the stop-loss was supposed to guarantee. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
