### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual end-user, allowing allowlist bypass or blocking legitimate allowlisted swappers — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is a production extension that gates `swap` access per pool by checking whether the `sender` argument passed from the pool is on the allowlist. However, `MetricOmmPool.swap()` always passes `msg.sender` as `sender` to `_beforeSwap`, and when users route through `MetricOmmSimpleRouter`, `msg.sender` at the pool level is the **router contract**, not the actual end user. The extension therefore checks the router's allowlist status, not the individual user's, breaking the per-user access control in both directions.

---

### Finding Description

**Call chain when a user swaps via `MetricOmmSimpleRouter.exactInputSingle()`:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — at this point `msg.sender` inside the pool is the **router**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, passing the router address as `sender`. [1](#0-0) 

4. `_beforeSwap` encodes `sender` (= router) and dispatches to the extension. [2](#0-1) 

5. `SwapAllowlistExtension.beforeSwap` receives `sender = router` and checks `allowedSwapper[pool][router]`. [3](#0-2) 

The router call site that causes `msg.sender` to be the router at the pool level: [4](#0-3) 

The extension's stated purpose is to gate by **swapper address**: [5](#0-4) 

---

### Impact Explanation

Two broken invariants arise simultaneously:

**Scenario A — Allowlist bypass (unauthorized access):**
The pool admin allowlists the router address (the only way to let router users swap at all). Because `allowedSwapper[pool][router]` is `true`, every user who calls through `MetricOmmSimpleRouter` passes the check regardless of their individual allowlist status. Non-KYC'd, blacklisted, or otherwise restricted addresses can freely swap by routing through the standard periphery.

**Scenario B — Broken swap functionality for legitimate users:**
The pool admin allowlists individual user addresses but does not allowlist the router. Allowlisted users who call `MetricOmmSimpleRouter.exactInputSingle()` have their swap rejected with `NotAllowedToSwap` because the extension sees `sender = router` (not allowlisted). The standard periphery is unusable for any pool with a per-user allowlist, breaking core swap flows.

Both outcomes are fund-impacting: Scenario A allows unauthorized parties to extract value from a restricted pool; Scenario B makes the pool's primary swap interface non-functional for legitimate LPs and traders.

---

### Likelihood Explanation

- `SwapAllowlistExtension` is a production periphery contract deployed alongside `MetricOmmSimpleRouter`.
- Any pool that configures `SwapAllowlistExtension` and expects users to interact via the router will immediately exhibit one of the two broken behaviors above.
- No special attacker capability is required — a normal `exactInputSingle` call through the router is sufficient to trigger the bypass.

---

### Recommendation

The router must forward the actual end-user address so the extension can check it. Two approaches:

**Option 1 — Pass user address through `extensionData`:**
`MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes the user address from `extensionData` and checks it instead of (or in addition to) `sender`.

**Option 2 — Explicit `msgSender` parameter on `swap`:**
Add an explicit `msgSender` parameter to `IMetricOmmPoolActions.swap()` (similar to Uniswap v4's approach), validated against `msg.sender` or a trusted router registry, and pass it as `sender` to extensions.

The same structural issue exists in `_afterSwap`, which also passes `msg.sender` as `sender` to `afterSwap` extensions. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is KYC'd
  allowedSwapper[pool][bob]   = false  // bob is not KYC'd
  allowedSwapper[pool][router] = true  // admin must set this for router users to work

Attack (Scenario A):
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(...)
  → pool calls _beforeSwap(msg.sender=router, ...)
  → SwapAllowlistExtension checks allowedSwapper[pool][router] == true → PASSES
  → bob's swap executes despite not being allowlisted

Broken functionality (Scenario B):
  admin does NOT allowlist router (tries to enforce per-user control)
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(...)
  → pool calls _beforeSwap(msg.sender=router, ...)
  → SwapAllowlistExtension checks allowedSwapper[pool][router] == false → REVERTS NotAllowedToSwap
  → alice cannot use the standard periphery despite being allowlisted
```
