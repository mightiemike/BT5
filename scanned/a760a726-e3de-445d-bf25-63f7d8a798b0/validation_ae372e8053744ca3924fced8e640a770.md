### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` from the pool's perspective. When any user routes through `MetricOmmSimpleRouter`, the pool sees the router as `msg.sender`, so the extension evaluates `allowedSwapper[pool][router]`. If the pool admin allowlists the router — a natural step to enable router-mediated swaps for their permitted users — every unpermitted user can bypass the allowlist by routing through the router.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`, which forwards it unchanged to every configured extension. [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [7](#0-6) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` — making the router `msg.sender` at the pool level: [8](#0-7) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. If the pool admin has allowlisted the router address (intending to permit router-mediated swaps for their approved users), the check passes for **every** caller regardless of their individual allowlist status.

The same substitution occurs for `exactInput` (all hops use `address(this)` = router as payer/caller) and `exactOutputSingle`/`exactOutput`. [9](#0-8) 

There is no mechanism in the extension or the pool to recover the original EOA once the router is the direct caller. The `extensionData` bytes are caller-controlled and cannot be trusted as an identity proof.

### Impact Explanation
Any unpermitted user can swap in a pool that has `SwapAllowlistExtension` active and the router allowlisted. The allowlist — the sole access-control gate on the swap path — is fully bypassed. Pools designed for restricted participants (e.g., institutional, KYC-gated, or partner-only pools) become open to the public. This is a direct admin-boundary break: an unprivileged path (routing through the public router) defeats a pool-admin-configured guard.

### Likelihood Explanation
The pool admin must allowlist the router. This is the natural and expected action when a pool admin wants their approved users to be able to use the standard periphery router rather than calling the pool directly. The admin's mental model — "I allowlist the router so my users can route through it" — is incorrect; the actual effect is "I allowlist every user who routes through it." The misunderstanding is easy to make because the router is a trusted, factory-registered contract, and the allowlist admin API (`setAllowedToSwap`) gives no indication that allowlisting an intermediary opens the gate to all downstream callers.

### Recommendation
The `SwapAllowlistExtension` should gate the **economically relevant actor** — the end user — not the intermediary. Two viable approaches:

1. **Router-forwarded identity**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have the extension decode and verify it (requires a trusted router registry or signature scheme to prevent spoofing).
2. **Direct-only allowlist**: Document clearly that `SwapAllowlistExtension` only works for direct pool calls, and that allowlisting the router defeats the gate. Pools that need router support should allowlist individual users, not the router contract.

### Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension at extension slot 1,
   BEFORE_SWAP_ORDER = extension 1.

2. Admin allowlists the router:
   swapExtension.setAllowedToSwap(pool, address(router), true);
   // Intent: "let my approved users route through the router"

3. Admin does NOT allowlist attacker:
   swapExtension.isAllowedToSwap(pool, attacker) == false

4. Attacker calls:
   router.exactInputSingle(ExactInputSingleParams({
       pool: pool,
       recipient: attacker,
       zeroForOne: true,
       amountIn: X,
       ...
   }));

5. Pool calls extension.beforeSwap(router, attacker, ...):
   allowedSwapper[pool][router] == true  →  check passes

6. Attacker receives output tokens. Allowlist fully bypassed.
``` [3](#0-2) [10](#0-9)
