### Title
Deposit Allowlist Bypassed via Attacker-Controlled `owner` Parameter — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument and gates only on `owner`. Because `owner` is a free caller-controlled parameter in `MetricOmmPool.addLiquidity`, any non-allowlisted address can deposit by nominating an allowlisted address as `owner`, fully bypassing the pool's deposit curation.

### Finding Description

**Call path:**

```
attacker.call → pool.addLiquidity(owner=allowlistedAddr, ...)
  → _beforeAddLiquidity(sender=attacker, owner=allowlistedAddr)
    → DepositAllowlistExtension.beforeAddLiquidity(/*sender ignored*/, owner=allowlistedAddr)
      → allowedDepositor[pool][allowlistedAddr] == true → PASSES
  → LiquidityLib.addLiquidity(owner=allowlistedAddr, ...)   // LP shares → allowlistedAddr
  → callback to msg.sender (attacker) to collect tokens     // attacker pays
```

In `MetricOmmPool.addLiquidity`, `owner` is a raw caller-supplied parameter with no restriction: [1](#0-0) 

`_beforeAddLiquidity` faithfully forwards both `sender` (= `msg.sender`, the actual caller) and `owner` to the extension: [2](#0-1) 

But `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter unnamed — it is silently discarded — and checks only `owner`: [3](#0-2) 

The check `allowedDepositor[msg.sender][owner]` (where `msg.sender` is the pool and `owner` is the attacker-chosen allowlisted address) passes, so the extension returns its selector and the deposit proceeds.

### Impact Explanation

- **Pool curation is broken**: The pool admin's invariant — "only allowlisted actors may inject tokens into the pool" — is violated. Any EOA or contract can deposit into a curated pool.
- **LP shares are minted to `allowlistedAddr`**: The attacker pays the tokens (via the `addLiquidity` callback to `msg.sender`) but LP shares are credited to the allowlisted address they nominated. The attacker loses funds; the allowlisted address gains LP shares for free without having initiated the deposit.
- **Pool accounting remains correct**: The pool receives the tokens and mints matching LP shares, so there is no insolvency. However, the allowlisted address now holds LP shares it did not request or pay for, and the pool's bin composition is altered by an unauthorized actor.
- **Severity ceiling**: Because the attacker self-harms (pays tokens, receives nothing) and the pool's token/share accounting is not broken, this does not meet the **High** direct-fund-loss threshold. It is a confirmed **Medium** — broken core pool functionality (allowlist bypass) with a real but indirect economic consequence (unauthorized bin injection, unsolicited LP share minting to third parties).

### Likelihood Explanation

Exploitable by any EOA with no special privileges. The only prerequisite is knowing one allowlisted address for the target pool, which is on-chain public state via `allowedDepositor`.

### Recommendation

Check `sender` (the actual depositor/payer) instead of `owner` (the LP share recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, gate on both `sender` and `owner` if the intent is that the LP share recipient must also be allowlisted.
