### Title
`ProtectedPriceProviderL2` Missing Sequencer Uptime Check Allows Bad-Price Swaps After Sequencer Restart - (File: `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

### Summary

`ProtectedPriceProviderL2` is the production L2 price provider for the Metric OMM protocol. It performs staleness checks via `MAX_TIME_DELTA` and `FUTURE_TOLERANCE`, but contains **no check against a Chainlink sequencer uptime feed**. The protocol README explicitly lists "swaps revert if... (L2) sequencer down" as a hard invariant, and the contract registry ABI contains a `ChainlinkVerifierL2` contract with `sequencerUptimeFeed` and `GRACE_PERIOD` fields — but no source file for it exists, and it is not integrated into `ProtectedPriceProviderL2`. The invariant is stated but not enforced in code.

### Finding Description

`ProtectedPriceProviderL2._computeBidAsk()` runs the following guards in order:

1. `_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)` — rejects prices older than `MAX_TIME_DELTA` or more than `FUTURE_TOLERANCE` seconds in the future.
2. `mid == 0 || spread >= ORACLE_BPS` — rejects zero price or stall marker.
3. `priceGuard` bounds check.
4. Confidence/marginStep shaping. [1](#0-0) 

There is no step that queries a sequencer uptime feed (`AggregatorV3Interface`) or enforces a grace period after sequencer restart. The `FUTURE_TOLERANCE` field is explicitly documented as handling **clock skew** (oracle `refTime` slightly ahead of `block.timestamp`), not sequencer downtime: [2](#0-1) 

The registry ABI confirms a `ChainlinkVerifierL2` contract with `sequencerUptimeFeed()` and `GRACE_PERIOD()` was designed but never integrated: [3](#0-2) 

`PriceProviderL2` (the superseded predecessor) is structurally identical — it also has no sequencer uptime check — confirming the check was never implemented in either L2 provider: [4](#0-3) 

The README states the invariant explicitly: [5](#0-4) 

### Impact Explanation

When the Arbitrum sequencer goes down, oracle price updates (pushed by off-chain keepers) stop being processed. When the sequencer restarts, there is a window — bounded by `MAX_TIME_DELTA` — during which the last pre-downtime price is still in oracle storage and passes the staleness check. During this window, a trader can execute swaps at the stale pre-downtime price. If the true market price moved significantly during the outage (e.g., a sharp ETH drop), the trader can buy at the old high price from LPs who cannot rebalance, extracting value directly from LP principal. This is a **bad-price execution** impact: the pool settles a trade at a price the oracle/bin curve should have rejected.

### Likelihood Explanation

Arbitrum has experienced sequencer downtime historically. The attack requires: (1) sequencer downtime, (2) meaningful price movement during the outage, (3) sequencer restart, (4) a swap executed before oracle keepers push a fresh price. Steps 3 and 4 are trivially achievable by any public trader monitoring the sequencer uptime feed. The window is bounded by `MAX_TIME_DELTA` (up to 7 days per the constructor bound), but even a short window of minutes is sufficient for a prepared attacker. Likelihood is **medium** (requires external sequencer event, but the attack is fully permissionless once it occurs).

### Recommendation

Integrate a sequencer uptime check into `ProtectedPriceProviderL2._computeBidAsk()` using the already-designed `ChainlinkVerifierL2` pattern. Before accepting any oracle price, call `sequencerUptimeFeed.latestRoundData()`, verify `answer == 0` (sequencer is up), and enforce that `block.timestamp - startedAt >= GRACE_PERIOD` (e.g., 3600 seconds) to ensure oracle prices have had time to update after a restart. Revert (fail closed) if either condition fails.

```solidity
// In ProtectedPriceProviderL2._computeBidAsk(), before staleness check:
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD) {
    return (0, type(uint128).max); // fail closed
}
```

### Proof of Concept

1. Deploy `ProtectedPriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600` (1 hour).
2. Oracle stores price `P0 = 3000e8` (ETH/USD) at `T0`.
3. Arbitrum sequencer goes down at `T0 + 1`. True market price drops to `P1 = 2000e8`.
4. Sequencer restarts at `T0 + 1800` (30 minutes later). Oracle storage still holds `P0`.
5. Attacker calls `pool.swap()` immediately. `ProtectedPriceProviderL2._computeBidAsk()` runs:
   - `_isStale(T0, T0+1800, 3600, futureTol)` → `1800 <= 3600` → **not stale, passes**.
   - No sequencer check → **no revert**.
   - Pool quotes bid/ask around `P0 = 3000e8`.
6. Attacker buys ETH from the pool at `~3000` when true price is `~2000`, extracting ~$1000 per ETH from LP principal.
7. Oracle keepers push `P1` shortly after; the window closes, but the loss is already realized. [6](#0-5) [7](#0-6)

```text
    // ── Staleness ───────────────────────────────────────────────────────

    /// @dev Pure staleness check. L2-aware: tolerates oracle refTime slightly ahead of block.timestamp.
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```

```text
    function _computeBidAsk(uint256 price, uint256 spread, uint256 refTime)
        internal view returns (uint128, uint128)
    {
        // 1. Staleness check
        if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
```
