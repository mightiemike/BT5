### Title
`SwapAllowlistExtension` gates the router address instead of the end-user when swaps are routed through `MetricOmmSimpleRouter`, allowing any user to bypass a curated pool's swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which equals `msg.sender` of the `pool.swap(...)` call. When `MetricOmmSimpleRouter` intermediates the swap, `msg.sender` to the pool is the router contract, not the end user. If the pool admin allowlists the router (required for any router-based swap to succeed), every user who routes through `MetricOmmSimpleRouter` bypasses the per-user swap allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the argument forwarded by the pool from its own `msg.sender`. The pool's `swap` function calls `_beforeSwap(msg.sender, recipient, ...)`, so `sender` = whoever called `pool.swap(...)`.

When `MetricOmmSimpleRouter` executes a swap on behalf of a user, it calls `pool.swap(...)` directly. The pool therefore sees `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Bypass path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Pool admin allowlists specific counterparties: `allowedSwapper[pool][marketMaker] = true`.
3. Pool admin also allowlists the router (`allowedSwapper[pool][router] = true`) so that allowlisted users can access the pool through the router for better UX or multi-hop routing.
4. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInput(...)` targeting this pool.
5. The router calls `pool.swap(...)` → pool passes `sender = router` to the extension.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. Non-allowlisted user successfully swaps against the curated pool.

The pool admin has no way to allow router-based swaps for their allowlisted users while simultaneously blocking non-allowlisted users from using the same router path, because the extension cannot see past the router to the originating user.

The `DepositAllowlistExtension` does not share this exact flaw — it checks `owner` (the position beneficiary), not `sender` (the payer), so the deposit allowlist correctly gates the economic recipient regardless of routing.

---

### Impact Explanation

A curated pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., a private market-making pool, an institutional-only venue, or a pool with a specific risk profile) can be accessed by any arbitrary user through `MetricOmmSimpleRouter`. This allows unauthorized users to trade against the pool's liquidity at oracle-derived prices the pool admin intended only for specific parties, directly extracting LP value and breaking the pool's curation invariant. The loss is direct and proportional to the pool's liquidity depth and the spread between oracle price and fair market value.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router — a reasonable operational step any admin would take to enable router-based swaps for their legitimate users. Once the router is allowlisted (which is necessary for any router-mediated swap to work on the pool), the bypass is available to every user with access to `MetricOmmSimpleRouter`. No special privileges, flash loans, or unusual token behavior are required. Any user can call the router's public `exactInput` / `exactOutput` functions.

---

### Recommendation

The `SwapAllowlistExtension` must gate the **originating user**, not the immediate caller of `pool.swap`. Two approaches:

**Option A — Pass the originating user through `extensionData`:** The router encodes the end-user address in `extensionData`; the extension decodes and checks it. This requires the pool to trust that the router correctly reports the originator, which introduces a trust assumption on the router.

**Option B — Check `sender` against a router registry and then verify the originator via a callback:** The extension, upon seeing a known router as `sender`, calls back into the router to retrieve the originating user and checks that address against the allowlist.

**Option C (simplest and most robust) — Do not allowlist the router; instead, require allowlisted users to call `pool.swap` directly.** Document that router-based swaps are incompatible with `SwapAllowlistExtension` pools. This is a breaking UX constraint but preserves the security invariant without code changes.

The core invariant that must hold: **every guard must key authorization to the same actor that the economic action is actually attributed to** — the end user, not the routing intermediary.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension in beforeSwap slot
  allowedSwapper[pool][marketMaker] = true
  allowedSwapper[pool][router]      = true   ← admin adds this to enable router UX
  allowedSwapper[pool][attacker]    = false

Attack:
  attacker calls MetricOmmSimpleRouter.exactInput({
      pool: curated_pool,
      zeroForOne: true,
      amountIn: large_amount,
      ...
  })

  Router calls pool.swap(recipient=attacker, ...)
  Pool calls _beforeSwap(sender=router, ...)
  Extension checks allowedSwapper[pool][router] → true → no revert
  Swap executes; attacker receives output tokens

Result:
  attacker bypassed the per-user allowlist and traded against the curated pool
  LP funds extracted at oracle prices intended only for allowlisted counterparties
