### Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Curated-Pool Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` = router address. If the pool admin allowlists the router (the natural action to enable router-mediated swaps for allowlisted users), every unprivileged user can bypass the swap allowlist by routing through the public router contract.

---

### Finding Description

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the end user
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the pool:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no forwarding of the original user's identity:

```solidity
// MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
```

When Alice (allowlisted) or Bob (not allowlisted) routes through the router, the pool sees `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][Alice/Bob]`.

This creates an irresolvable dilemma for the pool admin:

| Admin action | Effect on allowlisted users | Effect on non-allowlisted users |
|---|---|---|
| Do NOT allowlist router | Cannot use router at all | Correctly blocked |
| Allowlist router | Can use router | **Also bypass allowlist via router** |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

Any user who is not on the swap allowlist of a curated pool can bypass the access control by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) once the pool admin has allowlisted the router address. The unauthorized swap executes at live oracle prices against the pool's LP liquidity, directly extracting value from LP positions that were deposited under the assumption that only vetted counterparties could trade. This is a direct loss of LP principal and a broken core pool invariant.

---

### Likelihood Explanation

The trigger is a single, expected admin action: allowlisting the router so that allowlisted users can access the pool through the standard periphery path. Any pool admin who wants to support router-mediated swaps for their allowlisted users will naturally add the router to the allowlist, at which point the bypass is immediately available to every unprivileged user. No malicious setup is required; the attacker only needs to call a public router function.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the actual end-user identity, not the direct pool caller. Two viable approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`, and the extension decodes and checks that address when `sender` is a known router.
2. **Check `sender` only when it is not a known router; otherwise decode the real user from `extensionData`**: The extension maintains a registry of trusted routers and falls back to `extensionData`-encoded identity for router-mediated calls.

The `DepositAllowlistExtension` correctly gates on `owner` (the LP share recipient), which is the economically relevant actor for deposits. The swap allowlist must apply the same principle and gate on the economically relevant actor for swaps — the end user — not the intermediary contract.

---

### Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, alice, true)      // alice is allowlisted
  pool admin: setAllowedToSwap(pool, router, true)     // router allowlisted so alice can use it

Attack (bob, not allowlisted):
  bob calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)           // msg.sender = router
    → pool calls _beforeSwap(router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → checks allowedSwapper[pool][router] == true      // ✓ passes
    → swap executes against LP liquidity
    → bob receives output tokens, LP bears the trade

Result:
  bob (non-allowlisted) successfully swaps on a curated pool
  allowlist invariant broken; LP funds exposed to unauthorized counterparties
```

**Exact corrupted value**: `allowedSwapper[pool][router]` is `true` (set by admin for legitimate router access), but the extension interprets this as authorization for every user who routes through the router, not just the router contract itself. The guard checks the wrong identity and fails open for all router-mediated swaps. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
