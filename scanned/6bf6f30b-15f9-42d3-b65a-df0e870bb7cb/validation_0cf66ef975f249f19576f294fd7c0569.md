### Title
`purgeStateGuardRole` Does Not Clear `pendingStateGuard`, Allowing a Previously Nominated Guard to Seize the Oracle stateGuard Role After Purge — (File: `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`purgeStateGuardRole` in the providers `OracleBase` deletes `stateGuard[feedId]` but leaves `pendingStateGuard[feedId]` intact. Any address that was previously nominated via `setStateGuardRole` can call `acceptStateGuardRole` after the purge and become the new stateGuard — exactly the `__abdicate`/`pendingGov` pattern from the external report. The stateGuard controls `setPriceGuard`, which gates every bid/ask quote that reaches the pool through `ProtectedPriceProvider`.

---

### Finding Description

In `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `purgeStateGuardRole` only clears the active guard:

```solidity
// line 120-124
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];          // ← only this is cleared
    // pendingStateGuard[feedId] is NOT touched
    emit StateGuardDeleted(feedId);
}
``` [1](#0-0) 

`acceptStateGuardRole` has no guard against a post-purge acceptance — it only checks that the caller matches the pending slot:

```solidity
// line 111-118
function acceptStateGuardRole(bytes32 feedId) external {
    require(pendingStateGuard[feedId] == msg.sender, InvalidGuard(msg.sender));
    delete pendingStateGuard[feedId];
    stateGuard[feedId] = msg.sender;
    emit StateGuardUpdated(feedId, msg.sender);
}
``` [2](#0-1) 

After `purgeStateGuardRole`, `stateGuard[feedId]` is `address(0)` so `checkRole` falls back to `ADMIN_ROLE` for future guard-setter calls — but `pendingStateGuard[feedId]` still holds the previously nominated address, and `acceptStateGuardRole` is permissionless (no `checkRole` guard). The nominated address can therefore call `acceptStateGuardRole` at any time and install itself as the new stateGuard. [3](#0-2) 

**Attack sequence:**

1. `guard1` (current stateGuard) calls `setStateGuardRole(feedId, guard2)` → `pendingStateGuard[feedId] = guard2`
2. `guard1` calls `purgeStateGuardRole(feedId)` intending to fully relinquish control → `stateGuard[feedId] = address(0)`, but `pendingStateGuard[feedId]` remains `guard2`
3. `guard2` calls `acceptStateGuardRole(feedId)` → `stateGuard[feedId] = guard2`; `guard2` is now the stateGuard despite `guard1`'s intent to remove all guard state
4. `guard2` calls `setPriceGuard(feedId, ...)` to manipulate the price bounds for the feed

---

### Impact Explanation

The stateGuard's sole operative power is `setPriceGuard`. The price guard is enforced inside `ProtectedPriceProvider._computeBidAsk`, which is the function that produces every bid/ask quote consumed by the pool during swaps:

```solidity
// ProtectedPriceProvider.sol line 202-206
(uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(offchainFeedId);
guardMax = guardMax == 0 ? type(uint128).max : guardMax;
if (price < guardMin || price > guardMax) {
    return (0, type(uint128).max);   // ← stalled sentinel
}
``` [4](#0-3) 

`guard2` can exploit this in two directions:

- **DoS / pool freeze:** Set `priceGuard` to a range that excludes all live oracle prices (e.g., `min = current_price + 1, max = current_price + 2`). Every call to `getBidAndAskPrice` returns the stalled sentinel `(0, type(uint128).max)`. The pool cannot execute swaps; LP withdrawals that depend on a valid quote are also blocked. This is broken core pool functionality with direct impact on user funds locked in the pool.
- **Price-guard bypass:** Set `priceGuard` to `(1, type(uint128).max - 1)`, effectively disabling the guard. Any oracle price — including a manipulated one pushed by a compromised pusher — passes the guard check and reaches the pool's swap math unclamped.

---

### Likelihood Explanation

The trigger requires `guard1` to have previously called `setStateGuardRole` (nominating `guard2`) and then called `purgeStateGuardRole` without first calling `purgePendingStateGuardRole`. This is a realistic operational mistake: a guard intending to fully relinquish control would naturally call `purgeStateGuardRole`, not realising it leaves the pending slot live. The function name gives no indication that a separate `purgePendingStateGuardRole` call is also required. `guard2` (the previously nominated address) can then call `acceptStateGuardRole` at any block after the purge — there is no time window that closes.

---

### Recommendation

Clear `pendingStateGuard` inside `purgeStateGuardRole`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
+   delete pendingStateGuard[feedId];   // prevent stale nominee from accepting
    emit StateGuardDeleted(feedId);
}
``` [1](#0-0)

function test_purgeStateGuard_doesNotClearPending_allowsStaleAccept() public {
    bytes32 feedId = bytes32(uint256(1));
    address guard1 = makeAddr("guard1");
    address guard2 = makeAddr("guard2");

    // ADMIN sets guard1 as stateGuard
    oracle.setStateGuardRole(feedId, guard1);
    vm.prank(guard1);
    oracle.acceptStateGuardRole(feedId);
    assertEq(oracle.stateGuard(feedId), guard1);

    // guard1 nominates guard2 (legitimate transfer attempt)
    vm.prank(guard1);
    oracle.setStateGuardRole(feedId, guard2);
    assertEq(oracle.pendingStateGuard(feedId), guard2);

    // guard1 decides to cancel by purging the stateGuard role entirely
    vm.prank(guard1);

    vm.prank(guard2);
    oracle.acceptStateGuardRole(feedId);
    assertEq(oracle.stateGuard(feedId), guard2);

    vm.prank(guard2);
    oracle.setPriceGuard(feedId, type(uint128).max - 1, type(uint128).max);

    // Pool swaps revert; LP funds are locked
