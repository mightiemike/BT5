### Title
Unguarded `refundETH()` allows any caller to drain excess ETH left in `MetricOmmSimpleRouter` after a WETH swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` unconditionally transfers the entire native ETH balance of the router to `msg.sender` with no access control. When a user sends excess ETH with `exactInputSingle` (e.g., 2 ETH for a 1 ETH WETH swap) and omits `refundETH()` from their multicall, the leftover ETH sits in the router and any attacker can immediately drain it.

---

### Finding Description

`refundETH()` in `PeripheryPayments` is a public, payable function with no caller restriction: [1](#0-0) 

It sends `address(this).balance` to `msg.sender` — whoever calls it, not the original ETH depositor.

ETH enters the router legitimately via `exactInputSingle{value: X}(...)` (and the other payable swap functions). Inside the swap callback, `pay()` wraps only the exact `value` needed: [2](#0-1) 

Any ETH above `value` is **not consumed** and remains in the contract after `exactInputSingle` returns: [3](#0-2) 

The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does **not** prevent ETH from arriving via `msg.value` on a payable call: [4](#0-3) 

The intended safe pattern is `multicall([exactInputSingle(...), refundETH()])` in one transaction. If the user calls `exactInputSingle` directly (not via multicall) or forgets to append `refundETH()`, the excess ETH is stranded and claimable by anyone.

---

### Impact Explanation

Direct loss of user principal. The attacker receives native ETH that belongs to the user. No privileged role is required — the attacker only needs to observe the mempool (or even just call `refundETH()` in a subsequent block) and call the public function. The loss equals the full excess ETH the user sent.

---

### Likelihood Explanation

Users interacting with WETH swap paths commonly send a round-number ETH value slightly above the required `amountIn` to avoid slippage-induced reverts. Any such transaction that does not bundle `refundETH()` in the same multicall is immediately exploitable. Front-running or back-running the victim's transaction is trivial.

---

### Recommendation

Restrict `refundETH()` so it can only be called from within a `multicall` context (i.e., via `delegatecall` from the router itself), or record the original ETH sender in transient storage at the start of each payable entry point and enforce that only that address can call `refundETH()`. The simplest fix consistent with the existing multicall design is to add a `msg.sender == address(this)` guard (since `multicall` uses `delegatecall`, `msg.sender` inside the delegated call is the original caller, but a direct external call has `msg.sender != address(this)`):

```solidity
function refundETH() external payable override {
    // Only callable through multicall (delegatecall preserves msg.sender == original caller,
    // but a direct external call has address(this) != msg.sender only when not delegated).
    // Better: record payer in transient storage and enforce it here.
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The cleanest fix is to store the initiating `msg.sender` in transient storage at the top of each payable swap entry point and require `msg.sender == _storedInitiator` inside `refundETH()`.

function test_refundETH_drainByAttacker() public {
    // User sends 2 ETH for a 1 ETH WETH exactInputSingle swap (no multicall, no refundETH)
    router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
        pool: wethPool,
        tokenIn: WETH,
        recipient: user,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        zeroForOne: true,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));

    // 1 ETH remains in router after swap consumed exactly 1 ETH
    assertEq(address(router).balance, 1 ether);

    // Attacker (unrelated address) calls refundETH
    vm.prank(attacker);
    router.refundETH();

    // Attacker received the user's excess ETH
    assertEq(attacker.balance, 1 ether);
    assertEq(address(router).balance, 0);
}
