### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the router is `msg.sender`, so the extension evaluates the router's allowlist entry rather than the actual user's. A pool admin who allowlists the router (a natural configuration for "allow all periphery users") inadvertently opens the gate to every address, defeating the curation the extension was deployed to enforce.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces its guard as:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct pool-key scoping). `sender` is the first positional argument the pool passes when it calls the extension hook — and that argument is `msg.sender` of the originating `pool.swap()` call.

The `FullMetricExtensionTest` confirms this binding: the test allowlists `callers[0]` (the `TestCaller` contract that directly calls `pool.swap()`), not `users[0]` (the EOA behind it):

```solidity
// metric-periphery/test/extensions/FullMetricExtension.t.sol L70
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
// L73
_swap(0, users[0], false, int128(1000), type(uint128).max);
```

When a user instead routes through `MetricOmmSimpleRouter`:

```
user EOA → MetricOmmSimpleRouter.exactInput*()
         → pool.swap(recipient, ...)   [msg.sender = router]
         → extension.beforeSwap(router, ...)
         → allowedSwapper[pool][router]  ← checked, NOT the user
```

Two fund-impacting outcomes follow:

**Bypass path (primary impact):** A pool admin who allowlists the router address — a natural choice when the intent is "allow all users coming through the official periphery" — grants every address the ability to swap, because the extension never inspects the actual user. Any non-allowlisted EOA routes through the router and the check passes.

**Broken-functionality path (secondary impact):** A pool admin who allowlists individual user addresses but not the router blocks those same users from using the standard periphery path, making the pool's swap flow unusable for legitimate participants.

The same structural mismatch exists for `DepositAllowlistExtension`: the first (unnamed) argument to `beforeAddLiquidity` is the caller of `pool.addLiquidity()`, which becomes the liquidity adder contract when `MetricOmmPoolLiquidityAdder` is used, not the economic depositor. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. If the router is allowlisted (or if `allowAllSwappers` is set for the pool), the guard is permanently open to the public. Unauthorized traders can drain liquidity, execute arbitrage, or interact with pools that were designed to be private, resulting in direct loss of LP principal and protocol fees.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point documented and deployed by the protocol. Any pool admin who configures an allowlist and then allowlists the router (or who expects users to use the router without understanding the identity substitution) triggers the bypass. The router is a public, permissionless contract, so no privileged access is required. The trigger is a normal `exactInput` or `exactOutput` call.

---

### Recommendation

The pool should pass the **originating user** as `sender` to extension hooks, not `msg.sender` of the immediate caller. Two concrete approaches:

1. **Caller-supplied sender**: Add an explicit `sender` parameter to `pool.swap()` that the router populates with `msg.sender` before forwarding to the pool. The pool passes this value to extensions rather than its own `msg.sender`.

2. **Extension reads `tx.origin` or a trusted forwarder pattern**: Extensions verify the actual user through a trusted context rather than the pool-supplied argument. This is more complex but avoids relying on the pool's argument binding.

The `DepositAllowlistExtension` should apply the same fix: gate on the economic depositor (the address that will own the LP position and whose funds are at risk), not on the intermediate contract that called `pool.addLiquidity()`.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: swapExtension.setAllowedToSwap(pool, router, true)
    (intending to allow all periphery users)

Attack:
  attacker = address not in allowedSwapper[pool]
  attacker calls: router.exactInputSingle({pool: pool, ...})

Execution trace:
  router.exactInputSingle()
    → pool.swap(attacker_recipient, ...)   // msg.sender = router
    → extension.beforeSwap(router, ...)    // sender = router
    → allowedSwapper[pool][router] == true // PASSES
    → swap executes for attacker

Result:
  attacker swaps in a pool they were never individually allowlisted for.
  The allowlist guard is silently bypassed for every user who routes
  through the router, because the extension only ever sees the router address.
