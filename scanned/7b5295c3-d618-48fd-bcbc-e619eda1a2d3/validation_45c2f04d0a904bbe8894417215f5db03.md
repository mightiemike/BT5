### Title
`addLiquidity()` Bypasses `whenNotPaused` Guard, Allowing Deposits Into a Paused Pool — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

The `whenNotPaused` modifier is applied exclusively to `swap()`, leaving `addLiquidity()` unguarded. When the pool is paused via `setPause()`, users can still deposit principal into a pool that may be in a compromised or exploited state, directly losing funds.

---

### Finding Description

`MetricOmmPool` defines a `pauseLevel` state variable and a `_checkNotPaused()` guard:

```solidity
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
uint8 internal pauseLevel;

function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [1](#0-0) [2](#0-1) 

The `whenNotPaused` modifier is applied only to `swap()`:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
``` [3](#0-2) 

But `addLiquidity()` carries no such guard:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
``` [4](#0-3) 

The `_beforeAddLiquidity` extension hook (e.g., `DepositAllowlistExtension`) is still invoked, but the pause itself does not block the deposit path. The pool admin's emergency pause is therefore **misapplied**: it halts swaps but leaves the deposit surface fully open.

This is the direct analog to the external report: the `pauseLevel` field exists and is settable, but the guard it is supposed to enforce is wired only to one of the three fund-moving entry points.

---

### Impact Explanation

When a pool is paused due to a price manipulation, oracle failure, or active exploit, users can still call `addLiquidity()` and transfer tokens into the compromised pool. Those tokens are immediately subject to the same condition that triggered the pause (e.g., a bad oracle price draining the bin). This is a **direct loss of user principal** with no recourse, because the pool accepted the deposit without reverting.

---

### Likelihood Explanation

The pool is paused precisely when the risk of loss is highest. A user who observes that swaps are reverting with `PoolPaused` has no on-chain signal that deposits are still open. Any user or integrator that calls `addLiquidity()` during a pause window loses funds. The trigger (a paused pool) is a semi-trusted admin action, but the victim path (deposit) requires no privilege.

---

### Recommendation

Apply `whenNotPaused` to `addLiquidity()`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

If the design intent is to allow LP exits during a pause, `removeLiquidity()` can remain unguarded, but new deposits must be blocked.

---

### Proof of Concept

1. Pool admin calls `factory.setPause(pool, 1)` — pool enters paused state.
2. Any call to `swap()` reverts with `PoolPaused`.
3. Attacker (or innocent user) calls `pool.addLiquidity(owner, salt, deltas, callbackData, extensionData)` — **succeeds**, tokens transferred in.
4. Pool is in a compromised state (e.g., oracle returning a manipulated price); the deposited tokens are immediately at risk.
5. Attacker drains the pool via a mechanism that does not require `swap()` (e.g., a subsequent oracle correction that shifts bin accounting), or simply waits for the pause to lift and executes a favorable swap against the newly deposited liquidity. [5](#0-4) [6](#0-5)
