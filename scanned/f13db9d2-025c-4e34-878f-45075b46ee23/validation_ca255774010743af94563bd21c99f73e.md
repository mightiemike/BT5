### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is intended to gate who can add liquidity to a pool. However, it validates the `owner` parameter (the position beneficiary) rather than `sender` (the actual caller). Because `owner` is a free caller-supplied argument in `pool.addLiquidity(owner, ...)`, any unprivileged address can bypass the allowlist by specifying an allowlisted address as `owner` while paying the tokens themselves via the swap callback.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes both `msg.sender` (as `sender`) and the caller-supplied `owner` to the extension hook: [1](#0-0) 

The extension receives `sender` as its first argument but silently discards it (unnamed `address,`), then checks only `owner`: [2](#0-1) 

Because `owner` is a free parameter that any caller can set to any address, an unauthorized depositor can call:

```solidity
pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData);
```

The extension sees `owner = allowlistedAddress` → passes. The unauthorized caller pays the tokens via the `metricOmmSwapCallback` on `msg.sender`. Liquidity is credited to the allowlisted owner's position, but the actual depositor was never checked.

The contract's own NatSpec and setter name make the intent clear — it is supposed to gate the depositor (the actor adding funds), not the position owner: [3](#0-2) 

The `SwapAllowlistExtension`, by contrast, correctly checks `sender`: [4](#0-3) 

---

### Impact Explanation

Any address — including sanctioned, non-KYC'd, or otherwise restricted addresses — can add liquidity to a pool that has the `DepositAllowlistExtension` configured, completely nullifying the pool admin's access control. The unauthorized depositor pays the tokens, so there is no direct theft, but:

- The pool admin's regulatory or compositional restriction is violated.
- Liquidity is injected into bins the pool admin did not intend to allow, potentially altering the pool's price curve and harming existing LPs.
- The authorized `owner` receives LP shares they did not request; while they can withdraw them, the pool's liquidity composition has already been altered.

This is an **admin-boundary break**: an unprivileged path bypasses a pool-admin-configured guard.

---

### Likelihood Explanation

The bypass requires only a standard `addLiquidity` call with a known allowlisted address as `owner`. No special permissions, flash loans, or complex setup are needed. Any observer of on-chain allowlist events (`AllowedToDepositSet`) can identify valid `owner` values and exploit immediately. Likelihood is **high** whenever the extension is deployed and configured.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`. [4](#0-3) 

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. `_beforeAddLiquidity(bob, alice, ...)` is called; the extension checks `allowedDepositor[pool][alice]` → `true` → passes.
4. `LiquidityLib.addLiquidity` credits the position to `alice`; the callback fires on `bob`, who pays the tokens.
5. `bob` has successfully deposited into a pool that was supposed to deny him, bypassing the allowlist entirely. [2](#0-1) [5](#0-4)
