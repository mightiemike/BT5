### Title
`SwapAllowlistExtension` gates on the router address instead of the actual user, allowing any user to bypass per-user swap curation by routing through `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the actual user. If the pool admin allowlists the router (required for router-mediated swaps to function), every user â€” including those not individually allowlisted â€” can bypass the per-user swap gate by routing through the router.

---

### Finding Description

**Actor binding in `SwapAllowlistExtension`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

**What the pool passes as `sender`:**

`MetricOmmPool.swap` passes `msg.sender` (the direct caller of `pool.swap`) as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- this is the router when routed
    recipient,
    ...
);
``` [2](#0-1) 

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender` of the pool's `swap` call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
``` [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**The bypass chain:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses (e.g., Alice, Bob).
2. Pool admin also allowlists the `MetricOmmSimpleRouter` address so that allowlisted users can swap through the router.
3. Non-allowlisted user (Eve) calls `router.exactInputSingle({pool: curatedPool, ...})`.
4. Router calls `curatedPool.swap(...)` â€” pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. `SwapAllowlistExtension` checks `allowedSwapper[pool][router]` â†’ **passes** because the router is allowlisted.
7. Eve successfully swaps on the curated pool, bypassing the per-user allowlist entirely.

The `FullMetricExtensionTest` confirms the actor binding: it allowlists `address(callers[0])` (the `TestCaller` contract that directly calls `pool.swap`), not `users[0]` (the human user), demonstrating that the extension sees the direct caller, not the economic actor. [5](#0-4) 

---

### Impact Explanation

**Direct curation failure / High.** A pool admin who configures `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, institutional partners, or whitelisted market makers) loses that restriction entirely once the router is allowlisted. Any unprivileged user can trade on the curated pool by routing through `MetricOmmSimpleRouter`. This is a broken core pool functionality: the allowlist guard that the pool admin configured and relies upon does not enforce the intended policy on the standard periphery swap path.

---

### Likelihood Explanation

**High.** The `MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol. Pool admins who want to allow router-mediated swaps for their allowlisted users must allowlist the router. This is the expected operational pattern. Once the router is allowlisted, the bypass is unconditional and requires no special setup from the attacker â€” any user simply calls the router's standard `exactInputSingle` or `exactInput` functions.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the **economic actor** (the user who initiated the transaction), not the direct caller of `pool.swap`. Two viable approaches:

1. **Router-side fix**: Modify `MetricOmmSimpleRouter` to encode the original `msg.sender` into `extensionData` for each hop, and modify `SwapAllowlistExtension.beforeSwap` to decode and check that address when present (falling back to `sender` for direct pool calls).

2. **Extension-side fix**: Change the allowlist to gate on `sender` only for direct pool calls, and require the router to be separately ungated (i.e., do not allowlist the router; instead, require users to call the pool directly for curated pools). Document this constraint explicitly.

Note that `DepositAllowlistExtension` does **not** have this problem â€” it correctly checks `owner` (the position owner, the economic actor), not `sender` (the direct caller), which is the right binding for the deposit path. [6](#0-5)

### Proof of Concept

```solidity
// Scenario: curated pool with SwapAllowlistExtension
// Pool admin allowlists Alice and the router (so Alice can use the router)
swapExtension.setAllowedToSwap(address(pool), alice, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Eve is NOT allowlisted
assertFalse(swapExtension.isAllowedToSwap(address(pool), eve));

// Eve bypasses the allowlist by routing through the router
vm.prank(eve);
// This succeeds because pool sees msg.sender = router (allowlisted),
// not eve (not allowlisted)
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(pool),
    tokenIn:         address(token0),
    tokenOut:        address(token1),
    zeroForOne:      true,
    amountIn:        1_000,
    amountOutMinimum: 0,
    recipient:       eve,
    deadline:        block.timestamp + 1,
    priceLimitX64:   0,
    extensionData:   ""
}));
// Eve receives token1 output â€” allowlist bypassed
```

The root cause is at `SwapAllowlistExtension.sol:37` where `sender` (= router) is checked instead of the originating user, combined with `MetricOmmPool.sol:231` where `msg.sender` (= router) is passed as `sender` to the extension, and `MetricOmmSimpleRouter.sol:72-80` where the router calls `pool.swap` directly without forwarding the original user's identity.
