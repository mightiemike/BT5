### Title
Missing Minimum-Output Protection on `removeLiquidity` Enables Sandwich Extraction — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.removeLiquidity` carries no minimum-output parameters and no deadline. Because the amounts returned depend on the live bin cursor position — which is shifted by swaps — an attacker can sandwich any `removeLiquidity` call, move the cursor to an unfavorable position, let the withdrawal execute at the manipulated state, then reverse the swap and pocket the difference. No periphery wrapper exists that adds this protection for withdrawals.

---

### Finding Description

`addLiquidity` is protected by `MetricOmmPoolLiquidityAdder`, which stores `maxAmountToken0` / `maxAmountToken1` in transient storage and enforces them inside the callback: [1](#0-0) 

`removeLiquidity` has no equivalent wrapper. The pool exposes it as a bare function with only a `nonReentrant` guard: [2](#0-1) 

The function signature accepts `owner`, `salt`, `deltas`, and `extensionData` — no `minAmount0Out`, no `minAmount1Out`, no `deadline`. The returned `amount0Removed` / `amount1Removed` are computed from the live bin state at execution time: [3](#0-2) 

Each bin's token0/token1 balances change with every swap that crosses or partially fills that bin. A zeroForOne swap drains token0 from a bin and deposits token1; reversing it restores the original state. Between those two swaps, any `removeLiquidity` call on that bin receives a token1-heavy (lower-value) mix instead of the balanced mix the LP expected.

The `MetricOmmSimpleRouter` only wraps swaps: [4](#0-3) 

`MetricOmmPoolLiquidityAdder` only wraps `addLiquidity`: [5](#0-4) 

No periphery contract wraps `removeLiquidity`. LPs must call the pool directly and have no on-chain mechanism to bound their output.

---

### Impact Explanation

An LP withdrawing from a bin receives proportional shares of that bin's current token0 and token1 balances. A sandwich swap shifts those balances before the withdrawal and restores them after, extracting value from the LP. The LP receives a worse token mix with no revert path because no minimum-output check exists anywhere in the call stack. This is a direct loss of user principal, qualifying as a High/Medium impact under the contest gate (direct loss of LP assets above Sherlock thresholds).

---

### Likelihood Explanation

`removeLiquidity` is a public function callable by any position owner. Every withdrawal is visible in the mempool. The attack requires only two swaps bracketing the victim's transaction — a standard sandwich pattern executable by any MEV searcher. No special privilege or setup is needed. Likelihood is high on any chain with a public mempool.

---

### Recommendation

**Short term:** Add `minAmount0Out` and `minAmount1Out` parameters to `removeLiquidity` (either on the pool or in a new periphery wrapper) and revert if the computed amounts fall below those thresholds. Add an optional `deadline` parameter and enforce it before execution.

**Long term:** Create a `MetricOmmPoolLiquidityRemover` periphery contract mirroring `MetricOmmPoolLiquidityAdder`, which accepts user-defined output bounds and enforces them after the pool returns the actual amounts. Add tests that simulate a swap-sandwich around `removeLiquidity` and assert the call reverts when output bounds are violated.

---

### Proof of Concept

1. Alice holds shares in bin 0 of a pool (token0 = WETH, token1 = USDC). She submits `removeLiquidity` expecting ~50 WETH / 50,000 USDC.
2. Bob observes Alice's pending transaction. He front-runs with a large zeroForOne swap (WETH → USDC) that moves the cursor deep into bin 0, draining most of bin 0's WETH and filling it with USDC.
3. Alice's `removeLiquidity` executes. `LiquidityLib.removeLiquidity` computes her proportional share of the now-USDC-heavy bin. She receives ~5 WETH / 95,000 USDC — far less ETH-denominated value than expected. No minimum-output check exists to revert this: [2](#0-1) 

4. Bob back-runs with the reverse swap (USDC → WETH), restoring bin 0's composition and capturing the spread. Bob's profit equals Alice's loss minus the two spread fees paid. In a low-spread-fee pool this is net-positive for Bob.
