The vulnerability is real. Here is the complete trace:

**Call chain:**
1. `user` (non-allowlisted EOA) calls `router.exactInputSingle()`
2. Router calls `pool.swap(recipient, ...)` — pool sees `msg.sender = router`
3. Pool calls `_beforeSwap(msg.sender, ...)` — passes `sender = router`
4. Pool calls `extension.beforeSwap(sender=router, ...)` — extension sees `msg.sender = pool`, `sender = router`
5. Extension checks `allowedSwapper[pool][router]` — **router is allowlisted → passes**

The originating EOA is never checked.

---

### Title
SwapAllowlistExtension checks the router address instead of the originating EOA, allowing any user to bypass the per-pool swap allowlist via an allowlisted router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()` — the router contract, not the originating user. When `allowedSwapper[pool][router] == true` and `allowAllSwappers[pool] == false`, any unprivileged EOA can route through the allowlisted router and execute swaps the allowlist was designed to block.

### Finding Description
In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

This `sender` value (the router) is forwarded through `ExtensionCalling._beforeSwap` directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the router. The check passes whenever the router is in `allowedSwapper[pool]`, regardless of who called the router.

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap` directly — the originating EOA (`msg.sender` of the router call) is stored only in transient callback context for payment purposes, never forwarded to the pool as the swap initiator: [4](#0-3) 

### Impact Explanation
Any non-allowlisted user can execute swaps on a restricted pool by routing through `MetricOmmSimpleRouter` (or any other allowlisted contract). The allowlist invariant — *only explicitly approved addresses may swap* — is completely broken for pools that allowlist a router. Swaps drain token0/token1 LP balances, constituting direct loss of LP principal above Sherlock thresholds if the pool was configured to restrict access (e.g., private/institutional pools).

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the canonical periphery swap router and is the expected entry point for all users. Pool admins who configure `allowedSwapper[pool][router] = true` to permit normal routing while blocking specific addresses (or permitting only specific addresses) will unknowingly grant swap access to every user who can call the router. This is the default operational pattern.

### Recommendation
The pool should pass the originating caller identity — not just `msg.sender` — to extensions, or the router should forward the originating EOA explicitly. One approach: add an `originator` field to the swap call or extension data that the router populates with `msg.sender`, and have `SwapAllowlistExtension` check that field. Alternatively, the allowlist check should be performed at the router level before calling `pool.swap`, though this is weaker since it can be bypassed by calling the pool directly.

### Proof of Concept
```solidity
// Foundry integration test sketch
function test_allowlist_bypass_via_router() public {
    // Setup: pool with SwapAllowlistExtension, allowAllSwappers=false
    // allowedSwapper[pool][address(router)] = true
    // allowedSwapper[pool][user] = false (user is NOT allowlisted)

    vm.startPrank(user); // user is NOT in allowedSwapper[pool]
    uint256 balBefore = token1.balanceOf(user);

    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: user,
        tokenIn: address(token0),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        deadline: block.timestamp,
        extensionData: ""
    }));

    // Assert: swap succeeded despite user not being allowlisted
    assertGt(token1.balanceOf(user), balBefore);
    vm.stopPrank();
}
```

The pool passes `msg.sender = router` to `beforeSwap` as `sender`; the extension checks `allowedSwapper[pool][router]` which is `true`, so the revert never fires.
