### Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates the Wrong Actor — Payer Is Never Checked, Allowlist Bypassed via Operator Pattern - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the actual token payer and only validates the `owner` (position recipient). Because the pool's `addLiquidity` explicitly supports an operator pattern where `msg.sender` (payer) need not equal `owner`, any unprivileged address can deposit into a curated pool by supplying an allowed `owner` address, bypassing the deposit gate entirely.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives five parameters. The first — the actual payer, i.e. the `msg.sender` of `pool.addLiquidity` — is unnamed and discarded. Only the second parameter, `owner` (the position recipient), is checked against the allowlist:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The pool's own NatSpec for `addLiquidity` documents the operator pattern explicitly:

> `msg.sender` pays but need not equal `owner` (operator pattern). [2](#0-1) 

Because the extension ignores the first parameter entirely, the allowlist check is keyed only on who *receives* the LP position, not on who *pays* for it. Any address — including one explicitly excluded from the allowlist — can call `pool.addLiquidity(allowedOwner, salt, deltas, ...)` and the hook will pass, because `allowedOwner` is on the allowlist. The unauthorized payer's identity is never evaluated.

The `MetricOmmPoolLiquidityAdder` periphery contract amplifies this: it calls `pool.addLiquidity` with a caller-supplied `owner`, making the payer/owner separation trivially accessible to any public user without writing custom code. [3](#0-2) 

This is the direct structural analog to the external vault bug: the vault bug checked per-vault limits instead of the aggregate product limit (wrong level of granularity); here the extension checks the position recipient instead of the token depositor (wrong actor in the same call).

---

### Impact Explanation

The deposit allowlist is a pool-admin–configured curation boundary. Its stated purpose is to gate `addLiquidity` by depositor address. Because the payer is never checked, any unprivileged address can deposit tokens into a curated pool without appearing on the allowlist, as long as it names an allowed address as `owner`. The pool admin's access-control boundary is bypassed by a standard, publicly available call path. This constitutes an admin-boundary break: a pool-admin–enforced restriction is circumvented by an unprivileged actor through a supported periphery entrypoint.

Downstream consequences include unauthorized participation in compliance-gated or KYC-gated pools, and the ability to force liquidity into a curated pool from an unvetted source, which can distort bin balances and affect subsequent swap pricing against honest LPs.

---

### Likelihood Explanation

The operator pattern (`msg.sender != owner`) is a documented, first-class feature of the pool. Any user who knows a single allowed address (e.g., by reading on-chain allowlist events) can exploit this immediately with a direct pool call or through the public `MetricOmmPoolLiquidityAdder`. No privileged access, flash loan, or special token behavior is required. The `MetricOmmPoolLiquidityAdder` makes the payer/owner split trivially accessible to any EOA. [1](#0-0) 

---

### Recommendation

The `beforeAddLiquidity` hook must validate the actual payer — the first (currently unnamed) parameter — rather than, or in addition to, `owner`. The corrected check should be:

```solidity
function beforeAddLiquidity(address payer, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    address pool = msg.sender;
    if (!allowAllDepositors[pool]
        && !allowedDepositor[pool][payer]   // gate the token sender
        && !allowedDepositor[pool][owner])  // optionally also gate the recipient
    {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The exact policy (gate payer, gate owner, or gate both) should match the pool admin's intent, but the payer must not be silently ignored.

---

### Proof of Concept

1. Pool admin deploys a curated pool with `DepositAllowlistExtension` configured. Only `allowedOwner` is on the allowlist; `attacker` is not.
2. `attacker` calls `pool.addLiquidity(allowedOwner, salt, deltas, callbackData, extensionData)` directly.
3. The pool calls `extension.beforeAddLiquidity(attacker, allowedOwner, salt, deltas, extensionData)`.
4. The extension evaluates `allowedDepositor[pool][allowedOwner]` → `true`. The hook returns without reverting.
5. The pool invokes `IMetricOmmModifyLiquidityCallback` on `attacker` (the `msg.sender`), pulling tokens from `attacker`.
6. LP shares are minted to `allowedOwner`. `attacker` has deposited into the curated pool without being on the allowlist.
7. Alternatively, `attacker` calls `MetricOmmPoolLiquidityAdder.addLiquidity*(pool, allowedOwner, salt, deltas)` — same outcome with no custom contract required. [1](#0-0) [4](#0-3)
