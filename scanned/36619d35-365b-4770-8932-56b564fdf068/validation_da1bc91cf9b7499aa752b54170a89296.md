### Title
L2 Price Providers Lack Sequencer Uptime Feed Check, Allowing Stale Oracle Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` perform staleness validation by comparing the oracle's `refTime` against `block.timestamp`. On L2 chains (Arbitrum), when the sequencer is down, oracle push-data stops updating. If a transaction is submitted via the L1 delayed-inbox forced-inclusion path after sequencer downtime, `block.timestamp` advances to the current L1 time while `refTime` remains frozen at the last pre-downtime oracle update. If `MAX_TIME_DELTA` is configured at or above the forced-inclusion delay (~24 h on Arbitrum, up to 7 days is allowed by the constructor), the staleness check passes and a stale bid/ask price is delivered to the pool swap — the exact analog of the external report.

---

### Finding Description

Both L2 price providers implement `_isStale` as a pure timestamp comparison:

```solidity
// PriceProviderL2.sol lines 135-150
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

This check is the **only** oracle-health guard. There is no call to a Chainlink L2 Sequencer Uptime Feed or any equivalent liveness signal. The constructor enforces `MAX_TIME_DELTA <= 7 days`:

```solidity
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
``` [2](#0-1) 

The same pattern is present in `ProtectedPriceProviderL2`: [3](#0-2) [4](#0-3) 

The downstream `_getBidAndAskPrice` / `_computeBidAsk` functions pass the stale price directly into bid/ask computation and then into the pool swap if `_isStale` returns `false`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When the Arbitrum sequencer is down:

1. Push-based oracle data (Pyth Lazer / Chainlink Data Streams) stops being updated on-chain; `refTime` is frozen at the last pre-downtime value `T₀`.
2. After the forced-inclusion delay (~24 h), an attacker submits a swap transaction via the L1 delayed inbox. `block.timestamp` is now `T₀ + 24h+`.
3. `_isStale` computes `nowTs - refTime = 24h+`. If `MAX_TIME_DELTA ≥ 24h`, the check passes and the stale price is used.
4. The pool delivers a bid/ask derived from a price that may be significantly different from the current market price.
5. LPs suffer direct principal loss: the attacker buys the underpriced asset or sells the overpriced asset against LP liquidity at the stale rate.

This matches the **bad-price execution** and **direct loss of LP assets** impact categories.

---

### Likelihood Explanation

- Arbitrum sequencer outages have occurred historically (multiple incidents in 2023–2024).
- `MAX_TIME_DELTA` up to 7 days is permitted by the constructor; any value ≥ 24 h makes the attack viable.
- The forced-inclusion path is permissionless — any address can submit after the delay.
- No sequencer uptime feed or circuit-breaker exists anywhere in the L2 provider path.

Likelihood: **Medium** (requires sequencer downtime + `MAX_TIME_DELTA ≥ 24h` configuration, but both conditions are realistic).

---

### Recommendation

Add a Chainlink L2 Sequencer Uptime Feed check at the top of `_getBidAndAskPrice` / `_computeBidAsk` in both L2 providers. If the sequencer is reported down or the uptime feed answer timestamp is itself stale, return `(0, type(uint128).max)` (the existing stall sentinel) to halt swaps. Example pattern:

```solidity
// Chainlink sequencer uptime feed
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
// answer == 0 means sequencer is up; 1 means down
if (answer != 0) return (0, type(uint128).max);
// Grace period after sequencer recovery
if (block.timestamp - startedAt < GRACE_PERIOD) return (0, type(uint128).max);
```

Additionally, cap `MAX_TIME_DELTA` to a value well below the forced-inclusion delay (e.g., ≤ 1 hour) to reduce the exploitable window even without the uptime feed.

---

### Proof of Concept

**Setup:** Deploy `PriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 2 days`. Configure a pool using this provider.

**Attack sequence:**

1. At time `T₀`, the Arbitrum sequencer goes offline. The last oracle push sets `refTime = T₀` with price `P₀`.
2. Market price moves significantly (e.g., +20%) to `P₁` during the outage, but no oracle update reaches the chain.
3. At `T₀ + 24h`, the attacker submits a swap via the L1 delayed inbox (forced inclusion).
4. The transaction executes with `block.timestamp ≈ T₀ + 24h`.
5. `_isStale` evaluates: `(T₀ + 24h) - T₀ = 24h < MAX_TIME_DELTA (2 days)` → **not stale**.
6. `_getBidAndAskPrice` returns bid/ask derived from stale `P₀`.
7. The attacker buys the asset at `P₀` (20% below current market `P₁`), immediately profiting at LP expense.

        // 1. Read via the unified price(feedId, pool) path, forwarding the pool (msg.sender).
        //    refTime is already in seconds.
        (uint256 mid, uint256 spread, , uint256 refTime) =
            IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);

        // 2. Staleness check
        if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
            return (0, type(uint128).max);
        }

        // 3. Basic validity — price must be positive, spread must not be stalled marker
        if (mid == 0 || spread >= ORACLE_BPS) {
            return (0, type(uint128).max);
        }

        // 4. Price guard check (moved from oracle)
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(offchainFeedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) {
            return (0, type(uint128).max);
        }

        // 5. Compute bid/ask from mid + confidence-adjusted spread
        //    confidenceParam multiplies oracle spread; 0 means no spread
        uint256 adjustedSpread = spread * confidenceParam;
        (uint256 bid, uint256 ask) = _getBidAskFrom(mid, adjustedSpread);

        // 6. Apply marginStep adjustment
        (uint256 bidOut, bool bidOk) = _applyBidAdjustments(bid);
        if (!bidOk || bidOut > type(uint128).max) return (0, type(uint128).max);

        (uint256 askOut, bool askOk) = _applyAskAdjustments(ask);
        if (!askOk || askOut > type(uint128).max) return (0, type(uint128).max);

        // 7. Hard invariant: bid must be strictly less than ask.
        //    Can be violated when marginStep < 0 and confidence is too small.
        if (bidOut >= askOut) return (0, type(uint128).max);

        return (uint128(bidOut), uint128(askOut));
    }
