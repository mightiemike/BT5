### Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the end user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to restrict pool swaps to a curated set of addresses. However, the identity it checks is the **immediate caller of `pool.swap()`** — which is `MetricOmmSimpleRouter` when users route through the standard periphery — not the end user. This forces the pool admin into an impossible choice: either allowlist the router (enabling any user to bypass the gate) or leave the router off the allowlist (breaking the standard periphery path for every allowlisted user). Either way the guard fails to enforce its intended policy.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the **pool** (the pool calls the extension). `sender` is the first argument the pool forwards, which is `msg.sender` of the pool's own `swap()` call:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-L240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

When a user routes through `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`), the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-L80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the **router**, so the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`.

**Forced dilemma for the pool admin:**

| Admin choice | Consequence |
|---|---|
| Do **not** allowlist the router | Every allowlisted user who calls the router is blocked; the standard periphery path is broken for all curated users |
| **Allowlist the router** | `allowedSwapper[pool][router] == true` → any unprivileged user routes through the router and the check passes unconditionally |

There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set (e.g., KYC-verified counterparties, institutional LPs, or protocol-internal actors) is fully bypassable by any unprivileged user via `MetricOmmSimpleRouter`. The attacker receives the same oracle-priced execution as an allowlisted user, draining LP value at the pool's spread and notional fee rates without authorization. This constitutes a broken core pool functionality / admin-boundary break with direct loss of LP assets and protocol fees above Sherlock thresholds.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the canonical, publicly deployed periphery swap path.
- No special privileges, tokens, or setup are required; any EOA can call `exactInputSingle`.
- The bypass is deterministic: once the router is allowlisted (the only way to support the standard periphery for legitimate users), every subsequent router call passes the gate regardless of the actual caller.
- The codebase's own security analysis explicitly flags this vector: *"a curated pool's allowlist can be bypassed through a public router or liquidity-adder path … Enter through the supported periphery path rather than the direct pool call and see whether the identity check changes."*

---

### Recommendation

The extension must gate on the **economic actor** (the end user), not the transport layer (the router). Two sound approaches:

1. **Trusted forwarder pattern**: `MetricOmmSimpleRouter` encodes the original `msg.sender` into `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a known router.
2. **Check `sender` only for direct pool calls; require signed proof for router calls**: the extension verifies an EIP-712 signature from the real user embedded in `extensionData`.

The simplest safe fix is option 1: the router appends `abi.encode(msg.sender)` to `extensionData` for every hop, and the extension reads it when `sender` is a recognized periphery contract.

### Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin calls setAllowedToSwap(pool, alice, true)          // alice is the curated user
3. Admin calls setAllowedToSwap(pool, router, true)         // required so alice can use the router
4. bob (non-allowlisted) has token1 and wants to buy token0.

Attack
──────
5. bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool:        pool,
       zeroForOne:  false,
       amountIn:    X,
       recipient:   bob,
       ...
   });

6. Router calls pool.swap(bob, false, X, ...).
   pool.msg.sender = router.

7. Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...).
   Check: allowedSwapper[pool][router] == true  ✓  → no revert.

8. Swap executes at oracle price. bob receives token0.
   The allowlist guard never saw bob's address.
