### Title
`SwapAllowlistExtension` checks the router address instead of the actual end-user — any user bypasses the per-user swap gate on curated pools via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps on the `sender` argument. The pool sets `sender = msg.sender` of the `pool.swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end user. If the pool admin allowlists the router to enable router-mediated swaps — a natural and expected configuration — every user, including those not individually allowlisted, bypasses the per-user gate by routing through the router.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap` with its own `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` then forwards that value verbatim as the first argument to every configured extension: [2](#0-1) 

**Step 2 — `SwapAllowlistExtension` checks `sender` against the per-pool allowlist.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct). `sender` is the address the pool received as its own `msg.sender` — which is the **router**, not the end user, whenever the swap is router-mediated. [3](#0-2) 

**Step 3 — The router calls `pool.swap()` directly, making itself the pool's `msg.sender`.**

`exactInputSingle`, `exactInput`, and `exactOutputSingle` all call `IMetricOmmPoolActions(pool).swap(...)` from the router contract: [4](#0-3) [5](#0-4) 

In every case the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

**Step 4 — The bypass.**

A pool admin who wants to support both direct and router-mediated swaps must allowlist the router address:

```
allowedSwapper[pool][router] = true
```

Once the router is allowlisted, **every user** — including those not individually allowlisted — can bypass the gate by calling any of the router's `exact*` entry points. The extension sees `sender = router` (allowlisted) and passes unconditionally.

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole mechanism for curated pools to restrict who may trade. A complete bypass means:

- Unauthorized users can execute swaps on pools intended to be restricted (e.g., KYC-gated, institutional-only, or whitelist-only pools).
- LP funds in those pools are exposed to trades from actors the pool admin explicitly intended to exclude.
- The allowlist provides zero protection for any user who routes through the supported periphery path.

This matches the **allowlist bypass** and **wrong-actor binding** impact classes: the checked identity (`router`) is not the economically relevant actor (`end user`), and the bypass is reachable through the protocol's own supported public entrypoint.

---

### Likelihood Explanation

- The router (`MetricOmmSimpleRouter`) is the primary user-facing swap interface for the protocol.
- Any pool admin who deploys a curated pool and also wants to support router-mediated swaps must allowlist the router — this is the expected operational configuration.
- No special privileges, malicious setup, or non-standard tokens are required. Any unprivileged user can trigger the bypass by calling `exactInputSingle` or `exactInput` on the router.
- Likelihood: **Medium-High** (the vulnerable configuration is the natural one for any production curated pool).

---

### Recommendation

The extension must gate on the **actual end user**, not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Pass the original `msg.sender` through the router.** The router already stores the original caller in transient storage for the callback context (`_setNextCallbackContext(..., msg.sender, ...)`). Extend this to also expose the original initiator to the pool/extension, e.g., via an additional field in the swap call or a dedicated transient slot the extension can read.

2. **Alternatively, gate on `recipient` instead of `sender` when `sender` is a known router.** This is fragile; option 1 is preferred.

3. **Short-term mitigation:** Document that allowlisting the router address defeats per-user gating, and require pool admins to use `allowAllSwappers` only when they intend open access, never as a workaround for router support.

---

### Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension
// Pool admin allowlists alice for direct swaps AND the router for router-mediated swaps
allowlist.setAllowedToSwap(pool, alice, true);
allowlist.setAllowedToSwap(pool, address(router), true); // needed for router support

// Bob (not allowlisted) bypasses the gate via router
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: bob,
    amountIn: 1000,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// ✓ swap succeeds — extension saw sender=router (allowlisted), not bob (not allowlisted)
// Bob receives token1 from a pool he was explicitly excluded from
```
