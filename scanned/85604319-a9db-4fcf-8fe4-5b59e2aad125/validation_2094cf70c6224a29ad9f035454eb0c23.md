### Title
`SwapAllowlistExtension` Checks Router Address as `sender` Instead of End User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` ‚Äî (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the direct caller of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of the pool call, so `sender` = router address. If the pool admin whitelists the router to allow whitelisted users to use the standard periphery, every unprivileged user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The pool passes `msg.sender` of the `swap()` call as `sender` to `_beforeSwap` (confirmed by the `IMetricOmmPoolActions` NatSpec: *"Swap allowlist rejected `msg.sender`"*): [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`) calls `pool.swap()`, the router is `msg.sender` of that call: [2](#0-1) 

So the extension sees `sender` = router address, not the actual end user. The pool admin faces an impossible choice:

| Router whitelisted? | Effect |
|---|---|
| **No** | Whitelisted users cannot use the standard periphery at all ‚Äî broken core swap flow |
| **Yes** | Every user can bypass the per-user allowlist by routing through the router |

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the actual position owner, explicitly passed separately from `sender`): [3](#0-2) 

No equivalent "actual initiator" field exists in the swap path, so `SwapAllowlistExtension` is structurally bound to the wrong actor when an intermediary router is used.

The extension calling plumbing confirms `sender` is forwarded verbatim from the pool to the extension without any unwrapping of the original user: [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses loses that curation guarantee the moment the router is whitelisted. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool and the extension will approve the swap because it sees the whitelisted router as `sender`, not the non-whitelisted end user. This is a direct allowlist bypass on a production periphery path with no additional preconditions beyond the router being whitelisted ‚Äî a configuration the pool admin must make to allow any whitelisted user to use the standard periphery.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported swap entry point. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and wants whitelisted users to use the router must whitelist the router address. This is the natural operational path, making the bypass condition likely to be reached in practice. The attacker requires no special role ‚Äî only the ability to call the public router.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the actual end user, not the intermediary. Two complementary fixes:

1. **Pass the real initiator through `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when present, falling back to `sender` for direct pool calls.

2. **Alternatively, mirror the deposit pattern**: Introduce a dedicated `swapInitiator` field in the swap call (analogous to `owner` in `addLiquidity`) so the pool can pass the real user separately from the routing intermediary.

Until fixed, pool admins should not whitelist the router address in `SwapAllowlistExtension` and should instruct whitelisted users to call `pool.swap()` directly.

---

### Proof of Concept

```
Setup:
  - Pool P configured with SwapAllowlistExtension
  - allowedSwapper[P][alice] = true   (alice is whitelisted)
  - allowedSwapper[P][router] = true  (router whitelisted so alice can use it)

Attack:
  1. Bob (not whitelisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=bob, ...)  ‚Üí  msg.sender of pool.swap() = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[P][router] ‚Üí true  ‚úì
  5. Swap executes for Bob despite Bob not being in the allowlist

Result: SwapAllowlistExtension curation is fully bypassed for any user who routes
        through MetricOmmSimpleRouter, as long as the router is whitelisted.
``` [5](#0-4) [6](#0-5)
