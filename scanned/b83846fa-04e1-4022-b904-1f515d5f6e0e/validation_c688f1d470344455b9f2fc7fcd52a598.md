### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual token payer) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any address that is not on the allowlist can deposit tokens into a curated pool by nominating any allowlisted address as `owner`. The allowlist check passes on the allowlisted `owner`, while the actual depositor (`sender`) is never verified.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```solidity
// MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `msg.sender` — the actual caller who will pay tokens via the `metricOmmModifyLiquidityCallback`.
- `owner` — the address that receives the minted LP shares.

The pool's own NatSpec documents this split explicitly: *"`msg.sender` pays but need not equal `owner` (operator pattern)."*

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and never reads it:

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The guard resolves to `allowedDepositor[pool][owner]`. Because `owner` is a caller-supplied argument with no constraint tying it to `msg.sender`, an unauthorized caller can set `owner` to any allowlisted address and the check passes unconditionally.

The `MetricOmmPoolLiquidityAdder` amplifies the surface: its `_validateOwner` only rejects `address(0)`, and it explicitly supports depositing on behalf of a different `owner`:

```solidity
// MetricOmmPoolLiquidityAdder.sol:65-67
_validateOwner(owner);   // only checks owner != address(0)
_validateDeltas(deltas);
return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
```

The `addLiquidityWeighted` probe path also calls `pool.addLiquidity(owner, ...)` with `sender = address(LiquidityAdder)`, so the extension sees the adder contract as sender and the caller-supplied `owner` as the gated identity — the same bypass applies.

---

### Impact Explanation

A pool deployer configures `DepositAllowlistExtension` to restrict deposits to a curated set of addresses (e.g., KYC-verified LPs, institutional counterparties). The bypass allows any unpermissioned address to:

1. **Deposit tokens into the pool** without appearing on the allowlist — the curation policy is fully defeated.
2. **Force LP shares onto an allowlisted address** that did not initiate the deposit — a griefing vector that creates unwanted positions for the victim (who must spend gas to remove them and may be exposed to adverse price movement in the interim).
3. **Introduce tokens from non-allowlisted sources** into a pool whose compliance posture depends on knowing the origin of every deposit.

This matches the allowed impact category: *"High direct loss or curation failure if disallowed users can still trade or deposit."*

---

### Likelihood Explanation

The exploit requires no special privileges, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(allowlisted_address, ...)` directly or route through `MetricOmmPoolLiquidityAdder`. The only prerequisite is knowing one allowlisted address, which is publicly readable from `allowedDepositor` or observable on-chain. Likelihood is **High**.

---

### Recommendation

Gate on `sender` (the actual token payer) rather than — or in addition to — `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    // Check the actual payer, not just the position recipient.
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended policy is that both the payer and the position owner must be allowlisted (e.g., to prevent griefing of allowlisted owners), check both:

```solidity
if (!allowAllDepositors[msg.sender]
    && (!allowedDepositor[msg.sender][sender] || !allowedDepositor[msg.sender][owner])) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
```

### Proof of Concept

**Setup:**
- Pool deployed with `DepositAllowlistExtension` as `extension1`, `beforeAddLiquidity` order set to `1`.
- `alice` is allowlisted: `allowedDepositor[pool][alice] = true`.
- `bob` is **not** allowlisted.

**Attack:**
```solidity
// Bob calls addLiquidity directly, nominating Alice as owner.
// Bob is msg.sender (sender), Alice is owner.
pool.addLiquidity(
    alice,          // owner — passes allowlist check
    salt,
    deltas,
    callbackData,   // Bob's callback pays the tokens
    extensionData
);
// Extension sees: allowedDepositor[pool][alice] == true → passes.
// Bob's callback fires; Bob transfers tokens to the pool.
// Alice receives LP shares she never requested.
// Bob has deposited into a curated pool without being on the allowlist.
```

**Via LiquidityAdder (same result):**
```solidity
// Bob calls the periphery adder with alice as owner.
liquidityAdder.addLiquidityExactShares(
    address(pool),
    alice,          // owner — passes allowlist check
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    extensionData
);
// _validateOwner only checks alice != address(0) — passes.
// pool.addLiquidity(alice, ...) called with sender = address(liquidityAdder).
// Extension checks allowedDepositor[pool][alice] → true → passes.
// Bob (msg.sender of adder) pays tokens; Alice receives shares.
```
