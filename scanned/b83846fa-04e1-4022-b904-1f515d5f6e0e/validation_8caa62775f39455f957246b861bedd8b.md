### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Curated-Pool Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. The pool always passes `msg.sender` (the immediate caller of `pool.swap`) as `sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The extension therefore checks the router's address against the allowlist, not the actual user's address. If the pool admin allowlists the router (the natural step to enable periphery access), every unprivileged user can swap in the curated pool by calling the router, defeating the entire allowlist.

---

### Finding Description

**Actor binding in `SwapAllowlistExtension`**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

`msg.sender` here is the pool (correct for pool-namespacing). `sender` is the first argument the pool passes when it calls the extension.

**What the pool passes as `sender`**

Inside `MetricOmmPool.swap`, the pool calls:

```solidity
_beforeSwap(
    msg.sender,   // ‚Üê this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

`msg.sender` of the pool's `swap` call is whoever called `pool.swap(...)` ‚Äî the router, not the end user.

**How the router calls the pool**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
``` [3](#0-2) 

The pool's `msg.sender` is therefore the router contract address. The extension receives `sender = router`, not `sender = user`.

**The allowlist check is on the wrong actor**

The extension's allowlist is keyed `allowedSwapper[pool][swapper]`. The admin intends `swapper` to be the end user. But the extension receives and checks the router's address. The admin must either:

1. Allowlist the router ‚Üí every user can swap through the router, bypassing per-user curation entirely.
2. Not allowlist the router ‚Üí individually allowlisted users cannot use the standard periphery path at all.

Neither outcome matches the intended invariant.

---

### Impact Explanation

**Critical/High ‚Äî direct bypass of curated-pool access control.**

A pool configured with `SwapAllowlistExtension` is a curated pool: only approved addresses may trade. If the pool admin allowlists the router (the expected operational step to enable normal periphery usage), any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and swap against the pool. The allowlist provides zero protection against router-routed swaps. Unauthorized users can drain LP-owed output tokens from the pool at oracle prices, causing direct loss of LP principal and fee revenue.

---

### Likelihood Explanation

**High.** The `MetricOmmSimpleRouter` is the standard, documented swap entry point. Any pool admin who deploys a `SwapAllowlistExtension` and then allowlists the router (or any other intermediary) to enable normal user access triggers the bypass automatically. No special knowledge or privileged access is required ‚Äî any user who calls `exactInputSingle` or `exactInput` through the router is affected.

---

### Recommendation

The extension must check the actual economic actor, not the immediate pool caller. Two options:

1. **Check `recipient` instead of `sender`** ‚Äî the second argument to `beforeSwap` is `recipient`. For direct swaps this is the user; for router swaps the admin can allowlist recipients. However, `recipient` can be set to any address by the caller, so this is also gameable.

2. **Preferred: pass the original `msg.sender` through `extensionData`** ‚Äî the router should encode the original user address in `extensionData`, and the extension should decode and verify it. This requires a coordinated change to the router and extension.

3. **Simplest correct fix** ‚Äî change `SwapAllowlistExtension.beforeSwap` to check `recipient` (second param) and document that the allowlist gates who may *receive* output, or redesign the hook to accept a trusted-forwarder pattern where the router attests the real user.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, alice, true)` ‚Äî only Alice is allowed.
3. Admin calls `setAllowedToSwap(pool, router, true)` ‚Äî router is allowlisted so Alice can use the periphery.
4. Eve (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The pool calls `extension.beforeSwap(router, ...)`.
7. The extension checks `allowedSwapper[pool][router]` ‚Üí `true` ‚Üí swap proceeds.
8. Eve successfully swaps in the curated pool, receiving pool output tokens she was never authorized to receive.

The allowlist invariant is broken: Eve is not in `allowedSwapper[pool]`, yet she executes a full swap and receives output tokens from the pool. [4](#0-3) [5](#0-4) [6](#0-5)
