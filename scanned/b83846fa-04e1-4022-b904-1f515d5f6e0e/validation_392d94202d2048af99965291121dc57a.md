### Title
Swap Allowlist Bypassed via Router: `sender` Bound to Router Address Instead of Actual Swapper — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` — the direct caller of `MetricOmmPool.swap()`. When `MetricOmmSimpleRouter` mediates the swap, the router becomes `sender`. A pool admin who allowlists the router to support router-mediated swaps for curated users inadvertently grants every user the ability to bypass the per-user restriction by routing through the public router.

---

### Finding Description

**Root cause — wrong actor bound as `sender`:**

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks the allowlist keyed on that `sender`: [3](#0-2) 

**How the router breaks the invariant:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The actual user (`msg.sender` of the router call) is stored only in the transient callback context for payment purposes — it is never forwarded to the pool as `sender`. The extension therefore sees `sender = router address`, not the user's address.

The same pattern applies to `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput`: [5](#0-4) 

**The impossible configuration:**

A pool admin who wants to restrict swaps to a curated set of users faces an inescapable dilemma:

| Configuration | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| Allowlist the router | **Every** user can bypass the per-user restriction via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

Any user can bypass a pool's `SwapAllowlistExtension` by routing through `MetricOmmSimpleRouter`. The allowlist — the only mechanism for per-pool swap curation — is rendered ineffective for all router-mediated swaps. Unauthorized users can trade in pools intended to be restricted (e.g., for regulatory compliance, exclusive LP arrangements, or risk-tiered access), directly harming LP providers who deposited under the assumption that only authorized counterparties would trade against their liquidity.

This matches the allowed impact gate: **broken core pool functionality** (the allowlist extension does not enforce its policy on the supported periphery path) and **admin-boundary break** (an unprivileged user bypasses the pool admin's configured curation policy).

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any user can call it with any pool address.
- The bypass requires no privileged access, no special tokens, and no malicious setup.
- Any pool that configures `SwapAllowlistExtension` and expects router-mediated swaps to be subject to per-user restrictions is immediately vulnerable.
- Likelihood: **High**.

---

### Recommendation

Forward the original user's address through the router to the pool so the extension can gate the actual economic actor. One approach: add a `sender` override parameter to `pool.swap()` that the router populates with `msg.sender`, validated by requiring the caller to be a factory-registered router. Alternatively, the `SwapAllowlistExtension` should read the original initiator from a trusted transient context rather than relying on the pool's `msg.sender`.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as `beforeSwap` hook.
2. Pool admin allowlists `alice` (`allowedSwapper[pool][alice] = true`) and also allowlists the router (`allowedSwapper[pool][router] = true`) so that `alice` can use the router.
3. `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` is the router.
5. `_beforeSwap(router, ...)` is called; extension checks `allowedSwapper[pool][router]` → `true`.
6. Bob's swap executes successfully despite not being on the allowlist.

Alternatively, if the pool admin does **not** allowlist the router:

3. `alice` (allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Extension checks `allowedSwapper[pool][router]` → `false` → reverts with `NotAllowedToSwap`.
5. Alice cannot use the router at all, making the router unusable for any allowlisted pool.

Both outcomes demonstrate that the `SwapAllowlistExtension` cannot enforce per-user swap restrictions on the supported router path. [3](#0-2) [1](#0-0) [6](#0-5)
