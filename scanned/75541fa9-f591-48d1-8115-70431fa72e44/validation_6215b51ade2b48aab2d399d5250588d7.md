Looking at the `SwapAllowlistExtension`, the pool's `swap()` function, and the `MetricOmmSimpleRouter`, I can trace a concrete allowlist bypass.

**How the pool passes `sender` to the extension:**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, ...)`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of the pool's `swap()` call — so `sender` delivered to the extension is the **router's address**, not the actual end user.

**What the extension checks:**

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is whoever called `pool.swap()` — the router. So the check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**The bypass:**

A pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist. Once `allowedSwapper[pool][router] = true`, **every user** — including those explicitly not allowlisted — can bypass the gate by routing through `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput`. The router is a public, permissionless contract.

This is the direct analog to the InsurancePoolToken bug: the guard updates/checks state for one actor (the direct caller / the router) but leaves the economically relevant actor (the actual end user) unchecked, allowing the intended restriction to be bypassed by inserting an intermediary.

### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass Per-Pool Swap Allowlist via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` = `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, that value is the router's address. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users simultaneously grants swap access to every user on the network, defeating the allowlist entirely.

### Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then gates on that value: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender` inside the pool: [3](#0-2) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` / `_exactOutputIterateCallback`: [4](#0-3) [5](#0-4) 

Because the router is a public, permissionless contract, `allowedSwapper[pool][router] = true` is equivalent to `allowAllSwappers[pool] = true` for any user who knows to route through it.

### Impact Explanation
Any user not on the per-pool allowlist can trade in a curated pool by calling the router instead of the pool directly. LP providers who deployed a curated pool to restrict counterparties (e.g., KYC-only, institutional-only) lose that protection entirely. Depending on pool design, this can expose LP funds to adversarial flow, oracle-manipulation trades, or simply violate the legal/compliance intent of the curation, constituting a broken core pool functionality and curation failure.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who enables router-mediated swaps for their allowlisted users (the expected operational setup) triggers the bypass automatically. No special knowledge or privileged access is required — any user can call `exactInputSingle` with the curated pool address.

### Recommendation
Pass the **originating user** rather than the immediate caller as `sender` to the extension. Two options:

1. **Preferred:** Add a `swapper` parameter to `IMetricOmmPoolActions.swap()` that callers supply explicitly. The router passes `msg.sender` (the end user); direct callers pass `msg.sender` as well. The pool forwards this value to `_beforeSwap` instead of its own `msg.sender`.
2. **Alternative:** Have the router store the originating user in transient storage (alongside the existing callback context) and expose a `currentSwapper()` view that the extension can read. The extension calls back into the router to retrieve the real user identity.

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to allow allowlisted users to use the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)
   — alice is the only intended swapper.
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: curated_pool,
       recipient: bob,
       ...
     })
5. Router calls pool.swap(); msg.sender inside pool = router.
6. _beforeSwap passes sender = router to SwapAllowlistExtension.
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes despite not being on the allowlist.
