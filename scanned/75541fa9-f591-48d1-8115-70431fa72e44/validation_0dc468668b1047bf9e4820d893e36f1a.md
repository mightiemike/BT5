### Title
`SwapAllowlistExtension.beforeSwap` gates on the router's address instead of the actual user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user. If the pool admin allowlists the router to enable router-mediated swaps for their curated users, every unprivileged user can bypass the allowlist by routing through the same public router contract.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap()` forwards this value verbatim to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (or any other router entry point), the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

At the pool level `msg.sender = router`, so the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`.

**The extension has no visibility into who called the router.** A pool admin who wants to allow their allowlisted users to swap through the router must allowlist the router address. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether they are on the allowlist.

This is structurally identical to the seed bug: just as the `else` branch in `OneVsOneExactPrice` catches both `diff1 > diff2` and `diff1 == diff2` and awards the win to the opponent, the `allowedSwapper[pool][sender]` check catches both "allowlisted user via router" and "non-allowlisted user via router" and passes both — the wrong actor wins the identity check on the boundary case.

The `DepositAllowlistExtension` does not share this flaw because it gates on `owner` (the position owner explicitly passed by the caller), which the liquidity adder preserves correctly. The swap allowlist gates on `sender` (the immediate caller), which the router replaces with itself.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` cannot simultaneously:
1. Allow their allowlisted users to swap through `MetricOmmSimpleRouter` (for standard UX), and
2. Block non-allowlisted users from swapping through the same router.

If the router is allowlisted, every user — including those the pool admin explicitly excluded — can bypass the allowlist by calling any router entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). The curated pool's access control is fully defeated. LPs who deposited into a curated pool expecting restricted counterparty flow are exposed to unrestricted toxic flow, leading to direct LP principal loss through adverse selection.

---

### Likelihood Explanation

The likelihood is medium-high. Allowlisting the router is the natural and expected action for any pool admin who wants their curated users to access the pool through the standard periphery. The `MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who does not allowlist the router forces their allowlisted users to call the pool directly, bypassing the standard periphery — an unusual and friction-heavy requirement that most admins will not impose. The bypass is then reachable by any unprivileged user with a single public router call.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the economically relevant actor, not the immediate pool caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: Require the router to encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and check that address. The extension must also verify that `sender` is a trusted router before trusting the encoded identity.

2. **Gate on `recipient` as a proxy**: For single-hop swaps the recipient is often the actual user, but this breaks for multi-hop flows where intermediate recipients are the router itself.

3. **Maintain a router registry**: Allow the pool admin to register trusted routers; when `sender` is a registered router, require the actual user identity to be passed in `extensionData` and verified.

The `DepositAllowlistExtension` pattern (gating on `owner`, which is explicitly passed and preserved by the liquidity adder) is the correct model to follow.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)      // router allowlisted so alice can use it

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: curated_pool, ...})
  - Router calls pool.swap(...) with msg.sender = router
  - Pool calls _beforeSwap(router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
  - Swap proceeds — bob has bypassed the allowlist
```

The root cause is at:
- [1](#0-0)  — `sender` is checked, which is the immediate pool caller, not the actual user.
- [2](#0-1)  — pool passes `msg.sender` (the router) as `sender` to the extension.
- [3](#0-2)  — router calls `pool.swap()` directly, replacing the real user's identity with its own address.
