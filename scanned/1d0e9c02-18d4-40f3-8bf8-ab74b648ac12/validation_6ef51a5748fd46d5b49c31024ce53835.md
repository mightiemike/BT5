### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` — the immediate caller of `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks whether the **router** is allowlisted, not the actual user. If the router is allowlisted (a natural operational setup), any non-allowlisted user can bypass the curated-pool restriction by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (used as the per-pool namespace key — correct), and `sender` is the first argument forwarded by the pool's internal `_beforeSwap` dispatcher. [1](#0-0) 

In `ExtensionCalling._beforeSwap`, the pool encodes the call as:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
``` [2](#0-1) 

The pool's `swap` (and `simulateSwapAndRevert`) passes `msg.sender` as `sender`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap — the router, not the user
    recipient,
    ...
);
``` [3](#0-2) 

When `MetricOmmSimpleRouter` calls `pool.swap(recipient, ...)`, the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

This is structurally inconsistent with `DepositAllowlistExtension`, which correctly gates by `owner` — an explicit parameter that the `LiquidityAdder` sets to the actual user's address, not its own address:

```solidity
// DepositAllowlistExtension.sol line 38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

The deposit path correctly threads the real user through `owner`; the swap path has no equivalent — the real user is invisible to the extension.

---

### Impact Explanation

**Allowlist bypass (admin-boundary break):** A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers). The admin allowlists the router so that approved users can reach the pool through the standard periphery path. Because the extension checks the router address rather than the user, any non-allowlisted user can call `router.swap(pool, ...)` and the extension will pass — the router is allowlisted, so the check succeeds regardless of who the actual user is. The curated pool's entire access-control policy is nullified for every user who routes through the router.

**Inverse DoS:** If the admin does not allowlist the router (to avoid the above), then legitimately allowlisted users who call through the router are blocked, making the standard periphery path unusable for them.

Both outcomes break the core invariant that "a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary production entry point for swaps. Allowlisting the router is the natural operational step a pool admin would take to allow approved users to use the standard periphery. The bypass therefore activates under a routine, expected configuration — not an exotic or adversarial one. Any user aware of the router address can exploit it without any special privilege.

---

### Recommendation

The pool must thread the originating user's identity through to the extension in a way that cannot be spoofed by an intermediate router. Two options:

1. **Explicit `payer` parameter**: Add an explicit `payer` (or `originator`) address to `pool.swap` that the router sets to `msg.sender` (the actual user). The extension checks this field instead of `sender`. The pool validates that `payer == msg.sender` on direct calls and trusts the router to set it correctly (or the router is verified by the factory).

2. **Check `recipient` with router-awareness**: If the router always sets `recipient` to the actual user, the extension could check `recipient`. However, this is fragile because `recipient` is the output destination, not necessarily the initiating user.

The cleanest fix mirrors the deposit path: make the actual user an explicit, named parameter in the swap call so extensions can gate on the correct actor.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists the router:
       extension.setAllowedToSwap(pool, address(router), true)
   (natural step: "let router users reach my pool")
3. Non-allowlisted user Alice calls:
       router.swap(pool, recipient=Alice, zeroForOne=true, amount=X, ...)
4. Router calls pool.swap(recipient=Alice, ...) — pool sees msg.sender = router.
5. Pool calls _beforeSwap(sender=router, recipient=Alice, ...).
6. Extension evaluates:
       allowedSwapper[pool][router] == true  →  check passes, no revert.
7. Alice's swap executes on the curated pool despite never being allowlisted.
8. Pool admin's curation policy is completely bypassed for all router users.
``` [1](#0-0) [2](#0-1)
