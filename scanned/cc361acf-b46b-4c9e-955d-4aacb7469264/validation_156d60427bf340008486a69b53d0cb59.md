### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the **router contract**, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This wrong-actor binding either (a) silently bypasses the per-user allowlist for every user who routes through the router, or (b) blocks every allowlisted user from using the supported periphery path.

---

### Finding Description

**Call chain when a user swaps via the router:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ...) [msg.sender = router]
              → _beforeSwap(msg.sender=router, recipient, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → allowedSwapper[pool][router]  ← wrong actor
```

The pool passes its own `msg.sender` as the `sender` argument to every extension hook: [1](#0-0) 

The extension then gates on that forwarded `sender`: [2](#0-1) 

The router calls `pool.swap` directly with no mechanism to forward the original `msg.sender`: [3](#0-2) 

This produces two mutually exclusive failure modes depending on how the pool admin configures the allowlist:

**Mode A — Allowlist bypass (higher severity):** The pool admin allowlists the router address (a natural operational choice so that allowlisted users can use the standard periphery path). Because the router is a public, permissionless contract, every user — including those explicitly excluded from the allowlist — can now swap by calling `exactInputSingle` / `exactInput` / `exactOutput`. The per-user curation is completely defeated.

**Mode B — DoS on legitimate users:** The pool admin does not allowlist the router (correct intent, wrong outcome). Every allowlisted user who calls the router is rejected with `NotAllowedToSwap`, even though they are individually permitted. The only usable path is a direct `pool.swap()` call, which requires the caller to implement `IMetricOmmSwapCallback` — not a realistic option for ordinary EOAs.

The `DepositAllowlistExtension` does not share this flaw because it gates on `owner` (the position owner supplied by the caller), which the `MetricOmmPoolLiquidityAdder` preserves correctly as `msg.sender` of the adder call: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Mode A (bypass):** Any unprivileged user can trade in a curated pool by routing through `MetricOmmSimpleRouter`. LP funds in the pool are exposed to actors the pool admin explicitly excluded. This is a direct loss-of-curation-control impact with fund consequences (unauthorized price impact, fee extraction, or compliance violation on regulated pools).

**Mode B (DoS):** Every allowlisted EOA is blocked from the only practical swap path. This is broken core swap functionality for the intended user set.

Both modes are reachable by any public user with no special privileges.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that deploys `SwapAllowlistExtension` and expects users to interact through the router will immediately encounter one of the two failure modes. The bypass mode is triggered the moment the admin allowlists the router to "fix" the DoS, which is the intuitive remediation step.

---

### Recommendation

Pass the original end-user identity through the extension data or add a dedicated `originalSender` field to the `beforeSwap` hook arguments. A simpler short-term fix is to have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when present, falling back to `sender` for direct pool calls. Alternatively, gate on `tx.origin` as a last resort (with documented trust assumptions), or require direct pool interaction for allowlisted pools.

---

### Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension.
// Admin allowlists alice but NOT the router.
extension.setAllowedToSwap(pool, alice, true);

// Alice calls the router (standard periphery path):
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: alice,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// ↑ Reverts NotAllowedToSwap — alice is blocked despite being allowlisted.
// The extension checked allowedSwapper[pool][router] == false.

// Now admin allowlists the router to "fix" alice's access:
extension.setAllowedToSwap(pool, address(router), true);

// Bob (not allowlisted) now bypasses the allowlist:
vm.prank(bob);
router.exactInputSingle(...); // succeeds — allowlist fully bypassed
```
