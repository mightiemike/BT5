### Title
`SwapAllowlistExtension` gates the router contract address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter — which the pool sets to `msg.sender` of the `pool.swap` call — against the per-pool allowlist. When any user routes through `MetricOmmSimpleRouter`, `sender` resolves to the **router contract address**, not the actual user. If the router is allowlisted (a natural admin action to let allowlisted users use the official router), every unprivileged user can bypass the allowlist entirely by routing through the public router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
``` [2](#0-1) 

The pool sets `sender = msg.sender` of the `pool.swap` call. When `MetricOmmSimpleRouter.exactInputSingle` executes:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
``` [3](#0-2) 

the pool's `msg.sender` is the **router contract**, so the extension receives `sender = router_address`. The allowlist check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The same identity substitution occurs in `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

A pool admin who wants to let their allowlisted users use the official router will naturally call `setAllowedToSwap(pool, router, true)`. At that point the allowlist is completely open: any address can call `router.exactInputSingle({pool: restricted_pool, ...})` and the extension will see `sender = router` → allowed → swap proceeds.

The extension's own documentation states it "Gates `swap` by swapper address, per pool": [5](#0-4) 

but the identity it actually gates is the intermediary contract, not the swapper.

---

### Impact Explanation

Any unprivileged user can bypass a pool's `SwapAllowlistExtension` by routing through `MetricOmmSimpleRouter` whenever the router address is allowlisted. Pools that use the allowlist to restrict access to specific counterparties (e.g., KYC-gated, institutional, or compliance-restricted pools) are fully open to unauthorized swappers. LPs who provided liquidity under the assumption that only vetted counterparties could trade against them are exposed to unrestricted public flow, which can drain favorable inventory at oracle-quoted prices. This is a direct admin-boundary break: an unprivileged path (the public router) bypasses an admin-configured guard with fund-impacting consequences for LPs.

---

### Likelihood Explanation

Medium-to-High. The bypass requires the router to be allowlisted, which is the natural and expected configuration for any pool that wants its allowlisted users to access the protocol through the official periphery. The flaw is structural: the pool admin cannot simultaneously (a) allow allowlisted users to use the router and (b) block non-allowlisted users from using the router, because the router is a single address. The design gap is not documented, so pool admins are unlikely to anticipate it.

---

### Recommendation

1. **Check the actual user, not the intermediary.** The router should attest the real user's identity by encoding it in `extensionData`. The extension should decode and verify that address instead of trusting `sender`.
2. **Alternatively**, the pool's `swap` function could accept an explicit `swapper` argument (separate from `recipient`) that the router populates with `msg.sender`, and the extension checks that field.
3. **At minimum**, document clearly that `sender` is `msg.sender` of the `pool.swap` call (i.e., the router when the router is used), so pool admins understand that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls extension.setAllowedToSwap(pool, userA, true)   // allowlist userA
  admin calls extension.setAllowedToSwap(pool, router, true)  // allow userA to use the router

Attack:
  userB (not allowlisted) calls:
    router.exactInputSingle({pool: restricted_pool, recipient: userB, ...})

Execution trace:
  router → pool.swap(recipient=userB, ...)   [msg.sender = router]
  pool   → _beforeSwap(sender=router, recipient=userB, ...)
  extension checks: allowedSwapper[pool][router] == true  ✓
  swap executes — userB trades in a pool they were never authorized to access
```

`userB` receives tokens from a pool whose LP depositors expected only `userA`-class counterparties, extracting value at oracle-quoted prices with no authorization.
