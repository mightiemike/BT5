### Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, allowing any user to bypass the swap allowlist via the public router â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks `sender` against `allowedSwapper[pool][sender]`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `swap()` is called by the router, so `sender` is the router's address â€” not the actual user. A pool admin who adds the router to the allowlist (the only way to let legitimate users trade via the router) inadvertently opens the gate to every user on the network.

### Finding Description
The `_beforeSwap` dispatcher in `ExtensionCalling.sol` forwards `sender` verbatim to every registered extension:

```solidity
// ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified,
         priceLimitX64, packedSlot0Initial, bidPriceX64, askPriceX64, extensionData))
);
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
require(allowAll[pool] || allowedSwapper[pool][sender], NotAllowedToSwap());
```

When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract. That address is what the extension sees as `sender`. The actual end-user who initiated the transaction is never checked.

**Consequence:** To allow legitimate users to trade through the router, the pool admin must add the router's address to `allowedSwapper`. Once the router is allowlisted, the check degenerates to `allowedSwapper[pool][router] == true` for every caller â€” the allowlist is completely bypassed for router-mediated swaps.

This is structurally identical to the external self-trade bug: the guard correctly handles one execution path (direct `pool.swap()` calls) but silently fails for the other path (router-mediated calls), because the identity it checks (`sender`) is not the identity the pool designer intended to gate (the actual user).

### Impact Explanation
Any user â€” including those explicitly excluded from the allowlist â€” can trade against the pool's LP positions by routing through `MetricOmmSimpleRouter`. LP providers who deployed capital under the assumption that only allowlisted counterparties could trade against them suffer direct loss of principal. The allowlist's entire economic purpose (e.g., restricting to KYC'd users, specific market makers, or whitelisted protocols) is nullified.

### Likelihood Explanation
The router is a public, permissionless contract. No special privilege is required. The only precondition is that the pool admin has added the router to the allowlist â€” which is the expected and necessary configuration for any pool that wants legitimate users to access it via the standard periphery. The exploit path is a single call to `MetricOmmSimpleRouter.exactInputSingle()`.

### Recommendation
The router should forward the originating user's address as `sender` when calling `pool.swap()`, rather than relying on the pool to see `msg.sender` (the router). Concretely, `MetricOmmSimpleRouter` should pass `msg.sender` explicitly as the `sender` argument so the extension hook receives the actual swapper identity. Alternatively, `SwapAllowlistExtension` should check `recipient` when `sender` is a known router, but this is fragile; fixing the forwarding at the router level is the correct approach.

### Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` registered in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `allowedSwapper[pool][router] = true` so that legitimate users can trade via `MetricOmmSimpleRouter`.
3. Unauthorized user `U` (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(sender=router, recipient=U, ...)`.
5. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ no revert.
7. Swap executes; `U` receives output tokens. The allowlist provided zero protection. [6](#0-5) [2](#0-1) [3](#0-2)
