### Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Allowing Any User to Bypass Curated-Pool Swap Gate via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool always sets to `msg.sender` of the `pool.swap(...)` call. When `MetricOmmSimpleRouter` is the immediate caller, `sender` is the router's address, not the original user's address. A pool admin who allowlists the router (a natural step to let their curated users reach the pool through the standard periphery) inadvertently opens the gate to every user on-chain, because the extension never sees the original user's identity.

---

### Finding Description

**Actor binding in the extension:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist keyed by `msg.sender` (the pool): [1](#0-0) 

**What the pool passes as `sender`:**

`MetricOmmPool.swap` always passes `msg.sender` — the immediate caller of the pool — as the `sender` argument to `_beforeSwap`: [2](#0-1) 

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The router is therefore `msg.sender` inside the pool, so `sender` forwarded to the extension is the router's address, not the original EOA: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**The broken invariant:**

A pool admin who wants curated users to be able to use the standard router must allowlist the router address. Once `allowedSwapper[pool][router] = true`, the extension's check passes for every user who routes through the router — the original EOA is never inspected. Any non-allowlisted user can bypass the gate by calling any of the router's public `exact*` functions.

The contest scope explicitly states: *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."* [4](#0-3) 

---

### Impact Explanation

**Direct loss / broken core functionality.** The swap allowlist is the primary access-control mechanism for curated pools. Once bypassed, any user can execute swaps against a pool that was configured to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers, or protocol-internal actors). This breaks the pool's curation invariant and allows unauthorized users to extract value from the pool at oracle-anchored prices, which is a direct loss of LP principal relative to the pool's intended operating model.

---

### Likelihood Explanation

**Medium.** The bypass requires the pool admin to allowlist the router address. This is a natural and expected administrative action: without it, allowlisted users cannot use the standard periphery at all, which makes the pool effectively unusable through the supported interface. A pool admin who reads the allowlist docs and wants to support router-mediated swaps for their curated users will perform exactly this step, unknowingly opening the gate to everyone. The trigger is a single `setAllowedToSwap(pool, router, true)` call by the bounded pool admin.

---

### Recommendation

The extension must check the original user's identity, not the immediate pool caller. Two approaches:

1. **Preferred — pass original user through `extensionData`:** Have the router encode `msg.sender` into the `extensionData` it forwards to the pool. The extension decodes and checks that address. The pool admin allowlists individual users, not the router.

2. **Alternative — separate `originator` field in the hook interface:** Add an `originator` parameter to `beforeSwap` that the pool populates from a transient-storage context set by the router before calling `pool.swap`. The extension checks `originator` instead of `sender`.

Either approach ensures the allowlist gates the economically relevant actor regardless of which supported entrypoint is used.

---

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension configured
  alice = allowlisted user (allowedSwapper[pool][alice] = true)
  bob   = non-allowlisted user
  admin allowlists the router: allowedSwapper[pool][router] = true
    (so that alice can use the standard router interface)

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
    pool: pool,
    recipient: bob,
    ...
  })

  Router calls pool.swap(bob, ...) — msg.sender inside pool = router
  Pool calls _beforeSwap(sender=router, ...)
  Extension checks allowedSwapper[pool][router] → true
  Swap executes for bob despite bob not being allowlisted

Result:
  bob bypasses the curated-pool swap gate at zero additional cost.
  The allowlist provides no protection for router-mediated swaps.
