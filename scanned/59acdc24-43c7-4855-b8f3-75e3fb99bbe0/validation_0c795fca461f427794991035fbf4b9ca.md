### Title
`PriceVelocityGuardExtension.beforeSwap` Silently Bypasses Velocity Check When `lastMidPriceX64` Is Uninitialized — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally writes the current mid-price to storage and then gates the entire velocity check behind `if (prevMid != 0)`. When `lastMidPriceX64` has never been seeded — the default state for every newly configured pool — the guard is silently skipped for the first swap, and the current (potentially extreme) oracle price is committed as the new baseline. This is the direct structural analog to the TWAP cardinality-1 bug: just as a Uniswap V3 pool with cardinality=1 has no historical observation and falls back to the current spot price, a Metric pool with an unseeded velocity guard has no prior mid-price observation and falls back to unconstrained execution.

---

### Finding Description

`PriceVelocityGuardExtension` is a `beforeSwap` hook that caps how fast the oracle mid-price can move between blocks. Its state is keyed per pool in `priceVelocityState[pool_]`, which is a zero-initialized struct on first use. [1](#0-0) 

The critical sequence in `beforeSwap` is:

```solidity
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // reads 0 on first swap
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // ← state written unconditionally
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // ← guard skipped when prevMid == 0
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity check
    }
}
``` [2](#0-1) 

When `lastMidPriceX64` is zero (the default for any pool that has not had `setLastMidPrice` called by the admin), the entire velocity check is skipped. The current oracle mid-price — regardless of how far it has moved from the price at pool deployment — is written as the new baseline. Every subsequent swap is then checked against this potentially extreme baseline, not against the true market price at pool creation.

The extension has no `initialize` hook and no `onlyPool` guard on `beforeSwap` (unlike the base class which has `onlyPool`). The pool admin is expected to call `setLastMidPrice` before the first swap, but there is no enforcement of this ordering. [3](#0-2) 

The base class `BaseMetricExtension.beforeSwap` declares the function `virtual onlyPool`, but `PriceVelocityGuardExtension` overrides it without the `onlyPool` modifier, removing the factory-pool identity check from the override. [4](#0-3) 

---

### Impact Explanation

The velocity guard is the protocol's primary defense against rapid oracle price movement reaching live swaps. When `prevMid == 0`, the guard fails open: the first swap on any pool with this extension executes at whatever oracle price the price provider returns, with no velocity constraint applied. If the oracle price has moved significantly since pool deployment (e.g., a genuine market move, a brief oracle spike, or a Pyth update with a large confidence interval that passes staleness checks), LPs are exposed to a swap at a price the guard was configured to block. After this first swap, the extreme price is committed as the new baseline, potentially allowing a second swap that is within the velocity envelope relative to the manipulated baseline but still far from the true market price. This constitutes bad-price execution reaching a live pool swap, with direct LP principal loss.

---

### Likelihood Explanation

Any pool that deploys `PriceVelocityGuardExtension` without the admin calling `setLastMidPrice` before the first swap is vulnerable. This is the default state: the extension has no `initialize` hook, so the zero state is the starting condition for every pool. The first public swapper — any unprivileged user — triggers the bypass. No special permissions or oracle manipulation are required; the bypass is structural and reachable on every newly configured pool.

---

### Recommendation

Fail closed when no prior observation exists. Either:

1. **Revert if `prevMid == 0`** — require the admin to seed the price before the first swap is permitted:
   ```solidity
   if (prevMid == 0) revert NoPriorObservation();
   ```

2. **Add an `initialize` hook** — seed `lastMidPriceX64` from the pool's live oracle price at pool creation, analogous to requiring a minimum cardinality before a pool is eligible for guarded operations.

3. **Treat `prevMid == 0` as a full block** — if no prior observation exists, block the swap entirely rather than allowing unconstrained execution.

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured (`maxChangePerBlockE18 = 1e16`, i.e., 1% per block). Do **not** call `setLastMidPrice`.
2. Oracle price at pool deployment: `P0 = 1000 USD`. Oracle price at block N: `P1 = 2000 USD` (100% move, far exceeding the 1% cap).
3. Any user calls `pool.swap(...)` at block N. The pool fetches `bid/ask` from the oracle (P1-based). `PriceVelocityGuardExtension.beforeSwap` is called.
4. `prevMid = s.lastMidPriceX64 = 0`. The `if (prevMid != 0)` branch is skipped. The swap executes at P1 with no velocity check.
5. `s.lastMidPriceX64` is now set to `sqrt(bid1 * ask1)` ≈ P1.
6. LPs suffer losses from a swap at 2× the price the guard was configured to block.
7. Contrast: if `setLastMidPrice(pool, P0_encoded)` had been called before step 3, the swap at block N would revert with `PriceVelocityExceeded` because `(P1 - P0)^2 / P0^2 = 1.0 >> maxChange^2 * (1 + blockDiff)`. [5](#0-4)
