### Title
Router-Mediated Swap Passes Router Address as `sender` to `SwapAllowlistExtension`, Allowing Any User to Bypass Per-User Allowlist â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension::beforeSwap` hook gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the address the pool received as `msg.sender`. When a user swaps directly through the pool, `sender` is the user. When a user swaps through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the original user. If the router is allowlisted (which is required for any user to use the router), the per-user allowlist is completely bypassed for all router-mediated swaps.

The "two transactions / timed-threshold" framing in the question is inaccurate â€” the bypass is a single-transaction structural identity confusion, not a timing attack.

---

### Finding Description

**Call path â€” direct swap:**

`user â†’ pool.swap(...)` â†’ pool calls `_beforeSwap(msg.sender=user, ...)` â†’ extension checks `allowedSwapper[pool][user]` âś…

**Call path â€” router-mediated swap:**

`user â†’ router.exactInputSingle(...)` â†’ `router â†’ pool.swap(...)` â†’ pool calls `_beforeSwap(msg.sender=router, ...)` â†’ extension checks `allowedSwapper[pool][router]` âťŚ

In `MetricOmmPool::swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this as the first argument to the extension: [2](#0-1) 

The extension then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool â€” i.e., the router: [3](#0-2) 

The router calls `pool.swap(...)` directly with no mechanism to forward the original user's identity: [4](#0-3) 

---

### Impact Explanation

There are two failure modes, both harmful:

1. **Allowlist bypass (High impact):** The pool admin allowlists the router address so that users can use the router. Because the router is a public, permissionless contract, *any* address can call `exactInputSingle`/`exactOutputSingle`/`exactInput`/`exactOutput` and have the hook see `sender = router` (allowlisted). The per-user allowlist is completely defeated â€” any unprivileged user can swap on a pool that should have blocked them.

2. **Allowlisted users locked out of router (Medium impact):** If the pool admin does NOT allowlist the router, then even correctly allowlisted users cannot use the router â€” their swaps revert with `NotAllowedToSwap`. The router is unusable for allowlisted pools.

There is no configuration that correctly allows specific users to use the router while blocking others.

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router is affected. The router is the standard public entrypoint for swaps in the periphery. The pool admin has no way to fix this without replacing the extension, since the identity confusion is structural.

---

### Recommendation

The extension must receive the original user's identity, not the intermediary's. Two approaches:

1. **Pass `msg.sender` from the router as `extensionData`** and have the extension decode and verify it â€” but this is forgeable by any caller.
2. **Preferred:** The pool should pass both `msg.sender` (the direct caller) and an optional `originSender` (e.g., from `tx.origin` or a signed context). The extension can then check the origin. Alternatively, the router should be redesigned to call the pool with a verified user context, or the allowlist should gate at the router level with a separate `DepositAllowlistExtension`-style check that the router enforces before calling the pool.

The cleanest fix is for `beforeSwap` to receive a separate `origin` parameter (analogous to EIP-2771 trusted forwarder patterns) so the hook can check the true initiator.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for any router use
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack (single transaction):
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...)  // msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - extension checks allowedSwapper[pool][router] == true  âś… (passes)
  - attacker's swap executes on an allowlisted pool they were never authorized to use
```

The `allowAllSwappers` path has the same issue: if `allowAllSwappers[pool] = false` and only specific users are allowlisted, routing through the router bypasses the check entirely once the router itself is allowlisted.
