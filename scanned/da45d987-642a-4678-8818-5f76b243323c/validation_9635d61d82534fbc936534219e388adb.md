### Title
`DepositAllowlistExtension` Gates LP-Share Recipient (`owner`) Instead of Token Provider (`sender`), Allowing Non-Allowlisted Addresses to Bypass the Deposit Guard — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address actually providing tokens) and checks only `owner` (the LP-share recipient). Any address that is not on the allowlist can deposit tokens into a restricted pool by nominating an allowlisted address as `owner`. The pool admin's intent to gate token inflows is defeated.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` is declared with an unnamed, discarded first parameter:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol  L32-42
function beforeAddLiquidity(
    address,          // ← sender (actual caller / token provider) — IGNORED
    address owner,    // ← LP-share recipient — CHECKED
    uint80,
    LiquidityDelta calldata,
    bytes calldata
) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`msg.sender` inside the hook is the pool (the `onlyPool` pattern from `BaseMetricExtension`), so the effective check is:

```
allowedDepositor[pool][owner]
```

The address that actually transfers tokens into the pool — the `sender` forwarded by `ExtensionCalling._beforeAddLiquidity` — is never consulted.

**Attack path (direct pool call):**

1. Pool admin deploys a restricted pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. `bob` (not allowlisted) calls `pool.addLiquidity(owner = alice, salt, delta, extensionData)` directly.
3. The pool calls `extension.beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
4. The hook evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. `bob`'s tokens are transferred into the pool; LP shares are minted to `alice`.
6. `bob` has bypassed the deposit restriction and injected tokens into the pool without being allowlisted.

**Attack path (via `MetricOmmPoolLiquidityAdder`):**

The `MetricOmmPoolLiquidityAdder` accepts a caller-supplied `owner` and forwards it to the pool. The hook still sees `owner` (the user-supplied address) and `sender` = the LiquidityAdder contract. Because `sender` is ignored, any user can route through the adder, supply an allowlisted `owner`, and deposit tokens into the restricted pool. [2](#0-1) 

The `allowedDepositor` and `allowAllDepositors` mappings are keyed by `(pool, depositor)`, where the protocol's own documentation and the scanned-question control hint confirm the intended gated identity is the **economically relevant depositor** (the token provider), not the LP-share recipient:

> *"The attacker can separate payer from owner and can route through the liquidity adder, so the checked identity has to be exactly the one the pool intends to gate."* [3](#0-2) 

---

### Impact Explanation

- The deposit allowlist — the sole on-chain mechanism for restricting who may provide liquidity to a pool — is fully bypassable by any address that knows one allowlisted address.
- Tokens from unauthorized sources enter the pool, breaking the pool admin's access-control invariant.
- The non-allowlisted depositor loses their tokens (they accrue to the allowlisted `owner`'s LP position), making this a direct loss of user principal for the depositor and a broken core pool-access guarantee for the operator.
- Severity: **Medium** (access-control bypass with direct token loss for the depositor; no protocol-level insolvency, but the pool's restricted-access property is entirely negated).

---

### Likelihood Explanation

- Exploitable by any unprivileged address with no special setup beyond knowing one allowlisted address (which is readable from `allowedDepositor` public mapping or emitted events).
- Works through both the direct pool call and the public `MetricOmmPoolLiquidityAdder` path.
- No admin cooperation or malicious token required.

---

### Recommendation

Replace the unnamed, discarded first parameter with `sender` and gate on it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(
    address sender,   // actual token provider
    address owner,
    uint80,
    LiquidityDelta calldata,
    bytes calldata
) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the LP-share recipient, check both. Update `setAllowedToDeposit`, `isAllowedToDeposit`, and all documentation to reflect which identity is gated. [4](#0-3)
