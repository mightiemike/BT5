### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument forwarded by the pool, which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is the caller, `sender` is the **router address**, not the end user. This means the allowlist either (a) is bypassed by any user who routes through the router if the router is allowlisted, or (b) blocks all allowlisted users from using the router if the router is not allowlisted.

---

### Finding Description

The call chain for a router-mediated swap is:

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient, ...)          [msg.sender = router]
     → _beforeSwap(msg.sender=router, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← router address, not the end user
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, never touching the actual user's allowlist entry.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the position owner), which is the actual user regardless of who calls `addLiquidity`. The swap extension has no equivalent "actual user" parameter to check — the pool's `swap()` signature has no `owner`; the only user-identifying field is `sender`, which is the direct caller.

---

### Impact Explanation

**Bypass path (router is allowlisted):** A pool admin who wants to allow the router to be used by approved users will add the router to the allowlist: `allowedSwapper[pool][router] = true`. This immediately grants every user — including those explicitly not allowlisted — the ability to swap through the router. The allowlist is completely defeated. Any user can drain pool liquidity at oracle prices during a restricted phase (e.g., pre-DEX-listing, post-incident investigation).

**Lockout path (router is not allowlisted):** A pool admin who adds individual users to the allowlist (`allowedSwapper[pool][user] = true`) but does not add the router will find that those users cannot swap through `MetricOmmSimpleRouter` at all, because the extension sees `sender = router` and rejects it. Allowlisted users are forced to implement `IMetricOmmSwapCallback` themselves and call `pool.swap()` directly — the standard periphery entry point is broken for them.

Both outcomes are fund-impacting: the first allows unauthorized token extraction from the pool; the second renders the swap allowlist unusable with the production router.

---

### Likelihood Explanation

The `SwapAllowlistExtension` is a production periphery contract intended for real pool deployments. Any pool that configures it alongside `MetricOmmSimpleRouter` — the standard swap entry point — will encounter one of the two broken states above. The admin has no way to configure the allowlist correctly: allowlisting the router opens it to everyone; not allowlisting it blocks everyone who uses the router. The trigger requires only a normal user swap through the router, which is the expected usage pattern.

---

### Recommendation

The `beforeSwap` hook receives `sender` (the direct pool caller) and `recipient`. Neither is the end user when a router is involved. Two remediation options:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData` and the extension decodes it. This requires a trusted encoding convention and is fragile.

2. **Check `sender` as the authoritative identity and document that direct pool calls are required**: Remove the router from the trusted path and require allowlisted users to call `pool.swap()` directly. Document this constraint explicitly.

3. **Preferred — add a `payer` / `originator` field to the swap interface**: Extend `pool.swap()` with an explicit `originator` address (analogous to `owner` in `addLiquidity`) that the router sets to `msg.sender`. The extension then checks `allowedSwapper[pool][originator]`. This mirrors how the deposit allowlist correctly uses `owner`.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `swapExtension.setAllowedToSwap(pool, router, true)` — intending to allow router-mediated swaps for approved users.
3. A non-allowlisted user (`attacker`) calls `router.exactInputSingle(...)`.
4. The router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. `attacker` successfully swaps despite never being added to the allowlist.

Alternatively, if the admin does NOT allowlist the router:

1. Admin calls `swapExtension.setAllowedToSwap(pool, user1, true)`.
2. `user1` calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router]` → `false` → `NotAllowedToSwap` revert.
4. `user1` cannot swap through the standard router despite being explicitly allowlisted.
