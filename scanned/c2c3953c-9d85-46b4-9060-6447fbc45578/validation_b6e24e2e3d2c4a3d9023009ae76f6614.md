### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Original User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router**, not the original user. If the pool admin allowlists the router address (a natural action to enable router-mediated swaps), every unprivileged user can bypass the per-user swap gate by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool — which is `msg.sender` of the pool's own `swap()` call. [1](#0-0) 

The pool passes its own `msg.sender` as `sender` to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient, zeroForOne, ...
);
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

So `msg.sender` inside `pool.swap()` is the **router contract**, not the original user. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` and gates by `owner` (the economic actor), demonstrating the intended design:

```solidity
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [4](#0-3) 

The swap extension has no equivalent: it gates by `sender` (the immediate caller), not by the original user.

---

### Impact Explanation

A pool admin who wants to allow router-mediated swaps for their allowlisted users has only one option: `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, **every user** — including those not individually allowlisted — can call `router.exactInputSingle(pool, ...)` and the extension passes, because it sees `sender = router` which is allowlisted. The per-user curation policy is completely defeated. Any non-allowlisted user can trade on a pool that was designed to be permissioned (e.g., KYC-gated, institutional, or regulatory-compliant pools), causing a broken core pool functionality and potential regulatory/compliance loss.

---

### Likelihood Explanation

The trigger is a single, natural admin action: allowlisting the router so that allowlisted users can use the standard periphery. This is the expected operational pattern for any pool that wants to support the official router. The attacker needs no special access — any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the target pool. The router is a factory-validated contract, so `onlyPool` guards on the extension are satisfied. The bypass is fully unprivileged once the router is allowlisted.

---

### Recommendation

Gate the swap allowlist by the **original user** rather than the immediate caller. The pool already passes both `sender` (immediate caller) and `recipient` to the hook. For router-mediated swaps, the original user is `params.recipient` in single-hop exact-input flows, but this is not reliable across all router paths.

The correct fix is to mirror `DepositAllowlistExtension`: the pool should pass the **payer/originator** identity as a dedicated argument, or the `SwapAllowlistExtension` should check `recipient` (the economic beneficiary) rather than `sender`. Alternatively, the router should forward the original `msg.sender` as an authenticated field in `extensionData`, and the extension should decode and verify it.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Non-allowlisted attacker calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — `msg.sender` at the pool is the router.
5. Pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` checks `allowedSwapper[pool][router]` → `true` → passes.
7. Attacker's swap executes on a pool that should have blocked them.

The attacker is never individually allowlisted; the bypass is entirely through the router intermediary. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
