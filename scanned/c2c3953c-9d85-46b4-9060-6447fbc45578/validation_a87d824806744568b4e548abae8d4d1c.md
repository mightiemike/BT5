### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the originating user. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently opens the gate to every user on the internet, because the router's single address passes the check regardless of who called the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool level: [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The result is a structural mismatch: the allowlist is keyed on the direct caller of `pool.swap()`, but the economically relevant actor is the EOA who called the router. There is no mechanism in the router to forward the originating user's address as `sender`.

### Impact Explanation

**Allowlist bypass (High):** A pool admin who allowlists the router address — the natural step to permit router-mediated swaps for approved users — simultaneously permits every unapproved user to trade through the same router. The allowlist provides zero curation once the router is on it. Non-allowlisted users can drain liquidity from a pool that was designed to be restricted to a specific counterparty set.

**Broken allowlist for direct-EOA allowlisting (Medium):** If the admin instead allowlists individual EOA addresses (not the router), those EOAs cannot use the router at all, because the router's address fails the check. The pool's primary user-facing swap path is broken for every allowlisted user.

Both outcomes represent broken core pool functionality with direct fund-impact consequences: either unauthorized parties trade on a curated pool, or authorized parties cannot trade at all.

### Likelihood Explanation

The router is the canonical user-facing entrypoint documented in the protocol. Pool admins who configure a swap allowlist and want to support normal user flows will naturally allowlist the router. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — any public user can call `exactInputSingle` on the router pointing at the curated pool.

### Recommendation

The `sender` argument forwarded to extensions should represent the **originating user**, not the intermediary contract. Two complementary fixes:

1. **Router-side:** Have the router pass the originating `msg.sender` explicitly as a parameter to `pool.swap()` (e.g., via a dedicated `sender` field in the swap params), and have the pool forward that value to extensions instead of its own `msg.sender`.

2. **Extension-side (defense-in-depth):** `SwapAllowlistExtension` could accept an optional `sender` override embedded in `extensionData`, verified against a router-signed payload, so the extension can authenticate the true originator even when called through an intermediary.

Until fixed, pool admins must not allowlist the router address; they must require users to call `pool.swap()` directly, which defeats the purpose of the periphery layer.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to allow router-mediated swaps for approved users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls MetricOmmSimpleRouter.exactInputSingle({
        pool: curated_pool,
        recipient: attacker,
        ...
    })
  - Router calls pool.swap(...) → msg.sender at pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true ✓
  - Swap executes for attacker despite attacker never being allowlisted

Result:
  - attacker successfully trades on a pool designed to exclude them
  - allowlist provides no protection once the router is permitted
```

The existing unit test `test_allowedSwapSucceeds` in `FullMetricExtensionTest` allowlists `address(callers[0])` (a `TestCaller` contract that calls the pool directly), which masks the router-mediated bypass because the test never routes through `MetricOmmSimpleRouter`. [6](#0-5)
