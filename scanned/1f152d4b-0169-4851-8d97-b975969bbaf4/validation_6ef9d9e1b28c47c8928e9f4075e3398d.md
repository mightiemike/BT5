### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Trader, Allowing Any User to Bypass Curated-Pool Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router` and forwards that address as `sender` to the extension. The extension therefore checks whether the **router** is allowlisted, not the actual user. If the router is allowlisted (required for normal periphery operation), every user on the internet can bypass the curated-pool restriction by routing through the router.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ..., extensionData)   // msg.sender = router
              → _beforeSwap(msg.sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → allowedSwapper[pool][router]  ← checked, NOT the user
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly: [1](#0-0) 

The pool's `swap()` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

The allowlist is keyed on `(pool, router)` instead of `(pool, actual_user)`. This creates two broken states:

| Router allowlisted? | Effect |
|---|---|
| **Yes** (required for normal use) | Every user bypasses the allowlist by routing through the router |
| **No** | Individually allowlisted users cannot use the router at all |

The same structural flaw exists in `DepositAllowlistExtension.beforeAddLiquidity`, which ignores the `sender` argument entirely and checks `owner` — meaning the payer identity (the economically relevant actor for token pulls) is never gated: [5](#0-4) 

---

### Impact Explanation

A pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict trading to KYC'd addresses or specific counterparties. The admin must allowlist the router to enable normal user-facing operation. Once the router is allowlisted, any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` and trade against the pool's LP reserves without restriction. The allowlist provides zero protection. LP funds are exposed to the full universe of traders the admin intended to exclude, including adversarial actors who could exploit favorable oracle prices or drain specific bins.

**Impact: High** — direct loss of LP principal through unrestricted access to a pool whose entire security model depends on the allowlist.

---

### Likelihood Explanation

The router is the canonical user-facing entry point documented and deployed by the protocol. Any pool admin who wants users to swap through the router must allowlist it. The bypass requires no special knowledge: any user who calls `exactInputSingle` on an allowlisted-router pool automatically bypasses the guard. The trigger is a single standard router call.

**Likelihood: High** — the bypass is automatic for every router-mediated swap on any pool that has allowlisted the router.

---

### Recommendation

The extension must check the **original user**, not the intermediary. The pool already receives `msg.sender` from the router, but the router's `msg.sender` is the user. The fix requires the router to forward the real caller, or the extension must check `sender` only when the caller is a trusted periphery contract and fall back to a user-supplied identity otherwise.

The cleanest fix mirrors the BakerFi mitigation: enforce that the checked identity is always the economically relevant actor. For `SwapAllowlistExtension`, the pool should pass the original user address as `sender`. One approach is for the router to pass the user address in `extensionData` and have the extension decode it, with the pool verifying the binding. Alternatively, the pool can expose a `swapWithSender` entry point that accepts an explicit `sender` address and validates it against `msg.sender` or a trusted router registry.

For `DepositAllowlistExtension`, the `sender` (payer/caller) should also be checked, not only `owner`, since the payer is the actor whose token approval is consumed.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `alice` is allowlisted.
// Pool admin must allowlist the router for normal operation:
//   extension.setAllowedToSwap(pool, address(router), true);

// Attack: bob (not allowlisted) calls the router directly.
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:            address(pool),
        recipient:       bob,
        zeroForOne:      true,
        amountIn:        1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:   0,
        deadline:        block.timestamp,
        tokenIn:         token0,
        extensionData:   ""
    })
);
// The extension checks allowedSwapper[pool][router] == true → passes.
// Bob's swap executes against LP reserves despite not being allowlisted.
// Alice's intended exclusivity is completely bypassed.
```

The pool's `_beforeSwap` receives `sender = address(router)`, the extension checks `allowedSwapper[pool][router]`, finds it `true`, and allows the swap. The actual caller `bob` is never examined. [6](#0-5) [7](#0-6)
