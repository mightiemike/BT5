### Title
`DepositAllowlistExtension` gates on LP position `owner` instead of the actual depositor `sender`, allowing any unprivileged address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument (the actual caller who pays tokens) and instead checks the `owner` argument (the LP position recipient). Because `addLiquidity` lets any caller specify an arbitrary `owner`, any address not on the allowlist can bypass the guard by naming an allowed address as the position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts two distinct actor addresses:

- `msg.sender` → forwarded as `sender` to the extension; this is the address that pays tokens via `metricOmmModifyLiquidityCallback`.
- `owner` → the address that receives the LP position shares; freely chosen by the caller. [1](#0-0) 

The pool calls `_beforeAddLiquidity(msg.sender, owner, ...)`, passing both actors to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely (unnamed first parameter) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The protocol explicitly supports depositing on behalf of another address. `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a separate `owner` and `payer` (`msg.sender`), and the test suite confirms this pattern: [4](#0-3) [5](#0-4) 

Because the extension checks `allowedDepositor[pool][owner]` instead of `allowedDepositor[pool][sender]`, any address not on the allowlist can pass the guard by supplying an allowed address as `owner`.

---

### Impact Explanation

**Admin-boundary break / broken core pool functionality.**

The pool admin deploys `DepositAllowlistExtension` specifically to restrict who may deposit tokens into the pool. That invariant is fully defeated:

1. **Allowlist bypass**: An unauthorized address (e.g., a sanctioned or non-KYC'd address) calls `addLiquidityExactShares(pool, allowedAddress, ...)`. The extension sees `allowedDepositor[pool][allowedAddress] == true` and passes. The unauthorized address's tokens enter the pool.

2. **Forced LP position / fund loss for victim**: The LP shares are credited to `allowedAddress`, who did not consent. If the pool subsequently loses value (impermanent loss, bad oracle, stop-loss trigger), `allowedAddress` suffers a direct loss of principal on shares they never chose to hold.

3. **Compliance violation**: Pools using this extension for regulatory gating (KYC, sanctions screening) are rendered ineffective; the actual payer's identity is never checked.

---

### Likelihood Explanation

- The bypass requires no special privilege — any EOA or contract can call `addLiquidity` or `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an arbitrary `owner`.
- The `MetricOmmPoolLiquidityAdder` router explicitly exposes an `owner` parameter separate from `msg.sender`, making the attack path trivially accessible from the standard periphery.
- The attacker only needs to know one address that is on the allowlist (publicly readable via `allowedDepositor`).

---

### Recommendation

Check `sender` (the actual payer) instead of `owner` (the position recipient):

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) { ... }
}

// After (correct):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) { ... }
}
``` [3](#0-2) 

Note that `SwapAllowlistExtension.beforeSwap` already does this correctly — it checks `sender` (the actual caller), not `recipient`: [6](#0-5) 

---

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension (BEFORE_ADD_LIQUIDITY_ORDER set)
  - Pool admin calls setAllowedToDeposit(pool, alice, true)
  - Bob is NOT on the allowlist

Attack:
  1. Bob calls MetricOmmPoolLiquidityAdder.addLiquidityExactShares(
         pool,
         alice,   // owner — alice IS on the allowlist
         salt,
         deltas,
         maxAmount0, maxAmount1,
         ""
     )
  2. Pool calls _beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)
  3. DepositAllowlistExtension checks allowedDepositor[pool][alice] == true → PASSES
  4. Bob's tokens are pulled via metricOmmModifyLiquidityCallback
  5. Alice receives LP shares she did not request
  6. Bob has deposited into an allowlist-gated pool without being on the allowlist
  7. If the pool loses value, Alice suffers a direct loss on the forced position
```
