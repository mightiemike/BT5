### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unprivileged Actor to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` validates the LP-position `owner` (a caller-controlled parameter) against the allowlist instead of the actual depositor `sender` (the address that pays tokens via callback). Any unprivileged actor can bypass the allowlist by calling `pool.addLiquidity(owner = allowlistedAddress, …)`, depositing funds into a restricted pool while assigning the resulting LP shares to an allowlisted address.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an explicit `owner` parameter and passes both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the guard checks the second argument — `owner` — not the first argument `sender`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

`owner` is a free parameter supplied by the caller of `addLiquidity`. It is never required to equal `msg.sender`. The actual token payment flows through the `IMetricOmmModifyLiquidityCallback` invoked on `msg.sender` (the `sender`), not on `owner`. Therefore the allowlist check is performed on the wrong address: the address that receives LP shares, not the address that pays tokens.

Compare with `SwapAllowlistExtension`, which correctly checks `sender` (the address that initiates and pays for the swap):

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

The asymmetry between the two allowlist extensions confirms that checking `owner` in `DepositAllowlistExtension` is unintended.

---

### Impact Explanation

The deposit allowlist is the primary access-control mechanism for restricted pools. Its bypass means:

- Any unprivileged actor can inject liquidity into a pool the admin intended to restrict (e.g., for compliance, LP-composition, or anti-dilution reasons).
- The admin-configured guard is rendered entirely ineffective by a single caller-controlled parameter.
- The allowlisted `owner` receives LP shares they did not request and did not pay for; the unauthorized `sender` pays the tokens. This constitutes an admin-boundary break where an unprivileged path circumvents an admin-set guard with direct LP-asset consequences (unsolicited LP positions assigned to allowlisted addresses, unauthorized liquidity added to the pool).

---

### Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(owner = knownAllowlistedAddress, …)`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or by reading `allowedDepositor`. Likelihood is **High**.

---

### Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

If the intent is to gate LP-position ownership (not the payer), the extension name, NatSpec, and `setAllowedToDeposit` parameter name (`depositor`) should be updated to reflect that, and the bypass risk documented explicitly.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` configured in `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, Alice, true)` — Alice is the only allowlisted address.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner = Alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(Bob, Alice, …)` → extension receives `sender = Bob`, `owner = Alice`.
5. Extension evaluates `allowedDepositor[pool][Alice]` → `true` → no revert.
6. Pool processes the liquidity addition; Bob's callback is invoked and Bob pays the tokens.
7. Alice's LP position is credited with the new shares.
8. Bob has successfully deposited into a pool he is not allowlisted for; the allowlist is bypassed. [4](#0-3) [5](#0-4)

contract DepositAllowlistExtension is BaseMetricExtension, IDepositAllowlistExtension {
  mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;
  mapping(address pool => bool) public allowAllDepositors;

  constructor(address factory_) BaseMetricExtension(factory_) {}

  function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
    emit AllowedToDepositSet(pool_, depositor, allowed);
  }

  function setAllowAllDepositors(address pool_, bool allowed) external onlyPoolAdmin(pool_) {
    allowAllDepositors[pool_] = allowed;
