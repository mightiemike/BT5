### Title
`SwapAllowlistExtension` gates the router address instead of the end user, allowing any trader to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end user. A pool admin who allowlists the router to enable periphery-mediated swaps simultaneously opens the allowlist to every user on the network, because the extension can no longer distinguish individual end users.

---

### Finding Description

**Analog classification**: This is a direct "wrong-actor binding" analog to the Vyper signed-integer array-index bug. In Vyper, the compiler accepts a signed integer where only an unsigned index is valid, silently permitting access to unintended storage slots. Here, the extension accepts the router's address as the "swapper" identity where only the end user's address is valid, silently permitting access to a restricted pool.

**Root cause — `SwapAllowlistExtension.beforeSwap`**:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct — it is the caller of the extension). `sender` is the first argument forwarded by the pool. [1](#0-0) 

**The pool sets `sender` = `msg.sender` of the `swap` call**:

```solidity
_beforeSwap(
    msg.sender,   // <-- this is the router when called via periphery
    recipient,
    ...
);
``` [2](#0-1) 

**The router calls `pool.swap(...)` directly**, so `msg.sender` at the pool is the router:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

The end user's address (`msg.sender` of the router call) is **never forwarded** to the pool or the extension. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the second argument), not `sender`. The pool always passes the actual position owner as `owner` regardless of who calls `addLiquidity`, so the deposit allowlist correctly gates the economically relevant actor: [4](#0-3) 

The swap allowlist has no equivalent correct identity to check — the end user is simply absent from the hook arguments.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties, institutional traders) faces an irreconcilable dilemma:

1. **Do not allowlist the router** → all router-mediated swaps revert for every user, including allowlisted ones. The standard periphery path is broken.
2. **Allowlist the router** → the allowlist is bypassed for every user on the network. Any address can call `MetricOmmSimpleRouter.exactInputSingle` and trade on the restricted pool.

In case 2, the pool receives tokens from and sends tokens to any counterparty, defeating the curation policy entirely. This is a direct loss of the access-control invariant the pool admin paid to configure.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the documented, supported periphery path for swaps.
- Any user who discovers the pool is allowlist-restricted can trivially route through the router instead of calling the pool directly.
- No privileged access, no special tokens, no malicious setup required — a single `exactInputSingle` call suffices.
- The pool admin has no on-chain mechanism to fix this without disabling the router entirely (which breaks all legitimate router users).

---

### Recommendation

The `beforeSwap` hook must gate the **end user**, not the immediate pool caller. Two viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation and is trust-dependent.

2. **Check `recipient` instead of `sender`** (partial fix): For single-hop swaps the recipient is often the end user, but for multi-hop swaps it may be the router itself, so this is incomplete.

3. **Preferred — mirror the deposit pattern**: Introduce a dedicated `swapper` parameter (analogous to `owner` in `addLiquidity`) that the pool always sets to the economically relevant actor, independent of who calls `swap`. The router would pass `msg.sender` explicitly as this parameter.

Until fixed, pool admins should not rely on `SwapAllowlistExtension` to restrict access on pools where `MetricOmmSimpleRouter` is a reachable entry point.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  1. attacker (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: restrictedPool,
           recipient: attacker,
           ...
       })
  2. Router calls pool.swap(attacker, ...) with msg.sender = router.
  3. Pool calls _beforeSwap(router, attacker, ...).
  4. Extension checks allowedSwapper[pool][router] == true → passes.
  5. Swap executes. attacker receives output tokens.

Result:
  attacker successfully swaps on a pool they were never allowlisted for.
  The allowlist check passed because it evaluated the router's address,
  not the attacker's address — the wrong actor, the wrong "type" of identity.
``` [5](#0-4) [2](#0-1) [6](#0-5)

    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
