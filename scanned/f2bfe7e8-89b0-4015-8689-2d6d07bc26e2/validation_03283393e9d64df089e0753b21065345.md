### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. Because `MetricOmmPool.swap` always passes `msg.sender` (the immediate caller) as `sender`, any swap routed through `MetricOmmSimpleRouter` presents the router's address to the allowlist instead of the actual user's address. A pool admin who allowlists the router to enable router-mediated swaps for curated users simultaneously opens the pool to every unprivileged user who calls the router.

### Finding Description

**Call chain:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`).
2. The router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — here `msg.sender` seen by the pool is the **router**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, forwarding the router's address as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches to `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. `SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is the **router**, not the originating user.

The pool admin faces an impossible choice:

| Admin configuration | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all — broken functionality |
| Router **allowlisted** | Every unprivileged user bypasses the allowlist by routing through the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

**Relevant code:**

`MetricOmmPool.swap` passes `msg.sender` as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards it verbatim: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks the forwarded `sender` (the router) against the allowlist: [3](#0-2)

The same pattern applies to `exactInput` and `exactOutput`: [5](#0-4) 

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional traders, or protocol-internal actors) can be fully bypassed by any unprivileged user who routes through `MetricOmmSimpleRouter`. Once the router is allowlisted — the only way to make the router usable for legitimate users — the allowlist ceases to function as an access control boundary. Any user can trade in the restricted pool, violating the pool admin's configured policy. This constitutes an admin-boundary break where an unprivileged path bypasses a factory-configured guard, matching the allowed impact gate.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface documented and deployed by the protocol. Any pool operator who configures `SwapAllowlistExtension` and also wants their allowlisted users to be able to use the standard router must allowlist the router address. This is the natural and expected configuration, making the bypass reachable in any realistic curated-pool deployment. No special timing, flash loans, or privileged access is required — a single `exactInputSingle` call suffices.

### Recommendation

Pass the economically relevant actor — the originating user — rather than the immediate pool caller as the identity checked by the allowlist. Two complementary approaches:

1. **In the router:** Forward the originating user's address to the pool via an additional parameter or a dedicated field in `extensionData`, and have the extension decode it. This requires a protocol-level convention for "true sender" in extension payloads.

2. **In the extension:** Change `SwapAllowlistExtension.beforeSwap` to check `recipient` (the address that receives output tokens) rather than `sender` when the sender is a known router, or require the pool to pass the originating user explicitly.

3. **Simplest safe default:** Document that `SwapAllowlistExtension` only gates direct pool callers and must not be used with the public router if per-user curation is required. Provide a router-aware allowlist variant that decodes the true user from `extensionData`.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)      // router allowlisted so alice can use it

Attack:
  - bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(recipient=bob, ...)  →  msg.sender to pool = router
  - Pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  →  PASSES
  - Bob's swap executes successfully despite not being on the allowlist.

Expected: revert NotAllowedToSwap
Actual:   swap succeeds — allowlist bypassed
```
