### Title
Wrong-Actor Binding in `DepositAllowlistExtension.beforeAddLiquidity` Allows Unprivileged Deposit Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor/caller) and instead gates on `owner` (the LP position recipient, which is a caller-controlled argument). Any address not on the allowlist can bypass the deposit guard by supplying an allowlisted address as `owner`, while the pool admin's intended curation policy — gating by depositor — is never enforced.

---

### Finding Description

`IMetricOmmExtensions.beforeAddLiquidity` receives two distinct actor addresses:

- `sender` — `msg.sender` of the `pool.addLiquidity(...)` call; the address that actually initiates the deposit and must satisfy the callback.
- `owner` — the LP position recipient; a **caller-supplied** argument that can be set to any address. [1](#0-0) 

The pool wires these correctly in `ExtensionCalling._beforeAddLiquidity`: [2](#0-1) 

And in `MetricOmmPool.addLiquidity`, `msg.sender` is passed as `sender` and the caller-supplied `owner` is passed as `owner`: [3](#0-2) 

`DepositAllowlistExtension.beforeAddLiquidity` then **silently drops `sender`** (first parameter is unnamed) and checks only `owner`: [4](#0-3) 

The extension's own admin setter names the gated party `depositor`, confirming the intended check is on the depositor (sender), not the position owner: [5](#0-4) 

---

### Impact Explanation

**Allowlist bypass (unprivileged path):** An address `B` that is **not** on the allowlist calls:

```solidity
pool.addLiquidity(allowlistedAddressA, salt, deltas, callbackData, extensionData);
```

The extension evaluates `allowedDepositor[pool][A]` → `true` and does not revert. `B` satisfies the token callback and the position is minted to `A`. The pool admin's curation policy — restricting which addresses may deposit — is never enforced against the actual depositor.

**Inverse breakage (legitimate use blocked):** An allowlisted depositor `A` (e.g., a liquidity-manager contract) that calls `addLiquidity` with `owner = B` (a non-allowlisted beneficiary) is **incorrectly blocked**, because the extension checks `B` (owner) rather than `A` (sender). This makes the extension unusable for any router or helper that separates the depositor from the position owner.

---

### Likelihood Explanation

The `addLiquidity` function is permissionless — any address can call it with any `owner` argument. No privileged setup is required. The bypass is reachable in a single transaction by any external actor on any pool that has `DepositAllowlistExtension` configured. The only cost to the attacker is the token payment required by the callback, which is attributed to the allowlisted `owner` address.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns with the interface's documented semantics, the `setAllowedToDeposit(pool, depositor, allowed)` naming, and the analogous correct implementation in `SwapAllowlistExtension.beforeSwap`, which already gates on `sender`: [6](#0-5) 

---

### Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension; only `allowedDepositor` is on the list.
// `attacker` is NOT on the allowlist.

vm.startPrank(attacker);

// Attacker supplies `allowedDepositor` as owner — extension checks owner, not sender.
pool.addLiquidity(
    allowedDepositor,   // owner: allowlisted → extension passes
    salt,
    deltas,
    callbackData,       // attacker's contract pays tokens in callback
    extensionData
);

// Extension never checked `attacker` (sender). Deposit succeeded.
// LP position is minted to `allowedDepositor`, tokens came from `attacker`.
// Pool admin's curation policy is violated: an un-allowlisted address deposited.

vm.stopPrank();
```

The `removeLiquidity` path enforces `msg.sender == owner`, so the attacker cannot reclaim the tokens — but the allowlist invariant is broken regardless, and the pool has received liquidity from an address the admin explicitly did not authorize.
