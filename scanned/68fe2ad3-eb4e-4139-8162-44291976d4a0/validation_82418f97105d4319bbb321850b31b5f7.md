### Title
Swap Allowlist Guard Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass Pool Restrictions — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol` / `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which dispatches it to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool boundary is the **router contract**, not the originating user. The allowlist is keyed by `(pool, sender)`, so it evaluates the router's address — not the actual swapper's address. Any user who is not on the allowlist can bypass the restriction by routing through the public router.

---

### Finding Description

In `MetricOmmPool.swap()`, the `_beforeSwap` hook is invoked with `msg.sender` as the `sender` parameter:

```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of the pool
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes this `sender` and forwards it verbatim to every extension registered in `BEFORE_SWAP_ORDER`:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
     packedSlot0Initial, bidPriceX64, askPriceX64, extensionData)
)
``` [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` performs its allowlist lookup keyed by `(pool, sender)` — i.e., it checks whether the **direct caller of the pool** is permitted, not the originating end-user. When a user calls `MetricOmmSimpleRouter.exactInput*` (or any other router entry point), the router calls `pool.swap(...)`, making `msg.sender` at the pool equal to the router's address. The extension therefore evaluates the router's allowlist status, not the user's. [3](#0-2) 

---

### Impact Explanation

A pool operator who deploys a KYC-gated, compliance-restricted, or invite-only pool and attaches `SwapAllowlistExtension` intends to prevent non-allowlisted addresses from executing swaps. Because the guard evaluates the router's address instead of the originating user's address, any non-allowlisted user can:

1. Call `MetricOmmSimpleRouter.exactInput` or `exactOutput` targeting the restricted pool.
2. The router calls `pool.swap(...)` — `msg.sender` at the pool is the router.
3. If the router is allowlisted (or not explicitly blocked), `SwapAllowlistExtension.beforeSwap` passes.
4. The swap executes as if the user were permitted.

This breaks the core access-control invariant of the pool, allowing unauthorized extraction of liquidity at oracle-derived prices from a pool whose LP providers deposited under the assumption that only vetted counterparties could trade against them. This constitutes a direct loss of LP principal through unauthorized swap settlement.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any user can call it.
- No privileged setup is required beyond the pool operator having deployed a swap-allowlisted pool (a normal, intended configuration).
- The bypass requires only a single router call; no flash loans, callbacks, or multi-step exploits are needed.
- The router is a natural entry point for users, so the bypass is on the default happy path.

---

### Recommendation

`SwapAllowlistExtension.beforeSwap` must gate on the **originating user**, not the direct pool caller. Two approaches:

1. **Check `recipient` instead of `sender`**: For swap allowlists, the economically relevant actor is typically the recipient of output tokens. However, this can also be spoofed.
2. **Require the pool's periphery to forward the original `msg.sender` via `extensionData`**: The extension decodes the true originator from the payload and verifies it, with the pool's callback verifying the payer identity.
3. **Disallow router-mediated swaps on allowlisted pools at the router level**: The router checks whether the target pool has a swap allowlist and reverts if the caller is not on it — but this is fragile and easily bypassed by a custom router.

The most robust fix is option 2: pass the true originator through `extensionData` and have the extension verify it, combined with a trusted-forwarder pattern so the extension can authenticate the claim.

### Proof of Concept

```
Setup:
  - Pool P is deployed with SwapAllowlistExtension E.
  - E.allowedSwapper[P][alice] = true   (alice is allowed)
  - E.allowedSwapper[P][bob]   = false  (bob is NOT allowed)
  - MetricOmmSimpleRouter R is deployed (public, no access control).

Attack (bob bypasses allowlist):
  1. bob calls R.exactInputSingle({pool: P, ...})
  2. R calls P.swap(recipient=bob, ...) → msg.sender at P = address(R)
  3. P._beforeSwap(sender=address(R), ...) is dispatched to E.
  4. E checks allowedSwapper[P][address(R)]:
       - If R is allowlisted → passes immediately.
       - If allowAll[P] = true → passes immediately.
       - If R is not explicitly blocked → depends on default policy.
  5. Swap executes; bob receives token0/token1 from the restricted pool.
  6. LP providers who deposited expecting only KYC'd counterparties suffer
     unauthorized price impact and potential value extraction.

Direct call (correctly blocked):
  1. bob calls P.swap(...) directly → msg.sender = bob.
  2. E checks allowedSwapper[P][bob] = false → reverts.
``` [4](#0-3) [3](#0-2)
