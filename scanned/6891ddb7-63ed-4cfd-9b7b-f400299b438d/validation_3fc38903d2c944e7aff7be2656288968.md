### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist Guard — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the deposit allowlist against the `owner` parameter (the position beneficiary) rather than the `sender` parameter (the actual caller providing tokens). Because `sender` and `owner` can be distinct addresses, any unprivileged actor can bypass the allowlist by calling `addLiquidity` with an authorized `owner` address.

---

### Finding Description

The `beforeAddLiquidity` hook signature receives two address arguments: `sender` (the `msg.sender` of the pool call — the party actually initiating the deposit and providing tokens) and `owner` (the address that will own the resulting liquidity position). The extension silently discards `sender` and gates only on `owner`:

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The `sender` argument (first `address`) is unnamed and never read. The check `allowedDepositor[msg.sender][owner]` passes whenever `owner` is on the allowlist, regardless of who `sender` is.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`:

```solidity
// SwapAllowlistExtension.sol line 31-41
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [2](#0-1) 

The inconsistency is structural: the deposit guard checks the wrong actor.

---

### Impact Explanation

Any address not on the deposit allowlist can call `pool.addLiquidity(authorizedOwner, salt, deltas, ...)` where `authorizedOwner` is any address already approved. The extension sees `owner` = authorized address → passes. The unauthorized `sender` successfully injects liquidity into the pool, bypassing the permissioned deposit gate entirely.

Consequences:
- **Allowlist invariant broken**: The pool admin's intent to restrict depositors to a known set is nullified.
- **Pool state manipulation**: An unauthorized actor can shift bin balances, alter the pool's liquidity distribution, and affect pricing for other participants.
- **Permissioned pool compromise**: Pools designed for KYC'd or whitelisted LPs (the primary use-case of `DepositAllowlistExtension`) accept deposits from arbitrary addresses.

Severity: **Medium** — broken core access-control invariant with direct pool-state impact; no privileged setup required beyond knowing any authorized `owner` address (which is public on-chain from prior `setAllowedToDeposit` events).

---

### Likelihood Explanation

- The authorized `owner` addresses are discoverable from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`.
- No special role or token approval is needed beyond the standard ERC-20 approval to the pool.
- The attack path is a single direct call to `pool.addLiquidity`. [3](#0-2) 

---

### Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the position owner (not the caller), document this explicitly and ensure the pool enforces that `sender == owner` when the allowlist is active, or gate on both.

---

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension in beforeAddLiquidity slot.
  - Admin calls setAllowedToDeposit(pool, alice, true)  → alice is authorized.
  - bob is NOT on the allowlist.

Attack:
  1. bob calls pool.addLiquidity(
         owner    = alice,   // authorized address
         salt     = 0,
         deltas   = { binIdxs: [0], shares: [10_000] },
         ...
     )
  2. Pool calls extension.beforeAddLiquidity(
         sender = bob,    // actual caller — NOT checked
         owner  = alice,  // authorized — CHECK PASSES
         ...
     )
  3. Extension returns beforeAddLiquidity.selector → no revert.
  4. bob's tokens are deposited; liquidity is credited to alice's position.
  5. Deposit allowlist is bypassed.
```

The existing unit test `test_blocksDepositWhenDepositorNotAllowed` in `FullMetricExtensionTest` calls `_addLiquidity(0, ...)` where the caller and owner are the same address, so it does not exercise the `sender ≠ owner` path and does not catch this bypass. [4](#0-3) [1](#0-0)
