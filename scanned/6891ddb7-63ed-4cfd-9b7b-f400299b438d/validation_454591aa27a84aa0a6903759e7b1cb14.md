### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any Caller to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is `msg.sender` inside `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the **router contract**, not the end-user. The extension therefore checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router (required for legitimate router-mediated swaps to work) inadvertently opens the pool to every user, completely defeating the allowlist.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
// ExtensionCalling.sol L160-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool (`msg.sender` = pool):

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

At this point `msg.sender` inside the pool is the **router**, so the extension evaluates `allowedSwapper[pool][router]`. The actual end-user's address is never seen by the guard.

A pool admin who wants allowlisted users to be able to swap through the router must call `setAllowedToSwap(pool, router, true)`. The moment they do, every user on the network can call `router.exactInputSingle()` and the extension passes them through, because the router is allowlisted. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

---

### Impact Explanation

Any user can bypass the `SwapAllowlistExtension` on a curated pool by routing through `MetricOmmSimpleRouter`. The allowlist is the only on-chain mechanism preventing unauthorized swaps on such pools. Once bypassed, non-allowlisted users can execute swaps, draining LP value through price impact, extracting arbitrage, or violating regulatory/compliance constraints the pool admin intended to enforce. This is a direct loss of LP principal and a broken core pool invariant.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point documented and deployed for the protocol. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and wants their allowlisted users to use the router must allowlist the router address. This is the natural and expected operational pattern. The bypass is therefore triggered by normal, correct administrative setup and requires no privileged access or special conditions from the attacker — any EOA can call `exactInputSingle`.

---

### Recommendation

The extension must resolve the true end-user identity rather than trusting the `sender` argument when the caller is a known periphery contract. Two sound approaches:

1. **Pass-through identity in the router**: Have `MetricOmmSimpleRouter` include the original `msg.sender` inside `extensionData` and have the extension decode and verify it (requires a coordinated convention).

2. **Gate on `sender` only for direct pool calls; reject router calls**: The extension can check whether `sender` is a known router and revert unless the actual user is separately verified.

3. **Preferred — fix the identity at the pool level**: Introduce a separate `originator` field (the true EOA) that periphery contracts populate via a transient-storage pattern (similar to how the router already stores payer context), and pass it as a distinct argument to extension hooks so guards can always key on the economic actor.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists alice: setAllowedToSwap(pool, alice, true)
  - Pool admin allowlists router: setAllowedToSwap(pool, router, true)
    (required so alice can use the router)

Attack:
  - bob (not allowlisted) calls:
      router.exactInputSingle({pool: pool, ..., recipient: bob, ...})
  - Router calls pool.swap(bob, ...) → msg.sender in pool = router
  - Pool calls _beforeSwap(router, bob, ...)
  - Extension checks allowedSwapper[pool][router] → true ✓
  - Swap executes for bob despite bob not being allowlisted

Result:
  - bob swaps successfully on a pool that should have blocked him
  - SwapAllowlistExtension invariant broken
```
