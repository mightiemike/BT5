### Title
`SwapAllowlistExtension` checks the router's address instead of the actual user when swaps are routed through `MetricOmmSimpleRouter`, allowing curated-pool allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which equals `msg.sender` inside `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` in the pool, so the extension sees the router address — not the actual user. If the pool admin allowlists the router (a natural step when enabling router-mediated swaps on a curated pool), every unpermissioned user can bypass the per-user allowlist by routing through the public router contract.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument the pool passes to `_beforeSwap`, which is always `msg.sender` inside `MetricOmmPool.swap()`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
``` [3](#0-2) 

The router does not forward the original `msg.sender` to the pool in any way the extension can read. The pool therefore passes the router's address as `sender` to the extension. The allowlist check becomes `allowedSwapper[pool][router]` — a single boolean that, if true, opens the gate for every user on Earth who can call the public router.

A pool admin who wants to allow allowlisted users to trade via the router must allowlist the router address. Once they do, the per-user restriction is completely nullified: any address can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router and the extension will pass them through.

The same structural gap exists for multi-hop `exactInput` and `exactOutput` paths, where the router is `msg.sender` for every intermediate pool hop. [4](#0-3) 

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` is typically deployed to restrict swap access to trusted counterparties (e.g., known market makers, whitelisted institutions) to prevent toxic flow against LP positions. Once the router is allowlisted — a prerequisite for any allowlisted user to trade via the standard periphery — the restriction is void. Any unpermissioned address can execute swaps at oracle-derived prices, draining LP value through adverse selection. This is a direct loss of LP principal, satisfying the Critical/High impact gate.

---

### Likelihood Explanation

The scenario requires the pool admin to allowlist the router. This is a natural and expected operational step: without it, allowlisted users cannot use the standard periphery at all, making the pool effectively unusable for normal DeFi workflows. The admin is therefore likely to allowlist the router, unaware that doing so collapses the per-user gate entirely. The attacker needs no special role — only the ability to call the public `MetricOmmSimpleRouter`.

---

### Recommendation

The extension must gate on the economically relevant actor, not the proximate caller. Two sound approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into the `extensionData` it forwards to the pool, and the extension decodes and verifies it (requires a trusted router convention or a signed payload).
2. **Check `sender` only for direct pool calls; reject router-mediated calls on allowlisted pools**: The extension can detect that `sender` is a known router and revert unless the actual user is separately verified.
3. **Align with the deposit allowlist pattern**: `DepositAllowlistExtension` correctly gates on `owner` (the position beneficiary), not `sender`. The swap allowlist should similarly gate on a user-supplied and pool-verified identity rather than the raw `msg.sender` chain.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, trustedMaker, true)` — only `trustedMaker` should swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — necessary so `trustedMaker` can use the router.
4. Attacker (`badActor`, not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)` — `msg.sender` in the pool is the router.
6. Pool calls `_beforeSwap(router, recipient, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. Swap executes at oracle price; `badActor` receives output tokens, LP absorbs the adverse-selection loss.
9. Step 4–8 repeatable indefinitely; LP principal drains proportionally to toxic flow volume.
