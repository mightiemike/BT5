### Title
Wrong-Actor Binding in `DepositAllowlistExtension` Allows Unauthorized Depositors to Bypass the Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` by **depositor address**, but the implementation checks the `owner` (position owner) parameter instead of the first unnamed parameter (the actual caller/depositor). Any address not on the allowlist can bypass the guard by naming an allowed address as the position `owner`.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two address arguments: the first (unnamed) is the actual caller of `addLiquidity` on the pool, and the second is `owner` — the address that will hold the resulting LP position. The extension silently ignores the first parameter and gates on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, salt, deltas, ...)` overload explicitly accepts a caller-supplied `owner` argument that is forwarded directly to `pool.addLiquidity(positionOwner, ...)`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1, bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [2](#0-1) 

The `_addLiquidity` internal helper then calls `pool.addLiquidity(positionOwner, ...)`, so the pool's `beforeAddLiquidity` hook fires with `owner = positionOwner` (attacker-controlled) and the first unnamed parameter = the actual caller (the attacker). The extension checks only `owner`, so if the attacker supplies any address that is on the allowlist as `owner`, the check passes unconditionally. [3](#0-2) 

The test suite confirms the first parameter is the caller and the second is the owner, yet the extension ignores the first:

```solidity
// test: first arg is address(0) (caller), second is depositor (owner) — extension only checks owner
extension.beforeAddLiquidity(address(0), depositor, 0, emptyDelta, "");
``` [4](#0-3) 

---

### Impact Explanation

A curated pool that deploys `DepositAllowlistExtension` to restrict who may provide liquidity is fully bypassed. Any unprivileged address can deposit tokens into the pool by naming any allowlisted address as `owner`. The attacker's tokens enter the pool and the allowlisted address receives the LP position. The pool's curation invariant — that only approved depositors can add liquidity — is broken. Depending on the pool's design, this can result in:

- Unauthorized liquidity being injected into a restricted pool, diluting or manipulating the bin composition.
- The pool admin's access-control policy being rendered meaningless for any depositor willing to gift a position to an allowed address.

This is a **High** severity broken core pool functionality / admin-boundary break.

---

### Likelihood Explanation

The `MetricOmmPoolLiquidityAdder` is the canonical periphery path for adding liquidity and explicitly exposes the `owner` parameter to callers. No additional privilege, special token, or off-chain coordination is required. Any EOA can execute this in a single transaction. Likelihood is **High**.

---

### Recommendation

Change `beforeAddLiquidity` to check the first parameter (the actual caller/depositor) rather than `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

If the intent is to gate by position owner (not caller), the contract's NatSpec and name must be updated to reflect that, and the `isAllowedToDeposit` view must be renamed accordingly. The current documentation and name unambiguously state "depositor address," which is the caller, not the owner.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** on the allowlist.
3. Bob calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(
       pool,
       alice,   // owner = allowed address
       salt,
       deltas,
       maxAmount0,
       maxAmount1,
       ""
   );
   ```
4. Pool fires `beforeAddLiquidity(bob, alice, ...)`. Extension checks `allowedDepositor[pool][alice]` = `true` → no revert.
5. Bob's tokens are pulled from Bob's wallet; Alice receives the LP position. Bob has successfully deposited into a pool that was supposed to block him. [2](#0-1) [1](#0-0)
