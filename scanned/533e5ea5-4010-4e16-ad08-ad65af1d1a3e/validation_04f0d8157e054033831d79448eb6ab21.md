### Title
`SwapAllowlistExtension` checks the router address instead of the end-user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. This means the extension checks the router's address against the allowlist, not the actual trader's address. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users inadvertently opens the gate for every user on-chain.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool). `sender` is the first argument forwarded by the pool, which is set to `msg.sender` of the `pool.swap()` call in `MetricOmmPool.swap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- this becomes `sender` in the extension
    recipient,
    zeroForOne,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (or any `exact*` variant), the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

So the call chain is:

```
user → router.exactInputSingle() → pool.swap() [msg.sender = router]
                                        → _beforeSwap(router, ...)
                                            → extension.beforeSwap(sender=router, ...)
                                                → allowedSwapper[pool][router]  ← checked, NOT the user
```

The extension never sees the end user's address. It only sees the router's address.

---

### Impact Explanation

**Scenario A — Pool admin allowlists the router (to let allowlisted users use the router):**
Any non-allowlisted user can call `router.exactInputSingle()` and the extension will check `allowedSwapper[pool][router]`, which is `true`. The swap allowlist is completely bypassed for all users. This breaks the curated-pool invariant and allows unauthorized traders to execute swaps on pools that were intended to be restricted.

**Scenario B — Pool admin does NOT allowlist the router:**
Allowlisted users who call through the router will have their swaps rejected because `allowedSwapper[pool][router]` is `false`. The router — a supported, documented periphery contract — is effectively unusable for any allowlisted pool. This breaks core swap functionality for legitimate users.

In either scenario the allowlist guard fails to enforce its intended policy. Scenario A is the higher-severity path: a pool admin who follows the natural operational pattern (allowlist the router so allowlisted users can use it) silently opens the pool to all traders.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Pool admins deploying a curated pool with `SwapAllowlistExtension` will naturally need to decide whether to allowlist the router. The documentation and test suite do not flag this identity mismatch. Any pool admin who allowlists the router triggers the bypass immediately and unconditionally for every user on-chain. The trigger requires no special permissions, no malicious setup, and no unusual token behavior — only a standard router call.

---

### Recommendation

The extension must gate on the **end user's identity**, not the intermediary's. Two complementary fixes:

1. **Pass the original user through the router.** The router already knows `msg.sender` (the end user). It could forward the user's address in `extensionData`, and the extension could decode and check it. However, this is forgeable by any caller who constructs raw `extensionData`.

2. **Check `recipient` instead of `sender` for router flows, or require the pool to pass the payer.** The cleanest fix is to have the pool pass the economically relevant actor. For swaps, the payer (the address that will be debited in the callback) is the correct identity to gate. The router stores the payer in transient storage (`_getPayer()`) but does not forward it to the extension.

3. **Preferred: gate on `sender` and require direct pool calls for allowlisted pools.** Document explicitly that `SwapAllowlistExtension` is incompatible with router-mediated swaps and revert in `beforeSwap` if `sender` is a known router address. This is the safest short-term fix.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured on beforeSwap
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it
  - bob is NOT allowlisted

Attack:
  - bob calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient=bob, ...)
  - pool calls extension.beforeSwap(sender=router, ...)
  - extension checks allowedSwapper[pool][router] == true  → passes
  - bob's swap executes on the restricted pool

Result:
  - bob bypassed the swap allowlist entirely
  - The pool admin's curation policy is silently violated
```
