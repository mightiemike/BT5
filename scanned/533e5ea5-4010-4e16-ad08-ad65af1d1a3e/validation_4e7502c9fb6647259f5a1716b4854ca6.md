### Title
SwapAllowlistExtension gates the router address instead of the actual swapper, making the allowlist bypassable for all router-mediated swaps â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. The allowlist therefore gates the router's address, not the actual swapper's identity. A pool admin cannot simultaneously allow legitimate users to use the router and block non-allowlisted users: either the router is not allowlisted (breaking the router for everyone) or the router is allowlisted (bypassing the guard for everyone).

---

### Finding Description

**Hook dispatch â€” what `sender` the extension actually receives**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value as the first argument to every configured extension: [2](#0-1) 

**The guard â€” what it checks**

`SwapAllowlistExtension.beforeSwap` receives `sender` (the pool's `msg.sender`) and checks it against the per-pool allowlist: [3](#0-2) 

**The router â€” what it passes**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly. The pool's `msg.sender` is therefore the **router contract address**, not the EOA that called the router: [4](#0-3) 

The router stores the actual user only in transient callback context for payment settlement; it never forwards the user's identity to the pool or to extensions: [5](#0-4) 

**The impossible admin choice**

The pool admin faces a binary, both-wrong outcome:

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | Blocked (broken UX) | Blocked |
| Yes | Allowed | **Allowed â€” bypass** |

There is no configuration that allows allowlisted users to use the router while blocking non-allowlisted users.

---

### Impact Explanation

Any user can bypass a `SwapAllowlistExtension` guard by routing through `MetricOmmSimpleRouter`. If the pool admin allowlists the router to restore normal UX for legitimate users, the allowlist is completely ineffective. Non-allowlisted addresses (e.g., non-KYC'd, sanctioned, or otherwise restricted counterparties) can execute swaps against the pool, draining LP assets at oracle-quoted prices. This is a direct loss of LP principal and a broken core pool invariant: the configured access control does not gate the actual swapper.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any user who discovers the bypass (or simply uses the router normally) triggers it. No privileged access, no special setup, and no non-standard tokens are required. The trigger is a standard `exactInputSingle` or `exactInput` call.

---

### Recommendation

The `SwapAllowlistExtension` must check the identity the pool admin intends to gate â€” the end user â€” not the intermediary. Two complementary fixes:

1. **Extension-side**: Use `extensionData` to carry a signed or verified user identity, or have the extension read the actual payer from a trusted router registry.

2. **Simpler fix**: Change `beforeSwap` to check both `sender` (direct caller) and an optional user address passed via `extensionData`, so router-mediated swaps can still supply the real user identity for the allowlist check.

Alternatively, document clearly that `SwapAllowlistExtension` only gates direct pool callers and is incompatible with router-mediated flows, and provide a router-aware allowlist extension that verifies the end user.
