### Title
Deposit Allowlist Checks LP Owner Instead of Actual Depositor, Allowing Any Address to Bypass the Guard - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address that actually provides tokens) and instead validates the `owner` parameter (the LP position recipient). Because `MetricOmmPool.addLiquidity` lets any caller freely specify an arbitrary `owner`, any address not on the allowlist can bypass the guard by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding both the real caller (`sender`) and the caller-supplied LP recipient (`owner`) to the extension hook. [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first positional parameter (`sender`) is unnamed and ignored. The allowlist check is performed exclusively against `owner`: [2](#0-1) 

The mapping is keyed `allowedDepositor[pool][depositor]`, and the admin-facing setter is named `setAllowedToDeposit(pool, depositor, allowed)`, making the intent clear: the guard is supposed to restrict who **provides** tokens, not who **receives** shares. [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks the `sender` parameter (the actual swapper) and ignores the `recipient`: [4](#0-3) 

The asymmetry confirms the bug: the deposit extension checks the wrong address.

---

### Impact Explanation

The deposit allowlist is completely neutralised. Any address — regardless of whether it appears in `allowedDepositor` — can call `pool.addLiquidity(owner = <any_allowlisted_address>, ...)`, pass the hook, and inject liquidity into a restricted pool. The pool admin's access-control configuration is rendered ineffective. An attacker who can deposit freely can shift bin balances, alter the effective marginal price seen by subsequent swaps, and extract value from existing LPs through the oracle-anchored pricing mechanism, all while the allowlist appears to be enforced on-chain.

---

### Likelihood Explanation

Exploitation requires no special privileges and no complex setup. Any caller needs only to know one allowlisted address (e.g., the pool admin, which is publicly readable from the factory via `poolAdmin[pool]`). The bypass is a single parameter substitution in a standard `addLiquidity` call. [5](#0-4) 

---

### Recommendation

Rename the first parameter and check `sender` instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as its `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls:
   ```solidity
   pool.addLiquidity(
       owner        = alice,   // allowlisted address
       salt         = 0,
       deltas       = <desired bins/shares>,
       callbackData = <token transfer data>,
       extensionData = ""
   );
   ```
4. The pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The check evaluates `allowedDepositor[pool][alice]` → `true` → **passes**.
6. Bob's tokens are transferred into the pool; Alice receives the LP shares.
7. Bob has successfully deposited into a pool that was supposed to block him, bypassing the allowlist entirely. [2](#0-1) [6](#0-5)
