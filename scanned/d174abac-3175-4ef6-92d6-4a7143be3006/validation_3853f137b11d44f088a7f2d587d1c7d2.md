### Title
`SwapAllowlistExtension` gates the router address as `sender` instead of the actual user, allowing any unprivileged caller to bypass the curated-pool swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is the immediate caller of `pool.swap()`, so the extension sees the router's address as `sender` and checks `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actualUser]`. If the pool admin allowlists the router to enable router-based swaps, every user — including non-allowlisted ones — bypasses the guard. If the admin does not allowlist the router, individually allowlisted users are silently blocked from using the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool) and `sender` is the address the pool forwards from its own `swap()` invocation. The pool's `swap()` entry point receives `msg.sender = router` when a user routes through `MetricOmmSimpleRouter`. The pool passes that value as `sender` to `ExtensionCalling._beforeSwap`, which forwards it unchanged to the extension. [2](#0-1) 

The allowlist is keyed `allowedSwapper[pool][sender]`. When `sender` is the router, the extension is checking whether the router is allowlisted — not whether the actual user is allowlisted. [3](#0-2) 

This produces two mutually exclusive failure modes:

**Mode A — Bypass (higher impact):** The pool admin allowlists the router so that router-based swaps work. Because the extension checks the router address, every user who calls `MetricOmmSimpleRouter.exactInputSingle()` or `exactOutputSingle()` passes the guard regardless of whether they are individually allowlisted.

**Mode B — DoS on authorized users:** The pool admin allowlists only specific EOAs and does not allowlist the router. Individually allowlisted users who attempt to swap through the router are rejected because the extension sees `sender = router`, which is not on the list.

The research pivot in the repository's own audit scaffolding explicitly identifies this as the expected attack surface:

> *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."*
> *"Test direct swaps and router swaps on allowlisted pools and assert the hook cannot be bypassed by routing through an intermediate public contract."* [4](#0-3) 

---

### Impact Explanation

A curated pool's swap allowlist — the primary access-control boundary for KYC-gated, compliance-restricted, or partner-only pools — is rendered ineffective for any user who routes through the public `MetricOmmSimpleRouter`. In Mode A, an unprivileged path (the public router) bypasses the configured allowlist, allowing unauthorized users to trade against restricted liquidity. This is a direct admin-boundary break: the pool admin's intended access policy is silently overridden by a supported periphery contract. In Mode B, authorized users are permanently unable to use the router, breaking core swap functionality for the intended participants.

---

### Likelihood Explanation

High. `MetricOmmSimpleRouter` is a public, permissionless contract. No special privileges, flash loans, or complex setup are required — any user can call `exactInputSingle`. The pool admin is likely to allowlist the router to enable standard router-based swaps (Mode A bypass), or will discover that allowlisted users cannot use the router and be forced into a configuration that either breaks the allowlist or breaks router access (Mode B DoS). Both outcomes are reachable through normal, expected usage.

---

### Recommendation

The pool should forward the original transaction initiator — not the immediate `msg.sender` — as `sender` to the extension. Concretely:

- The router should pass the actual user's address as a dedicated parameter to `pool.swap()`, and the pool should forward that value to the extension rather than its own `msg.sender`.
- Alternatively, the extension can require the router to attest to the real user's address inside `extensionData`, and the extension verifies the attestation against a trusted-router registry.
- A simpler mitigation is to document that the allowlist gates the immediate pool caller, and require users of curated pools to call the pool directly — but this breaks the intended router UX.

---

### Proof of Concept

**Mode A — Allowlist bypass:**

1. Pool admin deploys a pool with `SwapAllowlistExtension`. Allowlists `alice` and `router` (to enable router-based swaps for `alice`).
2. `bob` (not allowlisted) calls `router.exactInputSingle({pool: curated_pool, tokenIn: USDC, ...})`.
3. Router calls `pool.swap(...)` — pool receives `msg.sender = router`.
4. Pool calls `extension.beforeSwap(router, ...)`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true`.
6. `bob`'s swap executes on the restricted pool despite never being allowlisted.

**Mode B — Authorized user DoS:**

1. Pool admin allowlists only `alice` (not the router).
2. `alice` calls `router.exactInputSingle({pool: curated_pool, ...})`.
3. Extension evaluates `allowedSwapper[pool][router]` → `false`.
4. Revert: `NotAllowedToSwap` — `alice` is blocked despite being explicitly permitted. [2](#0-1) [5](#0-4)
