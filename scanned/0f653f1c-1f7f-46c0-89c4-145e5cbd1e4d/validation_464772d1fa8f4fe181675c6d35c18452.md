### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any Caller to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router**, not the end user. If the router is allowlisted (the natural operational setup), every user—including those not on the allowlist—can bypass the gate by routing through the router.

---

### Finding Description

**Call chain when a user swaps through the router:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)` — `msg.sender = user`
2. Router calls `pool.swap(recipient, zeroForOne, amount, priceLimitX64, "", extensionData)` — `msg.sender = router`
3. Pool's `swap` function calls `_beforeSwap(msg.sender, ...)` — passes `router` as `sender`
4. `SwapAllowlistExtension.beforeSwap(sender = router, ...)` evaluates `allowedSwapper[pool][router]`
5. The actual end user is **never checked**

In `MetricOmmPool.swap`:

```solidity
_beforeSwap(
  msg.sender,   // ← router address, not the end user
  recipient,
  ...
);
``` [1](#0-0) 

In `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeSwap(address sender, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [2](#0-1) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument — the router address. The allowlist lookup is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the second argument), because the pool passes the actual position owner there:

```solidity
function beforeAddLiquidity(address, address owner, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [3](#0-2) 

The swap path has no equivalent `owner` argument — the only identity the pool forwards is `msg.sender`, which collapses to the router.

---

### Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict swaps to KYC'd, institutional, or otherwise vetted addresses is fully bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the restricted pool. The extension sees `sender = router`; if the router is allowlisted (the only way to let legitimate users use the router), the check passes for everyone. Unauthorized users execute swaps in a pool that was designed to exclude them, draining liquidity at oracle-derived prices that the pool admin intended only for approved counterparties.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point documented and deployed for the protocol. Any pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist. Once the router is allowlisted, the gate is open to all callers. The attack requires no special privileges, no flash loans, and no oracle manipulation — a single router call suffices.

---

### Recommendation

Pass the economically relevant identity through the swap path. Two options:

1. **Add a `swapper` field to the swap interface** (analogous to `owner` in `addLiquidity`) that the pool sets to `msg.sender` and the extension checks instead of `sender`.

2. **Check `recipient` as a proxy for the user** — but this is fragile since `recipient` can be set to any address.

The cleanest fix mirrors `DepositAllowlistExtension`: the pool should forward the true initiator (the address that called the router, recoverable via a trusted router registry or by requiring direct pool calls for allowlisted pools) so the extension can gate the correct identity.

---

### Proof of Concept

```
Setup:
  - Pool P configured with SwapAllowlistExtension
  - allowedSwapper[P][alice] = true   (alice is KYC'd)
  - allowedSwapper[P][router] = true  (router added so alice can use it)
  - bob is NOT on the allowlist

Attack:
  1. bob calls router.exactInputSingle({pool: P, ...})
  2. router calls P.swap(recipient=bob, ...)  →  msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[P][router] == true  →  PASSES
  5. bob's swap executes against the restricted pool

Result:
  - bob, an unauthorized user, swaps in a pool the admin intended to restrict
  - The allowlist provides zero protection for router-mediated swaps
```
