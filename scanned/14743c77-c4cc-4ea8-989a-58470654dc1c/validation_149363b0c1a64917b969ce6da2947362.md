### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension sees the router's address as the swapper — not the actual user. If the router is allowlisted (a natural admin action to enable "public" access through the official periphery), every non-allowlisted user can bypass the curated pool's swap gate by routing through the router.

---

### Finding Description

**Root cause — wrong actor binding in the swap allowlist hook:**

`SwapAllowlistExtension.beforeSwap` checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct — it namespaces the lookup per pool), and `sender` is the argument forwarded by the pool. The pool sets `sender = msg.sender` of the `swap()` call:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInput()` (or any `exact*` variant), the router calls `pool.swap(...)` directly. At that point `msg.sender` inside the pool is the **router contract**, not the end user. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Attack path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., KYC'd traders).
2. Pool admin also allowlists `MetricOmmSimpleRouter` (or any other public router) as a convenience entry point, believing the router is a neutral intermediary.
3. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInput()` targeting the curated pool.
4. The router calls `pool.swap()` → pool passes `msg.sender = router` as `sender` → extension checks `allowedSwapper[pool][router] == true` → swap proceeds.
5. The actual user's identity is never checked.

**Contrast with DepositAllowlistExtension:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (not `sender`), and `owner` is an explicit parameter that the liquidity adder passes as the actual beneficiary. This correctly gates the economic actor. The swap path has no equivalent — it only has `sender` (the direct caller of `pool.swap()`).

---

### Impact Explanation

**Direct loss of curation policy / unauthorized fund flows:**

- A curated pool (e.g., restricted to institutional traders, KYC'd users, or specific counterparties) can be accessed by any unpermissioned user through the public router.
- If the pool holds concentrated liquidity at oracle-anchored prices, unauthorized users can extract value at favorable oracle prices that the pool admin intended to reserve for allowlisted parties.
- LP funds are consumed by unauthorized swaps — LPs suffer adverse selection from actors the pool was explicitly designed to exclude.
- This matches the "wrong-actor binding" invariant break: the allowlist gates the router (intermediary) rather than the user (economic actor).

**Severity: High** — direct bypass of an explicit access-control guard with fund-impacting consequences on curated pools.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the canonical public swap entry point; pool admins are expected to allowlist it for normal operation.
- Any user who discovers the bypass can immediately exploit it with zero cost beyond gas.
- No privileged setup is required by the attacker; the vulnerability is triggered by a normal router call.
- The pool admin's allowlisting of the router is a natural, expected action — not a misconfiguration.

---

### Recommendation

Pass the **original user** through the swap call chain so the extension can gate the correct actor. Two options:

1. **Preferred:** Add a `swapper` parameter to `pool.swap()` that the router fills with `msg.sender` (the end user). The pool passes this value as `sender` to extensions. The pool enforces that `swapper == msg.sender` for direct calls, and the router sets it to the actual user.

2. **Alternative:** In `SwapAllowlistExtension.beforeSwap`, decode the actual user from `extensionData` (a signed or router-forwarded identity). This requires router cooperation and is more complex.

The `DepositAllowlistExtension` pattern (checking `owner`, an explicit beneficiary parameter) is the correct model — apply the same pattern to the swap path.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin: allowedSwapper[pool][router] = true   (router allowlisted)
  - Pool admin: allowedSwapper[pool][attacker] = false (attacker NOT allowlisted)
  - Pool has liquidity

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
    → router calls pool.swap(recipient=attacker, ...)
    → pool: _beforeSwap(msg.sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → PASSES
    → swap executes, attacker receives tokens

Result:
  Non-allowlisted attacker successfully swaps on a curated pool.
  The allowlist guard is completely bypassed.
  LP funds are consumed by an unauthorized actor.
