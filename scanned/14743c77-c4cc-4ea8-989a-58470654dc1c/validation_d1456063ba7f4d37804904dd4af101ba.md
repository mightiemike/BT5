### Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. The pool always passes `msg.sender` as `sender`, which is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the router contract, not the originating user. The extension therefore checks the router's allowlist status rather than the actual user's, breaking the allowlist invariant for every router-mediated swap.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [4](#0-3) 

The router never forwards the originating user's address to the pool as `sender`; it only stores the user as the payer in transient storage for the callback. The pool has no mechanism to receive a "true originator" field — `swap()` takes only `recipient`, not a separate `sender` override. [5](#0-4) 

This produces two distinct failure modes depending on how the pool admin configures the allowlist:

**Mode A — Full bypass:** The pool admin adds the router address to the allowlist (a natural choice when they want all "standard" users to be able to swap). Because the extension sees the router as the swapper for every router-mediated call, every user — including those the admin explicitly did not allowlist — can swap freely by calling `exactInputSingle` or `exactInput`. The allowlist provides zero protection.

**Mode B — Broken legitimate access:** The pool admin adds specific user EOA addresses to the allowlist (the intended design). Those users call the router (the standard periphery interface) and are rejected because the extension sees the router's address, which is not on the allowlist. Allowlisted users must call `pool.swap()` directly, bypassing the router's slippage, deadline, and multi-hop logic entirely.

---

### Impact Explanation

**Mode A** is a direct, complete bypass of a configured access-control guard on a curated pool. Any unprivileged user can trade on a pool that was designed to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties, protocol-owned addresses, or whitelisted market makers). This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's allowlist configuration is rendered ineffective by a supported public periphery path.

**Mode B** breaks the usability of the allowlist for its intended beneficiaries and forces them onto a lower-safety direct-pool path, but the primary contest-relevant impact is Mode A.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface described in the project documentation. Any user who wants to use deadline protection, slippage checks, or multi-hop routing will naturally go through it. A pool admin who configures `SwapAllowlistExtension` and adds the router to the allowlist (Mode A) or adds user EOAs (Mode B) triggers the vulnerability through entirely normal, expected usage. No special permissions, malicious setup, or non-standard tokens are required.

---

### Recommendation

The `SwapAllowlistExtension` should gate on the originating user rather than the direct pool caller. Two complementary fixes:

1. **Pool-side:** Add an optional `originator` parameter to `pool.swap()` (similar to how `addLiquidity` separates `sender` from `owner`). The pool passes `msg.sender` as the direct caller and the caller-supplied `originator` as the economic actor. Extensions can then check whichever identity is appropriate.

2. **Extension-side (short-term):** Until the pool interface is extended, document that `SwapAllowlistExtension` must allowlist the router address (not individual users) when router access is intended, and that per-user gating requires direct pool calls. Alternatively, gate on `recipient` if the pool admin controls who receives output — though this is semantically different.

The cleanest fix mirrors the `addLiquidity` pattern where `owner` (the economic actor) is explicitly separated from `sender` (the caller): [6](#0-5)

### Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension configured.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — intending to allow all "standard" router users.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack (Mode A — bypass):
  4. Attacker (not on allowlist) calls:
       router.exactInputSingle({pool: pool, ..., extensionData: ""})
  5. Router calls pool.swap(recipient, zeroForOne, amount, limit, "", "").
  6. Pool calls _beforeSwap(msg.sender=router, ...).
  7. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
  8. Swap executes. Attacker receives output tokens.
     The allowlist check never saw the attacker's address.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds.
```
