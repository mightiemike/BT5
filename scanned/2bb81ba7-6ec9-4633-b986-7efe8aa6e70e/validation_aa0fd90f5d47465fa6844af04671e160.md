### Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` â€” (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender`. When users swap through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user's address. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the allowlist to every user, completely defeating the per-user gate.

---

### Finding Description

**Root cause â€” wrong identity checked**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**How the router erases user identity**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly â€” the pool sees `msg.sender = router`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

So the extension always sees `sender = router`, never the originating EOA.

**The two broken configurations**

| Pool admin intent | Configuration | Actual result |
|---|---|---|
| Allow only specific EOAs | Allowlist EOAs, not router | Allowlisted EOAs cannot use the router at all; must call pool directly |
| Allow specific EOAs via router | Allowlist EOAs **and** router | Any user can bypass the gate by routing through the router |

The second row is the security-critical path: once the router is allowlisted, `allowedSwapper[pool][router] == true` for every call that arrives through the router, regardless of who initiated it.

**Contrast with `DepositAllowlistExtension`**

The deposit extension correctly gates on `owner` (the position owner â€” the economically relevant actor), not on `sender` (the payer/adder): [6](#0-5) 

The swap extension has no equivalent "real actor" field to fall back on; it only receives the immediate caller.

---

### Impact Explanation

When the router is allowlisted (the only way to support router-mediated swaps), the `SwapAllowlistExtension` is completely bypassed for every user. Any address can call `exactInputSingle` / `exactInput` / `exactOutput` and swap against the pool's liquidity regardless of whether they appear in the allowlist. Access controls intended to gate institutional-only pools, KYC-gated pools, or pools protected from sophisticated traders are rendered inoperative. Because the pool uses oracle pricing, individual swaps settle at fair value, but the allowlist's protective purpose â€” preventing specific actors from trading â€” is entirely lost, which can expose LPs to the exact counterparty risk the gate was meant to block.

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and wants users to interact via the standard router must allowlist the router address. This is the natural, expected configuration for a production pool. The pool admin has no indication from the extension's interface or documentation that doing so opens the gate to all users. The bypass is therefore reachable through a routine, non-malicious admin action.

---

### Recommendation

The extension must gate the originating user, not the intermediate caller. Two viable approaches:

1. **Forward the real user in `extensionData`**: The router encodes `msg.sender` into the `extensionData` it passes to `pool.swap`. The extension decodes and verifies this value. A signature or trusted-forwarder pattern prevents spoofing.

2. **Separate router allowlist from user allowlist**: Introduce a two-level check â€” if `sender` is an allowlisted router, decode the real user from `extensionData` and check that address against the per-user allowlist. If `sender` is not a router, check `sender` directly.

Either way, the extension must never treat the router's presence as proof that the originating user is authorized.

---

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  extension.setAllowedToSwap(pool, router, true)   // admin enables router-mediated swaps
  // Alice (attacker) is NOT in the allowlist

Attack:
  Alice calls router.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      ...
  })

  router â†’ pool.swap(recipient, ...)
    pool.msg.sender = router
    pool calls extension.beforeSwap(sender=router, ...)
    extension checks: allowedSwapper[pool][router] == true  âś“
    swap proceeds

Result:
  Alice swaps successfully against the pool despite not being allowlisted.
  The per-user gate is completely bypassed for every user who routes through the router.
```
