### Title
PriceVelocityGuardExtension per-swap `lastMidPriceX64` reset allows cumulative intra-block price movement to exceed the configured per-block cap — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` unconditionally writes `lastMidPriceX64 = midPrice` and `lastUpdateBlock = block.number` **before** the velocity check. Each subsequent swap in the same block therefore measures its delta against the *previous swap's* mid price, not the start-of-block mid price. An attacker who can push multiple oracle price updates within one block (e.g., via Pyth's permissionless `updatePriceFeeds`) can chain N swaps, each individually within the configured envelope, while the cumulative intra-block price movement reaches N × `maxChangePerBlockE18` — directly contradicting the documented invariant.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` (lines 57–58) writes the new state before the guard check:

```solidity
s.lastMidPriceX64 = midPrice;          // state committed before check
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {
    uint256 blockDiff = block.number - prevBlock;   // 0 for same-block swaps
    uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
    uint256 actualSq  = changeE18 * changeE18;
    uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
    if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
}
```

For two swaps in the same block:

| Swap | `prevMid` | `curMid` | `blockDiff` | `allowedSq` | passes? |
|------|-----------|----------|-------------|-------------|---------|
| 1 | P₀ | P₁ = P₀ × (1 + maxChange) | 0 | maxChange² | ✓ |
| 2 | **P₁** (just written) | P₂ = P₁ × (1 + maxChange) | 0 | maxChange² | ✓ |

After N swaps the oracle mid has moved from P₀ to P₀ × (1 + maxChange)^N, while the guard was documented to cap movement at `maxChangePerBlockE18` per block. The contract's own NatSpec states: *"Caps how fast the provided price can move **between blocks**"* — the implementation enforces only a per-swap cap.

The analog to the external Rigor bug is exact: in Rigor, `lastTimestamp` is reset to `block.timestamp` after each payment while `noOfDays` is truncated, so repeated small payments accumulate only half the intended interest. Here, `lastMidPriceX64` is reset to the current mid after each swap while the check is only incremental, so repeated small swaps accumulate N× the intended per-block price movement.

---

### Impact Explanation

The velocity guard is the primary on-chain protection against oracle-price manipulation on pools that configure it. Bypassing it allows a swap to execute at a mid price that is arbitrarily far from the start-of-block oracle price. Because the pool's bin pricing is anchored to the oracle mid, a sufficiently large intra-block price shift lets the attacker drain token reserves from LP bins at stale prices — a direct loss of LP principal. This matches the allowed-impact gate: *bad-price execution: stale, inverted, unbounded, or unclamped bid/ask quote reaches a pool swap*.

---

### Likelihood Explanation

Pyth Network price feeds are updated permissionlessly: any party holding a valid VAA (signed by Pyth's authorized publishers and freely available from the Pyth off-chain API) can call `updatePriceFeeds` on the Pyth contract. On chains with sub-second block times or on Ethereum with flashbots bundles, an attacker can interleave multiple `updatePriceFeeds` + `pool.swap` calls within a single block. No privileged role is required; the attacker only needs to be an allowed swapper (or the pool has no swap allowlist). The `smart-contracts-poc` oracle layer (`PythOracle.sol`) is in scope and supports this update pattern.

---

### Recommendation

Track the **start-of-block** mid price separately from the last-swap mid price. Only update `lastMidPriceX64` when `block.number > lastUpdateBlock` (i.e., a new block has started). Within the same block, compare the current mid against the value that was stored at the beginning of the block:

```solidity
if (block.number > prevBlock) {
    // new block: update reference and allow movement up to maxChange * sqrt(1 + blockDiff)
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
} else {
    // same block: compare against the start-of-block reference, do NOT update it
}
```

Alternatively, store a separate `blockStartMidPriceX64` that is only written once per block and always use it as the comparison baseline.

### Proof of Concept

1. Pool is configured with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 0.01e18` (1 % per block).
2. Oracle (Pyth) mid price at block N−1: P₀ = 1.00.
3. Attacker bundles in block N:
   - `updatePriceFeeds(VAA for P₁ = 1.01)` → `pool.swap(...)` → velocity check: Δ = 1 %, allowed = 1 %, passes; `lastMidPriceX64 = P₁`.
   - `updatePriceFeeds(VAA for P₂ = 1.0201)` → `pool.swap(...)` → velocity check: Δ = 1 % from P₁, passes; `lastMidPriceX64 = P₂`.
   - Repeat 10 times → final mid ≈ 1.1046 (10.46 % above P₀).
4. Each individual swap passed the 1 % cap; the cumulative intra-block movement is 10.46 %, 10× the intended limit.
