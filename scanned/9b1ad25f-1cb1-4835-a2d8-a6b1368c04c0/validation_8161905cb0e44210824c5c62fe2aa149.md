### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of Actual Depositor `sender`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` **by depositor address**. However, the hook reads the second ABI parameter (`owner`, the LP-share recipient) instead of the first (`sender`, the actual token-providing caller). Any address not on the allowlist can deposit into a restricted pool by simply naming an allowlisted address as `owner`, while a legitimately allowlisted depositor who routes LP shares to a non-allowlisted address is incorrectly blocked.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes the call as `(sender, owner, salt, deltas, extensionData)`: [2](#0-1) 

So the first `address` slot is `sender` (the `msg.sender` who provides tokens via callback) and the second is `owner` (the address that receives LP shares).

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first slot and names the second slot `owner`: [3](#0-2) 

The guard then checks `allowedDepositor[msg.sender][owner]` — i.e., whether the LP-share recipient is allowlisted — instead of whether the actual depositor (`sender`) is allowlisted. Compare with `SwapAllowlistExtension`, which correctly reads the first slot as `sender` and checks it: [4](#0-3) 

---

### Impact Explanation

**Allowlist bypass (unauthorized deposit):** Any address not on the allowlist calls `addLiquidity(owner = allowlisted_address, ...)`. The extension evaluates `allowedDepositor[pool][allowlisted_address]` → `true` and permits the call. The unauthorized caller provides the tokens via the swap callback; the allowlisted address receives the LP shares. The pool admin's intent to restrict who can inject liquidity is completely defeated.

**Authorized depositor blocked (false denial):** A legitimately allowlisted depositor who routes LP ownership to a separate vault or multisig address (a normal DeFi pattern) is blocked because the vault address is not on the allowlist, even though the actual token provider is authorized.

Both outcomes break the core invariant stated in the contract's own NatSpec: *"Gates `addLiquidity` by depositor address, per pool."* [5](#0-4) 

---

### Likelihood Explanation

The bypass requires only a single call to `addLiquidity` with any allowlisted address as `owner`. No special privileges, flash loans, or oracle manipulation are needed. Any observer of the allowlist (on-chain readable via `allowedDepositor`) can execute this immediately. Pools that deploy this extension specifically to restrict depositors are the target, making the trigger condition exactly the intended production use-case.

---

### Recommendation

Change the hook signature to name and use the first parameter (`sender`) instead of the second (`owner`):

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external
      view
      override
      returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`, which checks `sender` (the actual swapper) rather than `recipient`.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; only `alice` is allowlisted (`allowedDepositor[pool][alice] = true`).
2. `bob` (not allowlisted) calls `pool.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. The extension evaluates `allowedDepositor[pool][alice]` → `true` → hook returns the valid selector.
4. `LiquidityLib.addLiquidity` executes; the pool calls `bob.metricOmmSwapCallback(...)` and `bob` transfers the required tokens.
5. LP shares are minted to `alice`; `bob` has successfully deposited into a pool he is not authorized to access.
6. Conversely, `alice` calling `addLiquidity(owner = alice_vault, ...)` where `alice_vault` is not allowlisted is incorrectly reverted with `NotAllowedToDeposit`, blocking a legitimate operation. [3](#0-2) [6](#0-5)
