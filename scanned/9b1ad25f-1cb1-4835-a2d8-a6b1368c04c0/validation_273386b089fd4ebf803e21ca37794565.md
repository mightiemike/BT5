### Title
`SwapAllowlistExtension` checks the router address instead of the actual end-user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` - (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks whether the **router** is allowlisted — not the actual end user. Any unprivileged user can bypass a curated pool's swap allowlist by routing through the public router.

### Finding Description

The call chain is:

1. User calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. The router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — here `msg.sender` to the pool is the **router contract address**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passing the router as `sender`.
4. `ExtensionCalling._beforeSwap` encodes `sender = router` and dispatches to the configured extension.
5. `SwapAllowlistExtension.beforeSwap(address sender, ...)` evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]`.

The extension therefore checks whether the **router** is on the allowlist, not the actual user who initiated the transaction. There is no mechanism in the hook interface to recover the original `tx.origin` or the real end user.

**Bypass path:** A pool admin who wants to allow only KYC'd addresses to swap will allowlist those addresses individually. If the admin also allowlists the router (to let those users trade via the standard periphery), any non-KYC'd user can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. The individual-user allowlist is completely defeated.

**Inverse breakage:** If the admin does *not* allowlist the router, then even individually-allowlisted users cannot trade through the router, breaking the expected user flow.

The root cause is in `SwapAllowlistExtension.beforeSwap` at line 37: it reads `sender` which is `msg.sender` of the pool's `swap` call — the router — rather than the originating user.

### Impact Explanation

Any user can bypass a curated pool's swap allowlist by routing through `MetricOmmSimpleRouter`. This defeats the core purpose of the `SwapAllowlistExtension`: gating which addresses may trade against the pool. Pools using this extension for KYC, compliance, or access-control purposes are fully open to any public user via the router. This is a direct policy bypass with fund-impacting consequences (unauthorized parties execute swaps against LP assets).

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router (the normal periphery path) is immediately affected. The attacker needs no special privileges — only the ability to call the public router.

### Recommendation

The extension must gate the **economically relevant actor**, not the intermediary. Two options:

1. **Pass the real user through the router.** Modify `MetricOmmSimpleRouter` to encode the original `msg.sender` in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when the direct `sender` is a known router. This requires a trust model for the router.

2. **Check `sender` only when it is not a known intermediary, and require the real user to be encoded in `extensionData`.** The extension can be made aware of trusted forwarders (routers) and fall back to an `extensionData`-encoded identity when the direct caller is one of them.

The simplest safe fix is to require that the pool's swap allowlist is always checked against the address that will pay tokens — which for direct calls is `msg.sender` of `pool.swap`, and for router calls is the router's stored payer. The router should forward the real user identity in `extensionData` and the extension should consume it.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow router-mediated swaps for allowlisted users).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  - attacker (not allowlisted) calls:
      MetricOmmSimpleRouter.exactInputSingle({
          pool: pool,
          ...
      })
  - Router calls pool.swap(...) with msg.sender = router.
  - Pool calls _beforeSwap(router, ...).
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
  - Attacker's swap executes against LP assets despite not being allowlisted.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
