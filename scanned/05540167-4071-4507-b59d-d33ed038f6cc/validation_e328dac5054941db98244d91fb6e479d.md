### Title
Intra-block price reference reset in `PriceVelocityGuardExtension.beforeSwap` allows cumulative per-block price movement of N × `maxChangePerBlockE18` — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

### Summary

`PriceVelocityGuardExtension.beforeSwap` writes the new mid-price and current block number into storage **before** performing the velocity guard check. As a result, every subsequent swap within the same block compares against the already-moved price with `blockDiff = 0`, receiving a fresh full-block allowance. N swaps in one block can move the price by N × `maxChangePerBlockE18`, completely defeating the guard's stated invariant.

### Finding Description

The critical ordering in `beforeSwap`:

```solidity
// lines 57-58: state written FIRST
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

// lines 60-74: guard checked AFTER, using stale prevMid/prevBlock
if (prevMid != 0) {
    uint256 blockDiff = block.number - prevBlock;   // = 0 for swap 2+
    uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
    // allowedSq = maxChange^2 * 1  when blockDiff == 0
``` [1](#0-0) 

**Concrete two-swap trace in block N (prevBlock = N-1):**

| | Swap 1 | Swap 2 |
|---|---|---|
| `prevMid` | P₀ (block-start price) | P₁ (post-swap-1 price) |
| `prevBlock` | N-1 | **N** (written by swap 1) |
| `blockDiff` | 1 | **0** |
| `allowedSq` | `maxChange² × 2` | `maxChange² × 1` |
| Max allowed move | `maxChange × √2` from P₀ | `maxChange` from P₁ |

Combined worst-case intra-block move: `maxChange × (√2 + 1) ≈ 2.41 × maxChange` — more than double the intended per-block cap.

With three or more swaps the attacker accumulates further: each additional same-block swap contributes another `maxChange` deviation from the previous swap's price.

The root cause is that `s.lastMidPriceX64` and `s.lastUpdateBlock` are committed unconditionally before the guard runs, so the guard never sees the block-start reference price for any swap after the first. [2](#0-1) 

### Impact Explanation

The velocity guard exists to prevent oracle-price manipulation within a block. Bypassing it allows a sandwich attacker to push the pool's oracle-derived mid-price far beyond the configured cap in a single block, executing swaps at prices the pool admin explicitly intended to block. This constitutes **bad-price execution**: the pool settles trades at a manipulated mid-price, causing direct loss to LPs and counterparties in the pool.

### Likelihood Explanation

The path requires only two public `swap` calls in the same block — no privileged access, no malicious pool setup, no non-standard tokens. Any MEV bot or sandwich attacker can execute this on any pool using this extension. Likelihood is **high**.

### Recommendation

Move the state write to **after** the guard check, or track a separate `blockStartMidPrice` that is only updated when `block.number > lastUpdateBlock`, so all intra-block swaps are compared against the same block-start reference:

```solidity
// Only advance the reference when entering a new block
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// Guard always compares against block-start price
```

Alternatively, write the new state only after the guard passes, and keep `prevMid`/`prevBlock` as the block-start snapshot for the duration of the block.
