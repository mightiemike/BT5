### Title
`PriceVelocityGuardExtension` Uses `block.number` Which Returns L1 Block Number on Arbitrum, Causing Guard Bypass or Swap DoS — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` tracks price velocity using `block.number` as the block counter. On L2 chains like Arbitrum, `block.number` returns the **L1 block number** (~12 s cadence), not the L2 block number (~0.25 s cadence). This causes the guard to be systematically miscalibrated in two directions depending on how the pool admin sets `maxChangePerBlockE18`: either the guard is **under-constrained** (allowing ~48× more price movement per L2 block than intended, defeating the manipulation guard) or **over-constrained** (rejecting valid swaps and making the pool non-functional).

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` records and compares block numbers:

```solidity
// line 58
s.lastUpdateBlock = uint64(block.number);
// line 63
uint256 blockDiff = block.number - prevBlock;
// line 70
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
```

The allowed price movement scales as `maxChange × √(1 + blockDiff)`.

On Arbitrum, `block.number` is the **L1 block number**. One L1 block (~12 s) contains approximately 48 L2 blocks (~0.25 s each). This creates two failure modes:

**Mode A — Guard Bypass (under-constrained):**
If the pool admin calibrates `maxChangePerBlockE18` for L1 cadence (e.g., 1 % per 12 s), then within a single L1 block all L2 swaps see `blockDiff = 0`. The guard allows `maxChange` movement per L2 block (0.25 s), which is 48× more than the admin intended per L1 block. An attacker can execute 48 consecutive L2-block swaps, each moving the price by `maxChange`, accumulating a total price drift of `48 × maxChange` within one L1 block while the guard never fires.

**Mode B — Swap DoS (over-constrained):**
If the pool admin calibrates `maxChangePerBlockE18` for L2 cadence (e.g., 0.1 % per 0.25 s), then across L1 blocks `blockDiff` is the L1 block count, not the L2 block count. For 48 L2 blocks elapsed (= 1 L1 block), the guard allows `maxChange × √(1+1) ≈ 1.41 × maxChange`, but the intended allowance is `maxChange × √(1+48) ≈ 7 × maxChange`. The guard is ~5× more restrictive than intended, causing legitimate swaps to revert with `PriceVelocityExceeded` whenever the oracle price moves normally across an L1 block boundary.

The `setLastMidPrice` admin setter also writes `block.number` as the anchor:

```solidity
// line 32
s.lastUpdateBlock = uint64(block.number);
```

This compounds the problem: an admin reset on Arbitrum anchors to an L1 block, so the first swap in the next L1 block sees `blockDiff = 1` (L1 blocks) regardless of how many L2 blocks have elapsed.

---

### Impact Explanation

**Mode A (bypass):** The velocity guard — the sole on-chain defence against rapid oracle-price manipulation for pools using this extension — is rendered ineffective within any single L1 block. A price that moves `N × maxChange` across N L2 blocks within one L1 block passes unchecked. LPs are exposed to the exact manipulation scenario the guard was designed to prevent: a sequence of swaps draining the pool at a price that has moved far beyond the configured per-block cap.

**Mode B (DoS):** Every swap that crosses an L1 block boundary with a normal oracle price move reverts. The pool's swap functionality is broken for any market with routine inter-block volatility, qualifying as "unusable swap/liquidity flows." LPs cannot rebalance, traders cannot execute, and protocol fees are not collected.

Both impacts are reachable by any unprivileged caller (Mode A) or by normal market conditions (Mode B) on any L2 where `block.number` returns the L1 block number.

---

### Likelihood Explanation

The repository already ships L2-specific oracle contracts (`PriceProviderL2.sol`, `ProtectedPriceProviderL2.sol`) with explicit `FUTURE_TOLERANCE` handling for L2 sequencer clock skew, confirming the protocol targets L2 deployment. The `PriceVelocityGuardExtension` has no corresponding L2 awareness. Pool admins configuring `maxChangePerBlockE18` have no in-code signal that `block.number` behaves differently on their target chain; the NatSpec only says "Caps how fast the provided price can move between blocks" with no L2 caveat. Misconfiguration in either direction is a near-certain outcome for any Arbitrum deployment.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension` and restate the velocity formula in time units (e.g., seconds). This is chain-agnostic and consistent with how `OracleValueStopLossExtension` already tracks decay (`block.timestamp - hwmS.lastDecayTs`). Concretely:

- Rename `lastUpdateBlock` → `lastUpdateTs`, store `block.timestamp`.
- Replace `blockDiff = block.number - prevBlock` with `timeDiff = block.timestamp - prevTs`.
- Restate `maxChangePerBlockE18` as `maxChangePerSecondE18` and update the allowed-square formula accordingly.

---

### Proof of Concept

**Setup (Mode A — bypass on Arbitrum):**

1. Deploy `PriceVelocityGuardExtension` on Arbitrum; attach it to a pool as the `beforeSwap` extension.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e16)` — intending to cap price movement at 1 % per L1 block (~12 s).
3. Pool admin calls `setLastMidPrice(pool, initialMid)` — anchors `lastUpdateBlock` to current L1 block N.

**Attack (within L1 block N, across 48 L2 blocks):**

4. Attacker submits Swap 1 in L2 block 1 (still L1 block N): oracle mid moves +1 %. `blockDiff = N - N = 0`. `allowedSq = (1e16)² × 1`. `actualSq = (1e16)² × 1`. Guard passes (equality). `lastUpdateBlock` stays N.
5. Attacker submits Swap 2 in L2 block 2 (still L1 block N): oracle mid moves another +1 %. `blockDiff = 0` again. Guard passes again.
6. Repeat for all 48 L2 blocks within L1 block N. Total price drift: **+48 %** within 12 seconds. Guard never fires.

**Expected behaviour:** Guard should have fired after the first swap exceeded the 1 %/block cap.
**Actual behaviour:** Guard allows 48 % drift within one L1 block because `blockDiff` is always 0 for intra-L1-block L2 swaps. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
