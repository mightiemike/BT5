### Title
`SwapAllowlistExtension` checks the router address instead of the actual user, allowing any disallowed swapper to bypass the curated-pool gate via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the first argument forwarded from the pool — which equals `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the actual user. The extension therefore checks whether the router is allowlisted, not whether the individual user is allowlisted. Any disallowed user can bypass the curated-pool gate by calling any router entry-point instead of calling `pool.swap` directly.

---

### Finding Description

**Call path:**

```
User → MetricOmmSimpleRouter.exactInputSingle(...)
         → pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)
              [msg.sender = router]
              → ExtensionCalling._beforeSwap(msg.sender=router, recipient, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        checks: allowedSwapper[pool][router]
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value just described — the router address when the user entered through the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making itself the pool's `msg.sender`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

---

### Impact Explanation

The pool admin configures the allowlist to gate individual swapper addresses. The invariant is: only allowlisted addresses may trade on the curated pool. This invariant is broken in two symmetric ways:

1. **Allowlist bypass (direct loss path):** If the pool admin allowlists the router so that allowlisted users can trade through it, every non-allowlisted user can also trade through the router — the gate is fully open to the public.

2. **Broken core functionality:** If the pool admin does not allowlist the router, allowlisted users cannot use the router at all, even though the router is the canonical supported periphery path. The allowlist effectively becomes incompatible with the router.

In the first case, a disallowed user can execute swaps on a curated pool, draining LP value or extracting oracle-anchored profit that the pool admin intended to restrict to a specific counterparty set. This is a direct loss of LP principal and protocol fees above Sherlock thresholds.

---

### Likelihood Explanation

`SwapAllowlistExtension` is a production extension explicitly listed as a supported guard. `MetricOmmSimpleRouter` is the canonical user-facing swap interface. Any pool that deploys both — the expected production configuration for a curated pool — is immediately vulnerable. No special conditions, privileged access, or unusual token behavior is required. Any public user can call the router.

---

### Recommendation

The `beforeSwap` hook should gate on the **original end-user**, not on the immediate caller of the pool. Two options:

**Short term:** Pass the original user through `extensionData` and have the extension decode it, with the router populating `extensionData` with `msg.sender` before calling the pool. This requires a convention between the router and the extension.

**Long term (preferred):** Change the `sender` argument forwarded to extensions to carry the **economic actor** — i.e., the address that will pay for the swap — rather than the pool's `msg.sender`. The router already tracks the true payer in transient storage (`_getPayer()`); this value should be forwarded as `sender` to the pool's `swap` call, or the extension interface should include a separate `payer` field. The `DepositAllowlistExtension` correctly gates on `owner` (the position owner) rather than `sender`; the swap allowlist should follow the same pattern by gating on the true economic actor.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required for any router-mediated swap to succeed.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. The pool calls `_beforeSwap(msg.sender=router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true`.
7. The swap executes. The attacker, who is not on the allowlist, has successfully traded on the curated pool.

The `FullMetricExtensionTest` integration test at `metric-periphery/test/extensions/FullMetricExtension.t.sol` demonstrates the allowlist working correctly for direct pool calls but does not test the router-mediated path, leaving this bypass untested. [6](#0-5)
