### Title
`SwapAllowlistExtension` Gates Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract, not the actual user. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently opens the pool to every unprivileged caller.

---

### Finding Description

**Allowlist check in `SwapAllowlistExtension`:** [1](#0-0) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool; `sender` is whatever the pool passed as the first argument to `beforeSwap`.

**How the pool populates `sender`:** [2](#0-1) 

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

**How `ExtensionCalling` forwards it:** [3](#0-2) 

The `sender` forwarded to every extension is exactly `msg.sender` of the pool's `swap()` call.

**How the router calls the pool:** [4](#0-3) 

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The router calls `pool.swap()` directly. The pool sees `msg.sender = router`. The router's original caller (`msg.sender` of `exactInputSingle`) is stored only in transient storage for the payment callback — it is **never forwarded to the pool or to any extension**.

**Resulting identity mismatch:**

| Call path | `sender` seen by extension | Allowlist key checked |
|---|---|---|
| User → `pool.swap()` directly | actual user | `allowedSwapper[pool][user]` |
| User → `router.exactInputSingle()` → `pool.swap()` | router contract | `allowedSwapper[pool][router]` |

A pool admin who wants allowlisted users to be able to use the standard periphery router must add the router to `allowedSwapper`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` is `true` for every caller, so any unprivileged address can call `router.exactInputSingle()` and the extension passes unconditionally.

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain enforcement mechanism for restricting who may trade in a pool. Bypassing it allows any address to execute swaps in a pool the admin intended to be permissioned (e.g., KYC-gated, institutional-only, or whitelist-only). Unauthorized swaps drain pool liquidity and fees to unintended counterparties, constituting a direct loss of LP assets and a broken core swap flow. This satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" and "Broken core pool functionality causing loss of funds or unusable swap flows" impact gates.

---

### Likelihood Explanation

The bypass requires the pool admin to allowlist the router — a natural and expected action for any operator who wants their permitted users to access the standard periphery UX. The admin has no on-chain signal that doing so opens the pool to everyone. The router is a public, factory-validated contract, so any user can call it. Likelihood is **Medium**: the configuration is common and the exploit requires no special privilege or capital beyond a normal swap.

---

### Recommendation

The router must forward the original caller's identity so extensions can gate on it. Two concrete options:

1. **`extensionData` convention:** The router encodes `msg.sender` into `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap()` decodes and checks that address instead of `sender`.

2. **Dedicated originator field:** Add an `originator` parameter to `IMetricOmmExtensions.beforeSwap()` that the pool populates from a router-supplied value (e.g., via `callbackData` or a separate transient slot), and the extension checks `allowedSwapper[pool][originator]`.

Until one of these is implemented, pool admins using `SwapAllowlistExtension` must **not** allowlist the router address, accepting that allowlisted users can only swap by calling the pool directly.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension (allowAllSwappers = false)
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the only permitted swapper
  - Pool admin calls setAllowedToSwap(pool, router, true)  // to let alice use the router

Attack:
  1. bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({
         pool: restrictedPool,
         tokenIn: token0,
         amountIn: X,
         amountOutMinimum: 0,
         ...
     })
  2. Router calls restrictedPool.swap(recipient=bob, ..., sender=router)
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true  ✓
  5. Swap executes; bob receives token1 output from the restricted pool.

Result: bob, who is not on the allowlist, successfully swaps in a pool
        the admin intended to be permissioned. The allowlist guard is
        completely bypassed.
```
