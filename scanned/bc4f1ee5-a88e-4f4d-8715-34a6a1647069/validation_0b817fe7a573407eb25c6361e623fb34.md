### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller of `addLiquidity`) and validates `owner` (the position beneficiary) instead. Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender â‰  owner`, any address that is not on the allowlist can bypass the guard by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller who pays tokens via the callback; `owner` is the position beneficiary whose shares are credited. The pool's own NatSpec confirms the split: *"msg.sender pays but need not equal owner (operator pattern)."*

`DepositAllowlistExtension.beforeAddLiquidity` receives these as `(address sender, address owner, â€¦)` but discards `sender` entirely (unnamed first parameter) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's own NatSpec states it *"Gates `addLiquidity` by depositor address"* â€” the depositor is the caller (`sender`), not the position owner. The sibling `SwapAllowlistExtension` correctly checks `sender`:

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, â€¦) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    â€¦
}
```

The structural mismatch â€” `owner` checked in the deposit guard, `sender` checked in the swap guard â€” is the root cause.

---

### Impact Explanation

Any address not on the allowlist can call:

```
pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)
```

The extension sees `owner = allowlistedAddress` (allowlisted â†’ passes), ignores the actual caller, and the deposit proceeds. The unauthorized caller pays tokens; the allowlisted address receives LP shares it never requested. The deposit allowlist â€” the sole access-control mechanism for restricted pools â€” is rendered completely ineffective. Pools configured for KYC, regulatory, or counterparty-restriction purposes accept liquidity from arbitrary actors, breaking the admin-configured invariant and potentially altering pool composition in ways the admin explicitly prohibited.

---

### Likelihood Explanation

The operator pattern (`msg.sender â‰  owner`) is a documented, first-class feature of the pool. Any address can exploit this without any special privilege: it only needs to know one allowlisted address (which is public on-chain via `AllowedToDepositSet` events or `allowedDepositor` view calls). The exploit requires a single transaction with no preconditions beyond token approval.

---

### Recommendation

Replace the unnamed first parameter with `sender` and check it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the caller and the owner, both should be checked independently.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `alice`.
2. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. The pool calls `extension.beforeAddLiquidity(bob, alice, salt, deltas, "")`.
4. The extension evaluates `allowedDepositor[pool][alice]` â†’ `true` â†’ no revert.
5. `bob` pays tokens via the callback; `alice` receives LP shares.
6. The allowlist has been bypassed: `bob` deposited into a pool he was explicitly excluded from. [1](#0-0) [2](#0-1) [3](#0-2)
