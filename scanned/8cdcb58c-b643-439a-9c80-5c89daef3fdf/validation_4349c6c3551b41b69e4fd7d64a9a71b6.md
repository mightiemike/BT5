### Title
Pool Admin Bypasses Governance Fee Caps via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` lets the pool admin set per-bin additional spread fees (`addFeeBuyE6`, `addFeeSellE6`) with no cap validation, while every other admin fee setter enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can therefore charge swappers an effective fee that exceeds the governance-set cap, instantly and without a timelock.

### Finding Description

The factory owner governs fee caps via `setFeeCaps`, which sets `maxAdminSpreadFeeE6` (hard ceiling 200 000 E6 = 20%) and `maxAdminNotionalFeeE8`. Every admin fee path that touches the global spread or notional components enforces these caps:

```solidity
// setPoolAdminFees — enforces cap
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes the values straight through with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool) {
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, not the fee magnitude: [3](#0-2) 

These per-bin fees are added directly to the oracle-derived base fee in every swap path:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) 

The type is `uint16`, so the maximum settable value is 65 535 E6 ≈ **6.55%** additional fee per bin, applied on top of the oracle spread and the global `spreadFeeE6`. There is no timelock on this setter; the change takes effect for the very next swap.

### Impact Explanation

A pool admin can instantly raise the effective swap fee in any bin beyond the governance-enforced cap. Swappers trading in that bin pay the excess fee immediately, with no on-chain mechanism to warn them or delay execution. The excess fee accrues to LPs (of which the pool admin may be one), constituting a direct loss of user principal above the cap that governance intended to protect.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained by `maxAdminSpreadFeeE6`. The bypass path (`setPoolBinAdditionalFees`) is a standard admin operation, callable at any time with a single transaction. No special preconditions are required beyond holding the pool admin role.

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Additionally, consider whether a timelock (analogous to `priceProviderTimelock`) should apply to per-bin fee changes, so users can observe and react before the new rate takes effect.

### Proof of Concept

1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 10_000` (1%) and `adminSpreadFeeE6 = 5_000` (0.5%).
2. Pool admin calls `setPoolAdminFees(pool, 15_000, 0)` → reverts with `AdminFeeTooHigh` (correctly enforced).
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` → **succeeds** with no revert.
4. The active bin now charges `baseFeeX64 + 65535/1e6` ≈ oracle spread + 6.55% per swap, far above the 1% cap.
5. A user's next swap in bin 0 pays the uncapped fee; the excess is silently retained by LPs. [6](#0-5) [7](#0-6)
