### Title
`changeE18 * changeE18` Unchecked Squaring Overflows uint256, Permanently DoS-ing Pool Swaps ŌĆö (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` computes a squared price-change ratio to avoid an on-chain `sqrt`. The intermediate value `changeE18` is scaled by `1e18` and can reach `~3.4e56` under reachable oracle conditions. Squaring it (`changeE18 * changeE18`) silently overflows `uint256` in Solidity Ōēź 0.8, which instead **reverts** (checked arithmetic), causing every subsequent swap on the pool to revert until an admin manually resets `lastMidPriceX64`.

---

### Finding Description

The velocity guard computes:

```solidity
uint256 changeE18 = (delta * 1e18) / uint256(prevMid);   // line 67
uint256 actualSq  = changeE18 * changeE18;                // line 69
``` [1](#0-0) 

`delta` is the absolute difference of two `uint128` mid-prices, so it is bounded by `type(uint128).max Ōēł 3.4 ├Ś 10^38`. `prevMid` is also `uint128` and can be as small as `1` (a Q64.64 price of `~5.4 ├Ś 10^-20`).

When `prevMid` is small:

```
changeE18 = (3.4e38 ├Ś 1e18) / 1 = 3.4e56
actualSq  = (3.4e56)^2         = 1.16e113   ŌåÉ overflows uint256 (max Ōēł 1.15e77)
```

Because the code is **not** inside an `unchecked` block and the compiler version is `^0.8.35`, the multiplication reverts with a panic instead of wrapping. The `beforeSwap` hook therefore reverts, and since the pool calls this hook on every swap, **all swaps on the pool revert**.

The overflow threshold is `changeE18 > sqrt(type(uint256).max) Ōēł 1.157 ├Ś 10^38`, which is reached whenever:

```
delta / prevMid > 1.157e20
```

Concretely: if `prevMid` was recorded as `1` (or any value below `~3 ├Ś 10^18`) and the oracle now returns a normal mid-price of `2^64` (representing 1:1 in Q64.64), the ratio exceeds the threshold and the hook reverts.

`prevMid` is written on every swap:

```solidity
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);
``` [2](#0-1) 

So if the oracle ever returns a near-zero price (oracle malfunction, extreme market event, or a token pair with a very large price ratio), `prevMid` is persisted as that tiny value. On the very next swap, when the oracle has recovered to a normal price, the squaring overflows and the pool is bricked.

The only recovery path is the admin calling `setLastMidPrice`:

```solidity
function setLastMidPrice(address pool_, uint128 newLastMidPriceX64) external onlyPoolAdmin(pool_) {
``` [3](#0-2) 

Until that call is made, no swap can execute.

---

### Impact Explanation

Every swap on a pool that has `PriceVelocityGuardExtension` configured as its `beforeSwap` hook will revert. LPs cannot rebalance, traders cannot trade, and the pool's liquidity is effectively frozen. This matches the allowed impact category **"Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows"**: LPs suffer opportunity loss and cannot exit via swaps; the pool is insolvent from a usability standpoint until admin intervention.

---

### Likelihood Explanation

The trigger requires `prevMid` to be recorded at a very small value. This can occur via:

1. **Oracle malfunction / stale feed returning near-zero**: Pyth Lazer and Chainlink Data Streams can return stale or zero prices during network outages; a single swap at that moment writes the bad `prevMid`.
2. **Extreme token price ratio**: Token pairs where one token is priced at fractions of a wei in Q64.64 terms (e.g., high-decimal vs. low-decimal tokens, or a token that has crashed).
3. **Admin sets `lastMidPriceX64` to a small value**: `setLastMidPrice` is admin-only but has no lower-bound validation.

Condition 1 is realistic for any oracle-dependent protocol and requires no attacker ŌĆö a single swap during an oracle hiccup is sufficient to brick the pool.

---

### Recommendation

Replace the bare multiplication with `Math.mulDiv` (which uses full 512-bit intermediate arithmetic) or cap `changeE18` before squaring:

```solidity
// Option A: use mulDiv to avoid intermediate overflow
uint256 actualSq = Math.mulDiv(changeE18, changeE18, 1);  // still overflows if result > uint256

// Option B: cap changeE18 at sqrt(type(uint256).max) before squaring
uint256 MAX_CHANGE_E18 = 1.157e38;  // sqrt(type(uint256).max)
if (changeE18 > MAX_CHANGE_E18) {
    revert PriceVelocityExceeded(type(uint256).max, allowedSq);
}
uint256 actualSq = changeE18 * changeE18;

// Option C (preferred): compare without squaring, using cross-multiplication
// actualSq > allowedSq  Ō¤║  changeE18 > maxChange * sqrt(1 + blockDiff)
// Avoid squaring entirely by comparing changeE18 against the allowed bound directly.
```

Option B is the minimal fix: if `changeE18` already exceeds the maximum representable square root, the price change is definitionally beyond any configured `maxChange`, so the revert is correct ŌĆö but it should be a clean `PriceVelocityExceeded` revert, not a panic.

Also add a lower-bound check in `setLastMidPrice` (e.g., `require(newLastMidPriceX64 >= MIN_PRICE)`) to prevent an admin from inadvertently writing a near-zero `prevMid`.
