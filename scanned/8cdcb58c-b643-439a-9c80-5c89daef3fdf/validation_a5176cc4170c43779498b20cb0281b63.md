### Title
Router-Mediated Swaps Substitute Router Address for Original Caller in `SwapAllowlistExtension::beforeSwap`, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the address that called `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so `sender` = router address, not the original user. If the router is allowlisted (a natural admin action to enable router-mediated swaps), every unprivileged user bypasses the allowlist by routing through it.

### Finding Description

**Call chain:**

```
User (Alice) → MetricOmmSimpleRouter::exactInputSingle
                 → IMetricOmmPoolActions(pool).swap(...)   // msg.sender = router
                     → _beforeSwap(msg.sender=router, ...)
                         → SwapAllowlistExtension::beforeSwap(sender=router, ...)
                             → allowedSwapper[pool][router]  // checks router, not Alice
```

In `MetricOmmPool::swap`, the pool passes its own `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling::_beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

The hook then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = router: [3](#0-2) 

The router calls `pool.swap()` directly with no mechanism to forward the original user's address: [4](#0-3) 

### Impact Explanation

Two concrete failure modes:

1. **Allowlist bypass (High):** Pool admin allowlists the router address to enable router-mediated swaps for permitted users. Any unprivileged user can now bypass the allowlist by calling `exactInputSingle`/`exactInput`/`exactOutputSingle`/`exactOutput` through the router. The access control that was meant to restrict swaps to specific actors is fully defeated.

2. **Allowlisted users cannot use the router (Medium):** Pool admin allowlists specific user addresses. Those users cannot use the router because the router's address is not allowlisted, breaking core swap functionality for the intended beneficiaries.

### Likelihood Explanation

The router is the canonical public entry point for swaps. Any pool that uses `SwapAllowlistExtension` and also wants to support router-mediated swaps will encounter this. The bypass requires no special privileges — any public user can call the router.

### Recommendation

Pass the original user's address through the call chain. One approach: the router encodes the original `msg.sender` into `extensionData`, and the extension reads it from there. Alternatively, the pool could accept an explicit `swapper` parameter distinct from `msg.sender`. The hook must gate the actor the pool designers intended to allowlist, not the intermediate contract.

### Proof of Concept

```solidity
// Pool admin sets up allowlist: only Alice is allowed
extension.setAllowedToSwap(pool, alice, true);
// Router is NOT explicitly allowlisted

// Admin then allowlists the router to enable router-mediated swaps for Alice:
extension.setAllowedToSwap(pool, address(router), true);

// Now Bob (unprivileged) calls the router:
// router.exactInputSingle(...) → pool.swap(msg.sender=router) → beforeSwap(sender=router)
// allowedSwapper[pool][router] == true → PASSES
// Bob has bypassed the allowlist.
```

Direct swap by Bob (not through router) correctly reverts:
```solidity
// pool.swap(...) called directly by Bob → beforeSwap(sender=bob)
// allowedSwapper[pool][bob] == false → NotAllowedToSwap ✓
```

The two-transaction framing in the question is unnecessary — the bypass is atomic in a single transaction. The invariant broken is: **the `sender` identity checked by the hook must equal the economic actor initiating the swap**, which fails whenever the router intermediates.
