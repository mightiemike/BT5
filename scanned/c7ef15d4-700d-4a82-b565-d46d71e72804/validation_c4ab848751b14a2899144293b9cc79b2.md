### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension` is documented as gating deposits by the depositor (caller) address. Its `beforeAddLiquidity` hook receives both `sender` (the real `msg.sender` of `addLiquidity`) and `owner` (a free parameter the caller supplies). The implementation silently drops `sender` and checks `owner` against the allowlist. Because `owner` is attacker-controlled, any address can pass the gate by nominating an allowlisted address as `owner`.

### Finding Description
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` and passes `(msg.sender, owner)` to `_beforeAddLiquidity`, which forwards both to the extension. [4](#0-3) 

`DepositAllowlistExtension.beforeAddLiquidity` ignores the first argument entirely and evaluates `allowedDepositor[msg.sender][owner]`, where `msg.sender` is the pool and `owner` is the attacker-chosen value: [5](#0-4) 

If any address `alice` is allowlisted, an arbitrary operator can call `pool.addLiquidity(owner=alice, ...)`, the extension evaluates `allowedDepositor[pool][alice] == true`, and the deposit proceeds. The position is minted to `alice`; the operator supplies the tokens via the liquidity callback.

### Impact Explanation
The deposit allowlist is rendered completely ineffective as a sender gate. Any address can deposit into a pool that the admin intended to restrict, simply by specifying an allowlisted address as `owner`. This breaks the core access-control invariant of the extension and constitutes an admin-boundary break: an unprivileged path bypasses a pool-admin-configured restriction. Pools deployed with this extension for compliance or access-control purposes offer no real protection.

### Likelihood Explanation
Exploitation requires only knowledge of one allowlisted address (publicly readable from `allowedDepositor`) and a direct call to `pool.addLiquidity`. No privileged role, special token behavior, or off-chain data is needed. Likelihood is high.

### Recommendation
Replace the `owner` check with a `sender` check. The first parameter of `beforeAddLiquidity` is the real caller and should be named and used:

```solidity
// Before (broken):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```
