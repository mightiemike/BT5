### Title
Stop-Loss Guard Uses Arithmetic Mid Price While Pool Settles at Geometric Mid, Causing Silent Under-Protection During Wide-Spread Conditions — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the **arithmetic mean** of bid and ask, while the pool itself and `PriceVelocityGuardExtension` both compute it as the **geometric mean** via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because AM ≥ GM (AM-GM inequality), the stop-loss systematically overestimates `metricToken1` and underestimates `metricToken0` whenever the spread is non-zero. When the spread widens between the watermark-setting swap and a later value-extracting swap, the stop-loss fails to revert the later swap even though the geometric-mean-based value has fallen below the configured drawdown floor.

---

### Finding Description

**Pool settlement price (geometric mean):** [1](#0-0) 

```solidity
function midAndSpreadFeeX64FromBidAsk(uint256 bidPriceX64, uint256 askPriceX64)
    ...
{
    midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);   // geometric mean
    ...
}
```

**`PriceVelocityGuardExtension` (correct — uses geometric mean):** [2](#0-1) 

**`OracleValueStopLossExtension` (wrong — uses arithmetic mean):** [3](#0-2) 

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

The per-share metrics fed into the watermark comparison are: [4](#0-3) 

```
metricToken0 = t0*SCALE/shares + (t1 * Q64 / mid) * SCALE / shares
metricToken1 = (t0 * mid / Q64) * SCALE / shares + t1*SCALE/shares
```

Because `arithmeticMid ≥ geometricMid`:
- `metricToken1` is **inflated** (t0 is multiplied by a larger mid)
- `metricToken0` is **deflated** (t1 is divided by a larger mid)

The watermark ratchet and breach check: [5](#0-4) 

The breach condition `metric < hwm * floorMultiplier / E6` is evaluated against the inflated `metricToken1`. If the watermark was established during a tight-spread swap (arithmetic ≈ geometric), but the value-extracting swap occurs during a wide-spread period (arithmetic >> geometric), the inflated current metric can remain above the floor even though the geometric-mean-based value has already breached it.

---

### Impact Explanation

LPs configure the stop-loss to guarantee that per-share value cannot fall more than `drawdownE6` (e.g., 5%) below the high-watermark. During wide-spread conditions (volatile markets — exactly when protection is most needed), the arithmetic mean overestimates `metricToken1` relative to the pool's actual settlement price. A swap that extracts value from the pool at the geometric mid can push the true per-share value below the drawdown floor while the stop-loss, measuring at the inflated arithmetic mid, sees no breach and allows the swap to complete. LPs suffer losses exceeding the configured drawdown guarantee.

The magnitude of the discrepancy is `(bid + ask)/2 − sqrt(bid·ask)`. For a 10% spread this is ~0.125%; for a 50% spread it is ~3%. In high-volatility markets where oracle spreads widen significantly, the bypass can be material relative to typical drawdown configurations (e.g., 1–5%).

---

### Likelihood Explanation

The trigger is a public `swap` call timed to a period of elevated oracle spread. No privileged access is required. The attacker:
1. Waits for the oracle spread to widen (natural market event, no manipulation needed).
2. Executes a `!zeroForOne` swap (token0 in, token1 out) that would breach the geometric-mean-based `metricToken1` floor.
3. The stop-loss computes `metricToken1` at the inflated arithmetic mid, finds no breach, and returns without reverting.
4. The swap settles at the geometric mid, extracting more value than the drawdown floor permits.

The stop-loss is least protective precisely when spreads are widest — i.e., during the high-volatility events it is designed to guard against.

---

### Recommendation

Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the same geometric mean formula used by the pool and `PriceVelocityGuardExtension`:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct — matches pool settlement price):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64), uint256(askPriceX64)
);
``` [6](#0-5) 

This ensures the stop-loss measures value at the same price the pool uses for settlement, eliminating the systematic overestimation of `metricToken1` during wide-spread conditions.

### Proof of Concept

```
Setup:
  bid = 1·Q64, ask = 1·Q64 (tight spread, ~0%)
  t0 = 1000, t1 = 1000, shares = BIN_SHARES
  drawdownE6 = 50_000 (5%)

Step 1 — Watermark established (tight spread):
  arithmeticMid ≈ geometricMid ≈ 1·Q64
  metricT1 ≈ (1000·1·Q64/Q64)·SCALE/shares + 1000·SCALE/shares = 2000·SCALE/shares
  hwm1 = 2000·SCALE/shares

Step 2 — Spread widens (oracle volatility):
  bid = 1·Q64, ask = 2·Q64 (100% spread)
  geometricMid = sqrt(2)·Q64 ≈ 1.414·Q64
  arithmeticMid = 1.5·Q64

Step 3 — Value-extracting swap (!zeroForOne), t0=800, t1=800:
  metricT1 at arithmeticMid = (800·1.5)·SCALE/shares + 800·SCALE/shares
                             = 1200·SCALE/shares + 800·SCALE/shares = 2000·SCALE/shares
  floor = hwm1 · (1e6 - 50_000) / 1e6 = 2000 · 0.95 = 1900·SCALE/shares
  2000 ≥ 1900 → NO BREACH → swap allowed ✓ (wrong)

  metricT1 at geometricMid  = (800·1.414)·SCALE/shares + 800·SCALE/shares
                             = 1131·SCALE/shares + 800·SCALE/shares = 1931·SCALE/shares
  1931 ≥ 1900 → NO BREACH (barely, but the gap closes faster with more value leak)

  With t0=750, t1=750:
  arithmeticMid: metricT1 = (750·1.5 + 750)·SCALE/shares = 1875·SCALE/shares → 1875 < 1900 → BREACH
  geometricMid:  metricT1 = (750·1.414 + 750)·SCALE/shares = 1810·SCALE/shares → 1810 < 1900 → BREACH

  The stop-loss triggers ~65 SCALE/shares later with arithmetic mean than with geometric mean,
  allowing additional value extraction beyond the configured 5% drawdown floor.
```
