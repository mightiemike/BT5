### Title
`SwapAllowlistExtension` gates on the router address instead of the actual end-user, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is the caller, `sender` is the router's address, not the actual end-user. A pool admin who allowlists the router (a natural operational step to enable routing) simultaneously grants every user on the router the ability to bypass the per-user swap allowlist entirely.

---

### Finding Description

**Actor binding in the two allowlist extensions is inconsistent.**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` argument — the actual position beneficiary — because the pool passes it as the second parameter:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);

// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `swap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);

// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
);
```

So when a user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, and the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`.

**Bypass path:**

A pool admin who wants to allow routing adds the router to the allowlist:
```
swapExtension.setAllowedToSwap(pool, address(router), true);
```

From that moment, every user who calls `router.exactInputSingle(...)` passes the check, regardless of whether their own address is on the allowlist. The per-user curation is completely voided.

The multi-hop path (`exactInput`) compounds this: intermediate hops use `address(this)` (the router itself) as the payer, so the same router address is checked at every hop.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool — the admin intends to restrict swaps to a specific set of addresses (e.g., KYC-verified counterparties, whitelisted market makers, or protocol-controlled addresses). Allowlisting the router — a shared, permissionless contract — collapses the per-user gate to a single shared gate. Any user who can call the router can trade in the pool. This is a direct admin-boundary break: an unprivileged path (the public router) bypasses the access control the pool admin configured. The pool's LP assets are exposed to swaps from actors the admin explicitly did not authorize.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router address. This is a natural and expected operational step: without it, no user can route through the router to this pool. The admin has no way to simultaneously allow routing and enforce per-user restrictions using the current extension design. The mistake is structurally induced by the wrong-actor binding — the admin cannot fix it without abandoning the router entirely.

---

### Recommendation

Mirror the deposit allowlist pattern: pass the actual initiating user as a distinct parameter and check that address. One approach is to add a `swapper` field to `extensionData` that the router populates with `msg.sender` before calling the pool, and have the extension decode and verify it. A cleaner fix is to add an explicit `swapper` (analogous to `owner` in `addLiquidity`) to the `swap` signature so the pool can forward the real user identity to extensions, and the router sets it to `msg.sender`.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` to enable routing.
3. Attacker (address `0xBAD`, not on the allowlist) calls `router.exactInputSingle({pool: pool, recipient: 0xBAD, ...})`.
4. Router calls `pool.swap(recipient=0xBAD, ...)` with `msg.sender = router`.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes against the curated pool despite never being individually authorized.

**Relevant code locations:**

- Wrong-actor binding: [1](#0-0) 
- Pool passes `msg.sender` (router) as `sender`: [2](#0-1) 
- Router calls `pool.swap()` directly, becoming `msg.sender`: [3](#0-2) 
- Correct pattern (deposit checks `owner`, not caller): [4](#0-3) 
- Pool passes `owner` correctly to deposit hook: [5](#0-4)
