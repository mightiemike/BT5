Looking at the extension guard plumbing, I need to trace the `sender`/`owner` distinction through the hook path and compare `DepositAllowlistExtension` against `SwapAllowlistExtension`.

**Key observation:** `DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter and checks `owner` instead, while `SwapAllowlistExtension.beforeSwap` correctly checks `sender`. The mapping is named `allowedDepositor` and the setter takes a `depositor` argument — but the gate fires on the wrong address.

---

### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." The depositor is the `sender` — the address that actually calls `addLiquidity` and pays tokens. However, `beforeAddLiquidity` silently discards the `sender` argument and checks `owner` (the position beneficiary) instead. Any unprivileged caller can bypass the allowlist by specifying an allowlisted `owner` while themselves remaining non-allowlisted.

### Finding Description

`ExtensionCalling._beforeAddLiquidity` passes both `sender` and `owner` to the extension: [1](#0-0) 

The interface declares both parameters: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter `address` (unnamed/discarded) and checks only `owner`: [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender`: [4](#0-3) 

The admin-facing setter names the parameter `depositor`, confirming the intent was to gate by the depositing actor, not the position beneficiary: [5](#0-4) 

**Bypass path via `MetricOmmPoolLiquidityAdder`:**

`addLiquidityExactShares(pool, owner, ...)` accepts a caller-supplied `owner` and only validates `owner != address(0)`: [6](#0-5) 

The internal `_addLiquidity` call passes `positionOwner` (attacker-controlled) as the `owner` argument to the pool: [7](#0-6) 

Because the extension checks `allowedDepositor[pool][owner]` and `owner` is attacker-supplied, the gate is trivially satisfied by naming any allowlisted address as `owner`.

### Impact Explanation

The deposit allowlist is an admin-configured security boundary. A pool operator deploys it to restrict which addresses may add liquidity (e.g., KYC-gated pools, institutional-only pools, or pools where uncontrolled LP entry enables price manipulation). Because the check fires on `owner` rather than `sender`, any unprivileged address can deposit into the pool by nominating an allowlisted address as the position owner. The actual payer (`msg.sender` stored as `payer` in the transient context) is never validated against the allowlist. The admin-configured boundary is fully bypassed by an unprivileged path, matching the "admin-boundary break bypassed by an unprivileged path" impact category.

### Likelihood Explanation

The bypass requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an allowlisted `owner` address. The allowlisted address is public (emitted in `AllowedToDepositSet` events). Likelihood is high.

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the first, currently unnamed parameter) instead of `owner`, mirroring the pattern in `SwapAllowlistExtension`:

```solidity
// Before (buggy):
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

If the intent is to gate by both the depositor and the position owner, both should be checked.

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension as beforeAddLiquidity hook
  - Admin calls setAllowedToDeposit(pool, alice, true)
  - Bob (not allowlisted) holds token0 and token1 and has approved LiquidityAdder

Attack:
  1. Bob calls:
       liquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // <-- allowlisted owner; Bob is the actual payer
           salt,
           deltas,
           maxAmount0,
           maxAmount1,
           ""
       )

  2. LiquidityAdder calls pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")

  3. Pool calls _beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)

  4. DepositAllowlistExtension checks allowedDepositor[pool][alice] → true → passes

  5. Pool issues callback to LiquidityAdder; LiquidityAdder pulls tokens from Bob (payer)

  6. Alice receives the LP position; Bob's tokens are consumed.
     The allowlist check on the actual depositor (Bob) was never performed.
```

The allowlist is fully bypassed. Any non-allowlisted address can deposit into a restricted pool by supplying any allowlisted address as `owner`.
