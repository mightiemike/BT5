### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which equals the pool's `msg.sender` — the `MetricOmmSimpleRouter` — not the originating user. If the router is allowlisted on a curated pool (which is required for any router-mediated swap to succeed), every user can bypass the allowlist by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first parameter and checks it against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct — only the pool calls the extension), and `sender` is whatever the pool passes as the originating swapper. The pool's `swap` entry-point uses its own `msg.sender` as `sender` when it dispatches `_beforeSwap`. When a user calls `MetricOmmSimpleRouter.exact*`, the router becomes the pool's `msg.sender`, so the extension receives `sender = router`, not `sender = user`.

The allowlist admin configures entries keyed by `(pool, swapper)`:

```solidity
function setAllowedToSwap(address pool_, address swapper, bool allowed) external onlyPoolAdmin(pool_) {
    allowedSwapper[pool_][swapper] = allowed;
``` [2](#0-1) 

For router-mediated swaps to work at all on a curated pool, the admin must allowlist the router address. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for every user who routes through it, regardless of whether that user is individually permitted.

The pool's own interface documentation confirms the operator-pattern separation: `msg.sender` pays/initiates but need not equal the economic actor:

> `msg.sender` pays but need not equal `owner` (operator pattern). [3](#0-2) 

The `SwapAllowlistExtension` never resolves this separation — it blindly trusts the `sender` argument, which is the intermediary, not the end user.

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of addresses is fully bypassed. Any unpermissioned user can call `MetricOmmSimpleRouter.exactInput` / `exactOutput` targeting the curated pool. The router is the pool's `msg.sender`, so `allowedSwapper[pool][router] == true` passes the guard, and the swap executes. The pool's curation policy — which may exist to enforce KYC, rate limits, or price-sensitive access control — is rendered inoperative. This constitutes a broken core pool functionality with direct fund-impacting consequences: disallowed users can drain liquidity at any price the oracle permits, and LP providers who deposited under the assumption of a restricted pool face unrestricted counterparty exposure.

---

### Likelihood Explanation

The trigger requires only that the pool admin has allowlisted the router — a necessary step for any legitimate user to trade through the router on that pool. Any pool that intends to support both the router and a swap allowlist simultaneously is vulnerable. The attacker needs no special privilege: a single public call to `MetricOmmSimpleRouter` suffices.

---

### Recommendation

The extension must resolve the true end-user identity before checking the allowlist. Two sound approaches:

1. **Pass the originating user through the router.** The router should forward the original `msg.sender` as an explicit `sender` argument to `pool.swap`, and the pool should pass that value — not its own `msg.sender` — to `_beforeSwap`. This requires a pool-level change to accept a caller-supplied sender only from trusted periphery.

2. **Check the recipient instead of sender when routing.** Alternatively, the allowlist can gate the `recipient` field (which the router sets to the user), but this only works if recipient and economic actor are always the same, which is not guaranteed.

The safest fix is option 1: the pool should pass the verified originating user to extension hooks, not the raw `msg.sender`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Admin calls setAllowedToSwap(pool, router, true)   // required for any router swap
  - Admin does NOT call setAllowedToSwap(pool, alice, true)

Attack:
  - Alice (unpermissioned) calls MetricOmmSimpleRouter.exactInput(pool, ...)
  - Router calls pool.swap(recipient=alice, ...)
  - Pool dispatches _beforeSwap(sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] == true  → passes
  - Swap executes for Alice despite her not being on the allowlist

Result:
  - Alice trades on a curated pool she was explicitly excluded from.
  - Every other unpermissioned user can do the same through the router.
  - The swap allowlist provides zero protection for router-mediated flows.
``` [1](#0-0) [4](#0-3)
