### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any unprivileged user to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` mediates the swap, `sender` is the router's address, not the originating user. A pool admin who allowlists the router to let their users access it inadvertently opens the pool to every user, defeating the allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user routes through `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [4](#0-3) 

So the allowlist check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. A pool admin who adds the router to the allowlist (the natural step to let their allowlisted users use the router) simultaneously grants every user on-chain the ability to swap through that router, because the extension never inspects the originating EOA.

The extension's own NatSpec states it "Gates `swap` by swapper address, per pool," but the implementation gates by the immediate caller of `pool.swap()`, which is the router for all router-mediated flows. [5](#0-4) 

---

### Impact Explanation

Any user can bypass a pool's swap allowlist by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) on a pool that has `SwapAllowlistExtension` configured and the router address allowlisted. Unauthorized swappers gain full access to the pool's liquidity, enabling them to extract value from LPs through swaps the pool admin explicitly intended to block. This breaks the "Admin-boundary break" invariant: an unprivileged path (the public router) bypasses a configured access-control guard.

---

### Likelihood Explanation

The scenario is reachable whenever:
1. A pool is deployed with `SwapAllowlistExtension` as a configured extension (a deliberate, documented feature).
2. The pool admin adds the router to the allowlist — the natural and expected action to let their allowlisted users use the periphery router.

Both steps are normal operational choices. No malicious setup or non-standard token is required. Any user who observes the allowlist state on-chain can immediately exploit it.

---

### Recommendation

The extension must gate the originating user, not the intermediate contract. Two sound approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. The pool's `nonReentrant` guard prevents the router from being re-entered to forge this value.

2. **Check `sender` only for direct pool calls; require the router to forward the real user as a verified parameter**: Add a thin wrapper in the extension that, when `sender` is a known router, reads the real user from a router-provided field in `extensionData` and checks that address against the allowlist instead.

Either way, `allowedSwapper[pool][router]` must never be the terminal check when the router is acting on behalf of an arbitrary user.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, userA, true)` to restrict swaps to `userA`.
2. Pool admin also calls `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
3. `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. The router calls `pool.swap(recipient, zeroForOne, amount, limit, "", extensionData)` — `msg.sender` of this call is the router.
5. The pool calls `_beforeSwap(router, ...)`, which calls `SwapAllowlistExtension.beforeSwap(router, ...)`.
6. The extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. `userB`'s swap executes successfully despite never being allowlisted, draining LP funds the pool admin intended to protect. [6](#0-5) [4](#0-3)
