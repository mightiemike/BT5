### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the **router**, not the user. If the pool admin allowlists the router address to enable router-mediated swaps, every user — including those explicitly not on the allowlist — can bypass the per-user gate by going through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When a user calls `MetricOmmSimpleRouter.exactInput*`, the router calls `pool.swap(recipient, ...)`. At that point `msg.sender` inside the pool is the **router contract**, so `sender` delivered to the extension is the router's address, not the end user's address.

The allowlist check therefore becomes `allowedSwapper[pool][router]`.

**Contrast with `DepositAllowlistExtension`**, which deliberately ignores `sender` and instead checks `owner` — the LP position owner — which is the actual economic actor regardless of which intermediary called `addLiquidity`. The swap extension has no equivalent "owner" field, so it falls back to `sender`, which is wrong when an intermediary is involved.

Two broken outcomes follow:

1. **Allowlist bypass (critical):** The admin allowlists the router address so that their allowlisted users can swap via the standard periphery path. Because the check is `allowedSwapper[pool][router]`, every user — including those the admin explicitly excluded — can now swap through the router without restriction.

2. **Broken core functionality (high):** The admin allowlists individual user addresses. Those users call the router (the documented periphery path). The extension checks `allowedSwapper[pool][router]` → router is not allowlisted → swap reverts. Allowlisted users cannot use the standard swap path.

---

### Impact Explanation

**Scenario A — bypass:** A curated pool (e.g., KYC-gated or institution-only) deploys `SwapAllowlistExtension`. The admin allowlists the router so that approved users can trade via the UI. Any unapproved user calls `MetricOmmSimpleRouter.exactInput*` → pool sees `sender = router` → extension passes → unauthorized swap executes against LP liquidity. LP principals are exposed to trades the pool was designed to exclude.

**Scenario B — DoS:** Admin allowlists individual addresses. Those addresses call the router. Extension reverts because the router is not allowlisted. The allowlisted users' only practical swap path is broken.

Both outcomes are contest-relevant: Scenario A is a direct allowlist bypass with LP fund exposure; Scenario B is broken core swap functionality for a configured pool.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary documented user-facing swap path.
- Any pool that deploys `SwapAllowlistExtension` and expects users to use the router will hit one of the two broken outcomes immediately.
- No special privilege or unusual token behavior is required; any public user can trigger the bypass.

---

### Recommendation

The extension should check the actual end-user identity, not the intermediary caller. Two options:

1. **Check `recipient` instead of `sender`** — the router sets `recipient` to the user's address. This is imperfect (recipient can differ from the economic actor) but is closer to the intended gate.

2. **Pass the original user through `extensionData`** — the router encodes the real user's address in `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and extension.

3. **Mirror `DepositAllowlistExtension`'s pattern** — for swaps, define a canonical "owner" field (e.g., the address that will receive output tokens) and gate on that, requiring the router to always set `recipient` to the actual user.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Admin calls setAllowedToSwap(pool, router, true)
   — intending to allow router-mediated swaps for their allowlisted users.
3. Bob (address NOT in allowedSwapper[pool]) calls:
       MetricOmmSimpleRouter.exactInput(pool, zeroForOne, amount, ...)
4. Router calls pool.swap(recipient=Bob, ...).
5. Pool calls _beforeSwap(msg.sender=router, ...).
6. Extension checks allowedSwapper[pool][router] → true → passes.
7. Bob's swap executes against LP liquidity despite not being allowlisted.
```

Alternatively, if the admin allowlists Alice's address directly and Alice calls the router, the extension checks `allowedSwapper[pool][router]` → false → Alice's swap reverts even though she is explicitly allowlisted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
