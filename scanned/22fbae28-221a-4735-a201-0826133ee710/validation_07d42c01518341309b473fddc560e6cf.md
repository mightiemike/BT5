### Title
SwapAllowlistExtension Gates Router Address Instead of Actual User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the EOA. If the pool admin allowlists the router (the natural action to let curated users trade through the supported periphery), every unpermissioned user can bypass the allowlist by calling the router. Conversely, if the admin does not allowlist the router, even legitimately allowlisted EOAs cannot use the router at all.

---

### Finding Description

**Pool → Extension sender binding**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**Extension check**

`SwapAllowlistExtension.beforeSwap` then gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever called `pool.swap`: [3](#0-2) 

**Router is the caller of pool.swap**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly — making the router the `msg.sender` the pool sees, not the EOA: [4](#0-3) 

The same applies to `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Contrast with DepositAllowlistExtension**

`DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the economic beneficiary), not `sender` (the adder contract), so the deposit path does not share this flaw: [6](#0-5) 

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and then allowlists the router (the expected action to let approved users trade through the supported periphery) inadvertently opens the pool to every user who calls the router. Any EOA — including those explicitly blocked — can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. The allowlist provides zero protection against router-mediated swaps, allowing toxic or unauthorized flow to drain LP principal from a pool that was designed to be curated.

The alternative failure mode (admin does not allowlist the router) breaks core swap functionality for all legitimately allowlisted users who attempt to use the supported periphery path.

Both outcomes are direct fund-impacting consequences: LP assets are exposed to unauthorized counterparties, or the pool's swap path is rendered unusable for its intended participants.

---

### Likelihood Explanation

The trigger is fully unprivileged: any EOA calls `MetricOmmSimpleRouter.exactInputSingle` with the target pool address. No special role, no malicious setup, and no non-standard token behavior is required. The precondition — that the pool admin allowlists the router — is the natural and expected configuration for any curated pool whose users are meant to trade through the protocol's own periphery. The router is a canonical, factory-deployed contract, so allowlisting it is a routine administrative action, not an error.

---

### Recommendation

The extension must gate on the actual EOA, not the intermediary. Two approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` (the EOA) into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This requires a convention between router and extension but keeps the pool interface unchanged.

2. **Check `sender` only for direct pool calls; require routers to be transparent**: Define a standard interface where trusted routers forward the originating user as a typed field in `extensionData`, and the extension falls back to `sender` only when `extensionData` is empty (i.e., a direct pool call).

The deposit path already demonstrates the correct pattern: `DepositAllowlistExtension` gates on `owner` (the economic actor), not `sender` (the adder). The swap allowlist should adopt the same principle — gate on the address that economically benefits from the swap, not the contract that mechanically calls `pool.swap`.

---

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, router, true)   // allow router so curated users can trade
  pool admin: setAllowedToSwap(pool, alice, true)    // alice is an approved user
  bob = arbitrary EOA, NOT allowlisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)
    → pool calls _beforeSwap(msg.sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → checks allowedSwapper[pool][router] == true  ✓
    → swap proceeds — bob's trade executes against LP funds
    → allowlist is fully bypassed
```

Bob, a non-allowlisted user, successfully trades against a pool that was configured to be curated, exposing LP principal to unauthorized counterparty flow.
