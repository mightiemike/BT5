### Title
`DepositAllowlistExtension.beforeAddLiquidity` gates position `owner` instead of actual payer/caller, allowing any non-allowlisted address to bypass the deposit guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is the production deposit-gate hook for Metric OMM pools. Its `beforeAddLiquidity` implementation discards the actual caller identity and checks only the `owner` (position recipient) against the allowlist. Because `addLiquidity` accepts an arbitrary `owner` address, any non-allowlisted address can bypass the gate by naming an allowlisted address as the position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The first argument is `msg.sender` — the address that will pay tokens via the `metricOmmModifyLiquidityCallback`. The second argument is `owner` — the address that will receive the LP position.

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the caller and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The first parameter (`address`) is unnamed and never read. The allowlist lookup is `allowedDepositor[pool][owner]`, not `allowedDepositor[pool][caller]`.

The same bypass is reachable through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)`, which lets the caller supply an arbitrary `owner`:

```solidity
function addLiquidityExactShares(
    address pool, address owner, uint80 salt, ...
) external payable override returns (...) {
    _validateOwner(owner);
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [3](#0-2) 

Inside `_addLiquidity`, the pool is called with `positionOwner = owner` (the supplied address), so the extension sees the allowlisted owner and passes. [4](#0-3) 

The project's own audit-target document explicitly flags this exact separation:

> *"The attacker can separate payer from owner and can route through the liquidity adder, so the checked identity has to be exactly the one the pool intends to gate."* [5](#0-4) 

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who may provide liquidity (e.g., KYC/AML compliance, curated LP sets, or regulatory perimeters). With this bug:

- A non-allowlisted address calls `pool.addLiquidity(allowlistedOwner, ...)` or `liquidityAdder.addLiquidityExactShares(pool, allowlistedOwner, ...)`.
- The extension checks `allowedDepositor[pool][allowlistedOwner]` → passes.
- The non-allowlisted address pays tokens via callback; the allowlisted owner receives the LP position.
- `removeLiquidity` enforces `msg.sender == owner`, so the allowlisted owner can withdraw the full token value.

The allowlisted owner receives a funded LP position they did not pay for. The non-allowlisted address has interacted with a restricted pool as the economic depositor without being on the allowlist. The pool admin's access-control invariant — that only allowlisted addresses may deposit — is broken by any unprivileged caller who knows one allowlisted address (discoverable from on-chain `AllowedToDepositSet` events).

---

### Likelihood Explanation

- No special privilege is required; any EOA or contract can call `addLiquidity` or `addLiquidityExactShares`.
- Allowlisted addresses are publicly discoverable from emitted events.
- The `MetricOmmPoolLiquidityAdder` provides a clean, public entry point that explicitly separates `owner` from `msg.sender`.
- Likelihood is high whenever a pool is deployed with `DepositAllowlistExtension` in a non-`allowAllDepositors` configuration.

---

### Recommendation

Check the actual caller (`caller`, the first parameter) instead of `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the address that actually pays tokens — not the address that receives the position — is the one gated by the allowlist.

---

### Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` configured (not `allowAllDepositors`).
2. Pool admin calls `setAllowedToDeposit(pool, Alice, true)`. Bob is not allowlisted.
3. Bob calls `pool.addLiquidity(Alice, salt, deltas, callbackData, extensionData)` directly.
4. Pool calls `_beforeAddLiquidity(Bob, Alice, ...)` → extension checks `allowedDepositor[pool][Alice]` → `true` → no revert.
5. Pool calls `Bob.metricOmmModifyLiquidityCallback(amount0, amount1, ...)` → Bob pays tokens.
6. Alice's position is credited with the deposited shares.
7. Alice calls `pool.removeLiquidity(Alice, salt, deltas, ...)` and withdraws the full token value.
8. Bob has deposited into a restricted pool without being allowlisted; Alice received a funded position she did not pay for.
