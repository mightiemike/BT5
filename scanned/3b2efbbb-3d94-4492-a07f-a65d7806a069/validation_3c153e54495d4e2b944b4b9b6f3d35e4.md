### Title
`SwapAllowlistExtension` Checks Router Address as Swapper Identity, Enabling Full Per-User Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension.beforeSwap` hook gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When any user routes a swap through `MetricOmmSimpleRouter`, `sender` resolves to the router's address rather than the actual user's address. A pool admin who allowlists the router to enable router-mediated swaps for legitimate users inadvertently opens the gate to every non-allowlisted address, completely defeating the per-user access control.

---

### Finding Description

**Root cause — identity mismatch in the hook dispatch path**

`ExtensionCalling._beforeSwap` encodes and forwards `msg.sender` of `pool.swap()` as the `sender` argument to every configured extension: [1](#0-0) 

`MetricOmmPool.swap` passes `msg.sender` directly: [2](#0-1) 

When `MetricOmmSimpleRouter` calls `pool.swap(...)`, `msg.sender` inside the pool is the **router contract address**, not the end user. The `SwapAllowlistExtension.beforeSwap` hook receives `sender = router_address` and checks it against the per-pool allowlist (`allowedSwapper[pool][sender]`), as confirmed by the audit target description: [3](#0-2) 

**Bypass path**

A pool admin who wants to support router-mediated swaps for allowlisted users must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the extension's check becomes:

```
allowedSwapper[pool][router_address] == true  →  passes for every caller
```

Any non-allowlisted user can now bypass the per-user swap allowlist by routing their swap through `MetricOmmSimpleRouter`. The extension never sees the actual user's address.

**Contrast with `DepositAllowlistExtension`**

The deposit-side extension checks `owner` (the position owner), not `sender` (the payer), which is the documented operator-pattern design: [4](#0-3) 

The swap-side extension has no equivalent fallback — it checks only `sender`, which collapses to the router address for all router-mediated swaps.

**Explicit audit pivot match**

The protocol's own audit target document calls this out as the exact surface to validate: [5](#0-4) 

> *"the hook must gate the same actor the pool designers thought they were allowlisting … assert the hook cannot be bypassed by routing through an intermediate public contract."*

The code does not satisfy this invariant.

---

### Impact Explanation

**High/Medium.** The swap allowlist is completely bypassed for any user who routes through `MetricOmmSimpleRouter`. If the pool is configured with a swap allowlist to restrict trading to specific counterparties (e.g., to implement a whitelist-only liquidity venue, prevent adversarial flow, or satisfy regulatory requirements), any non-allowlisted address can execute swaps against the pool's LP inventory by using the public router. This directly exposes LP principal to swap flow the pool admin intended to block, constituting a broken core pool access-control invariant with potential direct loss of LP assets.

---

### Likelihood Explanation

**Medium.** The bypass requires the pool admin to allowlist the router address.
