### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Real User, Allowing Any Caller to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is the pool's own `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the pool admin allowlists the router to enable router-mediated swaps, every unprivileged user can bypass the allowlist by routing through the public router.

---

### Finding Description

`MetricOmmPool.swap()` captures `msg.sender` and forwards it as the `sender` argument to `_beforeSwap`, which relays it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ... {
    ...
    _beforeSwap(
      msg.sender,   // <-- whoever called pool.swap()
      recipient,
      ...
    );
```

`ExtensionCalling._beforeSwap` encodes that value and passes it to each extension:

```solidity
// ExtensionCalling.sol
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```

`SwapAllowlistExtension.beforeSwap` then uses that argument as the identity to gate:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the original `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // user's address is NOT passed to pool.swap()
    );
```

The pool's `swap()` has no explicit `sender` parameter — it uses `msg.sender`, which is the **router**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The broken invariant is identical in structure to the external NFT bug:** the external bug used `offer.nftId` (a collection-wide sentinel) instead of the specific NFT being purchased; here the extension uses the router address (a shared intermediary) instead of the specific user being gated.

---

### Impact Explanation

A pool admin who deploys a pool with `SwapAllowlistExtension` to restrict swaps to a specific set of users (e.g., KYC-verified traders, institutional counterparties) faces an irreconcilable dilemma:

- **If the router is NOT allowlisted**: allowlisted users cannot use the router at all; they must call `pool.swap()` directly. The router — the primary public entry point — is dead for this pool.
- **If the router IS allowlisted** (the natural step to enable router-mediated swaps): `allowedSwapper[pool][router] = true`, so `beforeSwap` passes for **any** `sender` value when `allowAllSwappers[pool]` is false but the router is listed. Every unprivileged user can call `router.exactInputSingle()` and the check resolves to `allowedSwapper[pool][router] == true` — the allowlist is fully bypassed.

Consequence: unauthorized users execute swaps on a pool designed to be restricted, draining LP-owned token reserves at oracle-anchored prices without the LP's consent to trade with those counterparties.

---

### Likelihood Explanation

- `SwapAllowlistExtension` is a production periphery contract, not a mock.
- `MetricOmmSimpleRouter` is the primary public swap entry point.
- Any operator who configures both on the same pool and allowlists the router (a natural operational step) is immediately vulnerable.
- No special privilege, flash loan, or unusual token behavior is required — a plain `exactInputSingle` call suffices.

---

### Recommendation

The extension must recover the true end-user identity. Two approaches:

1. **Pass the original caller through `pool.swap()`**: add an explicit `originator` field to the swap call or encode it in `extensionData`, and have the router populate it with `msg.sender`. The extension reads `originator` from `extensionData` instead of the `sender` argument.

2. **Gate on `recipient` instead of `sender`** (weaker, but avoids the router problem if recipient == user): only viable if the pool design guarantees recipient == the economically relevant actor.

The cleanest fix is option 1: the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension` decodes and checks that value when present, falling back to `sender` for direct pool calls.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // only alice allowed
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack (executed by bob, who is NOT allowlisted):
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
     → pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] == true  ✓
  5. Swap executes — bob has bypassed the allowlist entirely

Result: bob drains pool liquidity at oracle prices despite never being allowlisted.
```
