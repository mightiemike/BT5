### Title
Pool Admin Can Set Per-Bin Additional Fees Without Cap Validation, Bypassing Factory Fee Caps - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no validation against the factory's `maxAdminSpreadFeeE6` cap. The pool admin can set per-bin additional fees up to `uint16` max (65 535 = 6.55% in E6 scale) on any bin, regardless of what the factory owner has configured as the admin spread cap. Because these per-bin fees are additive to the global spread fee, the total effective fee on a specific bin can exceed the factory's hard ceiling of 20% (`HARD_MAX_SPREAD_FEE_E6 = 200_000`).

### Finding Description
The factory enforces a layered fee-cap system:

- `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is the absolute ceiling for the global spread fee. [1](#0-0) 
- `maxAdminSpreadFeeE6` is a configurable sub-cap that the factory owner can lower at any time. [2](#0-1) 
- `setPoolAdminFees` validates the new admin spread fee against `maxAdminSpreadFeeE6` before writing it. [3](#0-2) 

However, `setPoolBinAdditionalFees` performs **no such validation**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [4](#0-3) 

The per-bin fields `addFeeBuyE6` / `addFeeSellE6` are stored in `BinState` as `uint16` (max 65 535) and are documented as **additional** fees on top of the global spread fee:

> `addFeeBuyE6` — Additional fee for buying token0; 1e6 = 100% [5](#0-4) 

Because the factory's cap machinery (`maxAdminSpreadFeeE6`, `setFeeCaps`, `setPoolAdminFees`) is entirely bypassed for per-bin fees, the pool admin can:

1. Set the global admin spread fee to 0 (or have it capped at 0 by the factory owner).
2. Simultaneously set `addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` on every bin via `setPoolBinAdditionalFees`.
3. Charge ~6.55% additional fee per bin with no factory-level check.

Combined with the maximum global spread fee (20%), the total effective fee on a targeted bin reaches **~26.55%**, exceeding the hard ceiling the factory is supposed to enforce.

### Impact Explanation
Traders executing swaps through any bin where the pool admin has set `addFeeBuyE6` / `addFeeSellE6` to the `uint16` maximum pay up to 6.55% more than the factory's hard cap permits. This is a direct loss of trader principal on every affected swap. Because the factory's fee-cap system is the primary user-protection mechanism against admin fee abuse, its bypass constitutes a broken invariant: the factory owner's ability to constrain admin fees is rendered ineffective for the per-bin dimension.

### Likelihood Explanation
The pool admin is a semi-trusted role. The contest scope explicitly lists "pool admin exceeds caps" as an in-scope admin-boundary break. The call path is a single transaction by the pool admin through the factory's `setPoolBinAdditionalFees` — no special preconditions, no timelock, no co-signer required. Any pool whose admin is malicious or compromised is immediately exploitable.

### Recommendation
Add a cap check inside `setPoolBinAdditionalFees` (or inside the pool's `setBinAdditionalFees`) that enforces `addFeeBuyE6 <= maxAdminSpreadFeeE6` and `addFeeSellE6 <= maxAdminSpreadFeeE6`, mirroring the validation already present in `setPoolAdminFees`. Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap and validate against it, so the factory owner retains full control over the ceiling for all admin-controlled fee dimensions.

### Proof of Concept
```
// 1. Factory owner lowers the admin spread cap to 0 to prevent admin fee abuse.
factory.setFeeCaps(200_000, 0, 1_000_000, 1_000_000);

// 2. Pool admin bypasses the cap via per-bin fees — no revert.
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
// addFeeBuyE6 = 65535 ≈ 6.55% — stored on-chain, no validation triggered.

// 3. Any trader swapping through bin 0 now pays ~6.55% additional fee
//    on top of the global spread fee, despite the factory cap being 0%.
```

The factory's `setPoolAdminFees` would revert with `AdminFeeTooHigh` for the same effective amount, but `setPoolBinAdditionalFees` succeeds silently, confirming the missing validation.
