### Title
Unvalidated Price Provider Accepted by Factory Allows Manipulable Oracle to Execute Bad-Price Swaps Against LPs — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory._validatePriceProvider()` only verifies that the supplied price provider is non-zero and returns matching `token0()`/`token1()` addresses. There is no whitelist, no quality gate, and no community-review mechanism. Any contract that satisfies those two trivial checks — including ones with flash-loan-manipulable, donation-attack-vulnerable, or otherwise unbounded pricing logic — can be registered as a pool's oracle. When such a provider is manipulated, the corrupted bid/ask price flows directly into `MetricOmmPool._getBidAndAskPriceX64()` and the swap math with no further clamping, allowing an attacker to drain LP value through bad-price swaps.

---

### Finding Description

**Root cause — factory validation:**

`_validatePriceProvider()` is the sole guard applied both at pool creation (`createPool`) and at every subsequent price-provider update (`proposePoolPriceProvider` / `executePoolPriceProviderUpdate`):

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 541-546
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1) {
        revert PriceProviderTokenMismatch();
    }
}
```

The check is purely structural (non-zero address + token-pair match). It imposes no constraint on:
- whether the provider's price can be flash-loan-manipulated in the same block,
- whether the provider is a known, audited contract,
- whether the returned price is bounded or staleness-protected.

`createPool()` is permissionless — any caller can deploy a pool with any conforming price provider:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 156-158
function createPool(PoolParameters calldata params) external override returns (address pool) {
    if (poolDeployer == address(0)) revert PoolDeployerNotSet();
    _validatePoolParameters(params);   // calls _validatePriceProvider internally
```

**Propagation into the swap path:**

At swap time, `MetricOmmPool._getBidAndAskPriceX64()` fetches the live bid/ask from whatever provider is registered:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 804-813
function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
    address activePriceProvider = _resolvedPriceProvider();
    try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (uint128 bid, uint128 ask) {
        if (bid >= ask) revert BidGreaterThanAsk();
        if (bid == 0) revert BidIsZero();
        return (bid, ask);
    } catch (bytes memory reason) {
        revert PriceProviderFailed(reason);
    }
}
```

The only post-fetch guards are `bid >= ask` and `bid == 0`. A manipulated price that is still internally consistent (bid < ask, bid > 0) passes both checks and is used verbatim in `SwapMath.midAndSpreadFeeX64FromBidAsk()` to compute the mid-price and spread fee, which then drive all token-amount calculations for the swap.

There are no extension guards that are mandatory for every pool. `PriceVelocityGuardExtension` and `OracleValueStopLossExtension` are optional, opt-in extensions. A pool created without them has no circuit-breaker against a manipulated oracle price.

---

### Impact Explanation

An attacker who can transiently manipulate the price provider (e.g., via flash loan, donation, or direct price control in an underlying pool) can:

1. Inflate the ask price → buy token0 from the pool far below fair value → drain token0 from LPs.
2. Deflate the bid price → sell token1 to the pool far above fair value → drain token1 from LPs.

Because the corrupted bid/ask propagates through `SwapMath` without any bounds check, the attacker can extract the full imbalance in a single atomic transaction. LP principal is directly at risk; the loss is permanent and unrecoverable.

**Impact class**: Critical/High — direct loss of LP principal via bad-price swap execution.

---

### Likelihood Explanation

- `createPool()` is permissionless; any address can register any conforming price provider.
- A legitimate developer building a custom integration may deploy a price provider that is internally consistent (bid < ask) but flash-loan-manipulable (e.g., derived from a low-liquidity on-chain pool or a single-block TWAP).
- Once real liquidity accumulates, the attack becomes profitable. No privileged access is required to execute the manipulation + swap in one transaction.
- The absence of a mandatory velocity guard or stop-loss extension means no on-chain safety net exists unless the pool creator explicitly opts in.

**Likelihood**: Medium — requires a pool with a manipulable provider to attract liquidity, but the permissionless factory makes this scenario reachable without any privileged cooperation.

---

### Recommendation

1. **Whitelist price providers at the factory level**: maintain a mapping `isApprovedPriceProvider[address] → bool` settable only by the factory owner; `_validatePriceProvider()` should revert if the provider is not on the whitelist.
2. **Extend `_getBidAndAskPriceX64()` with bounds checks**: reject prices that deviate more than a configurable percentage from a reference or from the previous observation.
3. **Make at least one oracle guard extension mandatory** (e.g., `PriceVelocityGuardExtension`) for pools that use mutable price providers, rather than leaving it as an opt-in.

### Proof of Concept

```
Setup
─────
1. Attacker deploys `MalleableProvider` that:
   - returns correct token0()/token1() addresses (passes _validatePriceProvider)
   - stores a manipulable price backed by a flash-loan-drainable source
   - always returns bid < ask and bid > 0 (passes _getBidAndAskPriceX64 guards)

2. Attacker (or unsuspecting developer) calls factory.createPool({priceProvider: MalleableProvider, ...})
   → factory._validatePriceProvider passes (token match only)
   → pool is registered; LPs deposit liquidity

Attack (single transaction)
────────────────────────────
3. Attacker takes flash loan of the underlying asset that backs MalleableProvider's price.
4. Attacker manipulates MalleableProvider so getBidAndAskPrice() returns:
   - askPriceX64 = 1 (near-zero, token0 is "cheap")
5. Attacker calls pool.swap(zeroForOne=false, ...) — buys token0 at the manipulated near-zero ask.
   - _getBidAndAskPriceX64() fetches bid=0.5, ask=1 → passes both guards
   - SwapMath computes mid ≈ 0.75, baseFee ≈ tiny
   - Attacker receives token0 at a fraction of fair value
6. Attacker repays flash loan; net profit = (fair value − manipulated price) × swap volume
7. LPs' token0 balance is drained; their LP shares are now worth less than deposited principal
