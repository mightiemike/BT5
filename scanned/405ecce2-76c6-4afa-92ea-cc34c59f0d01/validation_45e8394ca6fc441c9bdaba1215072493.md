### Title
`DepositAllowlistExtension` gates on `owner` (position recipient) instead of `sender` (actual depositor/payer), allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual `msg.sender` to the pool, i.e., the entity paying tokens) and instead checks `owner` (the position recipient, a free caller-supplied parameter). Because `owner` is an arbitrary address passed by the caller to `pool.addLiquidity(owner, ...)`, any non-allowlisted address can bypass the deposit gate by naming any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts `owner` as a free parameter — it is not required to equal `msg.sender`:

```solidity
// MetricOmmPool.sol
function addLiquidity(
    address owner,          // ← free parameter, not msg.sender
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

The pool then calls the extension with both identities:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  sender      owner (free param)
```

`DepositAllowlistExtension.beforeAddLiquidity` silently drops `sender` (first argument) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address /*sender*/, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` here is the pool (the extension caller), so the check is `allowedDepositor[pool][owner]`. Because `owner` is caller-controlled, any non-allowlisted address can pass an allowlisted address as `owner` and the guard approves the call.

The `MetricOmmPoolLiquidityAdder` makes this trivially reachable for EOAs without needing to implement a callback:

```solidity
// MetricOmmPoolLiquidityAdder.sol
function addLiquidityExactShares(
    address pool,
    address owner,   // ← caller supplies any address
    uint80 salt,
    ...
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks != address(0)
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

`_validateOwner` only rejects `address(0)`. The payer stored in transient context is `msg.sender` (the non-allowlisted caller), while `owner` (the allowlisted address) is forwarded to the pool and checked by the extension.

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who can add liquidity. By checking `owner` (position recipient) instead of `sender` (actual payer), the guard is checking the wrong identity. Any non-allowlisted address can:

1. Call `liquidityAdder.addLiquidityExactShares(pool, allowlisted_address, ...)`.
2. The extension sees `owner = allowlisted_address` → passes.
3. The non-allowlisted caller's tokens are pulled and deposited into the pool.
4. LP shares are minted to the allowlisted address.

If the non-allowlisted depositor colludes with the allowlisted address (or controls it), they gain full LP exposure in a pool that was explicitly configured to exclude them. The allowlist — the sole on-chain guard for deposit access — is rendered completely ineffective. This is an admin-boundary break: an unprivileged path bypasses a pool-admin-configured access control, which is within the allowed impact gate.

---

### Likelihood Explanation

The exploit path is direct and requires no special privileges:
- Any EOA can call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an arbitrary `owner`.
- No admin cooperation, no flash loan, no oracle manipulation is needed.
- The only prerequisite is knowing one allowlisted address (which may be discoverable from on-chain events emitted by `setAllowedToDeposit`).

---

### Recommendation

Change `DepositAllowlistExtension.beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns with the NatSpec ("Gates `addLiquidity` by depositor address") and with the analogous `SwapAllowlistExtension`, which correctly checks `sender`.

---

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  allowedDepositor[pool][bob]   = false (bob is not allowlisted)

Attack:
  bob calls:
    liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extensionData)

  Internally:
    liquidityAdder → pool.addLiquidity(owner=alice, callbackData=KIND_PAY, ...)
    pool           → extension.beforeAddLiquidity(sender=liquidityAdder, owner=alice, ...)
    extension      → checks allowedDepositor[pool][alice] == true → PASSES
    pool           → liquidityAdder.metricOmmModifyLiquidityCallback(amount0, amount1, KIND_PAY)
    liquidityAdder → pulls tokens from bob (the stored payer)
    pool           → mints LP shares to alice

Result:
  bob's tokens are deposited into the pool despite bob not being allowlisted.
  alice receives LP shares she did not request.
  The deposit allowlist is fully bypassed.
```
