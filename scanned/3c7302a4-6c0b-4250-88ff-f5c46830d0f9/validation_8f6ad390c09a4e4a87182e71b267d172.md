### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unprivileged Bypass of the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` **by depositor address**. Its `beforeAddLiquidity` hook receives both `sender` (the actual `msg.sender` of `addLiquidity`) and `owner` (the position-recipient address). The hook silently discards `sender` and checks only `owner` against the allowlist. Any unprivileged address can therefore bypass the allowlist by passing any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, which encodes both values and forwards them to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` passes `sender` as the first argument and `owner` as the second: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter `address` (unnamed/discarded) and the second `owner`, then checks only `owner`: [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual caller): [4](#0-3) 

The asymmetry is the root cause. Because `addLiquidity` has **no** `msg.sender == owner` guard (unlike `removeLiquidity`, which does), any caller may supply an arbitrary `owner`: [5](#0-4) 

---

### Impact Explanation

The deposit allowlist is a production safety extension intended to restrict which addresses may provide liquidity to a permissioned pool (e.g., KYC-gated, institutional, or regulatory-compliance pools). Because the guard checks the position-recipient (`owner`) rather than the actual caller (`sender`), the invariant is broken:

- An unprivileged address calls `pool.addLiquidity(owner = allowlistedAddress, ...)`.
- The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` and does not revert.
- The unprivileged caller pays the tokens; the position shares are minted to `allowlistedAddress`.
- The pool receives liquidity from a source the admin explicitly intended to exclude.

This constitutes an **admin-boundary break**: an unprivileged path bypasses a pool-admin-configured access control, violating the pool's permissioned-deposit invariant. Downstream consequences include unauthorized alteration of the pool's liquidity distribution and bin state, which can affect swap pricing and LP value in pools where the allowlist was the primary admission control.

---

### Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with `owner` set to any address already in the allowlist. No special privileges, flash loans, or oracle manipulation are needed. The allowlist of any pool is publicly readable on-chain (`allowedDepositor` is a public mapping), so an attacker can trivially identify a valid `owner` to supply.

---

### Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on both the caller **and** the position recipient, both should be checked explicitly.

---

### Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(P, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `P.addLiquidity(owner = alice, salt = 0, deltas = ..., callbackData = ..., extensionData = ...)`.
4. The pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The hook evaluates `allowedDepositor[P][alice]` → `true`; no revert.
6. `LiquidityLib.addLiquidity` mints shares to `alice`; Bob's tokens are collected via callback.
7. Bob has successfully deposited into a pool he was explicitly excluded from, violating the pool admin's access-control invariant.
