### Title
`SwapAllowlistExtension` checks the router's address instead of the actual user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is always `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so the extension checks whether the **router** is allowlisted — not the actual user. If the pool admin allowlists the router (the only way to enable router-mediated swaps on an allowlisted pool), every user of the router bypasses the per-user access control.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the **router** the `msg.sender` seen by the pool and forwarded to the extension as `sender`: [4](#0-3) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` (all recursive hops): [5](#0-4) 

The router stores the **actual user** only in transient storage as the payer for the callback — it is never surfaced to the extension: [6](#0-5) 

This creates an irreconcilable dilemma for any pool admin who deploys `SwapAllowlistExtension`:

| Admin configuration | Result |
|---|---|
| Allowlist specific user addresses only | Router-mediated swaps revert for everyone (router not allowlisted) |
| Allowlist the router | Every user of the router bypasses the per-user allowlist |
| Allowlist specific users **and** the router | Specific users can swap directly; **all** users can swap via router — allowlist defeated |

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the position owner explicitly passed by the caller), not `sender` (the direct pool caller): [7](#0-6) 

No equivalent separation exists on the swap path.

---

### Impact Explanation

Any non-allowlisted user can execute swaps on a pool that has `SwapAllowlistExtension` configured, provided the router is allowlisted (the only configuration that makes the router usable on such a pool). The allowlist — the sole access-control mechanism for swap-restricted pools — is completely bypassed. Funds flow directly from the non-allowlisted user through the router to the pool and back, with no capital contribution from the allowlisted router contract itself. This is a direct loss of the access-control invariant the extension is designed to enforce, with fund-impacting consequences (non-permitted parties execute swaps and receive output tokens).

---

### Likelihood Explanation

A pool admin who deploys `SwapAllowlistExtension` and also wants users to be able to use the standard `MetricOmmSimpleRouter` must allowlist the router. This is the natural configuration for any production pool that intends to support both access control and router-mediated swaps. The bypass requires no special setup by the attacker: any user simply calls `exactInputSingle` (or any other router entry point) on the restricted pool. The router is a public, permissionless contract.

---

### Recommendation

The extension must receive the identity of the **economic actor** (the user who provides tokens), not the identity of the intermediary contract. Two viable approaches:

1. **Extension-data forwarding**: The router encodes the original `msg.sender` into `extensionData` and the extension decodes and checks it. This requires the router and extension to agree on an encoding convention.
2. **Separate `swapper` parameter**: Add an explicit `swapper` field to the pool's `swap` signature (analogous to `owner` in `addLiquidity`) that the router populates with `msg.sender` before calling the pool. The extension then checks `swapper` instead of `sender`.

Until one of these is implemented, the `SwapAllowlistExtension` should be documented as only effective for direct pool calls, and pools that require per-user access control should not be used with the public router.

---

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Pool admin allowlists the router so that router-mediated swaps work.
swapAllowlistExtension.setAllowedToSwap(address(pool), address(router), true);

// Alice is NOT individually allowlisted.
// Bob IS individually allowlisted (but irrelevant here).

// Alice calls the router directly — the extension sees router as sender, not Alice.
vm.startPrank(alice);
token1.approve(address(router), 1 ether);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        recipient: alice,
        zeroForOne: false,
        amountIn: 1 ether,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
vm.stopPrank();

// Alice successfully swapped despite not being on the allowlist.
// The extension checked allowedSwapper[pool][router] == true, not allowedSwapper[pool][alice].
assertGt(token0.balanceOf(alice), 0);
```
