### Title
`SwapAllowlistExtension` checks the router address instead of the actual end-user, making the swap allowlist ineffective for all router-mediated swaps — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the pool's `msg.sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, making the per-user allowlist either permanently broken (allowlisted users cannot use the router) or trivially bypassable (if the router is allowlisted, every user can swap).

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that forwarded `sender` against the per-pool allowlist: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point the pool's `msg.sender` is the **router contract**, so the extension evaluates `allowedSwapper[pool][router]` — never the actual end-user's address.

Contrast this with `DepositAllowlistExtension.beforeAddLiquidity`, which deliberately ignores the `sender` (first parameter, unnamed) and checks `owner` — the economically relevant actor: [5](#0-4) 

The asymmetry is the root cause: the deposit guard correctly identifies the economic actor; the swap guard does not.

---

### Impact Explanation

**Path A — Broken allowlist (allowlisted users blocked from the router):**
A pool admin allowlists specific user addresses (e.g., KYC'd traders). Those users call `router.exactInputSingle`. The extension sees `sender = router`, which is not in the allowlist → `NotAllowedToSwap` revert. Allowlisted users are permanently unable to use the standard periphery path; they must call `pool.swap` directly, which is not the intended UX and may not be supported by integrators.

**Path B — Allowlist bypass (any user can swap):**
A pool admin allowlists the router address (the natural workaround to enable router-mediated swaps for their users). Because `sender` is always the router for any user, every unpermissioned address can now call `router.exactInputSingle` and pass the extension check. The per-user allowlist is completely bypassed. Any user can trade on a pool that was intended to be restricted.

Both paths satisfy the contest's "broken core pool functionality" and "allowlist bypass" impact gates. Path B is a direct loss-of-curation impact on pools that rely on the allowlist for compliance or risk management.

---

### Likelihood Explanation

- The `SwapAllowlistExtension` is a production periphery contract explicitly listed as a reference implementation.
- Every pool that configures `SwapAllowlistExtension` and expects users to interact through the router will hit one of the two failure modes immediately, with no special preconditions.
- No privileged action, malicious setup, or non-standard token is required — only the normal router → pool call path.

---

### Recommendation

The extension must identify the actual end-user, not the intermediary. Two options:

1. **Router encodes the real user in `extensionData`**: `MetricOmmSimpleRouter` prepends `abi.encode(msg.sender)` to `extensionData` before forwarding to the pool. `SwapAllowlistExtension` decodes and checks that address. This requires a trusted router convention.

2. **Pool exposes an authenticated sender field**: Add an optional `realSender` parameter to `pool.swap` (or a separate authenticated-swap entry point) that the router populates with `msg.sender`. The extension reads this field instead of the raw `sender`.

Either way, the extension must check the address of the entity that initiated the economic action, not the address of the contract that relayed it — mirroring the correct pattern already used in `DepositAllowlistExtension`.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true).
3. Alice calls router.exactInputSingle({pool, tokenIn, ...}).
   → router calls pool.swap(recipient, ...) — pool's msg.sender = router.
   → _beforeSwap(sender=router, ...) is dispatched.
   → extension checks allowedSwapper[pool][router] → false → NotAllowedToSwap revert.
   Alice cannot swap despite being explicitly allowlisted.

4. Pool admin calls setAllowedToSwap(pool, address(router), true)
   (workaround to unblock router users).
5. Mallory (not allowlisted) calls router.exactInputSingle({pool, tokenIn, ...}).
   → extension checks allowedSwapper[pool][router] → true → passes.
   Mallory swaps successfully on a pool she was never authorized to access.
```
