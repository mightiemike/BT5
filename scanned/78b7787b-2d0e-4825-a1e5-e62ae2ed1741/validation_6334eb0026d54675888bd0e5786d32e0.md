### Title
SwapAllowlistExtension Allowlist Bypassed via Router: `sender` Is Always the Router, Not the Real User — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. However, when a swap is routed through `MetricOmmSimpleRouter`, the `sender` the extension receives is always the **router's address**, not the originating user. Any unprivileged user can therefore bypass the allowlist by routing through the public router.

---

### Finding Description

**Call chain:**

```
User (not allowlisted)
  → MetricOmmSimpleRouter.exactInputSingle()
      → pool.swap(recipient, ...)          // msg.sender in pool = router
          → _beforeSwap(msg.sender=router, recipient, ...)
              → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                  → checks allowedSwapper[pool][router]   ← router, NOT the user
```

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap`, it is the `msg.sender` of that call: [4](#0-3) 

The pool admin faces an impossible choice:
- **Do not allowlist the router** → all router-mediated swaps revert for every user, including allowlisted ones.
- **Allowlist the router** → every user, including non-allowlisted ones, can bypass the guard by routing through the router.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

---

### Impact Explanation

The `SwapAllowlistExtension` is the protocol's primary mechanism for restricting swap access to specific addresses (e.g., KYC'd users, designated market makers, private pool participants). When this guard is bypassed, any unprivileged user can execute swaps in pools that are intended to be restricted. Depending on the pool's purpose, this can lead to:

- Unauthorized users draining LP liquidity at oracle-quoted prices in pools designed for private or permissioned market-making.
- Complete nullification of the access-control invariant the pool admin configured, constituting broken core pool functionality.

---

### Likelihood Explanation

The bypass requires only that the pool admin has allowlisted the router (a necessary step for any allowlisted user to use the router). Any user who observes the allowlist configuration and the router's allowlisted status can immediately exploit this by calling `exactInputSingle` / `exactInput` / `exactOutput` / `exactOutputSingle` on the public router. No special privileges, flash loans, or complex setup are required.

---

### Recommendation

The `SwapAllowlistExtension` must check the **originating user**, not the immediate caller of `pool.swap`. Two complementary fixes:

1. **In the router:** pass the originating user's address as `callbackData` or use a dedicated field, and have the pool surface it to extensions. Alternatively, the router should not be allowlisted; instead, it should forward the real user identity.

2. **In `SwapAllowlistExtension`:** if the `sender` is a known router, extract and verify the real user from `extensionData` (which the router already forwards per-hop).

The simplest safe fix is to require that `sender == tx.origin` or that the router passes the real user address in `extensionData` and the extension validates it, rather than trusting the `sender` argument which is always the immediate `msg.sender` of `pool.swap`.

---

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Pool admin allowlists the router so that allowlisted users can use it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attacker (not individually allowlisted) calls the router directly.
// The extension sees sender=router, which IS allowlisted → swap succeeds.
vm.prank(attacker); // attacker is NOT in allowedSwapper[pool]
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 10_000,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Swap succeeds despite attacker not being individually allowlisted.
```

The `SwapAllowlistExtension` checks `allowedSwapper[pool][router]` (true) and passes, completely ignoring that the originating user is `attacker`. [5](#0-4) [6](#0-5)
