### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks whether the **router** is allowlisted — not the actual end-user. Any pool that allowlists the router (a natural operational choice) exposes its liquidity to every unpermissioned user who calls the router.

---

### Finding Description

The call chain is:

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ..., extensionData)   // msg.sender = router
              → _beforeSwap(msg.sender=router, recipient, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

Inside `MetricOmmPool.swap`, the pool passes its own `msg.sender` (the router) as the `sender` argument to the extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

If the pool admin allowlists the router address (e.g., so that standard router users can interact with the pool), the check `allowedSwapper[pool][router]` passes for **every** caller of the router, regardless of whether that caller was individually allowlisted. The router itself performs no identity forwarding — it simply calls `pool.swap` with `params.recipient` and its own address becomes the pool-level `sender`: [4](#0-3) 

---

### Impact Explanation

A pool deploying `SwapAllowlistExtension` intends to restrict which addresses may trade against its liquidity. Allowlisting the router collapses that restriction to zero: any unpermissioned address can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router and execute swaps against the pool's full liquidity at oracle-derived prices. The pool's LP funds are exposed to every public user, defeating the allowlist entirely.

---

### Likelihood Explanation

The scenario is operationally natural. A pool admin who wants to restrict direct pool access while still supporting the standard periphery router would allowlist the router address. The `SwapAllowlistExtension` documentation and interface give no indication that doing so opens the pool to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a call to the public router.

---

### Recommendation

The extension must gate the **original transaction initiator**, not the immediate pool caller. Two sound approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Check `tx.origin` as a fallback identity when `sender` is a known router**: Fragile and generally discouraged.
3. **Preferred — router-level allowlist enforcement**: The router checks the allowlist itself before calling the pool, so the pool-level extension only needs to gate direct (non-router) callers. The extension would then check `sender` for direct calls and trust the router to have pre-validated the user.

The cleanest fix is option 1: the router appends `abi.encode(msg.sender)` to `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a recognized router.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — a natural step to enable standard router usage.
3. Attacker (not individually allowlisted) calls `router.exactInputSingle(...)` targeting the pool.
4. Pool calls `_beforeSwap(msg.sender=router, ...)`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker receives pool output tokens; pool's LP balances are reduced without the allowlist having gated the actual trader. [5](#0-4) [6](#0-5)
