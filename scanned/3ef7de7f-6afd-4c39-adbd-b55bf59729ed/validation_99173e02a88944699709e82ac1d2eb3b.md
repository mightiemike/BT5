### Title
SwapAllowlistExtension Gates the Router Address Instead of the Real User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which equals `msg.sender` of the pool's `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, that caller is the **router contract**, not the end user. A pool admin who allowlists the router so that legitimate users can trade through it inadvertently opens the gate to every address on-chain, completely defeating the curation the extension was deployed to enforce.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every before-swap extension hook: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then uses that value to look up the per-pool allowlist: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry-point) calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` (all recursive hops): [4](#0-3) [5](#0-4) [6](#0-5) 

The result is that the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][realUser]`. A pool admin faces an impossible choice:

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | **Blocked** (broken UX) | Blocked |
| Yes | Passes | **Passes** (bypass) |

Allowlisting the router — the only way to let curated users trade through the standard periphery — simultaneously grants every address on-chain the ability to swap on the restricted pool.

---

### Impact Explanation

A pool protected by `SwapAllowlistExtension` is typically a curated venue: only vetted counterparties are permitted to trade against the LP's oracle-priced inventory. Once the router is allowlisted, any unprivileged address can call `exactInputSingle` (or any other router entry-point) and execute swaps against the pool's bins. LP principal is directly exposed to adversarial flow (e.g., informed traders, sandwich bots) that the allowlist was specifically deployed to exclude. This is a direct loss of LP assets above Sherlock thresholds whenever the pool carries meaningful liquidity.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical, documented swap entry-point for end users. Any pool admin who wants allowlisted users to trade through the router must allowlist it. The bypass is therefore reachable on every production pool that uses `SwapAllowlistExtension` with router support enabled — a configuration that is both natural and expected. No special privileges, no malicious setup, and no non-standard tokens are required; a plain `exactInputSingle` call suffices.

---

### Recommendation

Pass the **original user** through the call chain rather than the immediate `msg.sender`. Two complementary approaches:

1. **Router-side**: Store the originating user in transient storage (already done for the payer in `_setNextCallbackContext`) and forward it as a dedicated `swapper` field inside `extensionData`. The extension decodes and checks that field instead of `sender`.

2. **Extension-side**: Accept an explicit `swapper` address in the extension payload and verify it against the allowlist, ignoring the `sender` argument when a trusted router is the caller. The extension can verify the router is a known periphery contract (e.g., via a factory-registered router list) before trusting the embedded swapper identity.

Either approach must ensure the checked identity is the economically relevant actor — the address whose tokens are ultimately pulled — not the intermediate contract that relays the call.

---

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension as before-swap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so that allowlisted users can trade via the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended curated user.
   Pool admin does NOT allowlist bob.

Attack
──────
4. bob (non-allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: bob, amountIn: X, ...})

5. Router calls pool.swap(recipient=bob, ...) — msg.sender = router.

6. Pool calls extension.beforeSwap(sender=router, ...).

7. Extension evaluates:
     allowedSwapper[pool][router] == true  →  check passes.

8. Swap executes; bob receives output tokens from the LP's inventory.
   The allowlist provided zero protection.

Direct call (control)
─────────────────────
9. bob calls pool.swap(...) directly — msg.sender = bob.
10. Extension evaluates allowedSwapper[pool][bob] == false → reverts NotAllowedToSwap.
    Direct path is correctly blocked; router path is not.
``` [7](#0-6) [8](#0-7) [9](#0-8)
