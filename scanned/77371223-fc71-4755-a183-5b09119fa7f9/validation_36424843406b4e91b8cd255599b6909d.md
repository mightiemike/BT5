### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the actual end-user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is designed to restrict which addresses may swap on a curated pool. Its `beforeSwap` hook checks the `sender` argument, which is the `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the actual end-user. Any pool admin who allowlists the router (the only way to let allowlisted users use the standard periphery path) simultaneously opens the gate to every non-allowlisted user who routes through the same contract.

---

### Finding Description

**Hook check — `SwapAllowlistExtension.beforeSwap`** [1](#0-0) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the pool calls the extension). `sender` is the first argument forwarded by the pool — which is the `msg.sender` of the pool's own `swap()` call.

**Pool wires `msg.sender` as `sender`** [2](#0-1) 

```solidity
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
    extensionData
);
```

**Router calls `pool.swap()` as itself** [3](#0-2) 

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

When `exactInputSingle` (or any router entry point) is called, the router is `msg.sender` of `pool.swap()`. The pool therefore passes `sender = router` to `_beforeSwap`, and the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`.

**The inescapable dilemma for the pool admin:**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the standard periphery path at all |
| Allowlist the router | Every non-allowlisted user bypasses the guard by routing through the same contract |

There is no configuration that simultaneously permits allowlisted users to use `MetricOmmSimpleRouter` and blocks non-allowlisted users from doing the same. The `extensionData` bytes field is available in the hook signature but is unused by `SwapAllowlistExtension`, so there is no in-band mechanism to carry the real caller's identity. [4](#0-3) 

```solidity
mapping(address pool => mapping(address swapper => bool)) public allowedSwapper;
mapping(address pool => bool) public allowAllSwappers;
```

The allowlist is keyed by `(pool, swapper)` where `swapper` is the immediate caller of `pool.swap()`. There is no secondary key for the real originator.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and then allowlists the router (the natural step to let approved users trade via the standard periphery) inadvertently opens the pool to every address on-chain. Non-allowlisted users can execute swaps at oracle-derived prices on a pool that was explicitly configured to exclude them. This breaks the core allowlist invariant — the admin-configured guard is bypassed by an unprivileged path — and falls squarely within the "admin-boundary break" and "broken core pool functionality" impact categories.

---

### Likelihood Explanation

Medium. The bypass requires the pool admin to allowlist the router. However, allowlisting the router is the only way to let approved users trade through the standard periphery path (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). Any pool admin who wants their allowlisted users to use the router will trigger the vulnerability. The router is a public, permissionless contract, so once the router is allowlisted, any address can exploit the bypass without any further precondition.

---

### Recommendation

The `SwapAllowlistExtension` must gate the real end-user, not the immediate caller of `pool.swap()`. Two viable approaches:

1. **Pass the real caller via `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and verifies it against the allowlist. The pool admin must also configure the extension to trust the router as a forwarder.

2. **Check `sender` and fall back to a signed identity in `extensionData`**: The extension accepts either a directly allowlisted `sender` or a signed proof (e.g., EIP-712 permit) that the real originator is allowlisted, forwarded through `extensionData`.

Either approach requires a coordinated change to both the router and the extension.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin allowlists Alice: `setAllowedToSwap(pool, alice, true)`.
3. Pool admin allowlists the router so Alice can use the standard periphery: `setAllowedToSwap(pool, router, true)`.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` — `msg.sender` of `pool.swap` is the router.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes without checking Bob.
8. Bob's swap executes at oracle price on the curated pool, bypassing the allowlist entirely. [5](#0-4)
