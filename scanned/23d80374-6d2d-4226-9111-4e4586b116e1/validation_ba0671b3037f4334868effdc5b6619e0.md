### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
The `SwapAllowlistExtension.beforeSwap` hook gates swaps by checking the `sender` argument against a per-pool allowlist. When users swap through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so `sender` passed to the extension is the router address — not the actual user. This makes per-user access control impossible for router-mediated swaps: if the router is allowlisted (a natural admin action to enable router usage), every user on the network can bypass the allowlist.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with its own `msg.sender`: [1](#0-0) 

`_beforeSwap` then encodes that value as the `sender` argument forwarded to every configured extension: [2](#0-1) 

**Step 2 — Router calls `pool.swap` without exposing the actual user.**

`MetricOmmSimpleRouter.exactInputSingle` (and all other swap entry points) calls `pool.swap` directly. The actual user (`msg.sender` of the router call) is stored only in transient storage for the payment callback; it is never passed to the pool: [3](#0-2) 

So when the pool calls the extension, `sender = router address`.

**Step 3 — Extension checks the router address, not the actual user.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [4](#0-3) 

The allowlist is keyed `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. Individual user identities are invisible to the extension.

### Impact Explanation

Two broken states result from this mismatch:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot swap through the router; they must implement the swap callback themselves — broken core functionality |
| Router **allowlisted** | Every address on the network can bypass the allowlist by routing through the router — complete allowlist bypass |

The second state is the critical one. A pool admin who wants to allow specific users to use the router will naturally add the router to the allowlist. This single admin action silently opens the pool to all callers. Any non-allowlisted user can then trade on a pool that was supposed to be curated (e.g., restricted to institutional counterparties, KYC-verified addresses, or exclusive LPs). This matches the "admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact gate and "broken core pool functionality."

### Likelihood Explanation

The trigger is a pool admin allowlisting the router — a routine, non-malicious action that any operator would take to make the pool usable through the standard periphery. The admin has no indication that this action opens the pool to all users. The attacker needs no special privilege: any EOA can call `router.exactInputSingle` with the target pool.

### Recommendation

The extension must check the identity of the economic actor, not the intermediary contract. Two approaches:

1. **Pass the originating user through the pool.** Add an optional `originator` field to the swap call or encode it in `extensionData` in a tamper-evident way (e.g., signed by the router). The extension then checks `originator` instead of `sender`.
2. **Check `sender` only when it is a known EOA.** The extension could maintain a registry of trusted routers and, when `sender` is a trusted router, revert (since the real user cannot be identified). This forces curated pools to be accessed only via direct pool calls.

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)
   and setAllowedToSwap(pool, bob, true).
3. Pool admin calls setAllowedToSwap(pool, router, true)
   so that alice and bob can use the standard router.
4. charlie (not allowlisted) calls:
     router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(recipient, ...) → pool's msg.sender = router.
6. Pool calls extension.beforeSwap(router, ...).
7. Extension checks allowedSwapper[pool][router] → true.
8. charlie's swap executes on the curated pool.
9. The allowlist is completely bypassed.
```

The root cause — `sender` is the router, not the actual user — is structurally identical to the external report's "no check that `userOp.sender == msg.sender`": in both cases the guard checks a proxy identity rather than the economically relevant actor, and an attacker exploits the mismatch to consume a privilege they were never granted.
