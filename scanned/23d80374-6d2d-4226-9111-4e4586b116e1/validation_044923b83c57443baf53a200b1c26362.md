### Title
Missing Deadline Check in `addLiquidityExactShares` Allows Stale Liquidity Placement at Suboptimal Oracle Price — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` exposes `addLiquidityExactShares` (both overloads) without any expiration timestamp parameter or check. A user's add-liquidity transaction can sit in the mempool indefinitely and execute long after the oracle price has moved, placing the user's capital into bins that are now deeply out-of-range relative to the live oracle mid.

---

### Finding Description

The `MetricOmmSimpleRouter` correctly enforces a deadline on every swap entry point via `_checkDeadline`: [1](#0-0) 

All four swap functions call it at entry: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares`, however, accepts no `deadline` parameter and performs no timestamp check before calling into the pool: [6](#0-5) [7](#0-6) 

The `addLiquidityWeighted` overloads include cursor-bound guards (`minimalCurBin`/`maximalCurBin`) that check the pool's internal cursor position, but those bounds reflect pool state, not the live oracle price. If the oracle price has moved but no swap has yet shifted the pool cursor, the cursor check passes while the oracle mid is already stale. [8](#0-7) 

`addLiquidityExactShares` has no cursor-bound guard at all — only `maxAmountToken0`/`maxAmountToken1` caps, which prevent overpayment of tokens but do not prevent the user from receiving LP shares that are immediately worth less than the tokens deposited. [9](#0-8) 

---

### Impact Explanation

Metric OMM is an oracle-driven AMM: the bid/ask spread is set by `IPriceProvider` at swap time, not at liquidity-add time. A user who submits `addLiquidityExactShares` targeting bins near the current oracle mid can have that transaction delayed in the mempool. When it eventually executes:

1. The oracle price has moved; the user's chosen `binIdxs` are now far from the live mid.
2. The pool adds the user's tokens to those stale bins at the amounts the pool requests (bounded only by `maxAmountToken0`/`maxAmountToken1`).
3. Arbitrageurs immediately sweep the mispriced bins using the live oracle price, extracting value from the user's freshly deposited capital.
4. The user's LP position is worth materially less than the tokens they deposited — a direct loss of principal to arbitrage, not merely an opportunity cost.

This matches the allowed impact gate: **direct loss of user principal / LP assets** and **bad-price execution reaching a pool action**.

---

### Likelihood Explanation

- Any user interacting with `addLiquidityExactShares` during periods of network congestion or oracle price volatility is exposed.
- No privileged access is required; any normal user transaction can be delayed.
- The oracle price in Metric OMM can move continuously between the user's submission and execution, making this a routine risk rather than an edge case.

---

### Recommendation

Add a `uint256 deadline` parameter to both `addLiquidityExactShares` overloads and call `_checkDeadline` (or an equivalent inline check) before any pool interaction, mirroring the pattern already used in `MetricOmmSwapRouterBase`:

```solidity
modifier ensure(uint256 deadline) {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
    _;
}
```

Apply the same fix to `addLiquidityWeighted` overloads, since the cursor-bound check does not substitute for a deadline when the oracle price has moved without a corresponding pool cursor shift.

---

### Proof of Concept

1. Oracle mid is at price P; user calls `addLiquidityExactShares` targeting bins [−1, 0, 1] with `maxAmountToken0 = X`, `maxAmountToken1 = Y`.
2. Transaction is stuck in the mempool for 10 minutes due to gas price competition.
3. Oracle price moves 5% upward; pool cursor has not yet shifted (no swaps executed).
4. User's transaction executes: tokens are deposited into bins [−1, 0, 1], which are now below the live oracle mid.
5. An arbitrageur immediately calls `exactInputSingle` on the pool, sweeping token0 out of bins [−1, 0, 1] at the stale bin price, which is below the live oracle ask.
6. User's LP position now holds only token1 (the less-valuable leg at the new price), worth less than the X + Y tokens deposited. [6](#0-5) [9](#0-8)
