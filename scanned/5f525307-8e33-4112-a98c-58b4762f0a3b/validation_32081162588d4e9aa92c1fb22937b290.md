### Title
`SwapAllowlistExtension.beforeSwap` checks `sender` (the router) instead of the actual end-user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is a production extension that gates `swap` access to a per-pool allowlist of addresses. Its `beforeSwap` hook receives `sender` — the value the pool passes as `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the **router contract**, not the end user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. Any user who routes through the router either (a) is silently blocked even though they are individually allowlisted, or (b) can bypass the allowlist entirely if the router address is allowlisted.

---

### Finding Description

**Pool call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient=user, ..., callbackData)   // msg.sender = router
```

**Pool passes `msg.sender` as `sender` to the extension:**

`MetricOmmPool.sol` (swap function, line ~282):
```solidity
_beforeSwap(
  msg.sender,   // ← router address, not the end user
  recipient,
  ...
);
```

`ExtensionCalling.sol` (line 160–176):
```solidity
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)   // sender = router
  )
);
```

**Extension checks `sender` (the router):**

`SwapAllowlistExtension.sol` (lines 31–41):
```solidity
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` = pool (correct, used as pool key), but `sender` = router address (wrong actor). The pool admin calls `setAllowedToSwap(pool, userAddress, true)` to allowlist specific users. When those users route through `MetricOmmSimpleRouter`, the extension sees `sender = routerAddress`, which is not in the allowlist, so the swap reverts — even for allowlisted users.

Conversely, if the pool admin allowlists the router address (a natural operational choice to "allow the router"), **every user** can bypass the allowlist by routing through it, because the extension only checks whether the router is allowlisted, not the individual user.

Compare with `DepositAllowlistExtension.beforeAddLiquidity` (lines 32–42), which correctly checks `owner` (the position owner, the economically relevant actor for deposits). The swap extension should analogously check `recipient` (the beneficiary of the swap output), not `sender` (the intermediary router).

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of counterparties loses that restriction entirely for any user routing through `MetricOmmSimpleRouter`. An unauthorized user can:

- Execute swaps in a pool they are not permitted to access.
- Extract value from LP positions at oracle-anchored prices the pool admin intended to reserve for vetted counterparties.
- Drain LP principal if the pool's oracle pricing is favorable and the allowlist was the only access control.

This is a direct loss of LP principal and a complete bypass of a configured security boundary — matching the "allowlist bypass" and "wrong-actor binding" impact categories.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any user who follows normal protocol usage routes through it. No special privileges, malicious setup, or non-standard tokens are required. The bypass is triggered by ordinary router usage.

---

### Recommendation

In `SwapAllowlistExtension.beforeSwap`, check `recipient` (the second argument, the actual beneficiary of the swap) instead of `sender` (the router):

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`recipient` is the address that receives the output tokens and is the economically relevant actor for swap access control, consistent with how `DepositAllowlistExtension` uses `owner` for deposit gating.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is permitted to swap.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(recipient=bob, ...)` — `msg.sender` of pool.swap() = router.
5. Pool calls `extension.beforeSwap(sender=router, recipient=bob, ...)`.
6. Extension checks `allowedSwapper[pool][router]` — router is not in the allowlist, so this reverts for Bob. ✓ (correct block)
7. Admin now calls `setAllowedToSwap(pool, router, true)` to allow router-based swaps for Alice.
8. Bob again calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
9. Extension checks `allowedSwapper[pool][router]` — router IS allowlisted → swap succeeds for Bob. ✗ (allowlist bypassed)

The invariant "only allowlisted users may swap" is broken: allowlisting the router grants access to all users indiscriminately. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
