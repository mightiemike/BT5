### Title
`DepositAllowlistExtension` Gates on `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` argument (the address that actually provides tokens to the pool) and instead gates on `owner` (the address that receives the resulting LP shares). Because the pool's `addLiquidity` call allows `sender ≠ owner`, any non-allowlisted address can deposit funds into a permissioned pool simply by naming an allowlisted address as `owner`, bypassing the guard entirely.

---

### Finding Description

`DepositAllowlistExtension` is the production extension that enforces a per-pool deposit allowlist. Its `beforeAddLiquidity` hook receives two address arguments: `sender` (the caller who transfers tokens into the pool) and `owner` (the LP-position beneficiary). The implementation discards `sender` and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

Compare this with the sibling `SwapAllowlistExtension`, which correctly gates on `sender` (the actual swapper):

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [2](#0-1) 

The pool's extension-calling infrastructure passes `sender` and `owner` as distinct arguments to every hook:

```solidity
// metric-core/contracts/ExtensionCalling.sol
function _beforeAddLiquidity(
    address sender,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) internal {
    _callExtensionsInOrder(
        BEFORE_ADD_LIQUIDITY_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
    );
}
``` [3](#0-2) 

Because `sender` and `owner` are separate, the pool supports the pattern where the caller (`sender`) deposits tokens on behalf of a different beneficiary (`owner`). The extension's check on `owner` alone means the `sender`'s allowlist status is never evaluated.

---

### Impact Explanation

A non-allowlisted address can:

1. Approve the pool to spend its tokens.
2. Call `pool.addLiquidity(owner = <allowlisted address>, ...)`.
3. The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` → hook passes.
4. The non-allowlisted caller's tokens flow into the pool; the allowlisted address receives LP shares it did not request.

Consequences:
- **Admin-boundary break**: the pool admin's configured deposit guard is bypassed by any unprivileged address.
- **Pool insolvency risk**: a permissioned pool designed to accept liquidity only from vetted counterparties now accepts tokens from arbitrary sources, corrupting the pool's intended liquidity composition and potentially its price/bin state.
- **Griefing / forced LP position**: an attacker can force an allowlisted address to hold an LP position it never initiated, exposing that address to impermanent loss and fee obligations on removal.

---

### Likelihood Explanation

- Requires no special privilege — any EOA or contract can trigger this.
- The only precondition is knowing one allowlisted address for the target pool, which is discoverable on-chain from `AllowedToDepositSet` events.
- The `DepositAllowlistExtension` is a production periphery contract intended for real permissioned pools.

---

### Recommendation

Gate on `sender` (the token provider), consistent with how `SwapAllowlistExtension` gates on `sender` (the swapper):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate on the LP-position owner (i.e., control who may *hold* positions rather than who may *fund* them), the contract name, event names (`AllowedToDepositSet`), and admin setter (`setAllowedToDeposit`) should be updated to reflect that semantics, and the asymmetry with `SwapAllowlistExtension` should be explicitly documented.

---

### Proof of Concept

```
Setup:
  pool P uses DepositAllowlistExtension E.
  Pool admin allowlists Alice: allowedDepositor[P][Alice] = true.
  Bob (non-allowlisted) holds 1000 token0 and approves pool P.

Attack:
  Bob calls pool.addLiquidity(owner = Alice, salt = X, deltas = ..., extensionData = "")

Extension hook fires:
  msg.sender = P  (the pool)
  owner       = Alice
  sender      = Bob  (ignored by the extension)

Check: allowedDepositor[P][Alice] == true  →  hook returns selector, no revert.

Result:
  Bob's 1000 token0 enter the pool.
  Alice receives LP shares she did not request.
  Bob (non-allowlisted) has successfully deposited into a permissioned pool.
