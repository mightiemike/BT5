The call chain is fully traceable. Here is the analysis:

---

## Call Chain Trace

**Direct swap:**
`User → pool.swap()` → `_beforeSwap(msg.sender=User, ...)` → `SwapAllowlistExtension.beforeSwap(sender=User, ...)` → checks `allowedSwapper[pool][User]` ✓

**Router-mediated swap:**
`User → router.exactInput()` → `pool.swap()` → `_beforeSwap(msg.sender=Router, ...)` → `SwapAllowlistExtension.beforeSwap(sender=Router, ...)` → checks `allowedSwapper[pool][Router]`

The pool passes `msg.sender` (the router) as `sender` to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that `sender` directly to the extension: [2](#0-1) 

The extension checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = whoever called `pool.swap()`: [3](#0-2) 

---

## The Structural Problem

The `SwapAllowlistExtension` is designed to gate swaps by swapper identity. But `sender` is always the **immediate caller of `pool.swap()`**, not the originating EOA. When the router is the caller, `sender = router`.

This creates an inescapable dilemma for pool admins:

| Admin action | Direct swap result | Router swap result |
|---|---|---|
| Allowlist only specific users (not router) | Allowlisted users can swap ✓ | Nobody can use router ✗ |
| Allowlist the router | Direct-only users blocked ✓ | **Every user bypasses allowlist** ✗ |

There is no configuration that allows allowlisted users to use the router while blocking non-allowlisted users. If the router is allowlisted, `allowedSwapper[pool][router] = true` passes for every user regardless of their identity.

The router itself does not forward the originating user's address anywhere in the swap call — it passes `msg.sender` (the user) only as the `payer` in transient callback storage, which the extension never sees: [4](#0-3) 

---

## Verdict

### Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension.beforeSwap`, making per-user allowlisting unenforceable for any pool that permits router access — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]` where `sender` is the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable router usage inadvertently grants every unprivileged user the ability to bypass the per-user allowlist.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it unchanged to every configured extension. `SwapAllowlistExtension.beforeSwap` uses this value as the identity to check against `allowedSwapper[pool][sender]`. When `MetricOmmSimpleRouter.exactInput` (or any `exact*` variant) calls `pool.swap`, `msg.sender` at the pool is the router address. The extension therefore checks whether the router is allowlisted, not whether the originating user is allowlisted. Since the router is a single shared contract, allowlisting it opens the gate for all users simultaneously.

### Impact Explanation
Any pool that:
1. Configures `SwapAllowlistExtension` to restrict swaps to a specific set of addresses, and
2. Allowlists the router so that those addresses can use it

is fully open to any unprivileged user via the router. The protection boundary the pool admin believed they were enforcing does not exist for router-mediated swaps. Depending on pool economics (favorable rates, restricted liquidity, etc.), this can result in unauthorized extraction of value from the pool.

### Likelihood Explanation
The router is the standard user-facing entry point. Any pool admin who wants their allowlisted users to be able to use the router must allowlist it. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a standard `exactInput` or `exactInputSingle` call.

### Recommendation
Pass the originating user's address through the swap path so extensions can check it. One approach: add an `originSender` field to the `extensionData` that the router populates with `msg.sender` before calling `pool.swap`, and have `SwapAllowlistExtension` read and verify it (with the pool authenticating that the field was set by a trusted router). Alternatively, document clearly that `SwapAllowlistExtension` only gates direct callers of `pool.swap` and cannot enforce per-user restrictions for router-mediated swaps.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as `EXTENSION_1`, `BEFORE_SWAP_ORDER = 1`.
2. Call `setAllowedToSwap(pool, router, true)` — allowlist the router so users can use it.
3. As an address `attacker` that is **not** in `allowedSwapper[pool]`, call `router.exactInputSingle(...)` targeting the pool.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`. The extension checks `allowedSwapper[pool][router] == true` and returns `beforeSwap.selector` without reverting.
5. The swap executes successfully for the non-allowlisted attacker.
6. Confirm: calling `pool.swap(...)` directly from `attacker` reverts with `NotAllowedToSwap`, proving the allowlist is only bypassed via the router.
