### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Original User, Allowing Any Unprivileged User to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the pool's `msg.sender` — the router contract — not the original end-user. When a pool admin allowlists the router to enable router-mediated swaps for approved users, every unprivileged user gains the same access by routing through `MetricOmmSimpleRouter`, completely defeating the per-user allowlist guard.

### Finding Description

**Root cause — wrong actor binding in the hook.**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`ExtensionCalling._beforeSwap` forwards whatever `sender` the pool passes:

```solidity
// metric-core/contracts/ExtensionCalling.sol L149-177
function _beforeSwap(address sender, address recipient, ...) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
    );
}
```

The pool passes its own `msg.sender` as `sender`. When `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant) calls the pool, the pool's `msg.sender` is the router:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The original user's address (`msg.sender` in the router) is stored only in the callback context for payment settlement; it is never forwarded to the pool's `swap` call as the `sender` identity. The pool therefore calls `_beforeSwap(router_address, ...)`, and the extension checks `allowedSwapper[pool][router_address]`.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to a curated set of users (e.g., KYC-verified addresses `alice`, `bob`).
2. Pool admin allowlists the router (`setAllowedToSwap(pool, router, true)`) so that approved users can trade via the standard periphery path.
3. Attacker (not in the allowlist) calls `router.exactInputSingle(...)` targeting the restricted pool.
4. Router calls `pool.swap(...)` → pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap executes.
5. Attacker successfully trades on a pool they should be barred from.

For multi-hop swaps, the same bypass applies to every intermediate hop because the router is always the pool's `msg.sender`.

**Analog to the external report:** Just as `ecrecover` returning `address(0)` caused `verify` to pass for the zero-address (the wrong identity), here the pool passing `router` as `sender` causes `beforeSwap` to pass for the router address (the wrong actor). In both cases, the guard checks an identity that is structurally different from the one the protocol intended to gate, and the mismatch is exploitable by any unprivileged caller.

### Impact Explanation

Any user can bypass a curated pool's swap allowlist by routing through `MetricOmmSimpleRouter`. The allowlist guard — the sole mechanism preventing non-approved counterparties from trading — silently fails open for all router-mediated swaps whenever the router itself is allowlisted. This constitutes a broken core pool functionality and a direct policy bypass with fund-impacting consequences: non-approved users can drain or manipulate pools that were designed to be restricted to specific counterparties. Severity: **High**.

### Likelihood Explanation

The bypass is unconditional once the router is allowlisted. A pool admin who wants approved users to use the standard periphery path must allowlist the router, which simultaneously opens the pool to all users. There is no configuration that simultaneously allows router-mediated swaps for approved users and blocks unapproved users. Any production deployment of a curated pool that uses the router is affected. Likelihood: **High**.

### Recommendation

The pool should forward the original user's address — not the immediate `msg.sender` — as the `sender` identity to extension hooks. One approach is for the router to pass the original user's address explicitly in the `swap` call (e.g., encoded in `extensionData`), and for `SwapAllowlistExtension` to decode and check that address. Alternatively, the pool can expose a dedicated `swapOnBehalfOf(address realSender, ...)` entry point that trusted periphery contracts use, and the extension checks `realSender`. The fix must ensure the checked identity is the economically responsible actor, not the intermediary contract.

### Proof of Concept

```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, alice, true)       // alice is approved
  pool admin: setAllowedToSwap(pool, router, true)      // router allowlisted so alice can use it

Attack:
  attacker (not in allowlist) calls:
    router.exactInputSingle({
        pool: pool,
        zeroForOne: true,
        amountIn: X,
        ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, ...)          // pool.msg.sender = router
        → _beforeSwap(router, ...)
          → extension.beforeSwap(router, ...)
            → allowedSwapper[pool][router] == true  ← passes!
        → swap executes, attacker receives output tokens

Result: attacker trades on a pool they are not allowlisted for.
