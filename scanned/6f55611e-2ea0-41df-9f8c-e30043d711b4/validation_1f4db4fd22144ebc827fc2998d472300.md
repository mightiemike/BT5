### Title
Permissionless `register()` Clears ADMIN-Set Pool Blacklist, Bypassing Emergency Oracle Access Revocation — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase.register()` is a permissionless, fee-gated function that unconditionally clears `blacklisted[pool]` whenever a pool re-registers for a feed. Because `setBlacklist()` is the only mechanism for ADMIN to revoke a pool's oracle-price access, any actor (including the pool operator itself) can pay the default 1 wei registration fee to immediately undo an ADMIN blacklist, restoring the pool's ability to call `price()` and execute swaps against live oracle data.

---

### Finding Description

`OracleBase` in `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol` defines two separate, asymmetric paths for the pool blacklist:

**Setting the blacklist — ADMIN only:** [1](#0-0) 

**Clearing the blacklist — permissionless via `register()`:** [2](#0-1) 

The `register()` function requires only that `msg.value >= registrationFee` (default: 1 wei), that `factory` is approved, and that `factory.isPool(pool)` returns true. If the pool is blacklisted, lines 207–210 silently clear it:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
```

The `price()` function — the sole on-chain oracle read path for pools — enforces the blacklist at call time: [3](#0-2) 

Once `register()` clears `blacklisted[pool]`, the `require(!blacklisted[pool])` check in `price()` passes, and the pool resumes reading live oracle prices.

The registration fee defaults to 1 wei at construction: [4](#0-3) 

---

### Impact Explanation

The blacklist is the emergency mechanism by which ADMIN revokes a pool's oracle access — for example, when a pool is found to be malicious, compromised, or draining LP funds through bad-price execution. Once ADMIN blacklists a pool, the intended invariant is that the pool cannot call `price()` and therefore cannot execute swaps.

Because `register()` clears the blacklist permissionlessly for 1 wei, this invariant is broken:

- The blacklisted pool (or any actor on its behalf) calls `register(feedId, pool, approvedFactory)` with 1 wei.
- `blacklisted[pool]` is set to `false`.
- The pool immediately resumes calling `price()`, reading live oracle bid/ask, and executing swaps.
- LP funds continue to be exposed to the condition that caused ADMIN to blacklist the pool in the first place.

This is a direct admin-boundary break: an ADMIN-level security action is overridden by a permissionless, near-zero-cost call.

---

### Likelihood Explanation

The trigger requires only:
1. A pool that is recognized by an approved factory (`IPoolFactory(factory).isPool(pool)` returns `true`).
2. 1 wei (the default `registrationFee`).
3. A single public transaction.

The pool operator whose pool was blacklisted has both the motive and the trivial means to call `register()` immediately after the ADMIN blacklist. No privileged access, no timelock, no special role is required.

---

### Recommendation

Separate the blacklist-clearing logic from `register()`. The blacklist should only be clearable by `ADMIN_ROLE`, not as a side-effect of a permissionless fee payment. One approach:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    require(pool != address(0));
    require(approvedFactories.contains(factory), FactoryNotApproved(factory));
    require(IPoolFactory(factory).isPool(pool), NotAPool(pool));
-   if (blacklisted[pool]) {
-       blacklisted[pool] = false;
-       emit BlacklistUpdated(pool, false);
-   }
+   require(!blacklisted[pool], Blacklisted(pool));  // blacklisted pools cannot re-register
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

This mirrors the keep-core fix: only the higher-privileged role (ADMIN) can reset the blacklist state; the lower-privileged permissionless path cannot override it.

---

### Proof of Concept

```solidity
// 1. ADMIN blacklists the pool (emergency response)
oracle.setBlacklist(address(pool), true);
// pool.price() now reverts with Blacklisted(pool)

// 2. Anyone (including the pool operator) calls register() with 1 wei
oracle.register{value: 1 wei}(feedId, address(pool), address(approvedFactory));
// register() executes lines 207-210: blacklisted[pool] = false

// 3. Pool reads oracle price again — blacklist fully bypassed
(uint256 mid, uint256 spread,,) = oracle.price(feedId, address(pool));
// Succeeds. Pool can now execute swaps against live oracle data.
```

The only preconditions are that `approvedFactory.isPool(pool)` still returns `true` (the pool has not been removed from the factory) and that `registeredPool[feedId][pool]` was previously set (or is being set for the first time). Both are trivially satisfied for any legitimately deployed pool.
