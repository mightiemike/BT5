### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Callers to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is a production guard that gates `addLiquidity` by depositor address, per pool. Its `beforeAddLiquidity` hook silently drops the `sender` argument (the actual caller of `addLiquidity`) and instead validates the `owner` argument (the position recipient). Any address not on the allowlist can bypass the guard by naming an allowlisted address as `owner`, depositing tokens into that address's position without restriction.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two identity parameters: `sender` (the `msg.sender` of the `addLiquidity` call, i.e., the party providing tokens via the callback) and `owner` (the address that will hold the resulting LP position). The implementation discards `sender` entirely and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), not the `recipient`:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [2](#0-1) 

The pool calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding both the real caller and the chosen owner to the extension: [3](#0-2) 

Because the extension ignores the first argument, the guard is keyed entirely on `owner`, not on who is actually providing tokens.

---

### Impact Explanation

An address not on the allowlist (`Bob`) can call `pool.addLiquidity(owner = Alice, ...)` where `Alice` is an allowlisted address. The extension checks `allowedDepositor[pool][Alice]` → `true` and permits the call. Bob's callback provides the tokens; Alice receives the LP position. The pool's accounting is correct, but the deposit allowlist — the admin-configured security boundary — is fully bypassed by an unprivileged caller.

Concrete consequences:
- **Allowlist rendered ineffective**: Any caller can participate in a nominally restricted pool by naming any allowlisted address as `owner`.
- **Forced position on allowlisted LP**: Alice receives a position she did not initiate; she must spend gas to remove it.
- **Collusion path**: If Bob and Alice collude, Alice removes the liquidity and returns proceeds to Bob, giving Bob full LP economics in a pool he is explicitly barred from.
- **Liquidity-distribution manipulation**: Bob can deposit into specific bins to shift the pool's bin balance, affecting price and fee distribution, without being on the allowlist.

This satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" impact gate.

---

### Likelihood Explanation

- The allowlisted owner addresses are discoverable on-chain (emitted in `AllowedToDepositSet` events).
- No special privilege is required; any EOA or contract can call `addLiquidity` directly.
- The only cost to the attacker is the tokens deposited (which go to the named owner), making purely griefing variants free of financial risk to the attacker if the owner cooperates.

Likelihood: **Medium** (requires a known allowlisted address; attacker bears token cost unless colluding with the owner).

---

### Recommendation

Replace the `owner` check with a `sender` check, consistent with `SwapAllowlistExtension`:

```solidity
// Before (buggy):

    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

If the intended semantic is "only allowlisted addresses may hold positions" (i.e., `owner`-gating is deliberate), the contract comment must be corrected and the bypass documented, and a separate `sender` check should be added to prevent unauthorized token injection.

### Proof of Concept

```
Setup:
  pool P has DepositAllowlistExtension E configured
  E.allowedDepositor[P][Alice] = true
  E.allowedDepositor[P][Bob]   = false  (Bob is blocked)

Attack:
  1. Bob calls P.addLiquidity(owner = Alice, salt = 0, deltas = ..., callbackData = ..., extensionData = "")
  2. Pool calls E.beforeAddLiquidity(sender=Bob, owner=Alice, ...)
  3. E checks allowedDepositor[P][Alice] → true → no revert
  4. LiquidityLib.addLiquidity executes; pool calls Bob's metricOmmAddLiquidityCallback
  5. Bob transfers tokens to the pool inside the callback
  6. Alice's position is credited with the deposited shares
  7. Bob has successfully deposited into a pool he is explicitly barred from
  8. Alice calls P.removeLiquidity(owner=Alice, ...) and returns proceeds to Bob
``` [1](#0-0)
