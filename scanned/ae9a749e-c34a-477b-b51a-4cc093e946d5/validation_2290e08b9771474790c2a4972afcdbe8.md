### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates access by checking the `sender` parameter, which the pool sets to `msg.sender` of its own `swap` call. When users route through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the actual end user. This makes it impossible to enforce per-user allowlisting for router-mediated swaps: allowlisting the router opens the pool to every user, while not allowlisting it silently blocks all router-based swaps even for allowlisted users.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` argument to every configured extension: [2](#0-1) 

**Step 2 — Router calls `pool.swap` directly; its own address becomes `sender`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` with no mechanism to forward the original caller's address: [3](#0-2) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` (all callback hops). In every case the pool's `msg.sender` is the router contract.

**Step 3 — `SwapAllowlistExtension` checks the wrong actor.**

`beforeSwap` ignores the `recipient` and all other parameters and checks only `sender` against the per-pool allowlist: [4](#0-3) 

`msg.sender` inside the extension is the pool (the pool calls the extension), and `sender` is the router. The effective check is therefore `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

---

### Impact Explanation

**Bypass path (High impact):** A pool admin who wants allowlisted users to be able to use the router must add the router to the allowlist (`allowedSwapper[pool][router] = true`). Once the router is allowlisted, *every* user — including those explicitly excluded — can call `router.exactInputSingle` and pass the check, because the extension only sees the router address. The per-user curation is completely nullified for all router-mediated swaps.

**Broken-functionality path (Medium impact):** If the pool admin does not allowlist the router, every router-based swap reverts with `NotAllowedToSwap` even for addresses that are individually allowlisted. Allowlisted users are forced to call the pool directly, bypassing the standard periphery UX and any slippage/deadline protections the router provides.

Both outcomes violate the invariant stated in the contest scope: *"A curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it."*

---

### Likelihood Explanation

The bypass requires only that the pool admin allowlists the router — a natural and expected action for any curated pool that is meant to be usable through the standard periphery. The admin has no on-chain signal that doing so opens the pool to all users; the extension's interface gives no indication that `sender` will be the router rather than the end user. Any non-allowlisted user can then call `router.exactInputSingle` with no special setup.

---

### Recommendation

The extension must gate on the actual end user, not the immediate caller of `pool.swap`. Two viable approaches:

1. **Pass the real user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; `SwapAllowlistExtension` decodes and checks it. This requires a convention between the router and the extension.
2. **Check `sender` only for direct pool calls; require a signed proof for router calls:** The extension verifies an EIP-712 signature from the real user when `sender` is a known router.

The simplest safe fix is approach 1, combined with documentation that makes the `sender`-vs-user distinction explicit.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls:
       swapExt.setAllowedToSwap(pool, alice, true);   // alice is the intended user
       swapExt.setAllowedToSwap(pool, router, true);  // needed so alice can use the router
3. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...});
4. Inside pool.swap:
       _beforeSwap(msg.sender=router, ...)
       → SwapAllowlistExtension.beforeSwap(sender=router, ...)
       → allowedSwapper[pool][router] == true  ✓  (passes)
5. Bob's swap executes successfully despite not being on the allowlist.
```

Conversely, if step 2b is omitted (router not allowlisted), Alice's router call fails at step 4 because `allowedSwapper[pool][router] == false`, even though `allowedSwapper[pool][alice] == true`.
