### Title
SwapAllowlistExtension gates the router address instead of the real swapper, allowing any user to bypass the per-pool swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (the only way to let allowlisted users trade through the router), every user — including those explicitly excluded from the allowlist — can bypass the guard by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (the extension is called by the pool). `sender` is the value forwarded by `ExtensionCalling._beforeSwap`, which is `msg.sender` of the pool's `swap()` call:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

And `sender` is set in `MetricOmmPool.swap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(msg.sender, recipient, zeroForOne, ...);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

So `msg.sender` of the pool's `swap()` is the **router**, not the end user. The extension sees `sender = router`.

The pool admin faces an impossible dilemma:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user — including non-allowlisted ones — can bypass the guard via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same. The `extensionData` bytes are forwarded through the router but `SwapAllowlistExtension` never decodes them; it only reads `sender`.

---

### Impact Explanation

A pool admin who deploys a curated pool (e.g., KYC-gated, institution-only, or regulatory-restricted) and allowlists the router to support normal UX inadvertently opens the pool to every public user. Non-allowlisted users can execute swaps at oracle-anchored prices, extract value from the pool, front-run allowlisted LPs, and violate the pool's access-control invariant. This constitutes a broken core pool functionality and an admin-boundary break where an unprivileged path bypasses a configured protection.

---

### Likelihood Explanation

Allowlisting the router is the natural, expected action for any pool admin who wants allowlisted users to be able to trade through the supported periphery. The bypass is not obvious because the admin sees the router as a trusted intermediary, not as a bypass vector. Any production curated pool that enables router access is affected.

---

### Recommendation

The extension must identify the **economic actor**, not the immediate caller. Two viable approaches:

1. **Decode the original user from `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks that address instead of `sender`.
2. **Check `sender` only when `sender` is not a known router; otherwise check `recipient`**: The router always sets `recipient` to the end user's intended destination, which is a closer proxy for the economic actor.

The cleanest fix is approach 1: the router includes `abi.encode(msg.sender)` in `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a recognized router.

---

### Proof of Concept

```
Setup:
  pool = new MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
  admin calls setAllowedToSwap(pool, router, true)      // allow router for alice's UX

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})

  Router calls pool.swap(...) with msg.sender = router
  Extension checks allowedSwapper[pool][router] → true → PASSES
  Bob's swap executes at oracle price, draining pool value

Direct call check (for comparison):
  bob calls pool.swap(...) directly
  Extension checks allowedSwapper[pool][bob] → false → REVERTS ✓
```

The bypass is reachable by any public user on any curated pool whose admin has allowlisted the router — the only configuration that makes the router usable for legitimate allowlisted users. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
