### Title
`SwapAllowlistExtension` checks the router's address instead of the actual end-user's address, allowing any user to bypass the swap allowlist by routing through `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When `MetricOmmSimpleRouter` is the caller, `sender` is the router's address, not the actual end-user. A pool admin who allowlists the router so that legitimate users can trade through it inadvertently opens the gate for every user, defeating the curated-pool invariant entirely.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to every registered `beforeSwap` extension:

```solidity
// MetricOmmPool.sol
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ... {
    ...
    _beforeSwap(
        msg.sender,   // ← always the direct caller of pool.swap()
        recipient,
        ...
    );
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the allowlist:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry-point) calls `pool.swap()` directly, making the router itself the `msg.sender` seen by the pool:

```solidity
// MetricOmmSimpleRouter.sol
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,   // recipient = user-supplied
        params.zeroForOne,
        ...
    );
``` [3](#0-2) 

The `sender` the extension receives is therefore `address(MetricOmmSimpleRouter)`, not the EOA who called the router. The allowlist lookup `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user]`.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and wants legitimate allowlisted users to trade through the standard router must allowlist the router address. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for **every** caller of the router, regardless of whether that caller is on the allowlist. Any non-allowlisted user can bypass the curation policy by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry-point) instead of calling `pool.swap()` directly.

The result is a complete, unprivileged bypass of the swap allowlist on any curated pool that supports the router path — direct loss of the curation guarantee and potential direct financial loss to LPs who deposited under the assumption that only vetted counterparties could trade against their liquidity.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported periphery path for swaps. Any pool admin who configures `SwapAllowlistExtension` and also wants users to be able to use the router (the normal UX path) will allowlist the router. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — any EOA can exploit it by simply using the router instead of calling the pool directly.

---

### Recommendation

The extension must check the identity of the **economic actor** (the end-user), not the **transport layer** (the router). Two viable approaches:

1. **Check `recipient` instead of `sender`** — the router always sets `recipient` to the user-supplied address. However, `recipient` is the token-output destination, not necessarily the initiating user, so this is semantically imprecise.

2. **Pass the real user through `extensionData`** — the router encodes the original `msg.sender` into `extensionData`, and the extension decodes and checks it. This requires a coordinated change to the router and the extension.

3. **Check `tx.origin`** — simple but incompatible with smart-contract wallets and multicall patterns; generally discouraged.

The cleanest fix is option 2: the router encodes `msg.sender` into `extensionData` and the extension decodes it, falling back to the raw `sender` when no router-supplied identity is present.

---

### Proof of Concept

Setup:
- Deploy a pool with `SwapAllowlistExtension` registered on `beforeSwap`.
- Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` so that legitimate users can trade through the router.
- `alice` is NOT individually allowlisted: `allowedSwapper[pool][alice] == false`.

Attack:
1. `alice` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: alice, ...})`.
2. Router calls `pool.swap(alice, ...)` — pool's `msg.sender` = router.
3. Pool calls `_beforeSwap(router, alice, ...)`.
4. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
5. Swap executes; `alice` receives output tokens.

`alice` successfully traded against the curated pool despite never being allowlisted, because the extension checked the router's allowlist entry rather than hers. [4](#0-3) [1](#0-0) [5](#0-4)
