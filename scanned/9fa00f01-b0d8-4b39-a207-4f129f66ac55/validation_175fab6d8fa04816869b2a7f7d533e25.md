### Title
`SwapAllowlistExtension` checks router address instead of originating EOA, allowing any unprivileged user to bypass the per-pool swap gate when the router is allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating EOA. If the router address is allowlisted for a pool, every EOA — including those explicitly not allowlisted — can execute swaps against that pool by going through the router.

---

### Finding Description

`MetricOmmPool.swap` captures `msg.sender` and passes it as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-231
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap
    ...
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap`. When the path is `EOA → MetricOmmSimpleRouter.exactInputSingle → pool.swap`, `sender` is the router address. If `allowedSwapper[pool][router] = true`, the check passes for **any** EOA regardless of whether that EOA is individually allowlisted.

The pool admin may legitimately allowlist the router to permit router-mediated swaps for their allowlisted users, without realising that doing so grants swap access to every EOA that can call the public router.

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole per-pool swap gate. When bypassed, any EOA can execute swaps on a pool that the admin intended to restrict. This breaks the core access-control functionality of the extension and constitutes broken core pool functionality under the contest rules. Depending on the pool's purpose (e.g., private/institutional pools), this can lead to direct loss of LP value through unwanted price impact or arbitrage by unauthorised parties.

---

### Likelihood Explanation

The router is a public, permissionless contract. A pool admin who wants to allow router-mediated swaps for their allowlisted users has no other option than to allowlist the router address — the extension provides no mechanism to pass through the originating EOA. This is a natural and expected configuration, making the bypass reachable without any attacker privilege.

---

### Recommendation

`SwapAllowlistExtension` should check the originating EOA rather than (or in addition to) the direct caller. One approach: require callers to pass the intended swapper address in `extensionData`, and verify it against `tx.origin` or a signed permit. Alternatively, the extension should explicitly document that allowlisting a router grants access to all router users, and the pool admin UI/docs should warn against it. A more robust fix is to have the router forward the originating EOA in `extensionData` and have the extension validate it.

---

### Proof of Concept

1. Deploy `SwapAllowlistExtension` and a pool configured to use it.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting only the router.
3. `attackerEOA` (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Pool receives `pool.swap(...)` with `msg.sender = router` → extension sees `sender = router` → `allowedSwapper[pool][router] = true` → swap succeeds.
5. `attackerEOA` calls `pool.swap(...)` directly → extension sees `sender = attackerEOA` → `allowedSwapper[pool][attackerEOA] = false` → reverts with `NotAllowedToSwap`.

The per-user allowlist invariant is violated: the EOA that is blocked from direct swaps can freely swap through the router.
