### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the router's address, not the originating user. If the pool admin allowlists the router (a natural action to enable router-based swaps for curated pools), every unprivileged user bypasses the per-user allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates the allowlist keyed on that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router becomes `msg.sender` to the pool: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The actual economic actor — the user who initiated the call and whose funds are at stake — is never consulted.

The same substitution occurs in `exactInput` (intermediate hops use `address(this)` as payer), `exactOutputSingle`, and `exactOutput` (recursive callback hops call `pool.swap` from inside `metricOmmSwapCallback`): [5](#0-4) [6](#0-5) 

The `DepositAllowlistExtension` does not share this flaw because it keys on `owner` (the position owner explicitly passed by the caller), which the liquidity adder preserves correctly: [7](#0-6) 

---

### Impact Explanation

Two distinct impacts arise from the same root cause:

**Impact A — Allowlist bypass (High).** A pool admin who wants to allow router-based swaps for their curated pool will allowlist the router: `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call that arrives through the router, regardless of who the originating user is. Any address — including addresses the admin explicitly excluded — can trade on the curated pool by routing through `MetricOmmSimpleRouter`. The curated pool's access control is silently voided.

**Impact B — Broken core functionality (Medium).** If the admin does not allowlist the router, individually allowlisted users cannot use `MetricOmmSimpleRouter` at all: the extension sees the router as `sender`, finds it absent from the allowlist, and reverts with `NotAllowedToSwap`. The router — the primary supported swap entrypoint — is unusable for every curated pool that has `SwapAllowlistExtension` active, regardless of the user's individual allowlist status.

---

### Likelihood Explanation

The `SwapAllowlistExtension` is a production periphery contract explicitly listed in scope. `MetricOmmSimpleRouter` is the primary public swap entrypoint. Any pool admin who deploys a curated pool and wants to support router-based swaps will encounter Impact A the moment they allowlist the router. Any pool admin who does not allowlist the router will encounter Impact B immediately upon the first allowlisted user attempting a router swap. Both paths are reachable by normal, unprivileged user actions with no malicious setup required.

---

### Recommendation

The extension must resolve the originating user rather than the immediate pool caller. Two approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and checks that address. This requires a coordinated convention between the router and the extension.

2. **Check `recipient` instead of `sender` for swap allowlisting**: For exact-input swaps the recipient is the intended beneficiary; however, this does not generalize cleanly to exact-output or multi-hop flows.

3. **Preferred — dedicated router-aware allowlist**: The extension stores `allowedSwapper[pool][user]` and also accepts a signed or ABI-encoded user identity from trusted routers. The extension verifies `msg.sender` is a factory-registered pool and that the encoded user is allowlisted, rejecting any call where the identity cannot be verified.

At minimum, the `SwapAllowlistExtension` documentation must warn that allowlisting the router opens the pool to all users, and the test suite must cover the router path explicitly.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  allowedSwapper[pool][alice] = true   // alice is the only intended swapper
  allowedSwapper[pool][router] = true  // admin allowlists router to support router-based swaps for alice

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., extensionData: ""})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes, charlie receives tokens

Result:
  charlie swaps successfully on a pool that was supposed to be restricted to alice only.
  The allowlist is bypassed without any privileged action by charlie.
```

The bypass requires only that the pool admin has allowlisted the router — a natural and expected administrative action for any curated pool that intends to support the official periphery router. [3](#0-2) [1](#0-0) [4](#0-3)
