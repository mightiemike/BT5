### Title
SwapAllowlistExtension Gates the Router's Identity Instead of the End User's, Allowing Any User to Bypass the Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]` where `sender` is the first argument forwarded by the pool — always `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the allowlist checks the **router's** identity, not the **user's**. A pool admin who allowlists the router (the only way to let allowlisted users trade through the router) simultaneously opens the pool to every user on the router, completely defeating the access-control invariant.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to every extension.** [1](#0-0) 

`_beforeSwap` is called with `msg.sender` as the first argument. That value is forwarded verbatim to every configured extension. [2](#0-1) 

**Step 2 — `SwapAllowlistExtension` checks `sender`, not the end user.** [3](#0-2) 

`msg.sender` inside the extension is the pool; `sender` is the direct caller of `pool.swap()`. The guard is `allowedSwapper[pool][sender]`.

**Step 3 — The router calls `pool.swap()` directly, substituting its own address for the user's.** [4](#0-3) 

When `exactInputSingle` (or `exactInput`, `exactOutput`, `exactOutputSingle`) is called, the router is `msg.sender` to the pool. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The dilemma this creates for the pool admin:**

| Admin action | Effect |
|---|---|
| Does **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlists the router** | Every user on the router bypasses the allowlist |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` (e.g., a private institutional pool, a KYC-gated pool, or a pool restricted to specific market-makers) can be accessed by any unprivileged user simply by routing through `MetricOmmSimpleRouter`. Once inside, the unauthorized user can:

- Execute swaps against LP capital that was never intended to be exposed to them.
- Cause adverse selection losses for LPs (e.g., arbitrage against a pool designed only for low-frequency, informed traders).
- Drain one-sided liquidity from bins that the pool admin expected to be protected.

This is a direct admin-boundary break: the pool admin's access-control configuration is bypassed by an unprivileged path (the public router).

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract.
- No special privilege, token, or role is required to call it.
- The bypass requires only that the pool admin has allowlisted the router — a natural and expected operational step for any pool that wants allowlisted users to have a good UX.
- The attacker needs zero knowledge of the pool's internals; they only need to know the pool address and that the router exists.

---

### Recommendation

**Short term:** `SwapAllowlistExtension` should accept the true end-user identity through a verified channel. One approach: require the router to embed the user's address in `extensionData` and have the extension verify it against a known-router registry. Alternatively, restrict the allowlist check to `sender` only when `sender` is not a known router, and require a signed user identity when `sender` is a router.

**Long term:** Define a protocol-level convention for periphery contracts to attest the originating user's address in `extensionData` in a tamper-proof way (e.g., a signed payload or a transient-storage forwarding pattern), and update all identity-sensitive extensions to consume that convention instead of relying on `sender`.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)
   — only Alice is supposed to be able to swap.
3. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so Alice can use the router.
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) — msg.sender to pool = router.
6. Pool calls extension.beforeSwap(sender=router, ...)
7. Extension checks allowedSwapper[pool][router] → true → passes.
8. Bob's swap executes successfully against the "private" pool.
``` [5](#0-4) [6](#0-5) [1](#0-0)
