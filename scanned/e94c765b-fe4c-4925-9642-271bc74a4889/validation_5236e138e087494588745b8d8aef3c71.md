### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the original user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool call. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the original user. A pool admin who allowlists the router (the natural step to enable router-mediated swaps on a curated pool) inadvertently opens the allowlist to every user, since the extension never inspects the original caller's identity.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` is the on-chain gate for curated pools:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension's caller). `sender` is the first argument forwarded by the pool, which is set to `msg.sender` of the pool call:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

When a user swaps through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The pool's `msg.sender` is the **router**, so `sender = router` is what the extension receives. The extension then evaluates `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router (to permit router-mediated swaps), the check passes for **every user** regardless of their individual allowlist status.

The same flaw applies to `exactInput` (multi-hop) and `exactOutputSingle`/`exactOutput` paths — all router entry points call `pool.swap(...)` directly, making the pool see `msg.sender = router` on every hop.

**Contrast with `DepositAllowlistExtension`:** The deposit guard correctly checks `owner` (the position owner explicitly passed by the caller), not `sender`. `MetricOmmPoolLiquidityAdder` passes the original user as `owner`, so the deposit allowlist gates the economically relevant actor. The swap allowlist has no equivalent mechanism — it only has `sender`, which collapses to the router address on every router-mediated swap.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router to support standard periphery usage loses all access control over who can swap. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and trade on the pool as if they were allowlisted. This is a **direct loss of curation policy** and constitutes broken core pool functionality: the allowlist guard silently fails open for the primary supported swap path.

---

### Likelihood Explanation

The router is the primary supported periphery path for swaps. Pool admins who want to allow router-mediated swaps on a curated pool have no choice but to allowlist the router address. The moment they do, the allowlist is completely bypassed for all users. This is a predictable and likely operational configuration.

---

### Recommendation

The `SwapAllowlistExtension` must gate the **original user**, not the immediate pool caller. Two viable approaches:

1. **Pass the original user via `extensionData`**: Have the router encode `msg.sender` into `extensionData` before forwarding to the pool. The extension decodes and checks this address. This requires the extension to trust that the router correctly encodes the original caller, which can be enforced by only accepting `extensionData`-based identity from allowlisted routers.

2. **Align with the deposit pattern**: Introduce a dedicated `swapper` parameter (analogous to `owner` in `addLiquidity`) that the pool passes through to the extension, and have the router populate it with `msg.sender`. This is the cleanest fix and mirrors how `DepositAllowlistExtension` correctly gates by `owner`.

Until fixed, pool admins should **not** allowlist the router on curated pools and should require users to call `pool.swap()` directly.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists the router: allowedSwapper[pool][router] = true
  - Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] = false

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: attacker,
       ...
     })
  2. Router calls pool.swap(attacker, ...) — msg.sender to pool = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true → PASSES
  5. Swap executes; attacker receives output tokens

Result: attacker successfully swaps on a pool they should be blocked from,
        bypassing the curated allowlist entirely.
