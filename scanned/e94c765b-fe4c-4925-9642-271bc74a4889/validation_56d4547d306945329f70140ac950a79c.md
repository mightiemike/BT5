### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating Swapper, Enabling Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `sender`, which is the pool's `msg.sender`. When users swap through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension gates the router address rather than the actual user. A pool admin who allowlists the router to enable router-based swaps for their users inadvertently opens the pool to every unprivileged caller.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` into the call to each extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When a user routes through `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap` with `msg.sender = router`: [4](#0-3) 

So the extension receives `sender = router` and checks `allowedSwapper[pool][router]`. The actual originating user is invisible to the guard.

A pool admin who wants their allowlisted users to be able to use the standard router will call `setAllowedToSwap(pool, address(router), true)`. Once the router is allowlisted, the check passes for **every** caller of the router, regardless of whether that caller is individually allowlisted.

The allowlist admin setter: [5](#0-4) 

### Impact Explanation

The swap allowlist is the primary access-control boundary for restricted pools. Bypassing it lets any unprivileged user trade in a pool the admin intended to restrict (institutional-only, KYC-gated, regulatory-compliance pools). This is an admin-boundary break: an unprivileged path (`user → router → pool`) circumvents a pool-admin-configured guard. The pool's intended restriction is silently nullified the moment the router is allowlisted.

### Likelihood Explanation

The bypass requires the pool admin to allowlist the router — a natural and expected action for any admin who wants their allowlisted users to be able to use the standard periphery. The project's own audit-generation script flags this exact scenario as the primary validation concern for the swap allowlist gate: [6](#0-5) 

The documentation confirms: *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."* The current implementation does not satisfy this requirement.

### Recommendation

The extension must gate the originating user, not the intermediary router. Concrete options:

1. **`extensionData` forwarding**: Require the router to ABI-encode the originating user into `extensionData`; have the extension decode and check that address instead of `sender`.
2. **Direct-pool-only policy**: Document that pools using `SwapAllowlistExtension` must not allowlist any router or intermediary contract, and that allowlisted users must call `pool.swap` directly.
3. **Separate `originSender` field**: Extend the extension interface to carry a distinct `originSender` populated by the pool from a trusted source (with appropriate security analysis).

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured as the beforeSwap hook.

2. Pool admin allowlists the router (intending to enable router-based swaps
   for their individually allowlisted users):
     swapExtension.setAllowedToSwap(pool, address(router), true)

3. Attacker (address NOT individually allowlisted) calls:
     router.exactInputSingle(ExactInputSingleParams{ pool: pool, ... })

4. Router calls pool.swap(...) — msg.sender to the pool = router.

5. Pool calls extension.beforeSwap(sender=router, ...).

6. Extension evaluates allowedSwapper[pool][router] → true → no revert.

7. Attacker's swap executes in the restricted pool, bypassing the allowlist.
```

The root cause is in `SwapAllowlistExtension.beforeSwap` at line 37: `sender` is the router, not the user, so `allowedSwapper[pool][router]` is the effective gate rather than `allowedSwapper[pool][user]`.
