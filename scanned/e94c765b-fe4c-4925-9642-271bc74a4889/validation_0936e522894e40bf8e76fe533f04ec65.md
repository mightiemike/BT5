### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any user to bypass per-user swap restrictions via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. That argument is `msg.sender` of the pool's `swap()` call — the direct caller. When swaps are routed through `MetricOmmSimpleRouter`, the direct caller is the router, not the end user. If the pool admin allowlists the router address (a natural configuration to let allowlisted users benefit from multi-hop routing), every unprivileged user can bypass the per-user restriction by routing through the public router contract.

---

### Finding Description

**Root cause — wrong identity checked**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the allowlist keyed by `msg.sender` (the pool):

```solidity
// SwapAllowlistExtension.sol L37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`sender` is populated by `MetricOmmPool.swap`, which passes its own `msg.sender`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), NOT the end user
    recipient,
    ...
);
```

**Router path — identity substitution**

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` as `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool therefore passes `router` as `sender` to `_beforeSwap`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**The bypass**

A pool admin who wants allowlisted users to be able to use the router must allowlist the router address. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every call that arrives through the router — regardless of who the end user is. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` (a fully public contract with no access control) and the extension check passes unconditionally.

The pool admin cannot express "only these specific users may swap, whether directly or through the router" with the current design. The only choices are:
- Allowlist specific addresses → those users can only swap directly (not through the router).
- Allowlist the router → all users can swap through the router (allowlist is nullified).

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., a private institutional pool, a KYC-gated pool, or a pool with a stop-loss guard that relies on controlled swap flow) is fully open to any user the moment the router is allowlisted. Unauthorized swappers can:

- Execute adverse-selection trades against LP positions at oracle-derived prices.
- Drain LP funds by exploiting stale or manipulated oracle quotes that the allowlist was intended to gate.
- Trigger or circumvent the `OracleValueStopLossExtension` or `PriceVelocityGuardExtension` in ways the pool admin did not intend, since those guards assume only vetted counterparties reach the swap path.

This is a direct loss of LP principal and protocol fees above Sherlock medium thresholds.

---

### Likelihood Explanation

The configuration that enables the bypass — allowlisting the router — is the natural and expected setup for any restricted pool whose allowlisted users need multi-hop routing, slippage protection, or deadline enforcement. A pool admin who does not allowlist the router forces allowlisted users to call the pool directly, losing all periphery functionality. The bypass is therefore reachable in any realistic production deployment of a restricted pool that also supports the router. No privileged attacker action is required; any EOA can call the public router.

---

### Recommendation

The `SwapAllowlistExtension` must check the end user's identity, not the direct caller of `pool.swap()`. Two viable approaches:

1. **Extension-data forwarding**: The router encodes the end user's address in `extensionData`; the extension decodes and checks it. The pool admin must also configure the extension to trust the router as a forwarder.
2. **Separate originator field**: Add an `originator` parameter to the `beforeSwap` hook interface (distinct from `sender`) that the pool populates from a trusted forwarder registry or from `extensionData` when the direct caller is a known router.

Until fixed, pool admins must not allowlist the router address on any pool where per-user swap restrictions are security-critical.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, alice, true)      // alice is the intended swapper
  pool admin calls setAllowedToSwap(pool, router, true)     // to let alice use the router

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool:          restrictedPool,
        recipient:     bob,
        zeroForOne:    true,
        amountIn:      X,
        ...
    })

  → router calls pool.swap() with msg.sender = router
  → pool calls _beforeSwap(router, ...)
  → SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  → swap executes; bob receives tokens from LP funds
  → NotAllowedToSwap never fires
