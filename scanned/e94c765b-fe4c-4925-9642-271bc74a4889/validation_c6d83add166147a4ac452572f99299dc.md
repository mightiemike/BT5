### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore checks the router's address against the allowlist, not the actual swapper's address. This creates an irreconcilable dilemma for pool admins: if they allowlist the router so that allowlisted users can use it, every non-allowlisted user can also bypass the gate by routing through the same router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every registered extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever address called the pool: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same holds for every hop in `exactInput` and `exactOutput`: [5](#0-4) 

The result is that the extension always sees the router address as the swapper identity for any router-mediated trade. The pool admin faces an impossible choice:

| Admin action | Allowlisted users via router | Non-allowlisted users via router |
|---|---|---|
| Do **not** allowlist router | Blocked (broken UX) | Blocked |
| **Allowlist router** | Allowed | **Also allowed — bypass** |

There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

A pool deploying `SwapAllowlistExtension` intends to restrict swaps to a curated set of counterparties (e.g., KYC'd addresses, institutional LPs, or protocol-controlled addresses). Once the router is added to the allowlist — a necessary step for any allowlisted user to access the router — every address on the network can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the restricted pool. Unauthorized swappers can drain LP token reserves at oracle-quoted prices, causing direct loss of LP principal and owed assets. The allowlist guard is rendered completely ineffective for router-mediated paths.

---

### Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool admin who wants allowlisted users to access the router must add the router to the allowlist, at which point the bypass is immediately available to all users. The trigger requires no special privilege — any address can call the public router functions. The only mitigating factor is that the admin must have made the router-allowlisting mistake, but the protocol provides no warning or documentation that doing so opens the gate to everyone.

---

### Recommendation

Pass the original end-user address through the extension data or a dedicated field rather than relying on `msg.sender` of the pool call. One approach: have the router encode `msg.sender` (the actual user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when the caller is a known router. Alternatively, redesign the extension to check `tx.origin` only when `msg.sender` is a whitelisted router contract, or require direct pool interaction for allowlisted pools (removing router support for those pools entirely).

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension registered as a beforeSwap hook.
2. Admin calls setAllowedToSwap(pool, userA, true)       // allowlist a legitimate user
3. Admin calls setAllowedToSwap(pool, router, true)      // required so userA can use the router
4. Non-allowlisted userB calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...) — pool's msg.sender = router
6. Extension checks allowedSwapper[pool][router] == true  → passes
7. userB's swap executes on the restricted pool, bypassing the allowlist entirely.
```

The extension's `isAllowedToSwap(pool, userB)` view returns `false`, confirming userB is not on the allowlist, yet the swap succeeds. [6](#0-5) [7](#0-6) [8](#0-7)
