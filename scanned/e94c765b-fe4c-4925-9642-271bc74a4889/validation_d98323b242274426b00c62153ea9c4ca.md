### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the actual user. If the router is allowlisted for a pool, every user — including non-allowlisted ones — can bypass the per-user swap restriction by calling through the router.

---

### Finding Description

**Identity binding mismatch in the swap allowlist hook**

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][caller_of_pool_swap]`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly — the router is `msg.sender` to the pool: [4](#0-3) 

The pool therefore passes `sender = router_address` to the extension. The allowlist check becomes `allowedSwapper[pool][router]`, completely ignoring the actual end-user.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the economically relevant actor, explicitly passed by the caller), not `sender` (the immediate caller): [5](#0-4) 

The swap allowlist is inconsistent: it gates the wrong identity.

---

### Impact Explanation

**Bypass path (High impact):** A pool admin who allowlists the router address — a natural configuration when the intent is "allow all router-mediated swaps" — inadvertently opens the pool to every user. Any non-allowlisted address calls `router.exactInputSingle(pool, ...)` and the extension sees `allowedSwapper[pool][router] = true`, granting the swap. The per-user restriction is completely nullified.

**Broken-functionality path (Medium impact):** If the router is *not* allowlisted, individually allowlisted users cannot swap through the router at all, because the extension sees `allowedSwapper[pool][router] = false` and reverts with `NotAllowedToSwap`. The router is the primary user-facing entry point, so this renders the allowlist incompatible with normal usage.

Both paths have direct fund-impact consequences: the bypass allows unauthorized parties to drain favorable pricing from restricted pools; the broken path prevents legitimate LPs and traders from executing swaps through the standard periphery.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary user-facing swap entry point documented in the protocol.
- Pool admins configuring a restricted pool are likely to allowlist the router to enable normal UX, without realizing this collapses per-user gating.
- No special privilege or malicious setup is required: any EOA can call the public router.
- The `DepositAllowlistExtension` works correctly (checks `owner`), so the flaw is not obvious from reading one extension in isolation.

---

### Recommendation

**Short term:** In `SwapAllowlistExtension.beforeSwap`, check `recipient` (the second parameter) instead of `sender`, or add an explicit `originalSender` field to the extension data that the router populates with `msg.sender` before calling the pool.

**Long term:** Define a protocol-level convention for "the economically responsible actor" on each hook type and enforce it consistently across all extensions. For swaps, the router should forward the originating user address in `extensionData` so guards can gate the correct identity regardless of call depth.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Pool admin calls:
       extension.setAllowedToSwap(pool, router, true)
   (allowlists the router, intending to permit router-mediated swaps)
3. Non-allowlisted user `alice` calls:
       router.exactInputSingle({pool: pool, recipient: alice, ...})
4. Router executes:
       pool.swap(alice, zeroForOne, amount, limit, "", extensionData)
   with msg.sender = router.
5. Pool calls _beforeSwap(sender=router, ...).
6. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true
   → swap proceeds.
7. Alice, who was never individually allowlisted, successfully swaps on the
   restricted pool. Any non-allowlisted user can repeat this.
``` [6](#0-5) [4](#0-3) [1](#0-0)
