### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is intended to gate who may call `addLiquidity` on a private pool. Its `beforeAddLiquidity` hook validates the `owner` parameter (the LP-position recipient) against the allowlist, but the actual caller (`sender`) is never checked. Because `addLiquidity` accepts an arbitrary `owner` address, any unpermissioned address can bypass the allowlist entirely by supplying an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook: [1](#0-0) 

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData);
```

`sender` = the address that called `addLiquidity` (the router or EOA providing tokens via callback).
`owner` = the address that will hold the resulting LP shares (caller-supplied, unconstrained).

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely and validates only `owner`: [2](#0-1) 

```solidity
function beforeAddLiquidity(address /*sender*/, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The admin-facing setter names the parameter `depositor`, signalling the intent to restrict the calling party: [3](#0-2) 

```solidity
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
```

Yet the hook reads `allowedDepositor[pool][owner]`, not `allowedDepositor[pool][sender]`. The guard and the configured allowlist operate on incompatible entities — a structural mismatch identical in class to the quadratic-voting / linear-quorum incompatibility in the seed report.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly validates `sender`: [4](#0-3) 

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

The asymmetry confirms the deposit hook is checking the wrong field.

---

### Impact Explanation

**Direct impacts:**

1. **Allowlist bypass / unauthorized liquidity injection into private pools.** Any address — including sanctioned, unverified, or competitor addresses — can add liquidity to a pool whose admin believed it was restricted. The attacker calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The hook sees `owner = allowlistedAddress` (passes), the attacker's callback provides the tokens, and the allowlisted address receives LP shares it never requested.

2. **Forced LP-position creation on allowlisted addresses.** The allowlisted address receives an LP position it did not initiate. If the attacker chooses a salt that conflicts with an existing position, it modifies that position's composition without the owner's consent.

3. **Pool-composition manipulation in oracle-anchored pools.** Because Metric OMM bins are oracle-anchored, an attacker can inject liquidity at a chosen bin distribution, shifting `curBinIdx` / `curPosInBin` and altering the effective spread seen by subsequent swappers, potentially extracting value from existing LPs through adverse price execution.

4. **Regulatory / compliance failure.** Pools deployed for KYC/AML-gated institutional LPs are rendered non-compliant; the pool admin has no on-chain mechanism to prevent unauthorized deposits.

---

### Likelihood Explanation

- `addLiquidity` is a public, permissionless entry point — no special role or token is required to call it.
- The attacker only needs to know one allowlisted address (trivially discoverable from past `AllowedToDepositSet` events).
- No flash loan, oracle manipulation, or multi-step setup is required.
- The bypass is deterministic and repeatable every block.

---

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual caller) instead of `owner`, consistent with how `SwapAllowlistExtension` validates `sender` in `beforeSwap`:

```solidity
// BEFORE (wrong entity checked)
function beforeAddLiquidity(address /*sender*/, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// AFTER (correct entity checked)
function beforeAddLiquidity(address sender, address /*owner*/, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

If the intended semantic is genuinely "restrict which addresses may *own* positions" (not "restrict which addresses may *call* addLiquidity"), the admin-facing function and all documentation must be updated to reflect that, and a separate `sender` check should be added to prevent token injection by unauthorized callers.

### Proof of Concept

```
Setup:
  pool P has DepositAllowlistExtension E configured.
  Admin calls E.setAllowedToDeposit(P, alice, true).
  bob is NOT in the allowlist.

Attack:
  bob calls P.addLiquidity(
      owner       = alice,   // allowlisted → hook passes
      salt        = 0,
      deltas      = <any valid bin deltas>,
      callbackData = <bob's callback pays the tokens>,
      extensionData = ""
  );

Result:
  E.beforeAddLiquidity(sender=bob, owner=alice, ...) checks allowedDepositor[P][alice] == true → no revert.
  Pool calls bob's callback; bob transfers tokens to pool.
  alice receives LP shares for a position she never initiated.
  bob has successfully deposited into a pool he is not permitted to access.
```

The `removeLiquidity` path enforces `msg.sender == owner` (line 206 of `MetricOmmPool.sol`), so the attacker cannot withdraw alice's shares — but the deposit restriction is fully defeated. [2](#0-1) [4](#0-3) [5](#0-4) [6](#0-5)
