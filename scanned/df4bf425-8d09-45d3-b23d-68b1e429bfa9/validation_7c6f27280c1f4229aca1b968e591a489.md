### Title
SwapAllowlistExtension Bypass via Router: `sender` Is Always the Router Address, Not the Actual User - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When any user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the actual user. This means the allowlist can only gate the router address as a whole — it cannot gate individual users who route through the router. Any unprivileged user can bypass a per-user swap allowlist by calling the public router.

### Finding Description

**Root cause — identity mismatch in the hook dispatch chain:**

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`_beforeSwap` forwards this value unchanged to every configured extension:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
``` [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutput`, `exactOutputSingle`) calls the pool, `msg.sender` at the pool is the **router address**:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The allowlist check therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**The two broken states this creates:**

| Pool admin configuration | Effect |
|---|---|
| Router address NOT allowlisted | All router-based swaps revert — even for users who are individually allowlisted |
| Router address IS allowlisted | Every user on the network can swap through the router, bypassing the per-user gate entirely |

Neither state matches the intended design. The protocol's own audit-target description confirms the intended invariant: *"the hook must gate the same actor the pool designers thought they were allowlisting"* and *"assert the hook cannot be bypassed by routing through an intermediate public contract."* [5](#0-4) 

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., a private market-making pool, a KYC-gated pool, or a pool with a specific LP strategy) loses its access control entirely for any user who routes through the public `MetricOmmSimpleRouter`. An unauthorized trader can:

1. Execute swaps in a pool that was designed to reject them.
2. Extract value from LP positions at oracle-derived prices the pool admin intended to offer only to trusted counterparties.
3. Repeat until the pool's liquidity is drained or the price moves unfavorably.

This is a direct loss of LP principal — the same class as the external report's zero-shares bypass — because the guard that was supposed to prevent unauthorized value extraction is silently skipped.

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract. Any EOA or contract can call it.
- No special setup is required: the attacker simply calls `exactInputSingle` with the target pool address.
- The bypass is unconditional once the router is allowlisted (which is required for any legitimate router user to trade).
- The pool admin has no on-chain mechanism to distinguish "router call from allowlisted user" from "router call from attacker."

### Recommendation

The `sender` forwarded to extensions must be the **originating user**, not the intermediate router. Two complementary fixes:

1. **In the router:** pass the actual user's address as `recipient` of the extension context, or use a dedicated forwarding field. More practically, the router should pass the real `msg.sender` through `callbackData` or a separate transient slot so the pool can forward it to extensions.

2. **In `SwapAllowlistExtension`:** gate on the `recipient` parameter (which the router sets to the actual beneficiary) if the pool's design allows it, or require the pool to expose a "true originator" field in the extension call.

3. **Alternatively:** document that `SwapAllowlistExtension` only works for direct pool calls and must not be used with the router, and add a factory-level check or extension flag to enforce this.

### Proof of Concept

```solidity
// Pool is deployed with SwapAllowlistExtension.
// Admin allowlists the router so that legitimate users can trade via the router.
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not individually allowlisted) calls the router directly.
// The pool sees msg.sender = router (allowlisted), not attacker (not allowlisted).
// beforeSwap passes: allowedSwapper[pool][router] == true → no revert.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: largeAmount,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    tokenIn: token0,
    extensionData: ""
}));
// Attacker receives token1 from a pool that was supposed to reject them.
```
