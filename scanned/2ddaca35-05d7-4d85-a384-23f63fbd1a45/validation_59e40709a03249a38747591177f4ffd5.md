### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of Actual Caller `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates the caller-supplied `owner` field against the allowlist. Because `owner` is a free parameter that any caller can set to any allowlisted address, an unauthorized depositor can bypass the allowlist entirely by naming an allowlisted address as the LP position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct identity fields to the `beforeAddLiquidity` hook:

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `msg.sender` — the actual on-chain caller (the depositor who will pay tokens via callback)
- `owner` — a caller-supplied parameter that determines whose LP position receives the shares [1](#0-0) 

The hook interface forwards both:

```
IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData)
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address,`). It then checks `owner` — the caller-controlled field — against the allowlist:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The correct field to check is `sender` (the actual depositor paying tokens). By checking `owner` instead, the guard is trivially bypassed: any caller can pass any allowlisted address as `owner`, and the check passes regardless of whether the caller is authorized.

The `SwapAllowlistExtension` does not share this flaw — it correctly checks `sender` and discards `recipient`. [4](#0-3) 

---

### Impact Explanation

The `DepositAllowlistExtension` is the production mechanism for pool admins to restrict which addresses may deposit liquidity (e.g., KYC gating, whitelist-only pools). With this bug, the restriction is completely defeated: any unprivileged address can call `pool.addLiquidity(allowlisted_address, salt, deltas, ...)` directly, pass the extension check, pay tokens via callback, and mint LP shares into the allowlisted address's position. The pool admin's access control boundary is broken by an unprivileged path. Additionally, allowlisted users can have LP shares forced into their positions without consent (griefing).

---

### Likelihood Explanation

The pool's `addLiquidity` function is permissionless with respect to the `owner` parameter — any caller can supply any address. The `MetricOmmPoolLiquidityAdder` explicitly supports and tests adding liquidity on behalf of another owner (`test_exactShares_canAddOnBehalfOfAnotherOwner`). [5](#0-4) 

No special privilege, flash loan, or oracle manipulation is required. Any address that can call `pool.addLiquidity` directly (bypassing the `LiquidityAdder`) and fund the callback can exploit this.

---

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`, which checks `sender` and discards `recipient`.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Eve is **not** allowlisted.
3. Eve calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly.
4. Pool calls `_beforeAddLiquidity(eve, alice, ...)` → extension checks `allowedDepositor[pool][alice]` = `true` → **passes**.
5. Pool calls `metricOmmModifyLiquidityCallback` on Eve; Eve pays tokens.
6. LP shares are minted to Alice's position.
7. Eve has successfully deposited into a deposit-gated pool without being allowlisted. [3](#0-2) [6](#0-5)
