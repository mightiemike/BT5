### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass a Curated Pool's Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. If the pool admin allowlists the router (a natural action to enable standard periphery usage), every non-allowlisted user can bypass the curation gate by routing through the router.

### Finding Description

**Call chain:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
         → IMetricOmmPoolActions(pool).swap(recipient, ...)   // msg.sender = router
              → MetricOmmPool._beforeSwap(msg.sender=router, ...)
                   → ExtensionCalling._callExtensionsInOrder(...)
                        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether `sender` (the router) is on the allowlist for the pool (`msg.sender` inside the extension is the pool): [3](#0-2) 

The router never forwards the originating user's address to the pool — it only stores it in transient callback context for payment purposes: [4](#0-3) 

So the extension is structurally incapable of distinguishing which EOA initiated the router call. The allowlist entry that matters is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Analog to the external bug:** The external report shows `remove_lock_record_for_investor` writing `lock.value` back to `lock_index` (the same slot it read from) instead of the last lock's slot — the wrong entity is targeted. Here, `SwapAllowlistExtension` reads `sender` (the router) and checks it against the allowlist instead of the actual user — the wrong actor is gated.

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and also allowlists the router (to let approved users use the standard periphery) inadvertently opens the gate to every user. Any address can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. The intended curation policy — restricting swaps to specific KYC'd or institutional addresses — is completely bypassed. LPs who deposited under the assumption that only approved counterparties would trade against them are exposed to unrestricted public flow.

### Likelihood Explanation

The trigger is a reasonable, non-malicious admin action: allowlisting the router so that approved users can use the standard periphery. The `setAllowedToSwap` admin function is the only path to configure the allowlist: [5](#0-4) 

Any pool admin who wants approved users to access the pool through the router will hit this configuration. The exploit itself requires no privilege — any EOA calls the public router.

### Recommendation

The extension must receive and check the economically relevant actor, not the intermediary. Two options:

1. **Pass the originating user through the router:** The router should forward `msg.sender` as an additional field in `extensionData` (or a dedicated parameter), and the extension should decode and verify it. This requires a coordinated interface change.

2. **Check `recipient` instead of `sender`:** For swap allowlists the recipient is often the economically relevant party. However, `recipient` can also be a third-party address, so this is only correct if the pool's curation intent is to gate the output destination.

3. **Document the limitation clearly:** If the design intent is that the allowlist gates the direct caller of `pool.swap()` (not the originating EOA), the extension and pool admin tooling must make clear that allowlisting the router opens the gate to all users, and admins must allowlist individual EOAs who call the pool directly.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `approvedUser` is allowlisted.
// Admin also allowlists the router so approvedUser can use periphery.
swapExtension.setAllowedToSwap(address(pool), approvedUser, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // ← opens the gate

// Exploit: bannedUser bypasses the allowlist via the router.
vm.prank(bannedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: bannedUser,
        tokenIn: address(token1),
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        deadline: block.timestamp + 1,
        extensionData: ""
    })
);
// ✓ Swap succeeds — extension saw sender=router (allowlisted), not bannedUser (blocked).
```

The `beforeSwap` check resolves to `allowedSwapper[pool][router] == true` and passes, while `allowedSwapper[pool][bannedUser] == false` was never consulted. [6](#0-5) [7](#0-6)
