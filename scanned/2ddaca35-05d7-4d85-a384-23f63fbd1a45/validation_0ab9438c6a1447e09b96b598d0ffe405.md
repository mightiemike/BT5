### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against the per-pool allowlist. The pool passes `msg.sender` as `sender` to the extension. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the original user. This means the extension always checks the router's address, not the actual swapper's address, making the allowlist trivially bypassable by any user who routes through the public router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` from the pool's perspective: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The extension therefore never sees the original `msg.sender` of the router call ‚Äî it only ever sees the router's address. This creates an irreconcilable configuration dilemma:

- **If the pool admin allowlists the router** (to let allowlisted users swap via the router): every user, including non-allowlisted ones, can call `router.exactInputSingle(...)` and the extension will approve the swap because `allowedSwapper[pool][router] == true`.
- **If the pool admin does not allowlist the router**: allowlisted users cannot swap through the router at all, breaking the intended user flow.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

---

### Impact Explanation

A curated pool (e.g., KYC-gated, institutional-only, or regulatory-restricted) that deploys `SwapAllowlistExtension` to restrict who may trade can be bypassed by any unprivileged user simply by calling `MetricOmmSimpleRouter.exactInputSingle` or any multi-hop variant. The non-allowlisted user executes a real swap, receives output tokens, and the pool's LP assets are consumed in an unauthorized trade. This is a direct loss of the pool's curation guarantee and constitutes unauthorized access to LP-owned liquidity.

---

### Likelihood Explanation

The router is a public, permissionless contract. Any user can call it at any time. No special privileges, flash loans, or multi-step setup are required. The bypass is reachable on every swap through the router on any pool that has `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER`. Likelihood is **High**.

---

### Recommendation

The extension must gate on the **original user**, not the intermediary caller. Two approaches:

1. **Pass the original initiator through the router**: Have the router encode the original `msg.sender` into `extensionData` and have the extension decode and check it. This requires a trust assumption that the router is the only allowed intermediary.

2. **Check `recipient` instead of `sender`** (partial mitigation only ‚Äî does not cover all cases).

3. **Preferred ‚Äî check both `sender` and `recipient`**: Require that at least one of the two is allowlisted, or require the pool admin to explicitly allowlist the router and accept that the router is a trusted forwarder.

The cleanest fix is to have the pool pass the original initiator as a separate field, or to have the router forward the real user address in `extensionData` and have the extension decode it when `sender` is a known trusted router.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   ‚Äî necessary so that allowlisted users can swap via the router.
3. Non-allowlisted attacker calls:
     router.exactInputSingle(ExactInputSingleParams{
       pool: pool,
       tokenIn: token0,
       zeroForOne: true,
       amountIn: 1_000,
       ...
     })
4. Pool calls pool.swap(...) with msg.sender = router.
5. _beforeSwap passes sender = router to SwapAllowlistExtension.
6. Extension checks allowedSwapper[pool][router] == true ‚Üí passes.
7. Attacker receives token1 output. Swap settles. Allowlist is bypassed.

Alternatively, if the admin does NOT allowlist the router:
3b. Allowlisted user calls router.exactInputSingle(...)
4b. Extension checks allowedSwapper[pool][router] == false ‚Üí reverts NotAllowedToSwap.
    Allowlisted user cannot use the router at all.
