### Title
SwapAllowlistExtension Bypass via MetricOmmSimpleRouter — Unauthorized Swappers Can Trade Against Allowlisted Pools (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension.beforeSwap` hook gates swaps by checking `sender`, which is the direct `msg.sender` of `pool.swap()`. When a user routes through the public `MetricOmmSimpleRouter`, the router contract becomes `msg.sender` to the pool. The allowlist therefore checks the router's address, not the actual end-user's address. If the router is allowlisted (a natural operational step for any pool that wants to support router-based trading), every user — including those the pool admin explicitly excluded — can bypass the allowlist by routing through the public router.

---

### Finding Description

**Hook argument binding in `ExtensionCalling._beforeSwap`:**

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value verbatim into the extension call: [2](#0-1) 

**What `SwapAllowlistExtension` checks:**

The extension's `beforeSwap` receives `sender` as its first argument and looks it up in `allowedSwapper[pool][sender]`. The test suite confirms this: the allowlist entry is keyed to `address(callers[0])`, the contract that directly calls `pool.swap()`: [3](#0-2) 

**The router intermediary breaks the identity:**

`MetricOmmSimpleRouter.exact*` calls `pool.swap(...)` with the router as `msg.sender`. The pool therefore passes the router's address as `sender` to the extension. The extension checks whether the **router** is allowlisted, not whether the **end user** is allowlisted.

Two broken outcomes follow:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Individually-allowlisted users **cannot** use the router; their swaps revert with `NotAllowedToSwap` even though they are on the list |
| Router **allowlisted** (pool admin adds it to support router trading) | **Any** user — including those the admin explicitly excluded — can bypass the guard by routing through the public router |

The second case is the fund-impacting path. A pool admin who wants to restrict swaps to a curated set of counterparties (e.g., a private LP pool, a KYC-gated venue, or a pool with a velocity guard that relies on the allowlist as a first line of defense) must allowlist the router to support normal periphery usage. Doing so silently opens the gate to every address.

The `generate_scanned_questions.py` audit pivot explicitly flags this path: [4](#0-3) 

---

### Impact Explanation

**Admin-boundary break with direct fund-impacting consequence.** The pool admin's swap allowlist — the primary access-control boundary for who may trade against pool liquidity — is silently nullified for any user who routes through the public `MetricOmmSimpleRouter`. Unauthorized users can execute swaps at oracle prices against LP capital that the pool admin intended to protect. In pools that combine the swap allowlist with a stop-loss or velocity guard (both of which also fire on the `beforeSwap`/`afterSwap` path), the allowlist bypass also removes the intended first-line filter, increasing the surface for those guards to be stressed by adversarial actors.

Severity: **High** — the guard is completely bypassed for the most common user entry point; no privileged action is required beyond the pool admin having allowlisted the router (a necessary operational step).

---

### Likelihood Explanation

Any pool that (a) deploys `SwapAllowlistExtension` and (b) wants users to trade through the standard periphery router must allowlist the router. This is the normal operational path. The bypass is therefore reachable on every such pool without any special setup. The attacker needs only to call `MetricOmmSimpleRouter.exactInput` or `exactOutput` targeting the pool.

---

### Recommendation

The extension must check the **economic actor** (the end user), not the intermediary. Two approaches:

1. **Pass the end-user identity through `extensionData`**: The router encodes `msg.sender` (the real user) into `extensionData`; the extension decodes and checks it. This requires the router to be trusted to populate the field honestly — acceptable because the router is a known, audited contract.

2. **Check `sender` only when `sender` is not a known router; otherwise decode the real caller from `extensionData`**: The extension maintains a registry of trusted routers and falls back to the encoded user identity for those callers.

Either way, the invariant that must hold is: **the address checked against the allowlist must be the address that economically initiates and benefits from the swap**, not the intermediary contract that relays it.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  allowedSwapper[pool][alice] = true   // alice is allowed
  allowedSwapper[pool][router] = true  // router allowlisted so normal users can trade

Attack:
  bob (not on allowlist) calls:
    router.exactInput(pool, zeroForOne=true, amountIn=X, ...)

Execution trace:
  router.exactInput(...)
    → pool.swap(recipient=bob, ..., msg.sender=router)
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes at oracle price
      → bob receives output tokens

Result:
  bob, who is NOT on the allowlist, successfully swaps against the pool.
  The pool admin's access control is completely bypassed.
  LP funds are traded against by an unauthorized counterparty.
