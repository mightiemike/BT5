### Title
Uninitialized `lastMidPriceX64` in `PriceVelocityGuardExtension` allows first swap to bypass velocity guard — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` silently skips the entire velocity check when `lastMidPriceX64 == 0` (its default, uninitialized value). Any swap that executes before the pool admin calls `setLastMidPrice` bypasses the guard unconditionally, regardless of how large the oracle price move is. This is the direct analog of the RocketJoe `lastRewardTimestamp = 0` bug: an uninitialized accumulator causes a protection invariant to fail open on the first interaction.

---

### Finding Description

`PriceVelocityGuardExtension` has no `initialize` hook called by the factory. Its per-pool state starts at the zero-value default:

```
priceVelocityState[pool] = PriceVelocityState { lastMidPriceX64: 0, lastUpdateBlock: 0, maxChangePerBlockE18: 0 }
```

The `beforeSwap` hook reads `prevMid` and **unconditionally writes the new mid-price before checking it**:

```solidity
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // state updated first
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                     // guard only fires when prevMid is non-zero
    ...
    if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
}
``` [1](#0-0) 

When `prevMid == 0`, the `if` block is never entered. The swap executes at whatever oracle price the pool currently quotes, with no velocity enforcement. After this first swap, `lastMidPriceX64` is set to the current mid-price and subsequent swaps are guarded normally.

The only way to pre-initialize the guard is the admin-only `setLastMidPrice`:

```solidity
function setLastMidPrice(address pool_, uint128 newLastMidPriceX64) external onlyPoolAdmin(pool_) {
    PriceVelocityState storage s = priceVelocityState[pool_];
    s.lastMidPriceX64 = newLastMidPriceX64;
    s.lastUpdateBlock = uint64(block.number);
    ...
}
``` [2](#0-1) 

There is no factory-called `initialize` function (contrast `OracleValueStopLossExtension`, which has one). The pool can go live with `PriceVelocityGuardExtension` configured but `lastMidPriceX64 == 0`, and there is no on-chain enforcement preventing a swap from landing before the admin's initialization transaction. [3](#0-2) 

The pool's `swap` function passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it to the extension. The extension's `beforeSwap` uses `msg.sender` (the pool) as the config key, so the guard is correctly scoped per pool — but the bypass is structural, not a key-mismatch issue. [4](#0-3) 

---

### Impact Explanation

The velocity guard's stated purpose is to cap how fast the oracle mid-price can move between blocks, protecting LPs from rapid price moves (whether from oracle latency, manipulation, or market events). When the guard is bypassed on the first swap, a trader can execute at an oracle price that has moved arbitrarily far from the last known price — exactly the scenario the guard was deployed to prevent. LPs who provided liquidity expecting the velocity cap to be active suffer direct principal loss through adverse selection at the unchecked price.

---

### Likelihood Explanation

- Any pool that deploys with `PriceVelocityGuardExtension` but whose admin does not atomically call `setLastMidPrice` in the same transaction as pool creation is vulnerable.
- The window is open from pool creation until the admin's initialization transaction is mined. On a busy chain, this can be multiple blocks.
- An attacker monitoring the mempool for new pool deployments can front-run or simply race the admin's `setLastMidPrice` call with a swap.
- No special privilege is required; any address can call `swap` on the pool.
- The oracle price need not be manipulated — a legitimate large price move (e.g., a volatile asset) is sufficient to make the bypass profitable.

---

### Recommendation

Add a factory-called `initialize` function to `PriceVelocityGuardExtension` (mirroring `OracleValueStopLossExtension`) that accepts the initial mid-price and sets both `lastMidPriceX64` and `lastUpdateBlock` atomically at pool creation:

```solidity
function initialize(address pool, bytes calldata data) external onlyFactory returns (bytes4) {
    uint128 initialMidPriceX64 = abi.decode(data, (uint128));
    require(initialMidPriceX64 != 0, "zero initial price");
    PriceVelocityState storage s = priceVelocityState[pool];
    s.lastMidPriceX64 = initialMidPriceX64;
    s.lastUpdateBlock = uint64(block.number);
    return IMetricOmmExtensions.initialize.selector;
}
```

Alternatively, add a guard in `beforeSwap` that reverts when `prevMid == 0` and `maxChangePerBlockE18 != 0`, forcing the admin to initialize before any swap can proceed.

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured with `maxChangePerBlockE18 = 1e16` (1% per block cap).
2. Admin calls `setMaxChangePerBlock` but has **not yet** called `setLastMidPrice`. `lastMidPriceX64 == 0`.
3. Oracle mid-price is currently `P = 2^64` (1:1 ratio). In the next block the oracle moves to `P' = 2 * 2^64` (100% move — far beyond the 1% cap).
4. Attacker calls `pool.swap(...)` before the admin's `setLastMidPrice` transaction is mined.
5. In `beforeSwap`: `prevMid = 0` → `if (prevMid != 0)` is false → velocity check skipped → swap executes at `P'`.
6. `lastMidPriceX64` is now set to `P'`. All subsequent swaps are guarded from `P'` onward, but the first swap at the 100% moved price has already extracted value from LPs.
7. Had `setLastMidPrice` been called with `P` before step 4, the swap at `P'` would have reverted with `PriceVelocityExceeded` because `changeE18^2 = (1e18)^2 = 1e36 >> maxChange^2 * (1 + blockDiff)`.
