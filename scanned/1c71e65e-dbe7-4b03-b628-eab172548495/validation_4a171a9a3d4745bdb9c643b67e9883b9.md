### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Non-Allowlisted Address to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` argument (the actual caller/payer) and gates on `owner` (the position recipient) instead. Because `MetricOmmPoolLiquidityAdder` explicitly supports depositing on behalf of a different address, any non-allowlisted actor can bypass the guard by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook as:

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData)
```

so `sender` = the immediate caller of the pool (e.g. `MetricOmmPoolLiquidityAdder`) and `owner` = the position recipient supplied by that caller. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values faithfully to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then **ignores** the first parameter (`sender`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` explicitly allows `owner != msg.sender` — the payer is always `msg.sender` of the adder, while `owner` is a caller-supplied argument: [4](#0-3) 

The interface even documents this split: *"The position `owner` may differ from `msg.sender`, but token pulls in callback are always sourced from `msg.sender`."* [5](#0-4) 

The contract-level NatSpec for `DepositAllowlistExtension` states its purpose is to gate by **depositor** address, and the storage mapping is named `allowedDepositor`. The `sender` is the depositor; `owner` is the beneficiary. Checking `owner` instead of `sender` inverts the intended invariant. [6](#0-5) 

For contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (not `recipient`): [7](#0-6) 

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism to restrict who may add liquidity (e.g., KYC gating, protocol-only LPs, regulatory compliance). With this bug the guard is completely ineffective: any non-allowlisted address can inject arbitrary liquidity into a restricted pool, altering its bin composition and depth without authorization. This is a broken core pool functionality / admin-boundary break with direct fund-flow consequences: unauthorized token deposits reach the pool and unauthorized LP positions are minted.

---

### Likelihood Explanation

Exploitation requires only knowing one allowlisted address (readable from on-chain `AllowedToDepositSet` events) and calling `addLiquidityExactShares` with that address as `owner`. No special privileges, flash loans, or complex setup are needed. The `MetricOmmPoolLiquidityAdder` is the standard periphery entry point, so the path is always available.

---

### Recommendation

Check `sender` (the actual depositor/payer) instead of `owner` (the position recipient):

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
``` [3](#0-2) 

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as `extension1` and `beforeAddLiquidity` order set to call it.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. `bob` is **not** allowlisted.
3. `bob` calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(
       pool,
       alice,   // owner — allowlisted
       salt,
       deltas,
       maxAmt0,
       maxAmt1,
       ""
   );
   ```
4. `MetricOmmPoolLiquidityAdder` calls `pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")`.
5. Pool calls `_beforeAddLiquidity(sender = address(liquidityAdder), owner = alice, ...)`.
6. Extension evaluates `allowedDepositor[pool][alice]` → `true` → **no revert**.
7. `bob`'s tokens are pulled in the callback; `alice` receives the LP shares.
8. `bob` has deposited into a pool that explicitly excluded him, with zero resistance from the allowlist guard.
