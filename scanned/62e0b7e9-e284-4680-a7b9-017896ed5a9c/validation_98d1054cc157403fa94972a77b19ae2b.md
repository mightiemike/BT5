### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the actual end-user. A pool admin who allowlists the router to enable router-mediated swaps for their users inadvertently opens the pool to **every user**, completely defeating the allowlist gate.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every extension hook: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point `msg.sender` inside `pool.swap` is the **router**, so `sender` forwarded to the extension is the router address — not the end-user. The actual end-user's identity is never visible to the extension.

Contrast this with `DepositAllowlistExtension`, which correctly gates on `owner` (the economically relevant LP identity) rather than `sender` (the caller): [5](#0-4) 

The asymmetry reveals the design flaw: for deposits the pool passes a separate `owner` field that survives router indirection; for swaps no such field exists, so the extension can only see the router.

**Bypass path (step-by-step):**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of trusted counterparties.
2. Admin calls `setAllowedToSwap(pool, alice, true)` and, wanting Alice to be able to use the router, also calls `setAllowedToSwap(pool, router, true)`.
3. Any unprivileged user `Bob` calls `router.exactInputSingle({pool: pool, ...})`.
4. The router calls `pool.swap(...)` — `sender` seen by the extension is the router address.
5. `allowedSwapper[pool][router] == true` → the check passes → Bob's swap executes.

The allowlist is fully bypassed without any privileged action by Bob.

**Broken invariant (no-bypass path):**

If the admin does **not** allowlist the router, Alice (who is individually allowlisted) cannot use the router at all — her router-mediated swap reverts because `allowedSwapper[pool][router] == false`. She is forced to call the pool directly, breaking the expected UX.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to gate access to specific counterparties (e.g., a private LP pool, a KYC-gated venue, or a pool with a stop-loss guard that relies on the allowlist as a first line of defense) can be fully bypassed by any user routing through `MetricOmmSimpleRouter`. Unauthorized users can swap against LP positions at oracle-derived prices, extracting value from LPs who believed their pool was access-controlled. This is a direct loss of LP principal and a broken core pool invariant.

---

### Likelihood Explanation

The likelihood is **medium**. Allowlisting the router is the natural, expected action for any pool admin who wants their allowlisted users to be able to use the standard periphery router. The admin has no reason to suspect this opens the pool to everyone. The `DepositAllowlistExtension`'s use of `owner` (not `sender`) further obscures the asymmetry, making the mistake easy to miss.

---

### Recommendation

1. **Document the limitation clearly**: `SwapAllowlistExtension` only gates direct `pool.swap()` callers; it cannot gate end-users of the router. Do not allowlist the router address.
2. **Structural fix**: Add an optional `trustedForwarder` mapping. If `sender` is a trusted forwarder, decode the real caller from `extensionData` (similar to ERC-2771). The router would need to encode `msg.sender` into `extensionData` before calling the pool.
3. **Alternative**: Expose a `recipient`-based allowlist variant for router flows, since `recipient` is the address that receives output tokens and is set by the end-user.

---

### Proof of Concept

```solidity
// Pool admin sets up an allowlist pool
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// ... deploy pool with ext as beforeSwap extension ...

// Admin allowlists Alice and the router (intending Alice to use the router)
ext.setAllowedToSwap(pool, alice, true);
ext.setAllowedToSwap(pool, address(router), true);

// Bob (not allowlisted) bypasses the gate via the router
vm.prank(bob);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        recipient: bob,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    })
);
// Bob's swap succeeds — allowlist bypassed
```

The `sender` seen by `SwapAllowlistExtension` is `address(router)`, which is allowlisted, so the check passes for Bob despite Bob never being granted access.
