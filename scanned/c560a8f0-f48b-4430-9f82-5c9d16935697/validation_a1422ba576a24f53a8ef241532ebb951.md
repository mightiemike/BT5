### Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User — Any Unprivileged User Bypasses the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the immediate `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the actual end user. If the pool admin allowlists the router (required for any router-based swap to succeed), every unprivileged address on the network can bypass the swap allowlist entirely by calling the router.

### Finding Description

The call chain is:

1. **User** calls `MetricOmmSimpleRouter.exactInputSingle(params)` (public, no access control).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — so `msg.sender` seen by the pool is the **router**.
3. Pool calls `_beforeSwap(msg.sender /*= router*/, recipient, ...)` and dispatches to the configured extension.
4. `SwapAllowlistExtension.beforeSwap(sender /*= router*/, ...)` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

If the pool admin allowlists the router to permit legitimate users to swap through it, the mapping entry `allowedSwapper[pool][router] = true` is set. From that point, **any caller** of `MetricOmmSimpleRouter.exactInputSingle` passes the guard, because the extension only sees the router's address — the actual end user's identity is never inspected.

The actual user address is not forwarded to the extension at all. The `beforeSwap` interface only exposes `sender` (immediate pool caller) and `recipient` (output receiver); neither is guaranteed to be the economic actor the pool admin intended to gate.

The `DepositAllowlistExtension` has the symmetric problem on the deposit side: it checks `owner` (the position recipient) and silently discards `sender` (the token payer), but the pool's own NatDoc explicitly documents that `owner` is the intended gated identity for deposits, so that side is by design. The swap side has no such documentation and the `generate_scanned_questions.py` audit target explicitly flags it as a bypass risk.

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., KYC-gated market makers, institutional participants) is fully open to any address the moment the pool admin allowlists the router. The allowlist guard is rendered a no-op for all router-mediated swaps. Unauthorized users can drain the pool's liquidity at oracle-anchored prices, extract fees, or manipulate the pool's bin cursor — all consequences that the allowlist was deployed to prevent.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard user-facing entry point for swaps. Any pool that wants to support normal UX must allowlist the router. The bypass is therefore reachable in every realistic deployment of `SwapAllowlistExtension` that also permits router usage. No special privileges, flash loans, or unusual token behavior are required — a single call to `exactInputSingle` suffices.

### Recommendation

The extension must gate the **economic actor**, not the immediate caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between the router and the extension.
2. **Check `recipient` as a proxy**: For exact-input swaps the recipient is typically the user. This is imperfect (recipient can differ from payer) but closer to the intended gate.
3. **Structural fix**: Add a `realSender` field to the `beforeSwap` hook that the pool populates from a trusted transient-storage slot set by the router, similar to how the liquidity adder stores payer context.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists only `trustedUser` via setAllowedToSwap(pool, trustedUser, true)
  - Pool admin also allowlists the router: setAllowedToSwap(pool, router, true)
    (required so trustedUser can swap through the router)

Attack:
  - `attacker` (not in allowlist) calls:
      MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: attacker, ...})
  - Router calls pool.swap(attacker, ...)  →  sender = router
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  →  passes
  - Attacker's swap executes; NotAllowedToSwap is never triggered

Result:
  - attacker receives output tokens from a pool that was supposed to be restricted
  - The allowlist guard is completely bypassed
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
