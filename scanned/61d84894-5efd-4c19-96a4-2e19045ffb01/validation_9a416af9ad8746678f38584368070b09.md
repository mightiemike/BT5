### Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any Caller to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` as seen by `MetricOmmPool.swap` — i.e., the immediate caller of the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router address to permit router-mediated swaps, every unpermissioned user can bypass the allowlist by routing through the router, because the extension sees only the router's address and not the actual trader's identity.

### Finding Description

**Call path:**

```
User (Bob, not allowlisted)
  → MetricOmmSimpleRouter.exactInputSingle(pool, ...)
      → MetricOmmPool.swap(recipient, ...)   [msg.sender = router]
          → ExtensionCalling._beforeSwap(sender = router, ...)
              → SwapAllowlistExtension.beforeSwap(sender = router, ...)
                  → allowedSwapper[pool][router]  ← checks ROUTER, not Bob
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to the extension's `beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When the router is the caller, `sender = router`. The check becomes `allowedSwapper[pool][router]`, which is entirely independent of the actual end user.

The router calls `pool.swap` directly without forwarding the original `msg.sender`: [4](#0-3) 

There is no mechanism in the router to pass the original caller's identity to the pool or extension.

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router address (the only way to permit router-mediated swaps for any user) inadvertently opens the pool to **all** callers. Any non-allowlisted user can call `router.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and the extension will pass because it sees `allowedSwapper[pool][router] = true`. The allowlist is completely neutralized for the router path. Non-allowlisted users can trade on pools intended to be restricted (e.g., KYC-gated, institutional-only, or regulatory-restricted pools), extracting value and violating the pool's curation invariant. This is a direct bypass of a configured access-control guard with fund-impacting consequences.

The alternative — not allowlisting the router — means even legitimately allowlisted users cannot use the router, breaking core swap functionality for the supported periphery path.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported swap interface. Any pool admin who configures `SwapAllowlistExtension` and wants their allowlisted users to be able to use the router must allowlist the router address. This is the natural and expected configuration. Once the router is allowlisted, the bypass is trivially reachable by any unpermissioned user with no special privileges, no malicious setup, and no non-standard tokens required.

### Recommendation

The `sender` identity passed to `beforeSwap` must reflect the economic actor (the end user), not the intermediate contract. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: propagate the original `msg.sender` through the `callbackData` or a dedicated field so the pool or extension can recover it. Alternatively, pass the original caller as the `recipient` and use a separate field for the actual output recipient.

2. **In `SwapAllowlistExtension.beforeSwap`**: if the `sender` is a known trusted router, decode the actual end-user identity from `extensionData` and check that identity against the allowlist instead of the router address.

The simplest correct fix is for the router to pass the original `msg.sender` as part of `extensionData` and for the extension to decode and verify it, similar to how Uniswap v4 hooks receive the originating caller.

### Proof of Concept

```
Setup:
  pool admin deploys pool with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, router, true)   // to allow router users
  pool admin calls setAllowedToSwap(pool, alice, true)    // alice is the intended allowlisted user
  bob is NOT allowlisted

Attack:
  bob calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(...)  [msg.sender = router]
    → extension checks allowedSwapper[pool][router] == true  → PASSES
    → bob's swap executes on the restricted pool

Result:
  bob (non-allowlisted) successfully swaps on a pool that was supposed to restrict access
  the allowlist guard is silently bypassed for all router-mediated swaps
```
