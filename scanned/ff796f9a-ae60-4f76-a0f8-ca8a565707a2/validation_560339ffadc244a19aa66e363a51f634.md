### Title
`SwapAllowlistExtension` Per-User Gate Bypassed via Router: Any User Can Swap Through Allowlisted Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` as seen by the pool — which is the **router address**, not the originating user. When a pool admin allowlists the router (a natural and expected configuration), any unprivileged user can bypass the per-user swap gate by routing through `MetricOmmSimpleRouter.exactInput` or `exactInputSingle`.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension:**

In `MetricOmmPool.swap()`, the `_beforeSwap` hook is called with `msg.sender` as the first argument: [1](#0-0) 

**Step 2 — Router is `msg.sender` when it calls `pool.swap()`:**

In `MetricOmmSimpleRouter.exactInput`, the router calls `pool.swap()` directly in a loop over all hops. The pool's `msg.sender` is therefore always the router contract, not the original caller: [2](#0-1) 

**Step 3 — Extension checks the router address, not the original user:**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (the pool) as the pool key and `sender` (the router) as the swapper to check: [3](#0-2) 

The check resolves to `allowedSwapper[pool][router]`. If the router is allowlisted, the gate passes unconditionally for every hop, regardless of who called the router.

---

### Impact Explanation

A pool admin who deploys a pool with `SwapAllowlistExtension` intending to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties, institutional traders) will naturally also allowlist the official `MetricOmmSimpleRouter` so that those users can trade via the router. Once the router is allowlisted, the per-user restriction is completely nullified: **any address** can call `exactInput` or `exactInputSingle` on the router and the extension will pass on every hop, because the pool only ever sees the router as the swapper. The extension's stated purpose — "Gates `swap` by swapper address, per pool" — is broken for all router-mediated paths. [4](#0-3) 

---

### Likelihood Explanation

The scenario is not contrived. Any pool that:
1. Uses `SwapAllowlistExtension` to restrict swaps to specific users, **and**
2. Allowlists the router so those users can trade via the standard router interface

...is fully exposed. This is the expected and documented usage pattern. The admin has no on-chain signal that allowlisting the router collapses the per-user gate.

---

### Recommendation

`SwapAllowlistExtension` must propagate the **original caller identity** through the router, not the router's own address. Two approaches:

1. **Pass original caller in `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and checks it. This requires a trusted encoding convention.
2. **Dedicated router-aware check**: Add a separate allowlist entry for "router + original payer" pairs, or require the router to pass the payer address as a verified field in `extensionData` that the extension validates against a known router registry.

The simplest safe fix is for the extension to treat the router as a transparent forwarder and require the actual payer identity to be passed and verified.
