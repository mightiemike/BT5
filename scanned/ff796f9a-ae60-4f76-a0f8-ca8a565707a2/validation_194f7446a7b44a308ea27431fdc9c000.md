### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any Caller to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` of the pool call is the **router contract**, not the end user. A pool admin who allowlists the router to enable router-mediated swaps for their curated users simultaneously opens the gate for every other user, because the extension sees only the router's address and approves it unconditionally.

---

### Finding Description

**Actor binding in the extension:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

**What the pool passes as `sender`:**

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller of the pool) as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router itself `msg.sender` of the pool call. The actual end user (`msg.sender` of the router call) is stored only in transient storage for the payment callback — it is never forwarded to the pool as the swap initiator:

```solidity
// MetricOmmSimpleRouter.sol line 71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    ...
);
``` [3](#0-2) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**The broken invariant:**

When the extension's `beforeSwap` fires, `sender = address(router)`. The check `allowedSwapper[pool][router]` is evaluated. If the pool admin has allowlisted the router (the only way to permit router-mediated swaps for their curated users), every caller of the router passes the check regardless of their own address.

The `onlyPool` modifier in `BaseMetricExtension` only verifies that the extension is called by a registered pool — it does not recover or validate the original end-user identity: [5](#0-4) 

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise vetted addresses is fully bypassed for any user who routes through `MetricOmmSimpleRouter`. Non-allowlisted users can execute swaps, extract liquidity at oracle prices, and interact with a pool that was designed to exclude them. This is a **direct loss of curation policy** with fund-impacting consequences: LP providers deposited into a curated pool under the assumption that only vetted counterparties would trade against them; the bypass exposes them to unrestricted counterparty flow.

---

### Likelihood Explanation

The scenario is highly realistic:

1. A pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists specific user addresses.
2. To support the standard periphery UX (router-mediated swaps), the admin must also allowlist the router address — there is no other mechanism to pass the check for router users.
3. Once the router is allowlisted, the bypass is open to every caller of the router with no further preconditions.
4. Any unprivileged user can trigger this by calling `exactInputSingle` or any other router entry point.

---

### Recommendation

The extension must check the **end user's identity**, not the intermediate contract's address. Two approaches:

1. **Pass the original initiator through the pool.** Add an `initiator` field to the `beforeSwap` hook arguments (distinct from `sender`) that the pool populates from a trusted source (e.g., transient storage set at the top of `swap`). The extension then gates on `initiator`.

2. **Check `recipient` instead of `sender` for router flows, or require direct pool calls only.** This is weaker but simpler: document that the allowlist only works for direct pool calls and revert if `sender` is not in the allowlist (which already happens for direct calls). The problem is that this breaks router UX entirely for curated pools.

The cleanest fix is approach 1: the pool should propagate the original transaction initiator (or require the router to forward it explicitly) so the extension can gate on the economically relevant actor.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
  - Pool admin calls setAllowedToSwap(pool, router, true)      // enable router UX for alice

Attack (bob is NOT allowlisted):
  - bob calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...) with msg.sender = router
  - pool calls extension.beforeSwap(sender=router, ...)
  - extension checks allowedSwapper[pool][router] == true  → passes
  - bob's swap executes on the curated pool

Result:
  - bob, a non-allowlisted user, successfully swaps on a pool
    that was configured to exclude him.
  - The SwapAllowlistExtension's curation policy is nullified
    for all router-mediated swaps.
