### Title
SwapAllowlistExtension gates on the router address instead of the originating user, enabling full allowlist bypass through the supported periphery path — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the value `msg.sender` of the pool passes through the hook. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`. A pool admin who adds the router to the allowlist (the natural remediation when allowlisted users report they cannot trade through the periphery) inadvertently opens the pool to every user on-chain, defeating the entire curation mechanism.

---

### Finding Description

**Hook argument binding in the pool**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← immediate caller, not the originating EOA
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim as the first positional argument to every configured extension:

```solidity
// ExtensionCalling.sol line 162-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, amountSpecified, ...)
)
```

**What the allowlist actually checks**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (the pool) as the namespace key and the first positional argument (`sender`) as the identity being gated:

```solidity
// SwapAllowlistExtension.sol line 37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

At this point `msg.sender` inside the pool is the **router**, so the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The originating EOA is stored only in the router's transient callback context (`_setNextCallbackContext`) and is never surfaced to the pool or the extension.

The same mismatch applies to `exactInput` (multi-hop) and `exactOutput` / `exactOutputSingle`.

**The impossible admin choice**

| Admin action | Effect |
|---|---|
| Do not allowlist the router | Allowlisted users cannot trade through the supported periphery |
| Allowlist the router | Every user on-chain can bypass individual gating by routing through the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users. The `extensionData` forwarded by the router is ignored by `SwapAllowlistExtension`, so there is no in-band channel to recover the originating user's identity.

---

### Impact Explanation

A pool deployed with `SwapAllowlistExtension` to enforce curated access loses that access control entirely once the router is added to the allowlist. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and execute swaps against the pool. This is a direct, fund-impacting bypass: the pool's LP positions are exposed to unrestricted swap flow that the pool admin explicitly intended to prevent, and the LP bears the resulting adverse-selection and inventory risk.

---

### Likelihood Explanation

The likelihood is medium-high. `MetricOmmSimpleRouter` is the canonical, documented swap entry point for end users. When allowlisted users discover they cannot trade through the router (because the router is not individually allowlisted), the natural support response is to add the router to the allowlist. Nothing in the admin interface or documentation warns that doing so opens the pool to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — any EOA can call the router.

---

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the originating user, not the immediate pool caller. Two complementary fixes:

1. **Pool-level**: Have `MetricOmmPool.swap` accept an explicit `originator` parameter (defaulting to `msg.sender` for direct calls) and pass that to `_beforeSwap` instead of `msg.sender`. The router would supply `msg.sender` (the EOA) as the originator.

2. **Extension-level (short-term)**: `SwapAllowlistExtension` can decode the originating address from `extensionData` when `sender` is a known router, falling back to `sender` for direct calls. The router would encode `msg.sender` into `extensionData` before forwarding.

Without one of these changes, `SwapAllowlistExtension` cannot enforce per-user swap gating on any pool that supports the router as an entry point.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwapOrder`.
2. Admin allowlists Alice: `swapExt.setAllowedToSwap(pool, alice, true)`.
3. Alice calls `router.exactInputSingle(...)`. The router calls `pool.swap(...)` with `msg.sender = router`. The extension evaluates `allowedSwapper[pool][router]` → `false` → revert. Alice cannot trade.
4. Admin adds the router: `swapExt.setAllowedToSwap(pool, router, true)` to unblock Alice.
5. Charlie (never allowlisted) calls `router.exactInputSingle(...)`. The extension evaluates `allowedSwapper[pool][router]` → `true` → passes. Charlie's swap executes against the curated pool.
6. The allowlist is fully bypassed for all users who route through the router.
