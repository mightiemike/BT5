### Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any Caller to Bypass the Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If a pool admin allowlists the router (the natural action to enable router-mediated swaps for permitted users), every user of the public router bypasses the per-user allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the extension is called by the pool). `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which passes `msg.sender` of the pool's `swap` call:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
         packedSlot0Initial, bidPriceX64, askPriceX64, extensionData))
);
```

And `sender` is set to `msg.sender` in `MetricOmmPool.swap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- this becomes `sender` in the extension
    recipient, ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the router. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Attack path:**

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` to restrict swaps to KYC'd users.
2. Admin allowlists specific users: `allowedSwapper[pool][alice] = true`.
3. Admin also allowlists the router: `allowedSwapper[pool][router] = true` — the natural action to let alice use the router.
4. Attacker (charlie, not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Extension sees `sender = router`, which is allowlisted → check passes.
6. Charlie swaps on the curated pool without restriction.

The router is a public, permissionless contract with no access control on any of its entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`).

---

### Impact Explanation

Any user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter`. The allowlist is the sole mechanism for curated pools to restrict who may trade. Once bypassed, unauthorized users can execute swaps against LP liquidity, extracting value from pools that were intended to be restricted (e.g., compliance-gated, institutional-only, or partner-only pools). This is a direct loss of LP principal and a broken core pool functionality invariant.

---

### Likelihood Explanation

The trigger requires the pool admin to have allowlisted the router address. This is the expected operational action for any admin who wants their allowlisted users to be able to use the standard periphery router — a common and documented use case. The `MetricOmmSimpleRouter` is the canonical swap entry point for end users. Any curated pool that enables router access is immediately vulnerable to this bypass by any public caller.

---

### Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor — the end user — not the intermediary contract. Two approaches:

1. **Pass the original initiator through the router.** Have the router forward the end user's address as part of `extensionData`, and have the extension decode and check it. This requires a coordinated change to the router and extension.

2. **Check `sender` only when it is not a known router; otherwise check a user field from `extensionData`.** This is fragile and requires maintaining a router registry.

3. **Simplest correct fix:** Do not allowlist the router address. Instead, require that all allowlisted users call the pool directly. Document that router-mediated swaps are incompatible with the per-user allowlist. Alternatively, deploy a router that forwards the original `msg.sender` as the `sender` argument (e.g., via a trusted forwarder pattern), so the extension always sees the end user.

### Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);

ext.setAllowedToSwap(pool, alice, true);
ext.setAllowedToSwap(pool, address(router), true); // to let alice use the router

// Attack: charlie (not allowlisted) bypasses the allowlist via the router
vm.prank(charlie);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({

    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
