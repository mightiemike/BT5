### Title
SwapAllowlistExtension Gates on Router Address Instead of End-User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` (the direct caller of `pool.swap()`). When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's. If the pool admin allowlists the router to enable legitimate users to swap through it, every unpermissioned user can bypass the allowlist by routing through the same router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct) and `sender` is the argument forwarded by `ExtensionCalling._beforeSwap`:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
     packedSlot0Initial, bidPriceX64, askPriceX64, extensionData))
``` [2](#0-1) 

The pool sets `sender = msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` calls `pool.swap()`, `sender` becomes the router's address, not the originating user. The `FullMetricExtensionTest` confirms this: it allowlists `callers[0]` (the intermediary `TestCaller` contract), not `users[0]` (the end user), for the swap to succeed:

```solidity
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
``` [3](#0-2) 

The structural analog to FTN-3: the guard check (`allowedSwapper[pool][sender]`) and the intended protection (blocking a specific end user) operate on **different identities** — the check passes for the router even when the end user is not allowlisted, so the guard is silently bypassed for every user who routes through an allowlisted router.

---

### Impact Explanation

A pool admin who deploys a swap allowlist intends to restrict swaps to a named set of addresses (e.g., KYC'd counterparties). To let those users access the router, the admin must allowlist the router address. Once the router is allowlisted, **any** address — including those explicitly not on the allowlist — can execute swaps on the restricted pool by calling `MetricOmmSimpleRouter.exactInputSingle` / `exactInput`. The allowlist is completely ineffective for router-mediated paths, allowing unauthorized traders to swap against pool liquidity, extract value, or front-run LP positions on a pool that was designed to be access-controlled.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary public entry point for swaps. Any pool that (a) deploys `SwapAllowlistExtension` and (b) needs to support router-based swaps for its allowlisted users must allowlist the router. Both conditions are the normal operating mode for a restricted pool with a UI. The bypass is therefore reachable by any unprivileged user with no special setup beyond knowing the router address. [4](#0-3) 

---

### Recommendation

Pass the **originating user** as `sender` rather than the direct pool caller. Two options:

1. **Preferred — router forwards the real sender**: `MetricOmmSimpleRouter` passes `msg.sender` (the end user) explicitly as the `sender` argument to `pool.swap()`, and the pool forwards it verbatim to the extension. The extension then checks the end user's allowlist entry.

2. **Alternative — extension reads `tx.origin`**: Replace `sender` with `tx.origin` inside `beforeSwap`. This is simpler but incompatible with smart-contract wallets and multicall patterns.

Option 1 is consistent with the `DepositAllowlistExtension` design, which correctly gates on `owner` (the position beneficiary) rather than `msg.sender` (the payer):

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [5](#0-4) 

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as the `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, routerAddress, true)` — necessary for any allowlisted user to use the router.
3. Admin does **not** call `setAllowedToSwap(pool, attackerAddress, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
5. The pool calls `_beforeSwap(sender=routerAddress, ...)`.
6. Extension evaluates `allowedSwapper[pool][routerAddress]` → `true` → no revert.
7. Attacker's swap executes against the restricted pool, bypassing the per-user allowlist entirely. [4](#0-3) [6](#0-5)
