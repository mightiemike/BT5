### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap` call. When users route through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the actual user. Any pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the gate to every user on the network, completely defeating the allowlist.

---

### Finding Description

**Root cause — wrong identity is checked**

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `sender` (the immediate caller of `pool.swap`) against the per-pool allowlist: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point), the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point `msg.sender` inside the pool is the **router contract**, so `sender` forwarded to the extension is the router's address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**The forced false choice**

A pool admin who wants to restrict swaps to a specific set of users faces an impossible configuration:

| Admin intent | Configuration | Actual effect |
|---|---|---|
| Allow Alice & Bob only, direct calls | `allowedSwapper[pool][Alice] = true`, `allowedSwapper[pool][Bob] = true` | Correct for direct calls; router calls revert for everyone |
| Allow Alice & Bob through the router | Must add `allowedSwapper[pool][router] = true` | Now **every** address can swap through the router |

There is no configuration that correctly restricts individual users who route through `MetricOmmSimpleRouter`.

**Contrast with DepositAllowlistExtension**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (the position owner), not `sender`: [5](#0-4) 

`MetricOmmPoolLiquidityAdder` passes the actual user as `owner` explicitly, so the deposit allowlist correctly gates the economically relevant depositor regardless of who the payer is. The swap path has no equivalent mechanism — there is no separate "actual user" field in the swap interface.

---

### Impact Explanation

Any non-allowlisted user can bypass a pool's swap allowlist by routing through the public `MetricOmmSimpleRouter`. If the pool admin allowlists the router (the only way to let their intended users trade through it), the allowlist provides zero protection. Unauthorized users gain full swap access to pools that were configured to be restricted — breaking the core access-control invariant of the extension system and enabling unauthorized extraction of value from pools that may carry favorable oracle-anchored pricing or lower fee tiers reserved for allowlisted participants.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool that uses `SwapAllowlistExtension` and also wants its allowlisted users to be able to use the router must allowlist the router address. This is a routine operational step, not an edge case. Once the router is allowlisted, the bypass is available to every address on the network with no special privileges required.

---

### Recommendation

The `sender` field passed through the hook chain must represent the **end user**, not the intermediary router. Two viable approaches:

1. **Extension-data convention**: Require the router to encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and check it when the immediate caller is a known router. This requires a protocol-level convention for the encoding.

2. **Separate originator field**: Add an `originator` parameter to `IMetricOmmExtensions.beforeSwap` (distinct from `sender`) that the pool always sets to `tx.origin` or that the router explicitly populates via a signed forwarding mechanism, and have the allowlist check `originator` instead of `sender`.

Either way, the allowlist must gate the actor who economically controls the swap, not the contract that mechanically calls `pool.swap`.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][Alice] = true   // Alice is the intended user
  allowedSwapper[pool][router] = true  // admin adds this so Alice can use the router

Attack:
  Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  → router calls pool.swap(...) with msg.sender = router
  → pool calls extension.beforeSwap(sender=router, ...)
  → extension checks allowedSwapper[pool][router] → true
  → swap proceeds for Bob, allowlist bypassed
```

The pool admin cannot remove `allowedSwapper[pool][router]` without also blocking Alice from using the router, so the bypass is permanent once the router is allowlisted.
