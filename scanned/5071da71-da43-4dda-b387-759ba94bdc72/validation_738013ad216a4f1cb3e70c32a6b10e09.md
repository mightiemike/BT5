### Title
Pool Admin Can Progressively Reduce Timelock to Zero Then Immediately Disable Stop-Loss Protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

The `OracleValueStopLossExtension` uses a timelock to give LPs a reaction window before parameter changes take effect. However, the timelock value itself is mutable via the same timelocked proposal flow, and there is no minimum timelock floor. A pool admin can propose `timelock=0`, wait the current timelock period, execute it, and then immediately propose and execute `drawdown=0` in the same block — completely disabling stop-loss protection with zero LP warning.

---

### Finding Description

The attack proceeds in two phases:

**Phase 1 — Reduce timelock to zero:**

`proposeOracleStopLossTimelock` snapshots the *current* timelock at proposal time:

```solidity
uint32 executeAfter = _afterTimelock(pool_);  // block.timestamp + current_timelock
sched.pendingTimelock = newTimelock;           // = 0
sched.pendingTimelockExecuteAfter = executeAfter;
``` [1](#0-0) 

After waiting `current_timelock` seconds, `executeOracleStopLossTimelock` passes `_requireElapsed` and writes `oracleStopLossConfig[pool_].timelock = 0`. [2](#0-1) 

There is no minimum timelock validation anywhere — `newTimelock=0` is accepted unconditionally.

**Phase 2 — Immediately disable drawdown:**

With `timelock=0`, `_afterTimelock` returns exactly `block.timestamp`:

```solidity
return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
// = block.timestamp + 0 = block.timestamp
``` [3](#0-2) 

`proposeOracleStopLossDrawdown(pool, 0)` passes `_validateDrawdown` (only rejects `> E6`, not `== 0`) and sets `pendingDrawdownExecuteAfter = block.timestamp`. [4](#0-3) 

`executeOracleStopLossDrawdown` is called in the same block. `_requireElapsed(block.timestamp)` checks `block.timestamp < block.timestamp` — false — so it passes. `drawdownE6` is set to `0`. [5](#0-4) 

**Phase 3 — Stop-loss silently disabled:**

Every subsequent swap hits the early-return guard:

```solidity
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;   // ← entire check skipped
``` [6](#0-5) 

All watermark breach checks are bypassed; value-draining swaps that would have reverted with `OracleStopLossTriggered` now execute freely.

---

### Impact Explanation

The stop-loss extension is the sole on-chain mechanism preventing per-bin value-per-share from falling below the configured floor. Disabling it allows an adversary (or a malicious/compromised pool admin) to drain LP value through repeated swaps at unfavorable oracle prices without any revert. This is a direct LP principal loss path and constitutes an admin-boundary break: the pool admin bypasses the LP protection window that the timelock is explicitly designed to enforce.

---

### Likelihood Explanation

Requires the pool admin to act maliciously or be compromised. However, the attack is fully permissionless once the admin key is controlled — no factory owner, oracle admin, or external dependency is needed. The two-phase sequence is straightforward and costs only one timelock wait period. Likelihood is **Medium** given the trusted-but-not-immutable admin role.

---

### Recommendation

Enforce a non-zero minimum timelock floor and prevent the timelock from being reduced below its current value without a separate governance override. Concretely:

1. Add a `MIN_TIMELOCK` constant (e.g., `1 days`) and reject proposals below it in `proposeOracleStopLossTimelock`.
2. Alternatively, require that a new timelock proposal must be `>= current timelock` (one-way ratchet), so the protection window can only grow, never shrink.
3. Reject `drawdown=0` in `_validateDrawdown` (treat zero as "no protection" and require an explicit factory-level opt-out instead).
