### Title
`SwapAllowlistExtension` checks the router address instead of the actual user, allowing any user to bypass the per-user swap allowlist — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router address**, not the actual user. If the pool admin allowlists the router (to permit router-based swaps), every user — including those not individually allowlisted — can bypass the per-user gate by calling any router entry point.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged as the first argument to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks whether that `sender` is allowlisted: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly: [4](#0-3) 

At that point `msg.sender` inside the pool is the **router contract**, so `sender` delivered to the extension is the router address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The same wrong-actor binding applies to `exactOutputSingle`, `exactInput`, and `exactOutput` — every router entry point calls `pool.swap()` directly, making the router the `sender` the extension sees.

The `DepositAllowlistExtension` does **not** share this flaw because it checks the `owner` parameter (the position owner explicitly supplied to `addLiquidity()`), which the `MetricOmmPoolLiquidityAdder` correctly sets to `msg.sender` or an explicit owner: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` intends to restrict trading to a specific set of addresses. To also support the standard periphery router, the admin must allowlist the router address. Once the router is allowlisted, **any** address — including those explicitly excluded from the per-user allowlist — can execute a live swap with real token transfers by calling any `MetricOmmSimpleRouter` entry point. The allowlist is completely defeated. Users who should be blocked receive pool output tokens and drain LP-owned liquidity at oracle prices, constituting a direct loss of LP principal and a broken core pool invariant (curated access control).

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the protocol's standard and only supported swap periphery. Any pool admin who wants end-users to interact via the router — the normal operational assumption — must allowlist it. The bypass requires no special privilege, no malicious setup, and no non-standard token behavior: any EOA calls the public router function. The combination of a natural admin action (allowlisting the router) and a public router call is the expected usage pattern, making exploitation trivially reachable.

---

### Recommendation

The `sender` parameter passed to `beforeSwap` represents the immediate caller of `pool.swap()`, which is the router when periphery is used. The extension must gate on the **economic actor** — the address that initiated the trade and will pay for it. Two complementary fixes:

1. **In the router**: pass the original `msg.sender` (the user) as the `recipient`-equivalent "initiator" through `extensionData`, and document a convention for allowlist extensions to read it. This is the cleanest approach because it does not require changing the core interface.

2. **In `SwapAllowlistExtension`**: check `sender` only when `sender` is not a known router; otherwise decode the real initiator from `extensionData`. This is fragile and not recommended as a standalone fix.

3. **Preferred**: add an `initiator` field to the `beforeSwap` signature (a core interface change) so the pool can pass both `msg.sender` (the immediate caller) and the original user address separately, letting extensions choose which actor to gate.

Until fixed, pool admins should **not** allowlist the router address on pools that use `SwapAllowlistExtension` for per-user access control, and should instead require users to call `pool.swap()` directly.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured on beforeSwap.
  - Pool admin calls swapExtension.setAllowedToSwap(pool, router, true)
    (to allow router-based swaps — the natural operational step).
  - Pool admin does NOT call setAllowedToSwap(pool, alice, true)
    (alice is intentionally excluded from the curated pool).

Attack:
  1. alice calls router.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(recipient, ...) — msg.sender in pool = router.
  3. Pool calls _beforeSwap(sender=router, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  5. Swap executes. Alice receives output tokens.

Expected: revert NotAllowedToSwap (alice is not allowlisted).
Actual:   swap succeeds because the router is allowlisted and the
          extension never sees alice's address.
