### Title
SwapAllowlistExtension gates the router address instead of the end-user, allowing any unprivileged swapper to bypass a curated pool's allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which equals the pool's `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address. A pool admin who allowlists the router to enable standard UX inadvertently grants every unprivileged user the ability to swap on a pool that was intended to be curated.

---

### Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

That value is forwarded to the extension as `sender`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

When a user calls the pool directly, `sender` = user — the check is correct. When the same user routes through `MetricOmmSimpleRouter`, `sender` = router contract address. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

A pool admin who wants allowlisted users to access the standard router UX must add the router to the allowlist. The moment the router address is allowlisted, the check `allowedSwapper[pool][router]` returns `true` for every caller — allowlisted or not — who routes through it. The per-user gate is silently voided.

The `DepositAllowlistExtension` has the inverse design: it ignores `sender` entirely and checks `owner`, which is the position beneficiary. That choice is consistent for deposits (the economically relevant actor is the owner). For swaps there is no owner/sender split — the swapper is both the economic actor and the caller — so checking `sender` should be correct, but the router indirection breaks the binding. [3](#0-2) 

---

### Impact Explanation

Any user not on the allowlist can swap on a curated pool by routing through `MetricOmmSimpleRouter` once the router is allowlisted. The pool's curation policy — which may exist to restrict access to institutional counterparties, comply with regulatory requirements, or protect LP capital from adversarial flow — is completely bypassed. Unrestricted swappers can extract value from the pool's oracle-priced bins, causing direct LP principal loss and breaking the core invariant that only approved actors interact with the pool.

---

### Likelihood Explanation

The trigger is a routine, semi-trusted admin action: allowlisting the router so that approved users can access the standard periphery UX. This is the expected operational path for any production pool that uses the allowlist extension alongside the router. The bypass is then available to every unprivileged user with no further preconditions, no special tokens, and no privileged access.

---

### Recommendation

Pass the original end-user's address through the call chain rather than the direct pool caller. One approach: have the router encode the originating user in `extensionData` and have the extension decode and verify it. A simpler approach: the pool could accept an explicit `swapper` parameter distinct from `msg.sender`, or the extension could require that `sender` is never a known router address and instead read the user from a trusted forwarding field. At minimum, document that allowlisting the router disables per-user gating, and provide a separate router-aware extension that re-derives the true user identity.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only alice is intended to swap.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — to let alice use the router UX.
4. Charlie (not allowlisted) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The pool's `swap` fires `_beforeSwap(msg.sender=router, ...)`.
6. The extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Charlie's swap executes successfully despite not being on the allowlist.
8. Charlie extracts value from the pool's oracle-priced bins; LP principal is reduced.

The check that should have blocked Charlie — `allowedSwapper[pool][charlie]` — is never evaluated because the router's address is substituted for Charlie's at the point the pool records `msg.sender`. [2](#0-1) [4](#0-3)
