### Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing any user to bypass per-user swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the end user. If the pool admin allowlists the router to support router-mediated swaps for their curated users, the check degenerates to "is the router allowlisted?" — granting every user on the public router access to a pool that was supposed to be restricted to specific addresses.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first argument to every configured extension: [2](#0-1) 

**Step 2 — When the router calls the pool, `msg.sender` = router.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly; the original user's address is stored only in transient callback context (for payment), never forwarded to the pool as the caller identity: [3](#0-2) 

**Step 3 — The extension checks the router's address, not the end user's.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` = pool and `sender` = router: [4](#0-3) 

**The dilemma the pool admin faces:**

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every user on the public router bypasses the per-user gate |

There is no configuration that simultaneously supports router-mediated swaps and enforces per-user restrictions. A pool admin who allowlists the router — a natural action when deploying a curated pool that is expected to integrate with the standard periphery — silently opens the pool to all users.

### Impact Explanation
Any user can swap on a pool whose admin intended to restrict access to a curated set of addresses (e.g., KYC-verified, institutional, or protocol-internal addresses). The attacker pays no special cost: they simply route through the public `MetricOmmSimpleRouter`. The pool receives tokens and settles the swap normally, so there is no revert — the bypass is silent. Depending on the pool's purpose, this can mean unauthorized price exposure, fee extraction from a restricted liquidity pool, or violation of regulatory/compliance requirements that the allowlist was meant to enforce.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the standard user-facing entry point documented and deployed alongside the core pool. Any pool admin who wants their allowlisted users to benefit from the router's slippage protection, multi-hop routing, or deadline enforcement must allowlist the router. The design flaw is not obvious: the admin sees "allowlist the router so my users can use it" without realizing this grants access to every user on the router. The trigger requires no privileged action beyond the admin's own reasonable configuration step.

### Recommendation
The extension must gate the **economic actor**, not the intermediary. Two complementary fixes:

1. **Router-side:** `MetricOmmSimpleRouter` should encode `msg.sender` (the original user) into `extensionData` for each hop, and `SwapAllowlistExtension.beforeSwap` should decode and check that value when `sender` is a known router.

2. **Extension-side (simpler):** Add an `onlyDirectSwapper` path: if `sender` is a registered periphery router, decode the real user from `extensionData` and check that address against the allowlist instead.

Either approach ensures the allowlist always gates the address that economically benefits from the swap, regardless of which supported entry point is used.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the curated user
  allowedSwapper[pool][router] = true  // admin adds router to support alice's router swaps

Attack:
  bob (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})

  pool.swap(recipient, ...) is called with msg.sender = router
  _beforeSwap(router, ...) is dispatched
  SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
  swap executes for bob — allowlist bypassed
```

The pool admin's intent was to allow only `alice`; the result is that every user of the public router can trade on the curated pool.
