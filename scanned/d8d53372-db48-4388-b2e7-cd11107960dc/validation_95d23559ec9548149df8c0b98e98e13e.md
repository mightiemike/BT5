### Title
`OracleValueStopLossExtension` Uses Arithmetic Mid-Price Instead of the Pool's Geometric Mid-Price, Causing the Stop-Loss Guard to Underprotect LPs in the `!zeroForOne` Direction — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as the arithmetic mean `(bid + ask) / 2`, while every other pricing surface in the protocol — the pool's swap engine, `PriceVelocityGuardExtension`, and `MetricOmmPoolDataProvider` — uses the geometric mean `sqrt(bid * ask)` via `SwapMath.midAndSpreadFeeX64FromBidAsk`. Because the arithmetic mean is always ≥ the geometric mean (AM-GM inequality), the stop-loss systematically overestimates `metricToken1`, causing the guard to fail to trigger for `!zeroForOne` swaps when it should, allowing LP value to drain below the configured drawdown floor.

---

### Finding Description

**The wrong mid-price formula in the stop-loss extension:**

In `OracleValueStopLossExtension._afterSwapOracleStopLoss` (line 218):

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

**The correct formula used everywhere else in the protocol:**

`SwapMath.midAndSpreadFeeX64FromBidAsk` computes the geometric mean:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

The pool's `swap` function uses this geometric mid for all swap math: [3](#0-2) 

`PriceVelocityGuardExtension` also correctly uses the geometric mid: [4](#0-3) 

**How the wrong mid-price corrupts the stop-loss metric:**

The `_metrics` function computes per-share value in token1 terms as:

```
metricToken1 = (t0 * mid / 2^64) * SCALE / shares + t1 * SCALE / shares
``` [5](#0-4) 

Because `M_arithmetic ≥ M_geometric` always (AM-GM), `metricToken1` is overestimated. The high-watermark ratchets up to this inflated value: [6](#0-5) 

The floor is `hwm * (1 - drawdown)`. Because the watermark was set with an inflated mid, the floor is also inflated. When token0 is subsequently drained by `!zeroForOne` swaps, the current `metricToken1` (also computed with the inflated arithmetic mid) stays above the inflated floor longer than it would with the correct geometric mid. The guard fails to revert when the actual LP value — measured at the pool's true geometric mid — has already fallen below the drawdown floor.

**Quantitative discrepancy:**

| Oracle spread | Arithmetic mid | Geometric mid | Relative error |
|---|---|---|---|
| 1% (bid=0.995, ask=1.005) | 1.000 | 0.99999 | ~0.001% |
| 10% (bid=0.95, ask=1.05) | 1.000 | 0.99875 | ~0.125% |
| 30% (bid=0.85, ask=1.15) | 1.000 | 0.98869 | ~1.13% |
| 56% (bid=0.80, ask=1.25) | 1.025 | 1.000 | ~2.5% |

For pools with wide oracle spreads (which are a normal operating condition for Metric OMM, since the spread is the primary fee mechanism), the effective drawdown floor is materially lower than the admin configured.

**The `afterSwap` hook is called on every swap:** [7](#0-6) 

Any unprivileged trader executing a `!zeroForOne` swap triggers the hook. The guard's decision to revert or not is made inside `_checkAndUpdateWatermarks`: [8](#0-7) 

---

### Impact Explanation

The stop-loss extension is the primary on-chain mechanism protecting LP principal from value leakage. Because it uses arithmetic mid instead of geometric mid, `metricToken1` is overestimated, the high-watermark is set too high, and the floor check `metric < hwm * (1 - drawdown)` is harder to breach than the pool admin intended. For pools with non-trivial oracle spreads, the guard allows `!zeroForOne` swaps to drain token0 from LP positions beyond the configured drawdown floor before reverting. The excess loss is proportional to `(M_arithmetic - M_geometric) / M_geometric`, which grows quadratically with spread width. For a 30% oracle spread, the effective floor is ~1.1% lower than configured; for a 56% spread, ~2.5% lower. LP principal loss above the configured floor is direct and unrecoverable.

---

### Likelihood Explanation

- Every pool that deploys `OracleValueStopLossExtension` is affected whenever the oracle spread is non-zero (i.e., always, since spread is the pool's fee source).
- Any unprivileged trader can trigger the `afterSwap` hook by executing a `!zeroForOne` swap.
- No special setup, privileged role, or malicious token is required.
- The discrepancy is present from the first swap and grows with spread width.

---

### Recommendation

Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the same geometric mid used by the pool's swap engine:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

This aligns the stop-loss metric with the pool's actual pricing, ensuring the drawdown floor is enforced at the value the pool admin configured.

  /// @notice Geometric mid price (Q64.64) and spread fee in Q64.64 from bid/ask oracle quotes.
  function midAndSpreadFeeX64FromBidAsk(uint256 bidPriceX64, uint256 askPriceX64)
    internal
    pure
    returns (uint256 midPriceX64, uint256 baseFeeX64)
  {
    midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
    baseFeeX64 = Math.mulDiv(askPriceX64, ONE_X64, midPriceX64, Math.Rounding.Ceil) - ONE_X64;
