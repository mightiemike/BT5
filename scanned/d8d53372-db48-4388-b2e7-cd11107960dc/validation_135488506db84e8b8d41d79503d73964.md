### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` parameter (the address that actually calls `addLiquidity` and pays tokens via the callback) and instead gates on `owner` (the LP-position recipient). Because `addLiquidity` imposes no constraint that `msg.sender == owner`, any address can bypass the allowlist by specifying an allowlisted `owner`. Symmetrically, a legitimately allowlisted `sender` is blocked whenever it specifies a non-allowlisted `owner`. This is the direct analog of the external report's `msg.sender`/`address(this)` mismatch: the wrong address entity is checked in the security-critical guard.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts two distinct address roles: [1](#0-0) 

- **`msg.sender`** (`sender`) — the address that calls `addLiquidity`, receives the swap callback, and pays the tokens into the pool.
- **`owner`** — the address that receives the minted LP-position shares; it may be any address, including one that is entirely different from `msg.sender`.

Both are forwarded to every extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed), and gates only on `owner`: [3](#0-2) 

The `SwapAllowlistExtension` correctly checks `sender` (the actual executor): [4](#0-3) 

The inconsistency is the root cause. `removeLiquidity` enforces `msg.sender == owner` before calling the hook, so the mismatch is harmless there. `addLiquidity` has no such constraint, making the mismatch exploitable. [5](#0-4) 

---

### Impact Explanation

**Bypass path (admin-boundary break):** An address `B` not in the allowlist calls `pool.addLiquidity(owner = A, ...)` where `A` is allowlisted. The extension checks `allowedDepositor[pool][A]` → passes. `B` pays tokens via the callback and successfully adds liquidity to a permissioned pool, violating the pool admin's access-control intent. `B` can use this to inject liquidity into specific bins, distorting the pool's bin distribution and affecting swap execution for all users.

**Blocking path (broken core functionality):** An allowlisted `sender` `A` calls `pool.addLiquidity(owner = B, ...)` (e.g., a router creating positions on behalf of users). The extension checks `allowedDepositor[pool][B]` → fails. `A` is blocked despite being explicitly allowlisted, making the `addLiquidity` flow unusable for any router or contract that separates the payer from the position owner.

Both paths match the allowed impact gate: admin-boundary bypass by an unprivileged path, and broken core pool liquidity flow.

---

### Likelihood Explanation

The trigger requires only a standard `addLiquidity` call with `owner != msg.sender`, which is the normal pattern for any router or liquidity-management contract (e.g., `MetricOmmPoolLiquidityAdder`). No special permissions, flash loans, or privileged access are needed. Any user interacting through a router against a pool that has `DepositAllowlistExtension` configured will encounter the blocking path immediately; the bypass path requires only knowledge of one allowlisted address.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external
      view
      override
      returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and any admin tooling to document that the allowlisted address is the **caller** of `addLiquidity`, not the LP-position owner.

---

### Proof of Concept

```solidity
// Pool is configured with DepositAllowlistExtension.
// Only `allowedSender` is allowlisted; `attacker` is not.
allowlist.setAllowedToDeposit(pool, allowedSender, true);

// Attacker calls addLiquidity specifying allowedSender as owner.
// Extension checks allowedDepositor[pool][allowedSender] → true → passes.
// Attacker pays tokens via callback; allowedSender receives LP shares.
// Permissioned pool now has liquidity injected by an unauthorized address.
pool.addLiquidity(
    allowedSender,   // owner  ← allowlisted, check passes
    0,               // salt
    deltas,
    callbackData,
    ""
); // called by attacker — succeeds despite attacker not being allowlisted
```
