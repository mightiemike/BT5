### Title
Router Address Substitution Bypasses SwapAllowlistExtension Per-User Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router address, not the originating EOA. A pool admin who allowlists the router (the natural configuration for a router-accessible curated pool) inadvertently grants every user access, defeating the allowlist entirely.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly — `msg.sender` to the pool is the router, not the originating EOA: [4](#0-3) 

When a pool admin sets `allowedSwapper[pool][router] = true` (the expected configuration to allow router-mediated swaps on a curated pool), the check `allowedSwapper[pool][router]` passes for **every** user who calls through the router, regardless of whether their own address is allowlisted.

### Impact Explanation

Any non-allowlisted EOA can swap on a pool that is intended to be restricted to specific users by routing through `MetricOmmSimpleRouter`. The pool's curation invariant is broken: the pool admin believes only allowlisted addresses can trade, but in practice the allowlist is bypassed for all router-mediated swaps. This enables unrestricted trading on pools that may be designed for specific counterparties (e.g., KYC'd users, institutional LPs), with direct fund impact if the pool's pricing or liquidity is calibrated for a restricted set of traders.

### Likelihood Explanation

The scenario requires only that the pool admin has allowlisted the router — the natural and expected configuration for any curated pool that is also meant to be accessible via the standard router. No privileged or malicious action is needed beyond normal pool setup. Any user can then call `exactInputSingle` from a non-allowlisted EOA.

### Recommendation

`SwapAllowlistExtension.beforeSwap` should check the **originating user** rather than the immediate caller of `pool.swap`. Two options:

1. **Pass the original payer through `extensionData`**: the router encodes `msg.sender` into `extensionData`, and the extension decodes and checks it. This requires a trusted encoding convention.
2. **Check `recipient` instead of `sender`**: if the pool's design intent is to gate who receives output, check the `recipient` parameter. If the intent is to gate who pays, the router must forward the original payer explicitly.
3. **Require direct pool interaction**: document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this via an `onlyEOA` check or by requiring `tx.origin == sender`.

### Proof of Concept
