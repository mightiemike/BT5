### Title
Unbounded `timelock` in `OracleValueStopLossExtension` Causes `uint32` Overflow, Allowing Pool Admin to Bypass LP-Protection Timelock Instantly - (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` enforces a per-pool timelock so LPs have advance notice before the pool admin can weaken the stop-loss guard (lower drawdown, raise decay, or reset watermarks). The timelock value is stored as `uint32` and is never bounded. `_afterTimelock()` computes `uint32(block.timestamp + timelock)`, which silently wraps when `timelock` is large enough, producing an `executeAfter` timestamp that is already in the past. Every subsequent `_requireElapsed()` check then passes immediately, giving the pool admin an instant bypass of the timelock for all future proposals.

---

### Finding Description

`_afterTimelock` is the single function that stamps every pending proposal with its earliest execution time:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space and then **truncated** to `uint32`. When `timelock = type(uint32).max` (4 294 967 295):

```
block.timestamp (Ôëł 1 753 000 000, July 2026)
+ type(uint32).max (4 294 967 295)
= 6 047 967 295
uint32(6 047 967 295) = 6 047 967 295 mod 4 294 967 296 Ôëł 1 752 999 999
```

The result is **one second in the past**. `_requireElapsed` then checks:

```solidity
if (block.timestamp < executeAfter) revert ...;
// 1_753_000_000 < 1_752_999_999  Ôćĺ  false  Ôćĺ  no revert
```

Every proposal stamped with this overflowed `executeAfter` can be executed in the same transaction it was proposed.

Neither `initialize()` nor `proposeOracleStopLossTimelock()` validates an upper bound on the timelock value:

```solidity
// initialize ÔÇö no _validateTimelock call
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock stored without any bound check
```

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    // newTimelock accepted without any upper-bound check
    sched.pendingTimelock = newTimelock;
    ...
}
```

Compare with `drawdownE6` and `decayPerSecondE8`, which both have explicit guards (`_validateDrawdown`, `_validateDecay`). The `timelock` field has no analogous guard.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value leakage. Its NatSpec explicitly states: *"Drawdown and decay changes are timelocked so LPs can react; monitor at least as often as the timelock or trust the pool admin."*

Once the pool admin sets `timelock = type(uint32).max` and the overflow is active:

1. **Drawdown can be set to 0 instantly** ÔÇö `_afterSwapOracleStopLoss` returns early when `drawdown == 0`, disabling all stop-loss checks.
2. **Decay can be maximised instantly** ÔÇö `decayPerSecondE8 = E8` causes watermarks to decay to zero in one second, making the guard permanently inactive.
3. **Watermarks can be zeroed instantly** ÔÇö removing the reference floor against which value loss is measured.

With the guard disabled, the pool admin can collude with an oracle update to move the mid-price adversarially, draining LP token balances through swaps that would otherwise have been blocked by `OracleStopLossTriggered`. This is a direct loss of LP principal.

---

### Likelihood Explanation

The pool admin is a semi-trusted role: the protocol explicitly warns LPs to "monitor at least as often as the timelock." The timelock is the only on-chain constraint limiting the admin's power over the stop-loss parameters. A malicious or compromised pool admin can execute this attack in two steps separated by the current timelock duration (which may be zero at deployment). No external oracle manipulation or special token behaviour is required ÔÇö only a valid `uint32` argument to `proposeOracleStopLossTimelock`.

---

### Recommendation

Add an upper-bound validation for `timelock` analogous to the existing `_validateDrawdown` and `_validateDecay` guards:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days; // or a protocol-chosen cap

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Call `_validateTimelock` in both `initialize()` and `proposeOracleStopLossTimelock()`. This prevents the `uint32` overflow and ensures the timelock remains within a range where LPs can meaningfully react.
