Looking at the `OracleValueStopLossExtension.sol` and how `executeOracleStopLossDecay` interacts with the lazy-decay watermark accounting, I can identify a direct analog to the Reserve Protocol bug.

---

### Title
`executeOracleStopLossDecay` applies new decay rate retroactively without first settling per-bin watermarks, weakening the stop-loss guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`OracleValueStopLossExtension` tracks per-bin high watermarks that decay lazily at `decayPerSecondE8`. When the pool admin executes a decay-rate change via `executeOracleStopLossDecay`, only `oracleStopLossConfig[pool_].decayPerSecondE8` is updated; the per-bin `highWatermarks[pool][binIdx].lastDecayTs` timestamps are **not** advanced and the old rate is **not** applied to the elapsed period. The next `afterSwap` call then applies the new rate to the entire elapsed time since the last watermark touch, including the interval that predates the rate change. If the rate is increased, the watermarks decay more than intended, weakening or completely disabling the stop-loss guard and allowing value-draining swaps that should have been blocked.

### Finding Description

The decay is applied lazily inside `_checkAndUpdateWatermarks`:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
``` [1](#0-0) 

`_decayed` uses the **current** `decayRate` for the full `dt` window:

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
}
``` [2](#0-1) 

`executeOracleStopLossDecay` writes the new rate but does **not** touch any `lastDecayTs`:

```solidity
function executeOracleStopLossDecay(address pool_) external onlyPoolAdmin(pool_) {
    ...
    uint32 decay = sched.pendingDecayPerSecondE8;
    oracleStopLossConfig[pool_].decayPerSecondE8 = decay;   // ← only this changes
    (sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
    emit OracleStopLossDecaySet(pool_, decay);
}
``` [3](#0-2) 

The correct amount of decay that should have been applied to the period `[lastDecayTs, T_change)` is `rate_old × (T_change − lastDecayTs)`. After the rate change, the period `[T_change, T_next)` should decay at `rate_new`. Instead, the entire window `[lastDecayTs, T_next)` is decayed at `rate_new`, retroactively misapplying the new rate to the past.

### Impact Explanation

When `rate_new > rate_old` (admin loosens the guard):

- `factor = rate_new × dt` can reach or exceed `1e8`, collapsing the watermark to `0`.
- A zero watermark means `_applyWatermark` always returns `(metric, false)` — no breach is ever detected.
- The stop-loss guard is silently disabled for the current swap, allowing a swap that drains LP value per share below the configured drawdown floor to execute without revert.
- LPs suffer direct principal loss that the extension was deployed to prevent.

When `rate_new < rate_old` (admin tightens the guard):

- Less decay is applied retroactively; watermarks remain artificially high.
- Legitimate swaps are blocked, breaking core pool swap functionality.

### Likelihood Explanation

The pool admin is a semi-trusted actor who can legitimately propose and execute decay-rate changes through the timelocked flow. A well-intentioned admin increasing the decay rate (e.g., to allow the pool to recover faster after a price move) would not expect the new rate to apply retroactively to the entire unsettled window. The timelock gives LPs notice of the *intent* to change the rate, but does not prevent the retroactive effect once executed. Any pool using `OracleValueStopLossExtension` with a non-zero decay rate and infrequent swap activity is exposed.

### Recommendation

Before writing the new decay rate, apply the old rate to all active bins up to `block.timestamp` — i.e., advance `lastDecayTs` and store the decayed watermark values. The minimal fix mirrors the Reserve Protocol mitigation: settle the old state before changing the slope.

```solidity
function executeOracleStopLossDecay(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingDecayExecuteAfter == 0) revert OracleStopLossNoPendingDecay(pool_);
    _requireElapsed(sched.pendingDecayExecuteAfter);

    // Settle all watermarks at the OLD rate before changing it
    uint32 oldDecay = oracleStopLossConfig[pool_].decayPerSecondE8;
    _settleAllBinWatermarks(pool_, oldDecay);   // advance lastDecayTs to block.timestamp

    uint32 decay = sched.pendingDecayPerSecondE8;
    oracleStopLossConfig[pool_].decayPerSecondE8 = decay;
    (sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
    emit OracleStopLossDecaySet(pool_, decay);
}
```

Alternatively, store a `rateChangedAt` timestamp alongside the new rate and split the `dt` window in `_decayed` at that boundary.

### Proof of Concept

1. Pool is deployed with `OracleValueStopLossExtension`; `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 1` (near-zero decay).
2. A swap executes at block T=0. `_checkAndUpdateWatermarks` sets `hwm0 = 1000`, `lastDecayTs = 0`.
3. No swaps occur for 86 400 seconds (1 day).
4. Pool admin proposes `decayPerSecondE8 = 1e8` (100 %/s). Timelock elapses. Admin calls `executeOracleStopLossDecay`.
   - `oracleStopLossConfig[pool_].decayPerSecondE8 = 1e8`.
   - `highWatermarks[pool][0].lastDecayTs` remains `0`.
5. At T = 86 401 a swap executes. `afterSwap` → `_checkAndUpdateWatermarks`:
   - `dt = 86 401 − 0 = 86 401`.
   - `factor = 1e8 × 86 401 = 8.64 × 10¹² ≥ 1e8` → `_decayed(1000, 1e8, 86401) = 0`.
   - `hwm0 = 0`; `_applyWatermark(metric, 0, floorMultiplier)` → `metric >= 0` → `breached = false`.
6. The stop-loss guard does not revert. A swap that reduces token0 value per share by 10 % (above the 5 % drawdown floor) executes successfully, causing LP principal loss. [3](#0-2) [4](#0-3) [2](#0-1)

  function _checkAndUpdateWatermarks(
    address pool_,
    int8 binIdx,
    uint256 metricT0,
    uint256 metricT1,
    uint256 floorMultiplier,
    uint256 decayRate,
    bool zeroForOne
  ) private {
    BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
    uint256 dt = block.timestamp - hwmS.lastDecayTs;

    (uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
    if (breach0 && zeroForOne) {
