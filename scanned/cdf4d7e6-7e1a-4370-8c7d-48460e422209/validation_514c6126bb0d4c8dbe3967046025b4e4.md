### Title
SwapAllowlistExtension Gates on Router Address Instead of Actual User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. Any user who routes through the router on a pool that has allowlisted the router (or any pool where the admin allowlists the router to support router-mediated swaps) bypasses the per-user allowlist entirely.

---

### Finding Description

**Pool swap call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
              msg.sender = router
              → _beforeSwap(msg.sender=router, recipient, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        checks: allowedSwapper[pool][router]
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension:

```solidity
// ExtensionCalling.sol line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**Contrast with `DepositAllowlistExtension`**, which correctly checks the `owner` parameter (the actual depositor, explicitly passed by the caller):

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

The deposit extension gates on `owner` (the economic actor). The swap extension gates on `sender` (the immediate caller), which is the router when routing is used.

**Two broken scenarios:**

1. **Bypass**: Pool admin allowlists the router (e.g., to support multi-hop routing for legitimate users). Any unprivileged user can now call `exactInputSingle` / `exactInput` / `exactOutput` through the router and bypass the per-user allowlist entirely, because the extension sees `allowedSwapper[pool][router] == true`.

2. **Lockout**: Pool admin does NOT allowlist the router. Allowlisted users who attempt to swap through the router are blocked (`allowedSwapper[pool][router] == false`), even though they are individually allowlisted. They can only swap by calling `pool.swap()` directly, which requires implementing the `IMetricOmmSwapCallback` interface — not possible for EOAs.

Both outcomes break the invariant that the allowlist gates the economically relevant actor.

---

### Impact Explanation

**Bypass path (scenario 1):** A curated pool with `SwapAllowlistExtension` is rendered open to all users. Any address can call `MetricOmmSimpleRouter.exactInputSingle()` targeting the pool. The extension sees the router (allowlisted) as the swapper and permits the trade. The pool admin's intent to restrict swaps to specific counterparties is completely defeated. This is a direct policy bypass on a curated pool — equivalent to a broken allowlist guard with fund-impacting consequences (unauthorized parties can drain liquidity at oracle prices).

**Lockout path (scenario 2):** Allowlisted users cannot use the router, which is the only practical swap entry point for EOAs (direct `pool.swap()` requires implementing a callback). This breaks core swap functionality for legitimate users.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool deployer who configures `SwapAllowlistExtension` and also wants to support router-mediated swaps faces this dilemma immediately. The bypass requires no special privileges — any user can call the public router functions. The lockout manifests for every allowlisted EOA who tries to use the router. Both failure modes are reachable on the first real use of a curated pool.

---

### Recommendation

The `beforeSwap` hook should gate on the actual initiating user, not the immediate caller. Two approaches:

**Option A — Pass the original `msg.sender` through the router as a dedicated field.** The router encodes the original caller in `extensionData` and the extension decodes it. This requires a protocol-level convention.

**Option B — Mirror the deposit extension pattern.** Add a `swapper` parameter to the swap interface that the pool passes explicitly (analogous to `owner` in `addLiquidity`), so the extension always sees the intended economic actor regardless of routing intermediaries.

**Option C — Gate on `tx.origin` as a stopgap.** Not recommended for production (breaks contract-to-contract flows), but illustrates the intent.

The cleanest fix is Option B: the pool should pass an explicit `swapper` address (defaulting to `msg.sender` for direct calls, but overridable by trusted routers) so the extension always checks the correct actor.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted
swapExtension.setAllowedToSwap(address(pool), allowedUser, true);
// Pool admin also allowlists the router to support multi-hop routing:
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: blockedUser routes through the router
vm.startPrank(blockedUser);
token0.approve(address(router), type(uint256).max);
// This succeeds — extension sees router (allowlisted), not blockedUser (not allowlisted)
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: blockedUser,
    tokenIn: address(token0),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// blockedUser successfully swapped on a pool they were not allowlisted for
```
