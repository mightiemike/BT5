### Title
`SwapAllowlistExtension` Gates Router Address Instead of End User — Allowlist Fully Bypassed When Router Is Allowlisted - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is always `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, the pool sees `msg.sender = router`, so the extension checks whether the **router address** is allowlisted — not the actual end user. A pool admin who allowlists the router to enable router-mediated swaps for their permitted users inadvertently opens the pool to **all** users, completely defeating the allowlist guard.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument — the router address: [3](#0-2) 

The allowlist is keyed `pool → swapper`. When the router is the immediate caller, the lookup is `allowedSwapper[pool][router]`. There is no path through which the extension can observe the actual end user who initiated the transaction.

This creates two mutually exclusive failure modes:

| Router allowlisted? | Effect |
|---|---|
| No | All router users blocked, even individually allowlisted ones — core swap flow broken |
| Yes | All users bypass individual restrictions via the router — allowlist defeated |

A pool admin who wants to allow their allowlisted users to also use the router must allowlist the router address. That single act silently opens the pool to every address on the network that can call the router.

The analog to the hardcoded USDC peg is exact: the USDC code assumed `identity(USDC) = 1 USD` (a fixed mapping that breaks when the peg deviates); this code assumes `identity(sender) = economic actor` (a fixed mapping that breaks when an intermediary is inserted).

### Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g., KYC'd traders, institutional partners, or a private liquidity arrangement) loses that restriction entirely the moment the router is allowlisted. Any unpermissioned address can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and execute swaps against the pool. LP funds are exposed to arbitrary counterparties at oracle-derived prices that may have been set assuming a restricted, trusted trading set. [4](#0-3) 

### Likelihood Explanation

Medium. The pool admin must allowlist the router — a privileged but entirely reasonable action. Any admin who wants their allowlisted users to access the router (the primary user-facing entry point) will perform this step. The documentation does not warn that allowlisting the router disables per-user gating. The misapplication is therefore a predictable operational mistake, not a malicious one.

### Recommendation

The extension must verify the **originating user**, not the immediate caller. Two viable approaches:

1. **Pass the original user through the router.** The router encodes `msg.sender` in `extensionData`; the extension decodes and verifies it. The pool must enforce that the extension data is not forgeable (e.g., by signing or by restricting who can set it).

2. **Check both the router and the originating user.** Extend the allowlist to a two-level mapping: `allowedSwapper[pool][router][endUser]`, and require the router to forward the end user identity in a verifiable way.

3. **Separate router-level and user-level allowlists.** Maintain one allowlist for trusted routers and a second for permitted end users; require both to pass.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is the intended permitted user
  allowedSwapper[pool][router] = true  // admin adds router to enable router-mediated swaps for alice

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, tokenIn: ..., ...})

  Execution path:
    router.exactInputSingle()
      → pool.swap(recipient, ...) [msg.sender = router]
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (passes)
        → swap executes, bob receives output tokens

Result:
  bob swaps successfully despite never being allowlisted.
  The allowlist guard is silently bypassed for all router users.
