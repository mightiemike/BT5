### Title
SwapAllowlistExtension gates on router address instead of original user, allowing any unprivileged user to bypass swap restrictions when the router is allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is the direct `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, that `sender` is the **router contract address**, not the original user. If the pool admin adds the router to the allowlist — a natural configuration for a pool that wants to support router-mediated swaps — any unprivileged user can bypass the restriction by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which is `msg.sender` of the `pool.swap()` call itself:

```solidity
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` or `exactInput()`, the router calls `pool.swap()` as `msg.sender = router`:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

The pool then calls `extension.beforeSwap(sender=router, ...)`. The extension checks `allowedSwapper[pool][router]`. The original user's address is stored only in transient storage for the callback payment and is **never forwarded to the pool or the extension**. The extension has no way to distinguish which user initiated the router call.

If the pool admin adds the router to the allowlist — a natural step for a pool that wants to support router-mediated swaps for its allowlisted users — the check passes for **every** user who routes through the router, regardless of whether they are individually allowlisted.

The same structural flaw applies to `exactInput` (multi-hop) and `exactOutput` / `exactOutputSingle`. [4](#0-3) 

---

### Impact Explanation

The `SwapAllowlistExtension` is a production-grade access-control guard. Its purpose is to restrict which addresses may trade on a pool — for example, to limit trading to specific market makers, enforce KYC/AML compliance, or prevent harmful trading patterns that would harm LPs. Once the router is allowlisted, the guard is silently nullified for all router-mediated swaps. Any unprivileged user can trade on the restricted pool, causing:

- **Direct LP fund loss** if the restriction was meant to prevent adversarial or uninformed order flow (e.g., only allowing a specific oracle-aware market maker).
- **Broken core pool invariant**: the access-control invariant the pool admin configured is violated on every router-mediated swap.
- **Compliance breach** if the allowlist enforces regulatory requirements.

---

### Likelihood Explanation

The scenario requires the pool admin to add the router to the allowlist. This is a **natural and expected configuration**: a pool admin who wants allowlisted users to be able to use the router (rather than calling the pool directly) must add the router to the allowlist. There is no documentation or on-chain guard warning that doing so opens the gate to all users. The admin's intent (allow specific users via the router) diverges silently from the implementation's effect (allow all users via the router). The trigger is unprivileged — any user can exploit it once the router is allowlisted.

---

### Recommendation

The extension must verify the **original user**, not the intermediary. Two concrete options:

1. **Pass the original user in `extensionData`**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and verifies it. This requires a coordinated convention between the router and the extension.

2. **Add an `originator` field to the swap interface**: Extend `pool.swap()` with an explicit `originator` address that the router sets to `msg.sender`. The extension checks `originator` instead of `sender`. This is the cleanest fix but requires a core interface change.

Until fixed, pool admins must be explicitly warned: **adding the router to the swap allowlist grants swap access to all users, not just those individually allowlisted**.

---

### Proof of Concept

**Setup**:
- Pool deployed with `SwapAllowlistExtension` configured.
- Pool admin calls `setAllowedToSwap(pool, userA, true)` — intending only `userA` to swap.
- Pool admin calls `setAllowedToSwap(pool, router, true)` — intending to let `userA` use the router.

**Attack** (executed by `userB`, who is NOT on the allowlist):

```solidity
// userB calls the router directly — no individual allowlist entry needed
router.exactInputSingle(ExactInputSingleParams({
    pool:           pool,
    recipient:      userB,
    zeroForOne:     true,
    amountIn:       1_000e18,
    amountOutMinimum: 0,
    priceLimitX64:  0,
    deadline:       block.timestamp,
    tokenIn:        token0,
    extensionData:  ""
}));
// Pool calls extension.beforeSwap(sender=router, ...)
// Extension checks allowedSwapper[pool][router] == true  ← passes
// userB's swap executes on the restricted pool
```

**Result**: `userB` successfully swaps on a pool the admin intended to restrict to `userA` only. The allowlist invariant is broken for every router-mediated swap. [5](#0-4) [6](#0-5) [7](#0-6)
