### Title
Swap Allowlist Bypassed via Router: `sender` Identity Mismatch in `SwapAllowlistExtension` тАФ (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the direct caller of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router's address**, not the user's address. If the pool admin allowlists the router (a necessary step for allowlisted users to use the router), every non-allowlisted user can bypass the per-user swap gate by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` captures `msg.sender` and forwards it as `sender` to `_beforeSwap`, which in turn passes it to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that `sender` value into the call to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap(...)` тАФ making the router the `msg.sender` of that call. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The identity mismatch:

| Layer | Address checked |
|---|---|
| Intended gate | Individual user (economic actor) |
| Actual gate | `MetricOmmSimpleRouter` (proximate caller) |

For allowlisted users to reach the pool through the router, the pool admin must add the router to the allowlist. Once the router is allowlisted, **every** user тАФ including those explicitly excluded тАФ can bypass the per-user gate by routing through the router.

This is structurally identical to H-19: one metric (the router's identity) is used for the guard check, while the economically relevant metric (the individual user's identity) drives the actual swap execution.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` and `allowAllSwappers[pool] = false` intends to restrict swaps to a curated set of counterparties (e.g., an institutional or private pool). Once the router is allowlisted, any address can execute swaps against that pool, bypassing the restriction entirely. This allows:

- Unauthorized parties to trade against LP positions that were never meant to be exposed to them.
- Potential LP value extraction if the pool's pricing or liquidity was calibrated for a specific, trusted counterparty set.
- Circumvention of any combined stop-loss or velocity guard that relies on the allowlist as a first line of defense.

This is a direct loss-of-principal risk for LPs in restricted pools.

---

### Likelihood Explanation

The bypass requires only that the router be allowlisted тАФ a configuration the pool admin is compelled to make if any allowlisted user is expected to use the router. The attack requires no special privileges, no flash loans, and no unusual token behavior. Any user who observes that the router is allowlisted (readable on-chain) can execute the bypass immediately.

---

### Recommendation

The extension must gate the **economic actor**, not the proximate caller. Two sound approaches:

1. **Require direct pool interaction**: Document that allowlisted pools must not allowlist the router; allowlisted users must call `pool.swap` directly.
2. **Pass the originating user through `extensionData`**: Have the router encode the user's address in `extensionData` and have the extension decode and check that address instead of `sender`. This requires the extension to trust the router's encoding, so the router itself must be separately verified.

The cleanest invariant: `allowedSwapper[pool][sender]` must resolve to the address that controls the economic outcome of the swap, which is the user, not the router.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Admin sets `allowAllSwappers[pool] = false`.
3. Admin allowlists `alice`: `allowedSwapper[pool][alice] = true`.
4. Admin allowlists the router: `allowedSwapper[pool][router] = true` (so alice can use the router).
5. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
6. Router calls `pool.swap(recipient=bob, ...)` тАФ `msg.sender` in the pool is `router`.
7. Pool calls `extension.beforeSwap(sender=router, ...)`.
8. Extension evaluates `allowedSwapper[pool][router] == true` тЖТ passes.
9. Bob's swap executes against the restricted pool, bypassing the per-user allowlist entirely. [3](#0-2) [4](#0-3)
