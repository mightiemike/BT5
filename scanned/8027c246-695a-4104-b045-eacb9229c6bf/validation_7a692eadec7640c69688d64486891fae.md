### Title
SwapAllowlistExtension gates router address instead of actual swapper, enabling full allowlist bypass via router â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `sender` â€” the immediate caller of `pool.swap()` â€” against the per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, `sender` is the **router address**, not the actual end user. A pool admin who allowlists the router to enable router-mediated swaps for their curated pool inadvertently grants every unprivileged user the ability to bypass the individual allowlist entirely.

---

### Finding Description

**Root cause â€” wrong actor bound in the allowlist check.**

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as `sender` to `_beforeSwap()`: [1](#0-0) 

`_beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user enters through `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [4](#0-3) 

So the extension evaluates `allowedSwapper[pool][router]`, **not** `allowedSwapper[pool][actualUser]`.

**The resulting dilemma for pool admins:**

| Admin configuration | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all â€” usability broken |
| Router **allowlisted** (to enable router usage) | **Every** user, including non-allowlisted ones, can bypass the individual allowlist via the router |

There is no configuration that simultaneously allows router-mediated swaps and enforces per-user allowlist restrictions. The extension's identity binding is structurally wrong for the router path.

This is the direct analog of the UlyssesToken off-by-one: in that bug, the wrong numeric ID (0-based instead of 1-based) was written back during `removeAsset`, corrupting the identity mapping for subsequent operations. Here, the wrong address (router instead of actual user) is bound as the gated identity, corrupting the allowlist check for every router-mediated swap.

---

### Impact Explanation

A pool admin who deploys a curated pool (e.g., KYC-gated, institutional-only) with `SwapAllowlistExtension` and also allowlists the router â€” a natural and expected configuration â€” exposes the pool to unrestricted swapping by any address. Any non-allowlisted user can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` and the extension will pass the check because `allowedSwapper[pool][router] = true`. The curation invariant is fully broken: the pool behaves as if `allowAllSwappers[pool] = true` for all router users.

---

### Likelihood Explanation

The trigger is unprivileged: any user can call the public router. The only precondition is that the pool admin has allowlisted the router address, which is the natural step a pool admin takes when they want their curated pool to be usable through the standard periphery. The admin is not malicious â€” they are unaware that allowlisting the router collapses the per-user allowlist for all router paths. This configuration is likely in any production deployment that intends to support the router
