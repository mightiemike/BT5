### Title
`SwapAllowlistExtension` gates the router address instead of the actual swapper, allowing any user to bypass the allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` from the pool's perspective — the **direct caller of `pool.swap()`**. When a user routes through the public `MetricOmmSimpleRouter`, `sender` is the router address, not the actual user. If the pool admin allowlists the router (so that allowlisted users can use it), every unprivileged user can bypass the allowlist entirely by routing through the same public contract.

---

### Finding Description

**Step 1 — Pool forwards `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`_beforeSwap` encodes that value as the `sender` argument and dispatches it to every configured extension: [2](#0-1) 

**Step 2 — The extension checks `allowedSwapper[pool][sender]`.**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (the pool) as the mapping key for the pool and `sender` (the direct caller of `pool.swap()`) as the identity being gated: [3](#0-2) 

**Step 3 — The router is the direct caller of `pool.swap()`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. From the pool's perspective, `msg.sender` is the router, so `sender` forwarded to the extension is the router address, not the originating user: [4](#0-3) 

**Step 4 — The identity mismatch creates an irreconcilable dilemma.**

The pool admin faces two mutually exclusive outcomes:

| Admin action | Result |
|---|---|
| Allowlist the router | Every unprivileged user bypasses the allowlist by routing through the public router |
| Do not allowlist the router | Allowlisted users cannot use the router; core swap UX is broken for them |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

A pool admin who deploys a restricted pool (e.g., for KYC-verified counterparties, specific market makers, or a private liquidity venue) and allowlists the router so that legitimate users can access the pool through the standard periphery will inadvertently open the pool to **all** users. Non-allowlisted users can execute swaps against LP positions that were never intended to face them, causing adverse-selection losses for LPs. This is a direct loss of LP principal above Sherlock thresholds and constitutes broken core pool functionality (the allowlist guard is the primary access-control mechanism for restricted pools).

---

### Likelihood Explanation

Exploitation requires no special privilege. Any user who observes that a pool has a `SwapAllowlistExtension` and that the router is allowlisted can call `MetricOmmSimpleRouter.exactInputSingle` with that pool address. The router is a public, permissionless contract. The bypass is trivially repeatable in every block.

The pool admin is likely to allowlist the router because without it, allowlisted users cannot use the standard periphery — making the allowlist and the router mutually exclusive in practice, which is a configuration trap that pushes admins toward the vulnerable state.

---

### Recommendation

Pass the **originating user** rather than the direct caller as `sender` to the extension hook. Two concrete options:

**Option A — Accept an explicit `swapper` parameter from the caller.**
Add a `swapper` field to the swap parameters that the router populates with `msg.sender` before calling the pool. The pool forwards this value as `sender` to the extension. The extension then checks the actual user identity.

**Option B — Check `sender` in the router and revert before reaching the pool.**
Add a view call to `SwapAllowlistExtension.isAllowedToSwap(pool, msg.sender)` inside `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutput`, `exactOutputSingle`) before calling `pool.swap`. Revert with `NotAllowedToSwap` if the check fails. This keeps the pool-level hook as a backstop for direct calls while enforcing the correct identity at the router level.

---

### Proof of Concept

```
Setup:
  1. Deploy a pool with SwapAllowlistExtension configured.
  2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
     so that allowlisted users can use the standard periphery.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. Attacker (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <restricted pool>,
           tokenIn:   token0,
           recipient: attacker,
           amountIn:  X,
           ...
       })
  5. Router calls pool.swap(...) with msg.sender = router.
  6. Pool calls SwapAllowlistExtension.beforeSwap(sender=router, ...).
  7. Extension checks allowedSwapper[pool][router] == true → passes.
  8. Swap executes. Attacker receives token1 output from the restricted pool.

Result:
  The attacker, who was never allowlisted, successfully swaps in a pool
  the admin intended to restrict. Repeated swaps cause adverse-selection
  losses for LPs in the restricted pool.
```
