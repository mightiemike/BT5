### Title
Router `pay()` Consumes Unattributed Native ETH Balance for WETH Swaps, Enabling Theft of Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **entire** native ETH balance — when settling a WETH swap obligation. Because this balance is never attributed to a specific user or transaction, any ETH stranded on the router from a prior user's overpayment is silently consumed by the next WETH swap, transferring that ETH to the pool on behalf of a different user who pays nothing.

---

### Finding Description

`PeripheryPayments.pay()` contains the following WETH branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire router balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`nativeBalance` is `address(this).balance` — the router's total native ETH at that moment — not the ETH contributed by the current caller's `msg.value`. The router's `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does **not** block ETH arriving via `msg.value` in any of the four payable swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). [2](#0-1) 

When a user calls `exactInputSingle{value: X}(...)` with `amountIn = Y < X` and omits `refundETH()`, the swap callback consumes `Y` ETH and `X − Y` ETH remains on the router with no owner record. The next caller who issues any WETH swap with `msg.value = 0` will have their `pay()` call satisfied — fully or partially — from that stranded ETH. [3](#0-2) 

The callback that triggers `pay()` is `_justPayCallback`, which is reached from `metricOmmSwapCallback` for every single-hop exact-input and exact-output swap: [4](#0-3) 

The same path is reached for the terminal hop of a multi-hop exact-output swap via `_exactOutputIterateCallback`: [5](#0-4)

### Impact Explanation

**Direct loss of user principal (High).**

- **Victim (User A)** calls `exactInputSingle{value: 1 000}(...)` with `tokenIn = WETH`, `amountIn = 500`. The swap callback calls `pay()`, which deposits 500 ETH as WETH and transfers it to the pool. The remaining 500 ETH is stranded on the router with no refund.
- **Attacker (User B)** calls `exactInputSingle{value: 0}(...)` with `tokenIn = WETH`, `amountIn = 500`. Inside `pay()`, `nativeBalance = 500 ≥ value = 500`, so the router deposits User A's 500 ETH as WETH and transfers it to the pool. User B's `safeTransferFrom` is never called — User B pays **zero WETH** and receives the full swap output.
- User A loses 500 ETH; User B receives a free swap. The pool's accounting is correct (it received the right WETH amount), so the loss is entirely borne by User A.

The `refundETH()` helper compounds this: it is a public, caller-chosen-recipient function that sweeps the router's entire ETH balance to `msg.sender`, giving any third party a direct theft path for the same stranded ETH. [6](#0-5) 

---

### Likelihood Explanation

**Medium-to-High.**

- All four swap entry points are `payable` and accept arbitrary `msg.value`. Users routinely send a safety-margin ETH amount for exact-output swaps where the exact input is unknown until the callback fires.
- The protocol's own test suite demonstrates the mixed native+WETH pattern (`test_mixedNativeAndWeth_exactInputSingle_wethForToken`, `test_mixedNativeAndWeth_exactOutputSingle_wethForToken`), confirming the design intentionally allows `msg.value > amountIn`. [7](#0-6) 

- `refundETH()` is a separate, optional call. Users who call `exactInputSingle` or `exactOutputSingle` directly (not via `multicall`) have no automatic refund mechanism and are especially exposed.
- An attacker can monitor the router's ETH balance on-chain and front-run or immediately follow any transaction that leaves ETH stranded.

---

### Recommendation

1. **Track per-transaction ETH in transient storage.** At the entry of each payable swap function, record `msg.value` in a transient slot. In `pay()`, consume only up to that recorded amount rather than `address(this).balance`.
2. **Auto-refund excess ETH.** After each swap function completes, transfer any remaining `address(this).balance` back to `msg.sender` (or require callers to use `multicall` + `refundETH()`).
3. **Restrict `refundETH()` to the original depositor.** Either record the depositor in transient storage and enforce it in `refundETH()`, or remove the public helper and rely on the auto-refund above.

### Proof of Concept

```
State before:
  router.balance = 0
  User A WETH balance = 0
  User B WETH balance = 1000

Step 1 — User A overpays:
  User A calls exactInputSingle{value: 1000}(tokenIn=WETH, amountIn=500, ...)
  → pay() sees nativeBalance=1000 >= value=500
  → deposits 500 ETH as WETH, transfers to pool
  → 500 ETH remains on router (no refundETH call)

State after step 1:
  router.balance = 500 ETH  (stranded, no owner record)
  User A net loss so far: 1000 ETH sent, 500 ETH consumed, 500 ETH stranded

Step 2 — Attacker exploits:
  User B calls exactInputSingle{value: 0}(tokenIn=WETH, amountIn=500, ...)
  → pay() sees nativeBalance=500 >= value=500
  → deposits User A's 500 ETH as WETH, transfers to pool
  → safeTransferFrom(User B, pool, 0) — never called
  → User B pays 0 WETH, receives full swap output

Final state:
  router.balance = 0
  User A: lost 1000 ETH total (500 for own swap + 500 stolen)
  User B: received swap output, paid nothing
  Pool: received correct WETH amount — no pool-side anomaly
```
