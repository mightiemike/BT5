### Title
`DepositAllowlistExtension` checks position `owner` instead of actual caller `sender`, allowing any address to bypass the deposit guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and gates only on `owner` (the position owner). Because `MetricOmmPool.addLiquidity` is an open operator-pattern function — any caller may add liquidity for any `owner` — a non-allowlisted address can trivially bypass the guard by naming an allowlisted address as `owner`. The parallel `SwapAllowlistExtension` correctly gates on `sender`; the deposit extension does not.

### Finding Description

**Pool entry point — operator pattern is open:**

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` from any `msg.sender` and passes both to the extension:

```solidity
// MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

**Extension hook — `sender` is silently dropped:**

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The first parameter (`sender`) is unnamed and never read. Only `owner` is checked. [2](#0-1) 

**Contrast with `SwapAllowlistExtension` — which correctly gates on `sender`:**

```solidity
// SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

**Periphery adder confirms any caller can name any owner:**

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` only validates `owner != address(0)` and then calls the pool with `positionOwner = owner`, `payer = msg.sender`. No check that `msg.sender` is allowlisted. [4](#0-3) 

**Extension interface confirms both addresses are available:**

`IMetricOmmExtensions.beforeAddLiquidity` receives `sender` and `owner` as distinct parameters; the extension has all the information needed to check the actual caller but ignores it. [5](#0-4) 

### Impact Explanation

The `DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool" and its storage mapping is named `allowedDepositor`. The intent is to restrict which addresses may deposit. Because the check is applied to `owner` rather than `sender`, any non-allowlisted address can deposit real tokens into the pool by naming an allowlisted address as the position owner. The pool admin's access-control boundary is fully bypassed by an unprivileged caller without any special privilege or malicious setup. [6](#0-5) 

### Likelihood Explanation

Exploitation requires only that the attacker know one allowlisted address (publicly readable from `allowedDepositor`) and have tokens approved to `MetricOmmPoolLiquidityAdder`. No flash loan, no privileged role, no special timing. Any non-allowlisted address can execute this in a single transaction.

### Recommendation

Name and check `sender` instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the caller and the owner, check both. Update `setAllowedToDeposit` / `allowedDepositor` naming to reflect whichever semantics are chosen.

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  alice allowlisted: allowedDepositor[pool][alice] = true
  bob NOT allowlisted

Attack (bob calls MetricOmmPoolLiquidityAdder):
  bob.approve(liquidityAdder, large_amount)
  liquidityAdder.addLiquidityExactShares(pool, alice /*owner*/, salt, deltas, max0, max1, "")

Pool flow:
  pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)
    → _beforeAddLiquidity(liquidityAdder /*sender*/, alice /*owner*/, ...)
      → extension.beforeAddLiquidity(liquidityAdder, alice, ...)
        → allowedDepositor[pool][alice] == true  ✓  (guard passes)
    → LiquidityLib.addLiquidity(ctx, alice, ...)   // position credited to alice
    → callback → liquidityAdder pulls tokens from bob

Result:
  bob (non-allowlisted) has deposited tokens into the pool
  alice holds a position she did not initiate
  DepositAllowlistExtension guard is fully bypassed
``` [2](#0-1) [7](#0-6) [4](#0-3)
