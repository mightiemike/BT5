### Title
`DepositAllowlistExtension` gates on `owner` (position owner) instead of `sender` (actual depositor/caller), allowing any unprivileged address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, its `beforeAddLiquidity` implementation silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead checks the caller-supplied `owner` (the position owner). Because `owner` is a free parameter that any caller can set to any address, any unprivileged address can bypass the allowlist by naming an already-allowed address as the position owner.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

- `sender` = `msg.sender` — the address that actually called `addLiquidity` and will be asked to pay tokens via the callback.
- `owner` — a caller-supplied argument that records who owns the resulting position. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops `sender` (first parameter unnamed `address,`) and checks only `owner`: [3](#0-2) 

Because `owner` is a free parameter supplied by the caller, any address — including one that is **not** on the allowlist — can call:

```solidity
pool.addLiquidity(allowedAddress, salt, deltas, callbackData, extensionData);
```

The extension evaluates `allowedDepositor[pool][allowedAddress]` → `true` and does not revert. The pool then calls back the **actual caller** (the unauthorized address) to pull tokens, and the position is recorded under `allowedAddress`. The guard is fully bypassed.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), not the `recipient`: [4](#0-3) 

The asymmetry confirms the deposit extension is checking the wrong actor.

---

### Impact Explanation

**Allowlist guard is fully bypassed for deposits.** Any address — regardless of allowlist status — can inject liquidity into a restricted pool by naming any already-allowed address as `owner`. Consequences:

1. **Compliance/KYC bypass**: Pools configured for permissioned LP sets (e.g., regulated venues) accept liquidity from unauthorized parties.
2. **Pool state manipulation**: The unauthorized depositor controls which bins receive liquidity, moving `curBinIdx`/`curPosInBin` and altering bin balances. This can affect swap pricing and, if `OracleValueStopLossExtension` is also configured, manipulate per-bin value metrics to suppress or prematurely trigger the stop-loss guard — blocking legitimate LP withdrawals.
3. **LP dilution**: Unauthorized liquidity added at favorable prices dilutes existing LPs' share of future fee income.

The unauthorized depositor permanently loses the deposited tokens (only `owner` can call `removeLiquidity`), so this is not a direct theft path for the attacker, but it is a reachable, unprivileged bypass of an admin-configured guard with fund-impacting pool-state consequences.

---

### Likelihood Explanation

- Requires no special role or privilege — any EOA or contract can call `pool.addLiquidity` directly.
- The bypass is trivially constructed: pick any address already on the allowlist as `owner`.
- The pool does not verify that `msg.sender == owner` for `addLiquidity` (only `removeLiquidity` enforces that). [5](#0-4) 

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns with the contract's own NatSpec ("Gates `addLiquidity` by **depositor** address") and mirrors the correct pattern used in `SwapAllowlistExtension` (which checks `sender`, not `recipient`).

---

### Proof of Concept

```solidity
// Setup: pool with DepositAllowlistExtension; only `allowedLP` is on the allowlist.
// `attacker` is NOT on the allowlist.

// Attacker calls addLiquidity directly, naming allowedLP as owner.
vm.prank(attacker);
pool.addLiquidity(
    allowedLP,          // owner — checked by extension → passes
    salt,
    deltas,
    callbackData,       // attacker's callback pays tokens
    extensionData
);
// Extension checks allowedDepositor[pool][allowedLP] == true → no revert.
// Pool pulls tokens from attacker via callback.
// Position recorded under allowedLP.
// Allowlist guard fully bypassed; attacker injected liquidity into a restricted pool.
``` [3](#0-2) [6](#0-5)
