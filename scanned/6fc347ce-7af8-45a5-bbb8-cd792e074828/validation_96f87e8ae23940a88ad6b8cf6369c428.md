### Title
`SwapAllowlistExtension.beforeSwap` Checks the Router Address Instead of the Actual User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` gates curated-pool swaps by checking the `sender` argument passed by the pool. Because `MetricOmmPool.swap()` passes `msg.sender` (the direct caller of the pool) as `sender`, any swap routed through `MetricOmmSimpleRouter` presents the **router's address** to the extension instead of the actual user's address. If the pool admin allowlists the router to enable standard periphery access, every user—including those not individually allowlisted—can bypass the per-user gate.

---

### Finding Description

**Step 1 – Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`_beforeSwap` forwards that value verbatim as the `sender` parameter of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

**Step 2 – The router is `msg.sender` of `pool.swap()`.**

Every entry-point in `MetricOmmSimpleRouter` calls `pool.swap()` directly, so the pool sees `msg.sender = router`: [3](#0-2) [4](#0-3) 

**Step 3 – The extension checks the router address, not the user.**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [5](#0-4) 

The extension therefore asks: *"Is the router allowlisted for this pool?"* — not *"Is the actual user allowlisted?"*

**Step 4 – Contrast with `DepositAllowlistExtension`.**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (the second parameter), which is the actual position owner supplied by the caller and forwarded unchanged through the pool. This correctly identifies the economic actor regardless of which periphery contract is used: [6](#0-5) 

The swap extension uses `sender` (first parameter = direct pool caller), while the deposit extension uses `owner` (second parameter = actual beneficiary). This asymmetry is the root cause.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (the natural action to let allowlisted users use the standard periphery) inadvertently opens the gate to **all** users. Any address can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and the extension will approve the swap because it sees `sender = router`, which is allowlisted. The per-user curation policy is completely nullified. This is an admin-boundary break: an unprivileged path (`MetricOmmSimpleRouter`) bypasses the pool admin's configured access control.

Conversely, if the pool admin does **not** allowlist the router, allowlisted users cannot use the router at all — the extension rejects every router-mediated swap — making the standard periphery unusable for curated pools.

---

### Likelihood Explanation

The trigger is a pool admin who:
1. Deploys a pool with `SwapAllowlistExtension` to restrict trading to specific addresses, and
2. Allowlists the router so those addresses can use the standard periphery interface.

Both steps are natural and expected for a production curated pool. The pool admin has no reason to suspect that allowlisting the router opens the gate to all users, because the extension's documented purpose is per-user gating. No attacker privilege is required beyond being able to call the public router.

---

### Recommendation

The `SwapAllowlistExtension` must check the **actual economic actor**, not the direct pool caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a coordinated change to the router and extension.

2. **Check `recipient` instead of `sender`**: For swaps, `recipient` is the address that receives output tokens and is the closest proxy for the actual user. However, `recipient` can also be a third-party address, so this is only correct if the pool admin's policy is to gate by output recipient.

3. **Require direct pool calls for curated pools**: Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the extension level by reverting when `sender` is a known router address.

The cleanest long-term fix is approach 1, mirroring how `DepositAllowlistExtension` correctly uses `owner` (the caller-supplied beneficiary) rather than `sender` (the direct pool caller).

---

### Proof of Concept

```solidity
// Setup: pool admin creates a curated pool
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// Admin allowlists alice (the intended user)
ext.setAllowedToSwap(pool, alice, true);
// Admin also allowlists the router so alice can use the standard interface
ext.setAllowedToSwap(pool, address(router), true);

// Attack: charlie (not allowlisted) swaps through the router
vm.prank(charlie);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: charlie,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// ✓ Succeeds: extension checks allowedSwapper[pool][router] == true
// ✗ Expected: revert NotAllowedToSwap because charlie is not allowlisted
```

The extension receives `sender = address(router)`, finds it allowlisted, and returns the success selector. Charlie's swap executes in a pool that was configured to exclude him.

    }
