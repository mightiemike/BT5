### Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. Because `MetricOmmPool.swap` always passes its own `msg.sender` as `sender`, any swap routed through `MetricOmmSimpleRouter` presents the router's address to the extension — not the actual end-user's address. If the pool admin allowlists the router (the natural action to let allowlisted users access the pool through the standard periphery), the gate is silently opened to every user on-chain.

---

### Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender`.** [1](#0-0) 

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` is the router contract, not the end-user.

**Step 2 — `ExtensionCalling` forwards that value verbatim as the `sender` argument.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`.** [3](#0-2) 

`sender` here is the router address. The extension has no visibility into the actual end-user (`msg.sender` of the router call).

**Step 4 — The router does not inject the real caller into `extensionData`.** [4](#0-3) 

`params.extensionData` is user-supplied and forwarded opaquely; the router never encodes `msg.sender` into it, and `SwapAllowlistExtension` ignores `extensionData` entirely.

**Resulting invariant break.** The pool admin intends to allowlist specific addresses (e.g. KYC'd counterparties). To let those users access the pool through the standard router, the admin must call `setAllowedToSwap(pool, router, true)`. From that moment, `allowedSwapper[pool][router] == true` satisfies the check for every caller of the router, regardless of their identity. The allowlist is fully bypassed.

Contrast with `DepositAllowlistExtension`, which correctly checks the `owner` argument (the position beneficiary), not `sender`: [5](#0-4) 

The asymmetry confirms the swap-side check is the defective one.

---

### Impact Explanation

Any user can swap against a pool whose admin intended to restrict access to a specific set of counterparties, simply by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`). The pool executes at the live oracle bid/ask, so unauthorized swappers can extract value from LPs at oracle-fair prices without the LP's consent. For pools designed for institutional or permissioned flow (e.g. RFQ-style pools, compliance-gated pools), this constitutes a direct breach of the access-control invariant and exposes LP principal to unintended counterparty risk.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who deploys a swap-allowlisted pool and wants allowlisted users to access it through the standard periphery will naturally allowlist the router. The bypass requires no special knowledge beyond knowing the router address and calling it normally. It is reachable by any unprivileged on-chain actor.

---

### Recommendation

Pass the actual end-user identity through the extension pipeline. Two options:

1. **Preferred — encode caller in `extensionData` at the router level.** The router encodes `msg.sender` into `extensionData` before forwarding to the pool; `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of (or in addition to) `sender`.

2. **Alternative — add a `realSender` field to the `beforeSwap` signature.** The pool stores the original `msg.sender` in transient storage before any callback and exposes it to extensions, similar to how Uniswap v4 passes `msgSender` through hook calls.

Either approach must be applied consistently across `exactInputSingle`, `exactInput`, `exactOutput`, and `exactOutputSingle` in the router.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, router, true)
   — intended to let allowlisted users swap via the router.
3. Attacker (not in allowlist) calls:
     router.exactInputSingle({pool: pool, ...})
4. Pool calls _beforeSwap(msg.sender=router, ...)
5. Extension checks allowedSwapper[pool][router] == true → passes.
6. Attacker's swap executes at oracle price against LP funds.
7. Allowlist invariant is broken; no allowlisted-user check was ever performed.
``` [6](#0-5) [1](#0-0) [7](#0-6)
