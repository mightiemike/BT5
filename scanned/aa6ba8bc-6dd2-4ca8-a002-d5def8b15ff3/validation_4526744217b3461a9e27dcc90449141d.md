### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual User, Allowing Any User to Bypass Per-User Swap Restrictions - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. If the pool admin allowlists the router (the natural step to enable router-mediated swaps), every user — including those not individually allowlisted — can bypass the per-user curation policy by routing through the router.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` then encodes that `sender` into the call to each extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` enforces the allowlist against that `sender` value: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly — making the router the `msg.sender` the pool observes: [4](#0-3) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and the outer call of `exactOutput`: [5](#0-4) 

For `exactOutput` multi-hop, intermediate hops are triggered from inside `_exactOutputIterateCallback`, where `msg.sender` is the previous pool — so those pools see a pool address as `sender`, not the user or the router: [6](#0-5) 

The result: the extension never sees the actual user's address when the swap is router-mediated. The allowlist check resolves to `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

### Impact Explanation

A pool admin who wants to restrict swaps to a curated set of counterparties (e.g., KYC-verified addresses, specific market makers, or regulated participants) configures `SwapAllowlistExtension` and allowlists those addresses. To also support the standard periphery router — which is the primary UX path — the admin must additionally allowlist the router address. Once the router is allowlisted, the per-user restriction is completely void: any address can call `router.exactInputSingle` and the extension passes because it sees `sender = router`. Unauthorized users can trade at oracle prices in a pool that was supposed to be restricted, extracting value from LPs or violating the pool's intended access policy. This is a direct bypass of a live loss-prevention and access-control mechanism with fund-impacting consequences.

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` for per-user curation and also wants to support the standard `MetricOmmSimpleRouter` will face this issue. Allowlisting the router is the natural and expected configuration step for router support. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — only a standard router call from any EOA.

### Recommendation

The `sender` argument passed to `beforeSwap` should represent the economic actor, not the intermediary. Two complementary fixes:

1. **In the router**: pass the original `msg.sender` (the user) as the `sender` argument to `pool.swap()` rather than relying on the pool to use `msg.sender`. This requires adding a `sender` parameter to the pool's `swap` interface, or having the router store the user in transient storage and expose it via a callback the extension can query.

2. **In `SwapAllowlistExtension`**: document clearly that `sender` is the direct caller of `pool.swap()`, not the end user, and that allowlisting the router opens the pool to all router users. Until the interface is changed, pool admins must not simultaneously allowlist the router and rely on per-user curation.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists user1 directly: allowedSwapper[pool][user1] = true
  - Pool admin allowlists router for router support: allowedSwapper[pool][router] = true

Attack (user2, not individually allowlisted):
  1. user2 calls router.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) — pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → TRUE
  5. Swap executes for user2 despite user2 not being individually allowlisted

Direct call (correctly blocked):
  1. user2 calls pool.swap(...) directly
  2. Pool calls _beforeSwap(sender=user2, ...)
  3. SwapAllowlistExtension checks allowedSwapper[pool][user2] → FALSE → revert NotAllowedToSwap

Conclusion: the router allowlist entry acts as a universal bypass for all users,
defeating the per-user curation the pool admin intended to enforce.
