### Title
`addLiquidity` and `removeLiquidity` bypass the `whenNotPaused` guard, allowing deposits into paused pools — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool` exposes a `pauseLevel` state variable and a `whenNotPaused` modifier that is intended to halt pool operations when the pool is paused by the admin (level 1) or protocol (level 2). However, the modifier is applied **only** to `swap`, leaving `addLiquidity` and `removeLiquidity` completely unguarded. This is a direct structural analog to the external report: a shutdown/pause flag is set but does not actually prevent the fund-moving operations it is supposed to block.

---

### Finding Description

`MetricOmmPool` defines a two-level pause mechanism:

```solidity
// metric-core/contracts/MetricOmmPool.sol  line 72
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
uint8 internal pauseLevel;
``` [1](#0-0) 

The guard is implemented as:

```solidity
modifier whenNotPaused() {
    _checkNotPaused();
    _;
}
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) [3](#0-2) 

`swap` correctly applies the modifier:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
``` [4](#0-3) 

But `addLiquidity` and `removeLiquidity` do **not**:

```solidity
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
``` [5](#0-4) [6](#0-5) 

The extension hooks (`_beforeAddLiquidity`, `_afterAddLiquidity`) are still called, but they are optional per-pool configurations — they are not a substitute for the core pause guard. A pool with no deposit-allowlist extension configured has zero protection on `addLiquidity` when paused. [7](#0-6) 

---

### Impact Explanation

When the admin or protocol pauses a pool (e.g., because the price provider is suspected to be compromised, returning stale or manipulated prices), users can still call `addLiquidity` and deposit principal into the pool. Once the pool is unpaused — even after the triggering issue is resolved — the newly deposited liquidity is immediately exposed to swaps. If the pause was triggered by a price oracle issue that is not fully remediated before unpausing, those deposits can be drained by swaps executing at bad bid/ask prices. This constitutes a direct loss of user principal deposited during the pause window.

---

### Likelihood Explanation

- The pool must be paused by the admin or protocol (a semi-trusted, reachable action).
- Users interacting with the pool during the pause window (e.g., via a router or liquidity adder that does not itself check `pauseLevel`) will unknowingly deposit into a paused pool.
- The `MetricOmmPoolLiquidityAdder` periphery contract calls `addLiquidity` directly and performs no pause check of its own, making this reachable by ordinary users.
- No attacker privilege is required; any user calling `addLiquidity` during a pause triggers the exposure.

---

### Recommendation

Apply `whenNotPaused` to both `addLiquidity` and `removeLiquidity` (or at minimum to `addLiquidity`, since allowing withdrawals during a pause is a common and defensible design choice):

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

This mirrors the fix recommended in the external report: enforce the shutdown/pause flag at the entry point of every fund-moving operation, not just swaps.

---

### Proof of Concept

1. Admin calls `factory.setPauseLevel(pool, 1)` — pool is paused due to a suspected oracle issue.
2. `pool.swap(...)` reverts with `PoolPaused()`. ✓
3. `pool.addLiquidity(owner, salt, deltas, callbackData, extensionData)` — **succeeds**, depositing user tokens into the paused pool. ✗
4. Admin resolves (or believes they have resolved) the oracle issue and calls `factory.setPauseLevel(pool, 0)`.
5. An attacker immediately calls `pool.swap(...)` at the still-bad oracle price, draining the liquidity deposited in step 3.
6. The user who deposited in step 3 suffers a direct loss of principal. [8](#0-7) [9](#0-8)
