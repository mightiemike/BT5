### Title
`PriceVelocityGuardExtension.beforeSwap` resets `lastMidPriceX64` on every swap, allowing the per-block velocity cap to be bypassed through intra-block oracle stepping — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` is supposed to cap how fast the oracle mid-price can move between blocks. The guard stores `lastMidPriceX64` and checks each swap's mid-price against it. However, the stored reference price is unconditionally overwritten **before** the velocity check on every swap. When the oracle price changes between two swap transactions in the same block — which is possible with Pyth's push model — each swap resets the reference anchor to the new mid, giving the next swap a fresh, tighter baseline. An attacker can chain N small swaps in one block, each one stepping the reference price by at most `maxChangePerBlockE18`, and achieve a cumulative price movement of N × `maxChangePerBlockE18` while every individual check passes. This is the direct analog of the OpenTerm "pulse" pattern: briefly satisfying the guard condition resets the state, allowing indefinite bypass of the intended cap.

---

### Finding Description

In `PriceVelocityGuardExtension.beforeSwap`, lines 57–58 write the new mid-price and block number into storage **before** the velocity check at lines 60–75:

```solidity
// PriceVelocityGuardExtension.sol lines 53-74
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // ← reference reset BEFORE check
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;
        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);
        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);
        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
    }
}
```

The pool fetches a fresh oracle quote on every `swap()` call (`MetricOmmPool.sol` line 228: `_getBidAndAskPriceX64()`). With Pyth's push model, anyone holding a valid signed price update can push it to the on-chain oracle between two swap transactions in the same block. Because `blockDiff = block.number - prevBlock = 0` for same-block swaps, the allowed squared change is `maxChange² × 1`. Each swap is checked only against the **previous swap's** mid-price, not the block-start price.

**Attack sequence (single block):**

| Step | Action | Guard sees | Passes? | New `lastMidPriceX64` |
|------|--------|-----------|---------|----------------------|
| 1 | Push Pyth update P0→P1 (+1%) | P1 vs P0 = 1% | ✓ | P1 |
| 2 | Swap at P1 | — | — | P1 |
| 3 | Push Pyth update P1→P2 (+1%) | P2 vs P1 = 1% | ✓ | P2 |
| 4 | Swap at P2 | — | — | P2 |
| … | … | … | … | … |
| N | Swap at P_N | P_N vs P_{N-1} = 1% | ✓ | P_N |

Total price movement in one block: N × 1%, while the guard was configured to allow only 1% per block.

The analog to the OpenTerm "pulse":
- **OpenTerm**: briefly restore liquidity → `delinquencyStartTime` resets to 0 → immediately borrow again → fresh grace window.
- **PriceVelocityGuard**: each swap updates `lastMidPriceX64` to the current oracle mid → next swap checks only against this new anchor → fresh velocity window from the new price.

---

### Impact Explanation

The velocity guard is the LP's primary protection against oracle-speed manipulation and adverse selection. By stepping through intermediate Pyth price updates in a single block, an attacker can execute swaps at a final price that is N × `maxChangePerBlockE18` away from the block-start price — a movement the guard was explicitly configured to block. LPs suffer direct principal loss through adverse selection: they are forced to trade at prices that moved far faster than the configured safety envelope, with no opportunity to react or withdraw.

**Severity: High** — direct loss of LP principal through a bypassed core protection hook.

---

### Likelihood Explanation

- Pyth's push model is permissionless: any actor can submit a valid signed price update obtained from Pyth's public price service.
- When the real market price moves significantly within a block's time window (~12 s on Ethereum, ~2 s on L2s), multiple valid intermediate Pyth updates exist and are publicly available.
- The attacker needs no privileged role, no special token, and no flash loan — only the ability to call `swap()` and push Pyth updates in sequence within one block, which is achievable via a single multicall or a purpose-built contract.
- On L2s (Arbitrum, Base) where blocks are ~250 ms, the attack window is even wider because many Pyth updates can accumulate per block.

---

### Recommendation

Track the **block-start** reference price separately from the per-swap reference price. Only advance `lastMidPriceX64` once per block (when `block.number > lastUpdateBlock`). All velocity checks within the same block must compare against the price recorded at the **start** of that block, not the price from the most recent swap:

```solidity
// Only update the reference when entering a new block
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// Always check against the block-start reference (prevMid)
```

This ensures that no matter how many swaps occur in a single block, the velocity check always measures the total movement from the block-start price, closing the stepping bypass.

### Proof of Concept

**Setup:**
- Pool with `PriceVelocityGuardExtension`, `maxChangePerBlockE18 = 1e16` (1%).
- Oracle (Pyth) at block N−1: mid-price P₀ = 100 USDC/ETH. `lastMidPriceX64 = P₀`.
- Real market price moves to P₁₀ ≈ 110 USDC/ETH within the same second (10% jump).
- Attacker collects 10 valid signed Pyth updates: P₀→P₁ (1%), P₁→P₂ (1%), …, P₉→P₁₀ (1%).

**Execution (all in block N):**
```
tx1: pushPythUpdate(P₁)          // oracle now at 101
tx2: pool.swap(...)               // beforeSwap: |P₁-P₀|/P₀ = 1% ≤ 1% ✓, lastMidPriceX64 = P₁
tx3: pushPythUpdate(P₂)          // oracle now at 102.01
tx4: pool.swap(...)               // beforeSwap: |P₂-P₁|/P₁ = 1% ≤ 1% ✓, lastMidPriceX64 = P₂
...
tx19: pushPythUpdate(P₁₀)        // oracle now at ~110
tx20: pool.swap(...)              // beforeSwap: |P₁₀-P₉|/P₉ = 1% ≤ 1% ✓, lastMidPriceX64 = P₁₀
```

**Result:** 10% total price movement executed in one block. The guard allowed it because each individual step was ≤ 1%. Without the stepping, a single swap from P₀ to P₁₀ would have triggered `PriceVelocityExceeded` (10² = 100 > 1² × 1 = 1). LPs sold ETH at prices ranging from P₁ to P₁₀ while the market had already moved to P₁₀, suffering the full adverse-selection loss the guard was designed to prevent. [1](#0-0) [2](#0-1) [3](#0-2)
