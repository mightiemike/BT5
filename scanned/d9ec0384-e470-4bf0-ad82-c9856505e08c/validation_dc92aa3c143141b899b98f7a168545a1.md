### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position Owner Instead of Actual Depositor, Allowing Deposit Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` **by depositor address**. However, its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual caller who transfers tokens) and instead validates the `owner` argument (the LP position recipient). Because `owner` is a free caller-controlled parameter in `MetricOmmPool.addLiquidity`, any address not on the allowlist can bypass the guard by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook: [1](#0-0) 

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension: [2](#0-1) 

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but names the first parameter `_` (discarded) and checks only `owner`: [3](#0-2) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`owner` is a free parameter supplied by the caller of `addLiquidity`. Any address — including one not on the allowlist — can call:

```solidity
pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData);
```

The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true` and permits the deposit. The actual token transfer originates from the unauthorized `msg.sender` via the swap callback, so the allowlist is fully circumvented.

The allowlist setter correctly maps `depositor → bool` per pool: [4](#0-3) 

but the enforcement hook reads the wrong key, making the mapping irrelevant for access control.

---

### Impact Explanation

Any address excluded from the deposit allowlist can deposit tokens into the pool by specifying any allowlisted address as `owner`. The deposited tokens are credited to the allowlisted `owner`'s LP position, not the actual depositor. The depositor cannot unilaterally recover the tokens because `removeLiquidity` enforces `msg.sender == owner`: [5](#0-4) 

However, if the attacker controls an allowlisted contract (e.g., a multisig or a purpose-built wrapper they deployed and had the pool admin allowlist), they can complete the round-trip autonomously: deposit via the unauthorized EOA → withdraw via the controlled allowlisted contract → sweep proceeds. Even without that, the allowlist is rendered meaningless as a deposit gate: any actor can inject liquidity into the pool at will, defeating regulatory, compliance, or pool-composition controls the pool admin intended to enforce.

---

### Likelihood Explanation

Exploitation requires no special privilege, no price manipulation, and no flash loan. Any caller of `addLiquidity` can supply an arbitrary `owner`. The allowlisted address need not cooperate for the deposit to succeed; cooperation is only needed to recover funds. Likelihood is **high** for the bypass itself; the attacker's ability to profit depends on whether they control an allowlisted address.

---

### Recommendation

Name and use the `sender` parameter (the actual depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`sender` is `msg.sender` of the originating `addLiquidity` call — the address that will transfer tokens via the callback — which is the correct entity to gate.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` configured; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. `beforeAddLiquidity` is invoked with `sender = Bob`, `owner = alice`.
5. The hook discards `Bob` and checks `allowedDepositor[pool][alice]` → `true`. No revert.
6. Bob's tokens are pulled via the callback; Alice's LP position is credited.
7. The deposit allowlist has been bypassed without any privileged access.
