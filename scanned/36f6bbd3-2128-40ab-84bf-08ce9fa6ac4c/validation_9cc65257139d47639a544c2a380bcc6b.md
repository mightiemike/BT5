### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router address**, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. If the pool admin allowlists the router to enable standard-UX access for their curated users, the allowlist is completely bypassed for every user on the network.

### Finding Description

**Allowlist check in `SwapAllowlistExtension`:** [1](#0-0) 

The extension receives `sender` from the pool and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Pool passes `msg.sender` as `sender` to the extension:** [2](#0-1) [3](#0-2) 

**Router calls `pool.swap()` directly — the original user's address is only stored in transient storage for the payment callback, never forwarded to the pool as the swap initiator:** [4](#0-3) 

The router stores `msg.sender` in transient storage via `_setNextCallbackContext` for token-pull purposes only. The actual `pool.swap()` call has `msg.sender = router`. The pool then passes `router` as `sender` to every `beforeSwap` extension hook.

**Result:** `allowedSwapper[pool][router]` is checked, not `allowedSwapper[pool][end_user]`.

### Impact Explanation

Two fund-impacting outcomes arise:

1. **Allowlist bypass (High):** A pool admin who allowlists the router address — a natural step to enable the standard periphery UX for their curated users — inadvertently opens the pool to every user on the network. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle/exactInput/exactOutput` and trade against a pool that was supposed to be restricted. LPs in a private/institutional pool are exposed to unauthorized counterparties, enabling adverse selection and direct LP principal loss.

2. **Allowlisted users blocked from router (Medium):** If the pool admin allowlists individual user addresses (not the router), those users cannot swap through the router at all — the extension sees the router address and reverts `NotAllowedToSwap`. The standard periphery path is broken for every legitimately allowlisted user, forcing direct pool calls and breaking the intended UX.

The first scenario matches the allowed impact gate: an unprivileged path (`MetricOmmSimpleRouter`) bypasses an admin-configured access control boundary (`SwapAllowlistExtension`), with direct LP fund exposure as the consequence.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool admin who deploys a `SwapAllowlistExtension` and wants their allowlisted users to use the standard router will naturally allowlist the router address. The bypass is then immediately available to every user. No special knowledge or privileged access is required — any address can call the public router functions.

### Recommendation

The `SwapAllowlistExtension` must gate the **economic actor** (the end user), not the intermediary. Two options:

1. **Pass the original initiator through `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and checks that address instead of `sender`. This requires a convention between the router and the extension.

2. **Check `sender` only for direct pool calls; require the router to forward the user identity:** Add a dedicated field or a standardized `extensionData` prefix that the router always populates with the originating user, and have the extension decode it when present.

The core invariant to restore: the allowlist must check the same identity regardless of whether the swap arrives via direct pool call or via the router.

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to let allowlisted users use the standard UX.
3. Attacker (not in any allowlist) calls:
     MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
4. pool.swap() is called with msg.sender = router.
5. _beforeSwap passes sender = router to SwapAllowlistExtension.
6. Extension checks allowedSwapper[pool][router] == true → passes.
7. Attacker's swap executes on the restricted pool.
   LP funds are exposed to an unauthorized counterparty.
```

The bypass requires zero privileged access and is reachable through the public `MetricOmmSimpleRouter` entrypoints (`exactInputSingle`, `exactInput`, `exactOutput`, `exactOutputSingle`). [5](#0-4) [6](#0-5) [2](#0-1)
