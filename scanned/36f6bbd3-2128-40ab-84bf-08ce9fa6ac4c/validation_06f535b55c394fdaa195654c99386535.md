### Title
SwapAllowlistExtension Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which equals `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the end user. If the pool admin allowlists the router (the natural step to let their allowlisted users use it), every unprivileged user can bypass the swap gate by routing through the public router contract.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any of the other `exact*` entry points), the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point the pool's `msg.sender` is the router contract, so `sender` forwarded to the extension is the router's address, not the end user's address. The extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

The pool admin must allowlist the router to let their approved users trade through it. Once the router is allowlisted, the check degenerates to a no-op for every caller of the public router, because the router address is always the one tested.

### Impact Explanation

Any unprivileged user can swap in a pool that is supposed to be restricted to a curated set of addresses. Consequences include:

- Unauthorized extraction of pool liquidity at oracle-anchored prices (direct loss of LP principal).
- Defeat of compliance or KYC-gated pool designs, which may expose the protocol to regulatory or contractual liability.
- Pool state (bin position, fee accrual) driven by actors the admin explicitly excluded.

### Likelihood Explanation

The trigger is fully unprivileged: any EOA or contract can call `MetricOmmSimpleRouter.exactInputSingle`. The precondition — the router being allowlisted — is the natural and expected admin action; without it, allowlisted users cannot use the router at all, making the allowlist and the router mutually exclusive. The admin is therefore forced into the vulnerable configuration to make the system usable.

### Recommendation

The extension must verify the original user, not the immediate pool caller. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted encoding convention.

2. **Check both `sender` and a user field**: Extend the `beforeSwap` signature (or a wrapper) so the router passes the real payer, and the extension checks that field when `sender` is a known router.

3. **Allowlist-per-caller model**: Instead of allowlisting the router as a single address, require the router to forward the real user identity in a verifiable way (e.g., a signed permit or a dedicated router that re-encodes the caller).

The simplest safe fix is to never allowlist the router as a blanket entry; instead, require direct pool calls for allowlisted users, and document this constraint explicitly in the extension's NatSpec.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  alice  → allowedSwapper[pool][alice]  = true   (legitimate user)
  router → allowedSwapper[pool][router] = true   (admin adds this so alice can use the router)
  bob    → allowedSwapper[pool][bob]   = false   (explicitly excluded)

Attack:
  bob calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, zeroForOne, amount, ...)
        pool.msg.sender = router
      → pool calls _beforeSwap(router, ...)
        → extension.beforeSwap(sender=router, ...)
          allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes for bob
      → pool calls router.metricOmmSwapCallback(...)
        → router pays from bob's tokens

Result: bob swaps successfully despite being explicitly excluded from the allowlist.
        LP funds flow to bob at oracle-anchored prices; allowlist invariant is broken.
