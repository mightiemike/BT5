### Title
`DepositAllowlistExtension` Checks LP Position `owner` Instead of Actual Depositor `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the allowlist against the LP position `owner` parameter rather than the actual caller (`sender`). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address distinct from `msg.sender`, any unprivileged address can add liquidity to a restricted pool by naming an already-allowed address as the position owner, completely defeating the guard.

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook: [1](#0-0) 

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `sender` = `msg.sender` — the address that actually calls `addLiquidity`, provides tokens via the swap callback, and is the true "depositor."
- `owner` — an arbitrary address supplied by the caller that receives the LP position shares.

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both: [2](#0-1) 

The extension hook receives `(sender, owner, …)` but silently discards `sender` (first parameter unnamed) and checks only `owner`: [3](#0-2) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The guard name (`setAllowedToDeposit`), the event (`AllowedToDepositSet`), and the NatSpec ("Gates `addLiquidity` by depositor address") all confirm the intent is to restrict the depositing party — `sender` — not the LP position recipient. [4](#0-3) 

### Impact Explanation

Any address can add liquidity to a pool whose admin configured a deposit allowlist by simply passing an allowed address as `owner`. The attacker provides the tokens (via the `metricOmmSwapCallback`), the allowed address receives the LP shares (and can later withdraw them), and the pool receives unauthorized liquidity. The pool admin's access control is entirely ineffective:

- Unauthorized parties can alter pool bin balances and `binTotals`, shifting the marginal price and affecting subsequent swap settlement for all LPs.
- Pools intended for permissioned participants (e.g., institutional or regulatory-restricted pools) have no enforceable deposit gate.
- The allowed `owner` receives unsolicited LP positions they did not initiate, which may create accounting or tax complications and cannot be refused.

### Likelihood Explanation

The bypass requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(allowedAddress, …)`. The only cost to the attacker is the token amount they deposit (which flows to the allowed owner's LP position). The attack is trivially repeatable.

### Recommendation

Change the allowlist check to validate `sender` (the first parameter) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swap initiator): [5](#0-4) 

### Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` configured; only `Alice` is in `allowedDepositor[P]`.
2. `Bob` (not in the allowlist) holds token0 and token1.
3. `Bob` calls `P.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. The pool calls `extension.beforeAddLiquidity(Bob, Alice, …)`.
5. The extension checks `allowedDepositor[P][Alice]` → `true` → no revert.
6. `LiquidityLib.addLiquidity` executes: Bob's tokens are pulled via `metricOmmSwapCallback`, LP shares are credited to Alice.
7. Bob has successfully added liquidity to a restricted pool. The allowlist guard is bypassed. [3](#0-2) [6](#0-5)
