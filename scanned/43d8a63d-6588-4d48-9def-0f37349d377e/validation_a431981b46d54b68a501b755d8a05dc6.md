The call chain is fully traceable. Here is the analysis:

**Call path:**

1. User → `MetricOmmSimpleRouter::exactInputSingle` → `IMetricOmmPoolActions(pool).swap(...)` — here `msg.sender` to the pool is the **router**
2. `MetricOmmPool::swap` calls `_beforeSwap(msg.sender, ...)` — `msg.sender` = router address
3. `ExtensionCalling::_beforeSwap` encodes `sender` = router address and calls the extension
4. `SwapAllowlistExtension::beforeSwap(sender=router, ...)` executes the check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = **router address** — not the original user.

**The identity confusion is real and structural.** The hook always checks whether the **router** is allowlisted, never the original EOA. This creates an irreconcilable dilemma for pool admins:

- If the router is **not** allowlisted: allowlisted users cannot swap through the router at all (DoS on the intended flow).
- If the router **is** allowlisted: any unprivileged user can bypass the per-user allowlist with a single transaction through the router.

**The "timed-threshold manipulation / two-transaction sequence" framing in the question is incorrect.** There are no thresholds, observations, or stale state involved. The bypass is a single-transaction structural flaw, not a race condition.

**Impact assessment:** The `SwapAllowlistExtension` is a live access-control gate. A pool admin who allowlists the router to enable router-mediated swaps for their allowlisted users inadvertently opens the gate to all users. This breaks the core invariant the extension is designed to enforce.

### Title
Router-mediated swaps pass the router's address as `sender` to `SwapAllowlistExtension::beforeSwap`, allowing any user to bypass per-user allowlists — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension::beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user swaps via `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the hook checks whether the **router** is allowlisted — not the original user. Any user can bypass a per-user allowlist by routing through the router if the router is allowlisted.

### Finding Description
In `MetricOmmPool::swap`, `_beforeSwap` is called with `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling::_beforeSwap` forwards this value unchanged as the first argument to the extension: [2](#0-1) 

`SwapAllowlistExtension::beforeSwap` then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = the immediate caller of the pool: [3](#0-2) 

When the router calls the pool, it passes no user identity — the pool only sees the router: [4](#0-3) 

### Impact Explanation
A pool admin who configures a per-user allowlist and also allowlists the router (to enable router-mediated swaps for their users) inadvertently grants swap access to all users. The allowlist is completely bypassed for any user who routes through `MetricOmmSimpleRouter`. This breaks the core access-control invariant of the extension and constitutes broken core pool functionality.

### Likelihood Explanation
Any pool using `SwapAllowlistExtension` with `allowAllSwappers = false` and `allowedSwapper[pool][router] = true` is immediately exploitable by any unprivileged user. The router is a standard periphery contract that users are expected to use, making this a realistic configuration.

### Recommendation
The extension must receive the original user's address, not the immediate pool caller. Options:
1. Pass the original user's address through `extensionData` and have the extension decode it (requires router cooperation and trust assumptions).
2. Have the router pass the original `msg.sender` as a verified parameter that the pool forwards to extensions as a separate `originator` field.
3. Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the factory level.

### Proof of Concept
```
1. Pool admin deploys pool with SwapAllowlistExtension, allowAllSwappers[pool] = false
2. Pool admin sets allowedSwapper[pool][alice] = true  (only Alice should swap)
3. Pool admin sets allowedSwapper[pool][router] = true (to let Alice use the router)
4. Attacker (Bob, not allowlisted) calls MetricOmmSimpleRouter::exactInputSingle with pool as target
5. Pool receives swap() call from router (msg.sender = router)
6. _beforeSwap passes sender = router to SwapAllowlistExtension
7. Hook checks allowedSwapper[pool][router] == true → passes
8. Bob's swap executes successfully, bypassing the allowlist
``` [5](#0-4)
