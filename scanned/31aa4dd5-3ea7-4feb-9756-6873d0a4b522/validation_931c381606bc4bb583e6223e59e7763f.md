### Title
`SwapAllowlistExtension` gates the router address instead of the originating user, allowing any actor to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the originating user. If the pool admin allowlists the router address (the natural step to enable router-mediated swaps for their curated users), every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

**Call chain:**

1. User (`charlie`, not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — `msg.sender` inside the pool is `address(router)`.
3. Pool calls `_beforeSwap(msg.sender, ...)` — passes `address(router)` as `sender`.
4. `ExtensionCalling._beforeSwap` encodes and dispatches `IMetricOmmExtensions.beforeSwap(sender=router, ...)` to the extension.
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]`. [1](#0-0) [2](#0-1) [3](#0-2) 

The pool passes `msg.sender` (the router) as `sender` to the extension. The extension then checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][charlie]`. [4](#0-3) 

The router never forwards the originating user's address to the pool or to `extensionData`; it only stores it in transient callback context for payment settlement.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and wants to support router-mediated swaps for their allowlisted users must allowlist the router address (`setAllowedToSwap(pool, router, true)`). Once the router is allowlisted, **any** address — including completely unprivileged actors — can call `router.exactInputSingle` and pass the allowlist check, because the extension sees `sender = router` and `allowedSwapper[pool][router] = true`.

LPs on curated pools rely on the allowlist to ensure only trusted counterparties trade against their liquidity. Bypassing it exposes LP principal to adverse selection from unintended actors, constituting a direct loss of LP value and a broken core pool invariant.

The alternative failure mode (router not allowlisted) means allowlisted users cannot use the router at all — broken core swap functionality for the intended user set.

---

### Likelihood Explanation

The trigger is a valid, non-privileged user action: calling the public `MetricOmmSimpleRouter.exactInputSingle`. The only precondition is that the pool admin has allowlisted the router, which is the natural and expected configuration for any curated pool that intends to support the protocol's own periphery. No malicious setup, non-standard tokens, or privileged access is required beyond the normal pool admin configuration.

---

### Recommendation

The `SwapAllowlistExtension` should not rely on the `sender` argument (which is the immediate pool caller) to identify the economic actor. Two viable fixes:

1. **Extension-data forwarding**: The router encodes `msg.sender` into `extensionData` for each hop; the extension decodes and checks it. This requires a coordinated change to the router and extension.
2. **Recipient-based check**: Gate on `recipient` instead of `sender` when the pool is configured for router use, since `recipient` is the address that receives output tokens and is more directly tied to the economic beneficiary.

The cleanest fix is to have the router always append the originating `msg.sender` to `extensionData` so extensions can verify the true initiator.

---

### Proof of Concept

```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)      // intended allowlisted user
  admin calls setAllowedToSwap(pool, router, true)     // to let alice use the router

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ...})

Execution:
  router → pool.swap(recipient, ...)   [msg.sender = router]
  pool   → _beforeSwap(router, ...)
  pool   → extension.beforeSwap(sender=router, ...)
  extension checks: allowedSwapper[pool][router] == true  ✓
  swap executes — charlie bypasses the allowlist
``` [5](#0-4) 

The check `allowedSwapper[msg.sender][sender]` where `sender = address(router)` passes for any user routing through the router, defeating the per-user curation the extension is designed to enforce.
