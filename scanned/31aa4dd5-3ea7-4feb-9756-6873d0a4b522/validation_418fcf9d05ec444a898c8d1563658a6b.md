### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any caller to bypass a curated pool's swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. The allowlist therefore gates the router address, not the actual swapper. If the router is allowlisted (the only way to let any allowlisted user use the router), every unprivileged caller can bypass the curated pool's swap restriction by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`), the router calls `pool.swap(recipient, ...)` directly: [4](#0-3) 

At the pool level `msg.sender` is the **router contract address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

This creates an irreconcilable dilemma for any pool admin who deploys a curated pool with `SwapAllowlistExtension`:

| Admin choice | Consequence |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all — broken core functionality |
| **Allowlist the router** | Every unprivileged caller can bypass the allowlist by routing through the router |

The second branch is the fund-impacting path: once the router is allowlisted (the only way to give allowlisted users router access), any address can call `exactInputSingle` / `exactInput` / `exactOutput` and the extension will pass because `allowedSwapper[pool][router] == true`.

---

### Impact Explanation

A curated pool (e.g., institutional-only, KYC-gated, or whitelist-restricted) that relies on `SwapAllowlistExtension` to enforce access control is fully bypassed for any caller who routes through `MetricOmmSimpleRouter`. The unauthorized swapper receives pool output tokens and the pool receives input tokens at oracle-anchored prices — a direct, real-money swap that the pool admin intended to block. This is a **High** direct loss / policy-bypass impact: unauthorized parties extract value from a pool designed to exclude them, and the pool's LP composition and fee revenue are affected by trades that should never have executed.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public entry point for swaps in the periphery layer. Any user who discovers that a pool uses `SwapAllowlistExtension` can trivially route through the router. No privileged access, no special setup, and no malicious token is required — a single `exactInputSingle` call suffices. Likelihood is **High**.

---

### Recommendation

The extension must gate the **economically relevant actor** — the end user — not the intermediary contract. Two sound approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.

2. **Check `sender` only for direct pool calls; require the router to pass the real user as `sender`**: Modify `MetricOmmPool.swap` to accept an explicit `swapper` parameter (separate from `msg.sender`) that the router populates with the original caller, and forward that to extensions.

The deposit-side extension (`DepositAllowlistExtension`) correctly gates `owner` (the position beneficiary), which is caller-supplied and pool-enforced — the swap extension should adopt the same pattern by gating the address that economically benefits from the swap.

---

### Proof of Concept

```solidity
// Pool is deployed with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin allowlists the router so allowedUser can use it.
swapAllowlist.setAllowedToSwap(pool, address(router), true);  // router allowlisted

// Attacker (not allowlisted) calls the router directly.
router.exactInputSingle(ExactInputSingleParams({
    pool:            pool,
    recipient:       attacker,
    zeroForOne:      true,
    amountIn:        1e18,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    deadline:        block.timestamp,
    tokenIn:         token0,
    extensionData:   ""
}));
// Extension checks allowedSwapper[pool][router] == true → passes.
// Attacker receives token1 output from the curated pool.
// allowedSwapper[pool][attacker] was never set.
```
