### Title
`SwapAllowlistExtension` gates on the router address instead of the actual swapper — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual user. If the router is allowlisted (the only way to enable router-based swaps on a curated pool), every user on-chain can bypass the per-user swap allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`, which forwards it as `sender` to every registered extension:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value:

```solidity
// SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls the pool, the pool's `msg.sender` is the router contract, so `sender` = router address. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The sister extension `DepositAllowlistExtension.beforeAddLiquidity` correctly handles this by ignoring `sender` and checking `owner` (the actual position owner):

```solidity
// DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

No equivalent "owner" field exists in the swap path — the pool's `swap()` signature only exposes `recipient` (who receives output tokens), which is also not the initiating user when going through the router. The actual user who called the router is never forwarded to the pool.

---

### Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension` to restrict swaps to a curated set of addresses faces a binary failure:

**Case A — Router is allowlisted** (the only way to let allowlisted users use the router): `allowedSwapper[pool][router] = true` passes the check for every caller, regardless of who they are. Any address on-chain can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) and swap against the curated pool. The allowlist is completely bypassed.

**Case B — Router is not allowlisted**: Every router-based swap reverts with `NotAllowedToSwap`, even for users who are individually allowlisted. Allowlisted users are forced to call the pool directly, losing access to slippage protection, multi-hop routing, and deadline enforcement provided by the router.

Case A is the critical path: the pool admin allowlists the router to enable normal UX, and in doing so silently opens the pool to all swappers.

---

### Likelihood Explanation

The `SwapAllowlistExtension` is a production periphery contract explicitly designed for curated pools. The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool admin who deploys both together and allowlists the router (the natural configuration) triggers the bypass. No special permissions, flash loans, or oracle manipulation are required — a standard `exactInputSingle` call from any EOA suffices.

---

### Recommendation

Mirror the `DepositAllowlistExtension` pattern: check the economic actor, not the direct pool caller. For swaps, the closest available field is `recipient` (the second parameter, currently discarded with `address`). However, `recipient` is also not the initiating user in all cases. The correct fix is to check `sender` only when the pool is called directly, or — better — add a `swapper` field to the swap hook signature that the pool populates from a trusted source (e.g., a transient-storage context set by the router before calling the pool, analogous to how the liquidity adder stores `payer` in transient storage).

As a minimal fix within the current interface, change `beforeSwap` to check `recipient` instead of `sender`, since `recipient` is the address that economically benefits from the swap output and is user-controlled even through the router:

```solidity
// current (wrong actor)
function beforeSwap(address sender, address, ...)
    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// fixed (economic beneficiary)
function beforeSwap(address, address recipient, ...)
    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` registered in `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` with `pool` as target.
5. The router calls `pool.swap(recipient, ...)` — pool passes `msg.sender` (router) as `sender` to `_beforeSwap`.
6. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Bob's swap executes successfully against the curated pool, bypassing the per-user allowlist.
