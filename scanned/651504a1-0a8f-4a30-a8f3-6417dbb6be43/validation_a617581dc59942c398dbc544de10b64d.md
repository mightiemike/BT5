### Title
Silent Output Cap in Exact-Output Swap Delivers Less Than Requested Without Revert — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`_swapToken1ForToken0SpecifiedOutput` silently clamps the requested output amount to the pool's total available token0 balance instead of reverting when the requested amount exceeds what the pool holds. This is the direct analog to M-11: a guard that should reject an out-of-bounds value instead silently caps it, allowing the operation to succeed with a materially different outcome than the caller specified.

---

### Finding Description

When a caller submits an exact-output swap (`zeroForOne = false`, `amountSpecified < 0`) requesting more token0 than the pool holds, `_swapToken1ForToken0SpecifiedOutput` executes the following at the top of its body:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;   // silent cap — no revert
}
``` [1](#0-0) 

After the cap, the function continues normally: it iterates bins, computes the reduced output, calls `_finalizeSwap`, and returns. Control returns to `swap()`, which fires the callback and emits `Swap`. The transaction succeeds. The caller receives fewer tokens than it requested, with no error signal.

The `priceLimitX64` guard that follows does **not** protect against this. It bounds the marginal price at which the swap executes, not the total output volume. A pool that is nearly empty can still satisfy the price limit while delivering a fraction of the requested output. [2](#0-1) 

The symmetric `_swapToken0ForToken1SpecifiedOutput` path (called when `zeroForOne = true`, `amountSpecified < 0`) is visible in the same file and follows the same structural pattern. [3](#0-2) 

---

### Impact Explanation

**Swap conservation failure / direct loss of user principal.**

An exact-output swap is a binding commitment: the caller specifies exactly how many tokens it must receive. Routers, aggregators, and on-chain integrators built on top of `MetricOmmPool` rely on this guarantee to enforce their own minimum-output checks. When the pool silently delivers a smaller amount:

- The caller's downstream logic (e.g., a multi-hop router that forwards the output into a second pool) receives less than expected and may revert, leaving the user's input token already transferred with no recourse.
- A caller that does not independently verify the returned delta (relying on the exact-output specification as the guarantee) suffers a direct shortfall in received tokens.
- The `Swap` event emits the capped `amount0Delta`, so off-chain monitoring also sees a "successful" swap at a reduced amount, masking the discrepancy. [4](#0-3) 

---

### Likelihood Explanation

No special permissions are required. Any user or contract can trigger this by submitting an exact-output swap for an amount larger than the pool's current token0 balance. This condition arises naturally as pool liquidity depletes through normal trading activity or after large removals. It is reachable through the public `swap()` entry point with no privileged setup. [5](#0-4) 

---

### Recommendation

Replace the silent cap with an explicit revert:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    revert InsufficientLiquidity();   // revert instead of cap
}
```

This mirrors the correct behavior described in M-11's mitigation: when the requested amount exceeds the available limit, the transaction must revert so the caller can handle the failure explicitly.

---

### Proof of Concept

1. Pool is deployed with `binTotals.scaledToken0 = 100e18` (100 token0 scaled).
2. Attacker/user calls `pool.swap(recipient, false, -200e6, priceLimitX64, callbackData, "")` — requesting 200 token0 output (exact-out).
3. `_executeSwap` computes `amountOutScaled = TOKEN_0_SCALE_MULTIPLIER * 200e6`.
4. `_swapToken1ForToken0SpecifiedOutput` is entered; the cap fires: `amountOutScaled` is silently reduced to `100e18`.
5. The swap loop executes against the capped amount, draining all token0 from bins.
6. `_finalizeSwap` writes the new bin state; `swap()` transfers only 100 token0 to `recipient`.
7. The callback is called with `amount0Delta = -100e6` (external units), not `-200e6`.
8. The transaction succeeds. The caller requested 200 token0 and received 100 — a 50% shortfall — with no revert and no on-chain error. [6](#0-5) [1](#0-0)
