### Title
`SwapAllowlistExtension` checks the router address as `sender` instead of the actual end-user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the value the pool passes from its own `msg.sender`. When `MetricOmmSimpleRouter` is the direct caller of `pool.swap()`, `sender` is the **router address**, not the actual end-user. If the pool admin allowlists the router to let legitimate users trade through it, every unpermissioned address on-chain can also bypass the guard by routing through the same router.

---

### Finding Description

**Root cause — pool passes `msg.sender` as `sender` to extensions**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, …)` and `_afterSwap(msg.sender, recipient, …)`. [1](#0-0) 

`simulateSwapAndRevert` shows the same pattern explicitly: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` argument to every configured extension: [3](#0-2) 

**Root cause — `SwapAllowlistExtension` checks `sender` (the router) not the real user**

```solidity
function beforeSwap(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    …
}
``` [4](#0-3) 

Here `msg.sender` is the pool (correct) and `sender` is what the pool forwarded — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry-point) calls the pool, `sender` = router address. [5](#0-4) 

The effective check therefore becomes:

```
allowedSwapper[pool][router]   // NOT allowedSwapper[pool][actual_user]
```

**The dilemma the pool admin faces**

| Admin action | Consequence |
|---|---|
| Allowlist the router | Every address on-chain can swap through the router — allowlist is nullified |
| Do not allowlist the router | Allowlisted users cannot use the router at all — core swap path is broken |

There is no configuration that simultaneously (a) lets allowlisted users trade through the router and (b) blocks non-allowlisted users from doing the same.

**Contrast with `DepositAllowlistExtension`**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the actual LP position owner), not `sender`: [6](#0-5) 

The swap extension has no equivalent separation between the intermediary caller and the real beneficiary.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, whitelisted market-makers) is fully bypassable by any address that routes through `MetricOmmSimpleRouter`. Once the router is allowlisted — the only way to let legitimate users trade through the standard periphery — the guard provides zero protection. Unauthorized swappers can drain the pool's liquidity at oracle-derived prices, causing direct loss of LP principal and fee revenue to liquidity providers who deposited under the assumption that only permitted counterparties could trade against them.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary user-facing swap interface for the protocol.
- Any pool admin who wants allowlisted users to use the router must allowlist the router address, which is the natural and expected configuration.
- Once that configuration is in place, the bypass requires no special knowledge or privilege — any EOA or contract can call `router.exactInputSingle(...)` targeting the restricted pool.
- The bypass is silent: no event distinguishes a router-mediated swap from a direct swap, so the pool admin has no on-chain signal that the guard has been circumvented.

---

### Recommendation

Pass the real end-user identity through the swap path. Two options:

1. **Preferred — use `recipient` as the identity to check.** The `recipient` is the address that receives output tokens and is the closest proxy for the economic beneficiary. Change `SwapAllowlistExtension.beforeSwap` to check `recipient` instead of `sender`. The router always sets `recipient` to the user-supplied address.

2. **Alternative — add a `payer` field to `extensionData`.** The router could encode `msg.sender` (the actual user) into `extensionData` and the extension could decode it. This requires a coordinated change to the router and extension.

Option 1 requires no router changes and is consistent with how `DepositAllowlistExtension` uses `owner` (the economic actor) rather than `sender` (the intermediary).

---

### Proof of Concept

```
Setup:
  - Pool P is deployed with SwapAllowlistExtension E.
  - Admin allowlists router R: E.setAllowedToSwap(P, router, true)
  - Admin does NOT allowlist attacker A.

Attack:
  1. Attacker A calls router.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=A, ...)  →  msg.sender = router
  3. Pool calls E.beforeSwap(sender=router, ...)
  4. Extension checks: allowedSwapper[P][router] == true  →  PASSES
  5. Swap executes; A receives output tokens from the restricted pool.

Result:
  - A, who is not on the allowlist, successfully swaps against the restricted pool.
  - LPs suffer losses from trades with an unauthorized counterparty.
  - The allowlist guard is completely nullified for all router-mediated swaps.
``` [4](#0-3) [7](#0-6) [3](#0-2)
