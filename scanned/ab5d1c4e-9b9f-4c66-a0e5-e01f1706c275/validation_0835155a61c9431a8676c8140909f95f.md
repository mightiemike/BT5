### Title
Pool Admin Can Set Unbounded Per-Bin Additional Fees, Bypassing Factory-Enforced Spread Fee Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards per-bin additional fees to the pool with **no validation against `maxAdminSpreadFeeE6`**, while the parallel base-spread setter `setPoolAdminFees` enforces that cap explicitly. A pool admin can therefore charge per-bin fees that exceed the factory-owner-imposed ceiling, causing traders to pay more than the protocol intends to allow.

---

### Finding Description

The factory maintains two separate admin-fee setters:

**`setPoolAdminFees`** — enforces the cap: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**`setPoolBinAdditionalFees`** — no cap check at all: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The per-bin fees are then **added on top of the base spread fee** inside every swap: [3](#0-2) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to any value from 0 to 65 535 (≈ 6.55 % in E6 units). The factory owner can set `maxAdminSpreadFeeE6` as low as 0 to prohibit admin spread fees entirely, yet the pool admin can still impose up to 6.55 % per-bin fee through this unchecked path, completely bypassing the cap.

The hard cap constants confirm the intent to bound fees: [4](#0-3) 

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000;
```

The bin-additional-fee path is the only admin-controlled fee path that carries no analogous guard.

---

### Impact Explanation

Every swap through an affected bin charges the trader `baseFee + addFeeBuyE6/addFeeSellE6`. If the factory owner has capped admin spread fees at, say, 0 % to protect users, the pool admin can still extract up to 6.55 % per swap per bin. This is a **direct, per-trade loss of user principal** routed to the pool's LP surplus (which the pool admin ultimately collects via `collectPoolFees`). The loss scales with swap volume and the number of bins configured with inflated fees.

---

### Likelihood Explanation

The pool admin is a semi-trusted role explicitly constrained by factory-owner-set caps. The factory owner can lower `maxAdminSpreadFeeE6` to protect users, but that protection is silently bypassed by `setPoolBinAdditionalFees`. Any pool admin who wishes to extract excess fees can do so immediately, without any timelock or further approval, making exploitation straightforward once a pool is deployed.

---

### Recommendation

Add the same cap enforcement to `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce a combined cap so that `adminSpreadFeeE6 + addFeeBuyE6 ≤ maxAdminSpreadFeeE6` for each bin.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(0, 0, 0, 0)` — intending to prohibit all admin fees.
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs; `setBinAdditionalFees` stores `addFeeBuyE6 = 65535` on bin 0.
4. A trader calls `swap(…, zeroForOne=false, …)` routing through bin 0.
5. The swap math applies `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — an effective ≈ 6.55 % additional fee — despite the factory owner having set the admin cap to 0.
6. The excess fee accumulates in the pool's token surplus and is later collected by the pool admin via `collectPoolFees`, constituting a direct loss to the trader.
