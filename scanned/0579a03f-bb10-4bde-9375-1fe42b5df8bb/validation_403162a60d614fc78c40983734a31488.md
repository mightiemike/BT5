### Title
`SwapAllowlistExtension` gates the router address instead of the real user, allowing any unprivileged caller to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. That argument is `msg.sender` of the pool's `swap()` call. When a user enters through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the originating user. If the pool admin allowlists the router (the only way to let allowlisted users reach the pool through the supported periphery path), the allowlist becomes a no-op: every user on the network can bypass it by routing through the same public router.

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension:

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (correct pool-key) and `sender` is the router (wrong actor):

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The allowlist is keyed on the router address. A pool admin who wants allowlisted users to reach the pool through the standard periphery path must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true` for every call that arrives through the router — regardless of who the originating user is. Any address on the network can then call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and the guard passes.

---

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a specific set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker receives the full output token amount from the pool at live oracle prices; the pool's LP providers bear the resulting exposure to unrestricted toxic or unauthorized flow. This is a direct loss of the pool's curation guarantee and constitutes broken core pool functionality (the allowlist guard that the pool was configured to enforce is silently bypassed).

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported periphery path for swaps. A pool admin who allowlists individual user EOAs but does not also allowlist the router will find that those users cannot use the router at all (the check fails with `NotAllowedToSwap` because the router is not in the allowlist). The natural operational fix — allowlisting the router — immediately opens the gate to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior: any EOA calls a standard router function.

---

### Recommendation

The `beforeSwap` hook receives both `sender` (the direct pool caller, i.e., the router) and the full call context. To gate the originating user, the extension must either:

1. **Check the original user via a trusted forwarding convention** — require the router to pass the real user's address in `extensionData`, and have the extension decode and verify it (with the pool or router attesting to its authenticity); or
2. **Gate `sender` only when `sender` is not a known trusted router**, and require the router to separately attest the real user; or
3. **Simplest correct fix** — document that `sender` is the direct caller and change the allowlist semantics so that the pool admin allowlists the router and the router enforces its own per-user access control, making the two layers explicit and non-conflatable.

The minimal on-chain fix consistent with the current design is to have the router forward the originating user's address in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of `sender` when `sender` is a recognized router.
