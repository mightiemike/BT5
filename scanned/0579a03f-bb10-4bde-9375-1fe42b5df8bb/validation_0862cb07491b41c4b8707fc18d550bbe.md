### Title
`SwapAllowlistExtension` Gates the Router's Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter against the per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension receives `sender = router` and checks `allowedSwapper[pool][router]` — not the actual user's address. If the router is allowlisted (the only way to enable router-mediated swaps), every user on-chain bypasses the per-user allowlist gate.

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool from its own `swap` call. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router, so the pool forwards `sender = router` to the extension. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The `FullMetricExtensionTest` confirms this binding: the test allowlists `callers[0]` (the `TestCaller` contract that wraps `users[0]`), not `users[0]` itself, because the pool passes the immediate caller as `sender`. [2](#0-1) 

This creates two mutually exclusive broken states for any pool using `SwapAllowlistExtension` with the router:

**State A — Router not allowlisted:** Pool admin allowlists individual user addresses. Router-mediated swaps revert for every user because the extension sees `sender = router` (not allowlisted). Core swap functionality through the supported periphery path is broken for all allowlisted users.

**State B — Router allowlisted:** Pool admin allowlists the router address to enable router-mediated swaps. Now `allowedSwapper[pool][router] = true`, so the check passes for every caller regardless of who they are. Any unprivileged user bypasses the per-user allowlist by routing through `MetricOmmSimpleRouter`.

The `DepositAllowlistExtension` does **not** share this flaw: it checks the `owner` parameter, which the liquidity adder passes explicitly as the LP position owner, correctly identifying the economic actor. [3](#0-2) [1](#0-0) 

The generate_scanned_questions audit pivot explicitly flags this exact concern:

> *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."* [4](#0-3) 

### Impact Explanation

**High.** A curated pool's entire swap allowlist policy is nullified. Any unprivileged user can swap in a pool intended to be restricted by routing through `MetricOmmSimpleRouter`. Unauthorized swaps against a curated pool can drain LP principal through adversarial trade selection, violate the pool's economic design, and cause direct loss of LP assets above Sherlock thresholds. This matches the "allowlist bypass" impact gate: *"High direct loss or curation failure if disallowed users can still trade or deposit."* [5](#0-4) 

### Likelihood Explanation

**Medium.** `MetricOmmSimpleRouter` is a public, permissionless contract. Any user who observes that a pool uses `SwapAllowlistExtension` can immediately attempt a router-mediated swap. The pool admin faces an impossible configuration choice: either the router is allowlisted (bypass enabled for everyone) or it is not (router-mediated swaps broken for everyone). No correct configuration exists.

### Recommendation

The pool must forward the original user's address — not `msg.sender` — as `sender` to the extension. Two approaches:

1. **Explicit originator forwarding:** Have `MetricOmmSimpleRouter` pass the user's address as a dedicated `originator` field in the swap call, and have the pool forward it to the extension as `sender` instead of `msg.sender`.
2. **Mirror the deposit pattern:** Follow `DepositAllowlistExtension`'s approach — check an explicitly passed identity parameter (like `owner`) rather than the immediate caller, so the liquidity adder and router can pass the true economic actor through.

### Proof of Concept

```
1. Pool admin deploys a pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)  — only alice should swap.
3. Pool admin calls setAllowedToSwap(pool, router, true) — required to enable router swaps.
4. bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) — pool's msg.sender = router.
6. Pool calls extension.beforeSwap(router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true  → passes.
8. bob's swap executes against the curated pool despite not being allowlisted.
   bob can drain LP funds through adversarial trade selection.
``` [1](#0-0) [3](#0-2) [6](#0-5) [7](#0-6)
