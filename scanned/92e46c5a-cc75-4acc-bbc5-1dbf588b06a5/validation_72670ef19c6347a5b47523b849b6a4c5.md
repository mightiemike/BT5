### Title
`DepositAllowlistExtension` gates position `owner` instead of the actual depositor (`sender`), allowing any unprivileged address to bypass the deposit allowlist — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool" and uses a mapping named `allowedDepositor`. However, its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of the pool's `addLiquidity` call, i.e., the token payer) and instead checks only `owner` (the position beneficiary). Because `addLiquidity` explicitly supports an operator pattern where `msg.sender` pays but `owner` receives the position, any address that is not in the allowlist can deposit into a curated pool simply by naming an allowed `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both values to every configured extension:

```solidity
// ExtensionCalling.sol lines 95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but declares it as an unnamed, unused `address`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The guard therefore only asks "is `owner` allowed?" and never asks "is the actual depositor/payer allowed?".

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(address pool, address owner, ...)` lets any caller supply an arbitrary `owner`:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56-68
function addLiquidityExactShares(
    address pool,
    address owner,
    ...
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks owner != address(0)
    ...
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

`_validateOwner` only rejects `address(0)`. There is no check that `msg.sender == owner`.

---

### Impact Explanation

A pool admin who deploys a curated pool with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook intends to restrict which addresses can supply liquidity. Because the guard checks `owner` rather than `sender`, any unprivileged address can:

1. Call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, allowedOwner, ...)` naming any address that is already in the allowlist as `owner`.
2. The extension sees `allowedDepositor[pool][allowedOwner] == true` and passes.
3. The unprivileged caller pays the tokens; `allowedOwner` receives the LP position.

The pool admin's curation boundary is fully broken: unauthorized liquidity enters the pool, the LP share distribution is altered, and the admin has no on-chain mechanism to prevent it. If the pool carries specific risk parameters or regulatory constraints tied to LP identity, those constraints are silently violated. The `allowedOwner` also receives an unsolicited position they did not initiate, which they may not remove promptly, leaving the pool's liquidity composition in an unintended state.

---

### Likelihood Explanation

The attack requires only that at least one address is already in the allowlist (a necessary precondition for the pool to be usable at all). The attacker needs no special role, no privileged key, and no cooperation from the allowlisted address. The `MetricOmmPoolLiquidityAdder` is a supported periphery contract that any user can call. Likelihood is high whenever a curated pool with this extension is live.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual token payer / `msg.sender` of the pool call) instead of, or in addition to, `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    // Gate the actual depositor (payer), not just the position beneficiary.
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended policy is to gate both the payer and the position owner, both checks should be applied. The mapping name `allowedDepositor` and the NatSpec "Gates `addLiquidity` by depositor address" both indicate the payer is the intended subject of the guard.

---

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension as beforeAddLiquidity hook
  allowedDepositor[pool][alice] = true
  bob is NOT in the allowlist

Attack:
  // Bob calls the periphery liquidity adder, naming alice as owner
  liquidityAdder.addLiquidityExactShares(
      pool,
      alice,   // owner — passes the allowlist check
      salt,
      deltas,
      maxAmount0,
      maxAmount1,
      ""
  );
  // Inside: pool.addLiquidity(alice, ...) is called with msg.sender = liquidityAdder
  // Extension checks allowedDepositor[pool][alice] == true → no revert
  // Bob's tokens are pulled; alice receives the LP position
  // Bob has successfully deposited into a pool he is not authorized to enter
