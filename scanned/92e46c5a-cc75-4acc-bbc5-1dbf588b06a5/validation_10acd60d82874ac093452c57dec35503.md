### Title
SwapAllowlistExtension gates the router address instead of the end user, enabling allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When swaps route through `MetricOmmSimpleRouter`, `sender` is the router address, not the originating user. If the pool admin allowlists the router to support router-based swaps, every user — including those explicitly excluded from the per-address allowlist — can bypass the gate by routing through the router.

---

### Finding Description

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards this value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks:

```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()`, so `sender = address(router)`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`. [4](#0-3) 

A pool admin who wants to support router-based swaps for allowlisted users must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, **any** user — regardless of whether their own address is on the allowlist — can call the router and pass the gate. The admin has no mechanism to allowlist the router for some users but not others, because the extension has no visibility into who called the router.

The same issue applies to multi-hop `exactInput()` calls, where the router is `msg.sender` for every hop: [5](#0-4) 

---

### Impact Explanation

Any user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter`, provided the router address has been added to the allowlist. The allowlist's curation guarantee — that only explicitly permitted addresses may trade — is completely defeated for all router-mediated swaps. Disallowed users gain full swap access to a pool designed to restrict them, which can result in direct loss of LP value on pools where the allowlist is the primary protection against adversarial flow (e.g., MEV-sensitive or KYC-gated pools).

---

### Likelihood Explanation

The bypass is reachable whenever the pool admin allowlists the router address. This is a natural and expected administrative action: the router is the primary user-facing entry point for swaps, and an admin who wants allowlisted users to be able to use the router must allowlist it. The admin has no way to allowlist the router for a subset of users; the only options are to allowlist the router globally (bypassing all per-user restrictions) or not at all (blocking all router users, even allowed ones). The design trap is subtle and the admin is likely to fall into it. [6](#0-5) 

---

### Recommendation

The extension should gate the originating user, not the intermediary. Options:

1. Require the router to encode the originating user's address in `extensionData`, and have the extension decode and check that address when `sender` is a known router.
2. Maintain a separate allowlist of trusted routers that are permitted to forward swaps on behalf of any allowlisted user, and require the router to attest the originator in `extensionData`.
3. Document clearly that allowlisting the router grants swap access to all router users, so admins understand the implication before doing so.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension as a beforeSwap hook.
2. Admin allowlists the router:
       swapAllowlistExtension.setAllowedToSwap(pool, address(router), true)
3. Attacker (address NOT in allowedSwapper[pool]) calls:
       router.exactInputSingle({pool: pool, recipient: attacker, ...})
4. Pool calls _beforeSwap(address(router), ...)
5. Extension evaluates allowedSwapper[pool][address(router)] == true → passes.
6. Attacker's swap executes on the curated pool despite not being individually allowlisted.
``` [7](#0-6) [1](#0-0)
