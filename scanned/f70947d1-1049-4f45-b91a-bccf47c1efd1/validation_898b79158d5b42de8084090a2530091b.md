### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool level is the router contract, not the originating user. If the pool admin allowlists the router (the natural step to let allowlisted users use the standard periphery), every user — including those explicitly excluded — can bypass the allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // whoever called pool.swap() — the router when routed
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) calls `pool.swap()`, the pool's `msg.sender` is the router:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The router never forwards the originating user's address to `pool.swap()`. The pool has no parameter for an originator. Therefore the extension always sees `sender = router_address`, not the actual user.

**Attack path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of users.
2. Admin allowlists `alice` directly: `allowedSwapper[pool][alice] = true`.
3. Admin also allowlists the router so that `alice` can use the standard periphery: `allowedSwapper[pool][router] = true`.
4. `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. `bob` swaps on the curated pool without being allowlisted.

The same bypass applies to `exactInput` (all hops call `pool.swap()` with `msg.sender = router`) and `exactOutput` (the recursive callback also calls subsequent pools with `msg.sender = router`).

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores the `sender` argument and checks `owner` instead, which is the position owner explicitly passed through the call stack and not overwritten by the LiquidityAdder. The swap extension has no equivalent stable identity to check.

---

### Impact Explanation

A pool admin who intends to restrict swaps to a curated set of users cannot do so if those users need to use the router. The only way to allow allowlisted users to use the router is to allowlist the router itself, which simultaneously opens the pool to every user. Any user can then execute swaps on a pool that was designed to be private, draining LP value at oracle-anchored prices that the pool admin assumed only trusted counterparties would access.

**Impact class:** High — direct loss of LP principal and protocol fees on curated pools; broken core pool functionality (the allowlist invariant is violated).

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the canonical periphery entry point. Pool admins who deploy a `SwapAllowlistExtension` and want their allowlisted users to use the router will naturally allowlist the router. This is the expected operational pattern, making the bypass reachable without any unusual configuration. No privileged access beyond the admin's own intended setup is required by the attacker.

---

### Recommendation

The pool's `swap()` interface does not carry an originator address, so the fix must be applied in the extension or the router:

1. **Preferred — router forwards originator in `extensionData`:** Define a standard encoding where the router prepends `msg.sender` (the actual user) to `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` decodes and checks this address instead of `sender`. The extension must also verify that `msg.sender` (the pool's caller) is a trusted router; otherwise any caller can forge the originator field.

2. **Alternative — check `recipient` as a proxy:** If the pool's design guarantees that the recipient is always the actual user, the extension could check `recipient`. However, this is not guaranteed (multi-hop routes use intermediate addresses as recipients).

3. **Minimal guard:** Document that `SwapAllowlistExtension` is incompatible with the router and revert in `beforeSwap` if `sender` is a known router address, forcing direct pool interaction only.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension
// Admin allowlists alice and the router
swapExtension.setAllowedToSwap(address(pool), alice, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// bob is NOT allowlisted

// bob bypasses the allowlist via the router
vm.prank(bob);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(pool),
    tokenIn:         address(token0),
    recipient:       bob,
    deadline:        block.timestamp + 1,
    zeroForOne:      true,
    amountIn:        1000,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    extensionData:   ""
}));
// Succeeds: pool.swap() sees msg.sender = router, which is allowlisted.
// bob swaps on the curated pool without being individually allowlisted.
```
