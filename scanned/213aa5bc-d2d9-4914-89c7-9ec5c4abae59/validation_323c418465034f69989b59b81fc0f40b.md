### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating Swapper, Enabling Full Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[pool][sender]` where `sender` is the `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router address, not the originating user. A pool admin who allowlists the router so that allowlisted users can access it via the router simultaneously opens the gate to every non-allowlisted user, completely defeating the access-control invariant the extension is meant to enforce.

### Finding Description

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to the extension dispatcher: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then evaluates the allowlist keyed on that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter` (a public periphery contract), the router calls `pool.swap(...)` with itself as `msg.sender`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates an irresolvable dilemma for the pool admin:

| Admin choice | Effect on allowlisted users | Effect on non-allowlisted users |
|---|---|---|
| Do **not** allowlist the router | Cannot use the router | Also cannot use the router |
| **Allowlist the router** | Can use the router | **Can also use the router — bypass complete** |

There is no configuration that simultaneously permits allowlisted users to route through the public router and blocks non-allowlisted users from doing the same.

The `SwapAllowlistExtension` stores its allowlist keyed by pool (using `msg.sender` inside the extension, which is the pool): [4](#0-3) 

The `onlyPool` guard in `BaseMetricExtension` only verifies the caller is a factory-registered pool; it does not verify the identity of the human initiating the swap: [5](#0-4) 

### Impact Explanation

Any non-allowlisted address can execute swaps on a pool whose admin intended to restrict trading to a curated set of counterparties. The pool's admin-boundary invariant — "only allowlisted addresses may swap" — is broken. Depending on the pool's purpose (e.g., institutional-only liquidity, regulatory-compliant venue, KYC-gated market), this allows unauthorized parties to extract value from the pool's liquidity, trade against restricted price feeds, or violate compliance requirements. This is a direct admin-boundary break with fund-impacting consequences: the pool's LPs are exposed to swap flow from actors the pool admin explicitly intended to exclude.

### Likelihood Explanation

Medium. The bypass requires the pool admin to allowlist the router — a natural and expected action when the pool is meant to be usable via the standard periphery. A pool admin who deploys `SwapAllowlistExtension` to restrict individual swappers will almost certainly also want those swappers to use the router for slippage protection and multicall. The connection between "allowlisting the router" and "opening the gate to everyone" is non-obvious and undocumented. No special privilege or front-running is required; any EOA can call the public router.

### Recommendation

The extension must check the identity of the economic actor, not the immediate caller of `pool.swap()`. Two sound approaches:

1. **Trusted-forwarder pattern**: `MetricOmmSimpleRouter` appends the originating `msg.sender` to `extensionData` in a tamper-evident way (e.g., ABI-encoded and signed, or via a dedicated router-identity slot). `SwapAllowlistExtension` reads and verifies that field when `sender` is a known router.

2. **Direct-call-only enforcement**: Document and enforce that pools using `SwapAllowlistExtension` must not allowlist any router or intermediary contract. The extension should expose a `setAllowedRouter` flag that, when false, causes the hook to revert if `sender` is not an EOA (i.e., `sender.code.length == 0`).

Either approach closes the gap between the identity the admin intends to gate and the identity the extension actually checks.

### Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension (BEFORE_SWAP_ORDER configured).
2. Admin calls setAllowedToSwap(pool, Alice, true)   // Alice is the intended swapper
3. Admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so Alice can use it

Attack
──────
4. Charlie (non-allowlisted EOA) calls:
       MetricOmmSimpleRouter.exactInput(
           pool, zeroForOne=true, amountIn, minOut, recipient=Charlie, extensionData=""
       )
   The router calls pool.swap(recipient=Charlie, ...).

5. Pool dispatches _beforeSwap(sender=router, ...).

6. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true   ✓

7. Swap executes. Charlie receives output tokens.
   The allowlist never evaluated Charlie's address.

Result: Charlie swaps freely on a pool the admin intended to restrict to Alice only.
