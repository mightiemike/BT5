### Title
`DepositAllowlistExtension` Guards `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor address**. Its `beforeAddLiquidity` hook silently drops the `sender` argument and checks `owner` instead. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, …)` accepts an arbitrary `owner` with no constraint beyond `owner != address(0)`, any unprivileged caller can satisfy the allowlist check by supplying an already-allowlisted address as `owner`, while they themselves remain the actual payer and the allowlist is never consulted for them.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` ignores its first parameter (`sender`) and evaluates the second (`owner`):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The pool passes `msg.sender` (the direct pool caller, e.g. the LiquidityAdder) as `sender` and the caller-supplied `owner` as the second argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the explicit-owner overload) accepts any non-zero `owner` from the caller with no further validation:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L56-68
function addLiquidityExactShares(
    address pool, address owner, uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);   // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
```

`_validateOwner` only rejects `address(0)`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

**Attack path:**

1. Pool is deployed with `DepositAllowlistExtension`; pool admin allowlists only `Alice`.
2. `Bob` (not allowlisted) calls `addLiquidityExactShares(pool, Alice, salt, deltas, max0, max1, data)`.
3. The LiquidityAdder calls `pool.addLiquidity(Alice, salt, deltas, …)`.
4. The pool calls `_beforeAddLiquidity(sender=LiquidityAdder, owner=Alice, …)`.
5. The extension evaluates `allowedDepositor[pool][Alice]` → `true` → hook passes.
6. Bob's tokens are pulled (Bob is the payer stored in transient context); the position is recorded under Alice's key.
7. The allowlist was never consulted for Bob.

The guard is completely ineffective: any caller can name any allowlisted address as `owner` and the check passes unconditionally for them.

---

### Impact Explanation

The `DepositAllowlistExtension` is the pool admin's primary mechanism for restricting who may provide liquidity (e.g., KYC/compliance gating, competitive exclusion, or regulatory controls). Because the hook checks the wrong field, the restriction is trivially bypassed by any unprivileged actor. This constitutes an **admin-boundary break**: a pool admin-configured guard is bypassed by an unprivileged path, violating the invariant that only allowlisted depositors may add liquidity to the pool.

Secondary consequence: Bob can force Alice into an unwanted LP position (Alice is the recorded owner and must actively call `removeLiquidity` to exit it, since `removeLiquidity` enforces `msg.sender == owner`). If the pool's value declines before Alice notices and removes the position, Alice suffers a direct loss of principal she never consented to deposit.

---

### Likelihood Explanation

- The bypass requires no special privilege, no flash loan, and no oracle manipulation.
- Any address that knows one allowlisted address (which is publicly readable from `allowedDepositor`) can execute the bypass in a single transaction.
- The LiquidityAdder explicitly documents that `owner` is caller-supplied with no factory or allowlist cross-check.

Likelihood: **High**.

---

### Recommendation

In `beforeAddLiquidity`, check `sender` (the actual depositing caller) rather than `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Additionally, `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (explicit-owner overload) should require `owner == msg.sender` or require the caller to prove authorization for the target owner, so that the `sender` field the pool passes to extensions is meaningful.

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  Bob is NOT in allowedDepositor

Attack:
  Bob calls:
    liquidityAdder.addLiquidityExactShares(
        pool,
        Alice,   // owner — allowlisted, so hook passes
        salt,
        deltas,
        max0, max1,
        extensionData
    )

  Pool calls:
    _beforeAddLiquidity(sender=liquidityAdder, owner=Alice, ...)

  Extension evaluates:
    allowedDepositor[pool][Alice] == true  →  no revert

  Callback pulls tokens from Bob (payer stored in transient slot).
  Position recorded under Alice's key.

Result:
  Bob deposited without being allowlisted.
  Alice holds an unwanted position she did not initiate.
