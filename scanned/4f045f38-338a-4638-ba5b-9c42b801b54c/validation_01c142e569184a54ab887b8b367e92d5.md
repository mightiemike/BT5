### Title
Wrong-actor binding in `SwapAllowlistExtension.beforeSwap`: router address checked instead of end user — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the first argument passed by the pool — which is `msg.sender` of the pool's `swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the hook checks the router's address, not the end user's address.

### Finding Description
In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // <-- this is the router when called via router
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value verbatim to the extension:

```solidity
// ExtensionCalling.sol:163-165
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When the router calls the pool, `sender` = router address. The allowlist is keyed by end-user addresses, so:

- **Path A (router not allowlisted):** Every router-mediated swap reverts with `NotAllowedToSwap`, even for users the pool admin explicitly allowlisted. Allowlisted users are permanently locked out of the router.
- **Path B (router allowlisted to fix Path A):** Any arbitrary user can bypass the allowlist entirely by routing through `MetricOmmSimpleRouter`, since the hook only sees the router address and passes.

### Impact Explanation
The allowlist is the pool designer's primary access-control mechanism for curated/permissioned pools. The wrong-actor binding means either (A) the allowlist breaks all router usage for legitimate users, or (B) the allowlist is trivially bypassed by any user who routes through the public router. In either case, the invariant "only allowlisted addresses may swap" is violated. For pools that use the allowlist to restrict trading to KYC'd counterparties or specific market makers, this constitutes a policy bypass enabling unauthorized swaps — a broken core functionality finding.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary public entrypoint for swaps. Any user who calls `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` triggers this bug. No special conditions, privileged access, or unusual token behavior is required. The attacker simply calls the public router.

### Recommendation
The pool cannot know the original end user once the router is the `msg.sender`. The fix must be applied at the router level: pass the actual end user's address through `extensionData` or a dedicated `sender` override field, and have the extension read it from there. Alternatively, the pool interface should accept an explicit `swapper` parameter distinct from `msg.sender`, authenticated by the router (e.g., via a trusted-forwarder pattern). The simplest scoped fix is for the router to encode `msg.sender` (the end user) into `extensionData` and for `SwapAllowlistExtension.beforeSwap` to decode and check that value when present, falling back to `sender` for direct pool calls.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` active.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowlisted.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` with himself as `recipient`.
4. The pool calls `_beforeSwap(router_address, ...)`.
5. The extension checks `allowedSwapper[pool][router_address]` — if the router was allowlisted to allow Alice to use it, this returns `true` and Bob's swap succeeds.
6. Bob, an un-allowlisted user, has successfully swapped on a permissioned pool.

Conversely, if the router is not allowlisted, Alice's swap via the router reverts at step 5 even though she is explicitly allowlisted.
