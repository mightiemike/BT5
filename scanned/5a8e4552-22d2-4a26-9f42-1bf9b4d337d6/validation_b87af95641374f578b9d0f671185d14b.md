### Title
`SwapAllowlistExtension` checks the router's address as `sender` instead of the original user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the first argument forwarded by the pool — which is `msg.sender` of `pool.swap()`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the **router contract**, not the original user. If the pool admin allowlists the router address (a natural configuration to enable router-mediated swaps for legitimate users), every user — including non-allowlisted ones — can bypass the curated allowlist by routing through the router.

---

### Finding Description

**Call chain:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
         → IMetricOmmPoolActions(pool).swap(recipient, ..., extensionData)
              [msg.sender = router]
         → MetricOmmPool.swap()
              → _beforeSwap(msg.sender=router, recipient, ...)
         → ExtensionCalling._beforeSwap(sender=router, ...)
         → SwapAllowlistExtension.beforeSwap(sender=router, ...)
              → checks allowedSwapper[pool][router]   ← wrong actor
```

**Pool passes `msg.sender` (the router) as `sender` to the extension:** [1](#0-0) 

**`ExtensionCalling._beforeSwap` forwards that value unchanged:** [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks `sender` (the router) against the allowlist:** [3](#0-2) 

**The router calls `pool.swap()` directly, making itself `msg.sender` of the pool:** [4](#0-3) 

The pool admin intends to allowlist specific user addresses. To also allow those users to trade through the supported periphery router, the admin must allowlist the router address. But `allowedSwapper[pool][router] = true` grants every user — including non-allowlisted ones — the ability to bypass the guard by routing through `MetricOmmSimpleRouter`.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` paths, all of which call `pool.swap()` with `msg.sender = router`. [5](#0-4) 

---

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that protection entirely once the router is allowlisted. Any unprivileged user can execute swaps against the pool's liquidity, draining LP value at oracle-anchored prices that were only intended to be accessible to vetted participants. This is a direct loss of LP principal and a complete failure of the pool's access-control invariant.

---

### Likelihood Explanation

The pool admin must allowlist the router to enable router-mediated swaps for legitimate users — a natural and expected operational step for any curated pool that supports the standard periphery. The bypass requires no special privileges, no malicious setup, and no non-standard tokens: any user can call `MetricOmmSimpleRouter.exactInputSingle()` targeting the pool. Likelihood is **Medium** (requires the admin to allowlist the router, but this is a routine configuration step, not an adversarial one).

---

### Recommendation

The extension should check the **original user** rather than the immediate caller of `pool.swap()`. Two approaches:

1. **Use `recipient` instead of `sender`** — the recipient is the address that receives swap output and is the economically relevant actor. However, `recipient` can also be set to a third party.

2. **Pass the original `msg.sender` through `extensionData`** — the router encodes the original caller in `extensionData`, and the extension decodes and checks it. This requires a coordinated encoding convention.

3. **Preferred: check `recipient` and require `recipient == sender`** — enforce that the router-level caller and the output recipient are the same address, then gate on that address. This is the cleanest invariant for a swap allowlist.

The `DepositAllowlistExtension` does not share this bug because it gates on `owner` (the second parameter), which is the explicit position owner passed by the caller — correctly preserved through the `MetricOmmPoolLiquidityAdder` path. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — intending to let allowlisted users trade via the router.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. attacker (non-allowlisted EOA) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool: pool,
           recipient: attacker,
           ...
       })
  5. Router calls pool.swap(attacker, ...) with msg.sender = router.
  6. Pool calls _beforeSwap(sender=router, ...).
  7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  8. Swap executes. Attacker receives pool output tokens.

Result:
  Non-allowlisted attacker successfully swaps against the curated pool,
  bypassing the SwapAllowlistExtension guard entirely.
```
