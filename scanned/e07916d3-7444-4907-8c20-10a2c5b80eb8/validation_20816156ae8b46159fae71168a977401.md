### Title
SwapAllowlistExtension Checks Router Address Instead of Original User, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router**, not the original user. If the pool admin allowlists the router (the natural configuration to enable router-mediated swaps), every unprivileged user can bypass the allowlist by calling any router entry point.

---

### Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap`, the pool calls:

```solidity
_beforeSwap(
  msg.sender,   // ← direct caller of pool.swap(), not the original user
  recipient,
  ...
);
``` [1](#0-0) 

**Step 2 — The extension checks that `sender` argument.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [2](#0-1) 

`msg.sender` inside the extension is the pool (correct). `sender` is whatever the pool received as its own `msg.sender` — which is the **router** when the user goes through `MetricOmmSimpleRouter`.

**Step 3 — The router calls `pool.swap()` directly, substituting itself as `msg.sender`.**

```solidity
// exactInputSingle
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

The original user's address (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes — it is **never forwarded** to the pool as `sender`. The pool sees `msg.sender = router`.

**Step 4 — The allowlist check therefore resolves against the router, not the user.**

The extension evaluates `allowedSwapper[pool][router]`. If the pool admin has allowlisted the router (the natural configuration to let users swap via the standard periphery), the check passes for **every caller of the router**, regardless of whether that caller is individually allowlisted.

The same substitution occurs in `exactInput` (all hops), `exactOutputSingle`, `exactOutput`, and the recursive `_exactOutputIterateCallback` path — every router entry point calls `pool.swap()` with `msg.sender = router`. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A curated pool that deploys `SwapAllowlistExtension` to restrict trading to a known set of addresses loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker receives real token output from the pool; the pool's LP assets are consumed by an actor the pool admin explicitly intended to exclude. This is a direct loss of LP principal and a complete failure of the pool's core access-control invariant.

---

### Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is the expected operational step for any curated pool that also wants to support the standard periphery: without allowlisting the router, even individually allowlisted users cannot use `exactInputSingle` or any multi-hop path. The pool admin faces a forced choice — either allowlist the router (opening the gate to everyone) or block the router (breaking the standard swap UX for legitimate users). There is no configuration that simultaneously allows router-mediated swaps for approved users and blocks them for unapproved users.

---

### Recommendation

The extension must receive and check the **original user's address**, not the direct caller of `pool.swap()`. Two approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.

2. **Add a `trustedForwarder` registry to the extension**: The extension checks whether `sender` (the direct caller) is a registered trusted forwarder; if so, it decodes the real user from `extensionData` and checks that address instead. This is the ERC-2771 meta-transaction pattern applied to hook guards.

Either way, the extension must never treat the router's address as the identity to gate when the router is a public, permissionless contract.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured in beforeSwap order.
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, charlie, true)

Attack:
  - Charlie (not allowlisted) calls:
      router.exactInputSingle(ExactInputSingleParams{
          pool: curated_pool,
          tokenIn: token0,
          tokenOut: token1,
          zeroForOne: true,
          amountIn: X,
          ...
      })

Execution trace:
  router.exactInputSingle()
    → pool.swap(recipient, zeroForOne, ...) [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (no revert)
      → swap executes, charlie receives token1 output

Result:
  Charlie successfully swaps on a pool that was supposed to block him.
  The allowlist is completely bypassed.
