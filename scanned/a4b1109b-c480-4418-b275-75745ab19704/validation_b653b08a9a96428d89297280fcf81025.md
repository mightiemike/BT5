### Title
`PriceVelocityGuardExtension.beforeSwap()` velocity guard is fully bypassed when `lastMidPriceX64` is zero — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` wraps its entire velocity check inside `if (prevMid != 0)`. Because `lastMidPriceX64` is zero-initialized by Solidity and the factory's `initialize()` path never seeds it, the guard is silently skipped on every pool's first swap. Any unprivileged user who executes that first swap bypasses the configured velocity cap entirely, regardless of how far the oracle price has moved.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap()` reads the stored reference price, writes the new mid-price, and then conditionally enforces the velocity cap:

```solidity
// metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol  lines 53-76
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // 0 on first swap
uint64  prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // state written before the guard
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // ← guard skipped when prevMid == 0
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // squared-change velocity check
        if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
    }
}
```

`PriceVelocityGuardExtension` does not override `initialize()`, so the inherited no-op from `BaseMetricExtension` is called at pool creation:

```solidity
// metric-periphery/contracts/extensions/base/BaseMetricExtension.sol  line 41-43
function initialize(address, bytes calldata) external virtual onlyFactory returns (bytes4) {
    return IMetricOmmExtensions.initialize.selector;
}
```

`lastMidPriceX64` is never seeded. The only setter is `setLastMidPrice`, which is `onlyPoolAdmin`. If the admin does not call it before the pool opens, `prevMid` is 0 on the first swap and the guard is unconditionally skipped.

This is the direct structural analog to the LibDiamond finding: just as `acceptanceTimes[hash] = 0` (either never proposed or reset by `rescindDiamondCut`) passes the `< block.timestamp` check, `lastMidPriceX64 = 0` (never initialized) passes the `!= 0` guard, making the protection fail-open.

---

### Impact Explanation

The velocity guard exists to block swaps when the oracle mid-price has moved faster than the configured cap between blocks — the primary defense against oracle-manipulation attacks that would let a trader extract value from LPs at a stale or manipulated price. On the first swap the guard is entirely absent: a trader can execute at any oracle price, no matter how large the single-block move. After that first swap `lastMidPriceX64` is populated and the guard activates for all subsequent swaps. The window is one swap per pool lifetime, but it is reachable by any unprivileged caller and is guaranteed to exist on every newly deployed pool that uses this extension.

---

### Likelihood Explanation

Every pool that deploys `PriceVelocityGuardExtension` in its `beforeSwap` order and has `maxChangePerBlockE18 > 0` is vulnerable from block 0 until the first swap lands. No special role or permission is required; the first public swap triggers the bypass automatically. The admin can close the window by calling `setLastMidPrice` before the pool opens, but there is no on-chain enforcement of that ordering.

---

### Recommendation

Seed `lastMidPriceX64` inside `initialize()` so the guard is active from the very first swap. Override `initialize` in `PriceVelocityGuardExtension` and require the deployer to supply an initial mid-price:

```solidity
function initialize(address pool, bytes calldata data) external override onlyFactory returns (bytes4) {
    uint128 initialMidPriceX64 = abi.decode(data, (uint128));
    require(initialMidPriceX64 != 0, "PriceVelocityGuard: zero initial price");
    PriceVelocityState storage s = priceVelocityState[pool];
    s.lastMidPriceX64 = initialMidPriceX64;
    s.lastUpdateBlock = uint64(block.number);
    return IMetricOmmExtensions.initialize.selector;
}
```

Alternatively, treat `prevMid == 0` as an uninitialized sentinel and revert rather than skip:

```solidity
if (prevMid == 0) revert PriceVelocityNotInitialized();
```

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` in `beforeSwap` order and `maxChangePerBlockE18 = 1e15` (0.1 % per block).
2. Do **not** call `setLastMidPrice`.
3. Wait one block; oracle price moves 50 % (e.g., due to a real flash crash or oracle manipulation).
4. Call `pool.swap(...)` as any unprivileged user.
5. Inside `beforeSwap`: `prevMid = priceVelocityState[pool].lastMidPriceX64 = 0`.
6. `if (prevMid != 0)` is false → velocity check is skipped entirely.
7. Swap executes at the 50 %-moved oracle price; `lastMidPriceX64` is now set to the new mid.
8. All subsequent swaps are correctly guarded — the bypass is consumed in one shot per pool lifetime.
