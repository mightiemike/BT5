### Title
`DepositAllowlistExtension` Gates Position Owner Instead of the Actual Payer, Allowing Unauthorized Deposits via the Operator Pattern — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` argument (position holder) against the allowlist, but the pool's `addLiquidity` explicitly supports an operator pattern where `msg.sender` (the payer) differs from `owner`. An unauthorized user can deposit funds into a curated pool by supplying an allowlisted address as `owner` while paying themselves, bypassing the deposit gate entirely.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The pool's `addLiquidity` explicitly documents and enforces the operator pattern — `msg.sender` pays but need not equal `owner`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-195
function addLiquidity(address owner, ...) external nonReentrant(...) {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(..., owner, ...);
    ...
}
``` [2](#0-1) 

The `MetricOmmPoolLiquidityAdder` makes this trivially reachable: its `addLiquidityExactShares(pool, owner, ...)` overload accepts an arbitrary `owner` while always using `msg.sender` as the payer:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L56-68
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [3](#0-2) 

The extension's own naming — mapping `allowedDepositor`, setter `setAllowedToDeposit`, NatSpec "Gates `addLiquidity` by depositor address" — all indicate the intended gate is the depositing economic actor (payer/sender), not the position holder. The implementation checks the wrong actor. [4](#0-3) 

---

### Impact Explanation

An unauthorized user (Bob) can deposit funds into a curated pool that has `DepositAllowlistExtension` configured:

1. Bob calls `LiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, ...)` where Alice is allowlisted.
2. The adder calls `pool.addLiquidity(alice, ...)` with `msg.sender = LiquidityAdder`.
3. The extension checks `allowedDepositor[pool][alice]` → true → passes.
4. The callback pulls tokens from Bob (the payer stored in transient context).
5. Alice receives the position; Bob's funds are now in the pool.

The pool admin's curation policy — restricting which economic actors can provide liquidity — is fully bypassed. Any unauthorized user can inject funds into the pool by nominating any allowlisted address as `owner`. This undermines the purpose of the allowlist (e.g., KYC/AML compliance, curated LP pools) and constitutes an admin-boundary break via an unprivileged path.

---

### Likelihood Explanation

The `MetricOmmPoolLiquidityAdder` is the documented production periphery for EOA deposits and explicitly exposes the `owner ≠ msg.sender` overload as a supported public entrypoint. No special privileges, malicious setup, or non-standard tokens are required. Any user who knows an allowlisted address can execute this in a single transaction. [5](#0-4) 

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate by position owner (not payer), rename the mapping and setter to `allowedOwner`/`setAllowedOwner` and update documentation to match, so pool admins are not misled about what the allowlist actually enforces.

---

### Proof of Concept

```
Setup:
  - Pool configured with DepositAllowlistExtension
  - Alice (0xAlice) is allowlisted: allowedDepositor[pool][alice] = true
  - Bob (0xBob) is NOT allowlisted

Attack:
  1. Bob approves LiquidityAdder for token0/token1
  2. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // owner = allowlisted address
           salt,
           deltas,
           maxAmount0,
           maxAmount1,
           ""
       )
  3. Pool calls _beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)
  4. Extension checks allowedDepositor[pool][alice] → true → no revert
  5. Callback pulls tokens from Bob (payer in transient context)
  6. Alice receives position shares; Bob's funds are in the pool

Result:
  - Bob (unauthorized) has deposited into a curated pool
  - The DepositAllowlistExtension did not block him
  - Pool admin's curation policy is violated
``` [1](#0-0) [6](#0-5)
