### Title
Router-Mediated Swaps Pass Router Address as `sender` to `SwapAllowlistExtension::beforeSwap`, Breaking Allowlist Identity Check — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension::beforeSwap` hook gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `MetricOmmPool::swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, so `sender` is the router's address — not the original user's address. The allowlist check therefore operates on the wrong identity, making it impossible to correctly gate individual users through the router.

---

### Finding Description

**Exact call chain:**

```
User
  → MetricOmmSimpleRouter::exactInputSingle / exactInput / exactOutputSingle / exactOutput
      → IMetricOmmPoolActions(pool).swap(...)          // msg.sender to pool = router
          → MetricOmmPool::swap()
              → _beforeSwap(msg.sender=router, ...)    // sender = router, not user
                  → SwapAllowlistExtension::beforeSwap(sender=router, ...)
                      → allowedSwapper[pool][router]   // WRONG: should be allowedSwapper[pool][user]
```

In `MetricOmmPool::swap`, the pool passes `msg.sender` (the router) as `sender` to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` encodes that value and forwards it to the extension: [2](#0-1) 

The extension then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = router: [3](#0-2) 

In the router, `msg.sender` (the original user) is never forwarded to the pool — the router calls `pool.swap(...)` directly, making itself the pool's `msg.sender`: [4](#0-3) 

**Two concrete failure modes result:**

**Mode A — Allowlist bypass:** A pool admin allowlists the router address (a natural action to enable router-mediated swaps for allowlisted users). Because `sender = router` for every router-mediated swap, any unprivileged user can bypass the allowlist simply by routing through `MetricOmmSimpleRouter`. The allowlist is completely ineffective.

**Mode B — Allowlisted users blocked:** A pool admin allowlists specific user addresses but does not allowlist the router. Allowlisted users who attempt to swap via the router are rejected because `allowedSwapper[pool][router]` is `false`. The router is unusable for any allowlisted pool.

---

### Impact Explanation

The `SwapAllowlistExtension` is the sole mechanism for restricting swap access on a pool. Its failure means either:
- Unauthorized parties execute swaps on a pool that was designed to be restricted (Mode A), potentially draining LP value or executing at unfavorable prices the pool admin did not intend to permit.
- Legitimate allowlisted users cannot use the canonical router, breaking core swap functionality (Mode B).

Both outcomes represent broken core pool functionality. Mode A in particular can lead to direct value leakage if the pool is configured to allow only trusted counterparties (e.g., market makers with agreed-upon price ranges).

---

### Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects router-mediated swaps to respect the allowlist is affected. The router is the primary public entrypoint for end users. The misconfiguration is not obvious — a pool admin who allowlists the router to "enable router swaps" will unknowingly open the pool to all users.

---

### Clarification on the "Paused Pool" Framing

The question's framing around "pool paused for swaps but LP withdrawals remain live" is **not a contributing factor**. The `whenNotPaused` modifier in `MetricOmmPool::swap` reverts before `_beforeSwap` is ever called when `pauseLevel != 0`: [5](#0-4) [6](#0-5) 

A paused pool blocks all swaps unconditionally; the allowlist is irrelevant in that state. The `OracleValueStopLossExtension` is an `afterSwap` hook and is also a separate concern from the allowlist. The real vulnerability is purely the router identity mismatch in `beforeSwap`.

---

### Recommendation

Pass the original user's address through the call chain. One approach: have the router encode the original `msg.sender` in `extensionData` and have the extension decode it. A cleaner protocol-level fix is to add an `originator` field to the swap call that the pool passes to hooks alongside `sender`, so extensions can distinguish the immediate caller from the end user.

---
