### Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

The `SwapAllowlistExtension.beforeSwap` hook gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The allowlist therefore checks the router's address rather than the economically relevant actor. If the pool admin allowlists the router (the only way to let any user swap through it), every user — including those explicitly excluded from the allowlist — can bypass the restriction by routing through the router.

---

### Finding Description

**Call chain when routing through the router:**

```
User → MetricOmmSimpleRouter.exactInput*(...)
         └─ pool.swap(recipient=user, ...)   ← msg.sender = router
               └─ _beforeSwap(sender=router, recipient=user, ...)
                     └─ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                           └─ isAllowedToSwap(pool, router)   ← checks router, not user
```

In `MetricOmmPool.swap`, the pool captures `msg.sender` and forwards it as `sender` to every `beforeSwap` hook: [1](#0-0) 

`ExtensionCalling._beforeSwap` then encodes that `sender` value directly into the hook call: [2](#0-1) 

The `SwapAllowlistExtension` (confirmed by its unit test) gates on the `sender` parameter — the first argument — not on the originating user: [3](#0-2) 

**The mismatch:** The pool's `sender` is the router when the user routes through `MetricOmmSimpleRouter`. The allowlist therefore evaluates the router's allowlist status, not the user's. Two broken outcomes follow:

1. **Allowlisted users cannot use the router** — because the router is not on the allowlist, their router-mediated swaps revert even though they are individually permitted.
2. **Non-allowlisted users bypass the allowlist** — if the pool admin allowlists the router (the only fix for outcome 1), every user can call `MetricOmmSimpleRouter` and the hook passes unconditionally, rendering the allowlist inert.

This is structurally identical to the external H-2 bug: a value that represents one thing (the router address / debt shares) is used in a guard that was designed to evaluate a different thing (the actual user / debt assets), causing the guard to either block legitimate actors or pass illegitimate ones.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` for compliance, KYC, or access-control purposes loses its enforcement guarantee the moment the router is allowlisted. Any address — including those the pool admin explicitly excluded — can execute swaps against the pool's liquidity by routing through `MetricOmmSimpleRouter`. This is a direct, unprivileged bypass of an admin-configured access boundary with fund-impacting consequences: unauthorized parties can drain arbitrage value or execute swaps the pool owner intended to prohibit.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard periphery entry point for end users. Any pool that deploys a swap allowlist and also wants to support router-mediated swaps (the common case) must allowlist the router, triggering the bypass. The attacker needs no special privileges — only the ability to call the public router.

---

### Recommendation

The `SwapAllowlistExtension` must gate on the identity of the economic actor, not the immediate caller. Two viable approaches:

1. **Check `recipient` instead of `sender`** — the recipient is the address that receives output tokens and is typically the actual user even when routing through the router. This requires the pool admin to allowlist recipients rather than senders.

2. **Pass the originating user through `extensionData`** — the router encodes the real caller into `extensionData`; the extension decodes and checks that address. This requires a coordinated convention between the router and the extension.

The simplest correct fix is to change the allowlist lookup from `sender` to `recipient` in `SwapAllowlistExtension.beforeSwap`, since `recipient` is the address the router sets to the actual user.

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; alice is NOT allowlisted; router IS allowlisted
// (admin had to allowlist the router so that bob, who IS allowlisted, can use it)

// Step 1: alice tries a direct swap — correctly blocked
vm.prank(alice);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(alice, true, 1000, 0, "", "");

// Step 2: alice routes through MetricOmmSimpleRouter — bypass succeeds
// sender seen by the hook = router (allowlisted), not alice (not allowlisted)
vm.prank(alice);
router.exactInputSingle(
    MetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: alice,
        extensionData: ""
    })
);
// alice's swap executes despite being excluded from the allowlist
