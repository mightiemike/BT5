### Title
`SwapAllowlistExtension` gates the router address instead of the real user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the value `MetricOmmPool.swap` passes as `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the real user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. If the pool admin allowlists the router so that their permitted users can trade through it, every unpermissioned user gains the same access by calling the same public router.

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // <-- immediate caller of pool.swap
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` uses that value to look up the allowlist:

```solidity
// SwapAllowlistExtension.sol line 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap` directly:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

The pool sees `msg.sender == router`, so `sender == router` reaches the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][real_user]`.

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists specific users.
2. To let those users trade through the public router, the admin also calls `setAllowedToSwap(pool, router, true)`.
3. Any unpermissioned user now calls `router.exactInputSingle(pool, ...)`. The extension sees `sender == router`, finds it allowlisted, and passes. The swap executes.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap` with the router as `msg.sender`.

### Impact Explanation

The swap allowlist is the primary access-control mechanism for permissioned pools. Bypassing it lets any unpermissioned address trade against a pool that was explicitly restricted. Depending on pool configuration this can:

- Drain LP value by allowing adversarial or high-frequency traders the pool admin explicitly excluded.
- Violate regulatory or compliance constraints the pool was designed to enforce.
- Allow sandwich or front-running actors the admin had blocked to re-enter.

This is a direct loss-of-access-control impact on LP principal and pool integrity, matching the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" gate.

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any EOA or contract can call it.
- The bypass requires only that the pool admin has allowlisted the router (the natural and expected configuration for a pool that wants its allowlisted users to trade through the standard router).
- No privileged access, no special tokens, no flash loans, no oracle manipulation required.
- The attacker needs only to call `exactInputSingle` with the target pool address.

### Recommendation

The pool must communicate the original user's address to the extension, not the immediate `msg.sender`. Two options:

**Option A (preferred):** Add an `originator` field to the swap callback data or to a transient storage slot in the pool, and pass it as a separate argument to `_beforeSwap`. The router already stores the real payer in transient storage (`_setNextCallbackContext(..., msg.sender, ...)`); the pool could read it or the router could encode it in `extensionData`.

**Option B:** Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the extension level by reverting when `msg.sender` (the pool) is called from a context where `sender` is a known router. This is fragile and not recommended.

The cleanest fix is for `MetricOmmPool.swap` to accept an explicit `originator` parameter (defaulting to `msg.sender` for direct calls) and pass it as `sender` to extensions, while the router supplies `msg.sender` (the real user) as that originator.

### Proof of Concept

```

  - Deploy pool with SwapAllowlistExtension (extension1, beforeSwap order = 1).
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allow router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls router.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      zeroForOne: true,
      amountIn: X,
      ...
  });

Trace:
  router.exactInputSingle
    → pool.swap(recipient, ...) [msg.sender = router]
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router)
          → allowedSwapper[pool][router] == true  ✓ (passes)
      → swap executes, attacker receives output tokens

Result: attacker bypasses the allowlist and executes a swap in a pool
        they were explicitly excluded from.
