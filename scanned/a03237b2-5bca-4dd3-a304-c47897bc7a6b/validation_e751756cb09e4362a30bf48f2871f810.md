### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user. If the pool admin allowlists the router (a natural step to support router-based swaps for permitted users), every unprivileged user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
    extensionData
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension = pool):

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router is the entity that calls `pool.swap()`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

So `sender` arriving at the extension is `address(router)`, not the actual end-user. The allowlist lookup becomes `allowedSwapper[pool][router]`.

**The contradictory predicate (analog to `IsFullyWithdrawn`):** The extension is named and documented as gating individual *swappers* (`setAllowedToSwap(pool, swapper, …)`), yet the identity it actually checks when the router is in the call path is the router's address. These two things cannot both be true simultaneously: you cannot gate individual users while the check resolves to a shared intermediary address. Any pool admin who allowlists the router to let permitted users trade via the router simultaneously opens the gate to every unpermitted user.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position beneficiary), not `sender` (the caller), demonstrating that the protocol does distinguish between the economic actor and the call-chain intermediary — but `SwapAllowlistExtension` does not apply the same logic.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers) is completely bypassed the moment the router is added to the allowlist. Any unpermitted user can:

1. Call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` targeting the restricted pool.
2. The extension sees `sender = router`, which is allowlisted, and passes.
3. The swap executes as if the user were permitted.

This breaks the core curation invariant of the pool, allows unauthorized trading, and can drain LP value from a pool that was supposed to be restricted to trusted counterparties. The loss is direct and repeatable.

---

### Likelihood Explanation

The trigger condition — the router being allowlisted — is not exotic. A pool admin who wants to support router-based swaps for their permitted users *must* allowlist the router, because without it, even permitted users cannot trade through the router (their individual addresses are not `sender`). The admin faces a forced choice: either block all router usage or open the allowlist to everyone. There is no correct configuration that achieves both goals with the current implementation. This makes the vulnerable state a natural operational outcome, not a misconfiguration edge case.

---

### Recommendation

The extension must resolve the actual end-user identity rather than the immediate caller. Two viable approaches:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` (the real user) into `extensionData`; the extension decodes and checks it. This requires a convention between the router and the extension.

2. **Mirror the deposit allowlist pattern:** Introduce a separate `recipient` or `originator` field that the pool populates with the economically relevant actor, analogous to how `beforeAddLiquidity` receives `owner` rather than `sender`.

At minimum, the NatSpec of `SwapAllowlistExtension` must warn that allowlisting the router grants access to all router users, not just individually permitted ones.

---

### Proof of Concept

```
Setup:
  pool  = MetricOmmPool with SwapAllowlistExtension (extension1)
  admin = pool admin
  alice = KYC-approved user
  bob   = unpermitted attacker

  admin calls swapExt.setAllowedToSwap(pool, alice, true)
  admin calls swapExt.setAllowedToSwap(pool, router, true)
    ↳ intent: alice can trade directly or via router
    ↳ actual effect: every user can trade via router

Step 2 – Bob bypasses allowlist:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
  })

Step 3 – Call chain:
  router.exactInputSingle()
    → pool.swap(msg.sender=router, ...)
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓ passes
      → swap executes, bob receives output tokens

Result: bob trades on a pool he is not individually permitted to access.
