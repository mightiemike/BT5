### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router address, not the actual end user. If the pool admin whitelists the router (the only way to allow legitimate users to swap through it), every user—including unauthorized ones—bypasses the per-user restriction entirely.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value verbatim as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks whether `sender` (the direct caller of `pool.swap()`) is in the per-pool allowlist, using `msg.sender` (the pool) as the mapping key: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` (or any multi-hop variant) calls `pool.swap()`, the pool's `msg.sender` is the router, so `sender = router`: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`. The pool admin faces an inescapable dilemma:

| Admin choice | Effect |
|---|---|
| Do **not** whitelist the router | No user can swap through the router for this pool |
| Whitelist the router | **Every** user, including unauthorized ones, bypasses the per-user restriction |

There is no configuration that simultaneously allows legitimate users to use the router and blocks unauthorized users.

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified addresses, institutional partners, or protocol-controlled addresses) loses that restriction entirely once the router is whitelisted. Any unprivileged address can call `router.exactInputSingle()` and execute swaps in the restricted pool. This breaks the core allowlist invariant: "swap allowlist checks must cover the exact actor/action intended and cannot be bypassed through router." Unauthorized swaps can move pool price, extract value from LPs through adversarial trading, and violate any regulatory or contractual access-control requirement the pool admin intended to enforce.

### Likelihood Explanation

The bypass is unconditional and requires no special setup beyond the pool admin having whitelisted the router—a step that is necessary for any legitimate user to use the standard periphery. The router is a public, permissionless contract. Any address can call it.

### Recommendation

The extension must check the actual end user, not the direct caller of `pool.swap()`. Two viable approaches:

1. **Pass end user in `extensionData`**: The router encodes `msg.sender` (the end user) into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Dedicated allowlist field**: Add a `swapOriginator` field to the pool's swap hook signature (analogous to how `addLiquidity` separates `sender` from `owner`), so the router can forward the real user address as a distinct, trusted parameter.

The `DepositAllowlistExtension` avoids this problem because `addLiquidity` already separates `sender` (the caller/payer) from `owner` (the position beneficiary), and the extension checks `owner`—the address that actually receives the position. The swap path has no equivalent separation. [5](#0-4) 

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension configured in beforeSwap order.
2. Pool admin calls extension.setAllowedToSwap(pool, allowedUser, true)
   — only allowedUser should be able to swap.
3. Pool admin calls extension.setAllowedToSwap(pool, router, true)
   — necessary so allowedUser can use the router.
4. unauthorizedUser calls router.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...); pool's msg.sender = router.
6. Pool calls extension.beforeSwap(router, ...).
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Swap executes for unauthorizedUser despite not being on the allowlist.
```

The bypass is also reachable through `exactInput`, `exactOutputSingle`, and `exactOutput` on the same router, and through any other contract that calls `pool.swap()` directly, since the check is always on the direct caller, never on the originating EOA.
