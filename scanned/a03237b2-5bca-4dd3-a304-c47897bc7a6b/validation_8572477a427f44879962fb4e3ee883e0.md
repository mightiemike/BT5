### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address" and its setter is named `setAllowedToDeposit(pool_, address depositor, bool allowed)`. However, the `beforeAddLiquidity` hook silently discards the `sender` argument (the actual caller who pays tokens) and instead checks `owner` (the LP position holder). Because `addLiquidity` lets any caller supply an arbitrary `owner`, any unprivileged address can bypass the allowlist by setting `owner` to an already-allowlisted address.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` parameter as `owner` to the extension: [1](#0-0) 

The extension receives both actors, but `DepositAllowlistExtension.beforeAddLiquidity` silently drops `sender` (first argument, unnamed `address`) and gates only on `owner`: [2](#0-1) 

The storage key and setter both use the name `depositor`, making the intent unambiguous — the guard is supposed to vet the caller, not the position holder: [3](#0-2) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the swap caller), confirming the intended pattern: [4](#0-3) 

Because `addLiquidity` imposes no restriction on who may supply an arbitrary `owner`: [5](#0-4) 

any caller can set `owner` to an allowlisted address and the extension will pass. The `MetricOmmPoolLiquidityAdder` makes this even more accessible — its `addLiquidityExactShares(pool, owner, ...)` overload accepts any non-zero `owner` with no further restriction: [6](#0-5) 

The existing test suite does not catch this because every test that exercises the allowlist uses a setup where `sender == owner` (the TestCaller is both the caller and the position holder), so the wrong field happens to produce the correct result: [7](#0-6) 

---

### Impact Explanation

An unprivileged caller bypasses the pool admin's deposit allowlist — an admin-boundary break. The caller pays the input tokens; the allowlisted address receives the LP position. Consequences include:

- Unauthorized liquidity provision to pools the admin intended to curate (e.g., regulatory-restricted or manipulation-resistant pools).
- Forced LP positions on allowlisted addresses without their consent (griefing).
- If the pool admin's intent was to prevent a specific actor from influencing pool depth (e.g., to stop sandwich-attack setups), the bypass nullifies that protection entirely.

---

### Likelihood Explanation

Exploitation requires only a direct call to `pool.addLiquidity` or `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with `owner` set to any already-allowlisted address. No special privileges, flash loans, or multi-step setup are needed. Any public user can execute this in a single transaction.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor/caller) instead of `owner`, matching the documented intent and the pattern used by `SwapAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol

function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` attached to `beforeAddLiquidity`.
2. Pool admin calls `depositExtension.setAllowedToDeposit(pool, alice, true)` — only `alice` is allowlisted.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly.
   - Pool passes `sender = bob`, `owner = alice` to the extension.
   - Extension evaluates `allowedDepositor[pool][alice]` → `true`.
   - `bob` pays the tokens; `alice` receives the LP position.
4. `bob` has deposited into a curated pool without being allowlisted, violating the admin's access policy.

The same bypass works through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, ...)` where `bob` is `msg.sender` and `alice` is the supplied `owner`.
