### Title
SwapAllowlistExtension Gates the Router Address Instead of the Originating User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the first argument forwarded by the pool — which is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` mediates a swap, it is the direct caller of `pool.swap()`, so the extension sees the router's address as `sender`, not the originating user. Any pool admin who allowlists the router to enable router-mediated swaps on a curated pool inadvertently opens the allowlist to every user who can call the router.

---

### Finding Description

**Call chain for a direct swap:**
```
user → pool.swap()
  pool passes msg.sender = user → _beforeSwap(sender=user, ...)
  extension checks allowedSwapper[pool][user]   ✓ correct
```

**Call chain through the router:**
```
user → MetricOmmSimpleRouter.exactInputSingle()
  router → pool.swap(recipient, ...)
  pool passes msg.sender = router → _beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router]  ✗ wrong actor
```

In `MetricOmmPool.swap`, the pool unconditionally passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks the `sender` argument (the router) against the per-pool allowlist: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no forwarding of the original `msg.sender`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and wants to support the official router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the extension's per-user gate is completely bypassed: any address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and the extension will pass because it only sees the router's address. The intended curation — restricting which counterparties can trade against the pool's LP liquidity — is silently defeated. LP funds are exposed to unrestricted swap flow, which is a direct loss of the policy protection the pool admin paid to configure.

---

### Likelihood Explanation

The scenario is not hypothetical. A pool admin who wants both a swap allowlist and router support must allowlist the router; there is no other supported path. The `MetricOmmSimpleRouter` is the canonical periphery entry point documented for end users. Any production deployment of a curated pool that also supports the router will trigger this bypass. The attacker needs no special privilege — only the ability to call the public router.

---

### Recommendation

The router must forward the originating user's identity to the pool so the extension can gate the correct actor. Two complementary fixes:

1. **Pool-side:** Add an optional `originator` field to the `swap` call signature (or a transient-storage slot set by the router before calling the pool) that the pool passes as `sender` to extensions instead of `msg.sender`.

2. **Extension-side (short-term):** `SwapAllowlistExtension.beforeSwap` should accept an `extensionData`-encoded originator address when `sender` is a known router, and fall back to checking that decoded address. This is fragile but deployable without a core change.

The cleanest fix is option 1: the pool should distinguish between the immediate caller (used for callback settlement) and the economic originator (used for extension gating), mirroring how `addLiquidity` already separates `msg.sender` (payer/sender) from `owner` (position beneficiary).

---

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin also allowlists the router so router swaps work.
extension.setAllowedToSwap(pool, allowedUser, true);
extension.setAllowedToSwap(pool, address(router), true); // required for router support

// Attack: blockedUser routes through the router.
// The extension sees sender = address(router), which IS allowlisted → passes.
vm.prank(blockedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             pool,
        recipient:        blockedUser,
        tokenIn:          token0,
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:    0,
        deadline:         block.timestamp,
        extensionData:    ""
    })
);
// Swap succeeds. blockedUser traded on a pool they were explicitly excluded from.
// The allowlist check passed because allowedSwapper[pool][router] == true,
// and sender == address(router), not blockedUser.
