### Title
`SwapAllowlistExtension` checks the router address as `sender` instead of the actual user, making per-user swap restrictions bypassable or permanently broken when `MetricOmmSimpleRouter` is used — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the **router contract**, not the actual user. This makes the per-user allowlist either permanently broken (allowlisted users blocked through the router) or trivially bypassable (any user passes if the router is allowlisted), while the analogous `DepositAllowlistExtension` correctly checks the position `owner` and is immune to this problem.

### Finding Description

**Root cause — wrong identity checked in `SwapAllowlistExtension`:**

`SwapAllowlistExtension.beforeSwap` reads the first argument (`sender`) and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (the pool calls the extension). `sender` is the first argument forwarded by the pool, which is always `msg.sender` of `pool.swap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- this becomes `sender` in the extension
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()`, so `msg.sender` of `pool.swap()` is the **router address**. The extension therefore checks whether the **router** is allowlisted, not the actual user.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` (second argument), which is the actual position owner passed explicitly by the caller — not `msg.sender` of `addLiquidity`. This correctly identifies the real user regardless of whether `MetricOmmPoolLiquidityAdder` or a direct call is used:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The pool passes `owner` as a separate, caller-supplied argument to `addLiquidity`, so the deposit allowlist is router-agnostic. No equivalent separate "actual user" argument exists for `swap` — the only identity available is `msg.sender` of `pool.swap()`, which collapses to the router.

**Two concrete broken invariants:**

**Invariant 1 — Bypass:** Pool admin allowlists `MetricOmmSimpleRouter` (a natural action: "allow the official router so users can swap"). Any non-allowlisted user calls `exactInputSingle()` → router calls `pool.swap()` → `sender` = router → router is allowlisted → `NotAllowedToSwap` is never raised → unauthorized user swaps successfully. The per-user allowlist is completely defeated.

**Invariant 2 — Broken functionality:** Pool admin allowlists specific user addresses (e.g., `userA`) but not the router. `userA` calls `MetricOmmSimpleRouter.exactInputSingle()` → `sender` = router → router not in allowlist → `NotAllowedToSwap` revert. `userA` is allowlisted but cannot use the router at all, making the primary swap interface unusable for them.

### Impact Explanation

**Bypass path:** An unprivileged, non-allowlisted user can swap in a pool that is supposed to be restricted (e.g., private pool, KYC-gated pool, market-maker-only pool). They can extract value from LPs at oracle-anchored prices that the pool admin intended to reserve for trusted counterparties. This is a direct loss of LP assets and a broken admin-boundary invariant.

**Broken-functionality path:** Allowlisted users are silently blocked from using `MetricOmmSimpleRouter`, the primary user-facing swap interface. Their only recourse is to implement a custom swap callback and call `pool.swap()` directly — a non-trivial requirement that effectively makes the swap flow unusable for them.

### Likelihood Explanation

Both paths are reachable by any user without any privileged setup:

- The bypass requires only that the pool admin allowlists the router — a natural and expected action for any pool that wants to support the official periphery.
- The broken-functionality path requires only that the pool admin allowlists individual users (the primary intended use of the extension) and that those users call through the router (the primary intended swap interface).

No malicious initial setup, non-standard tokens, or oracle manipulation is required.

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to accept the actual user identity through `extensionData` (caller-supplied), or redesign the check to mirror `DepositAllowlistExtension`'s approach. Since `swap` has no separate caller-supplied "owner" field, the cleanest fix is to require callers (including the router) to ABI-encode the actual user address in `extensionData` and verify it there, or to add a dedicated `swapper` field to the pool's `swap` signature analogous to `owner` in `addLiquidity`.

As an interim measure, document clearly that `SwapAllowlistExtension` gates by the **immediate caller of `pool.swap()`** (i.e., the router address, not the end user), so pool admins understand they must allowlist the router — not individual users — when using the periphery.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][userA] = true   // only userA is allowed
  allowedSwapper[pool][router] = true  // router also allowlisted (natural admin action)

Attack (Bypass):
  userB (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
      → pool.swap(recipient, ...) called with msg.sender = router
      → _beforeSwap(router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router, ...)
      → allowedSwapper[pool][router] == true → passes
      → userB swaps successfully despite not being allowlisted

Broken Functionality:
  allowedSwapper[pool][router] = false  // router not allowlisted
  userA (allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
      → pool.swap() called with msg.sender = router
      → SwapAllowlistExtension.beforeSwap(sender=router, ...)
      → allowedSwapper[pool][router] == false → NotAllowedToSwap revert
      → userA blocked despite being explicitly allowlisted
```
