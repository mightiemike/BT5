Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end-user, neutralizing per-user swap restrictions for router-mediated swaps - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When users route through `MetricOmmSimpleRouter`, the pool receives `sender = router address`. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the pool to every user, because the extension cannot distinguish individual end-users passing through the router. The per-user allowlist is completely neutralized.

## Finding Description
`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct), and `sender` is the first argument forwarded by `MetricOmmPool._beforeSwap`, which passes `msg.sender` of the pool's `swap()` call:

```solidity
_beforeSwap(
    msg.sender,   // ŌåÉ router address when called via router
    recipient,
    ...
);
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly, so the pool sees `sender = router address`: [3](#0-2) 

The same identity collapse occurs in `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). [4](#0-3) 

Once the router is allowlisted via `setAllowedToSwap(pool, address(router), true)`, the extension cannot distinguish between different end-users passing through it. Every user ŌĆö including those explicitly not in `allowedSwapper[pool]` ŌĆö can swap freely by routing through `MetricOmmSimpleRouter`. [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the economically relevant identity ŌĆö the position owner), not on `sender` (the immediate caller), so it does not share this flaw: [6](#0-5) 

The swap extension has no equivalent "economically relevant identity" parameter to fall back on, creating an asymmetry between the two allowlist extensions.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., a private institutional pool) is fully open to any user once the router is allowlisted. Unauthorized users can trade against the pool's LPs, extracting value through adverse selection or consuming liquidity the LPs did not intend to provide to the public. This constitutes broken core pool functionality (the allowlist does not restrict actual users for router-mediated swaps) and direct loss of LP principal ŌĆö a High-severity impact under Sherlock thresholds.

## Likelihood Explanation
The trigger is a pool admin allowlisting the router ŌĆö a natural and expected configuration step for any pool that wants to support the standard periphery UX. The admin has no reason to suspect this opens the gate to all users; `setAllowedToSwap` gives no indication that allowlisting a router-type contract is semantically different from allowlisting an EOA. The bypass requires no privileged access, no special tokens, and no flash loan; any user with the router address can exploit it in a single transaction.

## Recommendation
Gate on the actual end-user identity rather than the immediate caller. The cleanest fix is to have the router encode `msg.sender` into `extensionData` (e.g., `abi.encode(msg.sender)`) before forwarding to the pool, and have `SwapAllowlistExtension.beforeSwap` decode and verify it as the authoritative swapper identity. Alternatively, add an explicit guard in `setAllowedToSwap` that reverts or emits a warning if the target address is a registered periphery contract (router), preventing the unsafe configuration from being set silently.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, router allowlisted, Eve not allowlisted
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// Eve is NOT in allowedSwapper[pool]

// Eve bypasses the allowlist via the router
vm.prank(eve);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          address(token1),
        tokenOut:         address(token0),
        zeroForOne:       false,
        amountIn:         1_000,
        amountOutMinimum: 0,
        recipient:        eve,
        deadline:         block.timestamp + 1,
        priceLimitX64:    type(uint128).max,
        extensionData:    ""
    })
);
// Succeeds: pool.swap() sees sender=router, router is allowlisted, Eve trades freely.

// Direct call from Eve reverts as expected
vm.prank(eve);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(eve, false, 1_000, type(uint128).max, "", "");
// The asymmetry confirms the allowlist checks the wrong identity for router-mediated swaps.
```
