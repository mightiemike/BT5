### Title
Cross-Subaccount Self-Liquidation Allows Insurance Fund Drain - (File: `core/contracts/ClearinghouseLiq.sol`)

### Summary
`liquidateSubaccountImpl` only checks that `txn.sender != txn.liquidatee` (exact subaccount identity), but does not check that the owner address embedded in `txn.sender` differs from the owner address embedded in `txn.liquidatee`. A user who controls two subaccounts can use a healthy second subaccount to liquidate their own underwater subaccount, capturing the liquidation discount themselves and forcing the insurance fund to absorb the collateral shortfall.

### Finding Description

In Nado, every subaccount is a `bytes32` where the first 20 bytes are the owner's Ethereum address and the last 12 bytes are a name suffix:

```solidity
// Endpoint.sol:108-110
bytes32 subaccount = bytes32(
    abi.encodePacked(msg.sender, subaccountName)
);
```

A single address can therefore own multiple distinct subaccounts (e.g., `alice+"default"` and `alice+"second"`). The liquidation guard in `liquidateSubaccountImpl` is:

```solidity
// ClearinghouseLiq.sol:602
require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
```

This only rejects the case where the exact same `bytes32` appears on both sides. It does **not** reject the case where `bytes20(txn.sender) == bytes20(txn.liquidatee)` — i.e., two different subaccounts owned by the same address.

Compare this with `transferQuote`, which correctly enforces same-owner-only transfers:

```solidity
// Clearinghouse.sol:222-225
require(
    bytes20(txn.sender) == bytes20(txn.recipient),
    ERR_UNAUTHORIZED
);
```

No equivalent owner-equality guard exists in the liquidation path.

When a spot or spread liability is liquidated and the liquidatee's quote balance goes negative, the insurance fund covers the gap:

```solidity
// ClearinghouseLiq.sol:538-543
if (txn.amount < 0) {
    insurance = spotEngine.updateQuoteFromInsurance(
        txn.liquidatee,
        insurance
    );
}
```

The liquidator (subaccount B, same owner) simultaneously receives the asset at the discounted liquidation price:

```solidity
// ClearinghouseLiq.sol:530-536
spotEngine.updateBalance(txn.productId, txn.sender, txn.amount);
spotEngine.updateBalance(
    QUOTE_PRODUCT_ID,
    txn.sender,
    -v.liquidationPayment - v.liquidationFees
);
```

Because `v.liquidationPayment = liquidationPriceX18 * amount` and `liquidationPriceX18 < oraclePriceX18`, the liquidator buys below market. The discount that is supposed to incentivize external liquidators is instead captured by the same user.

### Impact Explanation

**Impact: High.**

A user with an underwater subaccount A and a funded subaccount B (same owner address) can:

1. Submit a `LiquidateSubaccount` transaction with `sender = B`, `liquidatee = A`.
2. Subaccount B acquires A's collateral at the liquidation discount (below oracle price).
3. If A's quote balance is insufficient to cover the liability buyback, `updateQuoteFromInsurance` draws from the insurance fund to make up the difference.
4. Net effect: the user retains the discounted asset in B, and the insurance fund absorbs the loss that should have been borne by the user.

Repeated across multiple positions or users, this drains the insurance fund — the last-resort solvency backstop for the entire protocol.

### Likelihood Explanation

**Likelihood: Low.**

The attacker must:
- Have a second subaccount with enough capital to pass the post-liquidation health check (`!isUnderInitial(txn.sender)` at line 575).
- Have their first subaccount genuinely underwater (`isUnderMaintenance(txn.liquidatee)` at line 603).
- Craft and sign a valid `LiquidateSubaccount` transaction accepted by the sequencer.

These conditions are realistic for a sophisticated user who deliberately positions themselves to exploit the gap.

### Recommendation

Add an owner-address equality check in `liquidateSubaccountImpl`, mirroring the pattern already used in `transferQuote`:

```solidity
require(
    bytes20(txn.sender) != bytes20(txn.liquidatee),
    ERR_UNAUTHORIZED
);
```

This ensures that no user can act as both liquidator and liquidatee regardless of which subaccount names they use.

### Proof of Concept

1. Alice deposits into `alice+"default"` (subaccount A) and opens a large spot liability position.
2. Alice deposits into `alice+"second"` (subaccount B) with sufficient USDC.
3. The oracle price drops; subaccount A's maintenance health falls below zero.
4. Alice signs a `LiquidateSubaccount` transaction: `sender = alice+"second"`, `liquidatee = alice+"default"`, `amount = <full liability>`.
5. `liquidateSubaccountImpl` passes the check `txn.sender != txn.liquidatee` (different `bytes32`).
6. `_handleLiquidationPayment` transfers the spot asset from A to B at `liquidationPriceX18 < oraclePriceX18`.
7. A's quote balance goes negative; `updateQuoteFromInsurance` draws from the insurance fund.
8. Alice's net position: she still holds the asset (now in B) at a discount, and the insurance fund absorbed the shortfall she would otherwise have owed.
