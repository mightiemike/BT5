### Title
Linked Signer Can Drain User Collateral Without Per-Transaction User Consent — (`File: core/contracts/EndpointTx.sol`)

---

### Summary
Once a user links a signer via a single `LinkSigner` transaction, that linked signer gains unlimited authority to withdraw the user's entire collateral balance across any product, without any further per-transaction user confirmation. This is the direct Nado analog to the "blanket ERC20 approve" anti-pattern described in the external report.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` (V1) handler calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 418–424
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allowLinkedSigner
);
``` [1](#0-0) 

`validateSignedTx` resolves the effective signer via `getLinkedSigner`, which returns the stored `linkedSigners[subaccount]` entry:

```solidity
// EndpointTx.sol lines 143–157
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return RiskHelper.isIsolatedSubaccount(subaccount)
        ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
        : linkedSigners[subaccount];
}
``` [2](#0-1) 

The linked signer is set once — either via the fast-path `processTransactionImpl` `LinkSigner` handler (lines 576–590) or via the slow-mode `processSlowModeTransactionImpl` `LinkSigner` handler (lines 232–239) — with no expiry, no per-operation scope, and no per-withdrawal user re-confirmation required. [3](#0-2) 

After the signature check passes, the withdrawal executes unconditionally:

```solidity
// EndpointTx.sol lines 430–436
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    address(0),
    nSubmissions
);
``` [4](#0-3) 

`Clearinghouse.withdrawCollateral` resolves `sendTo = address(0)` to the subaccount owner's wallet and transfers the ERC20 tokens out of the clearinghouse:

```solidity
// Clearinghouse.sol lines 404–408
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
}
handleWithdrawTransfer(token, sendTo, amount, idx);
``` [5](#0-4) 

The same `allowLinkedSigner = true` pattern also applies to `LiquidateSubaccount` (line 402), `MintNlp` (line 544), `BurnNlp` (line 564), and `TransferQuote` (line 604), giving the linked signer broad authority over the user's collateral with no per-operation user consent. [6](#0-5) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. **Force-drain all collateral** from the user's subaccount across every product, one `WithdrawCollateral` transaction per product, without the user signing any of them. Funds are sent to the user's own wallet, but the subaccount is left empty.
2. **Collapse open positions**: removing collateral causes the subaccount's health to drop below zero, making it immediately liquidatable by any third party.
3. **Charge repeated liquidation fees** from the user's account by signing `LiquidateSubaccount` transactions (also `allowLinkedSigner = true`), directly spending the user's collateral balance as protocol fees.

The corrupted state delta is: `spotEngine.balance[productId][sender]` is reduced to zero (or below health threshold) without any per-transaction user authorization. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The attack requires a compromised linked signer. Linked signers are commonly API keys or hot wallets used for automated trading — a realistic threat model. The user grants this authority once and has no on-chain mechanism to scope it to specific operations or amounts. Any key compromise, phishing, or malicious API integration immediately exposes the full collateral balance. [8](#0-7) 

---

### Recommendation

Require explicit per-withdrawal user confirmation by setting `allowLinkedSigner = false` for `WithdrawCollateral` (both V1 and V2 with `sendTo != address(0)`), so that only the subaccount owner's key can authorize collateral outflows. Alternatively, introduce a scoped delegation model where the linked signer's authority is bounded to specific transaction types (e.g., trading only) and explicitly excludes withdrawal and fee-charging operations. [1](#0-0) 

---

### Proof of Concept

1. User `Alice` owns subaccount `alice_subaccount` with 10,000 USDC collateral deposited.
2. Alice links `hotWallet` as her linked signer via a `LinkSigner` transaction (one-time user signature).
3. `hotWallet` is compromised by an attacker.
4. Attacker constructs a `WithdrawCollateral` transaction: `sender = alice_subaccount`, `productId = USDC_PRODUCT`, `amount = 10000e6`, `nonce = current_nonce`.
5. Attacker signs this transaction with `hotWallet`'s private key.
6. Attacker submits the transaction to the sequencer via the normal fast-path.
7. `validateSignedTx(..., true)` passes because `hotWallet == linkedSigners[alice_subaccount]`.
8. `clearinghouse.withdrawCollateral` executes, transferring 10,000 USDC to Alice's wallet address — but Alice's subaccount is now empty.
9. Any open perp or spot positions Alice held are now undercollateralized and subject to immediate liquidation by any third party.
10. Attacker repeats for every product Alice holds collateral in, with no further user interaction required. [9](#0-8) [10](#0-9)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
