### Title
Silent Failure of `DepositCollateral` Slow Mode Transactions Permanently Locks User Funds - (File: `core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.depositCollateralWithReferral`, user funds are transferred to the clearinghouse **before** the corresponding slow mode transaction is queued. If that slow mode transaction later fails during execution, the catch block in `_executeSlowModeTransaction` silently discards the failure with no refund logic — a comment in the code explicitly reads `// try return funds now removed`. The deposited collateral is permanently locked in the clearinghouse with no recovery path.

---

### Finding Description

`Endpoint.depositCollateralWithReferral` performs two sequential operations:

1. It calls `handleDepositTransfer`, which moves tokens from the user into the clearinghouse contract.
2. It enqueues a `SlowModeTx` with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (hardcoded to three days). [1](#0-0) 

`handleDepositTransfer` in `EndpointStorage` transfers funds from the user to the Endpoint and then immediately forwards them to the clearinghouse: [2](#0-1) 

When the slow mode transaction is later executed — either by the sequencer via `processTransaction(ExecuteSlowMode)` or by any user calling `executeSlowModeTransaction()` after the delay — `_executeSlowModeTransaction` wraps the call in a `try/catch`: [3](#0-2) 

If `processSlowModeTransaction` reverts (for any reason), the catch block checks only for out-of-gas conditions and then falls through. The comment `// try return funds now removed` at line 226 explicitly confirms that a prior refund mechanism was deliberately deleted. No alternative recovery path exists.

The slow mode processing path for `DepositCollateral` calls `clearinghouse.depositCollateral(txn)`: [4](#0-3) 

If `clearinghouse.depositCollateral` reverts — for example because the product was delisted in the 3-day window between deposit and processing — the funds already sitting in the clearinghouse are never credited to the user's subaccount and cannot be recovered.

---

### Impact Explanation

Any user whose `DepositCollateral` slow mode transaction fails during execution permanently loses their deposited collateral. The tokens are held by the clearinghouse contract but are not attributed to any subaccount balance. There is no owner-callable rescue function, no user-callable reclaim path, and no re-queuing mechanism. The asset delta is: user loses `amount` of the deposited token with zero on-chain recourse.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which the state of the protocol can change. A product delisting (`DelistProduct` slow mode tx) processed before the pending `DepositCollateral` tx is a concrete, realistic trigger. Additionally, any internal revert condition inside `clearinghouse.depositCollateral` (e.g., a product configuration change, a sanction applied to the subaccount address after deposit but before processing) would trigger the same outcome. The likelihood is **medium**: the window is long, the trigger conditions are realistic, and the comment `// try return funds now removed` confirms the refund path was intentionally removed rather than never considered.

---

### Recommendation

Restore a refund mechanism in the catch block of `_executeSlowModeTransaction` for `DepositCollateral` transactions. Specifically:

- Before deleting the slow mode tx, inspect its type.
- If it is a `DepositCollateral` tx and execution fails, decode the `productId` and `amount` and transfer the tokens back to `txn.sender` from the clearinghouse.
- Alternatively, store a `pendingRefunds` mapping keyed by sender and allow users to pull their refund explicitly, avoiding reentrancy concerns.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
2. `depositCollateralWithReferral` executes: `handleDepositTransfer` moves `amount` tokens from the user into the clearinghouse. A `SlowModeTx` is enqueued with `executableAt = block.timestamp + 3 days`.
3. Within the 3-day window, the owner submits a `DelistProduct` slow mode tx for `productId`, which is processed by the sequencer first (sequencer controls ordering).
4. After 3 days, the sequencer (or any user) calls `executeSlowModeTransaction()` for the `DepositCollateral` tx.
5. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(txn.sender, txn.tx)`.
6. Inside `processSlowModeTransactionImpl`, `clearinghouse.depositCollateral(txn)` reverts because `productId` is delisted.
7. The `catch` block in `_executeSlowModeTransaction` fires. Gas checks pass. Execution reaches `// try return funds now removed` and exits silently.
8. The `amount` tokens remain in the clearinghouse, unattributed. The user has no on-chain mechanism to recover them. [5](#0-4) [6](#0-5)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
