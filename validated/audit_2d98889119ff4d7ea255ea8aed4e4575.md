### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Enables 64-Byte Merkle Proof Forgery - (File: `contract/src/lib.rs`)

### Summary
The deprecated `verify_transaction_inclusion` function is still reachable by any unprivileged NEAR caller. It omits the coinbase-proof guard that `verify_transaction_inclusion_v2` was specifically introduced to enforce, leaving the 64-byte internal-node substitution attack fully open on the older entry point.

### Finding Description
`verify_transaction_inclusion` was deprecated at version 0.5.0 with an explicit note that `verify_transaction_inclusion_v2` must be used instead to close the 64-byte transaction Merkle-proof forgery vulnerability. Despite the deprecation, the function remains a live, publicly callable contract method protected only by the `#[pause]` gate (which is inactive by default). [1](#0-0) 

The v2 entry point closes the attack by first verifying a coinbase Merkle proof at index 0 against the block's stored `merkle_root`: [2](#0-1) 

The v1 entry point performs no such check. Its only guard is that the proof array is non-empty: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure positional hash-chain computation that treats any 32-byte value as a leaf: [4](#0-3) 

Because every internal Merkle-tree node is itself the double-SHA256 of two 32-byte children, an attacker can supply any internal node as `tx_id`, supply the correct sibling path to the root, and the function returns `true` — falsely asserting that the internal node is an included transaction.

### Impact Explanation
Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` to gate fund releases or cross-chain state transitions can be deceived into accepting a fabricated Bitcoin transaction. The attacker does not need to forge a block header or break PoW; they only need a real mainchain block (already stored by the relayer) and knowledge of its Merkle tree structure. The corrupted proof result is the boolean `true` returned for a `tx_id` that is not a real transaction.

### Likelihood Explanation
The 64-byte transaction Merkle-proof forgery is a publicly documented, practically executable attack (referenced in the contract's own deprecation notice and the linked BitMEX research). Any NEAR account can call `verify_transaction_inclusion` without any role or bond. The only prerequisite is a block already accepted into the mainchain, which the relayer continuously supplies. Likelihood is **high** given the zero-privilege entry path and the well-known attack technique.

### Recommendation
Remove `verify_transaction_inclusion` from the contract entirely, or gate it with an `env::panic_str` unconditional abort so it can no longer be invoked. All callers must migrate to `verify_transaction_inclusion_v2`. If backward compatibility is required during a transition window, add the identical coinbase-proof check to the v1 path before the Merkle root comparison.

### Proof of Concept

1. The relayer submits a real Bitcoin block `B` containing transactions `[T0, T1, T2, T3]`. The contract stores `B` in `headers_pool` and `mainchain_header_to_height`.
2. The Merkle tree of `B` has internal node `N = hash(T0 || T1)` at height 1, index 0.
3. The attacker computes the sibling path from `N` to the Merkle root: `proof = [hash(T2 || T3)]`.
4. The attacker calls `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash = B.hash`
   - `tx_index = 0`
   - `merkle_proof = [hash(T2 || T3)]`
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, 0, proof)` computes `hash(N || hash(T2||T3))` = actual Merkle root of `B`.
6. The comparison `== header.block_header.merkle_root` succeeds; the function returns `true`.
7. Any contract that trusted this result now believes the 32-byte internal node `N` is a confirmed Bitcoin transaction. [3](#0-2) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
