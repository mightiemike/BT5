### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unauthorized caller to bypass the deposit gate â€” (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently drops the `sender` parameter and gates only on `owner`. Because `addLiquidity` lets `msg.sender` (the token provider) differ from `owner` (the LP-share recipient), any address that is not on the allowlist can deposit tokens into a restricted pool by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the `beforeAddLiquidity` hook:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData);
```

`sender` = `msg.sender`, the actual caller who provides tokens via the swap callback.  
`owner` = the caller-supplied address that receives the LP shares. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed `address`). It only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

The allowlist mappings are keyed `allowedDepositor[pool][depositor]`, where the pool admin intends `depositor` to be the party interacting with the pool: [4](#0-3) 

Because `removeLiquidity` enforces `msg.sender == owner`, the sender/owner split only exists for `addLiquidity`, making this the sole exploitable entry point. [5](#0-4) 

---

### Impact Explanation

An unauthorized address (`sender`) calls `addLiquidity(allowlisted_address, ...)`. The extension checks `allowedDepositor[pool][allowlisted_address]` â†’ passes. The unauthorized sender provides tokens via callback; the allowlisted address receives LP shares it never requested. The pool admin's deposit gate â€” the sole mechanism restricting who may interact with a private/compliant pool â€” is fully bypassed by any unprivileged caller. This is a direct admin-boundary break: a pool admin-configured access control is circumvented through a valid, unprivileged call path with no special permissions required.

---

### Likelihood Explanation

The attack requires only knowing one allowlisted address (publicly readable from `allowedDepositor`) and being willing to provide tokens. No privileged role, flash loan, or oracle manipulation is needed. Any external actor can execute this against any pool that has deployed `DepositAllowlistExtension`.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the token provider and the LP-share recipient, check both `sender` and `owner`.

---

### Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension`. Pool admin allowlists only `alice`.
2. `allowedDepositor[P][alice] == true`; `allowedDepositor[P][bob] == false`.
3. Bob calls `P.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. Extension receives `(bob, alice, ...)` but checks only `allowedDepositor[P][alice]` â†’ `true` â†’ no revert.
6. Bob's callback provides tokens; Alice receives LP shares.
7. Bob has deposited into a pool he is explicitly barred from, bypassing the allowlist entirely.
