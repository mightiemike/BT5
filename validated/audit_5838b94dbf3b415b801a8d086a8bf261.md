### Title
Unvalidated Peer-Supplied Transaction Hash Accepted During P2P Block Sync - (`File: crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

During P2P block synchronization, the `TransactionStreamFactory` accepts the `transaction_hash` field directly from an untrusted peer's `FullTransaction` message without recomputing it from the transaction body. A malicious peer can supply an arbitrary hash for any transaction, causing the syncing node to persist a wrong hash-to-transaction binding in storage. This corrupts RPC responses, block commitment verification, and any downstream logic that relies on the stored transaction hash.

### Finding Description

In `crates/apollo_p2p_sync/src/client/transaction.rs`, `parse_data_for_block` receives `FullTransaction { transaction, transaction_output, transaction_hash }` from a P2P peer and pushes the peer-supplied `transaction_hash` directly into `block_body.transaction_hashes` without recomputing it from `transaction`:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
```

The code even carries an explicit TODO acknowledging the missing validation. [1](#0-0) 

The correct behavior — computing the hash from the transaction body — is enforced in every other ingestion path. In the gateway path, `convert_rpc_tx_to_internal` always recomputes the hash:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [2](#0-1) 

In the consensus proposal validation path, `convert_consensus_tx_to_internal_consensus_tx` also calls `convert_rpc_tx_to_internal`, which recomputes the hash from content. [3](#0-2) 

The P2P sync path is the only path that skips this recomputation and trusts the peer-supplied value. The `SyncBlock` type used for internally-produced blocks does carry pre-computed hashes, but those originate from the local sequencer and are trusted. [4](#0-3) 

The analog to the external report is exact: `liquidate` (gateway/consensus path) enforces the invariant (hash computed from content), while `liquidateFrom` (P2P sync path) is the publicly reachable entry point that skips it.

### Impact Explanation

The peer-supplied `transaction_hash` is written to storage via `block_body.transaction_hashes`. Once stored:

1. **Wrong RPC responses**: `get_transaction_by_hash` looks up transactions by the stored hash. A wrong hash means the transaction is unreachable under its real hash and reachable under the attacker-chosen hash. [5](#0-4) 

2. **Wrong block commitment**: The transaction commitment in the block hash is computed over the stored transaction hashes. A wrong hash produces a wrong `transaction_commitment`, causing the syncing node to compute a different block hash than the canonical chain. [6](#0-5) 

3. **Wrong state**: Any component that reads `get_block_transaction_hashes` from storage receives the attacker-controlled value, propagating the wrong hash into receipts, events, and re-execution.

This matches the allowed impact: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result** and **RPC execution returns an authoritative-looking wrong value**.

### Likelihood Explanation

- Any P2P peer the node connects to can trigger this — no special privilege is required.
- The node actively queries peers for transactions during sync (`parse_data_for_block` is called for every synced block).
- The TODO comment confirms the developers are aware the input is untrusted and the check is missing.
- The attack requires only sending a well-formed `FullTransaction` protobuf with a crafted `transaction_hash` field.

### Recommendation

After deserializing `FullTransaction`, recompute the transaction hash from the `transaction` body using `validate_transaction_hash` (already present in `crates/starknet_api/src/transaction_hash.rs`) and reject the peer with `BadPeerError` if the recomputed hash does not match the peer-supplied one:

```rust
// After: let Some(FullTransaction { transaction, transaction_output, transaction_hash }) = ...
let computed = get_transaction_hash(&transaction, chain_id, &TransactionOptions::default())?;
if computed != transaction_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }));
}
block_body.transaction_hashes.push(transaction_hash);
``` [7](#0-6) 

### Proof of Concept

1. A malicious P2P peer responds to a transaction sync query for block N.
2. For each `FullTransaction` in the response, the peer sets `transaction_hash` to an arbitrary value (e.g., `0xdeadbeef`) while keeping the `transaction` body valid.
3. The syncing node's `parse_data_for_block` accepts the response, pushes the crafted hash into `block_body.transaction_hashes`, and writes it to storage. [8](#0-7) 
4. A subsequent RPC call `starknet_getTransactionByHash(0xdeadbeef)` returns the transaction body, while the real hash returns `TRANSACTION_HASH_NOT_FOUND`.
5. The block hash computed by the syncing node diverges from the canonical chain because `transaction_commitment` is computed over the wrong hashes, causing the node to disagree with honest peers on block validity.

/// Validates the hash of a starknet transaction.
/// For transactions on testnet or those with a low block_number, we validate the
/// transaction hash against all potential historical hash computations. For recent
/// transactions on mainnet, the hash is validated by calculating the precise hash
/// based on the transaction version.
pub fn validate_transaction_hash(
    transaction: &Transaction,
    block_number: &BlockNumber,
    chain_id: &ChainId,
    expected_hash: TransactionHash,
    transaction_options: &TransactionOptions,
) -> Result<bool, StarknetApiError> {
    let mut possible_hashes = get_deprecated_transaction_hashes(
        chain_id,
        block_number,
        transaction,
        transaction_options,
    )?;
    possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
    Ok(possible_hashes.contains(&expected_hash))
}
