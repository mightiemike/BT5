### Title
Silent `unwrap_or_default()` on `starknet_version` parse substitutes `StarknetVersion::LATEST` for any unrecognized version string, causing wrong `VersionedConstants` selection and wrong block hash computation - (File: crates/native_blockifier/src/py_state_diff.rs)

### Summary

In `crates/native_blockifier/src/py_state_diff.rs`, the `TryFrom<PyBlockInfo> for BlockInfo` conversion silently substitutes `StarknetVersion::LATEST` whenever the version string fails to parse, instead of propagating the error. Because `StarknetVersion::default()` is defined as `Self::LATEST`, any unrecognized version string — including unknown intermediate versions, gap versions, or malformed strings — silently becomes the latest known version. This causes the blockifier to select the wrong `VersionedConstants` (wrong gas costs, step limits, syscall costs) and the wrong block hash algorithm, producing incorrect execution results and incorrect block hashes for any block whose version string does not exactly match a known `StarknetVersion` variant.

### Finding Description

`PyBlockInfo` is the Python-to-Rust bridge used by `native_blockifier` (the Python extension module consumed by the Starknet OS Python code for block execution and proving). Its `starknet_version` field is a raw `String` supplied by the Python caller.

At line 255 of `crates/native_blockifier/src/py_state_diff.rs`:

```rust
starknet_version: block_info.starknet_version.try_into().unwrap_or_default(),
```

`StarknetVersion::default()` is defined as `Self::LATEST` in `crates/starknet_api/src/block.rs`:

```rust
impl Default for StarknetVersion {
    fn default() -> Self {
        Self::LATEST   // currently V0_14_4
    }
}
```

`StarknetVersion::try_from(String)` returns `Err(StarknetApiError::InvalidStarknetVersion(...))` for any version string whose dot-separated segments do not match a known variant. The `unwrap_or_default()` swallows this error and substitutes `LATEST` silently, with no log, no metric, and no caller-visible signal.

This contrasts with the deliberate `starknet_version_or_latest` helper in `crates/starknet_transaction_prover/src/running/virtual_block_executor.rs`, which only falls back to `LATEST` for versions **strictly newer** than `LATEST`, and propagates errors for all other failures (unknown older versions, malformed strings, gap versions like `"0.13.7"`). The `native_blockifier` path has no such guard.

The resulting `BlockInfo` with the wrong `starknet_version` is then used in two critical downstream paths:

1. **`VersionedConstants` selection** — `VersionedConstants::get(&block_info.starknet_version)` selects execution parameters (gas costs per syscall, step limits, `invoke_tx_max_n_steps`, `validate_max_n_steps`, `min_sierra_version_for_sierra_gas`, etc.). Using LATEST constants for an older block changes execution semantics, gas accounting, and can change whether transactions succeed or revert.

2. **Block hash computation** — `calculate_block_hash` calls `BlockHashVersion::try_from(starknet_version)` to select the hash algorithm. `BlockHashVersion::V0_13_2` uses `STARKNET_BLOCK_HASH0` as the prefix and omits L2 gas prices; `BlockHashVersion::V0_13_4` uses `STARKNET_BLOCK_HASH1` and includes L2 gas prices via a nested Poseidon hash. Substituting LATEST for an older version crosses this gate, producing a completely different block hash.

### Impact Explanation

When the Python OS passes a `PyBlockInfo` with a version string that does not match any known `StarknetVersion` variant (e.g., a gap version `"0.13.7"`, a future version not yet in the binary, or a malformed string), the blockifier silently executes the block under `LATEST` semantics. This produces:

- Wrong execution results (different gas costs, different step limits, different syscall costs) — matching **"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**
- Wrong block hash (different hash prefix constant, different gas price hash structure) — matching **"Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution"** and wrong state commitment.

### Likelihood Explanation

The trigger is any block whose `starknet_version` string does not exactly match a known `StarknetVersion` variant. This occurs in practice during:
- Network upgrades where a new protocol version is deployed before the binary is updated (the binary sees an unknown newer version — but unlike `starknet_version_or_latest`, `unwrap_or_default` also silently accepts unknown **older** gap versions)
- Any intermediate or patch version not enumerated in the `starknet_version_enum!` macro
- Any malformed string passed from the Python layer

The Python OS code controls the version string and passes it without validation. No privilege is required beyond submitting a block with a non-standard version string.

### Recommendation

Replace the silent fallback with an explicit error:

```rust
// Before (silent fallback to LATEST):
starknet_version: block_info.starknet_version.try_into().unwrap_or_default(),

// After (propagate the error):
starknet_version: block_info.starknet_version.try_into().map_err(|e| {
    NativeBlockifierInputError::InvalidNativeBlockifierInputError(
        InvalidNativeBlockifierInputError::InvalidStarknetVersion(
            block_info.starknet_version.clone(),
        )
    )
})?,
```

If forward-compatibility with newer-than-LATEST versions is required, apply the same guarded fallback used in `starknet_version_or_latest`: only fall back to `LATEST` when the parsed segments are strictly greater than `LATEST`'s segments, and return an error for all other failures.

### Proof of Concept

1. Construct a `PyBlockInfo` with `starknet_version = "0.13.7"` (a gap version between `V0_13_6` and `V0_14_0`, not in the enum).
2. Call `BlockInfo::try_from(py_block_info)`.
3. Observe that the conversion succeeds (no error) and `block_info.starknet_version == StarknetVersion::V0_14_4` (LATEST).
4. Pass this `BlockInfo` to `VersionedConstants::get(&block_info.starknet_version)` — it returns the V0_14_4 constants instead of failing or returning V0_13_6 constants.
5. Pass this `BlockInfo` to `calculate_block_hash` — it uses `BlockHashVersion::V0_13_4` (LATEST maps to V0_13_4 in the block hash version gate), producing a hash with `STARKNET_BLOCK_HASH1` prefix and L2 gas prices included, which is wrong for a block that was originally hashed under V0_13_2 semantics.

The divergence is exact and deterministic: any version string not in the `starknet_version_enum!` macro produces `StarknetVersion::LATEST` silently, and all downstream version-gated logic operates on the wrong version. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
}

/// Returns the [StarknetVersion] matching the given version string, falling back to
/// [StarknetVersion::LATEST] for versions newer than `LATEST`
pub(crate) fn starknet_version_or_latest(
    version_string: &str,
) -> Result<StarknetVersion, StarknetApiError> {
    match StarknetVersion::try_from(version_string) {
        // `InvalidStarknetVersion` carries the parsed version segments, whose lexicographic order
        // matches version order.
        Err(StarknetApiError::InvalidStarknetVersion(segments))
            if segments > Vec::<u8>::from(StarknetVersion::LATEST) =>
        {
            Ok(StarknetVersion::LATEST)
        }
        result => result,
    }
}
