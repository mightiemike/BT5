Audit Report

## Title
Pool Admin Can Instantly Raise Swap Fees Without Timelock to Front-Run Traders — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`MetricOmmPoolFactory.setPoolAdminFees` applies new spread and notional fee rates atomically with no queuing or delay. Because the pool-admin role is permissionless (any caller of `createPool` can designate themselves), a malicious pool admin can front-run a pending swap by raising fees to the factory-enforced maximum in the same block, causing the trader to pay up to 20 % spread + 1 % notional instead of the rate they observed. The excess flows to the pool admin via `collectPoolFees`.

## Finding Description
`setPoolAdminFees` (lines 408–435) validates the new rates against `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`, settles accrued fees at the old rate via `collectFees`, then immediately writes the new rates to `poolFeeConfig[pool]` and pushes them to the pool via `setPoolFees`. There is no pending slot, no `executeAfter` timestamp, and no event that gives traders advance notice before the new rate takes effect. [1](#0-0) 

By contrast, price-provider changes go through a two-step propose/execute pattern gated by `priceProviderTimelock[pool]`, demonstrating the protocol is aware of the timelock pattern and deliberately applied it to one sensitive setter but not to fee changes. [2](#0-1) 

`MetricOmmPool.setPoolFees` applies the new rates immediately with no additional guard: [3](#0-2) 

The hard caps the pool admin can reach in a single call are: [4](#0-3) 

Both caps are the default values of `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` at construction: [5](#0-4) 

The pool-admin role is assigned from `params.admin` during `createPool`, making it permissionless — any caller can create a pool and designate themselves as admin: [6](#0-5) 

## Impact Explanation
A trader who queries `spreadFeeE6` / `notionalFeeE8` before submitting a swap sees the benign rate. The pool admin can atomically replace those rates with the maximum values (`200_000` spread + `1_000_000` notional) in the block before the swap lands. The swap executes at up to **20 % spread + 1 % notional** instead of the observed rate. The excess fee is credited to `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and the spread surplus, both of which flow to `poolAdminFeeDestination` on the next `collectPoolFees` call. This is a direct, quantifiable loss of trader principal — a Critical/High direct loss of user funds above Sherlock thresholds.

## Likelihood Explanation
- The pool-admin role is permissionless; any address that calls `createPool` can be its own admin.
- The attack requires a single `setPoolAdminFees` call with higher gas than the target swap — standard mempool front-running, available on all EVM chains.
- The admin can restore the original fee immediately after the target swap settles, making the manipulation hard to detect retrospectively.
- No special privilege beyond pool-admin status is required.

## Recommendation
Apply the same two-step timelock pattern already used for price-provider changes to fee updates:

1. Add `pendingAdminFeeConfig` and `pendingAdminFeeExecuteAfter` mappings per pool.
2. `setPoolAdminFees` writes to the pending slot and records `block.timestamp + feeTimelock`.
3. A separate `executePoolAdminFees` function (callable only after the delay) applies the change and calls `setPoolFees` on the pool.
4. Emit an event on proposal so LPs and traders can react before the new rate takes effect.

This mirrors the existing `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` pattern.

## Proof of Concept
1. Pool is deployed with `adminSpreadFeeE6 = 500` (0.05 %) and `adminNotionalFeeE8 = 0`.
2. Alice signs a large swap (e.g., 1 000 000 USDC → TOKEN1) and broadcasts it.
3. Pool admin sees Alice's transaction in the mempool and submits `setPoolAdminFees(pool, 200_000, 1_000_000)` with higher gas.
4. Pool admin's transaction lands first: `collectFees` settles tiny accrued fees at the old rate; `poolFeeConfig[pool].adminSpreadFeeE6` is set to `200_000`; `pool.setPoolFees(protocolSpread + 200_000, protocolNotional + 1_000_000)` is called immediately.
5. Alice's swap executes: she pays ≈ 20 % spread + 1 % notional instead of 0.05 %.
6. Pool admin calls `collectPoolFees` — the inflated fees flow to `poolAdminFeeDestination`.
7. Pool admin calls `setPoolAdminFees(pool, 500, 0)` to restore the normal rate.

Alice loses up to ~21 % of her swap value with no on-chain protection and no advance warning. A Foundry integration test can reproduce this by deploying a pool, broadcasting a swap, inserting `setPoolAdminFees` before it in the same block, and asserting the fee delta against the trader's received amount.

    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
      );

    c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
    c.adminNotionalFeeE8 = newAdminNotionalFeeE8;
    poolFeeConfig[pool] = c;

    IMetricOmmPoolFactoryActions(pool)
      .setPoolFees(c.protocolSpreadFeeE6 + c.adminSpreadFeeE6, c.protocolNotionalFeeE8 + c.adminNotionalFeeE8);
    emit PoolAdminSpreadFeeUpdated(pool, newAdminSpreadFeeE6);
    emit PoolAdminNotionalFeeUpdated(pool, newAdminNotionalFeeE8);
  }
