Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` Checks Router Address Instead of Actual User, Enabling Per-User Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension checks the router's allowlist status rather than the actual user's. A pool admin who allowlists the router inadvertently grants swap access to every user of the router, collapsing the per-user gate entirely.

## Finding Description
In `MetricOmmPool.swap()`, `msg.sender` is passed as `sender` to `_beforeSwap()`:
```solidity
_beforeSwap(msg.sender, recipient, ...);  // MetricOmmPool.sol:231
```
`ExtensionCalling._beforeSwap()` forwards this `sender` verbatim to the extension via `abi.encodeCall`. `SwapAllowlistExtension.beforeSwap()` then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) revert NotAllowedToSwap();
```
Here `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle()` is used, the router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly, making the router the `msg.sender` of `pool.swap()`. Therefore `sender = router` in `beforeSwap`, and the check becomes `allowedSwapper[pool][router]`. Any user calling the router passes this check if the router is allowlisted, regardless of their individual allowlist status. By contrast, `DepositAllowlistExtension.beforeAddLiquidity()` ignores `sender` entirely and checks `owner` — the economic actor — correctly gating the beneficiary rather than the intermediary.

## Impact Explanation
**Allowlist bypass (High):** A pool admin configuring a curated pool will naturally allowlist the router as the standard user-facing entry point. Doing so collapses the per-user gate: every unprivileged address can call `router.exactInputSingle()` and have the swap accepted because `allowedSwapper[pool][router] = true`. The curated-pool invariant — only approved counterparties may trade — is silently violated, exposing LP assets to unauthorized swap flow. This constitutes broken core pool functionality and an admin-boundary break reachable by any unprivileged caller.

**Broken allowlist for legitimate users (Medium):** If the admin allowlists individual users but not the router, those allowlisted users cannot swap through the router at all — the check sees the router address and reverts — making the standard periphery path unusable for the very users the admin intended to serve.

## Likelihood Explanation
The router is the canonical, documented user-facing swap entry point deployed alongside the pool. A pool admin configuring a curated pool will naturally allowlist the router as a trusted intermediary without realizing this collapses the per-user gate. The bypass requires no special privilege: any unprivileged address can call the public router. The condition (router allowlisted on a restricted pool) is the expected operational configuration, not an edge case.

## Recommendation
The swap allowlist must gate the actual initiating user, not the intermediary. The most robust fix mirrors `DepositAllowlistExtension`: pass the real user through `extensionData`. The router should encode `msg.sender` into `extensionData`; the extension decodes and checks it. Alternatively, if the allowlist is only meaningful for direct `pool.swap()` calls, the extension documentation must explicitly state this and the router must never be allowlisted on curated pools — but this is fragile and operationally error-prone. The `DepositAllowlistExtension` pattern of checking the economic actor (`owner`) rather than the intermediary (`sender`) should be the model.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension; allowAllSwappers[pool] = false.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — intending to allow the standard periphery path.
3. Charlie (not individually allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: charlie, ...})
4. Router executes: IMetricOmmPoolActions(pool).swap(charlie, zeroForOne, amount, ...)
     → msg.sender of pool.swap() = router
5. Pool calls _beforeSwap(sender=router, ...)
     → extension checks allowedSwapper[pool][router] = true → passes
6. Charlie's swap executes on the restricted pool.
   Any user of the router now bypasses the per-user allowlist.
```
