Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the originating EOA, allowing any unprivileged caller to bypass per-pool swap allowlists via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router contract — not the end-user EOA. When a pool admin allowlists the router address so that legitimate users can access the pool through `MetricOmmSimpleRouter`, every unprivileged caller gains identical access because the extension never inspects the original initiator. There is no configuration that simultaneously permits allowlisted users to use the router and blocks non-allowlisted users.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router — the original EOA is never visible: [3](#0-2) 

The router stores the real payer only in transient storage for payment settlement and never forwards it to the pool's `swap` call or to any extension hook: [4](#0-3) 

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`, allowlists `userA`, and also allowlists `address(router)` so `userA` can use the standard interface.
2. Attacker (not in allowlist) calls `router.exactInputSingle(...)`.
3. Router calls `pool.swap(...)` — pool's `msg.sender` is the router.
4. `_beforeSwap` passes `sender = address(router)` to the extension.
5. Extension checks `allowedSwapper[pool][router] == true` → passes. Attacker's swap executes.

The wrong value approved is `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][attacker_EOA]`.

## Impact Explanation

Any pool configured with `SwapAllowlistExtension` for access control (KYC-gated, institutional, closed-beta) loses that restriction entirely once the admin allowlists the router. Any unprivileged EOA can execute swaps against the pool via `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`, directly extracting token0/token1 from LP principal. This constitutes broken core swap functionality and unauthorized extraction of user principal — a direct loss-of-funds impact meeting Critical/High Sherlock thresholds.

## Likelihood Explanation

The precondition — admin allowlisting the router — is the natural and expected operational step for any pool intended to be accessible through the protocol's standard periphery. The integration test `test_allowedSwapSucceeds` confirms the gap: it allowlists `callers[0]` (a direct pool-caller wrapper) rather than the router, working only because the test bypasses the router entirely: [5](#0-4) 

Any real deployment routing through `MetricOmmSimpleRouter` hits this path. No special attacker capability is required beyond calling a public router function.

## Recommendation

The `sender` argument reaching `beforeSwap` must represent the economically relevant actor. Two viable approaches:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires a trust assumption that only the allowlisted router can supply this field; the extension must verify `msg.sender` (the pool) has the router registered as a trusted forwarder.

2. **Pool-side**: Add a separate `originator` field to the `swap` signature that the pool passes through to extensions, allowing the router to supply the real EOA. Extensions gate on `originator` instead of `sender`.

Either fix must prevent the router from being used to spoof an arbitrary allowlisted address.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only userA is allowlisted.
// Admin also allowlists the router so userA can use the standard interface.
swapExtension.setAllowedToSwap(address(pool), userA, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // necessary for userA to use router

// Attacker (not allowlisted) calls the router directly.
vm.prank(attacker); // attacker != userA, not in allowlist
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: token0,
        tokenOut: token1,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Swap succeeds: extension checked allowedSwapper[pool][router] == true,
// never inspecting `attacker`. Allowlist is bypassed.
```

The pool's `_beforeSwap` receives `sender = address(router)` from `MetricOmmPool.swap` line 231; the extension approves it because `allowedSwapper[pool][router] == true` at `SwapAllowlistExtension.sol` line 37, while `attacker` is never checked.
