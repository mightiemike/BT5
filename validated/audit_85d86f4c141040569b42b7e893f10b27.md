### Title
ECDSA Signature Malleability in `op_secp256k1_verify` and `op_secp256r1_verify` — No Low-S Enforcement - (File: src/secp_ops.rs)

### Summary

`op_secp256k1_verify` and `op_secp256r1_verify` in `src/secp_ops.rs` parse and verify ECDSA signatures without enforcing low-S normalization. For any valid signature `(r, s)`, the signature `(r, n - s)` — where `n` is the curve order — is also mathematically valid and will pass verification. This is the direct analog of the Solidity `ecrecover()` s-value malleability described in the external report.

### Finding Description

Both operators are implemented in `src/secp_ops.rs`. The secp256k1 path:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

The secp256r1 path uses the `hazmat::PrehashVerifier` interface — explicitly a low-level bypass of safety checks:

```rust
use p256::ecdsa::signature::hazmat::PrehashVerifier;
...
let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

Neither function contains any call to `normalize_s()`, any low-S range check, or any rejection of high-S signatures. A grep across all of `src/` for `normalize`, `low_s`, `high_s`, `malleable`, or `non_canonical` returns no matches in production code. The `hazmat` import for p256 is particularly significant: the `signature::hazmat` module is explicitly documented as bypassing safety checks that would otherwise be applied by the higher-level `Verifier` trait, including canonicality enforcement.

`K1Signature::from_slice` and `P1Signature::from_slice` accept any 64-byte compact `(r || s)` encoding where both components are in `[1, n-1]`. They do not restrict `s` to the lower half `[1, (n-1)/2]`. The `verify_prehash` call then verifies the signature as presented, without normalizing `s` first. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

In Chia, CLVM puzzles that use `secp256k1_verify` or `secp256r1_verify` to gate coin spends accept both `(r, s)` and `(r, n - s)` as valid authorizations for the same message and public key. An attacker who observes a valid spend bundle can compute the malleable counterpart signature and submit a second spend bundle that:

1. Passes the same `secp256k1_verify` / `secp256r1_verify` check inside the puzzle.
2. Has a **different spend bundle ID** (the solution bytes differ because the signature bytes differ).
3. Spends the same coin.

This breaks any off-chain system, mempool deduplication logic, or wallet that tracks spend bundles by ID. It also breaks any puzzle that uses the signature bytes as a nonce or unique identifier. The corrupted result is: two distinct byte strings, each a valid solution to the same puzzle, producing the same authorized coin spend — a direct consensus-layer signature non-uniqueness.

### Likelihood Explanation

The attack requires only that an attacker observe a valid spend bundle containing a `secp256k1_verify` or `secp256r1_verify` call. The malleable signature is trivially computed: given `(r, s)`, compute `s' = n - s mod n` and replace the last 32 bytes of the 64-byte signature atom. No private key material is needed. The attacker-controlled entry path is the signature atom passed as the third argument to either operator, which is fully attacker-controlled CLVM bytes.

### Recommendation

Add explicit low-S enforcement before calling `verify_prehash`. For secp256k1, use `k256::ecdsa::signature::hazmat::PrehashVerifier` only after calling `sig.normalize_s()` and rejecting if the signature was already high-S, or use `k256::ecdsa::VerifyingKey`'s higher-level `verify` interface with the `normalize-s` feature disabled and a manual check. For secp256r1, replace the `hazmat::PrehashVerifier` call with the safe `p256::ecdsa::VerifyingKey::verify_prehash` from the non-hazmat trait, and add an explicit check that `sig.s()` is in the lower half of the curve order. Reject any signature where `s > n/2`.

### Proof of Concept

Given a valid call that succeeds:
```
secp256k1_verify <pubkey> <msg_hash> <r || s>  => 0
```

Compute `s' = secp256k1_order - s`. Then:
```
secp256k1_verify <pubkey> <msg_hash> <r || s'>  => 0  (also succeeds)
```

Both 64-byte blobs are distinct CLVM atoms. Both pass `K1Signature::from_slice` (both are valid compact encodings). Both pass `verify_prehash` (ECDSA verification is symmetric in s). The two spend bundles containing these two solutions have different IDs but spend the same coin with the same authorization. [4](#0-3) [5](#0-4)

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
