### Title
Missing `max_l2_gas_amount` Admission Check for Declare Transactions Allows Gateway Bypass - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The `StatelessTransactionValidator::validate_resource_bounds` function enforces the `max_l2_gas_amount` cap for Invoke and DeployAccount transactions but explicitly skips this check for Declare transactions. An unprivileged user can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount` value that exceeds the configured `max_l2_gas_amount` limit, bypassing the gateway's admission control. No downstream component re-enforces this upper bound, so the transaction enters the mempool and block pipeline with an unchecked gas amount declaration.

### Finding Description

In `validate_resource_bounds`, the check for `l2_gas.max_amount` against `config.max_l2_gas_amount` is guarded by a type-dispatch that silently skips the entire check for `RpcTransaction::Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // nothing
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
```

The `min_gas_price` check immediately above this block applies to all transaction types including Declare, so the asymmetry is not intentional symmetry — it is a missing check. The TODO comment confirms the gap is known but unresolved.

The default `max_l2_gas_amount` is `1_210_000_000` (1.21 billion gas units). A Declare transaction can declare `l2_gas.max_amount = u64::MAX` and pass stateless validation. The stateful validator (`StatefulTransactionValidator::validate_resource_bounds`) only checks the L2 gas *price*, not the *amount*. The blockifier's pre-validation (`check_resource_bounds`) only checks that the declared amount is not *too low* (below the minimum needed), never that it is *too high*. The bouncer enforces actual block capacity but does not re-check the declared bound against the gateway limit.

The `validate_resource_bounds` flag is a shared pointer target that simultaneously controls the stateless validator, stateful validator, and mempool. When it is `true` (production default), the stateless validator runs but skips the amount check for Declare; when it is `false` (bootstrap mode), all resource-bound checks are disabled. There is no code path that enforces `max_l2_gas_amount` for Declare transactions in either mode.

### Impact Explanation

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

A Declare transaction with `l2_gas.max_amount` far exceeding `max_l2_gas_amount` is accepted by the gateway and forwarded to the mempool. The `max_possible_fee` computed from this inflated amount (used by `verify_can_pay_committed_bounds` in the blockifier) saturates to `Fee::MAX` for any non-zero price, causing the transaction to fail blockifier pre-validation with `InsufficientFeeTokenBalance` — but only after it has already consumed mempool capacity and been included in a block proposal. An attacker with sufficient balance to cover the inflated `max_possible_fee` can execute Declare transactions that the gateway was explicitly configured to reject, bypassing the operator-set admission limit entirely.

### Likelihood Explanation

The trigger is unprivileged: any user can submit a Declare transaction via the public HTTP gateway endpoint. The bypass requires only setting `l2_gas.max_amount` above `max_l2_gas_amount` in the transaction's resource bounds. The code path is reachable in production (the `validate_resource_bounds` flag defaults to `true`). The TODO comment confirms the gap is known and unmitigated.

### Recommendation

Remove the type-dispatch exemption and apply the `max_l2_gas_amount` check uniformly to all transaction types, including Declare:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas amount ceiling, introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

1. Construct a Declare V3 transaction with `resource_bounds.l2_gas.max_amount = max_l2_gas_amount + 1` (e.g., `1_210_000_001`) and `resource_bounds.l2_gas.max_price_per_unit >= min_gas_price` (e.g., `8_000_000_000`).
2. Submit it to the gateway's `add_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate_resource_bounds` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised because the Declare branch is empty.
4. The transaction is forwarded to the mempool and accepted.
5. Contrast with an Invoke transaction carrying the same `l2_gas.max_amount`: it is rejected with `StatelessTransactionValidatorError::MaxGasAmountTooHigh`.

The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` (lines 173–201) explicitly documents and asserts this bypass as passing behavior, confirming the gap is present and untested for rejection.
