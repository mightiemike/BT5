### Title
`TierUndelegate` Permanently Blocked by Insufficient Bonus Pool — (`x/tieredrewards/keeper/msg_server.go`, `claim_rewards.go`, `bonus_rewards.go`)

### Summary

`MsgTierUndelegate` unconditionally calls `claimRewards`, which calls `processEventsAndClaimBonus`, which hard-fails with `ErrInsufficientBonusPool` if the `RewardsPoolName` module account balance is less than the computed bonus. Because the error propagates back through the entire call stack, the undelegation is never executed. There is no alternative user-callable path to undelegate without first successfully claiming bonus rewards. The owner's principal is permanently locked inside the position's delegator sub-account.

---

### Finding Description

**Step 1 — Entrypoint**

`TierUndelegate` in `msg_server.go` calls `claimRewards` unconditionally before calling `undelegate`: [1](#0-0) 

**Step 2 — `claimRewards` calls `processEventsAndClaimBonus`**

`claimRewards` in `claim_rewards.go` calls `processEventsAndClaimBonus` and propagates any error: [2](#0-1) 

**Step 3 — Pool balance check hard-fails**

`processEventsAndClaimBonus` calls `sufficientBonusPoolBalance` after computing the total bonus. If the pool is short, it returns `ErrInsufficientBonusPool` and the function returns immediately — no bonus is paid, no checkpoint is advanced, and no error recovery is attempted: [3](#0-2) 

`sufficientBonusPoolBalance` in `bonus_rewards.go`: [4](#0-3) 

**Step 4 — Error propagates; `undelegate` is never reached**

Because `claimRewards` returns an error, `TierUndelegate` returns at line 167–169 before ever calling `ms.undelegate` or `pos.ClearBonusCheckpoints`: [5](#0-4) 

**Step 5 — No alternative user-callable exit path**

`ExitTierWithDelegation` has the identical coupling — it also calls `claimRewards` before transferring the delegation: [6](#0-5) 

`ForceFullExitWithDelegation` in `force_exit.go` also calls `claimRewards` first and is a migration/upgrade path, not a user-callable `MsgServer` method: [7](#0-6) 

There is no `MsgServer` method that allows undelegation while skipping or tolerating a failed bonus claim.

---

### Impact Explanation

When the `RewardsPoolName` module account balance is less than the bonus owed to a position, the owner of that position cannot:
- Call `TierUndelegate` (fails at `claimRewards`)
- Call `ExitTierWithDelegation` (same failure)

The owner's staked principal is locked inside the position's delegator sub-account indefinitely. The funds cannot leave the module boundary. This satisfies the Critical scope: *"module-account accounting flaw … lets value leave the intended module/account boundary"* — here, value is **prevented** from leaving when it must be able to.

---

### Likelihood Explanation

Two realistic triggers exist:

1. **Natural pool exhaustion**: The bonus pool is funded externally. If it is not topped up fast enough relative to accrued rewards across all positions, it runs dry. Any position with non-zero accrued bonus then becomes permanently stuck.

2. **Griefing via pool exhaustion**: An attacker with their own position(s) calls `ClaimTierRewards` repeatedly to drain the pool. Once the pool balance drops below the bonus owed to a victim's position, the victim's `TierUndelegate` fails. The attacker does not need any special privilege — `ClaimTierRewards` is a standard user-callable message. [8](#0-7) 

---

### Recommendation

Decouple undelegation from bonus claiming. Two options:

1. **Skip bonus on insufficient pool**: In `processEventsAndClaimBonus`, if `sufficientBonusPoolBalance` fails, advance the checkpoints (so the position is not re-charged later) but pay zero bonus and return `nil` instead of the error. Log the shortfall.

2. **Separate the operations**: Remove the `claimRewards` call from `TierUndelegate` entirely. Require the owner to call `ClaimTierRewards` first (which can fail), and then call `TierUndelegate` (which only calls `undelegate` and `ClearBonusCheckpoints`). This makes the two operations independently failable.

---

```go
func TestTierUndelegateBlockedByEmptyPool(t *testing.T) {

    //    (many SLASH events or long bonded duration)

        Owner:      owner,

    require.ErrorIs(t, err, types.ErrInsufficientBonusPool)
    // 5. Assert: position still delegated; owner cannot recover funds
    pos, _ := k.getPositionState(ctx, posId)
    require.True(t, pos.IsDelegated())
}
```

The exact state delta: `RewardsPoolName.balance < computedBonus` → `TierUndelegate` returns `ErrInsufficientBonusPool` → `pos.Delegation.Shares` unchanged → owner's tokens remain locked in the position delegator account with no user-callable recovery path.

```go
func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}
```

```go
func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
