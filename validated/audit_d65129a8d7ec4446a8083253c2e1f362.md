### Title
Unvalidated P2P Transaction Hash Stored and Served Without Recomputation — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

A malicious p2p peer can send a `TransactionInBlock` protobuf message with a valid transaction body but an arbitrary `transaction_hash` field. The p2p sync client takes this hash verbatim, stores it in the node's DB, and serves it via RPC and to downstream p2p peers. There is no recomputation or verification of the hash against the transaction body or the block header's `transaction_commitment`.

### Finding Description

**Deserialization — no hash check:**

In `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)`, the `transaction_hash` field is decoded directly from the protobuf wire value and returned alongside the decoded transaction body. No canonical hash is computed and compared. [1](#0-0) 

**P2P sync client — explicit TODO, no validation:**

In `parse_data_for_block`, the deserialized `transaction_hash` is pushed directly into `block_body.transaction_hashes`. The code contains an explicit acknowledgment that this is unvalidated: [2](#0-1) 

The only check performed is that the count of received transactions matches `n_transactions` from the stored block header. No check is made against the header's `transaction_commitment`. [3](#0-2) 

**Storage write — wrong hash committed:**

The `BlockBody` (with the attacker-supplied hash) is written directly to storage via `append_body`. The storage layer stores the hash in both `transaction_hash_to_idx_table` (used for hash-based lookups) and `TransactionMetadata.tx_hash` (used for hash-by-index lookups), with no hash recomputation. [4](#0-3) [5](#0-4) 

**No post-storage verification against `transaction_commitment`:**

The block header's `transaction_commitment` (a Merkle root over `(transaction_hash, signature)` pairs) is stored separately during the header sync phase. There is no code path that recomputes the commitment from the stored transaction hashes and compares it to the stored header commitment after the body sync phase. [6](#0-5) 

**RPC serves the wrong hash:**

`get_block_w_full_transactions_and_receipts` and `get_transaction_by_hash` read transaction hashes directly from storage. A lookup by the canonical hash returns "not found"; a lookup by the attacker-supplied hash returns the transaction body. [7](#0-6) [8](#0-7) 

**P2P server re-propagates the wrong hash:**

The p2p server reads `get_block_transaction_hashes` from storage and re-serves it to downstream peers, propagating the attacker-supplied hash further. [9](#0-8) 

### Impact Explanation

Any node running the p2p sync client (full nodes, archive nodes, or the sequencer node itself when syncing) that connects to a malicious peer will:

1. Store an arbitrary `TransactionHash` value in its DB for every transaction the attacker controls.
2. Serve wrong transaction hashes via all RPC endpoints (`starknet_getBlockWithTxHashes`, `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, etc.).
3. Re-propagate the wrong hashes to other p2p peers it serves, spreading the corruption.
4. Make canonical-hash-based lookups return "not found" for affected transactions.

The stored `transaction_commitment` in the block header remains correct (it was set during the separate header sync), creating a permanent inconsistency between the header commitment and the stored transaction hashes that is never detected.

### Likelihood Explanation

The p2p sync protocol is an unprivileged public network interface. Any peer that the syncing node connects to can exploit this. The attacker does not need any special credentials, keys, or operator access. The attack requires only the ability to respond to a `TransactionsRequest` query with a well-formed protobuf message containing a valid transaction body and an arbitrary `transaction_hash` field.

### Recommendation

In `parse_data_for_block` (or immediately after deserialization in the `TryFrom` impl), recompute the canonical transaction hash from the decoded transaction body and chain ID, and compare it to the peer-supplied hash. If they differ, treat it as a `BadPeerError` and disconnect/report the peer. The function `validate_transaction_hash` in `crates/starknet_api/src/transaction_hash.rs` already implements the multi-version hash validation logic needed for this check. [10](#0-9)

fn attacker_hash_passthrough() {
    use starknet_api::felt;
    // Build a minimal valid InvokeV1 protobuf body
    let invoke_v1 = protobuf::transaction_in_block::InvokeV1 {
        sender: Some(/* any valid address */),
        max_fee: Some(felt!(1u64).into()),
        signature: Some(protobuf::AccountSignature { parts: vec![] }),
        nonce: Some(felt!(0u64).into()),
        calldata: vec![],
    };
    let attacker_hash = felt!("0x1"); // Felt::ONE, not the canonical hash
    let tx_in_block = protobuf::TransactionInBlock {
        txn: Some(protobuf::transaction_in_block::Txn::InvokeV1(invoke_v1)),
        transaction_hash: Some(attacker_hash.into()),
    };
    let (_, stored_hash) = <(Transaction, TransactionHash)>::try_from(tx_in_block).unwrap();
    // Passes: stored_hash == TransactionHash(Felt::ONE), not the canonical hash
    assert_eq!(stored_hash.0, felt!("0x1"));
}
