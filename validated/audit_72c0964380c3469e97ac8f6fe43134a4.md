### Title
Zero NLP Minted for Non-Zero Quote Consumed Due to Fixed-Point Division Truncation — (`core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.mintNlp`, the NLP token amount is computed via fixed-point division of `quoteAmount` by `oraclePriceX18`. When `quoteAmount` is small enough that `quoteAmount * 1e18 < oraclePriceX18`, integer truncation yields `nlpAmount = 0`. The user's quote is still fully deducted and distributed to NLP pools, but the user receives zero NLP tokens. The reverse path in `burnNlp` has the same class of truncation: a tiny `nlpAmount` yields `quoteAmount = 0`, so the user burns NLP tokens and receives nothing back.

---

### Finding Description

**Root cause — `mintNlp`:**

In `Clearinghouse.sol` line 466:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

`MathSD21x18.div` computes:

```solidity
int256 result = (int256(x) * ONE_X18) / y;   // ONE_X18 = 1e18
```

So `nlpAmount = (quoteAmount × 1e18) / oraclePriceX18`. When `quoteAmount × 1e18 < oraclePriceX18`, Solidity integer division truncates to zero.

After this, the function unconditionally:
- Credits `nlpAmount = 0` to the sender (line 473) — no NLP received
- Debits the full `quoteAmount` from the sender (line 476) — quote is lost
- Distributes the full `quoteAmount` to NLP pools via `_applyNlpRebalance` (line 477)

There is no minimum-amount guard on `mintNlp`, unlike `depositCollateral` which has `isValidDepositAmount` / `checkMinDeposit`.

**Root cause — `burnNlp` (reverse operation):**

In `Clearinghouse.sol` line 502:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
```

`MathSD21x18.mul` computes `(nlpAmount × oraclePriceX18) / 1e18`. When `nlpAmount × oraclePriceX18 < 1e18`, this truncates to zero. The user's `nlpAmount` NLP tokens are still burned (line 511), but `quoteAmount = 0` means no quote is returned.

---

### Impact Explanation

**`mintNlp`:** A user who submits a `MintNlp` transaction with a `quoteAmount` below the truncation threshold loses their entire quote balance delta to the NLP pools while receiving zero NLP tokens. The corrupted state delta is: `sender.quoteBalance -= quoteAmount` (non-zero), `sender.nlpBalance += 0`. The quote is permanently redistributed to NLP pool subaccounts.

**`burnNlp`:** A user who submits a `BurnNlp` transaction with a tiny `nlpAmount` loses those NLP tokens to `N_ACCOUNT` while receiving zero quote in return. The corrupted state delta is: `sender.nlpBalance -= nlpAmount` (non-zero), `sender.quoteBalance += 0`.

Both paths result in direct, irreversible asset loss for the user with no protocol-level revert.

---

### Likelihood Explanation

The truncation threshold for `mintNlp` is `quoteAmount < oraclePriceX18 / 1e18`. Since `oraclePriceX18` is the NLP price expressed in 18-decimal fixed point (e.g., `1e18` = 1 USDC), any `quoteAmount` below 1 unit in the 18-decimal representation triggers the bug. This is reachable by any user who signs a `MintNlp` transaction with a small amount — whether due to a UI rounding error, a programmatic integration, or deliberate testing. No privileged access is required; the user's own signed transaction is the entry path.

---

### Recommendation

Add a minimum-amount check in `mintNlp` analogous to the existing `checkMinDeposit` guard in `depositCollateral`. Specifically, require `nlpAmount > 0` after the division, and revert if it is zero:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
require(nlpAmount > 0, ERR_DEPOSIT_TOO_SMALL);
```

Apply the symmetric check in `burnNlp`:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
require(quoteAmount > 0, ERR_AMOUNT_TOO_SMALL);
```

---

### Proof of Concept

Assume `oraclePriceX18 = 2e18` (NLP price = 2 USDC in 18-decimal fixed point).

**`mintNlp` path:**
- User submits `MintNlp` with `quoteAmount = 1` (1 unit in 18-decimal = 1e-18 USDC)
- `nlpAmount = (1 × 1e18) / 2e18 = 0` (integer truncation)
- Line 473: `updateBalance(NLP_PRODUCT_ID, sender, 0)` — sender gets 0 NLP
- Line 476: `updateBalance(QUOTE_PRODUCT_ID, sender, -1)` — sender loses 1 unit of quote
- Line 477: `_applyNlpRebalance(...)` distributes 1 unit of quote to NLP pools
- Net: sender lost `quoteAmount = 1`, received `nlpAmount = 0`

**`burnNlp` path:**
- User submits `BurnNlp` with `nlpAmount = 1`
- `quoteAmount = (1 × 2e18) / 1e18 = 2` — this case does not truncate to zero
- For truncation: use `nlpAmount = 1` with `oraclePriceX18 = 5e17` (NLP price = 0.5 USDC): `quoteAmount = (1 × 5e17) / 1e18 = 0`
- Line 511: `updateBalance(NLP_PRODUCT_ID, sender, -1)` — sender loses 1 NLP token
- Line 514: `quoteAmount > 0` is false, so no quote is credited
- Net: sender lost `nlpAmount = 1`, received `quoteAmount = 0`
