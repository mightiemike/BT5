### Title
NEAR Relayer Private Key Stored as Plaintext `String` with `Debug`/`Serialize` Derives Enables Key Exfiltration and Trusted-Relayer Impersonation ‚Äî (`File: relayer/src/config.rs`)

---

### Summary

The `NearConfig` struct holds the NEAR signing private key as a bare `String` field and derives `Debug`, `Clone`, `Serialize`, and `Deserialize`. The outer `Config` struct inherits the same derives. Any code path that formats the config with `{:?}` ‚Äî including Rust panic messages, `unwrap()` failures, or structured-log calls ‚Äî will emit the raw private key in plaintext. An attacker who reads those logs can then sign NEAR transactions as the trusted relayer and submit arbitrary Bitcoin block headers to the on-chain `BtcLightClient` contract, directly corrupting the canonical chain state.

---

### Finding Description

**Plaintext key field with unsafe derives ‚Äî `relayer/src/config.rs`**

```rust

    ...
    pub private_key: String,                       // line 44
    ...

``` [1](#0-0) 

The `Debug` derive means `format!("{:?}", config)` or `format!("{:?}", config.near)` ‚Äî which is the default output of any `unwrap()` / `expect()` panic on a value that contains the config ‚Äî will print the raw private key string. The `Serialize` derive means the key is emitted verbatim if the struct is ever serialised to JSON (e.g., for structured logging, telemetry, or configuration export).

**Key read as a plain intermediate `String` ‚Äî `relayer/src/near_client.rs`**

```rust

let private_key = near_crypto::SecretKey::from_str(private_key.as_str()).unwrap(); // line 106
``` [2](#0-1) 

The intermediate `String` is never zeroed. If `SecretKey::from_str` panics (e.g., malformed key), the panic message produced by `unwrap()` will include the raw key value. The resulting `InMemorySigner` then holds the key in heap memory for the entire process lifetime with no protection.

**The stolen key unlocks the trusted-relayer gate ‚Äî `contract/src/lib.rs`**

``` [4](#0-3) 

`submit_blocks` is gated by `#[trusted_relayer]`. An attacker who possesses the relayer's NEAR private key can sign and broadcast `submit_blocks` calls as the legitimate relayer, bypassing this gate entirely.

---

### Impact Explanation

A successful key exfiltration lets the attacker:

1. Call `submit_blocks` as the trusted relayer with attacker-crafted `BlockHeader` values.
2. Drive a chain reorganisation by submitting a fork with higher `chain_work`, corrupting `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height`.
3. Cause `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return `true` for transactions that never existed in the real Bitcoin chain, enabling fraudulent cross-chain proofs for any downstream consumer of the light client.

The corrupted canonical mapping is the `headers_pool` / `mainchain_height_to_header` state, and the broken invariant is that only headers with valid PoW and correct chain linkage should ever be accepted as canonical.

---

### Likelihood Explanation

The `Debug` derive is idiomatic Rust and is routinely triggered by:
- `unwrap()` / `expect()` on a `Result` or `Option` that contains or references the config.
- Structured logging frameworks that serialise the entire context object.
- Any future developer adding a `log::debug!("{:?}", config)` line during troubleshooting.

The relayer is a long-running daemon that connects to external RPC endpoints; transient network errors are common and frequently cause `unwrap()` panics or logged error contexts. The risk is therefore realistic, not merely theoretical.

---

### Recommendation

1. **Wrap the key in a redacting newtype.** Implement `Debug` manually to emit `[REDACTED]` and do not derive `Serialize`/`Deserialize` on the wrapper.
2. **Clear the intermediate `String` after use.** After constructing `SecretKey`, overwrite the `String` bytes before dropping it (or use a `zeroize`-enabled type).
3. **Remove `private_key` from `NearConfig` after the signer is built.** The field should not persist in the long-lived `Config` object; move it into a one-shot initialisation function.
4. **Do not derive `Serialize` on any struct that contains key material.** Use a separate DTO for serialisation that explicitly omits sensitive fields.
