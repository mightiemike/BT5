### Title
`op_unknown` Cost Guard Checks Pre-Multiplier Value While Returning Post-Multiplier Cost — (`src/more_ops.rs`)

### Summary
`op_unknown()` in `src/more_ops.rs` calls `check_cost(cost, max_cost)` against the **pre-multiplier** cost, then immediately multiplies that cost by `(cost_multiplier + 1)` and returns the **post-multiplier** value. The guard and the returned cost use two different metrics, exactly mirroring the original report's pattern: one value is used to decide "is this allowed?" and a different, larger value is what actually gets charged.

### Finding Description
In `op_unknown()`:

```rust
// line 260 — guard checks pre-multiplier cost
check_cost(cost, max_cost)?;
// line 261 — multiplier applied AFTER the guard
cost *= cost_multiplier + 1;
// line 265 — post-multiplier cost is what is returned to the caller
Ok(Reduction(cost as Cost, allocator.nil()))
``` [1](#0-0) 

`cost_multiplier` is decoded from the opcode bytes as a `u32` (up to `0xFFFF_FFFF`), so `cost_multiplier + 1` can be up to `4_294_967_296`. For `cost_function = 0`, the pre-multiplier cost is exactly `1`. The guard `check_cost(1, max_cost)` passes for any `max_cost ≥ 1`, yet the returned cost can be up to `u32::MAX = 4_294_967_295`. [2](#0-1) 

For `cost_function` values 1–3, the inner-loop `check_cost` calls also use the pre-multiplier accumulator: [3](#0-2) 

All of these checks are against the pre-multiplier cost, while the value ultimately returned to `run_program` is the post-multiplier cost.

### Impact Explanation
`op_unknown` is the handler for all unknown opcodes in lenient (consensus) mode. The caller, `apply_op`, passes `max_cost = effective_max_cost - current_cost` (the remaining budget): [4](#0-3) 

`op_unknown` can return a cost far exceeding that remaining budget without raising `CostExceeded` itself. The main `run_program` loop does catch the overage on the **next** iteration: [5](#0-4) 

However, the internal `check_cost` guard is structurally broken: it enforces a limit on the pre-multiplier cost while the actual charge is the post-multiplier cost. The invariant "if `check_cost` passes, the returned cost ≤ `max_cost`" — which holds for every other operator — is violated here. Any node logic that relies on `op_unknown` itself raising `CostExceeded` when the budget is tight (e.g., inside a softfork guard where `effective_max_cost = sf.expected_cost`) will see the guard silently pass and then have the overage caught one step later, creating a subtle ordering difference from all other operators. [6](#0-5) 

### Likelihood Explanation
The trigger is fully attacker-controlled: any CLVM program submitted in consensus mode can include an unknown opcode whose byte encoding sets `cost_function = 0` and `cost_multiplier = 0xFFFF_FFFE` (maximum valid multiplier before the `u32::MAX` overflow guard fires). The pre-multiplier cost is `1`, which passes any non-zero budget check, while the returned cost is `0xFFFF_FFFF = 4_294_967_295`. This is reachable on every full node that evaluates CLVM in lenient mode. [7](#0-6) 

### Recommendation
Move `check_cost` to after the multiplier is applied, and before the `u32::MAX` overflow guard:

```rust
assert!(cost > 0);
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;
Ok(Reduction(cost as Cost, allocator.nil()))
```

This makes the guard consistent with every other operator: the value checked is the value returned.

### Proof of Concept
Craft a 5-byte unknown opcode: bytes `[0x3F, 0xFF, 0xFF, 0xFF, 0x00]`.
- Last byte `0x00`: `cost_function = (0x00 >> 6) & 0b11 = 0` → constant cost = 1.
- First 4 bytes `[0x3F, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = 0x3FFF_FFFF`.
- Pre-multiplier cost = 1; `check_cost(1, max_cost)` passes for any `max_cost ≥ 1`.
- Post-multiplier cost = `1 * (0x3FFF_FFFF + 1) = 0x4000_0000 = 1_073_741_824`.

With `max_cost = 100`, `check_cost(1, 100)` passes inside `op_unknown`, but the function returns `Reduction(1_073_741_824, nil)`. The main loop then catches the overage — but `op_unknown` itself never raised `CostExceeded`, violating the operator contract that the guard and the returned cost use the same metric. [1](#0-0)

fn unknown_operator(
    allocator: &mut Allocator,
    o: NodePtr,
    args: NodePtr,
    flags: ClvmFlags,
    max_cost: Cost,

    if flags.contains(ClvmFlags::NO_UNKNOWN_OPS) {
        Err(EvalErr::Unimplemented(o))?
