### Title
`#[pause]` Modifier on `verify_transaction_inclusion_v2` Causes Permanent Failure for In-Flight Consumer Cross-Contract Calls — (`File: contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion_v2` and `verify_transaction_inclusion` are gated by the `#[pause]` macro. Consumer contracts (bridges, atomic swaps, cross-chain lending protocols) that call these functions via cross-contract calls will receive a permanent panic failure whenever the light client is paused. NEAR has no automatic retry mechanism for failed cross-contract receipts, so any consumer operation that depends on a successful verification result is permanently broken for the duration of the pause, with no built-in recovery path.

---

### Finding Description

`BtcLightClient` derives `Pausable` from `near-plugins` and applies the `#[pause]` attribute to both SPV verification entry points: [1](#0-0) [2](#0-1) 

When the contract is paused, any call to either function panics immediately before executing any logic. `verify_transaction_inclusion_v2` also internally calls `verify_transaction_inclusion` (which carries its own `#[pause]` guard), creating a double-gate: [3](#0-2) 

The wiki explicitly describes these functions as the foundational layer for "bridges, atomic swaps, and cross-chain lending protocols." These consumers invoke `verify_transaction_inclusion_v2` via NEAR cross-contract calls. In NEAR's asynchronous execution model, a scheduled receipt that panics is not retried — the failure is propagated back to the caller's callback. If the consumer contract does not handle the failure callback gracefully (e.g., by refunding or re-queuing), the associated funds or state transition are permanently stuck.

The problematic scenario:
1. A user initiates a Bitcoin-backed operation on a consumer contract (e.g., a bridge unlock or atomic swap).
2. The consumer contract schedules a cross-contract call to `verify_transaction_inclusion_v2`.
3. The `BtcLightClient` is paused (emergency or maintenance) before or during receipt execution.
4. The receipt panics due to `#[pause]`.
5. The consumer contract's failure callback fires; if it lacks a retry or refund path, the user's funds are stuck.
6. There is no automatic re-delivery mechanism in NEAR — the only recovery is manual intervention.

---

### Impact Explanation

Consumer contracts that gate fund releases or state transitions on a successful `verify_transaction_inclusion_v2` result will permanently fail for any in-flight call made while the light client is paused. Funds locked in such consumer contracts cannot be recovered automatically. The impact matches the reported class: assets become stuck with no built-in recovery mechanism, requiring manual intervention.

---

### Likelihood Explanation

The `PauseManager` role can pause the contract at any time for any reason (emergency, upgrade, bug response). The light client is described as infrastructure for multiple production protocols. Any pause — even a brief one — that coincides with an in-flight consumer cross-contract call triggers the failure. Given that the contract is designed to be paused during incidents, and that consumer protocols will continuously issue verification calls, the overlap probability is non-trivial in production.

---

### Recommendation

Remove `#[pause]` from `verify_transaction_inclusion_v2` and `verify_transaction_inclusion`. These are read-only verification functions; pausing them provides no meaningful protection against the threats that typically motivate a pause (e.g., a bad header being submitted), while creating a permanent failure risk for all downstream consumers.

If pause protection is still desired for these functions, implement a queuing or retry mechanism: allow the consumer to re-submit a failed verification request after the contract is unpaused, rather than requiring the entire upstream operation to be restarted.

---

### Proof of Concept

```
1. Deploy BtcLightClient with a valid genesis chain.
2. Deploy a consumer contract C that:
   a. Accepts a user deposit.
   b. Calls verify_transaction_inclusion_v2 on the light client.
   c. In its success callback: releases funds.
   d. In its failure callback: does nothing (or panics).
3. User calls C with a deposit; C schedules the cross-contract call.
4. PauseManager calls `pa_pause_feature` on BtcLightClient before the receipt executes.
5. The receipt for verify_transaction_inclusion_v2 panics ("Contract is paused").
6. C's failure callback fires; user's deposit is stuck in C with no recovery path.
7. Even after unpausing, the original receipt is gone — the user must restart from step 3,
   but C has no mechanism to re-trigger the verification for the already-deposited funds.
``` [4](#0-3) [5](#0-4)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]

    }
