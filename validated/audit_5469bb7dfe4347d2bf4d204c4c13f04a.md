### Title
Division-by-Zero in `socializeSubaccount` When `openInterest == 0` and `vQuoteBalance < 0` Permanently Blocks Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when attempting to spread a residual negative `vQuoteBalance` across other participants. No guard exists for the case where `state.openInterest == 0`. Because `_finalizeSubaccount` in `ClearinghouseLiq` *requires* all perp `balance.amount` values to be zero before calling `socializeSubaccount`, the liquidatee's own position contributes zero to `openInterest`. If no other participants hold open positions in that product, `openInterest` is zero, and the division reverts with `"DBZ"`, permanently blocking finalization.

---

### Finding Description

**Entry path:** `ClearinghouseLiq._finalizeSubaccount` → `perpEngine.socializeSubaccount`

**Step 1 — Precondition enforced by `_finalizeSubaccount`:**

`_finalizeSubaccount` requires every perp `balance.amount` for the liquidatee to be zero before proceeding: [1](#0-0) 

This means the liquidatee contributes **zero** to `state.openInterest` for every product.

**Step 2 — `socializeSubaccount` divides by `state.openInterest` without a zero-guard:** [2](#0-1) 

The division at line 166–168 is reached only when insurance is insufficient to cover `balance.vQuoteBalance`. There is no `if (state.openInterest == 0)` guard before it.

**Step 3 — `MathSD21x18.div` explicitly reverts on zero denominator:** [3](#0-2) 

`require(y != 0, ERR_DIV_BY_ZERO)` fires, reverting the entire `_finalizeSubaccount` call.

**Step 4 — `openInterest` can legitimately be zero:**

`state.openInterest` is the sum of absolute position sizes across all participants, maintained in `_updateBalance`: [4](#0-3) 

If the liquidatee is the sole (or last) participant in a product and their `balance.amount` is already zero, `openInterest` is zero. A non-zero `vQuoteBalance` can persist independently — it accumulates from funding payments, realized losses, or PnL settlement and is not zeroed when `balance.amount` reaches zero.

---

### Impact Explanation

Every subsequent attempt to finalize the liquidatee's subaccount reverts at the same point. Because the revert undoes all state changes, the subaccount is permanently stuck: collateral cannot be recovered, insurance allocated for the finalization is consumed by the failed transaction, and the liquidation flow is irrecoverably blocked for that subaccount.

---

### Likelihood Explanation

The condition is reachable in normal protocol operation:
- A product with a single active participant (or one where all other participants have closed positions) has `openInterest == 0`.
- The liquidatee can have `balance.amount == 0` (required by finalization) but `vQuoteBalance < 0` from accumulated funding or realized losses on a previously closed position.
- If the insurance fund is depleted or insufficient to cover the full deficit, the inner branch at line 164 is entered and the revert fires.

No privileged access, governance capture, or external dependency failure is required.

---

### Recommendation

Add an explicit guard before the division. If `state.openInterest == 0`, there are no counterparties to socialize against; the loss should be written off directly (e.g., zeroed out and tracked separately, or absorbed by the protocol):

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the residual loss
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```
