### Title
`_protocol_version` Silently Discarded in `calculate_reward` Permanently Locks `protocol_reward_rate` at Hardcoded `1/10` for Mainnet/Testnet, Breaking Protocol Upgrade Boundary — (`chain/epoch-manager/src/reward_calculator.rs`)

---

### Summary

`RewardCalculator::calculate_reward` accepts a `_protocol_version` parameter but silently discards it (the underscore prefix is explicit). Instead of using the current epoch's protocol version to look up the treasury split, it uses `self.genesis_protocol_version` — set once at node startup from `GenesisConfig` — to decide whether to apply a hardcoded `protocol_reward_rate` of `1/10`. For every mainnet and testnet node (`genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION == 29`), the treasury fraction is permanently locked at `Rational32::new_raw(1, 10)` regardless of any future protocol upgrade. Meanwhile, `max_inflation_rate` is correctly versioned through the epoch config store and changes across protocol versions. The two economic parameters are treated asymmetrically, breaking the protocol upgrade boundary for `protocol_reward_rate`.

---

### Finding Description

In `finalize_epoch`, the epoch manager calls `calculate_reward` with the current epoch's protocol version and the epoch-config-sourced `max_inflation_rate`: [1](#0-0) 

Inside `calculate_reward`, the `_protocol_version` argument is accepted but immediately discarded: [2](#0-1) 

The treasury split is then determined solely by the genesis protocol version, not the current epoch's version: [3](#0-2) 

`PROD_GENESIS_PROTOCOL_VERSION` is `29`, the genesis version for both mainnet and testnet: [4](#0-3) 

So for every mainnet/testnet node, `use_hardcoded_value` is always `true`, and `protocol_reward_rate` is always `Rational32::new_raw(1, 10)`, regardless of what any future epoch config says.

By contrast, `max_inflation_rate` is correctly read from the versioned epoch config at the call site and passed as a parameter — the `test_adjust_max_inflation` test explicitly verifies it changes from `1/20` to `1/40` across protocol versions: [5](#0-4) 

The test comment at line 597 even acknowledges the asymmetry: `// Unused, would only be used for genesis_protocol_version`. [6](#0-5) 

---

### Impact Explanation

`protocol_reward_rate` controls what fraction of `epoch_total_reward` is minted to the protocol treasury account versus distributed to validators: [7](#0-6) 

If a future protocol upgrade intends to change `protocol_reward_rate` (e.g., reduce treasury share to zero, or increase it), the change is silently ignored for all mainnet/testnet nodes. The hardcoded `1/10` persists. With approximately 2.5% annual inflation on ~1 billion NEAR, the treasury receives roughly 2.5 million NEAR per year. A protocol upgrade setting `protocol_reward_rate` to `0` (all rewards to validators) would still silently send ~2.5 million NEAR per year to the treasury. Conversely, a reduction intended to benefit validators would be silently suppressed. The divergence is permanent, produces no error or warning, and affects every epoch finalization on mainnet and testnet.

---

### Likelihood Explanation

`max_inflation_rate` has already been changed via protocol upgrades (halved from `1/20` to `1/40` at a recent protocol version), demonstrating that economic parameters in the epoch config are actively modified. `protocol_reward_rate` is a structurally identical parameter. Any future governance decision to adjust the treasury fraction — a plausible economic policy change — would trigger this silent divergence. The broken upgrade boundary is reachable on every epoch boundary on mainnet and testnet.

---

### Recommendation

Remove the underscore from `_protocol_version` and use it to look up `protocol_reward_rate` from the versioned epoch config, exactly as `max_inflation_rate` is already handled. The `protocol_reward_rate` should be added to the epoch config store (alongside `max_inflation_rate`) and passed as a parameter to `calculate_reward`, eliminating the genesis-version-based hardcoding entirely.

---

### Proof of Concept

1. A future protocol upgrade sets `protocol_reward_rate` to `0` in the epoch config for protocol version N.
2. `finalize_epoch` calls `calculate_reward(..., epoch_protocol_version=N, ..., epoch_config.max_inflation_rate)`.
3. Inside `calculate_reward`, `_protocol_version` (= N) is discarded.
4. `use_hardcoded_value = (self.genesis_protocol_version == 29)` evaluates to `true` for all mainnet/testnet nodes.
5. `protocol_reward_rate` is set to `Rational32::new_raw(1, 10)` instead of `0`.
6. `epoch_protocol_treasury` = `epoch_total_reward / 10` is minted to the treasury account every epoch, contrary to the protocol upgrade's intent.
7. Validators receive `9/10` of rewards instead of the full `10/10` intended by the upgrade.
8. No error is raised; the divergence is invisible to the protocol upgrade mechanism. [8](#0-7)

    pub fn calculate_reward(
        &self,
        validator_block_chunk_stats: HashMap<AccountId, BlockChunkValidatorStats>,
        validator_stake: &HashMap<AccountId, Balance>,
        total_supply: Balance,
        _protocol_version: ProtocolVersion,
        epoch_duration: u64,
        online_thresholds: ValidatorOnlineThresholds,
        max_inflation_rate: Rational32,
    ) -> (HashMap<AccountId, Balance>, Balance) {
        let mut res = HashMap::new();
        let num_validators = validator_block_chunk_stats.len();
        let use_hardcoded_value = self.genesis_protocol_version == PROD_GENESIS_PROTOCOL_VERSION;
        let protocol_reward_rate = if use_hardcoded_value {
            Rational32::new_raw(1, 10)
        } else {
            self.protocol_reward_rate
        };
        let epoch_total_reward = Balance::from_yoctonear(
            (U256::from(*max_inflation_rate.numer() as u64)
                * U256::from(total_supply.as_yoctonear())
                * U256::from(epoch_duration)
                / (U256::from(self.num_seconds_per_year)
                    * U256::from(*max_inflation_rate.denom() as u64)
                    * U256::from(NUM_NS_IN_SECOND)))
            .as_u128(),
        );
        let epoch_protocol_treasury = Balance::from_yoctonear(
            (U256::from(epoch_total_reward.as_yoctonear())
                * U256::from(*protocol_reward_rate.numer() as u64)
                / U256::from(*protocol_reward_rate.denom() as u64))
            .as_u128(),
        );
        res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
        if num_validators == 0 {
            return (res, Balance::ZERO);
        }
        let epoch_validator_reward =
            epoch_total_reward.checked_sub(epoch_protocol_treasury).unwrap();
