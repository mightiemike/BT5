### Title
`SwapAllowlistExtension` gates on the router address instead of the end-user, allowing any unprivileged user to bypass a curated pool's swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool level is the router contract, not the end-user. A pool admin who configures the allowlist to restrict swaps to specific addresses must either (a) allowlist the router — granting every user who calls through it unrestricted access — or (b) not allowlist the router — breaking the router for all users including legitimately allowlisted ones. Either branch defeats the allowlist's purpose.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is whatever the pool passed as the first argument to `_beforeSwap`. The pool always passes its own `msg.sender`:

```solidity
// MetricOmmPool.sol L230-240 (beforeSwap call)
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
)
```

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap(...)`. The pool's `msg.sender` is the router. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The pool's swap callback mechanism confirms the router is the direct caller:

```solidity
// MetricOmmPool.sol L258
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
```

`msg.sender` must implement `IMetricOmmSwapCallback`, which is the router — not the end-user.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the second parameter), which is the LP position owner explicitly passed by the caller and preserved through the periphery path. The swap extension has no equivalent explicit-owner parameter; it relies solely on `sender`, which collapses to the router address.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to gate swaps to KYC'd or otherwise curated addresses is fully bypassable by any user who calls through `MetricOmmSimpleRouter`:

- If the pool admin allowlists the router (necessary for the router to function), every user who routes through it inherits the router's allowlist status. The per-user restriction is nullified.
- Any non-allowlisted user can execute swaps in a curated pool, draining LP assets at oracle-derived prices that were only intended for approved counterparties.
- This is a direct loss of LP principal and a broken core pool invariant: the allowlist guard silently fails open for the primary supported swap path.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint described in the protocol architecture. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router is immediately affected. The attacker needs no special privilege — only the ability to call the public router.

---

### Recommendation

Pass the actual initiating user through the hook chain. Two options:

1. **Add an explicit `initiator` field** to the `beforeSwap` hook signature (analogous to `owner` in `beforeAddLiquidity`) and have the router supply `msg.sender` (the end-user) when calling `pool.swap`.

2. **Check `recipient` as a proxy** only if the protocol guarantees that the recipient is always the economic beneficiary and cannot be spoofed — but this is fragile and not currently guaranteed.

The cleanest fix mirrors the deposit path: the router should pass the actual user identity as a dedicated parameter that the pool forwards to extensions, rather than relying on `msg.sender` which is always the router.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only `alice`.
2. Pool admin also allowlists the `MetricOmmSimpleRouter` address (required for the router to function).
3. `bob` (not in the allowlist) calls `MetricOmmSimpleRouter.swap(pool, ...)`.
4. The router calls `pool.swap(recipient=bob, ...)`.
5. The pool calls `_beforeSwap(msg.sender=router, recipient=bob, ...)`.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router]` → `true`.
7. The swap executes. `bob` receives tokens from the curated pool despite never being allowlisted.

The allowlist guard is bypassed without any privileged action by `bob`.
