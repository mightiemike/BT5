### Title
Unchecked `u64` Multiplication in `op_unknown` Cost Calculation Enables Cost Undercharging — (File: `src/more_ops.rs`)

---

### Summary

The `op_unknown` function in `src/more_ops.rs` performs a plain (unchecked) `u64` multiplication `cost *= cost_multiplier + 1` after a pre-multiplication cost check. In Rust release builds, integer overflow wraps silently. An attacker who controls both the unknown opcode bytes (which encode `cost_multiplier`) and the opcode arguments (which drive the pre-multiplication `cost`) can engineer a wrap-around such that the post-multiplication value falls at or below `u32::MAX`, bypassing the intended cost cap and causing the operation to be charged a near-zero cost.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`), the cost for an unknown opcode is computed in two stages:

**Stage 1 — argument-driven cost accumulation** (lines 209–256):

```rust
let mut cost = match cost_function {
    0 => 1,
    1 => { /* ARITH_BASE_COST + n*ARITH_COST_PER_ARG + bytes*ARITH_COST_PER_BYTE */ }
    2 => { /* MUL-like quadratic */ }
    3 => { /* CONCAT_BASE_COST + n*CONCAT_COST_PER_ARG + bytes*CONCAT_COST_PER_BYTE */ }
    _ => 1,
};
``` [1](#0-0) 

**Stage 2 — multiplier application** (lines 258–266):

```rust
assert!(cost > 0);
check_cost(cost, max_cost)?;          // ensures cost ≤ max_cost (u64)
cost *= cost_multiplier + 1;          // ← UNCHECKED u64 multiplication
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [2](#0-1) 

`cost_multiplier` is derived from the opcode bytes via `u32_from_u8`, so it is at most `u32::MAX = 4,294,967,295`: [3](#0-2) 

Therefore `cost_multiplier + 1` is at most `4,294,967,296 = 2^32`. `Cost` is `u64`: [4](#0-3) 

The multiplication `cost *= cost_multiplier + 1` is a plain Rust `u64 *= u64`. In release mode, Rust wraps on overflow without panicking. If `cost` (post-`check_cost`, so ≤ `max_cost`) is large enough that `cost × (cost_multiplier + 1) > u64::MAX`, the result wraps modulo 2⁶⁴. If the wrapped value is ≤ `u32::MAX`, the subsequent guard `if cost > u32::MAX` passes, and the function returns `Ok(Reduction(wrapped_cost, nil))` — a drastically undercharged cost.

**Concrete overflow scenario:**

- `cost_multiplier = 0xFFFFFFFF` → `cost_multiplier + 1 = 2^32 = 4,294,967,296`
- Attacker crafts arguments for `cost_function = 3` (concat-like) to produce `cost = k × 2^32` for any integer `k ≥ 1`
- `cost × 2^32 mod 2^64 = (k × 2^32) × 2^32 mod 2^64 = k × 2^64 mod 2^64 = 0`
- Post-multiplication `cost = 0`, which satisfies `0 ≤ u32::MAX`
- Function returns `Ok(Reduction(0, nil))` — zero cost for the operation

With `cost_function = 3`: `cost = 142 + 135n + 3b`. To reach `cost = 4,294,967,296` (the smallest multiple of 2^32 above zero), the attacker needs approximately 31.8 million nil-atom arguments (within the allocator's `MAX_NUM_PAIRS = 62,500,000` limit) and a `max_cost` of at least 4.3 billion — achievable when `run_program` is called with `max_cost = 0` (which becomes `Cost::MAX`): [5](#0-4) [6](#0-5) 

---

### Impact Explanation

`op_unknown` is invoked in lenient mode (when `allow_unknown_ops()` returns true), which is used during mempool validation. The cost model is the primary spam-prevention mechanism. By engineering a cost wrap-around to 0 (or any small value), an attacker can include unknown opcodes in CLVM programs that pass mempool cost checks while being charged near-zero fees. This undermines the cost model's integrity for mempool admission, enabling cost-free spam of unknown-op-bearing transactions. The corrupted value is the `Cost` field of the returned `Reduction`.

---

### Likelihood Explanation

Moderate-low. The attacker must control both the opcode bytes (trivially done — they write the CLVM program) and must accumulate a pre-multiplication `cost` that is a precise multiple of `2^32 / gcd(2^32, cost_multiplier+1)`. This requires a large number of arguments (tens of millions for cost_function = 3), which is within allocator limits but requires a large program. The attack is more feasible when `max_cost` is large or uncapped (e.g., `max_cost = 0` passed by a caller). The vulnerability is latent in all release builds since Rust wraps silently.

---

### Recommendation

Replace the unchecked multiplication with a checked variant and treat overflow as an invalid opcode:

```rust
// Before:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .filter(|&c| c <= u32::MAX as u64)
    .ok_or(EvalErr::Invalid(o))?;
```

This ensures that any overflow — or any result exceeding `u32::MAX` — is uniformly rejected, eliminating the wrap-around bypass.

### Proof of Concept

Entry path:
1. Caller invokes `run_program` with `max_cost = 0` (→ `Cost::MAX`) and a CLVM program containing an unknown opcode.
2. The unknown opcode bytes are crafted as `[0xFF, 0xFF, 0xFF, 0xFF, 0x40]`: the first four bytes encode `cost_multiplier = 0xFFFFFFFF`; the last byte has bits 7–6 = `01` → `cost_function = 1`.
3. The program passes ~13.4 million atom arguments (each 0 bytes) to the unknown op, driving `cost = 99 + 320 × 13,421,772 = 4,294,967,139` — close to `2^32`. Fine-tuning with a few byte-length arguments reaches a multiple of `2^32`.
4. `cost *= 4,294,967,296` wraps to `0` in release mode.
5. `0 ≤ u32::MAX` → function returns `Ok(Reduction(0, nil))`.
6. The `run_program` loop adds 0 to the running cost; the program passes the cost budget check with zero cost charged for the unknown op. [2](#0-1) [7](#0-6) [8](#0-7)
