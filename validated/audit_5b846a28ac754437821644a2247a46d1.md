### Title
Native ETH Permanently Stuck in `DirectDepositV1` — `ContractOwner` Cannot Receive ETH - (File: `core/contracts/DirectDepositV1.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

`DirectDepositV1` is deployed with a fixed CREATE2 salt and explicitly handles the case where native ETH is pre-sent to its deterministic address before deployment. If wrapping that ETH fails, it stays in the contract silently. The only designated recovery path — `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` — always reverts because `DirectDepositV1.withdrawNative()` pushes ETH to `msg.sender` (i.e., `ContractOwner`), and `ContractOwner` has no `receive()` or `fallback()` function.

---

### Finding Description

`DirectDepositV1` is deployed by `ContractOwner.createDirectDepositV1()` using a fixed salt (`bytes32(uint256(1))`), making its CREATE2 address deterministic and publicly computable. [1](#0-0) 

The constructor of `DirectDepositV1` explicitly anticipates ETH being pre-sent to this address. If the wrapping call to `wrappedNative` fails, the ETH is silently retained in the contract (no revert, only an event emit). The comment even acknowledges the risk: *"shouldn't revert even if the transfer fails, otherwise the funds will be stuck in the DDA forever."* [2](#0-1) 

The intended recovery path is `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`, which calls `DirectDepositV1.withdrawNative()`: [3](#0-2) 

`withdrawNative()` pushes ETH to `msg.sender` via a low-level call with a hard `require(success, ...)`: [4](#0-3) 

Since `DirectDepositV1` is `Ownable` and its owner is `ContractOwner` (set at construction), `msg.sender` in `withdrawNative()` is always `ContractOwner`. However, `ContractOwner` declares no `receive()` or `fallback()` function — confirmed by a full-text search returning zero matches. The ETH push therefore always fails, `require(success, ...)` always reverts, and the native ETH withdrawal path is permanently broken.

A second ETH-entry vector exists via `selfdestruct` from any contract, which bypasses `receive()` entirely and forces ETH into `DirectDepositV1` with no wrapping attempted.

---

### Impact Explanation

Any native ETH that lands in `DirectDepositV1` — whether from a pre-deployment send to the deterministic address where wrapping fails, or from a `selfdestruct` force-send — is permanently irrecoverable. The only designated withdrawal function always reverts when invoked through `ContractOwner`. The ERC-20 `withdraw()` path does not cover native ETH. There is no alternative recovery mechanism.

**Asset delta:** native ETH balance of `DirectDepositV1` is permanently locked.

---

### Likelihood Explanation

The CREATE2 salt is hardcoded as `bytes32(uint256(1))`, making the deployment address fully predictable on any chain. Anyone can pre-send ETH to that address before `createDirectDepositV1()` is called. If the `wrappedNative` call in the constructor fails (e.g., the wrapped-native contract is paused, has a deposit cap, or is not yet deployed), the ETH stays silently. The constructor's own comment acknowledges this scenario as a known risk. Additionally, `selfdestruct`-based force-sends are always possible. The broken `withdrawNative()` path means recovery is impossible regardless of how the ETH arrived.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, restructure `withdrawNative()` to accept a `recipient` parameter instead of pushing to `msg.sender`, so the ETH can be sent directly to the multisig without routing through `ContractOwner`.

---

### Proof of Concept

1. Compute the deterministic CREATE2 address for `DirectDepositV1` for a given `subaccount` (salt is always `bytes32(uint256(1))`).
2. Send ETH to that address before `createDirectDepositV1(subaccount)` is called.
3. Call `ContractOwner.createDirectDepositV1(subaccount)`. The constructor detects `address(this).balance != 0` and attempts `wrappedNative.call{value: balance}("")`. If this fails (e.g., `wrappedNative` is paused), ETH stays in `DirectDepositV1` and `NativeTokenTransferFailed` is emitted.
4. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
5. This calls `DirectDepositV1.withdrawNative()`, which executes `ContractOwner.call{value: balance}("")`. Since `ContractOwner` has no `receive()` function, the call returns `success = false`.
6. `require(success, "Failed to transfer native token to owner")` reverts. The ETH remains permanently locked in `DirectDepositV1`.
