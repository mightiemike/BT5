Audit Report

## Title
SwapAllowlistExtension Gates `sender` (Router Address) Instead of the End User — Any Unprivileged User Bypasses the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the immediate caller of `pool.swap()`. When a user routes through the public, permissionless `MetricOmmSimpleRouter`, `sender` becomes the router's address rather than the end user's address. If the router is allowlisted — a natural operational state since it is the intended user-facing entry point — any unprivileged user can bypass the allowlist entirely by routing through it.

## Finding Description

`MetricOmmPool.swap` always passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this `sender` verbatim to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` keys its allowlist check on this `sender`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter` is a public, permissionless contract — no access control on `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput`. When any user calls these functions, the router becomes `msg.sender` of the downstream `pool.swap()` call, so `sender` seen by the extension is the router's address, not the end user's address. If the router is allowlisted (e.g., `allowedSwapper[pool][router] = true`), the allowlist check passes for every user who routes through it, regardless of whether they are individually allowlisted.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to approved counterparties (e.g., institutional-only or KYC-gated pools) can be freely traded against by any unprivileged user via `MetricOmmSimpleRouter`. This breaks the core access-control invariant the pool operator deployed the extension to enforce. Unauthorized swaps against a restricted pool can drain LP assets at oracle-determined prices, constituting a direct loss of LP principal. This meets the **Medium/High** threshold: broken core pool functionality causing loss of funds.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, deployed periphery contract requiring no special access.
- Any user blocked by a direct `pool.swap()` call can trivially re-route through the router.
- The only prerequisite is that the router is allowlisted, which is a natural operational state since the router is the intended user-facing entry point.
- The structural mismatch between `sender` (immediate caller) and the intended gated identity (end user) is permanent and cannot be patched by allowlist configuration alone.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must gate the end user, not the immediate pool caller. Two sound approaches:

1. **Check `recipient` instead of `sender`**: For typical swap flows, `recipient` is the address that receives output tokens and is controlled by the end user. This is only correct if the pool operator's intent is to gate the output recipient.
2. **Require the end user to pass their identity in `extensionData`** and verify it against the allowlist, with the pool or router attesting to the binding. This is the most robust approach and avoids reliance on `sender` when public router intermediaries are supported.

## Proof of Concept

```
1. Pool P is deployed with SwapAllowlistExtension; only address ALICE is allowlisted.
2. MetricOmmSimpleRouter R is also allowlisted (standard operational setup):
       allowedSwapper[P][R] = true
3. Attacker BOB (not allowlisted) calls:
       pool.swap(...)  → reverts: BOB not on allowlist (sender = BOB ✗)
4. BOB instead calls:
       MetricOmmSimpleRouter.exactInputSingle(pool=P, ...)
   The router calls pool.swap(...) with msg.sender = R.
   Extension checks allowedSwapper[P][R] → R is allowlisted → passes.
5. BOB's swap executes against the restricted pool.
   The allowlist invariant is broken with zero privileged access.
```
