### Title
Missing Event Emission in `UpdateParams` After Critical Parameter Change - (`x/tieredrewards/keeper/msg_server_auth.go`)

### Summary

The `UpdateParams` handler in the `x/tieredrewards` module silently updates `TargetBaseRewardsRate` — the parameter that governs per-block fund transfers from the rewards pool to the distribution module — without emitting any event. Every other governance-gated handler in the same file (`AddTier`, `UpdateTier`, `DeleteTier`) emits a typed event. The omission creates a blind spot for off-chain indexers, block explorers, and monitoring systems that rely on events to track critical parameter changes.

### Finding Description

In `x/tieredrewards/keeper/msg_server_auth.go`, `UpdateParams` (lines 21–31) calls `ms.SetParams(ctx, msg.Params)` and returns immediately with no call to `ctx.EventManager().EmitTypedEvent` or any equivalent: [1](#0-0) 

By contrast, every other governance handler in the same file emits a typed event via `emitTierChangedEvent`: [2](#0-1) [3](#0-2) [4](#0-3) 

The parameter being silently changed is `TargetBaseRewardsRate`: [5](#0-4) 

This rate is consumed every block by `topUpBaseRewards` in `abci.go`, which computes a shortfall and transfers tokens from the `RewardsPoolName` module account to the distribution module: [6](#0-5) [7](#0-6) 

### Impact Explanation

A change to `TargetBaseRewardsRate` directly and immediately alters the rate at which funds are drained from the rewards pool into the distribution module on every subsequent block. Off-chain systems (indexers, block explorers, monitoring bots, governance dashboards) that subscribe to events to detect parameter changes will receive no signal when this happens. The inconsistency is concrete: the three tier-management governance messages all emit `EventTierChanged`, but the parameter governance message emits nothing, making `UpdateParams` the only silent governance action in the module.

### Likelihood Explanation

The entry path is a standard governance proposal (`MsgUpdateParams`) executable by any governance participant. The omission is triggered on every successful execution of this message. No special conditions are required.

### Recommendation

Add a typed event (e.g., `EventParamsUpdated`) to `UpdateParams` after `SetParams` succeeds, mirroring the pattern used by `AddTier`, `UpdateTier`, and `DeleteTier`:

```go
func (ms msgServer) UpdateParams(ctx context.Context, msg *types.MsgUpdateParams) (*types.MsgUpdateParamsResponse, error) {
    if err := ms.requireAuthority(msg.Authority); err != nil {
        return nil, err
    }
    if err := ms.SetParams(ctx, msg.Params); err != nil {
        return nil, err
    }
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventParamsUpdated{
        Params: msg.Params,
    }); err != nil {
        return nil, err
    }
    return &types.MsgUpdateParamsResponse{}, nil
}
```

Define `EventParamsUpdated` in the proto events file and regenerate.

### Proof of Concept

1. Submit a governance proposal containing `MsgUpdateParams` with a new `TargetBaseRewardsRate`.
2. Proposal passes; `UpdateParams` executes.
3. Query the transaction result events — no `tieredrewards`-module event is present for the parameter change.
4. Compare with a `MsgAddTier` transaction: its result contains `EventTierChanged`.
5. The `BeginBlocker` immediately begins using the new rate to drain the rewards pool at the updated rate, with no on-chain event record of the change. [1](#0-0) [8](#0-7)
