### Title
`withdrawCollateral()` Performs Health Check Against Stale Interest Multipliers Without Calling `updateStates()` — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`withdrawCollateral()` in `Clearinghouse.sol` calls `getHealth()` to validate subaccount solvency after reducing a balance, but never triggers `updateStates()` on `SpotEngine` or `PerpEngine` first. Because SpotEngine stores balances in normalized form and converts them using `cumulativeDepositsMultiplierX18` / `cumulativeBorrowsMultiplierX18`, any accrued-but-unapplied interest causes the health check to operate on understated debt. This path is reachable by any unprivileged user through `executeSlowModeTransaction()`.

---

### Finding Description

SpotEngine stores every balance as a normalized value (`amountNormalized`) and converts it to a real amount by multiplying against the current cumulative multiplier:

```
actualBalance = amountNormalized × cumulativeMultiplierX18
``` [1](#0-0) 

The multipliers are advanced only when `_updateState()` is called, which happens inside `SpotEngine.updateStates()` — an `onlyEndpoint` function driven exclusively by the off-chain sequencer. [2](#0-1) 

`withdrawCollateral()` in `Clearinghouse.sol` reduces the caller's spot balance and then calls `getHealth()` to confirm the subaccount remains solvent: [3](#0-2) 

`getHealth()` delegates to `spotEngine.getHealthContribution()` and `perpEngine.getHealthContribution()`, both of which read the current (potentially stale) cumulative multipliers directly from storage: [4](#0-3) 

Neither `withdrawCollateral()` nor `getHealth()` calls `updateStates()` on either engine before performing this check. The same omission applies to PerpEngine: `cumulativeFundingLongX18` / `cumulativeFundingShortX18` are only advanced by the sequencer's `updateStates()` call, so pending funding payments are invisible to the health check. [5](#0-4) 

---

### Impact Explanation

Because `cumulativeBorrowsMultiplierX18` only grows over time, a stale multiplier **understates a borrower's actual debt**. When `getHealth()` runs against stale state, a subaccount that has accrued enough interest to be unhealthy can appear healthy. The health check passes, and the withdrawal proceeds, leaving the protocol holding more risk than the on-chain accounting reflects.

Concretely:
- A user with a large negative spot balance (borrowed assets) submits a `WithdrawCollateral` slow-mode transaction.
- After the 3-day timeout, the user calls `executeSlowModeTransaction()` — callable by anyone with no access control.
- If `updateStates()` has not been called in the same block (or recently enough), `cumulativeBorrowsMultiplierX18` is stale.
- The health check underestimates the user's debt, passes, and collateral is released. [6](#0-5) 

The same stale-state problem affects `getHealth()` calls inside `mintNlp()`, `burnNlp()`, `transferQuote()`, and `nlpProfitShare()`, all of which check health without first advancing the multipliers. [7](#0-6) 

---

### Likelihood Explanation

The slow-mode path (`executeSlowModeTransaction()`) is permissionless. Any user can queue a `WithdrawCollateral` transaction and execute it themselves after the timeout. The magnitude of the staleness depends on how recently the sequencer last submitted `updateStates()`. During normal operation the window is small (seconds to minutes), but the protocol provides no on-chain guarantee that `updateStates()` has been called before the health check runs. If the sequencer is delayed or the user times the call between sequencer batches, the stale multipliers can produce a materially incorrect health result. The 3-day slow-mode delay also means significant interest can accumulate between the last `updateStates()` and the moment the user executes the withdrawal.

---

### Recommendation

Explicitly call `spotEngine.updateStates()` and `perpEngine.updateStates()` (with `dt = 0` or the elapsed time) at the top of `withdrawCollateral()` before the balance mutation and health check, mirroring the fix applied in the referenced upstream commit. The same pattern should be applied to every function in `Clearinghouse.sol` that calls `getHealth()`.

---

### Proof of Concept

1. User borrows a large amount of a spot asset (negative `amountNormalized` in SpotEngine).
2. User calls `Endpoint.submitSlowModeTransaction(WithdrawCollateral{productId: quoteId, amount: X})`.
3. Three days pass. The sequencer calls `updateStates()` periodically, but the user waits until just after a sequencer batch is processed (so the next `updateStates()` has not yet run).
4. User calls `Endpoint.executeSlowModeTransaction()`.
5. `Clearinghouse.withdrawCollateral()` is invoked. It calls `spotEngine.updateBalance(productId, sender, -amountRealized)` then `getHealth(sender, INITIAL)`.
6. `getHealth()` reads `cumulativeBorrowsMultiplierX18` from storage — this value is stale by the time elapsed since the last sequencer `updateStates()` call.
7. The user's actual debt (`amountNormalized × freshMultiplier`) is larger than what `getHealth()` computes (`amountNormalized × staleMultiplier`).
8. The health check passes; collateral is transferred out.
9. When the sequencer next calls `updateStates()`, the true debt is revealed and the subaccount is undercollateralized. [8](#0-7) [9](#0-8)

        public
        returns (int128 health)
    {
        ISpotEngine spotEngine = _spotEngine();
        IPerpEngine perpEngine = _perpEngine();

        health = spotEngine.getHealthContribution(subaccount, healthType);
        // min health means that it is attempting to borrow a spot that exists outside
        // of the risk system -- return min health to error out this action
        if (health == -INF) {
            return health;
        }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
