### Title
Merkle Proof Leaf Format Lacks Node-Type Context, Enabling Proof-Verification Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a bare 32-byte `tx_id` as the Merkle leaf with no type discriminator to distinguish a real transaction hash (leaf node) from an internal Merkle tree node hash. Because Bitcoin block headers carry no transaction count, the contract cannot validate proof depth on-chain. An unprivileged NEAR caller can supply an internal node hash as `tx_id` with a truncated proof path and receive a `true` return value for a transaction that does not exist.

---

### Finding Description

The proof format used by `verify_transaction_inclusion` is:

```
ProofArgs { tx_id: H256, tx_block_blockhash: H256, tx_index: u64, merkle_proof: Vec<H256>, confirmations: u64 }
```

The leaf is just `tx_id` — a 32-byte hash — identical in structure to any internal Merkle node hash. The verification logic in `compute_root_from_merkle_proof` simply walks the supplied proof path upward and compares the result to the stored `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

There is no check that `merkle_proof.len()` equals the actual tree depth, and no check that `tx_id` is a leaf-level hash rather than an intermediate node. The only guard is `require!(!args.merkle_proof.is_empty(), ...)`, which only prevents a zero-length proof.

This is the direct analog to the external report: the leaf format `(tx_id)` is missing a type discriminator — just as `keccak256(abi.encodePacked(msg.sender, _amount))` was missing a denomination field. In both cases, the proof element is structurally indistinguishable from a different semantic object (internal node vs. leaf; ETH vs. ERC20 amount), and the verifier cannot resolve the ambiguity.

`verify_transaction_inclusion_v2` was introduced to mitigate this via a coinbase proof length check, but `verify_transaction_inclusion` (v1) remains a live, callable public method — it is only `#[deprecated]`, not removed or access-gated beyond the shared `#[pause]` flag.

---

### Impact Explanation

Any unprivileged NEAR caller can invoke `verify_transaction_inclusion` directly and receive `true` for a `tx_id` that corresponds to an internal Merkle node rather than a real transaction. Downstream contracts or off-chain bridges that consume this `true` result — e.g., to confirm a Bitcoin payment before releasing funds — are deceived into treating a non-existent transaction as confirmed. The corrupted value is the proof result itself: a false positive SPV inclusion proof.

---

### Likelihood Explanation

Medium. The function is publicly callable by any NEAR account without any role or stake requirement. It is deprecated but not paused or removed. Any integrator that has not yet migrated to v2 — or that calls v1 directly — is exposed. The attack requires only knowledge of a real block's Merkle tree structure, which is public Bitcoin data.

---

### Recommendation

1. **Remove or hard-disable `verify_transaction_inclusion` (v1)** rather than merely deprecating it. A `#[deprecated]` attribute in Rust is a compile-time lint for Rust callers; it does not prevent NEAR RPC callers from invoking the method directly.
2. **Alternatively**, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` at the top of v1 to make it unconditionally revert for external callers, or gate it behind a privileged role.
3. In `verify_transaction_inclusion_v2`, document explicitly that the equal-length coinbase proof check is the sole depth-validation mechanism, and add a note that single-transaction blocks (where both proof lengths are 0) cannot be verified due to the `is_empty` guard in v1.

---

### Proof of Concept

Given a real Bitcoin block with transactions `[T0, T1, T2, T3]` (4 txs), the Merkle tree is:

```
root = H(H(T0,T1), H(T2,T3))
internal_node_A = H(T0, T1)
internal_node_B = H(T2, T3)
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id` = `internal_node_A` (hash of an internal node, not a real txid)
- `tx_index` = `0`
- `merkle_proof` = `[internal_node_B]` (length 1, not the full depth-2 proof)
- `tx_block_blockhash` = the real block hash

`compute_root_from_merkle_proof(internal_node_A, 0, [internal_node_B])` computes `H(internal_node_A, internal_node_B)` = `root`. The comparison `== header.block_header.merkle_root` succeeds, and the function returns `true` — falsely asserting that `internal_node_A` is an included transaction.     #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
