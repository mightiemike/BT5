### Title
Division-by-Zero in `socializeSubaccount` When `openInterest == 0` Permanently Blocks Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when socializing a residual negative `vQuoteBalance`. No zero-guard exists. When a subaccount has `balance.amount == 0` and `balance.vQuoteBalance < 0` for a product whose `state.openInterest == 0`, the call reverts with `ERR_DIV_BY_ZERO`, permanently blocking `_finalizeSubaccount` and preventing bad-debt clearance.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after exhausting insurance coverage, the code computes:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest          // ← no zero-check
);
``` [1](#0-0) 

`MathSD21x18.div` explicitly enforces `require(y != 0, ERR_DIV_BY_ZERO)`: [2](#0-1) 

`state.openInterest` is the sum of absolute position sizes across all holders of that product. It reaches zero whenever every participant has closed their position. A subaccount that closed its position at a loss retains `balance.amount == 0` but `balance.vQuoteBalance < 0` — a fully valid on-chain state. `_updateBalance` called with zero deltas leaves `openInterest` unchanged: [3](#0-2) 

`_finalizeSubaccount` (called when `txn.productId == type(uint32).max`) explicitly requires `balance.amount == 0` for every perp before proceeding, then calls `perpEngine.socializeSubaccount` unconditionally: [4](#0-3) [5](#0-4) 

The revert propagates through `liquidateSubaccountImpl` → `liquidateSubaccount` (delegatecall), causing the entire liquidation transaction to revert. [6](#0-5) 

---

### Impact Explanation

- `_finalizeSubaccount` is permanently unreachable for the affected subaccount.
- The subaccount's negative `vQuoteBalance` (bad debt) can never be socialized or cleared.
- The protocol's accounting invariant — that any subaccount with negative net value must be finalizable — is broken.
- Bad debt accumulates on-chain with no recovery path, satisfying the Critical scope criterion of bad debt creation and locked liquidation flow.

---

### Likelihood Explanation

The precondition requires `state.openInterest == 0` for the specific product at the moment of finalization. This is reachable in:

1. **Low-liquidity or newly-launched perp markets** where the liquidatee was the last open participant.
2. **Adversarial setup**: a third party holding the only other open position closes it before the finalization transaction is submitted, driving `openInterest` to zero. This is a permissionless action requiring no special privileges.
3. **Market wind-down**: all participants exit a product, leaving only insolvent subaccounts with residual negative `vQuoteBalance`.

No admin access, sequencer compromise, or governance capture is required.

---

### Recommendation

Add a zero-guard before the division. If `openInterest == 0`, there are no other participants to socialize the loss against; the residual should be absorbed entirely by the insurance fund or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; zero out the residual
        // (insurance has already been applied above)
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

### Proof of Concept

State-machine test on an unmodified Hardhat setup:

1. Deploy the full protocol stack (Endpoint, Clearinghouse, ClearinghouseLiq, PerpEngine, SpotEngine).
2. Open a perp long position for subaccount A (product P): `updateBalance(P, A, +1e18, -100e18)` — sets `openInterest = 1e18`, `vQuoteBalance = -100e18`.
3. Close the position at a loss: `updateBalance(P, A, -1e18, +90e18)` — sets `balance.amount = 0`, `balance.vQuoteBalance = -10e18`, `openInterest = 0`.
4. Ensure no other subaccount holds an open position in product P (`openInterest` remains 0).
5. Drain subaccount A's spot USDC balance so it is under maintenance margin.
6. Call `endpoint.liquidateSubaccount({..., productId: type(uint32).max, ...})`.
7. Assert the transaction reverts with `"DBZ"`.

The revert occurs at `PerpEngine.sol:166` inside `socializeSubaccount`, called from `ClearinghouseLiq._finalizeSubaccount:386`, with `state.openInterest == 0` and `balance.vQuoteBalance == -10e18`. [7](#0-6)
