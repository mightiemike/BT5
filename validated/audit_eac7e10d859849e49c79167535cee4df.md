### Title
Inverted Fee Direction in `submitFastWithdrawal` Permanently Breaks Third-Party Relayer Incentive — (File: `core/contracts/BaseWithdrawPool.sol`)

### Summary
When a third-party relayer calls `submitFastWithdrawal` with `sendTo != msg.sender`, the protocol transfers the fast-withdrawal fee **from** the relayer's token balance **to** the pool, while sending the full withdrawal amount to the user. The relayer pays both gas and a token-denominated fee and receives nothing. The `FAST_WITHDRAWAL_FEE_RATE` is a hardcoded compile-time constant with no adjustment mechanism, making third-party fast-withdrawal relaying permanently and unconditionally economically irrational.

### Finding Description

`BaseWithdrawPool.submitFastWithdrawal` contains two fee-handling branches:

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);          // fee deducted from user's withdrawal
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // fee pulled FROM relayer
}
fees[productId] += fee;
handleWithdrawTransfer(token, sendTo, transferAmount);  // full amount sent to user
``` [1](#0-0) 

In the `sendTo != msg.sender` branch (the third-party relayer path), `safeTransferFrom` pulls the fee **from** `msg.sender` (the relayer) into the pool. The full `transferAmount` is then forwarded to `sendTo` (the user). The relayer receives zero compensation.

The fee is computed as:

```solidity
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;
int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
``` [2](#0-1) 

`FAST_WITHDRAWAL_FEE_RATE` is a Solidity `constant` hardcoded at `1_000_000_000_000_000` (0.1%): [3](#0-2) 

There is no admin-callable setter for this value anywhere in the codebase. It is immutably compiled into the bytecode.

For the standard `WithdrawCollateral` transaction type, `resolveFastWithdrawal` always resolves `sendTo` to the user's own address derived from the subaccount bytes:

```solidity
return (
    signedTx.tx.productId,
    address(uint160(bytes20(signedTx.tx.sender))),
    signedTx.tx.amount
);
``` [4](#0-3) 

This means any relayer calling `submitFastWithdrawal` for a `WithdrawCollateral` transaction will always land in the `sendTo != msg.sender` branch, paying the fee and receiving nothing.

### Impact Explanation

The fast-withdrawal system is designed to allow any caller to act as a relayer and provide users with immediate liquidity by front-running the sequencer settlement. The fee is the economic incentive for this service. Because the fee is transferred **from** the relayer rather than **to** the relayer, no rational actor will ever call `submitFastWithdrawal` on behalf of another user. The fast-withdrawal feature is permanently non-functional for third-party relayers. Users who cannot self-submit (e.g., those relying on meta-transaction infrastructure, smart-contract wallets without direct call capability, or users who need gas sponsorship) cannot access fast withdrawals. The `fees[productId]` mapping accumulates protocol revenue that was extracted from relayers who were deceived by the function's apparent purpose. [5](#0-4) 

### Likelihood Explanation

Certain. The fee direction is structurally inverted for every `WithdrawCollateral`-type fast withdrawal submitted by a third party. Any relayer who calls the function loses tokens proportional to the withdrawal amount (minimum 0.1%). The `FAST_WITHDRAWAL_FEE_RATE` constant cannot be changed without a contract upgrade, so there is no operational workaround. The condition `sendTo != msg.sender` is always true for a relayer acting on behalf of a user.

### Recommendation

1. **Fix fee direction**: In the `sendTo != msg.sender` branch, deduct the fee from `transferAmount` (as in the self-service branch) and transfer it to `msg.sender` (the relayer), not to the pool. This mirrors the incentive structure of the original bug's recommended fix.
2. **User-defined relayer fee**: Allow the signed withdrawal transaction to include a `relayerFeeX18` field (analogous to `WithdrawCollateralV2.feeX18`) so users can specify how much of their withdrawal they are willing to pay a relayer.
3. **Adjustable rate**: Replace the compile-time `FAST_WITHDRAWAL_FEE_RATE` constant with a governance-settable storage variable so the protocol can respond to changing gas conditions on the Ink network.

### Proof of Concept

1. User signs a `WithdrawCollateral` transaction: `sender = userSubaccount`, `amount = 1000 USDC`.
2. Relayer (address `R`) calls `submitFastWithdrawal(idx, transaction, signatures)`.
3. `resolveFastWithdrawal` returns `sendTo = address(uint160(bytes20(userSubaccount)))` — the user's address, not `R`.
4. `sendTo != msg.sender` → `safeTransferFrom(token, R, fee)` pulls `fee = 1 USDC` (0.1%) from `R`'s balance.
5. `fees[productId] += 1 USDC` — protocol collects the fee.
6. `handleWithdrawTransfer(token, user, 1000 USDC)` — user receives the full amount.
7. Relayer `R` has paid gas + 1 USDC and received nothing. No rational relayer will repeat this. [6](#0-5)

        uint64 idx,

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
