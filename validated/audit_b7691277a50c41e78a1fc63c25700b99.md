Audit Report

## Title
`SwapAllowlistExtension` checks router address as `sender` instead of original caller, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool receives the router contract as `msg.sender`, not the original user. A pool admin who allowlists the router to support router-mediated swaps for legitimate users inadvertently grants every caller of the router — including those explicitly not on the per-user allowlist — the ability to bypass the swap gate entirely.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `pool.swap(...)` directly without forwarding the original `msg.sender` as the `sender` identity — they only pass it as the payer in transient callback context, which the extension never reads: [4](#0-3) [5](#0-4) 

Therefore, the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`. Once the router is allowlisted, the check passes for every caller of the router regardless of their individual allowlist status. There is no mechanism in the extension to recover the original user's identity from `extensionData` or any other source.

## Impact Explanation

A curated pool using `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter` once the router is allowlisted. Unauthorized users can execute live swaps against oracle-priced liquidity, draining LP value at oracle-fair prices or trading in a pool designed to be closed. This is broken core pool functionality (the allowlist extension's sole purpose is defeated) with direct unauthorized-swap impact on LP principal.

## Likelihood Explanation

The pool admin has a legitimate, non-malicious reason to allowlist the router: without it, allowlisted users cannot use the router at all (no slippage protection, no multi-hop). The admin cannot simultaneously support router-mediated swaps for allowlisted users and block non-allowlisted users, because the router is a single shared address. Any pool that enables router support for its allowlisted users is immediately exposed. The attacker requires no privileged action — only a standard `MetricOmmSimpleRouter` call to the affected pool.

## Recommendation

The extension must identify the original economic actor, not the intermediary. Two approaches:

1. **Pass original caller through `extensionData`**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData`, and update `SwapAllowlistExtension.beforeSwap` to decode and check that value when `sender` is a known trusted router. This requires the extension to maintain a trusted-router registry.
2. **Transient-storage originator pattern**: The router writes the original caller into transient storage before calling `pool.swap`; the extension reads it back. This avoids trusting user-supplied `extensionData` and is consistent with the router's existing use of transient storage for callback context.

Either approach must ensure the extension verifies the identity of the economic actor, not the intermediary contract.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the intended sole swapper.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — to let Alice use `MetricOmmSimpleRouter`.
4. Charlie (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(recipient, ...)` — `msg.sender` of `pool.swap` is the router contract.
6. Pool calls `_beforeSwap(sender=router, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Charlie's swap executes successfully against the curated pool, bypassing the per-user allowlist entirely. [6](#0-5) [7](#0-6)
