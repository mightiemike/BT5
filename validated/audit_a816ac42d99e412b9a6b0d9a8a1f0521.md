### Title
Uncontrolled USDC Pull from `msg.sender` Based on On-Chain Balance in `replaceUsdcEWithUsdc` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is callable by any unprivileged user and pulls USDC from `msg.sender` in an amount determined entirely by the DDA's live on-chain USDCe balance — not by any user-specified parameter. This is a direct structural analog to the LibSwap pattern: a token pull whose magnitude is driven by a balance check rather than an explicit caller input, enabling a frontrunner to inflate the DDA's USDCe balance before the user's transaction executes and force the user to provide more USDC than intended.

---

### Finding Description

`replaceUsdcEWithUsdc` in `ContractOwner.sol` is a migration utility that swaps USDCe held in a `DirectDepositV1` (DDA) for USDC provided by `msg.sender`. The critical path is:

```solidity
// ContractOwner.sol lines 614-619
uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1); // ← amount set by on-chain state
if (balance > 0) {
    IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← pulls from caller
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
    IERC20Base(usdcE).safeTransfer(msg.sender, balance);
}
``` [1](#0-0) 

Three properties combine to create the vulnerability:

1. **No access control.** The function carries only a `block.chainid == 57073` guard — no `onlyOwner`, no check that `msg.sender` owns the target subaccount. Any address on Ink Chain can invoke it for any registered DDA. [2](#0-1) 

2. **Pull amount is not caller-specified.** The USDC amount pulled from `msg.sender` equals `IERC20Base(usdcE).balanceOf(directDepositV1)` at execution time. There is no `maxAmount` guard, no slippage parameter, and no snapshot of the balance taken at approval time. [3](#0-2) 

3. **DDA balance is externally writable.** Any address can send USDCe directly to a DDA address, increasing the balance that will be read by the function. The DDA's `receive()` and standard ERC20 `transfer` both accept arbitrary inbound tokens. [4](#0-3) 

This is structurally identical to the LibSwap root cause: `toDeposit = initialSendingAssetBalance < fromAmount ? fromAmount - initialSendingAssetBalance : 0` — a pull whose size is determined by a live balance check, not by the caller's explicit intent.

---

### Impact Explanation

A frontrunner observing a pending `replaceUsdcEWithUsdc` call can deposit additional USDCe into the target DDA before the user's transaction is mined. The user's transaction then pulls a larger USDC amount than the user approved for or intended to spend.

If USDCe is trading at a discount to USDC (a realistic scenario during a depeg event — the exact context this migration function was designed for), the user receives depegged USDCe in exchange for full-value USDC, suffering a direct, measurable financial loss proportional to the inflated amount and the depeg spread.

The corrupted asset delta: `msg.sender` loses `(inflated_balance - original_balance) × (1 - USDCe_price)` USDC in value.

---

### Likelihood Explanation

**Medium.** The preconditions are realistic and co-occur naturally:

- Users calling this migration function are doing so precisely because USDCe exists in their DDA — the depeg context that makes the loss meaningful is the same context that motivates the call.
- Infinite or large USDC approvals to `ContractOwner` are a standard DeFi UX pattern; users performing a migration are likely to approve a round number or `type(uint256).max`.
- Frontrunning on Ink Chain is feasible for any MEV-capable actor monitoring the mempool.
- The function is permissionlessly callable, so no privileged coordination is required.

---

### Recommendation

1. **Add a caller-specified `maxAmount` parameter** and revert if `balance > maxAmount`:
   ```solidity
   function replaceUsdcEWithUsdc(bytes32 subaccount, uint256 maxAmount) external {
       ...
       uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
       require(balance <= maxAmount, "balance exceeds maxAmount");
       ...
   }
   ```

2. **Add access control** — require `msg.sender` to be the address encoded in `subaccount` (i.e., `address(bytes20(subaccount)) == msg.sender`), or restrict to `onlyOwner`. The current design allows any caller to initiate a USDC pull from themselves against any DDA, which is unnecessary for a migration utility.

3. **Use `safeTransferFrom`** instead of bare `transferFrom` for the USDC pull to be consistent with the rest of the codebase's ERC20 handling. [5](#0-4) 

---

### Proof of Concept

1. User A's DDA (`directDepositV1Address[subaccount]`) holds 1,000 USDCe. USDCe is trading at $0.95 (depeg event).
2. User A approves `ContractOwner` to spend `type(uint256).max` USDC and submits `replaceUsdcEWithUsdc(subaccount)`.
3. Attacker observes the pending transaction in the mempool and frontruns it by transferring 9,000 USDCe directly to the DDA address.
4. User A's transaction executes. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` now returns 10,000.
5. `transferFrom(msg.sender, directDepositV1, 10_000e6)` pulls 10,000 USDC from User A.
6. User A receives 10,000 USDCe worth $9,500 at current market price.
7. **Net loss to User A: $500** (the 5% depeg spread on the 9,000 USDCe injected by the attacker).

The attacker's cost is the 9,000 USDCe they deposited; they recover it via the USDC now sitting in the DDA (which will be credited to the subaccount on the next `creditDeposit` call), netting the $450 spread as profit. [1](#0-0) [6](#0-5)
