### Title
`op_multiply` Undercharges Execution Cost via `limbs_for_int` Undercounting Intermediate Product Byte Length — (File: `src/more_ops.rs`)

### Summary

`op_multiply` tracks the running product's byte size using `limbs_for_int(&total)`, which calls `v.bits().div_ceil(8)`. For positive integers whose most-significant bit is set (e.g. 128, 255, 32768, 65535, …), `bits().div_ceil(8)` returns one fewer byte than the actual CLVM signed-big-endian atom representation requires (which needs a leading `0x00` byte to preserve the positive sign). Every subsequent multiplication step uses this undercount as `l0`, causing the cost formula `(l0 + l1) * MUL_LINEAR_COST_PER_BYTE + (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER` to charge less than the true work performed.

### Finding Description

In `src/more_ops.rs`, `op_multiply` initialises `l0` from `int_atom`, which returns the actual atom byte length. After each multiplication it updates `l0` with:

```rust
l0 = limbs_for_int(&total);
```

where:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`num_bigint::BigInt::bits()` returns the number of bits in the magnitude, excluding the sign bit. For a positive value whose high bit is set — e.g. `total = 128` (0x80) — `bits()` = 8, `div_ceil(8)` = 1. But the canonical CLVM atom for 128 is `[0x00, 0x80]` (2 bytes), because the high bit must be zero for a positive signed integer. The same off-by-one applies to every value in `[0x80, 0xFF]`, `[0x8000, 0xFFFF]`, `[0x800000, 0xFFFFFF]`, `[0x80000000, 0xFFFFFFFF]`.

By contrast, `len_for_value` (used correctly in `op_add`, `op_subtract`, `op_gr` fast paths) returns the correct CLVM byte length:

```rust
pub fn len_for_value(val: u32) -> usize {
    if val == 0 { 0 }
    else if val < 0x80 { 1 }
    else if val < 0x8000 { 2 }   // 0x80..0xFF → 2 bytes ✓
    ...
}
```

The mismatch is the direct analog of the external report: a conversion (`limbs_for_int`) produces a value smaller than the true quantity (`actual atom length`), and that smaller value is then used in the downstream cost computation for every subsequent multiplication step, instead of the correct larger value.

Additionally, the same undercount is used for the size-limit guard:

```rust
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
```

A product whose true CLVM atom length is 1025 bytes but whose `limbs_for_int` value is 1024 would pass this guard and proceed.

### Impact Explanation

**Impact: Medium.** The cost charged for a multi-step `op_multiply` is lower than the true computational work. An attacker who crafts a CLVM program whose intermediate products repeatedly land in the high-bit-set ranges accumulates an unbounded undercharge (one byte per step). If the Chia Python reference implementation charges the correct cost, clvm_rs will accept spend bundles that the reference rejects, producing a consensus divergence. Even within a single clvm_rs deployment, programs can consume more resources than the declared cost limit permits, enabling a resource-exhaustion vector against full nodes.

### Likelihood Explanation

**Likelihood: Medium.** Any attacker-controlled CLVM program invoking `*` with three or more arguments can trigger this path. Arranging for intermediate products to have their high bit set (e.g. multiplying by values that push the product into `[0x80, 0xFF]`, `[0x8000, 0xFFFF]`, etc.) is straightforward and requires no special privileges. The entry point is the public `run_program()` API, reachable from attacker-supplied CLVM bytes in a spend bundle.

### Recommendation

Replace `limbs_for_int` with a function that counts the actual CLVM signed-big-endian byte length, consistent with `len_for_value` and `new_number`. Concretely, add one byte whenever the magnitude's high bit is set:

```rust
fn limbs_for_int(v: &Number) -> usize {
    let bits = v.bits();
    if bits == 0 { return 0; }
    let bytes = bits.div_ceil(8) as usize;
    // If the top bit of the magnitude is set, a leading 0x00 byte is required
    // to keep the CLVM atom positive (signed big-endian).
    if bits % 8 == 0 { bytes + 1 } else { bytes }
}
```

This aligns `l0` with the actual atom size that `new_number` will produce, making the cost model consistent.
