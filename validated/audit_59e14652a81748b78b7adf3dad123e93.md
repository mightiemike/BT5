### Title
NLP Burn Fee Is Not Distributed to the Protocol — (`core/contracts/Clearinghouse.sol`)

### Summary
The `burnNlp` function computes a `burnFee` deducted from the user's payout, but this fee is entirely absorbed by the NLP system (`N_ACCOUNT` + NLP pools) and never credited to the protocol's fee accounting system (`X_ACCOUNT` or `insurance`). Every other fee-generating operation in the protocol distributes fees to the protocol, making this a direct analog to M-06.

### Finding Description
In `Clearinghouse.burnNlp()`, a `burnFee` is computed and deducted from the user's payout:

```solidity
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [1](#0-0) 

The user receives `quoteAmount` (reduced by `burnFee`). The NLP pools give up exactly `quoteAmount` in quote via `_applyNlpRebalance`, while `N_ACCOUNT` receives the full `nlpAmount` NLP tokens worth `quoteAmount + burnFee`. The NLP system is net positive by `burnFee`. [2](#0-1) 

No portion of `burnFee` is credited to:
- `X_ACCOUNT` — which receives trading fees via `dumpFees()` [3](#0-2) 
- `X_ACCOUNT` — which receives sequencer fees (withdrawal, liquidation, healthcheck) via `claimSequencerFees()` [4](#0-3) 
- `insurance` — which receives liquidation fees [5](#0-4) 

The `burnFee` is the only protocol-level fee that does not flow to the protocol's on-chain accounting.

The `HEALTHCHECK_FEE` charged at the `Endpoint` level for `BurnNlp` is separate and does reach `X_ACCOUNT` via `sequencerFee`. [6](#0-5)  The `burnFee` inside `Clearinghouse.burnNlp` is an additional, larger, percentage-based fee that is silently absorbed by the NLP system.

### Impact Explanation
The protocol's fee accounting system (`X_ACCOUNT`) does not receive any portion of the NLP burn fee. For large burns, this is material: a $1,000,000 burn generates a $1,000 `burnFee` (0.1%) that goes entirely to the NLP system. The protocol's revenue is permanently lower than it should be. The `burnFee` is at minimum `ONE` ($1 in 18-decimal fixed-point) and scales to 0.1% of the burn value. [7](#0-6) 

### Likelihood Explanation
Every NLP burn operation triggers this issue. Any user who calls `BurnNlp` through the sequencer pays a `burnFee` that goes entirely to the NLP system. The entry path is: user signs a `BurnNlp` transaction → sequencer submits via `submitTransactionsChecked()` → `Endpoint` → `EndpointTx.processTransactionImpl` → `Clearinghouse.burnNlp`. No privilege is required beyond holding NLP tokens. [8](#0-7) 

### Recommendation
Distribute a portion of the `burnFee` to the protocol's fee accounting system. For example, credit a protocol share to `X_ACCOUNT` and leave the remainder in the NLP system:

```solidity
int128 protocolFee = burnFee / 2; // example: 50% to protocol
spotEngine.updateBalance(QUOTE_PRODUCT_ID, X_ACCOUNT, protocolFee);
// quoteAmount already reduced by full burnFee; NLP pools absorb only (burnFee - protocolFee)
```

Alternatively, route the protocol share to `insurance` consistent with how liquidation fees are handled.

### Proof of Concept
1. User holds NLP tokens worth $1,000,000 at oracle price.
2. User submits `BurnNlp` transaction; sequencer processes it.
3. `burnFee = MathHelper.max(ONE, quoteAmount / 1000)` = $1,000 (0.1%).
4. `quoteAmount` is reduced to $999,000.
5. `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount)` — user loses NLP.
6. `spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount)` — N_ACCOUNT gains NLP worth $1,000,000.
7. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount)` — user gains $999,000.
8. `_applyNlpRebalance(...)` — NLP pools lose $999,000 in quote.
9. NLP system net: +$1,000,000 NLP − $999,000 quote = **+$1,000 (the burnFee)**.
10. `X_ACCOUNT` balance: unchanged. `insurance`: unchanged.
11. Compare: a $1,000,000 taker trade at 2 bps generates ~$200 in trading fees credited to `X_ACCOUNT` via `dumpFees()`. The burn fee is 5× larger and goes nowhere in the protocol's accounting. [9](#0-8)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
