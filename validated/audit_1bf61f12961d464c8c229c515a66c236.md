### Title
Stale `priceX18` Used in Cross-Product Health Checks Enables Over-Withdrawal and Liquidation Bypass - (`core/contracts/BaseEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`getHealth()` in `Clearinghouse.sol` iterates over every product a subaccount holds and computes health using the stored `risk.priceX18` value for each product. This stored price is only updated when an explicit `UpdatePrice` transaction is submitted by the sequencer. No health-sensitive operation — `withdrawCollateral`, `transferQuote`, `mintNlp`, `burnNlp`, or the liquidation eligibility check — triggers a price refresh for products other than the one directly being acted upon. A subaccount holding multiple products will have its health evaluated using a mix of current and arbitrarily stale prices, enabling over-withdrawal and liquidation bypass.

---

### Finding Description

In `BaseEngine._calculateProductHealth()`, health for each product is computed as:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [1](#0-0) 

`risk.priceX18` is read from `_risk().value[productId].priceX18`, a storage slot that is only written when `BaseEngine.updatePrice()` is called:

```solidity
function updatePrice(uint32 productId, int128 priceX18) external virtual {
    require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);
    _risk().value[productId].priceX18 = priceX18;
}
``` [2](#0-1) 

`updatePrice` is only triggered by an `UpdatePrice` transaction type routed through `EndpointTx.processTransactionImpl()`:

```solidity
} else if (txType == IEndpoint.TransactionType.UpdatePrice) {
    (uint32 productId, int128 newPriceX18) = clearinghouse.updatePrice(transaction);
``` [3](#0-2) 

This is a **separate, independent transaction** submitted by the sequencer. It is never called atomically before any health check.

The following operations call `getHealth()` without first refreshing prices for all products in the subaccount:

**`withdrawCollateral`** — updates only the withdrawn product's balance, then checks health across all products:

```solidity
spotEngine.updateBalance(productId, sender, amountRealized);
...
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [4](#0-3) 

**`mintNlp` / `burnNlp`** — explicitly updates only the NLP product price, then checks health across all products:

```solidity
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);
...
require(getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0, ERR_SUBACCT_HEALTH);
``` [5](#0-4) 

This is the most direct analog to the Venus pattern: one product's price is explicitly refreshed while all others remain stale.

**`liquidateSubaccountImpl`** — checks `isUnderMaintenance()` which calls `getHealth()` with no price refresh:

```solidity
require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
``` [6](#0-5) 

**`transferQuote`** — calls `_isAboveInitial()` → `getHealth()` with no price refresh:

```solidity
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
``` [7](#0-6) 

---

### Impact Explanation

**Over-withdrawal**: A user holds spot collateral (e.g., BTC) whose `priceX18` has not been updated after a market price drop. When `withdrawCollateral` is called for any product, `getHealth()` uses the stale (inflated) BTC price, making the subaccount appear healthier than it is. The user withdraws collateral they are not entitled to, leaving the protocol with an undercollateralized position.

**Liquidation bypass**: A subaccount whose collateral has fallen below the maintenance threshold cannot be liquidated if the stored `priceX18` for that collateral is stale (higher than actual). `isUnderMaintenance()` returns `false`, the liquidation reverts with `ERR_NOT_LIQUIDATABLE`, and the protocol accumulates bad debt.

**`mintNlp` / `burnNlp` health bypass**: Because only the NLP price is refreshed before the health check, a user with other collateral products whose prices have dropped can mint NLP or burn NLP while appearing healthier than they are, bypassing the INITIAL/MAINTENANCE health gate. [8](#0-7) 

---

### Likelihood Explanation

Price updates are submitted as discrete sequencer transactions (`UpdatePrice`). In any window between two `UpdatePrice` transactions for a given product — which can span multiple blocks or longer during sequencer delays, market inactivity, or high throughput — all health checks for subaccounts holding that product use stale prices. Slow-mode `WithdrawCollateral` transactions submitted directly by users are queued for 3 days and processed with whatever prices are stored at execution time, providing a concrete unprivileged entry path independent of sequencer ordering. The `mintNlp`/`burnNlp` path is particularly direct because the code itself demonstrates the pattern of updating one price and checking health across all products. [9](#0-8) 

---

### Recommendation

Before calling `getHealth()` in `withdrawCollateral`, `transferQuote`, `mintNlp`, `burnNlp`, and `liquidateSubaccountImpl`, iterate over all products held by the subaccount and assert or enforce that their stored `priceX18` values are within an acceptable staleness bound, or require the sequencer to batch an `UpdatePrice` for each held product atomically before the health-sensitive operation. At minimum, document and enforce an on-chain staleness check analogous to what Chainlink heartbeat checks provide, so that health calculations cannot silently use prices that are arbitrarily old.

---

### Proof of Concept

1. Subaccount `A` holds 1 BTC spot (product ID `P_BTC`) and has borrowed 40,000 USDC. The last `UpdatePrice` for `P_BTC` stored `priceX18 = 50,000e18`.

2. BTC market price drops to 38,000. No `UpdatePrice` transaction for `P_BTC` has been submitted yet.

3. User submits a slow-mode `WithdrawCollateral` for 1,000 USDC. After the 3-day delay, the sequencer executes it.

4. `withdrawCollateral` calls `spotEngine.updateBalance(QUOTE_PRODUCT_ID, sender, -1000e18)`, then calls `getHealth(sender, INITIAL)`.

5. `getHealth` → `getHealthContribution` → `_calculateProductHealth` for `P_BTC` reads `risk.priceX18 = 50,000e18` (stale). Health = `1 BTC × weight × 50,000 − 41,000 USDC > 0`. Check passes.

6. At actual price 38,000, health = `1 × weight × 38,000 − 41,000 < 0`. The withdrawal should have been rejected.

7. The protocol now holds an undercollateralized position. If BTC continues to fall, the insurance fund absorbs the loss. [10](#0-9) [11](#0-10)
