### Title
`run_with_cost` raises `IndexError` on malformed args bytes instead of `EvalError` — (`wheel/python/clvm_rs/program.py`, `wheel/src/api.rs`)

### Summary

`run_with_cost` catches `ValueError` from `run_serialized_chia_program` and unconditionally accesses `ve.args[1]`. However, `eval_to_py` in Rust constructs the `ValueError` with only **one** argument (the error string). When args bytes are malformed, `ve.args[1]` raises `IndexError`, not `EvalError`. Any caller that catches `EvalError` to handle CLVM failures will miss this exception entirely.

### Finding Description

**Step 1 — Attacker entry point:**

`Program.run_with_cost` in `wheel/python/clvm_rs/program.py`:

```python
try:
    cost, lazy_node = run_serialized_chia_program(
        prog_bytes, args_bytes, max_cost, flags
    )
    r = self.wrap(lazy_node)
except ValueError as ve:
    raise EvalError(ve.args[0], self.wrap(ve.args[1]))   # line 299
``` [1](#0-0) 

**Step 2 — Rust side raises a 1-arg ValueError:**

In `wheel/src/api.rs`, `eval_to_py` is:

```rust
fn eval_to_py(err: EvalErr) -> PyErr {
    pyo3::exceptions::PyValueError::new_err(err.to_string())
}
``` [2](#0-1) 

`PyValueError::new_err(string)` creates a Python `ValueError` whose `.args` is a **1-tuple**: `("bad encoding",)`.

**Step 3 — Malformed args bytes trigger the path:**

```rust
let args = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
``` [3](#0-2) 

If `args` bytes are truncated or otherwise malformed, `node_from_bytes` returns `Err(EvalErr::SerializationError)`, which `eval_to_py` converts to `ValueError("bad encoding")` — one arg only.

**Step 4 — IndexError fires:**

Back in Python, `ve.args` = `("bad encoding",)`. Accessing `ve.args[1]` raises `IndexError: tuple index out of range`. The `EvalError` is never raised.

**Step 5 — EvalError.__init__ is never reached:**

`EvalError.__init__` requires two arguments (`message`, `sexp`). The handler never gets there because `ve.args[1]` blows up first. [4](#0-3) 

### Impact Explanation

Any Python-based node or wallet that does:

```python
try:
    cost, result = program.run_with_cost(bad_args_bytes, max_cost)
except EvalError:
    reject_spend()
```

will **not** catch the `IndexError`. The exception propagates uncaught, crashing the handler. A direct Rust call returns `Err(EvalErr::SerializationError)` cleanly. This is a concrete exception-type divergence between the Python wheel and the Rust API for the same input.

The consensus split framing in the question is partially correct: a Python-based full node and a Rust-based full node will handle the same malformed spend differently — one crashes its handler (potentially causing the spend to be treated as an internal error rather than a clean rejection), the other returns a clean error. Whether this causes acceptance vs. rejection depends on how the upstream exception handler is written, but the exception-type mismatch is real and locally testable.

### Likelihood Explanation

The trigger is trivial: pass any truncated bytes as the `args` argument to `run_with_cost`. No special privileges or conditions are required. The bug is in production code on the critical execution path.

### Recommendation

Fix `run_with_cost` to handle the 1-arg `ValueError` case from deserialization failures separately from the 2-arg case from `adapt_response` (runtime CLVM errors). For example:

```python
except ValueError as ve:
    if len(ve.args) >= 2:
        raise EvalError(ve.args[0], self.wrap(ve.args[1]))
    else:
        raise EvalError(ve.args[0], self)
```

Or, better, change `eval_to_py` to always include a sentinel node so the Python side always receives two args.

### Proof of Concept

```python
from clvm_rs.program import Program

valid_program = bytes.fromhex("01")   # identity program: (q . 1)
malformed_args = b"\xff"              # truncated/malformed CLVM bytes

try:
    cost, result = Program.from_bytes(valid_program).run_with_cost(
        Program.from_bytes(b"\x80"),  # null
        max_cost=10_000_000
    )
except Exception as e:
    print(type(e).__name__, e)

# To trigger the bug directly:
from clvm_rs.clvm_rs import run_serialized_chia_program
try:
    run_serialized_chia_program(valid_program, malformed_args, 10_000_000, 0)
except ValueError as ve:
    print("ve.args =", ve.args)          # ('bad encoding',)  -- length 1
    print(ve.args[1])                    # IndexError here
```

The `ve.args` tuple has length 1, so `ve.args[1]` raises `IndexError` unconditionally whenever deserialization of either `program` or `args` bytes fails. [5](#0-4) [2](#0-1)
