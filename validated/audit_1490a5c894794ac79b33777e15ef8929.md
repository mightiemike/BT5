### Title
Missing Storage Gaps in Multi-Level Upgradeable Inheritance Chain Corrupts All User Balances on Upgrade - (`File: core/contracts/EndpointGated.sol`, `core/contracts/BaseEngine.sol`, `core/contracts/SpotEngineState.sol`, `core/contracts/PerpEngineState.sol`)

---

### Summary

The `SpotEngine` and `PerpEngine` contracts are deployed behind `TransparentUpgradeableProxy` instances and hold all user spot and perpetual balances. Their inheritance chains pass through multiple abstract parent contracts — `EndpointGated`, `BaseEngine`, `SpotEngineState`, and `PerpEngineState` — none of which define a `__gap` storage reserve. Adding any new state variable to any of these parent contracts during an upgrade will shift the storage layout of all child contracts, silently corrupting every user balance, product state, and configuration mapping stored in `SpotEngine` and `PerpEngine`.

---

### Finding Description

The Nado protocol deploys `SpotEngine` and `PerpEngine` as upgradeable contracts via `TransparentUpgradeableProxy`. Their full inheritance chains are:

```
SpotEngine
  └── SpotEngineState          ← no __gap (configs, states, balances, nlpLockedBalanceQueues)
        └── BaseEngine         ← no __gap (_clearinghouse, productIds, canApplyDeltas, nonZeroBalances)
              └── EndpointGated ← no __gap (address private endpoint)
                    └── OwnableUpgradeable  ← has OZ __gap (safe)

PerpEngine
  └── PerpEngineState          ← no __gap (states, balances, migrationFlag)
        └── BaseEngine         ← no __gap (same as above)
              └── EndpointGated ← no __gap (same as above)
                    └── OwnableUpgradeable  ← has OZ __gap (safe)
```

A global search for `__gap` across all `.sol` files returns **zero matches** — no storage gap is defined anywhere in the protocol.

The concrete storage layout of `SpotEngine` (in slot order after OZ's own gap) is:

| Slot offset | Variable | Contract |
|---|---|---|
| +0 | `endpoint` | `EndpointGated` |
| +1 | `_clearinghouse` | `BaseEngine` |
| +2 | `productIds` | `BaseEngine` |
| +3 | `canApplyDeltas` | `BaseEngine` |
| +4 | `nonZeroBalances` | `BaseEngine` |
| +5 | `configs` | `SpotEngineState` |
| +6 | `states` | `SpotEngineState` |
| +7 | `balances` | `SpotEngineState` |
| +8 | `nlpLockedBalanceQueues` | `SpotEngineState` |

If a developer adds even a single `address` or `bool` to `EndpointGated`, every slot from `_clearinghouse` onward shifts by one. The `balances` mapping — which stores every user's normalized spot balance — would now read from the slot previously occupied by `states`, returning completely wrong values. The same applies to `PerpEngine`'s `balances` and `states` mappings.

The same structural flaw exists in `ClearinghouseStorage` (inherited by `Clearinghouse`) and `EndpointStorage` (inherited by `Endpoint`), both of which are also upgradeable and carry no gaps.

---

### Impact Explanation

A storage corruption event on `SpotEngine` or `PerpEngine` would:

- Return wrong balances from `getBalance()` / `getStateAndBalance()`, causing `getHealthContribution()` in `Clearinghouse` to compute incorrect health scores for every subaccount.
- Allow previously healthy subaccounts to appear insolvent (triggering wrongful liquidations) or allow insolvent subaccounts to appear healthy (bypassing liquidation).
- Corrupt `totalDepositsNormalized` and `totalBorrowsNormalized` in `SpotEngineState`, breaking the interest rate model and `assertUtilization()` checks, potentially allowing unlimited withdrawals.
- Corrupt `PerpEngineState.states` (funding rates, open interest, `availableSettle`), causing incorrect PnL settlement for all perpetual positions.

All user funds held in the protocol — spot collateral, perpetual positions, and NLP balances — would be at risk of being permanently inaccessible or incorrectly valued.

---

### Likelihood Explanation

The likelihood is **medium**. The trigger requires a protocol upgrade that adds a new state variable to any of the gap-less parent contracts. This is a routine maintenance action (adding a new access control flag, a new configuration parameter, a new counter) that any developer could perform without realizing the storage layout consequence. The `ProxyManager` upgrade workflow does not include any automated storage layout compatibility check. The risk materializes the moment `migrateAll()` is called with a new implementation that has an expanded parent contract.

---

### Recommendation

Add a `uint256[N] private __gap;` at the end of each upgradeable parent contract's storage variable declarations, where `N` is chosen to reserve enough slots for future additions (typically 50):

```solidity
// EndpointGated.sol
abstract contract EndpointGated is OwnableUpgradeable, IEndpointGated {
    address private endpoint;
    uint256[49] private __gap; // reserve 49 more slots
}

// BaseEngine.sol
abstract contract BaseEngine is IProductEngine, EndpointGated {
    IClearinghouse internal _clearinghouse;
    uint32[] internal productIds;
    mapping(address => bool) internal canApplyDeltas;
    mapping(bytes32 => mapping(uint32 => uint256)) internal nonZeroBalances;
    bytes32 internal constant RISK_STORAGE = keccak256("nado.protocol.risk");
    uint256[46] private __gap;
}

// SpotEngineState.sol, PerpEngineState.sol, ClearinghouseStorage.sol, EndpointStorage.sol
// — same pattern applied to each
```

Also apply to `OffchainExchange` (which inherits `EndpointGated` and `EIP712Upgradeable` and defines ~15 state variables with no gap).

### Proof of Concept

**Step 1 — Confirm zero gaps exist:** [1](#0-0) 

`EndpointGated` declares `address private endpoint` with no `__gap` following it. [2](#0-1) 

`BaseEngine` declares four state variables with no `__gap`. [3](#0-2) 

`SpotEngineState` declares four mappings (including the critical `balances` mapping) with no `__gap`. [4](#0-3) 

`PerpEngineState` declares `states`, `balances`, and `migrationFlag` with no `__gap`.

**Step 2 — Confirm contracts are upgradeable:**

The wiki confirms `SpotEngine` and `PerpEngine` are registered as `TransparentUpgradeableProxy` instances in `ProxyManager` and are upgraded via `migrateAll()`. [5](#0-4) 

`SpotEngine` uses `initializer` pattern confirming proxy deployment.

**Step 3 — Trace the corrupted invariant:**

If `EndpointGated` gains one new `address` variable, the slot of `BaseEngine._clearinghouse` shifts from slot `X` to slot `X+1`. The slot previously holding `_clearinghouse` now holds the new variable (zero-initialized after upgrade). All calls to `_clearinghouse.registerProduct()`, `_clearinghouse.checkHealth()`, etc. would call `address(0)`, reverting or silently no-oping. Simultaneously, `SpotEngineState.balances` shifts one slot forward, reading from the slot that previously held `states`, returning corrupted balance data for every subaccount health check. [6](#0-5) 

`_initialize` stores `_clearinghouse` — this address would be read from a corrupted slot post-upgrade. [7](#0-6) 

`ClearinghouseStorage` similarly has no `__gap`, making `Clearinghouse` vulnerable to the same class of corruption.
