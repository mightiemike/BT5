### Title
Stale `node_map` Entries in `TreeCache` After `restore()` Enable Incorrect Back-Reference Paths When Allocator `NodePtr`s Are Reused — (`src/serde/tree_cache.rs`)

---

### Summary

`TreeCache.restore()` does not remove entries from `node_map` (or `node_entries`, `atom_lookup`, `pair_lookup`) that were inserted during the rolled-back `update()` call. When the `Allocator` is also restored to a prior checkpoint in the same session, newly allocated nodes receive the same `NodePtr` integer indices as the rolled-back nodes. On the next `update()` call, `TreeCache` silently reuses the stale `NodeEntry` objects — which carry the wrong SHA-1 tree hash, wrong serialized length, and wrong parent-chain — for the new, different nodes. `find_path()` then emits back-reference paths that point to the wrong subtree, producing a serialized byte string that deserializes to a different CLVM tree than the one that was serialized.

---

### Finding Description

`TreeCacheCheckpoint` captures only three fields:

```rust
pub struct TreeCacheCheckpoint {
    stack: Vec<u32>,
    serialized_nodes: BitSet,
    sentinel_entry: Option<u32>,
}
``` [1](#0-0) 

`TreeCache.restore()` uses this checkpoint to roll back `stack`, `serialized_nodes`, and the sentinel's `node_map` slot. It does **not** roll back `node_map`, `node_entries`, `atom_lookup`, or `pair_lookup`:

```rust
pub fn restore(&mut self, st: TreeCacheCheckpoint) {
    // ... adjusts on_stack counters ...
    self.stack = st.stack;
    // ...
    self.serialized_nodes = st.serialized_nodes;
    if let Some(sentinel_entry) = st.sentinel_entry {
        self.node_map.insert(self.sentinel_node.unwrap(), sentinel_entry);
    }
    // node_map entries added by the rolled-back update() are NOT removed
}
``` [2](#0-1) 

Meanwhile, `Allocator.restore_transparent_checkpoint()` truncates `atom_vec` and `pair_vec`, making the same integer indices available for reuse:

```rust
self.u8_vec.truncate(cp.u8s as usize);
self.pair_vec.truncate(cp.pairs as usize);
self.atom_vec.truncate(cp.atoms as usize);
``` [3](#0-2) 

The allocator test explicitly documents this reuse:

```rust
// since atom2 was removed, atom3 should actually be using that slot
assert_eq!(atom2, atom3);
``` [4](#0-3) 

When `update()` is called after the combined restore, it hits the early-exit path for any `NodePtr` that already has a stale `node_map` entry:

```rust
let e = match self.node_map.entry(node) {
    Entry::Occupied(e) => {
        // already traversed — reuse the old NodeEntry index
        let idx = *e.get();
        stack.push(idx);
        continue;
    }
    Entry::Vacant(e) => e,
};
``` [5](#0-4) 

The new node is silently aliased to the old `NodeEntry`, which holds the SHA-1 hash, serialized length, and parent chain of the **previous** (rolled-back) node. `find_path()` then navigates the parent chain of the old entry and returns a path that resolves to the wrong subtree in the deserialized output.

---

### Impact Explanation

The corrupted result is the back-reference byte sequence emitted by `find_path()`. A back-reference (`0xfe` prefix + path atom) that resolves to the wrong subtree causes `node_from_bytes_backrefs` to reconstruct a CLVM tree that differs from the one that was serialized. This is a **non-canonical serialization** defect: two nodes serializing the same logical spend bundle can produce different byte strings, or one node can produce a byte string that deserializes to a different program, breaking consensus equivalence across the network. [6](#0-5) 

---

### Likelihood Explanation

The trigger requires two simultaneous conditions:

1. `Serializer.restore()` is called to undo a partial `add()`.
2. The `Allocator` is also restored to a checkpoint taken before that `add()`, causing `NodePtr` index reuse.

The `Serializer` is explicitly designed for incremental, restartable serialization (the `restore()` API exists for this purpose), and the `Allocator` checkpoint/restore API is used in the same codebase for memory management during program evaluation. A caller that combines both — for example, a block-builder that speculatively serializes a spend and rolls back on failure — would trigger this bug with attacker-supplied CLVM content filling the reused `NodePtr` slots with different atoms or pairs. [7](#0-6) [8](#0-7) 

---

### Recommendation

`TreeCacheCheckpoint` must capture enough state to fully undo `update()`. The minimal fix is to record the lengths of `node_map`, `node_entries`, `atom_lookup`, and `pair_lookup` at checkpoint time and truncate/drain them on restore — mirroring exactly how the `Allocator` truncates its own vectors. Alternatively, add a monotonic generation counter to `node_map` entries and skip any entry whose generation exceeds the checkpoint's generation, analogous to adding the `_owner` key field recommended in the external report to scope the `handlerCan` mapping per owner. [1](#0-0) [2](#0-1)

### Proof of Concept

```rust
use clvmr::serde::Serializer;
use clvmr::Allocator;

let mut a = Allocator::new();

// Allocate a sentinel and a first atom BEFORE the checkpoint
let sentinel = a.new_pair(clvmr::NodePtr::NIL, clvmr::NodePtr::NIL).unwrap();
let alloc_cp = a.checkpoint();          // <-- checkpoint taken here

// Step 1: allocate atom_A with content b"AAAA" — gets NodePtr index N
let atom_a = a.new_atom(b"AAAA").unwrap();
let tree_a = a.new_pair(atom_a, sentinel).unwrap();

let mut ser = Serializer::new(Some(sentinel));
let (done, undo) = ser.add(&a, tree_a).unwrap();
assert!(!done);
// node_map now contains: atom_a -> NodeEntry{hash=H_A, len=6}

// Step 2: restore the serializer (rolls back stack/serialized_nodes but NOT node_map)
ser.restore(undo);

// Step 3: restore the allocator — atom_a's NodePtr index N is now free
a.restore_checkpoint(&alloc_cp);

// Step 4: allocate atom_B with DIFFERENT content b"BBBB" — reuses NodePtr index N
let atom_b = a.new_atom(b"BBBB").unwrap();
// atom_b == atom_a (same NodePtr integer)

let tree_b = a.new_pair(atom_b, sentinel).unwrap();

// Step 5: add tree_b — update() finds stale node_map[atom_b] -> NodeEntry{hash=H_A}
// find_path() will use H_A's parent chain, emitting a path that resolves to
// the wrong subtree on deserialization.
let (done, _) = ser.add(&a, tree_b).unwrap();
// serialized bytes contain a back-reference computed from H_A, not H_B
// => deserialization produces atom b"AAAA" where b"BBBB" is expected
```

The stale `node_map` entry for `atom_b` (carrying `NodeEntry` metadata for `atom_a`) is the direct analog of the stale `handlerCan` permissions that persist after a safe transfer: in both cases, a mapping keyed by an identifier (`NodePtr` / `safeHandler`) retains state from a previous owner/context that should have been cleared on the state transition. [9](#0-8) [5](#0-4)

pub struct TreeCache {
    /// caches extra metadata about a tree of nodes. The value is an index into
    /// the node_entries vector.
    node_map: HashMap<NodePtr, u32>,

    /// The metadata for all nodes in the tree. This is like a shadow tree
    /// structure to the NodePtr one. The most important difference is that
    /// identical nodes are merged, using the same NodeEntry, and additional
    /// metadata is kept, such as the tree hash.
    node_entries: Vec<NodeEntry>,

    /// maps tree-hashes to the index of the corresponding NodeEntry in the
    /// node_entries vector. For any given tree hash, we're only supposed to
    /// have a single NodeEntry. There may be multiple NodePtr referring to
    /// the same NodeEntry (if they are identical sub trees).
    atom_lookup: HashMap<Bytes20, u32, RandomState>,

    /// maps left + right child indices to the index of the pair with those
    /// children. This is the atom_lookup counterpart for pairs
    pair_lookup: HashMap<u64, u32>,

    /// When deserializing, we keep a stack of nodes we've parsed so far, this
    /// stack is maintaining that same state, since that's what back-references
    /// are pointing into.
    stack: Vec<u32>,

    /// This records which NodeEntries have been serialized so far. When we look
    /// for back-references, we can only pick nodes in this set. nodes with
    /// small serialized length are not inserted. This set is built and
    /// updated as we serialize, to ensure we only include nodes that *can* be
    /// referenced.
    serialized_nodes: BitSet,

    /// if the sentinel node is set, we can't compute the tree hashes or
    /// serialized length for this node nor any of its ancestors. When calling
    /// update(), the tree is assumed to be placed at the sentinel node in the
    /// previous call to update()
    pub sentinel_node: Option<NodePtr>,

    /// We compute hash-trees using SHA-1 in order to determine whether the
    /// trees are identical or not. To mitigate malicious SHA-1 hash collisions,
    /// we salt the hashes
    salt: [u8; 8],
}
