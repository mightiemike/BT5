### Title
`UnrestrictedSubmitBlocks` Role Cannot Bypass Pause on `submit_blocks` — (`contract/src/lib.rs`)

### Summary

The `UnrestrictedSubmitBlocks` role is documented as "Allows to use contract API even after contract is paused," but the `submit_blocks` function's `#[pause]` attribute has no `except(roles(Role::UnrestrictedSubmitBlocks))` clause. The role is only wired to the `#[trusted_relayer]` bypass, which governs a completely separate check. As a result, accounts granted `UnrestrictedSubmitBlocks` are silently blocked by the pause guard before the trusted-relayer check is ever reached, making the role's stated purpose unreachable.

### Finding Description

The contract defines the `UnrestrictedSubmitBlocks` role with an explicit intent: [1](#0-0) 

```rust
/// Allows to use contract API even after contract is paused
UnrestrictedSubmitBlocks,
```

The `#[trusted_relayer]` macro at the `impl` level lists this role as a `bypass_role`: [2](#0-1) 

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    ...
)]
```

However, `bypass_roles` in `#[trusted_relayer]` only bypasses the *trusted-relayer stake check*, not the *pause check*. The `submit_blocks` function carries both attributes: [3](#0-2) 

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
) -> PromiseOrValue<()> {
```

`#[pause]` is evaluated first and has **no** `except(roles(...))` clause. Contrast this with `run_mainchain_gc`, which correctly wires its bypass role into the pause attribute: [4](#0-3) 

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `UnrestrictedRunGC` role works because it appears inside `except(roles(...))`. `UnrestrictedSubmitBlocks` has no equivalent wiring in `submit_blocks`, so when the contract is paused the function panics unconditionally for every caller, including those holding `UnrestrictedSubmitBlocks`.

### Impact Explanation

When the contract is paused, accounts specifically granted `UnrestrictedSubmitBlocks` — intended to keep the light client synchronized during a pause — cannot submit block headers. The Bitcoin chain tip tracked by the contract falls behind the real chain. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive stale or incorrect confirmation counts, or will fail outright because the target block is no longer in the mainchain window. The emergency bypass mechanism the role was designed to provide is entirely non-functional. [5](#0-4) 

### Likelihood Explanation

The contract is designed to be pausable in production (it ships `PauseManager`, `DAO`, and `UnrestrictedSubmitBlocks` roles). Any pause event — whether for an upgrade, an incident, or a security response — triggers the broken invariant. The `RelayerManager` or `DAO` can grant `UnrestrictedSubmitBlocks` to a relayer account expecting it to keep working through a pause; that expectation is silently violated. No adversarial action is required beyond a normal pause operation.

### Recommendation

Add `except(roles(Role::UnrestrictedSubmitBlocks, Role::DAO))` to the `#[pause]` attribute on `submit_blocks`, mirroring the pattern already used for `run_mainchain_gc`:

```rust
#[payable]
#[pause(except(roles(Role::UnrestrictedSubmitBlocks, Role::DAO)))]
#[trusted_relayer]
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
) -> PromiseOrValue<()> {
```

This aligns the pause bypass with the role's documented purpose and with the existing `UnrestrictedRunGC` pattern.

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = true`.
2. Grant account `alice.near` the `UnrestrictedSubmitBlocks` role via `acl_grant_role`.
3. Call `pa_pause_feature` as `PauseManager` to pause `submit_blocks`.
4. Have `alice.near` call `submit_blocks` with a valid header batch.
5. Observe the transaction panics at the `#[pause]` guard — the `UnrestrictedSubmitBlocks` role provides no relief — even though `alice.near` is not subject to the trusted-relayer check.
6. Compare with `run_mainchain_gc`: grant `alice.near` the `UnrestrictedRunGC` role and observe that `run_mainchain_gc` succeeds while paused, confirming the `except(roles(...))` pattern works and its absence in `submit_blocks` is the root cause. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
