### Title
Protobuf P2P Sync Silently Downgrades `AllResources` V3 Transactions with Zero L2/L1-Data Gas to `L1Gas`, Producing Wrong Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter in the P2P sync path uses a zero-value check to distinguish pre-0.13.3 (`L1Gas`) from post-0.13.3 (`AllResources`) transactions. When a valid V3 transaction has `l2_gas = 0` and `l1_data_gas = 0`, it is silently deserialized as `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements for each variant (2 vs 3), the locally computed transaction hash diverges from the canonical hash, breaking hash validation and/or causing wrong execution behavior for any syncing node that receives such a transaction.

### Finding Description

**The vulnerable converter** in `crates/apollo_protobuf/src/converters/transaction.rs` decides the `ValidResourceBounds` variant purely by inspecting whether the deserialized `l2_gas` and `l1_data_gas` are both zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

This is the only signal used to distinguish a pre-0.13.3 transaction (which legitimately carries only L1 gas) from a post-0.13.3 transaction that happens to set zero L2 and L1-data gas bounds. There is no version field or explicit tag in the protobuf message to disambiguate them.

**The hash domain boundary** in `crates/starknet_api/src/transaction_hash.rs` produces structurally different preimages for the two variants:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements total
    }
});
``` [2](#0-1) 

- `L1Gas` path: `poseidon(tip, concat(L1_GAS, …), concat(L2_GAS, 0, 0))`
- `AllResources` path: `poseidon(tip, concat(L1_GAS, …), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0))`

Even when `l2_gas` and `l1_data_gas` are both zero, the `AllResources` hash includes the `L1_DATA_GAS` packed felt; the `L1Gas` hash does not. The two hashes are therefore always distinct for the same underlying field values.

**The codebase itself acknowledges the ambiguity.** The consensus round-trip test explicitly patches transactions to avoid the zero case before serializing:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
``` [3](#0-2) 

The consensus path avoids the problem because `RpcTransaction` stores `AllResourceBounds` (not `ValidResourceBounds`) and uses a separate converter (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs`) that always produces `AllResourceBounds` without any zero-value branching:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(...)?.try_into()?,
        })
    }
}
``` [4](#0-3) 

The P2P sync path for historical blocks uses `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (in `transaction.rs`) and has no such protection. A `FullTransaction` protobuf message carrying a V3 transaction with zero L2/L1-data gas bounds will be deserialized with the wrong variant, and the hash computed from the deserialized object will not match the canonical hash stored in the message.

Additionally, the wrong variant changes the `GasVectorComputationMode`:

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
    }
}
``` [5](#0-4) 

This means any re-execution of the stored transaction (e.g., block re-execution, state sync verification) will use `NoL2Gas` mode instead of `All`, producing wrong gas accounting and potentially wrong execution results.

### Impact Explanation

A syncing node that receives a valid V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` over P2P sync will:

1. Compute a wrong transaction hash (the `L1_DATA_GAS` element is absent from the poseidon preimage), causing hash validation against the stored hash to fail and the transaction to be rejected — or, if hash validation is skipped, the wrong variant is persisted.
2. Store `ValidResourceBounds::L1Gas` in the DB instead of `ValidResourceBounds::AllResources`, causing all subsequent re-executions to use `NoL2Gas` computation mode, producing wrong receipts, wrong fee accounting, and wrong state.

This matches: **Critical — Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input**, and **High — RPC execution, fee estimation, tracing, or simulation returns an authoritative-looking wrong value**.

### Likelihood Explanation

The gateway's stateless validator accepts a V3 transaction as long as at least one resource bound is non-zero:

```rust
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(StatelessTransactionValidatorError::ZeroResourceBounds { ... });
}
``` [6](#0-5) 

A transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` passes this check. Any user can craft and submit such a transaction. Once included in a block and propagated over P2P sync, every syncing node will mishandle it.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The protobuf schema should carry an explicit version or bounds-type tag, or the converter should always produce `AllResources` when all three resource-limit fields are present in the message (matching the behavior of the `AllResourceBounds` converter used in the consensus path). The TODO comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` indicates the zero-value fallback is already known to be a temporary workaround; it should be replaced with an explicit discriminant before 0.13.2 support is dropped. [7](#0-6) 

### Proof of Concept

1. Craft a valid V3 invoke transaction: `l1_gas = {max_amount: 1000, max_price_per_unit: 1}`, `l2_gas = {max_amount: 0, max_price_per_unit: 0}`, `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}`. The gateway accepts it (non-zero `l1_gas` satisfies the resource-bounds check).
2. The transaction is included in a block. Its canonical hash is computed with `AllResources`, including the `L1_DATA_GAS` packed felt: `poseidon(tip, concat(L1_GAS,…), concat(L2_GAS,0,0), concat(L1_DATA_GAS,0,0))`.
3. A syncing node receives the `FullTransaction` protobuf. The converter at line 431 of `transaction.rs` sees `l1_data_gas.is_zero() && l2_gas.is_zero()` → produces `ValidResourceBounds::L1Gas`.
4. The syncing node recomputes the hash with `L1Gas`: `poseidon(tip, concat(L1_GAS,…), concat(L2_GAS,0,0))` — the `L1_DATA_GAS` element is absent.
5. The two hashes differ. Hash validation fails and the transaction (and its block) is rejected by the syncing node, or the wrong variant is stored and all subsequent re-executions produce wrong results.
