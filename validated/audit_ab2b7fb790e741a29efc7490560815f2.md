### Title
Linked Signer Can Permanently Hijack Subaccount by Self-Overwriting via `LinkSigner` - (`core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` transaction in `processTransactionImpl` is processed with `allowLinkedSigner = true`, meaning the current linked signer can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. This gives the linked signer account-management authority that the user never intended to grant, and permanently transfers subaccount control to an attacker if the linked signer is ever compromised.

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer as the accepted alternate signer: [2](#0-1) 

`Verifier.validateSignature` accepts the recovered address if it equals either the subaccount owner address or the `linkedSigner`: [3](#0-2) 

Because the signature check passes for the linked signer, the subsequent write is executed unconditionally: [4](#0-3) 

This means the linked signer can overwrite `linkedSigners[sender]` to any arbitrary address — including an attacker-controlled one.

The slow-mode path for the same transaction type is correctly restricted: it uses `validateSender`, which requires `address(uint160(bytes20(txn.sender))) == sender`, i.e., only the actual subaccount owner (EOA) can submit it: [5](#0-4) 

This inconsistency between the sequencer path (`allowLinkedSigner = true`) and the slow-mode path (owner-only) confirms the sequencer path is the vulnerable one.

### Impact Explanation

Once the linked signer is changed to an attacker address, the attacker can sign any transaction that accepts `allowLinkedSigner = true`:

- `WithdrawCollateral` — drain all collateral from the subaccount [6](#0-5) 
- `TransferQuote` — transfer quote balance to any subaccount [7](#0-6) 
- `MintNlp` / `BurnNlp` — manipulate NLP positions [8](#0-7) 
- `LiquidateSubaccount` — initiate liquidations from the victim's subaccount [9](#0-8) 

The original owner cannot recover control through the sequencer path because the attacker controls the linked signer. Recovery is only possible via the slow-mode `LinkSigner` path, which requires the owner's EOA and a 3-day delay — during which the attacker can drain all assets.

### Likelihood Explanation

Linked signers are the standard mechanism for trading bots and automated strategies in Nado. These are hot wallets by design. A compromised hot wallet (e.g., via leaked API key, server breach, or malicious dependency) would normally allow only trading-level operations. The `allowLinkedSigner = true` on `LinkSigner` silently elevates that to full account takeover. The user has no on-chain indication that their linked signer holds this elevated power.

### Recommendation

Process the `LinkSigner` transaction with `allowLinkedSigner = false` in `processTransactionImpl`, consistent with the slow-mode path. Only the subaccount owner's EOA should be permitted to change the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // linked signer must NOT be able to change the linked signer
);
```

### Proof of Concept

1. Victim calls `submitSlowModeTransaction` with a `LinkSigner` transaction setting `linkedSigners[victim_subaccount] = hot_wallet_B`.
2. Attacker compromises `hot_wallet_B` (e.g., leaked server key for a trading bot).
3. Attacker uses `hot_wallet_B` to sign a new `LinkSigner` transaction off-chain: `{sender: victim_subaccount, signer: attacker_address, nonce: current_nonce}`.
4. Attacker submits this signed transaction to the sequencer's off-chain API.
5. Sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`. [10](#0-9) 
6. `validateSignedTx(..., allowLinkedSigner: true)` passes because `hot_wallet_B` is the current linked signer.
7. `linkedSigners[victim_subaccount] = attacker_address` is written. [4](#0-3) 
8. Attacker signs a `WithdrawCollateral` transaction from `victim_subaccount` and submits it to the sequencer, draining all collateral before the victim can react via slow mode.

```text
    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
```

```text
        uint64 idx,
        bytes[] calldata transactions,
        bytes32 e,
        bytes32 s,
        uint8 signerBitmask
    ) external {
        validateSubmissionIdx(idx);
        require(msg.sender == sequencer);
        // TODO: if one of these transactions fails this means the sequencer is in an error state
        // we should probably record this, and engage some sort of recovery mode

        bytes32 digest = keccak256(abi.encode(idx));
        for (uint256 i = 0; i < transactions.length; ++i) {
            digest = keccak256(abi.encodePacked(digest, transactions[i]));
        }
        verifier.requireValidSignature(digest, e, s, signerBitmask);

        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            nSubmissions += 1;
        }
    }
```
