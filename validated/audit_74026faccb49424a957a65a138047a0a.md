### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is still exposed as a live NEAR contract method despite being deprecated. It lacks the coinbase Merkle proof check that `verify_transaction_inclusion_v2` was introduced to provide. Any unprivileged NEAR caller can invoke the deprecated function directly via RPC, bypassing the 64-byte transaction Merkle proof forgery mitigation entirely.

### Finding Description

The contract exposes two transaction-inclusion verification methods:

- `verify_transaction_inclusion` (deprecated since v0.5.0) — verifies only that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == block.merkle_root`. [1](#0-0) 

- `verify_transaction_inclusion_v2` — additionally requires a coinbase Merkle proof at index 0 to match the same Merkle root, mitigating the 64-byte internal-node forgery attack. [2](#0-1) 

The `#[deprecated]` Rust attribute only generates a compiler warning for Rust callers. It does **not** remove the function from the NEAR ABI or prevent any external caller from invoking it via NEAR RPC. The function remains decorated with `#[pause]` (not `#[private]`), confirming it is intentionally public. [3](#0-2) 

The function's own `# Warning` comment acknowledges the flaw: *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [4](#0-3) 

The 64-byte attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) works as follows: every internal Merkle tree node is exactly 64 bytes (two concatenated 32-byte hashes). A 64-byte byte string can be interpreted as a valid-looking "transaction." An attacker picks any internal node `N` from a real block's Merkle tree, supplies it as `tx_id`, and provides the sibling path above `N` as `merkle_proof`. `compute_root_from_merkle_proof` will compute the correct Merkle root, and the function returns `true` — for a transaction that does not exist. [5](#0-4) 

The `ProofArgsV2 → ProofArgs` conversion used inside `verify_transaction_inclusion_v2` simply drops `coinbase_tx_id` and `coinbase_merkle_proof`, confirming that the coinbase check is entirely absent from the v1 path. [6](#0-5) 

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` (the deprecated v1) can be deceived into accepting a forged proof. The function returns `true` for a crafted `tx_id` that is actually an internal Merkle node, not a real transaction. This breaks the core security invariant of the light client: that a `true` result guarantees the transaction was actually included in a Bitcoin block. Downstream consumers (e.g., bridge contracts releasing funds, oracle contracts recording payment) would act on a false positive.

### Likelihood Explanation

The function is reachable by any unprivileged NEAR account via a standard NEAR RPC `call` with no special role or key required. The `#[deprecated]` marker is invisible to RPC callers. The 64-byte forgery technique is publicly documented and requires no mining power — only knowledge of a real block's Merkle tree structure, which is public data. Downstream integrators who read the contract ABI see two verification methods and may choose either one.

### Recommendation

Remove `verify_transaction_inclusion` from the public NEAR ABI entirely by adding `#[private]` to it, or delete the function body and replace it with `env::panic_str("use verify_transaction_inclusion_v2")`. The `#[deprecated]` attribute alone is insufficient because it has no effect on NEAR RPC accessibility.

### Proof of Concept

1. Obtain any confirmed Bitcoin block whose Merkle tree has depth ≥ 2 (i.e., more than one transaction).
2. Compute the Merkle tree. Pick any internal node `N` at depth 1 (the hash of two leaf transactions). `N` is 32 bytes but was computed from a 64-byte input.
3. Construct `ProofArgs`:
   - `tx_id` = `N`
   - `tx_block_blockhash` = the block's hash (already stored in the contract)
   - `tx_index` = the position of `N`'s left child in the leaf layer (so `tx_index` is even)
   - `merkle_proof` = the sibling path from `N`'s level up to the root (one element shorter than a leaf proof)
   - `confirmations` = 1
4. Call `verify_transaction_inclusion` via NEAR RPC with the above args.
5. `compute_root_from_merkle_proof(N, tx_index/2, proof_above_N)` correctly reconstructs the Merkle root. The function returns `true`. [7](#0-6) 

No mining, no privileged role, and no contract modification is required. The attack is fully executable by any NEAR account against the live deployed contract.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
