### Title
Bonus Reward Checkpoints Advance Without Payment When Pool Is Insufficient During Redelegation Slash - (File: `x/tieredrewards/keeper/slash.go`)

### Summary
In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent chain halt. However, by the time the error is returned, the position's bonus accounting checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) have already been advanced in the in-memory `pos` struct, and event reference counts have already been decremented in the store. The updated position is then persisted via `setPositionWithState`. This creates a permanent desynchronization: the position records that bonus was "claimed" up to the current block time, but the owner never received the corresponding bonus coins.

### Finding Description
`processEventsAndClaimBonus` in `x/tieredrewards/keeper/claim_rewards.go` performs checkpoint advancement **before** the pool sufficiency check:

1. Inside the event loop, `pos.UpdateLastEventSeq(entry.Seq)` advances `LastEventSeq` in memory and `k.decrementEventRefCount` mutates the store — both happen before the pool check.
2. After the loop, `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `LastBonusAccrual` and `pos.UpdateLastKnownBonded(bonded)` advances `LastKnownBonded` — again before the pool check.
3. Only then does `k.sufficientBonusPoolBalance(ctx, bonusCoins)` run; if it fails, the function returns `nil, err` with the in-memory `pos` already fully checkpointed. [1](#0-0) 

Back in `slashRedelegationPosition`, the `ErrInsufficientBonusPool` branch swallows the error and falls through: [2](#0-1) 

The code then calls `k.setPositionWithState(ctx, pos, ...)` with the already-checkpointed `pos`, permanently persisting `LastBonusAccrual = blockTime`, the advanced `LastEventSeq`, and the updated `LastKnownBonded` — even though zero bonus coins were transferred to the owner.

This is the exact analog to the external report: two values that must move in tandem (`pos.LastBonusAccrual` / `pos.LastEventSeq` representing "claimed up to" and the owner's actual received balance) are desynchronized in a specific code path.

For contrast, all user-driven paths (`ClaimTierRewards`, `AddToPosition`, `Undelegate`, `Redelegate`, `ClearPosition`) fail **atomically** on `ErrInsufficientBonusPool` — no state change, user retries after pool is replenished. [3](#0-2) 

### Impact Explanation
A delegator who holds a tier position, redelegates, and whose source validator is subsequently slashed while the bonus pool is insufficient will permanently lose all accrued bonus rewards for the period since their last claim. The checkpoints advance past the accrual window, so even after the pool is replenished, `processEventsAndClaimBonus` computes zero bonus for the already-consumed window. The user has no recourse. The corrupted value is `pos.LastBonusAccrual` (and `LastEventSeq`) in the `Positions` store, which diverges from the owner's actual received bonus balance. [4](#0-3) 

### Likelihood Explanation
Requires three concurrent conditions: (1) an active tier position with a pending redelegation, (2) the source validator is slashed during the redelegation's unbonding window, and (3) the bonus pool balance is below the accrued bonus at slash time. Validator slashing is a normal protocol event; bonus pool depletion is a realistic operational state. All three conditions can occur in production without any privileged action.

### Recommendation
In `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** persist the updated position checkpoints. Restore the pre-call checkpoint state on the in-memory `pos` before calling `setPositionWithState`, or restructure `processEventsAndClaimBonus` to advance checkpoints only after `SendCoinsFromModuleToAccount` succeeds. This mirrors the atomic-failure guarantee already provided on all user-driven paths. [5](#0-4) 

### Proof of Concept
1. User calls `MsgLockTier` → tier position created, bonus begins accruing.
2. User calls `MsgTierRedelegate` → redelegation entry created, `RedelegationMappings[unbondingId] = positionId` stored.
3. Bonus pool is depleted by other users claiming rewards (pool balance < accrued bonus for this position).
4. Source validator misbehaves → staking module calls `SlashRedelegation` → `BeforeRedelegationSlashed` hook fires → `slashRedelegationPosition` is called.
5. `processEventsAndClaimBonus` walks events, advances `LastEventSeq`, decrements event ref counts in store, sets `LastBonusAccrual = blockTime` — then returns `ErrInsufficientBonusPool`.
6. Error is swallowed; `setPositionWithState` persists the position with `LastBonusAccrual = blockTime`.
7. Pool is later replenished.
8. User calls `MsgClaimTierRewards` → `processEventsAndClaimBonus` computes `segmentStart = LastBonusAccrual = blockTime`, `segmentEnd = blockTime` → zero duration → zero bonus.
9. The entire accrued bonus for the pre-slash period is permanently lost. [6](#0-5) [7](#0-6)

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
