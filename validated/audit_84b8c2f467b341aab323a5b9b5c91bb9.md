### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing 64-Byte Merkle Proof Forgery Bypass — (`File: contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` was deprecated in v0.5.0 because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. The secure replacement, `verify_transaction_inclusion_v2`, adds a mandatory coinbase Merkle proof validation that closes this attack vector. However, the deprecated v1 function remains a live, publicly callable NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase proof check entirely and obtaining a `true` return value for a non-existent transaction.

### Finding Description
The contract documents the 64-byte Merkle proof forgery vulnerability in the NatSpec of `verify_transaction_inclusion`:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The fix was introduced in `verify_transaction_inclusion_v2`, which requires the caller to also supply a coinbase Merkle proof. The contract validates that the coinbase proof resolves to the block's `merkle_root` before proceeding: [2](#0-1) 

This coinbase check is the only guard against the forgery. However, `verify_transaction_inclusion` is still decorated with `#[pause]` — the NEAR SDK attribute that exposes it as a public contract method — and carries no access restriction beyond the global pause flag: [3](#0-2) 

The Rust `#[deprecated]` attribute is a **compiler hint only**. It produces a warning for Rust callers but has zero effect on the compiled WASM ABI. The method remains fully callable by any NEAR account via a standard function call transaction.

The `ProofArgs` struct accepted by v1 requires no coinbase fields: [4](#0-3) 

### Impact Explanation
An attacker who knows a real block hash on the canonical chain can:

1. Pick any 64-byte internal Merkle tree node hash `N` from that block's transaction tree.
2. Construct a valid Merkle sibling path from `N` up to the block's `merkle_root` (all intermediate hashes are computable from public block data).
3. Call `verify_transaction_inclusion` with `tx_id = N`, the real `tx_block_blockhash`, the crafted `merkle_proof`, and any `tx_index` consistent with the path.
4. The function computes `compute_root_from_merkle_proof(N, index, proof)` and compares it to `header.block_header.merkle_root`. The comparison succeeds and the function returns `true`. [5](#0-4) 

Any recipient NEAR contract that calls `verify_transaction_inclusion` to gate a financial action (e.g., a bridge unlock, a cross-chain settlement) will accept the forged proof as a valid Bitcoin transaction inclusion, enabling theft or unauthorized state transitions on the NEAR side.

### Likelihood Explanation
The entry path requires no privileges: a standard NEAR function-call transaction suffices. The block data needed to construct the forgery is entirely public. The deprecated function is not hidden behind any role check or allowlist. Likelihood is **high**.

### Recommendation
Remove the `#[pause]` attribute (and thus the public NEAR ABI exposure) from `verify_transaction_inclusion`, or add an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` as the first statement in the function body. The Rust `#[deprecated]` attribute alone provides no on-chain enforcement. The protection introduced in v2 must be the only reachable code path for external callers.

### Proof of Concept

```
# Attacker calls the deprecated v1 endpoint directly on the NEAR contract.
# No coinbase proof is required; the forgery succeeds.

near call <contract_id> verify_transaction_inclusion \
  --args-borsh <borsh_encoded_ProofArgs(
      tx_id         = <64-byte internal Merkle node hash N>,
      tx_block_blockhash = <real canonical block hash>,
      tx_index      = <index consistent with N's position>,
      merkle_proof  = <sibling path from N to merkle_root>,
      confirmations = 1
  )> \
  --accountId attacker.near

# Returns: true
# verify_transaction_inclusion_v2 would have rejected this because
# the coinbase proof check (lines 358-365) is never reached.
```

The analog to the reported issue is exact: the security guard (coinbase proof validation in v2 / bucket-active check in the original) exists only in the replacement path, while the deprecated path — still reachable by any unprivileged caller — skips it entirely.
