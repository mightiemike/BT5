### Title
`paid_fee_on_l1` Silently Dropped and Hardcoded to `Fee(1)` Across the `ConsensusTransaction → InternalConsensusTransaction` Serialization Boundary — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`ConsensusTransaction::L1Handler` carries only `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field. When a validator node converts a received `ConsensusTransaction::L1Handler` back into an `InternalConsensusTransaction::L1Handler` (which requires `executable_transaction::L1HandlerTransaction` with a `paid_fee_on_l1: Fee` field), the converter unconditionally injects `Fee(1)` as a placeholder. The actual fee paid on L1 — correctly captured by the L1 scraper and stored in the proposer's `ExecutableL1HandlerTransaction` — is permanently lost at this boundary. The blockifier's fee-sufficiency check then operates on the fabricated `Fee(1)` value rather than the real one.

---

### Finding Description

**Step 1 — The type boundary.**

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`:

```rust
// crates/starknet_api/src/consensus_transaction.rs
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // ← no paid_fee_on_l1
}
pub enum InternalConsensusTransaction {
    RpcTransaction(InternalRpcTransaction),
    L1Handler(executable_transaction::L1HandlerTransaction), // ← has paid_fee_on_l1
}
```

`transaction::L1HandlerTransaction` has no `paid_fee_on_l1` field. `executable_transaction::L1HandlerTransaction` does:

```rust
// crates/starknet_api/src/executable_transaction.rs:380-385
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,   // ← present here only
}
```

**Step 2 — The hardcoded injection.**

`convert_consensus_l1_handler_to_internal_l1_handler` in `TransactionConverter` bridges the gap by injecting `Fee(1)`:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

This is called unconditionally for every `ConsensusTransaction::L1Handler` received by a validator:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:197-200
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
```

**Step 3 — The same erasure in the RPC path.**

The RPC execution path (`starknet_estimateMessageFee`, `starknet_simulateTransactions`) independently hardcodes the same placeholder:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs:424-429
starknet_api::transaction::Transaction::L1Handler(value) => {
    // todo(yair): This is a temporary solution until we have a better way to get the l1 fee.
    let paid_fee_on_l1 = Fee(1);
    Ok(ExecutableTransactionInput::L1Handler(value, paid_fee_on_l1, false))
}
```

**Step 4 — The blockifier fee check operates on the fabricated value.**

The blockifier's `execute_raw` for `L1HandlerTransaction` reads `self.paid_fee_on_l1` and checks only that it is non-zero:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs:103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
```

Because `Fee(1) != Fee(0)`, the check always passes on the validator path regardless of what was actually paid on L1.

**Step 5 — The proposer path is unaffected; the divergence is structural.**

The proposer's `ProposeTransactionProvider` converts `ExecutableL1HandlerTransaction` (with the real fee from the L1 scraper) directly into `InternalConsensusTransaction::L1Handler` without going through `convert_consensus_l1_handler_to_internal_l1_handler`. The real `paid_fee_on_l1` is then serialized into `CentralL1HandlerTransaction.paid_fee_on_l1` for the cende recorder. Validators, however, always reconstruct with `Fee(1)`.

The protobuf serialization of `ConsensusTransaction::L1Handler` confirms the field is absent from the wire format:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:987-995
impl From<L1HandlerTransaction> for protobuf::L1HandlerV0 {
    fn from(value: L1HandlerTransaction) -> Self {
        Self {
            nonce: Some(value.nonce.0.into()),
            address: Some(value.contract_address.into()),
            entry_point_selector: Some(value.entry_point_selector.0.into()),
            calldata: value.calldata.0.iter().map(|calldata| (*calldata).into()).collect(),
            // paid_fee_on_l1 is not serialized
        }
    }
}
```

---

### Impact Explanation

The `paid_fee_on_l1` field is part of the executable `L1HandlerTransaction` payload. On the validator path, the converter binds the wrong value (`Fee(1)`) to this field instead of the actual fee paid on L1. This constitutes a conversion boundary that binds the wrong executable payload.

Concretely:

1. **Fee check bypass on validators**: The blockifier's `InsufficientFee` guard is permanently bypassed on every validator node because `Fee(1) != Fee(0)` always holds. If the L1 contract's own enforcement were ever relaxed or bypassed (e.g., via an upgrade or a bug), a zero-fee L1 handler transaction would be accepted by validators but rejected by the proposer, causing a consensus split.

2. **Wrong `paid_fee_on_l1` in the cende-recorded block**: If the validator path ever writes to the cende recorder (e.g., in a future topology change), the `CentralL1HandlerTransaction.paid_fee_on_l1` field would carry `Fee(1)` instead of the actual fee, producing a wrong on-chain record.

3. **Wrong RPC fee estimation**: `starknet_estimateMessageFee` and `starknet_simulateTransactions` use the same `Fee(1)` placeholder, meaning the fee-sufficiency check in simulated execution is always trivially satisfied regardless of the actual L1 fee, returning an authoritative-looking but incorrect result.

---

### Likelihood Explanation

Every L1 handler transaction processed by a validator node goes through `convert_consensus_l1_handler_to_internal_l1_handler`. The hardcoded `Fee(1)` is not gated by any version flag or configuration. The TODO comment (`// TODO(Gilad): Change this once we put real value in paid_fee_on_l1`) confirms this is a known placeholder that has not been resolved. The RPC path has an identical acknowledged placeholder. Both are in production code paths.

---

### Recommendation

1. Add `paid_fee_on_l1: Fee` to `ConsensusTransaction::L1Handler` (or to the protobuf `L1HandlerV0` message) so the field survives the consensus serialization round-trip.
2. Remove the hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` and use the deserialized value.
3. Fix the RPC path in `stored_txn_to_executable_txn` to read the actual `paid_fee_on_l1` from storage rather than hardcoding `Fee(1)`.
4. Add a round-trip test asserting that `paid_fee_on_l1` is preserved across `InternalConsensusTransaction → ConsensusTransaction → InternalConsensusTransaction`.

---

### Proof of Concept

```
1. L1 scraper observes sendMessageToL2 with ETH value = 1_000_000 wei.
   → ExecutableL1HandlerTransaction { paid_fee_on_l1: Fee(1_000_000), ... }

2. Proposer's ProposeTransactionProvider wraps it directly:
   → InternalConsensusTransaction::L1Handler { paid_fee_on_l1: Fee(1_000_000) }

3. Proposer broadcasts ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { ... })
   via protobuf (L1HandlerV0). paid_fee_on_l1 is NOT in the wire message.

4. Validator receives the protobuf, reconstructs ConsensusTransaction::L1Handler,
   calls convert_consensus_l1_handler_to_internal_l1_handler:
   → executable_transaction::L1HandlerTransaction::create(tx, &chain_id, Fee(1))
   → InternalConsensusTransaction::L1Handler { paid_fee_on_l1: Fee(1) }   ← WRONG

5. Blockifier executes on validator:
   paid_fee = Fee(1)
   if paid_fee == Fee(0) { ... }  // Fee(1) != Fee(0) → check passes unconditionally

6. Divergence: proposer holds paid_fee_on_l1 = Fee(1_000_000);
               validator holds paid_fee_on_l1 = Fee(1).
   CentralL1HandlerTransaction written by proposer carries Fee(1_000_000).
   Any future validator write would carry Fee(1).
