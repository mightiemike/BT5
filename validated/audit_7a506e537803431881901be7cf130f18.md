### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification entry points. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle-proof forgery vulnerability by requiring a coinbase proof. However, the original `verify_transaction_inclusion` (v1) is still a fully callable public NEAR method. Any unprivileged NEAR caller can invoke v1 directly, skipping the coinbase proof check entirely and re-opening the forgery path that v2 was designed to close.

---

### Finding Description

The contract documents the problem explicitly:

> *"Use `verify_transaction_inclusion_v2` instead."*
> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [1](#0-0) 

`verify_transaction_inclusion_v2` enforces the coinbase proof check before delegating to v1: [2](#0-1) 

The coinbase proof check in v2 is:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [3](#0-2) 

v1 has **no such check**. Its only guard is `require!(!args.merkle_proof.is_empty(), ...)`, which does not prevent an internal Merkle-tree node from being used as `tx_id`: [4](#0-3) 

The `#[deprecated]` attribute is a Rust **compiler warning only** — it imposes zero runtime restriction. The function is `pub` and decorated with `#[pause]` (not `#[private]`), so it remains reachable by any NEAR account as long as the contract is not paused. [5](#0-4) 

The Merkle proof computation in `merkle-tools` is purely positional — it accepts any 32-byte value as the starting leaf, including an internal tree node: [6](#0-5) 

This is the structural analog of M-18: the protocol defines two code paths, one of which (v1) skips the absolute security invariant (coinbase proof) that the other (v2) enforces. An attacker freely chooses the path that omits the check.

---

### Impact Explanation

A caller can supply a crafted `tx_id` equal to an internal Merkle-tree node and a matching `merkle_proof` such that `compute_root_from_merkle_proof` returns the block's real `merkle_root`. `verify_transaction_inclusion` returns `true` for a transaction that does not exist in the block. Any consumer contract or bridge that calls v1 to gate a cross-chain action (asset release, settlement, etc.) can be deceived into accepting a forged Bitcoin transaction proof.

---

### Likelihood Explanation

The entry point is unconditionally reachable by any unprivileged NEAR account. The 64-byte Merkle-proof forgery technique is publicly documented (referenced in the contract's own comments). No privileged role, leaked key, or social engineering is required — only knowledge of the contract ABI and a target block's Merkle tree structure.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI, or restrict it with `#[private]` so only the contract itself can call it (i.e., only via `verify_transaction_inclusion_v2`). If full removal is not possible for backward compatibility, add the coinbase proof check directly inside v1 as well, mirroring the guard already present in v2.

---

### Proof of Concept

1. Attacker selects a mainchain Bitcoin block already tracked by the light client (so `tx_block_blockhash` passes the mainchain check in v1).
2. Attacker computes an internal Merkle-tree node `N` at depth `d` in that block's transaction tree. `N` is 32 bytes and is a valid `H256`.
3. Attacker constructs a `merkle_proof` of length `d` such that `compute_root_from_merkle_proof(N, crafted_index, proof) == block.merkle_root`. This is feasible because the proof elements are the sibling hashes on the path from `N` to the root — all publicly available from the Bitcoin block.
4. Attacker calls `verify_transaction_inclusion` (v1) with `tx_id = N`, `tx_index = crafted_index`, `merkle_proof = proof`, `confirmations = 1`.
5. v1 passes the non-empty check, computes the root, finds it equal to `block.merkle_root`, and returns `true`.
6. Any consumer contract that called v1 now believes the forged transaction `N` is confirmed in the Bitcoin chain.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
