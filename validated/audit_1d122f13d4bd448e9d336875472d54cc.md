### Title
`DepositAllowlistExtension` Guards LP Owner Instead of Actual Depositor, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` validates the `owner` parameter (the LP position recipient) rather than the `sender` parameter (the actual caller of `addLiquidity`). Because `owner` is freely chosen by the caller, any unprivileged address can bypass the deposit allowlist by nominating an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

- `sender` = `msg.sender` of `addLiquidity` — the party actually calling the pool and supplying tokens via the swap callback
- `owner` = a caller-supplied parameter — the address that will hold the resulting LP shares [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension in order `(sender, owner, ...)`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and checks only the second (`owner`): [3](#0-2) 

The first parameter is unnamed (`address,`), confirming it is intentionally ignored. The guard therefore enforces: *"the LP position owner must be allowlisted"*, not *"the depositor must be allowlisted"* — the opposite of the stated contract invariant ("Gates `addLiquidity` by depositor address").

Contrast with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swapper): [4](#0-3) 

The structural analog to the external report is exact: just as `KoalaBearRangeCheck` validates the limbs but never asserts `value == highLimb * 2^24 + lowLimb`, `DepositAllowlistExtension` validates a sub-component of the call (`owner`) but never binds that check back to the entity the guard is supposed to constrain (`sender`/depositor).

---

### Impact Explanation

An address not on the allowlist can call `addLiquidity(owner = <any_allowlisted_address>, ...)`. The extension check passes because `allowedDepositor[pool][allowlisted_address]` is `true`. The pool proceeds to credit LP shares to the allowlisted address and pull tokens from the unauthorized caller via the swap callback. The pool admin's access-control intent — restricting which parties may inject liquidity — is fully circumvented by any unprivileged caller who knows one allowlisted address (a public mapping). This is an admin-boundary break: an admin-configured guard is bypassed by an unprivileged path, allowing unauthorized parties to alter pool bin balances, affect stop-loss watermarks, and dilute or manipulate the pool state in ways the admin explicitly intended to prevent.

---

### Likelihood Explanation

The allowlist is a public mapping (`allowedDepositor`), so any allowlisted address is trivially discoverable on-chain. The bypass requires only a single `addLiquidity` call with `owner` set to any allowlisted address. No special privileges, flash loans, or multi-step setup are needed. Any actor who is blocked by the allowlist can immediately circumvent it.

---

### Recommendation

Replace the `owner` check with a `sender` check (or check both, depending on intent):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`, which checks `sender` (the actual swapper) rather than `recipient`. [3](#0-2) 

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` configured; `allowAllDepositors[pool] = false`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Alice is the only allowlisted depositor.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner = alice, salt = 0, deltas = ..., ...)`.
4. Pool calls `_beforeAddLiquidity(msg.sender=Bob, owner=alice, ...)`.
5. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. `LiquidityLib.addLiquidity` credits LP shares to `alice`; tokens are pulled from Bob via the callback.
7. Bob has successfully deposited into the allowlisted pool despite being explicitly excluded. The allowlist is bypassed. [5](#0-4) [6](#0-5)
