Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Actual User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the allowlist evaluates the router's address rather than the actual user. If the pool admin allowlists the router (the only way to support router-mediated swaps), every user — including those explicitly excluded — can bypass the restriction by calling through the router.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

In `MetricOmmPool.swap`, `msg.sender` is captured and forwarded as `sender` to every `beforeSwap` hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient,
  ...
);
```

**`ExtensionCalling._beforeSwap` encodes that `sender` directly into the hook call:**

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

**`SwapAllowlistExtension.beforeSwap` gates on `sender` — the first argument:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the caller of the extension), and `sender` is whoever called `pool.swap()`.

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making itself `msg.sender`:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
  params.recipient,   // ← actual user set as recipient
  params.zeroForOne,
  ...
);
```

The router is `msg.sender` of `pool.swap()`, so `sender` seen by the extension is the router address, not the user. The allowlist therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Two broken outcomes:**
1. Allowlisted users cannot use the router — the router is not on the allowlist, so their router-mediated swaps revert even though they are individually permitted.
2. Non-allowlisted users bypass the allowlist — if the admin allowlists the router (the only fix for outcome 1), every user can call `MetricOmmSimpleRouter` and the hook passes unconditionally, rendering the allowlist inert.

No existing guard prevents this. The `onlyPool` modifier on the extension only verifies the caller is a registered pool; it does not validate which address is passed as `sender`.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for compliance, KYC, or access-control purposes loses its enforcement guarantee the moment the router is allowlisted. Any address — including those the pool admin explicitly excluded — can execute swaps against the pool's liquidity by routing through `MetricOmmSimpleRouter`. This is a direct, unprivileged bypass of an admin-configured access boundary: unauthorized parties can execute swaps the pool owner intended to prohibit, with direct fund-impacting consequences (unauthorized arbitrage, draining of pool value, or violation of regulatory access controls).

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard periphery entry point for end users. Any pool that deploys a swap allowlist and also wants to support router-mediated swaps (the common case) must allowlist the router, triggering the bypass. The attacker needs no special privileges — only the ability to call the public router. The precondition (router allowlisted) is the natural operational state for any pool that intends to serve normal users through the router.

## Recommendation
`SwapAllowlistExtension` must gate on the identity of the economic actor, not the immediate caller. Two viable approaches:

1. **Check `recipient` instead of `sender`** — the recipient is the address that receives output tokens and is set to the actual user by the router (`params.recipient`). Change line 37 of `SwapAllowlistExtension.sol` to check `recipient` instead of `sender`.

2. **Pass the originating user through `extensionData`** — the router encodes the real caller into `extensionData`; the extension decodes and checks that address. This requires a coordinated convention between the router and the extension.

The simplest correct fix is to change the allowlist lookup from `sender` to `recipient` in `SwapAllowlistExtension.beforeSwap`, since `recipient` is the address the router sets to the actual user.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; alice is NOT allowlisted; router IS allowlisted
// (admin had to allowlist the router so that bob, who IS allowlisted, can use it)

// Step 1: alice tries a direct swap — correctly blocked
vm.prank(alice);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(alice, true, 1000, 0, "", "");

// Step 2: alice routes through MetricOmmSimpleRouter — bypass succeeds
// sender seen by the hook = router (allowlisted), not alice (not allowlisted)
vm.prank(alice);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: alice,
        deadline: type(uint256).max,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// alice's swap executes despite being excluded from the allowlist
```

The call chain is: `alice → router.exactInputSingle → pool.swap(recipient=alice, ...)` where `msg.sender=router` → `_beforeSwap(sender=router, ...)` → `SwapAllowlistExtension.beforeSwap(sender=router, ...)` → `allowedSwapper[pool][router]` returns `true` → swap proceeds.
