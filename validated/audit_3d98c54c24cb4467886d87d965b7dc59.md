### Title
Difficulty Validation for Fork Blocks at Adjustment Boundaries Reads Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When a fork block is submitted at a difficulty adjustment boundary, `get_next_work_required` retrieves the boundary block's timestamp via `get_header_by_height`, which reads exclusively from the **mainchain** height-to-hash mapping. If the fork diverged before that boundary height, the mainchain's block at that height has a different timestamp than the fork's true ancestor. The resulting expected `bits` value is computed from the wrong block, allowing a relayer to submit fork blocks whose difficulty is validated against an incorrect target. A developer-placed `TODO` comment in `dogecoin.rs` explicitly acknowledges this ambiguity.

---

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It unconditionally resolves a height through `mainchain_height_to_header`, which only tracks the current canonical chain. There is no mechanism to walk the fork's own ancestry.

In `bitcoin.rs`, at every difficulty adjustment boundary, `get_next_work_required` calls:

```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [2](#0-1) 

If the fork diverged before `first_block_height`, the block returned is the **mainchain's** block at that height, not the fork's ancestor. The timestamp fed into `calculate_next_work_required` is therefore wrong, producing an incorrect expected `bits`.

The same pattern appears in `dogecoin.rs`, where the developer explicitly flagged the problem:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

The PoW check in `submit_block_header` runs **before** `submit_block_header_inner` and any reorg, so the mainchain mapping has not yet been updated when the difficulty is validated:

```rust
if !skip_pow_verification {
    self.check_target(&header, &prev_block_header);
    ...
}
...
self.submit_block_header_inner(current_header, &prev_block_header);
``` [4](#0-3) 

---

### Impact Explanation

**Concrete scenario (Bitcoin build):**

1. Mainchain exists at heights 0–4031 (two full 2016-block difficulty periods). The mainchain's block at height 2016 has timestamp `T_main`.
2. A fork diverges at height 2000. The fork's block at height 2016 has timestamp `T_fork ≠ T_main`.
3. A relayer submits the fork tip at height 4031. `check_pow` fires; `get_next_work_required` computes `first_block_height = 4031 − 2015 = 2016` and calls `get_header_by_height(2016)`, returning the **mainchain's** block with `T_main`.
4. If `T_main > T_fork`, the computed expected difficulty is **lower** (easier target) than the fork's true required difficulty. The relayer submits a fork block whose `bits` matches this easier target; the check passes.
5. Mining at the easier target requires less work per block. The attacker accumulates chainwork faster, eventually surpassing the mainchain's chainwork and triggering `reorg_chain`, corrupting the canonical chain pointer `mainchain_tip_blockhash` and all downstream `verify_transaction_inclusion` results.

For Dogecoin (post-height 145,000, per-block retargeting), the same flaw fires on **every** fork block submission, not just at 2016-block boundaries. [5](#0-4) 

---

### Likelihood Explanation

The `submit_blocks` entry point is gated by `#[trusted_relayer]`. [6](#0-5) 

The `RelayerManager` role description ("reject applications and update relayer config") implies a staking-based application process. If the default is permissive (applications accepted unless explicitly rejected), any NEAR account that stakes can become a relayer and trigger this path. Even under a strict allowlist, a compromised or malicious relayer — a realistic threat model for a bridge — can exploit this. For Dogecoin, the window is every block on a fork; for Bitcoin, every 2016 blocks (~2 weeks).

---

### Recommendation

Replace `get_header_by_height` (mainchain lookup) with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from the fork block being validated. This ensures the boundary block used in difficulty calculation is the fork's true ancestor, not the mainchain's block at the same height. The Dogecoin TODO comment already identifies this as the correct fix direction. [7](#0-6) 

---

### Proof of Concept

1. Deploy the contract (Bitcoin build, mainnet config, `skip_pow_verification = false`).
2. Submit a mainchain of 4032 blocks (two full difficulty periods). Record the mainchain's block at height 2016 with timestamp `T_main`.
3. Construct a fork starting at height 2000. Craft the fork's block at height 2016 with timestamp `T_fork < T_main` (earlier, so the true required difficulty for the fork's second period is **higher** than the mainchain's).
4. Compute the expected `bits` that the contract will accept: use `T_main` (mainchain boundary) in `calculate_next_work_required` — this yields a **lower** difficulty than the fork's true requirement.
5. Submit fork blocks 2001–4031 with `bits` set to the easier value from step 4. Each passes `check_pow` because the contract uses `T_main`.
6. Submit a fork block at height 4032 with chainwork exceeding the mainchain. `reorg_chain` fires; `mainchain_tip_blockhash` is updated to the fork tip.
7. `verify_transaction_inclusion` now resolves against the attacker-controlled fork chain. [8](#0-7) [9](#0-8)

```rust
fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}
