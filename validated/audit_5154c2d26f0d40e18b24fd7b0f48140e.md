### Title
Hardcoded `usize::MAX` Patch Override in Sierra Version Gate Silently Ignores Configured Upper Bound - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary
`StatelessTransactionValidator::validate_sierra_version` unconditionally overwrites the configured `max_sierra_version.patch` with `usize::MAX` before performing the upper-bound comparison. The operator-supplied patch ceiling is therefore never enforced: any patch level of the configured major.minor Sierra version is admitted, regardless of what the config says.

### Finding Description
In `validate_sierra_version` (lines 293–313), the code reads the configured `max_sierra_version`, then immediately replaces its `.patch` field with `usize::MAX`:

```rust
// Any patch version is valid. (i.e. when check version for upper bound, we ignore the Z
// part in a version X.Y.Z).
let mut max_sierra_version = self.config.max_sierra_version;
max_sierra_version.0.patch = usize::MAX;          // config value discarded

let sierra_version = VersionId::from_sierra_program(sierra_program)?;
if self.config.min_sierra_version <= sierra_version && sierra_version <= max_sierra_version {
    return Ok(());
}
``` [1](#0-0) 

The production config files set `max_sierra_version.patch = 0`: [2](#0-1) 

The canonical config schema serialises the Rust default (`usize::MAX` = `18446744073709551615`) as the patch value, confirming the override is always active: [3](#0-2) 

The `StatelessTransactionValidatorConfig::default()` also encodes `usize::MAX` directly: [4](#0-3) 

The result is a two-layer "unlimited" grant:
1. The Rust default already sets `patch = usize::MAX`.
2. Even if an operator overrides it to `0` (as the production JSON does), the runtime code re-sets it to `usize::MAX` before the comparison.

The config field `max_sierra_version.patch` is therefore a dead knob: it is documented, serialised, and deployed, but has zero effect on admission.

### Impact Explanation
The gateway's stateless validator is the first admission gate for `Declare` transactions. Because the patch ceiling is never enforced, a `Declare` carrying a Sierra program whose version is `(major=1, minor=9, patch=N)` for any `N > 0` passes the version check even when the operator has explicitly configured `patch = 0`. If a future Sierra 1.9.x release introduces a semantic change that the bundled compiler cannot handle correctly, the gateway will admit the transaction, the class manager will compile it, and the resulting CASM/class hash stored on-chain may be wrong. This maps to:

- **High – Mempool/gateway/RPC admission accepts invalid transactions before sequencing**: a declare with a Sierra version the operator intended to block is accepted.
- **Critical (conditional) – Wrong compiled class / CASM artifact / class hash selected for execution**: if the admitted Sierra version is incompatible with the compiler version in use, the compiled class stored and executed will be incorrect.

### Likelihood Explanation
The trigger is unprivileged: any user can submit a `Declare` transaction with a Sierra program whose version field encodes `patch > 0`. The production JSON configs set `patch = 0`, creating an operator expectation that such transactions are blocked. The code silently ignores that expectation on every invocation of `validate_sierra_version`. Likelihood is **medium** today (no Sierra 1.9.x patch releases exist yet) but becomes **high** the moment any such release appears.

### Recommendation
Remove the unconditional `max_sierra_version.0.patch = usize::MAX` override. If the intended policy is "any patch is acceptable", encode that as a sentinel value (e.g., `None`) in the config type and document it explicitly, rather than silently discarding the operator-supplied value at runtime. If patch-level gating is desired (as the production config implies), honour the configured value directly:

```rust
fn validate_sierra_version(&self, sierra_program: &[Felt]) -> ... {
    let sierra_version = VersionId::from_sierra_program(sierra_program)?;
    if self.config.min_sierra_version <= sierra_version
        && sierra_version <= self.config.max_sierra_version   // use config as-is
    {
        return Ok(());
    }
    Err(...)
}
```

If patch-agnostic matching is intentional, expose it as an explicit `ignore_patch_in_max_version: bool` config field so operators understand the semantics.

### Proof of Concept
1. Deploy the gateway with the production config (`max_sierra_version = {major:1, minor:9, patch:0}`).
2. Craft a `Declare` transaction whose Sierra program encodes version `1.9.999` in its first three felts.
3. Submit to the gateway's `/add_transaction` endpoint.
4. The `validate_sierra_version` call overwrites `patch` with `usize::MAX`, so `1.9.999 <= 1.9.usize::MAX` is `true`, and the transaction is admitted — contradicting the operator's configured `patch = 0` ceiling.

    fn validate_sierra_version(
        &self,
        sierra_program: &[Felt],
    ) -> StatelessTransactionValidatorResult<()> {
        // Any patch version is valid. (i.e. when check version for upper bound, we ignore the Z
