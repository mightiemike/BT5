Audit Report

## Title
Incomplete Pause Guard: `addLiquidity` and `removeLiquidity` Lack `whenNotPaused`, Allowing Fund-Impacting Operations on a Paused Pool — (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool` implements a three-level pause mechanism (`pauseLevel` 0/1/2) and a `whenNotPaused` modifier, but only applies it to `swap`. Both `addLiquidity` and `removeLiquidity` execute without any pause check, meaning all liquidity operations remain fully open when the pool is paused. This defeats the purpose of the pause mechanism and exposes LP principal to loss during security incidents.

## Finding Description
`MetricOmmPool` declares `pauseLevel` (L71-72) and the `whenNotPaused` modifier (L174-177), which calls `_checkNotPaused()` (L643-645) and reverts with `PoolPaused` if `pauseLevel != 0`. The `swap` function (L217-224) correctly applies `whenNotPaused`. However, `addLiquidity` (L182-196) and `removeLiquidity` (L199-212) only carry `nonReentrant` — no pause check is present. The `setPause` function (L455-461) is callable by the factory and can set `pauseLevel` to 1 or 2, making the pause mechanism reachable and functional. The gap is that two of the three fund-moving entry points are entirely unguarded.

**Exploit path for `removeLiquidity` race:**
1. Factory admin triggers `setPause(1)` in response to a security event (oracle compromise, accounting inconsistency, exploit in progress).
2. `swap` now reverts with `PoolPaused`.
3. Any LP calls `removeLiquidity(owner, salt, deltas, "")` — no `whenNotPaused` check, proceeds normally.
4. If pool accounting is already skewed at the time of the pause, early withdrawers receive a disproportionate share of bin tokens, leaving later LPs with a shortfall.

**Exploit path for `addLiquidity` inflation:**
1. Pool is paused (`pauseLevel = 1`) due to corrupted `BinState`.
2. Attacker calls `addLiquidity(attacker, salt, deltas, "", "")` — no `whenNotPaused` check, proceeds.
3. Shares are minted against the corrupted `BinState`, giving the attacker an inflated share count.
4. Protocol fixes accounting and calls `setPause(0)`.
5. Attacker calls `removeLiquidity`, redeeming inflated shares for more tokens than deposited, extracting value from honest LPs.

Existing guards (`nonReentrant`) do not address the pause bypass — they only prevent reentrancy within a single call.

## Impact Explanation
Direct loss of LP principal. The `removeLiquidity` race allows LPs to drain proportional shares from a paused pool whose accounting may be skewed, leaving remaining LPs with undercollateralized claims (pool insolvency). The `addLiquidity` path allows an attacker to mint inflated shares against a corrupted `BinState` and redeem them post-fix for a surplus, directly reducing honest LP claims. Both paths are reachable by any unprivileged address holding the relevant tokens. This matches the allowed impact: "Pool insolvency: balances fail to cover LP claims, owed fees, or owed LP assets" and "Broken core pool functionality causing loss of funds."

## Likelihood Explanation
The pause mechanism is explicitly designed for security incidents. Any time the factory admin or protocol triggers a pause (level 1 or 2), the incomplete guard is immediately exploitable by any LP or attacker. The `removeLiquidity` race is especially likely because sophisticated LPs monitor on-chain events and can react within the same block or the next block after a pause is set. No special role, privileged access, or complex setup is required beyond holding pool tokens.

## Recommendation
Add `whenNotPaused` to both `addLiquidity` and `removeLiquidity`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) { ... }

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (...) { ... }
```

If the protocol intentionally wants to allow LP exits during a pause, introduce a distinct `pauseLevel` value (e.g., level 1 = block swaps only, level 2 = block all fund-moving operations) and document the intent explicitly, rather than leaving the gap implicit.
