### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Allowlist Bypass and Breaking Router-Based Deposit Flows — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as gating `addLiquidity` by **depositor** address. Its `beforeAddLiquidity` hook silently discards the `sender` parameter (the actual caller who provides tokens via callback) and instead checks `owner` (the LP-position recipient). This is the inverse of what `SwapAllowlistExtension.beforeSwap` does, which correctly checks `sender`. The mismatch produces two fund-impacting failure modes: (1) an unauthorized depositor can bypass the allowlist by naming an allowlisted address as `owner`, and (2) a legitimately allowlisted router is blocked from depositing on behalf of any user whose address is not itself on the allowlist, rendering the pool's deposit path permanently broken.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook: [1](#0-0) 

```
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

- `sender` = `msg.sender` of `addLiquidity` — the address that will be called back to supply tokens.
- `owner` = the LP-position recipient, an arbitrary address supplied by the caller.

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely and gates on `owner`: [2](#0-1) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender`: [3](#0-2) 

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The structural inconsistency between the two sibling extensions confirms the wrong field is being read in `DepositAllowlistExtension`.

---

### Impact Explanation

**Failure mode A — Allowlist bypass (unauthorized tokens enter a permissioned pool):**

An unauthorized depositor `Bob` calls `pool.addLiquidity(owner = Alice, ...)` where `Alice` is on the allowlist. The extension evaluates `allowedDepositor[pool][Alice] == true` and permits the call. The pool's callback fires on `Bob` (the actual `msg.sender`), pulling Bob's tokens into the pool. The LP position is credited to `Alice`. Alice can then call `removeLiquidity` and extract Bob's tokens. The allowlist guard — intended to restrict which addresses may deposit — is fully bypassed; unauthorized funds enter the pool and are claimable by the allowlisted address.

**Failure mode B — Legitimate router-based deposits permanently blocked:**

A pool admin configures the allowlist to permit only a trusted router: `allowedDepositor[pool][router] = true`. A user calls `router.addLiquidity(pool, owner = user, ...)`. The router calls `pool.addLiquidity(owner = user, ...)`. The extension checks `allowedDepositor[pool][user]`, which is `false`, and reverts with `NotAllowedToDeposit`. Every deposit through the router fails. The pool's add-liquidity path is permanently broken for all users, making LP positions impossible to open and locking the pool into a state where existing LPs cannot be joined. [4](#0-3) 

---

### Likelihood Explanation

The `DepositAllowlistExtension` is a production extension shipped with the protocol. Any pool that (a) configures this extension and (b) routes deposits through a periphery contract (e.g., `MetricOmmPoolLiquidityAdder`) will hit failure mode B immediately upon the first deposit attempt. Failure mode A is reachable by any external caller who can observe the allowlist state and craft a call with a suitable `owner`. Both paths require no special privileges.

---

### Recommendation

Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `allowedDepositor` mapping documentation to clarify that the keyed address is the caller of `addLiquidity`, not the LP-position owner.

---

### Proof of Concept

**Setup:**
- Pool `P` is deployed with `DepositAllowlistExtension` as its `beforeAddLiquidity` hook.
- Pool admin calls `ext.setAllowedToDeposit(P, router, true)` — intending to allow only the trusted router to deposit.

**Failure mode B (broken deposit path):**
```
user → router.addLiquidity(pool=P, owner=user, ...)
  → P.addLiquidity(owner=user, ...)          // msg.sender = router
    → _beforeAddLiquidity(sender=router, owner=user, ...)
      → ext.beforeAddLiquidity(router, user, ...)
        → allowedDepositor[P][user] == false  // checks owner, not sender
        → revert NotAllowedToDeposit()        // router is allowlisted but irrelevant
```
Every deposit through the router reverts. The pool's liquidity-add path is permanently broken.

**Failure mode A (allowlist bypass):**
```
// Pool admin has set: allowedDepositor[P][Alice] = true
Bob → P.addLiquidity(owner=Alice, salt=X, deltas=..., ...)
  → _beforeAddLiquidity(sender=Bob, owner=Alice, ...)
    → ext.beforeAddLiquidity(Bob, Alice, ...)
      → allowedDepositor[P][Alice] == true    // checks owner, passes
  → callback fires on Bob, pulls Bob's tokens into P
  → LP position credited to Alice (key: owner=Alice, salt=X)
Alice → P.removeLiquidity(owner=Alice, salt=X, ...)
  → Alice receives Bob's tokens
``` [2](#0-1) [3](#0-2) [5](#0-4) [6](#0-5)
