Audit Report

## Title
Pool Admin Can Set Timelock to Zero, Eliminating LP Reaction Window and Atomically Disabling Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension` uses a per-pool timelock to give LPs a reaction window before the pool admin can weaken drawdown, decay, or watermark parameters. However, the timelock itself has no minimum enforced at initialization or on update, so it can be set to zero. With a zero timelock, `_afterTimelock` returns `block.timestamp`, `_requireElapsed` passes immediately, and the admin can propose and execute any parameter change atomically in a single transaction, collapsing the LP protection window entirely.

## Finding Description
`initialize` decodes and stores `timelock` without any floor validation:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no validation — zero accepted silently
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
```

`proposeOracleStopLossTimelock` and `executeOracleStopLossTimelock` similarly impose no minimum on `newTimelock` before writing it to storage (L78-94). Once `timelock == 0`:

- `_afterTimelock` returns `uint32(block.timestamp + 0)` = `block.timestamp` (L297-299).
- `_requireElapsed` checks `block.timestamp < block.timestamp`, which is always false — no revert (L301-303).

The admin can then call `proposeOracleStopLossDrawdown(pool, E6)` followed immediately by `executeOracleStopLossDrawdown(pool)` in the same transaction. This sets `drawdownE6 = E6`, making `floorMultiplier = E6 - E6 = 0` (L234). In `_applyWatermark`, the breach condition becomes `metric < (hwm * 0) / E6 = 0`, which is always false for any uint256 value (L334) — the stop-loss never triggers regardless of value loss. The protocol's own test confirms this path is reachable: `test_decayTimelockZeroExecutesImmediately` (L249-255) proposes and executes a decay change in the same block with no time warp.

## Impact Explanation
LPs deposit into a pool with `OracleValueStopLossExtension` expecting the stop-loss guard to protect their principal from oracle-price-driven value drain. The timelock is the sole mechanism bounding pool admin trust. Once the timelock is zero — either from initialization or after a single timelock-period wait — the admin can atomically set `drawdownE6 = E6`, making `floorMultiplier = 0` and permanently disabling the stop-loss. LP funds are then exposed to unlimited value loss with no protection and no exit window. This is a direct admin-boundary break: the pool admin bypasses the timelock mechanism that is explicitly designed to cap admin power, resulting in potential direct loss of LP principal.

## Likelihood Explanation
A pool can be deployed with `timelock = 0` from day one, making the stop-loss protection illusory for all LPs from the moment of deposit. For pools initialized with a non-zero timelock, the admin pays a one-time cost of waiting one timelock period to reduce it to zero via `proposeOracleStopLossTimelock(pool, 0)` + `executeOracleStopLossTimelock`, after which all subsequent parameter changes execute atomically with no further waiting. The NatDoc itself acknowledges the trust dependency ("monitor at least as often as the timelock or trust the pool admin"), confirming the timelock is the explicit trust boundary — and it can be eliminated.

## Recommendation
1. Enforce a minimum timelock at initialization in `initialize`: add `require(timelock >= MIN_TIMELOCK)` before storing the config.
2. Enforce the same floor in `executeOracleStopLossTimelock` before writing `oracleStopLossConfig[pool_].timelock = timelock`.
3. Optionally, prevent timelock reduction by requiring `newTimelock >= oracleStopLossConfig[pool_].timelock` in `proposeOracleStopLossTimelock`.

```solidity
uint32 constant MIN_TIMELOCK = 1 days;
require(timelock >= MIN_TIMELOCK, "TimelockTooShort");
```
