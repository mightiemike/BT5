### Title
Quote Token Price Permanently Hardcoded to $1 With No Oracle Update Path — (`SpotEngine.sol`, `Clearinghouse.sol`, `Endpoint.sol`)

---

### Summary

The Nado protocol hardcodes the price of `QUOTE_PRODUCT_ID` (the stablecoin quote token, e.g., USDC) to `ONE` (1e18 = $1) at initialization across three independent locations, with no on-chain mechanism to update it. Health calculations for all subaccounts use this stale price. If the quote token depegs, users can exploit the mispricing to over-borrow or drain value from the protocol, while other users suffer undetected undercollateralization.

---

### Finding Description

**Root Cause 1 — `SpotEngine.initialize` hardcodes `priceX18: ONE` for `QUOTE_PRODUCT_ID`:**

In `SpotEngine.sol` lines 32–38, the risk store for `QUOTE_PRODUCT_ID` is initialized with a fixed price of `ONE`:

```solidity
_risk().value[QUOTE_PRODUCT_ID] = RiskHelper.RiskStore({
    longWeightInitial: 1e9,
    shortWeightInitial: 1e9,
    longWeightMaintenance: 1e9,
    shortWeightMaintenance: 1e9,
    priceX18: ONE   // hardcoded $1
});
``` [1](#0-0) 

**Root Cause 2 — `Endpoint.initialize` hardcodes `priceX18[QUOTE_PRODUCT_ID] = ONE`:**

In `Endpoint.sol` line 54, the Endpoint's own price mapping for the quote product is also set to `ONE` at initialization and is never updated by any oracle path:

```solidity
priceX18[QUOTE_PRODUCT_ID] = ONE;
``` [2](#0-1) 

**Root Cause 3 — `Clearinghouse.checkMinDeposit` hardcodes `priceX18 = ONE` for `QUOTE_PRODUCT_ID`:**

In `Clearinghouse.sol` lines 709–711, the minimum deposit check explicitly skips the oracle call for the quote product and substitutes `ONE`:

```solidity
int128 priceX18 = ONE;
if (productId != QUOTE_PRODUCT_ID) {
    priceX18 = _getPriceX18(productId);
}
``` [3](#0-2) 

**Health Calculation Path:**

`BaseEngine._calculateProductHealth` computes health for every product including `QUOTE_PRODUCT_ID` using `risk.priceX18`:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [4](#0-3) 

Since `risk.priceX18` for `QUOTE_PRODUCT_ID` is permanently `ONE` (set in `SpotEngine.initialize` and never updated by any oracle), the health contribution of the quote token balance is always computed at $1 per unit regardless of actual market price.

**No Oracle Update Path for Quote Token:**

The `Clearinghouse.updatePrice` function dispatches price updates to the registered engine for a given `productId`. While `QUOTE_PRODUCT_ID` is registered in `SpotEngine`, the `checkMinDeposit` function ignores any such update and always uses `ONE`. More critically, the `Endpoint.priceX18[QUOTE_PRODUCT_ID]` is set once at initialization and the `updatePrice` sequencer transaction does not update `Endpoint.priceX18` — it only calls `engine.updatePrice`. The `getPriceX18` function on `Endpoint` returns `priceX18[productId]`, which for `QUOTE_PRODUCT_ID` is permanently `ONE`. [5](#0-4) 

---

### Impact Explanation

If the quote stablecoin (USDC) depegs to $0.95:

- A user deposits 1,000,000 USDC (actual value: $950,000).
- The protocol computes their health as if they deposited $1,000,000.
- The user can open leveraged positions or borrow against $1,000,000 of collateral value.
- When the position is settled or liquidated, the protocol absorbs the $50,000 shortfall.

If USDC trades at $1.01:

- A user depositing 1,000,000 USDC (actual value: $1,010,000) receives health credit for only $1,000,000.
- The user is underserved and may be incorrectly liquidated.

The corrupted state delta is: **subaccount health is overstated by `(1 - actualPrice) * quoteBalance * weight`** during a depeg, enabling undercollateralized borrowing and protocol insolvency. [6](#0-5) 

---

### Likelihood Explanation

Stablecoin depegs are historically common (USDC briefly depegged to ~$0.87 in March 2023; UST collapsed entirely). The Nado protocol explicitly uses a stablecoin as its quote token. Any depeg event — even a minor one — immediately creates an exploitable arbitrage window. The entry path is fully permissionless: any user can call `Endpoint.depositCollateral` followed by borrowing or trading operations. No privileged access is required. [7](#0-6) 

---

### Recommendation

- **Short Term:** Integrate a price oracle (e.g., Chainlink) for `QUOTE_PRODUCT_ID`. Remove the hardcoded `priceX18 = ONE` branch in `Clearinghouse.checkMinDeposit` and replace it with an oracle call. Ensure `Endpoint.priceX18[QUOTE_PRODUCT_ID]` is updated by the sequencer's `UpdatePrice` transaction path.
- **Long Term:** Enforce that all collateral prices — including the quote token — are sourced from a live oracle feed. Add a circuit breaker that halts deposits/borrows if the quote token price deviates beyond a configurable threshold from $1.

---

### Proof of Concept

1. USDC depegs to $0.95 on-chain.
2. Attacker calls `Endpoint.depositCollateral(subaccountName, QUOTE_PRODUCT_ID, 1_000_000e6)` depositing 1,000,000 USDC (cost: $950,000).
3. `Clearinghouse.depositCollateral` credits the subaccount with `1_000_000 * 1e12` (normalized to 18 decimals) of USDC balance.
4. `BaseEngine._calculateProductHealth` computes health as `1_000_000e18 * longWeightInitial * ONE` = $1,000,000 of health.
5. Attacker borrows or opens positions worth up to $1,000,000 against $950,000 of real collateral.
6. Protocol absorbs the $50,000 shortfall when positions are settled.

The hardcoded `priceX18: ONE` in `SpotEngine.initialize` is the necessary vulnerable step — it is the value read by `_calculateProductHealth` at line 174 of `BaseEngine.sol`, and it is never corrected by any oracle. [1](#0-0) [8](#0-7) [9](#0-8)
