Audit Report

## Title
Unbounded `blockDiff` silently disables velocity guard after pool inactivity — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` computes `allowedSq = maxChange² × (1 + blockDiff)` with no cap on `blockDiff`. After any sufficiently long idle period, `allowedSq` grows without bound, making it impossible for `actualSq > allowedSq` to trigger, regardless of how large the oracle price jump is. Additionally, the override drops the `onlyPool` modifier present in `BaseMetricExtension.beforeSwap`, allowing any address to call `beforeSwap` directly and write state under their own address as the pool key.

## Finding Description
In `PriceVelocityGuardExtension.beforeSwap` (L63–74), `blockDiff` is the raw difference between the current block and the block of the last swap, with no ceiling:

```solidity
uint256 blockDiff = block.number - prevBlock;                          // unbounded
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);
``` [1](#0-0) 

`lastMidPriceX64` is only updated inside `beforeSwap` (on each swap) or via the admin-only `setLastMidPrice`. There is no keeper or automatic refresh path. [2](#0-1) 

As `blockDiff` grows during idle periods, `allowedSq` grows without bound, making the guard trivially passable for any price jump. The reference price `lastMidPriceX64` becomes stale with no bound on staleness.

Additionally, `BaseMetricExtension.beforeSwap` declares `onlyPool`: [3](#0-2) 

But `PriceVelocityGuardExtension.beforeSwap` overrides it without the modifier, using `pool_ = msg.sender` as the storage key: [4](#0-3) 

Any address can call `beforeSwap` directly. While this does not corrupt a real pool's slot (the key is `msg.sender`), the authentication boundary is absent.

## Impact Explanation
The velocity guard is the sole on-chain mechanism capping oracle price movement speed for pools that configure this extension. When `blockDiff` is large enough, `allowedSq` exceeds any realistic `actualSq`, so a swap executing at an oracle price that has moved 100% or more from the stale reference passes the guard unchecked. LPs bear the full adverse-selection cost of the price jump with zero velocity-guard protection. This constitutes bad-price execution: an unclamped oracle-derived bid/ask quote reaches a pool swap, directly causing loss of LP principal through adverse selection.

Concrete arithmetic with `maxChangePerBlockE18 = 1e15` (0.1%/block) and `blockDiff = 1_000_000`:
```
allowedSq = (1e15)² × 1_000_001 = 1_000_001 × 1e30 ≈ 1.000001e36
actualSq  for 100% move = (1e18)² = 1e36
1e36 < 1.000001e36  →  guard passes
```

## Likelihood Explanation
No privileged access is required. Low-volume or newly deployed pools naturally go many blocks without a swap. Market downturns or protocol pauses create extended idle windows on any pool. Any user can be the first swapper after the idle period. The attacker does not need to manipulate the oracle — a genuine large price move occurring during the idle window is sufficient to cause LP losses through adverse selection.

## Recommendation
Cap `blockDiff` at a protocol-defined maximum (e.g., `MAX_BLOCK_DIFF = 7200` blocks ≈ 1 day on Ethereum). If `blockDiff` exceeds the cap, reset the anchor to the current price and skip the velocity check for that swap:

```solidity
uint256 blockDiff = block.number - prevBlock;
if (blockDiff > MAX_BLOCK_DIFF) {
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Additionally, restore the `onlyPool` modifier on `beforeSwap` to match the base class contract and prevent unauthenticated state writes.

## Proof of Concept
```
Setup:
  pool configured with PriceVelocityGuardExtension
  maxChangePerBlockE18 = 1e15  (0.1% per block)
  lastMidPriceX64 = P  (set at block N after last swap)

Attack:
  1. Wait until block N + 1_000_000 (no swaps occur).
  2. Oracle price moves from P to 2P (100% increase).
  3. Call pool.swap(...) — beforeSwap fires:
       blockDiff  = 1_000_000
       delta      = P
       changeE18  = (P × 1e18) / P = 1e18
       actualSq   = 1e36
       allowedSq  = (1e15)² × 1_000_001 = 1_000_001 × 1e30 > 1e36
       → guard passes, swap executes at 2P
  4. LPs suffer full adverse-selection loss on the 100% price move
     with zero velocity-guard protection.
```
