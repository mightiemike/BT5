### Title
Fee-on-Transfer Token Accounting Corruption in Deposit Flow — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential ERC20 transfers using the same nominal `amount` without measuring actual received balances at either hop. The slow-mode queue and `Clearinghouse.depositCollateral` then credit the full nominal `amount` to the subaccount. With a fee-on-transfer token, the Clearinghouse receives strictly less than `amount` but credits the full `amount`, creating an unbacked balance that can be withdrawn, draining other depositors.

---

### Finding Description

The deposit path in Nado is:

**Step 1 — `Endpoint.depositCollateralWithReferral`** calls `handleDepositTransfer` with the user-supplied `amount`, then enqueues a `SlowModeTx` encoding that same `amount`: [1](#0-0) 

**Step 2 — `EndpointStorage.handleDepositTransfer`** performs two transfers in sequence, both using the same `amount`:

```
safeTransferFrom(token, from, amount)          // user → Endpoint
safeTransferTo(token, address(clearinghouse), amount)  // Endpoint → Clearinghouse
``` [2](#0-1) 

With a fee-on-transfer token (e.g., USDT with fees enabled):
- Transfer 1: Endpoint receives `amount - fee1` (less than `amount`)
- Transfer 2: Endpoint attempts to forward the full `amount` to Clearinghouse, but only holds `amount - fee1`. This either reverts (if Endpoint has no prior balance of that token) or silently drains accumulated balances from prior depositors.
- Even if Transfer 2 succeeds, Clearinghouse receives `amount - fee2` (fee taken again on the second hop).

**Step 3 — `Clearinghouse.depositCollateral`** credits `txn.amount` (the original nominal value) to the subaccount's spot balance, with no check against actual received tokens: [3](#0-2) 

The `amountRealized` credited at line 205–207 is derived entirely from `txn.amount`, which was set at deposit time from the user-supplied parameter — never from a before/after balance measurement. [4](#0-3) 

The same nominal `amount` also flows through `chargeSlowModeFee` in `EndpointStorage`, which performs a `safeTransferFrom` without measuring actual receipt: [5](#0-4) 

---

### Impact Explanation

The Clearinghouse's on-chain token balance becomes less than the sum of all credited subaccount balances. Any user who deposited a fee-on-transfer token has an inflated spot balance. When they call `withdrawCollateral`, the Clearinghouse transfers the full credited `amount` out: [6](#0-5) 

This drains tokens that belong to other depositors, causing insolvency. The `assertUtilization` check at line 413 operates on the inflated accounting state and does not catch the real shortfall.

---

### Likelihood Explanation

USDT already has a fee switch in its contract that governance can enable. Any future spot product listed on Nado using such a token would be immediately exploitable by any unprivileged user calling `depositCollateral` or `depositCollateralWithReferral`. No special role or privileged access is required — the entry path is fully public. [7](#0-6) 

---

### Recommendation

In `handleDepositTransfer`, measure the actual balance received at each hop using before/after `balanceOf` snapshots, and propagate the real received amount (not the nominal `amount`) into the slow-mode queue entry and into `Clearinghouse.depositCollateral`. Specifically:

1. Record `balanceBefore = token.balanceOf(address(clearinghouse))` before the two transfers.
2. Execute both transfers.
3. Compute `actualReceived = token.balanceOf(address(clearinghouse)) - balanceBefore`.
4. Encode `actualReceived` (not `amount`) into the `DepositCollateral` slow-mode transaction.

Alternatively, restrict supported tokens to a whitelist that explicitly excludes fee-on-transfer tokens and enforce this at product listing time.

---

### Proof of Concept

1. A fee-on-transfer token (1% fee per transfer) is listed as a spot product.
2. Attacker calls `Endpoint.depositCollateral(subaccountName, productId, 1000e6)`.
3. Inside `handleDepositTransfer`:
   - Transfer 1: Endpoint receives `990e6` (1% fee deducted).
   - Transfer 2: Endpoint tries to send `1000e6` to Clearinghouse. If Endpoint has ≥ `10e6` of prior accumulated balance, this succeeds; Clearinghouse receives `990e6` (another 1% fee). Otherwise the call reverts.
4. The slow-mode queue records `amount = 1000e6`.
5. When the sequencer executes the slow-mode transaction, `Clearinghouse.depositCollateral` credits `amountRealized` derived from `1000e6` to the attacker's subaccount — despite the Clearinghouse holding only `980.1e6`.
6. Attacker calls `withdrawCollateral` for `1000e6`, receiving tokens that belong to other depositors. [2](#0-1) [8](#0-7) [9](#0-8)
