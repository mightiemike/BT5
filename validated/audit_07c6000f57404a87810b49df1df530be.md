### Title
`SwapAllowlistExtension.beforeSwap` checks the router address (proxy caller) instead of the actual user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on `sender`, which is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` = router address. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every non-allowlisted user can bypass the gate by routing through the same router. The extension checks the proxy (router) instead of the actual economic actor (user) — a direct structural analog to checking the iterator `t` instead of `tokens_[t]`.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, zeroForOne, ...)
)
``` [2](#0-1) 

`sender` is whatever address called `pool.swap()`. When a user routes through `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of the pool:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

So the extension receives `sender = router`, and the check becomes `allowedSwapper[pool][router]`.

**Attack path:**

1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC'd users.
2. Pool admin calls `setAllowedToSwap(pool, user1, true)` for each approved user.
3. Pool admin also calls `setAllowedToSwap(pool, router, true)` so that approved users can use the router (a natural usability requirement).
4. Attacker (not allowlisted) calls `router.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(...)` → pool calls `extension.beforeSwap(router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` = `true` → swap proceeds.
7. Attacker successfully swaps on a pool that was supposed to be restricted.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` with `msg.sender = router`. [4](#0-3) 

---

### Impact Explanation

**High.** A pool configured with `SwapAllowlistExtension` to enforce a curated or permissioned swap policy (e.g., KYC, institutional-only) is completely bypassed by any user who routes through the public `MetricOmmSimpleRouter`. The allowlist provides no protection once the router is allowlisted, which is the only way to let legitimate users use the router. Unauthorized users can execute swaps, draining pool liquidity at oracle-derived prices, directly harming LPs who deposited under the assumption that only approved counterparties could trade.

---

### Likelihood Explanation

**Medium.** The precondition — pool admin allowlisting the router — is a natural and expected operational step. Any pool that wants to support router-mediated swaps for its approved users must allowlist the router, at which point the bypass is unconditional and requires no special privileges or setup from the attacker. The attacker only needs to call the public router.

---

### Recommendation

The extension must check the actual economic actor, not the intermediary. Two viable approaches:

1. **Pass original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool, and the extension decodes and checks it. This requires a convention between router and extension.

2. **Check `recipient` instead of `sender`**: For swap allowlists, the recipient is the economic beneficiary. However, `recipient` can also be set to an arbitrary address, so this is not a complete fix.

3. **Preferred — dedicated router-aware allowlist**: The extension should accept a signed or router-attested user identity, or the pool should expose the original initiator address as a first-class parameter in the hook interface.

At minimum, document that allowlisting the router opens the gate to all users, and provide a separate `allowedRouter` flag distinct from `allowedSwapper` so admins understand the distinction.
