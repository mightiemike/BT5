### Title
Missing Declaration Existence Check for Cached Cairo0 Classes Allows Execution of Undeclared Classes After Reorg - (File: crates/blockifier/src/state/state_reader_and_contract_manager.rs)

### Summary

`StateReaderAndContractManager::get_compiled_from_class_manager` verifies that a cached Cairo1 class is still declared before returning it, but performs **no such check** for cached Cairo0 (`RunnableCompiledClass::V0`) classes. A Cairo0 class cached from a reverted block survives a chain reorganization and is returned to the blockifier without any declaration existence check, causing the sequencer to execute a class that is not declared in the canonical chain state.

### Finding Description

In `get_compiled_from_class_manager`, when a class is found in the `ContractClassManager` cache, the code branches on the class type:

```rust
// crates/blockifier/src/state/state_reader_and_contract_manager.rs, lines 73-83
match &runnable_class {
    RunnableCompiledClass::V0(_) => {}   // ‚Üê no existence check
    _ => {
        // The Cairo1 class is cached; verify it is declared,
        // since existence in the cache does not guarantee that
        // (it might contain a declared class from a reverted block, for example).
        if !self.state_reader.is_declared(class_hash)? {
            return Err(StateError::UndeclaredClassHash(class_hash));
        }
    }
}
```

The comment at lines 76‚Äì78 explicitly documents the reorg threat model, but the `V0` arm is an empty block ‚Äî the `is_declared` guard is never called for deprecated Cairo0 classes. [1](#0-0) 

The same omission exists in `ExecutionStateReader::get_compiled_class` in the class-manager path. For `ContractClass::V1`, `is_contract_class_declared` is called and the result gates the return. For `ContractClass::V0`, the class is returned unconditionally, with a developer TODO acknowledging the gap:

```rust
// crates/apollo_rpc_execution/src/state_reader.rs, lines 136-140
// TODO(shahak): Verify cairo0 as well after get_class_definition_block_number is
// fixed.
ContractClass::V0(deprecated_contract_class) => {
    Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
}
``` [2](#0-1) 

The `FetchCompiledClasses::is_declared` trait contract documents that Cairo0 classes always return `false`, meaning the check is intentionally asymmetric ‚Äî but the cache bypass for V0 is the bug, not the trait semantics. [3](#0-2) 

### Impact Explanation

After a chain reorganization, the `ContractClassManager` global cache retains Cairo0 classes that were declared in the reverted fork. When the sequencer processes a block on the canonical chain where those classes are **not** declared, any transaction that calls `get_compiled_class` for such a hash will receive the cached `RunnableCompiledClass::V0` object without any state check. The blockifier then executes the class as if it were declared, producing:

- A wrong execution result (transaction succeeds instead of reverting with `UndeclaredClassHash`)
- Wrong state diff, receipt, and events committed to the canonical chain
- Divergence between the sequencer's committed state and the provable OS state

This matches the impact category: **Critical ‚Äî Wrong state, receipt, event, or revert result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

The trigger requires a chain reorganization that reverts a block containing a Cairo0 `declare` transaction, followed by a new block on the canonical chain that does not re-declare the same class but contains a transaction that calls into it. Reorgs are a normal operational event in a live BFT network (e.g., during consensus view changes or network partitions). The `ContractClassManager` cache is process-global and is not invalidated on reorg. Cairo0 classes are still present in the Starknet ecosystem. The TODO comment in production code confirms the developers are aware the check is missing.

### Recommendation

Apply the same `is_declared` guard to the `RunnableCompiledClass::V0` branch in `get_compiled_from_class_manager`. Because `FetchCompiledClasses::is_declared` is defined to return `false` for Cairo0 classes, a separate `is_deprecated_declared` query (checking `deprecated_declared_classes_block_table`) is needed, or the trait must be extended. Concurrently, resolve the `TODO(shahak)` in `ExecutionStateReader::get_compiled_class` to add the `is_contract_class_declared` call for the `ContractClass::V0` arm once `get_class_definition_block_number` is fixed.

### Proof of Concept

1. Start the sequencer with a non-empty `ContractClassManager` cache.
2. Process block N that declares Cairo0 class `C` (hash `H`). The class is loaded into the cache via `set_and_compile`.
3. Trigger a reorg: revert block N. The cache is **not** flushed.
4. Process a replacement block N‚Ä≤ that does **not** declare `H`.
5. Submit an invoke transaction in block N‚Ä≤ whose execution calls `get_compiled_class(H)`.
6. `get_compiled_from_class_manager` finds `H` in the cache as `RunnableCompiledClass::V0`, hits the empty `V0` arm, skips `is_declared`, and returns the class.
7. The blockifier executes the class successfully; the transaction receipt shows success and emits events, whereas the correct result is `StateError::UndeclaredClassHash(H)`.

The `cairo_0_cached_scenario` test in `state_reader_and_contract_manager_test.rs` confirms that `is_declared_result: None` (i.e., the check is never called) is the expected behavior for a cached Cairo0 class ‚Äî demonstrating the missing guard is not tested for the post-reorg case. [4](#0-3)
