### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Enforcement - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` is marked `#[deprecated]` but remains a live, publicly callable NEAR contract method. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` enforces to prevent the 64-byte transaction Merkle forgery attack. This is a direct analog to the ContextModules bypass: just as ContextModules represent a code path that is never wrapped by LavaMoat's enforcement layer, `verify_transaction_inclusion` is a verification path that permanently skips the security control added in v2.

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase Merkle proof, verifying that the coinbase transaction sits at index 0 with the same proof depth: [1](#0-0) 

The old function, however, performs no such check. It only verifies that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root`: [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compile-time warning only**. It does not remove the function from the compiled WASM binary or prevent it from being dispatched at runtime. The function is `pub` and decorated with `#[pause]` (not permanently disabled), meaning it is active by default and callable by any NEAR account: [3](#0-2) 

The contract's own warning comment acknowledges the risk but incorrectly assumes it will be handled "at a higher level": [4](#0-3) 

No higher-level enforcement exists on-chain. Any caller that routes through `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2` receives no coinbase proof protection.

### Impact Explanation

An attacker can exploit the 64-byte transaction Merkle forgery attack through `verify_transaction_inclusion`:

1. Select any block already accepted into the contract's mainchain.
2. Identify an internal Merkle tree node (a 32-byte intermediate hash) within that block's transaction tree.
3. Treat that internal node as a `tx_id` and construct a valid Merkle proof path from it to the block's `merkle_root`.
4. Call `verify_transaction_inclusion` with this crafted `tx_id`, a valid `tx_block_blockhash`, and the forged proof.
5. The function computes `compute_root_from_merkle_proof(fake_tx_id, index, proof)`, which correctly reconstructs the real `merkle_root`, and returns `true`.

Any downstream contract or application that calls `verify_transaction_inclusion` to gate a financial action (e.g., cross-chain bridge release, SPV-based settlement) will accept a forged transaction inclusion proof as valid. The corrupted result is the boolean return value `true` for a transaction that does not exist. [5](#0-4) 

### Likelihood Explanation

Likelihood is **medium-high**. The function is publicly callable by any NEAR account with no privileged role required. Downstream integrators who read the contract ABI or documentation may use the older, simpler `verify_transaction_inclusion` (it takes fewer arguments). The 64-byte attack technique is well-documented and the necessary inputs (block Merkle tree structure) are publicly available from any Bitcoin node. The only prerequisite is that the target block is already in the contract's mainchain, which is the normal operating state.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely. Since it is already deprecated, it should be made `pub(crate)` or deleted. If backward compatibility is required, gate it behind a role check (e.g., `#[private]`) so it cannot be called by external accounts. The security invariant — that all SPV proofs must include a coinbase proof — must be enforced at the contract boundary, not delegated to callers.

### Proof of Concept

```javascript
// NEAR CLI call — no special role required
near call <contract_id> verify_transaction_inclusion \
  --args-borsh <borsh_encoded_ProofArgs_with_forged_tx_id> \
  --accountId attacker.near
```

Where `ProofArgs` is constructed as:
- `tx_id`: a 32-byte internal Merkle tree node from a real block (e.g., the hash of two real transaction IDs)
- `tx_block_blockhash`: the real block hash containing those transactions
- `tx_index`: the index corresponding to the internal node's position in the tree
- `merkle_proof`: the sibling hashes needed to reconstruct `merkle_root` from the internal node
- `confirmations`: any value ≤ `gc_threshold`

`compute_root_from_merkle_proof` will reconstruct the real `merkle_root` from the forged `tx_id`, and the function returns `true`. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
