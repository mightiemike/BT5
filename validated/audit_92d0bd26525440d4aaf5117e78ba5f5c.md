### Title
Deposit Accounting Corruption via Fee-on-Transfer Token: Clearinghouse Credited with Nominal Amount, Not Actual Received Amount — (`EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs a two-hop transfer (caller → Endpoint → Clearinghouse) and enqueues a slow-mode `DepositCollateral` transaction carrying the caller-supplied `amount`. No balance-before/after check is performed at either hop. For tokens that deduct a transfer fee from the recipient on `transfer` (but not on `transferFrom`), the Clearinghouse receives `amount - fee` while the slow-mode record — and the resulting on-chain credit — reflects the full `amount`. The Clearinghouse's real token balance becomes permanently less than the sum of all credited user balances, creating a solvency deficit that grows with every such deposit.

---

### Finding Description

**Entry point — `Endpoint.depositCollateral` / `depositCollateralWithReferral`**

Any unprivileged caller invokes `depositCollateral` (or `depositCollateralWithReferral`) with a user-chosen `amount`. [1](#0-0) 

`depositCollateralWithReferral` calls `handleDepositTransfer` with the raw `amount`, then immediately enqueues a slow-mode `DepositCollateral` record that also carries the raw `amount`: [2](#0-1) 

**Root cause — `EndpointStorage.handleDepositTransfer`**

```
safeTransferFrom(token, from, amount);               // hop 1: caller → Endpoint
safeTransferTo(token, address(clearinghouse), amount); // hop 2: Endpoint → Clearinghouse
``` [3](#0-2) 

Neither hop captures a balance snapshot. The function trusts that the Clearinghouse receives exactly `amount`.

`ERC20Helper.safeTransfer` (used by `safeTransferTo`) calls `token.transfer(to, amount)` and only checks the boolean return value — it does not verify the recipient's balance delta: [4](#0-3) 

**Accounting step — `Clearinghouse.depositCollateral`**

When the sequencer executes the slow-mode record, it credits `amountRealized = txn.amount * multiplier` — derived entirely from the enqueued nominal `amount`, with no reference to what the Clearinghouse actually holds: [5](#0-4) 

**The gap**

For a token that deducts a fee from the recipient on `transfer` (hop 2) but not on `transferFrom` (hop 1):

| Step | What happens |
|---|---|
| `safeTransferFrom(caller, endpoint, amount)` | Endpoint receives `amount` (no fee on `transferFrom`) |
| `safeTransferTo(endpoint, clearinghouse, amount)` | Clearinghouse receives `amount − fee` (fee on `transfer`) |
| Slow-mode record | Carries `amount` |
| `Clearinghouse.depositCollateral` | Credits user `amount × multiplier` |

The Clearinghouse's real balance is `amount − fee`; the user's internal balance is `amount`. The deficit is `fee` per deposit and compounds across all depositors.

---

### Impact Explanation

The Clearinghouse's on-chain token balance becomes structurally less than the aggregate of all user spot balances for the affected product. When users attempt to withdraw, the last withdrawers cannot be paid in full. The deficit is permanent and grows linearly with deposit volume. Because `spotEngine.updateBalance` is called with the inflated `amountRealized`, the corrupted balance also feeds into health checks, liquidation thresholds, and NLP rebalancing — all of which rely on the spot engine's balance state.

---

### Likelihood Explanation

Tokens that charge a fee on `transfer` but not on `transferFrom` exist in production (some governance and reward tokens implement this pattern to allow DEX router integrations while taxing direct transfers). Products are added by the protocol owner, but the owner may not audit every token's transfer implementation for asymmetric fee behavior. Once such a product is listed, any unprivileged user calling `depositCollateral` triggers the corruption with no further preconditions. The `DirectDepositV1.creditDeposit` path is an additional permissionless entry point that would trigger the same corruption for any DDA holding such a token. [6](#0-5) 

---

### Recommendation

Replace the two-hop fixed-amount pattern in `handleDepositTransfer` with a balance-snapshot approach on the Clearinghouse side:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) into the slow-mode record
}
```

The slow-mode `DepositCollateral` record and the resulting `Clearinghouse.depositCollateral` credit must use `actualReceived`, not the caller-supplied `amount`. Alternatively, restrict listed tokens to those that do not implement transfer fees.

---

### Proof of Concept

1. Protocol lists a product whose collateral token charges a 1 % fee on `transfer` but not on `transferFrom`.
2. Alice calls `Endpoint.depositCollateral(subaccountName, productId, 1000e6)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(Alice, Endpoint, 1000e6)` — Endpoint receives `1000e6` (no fee).
   - `safeTransferTo(Endpoint, Clearinghouse, 1000e6)` — Clearinghouse receives `990e6` (1 % fee deducted).
4. Slow-mode record is enqueued with `amount = 1000e6`.
5. Sequencer processes the record; `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(productId, Alice, 1000e6 × multiplier)`.
6. Alice's internal balance reflects `1000e6`; Clearinghouse holds only `990e6`.
7. Alice repeats 100 times: Clearinghouse deficit grows to `1000e6` (100 × 10e6), while Alice's credited balance is `100 000e6` against `99 000e6` of real tokens.
8. Alice withdraws `100 000e6`; the last `1000e6` cannot be paid, and other users' withdrawals fail. [3](#0-2) [2](#0-1) [7](#0-6)
