Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any caller to bypass per-user allowlist restrictions — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the allowlist entry consulted is `allowedSwapper[pool][router]`. A pool admin who allowlists the router to enable allowlisted users to reach the pool through the periphery simultaneously opens the pool to every unprivileged caller of the router, completely defeating the access-control guard.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. When the router is the direct caller, `msg.sender` is the router address. [1](#0-0) 

**`SwapAllowlistExtension.beforeSwap` checks the router, not the end user:**

The extension receives `sender = router` and evaluates `allowedSwapper[pool][router]`. If the admin has allowlisted the router, this check passes for every caller of the router regardless of whether they are individually allowlisted. [2](#0-1) 

**`MetricOmmSimpleRouter.exactInputSingle` does not forward the original `msg.sender`:**

The router calls `pool.swap(params.recipient, ...)` as a regular external call. The original end-user `msg.sender` is stored only in transient storage for the payment callback (`_setNextCallbackContext`) and is never passed to the pool's `swap` call as the economically relevant actor. [3](#0-2) 

**`multicall` uses `delegatecall` but does not change the outcome:**

`multicall` preserves `msg.sender` within the router's own execution frame via `delegatecall`, but the subsequent `pool.swap(...)` call is still a regular external call originating from the router contract, so the pool always sees `msg.sender = router`. [4](#0-3) 

**Exploit flow:**
```
attacker → router.exactInputSingle(pool=P, ...)
              └─ P.swap(recipient, ...)          // msg.sender = router
                    └─ _beforeSwap(sender=router, ...)
                          └─ SwapAllowlistExtension.beforeSwap(sender=router)
                                check: allowedSwapper[P][router] == true  ✓
                                // attacker's swap settles in full
```

No existing guard prevents this. The `allowedSwapper` mapping is keyed by the direct caller of `pool.swap()`, not by the economic actor. The `isAllowedToSwap` view function and `setAllowedToSwap` setter operate on the same flawed mapping. [5](#0-4) 

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties) and then allowlists the router so those addresses can use the periphery has, in effect, opened the pool to all users. Any unprivileged actor can call `router.exactInputSingle(...)` and receive pool output tokens. The attacker receives output tokens and the pool receives input tokens — a complete, fund-impacting bypass of the access-control guard. This constitutes broken core pool functionality causing loss of funds (unauthorized access to pool liquidity) and an admin-boundary break where the pool admin's intended restriction is rendered ineffective by a reachable unprivileged path.

## Likelihood Explanation
The `SwapAllowlistExtension` NatSpec states it "Gates `swap` by swapper address, per pool." A pool admin naturally interprets "swapper" as the end user. To let allowlisted users reach the pool through the standard periphery, the admin must allowlist the router — the exact action that collapses the guard. No privileged setup beyond the pool admin's own reasonable configuration is required. The attacker needs only to call the public router with any valid swap parameters. The exploit is repeatable on every swap.

## Recommendation
Pass the economically relevant actor — the end user — through the hook rather than the immediate `msg.sender`. Two complementary approaches:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and verifies it (with a pool-bound commitment so it cannot be spoofed from outside the router).

2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` already receives `extensionData`. Define a convention where the router encodes the real user in `extensionData`; the extension checks that field when `sender` is a known/trusted router, and falls back to `sender` for direct calls.

Either approach must ensure the encoded user identity cannot be spoofed by an arbitrary caller passing crafted `extensionData`.

## Proof of Concept
```solidity
// Setup: pool admin configures allowlist for user1 only, then allowlists router
// so user1 can use the periphery.
swapExtension.setAllowedToSwap(address(pool), user1, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // ← opens the gate

// Attacker (not allowlisted) bypasses the guard via the router:
vm.prank(attacker); // attacker is NOT in allowedSwapper[pool]
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          address(token0),
        tokenOut:         address(token1),
        zeroForOne:       true,
        amountIn:         1_000,
        amountOutMinimum: 0,
        recipient:        attacker,
        deadline:         block.timestamp + 1,
        priceLimitX64:    0,
        extensionData:    ""
    })
);
// Swap succeeds: allowedSwapper[pool][router] == true
// attacker receives token1 output despite not being on the allowlist
```

The `beforeSwap` hook receives `sender = address(router)`, looks up `allowedSwapper[pool][router] = true`, and returns the success selector — the attacker's swap settles in full.
