Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates LP Recipient (`owner`) Instead of Actual Depositor (`sender`), Allowing Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates access solely on `owner`, a free caller-supplied parameter. Because any caller can pass any allowlisted address as `owner`, the deposit allowlist is completely bypassable by any unprivileged address. This defeats the pool admin's intent to restrict depositors (e.g., KYC, compliance, institutional-only).

## Finding Description
`MetricOmmPool.addLiquidity` passes both `msg.sender` (as `sender`) and the caller-supplied `owner` to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol:191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` ignores the first parameter (`sender`) entirely — it is unnamed — and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol:32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

Inside the extension, `msg.sender` is the pool, so the effective check is `allowedDepositor[pool][owner]`. Since `owner` is a free parameter supplied by the caller with no pool-level restriction, any attacker can pass any allowlisted address as `owner` to satisfy the check.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swap initiator) and ignores `recipient`: [3](#0-2) 

The asymmetry confirms the deposit extension checks the wrong actor. No existing guard prevents a non-allowlisted caller from supplying an allowlisted `owner`.

## Impact Explanation
The deposit allowlist guard is completely ineffective. Any address not on the allowlist can call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)` directly. The extension sees `owner = allowlistedAddress` (allowlisted), passes the check, and the attacker's tokens are pulled via the modify-liquidity callback into the pool. This breaks the core pool functionality of access-restricted pools (KYC, compliance, institutional-only), allowing unauthorized parties to deposit into permissioned pools, manipulate bin depth and pool state, and introduce tokens from untrusted sources — a direct violation of the allowlist invariant the extension is designed to enforce.

## Likelihood Explanation
Exploitation requires only a direct call to `pool.addLiquidity` with any allowlisted address as `owner`. No special privileges, flash loans, or complex setup are needed. Pool addresses and allowlisted addresses are public on-chain. Any actor blocked by the allowlist can trivially and repeatedly bypass it.

## Recommendation
Check `sender` (the actual depositor) instead of `owner` (the LP position recipient), mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `setAllowedToDeposit` and `isAllowedToDeposit` documentation/semantics to reflect that the key is the depositor (caller), not the LP recipient.

## Proof of Concept

**Setup:**
- Pool deployed with `DepositAllowlistExtension` in `beforeAddLiquidity` order.
- `allowedDepositor[pool][alice] = true` (alice is allowlisted).
- `allowedDepositor[pool][attacker] = false` (attacker is NOT allowlisted).

**Attack:**
```solidity
// Attacker calls pool directly, setting owner = alice (allowlisted)
pool.addLiquidity(
    alice,        // owner — allowlisted, passes the guard
    salt,
    deltas,
    callbackData, // attacker implements the callback, pays tokens
    extensionData
);
// Pool calls: _beforeAddLiquidity(attacker /*sender*/, alice /*owner*/, ...)
// Extension checks: allowedDepositor[pool][alice] == true → passes
// Attacker's tokens are pulled; LP shares credited to alice
// Allowlist guard completely bypassed
```

The pool passes `msg.sender` (attacker) as `sender` and `alice` as `owner` to the hook. [1](#0-0) 

The extension ignores `sender` (attacker) and checks only `owner` (alice), which is allowlisted, so the check passes. [4](#0-3)
