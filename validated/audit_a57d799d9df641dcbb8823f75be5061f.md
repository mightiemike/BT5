### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Reentrancy via Publicly Callable `executeSlowModeTransaction` — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.sol::withdrawCollateral` performs the token transfer **before** updating the subaccount balance and running the health check. Because `Endpoint.sol::executeSlowModeTransaction` is publicly callable with no access restriction, an ERC-777 (or callback-enabled ERC-20) token's `tokensReceived` hook can re-enter `executeSlowModeTransaction` during the transfer, triggering a second `withdrawCollateral` call against the same subaccount while its on-chain balance is still stale (pre-deduction). If the subaccount holds additional collateral, both health checks pass and the attacker withdraws more tokens than they deposited.

---

### Finding Description

`Clearinghouse.sol::withdrawCollateral` executes in this order:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` → internally calls `token.safeTransfer(withdrawPool, amount)` then `BaseWithdrawPool.submitWithdrawal(...)` → which calls `token.safeTransfer(sendTo, amount)` (the actual user-facing transfer).
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — balance deducted **after** transfer.
3. **Line 419** — `require(getHealth(sender, healthType) >= 0)` — health check **after** transfer. [1](#0-0) 

The token transfer to the user (step 1) happens before the balance is decremented (step 2). This is a direct violation of the Checks-Effects-Interactions pattern.

`executeSlowModeTransaction` at line 231 of `Endpoint.sol` is externally callable by anyone with no access modifier:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [2](#0-1) 

`_executeSlowModeTransaction` dequeues and processes the next slow-mode transaction (including `WithdrawCollateral` type), ultimately calling `clearinghouse.withdrawCollateral`. [3](#0-2) 

The `handleWithdrawTransfer` in `Clearinghouse.sol` routes through `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(to, amount)` — the point at which an ERC-777 `tokensReceived` hook fires on the recipient contract. [4](#0-3) [5](#0-4) 

No `nonReentrant` guard exists on `withdrawCollateral`, `executeSlowModeTransaction`, or any function in the withdrawal call chain.

---

### Impact Explanation

An attacker holding a callback-enabled collateral token (ERC-777 or similar) and additional collateral (e.g., ETH) can:

1. Deposit 100 USDC (ERC-777) + sufficient ETH collateral.
2. Queue **two** slow-mode `WithdrawCollateral` transactions for 100 USDC each.
3. Call `executeSlowModeTransaction()` for the first withdrawal.
4. During the ERC-777 `tokensReceived` callback (fired when `withdrawPool` transfers USDC to the attacker), call `executeSlowModeTransaction()` again for the second withdrawal.
5. The second withdrawal executes with the **stale balance** (still 100 USDC, not yet decremented). `updateBalance` sets it to 0; `getHealth` passes because ETH covers the account.
6. Control returns to the first withdrawal. `updateBalance` sets USDC balance to −100. `getHealth` still passes because ETH collateral covers the deficit.
7. **Result**: 200 USDC withdrawn against 100 USDC deposited. The protocol absorbs a 100 USDC shortfall backed by the attacker's ETH — but the attacker can subsequently liquidate or withdraw the ETH through normal means, leaving the protocol insolvent for that product.

The corrupted state delta is: `spotEngine` balance for the subaccount's `productId` is decremented **twice** for a single deposit, creating an unbacked negative balance that passes the health check due to cross-margin collateral.

---

### Likelihood Explanation

- `executeSlowModeTransaction` is publicly callable — no privileged role required.
- Any user can queue slow-mode withdrawal transactions via `depositCollateralWithReferral` flow.
- ERC-777 tokens are ERC-20 compliant and can be listed as collateral products; the protocol makes no explicit exclusion.
- The attacker only needs to control a recipient contract that implements `tokensReceived` and calls `executeSlowModeTransaction` inside it.
- The cross-margin design (all collateral types contribute to a single health score) is what allows the health check to pass after the balance goes negative.

---

### Recommendation

1. **Apply `nonReentrant`** to `executeSlowModeTransaction` and `withdrawCollateral`.
2. **Follow CEI strictly** in `withdrawCollateral`: call `spotEngine.updateBalance` and `assertUtilization` and perform the health check **before** calling `handleWithdrawTransfer`.

Corrected order for `withdrawCollateral`:

```solidity
// 1. Checks
require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

// 2. Effects
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 3. Interactions (last)
handleWithdrawTransfer(token, sendTo, amount, idx);
```

### Proof of Concept

**Call stack for double-withdrawal via ERC-777 reentrancy:**

```
executeSlowModeTransaction()           ← public, no guard
  _executeSlowModeTransaction(idx=N)
    processSlowModeTransaction()
      clearinghouse.withdrawCollateral(sender, USDC, 100, ...)
        handleWithdrawTransfer(USDC, attacker, 100)
          USDC.safeTransfer(withdrawPool, 100)
          withdrawPool.submitWithdrawal(USDC, attacker, 100)
            USDC.safeTransfer(attacker, 100)
              !! ERC-777 tokensReceived callback fires !!
              !! attacker.tokensReceived() calls:
              executeSlowModeTransaction()   ← re-enters, idx=N+1
                clearinghouse.withdrawCollateral(sender, USDC, 100, ...)
                  !! balance still = 100 (not yet updated) !!
                  handleWithdrawTransfer(USDC, attacker, 100)  ← 2nd transfer
                  spotEngine.updateBalance(-100)  ← balance = 0
                  getHealth() >= 0  ← passes (ETH covers)
              !! returns to outer withdrawCollateral !!
        spotEngine.updateBalance(-100)  ← balance = -100
        getHealth() >= 0  ← passes (ETH still covers)
        !! 200 USDC transferred, only 100 deposited !!
``` [6](#0-5) [2](#0-1) [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);

            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
