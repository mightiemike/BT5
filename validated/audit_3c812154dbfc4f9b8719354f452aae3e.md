### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double Bonus Reward Drain and Checkpoint Reset — (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`ClaimTierRewards` accepts a caller-supplied slice of position IDs and processes them in two separate passes: a validation loop that loads all positions into memory, then a processing loop that claims rewards and writes updated checkpoints back to the store. Because there is no deduplication of position IDs between the two passes, submitting the same position ID twice causes the bonus reward computation to run twice against the same stale in-memory snapshot, paying the bonus twice. The second `setPosition` call then overwrites the store with the stale (pre-claim) checkpoint, resetting the position as if the claim never happened and allowing the attack to be repeated indefinitely.

---

### Finding Description

`MsgClaimTierRewards.PositionIds` is a user-supplied `[]uint64`. In `ClaimTierRewards`:

```
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)   // reads from store
    ...
    positions = append(positions, pos)             // stale snapshot appended
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

All positions are loaded into memory **before** any processing begins. If the same ID appears twice, two independent in-memory copies of the same position are appended, both carrying the same `LastEventSeq` and `LastBonusAccrual` checkpoint values.

`claimRewardsAndUpdatesPositions` then iterates over the slice:

```
for i := range positions {
    pos := &positions[i]
    base, err := k.claimBaseRewards(ctx, *pos)
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil {
``` [2](#0-1) 

**Iteration i=0** (first copy of position `id`):
- `processEventsAndClaimBonus` reads validator events since `pos.LastEventSeq`, computes and pays bonus, advances `pos.LastBonusAccrual` to `blockTime`, decrements event ref-counts.
- `setPosition` writes the updated checkpoint to the store.

**Iteration i=1** (second copy of position `id`, stale snapshot):
- `processEventsAndClaimBonus` is called again with the **original** `LastEventSeq` and `LastBonusAccrual`. Even if historical events were deleted by ref-count decrement, the current-segment bonus path still fires:

```go
if bonded && val.IsBonded() {
    currentRate, err := k.getTokensPerShare(ctx, valAddr)
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
    totalBonus = totalBonus.Add(bonus)
}
``` [3](#0-2) 

`segmentStart` equals the original `pos.LastBonusAccrual`, so the full accrued segment is computed and paid a second time.

- `setPosition` then **overwrites** the store with the stale position (original `LastBonusAccrual`), undoing the checkpoint advance written by iteration i=0. [4](#0-3) 

The net result: bonus is paid N times for N copies of the same ID, and the position's checkpoint is reset to its pre-claim state, making the attack repeatable in every subsequent transaction.

---

### Impact Explanation

The `RewardsPoolName` module account is drained at a rate proportional to the number of duplicate IDs submitted per transaction. Because the position checkpoint is reset after each attack transaction, the attacker can repeat the drain every block until the bonus pool is empty. Legitimate position holders lose access to their future bonus rewards. The corrupted `LastBonusAccrual` and `LastEventSeq` stored on-chain also permanently desynchronise the position's reward accounting.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, governance action, or IBC relay is required. The only prerequisite is owning at least one delegated position with accrued bonus rewards. The attack is trivially constructable by any user who inspects the message type.

---

### Recommendation

Add deduplication of `PositionIds` before the validation loop in `ClaimTierRewards`, or validate inside `claimRewardsAndUpdatesPositions` that each position ID is processed at most once. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(types.ErrInvalidPositionID, "duplicate position id %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Additionally, consider reading each position fresh from the store inside `claimRewardsAndUpdatesPositions` rather than operating on pre-loaded snapshots, so that any store update from a prior iteration is visible to subsequent ones.

---

### Proof of Concept

1. Alice owns position ID `42` with a bonded validator and 100 accrued bonus tokens.
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42, 42, 42, 42]}`.
3. The validation loop loads five identical copies of position 42 (all with `LastBonusAccrual = T0`).
4. `claimRewardsAndUpdatesPositions` iterates five times:
   - Each iteration computes the segment `[T0, blockTime]` and pays 100 bonus tokens.
   - Each `setPosition` call overwrites the store with `LastBonusAccrual = T0`.
5. Alice receives 500 bonus tokens instead of 100.
6. The store still shows `LastBonusAccrual = T0`, so Alice can repeat the transaction next block. [5](#0-4) [6](#0-5)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

// claimRewardsAndUpdatesPositions claims base and bonus rewards for multiple positions.
func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
