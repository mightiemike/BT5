### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` Permanently Blocks Finalization When `openInterest == 0` — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when a residual negative `vQuoteBalance` remains after insurance. If `openInterest` is zero at that moment, `MathSD21x18.div` explicitly reverts with `"DBZ"`, permanently blocking `_finalizeSubaccount` and trapping bad debt in the protocol.

---

### Finding Description

In `_finalizeSubaccount`, after all perp position amounts are confirmed zero and positive PnL is settled, the clearinghouse calls: [1](#0-0) 

Inside `socializeSubaccount`, for each product where `balance.vQuoteBalance < 0` after insurance is applied, the code attempts to spread the loss across remaining open-interest holders: [2](#0-1) 

The `div` implementation in `MathSD21x18` has an explicit guard: [3](#0-2) 

When `state.openInterest == 0`, this `require(y != 0, ERR_DIV_BY_ZERO)` fires, reverting the entire transaction.

**How `openInterest == 0` with `vQuoteBalance < 0` is reached:**

`_finalizeSubaccount` requires `balance.amount == 0` for every perp product before proceeding: [4](#0-3) 

A zero `balance.amount` means the liquidatee's position is closed, but `vQuoteBalance` retains the locked-in PnL from when the position was opened/closed. If all *other* participants have also closed their positions on that product, `state.openInterest` is exactly zero. `_updateBalance` in `PerpEngineState` tracks this: [5](#0-4) 

When every participant's `balance.amount` is zero, `openInterest` sums to zero. This is a realistic end-state for a low-activity or deprecated market.

---

### Impact Explanation

- `_finalizeSubaccount` always reverts for the affected product, making it impossible to close the subaccount.
- Insurance funds allocated to cover the deficit are never consumed — they remain locked in the clearinghouse variable `v.insurance` and are never written back.
- The bad debt (`vQuoteBalance < 0`) is permanently frozen in the protocol with no recovery path.
- No admin override or alternative finalization path exists in the scoped code.

This is a **Critical** solvency/accounting failure: bad debt is permanently locked, the subaccount cannot be closed, and the insurance fund is rendered ineffective for this case.

---

### Likelihood Explanation

The precondition is reachable through normal protocol operation:
1. A market winds down or loses all participants except the liquidatee.
2. The liquidatee's position is closed (amount = 0) but carries negative `vQuoteBalance`.
3. Insurance is insufficient to fully cover the deficit.
4. Any call to `liquidateSubaccountImpl` with `productId = type(uint32).max` will revert indefinitely.

No privileged access, oracle manipulation, or external dependency failure is required.

---

### Recommendation

Add a zero-check guard before the division in `socializeSubaccount`. If `openInterest == 0` and `vQuoteBalance < 0` after insurance, there are no counterparties to socialize against. The remaining loss should either be absorbed by the insurance fund entirely (even if it goes negative, to be reconciled later) or written off directly:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb loss directly
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The exact policy (write-off vs. insurance overdraft) should match the protocol's intended bad-debt accounting, but the division must be guarded unconditionally.
