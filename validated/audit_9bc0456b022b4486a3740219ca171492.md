### Title
Raw `approve()` Call via Boolean-Returning Interface Breaks `creditDeposit()` for Non-Standard ERC20 Tokens (e.g., USDT) — (`File: core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` through the `IIERC20Base` interface, which declares `approve` as returning `bool`. USDT's `approve` returns no value. Solidity's ABI decoder reverts when the returned data length does not match the declared return type, permanently breaking `creditDeposit()` for any DDA holding USDT and trapping those funds.

---

### Finding Description

`DirectDepositV1.sol` defines its own local interface `IIERC20Base` with `approve` declared as returning `bool`: [1](#0-0) 

Inside `creditDeposit()`, this interface is used to call `approve` directly — not through any safe wrapper: [2](#0-1) 

The same file already implements a `safeTransfer` helper that uses a low-level `.call()` and explicitly handles the `data.length == 0` case for non-standard tokens: [3](#0-2) 

No equivalent safe wrapper exists for `approve`. The raw interface call at line 92 causes Solidity's ABI decoder to check that the returned data is exactly 32 bytes (one `bool`). USDT's `approve` returns nothing (0 bytes), so the decoder reverts with `"function returned an unexpected amount of data"`.

The same pattern exists in `IERC20Base` used by `ContractOwner`: [4](#0-3) 

`ContractOwner.wrapVaultAsset()` also calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` through this interface with no access control: [5](#0-4) 

---

### Impact Explanation

Any `DirectDepositV1` (DDA) contract deployed for a subaccount where USDT is a registered spot product will have `creditDeposit()` permanently broken. Because `creditDeposit()` iterates over **all** product IDs and reverts on the first non-zero USDT balance, the entire deposit flow is blocked — not just for USDT but for every token in that DDA. USDT sent to the DDA address cannot be forwarded to the endpoint as collateral and is effectively trapped. [2](#0-1) 

---

### Likelihood Explanation

USDT is one of the most widely used stablecoins and a natural collateral candidate for a trading protocol. `ContractOwner.creditDepositV1()` has no access control — any user can call it to trigger `creditDeposit()`: [6](#0-5) 

Once USDT is registered as a spot product and a user sends USDT to a DDA address, the revert is deterministic and unrecoverable through the normal deposit path.

---

### Recommendation

Replace the raw `token.approve(...)` call in `creditDeposit()` with a safe wrapper analogous to the existing `safeTransfer` helper — using a low-level `.call()` and accepting `data.length == 0` as a valid success response:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to `IERC20Base` usages in `ContractOwner.wrapVaultAsset()` and `ContractOwner.depositInsurance()`.

---

### Proof of Concept

1. USDT is registered as a spot product in `SpotEngine`.
2. A user calls `ContractOwner.creditDepositV1(subaccount)` (no access control).
3. `DirectDepositV1.creditDeposit()` is invoked; it iterates product IDs and finds a non-zero USDT balance.
4. It executes `token.approve(address(endpoint), balance)` via `IIERC20Base` — which expects a `bool` return.
5. USDT's `approve` returns no data (0 bytes); Solidity's ABI decoder reverts: `"function returned an unexpected amount of data"`.
6. The entire `creditDeposit()` call reverts. USDT (and any other token in the same DDA) cannot be deposited. Funds are stuck.

This mirrors the confirmed PoC from the reference report, where `IERC20(token).approve(spender, 0)` compiled to code that checks `data.length != 1` and reverts when USDT returns 0 bytes.
