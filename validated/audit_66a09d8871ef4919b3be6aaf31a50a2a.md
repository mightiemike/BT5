Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks immediate caller (`sender`) instead of ultimate user, allowing router-mediated bypass of per-user swap gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is the `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension checks whether the **router** is allowlisted rather than the **actual user**. If the router is allowlisted (a natural operational step), every user who calls the router bypasses the per-user swap gate entirely, breaking the intended access control invariant.

## Finding Description
In `SwapAllowlistExtension.beforeSwap`, the check at line 37 is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the pool calls the extension), and `sender` is the value passed by the pool from its own `msg.sender` — whoever called `pool.swap()`.

In `MetricOmmPool.swap` (lines 230–240), the pool passes `msg.sender` directly as `sender` to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```

In `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80), the router calls `pool.swap()`, making the router `msg.sender` of `pool.swap()`. Therefore `sender = router` in the extension, and the check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` (line 38) correctly ignores `sender` and checks `owner` (the position owner), making it router-agnostic. `SwapAllowlistExtension` does the opposite and is therefore router-sensitive.

**Exploit path:**
1. Pool deployed with `SwapAllowlistExtension`; `allowedSwapper[pool][router] = true` (admin allowlists router for normal trading); `allowedSwapper[pool][alice] = false`
2. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`
3. Router calls `pool.swap(recipient=alice, ...)`
4. Pool calls `_beforeSwap(sender=router, ...)`
5. Extension checks `allowedSwapper[pool][router]` → `true`; swap proceeds
6. Alice successfully trades against the restricted pool without being individually allowlisted

## Impact Explanation
This is an admin-boundary break: the pool admin's configured per-user access control is bypassed by an unprivileged path. A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a specific set of addresses (KYC-gated pool, private institutional pool, market-maker-only pool) must allowlist the `MetricOmmSimpleRouter` to enable standard trading UX. Once the router is allowlisted, **every user who calls the router bypasses the individual user gate**, because the extension sees `sender = router`. Non-allowlisted users can freely trade against a pool intended to be restricted, extracting value from LPs who expected a controlled counterparty set. The admin has no way to simultaneously allow router-based trading and enforce per-user restrictions using this extension as written.

## Likelihood Explanation
Likelihood is medium. The `SwapAllowlistExtension` is a production periphery contract explicitly designed for deployment. Any pool that configures `SwapAllowlistExtension` to restrict swaps and allowlists `MetricOmmSimpleRouter` (a natural and expected operational step) is immediately vulnerable. The bypass requires no special privileges — any user with a wallet can call the router. The inconsistency with `DepositAllowlistExtension` (which correctly checks `owner`) confirms this is an unintentional design flaw rather than a documented limitation.

## Recommendation
`SwapAllowlistExtension.beforeSwap` should check the **ultimate user** rather than the immediate caller. The most consistent fix, aligned with `DepositAllowlistExtension`'s pattern, is to check `recipient` (the second parameter, currently ignored with `address`) instead of `sender`. Alternatively, the router can encode the real user's address in `extensionData` and the extension can decode and check it. At minimum, the extension must be documented to clarify that allowlisting the router opens the gate to all users, and pool admins must be warned not to allowlist the router if per-user restrictions are required.

```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - allowedSwapper[pool][router] = true   (admin allowlists router for normal trading)
  - allowedSwapper[pool][alice] = false   (alice is NOT individually allowlisted)

Attack:
  1. alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: alice, ...})
  2. Router calls pool.swap(recipient=alice, ...)                    [MetricOmmSimpleRouter.sol L72-80]
  3. pool.swap() calls _beforeSwap(sender=router, ...)              [MetricOmmPool.sol L230-240]
  4. SwapAllowlistExtension.beforeSwap(sender=router, ...)
     checks allowedSwapper[pool][router] → true                     [SwapAllowlistExtension.sol L37]
  5. Swap proceeds; alice successfully trades against the restricted pool

Expected: revert NotAllowedToSwap (alice is not allowlisted)
Actual:   swap succeeds (router is allowlisted, alice's identity is never checked)
```
