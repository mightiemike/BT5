Audit Report

## Title
SwapAllowlistExtension Gates the Router's Address Instead of the Actual Swapper When Trades Are Routed Through MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`, which is forwarded to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. This either allows unpermissioned users to bypass the allowlist (if the router is allowlisted) or blocks legitimate allowlisted users from using the supported router path.

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← pool's msg.sender, not the originating user
  recipient,
  zeroForOne,
  amountSpecified,
  priceLimitX64,
  packedSlot0Initial,
  bidPriceX64,
  askPriceX64,
  extensionData
);
``` [1](#0-0) 

This `msg.sender` value becomes the `sender` parameter in `SwapAllowlistExtension.beforeSwap`, which gates it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

`MetricOmmSimpleRouter` calls `pool.swap(...)` directly in all four swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), making the router the pool's `msg.sender`: [3](#0-2) [4](#0-3) 

The pool therefore passes `sender = router` to the extension. The extension checks `allowedSwapper[pool][router]`, never touching `allowedSwapper[pool][actual_user]`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` receives an explicit `owner` parameter that is separate from `msg.sender` (the operator pattern), so the liquidity path correctly gates the economic actor: [5](#0-4) 

The swap path has no equivalent explicit-originator parameter — it is always the pool's `msg.sender`.

## Impact Explanation

**Path A — Allowlist bypass (High):** A pool admin allowlists the router (a natural configuration to support router-mediated swaps). Any unpermissioned user routes through `MetricOmmSimpleRouter`; the extension sees `sender = router`, which is allowlisted, and passes. The actual user's address is never checked. Non-KYC'd or otherwise excluded users trade freely on a curated pool, directly violating the pool's access policy and exposing LPs to unauthorized counterparties — a direct loss of access-control integrity with fund-impacting consequences for LPs.

**Path B — Broken core swap functionality (Medium/High):** If the admin does not allowlist the router, every individually allowlisted user who routes through `MetricOmmSimpleRouter` receives `NotAllowedToSwap`, even though they are permitted. The router is a supported periphery path; blocking it breaks the core swap flow for legitimate users.

Both outcomes are reachable by any public user with no privileged access.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract callable by anyone.
- Pool admins configuring a swap allowlist will naturally want to support router-mediated swaps (multi-hop, slippage wrappers), making Path A a realistic and likely misconfiguration.
- Path B is triggered the moment any allowlisted user attempts a router swap, which is a normal usage pattern requiring no special setup.
- No malicious tokens, privileged roles, or non-standard ERC20 behavior are required.

## Recommendation

The swap path must carry the originating user's identity through to the extension, mirroring how `addLiquidity` separates `owner` from `msg.sender`:

1. **Router-side:** `MetricOmmSimpleRouter` should encode the original `msg.sender` (the user) into `extensionData` so the extension can decode and check it.
2. **Pool/interface-side (preferred):** Add an explicit `sender` override parameter to `pool.swap()` analogous to the `owner` parameter in `addLiquidity`, and have the pool forward it to extensions instead of its own `msg.sender`.
3. **Extension-side (interim):** `SwapAllowlistExtension.beforeSwap` can decode an originator address from `extensionData` when the immediate `sender` is a known router, falling back to `sender` for direct swaps.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, router, true)   // intends to allow router-mediated swaps
  admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack (Path A):
  attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  router calls pool.swap(...)  →  pool's msg.sender = router
  pool calls _beforeSwap(msg.sender=router, ...)
  extension checks allowedSwapper[pool][router]  →  true  →  passes
  attacker's swap executes on the curated pool

Result:
  attacker (not allowlisted) successfully swaps on a pool that was supposed to gate them
  the allowlist invariant is broken; LPs are exposed to unauthorized counterparties

Path B (no router allowlisting):
  allowlisted user calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  router calls pool.swap()  →  sender = router
  extension checks allowedSwapper[pool][router]  →  false  →  NotAllowedToSwap
  allowlisted user's swap is rejected through the supported periphery path
```

A Foundry test can confirm both paths by deploying a pool with `SwapAllowlistExtension`, calling `setAllowedToSwap(pool, router, true)`, then asserting that an address not in the allowlist can successfully swap via the router.
