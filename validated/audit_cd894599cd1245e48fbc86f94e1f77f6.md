Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap()` evaluates `allowedSwapper[msg.sender][sender]` where `sender` is the immediate caller of `MetricOmmPool.swap()`. When users interact through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual end-user. A pool admin who whitelists the router to allow their KYC'd users to swap through the standard interface inadvertently grants every unprivileged address the ability to bypass the allowlist entirely, enabling unauthorized swaps against LP-provided liquidity at oracle-derived prices.

## Finding Description

**Root cause:** `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at lines 230–231, forwarding its own `msg.sender` — the immediate caller — as `sender` to the extension. When `MetricOmmSimpleRouter.exactInputSingle()` is used, the router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` at lines 72–80, making the router the `msg.sender` to the pool. The extension therefore receives the router's address as `sender`, never the actual end-user.

**Extension check:** `SwapAllowlistExtension.beforeSwap()` at line 37 evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
This resolves to `allowedSwapper[pool][router]`. If the pool admin has called `setAllowedToSwap(pool, address(router), true)` to permit their whitelisted users to use the canonical interface, this single mapping entry satisfies the check for **every caller** of the router, regardless of whether they are individually whitelisted.

**Existing guards are insufficient:** There is no mechanism in the extension, the pool, or the router to propagate the real `msg.sender` of the router call into the `sender` field seen by the extension. The `extensionData` field is caller-supplied and unverified, so it cannot be trusted as a source of the real user identity without additional authentication logic that does not exist.

**Two broken outcomes:**

| Router allowlist state | Outcome |
|---|---|
| Router **not** whitelisted | All router users blocked, including individually whitelisted ones |
| Router **whitelisted** | Every user bypasses the allowlist — the guard is nullified |

## Impact Explanation

A pool deploying `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties or institutional partners). If the pool admin whitelists the router — the natural and necessary action to allow those users to use the standard interface — the allowlist is silently nullified. Any unprivileged address can call `exactInputSingle` or `exactInput` on the router and execute swaps in the supposedly restricted pool. Unauthorized swappers can drain LP-provided liquidity at the pool's oracle-derived bid/ask prices, causing direct loss of LP principal. This constitutes a direct loss of user principal above Sherlock thresholds and a broken core pool functionality (access-controlled swap flow).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical swap interface for the protocol. Any pool admin who configures `SwapAllowlistExtension` and also wants their allowlisted users to use the router will inevitably whitelist the router, triggering the bypass. The attacker requires no special role, no privileged tokens beyond the swap input amount, and no setup beyond calling the public router. The condition is self-inflicted by correct administrative use of the system.

## Recommendation

The extension must verify the actual end-user, not the immediate pool caller. Two complementary approaches:

1. **Pass the real user through `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling the pool. The extension reads and verifies this value, but must also validate that `sender` (the immediate caller) is a trusted, factory-registered router; otherwise any caller can forge a whitelisted address in `extensionData`.

2. **Direct-only constraint:** Remove router-level whitelisting entirely and require end-users to call `pool.swap()` directly when the allowlist extension is active, documenting this constraint clearly in the extension interface.

The simplest safe fix is option 2. Option 1 requires the extension to maintain a registry of trusted routers and verify the encoded real sender, which is a more complex but more flexible solution.

## Proof of Concept

```solidity
// Pool admin setup (legitimate, intended to allow KYC'd users to use the router):
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not KYC'd, not individually whitelisted):
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        recipient: attacker,
        tokenIn: token0,
        zeroForOne: true,
        amountIn: largeAmount,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Call chain:
// router.exactInputSingle() -> pool.swap(msg.sender=router) 
// -> _beforeSwap(sender=router) -> extension.beforeSwap(sender=router)
// -> allowedSwapper[pool][router] == true -> no revert
// Attacker swaps successfully
```
