Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mid-Price Instead of Geometric Mid-Price, Causing Stop-Loss to Fail for `!zeroForOne` Swaps — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterSwapOracleStopLoss` computes `midPriceX64` as the arithmetic mean `(bidPriceX64 + askPriceX64) / 2`, while `SwapMath.midAndSpreadFeeX64FromBidAsk` — used by both `MetricOmmPool` for swap execution and `PriceVelocityGuardExtension` — computes the geometric mean `sqrt(bid * ask)`. By AM-GM inequality the arithmetic mean is always ≥ the geometric mean, so `metricT1` is systematically overestimated, making the `!zeroForOne` stop-loss breach condition harder to satisfy and allowing token0 to drain past the configured drawdown floor.

## Finding Description

**Root cause:** In `_afterSwapOracleStopLoss`, the mid price is computed as:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
``` [1](#0-0) 

`SwapMath.midAndSpreadFeeX64FromBidAsk`, used everywhere else, computes the geometric mean:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`MetricOmmPool.swap` uses the geometric mean for actual swap execution: [3](#0-2) 

`PriceVelocityGuardExtension` also uses the geometric mean: [4](#0-3) 

**How the inflated mid corrupts `metricT1`:** The `_metrics` function computes `metricT1` by converting t0 into t1-denominated value using `midPriceX64`:

```solidity
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [5](#0-4) 

Since `mid_arithmetic ≥ mid_geometric`, the t0 contribution to `metricT1` is always overestimated. The stop-loss for `!zeroForOne` swaps only triggers when `breach1 && !zeroForOne`: [6](#0-5) 

An inflated `metricT1` raises the apparent per-share value, making `breach1` harder to satisfy. Swaps that should have been reverted are allowed to execute.

## Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism preventing per-share value from falling below the configured drawdown floor. When it fails to trigger for `!zeroForOne` swaps, token0 exits the pool without reversal. LP principal is permanently lost up to the magnitude of the overestimation times the watermark value. For a 50% oracle spread, the overestimation is ~3.3%, meaning a pool configured with a 3% drawdown floor could sustain a ~6.3% actual loss before the stop-loss triggers. This constitutes a direct loss of LP assets above Sherlock thresholds for pools with non-trivial oracle spreads.

## Likelihood Explanation

Every pool deploying `OracleValueStopLossExtension` is affected whenever `bid ≠ ask` (any real pool with non-zero spread). No special attacker setup is required — any ordinary `!zeroForOne` swap that crosses the drawdown boundary exploits the gap. The discrepancy scales with oracle spread, which can be large during volatile markets. No privileged access is needed.

## Recommendation

Replace the arithmetic mean with the same geometric-mean helper used by the pool and the velocity guard:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` configured with `drawdownE6 = 30_000` (3% floor).
2. Set the oracle to return `bid = 0.75 * ONE_X64`, `ask = 1.25 * ONE_X64` (50% spread). Arithmetic mid = `1.0 * ONE_X64`; geometric mid ≈ `0.9683 * ONE_X64`.
3. Seed the pool with token0 and token1 so that the initial watermarks are set at the arithmetic-mid metric values.
4. Execute a sequence of `!zeroForOne` swaps that drain token0 until the actual per-share value (at geometric mid) has fallen 6% below the watermark — 3% beyond the configured floor.
5. Observe: the stop-loss does not revert any of these swaps because `metricT1` computed with the inflated arithmetic mid still appears above the floor.
6. LP withdraws and receives ~3% less token0 than the stop-loss was supposed to guarantee.
