### Title
Imbalanced Open Interest Causes Funding Payment Accounting Mismatch, Leading to Protocol Undercollateralization — (`core/contracts/PerpEngineState.sol`)

---

### Summary

`updateStates` increments both `cumulativeFundingLongX18` and `cumulativeFundingShortX18` by the **same per-unit** `paymentAmount`. Because `_updateBalance` applies funding as `diffX18 * balance.amount`, the total funding debited from longs equals `paymentAmount × totalLongOI` and the total credited to shorts equals `paymentAmount × totalShortOI`. When these two quantities differ (i.e., OI is imbalanced), the net `vQuoteBalance` change across all participants is non-zero, and no mechanism compensates for the difference. This creates either phantom credits (shorts dominate → undercollateralization) or destroyed value (longs dominate → user loss).

---

### Finding Description

In `updateStates`:

```solidity
int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
state.cumulativeFundingLongX18 += paymentAmount;
state.cumulativeFundingShortX18 += paymentAmount;
``` [1](#0-0) 

Both accumulators receive the **same per-unit increment**. In `_updateBalance`, each position's funding settlement is:

```solidity
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
``` [2](#0-1) 

- A long with `amount = A > 0` has its `vQuoteBalance` changed by `−paymentAmount × A`.
- A short with `amount = −S < 0` has its `vQuoteBalance` changed by `+paymentAmount × S`.

Summing over all participants:

```
ΔvQuote_total = paymentAmount × (totalShortOI − totalLongOI)
```

When `totalShortOI > totalLongOI`, this sum is **positive**: shorts are credited more than longs are debited. No real USDC is added to `state.availableSettle` to back these phantom credits. [3](#0-2) 

Settlement in `getSettlementState` caps what a user can withdraw at `min(positionPnl, state.availableSettle)`:

```solidity
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
``` [4](#0-3) 

`state.availableSettle` is never increased by `updateStates`, so the phantom credits accumulate without backing. [5](#0-4) 

---

### Impact Explanation

**Undercollateralization (shorts dominate):** With 1 long of size 1 and 10 shorts of size 1 each, after one `updateStates` call with `paymentAmount = P`:

| Participant | vQuoteBalance Δ |
|---|---|
| 1 long | −P |
| 10 shorts | +P each = +10P total |
| **Net** | **+9P** |

The protocol owes `9P` more in aggregate `vQuoteBalance` than longs paid. This `9P` is unbacked. As funding accumulates over time, the gap grows, and eventually `state.availableSettle` is insufficient to honor all short settlements.

**Value destruction (longs dominate):** With 10 longs of size 1 and 1 short of size 1, longs collectively pay `10P` but the short only receives `P`. The `9P` difference is neither credited to the insurance fund nor to `availableSettle` — it is simply destroyed, causing longs to be systematically overcharged.

---

### Likelihood Explanation

OI imbalance is the normal state of any perpetual market. The protocol does not enforce balanced OI. Every `updateStates` call (which is a routine sequencer operation via `onlyEndpoint`) with any non-zero `priceDiff` and imbalanced OI silently widens the accounting gap. No special attacker action is required — a user simply needs to hold a short position when shorts dominate OI and collect funding that exceeds what longs paid. [6](#0-5) 

---

### Recommendation

The funding mechanism must distinguish between what longs pay and what shorts receive. The standard approach is to scale the per-unit rate by the respective side's OI:

```solidity
// Longs pay: paymentAmount per unit × totalLongOI
// Shorts receive: paymentAmount per unit × totalShortOI
// Difference goes to / from insurance fund
int128 longOI  = state.openInterest; // track separately if needed
int128 shortOI = state.openInterest; // same if symmetric, else track separately

int128 longPayment  = paymentAmount; // per-unit rate stays the same
int128 shortPayment = longOI >= shortOI
    ? paymentAmount.mul(longOI).div(shortOI)   // shorts receive proportionally more
    : paymentAmount;                            // or route surplus to insurance
```

Alternatively, route the imbalance explicitly to/from `state.availableSettle` or the insurance fund so the accounting identity `Σ vQuoteBalance = 0` is always maintained.

### Proof of Concept

```solidity
// Setup: 1 long of size 1, 10 shorts of size 1 each, same product
// All lastCumulativeFundingX18 = 0, cumulativeFundingLong/Short = 0

// Step 1: call updateStates with priceDiff > 0, dt = 1 day
// paymentAmount = priceDiff (for simplicity, priceDiff = 1e18)
// After: cumulativeFundingLongX18 = 1e18, cumulativeFundingShortX18 = 1e18

// Step 2: call _updateBalance(0, 0) for each participant to realize funding

// Long's vQuoteBalance delta:
//   diffX18 = 1e18 - 0 = 1e18
//   deltaQuote = 0 - 1e18 * 1 = -1e18  → long loses 1e18

// Each short's vQuoteBalance delta:
//   diffX18 = 1e18 - 0 = 1e18
//   deltaQuote = 0 - 1e18 * (-1) = +1e18  → each short gains 1e18

// Sum of all vQuoteBalance changes:
//   -1e18 (long) + 10 * 1e18 (shorts) = +9e18

// state.availableSettle was never increased → protocol is undercollateralized by 9e18
// assert(sum_of_all_vQuoteBalance_deltas == 0)  // FAILS: actual = +9e18
```

The invariant `Σ ΔvQuoteBalance = 0` fails whenever `totalLongOI ≠ totalShortOI`, which is the common case in any live market. [7](#0-6)

        State memory state,
        Balance memory balance,
        int128 balanceDelta,
        int128 vQuoteDelta

        // pre update
        state.openInterest -= balance.amount.abs();
        int128 cumulativeFundingAmountX18 = (balance.amount > 0)
            ? state.cumulativeFundingLongX18
            : state.cumulativeFundingShortX18;
        int128 diffX18 = cumulativeFundingAmountX18 -
            balance.lastCumulativeFundingX18;

        // apply delta
        balance.amount += balanceDelta;

        // apply vquote
        balance.vQuoteBalance += deltaQuote;

        // post update
        if (balance.amount > 0) {
            state.openInterest += balance.amount;
            balance.lastCumulativeFundingX18 = state.cumulativeFundingLongX18;
        } else {
            state.openInterest -= balance.amount;
            balance.lastCumulativeFundingX18 = state.cumulativeFundingShortX18;
        }
    }
