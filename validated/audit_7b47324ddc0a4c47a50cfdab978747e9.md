### Title
Fork Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor, Corrupting Difficulty Calculation for Fork Blocks - (File: `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When a fork block falls on a difficulty retarget boundary and the fork diverged before the start of the current retarget period, `get_header_by_height` returns the **mainchain** block at the period-start height rather than the fork's actual ancestor at that height. This causes the difficulty calculation to use the wrong timestamp, producing an incorrect `bits` target for the fork block. An unprivileged relayer can exploit this by crafting a fork whose mainchain period-start block has a favorable timestamp, causing the contract to accept fork blocks with a lower difficulty than the fork chain actually requires.

### Finding Description

All three Bitcoin-family difficulty modules call `get_header_by_height` to retrieve the block at the start of the retarget period:

**`contract/src/bitcoin.rs`** [1](#0-0) 

**`contract/src/litecoin.rs`** [2](#0-1) 

**`contract/src/dogecoin.rs`** [3](#0-2) 

The implementation of `get_header_by_height` always resolves the height through the **mainchain** index: [4](#0-3) 

This is correct only when the block being validated is on the mainchain. When validating a **fork block** at a retarget boundary, the fork may have diverged before `first_block_height`. In that case, the fork's ancestor at `first_block_height` is a fork block (`F_{first_block_height}`), not the mainchain block (`M_{first_block_height}`). The code silently uses `M_{first_block_height}.time` instead of `F_{first_block_height}.time`.

The developers themselves flagged this exact concern with a TODO comment: [5](#0-4) 

### Impact Explanation

The difficulty calculation computes:

```
actual_time_taken = prev_block_header.time - first_block_time
```

where `first_block_time` is taken from the wrong block. If `M_{first_block_height}.time` is earlier than `F_{first_block_height}.time`, the computed period appears longer than it actually is, and the retarget produces a **lower difficulty** (easier target) for the fork block. The attacker then mines the retarget block against this easier target, and all subsequent fork blocks inherit the reduced difficulty. With less PoW required per block, the attacker can accumulate chainwork faster than the honest chain expects, potentially triggering a chain reorganization that corrupts the canonical chain state stored in `mainchain_height_to_header` and `mainchain_tip_blockhash`. [6](#0-5) 

### Likelihood Explanation

Any unprivileged caller can invoke `submit_blocks` (subject to the trusted-relayer gate, but the `UnrestrictedSubmitBlocks` role exists and the relayer path is the primary production entry point): [7](#0-6) 

The attacker only needs to:
1. Identify a retarget boundary where the mainchain block at `first_block_height` has an earlier timestamp than the fork's ancestor would have.
2. Submit a fork that diverges before `first_block_height`.
3. Submit the retarget block with the artificially lowered `bits` value.

All of this is achievable with no privileged keys and no off-chain coordination beyond running a relayer.

### Recommendation

Replace the `get_header_by_height` call in all three retarget functions with a walk that follows `prev_block_hash` links backward from the current fork tip to the correct ancestor at `first_block_height`. This is exactly what the TODO comment in `dogecoin.rs` anticipates. The `get_prev_header` method already traverses `headers_pool` by hash and is fork-aware: [8](#0-7) 

The retarget functions should walk back `difficulty_adjustment_interval - 1` steps using `get_prev_header` starting from `prev_block_header`, rather than using the height-indexed mainchain lookup.

### Proof of Concept

**Setup (Bitcoin feature, 2016-block retarget):**

- Mainchain has blocks `M_0 … M_4031`. `M_2015` has timestamp `T_early` (e.g., 2 weeks before `M_4031`).
- Attacker creates a fork starting at height 1900 (before the retarget boundary at 2016).
- Fork blocks `F_1901 … F_2015` are mined at the mainchain difficulty.

**Trigger:**

- At height 2016 (retarget boundary), `get_next_work_required` computes:
  - `first_block_height = 2015 - 2015 = 0`
  - `get_header_by_height(0)` → `M_0` (correct, since fork started after height 0)

Let's use a more targeted example: fork starts at height 1000, retarget at height 2016:
- `first_block_height = 2015 - 2015 = 0` — still `M_0`, which is the common ancestor. OK.

More precisely, fork starts at height 100, retarget at height 4032 (second retarget):
- `first_block_height = 4031 - 2015 = 2016`
- `get_header_by_height(2016)` → `M_2016` (mainchain block)
- But fork's ancestor at height 2016 is `F_2016` (a fork block with attacker-controlled timestamp)
- If `M_2016.time` is earlier than `F_2016.time` would be, the period appears longer → lower difficulty
- Attacker mines `F_4032` against this lower difficulty, submits it via `submit_blocks`
- Contract accepts `F_4032` because `check_target` passes with the incorrectly computed `bits` [9](#0-8) [10](#0-9)

//https://github.com/bitcoin/bitcoin/blob/ae024137bda9fe189f4e7ccf26dbaffd44cbbeb6/src/pow.cpp#L14
fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    let first_block_height =
        prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
}

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
