### Title
`ValidResourceBounds::AllResources` with zero L2/L1DataGas silently collapses to `L1Gas` in protobuf deserialization, producing a divergent transaction hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — checking whether `l2_gas` and `l1_data_gas` are zero — to decide which enum variant to reconstruct. A post-0.13.3 transaction submitted with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is accepted by the gateway, hashed with a 4-element Poseidon preimage (tip + L1_GAS + L2_GAS + L1_DATA_GAS), and stored with that hash. When the same transaction is later serialized to protobuf and deserialized (P2P sync path), it silently becomes `L1Gas(X)`, whose hash uses only a 3-element preimage (tip + L1_GAS + L2_GAS). The two hashes are structurally different. Any code path that recomputes the hash from the stored transaction object — including `validate_transaction_hash`, blockifier re-execution, or RPC simulation — will produce a hash that does not match the one the user signed and the sequencer committed.

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`):

Both `AllResources(l1_gas=X, l2_gas=ZERO, l1_data_gas=ZERO)` and `L1Gas(X)` serialize to identical protobuf bytes: `l1_gas=X, l2_gas=ZERO, l1_data_gas=ZERO`. The variant tag is not preserved. [1](#0-0) 

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`):

The reconstructed variant is chosen purely by whether the numeric values are zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

**Hash divergence** (`get_tip_resource_bounds_hash`):

The function branches on the variant to decide how many elements enter the Poseidon hash:

- `AllResources` → `[tip, L1_GAS, L2_GAS, L1_DATA_GAS]` (4 elements)
- `L1Gas` → `[tip, L1_GAS, L2_GAS]` (3 elements) [3](#0-2) 

The two Poseidon outputs are cryptographically distinct. The transaction hash embedded in `InternalRpcTransaction.tx_hash` (computed at gateway ingress with `AllResources`) will never equal the hash recomputed from the protobuf-deserialized object (computed with `L1Gas`). [4](#0-3) 

**Trigger condition**: A user submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: ZERO, l1_data_gas: ZERO }`. The gateway accepts it (at least one bound is non-zero), converts it to `ValidResourceBounds::AllResources`, computes hash H1, and stores `InternalRpcTransaction { tx_hash: H1, ... }`. When the block is propagated via P2P sync, the `InvokeTransactionV3` is serialized to `protobuf::InvokeV3` and deserialized back as `L1Gas(X)`. Any subsequent hash recomputation yields H2 ≠ H1. [5](#0-4) 

**Storage serialization does not have this bug** — the `StorageSerde` for `InvokeTransactionV3` uses the `ValidResourceBounds` binary serializer which preserves the variant tag. The bug is specific to the protobuf path. [6](#0-5) 

### Impact Explanation

Any node that receives the transaction via P2P sync and then calls `validate_transaction_hash` (which recomputes the hash from the stored `Transaction` object) will compute H2 and fail to match the committed H1, causing it to reject a valid block or report an incorrect hash. RPC endpoints that return transaction data will expose the wrong `resource_bounds` variant (`L1Gas` instead of `AllResources`). If the transaction is re-executed (blockifier reexecution, simulation, fee estimation), it will use `GasVectorComputationMode::NoL2Gas` instead of `GasVectorComputationMode::All`, producing different gas accounting and potentially different execution results than the sequencer produced. [7](#0-6) 

### Likelihood Explanation

The trigger requires submitting a V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway's stateless validator only requires at least one bound to be non-zero, so such a transaction passes admission. The stateful validator's L2 gas price threshold check only fires when `l2_gas.max_price_per_unit` is below threshold — a transaction with `l2_gas.max_amount = 0` and `l2_gas.max_price_per_unit = 0` passes that check. Any user can craft this transaction without special privileges. [8](#0-7) 

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit variant tag in the protobuf schema, or add a dedicated boolean/enum field to the `ResourceBounds` protobuf message that records whether the transaction is a legacy `L1Gas` transaction. Until the schema is updated, the deserializer should default to `AllResources` when all three fields are present (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (the pre-0.13.3 wire format):

```rust
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

This preserves backward compatibility with 0.13.2 nodes (which omit `l1_data_gas`) while correctly reconstructing post-0.13.3 transactions that happen to have zero L2/L1DataGas values.

### Proof of Concept

1. Craft an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ZERO, l1_data_gas: ZERO }`.
2. Submit to the gateway. It is accepted; the converter computes hash H1 using `get_invoke_transaction_v3_hash` with `AllResources` → 4-element `tip_resource_bounds_hash`.
3. Serialize the resulting `InvokeTransactionV3` to `protobuf::InvokeV3` via `From<ValidResourceBounds> for protobuf::ResourceBounds`. Inspect the wire bytes: `l1_data_gas` field is present but zero.
4. Deserialize back via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true → result is `L1Gas(l1_gas)`.
5. Recompute the transaction hash from the deserialized object. The `tip_resource_bounds_hash` now uses only 3 elements → hash H2.
6. Assert H1 ≠ H2. The committed hash and the recomputed hash diverge, breaking `validate_transaction_hash` and any downstream consumer that recomputes the hash from stored transaction data. [9](#0-8)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

        }
