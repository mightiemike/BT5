### What `migrated_compiled_classes` means

During the Starknet 0.14.x migration, a class's compiled class hash can be updated **without** a new declare transaction. The entry `(class_hash, new_compiled_class_hash)` in `migrated_compiled_classes` tells the execution layer: "for this class hash, use this new compiled class hash." The TODO comment confirms this is a live, production-relevant field: [5](#0-4) 

The central sync path **does** handle `migrated_compiled_classes` correctly — `extract_compiled_classes_from_state_diff` chains over it: [6](#0-5) 

But the RPC pending path does not.

### Verdict

### Title
`migrated_compiled_classes` silently dropped in `client_pending_data_to_execution_pending_data`, causing wrong compiled class selected for pending-block RPC execution — (`crates/apollo_rpc/src/pending.rs`)

### Summary
`client_pending_data_to_execution_pending_data` omits `migrated_compiled_classes` when constructing `ExecutionPendingData`. Any class whose compiled class hash was updated via migration in the pending state diff will be executed with the stale compiled class hash during all pending-block RPC operations (`starknet_call`, `starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_traceTransaction` on pending).

### Finding Description
`ClientPendingData` carries `state_update.state_diff.migrated_compiled_classes: Vec<MigratedCompiledClassHashEntry>` deserialized from feeder-gateway JSON. The conversion function at `crates/apollo_rpc/src/pending.rs:5-24` constructs `ExecutionPendingData` by enumerating six state-diff fields explicitly; `migrated_compiled_classes` is not among them and `ExecutionPendingData` has no corresponding field. The blockifier therefore resolves the compiled class hash for any migrated class from the committed (pre-migration) state, selecting the old CASM/native artifact.

### Impact Explanation
**Critical — Wrong compiled class selected for execution.** During Starknet 0.14.x, `migrated_compiled_classes` entries change which compiled class hash is authoritative for a given class hash. Dropping them means the blockifier picks the pre-migration compiled class for pending-block execution. Every RPC call that simulates, estimates, or traces a transaction touching a migrated class in the pending block returns results computed against the wrong contract code.

### Likelihood Explanation
The Starknet network actively emits `migrated_compiled_classes` entries in pending state updates during the 0.14.x migration window. The feeder-gateway JSON path is the normal production data source; no attacker control is required — the legitimate network data triggers the bug. The `#[serde(default)]` annotation means the field is silently empty when absent, so the bug is dormant until the network produces a non-empty list, at which point it fires unconditionally.

### Recommendation
Add `migrated_compiled_classes` to `ExecutionPendingData` and forward it in `client_pending_data_to_execution_pending_data`. The blockifier's pending state overlay must apply the migrated compiled class hash mappings before resolving class code, mirroring the treatment already present in `extract_compiled_classes_from_state_diff` for the committed-block path.

fn extract_compiled_classes_from_state_diff(
    state_diff: &StateDiff,
) -> Vec<(ClassHash, CompiledClassHash)> {
    state_diff
        .declared_classes
        .iter()
        .map(|(class_hash, (compiled_class_hash, _class))| (*class_hash, *compiled_class_hash))
        .chain(
            state_diff
                .migrated_compiled_classes
                .iter()
                .map(|(class_hash, compiled_class_hash)| (*class_hash, *compiled_class_hash)),
        )
        .collect()
}
