### Title
Overly Strict Empty Merkle Proof Guard Causes Unnecessary Revert for Single-Transaction Blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` unconditionally rejects empty `merkle_proof` slices. For a Bitcoin-family block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty — the transaction hash *is* the merkle root. Because `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its own coinbase-proof check passes, any caller attempting to verify a transaction in a single-transaction block receives a NEAR panic instead of a valid `true` result.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` contains an unconditional guard:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This fires before `compute_root_from_merkle_proof` is ever called. However, `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` handles an empty proof correctly — the `for` loop over `merkle_proof` simply does not execute, and the function returns `transaction_hash` unchanged:

```rust
for proof_hash in merkle_proof {   // never entered when proof is empty
    ...
}
current_hash   // == transaction_hash
``` [2](#0-1) 

For a block with exactly one transaction, `merkle_root == coinbase_tx_id`, so `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == merkle_root` is mathematically correct and the proof is valid.

`verify_transaction_inclusion_v2` — the production, non-deprecated entry point — calls `verify_transaction_inclusion` after its own checks:

```rust
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
        ...
    );
    // coinbase proof validated here
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())   // panics if merkle_proof is empty
}
``` [3](#0-2) 

When both `merkle_proof` and `coinbase_merkle_proof` are empty (the correct inputs for a single-tx block):

1. Length check: `0 == 0` → passes.
2. Coinbase proof check: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == merkle_root` → passes (since `merkle_root == coinbase_tx_id`).
3. `verify_transaction_inclusion` is called → `require!(!args.merkle_proof.is_empty())` → **panics**.

The guard is the direct analog of the `MarketUtils.validatePoolAmountForDeposit` call in the reference report: a validation that fires in a condition where the underlying operation is valid and should succeed.

---

### Impact Explanation

Any consumer contract calling `verify_transaction_inclusion_v2` to verify a transaction in a single-transaction block will receive a NEAR panic (contract revert) instead of `true`. Cross-chain applications — atomic swaps, bridged-asset releases, or any protocol gating on Bitcoin transaction inclusion — are permanently blocked for this class of valid Bitcoin blocks. The contract's core verification guarantee is broken for a reachable, real-world input.

---

### Likelihood Explanation

Single-transaction blocks (coinbase-only) occur on Bitcoin mainnet and all supported chains, particularly during low-fee periods or early chain history. The genesis block and many early Bitcoin blocks are single-transaction blocks. Any relayer that has submitted such a block to the contract creates a permanently unverifiable state for coinbase transactions in those blocks. The trigger requires no privilege — any unprivileged NEAR account can call `verify_transaction_inclusion_v2`.

---

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`. The downstream call to `compute_root_from_merkle_proof` already handles empty proofs correctly and will return the transaction hash itself, which is the correct merkle root for a single-transaction block. [1](#0-0) 

---

### Proof of Concept

1. Submit a Bitcoin block header for a block containing only the coinbase transaction (e.g., any early mainnet block where `merkle_root == coinbase_txid`).
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_txid`
   - `tx_block_blockhash = block_hash` (must be on the stored mainchain)
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_txid`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. The length check passes (`0 == 0`). The coinbase proof check passes (`coinbase_txid == merkle_root`). Execution reaches `verify_transaction_inclusion`, which panics: `"Merkle proof is empty"`.
4. The call reverts despite the proof being mathematically valid and complete. [3](#0-2) [4](#0-3)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
