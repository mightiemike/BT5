### Title
Linked Signer Can Replace Itself via `LinkSigner` — Unauthorized Subaccount Takeover - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in the fast-path (`processTransactionImpl`) is validated with `allowLinkedSigner = true`, meaning the currently linked signer — a limited delegate — can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. Only the subaccount owner should be permitted to change the linked signer. This is the direct Nado analog of the EIP-721 `approve` bug: a limited delegate (approved address / linked signer) can perform a delegation-mutation operation that is reserved for the full owner (operator / subaccount owner).

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with the last argument hardcoded to `true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which accepts a signature from either the subaccount owner **or** the currently registered linked signer:

```solidity
// Verifier.sol lines 291–303
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // <-- linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

By contrast, the slow-mode path for `LinkSigner` correctly restricts the operation to the subaccount owner by calling `validateSender`, which requires the Ethereum address of the caller to match the subaccount owner address:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only check
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The slow-mode path enforces owner-only access; the fast-path does not. The two paths are inconsistent on the most sensitive operation in the linked-signer system.

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Craft a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address.
2. Sign it with the currently linked signer key (which the attacker controls).
3. Submit it to the sequencer. The sequencer accepts it because the signature is valid.
4. `linkedSigners[subaccount]` is now set to the attacker's address.
5. The attacker's address is now the linked signer and can sign `WithdrawCollateral` (also validated with `allowLinkedSigner = true`) to drain all collateral from the subaccount.

The original owner loses control of their subaccount's signing authority and all funds can be withdrawn by the attacker. This is a direct, concrete asset-theft path. [4](#0-3) 

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets or API keys used for automated trading — a higher-risk key class than the subaccount owner's cold wallet. Phishing, key leakage from a trading bot, or a malicious third-party integration are all realistic compromise vectors. Once the linked signer key is compromised, the attacker can escalate from "can trade on behalf of the account" to "can permanently replace the signer and drain funds" — a privilege escalation that the protocol's own slow-mode path explicitly prevents.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. The linked signer is a limited delegate and must not be permitted to mutate its own delegation. Only the subaccount owner (the address embedded in `sender`) should be able to change the linked signer.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // owner-only: linked signer must not re-link
);
``` [5](#0-4) 

---

### Proof of Concept

1. Alice owns subaccount `S` and has linked her hot-wallet `H` as the linked signer (`linkedSigners[S] = H`).
2. Attacker Eve compromises `H` (e.g., via phishing or a leaked API key).
3. Eve constructs `LinkSigner { sender: S, signer: Eve_address, nonce: current_nonce }`.
4. Eve signs the EIP-712 digest of this transaction with `H`.
5. Eve submits the signed transaction to the sequencer via the fast-path.
6. `validateSignedTx(..., true)` accepts the signature because `recovered == H == linkedSigners[S]`.
7. `linkedSigners[S]` is now set to `Eve_address`.
8. Eve signs a `WithdrawCollateral { sender: S, productId: QUOTE, amount: full_balance }` with `Eve_address`.
9. The sequencer processes it; `validateSignedTx(..., true)` accepts it; all collateral is sent to Eve.
10. Alice's subaccount is drained. Alice cannot recover control via the fast-path because Eve controls the linked signer slot. [6](#0-5)
