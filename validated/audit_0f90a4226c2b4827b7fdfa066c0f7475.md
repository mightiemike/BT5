### Title
Deprecated `verify_transaction_inclusion` Remains a Live Callable Endpoint and Enables Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a publicly reachable NEAR contract method. It does not validate that the caller-supplied `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply an internal node hash as `tx_id` with a shortened proof path, causing the function to return `true` for a Bitcoin transaction that was never included in any block.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated]` but remains a live public method on the contract: [1](#0-0) 

In Rust, `#[deprecated]` only emits a compiler warning at call sites in Rust source. It does **not** remove the function from the compiled WASM binary and does **not** prevent invocation via NEAR RPC. Any unprivileged NEAR account can call it at any time (subject only to the `#[pause]` gate, which is inactive by default).

The function's verification logic is: [2](#0-1) 

It computes `compute_root_from_merkle_proof(args.tx_id, args.tx_index, &args.merkle_proof)` and checks equality with `header.block_header.merkle_root`. There is **no check** that `args.tx_id` is a leaf-level hash (an actual transaction) rather than an internal Merkle tree node.

The `compute_root_from_merkle_proof` function in `merkle-tools` simply walks the proof path upward from whatever starting hash is given: [3](#0-2) 

If the attacker supplies an internal node N at height H from the leaves as `tx_id`, and provides the H-step proof path from N to the root, the computation reaches the correct root and the function returns `true`.

The v2 replacement `verify_transaction_inclusion_v2` correctly mitigates this by requiring a coinbase proof of equal length, forcing the proof depth to match the full tree depth: [4](#0-3) 

But the v1 function remains callable alongside it, with no such guard.

The code's own documentation acknowledges the broken invariant explicitly: [5](#0-4) 

---

### Impact Explanation

Any dApp or NEAR smart contract that calls `verify_transaction_inclusion` to gate asset releases, cross-chain operations, or any trust decision can be tricked into accepting a forged proof. The attacker can claim any internal Merkle node is a valid transaction, bypassing SPV verification entirely. The corrupted value is the boolean proof result returned to the caller — a `true` where the correct answer is `false`. This maps directly to the external report's class: an unrestricted execution path that corrupts a shared verification result consumed by downstream contracts.

---

### Likelihood Explanation

The attack requires no special privileges, no computational work, and no private keys. The attacker only needs the Merkle tree of any block already in the contract's mainchain — publicly available from any Bitcoin node or block explorer. The only practical barrier is that a downstream dApp must still be using the v1 function, which is a realistic scenario given the function was not removed and no on-chain enforcement prevents its use.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely. If backward compatibility is required, replace the function body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` so that any call reverts at runtime rather than returning a forged result. A Rust `#[deprecated]` attribute alone provides no runtime protection.

---

### Proof of Concept

1. Identify any block `B` in the contract's mainchain with at least 2 transactions (trivially satisfied by any real Bitcoin block).
2. Obtain the Merkle tree of `B` from a public Bitcoin node or block explorer.
3. Select any internal node `N` at height `H = 1` (the direct children of the root). Let `N = left_child` and `sibling = right_child`.
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = hash of `N`
   - `tx_block_blockhash` = hash of `B` (must be on the mainchain)
   - `tx_index` = `0`
   - `merkle_proof` = `[sibling]` (one-element proof)
   - `confirmations` = `1`
5. Inside the function, `compute_root_from_merkle_proof(N, 0, [sibling])` computes `SHA256d(N || sibling)` = `merkle_root` of `B`.
6. The equality check passes and the function returns `true`, even though `N` is not a real Bitcoin transaction. [6](#0-5) [3](#0-2)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
