### Title
AuxPoW Verification Bypassed When Block Has AuxPoW Flag Set But `aux_data = None` - (File: `contract/src/dogecoin.rs`)

### Summary

The Dogecoin `submit_block_header` function decides which PoW verification path to execute based on whether the **caller supplies** `aux_data`, rather than on whether the **block header itself** has the AuxPoW flag set in its `version` field. This is a direct structural analog to H-06: just as that bug took the ERC721 branch for a token that satisfied both interfaces, this code takes the standalone-PoW branch for a block that has the AuxPoW flag set, because the caller simply omits `aux_data`.

### Finding Description

In `contract/src/dogecoin.rs`, `submit_block_header` branches on the caller-supplied `Option<AuxData>`: [1](#0-0) 

```rust
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);

    if let Some(ref aux_data) = aux_data {
        self.check_aux(&block_header, aux_data);
    } else {
        let pow_hash = block_header.block_hash_pow();
        require!(
            U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            format!("block should have correct pow")
        );
    }
}
```

The AuxPoW flag check (`version & BLOCK_VERSION_AUXPOW != 0`) exists only **inside** `check_aux`: [2](#0-1) 

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
```

Because this check is gated behind `aux_data.is_some()`, it is never reached when `aux_data = None`. A caller can submit a block header with `version & 0x100 != 0` (AuxPoW flag set) while passing `aux_data = None`. The contract then falls into the `else` branch and checks the Dogecoin block's **own** SHA256d hash against the target â€” which is the wrong hash for an AuxPoW block. For AuxPoW blocks, the PoW proof lives in the parent Bitcoin block (`aux_data.parent_block`), not in the Dogecoin block header itself.

The public entry point is `submit_blocks`, which accepts `Vec<BlockHeader>` where `BlockHeader = (Header, Option<AuxData>)` for the Dogecoin build: [3](#0-2) 

```rust
#[cfg(feature = "dogecoin_header")]
pub type BlockHeader = (Header, Option<AuxData>);
```

The caller freely controls the `Option<AuxData>` field.

### Impact Explanation

An attacker who can call `submit_blocks` can craft a Dogecoin block header with:
- `version & 0x100 != 0` (AuxPoW flag set, as all real Dogecoin mainnet blocks since block 371337 have)
- A nonce chosen so that `SHA256d(header)` meets the declared `bits` target

Submitting this with `aux_data = None` causes the contract to accept the block without any AuxPoW verification. The light client's canonical chain (`mainchain_tip_blockhash`, `mainchain_height_to_header`, `mainchain_header_to_height`) is updated with a block that is not a valid Dogecoin block. This corrupts the canonical chain state and, consequently, any downstream `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` call that relies on that state. [4](#0-3) 

### Likelihood Explanation

The `submit_blocks` function carries a `#[trusted_relayer]` attribute. Whether any staked participant can become a relayer (permissionless staking) or whether the `RelayerManager` must approve each applicant affects the practical reachability. The role definition notes that `RelayerManager` may "reject applications," suggesting approval is required. However, the `bypass_roles` list includes `Role::UnrestrictedSubmitBlocks`, and the structural flaw exists regardless of access control: any relayer â€” honest or malicious â€” who submits a real Dogecoin mainnet AuxPoW block without its `aux_data` triggers the wrong verification path. The bug is also reachable during `init_genesis` via the `submit_blocks` field of `InitArgs`, which calls `submit_block_header` directly with `skip_pow_verification = true` for the initial blocks, but with `false` for subsequent ones. [5](#0-4) 

### Recommendation

Branch on the block header's own AuxPoW flag, not on the caller-supplied `Option<AuxData>`:

```rust
if block_header.version & BLOCK_VERSION_AUXPOW != 0 {
    // AuxPoW block: aux_data must be present and valid
    let aux = aux_data
        .as_ref()
        .unwrap_or_else(|| env::panic_str("AuxPoW block requires aux_data"));
    self.check_aux(&block_header, aux);
} else {
    // Standalone PoW block
    require!(aux_data.is_none(), "Non-AuxPoW block must not carry aux_data");
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

This mirrors the H-06 fix: check the entity's own declared property (AuxPoW flag in `version`) before deciding which verification path to take, rather than relying on caller-controlled input.

### Proof of Concept

1. Build the contract with `feature = "dogecoin"`.
2. Initialize with a valid genesis block.
3. Craft a `Header` with `version = 0x100` (AuxPoW flag set), valid `prev_block_hash`, and brute-force `nonce` until `SHA256d(header) <= target_from_bits(bits)`.
4. Call `submit_blocks` with `[(crafted_header, None)]`.
5. Observe that `submit_block_header` enters the `else` branch at line 182, checks the block's own hash (which was crafted to pass), and stores the block in `mainchain_height_to_header` and `headers_pool` â€” with no AuxPoW verification performed. [6](#0-5)

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
