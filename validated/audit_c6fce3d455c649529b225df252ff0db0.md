### Title
`DirectDepositV1.creditDeposit()` Permanently Bricked for USDT-Like Tokens Due to Missing Zero-Reset Before `approve()` — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens like USDT that forbid non-zero → non-zero allowance changes, a single silent-return path in `Endpoint.depositCollateralWithReferral()` leaves a residual non-zero allowance. Every subsequent call to `creditDeposit()` for that token then reverts permanently, bricking the DDA for that asset with no recovery path.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product tokens and, for each one with a non-zero balance, sets an approval and then calls the endpoint to deposit:

```solidity
// core/contracts/DirectDepositV1.sol : creditDeposit()
token.approve(address(endpoint), balance);          // line 92
endpoint.depositCollateralWithReferral(             // line 93
    subaccount, productId, uint128(balance), "-1"
);
``` [1](#0-0) 

`Endpoint.depositCollateralWithReferral()` contains a documented silent early-return path: if the deposit amount is below the minimum threshold, it returns without calling `handleDepositTransfer` and therefore without consuming any allowance:

```solidity
// core/contracts/Endpoint.sol : depositCollateralWithReferral()
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    return;
}
``` [2](#0-1) 

After this silent return, the allowance granted by `creditDeposit()` remains fully intact (non-zero). The next time `creditDeposit()` is called for the same token, `token.approve(address(endpoint), balance)` attempts a non-zero → non-zero allowance change, which reverts for USDT and any other token implementing the front-running protection guard.

The `endpoint` address is set in the constructor and stored as a plain state variable with no setter, making it effectively immutable for the lifetime of the deployed DDA:

```solidity
// core/contracts/DirectDepositV1.sol : constructor
endpoint = IIEndpoint(_endpoint);
``` [3](#0-2) 

There is no function in `DirectDepositV1` to reset the allowance independently, so once the residual allowance is set, the contract is permanently unable to credit further deposits of that token.

Notably, `ContractOwner.wrapVaultAsset()` in the same codebase correctly performs the zero-reset pattern before re-approving:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

This confirms the developers are aware of the pattern but did not apply it in `DirectDepositV1`.

---

### Impact Explanation

A `DirectDepositV1` instance handling a USDT-like token becomes permanently unable to credit deposits after the first silent-return event. All USDT sent to that DDA address afterward is permanently stranded — `creditDeposit()` reverts on the `approve` call, and there is no admin function to reset the allowance or change the endpoint. The only recovery is deploying a new DDA, which requires `ContractOwner.createDirectDepositV1()` and migrating the subaccount mapping.

---

### Likelihood Explanation

The silent-return path in `depositCollateralWithReferral` is explicitly designed to fire when a deposit is below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`. This is a normal operational condition — a user sending a small dust amount of USDT to the DDA before a larger deposit is enough to trigger it. `creditDeposit()` has no access control and can be called by anyone, so an adversary can deliberately trigger the condition by sending a sub-minimum USDT amount to a victim's DDA and then calling `creditDeposit()`, permanently bricking it. [5](#0-4) 

---

### Recommendation

Reset the allowance to zero before setting it to the new value, matching the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
// In creditDeposit(), replace:
token.approve(address(endpoint), balance);

// With:
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use OpenZeppelin's `SafeERC20.forceApprove()` which handles this automatically.

---

### Proof of Concept

1. A `DirectDepositV1` DDA is deployed for a subaccount, with USDT as one of the supported spot tokens.
2. An attacker sends 1 wei of USDT to the DDA address (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. The attacker calls `creditDeposit()` (no access control). The loop reaches the USDT product: `token.approve(endpoint, 1)` succeeds (allowance was 0 → 1). `depositCollateralWithReferral` silently returns because `1 wei` is below the minimum. Allowance remains at 1.
4. The legitimate user later sends 1000 USDT to the DDA and calls `creditDeposit()`. The loop reaches USDT: `token.approve(endpoint, 1000e6)` **reverts** because USDT's allowance is currently 1 (non-zero → non-zero disallowed).
5. The DDA is permanently bricked for USDT. All 1000 USDT is stranded. [5](#0-4) [6](#0-5)
