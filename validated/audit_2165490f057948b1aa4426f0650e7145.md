### Title
Unbounded `NlpLockedBalanceQueue` Growth Permanently Locks User NLP Tokens — (`core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.handleNlpLockedBalance` appends a new queue entry for every `mintNlp` call that occurs at a distinct oracle timestamp. `tryUnlockNlpBalance` drains the queue in a single unbounded `while` loop. Once the lock period expires for a large batch of entries, any call to `burnNlp` — which is forced to invoke `getNlpUnlockedBalance` → `tryUnlockNlpBalance` — exhausts the block gas limit, permanently locking the user's NLP tokens with no recovery path.

---

### Finding Description

**Queue growth path**

`SpotEngine.handleNlpLockedBalance` is called from `SpotEngine.updateBalance` whenever `productId == NLP_PRODUCT_ID` and `amountDelta > 0`: [1](#0-0) 

A new `NlpLockedBalance` entry is pushed to the per-subaccount queue only when the current oracle time differs from the last entry's `unlockedAt - NLP_LOCK_PERIOD`. Because the sequencer advances the oracle time with every `SpotTick`, each `mintNlp` submitted at a different oracle timestamp appends one entry. There is no cap on `balanceCount`.

**Unbounded drain loop**

`tryUnlockNlpBalance` processes every unlocked entry in a single `while` loop with no iteration limit: [2](#0-1) 

Each iteration performs multiple cold/warm `SLOAD`s and `SSTORE`s (delete + counter update). At ~10 000–15 000 gas per iteration, a queue of ≈2 000–3 000 entries exhausts a 30 M gas block.

**Forced call in the burn path**

`Clearinghouse.burnNlp` unconditionally calls `spotEngine.getNlpUnlockedBalance`, which calls `tryUnlockNlpBalance`: [3](#0-2) [4](#0-3) 

There is no alternative burn path and no way to process the queue in batches. Once the loop exceeds the gas limit, every subsequent `burnNlp` call reverts.

---

### Impact Explanation

A user whose `NlpLockedBalanceQueue` has grown beyond the gas-safe threshold cannot burn their NLP tokens. Because `burnNlp` is the only mechanism to redeem NLP for quote collateral, and no admin recovery function exists, the user's NLP balance is permanently frozen. The `tryUnlockNlpBalance` function is `public` but its single-pass `while` loop cannot be batched across multiple transactions, so there is no self-rescue path.

---

### Likelihood Explanation

A user must submit ≈2 000–3 000 `MintNlp` transactions at distinct oracle timestamps within a single lock period. Each transaction costs `HEALTHCHECK_FEE` in quote. The sequencer advances the oracle time with every `SpotTick`, so the oracle time changes frequently in normal operation. An active liquidity provider automating small recurring mints (e.g., via `DirectDepositV1`) could reach this threshold without realising the risk. The trigger is entirely within the user's own signed transaction stream — no admin or sequencer cooperation beyond normal operation is required.

---

### Recommendation

1. **Cap queue depth**: Enforce a `MAX_NLP_LOCK_ENTRIES` constant in `handleNlpLockedBalance` and revert or merge entries when the cap is reached.
2. **Bounded drain**: Refactor `tryUnlockNlpBalance` to accept a `maxEntries` parameter so callers can drain the queue incrementally across multiple transactions before invoking `burnNlp`.
3. **Merge on mint**: Merge all same-lock-period entries into a single slot regardless of oracle time granularity, reducing queue growth to at most one entry per lock period.

---

### Proof of Concept

1. User submits N `MintNlp` transactions (N ≈ 3 000), each processed by the sequencer at a distinct oracle timestamp. Each call appends one entry to `nlpLockedBalanceQueues[user]` via `handleNlpLockedBalance`. [5](#0-4) 

2. After `NLP_LOCK_PERIOD` elapses, all N entries satisfy `unlockedAt <= getOracleTime()`.

3. User submits a `BurnNlp` transaction. The sequencer processes it via `EndpointTx.processTransactionImpl`: [6](#0-5) 

4. `Clearinghouse.burnNlp` calls `spotEngine.getNlpUnlockedBalance(txn.sender)`: [3](#0-2) 

5. `getNlpUnlockedBalance` calls `tryUnlockNlpBalance`, which enters the `while` loop and iterates N times, exhausting the block gas limit. The transaction reverts. [7](#0-6) 

6. Every subsequent `burnNlp` call for this subaccount reverts identically. The user's NLP tokens are permanently locked.
