### Title
Unprivileged Caller Can Destroy Mainchain Block Headers via Unrestricted `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary
`run_mainchain_gc` is a public, state-mutating function that permanently deletes block headers from the contract's canonical chain state. It carries no caller-identity restriction when the contract is unpaused, so any unprivileged NEAR account can invoke it with an arbitrarily large `batch_size` and immediately purge all headers that exceed the GC threshold. This is a direct analog to the reported "minter burns tokens of other accounts" class: a destructive operation on shared state submitted by legitimate relayers is reachable by an account that has no authority over that state.

### Finding Description
`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`.

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` attribute gates the function when the contract is **paused**; it imposes no restriction on the caller when the contract is **live**. There is no `#[trusted_relayer]`, no ACL role check, and no `#[private]` guard. Any NEAR account — including one with no stake, no role, and no prior interaction with the contract — can call this function freely.

The function permanently removes entries from three storage structures:

```rust
self.remove_block_header(blockhash);          // removes from headers_pool + mainchain_header_to_height
self.mainchain_height_to_header.remove(&height);
```

and then advances `mainchain_initial_blockhash` to the new oldest surviving block. These deletions are irreversible.

The `batch_size` parameter is intended as a gas-bounding rate-limiter; `submit_blocks` passes `num_of_headers` (the count of headers just submitted) to keep each GC pass small. An unprivileged caller can pass `u64::MAX`, causing the function to compute:

```rust
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

which resolves to `total_amount_to_remove` — the entire backlog of headers past the threshold — and deletes them all in one transaction.

### Impact Explanation
The deleted headers are the inputs to `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`. Both functions look up the target block in `headers_pool` and `mainchain_header_to_height`:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
```

```rust
let header = self
    .headers_pool
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));
```

Once a header is GC'd by the attacker, every downstream consumer (bridge contracts, cross-chain applications) that calls either verification function for a transaction in that block will receive a hard panic. SPV proofs for those blocks are permanently invalidated. The attacker does not need to forge any data; they simply accelerate and maximize the GC pass that the system would otherwise apply gradually and in proportion to new block submissions.

### Likelihood Explanation
The entry path requires only a standard NEAR transaction to a live, unpaused contract. No role, no stake, no prior interaction is needed. The precondition — that the mainchain holds more than `gc_threshold` headers — is the normal operating state of a long-running deployment (the recommended `gc_threshold` is 52,704 blocks, approximately one year of Bitcoin production). Any observer of the NEAR chain can detect when this condition is met and immediately call `run_mainchain_gc(u64::MAX)`.

### Recommendation
Add an explicit caller-identity restriction to `run_mainchain_gc` that mirrors the guard already applied to `submit_blocks`. The simplest correct fix is to require the caller to hold a designated role (e.g., `UnrestrictedRunGC`, `DAO`, or a new `GCManager` role) before the function body executes. Alternatively, restrict the function to internal calls only (`#[private]`) and expose GC exclusively through `submit_blocks`, which is already gated by `#[trusted_relayer]`. The `batch_size` parameter should also be bounded by a contract-level maximum to prevent a single call from removing the entire eligible backlog regardless of who the caller is.

### Proof of Concept
1. Deploy the contract with `gc_threshold = 52704` (recommended production value).
2. Allow the relayer to submit blocks until `mainchain_size > gc_threshold` (normal operating condition).
3. From any unprivileged NEAR account `attacker.near`, call:
   ```
   run_mainchain_gc({ "batch_size": 18446744073709551615 })
   ```
4. The function computes `total_amount_to_remove = mainchain_size - gc_threshold` and, because `u64::MAX >= total_amount_to_remove`, sets `selected_amount_to_remove = total_amount_to_remove`.
5. All headers from `mainchain_initial_blockhash` up to `mainchain_tip_blockhash - gc_threshold` are deleted from `headers_pool` and `mainchain_height_to_header`.
6. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` referencing one of the deleted blocks panics with `"block does not belong to the current main chain"` or `"cannot find requested transaction block"`, permanently breaking SPV proof verification for those blocks.
