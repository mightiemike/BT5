### Title
NLP Burn Fee Not Extracted from Pool — Remaining NLP Holders Capture Protocol Fee Revenue - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.burnNlp()`, the computed `burnFee` is deducted from the amount the burner receives but is never routed to a fee account. It remains inside the NLP pool subaccounts, inflating the pool's quote balance and therefore the NLP oracle price. Every subsequent burner benefits from the accumulated fees of prior burners, paying a lower effective fee. The protocol collects nothing from the burn fee.

---

### Finding Description

`burnNlp` computes a fee and subtracts it from `quoteAmount` before paying the user:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [1](#0-0) 

The rebalance validation then enforces that NLP pool subaccounts are reduced by exactly `quoteAmount` (post-fee):

```solidity
_validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
``` [2](#0-1) 

`_validateNlpRebalance` only checks that `sum(nlpPoolRebalanceX18) == deltaQuoteAmount`, where `deltaQuoteAmount = -quoteAmount` (after fee). The `burnFee` portion is never subtracted from any pool subaccount and is never credited to `FEES_ACCOUNT`, `X_ACCOUNT`, or any treasury. [3](#0-2) 

Compare this with how every other fee type is handled: trading fees are accumulated in `collectedFees` and flushed to `X_ACCOUNT` via `dumpFees()`; interest fees are credited to `FEES_ACCOUNT` in `SpotEngineState`; sequencer fees are claimed via `claimSequencerFees()`. The `burnFee` has no equivalent routing. [4](#0-3) 

Because the fee stays in the NLP pool subaccounts, the pool's total quote balance is higher than it should be after a burn. The sequencer sets `oraclePriceX18` to reflect the pool's actual value (pool quote balance / NLP supply). Accumulated uncollected fees therefore raise the NLP price, and every subsequent burner redeems at that inflated price, capturing a portion of the fees paid by earlier burners.

---

### Impact Explanation

**Impact: Medium**

The protocol never receives the `burnFee` revenue. Instead, it is redistributed to remaining NLP holders through a higher NLP oracle price. A user who burns NLP after others have already burned pays a lower effective fee — in the limit (many prior burners, one late burner), the last burner's effective fee approaches zero. This is a direct, quantifiable loss of protocol fee revenue on every NLP burn.

The corrupted state delta: after each `burnNlp`, the NLP pool subaccounts hold `burnFee` more quote than they should, and `FEES_ACCOUNT` / `X_ACCOUNT` hold `burnFee` less than they should.

---

### Likelihood Explanation

**Likelihood: High**

NLP burns are a normal, user-initiated operation. Any NLP holder can sign a `BurnNlp` transaction and submit it to the sequencer. No special privilege is required. The bug is triggered on every single burn — it is not an edge case. The fee leakage is proportional to burn volume and compounds over time. [5](#0-4) 

The entry path: user signs `BurnNlp` → sequencer submits to `Endpoint` → `EndpointTx` dispatches to `Clearinghouse.burnNlp()`. [6](#0-5) 

---

### Recommendation

After computing `burnFee`, credit it to `FEES_ACCOUNT` (or `X_ACCOUNT`) via `spotEngine.updateBalance`, and include the fee in the rebalance delta so it is extracted from the NLP pool subaccounts:

```solidity
// Reduce pool by full gross amount (quoteAmount + burnFee)
_validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -(quoteAmount + burnFee));

// Credit fee to protocol fee account
spotEngine.updateBalance(QUOTE_PRODUCT_ID, FEES_ACCOUNT, burnFee);
```

This mirrors the pattern used for interest fees in `SpotEngineState` and trading fees in `OffchainExchange`. [7](#0-6) 

---

### Proof of Concept

Setup: two users each hold 100 NLP. Pool has 200 USDC. NLP price = 1 USDC. Burn fee = 0.1% (1/1000).

**Step 1 — User A burns 100 NLP:**
- `quoteAmount = 100 * 1 = 100 USDC`
- `burnFee = max(1e18, 100/1000) = 0.1 USDC`
- User A receives `100 - 0.1 = 99.9 USDC`
- Pool is reduced by 99.9 USDC → pool balance = `200 - 99.9 = 100.1 USDC`
- `burnFee = 0.1 USDC` stays in pool (not extracted)
- NLP supply = 100. New NLP price = `100.1 / 100 = 1.001 USDC`

**Step 2 — User B burns 100 NLP:**
- `quoteAmount = 100 * 1.001 = 100.1 USDC`
- `burnFee = 0.1001 USDC`
- User B receives `100.1 - 0.1001 = 99.9999 USDC`

**Result:**
- User A paid effective fee: `0.1 / 100 = 0.10%` ✓
- User B paid effective fee: `0.0001 / 100 = 0.0001%` — nearly zero
- Protocol collected: **0 USDC** from either burn
- User B captured `0.0999 USDC` of User A's fee [8](#0-7)
