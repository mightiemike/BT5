Looking at the `x/nft-transfer` module, specifically the `refundPacketToken` function in `packet.go`, there is a concrete analog to the external report's escrow/refund mismatch vulnerability class.

---

### Title
Sink-Chain NFT Refund Mints to Undefined `receiver` Instead of `sender`, Permanently Locking User NFTs on Failed/Timed-Out IBC Packets — (`File: x/nft-transfer/keeper/packet.go`)

---

### Summary

When a user sends an IBC NFT back toward its origin chain (sink-chain direction), the NFT is burned on the sending chain. If the IBC packet subsequently fails (error acknowledgement) or times out, `refundPacketToken` is invoked to restore the user's NFT. However, in the sink-chain branch of `refundPacketToken`, the code passes the undefined variable `receiver` as the NFT owner to `MintNFT` instead of the correctly-derived `sender`. The comment in the code itself reads *"mint voucher back to sender"*, confirming the intent, but the implementation uses the wrong variable. The original sender's NFT is never restored.

---

### Finding Description

**Send path** (`createOutgoingPacket`): When `isAwayFromOrigin` is `false` (sink chain — the NFT is being returned toward its origin), the NFT is burned: [1](#0-0) 

**Refund path** (`refundPacketToken`): On a failed or timed-out packet, the function correctly derives `sender` from `data.Sender`, but in the `else` branch (sink-chain case) it calls `MintNFT` with `receiver` — a variable that is never declared anywhere in the function scope: [2](#0-1) 

Specifically, line 45 reads:
```go
if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver); err != nil {
```

`sender` is the only address variable defined in the function (line 22). `receiver` is not defined in the function scope. The comment on line 43 explicitly states *"mint voucher back to sender"*, confirming `receiver` is a bug. The result is either a compile-time failure (making the entire module non-functional) or, if `receiver` resolves to a package-level zero-value `sdk.AccAddress`, the NFT is minted to the zero/null address — permanently destroying the user's asset with no recovery path.

The two IBC callbacks that invoke this broken path are: [3](#0-2) [4](#0-3) 

Both `OnAcknowledgementPacket` (error branch) and `OnTimeoutPacket` unconditionally delegate to `refundPacketToken`, so both failure modes are affected.

---

### Impact Explanation

A user who sends an IBC-received NFT back toward its origin chain (sink-chain direction) has their NFT burned at send time. If the packet fails or times out — both normal IBC operational events — the refund call mints the NFT to the wrong address. The original sender permanently loses the NFT with no recovery mechanism. This is a direct analog to the external report: assets are moved out of user custody at initiation, and the failure path does not restore them.

**Impact: Medium** — User NFT is permanently lost; no admin or user action can recover it after the broken refund executes.

---

### Likelihood Explanation

IBC packet timeouts and error acknowledgements are routine events on any live IBC-connected chain (relayer downtime, counterparty chain congestion, misconfigured timeout heights). Any user performing a sink-chain NFT transfer (returning an IBC NFT to its origin) is exposed. The trigger requires no special privileges — a standard `MsgTransfer` from any NFT owner suffices.

**Likelihood: Medium** — The sink-chain direction is a normal user operation; timeouts/failures are expected in production IBC environments.

---

### Recommendation

In `refundPacketToken`, replace `receiver` with `sender` in the sink-chain (`else`) branch:

```go
} else {
    // we are sink chain, mint voucher back to sender
    for i, tokenID := range data.TokenIds {
        if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, sender); err != nil {
            return err
        }
    }
}
```

This matches the stated intent in the comment and mirrors the `isAwayFromOrigin` branch which correctly uses `sender`.

---

### Proof of Concept

1. Chain A holds NFT `nftClass/token1` natively.
2. User transfers it to Chain B via IBC → Chain B now holds `ibc/<hash>/token1`; Chain A escrows the original.
3. User on Chain B calls `MsgTransfer` to send `ibc/<hash>/token1` back to Chain A (sink-chain direction).
4. `createOutgoingPacket` burns `ibc/<hash>/token1` on Chain B (`BurnNFTUnverified`, line 114).
5. The IBC packet times out (relayer offline, or timeout height reached).
6. IBC core calls `OnTimeoutPacket` → `refundPacketToken`.
7. `isAwayFromOrigin` is `false` (sink chain), so the `else` branch executes.
8. `MintNFT(..., receiver)` is called with undefined `receiver` — either compile error or zero-address mint.
9. The user's NFT is gone. No further message type exists to recover it. [5](#0-4) [6](#0-5)

func (k Keeper) refundPacketToken(ctx sdk.Context, packet channeltypes.Packet, data types.NonFungibleTokenPacketData) error {
	sender, err := sdk.AccAddressFromBech32(data.Sender)
	if err != nil {
		return err
	}

	classTrace := types.ParseClassTrace(data.ClassId)
	voucherClassID := classTrace.IBCClassID()

	isAwayFromOrigin := types.IsAwayFromOrigin(packet.GetSourcePort(),
		packet.GetSourceChannel(), data.ClassId)

	escrowAddress := types.GetEscrowAddress(packet.GetSourcePort(), packet.GetSourceChannel())

	if isAwayFromOrigin {
		// unescrow tokens back to the sender
		for _, tokenID := range data.TokenIds {
			if err := k.nftKeeper.TransferOwner(ctx, voucherClassID, tokenID, escrowAddress, sender); err != nil {
				return err
			}
		}
	} else {
		// we are sink chain, mint voucher back to sender

				return err
			}
		}
	}

	return nil
}

func (k Keeper) OnAcknowledgementPacket(ctx sdk.Context, channelVersion string, packet channeltypes.Packet, data types.NonFungibleTokenPacketData, ack channeltypes.Acknowledgement) error {
	switch ack.Response.(type) {
	case *channeltypes.Acknowledgement_Error:
		return k.refundPacketToken(ctx, packet, data)
	default:
		// the acknowledgement succeeded on the receiving chain so nothing
		// needs to be executed and no error needs to be returned
		return nil
	}
}
