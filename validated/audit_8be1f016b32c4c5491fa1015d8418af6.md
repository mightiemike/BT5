### Title
Signature Malleability in `op_secp256k1_verify` and `op_secp256r1_verify` via `hazmat::PrehashVerifier` — (File: `src/secp_ops.rs`)

---

### Summary

Both `op_secp256k1_verify` and `op_secp256r1_verify` in `src/secp_ops.rs` resolve `verify_prehash` through `signature::hazmat::PrehashVerifier`, which does **not** enforce low-s normalization. This is the direct analog of the `ecrecover` malleability class: for any valid compact ECDSA signature `(r, s)`, the signature `(r, n − s)` (where `n` is the curve group order) is also accepted as valid for the same message and public key. Two distinct 64-byte byte strings therefore both satisfy the opcode for the same inputs.

---

### Finding Description

In `src/secp_ops.rs`, the file imports:

```rust
use k256::ecdsa::{Signature as K1Signature, VerifyingKey as K1VerifyingKey};
use p256::ecdsa::signature::hazmat::PrehashVerifier;
use p256::ecdsa::{Signature as P1Signature, VerifyingKey as P1VerifyingKey};
``` [1](#0-0) 

The `hazmat::PrehashVerifier` trait is brought into scope from `p256::ecdsa::signature::hazmat`. Because `k256` and `p256` share the same underlying `signature` crate, this import makes `signature::hazmat::PrehashVerifier` the active `PrehashVerifier` trait for **both** `K1VerifyingKey` and `P1VerifyingKey`. The code compiles only because both verifier types implement `signature::hazmat::PrehashVerifier` (not the standard `signature::PrehashVerifier`), confirming that the `hazmat` path is taken at both call sites.

For `op_secp256k1_verify`:

```rust
let sig = K1Signature::from_slice(sig.as_ref())...?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
``` [2](#0-1) 

For `op_secp256r1_verify`:

```rust
let sig = P1Signature::from_slice(sig.as_ref())...?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
``` [3](#0-2) 

The `hazmat` module in the `signature` crate is explicitly documented as bypassing safety invariants, including low-s enforcement. Neither `K1Signature::from_slice` nor `P1Signature::from_slice` normalizes `s` to the lower half of the curve order. The `hazmat::PrehashVerifier::verify_prehash` implementation performs raw ECDSA verification without calling `normalize_s()`, so both `(r, s)` and `(r, n − s)` pass verification for the same `(pubkey, msg)` pair.

The `k256` dependency is pinned to `0.14.0-rc` (resolved to `0.14.0-rc.9` in `Cargo.lock`), and `p256` to the same release candidate series. [4](#0-3) 

---

### Impact Explanation

Any attacker-controlled CLVM program or spend solution that supplies a secp256k1 or secp256r1 signature to these opcodes can present two distinct 64-byte signatures that both return success (nil / cost) for the same `(pubkey, msg)` triple. Concretely:

1. **Spend-bundle malleability**: A valid spend bundle whose puzzle uses `secp256k1_verify` or `secp256r1_verify` can be re-signed with the high-s variant of the same signature. Both bundles are valid on-chain. Any system that tracks spend bundles by signature bytes (e.g., mempool deduplication keyed on solution hash, or off-chain indexers) will treat them as distinct events.

2. **CLVM program logic bypass**: A puzzle that embeds a specific expected signature as a quoted constant and then calls `secp256k1_verify` to confirm it cannot distinguish the canonical low-s form from the high-s variant. An attacker holding a valid low-s signature can derive the high-s form and satisfy the opcode, breaking any uniqueness assumption the puzzle author intended.

3. **Consensus-equivalence risk**: If a future hard fork or a parallel implementation enforces low-s (as Bitcoin Core has done since 2015 via BIP 146), nodes running this code and nodes running the stricter version will disagree on the validity of high-s-signed spends, producing a consensus split.

---

### Likelihood Explanation

The attack requires only arithmetic: given any valid compact signature `(r, s)`, compute `s' = n − s mod n` and submit `(r, s')`. No private key is needed. The attacker only needs to observe a valid spend bundle on the network. Both CLVM opcodes are reachable from attacker-controlled CLVM bytes passed to `run_program` via the Python API (`wheel/src/api.rs`) or directly through the Rust API. The `ENABLE_SECP_OPS` flag gates the single-byte opcode forms (64/65), but the 4-byte opcode forms (`0x13d61f00` / `0x1c3a8f00`) are always active. [5](#0-4) 

---

### Recommendation

1. After parsing, explicitly normalize `s` to the low half of the curve order before verification:

   ```rust
   // secp256k1
   let sig = K1Signature::from_slice(sig.as_ref())...?;
   let sig = sig.normalize_s().unwrap_or(sig);
   
   // secp256r1
   let sig = P1Signature::from_slice(sig.as_ref())...?;
   let sig = sig.normalize_s().unwrap_or(sig);
   ```

2. Alternatively, after normalization, reject signatures whose `s` was in the upper half (i.e., `normalize_s()` returned `Some(_)`), enforcing strict low-s-only acceptance.

3. Replace the `hazmat::PrehashVerifier` import with the standard `signature::PrehashVerifier` if a future version of `k256`/`p256` re-exposes it with low-s enforcement built in.

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
```
