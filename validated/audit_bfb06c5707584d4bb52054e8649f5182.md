### Title
Slow-Mode `LinkSigner` Silently Overwrites Fast-Path `LinkSigner` Due to Missing Nonce Check — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is handled by two separate code paths in `EndpointTx.sol`: a **fast path** (sequencer-submitted, nonce-gated) and a **slow-mode path** (user-submitted on-chain, no nonce). Because both paths write unconditionally to the same `linkedSigners[sender]` slot, a slow-mode `LinkSigner` queued before a fast-path `LinkSigner` will silently overwrite the fast-path result when it eventually executes after the mandatory 3-day delay. There is no cancellation mechanism for slow-mode transactions. The final `linkedSigners` state therefore depends on execution order, not on the user's most recent intent.

---

### Finding Description

**Fast-path `LinkSigner`** — `processTransactionImpl`, `EndpointTx.sol` lines 576–590:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,   // ← nonce consumed; strictly ordered
    transaction,
    signedTx.signature,
    true
);
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
``` [1](#0-0) 

**Slow-mode `LinkSigner`** — `processSlowModeTransactionImpl`, `EndpointTx.sol` lines 232–239:

```solidity
validateSender(txn.sender, sender);   // only checks msg.sender == subaccount owner
requireSubaccount(txn.sender);        // only checks subaccount is registered
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));  // ← no nonce
``` [2](#0-1) 

Both paths write to the same storage slot:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [3](#0-2) 

The slow-mode queue is strictly FIFO and non-cancellable. A slow-mode transaction submitted at time `t` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days) and can only be executed after that delay — either by the user calling `executeSlowModeTransaction()` or by the sequencer via `ExecuteSlowMode`. [4](#0-3) 

Because the slow-mode path carries **no nonce**, it has no awareness of any fast-path `LinkSigner` that may have already executed. When the slow-mode transaction fires, it unconditionally overwrites `linkedSigners[sender]` with its own (stale) signer address.

---

### Impact Explanation

The `linkedSigners` mapping is the sole authorization gate for session-key signing across the entire protocol. A linked signer can authorize:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drains subaccount collateral
- `LiquidateSubaccount` — triggers liquidation of other accounts
- `MatchOrders` — places and fills orders on behalf of the subaccount
- `TransferQuote` — moves quote balance to an arbitrary recipient
- `MintNlp` / `BurnNlp` — manages NLP positions [5](#0-4) 

If the slow-mode transaction restores a **compromised or revoked** signer key `A` after the user believed they had replaced it with key `B`, the holder of key `A` regains full signing authority over the subaccount. The user has no on-chain mechanism to cancel the pending slow-mode transaction once submitted.

---

### Likelihood Explanation

A user who relies on slow-mode for censorship resistance (e.g., submits `LinkSigner(signer=A)` via slow mode to ensure it lands even if the sequencer is unresponsive) and later updates their signer via the fast path (`LinkSigner(signer=B, nonce=N)`) will unknowingly leave a time-bomb in the slow-mode queue. After 3 days, the queue entry fires and silently reverts to `signer=A`. The user receives no on-chain event distinguishing a slow-mode execution from a fast-path one, so the reversion can go unnoticed — exactly the scenario Spearbit flagged in the reference report. The likelihood is **low-medium**: it requires a prior slow-mode `LinkSigner` submission, but the 3-day delay and lack of cancellation make the window for harm persistent.

---

### Recommendation

1. **Add a nonce check to the slow-mode `LinkSigner` path.** Mirror the fast-path by calling `validateNonce(txn.sender, txn.nonce)` before writing to `linkedSigners`. This ensures slow-mode and fast-path transactions are totally ordered per sender.

2. **Alternatively, add a slow-mode cancellation mechanism** that allows the subaccount owner to invalidate a pending slow-mode transaction by index before it executes.

3. **At minimum, document** that a pending slow-mode `LinkSigner` will overwrite any subsequent fast-path `LinkSigner` once the 3-day delay elapses, and that there is no cancellation path.

---

### Proof of Concept

```
t=0:   User submits slow-mode T1: LinkSigner(sender=S, signer=A)
       → queued at slowModeTxs[N], executableAt = t+3days
       → no nonce consumed

t=1:   User's key A is compromised.
       User submits fast-path T2: LinkSigner(sender=S, signer=B, nonce=0)
       → sequencer includes it; linkedSigners[S] = B
       → nonces[addr(S)] = 1

t=3days+1: Anyone calls executeSlowModeTransaction() (or sequencer includes ExecuteSlowMode)
       → processSlowModeTransactionImpl executes T1
       → validateSender passes (original msg.sender == addr(S))
       → requireSubaccount passes (S is registered)
       → linkedSigners[S] = A   ← overwrites B with no nonce guard

Result: Attacker holding key A now has full signing authority over subaccount S.
        User believes signer=B is active. No revert, no distinguishing event.
```

The root cause is the asymmetry between `processTransactionImpl` (line 581–587, nonce-gated) and `processSlowModeTransactionImpl` (line 237–239, no nonce), both writing to the same `linkedSigners` slot in `EndpointStorage`. [6](#0-5) [7](#0-6)
