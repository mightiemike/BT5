### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Re-Entrancy via ERC777 Token Callback — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the external token transfer (`handleWithdrawTransfer`) **before** updating the SpotEngine balance (`spotEngine.updateBalance`). With no `nonReentrant` guard anywhere in the codebase, an ERC777-compatible collateral token allows a recipient contract to re-enter `Endpoint.executeSlowModeTransaction` (a public, unguarded function) during the transfer callback. This causes a second `withdrawCollateral` to execute against the still-undeducted balance, allowing the attacker to withdraw more collateral than they hold in spot, with the resulting negative balance potentially passing the health check if the attacker holds sufficient perp-position health.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. `handleWithdrawTransfer(token, sendTo, amount, idx)` — external call chain that ultimately calls `token.safeTransfer(sendTo, amount)` via `BaseWithdrawPool.submitWithdrawal` → `BaseWithdrawPool.handleWithdrawTransfer`
2. `spotEngine.updateBalance(productId, sender, amountRealized)` — balance deduction
3. `spotEngine.assertUtilization(productId)`
4. `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)` — health check [1](#0-0) 

The token transfer to `sendTo` completes at step 1 before the balance is deducted at step 2. There is no `nonReentrant` modifier on any function in the protocol. [2](#0-1) 

`BaseWithdrawPool.submitWithdrawal` (called by `Clearinghouse.handleWithdrawTransfer`) calls `handleWithdrawTransfer(token, sendTo, amount)` which calls `token.safeTransfer(to, amount)` directly to the user-controlled `sendTo` address. [3](#0-2) [4](#0-3) 

`Endpoint.executeSlowModeTransaction` is a **public** function with no access control and no re-entrancy guard: [5](#0-4) 

No `nonReentrant` guard exists anywhere in the codebase — confirmed by absence of any `ReentrancyGuard` import or `nonReentrant` modifier.

---

### Impact Explanation

An attacker whose `sendTo` address is a contract with an ERC777 `tokensReceived` hook (or any token with transfer callbacks) can re-enter `Endpoint.executeSlowModeTransaction` during the token transfer. At that moment, `spotEngine.updateBalance` has not yet been called, so the SpotEngine still shows the pre-withdrawal balance. The re-entrant call processes a queued slow-mode `WithdrawCollateral` transaction, triggering a second `withdrawCollateral` against the stale balance. Both calls deduct `amount` from the balance. If the attacker holds a perp position providing sufficient health contribution, the final health check passes despite the resulting negative spot balance. The attacker withdraws `2 × amount` of collateral while only holding `amount` in spot, with the shortfall covered by perp health — a direct theft of protocol collateral.

---

### Likelihood Explanation

The attack requires:
- A supported collateral token with transfer callbacks (ERC777 or equivalent). The protocol's `IERC20Base` interface does not restrict token type, making this a realistic integration scenario.
- The attacker to have a queued slow-mode `WithdrawCollateral` transaction that is past its 3-day delay (`executableAt <= block.timestamp`).
- Sufficient perp-position health to pass the post-deduction health check.

All three conditions are attacker-controllable without any privileged access. The attacker submits the slow-mode tx themselves, waits the delay, and controls the `sendTo` contract. Likelihood is **medium** — it requires a specific token type but no admin or sequencer compromise.

---

### Recommendation

1. Add OpenZeppelin's `ReentrancyGuard` and apply `nonReentrant` to `Clearinghouse.withdrawCollateral`, `Endpoint.depositCollateralWithReferral`, `Endpoint.executeSlowModeTransaction`, and `BaseWithdrawPool.submitFastWithdrawal`.
2. Follow the checks-effects-interactions pattern in `withdrawCollateral`: call `spotEngine.updateBalance` and perform the health check **before** calling `handleWithdrawTransfer`.

---

### Proof of Concept

**Setup:**
- Collateral token is ERC777-compatible.
- Attacker's subaccount holds 100 USDC spot balance and a large long perp position providing +150 USDC of initial health.
- Attacker previously submitted a slow-mode `WithdrawCollateral` for 100 USDC; the 3-day delay has elapsed.

**Execution:**

1. Sequencer calls `submitTransactions` with a `WithdrawCollateral` tx for attacker (100 USDC, `idx=N`).
2. `Clearinghouse.withdrawCollateral` executes: `handleWithdrawTransfer` → `token.safeTransfer(withdrawPool, 100)` → `submitWithdrawal` → `token.safeTransfer(attacker_contract, 100)`.
3. ERC777 `tokensReceived` fires on attacker's contract. SpotEngine balance is still **100 USDC** (not yet deducted).
4. Attacker calls `Endpoint.executeSlowModeTransaction()` (public, no guard).
5. This processes the queued slow-mode `WithdrawCollateral` (100 USDC, `idx=M`) → `Clearinghouse.withdrawCollateral` called again.
6. SpotEngine balance is still 100 USDC → second 100 USDC transferred to attacker.
7. Inner call: `spotEngine.updateBalance(-100)` → balance = **0**. Health check: 0 spot + 150 perp = +150 ≥ 0. ✓ Passes.
8. Outer call resumes: `spotEngine.updateBalance(-100)` → balance = **-100**. Health check: -100 spot + 150 perp = +50 ≥ 0. ✓ Passes.

**Result:** Attacker received **200 USDC** from a 100 USDC spot balance. The 100 USDC deficit is absorbed by the protocol, backed by the perp position's health contribution — which can be closed separately, leaving the protocol insolvent by 100 USDC. [6](#0-5) [5](#0-4) [4](#0-3)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
