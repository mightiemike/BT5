### Title
Routed State-Sync Request Messages Permanently Bypass Per-Peer Rate Limiting, Enabling Global Throttle Exhaustion — (`File: chain/network/src/rate_limits/messages_limits.rs`)

---

### Summary

`T2MessageBody::StateHeaderRequest` and `T2MessageBody::StatePartRequest` — the routed variants of state-sync request messages — are hardcoded to return `None` from `get_key_and_token_cost()`, making them permanently invisible to the per-peer rate-limiting layer. Any connected peer can flood these messages without restriction. Each message unconditionally spawns a new async task in `PeerManagerActor` and forwards to the `StateRequestActor`, whose single global throttle (not per-peer) can be exhausted by the attacker, denying state-sync service to all legitimate syncing nodes.

---

### Finding Description

In `chain/network/src/rate_limits/messages_limits.rs`, the function `get_key_and_token_cost()` maps every `PeerMessage` variant to a `(RateLimitedPeerMessageKey, cost)` pair that the per-peer token-bucket checks before processing. Three routed T2 message types are explicitly excluded with `// TODO` comments:

```rust
T2MessageBody::StatePartRequest(_) => None, // TODO
T2MessageBody::StateHeaderRequest(_) => None, // TODO
T2MessageBody::StateRequestAck(_) => None,  // TODO
``` [1](#0-0) 

Because `is_allowed()` returns `true` whenever `get_key_and_token_cost` returns `None`, these messages are unconditionally passed through regardless of any operator-configured rate limits:

```rust
pub fn is_allowed(&mut self, message: &PeerMessage, now: Instant) -> bool {
    if let Some((key, cost)) = get_key_and_token_cost(message) {
        if let Some(bucket) = &mut self.buckets[key] {
            return bucket.acquire(cost, now);
        }
    }
    true  // always allowed when key is None
}
``` [2](#0-1) 

Note that the direct (non-routed) variants `PeerMessage::StateRequestHeader` and `PeerMessage::StateRequestPart` correctly return `Some(...)` and can be rate-limited. Only the routed T2 variants are excluded. [3](#0-2) 

When a routed `StateHeaderRequest` or `StatePartRequest` arrives, `network_state/mod.rs` dispatches a `Tier3Request` to the `PeerManagerActor`: [4](#0-3) 

The `PeerManagerActor` handler for `Tier3Request` unconditionally spawns a new async task per request with no concurrency cap:

```rust
self.handle.spawn("handle tier3 request", async move { ... });
``` [5](#0-4) 

Each spawned task calls `state.state_request_adapter.send_async(StateRequestHeader {...}).await` or the `StateRequestPart` equivalent, forwarding to `StateRequestActor`. [6](#0-5) 

`StateRequestActor` has a single **global** (not per-peer) sliding-window throttle:

```rust
fn throttle_state_sync_request(&self) -> bool {
    ...
    if timestamps.len() >= self.num_state_requests_per_throttle_period {
        return true;
    }
    timestamps.push_back(now);
    false
}
``` [7](#0-6) 

The throttle window and count are shared across all peers via a single `Arc<Mutex<VecDeque<Instant>>>`: [8](#0-7) 

The `standard_preset()` for rate limits only covers `EpochSyncRequest` and `EpochSyncResponse`; no default rate limit is configured for state-sync request messages: [9](#0-8) 

---

### Impact Explanation

**High.** An attacker who is any connected peer (Tier2) can send an unbounded stream of `PeerMessage::Routed` messages carrying `T2MessageBody::StateHeaderRequest` or `T2MessageBody::StatePartRequest`. These bypass the per-peer token-bucket entirely. Each message spawns an async task in `PeerManagerActor` and consumes one slot in the global `StateRequestActor` throttle. Once the global quota is exhausted, all legitimate state-sync requests from all other peers are rejected for the remainder of the throttle window. This denies state-sync service to nodes that are bootstrapping or recovering, which is a critical operation for network liveness.

---

### Likelihood Explanation

**High.** Any peer that completes a Tier2 handshake can send routed messages. No special privilege, validator key, or stake is required. The `RoutedMessageBody::StateHeaderRequest` and `RoutedMessageBody::StatePartRequest` variants are part of the stable Borsh-serialized protocol schema: [10](#0-9) 

The attack requires only a valid peer connection and knowledge of the message format, both of which are publicly documented.

---

### Recommendation

1. **Remove the `// TODO` stubs** in `get_key_and_token_cost()` and return `Some((StatePartRequest, 1))` and `Some((StateHeaderRequest, 1))` for the routed T2 variants, mirroring the treatment of the direct `PeerMessage::StateRequestHeader` and `PeerMessage::StateRequestPart` variants.

2. **Add default rate limits** for `StateRequestHeader` and `StatePartRequest` to `Config::standard_preset()`, analogous to the existing `EpochSyncRequest` limit (e.g., a small burst with a low refill rate per peer).

3. **Add a per-peer concurrency cap** on spawned `Tier3Request` handler tasks in `PeerManagerActor` (e.g., a per-peer semaphore or a bounded channel) so that a single peer cannot create an unbounded number of in-flight async tasks even if messages pass the token-bucket check.

---

### Proof of Concept

```
Attacker peer (any Tier2 connection):

for i in 0..N:
    send PeerMessage::Routed(
        body = T2MessageBody::StateHeaderRequest(StateHeaderRequest {
            shard_id: 0,
            sync_hash: <any valid CryptoHash>,
            addr: <attacker's addr>,
        })
    )

Effect:
  - get_key_and_token_cost() returns None for each message
    → per-peer token bucket is never consulted
  - network_state::process_incoming_routed() dispatches Tier3Request for each
  - PeerManagerActor::handle(Tier3Request) spawns N unbounded async tasks
  - StateRequestActor::throttle_state_sync_request() fills the global
    timestamps VecDeque to capacity within the throttle window
  - All subsequent StateRequestHeader/StateRequestPart from legitimate
    syncing nodes are rejected with Busy for the remainder of the window
```

The exact divergent value is the hardcoded `None` at lines 262–263 of `chain/network/src/rate_limits/messages_limits.rs` instead of `Some((StatePartRequest, 1))` / `Some((StateHeaderRequest, 1))`, which causes the per-peer `RateLimits::is_allowed()` gate to unconditionally return `true` for these message types.     fn throttle_state_sync_request(&self) -> bool {
        let mut timestamps = self.state_request_timestamps.lock();
        let now = self.clock.now();
        while let Some(&instant) = timestamps.front() {
            // Assume that time is linear. While in different threads there might be some small differences,
            // it should not matter in practice.
            if now - instant > self.throttle_period {
                timestamps.pop_front();
            } else {
                break;
            }
        }
        if timestamps.len() >= self.num_state_requests_per_throttle_period {
            return true;
        }
        timestamps.push_back(now);
        false
    }
