### Title
Incomplete Pause Guard: `addLiquidity` and `removeLiquidity` Lack `whenNotPaused`, Allowing Fund-Impacting Operations on a Paused Pool — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool` exposes a three-level pause mechanism (`pauseLevel` 0/1/2) and a `whenNotPaused` modifier, but only applies that modifier to `swap`. Both `addLiquidity` and `removeLiquidity` execute without any pause check. When the pool is paused in response to a security event, all liquidity operations remain fully open, defeating the purpose of the pause and exposing LP principal to loss.

---

### Finding Description

`MetricOmmPool` declares a `pauseLevel` state variable and a `whenNotPaused` modifier: [1](#0-0) [2](#0-1) [3](#0-2) 

`swap` is correctly guarded: [4](#0-3) 

But `addLiquidity` carries no `whenNotPaused`: [5](#0-4) 

And `removeLiquidity` carries no `whenNotPaused`: [6](#0-5) 

The factory's `setPause` can raise `pauseLevel` to 1 or 2: [7](#0-6) 

So the pause mechanism is reachable and functional — it simply does not cover the two remaining fund-moving entry points.

---

### Impact Explanation

When the pool is paused because of a security event (oracle compromise, accounting inconsistency, or an exploit in progress):

1. **Race-condition drain via `removeLiquidity`**: LPs who observe the pause event can immediately call `removeLiquidity` and withdraw their proportional share of bin tokens before the root cause is remediated. If the pool's internal accounting is already skewed (the reason for the pause), early withdrawers receive a disproportionate share, leaving later LPs with a shortfall — a direct loss of LP principal.

2. **Liquidity injection into a corrupted state via `addLiquidity`**: An attacker can call `addLiquidity` while the pool is paused and its bin accounting is inconsistent. Shares are minted against the corrupted `BinState`, giving the attacker an inflated share count. Once the pool is unpaused and accounting is corrected, the attacker redeems those inflated shares for more tokens than they deposited, extracting value from honest LPs.

Both paths are reachable by any unprivileged address with no special role or setup beyond holding the relevant tokens.

---

### Likelihood Explanation

The pool's pause mechanism is explicitly designed for security incidents (`pauseLevel` 1 = admin pause, 2 = protocol pause). Any time the factory admin or protocol triggers a pause, the incomplete guard is immediately exploitable. The `removeLiquidity` race is especially likely because sophisticated LPs monitor on-chain events and will react within the same block or the next block after a pause is set.

---

### Recommendation

Add `whenNotPaused` to both `addLiquidity` and `removeLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) { ... }

function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (...) { ... }
```

If the protocol intentionally wants to allow LP exits during a pause (a common design choice), a separate `pauseLevel` value (e.g., level 1 = block swaps only, level 2 = block everything) should be introduced and documented explicitly, rather than leaving the gap implicit.

---

### Proof of Concept

```
State: pauseLevel = 1 (admin pause active, swap reverts with PoolPaused)

1. Attacker observes pool is paused (e.g., oracle compromise in progress).
2. Attacker calls pool.addLiquidity(attacker, salt, deltas, "", "").
   - No whenNotPaused check → proceeds.
   - Shares minted against current (potentially skewed) BinState.
3. Protocol fixes oracle, calls factory.setPause(pool, 0).
4. Attacker calls pool.removeLiquidity(attacker, salt, deltas, "").
   - No whenNotPaused check → proceeds.
   - Attacker redeems inflated shares, receiving more tokens than deposited.
   - Honest LPs' proportional claims are reduced by the extracted surplus.
``` [8](#0-7)
