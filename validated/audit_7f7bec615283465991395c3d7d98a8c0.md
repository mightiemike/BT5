### Title
Premature Debt Socialization While Zero-Weight Spot Assets Remain Uncleared — (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_finalizeSubaccount` in `ClearinghouseLiq.sol` explicitly skips spot products whose `longWeightInitialX18 == 0` in every pre-finalization guard. A subaccount can therefore be finalized — and its entire negative quote balance socialized across all depositors — while it still holds a positive balance of zero-weight spot assets. After socialization those assets remain intact in the subaccount and can be sold or withdrawn by the owner, extracting value that should have offset the socialized loss.

---

### Finding Description

`_finalizeSubaccount` (lines 279–413) runs two spot-asset checks before proceeding to socialization.

**Check 1 — all non-zero-weight spot assets must be ≤ 0:**

```solidity
for (uint32 i = 1; i < v.spotIds.length; ++i) {
    uint32 spotId = v.spotIds[i];
    if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
        continue;          // ← zero-weight assets silently skipped
    }
    ISpotEngine.Balance memory balance = spotEngine.getBalance(spotId, txn.liquidatee);
    require(balance.amount <= 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
}
``` [1](#0-0) 

**Check 2 — when `canLiquidateMore` is true, non-zero-weight spot assets must be exactly 0:**

```solidity
if (v.canLiquidateMore) {
    for (uint32 i = 1; i < v.spotIds.length; ++i) {
        ...
        if (spotEngine.getRisk(spotId).longWeightInitialX18 == 0) {
            continue;      // ← zero-weight assets silently skipped again
        }
        require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
    }
}
``` [2](#0-1) 

In both checks, zero-weight spot assets are unconditionally skipped. Finalization then proceeds to socialize the subaccount's debt:

```solidity
v.insurance = perpEngine.socializeSubaccount(txn.liquidatee, v.insurance);
...
if (v.insurance <= 0) {
    spotEngine.socializeSubaccount(txn.liquidatee);
}
``` [3](#0-2) 

`SpotEngine.socializeSubaccount` only zeroes out **negative** balances; positive balances are untouched:

```solidity
if (balance.amount < 0) {
    // reduce cumulativeDepositsMultiplier, zero out the subaccount balance
    ...
    _setBalanceAndUpdateBitmap(productId, subaccount, BalanceNormalized({amountNormalized: 0}));
}
``` [4](#0-3) 

Result: after `_finalizeSubaccount` returns, the subaccount's negative quote balance has been socialized (spread as a loss to all depositors), but any positive zero-weight spot asset balances remain intact in the subaccount.

---

### Impact Explanation

The subaccount owner retains the full value of zero-weight spot assets even though those assets could have partially or fully offset the socialized debt. Every other depositor in the affected spot products absorbs a proportional loss via the reduced `cumulativeDepositsMultiplierX18`, while the insolvent subaccount owner walks away with assets that should have been applied to that shortfall. This is a direct solvency/accounting corruption: the protocol writes off a debt shortfall while recoverable collateral remains in the subaccount.

---

### Likelihood Explanation

Zero-weight spot products (`longWeightInitialX18 == 0`) are a first-class concept in the codebase — the skip logic appears in multiple places, confirming they are expected to exist. A subaccount holding such assets alongside a negative quote balance is a realistic state (e.g., a delisted token whose weight was set to zero after the position was opened, or a special product like `NLP_PRODUCT_ID`). The sequencer submits the finalization transaction following the on-chain rules, which explicitly permit finalization in this state. No sequencer key compromise is required; the sequencer is simply following the protocol's own permissive check.

---

### Recommendation

Before proceeding to socialization, `_finalizeSubaccount` must also require that **all** spot asset balances — including zero-weight ones — are ≤ 0 (or, when `canLiquidateMore` is true, exactly 0). The simplest fix is to remove the `longWeightInitialX18 == 0` guard from both loops, or to add a separate loop that enforces `balance.amount <= 0` for zero-weight assets before socialization is triggered.
