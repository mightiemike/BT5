### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Repeated Bonus Reward Withdrawal - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. Because there is no deduplication check on `msg.PositionIds`, a position owner can list the same position ID multiple times in a single transaction. Each duplicate entry is a separate in-memory copy carrying the original (pre-claim) `LastBonusAccrual` checkpoint. The bonus reward for the current accrual segment is then computed and paid out independently for every copy, draining the tieredrewards bonus pool by a multiple of the legitimate amount.

---

When the same position ID appears at index `i+1`, that element still holds the original `LastBonusAccrual` value loaded from the store before any processing began. `processEventsAndClaimBonus` therefore recomputes the identical segment and issues a second `SendCoinsFromModuleToAccount` from the rewards pool:

---

### Impact Explanation

A position owner submits one `MsgClaimTierRewards` transaction with `PositionIds = [X, X, X, …, X]` (N copies of the same ID). The tieredrewards bonus pool pays out N times the legitimate bonus for that position in a single block. The corrupted value is the `RewardsPoolName` module account balance and the owner's token balance. The pool can be drained to zero in a single transaction if N is chosen large enough.

Base rewards are not doubled because the Cosmos SDK distribution module resets the outstanding reward to zero after the first `WithdrawDelegationRewards` call; subsequent calls for the same delegator/validator pair return zero. Bonus rewards, however, are computed purely from the in-memory `LastBonusAccrual` field and are not protected by any analogous idempotency mechanism.

---

### Likelihood Explanation

Any position owner can trigger this with a single, standard `MsgClaimTierRewards` transaction. No special privileges, governance access, or timing are required. The only practical limit is the current pool balance and the per-transaction gas limit, both of which bound N but do not prevent the attack.

---

### Recommendation

Add a deduplication check on `msg.PositionIds` before the collection loop, or re-read each position from the store inside `claimRewardsAndUpdatesPositions` rather than relying on the pre-loaded in-memory slice. The simplest fix is in `MsgClaimTierRewards.Validate()` (or at the top of the handler):

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest,
            "duplicate position id %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, re-fetch each position from the store at the start of each loop iteration inside `claimRewardsAndUpdatesPositions` so that the updated checkpoint is always used.

---

### Proof of Concept

1. Owner holds position ID `42` with `LastBonusAccrual = T-1h` and the validator is bonded.
2. Owner broadcasts `MsgClaimTierRewards{Owner: owner, PositionIds: [42, 42, 42]}`.
3. The handler loads position `42` three times from the store; all three copies have `LastBonusAccrual = T-1h`.
4. First iteration: computes 1-hour bonus, sends it from `RewardsPoolName` to owner, advances in-memory checkpoint to `blockTime`.
5. Second iteration: the second copy still has `LastBonusAccrual = T-1h`; computes the same 1-hour bonus, sends it again.
6. Third iteration: same as second.
7. Owner receives 3× the legitimate bonus in one transaction; the rewards pool loses 3× the intended amount.
