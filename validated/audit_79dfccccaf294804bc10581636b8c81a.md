### Title
`promise_batch_action_transfer_to_gas_key` / `add_gas_key` host functions pass wire-format `public_key_len` to exec-fee helpers instead of `trie_id_len`, overcharging ML-DSA-65 gas-key actions ~59× - (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The host-function implementations of `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` forward the raw wire-format `public_key_len` (the borsh-encoded byte count as supplied by the calling contract) to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. Those helpers are explicitly documented to price the **trie read/write cost**, which for ML-DSA-65 keys is based on the 33-byte SHA3-256 hash stored in the trie, not the 1953-byte wire form. The transaction-conversion path in `exec_fee` (`runtime/runtime/src/config.rs`) correctly calls `action.public_key.trie_id_len()` (33 for ML-DSA-65). The host-function path uses `public_key_len as usize` (1953 for ML-DSA-65), producing a ~59× overcharge in the exec-fee component for every ML-DSA-65 gas-key action dispatched from a contract.

---

### Finding Description

`PublicKey` exposes two length methods with divergent semantics:

- `len()` — borsh-encoded wire length: 33 / 65 / **1953** bytes for ED25519 / SECP256K1 / ML-DSA-65.
- `trie_id_len()` — on-trie identifier length: 33 / 65 / **33** bytes (ML-DSA-65 is stored as `[tag=3] || sha3_256(domain || raw_pubkey)`). [1](#0-0) 

The exec-fee helpers are documented to price trie I/O:

> "Exec fee for TransferToGasKey / WithdrawFromGasKey actions. Based on the **access key trie key length** + estimated value length (what the receiver needs to read/write in the trie)." [2](#0-1) 

The transaction-conversion path in `exec_fee` correctly uses `trie_id_len()`:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [3](#0-2) 

The host-function path in `promise_batch_action_transfer_to_gas_key` uses the raw wire length instead:

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length (1953 for ML-DSA-65), not trie_id_len (33)
);
``` [4](#0-3) 

The same substitution error appears in the wasmtime path: [5](#0-4) 

And in the `add_gas_key` host functions, where `gas_key_add_key_exec_fee` is called with `public_key_len as usize` instead of `trie_id_len`: [6](#0-5) 

Compare with the correct call in `permission_exec_fees`:

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← correct
    gas_key_info.num_nonces,
);
``` [7](#0-6) 

The divergent value for ML-DSA-65: `public_key_len as usize` = **1953**, `trie_id_len()` = **33**. The exec per-byte component is overcharged by a factor of **1953 / 33 ≈ 59**.

---

### Impact Explanation

`pay_action_accumulated` charges `use_byte = send.per_byte + exec.per_byte` from the contract's gas counter. Because `exec.per_byte` is computed against 1953 bytes instead of 33, a contract dispatching a `TransferToGasKey` or `WithdrawFromGasKey` action with an ML-DSA-65 key has its gas budget reduced by ~59× more than the protocol intends for the exec component. This:

1. **Breaks the fee-domain invariant**: the exec fee is supposed to reflect trie I/O cost; using the wire length inflates it by ~59× for ML-DSA-65 keys.
2. **Makes ML-DSA-65 gas-key operations from contracts economically unviable**: contracts will exhaust their gas budget or pay far more than the equivalent ed25519 path.
3. **Creates a fee inconsistency** between the transaction-conversion path (correct) and the host-function path (incorrect), violating the protocol's deterministic pricing guarantee.

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature gates ML-DSA-65 key acceptance. Once that feature is active on mainnet/testnet, any unprivileged contract can call `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 key and trigger the overcharge. No special role or privileged access is required. The bug is latent until the feature activates; after activation it affects every such call.

---

### Recommendation

Replace `public_key_len as usize` with `public_key.trie_id_len()` in the exec-fee calls inside the host-function implementations. The decoded `public_key` is already available at that point (after `self.get_public_key(public_key_ptr, public_key_len)?`):

```rust
// In promise_batch_action_transfer_to_gas_key (logic.rs and wasmtime_runner/logic.rs):
let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
// ...
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
//                                                                ^^^ wire len: correct for send
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
-   public_key_len as usize,
+   public_key.decode()?.trie_id_len(),   // trie-id len: correct for exec
);
```

Apply the same fix to `gas_key_add_key_exec_fee` calls in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`.

---

### Proof of Concept

For an ML-DSA-65 public key:
- `public_key_len` (wire) = 1 + 1952 = **1953 bytes**
- `trie_id_len()` = 1 + 32 = **33 bytes** [8](#0-7) 

`gas_key_transfer_exec_fee` computes:
```
trie_key_len = access_key_key_len(account_id_len, public_key_len)
             = 2 + account_id_len + public_key_len
```

With `public_key_len = 1953` (bug): `trie_key_len = 2 + account_id_len + 1953`  
With `public_key_len = 33` (correct): `trie_key_len = 2 + account_id_len + 33`

The per-byte exec fee is multiplied by `trie_key_len + estimated_value_len`. For a 10-byte account ID and a typical `estimated_value_len`:

- Buggy: `(2 + 10 + 1953 + value_len) ≈ 1965 + value_len`
- Correct: `(2 + 10 + 33 + value_len) ≈ 45 + value_len`

Ratio ≈ **43×–59×** overcharge in the exec per-byte component, charged directly against the calling contract's gas budget. [9](#0-8)

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```
