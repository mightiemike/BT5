### Title
Protobuf `ValidResourceBounds` Round-Trip Silently Downgrades `AllResources` to `L1Gas`, Causing Transaction Hash Divergence for Zero-Gas V3 Transactions - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserialization of `ValidResourceBounds` converts `AllResources{l2_gas=0, l1_data_gas=0}` to `L1Gas`, but `get_tip_resource_bounds_hash` produces structurally different hash preimages for these two variants. `AllResources` includes a zero `L1_DATA_GAS` field (3 resource felts in the poseidon preimage), while `L1Gas` does not (2 resource felts). A valid V3 transaction with zero `l2_gas` and zero `l1_data_gas` is accepted by the gateway with hash H1 (computed over 3 felts), but after protobuf round-trip the variant is downgraded to `L1Gas` and any hash recalculation produces H2 ≠ H1, causing hash validation to fail on receiving peers.

---

### Finding Description

**Hash domain boundary — `get_tip_resource_bounds_hash`**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the `ValidResourceBounds` variant:

```
L1Gas      → poseidon(tip, L1_GAS_concat, L2_GAS_concat)                          // 2 resource felts
AllResources → poseidon(tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat)    // 3 resource felts
``` [1](#0-0) 

Even when `l1_data_gas` is zero, the `AllResources` branch appends a zero-valued `L1_DATA_GAS` felt, producing a different poseidon output than the `L1Gas` branch.

**Serialization path — `From<ValidResourceBounds> for protobuf::ResourceBounds`**

When `ValidResourceBounds::L1Gas` is serialized, `l2_gas` and `l1_data_gas` are both emitted as zero:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),          // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
``` [2](#0-1) 

**Deserialization path — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`**

On deserialization, the variant is inferred purely from field values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [3](#0-2) 

This means an `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` is **silently downgraded to `L1Gas`** after a protobuf round-trip. The two variants are indistinguishable in the wire format, but they produce different transaction hashes.

**Origin of the `AllResources` variant with zero gas fields**

The gateway always accepts `RpcTransaction` with `AllResourceBounds` (never `ValidResourceBounds`). The `InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` and always wraps it as `ValidResourceBounds::AllResources` when converting to `InvokeTransactionV3`:

```rust
impl From<InternalRpcInvokeTransactionV3> for InvokeTransactionV3 {
    fn from(tx: InternalRpcInvokeTransactionV3) -> Self {
        Self {
            resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
            ...
        }
    }
}
``` [4](#0-3) 

A user submitting a V3 invoke with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero — a valid and accepted configuration per gateway tests) causes the gateway to compute H1 using the `AllResources` branch (3 felts). After protobuf round-trip the `InvokeTransactionV3` carries `L1Gas`, and `validate_transaction_hash` recomputes H2 using the `L1Gas` branch (2 felts). H1 ≠ H2. [5](#0-4) 

The RPC conversion in `apollo_rpc` applies the same zero-check logic, confirming the pattern is consistent across conversion boundaries: [6](#0-5) 

---

### Impact Explanation

Any V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` (non-zero `l1_gas`) is accepted by the gateway with hash H1 (3-felt preimage). After protobuf serialization and deserialization — which occurs on every P2P hop and during state sync — the `ValidResourceBounds` variant changes to `L1Gas`. Any subsequent call to `validate_transaction_hash` or `get_transaction_hash` on the deserialized object produces H2 (2-felt preimage), H2 ≠ H1. This causes:

- **Hash validation failure during P2P state sync**: valid transactions are rejected by syncing peers.
- **Wrong hash bound to the executable payload**: the transaction hash stored by the receiving node diverges from the hash signed by the sender, breaking signature verification on the deserialized form.

This matches the allowed impact: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** (High).

---

### Likelihood Explanation

Any user submitting a V3 transaction with only `l1_gas` non-zero triggers this path. The gateway stateless validator explicitly accepts this configuration (test case `valid_l1_gas`). No special privileges are required. The bug fires on every protobuf round-trip of such a transaction, which happens on every P2P propagation hop and during state sync. [7](#0-6) 

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not infer the variant from field values alone. Instead, use the **presence or absence of the `l1_data_gas` field** in the protobuf message to distinguish legacy `L1Gas` (field absent, 0.13.2 era) from `AllResources` (field present, even if zero, 0.13.3+):

```rust
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),   // legacy: field absent
    Some(_) => ValidResourceBounds::AllResources(
        AllResourceBounds { l1_gas, l2_gas, l1_data_gas }
    ),
})
```

This preserves the `AllResources` variant for all transactions that explicitly carry a `l1_data_gas` field (even zero), matching the hash domain used at submission time. The `TODO(Shahak)` comment already anticipates removing the `unwrap_or_default` fallback once 0.13.2 support is dropped, which would make this fix straightforward. [8](#0-7) 

---

### Proof of Concept

1. Submit a V3 invoke transaction: `l1_gas = X` (non-zero), `l2_gas = 0`, `l1_data_gas = 0`.
2. Gateway accepts it. `InternalRpcInvokeTransactionV3::calculate_transaction_hash` calls `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`. Hash H1 = `poseidon(INVOKE, version, sender, poseidon(tip, L1_GAS_X, L2_GAS_0, L1_DATA_GAS_0), ...)` — **3 resource felts**.
3. Transaction is converted to `InvokeTransactionV3` with `resource_bounds = ValidResourceBounds::AllResources(...)` and stored with hash H1.
4. Transaction is serialized to protobuf: `l2_gas = 0`, `l1_data_gas = 0` (both present but zero).
5. Peer deserializes: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
6. Peer calls `validate_transaction_hash`. `get_transaction_hash` calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::L1Gas(...)`. Hash H2 = `poseidon(INVOKE, version, sender, poseidon(tip, L1_GAS_X, L2_GAS_0), ...)` — **2 resource felts**.
7. H1 ≠ H2. `possible_hashes.contains(&H1)` is false. Transaction rejected. [9](#0-8) [10](#0-9)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
