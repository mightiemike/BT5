### Title
`ENABLE_SECP_OPS` Flag Silently Bypassed by 4-Byte Opcode Form — (`File: src/chia_dialect.rs`)

### Summary

The `ClvmFlags::ENABLE_SECP_OPS` flag gates single-byte opcodes 64 and 65 for secp256k1/r1 verification, but the functionally identical 4-byte opcode forms (`0x13d61f00`, `0x1c3a8f00`) are dispatched to the same operator functions unconditionally, with no flag check. A caller who constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — expecting secp operations to be unavailable — receives no error when an attacker-supplied program uses the 4-byte encoding instead.

### Finding Description

In `ChiaDialect::op()`, the 4-byte opcode branch executes first:

```rust
// src/chia_dialect.rs lines 157–182
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The single-byte branch, reached only when `op_len == 1`, does check the flag:

```rust
// src/chia_dialect.rs lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The two branches call the same underlying functions (`op_secp256k1_verify`, `op_secp256r1_verify` in `src/secp_ops.rs`) with identical semantics and cost. The flag check is therefore structurally incomplete: it guards one encoding of the operation but not the other. No error is raised when the 4-byte form is used without the flag set; the secp verification simply succeeds.

The `ENABLE_SECP_OPS` flag documentation states: *"Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)."* A caller reading this naturally concludes that omitting the flag disables secp operations entirely. The 4-byte form is not mentioned, and no diagnostic is emitted when it is used.

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`:

```rust
// src/chia_dialect.rs lines 72–76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

This means secp operations are always reachable in mempool mode via the 4-byte encoding, regardless of whether the caller intended to permit them.

### Impact Explanation

**Consensus / mempool divergence.** A node operating in a mode where `ENABLE_SECP_OPS` is absent (e.g., pre-hardfork consensus, or any custom dialect that deliberately omits the flag) will silently accept and execute secp signature verification when the 4-byte opcode form is used. A node or tool that checks only the single-byte form will reject the same program. This produces a split in program acceptance between implementations or between flag configurations, which is a consensus-equivalence violation.

**Silent flag override.** The caller's explicit choice to omit `ENABLE_SECP_OPS` is overridden without any error or diagnostic, exactly mirroring the M-05 pattern: the caller's stated intent (disable secp ops) is silently ignored for one code path, and the operation proceeds as if the flag were set.

### Likelihood Explanation

The 4-byte opcode encoding is part of the documented "unknown operators with assigned cost" scheme and is a valid, parseable CLVM atom. Any attacker who can supply CLVM bytes to a node running without `ENABLE_SECP_OPS` can trivially use the 4-byte form. The encoding is deterministic and requires no special privileges.

### Recommendation

Add the same `ENABLE_SECP_OPS` guard to the 4-byte opcode branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

If the 4-byte form is intentionally always-on (pre-dating the flag), the flag documentation and the `ENABLE_SECP_OPS` name must be corrected to make clear that the flag only controls the single-byte aliases, and callers who want to fully disable secp operations must be given a separate mechanism to do so.

### Proof of Concept

1. Construct a `ChiaDialect` **without** `ENABLE_SECP_OPS` (e.g., `ChiaDialect::new(ClvmFlags::empty())`).
2. Craft a CLVM program that invokes secp256k1 verification using the 4-byte opcode `0x13d61f00` with a valid pubkey, message digest, and signature.
3. Call `run_program` with this dialect and program.
4. Observe: the secp verification executes and returns nil (success) with cost 1 300 000, with no error — despite `ENABLE_SECP_OPS` being absent.
5. Repeat with single-byte opcode 64: the program is routed to `unknown_operator` and rejected (or treated as a no-op), demonstrating the asymmetry. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
