### Title
CEI Violation in `withdrawCollateral` Enables Re-Entrancy via Token Transfer Hook, Corrupting `nSubmissions` and Permanently Invalidating Fast Withdrawals — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the external token transfer **before** updating the subaccount balance and running the health check. `Endpoint.executeSlowModeTransaction` has no re-entrancy guard and writes its `SlowModeConfig` memory copy back to storage only at the end of the call. A token with a transfer hook (e.g., ERC777 `tokensReceived`) can re-enter `executeSlowModeTransaction` mid-withdrawal, causing `nSubmissions` to be incremented twice. This skips an `idx` value, permanently invalidating any fast withdrawal that was pre-signed against the skipped index and desynchronising the sequencer's `nSubmissions` counter.

---

### Finding Description

`withdrawCollateral` in `Clearinghouse.sol` violates Checks-Effects-Interactions:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // ← external call FIRST
...
spotEngine.updateBalance(productId, sender, amountRealized);  // ← state update AFTER
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ...);     // ← health check AFTER
``` [1](#0-0) 

`handleWithdrawTransfer` chains two external calls: first `token.safeTransfer(withdrawPool, amount)`, then `withdrawPool.submitWithdrawal(...)`, which itself calls `token.safeTransfer(sendTo, amount)` — the final transfer to the user. [2](#0-1) [3](#0-2) 

`Endpoint.executeSlowModeTransaction` is publicly callable, has no re-entrancy guard, and writes its memory `_slowModeConfig` back to storage only at the very end:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;   // reads storage
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;                         // writes back at end
}
``` [4](#0-3) 

Inside `_executeSlowModeTransaction`, the slow-mode slot is deleted and `txUpTo` is incremented **only in the memory copy**:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
``` [5](#0-4) 

Because the storage `slowModeConfig.txUpTo` is not updated until the outer call returns, a re-entrant call to `executeSlowModeTransaction` reads the **original** `txUpTo = N` from storage, finds the already-deleted (empty) slot, processes it silently via the `try/catch`, and then **writes `txUpTo = N+1` back to storage**. The outer call later also writes `txUpTo = N+1` (no net change to `txUpTo`), but **both calls each increment `nSubmissions`**, advancing it by 2 instead of 1. [6](#0-5) 

`nSubmissions` is used as the `idx` argument to `withdrawCollateral` in both the slow-mode and sequencer paths: [7](#0-6) [8](#0-7) 

`submitWithdrawal` sets `minIdx = idx` for each sequencer-processed withdrawal: [9](#0-8) 

`submitFastWithdrawal` enforces `require(idx > minIdx)`: [10](#0-9) 

---

### Impact Explanation

After the re-entrancy, `nSubmissions` jumps from `K` to `K+2`. The next sequencer-processed withdrawal sets `minIdx = K+2`. Any fast withdrawal pre-signed by the verifier with `idx = K+1` now fails the `require(idx > minIdx)` check (`K+1 > K+2` is false), permanently invalidating it. The attacker can repeat this on every slow-mode withdrawal to continuously skip `idx` values, causing sustained DoS on the fast-withdrawal path. Additionally, the sequencer's off-chain `nSubmissions` tracking becomes desynchronised, causing subsequent sequencer-submitted withdrawals to use wrong `idx` values until manually corrected.

---

### Likelihood Explanation

The trigger requires a collateral token with a transfer hook (ERC777 `tokensReceived`, or a custom `_beforeTokenTransfer` that calls an external contract) and a `sendTo` address that is a contract implementing the hook. ERC777 tokens are in active use on EVM chains. The attacker controls both the token choice (by depositing a hook-bearing token as collateral) and the `sendTo` contract. No privileged access is required; any user can submit a slow-mode withdrawal and call `executeSlowModeTransaction` after the timeout.

---

### Recommendation

1. Apply a re-entrancy guard (e.g., OpenZeppelin `ReentrancyGuard`) to `Endpoint.executeSlowModeTransaction` and `Endpoint.executeSlowModeTransaction`'s internal path.
2. Move `spotEngine.updateBalance` and the health check **before** `handleWithdrawTransfer` in `Clearinghouse.withdrawCollateral` to restore CEI ordering.
3. Alternatively, write `slowModeConfig` and increment `nSubmissions` **before** calling `_executeSlowModeTransaction` so that re-entrant calls see the already-advanced state.

---

### Proof of Concept

**Setup:**
- Attacker deposits an ERC777 token as collateral into subaccount `S`.
- Attacker submits a slow-mode `WithdrawCollateral` transaction (queued at index `N`).
- Attacker deploys a receiver contract at address `R` that implements `tokensReceived` to call `Endpoint.executeSlowModeTransaction()`.
- `nSubmissions = K`, `minIdx = K-1`.

**Execution (after 3-day timeout):**

1. Attacker calls `Endpoint.executeSlowModeTransaction()`.
2. Outer call: reads `slowModeConfig.txUpTo = N`, deletes `slowModeTxs[N]`, processes withdrawal → `withdrawCollateral` → `handleWithdrawTransfer` → `token.safeTransfer(R, amount)`.
3. ERC777 hook fires on `R`: calls `Endpoint.executeSlowModeTransaction()` (re-entrant).
4. Inner call: reads `slowModeConfig.txUpTo = N` (storage unchanged), finds empty slot, processes silently, writes `slowModeConfig.txUpTo = N+1`, increments `nSubmissions` → `nSubmissions = K+1`.
5. Inner call returns.
6. Outer call: `spotEngine.updateBalance` runs, health check passes, `nSubmissions += 1` → `nSubmissions = K+2`, writes `slowModeConfig.txUpTo = N+1`.

**Result:**
- `nSubmissions = K+2` (skipped `K+1`).
- Next sequencer withdrawal uses `idx = K+2`, setting `minIdx = K+2`.
- Any fast withdrawal pre-signed with `idx = K+1` fails: `require(K+1 > K+2)` reverts.
