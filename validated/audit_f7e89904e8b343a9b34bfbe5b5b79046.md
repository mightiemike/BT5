### Title
`WithdrawCollateralV2` Defined and Accepted in Slow Mode but Never Executed — Silent Fee Loss and Withdrawal Failure - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` is fully declared in the `TransactionType` enum, has a complete struct definition, and is handled in the sequencer fast path (`processTransactionImpl`). However, it is entirely absent from `processSlowModeTransactionImpl`. Because `submitSlowModeTransactionImpl` accepts it without restriction (charging the slow mode fee), a user who submits a `WithdrawCollateralV2` via slow mode will have their fee deducted and their transaction queued — but the transaction will silently revert and be dropped when executed, leaving the withdrawal unprocessed.

---

### Finding Description

The `TransactionType` enum in `IEndpoint.sol` declares `WithdrawCollateralV2` as a first-class transaction type: [1](#0-0) 

Its struct is fully defined with `sender`, `productId`, `amount`, `nonce`, `sendTo`, and `appendix`: [2](#0-1) 

It is fully handled in the sequencer fast path inside `processTransactionImpl`: [3](#0-2) 

However, `processSlowModeTransactionImpl` handles only `WithdrawCollateral` (V1) and has no branch for `WithdrawCollateralV2`. Any unrecognized type hits the terminal `else { revert(); }`: [4](#0-3) [5](#0-4) 

Meanwhile, `submitSlowModeTransactionImpl` accepts `WithdrawCollateralV2` without restriction — it falls into the `else` branch that charges the slow mode fee and enqueues the transaction: [6](#0-5) 

When the queued transaction is later executed, `_executeSlowModeTransaction` calls `processSlowModeTransaction` inside a `try/catch`. On non-hardhat chains the revert is swallowed silently: [7](#0-6) 

The result is: fee charged, transaction queued, withdrawal silently dropped.

---

### Impact Explanation

Two concrete state deltas are corrupted:

1. **Slow mode fee loss** — `slowModeFees` is incremented and the user's spot balance is debited by `SLOW_MODE_FEE` at submission time. This debit is never reversed because the silent catch does not refund. The user permanently loses the fee for a transaction that can never succeed.

2. **Withdrawal failure with no on-chain signal** — The user's collateral remains locked. Because the failure is silent (no event, no revert propagated to the caller of `executeSlowModeTransaction`), the user has no on-chain indication that their withdrawal was dropped. The `sendTo` destination address receives nothing.

The `sendTo` field in `WithdrawCollateralV2` is specifically designed for users who need to direct funds to a different address (e.g., a cold wallet). This is most critical precisely when the sequencer is offline or censoring — the exact scenario where slow mode is the only available path.

---

### Likelihood Explanation

Any user who:
- Constructs a `WithdrawCollateralV2` transaction (the newer, documented withdrawal type),
- Submits it via `submitSlowModeTransaction` (the censorship-resistance path),

will trigger this. No special privileges are required. The `submitSlowModeTransactionImpl` function imposes no type-based restriction on `WithdrawCollateralV2`. The likelihood increases as `WithdrawCollateralV2` becomes the canonical withdrawal type used by front-ends and integrators, since users following current documentation would naturally use V2.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` in `EndpointTx.sol`, mirroring the existing `WithdrawCollateral` branch but decoding `SignedWithdrawCollateralV2` and forwarding `sendTo` to `clearinghouse.withdrawCollateral`. Alternatively, if `WithdrawCollateralV2` is intentionally sequencer-only, add an explicit `revert()` (or a named error) in `submitSlowModeTransactionImpl` for this type so the fee is never charged and the transaction is never silently accepted.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateralV2), abi.encode(signedV2Txn)))`.
2. `submitSlowModeTransactionImpl` hits the `else` branch at line 369 — slow mode fee is charged, transaction is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`.
3. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)` inside a `try/catch`.
5. `processSlowModeTransactionImpl` reaches `else { revert(); }` at line 328 — no `WithdrawCollateralV2` branch exists.
6. The revert is caught silently (line 207 `try ... {} catch {}`); `gasleft()` checks pass for a normal revert.
7. `nSubmissions` is incremented, the slow mode slot is deleted, but no withdrawal occurs and no fee is refunded.
8. User's collateral remains in the protocol; slow mode fee is permanently lost. [8](#0-7) [9](#0-8)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
