### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Systematically Understating `metricT0` and Weakening the Stop-Loss Guard for Token1 Outflow — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as the **arithmetic mean** of bid and ask, while every other price-sensitive path in the protocol — `MetricOmmPool.swap`, `MetricOmmPool.simulateSwapAndRevert`, and `PriceVelocityGuardExtension.beforeSwap` — uses the **geometric mean** via `SwapMath.midAndSpreadFeeX64FromBidAsk`. By the AM-GM inequality the arithmetic mean is always ≥ the geometric mean, so the stop-loss extension always prices token0 higher than the pool does. This causes the per-share value metric `metricT0` (total value in token0 terms) to be **understated**, setting the high-watermark and its drawdown floor lower than the configured `drawdownE6` actually requires. The stop-loss guard for `zeroForOne == true` swaps (token1 outflow) therefore fires later than intended, allowing LPs to suffer a larger real drawdown than the pool admin configured.

---

### Finding Description

**Divergent mid-price formulas**

`SwapMath.midAndSpreadFeeX64FromBidAsk` (used by the pool and `PriceVelocityGuardExtension`):

```solidity
// SwapMath.sol line 70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);   // geometric mean
```

`OracleValueStopLossExtension._afterSwapOracleStopLoss`:

```solidity
// OracleValueStopLossExtension.sol line 218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

**How the wrong price propagates into the watermark**

The arithmetic mid is passed directly into `_metrics`:

```solidity
// OracleValueStopLossExtension.sol lines 252-255
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

`metricT0` represents total bin value in token0 units per share: `t0 + t1 / price`. Dividing `t1` by a price that is **larger** than the true geometric mid yields a **smaller** token0-equivalent for the token1 leg, so `metricT0` is understated. The ratchet in `_applyWatermark` then records this understated value as the new high-watermark:

```solidity
// OracleValueStopLossExtension.sol lines 333-335
if (metric >= hwm) return (metric, false);   // ratchets up to understated value
breached = metric < (hwm * floorMultiplier) / E6;
```

The floor is `hwm * (E6 - drawdownE6) / E6`. Because `hwm` was set too low, the floor is also too low, and the breach condition fires later than the configured drawdown requires.

**Direction-aware blocking**

```solidity
// OracleValueStopLossExtension.sol lines 271-272
if (breach0 && zeroForOne) {
    revert OracleStopLossTriggered(...);
}
```

Only `metricT0` guards `zeroForOne == true` (token1 outflow). Because `metricT0` is the understated metric, this is precisely the direction where the guard is weakened.

---

### Impact Explanation

The stop-loss is the primary on-chain mechanism protecting LP principal from oracle-price-driven extraction. When the arithmetic mean exceeds the geometric mean by a relative factor `ε`, the effective drawdown floor is shifted down by approximately `ε * t1_fraction_of_value`. For a 10 % oracle spread (bid/ask ratio ≈ 0.9/1.1), `ε ≈ 0.12 %`; for a 50 % spread, `ε ≈ 3.5 %`; for a 100 % spread (bid = 0.5 × ask), `ε ≈ 10 %`. A pool configured with `drawdownE6 = 50 000` (5 %) and a 50 % oracle spread effectively allows ~8.5 % drawdown before the guard fires. LPs lose principal beyond the admin-configured threshold with no on-chain recourse.

---

### Likelihood Explanation

The bug is always active whenever `drawdownE6 > 0` and the oracle spread is non-zero. No special attacker action is required: any ordinary `zeroForOne` swap triggers `afterSwap`, which calls `_afterSwapOracleStopLoss` with the wrong mid-price. The magnitude of the bypass scales with the oracle spread, which can be large during volatile market conditions — exactly the scenario where the stop-loss is most needed.

---

### Recommendation

Replace the arithmetic mean with the same geometric mean formula used everywhere else in the protocol:

```solidity
// OracleValueStopLossExtension.sol line 218 — replace:
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// with:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This aligns the stop-loss metric with the price the pool actually uses for swap execution, ensuring the configured `drawdownE6` is the true protection threshold.

---

### Proof of Concept

Let `bid = 0.5 × Q64`, `ask = 1.5 × Q64` (100 % spread), a bin with `t0 = 0`, `t1 = 1000` scaled units, `shares = 1000`, `drawdownE6 = 50 000` (5 % drawdown allowed).

**Geometric mean (correct):**
```
mid_geom = sqrt(0.5 * 1.5) * Q64 = sqrt(0.75) * Q64 ≈ 0.866 * Q64
metricT0_true = t1 * Q64 / mid_geom / shares * METRIC_SCALE
             = 1000 * Q64 / (0.866 * Q64) / 1000 * 1e6
             ≈ 1.155 * 1e6
floor_true = 1.155e6 * (1e6 - 50_000) / 1e6 ≈ 1.097e6
```

**Arithmetic mean (buggy):**
```
mid_arith = (0.5 + 1.5) / 2 * Q64 = 1.0 * Q64
metricT0_bug = 1000 * Q64 / (1.0 * Q64) / 1000 * 1e6 = 1.0 * 1e6
floor_bug  = 1.0e6 * (1e6 - 50_000) / 1e6 = 0.95e6
```

The buggy floor (0.95e6) is ~13 % lower than the correct floor (1.097e6). An attacker can drain token1 until `metricT0` falls to 0.95e6 — a real drawdown of ~18 % — before the stop-loss fires, despite the admin configuring only 5 % drawdown protection.
