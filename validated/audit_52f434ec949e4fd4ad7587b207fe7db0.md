### Title
PromiseYield Timeout Queue Monotonicity Broken by Protocol Upgrade Changing `yield_timeout_length_in_blocks` — (`File: runtime/runtime/src/lib.rs`)

### Summary

The `resolve_promise_yield_timeouts` function in nearcore processes a persistent trie-backed queue of `PromiseYieldTimeout` entries ordered by `expires_at` block height. The processing loop hard-breaks on the first entry whose `expires_at > current_block_height`, relying on the invariant that the queue is monotonically ordered by `expires_at`. However, `expires_at` is computed at enqueue time as `block_height + yield_timeout_length_in_blocks`, where `yield_timeout_length_in_blocks` is a `RuntimeConfig` parameter that changes across protocol versions. When a protocol upgrade reduces `yield_timeout_length_in_blocks`, newly enqueued entries will have smaller `expires_at` values than older entries already in the queue, breaking the monotonicity invariant. The loop then stops at the first old (not-yet-expired) entry, permanently skipping all newer entries that have already expired, causing those yields to never time out until the blocking entry is processed.

### Finding Description

**Enqueue path** (`runtime/runtime/src/function_call.rs`, line 168):

```rust
apply_state.block_height
    + config.wasm_config.limit_config.yield_timeout_length_in_blocks,
```

Each `PromiseYield` receipt enqueues a `PromiseYieldTimeout` with `expires_at = current_block_height + yield_timeout_length_in_blocks`. The value of `yield_timeout_length_in_blocks` is taken from the `RuntimeConfig` active at the block where the yield is created.

**Dequeue path** (`runtime/runtime/src/lib.rs`, lines 2962–2964):

```rust
// Queue entries are ordered by expires_at
if queue_entry.expires_at > apply_state.block_height {
    break;
}
```

The comment explicitly states the ordering assumption. The loop breaks immediately on the first non-expired entry, never scanning past it.

**The invariant break**: When a protocol upgrade activates a new `RuntimeConfig` with a smaller `yield_timeout_length_in_blocks`, yields created after the upgrade will have `expires_at` values smaller than yields created before the upgrade that are still in the queue. The queue is ordered by insertion index, not by `expires_at`. After the upgrade, the queue looks like:

```
[old_entry: expires_at=H+200, new_entry: expires_at=H+100, ...]
```

When processing at block `H+150`, `old_entry.expires_at (H+200) > H+150` → `break`. The `new_entry` (already expired at `H+100`) is never reached and never processed.

**Protocol version context**: `yield_timeout_length_in_blocks` is set to 200 in all current protocol versions (mainnet/testnet snapshots confirm this), but the parameter is part of `RuntimeConfig` which is versioned per protocol version. Any future protocol upgrade that reduces this value (e.g., to 100 blocks) would trigger this bug for all yields created before the upgrade that have not yet timed out.

The `PromiseYieldTimeout` struct is Borsh-serialized into the trie:

```rust
pub struct PromiseYieldTimeout {
    pub account_id: AccountId,
    pub data_id: CryptoHash,
    pub expires_at: BlockHeight,
}
```

The `expires_at` value is baked in at creation time and never updated. There is no mechanism to re-sort or re-index the queue when the timeout parameter changes.

### Impact Explanation

Any `PromiseYield` receipt created after a protocol upgrade that reduces `yield_timeout_length_in_blocks` will have its timeout entry blocked behind older entries with larger `expires_at` values. The affected yields will never receive their timeout-triggered `PromiseResume` receipt until all older entries ahead of them in the queue expire. This means:

- Smart contract callbacks that depend on yield timeout (the `PromiseResult::Failed` path) are permanently delayed or indefinitely blocked.
- Funds or state locked behind those callbacks cannot be released.
- The severity scales with the number of old entries ahead in the queue and the magnitude of the timeout reduction.

This is a **High** severity issue: it causes permanent, deterministic DoS of the yield timeout mechanism for all yields created in the window between the protocol upgrade and the expiry of all pre-upgrade queue entries.

### Likelihood Explanation

The bug is latent and requires a protocol upgrade that reduces `yield_timeout_length_in_blocks`. Currently the value is 200 blocks across all protocol versions. However:

1. The parameter is explicitly designed to be configurable per protocol version.
2. Any future protocol upgrade reducing this value (e.g., to improve responsiveness) would trigger the bug.
3. The trigger is a normal, governance-approved protocol upgrade — not an attack. Any user who has an active `PromiseYield` at the time of such an upgrade is affected without any action on their part.

### Recommendation

Replace the hard `break` in `resolve_promise_yield_timeouts` with a `continue` (skip) when `queue_entry.expires_at > apply_state.block_height`, so that the loop scans the entire queue rather than stopping at the first non-expired entry:

```rust
if queue_entry.expires_at > apply_state.block_height {
    // Do not break; a later entry may have a smaller expires_at
    // due to a protocol upgrade reducing yield_timeout_length_in_blocks.
    promise_yield_indices.first_index += 1;
    continue;
}
```

Alternatively, store `expires_at` as the trie key (sorted order) rather than an insertion index, so the queue is always ordered by expiry time regardless of when entries were inserted. A simpler short-term fix is to scan the full queue and collect all expired entries before processing them.

### Proof of Concept

1. At block height `H`, protocol version `V` is active with `yield_timeout_length_in_blocks = 200`.
2. Contract A calls `promise_yield_create` → enqueues `PromiseYieldTimeout { expires_at: H + 200 }` at queue index 0.
3. At block `H + 50`, protocol upgrades to version `V+1` with `yield_timeout_length_in_blocks = 100`.
4. Contract B calls `promise_yield_create` → enqueues `PromiseYieldTimeout { expires_at: H + 150 }` at queue index 1.
5. At block `H + 160` (after B's timeout has passed, before A's):
   - `resolve_promise_yield_timeouts` reads queue index 0: `expires_at = H+200 > H+160` → **`break`**.
   - Queue index 1 (`expires_at = H+150 ≤ H+160`) is **never reached**.
   - Contract B's yield callback is never triggered by timeout.
6. Only at block `H + 200` does queue index 0 expire, allowing index 1 to be processed — 50 blocks late.

The exact divergent value is the `expires_at` field in the Borsh-serialized `PromiseYieldTimeout` stored at `TrieKey::PromiseYieldTimeout { index: 1 }`, which is smaller than the `expires_at` at `index: 0`, violating the ordering assumption at line 2962 of `runtime/runtime/src/lib.rs`. // Enqueues given timeout to the PromiseYield timeout queue
pub fn enqueue_promise_yield_timeout(
    state_update: &mut TrieUpdate,
    promise_yield_indices: &mut PromiseYieldIndices,
    account_id: AccountId,
    data_id: CryptoHash,
    expires_at: BlockHeight,
) {
    set(
        state_update,
        TrieKey::PromiseYieldTimeout { index: promise_yield_indices.next_available_index },
        &PromiseYieldTimeout { account_id, data_id, expires_at },
    );
    promise_yield_indices.next_available_index = promise_yield_indices
        .next_available_index
        .checked_add(1)
        .expect("Next available index for PromiseYield timeout queue exceeded the integer limit");
