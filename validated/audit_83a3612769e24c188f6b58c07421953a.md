Looking at the extension guard plumbing, I need to trace what `sender` the `SwapAllowlistExtension` actually receives when a swap goes through the router.

**Key trace:**

1. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` — the pool's `msg.sender` is the **router address**
2. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` — passes the router as `sender`
3. `ExtensionCalling._beforeSwap` encodes `sender` (router) and dispatches to the extension
4. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — checks if the **router** is allowlisted, not the actual user

The pool admin faces an inescapable catch-22: to let allowlisted users swap through the router, they must allowlist the router address — but doing so opens the gate to every user.

---

### Title
SwapAllowlistExtension gates the router address instead of the actual swapper, enabling full allowlist bypass via router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which equals `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the user. A pool admin who allowlists the router to support router-mediated swaps for their curated users inadvertently opens the allowlist to every address.

### Finding Description

In `MetricOmmPool.swap`, the before-swap hook is dispatched with `msg.sender` as the sender:

```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value verbatim and forwards it to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
``` [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted:

```solidity
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) calls `pool.swap`, the pool's `msg.sender` is the router:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The pool admin is forced into an inescapable choice:

| Configuration | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all (extension reverts on the router address) |
| Router **allowlisted** | Every address — including non-allowlisted ones — can bypass the allowlist by routing through the router |

There is no configuration that simultaneously allows router-mediated swaps for allowlisted users and blocks non-allowlisted users.

### Impact Explanation
A pool admin who allowlists the router to support router-mediated swaps for their curated users completely defeats the allowlist. Any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting the curated pool and the `beforeSwap` guard passes because the router is allowlisted. The allowlist — the sole access-control mechanism on the swap path — is rendered inoperative. Unauthorized swaps execute at the oracle-derived price, draining or manipulating the pool in ways the admin intended to prevent.

### Likelihood Explanation
Medium. A pool admin who deploys a `SwapAllowlistExtension` and wants their allowlisted users to access the pool through the standard periphery router will naturally call `setAllowedToSwap(pool, router, true)`. The documentation and interface give no indication that this collapses the per-user allowlist into an open gate. The misconfiguration is a predictable operational mistake, not a contrived edge case.

### Recommendation
The extension must gate the **original user**, not the immediate pool caller. Two viable approaches:

1. **Router-forwarded identity**: The router encodes `msg.sender` into `extensionData` for each hop; the extension reads and verifies that address (requires a trusted-router assumption or a signature).
2. **Recipient-based check**: For swap allowlists, check `recipient` instead of `sender` when `sender` is a known periphery contract — though this changes the semantic from "who initiates" to "who receives."
3. **Direct-only pools**: Document that `SwapAllowlistExtension` is incompatible with the router and must only be used on pools accessed directly.

The cleanest fix is to redesign the extension so that the router passes the originating user address in `extensionData` and the extension verifies it, keeping the per-user gate intact regardless of the call path.

### Proof of Concept

```solidity
// 1. Pool admin deploys pool with SwapAllowlistExtension
// 2. Pool admin allowlists user1 for direct access
swapAllowlist.setAllowedToSwap(pool, user1, true);

// 3. Pool admin allowlists the router so user1 can use it
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// 4. Non-allowlisted user2 bypasses the allowlist via the router
vm.prank(user2); // user2 is NOT in allowedSwapper
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: user2,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// ✓ swap succeeds — extension checked allowedSwapper[pool][router] == true
// ✓ user2 received token1 despite never being allowlisted
```

The `beforeSwap` hook receives `sender = address(router)`, finds it allowlisted, and returns the success selector — the non-allowlisted `user2` completes the swap.
