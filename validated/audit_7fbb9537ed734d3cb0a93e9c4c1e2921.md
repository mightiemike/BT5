### Title
Linked Signer Can Overwrite Its Own Authorization Entry via `LinkSigner` Fast-Path — (`core/contracts/EndpointTx.sol`)

### Summary
In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`. This means the current linked signer — a delegated hot-wallet or API key intended only for operational signing — can itself sign a new `LinkSigner` transaction and overwrite `linkedSigners[subaccount]` to any address it chooses. The same `linkedSigners` mapping that grants the linked signer its operational authority is also the target of the mutation it is permitted to perform, creating a privilege-escalation path from limited delegation to full subaccount control.

### Finding Description

In `EndpointTx.sol` the fast-path `LinkSigner` handler is:

```solidity
// EndpointTx.sol ~line 576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer via `getLinkedSigner`:

```solidity
// EndpointTx.sol ~line 143-157
function getLinkedSigner(bytes32 subaccount) public view virtual returns (address) {
    return
        RiskHelper.isIsolatedSubaccount(subaccount)
            ? linkedSigners[IOffchainExchange(offchainExchange).getParentSubaccount(subaccount)]
            : linkedSigners[subaccount];
}
``` [2](#0-1) 

The signature check therefore passes if the signature comes from **either** the subaccount owner address **or** the currently registered linked signer. Because the handler then unconditionally writes `linkedSigners[signedTx.tx.sender] = <attacker-chosen address>`, the linked signer can:

1. Craft a `SignedLinkSigner` transaction naming `sender = victim_subaccount`, `signer = attacker_address`.
2. Sign it with the currently registered linked-signer key.
3. Submit it through the sequencer fast path.
4. `linkedSigners[victim_subaccount]` is now set to `attacker_address`.
5. The attacker then signs `WithdrawCollateral` or `LiquidateSubaccount` transactions as the new linked signer to drain or destabilize the subaccount.

The root cause is structurally identical to the reference bug: a **single shared authorization mapping** (`linkedSigners`) is used both to authorize operational transactions (trading, withdrawals) **and** to authorize the mutation of that same mapping. There is no separate, higher-privilege check for the `LinkSigner` operation itself.

The isolated-subaccount path compounds the impact: because `getLinkedSigner` returns the **parent's** linked signer for any isolated subaccount, a single linked-signer key compromise propagates to every isolated subaccount derived from that parent. [3](#0-2) 

### Impact Explanation

A linked signer — typically a hot wallet or automated API key — can permanently replace itself with an attacker-controlled address. The attacker then holds a valid linked signer for the victim subaccount and can sign `WithdrawCollateral` (or `LiquidateSubaccount` against the victim) transactions, resulting in direct theft of collateral. For subaccounts with isolated children, the same key controls all child positions simultaneously.

**Severity: Medium** (mirrors the reference finding). Precondition is that the linked signer key is malicious or compromised, which is a realistic threat model for hot-wallet API keys used in automated trading.

### Likelihood Explanation

Linked signers are explicitly designed for automated/programmatic use (API bots, market-makers). These keys are higher-exposure than cold-wallet owner keys. A user who sets a linked signer for convenience and does not monitor `LinkSigner` events is fully exposed. The attack requires no on-chain capital and no admin access — only possession of the linked-signer private key.

### Recommendation

Remove `allowLinkedSigner = true` from the fast-path `LinkSigner` handler. The `LinkSigner` mutation is a privileged account-management operation and must only be authorized by the subaccount owner key, not by the currently delegated linked signer. Use `allowLinkedSigner = false` (or a dedicated owner-only validation path) for this transaction type, analogous to the recommendation in the reference report to use separate mappings for separate roles.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // owner signature required; linked signer must not self-escalate
);
```

### Proof of Concept

1. Alice owns subaccount `A`. She sets `linkedSigners[A] = hotWallet` via a slow-mode `LinkSigner` transaction.
2. Attacker obtains `hotWallet` private key (e.g., leaked API key).
3. Attacker constructs `SignedLinkSigner { sender: A, signer: attacker, nonce: next_nonce }` and signs it with `hotWallet`.
4. Attacker submits the transaction to the sequencer fast path.
5. `validateSignedTx(..., true)` passes because `recovered == hotWallet == getLinkedSigner(A)`.
6. `linkedSigners[A]` is overwritten to `attacker`.
7. Attacker signs a `WithdrawCollateral` transaction for subaccount `A` using `attacker` key.
8. `validateSignedTx(..., true)` passes because `recovered == attacker == getLinkedSigner(A)`.
9. Alice's collateral is withdrawn to the attacker's address. [4](#0-3)
