### Title
Gas-key nonce-fee computation uses wire length instead of trie-id length for ML-DSA-65 keys in host-function path — (`File: runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

### Summary

When a contract adds an ML-DSA-65 gas key via the `promise_batch_action_add_gas_key_with_full_access` or `promise_batch_action_add_gas_key_with_function_call` host functions, the per-nonce exec fee is computed using the raw wire/borsh length of the public key (1953 bytes) instead of the on-trie identifier length (33 bytes). The transaction path correctly uses `public_key.trie_id_len()` (33 bytes). The divergence is 1920 bytes per nonce, causing a ~59× overcharge in the per-byte component of the nonce-write exec fee for every ML-DSA-65 gas key added via a contract promise.

### Finding Description

`gas_key_add_key_exec_fee()` in `core/parameters/src/cost.rs` computes the exec fee for writing nonce trie entries as:

```
nonce_key_len = access_key_key_len(account_id_len, public_key_len) + sizeof(NonceIndex)
per_byte_fee  = gas_key_byte_exec × (nonce_key_len + NONCE_VALUE_LEN) × num_nonces
```

The `public_key_len` argument is supposed to reflect the actual on-trie key length. For ed25519 and secp256k1, the wire length equals the trie-id length (33 and 65 bytes respectively), so the two paths agree. For ML-DSA-65, they diverge: wire length = 1953 bytes, trie-id length = 33 bytes.

**Transaction path** (`runtime/runtime/src/config.rs`, `permission_exec_fees`): correctly passes `public_key.trie_id_len()` = 33 bytes. [1](#0-0) 

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`, `promise_batch_action_add_gas_key_with_full_access`): passes `public_key_len as usize`, which is the borsh wire length supplied by the calling contract — 1953 bytes for ML-DSA-65. [2](#0-1) 

The same error appears in the wasmtime path: [3](#0-2) 

The fee helper itself: [4](#0-3) 

The correct trie-id length for ML-DSA-65 is 33 bytes (1 tag + 32-byte SHA3-256 hash), not 1953 bytes: [5](#0-4) 

The parity test `test_gas_key_fee_parity` only exercises `KeyType::ED25519`, where wire length equals trie-id length, so the divergence is invisible to the test suite: [6](#0-5) 

### Impact Explanation

For each ML-DSA-65 gas key added via a host function with `N` nonces, the per-byte exec fee is computed over 1953 bytes instead of 33 bytes — a factor of ~59×. The overcharge per nonce is `1920 × gas_key_byte_exec_fee`. With `num_nonces` up to 65535 (u16::MAX), the total overcharge can far exceed the per-transaction gas limit (300 Tgas), making it impossible for any contract to successfully add an ML-DSA-65 gas key with more than a handful of nonces via a host function, even with the maximum gas attached. This breaks the protocol invariant that the same logical operation costs the same gas regardless of whether it is submitted as a transaction or emitted as a contract receipt.

### Likelihood Explanation

`PostQuantumSignatures` is stabilized at protocol version 85 and `GasKeys` is also a production feature. Any contract running on a node at or above that version can call `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key. No privileged role is required. The trigger is purely user-controlled (a contract call with a 1953-byte ML-DSA-65 pubkey blob).

### Recommendation

In both `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call` (in both `logic.rs` and `wasmtime_runner/logic.rs`), decode the public key first, then pass `public_key.decode()?.trie_id_len()` (or equivalently `PublicKeyHandle::from(&decoded_key).trie_id_len()`) to `gas_key_add_key_exec_fee()` instead of the raw `public_key_len as usize`. This mirrors what `permission_exec_fees()` already does in the transaction path.

### Proof of Concept

**Exact divergent values for ML-DSA-65:**

| Path | `public_key_len` passed to `gas_key_add_key_exec_fee` | Correct value |
|---|---|---|
| Transaction (`config.rs`) | `public_key.trie_id_len()` = **33** | 33 |
| Host fn (`logic.rs`) | `public_key_len as usize` = **1953** | 33 |

**Fee overcharge per nonce** = `(1953 − 33) × gas_key_byte_exec_fee` = `1920 × gas_key_byte_exec_fee`

**Concrete call path:**
1. Contract calls `promise_batch_action_add_gas_key_with_full_access(promise_idx, 1953, ptr_to_mldsa65_pubkey, num_nonces)`
2. Host function reads `public_key_len = 1953`
3. Calls `gas_key_add_key_exec_fee(fees, receiver_id.len(), 1953, num_nonces)` — wrong
4. Correct call would be `gas_key_add_key_exec_fee(fees, receiver_id.len(), 33, num_nonces)`
5. With `num_nonces = 100`, overcharge ≈ `192,000 × gas_key_byte_exec_fee` gas units

The storage-usage accounting in `add_gas_key` correctly uses `public_key.trie_id_len()` via `gas_key_storage_cost`, confirming the trie-id length is the right value to use: [7](#0-6)

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
```
