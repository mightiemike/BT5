### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Accepts Forged Merkle Proofs (64-Byte Internal-Node Attack) — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, unpermissioned NEAR contract method. It validates a Merkle proof by computing the root and comparing it to the block's `merkle_root`, but it never checks that the supplied `tx_id` is a leaf node (a real transaction) rather than an internal Merkle-tree node. Any unprivileged NEAR caller — or any recipient contract that has not migrated to v2 — can supply an internal node hash as `tx_id` with a matching sibling path and receive `true`, forging proof of a Bitcoin transaction that never existed.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust source, but `#[deprecated]` only emits compiler warnings for Rust callers; it does not remove the method from the on-chain ABI. The method is still decorated with `#[pause]` (not `#[private]`), meaning it is publicly callable by any NEAR account unless an operator has explicitly paused it. [1](#0-0) 

The verification logic reduces to a single comparison:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` simply walks the sibling path and hashes upward; it has no mechanism to distinguish a leaf hash from an internal-node hash: [3](#0-2) 

In Bitcoin's Merkle tree every internal node at depth *d* already satisfies a valid Merkle proof of length *d*. An attacker who knows the Merkle tree of any confirmed block (public information) can pick any internal node `N` at depth *d*, set `tx_id = N`, set `tx_index` to the correct parity path, and supply the *d* siblings. `compute_root_from_merkle_proof` will reproduce `merkle_root` exactly, and the function returns `true` — proof of a transaction that does not exist.

The only guard against an empty proof is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [4](#0-3) 

This does not prevent the attack; it only requires at least one sibling, which the attacker trivially provides.

The code's own warning acknowledges the broken invariant:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [5](#0-4) 

`verify_transaction_inclusion_v2` was introduced to close this gap by requiring a coinbase proof of equal depth, but it does not remove or gate the v1 method: [6](#0-5) 

---

### Impact Explanation

Any recipient contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., minting wrapped BTC, releasing escrowed funds, crediting a cross-chain deposit) can be made to accept a forged proof. The attacker does not need to produce a real Bitcoin transaction; they only need to identify an internal Merkle node in any block already stored by the light client and construct the corresponding sibling path. The corrupted state is the **proof result** returned to the caller: `true` is returned for a transaction that was never broadcast or confirmed on Bitcoin.

---

### Likelihood Explanation

- All required inputs (block Merkle tree structure, sibling hashes) are public on-chain Bitcoin data.
- No privileged role, private key, or social engineering is required.
- The method is callable by any NEAR account with no deposit or staking requirement.
- Likelihood is conditional on a recipient contract still using the v1 API, which is a realistic scenario during any migration window or for integrators who have not read the deprecation notice.

**Impact: High** (forged cross-chain transaction proof)
**Likelihood: Medium** (requires an un-migrated consumer; attack inputs are freely available)

---

### Recommendation

1. **Remove or gate the v1 method**: Add `#[private]` to `verify_transaction_inclusion`, or replace `#[pause]` with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` body, so it is no longer callable from outside the contract.
2. **Do not delegate to v1 from v2**: `verify_transaction_inclusion_v2` currently calls `self.verify_transaction_inclusion(args.into())` after the coinbase check. The transaction proof should be re-validated inline in v2 without routing through the deprecated path, to avoid any future regression.
3. **Enforce leaf-node validation**: Add a depth check — the proof length must equal `log2(block_tx_count).ceil()` — or require callers to supply the raw transaction preimage so the contract can hash it and confirm it matches `tx_id`.

---

### Proof of Concept

1. Identify any block `B` stored in the light client's `headers_pool` with at least two transactions (Merkle tree depth ≥ 1).
2. Obtain the Merkle tree of `B` from a Bitcoin node or block explorer (public data).
3. Select any internal node `N` at depth 1 (i.e., `N = hash(tx0, tx1)`).
4. The sibling of `N` at depth 1 is `S = hash(tx2, tx3)` (or the duplicate of `N` if the tree is odd).
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = B.block_hash`
   - `tx_index = 0` (even position → sibling is on the right)
   - `merkle_proof = [S, ...]` (remaining siblings up to root)
   - `confirmations = 1`
6. The function computes `hash(N, S) == merkle_root` → returns `true`, falsely asserting that `N` is a confirmed Bitcoin transaction in block `B`. [7](#0-6) [3](#0-2)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
