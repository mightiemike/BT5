### Title
Missing `gc_threshold` Validation in `init` Silently Breaks SPV Confirmation Security — (`contract/src/lib.rs`)

---

### Summary

The `init` function of `BtcLightClient` accepts `gc_threshold` from `InitArgs` and stores it without any non-zero guard. If `gc_threshold` is set to `0`, the `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` APIs permanently accept only `confirmations = 0`, stripping the SPV proof of its core security guarantee. This is a direct analog to the ENS constructor's missing parameter validation: a critical invariant is silently violated at initialization time, corrupting every downstream proof result.

---

### Finding Description

In `contract/src/lib.rs`, the `init` function assigns `gc_threshold` directly from caller-supplied `InitArgs` with no validation: [1](#0-0) 

`gc_threshold` is documented to represent how many blocks are retained in on-chain storage (recommended: `52704`, approximately one year of Bitcoin blocks). It is used in two places:

**1. SPV confirmation gate** — `verify_transaction_inclusion` enforces: [2](#0-1) 

If `gc_threshold = 0`, this `require!` passes only when `args.confirmations == 0`. Every call with `confirmations >= 1` panics with "The required number of confirmations exceeds the number of blocks stored in memory." The same path is exercised by `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after its coinbase check. [3](#0-2) 

**2. GC loop** — `run_mainchain_gc` (called automatically inside every `submit_blocks`) computes: [4](#0-3) 

With `gc_threshold = 0`, `amount_of_headers_we_store > 0` is always true after genesis, so the GC loop fires on every block submission and removes as many headers as the batch size allows. Over time this drains the chain down to a single block, making historical proof lookups impossible.

The `InitArgs` struct itself imposes no constraint on `gc_threshold`: [5](#0-4) 

---

### Impact Explanation

A contract initialized with `gc_threshold = 0` permanently:

- **Breaks SPV verification**: `verify_transaction_inclusion{,_v2}` rejects every call with `confirmations >= 1`. Any consumer contract or off-chain service that requests even a single confirmation receives a panic, not a `false` — it cannot distinguish "proof invalid" from "contract misconfigured."
- **Silently accepts 0-confirmation proofs**: If a caller passes `confirmations = 0`, the guard passes and the Merkle proof is evaluated against a block that may be the chain tip itself — providing no finality guarantee whatsoever.
- **Progressively destroys chain history**: The GC loop removes stored headers on every `submit_blocks` call, eventually making it impossible to look up any historical block, breaking `get_block_hash_by_height`, `get_height_by_block_hash`, and all proof paths that require a historical `tx_block_blockhash`.

The corrupted invariant is: *`gc_threshold` must be ≥ the maximum `confirmations` any caller will ever request*. Setting it to `0` violates this invariant permanently and irrecoverably (there is no setter for `gc_threshold` after `init`).

---

### Likelihood Explanation

The `init` function is `#[private]`, callable only by the contract account at deployment. This is the same deployment-time misconfiguration class as the ENS constructor issue. The risk is realistic because:

1. The parameter name `gc_threshold` does not obviously signal that `0` is catastrophic — a deployer might interpret it as "no GC" rather than "no confirmations allowed."
2. The only documentation is a comment (`/// gc_threshold = 52704`) with no enforced lower bound.
3. There is no post-deployment mechanism to correct the value; the contract must be redeployed.

---

### Recommendation

Add an explicit non-zero guard in `init` before assigning `gc_threshold`:

```rust
require!(
    args.gc_threshold > 0,
    "gc_threshold must be greater than zero"
);
```

A stronger guard would also enforce a minimum equal to `MEDIAN_TIME_SPAN` (currently `11`) to ensure the GC window is always large enough for MTP computation:

```rust
require!(
    args.gc_threshold >= btc_types::network::MEDIAN_TIME_SPAN as u64,
    "gc_threshold must be at least MEDIAN_TIME_SPAN"
);
``` [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract with `InitArgs { gc_threshold: 0, ... }` (all other fields valid).
2. Call `verify_transaction_inclusion` with `confirmations: 1`.
3. The call panics: `"The required number of confirmations exceeds the number of blocks stored in memory"` — even though blocks exist in the pool.
4. Call `verify_transaction_inclusion` with `confirmations: 0`.
5. The call succeeds and returns a Merkle proof result against the chain tip with zero finality, providing no security.
6. Submit 50 blocks in batches of 10. Observe that `run_mainchain_gc` removes 10 headers per batch (since `amount_of_headers_we_store > 0` always). After enough submissions, `get_block_hash_by_height` for early heights returns `None` and historical proof calls panic. [7](#0-6)
