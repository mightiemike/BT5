### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass the Swap Allowlist — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. The pool always passes `msg.sender` (the direct caller of `pool.swap()`) as `sender`. When a user routes through `MetricOmmSimpleRouter`, the direct caller is the router contract, not the user. If the router is allowlisted for a pool — the natural configuration for any pool that wants its allowlisted users to be able to use the standard periphery — every non-allowlisted user can bypass the guard by routing through the router.

---

### Finding Description

**Call chain:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ..., extensionData)   // msg.sender = router
              → _beforeSwap(msg.sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → allowedSwapper[pool][router]  ← checked, NOT the user
```

In `MetricOmmPool.swap`, the pool passes its own `msg.sender` as the `sender` argument to every before-swap hook:

```solidity
_beforeSwap(
  msg.sender,   // ← always the direct caller of pool.swap()
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` = pool and `sender` = the first argument = the router. The actual originating user is never consulted.

A pool admin who wants allowlisted users to be able to use `MetricOmmSimpleRouter` must add the router to `allowedSwapper[pool]`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for every call that arrives through the router, regardless of who called the router. Any non-allowlisted user can therefore execute a swap on the curated pool by calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router.

The same bypass applies to multi-hop `exactOutput` paths: intermediate hops are triggered from within `_exactOutputIterateCallback`, which is itself called by the pool's swap callback, so the pool still sees `msg.sender = router` for every hop.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool — the admin intends to restrict trading to a specific set of addresses (e.g., KYC-verified counterparties, whitelisted market makers, or regulated participants). Once the router is allowlisted (the only way to let allowlisted users use the standard periphery), the allowlist is completely nullified for router-mediated swaps. Any address can trade on the pool, draining LP value at oracle-anchored prices that were only intended to be accessible to vetted participants. This is a direct loss of the curation guarantee and constitutes a broken core pool functionality / admin-boundary break under the allowed impact gate.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the protocol's standard swap entry point. Any pool admin who deploys a `SwapAllowlistExtension` and wants their allowlisted users to be able to use the router must add the router to the allowlist. This is the expected operational configuration; the alternative (not allowlisting the router) silently breaks the router for all users of that pool. The bypass therefore activates under the most natural deployment scenario and requires no special privileges or unusual inputs from the attacker — a single call to `exactInputSingle` suffices.

---

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the originating economic actor, not the intermediary contract. Two complementary fixes:

1. **Router-side:** Have `MetricOmmSimpleRouter` forward the originating user's address through `extensionData` on every hop, and document this as the canonical identity field for allowlist extensions.
2. **Extension-side:** `SwapAllowlistExtension.beforeSwap` should decode the originating user from `extensionData` when `sender` is a known router, or alternatively reject any `sender` that is not itself an EOA or an explicitly allowlisted contract, so that router-mediated calls cannot pass unless the router is treated as a trusted forwarder with a verifiable user identity.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]  = true   // alice is KYC'd
  allowedSwapper[pool][router] = true   // router allowlisted so alice can use it
  allowedSwapper[pool][bob]    = false  // bob is NOT allowlisted

Attack (bob bypasses the allowlist):
  bob calls router.exactInputSingle({
      pool:      pool,
      recipient: bob,
      ...
  })

  pool.swap(msg.sender=router, ...)
  _beforeSwap(sender=router, ...)
  SwapAllowlistExtension.beforeSwap:
      allowedSwapper[pool][router] == true  → passes
  swap executes for bob
```

Bob successfully swaps on a pool he is not allowlisted for. The guard checked `allowedSwapper[pool][router]` (the wrong actor) instead of `allowedSwapper[pool][bob]` (the originating user), directly mirroring the external report's pattern of acting on the wrong account.
