Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument, which is the pool's `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable standard periphery access inadvertently grants every user who calls the router permission to swap, rendering the per-user allowlist a no-op and exposing curated pool liquidity to any unpermissioned trader.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (correct). `sender` is whatever the pool passed as the first argument.

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, binding the pool's `msg.sender` as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim as the first argument to every extension: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user's address: [3](#0-2) 

The pool's `msg.sender` is therefore the router contract. The extension sees `sender = router`, not `sender = user`. The check `allowedSwapper[pool][router]` passes for every user who calls the router, regardless of whether that user is individually permitted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` — an explicit parameter separate from `msg.sender` that the caller sets to the actual user: [4](#0-3) 

`swap` has no equivalent "owner" parameter; the only identity available to the extension is `sender`, which collapses to the router when the periphery is used. [5](#0-4) 

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties or protocol-controlled addresses) must allowlist the router to support standard periphery access. Once `allowedSwapper[pool][router] = true`, the check passes for every user who calls `exactInputSingle` or `exactInput` on the router, regardless of individual allowlist status. Any unpermissioned user can execute swaps against the curated pool's liquidity, constituting a direct loss of LP principal on pools whose security model depends on the allowlist. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool admin who wants to support normal user flows will allowlist the router. The bypass requires no special privileges, no flash loans, and no exotic token behavior — any EOA can call `exactInputSingle` on the router pointing at the curated pool. The precondition (router allowlisted) is the expected operational state, not an edge case.

## Recommendation
The extension must gate the economically relevant actor, not the intermediary. Two options:

1. **Pass the originating user through the pool**: Add a `swapper` or `payer` field to the swap call path (analogous to how `addLiquidity` separates `owner` from `msg.sender`) so the extension can check the real initiator.

2. **Check `recipient` as a proxy**: If the pool's design guarantees that the recipient is always the user, the extension could check `recipient` instead of `sender`. However, this is fragile because `recipient` can be set to any address.

The cleanest fix mirrors the deposit path: the pool should forward the originating user's address as a distinct parameter to swap extensions, separate from the caller (`msg.sender`).

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)   // enable router path
  - Pool admin does NOT allowlist attacker address
  - Pool admin adds liquidity from allowlisted LP

Attack:
  - Attacker (not in allowedSwapper) calls:
      router.exactInputSingle({pool: curatedPool, ...})
  - Router calls pool.swap(...) → pool's msg.sender = router
  - _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] = true → PASSES
  - Attacker executes swap against curated pool's liquidity
  - Allowlist is completely bypassed

The check that should have blocked the attacker — allowedSwapper[pool][attacker] — is never
evaluated. The extension evaluates allowedSwapper[pool][router] instead, which is true by
necessity of the pool's configuration.
```
