### Title
Pre-existing NLP Holders Permanently Blocked from Burning After Retroactive Lock-Check Introduction — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` enforces `getNlpUnlockedBalance(txn.sender).amount >= nlpAmount` for every caller. Users who minted NLP before the `handleNlpLockedBalance` queue mechanism was introduced (e.g., via a contract upgrade) have no entries in `nlpLockedBalanceQueues`, so their `unlockedBalanceSum` is permanently zero. These users are completely blocked from burning their NLP — a direct analog to the Penpie retroactive-condition bug.

---

### Finding Description

When a user mints NLP, `Clearinghouse.mintNlp` calls `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount)`. Inside `SpotEngine.updateBalance`, `handleNlpLockedBalance` is invoked for `NLP_PRODUCT_ID`: [1](#0-0) 

`handleNlpLockedBalance` pushes a `NlpLockedBalance` entry into `nlpLockedBalanceQueues[subaccount]` with `unlockedAt = getOracleTime() + NLP_LOCK_PERIOD` (4 days): [2](#0-1) 

`burnNlp` then gates every burn on the user having sufficient unlocked balance: [3](#0-2) 

`getNlpUnlockedBalance` calls `tryUnlockNlpBalance`, which iterates the queue and accumulates `unlockedBalanceSum`: [4](#0-3) 

**The gap**: If a user minted NLP before `handleNlpLockedBalance` was wired into `updateBalance` (i.e., before the upgrade that introduced the locking mechanism), their `nlpLockedBalanceQueues[subaccount]` is entirely empty — `balanceCount == 0`, `unlockedBalanceSum.amount == 0`. `tryUnlockNlpBalance` has nothing to process, so `unlockedBalanceSum` stays at zero forever. The `burnNlp` check then unconditionally reverts for these users, regardless of how large their actual NLP balance is.

There is no code path that can retroactively seed `unlockedBalanceSum` for pre-existing holders. The only way to add to the queue is through a new mint, and the only way to add to `unlockedBalanceSum` is by processing queue entries that do not exist.

---

### Impact Explanation

Any user who held NLP before the locking queue was introduced is permanently unable to burn their NLP. Their NLP balance (tracked in `balances[NLP_PRODUCT_ID][subaccount]`) is non-zero, but `getNlpUnlockedBalance` always returns zero. The `ERR_UNLOCKED_NLP_INSUFFICIENT` revert is unavoidable without a new mint to seed the queue. This constitutes a permanent, irreversible asset lock for affected users — a concrete, non-theoretical asset delta corruption. [5](#0-4) 

---

### Likelihood Explanation

The NLP locking mechanism (`handleNlpLockedBalance`, `NlpLockedBalanceQueue`, `tryUnlockNlpBalance`) is a non-trivial feature that was added to an upgradeable contract. Any user who minted NLP in a prior deployment version — before this feature was activated — is affected. Given that `NLP_PRODUCT_ID` is a live product and the contracts are upgradeable via `ProxyManager`, the scenario of pre-existing NLP holders is realistic and likely. The impact is total (100% burn block), not partial. [6](#0-5) 

---

### Recommendation

When the locking mechanism is introduced via upgrade, a migration step should initialize `nlpLockedBalanceQueues[subaccount].unlockedBalanceSum.amount` to the user's existing NLP balance for all pre-existing holders. Alternatively, `burnNlp` should only enforce the `getNlpUnlockedBalance` check for users whose `nlpLockedBalanceQueues[subaccount].balanceCount > 0` or whose first mint occurred after the locking mechanism was activated (tracked via a `lockingActivatedAt` timestamp). This mirrors the Penpie recommendation: only apply the new condition to users who interacted after the condition was set.

---

### Proof of Concept

1. At time T1, user Alice mints NLP via `MintNlp`. At this point, the `handleNlpLockedBalance` function does not yet exist in `updateBalance` (pre-upgrade). Alice's balance: `balances[NLP_PRODUCT_ID][alice] = 1000e18`, `nlpLockedBalanceQueues[alice]` = empty.

2. Protocol upgrades `SpotEngine` to add `handleNlpLockedBalance` into `updateBalance` for `NLP_PRODUCT_ID`.

3. Alice submits a `BurnNlp` transaction for `nlpAmount = 500e18`.

4. `burnNlp` calls `spotEngine.getNlpUnlockedBalance(alice)`:
   - `tryUnlockNlpBalance(alice)` runs: `queue.unlockedUpTo (0) < queue.balanceCount (0)` is false → loop never executes.
   - Returns `unlockedBalanceSum.amount = 0`.

5. Check: `require(0 >= 500e18, ERR_UNLOCKED_NLP_INSUFFICIENT)` → **reverts**.

6. Alice cannot burn any NLP. Her 1000e18 NLP is permanently locked. [7](#0-6) [8](#0-7)

```text
    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }
```
