### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by adding mandatory coinbase proof validation. However, the deprecated `verify_transaction_inclusion` (v1) remains a fully public, unrestricted NEAR contract method. Any unprivileged caller can invoke v1 directly, bypassing the coinbase proof guard entirely — an exact structural analog to M-26, where a protection check is present in one code path but absent in another reachable path.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced as the safe replacement for v1: [1](#0-0) 

Its doc-comment explicitly states the reason for deprecation: [2](#0-1) 

The v2 function enforces the coinbase proof check before delegating to v1: [3](#0-2) 

The critical issue is that v1 is still decorated with `#[pause]` (a runtime gate) but carries **no coinbase-proof gate**: [4](#0-3) 

In Rust, `#[deprecated]` is a **compile-time lint only**. It emits a compiler warning to crate authors who call the function in Rust source, but it imposes **zero runtime restriction** on NEAR callers invoking the method via a signed transaction. Any account on NEAR can call `verify_transaction_inclusion` directly, receiving a `bool` result without the coinbase proof ever being checked.

The v1 function's own warning acknowledges the risk: [5](#0-4) 

The assumption that "validation is performed at a higher level" is broken the moment v1 is callable as a standalone NEAR entry point — which it is.

---

### Impact Explanation

The 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to supply a `tx_id` that is actually the hash of an internal Merkle tree node rather than a real transaction. Because v1 computes only:

```
compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.merkle_root
```

without verifying that `tx_id` is a leaf (real transaction) rather than an internal node, a crafted proof can satisfy this equality for a transaction that was never broadcast or mined. Any downstream NEAR contract or protocol that calls `verify_transaction_inclusion` directly — rather than v2 — will accept this forged proof as `true`, potentially releasing funds or triggering irreversible state changes based on a Bitcoin transaction that does not exist.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can submit a signed transaction calling `verify_transaction_inclusion`. The function is not gated by `#[trusted_relayer]`, `#[private]`, or any role check beyond `#[pause]`. Downstream integrators reading the contract ABI will see both v1 and v2 as callable methods; those who call v1 (whether by mistake or by design) receive no runtime error and no indication that the coinbase guard was skipped. The deprecation notice is invisible at the NEAR RPC level.

---

### Recommendation

**Short term**: Add an explicit `env::panic_str` or `require!(false, "use verify_transaction_inclusion_v2")` at the top of `verify_transaction_inclusion` to make it a hard runtime error for any direct caller, preserving the internal call from v2 by extracting the shared logic into a private helper.

**Long term**: Refactor the shared Merkle-root comparison into a private `fn verify_tx_merkle_proof(...)` method. Remove the public `verify_transaction_inclusion` entry point entirely. This mirrors the M-26 long-term recommendation: move the protection into the only reachable code path rather than relying on callers to choose the correct entry point.

---

### Proof of Concept

1. Deploy the BTC light client contract with real Bitcoin headers.
2. Identify a block `B` in the main chain whose Merkle tree has at least one internal node `N` (64 bytes = two 32-byte child hashes concatenated).
3. Construct a Merkle proof `P` such that `compute_root_from_merkle_proof(N, index, P) == B.merkle_root`. This is feasible because `N` is itself a node in the tree.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = index`, `merkle_proof = P`, `confirmations = 1`.
5. The function returns `true` — confirming a Bitcoin transaction that was never mined.
6. Repeat step 4 calling `verify_transaction_inclusion_v2` with the same arguments but without a valid `coinbase_merkle_proof` — the call panics with "Incorrect coinbase merkle proof", demonstrating that v2 blocks the attack while v1 does not. [6](#0-5)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
