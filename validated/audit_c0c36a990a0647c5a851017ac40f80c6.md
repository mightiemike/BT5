### Title
`AllResources`-to-`L1Gas` silent coercion in protobuf deserializer produces a different transaction hash than the one committed at submission time ŌĆö (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `get_tip_resource_bounds_hash` function produces structurally different hashes for `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources` when `l1_data_gas = 0` and `l2_gas = 0`. The protobuf deserializer (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`) silently coerces `AllResources` with those zero values into `L1Gas`. A transaction submitted through the gateway with `AllResources` (only `l1_gas` non-zero) is hashed under the `AllResources` domain at admission time, but after a protobuf round-trip the same transaction is hashed under the `L1Gas` domain, producing a different hash. This breaks hash canonicalization across the serialization boundary.

---

### Finding Description

**Step 1 ŌĆō Submission path always uses `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (never `ValidResourceBounds::L1Gas`). [1](#0-0) 

`InternalRpcInvokeTransactionV3` also stores `AllResourceBounds`. Its `InvokeTransactionV3Trait` implementation always wraps it as `ValidResourceBounds::AllResources`. [2](#0-1) 

**Step 2 ŌĆō Hash domain for `AllResources` includes `L1_DATA_GAS`; `L1Gas` does not.**

`get_tip_resource_bounds_hash` appends the `L1_DATA_GAS` concat-field only for `AllResources`. For `L1Gas` that field is absent from the hash preimage entirely. [3](#0-2) 

Concretely, for a transaction with `l1_gas = X`, `l2_gas = 0`, `l1_data_gas = 0`:

- `AllResources` hash: `poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))`
- `L1Gas` hash: `poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0))`

These are distinct Poseidon outputs. [4](#0-3) 

**Step 3 ŌĆō Protobuf serialization erases the `AllResources`/`L1Gas` distinction.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `L1Gas` and `AllResources`-with-zeros to the same wire bytes (l1_gas present, l2_gas = 0, l1_data_gas = 0). [5](#0-4) 

**Step 4 ŌĆō Deserializer silently coerces back to `L1Gas`.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` returns `L1Gas` whenever `l1_data_gas.is_zero() && l2_gas.is_zero()`, regardless of whether the sender originally used `AllResources`. [6](#0-5) 

**Step 5 ŌĆō Hash recomputed under the wrong domain.**

`get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash` with the deserialized `ValidResourceBounds`. After the round-trip the variant is `L1Gas`, so the hash omits `L1_DATA_GAS` and diverges from the hash committed at submission time. [7](#0-6) 

The `validate_transaction_hash` function compares the recomputed hash against the stored one; the mismatch causes validation to return `false` for a legitimately submitted transaction. [8](#0-7) 

---

### Impact Explanation

A transaction submitted via the gateway with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted and its hash `H_all` is stored in the block. Any node that later receives this transaction over p2p (historical sync) deserializes it as `L1Gas`, recomputes hash `H_l1 ŌēĀ H_all`, and fails hash validation. This causes the syncing node to reject a valid committed transaction, breaking state synchronization and potentially causing a chain split. The impact matches: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

---

### Likelihood Explanation

The gateway's stateless validator explicitly accepts `AllResourceBounds` with only `l1_gas` non-zero (l2_gas = 0, l1_data_gas = 0) as a valid transaction. [9](#0-8)  Any user who submits such a transaction triggers the mismatch. No special privilege is required.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (`crates/apollo_protobuf/src/converters/transaction.rs`), remove the coercion to `L1Gas` for transactions that arrived as `AllResources`. The `L1Gas` variant should only be reconstructed when the wire message explicitly signals a pre-0.13.3 transaction (e.g., when `l1_data_gas` is absent from the protobuf message, not merely zero). The existing comment acknowledges this: `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.` [10](#0-9) 

Alternatively, `get_tip_resource_bounds_hash` should produce identical output for `L1Gas` and `AllResources`-with-zeros so that the two variants are hash-equivalent, eliminating the domain collision.

---

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);

        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
