### Title
Symmetric Funding Rate Application Creates Bad Debt When Long/Short OI Is Imbalanced — (`File: core/contracts/PerpEngineState.sol`)

### Summary
`PerpEngineState.updateStates` applies the same per-unit `paymentAmount` to both `cumulativeFundingLongX18` and `cumulativeFundingShortX18` without any scaling for OI imbalance. Because Nado tracks only a single combined `openInterest` (longs + shorts), when `totalLongOI != totalShortOI` the total funding debited from one side does not equal the total funding credited to the other side, creating unbacked virtual quote balances (bad debt).

### Finding Description

In `updateStates` (`PerpEngineState.sol` lines 130–132):

```solidity
int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
state.cumulativeFundingLongX18 += paymentAmount;
state.cumulativeFundingShortX18 += paymentAmount;
```

The same `paymentAmount` is added to both accumulators unconditionally. In `_updateBalance` (lines 31–36), each position's realized funding is:

```solidity
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
```

For a long with `amount = L > 0`: `vQuoteBalance -= paymentAmount * L`
For a short with `amount = -S < 0`: `vQuoteBalance += paymentAmount * S`

Net change across all traders = `paymentAmount * (totalShortOI - totalLongOI)`

When `totalShortOI > totalLongOI` and longs pay (`paymentAmount > 0`), shorts collectively receive more than longs collectively pay. The surplus is not backed by any real asset — `availableSettle` is never credited in `updateStates`, and no insurance mechanism compensates for this imbalance. The inverse holds when `totalLongOI > totalShortOI` and shorts pay.

Nado has no separate tracking of long vs. short OI — `state.openInterest` is the sum of all absolute position sizes — so the protocol has no information to apply the correct scaling factor.

### Impact Explanation

**Impact: Medium**

Each `updateStates` call where OI is imbalanced silently creates bad debt equal to:

```
|paymentAmount| * |totalLongOI - totalShortOI|
```

This bad debt accumulates in `vQuoteBalance` of traders on the receiving side. When those traders close positions or call `settlePnl`, the `availableSettle` pool (which was never credited with the surplus funding) is insufficient to cover settlement, causing either failed settlements or socialized losses via `socializeSubaccount`. Over time, with a maximum daily funding rate of 2% (`MAX_DAILY_FUNDING_RATE = 20000000000000000`) and persistent OI imbalance, the bad debt can become material.

### Likelihood Explanation

**Likelihood: High**

OI imbalance (`totalLongOI != totalShortOI`) is the normal state of any perpetual market. There is no mechanism in Nado that enforces or incentivizes OI balance. Every `updateStates` call during an imbalanced state silently corrupts the accounting. No special attacker action is required — ordinary trading activity is sufficient to trigger the condition.

### Recommendation

Track `openInterestLong` and `openInterestShort` separately in the `State` struct. In `updateStates`, scale the receiving side's cumulative rate so that total debited = total credited:

- When `paymentAmount > 0` (longs pay shorts):
  - `cumulativeFundingLongX18 += paymentAmount`
  - `cumulativeFundingShortX18 += paymentAmount * openInterestLong / openInterestShort`
- When `paymentAmount < 0` (shorts pay longs):
  - `cumulativeFundingShortX18 += paymentAmount`
  - `cumulativeFundingLongX18 += paymentAmount * openInterestShort / openInterestLong`

Update `_updateBalance` to maintain the separate OI counters accordingly.

### Proof of Concept

**Setup:** `totalLongOI = 100`, `totalShortOI = 200`, `availableSettle = 0`

1. Endpoint calls `updateStates` with `priceDiffX18 > 0` → `paymentAmount = 1`.
2. Both `cumulativeFundingLongX18 += 1` and `cumulativeFundingShortX18 += 1`.
3. Long trader (amount = 100) updates position: `deltaQuote = -(1)(100) = -100` → `vQuoteBalance -= 100`.
4. Short trader (amount = -200) updates position: `deltaQuote = -(1)(-200) = +200` → `vQuoteBalance += 200`.
5. Net `vQuoteBalance` change across all traders = **+100**, unbacked by any real asset.
6. Short trader calls `settlePnl`; `canSettle = min(positionPnl, state.availableSettle)`. Since `availableSettle = 0`, settlement fails or is zero — the 100 units of surplus funding the short trader "earned" cannot be realized, or if `availableSettle` had prior balance it is drained by the phantom surplus.
