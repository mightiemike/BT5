Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Complete Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument against its allowlist, but `sender` is `msg.sender` from the pool's perspective — the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, that caller is the router contract, not the end user. A pool admin who allowlists the router to enable normal UX for permitted users inadvertently grants swap access to every address on-chain, completely defeating per-user access control and exposing LP funds to unauthorized counterparties.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

**Step 2 — SwapAllowlistExtension checks that `sender` against its allowlist.**

`beforeSwap` receives `sender` (the immediate pool caller) and checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool: [2](#0-1) 

**Step 3 — MetricOmmSimpleRouter is the immediate caller of `pool.swap()`.**

`exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router `msg.sender` from the pool's perspective: [3](#0-2) 

**The mismatch:** The pool admin intends to allowlist specific end-users (e.g., KYC'd traders). To let those users use the router, the admin also calls `setAllowedToSwap(pool, router, true)`. At that point `allowedSwapper[pool][router] = true`, and the check in `beforeSwap` passes for **every** user who calls through the router — the actual end-user identity (`msg.sender` of the router call) is never inspected. The router stores the real payer in transient storage for callback purposes only; it is never forwarded to the pool or extension as the `sender`. [4](#0-3) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for private/gated access (institutional-only, KYC-gated, partner-only) loses all access control the moment the router is allowlisted. Any unpermissioned address can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` and execute swaps against the pool. LPs in such a pool suffer adverse selection from counterparties they explicitly excluded, resulting in direct loss of LP principal through unfavorable trades. This constitutes a broken core pool functionality causing loss of funds and an admin-boundary break where an unprivileged path bypasses the intended access control.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical user-facing entry point for swaps. A pool admin who deploys a gated pool and wants permitted users to have normal UX will naturally allowlist the router. The `SwapAllowlistExtension` NatDoc says it "Gates `swap` by swapper address" without clarifying that the swapper is the immediate pool caller, not the end user. [5](#0-4) 

The misconfiguration is a predictable, non-adversarial mistake by a legitimate pool admin. No special attacker capability is required — any EOA can exploit this by calling the public router.

## Recommendation
The extension must resolve the true end-user identity rather than the immediate pool caller. Two viable approaches:

1. **`extensionData` attestation**: Require the end-user to sign a permit passed through `extensionData`; the extension verifies the signature and checks the recovered address against the allowlist.
2. **Trusted router attestation**: The router encodes the real `msg.sender` into `extensionData`, and the extension trusts only calls from a known, immutably registered router address that attests the real sender.

Additionally, `DepositAllowlistExtension` has a symmetric issue: it checks `owner` (the position beneficiary supplied by the caller) rather than `sender` (the actual token provider), allowing any address to deposit tokens on behalf of an allowlisted owner. [6](#0-5) 

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls: setAllowedToSwap(pool, userA, true)    // permit userA
  pool admin calls: setAllowedToSwap(pool, router, true)   // permit router so userA can use it

Attack:
  userB (not in allowlist) calls:
    MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})

Execution trace:
  router.exactInputSingle()                          // msg.sender = userB
    → pool.swap(recipient, ...)                      // msg.sender = router
      → _beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true  ✓  (check passes)
      → swap executes, tokens transferred to userB

Result: userB successfully swaps against the gated pool.
        Per-user allowlist is completely bypassed.
        LP funds exposed to unauthorized counterparty.
```

Foundry test plan: deploy pool with `SwapAllowlistExtension`, allowlist `userA` and the router, then call `exactInputSingle` from `userB` (not allowlisted) and assert the swap succeeds without revert.
