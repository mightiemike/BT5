### Title
`DepositAllowlistExtension` silently ignores `sender` and gates only `owner`, allowing any disallowed payer to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` (the actual `msg.sender` of the pool call, i.e., the token payer) and `owner` (the position recipient) but discards `sender` entirely. The allowlist check is performed only against `owner`. Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any disallowed address can call `addLiquidity(allowedOwner, ...)`, pass the extension gate, and fund a position — bypassing the allowlist entirely.

---

### Finding Description

**Step 1 — Pool entry point has no sender == owner guard for `addLiquidity`.**

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no restriction on who `msg.sender` may be: [1](#0-0) 

Compare with `removeLiquidity`, which explicitly enforces `msg.sender == owner`: [2](#0-1) 

No equivalent guard exists for `addLiquidity`.

**Step 2 — Pool passes both `sender` and `owner` to the extension.**

`ExtensionCalling._beforeAddLiquidity` encodes `(sender, owner, ...)` and dispatches to the configured extension: [3](#0-2) 

**Step 3 — Extension silently drops `sender` and checks only `owner`.**

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed and unused. The gate checks `allowedDepositor[msg.sender][owner]` where `msg.sender` is the pool address: [4](#0-3) 

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address, per pool," and the mapping is named `allowedDepositor`. The depositor in economic terms is the address providing tokens (the `sender`/payer), not the position recipient (`owner`). The check is on the wrong actor.

**Exploit path:**

1. Pool admin allowlists `allowedOwner` via `setAllowedToDeposit(pool, allowedOwner, true)`.
2. Disallowed attacker calls `pool.addLiquidity(allowedOwner, salt, deltas, callbackData, extensionData)`.
3. Pool calls `extension.beforeAddLiquidity(attacker, allowedOwner, ...)`.
4. Extension evaluates `allowedDepositor[pool][allowedOwner]` → `true`. Gate passes.
5. `LiquidityLib.addLiquidity` credits LP shares to `allowedOwner`.
6. Pool invokes the liquidity callback on `attacker` (the actual `msg.sender`) to pull tokens.
7. Attacker has successfully deposited into a restricted pool; the allowlist is defeated for the payer dimension.

---

### Impact Explanation

The deposit allowlist is the primary access-control mechanism for restricted pools. A disallowed address can:

- Interact with a KYC/compliance-gated pool by funding positions for any allowlisted owner.
- Force LP shares onto an allowlisted address without their consent (griefing / dust-position spam).
- Circumvent any economic or regulatory restriction the pool admin intended to enforce on who may provide liquidity.

The allowlist invariant — that only permitted addresses may deposit — is broken for the payer/sender dimension. The extension receives the correct `sender` value but discards it, making the guard structurally ineffective.

---

### Likelihood Explanation

Exploitability is trivial: `addLiquidity` is a public function with no access control beyond the extension hook. Any address that knows an allowlisted owner address (which is on-chain public state via `AllowedToDepositSet` events) can execute the bypass with a single transaction. No privileged role, special token, or complex setup is required.

---

### Recommendation

Change `beforeAddLiquidity` to check the `sender` parameter (the actual payer) instead of, or in addition to, `owner`:

```solidity
// Current (broken): ignores sender, checks owner only
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// Fixed: check sender (the token payer)
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender]
        && !allowedDepositor[msg.sender][sender]   // gate the payer
        && !allowedDepositor[msg.sender][owner])   // optionally also gate the owner
    {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

The exact policy (gate sender, gate owner, or gate both) should match the pool admin's intent, but at minimum `sender` must be checked to prevent the bypass.
