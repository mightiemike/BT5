### Title
Tier `BonusApy` Update via Governance Retroactively Miscalculates Accrued Bonus Rewards for All Positions in the Tier — (`x/tieredrewards/keeper/tier.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When governance updates a tier's parameters (including `BonusApy`) via `SetTier`, the function overwrites the tier in state without first settling accrued bonus rewards for all positions in that tier. Because `processEventsAndClaimBonus` fetches the **current** tier at claim time and applies its `BonusApy` retroactively to **all** historical time segments since the position's last claim, a governance-driven reduction of `BonusApy` silently destroys legitimately accrued bonus rewards for every position in the affected tier.

---

### Finding Description

`SetTier` is the single write path for tier parameters and is explicitly documented as being used by governance messages, genesis, and chain upgrades: [1](#0-0) 

It performs no pre-claim step — it simply validates and overwrites the tier record.

When a position owner later calls `ClaimTierRewards` (or any operation that triggers `claimRewards`), `processEventsAndClaimBonus` fetches the **current** tier from state: [2](#0-1) 

It then walks every validator event since `pos.LastBonusAccrual` and computes the bonus for each historical segment using the **now-current** `tier.BonusApy`: [3](#0-2) 

The segment bonus formula in `computeSegmentBonus` multiplies `tier.BonusApy` directly against the elapsed duration: [4](#0-3) 

There is no snapshot of the `BonusApy` that was in effect during each historical segment. The rate in effect at claim time is applied uniformly to the entire unclaimed history.

---

### Impact Explanation

If governance passes a proposal that reduces `BonusApy` (e.g., from 10 % to 5 %), every position in that tier that has not claimed since before the governance change will have its entire historical accrual recomputed at the lower rate. The difference — rewards legitimately earned under the old rate — is silently lost to the position owner. The `RewardsPoolName` module account retains the tokens, but the owner has no mechanism to recover them.

The corrupted value is: **bonus rewards owed to position owners in the updated tier**, specifically the delta between `(old BonusApy − new BonusApy) × shares × tokensPerShare × elapsed_seconds / SecondsPerYear` for each position.

Conversely, a `BonusApy` increase causes the pool to overpay for the historical period, draining the `RewardsPoolName` module account faster than intended.

---

### Likelihood Explanation

Updating tier parameters is a routine governance operation — adjusting `BonusApy` in response to market conditions or protocol economics is an expected lifecycle event. Any governance proposal that lowers `BonusApy` will silently harm all position holders who have not claimed immediately before the proposal executes. The trigger is a standard `MsgSubmitProposal` / governance vote flow, requiring no special access beyond the normal governance quorum.

---

### Recommendation

Before overwriting a tier's parameters in `SetTier`, call `claimRewardsAndUpdateTierPositions` for the affected tier ID to settle all outstanding bonus accruals under the old rate: [5](#0-4) 

This mirrors the recommended fix in the original report: flush accrued state before replacing the calculator. Alternatively, store a per-position snapshot of the `BonusApy` at the time of the last checkpoint and use that snapshot for historical segments rather than the live tier value.

---

### Proof of Concept

1. Governance sets tier 1 with `BonusApy = 0.10` (10 %).
2. Alice calls `LockTier` and enters tier 1 with 1 000 000 tokens delegated. Her `LastBonusAccrual` is set to block time T₀.
3. 180 days pass (≈ 15 552 000 seconds). Alice does not claim. Her accrued bonus ≈ `1 000 000 × 0.10 × 15 552 000 / 31 536 000 ≈ 49 315` tokens.
4. Governance passes a proposal reducing `BonusApy` to `0.05` (5 %). `SetTier` overwrites the tier record with no pre-claim.
5. Alice calls `ClaimTierRewards`. `processEventsAndClaimBonus` fetches the current tier (`BonusApy = 0.05`) and computes her bonus for the entire 180-day segment at the new rate: ≈ `24 657` tokens.
6. Alice receives ≈ 24 657 tokens instead of ≈ 49 315 tokens. The ≈ 24 658-token difference is permanently lost to her. [6](#0-5) [1](#0-0)

// claimRewardsAndUpdateTierPositions claims base and bonus rewards for all
// delegated positions in the given tier.
func (k Keeper) claimRewardsAndUpdateTierPositions(ctx context.Context, tierId uint32) error {
	ids, err := k.getPositionsIdsByTier(ctx, tierId)
	if err != nil {
		return err
	}
	if len(ids) == 0 {
		return nil
	}

	for _, id := range ids {
		pos, err := k.getPositionState(ctx, id)
		if err != nil {
			return err
		}
		if !pos.IsDelegated() {
			continue
		}

		if _, err := k.claimBaseRewards(ctx, pos); err != nil {
			return err
		}
		if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
			return err
		}
		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return err
		}
	}

	return nil
}
