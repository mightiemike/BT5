### Title
Missing Staleness Check on Chain Tip Allows False-Positive Transaction Verification During Relayer Downtime — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` confirm Bitcoin transactions against the contract's stored chain tip without checking whether that tip is fresh. If the off-chain relayer stops submitting headers, the contract's view of the Bitcoin chain freezes. Any transaction that was accepted into the frozen tip but later reorganized out of the real Bitcoin chain will still return `true` from the verification functions, enabling a double-spend against any NEAR bridge or protocol that consumes these results.

---

### Finding Description

The `BtcLightClient` contract stores the canonical Bitcoin chain tip in `mainchain_tip_blockhash`. This pointer advances only when the relayer calls `submit_blocks`. The contract has no stored NEAR-side timestamp recording when the last block was submitted, and no staleness guard in the verification path.

Both verification functions read the tip and compute confirmations:

```rust

---

### Impact Explanation

Any NEAR contract (bridge, DEX, escrow) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate fund releases can be deceived:

1. Attacker deposits BTC into a NEAR bridge; the relayer submits the containing block B.
2. The relayer goes offline (network partition, infrastructure failure, or targeted disruption of the relayer's Bitcoin RPC endpoint).
3. The real Bitcoin chain reorganizes, orphaning block B; the attacker's BTC is returned or never actually spent.
4. The contract's tip is frozen at a height where block B is still canonical.
5. Attacker calls `verify_transaction_inclusion` for their deposit transaction → returns `true`.
6. The bridge releases NEAR-side tokens; the attacker keeps both the BTC and the bridged tokens.

The `confirmations` parameter provides no protection: the contract counts confirmations relative to its stale frozen tip, not the real chain tip.

---

### Likelihood Explanation

Bitcoin reorganizations of 1–2 blocks occur several times per year on mainnet. Relayer downtime (infrastructure failure, RPC node issues, network partition) is a realistic operational event. The two conditions do not need to be attacker-caused simultaneously — the attacker only needs to monitor both the real Bitcoin chain and the NEAR contract state, and act during the overlap window. The entry path (`verify_transaction_inclusion` / `verify_transaction_inclusion_v2`) is fully open to any unprivileged NEAR caller with no deposit or role requirement beyond the `#[pause]` gate being inactive.

---

### Recommendation

1. **Store a NEAR-side submission timestamp**: Add a `last_submission_timestamp_ms: u64` field to `BtcLightClient`, updated in `submit_blocks` via `env::block_timestamp_ms()`.
2. **Guard verification with a staleness check**: At the top of `verify_transaction_inclusion`, require that `env::block_timestamp_ms() - self.last_submission_timestamp_ms` is below a configurable threshold (e.g., 2–3 hours for Bitcoin, scaled by expected block time per chain).
3. **Expose a view function**: Provide `is_chain_tip_fresh() -> bool` so downstream contracts can independently gate on liveness before calling verification.

---

### Proof of Concept

```
State before relayer outage:
  mainchain_tip_blockhash → block B (height 100)
  block B contains attacker's deposit tx T

Real Bitcoin chain after relayer outage:
  block B reorganized out; height 100 is now block B' (different hash)
  tx T is no longer confirmed

Contract state (frozen):
  mainchain_tip_blockhash → block B (height 100)  ← stale

Attacker calls:
  verify_transaction_inclusion({
      tx_id: T,
      tx_block_blockhash: hash(B),
      tx_index: <index of T in B>,
      merkle_proof: <valid proof computed before reorg>,
      confirmations: 1,
  })

Contract evaluates:
  heaviest_block_header.block_height = 100  (stale tip)
  target_block_height = 100
  100 - 100 + 1 = 1 >= 1  ✓  → returns true

Bridge releases funds. Attacker double-spends.
```
