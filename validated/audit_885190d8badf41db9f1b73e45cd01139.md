### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Proof Forgery Protection — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a fully public, unguarded NEAR contract method. Any unprivileged caller can invoke it directly to bypass the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` enforces, making the 64-byte transaction Merkle proof forgery protection entirely optional and trivially circumventable.

---

### Finding Description

The contract exposes two proof-verification entry points:

**`verify_transaction_inclusion` (v1)** — deprecated since 0.5.0, but still `pub` and decorated only with `#[pause]` (no role restriction). It accepts a `ProofArgs` struct and verifies only that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == block.merkle_root`. It performs **no coinbase proof check**. [1](#0-0) 

**`verify_transaction_inclusion_v2`** — the secure replacement. Before delegating to v1, it first validates a coinbase Merkle proof to ensure the supplied `tx_id` is not an internal Merkle tree node (the 64-byte forgery attack vector). [2](#0-1) 

The `#[deprecated]` attribute in Rust is a **compile-time lint only**. It does not restrict runtime invocability on NEAR. Any NEAR account — including a malicious recipient contract consuming verification results — can call `verify_transaction_inclusion` directly, completely skipping the coinbase proof gate that v2 enforces.

The `ProofArgs` type accepted by v1 requires no coinbase fields at all: [3](#0-2) 

Whereas `ProofArgsV2` carries the additional `coinbase_tx_id` and `coinbase_merkle_proof` fields that v2 validates: [4](#0-3) 

The security invariant broken is: **every external proof verification call must validate the coinbase Merkle proof to prevent 64-byte internal-node forgery**. Because v1 remains publicly reachable, that invariant is not enforced at the contract boundary.

---

### Impact Explanation

An attacker can forge a transaction inclusion proof using the documented 64-byte transaction Merkle proof attack (https://www.bitmex.com/blog/64-Byte-Transactions): craft a `tx_id` that is actually the hash of an internal Merkle tree node, supply a valid-looking `merkle_proof` that hashes up to the block's `merkle_root`, and call `verify_transaction_inclusion` directly. The function returns `true`, falsely asserting that a non-existent transaction was confirmed in a Bitcoin block.

Any downstream NEAR contract that gates asset releases, bridge withdrawals, or cross-chain state transitions on the return value of this call is deceived into accepting a fabricated Bitcoin event. The corrupted proof result is the exact state delta: `verify_transaction_inclusion → true` for a transaction that was never mined. [5](#0-4) 

---

### Likelihood Explanation

The entry point is fully public with no role guard — any NEAR account can call it. The 64-byte Merkle forgery technique is well-documented and the contract's own NatSpec explicitly warns about it. A malicious bridge user or recipient contract has a direct, zero-privilege path to exploit this. Likelihood is high wherever downstream contracts rely on this contract's proof result without independently enforcing v2.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or restrict it to an internal/private function so it can only be reached through `verify_transaction_inclusion_v2`. The `#[deprecated]` attribute alone provides no runtime protection on NEAR. If backward compatibility must be preserved, at minimum gate v1 behind a role (e.g., `Role::DAO`) so unprivileged callers cannot reach it.

---

### Proof of Concept

1. Attacker identifies a Bitcoin block whose Merkle tree has an internal node `N` whose 32-byte hash, when treated as a "transaction hash," produces a valid Merkle path to `merkle_root`.
2. Attacker calls `verify_transaction_inclusion` on the NEAR contract with:
   - `tx_id = N` (internal node hash, not a real txid)
   - `tx_block_blockhash` = hash of the target block (already in the mainchain)
   - `tx_index`, `merkle_proof` = crafted to satisfy `compute_root_from_merkle_proof(N, index, proof) == merkle_root`
   - `confirmations = 1`
3. `verify_transaction_inclusion` performs no coinbase check and returns `true`.
4. A downstream bridge contract reads `true` and releases funds for a Bitcoin deposit that never occurred.

The v2 path would have rejected this at: [6](#0-5) 

but v1 is reachable directly, making that check optional.

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
