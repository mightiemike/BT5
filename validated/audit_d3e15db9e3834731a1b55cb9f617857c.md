### Title
Zero `confirmations` Bypasses Confirmation-Depth Security Check in SPV Verification — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept `confirmations = 0` without any minimum-value guard. When `confirmations` is zero, the confirmation-depth `require!` is trivially satisfied for every block in the chain, meaning any transaction in any on-chain block — including the unconfirmed tip — is returned as verified. This is a direct analog to the "empty committee" class: a zero value for a security-critical threshold parameter causes the corresponding security check to be unconditionally bypassed.

### Finding Description
`verify_transaction_inclusion` enforces confirmation depth with:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

Because `confirmations` is a `u64`, passing `0` makes the left-hand side (always ≥ 1 for any stored block) unconditionally satisfy `>= 0`. The guard becomes a no-op. No lower-bound check on `confirmations` exists anywhere in the function or in `ProofArgs`/`ProofArgsV2`. [2](#0-1) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` via `args.into()`, so it inherits the same flaw. [3](#0-2) 

The only upper-bound guard present is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [4](#0-3) 

This only prevents values that are *too large*; it does nothing to prevent `confirmations = 0`.

### Impact Explanation
A downstream NEAR contract that calls `verify_transaction_inclusion{_v2}` with `confirmations = 0` will receive `true` for any transaction whose `tx_id` and `merkle_proof` are valid, regardless of how recently the block was added. The tip block has zero depth of burial and is the most likely candidate for a chain reorganization. An attacker can:

1. Submit a fraudulent or double-spend transaction into a block.
2. Immediately call `verify_transaction_inclusion` with `confirmations = 0` before any reorg occurs.
3. The function returns `true`.
4. The consuming contract acts on the "verified" result (e.g., releases funds, mints tokens).
5. The block is subsequently reorganized away, invalidating the transaction.

The corrupted invariant is the SPV finality guarantee: the contract is supposed to assert that a transaction is buried under at least N blocks of proof-of-work, but with `confirmations = 0` that guarantee is completely absent.

### Likelihood Explanation
Any unprivileged NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` directly. No role, stake, or special permission is required. The `confirmations` field is caller-supplied with no on-chain enforcement of a minimum. Downstream integrators who do not independently enforce a minimum confirmation count in their own contracts are silently exposed.

### Recommendation
Add a minimum-confirmation guard at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

Alternatively, enforce a protocol-level minimum (e.g., 6 for Bitcoin finality) and document it clearly. The same guard should be applied before the delegation call in `verify_transaction_inclusion_v2`.

### Proof of Concept
1. Deploy the contract with any valid genesis and `gc_threshold >= 1`.
2. Submit one block header (the tip is now at height H).
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = the hash of that block
   - a valid `tx_id` + `merkle_proof` for a transaction in that block
   - `confirmations = 0`
4. The `require!` at line 304–308 evaluates `(H - H) + 1 = 1 >= 0` → passes.
5. The function returns `true` for a transaction with zero confirmation depth.
6. The same call with `confirmations = 1` would also pass (1 >= 1), but `confirmations = 0` means even a block that is immediately reorganized would have been "verified" at the moment of the call. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
