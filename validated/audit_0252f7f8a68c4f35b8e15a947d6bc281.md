### Title
`read_pending_data` Stale-Fallback Omits `l1_data_gas_price` and `l2_gas_price`, Causing Wrong RPC Fee Estimates — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending block is stale (its `parent_block_hash` no longer matches the latest committed block), `read_pending_data` synthesises a fallback `DeprecatedPendingBlock`. That type structurally carries no `l1_data_gas_price` or `l2_gas_price` fields, so both accessors return `GasPricePerToken::default()` (all-zero). The zero values propagate into the `BlockContext` used by `estimate_fee`, `simulate_transactions`, and `estimate_message_fee` for `block_id = Tag::Pending`, where they are silently replaced with `NonzeroGasPrice::MIN`. The result is an authoritative-looking but materially wrong fee estimate returned to callers.

---

### Finding Description

**Step 1 – Stale-pending fallback in `read_pending_data`** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash` (the normal transient window between a block being committed and the pending-data writer catching up), the function constructs a `DeprecatedPendingBlock`. It copies `l1_gas_price` (both wei and fri) from the latest committed header, but the `DeprecatedPendingBlock` struct has no `l1_data_gas_price` or `l2_gas_price` fields, so `..Default::default()` leaves them absent.

**Step 2 – `DeprecatedPendingBlock` accessors return zero** [2](#0-1) 

`l1_data_gas_price()` and `l2_gas_price()` on `PendingBlockOrDeprecated::Deprecated(_)` unconditionally return `GasPricePerToken::default()` (all fields zero). The comments say "In older versions, data gas price was 0" — correct for genuinely old blocks, but the fallback path uses this type for *any* stale-pending scenario, including modern blocks.

**Step 3 – Conversion propagates zeros** [3](#0-2) 

`client_pending_data_to_execution_pending_data` blindly forwards the zero `GasPricePerToken` values into `ExecutionPendingData.l1_data_gas_price` and `ExecutionPendingData.l2_gas_price`.

**Step 4 – Block-context construction silently clamps zeros to `NonzeroGasPrice::MIN`** [4](#0-3) 

`NonzeroGasPrice::new(0).unwrap_or(NonzeroGasPrice::MIN)` replaces every zero with the protocol minimum (1 fri / 1 wei). The block context used for fee estimation therefore carries `l1_data_gas_price = MIN` and `l2_gas_price = MIN` instead of the real values from the latest committed block header.

**Step 5 – Fee estimation and simulation consume the wrong block context** [5](#0-4) 

`estimate_fee`, `simulate_transactions`, and `estimate_message_fee` all call `read_pending_data` and pass the result into `exec_estimate_fee` / `exec_simulate_transactions`. When the pending block is stale, every one of these RPC methods returns fee estimates computed with `l1_data_gas_price = MIN` and `l2_gas_price = MIN`.

---

### Impact Explanation

`estimate_fee` and `simulate_transactions` are the primary tools users and wallets rely on to set `resource_bounds` before submitting v3 transactions. When the pending block is stale — a routine condition that occurs for every block transition — both `l1_data_gas_price` and `l2_gas_price` are reported as `NonzeroGasPrice::MIN` (effectively 1) instead of the real values (which can be on the order of gigaweis). A user who sets `max_resource_bounds` based on this estimate will either:

- **Under-bound**: transaction is rejected by the gateway or reverts with insufficient fee.
- **Over-bound**: if the user adds a large safety margin on top of an already-wrong estimate, they may overpay significantly.

The window during which the pending block is stale is the interval between a block being committed to storage and the pending-data writer publishing a new pending block with the correct parent hash. This is a normal, recurring, unprivileged condition — no attacker action is required to trigger it.

---

### Likelihood Explanation

The stale-pending window occurs on every block transition. Any caller of `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_estimateMessageFee` with `block_id = "pending"` during that window receives the wrong answer. The condition is not gated by any privilege, special transaction type, or unusual configuration.

---

### Recommendation

In the stale-pending fallback inside `read_pending_data`, use `PendingBlockOrDeprecated::Current` (i.e., `PendingBlock`) instead of `DeprecatedPendingBlock`, and populate `l1_data_gas_price` and `l2_gas_price` from the latest committed block header, exactly as `l1_gas_price` is already populated:

```rust
// Instead of DeprecatedPendingBlock, build a PendingBlock:
block: PendingBlockOrDeprecated::Current(PendingBlock {
    parent_block_hash: latest_header.block_hash,
    l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
    l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price, // ← add
    l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,           // ← add
    l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,               // ← add
    timestamp: latest_header.block_header_without_hash.timestamp,
    sequencer_address: latest_header.block_header_without_hash.sequencer,
    starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
    ..Default::default()
}),
```

This mirrors the fix applied to the Oracle `viewPrice` bug: the "view" path must apply the same price-selection logic as the execution path, rather than silently falling back to a type whose accessors return zero.

---

### Proof of Concept

1. Commit block N to storage (so `latest_header.block_hash = H_N`).
2. Do **not** update the in-memory `pending_data` (simulating the normal propagation delay).
3. Call `starknet_estimateFee` with `block_id = "pending"` for a v3 transaction that consumes blob DA.
4. `read_pending_data` detects `pending_data.block.parent_block_hash() != H_N` and constructs a `DeprecatedPendingBlock`.
5. `l1_data_gas_price()` returns `GasPricePerToken { price_in_wei: 0, price_in_fri: 0 }`.
6. The block context is built with `l1_data_gas_price = NonzeroGasPrice::MIN` (value = 1).
7. The returned `FeeEstimation.l1_data_gas_price` is `1` instead of the real value (e.g., `10_000_000_000`).
8. A wallet using this estimate to set `max_l1_data_gas` will under-bound the transaction, causing it to be rejected or reverted. [6](#0-5) [2](#0-1) [7](#0-6) [4](#0-3)
