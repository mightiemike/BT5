Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist via the Router â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` always sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract address, not the end user. If the router is allowlisted for a pool, every user â€” including those not individually allowlisted â€” can bypass the swap allowlist entirely by routing through the router.

## Finding Description

**Root cause â€” pool passes `msg.sender` as `sender` to extensions:**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // always the immediate caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every registered extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

**Check in `SwapAllowlistExtension.beforeSwap`:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the value the pool forwarded â€” the router's address, not the end user's.

**Router does not forward the original caller:**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly with no mechanism to pass the original `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

When this executes, `msg.sender` seen by the pool is the router contract address.

**Exploit flow:**

1. Pool admin deploys pool with `SwapAllowlistExtension` and allowlists alice: `allowedSwapper[pool][alice] = true`.
2. Pool admin also allowlists the router to enable periphery swaps: `allowedSwapper[pool][router] = true`.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` â€” pool sees `msg.sender = router`.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router]` â†’ `true` â†’ passes.
7. Bob's swap executes despite not being allowlisted.

**Existing guards are insufficient:** The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool â€” it does not verify the identity of the economic actor behind the call. The `allowedSwapper` mapping is keyed on the pool's `msg.sender`, which is structurally the router for all router-mediated swaps.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC-verified addresses, whitelisted market makers, or protocol-controlled addresses) can be fully bypassed by any user who routes through `MetricOmmSimpleRouter`. Once the router is allowlisted â€” the necessary operational step for any allowlisted pool that also wants to support the standard periphery â€” the allowlist provides zero protection. This constitutes a broken core pool functionality and an admin-boundary break: an unprivileged user bypasses an admin-configured access control, enabling unauthorized swaps against oracle-derived prices, potential liquidity drain, and interaction with the pool in ways the pool admin explicitly intended to prevent.

## Likelihood Explanation

The trigger requires the pool admin to allowlist the router, which is the expected and necessary operational step for any allowlisted pool that also wants to support the standard periphery swap path. The `SwapAllowlistExtension` provides no warning or mechanism to distinguish end users behind the router. Any pool that enables both the allowlist extension and the router is affected. The attacker needs no special privileges â€” only the ability to call `MetricOmmSimpleRouter`. The same bypass applies to all four router entry points: `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Recommendation

The `sender` identity passed to `beforeSwap` must represent the economic actor, not the intermediary. The cleanest fix is to add an optional `originSender` field to the swap call that the pool passes to extensions as a separate argument, allowing extensions to gate on the true economic actor. Alternatively, `MetricOmmSimpleRouter` could accept and forward an explicit `realSender` parameter to `pool.swap`, requiring a pool interface change. Without such a mechanism, `SwapAllowlistExtension` cannot distinguish end users behind the router and the allowlist is structurally broken for the router path.

## Proof of Concept

```
Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(recipient, ...) â€” pool sees msg.sender = router
  3. Pool calls extension.beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[pool][router] â†’ true â†’ passes
  5. Bob's swap executes successfully despite not being allowlisted

Result: bob bypasses the swap allowlist entirely by routing through the router.
```
