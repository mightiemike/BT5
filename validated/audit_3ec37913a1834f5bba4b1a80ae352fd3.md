### Title
Missing Blocked Module Account Check in IBC NFT Receive Path Permanently Locks NFTs — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` accepts any valid bech32 address as the NFT receiver, including addresses of blocked module accounts (e.g., `distribution`, `bonded_tokens_pool`). Unlike the bank module's fungible token transfer path — which explicitly rejects sends to blocked addresses — the NFT transfer keeper has no such guard. An NFT minted to a module account is permanently irrecoverable.

---

### Finding Description

The receive path is:

```
OnRecvPacket (ibc_module.go:153)
  → keeper.OnRecvPacket (relay.go:101)
    → data.ValidateBasic() (packet.go:41)
    → processReceivedPacket (packet.go:140)
```

**`ValidateBasic`** only checks that the receiver is a non-empty, parseable bech32 address: [1](#0-0) 

**`processReceivedPacket`** decodes the receiver and immediately passes it to `MintNFT` or `TransferOwner` with no blocked-address check: [2](#0-1) [3](#0-2) [4](#0-3) 

**`MintNFT` / `MintNFTUnverified`** in the NFT keeper perform no blocked-address check either: [5](#0-4) 

The `AccountKeeper` interface exposed to the nft-transfer keeper does **not** include a `BlockedAddr` method: [6](#0-5) 

By contrast, the bank keeper **is** constructed with `app.BlockedAddrs()`, which marks all module accounts (except the two explicitly whitelisted) as blocked receivers: [7](#0-6) [8](#0-7) 

The blocked set includes `distribution`, `bonded_tokens_pool`, `not_bonded_tokens_pool`, `fee_collector`, `mint`, `gov`, and `transfer`: [9](#0-8) 

None of these protections are wired into the nft-transfer keeper.

---

### Impact Explanation

A counterparty chain user sends an IBC ICS-721 packet whose `receiver` field is the bech32 address of, for example, the `distribution` module account. On Cronos POS Chain:

1. `processReceivedPacket` decodes the address successfully (it is a valid bech32).
2. `MintNFT` records the module account as the NFT owner in the NFT store.
3. The module account holds no private key and cannot sign a `MsgTransferNFT`. No governance or admin path exists to forcibly reassign NFT ownership.
4. The NFT is permanently locked. The sender's NFT on the counterparty chain was burned or escrowed; there is no refund because the receive succeeded and a success ACK is returned.

Exact state delta: `nft.Owner` is set to the module account address; `ownerStore[denomID][moduleAccAddr]` gains the token ID; supply increases. No path exists to reverse any of these writes.

---

### Likelihood Explanation

The exploit requires only a valid IBC channel and the ability to craft a packet with a module account address as `receiver`. Module account addresses are deterministic (`sha256(moduleName)[:20]` encoded as bech32) and publicly known. Any user on any connected counterparty chain can trigger this with a single IBC NFT transfer message. No special privileges are required.

---

### Recommendation

Add a blocked-address check in `processReceivedPacket` before calling `MintNFT` or `TransferOwner`. The `AccountKeeper` interface in `x/nft-transfer/types/expected_keepers.go` should be extended with a `BlockedAddr(addr sdk.AccAddress) bool` method (matching the method already present on `authkeeper.AccountKeeper`), and `processReceivedPacket` should return an error if `k.authKeeper.BlockedAddr(receiver)` is true — mirroring the guard present in the ICS-20 fungible token transfer module.

---

```go
// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
```
