### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Draining DDA usdcE Without Providing USDC ŌĆö (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is a permissionless function that is intended to let any caller swap USDC into a Direct Deposit Address (DDA) in exchange for the DDA's usdcE balance. The inbound USDC leg uses a raw `.transferFrom()` call whose return value is never checked. If the USDC token returns `false` on failure instead of reverting, the function continues to withdraw usdcE from the DDA and deliver it to the caller, who has provided nothing.

---

### Finding Description

`replaceUsdcEWithUsdc` is `external` with no access-control modifier; the only gate is a chain-ID check (`block.chainid == 57073`). [1](#0-0) 

The three-step swap sequence is:

1. **Line 616** ŌĆö raw `transferFrom` with unchecked return value:
   ```solidity
   IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
   ```
2. **Line 617** ŌĆö unconditional withdrawal of usdcE from the DDA to `ContractOwner`:
   ```solidity
   DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
   ```
3. **Line 618** ŌĆö usdcE forwarded to `msg.sender` via `safeTransfer`:
   ```solidity
   IERC20Base(usdcE).safeTransfer(msg.sender, balance);
   ``` [2](#0-1) 

If the USDC `transferFrom` at step 1 returns `false` (silent failure), execution does not revert. Steps 2 and 3 proceed unconditionally, so the caller receives the full usdcE balance of the DDA while contributing zero USDC.

The rest of the codebase consistently uses the safe wrappers. `ERC20Helper.safeTransferFrom` low-level-calls the token and requires `success && (data.length == 0 || abi.decode(data, (bool)))`: [3](#0-2) 

`ContractOwner` already imports and uses `ERC20Helper` (via `using ERC20Helper for IERC20Base`) for outbound transfers, but the inbound USDC leg at line 616 bypasses it entirely. [4](#0-3) 

---

### Impact Explanation

An attacker can drain the entire usdcE balance of any DDA on chain 57073 (Ink) without providing USDC. The DDA's usdcE is user-deposited collateral awaiting credit into the Nado subaccount. Draining it means the subaccount never receives the collateral, corrupting the deposit accounting for the affected subaccount and causing a direct asset loss to the depositor.

---

### Likelihood Explanation

The function is permissionless on-chain (any EOA or contract can call it). The only precondition is that a DDA holds a non-zero usdcE balance, which is the normal operating state for any pending deposit. The attacker needs zero USDC allowance ŌĆö the exploit works precisely because the allowance check is skipped when `transferFrom` returns `false`. Likelihood is **high** if the USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink can return `false` on failure rather than reverting.

---

### Recommendation

Replace the raw `transferFrom` call with the already-available `safeTransferFrom` from `ERC20Helper`:

```solidity
// Before (line 616)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper.safeTransferFrom` is already imported and in scope via `using ERC20Helper for IERC20Base`. [3](#0-2) 

---

### Proof of Concept

1. A user deposits usdcE into their DDA; `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` is non-zero.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073 with **zero USDC allowance**.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (no revert).
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers the full usdcE balance to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` delivers the usdcE to the attacker.
6. The DDA is now empty; the victim's subaccount never receives collateral credit; the attacker holds the usdcE. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
