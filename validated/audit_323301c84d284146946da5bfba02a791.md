### Title
Sanctioned Accounts Can Complete Queued Slow-Mode Withdrawals After Sanction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The Nado protocol enforces a sanctions check (`requireUnsanctioned`) only at **slow-mode transaction submission time**, not at **execution time**. A user who submits a `WithdrawCollateral` slow-mode transaction while unsanctioned, then becomes sanctioned during the mandatory 3-day delay, can still have their withdrawal executed and receive funds — because `processSlowModeTransactionImpl` performs no sanctions re-check.

---

### Finding Description

The protocol's sanctions enforcement is split across two phases of the slow-mode lifecycle:

**Phase 1 — Submission** (`submitSlowModeTransactionImpl`): The check is present. [1](#0-0) 

```solidity
IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
requireUnsanctioned(sender);   // ← sanctions checked HERE at submission
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({...});
```

**Phase 2 — Execution** (`processSlowModeTransactionImpl`): The check is **absent**. [2](#0-1) 

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:], (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(   // ← no requireUnsanctioned before this
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
```

`executeSlowModeTransaction` is callable by **anyone** with no access control: [3](#0-2) 

The 3-day delay is hardcoded: [4](#0-3) 

The `requireUnsanctioned` helper itself is defined in `EndpointStorage`: [5](#0-4) 

---

### Impact Explanation

A sanctioned address can withdraw collateral from the protocol, defeating the on-chain sanctions enforcement mechanism. The withdrawn funds are transferred out via `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal`, resulting in a real asset movement to a sanctioned recipient. This is the same broken invariant as M-1: the restriction is registered but not enforced at the point of actual asset transfer. [6](#0-5) 

---

### Likelihood Explanation

The slow-mode path is the protocol's **censorship-resistance escape hatch** — it is specifically designed to be usable without sequencer cooperation. A user anticipating sanctions (e.g., after seeing a pending OFAC designation) can pre-submit a `WithdrawCollateral` slow-mode transaction while still unsanctioned. After the 3-day window, the withdrawal executes on-chain with no re-check. This is a realistic, low-effort attack requiring no privileged access.

---

### Recommendation

Add a `requireUnsanctioned` call at the start of `processSlowModeTransactionImpl` (or specifically within the `WithdrawCollateral` branch) to re-validate the sender's sanctions status at execution time:

```diff
// EndpointTx.sol — processSlowModeTransactionImpl
function processSlowModeTransactionImpl(
    address sender,
    bytes calldata transaction
) public {
+   requireUnsanctioned(sender);
    IEndpoint.TransactionType txType = IEndpoint.TransactionType(
        uint8(transaction[0])
    );
    ...
``` [7](#0-6) 

---

### Proof of Concept

1. Alice holds collateral in Nado and is not yet sanctioned.
2. Alice calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload. The `requireUnsanctioned` check at line 375 of `EndpointTx.sol` passes. The transaction is queued with `executableAt = block.timestamp + 3 days`.
3. Alice is added to the Chainalysis sanctions list (e.g., OFAC designation). `sanctions.isSanctioned(alice)` now returns `true`.
4. After 3 days, anyone calls `executeSlowModeTransaction()` on `Endpoint.sol`.
5. `_executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl` executes the `WithdrawCollateral` branch with no sanctions re-check.
6. `clearinghouse.withdrawCollateral` transfers Alice's collateral to `BaseWithdrawPool`, which releases it to Alice's address — a sanctioned recipient receives protocol funds. [8](#0-7)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
```
