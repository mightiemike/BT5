### Title
Wrong Block Reference Used for Difficulty Interval Timestamp in Fork Validation — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork block at a difficulty-adjustment boundary, `get_next_work_required` fetches the "first block" of the retarget interval via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. If the fork diverges before that height, the fork's actual ancestor at that position is a different block with a different timestamp. The contract uses the mainchain's timestamp instead of the fork ancestor's timestamp, producing an incorrect `actual_time_taken` and therefore an incorrect expected difficulty for the fork.

---

### Finding Description

The difficulty retarget formula requires two timestamps: the last block of the interval (`prev_block_header.block_header.time`) and the first block of the interval (`first_block_time`). The difference is `actual_time_taken`, which drives the new target.

In all three affected chain modules, `first_block_time` is obtained by calling `get_header_by_height`:

**`bitcoin.rs`** — `get_next_work_required`: [1](#0-0) 

**`litecoin.rs`** — `get_next_work_required`: [2](#0-1) 

**`dogecoin.rs`** — `get_next_work_required` (with an explicit TODO acknowledging the problem): [3](#0-2) 

The `BlocksGetter` implementation for `BtcLightClient` resolves `get_header_by_height` exclusively from `mainchain_height_to_header`: [4](#0-3) 

This map only contains mainchain blocks. Fork blocks are stored in `headers_pool` but are never indexed by height. When a fork block is being validated at a retarget boundary and the fork diverges at or before `first_block_height`, the fork's actual ancestor at that height is a completely different block than the mainchain block at the same height. The contract silently uses the mainchain block's timestamp, not the fork ancestor's timestamp.

The correct approach — used by `get_prev_header` — is to walk the chain backwards through `prev_block_hash` links, which correctly follows the fork's own ancestry regardless of mainchain state. [5](#0-4) 

---

### Impact Explanation

`actual_time_taken` is computed as `prev_block_time − first_block_time`. If the mainchain's block at `first_block_height` has an **earlier** timestamp than the fork's actual ancestor at that height, `actual_time_taken` is inflated, and the retarget formula produces a **lower** required difficulty (larger target) for the fork than the protocol mandates. An attacker exploiting this can submit fork blocks that satisfy a weaker PoW target than they should, accumulate chain work faster than a legitimately-mined chain, and trigger a fraudulent chain reorganization. The canonical chain pointer (`mainchain_tip_blockhash`) and the height-to-header index are then corrupted to reflect the attacker's fork, causing all downstream SPV proofs (`verify_transaction_inclusion`) to operate against a fraudulent chain state.

---

### Likelihood Explanation

The condition requires: (a) a fork that diverges at or before the start of a 2016-block (Bitcoin/Litecoin) or per-block Digishield (Dogecoin) retarget interval, and (b) the mainchain's block at `first_block_height` having an earlier timestamp than the fork's ancestor at that height. Condition (b) is naturally satisfied whenever the fork's ancestor block was mined later than the mainchain block at the same height, which is the common case for any fork that branches off after the mainchain block at that height was already mined. The `submit_blocks` entry point is reachable by any account holding the `UnrestrictedSubmitBlocks` role or a trusted relayer role, both of which can be granted by the `RelayerManager`. [6](#0-5) 

---

### Recommendation

Replace `get_header_by_height` calls in all three `get_next_work_required` functions with an ancestor-walk using `get_prev_header`, starting from `prev_block_header` and stepping back `blocks_to_go_back` times. This correctly follows the fork's own chain of `prev_block_hash` links and retrieves the actual ancestor's timestamp regardless of what the mainchain holds at that height. The `get_prev_header` path already works correctly for fork blocks because it resolves via `headers_pool` (which stores all submitted headers, mainchain and fork alike).

---

### Proof of Concept

1. The mainchain is at height 4031 (one block before the second Bitcoin retarget at height 4032). The mainchain block at height 2016 (`first_block_height` for the next retarget) has `time = T_main`.

2. An attacker submits a fork starting at height 2015 (diverging before `first_block_height = 2016`). The fork's block at height 2016 has `time = T_fork`, where `T_fork > T_main` (the fork block was mined later).

3. When the attacker submits the fork block at height 4032 (the retarget boundary), `get_next_work_required` calls `get_header_by_height(2016)`, which returns the **mainchain** block with `time = T_main`.

4. `actual_time_taken = prev_block_time − T_main`. Because `T_main < T_fork`, this value is larger than the correct `prev_block_time − T_fork`.

5. The retarget formula (`calculate_next_work_required`) produces a larger target (lower difficulty) than the protocol requires for the fork. [7](#0-6) 

6. The attacker's fork block at height 4032 passes `check_pow` with a `bits` value encoding this artificially low difficulty, even though it would be rejected under the correct fork-ancestor timestamp.

7. Subsequent fork blocks accumulate chain work faster than they should. Once `current_header.chain_work > total_main_chain_chainwork`, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fork. [8](#0-7)

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
