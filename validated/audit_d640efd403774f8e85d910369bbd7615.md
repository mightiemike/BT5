### Title
`INTEREST_FEE_FRACTION` Applied to Entire Deposit Rate Instead of Spread Between Borrow and Deposit Rates — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState`, the protocol fee (`INTEREST_FEE_FRACTION = 20%`) is deducted from the **entire** `totalDepositRateX18` rather than from only the **spread** between the borrow rate and the deposit rate. This systematically overcharges depositors, with the overcharge ratio scaling as `utilization / (1 − utilization)` — reaching 4× at 80% utilization and 9× at 90% utilization.

---

### Finding Description

`_updateState` computes the deposit rate and fee as follows:

```solidity
int128 totalDepositRateX18 = utilizationRatioX18.mul(
    borrowRateMultiplierX18 - ONE
);

// deduct protocol fees
int128 realizedDepositRateX18 = totalDepositRateX18.mul(
    ONE - INTEREST_FEE_FRACTION
);

int128 feesAmt = totalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18
);
``` [1](#0-0) 

`totalDepositRateX18` is the fair deposit rate derived from the utilization-weighted borrow rate. The **spread** — the protocol's natural income — is the difference between what borrowers pay and what depositors receive without any fee:

```
spread = (borrowRateMultiplierX18 − ONE) − totalDepositRateX18
       = (borrowRateMultiplierX18 − ONE) × (1 − utilizationRatio)
```

The fee should be `INTEREST_FEE_FRACTION × spread`. Instead, the code applies `INTEREST_FEE_FRACTION` to the entire `totalDepositRateX18`, which equals `utilizationRatio × (borrowRateMultiplierX18 − ONE)`. The ratio of the actual fee taken to the correct fee is:

```
actualFee / correctFee = utilizationRatio / (1 − utilizationRatio)
```

`INTEREST_FEE_FRACTION` is hardcoded at 20% in `Constants.sol`:

```solidity
int128 constant INTEREST_FEE_FRACTION = 200_000_000_000_000_000; // 20%
``` [2](#0-1) 

---

### Impact Explanation

Depositors in the spot engine receive materially less interest than they are owed. The overcharge is not bounded to a small rounding error — it scales with utilization:

| Utilization | Overcharge multiplier | Depositor loss vs. correct |
|---|---|---|
| 50% | 1× | 10% of deposit earnings |
| 80% | 4× | ~15.8% of deposit earnings |
| 90% | 9× | ~18.9% of deposit earnings |
| 95% | 19× | ~19.5% of deposit earnings |

At 80% utilization with a 10% annualized borrow rate:
- `totalDepositRateX18 = 8%`
- **Current**: `realizedDepositRateX18 = 8% × 0.8 = 6.4%` (fee = 1.6%)
- **Correct**: spread = 2%, fee = 20% × 2% = 0.4%, `realizedDepositRateX18 = 7.6%`
- Depositors receive 6.4% instead of 7.6% — a 15.8% reduction in earnings

The corrupted state variable is `state.cumulativeDepositsMultiplierX18`, which is permanently set too low on every `updateStates` call. [3](#0-2) 

All balances computed via `balanceNormalizedToBalance` using this multiplier will reflect the understated accrual. [4](#0-3) 

---

### Likelihood Explanation

**High.** The bug is triggered on every invocation of `updateStates`, which is called by the endpoint as part of normal periodic state updates. [5](#0-4)  Every depositor in every spot product is affected continuously. No special attacker action is required — the protocol itself executes the flawed formula on each cycle.

---

### Recommendation

Apply `INTEREST_FEE_FRACTION` only to the spread between the borrow rate and the deposit rate:

```solidity
int128 totalDepositRateX18 = utilizationRatioX18.mul(
    borrowRateMultiplierX18 - ONE
);

// spread = what borrowers pay minus what depositors earn (protocol's natural income)
int128 spreadX18 = (borrowRateMultiplierX18 - ONE) - totalDepositRateX18;

// fee is taken only from the spread
int128 realizedDepositRateX18 = totalDepositRateX18 - spreadX18.mul(INTEREST_FEE_FRACTION);

int128 feesAmt = totalDeposits.mul(
    totalDepositRateX18 - realizedDepositRateX18
);
```

This mirrors the correct Morpho formula `a + (1/2 ± f)(b − a)`: the base depositor rate is preserved and the fee fraction is applied only to the spread.

---

### Proof of Concept

Concrete numeric trace at 80% utilization, 10% annualized borrow rate, `dt = 1 day`:

1. `borrowerRateX18 ≈ 10% / 365 per day`
2. `borrowRateMultiplierX18 − ONE ≈ 0.000274`
3. `totalDepositRateX18 = 0.8 × 0.000274 = 0.000219`
4. **Current** `realizedDepositRateX18 = 0.000219 × 0.8 = 0.0001752` → fee = `0.0000438`
5. **Correct** spread = `0.000274 − 0.000219 = 0.0000548`, fee = `0.2 × 0.0000548 = 0.00001096`, `realizedDepositRateX18 = 0.000219 − 0.00001096 = 0.0002080`

A depositor with $1,000,000 in the pool loses approximately **$29/day** relative to the correct formula — purely due to the misapplied fee base. Over a year this compounds to a ~$10,500 loss per $1M deposited, with the loss growing nonlinearly as utilization increases.
