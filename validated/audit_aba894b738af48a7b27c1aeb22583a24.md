### Title
Missing `ENABLE_SECP_OPS` Flag Guard on 4-Byte Secp Opcode Dispatch ŌĆö (`File: src/chia_dialect.rs`)

### Summary

In `src/chia_dialect.rs`, the 1-byte opcode handler correctly gates `op_secp256k1_verify` (opcode 64) and `op_secp256r1_verify` (opcode 65) behind `ClvmFlags::ENABLE_SECP_OPS`. However, the separate 4-byte opcode handler that maps `0x13d61f00` ŌåÆ `op_secp256k1_verify` and `0x1c3a8f00` ŌåÆ `op_secp256r1_verify` performs no such flag check. Any caller that has not yet activated `ENABLE_SECP_OPS` (i.e., the hard fork has not activated) will still execute live secp signature verification when a program uses the 4-byte opcode form, diverging from the expected "unknown operator ŌåÆ nil" behavior that pre-fork nodes produce.

### Finding Description

**Root cause ŌĆö missing precondition check on operator dispatch:**

In `ChiaDialect::op`, the 4-byte opcode branch is entered whenever `op_len == 4`:

```rust
// src/chia_dialect.rs lines 157ŌĆō182
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // ŌåÉ no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // ŌåÉ no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The 1-byte opcode branch, by contrast, correctly guards the same functions:

```rust
// src/chia_dialect.rs lines 248ŌĆō249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The secp functions themselves (`op_secp256r1_verify`, `op_secp256k1_verify` in `src/secp_ops.rs`) accept `_flags: ClvmFlags` but ignore it entirely ŌĆö they unconditionally perform cryptographic verification and return `EvalErr::Secp256Failed` on mismatch or `Reduction(cost, nil)` on success.

**Execution path:**

1. Attacker submits a CLVM program whose operator atom is the 4-byte literal `\x13\xd6\x1f\x00` (secp256k1) or `\x1c\x3a\x8f\x00` (secp256r1).
2. `ChiaDialect::op` is called; `op_len == 4` is true.
3. The opcode matches the hardcoded constant; `op_secp256k1_verify` / `op_secp256r1_verify` is selected.
4. The function is called with no flag check ŌĆö live secp verification executes regardless of whether `ENABLE_SECP_OPS` is set.

**Contrast with the expected pre-fork behavior:**

A node that does not set `ENABLE_SECP_OPS` (pre-hard-fork) expects 4-byte unknown operators to be dispatched through `unknown_operator`, which in consensus mode (`allow_unknown_ops = true`) returns `nil` with a formula-derived cost. Instead, the node executes real secp verification and either returns `nil` (valid signature) or raises `EvalErr::Secp256Failed` (invalid signature).

### Impact Explanation

**Consensus divergence (flag/operator wiring error):**

- **Pre-fork nodes running older clvm_rs** (without the 4-byte secp handler): `0x13d61f00` ŌåÆ `unknown_operator` ŌåÆ `nil`. The coin puzzle evaluates as if the secp check is a no-op.
- **Pre-fork nodes running the affected clvm_rs** (with the 4-byte handler, `ENABLE_SECP_OPS` not set): `0x13d61f00` ŌåÆ `op_secp256k1_verify` ŌåÆ either `nil` (valid sig) or `EvalErr::Secp256Failed` (invalid sig).

A coin puzzle that uses the 4-byte secp opcode to guard a spend will be evaluated differently across these two node populations:

- Old nodes accept a spend with an **invalid** secp signature (opcode is a no-op ŌåÆ nil ŌåÆ puzzle may succeed).
- Affected nodes reject the same spend (secp verification fails ŌåÆ `EvalErr::Secp256Failed`).

This is a concrete consensus split: the two populations disagree on whether a given spend is valid. An attacker who controls a coin secured by a 4-byte secp puzzle can spend it on old nodes without a valid secp signature, while affected nodes reject the transaction ŌĆö or vice versa depending on puzzle structure. Either direction constitutes a consensus integrity failure that can be exploited to double-spend or permanently freeze coins depending on which fork the majority follows.

### Likelihood Explanation

The attacker needs only to craft a CLVM program with a 4-byte opcode atom equal to `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00`. This is trivially achievable by any party who can submit a transaction to the Chia mempool ŌĆö no privileged access is required. The divergence is deterministic and reproducible: every node in the pre-fork state that runs the affected clvm_rs will behave differently from every node running older clvm_rs on the same input.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check inside the 4-byte opcode branch, mirroring the 1-byte branch:

```rust
// src/chia_dialect.rs ŌĆö inside the op_len == 4 block
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is not set, the 4-byte secp opcodes fall through to `unknown_operator`, producing the same `nil`-with-formula-cost result that pre-fork nodes expect.

### Proof of Concept

Construct a CLVM program whose operator is the 4-byte atom `\x13\xd6\x1f\x00` with three arguments: a valid secp256k1 pubkey, a 32-byte message digest, and a valid signature. Run it under `ChiaDialect::new(ClvmFlags::empty())` (no `ENABLE_SECP_OPS`). The affected code executes `op_secp256k1_verify` and returns `Reduction(1300000, nil)`. Run the same program under a dialect that routes 4-byte unknowns through `unknown_operator` (simulating an older node): it returns `Reduction(formula_cost, nil)` with a different cost and without any cryptographic check. The two results differ in cost and in the conditions under which they raise an error, confirming the consensus divergence.
