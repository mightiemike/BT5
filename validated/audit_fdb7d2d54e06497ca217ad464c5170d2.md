### Title
Unrestricted `run_mainchain_gc()` Allows Any Unprivileged Caller to Permanently Prune the Mainchain - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function on `BtcLightClient` that carries no role-based access control. Any unprivileged NEAR account can call it with `batch_size = u64::MAX` to immediately remove every block header that exceeds `gc_threshold` from the contract's storage. The removal is permanent and irreversible, and it directly breaks `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` for every pruned block.

---

### Finding Description

`submit_blocks` — the canonical entry point for adding headers — is gated behind `#[trusted_relayer]`, which enforces that only staked, approved relayers may submit chain data. [1](#0-0) 

`submit_blocks` calls `run_mainchain_gc` internally as a side-effect of each submission, passing only the number of headers in the current batch as `batch_size`. [2](#0-1) 

However, `run_mainchain_gc` is also exposed as a standalone public method decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`: [3](#0-2) 

The `#[pause]` decorator restricts the call only when the contract is **paused**; during normal operation (unpaused), the function is callable by any NEAR account with no role check whatsoever. The `UnrestrictedRunGC` role exists solely to bypass the pause — it does not restrict who may call the function when the contract is live.

When called with `batch_size = u64::MAX`, the function computes:

```
total_amount_to_remove = amount_of_headers_we_store - gc_threshold
selected_amount_to_remove = min(total_amount_to_remove, u64::MAX)
                          = total_amount_to_remove
``` [4](#0-3) 

It then iterates over every eligible height, calls `remove_block_header` (which deletes from both `headers_pool` and `mainchain_header_to_height`), removes the entry from `mainchain_height_to_header`, and finally advances `mainchain_initial_blockhash` to the new oldest block: [5](#0-4) 

All three storage maps are mutated permanently. There is no undo path.

---

### Impact Explanation

**Impact: High.**

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both require the target block to be present in `mainchain_header_to_height`: [6](#0-5) 

Once `run_mainchain_gc` removes a block, every SPV proof for a transaction in that block will permanently revert with `"block does not belong to the current main chain"`. The deleted entries cannot be re-inserted without re-deploying or re-initialising the contract. Any dApp or bridge that relies on `verify_transaction_inclusion` for historical settlement is broken for all pruned heights.

---

### Likelihood Explanation

**Likelihood: High.**

The function requires no deposit (`#[payable]` is absent), no staking, no privileged key, and no special knowledge. A single NEAR transaction from any account suffices. The contract is in normal (unpaused) operation the vast majority of the time, so the bypass path through `UnrestrictedRunGC` is irrelevant — the function is open to everyone by default.

---

### Recommendation

Restrict `run_mainchain_gc` to the same caller set that governs `submit_blocks`. The simplest fix mirrors the existing role pattern:

```rust
// Option A: restrict to trusted-relayer bypass roles
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[access_control_any(roles(Role::DAO, Role::UnrestrictedRunGC, Role::UnrestrictedSubmitBlocks))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { … }
```

Alternatively, remove the public exposure entirely and keep `run_mainchain_gc` as a private helper called only from `submit_blocks`, since that is the only legitimate trigger path.

---

### Proof of Concept

1. Deploy `BtcLightClient` with `gc_threshold = 52704`.
2. Allow the relayer to submit blocks until the chain height exceeds `gc_threshold` (i.e., more than 52 704 headers are stored).
3. From **any** unprivileged NEAR account (no staking, no role), call:
   ```
   run_mainchain_gc(batch_size: 18446744073709551615)
   ```
4. The contract immediately removes every header beyond `gc_threshold` in a single transaction, advancing `mainchain_initial_blockhash` to the new oldest block.
5. Call `verify_transaction_inclusion` for any transaction whose block was pruned in step 4 — the call panics with `"block does not belong to the current main chain"`.
6. The state is permanently corrupted; no recovery is possible without contract redeployment.
