### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claim of Bonus Rewards - (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions from the store **before** any rewards are claimed or state is updated. Because there is no deduplication of `PositionIds`, supplying the same position ID twice causes the position's stale pre-claim state to be used for a second reward computation, paying bonus rewards a second time from the `RewardsPoolName` module account.

---

### Finding Description

In `ClaimTierRewards`, the handler first loads every requested position into a local slice, then passes the entire slice to `claimRewardsAndUpdatesPositions`:

```go
// msg_server.go lines 434–448
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

There is no deduplication guard on `msg.PositionIds`. If the same ID appears twice, both entries in `positions` carry the **identical pre-claim snapshot** of the position (same `LastEventSeq`, same `LastBonusAccrual`).

`claimRewardsAndUpdatesPositions` then iterates the slice sequentially:

```go
// claim_rewards.go lines 110–134
for i := range positions {
    pos := &positions[i]
    base, err := k.claimBaseRewards(ctx, *pos)
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

**First iteration (i=0):**
- `processEventsAndClaimBonus` reads validator events with `seq > pos.LastEventSeq` from the store, computes bonus, calls `bankKeeper.SendCoinsFromModuleToAccount`, calls `decrementEventRefCount` for each event (reducing `ReferenceCount` by 1), and updates `pos.LastEventSeq` / `pos.LastBonusAccrual` in memory.
- `setPosition` persists the updated position to the store. [3](#0-2) 

**Second iteration (i=1, same position ID):**
- `pos` points to `positions[1]`, which still holds the **original** `LastEventSeq` and `LastBonusAccrual` (loaded before any updates).
- `getValidatorEventsSince(ctx, valAddr, original_LastEventSeq)` returns the **same events** from the store — they are still present whenever their `ReferenceCount` was > 1 after the first decrement (which occurs whenever at least two positions exist for the same validator).
- `bankKeeper.SendCoinsFromModuleToAccount` is called a **second time** for the same bonus amount.
- `decrementEventRefCount` is called again; if `ReferenceCount` is now 1, the event is deleted — consuming another position's reference prematurely.
- `setPosition` overwrites the store with the second iteration's result (which happens to be identical to the first, so the position's checkpoint is not permanently corrupted, but the double payment has already occurred). [4](#0-3) 

The `decrementEventRefCount` function silently no-ops on already-deleted events, so it does not surface an error that would revert the transaction:

```go
// validator_events.go lines 87–89
if errors.Is(err, collections.ErrNotFound) {
    return nil // already cleaned up
}
``` [5](#0-4) 

The `sufficientBonusPoolBalance` check only verifies the pool can cover the **current** bonus computation; it does not prevent the same amount from being paid twice across two loop iterations within the same transaction. [6](#0-5) 

---

### Impact Explanation

An attacker who owns a tiered-rewards position can drain the `RewardsPoolName` module account by repeatedly doubling their bonus claim within a single transaction. Each `MsgClaimTierRewards` with `PositionIds = [X, X, X, …, X]` (N copies) pays N times the legitimate bonus in one atomic transaction. The corrupted value is the `RewardsPoolName` module account balance (bond-denom tokens held for all stakers' bonus rewards). Other stakers' future bonus claims are underfunded or fail entirely once the pool is drained.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, leaked key, or social engineering is required. The precondition — at least two positions delegated to the same validator so that `ReferenceCount > 1` — is the normal operating state of the protocol (any two users locking to the same validator satisfy it). The attack is therefore trivially reachable on mainnet.

---

### Recommendation

Add a deduplication check in `ClaimTierRewards` before loading positions:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest,
            "duplicate position id %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Alternatively, re-read each position from the store inside `claimRewardsAndUpdatesPositions` rather than operating on a pre-loaded snapshot, so that the updated `LastEventSeq` written by the first iteration is visible to the second. [7](#0-6) 

---

### Proof of Concept

1. Attacker (Alice) creates two positions, both delegated to validator V. Validator V emits one bond event; each position's event entry now has `ReferenceCount = 2`.
2. Alice accrues bonus rewards on position P over several blocks.
3. Alice broadcasts `MsgClaimTierRewards{Owner: alice, PositionIds: [P, P]}`.
4. **Iteration 1**: `processEventsAndClaimBonus` finds the bond event (`ReferenceCount = 2`), computes bonus B, calls `SendCoinsFromModuleToAccount(RewardsPoolName → alice, B)`, decrements `ReferenceCount` to 1. `setPosition` persists updated `LastEventSeq`.
5. **Iteration 2**: `positions[1]` still holds the original `LastEventSeq`. `getValidatorEventsSince` returns the same bond event (still in store, `ReferenceCount = 1`). Bonus B is computed again. `SendCoinsFromModuleToAccount(RewardsPoolName → alice, B)` is called a second time. `decrementEventRefCount` reduces `ReferenceCount` to 0, deleting the event (consuming the reference belonging to Alice's second position).
6. Alice receives `2B` instead of `B`. The `RewardsPoolName` pool is short by `B`. Alice's second position can never claim its bonus for this event (the event was deleted).
7. Repeating with `PositionIds = [P, P, P, …]` (up to the pool balance) drains the pool in a single transaction. [8](#0-7) [9](#0-8)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}

// decrementEventRefCount decrements the reference count of a validator event.
// If the reference count reaches zero, the event is deleted.
func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
