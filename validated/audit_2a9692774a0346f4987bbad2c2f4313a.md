### Title
Protobuf `ValidResourceBounds` deserialization silently downgrades `AllResources` to `L1Gas` when L2/L1DataGas are zero, producing a divergent transaction hash — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter uses a value-based heuristic to choose between `ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources`. A post-0.13.3 transaction carrying `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` serializes to protobuf bytes that are byte-for-byte identical to a pre-0.13.3 `L1Gas` transaction. On deserialization the heuristic silently reconstructs the variant as `L1Gas`. Because `get_tip_resource_bounds_hash` hashes **two** resource elements for `L1Gas` but **three** for `AllResources`, any subsequent hash computation produces a value that does not match the original signed hash, breaking the hash-domain invariant across the serialization boundary.

---

### Finding Description

**Two variants, two hash preimages.**
`ValidResourceBounds` carries a protocol-version semantic:

```
L1Gas(ResourceBounds)          // pre-0.13.3 — 2 resource elements in hash
AllResources(AllResourceBounds) // post-0.13.3 — 3 resource elements in hash
```

`get_tip_resource_bounds_hash` branches on the variant:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 197-208
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
    ValidResourceBounds::AllResources(all) =>
        vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?],   // 3 elements
});
```

**Serialization produces identical bytes for two distinct variants.**
`From<ValidResourceBounds> for protobuf::ResourceBounds` emits explicit zero fields for `L1Gas`:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 474-477
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),          // zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
```

An `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction serializes identically (both fields zero). The wire representation is indistinguishable.

**Deserialization applies a value-only heuristic.**
The inverse converter reconstructs the variant purely from field values:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 427-435
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for post-0.13.3 tx
} else {
    ValidResourceBounds::AllResources(...)
})
```

A post-0.13.3 transaction with zero L2 and L1DataGas bounds is reconstructed as `L1Gas`. The TODO comment acknowledges the backward-compatibility hold but does not gate on a version field.

**Hash divergence.**
The original transaction hash was computed with `AllResources` (3 elements). After protobuf round-trip the reconstructed variant is `L1Gas` (2 elements). `get_tip_resource_bounds_hash` returns a different felt. Because this felt is chained into `get_invoke_transaction_v3_hash` / `get_declare_transaction_v3_hash` / `get_deploy_account_transaction_v3_hash`, the recomputed hash does not equal the original signed hash.

**Wrong `GasVectorComputationMode`.**
`ValidResourceBounds::get_gas_vector_computation_mode()` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. The wrong variant therefore also changes fee-check and resource-bound logic downstream (e.g., `check_resources_within_bounds`, `perform_pre_validation_stage`).

---

### Impact Explanation

Any node that receives the transaction via the state-sync protobuf path and then recomputes the transaction hash (e.g., `validate_transaction_hash`, OS re-execution, prover input construction) will derive a hash that does not match the original. This binds the wrong hash to the transaction after conversion — matching the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

Additionally, the wrong `GasVectorComputationMode` (`NoL2Gas` instead of `All`) alters fee-bound checks and resource accounting for any re-execution of the transaction, matching: **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The gateway's stateless validator accepts `AllResourceBounds` with only `l1_gas` non-zero (l2_gas and l1_data_gas both zero). Any unprivileged user can submit such a transaction. Once included in a block and propagated via the state-sync protobuf path, the deserialization silently changes the variant. No special privilege or malformed bytes are required.

---

### Recommendation

1. **Add an explicit variant discriminant to the protobuf schema.** Introduce a boolean or enum field (e.g., `bool use_all_resources`) in `ResourceBounds` so the deserializer can reconstruct the correct variant without relying on value heuristics.

2. **Alternatively, gate on the Starknet version.** The `ConsensusTransaction` / state-sync message already carries a `starknet_version` field in `ProposalInit`. Use it to decide whether to reconstruct `L1Gas` (pre-0.13.3) or `AllResources` (post-0.13.3) unconditionally, removing the value-based heuristic entirely.

3. **Enforce the TODO.** The comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2` should be resolved: once 0.13.2 support is dropped, treat a missing `l1_data_gas` field as a hard error rather than defaulting to zero, which would prevent the ambiguous reconstruction.
