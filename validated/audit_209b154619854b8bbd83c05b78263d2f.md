### Title
Missing Expiration and Price Bound on `MintNlp`/`BurnNlp` Allows Sequencer-Delayed Execution at Unfavorable Oracle Price — (`File: core/contracts/Verifier.sol`, `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`MintNlp` and `BurnNlp` are the Nado-native analogs of the reported `buy()`/`sell()` functions. Users sign these transactions committing only to an input amount and a nonce. The `oraclePriceX18` used to compute the output is **not covered by the user's signature** and is supplied by the sequencer at execution time. Neither struct contains an expiration/deadline field, and neither enforces a user-specified minimum output amount. This is strictly worse than the original report's scenario, which at least had slippage protection.

---

### Finding Description

The `MintNlp` and `BurnNlp` structs are defined as:

```solidity
struct MintNlp { bytes32 sender; uint128 quoteAmount; uint64 nonce; }
struct BurnNlp { bytes32 sender; uint128 nlpAmount;   uint64 nonce; }
``` [1](#0-0) 

Neither struct contains an expiration/deadline field. Compare this to the `Order` struct, which carries a `uint64 expiration` field and is checked via `_expired()` before matching. [2](#0-1) [3](#0-2) 

In `Verifier.computeDigest`, the EIP-712 digest for `MintNlp` covers only `{ sender, quoteAmount, nonce }`, and for `BurnNlp` only `{ sender, nlpAmount, nonce }`. The `oraclePriceX18` field present in `SignedMintNlp` / `SignedBurnNlp` is **excluded from the signed digest**:

```solidity
} else if (txType == IEndpoint.TransactionType.MintNlp) {
    ...
    digest = keccak256(abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce          // oraclePriceX18 is NOT here
    ));
} else if (txType == IEndpoint.TransactionType.BurnNlp) {
    ...
    digest = keccak256(abi.encode(
        keccak256(bytes(BURN_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.nlpAmount,
        signedTx.tx.nonce          // oraclePriceX18 is NOT here
    ));
}
``` [4](#0-3) 

The sequencer-supplied `oraclePriceX18` is then used directly in `Clearinghouse.mintNlp` / `burnNlp` to compute the output:

- `mintNlp`: `nlpAmount = quoteAmount / oraclePriceX18` — user pays fixed USDC, receives variable NLP
- `burnNlp`: `quoteAmount = nlpAmount * oraclePriceX18 - burnFee` — user burns fixed NLP, receives variable USDC [5](#0-4) [6](#0-5) 

There is no `minNlpOut`, `minQuoteOut`, or any equivalent slippage guard in either function.

---

### Impact Explanation

A user who signs a `MintNlp` or `BurnNlp` transaction has **no on-chain protection** against:

1. **Delayed execution**: The signed transaction sits in the sequencer queue. If the NLP oracle price moves adversely before the sequencer includes it, the user receives fewer tokens/USDC than they anticipated — with no recourse.
2. **No minimum output**: Unlike the original report's scenario (which had slippage protection), Nado's NLP mint/burn provides zero user-controlled price bound. The user cannot specify "give me at least X NLP for my USDC" or "give me at least Y USDC for my NLP."

For `mintNlp`: if NLP price rises between signing and execution, `quoteAmount / oraclePriceX18` yields fewer NLP tokens.
For `burnNlp`: if NLP price falls between signing and execution, `nlpAmount * oraclePriceX18` yields less USDC.

The corrupted asset delta is the output token balance of the user's subaccount — specifically `NLP_PRODUCT_ID` balance for mint and `QUOTE_PRODUCT_ID` balance for burn.

---

### Likelihood Explanation

**High.** Every user who calls `MintNlp` or `BurnNlp` is exposed. The sequencer queue can experience delays under normal operating conditions (network congestion, high throughput). No special attacker capability is required — the vulnerability is triggered by ordinary latency between signing and execution.

---

### Recommendation

1. Add an `expiration` field to `MintNlp` and `BurnNlp` structs (mirroring the `Order` struct pattern) and include it in the EIP-712 digest in `Verifier.computeDigest`. Enforce it in `Clearinghouse.mintNlp` / `burnNlp`.
2. Add a `minAmountOut` field to both structs (also covered by the signature) and enforce it in `Clearinghouse.mintNlp` / `burnNlp` before applying balance updates.

---

### Proof of Concept

1. User signs `MintNlp { sender: alice, quoteAmount: 1000e18, nonce: 5 }` when NLP oracle price is `1.00`.
2. Expected NLP received: `1000e18 / 1.00e18 = 1000 NLP`.
3. Transaction sits in sequencer queue for an extended period; NLP oracle price rises to `1.25`.
4. Sequencer submits `SignedMintNlp { tx: ..., oraclePriceX18: 1.25e18, ... }`.
5. `Verifier.computeDigest` hashes only `{ sender, quoteAmount, nonce }` — signature is valid.
6. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1.25e18 = 800 NLP`.
7. Alice receives 800 NLP instead of 1000 NLP, with no on-chain mechanism to have prevented or bounded this outcome. [7](#0-6) [8](#0-7)
