### Title
Linked Signer Can Hijack Subaccount Signing Authority via `LinkSigner` Transaction — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `processTransactionImpl` function in `EndpointTx.sol` processes `LinkSigner` transactions with `allowLinkedSigner = true`. This allows the currently-authorized linked signer to sign a new `LinkSigner` transaction that replaces the linked signer mapping for the subaccount with any address they choose — including one they control. Because linked signers are already authorized to sign withdrawals, transfers, and liquidations, this constitutes a full subaccount takeover path reachable by any unprivileged linked signer.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` transaction and calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol L576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` calls `validateSignature`, which accepts the signature as valid if the recovered address equals either the subaccount owner **or** the current `linkedSigner`:

```solidity
// Verifier.sol L297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

Because `allowLinkedSigner = true` is passed, the current linked signer's signature satisfies this check. The function then unconditionally writes `linkedSigners[signedTx.tx.sender] = <attacker-chosen address>`. The linked signer can therefore replace themselves — or set any address — as the new linked signer for the victim subaccount.

This is structurally identical to the zNS bug: the authorization check validates one scope of authority (linked signer for trading), but the state mutation operates on a different, more privileged scope (who controls the subaccount's signing authority going forward).

The contrast with the slow-mode path makes the flaw explicit. In `processSlowModeTransactionImpl`, the same `LinkSigner` type is handled with a strict owner-only check:

```solidity
// EndpointTx.sol L232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← msg.sender must be the owner address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

The slow-mode path correctly restricts `LinkSigner` to the account owner. The sequencer path does not.

---

### Impact Explanation

Once the linked signer has replaced the `linkedSigners` entry with an attacker-controlled address, the attacker's new address is accepted as a valid signer for all transactions that pass `allowLinkedSigner = true`. These include:

- `WithdrawCollateral` — drains collateral from the subaccount
- `WithdrawCollateralV2` — drains collateral to an arbitrary `sendTo` address (when `sendTo == address(0)` is not required)
- `TransferQuote` — moves quote balance to any registered subaccount
- `LiquidateSubaccount` — initiates liquidations signed by the attacker

For isolated subaccounts, the impact is amplified: `getLinkedSigner` inherits the parent's linked signer for all isolated subaccounts, so a single `LinkSigner` mutation on the parent compromises every isolated subaccount under it.

The corrupted state is `linkedSigners[victim_subaccount]` in `EndpointStorage`, and the resulting asset delta is the full collateral balance of the victim subaccount.

---

### Likelihood Explanation

Any address that has ever been granted linked signer status — including trading bots, hot wallets, or third-party integrations — can execute this attack unilaterally. No special conditions, governance capture, or admin access are required. The attacker only needs a valid nonce and their own private key. The sequencer will process the transaction in the normal flow.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Changing the linked signer is an ownership-level operation and must only be authorized by the subaccount owner's key, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `alice_sub` and links Bob's address as a signer via a legitimate `LinkSigner` transaction. `linkedSigners[alice_sub] = Bob`.
2. Bob (or an attacker who compromised Bob's key) constructs a `SignedLinkSigner` with `tx.sender = alice_sub`, `tx.signer = attacker_address`, `tx.nonce = current_nonce`, signed with Bob's key.
3. The sequencer calls `processTransactionImpl` with this transaction.
4. `validateSignedTx(..., true)` recovers Bob's address, matches it against `getLinkedSigner(alice_sub)` which returns Bob — validation passes.
5. `linkedSigners[alice_sub]` is overwritten with `attacker_address`.
6. The attacker now signs a `WithdrawCollateral` transaction for `alice_sub` using `attacker_address`'s key. `validateSignedTx(..., true)` resolves `getLinkedSigner(alice_sub)` to `attacker_address` — validation passes.
7. Alice's collateral is withdrawn to the attacker.
