Audit Report

## Title
Wrong-Actor Binding in `DepositAllowlistExtension.beforeAddLiquidity` Allows Unprivileged Deposit Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and instead gates on `owner`, which is a caller-controlled argument. Any address not on the allowlist can bypass the deposit guard by supplying an allowlisted address as `owner`. The pool admin's intended curation policy — restricting which addresses may deposit — is never enforced against the actual depositor.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` argument and passes `msg.sender` as `sender` to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both `sender` and `owner`, but leaves `sender` unnamed (discarded) and checks only `owner`: [2](#0-1) 

Because `owner` is a free argument any caller can set to any address, an unprivileged address `B` calls `pool.addLiquidity(allowlistedAddressA, ...)`. The extension evaluates `allowedDepositor[pool][A]` → `true` and does not revert. `B` is never checked. The admin setter `setAllowedToDeposit` names the gated party `depositor`, confirming the intended check is on the depositor (sender), not the position owner: [3](#0-2) 

The analogous `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`, not the second parameter, demonstrating the intended pattern: [4](#0-3) 

No existing guard compensates for this misbinding. The `removeLiquidity` path enforces `msg.sender == owner` (line 206), so the attacker cannot reclaim the deposited tokens, but the allowlist invariant is broken regardless. [5](#0-4) 

## Impact Explanation

The `DepositAllowlistExtension` is rendered completely ineffective as an access control mechanism. Any unprivileged address can deposit into a pool configured with this extension by supplying an allowlisted address as `owner`. The pool admin's curation policy — restricting which addresses may deposit — is never enforced against the actual depositor. Additionally, legitimate allowlisted depositors that use a router or helper contract (where `owner != sender`) are incorrectly blocked, making the extension unusable for standard router patterns. This constitutes broken core pool functionality causing loss of the admin-enforced access control invariant.

## Likelihood Explanation

`addLiquidity` is permissionless — any external address can call it with any `owner` argument. No privileged setup is required. The bypass is reachable in a single transaction by any external actor on any pool that has `DepositAllowlistExtension` configured. The only prerequisite is knowledge of at least one allowlisted address, which is publicly readable from `allowedDepositor` or observable from prior transactions.

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of `owner`, matching the pattern already used in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension.
// allowedDepositor[pool][allowlistedAddress] = true
// attacker is NOT on the allowlist.

vm.startPrank(attacker);

// Attacker supplies allowlistedAddress as owner.
// Extension checks allowedDepositor[pool][allowlistedAddress] → true, does not revert.
// attacker (msg.sender / sender) is never checked.
pool.addLiquidity(
    allowlistedAddress, // owner: allowlisted → extension passes
    salt,
    deltas,
    callbackData,       // attacker's contract pays tokens in callback
    extensionData
);

// Extension never checked attacker (sender). Deposit succeeded.
// LP position minted to allowlistedAddress; tokens came from attacker.
// Pool admin's allowlist invariant is violated.
vm.stopPrank();
```
