Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual User, Allowing Any Caller to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is `msg.sender` to the pool — the router contract when a swap is routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (required for any allowlisted user to use it), the per-user gate is unconditionally open to every address that calls through the router. Any unpermissioned address can bypass the swap allowlist by calling any of the router's `exact*` entry points.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // ← pool's msg.sender, not the original user
  ...
```

`ExtensionCalling._beforeSwap` encodes this value verbatim and dispatches it to the extension as `sender`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, the pool's `msg.sender` is the router:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData   // ← original msg.sender is NOT forwarded
  );
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**The dilemma:**
- Router **not** allowlisted → every allowlisted user is also blocked from using the router; the router is unusable for curated pools.
- Router **allowlisted** → the per-user gate is completely open; any EOA or contract can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` through the router and the extension passes unconditionally.

The same wrong-actor binding applies to every router entry point. In `exactInput` multi-hop, the pool's `msg.sender` is the router for every hop. In `exactOutput` recursive callbacks, the pool is called from inside `_exactOutputIterateCallback`, where `msg.sender` to the pool is again the router.

**Contrast with `DepositAllowlistExtension`:** that extension ignores the `sender` parameter entirely and gates on `owner` — the economically relevant actor passed explicitly by the caller — so it is not affected by router intermediation. `SwapAllowlistExtension` has no equivalent mechanism to recover the real user identity.

**No existing test covers this path.** `FullMetricExtension.t.sol` allowlists a `TestCaller` contract and calls the pool directly through it; the router path is never exercised against the allowlist.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker receives real token output from the pool's LP reserves without being on the allowlist. This is a direct loss of the curation invariant and constitutes unauthorized access to pool liquidity, which can drain LP value if the pool's pricing is calibrated for a specific counterparty set. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" criteria.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who deploy a curated pool and want their allowlisted users to have a normal UX will allowlist the router. The moment they do, the gate is open to everyone. The trigger requires no special privilege — any EOA or contract can call the router's public `exact*` functions. The precondition (router allowlisted) is the natural operational state for any curated pool that intends to support router-mediated swaps.

## Recommendation

The extension must recover the real user identity rather than trusting the `sender` argument, which is the router when the router is used. Options:

1. **Decode the real swapper from `extensionData`**: require the router to embed the original `msg.sender` in `extensionData` and have the extension verify and decode it (requires a trusted encoding scheme, e.g., the extension only trusts data from a known router address).
2. **Gate on `recipient` instead of `sender`** if the pool's curation intent is to restrict who receives output tokens (not who initiates the call).
3. **Document and enforce incompatibility**: prevent pools that configure `SwapAllowlistExtension` from being reachable through the router at the factory or router level.

## Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists the router so that allowlisted users can use it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// attacker is NOT on the per-user allowlist:
// swapExtension.allowedSwapper[pool][attacker] == false

// But attacker calls through the router:
vm.startPrank(attacker);
token0.approve(address(router), type(uint256).max);
// This succeeds: the extension sees sender=router, which IS allowlisted.
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(pool),
    recipient:       attacker,
    tokenIn:         address(token0),
    zeroForOne:      true,
    amountIn:        1000,
    amountOutMinimum: 0,
    priceLimitX64:   0,
    extensionData:   "",
    deadline:        block.timestamp + 1
}));
vm.stopPrank();
// attacker receives token1 output despite not being on the allowlist.
```
