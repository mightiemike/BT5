### Title
`check_proof_facts` in the OS unconditionally rejects `PROOF_VERSION_V0` while the blockifier's `allowed_proof_versions` gate still accepts it, causing OS execution failure for blockifier-accepted transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The Cairo OS `check_proof_facts` function hard-asserts `proof_header.proof_version = PROOF_VERSION_V1`, silently dropping the comment that says V0 is a "legacy" supported version. The Rust blockifier's `validate_proof_facts` uses the `allowed_proof_versions` list from `VersionedConstants` to gate proof-version acceptance. If that list still contains the V0 felt value, a user can submit an Invoke V3 transaction carrying V0 `proof_facts` that passes every blockifier pre-validation check, gets sequenced into a block, and then causes the OS to abort with `"Unsupported proof version"` during execution — producing a wrong execution result for an already-accepted transaction.

---

### Finding Description

**Blockifier side — accepts whatever `allowed_proof_versions` lists.**

`validate_proof_facts` in `crates/blockifier/src/transaction/account_transaction.rs` (line 314) checks:

```rust
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
```

`ProofFactsVariant::try_from` in `crates/starknet_api/src/transaction/fields.rs` (lines 741–748) parses the first felt as a `ProofVersion` and explicitly names both `ProofVersion::V0` and `ProofVersion::V1` as valid parse targets:

```rust
let proof_version = ProofVersion::try_from(*proof_version).map_err(|()| {
    StarknetApiError::InvalidProofFacts(format!(
        "Expected first field to be {} or {}, but got {}",
        ProofVersion::V0,
        ProofVersion::V1,
        proof_version,
    ))
})?;
```

So if `allowed_proof_versions` in any active `VersionedConstants` JSON contains the V0 felt, the blockifier will accept the transaction.

**OS side — hard-rejects V0.**

`check_proof_facts` in `execution_constraints.cairo` (lines 53–56) contains a direct contradiction:

```cairo
// Proof version may be V0 (legacy) or V1 (current).
with_attr error_message("Unsupported proof version") {
    assert proof_header.proof_version = PROOF_VERSION_V1;
}
```

The comment says V0 is a supported legacy version; the assertion rejects it unconditionally. There is no branch, no version-gated fallback, and no `or` condition — only a single equality check against `PROOF_VERSION_V1`.

`check_proof_facts` is called from `execute_invoke_function_transaction` (lines 313–318 of `transaction_impls.cairo`) during OS block execution, after the blockifier has already committed the transaction to the block.

**The divergence.**

| Layer | V0 proof_version | V1 proof_version |
|---|---|---|
| Blockifier `validate_proof_facts` | Accepted (if V0 ∈ `allowed_proof_versions`) | Accepted |
| OS `check_proof_facts` | **Aborts** (`"Unsupported proof version"`) | Accepted |

A transaction that clears every blockifier gate is later rejected by the OS, producing a wrong execution result for an input the sequencer already accepted.

---

### Impact Explanation

**Critical — Wrong revert result from blockifier/execution logic for accepted input.**

An unprivileged user submits a well-formed Invoke V3 transaction with `proof_facts[0] = PROOF_VERSION_V0`. The blockifier accepts and sequences it. The OS then aborts on `assert proof_header.proof_version = PROOF_VERSION_V1`, producing an incorrect execution outcome (wrong revert/state) for a transaction the sequencer committed to a block.

---

### Likelihood Explanation

**Medium.** The trigger requires `allowed_proof_versions` in the active `VersionedConstants` to contain the V0 felt. The comment in the Cairo code ("Proof version may be V0 (legacy) or V1 (current)") and the explicit V0 branch in the Rust parser both indicate V0 was intentionally supported and may still be listed. Any user who can submit an Invoke V3 transaction can craft the `proof_facts` payload; no privilege is required.

---

### Recommendation

Resolve the version-gate divergence at one authoritative boundary:

1. **If V0 is intentionally retired:** Remove `PROOF_VERSION_V0` from every `allowed_proof_versions` list in all active `VersionedConstants` JSON files so the blockifier rejects V0 before sequencing. Update the stale comment in `execution_constraints.cairo`.

2. **If V0 must remain supported:** Update `check_proof_facts` to accept both versions:
   ```cairo
   with_attr error_message("Unsupported proof version") {
       assert (proof_header.proof_version - PROOF_VERSION_V0) *
              (proof_header.proof_version - PROOF_VERSION_V1) = 0;
   }
   ```

### Proof of Concept

1. Construct an `RpcInvokeTransactionV3` where `proof_facts[0]` is the felt value of `PROOF_VERSION_V0` and the remaining fields form a structurally valid SNOS proof-facts payload.
2. Submit via the gateway. The blockifier's `validate_proof_facts` calls `ProofFactsVariant::try_from`, parses V0 successfully, and checks `allowed_proof_versions.contains(V0_felt)` — passes if V0 is listed.
3. The transaction is sequenced into a block.
4. During OS execution, `execute_invoke_function_transaction` calls `check_proof_facts`; the assertion `proof_header.proof_version = PROOF_VERSION_V1` fails for the V0 value, aborting OS execution with `"Unsupported proof version"` — a wrong execution result for an accepted transaction.
