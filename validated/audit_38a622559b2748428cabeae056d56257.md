### Title
Gas-key host-function fee path passes raw borsh `public_key_len` instead of `trie_id_len` to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`, causing ~59× exec-fee under-charge for ML-DSA-65 gas keys — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

When a WASM contract calls the `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` host functions with an ML-DSA-65 public key, the exec-fee calculation receives `public_key_len as usize` — the raw borsh-encoded wire length of the key (1953 bytes) — instead of `public_key.trie_id_len()` (33 bytes). The exec fee is supposed to price the trie read/write the *receiver* performs, which is keyed by the 33-byte hash form. The send fee correctly uses the wire length (what the sender sees in memory), but the exec fee uses the wrong dimension, producing a value ~59× too large relative to the actual trie work. The transaction-path (`runtime/runtime/src/config.rs`) correctly calls `action.public_key.trie_id_len()`, so the two paths diverge.

---

### Finding Description

The `PostQuantumSignatures` feature (protocol version 85) introduced ML-DSA-65 access keys. A key design invariant is that ML-DSA-65 keys are stored in the trie **by hash** (33 bytes: 1 tag + 32-byte SHA3-256 digest), not by their 1952-byte raw pubkey. Two separate length concepts therefore exist:

- `PublicKey::len()` / `trie_id_len()` diverge only for `MLDSA65`: `len()` = 1953, `trie_id_len()` = 33.

The exec-fee helpers `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee` accept a `public_key_len: usize` parameter that is supposed to be the **on-trie** key length, because the exec fee prices the trie work done by the receipt executor (reading/writing the trie entry keyed by the hash form).

**Transaction path (correct):** `runtime/runtime/src/config.rs` line 348 calls:
```rust
gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
```

**Host-function path (wrong):** `runtime/near-vm-runner/src/logic/logic.rs` lines 3092–3096 call:
```rust
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw borsh wire length from WASM memory, not trie_id_len
);
```

`public_key_len` here is the number of bytes the WASM contract passed to describe the key in its linear memory — for an ML-DSA-65 key that is 1953 bytes. The same pattern appears in `promise_batch_action_add_gas_key_with_full_access` (line 3158) and `promise_batch_action_add_gas_key_with_function_call` (line 3229), and is duplicated verbatim in the wasmtime runner at `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` lines 3325, 3398, and 3502.

The result: the exec portion of the gas fee for ML-DSA-65 gas-key operations via host functions is computed against a 1953-byte key length instead of 33 bytes, making it ~59× larger than the actual trie work. This is the **opposite** direction from the classic "under-charge" bug — the exec fee is **over-charged** relative to the trie cost. However, the invariant that is broken is the same one the codebase explicitly documents: "Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes." The host-function path is exactly such a caller.

The divergence between the transaction path and the host-function path means that the same logical operation (fund/add a gas key) costs a different amount of gas depending on whether it is submitted as a top-level transaction action or dispatched from a WASM contract via a host function. The `test_gas_key_fee_parity` integration test only exercises ED25519 keys (line 1043: `KeyType::ED25519`), so the ML-DSA-65 divergence is not caught.

---

### Impact Explanation

**Gas accounting inconsistency / economic invariant violation.** Any WASM contract that manages ML-DSA-65 gas keys via host functions will burn ~59× more exec gas than the equivalent transaction-level action. This breaks the protocol invariant that send+exec fees price the actual trie work, and creates a systematic discrepancy between the two action-dispatch paths. A contract that budgets gas based on the transaction-path cost will run out of gas when using the host-function path, or vice versa. Because the exec fee is over-charged, the practical effect is that ML-DSA-65 gas-key operations from WASM contracts are economically unusable — they consume far more gas than the trie work justifies, effectively blocking the feature for contract-driven use cases.

---

### Likelihood Explanation

The `PostQuantumSignatures` feature is stabilized at protocol version 85 (the current stable version). Any contract that calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 key will trigger the bug. The bug is reachable by any unprivileged user who deploys a contract and calls these host functions. The existing parity test (`test_gas_key_fee_parity`) does not cover ML-DSA-65 keys, so the divergence is not caught by CI.

---

### Recommendation

In both `runtime/near-vm-runner/src/logic/logic.rs` and `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`, replace `public_key_len as usize` with `public_key.trie_id_len()` (after decoding the key) in the calls to `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee`. The decoded `PublicKey` is already available at that point (stored in `public_key` after `get_public_key`). Mirror the pattern used in `runtime/runtime/src/config.rs`:

```rust
// Correct pattern (transaction path):
gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())

// Fix for host-function path:
let decoded_pk = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),  // ← use trie_id_len, not public_key_len
);
```

Add a `test_gas_key_fee_parity` variant that exercises `KeyType::MLDSA65` to prevent regression.

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
