### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Theft Without USDC Payment — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` is designed as a 1:1 swap: the caller provides USDC and receives usdcE from a `DirectDepositV1` address. However, the USDC pull uses a raw `transferFrom` call whose return value is never checked. If the USDC token at the hardcoded address returns `false` on failure (e.g., insufficient allowance) rather than reverting, the function continues execution, withdraws usdcE from the `DirectDepositV1` contract, and sends it to the caller — with no USDC ever transferred. This is a direct analog to the reported burn-accounting bug: in both cases, the `transferFrom` integration is broken such that the accounting invariant (payment ↔ receipt) is violated, causing one party to lose assets beyond what was intended.

---

### Finding Description

`replaceUsdcEWithUsdc` is `external` with no access control beyond a chain-ID check (`block.chainid == 57073`): [1](#0-0) 

The critical line is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

This is a raw `transferFrom` call — not `safeTransferFrom` from `ERC20Helper`. The return value (`bool`) is silently discarded. Contrast this with every other token transfer in the codebase, which uses `safeTransferFrom` / `safeTransfer` that check the return value: [3](#0-2) 

After the unchecked call, the function unconditionally:
1. Calls `DirectDepositV1(directDepositV1).withdraw(usdcE)` — which transfers all usdcE from the DDA to `ContractOwner` (the owner of the DDA).
2. Calls `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — which sends that usdcE to the caller. [4](#0-3) 

`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the deployer/owner of every DDA it creates, so the call succeeds: [5](#0-4) 

The broken invariant: the function is structured as "pay USDC → receive usdcE," but the USDC payment step is not enforced.

---

### Impact Explanation

Any caller on chain 57073 (Ink) can call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1Address` is set and holds usdcE. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on failure (e.g., zero allowance) rather than reverting, the caller receives the full usdcE balance of the DDA without providing any USDC. The corrupted asset delta is: caller gains `balance` usdcE; DDA loses `balance` usdcE; `ContractOwner` receives no USDC. This directly harms the subaccount owner whose DDA is drained.

---

### Likelihood Explanation

The function is callable by any unprivileged user on chain 57073 with no further preconditions beyond a non-empty DDA. The ERC20 standard permits tokens to return `false` on failure rather than reverting; whether the specific USDC deployment at the hardcoded address does so determines immediate exploitability. The Nado code is the necessary vulnerable step: it uses raw `transferFrom` instead of `safeTransferFrom`, which is the established safe pattern used everywhere else in the codebase. The risk is concrete and the entry path is fully permissionless.

---

### Recommendation

Replace the raw `transferFrom` call with `safeTransferFrom` from `ERC20Helper`, consistent with all other token transfers in the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

This ensures that if the USDC transfer fails for any reason, the transaction reverts before usdcE is withdrawn and sent to the caller.

---

### Proof of Concept

1. Identify a subaccount `S` on chain 57073 whose `directDepositV1Address[S]` is non-zero and whose DDA holds `N` usdcE.
2. Call `ContractOwner.replaceUsdcEWithUsdc(S)` with zero USDC allowance granted to `ContractOwner`.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, N)` returns `false` (insufficient allowance) — return value discarded.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` succeeds (caller is `ContractOwner`, which is the DDA owner) — `N` usdcE transferred to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, N)` succeeds — attacker receives `N` usdcE.
6. Attacker has gained `N` usdcE; DDA is drained; no USDC was paid. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
