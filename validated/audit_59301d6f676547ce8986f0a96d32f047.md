### Title
`SwapAllowlistExtension` Allowlist Bypassed via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. The pool always passes `msg.sender` of the `swap` call as `sender`. When any swap is routed through `MetricOmmSimpleRouter`, `msg.sender` of the pool's `swap` call is the **router**, not the original user. If the pool admin allowlists the router address (a natural step to enable router-based swaps), every unprivileged user can bypass the per-user allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same applies to `exactOutputSingle`, `exactOutput`, and every hop of `exactInput`: [5](#0-4) 

The extension therefore checks whether the **router** is allowlisted, not the original user. A pool admin who allowlists the router address â€” a natural action to enable router-based swaps for their allowlisted users â€” inadvertently grants every unprivileged user the ability to swap by routing through `MetricOmmSimpleRouter`. The original user's address is never visible to the extension.

---

### Impact Explanation

A pool protected by `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., to prevent toxic arbitrage flow, enforce KYC, or limit participation to specific market makers). Once the router is allowlisted, any address can call `router.exactInputSingle(...)` and the extension passes unconditionally. Unauthorized swappers can consume LP liquidity at oracle-anchored prices, extracting value from LPs who deposited under the assumption that only vetted counterparties could trade. This is a direct loss of LP principal through fee-free or adversarial flow that the allowlist was designed to block.

---

### Likelihood Explanation

The pool admin must allowlist the router for the bypass to be active. This is a predictable operational step: a pool admin who wants allowlisted users to be able to use the standard periphery router will allowlist the router address. The extension provides no mechanism to distinguish router-mediated swaps by allowlisted users from router-mediated swaps by anyone else, so the admin faces a forced choice: either block all router swaps (breaking the intended UX) or allowlist the router (opening the bypass). The structural mismatch makes the vulnerable configuration likely in any production deployment that uses both the allowlist extension and the router.

---

### Recommendation

The `sender` argument passed to `beforeSwap` must reflect the **original user**, not the immediate caller of `pool.swap`. Two complementary fixes:

1. **Extension-side**: `SwapAllowlistExtension` should check `recipient` (the address receiving output tokens) in addition to or instead of `sender` when `sender` is a known router, or require that `sender == recipient` for allowlist enforcement.

2. **Router-side**: `MetricOmmSimpleRouter` should forward the original `msg.sender` through `extensionData` so allowlist extensions can decode and verify the true initiator. The extension interface already accepts `extensionData` in `beforeSwap`.

3. **Documentation**: Until fixed, the `SwapAllowlistExtension` NatSpec must warn that allowlisting any intermediary contract (router, multicall, aggregator) grants that contract's entire user base access to the pool.

---
