### Title
`validate_resource_bounds()` skips `max_l2_gas_amount` upper-bound check for `Declare` transactions while enforcing it for `Invoke` and `DeployAccount` — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds()` enforces the `max_l2_gas_amount` ceiling on `Invoke` and `DeployAccount` transactions but contains an explicit carve-out that silently skips the same check for `Declare` transactions. Any user can submit a `Declare` transaction with `l2_gas.max_amount` set to an arbitrarily large value (e.g., `u64::MAX`) and the gateway will admit it, violating the configured admission policy.

---

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, the `validate_resource_bounds` function applies the `max_l2_gas_amount` ceiling asymmetrically:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `min_gas_price` check on line 71 applies to all three transaction types (the `resource_bounds` is read unconditionally from `tx.resource_bounds()`), but the `max_l2_gas_amount` upper-bound check is gated behind an `if let RpcTransaction::Declare(_) = tx { }` no-op branch that causes the check to be skipped entirely for Declare transactions. [2](#0-1) 

The test suite explicitly encodes and confirms this asymmetry: `valid_l2_gas_amount_on_declare` asserts that a Declare transaction with `l2_gas.max_amount = 200` passes validation when `max_l2_gas_amount = 100`, and `test_invalid_max_l2_gas_amount` only parameterises over `TransactionType::DeployAccount` and `TransactionType::Invoke`, deliberately excluding `Declare`. [3](#0-2) [4](#0-3) 

This is the direct sequencer analog of the Union Finance `updateTrust()` bug: `vouchees` was checked against `maxVouchers` but `vouchers` was not. Here, `Invoke`/`DeployAccount` are checked against `max_l2_gas_amount` but `Declare` is not, despite all three transaction types carrying the same `AllResourceBounds` field and being subject to the same gateway admission policy.

---

### Impact Explanation

The gateway's `max_l2_gas_amount` configuration parameter is the operator-controlled ceiling that prevents transactions from claiming excessive L2 gas amounts at admission time. A Declare transaction with `l2_gas.max_amount = u64::MAX` passes `StatelessTransactionValidator::validate()` without error and is forwarded to the mempool. The admission invariant — that no transaction with `l2_gas.max_amount > max_l2_gas_amount` enters the sequencer pipeline — is broken for the entire `Declare` transaction type.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."** [5](#0-4) 

---

### Likelihood Explanation

The trigger requires no privilege. Any user who can submit an RPC transaction can craft a `Declare` transaction with an arbitrarily large `l2_gas.max_amount`. The code path is unconditionally reachable through the public `starknet_addDeclareTransaction` endpoint. The TODO comment in the source confirms the developers are aware of the gap but have not closed it. [6](#0-5) 

---

### Recommendation

Remove the `Declare` carve-out and apply the `max_l2_gas_amount` check uniformly to all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions genuinely require a different ceiling (e.g., because Sierra compilation can consume more L2 gas), introduce a separate `max_l2_gas_amount_declare` config field and enforce it explicitly, rather than leaving the check absent.

---

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` is itself the proof of concept. It demonstrates that a Declare transaction with `l2_gas.max_amount = 200` passes validation when `max_l2_gas_amount = 100`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator_test.rs
#[case::l2_gas_amount_out_of_limit(
    StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        max_l2_gas_amount: 100,          // ceiling = 100
        ..*DEFAULT_VALIDATOR_CONFIG_FOR_TESTING
    },
    RpcTransactionArgs {
        resource_bounds: AllResourceBounds {
            l2_gas: ResourceBounds {
                max_amount: GasAmount(200), // declared amount = 200 > 100
                ..NON_EMPTY_RESOURCE_BOUNDS
            },
            ..Default::default()
        },
        ..Default::default()
    }
)]
fn valid_l2_gas_amount_on_declare(…) {
    // TransactionType::Declare — asserts Ok(()) despite exceeding the limit
    assert_matches!(tx_validator.validate(&tx), Ok(()));
}
``` [3](#0-2) 

A real attacker would submit via the RPC endpoint:

```json
{
  "type": "DECLARE",
  "resource_bounds": {
    "l2_gas": { "max_amount": "0xffffffffffffffff", "max_price_per_unit": "<min_gas_price>" },
    ...
  },
  ...
}
```

This transaction passes `StatelessTransactionValidator::validate()` and is admitted to the mempool, bypassing the `max_l2_gas_amount` admission gate that protects `Invoke` and `DeployAccount` transactions.

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
