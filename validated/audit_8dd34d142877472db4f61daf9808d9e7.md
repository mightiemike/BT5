### Title
`ChunkExtra::next_for_old_chunk()` carries over `proposed_split` for missing chunks, causing `InvalidChunkHeaderShardSplit` on the subsequent new chunk — (`core/primitives/src/types.rs`)

---

### Summary

Under `ProtocolFeature::DynamicResharding`, `ChunkExtra` gained a `proposed_split: Option<TrieSplit>` field (V5). When a chunk is present (new chunk path), `apply_chunk_postprocessing` freshly computes `proposed_split` via `compute_proposed_split()` and stores it in `ChunkExtra`. When a chunk is missing (old chunk path), `next_for_old_chunk()` clones the previous `ChunkExtra` and only updates `state_root`, silently carrying over whatever `proposed_split` was in the previous block. This is the exact nearcore analog of the external bug: the new-chunk path correctly updates the field; the old-chunk path does not.

---

### Finding Description

`ChunkExtra::next_for_old_chunk()` is defined as:

```rust
pub fn next_for_old_chunk(&self, state_root: StateRoot) -> Self {
    let mut new_extra = self.clone();
    *new_extra.state_root_mut() = state_root;
    new_extra
}
``` [1](#0-0) 

It clones every field of the previous `ChunkExtra` and only replaces `state_root`. The `proposed_split: Option<TrieSplit>` field in `ChunkExtraV5` is therefore inherited verbatim. [2](#0-1) 

This function is called in the `OldChunk` arm of `process_apply_chunk_result`:

```rust
ShardUpdateResult::OldChunk(OldChunkResult { shard_uid, apply_result }) => {
    let old_extra = self.chain_store_update.get_chunk_extra(prev_hash, &shard_uid)?;
    let new_extra = old_extra.next_for_old_chunk(apply_result.new_root);
    ...
    self.chain_store_update.save_chunk_extra(block_hash, &shard_uid, new_extra.into());
``` [3](#0-2) 

The new-chunk arm, by contrast, calls `apply_chunk_postprocessing` which invokes `apply_result.to_chunk_extra(gas_limit)` — a freshly computed `ChunkExtra` that includes the result of `compute_proposed_split()` for that block. [4](#0-3) 

The invariant that is broken: `proposed_split` in `ChunkExtra` must reflect what was computed during the application of **that specific chunk**. For a missing chunk, no split computation runs; the correct value is `None`. Carrying over a non-`None` value from the previous block corrupts the field.

The same stale-carry-over occurs in `set_state_finalize_on_height` (state-sync catch-up path):

```rust
let mut new_chunk_extra = ChunkExtra::clone(&chunk_extra);
*new_chunk_extra.state_root_mut() = apply_result.new_root;
self.chain_store_update.save_chunk_extra(..., new_chunk_extra.into());
``` [5](#0-4) 

And in the resharding manager when creating child `ChunkExtra` entries:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [6](#0-5) 

The project's own documentation acknowledges this at TODO #10: *"The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field)."* [7](#0-6) 

**Trigger sequence:**

1. Block N (near epoch boundary, shard above memory threshold): new chunk applied → `compute_proposed_split()` returns `Some(TrieSplit{boundary_account, ...})` → stored in `ChunkExtra[N].proposed_split`.
2. Block N+1: chunk is missing → `next_for_old_chunk()` clones `ChunkExtra[N]` → `ChunkExtra[N+1].proposed_split = Some(TrieSplit{...})` (stale carry-over).
3. Block N+2: new chunk produced. The chunk producer freshly runs `compute_proposed_split()` → returns `None` (epoch boundary has passed, cooldown active, or threshold not met). The chunk header is signed with `proposed_split = None`.
4. Chunk header validation enforces:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [8](#0-7) 

`prev_chunk_extra.proposed_split()` is `Some(TrieSplit{...})` (stale); `chunk_header.proposed_split()` is `None` (correct). The mismatch causes `InvalidChunkHeaderShardSplit`, rejecting the valid chunk.

The dynamic resharding documentation itself notes that `min_epochs_between_resharding > 0` is required precisely because of this carry-over: *"a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."* [9](#0-8) 

The cooldown is a partial mitigation for one specific sub-case (back-to-back reshardings), but it does not prevent the general case: any missing chunk in the window where `proposed_split` was non-`None` propagates the stale value forward.

---

### Impact Explanation

A valid chunk produced by a legitimate chunk producer is rejected with `InvalidChunkHeaderShardSplit` whenever a missing chunk follows a block that had a non-`None` `proposed_split`. This breaks chunk production for the affected shard, stalling block finality for that shard until the condition clears. Because `proposed_split` is only non-`None` near epoch boundaries, the window is narrow but deterministically reachable on any epoch where a shard crosses the memory threshold. The `DynamicResharding` feature is at protocol version 85, which is the current production version. [10](#0-9) 

---

### Likelihood Explanation

Missing chunks are a normal network occurrence (validator downtime, network partition, slow block production). The `proposed_split` window spans the last few blocks of an epoch for any shard above the memory threshold. The intersection of these two events — a non-`None` `proposed_split` followed by a missing chunk — is low-probability per epoch but becomes near-certain over many epochs as shard memory grows. No privileged action is required; natural network jitter suffices.

---

### Recommendation

Reset `proposed_split` to `None` in `next_for_old_chunk()`, since a missing chunk performs no split computation:

```rust
pub fn next_for_old_chunk(&self, state_root: StateRoot) -> Self {
    let mut new_extra = self.clone();
    *new_extra.state_root_mut() = state_root;
    // A missing chunk does not run compute_proposed_split(); reset to None.
    if let ChunkExtra::V5(ref mut v5) = new_extra {
        v5.proposed_split = None;
    }
    new_extra
}
```

Apply the same reset in `set_state_finalize_on_height` and in `ReshardingManager`'s child `ChunkExtra` construction.

---

### Proof of Concept

1. Enable `DynamicResharding` (protocol version ≥ 85) with a shard whose trie memory exceeds `memory_usage_threshold`.
2. Run the network until block N near an epoch boundary where `compute_proposed_split()` returns `Some(TrieSplit{...})` for shard S. Confirm `ChunkExtra[N].proposed_split = Some(...)`.
3. Simulate a missing chunk at block N+1 for shard S (e.g., via `adv_produce_chunks StopProduce`). Confirm `ChunkExtra[N+1].proposed_split = Some(...)` (stale carry-over via `next_for_old_chunk`).
4. Resume chunk production at block N+2. The chunk producer computes `proposed_split = None` and signs the header accordingly.
5. Observe `InvalidChunkHeaderShardSplit` in `validate_chunk_with_chunk_extra_and_receipts_root` at `chain/chain/src/validate.rs:176`, rejecting the valid chunk. [11](#0-10) [12](#0-11) [8](#0-7)
