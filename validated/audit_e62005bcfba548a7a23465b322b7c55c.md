### Title
`FillOrder` Event Emits Parent Subaccount While Balance Delta Lands on Isolated Subaccount, Permanently Desyncing Off-Chain Accounting — (`core/contracts/OffchainExchange.sol`)

### Summary

In `matchOrders`, `ordersInfo.taker.sender` is captured from `taker.order.sender` before the isolated-subaccount override. After the override, `_updateBalances` correctly credits the isolated subaccount, but `emitFillOrderEvent` emits the stale `ordersInfo.taker.sender` (the parent) as the `subaccount` field of `FillOrder`. Every isolated fill therefore emits an event attributing `baseDelta`/`quoteDelta` to the parent while the engine storage mutation targets the isolated subaccount.

### Finding Description

In `OffchainExchange.matchOrders`:

```
// Step 1 – snapshot sender into ordersInfo BEFORE override
ordersInfo.taker.sender = taker.order.sender;   // = parent P   (line 656)

// Step 2 – override taker.order.sender to isolated subaccount
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest]; // = iso I  (line 674)
}
// ordersInfo.taker.sender is NEVER updated → still = P

// Step 3 – balance update uses taker.order.sender = I
_updateBalances(callState, market.quoteId, taker.order.sender, ...); // (line 814)

// Step 4 – event uses ordersInfo.taker.sender = P
emitFillOrderEvent(callState, ordersInfo.taker, taker.order, true);  // (line 843)
// inside: emit FillOrder(..., orderInfo.sender, ...) → emits P    (line 855)
```

The `FillOrder` event definition in `IOffchainExchange.sol` documents `baseDelta` and `quoteDelta` as "change in **this** subaccount's balance from this fill." When `subaccount = P` but the engine storage mutation is on `I`, that invariant is broken for every isolated fill. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

Off-chain systems (indexers, risk engines, the sequencer's own position tracker) consume `FillOrder` events to maintain a real-time view of each subaccount's open position and collateral. Because every isolated fill emits `subaccount = P` with non-zero `baseDelta`/`quoteDelta`, those systems will:

- Accumulate a phantom position on the parent P that does not exist in engine storage
- Never record the fill against the isolated subaccount I (no `FillOrder` event is emitted for I)

The parent therefore appears to hold a position it does not hold on-chain. If the off-chain risk engine uses this view to decide whether P is healthy enough to avoid liquidation, P can be shielded from liquidation while its actual collateral sits in I. Conversely, I can accumulate bad debt that the off-chain system never attributes to P, preventing timely margin calls. This satisfies the "incorrect liquidation / bad debt creation" critical impact criterion. [5](#0-4) 

### Likelihood Explanation

This triggers on **every** isolated order fill — no special attacker setup is required beyond submitting a `CreateIsolatedSubaccount` followed by a `MatchOrders` referencing the same digest, both of which are standard supported protocol flows. The sequencer routes these transactions in normal operation; no privilege escalation or key compromise is needed. [6](#0-5) [7](#0-6) 

### Recommendation

After the isolated-subaccount override, synchronize `ordersInfo.taker.sender` to match `taker.order.sender`:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
    ordersInfo.taker.sender = taker.order.sender;  // ← add this line
}
if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
    maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
    ordersInfo.maker.sender = maker.order.sender;  // ← add this line
}
```

This ensures `emitFillOrderEvent` emits the isolated subaccount as `subaccount`, keeping the event's `baseDelta`/`quoteDelta` semantically consistent with the engine storage mutation. [8](#0-7) 

### Proof of Concept

1. Deploy full stack (Endpoint, OffchainExchange, SpotEngine/PerpEngine, Clearinghouse).
2. Parent subaccount P submits `CreateIsolatedSubaccount` for `productId X` → `digestToSubaccount[digest] = I`.
3. Sequencer submits `MatchOrders` referencing the same digest with `taker.order.sender = P`.
4. After the transaction, assert:
   - `perpEngine.getBalance(X, I).amount != 0` (isolated subaccount holds the position)
   - `perpEngine.getBalance(X, P).amount == 0` (parent has no on-chain position)
   - `FillOrder` event emitted with `subaccount = P` and non-zero `baseDelta` (event says parent's balance changed)
5. The contradiction between step 4b and step 4c is the confirmed desync: the event attributes the fill to P, but the engine state records it on I. [9](#0-8)
