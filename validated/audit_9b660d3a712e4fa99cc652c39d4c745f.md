### Title
Gateway Rejects Valid Transactions with Zero L2 Gas Amount Due to Unconditional `min_gas_price` Check - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` checks `l2_gas.max_price_per_unit < min_gas_price` unconditionally, without first verifying that `l2_gas.max_amount > 0`. When a transaction allocates zero L2 gas (making the L2 price field irrelevant), the gateway still rejects it with `MaxGasPriceTooLow`, mirroring the external bug's pattern of checking a pool's "units" without accounting for a zero allocation weight.

### Finding Description

In `validate_resource_bounds`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs, lines 71-76
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

The production default for `min_gas_price` is `8_000_000_000` (8 Gwei):

```rust
// crates/apollo_gateway_config/src/config.rs, line 192
min_gas_price: 8_000_000_000,
```

A transaction that sets `l2_gas.max_amount = 0` and `l2_gas.max_price_per_unit = 0` (paying entirely through `l1_gas`) passes the first guard (non-zero total fee from L1 gas) but is then unconditionally rejected because `0 < 8_000_000_000`. The L2 gas price is semantically irrelevant when `max_amount == 0` — no L2 gas will be consumed and no L2 fee will be charged — yet the check treats it as a hard rejection criterion.

The test suite confirms this scenario is intended to be valid: `test_positive_flow` includes a `valid_l1_gas` case that sets only `l1_gas: NON_EMPTY_RESOURCE_BOUNDS` with all other bounds at `Default::default()` (zero). That test only passes because `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` overrides `min_gas_price` to `0`. Under the production config (`min_gas_price: 8_000_000_000`), the identical transaction structure is rejected.

### Impact Explanation

**High. Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any transaction that legitimately sets `l2_gas.max_amount = 0` (e.g., a transaction structured to pay only in L1 gas) is permanently rejected at the gateway with a misleading `MaxGasPriceTooLow` error. The transaction never reaches the mempool or batcher. This is a gateway admission correctness failure: a structurally valid, fee-paying transaction is denied entry based on a price field that has no economic meaning when the corresponding amount is zero.

### Likelihood Explanation

The `AllResourceBounds` type is the only valid resource bounds format for v3 transactions. A user or SDK that constructs a transaction with `l2_gas.max_amount = 0` (relying solely on L1 gas for fee payment) will hit this rejection on every submission attempt against any node running the production config. The condition is deterministic and reproducible.

### Recommendation

Guard the price check with an amount check, mirroring the fix pattern from the external report:

```rust
// Only enforce the L2 gas price floor when L2 gas is actually being consumed.
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

### Proof of Concept

1. Construct a v3 `RpcInvokeTransaction` with:
   - `l1_gas: { max_amount: 1000, max_price_per_unit: 10_000_000_000 }` (above `min_gas_price`)
   - `l2_gas: { max_amount: 0, max_price_per_unit: 0 }` (zero amount, zero price)
   - `l1_data_gas: { max_amount: 0, max_price_per_unit: 0 }`

2. Submit to a gateway running with `StatelessTransactionValidatorConfig::default()` (`min_gas_price = 8_000_000_000`, `validate_resource_bounds = true`).

3. Observe: the first guard at line 66 passes (`max_possible_fee > 0` from L1 gas). The check at line 71 fires: `0 < 8_000_000_000` → `Err(MaxGasPriceTooLow { gas_price: 0, min_gas_price: 8_000_000_000 })`.

4. The transaction is rejected at the gateway despite being structurally valid and carrying sufficient L1 gas fee coverage.

The existing test `valid_l1_gas` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` (lines 70–82) documents exactly this transaction shape as expected-valid, but silently bypasses the production bug by setting `min_gas_price: 0` in the test config (line 57).     fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
