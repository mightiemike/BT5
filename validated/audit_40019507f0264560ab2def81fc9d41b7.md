### Title
Unprivileged Caller Can Trigger Mainchain GC to Permanently Corrupt SPV Proof Verification - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function with no caller access control when the contract is unpaused. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, immediately deleting all GC-eligible mainchain headers. This permanently advances `mainchain_initial_blockhash` and removes entries from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for any block whose header was deleted, and breaking chain reorganization if the common ancestor is among the removed headers.

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. [1](#0-0) 

The `#[pause(...)]` attribute controls behavior only when the contract is **paused** — it allows `UnrestrictedRunGC` role holders to bypass the pause. When the contract is **not paused** (the normal operating state), there is no caller restriction at all. Any NEAR account can invoke this method.

The function permanently deletes mainchain headers from all three state maps and advances `mainchain_initial_blockhash`: [2](#0-1) 

`remove_block_header` erases entries from both `mainchain_header_to_height` and `headers_pool`: [3](#0-2) 

By contrast, `submit_blocks` — the only other entry point that calls GC — is protected by `#[trusted_relayer]`, restricting it to staked relayers or accounts with `UnrestrictedSubmitBlocks`/`DAO` roles: [4](#0-3) 

The analog to the external report is exact: just as `executeProposal` lacked a caller guard and allowed any user to execute queued transactions, `run_mainchain_gc` lacks a caller guard and allows any user to execute a destructive state mutation.

### Impact Explanation

**`verify_transaction_inclusion` / `verify_transaction_inclusion_v2` permanently broken for GC'd blocks.**

Both functions look up the target block in `mainchain_header_to_height`: [5](#0-4) 

After GC, that entry no longer exists. The call panics with `"block does not belong to the current main chain"`. Because `mainchain_initial_blockhash` is permanently advanced, the deleted headers can never be recovered. Any dApp or bridge that relies on SPV proofs for those blocks receives a permanent hard failure.

**Chain reorganization broken if common ancestor is GC'd.**

`reorg_chain` walks the fork chain backward until it finds a block present in `mainchain_header_to_height`: [6](#0-5) 

If the common ancestor has been GC'd, the loop walks past it and eventually panics with `"previous fork block should be there"` when it tries to fetch the next predecessor from `headers_pool`. The CLAUDE.md explicitly acknowledges this failure mode:

> "If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound`." [7](#0-6) 

An attacker can front-run a pending reorg by calling `run_mainchain_gc(u64::MAX)` to remove the common ancestor before the fork tip accumulates enough work, permanently preventing the reorg from completing and locking the canonical chain on the weaker branch.

### Likelihood Explanation

Once the mainchain has grown beyond `gc_threshold` (recommended value: 52704 blocks, approximately one year of Bitcoin blocks), GC-eligible headers always exist. At that point, any NEAR account — with zero stake, no role, and no special permission — can call `run_mainchain_gc` at any time. The call costs only standard NEAR gas. An attacker can monitor the chain state and front-run any pending `verify_transaction_inclusion` call or any fork submission that is approaching the work threshold needed to trigger a reorg.

### Recommendation

Add a caller access-control guard to `run_mainchain_gc` that restricts direct external invocations to privileged roles (e.g., `DAO`, `RelayerManager`, or a dedicated `GCManager` role), while still allowing the internal call from `submit_blocks` to proceed without restriction. For example:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    // Add: require caller has GCManager or DAO role, OR is the contract itself
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || env::predecessor_account_id() == env::current_account_id(),
        "Unauthorized: only DAO or contract itself may run GC"
    );
    // ... existing logic
}
```

The internal call from `submit_blocks` (line 181) uses `self.run_mainchain_gc(num_of_headers)` where `predecessor_account_id()` is the relayer — this would need to be handled by checking `current_account_id()` or by splitting the function into a public guarded wrapper and a private internal implementation.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100`.
2. Submit 200 mainchain headers via the trusted relayer. Now `amount_of_headers_we_store = 200 > 100 = gc_threshold`, so 100 headers are GC-eligible.
3. From any unprivileged NEAR account (no role, no stake), call:
   ```
   run_mainchain_gc { batch_size: 18446744073709551615 }
   ```
4. The function removes all 100 GC-eligible headers immediately. `mainchain_initial_blockhash` is permanently advanced to height 100.
5. Call `verify_transaction_inclusion` for any block at height 0–99. The call panics: `"block does not belong to the current main chain"`.
6. Submit fork headers that branch from height 50 (now GC'd). Call `submit_blocks` with a fork tip that has higher chainwork. `reorg_chain` panics: `"previous fork block should be there"` when it walks back past height 100 and cannot find the common ancestor in `headers_pool`. [8](#0-7)

```rust
    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
```
