### Title
Phantom-Position Bypass in `verify_transaction_inclusion_v2`: Odd-Length Tree Duplicate-Leaf Allows Out-of-Bounds Index to Return `true` — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for `tx_index = n` (a phantom, nonexistent position) in any block whose transaction count `n` is odd and ≥ 3. The coinbase-proof guard does not prevent this because it only validates that the coinbase is correctly placed at index 0; it says nothing about whether the claimed `tx_index` for the target transaction is within `[0, n-1]`. Neither the contract nor `compute_root_from_merkle_proof` stores or checks the actual transaction count.

---

### Finding Description

**Root cause — no `tx_index` upper-bound check anywhere in the call chain.**

`verify_transaction_inclusion_v2` performs three checks before returning:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` [1](#0-0) 
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` [2](#0-1) 
3. Delegates to `verify_transaction_inclusion`, which only checks `!merkle_proof.is_empty()` and then computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` [3](#0-2) 

`compute_root_from_merkle_proof` itself is a pure position-parity walk with no knowledge of tree size:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
``` [4](#0-3) 

The contract stores only the block header (which contains the Merkle root) — it never stores the transaction count. There is therefore no way for any guard in the current code to detect that `tx_index ≥ n`.

**Why the phantom position produces the correct root.**

For a 3-tx block `[tx0, tx1, tx2]`, Bitcoin's Merkle tree duplicates the last leaf:

```
Leaves (padded): [tx0, tx1, tx2, tx2]
Level 1:         [H(tx0,tx1),  H(tx2,tx2)]
Root:             H(H(tx0,tx1), H(tx2,tx2))
```

The canonical proof for the real position 2 is `[tx2, H(tx0,tx1)]`:

| Step | pos | operation | result |
|------|-----|-----------|--------|
| 1 | 2 (even) | `H(tx2, proof[0]=tx2)` | `H(tx2,tx2)`, pos→1 |
| 2 | 1 (odd)  | `H(proof[1]=H(tx0,tx1), H(tx2,tx2))` | root |

The **same proof** `[tx2, H(tx0,tx1)]` applied at phantom position 3:

| Step | pos | operation | result |
|------|-----|-----------|--------|
| 1 | 3 (odd) | `H(proof[0]=tx2, tx2)` | `H(tx2,tx2)`, pos→1 |
| 2 | 1 (odd) | `H(proof[1]=H(tx0,tx1), H(tx2,tx2))` | root |

Both paths produce the same intermediate `H(tx2,tx2)` and therefore the same root. The proof is valid for both index 2 and the nonexistent index 3.

**Why the coinbase guard does not help.**

The coinbase proof for index 0 in the same tree is `[tx1, H(tx2,tx2)]`, depth 2 — identical depth to the phantom proof. The length-equality check passes. The coinbase root check passes because the coinbase proof is genuinely valid. Nothing in the guard touches `tx_index` at all. [5](#0-4) 

---

### Impact Explanation

`verify_transaction_inclusion_v2` returns `true` for `(tx_id=tx[n-1], tx_index=n)` — a wrong-index inclusion claim — in any odd-length block. Any downstream consumer (bridge, payment verifier, cross-chain relay) that relies on this function to confirm "transaction X is at position Y in block Z" can be deceived: the attacker presents a valid-looking proof for a position that does not exist, bypassing the v2 coinbase-proof mitigation entirely. This matches the scoped critical impact: *wrong-index inclusion claim that returns `true`*.

---

### Likelihood Explanation

The attack requires only:
- A canonical block with an odd number of transactions (extremely common on Bitcoin/Litecoin/Dogecoin mainnet — the majority of blocks have an odd tx count).
- Knowledge of the real proof for the last transaction (publicly derivable from any block explorer).
- A single unprivileged `view` call to `verify_transaction_inclusion_v2`.

No privileged role, no key compromise, no social engineering.

---

### Recommendation

Because the contract does not store transaction counts, the fix must be structural. Two complementary approaches:

1. **Require `tx_index < 2^(proof_depth - 1)` is insufficient alone** — the tree may not be full. The correct bound is `tx_index < n`, which requires storing `n` alongside the Merkle root in the block header, or passing it as a caller-supplied value that is itself committed to (e.g., via a separate on-chain record or a Merkle-root-of-tx-count commitment).

2. **Detect the phantom position algebraically**: after computing the root, also compute the root for `tx_index ^ 1` (the sibling). If both produce the same root, the position is ambiguous (i.e., it is a duplicate-leaf phantom) and the proof must be rejected.

3. **Require `tx_index` to be strictly less than `2^(proof_depth)` AND that the proof depth equals `ceil(log2(n))` with `n` supplied and verified** — this requires storing tx counts.

### Proof of Concept

```rust
// 3-tx block: [coinbase, tx1, tx2]
// Merkle tree (padded): [cb, tx1, tx2, tx2]
// Root = H(H(cb,tx1), H(tx2,tx2))

// Real proof for tx2 at index 2: [tx2, H(cb,tx1)]
// Phantom proof for tx2 at index 3: same [tx2, H(cb,tx1)]
// Coinbase proof for cb at index 0: [tx1, H(tx2,tx2)]

verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id:                  tx2,
    tx_block_blockhash:     block_hash,
    tx_index:               3,          // phantom — does not exist
    merkle_proof:           vec![tx2, H(cb, tx1)],
    coinbase_tx_id:         cb,
    coinbase_merkle_proof:  vec![tx1, H(tx2, tx2)],
    confirmations:          0,
})
// → returns true
```

Step-by-step verification of the phantom proof inside `compute_root_from_merkle_proof`:
- pos=3 (odd): `H(tx2, tx2)` → pos=1
- pos=1 (odd): `H(H(cb,tx1), H(tx2,tx2))` = root ✓

Step-by-step verification of the coinbase proof:
- pos=0 (even): `H(cb, tx1)` → pos=0
- pos=0 (even): `H(H(cb,tx1), H(tx2,tx2))` = root ✓

Both checks pass; `verify_transaction_inclusion_v2` returns `true` for the nonexistent index 3.
