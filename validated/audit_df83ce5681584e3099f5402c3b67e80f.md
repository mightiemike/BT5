### Title
Unchecked `transferFrom` Return Value and Missing Allowance/Balance Pre-Validation in `replaceUsdcEWithUsdc` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`replaceUsdcEWithUsdc` in `ContractOwner.sol` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without (a) pre-validating that `msg.sender` holds sufficient USDC or has approved `ContractOwner` for at least `balance`, and (b) checking the boolean return value of `transferFrom`. If the USDC token on chain 57073 (Ink) returns `false` on failure rather than reverting, the function continues executing and drains usdcE from the target DDA to the caller without the caller providing any USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an externally callable function with no `onlyOwner` guard — its only access restriction is `require(block.chainid == 57073, ERR_UNAUTHORIZED)`. Any unprivileged caller on chain 57073 can invoke it for any registered subaccount DDA.

The function body is:

```solidity
uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
if (balance > 0) {
    IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);   // ← unchecked bool
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));         // ← drains usdcE to ContractOwner
    IERC20Base(usdcE).safeTransfer(msg.sender, balance);                   // ← sends usdcE to caller
}
```

Three problems exist simultaneously:

1. **No allowance pre-check**: There is no `require(IERC20Base(usdc).allowance(msg.sender, address(this)) >= balance)` before the `transferFrom` call.
2. **No balance pre-check**: There is no `require(IERC20Base(usdc).balanceOf(msg.sender) >= balance)` before the `transferFrom` call.
3. **Unchecked return value**: The `bool` returned by `IERC20Base(usdc).transferFrom(...)` is silently discarded. The rest of the protocol uses `ERC20Helper.safeTransferFrom` which properly validates the return value, but this call bypasses that wrapper entirely.

The analog to the external report is exact: the report describes checking `balanceOf` but not `allowance` before a pull transfer. Here, Nado checks neither, and additionally discards the transfer result. [1](#0-0)

### Impact Explanation

If the USDC token deployed at `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 returns `false` on a failed `transferFrom` (rather than reverting — a common pattern in non-standard or bridged ERC20 tokens), execution continues past line 616. The subsequent calls:

- `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — transfers the entire usdcE balance of the DDA to `ContractOwner`
- `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — forwards that usdcE to the attacker

The attacker receives the victim DDA's usdcE without providing any USDC. The DDA is drained of its usdcE collateral, permanently corrupting the subaccount's pending deposit state. The subaccount owner loses their deposited usdcE with no USDC replacement credited. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The function is reachable by any unprivileged caller on chain 57073 with no further preconditions. The exploitability depends on whether the specific USDC deployment at the hardcoded address returns `false` or reverts on failure. Chain 57073 (Ink) is a newer chain where the USDC token may be a bridged or wrapped variant with non-reverting failure semantics. Even if the current USDC reverts, the missing pre-validation and unchecked return value represent a latent integration bug that would activate if the token address is ever updated or if the token's behavior changes.

---

### Recommendation

1. Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom` (already used throughout the protocol) to ensure the return value is validated:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

2. Add explicit pre-validation of both balance and allowance before the transfer:

```solidity
require(IERC20Base(usdc).balanceOf(msg.sender) >= balance, "Insufficient USDC balance");
require(IERC20Base(usdc).allowance(msg.sender, address(this)) >= balance, "Insufficient USDC allowance");
```

3. Consider adding `onlyOwner` to `replaceUsdcEWithUsdc` since it is a privileged migration operation, not a general-purpose user entrypoint. [1](#0-0)

### Proof of Concept

**Setup**: Chain 57073. A subaccount DDA (`directDepositV1`) holds 1000 usdcE. The USDC token at the hardcoded address returns `false` on failed `transferFrom` (non-reverting).

**Attacker steps**:
1. Call `replaceUsdcEWithUsdc(targetSubaccount)` with zero USDC balance and zero USDC approval.
2. `balance = usdcE.balanceOf(directDepositV1)` → `1000`.
3. `usdc.transferFrom(attacker, directDepositV1, 1000)` → returns `false` (attacker has no USDC / no approval). Return value is discarded.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → transfers 1000 usdcE from DDA to `ContractOwner`.
5. `usdcE.safeTransfer(attacker, 1000)` → transfers 1000 usdcE to attacker.

**Result**: Attacker receives 1000 usdcE at zero cost. The DDA is emptied. The subaccount's pending deposit is permanently lost. [3](#0-2) [4](#0-3)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
