### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two proof-verification entry points: the deprecated `verify_transaction_inclusion` (v1) and the secure `verify_transaction_inclusion_v2`. The 64-byte transaction Merkle forgery protection (coinbase proof validation) exists **only** in v2. Because Rust's `#[deprecated]` attribute is a compiler hint only — it does not remove the function from the NEAR ABI — any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase proof check entirely and obtaining a `true` result for a forged transaction inclusion proof.

This is structurally analogous to M-02: the critical enforcement logic (coinbase proof) lives in a non-mandatory path (v2), while the unprotected path (v1) remains fully reachable. Just as edition owners in M-02 could attach a fee-free forked minter to bypass platform fees, any NEAR caller here can route through v1 to bypass the forgery protection.

---

### Finding Description

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` and carries a warning comment: [1](#0-0) 

Despite the deprecation, it remains a `pub` function decorated with `#[pause]` and is therefore part of the live NEAR contract ABI: [2](#0-1) 

`verify_transaction_inclusion_v2` adds the coinbase proof check before delegating to v1: [3](#0-2) 

The coinbase proof requirement — `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` — is the **only** guard against the 64-byte internal-node forgery. It is absent in v1. An attacker who calls v1 directly supplies only `ProofArgs` (no `coinbase_tx_id`, no `coinbase_merkle_proof`): [4](#0-3) 

The v1 path then computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` and compares it to `header.block_header.merkle_root`. Because a 64-byte internal Merkle node can be crafted to hash to the correct root without corresponding to any real transaction, this check passes and the function returns `true`.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that is tricked into doing so — receives a `true` result for a transaction that was never included in the Bitcoin block. This corrupts the proof-verification guarantee that is the contract's sole security primitive. Bridge contracts, payment-verification contracts, or any application that gates actions on a `true` result from this API can be deceived into releasing funds or executing privileged operations based on a fabricated Bitcoin event.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` via a standard NEAR RPC `call` with a crafted `ProofArgs` payload. The 64-byte forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the contract's own comments acknowledge it. The only prerequisite is knowledge of a real block's Merkle root and the ability to construct a valid-looking internal-node proof path, which is straightforward given a Bitcoin full node or block explorer.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it behind a role that no external caller holds. Since `verify_transaction_inclusion_v2` already delegates to v1 internally, the v1 logic can be refactored into a private helper. Removing the public entry point closes the bypass path without any loss of functionality.

---

### Proof of Concept

1. Attacker identifies a real Bitcoin block whose Merkle tree has an even number of transactions (causing the last leaf to be duplicated — a standard Bitcoin Merkle tree property).
2. Attacker selects the duplicated internal node hash as `tx_id` and constructs a `merkle_proof` path that computes to the block's known `merkle_root`.
3. Attacker calls `verify_transaction_inclusion` (v1) on the NEAR contract with:
   - `tx_id` = crafted 64-byte internal node hash
   - `tx_block_blockhash` = hash of the real block (already in the contract's `headers_pool`)
   - `tx_index` = chosen index consistent with the crafted proof
   - `merkle_proof` = crafted path
   - `confirmations` = 1
4. The contract computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)`, which equals `header.block_header.merkle_root` by construction.
5. The function returns `true` — a false positive — for a transaction that does not exist.
6. Any downstream contract gating on this result executes its privileged action (e.g., releasing bridged funds) based on the forged proof. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
