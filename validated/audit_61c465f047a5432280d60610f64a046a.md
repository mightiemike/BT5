### Title
`DelegateAction` / `DelegateActionV2` Signed Payload Lacks Chain-ID Binding, Enabling Cross-Network Replay by a Malicious Relayer - (File: `core/primitives/src/action/delegate.rs`)

### Summary

`SignedDelegateAction` and `VersionedSignedDelegateAction` (NEP-366 / NEP-611 meta-transactions) produce a signed hash over a payload that contains no chain identifier. The `SignableMessage` wrapper prepends only a NEP-number discriminant. Because NEAR mainnet and testnet share the same account-ID namespace and users routinely reuse the same key-pairs across networks, a malicious relayer can take a `DelegateAction` signed by a user on one network and submit it verbatim on another network where the same account and key exist with a compatible nonce, executing the inner actions without the user's intent.

### Finding Description

`DelegateAction` is defined as:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [1](#0-0) 

The hash that is signed is produced by:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [2](#0-1) 

`SignableMessage` prepends only a 4-byte `MessageDiscriminant` (the NEP number, e.g. `NEP_366_META_TRANSACTIONS = 366`): [3](#0-2) 

The same pattern applies to `DelegateActionV2` / `VersionedDelegateActionPayload::get_nep461_hash()`: [4](#0-3) 

Neither the `DelegateAction` struct nor the `SignableMessage` wrapper includes a `chain_id` or any network-specific value. The verification path in `apply_delegate_action` calls `signed_delegate_action.verify()`, which only checks the cryptographic signature against the key and the chain-agnostic hash: [5](#0-4) 

By contrast, a regular `SignedTransaction` includes a `block_hash` field that implicitly binds it to a specific chain (the block must exist in the local chain store). `DelegateAction` has no equivalent binding — only `max_block_height` (a block-height ceiling, not a chain anchor) and a nonce (replay protection within one chain, not across chains).

### Impact Explanation

A malicious relayer who receives a `SignedDelegateAction` intended for testnet can submit it on mainnet (or any other NEAR network) if:

1. The `sender_id` account exists on the target network.
2. The `public_key` is registered as an access key on that account.
3. The current nonce on the target network is below `delegate_action.nonce`.
4. The current block height on the target network is below `max_block_height`.

All four conditions are routinely satisfied: NEAR testnet and mainnet share the account-ID namespace, developers and users commonly reuse the same key-pairs, and `max_block_height` values are typically set far in the future. The inner `actions` (token transfers, function calls, `AddKey`, `DeleteKey`, etc.) then execute on the target network with `sender_id` as the predecessor — without the user ever authorizing an action on that network.

### Likelihood Explanation

The relayer role is explicitly unprivileged and untrusted at the protocol level; the protocol's own documentation notes that "some level of trust between the relayer and its user is therefore required." A relayer that is malicious or compromised can trivially perform this replay. The same key-pair on mainnet and testnet is the common case for developers and many end-users. The nonce constraint is the only cross-chain barrier, and it is easily satisfied when the target-chain nonce is lower than the signed nonce (e.g., a fresh mainnet account).

### Recommendation

Include the network's `chain_id` (the string identifier such as `"mainnet"` or `"testnet"`) in the Borsh-serialized payload that is hashed and signed. The most backward-compatible approach is to add a new `SignableMessageType` variant (e.g., `DelegateActionV3`) whose payload wraps `(chain_id: String, delegate_action: DelegateActionV2)`, gated behind a new `ProtocolFeature`. The existing `DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) discriminants must remain unchanged for backward compatibility; the new variant's discriminant provides domain separation from both.

Alternatively, the `chain_id` can be prepended to the `SignableMessage` serialization at the `MessageDiscriminant` level, making all future `SignableMessageType` variants chain-bound by construction.

### Proof of Concept

1. Alice signs a `DelegateAction` on testnet: `sender_id = "alice.near"`, `receiver_id = "bob.near"`, `actions = [Transfer(1_000_000 yN)]`, `nonce = 5`, `max_block_height = 999_999_999`, `public_key = ed25519:AliceKey`.
2. The signed hash is `SHA256(LE32(NEP_366 + 2^30) || borsh(delegate_action))` — no chain identifier anywhere in the input.
3. Alice's mainnet account `alice.near` has the same `ed25519:AliceKey` registered with nonce 4.
4. The malicious relayer wraps the testnet `SignedDelegateAction` in a mainnet `SignedTransaction` (relayer pays gas) and submits it.
5. `apply_delegate_action` calls `signed_delegate_action.verify()` → passes (same bytes, same key, same hash).
6. Nonce 5 > 4 → passes. Block height < 999_999_999 → passes.
7. A `Transfer(1_000_000 yN)` receipt is emitted on mainnet from `alice.near` to `bob.near` — an action Alice never authorized on mainnet. [6](#0-5) [7](#0-6) [8](#0-7)

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }
