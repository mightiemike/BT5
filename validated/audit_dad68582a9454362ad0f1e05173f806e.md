### Title
Hard-Coded `softfork_kw` in `RuntimeDialect` Breaks Non-Standard Dialect Deployments — (`File: src/runtime_dialect.rs`)

---

### Summary

`RuntimeDialect::new()` accepts `quote_kw` and `apply_kw` as caller-supplied constructor arguments, signaling that it is designed for configurable, non-Chia-standard dialects. However, `softfork_kw` is unconditionally hard-coded to `vec![36]` at construction time. Any deployment that assigns opcode 36 to a non-softfork operator, or assigns softfork semantics to a different opcode, will silently mis-dispatch every affected instruction.

---

### Finding Description

In `src/runtime_dialect.rs`, the constructor is:

```rust
pub fn new(
    op_map: HashMap<String, Vec<u8>>,
    quote_kw: Vec<u8>,
    apply_kw: Vec<u8>,
    flags: ClvmFlags,
) -> RuntimeDialect {
    RuntimeDialect {
        f_lookup: f_lookup_for_hashmap(op_map),
        quote_kw,
        apply_kw,
        softfork_kw: vec![36], // softfork opcode  ← hard-coded
        flags,
    }
}
``` [1](#0-0) 

`quote_kw` and `apply_kw` are forwarded from the caller, but `softfork_kw` is always `[36]` — the Chia-mainnet opcode — regardless of what the caller intends.

In `src/run_program.rs`, the interpreter's hot dispatch path checks `softfork_kw()` before falling through to `dialect.op()`:

```rust
if op_atom == Some(self.dialect.apply_kw()) {
    ...
} else if op_atom == Some(self.dialect.softfork_kw()) {
    // full softfork guard logic: parse cost/extension/program/env
    ...
} else {
    let r = self.dialect.op(...)?;
    ...
}
``` [2](#0-1) 

Because `softfork_kw()` always returns `36`, two failure modes exist for any non-Chia dialect:

1. **Opcode 36 is not the softfork operator in the custom dialect.** Every program byte that encodes opcode 36 is intercepted by the softfork guard before `dialect.op()` is ever called. The guard attempts to parse four arguments (cost, extension, program, env); if the argument shape does not match, the call either errors or silently returns `nil` (depending on `allow_unknown_ops`), producing a wrong result.

2. **The custom dialect assigns softfork semantics to a different opcode.** That opcode falls through to `dialect.op()` and is dispatched as an ordinary or unknown operator, completely bypassing the softfork guard and its cost-accounting machinery.

The `Dialect` trait exposes `softfork_kw()` as a configurable method precisely so implementations can vary it:

```rust
pub trait Dialect {
    fn quote_kw(&self) -> u32;
    fn apply_kw(&self) -> u32;
    fn softfork_kw(&self) -> u32;
    ...
}
``` [3](#0-2) 

`ChiaDialect` hard-codes all three keywords because it is Chia-specific by design. `RuntimeDialect` is the general-purpose, caller-configurable implementation, yet it hard-codes only `softfork_kw`, creating an asymmetry that silently breaks any non-Chia deployment.

---

### Impact Explanation

**Medium.** A caller that supplies non-standard `quote_kw`/`apply_kw` to `RuntimeDialect` (the only reason those parameters exist) will experience silent mis-dispatch of every program that touches opcode 36 or the intended softfork opcode. Concretely:

- Programs that should invoke the softfork guard (with its cost enforcement and allocator-checkpoint rollback) bypass it entirely, allowing undercharged or unbounded execution.
- Programs that use opcode 36 for a non-softfork purpose are intercepted by the guard, producing wrong results or spurious errors.
- Consensus divergence between nodes running `ChiaDialect` and nodes running a misconfigured `RuntimeDialect` is possible.

---

### Likelihood Explanation

**Medium.** The `RuntimeDialect` constructor's signature — accepting `quote_kw` and `apply_kw` as arguments — explicitly advertises configurability for non-standard dialects. Any integrator who follows that API contract and supplies non-default keywords will trigger the bug without any further attacker involvement. The attacker's role is simply to submit CLVM bytes that exercise opcode 36 or the intended softfork opcode.

---

### Recommendation

Add `softfork_kw` as a constructor parameter, mirroring `quote_kw` and `apply_kw`:

```rust
pub fn new(
    op_map: HashMap<String, Vec<u8>>,
    quote_kw: Vec<u8>,
    apply_kw: Vec<u8>,
    softfork_kw: Vec<u8>,   // ← add this
    flags: ClvmFlags,
) -> RuntimeDialect {
    RuntimeDialect {
        f_lookup: f_lookup_for_hashmap(op_map),
        quote_kw,
        apply_kw,
        softfork_kw,          // ← forward it
        flags,
    }
}
```

Callers that want Chia-standard behavior pass `vec![36]`; callers with a different opcode layout pass their own value. At construction time, assert `!softfork_kw.is_empty()` to catch misconfiguration early.

---

### Proof of Concept

```python
from clvm_rs import run_program, RuntimeDialect

# Custom dialect: quote=1, apply=2, but softfork is intended at opcode 37,
# and opcode 36 is a user-defined operator registered in op_map.
dialect = RuntimeDialect(
    op_map={"my_op": bytes([36])},
    quote_kw=bytes([1]),
    apply_kw=bytes([2]),
    flags=0,
)

# Program: (36 (q . 1) (q . 2) (q . 3) (q . 4))
# Caller expects opcode 36 to dispatch to "my_op" via dialect.op().
# Actual behavior: the interpreter intercepts opcode 36 as softfork_kw,
# attempts to parse (1 2 3 4) as (cost extension program env),
# and either errors or returns nil — never reaching "my_op".
program = bytes.fromhex("...")  # encoding of the above
result = run_program(dialect, program, b"\x80", max_cost=10_000_000)
# result is wrong: nil or an error, not the output of "my_op"
```

The root cause is exclusively in `src/runtime_dialect.rs` line 30 (`softfork_kw: vec![36]`), which is a compile-time constant baked into the only configurable dialect implementation in the codebase. [4](#0-3)
