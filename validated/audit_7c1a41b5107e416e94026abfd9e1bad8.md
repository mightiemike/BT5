### Title
Upgrade Handler Halts Chain When Bonus Pool Is Empty During Vesting Position Force-Exit — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`)

### Summary

`ForceFullExitWithDelegation` unconditionally propagates `ErrInsufficientBonusPool` from `processEventsAndClaimBonus`. When called from the v8 upgrade migration for any vesting-owned position that has accrued bonus rewards and the `RewardsPoolName` module account is empty, the error bubbles through the entire upgrade handler call stack and halts the chain at upgrade height. The same risk was already identified and mitigated in `slashRedelegationPosition` but the fix was not applied to the migration path.

### Impact Explanation

If the `RewardsPoolName` module account balance is zero (or below the accrued bonus amount) at upgrade height, and at least one vesting-owned position has accrued non-zero bonus rewards, the v8 upgrade handler returns an error. In Cosmos SDK, an upgrade handler error at the upgrade height causes the chain to panic and halt. The position's delegation remains locked in the position's delegator address, which is a deterministic module address (`tieredrewards/position/<id>`). No standard `MsgServer` path exists to recover it post-halt because the upgrade never completes.

### Likelihood Explanation

The precondition is realistic and attacker-reachable:
1. Any user can drain the rewards pool by claiming their own bonus rewards via `MsgClaimTierRewards` before the upgrade height — this is a normal, permissionless transaction.
2. Vesting accounts with tier positions are an explicitly supported configuration (the entire `exitVestedAccountsPositions` migration exists to handle them).
3. The pool being empty at upgrade height is also a natural steady-state if it was never replenished.

No governance, operator, or privileged access is required. A single user with a vesting-owned position and a drained pool triggers the halt.

### Recommendation

In `ForceFullExitWithDelegation`, apply the same pattern already used in `slashRedelegationPosition`: catch `ErrInsufficientBonusPool` from `claimRewards`/`processEventsAndClaimBonus`, log it, and continue the exit without paying the bonus. The delegation must be returned to the owner regardless of whether the bonus pool can cover the accrued reward. The bonus can be forfeited or deferred, but the upgrade must not halt.

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        logger.Error("force-exit: insufficient bonus pool, forgoing bonus", "position_id", posID, "error", err)
        // re-run without bonus, or proceed with zero bonus
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

### Proof of Concept

1. Create a vesting account owner with a tier position that has accrued bonus rewards (advance block time past the accrual window).
2. Drain the `RewardsPoolName` module account to zero via `MsgClaimTierRewards` from other positions, or simply never fund it.
3. Call `v2.Migrate(ctx, positions, ak, keeper)` (i.e., trigger the v8 upgrade handler).
4. Observe: `exitVestedAccountsPositions` → `ForceFullExitWithDelegation` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`, which propagates to the upgrade handler, which returns an error, halting the chain.
5. The position's delegation remains locked in the position delegator address with no recovery path.
