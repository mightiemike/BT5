### Title
Unvalidated `tx_id` Input Enables Merkle Proof Forgery via Internal-Node Substitution — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and passes it directly into `compute_root_from_merkle_proof` with no check that it represents a real leaf transaction. An unprivileged NEAR caller can substitute any internal Merkle-tree node hash as `tx_id`, supply the correct sibling path to the root, and receive a `true` return value for a Bitcoin transaction that never existed.

---

### Finding Description

`verify_transaction_inclusion` is a public, pausable (but not access-controlled) contract method. It takes `ProofArgs` — including `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` — and delegates directly to `compute_root_from_merkle_proof`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

`compute_root_from_merkle_proof` is a pure hash-path traversal: it takes whatever hash is given as `transaction_hash` and walks up the tree using the supplied siblings, returning the computed root.

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof {
        ...
        current_hash = compute_hash(...);
        ...
    }
    current_hash
}
``` [2](#0-1) 

Neither function validates that `tx_id` is a leaf-level hash. The contract's own inline warning acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

The `#[deprecated]` Rust attribute is a compile-time lint only; it imposes no runtime restriction. The method remains callable by any NEAR account. [4](#0-3) 

---

### Impact Explanation

A recipient contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., token mint, bridge unlock) will receive `true` for a fabricated Bitcoin transaction. The attacker never needs to mine a real Bitcoin block or control any privileged role — they only need to know the Merkle tree structure of any block already stored in the light client's mainchain, which is public on-chain state.

**Impact: High** — proof-verification forgery; the canonical mapping `tx_id → included-in-block` is corrupted for any consumer of this API.

---

### Likelihood Explanation

**Likelihood: Low** — the function is deprecated and a correct replacement (`verify_transaction_inclusion_v2`) exists. However, any deployed consumer contract that still calls the deprecated method is immediately exploitable by any unprivileged NEAR account with knowledge of a stored block's Merkle tree (publicly derivable from Bitcoin RPC data). No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

1. **Remove** `verify_transaction_inclusion` entirely rather than leaving it deprecated-but-callable. A deprecated on-chain method is still a live attack surface.
2. If removal is not immediately possible, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` guard to make it unconditionally revert.
3. Ensure all downstream consumer contracts are migrated to `verify_transaction_inclusion_v2`, which mitigates the internal-node substitution by requiring a coinbase proof of equal depth. [5](#0-4) 

---

### Proof of Concept

**Setup:** Block `B` is stored in the light client mainchain. Its Merkle tree has transactions `[T0, T1, T2, T3]` with known internal node `N = hash(T2, T3)`.

**Attacker call:**
```
verify_transaction_inclusion({
  tx_id:             N,           // internal node hash, not a real tx
  tx_block_blockhash: B,          // real stored block
  tx_index:          2,           // position consistent with N's path
  merkle_proof:      [hash(T0,T1)], // sibling path from N to merkle_root
  confirmations:     1
})
```

**Result:** `compute_root_from_merkle_proof(N, 2, [hash(T0,T1)])` computes `hash(hash(T0,T1), N) == merkle_root` → returns `true`.

The function confirms inclusion of a Bitcoin transaction that does not exist, with zero Bitcoin mining required and no privileged NEAR role needed. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
