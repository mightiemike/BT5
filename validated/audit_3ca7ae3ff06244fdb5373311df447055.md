### Title
Authz Grantee Can Force Granter's Liquid Balance Into Tier Position via `MsgAddToTierPosition` â€” (`x/tieredrewards/keeper/msg_server.go`)

### Summary

`AddToTierPosition` derives the source of funds entirely from `msg.Owner`. Under a generic authz grant for `MsgAddToTierPosition`, a grantee can set `msg.Owner = granter`, pass the `IsOwner` check (because the position is legitimately owned by the granter), and cause `lockFunds` to send coins **from the granter's liquid balance** into the granter's tier position â€” without any per-transaction consent from the granter.

---

### Finding Description

The full call path in `AddToTierPosition` is:

**Step 1 â€” Owner check passes:**
`validateAddToPosition` only verifies `pos.IsOwner(owner)`, i.e., that the string `msg.Owner` matches `pos.Owner` in storage. It does **not** verify that the transaction signer equals `msg.Owner`. [1](#0-0) 

**Step 2 â€” `ownerAddr` is derived from `msg.Owner`:**
After the check passes, `ownerAddr` is decoded directly from `msg.Owner`. [2](#0-1) 

**Step 3 â€” `lockFunds` sends coins FROM `ownerAddr`:**
`lockFunds` calls `bankKeeper.SendCoins(ctx, ownerAddr, delAddr, ...)`, moving `amount` of the bond denom from the owner's liquid balance to the position's delegator account. [3](#0-2) 

**Step 4 â€” Delegation is executed from the position account:**
The funds are then delegated to the validator via `delegate(ctx, delAddr, valAddr, msg.Amount)`. [4](#0-3) 

**The authz path:**

In Cosmos SDK authz, `MsgAddToTierPosition` uses `owner` as its signer field (standard pattern; `GetSigners()` returns `msg.Owner`). When a grantee submits `MsgExec{msgs: [MsgAddToTierPosition{Owner: granter, PositionId: X, Amount: Y}]}`:

1. Authz verifies a grant exists from `granter â†’ grantee` for `MsgAddToTierPosition` âś“
2. Authz executes the inner message with the granter as the effective signer âś“
3. `validateAddToPosition` passes because `pos.Owner == granter` âś“
4. `lockFunds(ownerAddr=granter, ...)` sends `Y` tokens from the granter's liquid balance âś“
5. `delegate(delAddr, valAddr, Y)` locks those tokens into the granter's tier position âś“

There is no guard that restricts the `Amount` field, no per-execution spending limit enforced by the module itself, and no check that the effective signer is the same entity as the one whose balance is being spent. [5](#0-4) 

---

### Impact Explanation

A grantee holding a generic authz grant for `MsgAddToTierPosition` can:

- Lock an arbitrary amount of the granter's liquid (spendable) balance into the granter's existing tier position at any time, without further consent.
- Repeat this until the granter's liquid balance is exhausted.
- The granter's funds are not stolen (they remain in the granter's position), but they are **illiquid** for the full tier lock-up duration. The granter loses the ability to spend, transfer, or use those tokens until the exit process completes.

This directly violates the invariant that only the account whose key signs a transaction may have its liquid balance locked and delegated.

---

### Likelihood Explanation

Generic authz grants are a common pattern for delegation management (e.g., bots, multisigs, portfolio managers). A user who grants authz for `MsgAddToTierPosition` to allow a bot to top up their position on their behalf is fully exposed. The attacker needs only: (1) a valid grant, (2) knowledge of the granter's position ID, and (3) the granter to have a non-zero liquid balance. All three are observable on-chain.

---

### Recommendation

Add a `SpendLimit` authorization type specific to `MsgAddToTierPosition` (analogous to `SendAuthorization`) that caps the total `Amount` that can be spent across all executions under a single grant. Alternatively, enforce that the effective signer (extracted from the SDK context, not from `msg.Owner`) matches `msg.Owner` inside `AddToTierPosition`, which would make generic authz grants for this message type inoperable â€” forcing users to use a typed authorization with explicit spend limits.

func (k Keeper) validateAddToPosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if pos.HasTriggeredExit() {
		return types.ErrPositionTriggeredExit
	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	return nil
}

func (ms msgServer) AddToTierPosition(ctx context.Context, msg *types.MsgAddToTierPosition) (*types.MsgAddToTierPositionResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateAddToPosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(msg.Owner)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	if err := ms.lockFunds(ctx, ownerAddr, delAddr, msg.Amount); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	newShares, err := ms.delegate(ctx, delAddr, valAddr, msg.Amount)
	if err != nil {
		return nil, err
	}

	if err := ms.setPosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionAmountAdded{
		PositionId:  pos.Id,
		TierId:      pos.TierId,
		Owner:       pos.Owner,
		SharesAdded: newShares,
		AmountAdded: msg.Amount,
	}); err != nil {
		return nil, err
	}

	return &types.MsgAddToTierPositionResponse{PositionId: pos.Id}, nil
}

func (k Keeper) lockFunds(ctx context.Context, ownerAddr, delAddr sdk.AccAddress, amount math.Int) error {
	bondDenom, err := k.stakingKeeper.BondDenom(ctx)
	if err != nil {
		return err
	}
	return k.bankKeeper.SendCoins(ctx, ownerAddr, delAddr, sdk.NewCoins(sdk.NewCoin(bondDenom, amount)))
}

func (msg MsgAddToTierPosition) Validate() error {
	if _, err := sdk.AccAddressFromBech32(msg.Owner); err != nil {
		return errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	if !msg.Amount.IsPositive() {
		return errorsmod.Wrap(sdkerrors.ErrInvalidRequest, "amount must be positive")
	}

	return nil
}
