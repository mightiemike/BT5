### Title
`WithdrawCollateralV2` Slow Mode Handler Missing — Withdrawal Permanently Unexecutable via Censorship-Resistance Path, Fee Silently Consumed - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`TransactionType.WithdrawCollateralV2` is fully handled in the sequencer-submitted path (`processTransactionImpl`) but has **no handler** in `processSlowModeTransactionImpl`. Because `submitSlowModeTransactionImpl` does not gate `WithdrawCollateralV2` as owner-only, any user can submit it via slow mode, pay the slow mode fee, and have the transaction queued. When execution is attempted, `processSlowModeTransactionImpl` hits the terminal `else { revert(); }` branch. On all non-hardhat chains, `_executeSlowModeTransaction` silently swallows this revert via a bare `try/catch`. The user's withdrawal is permanently dropped and the slow mode fee is irrecoverably lost.

---

### Finding Description

The `TransactionType` enum in `IEndpoint.sol` defines `WithdrawCollateralV2` (index 32). [1](#0-0) 

`processTransactionImpl` in `EndpointTx.sol` handles it correctly for the sequencer path (lines 437–465). [2](#0-1) 

However, `processSlowModeTransactionImpl` handles only `DepositCollateral`, `WithdrawCollateral` (V1), `DepositInsurance`, `LinkSigner`, `WithdrawInsurance`, `DelistProduct`, `DumpFees`, `RebalanceXWithdraw`, `UpdateTierFeeRates`, `AddNlpPool`, `UpdateNlpPool`, `DeleteNlpPool`, `ForceRebalanceNlpPool`, `NlpProfitShare`, `UpdateBuilder`, and `ClaimBuilderFee`. `WithdrawCollateralV2` is absent. Any unrecognized type falls to `else { revert(); }`. [3](#0-2) 

In `submitSlowModeTransactionImpl`, `WithdrawCollateralV2` is **not** in the owner-only guard list, so it falls to the `else` branch: the slow mode fee is charged and the transaction is queued without restriction. [4](#0-3) 

When `_executeSlowModeTransaction` later fires, it calls `processSlowModeTransaction` inside a `try/catch`. On any non-hardhat chain, the revert from the missing handler is silently consumed — the comment even notes "try return funds now removed" — meaning the user receives no refund and no error. [5](#0-4) 

---

### Impact Explanation

A user who submits `WithdrawCollateralV2` via slow mode suffers two concrete asset losses:

1. **Slow mode fee is permanently consumed** — `chargeSlowModeFee` deducts the fee from the user's quote balance at submission time; it is never returned.
2. **Withdrawal is permanently dropped** — the V2 withdrawal (which supports `sendTo` routing and compact signatures) never executes; the user's collateral remains locked in the protocol.

The slow mode queue is the protocol's censorship-resistance guarantee. For `WithdrawCollateralV2` — the current, preferred withdrawal type — this guarantee is entirely broken. A user who cannot reach the sequencer and submits a V2 withdrawal via slow mode will lose their fee and remain unable to exit.

---

### Likelihood Explanation

Any user who holds a position and attempts to withdraw via `WithdrawCollateralV2` through the slow mode path (e.g., during sequencer downtime, censorship, or as a precaution) will trigger this. `WithdrawCollateralV2` is the newer, preferred withdrawal struct (it adds `sendTo` and `appendix` fields over V1), so off-chain tooling and frontends are likely to generate it by default. No special privileges or contract knowledge are required — only a standard slow mode submission call.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` in `EndpointTx.sol`, mirroring the existing `WithdrawCollateral` (V1) handler but decoding `SignedWithdrawCollateralV2` / `WithdrawCollateralV2` and forwarding to `clearinghouse.withdrawCollateral` with the `sendTo` field populated:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.WithdrawCollateralV2 memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateralV2)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender,
        txn.productId,
        txn.amount,
        txn.sendTo,
        nSubmissions
    );
}
```

Additionally, consider whether the fee-charging logic in `submitSlowModeTransactionImpl` should explicitly revert for transaction types that have no slow mode handler, rather than silently accepting the fee.

---

### Proof of Concept

**Step 1 — Submit `WithdrawCollateralV2` via slow mode (fee charged, tx queued):**

`submitSlowModeTransactionImpl` receives `txType == WithdrawCollateralV2`. It is not `DepositCollateral` (which reverts), not `DepositInsurance`, and not in the owner-only list. It falls to `else { chargeSlowModeFee(...); slowModeFees += SLOW_MODE_FEE; }`. The transaction is appended to `slowModeTxs`. [6](#0-5) 

**Step 2 — After 3-day delay, `executeSlowModeTransaction` is called:**

`_executeSlowModeTransaction` dequeues the transaction and calls `this.processSlowModeTransaction(txn.sender, txn.tx)` inside a `try/catch`. [7](#0-6) 

**Step 3 — `processSlowModeTransactionImpl` hits `else { revert(); }`:**

`WithdrawCollateralV2` matches none of the 16 handled branches. The function reverts. [3](#0-2) 

**Step 4 — Revert is silently swallowed:**

The `catch` block checks only for out-of-gas conditions. A normal revert passes the gas check and is discarded. The comment "try return funds now removed" confirms no refund path exists. [8](#0-7) 

**Result:** User's slow mode fee is consumed, withdrawal is permanently unexecuted, and the user has no on-chain recourse.
