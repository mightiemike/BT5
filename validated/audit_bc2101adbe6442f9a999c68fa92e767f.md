### Title
Caller-supplied `flags` u32 silently enables undocumented hard-fork dialect flags via `from_bits_truncate` — (File: `wheel/src/api.rs`)

---

### Summary

The Python-facing `run_serialized_chia_program` function accepts a raw `u32` flags value and converts it with `ClvmFlags::from_bits_truncate(flags)`. This silently accepts any *defined* flag bit, including hard-fork flags (`RELAXED_BLS` 0x0008, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` 0x0100, `MALACHITE` 0x1000) that are intentionally **not exported** to Python. Any Python caller can pass these raw bit values to activate restricted dialect behaviors, bypassing the only access control in place (omitting the constants from the module).

---

### Finding Description

In `wheel/src/api.rs`, the public Python function `run_serialized_chia_program` accepts a caller-supplied `flags: u32` and converts it:

```rust
let flags = ClvmFlags::from_bits_truncate(flags);
``` [1](#0-0) 

`from_bits_truncate` (from the `bitflags` crate) silently accepts any bit that corresponds to a *defined* flag and drops only truly unknown bits. The Python module registers only a subset of flag constants: [2](#0-1) 

The following flags are defined in `ClvmFlags` but deliberately **not exported** to Python:

| Flag | Bit | Comment in source |
|---|---|---|
| `RELAXED_BLS` | 0x0008 | "Hard-fork; enable only when it activates" |
| `ENABLE_GC` | 0x0020 | Enables allocator checkpoint/restore |
| `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` | 0x0100 | "Hard-fork; enable only when it activates" |
| `MALACHITE` | 0x1000 | Switches div/divmod/mod/modpow to malachite-bigint | [3](#0-2) 

Despite not being exported, any Python caller can pass the raw integer `0x0100` and `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` will be silently activated. The operator dispatch in `ChiaDialect::op` then routes opcode 62 to `op_keccak256` unconditionally: [4](#0-3) 

Similarly, `MALACHITE` (0x1000) is consumed in `more_ops.rs` across 34 call sites to switch the arithmetic backend from `num-bigint` to `malachite-bigint` for `div`, `divmod`, `mod`, and `modpow`.

The only "access control" is the omission of the constant names from the Python module — but since `from_bits_truncate` accepts any defined bit, passing the raw integer value bypasses this entirely.

---

### Impact Explanation

**Consensus divergence.** If a Python caller (e.g., a Chia full node, a mempool validator, or a downstream library) passes `flags=0x0100`:

- Opcode 62 (`keccak256`) is dispatched as a live operator **outside** the softfork guard.
- On nodes that do not set this flag (the current network majority before the hard fork activates), opcode 62 is an unknown operator — either a no-op with a fixed cost (consensus mode) or an outright error (`NO_UNKNOWN_OPS` / mempool mode).
- A program accepted by the flag-enabled node is rejected or produces a different result on all other nodes → **chain split / double-spend surface**.

Enabling `MALACHITE` (0x1000) prematurely introduces a second divergence vector: arithmetic results for `div`/`divmod`/`mod`/`modpow` may differ between `num-bigint` and `malachite-bigint` for edge-case inputs, producing different output atoms and therefore different tree hashes for the same program.

---

### Likelihood Explanation

**Low–Medium.** The raw bit values are visible in the public Rust source (`src/chia_dialect.rs`). Any developer reading the source to understand available flags can discover and pass them. The risk is highest in:

1. Downstream Python wrappers that forward a user-supplied integer directly as `flags`.
2. Testing or tooling code that constructs flag values by OR-ing known constants with additional bits.
3. A node operator who manually sets flags for debugging and inadvertently enables a hard-fork flag before network activation.

No memory corruption or cryptographic break is required — a single integer argument is sufficient.

---

### Recommendation

Replace `from_bits_truncate` with an explicit allowlist of caller-approved flags in the Python entry point:

```rust
// Only permit flags that are safe for external callers to set.
const CALLER_ALLOWED: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::ENABLE_SHA256_TREE)
    .union(ClvmFlags::ENABLE_SECP_OPS);

let raw = ClvmFlags::from_bits_truncate(flags);
if raw.difference(CALLER_ALLOWED) != ClvmFlags::empty() {
    return Err(PyValueError::new_err("unsupported flags"));
}
let flags = raw;
```

Hard-fork flags (`RELAXED_BLS`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `MALACHITE`) should only be enabled internally by the consensus layer when the corresponding network upgrade activates, not by arbitrary Python callers.

### Proof of Concept

```python
import clvm_rs

# ENABLE_KECCAK_OPS_OUTSIDE_GUARD = 0x0100 (not exported, but accepted)
HIDDEN_FLAG = 0x0100

# A CLVM program that calls opcode 62 (keccak256) directly,
# outside any softfork guard.
# (ff 3e ff 80 80) = (62 ()) in CLVM serialized form
program = bytes.fromhex("ff3eff8080")
args    = bytes.fromhex("80")          # nil environment

# Without the hidden flag: opcode 62 is unknown → no-op or error
cost_normal, _ = clvm_rs.run_serialized_chia_program(
    program, args, 10_000_000, clvm_rs.NO_UNKNOWN_OPS
)
# Raises: EvalErr (unknown op)

# With the hidden flag: keccak256 executes and returns a 32-byte hash
cost_hidden, result = clvm_rs.run_serialized_chia_program(
    program, args, 10_000_000, clvm_rs.NO_UNKNOWN_OPS | HIDDEN_FLAG
)
# Succeeds — result is the keccak256 hash of nil.
# Other nodes running without 0x0100 reject this program → consensus split.
``` [5](#0-4) [6](#0-5)
