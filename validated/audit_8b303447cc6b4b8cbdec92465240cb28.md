### Title
Reduce-Only Order Constraint Validated on Local Copy but Not Enforced in Execution — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange._validateOrder` checks the reduce-only constraint by capping `order.amount` to the current position size on a **local memory copy**. The actual execution in `matchOrders` uses the original signed `taker.order.amount`, bypassing the reduce-only cap entirely. A reduce-only order with `|amount| > |position|` passes validation but executes for the full signed amount, silently opening a new position in the opposite direction — the exact same silent-truncation-instead-of-revert class as the MochiVault borrow bug.

---

### Finding Description

In `_validateOrder`, when the reduce-only flag is set, the function creates a local copy of the order and caps its amount to the current position size:

```solidity
IEndpoint.Order memory order = signedOrder.order;
// ...
int128 filledAmount = filledAmounts[orderDigest];
order.amount -= filledAmount;

if (_isReduceOnly(order.appendix)) {
    int128 amount = callState.isPerp
        ? callState.perp.getBalance(callState.productId, order.sender).amount
        : callState.spot.getBalance(callState.productId, order.sender).amount;
    if ((order.amount > 0) == (amount > 0)) {
        order.amount = 0;
    } else if (order.amount > 0) {
        order.amount = MathHelper.min(order.amount, -amount);   // capped locally
    } else if (order.amount < 0) {
        order.amount = MathHelper.max(order.amount, -amount);   // capped locally
    }
}

return ... && (order.amount != 0) && ...;
``` [1](#0-0) 

The capped value is used only for the `(order.amount != 0)` validity gate. Because `order` is a local `memory` copy of `signedOrder.order`, the cap **never propagates back** to `taker.order` in `matchOrders`.

Back in `matchOrders`, the execution amount is computed directly from the original signed order:

```solidity
if (taker.order.amount > 0) {
    ordersInfo.taker.amountDelta = MathHelper.min(
        taker.order.amount,       // ← original, uncapped amount
        -maker.order.amount
    );
}
``` [2](#0-1) 

There is no post-validation enforcement of the reduce-only cap. The `isHealthy` guard in `matchOrders` is a no-op stub that always returns `true`:

```solidity
function isHealthy(bytes32 /* subaccount */) internal view virtual returns (bool) {
    return true;
}
``` [3](#0-2) 

So the contract provides zero on-chain enforcement of the reduce-only constraint at execution time.

---

### Impact Explanation

A reduce-only order signed to close a position of size `N` with a signed amount `M > N` will:

1. Pass `_validateOrder` (capped local copy is non-zero, so `order.amount != 0` is satisfied).
2. Execute in `matchOrders` for the full `M` units.
3. Close the `N`-unit position **and open a new `M − N` position in the opposite direction**.

The user's explicit intent — to only reduce exposure — is violated. The user ends up with an unexpected open position, increased margin usage, and potential exposure to liquidation. This is a direct accounting corruption: the subaccount's position state diverges from what the user authorized.

---

### Likelihood Explanation

Two realistic, unprivileged-user-reachable triggers exist:

1. **Race condition (no malice required):** A user signs a reduce-only order when their position is `N`. Before the sequencer submits it, a separate fill partially closes the position to `N'`. The signed amount `M = N > N'` now exceeds the remaining position. The sequencer submits the original signed order in good faith; the contract executes it for `M`, opening a `M − N'` position.

2. **Sequencer submits stale or oversized reduce-only orders:** The sequencer batches transactions and may not re-validate reduce-only amounts against live position state before submission. The contract is the last line of defense and fails to enforce the constraint.

Neither path requires key compromise, governance capture, or social engineering.

---

### Recommendation

Enforce the reduce-only cap **inside `matchOrders`** at execution time, not only inside `_validateOrder`. After computing `ordersInfo.taker.amountDelta`, check whether the taker's order is reduce-only and clamp the delta to the current position size — or revert if the delta would exceed it:

```solidity
if (_isReduceOnly(taker.order.appendix)) {
    int128 currentPosition = /* fetch taker's current balance */;
    // Revert if the execution would flip or exceed the position
    require(
        (ordersInfo.taker.amountDelta > 0) != (currentPosition > 0) &&
        ordersInfo.taker.amountDelta.abs() <= currentPosition.abs(),
        ERR_REDUCE_ONLY_VIOLATION
    );
}
```

Alternatively, have `_validateOrder` return the capped amount and use that capped amount as the upper bound for `amountDelta` in `matchOrders`.

---

### Proof of Concept

1. User holds a long perp position of **50 units** in product `P`.
2. User signs a reduce-only sell order for **100 units** at price `X` (intending to close the 50-unit long).
3. Sequencer submits the order via `matchOrders` against a maker with 200 units available.
4. `_validateOrder` is called: local copy is capped to `min(100, 50) = 50`; returns `true` (50 ≠ 0). [4](#0-3) 
5. `matchOrders` computes `amountDelta = min(100, 200) = 100` using the original signed amount. [5](#0-4) 
6. `_updateBalances` applies a −100 base delta to the taker: closes the 50-unit long and opens a **50-unit short**. [6](#0-5) 
7. `isHealthy` returns `true` unconditionally; no revert occurs. [7](#0-6) 
8. The user now holds an unexpected 50-unit short position, violating the reduce-only constraint they signed.
