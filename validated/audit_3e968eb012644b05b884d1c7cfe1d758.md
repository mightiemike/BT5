### Title
Gas Key Exec-Fee Uses Raw Borsh `public_key_len` Instead of On-Trie `trie_id_len()` for ML-DSA-65 Keys in Host-Function Path — (`core/parameters/src/cost.rs`, `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The exec-fee helpers `gas_key_transfer_exec_fee` and `gas_key_add_key_exec_fee` compute the trie-key length using the caller-supplied `public_key_len` (the raw borsh-encoded byte count). When invoked from the **transaction path**, the callers correctly substitute `action.public_key.trie_id_len()` — the actual on-trie identifier length. When invoked from the **host-function path** (`promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`), the callers pass the raw `public_key_len` argument from the contract. For ED25519 and SECP256K1 the two values are identical, so no divergence exists today. For ML-DSA-65 they differ by 1 920 bytes (borsh = 1 953, trie-id = 33), producing a massive exec-fee overcharge on every gas-key host-function call that carries an ML-DSA-65 public key.

---

### Finding Description

**Correct transaction path** — `runtime/runtime/src/config.rs` lines 347–354:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
WithdrawFromGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

`trie_id_len()` returns 33 for ML-DSA-65 (1 tag byte + 32-byte SHA3-256 digest).

**Broken host-function path** — `runtime/near-vm-runner/src/logic/logic.rs` lines 3091–3096:

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw borsh length, not trie_id_len()
);
```

`public_key_len` for an ML-DSA-65 key is 1 953 (1 tag + 1 952 raw bytes). The same pattern repeats in `promise_batch_action_add_gas_key_with_full_access` (line 3158) and `promise_batch_action_add_gas_key_with_function_call` (line 3229).

The fee helper `gas_key_transfer_exec_fee` computes:

```rust
let trie_key_len = access_key_key_len(account_id_len, public_key_len);
``` [1](#0-0) 

With `public_key_len = 1953` instead of `33`, `trie_key_len` is inflated by 1 920 bytes. The per-byte exec fee is then multiplied by this inflated length, producing an overcharge of `1920 × gas_key_byte.exec_fee()` per action.

The same inflation applies inside `gas_key_add_key_exec_fee`:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
``` [2](#0-1) 

multiplied by `num_nonces`, so the overcharge scales with the number of nonces.

The transaction path correctly uses `trie_id_len()`: [3](#0-2) 

The host-function path uses the raw borsh length: [4](#0-3) [5](#0-4) [6](#0-5) 

The wasmtime runner duplicates the same error: [7](#0-6) [8](#0-7) [9](#0-8) 

The design intent is documented: `trie_id_len()` is the correct measure for storage-cost paths, while `len()` (borsh) is for wire-size paths: [10](#0-9)

---

### Impact Explanation

Once `ProtocolFeature::PostQuantumSignatures` is enabled, any smart contract can call `promise_batch_action_transfer_to_gas_key` or `promise_batch_action_add_gas_key_with_full_access/function_call` with an ML-DSA-65 public key. The exec gas charged is computed against a 1 953-byte key length instead of the correct 33-byte trie-id length — a factor of ~59× overcharge on the per-byte exec component. For `gas_key_add_key_exec_fee` with multiple nonces the overcharge multiplies further. This breaks the fee-parity invariant between the transaction and host-function paths for the same logical operation, and makes ML-DSA-65 gas-key operations via host functions economically unusable (contracts exhaust prepaid gas or pay far more than the equivalent transaction).

---

### Likelihood Explanation

The trigger requires `PostQuantumSignatures` to be active at the current protocol version and a contract to call the affected host functions with an ML-DSA-65 public key. The feature is already compiled into the binary (no `#[cfg]` guard on the host-function registration or the fee helpers), and the ML-DSA-65 key type is fully supported. Any unprivileged contract author can construct the triggering call once the feature is live.

---

### Recommendation

In all three host-function call sites, decode the public key first (which is already done for `append_action_*`), then pass `public_key.trie_id_len()` to the exec-fee helpers instead of the raw `public_key_len`:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),   // ← correct
);
```

Apply the same fix to `gas_key_add_key_exec_fee` call sites in both the wasmer (`logic.rs`) and wasmtime (`wasmtime_runner/logic.rs`) paths. Add a fee-parity test that exercises ML-DSA-65 keys through the host-function path, analogous to the existing `test_gas_key_fee_parity` test.

---

### Proof of Concept

1. Enable `PostQuantumSignatures` (nightly build or after stabilization).
2. Deploy a contract that calls:
   ```
   promise_batch_action_transfer_to_gas_key(
       promise_idx,
       1953,                  // ML-DSA-65 borsh length
       <ml_dsa_65_pubkey_ptr>,
       <amount_ptr>
   )
   ```
3. Observe that `gas_burnt` on the resulting receipt is ~59× higher than the equivalent `TransferToGasKey` submitted as a transaction with the same ML-DSA-65 public key.
4. Concretely: `gas_key_transfer_exec_fee` computes `access_key_key_len(account_id_len, 1953)` = `2 + account_id_len + 1953` instead of `2 + account_id_len + 33`, inflating the per-byte exec cost by 1 920 × `gas_key_byte.exec_fee()` per call.

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
