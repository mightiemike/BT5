### Title
Fork Retarget Validation Uses Mainchain Historical Block Instead of Fork Branch Block — (`contract/src/lib.rs::get_header_by_height`, `contract/src/litecoin.rs::get_next_work_required`)

---

### Summary

`get_header_by_height` unconditionally reads from `mainchain_height_to_header`. When `get_next_work_required` calls it during fork-block validation at a retarget boundary, it retrieves the **mainchain's** block at the period-start height rather than the fork's block at that height. An attacker who controls a fork that diverges before the difficulty-period start can craft a fork whose retarget block carries `bits` that the contract accepts (because it computed expected difficulty from the mainchain's historical timestamp) but that the real Litecoin network would reject (because it would compute expected difficulty from the fork's own historical timestamp).

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

The function resolves height → hash exclusively through `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`: [2](#0-1) 

They are **never** inserted into `mainchain_height_to_header`, so `get_header_by_height` can never return a fork block.

**Where the wrong block is consumed — `get_next_work_required`:**

At every retarget boundary (`(prev_height + 1) % interval == 0`), the function computes `first_block_height` and fetches the period-start block: [3](#0-2) 

The returned block's `.time` is fed directly into `calculate_next_work_required` as `first_block_time`, which drives the entire `actual_time_taken` computation and therefore the accepted `bits` value.

**Call chain that reaches this during fork submission:**

`submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height` [4](#0-3) [5](#0-4) 

`check_pow` is called for **every** submitted block, including fork blocks, before `submit_block_header_inner` decides whether to route to `store_fork_header`.

**Concrete discrepancy:**

Let the difficulty period span heights `[H, H+2015]` (Litecoin interval = 2016). Suppose the fork diverges at height `H-1`. The fork's block at height `H` has timestamp `T_fork`; the mainchain's block at height `H` has timestamp `T_main`.

When the fork's retarget block at height `H+2016` is validated:
- Contract computes: `actual_time_taken = T_prev_fork - T_main`
- Litecoin network computes: `actual_time_taken = T_prev_fork - T_fork`

If `T_fork > T_main`, the contract's `actual_time_taken` is larger → it accepts an **easier** `bits` than the real network would require. The attacker mines the fork with this easier difficulty (scrypt-valid, but with a `bits` value the real chain rejects), and the contract stores it as valid.

---

### Impact Explanation

If the fork accumulates enough chainwork (easier difficulty means each block is cheaper to mine), `reorg_chain` promotes it to mainchain: [6](#0-5) 

After reorg, `mainchain_height_to_header` and `mainchain_header_to_height` point to the attacker's fork blocks. Any downstream bridge calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against these heights will treat the fork's transactions as finalized on the canonical chain, enabling double-spend or false-finality attacks.

---

### Likelihood Explanation

The `#[trusted_relayer]` guard on `submit_blocks` prevents a direct unprivileged NEAR call: [7](#0-6) 

However, the production relayer path (`Synchronizer::sync → NearClient::sign_submit_blocks → BtcLightClient::submit_blocks`) is explicitly listed as a valid attacker entry point under "trusted-relayer submission path with attacker-relevant data." The relayer observes the Litecoin P2P network and relays any fork it sees; it does not re-validate difficulty against the fork's own branch. An attacker who mines a fork with scrypt-valid headers (feasible with modest hashrate given the reduced difficulty the bug permits) will have those headers relayed by an honest relayer. The timestamp manipulation needed (`T_fork > T_main` at the period-start block) is constrained only by the MTP rule, which the attacker satisfies by controlling all fork blocks before height `H`.

---

### Recommendation

`get_header_by_height` must be fork-aware. During fork validation, the lookup for `first_block_height` must walk the fork's own `prev_block_hash` chain rather than the mainchain index. One approach: pass the fork's ancestor chain (or a closure that traverses `headers_pool` via `prev_block_hash`) into `get_next_work_required` instead of the `BlocksGetter` trait that is bound to `mainchain_height_to_header`. Alternatively, add a separate `get_header_by_height_from_tip(height, tip_hash)` that walks backwards through `headers_pool` from a given tip until it reaches the requested height.
