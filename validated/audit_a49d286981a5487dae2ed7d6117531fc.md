Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass the Deposit Guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the address that actually provides tokens via callback) and checks only `owner` (the LP-position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly supports the operator pattern where `msg.sender ≠ owner`, any non-allowlisted address can deposit into a permissioned pool by naming an allowlisted address as `owner`, bypassing the guard entirely.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an `owner` parameter distinct from `msg.sender` and passes both to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both as separate arguments to the hook:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but discards it (unnamed), checking only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-41
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

The operator pattern (`sender ≠ owner`) is explicitly supported and tested — `test_exactShares_canAddOnBehalfOfAnotherOwner` and `test_exactShares_usesMsgSenderAsPayerNotOwner` confirm Alice can pay while Bob receives LP shares. The token pull in `LiquidityLib.addLiquidity` is made against `msg.sender` (the caller/payer), not `owner`:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol L147-148
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
```

The sibling `SwapAllowlistExtension.beforeSwap` correctly gates on `sender` (the actual swapper/token provider), making the asymmetry clear evidence of a bug rather than intentional design.

Exploit path:
1. Pool P uses `DepositAllowlistExtension`. Admin allowlists Alice: `allowedDepositor[P][Alice] = true`. Bob is not allowlisted.
2. Bob calls `pool.addLiquidity(owner = Alice, salt = X, deltas = ..., callbackData = ..., extensionData = "")`.
3. Hook fires: `msg.sender = P`, `owner = Alice`, `sender = Bob` (ignored). Check: `allowedDepositor[P][Alice] == true` → no revert.
4. Pool calls `IMetricOmmModifyLiquidityCallback(Bob).metricOmmModifyLiquidityCallback(...)` — Bob's tokens enter the pool. Alice receives LP shares she never requested.

No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner`, but `addLiquidity` has no such constraint.

## Impact Explanation
This is an admin-boundary break: the pool admin's configured deposit allowlist is bypassed by any unprivileged address. Consequences include: (1) non-allowlisted addresses depositing into pools designed to accept liquidity only from vetted counterparties, corrupting the pool's intended liquidity composition; (2) forced LP positions on allowlisted addresses, exposing them to impermanent loss and fee obligations on removal they never consented to. This directly matches the "Admin-boundary break" allowed impact category.

## Likelihood Explanation
Requires no special privilege. The only precondition is knowing one allowlisted address for the target pool, which is trivially discoverable on-chain from `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. Any EOA or contract can execute this. The `DepositAllowlistExtension` is a production periphery contract intended for real permissioned pools.

## Recommendation
Gate on `sender` (the token provider), consistent with `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  Pool P uses DepositAllowlistExtension E.
  Pool admin calls E.setAllowedToDeposit(P, Alice, true).
  Bob (non-allowlisted) implements IMetricOmmModifyLiquidityCallback and holds tokens.

Attack:
  Bob calls pool.addLiquidity(owner = Alice, salt = X, deltas = ..., callbackData = ..., extensionData = "")

Extension hook fires:
  msg.sender = P  (the pool)
  owner       = Alice
  sender      = Bob  (discarded by the extension)

Check: allowedDepositor[P][Alice] == true → hook returns selector, no revert.

Result:
  Bob's tokens enter the pool via his callback.
  Alice receives LP shares she did not request.
  Bob (non-allowlisted) has successfully deposited into a permissioned pool.

Foundry test: fork the deployed pool, prank as Bob, call addLiquidity with owner=Alice,
assert no revert and Alice's positionBinShares increase while Bob's token balance decreases.
```
