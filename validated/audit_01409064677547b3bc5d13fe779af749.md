### Title
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` Always Panic for Single-Transaction Blocks Due to Unconditional Empty-Proof Rejection — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` unconditionally rejects empty Merkle proofs via a `require!` guard. For a block containing exactly one transaction, the valid Merkle proof is legitimately empty (the Merkle root equals the txid). Because `verify_transaction_inclusion_v2` — the current recommended API — calls `verify_transaction_inclusion` internally, both public proof-verification entry points always panic for single-transaction blocks even when the caller supplies a cryptographically correct proof.

---

### Finding Description

In `verify_transaction_inclusion`, before the Merkle root is computed, the following guard is applied unconditionally:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This check fires before `compute_root_from_merkle_proof` is ever called. However, for a block that contains exactly one transaction (e.g., a miner-only coinbase block), the Merkle tree has a single leaf. The Merkle root is identical to the transaction hash, and no sibling hashes are needed — the proof is legitimately empty. The underlying `compute_root_from_merkle_proof` function handles this correctly: with an empty proof slice, the loop body never executes and the input hash is returned unchanged. [2](#0-1) 

`verify_transaction_inclusion_v2` — the non-deprecated replacement — calls `verify_transaction_inclusion` after validating the coinbase proof:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

For a single-tx block, both `merkle_proof` and `coinbase_merkle_proof` are legitimately empty. The length-equality check in v2 passes (`0 == 0`), the coinbase check passes (`coinbase_tx_id == merkle_root` since the coinbase is the only tx), but the subsequent call to `verify_transaction_inclusion` immediately panics on the empty-proof guard. [4](#0-3) 

---

### Impact Explanation

Any NEAR caller — a dApp, bridge contract, or SPV proof consumer — invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in a single-transaction block will always receive a contract panic, regardless of proof correctness. The corrupted invariant is the contract's core guarantee: `verify_transaction_inclusion_v2` is documented as the authoritative SPV proof API, yet it permanently rejects a valid class of Bitcoin blocks. Downstream contracts that rely on a `true` return value to gate asset releases or cross-chain actions are blocked for this block class.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are rare but valid and occur in production Bitcoin (and Litecoin/Dogecoin/Zcash) chains. The relayer accepts and submits such blocks normally via `submit_blocks` — the contract stores them without issue. The failure is triggered only at proof-verification time by any unprivileged NEAR caller supplying a valid but empty proof. No special privileges or key material are required.

---

### Recommendation

Replace the unconditional empty-proof guard with a branch that allows the empty-proof case when the transaction hash equals the block's Merkle root:

```rust
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

This mirrors how `compute_root_from_merkle_proof` already behaves for an empty slice and correctly handles single-transaction blocks without weakening the proof check for multi-transaction blocks.

---

### Proof of Concept

1. A single-tx block (coinbase only) is submitted via `submit_blocks` — accepted normally.
2. A caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_txid` (= `merkle_root` for a single-tx block)
   - `tx_block_blockhash = block_hash`
   - `tx_index = 0`
   - `merkle_proof = []` (valid empty proof)
   - `coinbase_tx_id = coinbase_txid`
   - `coinbase_merkle_proof = []` (valid empty proof)
   - `confirmations = 1`
3. Length check: `0 == 0` → passes.
4. Coinbase check: `compute_root_from_merkle_proof(coinbase_txid, 0, &[])` returns `coinbase_txid`; `coinbase_txid == merkle_root` → passes.
5. `verify_transaction_inclusion` is called internally.
6. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` → **panics**, despite the proof being cryptographically valid. [5](#0-4)

    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);

            current_hash = compute_hash(proof_hash, &current_hash);
