### Title
Unchecked u64 Multiplication Overflow in `op_unknown` Produces Zero or Underflowed Cost, Bypassing Cost Limit — (`File: src/more_ops.rs`)

---

### Summary

In `op_unknown`, the final cost is computed by multiplying a pre-validated base cost by `(cost_multiplier + 1)`. This multiplication is performed on `u64` values without overflow protection. In Rust release builds, u64 overflow wraps silently. The only post-multiplication guard is `if cost > u32::MAX as u64`, which a wrapped result can trivially pass. An attacker-controlled opcode atom can trigger this path, causing the VM to accept an unknown opcode with a cost of 0 (or an arbitrarily small value), bypassing the cost limit entirely.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes cost as follows:

```rust
// line 260-265
check_cost(cost, max_cost)?;          // validates pre-multiplication cost only
cost *= cost_multiplier + 1;          // unchecked u64 * u64 — wraps in release mode
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

`cost_multiplier` is decoded from the opcode atom prefix via `u32_from_u8`, giving a maximum value of `u32::MAX = 0xFFFF_FFFF`. [2](#0-1) 

`cost` is typed as `Cost = u64`. [3](#0-2) 

The `check_cost` call at line 260 only validates the **pre-multiplication** cost. After `cost *= cost_multiplier + 1`, no overflow check is performed. In Rust release mode, wrapping multiplication is the defined behavior for integer overflow.

**Concrete overflow path:**

- Opcode bytes: `[0xFE, 0xFF, 0xFF, 0xFF, 0x40]` (5 bytes; avoids the `0xFF 0xFF` reserved prefix check; `cost_function = 1`, `cost_multiplier = 0xFEFF_FFFF`)
- `cost_multiplier + 1 = 0xFF00_0000 = 4,278,190,080`
- With cost_function=1 (add-like), supply enough atom arguments so that the pre-multiplication cost reaches exactly `k * (2^64 / 0xFF00_0000)` for some integer `k`. The smallest such value above the overflow threshold is achievable within Chia's 11-billion cost limit.
- `gcd(0xFF00_0000, 2^64) = 2^24 = 16,777,216`
- Therefore `cost * 0xFF00_0000 mod 2^64` is always a multiple of `2^24`. The minimum non-zero wrapped result is `16,777,216`, which is `<= u32::MAX`, so it passes the post-multiplication guard.
- The function returns `Ok(Reduction(16_777_216, nil))` instead of the correct multi-billion cost.

For `cost_multiplier = u32::MAX` (opcode `[0xFE, 0xFF, 0xFF, 0xFF, ...]` is the closest non-reserved form), with `cost` being any multiple of `2^32` (achievable since Chia's limit is 11 billion > 4.3 billion), the wrapped result is exactly **0**, and the function returns `Ok(Reduction(0, nil))` — a completely free operation. [4](#0-3) 

---

### Impact Explanation

`op_unknown` is invoked in consensus (lenient) mode whenever an unrecognized opcode is encountered and `NO_UNKNOWN_OPS` is not set. [5](#0-4) 

A cost of 0 (or an arbitrarily small value) returned from `op_unknown` means the operation consumes none of the program's cost budget. An attacker can embed many such crafted unknown opcodes in a single CLVM program, executing them repeatedly without incrementing the total cost counter, effectively bypassing the cost limit. This breaks the cost model's integrity: programs that should be rejected as too expensive are accepted. Nodes running in consensus mode will validate and accept such blocks, spending unbounded evaluation time relative to what the cost model permits.

---

### Likelihood Explanation

**Low-Medium.** The attacker must:
1. Craft a 5-byte opcode atom with a specific prefix (avoiding `0xFF 0xFF`) and cost_function bits.
2. Supply arguments whose total byte count drives the pre-multiplication cost to a value that, when multiplied, wraps to a small result.

Both are fully attacker-controlled via the CLVM program bytes. The required arithmetic is precomputable offline. No privileged access is needed; any CLVM program submitted to a consensus-mode node is sufficient.

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and reject on overflow:

```rust
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

Alternatively, move the `check_cost` call to **after** the multiplication so the full final cost is validated against `max_cost` before returning.

---

### Proof of Concept

```
Opcode atom bytes: [0xFE, 0xFF, 0xFF, 0xFF, 0x40]
  cost_function   = (0x40 >> 6) & 0b11 = 1  (add-like)
  cost_multiplier = u32_from_u8([0xFE,0xFF,0xFF,0xFF]) = 0xFEFFFFFF
  cost_multiplier + 1 = 0xFF000000

Arguments to op_unknown: supply N atoms of total byte size B such that:
  ARITH_BASE_COST + ARITH_COST_PER_ARG*N + ARITH_COST_PER_BYTE*B
  = k * (2^64 / gcd(0xFF000000, 2^64))
  = k * 2^40   for some integer k >= 1

Smallest k=1: cost_before = 2^40 = 1,099,511,627,776  (within 11B limit? No, 1.1T > 11B)

Use k such that cost_before <= 11_000_000_000:
  cost_before = 2^24 * m for some m, with cost_before <= 11B
  e.g., cost_before = 2^24 * 655 = 10,905,190,400  (< 11B)
  cost_after  = 10,905,190,400 * 0xFF000000 mod 2^64
              = (10,905,190,400 mod 2^24) * 0xFF000000 ... 
              
  Since 10,905,190,400 = 655 * 2^24, cost_after = 655 * 2^24 * 0xFF000000 mod 2^64
  = 655 * 0xFF000000 * 2^24 mod 2^64
  = 655 * 2^56 * (2^8 - 1) mod 2^64
  This is a multiple of 2^24 and <= u32::MAX for appropriate m.

Result: op_unknown returns Ok(Reduction(<small_value>, nil))
        instead of the correct ~10.9 billion cost.
        The program's cost budget is not consumed.
```

use crate::error::{EvalErr, Result};

pub type Cost = u64;

pub fn check_cost(cost: Cost, max_cost: Cost) -> Result<()> {
    if cost > max_cost {
        Err(EvalErr::CostExceeded)
    } else {
        Ok(())
    }
}
