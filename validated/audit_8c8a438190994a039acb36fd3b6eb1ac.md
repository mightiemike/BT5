### Title
Missing Access Control on `creditDepositV1()` Allows Anyone to Force Deposit for Any Subaccount — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.creditDepositV1(bytes32 subaccount)` is an `external` function with no caller restriction. Any unprivileged address can invoke it for any subaccount, triggering `DirectDepositV1.creditDeposit()`, which sweeps all token balances held in the user's `DirectDepositV1` contract and deposits them into the Nado protocol on behalf of that subaccount. The user has no ability to prevent or time this action.

---

### Finding Description

`creditDepositV1` in `ContractOwner.sol` accepts an arbitrary `subaccount` bytes32 value and immediately calls `creditDeposit()` on the associated `DirectDepositV1` contract without verifying that `msg.sender` is the owner of that subaccount:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [1](#0-0) 

`DirectDepositV1.creditDeposit()` is itself also `external` with no access control, and sweeps every supported token balance into the protocol for the hardcoded `subaccount`:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        ...
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(
                subaccount, productId, uint128(balance), "-1"
            );
        }
    }
}
``` [2](#0-1) 

Critically, the user **cannot** independently withdraw tokens from their `DirectDepositV1` contract — `withdraw()` and `withdrawNative()` are both `onlyOwner` (owner = `ContractOwner`/multisig): [3](#0-2) 

This means the only path for tokens sitting in a `DirectDepositV1` contract is to be deposited into the protocol. An attacker can force that path to execute at any moment, removing the user's sole remaining control: **timing**.

---

### Impact Explanation

A user who has sent tokens to their `DirectDepositV1` address may be deliberately deferring the deposit — for example, waiting for a favorable oracle price update, waiting to first close an existing position, or waiting to avoid having the collateral counted against a position they intend to unwind. An attacker can call `creditDepositV1(subaccount)` at any time to force the deposit immediately, bypassing the user's intent. The deposit is irreversible from the user's perspective (withdrawal requires a signed sequencer transaction and a delay). The funds reach the correct subaccount, but the user's ability to control the timing of their own collateral entry is eliminated. This matches the Sublime analog: funds go to the right place, but the forced action at an inopportune time constitutes a leak of value (lost optionality over collateral timing and position management).

---

### Likelihood Explanation

The entry point is a public `external` function on a deployed contract with no authentication. Any EOA or contract can call `ContractOwner.creditDepositV1(subaccount)` with any valid subaccount bytes32. No special role, signature, or prior state is required. The attacker only needs to know the target subaccount identifier, which is derivable from the user's address and subaccount name (both public on-chain).

---

### Recommendation

Add a caller check in `creditDepositV1` so that only the address encoded in the subaccount bytes32 can trigger the deposit:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    require(
        address(bytes20(subaccount)) == msg.sender,
        "Unauthorized: caller is not subaccount owner"
    );
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
```

Similarly, restrict `DirectDepositV1.creditDeposit()` to be callable only by the `ContractOwner` (its owner), so the access control is enforced at both layers.

---

### Proof of Concept

1. Alice sends 1000 USDC to her `DirectDepositV1` address, intending to deposit into the protocol only after she closes an existing leveraged position.
2. Bob (unprivileged attacker) calls `ContractOwner.creditDepositV1(aliceSubaccount)`.
3. `creditDeposit()` sweeps all 1000 USDC and calls `endpoint.depositCollateralWithReferral(aliceSubaccount, ...)`.
4. Alice's subaccount now has 1000 USDC of collateral she did not intend to deposit yet. Her position health and margin calculations change without her consent. She cannot reverse the deposit without going through the sequencer withdrawal flow (which incurs fees and a delay). [1](#0-0) [2](#0-1)
