Audit Report

## Title
Router Address Substitution Bypasses SwapAllowlistExtension Per-User Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is whoever called `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` resolves to the router address rather than the originating EOA. A pool admin who allowlists the router inadvertently grants every user access, defeating the per-user allowlist entirely.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly — `msg.sender` to the pool is the router contract, not the originating EOA: [4](#0-3) 

When a pool admin sets `allowedSwapper[pool][router] = true` (the expected configuration to allow router-mediated swaps on a curated pool), the check `allowedSwapper[pool][router]` passes for every user who calls through the router, regardless of whether their own address is allowlisted. The extension has no mechanism to recover the originating EOA — `extensionData` passed by the router is `""` (empty) by default. [5](#0-4) 

## Impact Explanation

Any non-allowlisted EOA can swap on a pool intended to be restricted to specific users by routing through `MetricOmmSimpleRouter`. The pool's curation invariant is broken: the pool admin believes only allowlisted addresses can trade, but in practice the allowlist is bypassed for all router-mediated swaps. This enables unrestricted trading on pools designed for specific counterparties (e.g., KYC'd users, institutional LPs), with direct fund impact if the pool's pricing or liquidity is calibrated for a restricted set of traders. This constitutes broken core pool functionality causing potential loss of funds, meeting Sherlock Medium/High thresholds.

## Likelihood Explanation

The scenario requires only that the pool admin has allowlisted the router — the natural and expected configuration for any curated pool also meant to be accessible via the standard router. No privileged or malicious action is needed beyond normal pool setup. Any user can then call `exactInputSingle` from a non-allowlisted EOA. The condition is trivially reachable and repeatable.

## Recommendation

`SwapAllowlistExtension.beforeSwap` should check the originating user rather than the immediate caller of `pool.swap`. Options:

1. **Pass the original payer through `extensionData`**: the router encodes `msg.sender` into `extensionData`, and the extension decodes and checks it. Requires a trusted encoding convention between router and extension.
2. **Use `tx.origin` as a fallback check**: if `sender` is a known router, fall back to `tx.origin`. This is fragile but avoids router-side changes.
3. **Document incompatibility**: explicitly document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this via an `onlyEOA` check (`require(tx.origin == sender)`) inside `beforeSwap`.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists only the router:
//    allowedSwapper[pool][router] = true
// 3. Non-allowlisted EOA calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: curated_pool,
    ...
}));
// 4. pool.swap(msg.sender=router) -> _beforeSwap(sender=router)
// 5. allowedSwapper[pool][router] == true -> passes
// 6. Swap executes for non-allowlisted EOA
// 7. Direct pool.swap() from same EOA reverts (allowedSwapper[pool][eoa] == false)
```
