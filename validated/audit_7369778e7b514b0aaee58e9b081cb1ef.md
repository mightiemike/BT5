### Title
Transaction-level `receipt.da_gas` uniformly stamped onto every nested `FunctionInvocation.execution_resources.da_gas_consumed` in simulation/trace responses — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`FunctionInvocation::try_from` receives the transaction-level `receipt.da_gas` as its `gas_vector` argument and passes it **unchanged** to every recursive inner-call conversion. The result is that every `FunctionInvocation` node in the returned trace tree — regardless of depth or actual storage activity — reports the same `da_gas_consumed` value equal to the whole transaction's DA gas. This is reachable by any unprivileged caller of `starknet_simulateTransactions` or `starknet_traceTransaction`.

---

### Finding Description

**Entry points** — all four trace constructors pass `receipt.da_gas` as the `gas_vector`: [1](#0-0) [2](#0-1) 

The same pattern is repeated for `DeclareTransactionTrace`, `DeployAccountTransactionTrace`, and `L1HandlerTransactionTrace`. [3](#0-2) [4](#0-3) 

**Recursive propagation** — inside `FunctionInvocation::try_from`, the same `gas_vector` is forwarded to every element of `inner_calls` without modification: [5](#0-4) 

**Stamping** — `vm_resources_to_execution_resources` then writes that value directly into `da_gas_consumed` for the call: [6](#0-5) 

**Concrete wrong value:** for a transaction whose `receipt.da_gas = { l1_gas: 1652, l1_data_gas: 1, l2_gas: 1 }`, every inner call — including ones that wrote zero storage slots — will report `da_gas_consumed = { l1_gas: 1652, l1_data_gas: 1, l2_gas: 1 }`.

**Public path:** `starknet_simulateTransactions` is an unauthenticated RPC method. Any caller can submit a transaction with nested contract calls and observe the inflated per-call DA gas values. [7](#0-6) 

---

### Impact Explanation

The `starknet_simulateTransactions` and `starknet_traceTransaction` responses are the authoritative source of per-call resource accounting for tooling, block explorers, gas profilers, and contract developers. Every inner `FunctionInvocation.execution_resources.da_gas_consumed` field carries the full transaction-level DA gas instead of the call's own contribution. A contract with two inner calls that each write one storage slot will show both inner calls consuming the same DA gas as the entire transaction, making it impossible to attribute DA costs to individual calls from the trace alone.

This falls squarely within: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Any invoke transaction that triggers at least one inner contract call (the common case for account abstraction wallets, multicall patterns, DeFi protocols) will produce the wrong per-call `da_gas_consumed`. The path requires no privilege — only a valid `starknet_simulateTransactions` or `starknet_traceTransaction` call.

---

### Recommendation

`CallInfo` does not carry per-call DA gas today (DA gas is a transaction-level aggregate computed from `StateChanges`). Two options:

1. **Short-term:** Set `da_gas_consumed` to `GasVector::ZERO` for all inner calls and document that the field is only meaningful at the top-level invocation, preventing the misleading inflation.
2. **Long-term:** Track per-call state-change counts inside `CallInfo` (storage writes, class-hash updates, etc.) and compute per-call DA gas during the `FunctionInvocation` conversion, summing correctly up the call tree.

---

Build a test with a contract that makes two inner calls with different numbers of storage writes, call `simulate_transactions`, and assert:
