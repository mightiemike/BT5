### Title
Migration Abort on Unbonding Validator: `ForceFullExitWithDelegation` Returns `ErrValidatorNotBonded` During v8 Upgrade, Halting the Chain — (`x/tieredrewards/migrations/v2/migrate.go`, `x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/keeper/transfer_delegation.go`)

### Summary

`Migrate1to2` calls `exitVestedAccountsPositions`, which calls `ForceFullExitWithDelegation` for every vesting-account-owned position. That function calls `transferDelegationFromPosition`, which contains a hard `IsBonded()` guard. If the position's validator is in unbonding or unbonded state at upgrade time, the guard returns `ErrValidatorNotBonded`, the error propagates unhandled through every layer, and the upgrade handler fails. In Cosmos SDK, a failed upgrade handler causes a chain halt.

`exitVestedAccountsPositions` iterates all positions, identifies those owned by vesting accounts, and calls `ForceFullExitWithDelegation` on each one with no error-type discrimination: [1](#0-0) 

`ForceFullExitWithDelegation` propagates the error from `transferDelegationFromPosition` without catching `ErrValidatorNotBonded`: [2](#0-1) 

`transferDelegationFromPosition` performs the hard bonding-state check: [3](#0-2) 

`ErrValidatorNotBonded` is a registered module error, not a sentinel that any caller wraps or ignores: [4](#0-3) 

The `Migrate` function wraps and returns the error without any recovery: [5](#0-4) 

### Impact Explanation

A validator enters unbonding state automatically when it is jailed for downtime — no attacker key or governance action is required. If any vesting account holds a tieredrewards position on such a validator at the moment the v8 upgrade block is processed, `Migrate1to2` returns a non-nil error, the upgrade handler fails, and the chain halts. All tieredrewards positions (including non-vesting ones) remain in the pre-migration v1 state and cannot be operated on. The existing migration test `TestMigrate1to2_ExitsVestedOwnerPositions` only exercises a bonded validator and does not cover this case: [6](#0-5) 

### Likelihood Explanation

Vesting accounts are common on Cosmos chains (genesis vesting, team allocations). Validator downtime jailing is a routine chain event. The intersection of these two conditions — a vesting account with a tieredrewards position on a currently-jailed validator — is realistic and requires no privileged action by any party.

### Recommendation

In `exitVestedAccountsPositions` (or in `ForceFullExitWithDelegation` itself), catch `ErrValidatorNotBonded` and handle the non-bonded case separately — for example, by initiating a standard unbonding delegation from the position's delegator address back to the owner, or by skipping the `IsBonded()` guard inside a migration-specific exit path. The migration must not propagate this error as a fatal failure.

### Proof of Concept

1. In an integration test, create a vesting account and fund it.
2. Create a tieredrewards position for that account on validator V.
3. Jail validator V (set its status to `Unbonding`).
4. Call `migrator.Migrate1to2(ctx)`.
5. Assert the returned error is non-nil and wraps `ErrValidatorNotBonded`.

This directly proves the upgrade handler would abort and halt the chain.
