### Title
`verify_transaction_inclusion` Always Reverts for Single-Transaction Blocks Due to Unconditional Empty-Proof Rejection - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2` which calls it) unconditionally panics when the supplied `merkle_proof` is empty. For a block containing exactly one transaction, the valid merkle proof **is** the empty list — the merkle root equals the transaction hash directly — so every SPV verification call for such blocks always reverts. This is structurally identical to M-9: a required data dependency (a non-empty proof) is unavailable for a specific, legitimate class of inputs, making the entire verification path permanently unusable for that class.

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The `compute_root_from_merkle_proof` implementation in `merkle-tools/src/lib.rs` handles an empty proof correctly — it simply returns the input hash unchanged:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof { ... }   // loop body never executes for empty proof
    current_hash                              // returns transaction_hash as-is
}
``` [2](#0-1) 

For a single-transaction block, `merkle_root == tx_id`. Passing `merkle_proof = []` would cause `compute_root_from_merkle_proof` to return `tx_id`, which correctly equals `merkle_root`. The comparison would return `true`. However, the `require!` guard fires first and reverts the call unconditionally, making correct verification impossible.

`verify_transaction_inclusion_v2` is equally affected: it validates the coinbase proof (which is also empty for a single-transaction block and passes correctly because `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == coinbase_tx_id == merkle_root`), then delegates to `verify_transaction_inclusion`, which panics. [3](#0-2) 

### Impact Explanation

Any unprivileged NEAR caller or downstream contract invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction residing in a single-transaction block will always receive a contract panic. The SPV verification path — the primary consumer-facing feature of this light client — is permanently broken for that block class. No funds are directly at risk, but the protocol's core verification guarantee is violated for a legitimate subset of Bitcoin blocks, matching the medium-severity profile of M-9.

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) occur regularly on Bitcoin and all supported chains (Litecoin, Dogecoin, Zcash), particularly during low-fee periods or early chain history. Any bridge, oracle, or application that needs to prove a coinbase transaction's inclusion in such a block will be permanently blocked. The trigger requires no privilege: any NEAR account can call `verify_transaction_inclusion` with a valid single-transaction block hash and an empty proof.

### Recommendation

Remove the blanket empty-proof rejection and instead handle the single-transaction case explicitly:

```rust
// Allow empty proof only when tx_id directly matches the merkle root
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

This preserves the existing behaviour for multi-transaction blocks while correctly returning `true` for the degenerate single-transaction case.

### Proof of Concept

1. A Bitcoin block with exactly one transaction (coinbase) has `merkle_root == coinbase_txid`.
2. The relayer submits this block header; it is accepted and stored in `headers_pool`.
3. A NEAR caller invokes `verify_transaction_inclusion` with:
   - `tx_id = coinbase_txid`
   - `tx_block_blockhash = <that block's hash>`
   - `merkle_proof = []` (correct proof for a single-transaction block)
   - `confirmations = 1`
4. Execution reaches `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` at line 315 and panics — the call reverts.
5. No alternative proof exists; the empty list is the only mathematically correct proof. The transaction can never be verified through this contract. [4](#0-3)

            args.confirmations <= self.gc_threshold,

            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1

            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
