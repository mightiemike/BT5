### Title
Raw `transferFrom` in `replaceUsdcEWithUsdc` Bypasses `safeTransferFrom` Wrapper, Silently Failing for Non-Boolean-Returning Tokens — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly at line 616, bypassing the protocol's own `ERC20Helper.safeTransferFrom` wrapper. The `IERC20Base` interface declares `transferFrom` as returning `bool`; in Solidity ^0.8.0, if the actual token returns no data, the ABI decoder reverts. The rest of the codebase consistently uses `ERC20Helper.safeTransferFrom` to handle this case via low-level `.call()`. The function has no `onlyOwner` guard, making it reachable by any unprivileged caller.

---

### Finding Description

`ERC20Helper.safeTransferFrom` is the protocol-wide safe wrapper:

```solidity
// core/contracts/libraries/ERC20Helper.sol
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transferFrom.selector, from, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
``` [1](#0-0) 

This wrapper correctly handles tokens that return no boolean by checking `data.length == 0 || abi.decode(data, (bool))`.

In `replaceUsdcEWithUsdc`, the inbound USDC leg uses the raw ABI call instead:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);  // line 616 — raw call
...
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                  // line 618 — safe call
``` [2](#0-1) 

The raw call at line 616 goes through the Solidity ABI decoder. If the token at `0x2D270e6886d130D724215A266106e6832161EAEd` (USDC on chain 57073 / Ink) returns no data or a non-standard return, the ABI decoder reverts the entire transaction. The outbound USDC.e leg on line 618 correctly uses `safeTransfer` from `ERC20Helper`.

The function carries no `onlyOwner` modifier — only a chain-ID gate — so any unprivileged caller on chain 57073 can invoke it:

```solidity
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
}
``` [3](#0-2) 

`IERC20Base` declares `transferFrom` as returning `bool`:

```solidity
function transferFrom(address from, address to, uint256 amount) external returns (bool);
``` [4](#0-3) 

---

### Impact Explanation

If the USDC token at the hardcoded address does not return a boolean on `transferFrom`, every call to `replaceUsdcEWithUsdc` reverts. The consequence is that USDC.e balances held in `directDepositV1` contracts cannot be migrated to USDC through this path. The USDC.e remains stranded in the `DirectDepositV1` contract until an owner-gated recovery path (`withdrawFromDirectDepositV1`) is used. This constitutes a token-transfer integration failure that blocks the intended migration flow and may strand user funds in the deposit adapter. [5](#0-4) 

---

### Likelihood Explanation

**Low.** The USDC address is hardcoded to a specific deployment on chain 57073 (Ink). Standard Circle USDC implementations do return a boolean. However, the inconsistency is concrete and structural: the same function uses `safeTransfer` for the outbound leg and raw `transferFrom` for the inbound leg, with no justification. If the Ink USDC deployment is a non-standard wrapper or proxy that omits the return value, the function permanently breaks. The permissionless entry path (no `onlyOwner`) means any user can trigger and observe the failure.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`: [6](#0-5) 

No additional imports are needed.

---

### Proof of Concept

1. Deploy or interact with `ContractOwner` on chain 57073.
2. Ensure a `directDepositV1` exists for a `subaccount` with a non-zero USDC.e balance.
3. Call `replaceUsdcEWithUsdc(subaccount)` as any unprivileged EOA with sufficient USDC allowance.
4. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` does not return a boolean, the ABI decoder at line 616 reverts with a generic decode error.
5. The USDC.e balance remains in the `DirectDepositV1` contract; the migration is permanently blocked through this path.
6. Contrast with line 618 (`safeTransfer`) which would succeed for the same token class, confirming the asymmetric handling is the root cause. [7](#0-6)

        require(block.chainid == 57073, ERR_UNAUTHORIZED);

            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
