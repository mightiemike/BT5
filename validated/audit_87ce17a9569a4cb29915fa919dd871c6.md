Audit Report

## Title
Zero Timelock Accepted at Initialization and via `proposeOracleStopLossTimelock` Allows Pool Admin to Bypass LP-Protection Timelock — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension.initialize` decodes a `timelock` parameter from caller-supplied `data` but applies no minimum-value validation, silently accepting `timelock = 0`. Additionally, `proposeOracleStopLossTimelock` imposes no lower-bound on `newTimelock`, allowing a pool admin to reduce a previously non-zero timelock to zero after the original delay elapses. With `timelock = 0`, the propose-then-execute flow collapses to a single block, letting the admin set `drawdownE6 = 1e6` and permanently disable the stop-loss guard for all future swaps.

## Finding Description

**Root cause 1 — `initialize` (L56–62):** `_validateDrawdown` and `_validateDecay` are called, but `timelock` is stored without any check:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) =
    abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // checked
_validateDecay(decayPerSecondE8); // checked
// timelock — NOT checked; zero silently accepted
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8,
    timelock: timelock, initialized: true
});
```

**Root cause 2 — `proposeOracleStopLossTimelock` (L78–84):** `newTimelock` is stored with no lower-bound validation, so an admin can propose `newTimelock = 0` on any already-deployed pool.

**Timelock bypass mechanics (L297–303):** `_afterTimelock` computes `block.timestamp + 0 = block.timestamp`. `_requireElapsed` checks `block.timestamp < executeAfter`; with `executeAfter = block.timestamp` this is `false`, so it never reverts. Propose and execute can occur in the same transaction.

**Stop-loss disabled (L234, L334):** Once `drawdownE6 = 1e6` is executed, `floorMultiplier = E6 − 1e6 = 0`. Inside `_applyWatermark`, `breached = metric < (hwm * 0) / E6` reduces to `metric < 0`, which is always `false` for `uint256`. The early-return guard at L217 (`if (drawdown == 0) return`) does not fire because `drawdown = 1e6 ≠ 0`, so the function proceeds to compute a permanently-false breach condition. Every subsequent `afterSwap` call passes without reverting regardless of how far LP bin value has fallen.

## Impact Explanation

The timelock is the sole on-chain mechanism giving LPs a reaction window before the pool admin can alter drawdown, decay, or watermark parameters. Eliminating it via `timelock = 0` lets the admin disable the stop-loss guard in a single block. After `drawdownE6 = 1e6` is executed, swaps that drain LP bins below any prior watermark proceed without revert, causing direct loss of LP token0 and token1 principal with no on-chain recourse. This is a confirmed admin-boundary break with direct fund loss, meeting the Critical/High threshold.

## Likelihood Explanation

**Path A (initialization):** The factory calls `initialize` with admin-supplied `data`; no factory-level validation of the `timelock` field exists. A pool creator who is also the pool admin passes `timelock = 0` at deployment. LPs deposit trusting the stop-loss, unaware the timelock window is zero.

**Path B (post-deployment reduction):** A pool deployed with a legitimate non-zero timelock (e.g., 24 hours) can have its timelock reduced to zero by the admin: call `proposeOracleStopLossTimelock(pool_, 0)`, wait 24 hours, call `executeOracleStopLossTimelock`. From that point the admin can propose and execute `drawdownE6 = 1e6` in the same block. Both paths require only pool-admin privilege — the exact role the timelock is designed to constrain.

## Recommendation

Add a minimum-timelock constant and a validation function, then apply it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 hours;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(timelock);
}
```

Call `_validateTimelock(timelock)` alongside `_validateDrawdown` / `_validateDecay` in `initialize`, and call `_validateTimelock(newTimelock)` at the top of `proposeOracleStopLossTimelock`.

## Proof of Concept

**Path A:**
1. Factory deploys pool and calls `initialize(pool, abi.encode(500_000, 58, 0))` — `timelock = 0`. No revert.
2. LPs deposit trusting the stop-loss guard.
3. Pool admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)`. `executeAfter = block.timestamp + 0 = block.timestamp`.
4. In the same transaction, pool admin calls `executeOracleStopLossDrawdown(pool)`. `_requireElapsed(block.timestamp)` → `block.timestamp < block.timestamp` → `false` → no revert. `drawdownE6` is set to `1_000_000`.
5. `floorMultiplier = 1e6 − 1e6 = 0`. All future `afterSwap` calls reach `_applyWatermark` with `floorMultiplier = 0`; `breached` is always `false`; stop-loss never fires.
6. Swaps draining LP bins below any prior watermark proceed without revert. LP principal is lost.

**Path B:**
1. Pool deployed with `timelock = 86400` (24 hours).
2. Admin calls `proposeOracleStopLossTimelock(pool_, 0)`. `executeAfter = block.timestamp + 86400`.
3. After 24 hours, admin calls `executeOracleStopLossTimelock(pool_)`. Timelock is now 0.
4. Admin immediately executes steps 3–6 from Path A in a single block.
