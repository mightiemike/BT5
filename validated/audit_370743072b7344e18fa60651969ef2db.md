### Title
Unexpected Panic in `verify_transaction_inclusion` Causes 3rd-Party NEAR Contract Reverts - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public boolean-returning verification functions. However, several of their internal validation checks call `env::panic_str` (a hard NEAR panic) instead of returning `false`. Any 3rd-party NEAR contract that calls these functions expecting a boolean result will instead receive an unexpected cross-contract panic, reverting its entire transaction and blocking the rest of its interactions.

---

### Finding Description

`verify_transaction_inclusion` is documented as returning `true` or `false` to indicate whether a transaction is included in a Bitcoin block. However, multiple validation paths inside it unconditionally panic rather than returning `false`:

**Panic 1 — block hash not in main chain** (`contract/src/lib.rs`, lines 299–301):
```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [1](#0-0) 

**Panic 2 — empty merkle proof** (`contract/src/lib.rs`, line 315):
```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [2](#0-1) 

**Panic 3 — insufficient confirmations** (`contract/src/lib.rs`, lines 304–308):
```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [3](#0-2) 

`verify_transaction_inclusion_v2` inherits all three panics because it delegates to `verify_transaction_inclusion` internally, and adds a fourth of its own — mismatched proof lengths:

**Panic 4 — proof length mismatch** (`contract/src/lib.rs`, lines 348–351):
```rust
require!(
    args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
    "Coinbase merkle proof and transaction merkle proof should have the same length"
);
``` [4](#0-3) 

The delegation from `verify_transaction_inclusion_v2` to `verify_transaction_inclusion` is at line 368:
```rust
self.verify_transaction_inclusion(args.into())
``` [5](#0-4) 

The most dangerous panic is Panic 1. The garbage-collection mechanism (`run_mainchain_gc`) actively removes old blocks from `mainchain_header_to_height` and `headers_pool`. A block hash that was valid and on the main chain at the time a 3rd-party contract was designed can become absent after GC runs, turning what was a valid call into a hard panic. [6](#0-5) 

---

### Impact Explanation

A 3rd-party NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` as part of a larger transaction (e.g., a bridge, a DeFi protocol gating on Bitcoin finality, or a cross-chain swap) will have its entire transaction reverted if any of the above panics fire. The contract author expects a `bool` return; instead they receive a propagated panic. This blocks all subsequent logic in the calling contract's transaction, including state changes, token releases, or callback chains that were expected to proceed regardless of the verification result.

---

### Likelihood Explanation

The trigger is realistic and unprivileged:

1. **GC-induced**: The contract's own `run_mainchain_gc` (called automatically inside `submit_blocks`) removes old blocks. A 3rd-party contract that stores a block hash and later calls `verify_transaction_inclusion` with it will panic once GC has removed that height. No attacker action is needed — normal relayer operation suffices. [7](#0-6) 

2. **Reorg-induced**: An adversary who can submit a chain reorganization (via `submit_blocks`, which is open to trusted relayers) can move a previously mainchain block onto a fork, making the block hash absent from `mainchain_header_to_height`, triggering Panic 1 in any 3rd-party contract that subsequently calls `verify_transaction_inclusion` with that hash. [8](#0-7) 

3. **Direct caller manipulation**: An unprivileged NEAR caller can directly supply an empty `merkle_proof` or mismatched proof lengths to trigger Panics 2 and 4 against any 3rd-party contract that passes caller-supplied arguments through to these functions.

---

### Recommendation

Replace all `env::panic_str` and `require!` calls inside `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` that represent "invalid but plausible" inputs with graceful `return false` paths. The functions are already typed to return `bool`; negative verification outcomes (block not on main chain, insufficient confirmations, empty proof) should be expressed as `false`, not as panics. Only truly unrecoverable internal invariant violations (e.g., a missing tip header that should always exist) warrant a panic.

---

### Proof of Concept

1. Alice deploys a NEAR contract `AliceBridge` that calls `verify_transaction_inclusion` and, if it returns `false`, proceeds with a refund path.
2. Alice's contract stores `tx_block_blockhash = H` (a valid mainchain block hash at the time of deployment).
3. The relayer submits new blocks; `run_mainchain_gc` removes block `H` from storage (it is now older than `gc_threshold` blocks).
4. Eve calls `AliceBridge` with a proof referencing `H`.
5. `AliceBridge` calls `verify_transaction_inclusion` with `tx_block_blockhash = H`.
6. Inside the contract, `mainchain_header_to_height.get(&H)` returns `None`; `unwrap_or_else(|| env::panic_str(...))` fires.
7. The panic propagates: `AliceBridge`'s entire transaction reverts. The refund path never executes. Alice's contract is permanently blocked for any proof referencing `H`.

No privileged access, leaked keys, or social engineering is required. Normal relayer operation (submitting blocks and triggering GC) is sufficient to create the condition.
