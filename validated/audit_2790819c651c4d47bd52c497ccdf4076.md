Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass a Curated Pool's Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but `MetricOmmPool.swap` supplies `msg.sender` (the direct caller of `pool.swap`) as `sender`. When `MetricOmmSimpleRouter` is used, `msg.sender` to the pool is the router contract, not the end user. If the pool admin allowlists the router to enable router-mediated swaps, every non-allowlisted user can bypass the per-user curation gate by routing through the router. There is no configuration that simultaneously permits router-mediated swaps for allowlisted users and blocks non-allowlisted users.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension is the pool): [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` to the pool: [3](#0-2) 

Consequently, `sender` delivered to the extension is always the router address, never the end user. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. The `extensionData` field is passed through but the extension never decodes it to recover the original caller. The router passes `""` as `callbackData` and forwards `params.extensionData` unchanged, providing no mechanism to embed the original `msg.sender`.

This creates an irreconcilable dilemma:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Legitimate allowlisted users cannot use the router at all |
| Router **allowlisted** | Every non-allowlisted user bypasses the per-user gate via the router |

Existing guards are insufficient: the only check in `beforeSwap` is `allowedSwapper[msg.sender][sender]`, and `sender` is structurally bound to the direct caller of `pool.swap`, which is the router.

## Impact Explanation

A curated pool using `SwapAllowlistExtension` is designed to restrict swaps to a specific set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled addresses). Once the pool admin allowlists the router — a necessary step for any user to swap through the standard periphery — the allowlist is completely ineffective. Any address can call `MetricOmmSimpleRouter.exactInputSingle()` and the extension will see `sender = router`, which is allowlisted, and pass the check. The non-allowlisted end user's address is never inspected. This is a direct, complete bypass of the curation invariant the pool was deployed to enforce, matching the "Admin-boundary break: pool admin exceeds caps, bypasses timelocks, or factory/oracle role checks are bypassed by an unprivileged path" allowed impact and the explicit Smart Audit Pivot: "allowlist checks must cover the exact actor/action intended and cannot be bypassed through router."

## Likelihood Explanation

The router is the standard, documented periphery entry point for swaps. Any pool admin who wants their allowlisted users to swap conveniently must allowlist the router. The bypass requires no special knowledge: any user simply calls the public router with the target pool address. No privileged access, no malicious setup, and no non-standard tokens are required. The attack is repeatable on every block.

## Recommendation

The extension must gate the economic actor (the end user), not the intermediary. Two concrete approaches:

1. **Pass the original initiator through `extensionData`**: Have `MetricOmmSimpleRouter` ABI-encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known router. This requires a convention between the router and the extension.

2. **Maintain a trusted-router registry in the extension**: When `sender` is a registered trusted router, require the end user's address to be encoded in `extensionData` and check `allowedSwapper[pool][decodedEndUser]` instead.

3. **Document incompatibility and add a revert guard**: If the design intent is that allowlisted pools must use direct `pool.swap()` calls only, document this explicitly and add a check in the router that reverts when the target pool has `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` — `msg.sender` to the pool is the router address.
6. Pool calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Attacker's swap executes against the curated pool despite never being allowlisted.

The attacker's address (`msg.sender` at the router level) is never inspected by the extension.
