### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claiming of Bonus Rewards - (File: x/tieredrewards/keeper/msg_server.go)

### Summary
`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. If an attacker submits the same position ID more than once in `msg.PositionIds`, each occurrence is loaded as a separate in-memory copy carrying identical stale checkpoint state (`LastEventSeq`, `LastBonusAccrual`). When `claimRewardsAndUpdatesPositions` iterates over the slice, the second copy's stale checkpoints cause the same validator events and the same current-segment bonus to be re-computed and re-paid, draining the `RewardsPoolName` module account.

### Finding Description

**The flaw**: when position ID `X` appears twice in `msg.PositionIds`, `positions[0]` and `positions[1]` are two independent in-memory copies both loaded with `LastEventSeq = N` and `LastBonusAccrual = T`. Processing `positions[0]` updates the store to `LastEventSeq = M`, `LastBonusAccrual = blockTime`, and pays the bonus. Processing `positions[1]` immediately after reads the same stale `LastEventSeq = N` and `LastBonusAccrual = T` from its own in-memory copy, re-computes the identical bonus, and pays it again. The final `setPosition` for `positions[1]` overwrites the store with the same terminal state, leaving no trace of the double payment.

There is no deduplication guard anywhere in the load loop or in `msg.Validate()`.
