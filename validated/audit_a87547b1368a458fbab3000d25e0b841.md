### Title
SwapAllowlistExtension Checks Router Identity Instead of Actual User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. Any pool admin who allowlists the router to enable router-mediated swaps for their curated users inadvertently opens the pool to every user who can call the router, completely defeating the allowlist.

### Finding Description

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to the extension:

```solidity
// ExtensionCalling.sol L160-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(params.recipient, ...)` directly:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

At this point `pool.msg.sender = router`, so the extension receives `sender = router` and checks `allowedSwapper[pool][router]`. The actual user who initiated the call is never examined.

**The trap for pool admins:** A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of users cannot simultaneously support router-mediated swaps without allowlisting the router address. The moment the admin calls `setAllowedToSwap(pool, router, true)` to let their allowlisted users trade via the router, every address on the network can bypass the allowlist by routing through `MetricOmmSimpleRouter`.

The same identity substitution occurs in `exactInput` for intermediate hops (where `msg.sender` is still the router) and in `exactOutput` / `_exactOutputIterateCallback` (where the pool is called from inside the callback, again with the router as `msg.sender`).

### Impact Explanation

Any user can bypass a pool's `SwapAllowlistExtension` by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) targeting the restricted pool, provided the router address is allowlisted. The pool's curation policy is silently voided: disallowed users can execute swaps, drain LP value at the pool's quoted prices, and extract tokens that the pool admin intended to restrict to a specific counterparty set. This is a direct loss of LP principal and a broken core pool functionality (the allowlist guard that LPs rely on for safety).

### Likelihood Explanation

The likelihood is high. The `SwapAllowlistExtension` is the production periphery contract for swap curation. Any pool admin who deploys it and also wants their allowlisted users to be able to use the standard router (the normal UX path) must allowlist the router. The documentation and audit targets explicitly call out the router as a supported entrypoint for allowlisted pools. There is no in-protocol mechanism to allowlist specific users through the router without also opening the pool to all router users.

### Recommendation

Pass the original user's identity through the call chain rather than the direct `msg.sender`. One approach: add a `swapper` field to the extension data that the router populates with `msg.sender` before forwarding, and have the extension verify it against a router-signed payload. A simpler approach: the pool should pass both `msg.sender` (the direct caller) and an optional `originator` (set by trusted periphery via a separate mechanism or transient storage) so the extension can gate on the true economic actor. At minimum, document that allowlisting the router is equivalent to `allowAllSwappers = true` and warn pool admins accordingly.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — intending only Alice to swap.
3. Pool admin also calls `setAllowedToSwap(pool, router, true)` — to let Alice use the router.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The extension receives `sender = router`, checks `allowedSwapper[pool][router] = true`, and passes.
7. Bob's swap executes successfully despite not being on the allowlist.
