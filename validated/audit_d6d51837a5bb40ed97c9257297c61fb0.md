### Title
Governance can retroactively extend `ExitDuration` on a tier with active positions, locking existing stakers' funds indefinitely — (`x/tieredrewards/keeper/tier.go`)

---

### Summary

`SetTier` in the tieredrewards keeper allows governance to overwrite all parameters of an existing tier — including `ExitDuration` — without checking whether the tier has active positions. Unlike `deleteTier`, which guards against modification when positions exist, `SetTier` has no such protection. A governance proposal that sets `ExitDuration` to an arbitrarily large value for a tier with active positions forces all subsequent `TriggerExitFromTier` calls to compute `ExitUnlockAt` far into the future, effectively locking stakers' funds indefinitely.

---

### Finding Description

`SetTier` writes tier parameters unconditionally after basic validation: [1](#0-0) 

The only validation performed is `tier.Validate()`, which checks that `ExitDuration > 0` but imposes **no upper bound**: [2](#0-1) 

In contrast, `deleteTier` explicitly guards against modification when active positions exist: [3](#0-2) 

When a user calls `TriggerExitFromTier`, the **current** tier state is fetched from the store and its `ExitDuration` is used to compute `ExitUnlockAt`: [4](#0-3) 

The `ExitDuration` is not snapshotted into the position at creation time — it is always read live from the tier store at the moment `TriggerExitFromTier` is executed. This means a governance-approved `SetTier` call that changes `ExitDuration` to `math.MaxInt64` (a valid value since only `> 0` is checked) will silently apply to all existing positions in that tier the next time their owners attempt to trigger an exit.

The asymmetry is stark: `deleteTier` protects existing positions from tier removal, but `SetTier` — the more dangerous operation for live positions — has no equivalent guard.

---

### Impact Explanation

All users who committed funds to a tier under specific exit terms (e.g., a 30-day `ExitDuration`) can have those terms changed retroactively by a governance proposal. If `ExitDuration` is set to `math.MaxInt64` nanoseconds (~292 years), every subsequent `TriggerExitFromTier` call for positions in that tier will set `ExitUnlockAt` to a time effectively in the infinite future. Users cannot withdraw their staked principal until `ExitUnlockAt` is reached, and `WithdrawFromTier` requires the position to have completed its exit sequence. The staked funds are not permanently destroyed, but they are rendered inaccessible for an arbitrarily long period — matching the Medium-severity framing of the reference report.

---

### Likelihood Explanation

The entry path is a standard Cosmos SDK governance proposal (`MsgUpdateTier` or equivalent) that calls `SetTier`. No leaked keys, social engineering, or off-chain coordination beyond a governance majority is required. The comment on `SetTier` itself — *"Used by governance messages, genesis, and chain upgrades"* — confirms this is a live, production-reachable code path. [5](#0-4) 

---

### Recommendation

Apply the same active-position guard used in `deleteTier` to `SetTier` when `ExitDuration` is being increased for an existing tier. Concretely:

1. In `SetTier`, if the tier already exists and `ExitDuration` is being changed, check `hasPositionsForTier` and reject the update (or restrict it to only decreasing `ExitDuration`).
2. Alternatively, snapshot `ExitDuration` into each `Position` at creation time so that the exit timeline is immutable per-position regardless of future governance changes to the tier.
3. Add an explicit upper-bound validation on `ExitDuration` in `Tier.Validate()` to prevent governance from setting astronomically large values.

---

### Proof of Concept

1. Governance creates Tier 1 with `ExitDuration = 30 days`. Users call `LockTier` or `CommitDelegationToTier` to open positions in Tier 1.
2. `PositionCountByTier[1] > 0` — active positions exist.
3. Governance submits and passes a proposal containing `MsgUpdateTier{Id: 1, ExitDuration: math.MaxInt64, ...}`.
4. `SetTier` is called; `Validate()` passes (only checks `> 0`); the tier is overwritten with no position-count check.
5. A user calls `TriggerExitFromTier` for their position. `getTier(ctx, pos.TierId)` returns the updated tier. `pos.TriggerExit(blockTime, math.MaxInt64)` sets `ExitUnlockAt ≈ year 292,277,026,596`.
6. The user's `WithdrawFromTier` call will never succeed within any realistic timeframe. Their staked principal is locked indefinitely.

```go
// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
	if err := tier.Validate(); err != nil {
		return err
	}
	if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil {
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tier.Id)
	}
	return nil
}
```

```go
func (k Keeper) deleteTier(ctx context.Context, tierId uint32) error {
	hasPositions, err := k.hasPositionsForTier(ctx, tierId)
	if err != nil {
		return err
	}
	if hasPositions {
		return types.ErrTierHasActivePositions
	}
	if err := k.Tiers.Remove(ctx, tierId); err != nil {
		if stderrors.Is(err, collections.ErrNotFound) {
			return errorsmod.Wrapf(types.ErrTierNotFound, "tier id %d", tierId)
		}
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tierId)
	}
	return nil
```
