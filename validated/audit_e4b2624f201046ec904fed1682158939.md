### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Proof Forgery Mitigation Enforced by `verify_transaction_inclusion_v2` - (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR function despite being deprecated. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` enforces to prevent the 64-byte transaction Merkle proof forgery attack. This is a direct structural analog to the reported pattern: a lower-level function is reachable without going through the higher-level wrapper that enforces the critical security check.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the [64-byte transaction Merkle proof forgery vulnerability](https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a valid coinbase Merkle proof, which anchors the proof to a real transaction position in the block: [1](#0-0) 

However, `verify_transaction_inclusion` (v1) is still decorated only with `#[pause]` — no role restriction, no deprecation enforcement at the protocol level — and remains a fully callable public NEAR function: [2](#0-1) 

The function's own docstring explicitly acknowledges the forgery risk it leaves open: [3](#0-2) 

The v1 function performs only a single Merkle root recomputation against the supplied `tx_id` and `merkle_proof`, with no check that `tx_id` corresponds to a leaf (real transaction) rather than an internal node: [4](#0-3) 

---

### Impact Explanation

An attacker can supply a crafted 64-byte value as `tx_id` that is actually the hash of an internal Merkle tree node. Because v1 does not validate via a coinbase proof, `verify_transaction_inclusion` returns `true` for a transaction that does not exist in the block. Any dApp or bridge contract that calls `verify_transaction_inclusion` to gate a payment, unlock, or state transition will be deceived into accepting a forged SPV proof. This corrupts the proof result — the core security guarantee of the light client.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` directly as a view or change call. The 64-byte forgery technique is well-documented and has known tooling. Any integrator who reads the contract ABI and calls v1 (whether intentionally or because they are unaware of the deprecation) is exposed. The risk is elevated because the function is still exported in the ABI alongside v2, making it indistinguishable to a naive consumer.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or add a hard `env::panic_str` at its entry point so it is unreachable at runtime. The `#[deprecated]` Rust attribute is a compile-time hint only; it has no effect on NEAR's runtime dispatch. `verify_transaction_inclusion_v2` should be the sole externally callable proof verification endpoint.

---

### Proof of Concept

1. Deploy the contract with a known chain state containing block `B` with Merkle root `R`.
2. Construct a 64-byte value `fake_tx_id` that is the SHA256d hash of the concatenation of two real sibling hashes at depth 1 of the Merkle tree (an internal node hash). This value is computable from public block data.
3. Call `verify_transaction_inclusion` directly (not v2) with:
   - `tx_id = fake_tx_id`
   - `tx_block_blockhash = B`
   - `tx_index` = the index corresponding to the internal node's position
   - `merkle_proof` = the remaining path to the root
4. The function computes `compute_root_from_merkle_proof(fake_tx_id, index, proof)` and matches `R`, returning `true`.
5. The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof check, but v1 is bypassed entirely. [5](#0-4) [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

            "Coinbase merkle proof and transaction merkle proof should have the same length"

            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
