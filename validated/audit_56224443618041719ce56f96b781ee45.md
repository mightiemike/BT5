### Title
`alreadyMatched` Reconstructed with Current Maker Price Instead of Actual Fill Prices Causes Taker Fee Miscalculation for Partially-Filled Orders — (`core/contracts/OffchainExchange.sol`)

---

### Summary

In `matchOrders`, the `alreadyMatched` value passed to `applyFee` is reconstructed by multiplying the **current** maker's price by the cumulative base amount already filled (`filledAmounts`). Because `filledAmounts` only tracks base units and previous fills may have executed at different prices, this reconstruction is incorrect whenever a taker order is partially filled across multiple `matchOrders` calls at different prices. The result is that the `minSize`-threshold accounting inside `applyFee` is wrong, causing the taker to pay either too little fee (protocol loss) or too much fee (user loss).

---

### Finding Description

`filledAmounts` is a mapping from order digest to cumulative **base** amount filled: [1](#0-0) 

After each match it is updated with the base delta of the current fill: [2](#0-1) 

When `applyFee` is called for the taker, `alreadyMatched` (in quote terms) is computed as: [3](#0-2) 

The formula `-maker.order.priceX18.mul(filledAmounts[ordersInfo.taker.digest])` converts the already-filled **base** amount to a **quote** amount using the **current** maker's price. This is only correct if every prior fill of this taker order also executed at exactly `maker.order.priceX18`. In practice, a taker order can be partially filled across many `matchOrders` calls, each potentially at a different maker price.

Inside `applyFee`, `alreadyMatched` drives the `minSize`-threshold logic: [4](#0-3) 

The percentage fee is only applied to the portion of the cumulative quote above `market.minSize`. If `alreadyMatched` is wrong, the boundary between the flat-fee region and the percentage-fee region shifts, producing an incorrect `feeApplied`.

**Concrete scenario — protocol loses fees (price falls between fills):**

| Step | Detail |
|---|---|
| `minSize` | 150 (quote units) |
| Fill 1 | 1 ETH at maker price $200 → `filledAmounts = 1`, actual quote matched = $200 |
| Fill 2 | 1 ETH at maker price $100 → `matchQuote = −100` |
| Computed `alreadyMatched` | `−100 × 1 = −100` (wrong; uses current price $100) |
| Correct `alreadyMatched` | `−200` (actual quote already paid) |
| `feeApplied` (wrong) | `abs(−100 + −100) − 150 = 50` → fee = `50 × rate` |
| `feeApplied` (correct) | `abs(−200 + −100) − 150 = 150`, capped at 100 → fee = `100 × rate` |

The taker pays **50 % less fee** on the second fill; the protocol absorbs the shortfall.

**Reverse scenario — user overpays (price rises between fills):**

| Step | Detail |
|---|---|
| `minSize` | 300 (quote units) |
| Fill 1 | 1 ETH at maker price $100 → `filledAmounts = 1`, actual quote matched = $100 |
| Fill 2 | 1 ETH at maker price $200 → `matchQuote = −200` |
| Computed `alreadyMatched` | `−200 × 1 = −200` (wrong; uses current price $200) |
| Correct `alreadyMatched` | `−100` (actual quote already paid) |
| `feeApplied` (wrong) | `abs(−200 + −200) − 300 = 100` → fee = `100 × rate` |
| `feeApplied` (correct) | `abs(−100 + −200) − 300 = 0` → fee = `0` |

The taker pays a **non-zero fee** when the correct fee is zero; the user loses funds.

---

### Impact Explanation

- **Protocol fee loss**: When market price falls between partial fills, `alreadyMatched` understates the already-consumed quote, shrinking the computed `feeApplied` below its correct value. The protocol collects less than the intended percentage fee.
- **User fund loss**: When market price rises between partial fills, `alreadyMatched` overstates the already-consumed quote, inflating `feeApplied` beyond its correct value. The taker pays more than the intended fee.

Both impacts are direct asset deltas: collected fees are a balance tracked in `market.collectedFees` and ultimately credited to the protocol; the taker's `quoteDelta` is reduced by the fee, directly affecting their on-chain balance. [5](#0-4) 

---

### Likelihood Explanation

Any taker order that is partially filled across two or more `matchOrders` calls at different execution prices triggers the bug. This is a routine occurrence in any active market with price movement. No special permissions, governance access, or sequencer compromise are required — the condition arises from normal order-book operation. The magnitude of the fee error scales with the price difference between fills and the size of `filledAmounts`, so it is larger in volatile markets or for large orders.

---

### Recommendation

Replace the single `filledAmounts` mapping (base units only) with a second mapping that tracks the cumulative **quote** amount already matched for each taker digest. Pass that stored quote total directly as `alreadyMatched` instead of reconstructing it from the current maker price:

```solidity
// new storage
mapping(bytes32 => int128) public filledQuoteAmounts;

// in matchOrders, before applyFee:
applyFee(
    callState.productId,
    ordersInfo.taker,
    market,
    filledQuoteAmounts[ordersInfo.taker.digest],   // actual quote already matched
    taker.order.appendix,
    true
);

// after applyFee, update both trackers:
filledAmounts[ordersInfo.taker.digest]      += ordersInfo.taker.amountDelta;
filledQuoteAmounts[ordersInfo.taker.digest] += ordersInfo.taker.quoteDelta; // pre-fee quote
```

This ensures `alreadyMatched` always reflects the true cumulative quote paid, regardless of price movement between fills.

---

### Proof of Concept

1. Deploy the protocol and create a market with `minSize = 150e18`.
2. User A submits a taker buy order for 2 ETH at limit price $300.
3. Sequencer matches Fill 1: 1 ETH against a maker at price $200. `filledAmounts[digest] = 1e18`. Fee on fill 1 is computed correctly (first fill, `alreadyMatched == 0`).
4. Market price moves to $100. Sequencer matches Fill 2: 1 ETH against a maker at price $100.
   - `alreadyMatched = −100e18 × 1e18 / 1e18 = −100e18` (wrong).
   - `feeApplied = abs(−100e18 + −100e18) − 150e18 = 50e18`.
   - Fee charged = `50e18 × takerRate`.
5. Correct `alreadyMatched` should be `−200e18` (actual quote paid in fill 1).
   - Correct `feeApplied = abs(−200e18 + −100e18) − 150e18 = 150e18`, capped at `100e18`.
   - Correct fee = `100e18 × takerRate`.
6. The protocol collects `50e18 × takerRate` instead of `100e18 × takerRate` — a 50 % shortfall on the second fill. [6](#0-5) [7](#0-6)
