### Title
Secp256k1/r1 Verify Reachable via 4-Byte Opcode Path Without `ENABLE_SECP_OPS` Flag Check — (`File: src/chia_dialect.rs`)

---

### Summary

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function dispatches secp signature-verification operators through two separate code paths. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the identical secp functions with **no flag check at all**. An attacker-controlled CLVM program can invoke secp verification unconditionally — including in mempool mode — by using the 4-byte encoding, bypassing the intended restriction entirely.

---

### Finding Description

`ChiaDialect::op()` in `src/chia_dialect.rs` handles operator dispatch in two branches:

**4-byte opcode branch (lines 157–183) — no flag guard:**

```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**1-byte opcode branch (lines 248–249) — correctly gated:**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as the gate for secp operations. The 1-byte aliases enforce it; the 4-byte path does not. `MEMPOOL_MODE` (lines 72–76) does not include `ENABLE_SECP_OPS`, so in mempool mode:

- Opcode `64` (1-byte) → falls to `unknown_operator` → `NO_UNKNOWN_OPS` set → **rejected**
- Opcode `0x13d61f00` (4-byte) → dispatched directly to `op_secp256k1_verify` → **executes**

The two encodings invoke the same underlying function but receive asymmetric access control.

---

### Impact Explanation

The concrete corrupted result is a **consensus/mempool divergence**: a CLVM program using the 4-byte secp opcode encoding is accepted and executed (returning `nil` on success) in contexts where the 1-byte encoding of the same operation is rejected. Any system that relies on `ENABLE_SECP_OPS` to gate secp operations — e.g., before a softfork activates, or in mempool validation — has that gate silently bypassed via the 4-byte path. This means:

1. Mempool validators running without `ENABLE_SECP_OPS` will accept programs that should be rejected.
2. If `ENABLE_SECP_OPS` is a softfork activation flag, pre-activation nodes will execute secp verification via 4-byte opcodes, diverging from nodes that correctly reject it.
3. The secp verification result (`nil` vs. `EvalErr::Secp256Failed`) directly affects coin spend validity on the Chia blockchain.

---

### Likelihood Explanation

The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source code comments. Any attacker reading the source can construct a CLVM program using these byte sequences as the operator atom. No special privileges, compromised nodes, or social engineering are required — only the ability to submit a CLVM program, which is the normal attacker surface for this library.

---

### Recommendation

Add an `ENABLE_SECP_OPS` flag check in the 4-byte opcode branch, mirroring the guard already present for the 1-byte aliases:

```rust
0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If the 4-byte opcodes are intended to be unconditionally available (i.e., hardforked in), then the 1-byte aliases should also be unconditionally available and the `ENABLE_SECP_OPS` guard on opcodes 64/65 should be removed. The two paths must be consistent.

---

### Proof of Concept

**Attacker-controlled CLVM bytes** (pseudocode):

```
; operator atom = 0x13d61f00 (4 bytes) — secp256k1_verify, no flag check
; args = (pubkey msg sig)
program = (0x13d61f00 pubkey msg sig)
```

Run with `ChiaDialect::new(MEMPOOL_MODE)` (which does NOT include `ENABLE_SECP_OPS`):

- The 4-byte branch at line 175–181 matches `0x13d61f00` and calls `op_secp256k1_verify` directly.
- No `ENABLE_SECP_OPS` check is performed.
- The function executes and returns `Reduction(1300000, nil)` on a valid signature — **accepted**.

Run the same operation with 1-byte opcode `64` under the same `MEMPOOL_MODE`:

- Line 248: `64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS)` — flag not set, falls through.
- Reaches `unknown_operator`; `NO_UNKNOWN_OPS` is set → `EvalErr::Unimplemented` — **rejected**.
