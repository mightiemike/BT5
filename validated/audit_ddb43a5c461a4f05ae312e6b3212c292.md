Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Swapper Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` becomes the router's address rather than the end user's address. A pool admin who allowlists the router to enable router-based UX for their allowlisted users inadvertently grants every address on the network the ability to bypass the per-user restriction.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` at the pool level: [4](#0-3) 

The real end user (`msg.sender` at the router level) is stored only in transient callback context for payment purposes and is never forwarded to the pool's `swap()` call as a parameter. The `extensionData` field passed to `pool.swap()` is taken directly from `params.extensionData`, which is fully user-controlled and cannot be trusted by the extension to identify the real swapper.

The result is a binary impossible choice for pool admins:
- **Allowlist individual users only**: those users cannot use the router (because `sender` = router, which is not allowlisted → `NotAllowedToSwap` revert).
- **Allowlist the router**: every address on the network can swap through the router, completely collapsing the per-user gate to a per-router gate.

No existing guard in the extension, pool, or router prevents this. The `allowAllSwappers` flag is a separate escape hatch and does not fix the actor-binding mismatch.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router address (a natural step to enable router-based UX for their allowlisted users) inadvertently opens the pool to all swappers. Any unprivileged address can call `router.exactInputSingle(...)` targeting the curated pool and the extension will pass because `allowedSwapper[pool][router]` is `true`. This constitutes a broken core pool invariant (the allowlist) and enables unauthorized traders to execute trades the pool was explicitly configured to block, resulting in direct loss of LP principal through arbitrage or unauthorized trading activity.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical production swap interface. A pool admin who wants allowlisted users to enjoy router UX (slippage protection, deadline checks) will naturally call `setAllowedToSwap(pool, router, true)`. The extension interface and documentation give no indication that doing so collapses the per-user gate. The misconfiguration is easy to make, hard to detect without auditing the actor-binding logic, and the exploit requires no special privileges — any EOA or contract can call `router.exactInputSingle`.

## Recommendation
The extension must check the economic actor (the end user), not the transport actor (the router). Two viable approaches:

1. **Trusted `swapper` field in pool's `swap` signature**: Add a `swapper` parameter to `IMetricOmmPoolActions.swap` that the router populates with `msg.sender`. The extension checks this field instead of `sender`. The pool enforces that only trusted callers (e.g., whitelisted routers) may set `swapper` to a value other than `msg.sender`.

2. **Router-signed `extensionData`**: The router encodes `msg.sender` into `extensionData` along with a signature or trusted-router attestation. The extension verifies the attestation before accepting the encoded swapper identity. This requires the pool to maintain a registry of trusted routers whose `extensionData` claims are accepted.

Until fixed, pool admins must be warned that allowlisting the router address in `SwapAllowlistExtension` is functionally equivalent to calling `setAllowAllSwappers(pool, true)`.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` to allowlist Alice.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` to let Alice use the router.
4. Bob (never allowlisted) calls `router.exactInputSingle({pool: curatedPool, recipient: bob, ...})`.
5. Router calls `pool.swap(bob, zeroForOne, amount, priceLimit, "", extensionData)` — `msg.sender` at pool = router.
6. Pool calls `_beforeSwap(router, bob, ...)` → extension receives `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. Bob's swap executes successfully in the curated pool despite never being allowlisted.

Foundry test outline:
```solidity
function test_allowlistBypass() public {
    // Setup: deploy pool with SwapAllowlistExtension, allowlist alice and router
    extension.setAllowedToSwap(address(pool), alice, true);
    extension.setAllowedToSwap(address(pool), address(router), true);

    // Bob (not allowlisted) swaps via router
    vm.prank(bob);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool), recipient: bob, ...
    }));
    // Assert: swap succeeded — bob bypassed the allowlist
}
```
