### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Accepts Internal Merkle Node Hashes as Valid Transaction Proofs — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function is still publicly callable by any unprivileged NEAR account. It performs no validation that the supplied `tx_id` is a leaf-level transaction hash, allowing an attacker to supply an internal Merkle tree node hash with a shortened proof path and receive a `true` return value for a transaction that was never included in the block. This is the direct analog to the "forced finalization with insufficient validation" class: just as `LightClient.force` bypasses the 2/3 signature threshold, `verify_transaction_inclusion` (v1) bypasses the coinbase-proof depth check introduced in v2, allowing a proof to be "forced" through with structurally insufficient evidence.

---

### Finding Description

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a live, public, unpermissioned entry point: [1](#0-0) 

Its only guard on the proof is: [2](#0-1) 

There is no check that `tx_id` is a leaf-level hash, and no check that `merkle_proof.len()` matches the actual tree depth. The underlying `compute_root_from_merkle_proof` in `merkle-tools` accepts any starting hash and any proof length: [3](#0-2) 

**Attack mechanics (Bitcoin Merkle second-preimage):**

For a block with transactions `[T0, T1, T2, T3]`:
- Level 1 internal node: `N01 = H(T0, T1)`
- Merkle root: `root = H(N01, H(T2, T3))`

An attacker calls `verify_transaction_inclusion` with:
- `tx_id = N01` (an internal node, not a real transaction)
- `tx_index = 0`
- `merkle_proof = [H(T2, T3)]` (length 1, not the full depth-2 proof)

`compute_root_from_merkle_proof(N01, 0, [H(T2,T3)])` computes `H(N01, H(T2,T3)) = root`, which matches `header.block_header.merkle_root`. The function returns `true`.

`verify_transaction_inclusion_v2` prevents this by requiring a coinbase proof of the same length as the tx proof, and verifying the coinbase proof reaches the actual merkle root — a shorter proof would compute an internal node, not the root, and fail: [4](#0-3) 

But v1 has no such guard and remains callable.

---

### Impact Explanation

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action (e.g., releasing bridged funds, confirming a Bitcoin deposit) can be deceived into accepting a fabricated transaction inclusion proof. The attacker does not need to mine any Bitcoin block — they only need to know the transaction list of any block already stored in the contract (publicly available on-chain). The false `true` result can be used to trigger unauthorized fund releases or other state changes in consuming contracts.

---

### Likelihood Explanation

- The function is callable by any unprivileged NEAR account with no role restriction and no deposit requirement (it is a `&self` view-compatible function).
- All inputs needed for the attack (block transaction hashes) are publicly available from Bitcoin's blockchain.
- No computational work is required; the attacker only needs to compute standard SHA-256d hashes.
- The attack is a well-documented class (Bitcoin Merkle second-preimage / CVE-2012-2459 family), meaning tooling and knowledge are widely available.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) entirely from the public interface. It is already deprecated and its replacement (`verify_transaction_inclusion_v2`) correctly mitigates the attack. Retaining the deprecated function as a live, unpermissioned entry point negates the security improvement introduced by v2. If removal is not immediately feasible, add an access-control guard (e.g., `#[trusted_relayer]` or a role check) to prevent unprivileged callers from invoking it.

---

### Proof of Concept

1. Identify any Bitcoin block stored in the contract that has ≥ 2 transactions. Its transaction hashes are publicly available.
2. Compute `N01 = double_sha256(T0 || T1)` (the level-1 internal node).
3. Compute the remaining proof siblings up to the root (e.g., `[H(T2,T3)]` for a 4-tx block).
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N01`
   - `tx_block_blockhash` = the stored block hash
   - `tx_index` = `0`
   - `merkle_proof` = `[H(T2,T3)]`
   - `confirmations` = `1`
5. The function returns `true` despite `N01` not being any real transaction.
6. A downstream contract consuming this result would treat the fabricated transaction as confirmed. [5](#0-4) [3](#0-2)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
