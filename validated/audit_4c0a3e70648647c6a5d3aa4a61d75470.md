Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass via Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` validates the `sender` argument forwarded by the pool, which is the pool's `msg.sender` at swap time. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to enable standard router-mediated swaps inadvertently opens the pool to every unpermissioned user, completely defeating the allowlist.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap` checks the wrong address:**

In `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol` L37:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` here is the pool; `sender` is the argument passed by the pool. The extension never inspects `extensionData` and has no way to recover the originating user.

**How `sender` is populated — `MetricOmmPool.swap` L230-231:**
```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via router
    ...
);
```
The pool passes its own `msg.sender` verbatim. When the call originates from `MetricOmmSimpleRouter`, this is the router's address.

**Router entry point — `MetricOmmSimpleRouter.exactInputSingle` L71-80:**
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., "", params.extensionData
);
```
The real `msg.sender` (the user) is stored only in transient storage for the payment callback. It is never encoded into `extensionData` or otherwise surfaced to the extension hook. The `extensionData` forwarded to the pool is entirely user-supplied and not authenticated.

**Full exploit call chain:**
```
attacker (not allowlisted)
  → router.exactInputSingle(pool=P, ...)
      → pool.swap(recipient, ...)          // msg.sender = router
          → _beforeSwap(sender=router, ...)
              → SwapAllowlistExtension.beforeSwap(sender=router)
                  → allowedSwapper[P][router] == true  ← passes
                      swap executes
```

The originating user address is never checked at any point in the hook path.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (KYC-verified addresses, whitelisted market makers, institutional participants) is completely unprotected once the router is allowlisted. Any address can execute swaps of arbitrary size against the pool's liquidity. Because the router is a shared, permissionless contract with no caller restrictions, a single allowlist entry for the router opens the gate to the entire internet. This constitutes broken core pool functionality and direct loss of LP principal at oracle-anchored prices without the pool admin's consent, meeting the contest's Critical/High direct-loss threshold.

## Likelihood Explanation

The scenario is highly likely in practice. Pool admins who want to support the standard periphery UX must allowlist the router — there is no other way to let legitimate users trade through it. The bypass requires no special privileges, no custom contract, and no unusual token behavior; any EOA can call `exactInputSingle` directly. The mismatch is invisible from the admin's perspective: `isAllowedToSwap(pool, router)` returns `true` as intended, but the invariant "only allowlisted users can swap" is silently broken.

## Recommendation

The extension must check the originating user, not the immediate pool caller. Options in increasing robustness:

1. **`extensionData` encoding (minimal fix):** Require that any allowlisted intermediary (router) encodes the real user address in `extensionData`. Modify `SwapAllowlistExtension.beforeSwap` to decode and check that address when `sender` is a known intermediary. This requires router cooperation and is fragile if other routers are added.

2. **Transient-storage `originalSender` (cleaner fix):** Add an `originalSender` slot to the router's transient context (analogous to how `MetricOmmSwapRouterBase` already stores the payer via `_setNextCallbackContext`). Expose it via a view function that the extension can call back into during `beforeSwap`. The extension would then check `IRouter(sender).originalSender()` when `sender` is a known router.

3. **Pool-level `originalSender` (most robust):** Have the pool expose a separate `originalSender` field that the router sets in transient storage before calling `swap`, and pass it as an additional argument to all extension hooks.

## Proof of Concept

```
Setup:
  pool P configured with SwapAllowlistExtension as beforeSwap hook
  pool admin calls: swapExtension.setAllowedToSwap(P, address(router), true)
    (to allow legitimate users to trade via the router)

Attack:
  attacker = address not in allowedSwapper[P]
  attacker calls:
    router.exactInputSingle({
      pool: P,
      recipient: attacker,
      zeroForOne: true,
      amountIn: X,
      extensionData: ""
    })

  Execution:
    router → pool.swap(attacker, true, X, ...)   // msg.sender = router
    pool   → _beforeSwap(sender=router, ...)
    ext    → allowedSwapper[P][router] == true  ← passes
    swap executes, attacker receives output tokens

Result:
  attacker swaps successfully despite never being individually allowlisted.
  The allowlist guard is completely bypassed for all router-mediated swaps.

Foundry test plan:
  1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
  2. Deploy MetricOmmSimpleRouter.
  3. Call swapExtension.setAllowedToSwap(pool, router, true).
  4. As an address NOT in allowedSwapper[pool], call router.exactInputSingle(...).
  5. Assert swap succeeds (no NotAllowedToSwap revert).
  6. Confirm the same address calling pool.swap() directly reverts with NotAllowedToSwap.
```
