### Title
Linked Signer Can Hijack Its Own Delegation — Unauthorized `linkedSigners` Mutation via `allowLinkedSigner = true` in Fast-Mode `LinkSigner` — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` handler in `EndpointTx.sol` passes `allowLinkedSigner = true` to `validateSignedTx`, permitting the **currently registered linked signer** (a session key) to sign a new `LinkSigner` transaction and replace itself with any arbitrary address. The slow-mode path for the same operation explicitly restricts authorization to the subaccount owner only. This inconsistency allows a compromised session key to permanently escalate its own authority by installing an attacker-controlled address as the new linked signer, without the subaccount owner's knowledge or consent.

---

### Finding Description

The `LinkSigner` transaction type is the mechanism by which a subaccount owner delegates signing authority to a session key. The `LinkSigner` struct carries a `sender` (the subaccount) and a `signer` (the address to be granted authority): [1](#0-0) 

In the fast-mode transaction processor (`processTransactionImpl`), the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` passes the currently registered linked signer address to `validateSignature`: [3](#0-2) 

`validateSignature` in `Verifier.sol` accepts a recovered ECDSA address that matches **either** the subaccount owner **or** the linked signer: [4](#0-3) 

This means the currently linked signer can sign a `LinkSigner` transaction naming any arbitrary `signer` bytes32 value, and the protocol will accept it and overwrite `linkedSigners[subaccount]` with the attacker-controlled address.

The slow-mode path for the same operation uses `validateSender`, which enforces that `msg.sender` must equal the first 20 bytes of `txn.sender` — i.e., only the subaccount owner can authorize it: [5](#0-4) 

This is a direct inconsistency: slow mode requires owner authorization for `LinkSigner`; fast mode does not.

---

### Impact Explanation

Once `linkedSigners[victim_subaccount]` is overwritten with an attacker-controlled address, the attacker can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` (when `sendTo == address(0)`) — drains collateral
- `TransferQuote` — transfers quote balance to any registered subaccount
- `LiquidateSubaccount` — initiates liquidation of the victim's positions
- `MintNlp` / `BurnNlp` — manipulates NLP positions [6](#0-5) [7](#0-6) 

The corrupted state variable is `linkedSigners[subaccount]` in `EndpointStorage`: [8](#0-7) 

Additionally, the attacker can use their newly installed linked signer to immediately sign another `LinkSigner` to re-install themselves if the victim attempts to revoke via slow mode, creating a persistent race condition through the sequencer's ordering control.

---

### Likelihood Explanation

Linked signers are session keys designed for browser-based or hot-wallet environments. They are materially more susceptible to compromise (XSS, phishing, local storage theft, clipboard hijacking) than the owner's hardware wallet or cold key. Any subaccount that has ever called `LinkSigner` to set a session key is exposed. The attack requires only a single signed transaction from the compromised session key — no special privileges, no admin access, no sequencer compromise.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-mode `LinkSigner` handler, consistent with the slow-mode path which already enforces owner-only authorization:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must NOT authorize its own replacement
);
``` [9](#0-8) 

This aligns fast-mode behavior with slow-mode behavior and ensures that only the subaccount owner (the address embedded in the first 20 bytes of `sender`) can modify the linked signer mapping.

---

### Proof of Concept

**Setup:**
- Victim subaccount: `bytes32(abi.encodePacked(victim_address, bytes12("default")))`
- Victim has previously called `LinkSigner` to set `linkedSigners[victim_subaccount] = session_key_address`
- Attacker has obtained `session_key_privkey` (e.g., via phishing)

**Attack:**
1. Attacker constructs a `LinkSigner` payload:
   ```
   sender = victim_subaccount
   signer = bytes32(uint256(uint160(attacker_address)))
   nonce  = current_nonce_for_victim_owner_address
   ```
2. Attacker signs the EIP-712 digest of this payload using `session_key_privkey`.
3. Attacker submits the signed transaction to the sequencer (or wraps it in a slow-mode submission).
4. `validateSignedTx(..., allowLinkedSigner = true)` recovers `session_key_address` from the signature, matches it against `getLinkedSigner(victim_subaccount)` → passes.
5. `linkedSigners[victim_subaccount]` is overwritten with `attacker_address`.
6. Attacker signs a `WithdrawCollateral` transaction using `attacker_privkey`, draining the victim's collateral. [2](#0-1) [10](#0-9)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
