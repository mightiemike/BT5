### Title
SwapAllowlistExtension Checks Router's Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. Because `sender` is `msg.sender` of the pool's `swap()` call, routing through `MetricOmmSimpleRouter` substitutes the router's address for the real user's address. If the pool admin allowlists the router to enable router-mediated swaps, every user — including those not on the allowlist — can bypass the gate. If the admin does not allowlist the router, every legitimately allowlisted user is silently blocked from using the router.

---

### Finding Description

**Allowlist check (SwapAllowlistExtension.sol line 37):**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct — enforced by `onlyPool`). `sender` is the first argument forwarded by the pool.

**Pool passes `msg.sender` as `sender` (MetricOmmPool.sol line 231):**

```solidity
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```

**Router calls `pool.swap()` directly (MetricOmmSimpleRouter.sol line 72-80):**

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

When a user calls `exactInputSingle`, the router becomes `msg.sender` to the pool. The pool passes `msg.sender` (= router) as `sender` to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. The extension then evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actual_user]`.

**Contrast with DepositAllowlistExtension (line 32-42):**

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

The deposit extension correctly checks `owner` (the position beneficiary), not `sender` (the caller). The swap extension has no equivalent — it only has `sender`, and that value is the router when routing is used.

---

### Impact Explanation

**Bypass path (high severity):** A pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of KYC'd or trusted counterparties. To allow those users to use the router (the primary UX path), the admin must allowlist the router address. Once the router is allowlisted, `allowedSwapper[pool][router] == true`, and the check passes for every caller regardless of who they are. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` → router calls `pool.swap()` → `sender = router` → allowlist passes → swap executes. The entire access control layer is nullified. LPs suffer swaps from counterparties the pool admin explicitly intended to exclude.

**False-rejection path (medium severity):** If the admin does not allowlist the router, every legitimately allowlisted user is blocked from using the router and must call the pool directly. This breaks the primary swap interface for all allowlisted users.

---

### Likelihood Explanation

The router is the standard user-facing entry point for swaps. Any pool that deploys `SwapAllowlistExtension` and expects users to route through `MetricOmmSimpleRouter` will encounter one of the two failure modes immediately. The bypass path is reachable by any unprivileged address with no special setup beyond calling the public router.

---

### Recommendation

Pass the original end-user identity through the swap path so the allowlist can check it. One approach: have the router encode the real `msg.sender` into `extensionData` and have the extension decode and verify it. A cleaner approach mirrors the deposit extension: introduce a `swapper` concept (analogous to `owner`) that the router explicitly sets and the pool forwards separately from `sender`. At minimum, document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and that allowlisting the router defeats the gate entirely.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, router, true)` — necessary for router users.
3. Non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. Router calls `pool.swap(recipient, ...)` → `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. `SwapAllowlistExtension.beforeSwap(sender=router, ...)` evaluates `allowedSwapper[pool][router] == true` → passes.
7. Swap executes for the non-allowlisted attacker. The allowlist is fully bypassed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
