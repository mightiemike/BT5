Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` inside `MetricOmmPool.swap`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router to enable standard periphery UX, every user — including those the allowlist was meant to exclude — can bypass the per-user gate entirely.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))`.

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly — the original `msg.sender` (the end user) is stored only in transient storage for the payment callback and is never forwarded to the pool as the swap initiator:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(
    params.recipient, params.zeroForOne, ..., params.extensionData
);
```

At the point `pool.swap` executes, `msg.sender` inside the pool is the **router contract address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same misbinding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Impact Explanation
A curated pool relying on `SwapAllowlistExtension` as its sole swap gate is fully bypassed once the pool admin allowlists the router. Any unprivileged EOA can call `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool and execute swaps. LP principal and fee revenue in the restricted pool are exposed to unauthorized traders — a broken core pool functionality / admin-boundary break with direct loss of LP assets.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. A pool admin deploying a curated pool who wants users to interact through the standard periphery will naturally allowlist the router. The bypass requires no special privilege, no flash loan, and no unusual token behavior — any EOA can exploit it immediately after the router is allowlisted.

## Recommendation
The extension must check the **economic actor** (the end user), not the intermediary. The cleanest fix mirrors the existing `_getPayer()` / `_getTokenToPay()` transient storage pattern already used in the router for callback settlement: the router stores the original `msg.sender` in transient storage before calling the pool, and the extension reads it via a trusted accessor. Alternatively, the pool can expose a trusted "original caller" slot populated by the router before each swap call, which the extension reads instead of the `sender` argument.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // Alice is KYC'd
  - Pool admin calls setAllowedToSwap(pool, router, true)  // Enable standard periphery

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({
         pool: restrictedPool,
         recipient: bob,
         zeroForOne: true,
         amountIn: X,
         ...
     })
  2. Router calls pool.swap(bob, true, X, ...) — msg.sender inside pool = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. Extension checks allowedSwapper[pool][router] → true (router is allowlisted)
  5. Swap executes; Bob receives tokens from the restricted pool

Result:
  - Bob bypassed the per-user allowlist entirely
  - allowedSwapper[pool][bob] was never consulted
  - LP funds in the restricted pool are exposed to unauthorized traders
