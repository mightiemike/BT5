### Title
`DISABLE_OP` Flag Check Is Unreachable Dead Code in `op_div`, `op_divmod`, `op_mod` — (`File: src/more_ops.rs`)

### Summary

The `DISABLE_OP` flag (`0x200`) is included in `MEMPOOL_MODE` and is intended to disable or restrict certain arithmetic operators in mempool (stricter) mode. It correctly and completely disables `op_modpow` (opcode 60) at the dispatch level in `src/chia_dialect.rs`. However, in `op_div`, `op_divmod`, and `op_mod`, the `DISABLE_OP` guard checks `a0_len > 2048`, which is unreachable dead code because an unconditional check at `a0_len > 256` always fires first. The flag has zero effect on these three operators, meaning they execute freely in mempool mode when they should be restricted or disabled.

### Impact Explanation

The `DISABLE_OP` flag is part of `MEMPOOL_MODE`, the stricter validation mode used when a node evaluates transactions before admitting them to the mempool. The flag's documented purpose is to disable or restrict expensive operators in that mode. It correctly disables `op_modpow` entirely. For `op_div`, `op_divmod`, and `op_mod`, the restriction is silently absent: the flag check is dead code and the operators execute without any additional mempool-mode restriction.

This creates an inconsistency in the enforcement of mempool policy: `op_modpow` is blocked, but the three division/modulo operators are not, even though all four share the same `DISABLE_OP` gating intent. An attacker-controlled CLVM program submitted to the mempool can freely invoke `op_div`, `op_divmod`, or `op_mod` with inputs up to the unconditional 256-byte limit, bypassing the intended mempool-mode restriction entirely. If the intent of `DISABLE_OP` was to completely disable these operators in mempool mode (mirroring `op_modpow`), then the mempool accepts programs it should reject, and those programs propagate to consensus nodes (which also accept them, since consensus mode lacks `DISABLE_OP`). The mempool protection is silently absent rather than enforced.

### Likelihood Explanation

The trigger is trivially attacker-controlled: any CLVM byte sequence that encodes a call to opcode 19 (`/`), 20 (`divmod`), or 61 (`mod`) with any valid arguments will bypass the dead `DISABLE_OP` check. No special privileges, configuration, or social engineering are required. The attacker only needs to submit a transaction to a node running in `MEMPOOL_MODE`.

### Recommendation

The `DISABLE_OP` checks in `op_div`, `op_divmod`, `op_mod` (and their Malachite variants) must be moved to execute **before** the unconditional size checks, or the operators must be completely disabled at the dispatch level (as `op_modpow` is), depending on the intended policy. The flag should also be documented with a clear description of which operators it disables and why.

For example, to mirror the `op_modpow` treatment, add a dispatch-level guard in `src/chia_dialect.rs`:

```rust
19 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_div
}
```

And remove the dead `DISABLE_OP` branches from inside `op_div`, `op_divmod`, `op_mod`, and their Malachite variants.

### Proof of Concept

```rust
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program;
use clvmr::allocator::Allocator;

// Encode: (/ (q . <256-byte atom>) (q . 1))
// In MEMPOOL_MODE, DISABLE_OP is set.
// op_modpow (opcode 60) would return Unimplemented.
// op_div (opcode 19) silently executes — DISABLE_OP check is dead code.
fn main() {
    let mut allocator = Allocator::new();
    let dialect = ChiaDialect::new(MEMPOOL_MODE);
    // craft a CLVM program calling opcode 19 (div) with valid args
    // ... program bytes for (/ (q . 0x<255 bytes>) (q . 1)) ...
    // run_program succeeds; DISABLE_OP had no effect.
}
```

The attacker submits any CLVM program invoking `/`, `divmod`, or `mod` to a mempool-mode node. The `DISABLE_OP` guard at `a0_len > 2048` is never reached because the unconditional `a0_len > 256` check would have to fire first — but inputs up to 256 bytes pass both checks, and inputs above 256 bytes are caught by the unconditional check before `DISABLE_OP` is ever evaluated. The flag is inert for all reachable inputs.
