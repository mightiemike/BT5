### Title
NLP Mint/Burn Executed at Sequencer-Supplied Price With No User-Controlled Bound — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`mintNlp()` and `burnNlp()` in `Clearinghouse.sol` compute the NLP tokens received (or quote returned) entirely from an `oraclePriceX18` value that is **supplied by the sequencer at execution time and is excluded from the user's EIP-712 signature**. The user signs only `{sender, quoteAmount, nonce}` (mint) or `{sender, nlpAmount, nonce}` (burn), with no price floor, price ceiling, minimum NLP out, or minimum quote out. The sequencer can therefore execute the signed intent at any price, and the user has no on-chain recourse.

---

### Finding Description

The `MintNlp` and `BurnNlp` structs that the user signs contain no price-related field:

```solidity
// IEndpoint.sol:112-116
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;   // user commits only to how much quote to spend
    uint64  nonce;
}

// IEndpoint.sol:125-129
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;     // user commits only to how many NLP to burn
    uint64  nonce;
}
```

The `SignedMintNlp` / `SignedBurnNlp` wrappers carry `oraclePriceX18` as an **unsigned, sequencer-appended field**:

```solidity
// IEndpoint.sol:118-136
struct SignedMintNlp {
    MintNlp tx;
    bytes   signature;
    int128  oraclePriceX18;        // NOT covered by user signature
    int128[] nlpPoolRebalanceX18;  // NOT covered by user signature
}

struct SignedBurnNlp {
    BurnNlp tx;
    bytes   signature;
    int128  oraclePriceX18;        // NOT covered by user signature
    int128[] nlpPoolRebalanceX18;
}
```

`Verifier.sol` confirms the digest covers only the inner `tx` fields:

```solidity
// Verifier.sol:378-385  (MintNlp)
digest = keccak256(abi.encode(
    keccak256(bytes(MINT_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.quoteAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));

// Verifier.sol:391-398  (BurnNlp)
digest = keccak256(abi.encode(
    keccak256(bytes(BURN_NLP_SIGNATURE)),
    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce          // oraclePriceX18 absent
));
```

`Clearinghouse.mintNlp()` then derives the NLP amount purely from the sequencer-supplied price:

```solidity
// Clearinghouse.sol:466
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
```

`Clearinghouse.burnNlp()` derives the quote returned the same way:

```solidity
// Clearinghouse.sol:502-504
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee     = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount        = MathHelper.max(0, quoteAmount - burnFee);
```

There is no check anywhere that the resulting `nlpAmount` (mint) or `quoteAmount` (burn) meets any user-specified minimum.

---

### Impact Explanation

**`mintNlp` (addLiquidity analog):** A user signs a commitment to spend `quoteAmount` of quote. If the sequencer supplies an inflated `oraclePriceX18`, the computed `nlpAmount = quoteAmount / oraclePriceX18` is smaller. The user spends the full `quoteAmount` but receives fewer NLP tokens than the price at signing implied. The quote balance delta is exact and irreversible; the NLP shortfall is a direct asset loss relative to the user's expectation.

**`burnNlp` (removeLiquidity analog):** A user signs a commitment to burn `nlpAmount` of NLP. If the sequencer supplies a deflated `oraclePriceX18`, the computed `quoteAmount = nlpAmount * oraclePriceX18 - burnFee` is smaller. The user burns the full `nlpAmount` but receives less quote than the price at signing implied. The NLP is permanently destroyed; the quote shortfall is a direct asset loss.

In both cases the corrupted state delta is the user's on-chain balance in `SpotEngine`: `updateBalance(NLP_PRODUCT_ID, txn.sender, nlpAmount)` (mint) and `updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount)` (burn).

---

### Likelihood Explanation

The sequencer is the only entity that can submit `MintNlp`/`BurnNlp` transactions to the endpoint. Even without malicious intent, the oracle price used at execution time can differ materially from the price at the moment the user signed — the user's signed message carries no expiry on the acceptable price range. A malicious or compromised sequencer can exploit this deterministically. A non-malicious sequencer still exposes users to unbounded price drift between signing and execution with no on-chain safety net. The signed transaction is valid for any `oraclePriceX18` the sequencer chooses to attach.

---

### Recommendation

1. Add a `minNlpAmount` field to `MintNlp` and include it in the EIP-712 digest. In `Clearinghouse.mintNlp()`, assert `nlpAmount >= txn.minNlpAmount`.
2. Add a `minQuoteAmount` field to `BurnNlp` and include it in the EIP-712 digest. In `Clearinghouse.burnNlp()`, assert `quoteAmount >= txn.minQuoteAmount`.
3. Alternatively, add a `maxOraclePriceX18` to `MintNlp` and `minOraclePriceX18` to `BurnNlp`, both covered by the user's signature, and enforce them before computing amounts.

---

### Proof of Concept

1. User signs `MintNlp{sender=alice, quoteAmount=10_000e18, nonce=1}` when the NLP oracle price is `1.00e18` (expecting ~10 000 NLP).
2. Sequencer batches the transaction but submits it with `oraclePriceX18 = 2.00e18`.
3. `Clearinghouse.mintNlp()` computes `nlpAmount = 10_000e18 / 2.00e18 = 5_000`.
4. Alice's quote balance is debited by the full `10_000e18`; her NLP balance is credited only `5_000` — half the expected amount.
5. No on-chain check rejects this execution because `oraclePriceX18` is not part of Alice's signature and no minimum-NLP guard exists.

Symmetric scenario for `burnNlp`: user signs `BurnNlp{sender=alice, nlpAmount=5_000, nonce=2}` at price `1.00e18` (expecting ~5 000 quote minus fee). Sequencer submits with `oraclePriceX18 = 0.50e18`; Alice receives only ~2 500 quote while burning the full 5 000 NLP.
