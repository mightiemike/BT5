Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router to support router-mediated swaps for curated users, every unprivileged user can bypass the per-pool swap allowlist by routing through the public router.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

**Step 2 — Extension checks `sender`, which is the router address when called via router.**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's caller (the router): [2](#0-1) 

**Step 3 — Router always calls the pool as itself.**

In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly. The pool's `msg.sender` is therefore the router address, not the end user: [3](#0-2) 

The same pattern holds for `exactInput` (all hops): [4](#0-3) 

And for `exactOutputSingle` and `exactOutput` recursive callback hops: [5](#0-4) 

**Step 4 — Irreconcilable configuration trap.**

A pool admin who wants to restrict swaps to specific users (e.g., KYC'd addresses A, B, C) faces two bad options:

- **Do not allowlist the router**: allowlisted users A, B, C cannot use `MetricOmmSimpleRouter` at all.
- **Allowlist the router**: the extension checks `allowedSwapper[pool][router] = true` for every swap routed through the router, so any unprivileged user D can bypass the allowlist by calling `router.exactInputSingle(...)`.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same, because the extension cannot distinguish between different end users behind the router.

## Impact Explanation

Any user can bypass a curated pool's swap allowlist by routing through the public `MetricOmmSimpleRouter`. For pools designed to restrict trading to specific counterparties (e.g., to prevent adverse selection against LPs, or to enforce regulatory access controls), this allows unauthorized traders to swap at oracle-derived prices. The allowlist guard — the sole mechanism protecting curated pools — silently fails open for all router-mediated swaps once the router is allowlisted. This constitutes broken core pool functionality causing potential loss of funds for LPs in curated pools.

## Likelihood Explanation

The likelihood is medium-high. `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who configures a `SwapAllowlistExtension` and also wants their allowlisted users to be able to use the router will naturally allowlist the router, triggering the bypass. The attacker requires no special privileges — only the ability to call a public router function.

## Recommendation

The `SwapAllowlistExtension` must gate the economically relevant actor, not the immediate pool caller. Two viable approaches:

1. **Forward the original user in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling the pool; the extension decodes and checks it. This requires a trusted router convention.
2. **Dedicated router-aware extension**: Implement a variant of `SwapAllowlistExtension` that accepts a signed user identity in `extensionData` and verifies it against the allowlist, making the check router-agnostic.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin allowlists address A (legitimate user)
  - Admin allowlists MetricOmmSimpleRouter (to support router-mediated swaps for A)

Attack:
  - Address B (not allowlisted) calls:
      router.exactInputSingle(ExactInputSingleParams({
          pool: curated_pool,
          tokenIn: token0,
          tokenOut: token1,
          zeroForOne: true,
          amountIn: X,
          amountOutMinimum: 0,
          recipient: address(B),
          deadline: block.timestamp,
          priceLimitX64: 0,
          extensionData: ""
      }))
  - Pool receives swap call with msg.sender = router
  - _beforeSwap passes sender = router to SwapAllowlistExtension
  - Extension checks allowedSwapper[pool][router] = true → passes
  - B successfully swaps on a pool they are not authorized to access
```
