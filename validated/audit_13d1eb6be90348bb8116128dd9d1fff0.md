### Title
Unsanitized `tx_id` Input in Deprecated `verify_transaction_inclusion` Enables Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function accepts a user-controlled `tx_id` that is never validated as a real leaf-level transaction hash. Any unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` with a crafted sibling path, causing the function to return `true` for a fabricated transaction inclusion proof. The function remains publicly callable despite its deprecation.

---

### Finding Description

`verify_transaction_inclusion` accepts `ProofArgs` containing `tx_id: H256` (fully attacker-controlled). Its sole verification step is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

There is no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. Bitcoin's internal nodes are `SHA256d(left_child || right_child)`. An attacker selects any internal node hash `N` at depth `d`, provides the `d`-step sibling path to the root, and the computed root equals the block's stored `merkle_root`, so the function returns `true`.

The contract's own documentation acknowledges this exact flaw:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

`verify_transaction_inclusion_v2` was introduced to close this gap via mandatory coinbase proof validation: [3](#0-2) 

However, the v1 function carries only a `#[pause]` attribute — no role restriction, no `#[private]` — so any NEAR account can call it directly: [4](#0-3) 

The `compute_root_from_merkle_proof` implementation in `merkle-tools` performs no leaf-vs-internal-node discrimination; it blindly hashes whatever `transaction_hash` it receives: [5](#0-4) 

---

### Impact Explanation

Any consumer contract or off-chain system that calls `verify_transaction_inclusion` to gate an action (e.g., releasing bridged assets, confirming a Bitcoin payment) can be deceived into accepting a forged proof. The attacker falsely proves that an arbitrary value `N` — which is never a real Bitcoin transaction — was included in a confirmed, mainchain block. The corrupted proof result is the direct output of a public contract entrypoint, not a theoretical edge case.

---

### Likelihood Explanation

The attack requires only:
1. Public Bitcoin blockchain data (the Merkle tree of any block already tracked by the contract).
2. The ability to call a public NEAR function — no role, no stake, no privileged key.

The 64-byte Merkle forgery technique is well-documented (see [BitMEX research](https://www.bitmex.com/blog/64-Byte-Transactions)), has known tooling, and is the exact reason `verify_transaction_inclusion_v2` was introduced. Likelihood is **medium-high** wherever any consumer still calls the v1 function.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract entirely, or gate it with `#[private]` so it cannot be called externally. All callers must migrate to `verify_transaction_inclusion_v2`. The deprecation annotation alone provides no runtime enforcement.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block whose hash is present in `mainchain_header_to_height` (e.g., block at height `H`).
2. Obtain the block's full Merkle tree from a Bitcoin node. Select any internal node `N` at depth `d` (where `d < tree_height`). Record its position index `idx` and the `(tree_height - d)` sibling hashes forming the path to the root.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash` = the block's hash (confirmed mainchain block)
   - `tx_index = idx`
   - `merkle_proof` = the sibling path from `N` to the root
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, idx, siblings)` reproduces the block's `merkle_root`; the function returns `true`.
5. The consumer contract accepts the proof, falsely believing transaction `N` was included in the Bitcoin block. [6](#0-5) [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
