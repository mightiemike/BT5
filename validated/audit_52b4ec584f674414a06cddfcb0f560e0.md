### Title
Permanent Unburnable NFT After Transfer: `BurnNFT` Requires Both Ownership AND Denom Creator â€” (`x/nft/keeper/keeper.go`)

### Summary

`Keeper.BurnNFT` enforces two independent AND-conditions: the caller must be both the current NFT owner (`IsOwner`) **and** the denom creator (`IsDenomCreator`). Once the denom creator transfers an NFT to any third party, neither the new owner nor the original creator can ever satisfy both conditions simultaneously. The NFT becomes permanently unburnable, and the denom's supply counter never decreases, corrupting the supply invariant.

### Finding Description

`Keeper.BurnNFT` at lines 141â€“161 of `x/nft/keeper/keeper.go` applies both checks sequentially with no OR-branch:

```go
nft, err := k.IsOwner(ctx, denomID, tokenID, owner)   // must be current owner
if err != nil { return err }

_, err = k.IsDenomCreator(ctx, denomID, owner)         // must also be denom creator
if err != nil { return err }
``` [1](#0-0) 

`IsDenomCreator` compares the caller against the immutable `denom.Creator` field stored at denom-issuance time and never updated: [2](#0-1) 

`TransferOwner` updates only the NFT's owner record; it never changes `denom.Creator`: [3](#0-2) 

After `A` (denom creator) transfers the NFT to `B`:

| Caller | `IsOwner` | `IsDenomCreator` | Result |
|--------|-----------|-----------------|--------|
| B (new owner) | âś… passes | âťŚ fails | `ErrUnauthorized` |
| A (denom creator) | âťŚ fails | âś… passes | `ErrUnauthorized` |

The only bypass, `BurnNFTUnverified`, skips the creator check but is **not reachable** from the user-facing `MsgBurnNFT` path â€” it is called exclusively from the IBC transfer module: [4](#0-3) 

The `msgServer.BurnNFT` handler unconditionally delegates to `k.Keeper.BurnNFT` (the broken path): [5](#0-4) 

### Impact Explanation

- The NFT is permanently locked in the new owner's account with no on-chain mechanism to destroy it.
- `k.decreaseSupply` is never reached, so the denom's supply counter remains inflated indefinitely.
- Any off-chain or on-chain logic that relies on supply accuracy (e.g., collection statistics, IBC escrow accounting) is corrupted.
- This is exploitable by any user who receives an NFT via `MsgTransferNFT` â€” no special privileges required.

### Likelihood Explanation

NFT transfers are a core, everyday operation. Any transfer of an NFT away from the denom creator triggers this condition. The attacker does not need to do anything beyond receiving a transferred NFT and observing that neither they nor the original creator can burn it.

### Recommendation

Remove the `IsDenomCreator` check from `BurnNFT`. Burning should require only ownership. The denom creator restriction is appropriate for minting and editing, but not for destruction of a token the caller legitimately owns. The existing `BurnNFTUnverified` already demonstrates the correct ownership-only pattern.

```go
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
    if !k.HasDenomID(ctx, denomID) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
    }
    nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
    if err != nil {
        return err
    }
    // Remove IsDenomCreator check â€” ownership is sufficient to burn
    k.deleteNFT(ctx, denomID, nft)
    k.deleteOwner(ctx, denomID, tokenID, owner)
    k.decreaseSupply(ctx, denomID)
    return nil
}
```

func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
