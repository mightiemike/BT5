### Title
Global-Contract Identifier Storage Not Subtracted in `action_delete_account` Deletion-Size Check, Permanently Locking Account Balance — (`File: runtime/runtime/src/actions.rs`)

### Summary

`action_delete_account` computes the account's non-contract storage usage to decide whether the account is small enough to delete. For accounts using a global contract (`AccountContract::Global` or `AccountContract::GlobalByAccount`), the legacy code path subtracts only local contract code bytes (which is zero for global contracts) and never subtracts the global-contract identifier overhead (32 bytes for a hash-based global contract, or `account_id.len()` bytes for an account-based one). The inflated storage figure is then compared against `MAX_ACCOUNT_DELETION_STORAGE_USAGE` (10 000 bytes). Any account whose `storage_usage` falls in the range `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_size]` is incorrectly rejected with `DeleteAccountWithLargeState`, permanently trapping the account's balance.

### Finding Description

In `action_delete_account`, the legacy branch (pre-`FixDeleteAccountGlobalContractStorageUsage`) computes:

```rust
// Legacy behavior: only subtracts local contract code, misses the
// global contract identifier overhead.
let account_storage_usage = account_ref.storage_usage();
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),
)?;
account_storage_usage.saturating_sub(code_len)
``` [1](#0-0) 

For a global contract, `local_contract_hash()` returns `None`, so `code_len` is 0 and the full `storage_usage` — which already includes the identifier overhead — is used unchanged. The identifier overhead is:

```rust
pub fn identifier_storage_usage(&self) -> u64 {
    match self {
        AccountContract::None | AccountContract::Local(_) => 0u64,
        AccountContract::Global(_) => 32u64,
        AccountContract::GlobalByAccount(id) => id.len() as u64,
    }
}
``` [2](#0-1) 

The corrected path (enabled by `FixDeleteAccountGlobalContractStorageUsage`) calls `get_contract_storage_usage`, which dispatches to `identifier_storage_usage()` for global contracts and subtracts the correct amount:

```rust
let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
account_ref.storage_usage().saturating_sub(contract_storage)
``` [3](#0-2) 

The exact divergent value is `identifier_storage_usage()` bytes — 32 for `AccountContract::Global(CryptoHash)` or `account_id.len()` for `AccountContract::GlobalByAccount(AccountId)`. The protocol feature that gates the fix is assigned to protocol version 85:

```rust
FixDeleteAccountGlobalContractStorageUsage,
``` [4](#0-3) 

Global contracts were introduced at protocol version 77 (`_DeprecatedGlobalContracts`), so the window of exposure is protocol versions 77–84. [5](#0-4) 

### Impact Explanation

When the deletion check fires incorrectly, `action_delete_account` returns `DeleteAccountWithLargeState` and the account is not removed. The account's `amount` balance is never forwarded to the `beneficiary_id` via `Receipt::new_balance_refund`. Because the account cannot be deleted, those tokens are permanently inaccessible to the owner — the exact "permanently locked funds" pattern from the external report. The account owner cannot reduce `storage_usage` below the threshold by any other means once a global contract is attached, because global contracts do not store code locally and the identifier bytes are fixed. [6](#0-5) [7](#0-6) 

### Likelihood Explanation

Any unprivileged user who:
1. Creates an account,
2. Attaches a global contract (`AccountContract::Global` or `AccountContract::GlobalByAccount`), and
3. Attempts to delete the account while `storage_usage` is in the range `(10 000, 10 000 + identifier_size]`

will trigger the bug on any node running protocol version 77–84. The condition is reachable without any privileged role. The 32-byte window for hash-based global contracts is narrow but deterministic and can be engineered by controlling the amount of contract data stored on the account.

### Recommendation

The fix is already present behind `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` (protocol version 85). All nodes should upgrade to a binary that includes this feature. No additional code change is needed beyond ensuring the feature is activated network-wide. [3](#0-2) 

### Proof of Concept

The existing unit test `test_delete_account_global_contract_protocol_transition` demonstrates the exact divergence:

```rust
let storage = Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32;
let enabled = ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage.protocol_version();

// Before the fix: the identifier is not subtracted, so `MAX + 32 > MAX`.
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    storage,
    enabled - 1,   // protocol version 84
);
expect_delete_account_too_large(&before);  // incorrectly blocked

// From the fix onwards: the 32-byte identifier is subtracted → MAX + 32 - 32 == MAX.
let after = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    storage,
    enabled,       // protocol version 85
);
assert!(after.result.is_ok());  // correctly allowed
``` [8](#0-7) 

An attacker or ordinary user can reproduce this on-chain by deploying a global contract to an account, ensuring `storage_usage` is exactly `MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32`, and submitting a `DeleteAccountAction`. On protocol versions 77–84 the deletion will be rejected and the balance will be permanently locked.
