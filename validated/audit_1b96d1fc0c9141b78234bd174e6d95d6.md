### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash in P2P Block Sync — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter uses a zero-value heuristic to reconstruct the `ValidResourceBounds` variant. When a V3 transaction carrying `AllResources` bounds with `l1_data_gas = 0` and `l2_gas = 0` is round-tripped through protobuf, the deserializer silently produces `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` includes the `l1_data_gas` field in the Poseidon preimage only for the `AllResources` variant, the hash recomputed from the deserialized transaction differs from the hash the sequencer originally committed. This is the direct sequencer analog of the external report's "distributeEx does not check the token address" pattern: a conversion boundary fails to preserve a type discriminant, causing a downstream computation to silently take the wrong branch.

---

### Finding Description

**Serialization boundary — `From<ValidResourceBounds> for protobuf::ResourceBounds`** [1](#0-0) 

Both `ValidResourceBounds::L1Gas(l1_gas)` and `ValidResourceBounds::AllResources` with `l1_data_gas = 0, l2_gas = 0` serialize to identical protobuf bytes: `l1_gas = X`, `l2_gas = 0`, `l1_data_gas = 0`. The variant discriminant is not encoded.

**Deserialization boundary — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** [2](#0-1) 

The deserializer reconstructs the variant purely from the zero-value heuristic at line 431: if `l1_data_gas.is_zero() && l2_gas.is_zero()` → `L1Gas`, otherwise `AllResources`. A transaction that was originally `AllResources` with both fields zero is therefore silently downgraded to `L1Gas` after any protobuf round-trip.

**Hash domain — `get_tip_resource_bounds_hash`** [3](#0-2) 

The function branches on the variant:
- `L1Gas` → Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed]` (2 resource elements)
- `AllResources` → Poseidon over `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (3 resource elements)

Even when `l1_data_gas = 0`, the packed encoding `get_concat_resource(&ResourceBounds::default(), L1_DATA_GAS)` is a non-zero felt (the resource-name bits are non-zero), so the two hashes are always distinct.

**Hash computation path for internal transactions** [4](#0-3) 

`InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` calls the per-type hasher, which calls `get_tip_resource_bounds_hash` with the stored `ValidResourceBounds`. If the variant was downgraded during deserialization, the recomputed hash diverges from the hash the proposer committed.

**Contrast with the RPC/mempool path (not affected)** [5](#0-4) 

The RPC-transaction protobuf converter uses `TryFrom<protobuf::ResourceBounds> for AllResourceBounds`, which always produces `AllResources` and requires `l1_data_gas` to be present. Only the block-transaction converter (`transaction.rs`) uses the lossy `ValidResourceBounds` path.

---

### Impact Explanation

Any V3 transaction submitted with `AllResourceBounds { l1_data_gas: 0, l2_gas: 0 }` (a valid, user-submittable transaction) receives hash **H₁** at the gateway. After the block is propagated via P2P and the transaction is deserialized as `InvokeTransactionV3` / `DeclareTransactionV3` / `DeployAccountTransactionV3` on a syncing or validating node, the `ValidResourceBounds` variant is `L1Gas`, and any hash recomputation yields **H₂ ≠ H₁**.

Consequences:
- **Wrong block hash**: if the syncing node recomputes transaction hashes to verify the block hash, it derives a different block hash than the proposer committed, causing block rejection or storage of an incorrect block hash.
- **Wrong RPC output**: `starknet_getTransactionByHash(H₁)` returns "not found"; `starknet_getTransactionByHash(H₂)` returns the transaction with a hash that does not match the on-chain commitment — an authoritative-looking wrong value.
- **`validate_transaction_hash` failure**: the function recomputes the hash and compares it to the expected value; it returns `false` for any such transaction, causing valid blocks to be rejected by syncing nodes. [6](#0-5) 

---

### Likelihood Explanation

The trigger condition — a V3 transaction with `AllResourceBounds` where `l1_data_gas = 0` and `l2_gas = 0` — is reachable by any unprivileged user submitting a standard V3 transaction that does not use L2 gas or L1 data gas. The `RpcTransaction` type always carries `AllResourceBounds` (never `L1Gas`), so any such submission from the public RPC gateway produces an `AllResources` internal transaction. The protobuf round-trip is exercised on every block propagated via P2P.

---

### Recommendation

1. **Encode the variant explicitly**: add a boolean or enum field to the protobuf `ResourceBounds` message (e.g., `bool is_all_resources`) so the deserializer can reconstruct the correct variant without relying on zero-value inference.
2. **Remove the heuristic**: the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter should not infer the variant from field values. If backward compatibility with 0.13.2 peers is required, use the explicit field with a default of `false` (= `L1Gas`) for old messages.
3. **Add a round-trip test**: assert that `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` survives a protobuf round-trip with the same variant and the same transaction hash.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
