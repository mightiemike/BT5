### Title
Storage-stake check in `verify_and_charge_tx_ephemeral` ignores in-flight pending balance, allowing chunk inclusion of transactions that fail at execution time — (File: `runtime/runtime/src/verifier.rs`)

---

### Summary

In `verify_and_charge_tx_ephemeral`, the balance-availability check correctly deducts `pending.paid_from_balance` (the sum of costs committed by in-flight, uncertified transactions) from the account balance. However, the immediately following `check_storage_stake` call is passed `new_amount = account.amount() - total_cost`, which does **not** subtract the pending committed balance. This inconsistency allows a transaction to pass chunk-production validation and be included in a chunk (burning gas for the conversion), yet fail at execution time with `LackBalanceForState`, wasting the signer's gas fees — the exact nearcore analog of the DCA "in-flight balance not deducted" pattern.

---

### Finding Description

`verify_and_charge_tx_ephemeral` (`runtime/runtime/src/verifier.rs`) contains two sequential balance checks that use different accounting bases:

**Step 1 — balance check (correct, accounts for pending):**
```rust

if available_balance < total_cost {
    return TxVerdict::Failed(InvalidTxError::NotEnoughBalance { … });
}
``` [1](#0-0) 

**Step 2 — `new_amount` computation (ignores pending):**
```rust
// line 320
let new_amount = account.amount().checked_sub(total_cost).unwrap();
``` [2](#0-1) 

**Step 3 — storage-stake check (uses the wrong base):**
```rust
// line 332
match check_storage_stake(account, new_amount, config) {
``` [3](#0-2) 

`check_storage_stake` computes `available_amount = account_balance + account.locked()` and compares it to the storage requirement: [4](#0-3) 

Because `new_amount = account.amount() - total_cost` (not `account.amount() - pending.paid_from_balance - total_cost`), the storage-stake check is evaluated against a balance that is **too high** by exactly `pending.paid_from_balance`. The same defect exists in `verify_and_charge_gas_key_tx_ephemeral` at the analogous `check_storage_stake(account, new_account_amount, config)` call: [5](#0-4) 

The code comment at line 307 explicitly acknowledges that on the consensus path `pending` is always `default()` (zero), so the subtraction is exact, and that on the RPC/chunk-production path it is "best-effort." However, the comment covers only the balance check; the storage-stake check was not updated to match: [6](#0-5) 

On the consensus path (`apply_transactions` in `runtime/runtime/src/lib.rs`), `PendingConstraints::default()` is always passed, so `paid_from_balance = 0` and there is no divergence: [7](#0-6) 

The divergence is live only on the chunk-production path (`prepare_transactions` in `chain/chain/src/runtime/mod.rs`), where the `PendingTxSession` supplies non-zero `paid_from_balance` derived from the `ShardedPendingTransactionQueue`: [8](#0-7) 

The `PendingConstraints` struct and its `paid_from_balance` field are defined in `runtime/runtime/src/lib.rs`: [9](#0-8) 

The `PendingTxSession::check_pending` correctly propagates `paid_from_balance` from the queue into the returned `PendingConstraints`: [10](#0-9)

### Impact Explanation

When an account has:
- a large storage footprint (high `S = storage_amount_per_byte × storage_usage`), and
- one or more uncertified in-flight transactions that have already committed `C₁` NEAR from its balance,

a new transaction costing `C₂` can satisfy:

| Check | Formula | Result |
|---|---|---|
| Balance check (correct) | `B − C₁ ≥ C₂` | pass |
| Storage-stake check (wrong base) | `(B − C₂) + L ≥ S` | pass |
| Storage-stake check (correct base) | `(B − C₁ − C₂) + L ≥ S` | **fail** |

The chunk producer includes the transaction and burns gas for the conversion to a receipt. When the transaction later executes on the consensus path (after the in-flight transactions have been applied, reducing the on-chain balance by `C₁`), the runtime's storage-stake check uses the real balance `B − C₁ − C₂`, which is below `S`, and the transaction fails with `LackBalanceForState`. The gas burnt for conversion is not refunded. The signer's fees are wasted — the direct nearcore analog of the DCA "in-flight balance not deducted" pattern.

---

### Likelihood Explanation

The condition is reachable by any unprivileged account holder on a node running with `protocol_feature_spice` (the Spice protocol, which activates the `ShardedPendingTransactionQueue`). The account must have non-trivial storage usage (e.g., a contract with many keys or large state) and must have at least one uncertified transaction pending. Both conditions are routine for active contract accounts. No privileged role is required; the attacker is the signer themselves (they lose their own fees, or a third party can craft the scenario to drain a victim's gas budget).

---

### Recommendation

Separate the balance used for the storage-stake check from the balance written into the ephemeral state:

```rust
// Correct: use pending-adjusted balance for the storage-stake check.
let storage_check_amount = available_balance   // already = account.amount() - pending.paid_from_balance
    .checked_sub(total_cost)
    .unwrap();

// Keep new_amount for the ephemeral state update (starts from certified state, not pending-adjusted).
let new_amount = account.amount().checked_sub(total_cost).unwrap();

match check_storage_stake(account, storage_check_amount, config) { … }
```

Apply the same fix to the `check_storage_stake` call in `verify_and_charge_gas_key_tx_ephemeral`.

### Proof of Concept

**Setup:**
- Account balance `B = 10 NEAR`, storage requirement `S = 8 NEAR`, locked `L = 0`.
- One uncertified pending transaction with `paid_from_balance = C₁ = 3 NEAR`.
- New transaction with `total_cost = C₂ = 1 NEAR`.

**Chunk-production path (buggy):**
1. `available_balance = 10 − 3 = 7 NEAR ≥ 1` → balance check passes.
2. `new_amount = 10 − 1 = 9 NEAR`.
3. `check_storage_stake(account, 9, config)`: `9 + 0 = 9 ≥ 8` → passes.
4. Transaction included in chunk; gas burnt for conversion.

**Execution path (consensus, correct):**
1. Pending tx executes first; on-chain balance becomes `10 − 3 = 7 NEAR`.
2. New tx executes; balance becomes `7 − 1 = 6 NEAR`.
3. `check_storage_stake(account, 6, config)`: `6 + 0 = 6 < 8` → **`LackBalanceForState`**.
4. Transaction fails; gas fees already burnt and unrecoverable.

pub fn check_storage_stake(
    account: &Account,
    account_balance: Balance,
    runtime_config: &RuntimeConfig,
) -> Result<(), StorageStakingError> {
    let billable_storage_bytes = account.storage_usage();
    let required_amount = runtime_config
        .storage_amount_per_byte()
        .checked_mul(u128::from(billable_storage_bytes))
        .ok_or_else(|| {
            format!(
                "Account's billable storage usage {} overflows multiplication",
                billable_storage_bytes
            )
        })
        .map_err(StorageStakingError::StorageError)?;
    let available_amount = account_balance
        .checked_add(account.locked())
        .ok_or_else(|| {
            format!(
                "Account's amount {} and locked {} overflow addition",
                account.amount(),
                account.locked(),
            )
        })
        .map_err(StorageStakingError::StorageError)?;
    if available_amount >= required_amount {
        Ok(())
    } else {
        if is_zero_balance_account(account) {
            return Ok(());
        }
        Err(StorageStakingError::LackBalanceForStorageStaking(
            required_amount.checked_sub(available_amount).unwrap(),
        ))
    }
}

pub struct PendingConstraints {
    /// Total balance already committed by this account's pending access key
    /// transactions (total_cost) plus pending gas key deposit costs.
    pub paid_from_balance: Balance,
    /// Total gas key cost already committed by pending gas key transactions
    /// signed with this key, plus any pending WithdrawFromGasKey amounts
    /// targeting this key.
    pub paid_from_gas_key: Balance,
    /// Maximum nonce seen among pending transactions for this (account, key,
    /// nonce_index) combination.
    pub max_nonce: Nonce,
}

impl Default for PendingConstraints {
    fn default() -> Self {
        Self { paid_from_balance: Balance::ZERO, paid_from_gas_key: Balance::ZERO, max_nonce: 0 }
    }
}
