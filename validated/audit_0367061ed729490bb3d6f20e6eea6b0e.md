### Title
Linked Signer Can Self-Replace via `LinkSigner`, Permanently Perpetuating Unauthorized Subaccount Access — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in `processTransactionImpl` is validated with `allowLinkedSigner = true`, meaning the currently active linked signer is permitted to sign a `LinkSigner` transaction and replace itself with any attacker-controlled address. This mirrors the external report's root cause exactly: the "key to the key" is stored alongside the protected data, making the protection illusory. Here, the delegated key (linked signer) can mutate the delegation itself, making revocation by the owner unreliable and allowing a compromised linked signer to permanently persist unauthorized access.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is processed as follows:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

The `validateSignedTx(..., true)` call routes through `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer to `Verifier.validateSignature`:

```solidity
// Verifier.sol lines 291–304
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

The `linkedSigners` mapping is the on-chain state being mutated: [3](#0-2) 

This means the currently linked signer — a hot wallet explicitly designed to be a lower-security delegated key — can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address it chooses, including a fresh attacker-controlled address. The delegation mechanism can modify itself.

The analog to the external report is direct: the passphrase/salt (the "key to the key") is stored alongside the encrypted private key, making the encryption illusory. Here, the linked signer (the "key to the subaccount") is permitted to modify the linked signer mapping (the "key storage"), making revocation illusory.

---

### Impact Explanation

A compromised linked signer can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to a fresh attacker-controlled address.
2. The `linkedSigners[subaccount]` mapping is overwritten to the attacker's new address.
3. The attacker now controls a new linked signer and can immediately use it to sign `WithdrawCollateral` or `WithdrawCollateralV2` (with `sendTo == address(0)`) transactions, draining all collateral from the subaccount.
4. If the legitimate owner attempts to revoke by submitting their own `LinkSigner` transaction through the sequencer, the attacker can race to re-link again, since both the owner and the current linked signer are valid signers for `LinkSigner`. The attacker controls the sequencer submission timing advantage.
5. The owner's only reliable recourse is the slow-mode path (3-day delay), during which the attacker retains full withdrawal authority.

**Corrupted state**: `linkedSigners[subaccount]` is set to an attacker-controlled address, granting persistent signing authority over all `allowLinkedSigner = true` operations including withdrawals, liquidations, transfers, and further `LinkSigner` mutations.

**Asset impact**: Full collateral balance of the subaccount is at risk of theft via `WithdrawCollateral` signed by the attacker's new linked signer.

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets — keys held in online, automated systems (e.g., trading bots, API keys). Hot wallets carry materially higher compromise risk than cold-wallet subaccount owners. The protocol's own design acknowledges this by restricting `WithdrawCollateralV2` with a non-zero `sendTo` to owner-only signatures: [4](#0-3) 

This shows the protocol already recognizes that linked signers should not have unrestricted authority for the most sensitive operations — yet `LinkSigner` itself, which controls the entire delegation chain, is left open to the linked signer. Any hot wallet compromise (leaked API key, server breach, phishing) directly enables this attack.

---

### Recommendation

- **Short term**: Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. The `LinkSigner` operation modifies the delegation authority itself and must only be authorized by the subaccount owner's primary key, not by the currently delegated key.

```solidity
// EndpointTx.sol — change allowLinkedSigner to false for LinkSigner
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // owner-only: linked signer must not modify its own delegation
);
```

- **Long term**: Establish a clear privilege hierarchy: operations that mutate signing authority (`LinkSigner`) must always require the root owner key. Delegated keys (linked signers) should never be able to modify the delegation that grants them authority — analogous to the principle that encryption keys must not be stored alongside the data they protect.

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...000` and has linked Bob (`0xBob`) as her hot-wallet signer via a prior `LinkSigner` transaction. `linkedSigners[0xAlice000...000] == 0xBob`.

2. An attacker compromises Bob's private key.

3. The attacker crafts a `LinkSigner` transaction:
   ```
   sender = 0xAlice000...000
   signer = 0xMallory  // attacker-controlled address
   nonce  = current nonce for Alice's address
   ```
   Signs it with Bob's key and submits to the sequencer.

4. `processTransactionImpl` reaches the `LinkSigner` branch. `validateSignedTx(..., true)` calls `validateSignature` with `linkedSigner = 0xBob`. The recovered address is `0xBob == linkedSigner`, so the check passes. [5](#0-4) 

5. `linkedSigners[0xAlice000...000]` is overwritten to `0xMallory`. [6](#0-5) 

6. The attacker immediately submits a `WithdrawCollateral` transaction signed by `0xMallory`, draining Alice's collateral. `validateSignature` accepts `0xMallory` as the linked signer. [7](#0-6) 

7. Alice notices and submits a `LinkSigner` transaction (owner-signed) to revoke. The attacker races to submit another `LinkSigner` (Mallory-signed) to re-link to `0xMallory2`. Since both are valid signers for `LinkSigner`, the outcome depends on sequencer ordering. Alice's only guaranteed recourse is the 3-day slow-mode path, during which the attacker retains full withdrawal authority.
