### Title
`setWithdrawPool` Allows Unrestricted Production Replacement Enabling Fast-Withdrawal Replay Drain — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` allows the owner to replace the live `withdrawPool` address with a fresh contract at any time, with no restriction on whether the pool has already been set. The new pool starts with empty `markedIdxs` and `minIdx = 0`, erasing the deduplication state that prevents fast-withdrawal replay. Any previously submitted fast withdrawal — whose transaction bytes and BLS signatures are permanently visible on-chain — can be replayed against the new pool by an unprivileged caller, double-spending tokens that accumulate in the new pool from subsequent slow withdrawals.

---

### Finding Description

`Clearinghouse.setWithdrawPool` is an unrestricted setter:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [1](#0-0) 

The `withdrawPool` address is used in `handleWithdrawTransfer`, which routes every withdrawal through the pool:

```solidity
function handleWithdrawTransfer(IERC20Base token, address to, uint128 amount, uint64 idx) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [2](#0-1) 

`BaseWithdrawPool` maintains two critical deduplication state variables:

- `markedIdxs` — a mapping that records every `idx` already processed (both slow and fast withdrawals)
- `minIdx` — the floor below which fast withdrawals are rejected [3](#0-2) 

`submitFastWithdrawal` enforces deduplication using both:

```solidity
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
markedIdxs[idx] = true;
``` [4](#0-3) 

A freshly deployed `WithdrawPool` has `markedIdxs` entirely empty and `minIdx = 0`. When `setWithdrawPool` points `Clearinghouse` at a new pool, all previously processed fast-withdrawal indices are unknown to the new pool. Because the BLS signatures submitted in prior `submitFastWithdrawal` calls are permanently recorded on-chain, any caller can replay them verbatim against the new pool.

`WithdrawPool` is initialized with a fixed `verifier` address:

```solidity
function _initialize(address _clearinghouse, address _verifier) internal initializer {
    __Ownable_init();
    clearinghouse = _clearinghouse;
    verifier = _verifier;
}
``` [5](#0-4) 

The verifier validates the same signatures that were already accepted on the old pool. Replaying them against the new pool passes signature verification because the signatures are cryptographically valid — they were never invalidated.

---

### Impact Explanation

After `setWithdrawPool(newPool)`:

1. Future slow withdrawals cause `Clearinghouse.handleWithdrawTransfer` to transfer tokens into `newPool`.
2. An attacker replays a previously submitted fast withdrawal by calling `newPool.submitFastWithdrawal(oldIdx, oldTx, oldSigs)`.
3. `newPool.markedIdxs[oldIdx]` is `false` and `oldIdx > 0 == newPool.minIdx`, so both guards pass.
4. The verifier accepts the original valid signatures.
5. Tokens are transferred to the original recipient a second time, draining `newPool`'s balance.

The corrupted state delta is: `newPool`'s token balance is drained by the amount of replayed withdrawals. Every fast withdrawal ever submitted against the old pool is replayable, bounded only by the token balance that accumulates in the new pool.

---

### Likelihood Explanation

The owner must call `setWithdrawPool` (e.g., to upgrade the pool contract). This is a plausible operational action. Once it occurs, the exploit requires no special privilege — any caller with access to historical on-chain calldata (publicly available) can replay old fast withdrawals. The attack window opens immediately after the pool switch and persists until the new pool is drained or the pool is switched again.

---

### Recommendation

Apply the same fix class as the NFTX report: make `withdrawPool` settable only once (immutable after first assignment):

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(withdrawPool == address(0), "withdrawPool is immutable");
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
```

If pool replacement in production is required, implement atomic state migration: copy `markedIdxs` for all live indices and `minIdx` from the old pool to the new pool before switching the pointer, or use an upgradeable proxy for the pool so state is preserved across implementation changes.

---

### Proof of Concept

1. Protocol is live. `Clearinghouse.withdrawPool` points to `OldPool`. Multiple fast withdrawals have been processed: `submitFastWithdrawal(idx=500, tx, sigs)` was called and `OldPool.markedIdxs[500] = true`, `OldPool.minIdx = 500`.

2. Owner calls `ContractOwner.setWithdrawPool(NewPool)` → `Clearinghouse.setWithdrawPool(NewPool)`. [6](#0-5) 

3. `NewPool` is freshly initialized: `markedIdxs` all `false`, `minIdx = 0`.

4. Users make slow withdrawals; `Clearinghouse.handleWithdrawTransfer` sends tokens to `NewPool`, accumulating balance. [7](#0-6) 

5. Attacker reads `idx=500`, `tx`, `sigs` from the original `submitFastWithdrawal` call on `OldPool` (public on-chain calldata).

6. Attacker calls `NewPool.submitFastWithdrawal(500, tx, sigs)`:
   - `!markedIdxs[500]` → `true` ✓
   - `500 > minIdx(0)` → `true` ✓
   - `verifier.requireValidTxSignatures(tx, 500, sigs)` → passes (signatures are valid) ✓
   - Tokens transferred to original recipient again. [8](#0-7) 

7. Every previously submitted fast withdrawal can be replayed in the same manner, draining `NewPool`'s entire token balance.

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
