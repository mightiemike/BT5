### Title
Accrued Bonus Rewards Permanently Lost When Redelegation Slash Fires With Insufficient Bonus Pool - (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is silently swallowed. However, `processEventsAndClaimBonus` has already advanced the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) in-memory before returning the error. The caller then persists those advanced checkpoints via `setPositionWithState`. The accrued bonus for the entire period up to the slash is permanently erased — the pool retains the tokens but the position's claim to them is irreversibly destroyed.

### Finding Description

`processEventsAndClaimBonus` updates the position's checkpoints **before** checking pool sufficiency: [1](#0-0) 

Specifically, `pos.UpdateLastEventSeq(entry.Seq)` (line 193), `applyBonusAccrualCheckpoint` (line 215), and `pos.UpdateLastKnownBonded` (line 217) all mutate the `pos` pointer before `sufficientBonusPoolBalance` is checked at line 230. When the pool check fails, the function returns `nil, err` — but the in-memory `pos` already has its checkpoints advanced past the accrual period. [2](#0-1) 

In `slashRedelegationPosition`, the `ErrInsufficientBonusPool` error is explicitly caught and swallowed to avoid a chain halt: [3](#0-2) 

After swallowing the error, execution continues and `setPositionWithState` is called with the already-mutated `pos`, persisting the advanced checkpoints to the store: [4](#0-3) 

Additionally, `decrementEventRefCount` is called inside the event-processing loop (line 196) via the real `ctx` — these store mutations are not rolled back when the error is returned, so event reference counts are decremented even though no bonus was paid. [5](#0-4) 

### Impact Explanation

The accrued bonus rewards for the period from `pos.LastBonusAccrual` up to the slash timestamp are permanently destroyed. The `RewardsPoolName` module account retains the tokens, but the position's checkpoints are advanced so the position can never re-compute or re-claim that bonus segment. The pool balance is not reduced, but the user's entitlement to those tokens is irreversibly erased. This is a permanent, unrecoverable loss of user funds. [6](#0-5) 

### Likelihood Explanation

The trigger requires two concurrent conditions: (1) a tier position has an active redelegation entry, and (2) the source validator is slashed while the bonus pool is insufficient. Condition (1) is a normal user action (`MsgTierRedelegate`). Condition (2) is a realistic operational scenario — the pool can be depleted by the `BeginBlocker` top-up mechanism or simply never replenished. The `BeforeRedelegationSlashed` hook fires automatically as part of the Cosmos SDK staking module's slash processing; no privileged action is required. [7](#0-6) 

### Recommendation

Wrap the `processEventsAndClaimBonus` call in `slashRedelegationPosition` with a cache context so that checkpoint mutations and event ref-count decrements are rolled back when the pool is insufficient. The position should be saved with its **original** (pre-call) checkpoints when the bonus cannot be paid, preserving the user's future claim:

```go
cacheCtx, commit := sdk.UnwrapSDKContext(ctx).CacheContext()
if _, err := k.processEventsAndClaimBonus(cacheCtx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // Do NOT commit — checkpoints and ref-counts are rolled back
        // pos retains its original checkpoints
    } else {
        return err
    }
} else {
    commit() // only commit if bonus was successfully paid
}
```

### Proof of Concept

1. User calls `MsgLockTier` to create a tier position on validator V1.
2. User calls `MsgTierRedelegate` to move the position to validator V2. This creates a redelegation entry and a `RedelegationMappings` entry.
3. The bonus pool is empty (or becomes empty via `BeginBlocker` top-up).
4. Validator V1 is slashed during the redelegation period. The Cosmos SDK staking module calls `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
5. Inside `processEventsAndClaimBonus`: the event loop runs, `pos.UpdateLastEventSeq` and `decrementEventRefCount` execute, then `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` advance the checkpoints. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` swallows the error and calls `setPositionWithState(ctx, pos, nil)` with the advanced checkpoints.
7. The user later calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts from the already-advanced `LastBonusAccrual` and `LastEventSeq` — the pre-slash accrual period is gone. The user receives zero bonus for that period. [8](#0-7)
