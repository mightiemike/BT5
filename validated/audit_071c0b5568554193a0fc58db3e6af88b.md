### Title
`verify_transaction_inclusion` Returns `bool` on Proof Failure Instead of Reverting, Enabling Silent Acceptance of Invalid Merkle Proofs — (`contract/src/lib.rs`)

### Summary
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return `false` when a supplied Merkle proof does not match the stored block Merkle root, rather than reverting the NEAR transaction. Any downstream contract that gates a state-changing action on the result of these calls without explicitly checking the returned boolean will silently proceed as if the proof were valid. An attacker can exploit this to trigger unauthorized state changes (e.g., minting bridged tokens, releasing escrowed funds) by supplying a fabricated transaction proof.

### Finding Description
`verify_transaction_inclusion` computes the Merkle root from the caller-supplied proof and compares it to the block's stored `merkle_root`. If they differ, the function returns `false` — it does not panic or revert: [1](#0-0) 

```rust
// compute merkle tree root and check if it matches block's original merkle tree root
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root   // returns bool, never panics on mismatch
```

`verify_transaction_inclusion_v2` partially mitigates the 64-byte forgery attack by first checking the coinbase proof with a hard `require!` that reverts on failure, then delegates to `verify_transaction_inclusion` for the actual transaction proof — which again returns `bool` on failure: [2](#0-1) 

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"   // panics here on coinbase failure
);

self.verify_transaction_inclusion(args.into())  // returns false, does NOT panic on tx failure
```

The inconsistency is the root cause: the coinbase check panics on failure, creating a reasonable expectation that the entire call reverts on any proof failure. A downstream contract author who observes the coinbase check panicking may assume the transaction check also panics — but it does not. The function silently returns `false`, and any caller that does not explicitly handle this return value proceeds as if the proof were valid.

This is the direct analog to the yAxis H-04 finding: just as the yAxis controller completed a withdrawal loop without reverting when liquidity was zero (burning shares while delivering nothing), the BTC light client completes proof verification without reverting when the Merkle proof is invalid — delivering a `false` result that callers may ignore.

### Impact Explanation
A downstream NEAR contract (bridge, DEX, escrow) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and does not explicitly check the returned `bool` will accept an invalid BTC transaction proof as valid. The attacker can:

1. Construct a valid coinbase proof for a real block (to pass the `require!` in v2).
2. Supply a fabricated transaction proof for a non-existent or unrelated transaction.
3. The function returns `false` instead of reverting.
4. The downstream contract proceeds with the state-changing action (e.g., minting tokens, releasing funds).

The corrupted proof result is the exact analog to the yAxis issue: a verification step that should block an invalid operation instead silently returns a failure indicator, allowing the caller to proceed with a harmful state change.

### Likelihood Explanation
Moderate. The inconsistency between the coinbase check (which panics) and the transaction check (which returns `bool`) creates a natural trap for downstream contract authors. A developer who tests only the coinbase failure path — which panics — may incorrectly assume the transaction failure path also panics, and omit the return value check. The attack requires no privileged access: any unprivileged NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with a fabricated proof. [3](#0-2) 

### Recommendation
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` should revert (via `require!`) when the computed Merkle root does not match the stored block Merkle root, instead of returning `false`:

```rust
// Instead of:
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root

// Use:
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root,
    "Invalid Merkle proof: computed root does not match block merkle root"
);
```

This eliminates the inconsistency between the coinbase check and the transaction check, and prevents downstream contracts from silently accepting invalid proofs.

### Proof of Concept
1. Attacker identifies a downstream NEAR contract that calls `verify_transaction_inclusion_v2` and uses the result to gate a state-changing action (e.g., minting bridged tokens) without checking the returned `bool`.
2. Attacker selects a real block already tracked by the light client and constructs a valid coinbase Merkle proof for it (to pass the `require!` coinbase check in v2).
3. Attacker supplies a fabricated `tx_id` and `merkle_proof` for a non-existent transaction in the same block.
4. The downstream contract calls `verify_transaction_inclusion_v2` with the fabricated arguments.
5. The coinbase `require!` passes; `verify_transaction_inclusion` returns `false` for the fabricated transaction proof without reverting.
6. The downstream contract ignores the `false` return value and executes the state-changing action.
7. The attacker receives minted tokens or released funds without a valid underlying BTC transaction. [4](#0-3) [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
