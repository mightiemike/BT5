Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the real user, allowing any unprivileged caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router address. If the pool admin allowlists the router (required for any router-mediated swap to succeed), every on-chain user can bypass the individual allowlist by calling through the router, completely defeating the access-control intent.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool. In `MetricOmmPool.swap`, the pool always passes its own `msg.sender` as `sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    ...
```

`ExtensionCalling._beforeSwap` faithfully forwards this value as the first argument to the extension. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

So `pool.swap()`'s `msg.sender` is the router, and the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The inescapable dilemma:**
- If the pool admin does **not** allowlist the router: individually allowlisted users cannot use the router at all.
- If the pool admin **does** allowlist the router (the only way to enable router-mediated swaps): every user on-chain can bypass the individual allowlist by routing through the router.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Impact Explanation
Any user not on the allowlist can bypass the `SwapAllowlistExtension` guard entirely by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router swap function) against a pool that has the router allowlisted. The pool admin's intent to restrict swaps to specific addresses is completely defeated. For pools designed to be KYC-gated, market-maker-only, or restricted-LP pools, this allows unauthorized parties to execute unrestricted swaps, draining LP value through trades the pool was explicitly configured to block. This is broken core pool functionality causing potential loss of LP funds — a direct match to the allowed impact gate.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard periphery entry point for swaps. Any pool admin who wants allowlisted users to be able to use the router must allowlist the router address. Once the router is allowlisted, the bypass is trivially reachable by any unprivileged user with no special setup, no privileged role, and no capital beyond the swap amount. The router is a public, permissionless contract. The bypass is repeatable on every block.

## Recommendation
The extension must check the **original transaction sender**, not the direct caller of `pool.swap()`. The simplest safe fix: the router should encode `msg.sender` into `extensionData`, and the extension should decode and verify it. Alternatively, pool admins must be explicitly documented (via NatSpec) that the router must never be allowlisted and allowlisted users must call the pool directly — but this is a usability constraint, not a code fix. The preferred code-level fix is for the router to forward the real user identity via `extensionData` and for `SwapAllowlistExtension.beforeSwap` to decode and check it when `sender` is a known router.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` to allowlist Alice.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` — `msg.sender` of `pool.swap()` is the router.
6. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender=router, ...)`.
7. `ExtensionCalling._beforeSwap` encodes `sender=router` and calls the extension.
8. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
9. Bob successfully swaps in a pool he was never authorized to access.
