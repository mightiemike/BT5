### Title
Deprecated `verify_transaction_inclusion` Lacks Leaf-Node Validation, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary
The `verify_transaction_inclusion` function (deprecated since v0.5.0) remains callable by any unprivileged NEAR account and performs no validation that the supplied `tx_id` is a genuine leaf node (transaction hash) rather than an internal Merkle tree node. An attacker can supply an internal node with a crafted proof and cause the function to return `true` for a transaction that was never included in the block. This is the direct analog of `ssh.InsecureIgnoreHostKey()`: both are production-callable functions that explicitly skip the one check that establishes authenticity of the remote data.

---

### Finding Description

`verify_transaction_inclusion` delegates proof verification entirely to `compute_root_from_merkle_proof`, which treats the caller-supplied `tx_id` as an opaque 32-byte value and hashes it upward through the tree without any check that it is a leaf:

```rust
// contract/src/lib.rs:318-322
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply iterates over the proof elements and hashes pairs — it has no concept of depth, tree size, or leaf vs. internal node:

```rust
// merkle-tools/src/lib.rs:39-51
let mut current_hash = transaction_hash;
let mut current_position = transaction_position;
for proof_hash in merkle_proof {
    if current_position % 2 == 0 {
        current_hash = compute_hash(&current_hash, proof_hash);
    } else {
        current_hash = compute_hash(proof_hash, &current_hash);
    }
    current_position /= 2;
}
current_hash
```

Bitcoin's Merkle tree does not tag leaf nodes differently from internal nodes. Any internal node `N` at depth `d` with index `i` in a confirmed block's Merkle tree can be used as `tx_id`. The attacker constructs a `merkle_proof` of length `d` such that `compute_root_from_merkle_proof(N, i, proof) == merkle_root`. The function then returns `true`, falsely asserting that `N` is an included transaction.

The contract's own docstring acknowledges this explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

Despite being deprecated, the function carries no on-chain access restriction beyond `#[pause]`. The `#[deprecated]` Rust attribute is a compile-time lint only; it does not prevent runtime invocation. Any NEAR account — including a consumer contract that has not yet migrated to `verify_transaction_inclusion_v2` — can call it directly.

`verify_transaction_inclusion_v2` does add a coinbase proof check, but it then delegates to the same deprecated function via `self.verify_transaction_inclusion(args.into())`, so the underlying leaf-node gap is inherited for the non-coinbase transaction proof path.

---

### Impact Explanation

Any consumer contract (bridge, atomic swap, cross-chain lending protocol) that gates a high-value action on a `true` result from `verify_transaction_inclusion` can be tricked into releasing funds or minting tokens for a Bitcoin transaction that never existed. The attacker does not need to submit any block headers, control any privileged role, or possess any private key. They only need to identify any confirmed block already stored in the contract's `headers_pool` and compute the Merkle tree of that block offline — all public information.

The corrupted proof result is: `verify_transaction_inclusion` returns `true` for a `tx_id` that is an internal Merkle tree node, not a real transaction hash.

---

### Likelihood Explanation

The Merkle tree structure of every Bitcoin block is fully public and can be computed from block data available on any block explorer. Constructing a valid forgery input requires only offline computation. The only prerequisite is that the target consumer contract still calls the deprecated `verify_transaction_inclusion` rather than `verify_transaction_inclusion_v2`. Given that the deprecation is recent (v0.5.0) and the function remains callable on-chain with no runtime guard, deployed consumer contracts are realistically still using it.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public ABI entirely, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")` to make it a hard runtime error. Consumer contracts must migrate to `verify_transaction_inclusion_v2`. Additionally, `verify_transaction_inclusion_v2` should not internally delegate to the deprecated function; it should perform the full proof check inline so that the leaf-node ambiguity cannot be reintroduced.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block `B` stored in the contract's `headers_pool` with Merkle root `R` and at least two transactions.
2. Compute the Merkle tree of `B` offline. Pick any internal node `N` at depth `d` with index `i` (e.g., the root of the left subtree at depth 1, index 0).
3. Construct `merkle_proof` of length `d` such that `compute_root_from_merkle_proof(N, i, proof) == R`. For depth 1, the proof is simply the right sibling of `N` at depth 1.
4. Call `verify_transaction_inclusion({ tx_id: N, tx_block_blockhash: B.hash, tx_index: i, merkle_proof: proof, confirmations: 1 })`.
5. The function returns `true`, falsely asserting that `N` is an included transaction in block `B`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
