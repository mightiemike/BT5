### Title
Missing Minimum Quote Output Protection in `burnNlp` — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` calculates the user's quote payout using an `oraclePriceX18` that is **not part of the user's signed message**. The user signs only `{sender, nlpAmount, nonce}`, while the sequencer appends `oraclePriceX18` at execution time. Because no `minQuoteAmount` field exists in the `BurnNlp` struct, users have no on-chain protection against receiving less quote than expected if the NLP oracle price moves adversely between signing and execution.

---

### Finding Description

The `BurnNlp` struct contains only `sender`, `nlpAmount`, and `nonce`: [1](#0-0) 

The `oraclePriceX18` lives outside the signed inner struct, in `SignedBurnNlp`. The `Verifier.sol` digest for `BurnNlp` confirms that `oraclePriceX18` is **excluded from the user's signature**: [2](#0-1) 

In `EndpointTx.sol`, the sequencer supplies `oraclePriceX18` when it constructs and submits the transaction: [3](#0-2) 

Inside `Clearinghouse.burnNlp()`, the entire quote payout is derived from this sequencer-provided price with no floor check: [4](#0-3) 

There is no `minQuoteAmount` parameter anywhere in the call chain. `ERR_SLIPPAGE_TOO_HIGH` is defined in `Errors.sol` but is never referenced in any NLP burn path: [5](#0-4) 

---

### Impact Explanation

A user who signs a `BurnNlp` for `nlpAmount` tokens commits to burning exactly that many NLP tokens. The quote they receive is `nlpAmount × oraclePriceX18 − burnFee`. If the NLP NAV (and thus `oraclePriceX18`) drops between signing and sequencer execution — due to trading losses in the underlying pools, adverse funding, or other NLP burns — the user receives materially less quote than they anticipated, with no recourse. The NLP tokens are already burned.

**Impact: High** — concrete, irreversible asset loss (quote underpayment) with no user-controlled bound.

---

### Likelihood Explanation

The NLP oracle price is a function of the NAV of the NLP pools, which changes continuously with perp/spot PnL, funding payments, and other mints/burns. In active markets, the price can shift meaningfully in the time between a user signing and the sequencer processing the transaction. No sequencer compromise is required; natural market activity is sufficient.

**Likelihood: Low** — requires adverse price movement in the execution window, but this is a realistic condition in volatile markets.

---

### Recommendation

Add a `minQuoteAmount` field to the `BurnNlp` struct so users can commit to a minimum acceptable payout in their signed message:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint128 minQuoteAmount; // slippage guard
    uint64 nonce;
}
```

Include `minQuoteAmount` in the `Verifier.sol` digest for `BurnNlp`, and add a revert in `Clearinghouse.burnNlp()`:

```solidity
require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE_TOO_HIGH);
```

This mirrors the existing `ERR_SLIPPAGE_TOO_HIGH` error already defined in `Errors.sol` and gives users the same protection that buy-side transactions have through their fixed `quoteAmount` input.

---

### Proof of Concept

1. User signs `BurnNlp{sender=alice, nlpAmount=1000e18, nonce=5}` when NLP oracle price is `1.00 USDC`. Expected payout ≈ `999 USDC` (after 0.1% burn fee).
2. Before the sequencer processes the transaction, the NLP pools suffer trading losses and the oracle price drops to `0.80 USDC`.
3. Sequencer submits `SignedBurnNlp` with `oraclePriceX18 = 0.80e18`.
4. `Clearinghouse.burnNlp()` computes `quoteAmount = 1000e18 × 0.80e18 = 800e18`, then subtracts `burnFee = max(1e18, 800e18/1000) = 0.8e18`, yielding `≈799.2 USDC`.
5. Alice receives `≈799 USDC` instead of `≈999 USDC` — a ~20% shortfall — with no on-chain protection having triggered. [6](#0-5)
