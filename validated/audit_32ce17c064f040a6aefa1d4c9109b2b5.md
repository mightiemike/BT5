Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is intended to gate `addLiquidity` by the actual token-providing address (`sender`, i.e. `msg.sender` of `addLiquidity`). However, `beforeAddLiquidity` silently discards the `sender` parameter and instead validates `owner`, which is a free caller-supplied argument. Because any caller can set `owner` to any allowlisted address, the allowlist is completely bypassed by any unprivileged address.

## Finding Description
`MetricOmmPool.addLiquidity` invokes the extension hook with both `msg.sender` (the actual depositor) and `owner` (the caller-supplied position recipient):

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` correctly forwards both values to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
  BEFORE_ADD_LIQUIDITY_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

`DepositAllowlistExtension.beforeAddLiquidity` then discards `sender` (unnamed first parameter) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` inside the extension is the pool address (the caller of the extension). `owner` is the position recipient supplied freely by the attacker. The check `allowedDepositor[pool][owner]` passes whenever `owner` is any address the pool admin has allowlisted — regardless of who is actually calling `addLiquidity` and providing tokens via the callback.

The `allowedDepositor` mapping and `setAllowedToDeposit` admin API are documented and named around the concept of a **depositor** — the entity providing tokens — which is `sender`, not `owner`. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
Any address can call `pool.addLiquidity(owner = <allowlisted_address>, ...)`, pass the allowlist check, provide tokens via the `IMetricOmmModifyLiquidityCallback`, and have an LP position minted to the allowlisted address. The deposit allowlist — the sole access-control mechanism for restricted pools (KYC'd, whitelisted, or protocol-controlled depositors) — is completely defeated. Additionally, the allowlisted address receives an unsolicited LP position it did not authorize, exposing it to impermanent loss and fee drag until it manually removes the position. This constitutes a broken core pool access-control mechanism causing direct fund-impacting consequences (unauthorized liquidity provision, unsolicited LP positions with associated losses). [4](#0-3) 

## Likelihood Explanation
`addLiquidity` is fully permissionless — any EOA or contract can call it. The bypass requires only setting `owner` to any address the pool admin has allowlisted, which is discoverable on-chain via past `AllowedToDepositSet` events. No privileged access, special tokens, or complex setup are required. Every pool deploying `DepositAllowlistExtension` without `allowAllDepositors = true` is immediately and trivially vulnerable. [5](#0-4) 

## Recommendation
Replace the unnamed first parameter with `sender` and validate it instead of `owner`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
  {
-   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
      revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
``` [1](#0-0) 

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured and `allowAllDepositors[pool] = false`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)` — only Alice is permitted.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(
       owner        = alice,   // any allowlisted address
       salt         = 0,
       deltas       = <valid bins/shares>,
       callbackData = <Bob's callback pays tokens>,
       extensionData = ""
   );
   ```
4. `beforeAddLiquidity` receives `sender = Bob, owner = Alice`. It evaluates `allowedDepositor[pool][alice]` → `true`. No revert.
5. Bob's `IMetricOmmModifyLiquidityCallback` pays the tokens. The pool mints shares to `(alice, 0)`.
6. Bob has bypassed the allowlist: he deposited into a restricted pool, and Alice holds an unwanted LP position she did not initiate.

A Foundry test can confirm this by: deploying the extension, allowlisting only Alice, calling `addLiquidity` from Bob with `owner = alice`, and asserting no revert and that Alice's position shares are non-zero. [6](#0-5) [1](#0-0)
