Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is designed to gate swap access on a per-user basis, but the `sender` argument it receives is `msg.sender` of the pool's `swap()` call — which is the router contract, not the end user, when swaps are routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every user on the network can bypass the per-user allowlist by routing through the router. There is no configuration that allows specific users to swap through the router while blocking others.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to every configured extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router — the end user's address is stored only in transient callback context (`_setNextCallbackContext`) and is never forwarded to the extension: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]` — a single boolean covering **all** users who route through the router — rather than the individual user's address. The same flaw applies to `exactInput` and `exactOutputSingle`/`exactOutput` paths. [5](#0-4) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC-verified counterparties, institutional LPs, or protocol-controlled addresses) has that restriction fully bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool. If the router is allowlisted — the only way to support router-mediated swaps for legitimate users — the extension passes unconditionally for the attacker. The attacker executes swaps against the pool's liquidity, extracting tokens at oracle-derived prices from a pool intended to be access-controlled. This constitutes broken core swap functionality and unauthorized access to pool liquidity, meeting the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" impact criteria.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery contract explicitly documented as the mechanism for gating swap access per pool. Pool admins who deploy it with a curated allowlist will naturally also need to support router-mediated swaps for their allowlisted users, making router allowlisting the expected operational pattern. The bypass requires only a standard call to the public `MetricOmmSimpleRouter` — no special privileges, no flash loans, no price manipulation. Any user who can read the allowlist configuration on-chain can immediately exploit it.

## Recommendation
The `sender` forwarded to `beforeSwap` must represent the economic actor (the end user), not the intermediary contract. Two viable approaches:

1. **Router-side fix**: Have the router encode the original `msg.sender` (the user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when the direct `sender` is a known trusted router.
2. **Core-side fix (preferred)**: Add an `originator` field to the swap call that the pool passes through to extensions, distinct from `msg.sender`, so the router can attest the real user without relying on `extensionData` conventions.

## Proof of Concept
```solidity
// Pool is configured with SwapAllowlistExtension.
// Pool admin allowlists only `trustedUser` for direct swaps.
// Pool admin also allowlists `address(router)` to support router-mediated swaps.

// Attacker (not in allowlist) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool:             restrictedPool,
    tokenIn:          token0,
    recipient:        attacker,
    amountIn:         1_000e18,
    amountOutMinimum: 0,
    zeroForOne:       true,
    priceLimitX64:    0,
    deadline:         block.timestamp,
    extensionData:    ""
}));

// Execution path:
// 1. router calls restrictedPool.swap(attacker, true, 1000e18, ...)
//    → msg.sender to pool = router
// 2. pool calls _beforeSwap(sender=router, ...)
// 3. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
// 4. Swap executes; attacker receives token1 output.
// SwapAllowlistExtension never sees the attacker's address.
```
