### Title
Unbounded per-voter position iteration in governance tally can permanently block proposal finalization — (`x/tieredrewards/keeper/gov_tally.go`)

---

### Summary

The custom governance tally function `NewCustomTallyTierVotesFn` calls `GetPositionStatesByOwner` for every voter, which iterates **all** tier positions owned by that address with no upper bound. Because `MsgLockTier` imposes no per-owner position cap, any user can accumulate an arbitrary number of positions. When a governance proposal's voting period ends, the EndBlocker tally must walk every position for every voter; a voter who has created a large number of positions makes the tally prohibitively expensive, and because the EndBlocker runs under the block gas meter, exceeding the limit causes a consensus panic and chain halt.

---

### Finding Description

**Step 1 — Unbounded iteration in the tally.**

`NewCustomTallyTierVotesFn` walks every vote for a proposal and, for each voter, calls:

```go
positions, err := tierKeeper.GetPositionStatesByOwner(ctx, voter)
``` [1](#0-0) 

`GetPositionStatesByOwner` collects every position ID for the owner and then calls `getPositionState` for each one:

```go

    ...
    for _, id := range ids {
        state, err := k.getPositionState(ctx, id)

``` [2](#0-1) 

`getPositionState` performs two KV-store reads per position (one from `Positions`, one from the staking store via `getDelegation`): [3](#0-2) 

`getPositionsIdsByOwner` iterates the entire `PositionsByOwner` prefix for the owner with no limit: [4](#0-3) 

**Step 2 — No per-owner position cap.**

`MsgLockTier` validation only checks `amount >= MinLockAmount`, that the validator is bonded, that the tier is not close-only, and that the owner is not a vesting account. There is no maximum number of positions per owner: [5](#0-4) 

The `MsgClaimTierRewards` message enforces a 500-position batch limit, but that limit applies only to the claim message, not to position creation: [6](#0-5) 

**Step 3 — Tally runs in EndBlocker.**

The tiered-rewards module's `BeginBlock` is registered before distribution, and governance's `EndBlocker` finalizes proposals. The custom tally function is wired into the governance keeper via `WithCustomCalculateVoteResultsAndVotingPowerFn`. When a proposal's voting period ends, the EndBlocker calls the tally, which triggers the unbounded per-voter iteration under the block gas meter: [7](#0-6) 

---

### Impact Explanation

For each voter on a proposal, the tally performs `2 × N` KV-store reads (positions store + staking store), where `N` is the voter's position count. If `N` is large enough to push the block's cumulative gas past the block gas limit, the EndBlocker panics and the chain halts. Even below the halt threshold, a voter with thousands of positions makes every proposal finalization extremely expensive, effectively freezing governance for as long as the attacker keeps their positions open and votes on each proposal.

The corrupted invariant is: **governance proposals must be finalizeable within a single block**. With an unbounded per-voter position set, that invariant is broken.

---

### Likelihood Explanation

The attack is permissionless. Any address can call `MsgLockTier` repeatedly, creating one position per call. The only cost is `N × MinLockAmount` of bonded tokens, which are recoverable after the exit duration elapses. The attacker can then vote on every active proposal. Because positions are not removed automatically and there is no per-owner cap, the attacker's position count grows monotonically until they choose to exit.

---

### Recommendation

1. **Add a `MaxPositionsPerOwner` parameter** enforced in `MsgLockTier` validation, analogous to the existing 500-position batch limit on `MsgClaimTierRewards`.
2. **Bound the tally iteration**: in `NewCustomTallyTierVotesFn`, cap the number of positions fetched per voter (e.g., skip or truncate beyond the governance-safe limit) and document the invariant.
3. **Alternatively**, pre-aggregate tier voting power into a single per-owner value updated lazily on position mutations, so the tally performs O(1) reads per voter instead of O(N).

---

### Proof of Concept

1. Alice calls `MsgLockTier` 10,000 times, each with `MinLockAmount`, creating 10,000 positions.
2. A governance proposal is submitted and enters its voting period.
3. Alice calls `MsgVote` on the proposal.
4. When the voting period ends, `gov.EndBlocker` calls `NewCustomTallyTierVotesFn`.
5. The tally reaches Alice's vote and calls `GetPositionStatesByOwner(ctx, alice)`.
6. `getPositionsIdsByOwner` iterates 10,000 entries in `PositionsByOwner`; `getPositionState` then performs 20,000 KV-store reads.
7. The cumulative gas for the block exceeds the block gas limit; the EndBlocker panics; the chain halts.
8. Alice can later exit her positions (after the exit duration) and recover her funds, while the chain remains halted until a coordinated emergency upgrade. [8](#0-7) [2](#0-1) [4](#0-3)

func (k Keeper) getPositionState(ctx context.Context, posId uint64) (types.PositionState, error) {
	pos, err := k.getPosition(ctx, posId)
	if err != nil {
		return types.PositionState{}, err
	}
	del, err := k.getDelegation(ctx, pos.DelegatorAddress)
	if err != nil {
		return types.PositionState{}, err
	}
	return types.PositionState{Position: pos, Delegation: del}, nil
}

func (k Keeper) GetPositionStatesByOwner(ctx context.Context, owner sdk.AccAddress) ([]types.PositionState, error) {
	ids, err := k.getPositionsIdsByOwner(ctx, owner)
	if err != nil {
		return nil, err
	}
	states := make([]types.PositionState, 0, len(ids))
	for _, id := range ids {
		state, err := k.getPositionState(ctx, id)
		if errors.Is(err, types.ErrPositionNotFound) {
			continue
		}
		if err != nil {
			return nil, err
		}
		states = append(states, state)
	}
	return states, nil
```
