### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `pool.addLiquidity`) and instead validates `owner` (the LP-position recipient). Any address can bypass the allowlist by calling `pool.addLiquidity` with an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  sender      owner (LP position recipient, may differ)
``` [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, …)` but drops `sender` (unnamed first parameter) and gates on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, …) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [2](#0-1) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly names and checks `sender`:

```solidity
function beforeSwap(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
``` [3](#0-2) 

The pool's `addLiquidity` signature accepts an arbitrary `owner` that need not equal `msg.sender`:

```solidity
function addLiquidity(address owner, uint80 salt, …) external nonReentrant …
``` [4](#0-3) 

`MetricOmmPoolLiquidityAdder` explicitly exposes this separation — `owner` is a free parameter while `msg.sender` is the payer:

```solidity
function addLiquidityExactShares(address pool, address owner, …) external payable {
    _validateOwner(owner);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, …);
``` [5](#0-4) 

---

### Impact Explanation

A pool configured with `DepositAllowlistExtension` to restrict deposits to a KYC/allowlisted set has no effective access control on who actually calls `addLiquidity`. Any unauthorized address can:

1. Call `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)` directly.
2. The hook checks `owner` = `allowlistedAddress` → passes.
3. The unauthorized address pays tokens via the modify-liquidity callback.
4. LP shares are credited to `allowlistedAddress`.
5. `allowlistedAddress` (colluding or controlled by the attacker) calls `removeLiquidity` and returns the tokens.

The deposit allowlist — the sole access-control mechanism for `addLiquidity` — is fully bypassed. Pools intended to be restricted (regulated, permissioned, or institutional) accept deposits from any address. This breaks the core pool invariant that only allowlisted parties may provide liquidity.

---

### Likelihood Explanation

- The `owner` parameter is freely settable by any caller of `pool.addLiquidity`.
- No other guard in the pool enforces `msg.sender == owner` for `addLiquidity` (only `removeLiquidity` enforces `msg.sender == owner`).
- The `MetricOmmPoolLiquidityAdder` router makes the `sender ≠ owner` pattern a first-class, documented use case.
- A pool admin who deploys `DepositAllowlistExtension` believing it gates depositors will be unaware the guard is misbound.

---

### Recommendation

Name and check `sender` (the first parameter), matching the pattern used by `SwapAllowlistExtension`:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, …) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {

// After (correct):
function beforeAddLiquidity(address sender, address, …) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
```

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  alice  → allowedDepositor[pool][alice] = true
  bob    → not allowlisted

Attack:
  bob calls pool.addLiquidity(alice, salt, deltas, callbackData, "")
    → _beforeAddLiquidity(bob /*sender*/, alice /*owner*/, …)
    → hook checks allowedDepositor[pool][alice] == true  ✓  (passes)
    → bob's callback pays token0/token1 into the pool
    → LP shares credited to alice

  alice calls pool.removeLiquidity(alice, salt, deltas, "")
    → tokens returned to alice

Result: bob deposited into a restricted pool; allowlist completely bypassed.
``` [2](#0-1) [6](#0-5)
