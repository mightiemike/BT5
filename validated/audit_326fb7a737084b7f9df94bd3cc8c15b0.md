Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the immediate pool caller (router) instead of the originating swapper, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the extension evaluates the router's allowlist status rather than the originating user's. A pool admin who allowlists the router to grant approved users router access inadvertently opens the pool to every user on the public, permissionless router, completely defeating the curated-pool allowlist.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every configured extension via `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // always the immediate caller of pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that exact value against the per-pool allowlist:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` inside the extension is the pool; `sender` is whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

The pool receives `msg.sender = MetricOmmSimpleRouter`. The extension therefore evaluates `allowedSwapper[pool][MetricOmmSimpleRouter]`, not `allowedSwapper[pool][originalUser]`. The `bytes calldata` `extensionData` parameter is accepted but entirely ignored by the extension — there is no mechanism to recover the original user's identity.

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the LP-share beneficiary) rather than `sender` (the payer), because the owner is the economically relevant actor for deposits:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

`SwapAllowlistExtension` has no equivalent: it checks `sender`, which collapses to the router address on every router-mediated swap. The same bypass applies to `exactInput` and `exactOutputSingle` / `exactOutput` paths in the router. [5](#0-4) 

## Impact Explanation

A curated pool protected by `SwapAllowlistExtension` is fully bypassed by any unprivileged user who routes through `MetricOmmSimpleRouter` once the pool admin allowlists the router. The allowlist's purpose — restricting swaps to KYC'd counterparties, institutional participants, or any other curated set — is defeated. LPs who deposited under the assumption that only approved counterparties would trade against them are exposed to the full public, including adversarial MEV bots and front-runners. This constitutes broken core pool functionality (the access-control extension produces the wrong extension decision — `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`) and a direct loss of the protection LPs relied upon.

## Likelihood Explanation

The trigger is a pool admin allowlisting the router — a natural, expected action for any operator who wants their approved users to enjoy the router's UX (slippage limits, multi-hop, deadline checks). The extension gives no warning that doing so opens the gate to everyone. The router is a public, permissionless contract. Once the router is allowlisted, any user can exploit the bypass without any further privileged action, repeatedly and at will.

## Recommendation

The extension must gate on the originating user, not the immediate pool caller. Two viable approaches:

1. **Router-aware allowlist**: Add a `mapping(address => bool)` of trusted routers registered by the factory or extension admin. When `sender` is a trusted router, decode the real user from `extensionData` and check that address instead.
2. **Signed identity in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension reads and verifies it against a factory-registered router list (trusted-forwarder pattern).

The simplest safe interim fix is to reject any `sender` that is not itself in the per-pool allowlist and document that router-mediated swaps are unsupported on allowlisted pools until a forwarding mechanism is added.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)   // Alice is the approved trader
3. Admin calls setAllowedToSwap(pool, router, true)  // Admin wants Alice to use the router

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <curated pool>,
           recipient: bob,
           ...
       })

5. Router calls pool.swap(bob, ...) with msg.sender = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true   ✓ (step 3)
   Extension does NOT revert.

8. Bob's swap executes on the curated pool.
   The allowlist is fully bypassed.

Foundry test sketch:
- Deploy SwapAllowlistExtension, pool, and MetricOmmSimpleRouter.
- setAllowedToSwap(pool, alice, true); setAllowedToSwap(pool, router, true).
- vm.prank(bob); router.exactInputSingle({pool: pool, ...});
- Assert swap succeeds (no NotAllowedToSwap revert) despite bob not being allowlisted.
```
