### Title
Unprotected `initialize` Functions in `SpotEngine` and `WithdrawPool` Allow Frontrunning to Seize Contract Ownership and Drain Funds - (File: `core/contracts/SpotEngine.sol`, `core/contracts/WithdrawPool.sol`)

---

### Summary

`SpotEngine.initialize()`, `PerpEngine.initialize()`, and `WithdrawPool.initialize()` are all `external` functions with no access control and no `initializer` modifier on the public entry point. Any unprivileged caller who observes the deployment mempool can frontrun the legitimate initialization transaction, inject attacker-controlled addresses, and gain privileged control over the contract's critical state.

---

### Finding Description

`SpotEngine.initialize()` is declared `external` with no modifier and no caller restriction: [1](#0-0) 

It delegates to `BaseEngine._initialize()`, which carries the `initializer` modifier: [2](#0-1) 

The `initializer` modifier from OpenZeppelin's `Initializable` is a **first-caller-wins** guard. It does not restrict *who* may call the function — it only prevents a second call. An attacker who calls `SpotEngine.initialize()` before the protocol's `Clearinghouse.addEngine()` transaction is mined will permanently set:

- `_clearinghouse` → attacker-controlled address (added to `canApplyDeltas`)
- `_offchainExchange` → attacker-controlled address (added to `canApplyDeltas`)
- `_endpoint` → attacker-controlled address (added to `canApplyDeltas`)
- `_admin` → attacker address (becomes `owner`)

The same pattern exists identically in `PerpEngine.initialize()`: [3](#0-2) 

And in `WithdrawPool.initialize()`, which has no `initializer` modifier at all on the public function and no access control: [4](#0-3) 

`BaseWithdrawPool._initialize()` carries the `initializer` guard: [5](#0-4) 

The `_disableInitializers()` call in `BaseWithdrawPool`'s constructor only protects the bare implementation contract, not the proxy's storage: [6](#0-5) 

---

### Impact Explanation

**SpotEngine / PerpEngine path:**

After a successful frontrun of `SpotEngine.initialize()`, the attacker is in `canApplyDeltas` and is the contract owner. The `_assertInternal()` guard used by every balance-mutating function checks only `canApplyDeltas[msg.sender]`: [7](#0-6) 

The attacker can therefore call `SpotEngine.updateBalance()` directly to credit themselves an arbitrary quote balance, then submit a legitimate `WithdrawCollateral` transaction through the real `Endpoint` to drain ERC-20 tokens from the `Clearinghouse`. The protocol's own `addEngine` call will revert, leaving the attacker in permanent control.

**WithdrawPool path:**

After frontrunning `WithdrawPool.initialize()`, the attacker sets `clearinghouse` to their own address. `submitWithdrawal` checks only `msg.sender == clearinghouse`: [8](#0-7) 

The attacker can immediately call `submitWithdrawal` to redirect any token balance held in the pool to an arbitrary address.

---

### Likelihood Explanation

The attack requires only mempool monitoring and a higher-gas frontrun transaction — no special privileges, no governance capture, no leaked keys. The window is the gap between proxy deployment and the first legitimate `initialize` call. This is a well-known and mechanically straightforward attack class. Likelihood is **Medium** (requires active mempool monitoring, but the technique is trivially automated).

---

### Recommendation

Add the `initializer` modifier directly to each public `initialize` function, not only to the internal delegate. Additionally, add a caller restriction (e.g., `require(msg.sender == deployer)` or an `onlyOwner` guard seeded at construction time) to all three functions. The pattern used in `ContractOwner.initialize()` is the correct reference: [9](#0-8) 

For `SpotEngine` and `PerpEngine`, add a `constructor()` that calls `_disableInitializers()` (as `BaseWithdrawPool` already does) and add the `initializer` modifier to the public `initialize` function. For `WithdrawPool`, add the `initializer` modifier directly to `WithdrawPool.initialize()`.

---

### Proof of Concept

1. Protocol deploys `SpotEngine` proxy and broadcasts `Clearinghouse.addEngine(spotEngineProxy, offchainExchange, SPOT)`.
2. Attacker observes the pending transaction in the mempool.
3. Attacker submits a higher-gas call:
   ```solidity
   SpotEngine(spotEngineProxy).initialize(
       address(realClearinghouse),  // keeps real clearinghouse in canApplyDeltas
       address(attacker),           // attacker added to canApplyDeltas
       quoteToken,
       address(realEndpoint),
       address(attacker)            // attacker becomes owner
   );
   ```
4. Attacker's transaction mines first. `_initialize` sets `initialized = true`.
5. Protocol's `addEngine` call reverts with "Initializable: contract is already initialized".
6. Attacker calls `SpotEngine.updateBalance(QUOTE_PRODUCT_ID, attackerSubaccount, 1_000_000e18)` directly (passes `_assertInternal()` since `canApplyDeltas[attacker] == true`).
7. Attacker calls `Endpoint.submitSlowModeTransaction(WithdrawCollateral{...})` to withdraw the credited balance, draining real tokens from the Clearinghouse.
