### Title
Protobuf `ValidResourceBounds` deserialization silently downcasts `AllResources` to `L1Gas` when `l2_gas=0` and `l1_data_gas=0`, causing consensus-path transaction rejection for valid transactions - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary
The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a value-based heuristic to select the `ValidResourceBounds` variant: if `l1_data_gas.is_zero() && l2_gas.is_zero()` it emits `ValidResourceBounds::L1Gas`, otherwise `ValidResourceBounds::AllResources`. A V3 invoke transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` is admitted as valid by the gateway (stateless validator explicitly allows it), but when the proposer serializes it into a consensus protobuf message and a validator deserializes it, the conversion produces `L1Gas` instead of `AllResources`. The downstream `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` then hard-fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR` because `RpcInvokeTransactionV3` requires `AllResources`. The validator rejects the block, breaking consensus liveness. The codebase itself acknowledges the defect in a test-only workaround.

### Finding Description

**The faulty conversion** is at `crates/apollo_protobuf/src/converters/transaction.rs:431`:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ŌåÉ wrong variant for AllResources tx
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

This heuristic was introduced to handle legacy pre-0.13.3 transactions stored in the DB, but it is also applied on the consensus deserialization path for `RpcInvokeTransactionV3`, `RpcDeclareTransactionV3`, and `RpcDeployAccountTransactionV3`, all of which go through `InvokeTransactionV3` / `DeclareTransactionV3Common` / `DeployAccountTransactionV3` (which carry `ValidResourceBounds`) before being converted to the RPC types.

**The gateway admits the trigger transaction.** The stateless validator explicitly marks a transaction with `l1_gas = NON_EMPTY`, `l2_gas = 0`, `l1_data_gas = 0` as valid: [2](#0-1) 

**The proposer computes the hash using `AllResources`.** `InternalRpcInvokeTransactionV3::resource_bounds()` unconditionally returns `ValidResourceBounds::AllResources(self.resource_bounds)`: [3](#0-2) 

`get_tip_resource_bounds_hash` hashes a **3-element** chain for `AllResources` (L1_GAS + L2_GAS + L1_DATA_GAS) but only a **2-element** chain for `L1Gas` (L1_GAS + L2_GAS): [4](#0-3) 

So the proposer stores hash H_A (3-element preimage).

**The validator's deserialization path fails.** When the consensus protobuf is received, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires and produces `L1Gas`. The subsequent `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` (same pattern as `RpcDeclareTransactionV3`) hard-errors: [5](#0-4) [6](#0-5) 

**The codebase acknowledges the defect.** The consensus round-trip test explicitly patches transactions to have `l2_gas.max_amount = GasAmount(1)` to avoid triggering this exact downcast: [7](#0-6) 

### Impact Explanation
A validator that receives a consensus proposal containing a transaction with `l2_gas=0` and `l1_data_gas=0` will fail to deserialize it and reject the entire block. Because the gateway admits such transactions and the proposer includes them, any block containing one will be permanently rejected by all validators. This breaks consensus liveness and maps to **High ŌĆō Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**, and **High ŌĆō Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**.

### Likelihood Explanation
Any unprivileged user can submit a V3 invoke with only `l1_gas` set (the other two bounds left at zero). The stateless validator passes it. No special account, contract, or privileged access is required. The trigger is a single well-formed RPC call.

### Recommendation
Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with a structural one: since the protobuf wire format always carries all three fields for V3 transactions, always produce `AllResources` when all three fields are present (even if zero-valued). Reserve `L1Gas` only for the legacy case where `l1_data_gas` is absent from the wire message (the `unwrap_or_default` branch already handles that). Alternatively, change the `RpcInvokeTransactionV3` protobuf deserialization to use `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` directly (which already always produces `AllResourceBounds`) instead of routing through `InvokeTransactionV3`.

### Proof of Concept
1. Submit via RPC:
   ```json
   { "type": "INVOKE", "version": "0x3",
     "resource_bounds": { "l1_gas": {"max_amount":"0x100","max_price_per_unit":"0x1"},
                          "l2_gas": {"max_amount":"0x0","max_price_per_unit":"0x0"},
                          "l1_data_gas": {"max_amount":"0x0","max_price_per_unit":"0x0"} },
     ... }
   ```
2. Gateway stateless validator passes it (`valid_l1_gas` test case).
3. Proposer computes hash H_A using `AllResources` (3-element preimage) and includes the transaction in a block proposal.
4. Validator receives the consensus protobuf; `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires with `l1_data_gas.is_zero() && l2_gas.is_zero()` ŌåÆ produces `L1Gas`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
6. Validator rejects the block; consensus stalls for any block containing such a transaction.
