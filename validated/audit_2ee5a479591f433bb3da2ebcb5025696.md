### Title
Deprecated `verify_transaction_inclusion` Remains a Live Callable Entry Point, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification functions. The newer `verify_transaction_inclusion_v2` adds a coinbase merkle proof check to prevent the 64-byte internal-node forgery attack. However, the original `verify_transaction_inclusion` (v1) was never removed or runtime-restricted — it remains a fully callable public function. Rust's `#[deprecated]` attribute is compile-time only; it imposes no runtime barrier. Any unprivileged NEAR account can call v1 directly, bypassing the coinbase proof check entirely and obtaining a `true` return value for a fabricated transaction hash.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated(since = "0.5.0")]` and carries an explicit warning in its doc comment:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

Despite this, the function is still decorated with `#[pause]` (not removed), is `pub`, and is reachable by any NEAR caller without any role restriction. [2](#0-1) 

The root cause is in `compute_root_from_merkle_proof`: it iterates over whatever proof slice is supplied without validating that the proof length equals the actual tree depth. [3](#0-2) 

This means an attacker can supply an **internal merkle node** as `tx_id` together with a **shorter-than-full-depth proof** and the computed root will still equal the block's `merkle_root`, causing v1 to return `true`.

The secure path, `verify_transaction_inclusion_v2`, closes this by requiring `merkle_proof.len() == coinbase_merkle_proof.len()` and independently verifying the coinbase transaction at index 0 (whose proof length equals the true tree depth). [4](#0-3) 

Because v1 is never called by v2 with the coinbase guard in place for the *direct* v1 call path, a caller who invokes v1 directly receives none of those protections.

---

### Impact Explanation

Any consumer NEAR contract that calls `verify_transaction_inclusion` (v1) — whether written before v2 existed or by a developer unaware of the deprecation — can be deceived into accepting a forged Bitcoin transaction inclusion proof. The attacker does not need to break SHA-256; they only need to identify an existing internal merkle node in any confirmed block and supply a proof one level shorter than the full tree depth. The function returns `true`, and the consumer contract acts on a false positive — potentially releasing bridged funds, minting tokens, or confirming a payment that never occurred on Bitcoin.

**Impact: 3** (matches the original report's score; high consequence for any consumer contract relying on this API).

---

### Likelihood Explanation

The function is unconditionally reachable by any NEAR account with no role gate. The `#[deprecated]` marker is invisible at runtime. Consumer contracts compiled against an older ABI or written without reading the deprecation notice will call v1 without knowing it is unsafe. The attack requires only knowledge of a real block's merkle tree structure, which is public Bitcoin data.

**Likelihood: 3** (matches the original report's score; the entry point is open and the exploit requires only public chain data).

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or replace its body with an unconditional `env::panic_str("Removed: use verify_transaction_inclusion_v2")` so that any direct call fails at runtime rather than silently returning a forgeable result. The `#[deprecated]` attribute alone is insufficient because it is a compile-time hint that provides no runtime enforcement.

---

### Proof of Concept

Consider a confirmed mainchain block whose merkle tree has four transactions `[T0, T1, T2, T3]`:

```
Level 2 (root): R  = H(H01, H23)
Level 1:        H01 = H(T0, T1),   H23 = H(T2, T3)
Level 0 (leaf): T0, T1, T2, T3
```

An attacker calls `verify_transaction_inclusion` directly (bypassing v2) with:

```
tx_id             = H01          // internal node, not a real transaction
tx_block_blockhash = <valid mainchain block hash>
tx_index          = 0
merkle_proof      = [H23]        // proof of length 1, not 2
confirmations     = 1
```

`compute_root_from_merkle_proof(H01, 0, [H23])` computes `H(H01, H23) = R`, which equals the block's `merkle_root`. [5](#0-4) 

The function returns `true` for the fabricated transaction `H01`. A consumer contract acting on this result would treat a non-existent Bitcoin transaction as confirmed.

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
