---

### Summary

`TransactionType.WithdrawCollateralV2` is fully defined in the interface and handled in the sequencer-submitted path (`processTransactionImpl`), but is entirely absent from the slow mode execution path (`processSlowModeTransactionImpl`). Because `submitSlowModeTransactionImpl` does not gate this transaction type, any user can queue it and pay the slow mode fee. When the transaction is later executed, `processSlowModeTransactionImpl` reaches the terminal `revert()` for unhandled types. ### Title
`WithdrawCollateralV2` Not Implemented in Slow Mode Processing Path Causes Silent Fee Loss and Censorship-Resistance Bypass - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` is declared in the `TransactionType` enum and fully handled in the sequencer-submitted path (`processTransactionImpl`), but is entirely absent from the slow mode processing path (`processSlowModeTransactionImpl`). A user who submits a `WithdrawCollateralV2` transaction via `submitSlowModeTransaction` will pay the slow mode fee, have their transaction queued, and then silently lose both the fee and the withdrawal when execution is attempted. The censorship-resistance path for `WithdrawCollateralV2` — including its `sendTo` feature — is non-functional.

---

### Finding Description

**Step 1 — Declaration exists.** `WithdrawCollateralV2` is declared in the `TransactionType` enum: [1](#0-0) 

**Step 2 — Sequencer path is fully implemented.** `processTransactionImpl` handles `WithdrawCollateralV2` with signature validation, fee charging, and `clearinghouse.withdrawCollateral` call: [2](#0-1) 

**Step 3 — Slow mode submission does not reject it.** In `submitSlowModeTransactionImpl`, `WithdrawCollateralV2` is not in the `DepositCollateral` revert branch and not in the owner-only branch, so it falls to the `else` clause that charges the slow mode fee and queues the transaction: [5](#0-4) 

**Step 4 — Slow mode execution path has no handler.** `processSlowModeTransactionImpl` handles `WithdrawCollateral` (V1) but has no branch for `WithdrawCollateralV2`. It falls to `else { revert(); }`: [6](#0-5) [3](#0-2) 

**Step 5 — The revert is silently swallowed.** `_executeSlowModeTransaction` wraps execution in a try/catch. The comment `// try return funds now removed` confirms that fee refund logic was deliberately removed. The revert from step 4 is caught and discarded: [7](#0-6) 

The net result: the user's slow mode fee is deducted, the transaction is queued, and on execution it silently fails with no refund and no withdrawal.

---

### Impact Explanation

1. **Concrete asset loss**: Any user who submits `WithdrawCollateralV2` via slow mode loses their slow mode fee in quote tokens with no recourse, because the fee is charged at submission time and the failure is silently swallowed at execution time.

2. **Censorship-resistance bypass**: The slow mode queue is the protocol's censorship-resistance mechanism. `WithdrawCollateralV2` adds a `sendTo` field allowing withdrawal to an arbitrary address. If the sequencer censors a user's `WithdrawCollateralV2` transactions, the user has no functional slow mode fallback — they can only fall back to `WithdrawCollateral` (V1), which forces withdrawal to their own address and eliminates the `sendTo` capability entirely. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** Any user who:
- Needs to withdraw to a custom `sendTo` address (e.g., a smart contract recipient), AND
- Finds the sequencer offline or unresponsive,

will naturally attempt `WithdrawCollateralV2` via slow mode. The fee loss is immediate and automatic. No special attacker capability is required — this is triggered by ordinary user behavior against a declared, publicly visible transaction type.

---

### Recommendation

Add a `WithdrawCollateralV2` handler in `processSlowModeTransactionImpl` in `EndpointTx.sol`, mirroring the existing `WithdrawCollateral` handler but decoding `SignedWithdrawCollateralV2` and passing `txn.sendTo` to `clearinghouse.withdrawCollateral`. Alternatively, explicitly revert in `submitSlowModeTransactionImpl` for `WithdrawCollateralV2` (as is done for `DepositCollateral`) so users are not charged a fee for an unprocessable transaction type.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(withdrawCollateralV2Bytes)`.
2. `submitSlowModeTransactionImpl` is reached via `delegatecall`. `WithdrawCollateralV2` is not `DepositCollateral` and not owner-only, so the `else` branch executes: slow mode fee is charged from the user's quote balance, `slowModeFees` is incremented, and the transaction is pushed to `slowModeTxs`.
3. After `SLOW_MODE_TX_DELAY` (3 days), anyone calls `executeSlowModeTransaction`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)` inside a try/catch.
5. `processSlowModeTransactionImpl` is reached. No branch matches `WithdrawCollateralV2`. Execution reaches `else { revert(); }`.
6. The revert propagates to the try/catch in `_executeSlowModeTransaction`. Gas check passes (not OOG). The catch block exits silently.
7. `nSubmissions` is incremented, `slowModeConfig` is updated. The user's slow mode fee is permanently lost. The withdrawal never executes. [9](#0-8) [10](#0-9)
