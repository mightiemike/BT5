### Title
`DepositAllowlistExtension` Registered in `beforeRemoveLiquidity` Order Permanently Locks All LP Funds - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is a production extension that only overrides `beforeAddLiquidity`. All other hook slots — including `beforeRemoveLiquidity` — are inherited from `BaseMetricExtension`, which unconditionally reverts with `ExtensionNotImplemented()`. The factory's `ValidateExtensionsConfig` validates only structural correctness of extension orders (index bounds, no duplicates) but never verifies that a registered extension actually implements the hook it is wired to. A pool can therefore be deployed with `DepositAllowlistExtension` in the `BEFORE_REMOVE_LIQUIDITY_ORDER`, causing every `removeLiquidity` call to revert permanently and locking all LP principal with no recovery path.

---

### Finding Description

`DepositAllowlistExtension` inherits from `BaseMetricExtension` and overrides only `beforeAddLiquidity`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
contract DepositAllowlistExtension is BaseMetricExtension, IDepositAllowlistExtension {
  function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
  { ... }
  // beforeRemoveLiquidity is NOT overridden
}
```

`BaseMetricExtension.beforeRemoveLiquidity` is a `virtual` function that always reverts:

```solidity
// metric-periphery/contracts/extensions/base/BaseMetricExtension.sol
function beforeRemoveLiquidity(address, address, uint80, LiquidityDelta calldata, bytes calldata)
  external virtual onlyPool returns (bytes4)
{
  revert ExtensionNotImplemented();
}
```

The factory's `ValidateExtensionsConfig.validateExtensionsConfig` only checks structural validity — that indices are in-bounds and non-duplicate — but never calls or inspects the extension to confirm it implements the hook:

```solidity
// metric-core/contracts/libraries/ValidateExtensionsConfig.sol
_validateExtensionOrder(orders.beforeAddLiquidity, extensions.length);
_validateExtensionOrder(orders.afterAddLiquidity, extensions.length);
_validateExtensionOrder(orders.beforeRemoveLiquidity, extensions.length);  // no semantic check
```

When `removeLiquidity` is called on a pool where `BEFORE_REMOVE_LIQUIDITY_ORDER` includes the `DepositAllowlistExtension` slot, the execution path is:

1. `MetricOmmPool.removeLiquidity` → `_beforeRemoveLiquidity`
2. `_callExtensionsInOrder(BEFORE_REMOVE_LIQUIDITY_ORDER, ...)` → `CallExtension.callExtension(extension, data)`
3. `DepositAllowlistExtension.beforeRemoveLiquidity` is called; `onlyPool` passes (pool is registered in factory)
4. `revert ExtensionNotImplemented()` is hit
5. `CallExtension.callExtension` bubbles the revert up via assembly
6. `removeLiquidity` reverts — permanently, for every caller, with no admin escape hatch

The pool's extension configuration is immutable (set at construction via `ExtensionCalling` constructor), so there is no way to fix this after deployment.

---

### Impact Explanation

All LP principal deposited into the pool is permanently locked. `removeLiquidity` is the sole withdrawal path for LPs; there is no alternative recovery function. Because `BEFORE_REMOVE_LIQUIDITY_ORDER` is an immutable set at construction time, no admin action — not even the factory owner — can unblock withdrawals after deployment. This is a direct, total loss of user principal with no upper bound on the amount at risk.

---

### Likelihood Explanation

`DepositAllowlistExtension` is the primary production allowlist extension. A pool creator intending to gate deposits would naturally register it as extension1 and set `beforeAddLiquidity = 1`. If they also set `beforeRemoveLiquidity = 1` (e.g., believing the extension governs all liquidity operations, or by copy-paste error), the pool is bricked. The factory provides no warning, no revert, and no simulation of the hook call during pool creation. The misconfiguration is silent at deployment and only manifests when the first LP attempts to withdraw.

---

### Recommendation

`ValidateExtensionsConfig.validateExtensionsConfig` should perform a dry-run call to each extension for each hook order it is registered in, verifying the extension returns the expected selector without reverting. Alternatively, `BaseMetricExtension` should return the correct selector (pass-through) for unimplemented hooks rather than reverting, making it safe to register in any order. The current "revert on unimplemented" default is appropriate for catching integration bugs during development but is catastrophic when it reaches a production pool with real funds.

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
```

  function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
      if (result.length > 0) {
        assembly ("memory-safe") {
          revert(add(result, 32), mload(result))
        }
      }
      revert ExtensionCallFailed();
    }
    if (result.length < 32) {
      revert InvalidExtensionResponse();
    }
    bytes4 returnedSelector;
    assembly ("memory-safe") {
      returnedSelector := mload(add(result, 32))
    }
    bytes4 expectedSelector;
    assembly ("memory-safe") {
      expectedSelector := mload(add(data, 32))
    }
    if (returnedSelector != expectedSelector) {
      revert InvalidExtensionResponse();
    }
  }
```
