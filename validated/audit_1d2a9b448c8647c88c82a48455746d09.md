All cited code is confirmed against the actual repository. The vulnerability is real and exploitable as described.

Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor/caller) and instead checks `owner` (the position owner), which is a free caller-supplied argument never authenticated by the pool. Any non-allowlisted address can bypass the allowlist by naming an already-allowlisted address as `owner`. The parallel `SwapAllowlistExtension` correctly checks `sender`, confirming the intended design was to gate on the actual actor.

## Finding Description
The extension's `beforeAddLiquidity` function drops the first parameter and checks `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The pool correctly passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`: [2](#0-1) 

`ExtensionCalling` faithfully forwards both: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts `owner` as a free caller-supplied argument with no authentication beyond a non-zero check, while `msg.sender` (the actual token payer) is stored separately in transient context: [4](#0-3) 

The `_validateOwner` check only rejects `address(0)`: [5](#0-4) 

The pool's `addLiquidity` has no `msg.sender == owner` guard (unlike `removeLiquidity` which enforces `msg.sender != owner` revert): [6](#0-5) 

**Exploit path:**
1. Pool is configured with `DepositAllowlistExtension`; `alice` is allowlisted, `bob` is not.
2. `bob` calls `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extData)`.
3. Router calls `pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), extData)`.
4. Pool calls `_beforeAddLiquidity(router, alice, ...)`.
5. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. Pool executes; callback pulls tokens from `bob` (the payer in transient context).
7. Position recorded under `alice`; `alice` later calls `removeLiquidity` and returns proceeds to `bob`.
8. `bob` has effectively provided liquidity to a pool that explicitly excluded him.

The wrong value checked is `allowedDepositor[msg.sender][owner]` — it should be `allowedDepositor[msg.sender][sender]`. [7](#0-6) 

## Impact Explanation
This is an admin-boundary break: the deposit allowlist is a pool admin access-control mechanism, and it is fully bypassed by an unprivileged path. In the collusion path, a non-allowlisted depositor earns LP fees from a pool they are explicitly barred from. In the griefing path, a non-allowlisted actor force-deposits tokens into any allowlisted address's position without consent, corrupting that LP's position accounting. Both paths violate the pool admin's access-control invariant and constitute a broken core pool functionality/admin-boundary break per the allowed impact gate. [8](#0-7) 

## Likelihood Explanation
Exploitation requires only knowing any allowlisted LP address — public on-chain information readable from `AllowedToDepositSet` events or direct `allowedDepositor` mapping reads. No privileged role, flash loan, oracle manipulation, or special setup is needed. `MetricOmmPoolLiquidityAdder` is the standard EOA entry point and is immediately usable. [9](#0-8) 

## Recommendation
Check `sender` (the actual depositor) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [10](#0-9) 

## Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`; call `setAllowedToDeposit(pool, alice, true)`.
2. As `bob` (not allowlisted), call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, extData)` with `bob` having approved the adder for token pulls.
3. Observe: extension checks `allowedDepositor[pool][alice]` → `true`, no revert, liquidity added with `bob`'s tokens under `alice`'s position.
4. `alice` calls `removeLiquidity` and transfers proceeds to `bob`.
5. Foundry test: assert `bob`'s token balance decreases by deposit amount and `alice`'s position shares increase, confirming the bypass. [11](#0-10)
