Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the intermediate router address instead of the originating user, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension` is intended to restrict swap access to a curated set of addresses per pool. When swaps are routed through `MetricOmmSimpleRouter`, the `sender` argument delivered to `beforeSwap` is the router's address — not the originating user — because the router is `msg.sender` of `pool.swap()`. Any pool admin who allowlists the router to enable router-mediated swaps for their allowlisted users simultaneously grants every user on the network unrestricted swap access to that pool.

## Finding Description

**Root cause — `msg.sender` of `pool.swap()` is forwarded as `sender` to extensions:**

In `MetricOmmPool.swap()`, `msg.sender` (the direct caller of the pool) is passed as `sender` to `_beforeSwap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← direct caller, not the originating user
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap()` encodes that value verbatim and dispatches it to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L162-176
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**The bypass path through `MetricOmmSimpleRouter`:**

When a user calls `router.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

At that call site, `msg.sender` inside `pool.swap()` is the **router address**, not the originating user. The extension therefore evaluates `allowedSwapper[pool][router_address]`, not `allowedSwapper[pool][actual_user]`.

**The dilemma this creates for pool admins:**

| Admin intent | Admin action | Actual result |
|---|---|---|
| Allow only Alice to swap via router | Allowlist Alice | Router-mediated swaps fail for Alice (router not allowlisted) |
| Allow Alice to swap via router | Allowlist the router | Every user can swap via the router — allowlist is void |

There is no configuration that simultaneously (a) allows allowlisted users to use the router and (b) blocks non-allowlisted users from using the router. The same misbinding applies to `exactInput`, `exactOutputSingle`, and `exactOutput` in `MetricOmmSimpleRouter`.

**Existing guards are insufficient:** There is no mechanism in `SwapAllowlistExtension` to decode or verify the originating user from `extensionData`, and the pool itself has no `originator` field separate from `sender`. The extension has no way to distinguish a router-mediated call from a direct call.

## Impact Explanation

Any user can trade on a pool whose admin configured `SwapAllowlistExtension` to restrict access to specific counterparties. The allowlist — the sole access-control mechanism on the swap path — is completely neutralised for router-mediated swaps. Pools designed for KYC'd users, institutional counterparties, or any curated set of traders are exposed to unrestricted public trading. This constitutes an admin-boundary break (factory/oracle role checks bypassed by an unprivileged path) and broken core pool functionality (the configured guard silently fails open for the standard periphery entry point), both of which are within the allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and expects users to interact via the router will be forced to allowlist the router, triggering the bypass. The attacker needs no special privilege, no malicious setup, and no non-standard token — only the publicly deployed router address. The bypass is repeatable on every such pool.

## Recommendation

The extension must check the **economically relevant actor** — the address that initiated the trade — not the intermediate contract that called `pool.swap()`.

Two viable approaches:

1. **Pass the originating user through `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into the `extensionData` field it forwards to the pool. `SwapAllowlistExtension.beforeSwap` decodes and verifies it. A separate `onlyRouter` guard or a signed payload can harden this against spoofing.

2. **Add an `originator` field to the pool's swap interface**: Extend `IMetricOmmPoolActions.swap` with an explicit `originator` address that the pool passes to extensions alongside `sender`. The router sets `originator = msg.sender`; direct callers set `originator = address(0)` (falling back to `sender`). Extensions check `originator` when non-zero.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true).
   // Alice is the only intended swapper.
3. Pool admin calls setAllowedToSwap(pool, router, true).
   // Necessary so Alice can use the standard router.
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ..., recipient: bob})
   // router calls pool.swap(); msg.sender inside pool = router.
5. SwapAllowlistExtension.beforeSwap receives sender = router.
   allowedSwapper[pool][router] == true  →  check passes.
6. Bob's swap executes successfully on the restricted pool.
   // Allowlist is completely bypassed.
```
