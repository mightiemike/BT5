### Title
`Clearinghouse::burnNlp` Doesn't Decrement `unlockedBalanceSum` After Burning, Allowing Users to Burn Locked NLP — (File: `core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngineState.sol`)

---

### Summary

`Clearinghouse::burnNlp` enforces that only unlocked NLP can be burned by checking `spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount`. However, after the burn executes via `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount)`, the `unlockedBalanceSum` field inside the user's `NlpLockedBalanceQueue` is never decremented. This leaves a stale, inflated unlock counter that can be exploited on a subsequent `burnNlp` call to burn NLP that is still within its lock period.

---

### Finding Description

The NLP system uses a per-user `NlpLockedBalanceQueue` (stored in `nlpLockedBalanceQueues` in `SpotEngineState`) to enforce a time-based lock on minted NLP. When the lock expires, `tryUnlockNlpBalance` moves eligible entries into `queue.unlockedBalanceSum`: [1](#0-0) 

`burnNlp` in `Clearinghouse.sol` gates the burn on this counter: [2](#0-1) 

The actual balance reduction is performed by `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount)`, which routes through `_updateBalanceNormalized`. That function only modifies `totalDepositsNormalized` / `totalBorrowsNormalized` on the `State` struct — it has no access to, and makes no modification to, `NlpLockedBalanceQueue.unlockedBalanceSum`: [3](#0-2) 

`_setBalanceAndUpdateBitmap` similarly only writes to `balances[productId][subaccount]` and the product bitmap: [4](#0-3) 

There is no other code path in the visible contract surface that decrements `unlockedBalanceSum`. After a successful burn, the counter retains its pre-burn value.

---

### Impact Explanation

An attacker can bypass the NLP lock period entirely:

1. Mint 100 NLP at time T=0 (lock expires at T=30 days).
2. At T=30 days, `tryUnlockNlpBalance` runs (called internally by `getNlpUnlockedBalance`), setting `unlockedBalanceSum = 100`.
3. Burn 100 NLP — passes the unlock check; NLP balance drops to 0; `unlockedBalanceSum` remains 100.
4. Immediately mint 100 fresh NLP (locked until T=60 days).
5. Call `burnNlp` for 100 NLP — the check `unlockedBalanceSum (100) >= 100` passes despite the NLP being locked; the burn succeeds.

The attacker exits a locked NLP position 30 days early. In practice this allows front-running of adverse pool events (e.g., large socialized losses), undermining the economic purpose of the lock. The `getBalance(NLP_PRODUCT_ID, sender).amount >= 0` post-check does not prevent this because the user holds a real (but locked) NLP balance. [5](#0-4) 

---

### Likelihood Explanation

Any NLP holder who has previously burned their unlocked NLP and then re-minted can trigger this. No privileged access is required. The entry path is the standard `BurnNlp` transaction type processed through `EndpointTx::processTransactionImpl` → `Clearinghouse::burnNlp`. [6](#0-5) 

---

### Recommendation

Decrement `unlockedBalanceSum` by `nlpAmount` inside `burnNlp` (or inside the NLP-specific `updateBalance` path in `SpotEngine`) after the burn is applied, mirroring the pattern used in `tryUnlockNlpBalance`:

```diff
// In Clearinghouse::burnNlp, after spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount):
+ spotEngine.decrementNlpUnlockedBalance(txn.sender, nlpAmount);
```

Alternatively, `getNlpUnlockedBalance` should return `min(unlockedBalanceSum, actualNlpBalance)` so the check is always bounded by the real balance, preventing the stale counter from granting excess unlock credit.

---

### Proof of Concept

```
T=0:   user calls mintNlp(100)
         → NlpLockedBalanceQueue entry added: {amount:100, unlockedAt: T+30d}
         → NLP balance = 100

T=30d: user calls burnNlp(100)
         → getNlpUnlockedBalance triggers tryUnlockNlpBalance
         → unlockedBalanceSum = 100
         → check: 100 >= 100 ✓
         → updateBalance(NLP_PRODUCT_ID, user, -100)
         → NLP balance = 0
         → unlockedBalanceSum = 100  ← NOT decremented

T=30d: user calls mintNlp(100)
         → new NlpLockedBalanceQueue entry: {amount:100, unlockedAt: T+60d}
         → NLP balance = 100

T=30d: user calls burnNlp(100)
         → getNlpUnlockedBalance: unlockedBalanceSum = 100 (stale)
         → check: 100 >= 100 ✓  ← passes despite NLP being locked until T+60d
         → updateBalance(NLP_PRODUCT_ID, user, -100)
         → NLP balance = 0
         → Lock period bypassed by 30 days
``` [7](#0-6) [8](#0-7)

        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;

        }
    }

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }
