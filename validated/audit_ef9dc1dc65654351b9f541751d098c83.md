### Title
CEI Violation in `withdrawCollateral` Enables Reentrancy Double-Withdrawal — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer to the recipient **before** decrementing the sender's on-chain balance in `SpotEngine`. An attacker controlling a recipient contract (e.g., via an ERC777 `tokensReceived` hook) can re-enter `Endpoint.executeSlowModeTransaction` during the transfer, processing a pre-queued slow-mode withdrawal against the still-undecremented balance, and receive double the entitled tokens.

---

### Finding Description

`withdrawCollateral` in `Clearinghouse.sol` violates the Checks-Effects-Interactions pattern. The function first calls `handleWithdrawTransfer`, which chains two external calls — a token transfer to `WithdrawPool` and then `submitWithdrawal`, which in turn transfers tokens to the final recipient. Only **after** all of this does the function call `spotEngine.updateBalance` to decrement the sender's balance.

```
withdrawCollateral (Clearinghouse.sol:391)
  └─ handleWithdrawTransfer (line 408)          ← EXTERNAL CALL FIRST
       ├─ token.safeTransfer(withdrawPool, ...)  ← tokens leave Clearinghouse
       └─ withdrawPool.submitWithdrawal(...)
            └─ token.safeTransfer(to, amount)   ← tokens reach recipient ← CALLBACK HERE
  └─ spotEngine.updateBalance(...)              ← STATE UPDATE AFTER
  └─ require(getHealth(...) >= 0)               ← HEALTH CHECK AFTER
```

The balance decrement and health check happen **after** the recipient has already received tokens.

`Endpoint.executeSlowModeTransaction` is a public, permissionless function with no reentrancy guard:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
```

It reads `slowModeConfig` from storage, processes the next queued transaction, and writes back. Because `slowModeConfig` is not updated in storage until after `_executeSlowModeTransaction` returns, a reentrant call reads the **pre-execution** state.

---

### Impact Explanation

An attacker can withdraw more collateral than they deposited, draining the protocol's token reserves. Concretely:

1. Attacker deposits `X` tokens.
2. Attacker submits a slow-mode `WithdrawCollateral` transaction for `X` tokens and waits for the 3-day delay to expire.
3. The sequencer (or the attacker via `executeSlowModeTransaction`) processes a withdrawal of `X` tokens for the attacker's subaccount. This triggers `withdrawCollateral` → `handleWithdrawTransfer` → `submitWithdrawal` → `token.safeTransfer(attackerContract, X)`.
4. The ERC777 `tokensReceived` hook fires on `attackerContract`. At this moment, `spotEngine` still shows balance = `X` (not yet decremented).
5. Inside the hook, the attacker calls `executeSlowModeTransaction()`. This processes the pre-queued slow-mode withdrawal. The balance check passes (balance is still `X`), and the attacker receives another `X` tokens.
6. The hook returns. The outer `withdrawCollateral` continues, decrements balance by `X` (to 0), and the health check passes.
7. Net result: attacker received `2X` tokens while only depositing `X`.

This can be repeated across multiple subaccounts or token products to drain the entire pool.

---

### Likelihood Explanation

The attack requires the collateral token to support transfer callbacks (ERC777 or a custom ERC20 with hooks). The protocol is designed to support multiple spot products, and any future listing of such a token would make this immediately exploitable by any depositor. No privileged access is required — `executeSlowModeTransaction` is callable by anyone, and the slow-mode queue is user-writable via `submitSlowModeTransaction`. The 3-day delay is a setup cost, not a mitigation.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern in `withdrawCollateral`: call `spotEngine.updateBalance` and perform the health check **before** calling `handleWithdrawTransfer`. Alternatively, add a `nonReentrant` modifier (from OpenZeppelin's `ReentrancyGuard`) to `withdrawCollateral` and `executeSlowModeTransaction`.
