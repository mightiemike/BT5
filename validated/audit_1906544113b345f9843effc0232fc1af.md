Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates access solely on `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary caller-supplied `owner` with no restriction on who the caller is, any non-allowlisted address can bypass the allowlist by specifying any allowlisted address as `owner`. The deposit allowlist — the sole on-chain mechanism for restricting liquidity provision — is rendered completely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first parameter but leaves it unnamed, silently discarding it. The allowlist check is performed only on `owner`: [2](#0-1) 

The guard evaluates `allowedDepositor[msg.sender][owner]`, not `allowedDepositor[msg.sender][sender]`. An attacker (address B, not on the allowlist) calls `pool.addLiquidity(owner=A, ...)` where A is an allowlisted address. The extension sees `owner=A`, finds it allowed, and passes. B pays the tokens via the liquidity callback; A receives the position they never requested.

`addLiquidity` imposes no restriction on who may call it with an arbitrary `owner`: [3](#0-2) 

Note that `removeLiquidity` correctly enforces `msg.sender == owner`: [4](#0-3) 

This asymmetry confirms the add-liquidity path is the only unguarded entry point.

## Impact Explanation

The `DepositAllowlistExtension` is the sole on-chain mechanism for restricting who may add liquidity to a pool (e.g., KYC-gated, institutional, or protocol-controlled pools). With this bug, the restriction is completely ineffective: any address can deposit into a restricted pool by naming any allowlisted address as `owner`. The allowlist invariant — that only approved addresses can add liquidity — is broken. Concrete consequences include non-allowlisted addresses injecting liquidity into restricted pools, and attackers griefing allowlisted LPs by front-running their deposits to force unfavorable bin compositions. This constitutes a broken core pool access-control mechanism causing direct loss of the pool admin's access-control intent and potential fund impact to legitimate LPs.

## Likelihood Explanation

The bypass requires only a public call to `pool.addLiquidity` with `owner` set to any allowlisted address. The `allowedDepositor` mapping is public, so any observer can enumerate allowlisted addresses. No privileged access, no special token behavior, no oracle manipulation is required. Any address that can call the pool can exploit this unconditionally and repeatably. [5](#0-4) 

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) rather than `owner` (the position recipient):

```solidity
// current (broken)

    external view override returns (bytes4)
