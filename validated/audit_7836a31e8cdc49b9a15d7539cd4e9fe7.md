### Title
`verify_transaction_inclusion` Missing Coinbase Merkle Proof Validation Enables 64-Byte Transaction Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still publicly callable despite being deprecated, and it lacks the coinbase merkle proof check that `verify_transaction_inclusion_v2` requires. Any unprivileged NEAR caller can invoke v1 directly, bypassing the coinbase validation entirely and exploiting the well-documented 64-byte transaction Merkle proof forgery vulnerability.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring a coinbase merkle proof at index 0, which constrains the Merkle tree structure and prevents an internal node from being passed off as a leaf transaction.

`verify_transaction_inclusion_v2` adds the coinbase validation before delegating to v1: [2](#0-1) 

The coinbase check itself: [3](#0-2) 

The `#[deprecated]` Rust attribute only emits a compiler warning for Rust callers. It does **not** restrict external NEAR contract calls or off-chain callers from invoking `verify_transaction_inclusion` directly via the WASM ABI. The function remains fully reachable with no access control beyond `#[pause]`.

This checks that the computed root matches the stored block's `merkle_root`, but it does not verify that the `tx_id` is a leaf node (a real transaction) rather than an internal 64-byte node.

---

### Impact Explanation

Any recipient contract or application that calls `verify_transaction_inclusion` can be deceived into accepting a forged transaction inclusion proof. An attacker constructs a 64-byte internal Merkle tree node from a real main-chain block and provides a merkle proof that makes it appear to be a valid transaction. The v1 function returns `true`. Any protocol that releases funds or takes irreversible on-chain actions based on this return value suffers a direct loss of funds.

---

### Likelihood Explanation

High. The function is callable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond the contract not being paused. The 64-byte forgery technique is publicly documented and requires no special mining capability — only knowledge of a real block's Merkle tree structure, which is public Bitcoin data.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add the same coinbase merkle proof check that `verify_transaction_inclusion_v2` performs before computing the transaction merkle root. Marking a function `#[deprecated]` is insufficient to prevent external callers from exploiting it.

---

### Proof of Concept

1. Attacker identifies a real Bitcoin block `B` whose hash is recorded in `mainchain_header_to_height` (i.e., it is on the current main chain).
2. Attacker inspects block `B`'s Merkle tree and selects an internal node `N` (the concatenation of two 32-byte child hashes = 64 bytes raw, 32-byte hash).
3. Attacker constructs a merkle proof `P` such that `compute_root_from_merkle_proof(N, index, P) == B.merkle_root`.
4. Attacker calls `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index = index`, `merkle_proof = P`, `confirmations = 1`.
5. The function passes all checks: block is in main chain, confirmations are sufficient, merkle proof computes to the correct root.
6. The function returns `true`, falsely certifying that `N` is a valid transaction included in block `B`.
7. A downstream protocol consuming this result (e.g., a bridge or escrow contract) releases funds based on the forged proof.

The missing step — present in `verify_transaction_inclusion_v2` at lines 358–365 but absent from `verify_transaction_inclusion` — is the exact analog of the missing `updatePoints` modifier in the external report: one sibling function performs the critical validation; the other does not, and the unprotected one remains reachable by any unprivileged caller.

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
