### Title
`verify_transaction_inclusion` Accepts Internal Merkle Tree Nodes as Valid Transaction Proofs, Enabling Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The public function `verify_transaction_inclusion` validates a Merkle inclusion proof by computing the root from the supplied path and comparing it to the stored block header's `merkle_root`. It does **not** verify that the supplied `tx_id` is a leaf node (i.e., a real transaction hash). An unprivileged NEAR caller can supply an internal Merkle tree node as `tx_id` together with a valid sibling path, causing the function to return `true` for a transaction that does not exist in the block. This is the well-documented 64-byte transaction Merkle proof forgery vulnerability.

---

### Finding Description

`verify_transaction_inclusion` (lines 288–323 of `contract/src/lib.rs`) performs three checks before delegating to `merkle_tools::compute_root_from_merkle_proof`:

1. Enough confirmations exist on the mainchain.
2. The target block is on the mainchain.
3. The Merkle proof is non-empty. [1](#0-0) 

It then computes:

```
compute_root_from_merkle_proof(args.tx_id, args.tx_index, &args.merkle_proof)
    == header.block_header.merkle_root
``` [2](#0-1) 

There is **no check** that `args.tx_id` is a leaf-level hash. Bitcoin's Merkle tree uses the same `double_sha256` function for both leaf nodes and internal nodes. An internal node at level 1 is `double_sha256(tx0 || tx1)` — a 64-byte input. An attacker who knows the transactions in a block can compute any internal node hash and supply it as `tx_id`, with the corresponding sibling path as `merkle_proof`. The computed root will equal the stored `merkle_root`, and the function returns `true`.

The contract's own documentation acknowledges this gap:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. [3](#0-2) 

The fixed version, `verify_transaction_inclusion_v2`, mitigates this by requiring a coinbase Merkle proof of equal length, which anchors the proof depth to the leaf level and prevents internal-node substitution. [4](#0-3) 

Despite being marked `#[deprecated]`, `verify_transaction_inclusion` remains a callable public function. Rust's `#[deprecated]` attribute emits only a compiler warning; it does not prevent invocation at runtime. [5](#0-4) 

---

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged funds, minting wrapped tokens, or confirming a cross-chain payment) can be deceived into accepting a forged proof. The attacker does not need to mine anything; all required data (block transactions, internal node hashes, sibling paths) is publicly available from the Bitcoin chain. The corrupted proof result (`true` for a non-existent transaction) is the direct output of this contract function, consumed by downstream logic.

---

### Likelihood Explanation

The function is reachable by any unprivileged NEAR caller with no special role or stake. All inputs needed to construct the attack (a mainchain block hash, its transaction list, and the derived internal node hashes) are public. The only precondition is that the target block has sufficient confirmations, which is trivially satisfiable for any block already in the mainchain. Existing integrations written before `v2` was introduced continue to call the `v1` function.

---

### Recommendation

1. **Remove** `verify_transaction_inclusion` entirely, or make it `private` / unreachable from external callers, forcing all consumers to use `verify_transaction_inclusion_v2`.
2. If backward compatibility must be preserved, add the same coinbase-proof length guard inside `verify_transaction_inclusion`:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id,
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Coinbase proof must be provided to prevent 64-byte forgery"
);
```

---

### Proof of Concept

Given a mainchain block at height H with transactions `[tx0, tx1, tx2, tx3]`:

1. Compute `internal_node = double_sha256(tx0_hash || tx1_hash)` — this is the level-1 left child.
2. Compute `sibling = double_sha256(tx2_hash || tx3_hash)` — the level-1 right child.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = internal_node`
   - `tx_block_blockhash` = the block's hash (on mainchain, with sufficient confirmations)
   - `tx_index = 0`
   - `merkle_proof = [sibling]`
4. `compute_root_from_merkle_proof(internal_node, 0, [sibling])` computes `double_sha256(internal_node || sibling)` = `merkle_root`.
5. The function returns **`true`** — confirming inclusion of a transaction that does not exist. [6](#0-5) [7](#0-6)

            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
