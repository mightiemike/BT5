### Title
`LiquidateSubaccount` Provides No Max-Price Bound for Liquidator — (`core/contracts/ClearinghouseLiq.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `LiquidateSubaccount` transaction struct contains no `maxPrice` field. The liquidation price is computed entirely at execution time from the live oracle price (`risk.priceX18`). Because the sequencer routinely interleaves `UpdatePrice` transactions with liquidation transactions, a liquidator can be charged significantly more quote than they anticipated when they signed the transaction.

---

### Finding Description

The `LiquidateSubaccount` struct in `IEndpoint.sol` is:

```solidity
struct LiquidateSubaccount {
    bytes32 sender;
    bytes32 liquidatee;
    uint32 productId;
    bool isEncodedSpread;
    int128 amount;
    uint64 nonce;
}
```

There is no `maxPriceX18` or equivalent slippage guard. [1](#0-0) 

When `liquidateSubaccountImpl` executes, it calls `getLiqPriceX18` (or `getSpreadLiqPriceX18`), which reads `risk.priceX18` — the current oracle price — at that moment:

```solidity
return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
``` [2](#0-1) 

The resulting `liquidationPayment` is then debited from the liquidator's quote balance with no comparison against any user-supplied bound:

```solidity
spotEngine.updateBalance(
    QUOTE_PRODUCT_ID,
    txn.sender,
    -v.liquidationPayment - v.liquidationFees
);
``` [3](#0-2) 

The same unchecked debit pattern applies to the perp path: [4](#0-3) 

The sequencer processes batches of transactions via `submitTransactions`. The `TransactionType` enum includes both `UpdatePrice` and `LiquidateSubaccount` as first-class transaction types: [5](#0-4) 

The `UpdatePrice` struct allows the sequencer to push a new oracle price for any product: [6](#0-5) 

Because the sequencer controls transaction ordering within a batch, an `UpdatePrice` transaction can be placed immediately before a `LiquidateSubaccount` transaction. The liquidator signed their transaction against the oracle price at signing time, but execution uses the post-update price. No on-chain check enforces that the execution price is within any bound the liquidator would have accepted.

---

### Impact Explanation

The liquidator's subaccount is debited `liquidationPayment + liquidationFees` computed from the updated (higher) oracle price. The excess debit is a direct, unrecoverable asset loss from the liquidator's quote balance. For large positions or volatile assets, the difference between the signed-time price and the execution-time price can be substantial. [7](#0-6) 

---

### Likelihood Explanation

`UpdatePrice` transactions are routine sequencer operations — they happen on every price tick. The sequencer naturally orders price updates before processing dependent transactions. No malicious intent is required: any liquidation transaction that arrives at the sequencer while a price update is pending will be executed at the new price. On EVM chains with frequent reorgs (L2s, sidechains), the signed transaction can also be replayed at a different block state with a higher oracle price. [8](#0-7) 

---

### Recommendation

Add a `maxPriceX18` field to `LiquidateSubaccount`. In `_handleLiquidationPayment`, after computing `v.liquidationPriceX18`, assert:

```solidity
require(
    v.liquidationPriceX18 <= txn.maxPriceX18,
    ERR_PRICE_TOO_HIGH
);
```

This gives the liquidator a signed, EIP-712-committed upper bound on the price they will pay, regardless of oracle updates that occur between signing and execution. [1](#0-0) 

---

### Proof of Concept

1. Oracle price for product X is 1000 USDC. Liquidator signs `LiquidateSubaccount { productId: X, amount: 100, ... }` expecting to pay ~`100 * 1000 * (1 - penalty)` quote.
2. Sequencer receives the liquidation transaction. Before including it, the sequencer includes `UpdatePrice { productId: X, priceX18: 1200e18 }` in the same batch (routine price update).
3. `liquidateSubaccountImpl` executes. `getLiqPriceX18` reads `risk.priceX18 = 1200e18`.
4. `liquidationPayment = 1200 * 100 * (1 - penalty)` — 20% more than the liquidator intended.
5. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -liquidationPayment - liquidationFees)` debits the excess from the liquidator's balance with no revert.
6. The liquidator has no recourse; the signed struct contained no price bound. [9](#0-8) [10](#0-9)

    function getLiqPriceX18(uint32 productId, int128 amount)
        internal
        returns (int128, int128)
    {
        RiskHelper.Risk memory risk = IProductEngine(productToEngine[productId])
            .getRisk(productId);
        int128 penaltyX18 = (RiskHelper._getWeightX18(
            risk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) / 5;
        if (penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_NON_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_NON_SPREAD_LIQ_PENALTY_X18;
            }
        }
        return (risk.priceX18.mul(ONE + penaltyX18), risk.priceX18);
    }

            ? address(0)
            : address(productToEngine[txn.productId]);

            uint32 spotId = txn.productId & 0xFFFF;
            uint32 perpId = txn.productId >> 16;

                v.oraclePriceX18,

            v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);

            v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
                .mul(LIQUIDATION_FEE_FRACTION)
                .mul(txn.amount);

            // transfer spot at the calculated liquidation price
            spotEngine.updateBalance(spotId, txn.liquidatee, -txn.amount);
            spotEngine.updateBalance(
                QUOTE_PRODUCT_ID,

                v.liquidationPayment
            );

                .mul(LIQUIDATION_FEE_FRACTION)
                .mul(txn.amount);

            spotEngine.updateBalance(
