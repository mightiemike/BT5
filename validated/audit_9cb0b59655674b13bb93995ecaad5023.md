### Title
SwapAllowlistExtension gates the router address instead of the end-user, allowing any unprivileged caller to bypass the swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The allowlist therefore gates the router's address, not the individual user's address. A pool admin cannot simultaneously allow allowlisted users to use the router and block non-allowlisted users from using the router: if the router is allowlisted, every user on-chain can bypass the allowlist.

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension = the pool): [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap()` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same pattern holds for `exactInput` (all hops), `exactOutputSingle`, and `exactOutput` (all recursive hops): [5](#0-4) [6](#0-5) 

Because the router is a single shared public contract, the allowlist entry `allowedSwapper[pool][router]` is either present or absent for all users simultaneously. The pool admin faces an inescapable dilemma:

| Router allowlisted? | Allowlisted users via router | Non-allowlisted users via router |
|---|---|---|
| No | Blocked (broken UX) | Blocked |
| Yes | Allowed | **Also allowed — bypass** |

There is no configuration that allows only allowlisted users to swap through the router while blocking others.

### Impact Explanation

Any user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router function) targeting a pool that has `SwapAllowlistExtension` configured. If the pool admin has added the router to the allowlist (the only way to let allowlisted users use the router), every address on-chain can swap in the restricted pool. The access-control invariant the pool admin intended — "only approved addresses may swap" — is fully defeated. This constitutes an admin-boundary break via an unprivileged public path.

### Likelihood Explanation

The router is a public, permissionless contract. No special role, token, or prior state is required. Any user who discovers the pool uses an allowlist can immediately route through the router to bypass it. Likelihood is high whenever the router is allowlisted (which is the only practical configuration for allowlisted users who want router access).

### Recommendation

Pass the economically meaningful actor — the end user — as `sender` to the extension, not the immediate `msg.sender` of `pool.swap()`. One approach: the router stores the originating user in transient storage (it already uses transient storage for callback context via `_setNextCallbackContext`) and the pool reads it from a trusted router registry, or the pool accepts an explicit `originator` parameter that the router fills with `msg.sender` before calling `pool.swap()`. Alternatively, `SwapAllowlistExtension` could be redesigned to gate on the payer identity surfaced through the callback rather than the `sender` hook argument.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required so that allowlisted user Alice can use the router.
3. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` — pool's `msg.sender` = router.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Bob's swap executes successfully despite not being on the allowlist. [7](#0-6) [8](#0-7)
