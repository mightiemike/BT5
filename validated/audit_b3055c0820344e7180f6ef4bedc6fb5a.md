### Title
Treasury Reward Inserted Before Validator-Count Guard Causes `minted_amount` / `total_supply` Divergence — (`File: chain/epoch-manager/src/reward_calculator.rs`)

### Summary

In `RewardCalculator::calculate_reward`, the protocol treasury reward is unconditionally inserted into the result map **before** the `num_validators == 0` early-return guard. When that guard fires, the function returns `Balance::ZERO` as `minted_amount`, but the treasury entry is already in the reward map. The runtime will credit the treasury account with `epoch_protocol_treasury` tokens that are never reflected in `EpochInfo::minted_amount`, causing `total_supply` in block headers to be permanently understated by exactly `epoch_protocol_treasury` yoctoNEAR.

### Finding Description

`RewardCalculator::calculate_reward` in `chain/epoch-manager/src/reward_calculator.rs` computes and inserts the treasury reward into `res` at line 84, then immediately checks whether there are any validators at line 85–86:

```rust
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);  // line 84
if num_validators == 0 {
    return (res, Balance::ZERO);   // line 86 — minted_amount is ZERO, but res already has treasury
}
``` [1](#0-0) 

When `num_validators == 0` the function returns `(res, Balance::ZERO)`. The caller in `finalize_epoch` destructures this as `(validator_reward, minted_amount)`: [2](#0-1) 

`minted_amount` (`Balance::ZERO`) is stored in `EpochInfo::minted_amount`: [3](#0-2) 

`EpochInfo::minted_amount` is then read by the chain to verify and update `total_supply` in block headers: [4](#0-3) 

Meanwhile, `validator_reward` (which contains `{treasury_account: epoch_protocol_treasury}`) is stored in `EpochInfo::validator_reward` and the runtime credits the treasury account with `epoch_protocol_treasury` tokens at the epoch boundary. Those tokens are minted into existence but `minted_amount` says zero, so `total_supply` is never incremented to match.

The normal path initialises `epoch_actual_reward` to `epoch_protocol_treasury` at line 90, accumulates validator rewards, and returns the correct total: [5](#0-4) 

The early-return path skips this initialisation entirely, returning `Balance::ZERO` instead.

### Impact Explanation

`total_supply` recorded in every subsequent block header is understated by exactly `epoch_protocol_treasury` yoctoNEAR. Because `verify_total_supply_checked` uses `minted_amount` from `EpochInfo` to validate the header field, any node that independently recomputes `total_supply` from the credited treasury balance will observe a mismatch. This breaks the deterministic `total_supply` invariant that all nodes must agree on for consensus, and can cause `InvalidTotalSupply` errors or silent supply-accounting drift depending on which side of the check is trusted.

### Likelihood Explanation

The `num_validators == 0` branch is reached when `validator_block_chunk_stats` is empty after kicked-out validators are removed in `finalize_epoch`. The exception rule in `compute_validators_to_reward_and_kickout` prevents all validators from being simultaneously kicked out in normal operation, so this path is not triggered on mainnet today. However, the guard exists precisely because the code considers the case reachable (e.g., genesis epoch, a chain bootstrapped with no validators, or a future protocol change that alters kickout logic). The bug is latent and deterministic: any execution that reaches line 86 with a non-zero `max_inflation_rate` and non-zero `protocol_reward_rate` will produce the divergence.

### Recommendation

Move the treasury reward insertion to **after** the `num_validators == 0` guard, or include `epoch_protocol_treasury` in the returned `minted_amount` even in the early-return path:

```rust
// Option A: return the treasury amount as minted even when no validators
if num_validators == 0 {
    return (res, epoch_protocol_treasury);
}

// Option B: defer insertion until after the guard
if num_validators == 0 {
    return (HashMap::new(), Balance::ZERO);
}
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
```

Either fix ensures `minted_amount` equals the sum of all tokens actually credited, preserving the `total_supply` invariant.

### Proof of Concept

Given `max_inflation_rate = 1/40`, `protocol_reward_rate = 1/10`, `total_supply = 1_000_000_000 yN`, `epoch_duration = 1s`:

1. `epoch_total_reward` = non-zero value T  
2. `epoch_protocol_treasury` = T/10 (non-zero)  
3. `res.insert(treasury, T/10)` — treasury entry written  
4. `num_validators == 0` → `return (res, Balance::ZERO)`  
5. Caller stores `minted_amount = 0` in `EpochInfo`  
6. Runtime credits treasury with T/10 yoctoNEAR  
7. `total_supply` in next block header = old `total_supply + 0` = understated by T/10  
8. Any node recomputing supply from account balances sees `total_supply + T/10`, diverging from the header value [6](#0-5)
