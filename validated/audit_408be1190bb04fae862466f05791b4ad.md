Looking at the external report's vulnerability class — **flag/parameter validation mismatch** where an attacker-controlled input satisfies one check but the actual operation uses a different stored value — I need to find an analog in `clvm_rs` where an attacker-controlled opcode form bypasses a flag gate that a parallel dispatch path correctly enforces.

The key finding is in `src/chia_dialect.rs`.

### Title
Secp256k1/r1 verify operations reachable via 4-byte opcode aliases, bypassing `ENABLE_SECP_OPS` flag gate — (File: `src/chia_dialect.rs`)

---

### Summary
`ChiaDialect::op` in `src/chia_dialect.rs` contains two separate dispatch paths for secp signature-verification operators. The 1-byte opcode aliases (opcodes `64` and `65`) are correctly gated behind `flags.contains(ClvmFlags::ENABLE_SECP_OPS)`. The 4-byte opcode aliases (`0x13d61f00` for `secp256k1_verify`, `0x1c3a8f00` for `secp256r1_verify`) are dispatched to the same underlying functions with **no flag check at all**. An attacker who controls CLVM program bytes can select the 4-byte opcode form to invoke live secp signature verification in any execution context — including `MEMPOOL_MODE` — where `ENABLE_SECP_OPS` is absent and the 1-byte form would be rejected.

---

### Finding Description

`ChiaDialect::op` has two structurally separate dispatch branches:

**Branch 1 — 4-byte opcodes (lines 157–182):** entered when `op_len == 4`. The raw 4-byte integer is matched and dispatched unconditionally:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
    0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** entered when `op_len == 1`. The same two functions are dispatched only when the flag is present:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag itself is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`MEMPOOL_MODE` — the stricter execution context used for transaction validation — does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are computed from the secp cost formula (multiplier × cost-function-0), so they are not reserved or rejected by `NO_UNKNOWN_OPS` — they are explicitly matched and dispatched. The result is that an attacker-controlled CLVM program can invoke `op_secp256k1_verify` or `op_secp256r1_verify` in full in any dialect that lacks `ENABLE_SECP_OPS`, including `MEMPOOL_MODE`.

The analog to the external report is exact: the attacker supplies an alternative "form" of the same operation (4-byte opcode instead of 1-byte alias) that satisfies the branch-entry condition (`op_len == 4`) but skips the flag validation that the intended path enforces. The actual executed function — `op_secp256k1_verify` / `op_secp256r1_verify` — is identical in both paths. [5](#0-4) 

---

### Impact Explanation

`ENABLE_SECP_OPS` is the softfork gate for secp signature verification. When a Chia node runs programs in `MEMPOOL_MODE` (or any pre-activation consensus mode that omits `ENABLE_SECP_OPS`), the intent is that secp operations are unavailable. The 4-byte bypass makes this gate ineffective:

- A CLVM coin puzzle or spend bundle can embed opcode `0x13d61f00` or `0x1c3a8f00` and have the secp verification execute and succeed/fail on its cryptographic merits, regardless of whether the softfork has activated.
- If mempool nodes and consensus nodes differ in whether they include `ENABLE_SECP_OPS`, the 4-byte form produces the same live verification result on both, masking the divergence — but any node that treats the 4-byte form as a no-op (e.g., an older implementation or a node that routes it through `op_unknown`) will disagree on program outcome, producing a consensus split.
- The secp operations themselves (`op_secp256k1_verify`, `op_secp256r1_verify`) raise a hard `EvalErr::Secp256Failed` on invalid signatures, so a program that passes the 4-byte form with a bad signature will abort rather than silently succeed — but a program with a valid signature will pass through the gate that was supposed to be closed. [6](#0-5) 

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the CLVM program bytes are the attack surface. Substituting the 4-byte opcode `0x13d61f00` for the 1-byte opcode `0x40` (64) requires only choosing a different byte sequence in the program. No privileged access, social engineering, or dependency compromise is needed. Any caller that submits CLVM programs to a node running without `ENABLE_SECP_OPS` can trigger this.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch, mirroring the 1-byte branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is absent, the 4-byte secp opcodes should fall through to `unknown_operator`, which charges cost and returns nil — consistent with how all other ungated unknown operators behave.

### Proof of Concept

A minimal CLVM program that triggers `op_secp256k1_verify` in `MEMPOOL_MODE` (which lacks `ENABLE_SECP_OPS`):

```
; opcode atom = 0x13d61f00  (4 bytes, secp256k1_verify cost formula)
; args = (pubkey msg sig)
(0x13d61f00 pubkey msg sig)
```

In Rust pseudocode:

```rust
let mut a = Allocator::new();
// Build opcode atom: 4 bytes = 0x13d61f00
let op = a.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
// Build args: valid secp256k1 pubkey, 32-byte msg hash, valid sig
let args = /* ... */;
let program = a.new_pair(op, args).unwrap();

// Run under MEMPOOL_MODE — ENABLE_SECP_OPS is NOT set
let dialect = ChiaDialect::new(MEMPOOL_MODE);
let result = run_program(&mut a, &dialect, program, a.nil(), u64::MAX);
// result is Ok(...) if signature is valid — secp verification ran despite missing flag
// Opcode 64 (1-byte form) with the same args would return Err(Unimplemented)
```

The 1-byte form (opcode `64`) under `MEMPOOL_MODE` falls to the `_` arm and returns `EvalErr::Unimplemented`. The 4-byte form `0x13d61f00` under the same flags executes the full cryptographic verification, confirming the bypass. [7](#0-6) [8](#0-7)

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
