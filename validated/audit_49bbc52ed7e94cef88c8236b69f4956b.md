Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the allowlist check is performed against the router address rather than the originating user. This creates an irreconcilable dilemma: allowlisting the router defeats per-user gating for all users, while not allowlisting it permanently breaks router-mediated swaps on curated pools.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly: [3](#0-2) 

This makes the router the `sender` passed to the extension. The check becomes `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

The allowlist state is keyed by `(pool, swapper)`: [5](#0-4) 

No mechanism exists to unwrap the originating user from `extensionData` or any other field — the extension only inspects `sender`.

## Impact Explanation

On any pool configured with `SwapAllowlistExtension` (KYC-gated, institutional, or otherwise curated), an unprivileged user blocked by direct `pool.swap()` can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The pool admin's intent to restrict swaps to vetted counterparties is completely defeated. LPs who deposited under the assumption that only allowlisted addresses could trade are exposed to unvetted counterparties executing swaps at oracle-derived prices — a direct loss of LP asset integrity and an admin-boundary break via an unprivileged path through a supported public contract. Severity: **High**.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported swap entrypoint. Any blocked user can trivially re-route through it in a single transaction with no special privileges, flash loans, or unusual token behavior. The bypass is deterministic and repeatable.

## Recommendation

The extension must check the actual originating user, not the intermediary. The cleanest fix is to have `MetricOmmSimpleRouter` encode the real `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify that address — gated by a trusted-router check (e.g., a factory-registered router list or `onlyRouter` modifier). Alternatively, a `tx.origin` fallback is acceptable only in strictly EOA-only contexts, but is not recommended for general use.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for any router swap to work
  - bob is NOT allowlisted

Step 1 — Direct call (guard works):
  bob calls pool.swap() directly
  → _beforeSwap(bob, ...)
  → allowedSwapper[pool][bob] == false → REVERT ✓

Step 2 — Router bypass (guard fails):
  bob calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  → router calls pool.swap()
  → _beforeSwap(router, ...)
  → allowedSwapper[pool][router] == true → PASSES ✗
  → bob's swap executes at oracle price on the curated pool
```
