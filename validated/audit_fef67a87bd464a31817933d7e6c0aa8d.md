### Title
`verify_transaction_inclusion` Returns `true` for Internal Merkle Tree Nodes Without Verifying Leaf-Level Transaction Existence — (File: `contract/src/lib.rs`)

---

### Summary

The publicly callable `verify_transaction_inclusion` function confirms Bitcoin transaction inclusion by computing a Merkle root from caller-supplied inputs and comparing it to the stored block header's `merkle_root`. It does not verify that the supplied `tx_id` is an actual leaf-level transaction hash. Any unprivileged NEAR caller can supply an internal Merkle tree node as `tx_id` with a correspondingly shorter proof path, causing the function to return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` at [1](#0-0)  accepts caller-controlled `tx_id`, `tx_index`, and `merkle_proof`, then delegates to `compute_root_from_merkle_proof` and compares the result to the block's stored `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` simply hashes the supplied `tx_id` up the tree using the supplied proof siblings, with no check that `tx_id` is a leaf node: [3](#0-2) 

Because the function only checks that the computed root matches the stored root, an attacker can supply any **internal Merkle node** as `tx_id` with a proof path that is one level shorter than a real leaf proof. The computed root will still equal the block's `merkle_root`, so the function returns `true`.

The contract itself documents this broken invariant explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [4](#0-3) 

Critically, Rust's `#[deprecated]` attribute is a **compiler warning only** — it imposes no runtime restriction. The function remains fully callable by any NEAR account. [5](#0-4) 

---

### Impact Explanation

Consumer contracts — bridges, atomic swaps, cross-chain lending protocols — that call `verify_transaction_inclusion` to gate fund releases or state transitions can be deceived into accepting a forged proof. An attacker can "prove" inclusion of a Bitcoin transaction that never existed, potentially:

- Unlocking bridge funds by faking a Bitcoin deposit transaction
- Satisfying a cross-chain lending repayment condition with a fabricated tx
- Manipulating any protocol that treats a `true` return as a confirmed on-chain event

The corrupted value is the **proof result** returned to the consumer: a `true` that asserts a canonical Bitcoin transaction exists when it does not. [1](#0-0) 

---

### Likelihood Explanation

The attack requires no privileged role, no leaked keys, and no social engineering. All inputs needed — the block's Merkle tree structure and internal node hashes — are publicly available from any Bitcoin full node. The attacker only needs to:

1. Identify a block already accepted into the contract's mainchain.
2. Compute an internal Merkle node from that block's public transaction list.
3. Call `verify_transaction_inclusion` directly on NEAR with the crafted inputs.

This is a low-effort, high-impact path for any unprivileged caller. Likelihood is **High** given the function is unrestricted and the required data is public.

---

### Recommendation

**Remove or gate the deprecated `verify_transaction_inclusion` function.** Since `verify_transaction_inclusion_v2` already exists as the correct replacement (it enforces a coinbase proof of equal length, anchoring the tree depth and ruling out internal-node forgeries), the deprecated function should either be:

- Removed entirely from the public interface, or
- Guarded with an access-control role so only trusted callers can invoke it.

Additionally, consider adding an explicit runtime `panic` or `require!(false, "use verify_transaction_inclusion_v2")` inside the deprecated function body to enforce the deprecation at runtime, not just at compile time. [6](#0-5) 

---

### Proof of Concept

Consider a block with four transactions `T0, T1, T2, T3` whose Merkle tree is:

```
merkle_root = hash(N01, N23)
  N01 = hash(T0, T1)
  N23 = hash(T2, T3)
```

An attacker calls `verify_transaction_inclusion` with:

| Field | Value |
|---|---|
| `tx_id` | `N01` (internal node — **not a real transaction**) |
| `tx_block_blockhash` | hash of the target block (already in mainchain) |
| `tx_index` | `0` |
| `merkle_proof` | `[N23]` (one sibling — one level shorter than a real leaf proof) |
| `confirmations` | `1` |

Execution trace through `compute_root_from_merkle_proof`:

```
current_hash = N01
current_position = 0

iteration 1: proof_hash = N23
  position 0 is even → current_hash = hash(N01, N23) = merkle_root
  current_position = 0

return merkle_root  ← equals header.block_header.merkle_root ✓
``` [7](#0-6) 

The function returns `true`. No transaction `N01` exists in the Bitcoin block; the attacker has forged a valid-looking inclusion proof for a phantom transaction. Any consumer contract gating on this result is compromised.

        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
