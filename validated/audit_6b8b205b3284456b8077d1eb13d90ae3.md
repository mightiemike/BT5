Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any Caller to Bypass Swap Allowlist via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` of `pool.swap()`, so the allowlist checks the router's address rather than the end user's address. If the router is allowlisted — a natural admin action to permit allowlisted users to use the standard periphery — every unprivileged user can bypass the per-user gate entirely by calling through the router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct). `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`, which encodes `sender` directly from `MetricOmmPool.swap`'s `msg.sender` (L231). When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` with itself as `msg.sender` and passes `""` as `callbackData` and the user-supplied `params.extensionData` — no end-user identity is encoded anywhere the extension can read. The extension therefore receives `sender = router address`. The check becomes `allowedSwapper[pool][router]`: if the router is allowlisted, every caller through the router passes, regardless of their own allowlist status. No existing guard in the pool, router, or extension decodes or validates the originating user from `extensionData`.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` in restrictive mode (KYC-gated, institutional, or counterparty-restricted) relies on the allowlist as its primary swap access control. If the pool admin allowlists the router to permit allowlisted users to use the standard periphery, the allowlist is rendered ineffective: any unprivileged user can call `router.exactInputSingle()` or `router.exactInput()` and the hook passes because it sees the router address. Unauthorized users can execute swaps the pool was designed to prohibit, extracting value from LP positions at prices the pool admin intended to reserve for specific counterparties only. This constitutes a direct loss of LP principal and broken core pool functionality (access-controlled swap flow).

## Likelihood Explanation
The trigger requires the pool admin to allowlist `MetricOmmSimpleRouter`. This is a natural, non-malicious configuration: any admin who wants allowlisted users to be able to use the standard router must allowlist the router address. Once that configuration is in place, any unprivileged user can exploit it with a single public call to `router.exactInputSingle()`. No special privileges, flash loans, or unusual tokens are required. The attack is repeatable on every block.

## Recommendation
The `beforeSwap` hook must gate the economically relevant actor — the end user — not the immediate caller of `pool.swap()`. The cleanest fix is to define a convention where the router encodes the originating `msg.sender` in `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a known router. Concretely: (1) `MetricOmmSimpleRouter` encodes `msg.sender` into `params.extensionData` before calling `pool.swap()`; (2) `SwapAllowlistExtension.beforeSwap` decodes the originating user from `extensionData` when `sender` is a registered router address, and checks that decoded address against `allowedSwapper`. Alternatively, remove the router from the allowlist and require allowlisted users to call `pool.swap()` directly, but this breaks router usability for restricted pools.

## Proof of Concept
```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - allowAllSwappers[pool] = false
  - allowedSwapper[pool][alice] = true        // alice is the only allowed swapper
  - allowedSwapper[pool][router] = true       // admin allowlists router so alice can use it

Attack:
  1. Bob (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams({
           pool: pool,
           recipient: bob,
           zeroForOne: true,
           amountIn: X,
           extensionData: ""
       }))
  2. Router calls pool.swap(recipient=bob, ..., extensionData="") with msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks: allowedSwapper[pool][router] == true → passes
  5. Bob's swap executes against the restricted pool
  6. Bob extracts value the pool admin intended to reserve for alice only

Foundry test sketch:
  vm.prank(bob);
  router.exactInputSingle(params);  // must not revert — demonstrates bypass
```
