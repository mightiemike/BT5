### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Merkle Proof Guard, Enabling 64-Byte Transaction Proof Forgery - (`contract/src/lib.rs`)

### Summary

The contract exposes two proof-verification entry points: the deprecated `verify_transaction_inclusion` (v1) and the current `verify_transaction_inclusion_v2`. The coinbase Merkle proof validation — the only guard against the 64-byte transaction Merkle forgery attack — was added exclusively to v2. The v1 function remains a live, unprivileged, publicly callable NEAR method. Any caller can bypass the security check entirely by invoking v1 directly, obtaining a `true` result for a transaction that does not exist in the claimed block.

### Finding Description

`verify_transaction_inclusion` (v1) computes a Merkle root from a caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block's `merkle_root`: [1](#0-0) 

It performs no validation that `tx_id` is a leaf node (an actual transaction) rather than an internal Merkle tree node. The 64-byte transaction attack (documented at the URL cited in the code itself) exploits this: an attacker crafts a 64-byte payload whose double-SHA256 equals an internal node `N` of the real Merkle tree, then supplies a proof starting from `N`'s parent. The computation reaches the correct root, and the function returns `true`.

`verify_transaction_inclusion_v2` closes this gap by first verifying a coinbase Merkle proof anchored at index 0: [2](#0-1) 

This coinbase anchor prevents the attack because the attacker cannot simultaneously satisfy both the coinbase proof (which must start from a real leaf) and the forged proof. However, v1 is still decorated only with `#[deprecated]` and `#[pause]` — it is not gated, removed, or restricted to any role: [3](#0-2) 

The desynchronization is exact: the security check exists in v2 but is absent in v1, and both paths are reachable by any unprivileged caller. This mirrors the external report's pattern where a critical guard was present for one code path (borrowers) but absent for another (suppliers), creating an exploitable asymmetry.

### Impact Explanation

Any downstream NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., releasing funds, minting tokens, unlocking a bridge transfer) can be deceived into accepting a forged proof. The attacker does not need to mine a real Bitcoin block or control any privileged role — they only need to identify an internal Merkle node in any already-accepted block and craft the corresponding 64-byte payload. The corrupted value is the proof result itself: the function returns `true` for a transaction that was never included in the block.

### Likelihood Explanation

The attack is realistic. The 64-byte transaction forgery technique is publicly documented (referenced in the contract's own comments at line 268). The v1 function is a standard NEAR public method reachable by any account. No staking, role, or special permission is required. The only precondition is that the target block is already in the contract's `headers_pool` and `mainchain_header_to_height`, which is the normal operating state. [4](#0-3) 

### Recommendation

Remove or fully gate `verify_transaction_inclusion`. The simplest safe option is to add a `#[private]` attribute so it is no longer externally callable, since `verify_transaction_inclusion_v2` already delegates to it internally after performing the coinbase check. Alternatively, inline the coinbase validation directly into v1 and remove the separate v2 entry point to eliminate the asymmetry entirely.

### Proof of Concept

1. Identify any block `B` already accepted into the contract's mainchain (e.g., via `get_block_hash_by_height`).
2. Obtain the Merkle tree of `B`. Select any internal node `N` at depth `d` (with `d ≥ 1`). `N` is the double-SHA256 of two 32-byte child hashes — a 64-byte input.
3. Craft a 64-byte payload `P` such that `double_sha256(P) == N`. Supply `P` as `tx_id`.
4. Construct `merkle_proof` as the sibling path from `N`'s parent up to the root (length `depth_of_tree - d`), and set `tx_index` to the position of `N` among its siblings at depth `d`.
5. Call `verify_transaction_inclusion` with `tx_block_blockhash = hash(B)`, the crafted `tx_id`, `tx_index`, `merkle_proof`, and `confirmations = 1`.
6. The function computes `compute_root_from_merkle_proof(P_hash, tx_index, proof)` which equals the real `merkle_root` of `B`, and returns `true` — despite `P` never having been a transaction in `B`. [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
