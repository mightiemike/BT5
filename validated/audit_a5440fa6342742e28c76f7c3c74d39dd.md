### Title
`verify_transaction_inclusion_v2` Permanently Reverts for Valid Single-Transaction Block Proofs Due to Inconsistent Empty-Proof Check in Delegated `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` permits an empty `merkle_proof` (when `coinbase_merkle_proof` is also empty), passes its own coinbase validation, then delegates to `verify_transaction_inclusion`, which unconditionally rejects any empty `merkle_proof`. For any Bitcoin block containing exactly one transaction (a coinbase-only / empty block), the mathematically correct Merkle proof is an empty vector. This means `verify_transaction_inclusion_v2` will always panic for such blocks, even though the block is validly stored and confirmed in the contract.

---

### Finding Description

**Step 1 — Block submission (less strict):**

`submit_blocks` accepts any valid block header and stores it in `headers_pool` and the mainchain maps with no restriction on the number of transactions in the block. [1](#0-0) 

**Step 2 — Proof verification (inconsistent strictness):**

`verify_transaction_inclusion_v2` first checks that both proof vectors have equal length — this allows both to be empty: [2](#0-1) 

It then validates the coinbase proof. For a single-transaction block, `merkle_root == coinbase_tx_id`, so `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id` directly — this check passes: [3](#0-2) 

It then delegates to `verify_transaction_inclusion` via `args.into()`, which passes `merkle_proof` unchanged: [4](#0-3) 

Inside `verify_transaction_inclusion`, there is an unconditional guard that panics on any empty proof: [5](#0-4) 

This guard has no awareness that an empty proof is mathematically correct when the block contains exactly one transaction. The result is that `verify_transaction_inclusion_v2` always reverts for single-transaction blocks, despite having already passed its own coinbase validation.

---

### Impact Explanation

Any downstream dApp or bridge contract calling `verify_transaction_inclusion_v2` to verify a transaction in a coinbase-only Bitcoin block will always receive a revert. The block is legitimately stored and confirmed in the contract; the proof is mathematically valid; yet the verification function permanently returns a panic. This breaks the core SPV guarantee the contract is designed to provide for an entire class of real Bitcoin blocks.

---

### Likelihood Explanation

Coinbase-only (empty) blocks are a routine occurrence on the Bitcoin mainnet — miners frequently produce them during low-mempool periods or to minimize orphan risk. Any relayer that submits such a block to the contract creates a permanently unverifiable block. Any caller (unprivileged NEAR account) can trigger the revert simply by supplying the correct, empty proof for the coinbase transaction.

---

### Recommendation

Remove or relax the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard in `verify_transaction_inclusion`. An empty proof is valid when the transaction hash equals the block's `merkle_root` (i.e., single-transaction blocks). The guard should be replaced with a check that the computed root matches the stored `merkle_root`, which already happens on the next line and handles the empty-proof case correctly via `compute_root_from_merkle_proof`. [6](#0-5) 

---

### Proof of Concept

1. Relayer submits a valid Bitcoin block whose `merkle_root` equals a single coinbase transaction hash (a coinbase-only block). The block is accepted and stored in `headers_pool`.
2. After sufficient confirmations, an unprivileged NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase tx hash
   - `tx_block_blockhash` = the stored block hash
   - `tx_index` = 0
   - `merkle_proof` = `[]` (correct empty proof for a single-tx block)
   - `coinbase_tx_id` = same coinbase tx hash
   - `coinbase_merkle_proof` = `[]` (correct empty proof)
   - `confirmations` = 1
3. The length check `0 == 0` passes. [2](#0-1) 
4. The coinbase check `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id == merkle_root` — passes. [3](#0-2) 
5. `verify_transaction_inclusion` is called with `merkle_proof = []`. [7](#0-6) 
6. The call panics: `"Merkle proof is empty"` — despite the proof being mathematically correct. [5](#0-4)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {

        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
