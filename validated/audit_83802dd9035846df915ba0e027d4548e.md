### Title
Missing Liquidator Health Check in Finalization Path Allows Bad Debt Creation — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`liquidateSubaccountImpl` returns early after `_finalizeSubaccount` without ever executing the liquidator solvency check that guards the normal liquidation path. When the finalization loop settles positive `vQuoteBalance` from the liquidatee against the liquidator, the liquidator's QUOTE balance can be driven arbitrarily negative with no revert, creating protocol bad debt that is later socialized onto depositors.

---

### Finding Description

**Entry point and early-return bypass**

`liquidateSubaccountImpl` is called with `txn.productId == type(uint32).max`:

```
liquidateSubaccountImpl
  └─ _finalizeSubaccount(txn, ...) → true
       └─ return   ← exits here, no health check on txn.sender
``` [1](#0-0) 

The liquidator health check exists only inside `_handleLiquidationPayment`, which is never reached when finalization succeeds: [2](#0-1) 

**Positive vQuoteBalance settlement loop**

Inside `_finalizeSubaccount`, every perp with `balance.vQuoteBalance > 0` is settled against the liquidator unconditionally: [3](#0-2) 

`_settlePnlAgainstLiquidator` deducts the full PnL from the liquidator's QUOTE balance: [4](#0-3) 

**`spotEngine.updateBalance` has no floor**

`SpotEngine.updateBalance` applies the delta unconditionally — there is no minimum-balance guard: [5](#0-4) 

**Health calculation confirms the precondition is reachable**

`PerpEngineState._getBalance` returns `(balance.amount, balance.vQuoteBalance)`: [6](#0-5) 

`BaseEngine._calculateProductHealth` adds `quoteAmount` (= `vQuoteBalance`) unconditionally, then adds the weighted position value only when `amount != 0`: [7](#0-6) 

For a closed perp (`amount == 0`, `vQuoteBalance == +V`), the health contribution is exactly `+V`. Therefore a liquidatee can be under maintenance while holding positive `vQuoteBalance`:

```
QUOTE balance:  -2000
Perp A (amount=0, vQuoteBalance=+1000)
Maintenance health = -2000 + 1000 = -1000 < 0  ✓ liquidatable
```

`_finalizeSubaccount` requires `balance.amount == 0` for all perps before proceeding: [8](#0-7) 

This state is therefore a valid finalization target.

---

### Impact Explanation

After finalization with an undercapitalized liquidator (QUOTE = 0):

| Account | Before | After |
|---|---|---|
| Liquidatee QUOTE | −2000 | −1000 (covered by insurance/socialization) |
| Liquidatee vQuoteBalance | +1000 | 0 |
| Liquidator QUOTE | 0 | **−1000 (bad debt, no check)** |

The liquidator's −1000 QUOTE bad debt persists in state. When the liquidator is subsequently liquidated, `SpotEngine.socializeSubaccount` dilutes the `cumulativeDepositsMultiplierX18` for all depositors: [9](#0-8) 

Net protocol loss doubles compared to a solvent liquidator: insurance covers the liquidatee's residual liability **and** the liquidator's bad debt is socialized separately. This is a direct, concrete loss of depositor funds.

---

### Likelihood Explanation

The precondition (liquidatee with closed perp positions, positive `vQuoteBalance`, and net negative health) is a normal end-state of a partially-liquidated account. An attacker controlling both the liquidatee and a fresh zero-balance liquidator subaccount can trigger this deterministically. No oracle manipulation, governance capture, or privileged access is required — only a standard `liquidateSubaccountImpl` call routed through the endpoint.

---

### Recommendation

Add a liquidator health check immediately after `_finalizeSubaccount` returns `true`, mirroring the guard already present in `_handleLiquidationPayment`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    require(
        txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
        ERR_SUBACCT_HEALTH
    );
    if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) { ... }
    return;
}
``` [1](#0-0)

### Proof of Concept

```
Setup:
  liquidatee.QUOTE          = -2000e18
  liquidatee.perpA.amount   = 0          // position already closed
  liquidatee.perpA.vQuote   = +1000e18   // profitable closed position
  → maintenance health = -1000e18 < 0   ✓

  liquidator.QUOTE          = 0          // undercapitalized

Attack:
  endpoint.submitSlowModeTransaction(
      LiquidateSubaccount{
          sender:    liquidator,
          liquidatee: liquidatee,
          productId: type(uint32).max,   // triggers _finalizeSubaccount
          ...
      }
  )

State after:
  liquidatee.perpA.vQuote   = 0
  liquidatee.QUOTE          = -1000e18   → covered by insurance/socialize
  liquidator.QUOTE          = -1000e18   → BAD DEBT, no revert

Assert:
  isUnderInitial(liquidator) == true     // invariant broken, no check performed
  liquidator bad debt socialized → depositor loss
```
