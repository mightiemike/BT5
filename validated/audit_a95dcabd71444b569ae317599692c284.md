### Title
Repeated `createIsolatedSubaccount` Calls with Distinct Order Nonces Drain Parent Quote Balance into Existing Isolated Subaccount — (`core/contracts/OffchainExchange.sol`)

### Summary
`createIsolatedSubaccount` deduplicates margin transfers by order digest, but the digest includes the order nonce. A trader can sign N orders with distinct nonces for the same `productId`, each carrying margin `M`. Each call passes the per-digest guard, finds the already-existing isolated subaccount for that product, and unconditionally transfers `M` from the parent's `QUOTE_PRODUCT_ID` balance. The parent loses `N×M` while the isolated subaccount accumulates `N×M`, with no health check on the parent after each transfer.

### Finding Description

The deduplication guard at line 1009 is keyed on the EIP-712 order digest: [1](#0-0) 

Because `getDigest` commits to `order.nonce` (among other fields), two orders with different nonces produce different digests and both pass this guard independently. [2](#0-1) 

After the guard, the function searches for an existing isolated subaccount for `txn.productId`: [3](#0-2) 

When one is found, `newIsolatedSubaccount` is set to the existing subaccount. The new-subaccount creation block is skipped. Execution then falls through unconditionally to the margin transfer: [4](#0-3) 

There is no guard here that checks whether the isolated subaccount was newly created or already existed. Every distinct-digest call transfers `M` from the parent.

At the `Endpoint` layer, `CreateIsolatedSubaccount` transactions are dispatched without calling `validateSignedTx`, so there is no monotonic transaction-level nonce enforcement: [5](#0-4) 

Compare this to `TransferQuote`, `WithdrawCollateral`, `LinkSigner`, etc., which all call `validateSignedTx` before acting.

### Impact Explanation

After `N` calls with distinct nonces, the parent's `QUOTE_PRODUCT_ID` balance is reduced by `N×M` while the isolated subaccount holds `N×M`. If the parent also holds open perp positions, those positions are now undercollateralized. The parent becomes eligible for liquidation. The attacker retains the `N×M` in the isolated subaccount (returned to them on close). This enables:

- **Bad debt creation**: Parent's perp positions are liquidated with no remaining collateral; the protocol absorbs the shortfall.
- **Unauthorized collateral extraction**: The attacker effectively removes collateral from under open positions without closing them, bypassing the health check that would normally prevent a direct withdrawal.

There is no post-transfer health check on the parent inside `createIsolatedSubaccount`, so the parent's health is never verified after the drain.

### Likelihood Explanation

The attack requires only that the attacker control a parent subaccount with an open perp position and sufficient quote balance. They sign `N` valid isolated orders with distinct nonces (all off-chain, no on-chain cost beyond sequencer submission), submit them as `CreateIsolatedSubaccount` transactions, and the drain executes atomically within the sequencer's batch. No privileged access, oracle manipulation, or external dependency is required.

### Recommendation

Add a guard that skips the margin transfer when the isolated subaccount for `txn.productId` already exists (i.e., when `newIsolatedSubaccount` was found in the loop rather than freshly created). Concretely, track whether the subaccount was newly created and only transfer margin in that branch:

```solidity
bool isNew = (newIsolatedSubaccount == bytes32(0));
// ... existing creation logic ...
digestToSubaccount[digest] = newIsolatedSubaccount;

if (isNew && margin > 0) {          // <-- guard on isNew
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
```

Additionally, add a parent health check after any margin transfer, consistent with how `TransferQuote` is handled.
