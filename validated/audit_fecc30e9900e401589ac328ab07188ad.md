### Title
Fee-on-Transfer Token Accounting Discrepancy in `depositCollateralWithReferral()` Overcredits Subaccount Collateral — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral()` in `Endpoint.sol` records the caller-supplied `amount` in the slow-mode transaction queue rather than the actual tokens received. When a fee-on-transfer token is used, the Clearinghouse is credited with less than `amount`, but the subaccount is credited with the full `amount`, breaking the invariant that the Clearinghouse's token balance must equal the sum of all subaccount balances.

---

### Finding Description

The deposit flow in Nado is a two-step process:

**Step 1 — On-chain token transfer** (`Endpoint.sol`, `depositCollateralWithReferral`):

`handleDepositTransfer` is called with the user-supplied `amount`:

```
safeTransferFrom(token, msg.sender, amount);          // user → Endpoint: receives amount - fee1
safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse: sends amount
```

With a fee-on-transfer token, the Endpoint receives `amount - fee1` in the first leg. The second leg then attempts to forward the full `amount` to the Clearinghouse. If the Endpoint holds no prior balance of that token, this second transfer reverts (DoS). If the Endpoint does hold accumulated balance (e.g., from slow-mode fees paid in the same token), the second transfer succeeds by silently consuming that surplus — and the Clearinghouse receives `amount - fee2` (after the second transfer fee).

**Step 2 — Slow-mode queue entry** (`Endpoint.sol`, lines 152–165):

Regardless of what was actually received, the slow-mode transaction is enqueued with the original caller-supplied `amount`:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    ...
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(
            DepositCollateral({
                sender: subaccount,
                productId: productId,
                amount: amount          // ← user-supplied, not actual received
            })
        )
    )
});
```

**Step 3 — Sequencer processes the slow-mode tx** (`Clearinghouse.sol`, lines 193–209):

`depositCollateral` blindly credits `txn.amount` (the original user-supplied value) to the subaccount:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

The Clearinghouse credits `amount` worth of collateral, but only holds `amount - fee1 - fee2` in actual tokens.

---

### Impact Explanation

The Clearinghouse's real token balance becomes less than the sum of all subaccount credited balances. This is the exact invariant broken in the reference report: `balance[before] != balance[after]`. Concretely:

- A depositor is credited more collateral than they deposited.
- They can use the phantom collateral to open leveraged positions or withdraw, extracting value that does not exist in the contract.
- As more users deposit with fee-on-transfer tokens, the deficit compounds, eventually making the protocol insolvent and preventing legitimate withdrawals.

---

### Likelihood Explanation

Medium. The protocol uses whitelisted tokens, which limits exposure. However, USDT (a commonly whitelisted stablecoin) has a dormant fee mechanism that can be activated by the USDT issuer. If USDT is whitelisted and its fee is enabled, every deposit via `depositCollateral` or `depositCollateralWithReferral` — callable by any unprivileged user — triggers the discrepancy. The `DirectDepositV1.creditDeposit()` path also feeds into the same vulnerable `depositCollateralWithReferral` entrypoint.

---

### Recommendation

Measure the actual received amount by comparing balances before and after the transfer, and record that value in the slow-mode transaction instead of the caller-supplied `amount`:

```solidity
uint256 before = token.balanceOf(address(clearinghouse));
handleDepositTransfer(token, msg.sender, uint256(amount));
uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
// encode actualReceived (not amount) into the slow-mode tx
```

Alternatively, enforce that whitelisted tokens never have transfer fees, and add an on-chain check that reverts if the received amount differs from the expected amount.

---

### Proof of Concept

1. Assume a fee-on-transfer token `FOT` (1% fee per transfer) is whitelisted and the Endpoint holds a prior balance of `FOT` (e.g., from slow-mode fees).
2. Attacker calls `depositCollateral(subaccountName, productId, 1000e18)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(FOT, attacker, 1000e18)` → Endpoint receives `990e18`.
   - `safeTransferTo(FOT, clearinghouse, 1000e18)` → Clearinghouse receives `990e18` (another 1% fee), consuming `1000e18` from Endpoint (draining `10e18` of Endpoint's prior balance).
4. Slow-mode tx is enqueued with `amount = 1000e18`.
5. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits `1000e18` to the attacker's subaccount.
6. Clearinghouse actually holds only `980.1e18` (`990e18 - 1%`), but the subaccount shows `1000e18`.
7. Attacker withdraws `1000e18`, extracting `~19.9e18` more than deposited, at the expense of other users' collateral.
