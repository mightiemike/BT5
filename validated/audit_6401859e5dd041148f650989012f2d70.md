Audit Report

## Title
`SwapAllowlistExtension` Per-User Allowlist Bypassed via Router When Router Is Allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of the pool's `swap()` call. When `MetricOmmSimpleRouter` mediates a swap, the pool sees the router as `msg.sender`, not the originating EOA. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user — including explicitly disallowed ones — full swap access, completely neutralizing the per-user allowlist.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the first argument forwarded by the pool. In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, recipient, ...)` — so `sender` in the extension equals `msg.sender` of the pool's `swap()` invocation.

When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly. At that point `msg.sender` seen by the pool is the **router contract**, not the originating EOA. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The same substitution occurs across all router paths:
- `exactInput` multi-hop: router calls each pool directly with `msg.sender = router`
- `exactOutputSingle`: router calls pool directly
- `exactOutput` recursive callback (`_exactOutputIterateCallback`): the router calls each intermediate pool with `msg.sender = router`

A pool admin who wants to allow router-mediated swaps will naturally call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether that caller is individually disallowed. The `SwapAllowlistExtension` NatSpec and interface provide no indication that allowlisting the router opens access to all users.

## Impact Explanation
The per-user swap allowlist — the core access control invariant of `SwapAllowlistExtension` — is completely neutralized for all router paths once the router is allowlisted. Any user, including those explicitly set to `allowedSwapper[pool][user] = false`, can call any router entry point and execute swaps on a pool designed to be restricted (e.g., KYC-gated, institutional, or whitelist-only). This enables unauthorized extraction of value from curated pools, front-running of allowlisted participants, and disruption of pools with restricted audiences. The broken invariant is `allowedSwapper[pool][user]` — a disallowed user's flag is never evaluated when the router intermediates.

## Likelihood Explanation
The scenario is highly likely. Any pool admin deploying a curated pool with `SwapAllowlistExtension` who also wants users to access the pool via the standard router will allowlist the router — this is the only way to enable router-mediated swaps. The bypass requires no special privileges, no flash loans, and no unusual token behavior. Any EOA can call `exactInputSingle` on the router. The admin action that triggers the vulnerability (`setAllowedToSwap(pool, router, true)`) is the expected and natural configuration step.

## Recommendation
The extension must gate the economically relevant actor, not the direct pool caller. Viable approaches:

1. **Extension-data forwarding**: The router encodes the originating user's address in `extensionData`; the extension decodes and checks it. Requires a trusted router or signed attestation.
2. **Recipient-based check**: Check `recipient` (the address receiving output tokens) in addition to or instead of `sender`, since the recipient is the economic beneficiary and is harder to spoof.
3. **Documentation + separate extension**: If the extension is intentionally caller-only, document that allowlisting the router opens access to all users, and provide a separate `RouterSwapAllowlistExtension` that decodes user identity from `extensionData`.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  pool admin: setAllowedToSwap(pool, router, true)   // to enable router swaps
  pool admin: setAllowedToSwap(pool, alice, false)   // alice is explicitly disallowed

Attack:
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool, recipient: alice, ...})
    → router calls pool.swap(alice, zeroForOne, amount, limit, "", extensionData)
    → pool calls _beforeSwap(msg.sender=router, alice, ...)
    → extension checks allowedSwapper[pool][router] == true  ✓
    → swap executes; alice receives output tokens

Result:
  alice swaps successfully on a pool that was supposed to block her.
  allowedSwapper[pool][alice] == false is never evaluated.
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension` configured in `beforeSwapOrder`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` and `setAllowedToSwap(pool, alice, false)`.
3. `vm.prank(alice)` → call `router.exactInputSingle(...)` targeting the pool.
4. Assert swap succeeds (no `NotAllowedToSwap` revert) and alice receives output tokens.
5. Confirm that calling `pool.swap(...)` directly as alice reverts with `NotAllowedToSwap`.
