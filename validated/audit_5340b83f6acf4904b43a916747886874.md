### Title
Sequencer-Supplied `oraclePriceX18` Is Excluded from User's EIP-712 Signature, Enabling NLP Pool Drain — (`core/contracts/Verifier.sol`)

---

### Summary

The `oraclePriceX18` field in `SignedMintNlp` and `SignedBurnNlp` is **not covered by the user's EIP-712 signature**. The sequencer — a restricted but not fully-trusted actor analogous to the Arrakis executor — can substitute any oracle price when submitting these transactions. By minting NLP tokens at an artificially low price and burning them at an artificially high price, the sequencer can drain the NLP pool.

---

### Finding Description

In `Verifier.computeDigest()`, the digest for a `MintNlp` transaction is computed over only three fields: `sender`, `quoteAmount`, and `nonce`: [1](#0-0) 

The `MINT_NLP_SIGNATURE` type string confirms this: [2](#0-1) 

The `oraclePriceX18` field present in `SignedMintNlp` is entirely absent from the digest: [3](#0-2) 

The same omission applies to `BurnNlp`: [4](#0-3) [5](#0-4) 

When the sequencer submits a `MintNlp` transaction via `submitTransactionsChecked`, it passes `oraclePriceX18` as a free parameter alongside the user's valid signature: [6](#0-5) 

Inside `Clearinghouse.mintNlp()`, the NLP amount issued is computed directly from the sequencer-supplied price: [7](#0-6) 

A sequencer-supplied `oraclePriceX18 = 1` (the minimum non-zero value) causes `nlpAmount = quoteAmount / 1 = quoteAmount`, issuing orders of magnitude more NLP tokens than the deposited quote warrants.

Symmetrically, in `Clearinghouse.burnNlp()`, the quote returned is: [8](#0-7) 

A sequencer-supplied `oraclePriceX18` equal to the true fair price at burn time returns the full fair value for the inflated NLP balance, extracting far more quote than was deposited.

---

### Impact Explanation

The NLP pool (`N_ACCOUNT`) acts as the protocol's liquidity reserve and counterparty to all traders. A malicious or compromised sequencer can drain it in a single mint-then-burn cycle:

1. User signs `MintNlp{quoteAmount=Q, nonce=N}` — signature covers only `(sender, Q, N)`.
2. Sequencer submits with `oraclePriceX18 = 1`. User receives `Q` NLP tokens instead of `Q / fair_price`.
3. User signs `BurnNlp{nlpAmount=Q, nonce=N+1}`.
4. Sequencer submits with `oraclePriceX18 = fair_price`. User receives `Q * fair_price` quote.
5. Net theft: `Q * fair_price − Q` quote extracted from the NLP pool per cycle.

The 4-day lock period (`NLP_LOCK_PERIOD`) does not prevent this attack because the sequencer controls when the burn transaction is processed — it can simply delay submission until the lock expires. [9](#0-8) 

---

### Likelihood Explanation

The sequencer is a restricted but not fully-trusted role (analogous to the Arrakis executor). It is a single centralized key that, if compromised or operated maliciously, has no on-chain constraint preventing it from supplying an arbitrary `oraclePriceX18`. No bounds check, no on-chain oracle comparison, and no user-signed commitment to the price exist anywhere in the execution path. [10](#0-9) 

---

### Recommendation

Include `oraclePriceX18` (and `nlpPoolRebalanceX18`) in the EIP-712 type hash and digest for both `MintNlp` and `BurnNlp`. The user must sign over the price they expect to receive, so the sequencer cannot substitute a different value without invalidating the signature.

```
MintNlp(bytes32 sender,uint128 quoteAmount,int128 oraclePriceX18,uint64 nonce)
BurnNlp(bytes32 sender,uint128 nlpAmount,int128 oraclePriceX18,uint64 nonce)
``` [11](#0-10) 

---

### Proof of Concept

```
// Precondition: NLP pool holds 1,000,000 USDC worth of quote.
// fair_price = 1e18 (1 NLP = 1 USDC for simplicity)

// Step 1: attacker deposits 1 USDC and signs MintNlp{quoteAmount=1e6, nonce=0}
// Step 2: sequencer submits with oraclePriceX18 = 1
//   → nlpAmount = 1e6 / 1 = 1e6 NLP tokens minted  (should be 1)

// Step 3: 4-day lock expires (sequencer delays BurnNlp submission)

// Step 4: attacker signs BurnNlp{nlpAmount=1e6, nonce=1}
// Step 5: sequencer submits with oraclePriceX18 = 1e18
//   → quoteAmount = 1e6 * 1e18 / 1e18 = 1e6 USDC returned
//   → burnFee = max(1, 1e6/1000) = 1000 USDC
//   → net received ≈ 999,000 USDC

// Attacker deposited 1 USDC, received ~999,000 USDC.
// NLP pool drained by ~999,000 USDC.
```

    function submitTransactionsChecked(
        uint64 idx,
        bytes[] calldata transactions,
        bytes32 e,
        bytes32 s,
        uint8 signerBitmask
    ) external {
        validateSubmissionIdx(idx);
        require(msg.sender == sequencer);
        // TODO: if one of these transactions fails this means the sequencer is in an error state
        // we should probably record this, and engage some sort of recovery mode

        bytes32 digest = keccak256(abi.encode(idx));
        for (uint256 i = 0; i < transactions.length; ++i) {
            digest = keccak256(abi.encodePacked(digest, transactions[i]));
        }
        verifier.requireValidSignature(digest, e, s, signerBitmask);

        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            nSubmissions += 1;
        }
    }
