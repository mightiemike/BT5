### Title
Instant Risk Parameter Updates Allow Immediate Liquidation of Users Without Warning — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.spotUpdateRisk` and `ContractOwner.perpUpdateRisk` update collateral risk weights directly on the engine contracts with no time delay, bypassing the slow-mode queue that governs all other sensitive admin operations. An owner can instantly reduce maintenance weights for any product, rendering previously healthy subaccounts immediately liquidatable before users have any opportunity to add collateral or close positions.

---

### Finding Description

The Nado protocol enforces a 3-day delay (`SLOW_MODE_TX_DELAY`) on sensitive administrative operations by routing them through the slow-mode queue via `_submitSlowModeTransaction`. This applies to fee updates, NLP pool changes, insurance withdrawals, and product delisting.

However, `spotUpdateRisk` and `perpUpdateRisk` in `ContractOwner.sol` call the engine contracts directly, completely bypassing this queue:

```solidity
// ContractOwner.sol lines 453–465
function spotUpdateRisk(
    uint32 productId,
    RiskHelper.RiskStore memory riskStore
) external onlyOwner {
    spotEngine.updateRisk(productId, riskStore);   // direct call, no delay
}

function perpUpdateRisk(
    uint32 productId,
    RiskHelper.RiskStore memory riskStore
) external onlyOwner {
    perpEngine.updateRisk(productId, riskStore);   // direct call, no delay
}
```

The `RiskStore` struct controls the collateral weights used in every health calculation:

```solidity
// RiskHelper.sol lines 14–24
struct RiskStore {
    int32 longWeightInitial;
    int32 shortWeightInitial;
    int32 longWeightMaintenance;
    int32 shortWeightMaintenance;
    int128 priceX18;
}
```

These weights are applied in `_getWeightX18` to compute both initial and maintenance health for every subaccount holding that product. Reducing `longWeightMaintenance` or `shortWeightMaintenance` for any product immediately lowers the maintenance health of all subaccounts with positions in that product.

Contrast with operations that do use the slow-mode queue:

```solidity
// ContractOwner.sol lines 418–430 — fee updates go through slow mode
_submitSlowModeTransaction(
    IEndpoint.TransactionType.UpdateTierFeeRates,
    abi.encode(_txn)
);
```

```solidity
// ContractOwner.sol lines 309–323 — NLP pool updates go through slow mode
_submitSlowModeTransaction(
    IEndpoint.TransactionType.UpdateNlpPool,
    abi.encode(_txn)
);
```

Risk weight updates are architecturally more impactful than fee or pool changes, yet they are the only category that skips the delay entirely.

---

### Impact Explanation

The `longWeightMaintenance` and `shortWeightMaintenance` fields directly determine whether a subaccount is subject to liquidation. If the owner calls `spotUpdateRisk` or `perpUpdateRisk` to reduce these weights to near zero for a widely-held collateral product, every subaccount holding that product can transition from healthy to liquidatable in the same block. Liquidators can then immediately call `LiquidateSubaccount` in the next sequencer batch, seizing collateral at a discount. Users suffer direct, irreversible asset loss with zero opportunity to respond.

The corrupted state delta is: `RiskHelper.Risk.longWeightMaintenanceX18` / `shortWeightMaintenanceX18` for the targeted product, which propagates into every health check performed by the Clearinghouse for all subaccounts holding that product.

---

### Likelihood Explanation

This requires the owner (multisig) to act maliciously or be compromised. While a multisig raises the bar, the protocol's own design acknowledges this threat class by routing all other sensitive operations through the slow-mode queue. The absence of a delay here is an inconsistency in the protocol's own trust model, not an external dependency. The attack requires no user interaction, no special market conditions, and executes atomically in a single transaction.

---

### Recommendation

Route `spotUpdateRisk` and `perpUpdateRisk` through `_submitSlowModeTransaction` using a new `UpdateRisk` transaction type, consistent with how `UpdateTierFeeRates`, `UpdateNlpPool`, and other sensitive operations are handled. This gives users the existing 3-day window to observe the pending change and exit positions before the new weights take effect.

---

### Proof of Concept

1. Owner calls `ContractOwner.spotUpdateRisk(productId, RiskStore{longWeightMaintenance: 1, ...})` — sets maintenance weight to effectively zero for a spot collateral product.
2. `spotEngine.updateRisk` executes immediately; `_risk().value[productId]` is updated in the same block.
3. In the next sequencer submission, a liquidator submits a `LiquidateSubaccount` transaction for any subaccount that held that product as collateral.
4. `clearinghouse.liquidateSubaccount` computes maintenance health using the new weight, finds the subaccount underwater, and transfers collateral to the liquidator at a discount.
5. Users whose positions were healthy under the prior weights have no on-chain recourse.

```text
    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```
