### Title
`FeeEstimation` Missing `l2_gas_consumed` Causes Authoritative-Looking Wrong Fee Breakdown in RPC Response - (File: `crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` is missing the `l2_gas_consumed` field. Both the in-code doc comment and the OpenRPC spec describe `overall_fee` as equaling `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. For V3 transactions with `AllResources` resource bounds, the actual `overall_fee` also includes an L2 gas component (`l2_gas * l2_gas_price`). The `l2_gas_price` is returned in the response but `l2_gas_consumed` is not, making the documented formula wrong and making it impossible for callers to reconstruct or verify the fee breakdown. Users who follow the documented formula to set their resource bounds will systematically underestimate the required fee and have their transactions rejected.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_price,                                          // price returned …
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // actual fee
    unit: tx_execution_output.price_unit,
})
```

`gas_vector.l2_gas` is never placed in the response. The `overall_fee` is taken from `receipt.fee`, which for V3 `AllResources` transactions is:

```
overall_fee = l1_gas * l1_gas_price + l2_gas * l2_gas_price + l1_data_gas * l1_data_gas_price
```

The struct's doc comment, however, states:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
pub overall_fee: Fee,
```

The OpenRPC spec at `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` repeats the same wrong formula:

```json
"overall_fee": {
    "description": "…equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price",
```

The `l2_gas_price` field is present in the response but `l2_gas_consumed` is absent, so callers:
1. Cannot reconstruct the fee from the returned components.
2. Cannot set `l2_gas.max_amount` in their V3 resource bounds from the estimation result.
3. If they follow the documented formula they compute `F' = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price`, which is strictly less than the actual `overall_fee` whenever L2 gas is non-zero.

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` that uses the documented formula or the returned component fields to derive resource bounds for a V3 `AllResources` transaction will produce a `max_fee` / `l2_gas.max_amount` that is too low. The blockifier's pre-validation check (`check_resources_within_bounds`) will then reject the transaction with `MaxGasAmountExceeded`. The `l2_gas_price` field returned in the response is rendered useless because its counterpart `l2_gas_consumed` is absent. This matches the impact category: **High — RPC fee estimation returns an authoritative-looking wrong value**.

### Likelihood Explanation

Every V3 transaction that consumes non-trivial L2 (Sierra) gas triggers this discrepancy. V3 with `AllResources` is the current production transaction format.
