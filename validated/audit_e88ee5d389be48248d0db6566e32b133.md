### Title
NFT Owner Who Is Not Denom Creator Cannot Burn Their Own NFT via MsgBurnNFT — (`x/nft/keeper/keeper.go`)

### Summary
`BurnNFT` enforces a conjunctive guard: the caller must be **both** the current NFT owner **and** the denom creator. After a legitimate `MsgTransferNFT`, the new owner (not the denom creator) is permanently blocked from burning via `MsgBurnNFT`. The denom creator (no longer the owner) is equally blocked. No standard transaction path exists for the owner to destroy their own asset.

### Finding Description
In `keeper.BurnNFT`:

```go
// x/nft/keeper/keeper.go:141-161
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
    ...
    nft, err := k.IsOwner(ctx, denomID, tokenID, owner)   // must be owner
    if err != nil { return err }

    _, err = k.IsDenomCreator(ctx, denomID, owner)         // must ALSO be denom creator
    if err != nil { return err }
    ...
}
```

`IsDenomCreator` is immutable — the denom creator is set at `IssueDenom` time and never changes. Once an NFT is transferred to a non-creator, neither party can satisfy both conditions simultaneously:

| Caller | `IsOwner` | `IsDenomCreator` | Result |
|---|---|---|---|
| Bob (owner, not creator) | ✓ | ✗ | `ErrUnauthorized` |
| Alice (creator, not owner) | ✗ | ✓ | `ErrUnauthorized` |

`BurnNFTUnverified` (which only checks `IsOwner`) is the correct logic for this case, but it is gated exclusively to the IBC module path and is not reachable via any `MsgServer` transaction.

### Impact Explanation
Any NFT that has been transferred away from its denom creator becomes unburnable by its owner through the standard `MsgBurnNFT` transaction. The owner's only recourse is to transfer the NFT back to the denom creator (requiring the creator's cooperation to then burn it), or to initiate an IBC transfer — an operation the owner may not have access to or that may not be available on the chain. This violates the fundamental ownership invariant that an asset owner can destroy their own asset.

### Likelihood Explanation
This is triggered by the ordinary, intended use of `MsgTransferNFT` followed by `MsgBurnNFT`. No special privileges or unusual conditions are required. Any user who receives an NFT via transfer and attempts to burn it will hit this.

### Recommendation
Remove the `IsDenomCreator` check from `BurnNFT`. Burning should require only that the caller is the current owner (`IsOwner`). The denom creator restriction is appropriate for minting and editing, but not for destruction of an asset the caller legitimately owns. Alternatively, allow the denom creator to burn any NFT in their denom regardless of current owner, as a separate privileged path.

### Proof of Concept
```go
// Keeper test (no IBC needed)

func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
