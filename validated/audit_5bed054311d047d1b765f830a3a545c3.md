### Title
Proof Forgery via Internal Merkle Node in Publicly Callable Deprecated `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains a live, unrestricted public NEAR entry point. Because `compute_root_from_merkle_proof` is a pure mathematical function that cannot distinguish a leaf-level transaction hash from an internal Merkle tree node, any unprivileged caller can supply an internal node as `tx_id` with a shortened proof path and receive `true` for a Bitcoin transaction that does not exist. The newer `verify_transaction_inclusion_v2` closes this path, but the old entry point is never removed or access-gated, so the fix is trivially bypassed.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated]` in Rust source. In Rust, `#[deprecated]` emits a compiler warning only; it has no effect on the compiled WASM binary and imposes no on-chain access restriction. The function remains `pub` and is guarded only by `#[pause]`, which is inactive by default. [1](#0-0) 

The function accepts four fully attacker-controlled fields — `tx_id`, `tx_block_blockhash`, `tx_index`, and `merkle_proof` — from any NEAR account: [2](#0-1) 

Verification is delegated entirely to `compute_root_from_merkle_proof`, which performs a pure iterative hash computation: [3](#0-2) 

This function takes whatever bytes are supplied as `transaction_hash` and hashes them up the tree. It has no mechanism to confirm that the starting value is a leaf-level transaction hash rather than an intermediate node. The contract's own docstring acknowledges this: [4](#0-3) 

The final check is a simple equality comparison against the stored `merkle_root`: [5](#0-4) 

`verify_transaction_inclusion_v2` was introduced to close this path by requiring a coinbase proof of equal length, which forces the attacker's proof to span the full tree depth: [6](#0-5) 

However, because `verify_transaction_inclusion` (v1) is never removed or access-controlled, the v2 fix is a dead letter — any caller can invoke v1 directly and bypass the coinbase check entirely.

---

### Impact Explanation

Any dApp (cross-chain bridge, payment processor, DEX settlement layer) that calls `verify_transaction_inclusion` to confirm a Bitcoin transaction before releasing assets on NEAR can be deceived into releasing those assets for a Bitcoin transaction that was never broadcast or mined. The attacker needs no privileged role, no leaked key, and no social engineering — only knowledge of a real block that has been submitted to the contract.

---

### Likelihood Explanation

- The function is publicly callable with no access control.
- The 64-byte transaction Merkle forgery technique is well-documented and has published tooling.
- The contract's own NatSpec explicitly names the vulnerability class.
- The attacker only needs one real block already in `headers_pool` (supplied by the honest relayer) to construct the forged proof.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with an access-control role that prevents external callers from invoking it. All external consumers should be directed exclusively to `verify_transaction_inclusion_v2`. A `#[deprecated]` annotation is insufficient — it must be paired with an on-chain restriction or removal.

---

### Proof of Concept

Assume a real Bitcoin block `B` with `merkle_root = R` and four transactions `[T0, T1, T2, T3]` has been submitted to the contract.

**Step 1 — Compute an internal node (public information from the block):**
```
H01 = double_sha256(T0 || T1)   // level-1 internal node
H23 = double_sha256(T2 || T3)   // level-1 internal node
R   = double_sha256(H01 || H23) // merkle root stored in the contract
```

**Step 2 — Construct the forged call:**
```
tx_id             = H01          // internal node, not a real transaction
tx_block_blockhash = hash(B)     // real block already in headers_pool
tx_index          = 0
merkle_proof      = [H23]        // one-element proof, starting from level 1
confirmations     = 1
```

**Step 3 — Trace through `compute_root_from_merkle_proof`:**
```
current_hash = H01
position 0 (even): current_hash = double_sha256(H01 || H23) = R
```

**Step 4 — Final check in `verify_transaction_inclusion`:**
```
R == header.block_header.merkle_root  →  true
```

The function returns `true` for `H01`, which is not a Bitcoin transaction. Any bridge or settlement contract that trusts this result and releases funds has been exploited. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
