Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass Per-User Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to restrict swaps to a per-pool allowlist of swapper addresses. However, `beforeSwap` checks `sender`, which is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router address (a natural configuration step to enable router-based swaps for permitted users) inadvertently opens the pool to all users, completely defeating the per-user restriction.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged as the first argument to the extension via `_callExtensionsInOrder`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
```

where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

The same pattern applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165). In all cases, the router is `msg.sender` of `pool.swap()`.

The result: the allowlist check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The check that should have blocked an unauthorized user — `allowedSwapper[pool][user]` — is never evaluated. [4](#0-3) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers) is fully bypassed by any user who routes through `MetricOmmSimpleRouter`. Once the router is allowlisted, every user routing through it is treated as the same allowlisted entity. Unauthorized users can execute swaps against restricted pools, receiving output tokens from LP positions that the pool admin explicitly intended to protect from unrestricted access. This constitutes a broken core pool functionality (allowlist enforcement) causing potential loss of LP value to unauthorized counterparties.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical periphery entry point. A pool admin deploying `SwapAllowlistExtension` to restrict individual users would naturally also need to allowlist the router so that permitted users can access the pool through the standard interface. This is the expected operational pattern, making the bypass condition a near-certain outcome of normal configuration. No special privileges or malicious setup are required — any user can call `router.exactInputSingle(...)` targeting the restricted pool.

## Recommendation
The `beforeSwap` hook should check the actual economic actor, not the intermediary caller. The cleanest options are:

1. **Pass the originating user through `extensionData`** — require the router to embed the originating `msg.sender` in `extensionData` and have the extension decode and verify it. This requires coordinated router + extension design.
2. **Check `recipient` in addition to `sender`** — require that at least one of `sender` or `recipient` is allowlisted, which prevents the router-as-proxy bypass while still allowing direct swaps.
3. **Redesign the extension interface** — have the pool pass the payer (the address that funds the swap callback) as a distinct argument, separate from the intermediary caller.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-based swaps for permitted users.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` to allowlist Alice.
4. Bob (not allowlisted) calls `router.exactInputSingle(...)` targeting the restricted pool.
5. Inside `pool.swap()`, `msg.sender` = router. The extension checks `allowedSwapper[pool][router]` → `true`. Bob's swap succeeds.
6. Bob has bypassed the per-user allowlist entirely by routing through the official periphery. The check `allowedSwapper[pool][bob]` is never evaluated.
