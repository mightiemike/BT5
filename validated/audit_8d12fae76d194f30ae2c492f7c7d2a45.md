### Title
Unverified Caller-Supplied `tx_id` Accepted as Merkle Leaf Enables SPV Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary
The publicly callable `verify_transaction_inclusion` function accepts a caller-supplied `tx_id` without verifying it is a genuine transaction hash (Merkle leaf). An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id`, causing the function to return `true` for a transaction that was never included in the block — a direct analog to the unverified-input-used-in-critical-call class.

---

### Finding Description
In `verify_transaction_inclusion`, the `args.tx_id` value is passed directly to `compute_root_from_merkle_proof` as the starting hash with no validation that it is a leaf node (i.e., a real transaction hash) rather than an internal Merkle tree node:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The underlying `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` is purely position-based: it hashes the supplied starting value up the tree using the proof path, with no distinction between leaf and internal nodes:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
``` [2](#0-1) 

Because any internal node hash from a real block's Merkle tree is a valid starting point for a shorter proof path that still reaches the correct root, an attacker can supply an internal node as `tx_id` with a truncated proof and receive a `true` result.

The only input guard on `tx_id` is the non-emptiness check on `merkle_proof`:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

This does not prevent the attack. The `tx_block_blockhash` is validated against `mainchain_header_to_height`, confirming the block is on the canonical chain, but `tx_id` itself is never validated. [4](#0-3) 

The function is explicitly deprecated with a note pointing to this exact class of vulnerability, but it remains callable:

```rust
#[deprecated(
    since = "0.5.0",
    note = "Use `verify_transaction_inclusion_v2` instead."
)]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [5](#0-4) 

The `#[pause]` attribute restricts it only when the contract is paused; when active, any unprivileged NEAR account can call it. There is no `#[trusted_relayer]` guard (unlike `submit_blocks`). [6](#0-5) 

The fixed version `verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof of equal depth, anchoring the tree height and preventing internal-node substitution:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [7](#0-6) 

`verify_transaction_inclusion` has no equivalent guard.

---

### Impact Explanation
Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` to gate an action — releasing bridged funds, minting wrapped tokens, processing cross-chain messages — can be deceived into accepting a forged Bitcoin SPV proof. The attacker does not need to mine a block or hold any privileged role; they only need to know the Merkle tree structure of any block already accepted by the light client (publicly available on-chain). The corrupted protocol value is the **proof result**: `true` is returned for a transaction that does not exist in the block.

---

### Likelihood Explanation
The function is reachable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond gas. The attack technique (the 64-byte / internal-node Merkle forgery) is well-documented and mechanically straightforward. The block data needed to construct the exploit is fully public. The only prerequisite is that the contract is not paused.

---

### Recommendation
Remove `verify_transaction_inclusion` from the public ABI entirely, or add an `env::panic_str` body that unconditionally reverts, forcing all callers to migrate to `verify_transaction_inclusion_v2`. Deprecation alone is insufficient because the function remains callable on-chain.

---

### Proof of Concept

Assume block B (on the canonical chain) has four transactions `[T0, T1, T2, T3]` with Merkle tree:

```
Root = H( H(H(T0)||H(T1)) || H(H(T2)||H(T3)) )
         \_____N_____/         \_____M_____/
```

**Attacker steps:**

1. Compute `N = H(H(T0) || H(T1))` — an internal node, not a transaction.
2. Compute `M = H(H(T2) || H(T3))`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N`
   - `tx_block_blockhash = B` (a real canonical block)
   - `tx_index = 0`
   - `merkle_proof = [M]`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, 0, [M])` computes `H(N || M) = Root` ✓
5. The function returns `true`, falsely certifying that `N` is a transaction included in block B.

The broken invariant: `verify_transaction_inclusion` must only return `true` when `tx_id` is a genuine leaf (transaction hash) in the block's Merkle tree. Because `tx_id` is accepted without leaf-node validation, this invariant is violated by any caller who supplies an internal node hash.
