### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Proof Forgery Guard Added in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase Merkle proof. However, the original `verify_transaction_inclusion` is still a fully public, unpermissioned entry point. Any NEAR caller can invoke it directly, skipping the coinbase check entirely, and obtain a `true` result for a fabricated transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces an additional guard before delegating to the old function:

```rust
// verify_transaction_inclusion_v2 (lines 347-369)
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);

#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

The coinbase proof at index 0 anchors the real tree depth, making it impossible to substitute an internal 64-byte node as a leaf. But `verify_transaction_inclusion` is still a public `#[pause]`-gated function with no access control:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

The function's own documentation acknowledges the risk:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.

The `#[deprecated]` attribute is a Rust compiler hint only. It has no effect on NEAR contract dispatch. The function remains callable by any account on-chain.

The pattern is structurally identical to the reported issue: `verify_transaction_inclusion_v2` is the "protected path" (analogous to `withdraw` with its timelock), and `verify_transaction_inclusion` is the "unprotected alternative path" (analogous to `rigidRedemption` without the timelock).

---

### Impact Explanation

An attacker can forge a Bitcoin transaction inclusion proof accepted by the contract:

1. Identify a Bitcoin block whose Merkle tree has depth ≥ 2.
2. Craft a 64-byte value that, when treated as a "transaction hash," is actually an internal Merkle node whose subtree hashes to the block's Merkle root via a valid proof path.
3. Call `verify_transaction_inclusion` directly (not v2), supplying the crafted `tx_id` and the corresponding proof path.
4. The contract returns `true` for a transaction that does not exist in the block.

Any downstream bridge or application contract that calls `verify_transaction_inclusion` to gate fund releases will accept this forged proof, enabling theft of bridged assets.

---

### Likelihood Explanation

- The function is unconditionally reachable by any unprivileged NEAR account.
- The 64-byte Merkle forgery technique is publicly documented and has known tooling.
- Downstream integrators may call the old function because it has a simpler interface (no coinbase proof required), or because they integrated before v2 existed.
- The contract itself signals the risk in its own docstring, confirming the attack surface is understood but the entry point was not closed.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it with an access-control role that prevents unprivileged callers from invoking it. A `#[private]` attribute or an explicit `require!(false, "use verify_transaction_inclusion_v2")` panic body would close the bypass path while preserving the internal delegation used by v2.

---

### Proof of Concept

**Attacker-controlled call (NEAR transaction):**

```
contract.verify_transaction_inclusion({
  tx_id:            <64-byte internal Merkle node crafted to satisfy the proof>,
  tx_block_blockhash: <hash of a real mainchain block>,
  tx_index:         <index matching the crafted node's position>,
  merkle_proof:     <proof path from the internal node to the Merkle root>,
  confirmations:    1
})
// Returns: true  (no coinbase anchor check performed)
```

**Why v2 would reject this:**

```
contract.verify_transaction_inclusion_v2({
  ...same fields...,
  coinbase_tx_id:        <must be real coinbase hash>,
  coinbase_merkle_proof: <must produce the same Merkle root at index 0>
})
// The coinbase proof at index 0 reveals the true tree depth,
// making the internal-node substitution detectable → panics with
// "Incorrect coinbase merkle proof"
```

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
