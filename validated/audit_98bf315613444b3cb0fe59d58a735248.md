### Title
`decayPerSecondE8` Change Retroactively Misapplies Decay to Historical Watermarks, Disarming the Oracle Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

When `executeOracleStopLossDecay` updates `decayPerSecondE8` mid-course, the stored per-bin watermarks (`hwmS.token0`, `hwmS.token1`) and their `lastDecayTs` timestamps are **not reset**. The new rate is then retroactively applied to the entire elapsed time since `lastDecayTs` on the next swap. This is the direct analog of the external `time_unit` bug: a configured time-unit parameter is changed mid-course while historical timestamps remain anchored to the old unit, producing incorrect calculations.

---

### Finding Description

The `_checkAndUpdateWatermarks` function computes decay lazily on every swap:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
(uint256 hwm0, bool breach0) = _applyWatermark(
    metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier
);
``` [1](#0-0) 

`decayRate` is read from the live `oracleStopLossConfig[pool_].decayPerSecondE8`: [2](#0-1) 

`executeOracleStopLossDecay` writes the new rate but **does not touch any bin's `lastDecayTs`**: [3](#0-2) 

The `_decayed` helper floors to zero whenever `ratePerSecondE8 * dt >= E8`: [4](#0-3) 

After the rate change, the first swap on any bin that has not been touched since `lastDecayTs` applies the **new** rate to the **entire** elapsed interval `dt`, not just the portion that elapsed after the rate change. If `newRate × dt ≥ 1e8`, the watermark collapses to zero. The ratchet then restores from the current live metric, effectively resetting the stop-loss guard as if no breach had ever occurred.

By contrast, `executeOracleStopLossHighWatermarks` correctly resets `lastDecayTs` to `block.timestamp` when it writes new watermarks: [5](#0-4) 

No equivalent reset exists in `executeOracleStopLossDecay`.

---

### Impact Explanation

The stop-loss guard is the primary on-chain mechanism preventing value extraction from LP bins. Disarming it allows swaps in directions that should be blocked, draining token0 or token1 from bins whose per-share value has already fallen below the configured drawdown floor. LPs suffer direct principal loss.

Concrete numbers: with `R_old = 1` (very slow decay) and `R_new = 39`, any bin whose `lastDecayTs` is more than `⌈1e8 / 39⌉ ≈ 2,564,103 seconds ≈ 29.7 days` in the past will have its watermarks zeroed on the next swap after the rate change. The guard is silently disarmed for those bins.

---

### Likelihood Explanation

The pool admin can execute this via the timelocked `proposeOracleStopLossDecay` → `executeOracleStopLossDecay` flow. The timelock is designed to give LPs time to react to parameter changes, but the retroactive application of the new rate to historical `lastDecayTs` values is not visible to LPs monitoring the proposal. An admin who increases the decay rate to allow faster recovery after a breach inadvertently disarms the guard for all bins that have not been touched during the timelock window. The NatSpec states "monitor at least as often as the timelock or trust the pool admin," but the retroactive impact is not predictable from the proposed rate alone.

---

### Recommendation

When `executeOracleStopLossDecay` is called, apply the **old** decay rate to the elapsed time for all existing watermarks before committing the new rate, or reset `lastDecayTs` for all active bins to `block.timestamp` so the new rate is only applied to future time. Alternatively, mirror the approach used in `executeOracleStopLossHighWatermarks` and require an explicit watermark reset whenever the decay rate changes.

---

### Proof of Concept

1. Pool is initialized with `drawdownE6 = 50_000`, `decayPerSecondE8 = 1` (very slow decay), `timelock = 0`.
2. A swap occurs at `T0`; bin 0 watermarks are set to `hwm0 = H`, `lastDecayTs = T0`.
3. 30 days pass with no swaps on bin 0 (`dt = 2,592,000 s`).
4. Admin proposes and immediately executes `decayPerSecondE8 = 39`.
   - Under the old rate: `1 × 2,592,000 / 1e8 = 2.59%` decay — watermarks still at ~97% of `H`.
   - Under the new rate: `39 × 2,592,000 = 101,088,000 > 1e8` — watermarks floor to 0.
5. Next swap on bin 0: `_decayed(H, 39, 2_592_000)` returns 0; `_applyWatermark` sees `metric >= 0`, ratchets up to the current metric, no breach detected.
6. Stop-loss guard is silently disarmed; the swap proceeds even if the bin's value per share is below the drawdown floor.
7. Subsequent swaps extract value from the pool without triggering `OracleStopLossTriggered`. [6](#0-5)

  function _checkAndUpdateWatermarks(
    address pool_,
    int8 binIdx,
    uint256 metricT0,
    uint256 metricT1,
    uint256 floorMultiplier,
    uint256 decayRate,
    bool zeroForOne
  ) private {
    BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
    uint256 dt = block.timestamp - hwmS.lastDecayTs;

    (uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
    if (breach0 && zeroForOne) {
      revert OracleStopLossTriggered(binIdx, true, metricT0, (hwm0 * floorMultiplier) / E6);
    }

    (uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
    if (breach1 && !zeroForOne) {
      revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
    }

    // forge-lint: disable-next-line(unsafe-typecast)
    hwmS.token0 = uint104(hwm0);
    // forge-lint: disable-next-line(unsafe-typecast)
    hwmS.token1 = uint104(hwm1);
    hwmS.lastDecayTs = uint32(block.timestamp);
  }

  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
