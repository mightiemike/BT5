### Title
Governance can retroactively extend `ExitDuration` for existing tier positions, trapping users while their delegated tokens remain subject to validator slashing — (File: `x/tieredrewards/keeper/msg_server_auth.go`)

---

### Summary

`UpdateTier` in `msg_server_auth.go` allows the governance authority to modify a tier's `ExitDuration` without checking for active positions. Because `TriggerExitFromTier` reads the **current** tier's `ExitDuration` at call time, governance can retroactively extend the exit lock period for all existing position holders. Users who have not yet triggered their exit are forced to wait far longer than they agreed to, while their delegated tokens remain subject to validator slashing throughout the extended period.

---

### Finding Description

`UpdateTier` updates a tier's full parameter set, including `ExitDuration`, without checking whether active positions exist in that tier. The only conditional guard is for `BonusApy` changes, which triggers a reward settlement flush: [1](#0-0) 

Specifically, lines 67–71 guard only `BonusApy`:

```go
if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
    if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
        return nil, err
    }
}
```

No equivalent guard exists for `ExitDuration` changes. The tier is then unconditionally overwritten via `SetTier`. [2](#0-1) 

Contrast this with `deleteTier`, which **does** check for active positions before proceeding: [3](#0-2) 

The protection exists for deletion but is absent for parameter mutation.

On the user side, `TriggerExitFromTier` reads the **current** tier's `ExitDuration` at the moment the user triggers their exit — the position does not snapshot the `ExitDuration` at creation time: [4](#0-3) 

```go
tier, err := ms.getTier(ctx, pos.TierId)
if err != nil {
    return nil, err
}
sdkCtx := sdk.UnwrapSDKContext(ctx)
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
```

If governance increases `ExitDuration` after a user has locked tokens, every subsequent call to `TriggerExitFromTier` for positions in that tier will compute `ExitUnlockAt` using the new, longer duration — not the duration in effect when the user entered.

---

### Impact Explanation

A user who locked tokens under a tier with `ExitDuration = 7 days` can be forced to wait `365 days` (or any duration governance sets) if a governance proposal increases `ExitDuration`. During this extended lock period:

- The user's delegated tokens remain subject to validator slashing via the staking module's slash hooks, which the tiered rewards module explicitly handles: [5](#0-4) 

- The user cannot exit their position within the originally agreed timeframe.
- The user cannot reduce their slashing exposure by exiting, even though they would have been able to do so under the original terms.

This is a direct financial risk: the user's principal can be reduced by validator slashing during an extended lock period they never consented to. The broken invariant is: *users who locked tokens under specific exit conditions must be able to exit under those same conditions*.

---

### Likelihood Explanation

Triggering this requires a governance proposal to pass `MsgUpdateTier`. Governance proposals to adjust tier parameters (e.g., to respond to market conditions or extend lock incentives) are expected operational actions. The lack of any on-chain warning or check means governance can inadvertently — or deliberately — retroactively extend exit durations for all existing position holders without realizing the impact. Likelihood is moderate: the action is a normal governance operation with no friction preventing it from affecting live positions.

---

### Recommendation

**Option 1 (preferred):** Snapshot `ExitDuration` into the `Position` struct at creation time (in `LockTier` / `CommitDelegationToTier`), and use the stored value in `TriggerExitFromTier` instead of reading the current tier. This ensures users always exit under the terms they agreed to.

**Option 2:** In `UpdateTier`, when `ExitDuration` changes, check `hasPositionsForTier` and return an error if active positions exist — mirroring the guard already present in `deleteTier`. [6](#0-5) 

---

### Proof of Concept

1. Governance creates a tier with `ExitDuration = 7 days`.
2. User calls `LockTier` or `CommitDelegationToTier` — tokens are locked and delegated to a validator.
3. Governance passes `MsgUpdateTier` setting `ExitDuration = 365 days`. No active-position check fires; the tier is updated.
4. User calls `TriggerExitFromTier`. `getTier` returns the updated tier; `pos.TriggerExit(blockTime, 365 days)` sets `ExitUnlockAt = now + 365 days`.
5. User cannot withdraw or clear their position for 365 days instead of 7 days.
6. During those 365 days, the user's delegated tokens remain subject to validator slashing. If the validator is slashed, the user suffers a principal loss they could have avoided by exiting within the original 7-day window.

The entry path is a standard governance `MsgUpdateTier` transaction — no special access beyond governance passage is required, and no code path prevents the retroactive application to existing positions.

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}

func (k Keeper) deleteTier(ctx context.Context, tierId uint32) error {
	hasPositions, err := k.hasPositionsForTier(ctx, tierId)
	if err != nil {
		return err
	}
	if hasPositions {
		return types.ErrTierHasActivePositions
	}
	if err := k.Tiers.Remove(ctx, tierId); err != nil {
		if stderrors.Is(err, collections.ErrNotFound) {
			return errorsmod.Wrapf(types.ErrTierNotFound, "tier id %d", tierId)
		}
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tierId)
	}
	return nil
}
