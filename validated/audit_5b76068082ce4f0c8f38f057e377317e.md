Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Originating User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router's address, not the end user's. A pool admin who allowlists the router to enable router-mediated swaps for permitted users inadvertently opens the pool to every caller, because the router's single allowlist entry passes for all users regardless of identity.

## Finding Description

`MetricOmmPool.swap` captures `msg.sender` and forwards it as `sender` to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` passes this value unchanged to the extension. `SwapAllowlistExtension.beforeSwap` then checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without encoding the originating user anywhere:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData   // extension data from caller, not checked by extension
    );
```

The extension ignores `extensionData` entirely (the parameter is unnamed and unused). When the router calls `pool.swap()`, the pool sees `msg.sender = router`, so `sender = router`. The check becomes `allowedSwapper[pool][router]`. If the router is allowlisted — the only way to permit any router-mediated swap on a curated pool — this check passes for every caller unconditionally. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput`, all of which call `pool.swap()` with `msg.sender = router`.

## Impact Explanation

The allowlist invariant — "only permitted addresses may trade in this pool" — is silently broken for all router-mediated swaps the moment the router is allowlisted. Any unpermissioned user, including those explicitly excluded, can execute swaps in a KYC-gated or institutionally restricted pool by routing through `MetricOmmSimpleRouter`. This constitutes broken core pool functionality with direct fund-access impact: restricted pools are fully open to arbitrary traders.

## Likelihood Explanation

The precondition (router allowlisted) is a necessary operational step for any pool admin who wants permitted users to trade via the router. There is no way to allowlist the router for specific users only; the entry is binary. Once the router is allowlisted, the bypass is available to any address with no privileged access required. It is repeatable and requires no special on-chain state beyond the router being allowlisted.

## Recommendation

The extension must gate on the economic actor, not the immediate caller. The preferred fix is to have the router encode the originating `msg.sender` in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it when `sender` is a known router. Because `beforeSwap` is only callable by a registered pool (`msg.sender` is validated via `onlyPool` in the base class), the pool is trusted to forward `extensionData` faithfully. Alternatively, add a complementary allowlist check inside `MetricOmmSimpleRouter` before calling `pool.swap()`, rejecting callers not permitted for the target pool, as a defense-in-depth layer.

## Proof of Concept

1. Deploy pool with `SwapAllowlistExtension`. Admin intends only `userA` to swap.
2. Admin calls `setAllowedToSwap(pool, userA, true)`.
3. Admin calls `setAllowedToSwap(pool, router, true)` so `userA` can use the router.
4. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
5. Router calls `pool.swap(userB, zeroForOne, amount, limit, "", extensionData)` — `msg.sender` to pool is `router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
8. `userB`'s swap executes, bypassing the allowlist entirely.
