### Title
Unconditional Non-Empty Merkle Proof Guard Blocks Valid Single-Transaction Block Proofs — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` unconditionally rejects an empty `merkle_proof` before reaching the actual computation. For a block containing exactly one transaction, the correct and only valid proof **is** empty (the merkle root equals the transaction hash directly). This mirrors M-18 exactly: a guard check fires even when the operation it protects against is not applicable, permanently blocking a class of legitimate callers.

---

### Finding Description

Inside `verify_transaction_inclusion`, after confirming the block is on the main chain and has enough confirmations, the following check fires unconditionally:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

Immediately after, the function delegates to:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` iterates over the proof slice and, when it is empty, simply returns `transaction_hash` unchanged:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    ...
    for proof_hash in merkle_proof { ... }
    current_hash
}
``` [3](#0-2) 

For a block containing exactly one transaction, the Bitcoin protocol defines `merkle_root = tx_hash`. An empty proof is therefore the **only** mathematically correct proof. The `require!` guard fires before the computation can confirm this, causing an unconditional panic.

The same code path is triggered when `verify_transaction_inclusion_v2` calls the deprecated v1 function internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

`verify_transaction_inclusion_v2` already validates the coinbase proof against the merkle root before this call, yet the unconditional guard in v1 still fires and panics. [5](#0-4) 

---

### Impact Explanation

Any NEAR caller — including downstream contracts that consume the light client's verification result — who submits a valid empty merkle proof for a transaction in a single-transaction block will receive a panic (`"Merkle proof is empty"`) instead of the correct `true` result. The core contract invariant — that a mathematically valid inclusion proof is accepted — is broken for this entire class of blocks. Because the check is unconditional, there is no caller-controlled workaround; the block cannot be verified through either the v1 or v2 API.

---

### Likelihood Explanation

Bitcoin blocks containing only a coinbase transaction are uncommon on mainnet but are valid protocol-level constructs. They appear in early mainnet history, on testnet, and during periods of low mempool activity. Any downstream bridge or application that needs to verify a transaction in such a block is permanently blocked. The entry path is fully unprivileged: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are both public, pausable-only functions callable by any NEAR account. [6](#0-5) [7](#0-6) 

---

### Recommendation

Remove the unconditional guard:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

`compute_root_from_merkle_proof` already handles the empty-proof case correctly by returning the transaction hash unchanged. The guard provides no additional safety and actively rejects valid inputs. If a non-empty proof is desired as a policy constraint, the check must be made conditional on the block's transaction count, which is not available on-chain here — making removal the correct fix.

---

### Proof of Concept

1. A Bitcoin block `B` contains exactly one transaction (the coinbase). Its `merkle_root = coinbase_tx_hash`.
2. The block is submitted to the light client and stored in `headers_pool`.
3. A NEAR caller invokes `verify_transaction_inclusion` with:
   - `tx_id = coinbase_tx_hash`
   - `tx_block_blockhash = B.hash`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `confirmations = 1`
4. The function passes the confirmation check, retrieves the header, then panics at:
   ```rust
   require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
   ``` [1](#0-0) 
5. Yet `compute_root_from_merkle_proof(coinbase_tx_hash, 0, &[])` would return `coinbase_tx_hash`, which equals `B.merkle_root` — a valid proof that the guard prevents from ever being evaluated.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
