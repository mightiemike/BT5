### Title
Fork-Block Difficulty Calculated Against Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

### Summary

When validating the proof-of-work difficulty for a fork block, `get_next_work_required` in the Dogecoin implementation fetches the boundary block for the difficulty window using `get_header_by_height`, which always returns the **mainchain** block at that height. For a fork block whose chain diverged before that boundary height, the correct block to use is the **fork's own ancestor** at that height. Using the mainchain block's timestamp instead of the fork ancestor's timestamp produces an incorrect `actual_timespan`, which in turn produces an incorrect expected difficulty. An attacker who controls the fork chain's timestamps can exploit this to submit fork blocks that satisfy a lower difficulty than the protocol requires.

The codebase itself acknowledges this with an explicit TODO comment at the exact line of the bug.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty for the next block by measuring the elapsed time between the previous block and the first block of the current difficulty window: [1](#0-0) 

```rust
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    ...;

// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as: [2](#0-1) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

This unconditionally returns the **mainchain** block at `height_first`. When the block being validated is a fork block whose chain diverged at or before `height_first`, the fork's ancestor at that height is a **different block** with a potentially different timestamp. The correct approach (used by the Zcash implementation) is to walk the chain backwards via `get_prev_header`, which follows `prev_block_hash` links through `headers_pool` and therefore correctly traverses the fork chain: [3](#0-2) 

The same flaw exists in the Bitcoin and Litecoin implementations: [4](#0-3) [5](#0-4) 

The Dogecoin post-145,000 protocol sets `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_height − 1`. A fork that diverges only **two blocks** before the tip already has a different ancestor at `height_first`, making this the most reachable variant.

---

### Impact Explanation

The difficulty calculation in `calculate_next_work_required` computes: [6](#0-5) 

```rust
let modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
// clamped to [min_timespan, max_timespan]
```

If the attacker sets the fork's ancestor at `height_first` to a timestamp **later** than the mainchain block at that height, the code computes `actual_timespan = prev_block_time − T_mainchain` (larger than correct), producing a larger `modulated_timespan` and therefore a **lower (easier) difficulty target**. The clamp allows up to `max_timespan = retarget_timespan × 1.5`, meaning the attacker can reduce the required difficulty by up to ~33% relative to the correct value.

A fork block meeting this artificially lowered difficulty is accepted into `headers_pool`. If the fork accumulates sufficient chainwork to trigger `reorg_chain`, it becomes the canonical chain, and `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` will return `true` for transactions in the fraudulent fork — the primary security guarantee of the light client is broken. [7](#0-6) 

---

### Likelihood Explanation

For the Dogecoin post-145,000 protocol the fork needs to diverge only two blocks before the tip. The attacker submits fork blocks via `submit_blocks`, which is the normal relayer entry point. Timestamp manipulation is constrained by the MTP rule and `MAX_FUTURE_BLOCK_TIME_LOCAL`, but within those bounds the attacker can reliably push `actual_timespan` to `max_timespan`, achieving the maximum difficulty reduction. The TODO comment at the exact line of the bug confirms the developers identified this as an open correctness question. [8](#0-7) 

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool`, mirroring the Zcash implementation in `zcash_get_next_work_required`. This ensures the boundary block is always the fork's own ancestor, regardless of what the current mainchain holds at that height.

---

### Proof of Concept

1. Mainchain state: `[H−3] → [H−2: time=T_main] → [H−1] → [H: tip]`
2. Attacker submits fork block at height `H−2` with `time = T_main + 3600` (1 hour later, within `MAX_FUTURE_BLOCK_TIME_LOCAL`). Fork diverges here.
3. Attacker submits fork block at height `H−1` with `time = T_main + 3660`.
4. Contract validates fork block at height `H`:
   - `prev_block_header` = fork block at `H−1`, `time = T_main + 3660`
   - `height_first = H−2`
   - `get_header_by_height(H−2)` → mainchain block, `time = T_main`
   - `actual_timespan = (T_main + 3660) − T_main = 3660 s` → clamped to `max_timespan = 90 s`
   - Correct `actual_timespan = (T_main + 3660) − (T_main + 3600) = 60 s`
   - Computed `modulated_timespan = 60 + (90−60)/8 = 63.75 s` vs correct `60 s`
   - Expected bits computed from the easier target; fork block at `H` with those bits is accepted.
5. If the fork's cumulative chainwork exceeds the mainchain's, `reorg_chain` promotes the fork. `verify_transaction_inclusion` then confirms attacker-chosen transactions as valid.
