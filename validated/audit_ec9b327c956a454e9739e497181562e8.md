### Title
Missing `ClassTrace.Validate()` call in `processReceivedPacket` allows counterparty chain to mint NFTs under a class trace with empty `BaseClassId` — (`x/nft-transfer/keeper/packet.go`)

### Summary

A counterparty chain can send an IBC NFT packet whose `ClassId` ends with a trailing slash (e.g., `"nft/channel-0/"`). `ValidateBasic()` only checks for blank strings and passes. `processReceivedPacket` calls `ParseClassTrace` on the prefixed class ID, which uses `strings.Split` and assigns the last empty segment as `BaseClassId`. `classTrace.Validate()` is never called. The malformed `ClassTrace{Path: "nft/channel-1/nft/channel-0", BaseClassId: ""}` is stored in state, and `IssueDenom`/`MintNFT` are called with the resulting `ibc/<hash>` class ID.

### Finding Description

`ParseClassTrace` in `x/nft-transfer/types/trace.go` (lines 61–75) splits on `/` and takes the last segment as `BaseClassId`. For any input ending in `/`, the last segment is `""`. The function has no guard against this. `processReceivedPacket` in `x/nft-transfer/keeper/packet.go` (lines 160–168) calls `ParseClassTrace`, then immediately calls `k.SetClassTrace` and `classTrace.IBCClassID()` without ever calling `classTrace.Validate()`. The only upstream guard is `data.ValidateBasic()` in `relay.go` (line 105), which only rejects a blank `ClassId` — a trailing slash is not blank.

### Impact Explanation

- A malformed `ClassTrace` with `BaseClassId = ""` is permanently written to chain state.
- NFTs are minted under `ibc/<hash_of_path_with_trailing_slash>`, a class ID that appears legitimate but has no traceable base class.
- When the receiver attempts to send these NFTs back, `ClassPathFromHash` returns the path with the trailing slash. `IsAwayFromOrigin` evaluates correctly (the NFT is burned on the receiving chain), but the packet sent to the counterparty has `ClassId = "nft/channel-1/nft/channel-0/"`. The counterparty chain sees this as `isAwayFromOrigin = true` (it does not start with `"nft/channel-0/"`) and mints a new voucher instead of unescrowing — the original NFT lifecycle is permanently broken and the user's NFTs are effectively locked.
- Additionally, the empty-`BaseClassId` trace collides in the hash space with any legitimate trace whose `GetFullClassPath()` happens to equal `"nft/channel-1/nft/channel-0/"` (i.e., a trace with `Path="nft/channel-1/nft/channel-0"` and `BaseClassId=""`), corrupting the class trace registry.

### Likelihood Explanation

Any chain with an established IBC channel to the target chain can send this packet. In the Cosmos ecosystem, establishing an IBC channel is permissionless. A malicious actor can deploy a chain, open a channel, and send a crafted packet. No governance or validator compromise of the *target* chain is required.

### Recommendation

Add a `classTrace.Validate()` call in `processReceivedPacket` immediately after `ParseClassTrace`, before `SetClassTrace` or `IBCClassID()` is called. Return an error (which becomes an error acknowledgement) if validation fails. Additionally, `ValidateBasic()` on `NonFungibleTokenPacketData` should reject `ClassId` values ending with `/`.

```go
func ParseClassTrace(rawClassID string) ClassTrace {
	classSplit := strings.Split(rawClassID, "/")

	if classSplit[0] == rawClassID {
		return ClassTrace{
			Path:        "",
			BaseClassId: rawClassID,
		}
	}

	return ClassTrace{
		Path:        strings.Join(classSplit[:len(classSplit)-1], "/"),
		BaseClassId: classSplit[len(classSplit)-1],
	}
}
```
