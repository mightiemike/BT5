### Title
`op_unknown` Cost Check Applied Before Multiplier Causes Returned Cost to Exceed `max_cost` — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the `check_cost` guard is called on the **pre-multiplied** base cost, but the value actually returned to the caller is the **post-multiplied** cost. This is a direct arithmetic-ordering analog to the external report's "division before multiplication" pattern: the guard sees a value that is `1/(cost_multiplier+1)` of the true cost, allowing `op_unknown` to return a `Reduction` whose cost exceeds `max_cost` without ever triggering `CostExceeded`.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes a base cost from the opcode's argument structure, then multiplies it by `(cost_multiplier + 1)` extracted from the opcode bytes. The single `check_cost` call occurs **before** that multiplication:

```rust
// line 260
check_cost(cost, max_cost)?;
// line 261
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

The `cost_multiplier` is decoded from the leading bytes of the opcode atom:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    None => { return Err(EvalErr::Invalid(o))?; }
};
``` [2](#0-1) 

`cost_multiplier` can be up to `u32::MAX` (4 294 967 295), so the multiplied cost can be up to `u32::MAX` times the base cost. The only post-multiplication guard is `cost > u32::MAX`, which caps the returned cost at 4 294 967 295 — but says nothing about whether that value exceeds `max_cost`.

**Concrete example** (cost_function = 0, constant cost):

| Parameter | Value |
|---|---|
| `max_cost` | 1 000 |
| base `cost` (cost_function=0) | 1 |
| `check_cost(1, 1000)` | passes |
| `cost_multiplier` | 1 500 |
| final `cost` | 1 501 |
| returned | `Ok(Reduction(1501, nil))` |

`op_unknown` returns a cost of 1 501 when `max_cost` is 1 000, with no `CostExceeded` error raised from within the function. [3](#0-2) 

The `check_cost` utility itself is correct; the bug is purely in call ordering: [4](#0-3) 

---

### Impact Explanation

An attacker-controlled CLVM byte sequence can encode an unknown opcode whose `cost_multiplier` field is chosen so that:

- the base cost passes `check_cost` (base cost ≤ `max_cost`), and  
- the multiplied cost exceeds `max_cost` by up to a factor of `u32::MAX`.

`op_unknown` returns `Ok(Reduction(cost, nil))` with `cost > max_cost`. Whether the run-program loop catches this depends on whether it re-checks the accumulated total after each operator call. If `max_cost` is passed down as a *remaining budget* and `op_unknown`'s internal check is the only guard for that budget slice, the program executes an operation whose true cost was never validated against the limit — an undercharged execution analogous to the truncated-shares bug in the external report.

Even if the outer loop catches the excess, the cost model invariant — that no single operator call returns a cost exceeding the budget it was given — is broken, which is a consensus-relevant property in a deterministic VM.

---

### Likelihood Explanation

- The entry path is fully attacker-controlled: any CLVM program byte sequence that includes an unknown opcode (any opcode not starting with `0xffff`, with a non-empty body) reaches `op_unknown`.  
- The `cost_multiplier` is decoded directly from the opcode bytes with no sanitization beyond a 4-byte length limit.  
- No special privileges, configuration, or social engineering are required; a standard `run_program` call with a crafted program suffices. [5](#0-4) 

---

### Recommendation

Move `check_cost` to after the multiplier is applied, mirroring the correct order used everywhere else in the file:

```rust
// Before (wrong order):
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;

// After (correct order):
cost *= cost_multiplier + 1;
check_cost(cost, max_cost)?;
```

The `cost > u32::MAX` guard should remain in place (or be folded into `check_cost` since `max_cost` is already `u64`).

---

### Proof of Concept

Craft an opcode atom where:
- bytes `[0..len-1]` encode `cost_multiplier = max_cost` (e.g., `[0x00, 0x00, 0x03, 0xe8]` for `max_cost = 1000`)  
- the last byte has bits `[7:6] = 0b00` (cost_function = 0, constant cost = 1)

Base cost = 1 ≤ 1000 → `check_cost` passes.  
Multiplied cost = 1 × (1000 + 1) = 1001 > `max_cost`.  
`op_unknown` returns `Ok(Reduction(1001, nil))` — cost exceeds budget with no error. [6](#0-5)
