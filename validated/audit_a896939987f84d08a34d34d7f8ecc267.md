### Title
ML-DSA-65 Gas-Key Host Functions Use Wire Length Instead of Trie-ID Length for Exec-Fee Calculation - (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The host-function path for gas-key actions (`promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`) passes the raw borsh wire length of the public key (`public_key_len as usize`, which is **1953 bytes** for ML-DSA-65) to exec-fee helpers that are documented to expect the **on-trie identifier length** (33 bytes for ML-DSA-65). The transaction path correctly calls `public_key.trie_id_len()` (33 bytes). The divergence is ~1920 bytes per call, causing a systematic exec-fee overcharge of `1920 × gas_key_byte_exec_rate` gas for every contract that adds or funds an ML-DSA-65 gas key via a host function.

---

### Finding Description

`PublicKey::len()` returns the borsh-encoded wire length: 33 bytes for ED25519, 65 for SECP256K1, and **1953 bytes** for ML-DSA-65 (1 tag + 1952-byte raw key). `PublicKey::trie_id_len()` returns the on-trie identifier length: 33 bytes for ED25519, 65 for SECP256K1, and **33 bytes** for ML-DSA-65 (1 tag + 32-byte SHA3-256 hash). The two values diverge only for ML-DSA-65. [1](#0-0) 

The exec-fee helpers are explicitly documented to price trie I/O, not wire I/O: [2](#0-1) 

The **transaction path** in `exec_fee()` correctly uses `trie_id_len()`: [3](#0-2) 

The **host-function path** in `VMLogic::promise_batch_action_transfer_to_gas_key` passes the raw wire length: [4](#0-3) 

The same error appears in `promise_batch_action_add_gas_key_with_full_access`: [5](#0-4) 

And in `promise_batch_action_add_gas_key_with_function_call`: [6](#0-5) 

The identical pattern is present in the wasmtime runner: [7](#0-6) [8](#0-7) [9](#0-8) 

The storage-usage accounting in `access_key_storage_usage` and `gas_key_storage_cost` already correctly uses `trie_id_len()`, and the existing fee-parity tests (`test_gas_key_fee_parity_full_access`, `test_gas_key_fee_parity_function_call`) only exercise ED25519 keys, where `len() == trie_id_len()`, so the discrepancy is invisible to the test suite. [10](#0-9) [11](#0-10) 

---

### Impact Explanation

For every contract call that invokes one of the three affected host functions with an ML-DSA-65 gas key:

- **Exec fee overcharge**: `(1953 − 33) × gas_key_byte_exec_rate` = `1920 × gas_key_byte_exec_rate` gas is charged above what the trie operation actually costs.
- **Fee parity broken**: the same logical action costs different gas depending on whether it is submitted as a signed transaction (correct, uses `trie_id_len()`) or via a cross-contract promise (incorrect, uses wire length). This is a deterministic, protocol-level invariant violation.
- **Transaction failure**: if a contract budgets gas based on the expected trie cost (33 bytes), the actual charge (1953 bytes) may exhaust the gas limit, causing the receipt to fail with `GasExceeded`.
- **Excess gas burnt**: exec gas is not refunded; the overcharge is permanently burnt from the caller's balance.

---

### Likelihood Explanation

The `PostQuantumSignatures` feature is stabilized at protocol version 85. Once active on any network, any deployed contract can call these host functions with an ML-DSA-65 key. No privileged role is required. The trigger is purely user-controlled: a contract that constructs a gas-key action via a promise batch with an ML-DSA-65 public key. The existing fee-parity integration tests do not cover ML-DSA-65 keys, so the bug is not caught by the current test suite. [12](#0-11) 

---

### Recommendation

After reading the public key bytes from guest memory, derive the trie-id length from the parsed key rather than from the raw byte count. In `PublicKeyBuffer`, expose a `trie_id_len()` helper that delegates to `PublicKey::trie_id_len()` on the successfully parsed key (or returns the wire length as a safe fallback if parsing failed, since the error will be surfaced at `decode()` time). Replace every occurrence of `public_key_len as usize` in exec-fee calls with this value:

```rust
// Before (wrong for ML-DSA-65):
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), public_key_len as usize);

// After (correct):
let pk_trie_id_len = public_key.trie_id_len(); // new helper on PublicKeyBuffer
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), pk_trie_id_len);
```

The send-fee call (`gas_key_transfer_send_fee`) intentionally uses the wire length ("what the sender sees") and should remain unchanged.

Add a fee-parity test that exercises all three host functions with an ML-DSA-65 key and asserts `gas_burnt` equality with the equivalent signed-transaction path.

---

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with a borsh-encoded ML-DSA-65 public key (1953 bytes on the wire).
2. The host function receives `public_key_len = 1953`.
3. `gas_key_add_key_exec_fee` is called with `public_key_len = 1953`, computing `nonce_key_len = access_key_key_len(account_id_len, 1953) + sizeof(NonceIndex)`.
4. The actual trie nonce key uses the 33-byte hash form, so the correct call would pass `33`.
5. The overcharge per nonce entry is `(1953 − 33) × gas_key_byte_exec_rate` gas.
6. Submit the same `AddKey` action as a signed transaction: `exec_fee` calls `gas_key_add_key_exec_fee(..., public_key.trie_id_len(), ...)` = `gas_key_add_key_exec_fee(..., 33, ...)`.
7. The two `gas_burnt` values differ by `num_nonces × 1920 × gas_key_byte_exec_rate`, confirming the parity violation. [13](#0-12) [14](#0-13)

```rust
    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```

```rust
/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```rust
fn access_key_storage_usage(
    fee_config: &RuntimeFeesConfig,
    public_key: &PublicKey,
    access_key: &AccessKey,
) -> StorageUsage {
    let storage_usage_config = &fee_config.storage_usage_config;
    // Use the on-trie identifier length, not the borsh-serialized pubkey
    // length: ML-DSA-65 access keys live in the trie as a SHA3-256 hash
    // (33 bytes incl. type tag), not as a 1953-byte full pubkey.
    public_key.trie_id_len() as u64
        + borsh::object_length(access_key).unwrap() as u64
        + storage_usage_config.num_extra_bytes_record
}
```

```rust
fn gas_key_storage_cost(
    fee_config: &RuntimeFeesConfig,
    public_key: &PublicKey,
    access_key: &AccessKey,
    num_nonces: NonceIndex,
) -> StorageUsage {
    let storage_config = &fee_config.storage_usage_config;
    let per_nonce_value_size = borsh::object_length(&(0 as Nonce)).unwrap() as u64;
    let per_nonce_key_size = public_key.trie_id_len() as u64 + size_of::<NonceIndex>() as u64;

    num_nonces as u64
        * (per_nonce_key_size + per_nonce_value_size + storage_config.num_extra_bytes_record)
        + access_key_storage_usage(fee_config, public_key, access_key)
}
```
