### Title
Linked Signer Can Overwrite Subaccount's Linked Signer, Enabling Unauthorized Collateral Drain — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction path accepts a signature from the **existing linked signer** (`allowLinkedSigner = true`). This means a linked signer can unilaterally replace itself with an attacker-controlled address, after which the new signer can sign `WithdrawCollateral`, `BurnNlp`, or any other asset-moving transaction against the victim subaccount — without the subaccount owner's knowledge or consent.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 581–590
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true                          // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateCompactSignature`, which accepts a signature from **either** the subaccount owner address **or** the current linked signer:

```solidity
// Verifier.sol lines 312–318
address recovered = ECDSA.recover(digest, signature.r, signature.vs);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

Because the linked signer's signature is sufficient to pass `validateSignedTx`, the linked signer can craft and submit a `SignedLinkSigner` transaction that sets `signedTx.tx.signer` to any address it controls. The `linkedSigners` mapping is then overwritten with that new address. [3](#0-2) 

Once the new address is installed as the linked signer, it can sign `WithdrawCollateral` (fast mode, `allowLinkedSigner = true`): [4](#0-3) 

or `BurnNlp` (fast mode, `allowLinkedSigner = true`): [5](#0-4) 

to drain or destroy the victim subaccount's assets.

---

### Impact Explanation

A compromised or malicious linked signer can:
1. Replace itself with an attacker-controlled address in a single sequenced transaction.
2. Use the new address to sign `WithdrawCollateral` or `BurnNlp` against the victim subaccount.
3. Drain all collateral or burn all NLP tokens from the victim subaccount without the owner's consent.

The corrupted state is `linkedSigners[victimSubaccount]` and subsequently the victim's spot/NLP balances via `spotEngine.updateBalance`. [6](#0-5) 

---

### Likelihood Explanation

Any entity that has ever been granted linked-signer status (e.g., a trading bot, a market-maker key, a third-party integration) can execute this attack unilaterally. The only prerequisite is that the victim previously called `LinkSigner` to delegate signing authority. This is a common operational pattern in the protocol. The attacker needs no additional privileges, no oracle manipulation, and no governance capture.

---

### Recommendation

Set `allowLinkedSigner = false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner's own key should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`:

```solidity
// EndpointTx.sol lines 237–239 (slow mode — correct)
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [7](#0-6) 

The fast-mode path should mirror this restriction by passing `false` as the `allowLinkedSigner` argument to `validateSignedTx`.

---

### Proof of Concept

1. **Setup**: Alice owns subaccount `0xAlice000...`. She sets Bob (`0xBob`) as her linked signer via a `LinkSigner` slow-mode transaction.
2. **Attack step 1**: Bob constructs a `SignedLinkSigner` with `tx.sender = 0xAlice000...`, `tx.signer = 0xAttacker`, and signs it with Bob's key.
3. **Attack step 2**: Bob submits this transaction to the sequencer. `validateSignedTx(..., allowLinkedSigner=true)` passes because Bob is the current linked signer. `linkedSigners[0xAlice000...]` is overwritten with `0xAttacker`.
4. **Attack step 3**: The attacker (`0xAttacker`) constructs a `SignedWithdrawCollateral` for Alice's subaccount and signs it. `validateSignedTx(..., allowLinkedSigner=true)` passes because `0xAttacker` is now the linked signer.
5. **Result**: `clearinghouse.withdrawCollateral` is called with Alice's subaccount as sender, draining her collateral to an attacker-controlled address — without Alice's consent and without Alice's private key ever being involved. [8](#0-7) [4](#0-3)
