### Title
`Treehasher.sha256_treehash` Ignores Configured Prefixes, Always Using CHIA Domain Separators — (`File: wheel/python/clvm_rs/tree_hash.py`)

### Summary

The `Treehasher` class in the Python wheel is designed to be configurable with custom `atom_prefix` and `pair_prefix` values for domain-separated tree hashing. However, the `sha256_treehash` method's inner closures call the **module-level** `shatree_atom` and `shatree_pair` functions — which are permanently bound to `CHIA_TREEHASHER`'s methods — instead of `self.shatree_atom` and `self.shatree_pair`. As a result, every `Treehasher` instance, regardless of its configured prefixes, silently produces the same CHIA-standard hashes (`0x01`/`0x02`). This is the direct analog of the external report's "same seed for two independent outputs" wiring error: the custom prefix is wired to the wrong output path and is never applied.

### Finding Description

In `wheel/python/clvm_rs/tree_hash.py`, the `Treehasher` class accepts `atom_prefix` and `pair_prefix` at construction time and correctly uses them in `shatree_atom` (line 39) and `shatree_pair` (line 45). However, the `sha256_treehash` method (line 50) defines two inner closures, `handle_obj` and `handle_pair`, that call the **module-level names** `shatree_atom` (line 63) and `shatree_pair` (line 86). These module-level names are assigned at the bottom of the file:

```python
shatree_atom = CHIA_TREEHASHER.shatree_atom   # line 108
shatree_pair = CHIA_TREEHASHER.shatree_pair   # line 109
```

They are permanently bound to `CHIA_TREEHASHER`'s methods, which use `atom_prefix = b"\x01"` and `pair_prefix = b"\x02"`. Any custom `Treehasher` instance — e.g., `Treehasher(b"\x03", b"\x04")` — will have its configured prefixes silently ignored inside `sha256_treehash`. The fix would be to call `self.shatree_atom` and `self.shatree_pair` inside the closures instead.

Additionally, `handle_obj` writes the computed (CHIA-standard) hash into `obj._cached_sha256_treehash` (line 66). This means a custom `Treehasher` poisons the cache of shared CLVM objects with CHIA-standard hashes, and any subsequent call to `CHIA_TREEHASHER.sha256_treehash` on the same object will read the cached value without recomputing — which happens to be correct for CHIA but was set by the wrong hasher.

The attacker-controlled entry path is: an external caller submits attacker-crafted CLVM bytes to a downstream application that uses a custom `Treehasher` for domain separation (e.g., to distinguish puzzle types or protocol versions). The application computes a tree hash expecting the custom domain, but receives the CHIA-standard hash. If the application uses this hash for authorization, signature verification, or state-root comparison, the domain boundary is silently collapsed.

### Impact Explanation

Any downstream Python application that instantiates `Treehasher` with non-CHIA prefixes for domain separation will silently compute CHIA-standard hashes instead. This collapses the intended domain boundary between hash spaces. Concretely:

- A puzzle hash computed under a custom domain is identical to the CHIA-standard puzzle hash for the same tree, making cross-domain hash collisions trivially constructable.
- If the application uses the custom-domain hash for authorization or signature binding, an attacker can present a CLVM program whose CHIA-standard hash matches the expected custom-domain hash, bypassing the domain check.
- The `_cached_sha256_treehash` cache poisoning means the incorrect hash persists across calls, making the failure non-transient and hard to detect.

### Likelihood Explanation

The `Treehasher` class is exported from the Python wheel as a public API and is explicitly designed to be instantiated with custom prefixes (the constructor takes `atom_prefix` and `pair_prefix` as required arguments). Any integrator who reads the class signature and creates a custom instance for domain separation will be silently affected. The bug requires no special attacker capability beyond submitting attacker-controlled CLVM bytes to an application that uses a custom `Treehasher`.

### Recommendation

In `sha256_treehash`, replace the module-level function calls inside the inner closures with `self`-bound method calls:

```python
# line 63: change
r = shatree_atom(obj.atom)
# to
r = self.shatree_atom(obj.atom)

# line 86: change
r = shatree_pair(p0, p1)
# to
r = self.shatree_pair(p0, p1)
```

This ensures the configured `atom_prefix` and `pair_prefix` are actually applied during tree hashing, restoring the intended domain separation.

### Proof of Concept

```python
from clvm_rs.tree_hash import Treehasher, CHIA_TREEHASHER
from clvm_rs.clvm_storage import CLVMStorage  # or any CLVMStorage-compatible object

# Create a custom Treehasher with different prefixes
custom_hasher = Treehasher(b"\x03", b"\x04")

# Build a simple atom node
class SimpleAtom:
    atom = b"hello"
    pair = None

node = SimpleAtom()

chia_hash   = CHIA_TREEHASHER.sha256_treehash(node)
custom_hash = custom_hasher.sha256_treehash(node)

# Expected: custom_hash != chia_hash (different domain separators)
# Actual:   custom_hash == chia_hash (bug: module-level shatree_atom used)
assert chia_hash == custom_hash, "BUG: custom prefix silently ignored"

# Verify the custom hasher's own shatree_atom DOES use the custom prefix
direct_custom = custom_hasher.shatree_atom(b"hello")
assert direct_custom != chia_hash, "shatree_atom itself is correct"
# This confirms the wiring error is only inside sha256_treehash's closures

    def shatree_atom(self, atom: bytes) -> bytes:
        s = sha256()
        s.update(self.atom_prefix)
        s.update(atom)
        return s.digest()

    def shatree_pair(self, left_hash: bytes, right_hash: bytes) -> bytes:
        s = sha256()
        s.update(self.pair_prefix)
        s.update(left_hash)
        s.update(right_hash)
        return s.digest()
