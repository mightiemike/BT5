### Title
Bonus pool depletion blocks all exit paths in `x/tieredrewards`, permanently locking user principal — (File: x/tieredrewards/keeper/msg_server.go)

### Summary
Every exit path from a delegated tier position mandates bonus reward settlement before the principal can be recovered. When the bonus pool is empty, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, causing `MsgTierUndelegate` and `MsgExitTierWithDelegation` to fail atomically. Because `MsgWithdrawFromTier` requires the position to already be undelegated, and no exit path bypasses reward settlement, the user's locked principal becomes inaccessible for as long as the pool remains empty — with no protocol-level escape hatch.

### Finding Description

Every user-driven exit path calls `claimRewards` before mutating position state:

**`MsgTierUndelegate`** — the ADR and code confirm it "claims rewards first": [1](#0-0) 

**`MsgExitTierWithDelegation`** — the flow explicitly states "Claim rewards for position (settle base + bonus)" as the first step: [2](#0-1) 

Both paths reach `processEventsAndClaimBonus`, which fails atomically when accrued bonus exceeds pool balance: [3](#0-2) 

The ADR explicitly documents this behavior for user-driven paths: [4](#0-3) 

`MsgWithdrawFromTier` does not call `claimRewards`, but it requires the position to be undelegated and unbonding-complete before it can execute: [5](#0-4) 

Because `MsgTierUndelegate` fails when the pool is empty, the position never reaches the undelegated state, so `MsgWithdrawFromTier` is also blocked (`ErrPositionDelegated`).

The `BeforeRedelegationSlashed` hook demonstrates that the codebase already knows how to forfeit bonus silently when the pool is insufficient — but this safety valve is deliberately absent from user-driven exit paths: [6](#0-5) 

### Impact Explanation

When the bonus pool is empty and a position has accrued non-zero bonus:

- `MsgTierUndelegate` → `ErrInsufficientBonusPool` → position stays delegated
- `MsgExitTierWithDelegation` → `ErrInsufficientBonusPool` → delegation stays in module
- `MsgWithdrawFromTier` → `ErrPositionDelegated` → principal inaccessible
- `MsgTierRedelegate` → `ErrInsufficientBonusPool` (also claims rewards first)
- `MsgClearPosition` (exit triggered) → `ErrInsufficientBonusPool` (also claims rewards first)

The user's **principal** — not just their bonus rewards — is locked inside the tier module account with no protocol-level escape hatch. The only resolution is for an external party to replenish the pool, mirroring the TimeLock report's finding that "assets become inaccessible to the beneficiary until the manual intervention of the smart contract owner."

The corrupted invariant: after `ExitUnlockAt` elapses, a position owner must be able to recover their principal regardless of the bonus pool state.

### Likelihood Explanation

The bonus pool is a finite module account balance. It can reach zero through:

1. **Normal operation**: many users claiming rewards over time without the pool being topped up.
2. **Deliberate drain**: a holder of a large position claims all accrued bonus, emptying the pool, then other users' exit attempts fail.

The `BeginBlocker` top-up only covers base rewards, not the bonus pool: [7](#0-6) 

The bonus pool has no automatic replenishment mechanism. If the external funder stops or governance does not act, the pool stays empty indefinitely.

### Recommendation

Decouple principal recovery from bonus settlement. Specifically:

- In `MsgTierUndelegate` and `MsgExitTierWithDelegation`, if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, silently forfeit the unpayable bonus (advance checkpoints, persist position, proceed with undelegation/exit) rather than aborting the entire transaction.
- This mirrors the existing behavior in `BeforeRedelegationSlashed` and ensures the user can always recover their principal after the exit commitment elapses.
- Alternatively, add a separate `MsgForfeitBonusAndExit` message that skips bonus settlement entirely, allowing the owner to exit at the cost of forfeiting accrued bonus.

### Proof of Concept

1. Alice calls `MsgLockTier` with `trigger_exit_immediately = true`, locking 10,000 basecro.
2. One year passes; Alice accrues significant bonus rewards. The bonus pool is funded initially.
3. Other users (or a single large holder) claim their rewards, draining the pool to zero.
4. Alice's exit commitment elapses (`block_time >= ExitUnlockAt`).
5. Alice calls `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated).
   - `claimRewards` → `processEventsAndClaimBonus` → `totalBonus > 0`, pool balance = 0 → `ErrInsufficientBonusPool`.
   - Transaction reverts. Position remains delegated.
6. Alice calls `MsgExitTierWithDelegation`:
   - `validateExitTierWithDelegation` passes.
   - `claimRewards` → same failure.
7. Alice calls `MsgWithdrawFromTier`:
   - `validateWithdrawFromTier` → `pos.IsDelegated()` → `ErrPositionDelegated`.
8. Alice's 10,000 basecro principal is inaccessible. No protocol path exists to recover it without pool replenishment.

func (k Keeper) validateWithdrawFromTier(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	if pos.IsDelegated() {
		return types.ErrPositionDelegated
	}

	isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isUnbonding {
		return types.ErrPositionUnbonding
	}

	return nil

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (ms msgServer) WithdrawFromTier(ctx context.Context, msg *types.MsgWithdrawFromTier) (*types.MsgWithdrawFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateWithdrawFromTier(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}
