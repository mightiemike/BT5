### Title
Malicious Sync Peer Can Corrupt Stored DeclareV1 Transaction by Sending DeclareV0 Protobuf Variant, Zeroing the Nonce — (`crates/apollo_protobuf/src/converters/transaction.rs`, `crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

A low-trust sync peer can send a `TransactionInBlock` protobuf message with `txn = DeclareV0(...)` (no nonce field) while supplying the real `transaction_hash` of a `DeclareV1` transaction. Because the sync client never verifies the supplied hash against the deserialized transaction content, the node stores a `DeclareTransaction::V0` with `nonce = Nonce(0)` indexed under the real V1 hash. Any subsequent RPC call for that hash returns the wrong transaction type and a zeroed nonce.

---

### Finding Description

**Step 1 — Peer-supplied hash is accepted without verification.**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` reads `transaction_hash` directly from the peer-supplied protobuf field and returns it alongside the deserialized transaction with no cross-check: [1](#0-0) 

**Step 2 — `DeclareV0` arm hardcodes `nonce = Nonce::default()`.**

When the `txn` oneof is `DeclareV0`, the converter unconditionally sets `nonce = Nonce::default()` (= `Nonce(0)`), regardless of what the actual transaction's nonce was: [2](#0-1) 

**Step 3 — Sync client stores the corrupted transaction without hash validation.**

`parse_data_for_block` pushes the peer-supplied `transaction_hash` directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing guard: [3](#0-2) 

**Step 4 — Storage indexes the corrupted transaction under the peer-supplied hash.**

`write_transactions` inserts `tx_hash` (the peer-supplied value) as the key in `transaction_hash_to_idx_table`, so the corrupted `DeclareV0` body with `nonce=0` is permanently indexed under the real V1 hash: [4](#0-3) 

**Step 5 — RPC serves the corrupted data.**

`get_transaction_by_hash` looks up the transaction by hash and returns whatever is stored — the wrong type (`V0` instead of `V1`) and wrong nonce (`0` instead of the real value): [5](#0-4) 

---

### Impact Explanation

The node permanently stores a `DeclareTransaction::V0` with `nonce = Nonce(0)` in place of the real `DeclareTransaction::V1` with its actual nonce. This affects:

- **RPC `starknet_getTransactionByHash`**: returns wrong transaction type and nonce — an authoritative-looking wrong value.
- **RPC `starknet_simulateTransactions` / `starknet_estimateFee` / `starknet_traceTransaction`**: if the stored transaction is replayed for simulation or tracing, the wrong nonce and wrong version tag are used, producing incorrect results.
- **Transaction hash / signature domain**: the stored transaction body no longer matches its hash. Any component that recomputes the hash from the stored body (e.g., for verification or re-execution) will produce a divergent hash, binding the wrong payload to the hash identity.

Applicable impact categories:
- **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**
- **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

Any peer that the syncing node connects to can trigger this. The p2p sync protocol is designed to accept data from untrusted peers; the only guard is the `n_transactions` count from the stored block header. The TODO comment at line 88 of `transaction.rs` confirms the developers are aware that hash validation is absent. No operator privilege is required. [6](#0-5) 

---

### Recommendation

After deserializing each `FullTransaction` from the peer, recompute the transaction hash from the deserialized transaction body using `calculate_transaction_hash` and compare it against the peer-supplied `transaction_hash`. Reject (and penalize) the peer if they differ. This is exactly what the existing TODO at line 88 calls for.

### Proof of Concept

```rust
// In crates/apollo_protobuf/src/converters/transaction.rs (or a test module)
use starknet_api::transaction::{DeclareTransactionV0V1, Fee, Nonce, TransactionSignature, ClassHash};
use starknet_api::core::ContractAddress;
use starknet_api::felt;
use crate::protobuf;

let original = DeclareTransactionV0V1 {
    max_fee: Fee(1000),
    signature: TransactionSignature::default(),
    nonce: Nonce(felt!("0x5")),   // nonce = 5
    class_hash: ClassHash::default(),
    sender_address: ContractAddress::default(),
};

// Attacker serializes as DeclareV0 (drops nonce)
let proto_v0: protobuf::transaction_in_block::DeclareV0WithoutClass = original.clone().into();

// Attacker wraps it in TransactionInBlock with the real V1 hash
let proto_block = protobuf::TransactionInBlock {
    txn: Some(protobuf::transaction_in_block::Txn::DeclareV0(proto_v0)),
    transaction_hash: Some(/* real V1 hash */ ...),
};

// Node deserializes
let (tx, _hash) = <(Transaction, TransactionHash)>::try_from(proto_block).unwrap();
if let Transaction::Declare(DeclareTransaction::V0(inner)) = tx {
    // nonce is Nonce(0), not Nonce(5)
    assert_eq!(inner.nonce, Nonce::default()); // passes — nonce is LOST
}
```

The exact divergent value is `nonce = Nonce(0)` for any `DeclareV1` transaction whose actual nonce is nonzero. [7](#0-6) [2](#0-1)
