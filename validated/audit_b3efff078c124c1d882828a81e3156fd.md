### Title
Incorrect `MAX_FUTURE_BLOCK_TIME_MTP` Constant Causes Contract to Reject Valid Zcash Blocks, Diverging from Real Chain — (`btc-types/src/network.rs`, `contract/src/zcash.rs`)

### Summary

`MAX_FUTURE_BLOCK_TIME_MTP` is hardcoded to `5400` seconds (90 minutes) in the shared constants file, but the real Zcash protocol uses `129600` seconds (36 hours). The `check_pow` function enforces this tighter bound, so any valid Zcash block whose timestamp falls in the range `(prev_mtp + 5400, prev_mtp + 129600]` is rejected by the contract even though it is fully valid on the real Zcash network. The code comment directly contradicts the constant value, confirming this is an implementation error.

### Finding Description

`MAX_FUTURE_BLOCK_TIME_MTP` is defined as:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60; // = 5400 seconds
``` [1](#0-0) 

This constant is consumed directly in `check_pow` for the Zcash feature:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment on line 40 explicitly states the correct Zcash value is `129600` seconds, yet the enforced value is `5400`. The contract therefore applies a rule that is **24× stricter** than the actual Zcash consensus rule.

The same constant is also shared with Bitcoin/Litecoin/Dogecoin paths (where 90 minutes is correct for Bitcoin), but for the Zcash feature it is wrong.

### Impact Explanation

If a valid Zcash block is mined with a timestamp `T` such that:

```
prev_mtp + 5400 < T <= prev_mtp + 129600
```

- The real Zcash network accepts it (valid per consensus).
- The contract's `check_pow` panics with `"time-too-far-ahead-of-mtp"`.
- The block cannot be submitted; the contract's canonical tip is stuck at the last accepted block.
- `verify_transaction_inclusion` returns `false` for any transaction confirmed in that block or any descendant, even though those transactions are fully confirmed on Zcash mainnet.
- Any bridge or settlement layer relying on the contract's canonical state would incorrectly treat confirmed Zcash transactions as unconfirmed. [3](#0-2) 

### Likelihood Explanation

In normal Zcash operation (75-second block target), miners set timestamps close to wall-clock time, so the MTP is typically only a few minutes behind the tip. A timestamp more than 90 minutes ahead of MTP is unusual in steady-state operation. However:

- The Zcash protocol explicitly permits it up to 36 hours.
- During network disruptions, slow block periods, or deliberate timestamp manipulation by a miner, timestamps can legally drift well beyond 90 minutes from MTP.
- The contract has no fallback or override path once it diverges; it would remain stuck until the constant is corrected and the contract redeployed.

### Recommendation

Replace the shared constant with a Zcash-specific value, or introduce a per-chain constant:

```rust
// For Zcash: 36 hours, matching zcash/zcash src/main.cpp MAX_FUTURE_BLOCK_TIME_MTP
pub const ZCASH_MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600;

// For Bitcoin/Litecoin/Dogecoin: 90 minutes
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;
```

Then update `contract/src/zcash.rs` to import and use `ZCASH_MAX_FUTURE_BLOCK_TIME_MTP` instead of `MAX_FUTURE_BLOCK_TIME_MTP`. [4](#0-3) 

### Proof of Concept

1. Deploy the contract with `network: Mainnet` and the Zcash feature enabled.
2. Initialize with a chain of real Zcash headers so `prev_mtp` is established.
3. Construct a Zcash header with `time = prev_mtp + 6000` (valid on real Zcash since `6000 < 129600`), a valid Equihash solution, and correct difficulty bits.
4. Call `submit_blocks([header])`.
5. **Observed**: contract panics with `"time-too-far-ahead-of-mtp"`.
6. **Expected**: contract accepts the block (as the real Zcash network would).
7. Consequence: the contract's `mainchain_tip_blockhash` remains at the pre-submission value; any subsequent `verify_transaction_inclusion` call for a transaction in that block returns `false`, while the real Zcash network considers it confirmed. [5](#0-4) [1](#0-0)
