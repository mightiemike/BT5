### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Exploitable via 64-Byte Transaction Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a live, publicly reachable NEAR contract method. It lacks the coinbase-proof guard that `verify_transaction_inclusion_v2` was introduced to provide. Any unprivileged NEAR caller can invoke it directly and obtain a `true` result for a forged transaction inclusion proof by supplying an internal Merkle tree node as the `tx_id`.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated(since = "0.5.0", ...)]` at line 283 of `contract/src/lib.rs`, but the Rust `#[deprecated]` attribute only emits a compiler warning for Rust-language callers. It does **not** remove the function from the compiled WASM binary or prevent it from being dispatched by the NEAR runtime. The function remains a fully callable public contract method accessible to any NEAR account via RPC. [1](#0-0) 

The function's own documentation acknowledges the flaw explicitly:

> **Warning**: This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification. [2](#0-1) 

The only guard against the trivial case is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

This does not prevent the 64-byte attack, which uses a **non-empty** proof path rooted at an internal Merkle node. The verification reduces to:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no structural constraint on whether the leaf is a real transaction or an internal node: [5](#0-4) 

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase proof of the same depth, which forces the prover to demonstrate knowledge of a real leaf at depth 0: [6](#0-5) 

Because the v1 path remains callable, the v2 mitigation is bypassable by any caller who simply invokes the older method directly.

---

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` to authorize a cross-chain action (bridge withdrawal, token mint, settlement) can be deceived into accepting a forged proof. The attacker does not need to mine Bitcoin blocks or compromise any key. They only need to:

1. Identify a real, confirmed Bitcoin block already tracked by the light client.
2. Compute an internal Merkle node at any depth in that block's transaction tree.
3. Construct a valid proof path from that internal node to the Merkle root.
4. Submit the internal node hash as `tx_id` to `verify_transaction_inclusion`.

The function returns `true`, and any relying contract treats the fabricated transaction as confirmed on-chain.

---

### Likelihood Explanation

**High.** The entry point is a public, permissionless NEAR contract method. The 64-byte transaction attack is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions) and requires no special resources beyond knowledge of a block's transaction set, which is public. The contract itself documents the vulnerability in the function's warning comment, confirming the attack surface is known and unmitigated at the contract level.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract API entirely. If backward compatibility is required for a transition period, gate it with `#[private]` or an access-control role so it cannot be called by unprivileged external accounts. All callers should be migrated to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Let block `B` be any block tracked by the light client with at least 4 transactions `[T0, T1, T2, T3]`.
2. Compute the intermediate Merkle node `N01 = SHA256d(T0 || T1)`.
3. Compute the sibling `N23 = SHA256d(T2 || T3)`.
4. Observe that `SHA256d(N01 || N23) == merkle_root(B)`.
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N01` (an internal node, not a real txid)
   - `tx_block_blockhash = hash(B)`
   - `tx_index = 0` (even position → left child)
   - `merkle_proof = [N23]`
   - `confirmations = 1`
6. Inside `compute_root_from_merkle_proof`: `SHA256d(N01 || N23) == merkle_root` → returns `true`.
7. The contract confirms a transaction that does not exist. [7](#0-6) [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
