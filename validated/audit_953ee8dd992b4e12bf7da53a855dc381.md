### Title
Reentrancy in `submitFastWithdrawal` via Callback-Bearing Token Allows Double-Withdrawal Drain — (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`submitFastWithdrawal` sets `markedIdxs[idx1] = true` before the external token transfer, which prevents replay of `idx1`. However, it provides **no protection against a reentrant call using a distinct, independently valid `idx2`**. There is no `nonReentrant` guard anywhere in `BaseWithdrawPool` or `WithdrawPool`. If the token used for a product triggers a recipient callback on `transfer` (e.g., ERC777 `tokensReceived`), an attacker who controls the `sendTo` address can reenter `submitFastWithdrawal` with a second pre-signed withdrawal index during the first transfer, draining the pool of two withdrawal amounts in a single transaction.

---

### Finding Description

`submitFastWithdrawal` follows a partial CEI pattern: [1](#0-0) 

`markedIdxs[idx1] = true` is written before the transfer, so the same `idx1` cannot be replayed. But the function then calls: [2](#0-1) 

which resolves to: [3](#0-2) 

`ERC20Helper.safeTransfer` issues a low-level `.call` to `token.transfer(to, amount)`: [4](#0-3) 

For a callback-bearing token (ERC777), this invokes `tokensReceived` on the recipient contract before returning. At that moment, `markedIdxs[idx2]` is still `false` and `minIdx` is unchanged (it is only mutated by `submitWithdrawal`, not by `submitFastWithdrawal`): [5](#0-4) 

So a reentrant call to `submitFastWithdrawal(idx2, tx2, sigs2)` passes every guard:
- `!markedIdxs[idx2]` → `true`
- `idx2 > minIdx` → `true`
- `v.requireValidTxSignatures(tx2, idx2, sigs2)` → passes (attacker holds a second valid pre-signed withdrawal)

The second transfer executes, draining an additional `amount` from the pool. Neither `BaseWithdrawPool` nor `WithdrawPool` inherits `ReentrancyGuard` or applies `nonReentrant`: [6](#0-5) 

---

### Impact Explanation

The WithdrawPool's token balance decreases by `2 × amount` while only one withdrawal's worth of liquidity was legitimately owed. Repeated across multiple reentrant depths (if the pool holds enough), the attacker can drain the entire pool balance for that product. This is a direct theft of protocol-held user funds.

---

### Likelihood Explanation

The preconditions are:
1. **Two valid verifier-signed withdrawal transactions** for `idx1` and `idx2` — both are legitimately obtainable by any user with real balances.
2. **A callback-bearing token** registered for a product — `IERC20Base` is a minimal ERC20 interface that does not exclude ERC777 tokens; ERC777 is backward-compatible with ERC20 and could be registered as a supported product token.
3. **Attacker controls `sendTo`** — trivially satisfied by using a contract address as the withdrawal destination (supported via `WithdrawCollateralV2.sendTo`). [7](#0-6) 

---

### Recommendation

Apply OpenZeppelin's `ReentrancyGuardUpgradeable` to `BaseWithdrawPool` and add `nonReentrant` to `submitFastWithdrawal`, `submitWithdrawal`, and `removeLiquidity`. Alternatively, move `handleWithdrawTransfer` to the very end of `submitFastWithdrawal` after all state writes (full CEI), though a mutex is the more robust fix given the `virtual` override surface of `handleWithdrawTransfer`.

---

### Proof of Concept

```
1. Attacker deploys AttackerContract implementing ERC1820 tokensReceived.
2. Attacker obtains verifier-signed (tx1, idx1) and (tx2, idx2) for their subaccount,
   both with sendTo = address(AttackerContract), idx2 > idx1 > minIdx.
3. AttackerContract.attack():
     WithdrawPool.submitFastWithdrawal(idx1, tx1, sigs1)
       → markedIdxs[idx1] = true
       → handleWithdrawTransfer → token.transfer(AttackerContract, amount)
         → tokensReceived callback fires:
             WithdrawPool.submitFastWithdrawal(idx2, tx2, sigs2)
               → markedIdxs[idx2] = false  ✓
               → idx2 > minIdx             ✓
               → markedIdxs[idx2] = true
               → token.transfer(AttackerContract, amount)  ← second drain
         ← callback returns
       ← outer transfer returns
4. Assert: pool token balance decreased by 2 × amount.
   AttackerContract holds 2 × amount.
```
