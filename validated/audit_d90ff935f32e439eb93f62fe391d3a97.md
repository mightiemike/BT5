### Title
Unauthenticated peer can poison `my_public_addr` via `PeersResponse` with spoofed local `PeerId` — (`chain/network/src/peer/peer_actor.rs`)

### Summary

Any peer that completes a T2 handshake (with its own legitimate key) can send a `PeersResponse` whose `direct_peers` list contains a `PeerInfo` with `id` equal to the victim node's `PeerId` and an attacker-controlled `addr`. Because the code checks only `peer_info.id == node_id` with no signature or authentication of the `PeerInfo` contents, and because `my_public_addr` is set on first write only, the attacker can permanently set the victim node's `my_public_addr` to an arbitrary `SocketAddr` before any legitimate peer does.

### Finding Description

`PeerInfo` entries inside a `PeersResponse` are unsigned, unverified data. A node's `PeerId` is its public key and is publicly known (broadcast in routing tables). An attacker who establishes a normal T2 connection can craft:

```
PeersResponse {
    peers: [],
    direct_peers: [PeerInfo { id: <victim_node_id>, addr: Some(<attacker_addr>) }]
}
```

The handler at line 1099 checks `my_public_addr.read().is_none()`, then at line 1101 finds the entry by `peer_info.id == node_id`, and at line 1105 writes `*my_public_addr = Some(addr)` — the attacker-controlled address — with no further validation. [6](#0-5) 

### Impact Explanation

`my_public_addr` is the address the node advertises for Tier3 state sync. If poisoned:
- The victim node advertises the attacker's `SocketAddr` to peers requesting state sync.
- Peers attempting to sync state from the victim are redirected to the attacker-controlled address.
- The victim node itself may fail to participate correctly in Tier3 state sync.
- Because of the first-write-wins guard, the poisoned value persists for the lifetime of the process (until restart).

### Likelihood Explanation

- Requires only a standard T2 peer connection — no special privileges.
- The victim's `PeerId` is publicly available from routing table gossip.
- The attack window is the period between node startup and the first legitimate `PeersResponse` that sets `my_public_addr`. On a freshly started node with no operator-configured address, this window is open until the first honest peer responds.
- The race is easily won by connecting early and sending the spoofed message immediately after handshake.

### Recommendation

Validate that the `addr` in a `PeerInfo` from `direct_peers` was actually observed from the connection itself, not from peer-supplied data. Specifically:
- Only accept `my_public_addr` from the address the remote peer reports seeing for the local node via the TCP connection metadata (e.g., the source address of the incoming connection as seen by the peer), not from an arbitrary `PeerInfo.id` match in a gossip message.
- Alternatively, require the `PeerInfo` entry to be signed by the key matching `peer_info.id` before trusting it for self-address discovery.
- At minimum, cross-validate against multiple independent peers before committing `my_public_addr`.

### Proof of Concept

1. Connect to the victim node as a T2 peer (complete normal handshake with your own key).
2. Immediately send:
   ```
   PeersResponse {
       peers: [],
       direct_peers: [PeerInfo { id: victim_node_id, addr: Some("1.2.3.4:9999") }]
   }
   ```
3. Assert that `network_state.my_public_addr` is now `Some(1.2.3.4:9999)`.
4. Observe that the victim node now advertises `1.2.3.4:9999` for Tier3 state sync, redirecting state sync connections to the attacker-controlled address.
