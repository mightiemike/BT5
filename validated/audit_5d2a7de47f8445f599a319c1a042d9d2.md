Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual user, making per-user allowlist unenforceable for router-mediated swaps - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When `MetricOmmSimpleRouter` calls `pool.swap()`, the pool's `msg.sender` is the router contract, so the extension checks whether the router is allowlisted—not the actual user. If the pool admin allowlists the router to enable router-mediated swaps, any user (including non-allowlisted ones) can bypass the per-user gate by routing through the router.

## Finding Description

**Root cause:** `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient, ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is allowlisted for the calling pool:

```solidity
// SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()`, the pool's `msg.sender` is the router contract address:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [4](#0-3) 

So `sender` arriving at the extension is the router address, not the actual user. The pool admin faces an irresolvable dilemma:

| Configuration | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all |
| Router **allowlisted** | Any user (including non-allowlisted) can swap through the router |

There is no configuration of `SwapAllowlistExtension` that allows specific users to swap through the router while blocking others. The per-user allowlist check is structurally dead code for router-mediated swaps.

## Impact Explanation

Any non-allowlisted user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter`. The pool admin's intended access control policy is silently nullified for all router-mediated swaps. Curated pools (e.g., KYC-gated, institutional, or restricted-access pools) lose their curation guarantee. Unauthorized users can trade against the pool's LPs, which may violate regulatory or contractual obligations and expose LPs to counterparty risk they explicitly opted out of. This constitutes broken core pool functionality (access control) causing potential fund-impacting exposure.

## Likelihood Explanation

Medium. The pool admin must allowlist the router to enable router-mediated swaps for their allowlisted users—this is a natural, expected configuration. Once the router is allowlisted, any unprivileged user can exploit the bypass by simply calling `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the restricted pool. No special privileges or setup are required beyond the pool admin's own natural configuration choice.

## Recommendation

Pass the original user's address through the call chain rather than `msg.sender`. Options include:

1. **Router passes user via `extensionData`:** The router encodes `msg.sender` into `extensionData`, and the extension decodes and checks it. This requires a convention between router and extension.
2. **Separate `swapper` argument:** Add an explicit `swapper` parameter to `pool.swap()` that the router populates with `msg.sender`, and the pool validates that `msg.sender` is an approved router before trusting this value.
3. **Check `tx.origin` as a fallback (not recommended):** Using `tx.origin` is generally discouraged but would correctly identify the EOA initiating the transaction.

The cleanest fix is option 2: the pool accepts a `swapper` address from trusted routers (verified via a factory-maintained registry), and the extension checks that address instead of `sender`.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension configured
// 2. Pool admin calls setAllowedToSwap(pool, alice, true)  — alice is allowlisted
// 3. Pool admin calls setAllowedToSwap(pool, router, true) — router allowlisted to enable router swaps
// 4. Bob (non-allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
// 5. Router calls pool.swap() → pool's msg.sender = router address
// 6. Extension checks allowedSwapper[pool][router] == true → passes
// 7. Bob's swap succeeds despite not being allowlisted
// Assert: bob successfully swapped on a pool he should be blocked from
```

Foundry test outline:
- Deploy `SwapAllowlistExtension`, configure pool with it
- `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, address(router), true)`
- Prank as `bob` (not allowlisted), call `router.exactInputSingle(...)` targeting the pool
- Assert the swap succeeds (no `NotAllowedToSwap` revert)
- Contrast: prank as `bob`, call `pool.swap()` directly → assert it reverts with `NotAllowedToSwap`
