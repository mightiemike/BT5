### Title
Slow-Mode Delay and NLP Lock Period Rely on `block.timestamp` Which Is Unreliable on zkSync, Breaking Censorship-Resistance and Fund-Unlock Guarantees - (File: `core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/common/Constants.sol`)

---

### Summary

Nado is deployed on zkSync. Two critical time-sensitive constants — `SLOW_MODE_TX_DELAY` (3 days) and `NLP_LOCK_PERIOD` (4 days) — are enforced using `block.timestamp`. On zkSync, `block.timestamp` can be identical across consecutive blocks (time can be frozen within a batch), and block production intervals are not time-bounded the way Ethereum's ~12-second blocks are. This means the 3-day slow-mode window and the 4-day NLP lock period are not reliably enforced in wall-clock time, breaking the censorship-resistance guarantee of the slow-mode mechanism and the NLP redemption lock.

---

### Finding Description

`SLOW_MODE_TX_DELAY` and `NLP_LOCK_PERIOD` are defined as fixed second-counts:

```solidity
uint64 constant SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60; // 3 days
uint64 constant NLP_LOCK_PERIOD    = 4 * 24 * 60 * 60; // 4 days
``` [1](#0-0) 

When a user submits a slow-mode transaction (e.g., a withdrawal), `executableAt` is stamped as:

```solidity
executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
``` [2](#0-1) 

The same pattern appears in `depositCollateral`: [3](#0-2) 

The user-facing execution gate checks:

```solidity
require(
    fromSequencer || (txn.executableAt <= block.timestamp),
    ERR_SLOW_TX_TOO_RECENT
);
``` [4](#0-3) 

On zkSync, `block.timestamp` is set by the operator when committing batches. Within a batch, multiple L2 blocks can share the same `block.timestamp`. The operator is only required to keep the timestamp within a loose boundary when committing to L1 — it is not required to advance by one second per block. As a result:

- `block.timestamp` can be frozen across many L2 blocks.
- The wall-clock time elapsed since a slow-mode transaction was submitted may far exceed 3 days, yet `block.timestamp` has not advanced enough to satisfy `txn.executableAt <= block.timestamp`.
- Conversely, if the operator commits a batch with a large timestamp jump, the lock period could expire sooner than intended.

---

### Impact Explanation

The slow-mode mechanism is Nado's **only censorship-resistance path**: if the sequencer refuses to include a user's withdrawal or other transaction, the user submits it as a slow-mode transaction and calls `executeSlowModeTransaction()` after 3 days — without sequencer approval. If `block.timestamp` does not advance reliably, this path is broken:

- A user whose withdrawal is being censored submits a slow-mode transaction.
- 3+ real-world days pass, but `block.timestamp` has not advanced sufficiently on zkSync.
- `executeSlowModeTransaction()` reverts with `ERR_SLOW_TX_TOO_RECENT`.
- The user's funds remain locked with no recourse, as long as the sequencer continues censoring and the timestamp remains frozen.

The NLP lock period (`NLP_LOCK_PERIOD`) is similarly affected: NLP holders who minted and wish to burn after 4 days may find the lock has not expired per `block.timestamp`, or may find it expires prematurely if the operator jumps the timestamp. [5](#0-4) 

---

### Likelihood Explanation

Nado is explicitly deployed on zkSync (confirmed by `hardhat.config.ts` and `block.chainid` checks in `Endpoint.sol`). The zkSync operator controls batch timestamp progression within the allowed boundary. Under normal operation, timestamps advance roughly in real time. However, during periods of low throughput or operator-side issues, timestamps can stall. A malicious or faulty operator can freeze timestamps for extended periods while still producing L2 blocks, directly preventing users from executing slow-mode transactions. This is a realistic scenario on any zkEVM rollup.

---

### Recommendation

1. **Document the zkSync timestamp caveat** explicitly in the contract and protocol documentation: the 3-day and 4-day delays are measured in `block.timestamp` units, which may not correspond to wall-clock time on zkSync.
2. **Consider using an off-chain time oracle or L1 block timestamp** (via zkSync's L1 messenger) as the reference for the slow-mode delay, rather than `block.timestamp`.
3. **Add a maximum staleness bound**: if `block.timestamp` has not advanced within a configurable window, allow slow-mode transactions to be executed unconditionally, preventing indefinite fund lockup.
4. Alternatively, **reduce reliance on `block.timestamp`** for the slow-mode gate by using a submission-count-based or L1-block-number-based delay.

---

### Proof of Concept

1. User calls `submitSlowModeTransaction(withdrawalTx)` on zkSync at L2 block N, where `block.timestamp = T`.
2. `executableAt` is stored as `T + 259200` (3 days in seconds).
3. The zkSync operator produces L2 blocks N+1, N+2, … N+10000, all with `block.timestamp = T` (frozen within a batch or across low-throughput batches).
4. 3 real-world days pass. User calls `executeSlowModeTransaction()`.
5. The check `txn.executableAt <= block.timestamp` evaluates as `T + 259200 <= T` → **false**.
6. The call reverts with `ERR_SLOW_TX_TOO_RECENT`.
7. The user's withdrawal is permanently blocked without sequencer cooperation, defeating the censorship-resistance guarantee. [6](#0-5)
