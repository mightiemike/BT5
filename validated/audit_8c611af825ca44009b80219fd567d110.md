Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass via Router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. Any pool admin who allowlists the router to enable standard periphery UX inadvertently opens the pool to all unpermissioned users, completely defeating the allowlist invariant.

## Finding Description
The pool passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // router address when called via router
  ...
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension (L162-165). `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
// msg.sender = pool, sender = router — the actual user is never checked
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to inject the real caller into the checked field:

```solidity
// MetricOmmSimpleRouter.sol L72-80
IMetricOmmPoolActions(params.pool).swap(
  params.recipient,
  params.zeroForOne,
  ...
  params.extensionData   // user-supplied, not decoded by extension
);
```

The router stores `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), not in any field the extension can read. The extension never decodes `extensionData`. Therefore, `allowedSwapper[pool][router]` is the only check that runs, and it passes for every caller once the router is allowlisted.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict swaps to KYC-verified addresses, whitelisted market makers, or institutional counterparties loses all access control the moment the router is allowlisted. Any EOA or contract can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the public, permissionless router and execute swaps of arbitrary size against the pool's liquidity at oracle-anchored prices. This constitutes a broken core pool access-control extension causing potential direct loss of LP value and a complete admin-boundary break: the pool admin's intended restriction is silently nullified by an unprivileged path.

## Likelihood Explanation
The scenario is highly likely: pool admins who want to support the standard periphery UX must allowlist the router — there is no alternative path for router-mediated swaps. The bypass requires no special privileges, no custom contract, and no unusual token behavior. Any EOA can call `exactInputSingle` directly. The mismatch is invisible from the admin's perspective: `isAllowedToSwap(pool, router)` returns `true` as intended, but the invariant "only allowlisted users can swap" is silently broken for all router-mediated swaps.

## Recommendation
The extension must check the originating user, not the immediate pool caller. Options in increasing robustness:
1. Require the router to encode the real user address in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known intermediary.
2. Add an `originalSender` slot to the router's transient context (analogous to the existing payer slot set by `_setNextCallbackContext`) and expose it via a view function that the extension calls back into during `beforeSwap`.
3. At the pool level, expose a separate `originalSender` field that the router sets in transient storage before calling `swap`, and pass it as an additional argument to extension hooks.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  pool admin calls: swapExtension.setAllowedToSwap(pool, address(router), true)
    (to allow legitimate users to trade via the router)

Attack:
  attacker = address not in allowedSwapper[pool]
  attacker calls:
    router.exactInputSingle({
      pool: pool,
      recipient: attacker,
      zeroForOne: true,
      amountIn: X,
      ...
    })

  Execution:
    router → pool.swap(attacker, true, X, ...)   // msg.sender = router
    pool   → _beforeSwap(sender=router, ...)      // MetricOmmPool.sol L230-231
    ext    → allowedSwapper[pool][router] == true  ← passes (ExtensionCalling.sol L162-165, SwapAllowlistExtension.sol L37)
    swap executes, attacker receives output tokens

Result:
  attacker swaps successfully despite never being individually allowlisted.
  The allowlist guard is completely bypassed for all router-mediated swaps.
```
