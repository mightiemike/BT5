### Title
Unprivileged Caller Can Permanently Prune Mainchain Block Headers via `run_mainchain_gc` - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function with no caller authorization check. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, permanently deleting mainchain block headers from storage, advancing `mainchain_initial_blockhash`, and breaking SPV proof verification for any pruned block.

### Finding Description
The function `run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. The `#[pause]` macro from `near_plugins` gates the function only when the contract is **paused** — it imposes no caller restriction when the contract is running normally. There is no `#[trusted_relayer]`, no role check, and no `env::predecessor_account_id()` guard. [1](#0-0) 

Compare this with `submit_blocks`, which carries both `#[pause]` **and** `#[trusted_relayer]`: [2](#0-1) 

`run_mainchain_gc` permanently removes entries from `headers_pool` and `mainchain_height_to_header`, and advances `mainchain_initial_blockhash`: [3](#0-2) 

Once deleted, these entries cannot be recovered — there is no re-insertion path.

### Impact Explanation
An attacker calling `run_mainchain_gc(u64::MAX)` causes:

1. **SPV proof breakage**: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up `args.tx_block_blockhash` in `mainchain_header_to_height`. After GC, that lookup panics with `"block does not belong to the current main chain"` for any pruned block, making all historical SPV proofs permanently unverifiable. [4](#0-3) 

2. **Chain reorg breakage**: `reorg_chain` walks fork headers back to the common ancestor via `headers_pool`. If the common ancestor was pruned, the walk panics with `"previous fork block should be there"`, permanently breaking the reorg path. [5](#0-4) 

3. **Canonical state corruption**: `mainchain_initial_blockhash` is permanently advanced, destroying the integrity invariant that the stored window covers a contiguous range of the canonical chain. [6](#0-5) 

### Likelihood Explanation
The entry point is a plain public NEAR contract method callable by any account with no deposit or staking requirement. The attacker needs only to know the contract address and call `run_mainchain_gc` with `batch_size = u64::MAX`. No privileged key, no leaked secret, no social engineering is required. The attack is a single transaction.

### Recommendation
Add a caller authorization check at the top of `run_mainchain_gc`, analogous to the `#[trusted_relayer]` guard already used on `submit_blocks`. The simplest fix is to restrict it to accounts holding a designated role (e.g., `Role::DAO` or a new `GCManager` role):

```rust
// Before the existing pause attribute:
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
+   require!(
+       self.acl_has_role("DAO", &env::predecessor_account_id())
+           || self.acl_has_role("UnrestrictedRunGC", &env::predecessor_account_id()),
+       "Unauthorized: caller does not have permission to run GC"
+   );
    ...
}
```

Alternatively, apply a `#[trusted_relayer]`-equivalent macro or a dedicated `#[access_control_any(roles(Role::DAO, Role::GCManager))]` attribute consistent with the project's existing `near_plugins` patterns.

### Proof of Concept

```
// Any unprivileged NEAR account executes this single transaction:
let outcome = attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({ "batch_size": u64::MAX }))
    .max_gas()
    .transact()
    .await?;
// outcome.is_success() == true

// Now any SPV proof for a block below the new mainchain_initial_blockhash panics:
let result = contract
    .view("verify_transaction_inclusion_v2")
    .args_borsh(proof_for_pruned_block)
    .await;
// result.is_err() == true  ("block does not belong to the current main chain")
```

The `batch_size` is bounded only by `std::cmp::min(total_amount_to_remove, batch_size)` at line 393, so passing `u64::MAX` removes every block above `gc_threshold` in a single call — or, if `gc_threshold` itself is large, repeated calls drain the entire window. [7](#0-6)
