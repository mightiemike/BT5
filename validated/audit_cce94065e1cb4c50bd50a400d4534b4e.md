### Title
Fee-Free `depositCollateralWithReferral` Allows Cheap Slow Mode Queue Flooding, Blocking Censorship-Resistant Withdrawals - (File: `core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` inserts entries directly into the strictly-FIFO `slowModeTxs` queue without charging the `SLOW_MODE_FEE` ($1). The only cost is the minimum deposit amount ($0.1 for an existing subaccount), which is **fully recoverable** via the sequencer. Because the queue has no priority mechanism and is drained one entry at a time, an attacker can flood it at near-zero net cost, indefinitely delaying every subsequent slow mode transaction â€” including the `WithdrawCollateral` entries that users depend on as their only censorship-resistant exit path.

---

### Finding Description

The protocol enforces a $1 slow mode fee in `submitSlowModeTransactionImpl` for every non-admin slow mode submission:

```solidity
// EndpointTx.sol lines 369-372
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

`DepositCollateral` is explicitly blocked from this path (`revert()` at line 343-344), because deposits are supposed to go through `depositCollateralWithReferral` instead. However, `depositCollateralWithReferral` adds to the same queue with **no fee**:

```solidity
// Endpoint.sol lines 149-166
// copy from submitSlowModeTransaction
SlowModeConfig memory _slowModeConfig = slowModeConfig;
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: abi.encodePacked(...)
});
slowModeConfig = _slowModeConfig;
```

The only gate is `isValidDepositAmount`, which requires `MIN_DEPOSIT_AMOUNT` ($0.1) for an existing subaccount. That $0.1 is transferred to the clearinghouse and credited to the attacker's own subaccount â€” it is **recoverable** via a normal sequencer-routed withdrawal. The net cost per queue entry is therefore gas only.

The queue is strictly FIFO. `_executeSlowModeTransaction` always pops `slowModeTxs[txUpTo]` with no skip or priority logic:

```solidity
// Endpoint.sol lines 193-194
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];
```

`executeSlowModeTransaction()` (line 231) is callable by anyone but processes exactly one entry per call, and requires `txn.executableAt <= block.timestamp` when called outside the sequencer. There is no mechanism to jump ahead of attacker entries.

---

### Impact Explanation

The slow mode queue is the protocol's censorship-resistance guarantee. When the sequencer refuses to include a user's withdrawal, the user's only recourse is to submit a `WithdrawCollateral` slow mode transaction and wait 3 days before calling `executeSlowModeTransaction()` directly. If the queue ahead of the user's entry contains N attacker-inserted `DepositCollateral` entries, the user must drain all N entries first â€” each requiring a separate on-chain call and each having its own 3-day `executableAt` window.

An attacker who pre-floods the queue with N entries before the user submits can delay the user's withdrawal by an unbounded period. During market stress (price crash, liquidation risk), a user who cannot withdraw collateral suffers direct, quantifiable financial loss: positions cannot be closed, margin cannot be recovered, and liquidation cannot be avoided. This is not gas-only degradation â€” it is asset inaccessibility with a concrete financial consequence.

---

### Likelihood Explanation

The entry point is `public` with no access control beyond the unsanctioned check. Any address holding the minimum deposit token balance can call it. The attacker's capital is not destroyed â€” it is deposited to their own subaccount and retrievable via the sequencer when the attack is complete. The effective cost is gas per call on Ink Chain (an OP-stack L2 with low fees). No privileged role, leaked key, or governance action is required.

---

### Recommendation

1. **Charge the slow mode fee inside `depositCollateralWithReferral`** in addition to the deposit transfer, mirroring the protection in `submitSlowModeTransactionImpl`.
2. Alternatively, **add per-address rate limiting** on slow mode queue insertions (e.g., max N entries per address per time window).
3. Consider a **priority lane** for `WithdrawCollateral` slow mode entries so that user exits cannot be blocked by deposit spam.

---

### Proof of Concept

```
1. Attacker holds 1000 * $0.10 = $100 of the quote token.

2. Attacker calls depositCollateralWithReferral(attackerSubaccount, QUOTE_PRODUCT_ID, MIN_AMOUNT)
   1000 times in rapid succession.
   â†’ Each call: transfers $0.10 to clearinghouse, appends one SlowModeTx to the queue.
   â†’ No SLOW_MODE_FEE charged. Net cost: gas only (funds credited to attacker's subaccount).

3. Victim (censored by sequencer) calls submitSlowModeTransaction(WithdrawCollateral(...)).
   â†’ Victim's entry lands at queue index txCount = 1000.

4. After 3 days, victim calls executeSlowModeTransaction().
   â†’ Processes queue[0] (attacker deposit). Victim's entry is still at index 1000.

5. Victim must call executeSlowModeTransaction() 1000 more times to drain attacker entries
   before their own withdrawal is reached.

6. Attacker recovers $100 via a normal sequencer withdrawal at any time.
   Total attacker cost: gas for 1000 calls (~negligible on Ink Chain L2).
```

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
