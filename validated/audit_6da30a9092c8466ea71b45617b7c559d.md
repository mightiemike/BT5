### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Aliases — (`File: src/chia_dialect.rs`)

---

### Summary

The `ENABLE_SECP_OPS` flag is intended to gate access to the secp256k1 and secp256r1 signature-verification operators. However, the same two operators are unconditionally reachable through their 4-byte opcode encodings (`0x13d61f00` and `0x1c3a8f00`), which are dispatched directly without any flag check. An attacker-controlled CLVM program can invoke full secp signature verification regardless of whether `ENABLE_SECP_OPS` is set, including in `MEMPOOL_MODE`.

---

### Finding Description

`ChiaDialect::op` in `src/chia_dialect.rs` contains two separate dispatch paths for secp operators.

**Path 1 — 1-byte opcodes (flag-gated):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator`, which either errors (`NO_UNKNOWN_OPS`) or executes a no-op cost formula.

**Path 2 — 4-byte opcodes (no flag check):**

```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(...);
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => return unknown_operator(...),
    };
    return f(allocator, argument_list, max_cost, flags);  // no ENABLE_SECP_OPS check
}
```

`op_secp256k1_verify` and `op_secp256r1_verify` are called directly. There is no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard anywhere in this branch. [1](#0-0) [2](#0-1) 

`MEMPOOL_MODE` is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

`ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE`. [3](#0-2) 

The 4-byte opcodes are not routed through `unknown_operator`, so `NO_UNKNOWN_OPS` does not block them either. Both secp operators execute fully in mempool mode when addressed by their 4-byte encoding.

The secp operator implementations themselves perform real cryptographic work and return `nil` on success or `EvalErr::Secp256Failed` on failure — they are not no-ops. [4](#0-3) 

---

### Impact Explanation

**Consensus / mempool divergence.** Any caller that withholds `ENABLE_SECP_OPS` (e.g., a node running before a hardfork activation, or any caller using `MEMPOOL_MODE` via `run_serialized_chia_program`) expects secp operations to be unavailable. A CLVM program using the 4-byte encoding executes the full secp verification anyway. The result — `nil` on valid signature, `EvalErr` on invalid — is identical to what the 1-byte form produces when the flag is set. Two nodes that differ only in `ENABLE_SECP_OPS` will reach the same result for 4-byte-encoded programs but different results for 1-byte-encoded programs, producing a split in accepted/rejected transaction sets.

**Flag contract broken.** The Python API exports `ENABLE_SECP_OPS` as a named constant and passes it through `ClvmFlags::from_bits_truncate` into `ChiaDialect`. [5](#0-4)  Any downstream caller that relies on the absence of `ENABLE_SECP_OPS` to enforce "no secp operations permitted" is silently wrong for 4-byte-encoded programs.

**Exact corrupted invariant:** `flags.contains(ENABLE_SECP_OPS) == false` does not imply `op_secp256k1_verify` / `op_secp256r1_verify` are unreachable. The boolean is false; the operator executes.

---

### Likelihood Explanation

The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source comments and appear in the fuzzing library and the benchmark-cost tool. [6](#0-5)  Any party reading the source or the cost-benchmark binary can derive the encoding. Crafting a CLVM program that uses a 4-byte atom as its operator requires only serializing the 4-byte value as an atom — a trivial operation. The entry path is fully attacker-controlled CLVM bytes submitted through `run_serialized_chia_program`.

---

### Recommendation

Add the same `ENABLE_SECP_OPS` guard to the 4-byte dispatch path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => return unknown_operator(allocator, o, argument_list, flags, max_cost),
};
```

If the 4-byte form is intentionally always available (pre-hardfork legacy), that design decision must be explicitly documented and the `ENABLE_SECP_OPS` flag comment must be corrected to state it only gates the 1-byte aliases, not the operators themselves.

---

### Proof of Concept

Serialize the following CLVM program (pseudocode):

```
; operator atom = 0x13d61f00  (4 bytes)
; args = (pubkey msg sig)
(0x13d61f00 pubkey msg sig)
```

Call `run_serialized_chia_program(program, args, max_cost, flags=MEMPOOL_MODE)`.

- `MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`.
- `op_len == 4`, opcode matches `0x13d61f00`.
- `op_secp256k1_verify` is called directly; no flag check occurs.
- With a valid `(pubkey, msg, sig)` triple, the call returns `Reduction(1300000, nil)` — a successful secp256k1 signature verification — despite `ENABLE_SECP_OPS` being absent.

The same program encoded with 1-byte opcode `64` instead would fall through to `unknown_operator` and, under `NO_UNKNOWN_OPS`, return `EvalErr::Unimplemented`. The two encodings of the same semantic operation produce different acceptance outcomes under identical flags, confirming the broken invariant.

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
