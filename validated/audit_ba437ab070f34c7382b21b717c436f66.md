### Title
Stale Spot Balance Reads in `getStateAndBalance()` and `getBalance()` Due to Missing `_updateState()` Call — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary
`SpotEngineState.getStateAndBalance()` and `getBalance()` return balances computed from stale cumulative interest multipliers because they read `states[productId]` directly from storage without first calling `_updateState()`. This is the direct analog of Compound's `borrowRatePerBlock`/`supplyRatePerBlock` returning stale rates because `accrueInterest` was not called first. In Nado, the stale balances propagate into every health calculation, causing borrowers' actual accrued debt to be understated and their health to be overstated between sequencer `updateStates()` calls.

---

### Finding Description

`SpotEngineState` stores balances in normalized form (`amountNormalized`) and reconstructs real balances by multiplying against `cumulativeDepositsMultiplierX18` or `cumulativeBorrowsMultiplierX18` from the `State` struct. These multipliers are only advanced when the sequencer submits an `updateStates(dt)` transaction, which calls `_updateState()` to compound interest for the elapsed time `dt`.

`getStateAndBalance()` and `getBalance()` are `public view` functions that read `states[productId]` directly from storage without triggering `_updateState()`:

```solidity
// SpotEngineState.sol line 236-244
function getStateAndBalance(uint32 productId, bytes32 subaccount)
    public view returns (State memory, Balance memory)
{
    State memory state = states[productId];                        // stale multipliers
    BalanceNormalized memory balance = balances[productId][subaccount];
    return (state, balanceNormalizedToBalance(state, balance));    // stale real balance
}
```

`balanceNormalizedToBalance()` multiplies `amountNormalized` by whichever stale multiplier applies:

```solidity
// SpotEngineState.sol line 180-192
function balanceNormalizedToBalance(...) internal pure returns (Balance memory) {
    ...
    return Balance(balance.amountNormalized.mul(cumulativeMultiplierX18));
}
```

`_getBalance()` delegates to `getBalance()`, which uses the same stale path:

```solidity
// SpotEngineState.sol line 256-263
function _getBalance(uint32 productId, bytes32 subaccount)
    internal view override returns (int128, int128)
{
    return (getBalance(productId, subaccount).amount, 0);
}
```

`_getBalance()` is called by `BaseEngine._calculateProductHealth()`, which feeds `getHealthContribution()`, which feeds `Clearinghouse.getHealth()`:

```solidity
// BaseEngine.sol line 163-166
(int128 amount, int128 quoteAmount) = _getBalance(productId, subaccount);
int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
health += quoteAmount;
health += amount.mul(weight).mul(risk.priceX18);
```

`Clearinghouse.getHealth()` is `public` and `nonpayable`, callable by any address, and is the function used by the sequencer for all liquidation eligibility checks (`_isAboveInitial`, `isUnderMaintenance`).

By contrast, `PerpEngineState.getStateAndBalance()` and `getBalance()` do apply a zero-delta `_updateBalance()` call in memory to settle funding, so the perp engine partially mitigates this for its own balances. The spot engine has no equivalent mitigation.

---

### Impact Explanation

For a borrower (`amountNormalized < 0`), the true real balance is:

```
actual_debt = amountNormalized × cumulativeBorrowsMultiplierX18_current
```

Because `cumulativeBorrowsMultiplierX18` grows monotonically with time, the stale value is always less than the current value. `getBalance()` therefore returns a debt magnitude that is **smaller than the true debt**. The health contribution computed in `_calculateProductHealth()` is correspondingly overstated — the borrower appears healthier than they actually are.

Concrete corrupted value: the `health` return value of `Clearinghouse.getHealth()` is inflated for any subaccount with an outstanding spot borrow, by an amount proportional to the unaccrued interest since the last `updateStates()` call. This directly corrupts the liquidation eligibility check.

---

### Likelihood Explanation

`updateStates()` is called periodically by the sequencer, not on every block. Any window between two `updateStates()` calls — which can span hours in practice — is a window during which `getBalance()` returns stale data. The sequencer's liquidation transactions use `getHealth()` which reads through this stale path. A borrower whose position crosses the liquidation threshold strictly due to accrued interest (not a price move) will not be identified as liquidatable until the next `updateStates()` call, allowing the position to accumulate additional bad debt. The longer the `updateStates()` interval, the larger the discrepancy.

---

### Recommendation

`getStateAndBalance()` and `getBalance()` in `SpotEngineState` should compute the up-to-date multipliers before reconstructing the real balance, analogous to how `PerpEngineState.getBalance()` calls `_updateBalance(state, balance, 0, 0)` in memory before returning. Specifically, `_updateState()` should be applied to a memory copy of `state` using the elapsed time (`block.timestamp` minus a stored last-update timestamp) before passing the multipliers to `balanceNormalizedToBalance()`. This mirrors the Compound recommendation of calling `accrueInterest` at the start of `borrowRatePerBlock`/`supplyRatePerBlock`.

---

### Proof of Concept

1. At time T₀, sequencer calls `updateStates(dt)`. `cumulativeBorrowsMultiplierX18` is advanced to value M₀.
2. Borrower opens a spot borrow position. `amountNormalized = -N`.
3. Time passes. No `updateStates()` call. True multiplier would be M₁ > M₀.
4. Borrower's true debt = `N × M₁`. Borrower's health should be negative (insolvent).
5. Any caller invokes `Clearinghouse.getHealth(subaccount, MAINTENANCE)`.
6. Call chain: `getHealth()` → `spotEngine.getHealthContribution()` → `_calculateProductHealth()` → `_getBalance()` → `SpotEngineState.getBalance()` → reads `states[productId]` with stale M₀ → returns debt = `N × M₀ < N × M₁`.
7. Health is computed as positive (solvent) using the understated debt.
8. Sequencer's liquidation check passes the borrower as healthy; liquidation is not triggered.

    function balanceNormalizedToBalance(
        State memory state,
        BalanceNormalized memory balance
    ) internal pure returns (Balance memory) {
        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }

    function _calculateProductHealth(
        uint32 productId,
        bytes32 subaccount,
        IProductEngine.HealthType healthType
    ) internal returns (int128 health) {
        RiskHelper.Risk memory risk = _risk(productId);
        (int128 amount, int128 quoteAmount) = _getBalance(
            productId,
            subaccount
        );
        int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
        health += quoteAmount;

        if (amount != 0) {
            if (weight == 2 * ONE) {
                return -INF;
            }
            health += amount.mul(weight).mul(risk.priceX18);
            emit PriceQuery(productId);
        }
    }
