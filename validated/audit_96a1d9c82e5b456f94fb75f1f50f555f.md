Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `pool.swap()` — the direct caller. When `MetricOmmSimpleRouter` is the direct caller, the extension evaluates the router's allowlist entry, not the originating user's. A pool admin who allowlists the router to enable router-mediated swaps for their allowlisted users inadvertently grants all router users the ability to bypass per-user revocations.

## Finding Description
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the extension's caller) and `sender` is whatever the pool passed. `MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`. `ExtensionCalling._beforeSwap` forwards that value unchanged via `abi.encodeCall` to the extension. When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router `msg.sender` inside the pool. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same misbinding applies to `exactOutputSingle`, `exactInput`, and `exactOutput`. There is no mechanism in the router or extension to pass or verify the originating user's identity. `DepositAllowlistExtension` avoids this flaw by checking `owner` (the position owner, supplied explicitly), which is the economically relevant actor regardless of which periphery contract initiates the call.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for allowlisted users, and `setAllowedToSwap(pool, userA, true)`.
3. Pool admin revokes userA: `setAllowedToSwap(pool, userA, false)`.
4. UserA calls `router.exactInputSingle(...)` — extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds despite revocation.

## Impact Explanation
The core invariant of `SwapAllowlistExtension` — "only explicitly permitted addresses may swap" — is broken for every pool that allowlists the router. Any user whose individual swap permission has been revoked can continue trading on a curated pool by routing through `MetricOmmSimpleRouter`. For compliance-gated, institutional-only, or risk-controlled pools, this allows unauthorized principals to move funds through the pool, defeating the entire purpose of the extension. This constitutes broken core pool functionality causing loss of the access-control guarantee the extension is designed to enforce.

## Likelihood Explanation
Medium. The bypass requires the pool admin to have allowlisted the router, which is the only available mechanism to allow allowlisted users to trade via the router. The admin may not realize that allowlisting the router grants access to all router users, not just individually allowlisted ones. Once the router is allowlisted, the bypass is trivially reachable by any user with no further preconditions and is repeatable indefinitely.

## Recommendation
The extension must check the economically relevant actor — the originating user — not the intermediary contract. Two viable approaches:

1. **Extension-data forwarding**: Require the router to encode the originating user's address in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of (or in addition to) `sender`. The pool admin would configure the extension to trust the router as a forwarding agent only.
2. **Separate router-aware allowlist**: Introduce a second allowlist tier keyed by `(pool, router, user)` so the extension can verify both that the router is trusted and that the underlying user is permitted, using the decoded address from `extensionData`.
