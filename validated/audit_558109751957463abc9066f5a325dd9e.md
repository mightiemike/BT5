### Title
`SwapAllowlistExtension.beforeSwap` checks the router's address instead of the end-user's identity, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the immediate `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. If the pool admin allowlists the router address (a natural step to enable router-based swaps for their users), the per-user gate is completely bypassed and any unpermissioned user can swap against the restricted pool.

---

### Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, recipient, ...)`: [1](#0-0) 

`msg.sender` here is whoever called `pool.swap()` — the router when a user goes through the periphery.

**Step 2 — The extension checks that caller-supplied `sender` argument.**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` as the pool key and `sender` (the first argument) as the identity to gate: [2](#0-1) 

**Step 3 — The router is the immediate caller of `pool.swap()`.**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the router `msg.sender` to the pool: [3](#0-2) 

**The identity mismatch:** The extension receives `sender = address(router)`, not the actual end user. The allowlist lookup becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The forced dilemma for pool admins:**

| Admin action | Result |
|---|---|
| Allowlist specific user addresses only | Allowlisted users **cannot** use the router (router not in allowlist → reverts) |
| Allowlist the router address | **Any** user can bypass the per-user gate via the router |
| `setAllowAllSwappers = true` | Defeats the entire purpose of the allowlist |

There is no configuration that simultaneously (a) allows allowlisted users to use the router and (b) blocks non-allowlisted users from using the router.

---

### Impact Explanation

A pool admin who deploys a `SwapAllowlistExtension`-gated pool (e.g., for regulatory KYC compliance or restricted LP access) and allowlists the router to enable router-based swaps for their users inadvertently opens the gate to **all** users. Any unpermissioned address can call `MetricOmmSimpleRouter.exactInputSingle()` and execute swaps against the restricted pool. This constitutes a broken core pool invariant — the allowlist guard is misapplied and fails to gate the economically relevant actor — with direct consequences: unauthorized parties can drain LP positions or execute trades the pool was designed to prohibit.

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router, which is a natural and expected operational step for any pool that wants its allowlisted users to access multi-hop routing. The `MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who enables router-based swaps for their users will inadvertently enable the bypass. The attacker needs no special privileges — a single call to `exactInputSingle()` suffices.

---

### Recommendation

The `SwapAllowlistExtension` should gate the **end user**, not the immediate caller of `pool.swap()`. Two approaches:

1. **Check `recipient` instead of `sender`** — but `recipient` is the output address, not the payer, so this is also wrong for multi-hop.
2. **Pass the original `msg.sender` through `extensionData`** — the router encodes the original caller in `extensionData`, and the extension decodes and checks it. This requires a coordinated change to the router and extension.
3. **Preferred:** The pool's `swap()` interface should expose the original initiator as a dedicated argument (separate from `sender`/`recipient`), and the extension should check that field. Until then, document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used with direct pool calls.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin: setAllowedToSwap(pool, userA, true)       // allowlist userA
  - Admin: setAllowedToSwap(pool, router, true)       // enable router swaps for userA

Attack:
  - userC (not allowlisted) calls:
      router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap() → msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks: allowedSwapper[pool][router] == true → PASSES
  - userC executes an unauthorized swap against the restricted pool

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
