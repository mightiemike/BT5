### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Depositors to Bypass the Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is designed to gate `addLiquidity` by depositor address. However, the `beforeAddLiquidity` hook silently drops the `sender` parameter (the actual caller of `addLiquidity`) and instead validates `owner` (the LP-position recipient). Because `sender` and `owner` can differ — e.g., when a router or multicall contract calls `addLiquidity` on behalf of a user — any address can bypass the allowlist by specifying an allowlisted address as `owner`.

---

### Finding Description

`setAllowedToDeposit` stores permissions under the key `depositor`: [1](#0-0) 

The `beforeAddLiquidity` hook receives `sender` as its first argument but discards it (unnamed `address`). The guard then reads `allowedDepositor[msg.sender][owner]`, where `owner` is the LP-position recipient, not the depositing caller: [2](#0-1) 

The pool's extension-calling layer forwards both `sender` and `owner` as distinct arguments: [3](#0-2) 

Because the hook ignores `sender` entirely, the allowlist check is applied to the wrong principal.

---

### Impact Explanation

Any unprivileged address can call `addLiquidity` on a restricted pool (one protected by `DepositAllowlistExtension`) by passing an allowlisted address as `owner`. The actual depositor — who is not on the allowlist — is never checked. This completely nullifies the deposit-gating invariant the extension is meant to enforce. Pools relying on this extension for compliance, KYC gating, or LP-set restriction have no effective protection.

---

### Likelihood Explanation

The bypass requires only that `sender != owner`, which is the normal case whenever a router, aggregator, or multicall contract is used (the standard interaction pattern in DeFi). No special privileges, flash loans, or exotic tokens are needed. Any user aware of the mismatch can exploit it immediately.

---

### Recommendation

Replace the `owner` check with a `sender` check to match the semantic intent of `allowedDepositor`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict LP-position ownership (not the caller), rename the mapping and setter to `allowedOwner` / `setAllowedToOwn` to make the design explicit and avoid future confusion.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `trustedLP`.
2. `attacker` (not allowlisted) calls `pool.addLiquidity(deltas, extensionData)` with `owner = trustedLP`.
3. The pool calls `extension.beforeAddLiquidity(attacker, trustedLP, ...)`.
4. The hook checks `allowedDepositor[pool][trustedLP]` → `true` → no revert.
5. `attacker` successfully adds liquidity to the restricted pool despite never being allowlisted. [2](#0-1)
