Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user, enabling allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. The pool always passes `msg.sender` (the immediate caller) as `sender`, so when any swap is routed through `MetricOmmSimpleRouter`, the extension checks whether the **router contract** is allowlisted rather than the actual end-user. If the pool admin allowlists the router to enable router-mediated swaps for their permitted users, every unprivileged address can bypass the per-user allowlist by calling the router.

## Finding Description

**Root cause – wrong identity propagated to the extension**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol  line 230-240
_beforeSwap(
    msg.sender,   // ← always the immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol  line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks that forwarded `sender` against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol  line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Router call path**

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol  line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

When this executes, `msg.sender` inside `MetricOmmPool.swap` is the **router address**, not the end-user. The router stores the real user in transient storage only for the payment callback; it is never surfaced to the extension.

**Exploit flow**

1. Pool is deployed with `SwapAllowlistExtension` configured; only specific addresses are meant to swap.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for their permitted users (a natural operational step).
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle` targeting the pool.
4. Pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. Attacker successfully swaps on a pool that was supposed to be restricted to specific addresses.

**Why existing guards are insufficient**

`BaseMetricExtension.onlyPool` only verifies that `msg.sender` is a factory-registered pool; it does not validate the identity of the human initiating the swap. There is no mechanism in the pool or router to pass the original `tx.origin` or user address through to the extension in a trustless way.

## Impact Explanation
Any unprivileged address can trade on a pool whose `SwapAllowlistExtension` is configured to restrict swappers, provided the router is allowlisted (which is the only way to enable router-mediated swaps). The allowlist—a core access-control feature of the extension framework—is rendered completely ineffective for the router path. This constitutes broken core pool functionality and an admin-boundary break where an unprivileged path bypasses an intended access restriction.

## Likelihood Explanation
The condition is realistic: a pool admin who wants to allow their whitelisted users to trade via the standard router must allowlist the router address. The moment they do, the gate is open to everyone. The attacker needs no special capability beyond calling a public router function. The bypass is repeatable on every swap.

## Recommendation
The extension must receive the true end-user identity. Two options:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData` before forwarding to the pool; `SwapAllowlistExtension` decodes and checks it. This requires a trusted router convention and is fragile.
2. **Check `recipient` instead of `sender`** (if the pool's design intent is to gate who receives output): change the allowlist key to `recipient`.
3. **Preferred – dedicated `originalSender` field**: Add an `originalSender` parameter to the `beforeSwap` hook signature that the pool populates from a trusted periphery context (e.g., transient storage written by the router before calling `swap`), and have `SwapAllowlistExtension` check that field.

## Proof of Concept

```solidity
// Foundry test sketch
function test_routerBypassesSwapAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, only `allowedUser` is permitted
    extension.setAllowedToSwap(address(pool), allowedUser, true);
    // Admin allowlists the router so allowedUser can use it
    extension.setAllowedToSwap(address(pool), address(router), true);

    // Attacker (not in allowlist) calls the router
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp + 1,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```
