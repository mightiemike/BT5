Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via Router â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool, which is the pool's own `msg.sender`. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates the router's allowlist status rather than the originating EOA's. Any pool admin who allowlists the router to enable router-mediated swaps for their curated pool inadvertently grants every unprivileged user full swap access, completely defeating the allowlist invariant.

## Finding Description

**Root cause â€” three-step call chain:**

1. `MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

2. `ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever called the pool: [3](#0-2) 

**Router entry point â€” the router calls the pool directly:**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` with no forwarding of the original EOA: [4](#0-3) 

At that point `msg.sender` inside the pool is the router address. The router does store the original caller in transient storage via `_setNextCallbackContext` (L71), but that context is used exclusively for the payment callback â€” it is never surfaced to extensions. The extension has no access to the originating EOA.

**Why existing guards are insufficient:**

The `allowAllSwappers` flag is the only alternative path in the check, and it is a pool-wide open gate. There is no mechanism in the extension to distinguish a direct EOA call from a router-mediated call, nor to read the router's transient payer context. The `extensionData` field is caller-supplied and unverifiable, so it cannot be trusted as a forwarding channel without an explicit convention enforced by the router.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (the necessary step to let approved users trade through the standard UI) creates a condition where `allowedSwapper[pool][router] == true` satisfies the check for every call arriving through the router, regardless of the actual EOA. Any non-allowlisted address can bypass the curation gate with a single `exactInputSingle` call. The entire allowlist invariant â€” that only approved addresses may trade on the pool â€” is broken. This constitutes broken core pool functionality and an admin-boundary break: an unprivileged path (the public router) bypasses an explicit access-control gate set by the pool admin.

## Likelihood Explanation

The router is the primary user-facing swap interface. Any pool admin who wants allowlisted users to access the pool through the router must allowlist the router address â€” this is the natural, expected configuration. The bypass requires no special privileges, no flash loans, and no multi-transaction setup. A single `exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) call from any EOA is sufficient. The precondition is the default operational state for any curated pool that is not exclusively accessed via direct pool calls.

## Recommendation

The extension must gate on the economically relevant actor â€” the EOA who initiated the swap â€” not the intermediate contract that called the pool. Two viable approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` (the EOA) into `extensionData` for each hop, and the extension decodes and checks that value instead of `sender`. This requires a trusted convention between the router and the extension (e.g., a known ABI prefix that the extension verifies was set by a recognized router).

2. **Router-aware transient forwarding**: Add a router-aware path in the extension that reads the real user from the router's transient storage (the payer stored via `_setNextCallbackContext`), similar to how the router already stores payer context. The extension would call back to a trusted router to retrieve the original caller.

The simplest safe default is to document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and must only be used on pools accessed directly, until a forwarding mechanism is implemented.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwapOrder`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` â€” only Alice is intended to be allowed.
3. Admin calls `setAllowedToSwap(pool, router, true)` â€” router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(recipient, ...)` â€” inside the pool, `msg.sender == router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension evaluates `allowedSwapper[pool][router] == true` â†’ check passes.
8. Bob's swap executes successfully despite not being on the allowlist.

Foundry test outline:
```solidity
// Setup
SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
// deploy pool with ext in beforeSwapOrder
ext.setAllowedToSwap(pool, alice, true);
ext.setAllowedToSwap(pool, address(router), true);

// Attack
vm.prank(bob); // bob is NOT allowlisted
router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
// Expect: swap succeeds â€” bob bypassed the allowlist
```
