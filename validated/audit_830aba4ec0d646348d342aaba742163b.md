### Title
Unrestricted `GenericAuthorization` for `MsgTransfer` Allows Grantee to Drain All Granter NFTs — (`x/nft-transfer/types/msgs.go`, `x/nft-transfer/keeper/msg_server.go`, `x/nft-transfer/keeper/packet.go`)

---

### Summary

A grantee holding a `GenericAuthorization` for `MsgTransfer` can submit a `MsgExec` wrapping a `MsgTransfer` where `msg.Sender` is set to the granter's address. Because no `NFTTransferAuthorization` type exists in `x/nft-transfer/types`, there is no mechanism to restrict the grant to specific `classID`s or `tokenID`s. The keeper derives the sender directly from `msg.Sender`, and the only ownership guard (`sender.Equals(owner)`) passes trivially because the granter owns the NFTs. The grantee can therefore transfer every NFT owned by the granter to an attacker-controlled receiver.

---

### Finding Description

**Step 1 — `GetSigners` returns `msg.Sender`**

`MsgTransfer.GetSigners` returns the address parsed from `msg.Sender`: [1](#0-0) 

When the Cosmos SDK authz module processes `MsgExec`, it calls `GetSigners()` on the inner message to identify the granter, then verifies the grantee holds a grant from that granter for that message type. A `GenericAuthorization` for `MsgTransfer` satisfies this check with no further restrictions.

**Step 2 — `keeper.Transfer` derives sender from `msg.Sender` unconditionally** [2](#0-1) 

There is no check that the transaction signer (the grantee) equals `msg.Sender`. The sender passed to `SendTransfer` is always the granter's address.

**Step 3 — Ownership check in `createOutgoingPacket` passes for the granter** [3](#0-2) 

Because `sender` is the granter's address and the granter owns the NFTs, `sender.Equals(owner)` is always `true`. The grantee can specify any `classID` and any `tokenID` owned by the granter.

**Step 4 — No `NFTTransferAuthorization` type exists**

`codec.go` registers only `MsgTransfer` as an `sdk.Msg` implementation. There is no scoped authorization type that would let a granter restrict a grant to specific classes or tokens: [4](#0-3) 

A `GenericAuthorization` for `MsgTransfer` is therefore a blanket grant over every NFT the granter currently owns or will own in the future.

---

### Impact Explanation

A grantee with a single `GenericAuthorization` for `MsgTransfer` can:
- Enumerate all `classID`s and `tokenID`s owned by the granter (public chain state).
- Submit one `MsgExec(MsgTransfer)` per class, setting `msg.Sender` = granter, `msg.Receiver` = attacker address, across any IBC channel.
- Transfer every NFT owned by the granter to an attacker-controlled address on a remote chain, with no per-class or per-token restriction.

This is a complete, irreversible loss of all NFT assets owned by the granter.

---

### Likelihood Explanation

`GenericAuthorization` for `MsgTransfer` is a standard, user-facing authz grant that any user might issue to a trusted delegate (e.g., a relayer, a dApp, a multisig co-signer). The absence of a scoped `NFTTransferAuthorization` means there is no safe way to issue a limited grant. Any such grant is exploitable by the grantee.

---

### Recommendation

Implement an `NFTTransferAuthorization` type (analogous to `SendAuthorization` in `x/bank`) that:
- Restricts grants to a specific list of `(classID, tokenID)` pairs or at minimum a list of `classID`s.
- Decrements or removes the authorization after each successful transfer.
- Registers the type in `codec.go` via `RegisterInterfaces` so it can be used in place of `GenericAuthorization`.

Until this type exists, `GenericAuthorization` for `MsgTransfer` should be explicitly blocked or documented as unsafe.

---

### Proof of Concept

```
1. Granter issues: MsgGrant { granter, grantee, GenericAuthorization{MsgTypeURL: "/nft.transfer.v1.MsgTransfer"} }
2. Granter mints NFTs: classA/token1, classA/token2, classB/token1 (all owned by granter)
3. Grantee submits:
     MsgExec { grantee, msgs: [
       MsgTransfer { Sender: granter, ClassId: "classA", TokenIds: ["token1","token2"], Receiver: attacker, ... },
       MsgTransfer { Sender: granter, ClassId: "classB", TokenIds: ["token1"],          Receiver: attacker, ... },
     ]}
4. Authz: GetSigners() → granter; grant found for grantee → passes.
5. keeper.Transfer: sender = granter; SendTransfer → createOutgoingPacket.
6. createOutgoingPacket: sender.Equals(owner) → true for all tokens → escrow/burn succeeds.
7. All NFTs are now in-flight to attacker's receiver address on the remote chain.
```

func (k Keeper) Transfer(goCtx context.Context, msg *types.MsgTransfer) (*types.MsgTransferResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)

	sender, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return nil, err
	}
	if err := k.SendTransfer(
		ctx, msg.SourcePort, msg.SourceChannel, msg.ClassId, msg.TokenIds,
		sender, msg.Receiver, msg.TimeoutHeight, msg.TimeoutTimestamp,
	); err != nil {
		return nil, err
	}
