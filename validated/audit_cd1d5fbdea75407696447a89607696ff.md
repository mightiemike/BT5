### Title
`P2pSyncClient` Never Registers the `Event` Protocol, Silently Omitting Event Sync While the Server Fully Handles It - (`crates/apollo_p2p_sync/src/client/mod.rs`)

### Summary

The `Protocol` enum defines five P2P sync protocols including `Event`. The `P2pSyncServer` registers and serves all five. The `P2pSyncClient` registers only four — it never registers `Protocol::Event` as a client, never adds an `event_sender` to `P2pSyncClientChannels`, and never creates an event stream. A node syncing via P2P therefore never requests event data from peers, causing `starknet_getEvents` and receipt-level event fields served by the RPC to return authoritative-looking empty or wrong values for P2P-synced blocks.

### Finding Description

The `Protocol` enum in `crates/apollo_p2p_sync/src/lib.rs` enumerates all five sync protocols:

```rust
pub enum Protocol {
    SignedBlockHeader,
    StateDiff,
    Transaction,
    Class,
    Event,          // ← defined
}
``` [1](#0-0) 

`P2pSyncServerChannels` carries a receiver for every protocol, including `event_receiver`:

```rust
pub struct P2pSyncServerChannels {
    header_receiver: HeaderReceiver,
    state_diff_receiver: StateDiffReceiver,
    transaction_receiver: TransactionReceiver,
    class_receiver: ClassReceiver,
    event_receiver: EventReceiver,   // ← present
}
``` [2](#0-1) 

`P2pSyncClientChannels` carries a sender for only four protocols — `Event` is absent:

```rust
pub struct P2pSyncClientChannels {
    header_sender: HeaderSqmrSender,
    state_diff_sender: StateSqmrDiffSender,
    transaction_sender: TransactionSqmrSender,
    class_sender: ClassSqmrSender,
    // no event_sender
}
``` [3](#0-2) 

`create_stream` merges only four streams; no event stream is ever created or merged:

```rust
header_stream.merge(state_diff_stream).merge(transaction_stream).merge(class_stream)
// Event stream is never created or merged
``` [4](#0-3) 

`new_p2p_state_sync_client` in the runner registers four protocols with the `NetworkManager` but never calls `register_sqmr_protocol_client` for `Protocol::Event`:

```rust
let header_client_sender = network_manager
    .register_sqmr_protocol_client(Protocol::SignedBlockHeader.into(), BUFFER_SIZE);
let state_diff_client_sender =
    network_manager.register_sqmr_protocol_client(Protocol::StateDiff.into(), BUFFER_SIZE);
let transaction_client_sender = network_manager
    .register_sqmr_protocol_client(Protocol::Transaction.into(), BUFFER_SIZE);
let class_client_sender =
    network_manager.register_sqmr_protocol_client(Protocol::Class.into(), BUFFER_SIZE);
// Protocol::Event is never registered
``` [5](#0-4) 

By contrast, `new_p2p_state_sync_server` registers all five protocols including `Protocol::Event`: [6](#0-5) 

The server's `run` loop actively handles event queries and serves `DataOrFin<(Event, TransactionHash)>` responses: [7](#0-6) 

### Impact Explanation

A node operating in P2P-sync mode never issues event queries to peers. Event data for P2P-synced blocks is therefore never written to storage. Any subsequent RPC call that reads events from storage — `starknet_getEvents`, `starknet_getTransactionReceipt` (event fields) — returns an authoritative-looking empty or wrong result for those blocks. This matches the **High** impact: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

### Likelihood Explanation

Every node that uses P2P sync (rather than central/feeder-gateway sync) is affected unconditionally. No special transaction type or attacker action is required — the omission is structural. The `Protocol::Event` variant exists, the server is fully wired, and the client registration call is simply absent.

### Recommendation

In `new_p2p_state_sync_client` (`crates/apollo_state_sync/src/runner/mod.rs`), register the event protocol client and thread the resulting sender through `P2pSyncClientChannels`:

```rust
let event_client_sender =
    network_manager.register_sqmr_protocol_client(Protocol::Event.into(), BUFFER_SIZE);
```

Add `event_sender` to `P2pSyncClientChannels`, implement an `EventStreamBuilder` analogous to `ClassStreamBuilder`, and merge the resulting event stream in `create_stream`.

### Proof of Concept

1. Start a node configured for P2P sync only (no central sync).
2. Let it sync several blocks from a peer that has event-emitting transactions.
3. Call `starknet_getEvents` with a filter covering those blocks.
4. Observe that the response is empty despite the peer serving events correctly via the `Event` protocol — because the client never issued an `EventQuery`.
