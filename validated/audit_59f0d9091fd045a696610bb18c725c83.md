### Title
`paid_fee_on_l1` Dropped and Hardcoded to `Fee(1)` in Consensus L1Handler Conversion — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `paid_fee_on_l1` field of an L1 handler transaction is structurally absent from `ConsensusTransaction::L1Handler` and the protobuf `L1HandlerV0` wire format. When the validator side reconstructs an `InternalConsensusTransaction::L1Handler` from a received consensus message, `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally substitutes `Fee(1)` for the real fee paid on L1. The proposer executes the same transaction with the authentic `paid_fee_on_l1` scraped from L1. This is a direct analog of the external report's "missing value forwarding" pattern: a critical field is silently zeroed/replaced at a conversion boundary, causing the two sides of a cross-component call to operate on divergent values.

### Finding Description

**Type hierarchy mismatch at the consensus boundary**

`ConsensusTransaction` and `InternalConsensusTransaction` use different underlying types for L1 handler transactions:

```
ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction)
    ↑ no paid_fee_on_l1 field

InternalConsensusTransaction::L1Handler(executable_transaction::L1HandlerTransaction)
    ↑ has paid_fee_on_l1: Fee
``` [1](#0-0) 

`executable_transaction::L1HandlerTransaction` carries `paid_fee_on_l1`: [2](#0-1) 

**Proposer-side: real fee is present**

The L1 events scraper populates `paid_fee_on_l1` from the actual ETH value sent to the Starknet core contract on L1: [3](#0-2) 

**Outbound conversion: `paid_fee_on_l1` is silently dropped**

When the proposer converts its `InternalConsensusTransaction::L1Handler` to a `ConsensusTransaction::L1Handler` for broadcast, only `tx.tx` (the inner `transaction::L1HandlerTransaction`) is kept; `paid_fee_on_l1` is discarded: [4](#0-3) 

**Protobuf wire format: field is absent**

The protobuf `L1HandlerV0` message carries only `nonce`, `address`, `entry_point_selector`, and `calldata` — no `paid_fee_on_l1`: [5](#0-4) 

**Inbound conversion: hardcoded `Fee(1)` substituted**

When a validator reconstructs the executable transaction from the received `ConsensusTransaction::L1Handler`, the converter unconditionally injects `Fee(1)`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [6](#0-5) 

**Blockifier fee check**

The blockifier's `execute_raw` for L1 handler transactions checks `paid_fee_on_l1` after execution:

```rust
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [7](#0-6) 

The comment explicitly acknowledges the check is intentionally weak: "For now, assert only that any amount of fee was paid." `Fee(1)` always satisfies `!= Fee(0)`, so the substitution currently bypasses the check silently.

### Impact Explanation

**Divergent executable payload between proposer and validators.** The proposer executes every L1 handler with the authentic `paid_fee_on_l1` from L1 (e.g., `Fee(1_000_000_000)`). Every validator executes the same transaction with `Fee(1)`. The two sides of the consensus protocol operate on structurally different `executable_transaction::L1HandlerTransaction` objects for the same block.

Under the current weak check (`paid_fee != 0`), both sides reach the same execution outcome, so no state divergence is observable today. However:

1. **Latent consensus safety hazard.** If the fee check is tightened to `paid_fee >= actual_fee` (the semantically correct check, and the direction the TODO comment points), the proposer and validators will produce divergent revert/success results for any L1 handler whose real `paid_fee_on_l1` is below the actual execution cost. This is a "wrong revert result from blockifier execution logic" impact.

2. **Broken conversion canonicalization invariant.** The round-trip `InternalConsensusTransaction::L1Handler → ConsensusTransaction::L1Handler → InternalConsensusTransaction::L1Handler` is not lossless. The `paid_fee_on_l1` field is permanently destroyed at the first step and replaced with a synthetic constant at the last step. Any future logic that reads `paid_fee_on_l1` from the validator's reconstructed transaction (e.g., for receipt generation, OS hints, or fee accounting) will receive the wrong value.

3. **Incorrect fee accounting in receipts.** The `CendePreconfirmedTransaction::L1Handler` conversion already drops `paid_fee_on_l1` from the preconfirmation payload sent to the Cende service: [8](#0-7) 

### Likelihood Explanation

Every L1 handler transaction that passes through the consensus path (proposer → protobuf broadcast → validator reconstruction) triggers this substitution. The trigger is unconditional and requires no special attacker input — it fires for every legitimate L1-to-L2 message included in a block proposal. The TODO comment confirms the developers are aware the value is wrong and intend to fix it.

### Recommendation

1. Add `paid_fee_on_l1` to `ConsensusTransaction::L1Handler`. Change the variant to hold `executable_transaction::L1HandlerTransaction` (which already carries the field) instead of `transaction::L1HandlerTransaction`.

2. Add a `paid_fee` field to the protobuf `L1HandlerV0` message and update the `From`/`TryFrom` converters in `crates/apollo_protobuf/src/converters/transaction.rs` to serialize and deserialize it.

3. Remove the hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` and forward the real value from the deserialized consensus transaction.

4. Add a round-trip test asserting that `paid_fee_on_l1` is preserved across `InternalConsensusTransaction → ConsensusTransaction → InternalConsensusTransaction`.

### Proof of Concept

Trace the value through the production code path:

**Step 1 — Proposer creates `InternalConsensusTransaction::L1Handler` with real fee (e.g., from L1 events scraper):**
```
InternalConsensusTransaction::L1Handler(
    executable_transaction::L1HandlerTransaction { paid_fee_on_l1: Fee(1_000_000), ... }
)
```

**Step 2 — `convert_internal_consensus_tx_to_consensus_tx` strips the fee:**
```rust
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))  // tx.paid_fee_on_l1 is gone
}
``` [4](#0-3) 

**Step 3 — Protobuf serialization (`From<L1HandlerTransaction> for protobuf::L1HandlerV0`) omits the field entirely:** [5](#0-4) 

**Step 4 — Validator deserializes and calls `convert_consensus_tx_to_internal_consensus_tx`:**
```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    // internal_tx.paid_fee_on_l1 == Fee(1), not Fee(1_000_000)
}
``` [9](#0-8) 

**Step 5 — Blockifier on validator side executes with `Fee(1)`; check passes only because the threshold is `!= 0`:** [7](#0-6) 

The proposer executed with `Fee(1_000_000)`; the validator executed with `Fee(1)`. The executable payloads are divergent. When the fee check is tightened to enforce `paid_fee >= actual_fee`, any L1 handler whose real fee is below the actual execution cost will revert on the proposer but succeed on validators (or vice versa), breaking consensus safety.
