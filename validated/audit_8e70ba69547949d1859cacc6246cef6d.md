Audit Report

## Title
Division-Before-Multiplication in `_metrics` Causes Precision Loss That Can Falsely Trigger Stop-Loss, Blocking Legitimate Swaps — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._metrics()` uses nested `Math.mulDiv` calls where the inner call performs a floor division before the outer multiplication by `METRIC_SCALE`, discarding the fractional component permanently. This causes the computed per-share metric to be systematically underestimated by up to `METRIC_SCALE / shares`. When the true metric sits just above the drawdown floor, the truncated value can fall below it, causing `_checkAndUpdateWatermarks` to revert with `OracleStopLossTriggered` on a legitimate swap, blocking that swap direction until admin intervention.

## Finding Description

In `_metrics` at lines 254–255:

```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
```

The inner `Math.mulDiv(uint256(t1), Q64, midPriceX64)` computes `floor(t1 × Q64 / midPriceX64)`, discarding the fractional remainder. The outer `Math.mulDiv(..., METRIC_SCALE, shares)` then multiplies the already-truncated integer, so the fractional part is never recovered. The correct computation is:

```
t1 × Q64 × METRIC_SCALE / (midPriceX64 × shares)
```

but the code computes:

```
floor(t1 × Q64 / midPriceX64) × METRIC_SCALE / shares
```

The maximum absolute error is `METRIC_SCALE / shares`, reaching up to `1e6` when `shares = minimalMintableLiquidity = 1`.

The underestimated metric is then passed to `_applyWatermark` at line 334:

```solidity
breached = metric < (hwm * floorMultiplier) / E6;
```

If the computed metric falls below `hwm × (1 − drawdown)` while the true metric does not, `_checkAndUpdateWatermarks` reverts with `OracleStopLossTriggered` at lines 271–273 or 276–278, reverting the entire swap.

No existing guard compensates for this truncation: `_clampMetric` only caps at `METRIC_MAX` and does not protect against underestimation, and `_applyWatermark` performs a direct comparison with no rounding tolerance.

## Impact Explanation

The falsely-triggered stop-loss causes the pool's `afterSwap` hook to revert, making the affected swap direction (`zeroForOne` for `metricT0`, `!zeroForOne` for `metricT1`) permanently unusable until the watermark decays below the underestimated metric or an admin resets the watermark. This constitutes broken core pool functionality — a swap direction is rendered unusable by a precision artifact rather than a genuine drawdown event.

## Likelihood Explanation

The false positive requires the true per-share metric to lie in the window `[hwm × (1 − drawdown), hwm × (1 − drawdown) + METRIC_SCALE/shares)`. This window is widest (up to `1e6` units) when `shares = minimalMintableLiquidity = 1`. Pools configured with tight drawdowns (1–10%) and low minimum liquidity are most exposed. The condition is reachable by any unprivileged user calling `swap` when the bin's true metric is near the floor — no special privileges or setup beyond normal pool operation are required.

## Recommendation

Delay all divisions to the final step by using a single three-argument `mulDiv` that keeps the full numerator precision:

```solidity
uint256 t1InT0Scaled = Math.mulDiv(uint256(t1), Q64 * METRIC_SCALE, midPriceX64);
metricT0 = _clampMetric(t0ps + t1InT0Scaled / shares);

uint256 t0InT1Scaled = Math.mulDiv(uint256(t0), midPriceX64 * METRIC_SCALE, Q64);
metricT1 = _clampMetric(t0InT1Scaled / shares + t1ps);
```

`Q64 * METRIC_SCALE = 2^64 × 1e6 ≈ 1.8e25` fits in `uint256`, and with `t1 ≤ 2^104` the numerator `t1 × Q64 × METRIC_SCALE < 2^256`, so no overflow occurs. The denominator `midPriceX64 × shares` should be checked for overflow; if it can exceed `uint256`, use a two-step fallback.

## Proof of Concept

**Setup:**
- Pool with `minimalMintableLiquidity = 1`, `drawdownE6 = 100_000` (10%).
- Bin 0: `t0 = 0`, `t1 = 1`, `totalShares = 1`.
- `midPriceX64 = 3000 × 2^64`.
- Admin sets watermark `hwm0 = 300`.

**Current code computation:**
```
inner = Math.mulDiv(1, 2^64, 3000 × 2^64) = floor(1/3000) = 0
metricT0 = 0 + Math.mulDiv(0, 1e6, 1) = 0
```

**True metric:**
```
true_metricT0 = 1 × 2^64 × 1e6 / (3000 × 2^64 × 1) = 1e6/3000 ≈ 333
```

**Stop-loss check:**
```
floor = 300 × (1e6 − 100_000) / 1e6 = 270
computed metricT0 = 0 < 270  →  OracleStopLossTriggered fires (false positive)
true metricT0     ≈ 333 > 270 →  no breach (correct behavior suppressed)
```

Any `zeroForOne` swap on this pool reverts. The pool's sell direction is blocked until admin resets the watermark. A Foundry unit test can reproduce this by deploying the extension with the above parameters, setting the watermark to 300 via the admin setter, then calling `swap` with `zeroForOne = true` and asserting the revert.
