### Title
Zero-Value Approval Causes Permanent DoS for ERC4626 Vault Products with Revert-on-Zero-Approval Underlying Assets - (File: `core/contracts/ContractOwner.sol`)

### Summary

`ContractOwner.wrapVaultAsset` unconditionally calls `assetToken.approve(tokenAddr, 0)` before setting the real allowance. For underlying asset tokens that revert on zero-value approvals (e.g., BNB), this call always reverts, permanently bricking the vault-wrapping flow for those products.

### Finding Description

In `ContractOwner.wrapVaultAsset`, after pulling the raw asset from the `DirectDepositV1` contract, the code resets the allowance to zero before granting the actual deposit allowance:

```solidity
IERC20Base assetToken = IERC20Base(assetTokenAddr);
assetToken.approve(tokenAddr, 0);   // ← reverts for BNB-like tokens
assetToken.approve(tokenAddr, assetBalance);
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
``` [1](#0-0) 

The zero-value `approve` is a standard "safe approval" pattern intended to clear a prior allowance before setting a new one. However, tokens such as BNB revert when the approval amount is exactly `0`. Because the call is unconditional and there is no try/catch or skip-if-zero guard, any ERC4626 product whose underlying asset exhibits this behaviour will always revert at this line.

`wrapVaultAsset` carries no `onlyOwner` or similar access modifier — it is a plain `external` function callable by any address:

```solidity
function wrapVaultAsset(bytes32 subaccount, uint32 productId) external {
``` [2](#0-1) 

### Impact Explanation

Any ERC4626 vault product registered in `SpotEngine` whose `asset()` is a token that reverts on zero-value approvals cannot be deposited through the `wrapVaultAsset` path. The entire function reverts, so users can never convert their raw asset balance (held in the `DirectDepositV1` contract) into the wrapped vault token required for collateral deposit. The affected product is effectively non-functional for all users.

### Likelihood Explanation

The likelihood is medium. The Nado protocol explicitly supports ERC4626 vault products via `spotEngine.getToken(productId)` and the `IERC4626Base` interface. [3](#0-2)  If any such vault is deployed with BNB (or another revert-on-zero-approval token) as its underlying asset, the bug is deterministically triggered on every call to `wrapVaultAsset`. No special attacker action is required — the revert happens for every legitimate user attempting to use the flow.

### Recommendation

Skip the zero-value approval when the current allowance is already zero, or check the current allowance before resetting it:

```solidity
IERC20Base assetToken = IERC20Base(assetTokenAddr);
uint256 currentAllowance = assetToken.allowance(address(this), tokenAddr);
if (currentAllowance != 0) {
    assetToken.approve(tokenAddr, 0);
}
assetToken.approve(tokenAddr, assetBalance);
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
```

Alternatively, use OpenZeppelin's `SafeERC20.forceApprove` which already handles this edge case correctly, or simply call `approve(tokenAddr, assetBalance)` directly without the preceding zero-reset (since the prior allowance is always 0 at this point — no prior approval was granted in this execution path).

### Proof of Concept

1. A vault product is registered in `SpotEngine` where `tokenAddr` is an ERC4626 vault whose `asset()` returns the BNB token address.
2. A user sends BNB tokens to their `DirectDepositV1` contract address.
3. Any caller invokes `ContractOwner.wrapVaultAsset(subaccount, productId)`.
4. The function passes the `previewDeposit` guard at line 525 (balance > 0).
5. `DirectDepositV1.withdraw` transfers BNB tokens to `ContractOwner` (line 526–528).
6. Execution reaches `assetToken.approve(tokenAddr, 0)` at line 530.
7. BNB's `approve` reverts because the value is `0`.
8. The entire transaction reverts; the BNB tokens are now stranded in `ContractOwner` with no recovery path through this function. [4](#0-3)
