Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and enforces the allowlist only against `owner`, a free caller-supplied argument. Because any caller can name any allowlisted address as `owner`, the allowlist check always passes for any non-allowlisted address that knows one allowlisted address, completely nullifying the access-control boundary the extension is designed to enforce.

## Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook with `msg.sender` as the first argument and the caller-supplied `owner` as the second:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
//                  ^^^^^^^^^^  ^^^^^
//                  actual caller  caller-supplied beneficiary
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both addresses: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` then ignores the first argument entirely (unnamed `address`) and enforces the allowlist only on `owner`: [2](#0-1) 

Since `owner` is a free parameter supplied by the caller, any non-allowlisted address can pass the check by naming any allowlisted address as `owner`. The `isAllowedToDeposit` view function names its second parameter `depositor`, confirming the intended semantic is to gate the actual depositing address: [3](#0-2) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), not the recipient: [4](#0-3) 

The asymmetry is the root cause: `SwapAllowlistExtension` gates the actual actor; `DepositAllowlistExtension` gates only the beneficiary label, which the actor controls freely.

## Impact Explanation

The deposit allowlist is completely bypassed. Any address not on the allowlist can deposit into a pool configured with `DepositAllowlistExtension` by supplying any allowlisted address as `owner`. The pool admin's access-control boundary is fully nullified: the core invariant of the extension — that only allowlisted addresses may add liquidity — is broken. This constitutes an admin-boundary break where an unprivileged path circumvents an admin-configured guard, allowing unauthorized liquidity positions to be created in pools intended to be restricted.

## Likelihood Explanation

Exploitation requires only: (1) knowing any one allowlisted address, which is observable on-chain from prior `setAllowedToDeposit` events or successful deposits, and (2) calling `addLiquidity` with that address as `owner` while providing the required tokens via the callback. No special privileges, flash loans, or oracle manipulation are needed. Any EOA or contract can execute this in a single transaction, and it is repeatable indefinitely.

## Recommendation

Replace the unnamed first argument with `sender` and enforce the allowlist against it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Deploy a pool with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` and leaves `bob` unlisted.
3. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. `beforeAddLiquidity` is invoked with `sender = bob` (ignored), `owner = alice`.
5. The check evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob` satisfies the token-transfer callback; the position is minted under `alice`.
7. `bob` has successfully deposited into the allowlist-restricted pool despite being explicitly excluded.
