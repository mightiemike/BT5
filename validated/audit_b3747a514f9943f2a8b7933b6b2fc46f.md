### Title
Merkle Leaf/Internal-Node Hash Collision Allows SPV Proof Forgery — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_hash` in `merkle-tools` applies `double_sha256` to a raw 64-byte concatenation of two child hashes with no domain-separation prefix. Because Bitcoin's Merkle tree uses the same hash operation for both leaf nodes (transaction hashes) and internal nodes (pairs of child hashes), a 64-byte crafted transaction is cryptographically indistinguishable from an internal node. The still-callable `verify_transaction_inclusion` function accepts a single Merkle path and has no defence against this collision, allowing any unprivileged NEAR caller to forge a valid-looking SPV proof for a transaction that was never included in a Bitcoin block.

---

### Finding Description

`compute_hash` concatenates two 32-byte hashes into a 64-byte buffer and hashes the result: [1](#0-0) 

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);
    double_sha256(&concat_inputs)
}
```

This is identical to how Bitcoin hashes a raw 64-byte transaction: `double_sha256(raw_tx_bytes)`. There is no leaf-prefix or internal-node-prefix to distinguish the two cases.

`compute_root_from_merkle_proof` calls `compute_hash` at every level of the path without any such distinction: [2](#0-1) 

`verify_transaction_inclusion` (the original, non-v2 function) accepts a `ProofArgs` struct that contains only a single Merkle path — no coinbase proof, no path-length cross-check: [3](#0-2) 

Because the function is still present in the contract (confirmed by five references in `contract/src/lib.rs`) and is a public NEAR entry point, any caller can invoke it. The function computes the Merkle root from the supplied path and compares it to the stored header's Merkle root — nothing more.

---

### Impact Explanation

An attacker who knows the Merkle tree of any confirmed Bitcoin block can:

1. Pick any internal node `N` at depth `d` in the tree.
2. Construct a 64-byte raw transaction whose `double_sha256` equals `N` (this is the known CVE-2012-2459 / BIP-0098 attack).
3. Submit a `ProofArgs` with `tx_id = hash(crafted_tx)`, a Merkle path of length `d` that terminates at `N`, and the real block hash.
4. `verify_transaction_inclusion` returns `true`.

The contract then attests that a fabricated transaction was included in a real Bitcoin block. Any downstream NEAR dApp that calls `verify_transaction_inclusion` and acts on its result (e.g., releasing funds, minting tokens) is defrauded.

---

### Likelihood Explanation

- The entry point is a public, permissionless NEAR contract function.
- The required Bitcoin block data (Merkle tree, internal nodes) is fully public.
- Constructing a 64-byte transaction that hashes to a chosen internal node is the well-documented CVE-2012-2459 technique; no cryptographic break is required.
- The only barrier is that `verify_transaction_inclusion` is labelled "deprecated" in documentation, but it remains callable on-chain.

---

### Recommendation

1. **Remove `verify_transaction_inclusion`** from the contract entirely, or add a hard `panic!` / `require!(false, "deprecated")` guard so it can never return `true`.
2. In `compute_hash`, add a domain-separation prefix distinguishing leaf nodes from internal nodes (e.g., `0x00` prefix for leaves, `0x01` for internal nodes), mirroring RFC 6962 / Bitcoin's own BIP-0098 proposal.

```rust
fn compute_leaf_hash(tx_hash: &H256) -> H256 {
    let mut buf = Vec::with_capacity(33);
    buf.push(0x00u8);
    buf.extend(tx_hash.0);
    double_sha256(&buf)
}

fn compute_internal_hash(left: &H256, right: &H256) -> H256 {
    let mut buf = Vec::with_capacity(65);
    buf.push(0x01u8);
    buf.extend(left.0);
    buf.extend(right.0);
    double_sha256(&buf)
}
```

### Proof of Concept

Given a real Bitcoin block with Merkle tree:

```
        Root
       /    \
      A      B        <- internal nodes (each is double_sha256 of two 32-byte children)
     / \    / \
   T0  T1  T2  T3    <- leaf transaction hashes
```

Internal node `A = double_sha256(T0 || T1)` is a 32-byte value. Craft a 64-byte raw transaction `crafted_tx` such that `double_sha256(crafted_tx) == A`. Submit to `verify_transaction_inclusion`:

```
ProofArgs {
    tx_id:              double_sha256(crafted_tx),  // == A
    tx_block_blockhash: <real block hash>,
    tx_index:           0,
    merkle_proof:       [B],                        // one-element path: A + B = Root
    confirmations:      6,
}
```

`compute_root_from_merkle_proof(A, 0, [B])` computes `compute_hash(A, B) = Root`. The stored header's `merkle_root` equals `Root`. The function returns `true` for a transaction that does not exist. [1](#0-0) [4](#0-3) [3](#0-2)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
