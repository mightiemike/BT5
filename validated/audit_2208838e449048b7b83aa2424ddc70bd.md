### Title
Linked Signer Can Overwrite Its Own Delegation via `LinkSigner` — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in the sequencer path (`processTransactionImpl`) is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to replace itself with any address. Only the subaccount owner should be able to change the linked signer, but the missing self-check allows a compromised linked signer to permanently hijack signing authority over a subaccount.

---

### Finding Description

In `EndpointTx.sol`, the sequencer-path handler for `LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier: [2](#0-1) 

The verifier accepts a signature from **either** the subaccount owner address **or** the linked signer: [3](#0-2) 

This means the current linked signer can produce a valid `SignedLinkSigner` transaction that replaces `linkedSigners[sender]` with any address of the attacker's choosing.

By contrast, the slow-mode path for `LinkSigner` correctly enforces that only the subaccount owner (`msg.sender`) can change the linked signer: [4](#0-3) 

The asymmetry between the two paths reveals the missing invariant check: the sequencer path should also reject `LinkSigner` transactions signed by the current linked signer (i.e., `require(recovered == address(uint160(bytes20(sender))))`), or equivalently, `LinkSigner` should be validated with `allowLinkedSigner = false`.

The `linkedSigners` mapping is stored in `EndpointStorage`: [5](#0-4) 

---

### Impact Explanation

A compromised linked signer can:
1. Sign a `LinkSigner` transaction pointing `linkedSigners[victim_subaccount]` to an attacker-controlled address.
2. Immediately use that new address to sign `WithdrawCollateral` or `WithdrawCollateralV2` transactions (both validated with `allowLinkedSigner = true`) to drain the subaccount.

The subaccount owner's only recovery path is the slow-mode queue, which has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window the attacker retains full signing authority and can execute withdrawals before the owner can revoke the linked signer. [6](#0-5) 

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets used for high-frequency trading. Hot wallets have materially higher key-compromise risk than cold-storage subaccount owners. Any user who has linked a signer is exposed. The attack requires only a single sequencer-submitted transaction from the compromised key — no admin access, no governance, no oracle manipulation.

---

### Recommendation

Change the `LinkSigner` handler in `processTransactionImpl` to pass `allowLinkedSigner = false` to `validateSignedTx`, so that only the subaccount owner's key can authorize a signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only owner may change linked signer
);
```

This mirrors the invariant already enforced in the slow-mode path via `validateSender`.

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...0001` and links hot wallet `0xHot` via `LinkSigner`.
2. Attacker compromises `0xHot`.
3. Attacker constructs `SignedLinkSigner { sender: 0xAlice000...0001, signer: 0xAttacker, nonce: N }` and signs it with `0xHot`.
4. Attacker submits the transaction to the sequencer. `validateSignedTx` passes because `recovered == linkedSigner` (`0xHot`) satisfies the verifier check at `Verifier.sol:300-301`.
5. `linkedSigners[0xAlice000...0001]` is now set to `0xAttacker`.
6. Attacker immediately signs a `WithdrawCollateral` transaction with `0xAttacker`, draining Alice's collateral before she can submit a slow-mode `LinkSigner` to revoke. [7](#0-6) [8](#0-7)

```text
    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
```
