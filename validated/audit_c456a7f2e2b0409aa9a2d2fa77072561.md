## Title

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as the arithmetic mean of bid and ask at line 218, while `MetricOmmPool.swap` computes the oracle mid-price via `SwapMath.midAndSpreadFeeX64FromBidAsk` (geometric mean) at line 243. By AM-GM inequality, arithmetic mean â‰¥ geometric mean for any non-zero spread, causing the stop-loss to overestimate `metricToken1` and underestimate `metricToken0` relative to the pool's actual settled prices. This allows the guard to silently pass even when the true per-share value has fallen below the configured drawdown floor, leaking LP principal beyond the intended protection boundary.

**Pool swap path â€” geometric mean:**

`MetricOmmPool.swap` calls `_getBidAndAskPriceX64()` and then at lines 242â€“243 passes the result to `SwapMath.midAndSpreadFeeX64FromBidAsk`:


**Stop-loss afterSwap hook â€” arithmetic mean:**

`OracleValueStopLossExtension._afterSwapOracleStopLoss` receives the same `bidPriceX64`/`askPriceX64` forwarded by the pool but computes mid differently:


This arithmetic mid is fed into `_metrics` at lines 254â€“255:


For any spread > 0: `arith_mid > geo_mid`, so `metricToken1` (which includes `t0 Ă— mid / shares`) is inflated relative to the pool's actual geometric-mid value.

**Watermark ratchet and breach check:**

The watermark for token1 is ratcheted up to the inflated arithmetic metric at lines 275â€“278. The floor is `hwm1 Ă— floorMultiplier / E6`. When the spread is non-zero, the computed `metricToken1` remains inflated relative to the true geometric-mid value, so the guard can fail to detect a real breach.

**Exploit path:**
1. Watermark established during a period of small spread (arithmetic â‰ˆ geometric). `hwm1` is set close to the true geometric value.
2. Oracle spread widens (volatile market, oracle latency).
3. Attacker executes a `!zeroForOne` swap that causes value leakage.
4. Stop-loss computes `metricToken1` using arithmetic mid (now materially higher than geometric mid).
5. Computed `metricToken1` remains above `hwm1 Ă— floorMultiplier / E6` even though the true geometric-mid value has fallen below the floor.
6. `OracleStopLossTriggered` is not emitted; the swap is not reverted; LP principal leaks beyond the configured drawdown.

No existing guard checks for this mid-price inconsistency. The `_requireInitialized` check only validates pool registration, not price computation correctness.

## Impact Explanation

The stop-loss is the primary on-chain protection for LP capital against value leakage. When it fails to trigger, LPs lose principal beyond the drawdown they accepted. The magnitude of the error scales with the spread: a 20% spread produces ~1% metric inflation. For pools configured with tight drawdowns (e.g., 1â€“5%), a 10â€“20% oracle spread can suppress the guard entirely at the threshold, constituting a direct loss of LP principal above the Sherlock medium threshold.

## Likelihood Explanation

Any pool deploying `OracleValueStopLossExtension` is affected whenever the oracle spread is non-zero. Pools on volatile pairs or during periods of oracle stress â€” the exact conditions the stop-loss is meant to guard against â€” experience the largest spread and therefore the largest guard error. No privileged action is required; any public swap at the right moment triggers the path.

## Recommendation

Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (line 218):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After:
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This makes the stop-loss metrics consistent with the prices the pool actually used to settle the swap, eliminating the spread-dependent bias.
