### Title
`get_split_parent_shard_uids` Uses Current Layout Version Instead of Previous Layout Version for Parent Shard UIDs — (`core/primitives/src/shard_layout/mod.rs`)

### Summary

`ShardLayout::get_split_parent_shard_uids` constructs `ShardUId` values for parent shards using `self.version()` — the **current** (new) shard layout's version — but parent shards belong to the **previous** shard layout and must carry the previous layout's version. This produces `ShardUId` values that are structurally incompatible with the database keys under which parent-shard state is actually stored, breaking the shard-layout compatibility invariant at every resharding boundary.

### Finding Description

`ShardUId` is the canonical cross-epoch shard identifier used as a database key prefix. Its `version` field must equal the shard layout version in which the shard existed. A parent shard that lived in layout version N has its state stored under `ShardUId { version: N, shard_id: X }`. After resharding, the new layout has version N+1.

`get_split_parent_shard_uids` is called on the **new** layout to return the UIDs of the shards that were split:

```rust
// core/primitives/src/shard_layout/mod.rs
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))  // ← uses current version N+1
        .collect()
}
``` [1](#0-0) 

`self.version()` here is the **new** layout's version (N+1). Parent shards do not exist in the new layout; they existed in the previous layout (version N). The returned `ShardUId { version: N+1, shard_id: X }` does not match any real shard in any layout and does not match the database key `ShardUId { version: N, shard_id: X }` under which the parent shard's state is stored.

The sibling function `get_split_child_shard_uids` correctly uses `self` (the current layout) for child shards via `get_children_shards_uids` → `ShardUId::from_shard_id_and_layout(id, self)`, which asserts the shard exists in the layout before constructing the UID:

```rust
pub fn get_split_child_shard_uids(&self) -> BTreeSet<ShardUId> {
    self.get_split_parent_shard_ids()
        .iter()
        .flat_map(|parent_shard_id| {
            self.get_children_shards_uids(*parent_shard_id).unwrap_or_default()
        })
        .collect()
}
``` [2](#0-1) 

`from_shard_id_and_layout` even asserts the shard exists in the given layout:

```rust
pub fn from_shard_id_and_layout(shard_id: ShardId, shard_layout: &ShardLayout) -> Self {
    assert!(shard_layout.shard_ids().any(|i| i == shard_id));
    Self::new(shard_layout.version(), shard_id)
}
``` [3](#0-2) 

`get_split_parent_shard_uids` bypasses this assertion by calling `ShardUId::new` directly, silently producing UIDs with the wrong version. The exact divergent value is: `ShardUId { version: N+1, shard_id: parent_id }` instead of the correct `ShardUId { version: N, shard_id: parent_id }`.

The `ShardUId` byte encoding is `[version_le_u32 || shard_id_le_u32]`:

```rust
pub fn to_bytes(&self) -> [u8; 8] {
    let mut res = [0; 8];
    res[0..4].copy_from_slice(&u32::to_le_bytes(self.version));
    res[4..].copy_from_slice(&u32::to_le_bytes(self.shard_id));
    res
}
``` [4](#0-3) 

A one-version mismatch in the first four bytes produces a completely different database key prefix, so any state lookup keyed by the returned UIDs silently misses the parent shard's data.

### Impact Explanation

Any resharding code path that calls `get_split_parent_shard_uids` to locate parent-shard state in the database (e.g., for state migration, trie copying, or split-shard validation) will receive UIDs with the wrong version. Database lookups using these UIDs will find no data (or, in the worst case, data belonging to a different shard that happens to share the same `shard_id` integer under a different layout version). This can cause resharding to silently skip parent-shard state or produce an inconsistent post-split state, breaking the shard-layout compatibility invariant that `ShardUId` is designed to enforce. The scope is **shard-layout compatibility** — a listed High-impact category.

### Likelihood Explanation

The bug fires at every shard layout change (resharding event). The function is public production code in `core/primitives/src/shard_layout/mod.rs`. No existing gate or canonicalization corrects the version after the fact. The mismatch is structural: the function is called on the new layout but must produce UIDs valid in the old layout, and it has no access to the old layout's version without it being passed in.

### Recommendation

`get_split_parent_shard_uids` must use the **previous** layout's version when constructing parent shard UIDs. The simplest fix is to accept the previous `ShardLayout` as a parameter and delegate to `ShardUId::from_shard_id_and_layout(shard_id, prev_layout)`, which both uses the correct version and asserts the shard actually exists in that layout. Alternatively, derive the previous version from the split history already stored in `ShardLayoutV3::shards_ancestor_map`.

### Proof of Concept

Suppose the network reshards from layout version 1 (shards `{3, 5}`) to layout version 2 (shards `{4, 6, 5}`, where shard 3 split into 4 and 6):

- Parent shard 3 is stored in the database under key prefix `ShardUId { version: 1, shard_id: 3 }` → bytes `[01 00 00 00 | 03 00 00 00]`.
- `new_layout.get_split_parent_shard_uids()` returns `ShardUId { version: 2, shard_id: 3 }` → bytes `[02 00 00 00 | 03 00 00 00]`.
- Any state lookup using the returned UID queries a key prefix that does not exist in the database, silently returning no data for the parent shard.
- The correct call would be `ShardUId::new(prev_layout.version(), 3)` → `[01 00 00 00 | 03 00 00 00]`, matching the stored key. [1](#0-0) [5](#0-4)
