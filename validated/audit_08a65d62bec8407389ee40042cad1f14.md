### Title
Bonus Reward Push Embedded in Critical Position-Management Path Blocks Fund Access When Pool Is Depleted — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` pushes bonus rewards directly to the position owner via `bankKeeper.SendCoinsFromModuleToAccount` as part of `claimRewards`. Because `claimRewards` is called as a mandatory prerequisite inside `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`, any failure of that push — specifically when the bonus rewards pool (`RewardsPoolName`) is depleted — causes all of those operations to revert. Users with accrued bonus rewards cannot undelegate, redelegate, add to, clear, or exit their positions, making their staked funds inaccessible.

---

### Finding Description

In `x/tieredrewards/keeper/claim_rewards.go`, `processEventsAndClaimBonus` computes accrued bonus rewards and, when non-zero, first checks pool sufficiency and then immediately sends the coins to the owner:

```go
// claim_rewards.go lines 230–241
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
...
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
    return nil, err
}
``` [1](#0-0) 

This push is invoked unconditionally by `claimRewards`:

```go
// claim_rewards.go lines 92–102
base, err := k.claimBaseRewards(ctx, pos)
...
bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
if err != nil {
    return types.PositionState{}, nil, nil, err
}
``` [2](#0-1) 

`claimRewards` is then called as a hard prerequisite — before any state mutation — in every critical position-management message handler:

| Handler | Line |
|---|---|
| `TierUndelegate` | 166 |
| `TierRedelegate` | 229 |
| `AddToTierPosition` | 314 |
| `ClearPosition` | 406 |
| `ExitTierWithDelegation` | 532 | [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

If `sufficientBonusPoolBalance` returns an error (pool balance < accrued bonus), the error propagates through `claimRewards` and causes the entire message handler to revert. The position's staked delegation is never touched; it remains locked in the position's sub-delegator account with no exit path.

---

### Impact Explanation

Any position that has accrued a non-zero bonus reward while the `RewardsPoolName` module account is depleted becomes permanently unmanageable. The owner cannot:

- Undelegate (`TierUndelegate`) — staked tokens remain locked.
- Redelegate (`TierRedelegate`) — cannot move to a different validator.
- Add to the position (`AddToTierPosition`) — cannot compound.
- Clear the exit (`ClearPosition`) — exit window cannot be reset.
- Exit with delegation (`ExitTierWithDelegation`) — cannot recover the delegation.

The staked funds held by the position's sub-delegator account are inaccessible to the owner for as long as the pool remains depleted. The corrupted invariant is: **position.Delegation.Shares > 0 but no exit path exists**, breaking the guarantee that a position owner can always recover their principal.

---

### Likelihood Explanation

The `RewardsPoolName` pool is a finite module account funded by inflation/governance. It can be depleted by:

1. **Natural exhaustion** — if the inflation module stops minting or the pool is not replenished, the pool drains as users claim rewards.
2. **Governance parameter change** — reducing `TargetBaseRewardsRate` or stopping inflation does not retroactively zero out accrued bonus checkpoints; positions that accrued rewards before the change still compute non-zero `totalBonus` on the next claim attempt.
3. **Race condition** — a large number of simultaneous claims can drain the pool between the `sufficientBonusPoolBalance` check and the actual send (though within a single block this is deterministic, the pool can be empty at the start of a block).

The trigger is reachable by any unprivileged position owner submitting a standard `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`, or `MsgExitTierWithDelegation` transaction.

---

### Recommendation

Decouple reward distribution from position management using a pull pattern:

1. **Rename `processEventsAndClaimBonus` to `accumulateBonusRewards`** — instead of sending coins, accumulate the computed bonus in a per-owner mapping (e.g., `PendingBonusRewards: Map[ownerAddr]sdk.Coins`).
2. **Remove the `sufficientBonusPoolBalance` check and `SendCoinsFromModuleToAccount` call** from the critical path. The accounting (checkpoint advancement, event seq decrement) can still happen; only the coin transfer is deferred.
3. **Add a `ClaimBonusRewards` message** that reads from the mapping, checks pool balance, sends coins, and clears the entry — callable independently by the owner at any time.
4. **Ensure `TierUndelegate`, `TierRedelegate`, etc.** only call the accumulation step, never the send step.

This mirrors the recommendation in the external report exactly: rename, accumulate in a mapping, and let the beneficiary pull funds via a dedicated claim entry point.

---

### Proof of Concept

1. Alice creates a tier position via `MsgLockTier`, delegating to validator V.
2. Over several blocks, Alice's position accrues non-zero bonus rewards (validator V remains bonded, `totalBonus > 0`).
3. The `RewardsPoolName` module account balance drops to zero (e.g., inflation stops, pool is drained by other claimants).
4. Alice submits `MsgTierUndelegate` for her position.
5. `TierUndelegate` calls `ms.claimRewards(ctx, pos)` at line 166.
6. `claimRewards` calls `processEventsAndClaimBonus`, which computes `totalBonus > 0` and reaches `sufficientBonusPoolBalance` at line 230.
7. `sufficientBonusPoolBalance` returns an error because the pool is empty.
8. The error propagates: `processEventsAndClaimBonus` → `claimRewards` → `TierUndelegate` → transaction reverts.
9. Alice's delegation remains locked in the position sub-delegator account. She cannot undelegate, redelegate, or exit. Her principal is inaccessible. [8](#0-7) [9](#0-8)
