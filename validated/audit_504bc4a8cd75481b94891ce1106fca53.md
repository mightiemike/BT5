### Title
Missing Epoch Domain Separator in Propeller Signature Payload Enables Cross-Epoch Signature Replay — (File: `crates/apollo_propeller/src/signature.rs`)

---

### Summary

`build_signed_payload` in `crates/apollo_propeller/src/signature.rs` constructs a signed payload that contains no epoch number and no network/chain identifier. Because `CommitteeId` is derived solely from committee members' peer IDs and weights (no epoch, no chain), the same `CommitteeId` recurs whenever the same validator set is elected in a later epoch. Combined with an in-memory-only, per-peer nonce cache that resets on restart, a captured signature from epoch N is cryptographically valid and nonce-acceptable in epoch N+1 after any node restart, allowing a replayer to inject a stale message into the consensus layer.

---

### Finding Description

**Signed payload construction** — `build_signed_payload` at line 88–93 of `crates/apollo_propeller/src/signature.rs`:

```rust
fn build_signed_payload(
    message_id: &MessageRoot,
    committee_id: CommitteeId,
    nonce: u64,
) -> Vec<u8> {
    [SIGNING_PREFIX, &message_id.0, &committee_id.0, &nonce.to_be_bytes(), SIGNING_POSTFIX].concat()
}
```

The payload is `<propeller> || message_root (32 B) || committee_id (32 B) || nonce (8 B) || </propeller>`. No epoch number, no chain/network identifier, and no protocol version are included. [1](#0-0) 

**`CommitteeId` computation** — `compute_committee_id` in `crates/apollo_network/src/active_committees/store.rs` hashes only `peer_id` and `weight` for each member:

```rust

        hasher.update(member.weight.0.to_be_bytes());

    CommitteeId(hasher.finalize().into())
}

**Nonce tracking** — `engine.rs` tracks nonces in an in-memory LRU cache keyed only by `PeerId`, not by `(PeerId, CommitteeId, EpochId)`:

### Impact Explanation

When the same validator set is elected in epoch N+1 as in epoch N, `CommitteeId` is byte-for-byte identical. Any node that restarts between epochs (normal operational event) loses its nonce cache. A replayer that captured a legitimate signature `(message_root_N, committee_id, nonce_X, sig_N)` from epoch N can re-broadcast it in epoch N+1. The receiving engine will:

1. Compute `CommitteeId` for epoch N+1 — identical to epoch N.
2. Look up stored nonce for the publisher — 0 (cache cleared).
3. Check `0 >= nonce_X` — false, so the unit is accepted.
4. Verify the signature against the publisher's public key — valid (it was legitimately created).
5. Reconstruct the epoch-N message and emit `Event::MessageReceived` to the consensus layer.

The consensus layer receives a stale message (epoch-N block proposal or other consensus payload) as if it were a fresh epoch-N+1 message. Depending on whether the consensus layer independently validates the block number embedded in the payload, this can cause the wrong block data to be processed, binding the wrong executable payload to the current round.

---

### Likelihood Explanation

- Validator sets are stable across consecutive epochs in practice (especially in a small or slowly-rotating set), making `CommitteeId` collisions across epochs routine, not exceptional.
- Node restarts between epochs are a normal operational event (upgrades, crashes, maintenance).
- Any committee member — or any node that observed the P2P traffic — can capture and store a valid unit signature for later replay.
- The two TODO comments (`types.rs` line 38 and `engine.rs` line 123) confirm the developers are aware of both the epoch-collision problem and the per-committee nonce gap, but neither has been fixed.

---

### Recommendation

1. **Include epoch in `CommitteeId`**: Mix the `EpochId` into `compute_committee_id` so that the same member set in different epochs produces distinct committee identifiers.
2. **Include epoch and chain ID in the signed payload**: Extend `build_signed_payload` to include the epoch number and a network/chain identifier, analogous to EIP-712's domain separator, so a signature is bound to exactly one epoch on one network.
3. **Track nonces per `(PeerId, CommitteeId)`** (as the existing TODO in `engine.rs` line 123 suggests): This prevents cross-committee replay even if `CommitteeId` collisions remain.
4. **Persist nonces across restarts**: Use durable storage for the nonce high-water mark so that a restart does not open a replay window.

### Proof of Concept

```
Epoch N:
  members = {A, B, C} (same weights)
  CommitteeId_N = SHA256(A.peer_id||A.w || B.peer_id||B.w || C.peer_id||C.w)
  Publisher A signs: payload = <propeller>||root_N||CommitteeId_N||nonce_X||</propeller>
  Attacker captures (root_N, CommitteeId_N, nonce_X, sig_A).

Node restart (peer_nonce cache cleared for all peers).

Epoch N+1:
  members = {A, B, C} (same set, same weights)
  CommitteeId_{N+1} = SHA256(...) == CommitteeId_N  ← identical

Attacker replays (root_N, CommitteeId_N, nonce_X, sig_A) to a target node.

Target node:
  stored_nonce(A) = 0  (cache was cleared)
  0 >= nonce_X?  → false  → unit accepted
  verify(sig_A, payload, pubkey_A) → valid
  → message root_N reconstructed and delivered to consensus layer as epoch N+1 data
``` [7](#0-6) [2](#0-1) [3](#0-2) [5](#0-4) [6](#0-5)

pub fn verify_message_id_signature(
    message_id: &MessageRoot,
    committee_id: CommitteeId,
    nonce: u64,
    signature: &[u8],
    public_key: &PublicKey,
) -> Result<(), SignatureVerificationError> {
    if signature.is_empty() {
        return Err(SignatureVerificationError::VerificationFailed);
    }
    let msg = build_signed_payload(message_id, committee_id, nonce);
    let signature_valid = public_key.verify(&msg, signature);
    if signature_valid { Ok(()) } else { Err(SignatureVerificationError::VerificationFailed) }
}
