### Title
`PriceVelocityGuardExtension.beforeSwap` Drops `onlyPool` Modifier on Override, Allowing Any Caller to Reset Velocity State and Bypass the Guard - (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` overrides the base implementation but silently drops the `onlyPool` modifier. Any external caller — not just a registered pool — can invoke it directly to overwrite `lastMidPriceX64` and `lastUpdateBlock` with arbitrary values. This lets an attacker zero out the apparent price delta immediately before a real pool swap, bypassing the velocity cap entirely and executing swaps that the guard was designed to block.

---

### Finding Description

`BaseMetricExtension` declares `beforeSwap` as `external virtual onlyPool`: [1](#0-0) 

The `onlyPool` modifier verifies `IMetricOmmPoolFactory(FACTORY).isPool(msg.sender)`: [2](#0-1) 

`PriceVelocityGuardExtension.beforeSwap` overrides this function but omits `onlyPool`: [3](#0-2) 

In Solidity, modifiers are **not** inherited through overrides; the override must re-declare them explicitly. Because `onlyPool` is absent, the function is callable by any EOA or contract. The two state writes that happen unconditionally before the velocity check are:

```solidity
s.lastMidPriceX64 = midPrice;   // line 57
s.lastUpdateBlock = uint64(block.number); // line 58
```

These writes occur before the guard logic at lines 60–75, so an attacker can pre-seed the state to make the subsequent pool-triggered call see a zero delta. [4](#0-3) 

---

### Impact Explanation

The velocity guard is the sole on-chain protection against large oracle price moves being exploited against LPs. Bypassing it allows a swap to execute at a price that moved far beyond the configured `maxChangePerBlockE18` cap. LPs suffer direct principal loss because they are forced to trade at a price the guard was specifically configured to reject. The loss magnitude scales with the size of the oracle move and the pool's liquidity depth.

---

### Likelihood Explanation

The attack requires no special role, no privileged setup, and no non-standard token behavior. Any unprivileged address can call `PriceVelocityGuardExtension.beforeSwap` directly in the same transaction as a pool swap. The attacker only needs to observe a large oracle price move (or cause one) and front-run the next legitimate swap. This is a straightforward, low-cost, single-transaction exploit.

---

### Recommendation

Re-apply the `onlyPool` modifier on the override:

```solidity
function beforeSwap(
    address,
    address,
    bool,
    int128,
    uint128,
    uint256,
    uint128 bidPriceX64,
    uint128 askPriceX64,
    bytes calldata
) external override onlyPool returns (bytes4) {
```

This mirrors the pattern used in `DepositAllowlistExtension.beforeAddLiquidity` and `SwapAllowlistExtension.beforeSwap`, which both carry `override` and the appropriate access modifier.

---

### Proof of Concept

1. Pool is deployed with `PriceVelocityGuardExtension` as a `beforeSwap` hook. `maxChangePerBlockE18` is set to 5% per block. At block N−1 the last recorded mid price is `P_old`.

2. At block N the oracle price jumps to `P_new` (20% move, exceeding the cap). A legitimate swap through the pool would revert with `PriceVelocityExceeded`.

3. Attacker constructs `bidPriceX64` and `askPriceX64` whose geometric mid equals `P_new` and calls `PriceVelocityGuardExtension.beforeSwap(pool, ...)` **directly** (not through the pool). Because `onlyPool` is absent, the call succeeds and writes:
   - `s.lastMidPriceX64 = P_new`
   - `s.lastUpdateBlock = N`

4. In the same transaction the attacker calls `pool.swap(...)`. The pool invokes `extension.beforeSwap(...)` with the current oracle prices (still `P_new`). The extension computes `delta = P_new − P_new = 0`, so `actualSq = 0 ≤ allowedSq`. The guard passes.

5. The swap executes at the 20%-moved price. LPs absorb the loss that the velocity guard was configured to prevent. [5](#0-4)

```text
  function beforeSwap(
    address,
    address,
    bool,
    int128,
    uint128,
    uint256,
    uint128 bidPriceX64,
    uint128 askPriceX64,
    bytes calldata
  ) external override returns (bytes4) {
    address pool_ = msg.sender;
    (uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
    // casting to `uint128` is safe: geometric mid of two uint128 bid/ask quotes fits uint128 (same bound as pool)
    // forge-lint: disable-next-line(unsafe-typecast)
    uint128 midPrice = uint128(midPriceX64);

    PriceVelocityState storage s = priceVelocityState[pool_];
    uint128 prevMid = s.lastMidPriceX64;
    uint64 prevBlock = s.lastUpdateBlock;

    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);

    if (prevMid != 0) {
      uint64 maxChange = s.maxChangePerBlockE18;
      if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;

        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);

        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);

        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);

        if (actualSq > allowedSq) {
          revert PriceVelocityExceeded(actualSq, allowedSq);
        }
      }
    }

    return IMetricOmmExtensions.beforeSwap.selector;
  }
```
