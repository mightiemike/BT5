### Title
Odd-Leaf Duplicate Position Ambiguity Allows Same Proof to Verify at Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` never validates that the caller-supplied `tx_index` is within the actual transaction count of the block. In any block whose transaction count is odd, Bitcoin's Merkle tree duplicates the last leaf. This makes the last real leaf (index `N-1`) and the phantom duplicate slot (index `N`) produce identical intermediate hashes at every level, so the **same `(tx_id, merkle_proof)` pair verifies as `true` for both indices**. `verify_transaction_inclusion_v2` inherits this flaw without adding any position-range guard.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`** [1](#0-0) 

The function uses `current_position` only to decide left/right placement (`% 2`). It never checks that the supplied position is less than the actual number of transactions in the block. No transaction-count is stored in the on-chain header; the `LightHeader` only carries `merkle_root`. [2](#0-1) 

**How the duplicate arises**

`merkle_proof_calculator` pads an odd-length list by appending the last element to itself before hashing each level: [3](#0-2) 

For a 3-tx block `[T0, T1, T2]` the padded level-0 is `[T0, T1, T2, T2]`. The canonical proof for `T2` at index 2 is `[T2, H(T0,T1)]` (sibling is the duplicate copy of `T2`).

**Why both index 2 and index 3 verify identically**

```
Verify T2 at index 2 (even):
  step 1: H(T2, proof[0]=T2) → H(T2,T2);  position → 1
  step 2: H(proof[1]=H(T0,T1), H(T2,T2)) → Root  ✓

Verify T2 at index 3 (odd):
  step 1: H(proof[0]=T2, T2) → H(T2,T2);  position → 1
  step 2: H(proof[1]=H(T0,T1), H(T2,T2)) → Root  ✓
```

Both calls return `true`. Index 3 is a phantom slot that does not correspond to any real transaction.

**`verify_transaction_inclusion_v2` does not close the gap**

The coinbase guard only pins the coinbase to position 0; it places no constraint on the target `tx_index`: [4](#0-3) 

The inner call then passes the attacker-chosen `tx_index` directly to `compute_root_from_merkle_proof` without a range check: [5](#0-4) 

`tx_index` is a caller-controlled `u64` field with no upper-bound validation anywhere in the call chain: [6](#0-5) 

---

### Impact Explanation

The invariant **"a Merkle proof must bind to exactly one transaction position"** is broken. Any downstream bridge, unlock, mint, or withdrawal contract that uses `(tx_id, tx_index, block_hash)` as a replay-protection key will accept the same economic event twice: once at the real index `N-1` and once at the phantom index `N`. The light client itself returns `true` for a position that does not exist in the block, constituting a **light client verification bypass**.

---

### Likelihood Explanation

- The attack requires only a block with an odd transaction count, which is the majority of Bitcoin blocks.
- The attacker needs no special role, no relayer key, and no privileged access — `verify_transaction_inclusion_v2` is a public, unpermissioned view call.
- The proof to supply is the standard canonical proof for the last transaction; no forgery is required.
- The only precondition is that the target block is on the current main chain (enforced by `mainchain_header_to_height`), which is trivially satisfied for any confirmed block. [7](#0-6) 

---

### Recommendation

Add a position-range guard inside `compute_root_from_merkle_proof` or at the call sites. The total number of leaf slots at depth `d` is `2^d`; the real transaction count must be passed alongside the proof so the verifier can reject any `tx_index >= tx_count`. Concretely:

1. Extend `ProofArgs` / `ProofArgsV2` with a `tx_count: u64` field.
2. Before calling `compute_root_from_merkle_proof`, assert `args.tx_index < args.tx_count`.
3. Assert `args.tx_count` is consistent with the proof depth: `(1 << proof.len()) >= tx_count > (1 << (proof.len()-1))`.
4. Reject any `tx_index` that falls in the phantom duplicate range, i.e., `tx_index >= tx_count`.

---

### Proof of Concept

Construct a 3-transaction block (coinbase + 2 txs). Submit its header. Then call `verify_transaction_inclusion_v2` twice with identical `(tx_id, merkle_proof)` but `tx_index=2` and `tx_index=3` respectively. Both calls return `true`, demonstrating that the phantom position 3 is accepted as a valid inclusion proof for the same transaction.

```rust
// Block txs: [coinbase, tx1, tx2]  (odd count → tx2 is duplicated)
// Canonical proof for tx2 at index 2: [tx2_hash, H(coinbase, tx1)]

let proof = vec![tx2_hash.clone(), h_coinbase_tx1.clone()];
let coinbase_proof = vec![tx1_hash.clone(), /* ... */];

// Call 1: legitimate position
let r1 = verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: tx2_hash, tx_index: 2, merkle_proof: proof.clone(),
    coinbase_tx_id, coinbase_merkle_proof: coinbase_proof.clone(), ..
});
assert!(r1); // true — expected

// Call 2: phantom duplicate position
let r2 = verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: tx2_hash, tx_index: 3, merkle_proof: proof,   // same proof
    coinbase_tx_id, coinbase_merkle_proof: coinbase_proof, ..
});
assert!(r2); // true — SHOULD be false; index 3 does not exist
```

Both assertions pass against the unmodified contract, confirming the broken position-binding invariant.

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
