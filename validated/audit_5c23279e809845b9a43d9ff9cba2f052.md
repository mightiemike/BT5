Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Validates Position `owner` Instead of Token Payer `sender`, Allowing Any Address to Bypass the Deposit Allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`, who pays tokens) and validates only `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any address not on the allowlist can bypass the guard by supplying an allowed `owner` address, completely nullifying the pool-admin-configured LP access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes two distinct actors to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`msg.sender` is the operator who triggers the call and pays tokens; `owner` is the address whose position is credited. Unlike `removeLiquidity`, which enforces `if (msg.sender != owner) revert NotPositionOwner()`, `addLiquidity` imposes no such constraint. [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both actors to every configured extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [3](#0-2) 

`DepositAllowlistExtension.beforeAddLiquidity` then receives `sender` as its first argument but discards it (unnamed `address,`), checking only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [4](#0-3) 

The contract's own NatSpec states it "Gates `addLiquidity` by depositor address", and its admin API (`setAllowedToDeposit`, `allowedDepositor`) is named around the depositor/payer concept — confirming the intent is to restrict the payer, not the beneficiary. [5](#0-4) 

The wrong value checked is `allowedDepositor[msg.sender][owner]` — it should be `allowedDepositor[msg.sender][sender]`.

## Impact Explanation
Any address not on the allowlist can call `pool.addLiquidity(allowedOwner, salt, deltas, ...)`, pay tokens through the modify-liquidity callback, and have the position credited to `allowedOwner`. The extension guard passes because it only validates `allowedOwner`, which is on the allowlist. Every pool that relies solely on `DepositAllowlistExtension` for LP access control is open to any depositor who can identify one allowed owner address (trivially discoverable from on-chain `LiquidityAdded` events). This is an admin-boundary break: an unprivileged actor bypasses a pool-admin-configured access control gate, depositing tokens into a pool that was intended to be restricted.

## Likelihood Explanation
Likelihood is high. The operator pattern (`msg.sender ≠ owner`) is a first-class feature of the pool with no guard in `addLiquidity`. Allowed owner addresses are publicly visible from on-chain events. No special privilege, flash loan, or oracle manipulation is required — a standard `addLiquidity` call with a known allowed `owner` suffices. The attack is repeatable by any EOA or contract.

## Recommendation
Replace the discarded first parameter with a named `sender` and validate it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the payer and the position owner, both should be checked independently.

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** added to the allowlist.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(Bob, alice, ...)` → extension receives `sender=Bob, owner=alice`.
5. Extension discards `Bob`, checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's tokens are pulled via the modify-liquidity callback; Alice's position is credited.
7. Bob has successfully deposited despite being absent from the allowlist, bypassing the admin-configured access control gate.
