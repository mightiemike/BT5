Looking at the `SwapAllowlistExtension` hook, the pool's `swap()` function, and the `MetricOmmSimpleRouter`, I can trace a clear wrong-actor binding vulnerability.

**Key trace:**

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly, making the **router** the `msg.sender` to the pool: [3](#0-2) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is that caller. If the pool admin allowlists the router so that allowlisted users can use it, every unprivileged user can bypass the swap allowlist by routing through the same router.

### Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` is the pool and `sender` is the direct caller. When a user swaps through `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

The router is `msg.sender` to the pool, so `sender` in `beforeSwap` is the router's address. The extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

The pool admin faces an impossible choice:
- **Don't allowlist the router**: Allowlisted EOAs cannot use the router at all — core swap functionality is broken for them.
- **Allowlist the router**: Every user, including non-allowlisted ones, can bypass the allowlist by routing through the router.

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput` in the router. [4](#0-3) [5](#0-4) 

### Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict swaps to specific counterparties is fully bypassed. Any unprivileged user calls `router.exactInputSingle()` on the curated pool; the router's address passes the allowlist check; the swap executes. LP funds in the curated pool are exposed to unrestricted trading that the pool admin explicitly intended to prevent. This is a direct loss-of-policy / broken-core-functionality finding above Sherlock thresholds.

### Likelihood Explanation
High. `MetricOmmSimpleRouter` is the standard, documented periphery swap interface. Any user who wants to swap on a curated pool will naturally use it. No special privilege, flash loan, or multi-step setup is required — a single `exactInputSingle` call suffices.

### Recommendation
The extension must gate the economically relevant actor, not the intermediary. Options:

1. **Router-injected identity**: Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and verify it (requires trusting the router, which can be enforced by checking `msg.sender` is a factory-registered pool whose callback context names the real payer).
2. **Transient payer forwarding**: The router already stores the real payer in transient storage for callback settlement; expose a read function so extensions can query it.
3. **Direct-pool-only policy**: Document that `SwapAllowlistExtension` only works correctly when users call `pool.swap()` directly, and add a guard that reverts if `msg.sender` (the pool's caller) is not an EOA or a known non-router contract.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension.
2. Admin: setAllowedToSwap(pool, alice, true)          // alice is the intended grantee
3. Admin: setAllowedToSwap(pool, router, true)          // required so alice can use the router
4. bob (non-allowlisted) calls:
       router.exactInputSingle({pool: pool, tokenIn: ..., amountIn: X, ...})
5. Router calls pool.swap(recipient, ...) — msg.sender to pool = router
6. Pool calls _beforeSwap(sender=router, ...)
7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
8. Bob's swap executes on the curated pool, bypassing the allowlist entirely.
```

The corrupted value is the identity checked by the guard: `router` is substituted for `bob`, so the allowlist invariant `allowedSwapper[pool][bob]` is never evaluated. [2](#0-1) [1](#0-0) [3](#0-2)

      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(

          zeroForOne,

          "",
          params.extensionDatas[i]
