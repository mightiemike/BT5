### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, `sender` resolves to the **router's address**, not the actual user's address. If the pool admin allowlists the router (the natural step to enable router-mediated swaps for curated pools), every user — including non-allowlisted ones — can bypass the per-user curation policy.

---

### Finding Description

**Root cause — wrong actor bound in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

`msg.sender` here is the pool (the extension is called by the pool). `sender` is whatever `msg.sender` the pool received when `swap` was called.

**How the pool populates `sender`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← sender = immediate caller of pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

**How the router calls the pool:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [3](#0-2) 

When `exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) is called, the router is `msg.sender` of `pool.swap`. Therefore `sender` in `beforeSwap` = **router address**, not the original user.

**Contrast with `DepositAllowlistExtension`:**

The deposit allowlist correctly gates on `owner` (the position owner), not `sender` (the payer/caller):

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [4](#0-3) 

The swap allowlist has no equivalent forwarding of the original user identity — it only sees the router.

---

### Impact Explanation

**Bypass path:** A pool admin deploying a curated pool with `SwapAllowlistExtension` must allowlist the router address to permit allowlisted users to trade via the standard periphery. Once `allowedSwapper[pool][router] = true`, the extension's check becomes:

```
allowedSwapper[pool][router] == true  →  passes for every user routing through the router
```

Any non-allowlisted user can now swap on the curated pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point). The per-user curation policy is entirely nullified.

**Denial path (secondary):** If the pool admin does *not* allowlist the router, allowlisted users are silently blocked when they use the router (the router address fails the check), breaking the expected user flow without any error that identifies the root cause.

Both outcomes are fund-impacting: the bypass allows unauthorized traders to interact with pools designed to restrict access (e.g., institutional-only, KYC-gated, or risk-managed pools), and LP funds in those pools are exposed to trades the pool admin explicitly intended to prevent.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is the primary supported swap interface for end users.
- A pool admin who configures `SwapAllowlistExtension` and wants allowlisted users to use the router **must** allowlist the router address — there is no other mechanism to forward the original user identity.
- This is a natural, expected operational step, making the bypass reachable without any adversarial setup beyond using the public router.

---

### Recommendation

Pass the original user's address through the hook rather than the immediate pool caller. Two options:

1. **Preferred — add an `originator` field to `extensionData` or a dedicated hook parameter:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated extension/router convention.

2. **Simpler — mirror the deposit allowlist pattern:** Change `beforeSwap` to check `recipient` (the economic beneficiary) instead of `sender`, or add a dedicated `originator` argument to `IMetricOmmExtensions.beforeSwap` that the pool populates from a transient-storage originator set by the router before calling `pool.swap`.

The deposit allowlist's design (checking `owner`, not `sender`) is the correct model; the swap allowlist should adopt the same principle.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as `beforeSwap` extension.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — allowlisting the router so that allowlisted users can trade via the standard periphery.
3. Pool admin does **not** call `setAllowedToSwap(pool, alice, true)` — Alice is not an allowlisted user.
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. `_beforeSwap(router, ...)` is dispatched; `SwapAllowlistExtension.beforeSwap` receives `sender = router`.
7. Check: `allowedSwapper[pool][router] == true` → **passes**.
8. Alice's swap executes on the curated pool despite never being allowlisted.

Expected (correct) behavior: Alice's swap should revert with `NotAllowedToSwap`.
Actual behavior: Alice's swap succeeds because the router address is allowlisted, not Alice's address.
