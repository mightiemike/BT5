Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap` invocation. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap`, so `sender` arriving at the extension is the router address, not the originating user. If the router is allowlisted (required for any router-mediated swap to work), every user — including non-allowlisted ones — can bypass the gate by routing through the public router.

## Finding Description
`ExtensionCalling._beforeSwap` receives `sender` and forwards it verbatim into the extension call: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed in: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the router contract `msg.sender` from the pool's perspective: [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

Therefore `allowedSwapper[pool][router]` is evaluated. If the router is allowlisted (a necessary operational condition for any router-mediated swap to succeed), the check passes for every user regardless of their individual allowlist status. The existing `allowAllSwappers` escape hatch does not mitigate this — it is a separate flag. There are no other guards in the extension.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the LP-share recipient), not `sender` (the caller), making it robust to intermediary contracts: [5](#0-4) 

`SwapAllowlistExtension` has no equivalent indirection.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers) is fully bypassed. Any unprivileged user routes through `MetricOmmSimpleRouter`, the extension sees the allowlisted router address, and the swap executes. This is an **admin-boundary break**: the pool admin's access control is circumvented by an unprivileged path (the public router). Consequences include unauthorized price impact on restricted pools, unauthorized extraction of LP value, and violation of regulatory or operational constraints the allowlist was designed to enforce — constituting direct loss of protocol integrity and potentially LP principal.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, publicly documented entry point for swaps. No special privilege is required. The pool admin has no on-chain mechanism to prevent router-mediated calls while keeping the router usable for allowlisted users — allowlisting the router is a prerequisite for normal operation. Any user aware of the router can exploit this unconditionally and repeatably.

## Recommendation
Replace the `sender` check with the originating user identity. Two options:

1. **Use `recipient` instead of `sender`**: For exact-input swaps the recipient is the user. Mirror `DepositAllowlistExtension`'s approach — gate the economically relevant actor (the token recipient), not the intermediate dispatcher. The extension interface already delivers both `sender` and `recipient` to `beforeSwap`.

2. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires router cooperation but is trustless if the extension validates the encoding.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` — required for any router-mediated swap to function.
3. Pool admin does **not** allowlist `attacker`.
4. `attacker` calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(recipient=attacker, ...)` — router is `msg.sender` in the pool.
6. Pool calls `_beforeSwap(sender=router, ...)` → extension receives `sender=router`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. Swap executes for `attacker` despite `allowedSwapper[pool][attacker] == false`.

The allowlist is fully bypassed. Reproducible as a Foundry integration test deploying the pool with the extension, setting up the allowlist as described, and asserting the swap succeeds for a non-allowlisted caller via the router.
