Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Transaction `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks the caller-supplied `owner` argument against the allowlist. Because `owner` is a free parameter with no enforced relationship to `msg.sender` in `addLiquidity`, any un-allowlisted address can bypass the guard by naming any allowlisted address as `owner`. The pool admin's deposit restriction is rendered completely ineffective.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards `sender` (first parameter is unnamed) and checks only `owner`: [3](#0-2) 

The allowlist mapping is keyed by `depositor` (which resolves to `owner`, not `sender`): [4](#0-3) 

There is no `msg.sender == owner` enforcement in `addLiquidity`. That check exists only in `removeLiquidity`: [5](#0-4) 

The exploit path is: un-allowlisted Bob calls `addLiquidity(owner=Alice, ...)` where Alice is allowlisted. The extension checks `allowedDepositor[pool][Alice]` → `true` → no revert. Bob's tokens are pulled via the swap callback; LP shares are credited to Alice. Bob has deposited without being on the allowlist.

## Impact Explanation
The deposit allowlist — the core access-control mechanism pool admins use to restrict liquidity provision to curated addresses (e.g., KYC'd LPs) — is fully bypassed by any caller who knows one allowlisted address. The pool admin's configuration has no effect on who can actually deposit. This constitutes broken core pool access-control functionality. No direct theft of existing LP principal occurs, but the intended restriction is silently inert, meeting the Medium threshold for broken core pool functionality.

## Likelihood Explanation
Exploitation requires only: (1) knowledge of one allowlisted address, readable from the public `allowedDepositor` mapping or on-chain events, and (2) the ability to call `addLiquidity` directly or through any router. No special role, privileged access, or admin cooperation is needed. Any EOA or contract can execute this in a single transaction, unconditionally and repeatably.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual transaction originator) instead of `owner`:

```solidity
// Fixed: check the actual caller (sender = msg.sender of addLiquidity)
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the caller and the position beneficiary, both `sender` and `owner` should be checked.

## Proof of Concept

```
Setup:
  pool  = MetricOmmPool with DepositAllowlistExtension configured
  Alice = allowedDepositor[pool][Alice] = true   (allowlisted)
  Bob   = allowedDepositor[pool][Bob]   = false  (not allowlisted)

Attack (single tx, no special role):
  Bob calls pool.addLiquidity(
      owner        = Alice,   // allowlisted — passes the guard
      salt         = 0,
      deltas       = <desired liquidity>,
      callbackData = <Bob pays tokens in metricOmmSwapCallback>,
      extensionData = ""
  )

Execution path:
  1. addLiquidity calls _beforeAddLiquidity(msg.sender=Bob, owner=Alice, ...)
  2. ExtensionCalling forwards (sender=Bob, owner=Alice) to DepositAllowlistExtension
  3. beforeAddLiquidity ignores sender=Bob; checks allowedDepositor[pool][Alice] → true → no revert
  4. LiquidityLib.addLiquidity credits LP shares to Alice's position
  5. Bob's tokens are transferred to the pool via callback

Result:
  - Bob bypasses the deposit allowlist entirely
  - Pool admin's restriction on Bob is silently ineffective
```
