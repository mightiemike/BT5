### Title
`verify_transaction_inclusion` Allows Zero-Confirmation Proof Bypass — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`, which delegates to it) accepts a caller-supplied `confirmations: u64` parameter but never enforces that it is non-zero. When `confirmations == 0`, the confirmation-depth guard is trivially satisfied for every block in the pool, allowing any unprivileged NEAR caller to obtain a `true` SPV proof result with zero confirmation depth.

---

### Finding Description

The confirmation-depth guard in `verify_transaction_inclusion` is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

`args.confirmations` is a `u64` field in `ProofArgs` with no lower-bound constraint:

```rust
pub struct ProofArgs {
    ...
    pub confirmations: u64,
}
``` [2](#0-1) 

When a caller supplies `confirmations = 0`, the right-hand side of the `>=` comparison is `0`. Because the left-hand side is a `u64` expression (`saturating_sub(...) + 1`), it is always `>= 0`. The guard **unconditionally passes** regardless of how recently the target block was submitted or how deep it sits in the chain.

The preceding guard:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [3](#0-2) 

also trivially passes when `confirmations == 0` (since `0 <= gc_threshold` for any valid `gc_threshold`).

`verify_transaction_inclusion_v2` inherits the same flaw because it converts its args and calls `verify_transaction_inclusion` directly:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

The existing test suite even demonstrates the path: `confirmations: 0` is passed in a test call, and the test expects `false` only because the Merkle proof itself does not match — not because the confirmation check rejects it. [5](#0-4) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` — or that allows its users to supply the `confirmations` value — will receive `true` for a transaction whose block has zero confirmation depth. This defeats the entire purpose of the confirmation parameter, which exists to protect against chain reorganizations. An attacker can:

1. Submit a Bitcoin block header to the light client (via the relayer path or directly if `UnrestrictedSubmitBlocks` is granted).
2. Immediately call `verify_transaction_inclusion` with `confirmations = 0` targeting that block.
3. The function returns `true` even though the block may be reorganized away moments later.

Downstream bridge or settlement contracts that rely on this result to release funds or finalize state transitions are exposed to double-spend via reorg.

---

### Likelihood Explanation

The entry point is a public, unprivileged `view` call — no role, deposit, or special permission is required to call `verify_transaction_inclusion`. The `confirmations` field is a plain `u64` in a Borsh-serialized struct; any caller can set it to `0`. The analog to the original report is direct: just as `undelegationBlock == 0` made the time-lock trivially pass, `confirmations == 0` makes the depth guard trivially pass.

---

### Recommendation

Add a non-zero guard at the top of `verify_transaction_inclusion`, mirroring the fix applied in keep-network/keep-core#1521:

```rust
require!(
    args.confirmations > 0,
    "Confirmations must be greater than zero"
);
```

This should be placed before the `gc_threshold` check in `contract/src/lib.rs`. [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract with any valid genesis (e.g., `skip_pow_verification: true`, `gc_threshold: 100`).
2. The genesis block is at height 0 and is in `headers_pool`.
3. Call (as any unprivileged account):

```json
verify_transaction_inclusion({
  "tx_id": "<any hash present in genesis merkle root>",
  "tx_block_blockhash": "<genesis block hash>",
  "tx_index": 0,
  "merkle_proof": [...],
  "confirmations": 0
})
```

4. The `confirmations <= gc_threshold` check passes (`0 <= 100`).
5. The depth check evaluates `(0 - 0) + 1 >= 0` → `1 >= 0` → `true`.
6. The function proceeds to Merkle verification and returns `true` if the proof is valid — with zero confirmation depth enforced.

The same applies to `verify_transaction_inclusion_v2` since it converts `ProofArgsV2` (which also carries `confirmations: u64`) into `ProofArgs` and delegates. [7](#0-6)
