### Title
Missing `chunk.shard_id()` vs. `shard_id` identity check in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts an attacker-supplied `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard B, stores it under the DB key for shard A, and never detects the mismatch. Every validation gate in the function either ignores `shard_id` entirely or uses the *parameter* value rather than the chunk's own `shard_id()`, so a crafted header passes all checks and poisons `DBCol::StateHeaders` with a cross-shard entry.

---

### Finding Description

`set_state_header` performs five validation steps before writing to `DBCol::StateHeaders`:

**Step 1-2 — `validate_chunk_proofs`** [1](#0-0) 

This function checks the chunk's internal hash consistency, tx root, and receipts root. It has no parameter for an expected `shard_id` and never reads `chunk.shard_id()` against any external value. A self-consistent chunk for shard B passes unconditionally.

**Step 3a — `verify_path` against `chunk_headers_root`** [2](#0-1) 

This call uses the plain `verify_path`, not `verify_path_with_index`: [3](#0-2) 

`verify_path` only reconstructs the root from the path's Left/Right directions and compares it to the stored root. It does **not** verify which leaf index (i.e., which shard position) the proof corresponds to. A valid Merkle proof for shard B's chunk at position B in the block's chunk-headers tree will satisfy this check even when the caller passes `shard_id = A`. The position-aware variant `verify_path_with_index` (line 121) is available but not used here.

**Step 4 — receipt proof validation** [4](#0-3) 

The receipt hash is computed with the *parameter* `shard_id`, not `chunk.shard_id()`. An attacker who has read access to the canonical chain (any full node) can supply the real incoming receipts for shard A together with their valid Merkle proofs against `prev_chunk_outgoing_receipts_root`, satisfying steps 4b–4g while the embedded chunk still belongs to shard B.

**Step 5 — state root node validation** [5](#0-4) 

Validates the `state_root_node` against `chunk_inner.prev_state_root()`. Both values come from the attacker-supplied header; no cross-check against the canonical shard A state root is performed.

**Storage** [6](#0-5) 

The key is `StateHeaderKey(shard_id, sync_hash)` — the *parameter* value — regardless of what `chunk.shard_id()` contains. The result is a `DBCol::StateHeaders` entry keyed by `(shard_id_A, sync_hash)` that holds shard B's chunk.

There is no line anywhere in `set_state_header` that reads `chunk.shard_id()` and compares it to the `shard_id` parameter.

---

### Impact Explanation

A syncing node that accepts this poisoned header will subsequently:

1. Call `set_state_part` — which reads the stored header, extracts shard B's `prev_state_root`, and validates incoming state parts against it. A malicious peer can supply valid state parts for shard B, which pass validation.
2. Call `set_state_finalize` — which applies the chunk using the *parameter* `shard_id` to derive `shard_uid` but uses shard B's chunk data (transactions, receipts, gas limit, state root). The result is shard A's trie being overwritten with shard B's state. [7](#0-6) 

The syncing node ends up with a corrupted state for shard A, causing it to diverge from consensus on every subsequent block that touches shard A.

---

### Likelihood Explanation

State sync is performed with untrusted peers. Any peer that a syncing node contacts for a `StateRequestHeader` can respond with a crafted `ShardStateSyncResponseHeader`. No validator or privileged role is required — the attacker only needs to be a reachable peer. All inputs needed to craft the attack (canonical block headers, chunk Merkle paths, outgoing receipt proofs) are publicly available on-chain.

---

### Recommendation

Add an explicit identity check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the plain `verify_path` call at step 3a with `verify_path_with_index`, passing the shard index derived from `shard_id` and the total number of shards, so that the Merkle proof is bound to the correct leaf position.

---

### Proof of Concept

```
1. Syncing node requests StateRequestHeader for shard_id=0, sync_hash=H.
2. Attacker (malicious peer) constructs a ShardStateSyncResponseHeader where:
   - chunk = a valid ShardChunk for shard 1 (shard_id()=1), self-consistent
     (validate_chunk_proofs passes).
   - chunk_proof = the real Merkle path for shard 1's chunk in block H-1's
     chunk_headers_root (verify_path passes because it is position-agnostic).
   - prev_chunk_header / prev_chunk_proof = real values for shard 1's prev chunk.
   - incoming_receipts_proofs / root_proofs = real incoming receipts for shard 0
     (the parameter), with valid Merkle proofs against prev_chunk_outgoing_receipts_root
     (step 4 passes because receipts_hash uses the parameter shard_id=0).
   - state_root_node = a valid node for shard 1's prev_state_root.
3. set_state_header returns Ok(()).
4. DBCol::StateHeaders now contains an entry keyed (shard_id=0, H) holding shard 1's chunk.
5. Attacker supplies valid state parts for shard 1; set_state_part accepts them.
6. set_state_finalize applies shard 1's chunk under shard_uid for shard 0,
   corrupting shard 0's state trie.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
