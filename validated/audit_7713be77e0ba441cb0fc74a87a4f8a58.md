### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash on P2P-Synced Nodes — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to decide which enum variant to reconstruct. When a V3 transaction is submitted via RPC with `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }`, the gateway computes its hash using the `AllResources` variant (which includes `L1_DATA_GAS` in the Poseidon hash chain). After the transaction is serialized to protobuf and deserialized by a P2P-syncing node, the same wire bytes are reconstructed as the `L1Gas` variant (which omits `L1_DATA_GAS` from the hash chain). The two variants produce structurally different hash preimages, so any node that recomputes the transaction hash from the deserialized `InvokeTransactionV3` / `DeployAccountTransactionV3` / `DeclareTransactionV3` gets a different value than the canonical hash stored in the block.

### Finding Description

**Step 1 — Gateway always hashes with `AllResources`.**

`InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` and its `InvokeTransactionV3Trait::resource_bounds()` implementation unconditionally wraps it in `ValidResourceBounds::AllResources`: [1](#0-0) 

The hash function `get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash` with this value. When the variant is `AllResources`, the function appends a third element — the `L1_DATA_GAS` concat-felt — to the Poseidon hash chain: [2](#0-1) 

So for `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the hash chain is:

```
poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0), L1_DATA_GAS_concat(0))   // H1
```

**Step 2 — Protobuf serialization is lossless for the wire bytes.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources` with all three fields present (even when zero): [3](#0-2) 

**Step 3 — Protobuf deserialization silently downgrades the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` inspects the *values* of the deserialized fields and picks the variant based on whether `l2_gas` and `l1_data_gas` are zero: [4](#0-3) 

When `l2_gas = 0` and `l1_data_gas = 0`, the result is `ValidResourceBounds::L1Gas(l1_gas)` — even though the sender originally used `AllResources`. This converter is called directly when deserializing `InvokeV3`, `DeployAccountV3`, and `DeclareV3` from P2P sync messages: [5](#0-4) [6](#0-5) 

**Step 4 — The `L1Gas` variant produces a different hash.**

After deserialization, `InvokeTransactionV3::resource_bounds` is `L1Gas`. When `get_tip_resource_bounds_hash` is called on this variant, the `L1_DATA_GAS` element is omitted: [7](#0-6) 

The hash chain becomes:

```
poseidon(tip, L1_GAS_concat(X), L2_GAS_concat(0))   // H2
```

`H1 ≠ H2` because Poseidon over 3 elements ≠ Poseidon over 2 elements, even when the third element is the zero-bounds felt.

**Step 5 — The `InvokeTransactionV3` struct uses `ValidResourceBounds`, not `AllResourceBounds`.** [8](#0-7) 

Its `TransactionHasher` implementation delegates to the same `get_invoke_transaction_v3_hash`, so the wrong variant directly produces the wrong hash: [9](#0-8) 

### Impact Explanation

Any component on a P2P-syncing node that calls `calculate_transaction_hash` or `validate_transaction_hash` on the deserialized `InvokeTransactionV3` / `DeployAccountTransactionV3` / `DeclareTransactionV3` will compute H2 instead of H1. Concretely:

- **`validate_transaction_hash`** returns `false` for a valid, committed transaction, causing the syncing node to treat it as corrupt.
- **RPC fee estimation / simulation / tracing** (`starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_traceTransaction`) re-execute the stored transaction. The wrong `ValidResourceBounds` variant changes `get_gas_vector_computation_mode()` from `All` to `NoL2Gas`, altering the fee-check path in `check_resources_within_bounds` and potentially producing a different fee estimate or a spurious revert — an authoritative-looking wrong value served to clients.
- **Transaction commitment hash** in block headers is computed from transaction hashes. A node that recomputes commitment hashes from stored transactions would derive a wrong commitment, causing header verification to fail.

This matches the allowed impacts: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** and **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload**.

### Likelihood Explanation

The trigger requires a V3 transaction with `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }`. The `RpcInvokeTransactionV3` struct accepts `AllResourceBounds` directly with no minimum-value validation, so any user can submit such a transaction. The gateway accepts it (the `InternalRpcInvokeTransactionV3` path never rejects zero l2/l1_data gas). Once included in a block, every syncing node that processes it via P2P will silently downgrade the variant. The condition is easy to satisfy deliberately or accidentally (e.g., a wallet that sets only l1_gas for a legacy-style V3 transaction).

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminant field in the protobuf schema, or always deserialize to `AllResources` when all three fields are present in the wire message (regardless of their values). The `l1_data_gas` field being present and explicitly zero is semantically different from it being absent (pre-0.13.3 wire format). The TODO comment at line 426 acknowledges this is a temporary compatibility shim; until it is removed, the deserializer must not use zero-value checks to infer the variant, because doing so conflates two distinct hash preimage structures.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds, GasAmount, GasPrice, Tip,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;

fn main() {
    let l1_gas = ResourceBounds {
        max_amount: GasAmount(1000),
        max_price_per_unit: GasPrice(1),
    };
    let tip = Tip(0);

    // Gateway path: AllResources with zero l2/l1_data gas
    let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas,
        l2_gas: ResourceBounds::default(),      // zero
        l1_data_gas: ResourceBounds::default(), // zero
    });
    let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();

    // P2P-synced path: same wire values, but deserialized as L1Gas
    let l1_gas_variant = ValidResourceBounds::L1Gas(l1_gas);
    let h2 = get_tip_resource_bounds_hash(&l1_gas_variant, &tip).unwrap();

    // H1 != H2 — same transaction data, different hash preimage
    assert_ne!(h1, h2, "Hash divergence confirmed: AllResources != L1Gas for zero l2/l1_data");
    println!("AllResources hash: {:?}", h1);
    println!("L1Gas hash:        {:?}", h2);
}
```

The `assert_ne!` will pass, confirming that the protobuf round-trip silently changes the hash preimage for any V3 transaction submitted with `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }`.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
