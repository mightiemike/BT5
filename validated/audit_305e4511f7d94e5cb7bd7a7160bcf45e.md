### Title
Stale `lastLiquidationFees` Corrupts Finalization Logic for Subsequent Subaccounts — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a global storage variable set during every non-finalization liquidation call. It is never reset to zero after a finalization completes. Because `_finalizeSubaccount` uses this stale value to compute `v.canLiquidateMore`, a large `lastLiquidationFees` left over from a prior subaccount's liquidation can cause the strict spot-balance check to be incorrectly skipped during a later subaccount's finalization, allowing finalization with unresolved spot liabilities.

---

### Finding Description

`_handleLiquidationPayment` unconditionally writes to `lastLiquidationFees` at the end of every non-finalization liquidation: [1](#0-0) 

`_finalizeSubaccount` then reads this global value to compute the effective insurance available and the `canLiquidateMore` flag: [2](#0-1) 

If `v.canLiquidateMore` is `true`, the function enforces a strict invariant that all spot balances must be exactly zero before finalization proceeds: [3](#0-2) 

At the end of `_finalizeSubaccount`, `lastLiquidationFees` is added back to `v.insurance` and written to the global `insurance`, but `lastLiquidationFees` itself is never cleared: [4](#0-3) 

Because `lastLiquidationFees` is a single global slot (not per-subaccount), any non-finalization liquidation of **any** subaccount overwrites it. When a different subaccount is subsequently finalized, `_finalizeSubaccount` subtracts this stale value from the real insurance balance before evaluating `v.canLiquidateMore`. If the stale `lastLiquidationFees` is large enough that `quoteBalance.amount + (insurance − lastLiquidationFees) ≤ 0`, `v.canLiquidateMore` evaluates to `false` even though the true condition `quoteBalance.amount + insurance > 0` holds. The strict `require(balance.amount == 0)` guard is then bypassed, and the subaccount is finalized while still carrying negative spot balances (liabilities).

The earlier, weaker guard only requires `balance.amount <= 0`: [5](#0-4) 

So spot liabilities pass the first check but should be blocked by the second. With `v.canLiquidateMore` incorrectly false, the second check is never reached.

---

### Impact Explanation

A subaccount finalized with unresolved negative spot balances leaves those liabilities permanently in the protocol's accounting. The insurance coverage step only covers the quote balance: [6](#0-5) 

Spot liabilities are only socialized if `v.insurance <= 0`: [7](#0-6) 

When `v.canLiquidateMore` is incorrectly false, `v.insurance` is artificially low, so `v.insurance` may still be positive after the insurance-cover step, meaning `spotEngine.socializeSubaccount` is never called. The spot liabilities remain as phantom debt in the system, corrupting `totalBorrowsNormalized` in `SpotEngine` and causing all depositors to silently absorb losses they should not bear.

---

### Likelihood Explanation

Every non-finalization liquidation (`productId != type(uint32).max`) sets `lastLiquidationFees` to a non-zero value. In any active market, liquidations are frequent. An attacker can deliberately liquidate a large position to inflate `lastLiquidationFees`, then immediately trigger finalization (`productId = type(uint32).max`) of a different underwater subaccount. The entry path — `liquidateSubaccountImpl` via `EndpointTx.processTransactionImpl` — is reachable by any unprivileged liquidator: [8](#0-7) [9](#0-8) 

---

### Recommendation

Reset `lastLiquidationFees` to zero at the start of `_finalizeSubaccount` (or immediately after finalization completes), so that the insurance deduction only reflects fees from the current subaccount's own liquidation sequence:

```solidity
// At the start of _finalizeSubaccount, before using lastLiquidationFees:
lastLiquidationFees = 0;
```

Alternatively, track `lastLiquidationFees` per-subaccount (`mapping(bytes32 => int128)`) so that stale values from unrelated subaccounts cannot contaminate the finalization of a different subaccount.

---

### Proof of Concept

1. Subaccount **A** is underwater. Liquidator calls `liquidateSubaccountImpl` with a large `productId` (non-finalization). `_handleLiquidationPayment` executes and sets `lastLiquidationFees = L` (a large value).

2. Subaccount **B** is also underwater and ready for finalization (all perp positions closed, spot balances ≤ 0 but some are negative). Liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max`.

3. Inside `_finalizeSubaccount`:
   - `v.insurance = insurance` (e.g., `I`)
   - `v.insurance -= lastLiquidationFees` → `v.insurance = I − L`
   - If `L > quoteBalance.amount + I`, then `v.canLiquidateMore = false`

4. The `if (v.canLiquidateMore)` block is skipped. The `require(balance.amount == 0)` check for spot balances is never executed.

5. Subaccount **B** is finalized with remaining negative spot balances. `spotEngine.socializeSubaccount` may not be called (if `v.insurance > 0`). The spot liabilities remain in the system, corrupting `SpotEngine`'s accounting and silently distributing losses to all depositors. [10](#0-9)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
