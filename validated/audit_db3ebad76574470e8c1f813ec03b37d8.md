### Title
`registerMerkleRoot` Accepts Merkle Root Without Validating Sufficient Token Balance, Causing Claim Failures for Valid Recipients — (File: `core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.sol`'s `registerMerkleRoot()` stores a Merkle root encoding per-user token allocations without verifying that the contract's token balance is sufficient to cover the total claimable amount encoded in the tree. If the contract is underfunded, valid claimants will have their `claim()` transactions revert after earlier claimants exhaust the available balance.

---

### Finding Description

`registerMerkleRoot()` accepts a `merkleRoot` and stores it with no balance check: [1](#0-0) 

There is no on-chain invariant enforcing that `IERC20(token).balanceOf(address(this)) >= sum(all leaf amounts in the Merkle tree)` at the time of registration. The Merkle root is opaque to the contract — it encodes arbitrary per-user `totalAmount` values that are only revealed at claim time.

When a user calls `claim()`, the internal `_claim()` function calls `SafeERC20.safeTransfer` directly against the contract's live token balance: [2](#0-1) 

If the contract's balance has been exhausted by earlier claimants, this transfer reverts. The `claimed[week][sender]` state write in `_verifyProof` is part of the same transaction and also reverts, so the user's claim slot is not consumed — but they still cannot receive their tokens. [3](#0-2) 

The public `claim()` entrypoint is callable by any unprivileged user: [4](#0-3) 

---

### Impact Explanation

Valid recipients holding correct Merkle proofs for a registered week cannot claim their allocated tokens if the contract is underfunded. The first claimants drain the available balance; all subsequent claimants' transactions revert. Users lose expected token rewards that are visible to them off-chain via the Merkle tree data. This is a direct asset-denial impact on unprivileged users interacting through the supported `claim()` entrypoint.

---

### Likelihood Explanation

Medium. The condition is triggered by any operational error where the owner registers a Merkle root before depositing sufficient tokens, deposits the wrong amount, or sends tokens to the wrong address. There is no on-chain guard preventing this. The protocol's own `getClaimed()` view function further reinforces user expectations by surfacing per-week allocation data, making the mismatch more impactful from a user-trust perspective. [5](#0-4) 

---

### Recommendation

Add a `totalWeekAmount` parameter to `registerMerkleRoot()` and require that the contract's current token balance is at least equal to that amount at registration time:

```solidity
function registerMerkleRoot(uint32 week, bytes32 merkleRoot, uint256 totalWeekAmount)
    external
    onlyOwner
{
    require(
        IERC20(token).balanceOf(address(this)) >= totalWeekAmount,
        "Insufficient balance for week distribution"
    );
    pastWeeks += 1;
    require(week == pastWeeks, "Invalid week provided.");
    merkleRoots[week] = merkleRoot;
}
```

Alternatively, at minimum, document clearly in user-facing materials that the owner must ensure sufficient token balance is deposited before calling `registerMerkleRoot()`, and that claims may fail if this invariant is violated.

---

### Proof of Concept

1. Owner calls `registerMerkleRoot(1, merkleRoot)` where `merkleRoot` encodes 100 users each entitled to 100 tokens (total: 10,000 tokens).
2. The contract holds only 5,000 tokens at the time of registration — no check prevents this.
3. Users 1–50 call `claim()` with valid proofs and successfully receive 100 tokens each, exhausting the balance.
4. User 51 calls `claim()` with a valid proof; `SafeERC20.safeTransfer` reverts due to zero remaining balance.
5. Users 51–100 cannot claim their tokens despite holding valid Merkle proofs for a registered week.

The root cause is the absence of a balance sufficiency check in `registerMerkleRoot()` at `core/contracts/Airdrop.sol` line 33–40, directly analogous to the missing validation in `createDistribution()` identified in the external report.

```text
    function _verifyProof(
        uint32 week,
        address sender,
        uint256 totalAmount,
        bytes32[] calldata proof
    ) internal {
        require(claimed[week][sender] == 0, "Already claimed.");
        require(
            merkleRoots[week] != bytes32(0),
            "Week hasn't been registered."
        );
        require(
            !ISanctionsList(sanctions).isSanctioned(sender),
            "address is sanctioned."
        );
        bytes32 leaf = keccak256(
            bytes.concat(keccak256(abi.encode(sender, totalAmount)))
        );
        bool isValidLeaf = MerkleProof.verify(proof, merkleRoots[week], leaf);
        require(isValidLeaf, "Invalid proof.");
        claimed[week][sender] = totalAmount;
    }
```
