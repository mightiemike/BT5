### Title
Missing Contract Address in Fast-Withdrawal Signature Digest Enables Cross-Pool Replay — (File: `core/contracts/Verifier.sol`)

---

### Summary

`Verifier.requireValidTxSignatures` constructs its message hash as `keccak256(chainid, idx, txn)` without committing to the calling `WithdrawPool` contract address. This is the direct analog to the PLONK bug: a critical context parameter is absent from the hash transcript. Any valid fast-withdrawal signature produced for one `WithdrawPool` instance is equally valid on any other `WithdrawPool` instance that shares the same `Verifier` contract, enabling an unprivileged attacker to replay the signature and drain tokens from the second pool.

---

### Finding Description

In `Verifier.requireValidTxSignatures`, the signed message is constructed as:

```solidity
bytes32 data = keccak256(
    abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
);
bytes32 hashedMsg = keccak256(
    abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
);
``` [1](#0-0) 

The hash commits to `chainid`, the withdrawal index `idx`, and the raw transaction bytes `txn`. It does **not** commit to the address of the `WithdrawPool` contract that is requesting the verification.

This function is called from `BaseWithdrawPool.submitFastWithdrawal`:

```solidity
v.requireValidTxSignatures(transaction, idx, signatures);
``` [2](#0-1) 

`submitFastWithdrawal` is a `public` function callable by any address. Its replay-prevention relies on two guards:

1. `require(!markedIdxs[idx])` — per-contract state, not shared across pool instances.
2. `require(idx > minIdx)` — also per-contract state. [3](#0-2) 

Because neither guard nor the hash itself encodes *which* `WithdrawPool` contract is the target, a signature `(e, s, signerBitmask)` that the sequencer's quorum produced for `(chainid, idx=N, txn=<withdraw X tokens to Alice>)` on Pool A is cryptographically indistinguishable from a signature for the same tuple on Pool B. The `Verifier` contract has no way to distinguish the two calls.

The `Verifier.requireValidTxSignatures` function is `public view`, so it can be called by any contract or EOA, and the `BaseWithdrawPool.submitFastWithdrawal` function is also `public`. [4](#0-3) 

---

### Impact Explanation

If two `WithdrawPool` instances share the same `Verifier` contract and both hold the same product token (e.g., USDC, `productId=0`), an attacker can:

1. Observe a legitimate fast-withdrawal submission on Pool A for `(idx=N, txn=<withdraw X USDC to Alice>)` along with its valid sequencer signatures.
2. Call `Pool_B.submitFastWithdrawal(N, txn, signatures)` before Pool B's `markedIdxs[N]` is set.
3. Pool B's signature check passes (same hash, same signatures), `markedIdxs[N]` is unset, and `N > minIdx` if Pool B is newer.
4. Pool B transfers X USDC to Alice, with no corresponding debit from any subaccount balance on Pool B.

The corrupted state delta is: Pool B's token balance is reduced by `X` without any authorized on-chain debit. The recipient receives a double payment. This is a direct, concrete asset loss from Pool B.

---

### Likelihood Explanation

Medium. The attack requires two `WithdrawPool` instances using the same `Verifier` and the same product token. This is a realistic scenario during:

- Protocol upgrades where a v2 `WithdrawPool` is deployed alongside a v1 instance before migration is complete.
- Any multi-pool deployment where the same `Verifier` key set is reused across pool contracts.

The attacker requires no special privileges — only the ability to observe on-chain transactions and call `submitFastWithdrawal` on the second pool. The signature is already public on-chain.

---

### Recommendation

Include the calling contract's address in the hash. Since `requireValidTxSignatures` is in `Verifier.sol` and does not inherently know the caller, the caller address should be passed as an explicit parameter:

```solidity
function requireValidTxSignatures(
    bytes calldata txn,
    uint64 idx,
    bytes[] calldata signatures,
    address callerContract   // <-- add this
) public view {
    bytes32 data = keccak256(
        abi.encodePacked(
            uint256(block.chainid),
            callerContract,        // <-- bind to specific pool
            uint256(idx),
            txn
        )
    );
    ...
}
```

`BaseWithdrawPool.submitFastWithdrawal` would then pass `address(this)`. This mirrors the standard EIP-712 domain separator pattern, which always includes `verifyingContract`.

```

  - Verifier V (shared)
  - WithdrawPool_A (productId=0, USDC, uses V)
  - WithdrawPool_B (productId=0, USDC, uses V) — newly deployed, minIdx=0

Step 1: Alice submits a legitimate fast withdrawal on Pool A:
  Pool_A.submitFastWithdrawal(
      idx = 50,
      txn = encode(WithdrawCollateral{sender=Alice, productId=0, amount=1000e6}),
      signatures = [sig_0, sig_1, ...]   // sequencer quorum signs keccak256(chainid, 50, txn)
  )
  → Pool A marks markedIdxs[50] = true, transfers 1000 USDC to Alice. ✓

Step 2: Attacker replays the same call on Pool B:
  Pool_B.submitFastWithdrawal(
      idx = 50,
      txn = <same bytes>,
      signatures = <same signatures>
  )
  → Pool B: markedIdxs[50] == false ✓, 50 > minIdx(0) ✓
  → v.requireValidTxSignatures(txn, 50, signatures) passes (identical hash)
  → Pool B transfers 1000 USDC to Alice. ✓ (unauthorized)

Result: Pool B drained of 1000 USDC. Alice receives 2000 USDC total.
        No subaccount balance was debited on Pool B.

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
