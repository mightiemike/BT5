### Title
Missing `l1_data_gas` in Protobuf Deserialization Silently Defaults to Zero, Producing Wrong Transaction Hash for V3 Transactions — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses `unwrap_or_default()` for the `optional` `l1_data_gas` protobuf field. For any V3 transaction with non-zero `l2_gas` (post-0.13.3 `AllResources` transactions), a peer that omits `l1_data_gas` from the protobuf message causes the receiving node to reconstruct the transaction with `l1_data_gas = 0`. The transaction hash is then recomputed from this wrong resource-bounds value, diverging from the hash the user originally signed. The blockifier passes this wrong hash to the account contract's `__validate__` entrypoint, causing signature verification to fail and the transaction to revert — a valid transaction producing a wrong revert result.

---

### Finding Description

**Root cause — `crates/apollo_protobuf/src/converters/transaction.rs`, line 427:**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← silent zero default
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
``` [1](#0-0) 

The protobuf schema declares `l1_data_gas` as `optional`:

```proto
message ResourceBounds {
    ResourceLimits l1_gas = 1;
    // This can be None only in transactions that don't support l2 gas.
    // Starting from 0.14.0, MempoolTransaction and ConsensusTransaction shouldn't have None here.
    optional ResourceLimits l1_data_gas = 2;
    ResourceLimits l2_gas = 3;
}
``` [2](#0-1) 

When `l1_data_gas` is absent, `unwrap_or_default()` produces `ResourceBounds { max_amount: 0, max_price_per_unit: 0 }`. Because `l2_gas` is non-zero for any modern V3 transaction, the branch at line 431 falls into `AllResources` — carrying `l1_data_gas = 0` instead of the actual signed value.

**Hash divergence — `crates/starknet_api/src/transaction_hash.rs`, lines 188–211:**

`get_tip_resource_bounds_hash` unconditionally includes `l1_data_gas` in the Poseidon hash for `AllResources` transactions:

```rust

    ValidResourceBounds::AllResources(all_resources) => {

Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [3](#0-2) 

This hash feeds directly into `get_invoke_transaction_v3_hash` (and the declare/deploy-account equivalents), which is the canonical Starknet transaction hash committed to the chain: [4](#0-3) 

**Hash is recomputed from deserialized data — `crates/apollo_transaction_converter/src/transaction_converter.rs`, line 391:**

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [5](#0-4) 

There is no transmitted hash field in `InvokeV3` protobuf to cross-check against; the hash is always derived from the deserialized fields. A zero `l1_data_gas` therefore silently produces a different `tx_hash` than the one the user signed.

---

### Impact Explanation

A P2P peer (any node participating in mempool propagation or consensus) can relay a V3 transaction protobuf with `l1_data_gas` omitted. The receiving sequencer node:

1. Deserializes `l1_data_gas = 0` via `unwrap_or_default()`.
2. Constructs `AllResources { l1_gas, l2_gas, l1_data_gas: 0 }` (because `l2_gas ≠ 0`).
3. Recomputes `tx_hash` with `l1_data_gas = 0` — a value that diverges from the hash the user signed.
4. Stores and sequences the transaction under this wrong hash.
5. The blockifier passes the wrong hash to the account contract's `__validate__` entrypoint via `TxInfo`.
6. The account contract verifies the ECDSA/Stark signature against the wrong hash → verification fails → transaction reverts.

The result is a valid, correctly-signed transaction producing a wrong revert result from blockifier execution logic — matching the **Critical** impact scope: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

It also matches the **High** impact scope: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The `l1_data_gas` field is `optional` in the protobuf schema and the TODO comment explicitly defers enforcement: *"Assert data gas is not none once we remove support for 0.13.2."* Any peer running a pre-0.13.3 client version, or any peer that deliberately omits the field, can trigger this path. The serialization side (`From<ValidResourceBounds> for protobuf::ResourceBounds`) always emits `l1_data_gas`, so a conforming modern node would not trigger this — but the deserialization side has no guard, making the invariant one-sided and fragile. [6](#0-5) 

---

### Recommendation

Replace the silent `unwrap_or_default()` with an explicit error for transactions that carry non-zero `l2_gas` (i.e., post-0.13.3 `AllResources` transactions):

```rust
// After resolving l2_gas:
let l1_data_gas = match value.l1_data_gas {
    Some(v) => v,
    None if l2_gas_raw.is_zero() => protobuf::ResourceLimits::default(), // pre-0.13.3 compat
    None => return Err(missing("ResourceBounds::l1_data_gas")),           // modern tx: reject
};
```

This matches the intent already expressed in the TODO comment and mirrors the strict handling applied to `l1_gas` and `l2_gas`.

---

### Proof of Concept

```
1. User submits InvokeV3 with:
     l1_gas  = { max_amount: 1000, max_price_per_unit: 1 }
     l2_gas  = { max_amount: 500,  max_price_per_unit: 1 }   // non-zero → AllResources
     l1_data_gas = { max_amount: 200, max_price_per_unit: 1 } // signed into tx_hash_A

2. Peer serializes the transaction to protobuf::InvokeV3 but omits l1_data_gas
   (field is optional; proto3 default = absent).

3. Receiving sequencer node calls:
     TryFrom<protobuf::ResourceBounds> for ValidResourceBounds
   → l1_data_gas = unwrap_or_default() = { max_amount: 0, max_price_per_unit: 0 }
   → l2_gas ≠ 0 → AllResources { l1_gas, l2_gas, l1_data_gas: 0 }

4. calculate_transaction_hash() computes tx_hash_B ≠ tx_hash_A
   (l1_data_gas contribution in get_tip_resource_bounds_hash differs).

5. InternalRpcTransaction { tx, tx_hash: tx_hash_B } is stored and sequenced.

6. Blockifier executes __validate__(tx_hash_B).
   Account contract: verify_ecdsa(signature, tx_hash_B) → FAIL (signed over tx_hash_A).
   Transaction reverts with "Invalid signature".
```

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
