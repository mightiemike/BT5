### Title
Deprecated `verify_transaction_inclusion` Remains a Live Public API Vulnerable to 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a callable public NEAR entry point despite being explicitly deprecated. Its underlying `compute_hash` function concatenates two 32-byte values without any length tag or domain separator — the exact same structural flaw as `abi.encodePacked(a, b)` with two dynamic types. An unprivileged NEAR caller can supply an internal Merkle-tree node as `tx_id` with a one-level-shorter proof and cause the function to return `true` for a transaction that was never included in any Bitcoin block.

---

### Finding Description

`compute_hash` in `merkle-tools/src/lib.rs` builds its hash input by raw concatenation:

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);   // 32 bytes
    concat_inputs.extend(second_tx_hash.0);  // 32 bytes
    double_sha256(&concat_inputs)            // 64-byte input
}
``` [1](#0-0) 

Because every internal node is itself `double_sha256(left_child || right_child)`, any internal node hash is a valid 32-byte value that can be passed as `transaction_hash` to `compute_root_from_merkle_proof`. With a proof that is one level shorter than the real tree, the computation still reaches the correct Merkle root. This is the Bitcoin-specific analog of `abi.encodePacked(a, b)` collision: the 64-byte input is ambiguous between a leaf-level pair and a higher-level pair.

`verify_transaction_inclusion` is the live entry point that exposes this:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    ...
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
``` [2](#0-1) 

`#[deprecated]` in Rust is a **compiler lint only** — it does not remove the function from the compiled WASM binary or prevent runtime calls. The function remains a fully reachable public NEAR method. Any NEAR account can call it directly, bypassing `verify_transaction_inclusion_v2` entirely.

The contract's own documentation acknowledges the flaw:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [3](#0-2) 

But the contract does not enforce the stated assumption ("validation … is performed at a higher level") — it accepts any caller-supplied `tx_id` without restriction.

---

### Impact Explanation

An attacker can forge an SPV inclusion proof for a Bitcoin transaction that does not exist. Given a real block with 4 transactions whose Merkle tree has depth 2, the attacker supplies:

- `tx_id` = the internal node `H(T0 || T1)` (a valid 32-byte hash, but an internal node)
- `tx_index` = 0
- `merkle_proof` = `[H(T2 || T3)]` (depth-1 proof, one element)

`compute_root_from_merkle_proof(H(T0||T1), 0, [H(T2||T3)])` computes `H(H(T0||T1) || H(T2||T3))` = real `merkle_root`. The function returns `true`.

Any downstream NEAR contract that gates payments, minting, or authorization on `verify_transaction_inclusion` returning `true` can be deceived into accepting a fabricated Bitcoin transaction as confirmed.

---

### Likelihood Explanation

The attack requires no privileged access, no leaked keys, and no social engineering. The 64-byte Merkle forgery technique is publicly documented (referenced in the contract's own deprecation notice: https://www.bitmex.com/blog/64-Byte-Transactions). The attacker only needs to know a real block's Merkle tree structure, which is public Bitcoin data. The entry point is open to any NEAR account.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public API entirely (or gate it with an access-control role so it cannot be called by unprivileged accounts). Rust's `#[deprecated]` attribute does not prevent runtime invocation. The safe path is to either:

1. Delete the function body and replace it with `env::panic_str("removed: use verify_transaction_inclusion_v2")`, or
2. Apply `#[private]` so only the contract itself can call it (which effectively removes external access).

The `verify_transaction_inclusion_v2` path correctly anchors tree depth via the coinbase proof length check and should be the only externally callable verification endpoint. [4](#0-3) 

---

### Proof of Concept

Given a real Bitcoin block with 4 transactions `[T0, T1, T2, T3]`:

```
Merkle tree:
  root = H(H(T0||T1) || H(T2||T3))
  internal_node_A = H(T0 || T1)
  internal_node_B = H(T2 || T3)
```

Attacker calls `verify_transaction_inclusion` with:
```json
{
  "tx_id": "<internal_node_A>",
  "tx_block_blockhash": "<real_block_hash>",
  "tx_index": 0,
  "merkle_proof": ["<internal_node_B>"],
  "confirmations": 1
}
```

`compute_root_from_merkle_proof(internal_node_A, 0, [internal_node_B])`:
- position 0 is even → `compute_hash(internal_node_A, internal_node_B)`
- = `double_sha256(H(T0||T1) || H(T2||T3))` = `root` ✓

The function returns `true` for a transaction (`internal_node_A`) that is not a real transaction in the block. The forged proof is indistinguishable from a legitimate one at the contract level. [5](#0-4) [6](#0-5)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
