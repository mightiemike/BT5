### Title
Missing Parent Subaccount Initial Health Check in `createIsolatedSubaccount` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.createIsolatedSubaccount` deducts `margin` from the parent subaccount's quote balance with no subsequent initial-health assertion on the parent. The directly analogous collateral-transfer path — `Clearinghouse.transferQuote` — enforces `require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH)` after the identical `spotEngine.updateBalance` call. The omission in `createIsolatedSubaccount` allows a parent subaccount to be left below initial health without reverting.

---

### Finding Description

**`Clearinghouse.transferQuote` (the reference path):**

After deducting from the sender's quote balance, the function enforces the invariant:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -toTransfer);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.recipient, toTransfer);
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);   // ← guard present
``` [1](#0-0) 

**`OffchainExchange.createIsolatedSubaccount` (the vulnerable path):**

The same economic operation — moving quote from parent to isolated subaccount — is performed with no health check:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);   // parent loses quote
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    // ← no _isAboveInitial(txn.order.sender) check
}
``` [2](#0-1) 

**`EndpointTx` handler adds nothing:**

After calling `createIsolatedSubaccount`, the handler only records the new subaccount — no health assertion on the parent is performed at any layer:

```solidity
bytes32 newIsolatedSubaccount = IOffchainExchange(offchainExchange)
    .createIsolatedSubaccount(txn, getLinkedSigner(txn.order.sender));
_recordSubaccount(newIsolatedSubaccount);   // ← no health check on txn.order.sender
``` [3](#0-2) 

---

### Impact Explanation

A parent subaccount whose initial health is at or near zero can have its quote balance reduced below the initial health threshold by encoding a non-zero `margin` in the order appendix. After the transaction is processed on-chain, the parent's health is negative with no revert. The parent is then:

- **Immediately liquidatable** — `isUnderInitial` returns `true`, satisfying the liquidation precondition in `ClearinghouseLiq`.
- **Blocked from further collateral-reducing operations** — any subsequent `transferQuote` or order placement that checks `_isAboveInitial` will revert.

The isolated subaccount holds the transferred margin and is unaffected; only the parent is harmed. [4](#0-3) 

---

### Likelihood Explanation

The attack path is fully on-chain and user-controlled:

1. The trader signs a `CreateIsolatedSubaccount` order with `margin` encoded in the appendix — this is a standard user-signed message, no privileged access required.
2. The sequencer processes the transaction as a normal protocol operation; no sequencer compromise is needed.
3. The margin value is attacker-controlled (any `margin > 0` that pushes parent health below zero suffices).
4. The precondition (parent health near zero) is reachable by any subaccount that has consumed most of its collateral through open positions or prior transfers. [5](#0-4) 

---

### Recommendation

Add an initial health assertion on the parent subaccount immediately after the margin deduction in `createIsolatedSubaccount`, mirroring the guard in `transferQuote`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [6](#0-5)
