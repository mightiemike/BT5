### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Validation — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification entry points. The newer `verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check to mitigate the 64-byte transaction Merkle-proof forgery vulnerability. The older `verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a live, unrestricted public method. Any unprivileged NEAR caller can invoke v1 directly, bypassing the protective coinbase-proof constraint entirely and obtaining a `true` result for a forged transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle-proof forgery vector:

```
/// # Deprecated
/// Use `verify_transaction_inclusion_v2` instead, which includes coinbase merkle proof validation
/// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
/// https://www.bitmex.com/blog/64-Byte-Transactions
``` [1](#0-0) 

The v2 function enforces two guards before delegating to v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()`
2. The coinbase transaction must be provably at position 0 in the block's Merkle tree. [2](#0-1) 

After both guards pass, v2 calls v1 internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

However, v1 itself is still declared `pub` with no access-control restriction beyond `#[pause]`: [4](#0-3) 

The `#[deprecated]` attribute in Rust emits a **compiler warning only**. It has no runtime effect. NEAR callers invoking the contract via JSON-RPC or borsh are not subject to any Rust compiler warning; they can call `verify_transaction_inclusion` directly, skipping the coinbase proof validation entirely.

The v1 function's own documentation acknowledges the risk:

```
/// # Warning
/// This function may return `true` if the provided `tx_id` is a hash of an internal node
/// in the Merkle tree rather than a valid transaction hash.
``` [5](#0-4) 

The Merkle root check in v1 is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [6](#0-5) 

`compute_root_from_merkle_proof` is purely positional — it does not distinguish leaf nodes (real transactions) from internal nodes: [7](#0-6) 

An attacker who supplies an internal Merkle tree node hash as `tx_id` with a correctly constructed sibling proof will cause `compute_root_from_merkle_proof` to reproduce the block's `merkle_root`, and v1 returns `true`.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a state-changing action (e.g., releasing funds upon proof of a Bitcoin deposit) can be deceived. An attacker can claim that a fabricated "transaction" — which is actually an internal Merkle tree node from a real, mainchain block — was included in that block. The contract returns `true`, and the downstream protocol acts on a false proof. The protective invariant (coinbase-proof validation) is present in v2 but completely absent from the still-reachable v1 path.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of any real block in the mainchain (publicly available).
2. Computation of the block's Merkle tree to extract an internal node and its sibling path (trivial with standard Bitcoin libraries).
3. A direct NEAR RPC call to `verify_transaction_inclusion` — no privileged role, no staking, no special setup.

The 64-byte Merkle forgery technique is well-documented and tooling exists. Likelihood is **medium-high** for any deployment where downstream contracts consume v1's result.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public interface entirely, or gate it behind an access-control role so it cannot be called by unprivileged accounts. Only `verify_transaction_inclusion_v2` should remain publicly reachable. Alternatively, add a runtime `env::panic_str` at the top of v1 to make it unconditionally revert when called externally, preserving the internal delegation path from v2.

---

### Proof of Concept

1. Obtain any block hash `B` present in `mainchain_header_to_height` (e.g., via `get_block_hash_by_height`).
2. Retrieve the full transaction list for block `B` from a Bitcoin node.
3. Build the Merkle tree; select any internal node `N` at tree depth `d` and position `p`. Compute the sibling proof of length `(total_depth - d)`.
4. Call `verify_transaction_inclusion` on NEAR with:
   - `tx_id = N` (the internal node hash, a valid H256)
   - `tx_block_blockhash = B`
   - `tx_index = p * 2^d` (the leaf-equivalent index for this internal node)
   - `merkle_proof = <sibling path computed in step 3>`
   - `confirmations = 1`
5. The function returns `true` — falsely asserting that the internal node `N` is a confirmed transaction in block `B`.

The coinbase-proof guard in `verify_transaction_inclusion_v2` is never reached because the attacker calls v1 directly. [8](#0-7)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
