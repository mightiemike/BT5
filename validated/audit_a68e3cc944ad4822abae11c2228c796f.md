### Title
Linked Signer Can Irrevocably Perpetuate Its Own Authorization via `LinkSigner` Self-Rotation - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction is validated with `allowLinkedSigner=true`. This means the **currently active linked signer** can sign a new `LinkSigner` transaction to rotate the linked signer to any address it controls — before or after the subaccount owner attempts revocation — making revocation permanently ineffective.

---

### Finding Description

The `processTransactionImpl` function in `EndpointTx.sol` handles the fast-path `LinkSigner` transaction type as follows:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature`, which passes the current linked signer address to `Verifier.validateSignature`:

```solidity
function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the recovered signer if it equals either the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The `linkedSigners` mapping is the sole authorization state for this mechanism: [4](#0-3) 

Because `allowLinkedSigner=true` is passed for `LinkSigner` itself, the currently authorized linked signer can sign a valid `LinkSigner` transaction to replace itself with any address it controls. This is structurally identical to the ERC20 approve race condition: just as a spender can call `transferFrom` with the old allowance before a new `approve` is mined, a linked signer can submit a `LinkSigner` rotation before the owner's revocation is sequenced — and unlike ERC20, the linked signer can do this **proactively at any time**, not just reactively.

---

### Impact Explanation

A malicious linked signer can:

1. **Proactively rotate** to a new address it controls (no race required), making the owner's revocation target stale.
2. **Reactively front-run** the owner's `LinkSigner(address(0))` revocation by submitting `LinkSigner(newAddress)` with the same nonce to the sequencer first.
3. **Repeat indefinitely**: each time the owner submits a revocation, the linked signer consumes the nonce with a rotation, forcing the owner to chase an ever-changing address.

Once persistent access is established, the linked signer can sign:
- `WithdrawCollateral` / `WithdrawCollateralV2` — drain collateral from the subaccount
- `TransferQuote` — move quote balances to arbitrary recipients
- `MintNlp` / `BurnNlp` — manipulate NLP positions [5](#0-4) 

The corrupted state is `linkedSigners[subaccount]` and the resulting asset delta is the full collateral balance of the victim subaccount.

---

### Likelihood Explanation

Any linked signer that turns malicious (or is compromised) can exploit this immediately. No special privileges, no admin access, and no sequencer compromise is required — the linked signer only needs to submit a signed `LinkSigner` transaction with the next valid nonce, which is a normal protocol operation. The nonce is per-address and publicly readable from `nonces[address]`. [6](#0-5) 

Likelihood is **Medium**: it requires a malicious or compromised linked signer, but once that condition holds, exploitation is trivial and unstoppable on-chain.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the first 20 bytes of the `sender` subaccount) should be permitted to change the linked signer. The linked signer is a delegated key — it must not be able to re-delegate or perpetuate its own authorization.

```solidity
// Before (vulnerable):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // linked signer can change linked signer
);

// After (fixed):
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner can change linked signer
);
``` [7](#0-6)

### Proof of Concept

**Setup**: Alice's subaccount is `aliceSub`. Alice has linked Bob (`bobAddr`) as her linked signer. Current nonce for Alice's address is `N`.

**Step 1 — Bob proactively rotates (no race needed)**:
Bob signs `LinkSigner{ sender: aliceSub, signer: carolAddr, nonce: N }` using `bobAddr`'s private key, where `carolAddr` is another address Bob controls. Bob submits this to the sequencer.

**Validation path**:
- `validateNonce(aliceSub, N)` → passes (N is current nonce, increments to N+1)
- `validateSignature(aliceSub, digest, sig, true)` → `getLinkedSigner(aliceSub)` returns `bobAddr`; `ECDSA.recover(digest, sig)` returns `bobAddr`; check passes
- `linkedSigners[aliceSub] = carolAddr` [8](#0-7) 

**Step 2 — Alice attempts revocation**:
Alice signs `LinkSigner{ sender: aliceSub, signer: address(0), nonce: N+1 }`. Bob (as Carol) immediately signs `LinkSigner{ sender: aliceSub, signer: daveAddr, nonce: N+1 }` and submits it first.

**Result**: Nonce N+1 is consumed by Bob's rotation. Alice's revocation fails with `ERR_WRONG_NONCE`. `linkedSigners[aliceSub]` is now `daveAddr` (Bob's third address). Bob retains full signing authority indefinitely.

**Step 3 — Asset extraction**:
Bob (as `daveAddr`) signs `WithdrawCollateral{ sender: aliceSub, productId: QUOTE_PRODUCT_ID, amount: fullBalance, nonce: N+2 }` and submits it. The withdrawal is processed, draining Alice's subaccount. [9](#0-8)
