### Title
Duplicate Position IDs in `MsgClaimTierRewards` Allow Double-Claiming Tier Rewards - (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary
The `ClaimTierRewards` message handler iterates over a caller-supplied `PositionIds` slice without any deduplication check. A position owner can include the same position ID multiple times in a single `MsgClaimTierRewards` transaction, causing the reward computation to run against the same stale checkpoint state N times and pay out N times the entitled reward amount.

---

### Finding Description
In `ClaimTierRewards` (`x/tieredrewards/keeper/msg_server.go`), the handler builds a `positions` slice by fetching each ID from `msg.PositionIds` in a loop:

```go
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

There is no set-membership check or deduplication guard before appending. If `msg.PositionIds = [42, 42, 42]`, the same `PositionState` (with the same stale `LastEventSeq` checkpoint) is appended three times. `claimRewardsAndUpdatesPositions` then processes all three entries from the in-memory slice — each copy carries the same pre-claim checkpoint — and computes the full reward amount three times, sending three separate reward payouts to the owner before the checkpoint is finally written back.

The only per-entry validation performed inside the loop is an ownership check:

```go
if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
    return nil, err
}
``` [2](#0-1) 

This confirms no duplicate-ID rejection exists at the message-server level.

---

### Impact Explanation
A position owner submits one `MsgClaimTierRewards` with the same position ID repeated N times. The tiered rewards pool (`types.RewardsPoolName` module account) is debited N × (legitimate reward amount). The owner receives N times the rewards they are entitled to, directly draining the shared rewards pool and reducing the rewards available to all other delegators.

The corrupted value is the balance of the `RewardsPoolName` module account and the `totalBase` / `totalBonus` coin amounts transferred to the attacker. [3](#0-2) 

---

### Likelihood Explanation
Any position owner can trigger this with a single standard Cosmos SDK transaction. No privileged role, leaked key, or social engineering is required. The attacker only needs to own at least one tiered-rewards position with accrued rewards. The entry path is the normal `MsgClaimTierRewards` CLI/gRPC transaction path.

---

### Recommendation
Add a deduplication check before the accumulation loop. Reject the transaction if any position ID appears more than once in `msg.PositionIds`:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest,
            "duplicate position id %d in claim request", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Alternatively, enforce uniqueness at the `MsgClaimTierRewards.Validate()` level so the check is applied uniformly across all entry paths (direct tx, authz, etc.). [4](#0-3) 

---

### Proof of Concept

1. Alice creates a tiered-rewards position (ID = 42) and accrues, say, 100 BASECRO in rewards.
2. Alice broadcasts a single `MsgClaimTierRewards` with `position_ids = [42, 42, 42]` and `owner = alice`.
3. The loop fetches the same `PositionState{LastEventSeq: X}` three times and appends all three copies to `positions`.
4. `claimRewardsAndUpdatesPositions` processes each copy independently; each copy computes 100 BASECRO from the same stale checkpoint.
5. Alice receives 300 BASECRO instead of 100 BASECRO; the rewards pool is debited 300 BASECRO.
6. The position's checkpoint is updated once (the last write wins), so the position appears correctly updated on-chain, leaving no obvious trace of the over-payment. [5](#0-4)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}
```
