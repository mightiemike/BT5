### Title
`ValidResourceBounds` Protobuf Round-Trip Silently Demotes `AllResources` to `L1Gas`, Causing Transaction Hash Divergence Between Consensus and Sync Paths — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` in the P2P sync path uses a zero-value heuristic to decide between `L1Gas` and `AllResources`. A V3 transaction legitimately carrying `AllResourceBounds { l2_gas: 0, l1_data_gas: 0 }` is hashed as `AllResources` (4-element Poseidon preimage) at the gateway/consensus layer, but after a protobuf round-trip it is silently re-typed as `L1Gas` (3-element preimage), producing a structurally different hash. The two representations are not equivalent under `get_tip_resource_bounds_hash`, so any downstream component that recomputes the hash from the deserialized `Transaction` object — including re-execution, block-hash verification, and state-sync validation — will derive a value that does not match the hash committed to the block.

### Finding Description

**Serialization boundary.** `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436 implement `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 path
} else {
    ValidResourceBounds::AllResources(...)       // ← post-0.13.3 path
})
```

The heuristic was introduced to handle legacy 0.13.2 transactions that only carry L1 gas. The TODO comment on line 426 acknowledges this is temporary.

**Hash function sensitivity.** `get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` lines 188–211 produces a **different number of Poseidon inputs** depending on the variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements total
    ValidResourceBounds::AllResources(all) => {
        vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]     // 3 elements total
    }
});
```

When `l2_gas = 0` and `l1_data_gas = 0`, the packed felts for those two resources are both `0x0`, but the `AllResources` path still appends the `l1_data_gas` felt, making the Poseidon input vector length 3 instead of 2. The resulting hashes are distinct.

**Divergent paths for the same transaction.**

| Layer | Type used | `resource_bounds` variant | Hash preimage length |
|---|---|---|---|
| Gateway / Consensus (`InternalRpcInvokeTransactionV3`) | `AllResourceBounds` | always `AllResources` | 3 felts (tip + l1 + l2 + l1_data) |
| Storage / Sync (`InvokeTransactionV3`) after protobuf round-trip | `ValidResourceBounds` | `L1Gas` (when l2=0, l1_data=0) | 2 felts (tip + l1 + l2) |

The gateway always converts `RpcInvokeTransactionV3 → InternalRpcInvokeTransactionV3` preserving `AllResourceBounds`, and the hash is computed via `InvokeTransactionV3Trait::resource_bounds()` which returns `ValidResourceBounds::AllResources(self.resource_bounds)` (line 637–639 of `rpc_transaction.rs`). The storage layer converts to `InvokeTransactionV3` with `resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds)` (line 682 of `rpc_transaction.rs`). After a protobuf serialise/deserialise cycle the variant is silently changed to `L1Gas`, and any subsequent call to `Transaction::calculate_transaction_hash` or `validate_transaction_hash` produces a different value.

### Impact Explanation

Any component that reads a `Transaction` from the protobuf sync layer and recomputes its hash — including `validate_transaction_hash` used during block verification, `blockifier_reexecution` replay, and RPC `starknet_getTransactionByHash` hash cross-checks — will derive a hash that does not match the one committed to the block. This constitutes:

- **Wrong hash bound to the transaction**: the hash stored in the block differs from the hash recomputed from the deserialized transaction body, breaking hash-based integrity checks.
- **Wrong gas-vector computation mode**: `ValidResourceBounds::L1Gas` returns `GasVectorComputationMode::NoL2Gas` while `AllResources` returns `GasVectorComputationMode::All`, causing different fee and resource accounting during re-execution.
- **Wrong `calculate_resource_bounds` output** fed to the Cairo VM (`native/utils.rs` lines 92–103): `L1Gas` omits the `l1_data_gas` element from the resource-bounds vector, so the VM sees a 2-element vector instead of 3.

These map to: *Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input* (Critical) and *RPC execution or fee estimation returns an authoritative-looking wrong value* (High).

### Likelihood Explanation

The trigger condition — a V3 transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0` — is reachable by any unprivileged user submitting a standard invoke transaction that sets only L1 gas bounds (a common pattern for wallets that have not yet adopted the full 0.13.3 resource-bounds model). The gateway accepts such transactions without error because `AllResourceBounds` imposes no constraint on zero values. The type demotion is silent and produces no error.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, either:

1. Add an explicit discriminant field to the `ResourceBounds` protobuf message (e.g., `bool is_all_resources`) so the variant is preserved across serialization, or
2. Derive the variant from the transaction version: transactions at version ≥ 0x100000000000000000000000000000003 (the 0.13.3 boundary) always use `AllResources`; older ones use `L1Gas`. The version is already present in the enclosing transaction message and can be threaded into the converter.

The parallel converter in `rpc_transaction.rs` (lines 212–223) already takes the correct approach by deserializing directly into `AllResourceBounds` without any zero-value discrimination; the `transaction.rs` converter should be aligned with it for post-0.13.3 transactions.

### Proof of Concept

1. Submit an `InvokeV3` transaction with `resource_bounds = AllResourceBounds { l1_gas: (amount=1, price=1), l2_gas: (0,0), l1_data_gas: (0,0) }` via the RPC gateway.
2. The gateway calls `InternalRpcInvokeTransactionV3::calculate_transaction_hash` → `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(AllResources(...))` → 3-felt preimage → hash **H₁**.
3. The transaction is stored as `InvokeTransactionV3 { resource_bounds: ValidResourceBounds::AllResources(...) }` and included in a block with committed hash **H₁**.
4. A syncing node receives the block over P2P. The transaction is deserialized via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: since `l2_gas.is_zero() && l1_data_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`.
5. The syncing node calls `Transaction::calculate_transaction_hash` → `get_tip_resource_bounds_hash(L1Gas(...))` → 2-felt preimage → hash **H₂**.
6. **H₁ ≠ H₂**. The syncing node's hash-verification step rejects the transaction (or stores it under the wrong hash), causing a state divergence from the proposing node. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
