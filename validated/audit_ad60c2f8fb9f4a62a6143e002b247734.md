### Title
Deprecated `verify_transaction_inclusion` Remains Callable, Bypassing Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR function despite being deprecated. It lacks the coinbase Merkle proof check that `verify_transaction_inclusion_v2` enforces to prevent the 64-byte transaction Merkle forgery attack. Any unprivileged NEAR caller can invoke v1 directly, bypassing the security control entirely.

---

### Finding Description

The contract introduced `verify_transaction_inclusion_v2` specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (cited inline: https://www.bitmex.com/blog/64-Byte-Transactions). The v2 function requires the caller to supply a valid coinbase Merkle proof, which anchors the proof to a real transaction at index 0 and prevents an attacker from substituting an internal Merkle tree node as a fake `tx_id`.

However, the original `verify_transaction_inclusion` (v1) was only marked `#[deprecated]` — it was **not removed and not access-restricted**. It retains its `#[pause]` attribute and remains a fully callable public function: [1](#0-0) 

The v1 function's own documentation acknowledges the risk: [2](#0-1) 

The v2 function enforces the coinbase proof check before delegating to v1: [3](#0-2) 

But because v1 is independently reachable, the coinbase proof check in v2 is entirely optional from the caller's perspective. An attacker simply calls v1 directly and the coinbase validation is never executed.

The underlying Merkle computation in `merkle-tools` is a pure hash traversal with no structural constraints on the leaf value: [4](#0-3) 

`compute_root_from_merkle_proof` accepts any 32-byte value as `transaction_hash`. It does not distinguish between a real transaction hash and an internal Merkle node (which is also a 32-byte double-SHA256 output). This is the root cause that the coinbase proof check in v2 was designed to close.

---

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion` (v1) to gate an action — such as releasing bridged funds after a claimed Bitcoin payment — can be deceived into accepting a forged proof. The attacker never sends any Bitcoin. They supply an internal Merkle node of a real, already-confirmed block as the `tx_id`, paired with a crafted `merkle_proof` that reconstructs the block's known `merkle_root`. v1 returns `true`, and the recipient contract acts on a fabricated transaction.

The corrupted invariant is the proof result returned by `verify_transaction_inclusion`: it transitions from `false` (no such transaction) to `true` (transaction confirmed) based on a forged input, with no on-chain state change required to enable the attack.

---

### Likelihood Explanation

The 64-byte forgery technique is publicly documented and the contract itself cites the BitMEX writeup. The entry point (`verify_transaction_inclusion`) is a standard public NEAR view/call function requiring no role, no stake, and no privileged access. Any NEAR account can call it. The attacker only needs a real confirmed Bitcoin block (freely available) and the ability to locate an internal Merkle node — a straightforward offline computation.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) entirely, or restrict it so it is no longer externally callable (e.g., make it `pub(crate)` or gate it behind a role that no external caller holds). Deprecation alone does not prevent invocation. All external callers must be directed exclusively to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

1. Identify a real Bitcoin block `B` already tracked by the contract, with known `merkle_root = R`.
2. Obtain the full transaction list of `B`. Compute the Merkle tree offline. Select any internal node `N` at depth `d` (a 32-byte value).
3. Construct `merkle_proof` such that `compute_root_from_merkle_proof(N, tx_index, merkle_proof) == R`. This is trivially possible because `N` is itself a product of the same hash function used by `compute_root_from_merkle_proof`.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B.hash`, `tx_index`, `merkle_proof`, `confirmations = 1`.
5. v1 checks: block is on main chain ✓, confirmations satisfied ✓, `merkle_proof` non-empty ✓, `compute_root_from_merkle_proof(N, ...) == R` ✓ → returns `true`.
6. No coinbase proof was ever requested. The forgery succeeds. Any recipient contract consuming this result treats the non-existent transaction as confirmed. [5](#0-4)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
