### Title
Unprivileged `MsgSend` to `BurnedAddresses` entry bypasses `MaxSupply` cap in `BeginBlocker` — (`x/inflation/abci.go`)

### Summary

The `BeginBlocker` in `x/inflation/abci.go` computes effective circulating supply as `GetSupply(denom) - Σ GetBalance(burned_addr, denom)`. Because `BurnedAddresses` entries are validated only as valid bech32 addresses (not as bank-blocked accounts), any unprivileged actor can send bond-denom tokens to a burned address via a standard `MsgSend`, permanently inflating the `burned` subtrahend. This suppresses the apparent circulating supply and prevents the panic guard from triggering, allowing the SDK `x/mint` module to continue minting beyond `MaxSupply` indefinitely.

---

### Finding Description

**Entrypoint:** Standard `MsgSend` (bank module) from any account to any address listed in `params.BurnedAddresses`.

**Root cause — `validateBurnedAddresses` imposes no blocked-address constraint:** [1](#0-0) 

The only validation is `sdk.AccAddressFromBech32` — no check that the address is a module account or a bank-blocked address. Any regular account can therefore receive tokens.

**Root cause — `BeginBlocker` reads live balance of burned addresses:** [2](#0-1) 

`burned` is the sum of live balances of `BurnedAddresses`. `totalsupply` (the full on-chain supply) is then reduced by `burned` before the `MaxSupply` comparison. Sending tokens to a burned address increases `burned` without changing `totalsupply`, so `totalsupply - burned` decreases.

**Root cause — `GetAddressBalance` reads the live bank balance with no access control:** [3](#0-2) 

---

### Impact Explanation

The `BeginBlocker` is the **sole enforcement mechanism** for `MaxSupply`. It does not mint; it panics (chain halt) when `circulating_supply > MaxSupply`. The SDK `x/mint` module mints tokens every block with no awareness of this cap.

Attack steps:
1. Attacker monitors total supply approaching `MaxSupply`.
2. Attacker sends `Y` bond-denom tokens to a `BurnedAddresses` entry via `MsgSend`.
3. `burned` increases by `Y`; `totalsupply` is unchanged.
4. SDK mint module mints `X` tokens, pushing total supply to `MaxSupply + X`.
5. `BeginBlocker` computes `(MaxSupply + X) - (burned_before + Y)`. If `Y ≥ X`, the check `> MaxSupply` does not trigger.
6. No panic occurs; the chain continues operating; the SDK mint module mints again next block.
7. The attacker can repeat or pre-fund a large enough `Y` to suppress the guard indefinitely.

Net result: total on-chain supply exceeds `MaxSupply` by an unbounded amount, diluting all CRO holders. The attacker sacrifices `Y` tokens but, as a validator or delegator, recovers value from the newly minted inflation rewards.

---

### Likelihood Explanation

- Requires only a standard `MsgSend` — no special permissions, no governance, no key compromise.
- The burned address must be a regular (non-blocked) account, which `validateBurnedAddresses` does not prevent.
- The attacker needs to hold bond-denom tokens, which is trivially satisfied by any market participant.
- The attack is persistent: once tokens are sent to the burned address, the suppression is permanent (the burned address cannot spend them).

---

### Recommendation

1. **Require burned addresses to be bank-blocked accounts.** In `validateBurnedAddresses`, call `bankKeeper.BlockedAddr(addr)` and reject any address that is not blocked. Blocked addresses cannot receive tokens via `MsgSend`, making their balance immutable except through module-level minting/burning.

2. **Alternatively, track burned supply independently.** Instead of reading live balances, maintain a separate on-chain counter that is incremented only by an authorized burn operation (e.g., a dedicated `MsgBurn` gated by the module authority). This removes the attacker-controllable input entirely.

3. **Add an invariant check** that asserts `GetSupply(denom) - Σ burned_balances ≤ MaxSupply` and register it with the crisis module so it is enforced independently of `BeginBlocker`.

The exact state delta: `GetSupply` = `MaxSupply + X`, `burned` = `X`, `totalsupply - burned` = `MaxSupply` — the invariant `total minted supply - burned ≤ MaxSupply` is violated while the guard reports compliance.

```go
func validateBurnedAddresses(v any) error {
	burnedAddresses, ok := v.([]string)
	if !ok {
		return fmt.Errorf("invalid parameter type: %T", v)
	}

	addressMap := make(map[string]bool)

	for i, addr := range burnedAddresses {
		if _, err := sdk.AccAddressFromBech32(addr); err != nil {
			return fmt.Errorf("invalid burned address at index %d: %s, error: %w", i, addr, err)
		}

		if addressMap[addr] {
			return fmt.Errorf("duplicate burned address found: %s", addr)
		}
		addressMap[addr] = true

	}

	return nil
}
```
