### Title
`MsgTierUndelegate` and `MsgExitTierWithDelegation` Revert When Bonus Pool Is Insufficient, Blocking Users From Accessing Staked Tokens — (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

The `TierUndelegate`, `ExitTierWithDelegation`, `TierRedelegate`, `AddToTierPosition`, and `ClearPosition` handlers unconditionally call `claimRewards` before executing the user's requested operation. `claimRewards` internally calls `processEventsAndClaimBonus`, which hard-fails with `ErrInsufficientBonusPool` when the rewards pool cannot cover accrued bonus. Because the reward-settlement step is not separable from the operational step, a user whose bonus exceeds the pool balance is entirely blocked from undelegating, exiting, or redelegating their position — even though those operations are valid and independent of the pool balance.

---

### Finding Description

Every mutation handler that modifies a delegated position first settles rewards:

```go
// msg_server.go – TierUndelegate
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err          // ← blocks undelegate entirely
}
``` [1](#0-0) 

The same pattern appears in `TierRedelegate`, `AddToTierPosition`, `ClearPosition`, and `ExitTierWithDelegation`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

`claimRewards` calls `processEventsAndClaimBonus`, which checks the pool balance and returns a hard error if it is insufficient:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [6](#0-5) 

`sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` whenever `poolBalance < bonus`:

```go
if !poolBalance.IsAllGTE(bonus) {
    return errorsmod.Wrapf(types.ErrInsufficientBonusPool, ...)
}
``` [7](#0-6) 

There is no code path in any of these handlers that allows the operation to proceed while skipping or deferring the bonus claim. The user has no message available to undelegate or exit without first settling bonus rewards.

---

### Impact Explanation

A user who has a delegated tier position with accrued bonus rewards exceeding the current pool balance cannot:

- Undelegate their position (`MsgTierUndelegate`)
- Exit with delegation transfer (`MsgExitTierWithDelegation`)
- Redelegate to a different validator (`MsgTierRedelegate`)
- Add tokens to their position (`MsgAddToTierPosition`)
- Cancel an exit commitment (`MsgClearPosition`)

Their staked tokens remain locked inside the tier module's per-position delegator address until the rewards pool is replenished. The user's base rewards (routed through `x/distribution`) are also inaccessible via these paths because the entire transaction reverts. The corrupted invariant is: **a user's right to retrieve their own staked principal is gated on an unrelated module-account balance (`tieredrewards` rewards pool)**.

---

### Likelihood Explanation

The rewards pool is funded by inflation and governance. If the pool is depleted — either through sustained high claim volume, a governance decision to stop funding it, or a period of low inflation — every user with a delegated position whose accrued bonus exceeds the remaining pool balance is simultaneously blocked from all exit paths. This is a realistic operational condition, not a theoretical one: the ADR itself acknowledges it and instructs users to "retry after pool is replenished." [8](#0-7) 

The condition is reachable by any unprivileged account that holds a tier position; no privileged role is required to trigger it.

---

### Recommendation

Decouple the reward-settlement step from the operational step in `TierUndelegate`, `ExitTierWithDelegation`, `TierRedelegate`, `AddToTierPosition`, and `ClearPosition`. When `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the handler should either:

1. Proceed with the operation and defer the bonus claim (recording the owed amount for later settlement), or
2. Proceed with the operation and forfeit the bonus for that period (with explicit user consent via a message flag), or
3. At minimum, allow `MsgTierUndelegate` and `MsgExitTierWithDelegation` — the only paths to recover staked principal — to skip bonus settlement when the pool is insufficient, consistent with how `BeforeRedelegationSlashed` already handles this case:

```go
// slash.go – already forfeits bonus silently on pool shortage to avoid chain halt
if errors.Is(err, types.ErrInsufficientBonusPool) {
    k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
} else {
    return err
}
``` [9](#0-8) 

The same "proceed without bonus" logic applied in the slash hook should be available to user-driven exit paths.

---

### Proof of Concept

1. Alice creates a tier position via `MsgLockTier`, locking 1 000 000 uatom.
2. Time advances; Alice accrues 500 uatom in bonus rewards.
3. The rewards pool balance drops to 400 uatom (e.g., other users have claimed, or the pool was never fully funded).
4. Alice triggers exit via `MsgTriggerExitFromTier` and waits for the exit duration to elapse.
5. Alice submits `MsgTierUndelegate` to recover her principal.
6. `TierUndelegate` calls `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` (500 > 400).
7. `TierUndelegate` returns the error; the transaction reverts.
8. Alice's 1 000 000 uatom remain locked in the position's delegator address. She has no alternative message to undelegate without settling bonus rewards.
9. Alice is blocked until governance replenishes the pool — an event outside her control. [10](#0-9) [11](#0-10) [12](#0-11)

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil
}

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
