### Title
`isHealthy()` Always Returns `true`, Permanently Disabling Post-Trade Health Enforcement — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.isHealthy()` is a stub that unconditionally returns `true` and is never overridden anywhere in the codebase. The two `require(isHealthy(...))` guards in `matchOrders` that are supposed to enforce post-trade collateral health are therefore permanently inert, leaving the protocol with no on-chain defense against undercollateralized positions being created through order matching.

---

### Finding Description

In `OffchainExchange.sol`, after all balance deltas for a matched trade are applied, two health checks are executed:

```solidity
require(isHealthy(taker.order.sender), ERR_INVALID_TAKER);
require(isHealthy(maker.order.sender), ERR_INVALID_MAKER);
``` [1](#0-0) 

These calls invoke:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [2](#0-1) 

The function is declared `virtual`, which signals it was designed to be overridden by a concrete subclass with real health-checking logic (e.g., querying the clearinghouse for initial-margin sufficiency). However, a codebase-wide search for `isHealthy` returns **only matches inside `OffchainExchange.sol`** — no derived contract ever overrides it. The stub is the live implementation.

This is structurally identical to the reference bug: a safety flag/function that should be restrictive defaults to permissive (`true`), and the intended restriction is never applied.

---

### Impact Explanation

After `_updateBalances` writes the taker and maker balance deltas: [3](#0-2) 

…the two `require(isHealthy(...))` calls are supposed to revert the entire `matchOrders` call if either party's resulting position is undercollateralized. Because `isHealthy` always returns `true`, this revert path is unreachable. Any trade that passes signature and order-crossing validation will be committed to state regardless of the resulting collateral ratio. The protocol's on-chain solvency invariant after order matching is completely absent, enabling bad debt to accumulate silently.

---

### Likelihood Explanation

`matchOrders` is gated by `onlyEndpoint`: [4](#0-3) 

The Endpoint processes sequencer-submitted batches. The sequencer is expected to enforce health off-chain before matching, but the on-chain check is the protocol's last line of defense against sequencer bugs, stale price data at match time, or a compromised sequencer. With `isHealthy` permanently returning `true`, that defense does not exist. Any sequencer error or compromise that produces an undercollateralized match will succeed on-chain with no revert.

---

### Recommendation

Override `isHealthy` in `OffchainExchange` (or a concrete subclass) to perform a real collateral check via the clearinghouse, for example:

```solidity
function isHealthy(bytes32 subaccount)
    internal
    view
    override
    returns (bool)
{
    return IClearinghouse(clearinghouse)
        .getHealth(subaccount, IProductEngine.HealthType.INITIAL) >= 0;
}
```

Until a real implementation is provided, the `virtual` stub should be replaced with an explicit `revert` or `abstract` declaration so the missing override is caught at compile time rather than silently bypassed at runtime.

---

### Proof of Concept

1. Sequencer (or compromised sequencer) submits a `MatchOrders` transaction pairing a taker order whose resulting position would be undercollateralized.
2. `matchOrders` validates signatures, crosses the orders, applies balance deltas via `_updateBalances`.
3. `require(isHealthy(taker.order.sender))` is reached — `isHealthy` returns `true` unconditionally.
4. The transaction succeeds; the undercollateralized position is committed to the engine's balance storage.
5. The protocol now holds bad debt with no on-chain record of the health violation. [2](#0-1) [1](#0-0)
