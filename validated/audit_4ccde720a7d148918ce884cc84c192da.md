### Title
`verify_transaction_inclusion` hardcodes a non-empty merkle proof assumption, blocking SPV verification of single-transaction blocks — (File: `contract/src/lib.rs`)

---

### Summary
Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` unconditionally reject any call where `merkle_proof` is empty. For a block containing exactly one transaction, the correct merkle proof **is** the empty vector (the transaction hash equals the merkle root by definition). The hardcoded `require!(!args.merkle_proof.is_empty())` guard is the direct analog of the Curve bug: a fixed-size assumption about a variable-length structure that silently excludes a valid subset of inputs.

---

### Finding Description

In `verify_transaction_inclusion`, the contract enforces:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

`verify_transaction_inclusion_v2` delegates to this deprecated function after its own coinbase check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

For a block with exactly one transaction (e.g., a coinbase-only block), the merkle tree has depth 0. `merkle_proof_calculator` correctly returns an empty `Vec` in this case because the `while current_hashes.len() > 1` loop never executes: [3](#0-2) 

Likewise, `compute_root_from_merkle_proof` with an empty proof simply returns the input `transaction_hash` unchanged, which equals the block's `merkle_root` when there is only one transaction: [4](#0-3) 

The guard on line 315 therefore rejects a mathematically valid proof, making it impossible for any caller to verify a transaction in a single-transaction block through either public API.

---

### Impact Explanation

**Impact: Low.** No funds are lost and no chain state is corrupted. However, the contract cannot fulfill its core SPV promise for a well-defined class of real Bitcoin blocks. Any downstream NEAR contract or user that needs to prove inclusion of a transaction in a coinbase-only block will receive a panic instead of a result, and the contract would require redeployment to fix this.

---

### Likelihood Explanation

**Likelihood: Low-to-moderate.** Coinbase-only blocks (blocks mined with no user transactions) occur on mainnet during periods of very low mempool activity and have appeared historically. They are a standard, protocol-valid block type. Any relayer that submits such a block header to the light client will cause the header to be stored correctly, but no caller will ever be able to call `verify_transaction_inclusion_v2` against it successfully.

---

### Recommendation

Remove the blanket `require!(!args.merkle_proof.is_empty())` guard and instead handle the single-transaction case explicitly: if `merkle_proof` is empty, directly compare `tx_id` to `header.block_header.merkle_root` and return the result. This mirrors how `compute_root_from_merkle_proof` already behaves for an empty proof slice.

---

### Proof of Concept

Attacker-controlled entry path (unprivileged NEAR caller):

1. A relayer submits a real Bitcoin block header for a coinbase-only block. The header is accepted and stored normally.
2. A caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase tx hash (= merkle root)
   - `tx_index` = 0
   - `merkle_proof` = `[]`
   - `coinbase_tx_id` = same coinbase tx hash
   - `coinbase_merkle_proof` = `[]`
3. The length check `merkle_proof.len() == coinbase_merkle_proof.len()` passes (`0 == 0`). [5](#0-4) 
4. The coinbase check `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[]) == merkle_root` passes because the empty-proof path returns `coinbase_tx_id` directly, which equals `merkle_root`. [6](#0-5) 
5. The call delegates to `verify_transaction_inclusion`, which panics at `require!(!args.merkle_proof.is_empty())`. [1](#0-0) 

The proof is valid; the contract rejects it solely because of the hardcoded non-empty assumption.
