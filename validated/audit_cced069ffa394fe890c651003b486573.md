### Title
Unhandled Return Value of `approve` in `creditDeposit` Permanently Blocks Collateral Deposits for Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` via the raw `IIERC20Base` interface without checking the boolean return value. No `safeApprove` equivalent exists anywhere in the project for this code path. For any spot-listed token whose `approve` silently returns `false` instead of reverting, the allowance is never set, the subsequent `depositCollateralWithReferral` call reverts due to zero allowance, and user funds sent to the DDA address are permanently inaccessible through the normal deposit flow.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is the core function that sweeps token balances held by a Direct Deposit Address (DDA) into the Nado protocol on behalf of a subaccount. [1](#0-0) 

Inside the loop, the raw `IIERC20Base.approve` call is made and its return value is silently discarded: [2](#0-1) 

The `IIERC20Base` interface declares `approve` as returning `bool`: [3](#0-2) 

The project's `ERC20Helper` library implements `safeTransfer` and `safeTransferFrom` with proper return-value checking, but provides **no** `safeApprove` equivalent: [4](#0-3) 

The same pattern appears in `ContractOwner.wrapVaultAsset()`, which also calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` without checking return values, and has no access modifier (callable by anyone): [5](#0-4) 

---

### Impact Explanation

For any spot-listed ERC20 token whose `approve` implementation silently returns `false` (a known pattern in non-standard tokens such as early USDT, BNB, and others):

1. `token.approve(address(endpoint), balance)` executes, returns `false`, and the allowance remains at zero — no revert, no error.
2. `endpoint.depositCollateralWithReferral(...)` is called; the endpoint's internal `transferFrom` on the DDA contract reverts because allowance is zero.
3. The entire `creditDeposit()` transaction reverts.
4. The user's tokens are held in the DDA contract with no way to deposit them through the normal protocol flow.
5. Recovery requires the contract owner to call `withdraw()` manually — the user cannot self-rescue.

This constitutes a **collateral lock**: user funds are trapped in the DDA contract indefinitely until privileged intervention.

---

### Likelihood Explanation

- `creditDeposit()` has **no access control** — any address can trigger it.
- `ContractOwner.creditDepositV1(bytes32 subaccount)` also has no access control and routes directly to `creditDeposit()`. [6](#0-5) 
- The Nado protocol is designed to support multiple spot tokens. If any listed token has a non-reverting `approve` that returns `false`, every deposit attempt for that token via the DDA flow will fail.
- The likelihood is **medium**: standard tokens (USDC, WETH) revert on failure, but the protocol's design intent to support arbitrary spot tokens makes this a realistic risk as new tokens are listed.

---

### Recommendation

Add a `safeApprove` function to `ERC20Helper.sol` mirroring the existing `safeTransfer` pattern, and replace the bare `approve` calls in `DirectDepositV1.creditDeposit()` and `ContractOwner.wrapVaultAsset()` with it:

```solidity
// In ERC20Helper.sol
function safeApprove(
    IERC20Base self,
    address spender,
    uint256 amount
) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Apply the same fix to `DirectDepositV1.sol` using its local `IIERC20Base` interface, consistent with the existing `safeTransfer` helper already defined in that file: [7](#0-6) 

---

### Proof of Concept

1. A non-standard ERC20 token `T` is listed as a spot product. Its `approve` always returns `false` without reverting.
2. A user sends `1000 T` to their DDA address.
3. Anyone calls `ContractOwner.creditDepositV1(subaccount)` (no access control).
4. Inside `DirectDepositV1.creditDeposit()`, `token.approve(endpoint, 1000)` is called — returns `false`, allowance stays at 0, no revert.
5. `endpoint.depositCollateralWithReferral(subaccount, productId, 1000, "-1")` is called; the endpoint's `transferFrom(dda, ..., 1000)` reverts: `ERC20: insufficient allowance`.
6. The entire transaction reverts. The `1000 T` remain in the DDA contract.
7. Step 3–6 can be repeated indefinitely; the deposit can never succeed.
8. Only the owner can recover the tokens via `DirectDepositV1.withdraw()`.
