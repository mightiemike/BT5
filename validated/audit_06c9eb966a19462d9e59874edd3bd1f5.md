### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is still a live, unpermissioned public entry point despite being deprecated in favour of `verify_transaction_inclusion_v2`. Any NEAR caller can invoke it directly, bypassing the coinbase Merkle proof check that `v2` adds to block the 64-byte transaction Merkle proof forgery attack. The contract's own documentation acknowledges the function can return `true` for an internal Merkle tree node rather than a real transaction hash.

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` but remains `pub` and `#[pause]`-gated (i.e., callable by anyone when the contract is not paused). [1](#0-0) 

The function's own warning block states: [2](#0-1) 

The verification logic in `verify_transaction_inclusion` passes the caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` with no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle node: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools` is a pure hash-chain computation that treats every input identically — it cannot distinguish a real transaction hash from an internal node: [4](#0-3) 

`verify_transaction_inclusion_v2` closes this gap by first verifying a coinbase proof at index 0 against the same Merkle root before delegating to `v1`: [5](#0-4) 

Because the coinbase is always a leaf at index 0, a valid coinbase proof of the same depth as the target proof guarantees the target is also a leaf. Callers who invoke `v1` directly skip this guard entirely.

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion` (e.g., a cross-chain bridge or token-unlock contract) can be deceived into accepting a forged Bitcoin transaction inclusion proof. The attacker supplies a 64-byte internal Merkle node as `tx_id`; the function returns `true` for a Bitcoin transaction that never existed. In a bridge context this directly enables theft of locked assets on the NEAR side.

### Likelihood Explanation

The entry point is public, requires no role, no stake, and no privileged key — any NEAR account can call it. The 64-byte Merkle forgery technique is well-documented (https://www.bitmex.com/blog/64-Byte-Transactions) and the contract's own comments acknowledge the exact attack vector. Downstream integrators who read the ABI or copy older example code will naturally reach for the simpler `v1` function.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add an `#[private]` attribute so it is only callable by the contract itself (as it is already used internally by `verify_transaction_inclusion_v2`). All external callers must be migrated to `verify_transaction_inclusion_v2`. [6](#0-5) 

### Proof of Concept

1. Deploy the contract with a real Bitcoin block header whose Merkle root is known.
2. Identify two adjacent internal Merkle nodes `N_L` and `N_R` at depth `d-1` (one level above the leaves) such that `double_sha256(N_L || N_R)` equals the block's Merkle root.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N_L` (an internal node, not a real transaction)
   - `tx_index = 0`
   - `merkle_proof = [N_R]` (a single-element proof)
   - `tx_block_blockhash` = the real block hash
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N_L, 0, [N_R])` computes `double_sha256(N_L || N_R)` = Merkle root → the function returns `true`.
5. No real transaction with hash `N_L` exists; the proof is entirely forged.

The same call to `verify_transaction_inclusion_v2` would fail at the coinbase proof length check or the coinbase root comparison, blocking the attack. [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
