### Title
`DirectDepositV1.creditDeposit()` Deposits to First Matching Product ID When Same Token Is Registered Under Multiple Products — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates over all spot product IDs and deposits any token balance it finds to the **first** product ID whose token matches. Because `SpotEngine` does not enforce token-address uniqueness across product IDs, the same token can be registered under multiple products. When that happens, the deposit is silently credited to the wrong product ID — the user has no mechanism to specify their intended product.

---

### Finding Description

`creditDeposit()` is an unrestricted `external` function that loops over every product ID returned by `spotEngine.getProductIds()`:

```solidity
// DirectDepositV1.sol lines 83-101
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(
                subaccount,
                productId,
                uint128(balance),
                "-1"
            );
        }
    }
}
``` [1](#0-0) 

On the first iteration that finds a non-zero balance for a given token, the **entire balance** is transferred to the `Endpoint` and credited to that product ID. All subsequent iterations for the same token address find a zero balance and skip. The product ID that receives the deposit is therefore determined solely by its position in the `productIds` array — not by any user intent.

`SpotEngine.addOrUpdateProduct()` stores `configs[productId] = config` with no uniqueness check on `config.token`:

```solidity
// SpotEngine.sol lines 68-97
function addOrUpdateProduct(
    uint32 productId, uint32 quoteId, int128 sizeIncrement, int128 minSize,
    Config calldata config, RiskHelper.RiskStore calldata riskStore
) public onlyOwner {
    bool isNewProduct = _addOrUpdateProduct(...);
    configs[productId] = config;   // no token-uniqueness enforcement
    ...
}
``` [2](#0-1) 

This means the same token address can legitimately appear under two different product IDs (e.g., the same stablecoin listed under a standard product and a vault-wrapped product with different risk weights). When that configuration exists, `creditDeposit()` will always credit the lower-indexed product ID, regardless of which one the user intended.

A secondary surface: `creditDeposit()` carries no access control modifier. Any external caller — including a griefing attacker — can invoke it at any time, forcing the deposit to be processed before the user is ready or before the intended product ID is the first match. [3](#0-2) 

`ContractOwner.creditDepositV1()` also exposes this path with no access control:

```solidity
// ContractOwner.sol lines 502-508
function creditDepositV1(bytes32 subaccount) external {
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [4](#0-3) 

---

### Impact Explanation

Different product IDs carry different risk weights (`longWeightInitialX18`, `shortWeightInitialX18`), interest rates, and `withdrawFeeX18` values. A deposit credited to the wrong product ID results in:

- **Incorrect collateral health**: a lower-weight product ID reduces the user's initial/maintenance health, potentially triggering unexpected liquidation.
- **Wrong interest accrual**: the deposited balance accrues interest under the wrong product's rate model.
- **Inability to use collateral for the intended product**: the user cannot easily move the balance to the correct product without going through a withdrawal-and-redeposit cycle.

Funds are not permanently lost, but the user's on-chain accounting state is corrupted relative to their intent.

**Impact: Medium**

---

### Likelihood Explanation

The precondition — the same token registered under two product IDs — is an admin-controlled configuration, but the protocol provides no guard against it. The `wrapVaultAsset` flow in `ContractOwner` explicitly handles ERC-4626 vault tokens whose underlying asset may also be a registered spot token, making the dual-registration scenario realistic in production. The permissionless `creditDeposit()` entry point means no privileged actor is needed to trigger the wrong-product-ID path once the configuration exists.

**Likelihood: Medium**

---

### Recommendation

1. **Add a `productId` parameter to `creditDeposit()`** so the caller explicitly specifies which product to deposit to, eliminating ambiguous iteration-order selection.
2. **Enforce token-address uniqueness** in `SpotEngine.addOrUpdateProduct()` by maintaining a reverse mapping `tokenToProductId` and reverting if the token is already registered under a different product ID.
3. **Add access control** to `creditDeposit()` (e.g., `onlyOwner` or restrict to the `ContractOwner` proxy) to prevent unauthorized callers from triggering deposits at unintended times.

---

### Proof of Concept

1. Admin calls `SpotEngine.addOrUpdateProduct()` registering token `T` under product ID `P1` (low `longWeightInitialX18 = 0.5e18`) and again under product ID `P2` (high `longWeightInitialX18 = 0.9e18`). `P1 < P2` so `P1` appears first in `productIds`.
2. User sends `1000 T` to their `DirectDepositV1` contract, intending to deposit to `P2` for higher collateral value.
3. Any caller (including an attacker) calls `creditDeposit()` or `ContractOwner.creditDepositV1(subaccount)`.
4. Iteration hits `P1` first: `balanceOf(DDA) = 1000 T > 0` → approves and calls `endpoint.depositCollateralWithReferral(subaccount, P1, 1000T, "-1")`. Full balance transferred.
5. Iteration hits `P2`: `balanceOf(DDA) = 0` → skipped.
6. User's subaccount now holds `1000 T` under `P1` with `longWeightInitialX18 = 0.5e18` instead of the intended `P2` with `0.9e18`. The user's initial health is computed at half the expected collateral value, exposing them to premature liquidation. [1](#0-0) [5](#0-4)
