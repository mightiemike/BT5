### Title
Linked Signer Can Self-Escalate Authority via `LinkSigner` Fast Path — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path handler passes `allowLinkedSigner=true` to `validateSignedTx`, which means the current linked signer's signature is accepted as valid authorization to overwrite `linkedSigners[sender]`. A compromised linked signer can therefore replace itself with an attacker-controlled address without any consent from the subaccount owner.

---

### Finding Description

In the sequencer fast path, the `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with `allowLinkedSigner=true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer address to `verifier.validateSignature`: [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals **either** the subaccount owner address **or** the current linked signer: [3](#0-2) 

The `LinkSigner` EIP-712 digest commits only to `sender`, `signer`, and `nonce` — there is no field that restricts who the new `signer` value may be: [4](#0-3) 

After validation passes, `linkedSigners[sender]` is unconditionally overwritten with the attacker-supplied address: [5](#0-4) 

By contrast, the slow-mode `LinkSigner` path uses `validateSender`, which enforces that `msg.sender` is the subaccount owner — it does **not** accept a linked signer signature: [6](#0-5) 

This asymmetry is the root cause: the slow-mode path correctly restricts `LinkSigner` to the owner, but the fast path does not.

---

### Impact Explanation

A compromised linked signer key can:

1. Craft `LinkSigner(sender=S, signer=attacker, nonce=N)` and sign it with the compromised linked signer key.
2. Submit via the sequencer; `validateSignedTx` passes because `recovered == linkedSigner`.
3. `linkedSigners[S]` is now the attacker's address.
4. The attacker immediately uses the new linked signer authority to sign `WithdrawCollateral` or `TransferQuote` transactions, draining the subaccount before the owner can react.

**Clarification on "permanent loss":** The owner's address (derived from `bytes20(sender)`) is still accepted by `validateSignature`, so the owner retains the ability to sign transactions directly and can issue a new `LinkSigner` to revoke the attacker. However, funds can be drained in the window between the attacker's `LinkSigner` and the owner's recovery transaction. The invariant broken is that **a linked signer must not be able to escalate or transfer its own authority** — this invariant is concretely violated.

---

### Likelihood Explanation

Any party that obtains the linked signer private key (e.g., a compromised trading bot, a leaked API key, a malicious service that was granted linked signer status) can execute this in a single sequencer submission. No owner interaction, governance action, or additional privilege is required beyond possession of the linked signer key.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` fast-path handler, consistent with the slow-mode path:

```solidity
// EndpointTx.sol ~line 581
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the owner may set/change the linked signer
);
```

This ensures that only the subaccount owner (whose address is embedded in `sender`) can authorize a change to `linkedSigners`, matching the security model of the slow-mode path.

---

### Proof of Concept

```solidity
// Hardhat pseudocode — no modifications to protocol required
const subaccountOwner = signers[0];
const linkedSigner   = signers[1];   // compromised
const attacker       = signers[2];

// 1. Owner legitimately sets linkedSigner
await endpoint.connect(subaccountOwner).submitSlowModeTransaction(
    encodeLinkSigner(subaccountS, linkedSigner.address)
);
assert(await endpoint.linkedSigners(subaccountS) == linkedSigner.address);

// 2. Compromised linkedSigner crafts and signs a LinkSigner tx pointing to attacker
const nonce = await endpoint.nonces(ownerAddress);
const tx = encodeLinkSignerFastPath(subaccountS, attacker.address, nonce);
const sig = await linkedSigner.signTypedData(..., tx);  // linked signer signs

// 3. Sequencer submits it
await endpoint.submitTransactions([tx + sig]);

// 4. Assert: linkedSigners[S] is now attacker — no owner consent obtained
assert(await endpoint.linkedSigners(subaccountS) == attacker.address);

// 5. Attacker drains funds using new linked signer authority
const withdrawTx = encodeWithdrawCollateral(subaccountS, MAX_AMOUNT, nonce+1);
const attackerSig = await attacker.signTypedData(..., withdrawTx);
await endpoint.submitTransactions([withdrawTx + attackerSig]);
```

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
