Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-supplied `owner` instead of real `sender`, allowing full allowlist bypass - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (`sender`, the real caller) and instead gates on `owner`, a value freely supplied by the caller of `addLiquidity`. Because `owner` is unconstrained, any unprivileged address can bypass the deposit allowlist by passing an already-allowlisted address as `owner`, completely defeating the pool admin's access control.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` parameter and at line 191 calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding the real caller as `sender` and the caller-chosen value as `owner`. [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded. The allowlist check at line 38 reads `allowedDepositor[msg.sender][owner]`, gating on the caller-controlled `owner` value rather than the real caller. [2](#0-1) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender` (the real caller): `allowedSwapper[msg.sender][sender]`. [3](#0-2) 

Exploit path:
1. Pool is deployed with `DepositAllowlistExtension`; only `allowedLP` is allowlisted.
2. `attacker` (not allowlisted) calls `pool.addLiquidity(owner = allowedLP, salt, deltas, callbackData, "")`.
3. `_beforeAddLiquidity(attacker, allowedLP, ...)` is called; the extension checks `allowedDepositor[pool][allowedLP]` → `true`.
4. The guard passes; `LiquidityLib.addLiquidity` credits LP shares to `allowedLP` and pulls tokens from `attacker` via callback.
5. `removeLiquidity` enforces `msg.sender == owner`, so `attacker` cannot reclaim the deposited tokens. [4](#0-3) 

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism to restrict who may provide liquidity. With this bug, any address can deposit into a restricted pool by naming an allowlisted address as `owner`. The allowlist invariant is completely broken. Additionally, an attacker can inject liquidity into specific bins credited to an allowlisted address without that address's consent, altering the pool's liquidity distribution and potentially affecting LP returns for legitimate participants. This constitutes a broken core pool functionality (access control) with direct impact on pool state and LP positions, meeting the contest's High severity threshold. [5](#0-4) 

## Likelihood Explanation

- The `owner` parameter in `addLiquidity` is completely unconstrained; no validation prevents an arbitrary address from being supplied.
- Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` reads.
- No special privilege or precondition is required — any EOA or contract can execute the bypass in a single transaction.
- The `DepositAllowlistExtension` is a production periphery contract intended for real pool deployments. [6](#0-5) 

## Recommendation

Change `beforeAddLiquidity` to gate on the first parameter (`sender`, the real caller) rather than `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

## Proof of Concept

```solidity
// Pool deployed with DepositAllowlistExtension.
// allowedLP is allowlisted; attacker is not.

vm.prank(attacker);
pool.addLiquidity(
    allowedLP,       // owner = allowlisted address — passes the guard
    someArbitrarySalt,
    deltas,
    callbackData,    // attacker pays tokens via callback
    ""
);
// Result: deposit succeeds despite attacker not being allowlisted.
// LP shares credited to allowedLP; attacker's tokens consumed by pool.
// allowedDepositor[pool][allowedLP] == true → NotAllowedToDeposit never reverts.
``` [7](#0-6)
