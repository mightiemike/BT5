### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Alternative Entry Path — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` gates the 1-byte secp opcodes (64 = `secp256k1_verify`, 65 = `secp256r1_verify`) behind the `ENABLE_SECP_OPS` flag. However, the same two secp functions are also reachable via their 4-byte opcode aliases (`0x13d61f00`, `0x1c3a8f00`) through a completely separate dispatch branch that performs **no flag check at all**. An attacker-controlled CLVM program can invoke secp operations unconditionally by using the 4-byte form, bypassing the `ENABLE_SECP_OPS` guard entirely.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method contains two independent dispatch branches:

**Branch 1 — 4-byte opcodes (lines 157–182):** Entered when `op_len == 4`. The two secp opcodes are matched and dispatched directly with no flag check:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);
``` [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** Entered when `op_len == 1`. The same two secp functions are gated by `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag is defined as:

> "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)." [3](#0-2) 

The `MEMPOOL_MODE` constant does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

This means in mempool mode, 1-byte secp opcodes are rejected (they fall through to `unknown_operator`, which returns `Err(EvalErr::Unimplemented)` because `NO_UNKNOWN_OPS` is set), but the 4-byte secp opcodes execute unconditionally.

---

### Impact Explanation

**Vulnerability class: flag/operator wiring error — missing guard on an alternative entry path.**

The concrete broken invariant: `ENABLE_SECP_OPS = false` is supposed to prevent secp signature verification from executing. The 4-byte opcode path violates this invariant. Any caller (including the Python wheel via `api.rs`) that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` and passes attacker-controlled CLVM bytes containing the 4-byte secp opcodes will execute `op_secp256k1_verify` or `op_secp256r1_verify` anyway.

The corrupted result is the `Response` value: instead of `Err(EvalErr::Unimplemented)` (the expected outcome when the flag is absent), the evaluator returns `Ok(Reduction(cost, nil))` or `Err(EvalErr::Secp256Failed)` — the actual secp verification result. This is a consensus/mempool divergence: a program using 4-byte secp opcodes is accepted where the equivalent 1-byte form would be rejected. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are explicitly documented in the source code comments and appear in the fuzzing harness (`clvm-fuzzing/src/make_tree.rs`) and benchmark tool (`tools/src/bin/benchmark-clvm-cost.rs`), making them discoverable. Any attacker who can submit CLVM programs (the standard attacker model for this codebase) can craft a program using the 4-byte form. No special privileges are required beyond the ability to submit a transaction.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch branch, mirroring the guard already present for the 1-byte aliases:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

If the 4-byte forms are intentionally always available (hardforked in unconditionally), then the `ENABLE_SECP_OPS` flag documentation and the 1-byte guard should be updated to reflect that the flag only controls the 1-byte aliases, not secp availability in general, to prevent callers from relying on the flag as a complete secp gate.

---

### Proof of Concept

A CLVM program that invokes `secp256k1_verify` via the 4-byte opcode `0x13d61f00` will execute successfully even when `ChiaDialect` is constructed without `ENABLE_SECP_OPS`:

```rust
// Attacker-controlled CLVM bytes: (0x13d61f00 pubkey msg sig)
// ChiaDialect constructed with MEMPOOL_MODE (no ENABLE_SECP_OPS)
let dialect = ChiaDialect::new(MEMPOOL_MODE);
// run_program with a program whose operator atom is the 4-byte 0x13d61f00
// → dispatches to op_secp256k1_verify unconditionally
// → returns Ok(Reduction(1300000, nil)) or Err(Secp256Failed)
// Expected: Err(EvalErr::Unimplemented) — the same result as opcode 64 without ENABLE_SECP_OPS
```

The 4-byte opcode path at lines 175–182 returns before reaching the `ENABLE_SECP_OPS` check at lines 248–249, so the guard is never evaluated. [6](#0-5)

```rust
pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
