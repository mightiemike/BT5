Audit Report

## Title
SwapAllowlistExtension Checks Immediate Caller Instead of End User, Enabling Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool binds to `msg.sender` of `pool.swap()` — the immediate caller. When `MetricOmmSimpleRouter` is used, the router is `msg.sender`, so the extension checks the router's address rather than the end user's. Any pool admin who allowlists the router to enable router-mediated swaps for permitted users simultaneously grants every unpermitted user the ability to bypass the individual allowlist by calling the same public router.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // <-- immediate caller of pool.swap()
  recipient,
  ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` value:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`), the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The router is `msg.sender` to the pool, so `sender` passed to the extension equals `address(router)`. If the pool admin allowlists the router (`allowedSwapper[pool][router] = true`) to enable any router-mediated swap for permitted users, the check passes for **every** caller of the router regardless of their individual allowlist status.

The `FullMetricExtensionTest.test_allowedSwapSucceeds` test confirms this binding: it allowlists `address(callers[0])` (the `TestCaller` wrapper contract that calls `pool.swap()`), not `users[0]` (the human address), because the pool passes the immediate caller as `sender`.

`DepositAllowlistExtension.beforeAddLiquidity` avoids this problem by ignoring `sender` and checking `owner` — the LP position owner — which correctly identifies the economic beneficiary regardless of who the immediate caller is:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol:32-41
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  ...
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
```

The swap extension has no equivalent "owner" binding — it only receives `sender` (immediate caller) and `recipient` (output destination), neither of which reliably identifies the originating user when routing through an intermediary.

## Impact Explanation

A pool admin configuring `SwapAllowlistExtension` to restrict swaps to a curated set of users faces an unresolvable configuration dilemma:

- **Do not allowlist the router** → all router-mediated swaps revert, even for permitted users. The primary periphery swap entrypoint is unusable on this pool.
- **Allowlist the router** → `allowedSwapper[pool][router] = true` passes for any `sender = router`, regardless of who called the router. Every non-permitted user bypasses the individual allowlist by calling `MetricOmmSimpleRouter.exactInput/exactOutput`, which is a public, permissionless contract.

There is no configuration that simultaneously allows permitted users to swap through the router and blocks non-permitted users from doing the same. The allowlist is therefore unenforceable on any pool that supports router-mediated swaps. Consequences include unauthorized fee extraction, unauthorized price impact on LP positions, and violation of compliance or regulatory restrictions the pool admin intended to enforce — all constituting direct fund-level impact on LP positions.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entrypoint in the periphery layer. Any pool that configures `SwapAllowlistExtension` and also wants to support router swaps — the common case — must allowlist the router, triggering the bypass. The attacker requires no special privileges: they call the public router with a valid swap, and the extension passes because the router is allowlisted. The attack is repeatable on every block.

## Recommendation

The extension must check the identity of the economic actor, not the intermediate relay. Two approaches:

1. **Extension-level fix (mirrors deposit allowlist design)**: Add an explicit `swapper` field to `extensionData` that the router populates with `msg.sender` (the actual user) before forwarding. The extension reads this field and checks it against the allowlist instead of checking `sender`. This mirrors how `DepositAllowlistExtension` checks `owner` rather than the immediate caller.

2. **Pool-level fix**: Have `pool.swap()` accept an explicit `originator` parameter (the end user) that the router forwards as `msg.sender`. The extension then checks the true originator. This requires a core interface change.

Either way, the invariant must be: the identity checked by `SwapAllowlistExtension` is the address that economically initiates the swap, not the intermediate contract that relays it.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as `extension2` and `beforeSwap` order set to `2`.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, router, true)` — necessary to allow any router-mediated swap for permitted users.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. Router calls `pool.swap(...)` → pool passes `sender = address(router)` to `_beforeSwap` → extension checks `allowedSwapper[pool][router]` → `true` → swap succeeds.
6. Attacker has swapped on a pool they were explicitly not permitted to access.

Foundry test plan: extend `FullMetricExtensionTest` — allowlist `address(router)` on the pool, do **not** allowlist `users[1]`, call `router.exactInputSingle` from `users[1]`, and assert the swap succeeds (demonstrating the bypass). Then assert a direct `pool.swap()` call from `users[1]` reverts with `NotAllowedToSwap` (demonstrating the check works correctly for direct calls).
