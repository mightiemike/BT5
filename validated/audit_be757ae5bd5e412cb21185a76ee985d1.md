Audit Report

## Title
Swap allowlist bypassed via router: `sender` is router address, not originating EOA — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap` call. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating EOA. Any EOA can therefore bypass the allowlist by routing through an allowlisted router, completely defeating the extension's core invariant.

## Finding Description
The call chain is confirmed by production code:

**Step 1:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. [1](#0-0) 

At this point, `msg.sender` inside the pool is the router address, not the originating EOA.

**Step 2:** `MetricOmmPool.swap` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`. [2](#0-1) 

**Step 3:** `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension. [3](#0-2) 

**Step 4:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router address. [4](#0-3) 

If the pool admin has allowlisted the router (`allowedSwapper[pool][router] = true`), the check passes for every EOA that calls through the router, regardless of whether that EOA is individually blocked. The originating EOA identity is never checked.

The existing `allowAllSwappers` escape hatch is a separate flag and does not mitigate this — the bug is in the `allowedSwapper` path used for curated pools that still want router UX. [5](#0-4) 

## Impact Explanation
The swap allowlist extension's core invariant — that only explicitly permitted addresses may swap on a curated pool — is completely broken for any pool that allowlists the router. A disallowed EOA executes swaps that the pool admin intended to block. This is broken core functionality of a production extension, matching the "Admin-boundary break" and "Broken core pool functionality" allowed impact categories. The contest's own target scope explicitly identifies this exact vector: "Test direct swaps and router swaps on allowlisted pools and assert the hook cannot be bypassed by routing through an intermediate public contract." [6](#0-5) 

## Likelihood Explanation
Allowlisting the router is the expected operational pattern for a curated pool that still wants to support standard periphery UX. The bypass requires no special privileges, no malicious setup, and no non-standard tokens — only calling `exactInputSingle` (or any other router function) from a disallowed EOA. The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` since all call `pool.swap` with the router as `msg.sender`. [7](#0-6) 

## Recommendation
Pass the originating caller through the call chain rather than `msg.sender`. One approach: add a `swapper` parameter to `IMetricOmmPoolActions.swap` that the pool passes to extensions, and have the router supply `msg.sender` explicitly. Alternatively, `SwapAllowlistExtension.beforeSwap` should reject calls where `sender` is a known router/contract unless the pool admin has explicitly opted into router-mediated allowlist semantics. [8](#0-7) 

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension configured
// 2. allowedSwapper[pool][router] = true   (router is allowlisted — expected operational config)
// 3. allowedSwapper[pool][eoa]    = false  (EOA is NOT allowlisted)
// 4. vm.prank(eoa);
//    router.exactInputSingle(ExactInputSingleParams({ pool: pool, ... }));
// 5. Assert: swap succeeds — eoa bypassed the allowlist via the router
//    (pool.swap sees msg.sender = router; extension checks allowedSwapper[pool][router] = true)
``` [9](#0-8)

        "",

    amountOut = MetricOmmSwapInputs.int128ToUint128(out);

  }
