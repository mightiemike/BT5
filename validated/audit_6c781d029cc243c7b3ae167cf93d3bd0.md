### Title
Liquidator Can Liquidate Disabled Perp Products Without Active-Status Validation — (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_assertLiquidationAmount()` in `ClearinghouseLiq.sol` validates spot product activity via `longWeightInitialX18 != 0` before allowing liquidation, but applies no equivalent check for perp products. A liquidator can liquidate a perp position in a product that has been disabled (weights zeroed via `updateRisk()`), extracting value from the liquidatee without improving their health.

---

### Finding Description

In `_assertLiquidationAmount()`, when the product being liquidated is a **spot** product or spread, the function explicitly checks:

```solidity
require(
    spotEngine.getRisk(spotId).longWeightInitialX18 != 0,
    ERR_INVALID_PRODUCT
);
``` [1](#0-0) 

This prevents liquidation of disabled spot products (those whose weights were zeroed via `updateRisk()`).

However, for **perp** products (the `else` branch), no equivalent check exists. The only validation is:

```solidity
address engine = txn.isEncodedSpread
    ? address(0)
    : address(productToEngine[txn.productId]);
require(
    txn.isEncodedSpread || engine != address(0),
    ERR_INVALID_LIQUIDATION_PARAMS
);
``` [2](#0-1) 

The perp-only branch then proceeds directly to:

```solidity
} else {
    require(
        txn.amount % perpSizeIncrement == 0,
        ERR_INVALID_LIQUIDATION_AMOUNT
    );
    maxLiquidatable = perpAmount;
}
``` [3](#0-2) 

**Root cause — desynchronization between "registered" and "active" states:**

`productToEngine[productId]` is set permanently when a product is registered via `registerProduct()` and is **never cleared**: [4](#0-3) 

A perp product can be disabled by calling `updateRisk()` with zero weights: [5](#0-4) 

After `updateRisk()` zeros the weights:
- The product contributes **0** to health calculations (zero weight × any amount = 0)
- But `productToEngine[productId]` remains non-zero
- The `engine != address(0)` check in `_assertLiquidationAmount()` still passes
- No weight check exists for perp products — liquidation proceeds

This is the exact structural analog to the reported bug: `isRegisteredCurrency()` only checking `tokenAddresses[_ccy] != address(0)` rather than whether the currency is still accepted as collateral.

---

### Impact Explanation

A liquidator can call `liquidateSubaccountImpl()` targeting a disabled perp product:

1. The liquidatee's health is calculated **without** the disabled perp position (zero weights), so the position does not contribute positively to health.
2. The liquidation proceeds at the stored price (which may be stale if the sequencer stopped updating prices for the disabled product).
3. The liquidatee loses their perp position **without any health improvement** — their health is unchanged after the liquidation.
4. The liquidatee remains under maintenance health and can be liquidated again for their actual problematic positions.
5. The liquidator acquires the perp position at a discount, extracting value from the liquidatee.

The post-liquidation health checks at lines 573–576 do not prevent this: [6](#0-5) 

Since the disabled perp contributes 0 to health, liquidating it does not push the liquidatee above initial health, so `require(!isAboveInitial(txn.liquidatee))` passes. The liquidator's health check also passes because the acquired disabled perp position contributes 0 to their health.

---

### Likelihood Explanation

**Medium.** The scenario requires:
1. A perp product to be disabled via `updateRisk()` (an admin action, but a realistic operational event — e.g., delisting a market).
2. Users to still hold positions in the disabled product at the time of disabling.
3. Those users to be under maintenance health (which can be caused directly by the disabling, since the positive health contribution from the long perp position drops to 0).

Condition 3 is self-reinforcing: disabling a product with non-zero weights can itself push users holding long positions in that product below maintenance health, creating the liquidation opportunity immediately.

---

### Recommendation

Add a weight check for perp products in `_assertLiquidationAmount()`, mirroring the existing check for spot products:

```solidity
} else {
    require(
        perpEngine.getRisk(perpId).longWeightInitialX18 != 0,
        ERR_INVALID_PRODUCT
    );
    require(
        txn.amount % perpSizeIncrement == 0,
        ERR_INVALID_LIQUIDATION_AMOUNT
    );
    maxLiquidatable = perpAmount;
}
```

---

### Proof of Concept

1. Perp product X is active with non-zero weights; User A holds a long position contributing positively to their health.
2. Admin calls `updateRisk(perpProductX, RiskStore{longWeightInitial: 0, shortWeightInitial: 0, longWeightMaintenance: 0, shortWeightMaintenance: 0, priceX18: lastPrice})`.
3. User A's health drops to below maintenance because the positive perp contribution is now 0.
4. Liquidator B submits a `LiquidateSubaccount` transaction with `productId = perpProductX`.
5. `liquidateSubaccountImpl()` passes `isUnderMaintenance(txn.liquidatee)` → true.
6. `_assertLiquidationAmount()` evaluates `engine = address(productToEngine[perpProductX])` → non-zero → `require(engine != address(0))` passes.
7. The perp `else` branch executes with no weight check; `maxLiquidatable = perpAmount`.
8. `_handleLiquidationPayment()` calls `getLiqPriceX18(perpProductX, amount)` using the stored (potentially stale) price; Liquidator B acquires the position at a discount.
9. User A's health is unchanged (disabled perp still contributes 0); User A remains under maintenance health.
10. Liquidator B has extracted value from User A's disabled perp position without improving User A's solvency.
