### Title
Users Cannot Provide Slippage Protection for NLP Mint/Burn Oracle Price — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

When a user mints or burns NLP tokens, the `oraclePriceX18` that determines the exchange rate is supplied by the sequencer at execution time and is **not covered by the user's EIP-712 signature**. The user signs only the inner `MintNlp`/`BurnNlp` struct (amount + nonce), with no way to specify a minimum NLP received (for mint) or minimum quote received (for burn). No on-chain bound check exists on `oraclePriceX18`, unlike the analogous `WithdrawCollateralV2` path which does enforce a fee cap.

---

### Finding Description

The user-signed structs for NLP operations contain only:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;   // how much quote to spend — signed
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;     // how many NLP to burn — signed
    uint64 nonce;
}
``` [1](#0-0) 

The outer `SignedMintNlp` / `SignedBurnNlp` structs carry `oraclePriceX18` and `nlpPoolRebalanceX18` as sequencer-appended fields that are **not part of the EIP-712 digest the user signs**:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;        // sequencer-provided, not signed
    int128[] nlpPoolRebalanceX18; // sequencer-provided, not signed
}
``` [2](#0-1) 

In `EndpointTx.sol`, after signature validation the sequencer-supplied price is used directly with no user-specified bound:

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,   // no lower-bound check
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [3](#0-2) 

The same pattern applies to `BurnNlp`: [4](#0-3) 

**Contrast with `WithdrawCollateralV2`**, where the sequencer-provided `feeX18` is also outside the signed struct but is explicitly bounded on-chain:

```solidity
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);   // user protected
``` [5](#0-4) 

No equivalent protection exists for `oraclePriceX18` in the mint/burn paths.

---

### Impact Explanation

- **MintNlp**: A user commits to spending `quoteAmount` of quote. The number of NLP tokens minted is determined by `oraclePriceX18`. If the sequencer supplies a depressed price, the user receives far fewer NLP tokens than the market rate warrants — a direct asset loss relative to fair value.
- **BurnNlp**: A user commits to burning `nlpAmount` NLP. The quote returned is determined by `oraclePriceX18`. If the sequencer supplies an inflated price (NLP overvalued), the user receives less quote than expected.

In both cases the user's signed commitment is irrevocable once the nonce is consumed, and the corrupted state delta is the quote/NLP balance in the `SpotEngine` / `PerpEngine` after `clearinghouse.mintNlp` / `clearinghouse.burnNlp` executes.

---

### Likelihood Explanation

Medium. The sequencer is a semi-trusted off-chain component. Even without malicious intent, oracle prices can shift materially between the moment a user signs and the moment the sequencer batches the transaction. Because the user has no on-chain recourse (no minimum-out parameter, no revert condition), any adverse price movement is silently absorbed. A compromised or front-running sequencer can exploit this systematically.

---

### Recommendation

1. Add `minNlpAmount` to `MintNlp` and `minQuoteAmount` to `BurnNlp` as user-signed fields.
2. After `clearinghouse.mintNlp` / `clearinghouse.burnNlp`, assert the received amount meets the user's minimum.
3. Alternatively, mirror the `WithdrawCollateralV2` pattern: keep `oraclePriceX18` sequencer-provided but add an on-chain check bounding it against a trusted on-chain oracle or a user-signed acceptable range.

---

### Proof of Concept

1. User signs `MintNlp { sender, quoteAmount: 10_000e18, nonce: N }` expecting ~100 NLP at a fair price of 100 USDC/NLP.
2. Sequencer batches the transaction but supplies `oraclePriceX18 = 1000e18` (10× inflated NLP price).
3. `clearinghouse.mintNlp` mints only ~10 NLP for the user's 10,000 USDC.
4. No on-chain check reverts the transaction; the user's nonce is consumed and the balance update is final.
5. The sequencer (or a colluding party) can capture the difference in value.

The entry path is fully unprivileged: any user calling `submitSlowModeTransaction` with a `MintNlp` or `BurnNlp` payload, or any user whose signed order is included in a sequencer batch via `submitTransactions`, is exposed. [6](#0-5)
