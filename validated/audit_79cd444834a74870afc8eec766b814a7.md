### Title
Missing Zero-Address Validation on `owner` in `addLiquidity` Permanently Locks LP Funds — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and mints LP position shares to it without ever checking `owner != address(0)`. Because `removeLiquidity` enforces `msg.sender == owner`, a position minted to `address(0)` can never be withdrawn. The deposited tokens enter the pool's bin accounting permanently, constituting a direct, irrecoverable loss of user principal.

---

### Finding Description

`addLiquidity` passes `owner` straight into `LiquidityLib.addLiquidity`, which records shares under the position key derived from `(owner, salt)` and updates `binTotals`: [1](#0-0) 

No zero-address guard exists on `owner` anywhere in this path. The extension hooks `_beforeAddLiquidity` / `_afterAddLiquidity` receive `owner` as a parameter but are optional and pool-specific; they do not substitute for a core invariant check.

`removeLiquidity`, by contrast, enforces:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

Because `msg.sender` can never equal `address(0)` in a live EVM call, any position whose `owner` is `address(0)` is permanently unremovable. The tokens deposited via the `addLiquidity` callback are credited to `binTotals` and remain in the pool forever, inaccessible to any party.

This is the direct structural analog of the seeded bug: just as minting ERC-20 tokens to `address(0)` increments `totalSupply` while making the tokens unspendable, minting LP shares to `address(0)` increments `binTotals` while making the underlying liquidity unwithdrawable.

---

### Impact Explanation

**Critical / High — direct, permanent loss of user principal.**

Any tokens deposited through `addLiquidity(address(0), ...)` are locked in the pool forever. The pool's internal accounting (`binTotals.scaledToken0` / `scaledToken1`) is incremented correctly, so the pool appears solvent, but the LP claim backing those tokens is held by an address that can never sign a transaction. The loss is total and irreversible.

---

### Likelihood Explanation

**Medium.** The function is permissionless — any EOA, router, or integration contract can call it. A router that forwards a user-supplied `owner` field without sanitising it, or a misconfigured integration that passes `address(0)` as a default, will silently destroy the deposit. No privileged access or malicious setup is required; a single honest mistake suffices.

---

### Recommendation

Add a zero-address guard at the top of `addLiquidity`, mirroring the ownership check already present in `removeLiquidity`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
+   require(owner != address(0), "addLiquidity: owner is zero address");
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
``` [1](#0-0) 

---

### Proof of Concept

1. Any caller invokes `pool.addLiquidity(address(0), salt, deltas, callbackData, extensionData)`.
2. `_beforeAddLiquidity` fires (no core guard stops it).
3. `LiquidityLib.addLiquidity` records shares under the key `keccak256(abi.encode(address(0), salt, binIdx))` and increases `binTotals.scaledToken0` / `scaledToken1`.
4. The pool's `addLiquidity` callback pulls tokens from the caller; tokens now reside in the pool.
5. Caller attempts `pool.removeLiquidity(address(0), salt, deltas, extensionData)`.
6. `msg.sender != address(0)` → `revert NotPositionOwner()`.
7. No other address can ever satisfy `msg.sender == address(0)`.
8. Deposited tokens are permanently locked; `binTotals` is inflated by the locked amount, diluting future LP withdrawals proportionally.
