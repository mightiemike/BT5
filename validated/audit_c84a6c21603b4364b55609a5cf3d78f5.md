### Title
Router-mediated swaps bypass per-user `SwapAllowlistExtension` gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool sees `msg.sender = router`, so the extension checks the router's allowlist entry, not the user's. Any user blocked individually can bypass the block by routing through an allowlisted router.

### Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it unchanged to every configured extension. `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][caller_of_pool_swap]`. When the call originates from `MetricOmmSimpleRouter.exactInputSingle` (or any other `exact*` entry point), the pool's `msg.sender` is the router contract, not the end-user. If the pool admin has allowlisted the router address to permit router-based swaps, the check becomes `allowedSwapper[pool][router] == true`, which passes unconditionally for every user of that router, including those explicitly blocked.

### Impact Explanation
The `SwapAllowlistExtension` is documented as gating `swap` **by swapper address, per pool**. Allowlisting the router to enable router-based trading simultaneously and silently removes per-user enforcement for all users of that router. A pool admin who intends to block specific addresses (e.g., sanctioned wallets, competitors, or during an incident response) cannot achieve that goal if the router is also allowlisted. The core access-control invariant of the extension is broken for the router path.

### Likelihood Explanation
Allowlisting the router is the natural, expected configuration for any pool that wants to support the standard periphery swap flow. The pool admin has no on-chain signal that doing so opens the gate to all router users. The bypass requires no special privileges — any EOA can call `MetricOmmSimpleRouter.exactInputSingle`.

### Recommendation
The extension should gate on the **economic initiator**, not the immediate `pool.swap()` caller. Two options:

1. **Pass the original user through the router**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` read and verify it. This requires a coordinated convention between router and extension.
2. **Check both router and user**: Extend the allowlist check so that when `sender` is a known router, the extension additionally verifies the user identity supplied in `extensionData`.
3. **Document the limitation clearly**: If the design intent is that allowlisting the router means "all router users may swap," document this explicitly so pool admins understand the trade-off.
