### Title
Single-Step `LinkSigner` Assignment Grants Immediate Signing Authority Without New-Signer Confirmation - (File: core/contracts/EndpointTx.sol)

---

### Summary

The `LinkSigner` transaction type in Nado assigns a new linked signer to a subaccount in a single atomic step. The new address immediately acquires full signing authority over the subaccount — including the ability to withdraw collateral, transfer quote, liquidate, and re-link to yet another signer — with no confirmation required from the new signer address. This is a direct analog to the `setWhiteHackGroupAddress` single-step setter described in the report.

---

### Finding Description

Both execution paths for `LinkSigner` perform an unconditional single-step write to `linkedSigners`:

**Fast path** (sequencer-submitted, `processTransactionImpl`):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

**Slow path** (user-submitted directly on-chain, `processSlowModeTransactionImpl`):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

In both cases, the new signer address is written directly into `linkedSigners` with no pending-state, no acceptance call, and no time delay. The new address gains authority the moment the transaction is processed.

A compounding issue: `validateSignedTx` is called with `allowLinkedSigner = true` on the fast path. This means the **current** linked signer — not just the subaccount owner — can submit a `LinkSigner` transaction to replace itself with any arbitrary address. [3](#0-2) 

The `linkedSigners` mapping is the authoritative source consulted by `getLinkedSigner`, which is used in `validateSignature` and `validateCompactSignature` for every signed transaction type. [4](#0-3) 

---

### Impact Explanation

The linked signer can sign every privileged transaction type for the subaccount, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drains all collateral to an arbitrary address
- `TransferQuote` — moves quote balance to any registered subaccount
- `LiquidateSubaccount` — initiates liquidation of other accounts
- `MintNlp` / `BurnNlp` — manipulates NLP pool positions
- Another `LinkSigner` — permanently replaces the signer again, locking out the original owner [5](#0-4) 

If a user accidentally provides a wrong signer address, or if the current linked signer is compromised, the new address acquires all of the above capabilities immediately and irrevocably (unless the original owner acts first with a corrective `LinkSigner`). Because the fast path also allows the current linked signer to re-link, a compromised signer can race to lock out the owner permanently.

---

### Likelihood Explanation

The slow-mode path (`submitSlowModeTransaction`) is directly callable by any subaccount owner without sequencer involvement. A user who mistypes or copy-pastes the wrong address when linking a signer triggers the vulnerability with no further preconditions. The fast path is equally reachable for any user who submits a signed `LinkSigner` transaction to the sequencer. No privileged access, no leaked keys, and no admin action is required to reach the vulnerable code path. [6](#0-5) 

---

### Recommendation

Implement a two-step procedure for `LinkSigner`:

1. **Step 1 — Propose**: The subaccount owner (or current linked signer) proposes a `pendingLinkedSigner[sender] = newSigner` without activating it.
2. **Step 2 — Accept**: The proposed new signer address must call a separate acceptance function, proving it controls the key before `linkedSigners[sender]` is updated.

This mirrors the `approve`/`transferFrom` pattern cited in the original report and ensures that a mistyped or attacker-controlled address can never silently acquire signing authority.

Additionally, restrict the fast-path `LinkSigner` so that `allowLinkedSigner = false` — only the subaccount owner's key should be permitted to nominate a new linked signer, preventing a compromised linked signer from self-perpetuating its access. [3](#0-2) 

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...` and wants to link her hot wallet `0xHotWallet`.
2. She submits a slow-mode `LinkSigner` transaction via `submitSlowModeTransaction` with `signer = 0xWrongAddress` (a typo).
3. After the slow-mode delay, `processSlowModeTransactionImpl` executes: `linkedSigners[0xAlice000...] = 0xWrongAddress`.
4. `0xWrongAddress` immediately satisfies the `recovered == linkedSigner` check in `Verifier.validateSignature`.
5. The controller of `0xWrongAddress` signs a `WithdrawCollateral` transaction for Alice's subaccount and submits it to the sequencer.
6. The sequencer processes it; Alice's entire collateral balance is withdrawn to an address the attacker controls.
7. The attacker then submits a second `LinkSigner` (signed by `0xWrongAddress`, allowed because `allowLinkedSigner = true`) pointing to a fresh attacker address, permanently locking Alice out. [2](#0-1) [7](#0-6)
