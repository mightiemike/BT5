### Title
Linked Signer Can Overwrite Its Own Entry to Hijack or Disrupt Subaccount Signing Authority - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type in `EndpointTx.sol` is processed in the sequencer path with `allowLinkedSigner = true`, meaning the currently registered linked signer can sign a `LinkSigner` transaction to replace the linked signer with any arbitrary address. Only the subaccount owner should be able to change the linked signer, but the current implementation grants this power to the linked signer itself.

---

### Finding Description

In `EndpointTx.sol`, the sequencer path processes `LinkSigner` transactions as follows:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` as the permitted alternate signer to `verifier.validateSignature`. This means the currently registered linked signer is a valid signer for this transaction type. [1](#0-0) [2](#0-1) 

Because the linked signer can produce a valid signature for a `LinkSigner` transaction, it can set `linkedSigners[subaccount]` to any address it chooses — including an attacker-controlled address or `address(0)`.

By contrast, the **slow-mode path** for the same transaction type correctly restricts this operation to the subaccount owner via `validateSender(txn.sender, sender)`, which verifies that `msg.sender` is the owner of the subaccount:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
    validateSender(txn.sender, sender);   // owner-only check
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [3](#0-2) 

The sequencer path lacks this owner-only guard, creating an asymmetry: the slow-mode path is safe, but the sequencer path is not.

The `linkedSigners` mapping is stored in `EndpointStorage.sol`:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [4](#0-3) 

---

### Impact Explanation

**Impact: Medium**

A malicious or compromised linked signer can:

1. **Hijack signing authority**: Replace itself with an attacker-controlled address. All future signed transactions for the subaccount (withdrawals, liquidations, transfers, NLP operations) will be validated against the attacker's key, not the owner's. The owner loses the ability to trade or manage the subaccount until they submit a slow-mode `LinkSigner` to reclaim control.

2. **Disrupt operations (DoS)**: Set the linked signer to `address(0)`, removing the linked signer entirely. Any pending signed transactions that relied on the linked signer's key will fail signature validation, disrupting legitimate in-flight operations.

The corrupted state is `linkedSigners[subaccount]` — a concrete, persistent on-chain mutation that affects all subsequent signature validations for the subaccount across every transaction type that uses `allowLinkedSigner = true`. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium**

The linked signer is a trusted entity designated by the subaccount owner, typically an automated trading bot or hot wallet. However:

- The linked signer is explicitly a **limited-trust** role: it is expected to have trading permissions only, not account management permissions.
- A compromised linked signer key (e.g., a hot wallet breach) immediately enables this attack with no further preconditions.
- The attack requires no admin access, no governance capture, and no sequencer compromise — only possession of the linked signer's private key.
- The attacker-controlled entry path is fully reachable: sign a `LinkSigner` transaction with the linked signer's key and submit it to the sequencer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in the sequencer path, so that only the subaccount owner's key can authorize a change to the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This aligns the sequencer path with the slow-mode path, which already enforces owner-only access via `validateSender`. [6](#0-5) 

---

### Proof of Concept

1. Subaccount owner (`owner`) registers `alice` as the linked signer for `subaccount = keccak256(abi.encodePacked(owner, "default"))` via a slow-mode `LinkSigner` transaction. `linkedSigners[subaccount] = alice`.

2. `alice` (the linked signer) constructs a `SignedLinkSigner` transaction:
   - `signedTx.tx.sender = subaccount`
   - `signedTx.tx.signer = bytes32(uint256(uint160(attacker)))` (attacker-controlled address)
   - `signedTx.tx.nonce = currentNonce`
   - `signedTx.signature` = signed by `alice`'s key

3. `alice` submits this to the sequencer. The sequencer calls `processTransactionImpl` with this transaction.

4. `validateSignedTx(..., true)` passes because `getLinkedSigner(subaccount)` returns `alice`, and the signature is valid for `alice`.

5. `linkedSigners[subaccount]` is overwritten with `attacker`.

6. All future signed transactions for `subaccount` — including `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount` — now accept signatures from `attacker`. The owner's key is no longer recognized as a valid signer unless they submit a slow-mode `LinkSigner` to reclaim control. [1](#0-0) [7](#0-6)
