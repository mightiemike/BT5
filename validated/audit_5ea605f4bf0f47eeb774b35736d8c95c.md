### Title
Linked Signer Can Overwrite Its Own Authorization on Any Subaccount Without Owner Consent — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` handler in `EndpointTx.processTransactionImpl` accepts a signature from the **current linked signer** to overwrite the `linkedSigners` mapping for the subaccount. Because `allowLinkedSigner=true` is passed to `validateSignedTx`, the linked signer — a delegated, lower-trust key — can unilaterally replace itself (or any address) as the authorized signer for the subaccount owner's account, without the subaccount owner ever signing or consenting. This is the on-chain analog of the clickjacking report: an attacker with a limited delegated role initiates a privileged state mutation on behalf of a victim without their knowledge.

---

### Finding Description

**Root cause — `EndpointTx.sol` lines 576–590:**

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` with `allowLinkedSigner=true` calls `validateSignature(sender, digest, sig, true)`, which resolves to:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` then accepts the recovered signer if it equals **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means the current linked signer can produce a valid `SignedLinkSigner` transaction setting `signedTx.tx.signer` to any address — including an attacker-controlled one — and the protocol will write it unconditionally to `linkedSigners[subaccount]`.

**Contrast with the slow-mode path** (lines 232–239), which correctly restricts this operation to the subaccount owner:

```solidity
validateSender(txn.sender, sender);   // msg.sender must equal subaccount owner address
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The slow-mode path enforces owner-only authorization. The fast path does not.

**Exploit flow:**

1. Victim (subaccount owner) links a trading bot or API key as their linked signer — a normal, expected protocol action.
2. Attacker controls the linked signer key (e.g., a compromised API key, a malicious service, or a colluding counterparty).
3. Attacker crafts a `SignedLinkSigner` transaction where `signedTx.tx.sender = victim_subaccount` and `signedTx.tx.signer = attacker_address`, signed with the current linked signer key.
4. Attacker submits this to the sequencer (or waits for it to be included in `submitTransactions`).
5. `linkedSigners[victim_subaccount]` is overwritten with `attacker_address`.
6. Attacker now signs `WithdrawCollateral` (fast path, `allowLinkedSigner=true`, lines 413–436) or `TransferQuote` (lines 593–614, `allowLinkedSigner=true`) transactions as the new linked signer, draining the subaccount.

---

### Impact Explanation

The corrupted state is `linkedSigners[subaccount]` — the mapping that grants signing authority over all fast-path operations for a subaccount. Once overwritten, the attacker's address is accepted as a valid signer for `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `MintNlp`, `BurnNlp`, and future `LinkSigner` transactions. The subaccount owner loses control of their account without any on-chain action on their part. Collateral can be fully drained. The owner cannot recover via the fast path because the linked signer slot is already attacker-controlled; recovery requires a slow-mode `LinkSigner` (3-day delay), during which the attacker can complete the drain.

**Impact: High** — direct, complete theft of all collateral in the subaccount.

---

### Likelihood Explanation

The attacker must already hold the current linked signer key for the target subaccount. This is a realistic precondition: linked signers are explicitly designed for API keys, trading bots, and third-party integrators. A malicious or compromised service provider, a leaked API key, or a colluding counterparty all satisfy this precondition without requiring any privileged protocol access. The sequencer is the only gating step, and it has no reason to reject a structurally valid `LinkSigner` transaction.

**Likelihood: Medium** — requires holding a linked signer key, which is a common, intended use case.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the fast path, matching the slow-mode path's owner-only enforcement:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that only the subaccount owner's key can authorize a change to the linked signer, regardless of whether the transaction arrives via the fast or slow path.

### Proof of Concept

**Preconditions:**
- `victim_subaccount` = `0xAlice000...000` (Alice's subaccount, registered)
- `linkedSigners[victim_subaccount]` = `bot_address` (Alice linked her trading bot)
- Attacker controls `bot_key` (the private key for `bot_address`)

**Step 1 — Attacker constructs the payload:**
```
txType = LinkSigner (byte prefix)
signedTx.tx.sender  = victim_subaccount
signedTx.tx.signer  = attacker_address   // attacker's new key
signedTx.tx.nonce   = nonces[alice_address]  // current nonce, readable on-chain
signedTx.signature  = ECDSA.sign(digest, bot_key)
```

**Step 2 — Sequencer includes the transaction in `submitTransactions`.**

**Step 3 — `processTransactionImpl` executes:**
- `validateSignedTx` passes: `recovered == bot_address == linkedSigner` ✓
- `linkedSigners[victim_subaccount] = attacker_address` is written.

**Step 4 — Attacker immediately submits `WithdrawCollateral`:**
```
signedTx.tx.sender  = victim_subaccount
signedTx.tx.amount  = full_balance
signedTx.signature  = ECDSA.sign(digest, attacker_key)
```
- `validateSignedTx(..., true)` passes: `recovered == attacker_address == linkedSigner` ✓
- Alice's collateral is withdrawn to the protocol's withdraw pool, claimable by the attacker.

**Corrupted state:** `linkedSigners[victim_subaccount]` overwritten without owner consent; full collateral drained.
