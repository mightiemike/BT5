### Title
Bonus Checkpoints Advanced Without Payment on Insufficient Pool During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`)

---

### Summary

`slashRedelegationPosition` swallows `ErrInsufficientBonusPool` from `processEventsAndClaimBonus`, but by the time that error is returned the position's in-memory bonus checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) have already been advanced and validator-event reference counts have already been decremented in the store. The position is then persisted with those advanced checkpoints. The bonus owed for the pre-slash period is permanently unclaimable.

---

### Finding Description

`processEventsAndClaimBonus` walks pending validator events, updating the in-memory position and writing to the store, **before** it checks whether the rewards pool can cover the computed bonus: [1](#0-0) 

Inside the loop, two irreversible side-effects occur before the pool check:

1. `pos.UpdateLastEventSeq(entry.Seq)` advances the in-memory checkpoint past each processed event.
2. `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` writes the decremented reference count to the store; if it reaches zero the event is garbage-collected.

After the loop, `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` further advance the in-memory position: [2](#0-1) 

Only then is the pool balance checked: [3](#0-2) 

When the pool is short, `ErrInsufficientBonusPool` is returned with `pos` already mutated and store writes already committed.

Back in `slashRedelegationPosition`, that error is explicitly swallowed: [4](#0-3) 

For a **partial** slash the function then calls `setPositionWithState` with the already-mutated `pos`: [5](#0-4) 

This persists the advanced checkpoints. The position will never replay those events again, and the bonus for the pre-slash bonded segment is permanently lost.

For a **full** slash `pos.ClearBonusCheckpoints()` is called before saving, so the checkpoint advancement is overwritten — but the store-side event reference count decrements are still committed, which can prematurely garbage-collect events that other positions have not yet processed. [6](#0-5) 

---

### Impact Explanation

The corrupted values are:
- `pos.LastEventSeq` — advanced past events the position never received payment for.
- `pos.LastBonusAccrual` — advanced to `blockTime`, closing the accrual window permanently.
- `pos.LastKnownBonded` — updated to the post-event bonded state.
- `ValidatorEvents[valAddr, seq].ReferenceCount` — decremented in the store even though the bonus was not disbursed; events may be deleted before other positions process them.

The position owner permanently loses the bonus rewards that accrued between `LastBonusAccrual` and the slash event. There is no retry path: the checkpoints have moved past the events, and the events themselves may have been garbage-collected.

---

### Likelihood Explanation

The trigger requires three concurrent conditions, all of which are normal protocol events:

1. A tier position holder has an active redelegation (reachable via `MsgTierRedelegate`).
2. The destination validator is slashed during the redelegation window (standard Cosmos SDK slashing).
3. The `tieredrewards` rewards pool balance is below the computed bonus at that moment.

Condition 3 is realistic: the pool is funded by governance and can be depleted or temporarily empty. Partial validator slashes (downtime) are far more common than full slashes, making the partial-slash code path the dominant case.

---

### Recommendation

Move all store writes and in-memory checkpoint mutations to **after** the pool balance check, or use a two-phase approach: compute the bonus amount first, verify pool sufficiency, and only then walk events and advance checkpoints. If the pool is insufficient and the forfeit is intentional, the checkpoints must **not** be advanced — the position should retain its current `LastEventSeq` and `LastBonusAccrual` so it can retry once the pool is replenished, consistent with the atomically-failing user-driven paths documented in the ADR.

---

### Proof of Concept

1. Alice creates a tier position delegated to validator V.
2. Alice calls `MsgTierRedelegate`; a redelegation entry is created and a `RedelegationMapping` is stored.
3. The rewards pool is drained (e.g., by many concurrent claims or simply not yet funded).
4. Validator V is slashed for downtime (partial slash, e.g. 1%).
5. The staking module fires `BeforeRedelegationSlashed` → `slashRedelegationPosition`.
6. `processEventsAndClaimBonus` walks the pending SLASH event, decrements its reference count in the store, advances `pos.LastEventSeq` and `pos.LastBonusAccrual` in memory, then returns `ErrInsufficientBonusPool`.
7. `slashRedelegationPosition` swallows the error and calls `setPositionWithState(ctx, pos, nil)` (partial-slash branch), persisting the advanced checkpoints.
8. The pool is later replenished. Alice calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` starts from the now-advanced `LastEventSeq` and finds no pending events for the pre-slash period. Zero bonus is paid. The bonus for the bonded segment before the slash is permanently lost.
