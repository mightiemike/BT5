Audit Report

## Title
`SwapAllowlistExtension::beforeSwap` Checks Router Address Instead of End-User, Breaking Per-User Allowlist Enforcement â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool sets to `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][routerAddress]` instead of `allowedSwapper[pool][userAddress]`. This makes per-user allowlist enforcement either completely broken (allowlisted users cannot swap via the router) or trivially bypassable (allowlisting the router grants every user access).

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- this is the router when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this directly to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool (correct key) and `sender` is the router address â€” not the end-user. Meanwhile `MetricOmmSimpleRouter.exactInputSingle` calls:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The real user's address (`msg.sender`) is stored only in transient callback context for payment purposes â€” it is never forwarded to the pool as `sender`. The pool's `msg.sender` is the router.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the explicit `owner` parameter (the actual LP), not `sender` (the direct caller):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

This asymmetry is the root cause. The `addLiquidity` interface carries a separate `owner` field; the `swap` interface has no analogous "real initiator" field, so the extension has no way to recover the true end-user from `sender` alone when a router is in the call path.

## Impact Explanation
Two fund-impacting outcomes:

**Broken core functionality (High):** A pool admin allowlists individual user addresses via `setAllowedToSwap(pool, userAddress, true)`. Those users call `router.exactInputSingle(...)`. The extension checks `allowedSwapper[pool][router]` â†’ `false` â†’ `NotAllowedToSwap` revert. Legitimate, allowlisted users are permanently blocked from the standard swap path. LP assets in the pool become inaccessible to the intended participants.

**Allowlist bypass (Critical):** A pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to enable router-based swaps. This single entry grants every address â€” including those the admin explicitly never allowlisted â€” the ability to swap through the router. Any attacker calls `router.exactInputSingle(...)`, the extension checks `allowedSwapper[pool][router]` â†’ `true`, and the swap proceeds. The per-user restriction is completely nullified; unauthorized parties can drain liquidity from a pool intended to be restricted.

Both outcomes meet Sherlock High/Critical thresholds: direct loss of user principal or complete failure of a core pool access-control mechanism.

## Likelihood Explanation
`SwapAllowlistExtension` is a production periphery contract. Any pool that configures it expecting per-user gating is affected on every router-mediated swap â€” no special attacker capability is required. `MetricOmmSimpleRouter` is the standard swap entry point for end-users, so the mismatch is triggered on every normal interaction. The broken-functionality path requires zero attacker action; the bypass path requires only a standard `exactInputSingle` call.

## Recommendation
The `swap` interface must carry the real initiator separately, analogous to `addLiquidity`'s `owner` parameter. Options in increasing robustness:

1. **Minimal fix â€” check `recipient`:** If `recipient` reliably identifies the economic beneficiary, replace `sender` with `recipient` in the allowlist check. This is only safe if the pool guarantees `recipient` is always the end-user.

2. **Router encodes real user in `extensionData`:** Have `MetricOmmSimpleRouter` ABI-encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. The pool's transient callback context (already storing the real payer) can be used to authenticate the binding.

3. **Protocol-level fix:** Add an explicit `initiator` field to the `swap` interface (analogous to `owner` in `addLiquidity`) so the pool can forward the real end-user to all extensions without relying on `msg.sender`.

```diff
- if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+ address realUser = _decodeInitiatorFromExtensionData(extensionData); // router encodes msg.sender
+ if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][realUser]) {
```

## Proof of Concept
**Broken functionality path:**
1. Deploy pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)`.
3. Alice calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient=alice, ...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` â†’ `false` â†’ `NotAllowedToSwap` revert.
6. Alice, despite being allowlisted, cannot use the standard swap path.

**Bypass path:**
1. Deploy pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to enable router swaps.
3. Attacker (never allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient=attacker, ...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` â†’ `true` â†’ swap proceeds.
6. Attacker successfully swaps in a pool intended to restrict them.
