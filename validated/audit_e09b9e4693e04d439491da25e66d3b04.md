### Title
Unconstrained `skip_pow_verification` Flag in `init` Permanently Disables PoW Validation for All Subsequent Block Submissions - (File: `contract/src/lib.rs`)

---

### Summary

The `init` function accepts a caller-supplied `skip_pow_verification: true` flag with no on-chain enforcement that it must be `false` for mainnet deployments. Once stored in contract state, this flag causes every subsequent `submit_blocks` call to bypass both difficulty-target validation and the PoW hash check entirely. Any account holding the `UnrestrictedSubmitBlocks` role can then insert arbitrary block headers into the canonical chain, corrupting the light-client state and causing `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` to return `true` for fabricated transactions.

---

### Finding Description

`InitArgs` exposes `skip_pow_verification` as an unconstrained boolean field. [1](#0-0) 

The `init` function stores this value verbatim into contract state with no guard against `true` on a mainnet deployment. [2](#0-1) 

The documentation comment on `init` only *recommends* `false` for production; it does not enforce it. [3](#0-2) 

Every call to `submit_blocks` reads this stored flag and passes it directly to `submit_block_header`. [4](#0-3) 

Inside `submit_block_header`, when `skip_pow_verification` is `true`, both `check_target()` (difficulty bits validation) and the PoW hash comparison are skipped entirely. [5](#0-4) 

`check_target` calls `check_pow`, which enforces correct `bits`, MTP, future-time, and version constraints — all of which are bypassed. [6](#0-5) 

The `init_genesis` helper always submits the seed blocks with `skip_pow = true` (hardcoded), which is intentional for the genesis window. The problem is that the *persistent* flag governs all post-init submissions. [7](#0-6) 

There is no setter to correct `skip_pow_verification` after deployment, and the `migrate` function carries the flag forward unchanged. [8](#0-7) 

---

### Impact Explanation

If the contract is deployed with `skip_pow_verification: true` (a configuration used routinely in tests), every trusted relayer with the `UnrestrictedSubmitBlocks` role can submit block headers with:

- Any `bits` value (no difficulty check)
- Any timestamp (no MTP or future-time check)
- Any hash (no PoW hash ≤ target check)

These headers are accepted into `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`, corrupting the canonical chain tip (`mainchain_tip_blockhash`). Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` receive `true` for transactions in fabricated blocks, breaking the core security guarantee of the light client — that only Bitcoin-valid blocks anchor transaction proofs.

---

### Likelihood Explanation

Medium. The deployer must supply `skip_pow_verification: true` during `init`. However:

1. Tests throughout the repository use `skip_pow_verification: true` as the default convenience configuration. [9](#0-8) 

2. The relayer's `init_contract` function passes `skip_pow_verification` directly from its config file with no validation. [10](#0-9) 

3. There is no on-chain guard preventing a test-mode configuration from reaching production. A deployer reusing a test config or misconfiguring the relayer's init parameters produces a permanently insecure contract with no recovery path short of redeployment.

---

### Recommendation

Add an on-chain assertion in `init` that rejects `skip_pow_verification: true` when `network == Network::Mainnet`:

```rust
require!(
    !args.skip_pow_verification || args.network != Network::Mainnet,
    "skip_pow_verification must be false for mainnet deployments"
);
```

Alternatively, remove `skip_pow_verification` from `InitArgs` entirely and enforce PoW verification unconditionally in production builds via a compile-time feature flag.

---

### Proof of Concept

1. Deploy the contract with:
   ```json
   { "skip_pow_verification": true, "network": "Mainnet", "genesis_block_height": 0, ... }
   ```
2. Grant `UnrestrictedSubmitBlocks` to a relayer account (standard post-deploy step).
3. The relayer calls `submit_blocks` with a fabricated header:
   ```json
   { "version": 4, "prev_block_hash": "<tip_hash>", "merkle_root": "<attacker_root>",
     "time": <any>, "bits": 1, "nonce": 0 }
   ```
4. `submit_block_header` is called with `skip_pow_verification = true`; neither `check_target` nor the hash ≤ target check runs. The header is stored as the new canonical tip.
5. A consumer calls `verify_transaction_inclusion_v2` with a crafted Merkle proof matching `<attacker_root>`. The call returns `true`, falsely certifying a non-existent Bitcoin transaction.

    pub(crate) fn check_pow(&self, block_header: &Header, prev_block_header: &ExtendedHeader) {
        let config = self.get_config();
        let expected_bits = get_next_work_required(&config, block_header, prev_block_header, self);

        require!(
            expected_bits == block_header.bits,
            "bad-diffbits: incorrect proof of work"
        );

        // Check timestamp against prev
        require!(
            block_header.time > get_median_time_past(prev_block_header.clone(), self),
            "time-too-old: block's timestamp is too early"
        );

        // Check timestamp
        let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap(); // Convert to seconds
        require!(
            block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
            "time-too-new: block timestamp too far in the future"
        );

        // Reject blocks with outdated version
        require!(
            block_header.version >= 4,
            "bad-version: block version must be at least 4"
        );
    }
