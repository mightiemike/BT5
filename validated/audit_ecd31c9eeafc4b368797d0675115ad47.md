### Title
`tx_index` Aliasing in `compute_root_from_merkle_proof` Allows Out-of-Range Index to Pass Inclusion Verification — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` uses only the low `k` bits of `transaction_position` (where `k` is the proof length) to determine left/right branching at each level. Any two indices that differ by a multiple of `2^k` produce an identical parity sequence and therefore an identical computed root. Because neither `verify_transaction_inclusion` nor `verify_transaction_inclusion_v2` bounds-checks `tx_index` against the actual tree size, an unprivileged NEAR caller can supply `tx_index = real_index + 2^proof_length` with the legitimate proof and receive `true`.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates exactly `proof_length` times:

```rust
for proof_hash in merkle_proof {
    if current_position % 2 == 0 { … } else { … }
    current_position /= 2;
}
``` [1](#0-0) 

At loop step `i`, the branching decision is `(current_position >> i) % 2`. For `transaction_position = n + 2^k` (where `k = proof_length`):

```
step i: current_position = (n + 2^k) >> i = (n >> i) + 2^(k-i)
parity = ((n >> i) + 2^(k-i)) % 2
       = (n >> i) % 2          // because 2^(k-i) is even for all i < k
```

The extra `2^k` term is always even throughout all `k` iterations, so the parity sequence — and therefore the entire hash computation — is identical to that of `n`. The loop terminates before the extra term becomes odd (which would only happen at step `k`, one step past the end).

`verify_transaction_inclusion` passes `tx_index` directly to this function with no upper-bound check:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

The contract stores only the Merkle root, not the transaction count, so it has no way to detect that the supplied index is out of range.

`verify_transaction_inclusion_v2` calls `verify_transaction_inclusion` after its coinbase check, so it inherits the same flaw: [3](#0-2) 

`tx_index` is typed `u64` in `ProofArgs` / `ProofArgsV2` with no constraint: [4](#0-3) 

---

### Impact Explanation

The function returns `true` for a `tx_index` that is out of range for the actual block, violating the invariant that `tx_index` uniquely identifies the transaction's position in the tree. The concrete downstream risk is replay-attack bypass: any protocol built on top of this contract that uses `(tx_id, tx_block_blockhash, tx_index)` as a unique key to prevent double-claiming (e.g., a BTC-backed token bridge that marks a deposit as spent by its index) can be bypassed. An attacker who already used the real index `n` can re-submit the same proof with `tx_index = n + 2^proof_length` and receive a second `true`, allowing the same Bitcoin deposit to be claimed twice.

---

### Likelihood Explanation

The attack requires only public information (a real transaction, its block hash, and its Merkle proof, all available from any Bitcoin node). No privileged role, leaked key, or social engineering is needed. The attacker calls a public, unprivileged NEAR view or change method. The mathematical property is deterministic and always holds.

---

### Recommendation

Add an upper-bound check on `tx_index` before calling `compute_root_from_merkle_proof`. The maximum valid index for a proof of length `k` is `2^k - 1`:

```rust
let proof_len = args.merkle_proof.len();
require!(
    (args.tx_index as usize) < (1usize << proof_len),
    "tx_index out of range for the given proof length"
);
```

Alternatively, enforce the bound inside `compute_root_from_merkle_proof` itself by returning an error (or panicking) when `transaction_position >= (1 << merkle_proof.len())`.

---

### Proof of Concept

```rust
#[test]
fn test_tx_index_aliasing() {
    // 4-tx tree → proof length = 2 → alias offset = 2^2 = 4
    let tx0 = H256([1u8; 32]);
    let tx1 = H256([2u8; 32]);
    let tx2 = H256([3u8; 32]);
    let tx3 = H256([4u8; 32]);

    let proof_for_index_0 = merkle_proof_calculator(
        vec![tx0.clone(), tx1.clone(), tx2.clone(), tx3.clone()], 0
    );
    // proof_for_index_0 has length 2, so alias offset = 4

    let root_real  = compute_root_from_merkle_proof(tx0.clone(), 0, &proof_for_index_0);
    let root_alias = compute_root_from_merkle_proof(tx0.clone(), 4, &proof_for_index_0);

    // Both return the same root → verify_transaction_inclusion returns true for tx_index=4
    assert_eq!(root_real, root_alias,
        "tx_index=4 aliases tx_index=0 for a proof of length 2");
}
```

`tx_index = 4` is out of range for a 4-transaction tree (valid indices: 0–3), yet `compute_root_from_merkle_proof` returns the correct Merkle root, causing `verify_transaction_inclusion` to return `true`.
