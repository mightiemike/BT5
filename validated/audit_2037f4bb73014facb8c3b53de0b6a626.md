### Title
Unprivileged Caller Can Trigger Full Mainchain GC, Bypassing the Trusted-Relayer State-Modification Restriction — (File: `contract/src/lib.rs`)

---

### Summary

`submit_blocks` is the only role-gated path for modifying mainchain state, restricted to trusted relayers via `#[trusted_relayer]`. However, `run_mainchain_gc`, which is called internally by `submit_blocks` and directly mutates the same mainchain state, is a public function callable by **any** unprivileged NEAR account when the contract is not paused. An attacker can call `run_mainchain_gc(u64::MAX)` to flush all GC-eligible mainchain blocks in a single transaction, permanently corrupting the SPV-verification surface and breaking reorg safety.

---

### Finding Description

`submit_blocks` carries two access-control attributes:

```rust
#[pause]
#[trusted_relayer]
pub fn submit_blocks(&mut self, ...) -> PromiseOrValue<()> {
    ...
    self.run_mainchain_gc(num_of_headers);   // line 181
    ...
}
``` [1](#0-0) 

The `#[trusted_relayer]` guard enforces that only registered trusted relayers — or accounts holding `Role::DAO` / `Role::UnrestrictedSubmitBlocks` — may submit headers and trigger GC. The `batch_size` passed internally is `num_of_headers`, a small, relayer-controlled value.

`run_mainchain_gc` is separately exposed as a public method:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [2](#0-1) 

The `#[pause(except(...))]` attribute only restricts the function when the contract **is paused**; it grants `UnrestrictedRunGC` holders the ability to call it even while paused. When the contract is **not paused** — the normal production state — the attribute imposes **no caller restriction at all**. Any NEAR account can call `run_mainchain_gc` with an arbitrary `batch_size`.

Inside the function, the only bound on how many blocks are removed is:

```rust
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [3](#0-2) 

Passing `batch_size = u64::MAX` sets `selected_amount_to_remove = total_amount_to_remove`, removing every GC-eligible block in one call. The function then deletes those entries from `mainchain_height_to_header`, `mainchain_header_to_height`, and `headers_pool`, and advances `mainchain_initial_blockhash` to the new oldest block:

```rust
self.mainchain_initial_blockhash = self
    .mainchain_height_to_header
    .get(&end_removal_height)
    ...;
``` [4](#0-3) 

This state change is **permanent** — there is no undo path.

---

### Impact Explanation

Two concrete invariants are broken:

1. **SPV proof verification is destroyed for pruned blocks.** `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the block in `mainchain_header_to_height` and panic with `"block does not belong to the current main chain"` if it is absent. Any downstream contract or user that submitted a proof request for a block that was aggressively GC'd will receive a hard failure.

2. **Reorg safety is broken.** The CLAUDE.md explicitly documents: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor."* An attacker who calls `run_mainchain_gc(u64::MAX)` immediately after a fork appears can ensure the common ancestor is gone before the relayer submits the competing chain, making the reorg permanently unprocessable. [5](#0-4) 

---

### Likelihood Explanation

The entry path requires no privileges, no staking, and no special role. Any NEAR account can call `run_mainchain_gc` with a single transaction at zero cost beyond gas. The contract is expected to run continuously in a non-paused state, so the window is always open. The attack is trivially repeatable after each new batch of blocks is submitted by the relayer.

---

### Recommendation

Add an explicit caller restriction to `run_mainchain_gc` that mirrors the restriction on `submit_blocks`. The simplest correct fix is to require the caller to hold `Role::UnrestrictedSubmitBlocks`, `Role::DAO`, or to be an active trusted relayer — the same set that gates `submit_blocks`. The `UnrestrictedRunGC` role should remain as the bypass for the paused-contract case, but the function must not be open to arbitrary callers in the unpaused state.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and a synced mainchain of 60 000 blocks (normal production state).
2. From any unprivileged NEAR account, call:
   ```
   run_mainchain_gc(u64::MAX)
   ```
3. The contract removes `60000 - 52704 = 7296` blocks in a single call, advancing `mainchain_initial_blockhash` by 7296 heights.
4. Any `verify_transaction_inclusion` call for a block in that pruned range now panics.
5. If a fork whose common ancestor falls within those 7296 blocks is submitted by the relayer, the reorg walk panics with `PrevBlockNotFound` and the fork can never be resolved.
