Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` Returning L1 Block Number on Arbitrum, Enabling Guard Bypass or Swap DoS — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension` tracks price velocity using `block.number` as the block counter. On Arbitrum, `block.number` returns the L1 block number (~12 s cadence) rather than the L2 block number (~0.25 s cadence). Because multiple L2 blocks share the same L1 block number, `blockDiff` is zero for all intra-L1-block swaps, allowing cumulative price drift of up to `N × maxChangePerBlockE18` across N L2 blocks within one L1 block while the guard never fires. Alternatively, if the admin calibrates for L2 cadence, the guard becomes ~5× over-restrictive across L1 block boundaries, causing legitimate swaps to revert.

## Finding Description

In `beforeSwap`, the extension records and compares block numbers:

```solidity
// PriceVelocityGuardExtension.sol L55-70
uint64 prevBlock = s.lastUpdateBlock;
s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);   // L58 — L1 block on Arbitrum

uint256 blockDiff = block.number - prevBlock; // L63 — always 0 within one L1 block

uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff); // L70
``` [1](#0-0) 

The same pattern appears in `setLastMidPrice`:

```solidity
s.lastUpdateBlock = uint64(block.number); // L32
``` [2](#0-1) 

On Arbitrum, one L1 block (~12 s) spans approximately 48 L2 blocks (~0.25 s each). For every swap within the same L1 block, `blockDiff = 0`, so `allowedSq = maxChange²`. The guard checks each swap's delta independently against `maxChange`, but `s.lastMidPriceX64` is updated after each swap. An attacker can therefore execute 48 consecutive L2-block swaps, each moving the price by exactly `maxChange` from the updated anchor, accumulating a total drift of `48 × maxChange` within 12 seconds while the guard never triggers.

**Mode B (DoS):** If the admin instead calibrates `maxChangePerBlockE18` for L2 cadence (small per-0.25 s value), then across an L1 block boundary `blockDiff = 1` (L1 blocks), but 48 L2 blocks have elapsed. The guard allows `maxChange × √2 ≈ 1.41 × maxChange` when the intended allowance is `maxChange × √49 = 7 × maxChange`, making the guard ~5× more restrictive than intended and causing legitimate swaps to revert with `PriceVelocityExceeded`.

The repository already ships L2-specific oracle contracts with explicit `FUTURE_TOLERANCE` handling for L2 sequencer clock skew, confirming L2 deployment intent: [3](#0-2) 

`PriceVelocityGuardExtension` has no corresponding L2 awareness. By contrast, `OracleValueStopLossExtension` already uses `block.timestamp` for its decay mechanism: [4](#0-3) 

## Impact Explanation

**Mode A (bypass):** The velocity guard — the sole on-chain defence against rapid oracle-price manipulation for pools using this extension — is rendered ineffective within any single L1 block. An attacker can accumulate `N × maxChangePerBlockE18` price drift across N L2 blocks within one L1 block, draining LP funds at a price far beyond the configured cap. This constitutes direct loss of LP principal and bad-price execution, both within the allowed impact gate.

**Mode B (DoS):** Every swap crossing an L1 block boundary with a normal oracle price move reverts. The pool's swap functionality is broken for any market with routine inter-block volatility, qualifying as unusable swap flows and loss of protocol fees.

## Likelihood Explanation

The repository explicitly targets L2 deployment (`PriceProviderL2.sol`, `ProtectedPriceProviderL2.sol`). Pool admins configuring `maxChangePerBlockE18` receive no in-code signal that `block.number` behaves differently on Arbitrum; the NatSpec only states "Caps how fast the provided price can move between blocks" with no L2 caveat. Misconfiguration in either direction is near-certain for any Arbitrum deployment. Mode A is exploitable by any unprivileged trader submitting multiple swaps within one L1 block; Mode B is triggered by normal market conditions.

## Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension` and restate the velocity formula in time units (seconds), consistent with how `OracleValueStopLossExtension` already tracks decay via `block.timestamp`. Concretely:

- Rename `lastUpdateBlock` → `lastUpdateTs`, store `block.timestamp`.
- Replace `blockDiff = block.number - prevBlock` with `timeDiff = block.timestamp - prevTs`.
- Rename `maxChangePerBlockE18` → `maxChangePerSecondE18` and update `allowedSq = maxChange² × (1 + timeDiff)` accordingly.

This is chain-agnostic and eliminates the L1/L2 block number discrepancy entirely.

## Proof of Concept

**Setup (Mode A — bypass on Arbitrum):**

1. Deploy `PriceVelocityGuardExtension` on Arbitrum; attach as `beforeSwap` extension.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e16)` — intending 1% cap per L1 block (~12 s).
3. Pool admin calls `setLastMidPrice(pool, initialMid)` — anchors `lastUpdateBlock` to L1 block N.

**Attack (within L1 block N, across 48 L2 blocks):**

4. Attacker submits Swap 1 in L2 block 1 (L1 block N): oracle mid moves +1%. `blockDiff = N - N = 0`. `allowedSq = (1e16)²`. `actualSq = (1e16)²`. Guard passes. `lastMidPriceX64` updated to new mid.
5. Attacker submits Swap 2 in L2 block 2 (still L1 block N): oracle mid moves another +1% from updated anchor. `blockDiff = 0` again. Guard passes again.
6. Repeat for all 48 L2 blocks. Total price drift: **+48%** within 12 seconds. Guard never fires.

**Expected:** Guard fires after first swap exceeds 1%/block cap.
**Actual:** Guard allows 48% cumulative drift within one L1 block because `blockDiff = 0` for all intra-L1-block L2 swaps and the anchor is updated after each passing swap.
