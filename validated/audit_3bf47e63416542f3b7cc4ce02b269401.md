### Title
P2P Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash Domain - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` transaction (with zero `l1_data_gas` and zero `l2_gas`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` includes the `l1_data_gas` element in the Poseidon hash only for `AllResources`, a transaction originally signed and hashed under the `AllResources` domain produces a **different hash** after a P2P protobuf round-trip. This is the sequencer analog of the Trader Joe bug: just as the LBPair address was silently exempted from `_cacheFees` (skipping debt initialization and allowing fee collection from zero), here a specific class of transactions is silently exempted from the three-resource hash domain and re-classified into the two-resource domain, breaking the hash invariant.

---

### Finding Description

**Root cause — protobuf converter silently changes the variant:**

In `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← silently zero when absent
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)                      // ← variant changed
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
```

When a peer sends a V3 transaction whose `l1_data_gas` field is absent (or explicitly zero) and whose `l2_gas` is zero, the converter produces `ValidResourceBounds::L1Gas` regardless of whether the original submitter signed it as `AllResources`.

**Hash domain divergence — `get_tip_resource_bounds_hash`:**

In `crates/starknet_api/src/transaction_hash.rs` lines 188–211:

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // ← l1_data_gas ABSENT
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← l1_data_gas PRESENT
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

The two variants produce structurally different Poseidon inputs:

| Variant | Hash input elements |
|---|---|
| `L1Gas(l1_gas)` | `[tip, concat(L1_GAS, l1_gas), concat(L2_GAS, 0)]` — **2 resource entries** |
| `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` | `[tip, concat(L1_GAS, l1_gas), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0)]` — **3 resource entries** |

Because `poseidon([a, b, c]) ≠ poseidon([a, b, c, 0])` in general, the two hashes are distinct.

**The conversion chain that exposes this:**

A V3 transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is stored internally as `InternalRpcDeclareTransactionV3` / `InternalRpcInvokeTransactionV3`, both of which hold `resource_bounds: AllResourceBounds` and always call `ValidResourceBounds::AllResources(self.resource_bounds)` in their `DeclareTransactionV3Trait` / hash implementations. The hash H₁ is computed over three resource entries.

When this committed transaction is serialized to protobuf for P2P sync via `From<ValidResourceBounds> for protobuf::ResourceBounds` (lines 471–489), the `L1Gas` arm emits `l1_data_gas: Some(ResourceBounds::default().into())` — a zero value. On the receiving node, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l1_data_gas.is_zero() && l2_gas.is_zero()` and reconstructs `L1Gas`. Any subsequent hash recomputation on the receiving node produces H₂ ≠ H₁.

---

### Impact Explanation

**Matching scope: "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result" and "RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

1. **Hash divergence between nodes.** A syncing node that recomputes the transaction hash from the deserialized `ValidResourceBounds::L1Gas` object will derive H₂ ≠ H₁. If this recomputed hash is used to populate block receipts or transaction indices, the syncing node's state diverges from the sequencer's canonical state.

2. **Wrong fee/resource-bounds semantics.** `ValidResourceBounds::L1Gas` returns `GasVectorComputationMode::NoL2Gas` from `get_gas_vector_computation_mode()`, while `AllResources` returns `GasVectorComputationMode::All`. Fee estimation, bouncer accounting, and resource-bound validation all branch on this mode. A syncing node serving `starknet_estimateFee` or `starknet_simulateTransactions` for such a transaction would apply the wrong computation mode, returning an authoritative-looking but incorrect fee estimate.

3. **Signature domain mismatch.** The transaction hash is the message the account contract signs. If the hash served by a syncing node differs from the hash the signer committed to, any downstream signature verification (e.g., in proof generation or re-execution) will fail or silently accept the wrong preimage.

---

### Likelihood Explanation

The trigger condition — a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` — is entirely valid and accepted by the gateway's stateless validator (the validator only requires at least one non-zero resource bound). Such transactions are common: any user who sets only `l1_gas` and leaves the other two bounds at zero satisfies the condition. The P2P sync path is exercised by every non-sequencer full node. No special privileges are required; any submitted transaction that meets the zero-bound condition will trigger the mismatch on every syncing peer.

---

### Recommendation

1. **Preserve the original variant across protobuf round-trips.** Add a discriminator field (e.g., a boolean `is_all_resources`) to the protobuf `ResourceBounds` message, or use a separate oneof, so the receiver can reconstruct the exact variant the sender used.

2. **Remove the silent downgrade.** The `unwrap_or_default()` fallback for `l1_data_gas` was introduced for 0.13.2 backward compatibility. Once that support is dropped (as the TODO comment acknowledges), the converter should require `l1_data_gas` to be present and should never silently produce `L1Gas` for a message that carries all three resource fields.

3. **Canonicalize the hash domain.** Alternatively, make `get_tip_resource_bounds_hash` always include the `l1_data_gas` element (even as zero) for all V3 transactions, eliminating the variant-dependent branching. This would make `L1Gas` and `AllResources(_, 0, 0)` hash-equivalent and remove the divergence.

### Proof of Concept

```
1. Submit InvokeV3 with AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }.
   Gateway computes:
     fee_hash = poseidon([tip, concat(L1_GAS,X), concat(L2_GAS,0), concat(L1_DATA_GAS,0)])
     H1 = poseidon([INVOKE, version=3, sender, fee_hash, paymaster_hash, chain_id, nonce, da_mode, calldata_hash])

2. Transaction is included in block B. Block B is propagated via P2P protobuf.

3. Syncing node deserializes the transaction:
     protobuf::ResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }
     → TryFrom: l1_data_gas.is_zero() && l2_gas.is_zero() → ValidResourceBounds::L1Gas(X)

4. Syncing node recomputes:
     fee_hash' = poseidon([tip, concat(L1_GAS,X), concat(L2_GAS,0)])   ← only 2 resource entries
     H2 = poseidon([INVOKE, version=3, sender, fee_hash', paymaster_hash, chain_id, nonce, da_mode, calldata_hash])

5. H1 ≠ H2.
   - starknet_getTransactionReceipt on the syncing node returns H2 ≠ H1 (wrong hash).
   - starknet_estimateFee uses GasVectorComputationMode::NoL2Gas instead of All (wrong fee).
   - Any proof or re-execution that verifies the signature against H2 fails or accepts the wrong preimage.

    let l1_resource_bounds = resource_bounds.get_l1_bounds();

        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

#[derive(Clone, Copy, Debug, Eq, PartialEq, Hash, Ord, PartialOrd)]
pub enum ValidResourceBounds {
    L1Gas(ResourceBounds), // Pre 0.13.3. Only L1 gas. L2 bounds are signed but never used.
    AllResources(AllResourceBounds),
}

impl From<AllResourceBounds> for ValidResourceBounds {
    fn from(value: AllResourceBounds) -> Self {
        Self::AllResources(value)
    }
}

impl ValidResourceBounds {
    pub fn get_l1_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(l1_bounds) => *l1_bounds,
            Self::AllResources(AllResourceBounds { l1_gas, .. }) => *l1_gas,
        }
    }

    pub fn get_l2_bounds(&self) -> ResourceBounds {
        match self {
            Self::L1Gas(_) => ResourceBounds::default(),
            Self::AllResources(AllResourceBounds { l2_gas, .. }) => *l2_gas,
        }
    }

    /// Returns the maximum possible fee that can be charged for the transaction.
    /// The computation is saturating, meaning that if the result is larger than the maximum
    /// possible fee, the maximum possible fee is returned.
    pub fn max_possible_fee(&self, tip: Tip) -> Fee {
        match self {
            ValidResourceBounds::L1Gas(l1_bounds) => {
                l1_bounds.max_amount.saturating_mul(l1_bounds.max_price_per_unit)
            }
            ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas,
            }) => l1_gas
                .max_amount
                .saturating_mul(l1_gas.max_price_per_unit)
                .saturating_add(
                    l2_gas
                        .max_amount
                        .saturating_mul(l2_gas.max_price_per_unit.saturating_add(tip.into())),
                )
                .saturating_add(
                    l1_data_gas.max_amount.saturating_mul(l1_data_gas.max_price_per_unit),
                ),
        }
    }

    pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
        match self {
            Self::AllResources(_) => GasVectorComputationMode::All,
            Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
        }
