### Title
SwapAllowlistExtension gates on the router address instead of the actual user when swaps are routed through MetricOmmSimpleRouter, enabling allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension sees the router as `sender` — not the actual end-user. This creates a wrong-actor binding: the allowlist gates on the router's address rather than the address of the user who initiated the trade.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` is:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct — used as the mapping key), and `sender` is the first argument forwarded by the pool from its own `msg.sender` at the time `swap` is called. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls the pool, the pool's `msg.sender` is the router:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [2](#0-1) 

The pool therefore calls `_beforeSwap(msg.sender /*= router*/, recipient, ...)`, and the extension evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][actualUser]`. [3](#0-2) 

The same wrong-actor binding applies to every multi-hop path in `exactInput` (intermediate hops use `address(this)` = router as recipient/sender) and to `exactOutput` recursive callbacks. [4](#0-3) 

---

### Impact Explanation

Two fund-impacting consequences follow directly:

**Path A — Allowlist bypass (High):** A pool admin who wants to support router-mediated swaps must allowlist the router address. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true`, so every user — including those the admin explicitly never allowlisted — can swap through the router and pass the guard. The curated pool's access control is fully defeated for all router-mediated volume.

**Path B — Broken core functionality (Medium):** A pool admin who allowlists only specific end-users (e.g., KYC'd addresses) will find that those users cannot use the router at all, because the extension sees the router as `sender` and the router is not in the allowlist. Allowlisted users are forced to call the pool directly, bypassing the standard periphery path; any user who cannot do so is effectively locked out of the pool despite being authorized.

Both paths satisfy the allowed impact gate: Path A is a direct allowlist bypass enabling unauthorized trading; Path B is broken core swap functionality for authorized users.

---

### Likelihood Explanation

The trigger is unprivileged: any user can call `MetricOmmSimpleRouter` without any special role. Path A requires the pool admin to have allowlisted the router (a natural action when the admin wants to support the standard periphery), and Path B is triggered by any allowlisted user simply using the router. Neither path requires malicious setup or non-standard tokens.

---

### Recommendation

The extension must gate on the **actual end-user**, not the intermediary. Two complementary fixes:

1. **Pass the originating user through the pool.** Add an explicit `swapper` parameter to the pool's `swap` call (separate from `recipient`), set by the router to `msg.sender` before forwarding to the pool, and have the pool pass that value as `sender` to the extension.

2. **Alternatively, check `tx.origin` as a fallback** only when `sender` is a known periphery contract — though `tx.origin` gating has its own risks and is generally discouraged.

The cleanest fix is option 1: the router stores the originating user in transient storage (analogous to how it already stores the payer via `_setNextCallbackContext`) and the pool reads it to populate the `sender` field forwarded to extensions. [5](#0-4)

### Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension configured as beforeSwap hook.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — a natural action to enable router-mediated swaps.
  3. Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  4. attacker (non-allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: attacker, ...})
  5. Router calls pool.swap(attacker, zeroForOne, amount, limit, "", extensionData)
     → pool's msg.sender = router
  6. Pool calls _beforeSwap(router /*sender*/, attacker /*recipient*/, ...)
  7. Extension evaluates:
       allowedSwapper[pool][router] == true  → guard passes
  8. Swap executes. attacker receives output tokens.

Expected: revert NotAllowedToSwap (attacker is not allowlisted)
Actual:   swap succeeds — allowlist fully bypassed via router indirection.
```

The symmetric broken-functionality PoC: allowlist `alice` but not the router, then call `router.exactInputSingle` as `alice` — the extension sees `sender = router`, finds it not allowlisted, and reverts `NotAllowedToSwap`, locking the authorized user out of the standard periphery path.
