### Title
Linked Signer Can Overwrite Owner's `linkedSigners` Mapping via `LinkSigner` Transaction - (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This permits the current linked signer — not just the subaccount owner — to sign and submit a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with an attacker-controlled address. The slow-mode path for the same transaction type correctly restricts this operation to the owner only, exposing an inconsistency that constitutes an unauthorized subaccount state mutation.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch at line 576 calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier, meaning the verifier accepts a signature from **either** the subaccount owner **or** the current linked signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

Because the signature check passes for the linked signer, the subsequent write `linkedSigners[signedTx.tx.sender] = ...` executes unconditionally, replacing the owner's linked signer with any address the linked signer specifies.

In contrast, the slow-mode path for the same `LinkSigner` transaction type uses `validateSender(txn.sender, sender)`, which requires `msg.sender` to equal `address(uint160(bytes20(txn.sender)))` — i.e., the actual subaccount owner address:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

`validateSender` enforces owner-only access:

```solidity
function validateSender(bytes32 txSender, address sender) internal view {
    require(
        address(uint160(bytes20(txSender))) == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [4](#0-3) 

The slow-mode path correctly restricts `LinkSigner` to the owner. The sequencer (fast) path does not. This is the root cause.

---

### Impact Explanation

A malicious or compromised linked signer can:

1. Sign a `LinkSigner` transaction specifying an attacker-controlled address as the new signer.
2. The sequencer processes it; `validateSignedTx` passes because the current linked signer's signature is accepted.
3. `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.
4. The attacker's address can now sign `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `MintNlp`, `BurnNlp`, and `LiquidateSubaccount` transactions on behalf of the victim.

The corrupted state is `linkedSigners[subaccount]` in `EndpointStorage`. The asset delta is full collateral theft from the victim's subaccount. The victim retains their owner key and can re-link a signer, but the window between the overwrite and detection allows complete fund drainage. [5](#0-4) 

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and hot wallets in Nado. Private key compromise of a hot wallet is a realistic and common threat. Once a linked signer key is compromised, the attacker can escalate to persistent access by overwriting the linked signer before the owner notices. The attack requires only a single valid linked-signer signature and sequencer inclusion — no admin access, no governance, no oracle manipulation.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`, consistent with the slow-mode path:

```solidity
// In processTransactionImpl, LinkSigner branch:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true    // allowLinkedSigner
+   false   // only the subaccount owner may change the linked signer
);
```

This ensures that only the subaccount owner's key can authorize a change to the linked signer, matching the invariant already enforced in the slow-mode path.

---

### Proof of Concept

1. Alice owns subaccount `alice_subaccount`. She links Bob's hot wallet as her linked signer for automated trading: `linkedSigners[alice_subaccount] = Bob`.
2. An attacker compromises Bob's private key.
3. The attacker constructs a `LinkSigner` transaction: `{sender: alice_subaccount, nonce: current_nonce, signer: attacker_address}`.
4. The attacker signs this transaction with Bob's key and submits it to the sequencer.
5. The sequencer calls `processTransactionImpl`. `validateSignedTx(..., true)` passes because Bob (the current linked signer) signed it.
6. `linkedSigners[alice_subaccount] = attacker_address` is written.
7. The attacker signs a `WithdrawCollateralV2` transaction using `attacker_address`'s key, specifying `sendTo: attacker_wallet`.
8. The sequencer processes it; `validateSignedTx(..., true)` passes because `attacker_address` is now the linked signer.
9. Alice's entire collateral is withdrawn to the attacker's wallet.

Alice's owner key is never used. The attack is complete before Alice can react.
