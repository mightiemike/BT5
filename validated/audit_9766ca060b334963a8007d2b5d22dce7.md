Audit Report

## Title
Atomic fee delivery in `collectFees` can be blocked by a blacklisted `adminFeeDestination`, locking protocol fees and bricking fee-update flows — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool.collectFees` performs four sequential `safeTransfer` calls with no isolation between them. If any single transfer reverts — for example because `adminFeeDestination` is USDC-blacklisted — the entire function reverts and no state is mutated. Because `setPoolAdminFees` and `setPoolProtocolFee` in the factory unconditionally call `collectFees` before writing new fee rates, a blocked `collectFees` also permanently freezes fee-rate changes for that pool until the pool admin rotates the destination.

## Finding Description

`MetricOmmPool.collectFees` (callable only by the FACTORY) performs four sequential token transfers inside a single `unchecked` block:

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // (1)
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // (2)
}
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // (3)
}
if (totalFee1ToProtocol > 0) {
    transferToken1(FACTORY, totalFee1ToProtocol);             // (4)
}
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
``` [1](#0-0) 

`transferToken0` and `transferToken1` both use `IERC20.safeTransfer`, which reverts on USDC/USDT blacklisted recipients: [2](#0-1) 

The state reset (`notionalFeeToken0Scaled = 0`) only executes after all four transfers succeed. A revert on transfer (1) or (2) leaves all fee counters unchanged and no fees distributed.

Three factory entry points all route through this same `collectFees` call as a mandatory first step:

- `collectPoolFees` (open to any caller): [3](#0-2) 

- `setPoolProtocolFee` (factory owner only), calls `collectFees` before writing new rates: [4](#0-3) 

- `setPoolAdminFees` (pool admin only), calls `collectFees` before writing new rates: [5](#0-4) 

The only recovery path is `setPoolAdminFeeDestination`, which does **not** call `collectFees` and can rotate the destination to a non-blacklisted address: [6](#0-5) 

However, this recovery requires a responsive and cooperative pool admin.

## Impact Explanation

- **Protocol fees locked**: Accrued spread and notional fees sit in the pool and cannot be transferred to the FACTORY or the admin destination. The fees are not permanently lost but are inaccessible for an indefinite period.
- **Fee-rate management bricked**: `setPoolAdminFees` and `setPoolProtocolFee` both revert, so neither the pool admin nor the factory owner can adjust fee rates for the affected pool until the blockage is resolved.
- **`collectPoolFees` DoS**: Any caller attempting to trigger fee collection is blocked.

This constitutes broken core pool functionality causing loss of access to protocol fees and unusable fee-management flows, meeting the Medium/High threshold under Sherlock contest rules.

## Likelihood Explanation

USDC and USDT both implement address blacklists and are explicitly in scope. `adminFeeDestination` is set by the pool admin and can be any address — an address valid at pool creation time may later be blacklisted due to regulatory action or a compromised key. No privileged escalation is required: the trigger is a standard USDC blacklist event combined with a non-zero `adminFeeDestination`. The blockage is recoverable only if the pool admin acts promptly and in good faith; an unresponsive or adversarial pool admin can sustain the DoS indefinitely.

## Recommendation

Isolate each transfer so that a failure in one leg does not block the others:

1. **Wrap each transfer in a try/catch** and emit an event on failure, allowing partial collection and retrying the failed leg later.
2. **Separate token and recipient selection**: add a `token` and `recipient` parameter so callers can collect one token to one recipient at a time, keeping independent accounting for each leg.
3. **Per-token counter reset**: separate `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` resets so a successful token1 collection can still zero its counter even if the token0 leg fails.

## Proof of Concept

1. Deploy pool with `token0 = USDC`, `token1 = WETH`.
2. Pool admin sets `adminFeeDestination` to address `A` (valid at the time).
3. USDC blacklists address `A`.
4. Swaps accrue spread and notional fees in the pool.
5. Anyone calls `MetricOmmPoolFactory.collectPoolFees(pool)`.
   - Factory calls `pool.collectFees(...)` with `adminFeeDestination = A`.
   - `transferToken0(A, totalFee0ToAdmin)` → USDC `safeTransfer` reverts (blacklisted).
   - Entire `collectFees` reverts; `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are unchanged.
6. Factory owner calls `setPoolProtocolFee(pool, newFee, newNotional)` to adjust rates.
   - Same `collectFees` call is made first → same revert.
   - Fee-rate update is blocked.
7. Fees continue to accumulate but cannot be distributed; fee-rate management is frozen until the pool admin calls `setPoolAdminFeeDestination(pool, nonBlacklistedAddress)`.
