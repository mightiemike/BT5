### Title
Deprecated `verify_transaction_inclusion` Lacks Coinbase Proof Validation, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function remains publicly callable on-chain and does not implement the coinbase Merkle proof validation that is required to prevent the 64-byte transaction Merkle proof forgery attack. An unprivileged NEAR caller can supply a crafted proof where `tx_id` is an internal Merkle tree node hash, causing the function to return `true` for a transaction that does not exist.

---

### Finding Description

`BtcLightClient` exposes two transaction-verification entry points:

- `verify_transaction_inclusion` (v1, deprecated since 0.5.0) — no coinbase proof check
- `verify_transaction_inclusion_v2` (current) — adds coinbase proof check, then delegates to v1

The v1 function is still a public, externally callable NEAR method. Its only structural guard is that the Merkle proof must be non-empty:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

After that it computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block Merkle root:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

The function never validates that `tx_id` is a leaf node (an actual transaction) rather than an internal Merkle tree node. Because internal nodes are exactly 64 bytes (two concatenated 32-byte hashes), they can be interpreted as valid 64-byte "transactions" under the Bitcoin Merkle tree construction. An attacker who knows the Merkle tree of any confirmed block can present an internal node hash as `tx_id`, supply a valid sibling path, and the root comparison will succeed.

The v2 function addresses this by first verifying a coinbase proof before calling v1:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

But v1 is still directly reachable and bypasses this guard entirely.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a state-changing action (releasing funds, minting tokens, authorizing a cross-chain operation) can be deceived into accepting a forged proof. The attacker proves "inclusion" of a non-existent transaction in a real, confirmed Bitcoin block, satisfying the confirmation-depth check and the Merkle root comparison, while the `tx_id` corresponds to no actual on-chain Bitcoin transaction.

The corrupted value is the boolean verification result returned to the caller: it is `true` when it must be `false`.

---

### Likelihood Explanation

Medium. The function is callable by any unprivileged NEAR account with no role restriction beyond the pausable flag (which is off by default). The 64-byte forgery technique is publicly documented and the required inputs — internal Merkle node hashes and sibling paths — are derivable from publicly available Bitcoin block data. The constraint is that a downstream consumer must be using the deprecated v1 path rather than v2.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or gate it with an access-control role so it cannot be called externally. If removal breaks existing integrations, apply the same coinbase proof validation that `verify_transaction_inclusion_v2` performs before delegating to the shared Merkle root comparison.

---

### Proof of Concept

1. Select any confirmed Bitcoin block `B` stored in the contract's main chain with at least two transactions `T0`, `T1`.
2. Compute the internal Merkle node `N = double_sha256(T0 || T1)` — this is the parent of `T0` and `T1` at depth 1.
3. Build a sibling path from `N`'s position upward to the Merkle root (all siblings are publicly known from the block).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash, not a real transaction)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `N` at depth 1)
   - `merkle_proof = [siblings from depth 1 to root]`
   - `confirmations = 1`
5. The function returns `true`, falsely asserting that `N` is a confirmed transaction in block `B`. [1](#0-0) [2](#0-1) [3](#0-2)

            .get(&self.mainchain_tip_blockhash)

            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
