### Title
Protobuf `ValidResourceBounds` Deserialization Produces Wrong Hash Variant for V3 Transactions with Zero L2/L1DataGas Bounds — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf deserialization of `ValidResourceBounds` uses a heuristic (zero-check on `l2_gas` and `l1_data_gas`) to decide between the `L1Gas` and `AllResources` variants. A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is hashed at the gateway using the `AllResources` variant (3 resource felts in the Poseidon preimage), but after protobuf round-trip in the block-sync path it is deserialized as `L1Gas` (2 resource felts). The two variants produce structurally different hash preimages, so the recomputed hash diverges from the committed hash.

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471–490):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),       // → zeros
    l1_data_gas: Some(ResourceBounds::default().into()), // → zeros
},
```

`L1Gas` serializes with `l2_gas = 0` and `l1_data_gas = 0`. An `AllResources` transaction where both happen to be zero serializes to **identical protobuf bytes**.

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, lines 431–435):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
} else {
    ValidResourceBounds::AllResources(...)
})
```

Any `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` is silently re-classified as `L1Gas` on the receiving side.

**Hash preimage divergence** (`get_tip_resource_bounds_hash`, lines 187–211):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all) => {
        vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]    // 3 felts total
    }
});
```

`L1Gas` hashes `[tip, L1_GAS_felt, L2_GAS_felt(0)]` (2 resource felts).  
`AllResources` with `l2_gas=0, l1_data_gas=0` hashes `[tip, L1_GAS_felt, L2_GAS_felt(0), L1_DATA_GAS_felt(0)]` (3 resource felts).  
These produce different Poseidon digests even when all numeric values are identical.

**Attacker-controlled path:**

1. Submit a V3 `InvokeV3` or `DeclareV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` via RPC. The gateway accepts it; `RpcTransaction` carries `AllResourceBounds` directly.
2. `convert_rpc_tx_to_internal` wraps it as `ValidResourceBounds::AllResources(tx.resource_bounds)` and calls `calculate_transaction_hash` → hash H₁ (3-felt preimage). This hash is stored in `InternalRpcTransaction.tx_hash`.
3. The transaction is included in a block and propagated via P2P block-sync as a `Transaction::InvokeV3` / `DeclareV3` whose `resource_bounds: ValidResourceBounds` field is serialized to protobuf.
4. The receiving node deserializes `resource_bounds` via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` → `L1Gas` variant.
5. Any code path that recomputes the hash (e.g., `validate_transaction_hash`, re-execution, RPC tracing) produces H₂ (2-felt preimage). H₁ ≠ H₂.

### Impact Explanation

- **Wrong transaction hash on syncing nodes**: Any node that recomputes the transaction hash after block-sync (e.g., `validate_transaction_hash` in the central-sync path) will compute H₂ ≠ H₁ and reject the block as invalid, causing a chain-split or DoS for syncing nodes.
- **Wrong gas vector computation mode**: `ValidResourceBounds::L1Gas` returns `GasVectorComputationMode::NoL2Gas` while `AllResources` returns `All`. A syncing node that re-executes the transaction uses the wrong mode, producing wrong fee/gas accounting and potentially wrong receipts.
- **Wrong RPC responses**: `starknet_getTransactionByHash` and simulation endpoints would return `resource_bounds` with the wrong variant semantics.

### Likelihood Explanation

A V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` is valid and accepted by the gateway (the stateless validator only requires at least one non-zero bound). Such transactions exist on mainnet (e.g., the block fixture at `crates/apollo_starknet_client/resources/reader/block_post_0_14_3.json` shows V3 transactions with `L1_GAS: 0x0` and `L1_DATA_GAS: 0x0`). Any user can trigger this condition without special privileges.

### Recommendation

Replace the heuristic zero-check with an explicit discriminant field in the protobuf schema (e.g., a `bounds_type` enum: `L1_GAS_ONLY` vs `ALL_RESOURCES`). Until the schema is updated, the deserializer should default to `AllResources` when all three fields are present in the protobuf message, reserving `L1Gas` only for messages where `l2_gas` is absent (matching the pre-0.13.3 wire format). The comment at line 426 (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) confirms this is a known transitional state that has not been resolved.

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
