### Title
Migration Abort via Unbonding Validator in `ForceFullExitWithDelegation` — (`x/tieredrewards/keeper/force_exit.go`, `x/tieredrewards/migrations/v2/migrate.go`)

### Summary

`Migrate1to2` calls `exitVestedAccountsPositions`, which calls `ForceFullExitWithDelegation` for every vesting-owned position. That function calls `transferDelegationFromPosition`, which unconditionally checks `validator.IsBonded()` and returns `ErrValidatorNotBonded` if the validator is not bonded. The error propagates without any recovery path, causing `Migrate1to2` to return a non-nil error. In Cosmos SDK, a module migration returning an error during `RunMigrations` causes the upgrade handler to panic, halting the chain.

### Finding Description

**Step 1 — Migration entry point.**
`Migrate1to2` delegates entirely to `v2.Migrate`: [1](#0-0) 

`v2.Migrate` calls `backfillDelegatorAddress` (safe) then `exitVestedAccountsPositions`: [2](#0-1) 

**Step 2 — Force-exit loop.**
`exitVestedAccountsPositions` iterates all positions, identifies vesting-owned ones, and calls `ForceFullExitWithDelegation` for each. Any returned error is immediately propagated: [3](#0-2) 

**Step 3 — Hard `IsBonded()` guard in `transferDelegationFromPosition`.**
`ForceFullExitWithDelegation` calls `transferDelegationFromPosition` and propagates its error: [4](#0-3) 

Inside `transferDelegationFromPosition`, the validator status is checked unconditionally: [5](#0-4) 

If the validator is unbonding or unbonded, `ErrValidatorNotBonded` is returned with no fallback path.

**Step 4 — Migration registration.**
The migration is registered via `cfg.RegisterMigration`; a non-nil return from `Migrate1to2` causes the Cosmos SDK upgrade handler to panic: [6](#0-5) 

### Impact Explanation

If any vesting-owned position exists whose validator is in unbonding or unbonded state at upgrade height, the entire v8 upgrade panics and the chain halts. No tieredrewards operations can resume because the module's consensus version is never incremented to 2.

### Likelihood Explanation

This is reachable through normal chain operation:
- Validators naturally transition to unbonding (jailing, voluntary exit, tombstoning).
- Vesting accounts can hold tiered rewards positions.
- An adversary can deliberately trigger it: create a vesting-owned position on a self-operated validator, then jail or voluntarily unbond that validator immediately before the upgrade block.

No governance, operator compromise, or privileged access is required beyond owning a vesting account and a validator.

### Recommendation

In `ForceFullExitWithDelegation` (or in `exitVestedAccountsPositions`), handle the non-bonded validator case explicitly. Options:

1. **Skip the transfer, leave delegation in place**: if `transferDelegationFromPosition` returns `ErrValidatorNotBonded`, log a warning and continue rather than aborting. The position can be cleaned up post-upgrade via a separate governance action.
2. **Initiate a standard unbonding**: instead of the instant transfer path, call `stakingKeeper.Undelegate` for the position's delegator address, which works regardless of validator status.
3. **Pre-flight check**: before calling `ForceFullExitWithDelegation`, check validator status and skip (with a log) if not bonded.
