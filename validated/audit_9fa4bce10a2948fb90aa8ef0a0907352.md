Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Sender Instead of Actual End User, Allowing Any User to Bypass Per-User Swap Gating via `MetricOmmSimpleRouter` ‚Äî (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` always passes `msg.sender` as the `sender` argument to `_beforeSwap`, so when `MetricOmmSimpleRouter` calls `pool.swap`, the extension receives the router's address ‚Äî not the end user's address. A pool admin who adds the router to the allowlist (required for any allowlisted user to use the router) inadvertently grants every user on the network the ability to bypass the per-user gate by routing through the router.

## Finding Description
**Step 1 ‚Äî Pool hardcodes `msg.sender` as `sender`.**
In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the first argument: [1](#0-0) 

**Step 2 ‚Äî `ExtensionCalling._beforeSwap` forwards that value unchanged.**
The `sender` value is passed directly into the extension call without modification: [2](#0-1) 

**Step 3 ‚Äî `SwapAllowlistExtension.beforeSwap` checks that forwarded `sender`.**
The extension checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whatever the pool passed ‚Äî the router's address when called via the router: [3](#0-2) 

**Step 4 ‚Äî `MetricOmmSimpleRouter` calls `pool.swap` directly, making itself `msg.sender`.**
In `exactInputSingle`, the router stores the real user in transient callback context for payment only, then calls `pool.swap` directly ‚Äî the real user's address is never forwarded to the pool as `sender`: [4](#0-3) 

The same pattern applies in `exactInput`: [5](#0-4) 

The real user's address is stored via `_setNextCallbackContext` solely for payment routing and is never surfaced to extensions. There is no mechanism for the extension to recover the actual end user's identity when the call originates from the router.

**The forced dilemma for pool admins:**

| Admin choice | Effect |
|---|---|
| Do **not** add router to allowlist | Allowlisted users cannot use the router at all |
| **Add router to allowlist** | Every user can bypass the per-user gate via the router |

## Impact Explanation
A pool configured with `SwapAllowlistExtension` for KYC/compliance gating or market-maker-only access loses all enforcement the moment the router is added to the allowlist. Any non-allowlisted user can call `exactInputSingle`, `exactInput`, or `exactOutput` on `MetricOmmSimpleRouter` targeting the restricted pool and execute swaps that the allowlist was designed to block. This constitutes broken core pool functionality ‚Äî the extension's access control decision (`allowedSwapper[pool][sender]`) is the exact wrong value (router address instead of end user address), allowing unauthorized parties to trade against LP assets in an explicitly restricted pool.

## Likelihood Explanation
The likelihood is high for any pool that deploys `SwapAllowlistExtension` and also wants to support the standard periphery router for its allowlisted users ‚Äî both conditions are the natural, expected use case. The admin has no way to discover the bypass from the extension's interface alone: `isAllowedToSwap(pool, router)` returning `true` looks correct from the admin's perspective. The attack requires no special privileges ‚Äî any unprivileged user with access to the router can exploit it.

## Recommendation
The extension must check the actual end user, not the intermediary. Two complementary approaches:

1. **In `MetricOmmPool.swap`**: Accept an explicit `sender` parameter (similar to how `addLiquidity` accepts an explicit `owner`) so the router can forward the real user's address. The pool should validate that `msg.sender` is a trusted router before accepting a caller-supplied `sender`.

2. **In `MetricOmmSimpleRouter`**: Encode the real user's address into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode it when `msg.sender` (the pool's caller, i.e., the extension's `msg.sender` is the pool, but the pool's caller is the router) is a known trusted router. The extension would need a registry of trusted routers to avoid spoofing.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  alice  ‚Üí setAllowedToSwap(pool, alice, true)
  router ‚Üí setAllowedToSwap(pool, router, true)   ‚Üê admin adds router to support alice's router swaps

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: pool, ..., recipient: bob})

Trace:
  router.exactInputSingle()
    ‚Üí _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, bob, tokenIn)  // bob stored for payment only
    ‚Üí pool.swap(recipient=bob, ...)          // msg.sender inside pool = router
      ‚Üí _beforeSwap(sender=router, ...)
        ‚Üí SwapAllowlistExtension.beforeSwap(sender=router, ...)
          ‚Üí allowedSwapper[pool][router] == true  ‚úì  (passes!)
      ‚Üí swap executes, bob receives output tokens

Result:
  bob swaps successfully on a pool that explicitly disallowed him.
  The allowlist check passed because it saw the router address, not bob.
```

A Foundry integration test can reproduce this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the `router`, then calling `router.exactInputSingle` from a `bob` address and asserting the swap succeeds (demonstrating the bypass).
