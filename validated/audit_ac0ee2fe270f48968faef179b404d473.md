### Title
Stale Mainchain State Used for Fork Difficulty Retarget Validation — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating a fork block that crosses a difficulty retarget boundary, the contract's `get_next_work_required()` functions call `get_header_by_height()` to retrieve the first block of the retarget period. This function reads exclusively from `mainchain_height_to_header` — the mainchain index — rather than walking the fork's actual ancestry. If the fork diverged from the mainchain before the retarget reference height, the contract uses the wrong block's timestamp to compute the expected difficulty, allowing fork blocks with an incorrect `bits` value to pass validation.

---

### Finding Description

**Root cause:**

`get_header_by_height()` in `contract/src/lib.rs` (lines 677–682) reads from `mainchain_height_to_header`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

All three retarget implementations call this function to retrieve the first block of the retarget window:

- `contract/src/litecoin.rs` line 88: `blocks_getter.get_header_by_height(first_block_height)`
- `contract/src/dogecoin.rs` lines 292–295: `blocks_getter.get_header_by_height(height_first)`
- `contract/src/bitcoin.rs` (analogous pattern)

The developers themselves flagged this in `contract/src/dogecoin.rs` line 291:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
```

**Exploit path:**

1. The mainchain has blocks at heights 0–N. A retarget boundary exists at height R (e.g., 2016 for Bitcoin/Litecoin, 240 for Dogecoin pre-145k). The retarget uses the block at height `R - interval` as the reference timestamp.
2. An attacker submits a fork that diverges from the mainchain at height `F < (R - interval)`. The fork's block at height `R - interval` is a different block than the mainchain's block at that height, with a different timestamp.
3. When the fork reaches height R (the retarget boundary), `get_next_work_required()` calls `get_header_by_height(R - interval)`, which returns the **mainchain** block at that height — not the fork's ancestor.
4. If the mainchain's reference block has an earlier timestamp than the fork's reference block, `actual_time_taken` is computed as larger, yielding an **easier** difficulty target than the protocol requires for the fork.
5. The attacker submits a fork block at height R with a `bits` field matching this easier (incorrect) target. The PoW check passes because the hash satisfies the easier target, and the difficulty adjustment check passes because the contract computed the same wrong target.
6. The block is stored. If the fork accumulates more `chain_work` than the mainchain, `reorg_chain()` is triggered, replacing the canonical chain with the attacker's fork.

**The period-transition analog:** The external report describes `rewardRate` not being zeroed when a new reward period starts — stale rate state from the previous period bleeds into the new one. Here, the stale state is the mainchain's height-indexed block at the retarget reference height. When the fork enters a new retarget period, the contract does not reset/re-derive the reference block from the fork's actual ancestry; it carries over the mainchain's reference block, corrupting the difficulty computation for the new period.

---

### Impact Explanation

A fork block with an incorrect (too easy) `bits` value passes both PoW verification and difficulty adjustment validation. If the fork accumulates sufficient `chain_work` (which is easier to achieve with a lower difficulty target), `reorg_chain()` promotes it to the mainchain. All subsequent calls to `verify_transaction_inclusion()` operate against this corrupted canonical chain, returning `true` for transactions in attacker-controlled blocks that do not represent the true Bitcoin/Litecoin/Dogecoin mainchain. Downstream NEAR contracts relying on SPV proofs (e.g., bridge contracts releasing funds on proof of deposit) would accept fraudulent proofs.

---

### Likelihood Explanation

For **Dogecoin pre-145k** (240-block retarget interval): the attacker needs a fork diverging before a 240-block boundary and reaching the next 240-block boundary — approximately 480 Dogecoin blocks. Dogecoin's low difficulty makes this feasible for a moderately resourced attacker.

For **Bitcoin/Litecoin** (2016-block interval): the attacker needs ~4032 blocks of fork mining, requiring significant hashrate. This is a higher bar but not impossible for a nation-state or well-funded actor targeting a bridge contract.

The TODO comment at `contract/src/dogecoin.rs` line 291 confirms the developers identified this as an open correctness question, indicating the issue is real and unresolved.

---

### Recommendation

Replace `get_header_by_height()` calls inside `get_next_work_required()` with a function that walks the chain ancestry via `get_prev_header()` starting from `prev_block_header`, traversing backward the required number of steps to reach the retarget reference block. This ensures the correct ancestor is used regardless of whether the block being validated is on the mainchain or a fork.

---

### Proof of Concept

**Setup (Bitcoin, 2016-block retarget):**

1. Contract is initialized with mainchain blocks 0–2015. Mainchain block at height 0 has `time = T0`.
2. Attacker builds a fork starting at height 0 (or any height before 2016). Fork block at height 0 has `time = T0 + Δ` where `Δ > 0` (later timestamp).
3. Attacker mines fork blocks 1–2015 with valid PoW (same `bits` as mainchain, since no retarget yet).
4. At fork height 2016 (retarget boundary), correct `actual_time_taken = time[2015] - (T0 + Δ)` → shorter → **harder** difficulty.
5. Contract computes `actual_time_taken = time[2015] - T0` (using mainchain block 0) → longer → **easier** difficulty.
6. Attacker sets `bits` in fork block 2016 to match the easier target. Block hash satisfies the easier target. Both checks pass.
7. Attacker continues mining the fork with the easier difficulty. Once fork `chain_work > mainchain chain_work`, `reorg_chain()` is triggered.
8. `verify_transaction_inclusion()` now operates against the attacker's fork. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
