### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` — No Claim/Withdrawal Mechanism for Accumulated `fees[productId]` - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` accumulates fast-withdrawal fees into `fees[productId]` on every call, but no function in `BaseWithdrawPool` or `WithdrawPool` ever reads from that mapping to transfer the collected fees to a designated recipient or treasury. The fees are permanently locked in the `WithdrawPool` contract.

---

### Finding Description

In `BaseWithdrawPool.submitFastWithdrawal`, a fee is computed and either deducted from the user's transfer amount or pulled from the caller via `safeTransferFrom`, then credited to the on-chain accounting mapping:

```solidity
// BaseWithdrawPool.sol line 111
fees[productId] += fee;
``` [1](#0-0) 

The `fees` mapping is declared as:

```solidity
// collected withdrawal fees in native token decimals
mapping(uint32 => int128) public fees;
``` [2](#0-1) 

Searching the entire `BaseWithdrawPool.sol` and `WithdrawPool.sol`, there is **no function** that reads `fees[productId]` and transfers the corresponding token balance to any recipient. The only token-outflow function available to the owner is `removeLiquidity`:

```solidity
function removeLiquidity(uint32 productId, uint128 amount, address sendTo)
    external onlyOwner {
    handleWithdrawTransfer(getToken(productId), sendTo, amount);
}
``` [3](#0-2) 

`removeLiquidity` does not reference `fees[productId]` at all — it accepts an arbitrary `amount` parameter, meaning:

1. The `fees` accounting is **never decremented**, making it a permanently growing dead variable.
2. There is no on-chain path to claim exactly the accumulated fee balance to a treasury or protocol-controlled address.
3. The actual ERC-20 tokens representing those fees sit in the `WithdrawPool` contract with no dedicated exit path.

`WithdrawPool.sol` adds no new functions beyond the `initialize` call: [4](#0-3) 

---

### Impact Explanation

Every `submitFastWithdrawal` call permanently locks a portion of ERC-20 tokens (the fast-withdrawal fee) inside the `WithdrawPool` contract. Over time, as fast withdrawals accumulate, the total locked fee balance grows without bound. The protocol has no on-chain mechanism to route these tokens to a treasury, sequencer, or any designated recipient. This is a direct asset-lock: real ERC-20 value enters the contract and cannot exit through any sanctioned fee-claim path.

---

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function callable by any liquidity provider servicing a fast withdrawal. Every successful call increments `fees[productId]`. The condition is triggered on every normal protocol usage of the fast-withdrawal feature, making this a certainty rather than a conditional edge case. [5](#0-4) 

---

### Recommendation

Add a `claimFees` function (restricted to owner or a designated fee recipient) that reads `fees[productId]`, resets the mapping to zero, and transfers the corresponding token amount to the recipient:

```solidity
function claimFees(uint32 productId, address recipient) external onlyOwner {
    int128 amount = fees[productId];
    require(amount > 0, "no fees");
    fees[productId] = 0;
    handleWithdrawTransfer(getToken(productId), recipient, uint128(amount));
}
```

This mirrors the fix applied in the referenced `early-purchase` program, which added a `withdraw_funds` instruction to drain collected payments to a designated address.

---

### Proof of Concept

1. Liquidity provider calls `submitFastWithdrawal(idx, transaction, signatures)` with a valid signed withdrawal for `productId = 1`, `transferAmount = 1000e6` USDC.
2. `fastWithdrawalFeeAmount` returns `fee = 5e6` (5 USDC).
3. Since `sendTo != msg.sender`, `safeTransferFrom(token, msg.sender, 5e6)` pulls 5 USDC from the caller into `WithdrawPool`.
4. `fees[1] += 5e6` — the accounting mapping is updated.
5. After 1000 such operations, `fees[1] = 5000e6` (5000 USDC locked).
6. No function in `BaseWithdrawPool` or `WithdrawPool` accepts `fees[1]` as input and transfers those 5000 USDC out. The owner can call `removeLiquidity(1, X, recipient)` with an arbitrary `X`, but this bypasses the `fees` accounting entirely and risks draining user liquidity rather than only the fee balance.
7. The 5000 USDC is permanently locked with no sanctioned claim path. [6](#0-5)

        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);

        }

        fees[productId] += fee;
