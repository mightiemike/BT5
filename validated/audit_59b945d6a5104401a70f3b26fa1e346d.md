### Title
`SwapAllowlistExtension` gates the router address instead of the end user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument — which is `msg.sender` of the `pool.swap()` call — against the per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, the pool's `swap()` is called by the router, so `sender = router`. If the pool admin allowlists the router (the natural step to enable normal router usage), every user in the world can bypass the individual-user allowlist by routing through the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every `beforeSwap` extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← this is the router when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks the allowlist keyed by `(msg.sender = pool, sender = router)`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap(...)`, the pool receives `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

This creates two mutually exclusive failure modes:

1. **Bypass**: If the pool admin allowlists the router (e.g., `setAllowedToSwap(pool, router, true)`) to permit normal router usage, every unpermissioned user can swap by routing through the router — the individual-user allowlist is completely defeated.
2. **DoS**: If the pool admin does not allowlist the router, every allowlisted user is silently blocked from using the router, even though they are individually permitted.

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing so. The extension's NatSpec states it "Gates `swap` by swapper address, per pool," but the actual gated identity is the intermediate contract, not the economic actor.

Note: `DepositAllowlistExtension.beforeAddLiquidity` has the same missing `onlyPool` override but gates by the `owner` (position owner) parameter rather than `sender`, so the deposit path does not share this bypass.

### Impact Explanation

Any user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter`. If the pool was deployed to restrict swaps to a specific set of actors (e.g., KYC-verified counterparties, institutional LPs, or whitelisted market makers), the restriction is nullified for all router-mediated swaps. This breaks the core access-control invariant the pool admin configured and can allow unauthorized parties to interact with pools that carry special pricing, fee, or compliance requirements.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Pool admins who deploy a `SwapAllowlistExtension` and also want users to be able to use the router will naturally allowlist the router, triggering the bypass. The router is a public, permissionless contract, so any user can exploit the bypass once the router is allowlisted.

### Recommendation

Pass the economically relevant actor — the end user — through the hook chain rather than the immediate `msg.sender`. One approach is to add a `swapper` field to the `extensionData` that the router populates with `msg.sender` before calling the pool, and have the extension read that field. A simpler alternative is to have the pool expose a `swap(address swapper, ...)` overload where the router explicitly passes the originating user, and the extension gates on that value. Either way, the allowlist must be keyed on the address that controls the economic decision, not the forwarding contract.

### Proof of Concept

```
Setup:
  pool  = MetricOmmPool with SwapAllowlistExtension
  alice = allowlisted user
  bob   = NOT allowlisted

Pool admin calls:
  ext.setAllowedToSwap(pool, alice,  true)   // alice allowed
  ext.setAllowedToSwap(pool, router, true)   // router allowed so alice can use it

Attack:
  vm.prank(bob);
  router.exactInputSingle(ExactInputSingleParams({
      pool:            address(pool),
      tokenIn:         token0,
      tokenOut:        token1,
      zeroForOne:      true,
      amountIn:        1_000,
      amountOutMinimum: 0,
      recipient:       bob,
      deadline:        block.timestamp + 1,
      priceLimitX64:   0,
      extensionData:   ""
  }));
  // pool.swap is called with msg.sender = router
  // extension checks allowedSwapper[pool][router] → true
  // bob's swap succeeds despite not being allowlisted
```

The check that should fire — `allowedSwapper[pool][bob]` — is never evaluated because `sender` is the router, not `bob`.
