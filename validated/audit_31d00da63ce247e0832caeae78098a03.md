### Title
Fork Difficulty Calculated Using Mainchain Ancestor Timestamps Instead of Fork-Chain Ancestors — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating a submitted fork block's proof-of-work difficulty, the contract fetches the boundary block's timestamp via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. If the fork diverges before the difficulty-adjustment boundary, the fork's actual ancestor at that height is a different block with a different timestamp. The difficulty calculation therefore uses stale, incorrect mainchain data — directly analogous to the original report's pattern of continuing to use corrupted/invalid state in a critical calculation.

---

### Finding Description

In `get_next_work_required` (Dogecoin, post-height 145,000 where `difficulty_adjustment_interval = 1`):

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always resolves through `mainchain_height_to_header`, which maps heights only for the canonical chain. Fork blocks are stored only in `headers_pool` via `store_fork_header` and are never registered in `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [3](#0-2) 

The same pattern exists in Bitcoin's `get_next_work_required`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [4](#0-3) 

The `check_pow` path is reached for every submitted block (fork or mainchain) when `skip_pow_verification = false`:

```rust
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

For Dogecoin post-height 145,000 (`difficulty_adjustment_interval = 1`), `height_first = prev_block_header.block_height - 1`. If the fork diverges at height H-3 or earlier, the mainchain block at H-2 is a **different block** from the fork's ancestor at H-2, with a potentially different timestamp.

The Digishield retarget formula is:

```
modulated_timespan = retarget_timespan + (time(prev) - time(first) - retarget_timespan) / 8
``` [6](#0-5) 

If the attacker's fork block at H-2 carries a timestamp **later** than the mainchain block at H-2, the incorrect `time(first)` (mainchain's earlier timestamp) inflates `modulated_timespan`, which lowers the computed difficulty target. The attacker then sets their fork block's `bits` to match this artificially lowered expected difficulty, mines it with less work, and — if the fork accumulates sufficient chainwork — triggers `reorg_chain`, corrupting the canonical chain accepted by all downstream `verify_transaction_inclusion` callers. [7](#0-6) 

---

### Likelihood Explanation

- The Dogecoin deployment is production-targeted and the 1-block adjustment interval means **every fork block beyond depth 2** triggers the wrong timestamp lookup.
- The entry point is `submit_blocks`, callable by any trusted relayer (an unprivileged NEAR account that has staked). The `trusted_relayer` macro gates submission but does not prevent adversarial header content.
- The developers themselves flagged this with a `TODO` comment at the exact line, confirming awareness of the desynchronization.
- No privileged key or social engineering is required — only the ability to submit a sequence of crafted fork headers. [8](#0-7) 

---

### Recommendation

Replace the `get_header_by_height` call (mainchain-only lookup) with an ancestor traversal that walks the fork's own `prev_block_hash` chain backward from `prev_block_header` until reaching `height_first`. This mirrors how Bitcoin Core resolves the retarget boundary block via `GetAncestor`. The fix must be applied in both `dogecoin.rs` and `bitcoin.rs`.

---

### Proof of Concept

1. Relayer submits a valid Dogecoin mainchain up to height H (post-145,000). Mainchain block at H-2 has timestamp `T_main`.
2. Attacker submits a fork block at height H-2 (diverging from the mainchain at H-3) with timestamp `T_fork > T_main`. This fork block's own difficulty is validated using `get_header_by_height(H-3)` — the mainchain block at H-3 — which may also be incorrect, but the attacker can tune timestamps to pass.
3. Attacker submits a fork block at height H-1 with timestamp `T1`.
4. When the contract validates the fork block at height H, it calls `get_next_work_required` → `get_header_by_height(H-2)` → returns the **mainchain** block at H-2 with timestamp `T_main` (not the fork's block with `T_fork`).
5. `modulated_timespan = T1 - T_main` (inflated, since `T_main < T_fork`), producing a lower `expected_bits` (easier target).
6. Attacker sets their fork block at H's `bits` to this lower value and mines it against the easier target.
7. If the fork's cumulative `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain with the attacker's fork.
8. All subsequent `verify_transaction_inclusion` calls now operate against the attacker-controlled canonical chain. [9](#0-8)
