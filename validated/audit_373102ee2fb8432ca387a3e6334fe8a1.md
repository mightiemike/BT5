### Title
`SwapAllowlistExtension` Checks Router Address Instead of End-User, Allowing Any User to Bypass Per-User Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the swap, `msg.sender` of `pool.swap()` is the **router contract**, not the end user. If the pool admin allowlists the router address (a necessary step to let any user reach the pool through the router), every unprivileged user can bypass the per-user allowlist entirely.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and every other router entry point) calls `pool.swap()` directly without forwarding the end user's address:

```solidity
// MetricOmmSimpleRouter.sol
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

`msg.sender` of `pool.swap()` is the **router address**. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

**Consequence:** The pool admin faces an impossible choice:
- Do **not** allowlist the router → no user can reach the pool through the router, breaking the intended periphery flow.
- **Allowlist the router** → every user, including those the allowlist was meant to exclude, can swap freely by routing through `MetricOmmSimpleRouter`.

The end-user identity is structurally absent from the field the extension inspects, exactly mirroring the external report's pattern where a critical field (`metaHash`) was omitted from the signed hash.

### Impact Explanation

Any user who is explicitly **not** on the allowlist can bypass the swap gate by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) on a pool whose admin has allowlisted the router address. The user executes a live swap against pool liquidity, draining LP assets at oracle-derived prices without the access restriction the pool admin configured. This is a direct breach of the pool's access-control invariant with fund-impacting consequences for LP holders who deposited under the assumption that only vetted counterparties could trade.

### Likelihood Explanation

The router is the canonical user-facing entry point documented and expected by the protocol. Any pool admin who wants allowlisted users to be able to use the standard periphery must allowlist the router. The bypass is therefore reachable in every realistic restricted-pool deployment that also supports router access. No special privileges, flash loans, or non-standard tokens are required; a single `exactInputSingle` call suffices.

### Recommendation

Pass the original end-user address through the swap call chain so extensions can gate on it. Two concrete approaches:

1. **Add a `payer` / `originator` field to `pool.swap()`** that the router populates with `msg.sender` before calling the pool. The pool forwards this field to `_beforeSwap` alongside the existing `sender`. Extensions can then check the originator instead of (or in addition to) the direct caller.

2. **Encode the end-user address inside `extensionData`** and have the `SwapAllowlistExtension` decode and verify it, combined with a router-level signature or transient-storage attestation so the field cannot be spoofed by an arbitrary caller.

Until one of these is implemented, document clearly that `SwapAllowlistExtension` gates the **direct caller of `pool.swap()`**, not the economic end user, and that allowlisting the router opens the pool to all users.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)       // alice is the only allowed swapper
  admin calls setAllowedToSwap(pool, router, true)      // router allowlisted so alice can use it

Attack (executed by bob, who is NOT allowlisted):
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      ...
  })
  → router calls pool.swap(recipient, ...)
  → pool calls _beforeSwap(msg.sender=router, ...)
  → SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  → swap executes successfully for bob
  → bob receives pool output tokens; LP assets drained by an unauthorized party
```
