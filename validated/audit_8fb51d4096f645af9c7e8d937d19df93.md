Audit Report

## Title
`SwapAllowlistExtension` gates on the router address instead of the originating user, allowing any caller to bypass per-user swap allowlists via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the pool's immediate `msg.sender`, which is the router contract when swaps are routed through `MetricOmmSimpleRouter`. When a pool admin allowlists the router address to support standard periphery swaps, the check `allowedSwapper[pool][router]` passes for every caller regardless of whether the originating user is individually allowlisted, completely defeating the per-user access control the extension is designed to enforce.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- this is the router when called via MetricOmmSimpleRouter
  recipient,
  ...
  extensionData
);
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly, making the pool's `msg.sender` the router contract, not the end user:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The `extensionData` bytes parameter is accepted but entirely ignored by the extension — there is no fallback path to decode an originating user address.

This creates an irreconcilable dilemma for the pool admin:

| Admin action | Effect |
|---|---|
| Allowlist individual users only | Allowlisted users **cannot** swap through the router (broken functionality) |
| Allowlist the router to support periphery path | **Every** user bypasses the allowlist through the router |

`DepositAllowlistExtension` is not affected because it gates on `owner` (the position recipient), which is passed explicitly by the caller and enforced separately by the pool.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd users, whitelisted market makers) is fully bypassed once the pool admin allowlists the router. Any unpermissioned user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps on the curated pool at oracle-derived prices without being individually allowlisted. This is a direct break of the access-control invariant the extension was deployed to enforce, with fund-impacting consequences for LPs who deposited under the assumption that only vetted counterparties could trade. This qualifies as an admin-boundary break by an unprivileged path and broken core pool functionality causing potential loss of LP value.

## Likelihood Explanation
Medium. The router is the canonical periphery swap path. A pool admin who wants to support standard tooling (wallets, aggregators, front-ends) while maintaining an allowlist will naturally attempt to allowlist the router address. The admin has no on-chain signal that doing so opens the allowlist to all users; `isAllowedToSwap(pool, router)` returns `true` with no indication of the bypass. The scenario requires a deliberate admin action, but it is the expected action for any operator who wants both access control and router compatibility.

## Recommendation
Pass the originating user's address through the swap path so the extension can gate on the actual economic actor:

1. **Router-side**: Add an optional `originSender` field to swap parameters; the router populates it with `msg.sender` and passes it encoded in `extensionData`. The extension decodes and checks it.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should decode an `originSender` from `extensionData` when present and fall back to `sender` only when no override is present, so router-mediated swaps carry the real user identity.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  pool admin calls setAllowedToSwap(pool, router, true)   // to support periphery path
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool: pool,
        recipient: attacker,
        zeroForOne: true,
        amountIn: X,
        ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient=attacker, ...)   // msg.sender = router
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes at oracle price
        → attacker receives token output

  Result: attacker swaps on a curated pool without being individually allowlisted.
