### Title
Unprivileged Caller Can Front-Run `wrapVaultAsset` by Calling `creditDepositV1` First, Crediting Raw Underlying Asset as Wrong Spot Product — (`core/contracts/ContractOwner.sol` / `core/contracts/DirectDepositV1.sol`)

---

### Summary

`ContractOwner.creditDepositV1` carries no access-control modifier, so any external account can invoke it at any time. When a DDA holds raw underlying-asset tokens (e.g. USDC) that are also registered as a spot product in `SpotEngine`, calling `creditDepositV1` before `wrapVaultAsset` causes `DirectDepositV1.creditDeposit()` to match the raw token to its own `productId` and deposit it directly — bypassing the vault-wrapping step entirely. The subaccount receives collateral credit for the unwrapped token under the wrong product, and the vault-share product receives nothing.

---

### Finding Description

**Root cause 1 — no access control on `creditDepositV1`:** [1](#0-0) 

`creditDepositV1` is declared `external` with no `onlyOwner` (or any other) guard. Any EOA or contract can call it for any `subaccount` at any time.

**Root cause 2 — `creditDeposit()` blindly iterates all spot products:** [2](#0-1) 

`creditDeposit()` calls `spotEngine.getProductIds()`, then for every product whose token the DDA holds a non-zero balance of, it immediately approves and calls `endpoint.depositCollateralWithReferral` with that `productId`. There is no check distinguishing a raw underlying asset from a vault-share token.

**Root cause 3 — `wrapVaultAsset` is a separate, unguarded step:** [3](#0-2) 

`wrapVaultAsset` is also `external` with no access control, but more importantly it is a completely separate transaction. The protocol relies on callers invoking it *before* `creditDepositV1`. Nothing in the code enforces this ordering.

**Attack sequence:**

1. USDC (productId=1) and vault-share token (productId=2, whose `asset()` == USDC) are both registered in `SpotEngine`.
2. A user sends USDC to the DDA (e.g. via direct transfer or the normal deposit flow).
3. The attacker calls `ContractOwner.creditDepositV1(subaccount)` before anyone calls `wrapVaultAsset`.
4. Inside `creditDeposit()`, the loop reaches productId=1 (USDC), finds a non-zero balance, and calls `endpoint.depositCollateralWithReferral(subaccount, 1, balance, "-1")`.
5. USDC is consumed and credited as productId=1. The DDA now holds zero USDC.
6. `wrapVaultAsset` can no longer wrap anything (balance is 0); vault-share product (productId=2) receives zero collateral.

---

### Impact Explanation

- The subaccount's collateral is recorded under productId=1 (raw USDC) instead of productId=2 (vault shares). Risk weights, margin calculations, and any product-specific logic that applies to vault-share collateral are bypassed.
- The vault-share product receives no deposit, breaking the protocol invariant that vault-wrapped collateral must be credited only as the vault-share product.
- The USDC is irrevocably consumed by the endpoint; `wrapVaultAsset` cannot recover it. The mis-credited state is permanent unless the protocol has an admin recovery path.
- If vault-share collateral carries different (e.g. higher) risk parameters than raw USDC, the subaccount may be under-collateralized relative to what the protocol intended, or may gain unintended collateral value.

---

### Likelihood Explanation

- Requires USDC (or any ERC4626 underlying) to be registered as a spot product alongside its vault-share wrapper — a realistic and common Nado deployment configuration.
- The attacker needs no special privileges, no signature, and no governance access. A single public `external` call suffices.
- The attack is trivially front-runnable: any mempool observer who sees a `wrapVaultAsset` pending transaction can insert `creditDepositV1` ahead of it.
- The DDA address is deterministic (CREATE2 with fixed salt), so the attacker can pre-compute it and monitor for incoming USDC balances.

---

### Recommendation

1. **Add `onlyOwner` to `creditDepositV1`** in `ContractOwner.sol`:

```solidity
function creditDepositV1(bytes32 subaccount) external onlyOwner {
```

2. **Atomically wrap-then-credit** inside `creditDepositV1`: if a vault-share product is registered whose `asset()` matches a token held by the DDA, perform the `wrapVaultAsset` logic inline before calling `creditDeposit()`.

3. **Guard `creditDeposit()` in `DirectDepositV1`** so only the owner (`ContractOwner`) can invoke it:

```solidity
function creditDeposit() external onlyOwner {
```

This prevents any external party from triggering the deposit loop out of sequence.

---

### Proof of Concept

```solidity
// 1. Deploy mock ERC4626 vault: asset() == USDC
// 2. Register USDC as productId=1, vault-share as productId=2 in SpotEngine
// 3. Send 1000e6 USDC to the DDA address

// Attacker (any EOA):
contractOwner.creditDepositV1(subaccount);   // no onlyOwner guard

// Assert wrong product credited:
assert(spotEngine.getBalance(1, subaccount).amount > 0);  // USDC credited — wrong
assert(spotEngine.getBalance(2, subaccount).amount == 0); // vault-share — nothing

// wrapVaultAsset now does nothing (DDA USDC balance == 0)
contractOwner.wrapVaultAsset(subaccount, 2);
assert(spotEngine.getBalance(2, subaccount).amount == 0); // still zero

        address payable directDepositV1 = directDepositV1Address[subaccount];
        if (directDepositV1 == address(0)) {
            directDepositV1 = createDirectDepositV1(subaccount);
        }

        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0));

        address assetTokenAddr = IERC4626Base(tokenAddr).asset();
        require(assetTokenAddr != address(0));

        uint256 assetBalance = IERC20Base(assetTokenAddr).balanceOf(
            directDepositV1
        );
        if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
            DirectDepositV1(directDepositV1).withdraw(
                IIERC20Base(assetTokenAddr)
            );
            IERC20Base assetToken = IERC20Base(assetTokenAddr);
            assetToken.approve(tokenAddr, 0);
            assetToken.approve(tokenAddr, assetBalance);
            IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
        }
    }

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
