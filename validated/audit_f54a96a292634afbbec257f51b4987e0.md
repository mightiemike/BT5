### Title
`nSubmissions` Incremented on Reverted Slow-Mode Transactions Breaks Fast-Withdrawal Repayment Alignment — (`core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The public `executeSlowModeTransaction()` function unconditionally increments `nSubmissions` even when the inner slow-mode transaction reverts (silently caught by try/catch on non-hardhat chains). Because `nSubmissions` is the sole `idx` passed to `withdrawCollateral` → `submitWithdrawal`, any unexpected increment permanently desynchronizes the idx used by the sequencer from the idx a fast-withdrawal provider committed to in `submitFastWithdrawal`. The result is that the fast-withdrawal provider is never repaid and the user receives a double payment.

---

### Finding Description

**`executeSlowModeTransaction()` increments `nSubmissions` unconditionally:** [1](#0-0) 

`_executeSlowModeTransaction` wraps the inner call in a try/catch that silently swallows all non-OOG reverts on production chains: [2](#0-1) 

There is no access control on `executeSlowModeTransaction()` — any address can call it.

**`nSubmissions` is the idx passed to `withdrawCollateral` in both `WithdrawCollateral` and `WithdrawCollateralV2` paths:** [3](#0-2) 

**`handleWithdrawTransfer` first transfers tokens to the pool, then calls `submitWithdrawal`:** [4](#0-3) 

**`submitWithdrawal` uses `markedIdxs[idx]` to decide whether to forward tokens to the user or hold them as repayment for the fast-withdrawal provider:** [5](#0-4) 

**`submitFastWithdrawal` marks `markedIdxs[idx] = true` and immediately pays the user:** [6](#0-5) 

**`validateSubmissionIdx` only checks that the sequencer's batch starts at the current `nSubmissions`** — it does not prevent `nSubmissions` from being shifted by the public function before the batch is submitted: [7](#0-6) 

---

### Impact Explanation

When `nSubmissions` is shifted by one via a reverted `executeSlowModeTransaction()` call:

- The fast-withdrawal provider committed to `markedIdxs[N] = true` and paid the user.
- The sequencer reads the new on-chain `nSubmissions = N+1` and submits its next batch with `idx = N+1`.
- The withdrawal is processed with `nSubmissions = N+1` → `submitWithdrawal(..., idx=N+1)`.
- `markedIdxs[N+1]` is `false` → tokens are forwarded to the user a second time.
- `markedIdxs[N]` is `true` but is never matched by the sequencer → the fast-withdrawal provider is permanently unrepaid.

**Net asset delta:**
- User: receives tokens twice (once from fast-withdrawal provider, once from the slow path).
- Fast-withdrawal provider: fronted tokens, never repaid — direct loss of the full withdrawal amount.
- Protocol/WithdrawPool: tokens transferred from Clearinghouse to WithdrawPool are forwarded to the user instead of being held as repayment.

---

### Likelihood Explanation

The attack requires:
1. Submitting a slow-mode transaction that will revert at execution time (e.g., a `WithdrawCollateral` where health check fails due to price movement, or any transaction with invalid state at execution time). No special privilege is needed — `submitSlowModeTransaction` is public.
2. Waiting the 3-day `SLOW_MODE_TX_DELAY`.
3. Calling `executeSlowModeTransaction()` at the right moment — specifically when a fast-withdrawal is pending with a known `idx`.

The 3-day delay is a meaningful barrier but not a blocker for a motivated attacker. The attack is fully permissionless, requires no admin access, and can be planned in advance. A griefing variant (no profit to attacker, but provider loses funds) is also possible without coordination with the user.

---

### Recommendation

1. **Do not increment `nSubmissions` when the inner slow-mode transaction reverts.** Track whether `processSlowModeTransaction` succeeded and only increment on success, or use a separate counter for slow-mode executions that is not used as the withdrawal idx.
2. **Alternatively, gate `executeSlowModeTransaction()` to the sequencer only**, removing the public execution path that bypasses sequencer control of `nSubmissions`.
3. **Or pass a separate, monotonically-increasing withdrawal-specific nonce** to `withdrawCollateral` instead of reusing `nSubmissions`, so that slow-mode execution failures cannot shift the withdrawal idx.

---

### Proof of Concept

```
State: nSubmissions = N

Step 1: Fast-withdrawal provider calls:
  WithdrawPool.submitFastWithdrawal(idx=N, withdrawalTx, signatures)
  → markedIdxs[N] = true
  → user receives tokens immediately from provider

Step 2: Attacker submits a reverting slow-mode tx (e.g., WithdrawCollateral
        with amount > balance, or any tx that will fail health check):
  Endpoint.submitSlowModeTransaction(revertingTx)
  → slowModeTxs[txCount] = revertingTx; txCount++

Step 3: After 3-day delay, attacker calls:
  Endpoint.executeSlowModeTransaction()
  → _executeSlowModeTransaction: txUpTo++
  → try this.processSlowModeTransaction(...) → REVERTS (caught silently)
  → nSubmissions += 1   ← nSubmissions is now N+1

Step 4: Sequencer reads nSubmissions = N+1, submits batch with idx=N+1:
  Endpoint.submitTransactionsChecked(idx=N+1, [withdrawalTx], ...)
  → validateSubmissionIdx: N+1 == N+1 ✓
  → processTransaction(withdrawalTx):
      EndpointTx: clearinghouse.withdrawCollateral(..., nSubmissions=N+1)
      Clearinghouse.handleWithdrawTransfer(token, user, amount, idx=N+1):
        token.safeTransfer(withdrawPool, amount)   ← tokens sent to pool
        withdrawPool.submitWithdrawal(token, user, amount, idx=N+1)
          markedIdxs[N+1] == false → user receives tokens AGAIN
  → nSubmissions += 1   ← nSubmissions = N+2

Result:
  - markedIdxs[N]   = true  (never matched by sequencer → provider unrepaid)
  - markedIdxs[N+1] = true  (matched, user paid twice)
  - Fast-withdrawal provider: lost `amount` tokens
  - User: received `amount` tokens twice
```

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
