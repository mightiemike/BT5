### Title
Phantom Duplicate-Leaf Index Bypass in Odd-Length Merkle Tree Allows Double-Inclusion Proof — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a phantom `tx_index=N` (which does not exist in the block) when a block has an odd number of transactions `N`, because `compute_root_from_merkle_proof` is index-position-only arithmetic with no upper-bound guard, and the contract stores no transaction count per block.

---

### Finding Description

Bitcoin's Merkle tree construction duplicates the last leaf when the transaction count is odd. For a 3-tx block `[T0, T1, T2]`, the padded leaf layer is `[T0, T1, T2, T2]`, producing root `hash(hash(T0,T1), hash(T2,T2))`.

`compute_root_from_merkle_proof` computes the root purely from `(tx_id, tx_index, proof_path)` using only parity of the index at each level: [1](#0-0) 

For `tx_id=T2` with proof `[T2, hash(T0,T1)]`:

- **Index 2 (real):** position 2 is even → `hash(T2, T2)`, then position 1 is odd → `hash(hash(T0,T1), hash(T2,T2))` = root ✓
- **Index 3 (phantom):** position 3 is odd → `hash(T2, T2)`, then position 1 is odd → `hash(hash(T0,T1), hash(T2,T2))` = root ✓

Both produce the same root. The contract has no knowledge of how many transactions are in a block — it only stores the block header (merkle root, bits, etc.): [2](#0-1) 

Neither `verify_transaction_inclusion_v2` nor `verify_transaction_inclusion` performs any bound check on `tx_index`: [3](#0-2) [4](#0-3) 

`ProofArgsV2.tx_index` is an unconstrained `u64`: [5](#0-4) 

The coinbase proof check in `verify_transaction_inclusion_v2` validates only that `coinbase_tx_id` at index 0 reaches the merkle root — it is entirely orthogonal to the `tx_index` of the target transaction and provides no protection against a phantom index: [6](#0-5) 

---

### Impact Explanation

Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` with `tx_id=T_last` and `tx_index=N` (one past the last real transaction) for any odd-transaction-count block, and the contract returns `true`. A downstream protocol that relies on this return value to authorize a payment, bridge withdrawal, or token mint can be triggered twice for the same Bitcoin transaction — once at the real index and once at the phantom index — enabling a double-spend or replay.

---

### Likelihood Explanation

The call is fully permissionless (`verify_transaction_inclusion_v2` has no `#[private]` or `#[trusted_relayer]` restriction). The attacker needs only the block hash and the standard Merkle proof for `T_last` (index `N-1`), which is public Bitcoin data. The same proof bytes work unchanged for the phantom index `N`. Any Bitcoin block with an odd transaction count (the majority of real blocks) is exploitable.

---

### Recommendation

The contract must validate that `tx_index < tx_count` before calling `compute_root_from_merkle_proof`. Since the contract does not store `tx_count`, one of the following is required:

1. **Store `tx_count` in `ExtendedHeader`** at block submission time and enforce `args.tx_index < header.tx_count` in both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2`.
2. **Require the caller to supply `tx_count`** as part of `ProofArgsV2`, and validate it against the Merkle proof depth: `tx_count` must satisfy `2^(proof.len()-1) < tx_count <= 2^proof.len()`, and `tx_index < tx_count`.

---

### Proof of Concept

For a 3-tx block `[T0, T1, T2]`:

```
merkle tree (padded): [T0, T1, T2, T2]
level 1:              [hash(T0,T1), hash(T2,T2)]
root:                 hash(hash(T0,T1), hash(T2,T2))

proof for T2 at index 2: [T2, hash(T0,T1)]
proof for T2 at index 3: [T2, hash(T0,T1)]   ← identical
```

Both calls below return `true`:

```rust
// Call 1: valid
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: T2,
    tx_index: 2,          // real last tx
    merkle_proof: [T2, hash(T0,T1)],
    coinbase_tx_id: T0,
    coinbase_merkle_proof: [T1, hash(T2,T2)],  // valid coinbase proof
    ...
})

// Call 2: phantom duplicate — contract returns true, tx_index=3 does not exist
verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: T2,
    tx_index: 3,          // phantom, N == 3 transactions
    merkle_proof: [T2, hash(T0,T1)],
    coinbase_tx_id: T0,
    coinbase_merkle_proof: [T1, hash(T2,T2)],
    ...
})
```

The arithmetic trace through `compute_root_from_merkle_proof` at index 3 is:

```
position=3 (odd)  → hash(proof[0]=T2, current=T2) = hash(T2,T2); position→1
position=1 (odd)  → hash(proof[1]=hash(T0,T1), current=hash(T2,T2)) = root ✓
``` [7](#0-6)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
