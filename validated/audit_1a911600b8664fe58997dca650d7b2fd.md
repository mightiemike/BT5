Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End User, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` validates the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender` — the router contract address when swaps are routed through `MetricOmmSimpleRouter`. Any pool admin who allowlists the router to support router-mediated swaps for permitted users simultaneously grants every address on-chain the ability to bypass the individual allowlist, because the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`. The `NotAllowedToSwap` guard is never triggered, and the bypass leaves no on-chain signal.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // router address when called via MetricOmmSimpleRouter
    recipient,
    ...
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension. [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that argument against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no originating-user forwarding:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData   // end-user address is NOT encoded here
  );
``` [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. A pool admin who wants allowlisted users to trade through the router must call `setAllowedToSwap(pool, router, true)`. The moment they do, the check returns `true` for every caller — allowlisted or not — because the router is a single shared contract with no mechanism to distinguish which end user initiated the call. [4](#0-3) 

## Impact Explanation
The swap allowlist is a first-class pool security feature. Its silent failure on the primary public swap path (the router) renders it completely ineffective for any pool that supports router access. LP principal is at risk: the allowlist may have been configured to exclude toxic-flow sources, front-runners, or non-KYC'd addresses. Bypassing it exposes LP positions to exactly the trading patterns the guard was meant to block, with no `NotAllowedToSwap` revert and no on-chain signal of the bypass. This constitutes broken core pool functionality causing potential loss of LP funds, meeting the contest's allowed impact gate.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who wants allowlisted users to use the router — the expected production configuration — must allowlist the router address. This is a routine, non-malicious configuration step. Once done, the bypass requires no privilege: a single `exactInputSingle` call from any address suffices. The condition is reachable by any unprivileged trader on any pool that has both `SwapAllowlistExtension` and router support enabled.

## Recommendation
The extension must check the originating end user, not the intermediate router. Two viable approaches:

1. **Caller forwarding via `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling `pool.swap`. The extension decodes and checks that address. This must be paired with a check that `sender` (the pool's `msg.sender`) is a factory-registered trusted router, to prevent an attacker from forging the originator field.

2. **Transient-storage originator:** The router writes the originating caller into a transient storage slot before calling the pool. The extension reads that slot. This is consistent with the protocol's existing EIP-1153 usage for callback context in `MetricOmmSwapRouterBase`. The same trust check (verify `sender` is a registered router) is required.

## Proof of Concept
```
Setup:
  pool admin deploys pool with SwapAllowlistExtension as beforeSwap hook
  pool admin calls setAllowedToSwap(pool, userA, true)   // allowlist userA
  pool admin calls setAllowedToSwap(pool, router, true)  // enable router-mediated swaps for userA

Attack (userB, not allowlisted):
  userB calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...) with msg.sender = router
    → pool calls _beforeSwap(sender=router, ...)
    → extension checks allowedSwapper[pool][router] → true
    → swap executes; userB receives output tokens

Result:
  userB traded on a curated pool despite not being individually allowlisted.
  NotAllowedToSwap was never triggered.
  LPs are exposed to the trading patterns the allowlist was meant to block.
```

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist `userA` and the router, then call `router.exactInputSingle` from `userB` (not allowlisted) and assert the swap succeeds and `userB` receives output tokens.
