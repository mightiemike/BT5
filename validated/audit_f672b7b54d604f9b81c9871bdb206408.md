### Title
Missing Slippage Protection in `mintNlp()` and `burnNlp()`: `oraclePriceX18` Is Not Committed to by User Signature — (`core/contracts/Clearinghouse.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` user-signed transaction structs do not include a minimum output amount or a price bound. The `oraclePriceX18` used to compute the NLP-to-quote conversion is supplied by the sequencer outside the user's signed payload and is never verified against any user-specified limit. A user minting NLP can receive far fewer tokens than expected; a user burning NLP can receive far less quote than expected, with no on-chain protection.

---

### Finding Description

In the Nado protocol, users interact with the NLP liquidity token through two signed transaction types: `MintNlp` and `BurnNlp`.

**`MintNlp` flow:**
The user signs a `MintNlp` struct containing only `{sender, quoteAmount, nonce}`. [1](#0-0) 

The sequencer wraps this in a `SignedMintNlp` struct, appending `oraclePriceX18` and `nlpPoolRebalanceX18` — both outside the user's signed payload: [2](#0-1) 

In `Verifier.sol`, the EIP-712 digest for `MintNlp` covers only `(MINT_NLP_SIGNATURE, sender, quoteAmount, nonce)` — `oraclePriceX18` is explicitly excluded: [3](#0-2) 

In `Clearinghouse.mintNlp()`, the NLP amount minted is computed as:
```
nlpAmount = quoteAmount / oraclePriceX18
``` [4](#0-3) 

Because `oraclePriceX18` is sequencer-supplied and not signed by the user, a higher price yields fewer NLP tokens for the same `quoteAmount` spent, with no floor check.

**`BurnNlp` flow:**
The user signs a `BurnNlp` struct containing only `{sender, nlpAmount, nonce}`. [5](#0-4) 

The `BurnNlp` digest also excludes `oraclePriceX18`: [6](#0-5) 

In `Clearinghouse.burnNlp()`, the quote returned is:
```
quoteAmount = nlpAmount * oraclePriceX18 - burnFee
``` [7](#0-6) 

A lower sequencer-supplied `oraclePriceX18` yields fewer quote tokens returned for the same `nlpAmount` burned, with no minimum quote check.

---

### Impact Explanation

**Impact: Medium.**

- **`mintNlp`**: A user spending `quoteAmount` of quote tokens receives `quoteAmount / oraclePriceX18` NLP tokens. If the sequencer uses a price significantly above the fair market price, the user receives materially fewer NLP tokens than expected — a direct asset loss relative to the user's intent.
- **`burnNlp`**: A user burning `nlpAmount` of NLP tokens receives `nlpAmount * oraclePriceX18 - burnFee` quote tokens. If the sequencer uses a price significantly below the fair market price, the user receives materially fewer quote tokens — a direct asset loss.

In both cases, the user's signed transaction commits only to the input amount and nonce. There is no on-chain mechanism for the user to specify a minimum acceptable output, so the execution price is entirely at the sequencer's discretion.

---

### Likelihood Explanation

**Likelihood: Medium.**

The sequencer controls `oraclePriceX18` in the submitted `SignedMintNlp` / `SignedBurnNlp` transaction. A malicious or compromised sequencer can set this value arbitrarily. Even without malice, a stale oracle price submitted during volatile market conditions can cause significant slippage. The user's signed message provides no protection against either scenario.

---

### Recommendation

Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp`, include these fields in the EIP-712 signed digest in `Verifier.sol`, and enforce them in `Clearinghouse.mintNlp()` and `Clearinghouse.burnNlp()`:

```solidity
// MintNlp
require(nlpAmount >= txn.minNlpAmount, "ERR_SLIPPAGE_MINT_NLP");

// BurnNlp
require(quoteAmount >= txn.minQuoteAmount, "ERR_SLIPPAGE_BURN_NLP");
```

This ensures the user's signed intent bounds the worst-case conversion rate they will accept.

---

### Proof of Concept

1. User signs `MintNlp{sender: alice, quoteAmount: 1000e18, nonce: 1}` intending to mint NLP at a fair price of `1.00` (expecting ~1000 NLP).
2. Sequencer submits `SignedMintNlp` with `oraclePriceX18 = 10e18` (10x the fair price).
3. `Clearinghouse.mintNlp()` computes `nlpAmount = 1000e18 / 10e18 = 100` NLP.
4. Alice's `quoteAmount` is fully debited, but she receives only 100 NLP instead of ~1000 NLP.
5. No on-chain check rejects this execution because `oraclePriceX18` is not part of Alice's signed digest and no minimum output is enforced. [8](#0-7) [9](#0-8)
