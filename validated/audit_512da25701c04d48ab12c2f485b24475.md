### Title
Sanctioned Users Can Bypass Compliance Controls via `submitFastWithdrawal` — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` performs **no sanction check** on either the caller (`msg.sender`) or the resolved `sendTo` recipient, while `Endpoint.depositCollateralWithReferral` correctly enforces `requireUnsanctioned` on both the depositor and the subaccount owner. This asymmetry allows a sanctioned user to receive withdrawn collateral through the fast-withdrawal path, bypassing the protocol's on-chain compliance controls.

---

### Finding Description

`Endpoint.depositCollateralWithReferral` enforces sanctions on both parties:

```solidity
// Endpoint.sol lines 133-135
// depositor / depositee need to be unsanctioned
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [1](#0-0) 

`requireUnsanctioned` is defined in `EndpointStorage`:

```solidity
// EndpointStorage.sol line 121-123
function requireUnsanctioned(address sender) internal view virtual {
    require(!sanctions.isSanctioned(sender), ERR_WALLET_SANCTIONED);
}
``` [2](#0-1) 

However, `BaseWithdrawPool.submitFastWithdrawal` — a `public` function callable by anyone — contains **no sanction check** at any point:

```solidity
// BaseWithdrawPool.sol lines 81-114
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;

    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);

    (uint32 productId, address sendTo, uint128 transferAmount) =
        resolveFastWithdrawal(transaction);
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);  // ← no sanction check on sendTo
}
``` [3](#0-2) 

For `WithdrawCollateralV2`, `sendTo` is an **arbitrary caller-controlled address** embedded in the signed transaction:

```solidity
// BaseWithdrawPool.sol lines 67-77
if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    ...
    address resolvedSendTo = signedTx.tx.sendTo == address(0)
        ? address(uint160(bytes20(signedTx.tx.sender)))
        : signedTx.tx.sendTo;
    return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
}
``` [4](#0-3) 

Neither `sendTo` nor `msg.sender` is checked against the sanctions list before `handleWithdrawTransfer` executes the token transfer. [5](#0-4) 

---

### Impact Explanation

A sanctioned user can receive collateral via the fast-withdrawal path without triggering any on-chain compliance check. The deposit path enforces sanctions on both parties; the fast-withdrawal path enforces them on neither. This directly mirrors the Neutrl bug class: one enforcement path is complete, the other is missing the check entirely.

Concretely:
- A user sanctioned **after** a sequencer-signed `WithdrawCollateralV2` transaction was created can still receive funds at `sendTo`.
- A user can set `sendTo` to a sanctioned address in a `WithdrawCollateralV2` transaction, routing funds to a sanctioned wallet with no on-chain block.
- The protocol's stated compliance posture (enforced at deposit) is silently bypassed at withdrawal.

---

### Likelihood Explanation

`submitFastWithdrawal` is `public` and callable by any address. The only gate is `v.requireValidTxSignatures`, which verifies sequencer signatures. However:
1. Sanctions can be applied **after** the sequencer signs a withdrawal — the on-chain contract is the last line of defense.
2. `WithdrawCollateralV2` allows an explicit `sendTo` address, meaning a non-sanctioned user can route funds to a sanctioned address in a single signed transaction.
3. No admin action is required; the attacker only needs a valid sequencer-signed withdrawal transaction.

---

### Recommendation

Add `requireUnsanctioned` checks in `submitFastWithdrawal` for both the caller and the resolved `sendTo` address, mirroring the deposit path:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;

    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);

    (uint32 productId, address sendTo, uint128 transferAmount) =
        resolveFastWithdrawal(transaction);

+   requireUnsanctioned(msg.sender);   // caller must not be sanctioned
+   requireUnsanctioned(sendTo);       // recipient must not be sanctioned

    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);
}
```

`BaseWithdrawPool` does not currently inherit `EndpointStorage`, so `requireUnsanctioned` should either be added as an internal function referencing the same `ISanctionsList` oracle, or the sanctions address should be stored and checked directly in `BaseWithdrawPool`.

---

### Proof of Concept

```solidity
function test_SanctionedUserFastWithdrawalBypass() external {
    // 1. Sanctioned user has an existing subaccount with collateral
    // 2. Sequencer signs a WithdrawCollateralV2 with sendTo = sanctionedUser
    // 3. Admin sanctions the user AFTER the sequencer signature is produced
    vm.prank(admin);
    sanctions.addToSanctionsList(sanctionedUser);

    // 4. Anyone calls submitFastWithdrawal — no sanction check fires
    uint256 balanceBefore = token.balanceOf(sanctionedUser);
    withdrawPool.submitFastWithdrawal(idx, signedWithdrawV2Tx, sequencerSigs);
    uint256 balanceAfter = token.balanceOf(sanctionedUser);

    // Sanctioned user received funds despite being on the sanctions list
    assertGt(balanceAfter, balanceBefore, "Sanctioned user received funds");
}
```
