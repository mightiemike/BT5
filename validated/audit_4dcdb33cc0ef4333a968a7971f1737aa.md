### Title
`action_delete_account` subtracts only local-contract code length, not global-contract identifier storage, overstating `account_storage_usage` and permanently locking funds in accounts near the deletion limit â€” (`File: runtime/runtime/src/actions.rs`)

---

### Summary

`action_delete_account` computes the "net" storage usage to compare against `MAX_ACCOUNT_DELETION_STORAGE_USAGE` by subtracting only the local contract code length. For accounts whose contract is `AccountContract::Global` (32-byte hash identifier) or `AccountContract::GlobalByAccount` (variable-length account-id identifier), the identifier bytes are **not subtracted**. The resulting `account_storage_usage` is overstated by exactly the identifier size, causing the deletion guard to fire when it should not, and permanently locking the account's funds.

---

### Finding Description

In `action_delete_account` (legacy path, protocol version < 85), the storage-usage calculation is:

```rust
// runtime/runtime/src/actions.rs  lines 316-331
let account_storage_usage = account_ref.storage_usage();
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),
)?;
account_storage_usage.saturating_sub(code_len)
``` [1](#0-0) 

`get_code_len_or_default` returns the byte-length of the **local** contract code stored in the trie. For a `Global` or `GlobalByAccount` contract there is no local code entry, so `code_len` is 0 and nothing is subtracted. However, when the account was assigned a global contract via `use_global_contract`, the identifier's byte length **was** added to `storage_usage`:

```rust
// runtime/runtime/src/global_contracts.rs  lines 97-104
account.set_storage_usage(
    account.storage_usage().checked_add(contract_identifier.len() as u64)â€¦
);
``` [2](#0-1) 

So `account.storage_usage()` already includes the identifier overhead, but the deletion path never removes it. The guard then fires incorrectly:

```rust
// runtime/runtime/src/actions.rs  line 333
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
    result.result = Err(ActionErrorKind::DeleteAccountWithLargeState { â€¦ }.into());
    return Ok(());
}
``` [3](#0-2) 

The correct subtraction â€” `get_contract_storage_usage`, which handles all three contract variants â€” is used only after `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` is enabled:

```rust
// runtime/runtime/src/actions.rs  lines 311-315
let account_storage_usage = if ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage
    .enabled(current_protocol_version)
{
    let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
    account_ref.storage_usage().saturating_sub(contract_storage)
``` [4](#0-3) 

The fix is gated at protocol version 85:

```rust
// core/primitives-core/src/version.rs  lines 555-571
ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage â€¦ => 85,
``` [5](#0-4) 

The divergent value is exact: **32 bytes** for `AccountContract::Global` (a `CryptoHash`), or **`account_id.len()` bytes** for `AccountContract::GlobalByAccount`. Any account whose `storage_usage` falls in the half-open interval `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_len]` is permanently undeletable on nodes running protocol version < 85.

The in-tree test confirms the exact boundary:

```rust
// runtime/runtime/src/actions.rs  lines 1097-1104
// Before the fix: the identifier is not subtracted, so `MAX + 32 > MAX`.
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    storage,          // = MAX + 32
    enabled - 1,      // protocol version before the fix
);
expect_delete_account_too_large(&before);
``` [6](#0-5) 

---

### Impact Explanation

An account whose `storage_usage` is in `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32]` (for `Global`) or `(MAX, MAX + len(account_id)]` (for `GlobalByAccount`) cannot be deleted. `DeleteAccountAction` always returns `DeleteAccountWithLargeState`, so the account's `amount` balance can never be sent to a beneficiary via this path. The funds are permanently locked in the account on any node running protocol version < 85.

`MAX_ACCOUNT_DELETION_STORAGE_USAGE` is 10,000 bytes. [7](#0-6) 

---

### Likelihood Explanation

Any unprivileged user can reach this state by:
1. Calling `UseGlobalContractAction` to assign a global contract to their account (adds identifier bytes to `storage_usage`).
2. Accumulating contract-data storage until `storage_usage` lands in the vulnerable window.
3. Attempting `DeleteAccountAction` â€” it is permanently rejected.

No privileged role is required. The `UseGlobalContractAction` path is public and the storage window is narrow but deterministically reachable.

---

### Recommendation

The fix is already present in the codebase: replace the local-only subtraction with `get_contract_storage_usage`, which correctly handles `None`, `Local`, `Global`, and `GlobalByAccount` variants. This is gated behind `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` (protocol version 85). Ensure this protocol version is activated on all networks. The legacy branch should be retained only for replay of historical blocks at protocol version < 85.

---

### Proof of Concept

```rust
// Construct an account with a Global contract whose storage_usage = MAX + 32

let account = Account::new(
    Balance::from_near(10),
    Balance::ZERO,
    AccountContract::Global(CryptoHash::default()), // 32-byte identifier
    storage,
);
// On protocol version 84 (< fix), action_delete_account computes:
//   code_len = 0  (no local code in trie)
//   account_storage_usage = 10_032 - 0 = 10_032 > 10_000  â†’ DeleteAccountWithLargeState
// The 10 NEAR in `amount` is permanently locked.
```

The exact divergent bytes: the 32-byte `CryptoHash` identifier stored in `account.storage_usage()` but never subtracted in the legacy deletion path.
