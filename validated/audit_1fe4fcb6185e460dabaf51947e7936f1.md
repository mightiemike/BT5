### Title
`validate_delegate_action` Validates Inner `DeterministicStateInitAction` Against the Outer Transaction Receiver Instead of the Delegate's Inner Receiver — (`File: runtime/runtime/src/action_validation.rs`)

### Summary

Before `ProtocolFeature::FixDelegatedDeterministicStateInit` activates, `validate_delegate_action` passes the **outer transaction's `receiver_id`** (which equals `delegate_action.sender_id`, the meta-tx sender's own account) as the `receiver` argument to `validate_actions_with_mode` for the inner actions. The `DeterministicStateInitAction` validator then checks `derive_near_deterministic_account_id(state_init) == outer_receiver` instead of `== delegate_action.receiver_id()`. This is the exact same wrong-identity substitution as the Solidity bug: a wrapper/intermediary's own address is used in place of the actual target's address.

### Finding Description

`validate_delegate_action` in `runtime/runtime/src/action_validation.rs` receives two account IDs:

- `receiver` — the outer transaction's `receiver_id`, which at the point of call equals `delegate_action.sender_id` (the meta-tx sender's account, i.e., the "wrapper" in the analogy).
- `delegate_action.receiver_id()` — the actual target of the inner receipt that will be created.

Before the fix, the function selects `receiver` (the outer/wrong ID) as `inner_receiver`:

```rust
// runtime/runtime/src/action_validation.rs lines 188-199
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // BUG: outer tx receiver, not inner receipt receiver
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
```

`validate_deterministic_state_init` then enforces:

```rust
// runtime/runtime/src/action_validation.rs lines 420-427
let derived_id = derive_near_deterministic_account_id(&action.state_init);
if derived_id != *receiver_id {
    return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
}
```

Because `receiver_id` is the outer tx receiver (the sender's own named account, e.g. `det_account_b`), not the inner receipt's target (`det_account_a`), two divergent outcomes arise:

1. **Legitimate meta-tx blocked**: A user who correctly sets `delegate_action.receiver_id = derive(state_init)` will have their tx rejected at initial validation because `outer_receiver != derive(state_init)` (the outer receiver is the sender's account, not the deterministic account).

2. **Malformed meta-tx passes initial validation**: An attacker who sets `outer_tx.receiver_id = derive(state_init_b)` (making the outer receiver match the derived ID) but sets `delegate_action.receiver_id = det_account_a` (a different account) passes the wrong check. The tx is accepted and forwarded; the receipt then fails at `validate_receipt` → `NewReceiptValidationError::ActionsValidation(InvalidDeterministicStateInitReceiver)`. The relayer's gas is consumed for a tx that was structurally invalid.

The test `try_meta_tx_deterministic_receiver_exploit` in `test-loop-tests/src/tests/deterministic_account_id.rs` (lines 183–266) explicitly demonstrates both outcomes.

### Impact Explanation

- **Feature DoS**: Any relayer or user attempting to create a deterministic account via a meta-transaction (`Action::Delegate` wrapping `DeterministicStateInitAction`) is unconditionally blocked on pre-fix protocol versions. The feature is entirely unusable through the meta-transaction path.
- **Validation bypass at tx admission**: A crafted meta-tx with a mismatched inner receiver passes the tx-admission gate (gas is charged, tx is forwarded to the shard), then fails at receipt execution. This wastes relayer gas and can be used to grief relayer services that trust tx-admission as a pre-filter.

### Likelihood Explanation

Any unprivileged user can submit a `SignedTransaction` containing a `DelegateAction`. No special role is required. The wrong-identity substitution is structural and deterministic: every meta-tx targeting a deterministic account is affected on pre-fix protocol versions. The exploit path (crafting a tx that passes admission but fails at receipt) requires only knowledge of the bug and a valid `DeterministicAccountStateInit`.

### Recommendation

The fix is already present and gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit`: use `delegate_action.receiver_id()` as the `inner_receiver` unconditionally. The protocol-version gate should be removed once the feature is fully stabilized and the pre-fix path is no longer reachable on any supported network.

### Proof of Concept

The divergent value is exact:

| Path | `inner_receiver` passed to `validate_deterministic_state_init` | Expected |
|---|---|---|
| Pre-fix | `outer_tx.receiver_id` = `delegate_action.sender_id` (e.g. `det_account_b`) | `delegate_action.receiver_id` (e.g. `det_account_a`) |
| Post-fix | `delegate_action.receiver_id()` (correct) | same |

**Exploit scenario** (from `test-loop-tests/src/tests/deterministic_account_id.rs`):

```
outer_tx.receiver_id  = det_account_b  = derive(state_init_b)
delegate_action.receiver_id = det_account_a  ← wrong target
inner action: DeterministicStateInitAction { state_init: state_init_b }

Pre-fix check: derive(state_init_b) == det_account_b  → PASSES (wrong)
Receipt check: derive(state_init_b) == det_account_a  → FAILS (correct)
```

The test `test_deterministic_state_init_meta_tx_receiver_check_pre_fix` confirms the exploit tx returns `Ok(outcome)` from `try_execute_tx` (tx admitted) but the outcome status is `Failure(NewReceiptValidationError(InvalidDeterministicStateInitReceiver))`. fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            // This is the correct receiver id to use for the check.
            delegate_action.receiver_id()
        } else {
            // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
            // validated against the wrong id. This makes it impossible to
            // initialize deterministic accounts from meta transactions.
            // The bug cannot be abused, if someone crafts a state init that passes
            // validation here, it will fail when it is checked as incoming receipt.
            receiver
        };
    validate_actions_with_mode(
        limit_config,
        &actions,
        inner_receiver,
        current_protocol_version,
        mode,
    )?;
    Ok(())
}

            ),
            ..
