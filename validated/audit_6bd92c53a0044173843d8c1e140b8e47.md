Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call) and instead validates `owner`, a free caller-supplied argument that designates who receives LP shares. Because any caller can pass an allowlisted address as `owner`, the allowlist check trivially passes for every caller, rendering the deposit restriction inoperative.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both addresses and forwards them to the extension hook in order `(sender, owner, ...)`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` names the first parameter `_` (unnamed/discarded) and the second `owner`, then guards only on `owner`: [3](#0-2) 

The contract's own NatSpec states the intent is to gate by **depositor** address: [4](#0-3) 

The depositor is `sender` (`msg.sender` of the pool call), not `owner`. Since `owner` is a free parameter, any caller can supply an allowlisted address as `owner` and the guard evaluates `allowedDepositor[pool][allowlisted_address] == true` and does not revert. The wrong value is the `extension decision` — the extension returns `beforeAddLiquidity.selector` (allow) when it should revert.

## Impact Explanation

The deposit allowlist — the sole access-control mechanism for liquidity provisioning on permissioned pools — is completely bypassed. Any unauthorized EOA or contract can add liquidity to a pool that the admin explicitly restricted. The attacker pays tokens via the pool's swap-callback mechanism (tokens are pulled from `msg.sender`, i.e., the attacker), and the allowlisted address receives LP shares it never requested. This constitutes a broken core pool functionality and an admin-boundary break: the pool admin's explicit access restriction is defeated by an unprivileged caller with no special setup.

## Likelihood Explanation

Exploitation requires no special privilege, no flash loan, and no oracle manipulation. Any EOA or contract can call `pool.addLiquidity(allowlisted_address, salt, deltas, callbackData, extensionData)`. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attack is repeatable at will. Likelihood is **High**. [5](#0-4) 

## Recommendation

Replace the `owner` check with a `sender` check so the guard validates the actual depositing address:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only Alice:
   ```solidity
   extension.setAllowedToDeposit(pool, alice, true);
   ```
2. Bob (not allowlisted) implements `IMetricOmmAddLiquidityCallback` so the pool's callback pulls tokens from Bob's balance.
3. Bob calls:
   ```solidity
   pool.addLiquidity(alice, salt, deltas, callbackData, extensionData);
   ```
4. The pool calls `extension.beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. The guard evaluates `allowedDepositor[pool][alice] == true` → no revert.
6. Bob's tokens are transferred into the pool; Alice receives LP shares.
7. Bob has deposited into a pool he is not authorized to touch. The attack is repeatable for any bin configuration. [6](#0-5) [3](#0-2)
