Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any router caller to bypass the per-user swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `sender`, which is `msg.sender` of `pool.swap()`. When a user trades through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the end user. A pool admin who allowlists the router to enable standard UX for their curated users inadvertently grants swap access to every router caller, completely defeating the per-user allowlist.

## Finding Description
The call chain is confirmed by the production code:

**Step 1 — Router calls pool with itself as `msg.sender`:**
`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` directly. [1](#0-0) 

**Step 2 — Pool forwards `msg.sender` (the router) as `sender` to `_beforeSwap`:**
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, recipient, ...)`. [2](#0-1) 

**Step 3 — `ExtensionCalling._beforeSwap` encodes and forwards `sender` (router) to the extension:** [3](#0-2) 

**Step 4 — Extension checks `allowedSwapper[pool][sender]` where `sender` is the router:**
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — `msg.sender` is the pool, `sender` is the router address. [4](#0-3) 

The allowlist mapping is keyed `allowedSwapper[pool][swapper]`. [5](#0-4) 

When the admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps for their curated users, the check `allowedSwapper[pool][router]` returns `true` for **every** caller of the router, not just the intended allowlisted users. There is no configuration that achieves "only allowlisted users can swap through the router" — the admin either blocks all router users (by not allowlisting the router) or opens the pool to all router users (by allowlisting it).

## Impact Explanation
This constitutes broken core pool functionality: the `SwapAllowlistExtension` is a deployed, supported extension whose stated purpose is to gate swaps by swapper address per pool. When used with the supported public entrypoint (`MetricOmmSimpleRouter`), the gate does not hold. Any unprivileged address can call `router.exactInputSingle(...)` and execute swaps in a pool intended to be restricted. For pools curated to exclude informed traders (e.g., to protect LPs from adverse selection), those traders can enter through the router, extracting value from LPs. This is a broken core pool security boundary with direct LP fund impact.

## Likelihood Explanation
Medium. The pool admin must allowlist the router — a plausible and expected configuration step when deploying a curated pool intended to be usable via the standard periphery. The `setAllowedToSwap(pool, router, true)` call is syntactically identical to allowlisting any individual user; the API gives no indication that it grants access to all router users. The bypass is repeatable by any address with no special privileges.

## Recommendation
The extension must gate on the economic actor (the end user), not the intermediary contract. The cleanest fix is to require the router to encode `msg.sender` into `extensionData`, and have the extension decode and verify it when `sender` is a known router address. Alternatively, document that `SwapAllowlistExtension` is incompatible with router-based swaps and enforce direct pool calls for curated pools — but this breaks standard UX.

## Proof of Concept
```
Setup:
  - Pool deployed with SwapAllowlistExtension
  - Pool admin: setAllowedToSwap(pool, alice, true)
  - Pool admin: setAllowedToSwap(pool, router, true)
    (to allow alice to use the standard router UX)

Attack:
  - charlie (not allowlisted) calls:
      router.exactInputSingle({pool: pool, ..., recipient: charlie})
  - Router calls pool.swap(charlie, ...) with msg.sender = router
  - Pool calls _beforeSwap(sender=router, ...)
  - Extension checks allowedSwapper[pool][router] → true → PASSES
  - charlie successfully swaps in the restricted pool

Result:
  - charlie receives output tokens
  - The per-user allowlist is completely bypassed
  - LPs in a curated pool are exposed to uninvited counterparties
```
