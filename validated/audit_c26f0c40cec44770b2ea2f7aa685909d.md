### Title
CEI Violation in `withdrawCollateral` Allows Health-Check Bypass via Reentrancy — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer to the caller-controlled `sendTo` address **before** debiting the subaccount balance in `SpotEngine`. No reentrancy guard exists anywhere in the Endpoint or Clearinghouse. With a token that has transfer callbacks (ERC777 or equivalent), an attacker can reenter the protocol during the transfer, observe an inflated pre-debit balance, and trigger a second withdrawal that passes the health check it should fail — draining more collateral than the subaccount legitimately holds.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — transfers tokens to `sendTo` (attacker-controlled)
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` — debits the subaccount balance
3. **Line 419** — `require(getHealth(sender, healthType) >= 0, ...)` — health check [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse` calls `token.safeTransfer(withdrawPool, amount)` and then `BaseWithdrawPool.submitWithdrawal`, which in turn calls `token.safeTransfer(sendTo, amount)`. [2](#0-1) 

The final `token.safeTransfer(to, amount)` in `BaseWithdrawPool.handleWithdrawTransfer` is the reentrancy entry point. [3](#0-2) 

A grep for `nonReentrant`, `ReentrancyGuard`, and `_nonReentrantBefore` across the entire repository returns **zero matches** — neither `Endpoint` nor `Clearinghouse` carries any reentrancy protection. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An attacker with subaccount balance `B` queues two slow-mode withdrawals of `A` each, where `B − A ≥ 0` (first is healthy) but `B − 2A < 0` (second should fail). During the ERC777 `tokensReceived` callback triggered by the first withdrawal's token transfer:

- `spotEngine` still records balance `B` (debit has not occurred)
- The attacker triggers the second withdrawal (e.g., by calling `Endpoint.executeSlowModeTransaction` — no reentrancy guard blocks this)
- The second `withdrawCollateral` call reads balance `B`, computes health as `B − A ≥ 0`, and passes
- Both withdrawals complete; the subaccount is debited twice, ending at `B − 2A < 0`
- The attacker has received `2A` tokens while only holding `B < 2A`

The corrupted state is `spotEngine`'s balance for the subaccount, which is driven negative beyond what the health check should permit, constituting direct collateral theft from the protocol's insurance or other depositors.

---

### Likelihood Explanation

**Medium.** The precondition is a supported token with transfer callbacks (ERC777 or a vault share token implementing `tokensReceived`/`afterTokenTransfer`). The Nado codebase already references ERC4626 vault tokens (`IERC4626Base`) as supported collateral types. [6](#0-5) 

ERC4626 vaults commonly implement `afterDeposit`/`afterWithdraw` hooks. If any such token is listed as a supported product, the precondition is met. The attacker entry path (submitting slow-mode withdrawals) is fully unprivileged.

---

### Recommendation

Move `spotEngine.updateBalance` and `spotEngine.assertUtilization` to **before** `handleWithdrawTransfer`, following the Checks-Effects-Interactions pattern:

```solidity
// EFFECTS first
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// INTERACTIONS last
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to `Endpoint.depositCollateral`, `Endpoint.executeSlowModeTransaction`, and `Clearinghouse.withdrawCollateral`.

### Proof of Concept

```
Attacker subaccount balance: B = 1000 USDC
Attacker queues: withdrawal₁(A=600), withdrawal₂(A=600)
  → B − 600 = 400 ≥ 0  (healthy)
  → B − 1200 = −200 < 0 (should fail)

Sequencer executes withdrawal₁:
  Clearinghouse.withdrawCollateral(sender, USDC, 600, AttackerContract, idx1)
    → handleWithdrawTransfer → token.safeTransfer(AttackerContract, 600)
      [ERC777 tokensReceived fires]
      AttackerContract calls Endpoint.executeSlowModeTransaction(withdrawal₂)
        Clearinghouse.withdrawCollateral(sender, USDC, 600, AttackerContract, idx2)
          spotEngine.balance[sender] = 1000  ← not yet debited!
          handleWithdrawTransfer → token.safeTransfer(AttackerContract, 600)
          spotEngine.updateBalance → balance = 1000 − 600 = 400
          getHealth(400) ≥ 0 ✓  ← passes incorrectly
      [callback returns]
    spotEngine.updateBalance → balance = 400 − 600 = −200  ← protocol insolvent

Result: attacker received 1200 USDC, protocol balance −200 USDC
``` [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
