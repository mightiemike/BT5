### Title
`updateSubmitter` Allows Setting `submitter` to Zero Address, Permanently Bricking the Upgrade Mechanism — (`File: core/contracts/BaseProxyManager.sol`)

### Summary
`BaseProxyManager.updateSubmitter` accepts `address(0)` as a valid new submitter. Because `submitImpl` is gated by `onlySubmitter`, setting `submitter` to the zero address permanently prevents any new implementation from ever being submitted, freezing the protocol's upgrade path.

### Finding Description
`BaseProxyManager.updateSubmitter` (line 181–183) performs no zero-address validation before writing to `submitter`:

```solidity
function updateSubmitter(address newSubmitter) external onlyOwner {
    submitter = newSubmitter;   // no require(newSubmitter != address(0))
}
```

The `onlySubmitter` modifier (lines 89–95) enforces `msg.sender == submitter`. If `submitter` is `address(0)`, no EOA or contract can ever satisfy that check, so `submitImpl` becomes permanently uncallable. Because `migrateAll` depends on `pendingImpls` being populated by `submitImpl`, the entire upgrade pipeline is bricked with no recovery path short of a full proxy admin intervention. [1](#0-0) [2](#0-1) 

### Impact Explanation
- `submitter` is the sole address authorized to call `submitImpl`, which populates `pendingImpls` and `pendingHashes`.
- `migrateAll` (line 189) iterates over `pendingImpls`; without a valid submitter, no new implementation can ever be staged.
- Setting `submitter = address(0)` permanently disables the protocol's upgrade mechanism. Any critical bug fix, security patch, or feature upgrade becomes impossible. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The owner calling `updateSubmitter(address(0))` is a realistic operational mistake (e.g., passing an uninitialized variable, a scripting error, or a zero-address placeholder during a migration). There is no on-chain safeguard to catch it, and the effect is irreversible within the contract's own logic.

### Recommendation
Add a zero-address guard identical in spirit to the external report's recommendation:

```solidity
function updateSubmitter(address newSubmitter) external onlyOwner {
    require(newSubmitter != address(0), "Incorrect submitter address");
    submitter = newSubmitter;
}
``` [1](#0-0) 

### Proof of Concept
1. Owner calls `updateSubmitter(address(0))`.
2. `submitter` is now `address(0)`.
3. Any subsequent call to `submitImpl(name, impl)` hits `onlySubmitter`, which checks `msg.sender == submitter` → `msg.sender == address(0)` → always reverts.
4. `pendingImpls[name]` can never be updated.
5. `migrateAll` can never stage or apply any new implementation.
6. The entire upgrade mechanism is permanently frozen. [5](#0-4) [1](#0-0)
