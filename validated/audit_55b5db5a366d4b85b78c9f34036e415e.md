Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on LP-Position `owner` Instead of Caller `sender`, Enabling Allowlist Bypass and Blocking Legitimate Routers — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`, who pays tokens via callback) and instead checks `owner` (the LP-position recipient). Because `MetricOmmPool.addLiquidity` imposes no constraint that `msg.sender == owner`, any non-allowlisted address can bypass the guard by naming an allowlisted address as `owner`, while any allowlisted router that separates payer from position owner is incorrectly blocked.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient to `_beforeAddLiquidity`: [1](#0-0) 

There is no `msg.sender == owner` guard in `addLiquidity`, unlike `removeLiquidity` which enforces it: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but leaves it unnamed and unused, checking only `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`: [4](#0-3) 

**Bypass path:** Non-allowlisted attacker `B` calls `pool.addLiquidity(owner = A, ...)` where `A` is allowlisted. The extension evaluates `allowedDepositor[pool][A]` → `true` → passes. `B` pays tokens via the swap callback and successfully injects liquidity into a permissioned pool.

**Blocking path:** Allowlisted router `A` calls `pool.addLiquidity(owner = B, ...)` to create a position on behalf of user `B`. The extension evaluates `allowedDepositor[pool][B]` → `false` → reverts. The router is blocked despite being explicitly allowlisted, making the `addLiquidity` flow unusable for any contract that separates payer from position owner.

## Impact Explanation
The bypass path constitutes an admin-boundary break: an unprivileged address circumvents the pool admin's access-control intent and injects liquidity into a permissioned pool, distorting bin distribution and affecting swap execution for all users. The blocking path constitutes broken core pool liquidity functionality: any router or liquidity-management contract (e.g., `MetricOmmPoolLiquidityAdder`) that uses `owner != msg.sender` is rendered non-functional against pools with this extension. Both match the allowed impact gate.

## Likelihood Explanation
The bypass requires only knowledge of one allowlisted address and a standard `addLiquidity` call — no special permissions, flash loans, or privileged access. The blocking path is triggered immediately by any allowlisted router operating normally. Both paths are reachable by any unprivileged external caller against any pool configured with `DepositAllowlistExtension`.

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

Also update `setAllowedToDeposit`, `isAllowedToDeposit`, and admin documentation to clarify that the allowlisted address is the **caller** of `addLiquidity`, not the LP-position owner.

## Proof of Concept
```solidity
// Pool configured with DepositAllowlistExtension.
// Only allowedSender is allowlisted; attacker is not.
extension.setAllowedToDeposit(pool, allowedSender, true);

// Attacker calls addLiquidity specifying allowedSender as owner.
// Extension checks allowedDepositor[pool][allowedSender] → true → passes.
// Attacker pays tokens via callback; allowedSender receives LP shares.
pool.addLiquidity(
    allowedSender,  // owner ← allowlisted, check passes
    0,
    deltas,
    callbackData,
    ""
); // called by attacker — succeeds despite attacker not being allowlisted

// Blocking path: allowlisted router blocked when owner != router
extension.setAllowedToDeposit(pool, router, true);
vm.prank(router);
pool.addLiquidity(user, 0, deltas, callbackData, "");
// Extension checks allowedDepositor[pool][user] → false → NotAllowedToDeposit revert
```
