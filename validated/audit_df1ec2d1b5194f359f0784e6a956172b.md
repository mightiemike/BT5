### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Repeated Bonus Reward Payout — (`File: x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before any writes occur. Because there is no deduplication check on `msg.PositionIds`, an attacker who includes the same position ID multiple times causes `processEventsAndClaimBonus` to execute against stale in-memory copies of the position — each copy still carrying the old `LastBonusAccrual` checkpoint — resulting in the same bonus interval being paid out once per duplicate entry, draining the `BonusRewardsPool` module account.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` first collects all positions into a slice, then delegates to `claimRewardsAndUpdatesPositions`:

```go
// msg_server.go:434-448
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)   // reads from store
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

All reads happen before any writes. If `msg.PositionIds = [42, 42]`, both `positions[0]` and `positions[1]` are independent in-memory copies of the same on-chain state, both carrying the same stale `LastBonusAccrual`.

`claimRewardsAndUpdatesPositions` then iterates the slice:

```go

When the loop reaches `positions[1]` (the second copy of the same position), it was loaded before the first `setPosition` write, so its `LastBonusAccrual` is still the original stale value. `processEventsAndClaimBonus` recomputes the identical bonus interval `[old_LastBonusAccrual, blockTime]` and issues a second `SendCoinsFromModuleToAccount` transfer from `RewardsPoolName` to the attacker.

The `sufficientBonusPoolBalance` guard runs per-iteration against the live pool balance, so it does not prevent the double-pay — it only prevents the pool from going negative. An attacker calibrates the number of duplicates to exactly drain the pool without triggering the guard. [5](#0-4) 

---

### Impact Explanation

The `BonusRewardsPool` module account holds the pre-funded bonus APY reserve for all tiered stakers. An attacker with any single valid position can drain the entire pool in one transaction by submitting `MsgClaimTierRewards` with the position ID repeated N times (where N × per-claim-bonus ≤ pool balance). This constitutes direct theft of funds held in the module account, depriving all other legitimate stakers of their earned bonus rewards.

---

### Likelihood Explanation

The entry path requires only a standard `MsgClaimTierRewards` transaction signed by any account that owns at least one active tiered position. No privileged role, governance action, or special timing is needed. The attack is executable in a single transaction and is trivially discoverable by reading the message handler.

---

### Recommendation

Deduplicate `msg.PositionIds` before loading positions, or re-fetch each position from the store inside `claimRewardsAndUpdatesPositions` rather than operating on a pre-loaded snapshot. The simplest fix is to add a uniqueness check in `MsgClaimTierRewards.Validate()` or at the top of the handler:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, reload each position from the store immediately before calling `processEventsAndClaimBonus` so that the second occurrence of the same ID reads the already-updated `LastBonusAccrual`.

---

### Proof of Concept

1. Attacker holds position ID `42` with `LastBonusAccrual = T0` and accrued bonus `B`.
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42, ..., 42]}` (repeated K times, where `K × B ≤ pool_balance`).
3. Handler loads K identical copies of position 42 (all with `LastBonusAccrual = T0`) into `positions[]`.
4. Loop iteration 0: computes bonus `B` for `[T0, blockTime]`, pays `B` to attacker, writes `LastBonusAccrual = blockTime` to store.
5. Loop iteration 1: operates on stale in-memory copy with `LastBonusAccrual = T0`, computes bonus `B` again, pays `B` to attacker again.
6. Iterations 2…K−1: same.
7. Attacker receives `K × B` tokens; legitimate stakers' bonus pool is drained.

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
