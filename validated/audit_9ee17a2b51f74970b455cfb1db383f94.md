### Title
Funds Sent to Legacy `tieredrewards` Module Account Are Permanently Locked Due to Unblocked Receive and No Active Keeper — (`app/app.go`)

---

### Summary

`BlockedAddrs()` returns `false` for `tieredrewardstypes.ModuleName` ("tieredrewards") because `moduleAccsAllowedToReceiveExternalFunds` maps it to `true`. Any unprivileged account can therefore execute a standard `MsgSend` to `authtypes.NewModuleAddress("tieredrewards")`, the bank keeper's blocked-address guard passes, and the funds are deposited into a legacy module account for which **no active keeper operation exists to withdraw them**. The funds are permanently locked.

---

### Finding Description

In `app/app.go`, two maps govern the legacy module account:

```go
// app/app.go lines 153-165
maccPerms = map[string][]string{
    ...
    // Legacy, not used anymore.
    tieredrewardstypes.ModuleName: {authtypes.Staking},
}

// app/app.go lines 169-173
moduleAccsAllowedToReceiveExternalFunds = map[string]bool{
    tieredrewardstypes.RewardsPoolName: true,
    // Legacy, not used anymore.
    tieredrewardstypes.ModuleName: true,
}
``` [1](#0-0) 

`BlockedAddrs()` computes:

```go
blockedAddrs[authtypes.NewModuleAddress(acc).String()] = !moduleAccsAllowedToReceiveExternalFunds[acc]
```

For `tieredrewardstypes.ModuleName`, this evaluates to `!true = false`, meaning the address is **not blocked**. [2](#0-1) 

The active tieredrewards keeper (`NewKeeper`) only validates the existence of `RewardsPoolName`, not `ModuleName`:

```go
if addr := accountKeeper.GetModuleAddress(types.RewardsPoolName); addr == nil {
    panic(...)
}
``` [3](#0-2) 

A search across all keeper files (`msg_server.go`, `abci.go`, `delegation.go`, `bonus_rewards.go`, `claim_rewards.go`, `claim_rewards.go`) confirms that **no keeper operation references `types.ModuleName` for any bank send, module-account withdrawal, or fund sweep**. The active module exclusively uses `RewardsPoolName` and per-position delegator addresses derived via `createPositionDelegatorAccount`. [4](#0-3) 

---

### Impact Explanation

Any user who sends `basecro` (or any coin) to `authtypes.NewModuleAddress("tieredrewards")` via `MsgSend`:

1. Passes the bank keeper's blocked-address check (returns `false`).
2. Has their funds deposited into the legacy module account.
3. Has **no on-chain path to recover those funds** — no `MsgServer` method, no `BeginBlocker`/`EndBlocker` sweep, and no governance message targets this account.

The result is **permanent, irrecoverable user fund loss** for any amount sent to this address. [5](#0-4) 

---

### Likelihood Explanation

The path requires a user to send funds directly to a module account address. This is an unusual action but is fully reachable via a standard `MsgSend` transaction with no special privileges. A confused user, a buggy client, or a malicious contract routing funds there would trigger the loss. The address is deterministic and publicly derivable from `authtypes.NewModuleAddress("tieredrewards")`.

---

### Recommendation

Remove `tieredrewardstypes.ModuleName` from `moduleAccsAllowedToReceiveExternalFunds`. Since the module is explicitly annotated "Legacy, not used anymore," there is no legitimate reason for external accounts to send funds to it. This single-line removal causes `BlockedAddrs()` to return `true` for that address, blocking all external sends:

```go
moduleAccsAllowedToReceiveExternalFunds = map[string]bool{
    tieredrewardstypes.RewardsPoolName: true,
    // tieredrewardstypes.ModuleName removed — legacy, no active keeper
}
``` [6](#0-5) 

Additionally, consider whether `tieredrewardstypes.ModuleName` should remain in `maccPerms` at all. If the legacy account holds no funds on-chain, removing it from `maccPerms` (after confirming zero balance via a migration check) would eliminate the account entirely from the blocked-address surface.
