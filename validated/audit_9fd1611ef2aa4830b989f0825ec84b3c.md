### Title
Untrusted `_cached_sha256_treehash` Blindly Inherited in `Program.wrap()` Produces Wrong Puzzle Hash - (File: `wheel/python/clvm_rs/program.py`)

### Summary

`Program.wrap()` unconditionally copies `_cached_sha256_treehash` from any caller-supplied `CLVMStorage` object without verifying it against the actual tree content. `Program.tree_hash()` then returns this cached value directly, skipping recomputation. An attacker who can pass a Python object with a forged `_cached_sha256_treehash` attribute to `Program.wrap()` causes every downstream consumer of `tree_hash()` — including `__eq__`, `curry_hash()`, and puzzle-hash verification — to operate on a wrong hash.

### Finding Description

In `Program.wrap()`, the cache field is inherited unconditionally:

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
``` [1](#0-0) 

`tree_hash()` then short-circuits on this value without any recomputation:

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
``` [2](#0-1) 

The same blind-trust pattern exists in `Treehasher.sha256_treehash()`, which reads `_cached_sha256_treehash` from every sub-object in the tree without verification:

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    hash_stack.append(r)
    return
``` [3](#0-2) 

`CLVMStorage` is a plain Python protocol — any object satisfying it can be passed to `Program.wrap()` or used as a sub-node in a tree. There is no mechanism that prevents a caller from setting `_cached_sha256_treehash` to an arbitrary value before wrapping.

The broken invariant is: `program.tree_hash()` must always equal the canonical SHA-256 tree hash of the program's actual content. This invariant is violated whenever the wrapped object carries a pre-set, unverified `_cached_sha256_treehash`.

### Impact Explanation

Every security-critical operation that relies on `tree_hash()` is affected:

- **`Program.__eq__`** compares programs by `tree_hash()`, so two programs with different content but the same injected hash compare as equal. [4](#0-3) 

- **`Program.curry_hash()`** feeds `self.tree_hash()` into the curry-hash computation, producing a wrong puzzle hash for any curried coin. [5](#0-4) 

- Any wallet or node code that calls `program.tree_hash()` to derive or verify a coin's puzzle hash will receive the wrong value, enabling coin misidentification or puzzle-hash bypass.

### Likelihood Explanation

The `CLVMStorage` protocol is intentionally open: any Python object with `atom` and `pair` attributes qualifies. Wallet and node code routinely wraps externally-sourced objects via `Program.wrap()` or `Program.to()`. A library consumer who receives a `CLVMStorage`-compatible object from an untrusted source (e.g., a peer, a serialized message, or a plugin) and wraps it will silently inherit whatever `_cached_sha256_treehash` that object carries. No privilege is required; the attacker only needs to supply a conforming Python object.

### Recommendation

Remove the unconditional cache inheritance in `Program.wrap()`. Either:

1. Do not copy `_cached_sha256_treehash` from the wrapped object at all (always recompute on first call), or
2. Copy it only from objects whose type is already trusted (e.g., `CLVMTree` produced by the internal deserializer), and set it to `None` for all other `CLVMStorage` inputs.

The same fix must be applied to `Treehasher.sha256_treehash()`: it must not short-circuit on `_cached_sha256_treehash` values sourced from objects it did not itself hash.

### Proof of Concept

```python
from clvm_rs.clvm_storage import CLVMStorage
from clvm_rs.program import Program

class MaliciousStorage(CLVMStorage):
    atom = b"real_data"
    pair = None
    # Inject a completely wrong hash
    _cached_sha256_treehash = b"\xde\xad" * 16

obj = MaliciousStorage()
p = Program.wrap(obj)

# Returns the injected fake hash, not sha256tree(b"real_data")
print(p.tree_hash().hex())  # deaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddeaddead

# Equality check is now broken: two programs with different content
# but the same injected hash compare as equal
p2 = Program.wrap(MaliciousStorage())
assert p == p2  # passes even if content differs

# curry_hash() derives the wrong puzzle hash
wrong_curry = p.curry_hash()
```

The injected hash propagates through `__eq__` and `curry_hash()` without any recomputation, exactly mirroring the oracle report's pattern: a cached/dampened value is trusted and reused in security-critical paths instead of the current correct value.
