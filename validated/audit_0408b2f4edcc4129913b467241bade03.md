### Title
Compromised Linked Signer Can Permanently Retain Signing Authority by Self-Issuing `LinkSigner` Transactions — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The sequenced `LinkSigner` transaction handler in `EndpointTx.sol` accepts signatures from the **current linked signer** (`allowLinkedSigner = true`). This means a compromised linked signer can sign a new `LinkSigner` transaction to redirect the `linkedSigners` mapping to a fresh address they control, racing against the subaccount owner's removal attempt and indefinitely retaining signing authority over the subaccount.

---

### Finding Description

In the sequenced (fast-path) transaction handler, `LinkSigner` is processed at: [1](#0-0) 

The call to `validateSignedTx` passes `allowLinkedSigner = true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` resolves the permitted signer as either the subaccount owner **or** the current linked signer: [3](#0-2) 

`getLinkedSigner` returns `linkedSigners[subaccount]` — the very address whose removal is being attempted: [4](#0-3) 

Because the linked signer can sign a `LinkSigner` transaction, it can atomically redirect `linkedSigners[subaccount]` to a new address it controls before the owner's removal transaction is sequenced. The nonce is per-subaccount address, so both the owner's removal and the attacker's re-delegation consume the same nonce slot. Whichever the sequencer processes first wins. The attacker can repeat this for every subsequent nonce the owner submits, permanently blocking removal.

By contrast, the slow-mode `LinkSigner` path correctly restricts the operation to the subaccount owner via `validateSender`, which checks `address(uint160(bytes20(txSender))) == sender`: [5](#0-4) [6](#0-5) 

The inconsistency between the two paths is the root cause.

---

### Impact Explanation

A linked signer with retained authority can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `LinkSigner` — to perpetuate the attack
- `TransferQuote` — to move quote balances out of the subaccount
- `MintNlp` / `BurnNlp` — to manipulate NLP positions
- Order matching via `getLinkedSignerOrNlpSigner` — to place and fill trades that drain the subaccount [7](#0-6) 

The concrete corrupted state is `linkedSigners[subaccount]` being permanently controlled by the attacker, enabling unauthorized balance mutations across spot and perp engines for the victim subaccount.

---

### Likelihood Explanation

The attacker only needs to have been a legitimately granted linked signer at some point (e.g., a trading API key). Once the owner attempts revocation, the attacker submits a competing `LinkSigner` transaction to the sequencer with the same nonce, pointing to a fresh address they control. Since sequencer ordering is not guaranteed to be strictly FIFO from the owner's perspective, and the attacker can submit at high frequency, they can reliably win the race. No privileged access beyond the already-held linked signer key is required.

---

### Recommendation

Remove `allowLinkedSigner` from the sequenced `LinkSigner` handler. Only the subaccount owner (the address whose first 20 bytes match the subaccount) should be permitted to authorize a change to the linked signer mapping. This mirrors the correct restriction already enforced in the slow-mode path via `validateSender`.

Change line 586 from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may change linked signer
);
```

---

### Proof of Concept

1. Alice owns subaccount `S`. She grants linked signer `L` (a trading bot key).
2. `L` is compromised. Alice submits a sequenced `LinkSigner(sender=S, signer=address(0), nonce=N)` signed by herself to revoke `L`.
3. The attacker, controlling `L`, simultaneously submits `LinkSigner(sender=S, signer=L2, nonce=N)` signed by `L` (valid because `allowLinkedSigner = true`).
4. The sequencer processes the attacker's transaction first. `linkedSigners[S] = L2`. Alice's transaction fails: nonce `N` is consumed.
5. Alice submits `LinkSigner(sender=S, signer=address(0), nonce=N+1)`. The attacker submits `LinkSigner(sender=S, signer=L3, nonce=N+1)` signed by `L2`.
6. This repeats indefinitely. The attacker retains signing authority and uses it to call `TransferQuote` or place orders via `matchOrders`, draining Alice's subaccount balance. [1](#0-0) [8](#0-7)
