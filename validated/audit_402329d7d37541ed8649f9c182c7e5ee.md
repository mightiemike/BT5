### Title
Arithmetic Overflow in `op_unknown` Cost Multiplication Enables Cost Limit Bypass - (File: src/more_ops.rs)

### Summary
The `op_unknown` function in `src/more_ops.rs` performs an unchecked `u64` multiplication when computing the final cost for unknown opcodes. In Rust release builds, integer overflow wraps silently. An attacker who controls CLVM program bytes can craft an unknown opcode whose inner cost accumulates to a value that, when multiplied by `cost_multiplier + 1`, wraps around `u64::MAX` to a value ≤ `u32::MAX`, bypassing the subsequent validity check and returning a fraudulently small (potentially zero) cost.

### Finding Description
At line 261 of `src/more_ops.rs`, after computing the inner cost via one of four cost functions and verifying it against `max_cost`, the code performs:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is a `u64` derived from `u32_from_u8(&op[0..op.len() - 1])`, capped at `u32::MAX = 4_294_967_295`. Thus `cost_multiplier + 1` is at most `2^32 = 4_294_967_296`. The multiplication `cost *= cost_multiplier + 1` is performed on two `u64` values with no overflow guard.

In Rust release builds, `u64` overflow wraps modulo `2^64`. The check `cost > u32::MAX` is applied **after** the multiplication, so a wrapped result that happens to be ≤ `u32::MAX` passes silently and is returned as the charged cost.

The inner cost functions (cost_function 1, 2, 3) accumulate cost proportional to argument count and byte sizes, all bounded by `check_cost(cost, max_cost)` inside the loop. The attacker fully controls the argument list and can therefore drive the pre-multiplication `cost` to any specific value in `[base_cost, max_cost]`.

For a concrete trigger: choose `cost_multiplier = 0xfffeffff` (opcode bytes `[0xff, 0xfe, 0xff, 0xff, XX]`, which avoids the `0xffff`-prefix reserved check), giving `M = cost_multiplier + 1 = 0xffff0000`. The attacker then solves for a `cost` value in `[99, max_cost]` such that `cost * 0xffff0000 mod 2^64 ≤ 0xffffffff`. Approximately `max_cost * u32::MAX / 2^64 ≈ 2–3` such values exist in the reachable range; the attacker can hit any of them precisely by choosing the number and sizes of arguments to the unknown opcode (cost_function = 1: `cost = 99 + n * 320 + total_bytes * 3`). ### Impact Explanation
`Cost` is `u64` throughout the engine. [3](#0-2) 

When the multiplication wraps, the returned `Reduction` carries a cost far below what the opcode should charge. Because the Chia block validator sums per-opcode costs against a block-level `max_cost` budget, an attacker can include unknown opcodes that consume near-zero cost budget while the node still processes them. This allows a crafted transaction to pack more total operations into a block than the cost model permits, undermining the DoS-prevention guarantee of the cost system. In Rust debug builds the same path panics, creating a consensus split between debug and release nodes.

### Likelihood Explanation
The attacker controls the full CLVM program bytes, including the opcode atom (choosing `cost_multiplier`) and the argument list (choosing the accumulated inner cost). The reserved-opcode filter (`0xffff` prefix) reduces the maximum reachable `cost_multiplier` but does not eliminate the overflow window. The attacker must solve a modular arithmetic equation to find a valid `cost` target, but the equation has solutions in the reachable range and the cost is continuously tunable via argument count/size. Likelihood is **Medium**: requires deliberate crafting but no privileged access.

### Recommendation
Replace the bare multiplication with an explicit overflow check before applying it:

```rust
// Instead of: cost *= cost_multiplier + 1;
let multiplied = cost.checked_mul(cost_multiplier + 1);
match multiplied {
    None | Some(u32::MAX_AS_U64_PLUS_ONE..) => return Err(EvalErr::Invalid(o))?,
    Some(c) => cost = c,
}
```

Or use `cost.saturating_mul(cost_multiplier + 1)` and then apply the `> u32::MAX` check, which is safe because saturation at `u64::MAX` will always exceed `u32::MAX`.

### Proof of Concept

1. Construct a CLVM program that calls an unknown opcode with atom bytes `[0xff, 0xfe, 0xff, 0xff, 0x40]`:
   - `cost_function = (0x40 >> 6) = 1` (arith-like)
   - `cost_multiplier = u32_from_u8(&[0xff, 0xfe, 0xff, 0xff]) = 0xfffeffff`
   - `M = cost_multiplier + 1 = 0xffff0000`

2. Provide arguments to the opcode such that the inner cost loop accumulates a `cost` value `c` satisfying `c * 0xffff0000 mod 2^64 ≤ 0xffffffff`. One such value is `c = 2^48` (if reachable within `max_cost`); for Chia's `max_cost = 11_000_000_000`, find `c` via: `c ≡ 0 (mod 2^48 / gcd(0xffff0000, 2^64))`.

3. With cost_function = 1: set `n` arguments and `total_bytes` so that `99 + n * 320 + total_bytes * 3 = c`. The attacker controls both `n` and argument sizes.

4. In a release build, `cost *= 0xffff0000` wraps; the result is ≤ `u32::MAX`; the function returns `Ok(Reduction(wrapped_cost, nil))` instead of `Err(EvalErr::Invalid)`, charging the attacker a near-zero cost for the opcode. [4](#0-3) [1](#0-0)

    // | multiplier    |XX | XXXXXX |

    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // is since cost may not be 0.

    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();
