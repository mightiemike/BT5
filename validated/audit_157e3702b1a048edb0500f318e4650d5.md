### Title
Sanctioned Users Cannot Submit Slow Mode Withdrawal, Permanently Locking Deposited Funds — (`core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` applies `requireUnsanctioned(sender)` unconditionally to every slow mode transaction type, including `WithdrawCollateral`. Because slow mode is the only censorship-resistant withdrawal path available to users, a user who deposits collateral and is subsequently added to the sanctions list loses all ability to recover their funds.

---

### Finding Description

In `EndpointTx.submitSlowModeTransactionImpl`, after optionally charging a slow mode fee, the function calls `requireUnsanctioned(sender)` before queuing any slow mode transaction: [1](#0-0) 

The check is unconditional — it applies to every transaction type that reaches that code path, including `WithdrawCollateral` (which is not listed in any special-case branch and therefore falls through to the general `else` block): [2](#0-1) 

`requireUnsanctioned` is defined in `EndpointStorage` and hard-reverts for any address on the Chainalysis/OFAC sanctions list: [3](#0-2) 

The sequencer path (`processTransactionImpl`) does **not** call `requireUnsanctioned`, so the sequencer is technically capable of processing a sanctioned user's withdrawal. However, the sequencer is expected to censor sanctioned addresses — that is precisely why the slow mode path exists as a censorship-resistance fallback. Blocking slow mode for sanctioned users eliminates the only self-service recovery path.

There is no admin function visible in the codebase that force-withdraws collateral on behalf of a sanctioned user, and the `_executeSlowModeTransaction` catch block explicitly notes that the "return funds" mechanism was removed: [4](#0-3) 

---

### Impact Explanation

A user who deposits collateral via `depositCollateralWithReferral` and is later sanctioned:

1. Cannot use the sequencer path (sequencer censors sanctioned addresses by design).
2. Cannot use slow mode — `requireUnsanctioned(sender)` reverts at line 375 of `EndpointTx.sol`.
3. Has no other on-chain withdrawal path.

Result: **permanent lock of all deposited collateral** with no recovery mechanism. The corrupted state is the user's spot balance in `SpotEngine` — it remains positive but is unreachable. [5](#0-4) 

---

### Likelihood Explanation

Retroactive sanctions are a documented real-world event (e.g., Tornado Cash address sanctioning in 2022 affected thousands of addresses that had previously interacted with the protocol). Any user whose address is added to the OFAC/Chainalysis list after depositing into Nado is immediately affected. No attacker action is required — the trigger is an external regulatory event combined with the protocol's own access control logic.

---

### Recommendation

Mirror the external report's fix pattern: exclude `WithdrawCollateral` (and `BurnNlp`) from the `requireUnsanctioned` check inside `submitSlowModeTransactionImpl`, so that users can always reclaim their own funds regardless of sanctions status. Concretely, move the `requireUnsanctioned` call into the branches that cover deposit and fee-charging operations, but skip it when `txType == WithdrawCollateral` or `txType == BurnNlp`. [1](#0-0) 

---

### Proof of Concept

1. Alice calls `Endpoint.depositCollateralWithReferral(subaccount, USDC_PRODUCT_ID, 10_000e6, "")`.
   - `requireUnsanctioned` passes (Alice is not yet sanctioned).
   - 10,000 USDC is transferred to the clearinghouse; a slow mode `DepositCollateral` tx is queued and processed by the sequencer, crediting Alice's subaccount.
2. Alice's address is added to the Chainalysis sanctions list.
3. The sequencer begins censoring Alice's signed `WithdrawCollateral` transactions.
4. Alice encodes a `WithdrawCollateral` transaction and calls `Endpoint.submitSlowModeTransaction(encodedTx)`.
5. Inside `submitSlowModeTransactionImpl` (line 375): `requireUnsanctioned(Alice)` → `sanctions.isSanctioned(Alice) == true` → **revert with `ERR_WALLET_SANCTIONED`**.
6. Alice cannot queue the withdrawal. Her 10,000 USDC remains locked in the clearinghouse indefinitely with no on-chain recovery path. [6](#0-5) [3](#0-2)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
