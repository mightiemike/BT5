Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to the pool's direct `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. If the pool admin allowlists the router to enable router-mediated swaps for permitted users, any unpermitted user can bypass the allowlist entirely by calling the router.

## Finding Description
In `SwapAllowlistExtension.beforeSwap`, the check is:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct), and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap`: [2](#0-1) 

The `sender` value originates from whoever called `pool.swap()`. In `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [3](#0-2) 

This makes the router contract itself the `msg.sender` of `pool.swap()`, so `sender` passed to the extension is the router address, not the end user. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][endUser]`.

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [4](#0-3) 

A pool admin who wants permitted users to use the router has no selective mechanism — the only option is to allowlist the router address itself. Once `allowedSwapper[pool][router] == true`, the check passes for every caller of the router regardless of identity.

## Impact Explanation
Any user blocked by `SwapAllowlistExtension` can bypass the gate by calling `MetricOmmSimpleRouter` instead of the pool directly. The extension sees the router as the swapper and passes the check if the router is allowlisted. The blocked user executes a swap in a pool intended to restrict them. This is a direct break of the pool's access-control invariant and constitutes an admin-boundary break by an unprivileged path, matching the allowed impact gate.

## Likelihood Explanation
The bypass requires the pool admin to have allowlisted the router. This is a natural and expected configuration: a pool admin who wants permitted users to use the standard router must allowlist the router address, as there is no other mechanism to enable router-mediated swaps for specific users. Once the router is allowlisted, the trigger (calling the router) is available to any unprivileged user with no special setup, making this repeatable and low-effort.

## Recommendation
The extension must gate the economic actor (the end user), not the intermediary. Two approaches:

1. **Pass original caller through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Pool-level originating user field**: The pool exposes a separate field for the "originating user" distinct from the direct `msg.sender`, allowing extensions to check the true initiator. This is more invasive but eliminates the ambiguity at the protocol level.

Until resolved, pool admins must not allowlist the router address in `SwapAllowlistExtension`; permitted users must call the pool directly.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` as a `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, allowedUser, true)` — only `allowedUser` should be able to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — intending to let `allowedUser` use the router.
4. `blockedUser` (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. The router calls `pool.swap(...)` with `msg.sender == router`.
6. The pool calls `_beforeSwap(router, ...)`.
7. The extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. `blockedUser`'s swap executes successfully, bypassing the allowlist. [5](#0-4) [6](#0-5)
