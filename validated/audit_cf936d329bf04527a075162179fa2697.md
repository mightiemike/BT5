### Title
Missing `ENABLE_SECP_OPS` Flag Enforcement on 4-Byte Secp Opcode Path — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` unconditionally when the caller supplies a 4-byte opcode (`0x13d61f00` / `0x1c3a8f00`), while the 1-byte opcode path (opcodes 64 / 65) correctly gates the same functions behind `ClvmFlags::ENABLE_SECP_OPS`. The flag check is computed and enforced in one branch but is entirely absent in the other, mirroring the `onlyMinter` pattern where the guard expression exists but is never applied.

---

### Finding Description

`ChiaDialect::op()` contains two separate dispatch branches for secp operators.

**Branch 1 — 1-byte opcodes (correctly gated):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

**Branch 2 — 4-byte opcodes (flag check absent):** [2](#0-1) 

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no flag check
        0x1c3a8f00 => op_secp256r1_verify,   // no flag check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The 4-byte opcode block is entered before the 1-byte block and returns immediately, so the `ENABLE_SECP_OPS` guard on lines 248–249 is never reached for these encodings. The flag is defined and documented as the hard-fork gate for secp operators: [3](#0-2) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The broken invariant is: *secp verification operators must not execute unless `ENABLE_SECP_OPS` is set*. The 4-byte path violates this invariant unconditionally.

---

### Impact Explanation

Any caller that constructs a CLVM program with a 4-byte atom `0x13d61f00` or `0x1c3a8f00` as the operator can invoke secp signature verification regardless of the dialect flags passed to `run_program`. This produces a **consensus divergence**: a node running without `ENABLE_SECP_OPS` (pre-hard-fork consensus mode) will still execute secp verification via the 4-byte path, while the intended behavior is to treat those opcodes as unknown operators (returning nil or erroring under `NO_UNKNOWN_OPS`). A spend bundle that should be rejected pre-fork is instead accepted, breaking the fork boundary.

The corrupted result is the `Reduction(cost, a.nil())` returned by `op_secp256k1_verify` / `op_secp256r1_verify` when the flag is absent but the 4-byte opcode is used — a successful execution where the consensus rules require a no-op or rejection.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the CLVM program bytes are supplied externally (e.g., via a coin spend). Constructing a 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]` as the operator in a CLVM program is trivial. No special privileges, keys, or social engineering are required. Any node that evaluates attacker-submitted CLVM programs (mempool validation, block validation) is reachable.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch block, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths enforce the same hard-fork gate.

---

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags};
use clvmr::run_program::run_program;

// Build a CLVM program: (0x13d61f00 pubkey msg sig)
// using the 4-byte secp256k1_verify opcode directly,
// WITHOUT setting ENABLE_SECP_OPS.
fn main() {
    let mut alloc = Allocator::new();

    // 4-byte opcode atom: secp256k1_verify via assigned-cost encoding
    let op = alloc.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();

    // valid pubkey / msg / sig atoms (33 + 32 + 64 bytes)
    let pubkey = alloc.new_atom(&[/* valid compressed secp256k1 pubkey */]).unwrap();
    let msg    = alloc.new_atom(&[0u8; 32]).unwrap();
    let sig    = alloc.new_atom(&[/* valid DER/compact sig */]).unwrap();

    let args = /* build list (pubkey msg sig) */;
    let program = alloc.new_pair(op, args).unwrap();

    // No ENABLE_SECP_OPS flag — secp ops should be unavailable
    let dialect = ChiaDialect::new(ClvmFlags::NO_UNKNOWN_OPS);

    // Expected: error (unimplemented operator or unknown op)
    // Actual:   Ok(Reduction(...)) — secp verification executes anyway
    let result = run_program(&mut alloc, &dialect, program, alloc.nil(), 10_000_000);
    assert!(result.is_err(), "secp op executed without ENABLE_SECP_OPS via 4-byte path");
}
```

The assertion fails because `op_secp256k1_verify` is dispatched through the 4-byte branch at [4](#0-3) 

before the `ENABLE_SECP_OPS` guard at [1](#0-0) 

is ever evaluated.
