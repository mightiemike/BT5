### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without Providing USDC — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` directly through the interface without using the project's own `ERC20Helper.safeTransferFrom` wrapper. The return value is never checked. If the `transferFrom` call returns `false` (indicating failure), execution continues and the function proceeds to withdraw usdcE from the DDA and send it to `msg.sender` — without the caller having actually provided any USDC.

---

### Finding Description

`ContractOwner.sol` defines a `replaceUsdcEWithUsdc` function callable by any external address on chain 57073. Its purpose is to let a caller swap usdcE held in a Direct Deposit Account (DDA) for USDC: the caller provides USDC, the DDA's usdcE is withdrawn and returned to the caller.

The critical transfer step at line 616 is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

This is a raw interface call — not routed through `ERC20Helper.safeTransferFrom`. The return value (`bool`) is silently discarded. The rest of the function then unconditionally executes:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

If `transferFrom` returns `false` (e.g., insufficient allowance, token-level rejection, or a non-standard ERC20 that does not revert on failure), the caller receives `balance` worth of usdcE from the DDA while having transferred zero USDC.

Contrast this with every other transfer site in the codebase, which consistently uses `ERC20Helper.safeTransferFrom` / `ERC20Helper.safeTransfer` — a low-level-call wrapper that checks both `success` and the decoded return value:

```solidity
// ERC20Helper.sol lines 23-42
(bool success, bytes memory data) = address(self).call(...);
require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
```

`replaceUsdcEWithUsdc` is the sole production transfer site that bypasses this wrapper.

---

### Impact Explanation

An attacker who has zero USDC allowance (or whose `transferFrom` returns `false` for any reason) can call `replaceUsdcEWithUsdc` for any DDA that holds usdcE. The function will:

1. Silently skip the USDC collection step.
2. Withdraw all usdcE from the DDA via `DirectDepositV1.withdraw`.
3. Transfer that usdcE to the attacker via `safeTransfer`.

The corrupted asset delta is: usdcE balance of the targeted DDA is reduced to zero; the attacker's usdcE balance increases by `balance`; the DDA receives no USDC in return. The DDA owner's deposited collateral is permanently lost.

---

### Likelihood Explanation

The function is `external` with no access-control modifier beyond the chain-id check (`block.chainid == 57073`). Any address on that chain can call it for any registered DDA. The attacker only needs to:

1. Identify a DDA with a non-zero usdcE balance.
2. Call `replaceUsdcEWithUsdc(subaccount)` without having approved any USDC.

If the USDC token at the hardcoded address returns `false` on a failed `transferFrom` (rather than reverting), the exploit is trivially executable. Standard Circle USDC reverts on failure, which would prevent silent drain — but the code makes no guarantee about which USDC deployment is at that address, and the pattern is architecturally broken regardless.

---

### Recommendation

Replace the raw interface call with the project's existing safe wrapper:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
// where safeTransferFrom is ERC20Helper.safeTransferFrom, already imported and used elsewhere
```

This ensures that a failed or non-bool-returning `transferFrom` always reverts, preventing the usdcE withdrawal from proceeding.

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
