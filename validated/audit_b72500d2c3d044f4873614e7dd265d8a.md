### Title
Read-Only Reentrancy via ERC-777 Callback in `withdrawCollateral` — Balance Updated After Token Transfer - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to the recipient **before** decrementing the subaccount's balance in `SpotEngine` storage. With an ERC-777 token, the recipient's `tokensReceived` hook fires during the transfer. At that moment, `SpotEngine.getBalance` and `Clearinghouse.getHealth` return the pre-withdrawal (inflated) balance, creating a read-only reentrancy window where any external observer or integrating contract reads a stale, inconsistent state.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

```
handleWithdrawTransfer(token, sendTo, amount, idx);   // line 408 — token transfer FIRST
...
spotEngine.updateBalance(productId, sender, amountRealized);  // line 412 — storage update AFTER
spotEngine.assertUtilization(productId);              // line 413
```

`handleWithdrawTransfer` (lines 377–385) calls `token.safeTransfer(withdrawPool, amount)` and then `BaseWithdrawPool(withdrawPool).submitWithdrawal(...)`, which in turn calls `token.safeTransfer(sendTo, amount)` via `BaseWithdrawPool.handleWithdrawTransfer` (line 189). This final `safeTransfer` to `sendTo` triggers the ERC-777 `tokensReceived` hook on the recipient contract.

During that hook, `SpotEngine.getBalance` (lines 246–254 of `SpotEngineState.sol`) reads directly from `balances[productId][subaccount]`, which has **not yet been decremented**. The same applies to `Clearinghouse.getHealth`, which calls `spotEngine.getHealthContribution` using the same stale storage. The subaccount appears to still hold the full pre-withdrawal collateral.

The `sendTo` address is attacker-controlled: in `WithdrawCollateralV2`, `signedTx.tx.sendTo` is an explicit user-supplied field. In `WithdrawCollateral` (V1), it resolves to `address(uint160(bytes20(signedTx.tx.sender)))`, which is the user's own address — valid if the user operates from a contract wallet implementing `IERC777Recipient`.

---

### Impact Explanation

Any external protocol that integrates with Nado and reads `SpotEngine.getBalance(productId, subaccount)` or `Clearinghouse.getHealth(subaccount, healthType)` as part of a collateral or solvency check will observe an inflated balance during the ERC-777 callback window. This can cause:

- **Overcollateralization illusion**: An integrating lending protocol reads the subaccount's balance as if the withdrawal never happened, allowing the attacker to borrow against collateral that has already been withdrawn.
- **Liquidation bypass**: A liquidation bot or contract that checks health before acting sees a healthy subaccount that is actually undercollateralized post-withdrawal.
- **Accounting desynchronization**: Any protocol that caches or acts on Nado's view functions during this window stores a value that is immediately stale once the callback returns.

The corrupted state is the subaccount's spot balance in `SpotEngine.balances[productId][subaccount]` — it reads as `X` when the true post-withdrawal value is `X - amountRealized`.

---

### Likelihood Explanation

- The attacker only needs to submit a valid signed `WithdrawCollateralV2` transaction with `sendTo` pointing to a contract they control that implements `IERC777Recipient`.
- No admin access, sequencer compromise, or governance capture is required — this is a standard user-initiated withdrawal.
- The precondition is that the withdrawn token is ERC-777 compatible. ERC-777 tokens are backward-compatible with ERC-20 and can be listed as spot products.
- The window is narrow (one call frame), but it is deterministic and fully attacker-controlled.

---

### Recommendation

Update the subaccount's balance in `SpotEngine` storage **before** calling `handleWithdrawTransfer`. The corrected order in `Clearinghouse.withdrawCollateral` should be:

1. Compute `amountRealized` and call `spotEngine.updateBalance(productId, sender, amountRealized)`.
2. Call `spotEngine.assertUtilization(productId)`.
3. Check `getHealth(sender, healthType) >= 0`.
4. Only then call `handleWithdrawTransfer(token, sendTo, amount, idx)`.

This mirrors the checks-effects-interactions pattern and ensures that any reentrant read of `SpotEngine.getBalance` or `Clearinghouse.getHealth` during the token transfer reflects the post-withdrawal state.

### Proof of Concept

**Root cause — transfer before storage update:** [1](#0-0) 

**Token transfer chain that triggers ERC-777 callback on `sendTo`:** [2](#0-1) [3](#0-2) 

**View function that returns stale balance during the callback window:** [4](#0-3) 

**Attack path — user-controlled `sendTo` in V2 withdrawal:** [5](#0-4) 

**Step-by-step:**

1. Attacker deploys `MaliciousRecipient` implementing `IERC777Recipient.tokensReceived`. Inside the hook, it calls `SpotEngine.getBalance(productId, victimSubaccount)` and records the result, or calls into an integrating protocol that uses this value for a credit decision.
2. Attacker submits a signed `WithdrawCollateralV2` transaction with `sendTo = address(MaliciousRecipient)`.
3. Sequencer processes it; `Clearinghouse.withdrawCollateral` executes.
4. `handleWithdrawTransfer` → `withdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousRecipient, amount)` fires `tokensReceived`.
5. Inside `tokensReceived`: `SpotEngine.getBalance` returns the **pre-withdrawal** balance (storage not yet updated). Any integrator acting on this value makes decisions based on inflated collateral.
6. Callback returns; `spotEngine.updateBalance` finally decrements the balance — too late for any integrator that already acted.
