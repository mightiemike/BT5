Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Supplied `owner` Instead of Actual `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and enforces the allowlist only against `owner`, a free caller-supplied argument. Because any address can pass an allowlisted address as `owner`, the allowlist is fully bypassed. The sibling `SwapAllowlistExtension` correctly gates on `sender`, confirming the asymmetry is a defect.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as separate arguments into the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops `sender` (unnamed first argument) and enforces the allowlist only on `owner`: [3](#0-2) 

`addLiquidity` has no `msg.sender == owner` guard (unlike `removeLiquidity` which enforces `if (msg.sender != owner) revert NotPositionOwner()`), so any caller can freely supply any `owner` value: [4](#0-3) 

`SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [5](#0-4) 

The `isAllowedToDeposit` view names its second parameter `depositor`, confirming the intended semantic is to gate the actual depositing address, not the position beneficiary: [6](#0-5) 

## Impact Explanation

Any address not on the allowlist can call `pool.addLiquidity(allowlisted_address, salt, deltas, callbackData, "")`. The hook evaluates `allowedDepositor[pool][allowlisted_address]` → `true` → no revert. The caller satisfies the token-transfer callback and the position is minted. The pool admin's access-control boundary is fully bypassed: the allowlist no longer restricts who can deposit into the pool. This is an admin-boundary break where an unprivileged path circumvents an admin-configured guard, constituting a broken core pool functionality impact.

## Likelihood Explanation

The attack requires only knowing one allowlisted address (observable on-chain from `setAllowedToDeposit` events or prior successful deposits) and calling `addLiquidity` with that address as `owner` while providing the required tokens via the callback. No special privileges, flash loans, or oracle manipulation are needed. Any EOA or contract can execute this in a single transaction, making exploitation trivial and repeatable.

## Recommendation

Name the first argument `sender` and enforce the allowlist against it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` and leaves `bob` unlisted.
3. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. `beforeAddLiquidity` is invoked with `sender = bob` (ignored), `owner = alice`.
5. Check evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. `bob` satisfies the token callback; position is minted under `alice`.
7. `bob` has deposited into the allowlist-gated pool despite being explicitly excluded.
