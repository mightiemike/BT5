### Title
Inconsistent `public_key_len` Representation in Gas Fee Calculation Between Transaction Path and Host-Function Path for `TransferToGasKey` — (File: `runtime/runtime/src/config.rs`, `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The `TransferToGasKey` action computes its per-byte gas fee using two different public-key length representations depending on whether the action is submitted as a transaction or invoked via the `promise_batch_action_transfer_to_gas_key` host function. The transaction path uses `PublicKey::trie_id_len()` (33 bytes for ML-DSA-65), while the host-function path uses the raw borsh-encoded wire length (`public_key_len as usize`, 1953 bytes for ML-DSA-65). This creates a ~59× divergence in the per-byte fee component for ML-DSA-65 keys across the two entry points for the same on-chain action.

### Finding Description

**Transaction path** — `total_send_fees` and `exec_fee` in `runtime/runtime/src/config.rs`:

```rust
// send fee
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
// exec fee
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
```

Both calls pass `trie_id_len()` = **33 bytes** for an ML-DSA-65 key. [1](#0-0) [2](#0-1) 

**Host-function path** — `promise_batch_action_transfer_to_gas_key` in `runtime/near-vm-runner/src/logic/logic.rs`:

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // raw borsh wire length
);
```

Both calls pass `public_key_len as usize` = **1953 bytes** for an ML-DSA-65 key (the full borsh-encoded pubkey the contract supplies). [3](#0-2) 

The two fee helpers are defined in `core/parameters/src/cost.rs`:

- `gas_key_transfer_send_fee` — comment: *"Based on the public key length (what the sender sees)."* Correct value is the wire length (1953). Transaction path passes 33 — **undercharges send fee ~59×**.
- `gas_key_transfer_exec_fee` — comment: *"Based on the access key trie key length + estimated value length (what the receiver needs to read/write in the trie)."* Correct value is the trie-id length (33). Host-function path passes 1953 — **overcharges exec fee ~59×**. [4](#0-3) 

The divergence exists because `PublicKey::trie_id_len()` and `PublicKey::len()` are identical for ED25519 (33) and SECP256K1 (65), but diverge for ML-DSA-65 (33 vs 1953): [5](#0-4) 

The `PostQuantumSignatures` protocol feature gates ML-DSA-65 key admission into state, so the divergence is latent until that feature activates. [6](#0-5) 

The parity test `test_gas_key_fee_parity` only exercises ED25519 keys and therefore does not catch this discrepancy: [7](#0-6) 

### Impact Explanation

**Transaction path (undercharges send fee):** A user submitting a `TransferToGasKey` transaction with an ML-DSA-65 key pays the per-byte send fee on 33 bytes instead of 1953 bytes — a ~59× undercharge on that component. This allows the action to be submitted at a gas cost far below what the fee model intends, breaking the economic invariant that gas prices reflect actual resource consumption.

**Host-function path (overcharges exec fee):** A contract calling `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 key is charged the per-byte exec fee on 1953 bytes instead of 33 bytes — a ~59× overcharge. This makes the host function economically unusable for ML-DSA-65 gas keys from contracts, creating a capability asymmetry between the two entry points for the same action.

### Likelihood Explanation

Likelihood is **Medium** once `PostQuantumSignatures` activates. Any unprivileged user can craft a `TransferToGasKey` transaction with an ML-DSA-65 key and observe the underpriced send fee. No special role or privileged access is required. The discrepancy is structural and deterministic — it will manifest on every such transaction.

### Recommendation

1. In `total_send_fees` (`runtime/runtime/src/config.rs`), change `action.public_key.trie_id_len()` to `action.public_key.len()` for the `TransferToGasKey` send-fee call, so the send fee reflects the wire length the sender supplies.
2. In `promise_batch_action_transfer_to_gas_key` (`runtime/near-vm-runner/src/logic/logic.rs`), decode the public key first (as is already done for `public_key.decode()?` at line 3114), then pass `public_key.trie_id_len()` to `gas_key_transfer_exec_fee` so the exec fee reflects the actual trie-key length.
3. Add a parity test that exercises ML-DSA-65 keys (in addition to ED25519) in `test_gas_key_fee_parity` to catch future regressions.

### Proof of Concept

1. Enable `PostQuantumSignatures` (nightly build).
2. Generate an ML-DSA-65 key pair.
3. Submit a `TransferToGasKey` transaction referencing the ML-DSA-65 public key. Record `gas_burnt`.
4. From a contract, call `promise_batch_action_transfer_to_gas_key` with the same ML-DSA-65 public key (borsh-encoded, 1953 bytes). Record `gas_burnt` on the execution receipt.
5. Observe that the per-byte fee component differs by a factor of ~59 (1953 / 33) between the two paths, confirming the divergence.

The exact divergent values are: **33 bytes** (`trie_id_len()` for ML-DSA-65, used by the transaction path) vs **1953 bytes** (borsh wire length for ML-DSA-65, used by the host-function path).

/// Send fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the public key length (what the sender sees).
pub fn gas_key_transfer_send_fee(
    cfg: &RuntimeFeesConfig,
    sender_is_receiver: bool,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).send_fee(sender_is_receiver);
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .send_fee(sender_is_receiver)
        .checked_mul(public_key_len as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }

### 5. Storage usage and fee plumbing

The storage-stake calculation
(`runtime/runtime/src/access_keys.rs::access_key_storage_usage`) and the
gas-key fee helpers (`gas_key_*_fee` in `runtime/runtime/src/config.rs`) use
`PublicKey::trie_id_len()` rather than `PublicKey::len()`:

- `len()` reports the borsh-encoded length (33 / 65 / 1953 across the
  three `PublicKey` variants).
- `trie_id_len()` reports the on-trie length (33 / 65 / **33**).

The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and
trie-byte-priced fee path was updated to call `trie_id_len()`.
