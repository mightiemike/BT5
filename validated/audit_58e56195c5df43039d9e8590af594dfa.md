### Title
`ValidResourceBounds` Protobuf Deserializer Uses Value-Based Heuristic to Select Hash Preimage Variant, Producing Wrong Transaction Hash for `AllResources` Transactions with Zero L2/L1-Data Gas - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — checking whether `l1_data_gas` and `l2_gas` are both zero — to decide between the `L1Gas` (pre-0.13.3) and `AllResources` (0.13.3+) variants. A V3 transaction legitimately submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is silently downgraded to `L1Gas(X)` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces a structurally different Poseidon preimage for each variant (2 vs 3 resource entries), the transaction hash computed before serialization diverges from the hash recomputed after deserialization, breaking hash canonicalization across the block-sync boundary.

---

### Finding Description

**Step 1 — Origination (gateway/sequencer side)**

When a user submits a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the gateway converts it to `InternalRpcInvokeTransactionV3`. Its `InvokeTransactionV3Trait::resource_bounds()` implementation always returns `ValidResourceBounds::AllResources(self.resource_bounds)`: [1](#0-0) 

The hash is then computed via `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash`: [2](#0-1) 

For `AllResources`, the function builds a 3-element resource vector `[L1_GAS_packed, L2_GAS_packed(=0), L1_DATA_GAS_packed(=0)]` and hashes it together with `tip`. Call this hash **H₁**.

**Step 2 — Protobuf serialization (correct)**

`ValidResourceBounds::AllResources` serializes all three fields as `Some(...)`: [3](#0-2) 

**Step 3 — Protobuf deserialization (broken)**

On the receiving side (block-sync peer), the deserializer applies a value-based heuristic: [4](#0-3) 

Because `l1_data_gas` is `Some(zero)` → `unwrap_or_default()` → zero, and `l2_gas` is also zero, the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is `true`. The deserializer returns `ValidResourceBounds::L1Gas(l1_gas)` — the pre-0.13.3 variant — even though the transaction was originally an `AllResources` V3 transaction.

The comment at line 426 acknowledges this is a temporary backward-compatibility measure: [5](#0-4) 

**Step 4 — Hash recomputation produces H₂ ≠ H₁**

When the peer recomputes the hash using the deserialized `L1Gas` variant, `get_tip_resource_bounds_hash` takes the `L1Gas` branch: [6](#0-5) 

For `L1Gas`, the resource vector has only 2 elements `[L1_GAS_packed, L2_GAS_packed(=0)]` — the `L1_DATA_GAS_packed(=0)` entry is absent. The resulting Poseidon hash **H₂ ≠ H₁**.

**Step 5 — Hash validation fails**

`validate_transaction_hash` computes the canonical hash from the deserialized transaction and checks it against the expected hash stored in the block: [7](#0-6) 

For V3 invoke transactions, `get_deprecated_transaction_hashes` returns an empty list (no deprecated hashes for V3 invoke). The only candidate is H₂, which does not equal H₁. Validation fails.

The `ValidResourceBounds` enum documents the version boundary explicitly: [8](#0-7) 

---

### Impact Explanation

A V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` — a valid user choice — has its `ValidResourceBounds` variant silently changed from `AllResources` to `L1Gas` during protobuf deserialization. This produces a different Poseidon hash preimage (3 vs 2 resource entries), causing the recomputed hash to diverge from the original. Any component that recomputes the hash after deserialization (block-sync validation, RPC `starknet_getTransactionByHash`, fee estimation replay) will produce or serve an authoritative-looking wrong hash value. This matches: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** and **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The trigger is a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0`. The gateway's stateless validator permits zero resource bounds for individual resources. Any user who submits a V3 transaction relying solely on L1 gas (a common pattern during the transition from pre-0.13.3 to 0.13.3+) can trigger this. The protobuf round-trip occurs on every block-sync peer that processes the block containing such a transaction.

---

### Recommendation

Replace the value-based heuristic with an explicit version/type discriminator. The protobuf `ResourceBounds` message should carry a boolean or enum field indicating whether the transaction is `L1Gas`-only or `AllResources`. Until the protobuf schema is updated, the deserializer should default to `AllResources` (not `L1Gas`) when `l1_data_gas` is absent, since `AllResources` with zero fields is the correct representation for a V3 transaction that happens to set those bounds to zero:

```rust
// Instead of:
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(...)
})

// Use:
Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
    // Only downgrade to L1Gas if l1_data_gas was truly absent (0.13.2 wire format)
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer already sets `l1_data_gas: Some(...)` for both variants, so the `None` check correctly identifies legacy 0.13.2 messages.

---

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: GasAmount(100)/GasPrice(1), l2_gas: 0/0, l1_data_gas: 0/0 }`.
2. The gateway computes hash H₁ using `AllResources` path: `Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)])`.
3. The transaction is included in a block and propagated to a syncing peer via block-sync protobuf.
4. The peer deserializes: `l1_data_gas = unwrap_or_default() = 0`, `l2_gas = 0` → `ValidResourceBounds::L1Gas(l1_gas)`.
5. The peer recomputes hash H₂ using `L1Gas` path: `Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)])`.
6. H₁ ≠ H₂ — `validate_transaction_hash` returns `false` for the transaction, causing block-sync failure or wrong hash storage.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
