### Title
Missing Deadline and Oracle Price Commitment in `MintNlp`/`BurnNlp` Signed Digest Allows Stale Execution at Unfavorable NLP Price â€” (`File: core/contracts/Verifier.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The EIP-712 digests for `MintNlp` and `BurnNlp` signed transactions omit both a deadline and the `oraclePriceX18` that directly determines how many NLP tokens the user receives (or how much quote they get back). A user who signs a `MintNlp` or `BurnNlp` transaction has no on-chain protection against execution at a stale oracle price, and cannot cancel the pending signature because the nonce is only consumed at execution time.

---

### Finding Description

In `Verifier.sol`, the type strings and digest computation for `MintNlp` and `BurnNlp` are:

```
MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)
BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)
``` [1](#0-0) 

Neither type string includes a deadline field. The `computeDigest` function encodes only `{sender, quoteAmount/nlpAmount, nonce}`: [2](#0-1) 

However, the `SignedMintNlp` and `SignedBurnNlp` structs carry an `oraclePriceX18` field that is **not** part of the user's signed digest â€” it is provided by the sequencer at execution time: [3](#0-2) 

In `Clearinghouse.mintNlp`, the NLP amount received is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

And in `Clearinghouse.burnNlp`, the quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [5](#0-4) 

The sequencer passes `signedTx.oraclePriceX18` directly into both calls without any on-chain staleness check: [6](#0-5) [7](#0-6) 

The nonce mechanism only prevents replay â€” it does not prevent delayed execution. `validateNonce` increments the nonce at execution time, so the user cannot cancel a pending signed transaction by pre-incrementing their nonce: [8](#0-7) 

---

### Impact Explanation

**`MintNlp`:** A user signs `{sender, quoteAmount=X, nonce=N}` intending to receive `X / P_current` NLP tokens at the current oracle price `P_current`. If the sequencer delays execution until the oracle price rises to `P_stale > P_current`, the user receives `X / P_stale` NLP tokens â€” a direct, measurable reduction in NLP tokens received. For example, if `P_current = 1.00` and `P_stale = 2.00`, the user receives 50% fewer NLP tokens than expected.

**`BurnNlp`:** A user signs `{sender, nlpAmount=Y, nonce=N}` intending to receive `Y * P_current` quote tokens. If the oracle price falls to `P_stale < P_current` at execution time, the user receives `Y * P_stale` quote â€” a direct loss of quote tokens. The burn fee (`max(1, quoteAmount / 1000)`) compounds the loss.

The corrupted state delta is: `spotEngine.balance[NLP_PRODUCT_ID][sender]` and `spotEngine.balance[QUOTE_PRODUCT_ID][sender]` are set to values the user never agreed to when signing.

---

### Likelihood Explanation

The sequencer is a trusted but not infallible entity. Sequencer backlogs, network congestion, or deliberate ordering decisions can cause `MintNlp`/`BurnNlp` transactions to be executed significantly after the user signed them. NLP oracle prices are volatile by nature (they reflect the value of the underlying liquidity pool). There is no on-chain mechanism for the user to cancel a pending signed transaction or bound the oracle price at which their transaction executes. Any user who has submitted a `MintNlp` or `BurnNlp` request during a period of oracle price movement is affected.

---

### Recommendation

1. Add a `uint64 deadline` field to both `MintNlp` and `BurnNlp` structs in `IEndpoint.sol`.
2. Include `deadline` in the EIP-712 type strings and `computeDigest` encoding in `Verifier.sol`.
3. In `EndpointTx.processTransactionImpl`, check `block.timestamp <= signedTx.tx.deadline` before calling `validateSignedTx`.
4. Optionally, include a user-specified `minNlpAmount` (for mint) or `minQuoteAmount` (for burn) in the signed digest to bound the acceptable oracle price range, analogous to slippage protection in AMMs.

```solidity
// Verifier.sol
string internal constant MINT_NLP_SIGNATURE =
    "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce,uint64 deadline)";

// EndpointTx.sol (MintNlp branch)
require(block.timestamp <= signedTx.tx.deadline, ERR_SIGNATURE_EXPIRED);
validateSignedTx(...);
```

### Proof of Concept

1. User signs `MintNlp{sender=Alice, quoteAmount=10_000e18, nonce=0}` when NLP oracle price is `1e18` (1 USDC/NLP). User expects to receive 10,000 NLP tokens.
2. User submits the signed transaction to the sequencer off-chain.
3. Sequencer experiences a delay (or intentionally defers the transaction).
4. NLP oracle price rises to `2e18` (2 USDC/NLP).
5. Sequencer calls `processTransaction` with `signedTx.oraclePriceX18 = 2e18`.
6. `validateSignedTx` passes â€” the digest only covers `{sender, quoteAmount, nonce}`, all of which are unchanged.
7. `Clearinghouse.mintNlp` computes `nlpAmount = 10_000e18 / 2e18 = 5_000` NLP.
8. Alice receives 5,000 NLP instead of 10,000 NLP â€” a 50% loss â€” with no on-chain recourse. [9](#0-8) [10](#0-9)
