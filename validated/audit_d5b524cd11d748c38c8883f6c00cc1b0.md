### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the Coinbase Merkle Proof Guard Added in `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a fully public NEAR contract method despite being marked `#[deprecated]`. Any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase Merkle proof check that `verify_transaction_inclusion_v2` was introduced to enforce. This is a direct structural analog to M-07: a guard exists in one entry point but is absent in a second, alternative entry point that reaches the same verification logic.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented in the code's own warning comment). It does so by requiring the caller to also supply a valid Merkle proof for the coinbase transaction (always at index 0), which an attacker cannot forge: [1](#0-0) 

The original `verify_transaction_inclusion` has no such coinbase guard. It only checks that the `merkle_proof` field is non-empty, then directly computes the Merkle root and compares it to the stored block header: [2](#0-1) 

The `#[deprecated]` Rust attribute is a **compile-time lint only**. It does not restrict on-chain access. The function remains `pub` and is exposed as a callable NEAR contract method: [3](#0-2) 

`verify_transaction_inclusion_v2` does **not** call `verify_transaction_inclusion` as a sub-step that would inherit the coinbase guard — it calls `verify_transaction_inclusion` only after its own coinbase check passes. But the reverse path — calling `verify_transaction_inclusion` directly — skips the coinbase check entirely: [4](#0-3) 

---

### Impact Explanation

The 64-byte transaction attack allows an adversary to construct a 64-byte blob that is simultaneously a valid serialized internal Merkle tree node and a plausible "transaction hash." By supplying this blob as `tx_id` along with a crafted `merkle_proof`, the attacker can make `compute_root_from_merkle_proof` return the genuine `merkle_root` of a real Bitcoin block — causing `verify_transaction_inclusion` to return `true` for a transaction that never existed on-chain.

Any NEAR DApp that calls `verify_transaction_inclusion` (rather than `verify_transaction_inclusion_v2`) — whether by choice, by dependency on an older SDK binding, or because the attacker instructs it to — receives a forged `true` result. This corrupts the proof-verification guarantee that is the contract's sole security primitive. [5](#0-4) 

---

### Likelihood Explanation

The entry point is reachable by any unprivileged NEAR account with no staking, no role, and no special permission. The `#[pause]` guard can be bypassed only if the contract is unpaused, which is the normal operating state. The 64-byte transaction attack technique is publicly documented (referenced in the contract's own comments). A recipient DApp that was written against an older version of the contract ABI, or that explicitly calls the v1 method, is the realistic trigger. [6](#0-5) 

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or replace it with `pub(crate)` / make it `#[private]` so it is no longer callable as an external NEAR contract method. All external callers must be forced through `verify_transaction_inclusion_v2`. Alternatively, inline the coinbase guard directly into `verify_transaction_inclusion` and remove the separate v2 entry point. [7](#0-6) 

---

### Proof of Concept

1. Identify a real Bitcoin block `B` with known `merkle_root`.
2. Construct a 64-byte blob `fake_tx` such that `compute_root_from_merkle_proof(fake_tx, idx, crafted_proof) == merkle_root` of block `B` (standard 64-byte Merkle forgery technique).
3. Call `verify_transaction_inclusion` directly on the NEAR contract with:
   - `tx_id = fake_tx`
   - `tx_block_blockhash = B`
   - `tx_index = idx`
   - `merkle_proof = crafted_proof` (non-empty, so the `is_empty` guard passes)
   - `confirmations = 1`
4. The function returns `true` — a forged proof accepted — because no coinbase cross-check is performed.
5. Calling `verify_transaction_inclusion_v2` with the same inputs (and no valid `coinbase_merkle_proof`) would fail at the coinbase guard and return an error. [8](#0-7) [9](#0-8)

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
