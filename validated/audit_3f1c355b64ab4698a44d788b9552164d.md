### Title
`ERC20Helper.safeTransferFrom` Passes Silently for Tokens Returning No Data Without Transferring, Enabling Uncollateralized Balance Credits — (File: `core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransferFrom` uses a `data.length == 0` branch to accommodate non-standard ERC20 tokens that omit a boolean return value. This branch also passes unconditionally for any token whose `transferFrom` call succeeds (no revert), returns no data, but performs no actual token movement. In the deposit path (`Endpoint.depositCollateralWithReferral` → `handleDepositTransfer`), this allows a slow-mode deposit transaction to be queued and later credited to a user's spot balance even though the Clearinghouse received zero tokens, breaking the protocol's core collateral invariant.

---

### Finding Description

`ERC20Helper.safeTransferFrom` is the sole transfer-success guard used throughout the deposit path:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) 

The `data.length == 0` arm is intentionally designed to handle tokens like USDT that do not return a boolean. However, it is structurally indistinguishable from a token whose `transferFrom`:

1. Does not revert (`success = true`)
2. Returns no ABI-encoded data (`data.length == 0`)
3. Moves zero tokens

No balance-before/balance-after check exists anywhere in the deposit path to detect this case.

`Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer`, which calls `safeTransferFrom` and then `safeTransferTo`:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
``` [2](#0-1) 

`EndpointStorage.handleDepositTransfer` performs both legs of the transfer with no receipt verification:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
}
``` [3](#0-2) 

Immediately after, a slow-mode `DepositCollateral` transaction is unconditionally enqueued:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(DepositCollateral({ sender: subaccount, productId: productId, amount: amount }))
    )
});
``` [4](#0-3) 

When the sequencer (or the user after the delay) executes this slow-mode transaction, `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance` to credit the full `amount` to the subaccount:

```solidity
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
``` [5](#0-4) 

The credit is applied regardless of whether the Clearinghouse actually holds the corresponding tokens.

---

### Impact Explanation

The broken invariant is: **a subaccount's spot balance must be backed by tokens physically held in the Clearinghouse**. If `safeTransferFrom` silently no-ops, the Clearinghouse holds fewer tokens than the sum of credited balances. The attacker can then:

- Trade against other users using phantom collateral, socializing losses onto the protocol
- Withdraw real tokens held by the Clearinghouse (backed by other users' legitimate deposits), draining protocol reserves
- Trigger `assertUtilization` failures for legitimate withdrawals by other users, since `totalDeposits` in the SpotEngine exceeds the actual on-chain token balance

The corrupted state is the `totalDepositsNormalized` field in `SpotEngine.states[productId]` and the individual `balances[productId][subaccount]` entry, both of which are inflated without a corresponding on-chain token receipt.

---

### Likelihood Explanation

The trigger requires a supported product token whose `transferFrom`:
- Does not revert on call
- Returns empty calldata (no boolean)
- Performs no token movement

Concrete realistic cases:
1. A token with a "soft pause" mechanism that silently no-ops transfers instead of reverting (some older or non-standard tokens exhibit this)
2. A token with a fee-exemption or blacklist path that silently skips the transfer for certain callers without reverting
3. A malicious or misconfigured token listed as a supported product

Products are added via `SpotEngine.addOrUpdateProduct` (`onlyOwner`), so the token must be admin-listed. However, the admin may not anticipate that a legitimate-looking token has a silent-failure transfer path, and the vulnerability is exploitable by any unprivileged user once such a token is listed.

---

### Recommendation

Add a balance-before/balance-after check in `handleDepositTransfer` to verify that the Clearinghouse actually received the declared amount:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    require(
        token.balanceOf(address(clearinghouse)) - before >= amount,
        ERR_TRANSFER_FAILED
    );
}
```

This mirrors the mitigation recommended in the original Infinity report and closes the silent-no-op path regardless of the token's return-value behavior.

---

### Proof of Concept

1. Deploy `SilentToken`: an ERC20 whose `transferFrom` executes `return` with no data and moves no tokens (assembly `return(0,0)`).
2. Admin calls `SpotEngine.addOrUpdateProduct` listing `SilentToken` as a supported product.
3. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, productId, 1_000_000e18, "")`.
4. `handleDepositTransfer` calls `ERC20Helper.safeTransferFrom(SilentToken, attacker, endpoint, 1_000_000e18)`. The call succeeds, returns no data; `data.length == 0` → `require` passes. No tokens move.
5. `safeTransferTo` similarly no-ops. Clearinghouse token balance is unchanged.
6. A `DepositCollateral` slow-mode tx for `1_000_000e18` is enqueued.
7. After `SLOW_MODE_TX_DELAY`, attacker calls `executeSlowModeTransaction`. `Clearinghouse.depositCollateral` credits `1_000_000e18 * multiplier` to the attacker's subaccount via `spotEngine.updateBalance`.
8. Attacker now holds a fully credited spot balance backed by zero real tokens and can withdraw legitimate collateral deposited by other users.
