### Title
`WithdrawCollateral` (V1) Fast Withdrawal Hardcodes `sendTo` to Subaccount-Embedded Address, Risking Token Loss for Smart Contract Wallet Depositors — (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.resolveFastWithdrawal()`, when processing a legacy `WithdrawCollateral` (V1) transaction, the withdrawal destination is unconditionally derived as `address(uint160(bytes20(signedTx.tx.sender)))` — the address embedded in the subaccount identifier. The V1 struct has no `sendTo` field, so the user cannot specify a different recipient. If the subaccount was opened by a smart contract wallet (e.g., a Gnosis Safe multisig), the collateral will always be sent back to that contract address. If the contract cannot receive ERC20 tokens, the funds are permanently locked. In the worst case — if the contract was deployed via `CREATE2` and has since been self-destructed — an attacker can redeploy at the same address and drain the withdrawal.

---

### Finding Description

`BaseWithdrawPool.resolveFastWithdrawal()` handles two withdrawal transaction types. For `WithdrawCollateral` (V1):

```solidity
if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.SignedWithdrawCollateral memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedWithdrawCollateral)
    );
    return (
        signedTx.tx.productId,
        address(uint160(bytes20(signedTx.tx.sender))), // hardcoded to subaccount address
        signedTx.tx.amount
    );
}
``` [1](#0-0) 

The `WithdrawCollateral` struct has no `sendTo` field:

```solidity
struct WithdrawCollateral {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
}
``` [2](#0-1) 

By contrast, `WithdrawCollateralV2` introduced an explicit `sendTo` field:

```solidity
struct WithdrawCollateralV2 {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
    address sendTo;
    uint128 appendix;
}
``` [3](#0-2) 

And `resolveFastWithdrawal()` for V2 correctly allows a custom destination:

```solidity
address resolvedSendTo = signedTx.tx.sendTo == address(0)
    ? address(uint160(bytes20(signedTx.tx.sender)))
    : signedTx.tx.sendTo;
``` [4](#0-3) 

The V1 path is still live and reachable via the public `submitFastWithdrawal()` function: [5](#0-4) 

A subaccount is formed as `bytes32(abi.encodePacked(msg.sender, subaccountName))` at deposit time: [6](#0-5) 

So if `msg.sender` at deposit time was a smart contract (multisig, proxy, etc.), the V1 fast-withdrawal path will always route funds back to that contract address with no override possible.

The same hardcoded fallback also exists in the sequencer-processed V1 path in `EndpointTx`, which passes `address(0)` as `sendTo` to `Clearinghouse.withdrawCollateral()`, which then resolves it identically:

```solidity
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
}
``` [7](#0-6) 

---

### Impact Explanation

**High.** If the subaccount-owning contract cannot receive ERC20 tokens (no fallback, no token-handling logic), the `safeTransfer` in `handleWithdrawTransfer` will either revert (blocking the fast withdrawal entirely) or silently succeed while the tokens sit unreachable inside the contract. In the Wintermute-style worst case: if the contract was deployed via `CREATE2` and has been self-destructed, an attacker can redeploy a malicious contract at the same address before the withdrawal is triggered and capture the full collateral amount. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** Smart contract wallets (Gnosis Safe, custom multisigs, proxy-based accounts) are a standard pattern for institutional and DAO depositors. Any such depositor who signed a V1 `WithdrawCollateral` transaction — or whose client software only generates V1 transactions — is exposed. The self-destruct scenario is lower probability but non-zero given the history of `CREATE2`-deployed multisigs being compromised or rotated.

---

### Recommendation

1. **Deprecate V1 fast withdrawals** in `resolveFastWithdrawal()` and require callers to use `WithdrawCollateralV2` which already carries an explicit `sendTo` field.
2. If V1 must remain supported, add a `sendTo` override parameter to `submitFastWithdrawal()` that is validated against the signed transaction's sender (similar to how V2 restricts linked-signer usage when `sendTo != address(0)`).

---

### Proof of Concept

1. A Gnosis Safe multisig at address `0xSafe` calls `endpoint.depositCollateral("default", productId, amount)`. The subaccount becomes `bytes32(abi.encodePacked(0xSafe, "default"))`.
2. The Safe's owners sign a `WithdrawCollateral` (V1) transaction (no `sendTo` field available in the struct).
3. A fast-withdrawal provider calls `withdrawPool.submitFastWithdrawal(idx, v1Transaction, signatures)`.
4. `resolveFastWithdrawal()` computes `sendTo = address(uint160(bytes20(subaccount))) = 0xSafe`.
5. `handleWithdrawTransfer` calls `token.safeTransfer(0xSafe, amount)`.
6. If `0xSafe` has no ERC20 receive logic, the tokens are permanently locked. If `0xSafe` was a `CREATE2`-deployed contract that has since been self-destructed, an attacker who redeploys at `0xSafe` receives the full collateral.

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
