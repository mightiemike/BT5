### Title
linkedSigner Can Overwrite Its Own Authorization via `LinkSigner` with `allowLinkedSigner = true` - (File: core/contracts/EndpointTx.sol)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-path transaction is validated with `allowLinkedSigner = true`. This means the currently registered `linkedSigner` for a subaccount can sign a new `LinkSigner` transaction to replace the linked signer with any arbitrary address — including an attacker-controlled one — without the subaccount owner's involvement. The linked signer was intended to be a delegated trading key, not an entity with authority to reassign signing rights. This is a direct analog to the `trustedForwarder` impersonation issue: a trusted-but-limited entity can assume a role (subaccount controller) it was never meant to hold.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` resolves the allowed signer via `getLinkedSigner(sender)` and passes it to `verifier.validateSignature`: [2](#0-1) 

`verifier.validateSignature` accepts either the subaccount owner address **or** the `linkedSigner` as a valid signer: [3](#0-2) 

Because `LinkSigner` itself is validated with `allowLinkedSigner = true`, the current `linkedSigner` can produce a valid signature over a `LinkSigner` payload that sets `signedTx.tx.signer` to any address. The result is written unconditionally: [4](#0-3) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [5](#0-4) 

By contrast, the slow-mode `LinkSigner` path uses `validateSender`, which requires `msg.sender` to be the actual subaccount owner address — not the linked signer: [6](#0-5) 

The fast path has no equivalent restriction.

---

### Impact Explanation

Once the linked signer replaces itself with an attacker-controlled address, the attacker holds full signing authority over the subaccount for every operation that accepts `allowLinkedSigner = true`. This includes:

- **`TransferQuote`** (line 599, `allowLinkedSigner = true`): attacker transfers the victim's entire quote balance to an attacker-controlled subaccount, then withdraws it. [7](#0-6) 
- **`WithdrawCollateral` / `WithdrawCollateralV2` with `sendTo == address(0)`**: attacker initiates withdrawals. [8](#0-7) 
- **`LiquidateSubaccount`**: attacker liquidates other accounts on behalf of the victim, consuming the victim's nonce and fees. [9](#0-8) 
- **Permanent lock-out**: the subaccount owner can no longer reclaim signing authority through the fast path because the attacker controls the linked signer and can keep re-signing `LinkSigner` to maintain control.

The corrupted state is `linkedSigners[subaccount]` and, downstream, the subaccount's collateral balances via `TransferQuote`.

---

### Likelihood Explanation

Any address that has ever been granted `linkedSigner` status — including trading bots, hot wallets, or third-party integrations — can execute this attack unilaterally. No additional privileges, governance capture, or key leakage are required. The attacker only needs to submit a valid EIP-712 `LinkSigner` signature to the sequencer, which is a normal operational flow. The sequencer processes it without distinguishing whether the signer is the owner or the linked signer.

---

### Recommendation

Process `LinkSigner` transactions with `allowLinkedSigner = false` in the fast path, mirroring the slow-mode path's `validateSender` restriction. Only the subaccount owner (the address embedded in the first 20 bytes of the `sender` bytes32) should be permitted to change the linked signer. The linked signer's authority should be scoped to trading operations, not to self-perpetuating signing-rights delegation.

---

### Proof of Concept

1. Victim subaccount `S` has owner `O` and linked signer `L` (e.g., a trading bot).
2. `L` constructs a `SignedLinkSigner` payload: `{ sender: S, signer: bytes32(uint256(uint160(attacker))), nonce: currentNonce }`.
3. `L` signs the EIP-712 digest of this payload.
4. The sequencer includes this transaction in a `submitTransactionsChecked` batch.
5. `processTransactionImpl` validates the signature — `ECDSA.recover` returns `L`, which equals `getLinkedSigner(S)`, so validation passes.
6. `linkedSigners[S]` is now set to `attacker`.
7. `attacker` signs a `SignedTransferQuote` payload: `{ sender: S, recipient: attackerSubaccount, amount: fullBalance, nonce: nextNonce }`.
8. Validation passes because `attacker == getLinkedSigner(S)`.
9. `clearinghouse.transferQuote` moves the victim's entire quote balance to `attackerSubaccount`.
10. `attacker` withdraws from `attackerSubaccount` via normal withdrawal flow.

```text
    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
```
