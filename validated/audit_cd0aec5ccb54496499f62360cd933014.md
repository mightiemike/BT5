Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks whether the router is allowlisted — never the actual end user. Any non-allowlisted user can bypass the per-user gate by routing through the canonical periphery router.

## Finding Description
**Root cause — pool passes `msg.sender` (the router) as `sender`:**

In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the first argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when called via router
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` encodes that value verbatim as the `sender` forwarded to every configured extension (`metric-core/contracts/ExtensionCalling.sol` L149-177).

**Router never forwards the original caller:**

All four public entry points in `MetricOmmSimpleRouter` call `pool.swap(...)` directly with no mechanism to forward `msg.sender` as `sender`. For example, `exactInputSingle` (L72-80) calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — the pool sees `msg.sender = router`.

**Extension checks the wrong address:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

`msg.sender` here is the pool (enforced by `onlyPool`). `sender` is the router. The check resolves to `allowedSwapper[pool][router]`. The actual end user's address is never consulted.

**Two broken states:**

| Admin configuration | Effect |
|---|---|
| Router allowlisted (to support standard periphery) | Every user, including those explicitly excluded, bypasses the allowlist via the router |
| Router NOT allowlisted | Explicitly allowlisted users cannot swap through the router; standard periphery is broken |

Neither configuration achieves per-user gating.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified counterparties, institutional LPs, whitelisted market makers) loses that protection entirely for any user routing through `MetricOmmSimpleRouter`. The attacker receives real token output from the pool's LP reserves at oracle-derived prices. This constitutes broken core pool functionality — the access control mechanism is completely defeated — and enables unauthorized execution on pools that should have been restricted, meeting the "broken core pool functionality causing loss of funds or unusable swap flows" impact criterion.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical, publicly deployed periphery entry point. No privileged access, special tokens, or unusual setup is required. Any user who encounters a `NotAllowedToSwap` revert on a direct pool call can trivially re-route through the router. The attack is repeatable with a standard swap call and knowledge of the router address.

## Recommendation
The extension must gate on the economically relevant actor. Two sound approaches:

1. **Forward the original caller through the router.** Add a `swapper` field passed as `callbackData` or a dedicated parameter so the pool can forward it as `sender`. Requires a pool-level interface change.
2. **Check `recipient` in addition to `sender`.** For single-hop swaps, `recipient` is the end user. However, multi-hop `exactOutput` routes intermediate tokens through the router as recipient, so this is not universally correct.
3. **Simplest safe fix:** In `SwapAllowlistExtension.beforeSwap`, check **both** `sender` and `recipient` — if either is not allowlisted, revert. This prevents the router from acting as a pass-through for non-allowlisted users.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to allow standard periphery usage).
  - Pool admin does NOT allowlist attacker address.

Attack:
  1. Attacker calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: attacker,
       ...
     })
  2. Router calls pool.swap(attacker, ...) — msg.sender = router.
  3. Pool calls _beforeSwap(sender=router, recipient=attacker, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  5. Swap executes. Attacker receives token output.

Result:
  The attacker, who is not on the allowlist, successfully swaps on a
  curated pool. The per-user allowlist is completely bypassed.
```
