### Title
`op_unknown` Cost Check Applied to Pre-Multiplication Base Cost, Allowing Returned Cost to Exceed `max_cost` - (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the cost guard (`check_cost`) is called on the **base cost** before it is multiplied by `(cost_multiplier + 1)`. The function then returns the **post-multiplication cost** as the consumed cost. An attacker who controls the opcode bytes can craft an unknown opcode whose base cost passes the guard but whose final returned cost exceeds `max_cost`, violating the cost-budget invariant that every operator is required to enforce.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` computes cost in two stages:

1. A base cost is computed from the `cost_function` field (bits 7–6 of the last opcode byte), ranging from 1 (constant) up to the arithmetic/mul/concat formula applied to the arguments.
2. That base cost is multiplied by `(cost_multiplier + 1)`, where `cost_multiplier` is the big-endian unsigned integer encoded in all bytes of the opcode except the last.

The critical ordering is:

```
check_cost(cost, max_cost)?;      // line 260 — guards the BASE cost
cost *= cost_multiplier + 1;      // line 261 — multiplies AFTER the guard
if cost > u32::MAX as u64 {       // line 262 — only catches u32 overflow
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))   // line 265 — returns multiplied cost
}
``` [1](#0-0) 

The guard at line 260 checks whether the **base cost** (e.g., `1` for `cost_function == 0`) exceeds `max_cost`. It does not check whether the **final returned cost** (`base_cost * (cost_multiplier + 1)`) exceeds `max_cost`. The only post-multiplication check is `cost > u32::MAX`, which is a completely different bound.

The `cost_multiplier` is derived from attacker-controlled opcode bytes via `u32_from_u8`:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    None => { return Err(EvalErr::Invalid(o))?; }
};
``` [2](#0-1) 

For a 5-byte opcode, `op[0..4]` is parsed as a 4-byte big-endian integer, giving a maximum `cost_multiplier` of `0xFFFFFFFF = 4,294,967,295`. The final cost can therefore be up to `base_cost * 4,294,967,296`, which is bounded only by `u32::MAX = 4,294,967,295` — not by `max_cost`.

**Concrete example:**
- `max_cost = 500` (remaining budget passed from `run_program`)
- Opcode bytes: `[0x00, 0x00, 0x01, 0xF4, 0x00]` → `cost_function = 0`, `cost_multiplier = 500`
- Base cost = `1`
- `check_cost(1, 500)` → passes (1 ≤ 500)
- `cost = 1 * (500 + 1) = 501`
- `501 ≤ u32::MAX` → returns `Ok(Reduction(501, nil))`
- **Returned cost 501 > max_cost 500, but no error was raised.**

Every other operator in the codebase calls `check_cost` on the **final** cost it will return. For example, `op_add` calls `check_cost(cost, max_cost)` after accumulating per-argument and per-byte costs, and returns exactly that checked cost. [3](#0-2)  `op_unknown` is the only operator that checks a different (smaller) value than what it returns.

---

### Impact Explanation

The invariant broken is: **every operator must not return a cost greater than `max_cost`**. This invariant is what allows `run_program` to trust operator-returned costs for budget accounting. When `op_unknown` returns a cost exceeding `max_cost`, the cost ledger in `run_program` is corrupted. Depending on how `run_program` accumulates costs (subtraction from remaining budget vs. addition to a running total), this can cause:

- **Consensus divergence**: nodes that re-check the returned cost against the remaining budget will reject the transaction; nodes that trust the operator's internal guard will accept it. This is a split-brain scenario on a consensus-critical execution path.
- **Undercharged execution**: a program that should have been rejected for exceeding the cost limit executes successfully, because the per-operator guard fires on the wrong value.

Unknown opcodes are reachable in non-mempool (consensus) mode — `NO_UNKNOWN_OPS` is not set by default, and `op_unknown` is explicitly called from `ChiaDialect::op` for any opcode that does not match a known 1-byte or 4-byte handler. [4](#0-3) 

---

### Likelihood Explanation

Any attacker who can submit a CLVM program to a Chia full node can trigger this path. Unknown opcodes are valid in consensus mode (non-mempool). The attacker only needs to include a 2–5 byte opcode atom that does not collide with a known opcode and is not `0xffff`-prefixed. The `cost_multiplier` is directly encoded in the opcode bytes, so the attacker has full control over the ratio between the checked base cost and the returned final cost. No special privileges, keys, or social engineering are required.

---

### Recommendation

Move `check_cost` to **after** the multiplication, so it guards the value that is actually returned:

```rust
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}
check_cost(cost, max_cost)?;   // guard the final cost, not the base cost
Ok(Reduction(cost as Cost, allocator.nil()))
``` [1](#0-0)

### Proof of Concept

```
# 5-byte unknown opcode: cost_function=0, cost_multiplier=0x000001F4 (500)
# Opcode bytes: [0x00, 0x00, 0x01, 0xF4, 0x00]
# Encoded as CLVM atom: 0x85 00 00 01 F4 00
# Program: (unknown_op_500x)  with max_cost = 500

# Step 1: cost_function = (0x00 & 0xC0) >> 6 = 0  → base cost = 1
# Step 2: cost_multiplier = u32_from_u8([0x00, 0x00, 0x01, 0xF4]) = 500
# Step 3: check_cost(1, 500) → Ok  (1 ≤ 500, passes)
# Step 4: cost = 1 * (500 + 1) = 501
# Step 5: 501 ≤ 4294967295 → Ok(Reduction(501, nil))
# Result: returned cost 501 > max_cost 500, budget invariant violated, no error raised.
```

The attacker can scale this to any ratio: with `cost_multiplier = 4294967294` and `max_cost = 4294967294`, the base cost of `1` passes the guard, and the returned cost is `4294967295` — the maximum possible — regardless of how small the remaining budget is.
