Audit Report

## Title
`OracleValueStopLossExtension._afterSwapOracleStopLoss` Silently Disables Stop-Loss When `drawdownE6 == 0` Instead of Enforcing Strictest Protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterSwapOracleStopLoss` returns early at line 217 when `drawdownE6 == 0`, bypassing all watermark checks. Mathematically, `drawdownE6 = 0` produces `floorMultiplier = 1e6` (100% floor — strictest possible protection), not a disabled state. A pool initialized with `drawdownE6 = 0` has its entire stop-loss silently bypassed, allowing any public swapper to drain LP value without triggering the configured protection.

## Finding Description

In `_afterSwapOracleStopLoss`, after loading config, the function immediately returns when `drawdown == 0`: [1](#0-0) 

The floor used in the actual breach check is: [2](#0-1) 

When `drawdown == 0`, `floorMultiplier = 1e6 = E6`, meaning the floor equals 100% of the watermark — any metric drop below the watermark triggers a breach. This is the **strictest** possible configuration. The actual disabled state is `drawdown == E6` (100% drawdown allowed), which makes `floorMultiplier = 0` and the breach condition `metric < 0` — never true.

`_validateDrawdown` explicitly permits `0` as a valid input with no "disables the guard" annotation: [3](#0-2) 

Contrast with the decay parameter, which **does** document that zero disables it: [4](#0-3) 

No equivalent documentation exists for `drawdownE6 = 0`. Because `_validateDrawdown` accepts 0 and the mathematical semantics of 0 is "strictest protection," a pool admin setting `drawdownE6 = 0` receives zero protection. The `_checkAndUpdateWatermarks` and `_applyWatermark` calls that would enforce the breach are never reached: [5](#0-4) 

## Impact Explanation

Any public swapper on a pool whose `OracleValueStopLossExtension` was initialized with `drawdownE6 = 0` can execute swaps that drain LP value without the stop-loss ever triggering. Watermarks are never updated and no revert is issued regardless of how far the per-share metric falls. This constitutes direct loss of LP principal — a High/Critical impact under the allowed impact gate (broken core pool functionality causing loss of funds; stop-loss guard fails open).

## Likelihood Explanation

`drawdownE6 = 0` is a valid, accepted initialization value that passes `_validateDrawdown`. A pool admin intending the strictest protection — "no drawdown tolerated" — would naturally supply `0`, mirroring the decay parameter's convention where `0` is a meaningful boundary value. The absence of any "0 disables" documentation makes this a realistic and likely misconfiguration. The trigger is any public `swap` call on such a pool; no special privilege is required beyond being a normal swapper.

## Recommendation

Replace the early exit with the semantically correct disabled check:

```solidity
// Disabled state: 100% drawdown allowed → floor is always 0 → never breaches
if (drawdown >= E6) return;
```

Alternatively, if `0` is intentionally a sentinel for "disabled," add explicit validation rejecting `0` in `_validateDrawdown` and add NatSpec documentation mirroring the decay parameter's pattern (`0 disables stop-loss`).

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` initialized via `abi.encode(uint32(0), uint32(58), uint32(0))` — `drawdownE6 = 0`, decay enabled, no timelock.
2. Add liquidity to the pool.
3. Execute adversarial swaps that move the oracle mid-price sharply, causing per-share bin value to drop well below the initial watermark.
4. Observe: `afterSwap` is called, `_afterSwapOracleStopLoss` loads `drawdown = 0` at line 217, immediately returns, and no `OracleStopLossTriggered` revert is issued.
5. Repeat with `drawdownE6 = 50_000` (5%) under identical conditions — the same swap sequence reverts with `OracleStopLossTriggered`, confirming the guard works for non-zero values but fails open at `0`.
