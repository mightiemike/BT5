Audit Report

## Title
`PriceVelocityGuardExtension` velocity guard silently bypassed due to uninitialized `maxChangePerBlockE18` and `lastMidPriceX64` — (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

## Summary

`PriceVelocityGuardExtension` does not override `initialize()`, so the factory's initialization call falls through to `BaseMetricExtension.initialize`, which is a no-op. Both `maxChangePerBlockE18` and `lastMidPriceX64` remain zero for every newly created pool. Two independent zero-guards in `beforeSwap` each independently short-circuit the entire velocity check, meaning any oracle price movement is accepted without restriction until the pool admin makes two separate post-deployment calls. The first swap against any pool using this extension is always unguarded regardless of admin action, because `lastMidPriceX64` starts at zero and the `prevMid != 0` guard fails.

## Finding Description

`MetricOmmPoolFactory.createPool` calls `initialize(pool, extensionInitData[i])` on every registered extension immediately after deployment:

```solidity
// MetricOmmPoolFactory.sol L206-210
for (uint256 i = 0; i < params.extensions.length; i++) {
    CallExtension.callExtension(
        params.extensions[i], abi.encodeCall(IMetricOmmExtensions.initialize, (pool, params.extensionInitData[i]))
    );
}
```

`PriceVelocityGuardExtension` has no `initialize()` override, so the call falls through to `BaseMetricExtension.initialize` (L41-43), which does nothing except return the selector. `priceVelocityState[pool]` is the zero-value struct `{maxChangePerBlockE18: 0, lastMidPriceX64: 0, lastUpdateBlock: 0}` for every new pool.

Contrast with `OracleValueStopLossExtension.initialize` (L46-68), which properly decodes and stores `drawdownE6`, `decayPerSecondE8`, and `timelock` atomically at pool creation, and reverts if called twice.

Inside `PriceVelocityGuardExtension.beforeSwap` (L60-76), two independent zero-guards each bypass the entire check:

```solidity
if (prevMid != 0) {           // Guard 1: skips if no prior price recorded (always true on first swap)
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {     // Guard 2: skips if cap was never set
        ...
        if (actualSq > allowedSq) {
            revert PriceVelocityExceeded(actualSq, allowedSq);
        }
    }
}
```

The two admin setters that would activate the guard — `setMaxChangePerBlock` and `setLastMidPrice` — are never called by the factory and are not required before the pool accepts swaps. `MetricOmmPool.swap` calls `_beforeSwap` unconditionally before executing any trade (L230-240).

Additionally, `PriceVelocityGuardExtension.beforeSwap` drops the `onlyPool` modifier that `BaseMetricExtension` declares on its virtual stub (L81-88), meaning any external caller can invoke it and write to `priceVelocityState[msg.sender]`, though the primary impact is the initialization gap.

## Impact Explanation

A pool that registers `PriceVelocityGuardExtension` in its `BEFORE_SWAP_ORDER` provides zero velocity protection until the pool admin makes two separate post-deployment transactions. During the unconfigured window, the oracle's raw bid/ask prices — however fast they move — are accepted without any cap. If the oracle delivers a price that has moved far beyond what the pool's LP positions can absorb at fair value (e.g., due to a flash oracle update, a stale feed, or a price spike), swaps execute at that bad price. LPs receive less than the oracle mid-price implies they are owed, constituting a direct loss of LP principal. This matches the "bad-price execution" and "direct loss of user principal" impact categories.

## Likelihood Explanation

The misconfiguration is the default state of every pool that uses this extension. A pool admin who deploys with `PriceVelocityGuardExtension` expecting velocity protection receives none until they discover and execute two separate admin calls. The first swap after pool creation is always unguarded regardless of whether `setMaxChangePerBlock` was called, because `lastMidPriceX64` starts at zero and `prevMid != 0` is false. The window is therefore at minimum one block wide for every pool using this extension, and potentially indefinite if the admin is unaware of the required post-deployment steps. No on-chain mechanism enforces or signals that the guard is inactive.

## Recommendation

Override `initialize()` in `PriceVelocityGuardExtension` to decode and store `maxChangePerBlockE18` and an initial `lastMidPriceX64` from the `data` parameter, mirroring the pattern used by `OracleValueStopLossExtension`. Require `maxChangePerBlockE18 > 0` during initialization so that a pool cannot be created with a silently disabled guard:

```solidity
function initialize(address pool, bytes calldata data)
    external
    override
    onlyFactory
    returns (bytes4)
{
    (uint64 maxChange, uint128 initialMid) = abi.decode(data, (uint64, uint128));
    require(maxChange > 0, "PVG: maxChange must be nonzero");
    PriceVelocityState storage s = priceVelocityState[pool];
    s.maxChangePerBlockE18 = maxChange;
    s.lastMidPriceX64 = initialMid;
    s.lastUpdateBlock = uint64(block.number);
    return IMetricOmmExtensions.initialize.selector;
}
```

Also restore the `onlyPool` modifier on `beforeSwap` to match the base contract's access control intent.

## Proof of Concept

1. Pool admin calls `factory.createPool(params)` with `PriceVelocityGuardExtension` listed in `params.extensions` and wired into `BEFORE_SWAP_ORDER`. `extensionInitData[i]` is empty (or ignored, since `initialize` is not overridden).
2. Factory calls `PriceVelocityGuardExtension.initialize(pool, "")` → falls through to `BaseMetricExtension.initialize` (L41-43), a no-op. `priceVelocityState[pool]` remains `{maxChangePerBlockE18: 0, lastMidPriceX64: 0, lastUpdateBlock: 0}`.
3. Admin does not call `setMaxChangePerBlock` or `setLastMidPrice` (or calls only one of them).
4. Oracle delivers a price that has moved 40% in one block.
5. Attacker calls `pool.swap(...)`.
6. Pool calls `_beforeSwap` (L230-240) → `PriceVelocityGuardExtension.beforeSwap` executes.
7. `prevMid = s.lastMidPriceX64 == 0` → `if (prevMid != 0)` is false (L60) → entire velocity check skipped, function returns `beforeSwap.selector`.
8. Swap executes at the 40%-moved oracle price. LPs receive far less token output than the pre-spike price would have entitled them to. Attacker profits from the price discrepancy at LP expense.
