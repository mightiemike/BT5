### Title
Missing Slippage Protection in NLP Mint and Burn Operations Exposes Users to Unfavorable Exchange Rates — (`core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `mintNlp` and `burnNlp` functions in `Clearinghouse.sol` execute NLP share conversions at a sequencer-supplied `oraclePriceX18` that is **not included in the user's EIP-712 signature**. Users have no mechanism to specify a minimum acceptable NLP output (for minting) or minimum acceptable quote output (for burning). Price movement between the time a user signs a request and the time the sequencer processes it can cause users to receive materially fewer tokens than expected, with no on-chain recourse.

---

### Finding Description

The `MintNlp` and `BurnNlp` structs that users sign contain only `sender`, an amount, and a `nonce`:

```solidity
// IEndpoint.sol
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The EIP-712 digest computed in `Verifier.sol` confirms that `oraclePriceX18` is **excluded** from what the user signs:

```solidity
string internal constant MINT_NLP_SIGNATURE =
    "MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)";
string internal constant BURN_NLP_SIGNATURE =
    "BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)";
``` [2](#0-1) 

The digest computation for `MintNlp` hashes only `sender`, `quoteAmount`, and `nonce` — no price bound: [3](#0-2) 

The `oraclePriceX18` lives in the outer `SignedMintNlp`/`SignedBurnNlp` wrapper, which is constructed and submitted by the sequencer: [4](#0-3) 

In `EndpointTx.sol`, the sequencer-supplied `signedTx.oraclePriceX18` is passed directly to `clearinghouse.mintNlp` / `clearinghouse.burnNlp` without any user-defined bound check: [5](#0-4) 

In `Clearinghouse.sol`, the NLP amount received by the user is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);   // mintNlp
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);   // burnNlp (before fee)
``` [6](#0-5) [7](#0-6) 

There is no check of the form `require(nlpAmount >= minNlpOut)` or `require(quoteAmount >= minQuoteOut)`.

---

### Impact Explanation

**For `mintNlp`:** A user signs a commitment to spend `quoteAmount` of quote tokens. If the NLP oracle price rises between signing and sequencer execution, the user receives `quoteAmount / higherPrice` NLP — fewer shares than anticipated — with no ability to revert the transaction.

**For `burnNlp`:** A user signs a commitment to burn `nlpAmount` of NLP. If the NLP oracle price falls between signing and execution, the user receives `nlpAmount * lowerPrice - burnFee` quote — fewer assets than anticipated — with no ability to revert.

In both cases the user's signed intent commits only to the input amount; the output is entirely determined by a sequencer-supplied price that the user cannot constrain. The corrupted state delta is a direct reduction in the user's `QUOTE_PRODUCT_ID` or `NLP_PRODUCT_ID` balance relative to what they expected at signing time.

---

### Likelihood Explanation

NLP price (`oraclePriceX18`) is updated continuously by the sequencer via `UpdatePrice` transactions. Because Nado is an off-chain order-book protocol, there is an inherent and normal delay between when a user signs a `MintNlp`/`BurnNlp` request and when the sequencer batches and submits it on-chain. During periods of NLP price volatility this delay is sufficient to produce meaningful slippage. No malicious sequencer behavior is required — an honest sequencer using the current oracle price at execution time is enough to trigger the loss.

---

### Recommendation

1. Add `minNlpOut` to `MintNlp` and `minQuoteOut` to `BurnNlp` in `IEndpoint.sol`.
2. Include these fields in the EIP-712 type strings and digest computation in `Verifier.sol` so users cryptographically commit to their acceptable bounds.
3. In `Clearinghouse.mintNlp`, add `require(nlpAmount >= txn.minNlpOut, "slippage")` after computing `nlpAmount`.
4. In `Clearinghouse.burnNlp`, add `require(quoteAmount >= txn.minQuoteOut, "slippage")` after computing `quoteAmount`.

---

### Proof of Concept

1. NLP oracle price is currently `1.00 USDC/NLP`.
2. User signs `MintNlp{sender, quoteAmount=1000e18, nonce=N}` expecting to receive `~1000 NLP`.
3. Before the sequencer processes the transaction, the NLP price rises to `1.10 USDC/NLP`.
4. Sequencer submits `SignedMintNlp{tx: <user-signed>, oraclePriceX18: 1.10e18, ...}`.
5. `Clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 1.10e18 ≈ 909 NLP`.
6. User's balance is credited with `909 NLP` instead of the expected `1000 NLP` — a ~9% loss — and the transaction cannot be reverted because no minimum was enforced.

The user's signature is valid (it covers only `sender`, `quoteAmount`, `nonce`), the sequencer acted honestly, and yet the user suffers a concrete asset loss with no on-chain protection.
