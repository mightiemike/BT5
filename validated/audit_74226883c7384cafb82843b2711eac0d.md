### Title
Protobuf `ValidResourceBounds` Deserializer Silently Converts `AllResources` to `L1Gas` When L2/Data Gas Are Zero, Producing a Different Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently changes the resource-bounds variant from `AllResources` to `L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` produces structurally different hash inputs for the two variants (the `AllResources` path appends an extra `L1_DATA_GAS` element), the transaction hash computed after a protobuf round-trip differs from the hash computed at submission time. Any component that re-derives the hash from the deserialized transaction — including the blockifier's account-validation call and the consensus orchestrator's `convert_rpc_tx_to_internal` path — will bind the wrong hash, causing valid transactions to be rejected or committed under an incorrect identifier.

---

### Finding Description

**Step 1 — Two variants, two different hashes.**

`get_tip_resource_bounds_hash` branches on the variant tag:

```
L1Gas(X):
  poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS))          // 3 elements

AllResources(X, 0, 0):
  poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS), concat(0, L1_DATA_GAS))  // 4 elements
```

Even when `l2_gas` and `l1_data_gas` are both zero, the `AllResources` path feeds one extra felt into the Poseidon sponge, producing a hash `H_all ≠ H_l1`. [1](#0-0) 

**Step 2 — The protobuf deserializer silently changes the variant.**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← variant changed
} else {
    ValidResourceBounds::AllResources(...)
})
```

A transaction that was originally `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is deserialized as `L1Gas(X)`. [2](#0-1) 

**Step 3 — The hash is recomputed from the deserialized form.**

In `convert_rpc_tx_to_internal`, after the protobuf round-trip through the consensus P2P layer, the hash is freshly derived from the deserialized `InternalRpcTransactionWithoutTxHash`:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [3](#0-2) 

For the consensus path the `RpcInvokeTransactionV3` struct carries `AllResourceBounds` (not `ValidResourceBounds`), so the variant-collapsing converter is not triggered there. However, the sync path deserializes directly into `InvokeTransactionV3` which carries `ValidResourceBounds`, and the hash stored in the protobuf `transaction_hash` field is the original `H_all`. Any subsequent call to `validate_transaction_hash` with the deserialized object will recompute `H_l1 ≠ H_all` and reject the transaction. [4](#0-3) 

**Step 4 — The RPC serialization path also loses the variant.**

`From<ValidResourceBounds> for ResourceBoundsMapping` (used when serving transactions over the JSON-RPC) maps `L1Gas` to a mapping with zero `l2_gas` and zero `l1_data_gas`, so a syncing node that stored the transaction as `L1Gas` will serve it with the wrong resource-bounds structure, and any client that recomputes the hash from the RPC response will obtain `H_l1` instead of `H_all`. [5](#0-4) 

---

### Impact Explanation

A transaction submitted via the RPC with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is signed over `H_all`. After the transaction is included in a block and propagated via the sync P2P layer, the deserialized `InvokeTransactionV3` carries `L1Gas(X)`. Any node that re-derives the hash (for block-hash computation, receipt lookup, or `validate_transaction_hash`) obtains `H_l1 ≠ H_all`. Concretely:

- **Wrong receipt / state**: the committed transaction hash diverges from the signed hash, so the receipt is indexed under the wrong key and the account's nonce is advanced under a hash the user never signed.
- **Wrong RPC value**: `starknet_getTransactionByHash(H_all)` returns not-found on syncing nodes; `starknet_getTransactionByHash(H_l1)` returns the transaction with wrong resource-bounds type — an authoritative-looking wrong value.
- **Signature-domain mismatch**: the blockifier's account-validation entry point receives `H_l1` as the transaction hash, but the account contract verifies the user's ECDSA signature over `H_all`, causing every such transaction to revert with an invalid-signature error on syncing nodes.

---

### Likelihood Explanation

The trigger condition — `l2_gas.max_amount == 0 && l2_gas.max_price_per_unit == 0 && l1_data_gas.max_amount == 0 && l1_data_gas.max_price_per_unit == 0` — is reachable by any user who submits a V3 invoke transaction relying solely on L1 gas (a pattern that was common before 0.13.3 and is still valid). The `AllResourceBounds::new_unlimited_gas_no_fee_enforcement` helper sets `l1_gas.max_amount = 0` and `l1_data_gas.max_amount = 0`, which would also trigger the conversion if `l2_gas` happened to be zero. No privileged access is required; a normal user transaction is sufficient.

---

### Recommendation

Remove the variant-collapsing logic from the protobuf deserializer. The `ValidResourceBounds` variant must be preserved exactly as serialized, regardless of whether the numeric values are zero:

```rust
// Before (buggy):
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})

// After (correct):
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

The `L1Gas` variant should only be produced when the serialized protobuf message was originally created from a `L1Gas` transaction (pre-0.13.3), which requires a separate discriminator field in the protobuf schema, not inference from zero values.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::block::GasPrice;
use starknet_api::execution_resources::GasAmount;
use starknet_api::transaction::fields::Tip;
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;

fn main() {
    let l1_gas = ResourceBounds {
        max_amount: GasAmount(100),
        max_price_per_unit: GasPrice(1),
    };
    let zero = ResourceBounds::default(); // max_amount=0, max_price_per_unit=0
    let tip = Tip(0);

    // Original AllResources with zero L2/data gas (as submitted via RPC)
    let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas,
        l2_gas: zero,
        l1_data_gas: zero,
    });

    // After protobuf round-trip: converter collapses to L1Gas
    let l1_gas_variant = ValidResourceBounds::L1Gas(l1_gas);

    let h_all = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();
    let h_l1  = get_tip_resource_bounds_hash(&l1_gas_variant, &tip).unwrap();

    // These are different — the transaction hash changes across the protobuf boundary
    assert_ne!(h_all, h_l1, "Hash mismatch: protobuf round-trip changes the tx hash");
}
```

The `assert_ne!` passes, demonstrating that the protobuf deserializer silently changes the hash domain of the transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
