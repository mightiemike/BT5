### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` Permanently Blocks Bankrupt Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides `balance.vQuoteBalance` by `state.openInterest` when insurance is insufficient to cover a negative vQuote balance. Because `_finalizeSubaccount` requires all perp `balance.amount` values to be zero before calling `socializeSubaccount`, the liquidatee contributes zero to `openInterest`. If no other subaccount holds an open position in that product, `state.openInterest == 0`, and `MathSD21x18.div` reverts with `ERR_DIV_BY_ZERO`, permanently blocking finalization.

---

### Finding Description

**Entry point:** `Clearinghouse.liquidateSubaccount` → `delegatecall` → `ClearinghouseLiq.liquidateSubaccountImpl` → `_finalizeSubaccount` (triggered when `txn.productId == type(uint32).max`). [1](#0-0) 

`_finalizeSubaccount` enforces that every perp product's `balance.amount == 0` before proceeding: [2](#0-1) 

It then calls `perpEngine.socializeSubaccount` unconditionally: [3](#0-2) 

Inside `socializeSubaccount`, for each product where `balance.vQuoteBalance < 0` and insurance is insufficient, the code attempts to socialize the residual loss across open-interest holders: [4](#0-3) 

`state.openInterest` is computed by `getStateAndBalance` → `_updateBalance(state, balance, 0, 0)`. Because `balance.amount == 0` (enforced above), the liquidatee contributes zero to `openInterest`. If no other subaccount holds an open position in that product, `state.openInterest == 0`. [5](#0-4) 

`MathSD21x18.div` explicitly reverts on a zero denominator: [6](#0-5) 

The revert propagates all the way back through `delegatecall` in `Clearinghouse.liquidateSubaccount`: [7](#0-6) 

The transaction reverts, the subaccount state is unchanged, and the call can never succeed as long as `openInterest == 0` for that product.

---

### Impact Explanation

The bankrupt subaccount's negative `vQuoteBalance` is never cleared and never socialized. The finalization path is permanently bricked for this subaccount. The protocol's accounting invariant — that insolvent subaccounts are resolved and losses distributed — is broken. The uncleared negative `vQuoteBalance` represents an unaccounted liability that leaves the protocol undercollateralized relative to its recorded state.

---

### Likelihood Explanation

The preconditions are reachable in normal operation:

1. A subaccount closes its perp position (`balance.amount → 0`) but retains a negative `vQuoteBalance` (realized loss).
2. All other participants in that specific perp market also have `balance.amount == 0` (market wound down, low liquidity, or all positions closed).
3. Insurance is insufficient to cover the full loss.
4. A liquidator submits `productId == type(uint32).max` to finalize.

Low-liquidity or newly-launched perp markets are the most realistic trigger. The liquidatee itself being the last open-interest holder before closing is a common end-of-life scenario for a position.

---

### Recommendation

Add a zero-check for `state.openInterest` before the division. If `openInterest == 0`, there is no one to socialize the loss onto; the loss should either be absorbed entirely by insurance, written off, or handled via a fallback path:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb with insurance or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```
