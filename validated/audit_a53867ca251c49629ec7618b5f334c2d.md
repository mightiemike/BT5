### Title
Missing `ENABLE_SECP_OPS` Flag Check in 4-Byte Secp Opcode Dispatch Allows Activation Gate Bypass — (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches 4-byte opcodes `0x13d61f00` (secp256k1\_verify) and `0x1c3a8f00` (secp256r1\_verify) directly to their implementation functions without checking `ClvmFlags::ENABLE_SECP_OPS`. The 1-byte opcode forms (64 and 65) correctly gate on this flag. An attacker can craft CLVM bytes using the 4-byte opcode form to invoke secp signature verification regardless of whether `ENABLE_SECP_OPS` is set, producing execution outcomes that diverge from the expected no-op behavior and from programs using the 1-byte form under the same flag configuration.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` has two separate dispatch paths for the secp operators:

**Path 1 — 1-byte opcodes (correctly gated):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

When `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator`, which returns a no-op result in consensus mode or `Err(EvalErr::Unimplemented)` in mempool mode (`NO_UNKNOWN_OPS`). [1](#0-0) 

**Path 2 — 4-byte opcodes (flag check absent):**

```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

No `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` check is present. The secp functions execute unconditionally for these opcodes. [2](#0-1) 

The `ENABLE_SECP_OPS` flag (`0x0800`) is defined as the activation gate for secp operations and is absent from `MEMPOOL_MODE` and from all softfork extension flag sets. [3](#0-2) 

The root cause is a structural asymmetry: the 1-byte dispatch table uses Rust pattern guards (`if flags.contains(...)`) to enforce the flag, but the 4-byte dispatch block — added separately to handle the "unknown operator with assigned cost" form — omits the equivalent guard entirely.

---

### Impact Explanation

When `ENABLE_SECP_OPS` is not set (pre-activation or any caller that omits the flag):

| Opcode form | `ENABLE_SECP_OPS` absent, consensus mode | `ENABLE_SECP_OPS` absent, mempool mode |
|---|---|---|
| 1-byte (64/65) | no-op, returns nil | `Err(Unimplemented)` |
| 4-byte (`0x13d61f00`/`0x1c3a8f00`) | **executes secp verify** | **executes secp verify** |

Concrete divergences:

1. **Consensus divergence between opcode forms.** A program using the 4-byte form with an invalid secp signature raises `EvalErr::Secp256Failed` even when `ENABLE_SECP_OPS` is not set. The identical logical program using the 1-byte form returns nil (no-op). Two programs that should be semantically equivalent produce different outcomes. [4](#0-3) 

2. **Activation gate bypass.** Before `ENABLE_SECP_OPS` is activated on-chain, an attacker can submit programs using the 4-byte opcode form that execute real secp signature verification. This allows programs to depend on secp semantics (raise on bad sig, succeed on good sig) before the soft-fork has activated, violating the invariant that pre-activation programs must not observe secp behavior.

3. **Mempool/consensus split.** In mempool mode (`NO_UNKNOWN_OPS` set, `ENABLE_SECP_OPS` absent), the 1-byte form is rejected as unimplemented, but the 4-byte form executes. A transaction that would be rejected by the mempool's 1-byte path can be submitted using the 4-byte path and execute successfully, bypassing mempool policy. [5](#0-4) 

---

### Likelihood Explanation

The attacker requires only the ability to submit CLVM programs — the standard, unprivileged capability of any Chia network participant. No keys, credentials, or privileged access are needed. The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are derivable from the cost formula documented in the source comments. The trigger is deterministic and reproducible with a small, hand-crafted CLVM atom.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch block, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

When `ENABLE_SECP_OPS` is absent, the 4-byte opcodes fall through to `unknown_operator`, producing the same no-op/error behavior as the 1-byte form, restoring the invariant that the activation flag uniformly gates secp execution regardless of opcode encoding. [6](#0-5)

### Proof of Concept

```rust
use clvmr::{Allocator, run_program};
use clvmr::chia_dialect::{ChiaDialect, ClvmFlags};

let mut a = Allocator::new();

// Build: (0x13d61f00 pubkey msg bad_sig)
// 4-byte opcode form of secp256k1_verify, invalid signature
let op = a.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
let pubkey = a.new_atom(&[0u8; 33]).unwrap();   // compressed G1 point (invalid)
let msg    = a.new_atom(&[0u8; 32]).unwrap();
let sig    = a.new_atom(&[0u8; 64]).unwrap();
// build arg list: (pubkey msg sig)
let nil = a.nil();
let l3  = a.new_pair(sig, nil).unwrap();
let l2  = a.new_pair(msg, l3).unwrap();
let l1  = a.new_pair(pubkey, l2).unwrap();
let prg = a.new_pair(op, l1).unwrap();

// No ENABLE_SECP_OPS — secp should be a no-op
let dialect = ChiaDialect::new(ClvmFlags::empty());
let result = run_program(&mut a, &dialect, prg, a.nil(), 10_000_000);

// Expected (with correct fix): Ok(nil) — no-op
// Actual (current code):       Err(Secp256Failed) — secp executed and rejected bad sig
println!("{:?}", result);
```

The program raises `Secp256Failed` despite `ENABLE_SECP_OPS` being absent, confirming the flag check is bypassed via the 4-byte opcode path. The same program rewritten with 1-byte opcode `\x40` (64) returns `Ok(nil)` under identical flags. [2](#0-1)
