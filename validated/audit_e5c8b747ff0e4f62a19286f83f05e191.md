Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of real swapper, breaking allowlist enforcement for router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` forwards `msg.sender` as the `sender` argument to every configured extension. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract address, not the actual user. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][sender]` — i.e., the router's allowlist status — producing two simultaneous broken invariants: allowlisted users are silently blocked from using the router, and allowlisting the router to restore access grants every user on the internet a bypass.

## Finding Description

**Root cause — pool forwards `msg.sender` (the router) as `sender` to extensions:**

In `MetricOmmPool.swap` (lines 230–240), the pool calls:
```solidity
_beforeSwap(
  msg.sender,   // router address when called through the router
  recipient,
  ...
  extensionData
);
``` [1](#0-0) 

**Extension checks `sender` (= router) against the per-pool allowlist:**

`SwapAllowlistExtension.beforeSwap` at lines 37–38:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool; `sender` is whatever the pool forwarded — the router address for every `MetricOmmSimpleRouter` call. [2](#0-1) 

**Router calls `pool.swap` with itself as `msg.sender`:**

`exactInputSingle` (lines 72–80) calls `pool.swap(...)` directly, making the router the `msg.sender` at the pool level. The real user's address is stored only in transient callback context (`_setNextCallbackContext`) and is never passed to the pool or extension. [3](#0-2) 

The same pattern applies to `exactOutputSingle` (line 136), `exactInput` (line 104), and the recursive callback path in `_exactOutputIterateCallback` (line 220). [4](#0-3) [5](#0-4) 

**No existing guard corrects this:** The extension has no mechanism to decode the real user from `extensionData`, and the pool provides no alternative identity channel to extensions. [6](#0-5) 

## Impact Explanation

**Broken invariant A — allowlisted users cannot use the router (broken core swap flow):**
A pool admin allowlists `alice` via `setAllowedToSwap(pool, alice, true)`. Alice calls `router.exactInputSingle`. The pool passes `sender = router`. The extension checks `allowedSwapper[pool][router]` → `false` → reverts `NotAllowedToSwap`. Alice is permanently locked out of the primary supported periphery path despite being individually allowlisted. This is a broken core swap flow for every curated pool using `SwapAllowlistExtension`.

**Broken invariant B — allowlist bypass when the router is allowlisted:**
To restore router access, the pool admin adds the router: `setAllowedToSwap(pool, router, true)`. Now `allowedSwapper[pool][router]` is `true` for every caller routing through the router — including addresses never individually allowlisted. The entire curation policy collapses: any user bypasses the allowlist by calling `router.exactInputSingle` instead of `pool.swap` directly.

Both impacts are reachable through normal, unprivileged user actions on any pool with `SwapAllowlistExtension` wired as a `beforeSwap` hook. This constitutes broken core pool functionality causing an unusable swap flow and an access-control bypass.

## Likelihood Explanation

`SwapAllowlistExtension` is a production periphery extension explicitly designed for curated pools. Any pool that deploys it and expects users to interact through `MetricOmmSimpleRouter` (the primary supported periphery path) immediately hits invariant A. Invariant B follows as soon as the admin takes the natural remediation step of allowlisting the router. No special token behavior, no malicious setup, and no privileged attacker role is required — only a standard router call from any EOA.

## Recommendation

The extension must gate the economically relevant actor, not the immediate `pool.swap` caller. Two complementary fixes:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the real user (`msg.sender`) into `extensionData` for every hop, and document this as the canonical way to pass user identity to extensions.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should decode the real user from `extensionData` when present, falling back to `sender` only for direct pool calls. Alternatively, add a separate `allowedRouter` mapping so the extension can distinguish "this router is trusted to forward allowlisted users" from "this address is an allowlisted swapper."

## Proof of Concept

**Setup:**
- Pool deployed with `SwapAllowlistExtension` as `beforeSwap` hook.
- Pool admin allowlists `alice`: `setAllowedToSwap(pool, alice, true)`.

**Step 1 — Broken functionality (invariant A):**
```
alice → router.exactInputSingle({pool, ...})
  router → pool.swap(recipient, ...)   // msg.sender = router
    pool → extension.beforeSwap(sender=router, ...)
      allowedSwapper[pool][router] == false → revert NotAllowedToSwap
```
Alice is allowlisted but cannot use the router.

**Step 2 — Admin remediation that opens the bypass (invariant B):**
```
admin → setAllowedToSwap(pool, router, true)
```

**Step 3 — Bypass by non-allowlisted user:**
```
eve (not allowlisted) → router.exactInputSingle({pool, ...})
  router → pool.swap(recipient, ...)   // msg.sender = router
    pool → extension.beforeSwap(sender=router, ...)
      allowedSwapper[pool][router] == true → passes
```
Eve trades on the curated pool without ever being individually allowlisted, violating the pool's access-control invariant.
