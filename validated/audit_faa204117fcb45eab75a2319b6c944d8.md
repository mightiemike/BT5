### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Bypass of 64-Byte Merkle Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a coinbase Merkle proof check to block the well-known 64-byte transaction forgery attack. However, the old function is still declared `pub` and remains a live, callable NEAR contract method. Any unprivileged caller can invoke it directly, completely skipping the coinbase proof guard and obtaining a forged `true` inclusion result.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle forgery vector documented at https://www.bitmex.com/blog/64-Byte-Transactions. It validates a coinbase Merkle proof before delegating to the old logic: [1](#0-0) 

The guard it adds is: [2](#0-1) 

The deprecated function, however, performs no such check — it only computes the Merkle root from the caller-supplied `tx_id` and compares it to the stored root: [3](#0-2) 

Critically, `verify_transaction_inclusion` is still `pub` and carries only a Rust `#[deprecated]` attribute: [4](#0-3) 

`#[deprecated]` is a **compile-time lint** for Rust callers. It has zero effect on the NEAR ABI: the method is still exported and callable by any account on-chain. Any caller that wants to skip the coinbase proof check simply calls `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2`.

The `compute_root_from_merkle_proof` function in `merkle-tools` accepts any 32-byte value as the leaf: [5](#0-4) 

Because Bitcoin's Merkle tree hashes 64-byte concatenations of child nodes, an internal node hash is itself a valid 32-byte value that can be supplied as `tx_id`. Without the coinbase anchor, there is no way to distinguish a real transaction leaf from an internal node.

---

### Impact Explanation

A consumer contract (bridge, atomic swap, cross-chain lending protocol) that calls `verify_transaction_inclusion` receives a forged `true` for a transaction that does not exist. The attacker can fabricate proof of a Bitcoin payment and trigger fund release on the NEAR side. The contract's own documentation acknowledges this:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [6](#0-5) 

**Impact: Medium** — direct financial loss is possible for any downstream consumer that has not independently migrated to `verify_transaction_inclusion_v2`.

---

### Likelihood Explanation

**Likelihood: Medium** — the entry point requires no privileges; any NEAR account can call it. The 64-byte forgery technique is publicly documented and the required crafted inputs are computable offline. The only friction is that the attacker must find a downstream consumer still using the v1 method, which is realistic given the function remains live and the deprecation is invisible at the protocol level.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with a hard `require!(false, "use verify_transaction_inclusion_v2")` panic so it cannot be called on-chain. A Rust `#[deprecated]` attribute alone is insufficient to prevent on-chain invocation.

---

### Proof of Concept

1. Identify a Bitcoin block whose Merkle tree has at least two transactions. Let `N` be an internal node hash at depth 1 (i.e., `double_sha256(tx0 || tx1)`).
2. Construct a `ProofArgs` where:
   - `tx_id` = `N` (the internal node, 32 bytes)
   - `tx_block_blockhash` = hash of the target block (already in the light client)
   - `tx_index` = 0
   - `merkle_proof` = the remaining Merkle path from `N` up to the root (one element shorter than a normal proof)
   - `confirmations` = 1
3. Call `verify_transaction_inclusion(args)` directly on the NEAR contract.
4. `compute_root_from_merkle_proof(N, 0, proof)` reconstructs the correct Merkle root, so the function returns `true` — falsely asserting that `N` is an included transaction.
5. A bridge contract consuming this result releases funds for a Bitcoin payment that never occurred. [7](#0-6) [8](#0-7)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
