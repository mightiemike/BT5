### Title
`verify_transaction_inclusion_v2` Always Panics for Single-Transaction Blocks Due to Missing Empty-Proof Guard Before Delegating to `verify_transaction_inclusion` - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` unconditionally delegates to the deprecated `verify_transaction_inclusion`, which contains a hard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. For a single-transaction (coinbase-only) block, the correct and only valid merkle proof is an empty vector. `verify_transaction_inclusion_v2` does not check for this edge case before delegating, so every legitimate call to verify a transaction in a single-transaction block panics and reverts, making the recommended non-deprecated API permanently broken for that class of blocks.

### Finding Description

`verify_transaction_inclusion_v2` is the recommended replacement for the deprecated `verify_transaction_inclusion`. It adds a coinbase merkle proof check to mitigate the 64-byte transaction forgery vulnerability. [1](#0-0) 

The function:
1. Checks `merkle_proof.len() == coinbase_merkle_proof.len()` — passes when both are empty (0 == 0).
2. Verifies the coinbase proof via `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof)`. [2](#0-1) 

When `coinbase_merkle_proof` is empty, `compute_root_from_merkle_proof` skips the loop entirely and returns `coinbase_tx_id` unchanged. For a single-transaction block the merkle root **is** the coinbase transaction hash, so this check passes correctly.

3. Then delegates unconditionally to `verify_transaction_inclusion(args.into())`. [3](#0-2) 

Inside `verify_transaction_inclusion`, the very first substantive guard after the confirmation checks is: [4](#0-3) 

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

Because `ProofArgsV2::into()` maps `merkle_proof` directly: [5](#0-4) 

the empty `merkle_proof` from the v2 call is forwarded verbatim, and `require!` panics unconditionally. There is no path through `verify_transaction_inclusion_v2` that can succeed for a single-transaction block.

### Impact Explanation

Any consumer of the NEAR contract that calls `verify_transaction_inclusion_v2` to verify a transaction in a coinbase-only block receives a panic/revert instead of a boolean result. Because `verify_transaction_inclusion` is deprecated (since 0.5.0) and carries the 64-byte forgery warning, downstream integrators are expected to use v2 exclusively. The proof result for an entire class of valid Bitcoin blocks is permanently broken through the recommended API entry point.

### Likelihood Explanation

Coinbase-only blocks occur regularly on Bitcoin mainnet during low-fee periods and are the norm on many altcoin chains supported by this contract (Dogecoin, Litecoin). Any relayer or cross-chain bridge that submits such a block and then attempts to verify its coinbase transaction via `verify_transaction_inclusion_v2` will trigger the panic on every attempt.

### Recommendation

Add an early-return or bypass of the empty-proof guard inside `verify_transaction_inclusion_v2` before delegating, mirroring the fix pattern from the referenced report:

```rust
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
        "Coinbase merkle proof and transaction merkle proof should have the same length"
    );

    let header = self
        .headers_pool
        .get(&args.tx_block_blockhash)
        .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(),
            0usize,
            &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );

    // Single-transaction block: merkle root == tx hash, proof is legitimately empty.
    if args.merkle_proof.is_empty() {
        return args.tx_id == header.block_header.merkle_root;
    }

    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())
}
```

### Proof of Concept

1. Initialize the contract with a block whose merkle root equals a known coinbase tx hash `C` (i.e., a single-transaction block).
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = C`
   - `tx_block_blockhash = <that block's hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = C`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. Observe: length check passes (0 == 0); coinbase proof check passes (`C == merkle_root`); delegation to `verify_transaction_inclusion` is made with `merkle_proof = []`; `require!(!args.merkle_proof.is_empty(), ...)` fires and the transaction panics with `"Merkle proof is empty"`.     transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
