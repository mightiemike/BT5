### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain Without USDC Payment — (`core/contracts/ContractOwner.sol`)

### Summary
`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token returns `false` on failure rather than reverting, the function continues execution, withdrawing usdcE from the victim's Direct Deposit Account (DDA) and sending it to the caller — without the caller ever providing USDC.

### Finding Description
`replaceUsdcEWithUsdc` is an externally callable function (no access control beyond a chain ID check) intended to let any caller migrate a subaccount's usdcE balance to USDC. The intended flow is:

1. Transfer USDC from `msg.sender` into the DDA.
2. Withdraw usdcE from the DDA to `ContractOwner`.
3. Forward usdcE from `ContractOwner` to `msg.sender`.

At step 1, the code makes a direct interface call:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value (`bool`) is silently discarded. [1](#0-0) 

Steps 2 and 3 then unconditionally execute regardless of whether step 1 succeeded:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [2](#0-1) 

This contrasts with the rest of the codebase, which consistently uses `ERC20Helper.safeTransfer` / `safeTransferFrom` (low-level calls that check both `success` and the decoded bool return value):

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [3](#0-2) 

The `IERC20Base` interface declares `transferFrom` as returning `bool`, but the call site ignores it entirely. [4](#0-3) 

### Impact Explanation
If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073) returns `false` on a failed `transferFrom` rather than reverting — for example, when the caller has no USDC balance or has not approved the contract — the caller receives the full usdcE balance of the victim's DDA for free. The DDA's usdcE is permanently drained. The corrupted asset delta is: victim DDA loses `balance` usdcE; caller gains `balance` usdcE; no USDC is transferred.

### Likelihood Explanation
The function is callable by any unprivileged external address on chain 57073 (no `onlyOwner` guard). [5](#0-4)  The likelihood of exploitation is **low** because standard USDC implementations revert on failure rather than returning `false`. However, the code defect is real and the exploitability is entirely contingent on the specific token behavior at the hardcoded address — a property Nado's code does not verify or enforce.

### Recommendation
Replace the bare interface call with the `safeTransferFrom` wrapper already used throughout the codebase:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This ensures the transaction reverts if the transfer fails for any reason, consistent with how all other token transfers in the protocol are handled. [6](#0-5) 

### Proof of Concept
1. Identify a subaccount whose DDA holds a non-zero usdcE balance on chain 57073.
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an address with zero USDC balance (or zero approval).
3. If the USDC token returns `false` instead of reverting, `transferFrom` silently fails.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, moving usdcE to `ContractOwner`. [7](#0-6) 
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` sends the full usdcE amount to the attacker.
6. Attacker has received usdcE without providing any USDC; victim's DDA is drained.
