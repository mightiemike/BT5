### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` Silently Downgrades `AllResources` to `L1Gas` When L2/L1_DATA Gas Are Zero, Producing a Different Transaction Hash After P2P Deserialization — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic that silently converts `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` produces structurally different hash preimages for these two variants (3 vs. 4 elements), the transaction hash computed on the originating node diverges from the hash recomputed on any node that received the transaction via P2P state-sync. The mismatch is silent — no error is returned.

### Finding Description

**Root cause — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The variant is inferred from the *values* of the fields, not from any discriminator. A transaction originally carrying `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is round-tripped to `L1Gas(X)` after one protobuf encode/decode cycle.

**Why the hash changes — `get_tip_resource_bounds_hash`** [2](#0-1) 

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
// For new V3 txs, need to also hash the data gas bounds.
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ← 3 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 4 elements
    }
});
```

| Variant at hash time | Elements hashed | Result |
|---|---|---|
| `AllResources(l1=X, l2=0, l1_data=0)` | `[tip, L1_GAS, L2_GAS(0), L1_DATA_GAS(0)]` | H1 |
| `L1Gas(X)` | `[tip, L1_GAS, L2_GAS(0)]` | H2 ≠ H1 |

This hash feeds directly into `get_invoke_transaction_v3_hash`, `get_declare_transaction_v3_hash`, and `get_deploy_account_transaction_v3_hash`. [3](#0-2) 

**The state-sync deserialization path that triggers the downgrade** [4](#0-3) 

```rust
impl TryFrom<protobuf::InvokeV3> for InvokeTransactionV3 {
    fn try_from(value: protobuf::InvokeV3) -> Result<Self, Self::Error> {
        let resource_bounds = ValidResourceBounds::try_from(
            value.resource_bounds.ok_or(missing("InvokeV3::resource_bounds"))?,
        )?;
```

This is the converter used for `TransactionInBlock` (state-sync) messages. The consensus path uses a separate `AllResourceBounds::try_from` in `rpc_transaction.rs` that always preserves `AllResources`, but the state-sync path does not. [5](#0-4) 

**The developers are aware of the ambiguity** — the consensus test explicitly works around it: [6](#0-5) 

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);
```

The workaround is applied only in the test; the production state-sync path has no such guard.

**Gateway accepts the triggering transaction**

The stateless validator accepts V3 transactions with `AllResourceBounds { l1_gas: NON_EMPTY, l2_gas: default(), l1_data_gas: default() }`: [7](#0-6) 

### Impact Explanation

After P2P state-sync, the stored `InvokeTransactionV3` carries `L1Gas(X)` instead of `AllResources(X, 0, 0)`. Any subsequent call to `calculate_transaction_hash` or `validate_transaction_hash` on the stored object produces H2 ≠ H1. Concretely:

- **`validate_transaction_hash`** (used by the feeder-gateway client and RPC verification) returns `false` for a valid transaction, causing the node to treat a legitimate transaction as having an invalid hash.
- **RPC simulation / tracing / `get_transaction_by_hash`** that recomputes the hash from the stored transaction returns an authoritative-looking wrong hash to callers.
- Any component that re-derives the transaction hash from the stored `InvokeTransactionV3` (e.g., for block-hash re-verification) will compute the wrong value silently.

### Likelihood Explanation

A V3 invoke transaction with non-zero L1 gas but zero L2 and L1_DATA gas is a valid, accepted transaction (pre-0.13.3 style). Any such transaction included in a block and synced via P2P triggers the downgrade. No special privileges are required; a normal user submitting a standard V3 transaction with only L1 gas bounds set is sufficient.

### Recommendation

Replace the value-based heuristic with a discriminator. The simplest fix is to use `AllResourceBounds::try_from` (which always returns `AllResources`) in the `InvokeTransactionV3` state-sync deserializer, matching the behavior already used in the consensus path:

```rust
// In TryFrom<protobuf::InvokeV3> for InvokeTransactionV3
let resource_bounds = ValidResourceBounds::AllResources(
    AllResourceBounds::try_from(
        value.resource_bounds.ok_or(missing("InvokeV3::resource_bounds"))?,
    )?
);
```

Alternatively, add an explicit variant tag to the `ResourceBounds` protobuf message so the deserializer can reconstruct the correct variant without inspecting field values.

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. The gateway accepts it. `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash preimage has 4 elements → hash **H1** is stored.
3. The transaction is included in a block. The block is serialized to `TransactionInBlock` protobuf and sent to a syncing peer.
4. The syncing peer calls `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` → `ValidResourceBounds::try_from` → `l2_gas.is_zero() && l1_data_gas.is_zero()` → returns `L1Gas(1000@1)`.
5. The syncing peer calls `calculate_transaction_hash` on the stored transaction → `get_tip_resource_bounds_hash` with `L1Gas` → hash preimage has 3 elements → hash **H2**.
6. **H1 ≠ H2**. `validate_transaction_hash(stored_tx, ..., H1, ...)` returns `false`; any RPC endpoint that recomputes the hash returns H2 instead of H1.
