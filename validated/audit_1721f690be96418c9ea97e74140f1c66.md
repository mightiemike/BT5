### Title
P2P Sync Client Accepts Attacker-Controlled `protocol_version` Without Block-Hash or Signature Verification, Causing Wrong `VersionedConstants` for RPC Execution — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

The p2p sync client accepts `SignedBlockHeader` messages from peers and stores the peer-supplied `protocol_version` as `starknet_version` without ever verifying the block hash or the sequencer signature. A malicious peer can inject any valid `StarknetVersion` string (e.g., `'0.13.0'` for a block that is actually `'0.13.2'`), causing the node to store the wrong version and subsequently select the wrong `VersionedConstants` for all RPC execution calls (`estimate_fee`, `simulate_transactions`, `call`) against that block.

### Finding Description

**Step 1 — Deserialization accepts any valid version string.**

`TryFrom<protobuf::SignedBlockHeader>` in `converters/header.rs` lines 126–134 calls `StarknetVersion::try_from(value.protocol_version)`. Any syntactically valid version string (e.g., `"0.13.0"`) succeeds and is stored verbatim as `starknet_version` in the resulting `BlockHeader`. There is no check that the version matches the real on-chain value. [1](#0-0) 

**Step 2 — The p2p sync client performs no block-hash or signature verification.**

`parse_data_for_block` in `apollo_p2p_sync/src/client/header.rs` only checks (a) that the block number is in order and (b) that exactly one signature is present. It does **not** recompute the block hash from the header fields and compare it against the provided `block_hash`, and it does **not** call `verify_block_signature` against the known sequencer public key. The `TODO` comment about checking `parent_hash` is also unimplemented. [2](#0-1) 

**Step 3 — The wrong version is written to the `starknet_version` storage table.**

`write_to_storage` calls `append_header`, which calls `update_starknet_version` with `block_header.block_header_without_hash.starknet_version` — the attacker-controlled value. [3](#0-2) 

**Step 4 — RPC execution reads the stored version and selects `VersionedConstants`.**

`apollo_rpc_execution/src/lib.rs` reads `starknet_version` from the `starknet_version` table (line 370–373) and immediately passes it to `VersionedConstants::get(&starknet_version)` (line 408) to build the `BlockContext` used for fee estimation and simulation. [4](#0-3) 

**Why the block-hash guard does not protect here.**

`starknet_version` is a committed field in the block hash (it is hashed at position 12 in the Poseidon chain in `calculate_block_hash`). [5](#0-4) 

The sequencer signature covers `Poseidon(block_hash, state_diff_commitment)`. [6](#0-5) 

An attacker who controls a peer can supply a self-consistent but fake `(block_hash, protocol_version, signature)` triple — the client never checks any of them against an authoritative source.

### Impact Explanation

Any RPC call that targets the poisoned block — `starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_call` — will execute under the wrong `VersionedConstants`. The concrete difference between, e.g., `V0_13_0` (`invoke_tx_max_n_steps = 3_000_000`) and `V0_13_2` (`invoke_tx_max_n_steps = 10_000_000`) is large enough to flip fee estimates and simulation results, producing authoritative-looking wrong values to callers. [7](#0-6) 

### Likelihood Explanation

Any node reachable on the p2p network can act as a peer. The attacker needs only to (a) establish a p2p connection to the victim node and (b) respond to a `BlockHeadersRequest` with a crafted `SignedBlockHeader`. No operator, validator, or admin privilege is required. The victim node will accept the response as long as the block number is in sequence and exactly one signature element is present.

### Recommendation

1. **Verify the block hash**: After deserializing a `SignedBlockHeader`, recompute the block hash from the received fields (using `calculate_block_hash`) and reject the peer if it does not match the provided `block_hash`.
2. **Verify the sequencer signature**: Call `verify_block_signature` against the known sequencer public key before accepting any header. The central sync path already has this infrastructure.
3. **Implement the `parent_hash` check**: The existing `TODO` comment in `parse_data_for_block` should be resolved to prevent chain-level forgery.
