Based on my analysis of the codebase, I can identify a direct structural analog to the Opus M-05 bug.

### Title
Permissionless `addLiquidity` Allows Any Allowlisted Sender to Mint Shares Into Arbitrary Non-Allowlisted Owner Positions, Bypassing `DepositAllowlistExtension` — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no check that `msg.sender == owner`. The `_beforeAddLiquidity` hook forwards `msg.sender` as `sender` and the caller-supplied address as `owner` to the `DepositAllowlistExtension`. Because the allowlist guard evaluates `sender` (the initiating caller), an allowlisted address can mint shares into any non-allowlisted address's position key, granting that address pool participation it was never approved for.

### Finding Description

`removeLiquidity` correctly enforces ownership:

```solidity
// MetricOmmPool.sol line 206
if (msg.sender != owner) revert NotPositionOwner();
```

`addLiquidity` has no equivalent guard:

```solidity
// MetricOmmPool.sol lines 188–196
function addLiquidity(
    address owner,          // ← arbitrary, caller-supplied
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, ...   // shares keyed to owner
    );
    _afterAddLiquidity(msg.sender, owner, salt, ...);
}
```

The extension hook encodes `msg.sender` as `sender` and the arbitrary `owner` as a separate field:

```solidity
// ExtensionCalling.sol lines 95–98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity,
        (sender, owner, salt, deltas, extensionData))
);
```

If `DepositAllowlistExtension` gates on `sender` (the standard pattern for a deposit allowlist), an allowlisted caller passes the check while the `owner` whose position receives the shares is never evaluated. Shares are then keyed to `(owner, salt)` in `_positionBinShares`. The non-allowlisted `owner` can subsequently call `removeLiquidity` — which only checks `msg.sender == owner`, not the allowlist — and withdraw real tokens from the pool.

### Impact Explanation

The `DepositAllowlistExtension` is the pool's primary access-control boundary for liquidity provision. Bypassing it for the position recipient means a non-approved address can hold and redeem pool shares, directly extracting token0/token1 from the pool. This is a broken core pool functionality / admin-boundary break with direct loss of LP assets to non-approved participants.

### Likelihood Explanation

Any single allowlisted address (e.g., a router, an integrator, or a compromised LP) can trigger this for any target `owner`. The attacker supplies the tokens via the swap callback and the victim receives the shares; the attacker's cost is the deposited capital, but the victim gains unrestricted pool access. Likelihood is medium: it requires a willing allowlisted initiator, but no privileged role or special setup beyond pool deployment with a `DepositAllowlistExtension`.

### Recommendation

Add the same ownership check to `addLiquidity` that already exists in `removeLiquidity`:

```solidity
function addLiquidity(address owner, ...) external ... {
    if (msg.sender != owner) revert NotPositionOwner(); // add this
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    ...
}
```

Alternatively, the `DepositAllowlistExtension` must check **both** `sender` and `owner` against the allowlist so that neither the initiating caller nor the receiving position owner can be non-approved.

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; Alice is allowlisted, Bob is not.
2. Alice calls `pool.addLiquidity(bob, salt, deltas, callbackData, extensionData)`.
3. `_beforeAddLiquidity(alice, bob, ...)` fires; extension checks `sender = alice` → allowlisted → passes. Bob's address is never evaluated.
4. `LiquidityLib.addLiquidity` credits shares to position key `(bob, salt)`.
5. Alice's tokens are pulled via the swap callback.
6. Bob calls `pool.removeLiquidity(bob, salt, deltas, extensionData)`.
7. `msg.sender == owner` check passes (Bob == Bob). No allowlist hook fires on remove for Bob.
8. Bob receives token0/token1 from the pool without ever being on the allowlist. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
