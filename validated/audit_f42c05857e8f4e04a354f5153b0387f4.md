### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double-Claiming Bonus Rewards from the Rewards Pool — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before any processing begins. Because there is no deduplication check on `msg.PositionIds`, a position owner can include the same position ID multiple times in a single transaction. Each duplicate entry in the slice carries the same pre-processing `LastEventSeq` checkpoint, causing `processEventsAndClaimBonus` to re-process the same validator events and transfer bonus coins from the `RewardsPoolName` module account a second (or Nth) time.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` iterates over `msg.PositionIds`, loads each position from state, and appends it to a local slice: [1](#0-0) 

The slice is then passed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, each element of the slice is processed independently: [3](#0-2) 

For each element, `processEventsAndClaimBonus` walks validator events starting from `pos.LastEventSeq`: [4](#0-3) 

It computes a bonus and immediately transfers it from the module account to the owner: [5](#0-4) 

When `msg.PositionIds = [X, X]`, both slice entries are loaded from state before any processing, so both carry the same `LastEventSeq`. The first iteration processes all pending events, updates the in-memory copy, and calls `setPosition` to persist the new checkpoint. The second iteration operates on the **stale in-memory snapshot** (same `LastEventSeq`), re-processes the same events, and issues a second `SendCoinsFromModuleToAccount` transfer. The on-chain state written by the first `setPosition` call is never re-read for the duplicate entry.

A secondary corruption also occurs: `decrementEventRefCount` is called once per event per iteration, so duplicate processing decrements each event's reference count twice, potentially causing premature garbage collection of events that other positions have not yet consumed. [6](#0-5) 

---

### Impact Explanation

The `RewardsPoolName` module account is drained by a multiple of the legitimate bonus entitlement for each duplicated position ID. With N copies of the same position ID, the attacker receives N× the bonus owed. This directly corrupts the bonus pool balance, reducing rewards available to all other participants. The corrupted value is the `RewardsPoolName` module account balance and the `LastEventSeq` / event reference counts stored per validator.

---

### Likelihood Explanation

Any position owner can craft a `MsgClaimTierRewards` transaction with a repeated position ID. No special privilege, leaked key, or social engineering is required. The transaction is a standard Cosmos SDK message submitted through the normal CLI or gRPC path. The only limiting factor is whether the bonus pool holds sufficient balance to satisfy the repeated transfer; if it does, every duplicate succeeds.

---

### Recommendation

Add a deduplication check on `msg.PositionIds` before the processing loop in `ClaimTierRewards`. The simplest fix is to build a `map[uint64]struct{}` of seen IDs and return an error if any ID appears more than once. Alternatively, enforce uniqueness inside `MsgClaimTierRewards.Validate()` so the invariant is checked at the message-validation layer before any keeper logic runs.

---

### Proof of Concept

1. Alice creates a tiered-rewards position with ID `42` and accumulates bonus rewards.
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42, 42]}`.
3. `ClaimTierRewards` loads position 42 three times into the `positions` slice; all three copies share the same `LastEventSeq`.
4. `claimRewardsAndUpdatesPositions` iterates:
   - **Iteration 1**: processes pending validator events for position 42, computes bonus B, transfers B from `RewardsPoolName` to Alice, calls `setPosition` with updated checkpoint.
   - **Iteration 2**: operates on the stale in-memory copy (old `LastEventSeq`), re-processes the same events, computes bonus B again, transfers B from `RewardsPoolName` to Alice a second time.
   - **Iteration 3**: same as iteration 2, transfers B a third time.
5. Alice receives 3B instead of B, draining the bonus pool by 2B beyond her entitlement.
6. Each event's reference count is decremented three times instead of once, corrupting the event lifecycle for other positions delegated to the same validator.

```go
func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
```
