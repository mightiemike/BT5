### Title
`LastKnownBonded` Hardcoded to `true` in `createDelegatedPosition` Enables Bonus Overpayment via `MsgCommitDelegationToTier` on Unbonded Validator — (`x/tieredrewards/keeper/position.go`)

---

### Summary

`createDelegatedPosition` unconditionally initialises `LastKnownBonded = true` for every new position. `MsgCommitDelegationToTier` — unlike `MsgLockTier` — carries no "validator bonded" precondition, so a user can commit an existing delegation to an unbonded validator. The resulting position starts with an incorrect bonded-state assumption. When the validator later rebonds, `processEventsAndClaimBonus` treats the entire gap between position creation and the BOND event as a bonded segment and pays out bonus rewards for a period during which the validator was actually unbonded.

---

### Finding Description

**Root cause — hardcoded initial state in `createDelegatedPosition`:**

```go
// x/tieredrewards/keeper/position.go  line 56
pos := types.NewPosition(id, owner, tier.Id, delAddr.String(),
    blockHeight, lastEventSeq, blockTime,
    true,       // ← LastKnownBonded hardcoded to true
    blockTime)
```

`LastKnownBonded` is the seed value for the bonded-state machine in `processEventsAndClaimBonus`. It is set to `true` regardless of whether the target validator is actually bonded at position-creation time.

**Missing precondition in `MsgCommitDelegationToTier`:**

The ADR documents the validation rules for each message:

| Message | "validator bonded" check |
|---|---|
| `MsgLockTier` | ✓ required |
| `MsgCommitDelegationToTier` | ✗ absent |

`MsgCommitDelegationToTier` only validates `amount <= user's delegation`, `amount >= MinLockAmount`, tier not close-only, and delegator not a vesting account. A user whose existing delegation targets an unbonded validator can therefore call this message successfully.

**How the incorrect state propagates through `processEventsAndClaimBonus`:**

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 164-213
bonded := pos.LastKnownBonded   // ← starts as true even if validator is unbonded
segmentStart := pos.LastBonusAccrual

for _, entry := range events {
    evt := entry.Event
    if bonded {
        // bonus computed for this segment — no IsBonded() guard here
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        totalBonus = totalBonus.Add(bonus)
    }
    switch evt.EventType {
    case UNBOND: bonded = false
    case BOND:   bonded = true
    case SLASH:  // unchanged
    }
    segmentStart = evt.Timestamp
}

// Defensive guard exists ONLY for the final open segment
if bonded && val.IsBonded() { ... }
```

Because `LastEventSeq` is set to the latest event sequence at position-creation time, the UNBOND event that put the validator into the unbonded state is already behind the position's cursor — it will never be replayed. The first event the position will ever see is the future BOND event when the validator rejoins. At that point the loop body executes with `bonded = true`, computing bonus for the entire interval `[positionCreationTime, bondTime]` — a period during which the validator was unbonded and no bonus should accrue.

The `val.IsBonded()` guard on line 206 only protects the trailing open segment; it does not protect intermediate segments between stored events.

---

### Impact Explanation

Bonus rewards are paid from the module's `RewardsPoolName` account. Every incorrectly computed segment drains the pool by:

```
overpayment = shares × tokensPerShare × bonusApy × unbondedDurationSeconds / SecondsPerYear
```

For a 1 000 000-token position at 4 % APY over a 30-day unbonded window this is ≈ 3 288 tokens extracted from the pool per position. Multiple positions on the same unbonded validator multiply the drain. The pool is a finite shared resource; premature exhaustion causes all legitimate bonus claims to fail atomically (`ErrInsufficientBonusPool`), blocking every honest staker from collecting earned rewards until the pool is replenished by governance.

---

### Likelihood Explanation

The trigger is a normal, permissionless user action: calling `MsgCommitDelegationToTier` with a delegation whose validator is currently jailed/unbonded. Validators are jailed for double-signing or extended downtime — both routine on a live chain. A user whose validator was jailed after they delegated can immediately exploit this without any privileged access or coordination. The message is reachable via CLI, gRPC, and REST.

---

### Recommendation

1. **Check validator bonded status in `MsgCommitDelegationToTier`** — add the same "validator bonded" precondition that `MsgLockTier` already enforces.
2. **Read actual bonded state in `createDelegatedPosition`** — instead of hardcoding `true`, query `stakingKeeper.GetValidator` and pass the real bonded state to `NewPosition`.
3. **Add an `IsBonded()` guard to intermediate segments** — or, equivalently, verify that `LastKnownBonded` is always consistent with the validator's live state before entering the event-replay loop.
