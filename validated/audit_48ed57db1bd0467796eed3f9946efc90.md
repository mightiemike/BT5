### Title
`nonDefaultFeeTierMask` Incorrectly Bounded to 128 Tiers Silently Discards All Custom Fee Rates for Tiers ≥ 128 — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`nonDefaultFeeTierMask` is declared as `uint128`, capping the effective fee-tier namespace to 128 entries (bits 0–127). When `updateTierFeeRates` is called with `txn.tier >= 128`, the expression `uint128(1) << txn.tier` evaluates to `0` in Solidity 0.8.x (silent truncation, no revert), so the mask is never updated. Consequently, `getTierFeeRateX18` always falls through to the hardcoded default rate (2 bps taker / 0 maker) for every tier ≥ 128, even though the rates were stored in `feeRates[tier][productId]`. The stored configuration is permanently unreachable.

---

### Finding Description

**Root cause — `updateTierFeeRates`:** [1](#0-0) 

```solidity
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
```

`txn.tier` is `uint32`. In Solidity 0.8.x, shifting a `uint128` value by a count ≥ 128 produces `0` without reverting. Therefore, for any `txn.tier >= 128`:

```
uint128(1) << 128  →  0
nonDefaultFeeTierMask |= 0  →  mask unchanged
```

The fee rates are written to `feeRates[txn.tier][productId]` correctly, but the sentinel bit that marks the tier as non-default is never set.

**Root cause — `getTierFeeRateX18`:** [2](#0-1) 

```solidity
if (nonDefaultFeeTierMask & (1 << tier) != 0) {
    return feeRates[tier][productId];
}
return FeeRates({
    makerRateX18: 0,
    takerRateX18: 200_000_000_000_000 // 2 bps
});
```

Because the mask bit for `tier >= 128` is never set, the condition is always `false`, and the function always returns the hardcoded default. The custom rates stored in `feeRates` are permanently shadowed.

**Fee tier assignment path (sequencer-driven):** [3](#0-2) 

The sequencer submits `UpdateFeeTier` transactions that call `updateFeeTier(user, newTier)` with an arbitrary `uint32` tier value. There is no on-chain enforcement that `newTier < 128`.

**Type declaration:** [4](#0-3) 

```solidity
uint128 internal nonDefaultFeeTierMask;
```

---

### Impact Explanation

Any user assigned a fee tier ≥ 128 by the sequencer will always be charged the hardcoded default taker rate of 2 bps and receive 0 maker rebate, regardless of what custom rates the owner configured for that tier. Concretely:

- If the owner configured a VIP tier (e.g., tier 200) with a 0 bps taker rate, VIP users are silently overcharged 2 bps on every trade.
- If the owner configured a punitive tier with a higher rate, those users are undercharged.
- The `feeRates[tier][productId]` storage is written but permanently unreachable, making the entire high-tier fee configuration system non-functional above tier 127.
- Fee revenue collected by the protocol (`market.collectedFees`, later swept to `X_ACCOUNT` via `dumpFees`) is systematically incorrect for all affected users. [5](#0-4) 

---

### Likelihood Explanation

The protocol already supports `uint32` tier IDs with no upper bound enforced at the assignment site. As the protocol scales and adds more fee tiers (e.g., per-builder tiers, institutional tiers, tiered rebate programs), tier values ≥ 128 are a natural progression. The owner has no on-chain signal that `updateTierFeeRates` silently failed to register the tier — the transaction succeeds, rates are stored, but the mask is never updated. This is a silent misconfiguration that would be extremely difficult to detect without off-chain monitoring of the mask state.

---

### Recommendation

Change `nonDefaultFeeTierMask` from `uint128` to `uint256` to support the full `uint32` tier space (up to 256 tiers with a bitmask, or use a `mapping(uint32 => bool)` for unbounded support):

```solidity
// Option A: extend to uint256 (supports tiers 0–255)
uint256 internal nonDefaultFeeTierMask;

// Option B: unbounded (supports all uint32 tiers)
mapping(uint32 => bool) internal isNonDefaultFeeTier;
```

Update both `updateTierFeeRates` and `getTierFeeRateX18` consistently. Add an explicit `require(txn.tier < 256, "tier out of range")` guard if Option A is chosen, so misconfiguration fails loudly rather than silently.

---

### Proof of Concept

1. Owner submits slow-mode `UpdateTierFeeRates{tier: 200, makerRateX18: 0, takerRateX18: 0}` — intending to grant zero-fee trading to VIP tier 200.
2. On-chain: `feeRates[200][productId] = FeeRates(0, 0)` is stored. But `uint128(1) << 200 == 0`, so `nonDefaultFeeTierMask` is unchanged (bit 200 is never set).
3. Sequencer submits `UpdateFeeTier{user: alice, newTier: 200}` — Alice is assigned tier 200.
4. Alice submits a `MatchOrders` trade. `applyFee` calls `getTierFeeRateX18(200, productId)`.
5. `nonDefaultFeeTierMask & (1 << 200)` — bit 200 was never set → condition is `false` → returns default `FeeRates(0, 200_000_000_000_000)`.
6. Alice is charged 2 bps taker fee despite the owner having configured 0 bps for her tier. The fee is credited to `market.collectedFees` and eventually swept to `X_ACCOUNT`, constituting a systematic overcharge on every trade Alice executes. [6](#0-5) [7](#0-6)
