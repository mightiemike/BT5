### Title
`ValidResourceBounds` Protobuf Deserialization Misclassifies `AllResources` as `L1Gas`, Producing Wrong Transaction Hash ŌĆö (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-`ValidResourceBounds` converter uses a value-based heuristic (`l2_gas.is_zero() && l1_data_gas.is_zero()`) to decide whether a received transaction carries `L1Gas` (pre-0.13.3) or `AllResources` (post-0.13.3) bounds. A post-0.13.3 `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero is silently downgraded to `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` conditionally omits the `L1_DATA_GAS` felt from the hash preimage for `L1Gas` variants, the hash recomputed by the receiving node diverges from the hash the sender computed and signed over. This is a direct analog of the LSSVMPairETH conditional-skip bug: a branch that was meant to handle a legacy case silently drops a field from an accounting/hash computation, producing a wrong value.

---

### Finding Description

**Root cause ŌĆö protobuf converter (`transaction.rs` lines 417ŌĆō436):**

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // ...
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ŌåÉ wrong for post-0.13.3 txs
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

Any post-0.13.3 `AllResources` transaction whose `l2_gas = 0` and `l1_data_gas = 0` is converted to `ValidResourceBounds::L1Gas` on the receiving side.

**Hash divergence ŌĆö `get_tip_resource_bounds_hash` (`transaction_hash.rs` lines 188ŌĆō211):**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1_DATA_GAS omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // included
    }
});
```

`get_concat_resource` encodes the 7-byte ASCII resource name `b"L1_DATA"` into the felt even when `max_amount = 0` and `max_price_per_unit = 0`, so the felt is **non-zero**. The two hash preimages therefore differ in length and content:

| Variant | Preimage elements |
|---|---|
| `AllResources` (l1_data_gas=0) | `[tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_felt]` |
| `L1Gas` | `[tip, L1_GAS_felt, L2_GAS_felt]` |

The final Poseidon hash over these two distinct sequences is different.

**Serialization round-trip confirms the mismatch.** The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer always emits `l1_data_gas = Some(default)` for both variants:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),  // zero, but present
},
```

So a node that serializes an `AllResources` (l2_gas=0, l1_data_gas=0) transaction and a node that serializes an `L1Gas` transaction produce **identical protobuf bytes**. The deserializer cannot distinguish them and always produces `L1Gas`, breaking the round-trip for the `AllResources` case.

---

### Impact Explanation

When a transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is propagated over P2P:

1. The **sending node** computed and stored `tx_hash_A` using `AllResources` (L1_DATA_GAS felt included).
2. The **receiving node** deserializes the protobuf, obtains `L1Gas`, recomputes `tx_hash_B` (L1_DATA_GAS felt absent).
3. `tx_hash_A ŌēĀ tx_hash_B`.

Consequences:
- Any hash-validation step on the receiving side (e.g., `validate_transaction_hash`) rejects a legitimately signed transaction ŌĆö **valid transaction rejected before sequencing** (High).
- If the hash is not re-validated but is re-derived from the deserialized object, the node stores the transaction under the wrong key, causing **wrong receipt/state/event lookup results** (Critical).
- The wrong hash is bound to the wrong executable payload, satisfying the "Transaction conversion or signature/hash logic binds the wrong hash or executable payload" criterion (High).

---

### Likelihood Explanation

The gateway's stateless validator explicitly accepts `AllResourceBounds` with `l2_gas = 0` and `l1_data_gas = 0` (only one non-zero resource bound is required). Any user submitting a transaction that specifies only L1 gas under the post-0.13.3 `AllResources` encoding triggers this path. No special privilege is required; the transaction is valid and accepted by the gateway.

---

### Recommendation

1. **Add an explicit discriminant to the protobuf schema** ŌĆö e.g., a boolean `is_all_resources` field or a version tag ŌĆö so the deserializer does not rely on value-based heuristics.
2. **Remove the `unwrap_or_default` fallback** once 0.13.2 support is dropped (as the existing TODO notes), and instead require `l1_data_gas` to be present for all post-0.13.3 transactions.
3. **Assert the round-trip invariant** in tests: serialize an `AllResources` (l2_gas=0, l1_data_gas=0) value, deserialize it, and assert the result is still `AllResources`.
