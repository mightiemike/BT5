### Title
Integer Overflow in `op_unknown` Cost Multiplier Multiplication Produces Undercharged Cost - (File: `src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` computes a final cost by multiplying a base cost (up to `max_cost`, ~11 billion on Chia mainnet) by `cost_multiplier + 1` (up to `2^32`). This multiplication is performed in unchecked `u64` arithmetic. In Rust release builds, overflow wraps silently. The post-multiplication guard only checks `cost > u32::MAX`, which is insufficient to detect a wrapped result. An attacker-controlled opcode byte sequence can cause the product to wrap to zero or a small value, making an expensive unknown op appear to cost nothing.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`), the final cost is computed as:

```rust
check_cost(cost, max_cost)?;   // line 260 — ensures cost ≤ max_cost
cost *= cost_multiplier + 1;   // line 261 — unchecked u64 multiplication
if cost > u32::MAX as u64 {    // line 262 — checks wrapped result only
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode bytes via `u32_from_u8`, so it is at most `u32::MAX = 4_294_967_295`, making `cost_multiplier + 1` at most `2^32 = 4_294_967_296`.

`cost` before the multiplication is bounded by `max_cost` (≈ 11 × 10⁹ on mainnet). The product can reach:

```
11 × 10⁹ × 2^32 ≈ 4.73 × 10¹⁹  >  u64::MAX ≈ 1.84 × 10¹⁹
```

In Rust release mode, this wraps modulo `2^64`. The post-multiplication check `cost > u32::MAX` only tests the *wrapped* value, not whether overflow occurred.

**Concrete overflow to zero:** With `cost_multiplier = u32::MAX` (so `cost_multiplier + 1 = 2^32`) and `cost` equal to any multiple of `2^32`, the product is an exact multiple of `2^64`, wrapping to `0`. Since `0 ≤ u32::MAX`, the function returns `Ok(Reduction(0, nil))` — a zero-cost unknown op.

**Achieving `cost ≡ 0 (mod 2^32)` with `cost_function = 2`:** The cost formula for cost_function 2 is:

```
cost = MUL_BASE_COST + MUL_COST_PER_OP + (l0 + l1) × 6 + (l0 × l1) / 128
```

With two equal-length atoms of size `l ≈ 724 KB`, the quadratic term `l²/128 ≈ 2^32`. The attacker can solve the congruence `977 + 12l + l²/128 ≡ 0 (mod 2^32)` offline to find the exact atom size needed. This is a deterministic, precomputable attack.

The size check `if bsize > 256 || esize > 256 || msize > 256` in `op_modpow` does not apply here; `op_unknown` has no per-argument size limit.

---

### Impact Explanation

An attacker can submit a CLVM program containing an unknown opcode (valid in lenient/consensus mode) with crafted opcode bytes and ~724 KB arguments. The cost computation overflows to 0 (or a small value), so the program is accepted even though its true cost should be billions of cost units — far above the block cost limit. This enables:

1. **Cost undercharging / fee bypass:** Programs that should be rejected for exceeding the cost limit are accepted for free.
2. **Network spam / DoS:** An attacker can flood the mempool with programs that appear cheap but consume significant node resources.
3. **Consensus risk:** Any implementation that computes cost correctly (e.g., a future reference implementation or a node running in debug mode where overflow panics) would disagree on program validity, causing a chain split.

The corrupted value is the `Cost` returned in `Reduction(cost, allocator.nil())`, which is directly used by `run_program` to track cumulative execution cost.

---

### Likelihood Explanation

The attack is fully deterministic and precomputable offline. The attacker needs only to:
- Set the opcode's top bytes to encode `cost_multiplier = u32::MAX` (4 bytes: `0xFF 0xFF 0xFF 0xFF`) and `cost_function = 2` (bits 7–6 of the last byte).
- Provide two atom arguments of a precomputed size (~724 KB each) to drive `cost` to the target multiple of `2^32`.

No special privileges, keys, or network access beyond submitting a transaction are required. The opcode is reachable via the standard `op_unknown` path in lenient mode, which is the production consensus mode for Chia.

---

### Recommendation

Replace the unchecked multiplication with an overflow-safe variant and reject on overflow:

```rust
let Some(new_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o));
};
cost = new_cost;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

Alternatively, use `saturating_mul` and let the existing `u32::MAX` guard handle the rejection, but `checked_mul` with an explicit error is cleaner and more auditable.

### Proof of Concept

**Opcode construction:**
- Opcode bytes: `[0xFF, 0xFF, 0xFF, 0xFF, 0x80]`
  - Bytes `[0..4]` = `0xFFFFFFFF` → `cost_multiplier = u32::MAX = 4294967295`
  - Last byte `0x80` → bits 7–6 = `10` → `cost_function = 2` (mul-like)
- `cost_multiplier + 1 = 2^32`

**Argument construction:**
- Solve `977 + 12l + l²/128 ≡ 0 (mod 2^32)` for integer `l`.
- Provide two atoms of exactly `l` bytes as arguments to the opcode.
- The loop in cost_function 2 computes `cost ≡ 0 (mod 2^32)`.

**Overflow:**
```
cost × (cost_multiplier + 1) = (k × 2^32) × 2^32 = k × 2^64 ≡ 0 (mod 2^64)
```
Wrapped cost = 0. Check `0 > u32::MAX` is false. Returns `Ok(Reduction(0, nil))`.

**Result:** The program is accepted with cost 0, bypassing any cost limit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
