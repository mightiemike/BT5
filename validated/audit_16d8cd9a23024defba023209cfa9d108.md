### Title
Unbounded `while` Loop in `tryUnlockNlpBalance` Permanently Locks NLP Tokens for Any Subaccount - (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

`tryUnlockNlpBalance` iterates over every expired entry in a subaccount's `NlpLockedBalanceQueue` in a single `while` loop. Because `balanceCount` grows by one for every NLP deposit made at a distinct oracle timestamp — with no cap — a subaccount that has accumulated enough queue entries will cause every subsequent NLP balance update to revert on gas exhaustion, permanently locking that subaccount's NLP tokens.

---

### Finding Description

`NlpLockedBalanceQueue` is a per-subaccount structure that records each NLP deposit as a time-locked entry:

```solidity
struct NlpLockedBalanceQueue {
    mapping(uint64 => NlpLockedBalance) balances;
    uint64 balanceCount;
    uint64 unlockedUpTo;
    Balance unlockedBalanceSum;
}
``` [1](#0-0) 

`handleNlpLockedBalance` appends a **new** entry to the queue whenever a positive `amountDelta` arrives and the last entry's `unlockedAt` timestamp does not equal `getOracleTime() + NLP_LOCK_PERIOD`. Because oracle time advances continuously, every deposit made at a different oracle second creates a distinct entry, incrementing `balanceCount` with no upper bound:

```solidity
queue.balances[queue.balanceCount] = NlpLockedBalance({...});
queue.balanceCount++;
``` [2](#0-1) 

`tryUnlockNlpBalance` is called at the start of every `handleNlpLockedBalance` invocation. It processes all expired entries in a single unbounded `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [3](#0-2) 

`handleNlpLockedBalance` is invoked from both `updateBalance` overloads whenever `productId == NLP_PRODUCT_ID`, meaning every NLP deposit or withdrawal triggers the loop: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Once a subaccount's `balanceCount` is large enough that iterating all expired entries in one transaction exceeds the block gas limit, every call to `updateBalance` for that subaccount with `NLP_PRODUCT_ID` reverts. This permanently blocks:

- NLP withdrawals (negative `amountDelta` path still calls `handleNlpLockedBalance` → `tryUnlockNlpBalance`)
- Any further NLP deposits

The subaccount's NLP tokens are irreversibly locked. There is no administrative escape hatch in the contract to drain or reset the queue.

---

### Likelihood Explanation

Any user who makes NLP deposits across many distinct oracle timestamps accumulates queue entries. The oracle timestamp advances with each block (or more frequently if sub-block granularity is used). A user making N deposits across N distinct oracle seconds produces N queue entries. At typical Ethereum gas limits (~30M gas), a few thousand entries are sufficient to exceed the limit. This is reachable by any unprivileged user through normal protocol interaction — no special permissions are required.

---

### Recommendation

Replace the unbounded `while` loop with a precalculated or lazily-updated accumulator, analogous to the fix recommended in the referenced report. Specifically:

1. Process at most a fixed number of queue entries per call (e.g., a `MAX_UNLOCK_STEPS` cap), or
2. Maintain a running `totalLockedAmount` that is decremented on each deposit's expiry without requiring iteration, so that `tryUnlockNlpBalance` achieves O(1) complexity.

---

### Proof of Concept

1. Attacker (or any user) calls the NLP deposit entrypoint N times, each at a different oracle timestamp (e.g., one per block). Each call reaches `handleNlpLockedBalance` with `amountDelta > 0` and a distinct `getOracleTime()`, appending a new entry: `queue.balanceCount` reaches N.
2. After `NLP_LOCK_PERIOD` elapses, all N entries have `unlockedAt <= getOracleTime()`.
3. The user attempts to withdraw NLP tokens. The call reaches `updateBalance(NLP_PRODUCT_ID, subaccount, -amount)` → `handleNlpLockedBalance` → `tryUnlockNlpBalance`.
4. The `while` loop iterates N times. For sufficiently large N, the transaction reverts with out-of-gas.
5. All future NLP balance updates for this subaccount revert identically. The subaccount's NLP tokens are permanently locked. [6](#0-5) [7](#0-6)

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()

            // we can unlock this balance

                .amount;

            queue.unlockedUpTo++;
