### Title
Retarget Difficulty Uses Mainchain Ancestor Timestamp Instead of Fork Ancestor Timestamp, Allowing Incorrect `bits` Acceptance on Fork Headers — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating a fork header at a retarget boundary, `get_next_work_required` calls `blocks_getter.get_header_by_height(first_block_height)` to obtain the timestamp of the first block in the retarget interval. The sole implementation of `get_header_by_height` reads exclusively from `mainchain_height_to_header`, which only contains mainchain blocks. If the fork diverges before the retarget boundary, the fork's ancestor at `first_block_height` is a different block than the mainchain block at that height. The contract silently uses the mainchain block's timestamp, computing a wrong `expected_bits` for the fork. A developer-placed TODO in `dogecoin.rs` explicitly acknowledges this concern.

---

### Finding Description

**Root cause — `get_header_by_height` always reads mainchain storage:** [1](#0-0) 

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← only mainchain hashes
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

`mainchain_height_to_header` is a `LookupMap<u64, H256>` that is only written by `store_block_header` (mainchain path) and `reorg_chain`. Fork blocks are stored only in `headers_pool` via `store_fork_header`, with no height-indexed entry. [2](#0-1) 

**Bitcoin retarget call-site:** [3](#0-2) 

At a retarget boundary, `first_block_height` is computed from `prev_block_header.block_height`, then `get_header_by_height(first_block_height)` is called. If the fork diverged at or before `first_block_height`, the returned header is the **mainchain** block at that height, not the fork's ancestor.

**Litecoin retarget call-site (identical pattern):** [4](#0-3) 

**Dogecoin retarget call-site — developer-acknowledged TODO:** [5](#0-4) 

The comment `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor` is a direct developer acknowledgment of this exact ambiguity.

**Difficulty calculation uses the wrong timestamp:** [6](#0-5) 

`actual_time_taken = prev_block_time - first_block_time`. If the mainchain block at `first_block_height` has an earlier timestamp than the fork's true ancestor at that height, `actual_time_taken` is inflated, and the computed target is easier (higher). The fork header's `bits` field is then checked against this wrong `expected_bits`.

**PoW check enforces the wrong expected bits:** [7](#0-6) 

---

### Impact Explanation

An attacker who controls a fork chain can craft fork ancestors at `first_block_height` with timestamps later than the corresponding mainchain block. The contract then computes an easier `expected_bits` (using the mainchain's earlier timestamp), and accepts a fork header whose `bits` encodes a lower difficulty than the fork's true consensus rules require. Subsequent fork blocks are mined at this reduced difficulty. If the attacker accumulates enough chainwork (easier per-block, but still requires real PoW at the reduced target), `reorg_chain` is triggered and the fork is promoted to `mainchain_tip_blockhash`, corrupting the canonical chain state used by all downstream `verify_transaction_inclusion` calls. [8](#0-7) 

---

### Likelihood Explanation

The attack requires:
1. Submitting fork blocks before a retarget boundary — this is a supported, open relayer path with no access control.
2. Crafting fork ancestor timestamps later than the mainchain block at the same height — feasible within MTP and future-timestamp bounds.
3. Mining blocks at the reduced difficulty — real PoW is still required, but at a lower threshold.

For Dogecoin after block 145,000, `difficulty_adjustment_interval` is 1, meaning **every single fork block submission** hits this code path, making the attack surface continuous rather than periodic. [9](#0-8) 

---

### Recommendation

Replace the `get_header_by_height` call in all three retarget functions with an ancestor walk that follows `prev_block_hash` links from `prev_block_header` backward by `blocks_to_go_back` steps, using `get_prev_header`. This mirrors how Bitcoin Core actually traverses the chain: it walks the active chain tip's ancestor list, which for a fork means walking the fork's own ancestor chain, not a height-indexed mainchain lookup. The `get_prev_header` implementation already correctly follows `prev_block_hash` through `headers_pool`, which contains both mainchain and fork blocks. [10](#0-9)

### Proof of Concept

```
Setup (Bitcoin, difficulty_adjustment_interval = 2016):
  - Mainchain: blocks 0..2016 stored, mainchain_height_to_header[0] → block M0 (timestamp T_main)
  - Fork diverges at height 1: fork block F0 at height 0 has timestamp T_fork > T_main

Attack:
  1. Relayer submits fork blocks F1..F2015 (heights 1..2015), all accepted as fork headers.
  2. At height 2016 (retarget boundary), get_next_work_required is called.
  3. first_block_height = 2016 - (2016 - 1) = 1  [Bitcoin] or 0 [Litecoin full-period]
  4. get_header_by_height(first_block_height) returns M0 with timestamp T_main.
  5. actual_time_taken = prev_block_time - T_main  (inflated, since T_main < T_fork)
  6. expected_bits computed at easier difficulty than fork's true required difficulty.
  7. Attacker submits fork header F2016 with bits = easier expected_bits → passes check_pow.
  8. Attacker mines F2017..FN at easier difficulty, accumulates chainwork > mainchain.
  9. reorg_chain promotes fork to mainchain_tip_blockhash.

Assert: expected_bits for fork ≠ expected_bits for mainchain at the same retarget height,
        yet the fork header is accepted.
```

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }
