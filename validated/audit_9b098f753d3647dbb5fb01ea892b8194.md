### Title
`ValidResourceBounds` Variant Lost in Protobuf Round-Trip Causes Wrong Transaction Hash and Consensus Rejection - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant: if both `l2_gas` and `l1_data_gas` are zero it emits `L1Gas`, otherwise `AllResources`. A V3 transaction legitimately signed with `AllResources { l2_gas: 0, l1_data_gas: 0 }` is therefore deserialized as `L1Gas` on the receiving side. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for the two variants (two resource felts vs. three), the transaction hash recomputed by the validator diverges from the hash computed by the proposer, and the downstream conversion back to `RpcTransaction` returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing the validator to reject a block that the proposer considers valid.

### Finding Description

**Root cause — unsafe variant reconstruction in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`** [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The protobuf wire format carries no tag distinguishing `L1Gas` from `AllResources`; the variant is inferred purely from whether the numeric fields are zero. This is the serialization analog of the external report's unsafe down-cast: type information is silently discarded on the write side and incorrectly reconstructed on the read side.

The codebase itself acknowledges the problem in the consensus test helper: [2](#0-1) 

```
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
```

The workaround used in tests (forcing `l2_gas.max_amount = GasAmount(1)`) is not applied in production.

**Hash divergence — `get_tip_resource_bounds_hash` produces different preimages for the two variants** [3](#0-2) 

For `L1Gas` the hash chain contains two resource felts (`L1_GAS`, `L2_GAS`). For `AllResources` it contains three (`L1_GAS`, `L2_GAS`, `L1_DATA_GAS`). Even when `l2_gas = 0` and `l1_data_gas = 0`, the `AllResources` path appends an extra felt encoding the zero-valued `L1_DATA_GAS` resource, so the two hashes are always distinct.

**Consensus serialization path — hash is not transmitted, it is recomputed** [4](#0-3) 

`transaction_hash: None` — the hash is omitted from the wire message. The validator recomputes it via `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal` → `tx_without_hash.calculate_transaction_hash`. [5](#0-4) [6](#0-5) 

**Downstream rejection — conversion back to `RpcTransaction` enforces `AllResources`**

After deserialization the `ValidResourceBounds::L1Gas` value is passed to the `RpcDeclareTransactionV3` / `RpcDeployAccountTransactionV3` converters, which explicitly reject anything that is not `AllResources`: [7](#0-6) [8](#0-7) 

The constant `DEPRECATED_RESOURCE_BOUNDS_ERROR` is returned, causing the validator to reject the entire block.

### Impact Explanation

A proposer that includes a valid V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` will:

1. Compute hash H1 (three-felt `AllResources` preimage) and store it in `InternalRpcTransaction.tx_hash`.
2. Serialize the `ConsensusTransaction` with `transaction_hash: None`.

The validator will:

1. Deserialize `ValidResourceBounds::L1Gas` (two-felt preimage).
2. Fail with `DEPRECATED_RESOURCE_BOUNDS_ERROR` when converting back to `RpcTransaction`.
3. Reject the block.

Even if the conversion did not fail, the recomputed hash H2 (two-felt preimage) would differ from H1, producing a wrong transaction hash that propagates into the transaction commitment, block hash, and execution receipt — matching the impact category **"Transaction conversion or signature/hash logic binds the wrong hash or executable payload"** and **"Wrong receipt/state from blockifier execution logic"**.

### Likelihood Explanation

The trigger is an unprivileged, protocol-valid action: submitting any V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero. This is a natural choice for a user who does not need L2 gas or data gas. No special encoding, malformed bytes, or privileged access is required. The gateway accepts such transactions; only the consensus P2P deserialization path is broken.

### Recommendation

Preserve the variant tag explicitly in the protobuf encoding. The simplest fix is to add a boolean or enum field (e.g., `is_all_resources`) to `protobuf::ResourceBounds` and set/check it unconditionally, so the variant is never inferred from numeric values. Alternatively, use two separate protobuf message types for `L1Gas` and `AllResources` and use a `oneof` field. The value-based heuristic must be removed; it is the direct analog of the unsafe down-cast described in the external report.

### Proof of Concept

```
1. User submits InvokeV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Gateway calls calculate_transaction_hash → get_tip_resource_bounds_hash
   with ValidResourceBounds::AllResources → hash chain has 3 resource felts
   → H1 = Poseidon(INVOKE, version, sender, H1_tip_bounds, ...)

3. Proposer serializes ConsensusTransaction::RpcTransaction(Invoke(V3(tx)))
   → protobuf::InvokeV3 { resource_bounds: { l1_gas: 1000/1, l2_gas: 0/0, l1_data_gas: 0/0 } }
   → transaction_hash: None

4. Validator deserializes:
   TryFrom<protobuf::ResourceBounds> for ValidResourceBounds
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas(l1_gas)

5. Conversion to RpcInvokeTransactionV3 checks for AllResources
   → returns DEPRECATED_RESOURCE_BOUNDS_ERROR
   → validator rejects the block

   OR (if conversion succeeds for Invoke):
   → recomputed hash H2 = Poseidon(INVOKE, version, sender, H2_tip_bounds, ...)
     where H2_tip_bounds uses only 2 resource felts (no l1_data_gas)
   → H1 ≠ H2 → wrong tx_hash stored → wrong transaction commitment → wrong block hash
```

    type Error = ProtobufConversionError;

            return Err(missing("ResourceBounds::l1_gas"));

            return Err(missing("ResourceBounds::l2_gas"));

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
