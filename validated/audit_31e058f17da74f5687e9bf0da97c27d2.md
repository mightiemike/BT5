Audit Report

## Title
`SwapAllowlistExtension` evaluates the router's address instead of the originating user, allowing any caller to bypass per-pool swap allowlists via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes its own `msg.sender` (the router contract) as the `sender` argument to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap()`. The extension checks `allowedSwapper[pool][sender]`, so it evaluates the router's identity rather than the originating user's identity. Any unprivileged user who calls `MetricOmmSimpleRouter` while the router is allowlisted on a restricted pool will pass the allowlist check, completely bypassing the intended access control and executing unauthorized swaps against LP capital.

## Finding Description
`MetricOmmPool.swap()` calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value verbatim as `sender` to every configured extension via `abi.encodeCall`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is whatever the pool passed — the router contract: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap(...)`, the pool's `msg.sender` is the router, not the end user: [4](#0-3) 

For multi-hop `exactInput`, intermediate hops use `address(this)` (the router) as payer, so the router is always the `sender` seen by every intermediate pool's extension: [5](#0-4) 

The allowlist check `allowedSwapper[pool][router]` evaluates to `true` for any caller of the router, not just the intended allowlisted users. There is no existing guard that distinguishes which end user invoked the router.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, whitelisted market makers) is fully open to any user who routes through `MetricOmmSimpleRouter`. The attacker executes swaps at oracle-derived prices against LP capital without ever being allowlisted. This is a direct violation of the pool's intended access-control invariant with fund-impacting consequences: unauthorized swaps drain pool liquidity or extract value the pool admin intended to gate. This meets the "Admin-boundary break" criterion — the pool admin's access control is bypassed by an unprivileged path.

## Likelihood Explanation
The scenario requires the pool admin to allowlist the router address. This is the natural operational step: a pool admin who deploys a restricted pool but still wants their allowlisted users to access it through the standard periphery router will allowlist the router address. The admin's mental model is "the router is a trusted intermediary for my users," but the extension has no mechanism to distinguish which user invoked the router. Once the router is allowlisted, the bypass is available to any unprivileged address with no special setup, no privileged access, and no capital requirement beyond the swap input tokens.

## Recommendation
The `SwapAllowlistExtension` must gate on the originating user, not the intermediary. The preferred fix is to have `MetricOmmPool.swap()` accept an explicit `originator` parameter that the router fills with its own `msg.sender` (the end user), and pass that value — not the pool's `msg.sender` — to extension hooks. Alternatively, `MetricOmmSimpleRouter` should encode the end user's address in `extensionData`, and `SwapAllowlistExtension` should decode and verify it (requiring the extension to trust the router). Until fixed, pool admins must not allowlist the router address; instead, allowlisted users must call the pool directly, which breaks the intended UX.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured. Pool admin calls `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)` so that `alice` can use the router.
2. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: restrictedPool, ...})`.
3. The router calls `pool.swap(...)`. The pool's `msg.sender` is the router.
4. `_beforeSwap` is called with `sender = router` (line 231 of `MetricOmmPool.sol`).
5. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` (line 37 of `SwapAllowlistExtension.sol`).
6. The swap executes. `bob` receives output tokens from the restricted pool without ever being allowlisted.

The allowlist is completely bypassed. The only guard that would prevent this is if the admin never allowlists the router — but then no user can use the router at all, breaking the intended UX.
