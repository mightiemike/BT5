### Title
`compute_meta_tx_v0_hash` Omits Nonce from Signature Domain, Enabling Cross-Transaction Signature Replay via `meta_tx_v0` Syscall - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo` and `crates/blockifier/src/execution/syscalls/syscall_base.rs`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that deliberately omits any nonce from its preimage. Because the OS also explicitly skips nonce checking and incrementing for version-0 transactions, there is no on-chain state that tracks whether a given meta-tx signature has already been consumed. Any party who observes a valid `(contract_address, calldata, signature)` tuple from a past transaction can replay it in an unlimited number of new outer transactions, causing the target contract's `__execute__` entry point to be invoked repeatedly with the same authorized signature.

---

### Finding Description

`compute_meta_tx_v0_hash` in the Cairo OS program computes the meta-transaction hash as:

```
hash(INVOKE_HASH_PREFIX, version=0, contract_address, entry_point_selector,
     calldata_hash, max_fee=0, chain_id,
     additional_data_size=0, additional_data=[])   ← no nonce
``` [1](#0-0) 

The Rust blockifier mirrors this exactly: it constructs an `InvokeTransactionV0` (which has no `nonce` field) and calls `calculate_transaction_hash`, then sets `nonce: Nonce(0.into())` in the replacement `TxInfo`. [2](#0-1) 

The OS nonce gate explicitly skips version-0 transactions:

```cairo
func check_and_increment_nonce{...}(tx_info: TxInfo*) {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

The `execute_meta_tx_v0` OS function also sets `nonce=0` in the synthetic `TxInfo` and never calls `check_and_increment_nonce` for the meta-tx target: [4](#0-3) 

The result is that the hash `H = hash(INVOKE_PREFIX, 0, addr, selector, calldata_hash, 0, chain_id)` is **fully determined by public, static inputs**. Once a valid signature `σ` over `H` appears in any on-chain transaction's calldata, any party can replay `(addr, calldata, σ)` in a new outer transaction. The outer transaction carries its own nonce (preventing outer-tx replay), but the inner meta-tx has no such protection.

Compare with `compute_l1_handler_transaction_hash`, which passes `additional_data_size=1, additional_data=&nonce` to prevent L1-handler replay: [5](#0-4) 

The meta-tx hash has no equivalent protection.

---

### Impact Explanation

Any contract whose `__execute__` entry point performs state-changing operations (token transfers, authorization grants, storage writes) based on the meta-tx calldata is vulnerable. The target contract sees `nonce=0` and `transaction_hash=H` every time; it cannot distinguish the first execution from a replay. An attacker who extracts `(addr, calldata, σ)` from a past block can submit new outer transactions that replay the meta-tx indefinitely, causing repeated unauthorized state changes.

This matches the **Critical** impact: "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" and "Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce, chain id, fee/resource bound, paymaster, or account-deployment logic."

---

### Likelihood Explanation

- The `meta_tx_v0` syscall is a production feature callable by any deployed Cairo 1 contract.
- Signatures are embedded in outer-transaction calldata, which is public on-chain.
- No privileged access is required; any account can submit a new outer transaction wrapping the replayed meta-tx.
- The only mitigation available today is for each target contract to implement its own replay-protection storage, but the protocol provides no enforcement of this.

---

### Recommendation

Add a nonce to the `compute_meta_tx_v0_hash` preimage. The most natural choice is the current on-chain nonce of the target contract at the time the meta-tx is first executed. Concretely:

1. In `compute_meta_tx_v0_hash` (Cairo), pass `additional_data_size=1` and `additional_data=&current_nonce` (read from `contract_state_changes`).
2. In `execute_meta_tx_v0` (Cairo), call `check_and_increment_nonce` on the target contract after computing the hash, so the nonce is consumed.
3. In `syscall_base.rs` (Rust), read the target contract's nonce from state before computing the hash, include it in the `InvokeTransactionV0`-equivalent preimage, and increment it afterward.

This mirrors how `compute_l1_handler_transaction_hash` includes the L1 nonce and how the external report's fix added `escrowNonce` to the signed data.

---

### Proof of Concept

1. Alice deploys a contract `B` whose `__execute__` transfers 100 tokens to the caller when invoked with `calldata = [transfer_100]`.
2. Alice signs `σ = sign(compute_meta_tx_v0_hash(B, __execute__, [transfer_100], chain_id))`.
3. Bob submits outer tx 1 (nonce=5): calls `execute_meta_tx_v0(B, [transfer_100], σ)` → 100 tokens transferred.
4. The signature `σ` and calldata are now visible on-chain.
5. Bob submits outer tx 2 (nonce=6): calls `execute_meta_tx_v0(B, [transfer_100], σ)` again.
6. The blockifier computes the same hash `H`, the same `σ` verifies, `check_and_increment_nonce` is skipped (version=0), and 100 more tokens are transferred.
7. Steps 5–6 repeat until the source account is drained.
