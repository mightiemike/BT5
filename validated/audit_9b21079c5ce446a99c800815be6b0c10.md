### Title
Difficulty Retarget Uses Mainchain Ancestor Instead of Fork Ancestor During Fork Block Validation — (`contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When a fork block is submitted and its difficulty is validated, both the Bitcoin and Dogecoin implementations fetch the "first block in the difficulty adjustment interval" from the **mainchain** (`get_header_by_height`) rather than walking the fork's own ancestor chain. If the fork diverged before that boundary height, the mainchain block at that height is a different block than the fork's true ancestor, causing the difficulty target to be computed from wrong timestamp data. This is a direct analog to H-37: state (difficulty) is computed in one chain context (mainchain) but applied to a block that belongs to a different chain context (fork).

### Finding Description

**Root cause — `bitcoin.rs` line 81:**

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
```

**Root cause — `dogecoin.rs` lines 292–295 (with the developers' own TODO acknowledging the problem):**

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
```

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It always reads from `mainchain_height_to_header`, which maps heights to **mainchain** block hashes only. Fork blocks are stored in `headers_pool` but are never inserted into `mainchain_height_to_header` (only `store_block_header` does that; `store_fork_header` does not). Therefore, when a fork block at height F is being validated and the difficulty calculation needs the ancestor at height `F - interval`, it silently retrieves the **mainchain** block at that height instead of the fork's true ancestor.

The broken invariant: *the difficulty target for a fork block must be derived exclusively from that fork's own ancestor chain*. The code violates this by mixing mainchain ancestor data into fork difficulty computation.

### Impact Explanation

The `calculate_next_work_required` functions in both `bitcoin.rs` and `dogecoin.rs` compute the new difficulty target as:

```
new_target = prev_target * actual_time_taken / pow_target_timespan
```

where `actual_time_taken = prev_block_time - first_block_time`. If the mainchain block at `first_block_height` has an **older** timestamp than the fork's true ancestor at that height, `actual_time_taken` is inflated, producing a **larger target** (easier difficulty). An attacker can exploit this to submit fork blocks that satisfy a lower difficulty than the fork's chain actually requires. With reduced per-block work, the attacker can accumulate enough chainwork to trigger `reorg_chain`, replacing the canonical mainchain with the attacker's fork. This corrupts the canonical chain state and invalidates previously confirmed transaction inclusion proofs.

For Dogecoin (difficulty adjusts every block, `difficulty_adjustment_interval = 1` for height ≥ 145 000), the wrong ancestor is used for **every single fork block** submitted after the fork divergence point, making the attack surface continuous rather than periodic.

### Likelihood Explanation

- Any unprivileged NEAR caller can invoke `submit_blocks` (subject to the trusted-relayer check, but the relayer path is the normal production entry point and the contract is designed to accept headers from it).
- The attacker only needs to control the fork's block submissions and find a scenario where the mainchain block at the boundary height has an older timestamp than the fork's ancestor — a natural condition whenever the fork's blocks were produced faster than the mainchain's blocks at the same heights.
- For Dogecoin, the condition is triggered on every fork block after the second block past the divergence point, requiring no special timing.
- The developers themselves flagged this with a `TODO` comment, confirming awareness of the ambiguity.

### Recommendation

Replace the `get_header_by_height` call in both `bitcoin.rs` and `dogecoin.rs` with a backward walk along the fork's own ancestor chain. Starting from `prev_block_header`, follow `prev_block_hash` links through `headers_pool` until reaching the block at `first_block_height`. This ensures the difficulty calculation is always grounded in the fork's own history, not the mainchain's.

### Proof of Concept

1. Mainchain: `M0(t=1000) → M1(t=1600) → … → M2015(t=…) → M2016(t=…)` (Bitcoin, 2016-block interval).
2. Attacker submits a fork starting at `M0`: `F1(t=2000) → F2(t=2600) → … → F2015(t=…)`.
3. At height 2016, the difficulty retarget for `F2016` calls `get_header_by_height(0)`, which returns `M0` (timestamp 1000) instead of the fork's genesis (which is `M0` itself here — but for a fork diverging at height 100, `get_header_by_height(100)` returns the mainchain block at 100, not the fork's ancestor at 100).
4. If `M100.time < F100.time`, `actual_time_taken` is larger than the fork's true elapsed time, producing a lower difficulty target for `F2016`.
5. The attacker mines `F2016` against this reduced target, satisfying `check_pow`.
6. If the fork's accumulated `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
