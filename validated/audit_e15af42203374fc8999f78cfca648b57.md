### Title
Linked Signer Can Escalate Its Own Privilege by Re-Linking to an Attacker-Controlled Address — (`core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` handler in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`, meaning the currently-registered linked signer is accepted as a valid signer for a transaction that **mutates the `linkedSigners` mapping itself**. A linked signer (a lower-privilege delegated key) can therefore replace itself with any attacker-controlled address, permanently hijacking the subaccount's signing authority without the owner's consent.

---

### Finding Description

In `EndpointTx.sol`, the fast-mode `LinkSigner` branch decodes the transaction and calls `validateSignedTx` with the fifth argument hardcoded to `true`: [1](#0-0) 

The `true` flag propagates through `validateSignedTx` → `validateSignature` → `Verifier.validateSignature`, where the check accepts a signature from **either** the subaccount owner address **or** the currently-registered `linkedSigner`: [2](#0-1) 

After passing validation, the handler unconditionally overwrites the `linkedSigners` entry: [3](#0-2) 

The `linkedSigners` mapping is the privileged access-control state that gates all subsequent signed operations (withdrawals, transfers, liquidations) for the subaccount: [4](#0-3) 

By contrast, the **slow-mode** `LinkSigner` path uses `validateSender`, which enforces that only the actual owner address (`msg.sender == address(uint160(bytes20(txn.sender)))`) can submit the transaction: [5](#0-4) 

This asymmetry is the root cause: the slow-mode path correctly restricts `LinkSigner` to the owner, but the fast-mode path allows the linked signer to perform the same privileged mutation.

---

### Impact Explanation

A linked signer (e.g., a trading API key, a bot key, or any delegated address) can sign and submit a fast-mode `LinkSigner` transaction naming an attacker-controlled address as the new signer. Once accepted by the sequencer, the attacker's address becomes the registered linked signer. The attacker can then sign `WithdrawCollateral`, `WithdrawCollateralV2`, or `TransferQuote` transactions on behalf of the victim subaccount, draining all collateral. The victim's only recourse is to race the attacker with a new `LinkSigner` from the owner key — but if the owner key is not immediately available (e.g., cold storage, hardware wallet), the attacker wins the race.

**Corrupted state:** `linkedSigners[subaccount]` is set to an attacker-controlled address, granting full signing authority over the subaccount's assets.

---

### Likelihood Explanation

Linked signers are a standard feature for API-based trading (hot keys, bots, programmatic traders). Any scenario where a linked signer key is leaked, compromised, or operated by a malicious third-party service (e.g., a trading bot provider) immediately enables this escalation. The attack requires no privileged protocol access — only possession of the linked signer's private key and the ability to submit a transaction to the sequencer, which is a normal user capability.

---

### Recommendation

Change `allowLinkedSigner` to `false` in the fast-mode `LinkSigner` handler so that only the subaccount owner can mutate the `linkedSigners` mapping:

```solidity
// EndpointTx.sol, fast-mode LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // <-- was true; linked signer must NOT be able to re-link
);
```

This aligns the fast-mode path with the slow-mode path's intent, where `validateSender` already enforces owner-only access for `LinkSigner`.

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...` and has registered Bob's key as her linked signer via a prior `LinkSigner` transaction.
2. Bob's key is compromised by an attacker (Eve).
3. Eve uses Bob's key to sign a `LinkSigner` transaction: `{ sender: 0xAlice000..., signer: 0xEve000..., nonce: N }`.
4. Eve submits this to the sequencer as a fast-mode transaction.
5. `validateSignedTx(..., true)` passes because Eve's recovered address equals the registered `linkedSigner` (Bob's address).
6. `linkedSigners[0xAlice000...]` is overwritten with `0xEve000...`.
7. Eve signs a `WithdrawCollateral` transaction from Alice's subaccount, draining all collateral to Eve's wallet.
8. Alice's owner key can submit a corrective `LinkSigner`, but only if she acts before Eve's withdrawal is sequenced.

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
