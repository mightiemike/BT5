### Title
Missing `ENABLE_SECP_OPS` Flag Check on 4-Byte Secp Opcode Dispatch Path - (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` enforces the `ENABLE_SECP_OPS` flag for the 1-byte opcode forms of `secp256k1_verify` (opcode 64) and `secp256r1_verify` (opcode 65), but omits the same flag check for the equivalent 4-byte opcode forms (`0x13d61f00` and `0x1c3a8f00`). Any attacker-controlled CLVM program can invoke the secp verification operators via the 4-byte opcode path regardless of whether the dialect has `ENABLE_SECP_OPS` set, bypassing the intended hard-fork access gate.

---

### Finding Description

In `ChiaDialect::op()`, operator dispatch is split into two paths based on opcode length.

**1-byte path (gated correctly):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
_ => {
    return unknown_operator(allocator, o, argument_list, flags, max_cost);
}
``` [1](#0-0) 

**4-byte path (flag check absent):**

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is documented as a hard-fork gate: "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify). Hard-fork; enable only when it activates." [3](#0-2) 

The `MEMPOOL_MODE` constant, used for stricter mempool validation, does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

This means that in mempool mode (or any pre-hard-fork consensus mode), a CLVM program using the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` will successfully execute `op_secp256k1_verify` or `op_secp256r1_verify`, while the same operation via 1-byte opcodes 64/65 would be rejected with `EvalErr::Unimplemented`.

---

### Impact Explanation

**Consensus divergence / flag bypass:** The `ENABLE_SECP_OPS` flag is a hard-fork gate. Before the fork activates, secp operations must not be executable. The 4-byte opcode path bypasses this gate entirely. A CLVM program using `0x13d61f00` or `0x1c3a8f00` will:

- In mempool mode (no `ENABLE_SECP_OPS`, with `NO_UNKNOWN_OPS`): **execute secp verification** via the 4-byte path, while the 1-byte form would be rejected.
- In consensus mode (no `ENABLE_SECP_OPS`, without `NO_UNKNOWN_OPS`): **execute secp verification** via the 4-byte path, while the 1-byte form would be a no-op with cost.

This creates a concrete divergence in program semantics depending solely on which opcode encoding is used, breaking the invariant that `ENABLE_SECP_OPS` controls all access to secp operations. A program that should be rejected (or treated as a no-op) before the hard fork activates instead performs real cryptographic verification, potentially producing a different `Reduction` result or error state than expected.

---

### Likelihood Explanation

The 4-byte opcode encoding is a documented feature of the CLVM cost model (the "unknown operators with assigned cost" formula). Any party who reads the source or the cost specification can discover that `0x13d61f00` and `0x1c3a8f00` are the 4-byte secp opcode forms. Crafting a CLVM program that uses these opcodes requires only constructing a 4-byte atom operator — a trivial operation with attacker-controlled CLVM bytes. No special privileges, social engineering, or compromised nodes are required.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag guard to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode encodings of the secp operators are uniformly gated behind the hard-fork flag, eliminating the bypass path.

---

### Proof of Concept

A CLVM program that invokes `secp256k1_verify` via the 4-byte opcode form, run against a `ChiaDialect` without `ENABLE_SECP_OPS` (e.g., `MEMPOOL_MODE`):

```rust
use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;

let mut allocator = Allocator::new();
let dialect = ChiaDialect::new(MEMPOOL_MODE); // no ENABLE_SECP_OPS

// Build (0x13d61f00 pubkey msg sig) — 4-byte secp256k1_verify opcode
// The 4-byte opcode atom: [0x13, 0xd6, 0x1f, 0x00]
let op = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
// ... build args (pubkey, msg, sig) ...
let program = allocator.new_pair(op, args).unwrap();

// This succeeds (executes secp verification) despite ENABLE_SECP_OPS being absent.
// Using 1-byte opcode 64 instead would return EvalErr::Unimplemented.
let result = run_program(&mut allocator, &dialect, program, env, u64::MAX);
```

The corrupted invariant: `result` is `Ok(Reduction(1300000, nil))` (secp verification executed) instead of `Err(EvalErr::Unimplemented)`, which is what the 1-byte opcode 64 would produce under the same `MEMPOOL_MODE` flags. [5](#0-4)
