Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and gates on `owner` (the LP position recipient), which is a free caller-supplied parameter. Any address not on the allowlist can bypass the restriction by calling `addLiquidity` with `owner` set to any allowlisted address, completely defeating the pool admin's access control.

## Finding Description
`MetricOmmPool.addLiquidity` passes the real caller as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both and forwards them to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed, discarding it entirely, and evaluates `allowedDepositor[msg.sender][owner]` — where `msg.sender` is the pool and `owner` is the attacker-controlled recipient: [3](#0-2) 

The pool imposes no restriction on who `owner` can be set to. The peripheral router `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` explicitly supports depositing on behalf of an arbitrary `owner`, with only a non-zero address check (`_validateOwner`): [4](#0-3) 

The existing unit tests confirm the design intent is to check `owner`, but they never test the case where `sender ≠ owner` with an unlisted sender: [5](#0-4) 

The contract's own NatSpec documents the intent as gating by **depositor address**: [6](#0-5) 

## Impact Explanation
The `DepositAllowlistExtension` allowlist is completely bypassed. Any address — regardless of allowlist status — can add liquidity to a restricted pool by naming any allowlisted address as `owner`. The pool admin's access control (e.g., KYC/compliance gating) is rendered entirely ineffective. The unauthorized caller's tokens are deposited and credited to the allowlisted address's LP position without that address's consent, constituting unsolicited griefing of the position owner's accounting. This breaks a core pool access-control invariant with direct fund-flow consequences.

## Likelihood Explanation
The attack requires only a public `addLiquidity` call with a known allowlisted address as `owner`. No privileged access, special tokens, or oracle manipulation is needed. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events. The attacker can call the pool directly or through the router. Likelihood is high.

## Recommendation
Replace the `owner` check with the `sender` argument (the actual caller):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// attacker is NOT on the allowlist; allowlisted IS on the allowlist
address attacker = makeAddr("attacker");
address allowlisted = makeAddr("allowlisted");

// pool admin allowlists only `allowlisted`
vm.prank(admin);
depositExtension.setAllowedToDeposit(address(pool), allowlisted, true);

// attacker calls addLiquidity with owner = allowlisted
// extension checks allowedDepositor[pool][allowlisted] == true → passes
vm.prank(attacker);
pool.addLiquidity(
    allowlisted,   // owner = allowlisted address
    0,
    deltas,
    callbackData,
    ""
);

// attacker successfully deposited despite not being on the allowlist
uint256 shares = pool.positionBinShares(allowlisted, 0, binIdx);
assertGt(shares, 0); // passes — allowlist bypassed
```
