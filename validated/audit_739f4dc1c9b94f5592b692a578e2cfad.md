### Title
`ghost_heap` Not Incremented in `new_substr` SmallAtomŌåÆSmallAtom Path, Causing Under-Counted Heap Limit ŌĆö (`File: src/allocator.rs`)

---

### Summary

`Allocator::new_atom` and `Allocator::new_small_number` both increment `ghost_heap` when producing a `SmallAtom`. `Allocator::new_substr`, when called on a `SmallAtom` source and producing a `SmallAtom` result, increments `ghost_atoms` but **never increments `ghost_heap`**. Because `ghost_heap` is the sole mechanism that keeps the heap-limit check consensus-accurate after the small-atom optimisation, this asymmetry causes the limit to be enforced less strictly than intended, producing a consensus divergence between optimised and non-optimised evaluators.

---

### Finding Description

The `ghost_heap` counter exists specifically to maintain consensus parity with a hypothetical non-optimised evaluator that stores every atom ŌĆö including small ones ŌĆö in the real heap (`u8_vec`). The allocator comment states this explicitly:

> *"the ghost counters are pretend atoms/pairs, that were optimized out. We still account for them to not affect the limits of atoms and pairs. Those limits must stay the same for consensus purpose."* [1](#0-0) 

**`new_atom` (correct):** when the value fits in a `SmallAtom`, both `ghost_atoms` and `ghost_heap` are incremented:

```rust
if let Some(ret) = fits_in_small_atom(v) {
    self.ghost_atoms += 1;
    self.ghost_heap += v.len();   // ŌåÉ ghost_heap updated
    Ok(self.mk_node(ObjectType::SmallAtom, ret as usize))
}
``` [2](#0-1) 

**`new_small_number` (correct):** same pattern ŌĆö both counters updated:

```rust
self.ghost_atoms += 1;
self.ghost_heap += len;           // ŌåÉ ghost_heap updated
``` [3](#0-2) 

**`new_substr` on `SmallAtom` ŌåÆ `SmallAtom` (broken):** only `ghost_atoms` is incremented; `ghost_heap` is silently skipped:

```rust
ObjectType::SmallAtom => {
    ...
    if let Some(new_val) = fits_in_small_atom(substr) {
        self.ghost_atoms += 1;
        // ghost_heap NOT incremented  ŌåÉ asymmetry
        Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
    }
``` [4](#0-3) 

The heap-limit guard in `new_atom` (and `new_concat`) is:

```rust
if start as usize + self.ghost_heap + v.len() > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
``` [5](#0-4) 

Because `ghost_heap` is under-counted every time `new_substr` produces a `SmallAtom` from a `SmallAtom` source, the effective limit seen by subsequent `new_atom` / `new_concat` calls is artificially relaxed. A non-optimised evaluator (or a future evaluator that stores all atoms on the heap) would count those same bytes as real heap usage and would reject the program earlier.

---

### Impact Explanation

`ghost_heap` is the consensus-critical accounting variable that bridges the optimised and non-optimised execution models. Under-counting it means:

- An optimised node accepts a CLVM program that a non-optimised node (or a reference implementation) rejects with `OutOfMemory`.
- This is a **consensus divergence**: two honest nodes evaluating the same spend bundle reach different accept/reject decisions, which is equivalent in severity to a hash-mismatch or cost-mismatch bug in the Chia context.
- The divergence is permanent for any block containing such a spend bundle: one class of nodes forks away.

---

### Likelihood Explanation

The CLVM `substr` operator is a standard, unconditionally available operator. Any attacker who can submit a spend bundle to the mempool can craft a program that:

1. Constructs a small integer atom (e.g., `(q . 1)` ŌåÆ value `1`, one byte).
2. Calls `(substr atom 0 1)` in a loop ŌĆö each call goes through the `SmallAtom ŌåÆ SmallAtom` branch of `new_substr`, incrementing `ghost_atoms` but not `ghost_heap`.
3. Interleaves real heap allocations (e.g., large `concat` results) that are gated by the under-counted `ghost_heap`.

The cost model charges per `substr` call, so the attacker is bounded by the block cost limit, but the per-call `ghost_heap` deficit accumulates linearly. The practical exploitability depends on the configured `heap_limit`; with a tighter limit (e.g., via `Allocator::new_limited`) the divergence is reachable within normal block cost budgets. [6](#0-5) 

---

### Recommendation

In the `SmallAtom ŌåÆ SmallAtom` branch of `new_substr`, add the missing `ghost_heap` increment, mirroring the pattern in `new_atom`:

```rust
if let Some(new_val) = fits_in_small_atom(substr) {
    self.ghost_atoms += 1;
    self.ghost_heap += (end - start) as usize;  // add this line
    Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
}
``` [7](#0-6) 

Additionally, add a unit test that creates a `SmallAtom`, calls `new_substr` on it to produce another `SmallAtom`, and asserts that `heap_size()` equals the value it would have if `new_atom` had been called with the same bytes ŌĆö confirming the ghost accounting is symmetric across all atom-creation paths.

---

### Proof of Concept

```rust
// Demonstrates ghost_heap asymmetry between new_atom and new_substr
let mut a1 = Allocator::new();
let mut a2 = Allocator::new();

// Path 1: create a 1-byte SmallAtom via new_atom
// ghost_heap += 1
let _ = a1.new_atom(&[0x01]).unwrap();

// Path 2: create a 2-byte SmallAtom, then substr it to 1 byte
// new_atom: ghost_heap += 2
// new_substr (SmallAtomŌåÆSmallAtom): ghost_heap += 0  ŌåÉ BUG
let src = a2.new_atom(&[0x00, 0x01]).unwrap();
let _ = a2.new_substr(src, 1, 2).unwrap();

// Both allocators have created one 1-byte SmallAtom.
// heap_size() should be equal, but it is not:
assert_eq!(a1.heap_size(), a2.heap_size()); // FAILS: a2.heap_size() is 1 less
```

The failing assertion confirms that `ghost_heap` is not updated symmetrically, meaning the heap-limit check in `a2` is one byte more permissive than in `a1` for every `substr`-produced `SmallAtom`. Repeated over many iterations within a single program, this gap accumulates and allows real heap allocations that the non-optimised accounting model would have blocked.
