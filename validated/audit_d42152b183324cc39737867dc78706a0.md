### Title
Linked Signer Can Overwrite Its Own Authorization — (`File: core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` transaction type in `EndpointTx.processTransactionImpl` is validated with `allowLinkedSigner: true`. This means the currently-registered linked signer for a subaccount is accepted as a valid signer to issue a new `LinkSigner` transaction — including one that replaces itself with an attacker-controlled address. A compromised linked signer can therefore permanently escalate its own authority and drain the subaccount, with no on-chain mechanism to atomically revoke it.

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` struct and calls `validateSignedTx` with `allowLinkedSigner: true`: [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which in turn calls `Verifier.validateSignature` with the current linked signer as the second accepted signer: [2](#0-1) 

`Verifier.validateSignature` accepts the recovered address if it equals either the subaccount owner or the linked signer: [3](#0-2) 

Because `allowLinkedSigner: true` is passed for `LinkSigner`, the linked signer itself is a valid signer for the transaction that mutates `linkedSigners[subaccount]`. There is no restriction preventing the linked signer from replacing itself with a different address.

The linked signer is stored in `EndpointStorage`: [4](#0-3) 

For isolated subaccounts, `getLinkedSigner` inherits the parent's linked signer, so a single compromise propagates to all child isolated subaccounts: [5](#0-4) 

### Impact Explanation

Once the linked signer is rotated to an attacker-controlled address, the attacker can sign any transaction that accepts a linked signer, including `WithdrawCollateral` (`allowLinkedSigner: true`): [6](#0-5) 

This allows the attacker to drain the subaccount's collateral to an arbitrary address via `WithdrawCollateralV2` with `sendTo == address(0)` (which routes to the owner's address) or via `WithdrawCollateral`. The attacker can also sign `LiquidateSubaccount` and `MintNlp`/`BurnNlp` transactions, corrupting the subaccount's position. The compromised linked signer can also re-rotate the linked signer after any attempted revocation, maintaining persistent access.

**Impact: 4** — direct collateral theft and persistent subaccount takeover.

### Likelihood Explanation

Linked signers are explicitly designed as hot wallets for high-frequency trading, meaning they hold private keys in memory on internet-connected infrastructure. This makes them materially more likely to be compromised than the cold-wallet subaccount owner. The sequencer path provides near-instant execution, so the attacker can rotate the linked signer and drain funds before the owner can react. The slow-mode revocation path has a hardcoded 3-day delay: [7](#0-6) 

During those 3 days, the attacker retains full signing authority and can re-rotate the linked signer each time the owner queues a revocation.

**Likelihood: 3** — requires linked signer key compromise, but the attack surface is a hot wallet by design.

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address encoded in the first 20 bytes of the `sender` field) should be permitted to change the linked signer. The linked signer's authority should be scoped to order signing, not to mutation of the authorization mapping itself.

```solidity
// Before (line 581):
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);

// After:
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, false);
```

### Proof of Concept

1. User A owns subaccount `S = bytes32(abi.encodePacked(addrA, bytes12(...)))` and registers linked signer `addrB` (hot wallet) via a `LinkSigner` transaction signed by `addrA`.
2. Attacker compromises `addrB`.
3. Attacker constructs `LinkSigner { sender: S, signer: bytes32(addrC), nonce: currentNonce }` and signs it with `addrB`.
4. Attacker submits the transaction to the sequencer. `validateSignedTx` accepts `addrB`'s signature because `allowLinkedSigner: true` and `linkedSigners[S] == addrB`.
5. `linkedSigners[S]` is now set to `addrC` (attacker-controlled).
6. Attacker signs `WithdrawCollateral { sender: S, productId: QUOTE, amount: fullBalance, nonce: N+1 }` with `addrC`.
7. `validateSignedTx` accepts `addrC`'s signature. `Clearinghouse.withdrawCollateral` transfers the full balance.
8. If User A queues a slow-mode `LinkSigner` revocation, the attacker re-rotates the linked signer before the 3-day delay expires, maintaining persistent access. [1](#0-0) [8](#0-7)

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
