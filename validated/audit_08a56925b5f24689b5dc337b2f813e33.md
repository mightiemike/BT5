### Title
`MIN_FIRST_DEPOSIT_AMOUNT` Does Not Enforce a Minimum Maintained Balance — Subaccounts Can Be Registered for Near-Free - (File: `core/contracts/Endpoint.sol`)

---

### Summary

The `MIN_FIRST_DEPOSIT_AMOUNT = $5` check in `Endpoint.isValidDepositAmount()` is enforced only at deposit time to gate new subaccount creation. After the deposit is processed and the subaccount is registered, the full $5 can be immediately withdrawn because `Clearinghouse.withdrawCollateral()` only requires `getHealth(sender) >= 0`, which is satisfied at a zero balance with no open positions. This allows any unprivileged user to register an unlimited number of subaccounts for the cost of gas and the $1 slow-mode fee, bypassing the intended $5 barrier entirely.

---

### Finding Description

`Endpoint.isValidDepositAmount()` enforces a higher minimum for new subaccounts:

```solidity
// Endpoint.sol lines 90-101
function isValidDepositAmount(...) internal returns (bool) {
    int256 minDepositAmount = MIN_DEPOSIT_AMOUNT;          // $0.1
    if (subaccount != X_ACCOUNT && (subaccountIds[subaccount] == 0)) {
        minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT;       // $5
    }
    return clearinghouse.checkMinDeposit(productId, amount, minDepositAmount);
}
``` [1](#0-0) 

Once the slow-mode deposit is processed by the sequencer, `_recordSubaccount()` permanently registers the subaccount:

```solidity
// EndpointStorage.sol lines 67-72
function _recordSubaccount(bytes32 subaccount) internal {
    if (subaccountIds[subaccount] == 0) {
        subaccountIds[subaccount] = ++numSubaccounts;
        subaccounts[numSubaccounts] = subaccount;
    }
}
``` [2](#0-1) 

The only post-withdrawal constraint in `Clearinghouse.withdrawCollateral()` is a health check:

```solidity
// Clearinghouse.sol lines 419
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [3](#0-2) 

With no open positions, a subaccount holding exactly $5 USDC has health = `$5 × longWeightInitial`. After withdrawing the full $5, health = 0, which satisfies `>= 0`. The withdrawal succeeds, the subaccount retains its registered ID, and the attacker recovers their $5 minus only the $1 slow-mode fee and gas.

There is no mechanism to deregister a subaccount or decrement `numSubaccounts`. The registration is permanent. [4](#0-3) 

---

### Impact Explanation

The `MIN_FIRST_DEPOSIT_AMOUNT` is the protocol's only on-chain barrier against sybil creation of subaccounts. Once bypassed, an attacker can:

1. **Inflate `numSubaccounts` without bound** — the counter is `uint64` and never decremented. Any off-chain or on-chain system iterating over the `subaccounts` mapping is exposed to unbounded growth.
2. **Cheaply acquire registered subaccounts** — each costs only gas + `SLOW_MODE_FEE = $1` instead of the intended $5. At scale this is a 5× reduction in sybil cost.
3. **Undermine the liquidation sybil deterrent** — the codebase explicitly notes that `LIQUIDATION_FEE = $1` deters sybil liquidators. A registered subaccount with a residual $1 balance can pay this fee, meaning the effective cost per liquidator identity drops from $5 to ~$2 (gas + slow-mode fee + $1 residual). [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The attack is fully permissionless, requires no privileged access, and is executable by any EOA. The only cost is gas and the $1 slow-mode fee per subaccount. The sequencer processes deposits quickly in practice (it can execute slow-mode transactions immediately via `fromSequencer = true`), so the 3-day slow-mode delay is not a real barrier. The attack is straightforward to script and loop. [7](#0-6) 

---

### Recommendation

Enforce a minimum maintained balance as an ongoing invariant, not just at deposit time. One approach: in `Clearinghouse.withdrawCollateral()`, after the health check, additionally require that the subaccount's quote balance does not fall below a protocol-defined floor (e.g., `MIN_FIRST_DEPOSIT_AMOUNT`) unless the subaccount has no remaining positions and is being fully exited. Alternatively, introduce a non-refundable registration fee that is not credited as collateral, so it cannot be withdrawn. [8](#0-7) 

---

### Proof of Concept

```
1. Attacker calls Endpoint.depositCollateral(subaccountName, USDC_PRODUCT_ID, 5e6)
   → $5 USDC transferred to Clearinghouse
   → Slow-mode tx queued (executableAt = now + 3 days, but sequencer processes immediately)

2. Sequencer calls processSlowModeTransaction → processSlowModeTransactionImpl:
   → _recordSubaccount(subaccount)  // subaccountIds[subaccount] = ++numSubaccounts
   → clearinghouse.depositCollateral(txn)  // balance += $5

3. Attacker signs and submits WithdrawCollateral(sender=subaccount, productId=USDC, amount=5e6)
   → Sequencer calls processTransactionImpl → WithdrawCollateral branch
   → chargeFee(sender, withdrawFeeX18)  // assume 0 or negligible
   → clearinghouse.withdrawCollateral(sender, USDC, 5e6, address(0), nSubmissions)
      → spotEngine.updateBalance(USDC, sender, -$5)  // balance = 0
      → getHealth(sender, INITIAL) = 0 >= 0  ✓  // passes
      → $5 USDC transferred back to attacker

Result: subaccountIds[subaccount] != 0 permanently.
        Attacker spent: gas + $1 slow-mode fee.
        Intended cost: $5.
        Loop N times with different subaccountNames to register N subaccounts.
``` [9](#0-8) [10](#0-9) [11](#0-10)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
