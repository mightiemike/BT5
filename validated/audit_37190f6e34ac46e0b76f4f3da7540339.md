### Title
Fork Retarget Difficulty Computed from Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height` is implemented to read exclusively from `mainchain_height_to_header`. When a fork block lands on a retarget boundary, `get_next_work_required` calls `get_header_by_height(first_block_height)` to obtain the interval-start timestamp for `calculate_next_work_required`. Because this lookup always returns the **mainchain** block at that height — never the fork's own ancestor — the difficulty for the fork's retarget block is computed from the wrong chain history. The developers themselves flagged this in a `TODO` comment in `dogecoin.rs`.

---

### Finding Description

`BlocksGetter::get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It only consults `mainchain_height_to_header` — the height-indexed map that is populated exclusively for mainchain blocks via `store_block_header`. Fork blocks are stored only in `headers_pool` via `store_fork_header`, with no height-index entry. [2](#0-1) 

At every retarget boundary, all three affected chain modules call `get_header_by_height` to fetch the interval-start block:

**Bitcoin** (`bitcoin.rs` lines 78–86):
```rust
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header,
    interval_tail_extend_header.block_header.time.into())
``` [3](#0-2) 

**Litecoin** (`litecoin.rs` lines 86–93): [4](#0-3) 

**Dogecoin** (`dogecoin.rs` lines 291–297) — with an explicit developer acknowledgement:
```rust
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header.time;
``` [5](#0-4) 

When the fork diverges at any height ≤ `first_block_height`, the fork's block at `first_block_height` is a different header (with a different timestamp) than the mainchain's block at the same height. The contract silently uses the mainchain's timestamp, violating the invariant that difficulty must be derived from the fork's own chain history.

---

### Impact Explanation

The attacker constructs a fork that diverges at height `H+1` (one block after the genesis/initial block). The fork's block at `H+1` carries a timestamp `T_fork` that is **later** than the mainchain's `T_main` at the same height. The fork then builds up to height `H+2015` (the block before the retarget at `H+2016`).

At the retarget boundary (`H+2016`):

- `first_block_height` = `H+1`
- `get_header_by_height(H+1)` returns the **mainchain** block with timestamp `T_main`
- `actual_time_taken` = `fork_H2015.time − T_main` (larger than the correct `fork_H2015.time − T_fork`)
- A larger `actual_time_taken` → larger new target → **easier required difficulty**

The attacker sets `block_header.bits` to the incorrectly computed (easier) value. The `check_pow` call validates `expected_bits == block_header.bits` using the same wrong calculation, so it passes. The PoW hash check then uses `target_from_bits(header.bits)` with the easier target, requiring less mining work. The fork block is accepted with a `bits` value that encodes a lower difficulty than the network would enforce for a fork with those timestamps. [6](#0-5) 

---

### Likelihood Explanation

The entry point is `submit_blocks`, which is gated by `#[trusted_relayer]`. The trusted relayer system uses a staking mechanism with a `RelayerManager` who can reject applications. However, the question scope explicitly includes the trusted-relayer submission path as a valid attacker entry point. Once a relayer role is obtained (via staking), the attacker can submit arbitrary valid-looking fork headers. No additional privileged access is needed beyond the relayer role itself. The bug is unconditional — it fires on every fork retarget boundary without any special precondition beyond the fork diverging before `first_block_height`. [7](#0-6) 

---

### Recommendation

Replace the `get_header_by_height` call in all three retarget functions with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from `prev_block_header` back `(difficulty_adjustment_interval - 1)` steps. This is the same pattern already used correctly by `get_prev_header` and by Zcash's `zcash_get_next_work_required`, which walks backward through `get_prev_header` rather than using the height index. [8](#0-7) 

---

### Proof of Concept

State-machine test outline (NEAR SDK sandbox, Bitcoin feature):

1. Initialize contract at height `H` (a multiple of 2016) with `skip_pow_verification = false`.
2. Submit mainchain blocks `H+1` through `H+2016` with timestamps `T_main[i]`.
3. Submit fork block at `H+1` with timestamp `T_fork_1 = T_main[1] + 3600` (1 hour later), building on the same `H` genesis.
4. Submit fork blocks `H+2` through `H+2015` with timestamps advancing normally.
5. At `H+2016`, compute the **correct** `expected_bits` using `T_fork_1` as `first_block_time`, and the **buggy** `expected_bits` using `T_main[1]`.
6. Submit a fork block at `H+2016` with `bits` = buggy value. Assert the contract **accepts** it.
7. Submit the same fork block with `bits` = correct value. Assert the contract also accepts it (or rejects the buggy one if the fix is applied).

The test demonstrates that `check_target` passes for the fork block carrying the mainchain-derived (incorrect) `bits`, confirming the invariant is broken. [9](#0-8)
