### Title
`op_modpow` (opcode 60) Disabled in Mempool Mode Despite Being a Hardforked Operator — (`File: src/chia_dialect.rs`)

### Summary
`op_modpow` (opcode 60) is a fully hardforked CLVM operator — it is dispatched unconditionally in consensus mode with no activation-flag guard. However, `MEMPOOL_MODE` includes `ClvmFlags::DISABLE_OP`, and the dispatch table for opcode 60 checks this flag and returns `EvalErr::Unimplemented` when it is set. This creates a mempool-consensus divergence: a coin spend that uses `modpow` is accepted on-chain but rejected by the mempool, mirroring the ERC-721 bug class where an overly strict capability check causes valid operations to fail.

### Finding Description

In `src/chia_dialect.rs`, the `MEMPOOL_MODE` constant is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)       // ← included here
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [1](#0-0) 

The operator dispatch table in `ChiaDialect::op()` handles opcode 60 as:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [2](#0-1) 

Because `MEMPOOL_MODE` includes `DISABLE_OP`, any call to opcode 60 in mempool mode returns `Unimplemented` — a hard rejection.

Contrast this with every other operator that is gated behind a hard-fork flag. Those operators use a conditional arm in the `match` that falls through to `unknown_operator` when the flag is absent:

```rust
62 if flags.contains(ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD) => op_keccak256,
63 if flags.contains(ClvmFlags::ENABLE_SHA256_TREE) => op_sha256_tree,
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
``` [3](#0-2) 

`op_modpow` has **no such activation guard** — it is dispatched unconditionally in consensus mode. This means it is a hardforked operator. Yet `DISABLE_OP` in `MEMPOOL_MODE` silently blocks it only in the mempool path.

The `DISABLE_OP` flag carries **no doc comment**, unlike every other flag in the `ClvmFlags` bitfield: [4](#0-3) 

This is the structural analog to the ERC-721 bug: an extra capability check (`DISABLE_OP`) is applied on top of the standard dispatch path, causing a valid, fully-activated operator to be rejected in a context (mempool) where it should be accepted.

### Impact Explanation

Any CLVM coin spend that invokes opcode 60 (`modpow`) will be rejected by the mempool with `EvalErr::Unimplemented` when the node runs in `MEMPOOL_MODE`. The same spend is accepted by the consensus engine (which does not set `DISABLE_OP`). This is a **mempool-consensus divergence**:

- The spend is valid on-chain and will be included in a block if it reaches a full node directly.
- The spend is rejected by the mempool and will not be propagated through the peer-to-peer network under normal conditions.

This effectively makes `modpow`-based puzzles non-functional for ordinary users who submit transactions through the standard mempool path, even though the operator is fully hardforked and consensus-valid.

### Likelihood Explanation

The `DISABLE_OP` flag is included in `MEMPOOL_MODE`, which is the standard mode used by Chia full nodes for mempool validation. Any coin spend using `modpow` submitted to a standard node will be rejected. The trigger is attacker-controllable: an adversary can craft a CLVM program with opcode 60 and submit it to observe the divergence, or a legitimate user will encounter the rejection when trying to use `modpow` in a puzzle.

### Recommendation

Remove `ClvmFlags::DISABLE_OP` from `MEMPOOL_MODE`, or add a proper doc comment and activation-flag guard to `op_modpow` consistent with the pattern used for `keccak256`, `sha256_tree`, and the secp operators. If `modpow` is intended to be soft-forked (not yet active), it should be gated behind a named activation flag (e.g., `ENABLE_MODPOW`) and fall through to `unknown_operator` when absent — not return `Unimplemented`. If it is already hardforked, `DISABLE_OP` must be removed from `MEMPOOL_MODE`.

### Proof of Concept

```
# Consensus mode (no DISABLE_OP): succeeds
run_program(
    program = "(modpow (q . 3) (q . 10) (q . 7))",  # opcode 60
    args    = "()",
    flags   = ClvmFlags::empty(),
)
# → Ok(Reduction(cost, atom[4]))   # 3^10 mod 7 = 4

# Mempool mode (DISABLE_OP set via MEMPOOL_MODE): fails
run_program(
    program = "(modpow (q . 3) (q . 10) (q . 7))",
    args    = "()",
    flags   = MEMPOOL_MODE,         # includes DISABLE_OP
)
# → Err(EvalErr::Unimplemented(op_node))
```

pub fn op_modpow(a: &mut Allocator, input: NodePtr, max_cost: Cost, flags: ClvmFlags) -> Response {
    if flags.contains(ClvmFlags::MALACHITE) {
        return op_modpow_malachite(a, input, max_cost, flags);
    }
    let [base, exponent, modulus] = get_args::<3>(a, input, "modpow")?;

    let mut cost = MODPOW_BASE_COST;
    let (base, bsize) = int_atom(a, base, "modpow")?;
    cost += bsize as Cost * MODPOW_COST_PER_BYTE_BASE_VALUE;
    let (exponent, esize) = int_atom(a, exponent, "modpow")?;
    cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;
    check_cost(cost, max_cost)?;
    let (modulus, msize) = int_atom(a, modulus, "modpow")?;
    cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;
    check_cost(cost, max_cost)?;

    if bsize > 256 || esize > 256 || msize > 256 {
        return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
    }

    if exponent.sign() == Sign::Minus {
        return Err(EvalErr::InvalidOpArg(
            input,
            "ModPow with Negative Exponent".to_string(),
        ));
    }

    if modulus.sign() == Sign::NoSign {
        return Err(EvalErr::DivisionByZero(input));
    }

    let ret = base.modpow(&exponent, &modulus);
    let ret = a.new_number(ret)?;
    Ok(malloc_cost(a, cost, ret))
}
