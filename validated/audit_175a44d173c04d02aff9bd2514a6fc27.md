Looking at the `x/tieredrewards` module, specifically `ClaimTierRewards` in `msg_server.go` and `claimRewardsAndUpdatesPositions` in `claim_rewards.go`, there is a concrete analog to the duplicate-withdrawal class of the reported reentrancy bug.

---

### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double Bonus Reward Withdrawal - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary
`ClaimTierRewards` loads all requested positions into an in-memory slice before processing them. If `msg.PositionIds` contains the same ID more than once, the same position is loaded as two independent in-memory copies and processed sequentially. When the targeted validator has events with a reference count greater than one (i.e., other positions also reference the same validator), the bonus reward payment executes twice in a single transaction, draining the `RewardsPoolName` module account beyond what is owed.

### Finding Description
In `msg_server.go`, `ClaimTierRewards` first iterates over `msg.PositionIds` to load and validate each position: [1](#0-0) 

There is no deduplication of `msg.PositionIds`. If the same ID appears twice, `getPositionState` is called twice, producing two independent in-memory copies of the same position — both carrying the **old** `LastEventSeq` and `LastBonusAccrual` checkpoints — and both are appended to the `positions` slice.

`claimRewardsAndUpdatesPositions` then iterates over this slice: [2](#0-1) 

For each element it calls `claimBaseRewards` (which calls `WithdrawDelegationRewards` — idempotent, returns 0 on the second call), then `processEventsAndClaimBonus`, then `setPosition`.

Inside `processEventsAndClaimBonus`, the bonus is computed using `pos.LastEventSeq` from the **in-memory copy**: [3](#0-2) 

The first iteration processes all validator events since the old `LastEventSeq`, decrements their reference counts, pays bonus coins via `bankKeeper.SendCoinsFromModuleToAccount`, and persists the updated position via `setPosition`. [4](#0-3) 

The second iteration operates on the **second in-memory copy**, which still holds the old `LastEventSeq`. It calls `getValidatorEventsSince` with the old sequence. If the validator's events have a reference count ≥ 2 (because at least one other position also references them — a normal condition when multiple positions share a validator), those events were not deleted by the first iteration's `decrementEventRefCount` calls. The second iteration therefore finds the same events, computes the same bonus, and issues a second `SendCoinsFromModuleToAccount` payment. It then calls `setPosition` with the **old** checkpoint state, overwriting the correctly-advanced state written by the first iteration. [5](#0-4) 

### Impact Explanation
The `RewardsPoolName` module account is debited twice for the same accrued bonus. The corrupted values are:
- The `RewardsPoolName` balance (drained by 2× the owed bonus per duplicated position per claim).
- `pos.LastEventSeq` — reset to the old value by the second `setPosition`, meaning future claims can re-process the same events if their reference counts are still non-zero, compounding the drain over multiple transactions.

### Likelihood Explanation
Any unprivileged user who owns at least one tiered-rewards position on a validator that also has at least one other position (their own or another user's) can trigger this. Multi-position validators are the normal operating state of the tiered-rewards system. The attacker only needs to submit a `MsgClaimTierRewards` transaction with a repeated position ID. No privileged access, leaked keys, or social engineering is required. The only mitigating uncertainty is whether `msg.Validate()` (defined in `msg_validate.go`) rejects duplicate IDs; the server-side handler contains no such check. [6](#0-5) 

### Recommendation
1. In `MsgClaimTierRewards.Validate()` (or at the top of the `ClaimTierRewards` handler), reject any `PositionIds` slice that contains duplicate values.
2. Alternatively, deduplicate `msg.PositionIds` before the loading loop in `ClaimTierRewards`.
3. As a defense-in-depth measure, `claimRewardsAndUpdatesPositions` should re-fetch each position from the store immediately before processing it (rather than operating on a pre-loaded in-memory snapshot), so that the second processing of a duplicate ID sees the already-advanced checkpoint written by the first processing and computes zero additional bonus.

### Proof of Concept
1. Attacker owns position `P` delegated to validator `V`. At least one other position (e.g., `Q`) is also delegated to `V`, so validator events for `V` have reference count ≥ 2.
2. Validator `V` has accrued events since `P.LastEventSeq`, entitling `P` to bonus `B`.
3. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [P, P]}`.
4. Server loads two copies of `P` (both with old `LastEventSeq`) into `positions[0]` and `positions[1]`.
5. `claimRewardsAndUpdatesPositions` processes `positions[0]`: pays bonus `B`, decrements event ref counts (still ≥ 1 because of `Q`), persists updated `P`.
6. `claimRewardsAndUpdatesPositions` processes `positions[1]` (old checkpoint): finds the same events (ref count still ≥ 1), pays bonus `B` again, persists old checkpoint (overwriting the updated state).
7. Attacker receives `2B` instead of `B` from `RewardsPoolName`. The persisted `LastEventSeq` is reset to the old value, enabling the attack to be repeated in subsequent blocks. [7](#0-6) [2](#0-1)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
