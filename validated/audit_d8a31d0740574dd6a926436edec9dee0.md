### Title
`verify_transaction_inclusion` unconditionally rejects valid empty Merkle proofs, permanently blocking SPV verification for single-transaction blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains an unconditional guard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` that fires even when an empty proof is the mathematically correct proof (a block containing exactly one transaction). Because `verify_transaction_inclusion_v2` — the only non-deprecated entry point — delegates to `verify_transaction_inclusion` internally, no caller can ever obtain a `true` result for a valid single-transaction block. This is the direct analog of H-1: a check that should be conditional on a secondary circumstance (collateral withdrawal / proof length > 0) is applied unconditionally, blocking a legitimate user action.

---

### Finding Description

**Root cause — `verify_transaction_inclusion`, line 315:**

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This guard fires before the actual Merkle computation on line 318–322:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` is implemented as a simple loop over the proof elements:

```rust
for proof_hash in merkle_proof {
    ...
}
current_hash   // returned unchanged when proof is empty
``` [3](#0-2) 

When `merkle_proof` is empty the loop does not execute and the function returns `transaction_hash` unchanged. For a block whose Merkle tree contains exactly one transaction, `merkle_root == tx_hash` by definition, so `compute_root_from_merkle_proof(tx_id, 0, &[]) == header.block_header.merkle_root` would evaluate to `true` — but the call never reaches that line because the `require!` panics first.

**Propagation through `verify_transaction_inclusion_v2`:**

`verify_transaction_inclusion_v2` is the only non-deprecated public entry point for SPV proofs. It validates the coinbase sub-proof, then unconditionally delegates to v1:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

For a single-transaction block both `merkle_proof` and `coinbase_merkle_proof` are empty vectors. The v2 length-equality check passes (`0 == 0`), and the coinbase sub-proof check also passes because `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id` which equals `merkle_root`. Execution then enters v1 and panics on line 315. [5](#0-4) 

**`merkle_proof_calculator` confirms empty proof is correct for single-transaction blocks:**

```rust
while current_hashes.len() > 1 {   // never entered when len == 1
    ...
}
merkle_proof   // empty Vec returned
``` [6](#0-5) 

The library's own proof generator produces an empty vector for a single-transaction block, yet the verifier unconditionally rejects it.

---

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion_v2` with a cryptographically valid proof for a single-transaction block receives a panic instead of `true`. Downstream contracts that gate cross-chain fund releases or state transitions on this verification result are permanently blocked for such blocks, even though the proof is correct. The corrupted proof result is a false rejection (`panic`) where the correct result is `true`.

---

### Likelihood Explanation

Single-transaction blocks (coinbase only) occur during periods of extremely low mempool activity and are common on testnets and early-stage deployments of the supported chains (Bitcoin, Litecoin, Dogecoin, Zcash). The contract is explicitly designed for multi-chain use via feature flags. [7](#0-6) 

On testnets and during chain bootstrapping these blocks appear regularly. Any relayer that submits such a block header creates a permanently unverifiable block in the pool — the header is accepted, but no SPV proof against it can ever succeed.

---

### Recommendation

Remove the unconditional guard on line 315, or make it conditional on the block containing more than one transaction. `compute_root_from_merkle_proof` already handles the empty-proof case correctly and no additional logic is needed:

```rust
// Before (overly strict):
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

// After (correct): remove the require entirely; the computation below
// already returns tx_id unchanged for an empty proof, which is the
// correct result for a single-transaction block.
```

### Proof of Concept

1. A block is submitted and accepted with exactly one transaction (the coinbase). Its `merkle_root` equals `coinbase_tx_hash`.
2. A NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_hash`
   - `tx_block_blockhash = <accepted block hash>`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_tx_hash`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. Length check in v2: `0 == 0` → passes.
4. Coinbase sub-proof in v2: `compute_root_from_merkle_proof(coinbase_tx_hash, 0, &[])` returns `coinbase_tx_hash` == `merkle_root` → passes.
5. Execution enters v1 via `self.verify_transaction_inclusion(args.into())`.
6. Line 315: `require!(!vec![].is_empty(), ...)` → **panics with "Merkle proof is empty"**.
7. A cryptographically valid inclusion proof is permanently rejected; any contract awaiting `true` is blocked.

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
