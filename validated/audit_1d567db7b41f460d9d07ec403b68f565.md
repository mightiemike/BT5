### Title
Insufficient GC-window guard in `verify_transaction_inclusion` causes permanent proof lockout for GC'd blocks — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` guards against over-confirmation requests with `args.confirmations <= self.gc_threshold`, but this check is **not sufficient** to guarantee the queried block is still in storage. After `run_mainchain_gc` removes the oldest block, any caller who queries that removed block with `confirmations <= gc_threshold` bypasses the guard and hits an unconditional panic at the `mainchain_header_to_height` lookup.

---

### Finding Description

The guard in `verify_transaction_inclusion` is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

This only bounds the *requested* confirmation count against `gc_threshold`. It does **not** verify that the specific block hash being queried is still present in `mainchain_header_to_height`.

`run_mainchain_gc` removes the oldest block from all three maps via `remove_block_header`:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
``` [2](#0-1) 

`remove_block_header` deletes the entry from both `mainchain_header_to_height` and `headers_pool`:

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [3](#0-2) 

After GC, the subsequent lookup in `verify_transaction_inclusion` unconditionally panics:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [4](#0-3) 

---

### Impact Explanation

**Exact state trace with `gc_threshold = T`, chain at heights `h..h+T` (T+1 blocks):**

1. Trusted relayer calls `submit_blocks([header])` → block at height `h+T+1` appended; chain is now `T+2` blocks.
2. `run_mainchain_gc(1)` is called automatically with `batch_size=1`.
   - `amount_of_headers_we_store = T+2`, `total_amount_to_remove = 2`, `selected_amount_to_remove = min(2,1) = 1`.
   - Block at height `h` is removed from `mainchain_header_to_height`, `mainchain_header_to_height`, and `headers_pool`.
   - `mainchain_initial_blockhash` advances to height `h+1`. [5](#0-4) 

3. Chain now holds `T+1` blocks (`h+1..h+T+1`). Block at height `h` is permanently gone.
4. Proof consumer calls `verify_transaction_inclusion` for `hash_h` with `confirmations=T`.
   - Guard: `T <= T` → **passes**.
   - `mainchain_header_to_height.get(&hash_h)` → `None` → **panics**: `"block does not belong to the current main chain"`.

The panic is permanent and unrecoverable. No future call can verify any transaction in the GC'd block. `verify_transaction_inclusion_v2` is equally affected because it delegates to `verify_transaction_inclusion` internally. [6](#0-5) 

---

### Likelihood Explanation

- GC runs **automatically** on every `submit_blocks` call; no special attacker action is needed beyond normal relayer operation.
- The scenario requires only that the chain size exceeds `gc_threshold` by more than `batch_size`, which is the normal steady-state condition.
- Any downstream contract or off-chain service that caches a block hash and later calls `verify_transaction_inclusion` after GC has advanced past that block will hit this panic.
- The misleading guard message (`"The required number of confirmations exceeds the number of blocks stored in memory"`) gives callers a false guarantee that `confirmations <= gc_threshold` is sufficient.

---

### Recommendation

Replace the single-scalar guard with a check that the queried block's height is within the currently retained window. After fetching the initial block height, add:

```rust
let initial_block_height = self
    .headers_pool
    .get(&self.mainchain_initial_blockhash)
    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
    .block_height;

let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

require!(
    target_block_height >= initial_block_height,
    "block has been garbage collected and is no longer available for proof verification"
);
```

This ensures the error is surfaced with a correct, actionable message rather than a misleading panic, and the guard accurately reflects the actual retention window rather than just the threshold scalar.

---

### Proof of Concept

```rust
#[test]
fn gc_proof_lockout() {
    // gc_threshold = 3, init with 4 blocks (heights 0..3) → T+1 = 4 blocks
    let mut contract = init_with_n_blocks(4, /*gc_threshold=*/3);

    // submit 1 more block → chain = 5 blocks, run_mainchain_gc(1) removes height 0
    contract.submit_blocks(vec![make_block(4)]);

    // chain is now heights 1..4 (T+1 = 4 blocks); height 0 is gone

    // guard passes: confirmations=3 <= gc_threshold=3
    // but block at height 0 is GC'd → panics "block does not belong to the current main chain"
    let result = std::panic::catch_unwind(|| {
        contract.verify_transaction_inclusion(ProofArgs {
            tx_block_blockhash: hash_at_height(0),
            confirmations: 3,
            ..default_proof()
        })
    });
    assert!(result.is_err(), "expected panic for GC'd block");
}
```
