### Title
Zero-Amount Fee Transfer in `submitFastWithdrawal` Can Block Fast Withdrawals for Zero-Revert Tokens — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.submitFastWithdrawal()`, when a third-party caller submits a fast withdrawal on behalf of a recipient (`sendTo != msg.sender`), the protocol collects the fee via `safeTransferFrom(token, msg.sender, uint128(fee))` without first checking whether `fee > 0`. If `fastWithdrawalFeeAmount()` returns `0` — possible when `withdrawFeeX18 = 0` for a product and the withdrawal amount is small enough to round the proportional fee to zero — and the collateral token reverts on zero-value transfers, the entire `submitFastWithdrawal` call reverts. This blocks the fast-withdrawal path for that product until the fee configuration changes or a non-zero fee is produced.

---

### Finding Description

`submitFastWithdrawal` is a `public` function with no access control. When the caller is not the intended recipient, the fee is charged from the caller: [1](#0-0) 

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // ← no zero-check
}
```

The fee is computed by `fastWithdrawalFeeAmount()`: [2](#0-1) 

```solidity
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;
int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
return feeX18 / int128(multiplier);
```

If `withdrawFeeX18 = 0` for a product (making `minFeeX18 = 0`) and the withdrawal amount is small enough that `proportionalFeeX18` rounds to zero after integer division, `fee` is `0`. The subsequent `safeTransferFrom` call with `amount = 0` is then forwarded through `ERC20Helper.safeTransferFrom`: [3](#0-2) 

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transferFrom.selector, from, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
```

For tokens that revert on zero-value transfers (a known ERC-20 non-standard behaviour), `success` is `false`, the `require` fails, and the entire `submitFastWithdrawal` call reverts.

---

### Impact Explanation

Fast withdrawals for the affected product become completely unavailable via the third-party submission path (`sendTo != msg.sender`) until either the product's `withdrawFeeX18` is raised above zero or the withdrawal amount is large enough to produce a non-zero fee. Users relying on the fast-withdrawal path to access their collateral quickly are denied that functionality. The slow-withdrawal path through the sequencer remains available, but the fast-withdrawal guarantee — a core protocol feature — is broken for the affected product.

---

### Likelihood Explanation

The condition requires two concurrent factors: (1) `withdrawFeeX18 = 0` for a listed product, which is a plausible default or intentional configuration for certain collateral types, and (2) a collateral token that reverts on zero-value transfers, a documented class of non-standard ERC-20 tokens. Neither condition requires privileged access or attacker control; both can arise from normal protocol configuration and token selection. Any unprivileged caller attempting a third-party fast withdrawal under these conditions will trigger the revert.

---

### Recommendation

Add a zero-amount guard before the fee transfer in the `sendTo != msg.sender` branch:

```solidity
} else {
    if (uint128(fee) > 0) {
        safeTransferFrom(token, msg.sender, uint128(fee));
    }
}
```

This mirrors the recommended fix in the referenced report and ensures the transfer is only attempted when there is a positive fee to collect.

---

### Proof of Concept

1. A product is configured with `withdrawFeeX18 = 0`.
2. A user signs a `WithdrawCollateral` transaction for a small amount of a token that reverts on zero-value transfers.
3. A third-party relayer calls `submitFastWithdrawal(idx, transaction, signatures)` where `sendTo` (the user's address) differs from `msg.sender` (the relayer).
4. `fastWithdrawalFeeAmount()` returns `0` due to zero `withdrawFeeX18` and small amount rounding.
5. `safeTransferFrom(token, msg.sender, 0)` is called; the token reverts; `ERC20Helper.safeTransferFrom` propagates the revert via `require(success …, ERR_TRANSFER_FAILED)`.
6. The fast withdrawal is blocked. No relayer can successfully submit this withdrawal until the fee configuration changes. [4](#0-3) [2](#0-1) [3](#0-2)

```text
    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```
