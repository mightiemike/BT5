Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the direct caller of `MetricOmmPool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router contract address, not the originating user. If the router is allowlisted (the natural admin action to enable router-mediated swaps), every user on-chain can bypass the allowlist by routing through the router. Additionally, `SwapAllowlistExtension.beforeSwap` drops the `onlyPool` modifier present in `BaseMetricExtension.beforeSwap`.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // direct pool caller — the router when routed
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` forwards `sender` unchanged into the extension call:

```solidity
// metric-core/contracts/ExtensionCalling.sol L162-165
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, recipient, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant) calls the pool, the pool's `msg.sender` is the router:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The effective check becomes `allowedSwapper[pool][router]`, never consulting the actual user's address. The router does not forward the original `msg.sender` to the pool in any parameter.

Additionally, `SwapAllowlistExtension.beforeSwap` is declared `external view override` without the `onlyPool` modifier that `BaseMetricExtension.beforeSwap` carries:

```solidity
// BaseMetricExtension.sol L81-88 — has onlyPool
function beforeSwap(...) external virtual onlyPool returns (bytes4) { ... }

// SwapAllowlistExtension.sol L31-35 — no onlyPool
function beforeSwap(address sender, ...) external view override returns (bytes4) { ... }
```

## Impact Explanation
A pool admin deploying a curated pool with `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses (e.g., KYC-verified users, protocol partners). Allowlisting `MetricOmmSimpleRouter` — the canonical periphery path — causes the allowlist to be completely ineffective for all router-mediated swaps. Any address explicitly denied can execute swaps by calling `router.exactInputSingle` or any `exact*` variant. This is a broken core pool functionality and admin-boundary break: the pool admin's access control is silently defeated, exposing LP funds to swaps from actors the admin explicitly intended to exclude.

## Likelihood Explanation
The exploit requires only: (1) a pool configured with `SwapAllowlistExtension`, (2) the pool admin allowlisting `MetricOmmSimpleRouter` (the expected action for normal usage), and (3) any disallowed user calling any `exact*` function on the router. All three conditions are reachable by unprivileged actors using only the documented public interface. No special tokens, malicious setup, or privileged access is required. The trigger is a natural admin configuration, making exploitation highly likely in production.

## Recommendation
The original user's address must be threaded through the call chain so the extension can gate on the economically relevant actor:

1. **Pool-level fix**: Add an `originator` parameter to `swap()` that the pool populates with `msg.sender` and passes separately to `_beforeSwap`, or store the original caller in transient storage before the extension call.
2. **Extension-level fix**: Re-add the `onlyPool` modifier to `SwapAllowlistExtension.beforeSwap` (restoring the inherited guard), and check a separately forwarded original-caller field rather than `sender`.
3. **Router-level (insufficient alone)**: The router could pass `msg.sender` via `extensionData`, but this is opt-in and bypassable; a protocol-level fix in the pool is required.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
3. Pool admin does NOT call setAllowedToSwap(pool, alice, true)  // alice is denied

4. alice calls router.exactInputSingle({pool: pool, ...})
   → router calls pool.swap(recipient, ...) with msg.sender = router
   → pool calls _beforeSwap(sender=router, ...)
   → extension checks allowedSwapper[pool][router] == true  ✓
   → swap executes — alice's trade goes through despite being denied

5. alice directly calls pool.swap(...)
   → pool calls _beforeSwap(sender=alice, ...)
   → extension checks allowedSwapper[pool][alice] == false  ✗
   → revert NotAllowedToSwap — correctly blocked
```
