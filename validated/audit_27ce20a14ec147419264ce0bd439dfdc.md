### Title
Insufficient Coinbase Identity Validation in `verify_transaction_inclusion_v2` Allows 64-Byte Merkle Proof Forgery Bypass — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability. Its coinbase guard checks only that `coinbase_tx_id` at position 0 produces the correct Merkle root, and that both proofs share the same length. It does **not** check that `coinbase_tx_id` is the actual coinbase transaction of the block. An unprivileged NEAR caller can supply two different internal Merkle tree nodes — one as `coinbase_tx_id` and one as `tx_id` — both of which satisfy the structural checks, causing the function to return `true` for a transaction that was never broadcast or mined.

---

### Finding Description

The analog to the stablecoin shorting bug is exact:

| Original | This Repository |
|---|---|
| Check: `custody_key != collateral_custody_key` (different keys) | Check: `merkle_proof.len() == coinbase_merkle_proof.len()` (same depth) |
| Missing: `custody` must not itself be a stablecoin | Missing: `coinbase_tx_id` must be the actual coinbase transaction |
| Bypass: two different stablecoins | Bypass: two different internal Merkle nodes |

The relevant code in `contract/src/lib.rs`:

Neither check verifies that `coinbase_tx_id` is the actual first transaction (coinbase) of the block. The function then delegates to the deprecated v1 path:

```rust

```

which checks only that `tx_id` at `tx_index` produces the correct Merkle root — the same structural check the attacker already satisfies.

### Impact Explanation

Consider a real on-chain block with 4 transactions `[A, B, C, D]` and Merkle tree:

```
        Root
       /    \
    AB        CD
   /  \      /  \
  A    B    C    D
```

where `AB = hash(A, B)` and `CD = hash(C, D)`.

An attacker submits:
- `coinbase_tx_id = AB`, `coinbase_merkle_proof = [CD]`
- `tx_id = CD`, `tx_index = 1`, `merkle_proof = [AB]`

Step-by-step evaluation:

1. **Length check**: `[CD].len() == [AB].len()` → `1 == 1` ✓
2. **Coinbase check**: `compute_root_from_merkle_proof(AB, 0, [CD])` → position 0 is even → `hash(AB, CD) = Root` ✓
3. **Transaction check** (inside v1): `compute_root_from_merkle_proof(CD, 1, [AB])` → position 1 is odd → `hash(AB, CD) = Root` ✓

The function returns `true`, asserting that `CD` — an internal Merkle node, not a real transaction — is included in the block at position 1. Any downstream NEAR contract consuming this result (e.g., a bridge that releases funds upon proof of a Bitcoin deposit) would be deceived into acting on a fabricated transaction.

---

### Likelihood Explanation

- The function is publicly callable by any NEAR account (no role restriction, only `#[pause]`).
- The attack requires only public information: any real block already accepted into the contract's main chain, plus its transaction list to compute internal Merkle nodes.
- No cryptographic hardness assumption is broken; the attacker performs only SHA-256d hashes that are already public.
- The attacker does not need to submit any new block headers or interact with the relayer.

---

### Recommendation

After verifying the coinbase proof, assert that `coinbase_tx_id` equals the actual coinbase transaction of the block. Since the contract does not store individual transactions, the simplest sound fix is to require that `coinbase_tx_id != tx_id` **and** that `tx_index != 0` (position 0 is reserved for the coinbase). More robustly, require callers to supply the coinbase transaction bytes and derive `coinbase_tx_id` on-chain, or require that the coinbase proof depth equals `floor(log2(N))` for the known transaction count. At minimum, add:

```rust
require!(
    args.coinbase_tx_id != args.tx_id,
    "coinbase_tx_id must differ from tx_id"
);
require!(
    args.tx_index != 0,
    "tx_index 0 is reserved for the coinbase transaction"
);
```

---

### Proof of Concept

Given any real block already in the contract's main chain with ≥ 2 transactions, compute `AB = hash(tx[0], tx[1])` and `CD = hash(tx[2], tx[3])` (or the appropriate internal nodes for the block's tree depth). Call:

```json
{
  "tx_id":                "CD_hex",
  "tx_block_blockhash":   "<real block hash>",
  "tx_index":             1,
  "merkle_proof":         ["AB_hex"],
  "coinbase_tx_id":       "AB_hex",
  "coinbase_merkle_proof":["CD_hex"],
  "confirmations":        1
}
```

`verify_transaction_inclusion_v2` returns `true` for the internal node `CD`, which is not a real transaction.

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
