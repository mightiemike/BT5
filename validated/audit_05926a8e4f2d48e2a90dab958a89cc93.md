### Title
Child `ChunkExtra.proposed_split` Not Reset After Shard Split — Guaranteed `InvalidChunkHeaderShardSplit` on First Child Chunk - (File: chain/chain/src/resharding/manager.rs)

### Summary

During dynamic resharding, `process_memtrie_resharding_storage_update` creates child `ChunkExtra` by cloning the parent's `ChunkExtra` but never resets the `proposed_split` field to `None`. Because the parent's `proposed_split` is `Some(split)` (that is what triggered the resharding), every child shard's initial `ChunkExtra` carries a stale split proposal. The first chunk produced for each child shard freshly computes `proposed_split = None` (new shard, below threshold), but `validate_chunk_with_chunk_extra_and_receipts_root` performs a strict equality check between `prev_chunk_extra.proposed_split()` and `chunk_header.proposed_split()`, producing `InvalidChunkHeaderShardSplit` and rejecting the chunk. This is a guaranteed chain-halt condition on the first epoch boundary after any dynamic resharding event.

### Finding Description

In `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is built by cloning the parent and updating only `state_root` and `congestion_info`:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// proposed_split is NOT cleared — inherits Some(parent_split)
``` [1](#0-0) 

This `ChunkExtra` is persisted under the child `ShardUId` at the resharding block hash and becomes the `prev_chunk_extra` for the first chunk of the child shard in epoch N+2.

`ChunkExtraV5.proposed_split` is defined as the split proposal computed during chunk application for that specific shard: [2](#0-1) 

The validation gate in `validate_chunk_with_chunk_extra_and_receipts_root` performs a strict equality check with no special-case for the first chunk of a freshly-created child shard:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(format!(
        "header has {:?}, expected {:?} ...",
        chunk_header.proposed_split(),
        prev_chunk_extra.proposed_split(),
    )));
}
``` [3](#0-2) 

For the first chunk of a child shard in epoch N+2:
- `prev_chunk_extra.proposed_split()` = `Some(parent_split)` — stale, inherited from parent
- `chunk_header.proposed_split()` = `None` — freshly computed by the chunk producer for the new shard (below threshold)

The mismatch is deterministic and unavoidable.

The `DynamicReshardingConfig` comment acknowledges the stale-inheritance problem but attributes it only to "back-to-back reshardings" and relies on `min_epochs_between_resharding: NonZeroEpochHeight` as a workaround: [4](#0-3) 

However, the cooldown only prevents a *second* resharding from being scheduled; it does not reset the stale `proposed_split` already written into the child `ChunkExtra`. The validation failure occurs on the first chunk of the child shard after *any* resharding, regardless of the cooldown value.

The `ShardChunkHeaderInnerV5.proposed_split` field is part of the Borsh-serialized, network-broadcast chunk header: [5](#0-4) 

### Impact Explanation

When `ShardLayoutConfig::Dynamic` is activated (gated behind `ProtocolFeature::DynamicResharding`, protocol version 85, currently stable), the first resharding event will produce child shards whose initial `ChunkExtra` carries a stale `proposed_split`. Every chunk producer for those child shards will produce a valid chunk with `proposed_split = None`, which will be rejected by all validators with `InvalidChunkHeaderShardSplit`. No chunk for the child shard can ever be accepted, halting those shards permanently. Because the child shards cover the full account space of the parent, this is a network-wide chain halt.

### Likelihood Explanation

`ProtocolFeature::DynamicResharding` is at stable protocol version 85 (current stable is 86). The bug is latent: it fires on the first epoch boundary after any dynamic resharding event once `ShardLayoutConfig::Dynamic` is deployed in an epoch config. No privileged action beyond the governance decision to enable dynamic resharding is required to trigger it; the resharding itself is driven automatically by trie memory thresholds.

### Recommendation

In `process_memtrie_resharding_storage_update`, explicitly reset `proposed_split` to `None` on the child `ChunkExtra` after cloning from the parent:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None; // child shard has no prior split proposal
```

This mirrors the invariant that `proposed_split` reflects the split computed during chunk application for *that* shard, not inherited from an ancestor.

### Proof of Concept

1. Configure an epoch with `ShardLayoutConfig::Dynamic` and a low `memory_usage_threshold` so that a shard exceeds it within a few blocks.
2. Run the chain until the last block of epoch N embeds `shard_split = Some((shard_id, boundary))` in `BlockHeaderInnerRestV6`.
3. Observe `finalize_epoch` writing `EpochInfoV5.shard_layout` for epoch N+2 with the two child shards.
4. At the first block of epoch N+2, the chunk producer for each child shard produces a chunk with `proposed_split = None` (fresh computation on a new, sub-threshold shard).
5. `validate_chunk_with_chunk_extra_and_receipts_root` reads `prev_chunk_extra` for the child shard (saved at the resharding block with `proposed_split = Some(parent_split)`) and returns `Err(InvalidChunkHeaderShardSplit(...))`.
6. All validators reject the chunk; the child shards never produce an accepted chunk. [1](#0-0) [3](#0-2) [2](#0-1)
