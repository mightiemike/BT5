### Title
Heap-Limit Restriction Enforced in `new_atom`/`new_concat` but Bypassed in `new_substr` SmallAtom Path — (`File: src/allocator.rs`)

### Summary

The `Allocator` enforces a `heap_limit` in every allocation path that writes new bytes to `u8_vec` — except one: `new_substr` when the source node is a `SmallAtom` and the resulting sub-slice does not fit back into a small atom. That branch writes directly to `u8_vec` without the heap-limit guard present in every other allocation function. An attacker-controlled CLVM program using the `substr` operator (opcode 12) can repeatedly trigger this path to push `u8_vec.len()` past the configured `heap_limit`.

### Finding Description

Every allocation function that physically writes bytes to `u8_vec` checks the heap limit before doing so:

**`new_atom`** — [1](#0-0) 

```rust
let start = self.u8_vec.len() as u32;
if start as usize + self.ghost_heap + v.len() > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
```

**`new_small_number`** — [2](#0-1) 

```rust
if self.u8_vec.len() + self.ghost_heap + len > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
```

**`new_concat`** — [3](#0-2) 

```rust
if start + self.ghost_heap + new_size > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
```

`new_substr`, however, has no such guard in its `SmallAtom` branch when the sub-slice cannot be stored as a small atom: [4](#0-3) 

```rust
ObjectType::SmallAtom => {
    let val = node.index();
    let len = len_for_value(val) as u32;
    bounds_check(node, start, end, len)?;
    let buf: [u8; 4] = val.to_be_bytes();
    let buf = &buf[4 - len as usize..];
    let substr = &buf[start as usize..end as usize];
    if let Some(new_val) = fits_in_small_atom(substr) {
        self.ghost_atoms += 1;
        Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
    } else {
        // ← NO heap-limit check here
        let start = self.u8_vec.len();
        let end = start + substr.len();
        self.u8_vec.extend_from_slice(substr);   // writes to heap unconditionally
        let idx = self.atom_vec.len();
        self.atom_vec.push(AtomBuf { start: start as u32, end: end as u32 });
        Ok(self.mk_node(ObjectType::Bytes, idx))
    }
}
```

The `ObjectType::Bytes` branch of `new_substr` is safe because it creates a new `AtomBuf` window into already-allocated bytes without writing new data. The `SmallAtom`-to-`Bytes` branch is the only allocation path that physically extends `u8_vec` without first checking `heap_limit`.

### Impact Explanation

When a caller constructs an `Allocator` via `Allocator::new_limited(heap_limit)` — as done in `intern_tree_limited` and as implied by the `LIMIT_HEAP` flag in `MEMPOOL_MODE` — the configured limit can be silently exceeded. Each bypassing call writes at most 5 bytes (the maximum byte-length of a `SmallAtom` value, per `len_for_value`). The total bypass is bounded by `MAX_NUM_ATOMS × 5` bytes because `check_atom_limit()` is still called at the top of `new_substr`. The corrupted invariant is: `u8_vec.len() > heap_limit` after a sequence of `substr` calls on SmallAtom nodes, violating the guarantee that `Allocator::new_limited` provides.

**Impact: Low** — bounded bypass of a resource limit; does not cause consensus divergence or signature mismatch, but can cause the allocator to exceed its configured memory ceiling in mempool/limited-heap mode.

### Likelihood Explanation

**Likelihood: High** — the `substr` operator (opcode 12) is a standard CLVM operator reachable by any attacker-supplied program. Constructing a SmallAtom whose sub-slice has a high bit set (e.g., `0x80`) is trivial. The attacker only needs to supply a program that calls `substr` on such a value repeatedly. [5](#0-4) 

```rust
12 => op_substr,
```

### Recommendation

Add the same heap-limit guard to the `SmallAtom`-to-`Bytes` branch of `new_substr`, mirroring the pattern used in `new_atom`:

```rust
} else {
    let start = self.u8_vec.len();
    // Add this guard:
    if start + self.ghost_heap + substr.len() > self.heap_limit {
        return Err(EvalErr::OutOfMemory);
    }
    let end = start + substr.len();
    self.u8_vec.extend_from_slice(substr);
    ...
}
```

Also add a corresponding test analogous to `test_new_concat_heap_limit` and `test_new_atom_heap_limit` that exercises `new_substr` on a SmallAtom against a limited allocator.

### Proof of Concept

```rust
// Construct a SmallAtom whose bytes contain 0x80 (does not fit in small atom).
// SmallAtom for value 0x80 = 128: len_for_value(128) = 2 bytes → [0x00, 0x80]
// substr([0x00, 0x80], 1, 2) = [0x80] → fits_in_small_atom returns None
// → writes 1 byte to u8_vec without checking heap_limit

let mut a = Allocator::new_limited(1); // heap_limit = 1 byte
// ghost_heap starts at 1, so 0 bytes are available for new_atom
// But new_substr on a SmallAtom bypasses this:
let small = a.new_small_number(128).unwrap_err(); // correctly fails: OutOfMemory

// However, if we pre-create the SmallAtom before the limit is hit:
let mut a = Allocator::new_limited(10);
let small = a.new_small_number(128).unwrap(); // ghost_heap += 2, still under limit
// Now exhaust remaining space via new_atom to reach the limit...
// Then call new_substr(small, 1, 2) → writes 0x80 to u8_vec without checking limit
// u8_vec.len() now exceeds heap_limit
```

The attacker-controlled CLVM program:
```
(substr (q . 0x0080) (q . 1) (q . 2))  ; substr of SmallAtom 0x0080, bytes [1..2] = [0x80]
```

Repeated invocations via a loop accumulate heap bytes past the configured limit, bypassing the restriction that `new_atom`, `new_small_number`, and `new_concat` all enforce.
