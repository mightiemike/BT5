Audit Report

## Title
First-swap watermark bootstrap bypasses stop-loss on every bin — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension` initialises per-bin watermarks lazily: the first swap touching any bin always passes the stop-loss check because the stored watermark is `0`, and `uint256 metric >= 0` is unconditionally true. The watermark is then written to the post-drain metric, permanently anchoring the baseline to the already-drained value and violating the NatSpec guarantee at lines 27–28.

## Finding Description
`_checkAndUpdateWatermarks` reads `hwmS.token0` and `hwmS.token1` from storage. For any bin that has never been touched, both are `0`. The call chain is:

`_decayed(0, rate, dt)` at line 320 short-circuits on `hwm == 0` and returns `0`. [1](#0-0) 

`_applyWatermark(metric, 0, floorMultiplier)` at line 333 evaluates `metric >= 0`, which is unconditionally true for `uint256`, and returns `(metric, false)` — no breach, no revert. [2](#0-1) 

Lines 281–284 then write `hwmS.token0 = uint104(hwm0)` where `hwm0` equals the post-drain metric, permanently anchoring the baseline to the already-drained value. [3](#0-2) 

The contract implements no `afterAddLiquidity` hook to auto-seed watermarks on deposit. The only seeding path is `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks`, which requires pool-admin privilege and a full timelock delay, making it impossible to atomically precede the first public swap. [4](#0-3) 

## Impact Explanation
An unprivileged trader who executes the first swap on any bin can cause an arbitrarily large value-per-share drop without triggering the stop-loss revert. LPs suffer direct principal loss equal to the drained value. The watermark is then set to the post-drain metric, so the drawdown floor for all subsequent swaps is computed relative to the already-drained baseline rather than the pre-drain LP value. This directly violates the documented invariant: *"value per share at oracle marks cannot fall faster than drawdown (one-time) + decay * t."* [5](#0-4) 

## Likelihood Explanation
Every newly created pool starts with all-zero watermarks for every bin. Any public swap on a fresh pool or on a bin that has never been touched is sufficient to exploit this. No special role, oracle manipulation, or non-standard token is required. The attacker only needs to be the first caller of `swap` on the target bin, which is trivially achievable by monitoring the mempool or acting immediately after pool creation.

## Recommendation
Seed watermarks at the time liquidity is first added to a bin by implementing `afterAddLiquidity` in `OracleValueStopLossExtension`. For each bin in the liquidity delta whose stored watermark is still `0`, compute the current metric and write it as the initial watermark. Alternatively, treat `hwm == 0` as "unset" inside `_applyWatermark` and skip the breach check while still writing the metric — but also emit an event so the admin can verify the seed value. The admin-driven `proposeOracleStopLossHighWatermarks` path is insufficient on its own because it cannot atomically precede the first public swap. [6](#0-5)

    // 3. Attacker executes first swap — drains bin to near-zero
    //    (simulate post-drain state: 100 t0, 100 t1)
    _storeBin(0, 100, 100, BIN_SHARES);

    // 4. afterSwap hook fires — should revert but does NOT
    //    because hwmS.token0 == 0 → _decayed returns 0
    //    → _applyWatermark: metric >= 0 always true → (metric, false)
    _exposeStopLoss(0, 0, uint128(Q64), true); // no revert

    (uint256 hwm0,) = extension.currentHighWatermarks(address(mockPool), 0);

    assertEq(hwm0, drainedMetric); // watermark == post-drain, not pre-drain
}
