### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable and Accepts Internal Node Hashes as Valid Transaction IDs — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully public, externally callable NEAR contract method. Rust's `#[deprecated]` attribute is a compile-time lint only — it provides zero runtime access restriction. Any unprivileged NEAR account can call it directly. The function passes the caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` without any leaf-level validation, making it trivially exploitable via the 64-byte transaction forgery attack.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub fn` with only a `#[pause]` guard (no `#[private]`, no role restriction): [1](#0-0) 

The `#[deprecated]` attribute at line 283 emits a compiler warning to callers of the Rust crate — it does **not** prevent the NEAR runtime from dispatching calls to this method. The method name `verify_transaction_inclusion` remains registered in the contract's ABI and is reachable via any NEAR RPC `call` or `view` transaction.

Inside the function, `args.tx_id` is passed verbatim to `compute_root_from_merkle_proof`: [2](#0-1) 

`compute_root_from_merkle_proof` treats its first argument as an opaque 32-byte value and iteratively hashes it with the proof siblings — it has no mechanism to distinguish a leaf transaction hash from an internal Merkle node hash: [3](#0-2) 

The function's own documentation acknowledges this at line 278: [4](#0-3) 

The v2 function mitigates this by requiring a coinbase proof at depth 0, which cryptographically binds the proof depth and prevents an internal node from being presented as a leaf: [5](#0-4) 

v1 has no equivalent guard.

---

### Impact Explanation

**Concrete attack scenario** — given a real Bitcoin block with transactions `[T0, T1, T2, T3]` whose Merkle root is `R`:

1. Compute `N01 = double_sha256(T0 || T1)` (a real internal node, derivable from any block explorer).
2. Compute `N23 = double_sha256(T2 || T3)`.
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N01`
   - `tx_index = 0`
   - `merkle_proof = [N23]`
   - `tx_block_blockhash` = any block hash already stored in the light client where `merkle_root == R`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N01, 0, [N23])` = `double_sha256(N01 || N23)` = `R` = `header.block_header.merkle_root` → returns **`true`**.

Any consumer contract that gates an action (e.g., releasing bridged funds, confirming a cross-chain payment) on a `true` return from `verify_transaction_inclusion` v1 can be deceived into accepting `N01` — a 64-byte concatenation of two real transaction IDs — as a confirmed transaction that never existed.

---

### Likelihood Explanation

- The function is callable by any NEAR account with no role or key restriction.
- The required inputs (internal node hashes) are fully derivable from public Bitcoin block data.
- No privileged access, leaked keys, or social engineering is required.
- The contract's own documentation explicitly warns that v1 is vulnerable, yet the method remains live.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or add `#[private]` to prevent external dispatch. Keeping a deprecated-but-callable method that the codebase itself documents as exploitable creates an ongoing attack surface for any integrator that has not yet migrated to v2.

---

### Proof of Concept

```rust
// Pseudocode for a near-workspaces integration test
let block = /* submit a real BTC block with ≥4 txs */;
let t0 = /* coinbase tx hash */;
let t1 = /* tx[1] hash */;
let t2 = /* tx[2] hash */;
let t3 = /* tx[3] hash */;

let n01 = double_sha256([t0, t1].concat());  // internal node — NOT a real tx
let n23 = double_sha256([t2, t3].concat());

// v1: returns true for an internal node hash
let result: bool = user.view(contract.id(), "verify_transaction_inclusion")
    .args_borsh(ProofArgs {
        tx_id: n01,
        tx_block_blockhash: block.block_hash(),
        tx_index: 0,
        merkle_proof: vec![n23],
        confirmations: 1,
    })
    .await?.json()?;
assert!(result);  // passes — forgery accepted

// v2: rejects the same internal node
let result_v2 = user.view(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(ProofArgsV2 {
        tx_id: n01,
        tx_block_blockhash: block.block_hash(),
        tx_index: 0,
        merkle_proof: vec![n23],
        coinbase_tx_id: t0,
        coinbase_merkle_proof: vec![/* valid coinbase siblings */],
        confirmations: 1,
    })
    .await;
// coinbase proof depth != tx proof depth → panics or returns false
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
