### Title
`DISABLE_OP` Permanently Disables `modpow` in `MEMPOOL_MODE` With No Re-Enable Mechanism, Creating Irremovable Mempool/Consensus Divergence — (`File: src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::DISABLE_OP` (0x200) is hardwired into the public `MEMPOOL_MODE` constant and unconditionally rejects opcode 60 (`modpow`) with `EvalErr::Unimplemented`. There is no corresponding flag or API to re-enable `modpow` within mempool mode. The flag carries no documentation comment. Once `MEMPOOL_MODE` is in use, the `modpow` restriction cannot be lifted without a source-level change to `MEMPOOL_MODE` itself — a direct analog to the "whitelist that can be enabled but never disabled" pattern in the reference report.

---

### Finding Description

`DISABLE_OP` is defined without any doc comment, unlike every other flag in the `ClvmFlags` bitset:

```rust
const DISABLE_OP = 0x200;   // ← no documentation
``` [1](#0-0) 

It is unconditionally included in the public `MEMPOOL_MODE` constant:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)          // ← always present
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

Inside `ChiaDialect::op`, when `DISABLE_OP` is set, opcode 60 (`modpow`) is hard-rejected before the operator function is even called:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [3](#0-2) 

There is no `ENABLE_MODPOW` flag, no `ENABLE_OP` counterpart, and no runtime path that allows a caller using `MEMPOOL_MODE` to selectively re-enable opcode 60. The only escape is to construct a fully custom `ClvmFlags` set from scratch — but `DISABLE_OP` is undocumented, so callers have no indication which bit to omit.

`DISABLE_OP` is also exported to Python callers as a named constant:

```rust
m.add("DISABLE_OP", ClvmFlags::DISABLE_OP.bits())?;
``` [4](#0-3) 

making it part of the public API surface with no explanation of its semantics.

**Secondary observation — dead-code size checks:** `DISABLE_OP` also gates a 2048-byte operand check inside `op_div`, `op_divmod`, and `op_mod`:

```rust
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if a0_len > 256 || a1_len > 1024 {   // unconditional, stricter
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
``` [5](#0-4) 

The `DISABLE_OP`-gated check (2048 bytes) is dead code: the unconditional check (256 bytes) is always stricter and fires first. This means `DISABLE_OP`'s only live effect is the total suppression of `modpow`.

---

### Impact Explanation

**Vulnerability class:** flag/operator wiring error — one-directional restriction with no re-enable path.

After the `modpow` soft-fork activates on-chain:

- Consensus nodes (running without `DISABLE_OP`) will **accept** any CLVM spend bundle that calls opcode 60.
- Mempool nodes using `MEMPOOL_MODE` will **permanently reject** those same bundles with `EvalErr::Unimplemented`, because `DISABLE_OP` is baked into `MEMPOOL_MODE` with no runtime toggle to remove it.

This is a concrete mempool/consensus divergence: attacker-crafted CLVM bytes invoking opcode 60 bypass mempool validation entirely and land directly in consensus. The exact corrupted result is `EvalErr::Unimplemented` returned by the mempool for a program that is valid and accepted by consensus.

The `new-operator-checklist.md` explicitly states the intent to eventually remove soft-fork activation logic:

> "Once a soft-fork has activated… all logic surrounding activating or deactivating the soft-fork should be removed." [6](#0-5) 

But the current design provides no mechanism to do this at runtime — the restriction is structurally irremovable from `MEMPOOL_MODE` without a source change.

**Impact:** Medium — permanent mempool/consensus divergence for all programs using `modpow` after activation.

---

### Likelihood Explanation

**Likelihood:** Medium.

- The trigger is straightforward: any CLVM program serialized with opcode byte `0x3c` (60 = `modpow`) submitted to a mempool node running `MEMPOOL_MODE`.
- The divergence is latent today (pre-activation) but becomes active the moment the `modpow` soft-fork activates on the Chia network.
- The `modpow` operator is already fully implemented, has test vectors, and is benchmarked — activation is a planned near-term event.
- The Python API exports `DISABLE_OP` without documentation, increasing the chance that downstream integrators (e.g., `chia_rs`) will propagate the flag without understanding its irremovable effect on opcode 60.

---

### Recommendation

1. **Add a dedicated `ENABLE_MODPOW` flag** (or rename `DISABLE_OP` to `DISABLE_MODPOW`) so the restriction on opcode 60 can be independently toggled from any other mempool-mode restrictions.
2. **Add a doc comment to `DISABLE_OP`** matching the style of every other flag in the bitset, explaining exactly which opcode(s) it suppresses and under what activation conditions it should be removed from `MEMPOOL_MODE`.
3. **Remove the dead-code size checks** in `op_div`, `op_divmod`, and `op_mod` that are gated on `DISABLE_OP` but can never fire (the unconditional 256-byte limit is always stricter).

Example corrected design:

```rust
/// Disables modpow (opcode 60) in mempool mode until the modpow soft-fork activates.
/// Remove from MEMPOOL_MODE once the fork is live on-chain.
const DISABLE_MODPOW = 0x200;

pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_MODPOW)   // remove after modpow activation
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

### Proof of Concept

**Entry path:** attacker submits serialized CLVM bytes to a mempool node running `MEMPOOL_MODE`.

```
Program bytes (CLVM): (modpow (q . 2) (q . 10) (q . 1000))
                       opcode 60 = 0x3c
```

**Mempool node** (`MEMPOOL_MODE` → `DISABLE_OP` set):
```
ChiaDialect::op → match op { 60 => { DISABLE_OP set → EvalErr::Unimplemented } }
→ transaction REJECTED
```

**Consensus node** (no `DISABLE_OP`):
```
ChiaDialect::op → match op { 60 => op_modpow }
→ op_modpow(2, 10, 1000) = 24 → transaction ACCEPTED
```

The exact divergence point is `src/chia_dialect.rs` lines 239–244: the `DISABLE_OP` branch fires in mempool, is absent in consensus, and there is no flag that can bridge the gap within `MEMPOOL_MODE`. [3](#0-2) [2](#0-1) [7](#0-6)

pub fn op_modpow(a: &mut Allocator, input: NodePtr, max_cost: Cost, flags: ClvmFlags) -> Response {
    if flags.contains(ClvmFlags::MALACHITE) {
        return op_modpow_malachite(a, input, max_cost, flags);
    }
    let [base, exponent, modulus] = get_args::<3>(a, input, "modpow")?;

    let mut cost = MODPOW_BASE_COST;
    let (base, bsize) = int_atom(a, base, "modpow")?;
    cost += bsize as Cost * MODPOW_COST_PER_BYTE_BASE_VALUE;
    let (exponent, esize) = int_atom(a, exponent, "modpow")?;
    cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;
    check_cost(cost, max_cost)?;
    let (modulus, msize) = int_atom(a, modulus, "modpow")?;
    cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;
    check_cost(cost, max_cost)?;

    if bsize > 256 || esize > 256 || msize > 256 {
        return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
    }

    if exponent.sign() == Sign::Minus {
        return Err(EvalErr::InvalidOpArg(
            input,
            "ModPow with Negative Exponent".to_string(),
        ));
    }

    if modulus.sign() == Sign::NoSign {
        return Err(EvalErr::DivisionByZero(input));
    }

    let ret = base.modpow(&exponent, &modulus);
    let ret = a.new_number(ret)?;
    Ok(malloc_cost(a, cost, ret))
}
