### Title
Unrestricted `run_mainchain_gc` Allows Any Caller to Permanently Prune Mainchain Headers, Breaking `verify_transaction_inclusion` for Targeted Blocks - (`contract/src/lib.rs`)

### Summary

`run_mainchain_gc` in `contract/src/lib.rs` carries no caller restriction when the contract is unpaused — any NEAR account can invoke it with an arbitrarily large `batch_size`. This lets an attacker aggressively prune mainchain block headers that are within the GC window but have not yet been removed by the normal gradual path. Once pruned, both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` permanently panic for those blocks because `mainchain_header_to_height` no longer contains them. Any dApp that gates asset releases on an SPV proof for a targeted block is permanently blocked.

### Finding Description

**Root cause — missing caller guard on a destructive state-mutation function.**

`submit_blocks` is gated by `#[trusted_relayer]`, meaning only accounts that have been granted a trusted-relayer role can add block headers. [1](#0-0) 

`run_mainchain_gc`, which *removes* block headers, carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. That attribute controls behaviour when the contract is paused; it imposes **no caller restriction when the contract is running normally**. Any NEAR account can call it. [2](#0-1) 

**How GC works.**

When `amount_of_headers_we_store > gc_threshold`, the function removes `min(total_amount_to_remove, batch_size)` of the oldest mainchain blocks. It deletes each block from `headers_pool` and `mainchain_height_to_header`, then advances `mainchain_initial_blockhash`. [3](#0-2) 

The normal call-site inside `submit_blocks` passes `batch_size = num_of_headers` — the count of headers in that single submission batch — so GC is gradual. [4](#0-3) 

An attacker bypasses this by calling `run_mainchain_gc` directly with `batch_size = u64::MAX`, collapsing all eligible blocks in one transaction.

**How this breaks verification.**

`verify_transaction_inclusion` looks up the target block in `mainchain_header_to_height`. If the block has been GC'd, the lookup returns `None` and the function panics with `"block does not belong to the current main chain"`. [5](#0-4) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` internally, so it is equally affected. [6](#0-5) 

The removal is **permanent**. A GC'd block's parent is no longer the mainchain tip, so re-submitting it lands in the fork path and is never inserted into `mainchain_header_to_height`. The block is unrecoverable for proof purposes. [7](#0-6) 

**End-to-end exploit path.**

1. Contract is live with `gc_threshold = N`; it currently stores `N + K` blocks (`K` blocks are eligible for GC but have not been removed yet because the relayer submits in small batches).
2. A user (or a dApp making a cross-contract call) is about to call `verify_transaction_inclusion` for a block at height `tip − N − j` (where `1 ≤ j ≤ K`), which is still present in storage.
3. Attacker calls `run_mainchain_gc(u64::MAX)` from any unprivileged NEAR account. All `K` eligible blocks are deleted in one shot, including the target block.
4. The user's subsequent call to `verify_transaction_inclusion` panics permanently. Any dApp cross-contract call that depended on this proof fails and cannot be retried for that block.

No privileged keys, no leaked secrets, no social engineering — only a single public NEAR transaction is required.

### Impact Explanation

The `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` API is the contract's sole security primitive for SPV proofs. dApps that gate NEAR-side asset releases on a successful proof call will be permanently blocked for any block an attacker chooses to prune. Because the removal is irreversible and the function panics (not returns `false`), the dApp's cross-contract call fails with no recovery path for that specific block. This constitutes permanent loss of the ability to settle Bitcoin-backed claims on NEAR for targeted transactions.

### Likelihood Explanation

The attack requires no privileged role, no staked assets, and no special knowledge beyond identifying which block hash a target user or dApp intends to verify. The function is unconditionally callable by any NEAR account when the contract is unpaused. The asymmetry — `submit_blocks` is role-gated while `run_mainchain_gc` is open — makes this straightforward to discover and exploit.

### Recommendation

Apply the same `#[trusted_relayer]` guard (or an equivalent role check) to `run_mainchain_gc` so that only accounts authorised to submit blocks can also trigger GC. Alternatively, remove the public entry point entirely and rely solely on the automatic GC invocation inside `submit_blocks`, which already bounds `batch_size` to the number of submitted headers.

```rust
// Before
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { … }

// After — restrict to trusted relayers (same guard as submit_blocks)
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[trusted_relayer]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { … }
```

### Proof of Concept

```
// Attacker account: any_near_account.testnet (no role required)
// Target: block at height (tip - gc_threshold - 1), still in storage

near call <contract_id> run_mainchain_gc \
  '{"batch_size": 18446744073709551615}' \
  --accountId any_near_account.testnet

// Now call verify_transaction_inclusion for the target block:
near call <contract_id> verify_transaction_inclusion \
  '<borsh-encoded ProofArgs with tx_block_blockhash = target_block>' \
  --accountId victim.testnet
// → panics: "block does not belong to the current main chain"
// Permanent. No retry possible for this block.
```
