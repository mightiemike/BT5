### Title
Slow Mode Fees Permanently Locked in `Endpoint` Contract — No Withdrawal Mechanism — (File: `core/contracts/EndpointStorage.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

Every non-privileged slow mode transaction charges a `SLOW_MODE_FEE` in USDC, which is transferred into the `Endpoint` contract and tracked in `slowModeFees`. However, unlike `sequencerFee` (which is claimed via the `DumpFees` path), `slowModeFees` is never decremented, claimed, or forwarded anywhere. The collected USDC is permanently locked in the `Endpoint` contract.

---

### Finding Description

In `EndpointStorage.chargeSlowModeFee`, the slow mode fee is pulled from the caller and deposited directly into `address(this)` (the Endpoint):

```solidity
function chargeSlowModeFee(IERC20Base token, address from) internal virtual {
    require(address(token) != address(0));
    token.safeTransferFrom(from, address(this), clearinghouse.getSlowModeFee());
}
``` [1](#0-0) 

In `EndpointTx.submitSlowModeTransactionImpl`, this fee is charged for every slow mode transaction that is not owner-gated, and the accounting variable is incremented:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;
``` [2](#0-1) 

The `DumpFees` slow mode transaction type — the only fee-claiming path in the protocol — handles only the per-product `sequencerFee` mapping and forwards those to the clearinghouse via `claimSequencerFees`. It does **not** touch `slowModeFees`:

```solidity
} else if (txType == IEndpoint.TransactionType.DumpFees) {
    IOffchainExchange(offchainExchange).dumpFees();
    uint32[] memory spotIds = spotEngine.getProductIds();
    int128[] memory fees = new int128[](spotIds.length);
    for (uint256 i = 0; i < spotIds.length; i++) {
        fees[i] = sequencerFee[spotIds[i]];
        sequencerFee[spotIds[i]] = 0;
    }
    requireSubaccount(X_ACCOUNT);
    clearinghouse.claimSequencerFees(fees);
}
``` [3](#0-2) 

`slowModeFees` is declared as `int128 internal slowModeFees` and is only ever incremented — never decremented, never forwarded, never claimed: [4](#0-3) 

There is no function in `EndpointTx`, `EndpointStorage`, or `WithdrawPool` that withdraws or routes the USDC tokens accumulated from slow mode fees out of the Endpoint contract.

---

### Impact Explanation

**Impact: Medium.** Real USDC (the quote token) is transferred into the Endpoint contract on every slow mode submission by an unprivileged user. Because no withdrawal or claim path exists for `slowModeFees`, this USDC accumulates and is permanently locked. The protocol loses the ability to recover or redistribute these fees (e.g., to the sequencer or insurance fund), representing a direct, ongoing loss of protocol revenue.

---

### Likelihood Explanation

**Likelihood: High.** Every slow mode transaction submitted by a non-owner user — including `WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`, and others — triggers `chargeSlowModeFee`. This is a normal, expected user flow. The fee accumulation begins from the first user-submitted slow mode transaction and continues indefinitely. [5](#0-4) 

---

### Recommendation

Add a claim path for `slowModeFees` analogous to the existing `DumpFees` handler for `sequencerFee`. Specifically:

1. In `processSlowModeTransactionImpl`, extend the `DumpFees` branch (or add a new transaction type) to read `slowModeFees`, reset it to zero, and forward the corresponding USDC to the clearinghouse or sequencer address.
2. Alternatively, in `chargeSlowModeFee`, route the fee directly to the clearinghouse rather than holding it in the Endpoint, consistent with how deposit collateral flows work via `handleDepositTransfer`.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction` with a `WithdrawCollateral` transaction.
2. `submitSlowModeTransactionImpl` reaches the `else` branch, calls `chargeSlowModeFee(_getQuote(), sender)`, transferring `SLOW_MODE_FEE` USDC from the user to `address(this)` (Endpoint), and increments `slowModeFees`.
3. Sequencer later calls `submitSlowModeTransaction` with a `DumpFees` transaction.
4. `processSlowModeTransactionImpl` handles `DumpFees`: iterates `sequencerFee` per product, resets them, and calls `clearinghouse.claimSequencerFees(fees)`. `slowModeFees` is never read or reset.
5. The USDC from step 2 remains in the Endpoint contract with no reachable path to recover it. [1](#0-0) [3](#0-2) [5](#0-4)
