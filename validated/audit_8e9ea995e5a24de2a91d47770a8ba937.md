Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on Caller-Controlled `owner` Instead of Actual Depositor `sender`, Fully Bypassing the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `pool.addLiquidity()`) and instead validates the caller-supplied `owner` argument. Because `owner` is a free parameter with no binding constraint in `addLiquidity`, any unprivileged actor can pass any allowlisted address as `owner`, causing the allowlist check to pass while the actor's own tokens are deposited into the pool. This completely nullifies the admin-configured deposit restriction.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes `msg.sender` as `sender` to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

There is no `msg.sender == owner` check on the add path. By contrast, `removeLiquidity` enforces this identity:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` ABI-encodes both `sender` and `owner` and forwards them verbatim to every configured extension: [3](#0-2) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed, discarding it entirely, and gates only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [4](#0-3) 

Here `msg.sender` inside the extension is the pool address (the caller of the extension), so the check resolves to `allowedDepositor[pool][owner]`. Since `owner` is freely chosen by the attacker, setting it to any known allowlisted address makes the check pass unconditionally. The actual depositor identity (`sender`) is never validated.

## Impact Explanation

**Admin-boundary break / broken core pool functionality.** The `DepositAllowlistExtension` is the sole mechanism by which a pool admin restricts LP entry (e.g., KYC/AML, whitelist-only pools). The bypass is total:

1. Any unprivileged actor calls `pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)`. The extension sees `allowedDepositor[pool][allowlistedAddress] == true` and does not revert.
2. The actor's tokens are pulled from the actor's contract via the add-liquidity callback mechanism, deposited into the pool, and LP shares are credited to `allowlistedAddress`.
3. Pool bin state (`curBinIdx`, `curPosInBin`, `binTotals`) is mutated by an unauthorized actor, affecting the price curve for all subsequent swappers.
4. Allowlisted LPs may be force-credited with positions they did not request, diluting or altering their bin exposure without consent.

The attacker cannot withdraw (blocked by `msg.sender != owner` in `removeLiquidity`), but the attack is a zero-cost pool-state manipulation tool that completely nullifies the admin's security boundary.

## Likelihood Explanation

**High.** The bypass requires only a single direct call to `pool.addLiquidity` with `owner` set to any known allowlisted address. No special role, flash loan, or oracle manipulation is needed. The `allowedDepositor` mapping is public storage, so any on-chain observer can enumerate valid `owner` values. The pool imposes no constraint tying `msg.sender` to `owner` on the add path. [5](#0-4) 

## Recommendation

Replace the unnamed first parameter with `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended semantic is "only allowlisted addresses may own positions", both `sender` and `owner` should be checked. If the intended semantic is "only allowlisted actors may initiate deposits", checking `sender` alone is sufficient.

## Proof of Concept

**Setup:**
- Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
- Pool admin calls `setAllowedToDeposit(P, alice, true)` — Alice is the only allowlisted depositor.
- Bob is not allowlisted (`allowedDepositor[P][bob] == false`).

**Attack:**
```solidity
// Bob (not allowlisted) calls addLiquidity with owner = alice
pool.addLiquidity(
    alice,          // owner — allowlisted, passes the guard
    0,              // salt
    deltas,         // bin shares Bob wants to inject
    callbackData,   // Bob's contract pays tokens in the callback
    extensionData
);
// Extension checks: allowedDepositor[pool][alice] == true → no revert
// Bob's tokens are pulled from Bob's contract via the add-liquidity callback
// LP shares are credited to alice at key keccak256(alice, 0)
// Bob has bypassed the allowlist and manipulated pool bin state
```

**Verification:** A Foundry integration test deploying a pool with `DepositAllowlistExtension`, allowlisting only Alice, then calling `addLiquidity` from Bob with `owner = alice` will confirm the call succeeds without revert, with Bob's token balance decreasing and pool `binTotals` updated.
