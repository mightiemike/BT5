Audit Report

## Title
`uint32` Timestamp Truncation in `lastDecayTs` Inflates `dt`, Permanently Disabling the Oracle Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`BinHighWatermarks.lastDecayTs` is stored as `uint32` but elapsed time `dt` is computed in `uint256` space without compensating for wrap-around. After year 2106, when `block.timestamp > type(uint32).max`, the stored `lastDecayTs` wraps to a small value while `block.timestamp` does not, inflating `dt` to approximately `4.3 × 10^9` seconds. This forces `_decayed()` to always return `0`, collapsing every per-bin high watermark to zero and permanently disabling the stop-loss guard for every pool using this extension.

## Finding Description

The write path truncates `block.timestamp` to `uint32`:

```solidity
// Line 284
hwmS.lastDecayTs = uint32(block.timestamp);

// Line 174 (executeOracleStopLossHighWatermarks)
BinHighWatermarks({..., lastDecayTs: uint32(block.timestamp)});
```

The read path computes `dt` in `uint256` space without accounting for the wrap:

```solidity
// Line 268 (_checkAndUpdateWatermarks)
uint256 dt = block.timestamp - hwmS.lastDecayTs;

// Line 74 (currentHighWatermarks)
uint256 dt = block.timestamp - hwm.lastDecayTs;
```

After `block.timestamp` exceeds `2^32 − 1`, `uint32(block.timestamp)` wraps to `block.timestamp % 2^32` (a small value, e.g. `1`), while the subtraction `block.timestamp - uint256(lastDecayTs)` is performed in full `uint256` space, yielding `dt ≈ block.timestamp ≈ 4.3 × 10^9`.

`_decayed` then evaluates:

```solidity
// Lines 321-322
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;
```

With `dt ≈ 4.3 × 10^9` and even the minimum non-zero `ratePerSecondE8 = 1`, `factor = 4.3 × 10^9 ≫ E8 = 10^8`, so `_decayed` always returns `0`. Every watermark collapses to zero.

`_applyWatermark` then receives `hwm = 0`:

```solidity
// Line 333
if (metric >= hwm) return (metric, false);
```

Since `metric >= 0` is always true, no breach is ever reported. The stop-loss guard is silently and permanently disabled for every subsequent `afterSwap` call.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from oracle-driven value drain. Once `lastDecayTs` wraps, every `afterSwap` hook call silently passes without checking any watermark. Any swap — including one that drains a bin's value far below the configured drawdown floor — proceeds without triggering the guard. LP funds that the extension was configured to protect are exposed to unbounded loss with no on-chain recourse. This matches the allowed impact gate: broken core pool functionality causing loss of funds, and oracle guard path failing open instead of closed.

## Likelihood Explanation

The trigger is `block.timestamp > uint32.max`, occurring in year 2106. The contract is not described as upgradeable in the periphery layer, so there is no automatic remediation path. Once `block.timestamp` exceeds `uint32.max`, every subsequent `afterSwap` call is affected for the lifetime of every pool using this extension. Any unprivileged trader executing a swap after that point benefits from the disabled guard, with no special capability required.

## Recommendation

Store `lastDecayTs` as `uint256` to eliminate truncation entirely:

```solidity
// In BinHighWatermarks struct: change uint32 lastDecayTs → uint256 lastDecayTs
hwmS.lastDecayTs = block.timestamp;
```

Alternatively, if storage packing requires `uint32`, compute `dt` in `uint32` space with `unchecked` to make wrap-around arithmetic intentional and consistent on both sides:

```solidity
uint32 dt;
unchecked {
    dt = uint32(block.timestamp) - hwmS.lastDecayTs;
}
```

## Proof of Concept

1. Deploy `OracleValueStopLossExtension` with non-zero `drawdownE6` and `decayPerSecondE8 = 1`.
2. Call `executeOracleStopLossHighWatermarks` to set a non-zero watermark for a bin; `lastDecayTs` is written as `uint32(block.timestamp)`.
3. Warp `block.timestamp` to `4_294_967_297` (just past `uint32.max`). `uint32(block.timestamp) = 1`.
4. Call `afterSwap` on the pool. Inside `_checkAndUpdateWatermarks`:
   - `dt = 4_294_967_297 - 1 = 4_294_967_296`
   - `factor = 1 * 4_294_967_296 = 4_294_967_296 >= 1e8` → `_decayed` returns `0`
   - `_applyWatermark(metric, 0, floorMultiplier)` → `metric >= 0` → returns `(metric, false)`
5. The swap completes even though the bin's value per share has fallen below the configured drawdown floor. The stop-loss guard never fires. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
