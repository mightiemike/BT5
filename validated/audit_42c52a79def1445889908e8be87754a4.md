Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks attacker-controlled `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller providing tokens) and instead validates `owner` (the LP-position recipient), which `addLiquidity` accepts from any caller without restriction. Any unprivileged address can bypass the deposit allowlist by supplying an allowlisted address as `owner`, depositing tokens via callback, and causing LP shares to be minted to that address.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`addLiquidity` imposes no constraint on the `owner` argument — unlike `removeLiquidity`, which enforces `msg.sender == owner`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `(sender, owner, ...)` but leaves `sender` unnamed and checks only `owner`: [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it names and checks `sender`: [4](#0-3) 

Because `owner` is fully attacker-controlled, an attacker sets `owner = allowlistedAddress`, causing the extension to evaluate `allowedDepositor[pool][allowlistedAddress]` → `true`, and the deposit proceeds. The attacker's callback transfers the tokens; LP shares are minted to the allowlisted address.

## Impact Explanation
The deposit allowlist is completely ineffective. Any unprivileged address can add liquidity to a pool the admin intended to restrict. Concrete consequences include: (1) pool state manipulation — attacker deposits into specific bins to shift the pool's internal price position and then swaps against those bins, with LP loss borne by the involuntary allowlisted position holder; (2) griefing of allowlisted LPs — allowlisted addresses receive unsolicited LP positions in bins they did not choose, exposing them to impermanent loss and fee dilution; (3) the pool admin's core access-control invariant (restricted liquidity provision) is silently voided.

## Likelihood Explanation
Exploitation requires only a standard `addLiquidity` call with `owner` set to any address present in `allowedDepositor`. Allowlisted addresses are discoverable from on-chain `AllowedToDepositSet` events. No special privilege, flash loan, or oracle manipulation is required. Any unprivileged address can trigger this at any time, repeatably.

## Recommendation
Replace the `owner` check with a `sender` check, matching the contract's stated intent and the `SwapAllowlistExtension` pattern:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool P is deployed with `DepositAllowlistExtension`; `allowedDepositor[P][alice] = true`; `allowedDepositor[P][attacker] = false`.
2. Attacker calls `P.addLiquidity(owner = alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(msg.sender=attacker, owner=alice, ...)`.
4. Extension receives `(sender=attacker, owner=alice)`, ignores `attacker`, checks `allowedDepositor[P][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` executes; attacker's callback transfers tokens; LP shares are minted to `alice`.
6. Attacker has deposited into a restricted pool, bypassing the allowlist entirely.
