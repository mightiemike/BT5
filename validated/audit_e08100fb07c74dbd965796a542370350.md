### Title
Sequencer Can Overfill Orders Beyond Signed Amount in `matchOrders()` — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary
In `OffchainExchange.matchOrders()`, the fill amount (`amountDelta`) is computed from the **original signed** order amounts without subtracting the already-filled amounts tracked in `filledAmounts`. The sequencer can therefore fill an order beyond its signed total, causing traders to trade more than they authorized.

---

### Finding Description

`_validateOrder()` subtracts `filledAmounts[orderDigest]` from a **local copy** of `order.amount` to verify the order has remaining capacity, but this subtraction never propagates back to the `taker` and `maker` variables in `matchOrders()`.

Inside `_validateOrder()`:

```solidity
IEndpoint.Order memory order = signedOrder.order;   // local copy
// ...
int128 filledAmount = filledAmounts[orderDigest];
order.amount -= filledAmount;                        // modifies local copy only
// ...
return ... && (order.amount != 0) && ...;
``` [1](#0-0) 

Because `order` is a local `memory` copy of `signedOrder.order`, the subtraction does not affect `taker.order.amount` or `maker.order.amount` back in `matchOrders()`. After `_validateOrder` returns, both still hold the **original signed amounts**.

For **non-TWAP** orders, `amountDelta` is then computed as:

```solidity
ordersInfo.taker.amountDelta = MathHelper.min(
    taker.order.amount,    // original signed amount — NOT remaining
    -maker.order.amount
);
``` [2](#0-1) 

For **TWAP** orders, the bound check is also against the original amount:

```solidity
require(
    taker.order.amount >= txn.takerAmountDelta &&   // original, NOT remaining
        maker.order.amount <= -txn.takerAmountDelta,
    ERR_INVALID_TAKER
);
``` [3](#0-2) 

In both paths, if `filledAmounts[digest] > 0`, the sequencer can submit a fill whose `amountDelta` exceeds the remaining capacity, causing `filledAmounts[digest]` to exceed the original signed `order.amount`.

There is no compensating health check: `isHealthy()` in `OffchainExchange` unconditionally returns `true` and provides no backstop. [4](#0-3) 

The `filledAmounts` update at the end of `matchOrders` then records the inflated delta:

```solidity
filledAmounts[ordersInfo.taker.digest] += ordersInfo.taker.amountDelta;
``` [5](#0-4) 

---

### Impact Explanation

A malicious or compromised sequencer can overfill any partially-filled order. For example, a trader who signed an order for 100 units and already had 80 filled could be forced to fill another 100 units (total 180), spending more quote and receiving more base than they authorized. This is a direct, concrete corruption of trader balances — the exact same accounting invariant broken in the external report (`curFillAmount > quoteTokenData.toTrader`).

---

### Likelihood Explanation

Exploitation requires the sequencer to act maliciously or be compromised. The sequencer is the off-chain operator that submits `matchOrders` via `Endpoint.submitTransactions()`. This is the direct analog of the external report's "Solver" — a trusted but potentially adversarial entity that controls fill amounts. The external report treats this actor class as a valid threat; the same applies here.

---

### Recommendation

In `matchOrders()`, subtract `filledAmounts[digest]` from `taker.order.amount` and `maker.order.amount` **before** computing `amountDelta`, mirroring the approach already used in `_validateOrder()`. This caps each fill to the remaining unfilled amount:

```solidity
taker.order.amount -= filledAmounts[ordersInfo.taker.digest];
maker.order.amount -= filledAmounts[ordersInfo.maker.digest];
```

Apply this adjustment after signature validation and before the `amountDelta` computation block.

---

### Proof of Concept

1. Trader signs order: `amount = 100`, digest `D`.
2. First `matchOrders` call: `amountDelta = 60` → `filledAmounts[D] = 60`.
3. Second `matchOrders` call: `_validateOrder` computes remaining `= 100 − 60 = 40` (non-zero → valid). Returns `true`.
4. Back in `matchOrders`, `taker.order.amount` is still `100`. Maker order also has `amount = −100`.
5. `amountDelta = min(100, 100) = 100` — uses original amount, not remaining `40`.
6. `_updateBalances` credits/debits the taker by `100` more units.
7. `filledAmounts[D] += 100` → `filledAmounts[D] = 160`, exceeding the signed `100`.
8. Trader has traded `160` units total against a signed authorization of `100`. [6](#0-5)
