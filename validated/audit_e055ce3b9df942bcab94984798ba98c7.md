### Title
Fee-on-Transfer Token Double-Transfer in `handleDepositTransfer` Causes Deposit DoS and Balance Inflation — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential ERC20 transfers using the same flat `amount` parameter. For fee-on-transfer (FoT) tokens, the first transfer leaves the `Endpoint` holding `amount - fee`, but the second transfer attempts to forward the original `amount` to `Clearinghouse`, causing an unconditional revert. If the revert were bypassed (e.g., via pre-existing balance), `Clearinghouse.depositCollateral` would credit the full `txn.amount` to the subaccount — inflating the on-chain balance beyond actual tokens received.

---

### Finding Description

`EndpointStorage.handleDepositTransfer` is the internal function called for every user deposit:

```solidity
// core/contracts/EndpointStorage.sol L111-119
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);                    // Step 1: user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount);    // Step 2: Endpoint → Clearinghouse
}
```

With a fee-on-transfer token:

- **Step 1**: `safeTransferFrom(token, from, address(this), amount)` — the Endpoint receives `amount - fee`, not `amount`.
- **Step 2**: `safeTransferTo(token, address(clearinghouse), amount)` — the Endpoint attempts to forward the original `amount`, but only holds `amount - fee`. This reverts unconditionally (assuming no pre-existing balance).

Even if the revert were avoided, `Clearinghouse.depositCollateral` credits the full `txn.amount` to the subaccount:

```solidity
// core/contracts/Clearinghouse.sol L205-207
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

The Clearinghouse received `amount - 2*fee` (two FoT deductions across both hops), but the subaccount is credited `amount`. This inflates the subaccount's collateral balance beyond actual token holdings.

The same double-transfer pattern exists in `Clearinghouse.handleWithdrawTransfer`:

```solidity
// core/contracts/Clearinghouse.sol L383-385
token.safeTransfer(withdrawPool, uint256(amount));
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
```

`WithdrawPool` receives `amount - fee` but `submitWithdrawal` calls `handleWithdrawTransfer` which tries to send `amount` to the recipient — reverting withdrawals for FoT tokens as well.

`DirectDepositV1.creditDeposit` compounds the issue by reading the raw token balance and passing it directly as `amount`:

```solidity
// core/contracts/DirectDepositV1.sol L90-98
uint256 balance = token.balanceOf(address(this));
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
}
```

This passes the full `balance` as `amount` into `handleDepositTransfer`, triggering the same revert.

---

### Impact Explanation

**Primary impact — Deposit DoS**: Any user attempting to deposit a fee-on-transfer token via `Endpoint.depositCollateral` or `depositCollateralWithReferral` will have their transaction unconditionally reverted at the second transfer in `handleDepositTransfer`. The token is listed as a valid spot product but is completely unusable as collateral.

**Secondary impact — Balance inflation / undercollateralization**: If the Endpoint held a pre-existing balance of the FoT token (e.g., from a prior partial deposit or airdrop), the second transfer would succeed, but `Clearinghouse.depositCollateral` would credit the full `txn.amount` to the subaccount. The Clearinghouse would hold fewer tokens than the sum of all credited balances, creating a protocol-wide insolvency gap exploitable by the inflated-balance subaccount withdrawing more than it deposited.

**Withdrawal DoS**: The same pattern in `Clearinghouse.handleWithdrawTransfer` means withdrawals of FoT tokens also revert, permanently locking any FoT collateral already held by the Clearinghouse.

---

### Likelihood Explanation

Medium. The protocol does not restrict which ERC20 tokens can be registered as spot products. Fee-on-transfer tokens (e.g., tokens with deflationary mechanics, certain rebasing tokens, or tokens with configurable transfer fees) are a known token class. Any operator or governance action that lists such a token as a spot product immediately exposes all depositors to the DoS. The entry path is fully unprivileged — any user calling `depositCollateral` triggers it.

---

### Recommendation

1. **Measure-after-transfer pattern**: In `handleDepositTransfer`, record the Clearinghouse's token balance before and after the transfer chain, and use the delta as the credited amount:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), token.balanceOf(address(this)));
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // pass actualReceived (not amount) to the slow-mode DepositCollateral tx
}
```

2. **Single-hop transfer**: Eliminate the intermediate Endpoint hop by transferring directly from the user to the Clearinghouse in one call, removing one fee deduction.

3. **Explicit FoT token blocklist**: Document and enforce that fee-on-transfer and rebasing tokens are not supported as spot products, consistent with the Timeswap judge's recommendation.

---

### Proof of Concept

**Trigger path**:

1. A fee-on-transfer token (2% fee per transfer) is registered as `productId = 5` in `SpotEngine`.
2. User calls `Endpoint.depositCollateral("default", 5, 1000e18)`.
3. `depositCollateralWithReferral` calls `handleDepositTransfer(token, user, 1000e18)`.
4. `safeTransferFrom(token, user, Endpoint, 1000e18)` → Endpoint receives `980e18` (2% fee deducted).
5. `safeTransferTo(token, Clearinghouse, 1000e18)` → Endpoint only holds `980e18`; transfer of `1000e18` reverts.
6. Entire deposit transaction reverts. User cannot deposit.

**Balance inflation path** (if Endpoint has pre-existing balance ≥ 20e18):

5. `safeTransferTo(token, Clearinghouse, 1000e18)` succeeds (drains pre-existing balance); Clearinghouse receives `980e18` (another 2% fee).
6. Slow-mode `DepositCollateral` tx is queued with `amount = 1000e18`.
7. Sequencer processes it; `Clearinghouse.depositCollateral` credits `1000e18 * multiplier` to subaccount.
8. Subaccount holds `1000e18` in book value but only `980e18` of real tokens backed it — `20e18` gap is socialized across the protocol.
