### Title
Validator-Signed `AccountData.proxies` Accepts Arbitrary Private/Loopback IP Addresses, Causing Network-Wide SSRF-Like TCP Connection Attempts — (File: `chain/network/src/peer_manager/network_state/tier1.rs`)

---

### Summary

The nearcore TIER1 network uses `AccountData` — signed by validators and gossiped across the entire TIER2 network — to advertise proxy `SocketAddr` values that all TIER1 nodes use to establish direct connections. The local configuration path validates that `public_addrs` does not contain private or loopback IPs. However, no equivalent validation is applied to the `proxies` field of **received** `AccountData` before those addresses are used to open TCP connections. A validator can sign and broadcast `AccountData` with arbitrary private, loopback, or link-local IP addresses, causing every TIER1 node in the network to attempt TCP connections to attacker-specified internal endpoints.

---

### Finding Description

**Local config validates IPs; received network data does not.**

In `NetworkConfig::new()`, before a node starts, the `public_addrs` list is validated: [1](#0-0) 

This correctly rejects loopback, unspecified, and private IPv4 addresses. However, this check applies only to the **locally configured** `public_addrs`. It has no effect on `AccountData.proxies` received from the network.

**The `AccountData` structure** carries a `proxies: Vec<PeerAddr>` field, where each `PeerAddr` is a `(peer_id: PeerId, addr: SocketAddr)` pair: [2](#0-1) 

Validators sign `VersionedAccountData` and broadcast it via `SyncAccountsData` messages over the TIER2 gossip network. Every node receives and re-broadcasts this data.

**The `verify()` function in `accounts_data/mod.rs`** checks only three things: serialized size, duplicate `account_key` entries, and cryptographic signature validity. It performs **no validation of the IP addresses** inside `proxies`: [3](#0-2) 

**The `tier1_connect()` function** then iterates over all received `AccountData`, extracts proxy addresses, and calls `transport.connect_to_peer()` directly with the attacker-controlled `proxy.addr`: [4](#0-3) 

There is no IP-range check between receiving the `AccountData` and opening the TCP connection.

**Additionally**, the local config check itself has a known gap for IPv6 — private IPv6 addresses (`is_unique_local()`) are explicitly skipped with a `TODO`: [5](#0-4) 

This means even the local-config path does not guard against IPv6 private addresses, and the received-data path has no guard at all.

---

### Impact Explanation

A validator (any participant with sufficient stake — a permissionless role) can:

1. Craft a `VersionedAccountData` with `proxies` containing arbitrary `SocketAddr` values such as `127.0.0.1:8080`, `10.0.0.1:22`, `192.168.1.1:80`, or IPv6 link-local addresses.
2. Sign it with their valid validator key.
3. Inject it into the network via a `SyncAccountsData` message to any peer.

The signed data passes signature verification and is re-gossiped to all nodes. Every TIER1 node that runs `tier1_connect()` will attempt a TCP connection to the attacker-specified address **on its own host or LAN**. This enables:

- **Network-wide SSRF**: All TIER1 validators simultaneously probe attacker-specified internal endpoints (metadata services, admin APIs, local daemons).
- **Internal service enumeration**: By rotating proxy addresses across versions, an attacker can map the internal network topology of every TIER1 validator operator.
- **Loopback service exploitation**: Connections to `127.0.0.1` on common ports (e.g., 8545 for local RPC, 9090 for Prometheus) may trigger unintended interactions with local services that assume loopback traffic is trusted.

**Scope**: High. The impact affects every TIER1 node in the network simultaneously, triggered by a single signed message from one validator.

---

### Likelihood Explanation

Becoming a validator in NEAR is permissionless — any account with sufficient staked NEAR qualifies. The attack requires no special on-chain privilege beyond validator status. The malicious `AccountData` only needs to pass signature verification (trivially satisfied by the attacker's own key) and the size check. No other gate exists on the proxy IP addresses in the received-data path.

---

### Recommendation

Apply the same IP-range validation that exists in `NetworkConfig::new()` to the `proxies` field of received `AccountData` before those addresses are used to establish connections. Concretely, in `accounts_data/mod.rs` `verify()` (or in `tier1_connect()` before calling `connect_to_peer`), reject any `PeerAddr` whose `addr.ip()` is loopback, unspecified, or private (IPv4 and IPv6). The existing validation logic in `config.rs` lines 325–334 can be extracted into a shared predicate and reused at both sites. Additionally, the IPv6 `is_unique_local()` gap in the local-config path should be resolved once the API stabilizes.

---

### Proof of Concept

1. Obtain a validator key that is in the current epoch's `accounts_data.keys` set.
2. Construct a `VersionedAccountData`:
   ```rust
   VersionedAccountData {
       data: AccountData {
           peer_id: <attacker_peer_id>,
           proxies: vec![PeerAddr {
               peer_id: <attacker_peer_id>,
               addr: "127.0.0.1:8545".parse().unwrap(), // local ETH RPC or any local service
           }],
       },
       account_key: <validator_public_key>,
       version: 9999,
       timestamp: <now>,
   }
   ```
3. Sign it with the validator's secret key (bypassing `NetworkConfig::new()` entirely — the config check only runs at startup for the local node's own config).
4. Send a `PeerMessage::SyncAccountsData` containing this signed entry to any connected peer.
5. The data passes `verify()` (valid signature, within size limit, single entry per account key) and is stored and re-gossiped.
6. On the next `tier1_connect()` tick on every TIER1 node, `transport.connect_to_peer(clock, PeerInfo { addr: Some("127.0.0.1:8545"), ... }, tcp::Tier::T1)` is called — opening a TCP connection to the local loopback on each validator's machine. [6](#0-5) [7](#0-6)

    async fn verify(
        &self,
        data: Vec<Arc<SignedAccountData>>,
    ) -> (Vec<Arc<SignedAccountData>>, Option<AccountDataError>) {
        // Filter out non-interesting data, so that we never check signatures for valid non-interesting data.
        // Bad peers may force us to check signatures for fake data anyway, but we will ban them after first invalid signature.
        let mut new_data = HashMap::new();
        let inner = self.0.load();
        for d in data {
            // There is a limit on the amount of RAM occupied by per-account datasets.
            // Broadcasting larger datasets is considered malicious behavior.
            if d.payload().len() > network_protocol::MAX_ACCOUNT_DATA_SIZE_BYTES {
                return (vec![], Some(AccountDataError::DataTooLarge));
            }
            // We want the communication needed for broadcasting per-account data to be minimal.
            // Therefore broadcasting multiple datasets per account is considered malicious
            // behavior, since all but one are obviously outdated.
            if new_data.contains_key(&d.account_key) {
                return (vec![], Some(AccountDataError::SingleAccountMultipleData));
            }
            // It is fine to broadcast data we already know about.
            // It is fine to broadcast account data that we don't care about.
            if inner.is_new(&d) {
                new_data.insert(d.account_key.clone(), d);
            }
        }

        // Verify the signatures in parallel.
        // Verification will stop at the first encountered error.
        let (data, ok) = concurrency::rayon::run(move || {
            concurrency::rayon::try_map(new_data.into_values().par_bridge(), |d| {
                match d.payload().verify(&d.account_key) {
                    Ok(()) => Some(d),
                    Err(()) => None,
                }
            })
        })
        .await;
        if !ok {
            return (data, Some(AccountDataError::InvalidSignature));
        }
        (data, None)
    }
