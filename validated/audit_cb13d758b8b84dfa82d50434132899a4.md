Audit Report

## Title
Pool Admin Transfer Does Not Update Fee Destination, Enabling Stale Routing of Accumulated Admin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`acceptPoolAdmin` updates only `poolAdmin[pool]` but leaves `poolAdminFeeDestination[pool]` pointing to the previous admin's address. Because `collectPoolFees` carries no access-control modifier, the displaced admin — or any third party — can call it immediately after the transfer and route all accumulated admin fees to the stale destination before the new admin can invoke `setPoolAdminFeeDestination`.

## Finding Description

**Partial state update in `acceptPoolAdmin` (L518–526):**

`acceptPoolAdmin` writes `poolAdmin[pool] = pending` and clears `pendingPoolAdmin[pool]`, but never touches `poolAdminFeeDestination[pool]`. [1](#0-0) 

`poolAdminFeeDestination[pool]` is written once at pool creation (L220) and is only mutable via `setPoolAdminFeeDestination`, which requires `onlyPoolAdmin`. [2](#0-1) 

**Permissionless `collectPoolFees` (L379–389):**

The function carries only `nonReentrant`; no role check exists. The interface explicitly documents it as "Callable by any address (keepers, admins, or bots)." [3](#0-2) [4](#0-3) 

**No atomic remedy:**

`setPoolAdminFeeDestination` requires `onlyPoolAdmin(pool)`, so the new admin must issue a separate transaction after `acceptPoolAdmin`. There is no path to atomically accept the role and update the destination. [5](#0-4) 

**Exploit path:**

1. Pool accumulates admin spread/notional fees over time.
2. Old admin calls `proposePoolAdminTransfer(pool, newAdmin)`.
3. New admin calls `acceptPoolAdmin(pool)` — `poolAdmin[pool]` becomes `newAdmin`; `poolAdminFeeDestination[pool]` still equals old admin's address.
4. Old admin (now unprivileged) or any MEV bot observes `PoolAdminTransferred` and calls `collectPoolFees(pool)` in the same or next block.
5. All accumulated admin fees are transferred to `poolAdminFeeDestination[pool]` — the old admin's address — with no revert.
6. New admin calls `setPoolAdminFeeDestination` — succeeds, but all pre-transfer fees are already gone.

## Impact Explanation

Direct, irreversible loss of owed admin-fee principal for the new pool admin. Every admin fee token accrued before the transfer is at risk. The wrong value is `poolAdminFeeDestination[pool]` remaining stale, causing `collectFees` to credit the old admin's address instead of the new admin's. This matches the allowed impact gate: direct loss of owed protocol/admin fees above Sherlock thresholds.

## Likelihood Explanation

The window opens on every pool admin transfer, which is a normal protocol operation. The old admin has a direct financial incentive to front-run `setPoolAdminFeeDestination`. Because `collectPoolFees` is permissionless, even a neutral MEV searcher can execute the drain without any special role. No special setup is required beyond a pending admin transfer.

## Recommendation

In `acceptPoolAdmin`, atomically settle fees to the old destination before handing over the role, then reset `poolAdminFeeDestination[pool]` to `address(0)` (or to a new-admin-supplied address):

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);

    // Settle all accrued fees to the current (old) destination before transfer
    _collectPoolFeesInternal(pool);

    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    // Reset destination so new admin must explicitly set it
    delete poolAdminFeeDestination[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

Alternatively, add an access-control modifier to `collectPoolFees` restricting callers to the current pool admin or factory owner, eliminating the permissionless drain vector.

## Proof of Concept

Foundry integration test outline:

```solidity
function test_adminTransfer_feeDestinationStale() public {
    address pool = _createPool(); // adminFeeDestination = oldDest
    // Accumulate fees via swaps
    _doSwaps(pool);

    // Propose and accept admin transfer
    vm.prank(admin);
    factory.proposePoolAdminTransfer(pool, newAdmin);
    vm.prank(newAdmin);
    factory.acceptPoolAdmin(pool);

    // Old admin (now unprivileged) drains fees
    uint256 oldDestBefore = token0.balanceOf(oldDest);
    factory.collectPoolFees(pool); // no access control — succeeds
    uint256 oldDestAfter = token0.balanceOf(oldDest);

    assertGt(oldDestAfter, oldDestBefore); // old dest received fees
    // New admin's subsequent setPoolAdminFeeDestination cannot recover drained fees
}
```
