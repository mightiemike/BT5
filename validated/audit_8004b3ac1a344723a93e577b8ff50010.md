Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the immediate pool caller (router) instead of the end user, allowing full allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the address that called `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension evaluates the router's allowlist entry rather than the actual end user's. If the pool admin allowlists the router — the expected operational step to let allowlisted users trade through the standard periphery — every unprivileged address can bypass the allowlist by calling the router.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (namespace key) and `sender` is whoever called `pool.swap()`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making `msg.sender` to the pool the router contract address, not the end user: [3](#0-2) 

The same applies to `exactOutputSingle`, `exactInput`, and the recursive `_exactOutputIterateCallback` path where intermediate hops also call `pool.swap` with `msg.sender = router`: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]` — the router's allowlist entry — rather than the actual end user's entry. There is no mechanism in the extension or the pool to recover the original `msg.sender` of the router call. The `isAllowedToSwap` view function gives a false sense of security, returning `true` only for explicitly listed addresses while the bypass is fully operational: [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` argument (the economic beneficiary of the position), which the pool passes through correctly regardless of who `msg.sender` is, so it does not share this flaw: [6](#0-5) 

## Impact Explanation

A curated pool with `SwapAllowlistExtension` is designed to restrict trading to specific counterparties (e.g., KYC-verified addresses, institutional traders, or protocol-controlled addresses). Once the pool admin allowlists the router to let those users trade through the standard periphery, the allowlist is fully bypassed for every address. Unauthorized users can trade on a pool whose LP positions were sized and priced assuming only specific, trusted counterparties, causing direct LP principal loss through adverse selection or arbitrage that the allowlist was intended to prevent. This meets the "Admin-boundary break" and "direct loss of LP assets" impact criteria.

## Likelihood Explanation

Likelihood is high. `MetricOmmSimpleRouter` is the primary supported swap interface; pool admins are expected to allowlist it so that their allowlisted users can trade through the standard UX. The bypass requires no special privileges, no flash loans, and no multi-step setup — any address calls `exactInputSingle` on the router pointing at the curated pool. The pool admin has no on-chain signal that allowlisting the router opens the gate to all users.

## Recommendation

The extension must resolve the actual end user, not the immediate pool caller. Two sound approaches:

1. **Pass the original `msg.sender` through the router.** Add a `swapper` field to the router's swap parameters and forward it inside `extensionData`; the extension decodes it and verifies it against the allowlist. The pool's reentrancy guard and callback verification already authenticate the router, so the decoded address can be trusted within that call.

2. **Require the router to forward the real user via `extensionData`.** The extension receives both `sender` (router) and the decoded real user from `extensionData`, then checks the real user against the allowlist. This keeps the extension stateless with respect to router identity.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin allowlists Alice (0xAlice) and the router (0xRouter):
      setAllowedToSwap(pool, 0xAlice, true)
      setAllowedToSwap(pool, 0xRouter, true)   ← required for Alice to use the router
  - Eve (0xEve) is NOT allowlisted.

Attack:
  1. Eve calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(recipient, ...) — msg.sender to pool = 0xRouter.
  3. Pool calls _beforeSwap(sender=0xRouter, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][0xRouter] → true.
  5. Swap executes. Eve has bypassed the allowlist.

Direct call (control):
  1. Eve calls pool.swap(...) directly.
  2. Pool calls _beforeSwap(sender=0xEve, ...).
  3. SwapAllowlistExtension checks allowedSwapper[pool][0xEve] → false → revert NotAllowedToSwap.
```
