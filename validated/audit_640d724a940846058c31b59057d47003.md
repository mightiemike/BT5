### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still exported as a public NEAR contract method despite being superseded by `verify_transaction_inclusion_v2`. The v2 path adds a mandatory coinbase Merkle proof check to prevent the 64-byte transaction forgery attack. Because v1 remains directly callable by any unprivileged NEAR account, an attacker can bypass that check entirely and obtain a `true` proof-verification result for a transaction that was never included in a Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It enforces a coinbase proof check before delegating to v1:

```
verify_transaction_inclusion_v2  ──(coinbase check)──►  verify_transaction_inclusion (v1)
``` [1](#0-0) 

The coinbase check at lines 358–365 is the only guard against the forgery: [2](#0-1) 

However, v1 is still declared `pub` and carries only a Rust `#[deprecated]` attribute: [3](#0-2) 

`#[deprecated]` is a **compile-time Rust lint** that emits warnings to Rust callers. It has no effect on the NEAR runtime ABI. The method remains a fully reachable on-chain entry point callable by any NEAR account or cross-contract call. The `#[pause]` macro is the only runtime gate, and it does not restrict who may call the function — only whether the contract is paused. [4](#0-3) 

The v1 function computes the Merkle root from the supplied `tx_id` and proof and compares it to `header.block_header.merkle_root` with no coinbase anchor: [5](#0-4) 

---

### Impact Explanation

The 64-byte forgery attack works as follows: Bitcoin's Merkle tree does not distinguish between leaf nodes (transactions) and internal nodes. A crafted 64-byte blob whose double-SHA256 hash equals an internal Merkle node can be presented as a `tx_id`. A valid Merkle proof path can then be constructed that makes `compute_root_from_merkle_proof(fake_tx_id, index, proof) == merkle_root` evaluate to `true`. Without the coinbase anchor, v1 cannot detect this forgery.

Any downstream NEAR contract that calls `verify_transaction_inclusion` directly — or any integrator that has not yet migrated to v2 — will receive `true` for a transaction that does not exist in the Bitcoin block. This corrupts the core security guarantee of the light client: the proof-verification result is forged, and any protocol action gated on that result (e.g., cross-chain asset release, settlement) can be triggered by an attacker with no real Bitcoin transaction.

---

### Likelihood Explanation

- The entry point is reachable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond the `#[pause]` gate.
- The 64-byte forgery is a well-documented, publicly known Bitcoin Merkle tree weakness with published construction techniques.
- The function is still present in the deployed ABI; any integrator who has not explicitly switched to v2 is exposed.
- The `#[deprecated]` attribute provides no on-chain protection.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public NEAR ABI. The function should be made `pub(crate)` or `private` so it is only reachable internally from `verify_transaction_inclusion_v2`. All external callers must go through the v2 path, which enforces the coinbase proof check. [3](#0-2) 

---

### Proof of Concept

1. Identify a Bitcoin block `B` whose Merkle root is `R` and which has at least two transactions.
2. Construct a 64-byte blob `fake_tx` such that `double_sha256(fake_tx) == internal_node_hash` for some internal node in `B`'s Merkle tree.
3. Build a Merkle proof `proof` and index `idx` such that `compute_root_from_merkle_proof(double_sha256(fake_tx), idx, proof) == R`.
4. Call `verify_transaction_inclusion` directly on the NEAR contract with `tx_id = double_sha256(fake_tx)`, `tx_block_blockhash = hash(B)`, `tx_index = idx`, `merkle_proof = proof`, `confirmations = 1`.
5. The function returns `true` — confirming inclusion of a transaction that was never broadcast or mined.

Step 4 is blocked by `verify_transaction_inclusion_v2` (which would reject the call at the coinbase proof check) but is fully open through the direct v1 entry point. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
