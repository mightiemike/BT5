### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` validates the LP position `owner` address against the allowlist instead of the actual transaction initiator (`sender`). Because `addLiquidity` accepts a caller-controlled `owner` parameter that is distinct from `msg.sender`, any address can bypass the deposit allowlist by nominating an already-allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts two distinct address parameters:

- `msg.sender` — the actual caller who provides tokens via the swap callback and is forwarded to extensions as `sender`
- `owner` — the LP position beneficiary, freely chosen by the caller [1](#0-0) 

The pool forwards both to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but silently discards it (unnamed `address`). The allowlist check is performed only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Compare with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual swapper): [4](#0-3) 

The asymmetry is the root cause. Because `owner` is a free parameter supplied by the caller, any address can pass the guard by specifying any allowlisted address as `owner`.

---

### Impact Explanation

An unauthorized depositor calls `pool.addLiquidity(allowlistedAddress, salt, deltas, ...)`. The extension checks `allowedDepositor[pool][allowlistedAddress]`, which passes. The unauthorized caller provides tokens via the callback; LP shares are credited to `allowlistedAddress`. The pool admin's access control is fully bypassed:

1. **Allowlist invariant broken**: The pool accepts liquidity from addresses the admin explicitly excluded.
2. **Pool state manipulation**: Bin balances, `curPosInBin`, and `binTotals` are modified by an unauthorized party, potentially affecting oracle-derived prices, stop-loss watermarks, and LP claims for existing LPs.
3. **Griefing / forced liquidity**: The unauthorized depositor can force the pool to hold liquidity it was not configured to accept, disrupting pool economics.

---

### Likelihood Explanation

The `addLiquidity` interface is public and `owner` is a free parameter — no special role or privileged access is required. Any address that can call the pool can exploit this. The only cost to the attacker is the tokens deposited (which become LP shares owned by the nominated `owner`). Pools that deploy `DepositAllowlistExtension` for KYC/compliance or liquidity-concentration purposes are directly affected.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; `allowedDepositor[pool][alice] = true`; `allowedDepositor[pool][attacker] = false`.
2. `attacker` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `extension.beforeAddLiquidity(attacker, alice, salt, deltas, extensionData)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Pool proceeds; `attacker` provides tokens via callback; LP shares are minted to `alice`.
6. `attacker` has successfully deposited into a pool that was supposed to block them, modifying bin balances and pool state without authorization. [3](#0-2) [5](#0-4)
