### Title
64-Byte Internal Merkle Node Forgery via Deprecated `verify_transaction_inclusion` - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is a publicly callable NEAR contract method with no caller restriction beyond the `#[pause]` gate. It accepts an arbitrary `tx_id` and passes it directly to `compute_root_from_merkle_proof` without any check that the supplied hash corresponds to a leaf-level transaction rather than an internal Merkle tree node. An attacker can supply a 64-byte internal node hash as `tx_id` with a shortened proof path, causing the function to reconstruct the correct `merkle_root` and return `true` for a transaction that was never included in the block.

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` and `#[pause]` but remains `pub` with no `#[private]` or `#[trusted_relayer]` restriction. [1](#0-0) 

The function body performs exactly one check: does `compute_root_from_merkle_proof(args.tx_id, tx_index, &args.merkle_proof)` equal the stored `merkle_root`? [2](#0-1) 

`compute_root_from_merkle_proof` is a pure hash-chain computation with no leaf/node distinction: [3](#0-2) 

`compute_hash` concatenates two 32-byte values and double-SHA256s them — identical to how Bitcoin constructs internal nodes: [4](#0-3) 

The contract itself documents the exact weakness in the `# Warning` comment: [5](#0-4) 

The v2 function was introduced specifically to close this gap by requiring a coinbase proof at index 0 (depth = full tree depth), which forces the proof length to match the tree depth and prevents an internal node from being presented as a leaf: [6](#0-5) 

However, v1 remains callable and unguarded.

### Impact Explanation

Any downstream contract or off-chain system that calls `verify_transaction_inclusion` (v1) to gate asset release (e.g., a bridge unlocking tokens upon BTC deposit confirmation) can be deceived into releasing assets for a Bitcoin transaction that was never broadcast or confirmed. The attacker needs only a valid block hash already stored in the contract's mainchain and knowledge of that block's Merkle tree structure — both are public on-chain data.

### Likelihood Explanation

- The method is `pub` with no access-control restriction beyond `#[pause]` (not paused by default).
- The required inputs (block hash, internal node hash, shortened proof) are all derivable from public Bitcoin blockchain data.
- The `#[deprecated]` Rust attribute is a compiler hint only; it does not prevent NEAR RPC callers from invoking the method.
- The relayer client code also retains a `verify_transaction_inclusion` wrapper marked `#[allow(dead_code)]`, indicating the v1 path is still wired up. [7](#0-6) 

### Recommendation

1. **Remove or gate v1**: Add `#[private]` to `verify_transaction_inclusion` so it can only be called as an internal cross-contract call, or remove the public export entirely. The v2 path already supersedes it.
2. **Enforce proof-length equality**: In v1 (if kept), require `merkle_proof.len()` to equal `log2(block_tx_count).ceil()`, matching the full tree depth, so a shortened internal-node proof is rejected.
3. **Deprecation is not removal**: The `#[deprecated]` attribute only produces a Rust compiler warning for Rust callers; it has no effect on NEAR RPC accessibility.

### Proof of Concept

Given a real Bitcoin block with transactions `[T0, T1, T2, T3]` stored in the contract:

```
Level 2 (root):  R  = H(N01, N23)
Level 1:         N01 = H(T0, T1),   N23 = H(T2, T3)
Level 0 (leaf):  T0, T1, T2, T3
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id`      = `N01`  (internal node hash — not a real txid)
- `tx_index`   = `0`    (left child at level 1)
- `merkle_proof` = `[N23]`  (one element, not two)
- `tx_block_blockhash` = the block's hash (already in contract state)
- `confirmations` = `0`

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `H(N01, N23) = R`, which equals `header.block_header.merkle_root`. The contract returns `true` despite `N01` never being a real transaction. [8](#0-7)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
```
