### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Read-Only Reentrancy via Token Transfer Callback — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the full token transfer to the recipient **before** decrementing the subaccount balance in `SpotEngine`. If a supported collateral token implements a transfer callback (e.g., ERC777 `tokensReceived`), the recipient executes arbitrary code while `SpotEngine` still reflects the pre-withdrawal (inflated) balance. During this window, `getBalance()`, `getStateAndBalance()`, and `getHealth()` all return stale state — a direct analog to the Balancer read-only reentrancy pattern.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the external token transfer is dispatched at line 408 via `handleWithdrawTransfer`, but `spotEngine.updateBalance` is not called until line 412:

```
408:  handleWithdrawTransfer(token, sendTo, amount, idx);   // external call
...
412:  spotEngine.updateBalance(productId, sender, amountRealized);  // state update
```

`handleWithdrawTransfer` (Clearinghouse.sol:377–385) first calls `token.safeTransfer(withdrawPool, amount)`, then calls `BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx)`. Inside `submitWithdrawal` (BaseWithdrawPool.sol:116–132), `handleWithdrawTransfer` (BaseWithdrawPool.sol:184–190) is called, which executes `token.safeTransfer(to, amount)` — transferring tokens directly to the user.

If the collateral token is ERC777 or has any `tokensReceived`/transfer hook, the recipient's hook fires during `token.safeTransfer(to, amount)`. At this exact point in the call stack:

- `Clearinghouse.withdrawCollateral` is mid-execution
- `spotEngine.updateBalance` (line 412) has **not** been called
- The subaccount's balance in `SpotEngineState` still reflects the full pre-withdrawal amount

Any call to `SpotEngineState.getBalance()` (line 246), `SpotEngineState.getStateAndBalance()` (line 236), or `Clearinghouse.getHealth()` (line 71) during this window returns the inflated, pre-withdrawal state.

---

### Impact Explanation

External contracts or protocols that integrate with Nado and read subaccount state during the token transfer callback observe an inflated balance and an artificially healthy subaccount. Concretely:

- `getBalance(productId, sender)` returns the full pre-withdrawal amount instead of the post-withdrawal amount.
- `getHealth(sender, healthType)` returns a health value computed from the inflated balance.

An attacker who controls a recipient contract with a `tokensReceived` hook can exploit this window to present a falsely healthy subaccount to any external protocol (e.g., a lending protocol, a vault, or a cross-margin integrator) that trusts Nado's state-reading functions. This is the exact analog to the Balancer read-only reentrancy: state-reading functions are not protected against being called during the reentrancy window, and they return incorrect state that downstream consumers can act on.

Within the Nado protocol itself, the health check at line 419 executes after the balance update at line 412, so the internal invariant is preserved. The risk is scoped to **external integrators** reading Nado state during the callback window.

---

### Likelihood Explanation

Medium. The trigger requires a collateral token with a transfer callback (ERC777 or equivalent). The protocol's `IERC20Base` interface is generic and does not restrict token types. If any listed collateral token supports hooks, the window is reachable by any user who submits a withdrawal (via slow mode or sequencer-processed path) with a recipient contract that implements the hook. No privileged access is required.

---

### Recommendation

Apply the Checks-Effects-Interactions pattern in `withdrawCollateral`: move `spotEngine.updateBalance` and `spotEngine.assertUtilization` **before** `handleWithdrawTransfer`. This ensures the SpotEngine state is consistent before any external call is made:

```solidity
// 1. Update state first
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 2. Then perform external transfer
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Alternatively, add a reentrancy guard (`nonReentrant`) to `withdrawCollateral` and any other function that performs external token transfers before state updates.

---

### Proof of Concept

**Call chain establishing the reentrancy window:**

1. Attacker holds collateral in an ERC777 token and submits a `WithdrawCollateral` slow-mode transaction with `sendTo` set to an attacker-controlled contract.
2. Sequencer processes the transaction → `Endpoint` → `Clearinghouse.withdrawCollateral` (line 391).
3. Line 408: `handleWithdrawTransfer(token, sendTo, amount, idx)` is called.
4. Clearinghouse's `handleWithdrawTransfer` (line 383): `token.safeTransfer(withdrawPool, amount)` — tokens move to `WithdrawPool`.
5. Line 384: `BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx)`.
6. `submitWithdrawal` (BaseWithdrawPool.sol:131): `handleWithdrawTransfer(token, to, amount)`.
7. BaseWithdrawPool's `handleWithdrawTransfer` (line 189): `token.safeTransfer(to, amount)` — ERC777 fires `tokensReceived` on the attacker's contract.
8. **Inside the hook** — `spotEngine.updateBalance` (Clearinghouse.sol:412) has not executed yet.
9. Attacker calls `spotEngine.getBalance(productId, attackerSubaccount)` → returns pre-withdrawal (inflated) balance.
10. Attacker calls `clearinghouse.getHealth(attackerSubaccount, INITIAL)` → returns inflated health.
11. Attacker presents this stale state to an external protocol to obtain credit or bypass a health gate.

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
