### Title
Missing Parent Subaccount Health Check After Margin Transfer in `createIsolatedSubaccount()` - (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.createIsolatedSubaccount()` transfers quote margin from a parent subaccount to a newly created isolated subaccount without performing any health check on the parent afterward. This is the direct analog of the reported vulnerability class: a safety check that must fire when collateral is removed from an account is absent in exactly that scenario, while the analogous `transferQuote()` path in `Clearinghouse.sol` correctly enforces the check.

---

### Finding Description

In `OffchainExchange.sol`, `createIsolatedSubaccount()` encodes the margin amount from the user-controlled `appendix` field of the signed order:

```solidity
function _isolatedMargin(uint128 appendix) internal pure returns (uint128) {
    return (appendix >> 64) * (10**12);
}
``` [1](#0-0) 

When `margin > 0`, the function deducts that amount from the parent's quote balance and credits it to the isolated subaccount:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [2](#0-1) 

After this balance mutation, **no health check is performed on `txn.order.sender`**. The function simply returns the new isolated subaccount address.

Contrast this with `Clearinghouse.transferQuote()`, which performs an identical quote balance reduction and then immediately enforces the initial health invariant:

```solidity
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -toTransfer);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.recipient, toTransfer);
require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH);
``` [3](#0-2) 

The `_isAboveInitial` helper used there checks `INITIAL` health: [4](#0-3) 

The same check is absent in `createIsolatedSubaccount()`.

---

### Impact Explanation

A user who signs an order with a large margin value encoded in the `appendix` can cause the sequencer to call `createIsolatedSubaccount()`, which will drain the parent subaccount's quote balance below initial health — or even below maintenance health — with no on-chain rejection. The parent subaccount is then immediately eligible for liquidation, and the margin locked in the isolated subaccount is inaccessible until the isolated position closes. This constitutes a solvency/accounting corruption: the parent's collateral is reduced without the protocol enforcing that the remaining collateral is sufficient to cover its open positions. The isolated subaccount's margin is also at risk if the parent is subsequently socialized.

---

### Likelihood Explanation

The margin amount is entirely user-controlled via the `appendix` field of a signed `IEndpoint.Order`. The user signs the order off-chain and submits it to the sequencer. The sequencer calls `createIsolatedSubaccount()` on-chain. Because the on-chain contract performs no health check, any user who holds open positions (giving them positive initial health) can set a margin value that exceeds their free collateral. The sequencer may apply off-chain pre-checks, but the on-chain contract is the authoritative enforcement layer and it is missing the guard entirely. The entry path is a standard user-facing isolated margin order, making this reachable by any unprivileged trader.

---

### Recommendation

After the margin transfer in `createIsolatedSubaccount()`, add an initial health check on the parent subaccount, mirroring the pattern used in `Clearinghouse.transferQuote()`:

```solidity
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    // ADD: enforce parent health after margin transfer
    require(
        IClearinghouse(clearinghouse).getHealth(
            txn.order.sender,
            IProductEngine.HealthType.INITIAL
        ) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

---

### Proof of Concept

1. Alice holds 1000 USDC quote balance and an open perp position worth 900 USDC in initial health contribution. Her initial health is +100.
2. Alice signs an isolated order with `appendix` encoding `margin = 200 * 10^12` (200 USDC).
3. The sequencer calls `createIsolatedSubaccount()`. The contract deducts 200 USDC from Alice's parent quote balance and credits the isolated subaccount.
4. Alice's parent quote balance is now −200 USDC (net of the perp position her initial health is −100). No on-chain check fires.
5. Alice's parent subaccount is now below initial health and eligible for liquidation. A liquidator calls `liquidateSubaccount()`, which passes `isUnderMaintenance()` if health is also below maintenance.
6. The 200 USDC in the isolated subaccount is locked until the isolated position closes, while the parent account is being liquidated — resulting in a net loss of funds for Alice and potential bad debt for the protocol if insurance is insufficient. [2](#0-1) [3](#0-2)
