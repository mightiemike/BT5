Audit Report

## Title
Router Identity Substitution Bypasses Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension::beforeSwap` checks the `sender` argument against a per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router`, so the hook checks the router's address rather than the original user's address. Allowlisting the router — required for any router-mediated swap to succeed — grants every user bypass access to the per-user allowlist, breaking the extension's core access-control invariant.

## Finding Description
**Step 1 — Router calls pool directly:**
`MetricOmmSimpleRouter::exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` at line 72–80 of `metric-periphery/contracts/MetricOmmSimpleRouter.sol`. The pool's `msg.sender` is the router, not the original user.

**Step 2 — Pool passes `msg.sender` as `sender` to the hook:**
`MetricOmmPool::swap` at lines 230–240 of `metric-core/contracts/MetricOmmPool.sol` calls `_beforeSwap(msg.sender, ...)`, forwarding the router address as `sender`.

**Step 3 — ExtensionCalling forwards it unchanged:**
`ExtensionCalling::_beforeSwap` at lines 149–177 of `metric-core/contracts/ExtensionCalling.sol` encodes `sender` (the router) directly into the `abi.encodeCall` for `IMetricOmmExtensions.beforeSwap`.

**Step 4 — Extension checks the wrong address:**
`SwapAllowlistExtension::beforeSwap` at line 37 of `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router. If the router is allowlisted (necessary for any router-mediated swap), the check passes for all callers regardless of their individual allowlist status.

**Existing guards are insufficient:** There is no mechanism in the pool, router, or extension to recover the original EOA/contract that initiated the transaction. The `allowedSwapper` mapping is keyed on the intermediary (router), not the end user.

## Impact Explanation
Any user can bypass a per-user swap allowlist by routing through `MetricOmmSimpleRouter`. If the allowlist is used as a loss-prevention or compliance control (e.g., restricting swaps to KYC'd addresses or specific market makers), this bypass allows unauthorized parties to trade against the pool. This breaks the core functionality of `SwapAllowlistExtension` — the extension cannot simultaneously (a) permit router-mediated swaps and (b) enforce per-user gating. This constitutes broken core pool/extension functionality under the allowed impact gate.

## Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` with per-user restrictions and also needs to support router-mediated swaps must allowlist the router, making the bypass trivially reachable by any public user in a single transaction. No special privileges or setup are required beyond the admin having allowlisted the router.

## Recommendation
Pass the original user's address through the call chain. Options include: (1) add a dedicated `originalSender` field to the hook signature populated from `tx.origin` or a trusted router-supplied parameter with pool-level authentication; (2) have the router encode the original `msg.sender` into `extensionData` and have the extension read and authenticate it; or (3) document explicitly that allowlisting the router grants universal access and that per-user gating requires direct pool interaction only, removing the false security guarantee.

## Proof of Concept
```solidity
// Setup (pool admin):
SwapAllowlistExtension.setAllowedToSwap(pool, alice, true);
SwapAllowlistExtension.setAllowedToSwap(pool, router, true); // required for router swaps

// Attack (charlie, not allowlisted):
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: charlie,
    tokenIn: token0,

    amountIn: X,
    ...
}));
