### Title
Paused Collateral Token Blocks All User Withdrawals — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `withdrawCollateral()` function in `Clearinghouse.sol` calls `token.safeTransfer()` with no try/catch. If a supported collateral token (e.g., USDC, which has a Circle-controlled pause mechanism) is paused, every withdrawal attempt for that token will revert, permanently locking user funds in the protocol until the token is unpaused.

---

### Finding Description

The withdrawal execution path is:

1. `EndpointTx.processTransactionImpl()` handles `WithdrawCollateral` / `WithdrawCollateralV2` transaction types and calls `clearinghouse.withdrawCollateral()`
2. `Clearinghouse.withdrawCollateral()` calls `handleWithdrawTransfer(token, sendTo, amount, idx)`
3. `Clearinghouse.handleWithdrawTransfer()` calls `token.safeTransfer(withdrawPool, uint256(amount))`
4. `ERC20Helper.safeTransfer()` performs a low-level `.call()` and then `require(success && ...)` — if the token is paused, `success == false` and the call reverts with `ERR_TRANSFER_FAILED` [1](#0-0) 

The `ERC20Helper.safeTransfer` wrapper unconditionally requires the transfer to succeed: [2](#0-1) 

There is no try/catch, no fallback queue, and no skip-on-failure logic anywhere in this path. The same pattern applies to `BaseWithdrawPool.handleWithdrawTransfer()` (the second hop, WithdrawPool → user) and to `withdrawInsurance()`, which also routes through `handleWithdrawTransfer`: [3](#0-2) [4](#0-3) 

The full withdrawal call in `Clearinghouse.withdrawCollateral()` — including the balance deduction — is atomic. If the transfer reverts, the balance is never decremented and the user's funds remain locked: [5](#0-4) 

---

### Impact Explanation

**High.** Any user holding a paused collateral token (e.g., USDC) cannot withdraw their funds. All `WithdrawCollateral`, `WithdrawCollateralV2`, `RebalanceXWithdraw`, and `WithdrawInsurance` calls for the affected token revert. Fast withdrawals via `BaseWithdrawPool.submitFastWithdrawal()` are equally blocked since they also call `handleWithdrawTransfer`: [6](#0-5) 

User collateral is effectively frozen in the Clearinghouse with no on-chain recourse until the token is unpaused.

---

### Likelihood Explanation

**Low.** Token pauses are rare but real: USDC (Circle), USDT (Tether), and similar regulated stablecoins all have admin-controlled pause mechanisms. A regulatory action, exploit response, or emergency freeze by the token issuer would trigger this condition without any action by Nado governance or users.

---

### Recommendation

Wrap the `token.safeTransfer()` call in `Clearinghouse.handleWithdrawTransfer()` in a try/catch (or equivalent low-level success check without revert), and queue failed withdrawals for later retry. Alternatively, implement a pull-based withdrawal pattern where the token transfer is separated from the accounting update, so a failed transfer does not block the accounting state machine.

---

### Proof of Concept

1. User deposits USDC as collateral via `Endpoint.depositCollateralWithReferral()`.
2. Circle pauses USDC (e.g., in response to a regulatory order).
3. User submits a `WithdrawCollateral` signed transaction through `Endpoint`.
4. Sequencer calls `processTransactionImpl()` → `clearinghouse.withdrawCollateral()` → `handleWithdrawTransfer()` → `token.safeTransfer(withdrawPool, amount)`.
5. `ERC20Helper.safeTransfer` executes `address(token).call(transfer(...))` — the paused USDC contract reverts, `success == false`, and `require(success && ...)` throws `ERR_TRANSFER_FAILED`.
6. The entire `withdrawCollateral` call reverts. The user's balance in `SpotEngine` is unchanged and their USDC remains locked in the Clearinghouse.
7. Every subsequent withdrawal attempt for USDC by any user produces the same revert until Circle unpauses the token. [1](#0-0) [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
