### Title
Missing Lower Bound on `confirmations` Parameter Allows Zero-Confirmation SPV Proof Acceptance - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (and its v2 variant) accepts a caller-supplied `confirmations: u64` value with only an upper-bound check (`confirmations <= gc_threshold`). There is no lower-bound check enforcing `confirmations >= 1`. Passing `confirmations = 0` causes the confirmation-depth guard to trivially pass for every block in the mainchain, returning `true` for a transaction that has zero confirmation depth and may still be reorganized away.

### Finding Description
The function `verify_transaction_inclusion` performs two sequential checks on the caller-supplied `confirmations` value:

**Check 1 — upper bound only:** [1](#0-0) 

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

**Check 2 — confirmation depth:** [2](#0-1) 

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

When `args.confirmations = 0`, Check 1 passes because `0 <= gc_threshold` is always true for any valid `gc_threshold`. Check 2 evaluates to `X >= 0` where `X` is a `u64`, which is unconditionally true in Rust. Both guards are bypassed, and the function proceeds to the Merkle proof check — which is purely about inclusion, not depth. The function returns `true` for any transaction in any mainchain block, including the current tip, regardless of how many blocks have been built on top of it.

The same flaw is present in `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` internally after its coinbase proof check: [3](#0-2) 

The `ProofArgs` struct that carries the unchecked field: [4](#0-3) 

### Impact Explanation
Any unprivileged NEAR caller can submit a `ProofArgs` or `ProofArgsV2` with `confirmations = 0`. The contract will return `true` for a transaction that is only in the current chain tip — a block that has not yet been buried under any subsequent work and is therefore fully susceptible to reorganization. Any downstream contract (e.g., a cross-chain bridge or asset-release contract) that trusts this return value as proof of finality will accept a transaction that may be rolled back, enabling double-spend or asset theft.

The corrupted proof result is concrete: `verify_transaction_inclusion` returns `true` when the correct answer for a security-sensitive caller is "insufficient confirmations."

### Likelihood Explanation
The entry point is a public, unprivileged NEAR contract method. No role, stake, or key is required. The exploit requires only constructing a `ProofArgs` with `confirmations = 0` and a valid Merkle inclusion proof for a transaction in any mainchain block. Merkle proofs for real Bitcoin transactions are publicly available from any Bitcoin full node. The attack is trivially reproducible.

### Recommendation
Add an explicit lower-bound check immediately after deserialization, mirroring the pattern used for the upper bound:

```rust
require!(args.confirmations > 0, "confirmations must be at least 1");
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

This enforces `0 < confirmations <= gc_threshold`, analogous to the AMM report's recommendation of `0 < α < 1 WAD`. Additionally, document the minimum safe confirmation depth for each supported chain (e.g., 6 for Bitcoin mainnet) and consider enforcing a protocol-level minimum rather than leaving it entirely to callers.

### Proof of Concept

1. Deploy the contract (Bitcoin feature) with any valid genesis and `gc_threshold >= 1`.
2. Submit a single block header so the mainchain tip is at height `H`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the tip block (height `H`)
   - `tx_id`, `tx_index`, `merkle_proof` = a valid Merkle inclusion proof for any transaction in that block
   - `confirmations = 0`
4. Observe: Check 1 passes (`0 <= gc_threshold`). Check 2 evaluates `(H - H) + 1 = 1 >= 0` → `true`. The function returns `true`.
5. The tip block has zero blocks built on top of it and can be reorganized. The contract has certified a zero-confirmation transaction as "verified."
