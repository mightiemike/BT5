### Title
Unsafe `approve()` in `DirectDepositV1.creditDeposit()` Breaks Deposit Flow for Non-Standard ERC20 Tokens — (`File: core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve()` directly via the `IIERC20Base` interface, which expects a `bool` return value. Non-standard ERC20 tokens (e.g., USDT) that do not return a value from `approve()` will cause the call to revert unconditionally. Any user who sends such a token to a `DirectDepositV1` address will have their funds stuck in the contract, unable to be credited as collateral, until the multisig manually rescues them.

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product tokens and, for each non-zero balance, calls `token.approve(address(endpoint), balance)` directly: [1](#0-0) 

The `IIERC20Base` interface declares `approve()` as returning `bool`: [2](#0-1) 

In Solidity `^0.8.0`, when the ABI expects a `bool` return but the called contract returns nothing (as USDT does), the ABI decoder reverts. This means `creditDeposit()` will always revert for any non-standard ERC20 token registered as a spot product.

By contrast, the same contract correctly implements a `safeTransfer` helper that tolerates missing return values: [3](#0-2) 

The `ERC20Helper` library also provides `safeTransfer` and `safeTransferFrom` with the same tolerance pattern, but provides **no `safeApprove`**: [4](#0-3) 

The same unsafe pattern appears in `ContractOwner.wrapVaultAsset()`, which calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` directly with no safe wrapper: [5](#0-4) 

### Impact Explanation

Any user who sends a non-standard ERC20 token (one that omits the `bool` return from `approve()`) to a `DirectDepositV1` address will find that:

1. `creditDeposit()` always reverts — the token balance is never forwarded to the endpoint as collateral.
2. The tokens are stuck in the `DirectDepositV1` contract until the multisig owner manually calls `ContractOwner.withdrawFromDirectDepositV1()` to rescue them. [6](#0-5) 

The deposit flow is permanently broken for the affected token type. Users lose access to their funds for an indefinite period and cannot use them as collateral.

### Likelihood Explanation

`creditDeposit()` has no access control — it is callable by any address: [7](#0-6) 

`ContractOwner.creditDepositV1()` also has no access control and routes directly to `creditDeposit()`: [8](#0-7) 

If any non-standard ERC20 token (USDT is the canonical example, and is a common collateral asset in DeFi protocols) is ever registered as a spot product, every deposit attempt via the `DirectDepositV1` path will fail. The likelihood is **medium**: it depends on whether such a token is listed, but the protocol's architecture explicitly supports multiple spot tokens and the `DirectDepositV1` path is a primary deposit mechanism.

### Recommendation

Add a `safeApprove` helper to `ERC20Helper` (or to `DirectDepositV1` itself) using the same low-level `call` pattern already used for `safeTransfer`:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace the raw `token.approve(address(endpoint), balance)` call in `DirectDepositV1.creditDeposit()` with this safe variant. Apply the same fix to `ContractOwner.wrapVaultAsset()` for both `approve` calls. Additionally, reset the allowance to zero before setting a new one (required by some non-standard tokens such as USDT).

### Proof of Concept

1. A non-standard ERC20 token (e.g., USDT-like, no `bool` return from `approve()`) is registered as a spot product in `SpotEngine`.
2. A user sends 1000 USDT to the `DirectDepositV1` address for their subaccount.
3. Anyone calls `DirectDepositV1.creditDeposit()` (or `ContractOwner.creditDepositV1(subaccount)`).
4. Inside `creditDeposit()`, `token.approve(address(endpoint), 1000)` is called. The token returns no data; Solidity's ABI decoder attempts to decode a `bool` from empty returndata and reverts.
5. The transaction reverts. The 1000 USDT remain in the `DirectDepositV1` contract.
6. The user has no way to recover their funds without the multisig calling `withdrawFromDirectDepositV1()`. [9](#0-8)

        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");

            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");

            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
