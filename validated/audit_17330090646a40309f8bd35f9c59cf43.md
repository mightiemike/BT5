### Title
`SwapAllowlistExtension` Bypass via `MetricOmmSimpleRouter`: Any User Can Swap on Allowlisted Pools — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so the extension checks whether the **router** is allowlisted — not the actual user. If the pool admin allowlists the router (a natural action to let allowlisted users access the router), any unprivileged user can bypass the allowlist entirely by calling the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (correct) and `sender` is the argument forwarded from `MetricOmmPool.swap`:

```solidity
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient,
  ...
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()`, so `msg.sender` to the pool is the **router address**:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
``` [3](#0-2) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. A pool admin who allowlists the router — a necessary step to let any allowlisted user access the router — simultaneously opens the gate to **every** user, because the router is a public, permissionless contract.

The same path exists for `exactInput`, `exactOutputSingle`, and `exactOutput`.

---

### Impact Explanation

The `SwapAllowlistExtension` is documented as "Gates `swap` by swapper address, per pool." [4](#0-3) 

Once the router is allowlisted, the gate is completely ineffective: any address can execute swaps against a pool that was intended to be restricted. Pools that rely on the allowlist for compliance, rate-limiting, or LP protection lose that guarantee entirely. This constitutes broken core pool functionality — the allowlist invariant is violated for every router-mediated swap.

---

### Likelihood Explanation

The pool admin must allowlist the router to let allowlisted users benefit from multi-hop routing, slippage protection, and deadline enforcement. There is no mechanism to say "allow the router only for specific end-users." The admin is forced to choose between blocking the router entirely (degraded UX for legitimate users) or allowlisting it (full bypass for everyone). A pool admin who chooses usability inadvertently enables the bypass. This is a realistic, semi-trusted trigger with no malicious setup required.

---

### Recommendation

The extension must verify the **economic actor**, not the immediate caller. Two approaches:

1. **Pass the original user through the router.** The router could forward the real `msg.sender` in `extensionData`, and the extension could decode and check it. This requires a coordinated change to the router and extension.

2. **Check `sender` against the allowlist; if `sender` is the router, reject or require the real user to be encoded in `extensionData`.** The extension can detect known router addresses and apply a secondary check.

3. **Adopt the same pattern as `DepositAllowlistExtension`**, which gates on `owner` (the economic beneficiary) rather than `sender` (the caller). For swaps, the analogous field would be the `recipient` or a user-supplied identity in `extensionData`.

---

### Proof of Concept

```solidity
// Pool admin sets up allowlist: only Alice is allowed to swap
swapAllowlist.setAllowedToSwap(pool, alice, true);

// Admin also allowlists the router so Alice can use it
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Mallory (not allowlisted) calls the router directly
// The pool sees msg.sender = router → extension checks allowedSwapper[pool][router] → true
// Mallory's swap succeeds despite not being on the allowlist
vm.prank(mallory);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: mallory,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// → succeeds; allowlist completely bypassed
``` [5](#0-4) [6](#0-5)
