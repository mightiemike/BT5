### Title
Deprecated `verify_transaction_inclusion()` Remains Publicly Callable and Vulnerable to 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion()` function is explicitly marked `#[deprecated]` in favour of `verify_transaction_inclusion_v2()`, which adds coinbase Merkle proof validation to close the 64-byte transaction Merkle proof forgery attack. However, the deprecated function remains a live, unpermissioned public entry point. Any NEAR caller can invoke it directly, bypassing the coinbase guard entirely, and supply an internal Merkle-tree node as `tx_id` together with a crafted proof path that hashes up to the real block Merkle root. The function returns `true`, falsely certifying that a non-existent transaction was included in a confirmed Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion()` is annotated with `#[deprecated(since = "0.5.0", note = "Use verify_transaction_inclusion_v2 instead.")]` and carries an explicit `# Warning` in its doc-comment:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

Despite this, the function is decorated only with `#[pause]` — no role restriction, no `#[private]`, no access-control gate — and is therefore callable by any NEAR account. [2](#0-1) 

The root cause is that `compute_root_from_merkle_proof()` is a pure hash-path computation: it accepts any 32-byte value as the leaf and walks the proof upward. [3](#0-2) 

If an attacker supplies an internal Merkle-tree node `N` (a 32-byte hash of two real child transactions) as `tx_id`, and supplies the sibling path that leads from `N` to the real Merkle root, `compute_root_from_merkle_proof(N, index, proof)` will equal `header.block_header.merkle_root` and the function returns `true`. [4](#0-3) 

`verify_transaction_inclusion_v2()` closes this gap by first verifying a coinbase proof at index 0, which anchors the tree depth and makes internal-node substitution impossible. It then delegates to the deprecated v1 path. [5](#0-4) 

Because v1 is still independently reachable, the v2 fix is entirely bypassable.

---

### Impact Explanation

**Impact: High.**

Any downstream NEAR contract that calls `verify_transaction_inclusion()` to gate a cross-chain action (bridge unlock, custody release, oracle update) will accept a forged SPV proof as valid. The attacker does not need to mine a block or control any privileged role; they only need a real confirmed block hash and the ability to compute an internal Merkle node from publicly available transaction data. The corrupted protocol invariant is the boolean proof result: `true` is returned for a transaction that does not exist.

---

### Likelihood Explanation

**Likelihood: Medium.**

The 64-byte Merkle proof forgery is a well-documented Bitcoin attack (https://www.bitmex.com/blog/64-Byte-Transactions). All inputs required — block hash, Merkle root, sibling hashes — are publicly available from any Bitcoin block explorer. The deprecated function is callable by any NEAR account with no deposit or role requirement. The only friction is that the attacker must identify a consumer contract that still calls the v1 API rather than v2.

---

### Recommendation

1. **Remove the public entry point.** Add `#[private]` to `verify_transaction_inclusion()` so it can only be called internally (it is already called internally by `verify_transaction_inclusion_v2()`), or remove it entirely from the public ABI.
2. **If removal is not immediately possible**, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` guard at the top of the function body to make any direct external call panic.
3. **Audit all consumer contracts** to confirm none still call the v1 API.

---

### Proof of Concept

Given a real confirmed Bitcoin block `B` with Merkle root `R` and at least two transactions `T0`, `T1`:

1. Compute `N = SHA256d(T0 || T1)` — this is the internal node one level above the leaves.
2. Obtain the sibling path from `N` up to `R` (one fewer step than a full leaf proof).
3. Call on NEAR:

```
verify_transaction_inclusion({
    tx_id:             N,          // internal node, not a real txid
    tx_block_blockhash: B,
    tx_index:          0,          // position of N in the reduced tree
    merkle_proof:      [siblings…], // path from N to R
    confirmations:     1
})
```

`compute_root_from_merkle_proof(N, 0, siblings)` returns `R`, which equals `header.block_header.merkle_root`. [6](#0-5) 

The function returns `true`, falsely certifying that the non-existent transaction `N` was confirmed in block `B`.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
