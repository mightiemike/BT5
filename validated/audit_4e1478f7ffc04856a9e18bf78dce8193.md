### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Transaction Forgery Protection Added in v2 - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` (v1) remains a fully public, unrestricted NEAR contract method. Any unprivileged caller can invoke v1 directly, completely bypassing the coinbase proof check, and obtain a forged `true` proof-verification result for a non-existent transaction.

### Finding Description
The contract explicitly documents that `verify_transaction_inclusion` is vulnerable to the 64-byte transaction Merkle proof forgery attack and that `verify_transaction_inclusion_v2` was introduced to fix it by requiring a coinbase merkle proof: [1](#0-0) 

The v2 function enforces the coinbase proof check before delegating to v1: [2](#0-1) 

However, v1 is still declared `pub fn` with only a `#[pause]` attribute and no access-control restriction: [3](#0-2) 

In Rust, `#[deprecated]` is a compile-time lint warning only — it has zero runtime enforcement. The NEAR runtime exposes every `pub fn` in a `#[near]` `impl` block as a callable contract method regardless of deprecation annotations. Therefore, any NEAR account can call `verify_transaction_inclusion` directly, supplying only `ProofArgs` (which has no `coinbase_tx_id` or `coinbase_merkle_proof` fields): [4](#0-3) 

The v1 function performs no coinbase validation whatsoever — it only checks that the computed Merkle root matches the block header's `merkle_root`: [5](#0-4) 

The `compute_root_from_merkle_proof` function in `merkle-tools` is a pure hash-chain computation with no structural validation of whether the input is a real transaction hash or an internal tree node: [6](#0-5) 

### Impact Explanation
The 64-byte transaction forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions) allows an attacker to craft a `tx_id` that is actually the concatenation of two 32-byte internal Merkle tree nodes. Because `compute_root_from_merkle_proof` treats this value identically to a real transaction hash, the attacker can produce a valid-looking Merkle proof for a transaction that does not exist in the block. Calling `verify_transaction_inclusion` directly with such a crafted `tx_id` and a shorter proof path returns `true`. Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing funds, minting tokens, recording a Bitcoin payment) will be deceived into accepting a forged Bitcoin transaction inclusion proof. The corrupted value is the boolean return of `verify_transaction_inclusion` — a `true` result for a non-existent transaction.

### Likelihood Explanation
The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` directly. Downstream contracts that integrated the light client before v2 was introduced are likely still calling v1. Even new integrators reading the ABI will see both methods exposed and may call v1 without understanding the security implication. The attack requires only knowledge of a real mainchain block's Merkle tree structure, which is public Bitcoin data.

### Recommendation
Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with an access-control role so it is no longer callable by unprivileged accounts. The preferred fix is to make it `pub(crate)` or `private`, since `verify_transaction_inclusion_v2` already calls it internally via `#[allow(deprecated)]`. Alternatively, add a hard `require!(false, "use verify_transaction_inclusion_v2")` panic at the top of v1 to make it a runtime no-op for external callers while keeping the internal call path through v2 intact (though making it non-public is cleaner).

### Proof of Concept
1. Identify any confirmed mainchain block `B` with at least two transactions. Obtain its full transaction list from a Bitcoin node.
2. Pick any two adjacent leaf hashes `L` and `R` in the Merkle tree. Concatenate them to form a 64-byte value; double-SHA256 this to get the internal node hash `N = SHA256d(L || R)`. This `N` is a valid internal Merkle tree node.
3. Construct a Merkle proof for `N` as if it were a transaction at the parent position in the tree (one level up, with the remaining sibling nodes as the proof path). This proof will correctly reconstruct the block's `merkle_root`.
4. Call `verify_transaction_inclusion` directly on the NEAR contract:
   ```
   verify_transaction_inclusion({
     tx_id: N,                  // forged internal node, not a real tx
     tx_block_blockhash: B,
     tx_index: <parent index>,
     merkle_proof: <shortened proof path>,
     confirmations: 1
   })
   ```
5. The function returns `true` because `compute_root_from_merkle_proof(N, parent_index, proof) == merkle_root` holds by construction — the coinbase proof check that would have caught this in v2 is never executed. [7](#0-6) [8](#0-7)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
