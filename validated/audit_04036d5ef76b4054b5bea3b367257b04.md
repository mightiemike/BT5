### Title
EIP-712 Domain Separator Binds Order Signatures to Wrong Entity (`verifyingContract` = `address(uint160(productId))` Instead of `address(this)`) — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs an EIP-712 domain separator that sets `verifyingContract` to `address(uint160(productId))` — a pseudo-address derived from the product ID — instead of `address(this)` (the actual `OffchainExchange` contract). This is the direct Nado analog of the ENS SOA-check-on-wrong-domain bug: a binding/verification check is performed against the wrong entity, so order signatures are not cryptographically tied to the specific contract that will execute them.

---

### Finding Description

`OffchainExchange.getDigest` is the sole function that produces the digest used to verify every order signature in `matchOrders`:

```solidity
// core/contracts/OffchainExchange.sol  lines 311-321
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← wrong entity
    )
);
return ECDSAUpgradeable.toTypedDataHash(domainSeparator, structHash);
```

For product ID `1` the `verifyingContract` field resolves to `0x0000000000000000000000000000000000000001`, not the deployed `OffchainExchange` address. The product ID is also absent from the struct hash, so the domain separator is the **only** field that differentiates products — but it does so at the cost of never binding the signature to the actual contract.

By contrast, the transaction-level signatures processed in `EndpointTx.validateSignedTx` correctly use `_hashTypedDataV4` (which internally uses `address(this)` — the `Endpoint` address) as the verifying contract:

```solidity
// core/contracts/EndpointTx.sol  lines 96-103
validateSignature(
    sender,
    _hashTypedDataV4(
        computeDigest(
            IEndpoint.TransactionType(uint8(transaction[0])),
            transaction[1:]
        )
    ),
    ...
```

Order signatures and transaction signatures therefore use structurally different domain separators: one is bound to the real contract, the other is not.

---

### Impact Explanation

Because the domain separator does not include the real `OffchainExchange` address, any signed order for product ID `P` produces an identical digest on **every** `OffchainExchange` deployment that has product ID `P` on the same chain. The concrete broken invariant is:

> A user's signed order must be valid only for the specific `OffchainExchange` instance the user intended to interact with.

When the protocol deploys a replacement `OffchainExchange` (not a proxy-upgrade, which preserves storage, but a fresh deployment — a routine operation given the ERC-1967 proxy architecture described in the wiki), the new contract starts with an empty `filledAmounts` mapping. The sequencer, operating normally, may submit previously signed and already-filled orders to the new contract. Because the digest is identical across deployments, `_checkSignature` passes, `filledAmounts[digest]` starts at zero, and the order is filled again — draining the user's collateral a second time without a new signature.

Additionally, users' wallets display `verifyingContract = 0x0000000000000000000000000000000000000001` (for product 1) during signing, making it impossible for users to verify they are authorising the correct contract.

---

### Likelihood Explanation

The trigger requires a fresh `OffchainExchange` deployment (not a proxy upgrade). The protocol explicitly supports upgradeable proxies and contract replacement. The sequencer does not need to be compromised — it simply processes the order queue against the new contract address, unaware that `filledAmounts` state was not migrated. This is a realistic operational scenario with medium likelihood.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator, and move product-ID binding into the struct hash:

```solidity
// domain separator — bind to the actual contract
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)          // ← correct verifyingContract
    )
);

// struct hash — include productId to prevent cross-product replay
bytes32 structHash = keccak256(
    abi.encode(
        keccak256(bytes(structType)),
        productId,             // ← add product binding here
        order.sender,
        order.priceX18,
        order.amount,
        order.expiration,
        order.nonce,
        order.appendix
    )
);
```

This preserves per-product signature isolation while correctly binding every order to the specific contract instance.

---

### Proof of Concept

1. User signs an order for product ID `1` on `OffchainExchange` at address `0xAAA…`. The digest is `D = hash(domainSep(chainId, 0x01), structHash(order))`.
2. Protocol deploys a new `OffchainExchange` at `0xBBB…` (same chain, same product IDs). `filledAmounts` is empty.
3. `getDigest(1, order)` on `0xBBB…` produces the same `D` because `verifyingContract = address(uint160(1)) = 0x01` in both cases.
4. Sequencer submits the old order to `0xBBB…`. `_checkSignature` passes (same digest, same signature). `filledAmounts[D] == 0`, so the order is treated as unfilled.
5. `_updateBalances` executes, debiting the user's collateral a second time. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    ) internal view virtual returns (bool) {
