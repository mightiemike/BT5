### Title
Hardcoded `ModuleAccounts` List in `x/supply` Omits `tieredrewards` Module Accounts, Causing Inflated Liquid Supply - (File: x/supply/keeper/keeper.go)

### Summary
The `x/supply` module computes `liquid_supply` by subtracting a **hardcoded, compile-time static list** of module account balances from total supply. This list was never updated to include the `rewards_pool` and `tieredrewards` module accounts introduced by the `x/tieredrewards` module. As a result, any `basecro` held in the rewards pool is counted as part of the liquid supply, causing the `LiquidSupply` query to return an inflated value.

### Finding Description
`GetLiquidSupply` in `x/supply/keeper/keeper.go` computes:

```
liquid_supply = total_supply - unvested_supply - module_account_balance(ModuleAccounts...)
```

The `ModuleAccounts` variable is a hardcoded slice:

```go
var ModuleAccounts = []string{
    authtypes.FeeCollectorName,
    distrtypes.ModuleName,
    stakingtypes.BondedPoolName,
    stakingtypes.NotBondedPoolName,
    minttypes.ModuleName,
    govtypes.ModuleName,
}
``` [1](#0-0) 

The `x/tieredrewards` module registers two module accounts: `"tieredrewards"` (module name) and `"rewards_pool"` (the bonus rewards pool). Neither appears in `ModuleAccounts`. [2](#0-1) 

The rewards pool is funded by normal bank transfers from any user or governance action, and is actively drained by the `BeginBlocker` and bonus reward claims. Its balance can be substantial. [3](#0-2) 

### Impact Explanation
Every call to `GET /chainmain/supply/v1/liquid` or `chain-maind query supply liquid` returns a `liquid_supply` value that is inflated by exactly the current balance of the `rewards_pool` (and `tieredrewards`) module accounts. The corrupted value is the `liquid_supply` field in `SupplyResponse`. Any exchange, analytics tool, or on-chain consumer that still queries this endpoint receives incorrect circulating supply data. The discrepancy grows proportionally to the rewards pool balance, which is funded by governance and user transfers. [4](#0-3) 

### Likelihood Explanation
The `x/tieredrewards` module is a new addition to the chain. The `x/supply` module's `ModuleAccounts` list was never updated when `x/tieredrewards` was introduced. Any user who funds the rewards pool (a normal bank send transaction) immediately widens the discrepancy. The `x/supply` query endpoint remains live (marked deprecated but still registered and serving responses). [5](#0-4) 

### Recommendation
Add `tieredrewards.ModuleName` and `tieredrewards.RewardsPoolName` to the `ModuleAccounts` slice in `x/supply/keeper/keeper.go`:

```go
var ModuleAccounts = []string{
    authtypes.FeeCollectorName,
    distrtypes.ModuleName,
    stakingtypes.BondedPoolName,
    stakingtypes.NotBondedPoolName,
    minttypes.ModuleName,
    govtypes.ModuleName,
    tieredrewardstypes.ModuleName,       // add
    tieredrewardstypes.RewardsPoolName,  // add
}
```

Alternatively, since the module is already marked deprecated, accelerate its removal per ADR-005. [6](#0-5) 

### Proof of Concept

1. Fund the rewards pool with a normal bank send:
   ```
   chain-maind tx bank send signer1 <rewards_pool_address> 1000000000basecro
   ```
2. Query liquid supply:
   ```
   chain-maind query supply liquid
   ```
3. Observe that the returned `liquid_supply` amount is 1,000,000,000 `basecro` higher than the true circulating supply, because `rewards_pool` balance is not subtracted.
4. The correct value would subtract the rewards pool balance, as `GetTotalModuleAccountBalance` only iterates the hardcoded `ModuleAccounts` list which omits `"rewards_pool"`. [7](#0-6)

// GetTotalModuleAccountBalance returns total balance of given module accounts
func (k Keeper) GetTotalModuleAccountBalance(ctx sdk.Context, moduleNames ...string) sdk.Coins {
	var balance sdk.Coins

	for _, moduleName := range moduleNames {
		balance = balance.Add(k.GetModuleAccountBalance(ctx, moduleName)...)
	}

	return balance
}

// GetLiquidSupply returns the total liquid supply in the system
func (k Keeper) GetLiquidSupply(ctx sdk.Context) sdk.Coins {
	totalSupply := k.GetTotalSupply(ctx)
	unvestedSupply := k.GetUnvestedSupply(ctx)
	moduleAccountBalance := k.GetTotalModuleAccountBalance(ctx, ModuleAccounts...)

	return totalSupply.Sub(unvestedSupply...).Sub(moduleAccountBalance...)

# ADR 005: Deprecate Cronos.org Chain's custom `x/supply` module

## Changelog
* 09-03-2022: Initial draft

## Context

The current `x/supply` module tracks liquid supply of a given token using the following formula:

```
liquid_supply = total_supply - (unvested_supply + module_account_balance)
```

where,

- `total_supply`: Total supply of a `denom` which is obtained from `x/bank` module.
- `unvested_supply`: The sum of tokens locked in vesting accounts (`x/supply` maintains a static list of vesting
  accounts configured in `genesis.json`, it does not support adding/removing vesting accounts).
- `module_account_balance`: The sum of tokens locked in module accounts of different modules (`x/supply` maintains a
  static list of module accounts that it uses to fetch total tokens locked in module accounts)

// ModuleAccounts defines the module accounts which will be queried to get liquid supply

	distrtypes.ModuleName,
	stakingtypes.BondedPoolName,
