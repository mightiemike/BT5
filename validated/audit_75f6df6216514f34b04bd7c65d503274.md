### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Security Check — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) is still a live, unpermissioned public entry point on the NEAR contract. It performs a Merkle proof check but omits the coinbase proof validation that `verify_transaction_inclusion_v2` was specifically introduced to enforce. Any unprivileged NEAR caller — including a consumer bridge, swap, or lending contract — can invoke v1 directly, bypassing the 64-byte transaction Merkle forgery protection entirely. The security check exists in the codebase but is checked against the wrong (incomplete) set of conditions, exactly mirroring the H-58 pattern.

### Finding Description

`verify_transaction_inclusion_v2` was introduced to mitigate the [64-byte transaction Merkle proof forgery vulnerability](https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase Merkle proof that must verify against the same block's `merkle_root` at index 0: [1](#0-0) 

Only after that coinbase check passes does v2 delegate to v1: [2](#0-1) 

However, v1 itself is still decorated with `#[pause]` (not `#[private]`) and carries only a Rust-compiler-level `#[deprecated]` attribute, which is invisible at the NEAR protocol level: [3](#0-2) 

The `#[deprecated]` attribute emits a compiler warning; it does **not** remove the method from the contract's ABI or prevent external callers from invoking it. Any NEAR account can call `verify_transaction_inclusion` directly with a `ProofArgs` payload, skipping the coinbase proof requirement entirely.

Inside v1, the only Merkle check performed is: [4](#0-3) 

`ProofArgs` has no `coinbase_tx_id` or `coinbase_merkle_proof` fields: [5](#0-4) 

So the coinbase proof check is structurally absent from v1 — it cannot be added without changing the function signature.

**H-58 analog mapping:**

| H-58 (TapiocaOFT) | This repository |
|---|---|
| Checks TapiocaOFT allowance (wrong state) | Checks tx Merkle proof only (incomplete condition set) |
| ERC20 allowance (the real restriction) is never checked | Coinbase proof (the real restriction) is never checked |
| TapiocaOFT allowance is never decremented → repeated exploitation | Coinbase proof is never required → forgery accepted on every call |
| Caller bypasses the intended ERC20 spend restriction | Caller bypasses the intended 64-byte forgery protection |

### Impact Explanation

A consumer contract (bridge, atomic swap, lending protocol) that calls `verify_transaction_inclusion` and acts on a `true` return value can be deceived. An attacker crafts a 64-byte payload that is simultaneously a valid serialized Bitcoin transaction and a valid internal Merkle tree node. Because internal nodes are computed as `SHA256d(left || right)` over 64 bytes, a carefully constructed fake transaction can produce a hash that equals an internal node of a real block's Merkle tree. The v1 function will compute the Merkle root from the attacker-supplied proof and compare it to the stored `header.block_header.merkle_root`, returning `true` for a transaction that does not exist in the block. Any asset release, bridge mint, or collateral unlock gated on this return value is at risk of theft.

### Likelihood Explanation

The function is reachable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond gas. The `#[pause]` guard is the only runtime gate, and the contract is not paused in normal operation. Consumer contracts that integrated before v2 was introduced, or that read the public ABI without noticing the deprecation comment, will call v1. The 64-byte forgery technique is publicly documented and has known tooling.

### Recommendation

Mark `verify_transaction_inclusion` with `#[private]` so it is only callable from within the contract itself (i.e., only via `verify_transaction_inclusion_v2`):

```rust
#[private]   // <-- add this
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

This mirrors the recommended fix in H-58: replace the insufficient check with a strict requirement that the caller must go through the safe path.

### Proof of Concept

1. Identify a real Bitcoin block `B` whose Merkle tree has at least two transactions. Let the Merkle root be `R`.
2. Craft a 64-byte payload `F` such that `SHA256d(F)` equals an internal Merkle node `N` of `B`'s tree. `F` is the attacker's fake "transaction."
3. Construct a Merkle proof `P` that, starting from `N` (treated as a leaf), walks up to `R`. This proof is valid because `N` is a genuine internal node.
4. Call `verify_transaction_inclusion` on the NEAR contract with:
   - `tx_id = SHA256d(F)` (the fake tx hash)
   - `tx_block_blockhash = hash(B)`
   - `tx_index` = position of `N` in the tree
   - `merkle_proof = P`
   - `confirmations = 1`
5. The function computes `compute_root_from_merkle_proof(tx_id, tx_index, P)` which equals `R`, matches `header.block_header.merkle_root`, and returns `true`.
6. The consumer contract releases funds for a transaction that never existed. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
