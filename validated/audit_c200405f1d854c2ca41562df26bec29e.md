Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual calling address) and gates access on `owner`, a free caller-supplied parameter. Because `owner` is not authenticated, any non-allowlisted address can add liquidity to an allowlist-protected pool by nominating any already-allowlisted address as `owner`. This completely nullifies the deposit allowlist as an access-control mechanism.

## Finding Description

`MetricOmmPool.addLiquidity` calls the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

This passes `msg.sender` (the actual caller) as `sender` and the user-supplied `owner` parameter as `owner`. The extension receives both but discards `sender` entirely:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

The guard checks `allowedDepositor[pool][owner]`, not `allowedDepositor[pool][sender]`. `owner` is a free parameter accepted directly from the caller by `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with no allowlist validation — `_validateOwner` only rejects `address(0)`: [3](#0-2) [4](#0-3) 

An attacker (Bob, not allowlisted) sets `owner` to any allowlisted address (Alice). The extension sees `allowedDepositor[pool][alice] == true`, the guard passes, and Bob's tokens are pulled via the callback (`payer = msg.sender` of the LiquidityAdder call). The LP position is minted to Alice. The same bypass works when calling `pool.addLiquidity` directly, since the pool passes `msg.sender` as `sender` and the attacker-supplied value as `owner`, and the extension only inspects `owner`.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [5](#0-4) 

The asymmetry confirms the deposit guard has the wrong field.

## Impact Explanation

`DepositAllowlistExtension` is the sole on-chain mechanism by which a pool admin restricts who may add liquidity. Bypassing it allows any unprivileged address to add liquidity to intentionally restricted pools (e.g., institutional or KYC-gated pools), violating the admin's access-control intent. This is a direct admin-boundary break: an unprivileged path circumvents an admin-configured access control. The attacker can alter the pool's liquidity distribution and effective depth at any price level, and can pair the bypass with other extensions (`OracleValueStopLossExtension`, `PriceVelocityGuardExtension`) to shift watermarks or trigger guard conditions, potentially blocking legitimate swaps.

## Likelihood Explanation

The bypass requires no special permissions, no flash loan, and no privileged role. The attacker only needs to know one allowlisted address, which is trivially discoverable from on-chain events or the public `allowedDepositor` mapping. The attack is a single transaction through the existing `MetricOmmPoolLiquidityAdder` router. Likelihood is **high**.

## Recommendation

Replace the `owner` check with a `sender` check in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// Setup: pool has DepositAllowlistExtension; only `alice` is allowlisted.
// Attacker (bob) is NOT allowlisted.

// Bob calls the LiquidityAdder, nominating alice as owner.
liquidityAdder.addLiquidityExactShares(
    pool,
    alice,      // owner — allowlisted, passes the guard
    0,          // salt
    deltas,
    maxAmount0,
    maxAmount1,
    ""
);
// Extension checks allowedDepositor[pool][alice] == true → no revert.
// Bob's tokens are pulled via callback (payer = bob).
// LP position minted to alice.
// Bob has successfully added liquidity despite not being allowlisted.
```
