Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender`/payer of the pool call) and gates only on `owner` (the caller-supplied LP-share recipient). Because `owner` is a free parameter in `pool.addLiquidity`, any address can pass an allowlisted address as `owner` and satisfy the check while the attacker's own tokens fund the position, completely defeating pool curation.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both verbatim to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as an **unnamed, discarded** `address` and checks only `owner`: [3](#0-2) 

The `allowedDepositor` mapping is keyed on `depositor`, and the contract NatDoc states "Gates `addLiquidity` by depositor address" — the depositor is the party paying tokens, i.e., `sender`, not `owner`. The implementation checks the wrong address. An attacker not on the allowlist calls `pool.addLiquidity(owner = allowedUser, ...)`. The extension evaluates `allowedDepositor[pool][allowedUser] == true` and passes. The attacker's tokens are pulled via the modify-liquidity callback (they are `msg.sender` in the pool), and LP shares are minted to `allowedUser`. No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner` but `addLiquidity` has no such check. [4](#0-3) 

## Impact Explanation
The deposit allowlist invariant is broken and pool curation is completely defeated. Any unprivileged address can inject liquidity into a curated pool by supplying an allowlisted address as `owner`. The attacker has no financial gain (they pay their own tokens), but the pool admin's core access-control mechanism is rendered non-functional. The allowlisted `owner` also receives unsolicited LP exposure they did not initiate. This constitutes broken core extension functionality — the `DepositAllowlistExtension` provides zero restriction on who can deposit. Severity is **Medium**: the allowlist invariant is broken and pool curation is defeated, but there is no direct loss of existing LP principal or protocol fees.

## Likelihood Explanation
Exploitable by any address holding tokens approved to the pool. No special role, oracle access, or malicious setup is required — only knowledge of one allowlisted address, which is publicly readable on-chain via `allowedDepositor`. [5](#0-4) 

## Recommendation
Check `sender` (the actual operator/payer) instead of `owner` in `beforeAddLiquidity`:

```solidity
// fixed
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
  external
  view
  override
  returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
function test_allowlistBypass() public {
    address allowedUser = makeAddr("allowedUser");
    address attacker    = makeAddr("attacker");

    // Pool admin allowlists only allowedUser
    depositExtension.setAllowedToDeposit(address(pool), allowedUser, true);

    // Fund and approve attacker
    token0.mint(attacker, 1e18);
    token1.mint(attacker, 1e18);
    vm.startPrank(attacker);
    token0.approve(address(pool), type(uint256).max);
    token1.approve(address(pool), type(uint256).max);

    // Attacker calls addLiquidity with owner = allowedUser
    // Extension checks allowedDepositor[pool][allowedUser] == true → passes
    // Attacker's tokens are pulled; LP shares go to allowedUser
    pool.addLiquidity(allowedUser, salt, deltas, callbackData, "");
    // ✓ succeeds — allowlist bypassed

    // Confirm attacker's own address is blocked
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToDeposit.selector);
    pool.addLiquidity(attacker, salt, deltas, callbackData, "");
    vm.stopPrank();
}
```
