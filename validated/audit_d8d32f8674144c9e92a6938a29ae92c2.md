### Title
Difficulty Retarget for Fork Blocks Uses Mainchain Ancestor Timestamp Instead of Fork's Actual Ancestor — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When validating a fork block, `get_next_work_required` fetches the first-interval boundary block via `get_header_by_height`, which always returns the **mainchain** block at that height. If the fork diverged before that boundary height, the mainchain block is a different block than the fork's actual ancestor, so the difficulty calculation uses the wrong timestamp. An attacker who controls the fork's block timestamps can exploit this discrepancy to obtain a lower required difficulty for their fork blocks than the protocol mandates, enabling a chain reorg with less accumulated proof-of-work than should be required.

---

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty adjustment interval boundary height and then fetches the boundary block:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

The same pattern appears in `contract/src/litecoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [2](#0-1) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [3](#0-2) 

This unconditionally reads from `mainchain_height_to_header`, which maps heights to the **canonical mainchain** block hashes. Fork blocks are stored only in `headers_pool` (via `store_fork_header`) and are never inserted into `mainchain_height_to_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [4](#0-3) 

Therefore, when a fork block at height H is being validated and the fork diverged at or before `height_first = H - blocks_to_go_back`, the call to `get_header_by_height(height_first)` returns the **mainchain's** block at that height, not the fork's actual ancestor. The difficulty retarget formula:

```
modulated_timespan = prev_block_time (fork tip) − first_block_time (mainchain block)
``` [5](#0-4) 

uses a `first_block_time` that belongs to a different chain branch than the block being validated. The codebase itself flags this with a `TODO` comment acknowledging the correctness concern.

---

### Impact Explanation

The difficulty retarget controls the `bits` value that every submitted fork block must satisfy. If `first_block_time` is artificially old (mainchain block has an older timestamp than the fork's actual ancestor at the same height), `modulated_timespan` is inflated, the computed target is easier, and the required PoW is lower than the protocol mandates.

A fork chain built with under-required PoW can still accumulate `chain_work` (computed from the accepted `bits` value) and, once its `chain_work` exceeds the mainchain tip's `chain_work`, trigger `reorg_chain`:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [6](#0-5) 

A successful reorg replaces the canonical chain tip and all `mainchain_height_to_header` / `mainchain_header_to_height` entries for the affected range. Any subsequent `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` call will verify against the attacker-controlled chain, returning `true` for transactions that were never confirmed on the real chain. This is a direct, concrete loss of the light client's security guarantee.

---

### Likelihood Explanation

The attack is reachable whenever a fork block is submitted whose divergence point is at or before the difficulty-interval boundary block (`height_first`). For Dogecoin (post-block 145,000, `difficulty_adjustment_interval = 1`), every single fork block triggers the vulnerable path because `blocks_to_go_back = 1` and `height_first = prev_height − 1`, meaning any fork that diverges two or more blocks back uses the wrong timestamp. For Litecoin (interval = 2016), the window is narrower but still reachable for any fork that spans a retarget boundary. The `TODO` comment in the production code confirms the developers themselves identified this as an open correctness question.

---

### Recommendation

Replace the `get_header_by_height` lookup with an ancestor walk that follows `prev_block_hash` links backward through `headers_pool` from the fork's previous block until the target height is reached. This ensures the boundary block is always the fork's actual ancestor, regardless of what the mainchain holds at that height. The walk is bounded by `blocks_to_go_back` steps, which is at most `difficulty_adjustment_interval` (2016 for Litecoin/Bitcoin, 1 for Dogecoin post-145k), so gas cost is predictable.

---

### Proof of Concept

**Setup (Dogecoin mainnet, post-block 145,000, `difficulty_adjustment_interval = 1`):**

1. Mainchain has blocks at heights …, H−2 (mainchain), H−1 (mainchain), H (mainchain tip). The mainchain block at H−2 has timestamp `T_mc` (e.g., 60 seconds before H−1).

2. Attacker submits a fork block at height H−2 (diverging from the mainchain at H−3). The fork block at H−2 has timestamp `T_fork = T_mc + 3600` (one hour later, within the `MAX_FUTURE_BLOCK_TIME_LOCAL = 7200 s` limit).

3. Attacker submits a fork block at height H−1 with timestamp `T_fork + 60`.

4. Attacker submits a fork block at height H with timestamp `T_fork + 120`. The contract calls `get_next_work_required`:
   - `prev_block_time = T_fork + 60` (fork's H−1 block — correct)
   - `height_first = H − 2`
   - `first_block_time = get_header_by_height(H−2).time = T_mc` ← **mainchain block, not fork ancestor**

5. `modulated_timespan = (T_fork + 60) − T_mc ≈ 3660 s` instead of the correct `(T_fork + 60) − T_fork = 60 s`.

6. Dogecoin's `calculate_next_work_required` clamps to `max_timespan = retarget_timespan + retarget_timespan/2 = 90 s`, but the inflated timespan still produces a target easier than the correct 60 s interval would yield.

7. The attacker's fork block is accepted with lower PoW than required. Repeating this across enough blocks accumulates sufficient `chain_work` to trigger `reorg_chain`, replacing the canonical chain with the attacker's fork. [7](#0-6) [8](#0-7)

fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))

fn get_next_work_required(
    config: &NetworkConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    if (prev_block_header.block_height + 1) % config.difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            if block_header.time
                > prev_block_header.block_header.time + 2 * config.pow_target_spacing
            {
                return config.proof_of_work_limit_bits;
            }

            let mut current_block_header = prev_block_header.clone();
            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            let last_bits = current_block_header.block_header.bits;
            return last_bits;
        }
        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = config.difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != config.difficulty_adjustment_interval {
        blocks_to_go_back = config.difficulty_adjustment_interval;
    }

    let first_block_height = prev_block_header.block_height - blocks_to_go_back;

    let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
    calculate_next_work_required(
        config,
        prev_block_header,
        interval_tail_extend_header.block_header.time.into(),
    )
