### Title
`ValidResourceBounds` Variant Flip During P2P Protobuf Deserialization Causes Valid Transaction Rejection and Hash Domain Inconsistency - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to select between the `L1Gas` and `AllResources` variants. A transaction submitted to the gateway with `AllResourceBounds { l1_gas: NON_EMPTY, l2_gas: ZERO, l1_data_gas: ZERO }` is accepted and hashed under the `AllResources` domain (which includes `L1_DATA_GAS` in the resource-bounds hash). When that same transaction is serialized to protobuf and received by a P2P peer, the deserializer silently flips the variant to `L1Gas`, causing the peer to reject the transaction with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The same flip would produce a different transaction hash if re-hashing were attempted, because `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` only for `AllResources`.

### Finding Description

**Step 1 – Gateway accepts the transaction under `AllResources`.**

`RpcInvokeTransactionV3` and `RpcInvokeTransactionV3` both store `resource_bounds: AllResourceBounds`. The gateway's stateless validator accepts a transaction where `l2_gas` and `l1_data_gas` are zero (the `valid_l1_gas` test case confirms this). [1](#0-0) 

**Step 2 – Hash is computed under `AllResources` domain.**

`InternalRpcInvokeTransactionV3::resource_bounds()` unconditionally returns `ValidResourceBounds::AllResources(self.resource_bounds)`. `get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash` with this value. For `AllResources`, the hash includes the `L1_DATA_GAS` resource felt; for `L1Gas` it does not. [2](#0-1) [3](#0-2) 

**Step 3 – Serialization preserves all three fields.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources { l1_gas, l2_gas: ZERO, l1_data_gas: ZERO }` as `{ l1_gas: Some(NON_EMPTY), l2_gas: Some(ZERO), l1_data_gas: Some(ZERO) }` — all three fields are present in the wire message. [4](#0-3) 

**Step 4 – Deserialization silently flips the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies a zero-value heuristic: if `l1_data_gas.is_zero() && l2_gas.is_zero()` it returns `ValidResourceBounds::L1Gas(l1_gas)` instead of `AllResources`. The original variant information is lost. [5](#0-4) 

**Step 5 – Downstream conversion fails.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()` which is `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion returns `Err` for any variant other than `AllResources`, mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The receiving peer rejects the transaction. [6](#0-5) 

This affects both the mempool P2P path (`MempoolTransaction::InvokeV3`) and the consensus P2P path (`ConsensusTransaction::InvokeV3`). The developers are aware of the symptom — the consensus test explicitly patches gas values to avoid it — but the root cause in the deserializer is unfixed. [7](#0-6) 

**Hash domain inconsistency (secondary).**

If a node were to re-hash the transaction after deserialization (e.g., during state sync verification), the `L1Gas` variant would produce a hash that omits `L1_DATA_GAS`, diverging from the hash stored in `InternalRpcTransaction::tx_hash`. This is the direct analog of the external report: a value (the resource-bounds hash domain) is fixed at submission time but re-evaluated under different parameters at a later processing stage. [8](#0-7) 

### Impact Explanation

A valid transaction accepted by the gateway is silently rejected by every P2P peer that receives it. In the consensus path, a proposer that includes such a transaction in a block will have that block rejected by all validators, because the `ConsensusTransaction` deserialization fails before the transaction can be executed. This matches:

- **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**
- **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

Any user who submits a V3 invoke transaction with non-zero `l1_gas` but zero `l2_gas` and `l1_data_gas` — a configuration the gateway explicitly accepts — triggers this path. No special privileges are required. The gateway's `valid_l1_gas` test case is exactly this configuration.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Since the RPC/consensus/mempool pipeline only accepts `AllResources` transactions (all three RPC transaction types store `AllResourceBounds`), the deserializer should always produce `AllResources` when all three fields are present in the protobuf message, regardless of their values. Alternatively, add an explicit discriminant field to the `ResourceBounds` protobuf message.

### Proof of Concept

```
1. Submit to gateway:
   RpcInvokeTransactionV3 {
     resource_bounds: AllResourceBounds {
       l1_gas:      ResourceBounds { max_amount: 1, max_price_per_unit: 1 },
       l2_gas:      ResourceBounds { max_amount: 0, max_price_per_unit: 0 },
       l1_data_gas: ResourceBounds { max_amount: 0, max_price_per_unit: 0 },
     },
     ...
   }

2. Gateway accepts (valid_l1_gas path). Hash H is computed via
   get_invoke_transaction_v3_hash → get_tip_resource_bounds_hash(AllResources)
   which includes L1_DATA_GAS=0 in the poseidon hash.

3. Transaction is serialized to protobuf::InvokeV3WithProof:
   resource_bounds = { l1_gas: Some(1,1), l2_gas: Some(0,0), l1_data_gas: Some(0,0) }

4. Peer receives protobuf. ValidResourceBounds::try_from sees
   l1_data_gas.is_zero() && l2_gas.is_zero() → returns L1Gas(l1_gas).

5. RpcInvokeTransactionV3::try_from(InvokeTransactionV3 { resource_bounds: L1Gas(..) })
   → Err(DEPRECATED_RESOURCE_BOUNDS_ERROR: "ResourceBounds::l1_data_gas")

6. Peer rejects the transaction. If this transaction was in a consensus block,
   the block is rejected by all validators.

7. If re-hashing were attempted with L1Gas variant:
   get_tip_resource_bounds_hash(L1Gas) omits L1_DATA_GAS → hash H' ≠ H.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
