### Title
Merkle Second-Preimage Attack: `verify_transaction_inclusion` Accepts Internal Node Hashes as Valid Transaction Proofs — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` applies no domain separation between leaf nodes and internal nodes. An unprivileged NEAR caller can supply an internal Merkle tree node hash as the `tx_id` argument to `verify_transaction_inclusion`, provide a shortened proof that skips the lower tree levels, and receive a `true` return value for a transaction that does not exist.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof elements and hashes them together unconditionally, treating the initial `transaction_hash` argument identically regardless of whether it is a leaf (real txid) or an internal node: [1](#0-0) 

There is no byte prefix, salt, or structural tag distinguishing a leaf hash from an intermediate node hash. The function simply computes `double_sha256(current_hash || proof_hash)` at each level.

`verify_transaction_inclusion` passes the caller-controlled `args.tx_id` directly as the starting hash into this function: [2](#0-1) 

Because the Bitcoin Merkle tree also uses `double_sha256` for internal nodes with no domain separation, any internal node hash `N` at depth `d` from the root is a valid starting point. An attacker who knows the tree structure can:

1. Pick any internal node `N` at depth `d`.
2. Set `tx_id = N`, `tx_index` = the position of `N` at its level.
3. Provide a proof of length `d` (the `d` sibling hashes from `N` up to the root), skipping all levels below `N`.
4. `compute_root_from_merkle_proof` will correctly recompute the root and the function returns `true`.

The contract's own developer documentation explicitly acknowledges this: [3](#0-2) 

Despite being marked `#[deprecated]`, `verify_transaction_inclusion` remains a `pub fn` decorated only with `#[pause]` (not role-restricted), meaning any unprivileged NEAR account can call it while the contract is unpaused: [4](#0-3) 

The `#[deprecated]` Rust attribute is a compiler lint only; it does not prevent external NEAR callers from invoking the function.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` and acts on a `true` result (e.g., minting wrapped BTC, releasing escrowed funds, authorizing a cross-chain action) can be deceived into accepting a fabricated "transaction inclusion" proof. The attacker does not need to forge a Bitcoin block or break any hash function — they only need knowledge of the Merkle tree structure of any real Bitcoin block already stored in the contract.

---

### Likelihood Explanation

The Bitcoin Merkle tree structure for any confirmed block is fully public. An attacker can compute all internal node hashes offline from block data, select a suitable internal node, and construct the shortened proof trivially. No privileged access, leaked keys, or social engineering is required. The entry point (`verify_transaction_inclusion`) is a public NEAR contract method callable by any account.

---

### Recommendation

1. **Remove or hard-disable `verify_transaction_inclusion` (v1).** It is already deprecated; it should be made uncallable (e.g., always panic, or remove from the ABI entirely) rather than merely discouraged.
2. **Enforce proof-depth validation in `compute_root_from_merkle_proof`.** The tree depth can be inferred from the coinbase proof length (as `verify_transaction_inclusion_v2` already does). Reject any proof whose length does not match the expected tree depth.
3. **Add domain separation.** Prepend a `0x00` byte before hashing leaf inputs and a `0x01` byte before hashing internal node inputs, consistent with RFC 6962 / the plasma-contracts fix referenced in the report.

---

### Proof of Concept

Given a real Bitcoin block with 8 transactions whose Merkle root is `R`:

```
Level 2 (root):        R = H(AB, CD)
Level 1:        AB = H(A,B)    CD = H(C,D)
Level 0 (leaves): A  B  C  D
```

Attacker calls `verify_transaction_inclusion` with:
- `tx_id = AB` (an internal node, not a real txid)
- `tx_index = 0` (position of `AB` at level 1)
- `merkle_proof = [CD]` (one element, skipping level 0 entirely)
- `tx_block_blockhash` = any valid stored block hash with root `R`
- `confirmations = 1`

`compute_root_from_merkle_proof(AB, 0, [CD])`:
- iteration 1: `0 % 2 == 0` → `current_hash = H(AB, CD) = R`
- proof exhausted → returns `R`

`R == header.block_header.merkle_root` → function returns `true`. [5](#0-4) [6](#0-5)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
