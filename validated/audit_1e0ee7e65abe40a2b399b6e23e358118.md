### Title
Single Failing Token Transfer

### Title
Single Failing Token Blocks All Deposits in `DirectDepositV1.creditDeposit()` — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates through every registered spot product ID and calls `token.approve` then `endpoint.depositCollateralWithReferral` for each token with a non-zero balance, with no try/catch around either call. If any single token's approve or transfer reverts, the entire loop reverts, leaving all other tokens stranded in the contract until the owner manually recovers them via `withdraw`.

---

### Finding Description

`creditDeposit()` is an unrestricted external function. It fetches all product IDs from the spot engine and, for each product whose token balance in the contract is non-zero, executes two sequential external calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
``` [1](#0-0) 

Neither call is wrapped in a try/catch. `depositCollateralWithReferral` internally calls `requireUnsanctioned(msg.sender)` and `requireUnsanctioned(sender)`, and then `handleDepositTransfer` which calls `token.safeTransferFrom`. [2](#0-1) 

Any of the following conditions causes a hard revert that aborts the entire loop:

- A registered token that implements USDT-style approve (reverts on non-zero → non-zero allowance transition), which can occur if a prior `creditDeposit` call left a residual allowance after a partial failure.
- A registered token that is paused or blacklists the `DirectDepositV1` contract address, causing `safeTransferFrom` to revert.
- The `DirectDepositV1` contract address or the subaccount's derived address appearing on the sanctions list, causing `requireUnsanctioned` to revert.

Because there is no error isolation, a revert on token `i` prevents tokens `i+1 … n` from ever being credited, even though those tokens are entirely unrelated to the failure.

---

### Impact Explanation

All tokens sitting in the `DirectDepositV1` contract are frozen. Users who sent funds expecting them to be credited to their subaccount cannot access those funds on-chain. Recovery requires the owner to call the `withdraw(token)` function individually for each stuck token — direct manual intervention analogous to the manager needing to update the `withdrawPriority` queue in M-02. [3](#0-2) 

The corrupted state is the token balances stranded in the `DirectDepositV1` contract that should have been credited to the subaccount's on-chain balance.

---

### Likelihood Explanation

`creditDeposit()` has no access control — any address can call it. The contract is designed to hold multiple tokens simultaneously. A realistic trigger exists whenever:

1. Any registered token implements non-standard approve semantics (USDT on Ethereum mainnet is a canonical example).
2. A token issuer pauses transfers (Circle can pause USDC).
3. The contract address or subaccount address is added to a sanctions list.

An adversary can also deliberately trigger the condition by sending a dust amount of any registered token whose `approve` or `transfer` is known to revert under current conditions, then calling `creditDeposit()` to freeze all other tokens in the contract.

---

### Recommendation

Wrap each iteration's external calls in a try/catch block so that a failure on one token is skipped and the loop continues to process the remaining tokens:

```solidity
try token.approve(address(endpoint), balance) {
    try endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1") {
        // success
    } catch { /* skip this token */ }
} catch { /* skip this token */ }
```

Alternatively, emit an event on failure so off-chain monitoring can detect and alert on stuck tokens.

---

### Proof of Concept

1. `DirectDepositV1` holds 1000 USDC (productId = 0) and 1 WETH (productId = 2).
2. A prior partial failure left a non-zero USDC allowance on the endpoint.
3. Anyone calls `creditDeposit()`.
4. Loop iteration for USDC: `token.approve(address(endpoint), 1000e6)` reverts because USDT/USDC-style tokens require allowance to be zeroed first.
5. The entire transaction reverts.
6. WETH (and any other tokens) remain stranded in the contract.
7. The owner must manually call `withdraw(USDC)` to clear the allowance issue, then call `creditDeposit()` again — direct manual intervention required, mirroring the M-02 pattern exactly. [1](#0-0)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

                uint8(TransactionType.DepositCollateral),
