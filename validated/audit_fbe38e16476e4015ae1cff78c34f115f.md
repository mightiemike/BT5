### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Tree Nodes as Valid `tx_id`, Enabling Proof-Verification Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable by any unprivileged NEAR account. It performs no validation that the caller-supplied `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. An attacker can supply an internal node hash as `tx_id` together with a valid proof path from that node to the block's Merkle root, causing the function to return `true` for a transaction that was never included in the block. Downstream NEAR DApps that consume this result are deceived into accepting a forged inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` accepts a `ProofArgs` struct containing `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations`. The only input validation applied to `tx_id` itself is the indirect check that `merkle_proof` is non-empty: [1](#0-0) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` treats whatever hash is passed as `transaction_hash` as the starting leaf and iterates through the proof elements to compute the root: [2](#0-1) 

There is no check at any layer that `tx_id` is a leaf-level hash (i.e., a real transaction) rather than an internal Merkle tree node. The contract's own inline documentation acknowledges this broken invariant: [3](#0-2) 

The function is annotated `#[deprecated]` but Rust's `#[deprecated]` attribute emits only a compiler warning; it imposes no runtime restriction. The function carries only `#[pause]`, meaning any unprivileged NEAR account can call it whenever the contract is not paused: [4](#0-3) 

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase Merkle proof, but it does not remove or gate the v1 path: [5](#0-4) 

The `ProofArgs` struct that carries the unvalidated `tx_id` is defined with no format or semantic constraints: [6](#0-5) 

This is the direct analog to the Rocketpool pattern: attacker-controlled input (`tx_id`) is accepted with only a trivial structural check (non-empty proof), stored as the starting point of a cryptographic computation, and the result is returned to downstream consumers without any validation that the input represents a semantically valid entity (a real transaction leaf, not an internal node).

---

### Impact Explanation

Any NEAR DApp (bridge, payment verifier, escrow) that calls `verify_transaction_inclusion` and acts on a `true` return value is deceived into accepting a forged transaction inclusion proof. The attacker does not need to mine a block or control a relayer. They only need to:

1. Identify a block already accepted into the contract's mainchain.
2. Compute the block's Merkle tree from its publicly available transaction list.
3. Select any internal node at depth *d* from the root.
4. Construct a *d*-element proof path from that internal node to the Merkle root.
5. Call `verify_transaction_inclusion` with the internal node hash as `tx_id`.

The function returns `true`. A downstream contract that gates fund release, cross-chain message acceptance, or any other action on this result will execute incorrectly. The corrupted value is the proof result itself: `true` is returned where `false` is the only correct answer.

---

### Likelihood Explanation

The attack is computationally trivial. Bitcoin block transaction lists are public. Computing the Merkle tree and extracting an internal node hash requires only standard double-SHA256 operations on publicly available data. No privileged access, no key material, and no mining power is required. The only prerequisite is that the target block is present in the contract's mainchain, which is the normal operating state. Any unprivileged NEAR account can submit the call.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it behind an access-control role so that no unprivileged caller can invoke it. All consumers must be migrated to `verify_transaction_inclusion_v2`. If backward compatibility is required during a transition period, add an explicit `require!(false, "use verify_transaction_inclusion_v2")` guard to make the deprecated path unconditionally revert rather than silently return a forgeable result.

---

### Proof of Concept

```
1. Let B be any block hash present in the contract's mainchain.
2. Fetch B's full transaction list from a Bitcoin node (public data).
3. Build the Merkle tree: compute all internal node hashes bottom-up using
   double_sha256(left_child || right_child).
4. Pick any internal node N at depth d (d >= 1) from the root.
   N's position in the level-d array is index i.
5. Build a d-element Merkle proof: for each level from d up to the root,
   append the sibling hash.
6. Call verify_transaction_inclusion with:
     tx_id              = N   (internal node hash, not a real transaction)
     tx_block_blockhash = B
     tx_index           = i   (position of N at depth d)
     merkle_proof       = [sibling_d, sibling_{d-1}, ..., sibling_1]
     confirmations      = 1
7. compute_root_from_merkle_proof(N, i, proof) == B.merkle_root  → true
8. verify_transaction_inclusion returns true.
9. Any downstream DApp acting on this result treats the forged "transaction"
   N as confirmed in block B.
``` [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
