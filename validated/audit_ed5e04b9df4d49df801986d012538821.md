Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any caller to bypass a pool's swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. A pool admin who allowlists the router (the only way to let any user reach the pool through the standard periphery path) inadvertently opens the gate to every address on the network, rendering the per-user allowlist a no-op.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router — not the originating user: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, so `msg.sender` at the pool is the router: [4](#0-3) 

This creates an inescapable dilemma for pool admins:
- If the router is **not** allowlisted: allowlisted users cannot use the router at all (the check fails with `NotAllowedToSwap` because the router is not in the allowlist).
- If the router **is** allowlisted: `allowedSwapper[pool][router]` is `true` for every call through the router, so any unprivileged address can bypass the per-user restriction.

No existing guard checks the originating user's address. The `extensionData` field passed by the router is an empty string `""`, so no user identity is forwarded: [5](#0-4) 

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The attacker receives the full output token amount from the pool at live oracle prices; the pool's LP providers bear the resulting exposure to unrestricted unauthorized flow. This constitutes broken core pool functionality — the allowlist guard the pool was configured to enforce is silently bypassed, matching the "Broken core pool functionality causing loss of funds or unusable swap flows" impact category.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary supported periphery path for swaps. The bypass requires no special privileges, no flash loans, and no unusual token behavior. Any EOA calls a standard router function. The operational pressure on pool admins to allowlist the router (otherwise their allowlisted users cannot use the supported periphery at all) makes the vulnerable configuration the natural and expected one.

## Recommendation
The `beforeSwap` hook receives `extensionData` which can carry the originating user's address. The minimal on-chain fix is to have the router encode `msg.sender` (the originating user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address when `sender` is a recognized trusted router. Alternatively, the allowlist semantics should be explicitly split: the router is allowlisted as a trusted forwarder, and the router enforces its own per-user access control, making the two layers non-conflatable and documented as such. [6](#0-5) 

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, allowedUser, true)
  pool admin calls setAllowedToSwap(pool, router, true)   ← required for allowedUser to use router

Attack:
  attacker (not in allowlist) calls:
    router.exactInputSingle({
      pool: pool,
      recipient: attacker,
      zeroForOne: true,
      amountIn: X,
      ...
    })

  pool.swap(msg.sender=router) fires
  _beforeSwap(sender=router, ...)
  SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  swap executes, attacker receives output tokens

Result:
  attacker bypasses the per-user allowlist and executes a full swap
  against a pool configured to restrict trading to allowedUser only
```
