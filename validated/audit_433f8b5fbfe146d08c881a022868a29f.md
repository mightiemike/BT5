### Title
Unrestricted `creditDeposit()` Allows Any Caller to Force-Deposit DDA Tokens, Permanently Blocking Owner Withdrawal — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` carries no access control modifier, making it callable by any external address. An attacker can front-run the contract owner's `withdraw()` call by invoking `creditDeposit()` first, sweeping all token balances held by the DDA into the protocol-side subaccount. The owner's `withdraw()` then transfers zero tokens. If the target subaccount is simultaneously unhealthy, the deposited tokens become inaccessible through the normal protocol withdrawal path as well, resulting in an effective asset lock.

---

### Finding Description

`DirectDepositV1` is a deposit-automation contract that holds ERC-20 tokens and credits them to a fixed `subaccount` set at construction time. Two mutually exclusive exit paths exist for tokens held by the contract:

- `creditDeposit()` — deposits all token balances into the protocol subaccount. **No access control.**
- `withdraw()` — transfers all token balances directly to the owner. **`onlyOwner`.** [1](#0-0) 

The asymmetry is the root cause: `withdraw()` is correctly gated, but `creditDeposit()` is not. Any unprivileged caller can invoke `creditDeposit()` at any time, draining the DDA's token balances into the protocol before the owner can call `withdraw()`.

The attack path:

1. Tokens accumulate in the DDA (e.g., a token sent by mistake, or a token the owner wishes to recover before depositing).
2. Owner submits a `withdraw()` transaction.
3. Attacker observes the pending transaction in the mempool and front-runs it with `creditDeposit()`.
4. `creditDeposit()` iterates over all `spotEngine` product IDs, approves the endpoint, and calls `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")` for each non-zero balance. [2](#0-1) 

5. The DDA's token balances are now zero.
6. Owner's `withdraw()` executes successfully but transfers `0` tokens. [3](#0-2) 

The tokens are now inside the protocol under `subaccount`. If that subaccount is at or below its maintenance margin at the time of the forced deposit, `withdrawCollateral` will revert on the health check: [4](#0-3) 

The owner cannot withdraw through the protocol either, creating an effective asset lock until the subaccount health recovers — which may never happen if the subaccount is being liquidated.

---

### Impact Explanation

- **Direct asset lock**: Tokens the owner intended to recover directly from the DDA are swept into the protocol. If the subaccount is unhealthy, the owner cannot withdraw them through the protocol either.
- **Forced protocol exposure**: Even in the healthy-subaccount case, the owner is forced to pay withdrawal fees and go through the protocol's withdrawal queue to recover tokens they never intended to deposit.
- **Griefing at zero cost**: The attacker spends only gas. No capital is required. The attack is repeatable every time the DDA accumulates a non-zero balance.

---

### Likelihood Explanation

- `DirectDepositV1` is a production contract deployed per-subaccount. Its address is deterministic and publicly observable on-chain.
- Token balances held by the DDA are visible via standard ERC-20 `balanceOf` queries.
- The attack requires only a single `creditDeposit()` call with no preconditions beyond gas.
- Front-running is straightforward on EVM chains where the mempool is public.
- The scenario where the owner wants to recover a mistakenly sent token (or delay a deposit) is realistic in production operation.

---

### Recommendation

Add an `onlyOwner` modifier to `creditDeposit()`, or introduce a separate permissioned role, so that only the contract owner can trigger the deposit sweep:

```solidity
function creditDeposit() external onlyOwner {
    ...
}
```

This preserves the censorship-resistance intent (the owner can always trigger deposits) while preventing unauthorized callers from forcing the DDA's funds into the protocol.

### Proof of Concept

```
Setup:
  - DDA deployed for subaccount S, owner = Alice
  - 1000 USDC sent to DDA by mistake; Alice wants to recover via withdraw()

Attack:
  1. Alice broadcasts: DDA.withdraw(USDC)
  2. Attacker sees tx in mempool, broadcasts with higher gas:
       DDA.creditDeposit()
  3. creditDeposit() executes first:
       - USDC.approve(endpoint, 1000e6)
       - endpoint.depositCollateralWithReferral(S, QUOTE_PRODUCT_ID, 1000e6, "-1")
       → 1000 USDC credited to subaccount S inside the protocol
  4. Alice's withdraw() executes:
       - USDC.balanceOf(DDA) == 0
       - safeTransfer(Alice, 0) — no tokens transferred
  5. Alice must now call WithdrawCollateral through the protocol,
     paying withdrawFeeX18 and subject to health checks on subaccount S.
  6. If subaccount S is under maintenance margin, withdrawal reverts:
       require(getHealth(sender, INITIAL) >= 0, ERR_SUBACCT_HEALTH)
     → 1000 USDC locked until S recovers health.
``` [1](#0-0) [5](#0-4)

    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
