### Title
SwapAllowlistExtension checks the router address instead of the actual user, allowing complete allowlist bypass through MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender`. When users swap through `MetricOmmSimpleRouter`, the router is the pool's `msg.sender`, so `sender = router`. The actual user's address is passed only as `recipient` and is silently ignored by the extension. This wrong-actor binding means: (a) if the admin allowlists the router to enable router-mediated swaps, every user on the network bypasses the allowlist; (b) if the admin does not allowlist the router, individually allowlisted users cannot use the standard periphery path at all.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every before-swap hook:

```solidity
// MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then gates on that `sender` value:

```solidity
// SwapAllowlistExtension.sol
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [2](#0-1) 

When a user routes through `MetricOmmSimpleRouter`, the call chain is:

```
user → router.exactInput*() → pool.swap(recipient=user, ...) → extension.beforeSwap(sender=router, recipient=user, ...)
```

The extension receives `sender = router` and `recipient = user`. It checks `allowedSwapper[pool][router]` and completely ignores the actual user. The `recipient` parameter — the only field that carries the real user's address — is the unnamed second argument and is never read. [2](#0-1) 

The allowlist storage is keyed `allowedSwapper[pool][swapper]` and is set only by the pool admin: [3](#0-2) 

This creates two mutually exclusive failure modes:

1. **Admin allowlists individual users, not the router.** Allowlisted users who call through the router are blocked (`allowedSwapper[pool][router] == false`). The standard periphery path is broken for every curated pool.

2. **Admin allowlists the router to fix mode 1.** `allowedSwapper[pool][router] == true` passes for every caller regardless of identity. Any non-allowlisted user bypasses the curated pool's access control by routing through the public router.

---

### Impact Explanation

Under failure mode 2 — the operationally natural fix — the swap allowlist is completely defeated. Any address on the network can execute swaps on a pool that was deployed specifically to restrict swap access. LP funds in the pool are exposed to actors the pool admin explicitly intended to exclude. This is a direct broken-invariant on the allowlist path with fund-impacting consequences (unauthorized swaps drain LP-owned assets at oracle prices).

---

### Likelihood Explanation

Medium. The admin must allowlist the router address. This is a natural operational step: a pool admin who wants allowlisted users to access the pool through the standard periphery router will add the router to the allowlist. The admin has no way to know this grants universal bypass, because the extension's documentation and interface give no indication that `sender` is the router rather than the user. The trigger requires no attacker privilege — any public address can call the router.

---

### Recommendation

The extension must gate the ultimate economic actor, not the intermediary. Two sound approaches:

1. **Check `recipient` for swap allowlisting.** `recipient` is the address that receives output tokens and is the closest proxy for the real user available in the current interface. Replace `sender` with the second (currently unnamed) parameter in `beforeSwap`.

2. **Decode user identity from `extensionData`.** The router encodes the original caller in `extensionData` before forwarding to the pool. The extension decodes and checks that address. This requires a coordinated change in the router and extension but preserves the full operator pattern.

Either way, the extension must not treat the router address as the identity to gate.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a before-swap hook.
2. Admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for allowlisted users.
3. Non-allowlisted attacker calls `router.exactInput(pool, tokenIn, amountIn, ...)`.
4. Router calls `pool.swap(recipient=attacker, ...)`.
5. Pool calls `extension.beforeSwap(sender=router, recipient=attacker, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes against LP liquidity on a pool that was supposed to block them.

The attacker needed zero allowlist approval. The only precondition is that the admin performed the natural operational step of allowlisting the router. [4](#0-3) [1](#0-0)
