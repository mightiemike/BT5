### Title
`burnNlp` Uses `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Collateral Extraction from Undercollateralized Subaccounts — (`core/contracts/Clearinghouse.sol`)

### Summary

`burnNlp` is a withdrawal-equivalent operation: the caller surrenders NLP tokens and receives quote collateral in return. Every other withdrawal-equivalent function in `Clearinghouse.sol` enforces a post-operation `HealthType.INITIAL` check. `burnNlp` enforces only `HealthType.MAINTENANCE`, allowing a subaccount that is already below the INITIAL threshold to extract quote collateral and deepen its undercollateralization.

### Finding Description

`burnNlp` removes NLP from the sender's balance and credits quote to the sender: [1](#0-0) 

The sole post-operation health guard is: [2](#0-1) 

This is `HealthType.MAINTENANCE`, not `HealthType.INITIAL`. Compare with every other withdrawal-equivalent path in the same contract:

- `withdrawCollateral` explicitly selects `HealthType.INITIAL` for all non-X_ACCOUNT senders: [3](#0-2) 

- `mintNlp` (the inverse operation, a deposit) uses `HealthType.INITIAL`: [4](#0-3) 

- `nlpProfitShare` uses `HealthType.INITIAL`: [5](#0-4) 

- `forceRebalanceNlpPool` uses `HealthType.INITIAL`: [6](#0-5) 

The `INITIAL` weight is stricter than `MAINTENANCE` (lower long-weight, higher short-weight per `RiskHelper._getWeightX18`): [7](#0-6) 

A subaccount can therefore satisfy `MAINTENANCE >= 0` while `INITIAL < 0`. The `burnNlp` check passes in that zone, but `withdrawCollateral` would revert.

### Impact Explanation

A user whose subaccount sits in the band `MAINTENANCE >= 0, INITIAL < 0` can call `burnNlp` to convert NLP holdings into quote collateral. The quote is credited to the subaccount at line 515: [8](#0-7) 

After the burn the subaccount's INITIAL health remains negative (or becomes more negative if NLP's initial weight exceeds quote's initial weight). The subaccount is now more deeply undercollateralized relative to its open positions. If those positions move adversely before liquidation, the shortfall exceeds the insurance fund's coverage, creating bad debt absorbed by the protocol.

### Likelihood Explanation

The path is fully permissionless. Any user who holds NLP tokens in a subaccount that has drifted below INITIAL health (e.g., due to adverse price moves on leveraged perp positions) can submit a signed `BurnNlp` transaction through the normal sequencer flow: [9](#0-8) 

No special privileges, governance actions, or sequencer compromise are required. The only precondition is holding unlocked NLP in a subaccount with `MAINTENANCE >= 0` and `INITIAL < 0`, which is a naturally occurring state during market stress.

### Recommendation

Replace `HealthType.MAINTENANCE` with `HealthType.INITIAL` at line 527 of `Clearinghouse.sol`:

```solidity
// Before (vulnerable)
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// After (correct)
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This aligns `burnNlp` with every other withdrawal-equivalent function in the contract.

### Proof of Concept

1. Deploy the protocol on a local Hardhat fork.
2. Create a subaccount; deposit quote collateral and open a leveraged perp position.
3. Mint NLP tokens into the same subaccount.
4. Move the oracle price so that `getHealth(sender, INITIAL) < 0` and `getHealth(sender, MAINTENANCE) >= 0`.
5. Submit a signed `BurnNlp` transaction for the full unlocked NLP balance.
6. Observe the transaction succeeds (MAINTENANCE check passes).
7. Assert `getHealth(sender, INITIAL) < 0` — the subaccount is still below INITIAL health after receiving quote.
8. Assert the quote balance of the subaccount increased by `quoteAmount` (line 515), confirming collateral was extracted from an undercollateralized position.
