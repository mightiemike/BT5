No Slippage Protection for `mintNlp`/`burnNlp` — `oraclePriceX18` Excluded from User Signature — (`core/contracts/Clearinghouse.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `mintNlp` and `burnNlp` functions in `Clearinghouse.sol` compute the user's output amount entirely from an `oraclePriceX18` value that is **supplied by the sequencer at execution time and is not committed to in the user's EIP-712 signature**. The user signs only `sender`, `quoteAmount`/`nlpAmount`, and `nonce`. There is no on-chain minimum-output guard. The user cannot specify an acceptable price bound, leaving them fully exposed to receiving fewer tokens than expected.

---

### Finding Description

The EIP-712 type strings for `MintNlp` and `BurnNlp` are defined as:

```
"MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)"
"BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)"
``` [1](#0-0) 

The corresponding `SignedMintNlp` and `SignedBurnNlp` wrapper structs carry an additional `oraclePriceX18` field that is **outside** the signed inner struct:

```solidity
struct SignedMintNlp {
    MintNlp tx;          // ← only this is signed
    bytes signature;
    int128 oraclePriceX18;       // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}
``` [2](#0-1) 

The `computeDigest` function in `Verifier.sol` confirms that only `sender`, `quoteAmount`/`nlpAmount`, and `nonce` are hashed into the digest: [3](#0-2) 

At execution time in `EndpointTx.sol`, the sequencer-supplied `signedTx.oraclePriceX18` is passed directly to `clearinghouse.mintNlp`/`burnNlp` without any user-side bound check: [4](#0-3) 

Inside `mintNlp`, the NLP output is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [5](#0-4) 

Inside `burnNlp`, the quote output is computed as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [6](#0-5) 

Neither function checks a user-specified minimum output. The user's signed message commits to the input amount but places no bound on the output amount.

---

### Impact Explanation

**`mintNlp`:** A user signs a transaction spending `quoteAmount` of quote tokens. If `oraclePriceX18` is set higher than the price the user observed when signing, `nlpAmount = quoteAmount / oraclePriceX18` is smaller — the user receives fewer NLP tokens for the same cost, with no recourse.

**`burnNlp`:** A user signs a transaction burning `nlpAmount` of NLP. If `oraclePriceX18` is set lower than the price the user observed when signing, `quoteAmount = nlpAmount * oraclePriceX18` is smaller — the user receives fewer quote tokens for the same NLP burned, with no recourse.

In both cases the user suffers a direct asset loss relative to their expectation at signing time, with no on-chain mechanism to reject the execution.

---

### Likelihood Explanation

The oracle price is legitimately volatile. Because Nado uses a sequencer-driven batching model, there is an inherent delay between when a user signs a `MintNlp`/`BurnNlp` transaction and when the sequencer includes it in an on-chain batch. During that window the NLP oracle price can move materially. Since `oraclePriceX18` is not part of the signed message, the user has no way to enforce that the price used at settlement matches the price they observed. Any price movement — benign or otherwise — between signing and settlement is silently absorbed by the user with no protection.

---

### Recommendation

Add a user-specified minimum output field to the `MintNlp` and `BurnNlp` structs and include it in the EIP-712 signed digest:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64  nonce;
    uint128 minNlpAmount;   // ← new: minimum NLP tokens to receive
}

struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64  nonce;
    uint128 minQuoteAmount; // ← new: minimum quote tokens to receive
}
```

Update the EIP-712 type strings in `Verifier.sol` to include the new fields, and add revert guards in `Clearinghouse.mintNlp`/`burnNlp`:

```solidity
// mintNlp
require(nlpAmount >= int128(txn.minNlpAmount), ERR_SLIPPAGE);

// burnNlp
require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE);
```

---

### Proof of Concept

1. User observes NLP oracle price = 1.00 USDC and signs `MintNlp { quoteAmount: 1000e18, nonce: N }`, expecting to receive 1000 NLP.
2. Before the sequencer batches the transaction, the oracle price moves to 2.00 USDC.
3. Sequencer submits the batch with `oraclePriceX18 = 2e18`.
4. `mintNlp` computes `nlpAmount = 1000e18 / 2e18 = 500`. The user's signature is valid (it only covers `quoteAmount` and `nonce`), so the transaction succeeds.
5. User pays 1000 USDC but receives only 500 NLP — a 50% shortfall — with no on-chain protection.

The symmetric case for `burnNlp`: user signs to burn 1000 NLP expecting 1000 USDC, oracle price drops to 0.50, user receives only 500 USDC minus the burn fee. [7](#0-6) [8](#0-7)

        int128 quoteAmount = int128(txn.quoteAmount);
        int128 nlpAmount = quoteAmount.div(oraclePriceX18);

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
