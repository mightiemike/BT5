### Title
Pausing `submit_blocks` Does Not Pause `verify_transaction_inclusion`, Leaving Malicious Headers Exploitable - (File: `contract/src/lib.rs`)

### Summary
The `BtcLightClient` contract uses `near_plugins`' per-function `#[pause]` macro, which creates an independent pausable feature for each annotated function. When an admin pauses `submit_blocks` to halt malicious header ingestion, the `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions retain their own separate, unpaused feature states. Any adversarial headers already committed to the chain remain fully usable for forging transaction inclusion proofs, making the pause action incomplete and ineffective against already-written malicious state.

### Finding Description

The contract derives `Pausable` from `near_plugins` and annotates three public functions with `#[pause]`:

- `submit_blocks` — the write path for ingesting Bitcoin headers
- `verify_transaction_inclusion` — the SPV proof consumer (deprecated)
- `verify_transaction_inclusion_v2` — the current SPV proof consumer [1](#0-0) [2](#0-1) [3](#0-2) 

The `near_plugins` `#[pause]` macro creates a **per-function** named feature (the feature name equals the function name). Calling `pa_pause("submit_blocks")` only blocks `submit_blocks`; it has no effect on the `"verify_transaction_inclusion"` or `"verify_transaction_inclusion_v2"` features. This is structurally confirmed by the existence of separate per-function bypass roles: [4](#0-3) 

`UnrestrictedSubmitBlocks` bypasses only `submit_blocks`; `UnrestrictedRunGC` bypasses only `run_mainchain_gc`. There is no unified global pause that covers all three functions simultaneously.

**Exploit flow:**

1. A malicious trusted relayer (valid entry point per scope: "relayer-path user supplying adversarial chain data") calls `submit_blocks` with a crafted chain of headers whose `merkle_root` fields are attacker-controlled.
2. The adversarial headers pass PoW validation (or `skip_pow_verification` is `true`) and are stored in `headers_pool` and `mainchain_height_to_header`.
3. The admin detects the attack and calls `pa_pause("submit_blocks")` to halt further ingestion.
4. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are **not paused** — they continue to execute against the now-corrupt chain state.
5. The attacker calls `verify_transaction_inclusion_v2` with a `tx_block_blockhash` pointing to a malicious header and a Merkle proof crafted to match the attacker-controlled `merkle_root`.
6. The function returns `true`, falsely certifying that an arbitrary transaction is included in a Bitcoin block. [5](#0-4) [6](#0-5) 

### Impact Explanation

Any consumer contract or bridge that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to gate asset releases, cross-chain swaps, or settlement logic will receive a fraudulent `true` result. The admin's pause action — intended to lock down the contract — provides no protection against the malicious headers already committed to `headers_pool`. The corrupted canonical chain mapping (`mainchain_height_to_header`, `mainchain_header_to_height`) remains intact and queryable, and the forged proof result is indistinguishable from a legitimate one. [7](#0-6) 

### Likelihood Explanation

The precondition is a compromised or malicious trusted relayer, which is explicitly listed as a valid attacker entry point in the audit scope ("relayer-path user supplying adversarial chain data"). Once a relayer is malicious, the window between header submission and admin detection is the only constraint. Given that the admin must manually identify the attack, call `pa_pause`, and then separately realize they must also pause the verification functions, the probability of the verification path remaining open during the response window is high. The admin has no single "emergency stop" that covers all three functions atomically.

### Recommendation

Add a single atomic emergency-pause path that covers both the write path (`submit_blocks`) and the read/consume path (`verify_transaction_inclusion`, `verify_transaction_inclusion_v2`) simultaneously. The simplest fix is to check the `submit_blocks` pause feature inside the verification functions as well, or introduce a shared global pause feature name used by all three `#[pause]` annotations:

```rust
// Option A: shared feature name
#[pause(name = "btc_light_client")]
pub fn submit_blocks(...) { ... }

#[pause(name = "btc_light_client")]
pub fn verify_transaction_inclusion(...) { ... }

#[pause(name = "btc_light_client")]
pub fn verify_transaction_inclusion_v2(...) { ... }
```

This ensures that a single `pa_pause("btc_light_client")` call atomically blocks both ingestion and consumption, matching the intended semantics of an emergency lockdown.

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = false` and a legitimate genesis.
2. Grant a test account the trusted relayer role.
3. From the relayer account, call `submit_blocks` with a batch of headers where `merkle_root` is set to `SHA256d(attacker_tx_id || attacker_tx_id)` (a single-leaf tree).
4. From the admin account, call `pa_pause("submit_blocks")`.
5. Confirm `submit_blocks` is now blocked.
6. From any unprivileged account, call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of the malicious header
   - `tx_id` = `attacker_tx_id`
   - `coinbase_tx_id` = `attacker_tx_id`
   - `merkle_proof` = `[]` (empty, single-leaf tree)
   - `coinbase_merkle_proof` = `[]`
   - `confirmations` = 1
7. The call succeeds and returns `true`, proving a fake transaction is certified as included despite the contract being "paused." [8](#0-7)
