Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and gates access on `owner` — a free caller-controlled argument specifying who receives the LP position. Because `owner` is arbitrary and `addLiquidity` imposes no `msg.sender == owner` constraint, any unauthorized address can bypass the allowlist by naming any already-allowlisted address as `owner`. The allowlisted address receives unwanted LP shares while the unauthorized depositor's tokens are pulled from them, rendering the allowlist completely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` into the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both parameters:

```solidity
// ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as unnamed, discarding it entirely, and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner`, `addLiquidity` imposes no such constraint:

```solidity
// MetricOmmPool.sol L206 — only removeLiquidity enforces this
if (msg.sender != owner) revert NotPositionOwner();
``` [4](#0-3) 

**Exploit path:**
1. Pool admin deploys pool with `DepositAllowlistExtension` and allowlists `alice`: `setAllowedToDeposit(pool, alice, true)`.
2. Unauthorized `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. The hook receives `sender = bob` (unnamed, discarded), `owner = alice`; it checks `allowedDepositor[pool][alice]` → `true`.
4. The check passes. `bob`'s tokens are pulled via the swap callback and deposited; LP shares are credited to `alice`.
5. `bob` has fully bypassed the deposit restriction. The wrong value is the extension decision: `allowedDepositor[pool][alice]` is evaluated instead of `allowedDepositor[pool][bob]`.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism to restrict who can provide liquidity (e.g., for regulatory compliance, curated LP sets, or preventing adversarial liquidity). Because the guard checks `owner` (recipient of LP shares) rather than `sender` (the actual depositor), the restriction is completely ineffective. Any address can deposit into a restricted pool by naming any allowlisted address as `owner`. The LP shares land with the allowlisted address — an unwanted, unsolicited position that exposes them to impermanent loss and oracle-price risk — while the unauthorized depositor circumvents the intended access control. This is a broken core pool invariant: the allowlist does not gate the actor it is designed to gate.

## Likelihood Explanation

High. The bypass requires a single standard `addLiquidity` call with `owner` set to any known allowlisted address. No special privileges, flash loans, or complex setup are needed. The allowlisted address is often discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. [5](#0-4) 

## Recommendation

Name and check `sender` instead of `owner` in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the actual depositor — not the LP position recipient — is the entity checked against the allowlist.

## Proof of Concept

```solidity
// Setup
pool.setAllowedToDeposit(poolAddr, alice, true);   // only alice is allowed

// Attack — bob is NOT allowlisted
vm.startPrank(bob);
token0.approve(address(pool), type(uint256).max);
token1.approve(address(pool), type(uint256).max);

// bob sets owner = alice; extension checks allowedDepositor[pool][alice] → true → passes
pool.addLiquidity(
    alice,          // owner: allowlisted address
    0,              // salt
    deltas,         // liquidity to add
    callbackData,
    extensionData
);
// Result: bob's tokens deposited, alice receives LP shares, allowlist bypassed
vm.stopPrank();
```

The `beforeAddLiquidity` hook passes because `allowedDepositor[pool][alice]` is `true`, even though `bob` (the actual depositor, passed as the unnamed first argument) is not allowlisted. [6](#0-5)
