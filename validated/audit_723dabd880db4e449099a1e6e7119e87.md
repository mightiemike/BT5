Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual swapper, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool` sets to `msg.sender` — the immediate caller of `pool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the originating user. Any pool admin who allowlists the router (the only way to enable router-mediated swaps) inadvertently grants every user on earth unrestricted swap access, completely defeating the allowlist.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` verbatim as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the `sender` field forwarded to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value above: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` with no forwarding of the original caller — the pool sees `msg.sender = router`: [4](#0-3) 

The same applies to `exactInput` (L103–112), `exactOutputSingle` (L135–137), and `exactOutput` (L165–181). In every case the pool receives `msg.sender = router`, so the extension evaluates `allowedSwapper[pool][router]` — the actual user's address is never consulted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly ignores the intermediary `sender` and checks `owner` (the LP position owner), the economically relevant actor: [5](#0-4) 

`SwapAllowlistExtension` has no equivalent correction.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., KYC-gated or institutional pools). Once the pool admin allowlists the router — the only way to make router-mediated swaps function — the allowlist is effectively disabled for all users. Any unprivileged address can call `exactInputSingle` (or any other router entry point) and execute a swap that the allowlist was supposed to block. This is a direct policy bypass with fund-impacting consequences: unauthorized parties can trade against restricted pools, violating the protocol invariant that only approved addresses may swap.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical periphery swap contract. Any pool operator who deploys a `SwapAllowlistExtension`-guarded pool and also wants to support the standard router must allowlist the router — this is the expected operational path. The bypass requires no special privileges, no malicious setup, and no non-standard token behavior. Any unprivileged user can exploit it with a single `exactInputSingle` call.

## Recommendation

The extension must check the identity of the economic actor, not the intermediary. Two viable approaches:

1. **Mirror `DepositAllowlistExtension`**: have the pool or router pass the original `msg.sender` as a dedicated field (e.g., the first word of `extensionData`). The extension decodes the real caller from `extensionData` when `sender` is a known router, and falls back to `sender` otherwise.

2. **Router-embedded caller**: require the router to encode `msg.sender` into `extensionData` before calling `pool.swap`, and have `SwapAllowlistExtension` decode and check that value when present.

Either way, the allowlist mapping key must resolve to the address that controls the economic decision to swap, not the address of the contract that mechanically forwards the call.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice]   = true   // intended gated user
  allowedSwapper[pool][router]  = true   // admin must set this for router swaps
  allowedSwapper[pool][attacker]= false  // attacker NOT on allowlist

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({
      pool:      pool,
      recipient: attacker,
      tokenIn:   token0,
      amountIn:  X,
      ...
  })

  router calls pool.swap(attacker_recipient, ...)
    pool: msg.sender = router
    pool calls _beforeSwap(sender=router, ...)
    extension checks allowedSwapper[pool][router] == true  ✓
    swap executes — attacker receives output tokens

Result:
  attacker bypassed the per-user allowlist entirely.
  The check evaluated the router's allowlist entry instead of attacker's.
```

A Foundry integration test can confirm this by deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the router, then calling `exactInputSingle` from an unlisted `attacker` address and asserting the swap succeeds (no `NotAllowedToSwap` revert).
