### Title
Repeated Margin Transfer to Same Isolated Subaccount via Distinct Digests — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` guards against exact digest replay but does **not** guard against multiple distinct digests (different order parameters) that all resolve to the same existing isolated subaccount for a given `productId`. Each new digest triggers an unconditional margin transfer from the parent to the isolated subaccount, allowing the parent's quote balance to be drained across N transactions.

---

### Finding Description

The guard at line 1009 only short-circuits if the **exact same digest** has already been processed:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];
}
``` [1](#0-0) 

After that check, the function loops over `isolatedSubaccountsMask` to find an existing isolated subaccount for the same `productId`. If one is found, it is reused:

```solidity
if (productId == txn.productId) {
    newIsolatedSubaccount = subaccount;
    break;
}
``` [2](#0-1) 

Regardless of whether `newIsolatedSubaccount` was freshly created or reused from the loop, the margin transfer block executes unconditionally for every new digest:

```solidity
digestToSubaccount[digest] = newIsolatedSubaccount;
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
}
``` [3](#0-2) 

`SpotEngine.updateBalance` (the single-delta overload) performs no floor check — it calls `_updateBalanceNormalized` directly, and the protocol's state struct explicitly tracks `totalBorrowsNormalized`, confirming negative balances are permitted: [4](#0-3) 

The `CreateIsolatedSubaccount` transaction type is dispatched from `processTransactionImpl` with **no** `validateSignedTx` / nonce check — only the per-order EIP-712 signature is verified inside `createIsolatedSubaccount` itself: [5](#0-4) 

---

### Impact Explanation

An attacker who controls a parent subaccount can sign N orders for the same `productId` with distinct parameters (different nonce, price, or amount → distinct EIP-712 digest). The sequencer processes each as a valid `CreateIsolatedSubaccount` transaction. For calls 2…N the existing isolated subaccount is reused, but `margin` is still debited from the parent and credited to the isolated subaccount each time. Because `updateBalance` allows negative balances, the parent can be driven below zero — effectively creating unbacked credit in the isolated subaccount. The isolated subaccount accumulates N × margin while the parent accrues N × margin of debt, violating the invariant that isolated margin must be bounded by the parent's available collateral.

---

### Likelihood Explanation

The attacker only needs to submit N legitimately signed orders with distinct parameters for the same `productId`. No sequencer compromise is required; the sequencer has no on-chain or off-chain reason to reject valid signed orders. The slow-mode path (`submitSlowModeTransaction`) is also available as an alternative entry point with no additional gating for this transaction type. [6](#0-5) 

---

### Recommendation

Before executing the margin transfer, add an explicit check that no isolated subaccount for `txn.productId` already exists for the sender. If one does exist, the function should either revert or return the existing subaccount without transferring margin:

```solidity
if (newIsolatedSubaccount != bytes32(0)) {
    // Subaccount already exists for this productId; record digest mapping only, no margin transfer
    digestToSubaccount[digest] = newIsolatedSubaccount;
    return newIsolatedSubaccount;
}
```

This ensures margin is transferred at most once per isolated subaccount, regardless of how many distinct digests reference the same `productId`.

---

### Proof of Concept

1. Parent subaccount `P` has 5000 USDC quote balance.
2. Attacker signs 5 orders for `productId = 2`, each with `margin = 1000e6` in the appendix, but with distinct nonces → 5 distinct EIP-712 digests `d1…d5`.
3. Sequencer submits 5 `CreateIsolatedSubaccount` transactions.
4. Call 1 (`d1`): `digestToSubaccount[d1] == 0` → loop finds no existing subaccount → creates `ISO`. Transfers 1000 USDC: P = 4000, ISO = 1000.
5. Call 2 (`d2`): `digestToSubaccount[d2] == 0` → loop finds ISO (same `productId`) → `newIsolatedSubaccount = ISO`. Transfers 1000 USDC: P = 3000, ISO = 2000.
6. Calls 3–5: same pattern. Final state: P = 0, ISO = 5000.
7. If margin per call exceeds P's balance, `updateBalance` drives P negative (e.g., 5 calls × 2000 USDC with P starting at 5000 → P = −5000, ISO = 10000), creating 5000 USDC of unbacked credit. [1](#0-0) [3](#0-2)
