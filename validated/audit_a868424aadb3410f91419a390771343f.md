### Title
Stale `lastLiquidationFees` Causes Insurance Fund to Bypass Coverage for Subsequent Liquidatee — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a **global** storage variable that is written by `_handleLiquidationPayment` but **never reset** after `_finalizeSubaccount` completes. When `_finalizeSubaccount` is called for a new liquidatee B whose finalization is the first action in its liquidation sequence (i.e., no `_handleLiquidationPayment` was called for B), the stale value from a prior liquidation of subaccount A is used to reduce `v.insurance` during B's finalization. This causes the insurance fund to provide zero coverage for B's negative quote balance, forcing socialization of losses onto all spot holders, while the insurance fund incorrectly retains its full balance.

---

### Finding Description

`lastLiquidationFees` is declared as a global state variable in `ClearinghouseStorage`: [1](#0-0) 

It is written unconditionally at the end of every `_handleLiquidationPayment` call: [2](#0-1) 

The design intent (per the inline comment) is to prevent the fees just added to insurance from being immediately consumed to cover the same liquidatee's deficit. The subtract/add-back pattern in `_finalizeSubaccount` implements this: [3](#0-2) 

The critical flaw: `_finalizeSubaccount` **never resets `lastLiquidationFees` to zero** after it completes. The variable retains the value from the last `_handleLiquidationPayment` call regardless of which liquidatee it belonged to.

`liquidateSubaccountImpl` calls `_finalizeSubaccount` **before** any `_handleLiquidationPayment` call: [4](#0-3) 

So when a liquidator calls `liquidateSubaccountImpl(productId=uint32.max)` on a new liquidatee B (whose positions are already zero, requiring only quote-balance cleanup), `_finalizeSubaccount` runs with the stale `lastLiquidationFees` from A's prior liquidation.

---

### Impact Explanation

**Concrete numeric trace:**

| Step | Variable | Value |
|------|----------|-------|
| After liquidating A (last step) | `insurance` | 500 |
| After liquidating A (last step) | `lastLiquidationFees` | 500 |
| After finalizing A | `insurance` | 500 (A's -600 was socialized; fees added back) |
| After finalizing A | `lastLiquidationFees` | **500 (never reset)** |
| Finalize B (`quoteBalance = -300`) | `v.insurance = 500 - 500` | **0** |
| | `insuranceCover = min(0, 300)` | **0** (no coverage!) |
| | `spotEngine.socializeSubaccount(B)` | called — B's -300 socialized |
| | `v.insurance += 500` | 500 |
| | `insurance = 500` | **500 (unchanged, should be 200)** |

**Correct behavior** (if `lastLiquidationFees` were reset to 0 after A's finalization):
- `v.insurance = 500`, `insuranceCover = min(500, 300) = 300`
- B's -300 USDC covered by insurance; `insurance` correctly decreases to 200

**Actual behavior:**
- B's -300 USDC is socialized across all spot holders via `spotEngine.socializeSubaccount`
- Insurance fund retains 500 USDC it should have spent
- All spot depositors suffer dilution of their `cumulativeDepositsMultiplierX18` [5](#0-4) 

This is a direct **balance bypass**: the insurance fund balance is not consumed when protocol invariants require it to be, and losses are incorrectly pushed onto innocent depositors.

---

### Likelihood Explanation

The preconditions are realistic in any stressed market:

1. A liquidation sequence on subaccount A ends with `lastLiquidationFees` set to a value ≥ current `insurance` (possible when insurance is nearly depleted covering A's losses).
2. Subaccount B exists with only a negative USDC balance and zero positions — a common end-state for accounts that borrowed USDC and had all collateral liquidated.
3. The liquidator (or anyone) calls `liquidateSubaccountImpl(productId=uint32.max)` on B. No special privileges required; `liquidateSubaccountImpl` is externally callable through the endpoint.

The sequencer processes transactions in order, so the attacker only needs to submit two sequential liquidation transactions. No reentrancy or callback is required.

---

### Recommendation

Reset `lastLiquidationFees` to zero at the end of `_finalizeSubaccount`, after the add-back and before writing `insurance`:

```solidity
v.insurance += lastLiquidationFees;
lastLiquidationFees = 0;   // ← add this
insurance = v.insurance;
```

This ensures that a finalization for one liquidatee cannot pollute the fee accounting for a subsequent, unrelated liquidatee.

---

### Proof of Concept

```
1. Deploy protocol with insurance = 500.
2. Liquidate subaccount A (perp position):
   - _handleLiquidationPayment sets insurance = 500, lastLiquidationFees = 500.
3. Finalize subaccount A (productId = uint32.max):
   - v.insurance = 500 - 500 = 0; insuranceCover = 0; socializeSubaccount(A) called.
   - v.insurance += 500; insurance = 500. lastLiquidationFees still = 500.
4. Subaccount B has quoteBalance = -300, all other positions = 0, under maintenance health.
5. Call liquidateSubaccountImpl(productId=uint32.max) for B:
   - _finalizeSubaccount: v.insurance = 500 - 500 = 0.
   - insuranceCover = min(0, 300) = 0. No coverage applied.
   - v.insurance <= 0 → spotEngine.socializeSubaccount(B) called.
   - v.insurance += 500; insurance = 500.
6. Assert: insurance == 500 (should be 200). B's -300 was socialized, not covered.
``` [6](#0-5) [2](#0-1) [7](#0-6)

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
