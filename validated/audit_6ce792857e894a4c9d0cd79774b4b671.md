### Title
`only_query` Hardcoded to `false` in `BroadcastedTransaction` → `ExecutableTransactionInput` Conversion, Causing Wrong Transaction Version in RPC Simulation and Fee Estimation - (File: `crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion unconditionally passes `only_query = false` for `Invoke` and `DeployAccount` transactions, even when the submitted transaction carries a query-version (bit 128 set, e.g. `0x100000000000000000000000000000003`). This conversion is the sole entry point for both `starknet_estimateFee` and `starknet_simulateTransactions`. As a result, the transaction version reported to contracts via the `get_execution_info` syscall during simulation and fee estimation is missing the query bit, causing any contract that branches on the query bit to execute the wrong code path and return an authoritative-looking wrong value to the caller.

---

### Finding Description

The Starknet protocol defines a "query version" for transactions submitted to simulation/estimation endpoints: the 128th bit of the transaction version is set (e.g. `0x100000000000000000000000000000003` for a V3 query transaction). The RPC type system already models this: `TransactionVersion1::Version1OnlyQuery`, `TransactionVersion2::Version2OnlyQuery`, etc. are defined in `crates/apollo_rpc/src/v0_8/transaction.rs`. [1](#0-0) 

The conversion from `BroadcastedTransaction` to `ExecutableTransactionInput` — the type consumed by both `estimate_fee` and `simulate_transactions` — ignores this bit entirely and hardcodes `false`: [2](#0-1) 

The TODO comment in the code itself acknowledges the defect: `// TODO(yair): pass the right value for only_query field.`

Both RPC handlers call this conversion without any override: [3](#0-2) [4](#0-3) 

The `only_query` value flows from `ExecutableTransactionInput` into `ExecutionFlags` inside `to_blockifier_tx`: [5](#0-4) 

`ExecutionFlags.only_query` is then stored in `CommonAccountFields.only_query` via `create_tx_info`, and `TransactionInfo::signed_version()` uses it to set bit 128: [6](#0-5) [7](#0-6) 

The `signed_version()` result is what `tx_version_for_get_execution_info` returns to the contract via the `get_execution_info` syscall: [8](#0-7) 

The blockifier test suite explicitly documents that `only_query` changes the version seen by the contract: [9](#0-8) 

---

### Impact Explanation

Any contract that inspects `tx_info.version` inside `get_execution_info` and branches on the query bit (e.g. to skip state-mutating side effects during simulation, or to return a dry-run-specific result) will receive version `0x3` instead of `0x100000000000000000000000000000003`. The simulation or fee estimation result returned to the caller is therefore wrong: it reflects the non-query execution path. This is a **High** impact: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

The trigger is unprivileged: any user can call `starknet_estimateFee` or `starknet_simulateTransactions` with a transaction whose version field carries the query bit. The RPC type system already accepts these versions. No special permissions or malformed bytes are required. The defect is present in every call to these two endpoints for `Invoke` and `DeployAccount` transactions.

---

### Recommendation

In `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput`, derive `only_query` from the transaction version rather than hardcoding `false`. For each transaction variant, inspect whether the submitted version is an `OnlyQuery` variant and pass the corresponding boolean:

```rust
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => {
                let only_query = tx.is_query(); // derive from version field
                Ok(Self::DeployAccount(tx.into(), only_query))
            }
            BroadcastedTransaction::Invoke(tx) => {
                let only_query = tx.is_query(); // derive from version field
                Ok(Self::Invoke(tx.into(), only_query))
            }
        }
    }
}
```

Add a negative test that submits a query-version transaction to `simulate_transactions` and asserts that the version seen by the contract inside `get_execution_info` has bit 128 set.

---

### Proof of Concept

1. Deploy a contract whose `execute` entry point reads `tx_info.version` via `get_execution_info` and reverts if the query bit is **not** set.
2. Call `starknet_simulateTransactions` with an `Invoke` transaction whose `version` is `0x100000000000000000000000000000003`.
3. Observe that the simulation **reverts** — because `only_query` is hardcoded to `false`, the version delivered to the contract is `0x3`, the query bit is absent, and the contract's guard triggers.
4. The caller receives a revert result from a transaction that, by the protocol's own definition, should succeed in query mode — a wrong authoritative value from the RPC simulation endpoint.

pub fn signed_tx_version(
    tx_version: &TransactionVersion,
    transaction_options: &TransactionOptions,
) -> TransactionVersion {
    // If only_query is true, set the 128-th bit.
    let query_only_bit = *QUERY_VERSION_BASE;
    assert_eq!(
        tx_version.0.to_biguint() & query_only_bit.to_biguint(),
        BigUint::from(0_u8),
        "Requested signed tx version with version that already has query bit set: {tx_version:?}."
    );
    if transaction_options.only_query {
        TransactionVersion(tx_version.0 + query_only_bit)
    } else {
        *tx_version
    }
}

    /// Returns the transaction version for the `get_execution_info` syscall.
    pub fn tx_version_for_get_execution_info(&self) -> TransactionVersion {
        let tx_context = &self.context.tx_context;
        // The transaction version, ignoring the only_query bit.
        let version = tx_context.tx_info.version();
        let versioned_constants = &tx_context.block_context.versioned_constants;
        // The set of v1-bound-accounts.
        let v1_bound_accounts = &versioned_constants.os_constants.v1_bound_accounts_cairo1;
        let class_hash = &self.call.class_hash;

        // If the transaction version is 3 and the account is in the v1-bound-accounts set,
        // the syscall should return transaction version 1 instead.
        if version == TransactionVersion::THREE && v1_bound_accounts.contains(class_hash) {
            let tip = match &tx_context.tx_info {
                TransactionInfo::Current(transaction_info) => transaction_info.tip,
                TransactionInfo::Deprecated(_) => {
                    panic!("Transaction info variant doesn't match transaction version")
                }
            };
            if tip <= versioned_constants.os_constants.v1_bound_accounts_max_tip {
                return signed_tx_version(
                    &TransactionVersion::ONE,
                    &TransactionOptions { only_query: tx_context.tx_info.only_query() },
                );
            }
        }

        tx_context.tx_info.signed_version()
    }
