### Title
Malicious P2P Peer Can Inject Arbitrary `block_hash` into Storage via Unverified `SignedBlockHeader` — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

The p2p sync client accepts and permanently stores a peer-supplied `block_hash` field from a `SignedBlockHeader` protobuf message without ever recomputing or verifying it against the canonical `calculate_block_hash(...)` value. A single malicious peer can cause the syncing node to store an attacker-chosen hash (e.g., `Felt::ZERO`) in the `headers` table and the `block_hash_to_number` reverse-index table, making the RPC return an authoritative-looking wrong block hash for every affected block.

---

### Finding Description

**Step 1 — Protobuf deserialization (no verification).**

`TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` decodes `block_hash` directly from the wire value and places it into `BlockHeader::block_hash` with no recomputation: [1](#0-0) 

**Step 2 — `parse_data_for_block` (no hash check).**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks on the received header: block-number ordering and signature-vector length. There is no call to `calculate_block_hash`, no call to `verify_block_signature`, and the `_storage_reader` parameter is unused (note the leading underscore): [2](#0-1) 

A `grep` across the entire `crates/apollo_p2p_sync/` tree confirms zero occurrences of either `verify_block_signature` or `calculate_block_hash`.

**Step 3 — `write_to_storage` → `append_header` (stores as-is).**

`BlockData::write_to_storage` for `SignedBlockHeader` calls `append_header` directly with the peer-supplied `block_header`: [3](#0-2) 

`append_header` copies `block_header.block_hash` verbatim into `StorageBlockHeader` and then calls `update_hash_mapping`, which inserts the attacker-supplied hash as the key in the `block_hash_to_number` table: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

After the attack:

- `get_block_header(N)` returns a `BlockHeader` with `block_hash = Felt::ZERO` (or any attacker value). Every RPC endpoint that reads a block header (e.g., `starknet_getBlockWithTxHashes`, `starknet_getBlockWithTxs`) returns this wrong hash as an authoritative value.
- `get_block_number_by_hash(Felt::ZERO)` returns `Some(N)`, while `get_block_number_by_hash(canonical_hash)` returns `None`. RPC calls that look up a block by its hash (e.g., `starknet_getBlockWithTxHashesByHash`) will fail for the canonical hash and succeed for the injected one.

This falls squarely within: **High — RPC execution returns an authoritative-looking wrong value.**

Note: the `get_block_hash` syscall and the batcher's `get_block_hash` read from the separate `block_hashes` table (written by `set_block_hash`, not `append_header`), so execution-layer block-hash reads are not directly affected by this path.

---

### Likelihood Explanation

Any node that participates in the p2p network as a peer can send a crafted `BlockHeadersResponse`. The syncing node has no mechanism to detect or reject a wrong `block_hash` — the only peer-reputation action taken is `report_peer` on `BadPeerError`, and a wrong `block_hash` never triggers that path. The attack requires only that the malicious peer is one of the peers the syncing node queries, which is a standard network-level capability.

---

### Recommendation

After decoding the `SignedBlockHeader` in `parse_data_for_block`, recompute the canonical block hash using `calculate_block_hash` (from `starknet_api::block_hash::block_hash_calculator`) over the decoded header fields and compare it to the received `block_hash`. If they differ, return `ParseDataError::BadPeer(...)` and call `report_peer`. Additionally, verify the `BlockSignature` over `Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0])` against a known sequencer public key before storing.

---
