### Title
Missing Contract Existence Check in `ERC20Helper` Allows Silent Deposit Without Token Transfer — (`core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransferFrom` and `ERC20Helper.safeTransfer` perform low-level `.call()` on a token address without first verifying that the address contains deployed code. Per EVM design, a call to a codeless address returns `success = true` with empty `returnData`. The post-call guard `success && (data.length == 0 || abi.decode(data, (bool)))` evaluates to `true` in this case, so the require passes silently and the function returns as if the transfer succeeded — even though no tokens moved.

---

### Finding Description

Both helpers in `ERC20Helper.sol` share the same pattern:

```solidity
(bool success, bytes memory data) = address(self).call(
    abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
);
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [1](#0-0) [2](#0-1) 

When `address(self)` has no deployed bytecode:
- The EVM returns `(true, "")` — `success = true`, `data = bytes("")`
- The guard becomes `true && (true || …)` → `true`
- `require` passes; the function returns without reverting

No contract existence check (e.g., `require(address(self).code.length > 0)`) is present in either function.

These helpers are consumed by `EndpointStorage.handleDepositTransfer`, which is the on-chain token-pull path for every user deposit:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);          // ← silent no-op if token has no code
    safeTransferTo(token, address(clearinghouse), amount); // ← silent no-op
}
``` [3](#0-2) 

The only guard is a zero-address check — not a code-existence check. `handleDepositTransfer` is called directly from `Endpoint.depositCollateralWithReferral` before the slow-mode transaction is enqueued: [4](#0-3) 

After `handleDepositTransfer` returns (silently, with no tokens moved), the slow-mode transaction is queued. When the sequencer later processes it, `Clearinghouse.depositCollateral` unconditionally credits the subaccount balance:

```solidity
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [5](#0-4) 

The clearinghouse has no way to know the token transfer silently failed.

The same silent-success path also exists in `BaseWithdrawPool.handleWithdrawTransfer` and `submitFastWithdrawal`, where `token.safeTransfer` is called to pay out withdrawals: [6](#0-5) 

---

### Impact Explanation

If a registered spot product's token address has no deployed code at call time, an attacker can call `depositCollateral` for that product and receive a credited subaccount balance without transferring any tokens. The clearinghouse's internal accounting diverges from its actual token holdings: it records a liability (the credited balance) backed by zero real collateral. The attacker can then use this phantom collateral to open leveraged positions, borrow against it, or withdraw quote via settlement — draining real assets from other users.

The same flaw in `handleWithdrawTransfer` means a withdrawal to a codeless token address is silently marked as completed (`markedIdxs[idx] = true`) while the recipient receives nothing, permanently locking the user's funds.

---

### Likelihood Explanation

The trigger condition — a registered token address with no code — can arise in two realistic ways:

1. **Token self-destruct:** A token contract that uses `selfdestruct` (common in older or upgradeable token designs) can be destroyed, leaving its address codeless. Any user can then exploit the deposit path for that product without any special privilege.
2. **Product registered with a non-contract address:** If a product is registered pointing to an address that has not yet been deployed (e.g., a CREATE2 address before deployment), the window between registration and deployment is exploitable by any caller.

In both cases the attacker needs only to call the public `depositCollateral` or `depositCollateralWithReferral` entrypoint — no privileged access is required.

---

### Recommendation

Add a contract existence check in both `ERC20Helper` functions before the low-level call:

```solidity
require(address(self).code.length > 0, "ERC20: token has no code");
```

This mirrors the fix recommended in the Solidity documentation and in OpenZeppelin's `Address.functionCall` helper. Alternatively, use OpenZeppelin's `SafeERC20` library, which already includes this guard.

---

### Proof of Concept

1. A spot product is registered with token address `T` (legitimate at registration time).
2. Token `T` is self-destructed; `T.code.length == 0`.
3. Attacker calls `Endpoint.depositCollateral(subaccountName, productId, 1_000_000)`.
4. `handleDepositTransfer(T, attacker, 1_000_000)` is reached.
5. `ERC20Helper.safeTransferFrom(T, attacker, 1_000_000)`:
   - `address(T).call(...)` → EVM returns `(true, "")` (no code at T)
   - Guard: `true && (true || …)` → passes silently; no tokens pulled
6. `ERC20Helper.safeTransfer(T, clearinghouse, 1_000_000)` → same silent success; no tokens sent to clearinghouse.
7. Slow-mode transaction is enqueued for `DepositCollateral{sender: attacker_subaccount, productId, amount: 1_000_000}`.
8. Sequencer processes it; `Clearinghouse.depositCollateral` calls `spotEngine.updateBalance(productId, attacker_subaccount, amountRealized)`.
9. Attacker's subaccount now shows `1_000_000` units of collateral backed by zero real tokens.
10. Attacker uses this phantom collateral to open positions or trigger withdrawals of real assets held by the clearinghouse from other depositors.
