Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates caller-controlled `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates `owner`, a parameter freely chosen by the caller. Because any caller can supply any allowlisted address as `owner`, the deposit allowlist provides zero access control: any unprivileged address can add liquidity to a pool the admin intended to restrict.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both into the hook call in the correct order `(sender, owner, ...)`: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` leaves the first `address` parameter (`sender`) unnamed and never reads it, then checks `allowedDepositor[msg.sender][owner]` — the caller-controlled position-owner address — instead of `allowedDepositor[msg.sender][sender]`: [3](#0-2) 

**Exploit path:**
1. Pool is deployed with `DepositAllowlistExtension`; admin allowlists `alice` via `setAllowedToDeposit`.
2. Unprivileged `attacker` calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
3. Pool calls `_beforeAddLiquidity(attacker /*sender*/, alice /*owner*/, ...)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. `LiquidityLib.addLiquidity` executes; the liquidity callback fires on `attacker`, pulling tokens from `attacker`; a position is minted under `alice`.
6. `removeLiquidity` enforces `msg.sender == owner`, so `attacker` cannot reclaim the deposited tokens. [4](#0-3) 

The allowlist was never consulted for `attacker`. The wrong value checked is `allowedDepositor[pool][owner]` (caller-controlled) instead of `allowedDepositor[pool][sender]` (the actual depositor).

## Impact Explanation

`DepositAllowlistExtension` is the sole mechanism for restricting who may provide liquidity to a pool. With the guard checking the wrong address, the restriction is entirely ineffective. Any unprivileged address can inject liquidity into a pool the admin intended to restrict, bypassing KYC/compliance controls and altering pool depth and bin state (`binTotals`, `curPosInBin`, `_binTotalShares`, `_positionBinShares`) in ways downstream extensions or integrators did not anticipate. This is a direct admin-boundary break: a pool-admin-configured guard is bypassed by an unprivileged path.

## Likelihood Explanation

The bypass requires no special conditions, no flash loan, and no privileged access. Any EOA or contract can trigger it in a single transaction by supplying any allowlisted address as `owner`. The allowlisted address need not cooperate. The allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events. Likelihood is high.

## Recommendation

Name and validate `sender` instead of `owner`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
      external view override returns (bytes4)
  {
-     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+     if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
          revert IMetricOmmPoolActions.NotAllowedToDeposit();
      }
      return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

```
Setup
─────
pool  = MetricOmmPool (DepositAllowlistExtension configured)
alice = allowlisted: allowedDepositor[pool][alice] = true
eve   = NOT allowlisted

Attack
──────
eve calls:
  pool.addLiquidity(
      alice,   // owner  ← eve controls this
      0,       // salt
      deltas,  // any valid bin deltas
      "",      // callbackData
      ""       // extensionData
  )

Hook fires:
  beforeAddLiquidity(eve /*sender — ignored*/, alice /*owner — checked*/)
  → allowedDepositor[pool][alice] == true → no revert

Result:
  LiquidityLib.addLiquidity executes; callback pulls tokens from eve;
  position minted: owner=alice, salt=0.
  eve deposited into a restricted pool; allowlist provided zero protection.
  eve's tokens are locked under alice's position (eve cannot call removeLiquidity).
```
