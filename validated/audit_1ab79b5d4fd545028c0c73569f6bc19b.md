Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the actual caller, allowing any address to bypass the deposit allowlist by nominating an allowlisted address as `owner` — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` receives the actual caller as its first (unnamed, discarded) argument and only checks the `owner` (position recipient) against the per-pool allowlist. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any unprivileged address can call `addLiquidity` with an allowlisted address as `owner`, pay the tokens via callback, and have the allowlist guard pass — entirely bypassing the pool admin's curation policy.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

There is no `msg.sender == owner` guard in `addLiquidity` (contrast with `removeLiquidity` at line 206 which enforces `if (msg.sender != owner) revert NotPositionOwner()`).

`DepositAllowlistExtension.beforeAddLiquidity` receives the actual caller as its first argument but leaves it unnamed and never reads it:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The check `allowedDepositor[msg.sender][owner]` uses `msg.sender` = the pool address and `owner` = the position recipient. The actual payer/initiator is never validated. Any caller who supplies an allowlisted address as `owner` will pass the guard regardless of their own allowlist status.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` further exposes this path: it accepts an arbitrary `owner` with only a non-zero check (`_validateOwner`), stores `msg.sender` as the payer in transient storage, and forwards the caller-supplied `owner` directly to the pool — meaning an unprivileged user can route through the supported periphery contract as well.

## Impact Explanation
This is an admin-boundary break. The pool admin deploys `DepositAllowlistExtension` as the sole mechanism to restrict who may deposit into a curated pool. An unprivileged address can deposit tokens into the pool by specifying any allowlisted address as `owner`. The attacker's capital enters the pool; the allowlisted address receives LP shares it did not request. The pool admin's curation policy — the only guard controlling participation — is silently violated through a publicly reachable path.

## Likelihood Explanation
Exploitation requires only knowledge of one allowlisted address, which is publicly readable from `allowedDepositor(pool, depositor)`. No special role, flash loan, oracle manipulation, or privileged access is needed. The path is reachable both directly through `MetricOmmPool.addLiquidity` and through the supported `MetricOmmPoolLiquidityAdder` periphery. Any user can trigger it at any time on any pool configured with `DepositAllowlistExtension`.

## Recommendation
The extension should gate on the actual caller (first argument), not only on `owner`. Change the check to validate the caller:

```solidity
function beforeAddLiquidity(address caller, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender]
        && !allowedDepositor[msg.sender][caller]
        && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This ensures the actual payer/initiator is always checked, while still permitting the `MetricOmmPoolLiquidityAdder` design where the adder is the caller and the user is the owner (both would need to be allowlisted, or the adder itself allowlisted).

## Proof of Concept
1. Pool admin deploys a curated pool with `DepositAllowlistExtension`. Bob (`0xBob`) is allowlisted; Alice (`0xAlice`) is not.
2. Alice calls `pool.addLiquidity(0xBob, salt, deltas, callbackData, extensionData)` directly.
3. The pool calls `_beforeAddLiquidity(0xAlice, 0xBob, ...)`, which calls `DepositAllowlistExtension.beforeAddLiquidity(0xAlice, 0xBob, ...)`.
4. The extension checks `allowedDepositor[pool][0xBob]` → `true`. The guard passes; `0xAlice` is never checked.
5. `LiquidityLib.addLiquidity` mints LP shares to `0xBob`.
6. The pool calls `IMetricOmmSwapCallback(0xAlice).metricOmmModifyLiquidityCallback(...)`. Alice's contract pays the required tokens.
7. Result: Alice's tokens are in the pool; Bob holds LP shares he did not request. The allowlist has been bypassed — Alice's capital entered a pool she was explicitly excluded from.
