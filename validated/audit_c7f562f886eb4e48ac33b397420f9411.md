### Title
Unchecked Integer Overflow in `op_unknown` Cost Multiplication Causes Undercharged Execution or Panic - (File: src/more_ops.rs)

### Summary
`op_unknown` in `src/more_ops.rs` performs a plain `u64` multiplication (`cost *= cost_multiplier + 1`) after a `check_cost` guard. Because Rust's integer arithmetic panics on overflow in debug builds and silently wraps in release builds, a crafted unknown opcode with a large `cost_multiplier` and arguments that push the pre-multiplication cost to a specific value can cause the final cost to wrap to zero (or a small value), bypassing the `if cost > u32::MAX as u64` rejection guard and returning an undercharged `Reduction`. This is the direct analog of the missing `unchecked` block in Solidity: arithmetic that is supposed to handle large intermediate values instead produces incorrect results due to missing overflow protection.

### Finding Description

In `src/more_ops.rs`, the `op_unknown` function computes a base cost from the opcode's `cost_function` field, then multiplies by `(cost_multiplier + 1)`:

```rust
// src/more_ops.rs ~line 258-266
assert!(cost > 0);

check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;          // ← plain u64 multiply, no overflow guard
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode bytes as a `u64` cast from `u32`, so `cost_multiplier + 1` can be as large as `2^32 = 4_294_967_296`. The `check_cost` call before the multiply only ensures `cost ≤ max_cost`; it does not bound the product. If `max_cost` is large (e.g., `u64::MAX`, which is what `run_program` uses when the caller passes `0`), then `cost` can be up to `u64::MAX` before the multiply.

**Concrete overflow path:**
- Set `cost_function = 2` (mul-like) and `cost_multiplier = u32::MAX` (opcode bytes `[0xff, 0xff, 0xff, 0xff, 0x80]`).
- Pass arguments to the unknown op such that the accumulated pre-multiplication cost equals exactly `2^32 = 4_294_967_296`.
- Then `cost *= cost_multiplier + 1` computes `2^32 * 2^32 = 2^64`, which wraps to **0** in release mode.
- `0 > u32::MAX` is false, so the function returns `Ok(Reduction(0, allocator.nil()))`.
- The main loop in `run_program` adds this 0 to `cost`, effectively charging nothing for the operation.

In debug mode the same multiply panics unconditionally, crashing the node.

### Impact Explanation

**Release build:** The wrapped cost (0 or a small value) is returned and accumulated into the main `cost` counter in `run_program`. The program is undercharged, allowing an attacker to execute arbitrarily expensive unknown-op programs without consuming the expected cost budget. This is a consensus-layer undercharged-execution vulnerability: nodes running release builds accept and process the transaction cheaply, while nodes running debug builds crash.

**Debug build:** The `cost *= cost_multiplier + 1` line panics with an arithmetic overflow, crashing the process. This is a remotely-triggerable denial of service.

**Impact: Medium** — undercharged execution in release mode (consensus divergence between build profiles) or node crash in debug mode.

### Likelihood Explanation

`op_unknown` is called in lenient/consensus mode whenever a CLVM program references an unrecognized opcode. An attacker can freely craft the opcode bytes (controlling `cost_function` and `cost_multiplier`) and the argument list (controlling the pre-multiplication cost). No special privilege is required beyond submitting a transaction to the network.

**Likelihood: Medium** — the attacker must tune the pre-multiplication cost to a value that causes the product to wrap to a small number, which requires knowledge of the cost constants but is entirely deterministic and offline-computable.

### Recommendation

Replace the plain multiply with a checked or saturating multiply, and treat overflow as an invalid opcode:

```rust
// Replace:
cost *= cost_multiplier + 1;

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the fix recommended in the external report (adding `unchecked` in Solidity to allow intentional overflow, or conversely, adding explicit overflow checks where overflow is not intended).

### Proof of Concept

1. Construct an unknown opcode atom with bytes `[0xff, 0xff, 0xff, 0xff, 0x80]`:
   - `op[op.len()-1] = 0x80` → `cost_function = (0x80 & 0xC0) >> 6 = 2` (mul-like)
   - `op[0..4] = [0xff, 0xff, 0xff, 0xff]` → `cost_multiplier = u32::MAX = 4_294_967_295`
2. Pass arguments to the op such that the cost_function=2 loop accumulates exactly `cost = 4_294_967_296 = 2^32` before `check_cost`.
3. `check_cost(4_294_967_296, u64::MAX)` passes.
4. `cost *= 4_294_967_296` → `4_294_967_296 * 4_294_967_296 = 2^64` → wraps to **0** in release mode.
5. `0 > u32::MAX` is false → returns `Ok(Reduction(0, nil))`.
6. The main loop in `run_program` adds 0 to `cost` — the operation is free.
