### Title
`creditDeposit()` Reverts for All Products When Any Spot Product Has a Zero Token Address — (`File: core/contracts/DirectDepositV1.sol`)

### Summary

`DirectDepositV1.creditDeposit()` loops over every product ID returned by `spotEngine.getProductIds()` and calls `require(tokenAddr != address(0), "Invalid productId.")` inside the loop. If any product in the list — such as the virtual NLP product — has no associated ERC20 token (`address(0)`), the entire call reverts, permanently blocking deposits for all other valid products.

### Finding Description

`creditDeposit()` is the core function of the `DirectDepositV1` contract. It iterates over all spot product IDs and attempts to deposit any held token balance into the protocol:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");   // ← hard revert
        ...
    }
}
``` [1](#0-0) 

`getProductIds()` returns the full set of registered spot products, which includes virtual products such as `NLP_PRODUCT_ID`. The `SpotEngineState.updateStates()` function explicitly skips `NLP_PRODUCT_ID` with a `continue` because it has no real ERC20 token:

```solidity
if (productId == NLP_PRODUCT_ID) {
    continue;
}
``` [2](#0-1) 

`creditDeposit()` performs no such skip. When the loop reaches `NLP_PRODUCT_ID` (or any other product with `address(0)` token), `spotEngine.getToken(productId)` returns `address(0)`, the `require` fires, and the entire transaction reverts — including all prior iterations that had not yet been committed (Solidity reverts all state changes).

The correct fix, matching the pattern already used in `updateStates()`, is to replace the `require` with a `continue`.

### Impact Explanation

The DDA's primary purpose is to automatically credit collateral into the protocol for its bound subaccount. With `creditDeposit()` permanently reverting, any ERC20 tokens sent to the DDA contract cannot be deposited into the protocol. The subaccount never receives the expected collateral, which can cause it to fall below margin requirements and become subject to liquidation. Funds accumulate in the DDA contract and are inaccessible to the protocol until the owner manually calls `withdraw()` — a recovery path that defeats the purpose of the DDA and requires active owner intervention.

### Likelihood Explanation

`NLP_PRODUCT_ID` is a registered spot product (evidenced by the explicit `continue` guard in `updateStates()`). If `getToken(NLP_PRODUCT_ID)` returns `address(0)` — the expected behavior for a virtual/synthetic product — then `creditDeposit()` is broken from the moment the NLP product is registered. The function is `external` with no access control, so any caller can observe and trigger the revert. No special privileges or unusual conditions are required.

### Recommendation

Replace the `require` with a `continue` so that products without a token address are skipped rather than causing the entire call to revert:

```solidity
address tokenAddr = spotEngine.getToken(productId);
if (tokenAddr == address(0)) {
    continue;  // skip virtual/synthetic products
}
```

This mirrors the guard already used in `SpotEngineState.updateStates()` for `NLP_PRODUCT_ID`.

### Proof of Concept

1. The NLP product is registered in the spot engine's product list (confirmed by the explicit `NLP_PRODUCT_ID` skip in `SpotEngineState.updateStates()`).
2. `getToken(NLP_PRODUCT_ID)` returns `address(0)` because NLP is a virtual product with no underlying ERC20.
3. Any caller invokes `creditDeposit()`.
4. The loop reaches `NLP_PRODUCT_ID`; `spotEngine.getToken(NLP_PRODUCT_ID)` returns `address(0)`.
5. `require(tokenAddr != address(0), "Invalid productId.")` reverts the entire transaction.
6. No product — including those with valid ERC20 tokens and non-zero balances held by the DDA — gets deposited into the protocol.
7. The subaccount bound to the DDA never receives its collateral; funds accumulate in the DDA contract indefinitely until the owner manually recovers them via `withdraw()`.
