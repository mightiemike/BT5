### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to Legacy `L1Gas` When Both `l2_gas` and `l1_data_gas` Are Zero, Producing a Divergent Transaction Hash in the Sync Path ‚ÄĒ (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` contains a legacy-compatibility gate that silently converts a new-style `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction into the pre-0.13.3 `L1Gas(X)` variant. Because `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage only for `AllResources`, the hash computed during block sync differs from the hash computed at submission time, breaking the hash-binding invariant for any transaction that legitimately sets both `l2_gas` and `l1_data_gas` to zero.

---

### Finding Description

**Root cause ‚ÄĒ the legacy gate in the protobuf deserializer**

In `crates/apollo_protobuf/src/converters/transaction.rs`, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` reads:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
let l1_gas: ResourceBounds = l1_gas.try_into()?;
let l2_gas: ResourceBounds = l2_gas.try_into()?;
let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ‚Üź legacy path
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The gate was introduced to handle pre-0.13.3 blocks where `l1_data_gas` was absent from the wire format. However, the serializer for `ValidResourceBounds::L1Gas` always emits `l1_data_gas: Some(ResourceBounds::default())` (i.e. `Some(zero)`), and the serializer for `AllResources` with zero bounds also emits `l1_data_gas: Some(zero)`:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),   // Some(zero)
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),             // Some(zero) when zero
    },
``` [2](#0-1) 

The serialized wire bytes are **identical** for `L1Gas(X)` and `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The deserializer therefore cannot distinguish them and always produces `L1Gas(X)` for the latter.

**Hash divergence ‚ÄĒ the version-gated preimage**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` includes `l1_data_gas` in the Poseidon preimage only for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // ‚Üź no l1_data_gas
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [3](#0-2) 

Consequently:

| Variant at hash time | Poseidon input elements | Hash |
|---|---|---|
| `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` | `[tip, L1_GAS‚ÄĖX, L2_GAS‚ÄĖ0, L1_DATA_GAS‚ÄĖ0]` | **H‚āĀ** |
| `L1Gas(X)` (after protobuf round-trip) | `[tip, L1_GAS‚ÄĖX, L2_GAS‚ÄĖ0]` | **H‚āā ‚Č† H‚āĀ** |

**Submission path vs. sync path**

- **Submission / consensus path**: `RpcTransaction` carries `AllResourceBounds` directly; `InternalRpcInvokeTransactionV3::resource_bounds` is typed `AllResourceBounds`, so `calculate_transaction_hash` always calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources(...)`, producing **H‚āĀ**. [4](#0-3) 

- **Sync path**: blocks are received as `transaction_in_block` protobuf messages, deserialized through `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. For the affected transaction, this produces `L1Gas(X)`, and any subsequent hash recomputation yields **H‚āā**.

The consensus-path protobuf converter (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs`) correctly requires `l1_data_gas` to be present and does not apply the legacy gate, so the consensus path is unaffected. The divergence is isolated to the block-sync path. [5](#0-4) 

---

### Impact Explanation

A syncing node that receives a block containing a transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` will recompute a transaction hash **H‚āā** that differs from the sequencer's canonical hash **H‚āĀ**. Depending on where hash validation occurs:

- If the sync layer validates the transaction hash against the block's transaction commitment, the block is rejected ‚Üí **chain split / DoS** against syncing nodes.
- If the hash is trusted from the block header but the transaction body is stored with the recomputed hash, the node's database holds the wrong hash ‚Üí **RPC responses (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`) return an authoritative-looking wrong value**.

This matches the allowed impact: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

Any user can submit a V3 transaction with `l2_gas = 0` and `l1_data_gas = 0` (paying only for L1 gas). The gateway's stateless validator accepts such transactions. The condition is therefore fully attacker-controlled and requires no privileged access. [6](#0-5) 

---

### Recommendation

Distinguish pre-0.13.3 legacy absence (`None`) from a new-style explicit zero (`Some(zero)`) at the serializer level. Change `ValidResourceBounds::L1Gas` serialization to emit `l1_data_gas: None` instead of `Some(zero)`:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: None,   // ‚Üź signals "pre-0.13.3, no data gas field"
},
```

Then update the deserializer to gate on `is_none()` rather than `is_zero()`:

```rust
Ok(match value.l1_data_gas {
    None if l2_gas.is_zero() => ValidResourceBounds::L1Gas(l1_gas),
    opt => {
        let l1_data_gas: ResourceBounds = opt.unwrap_or_default().try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
})
```

This preserves backward compatibility with genuine pre-0.13.3 blocks (which emit `l1_data_gas: None`) while correctly classifying new-style transactions that happen to have zero bounds as `AllResources`, keeping the hash preimage consistent across the submission and sync paths.

### Proof of Concept

1. Submit an invoke V3 transaction with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`.
2. The gateway's `convert_rpc_tx_to_internal` computes hash **H‚āĀ** via `get_invoke_transaction_v3_hash` ‚Üí `get_tip_resource_bounds_hash` with `AllResources`, Poseidon over 4 elements `[tip, L1_GAS‚ÄĖ‚Ä¶, L2_GAS‚ÄĖ0, L1_DATA_GAS‚ÄĖ0]`.
3. The transaction is included in block B; the block commitment encodes **H‚āĀ**.
4. A syncing peer receives block B. The `transaction_in_block` protobuf for this transaction has `l1_data_gas = Some({ max_amount: 0, max_price_per_unit: 0 })` and `l2_gas = Some({ 0, 0 })`.
5. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires: `l1_data_gas.is_zero() && l2_gas.is_zero()` ‚Üí `ValidResourceBounds::L1Gas(l1_gas)`.
6. The syncing peer recomputes the hash via `get_tip_resource_bounds_hash` with `L1Gas`, Poseidon over 3 elements `[tip, L1_GAS‚ÄĖ‚Ä¶, L2_GAS‚ÄĖ0]` ‚Üí **H‚āā ‚Č† H‚āĀ**.
7. Hash mismatch: the syncing peer either rejects block B or stores the wrong hash for the transaction.
