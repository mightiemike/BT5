Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. The allowlist therefore gates the router address rather than the actual swapper, creating an inescapable dilemma: either allowlisted users cannot use the router, or the router is allowlisted and every user on the network can bypass the curated pool's swap restriction.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the `sender` positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(recipient, ...)` directly, making `msg.sender` inside the pool the **router contract**: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`. The actual end user's address is never visible to the extension. By contrast, `DepositAllowlistExtension` correctly avoids this by keying on `owner` (the position owner, explicitly passed and independent of the call chain): [5](#0-4) 

The swap allowlist has no equivalent separation between the call-chain intermediary and the economic actor.

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set of addresses faces an inescapable dilemma. If the router is not allowlisted, allowlisted users cannot use `MetricOmmSimpleRouter` at all — breaking the primary documented swap entry point. If the router is allowlisted so that legitimate users can swap, every address on the network can call the router and bypass the allowlist entirely, defeating the curation policy and allowing unauthorized swaps that drain pool liquidity at oracle-determined prices. There is no configuration that simultaneously preserves router usability for allowlisted users and blocks non-allowlisted users. This constitutes broken core pool functionality and direct loss of pool assets to unauthorized swappers.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any user who calls `exactInputSingle` or `exactInput` through the router on a pool where the router has been allowlisted (for any legitimate user's benefit) can exploit this immediately without any privileged access, special tokens, or setup. The trigger is a normal router call available to any EOA or contract.

## Recommendation
Pass the original end-user address through the swap path so the extension can gate on it. The preferred fix is to add an explicit `swapper` (or `originator`) field to the swap parameters that the router populates with `msg.sender` before calling the pool, and forward it to extension hooks as a separate argument distinct from `sender`. Alternatively, document that `SwapAllowlistExtension` gates only the *direct pool caller* and prohibit curated pools from using the public router — but this is a severe usability restriction that must be made explicit in the extension interface and documentation.

## Proof of Concept
```
Setup:
  Pool P configured with SwapAllowlistExtension E
  Pool admin allowlists only Alice: allowedSwapper[P][Alice] = true
  Router R is NOT allowlisted

Step 1 — Alice tries to use the router:
  Alice calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  Router calls P.swap(recipient=Alice, ...)  ← msg.sender at pool = Router R
  Pool calls _beforeSwap(sender=R, ...)
  Extension checks allowedSwapper[P][R] → false → REVERTS
  Alice cannot use the router.

Step 2 — Admin adds router to fix Alice's UX:
  allowedSwapper[P][R] = true

Step 3 — Bob (not allowlisted) exploits:
  Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  Router calls P.swap(recipient=Bob, ...)  ← msg.sender at pool = Router R
  Pool calls _beforeSwap(sender=R, ...)
  Extension checks allowedSwapper[P][R] → true → PASSES
  Bob, who was never allowlisted, swaps freely against the curated pool.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension.
  2. Allowlist only Alice directly; do NOT allowlist Bob.
  3. Allowlist the router address (simulating admin enabling router for Alice).
  4. Call router.exactInputSingle from Bob's address.
  5. Assert swap succeeds — demonstrating full allowlist bypass.
```
