### Title
`ResourceBoundsMapping → ValidResourceBounds` Conversion Produces `L1Gas` Variant for Zero `l1_data_gas`/`l2_gas` Transactions, Causing Transaction Hash Divergence Between RPC Execution and Gateway Paths — (`File: crates/apollo_rpc/src/v0_8/transaction.rs`)

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently downgrades a V3 invoke transaction's resource bounds to `ValidResourceBounds::L1Gas` whenever `l1_data_gas == 0` and `l2_gas == 0`. This causes `get_tip_resource_bounds_hash` to hash only two resource elements (L1_GAS, L2_GAS) instead of three (L1_GAS, L2_GAS, L1_DATA_GAS), producing a different transaction hash than the gateway path, which always uses `ValidResourceBounds::AllResources`.

### Finding Description

**Divergent conversion — RPC layer:**

In `crates/apollo_rpc/src/v0_8/transaction.rs` lines 188–199:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← drops L1_DATA_GAS from hash
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
```

This conversion is consumed in `crates/apollo_rpc/src/v0_8/api/mod.rs` line 624 when converting an `InvokeTransaction::Version3` (the RPC read/execution type, which carries `resource_bounds: ResourceBoundsMapping`) into `starknet_api::transaction::InvokeTransactionV3`:

```rust
resource_bounds: resource_bounds.into(),  // ResourceBoundsMapping → ValidResourceBounds::L1Gas
```

**Gateway path always uses `AllResources`:**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` directly. The `InvokeTransactionV3Trait` implementation for `InternalRpcInvokeTransactionV3` unconditionally returns `ValidResourceBounds::AllResources(self.resource_bounds)` (line 637–639 of `crates/starknet_api/src/rpc_transaction.rs`). The gateway hash computation therefore always includes the L1_DATA_GAS element.

**Hash divergence in `get_tip_resource_bounds_hash`:**

In `crates/starknet_api/src/transaction_hash.rs` lines 188–210:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 elements
    }
});
```

For a transaction with `l1_data_gas=0` and `l2_gas=0`:
- **Gateway path** (`AllResources`): `poseidon(tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt)` — 3 resource elements, L1_DATA_GAS felt is zero but present.
- **RPC execution path** (`L1Gas`): `poseidon(tip, L1_GAS_felt, L2_GAS_felt)` — 2 resource elements, L1_DATA_GAS absent.

These produce different `tip_resource_bounds_hash` values, and therefore different final transaction hashes via `get_invoke_transaction_v3_hash` (lines 370–404 of `crates/starknet_api/src/transaction_hash.rs`).

### Impact Explanation

When `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_traceTransaction` process a V3 invoke transaction with `l1_data_gas=0` and `l2_gas=0`, the blockifier executes it with a transaction hash that differs from the canonical hash computed by the gateway. The `get_execution_info` syscall exposes this wrong hash to the contract during simulation/estimation. Any contract logic that branches on the transaction hash (e.g., replay-protection, meta-transaction routing, or client-side proving verification) will observe a different value in simulation than in actual execution, making fee estimation and simulation return authoritative-looking but incorrect results.

This matches the **High** impact scope: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Any V3 invoke transaction with `l1_data_gas=0` and `l2_gas=0` triggers the divergence. Pre-0.13.3-style transactions that set only L1 gas bounds are a common pattern. The trigger requires no privilege — any unprivileged user can submit such a transaction to the fee-estimation or simulation endpoint.

### Recommendation

Remove the `L1Gas` downgrade from `From<ResourceBoundsMapping> for ValidResourceBounds`. When all three resource fields are present in the JSON-RPC input (even if zero), always produce `ValidResourceBounds::AllResources`. The `L1Gas` variant should only be used when deserializing legacy on-disk transactions that genuinely predate the `AllResources` format, not when converting live RPC input.

Alternatively, align the RPC execution path to use `RpcInvokeTransactionV3` (with `AllResourceBounds`) as the input type for fee estimation and simulation, matching the gateway path, so the same `InvokeTransactionV3Trait` implementation is used for hash computation in all paths.

### Proof of Concept

1. Construct a V3 invoke transaction with `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }` and `l2_gas = { max_amount: 0, max_price_per_unit: 0 }`, and any non-zero `l1_gas`.
2. Submit it to the gateway (`add_transaction`). The gateway computes hash **H_gateway** using `ValidResourceBounds::AllResources`, which includes the zero L1_DATA_GAS felt in `get_tip_resource_bounds_hash`.
3. Submit the same transaction body to `starknet_estimateFee`. The RPC layer converts `ResourceBoundsMapping` → `ValidResourceBounds::L1Gas` (line 190–191 of `crates/apollo_rpc/src/v0_8/transaction.rs`), then computes hash **H_rpc** using only two resource elements.
4. Assert `H_gateway ≠ H_rpc`. The difference is in `tip_resource_bounds_hash`: the gateway version includes `get_concat_resource(&AllResourceBounds.l1_data_gas, L1_DATA)` (a zero-valued felt), while the RPC version omits it entirely.
5. Deploy a contract that reads `get_execution_info().tx_info.transaction_hash` and returns it. Observe that the hash returned during `starknet_simulateTransactions` differs from the hash stored on-chain after the transaction is sequenced. // An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

**File:** crates/starknet_api/src/transaction_hash.rs (L370-405)
```rust
pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}
```
