### Title
`op_secp256k1_verify` Accepts High-S (Malleable) Signatures Due to Missing Low-S Enforcement - (File: src/secp_ops.rs)

### Summary
`op_secp256k1_verify` in `src/secp_ops.rs` uses the `k256` crate's `Signature::from_slice()` and `verify_prehash()` without enforcing low-S normalization. For any valid ECDSA signature `(r, s)`, the malleable counterpart `(r, n−s)` (where `n` is the secp256k1 group order) is also accepted as valid for the same public key and message. This is the direct analog of the `ecrecover()` signature manipulation described in the external report: an attacker who observes a valid spend can produce a structurally different but equally valid spend, causing transaction malleability.

### Finding Description
In `src/secp_ops.rs`, `op_secp256k1_verify` parses the attacker-supplied 64-byte compact signature using `K1Signature::from_slice()` and verifies it with `verifier.verify_prehash()`:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

The `k256` crate's `Signature::from_slice()` does not enforce low-S normalization — it accepts any `(r, s)` pair where `s` is in `[1, n-1]`, including high-S values where `s > n/2`. The `PrehashVerifier::verify_prehash()` implementation likewise does not reject high-S signatures. No explicit low-S check exists anywhere in `op_secp256k1_verify`.

The same issue exists in `op_secp256r1_verify` via `p256::ecdsa::Signature::from_slice()` and `P1VerifyingKey::verify_prehash()`, which also lack low-S enforcement.

### Impact Explanation
For any CLVM puzzle that uses `secp256k1_verify` to authenticate a spend, an attacker who observes a valid spend bundle `(pubkey, msg, sig=(r,s))` can trivially compute the malleable counterpart `sig'=(r, n−s)` and submit a different spend bundle that passes the same puzzle. Because the spend bundle ID in Chia includes the solution (which contains the signature), the two spend bundles have different IDs despite having identical on-chain effects. This breaks:
- Off-chain protocols and payment channels that track spend bundles by ID
- Mempool deduplication logic that relies on spend bundle ID uniqueness
- Any application that assumes a confirmed spend bundle ID matches the one originally submitted

The attacker does not need any secret key material — only the ability to observe a valid spend bundle and flip the `s` component.

### Likelihood Explanation
The attack is trivially executable by any observer of the mempool or confirmed blocks. The computation `s' = n − s` requires only a single modular subtraction. No cryptographic secrets are needed. Any CLVM puzzle that uses `secp256k1_verify` (opcode `0x13d61f00` or opcode `64` with `ENABLE_SECP_OPS`) is affected. The operator is live on the Chia mainnet.

### Recommendation
After parsing the signature with `K1Signature::from_slice()`, explicitly enforce low-S normalization before calling `verify_prehash`. The `k256` crate provides `Signature::normalize_s()` for this purpose:

```rust
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
// Reject high-S signatures to prevent malleability
if sig.normalize_s().is_some() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "secp256k1_verify: signature has high-S value".to_string(),
    ))?;
}
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

Apply the same fix to `op_secp256r1_verify` using the equivalent `p256` API.

### Proof of Concept
Given a valid test vector from `op-tests/test-secp256k1.txt`:

```
pubkey = 0x02888b0c110ef0b4962e3fc6929cbba7a8bb25b4b2c885f55c76365018c909b439
msg    = 0x74c2941eb2ebe5aa4f2287a4c5e506a6290c045004058de97a7edf0122548668
sig    = 0x1acb7a6e...e6c018   (low-S, passes)
```

Compute `s' = n − s` where `n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`. The resulting `sig' = (r, s')` is a high-S signature. Submitting `sig'` to `op_secp256k1_verify` with the same `pubkey` and `msg` will return `Ok(Reduction(cost, nil()))` — accepted as valid — producing a spend bundle with a different ID but identical on-chain effect.
