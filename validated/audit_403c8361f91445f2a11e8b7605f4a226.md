Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead validates `owner` (the caller-supplied LP-share recipient). Because `owner` is a free parameter in `addLiquidity`, any non-allowlisted address can name any allowlisted address as `owner` and pass the hook unconditionally, depositing into a pool the admin intended to gate.

## Finding Description
The hook signature at line 32 unnamed the first argument (`sender`) and binds only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

The guard at line 38 then checks `allowedDepositor[msg.sender][owner]` — testing the LP-share recipient, not the depositor. [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` passes `sender` (the pool's `msg.sender`) as argument 0 and the caller-supplied `owner` as argument 1: [2](#0-1) 

Since `owner` is freely chosen by the caller, any address can set `owner` to any allowlisted address and the check passes. The contract's own NatSpec ("Gates `addLiquidity` by depositor address"), storage key naming (`allowedDepositor`), and setter/view functions all confirm the intent is to gate the caller, not the recipient: [3](#0-2) 

The companion `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (argument 0), confirming the deposit extension's check is wrong by direct comparison: [4](#0-3) 

## Impact Explanation
A pool admin deploys `DepositAllowlistExtension` to restrict which addresses may add liquidity (KYC-gated pool, private LP pool, or a pool in a sensitive state). The guard is silently inoperative: any non-allowlisted address can deposit freely by naming any allowlisted address as `owner`. The depositor's tokens enter the pool and LP shares are minted to the named allowlisted address — the state change the admin intended to block succeeds unconditionally. This is an admin-boundary break: an unprivileged path bypasses a pool admin's access control, breaking the core invariant the extension exists to enforce.

## Likelihood Explanation
The exploit requires no special permissions, no oracle manipulation, and no reentrancy. The allowlist mappings are public, so any observer can identify an allowlisted address. Any EOA or contract can call `addLiquidity` with `owner` set to that address. The attacker's only cost is the deposited capital. The attack is repeatable and unconditional.

## Recommendation
Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension`; only `alice` is allowlisted (`allowedDepositor[pool][alice] = true`).
2. `bob` (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(alice, 99, deltas, X, Y, "");
   //                ^owner = allowlisted address used as pass
   ```
3. Pool calls `extension.beforeAddLiquidity(bob, alice, 99, ...)`.
4. Hook evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. `bob`'s tokens are transferred into the pool; LP shares at `(alice, 99)` are minted.
6. `bob` has deposited into a pool the admin intended to restrict him from, bypassing the allowlist entirely.
