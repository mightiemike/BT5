### Title
`DirectDepositV1.creditDeposit` Does Not Operate in a Fail-Safe Manner — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` loops through all registered spot product IDs and makes bare external calls to each token's `approve()` and to `endpoint.depositCollateralWithReferral()`. If any single token's `approve()` reverts — for example because a token contract (USDC, USDT) blacklists the DDA address or is paused — the entire loop reverts, permanently blocking all other tokens sitting in the DDA from being credited to the subaccount.

---

### Finding Description

`creditDeposit()` is an unguarded `external` function with no access-control modifier. [1](#0-0) 

The loop iterates over every product ID returned by `spotEngine.getProductIds()`. For each product whose token balance is non-zero it executes two external calls in sequence:

```
token.approve(address(endpoint), balance);   // line 92
endpoint.depositCollateralWithReferral(...); // line 93-98
```

Neither call is wrapped in `try-catch` or `address.call`. If `token.approve()` reverts for any product — because the token contract has blacklisted the DDA address, is paused, or has any other revert condition — the entire transaction reverts. Every other token with a pending balance in the DDA is also blocked from being credited.

The protocol's own codebase already recognises this exact failure mode in two places:

1. The DDA **constructor** uses a low-level `.call` for the native-token transfer and explicitly comments: *"shouldn't revert even if the transfer fails, otherwise the funds will be stuck in the DDA forever."* [2](#0-1) 

2. `Endpoint.depositCollateralWithReferral()` returns early instead of reverting when a deposit is below the minimum, with the comment: *"we cannot revert here, otherwise direct deposit could be blocked when there are multiple assets awaiting credit but one of them is below the minimum deposit amount."* [3](#0-2) 

Despite this awareness, `creditDeposit()` itself does not apply the same fail-safe pattern to the `token.approve()` call.

`creditDeposit()` is also reachable through `ContractOwner.creditDepositV1(bytes32 subaccount)`, which is callable by any unprivileged address: [4](#0-3) 

---

### Impact Explanation

Any token registered as a spot product that reverts on `approve()` — due to blacklisting of the DDA address, a token-level pause, or any other external condition — causes `creditDeposit()` to revert entirely. All other tokens with pending balances in the DDA contract are also blocked from being credited to the subaccount. The DDA's primary function is broken until the problematic token condition is resolved. Funds are not permanently lost (the owner can call `withdraw()` per token), but the automated credit path is fully halted, requiring manual owner intervention for every affected token.

---

### Likelihood Explanation

USDC and USDT both implement blacklisting and pause functionality. If the DDA contract address is blacklisted by any one of the registered collateral tokens — whether by the token issuer, by a regulatory action, or by a protocol-level pause — `creditDeposit()` becomes permanently non-functional for all tokens. As the number of supported spot products grows, the probability of at least one token entering a revert-on-approve state increases. The entry path requires no privilege: any caller can trigger `creditDeposit()` or `creditDepositV1()`.

---

### Recommendation

Wrap each iteration's external calls in a `try-catch` block (or use low-level `address.call`) so that a failure on one token is silently skipped and the remaining tokens are still processed. This mirrors the pattern already applied in the DDA constructor and in `Endpoint.depositCollateralWithReferral()`:

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        if (tokenAddr == address(0)) continue;
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance == 0) continue;
        // Fail-safe: skip this token if approve or deposit reverts
        try token.approve(address(endpoint), balance) {
            try endpoint.depositCollateralWithReferral(
                subaccount, productId, uint128(balance), "-1"
            ) {} catch {}
        } catch {}
    }
}
```

---

### Proof of Concept

1. Protocol registers two spot products: product A (USDC) and product B (WETH).
2. A user sends 1000 USDC and 1 WETH to the DDA contract.
3. USDC's issuer blacklists the DDA contract address (or USDC is paused).
4. Any caller invokes `creditDeposit()` (or `ContractOwner.creditDepositV1(subaccount)`).
5. The loop reaches product A; `USDC.approve(endpoint, 1000e6)` reverts because the DDA is blacklisted.
6. The entire transaction reverts — the 1 WETH sitting in the DDA is also not credited.
7. The subaccount receives nothing. The owner must manually call `withdraw(WETH_token)` to recover the WETH, and the USDC remains stuck until the blacklist is lifted. [1](#0-0)
