Audit Report

## Title
Oracle Price Spike Permanently Inflates High Watermark, Blocking Legitimate Swaps — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`_checkAndUpdateWatermarks` unconditionally writes the new watermark to storage after every swap. When the oracle reports a transient price spike, `metricT1` is inflated and `hwm1` is ratcheted up permanently. After the oracle corrects, all `!zeroForOne` swaps revert with `OracleStopLossTriggered` because the true-price metric falls far below the inflated watermark floor, blocking one swap direction until admin intervention or slow decay.

## Finding Description

**Root cause — `_metrics` (lines 246–256):**

`metricT1` is computed as the token0 balance converted to token1 terms via the oracle mid-price:

```solidity
metricT1 = _clampMetric(
    Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps
);
```

When `midPriceX64` doubles (oracle spike), the t0 component doubles, so `metricT1` doubles. The NatSpec at lines 22–25 explicitly acknowledges this: *"A pure mid move pushes the metrics in opposite directions."* [1](#0-0) 

**Watermark ratchet — `_applyWatermark` (lines 333–335):**

During the spike, `metricT1 >= hwm1`, so `_applyWatermark` returns `(metricT1, false)` — the inflated metric becomes the new watermark with no breach flagged. [2](#0-1) 

**Unconditional storage write — `_checkAndUpdateWatermarks` (lines 280–284):**

The inflated `hwm1` is committed to storage regardless of direction or whether a breach occurred:

```solidity
hwmS.token0 = uint104(hwm0);
hwmS.token1 = uint104(hwm1);   // ← inflated hwm1 permanently stored
hwmS.lastDecayTs = uint32(block.timestamp);
``` [3](#0-2) 

**Post-correction breach — lines 275–278:**

After the oracle corrects, `metricT1_normal ≈ 0.5 × hwm1_inflated`. With `drawdownE6 = 50_000` (5%), the floor is `hwm1_inflated × 950_000 / 1_000_000 = 0.95 × hwm1_inflated`. Since `0.5 × hwm1_inflated < 0.95 × hwm1_inflated`, `breach1 = true`, and every `!zeroForOne` swap reverts:

```solidity
(uint256 hwm1, bool breach1) = _applyWatermark(metricT1, _decayed(hwmS.token1, decayRate, dt), floorMultiplier);
if (breach1 && !zeroForOne) {
    revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
}
``` [4](#0-3) 

**No spike filtering in the price path:**

`bidPriceX64` and `askPriceX64` fed to `afterSwap` come directly from the pool's price provider at swap time with no TWAP or median filter: [5](#0-4) [6](#0-5) 

**Design intent contradiction:**

The NatSpec states `metricToken1 breach (mid suspect-low) blocks zeroForOne == false`. But the actual behavior is the inverse: a transient HIGH oracle price inflates `hwm1`, and after correction the pool incorrectly treats the normal metric as a "suspect-low" breach, blocking `!zeroForOne` swaps permanently. [7](#0-6) 

## Impact Explanation

One swap direction (`!zeroForOne`) is permanently blocked after any transient oracle price spike. This is broken core pool functionality: traders cannot swap token1 for token0, LPs cannot rebalance, and the pool is effectively half-unusable. Recovery requires either (a) admin calling `proposeOracleStopLossHighWatermarks` + waiting the full timelock + `executeOracleStopLossHighWatermarks`, or (b) waiting for `_decayed` to erode the inflated watermark — at 58/E8 per second (~5%/day), a 2× spike takes approximately 20 days to decay below the floor. This meets the "broken core pool functionality" impact gate. [8](#0-7) 

## Likelihood Explanation

Any public trader who executes a swap during a natural oracle price spike (Chainlink/Pyth temporary high during volatile markets) triggers the watermark inflation. No oracle admin action, no malicious pool setup, and no incorrect oracle data is required — only a real transient price movement that the oracle faithfully reports. The oracle is reporting correct data; the bug is in how the extension handles the resulting metric inflation. This is a realistic market condition that occurs regularly.

## Recommendation

Before ratcheting `hwm1` upward, verify that the metric increase is consistent with genuine value growth rather than a pure oracle price move. Concretely: only ratchet `hwm1` when `metricT0` has not simultaneously decreased (since a pure mid-price increase pushes them in opposite directions). Alternatively, apply a TWAP/median filter to the oracle price before using it for watermark updates, or only update watermarks when both metrics are at or above their prior watermarks simultaneously.

## Proof of Concept

1. Deploy pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 0`.
2. Seed bin 0 with equal token0/token1 at oracle mid = `Q64` (1:1). Execute a swap; `hwm1` is set to baseline `B`.
3. Set oracle to `2·Q64` (2× spike). Execute any swap (either direction). `metricT1` doubles; `hwm1` is ratcheted to `2·B`.
4. Restore oracle to `Q64`. Attempt a `!zeroForOne` swap.
5. Assert revert with `OracleStopLossTriggered`: `metricT1_true ≈ 0.5·(2·B) = B < 0.95·(2·B) = floor`.

  /// @dev Linear decay; floors at 0 (ratchet restores from the live metric on next touch).
  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
```
