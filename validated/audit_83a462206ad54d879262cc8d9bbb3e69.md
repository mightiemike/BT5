### Title
Partial ERC4626 Vault Deposit Leaves Residual Assets Permanently Stranded in `ContractOwner` — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` moves all underlying assets out of `DirectDepositV1` into `ContractOwner` before calling `IERC4626Base.deposit`. If the target ERC4626 vault has a deposit cap and only accepts a fraction of the submitted assets, the remainder is permanently locked inside `ContractOwner` with no user-accessible recovery path. The subaccount is subsequently credited only for the partial vault-share balance, silently losing the difference.

---

### Finding Description

`wrapVaultAsset` executes the following sequence:

1. Reads `assetBalance` — the full underlying-token balance held by `directDepositV1`.
2. Guards with `previewDeposit(assetBalance) != 0` — this only confirms the vault is not fully paused; it does **not** confirm the vault can accept the **entire** `assetBalance`.
3. Calls `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr))`, which transfers the **entire** balance from `directDepositV1` to `ContractOwner` (the `msg.sender` of `withdraw`).
4. Approves `assetBalance` to the vault token.
5. Calls `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)`. [1](#0-0) 

The `IERC4626Base` interface exposed by the protocol contains only `deposit`, `previewDeposit`, and `asset` — **`maxDeposit` is absent**. [2](#0-1) 

Because `maxDeposit` is never queried, the code cannot verify that the vault can absorb the full `assetBalance`. If the vault enforces a deposit cap and only accepts `Y < assetBalance` tokens, the ERC4626 `deposit` call consumes only `Y` tokens from `ContractOwner`. The remaining `assetBalance − Y` tokens stay in `ContractOwner`.

`DirectDepositV1.withdraw` unconditionally sends the full balance to `msg.sender` (i.e., `ContractOwner`): [3](#0-2) 

After the partial deposit, `directDepositV1` holds vault shares only for `Y` tokens. When `creditDeposit` is subsequently called, it reads only the vault-share balance and credits the subaccount accordingly: [4](#0-3) 

The residual `assetBalance − Y` underlying tokens remain in `ContractOwner`. `ContractOwner` has no generic sweep or rescue function for arbitrary ERC-20 tokens held at its own address; `withdrawFromDirectDepositV1` only pulls from a `DirectDepositV1` instance, not from `ContractOwner` itself. [5](#0-4) 

The return value of `IERC4626Base.deposit` (the actual shares minted) is also discarded, so there is no post-deposit check that could detect the shortfall. [6](#0-5) 

---

### Impact Explanation

A user whose underlying assets are routed through `wrapVaultAsset` against a capacity-capped ERC4626 vault loses the portion of assets that exceed the vault's remaining capacity. Those tokens are permanently locked in `ContractOwner`. The subaccount balance in `SpotEngine` reflects only the partial deposit, understating the user's true collateral contribution. The lost tokens cannot be recovered through any user-callable path.

---

### Likelihood Explanation

`wrapVaultAsset` is an `external` function with no access control — any caller can invoke it for any subaccount at any time. [7](#0-6) 

ERC4626 vaults with deposit caps are common in production (Yearn, Aave, Morpho, etc.). The protocol's `IERC4626Base` interface deliberately omits `maxDeposit`, so the guard `previewDeposit(assetBalance) != 0` is the only check, and it passes even when the vault can only absorb a fraction of `assetBalance`. The scenario is realistic whenever a vault approaches its cap while a user has a large pending balance in their `DirectDepositV1`.

---

### Recommendation

1. Extend `IERC4626Base` to include `maxDeposit(address receiver) external view returns (uint256)`.
2. In `wrapVaultAsset`, cap the deposit amount to `min(assetBalance, maxDeposit(directDepositV1))`.
3. If `depositAmount < assetBalance`, transfer the remainder back to `directDepositV1` before returning, so it remains accessible for future `creditDeposit` calls.
4. Assert that the shares returned by `deposit` match `previewDeposit(depositAmount)` to detect any vault-side discrepancy.

---

### Proof of Concept

1. A user sends 1 000 USDC to their `directDepositV1` address.
2. The target ERC4626 vault has a remaining deposit capacity of 400 USDC.
3. Anyone calls `ContractOwner.wrapVaultAsset(subaccount, productId)`.
4. `previewDeposit(1000e6) != 0` → guard passes.
5. `DirectDepositV1.withdraw` moves all 1 000 USDC to `ContractOwner`.
6. `IERC4626Base.deposit(1000e6, directDepositV1)` — vault accepts only 400 USDC, mints shares for 400 USDC to `directDepositV1`, leaves 600 USDC in `ContractOwner`.
7. `creditDeposit()` is called; `directDepositV1` holds vault shares worth 400 USDC → subaccount is credited for 400 USDC.
8. The 600 USDC in `ContractOwner` is permanently inaccessible to the user.
