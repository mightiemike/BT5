### Title
Hardcoded `LastKnownBonded = true` in `TierRedelegate` Causes Overbilling of Bonus Rewards When Destination Validator Is Unbonded — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`TierRedelegate` unconditionally sets `LastKnownBonded = true` on the position after redelegating, regardless of the actual bonding state of the destination validator. If the destination validator is currently unbonded, the position's bonus-accrual checkpoint is poisoned: the next time `processEventsAndClaimBonus` replays events, it treats the entire unbonded gap as a bonded segment and pays out bonus rewards from the `BonusRewardsPool` for a period during which no bonded delegation existed.

---

### Finding Description

In `TierRedelegate`, after the staking redelegation is executed, the position's bonus checkpoints are updated:

```go
latestSeq, err := ms.getValidatorEventLatestSeq(ctx, dstValAddr)
// ...
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), true)   // ← hardcoded true
``` [1](#0-0) 

The third argument to `UpdateBonusCheckpoints` is `LastKnownBonded`. Passing `true` unconditionally asserts that the destination validator is bonded at the moment of redelegation. The Cosmos SDK's `BeginRedelegation` does not restrict the destination to bonded validators; a redelegation to an unbonded validator completes immediately. If the destination validator `V2` is unbonded at the time of the call, `LastKnownBonded` is set to `true` even though `V2` is not bonded.

Later, when `V2` transitions back to bonded, `AfterValidatorBonded` fires and appends a `BOND` event to `V2`'s event log:

```go
func (h Hooks) AfterValidatorBonded(ctx context.Context, _ sdk.ConsAddress, valAddr sdk.ValAddress) error {
    count, err := h.k.getPositionCountForValidator(ctx, valAddr)
    // ...
    _, err = h.k.appendValidatorEvent(ctx, valAddr, types.ValidatorEvent{
        EventType: types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND,
        // ...
    })
``` [2](#0-1) 

The position count for `V2` is non-zero because `setPosition` with `ValidatorTransition{PreviousAddress: srcValidator}` already incremented it during the redelegate call: [3](#0-2) 

When the user subsequently calls `MsgClaimTierRewards`, `processEventsAndClaimBonus` initialises the replay loop with `bonded = pos.LastKnownBonded` (i.e., `true`):

```go
bonded := pos.LastKnownBonded
segmentStart := pos.LastBonusAccrual
// ...
for _, entry := range events {
    if bonded {
        bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
        totalBonus = totalBonus.Add(bonus)
    }
    // ...
}
``` [4](#0-3) 

The loop processes the `BOND` event and computes bonus for the segment `[LastBonusAccrual, BOND_event_time]` with `bonded = true`. This entire segment corresponds to the period during which `V2` was unbonded — no bonded delegation existed, so no bonus should accrue. The defensive check `val.IsBonded()` only guards the **final** open segment after all events are replayed; it does not protect intermediate event-bounded segments inside the loop: [5](#0-4) 

The incorrect bonus is then paid out of the module's `BonusRewardsPool`: [6](#0-5) 

By contrast, `transferDelegationToPosition` (used by `CommitDelegationToTier`) explicitly enforces a bonded-only constraint on the destination validator, but `TierRedelegate` does not apply the same guard: [7](#0-6) 

---

### Impact Explanation

The `BonusRewardsPool` module account is drained by paying bonus rewards for a time window during which the position held no bonded delegation. The corrupted value is the `BonusRewardsPool` balance: it is decremented by `shares × tokensPerShare_at_bond × BonusApy × unbonded_duration / SecondsPerYear` per exploit cycle. Repeated exploitation across multiple positions or multiple unbond/rebond cycles compounds the drain. Other legitimate position holders who have earned real bonus rewards may find the pool insufficient and have their claims fail with `ErrInsufficientBonusPool`, causing a full transaction rollback that also blocks their base reward withdrawal.

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgTierRedelegate` transaction. Validators cycle between bonded and unbonded states in normal network operation (jailing, tombstoning, low self-delegation). An attacker who monitors validator state can time a redelegation to an unbonded validator and claim after it re-bonds. No privileged role, leaked key, or social engineering is required.

---

### Recommendation

After calling `redelegate`, query the actual bonding state of the destination validator and pass the correct boolean to `UpdateBonusCheckpoints`:

```go
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
```

Additionally, `validateRedelegatePosition` should be audited to confirm whether it already rejects unbonded destination validators; if not, adding that check is a defence-in-depth measure consistent with the constraint already enforced in `transferDelegationToPosition`.

---

### Proof of Concept

1. Validator `V1` is bonded; validator `V2` is unbonded (e.g., jailed).
2. User holds an active position delegating to `V1` with `LastKnownBonded = true`.
3. User submits `MsgTierRedelegate{DstValidator: V2}`.
   - `claimRewards` settles all rewards up to `T0`.
   - `stakingKeeper.BeginRedelegation` moves the delegation to `V2` (completes immediately since `V2` is unbonded).
   - `UpdateBonusCheckpoints(latestSeq, T0, true)` sets `LastKnownBonded = true`, `LastBonusAccrual = T0`.
   - `PositionCountByValidator[V2]` is incremented.
4. At time `T1 > T0`, `V2` re-bonds. `AfterValidatorBonded` fires and appends `BOND` event with timestamp `T1`.
5. User submits `MsgClaimTierRewards`.
   - `processEventsAndClaimBonus` starts with `bonded = true`, `segmentStart = T0`.
   - Processes `BOND` event at `T1`: computes `bonus = shares × rate × APY × (T1 − T0) / year` and adds to `totalBonus`.
   - `V2` is now bonded, so the final segment `[T1, blockTime]` also accrues correctly.
6. The module pays out the bonus for `[T0, T1]` from `BonusRewardsPool`, even though `V2` was unbonded during that entire interval and no bonded delegation existed.
