### Title
Unprotected `initialize()` Allows Anyone to Seize Ownership of `WithdrawPool` and Drain Pooled Collateral — (File: `core/contracts/WithdrawPool.sol`)

---

### Summary
`WithdrawPool.initialize()` is a public `external` function with no caller restriction and no deployer check. Any address that calls it before the legitimate deployer can become the contract owner and immediately drain all ERC20 collateral held in the pool via `removeLiquidity()`.

---

### Finding Description

`WithdrawPool` exposes a bare `external initialize()` that simply delegates to `BaseWithdrawPool._initialize()`: [1](#0-0) 

```solidity
contract WithdrawPool is BaseWithdrawPool {
    function initialize(address _clearinghouse, address _verifier) external {
        _initialize(_clearinghouse, _verifier);
    }
}
```

The `initializer` modifier lives only on the internal `_initialize` in `BaseWithdrawPool`, not on the public entry point: [2](#0-1) 

The outer `initialize()` carries **no access control** — no `onlyOwner`, no deployer check, no `require(msg.sender == deployer)`. Compare this with `ContractOwner.initialize()`, which explicitly enforces: [3](#0-2) 

```solidity
require(_deployer == msg.sender, "expected deployed to initialize");
```

`WithdrawPool` has no equivalent guard. An attacker who calls `initialize()` first — either by front-running the deployment transaction in the mempool or by racing against a two-step deploy-then-initialize pattern — will:

1. Become the `OwnableUpgradeable` owner (via `__Ownable_init()` inside `_initialize`).
2. Supply attacker-controlled `_clearinghouse` and `_verifier` addresses, poisoning all downstream trust assumptions.

Once owner, the attacker can immediately invoke: [4](#0-3) 

```solidity
function removeLiquidity(uint32 productId, uint128 amount, address sendTo)
    external onlyOwner {
    handleWithdrawTransfer(getToken(productId), sendTo, amount);
}
```

This transfers any ERC20 token held by the pool to an arbitrary address with no further checks.

The same unprotected pattern exists in `SpotEngine.initialize()` and `PerpEngine.initialize()`: [5](#0-4) [6](#0-5) 

Both delegate to `BaseEngine._initialize()` which carries the `initializer` modifier, but neither outer function restricts the caller.

---

### Impact Explanation

- **Direct asset theft**: `WithdrawPool` holds real ERC20 collateral deposited by users for fast-withdrawal liquidity. An attacker who seizes ownership can call `removeLiquidity()` to transfer the entire balance of any supported token to themselves.
- **Verifier substitution**: By supplying a malicious `_verifier` address, the attacker can make `submitFastWithdrawal()` accept forged multi-sig signatures, enabling unauthorized fast withdrawals.
- **Engine ownership takeover**: Seizing `SpotEngine` or `PerpEngine` ownership allows the attacker to call `addOrUpdateProduct()` with manipulated risk parameters, corrupting health calculations for all subaccounts.

---

### Likelihood Explanation

The attack window is the gap between contract deployment and the legitimate `initialize()` call. If deployment and initialization are separate transactions (common in upgradeable proxy workflows), a mempool-watching attacker can front-run the initialization. The attack requires no special privileges, no capital, and no social engineering — only the ability to submit a transaction before the deployer.

---

### Recommendation

Add a deployer check to `WithdrawPool.initialize()`, `SpotEngine.initialize()`, and `PerpEngine.initialize()`, mirroring the pattern already used in `ContractOwner`:

```solidity
// In WithdrawPool.sol
address private immutable _deployer;
constructor() { _deployer = msg.sender; }

function initialize(address _clearinghouse, address _verifier) external {
    require(msg.sender == _deployer, "only deployer can initialize");
    _initialize(_clearinghouse, _verifier);
}
```

Alternatively, move the `initializer` modifier to the public `initialize()` function and add an explicit `msg.sender == expectedDeployer` check, consistent with `ContractOwner.initialize()`.

---

### Proof of Concept

1. Deployer broadcasts a transaction to deploy `WithdrawPool` proxy.
2. Attacker observes the pending deployment in the mempool.
3. Attacker submits `WithdrawPool.initialize(attackerClearinghouse, attackerVerifier)` with higher gas, front-running the deployer's initialization call.
4. Attacker is now the `OwnableUpgradeable` owner of `WithdrawPool`.
5. Attacker calls `removeLiquidity(USDC_PRODUCT_ID, poolBalance, attackerAddress)`.
6. All USDC held in `WithdrawPool` is transferred to the attacker.
7. Legitimate `initialize()` call from the deployer reverts (OpenZeppelin `initializer` modifier blocks re-initialization), leaving the contract permanently under attacker control.
