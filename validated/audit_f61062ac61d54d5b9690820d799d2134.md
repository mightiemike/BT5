### Title
`MINIMUM_LIQUIDITY` Defined But Never Enforced — Pool Can Be Drained to Zero via `submitFastWithdrawal` - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`MINIMUM_LIQUIDITY` is declared in `Constants.sol` and imported by `BaseWithdrawPool.sol`, but is never referenced or enforced anywhere in the production codebase. The publicly callable `submitFastWithdrawal` function transfers tokens out of the pool with no post-withdrawal balance floor check, allowing the pool to be drained to zero through normal fast-withdrawal operations.

---

### Finding Description

`MINIMUM_LIQUIDITY = 10**3` is defined in `Constants.sol` line 15 and imported into `BaseWithdrawPool.sol` via `import "./common/Constants.sol"`. However, a codebase-wide search confirms the constant is referenced **only** in `Constants.sol` — it is never read, compared against, or enforced in any contract logic. [1](#0-0) 

`BaseWithdrawPool.submitFastWithdrawal` is the primary public outflow path. It accepts any caller who presents a sequencer-signed withdrawal transaction, verifies the signature, and immediately transfers `transferAmount` tokens to `sendTo` with no check that the remaining pool balance stays at or above `MINIMUM_LIQUIDITY`: [2](#0-1) 

The internal `handleWithdrawTransfer` simply calls `token.safeTransfer` with no floor guard: [3](#0-2) 

The owner-gated `removeLiquidity` path has the same absence of a minimum-balance check, though that path requires owner privilege: [4](#0-3) 

---

### Impact Explanation

Once the pool balance reaches zero (or below `MINIMUM_LIQUIDITY`), every subsequent call to `submitFastWithdrawal` will revert on the `safeTransfer` call due to insufficient token balance. This permanently breaks the fast-withdrawal service for all users until the owner manually re-funds the pool. The broken invariant is:

```
token.balanceOf(address(this)) >= MINIMUM_LIQUIDITY
```

after every outbound transfer. Because `MINIMUM_LIQUIDITY` is never enforced, this invariant can be violated by any sequence of legitimate fast withdrawals.

---

### Likelihood Explanation

The trigger path is normal protocol operation: users submit withdrawal requests, the sequencer signs them, and any fast-withdrawal provider (an unprivileged address) calls `submitFastWithdrawal`. No special privilege, governance capture, or key compromise is required beyond possessing a valid sequencer-signed transaction — which is produced as a matter of routine. A high-volume withdrawal period or a coordinated set of users requesting fast withdrawals simultaneously is sufficient to drain the pool. [5](#0-4) 

---

### Recommendation

Add a post-transfer minimum-liquidity guard inside `handleWithdrawTransfer` (or at the call sites in `submitFastWithdrawal` and `removeLiquidity`):

```solidity
function handleWithdrawTransfer(
    IERC20Base token,
    address to,
    uint128 amount
) internal virtual {
    token.safeTransfer(to, uint256(amount));
    require(
        token.balanceOf(address(this)) >= MINIMUM_LIQUIDITY,
        "Below minimum liquidity"
    );
}
```

This mirrors the fix recommended in the reference report and ensures the pool always retains a non-zero liquidity floor after every outbound transfer. [3](#0-2) [1](#0-0) 

---

### Proof of Concept

1. The `WithdrawPool` is funded with, say, 10,000 USDC by the owner.
2. Multiple users submit withdrawal requests to the sequencer; the sequencer signs each one (normal operation).
3. A fast-withdrawal provider (any address) calls `submitFastWithdrawal` repeatedly with each signed transaction, each time transferring tokens out of the pool.
4. After enough calls, `token.balanceOf(address(this))` drops to 0 — below `MINIMUM_LIQUIDITY = 1000`.
5. All subsequent `submitFastWithdrawal` calls revert on `safeTransfer`, breaking fast withdrawals for all users.
6. No check in `BaseWithdrawPool` ever compares the post-transfer balance against `MINIMUM_LIQUIDITY`. [2](#0-1) [1](#0-0)

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```
