### Title
Unbounded Loop in `reorg_chain` Enables Permanent Fork-Choice Corruption - (File: `contract/src/lib.rs`)

### Summary

The `reorg_chain` function contains two unbounded loops whose iteration count grows with fork depth. A relayer who builds a sufficiently deep fork across multiple `submit_blocks` calls can cause the single reorg-triggering transaction to exhaust NEAR's 300 TGas block gas limit, permanently preventing the contract from promoting the heavier fork to the main chain and leaving the canonical-chain pointer on a lower-work chain.

### Finding Description

`reorg_chain` is called from `submit_block_header_inner` whenever a newly submitted fork block's `chain_work` exceeds the current main chain tip's `chain_work`. [1](#0-0) 

Inside `reorg_chain`, two loops execute with no upper-bound parameter:

**Loop 1** — removes main-chain blocks above the fork tip height: [2](#0-1) 

**Loop 2** — walks every fork block from the fork tip back to the common ancestor, inserting each into the main-chain index and removing the displaced main-chain block: [3](#0-2) 

The number of iterations of Loop 2 equals the fork depth (number of fork blocks between the fork tip and the common ancestor). All fork blocks must already be present in `headers_pool` — they were accepted by prior `submit_blocks` calls. Each loop iteration performs 3–5 `LookupMap` storage writes/reads. At NEAR's storage-operation gas costs, roughly 10–20 iterations saturate the 300 TGas limit. There is no `batch_size` guard analogous to the one in `run_mainchain_gc`. [4](#0-3) 

### Impact Explanation

When the reorg transaction runs out of gas, NEAR reverts all state changes for that transaction. The fork blocks remain in `headers_pool` (they were committed in earlier transactions), but `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` are unchanged. The contract is left pointing to the lower-work chain.

Because the fork already has more cumulative work than the main chain, every subsequent block submitted on the fork will again trigger `reorg_chain` with an even longer fork, making the gas problem worse, not better. The contract cannot self-correct until the legitimate main chain accumulates enough work to exceed the fork — which may not happen if the fork represents a real network split.

During this window, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` evaluate Merkle proofs against the wrong canonical chain: [5](#0-4) 

A transaction that exists only on the fork (not on the true main chain) could pass SPV verification, and a transaction on the true main chain could fail it. This corrupts the contract's core security guarantee.

### Likelihood Explanation

`submit_blocks` is gated by the `#[trusted_relayer]` macro, which requires the caller to be a registered staking relayer or hold `Role::UnrestrictedSubmitBlocks` / `Role::DAO`: [6](#0-5) [7](#0-6) 

The prompt explicitly names "relayer-path user supplying adversarial chain data" as a valid entry path. A malicious or compromised trusted relayer can:

1. Submit a sequence of valid fork headers (each individually accepted, each within gas limits) over many `submit_blocks` calls, building up a deep fork in `headers_pool`.
2. Submit the final fork block whose cumulative `chain_work` exceeds the main chain tip, triggering `reorg_chain` with a loop depth that exceeds the gas limit.

For Dogecoin (AuxPoW merged mining), the per-block difficulty is low enough that building a 15–30 block fork is operationally feasible. For Bitcoin mainnet the bar is much higher, but the code path is identical across all chain variants.

### Recommendation

Apply the same `batch_size` pattern used in `run_mainchain_gc` to `reorg_chain`. Introduce a `reorg_batch_size` parameter (or a contract-level config field) that limits how many fork blocks are promoted per call. Persist intermediate reorg state (e.g., the current cursor hash) so the reorg can be resumed across multiple transactions. Alternatively, enforce a maximum fork depth at submission time, rejecting fork blocks that would create a fork deeper than a configurable threshold.

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and `skip_pow_verification = false` (production settings).
2. As a registered trusted relayer, submit a sequence of 20 valid fork headers (each referencing the same common ancestor on the main chain) across 20 separate `submit_blocks` calls. Each call succeeds individually.
3. Submit the 21st fork header whose cumulative `chain_work` exceeds the main chain tip's `chain_work`.
4. `submit_block_header_inner` calls `reorg_chain` with `fork_tip_header` at depth 21.
5. Loop 2 in `reorg_chain` attempts 21 iterations, each performing 4–5 `LookupMap` storage operations. The transaction runs out of gas and is reverted.
6. `mainchain_tip_blockhash` still points to the lower-work main chain.
7. Calling `verify_transaction_inclusion` with a transaction from the fork block at height N now returns `true` (the fork block is in `headers_pool` and, if it was previously on the main chain at that height, its hash may still be in `mainchain_height_to_header`), or returns an incorrect result depending on the exact reorg state, corrupting SPV proof outcomes. [8](#0-7)

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
