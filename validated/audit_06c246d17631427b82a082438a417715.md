### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, enabling full allowlist bypass via `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the end user. If the pool admin allowlists the router (the only way to permit router-mediated swaps), every user on the public internet can bypass the allowlist by calling the router.

---

### Finding Description

**Call chain:**

```
User ŌåÆ MetricOmmSimpleRouter.exactInputSingle()
         ŌåÆ pool.swap(recipient, ..., extensionData)   // msg.sender = router
              ŌåÆ _beforeSwap(msg.sender=router, recipient, ...)
                   ŌåÆ SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        ŌåÆ allowedSwapper[pool][router]  ŌåÉ checked, NOT the user
```

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no user-identity forwarding: [4](#0-3) 

The router is a **public, permissionless contract** ŌĆö anyone can call it. The pool admin has two choices, both broken:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | All router-mediated swaps revert for every user, including allowlisted ones |
| **Allowlist the router** | Every user on the internet bypasses the allowlist by routing through the router |

The protocol's own stated invariant is violated:

> "Allowlist path: deposit/swap allowlist checks must cover the exact actor/action intended and **cannot be bypassed through router**, multicall, callbacks, owner/salt separation, or alternate pool action." [5](#0-4) 

Note the contrast with `DepositAllowlistExtension`, which correctly checks `owner` (the position beneficiary) and is therefore **not** affected: [6](#0-5) 

---

### Impact Explanation

A curated pool (e.g., KYC-only, institutional, or RWA pool) deploys `SwapAllowlistExtension` to restrict trading to approved addresses. The pool admin allowlists the router so approved users can use the standard UX. Any non-approved user calls `MetricOmmSimpleRouter.exactInputSingle` with the curated pool as target. The extension sees `sender = router`, which is allowlisted, and the swap executes. The non-approved user trades on a pool they were explicitly barred from, draining liquidity or extracting value from LP positions that were priced for a restricted counterparty set. This is a direct loss of LP principal / unauthorized access to restricted pool assets, qualifying as **High** under the contest's "direct loss of user principal above Sherlock thresholds" gate.

---

### Likelihood Explanation

**High.** The `MetricOmmSimpleRouter` is a deployed, public, permissionless contract. No special setup is required beyond knowing the pool address. Any user who is blocked by the allowlist can trivially re-route through the router. The only precondition ŌĆö that the router is allowlisted ŌĆö is the natural operational state for any pool that intends to support router-mediated swaps.

---

### Recommendation

**Short term:** Change `SwapAllowlistExtension.beforeSwap` to check `recipient` if the intent is to gate the output beneficiary, or require the actual user identity to be passed in `extensionData` (signed or verified by the router) and decode it in the extension.

**Long term:** The extension framework should expose the true initiating user. One approach: the router encodes `msg.sender` into `extensionData` before forwarding to the pool, and the extension verifies the signature or trusts only the factory-registered router. Alternatively, gate by `recipient` (the address receiving output tokens) rather than `sender` (the intermediary contract), since `recipient` is always set by the end user and cannot be spoofed by the router.
