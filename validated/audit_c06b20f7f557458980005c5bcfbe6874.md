### Title
Fork Difficulty Calculation Uses Mainchain Ancestor Instead of Fork Ancestor, Causing Valid Fork Blocks to Be Rejected - (File: contract/src/dogecoin.rs, contract/src/bitcoin.rs, contract/src/litecoin.rs)

---

### Summary

When `get_next_work_required` computes the expected difficulty for a fork block at a difficulty-adjustment boundary, it calls `get_header_by_height` to retrieve the first block of the difficulty period. That function exclusively reads from `mainchain_height_to_header`, so it always returns the **mainchain** block at that height — not the fork's actual ancestor. If the fork diverged before the start of the difficulty period, the fork's ancestor has a different timestamp, the expected `bits` value is computed incorrectly, and the fork block is rejected even though it is fully valid under the fork chain's own consensus rules.

---

### Finding Description

`get_header_by_height` in `contract/src/lib.rs` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // mainchain-only map
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This is called inside `get_next_work_required` in all three chain modules at the difficulty-adjustment boundary:

**Bitcoin** (`contract/src/bitcoin.rs`):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [2](#0-1) 

**Dogecoin** (`contract/src/dogecoin.rs`):
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

**Litecoin** (`contract/src/litecoin.rs`):
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

The developers themselves flagged this with a `TODO` comment in the Dogecoin module, confirming awareness of the ambiguity.

After computing the (wrong) expected bits, `check_pow` enforces a strict equality:

```rust
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
``` [5](#0-4) 

A fork block whose `bits` field was set by the miner according to the fork chain's actual difficulty will not match the value computed from the mainchain ancestor, so `submit_block_header` panics and the block is rejected.

**Dogecoin-specific severity**: After block 145,000, Dogecoin adjusts difficulty every single block (`difficulty_adjustment_interval = 1`). [6](#0-5) 

With `blocks_to_go_back = 1`, `height_first = prev_block_header.block_height - 1`. For the third fork block onward (fork tip at height H+3, prev at H+2), `height_first = H+1`, which is a fork block — but `get_header_by_height` returns the mainchain block at H+1 instead. This means **every Dogecoin fork block from the third one onward** is validated against the wrong ancestor timestamp.

---

### Impact Explanation

A valid fork block that carries the correct `bits` value (computed from the fork's own ancestor chain) is rejected by the contract because the contract computes expected difficulty using the mainchain block at the same height. If the fork accumulates more chainwork than the mainchain, the contract cannot reorg to it, permanently locking the canonical chain pointer to a lower-work chain. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against the true heaviest chain will receive incorrect results (false negatives), breaking the trust-minimized bridge guarantee.

---

### Likelihood Explanation

For **Dogecoin** (post-145,000): any fork longer than two blocks triggers the bug on every subsequent block submission. This is the normal operating scenario for any competing chain tip. For **Bitcoin/Litecoin**: the fork must predate the start of the current 2016-block difficulty window, which is less frequent but realistic during deep reorgs or adversarial chain-split scenarios. The entry point (`submit_blocks`) is open to any trusted relayer and, depending on deployment, to any NEAR caller with the `UnrestrictedSubmitBlocks` role.

---

### Recommendation

Replace the mainchain-only `get_header_by_height` lookup with a backwards traversal along the fork's own ancestor chain to find the block at `height_first`. Since all fork blocks are stored in `headers_pool` with their `prev_block_hash` links, the correct ancestor can be reached by walking `get_prev_header` from `prev_block_header` until `block_height == height_first`. This mirrors how `get_median_time_past` already correctly traverses the fork chain via `get_prev_header` rather than `get_header_by_height`. [7](#0-6) 

---

### Proof of Concept

**Dogecoin scenario (post-145,000, difficulty adjusts every block):**

1. Contract is initialized with mainchain blocks at heights 145,000–145,011.
2. Relayer submits a fork block `F1` at height 145,012 (diverges from mainchain; stored as fork block).
3. Relayer submits fork block `F2` at height 145,013 (prev = `F1`).
4. Relayer submits fork block `F3` at height 145,014 (prev = `F2`).
   - `get_next_work_required` is called with `prev_block_header = F2` (height 145,013).
   - `height_first = 145,013 - 1 = 145,012`.
   - `get_header_by_height(145,012)` returns the **mainchain** block at 145,012 (call it `M1`), not the fork block `F1`.
   - `M1.time ≠ F1.time` (they are different blocks).
   - `calculate_next_work_required` produces `expected_bits` based on `M1.time`.
   - `F3.bits` was set by the miner using `F1.time` → `expected_bits ≠ F3.bits`.
   - `require!(expected_bits == block_header.bits, ...)` panics.
   - `F3` is rejected despite being a fully valid Dogecoin block under the fork chain's consensus rules.

pub fn get_median_time_past(
    block_header: ExtendedHeader,
    prev_block_getter: &impl BlocksGetter,
) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}
