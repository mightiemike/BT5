### Title
Unprivileged Caller Can Prematurely Drain the Entire Mainchain Block History via `run_mainchain_gc` - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function that carries no caller-identity guard. Any unprivileged NEAR account can call it with `batch_size = u64::MAX` to permanently delete every GC-eligible mainchain block header from the contract's storage in a single transaction. This irreversibly destroys the ability to verify SPV proofs for those blocks, breaking the core trust guarantee of the light client.

---

### Finding Description

`submit_blocks` is correctly gated behind `#[trusted_relayer]`, which enforces that only staked, authorized relayers may mutate chain state. However, `run_mainchain_gc` — which also permanently mutates chain state — carries only a `#[pause]` attribute:

```rust
// contract/src/lib.rs:376-377
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` macro only gates the function when the contract is in a paused state. When the contract is live (the normal operating condition), **any NEAR account with zero privileges can call this function with any `batch_size`**.

Inside the function, the attacker-controlled `batch_size` is used directly to compute how many blocks to remove:

```rust
// contract/src/lib.rs:392-393
let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
```

Passing `u64::MAX` as `batch_size` causes `selected_amount_to_remove` to equal `total_amount_to_remove` — the maximum number of blocks eligible for removal. The loop then permanently deletes every eligible block from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and advances `mainchain_initial_blockhash` to the new oldest block:

```rust
// contract/src/lib.rs:401-414
for height in start_removal_height..end_removal_height {
    let blockhash = &self.mainchain_height_to_header
        .get(&height).unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
    self.remove_block_header(blockhash);
    self.mainchain_height_to_header.remove(&height);
}
self.mainchain_initial_blockhash = self.mainchain_height_to_header
    .get(&end_removal_height).unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
```

`remove_block_header` deletes from both `mainchain_header_to_height` and `headers_pool`. There is no mechanism to re-insert pruned blocks. The deletion is permanent.

The contrast with the intended design is explicit: `submit_blocks` calls `run_mainchain_gc` internally with `batch_size = num_of_headers` (a small, bounded value equal to the number of headers just submitted). The external public entry point was intended to allow manual GC triggering but was never restricted to authorized callers.

---

### Impact Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both require the target block's hash to be present in `mainchain_header_to_height`:

```rust
// contract/src/lib.rs:299-301
let target_block_height = self.mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
```

Once a block is pruned by `run_mainchain_gc`, its hash is removed from `mainchain_header_to_height`. Any subsequent SPV proof call for that block panics with "block does not belong to the current main chain". This failure is permanent — the block cannot be re-submitted.

For any downstream contract (e.g., a cross-chain bridge) that calls `verify_transaction_inclusion` to authorize a transfer, an attacker can permanently block a specific user's cross-chain action by front-running the verification call with a `run_mainchain_gc(u64::MAX)` call that prunes the target block. The user's Bitcoin transaction is confirmed on-chain, but the NEAR-side proof can never succeed.

Additionally, the CLAUDE.md documentation explicitly notes that premature GC breaks chain reorganization:

> "If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor."

An attacker can therefore also prevent legitimate fork resolution by aggressively pruning blocks before a reorg is processed.

---

### Likelihood Explanation

The attack requires no privileged role, no staked funds, and no special knowledge beyond knowing the contract address. Any NEAR account can call `run_mainchain_gc` at any time the contract is not paused. The precondition — that the chain has grown beyond `gc_threshold` — is met in normal operation (the recommended `gc_threshold` is 52,704 blocks, approximately one year; after that point the attack window is permanently open). The call costs only standard NEAR gas fees.

---

### Recommendation

Add the same `#[trusted_relayer]` guard (or at minimum a dedicated role check) to `run_mainchain_gc` that is applied to `submit_blocks`. If the intent is to allow any caller to trigger GC (e.g., to reclaim storage), enforce a strict upper bound on `batch_size` per call and document the trust model explicitly. The simplest fix consistent with the existing access-control pattern:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
#[trusted_relayer]   // add this
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and allow the relayer to sync the chain until `mainchain_size > gc_threshold` (i.e., more than 52,704 blocks are stored).
2. As any unprivileged NEAR account, call:
   ```
   run_mainchain_gc({ "batch_size": 18446744073709551615 })
   ```
3. Observe that `get_mainchain_size()` drops to exactly `gc_threshold` in one transaction.
4. Attempt to call `verify_transaction_inclusion` for any block that was in the pruned range. The call panics with "block does not belong to the current main chain".
5. The pruned blocks cannot be recovered; the SPV proof for those blocks is permanently invalidated.

The test at `contract/tests/test_basics.rs:515-521` already demonstrates that an unprivileged `user_account` (not the contract itself) can successfully call `run_mainchain_gc` with an arbitrary `batch_size`, confirming the entry path is reachable without any role.
