### Title
Whitelisted `canApplyDeltas` Addresses in `BaseEngine` Cannot Be Revoked Once Granted — (`File: core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine._initialize()` permanently grants three addresses the ability to call `updateBalance` on the spot and perp engines via the `canApplyDeltas` mapping. No function exists to revoke this privilege. If any whitelisted contract is compromised or replaced, its old address retains unrestricted balance-write access indefinitely.

---

### Finding Description

In `BaseEngine.sol`, the `canApplyDeltas` mapping is a `mapping(address => bool)` that gates all balance mutations on the engines. It is populated once during `_initialize()`: [1](#0-0) 

```solidity
canApplyDeltas[_endpointAddr] = true;
canApplyDeltas[_clearinghouseAddr] = true;
canApplyDeltas[_offchainExchangeAddr] = true;
```

Access is enforced by `_assertInternal()`: [2](#0-1) 

```solidity
function _assertInternal() internal view virtual {
    require(canApplyDeltas[msg.sender], ERR_UNAUTHORIZED);
}
```

There is no function anywhere in `BaseEngine`, `SpotEngine`, or `PerpEngine` that sets `canApplyDeltas[addr] = false`. The grep for `canApplyDeltas[` returns matches exclusively inside `BaseEngine.sol`, confirming no revocation path exists. [3](#0-2) 

---

### Impact Explanation

`canApplyDeltas` is the sole on-chain access control protecting `updateBalance` on both `SpotEngine` and `PerpEngine`. Any address holding `canApplyDeltas[addr] == true` can arbitrarily credit or debit any subaccount's balance. If one of the three whitelisted contracts (`Endpoint`, `Clearinghouse`, `OffchainExchange`) is compromised — whether through a bug in that contract or a malicious upgrade — the protocol has no mechanism to revoke its engine-write access. The compromised contract can drain all user collateral and perp positions. The corrupted state is: `SpotEngine` and `PerpEngine` balance mappings for all subaccounts.

---

### Likelihood Explanation

Medium. The three whitelisted contracts are complex, upgradeable, and sequencer-driven. A bug in any one of them (e.g., an unchecked `updateBalance` call path) that is discovered post-deployment cannot be mitigated by revoking engine access — the only recourse is a full engine upgrade, which itself requires governance action and leaves a window of exposure. This mirrors exactly the Arborist scenario: the admin wants to revoke a misbehaving privileged actor but has no on-chain mechanism to do so.

---

### Recommendation

Add an owner-controlled function to `BaseEngine` that allows toggling `canApplyDeltas`:

```solidity
function setCanApplyDeltas(address addr, bool status) external onlyOwner {
    if (addr == address(0)) revert();
    canApplyDeltas[addr] = status;
}
```

This mirrors the recommended fix from the reference report: accept a `bool status` parameter so the same function handles both granting and revoking access.

---

### Proof of Concept

1. `Endpoint`, `Clearinghouse`, and `OffchainExchange` are whitelisted in `BaseEngine._initialize()`. [4](#0-3) 

2. `_assertInternal()` is the only guard on `updateBalance`; it checks `canApplyDeltas[msg.sender]`. [2](#0-1) 

3. Suppose `OffchainExchange` is found to have a bug allowing an attacker to trigger an arbitrary `updateBalance` call. The protocol owner calls `setCanApplyDeltas(offchainExchange, false)` — **this function does not exist**. The compromised `OffchainExchange` address retains `canApplyDeltas[offchainExchange] == true` permanently.

4. The attacker exploits the `OffchainExchange` bug to call `updateBalance` on `SpotEngine`, crediting their subaccount with arbitrary collateral, then withdraws via `WithdrawCollateral`.

5. No on-chain action short of upgrading the engine contracts themselves can stop this.
