Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual User, Enabling Full Allowlist Bypass via Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
When a swap is routed through `MetricOmmSimpleRouter`, the `SwapAllowlistExtension.beforeSwap` hook receives the router's address as `sender` rather than the originating user's address. The allowlist check is therefore performed against the router contract, not the actual trader. If the pool admin allowlists the router — a natural and expected action to enable router-mediated swaps on a curated pool — every unprivileged user can bypass the swap allowlist entirely by routing through the router.

## Finding Description

**Call path:**

1. Unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutputSingle` / `exactOutput`).
2. The router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — so `msg.sender` seen by the pool is the **router address**, not the user.
3. `MetricOmmPool.swap` invokes `_beforeSwap(msg.sender, ...)` passing the router address as `sender` (`MetricOmmPool.sol` line 230–240).
4. `ExtensionCalling._beforeSwap` ABI-encodes and forwards `sender` = router address to every configured extension (`ExtensionCalling.sol` lines 149–177).
5. `SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = **router address**. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Broken invariant:** The extension is supposed to gate the economically relevant actor — the user who initiates and pays for the swap. Instead it gates the router contract, which is a shared, permissionless intermediary.

**Two concrete failure modes:**

| Pool admin configuration | Result |
|---|---|
| `allowedSwapper[pool][router] = true` (to enable router swaps) | Every unprivileged user bypasses the allowlist by routing through the router |
| Only individual users allowlisted, router not allowlisted | Allowlisted users **cannot** use the router at all — broken core functionality |

**Why existing guards are insufficient:** The pool's `swap` function correctly passes `msg.sender` as `sender` to the extension. There is no mechanism in the router or the extension to recover the original `tx.origin` or the user's address in a trustworthy, non-spoofable way. The extension has no access to the router's transient callback context where the real payer is stored (`TransientCallbackPool.getPayer()`).

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to approved counterparties loses that protection entirely for all router-mediated swaps once the router is allowlisted. Any unprivileged user can trade on the pool by calling `MetricOmmSimpleRouter.exactInputSingle`. This constitutes broken core pool functionality (the allowlist gate) and an admin-boundary break where the intended access control policy is bypassed by an unprivileged public path. Severity: **Medium** — the allowlist policy is fully defeated; direct token loss per swap is not guaranteed but the protective invariant the pool was configured to enforce is nullified.

## Likelihood Explanation
The pool admin must allowlist the router to permit any router-mediated swap on the curated pool. This is a natural, expected configuration step — not a malicious or unusual action. Once done, the bypass is unconditional and repeatable by any caller with no special capability. Any user who knows the pool uses the allowlist extension can trivially route through the router to circumvent it.

## Recommendation
The `SwapAllowlistExtension` should not rely solely on the `sender` parameter (the direct pool caller) for identity. Two viable fixes:

1. **Extension-side:** Accept an optional `bytes calldata extensionData` payload carrying the verified original user address, signed or forwarded by the router in a trustworthy way, and check that address instead of `sender` when `sender` is a known router.
2. **Router-side:** The router should expose the originating user's address through a standardised mechanism (e.g., appending it to `extensionData`) so extensions can recover it. The extension then checks the embedded user address when `sender` matches a known router.
3. **Documentation/deployment guard:** At minimum, document that `SwapAllowlistExtension` cannot enforce per-user policy for router-mediated swaps and that allowlisting the router opens the pool to all router callers.

## Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension; only `alice` is allowlisted.
swapExtension.setAllowedToSwap(address(pool), alice, true);
// Pool admin also allowlists the router to enable router swaps.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attack: unprivileged `eve` (not allowlisted) routes through the router.
vm.startPrank(eve);
token0.approve(address(router), type(uint256).max);
// This should revert NotAllowedToSwap — but it succeeds because
// beforeSwap sees sender = router address, which IS allowlisted.
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    recipient: eve,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// Eve successfully swaps despite not being on the allowlist.
```
