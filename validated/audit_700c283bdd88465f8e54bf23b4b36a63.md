### Title
`verify_transaction_inclusion` Rejects Valid Proofs for Single-Transaction Blocks — (`contract/src/lib.rs`)

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` unconditionally reject empty merkle proofs via a hardcoded `require!`. For a Bitcoin block containing exactly one transaction (the coinbase), the merkle proof is legitimately empty — the merkle root is the transaction hash itself. This makes both public SPV verification entry points completely broken for single-transaction blocks.

### Finding Description

In `contract/src/lib.rs` at line 315, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This check unconditionally panics when the caller supplies an empty proof slice. However, for a block whose merkle tree has exactly one leaf (a single coinbase transaction), the merkle proof is correctly and necessarily empty: the merkle root equals the transaction hash directly, with no sibling hashes required. The underlying `compute_root_from_merkle_proof` function handles this case correctly — when `merkle_proof` is empty it simply returns `transaction_hash` unchanged — but the `require!` guard fires before that function is ever reached. [2](#0-1) 

`verify_transaction_inclusion_v2`, the non-deprecated replacement, inherits the same defect because it delegates to `verify_transaction_inclusion` as its final step:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The length-equality pre-check in `verify_transaction_inclusion_v2` passes for a single-transaction block (both `merkle_proof` and `coinbase_merkle_proof` are empty, so `0 == 0`), and the coinbase root check also passes (empty proof returns `coinbase_tx_id`, which equals `merkle_root` for a one-tx block). The call then falls through to `verify_transaction_inclusion`, which panics. [4](#0-3) 

### Impact Explanation

Any NEAR contract or unprivileged caller that invokes `verify_transaction_inclusion_v2` (or the deprecated `verify_transaction_inclusion`) with a legitimately valid proof for a transaction in a single-transaction Bitcoin block will always receive a contract panic. The proof result is corrupted from "true" to "panic/revert", making the entire SPV verification subsystem unusable for this class of blocks. Downstream contracts that gate actions on a `true` return value will be permanently blocked for those blocks.

### Likelihood Explanation

Bitcoin miners routinely produce empty blocks (containing only the coinbase transaction) when they find a block before receiving the previous block's transactions, or during periods of low mempool activity. These are valid, confirmed mainchain blocks. A relayer will submit them to the contract, they will be stored successfully, and any subsequent attempt to verify the coinbase transaction's inclusion will always panic. The entry path requires no privilege: `verify_transaction_inclusion_v2` is a public, unguarded (only `#[pause]`) method callable by any NEAR account. [5](#0-4) 

### Recommendation

Remove the blanket `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. An empty proof is valid when `tx_index == 0` and the block contains exactly one transaction. The `compute_root_from_merkle_proof` function already handles the empty-proof case correctly and will return the transaction hash unchanged, which will then be compared against `merkle_root` — producing the correct result without any special-casing.

### Proof of Concept

1. Submit a real Bitcoin empty block (coinbase-only) to the contract via `submit_blocks`. The block is accepted and stored normally.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id` = coinbase txid (= `merkle_root` of that block)
   - `tx_block_blockhash` = the block's hash
   - `tx_index` = 0
   - `merkle_proof` = `[]`
   - `coinbase_tx_id` = same coinbase txid
   - `coinbase_merkle_proof` = `[]`
   - `confirmations` = 1
3. The length-equality check passes (`0 == 0`). The coinbase root check passes (`compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id == merkle_root`). Execution reaches `verify_transaction_inclusion`, which panics: `"Merkle proof is empty"`.
4. The transaction is genuinely included in the block, the proof is mathematically correct, yet the contract always panics — mirroring the ConvexSpell pattern where a valid but "special-case" input causes an unconditional revert. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,

        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
