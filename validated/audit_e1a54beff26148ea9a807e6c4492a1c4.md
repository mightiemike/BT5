### Title
SHA-1 Used for Sub-Tree Identity in Back-Reference Serialization Cache — (`File: src/serde/tree_cache.rs`)

### Summary

`TreeCache`, the data structure used to deduplicate sub-trees during back-reference compressed serialization, uses **SHA-1** (a cryptographically broken 160-bit hash) to identify and merge identical atoms. A SHA-1 collision between two distinct attacker-controlled atoms causes the serializer to emit a back-reference to the wrong atom, producing serialized bytes that decode to a structurally different CLVM program than the one that was serialized.

### Finding Description

In `src/serde/tree_cache.rs`, the `hash_atom` function hashes atoms using SHA-1 and produces a 20-byte digest (`Bytes20`):

```rust
type Bytes20 = [u8; 20];

fn hash_atom(salt: &[u8], blob: &[u8]) -> Bytes20 {
    let mut ctx = Sha1::default();
    ctx.update(salt);
    ctx.update(blob);
    ctx.finalize().into()
}
``` [1](#0-0) 

These 20-byte SHA-1 digests are used as the key in `atom_lookup`, the HashMap that deduplicates atoms across the entire tree:

```rust
atom_lookup: HashMap<Bytes20, u32, RandomState>,
``` [2](#0-1) 

When a new atom is processed, the code checks `atom_lookup` for an existing entry with the same hash. If found, the new atom's `NodePtr` is silently mapped to the **existing** `NodeEntry` — the one belonging to the first atom:

```rust
let ne = match self.atom_lookup.entry(hash) {
    Entry::Occupied(ne) => {
        // we already have a node with this hash
        let idx = *ne.get();
        e.insert(idx);
        stack.push(idx);
        continue;
    }
    Entry::Vacant(ne) => ne,
};
``` [3](#0-2) 

The code's own comment acknowledges SHA-1 is broken and attempts to mitigate it with an 8-byte random salt:

```rust
/// We compute hash-trees using SHA-1 in order to determine whether the
/// trees are identical or not. To mitigate malicious SHA-1 hash collisions,
/// we salt the hashes
salt: [u8; 8],
``` [4](#0-3) 

The salt is generated at `TreeCache::new()` time using `rand::rng()`:

```rust
salt: rng.random(),
``` [5](#0-4) 

The `NodeEntry` stores the SHA-1 digest as the node's `tree_hash`, which is then used by `find_path` to locate back-reference candidates:

```rust
tree_hash: Some(hash),
``` [6](#0-5) 

`TreeCache` is part of the public API exported from `src/serde/mod.rs` and is used by the incremental serializer (`Serializer`): [7](#0-6) 

### Impact Explanation

If two distinct atoms A and B produce the same SHA-1 digest under the session salt, `atom_lookup` maps B's `NodePtr` to A's `NodeEntry`. When `find_path` is subsequently called for B, it returns a path to A. The serializer emits `0xfe <path-to-A>` instead of serializing B's bytes. A deserializer reading those bytes reconstructs A at the position where B should appear — a **different CLVM tree** than the one that was serialized.

The corrupted result is a concrete, identifiable artifact: the `[u8; 32]` SHA-256 tree hash of the deserialized program will differ from the SHA-256 tree hash of the original program. Any system that computes a program ID or coin puzzle hash from the re-serialized bytes will derive a wrong value, causing **consensus divergence** between nodes that serialize with back-references and those that do not.

### Likelihood Explanation

SHA-1 has been practically broken since the SHAttered attack (2017), which demonstrated chosen-prefix collisions. The 8-byte random salt raises the bar: an attacker cannot pre-compute a collision without knowing the salt. However:

- The salt is only 64 bits. If the salt can be observed or inferred (e.g., through repeated serialization calls on a long-lived `Serializer` instance), the search space collapses.
- SHA-1's internal structure is broken regardless of salting; chosen-prefix techniques can be adapted to salted variants with sufficient compute.
- The `TreeCache` is exported as a public API and can be instantiated by callers who may reuse a single instance across many programs, giving an attacker more oracle queries.

Likelihood is **low but non-zero** for a well-resourced attacker targeting a specific node.

### Recommendation

Replace SHA-1 with SHA-256 (or at minimum BLAKE3) for the `hash_atom` function and the `Bytes20` / `atom_lookup` types in `TreeCache`. The rest of the codebase already uses SHA-256 for all consensus-critical tree hashing (`tree_hash_atom`, `tree_hash_pair`, `ObjectCache<treehash>`). The internal deduplication cache should use the same primitive to eliminate the broken-hash surface entirely. The salt can be retained as an additional defense-in-depth measure.

### Proof of Concept

1. Instantiate a `TreeCache`.
2. Craft two distinct byte strings A and B such that `SHA1(salt || A) == SHA1(salt || B)` (feasible offline once the salt is known, using SHAttered-style chosen-prefix collision).
3. Build a CLVM tree `(A . B)` and call `TreeCache::update()`.
4. Simulate serialization: call `push(A_node)`, then call `find_path(B_node)`.
5. Observe that `find_path` returns a non-`None` path pointing to A, even though A ≠ B.
6. The serializer emits `0xfe <path>` for B; the deserializer reconstructs A at B's position.
7. Compute SHA-256 tree hashes of the original and deserialized trees — they differ, confirming consensus divergence. [8](#0-7) [9](#0-8) [10](#0-9)
