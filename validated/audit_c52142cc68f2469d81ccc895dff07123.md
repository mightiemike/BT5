### Title
Protobuf `ValidResourceBounds` Round-Trip Silently Downgrades `AllResources` to `L1Gas` When `l2_gas` and `l1_data_gas` Are Zero, Producing a Different Transaction Hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant (with `l2_gas = 0` and `l1_data_gas = 0`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` includes an extra `L1_DATA_GAS` element in the Poseidon preimage for `AllResources` but not for `L1Gas`, the two variants produce **different transaction hashes** even when all numeric field values are identical. Any V3 transaction submitted via the gateway with `AllResources(l2_gas=0, l1_data_gas=0)` receives hash H1 at the gateway; after a protobuf round-trip over P2P or block-sync, the receiving node recomputes hash H2 ≠ H1. This is a canonical hash-domain collision across the serialization boundary.

---

### Finding Description

**Step 1 – Originating node computes hash with `AllResources` path.**

`InternalRpcInvokeTransactionV3` always stores `resource_bounds: AllResourceBounds`, so `resource_bounds()` always returns `ValidResourceBounds::AllResources(...)`. [1](#0-0) 

`get_tip_resource_bounds_hash` for `AllResources` appends a third element `concat(l1_data_gas, L1_DATA_GAS)` to the Poseidon preimage: [2](#0-1) 

So for `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` the preimage is:
```
poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS), concat(0, L1_DATA_GAS))  → H1
```

**Step 2 – Serialization is lossy.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `L1Gas(X)` and `AllResources(X, 0, 0)` to an identical protobuf message: all three fields present, `l2_gas = 0`, `l1_data_gas = 0`. [3](#0-2) 

**Step 3 – Deserialization silently downgrades the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies `unwrap_or_default()` to `l1_data_gas` and then checks `if l1_data_gas.is_zero() && l2_gas.is_zero()`. When both are zero it returns `ValidResourceBounds::L1Gas(l1_gas)` — discarding the `AllResources` semantics: [4](#0-3) 

**Step 4 – Receiving node computes a different hash.**

After deserialization the receiving node calls `get_invoke_transaction_v3_hash` with `ValidResourceBounds::L1Gas(X)`. The `L1Gas` branch of `get_tip_resource_bounds_hash` omits the `L1_DATA_GAS` element:

```
poseidon(tip, concat(X, L1_GAS), concat(0, L2_GAS))  → H2
```

H1 ≠ H2 because the Poseidon hash over three elements differs from the hash over two elements even when the third element is zero. [5](#0-4) 

**Step 5 – Hash divergence propagates.**

The transaction hash is not included in the P2P mempool protobuf (`transaction_hash: None`), so the receiving node recomputes it and stores H2. [6](#0-5) 

The hash is recomputed inside `convert_rpc_tx_to_internal` on the receiving node: [7](#0-6) 

The blockifier's `__validate__` entry point receives the stored hash as part of `TxInfo`. If the stored hash is H2 but the user signed over H1, signature verification fails and the transaction reverts on every node that received it over P2P.

---

### Impact Explanation

**Critical – Wrong revert result from blockifier execution logic for accepted input.**

A V3 invoke transaction with `AllResources(l2_gas=0, l1_data_gas=0)` is valid and accepted by the gateway. On the originating node the blockifier executes it successfully (hash H1 matches the signature). On every node that received the transaction over P2P the blockifier executes it with hash H2 ≠ H1, causing `__validate__` to fail and the transaction to revert. The two nodes produce different receipts and different state roots for the same block, breaking consensus.

Additionally, if the block-sync path recomputes transaction hashes for verification, nodes syncing the block will compute H2 for this transaction and reject the block as having an invalid transaction hash, causing a chain split.

---

### Likelihood Explanation

Any user who submits a V3 invoke transaction and sets `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0` (a natural choice when only paying for L1 gas) triggers this path. The `InternalRpcInvokeTransactionV3` struct enforces `AllResourceBounds` for all V3 transactions, so the gateway always computes H1. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is trivially satisfied by any such transaction.

---

### Recommendation

Remove the variant-downgrade logic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The `L1Gas` vs `AllResources` distinction must be preserved across the serialization boundary because it changes the transaction hash. Two options:

1. **Add a discriminant field to the protobuf `ResourceBounds` message** (e.g., a boolean `is_all_resources`) so the deserializer can reconstruct the correct variant without inspecting numeric values.

2. **Make `get_tip_resource_bounds_hash` produce the same hash for `L1Gas(X)` and `AllResources(X, 0, 0)`** by always including the `L1_DATA_GAS` element (even when zero) for all V3 transactions. This eliminates the hash-domain split at the cost of a one-time hash migration for pre-existing `L1Gas` transactions.

Until fixed, the `unwrap_or_default()` fallback on line 427 should at minimum be changed to an explicit error when `l1_data_gas` is absent, so that the ambiguous case is rejected rather than silently misclassified: [8](#0-7)

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, GasAmount, GasPrice, ResourceBounds, ValidResourceBounds,
};
use apollo_protobuf::protobuf;

fn main() {
    // Originating node: AllResources with zero l2_gas and l1_data_gas.
    let original = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas:      ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
        l2_gas:      ResourceBounds::default(),   // zero
        l1_data_gas: ResourceBounds::default(),   // zero
    });

    // Serialize to protobuf (as happens during P2P propagation).
    let proto: protobuf::ResourceBounds = original.into();

    // Deserialize on receiving node.
    let received: ValidResourceBounds = proto.try_into().unwrap();

    // The variant has changed: AllResources → L1Gas.
    assert!(
        matches!(received, ValidResourceBounds::L1Gas(_)),
        "Receiving node silently downgraded AllResources to L1Gas"
    );

    // Hash divergence: compute hashes for both variants.
    use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
    use starknet_api::transaction::fields::Tip;

    let tip = Tip(0);
    let h1 = get_tip_resource_bounds_hash(&original,  &tip).unwrap();
    let h2 = get_tip_resource_bounds_hash(&received,  &tip).unwrap();

    assert_ne!(h1, h2,
        "Same numeric values, different variants → different transaction hashes: \
         gateway={h1:?}, p2p-node={h2:?}");
}
```

The assertion `h1 ≠ h2` holds because `get_tip_resource_bounds_hash` for `AllResources` chains three resource felts (L1_GAS, L2_GAS, L1_DATA_GAS) while for `L1Gas` it chains only two (L1_GAS, L2_GAS). The Poseidon hash of a three-element sequence differs from the hash of a two-element sequence even when the third element is zero.

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
