Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks `sender` (direct pool caller) instead of the originating user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `allowedSwapper[pool][sender]` where `sender` is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` collapses to the router address. Any pool admin who allowlists the router (the only way to let allowlisted users use the router) simultaneously grants every address on-chain the ability to bypass the per-user swap restriction by routing through the router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this as `sender` to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle()` is called, it calls `pool.swap()` directly with `msg.sender = router`: [4](#0-3) 

So `sender` in the extension is the router address, not the originating user. If the admin allowlists the router (`allowedSwapper[pool][router] = true`), the check passes for **every** caller regardless of whether they are individually allowlisted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the economic beneficiary), not `sender` (the direct caller): [5](#0-4) 

The inescapable catch-22: if the router is not allowlisted, allowlisted users cannot use the router. If the router is allowlisted, the entire per-user restriction is defeated for all router-mediated swaps.

## Impact Explanation
The `SwapAllowlistExtension` is designed to restrict which addresses can swap in a pool — e.g., KYC-gated pools or pools where the admin is protecting LPs from specific actors. When the router is allowlisted (the expected operational configuration for any pool that wants allowlisted users to use the standard router), the guard is completely defeated. Any non-allowlisted address can call `router.exactInputSingle()` or `exactInput()` against the restricted pool and the extension will pass. This is a broken core pool functionality / admin-boundary break: the pool admin's access control is bypassed by an unprivileged trader with no special permissions.

## Likelihood Explanation
Any pool that (a) deploys `SwapAllowlistExtension` to restrict swappers and (b) wants those allowlisted users to be able to use `MetricOmmSimpleRouter` must allowlist the router. Once the router is allowlisted, the bypass is available to every address on-chain with zero additional preconditions. No privileged access, no special tokens, no oracle manipulation required — a single public call to the router suffices.

## Recommendation
The `beforeSwap` interface receives both `sender` (direct pool caller) and `recipient` (output recipient). For swaps, the economic beneficiary is `recipient`, which is always the user-supplied address regardless of routing. The minimal fix is to check `recipient` instead of `sender`, mirroring the correct pattern in `DepositAllowlistExtension`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, require the router to embed the originating user's address in `extensionData` and decode it in the extension, but Option 1 is the minimal, consistent fix.

## Proof of Concept
```
1. Admin deploys pool with SwapAllowlistExtension attached.
2. Admin calls:
       extension.setAllowedToSwap(pool, userA, true)
       extension.setAllowedToSwap(pool, router, true)   // required for userA to use router
3. Non-allowlisted userB calls:
       router.exactInputSingle({pool: pool, recipient: userB, ...})
4. Router calls pool.swap(userB, ...) with msg.sender = router.
5. Pool calls _beforeSwap(router /*sender*/, userB /*recipient*/, ...).
6. Extension evaluates: allowedSwapper[pool][router] == true → passes.
7. Swap executes. userB bypassed the per-user allowlist entirely.
```
