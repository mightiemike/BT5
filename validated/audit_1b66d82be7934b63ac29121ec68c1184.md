### Title
Global `lastLiquidationFees` Cross-Subaccount Contamination Causes Incorrect Insurance Accounting During Finalization — (`core/contracts/ClearinghouseLiq.sol` / `core/contracts/ClearinghouseStorage.sol`)

---

### Summary

`lastLiquidationFees` is a single global `int128` in `ClearinghouseStorage`, not scoped per-subaccount. Every call to `_handleLiquidationPayment` overwrites it with the fees from whatever subaccount was just liquidated. When `_finalizeSubaccount` is subsequently called for a *different* subaccount, it reads this stale value, passes a wrong insurance amount to `perpEngine.socializeSubaccount`, and permanently mutates the cumulative funding rates of all open-interest holders in every affected perp market.

---

### Finding Description

**Storage declaration** — `lastLiquidationFees` is a single protocol-wide slot: [1](#0-0) 

**Write path** — every non-finalization liquidation overwrites it: [2](#0-1) 

**Read path** — `_finalizeSubaccount` uses it to compute the effective insurance passed to socialization: [3](#0-2) 

The design intent (per the inline comment at lines 581–585) is that `lastLiquidationFees` represents the fees added to insurance by the *last liquidation step of the subaccount currently being finalized*, so that those fees are temporarily excluded from the socialization budget. The add-back at line 410 restores them after socialization: [4](#0-3) 

**The bug:** because the variable is global, any intervening liquidation of *any other subaccount* between the last liquidation step of subaccount B and the finalization call for subaccount B will silently replace the value. The finalization then uses the wrong `F` in every downstream computation.

**Call sequence that triggers the bug:**

1. `liquidateSubaccountImpl(txn_A)` — normal liquidation of subaccount A → `_handleLiquidationPayment` → `lastLiquidationFees = fees_A` (large).
2. `liquidateSubaccountImpl(txn_B, productId=uint32.max)` — finalization of subaccount B → `_finalizeSubaccount` reads `lastLiquidationFees = fees_A` (stale).

---

### Impact Explanation

Let `I` = current `insurance`, `F_stale` = stale fees from subaccount A, `F_correct` = fees from the last liquidation of subaccount B.

**Effect 1 — Wrong insurance budget passed to `perpEngine.socializeSubaccount`:**

`perpEngine.socializeSubaccount` receives `I − F_stale` instead of `I − F_correct`. [5](#0-4) 

Inside `socializeSubaccount`, for each perp product with a negative `vQuoteBalance`, the function first draws from the insurance budget, then socializes any remainder by adjusting `cumulativeFundingLongX18` / `cumulativeFundingShortX18` across all open-interest holders. If `F_stale > F_correct`, the insurance budget is artificially deflated, so:
- Less insurance is consumed to cover negative vQuoteBalances.
- More residual loss is socialized via funding-rate mutation, permanently diluting every open-interest holder in those markets.
- The final `insurance` state ends up *higher* than it should be (the protocol retains funds it should have spent), while counterparties absorb losses they should not have borne — a direct bad-debt misallocation.

**Effect 2 — Wrong `canLiquidateMore` gate, skipping the zero-spot-balance check:** [6](#0-5) 

If `F_stale` is large enough to make `(quoteBalance.amount + I − F_stale) ≤ 0`, `canLiquidateMore` is forced to `false`. The `require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT)` guard for non-zero-weight spot products is then skipped entirely. A subaccount that still holds spot liabilities (negative balances) that should have been zeroed before finalization can pass through, leaving the spot engine in an inconsistent state.

---

### Likelihood Explanation

The trigger requires two consecutive liquidation transactions where the first is for a different subaccount than the second. In any active liquidation environment — or with a liquidator bot that processes multiple accounts — this ordering arises naturally without any adversarial intent. A deliberate attacker who controls a liquidator key can also manufacture it: submit a liquidation of any under-maintenance account (including one they themselves created and drove under water) to set `lastLiquidationFees` to a chosen large value, then immediately submit the finalization for the target subaccount. The sequencer processes transactions in submission order, so back-to-back submission is sufficient.

---

### Recommendation

Replace the single global `lastLiquidationFees` with a per-subaccount mapping:

```solidity
// ClearinghouseStorage.sol
mapping(bytes32 => int128) internal lastLiquidationFees;
```

In `_handleLiquidationPayment`, key the write by `txn.liquidatee`:
```solidity
lastLiquidationFees[txn.liquidatee] = v.liquidationFees;
```

In `_finalizeSubaccount`, read and clear the per-subaccount slot:
```solidity
int128 liqFees = lastLiquidationFees[txn.liquidatee];
lastLiquidationFees[txn.liquidatee] = 0;
v.insurance = insurance;
v.insurance -= liqFees;
// ...
v.insurance += liqFees;
insurance = v.insurance;
```

This ensures the reserved fee amount always corresponds to the subaccount being finalized, regardless of concurrent liquidations.

### Proof of Concept

```
State setup:
  insurance = 1000e18
  lastLiquidationFees = 0

Step 1 — liquidate subaccount A (large position, fees = 800e18):
  _handleLiquidationPayment(txn_A)
    insurance += 800e18  →  insurance = 1800e18
    lastLiquidationFees = 800e18          ← global overwrite

Step 2 — finalize subaccount B (productId = uint32.max):
  _finalizeSubaccount(txn_B)
    v.insurance = 1800e18
    v.insurance -= 800e18  →  v.insurance = 1000e18   ← uses stale fees_A
    canLiquidateMore = (quoteBalance_B + 1000e18) > 0  ← wrong threshold

    perpEngine.socializeSubaccount(B, 1000e18)
      // covers negative vQuoteBalance with only 1000e18 instead of correct budget
      // excess loss socialized to all open-interest holders  ← permanent state mutation

    v.insurance += 800e18  →  insurance = 1800e18 - consumed - cover + 800e18
    // insurance is inflated; counterparties bear losses they should not

Expected (with correct lastLiquidationFees = fees_B = 50e18):
    v.insurance = 1800e18 - 50e18 = 1750e18
    perpEngine.socializeSubaccount(B, 1750e18)
    // 750e18 more insurance available → less socialization → counterparties protected
```

Assert: `insurance_after_stale ≠ insurance_after_correct` and `cumulativeFundingLongX18_after_stale ≠ cumulativeFundingLongX18_after_correct` — both are locally verifiable in an unmodified Hardhat test.
