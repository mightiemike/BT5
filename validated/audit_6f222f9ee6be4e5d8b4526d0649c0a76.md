### Title
Governance-Mutable `BonusApy` Applied Retroactively to Historical Accrual Segments at Claim Time - (File: `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

The `processEventsAndClaimBonus` function fetches the **current** tier (including its `BonusApy`) from the store at claim time and applies it to **all** historical validator-event segments since the position's last claim. Because `BonusApy` is governance-mutable via `SetTier`, any governance-approved change to a tier's APY is silently applied retroactively to accrual periods that predate the change, making bonus rewards unpredictable and inconsistent with the rate in effect when the position was created.

---

### Finding Description

When a user creates a tiered-rewards position via `LockTier` or `CommitDelegationToTier`, the position stores only the tier's **ID** (`TierId`), not a snapshot of the tier's `BonusApy` at creation time. [1](#0-0) 

When rewards are later claimed (via `ClaimTierRewards`, `TierUndelegate`, `TierRedelegate`, `AddToTierPosition`, `ExitTierWithDelegation`, or `ClearPosition`), the call chain reaches `processEventsAndClaimBonus`, which fetches the **live** tier from the store: [2](#0-1) 

This single `tier` object — carrying the **current** `BonusApy` — is then passed into `computeSegmentBonus` for every historical validator-event segment that has accumulated since the last claim: [3](#0-2) 

And again for the trailing current segment: [4](#0-3) 

`computeSegmentBonus` multiplies `tier.BonusApy` directly into the reward calculation for each segment: [5](#0-4) 

Meanwhile, `SetTier` — explicitly documented as the target of governance messages — can update `BonusApy` at any time while positions are live: [6](#0-5) 

`BonusApy` is a first-class mutable field on the `Tier` struct with no upper bound beyond 100%: [7](#0-6) 

There is no mechanism that snapshots `BonusApy` into the `Position` record at creation time or at each validator event. The position lifecycle (lock → accrue → claim) spans an unbounded time window during which governance can freely alter the rate.

---

### Impact Explanation

**Corrupted value**: The `bonusCoins` amount transferred from `types.RewardsPoolName` to the position owner. [8](#0-7) 

- **APY increase via governance**: All historical segments are recomputed at the higher rate. Users receive more bonus tokens than they accrued, draining the `RewardsPoolName` module account faster than intended and potentially starving other position holders.
- **APY decrease via governance**: All historical segments are recomputed at the lower rate. Users receive fewer bonus tokens than they accrued during the higher-APY period, constituting a loss of earned rewards.

In both directions the `RewardsPoolName` balance is corrupted relative to the protocol's accounting invariant (accrued-at-rate-in-effect).

---

### Likelihood Explanation

Governance parameter changes to tier APY are a normal, expected operation — the `SetTier` path is explicitly designed for it. Any governance proposal that adjusts `BonusApy` (upward or downward) automatically triggers the retroactive mis-accounting for every position in that tier on the next claim. No special attacker capability is required beyond submitting and passing a governance proposal, which is an unprivileged on-chain action available to any token holder with sufficient stake.

---

### Recommendation

Snapshot `BonusApy` into the `Position` record at creation time (and update the snapshot on each `claimRewards` call that advances the accrual checkpoint). `processEventsAndClaimBonus` should use the snapshotted rate for all segments up to the last checkpoint, and only apply the current live rate to the trailing segment from the checkpoint to `blockTime`. This mirrors the fix applied in the referenced report: store the fee/rate in the position record so it cannot be altered during the position's lifetime.

---

### Proof of Concept

1. Governance sets tier 1 `BonusApy = 0.50` (50%).
2. Alice calls `LockTier` for tier 1 at block T=0. Her position stores `TierId = 1`.
3. Alice accrues bonus for 6 months at 50% APY (never claims).
4. Governance passes a proposal calling `SetTier` with `BonusApy = 0.01` (1%) at block T=6mo.
5. Alice calls `ClaimTierRewards` at T=6mo+1block.
6. `processEventsAndClaimBonus` fetches `tier.BonusApy = 0.01` and applies it to **all** segments from T=0 to T=6mo.
7. Alice receives ~2% of the bonus she accrued at 50% APY — a ~98% shortfall relative to the rate in effect during her lock period.

Conversely, if governance raises `BonusApy` from 1% to 50% at T=6mo, Alice retroactively receives 50× the bonus for the prior 6-month period accrued at 1%, draining the rewards pool. [9](#0-8) [10](#0-9)

// computeSegmentBonus computes bonus for a time segment using a snapshot rate.
// Formula: shares * tokensPerShare * tier.BonusApy * durationSeconds / SecondsPerYear
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}

// SetTier writes a tier after validation. Used by governance messages, genesis, and chain upgrades.
func (k Keeper) SetTier(ctx context.Context, tier types.Tier) error {
	if err := tier.Validate(); err != nil {
		return err
	}
	if err := k.Tiers.Set(ctx, tier.Id, tier); err != nil {
		return errorsmod.Wrapf(err, "%s (tier id %d)", types.ErrTierStore.Error(), tier.Id)
	}
	return nil
}
