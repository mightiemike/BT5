### Title
Linked Signer Can Overwrite Itself via `LinkSigner` with `allowLinkedSigner = true`, Enabling Persistent Subaccount Takeover — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. This is the direct EVM analog to the Algorand `RekeyTo` vulnerability: a delegated-but-limited signer can permanently mutate the subaccount's authorization state without the owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` handler is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx(..., true)` resolves to `validateSignature(..., getLinkedSigner(sender))`, which accepts a valid signature from **either** the subaccount owner **or** the current linked signer. [2](#0-1) 

The `linkedSigners` mapping stores a single delegated address per subaccount. Linked signers are intended as trading bots or API keys — not as entities with authority to mutate the subaccount's own authorization state. [3](#0-2) 

Because `allowLinkedSigner = true` is passed to `LinkSigner`, the current linked signer can sign a `LinkSigner` transaction that writes a new, attacker-controlled address into `linkedSigners[victim_subaccount]`. This is a permanent, on-chain authorization mutation — the exact analog of `RekeyTo` in the Algorand report.

The developers are demonstrably aware of the linked-signer theft risk: `Clearinghouse.transferQuote` contains an explicit guard:

```solidity
// require the sender address to be the same as the recipient address
// otherwise linked signers can transfer out
require(
    bytes20(txn.sender) == bytes20(txn.recipient),
    ERR_UNAUTHORIZED
);
``` [4](#0-3) 

That guard blocks direct quote exfiltration by a linked signer. However, no equivalent guard exists for `LinkSigner` itself — the operation that changes who the linked signer is.

---

### Impact Explanation

Once the attacker installs their address as the new linked signer, they can:

1. **Sign `WithdrawCollateral` (V1)** — `allowLinkedSigner = true`; funds are sent to `address(uint160(bytes20(sender)))` (the victim's own address), so no direct theft, but the attacker can drain the subaccount's spot balances, forcing the victim to re-deposit before trading. [5](#0-4) [6](#0-5) 

2. **Sign `MintNlp` / `BurnNlp`** — `allowLinkedSigner = true`; the attacker can mint or burn NLP on behalf of the victim, manipulating the victim's NLP exposure and incurring `HEALTHCHECK_FEE` charges. [7](#0-6) 

3. **Sign `LiquidateSubaccount`** — `allowLinkedSigner = true`; the attacker can initiate liquidations from the victim's subaccount, incurring `LIQUIDATION_FEE` charges against the victim. [8](#0-7) 

4. **Maintain a persistent backdoor** — even after the victim revokes the original linked signer (e.g., rotates an API key), the attacker's address remains in `linkedSigners`. The victim has no on-chain notification mechanism; the backdoor persists until the owner explicitly issues a new `LinkSigner` transaction from their own key.

The corrupted state is: `linkedSigners[victim_subaccount]` — a single storage slot that controls all delegated signing authority for the subaccount.

---

### Likelihood Explanation

Linked signers are the standard mechanism for trading bots, API keys, and programmatic order submission. These keys are:
- Stored in server environments, `.env` files, or CI secrets — all higher-risk than a hardware wallet
- Frequently rotated, meaning a window exists between compromise and revocation

A compromised or malicious linked signer (e.g., a rogue bot operator, a leaked API key, or a supply-chain attack on a trading library) can silently execute this mutation in a single sequencer-submitted transaction. The victim has no on-chain event distinguishing a legitimate `LinkSigner` rotation from a malicious one.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` handler:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode `LinkSigner` path, which already enforces `validateSender(txn.sender, sender)` — i.e., only the on-chain `msg.sender` (the owner) can set the linked signer. [9](#0-8) 

---

### Proof of Concept

1. **Setup**: Victim (`0xAlice`) has subaccount `alice_subaccount` with `linkedSigners[alice_subaccount] = 0xBot` (a trading bot API key).

2. **Compromise**: Attacker obtains `0xBot`'s private key (e.g., leaked `.env`).

3. **Mutation**: Attacker uses `0xBot` to sign and submit a `LinkSigner` transaction:
   ```
   LinkSigner {
     sender: alice_subaccount,
     signer: bytes32(uint256(uint160(0xAttacker))),
     nonce: current_nonce
   }
   ```
   `validateSignedTx(..., true)` accepts the signature because `0xBot == getLinkedSigner(alice_subaccount)`. [10](#0-9) 

4. **Backdoor installed**: `linkedSigners[alice_subaccount] = 0xAttacker`. Alice rotates `0xBot`'s key — but the backdoor persists.

5. **Exploitation**: Attacker signs `WithdrawCollateral` transactions to drain `alice_subaccount`'s spot balances (funds go to Alice's address, but her subaccount is emptied), or signs `MintNlp`/`BurnNlp`/`LiquidateSubaccount` to incur fees and manipulate positions, causing measurable financial loss to Alice's subaccount.
