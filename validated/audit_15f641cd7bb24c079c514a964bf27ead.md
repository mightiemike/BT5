### Title
Undercharged Execution via Integer Overflow in `op_unknown` Cost Multiplier — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown` (`src/more_ops.rs`), the budget check `check_cost(cost, max_cost)?` is performed on the **pre-multiplier** cost, but the final cost returned to the evaluator is the **post-multiplier** value `cost * (cost_multiplier + 1)`. In Rust release mode, this multiplication can silently wrap around to zero (or a small value) when the product overflows `u64`, causing the operation to be charged effectively zero cost. An attacker-controlled CLVM program submitted in consensus mode (without `NO_UNKNOWN_OPS`) can trigger this path, bypassing the block cost limit.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`):

```
260|    check_cost(cost, max_cost)?;
261|    cost *= cost_multiplier + 1;
262|    if cost > u32::MAX as u64 {
263|        Err(EvalErr::Invalid(o))?
264|    } else {
265|        Ok(Reduction(cost as Cost, allocator.nil()))
266|    }
``` [1](#0-0) 

The `cost_multiplier` is decoded from the opcode bytes as a `u32` cast to `u64`, so `cost_multiplier + 1` is at most `2^32 = 4,294,967,296`. [2](#0-1) 

The pre-multiplier `cost` is checked against `max_cost` at line 260. However, the post-multiplication value `cost * (cost_multiplier + 1)` is **not** re-checked against `max_cost`. The only post-multiplication guard is `if cost > u32::MAX as u64`, which is an overflow-magnitude guard, not a budget enforcement check.

**Overflow condition**: In Rust release mode, `u64` arithmetic wraps on overflow. If the pre-multiplier `cost` is a multiple of `2^32` and greater than `u32::MAX`, then:

```
cost * (cost_multiplier + 1)
= (k * 2^32) * 2^32
= k * 2^64
≡ 0  (mod 2^64)
```

The wrapped result `0` passes the `if cost > u32::MAX as u64` guard (since `0 ≤ u32::MAX`), and the function returns `Ok(Reduction(0, allocator.nil()))` — **zero cost** for the operation.

**Minimum triggering pre-multiplier cost**: The smallest multiple of `2^32` that exceeds `u32::MAX` is `2 * 2^32 = 8,589,934,592`. Chia's block cost limit is `11,000,000,000`, so this value is reachable within the budget.

For `cost_function = 3` (concat-like), the pre-multiplier cost formula is:

```
cost = CONCAT_BASE_COST + n * CONCAT_COST_PER_ARG + total_bytes * CONCAT_COST_PER_BYTE
     = 142 + 135n + 3B
``` [3](#0-2) 

Setting this equal to `8,589,934,592` is achievable with a large number of arguments or large atoms within the Allocator's limits.

---

### Impact Explanation

The returned cost from `op_unknown` is added to the running total in `run_program`:

```rust
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
``` [4](#0-3) 

If `op_unknown` returns `0`, the running cost does not increase. The attacker's program has consumed significant argument-iteration work (the pre-multiplier cost was up to `max_cost`) but is charged nothing. By repeating this pattern, an attacker can include arbitrarily many such operations in a block, each consuming real node resources while contributing zero to the cost counter, effectively bypassing the block cost limit. This is a **consensus-critical undercharged execution** bug.

---

### Likelihood Explanation

The trigger requires:
1. Submitting a CLVM program in consensus mode (without `NO_UNKNOWN_OPS` flag, which is the default for block validation).
2. An unknown opcode with `cost_multiplier = u32::MAX` (opcode bytes with all-`0xFF` multiplier prefix and appropriate `cost_function` bits in the last byte).
3. Arguments crafted so the pre-multiplier cost is exactly a multiple of `2^32`. [5](#0-4) 

Condition 3 requires careful crafting but is fully attacker-controlled via the argument list. The opcode bytes and argument sizes are both attacker-supplied inputs to `run_program`. The `NO_UNKNOWN_OPS` flag is absent in consensus mode (only set in `MEMPOOL_MODE`), so the path through `op_unknown` is reachable from on-chain programs. [6](#0-5) 

---

### Recommendation

After line 261, add a `check_cost` call on the post-multiplier value before returning:

```rust
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;
check_cost(cost, max_cost)?;   // <-- add this
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

Alternatively, use `checked_mul` and return `Err(EvalErr::CostExceeded)` on overflow, which also eliminates the wrapping risk entirely.

---

### Proof of Concept

Craft an unknown opcode with:
- Bytes: `[0xFF, 0xFF, 0xFF, 0xFF, 0xC0]` — 5-byte opcode, `cost_multiplier = 0xFFFFFFFF = u32::MAX`, `cost_function = 3` (concat-like, bits `11` in top 2 bits of last byte)
- Arguments: enough atoms totaling `B` bytes such that `142 + 135n + 3B = 2 * 2^32 = 8,589,934,592`

Execution in `op_unknown`:
1. `cost_multiplier = u32::MAX`, `cost_function = 3`
2. Pre-multiplier cost computed as `8,589,934,592` — passes `check_cost(cost, 11_000_000_000)?`
3. `cost *= u32::MAX + 1` → `8,589,934,592 * 4,294,967,296 = 2^33 * 2^32 = 2^65 ≡ 0 (mod 2^64)`
4. `0 > u32::MAX` is false → returns `Ok(Reduction(0, nil))`
5. Running cost in `run_program` unchanged; the operation cost zero. [7](#0-6)

fn unknown_operator(
    allocator: &mut Allocator,
    o: NodePtr,
    args: NodePtr,
    flags: ClvmFlags,
    max_cost: Cost,

    if flags.contains(ClvmFlags::NO_UNKNOWN_OPS) {
        Err(EvalErr::Unimplemented(o))?
