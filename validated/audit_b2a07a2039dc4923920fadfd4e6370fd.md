### Title
`DelegateAction` Signing Domain Omits Network Identifier, Enabling Cross-Network Meta-Transaction Replay — (`core/primitives/src/action/delegate.rs`)

### Summary

`DelegateAction::get_nep461_hash()` produces a signed payload that contains no chain-specific or network-specific identifier. A `SignedDelegateAction` obtained on one NEAR network (e.g., testnet) is cryptographically valid on any other NEAR network (e.g., mainnet) where the same account ID and access key exist with a lower nonce. This is the direct nearcore analog of the PhiFactory chain-ID-ignored replay: a context-binding field is structurally absent from the signed domain, so a signature obtained in one context is reusable in another.

### Finding Description

`DelegateAction::get_nep461_hash()` constructs the signed payload as:

```
SHA-256( borsh( MessageDiscriminant(NEP-366) ) || borsh( DelegateAction ) )
``` [1](#0-0) 

The `MessageDiscriminant` is a fixed compile-time constant derived from the NEP number: [2](#0-1) 

The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`: [3](#0-2) 

None of these fields are network-specific. The discriminant is a fixed integer (`1 << 30 + 366 = 1073742222`), identical on every NEAR network. There is no `chain_id`, `genesis_hash`, or `network_id` anywhere in the signed bytes.

By contrast, regular `SignedTransaction` includes a `block_hash` field in the signed body: [4](#0-3) 

This `block_hash` is chain-specific — a hash of a block that only exists on one network — providing the domain separation that `DelegateAction` lacks.

The runtime verifies the `SignedDelegateAction` signature by calling `signed_delegate_action.verify()`, which calls `get_nep461_hash()` and checks the signature against the sender's public key: [5](#0-4) [6](#0-5) 

If the same `sender_id` account exists on two NEAR networks with the same access key and a lower nonce on the target network, the signature check passes on both networks for the same `SignedDelegateAction`.

The `VersionedSignedDelegateAction` (DelegateV2 / NEP-611) has the same structural gap — its `get_nep461_hash()` uses `SignableMessageType::DelegateActionV2` (another fixed constant) and the `VersionedDelegateActionPayload` body, again with no network identifier: [7](#0-6) 

### Impact Explanation

An attacker who observes a valid `SignedDelegateAction` broadcast on network A (e.g., from a public relayer's mempool or an on-chain receipt) can submit it unchanged to network B. If the victim's account exists on both networks with the same key and a valid nonce on B, the inner actions execute on B under the victim's identity. Inner actions can include `Transfer`, `FunctionCall`, `AddKey`, `DeleteKey`, etc. — any action that can be wrapped in a `DelegateAction`. This results in unauthorized fund transfers or account-state mutations on the target network.

### Likelihood Explanation

The attack requires: (1) the victim has the same account ID on two NEAR networks, (2) the same access key on both, (3) the nonce on the target network is lower than the `DelegateAction`'s nonce, and (4) `max_block_height` has not expired on the target network. Conditions (1) and (2) are common for developers who use the same seed/key on mainnet and testnet. Condition (3) is achievable if the account is newer on the target network. Condition (4) requires the attacker to act within the validity window. Public relayers broadcast `SignedDelegateAction` objects, making observation straightforward.

### Recommendation

Include a network-specific identifier in the `DelegateAction` signed payload. The canonical approach is to add a `chain_id` or `genesis_hash` field to `DelegateAction` (and `DelegateActionV2`) and incorporate it into `get_nep461_hash()`. Alternatively, `get_nep461_hash()` can mix in the genesis block hash at hash-construction time (without changing the struct layout, by prepending it to the `SignableMessage` bytes before hashing). Either approach makes the signed digest network-specific, matching the protection that `block_hash` provides for regular `SignedTransaction`.

### Proof of Concept

1. Alice has account `alice.near` on both mainnet and testnet, with the same ed25519 key, and a lower nonce on mainnet (e.g., mainnet nonce = 0, testnet nonce = 5).
2. Alice signs a `DelegateAction` on testnet: transfer 100 NEAR to Bob, nonce = 6, max_block_height = testnet_height + 1000.
3. The relayer broadcasts the `SignedDelegateAction` on testnet; an attacker observes it.
4. The attacker submits the identical `SignedDelegateAction` to mainnet. The runtime calls `apply_delegate_action` → `signed_delegate_action.verify()` → `get_nep461_hash()`. The hash is identical (same discriminant, same struct bytes, no network field). The signature verifies against Alice's mainnet key. Nonce 6 > mainnet nonce 0. `max_block_height` is still in the future on mainnet. All checks pass.
5. 100 NEAR is transferred from Alice's mainnet account to Bob without Alice's authorization on mainnet. [1](#0-0) [8](#0-7) [9](#0-8)

pub struct SignedDelegateAction {
    pub delegate_action: DelegateAction,
    pub signature: Signature,
}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
