### Title
`burnNlp` Uses Weaker `MAINTENANCE` Health Check Instead of `INITIAL`, Allowing Under-Initial Subaccounts to Extract Quote — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` in `Clearinghouse.sol` enforces only a `MAINTENANCE` health check after converting a user's NLP tokens into quote. Every other value-extracting operation in the same contract (`withdrawCollateral`, `transferQuote`, `mintNlp`) enforces the stricter `INITIAL` health check. A subaccount whose `INITIAL` health is already negative — meaning it is blocked from withdrawing collateral or transferring quote — can still call `burnNlp` and receive quote tokens, bypassing the protocol's intended risk boundary.

---

### Finding Description

`burnNlp` burns the caller's NLP balance and credits them with quote tokens (minus a burn fee). After the state changes it runs a single post-condition:

```solidity
// core/contracts/Clearinghouse.sol  lines 519–529
require(
    spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
    ERR_SUBACCT_HEALTH
);
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [1](#0-0) 

The `MAINTENANCE` weight is always looser than the `INITIAL` weight. `RiskHelper._getWeightX18` returns `longWeightMaintenanceX18` / `shortWeightMaintenanceX18` for `MAINTENANCE` and the tighter `longWeightInitialX18` / `shortWeightInitialX18` for `INITIAL`:

```solidity
// core/contracts/libraries/RiskHelper.sol  lines 44–54
if (amount >= 0) {
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.longWeightInitialX18
        : risk.longWeightMaintenanceX18;
} else {
    weight = healthType == IProductEngine.HealthType.INITIAL
        ? risk.shortWeightInitialX18
        : risk.shortWeightMaintenanceX18;
}
``` [2](#0-1) 

`BaseEngine._addOrUpdateProduct` enforces `longWeightInitial ≤ longWeightMaintenance`, so there is always a gap between the two health values: [3](#0-2) 

Every other value-extracting path uses `INITIAL`:

| Function | Health type used |
|---|---|
| `withdrawCollateral` | `INITIAL` (line 417–419) |
| `transferQuote` | `INITIAL` via `_isAboveInitial` (line 249) |
| `mintNlp` | `INITIAL` (line 480) |
| `burnNlp` | **`MAINTENANCE`** ← inconsistent | [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A subaccount in the "under-initial, above-maintenance" band — i.e., `INITIAL health < 0` and `MAINTENANCE health ≥ 0` — is blocked from withdrawing collateral and transferring quote by every other function. However, `burnNlp` lets it convert NLP into quote freely. The received quote directly improves `INITIAL` health (quote carries weight `ONE` in all health types), potentially lifting the account back above the `INITIAL` threshold and enabling a subsequent `withdrawCollateral` or `transferQuote` that would otherwise be blocked. This is a solvency/accounting corruption: the protocol's risk boundary between "healthy enough to extract value" and "not healthy enough" is bypassed for the NLP redemption path.

---

### Likelihood Explanation

The "under-initial, above-maintenance" band is a normal operating state for any subaccount that has consumed its initial margin buffer but has not yet crossed the liquidation threshold. Any user who monitors their own health can deliberately enter this band (e.g., by opening a large position) and then call `burnNlp` to extract quote that the protocol intends to withhold. The entry path is fully unprivileged: `burnNlp` is dispatched by the sequencer on behalf of the user via `Endpoint`, and the user controls the transaction submission.

---

### Recommendation

Replace the `MAINTENANCE` health type in `burnNlp` with `INITIAL`, consistent with every other value-extracting function:

```solidity
// core/contracts/Clearinghouse.sol
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [6](#0-5) 

---

### Proof of Concept

1. User opens a large perp long position, consuming initial margin so that `INITIAL health = -ε` (slightly negative) while `MAINTENANCE health = +δ` (positive, above liquidation threshold).
2. User submits a `BurnNlp` transaction via the `Endpoint`.
3. `burnNlp` burns the user's NLP and credits `quoteAmount` to `QUOTE_PRODUCT_ID`.
4. The post-condition checks `MAINTENANCE health ≥ 0` — this passes because the user is above the liquidation threshold.
5. The `INITIAL health` check is never performed; the user has received quote tokens that `withdrawCollateral` or `transferQuote` would have refused to release.
6. With the newly credited quote, the user's `INITIAL health` may now be `≥ 0`, allowing a subsequent `withdrawCollateral` to succeed — extracting value that the protocol's risk model was designed to retain as margin. [7](#0-6)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
