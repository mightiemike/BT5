Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router as `sender` Instead of End User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `sender` is the direct caller of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of the pool call, so `sender` equals the router address. If the pool admin whitelists the router so that allowlisted users can use the standard periphery, every unprivileged user can bypass the per-user allowlist by routing through the router. The `DepositAllowlistExtension` avoids this problem by checking `owner` (the actual position owner, passed separately from `sender`), but no equivalent field exists in the swap path.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
  msg.sender,   // <-- direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards `sender` verbatim to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` checks this `sender` against the allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly, making the router `msg.sender` of that call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

So the extension sees `sender` = router address, not the actual end user. The pool admin faces an impossible choice: if the router is not whitelisted, allowlisted users cannot use the standard periphery at all; if the router is whitelisted, any unprivileged user can bypass the per-user allowlist by routing through the router.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the actual position owner, explicitly passed as a separate parameter from `sender`):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol:38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

No equivalent "actual initiator" field exists in the swap interface — `beforeSwap` receives only `sender` (the direct pool caller) and `recipient` (the output destination), neither of which identifies the true end user when an intermediary router is used.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses loses that curation guarantee the moment the router is whitelisted. Any unprivileged address can call `MetricOmmSimpleRouter.exactInputSingle` targeting the curated pool and the extension will approve the swap because it sees the whitelisted router as `sender`, not the non-whitelisted end user. This is a direct bypass of an access control mechanism on a production periphery path. The allowlist extension's core invariant — that only explicitly approved addresses may swap — is fully broken for any pool that must support router-mediated swaps.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported swap entry point. Any pool admin who deploys a `SwapAllowlistExtension`-gated pool and wants allowlisted users to use the router must whitelist the router address — this is the natural operational path. The attacker requires no special role, only the ability to call the public router. The bypass is repeatable and requires no additional preconditions beyond the router being whitelisted.

## Recommendation

`SwapAllowlistExtension` must gate on the actual end user, not the intermediary. Two complementary fixes:

1. **Pass the real initiator through `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when present, falling back to `sender` for direct pool calls.

2. **Mirror the deposit pattern**: Introduce a dedicated `swapInitiator` field in the swap call (analogous to `owner` in `addLiquidity`) so the pool can pass the real user separately from the routing intermediary.

Until fixed, pool admins should not whitelist the router address in `SwapAllowlistExtension` and should instruct allowlisted users to call `pool.swap()` directly.

## Proof of Concept

```
Setup:
  - Pool P configured with SwapAllowlistExtension
  - allowedSwapper[P][alice] = true   (alice is whitelisted)
  - allowedSwapper[P][router] = true  (router whitelisted so alice can use it)

Attack:
  1. Bob (not whitelisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: P, ...})
  2. Router calls P.swap(recipient=bob, ...)  →  msg.sender of pool.swap() = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[P][router] → true  ✓
  5. Swap executes for Bob despite Bob not being in the allowlist

Result: SwapAllowlistExtension curation is fully bypassed for any user who routes
        through MetricOmmSimpleRouter, as long as the router is whitelisted.
```

Foundry test plan: deploy a pool with `SwapAllowlistExtension`, whitelist only `alice` and the router, then call `MetricOmmSimpleRouter.exactInputSingle` from `bob` (not whitelisted) targeting the pool and assert the swap succeeds — confirming the bypass.
