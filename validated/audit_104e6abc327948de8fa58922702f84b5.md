### Title
Attacker-Controlled `tx_id` Accepted as Internal Merkle Node Enables Proof-Verification Forgery — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` accepts a caller-supplied `tx_id` and passes it directly into `compute_root_from_merkle_proof` without validating that it represents a real leaf-level transaction hash. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id`, pair it with a crafted proof, and cause the function to return `true` for a transaction that does not exist. Any consuming contract that relies on this result is deceived into accepting a forged Bitcoin transaction inclusion proof.

---

### Finding Description

The vulnerability class from the external report is **unsanitized attacker-controlled data passed through a trusted verification path without semantic validation**, causing a trusting consumer to accept incorrect output. The analog here is structurally identical: the caller controls `tx_id` in `ProofArgs`, and the contract passes it directly into the Merkle proof verifier with no check that it is a leaf node.

`verify_transaction_inclusion` is a public, unpermissioned method (no `#[private]`, no `#[trusted_relayer]`): [1](#0-0) 

The only guard on `merkle_proof` is an emptiness check: [2](#0-1) 

`compute_root_from_merkle_proof` in `merkle-tools` takes whatever hash it is given and walks the proof path upward. It has no mechanism to distinguish a leaf-level transaction hash from an internal node hash: [3](#0-2) 

The internal `compute_hash` function concatenates two 32-byte values and double-SHA256s them — the same operation used to build every internal node in the Bitcoin Merkle tree: [4](#0-3) 

Because internal nodes and leaf nodes are indistinguishable to the verifier, an attacker can:

1. Identify any confirmed block `B` in the mainchain (its hash is stored in `mainchain_header_to_height`).
2. Obtain `B`'s Merkle tree structure (public Bitcoin data).
3. Select any internal node `N` at depth `d` in `B`'s tree.
4. Construct a `merkle_proof` of length `d − 1` that walks from `N` to the Merkle root `R` stored in `B`'s header.
5. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_block_blockhash = B`, `tx_index` matching `N`'s position, and the crafted proof.
6. `compute_root_from_merkle_proof(N, index, proof)` evaluates to `R`, which equals `header.block_header.merkle_root`, so the function returns `true`.

The contract itself documents this broken invariant in the `# Warning` comment on the function: [5](#0-4) 

The `#[deprecated]` attribute is a compiler hint only; it does not prevent runtime calls. The function remains fully reachable on-chain.

---

### Impact Explanation

Any NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged funds, minting tokens, recording a cross-chain event) will receive `true` for a Bitcoin transaction that was never broadcast or confirmed. The attacker needs no special role, no staked position, and no access to private keys. The corrupted value is the boolean proof result returned to the consuming contract, which it treats as a cryptographic guarantee of Bitcoin transaction inclusion.

---

### Likelihood Explanation

The attack requires only public Bitcoin block data and the ability to call a NEAR contract method. The Merkle tree structure of any confirmed block is freely available. The crafted proof is straightforward to compute offline. The function is callable by any NEAR account when the contract is not paused. Likelihood is **high** for any deployment where a consuming contract uses `verify_transaction_inclusion` without independently re-validating the result.

---

### Recommendation

1. **Remove or gate `verify_transaction_inclusion`**: Since `verify_transaction_inclusion_v2` exists and mitigates this attack via coinbase proof validation, the v1 function should be made `#[private]` or removed entirely to prevent direct external calls.
2. **Enforce proof length bounds**: Require `merkle_proof.len()` to be within the expected range for the block's transaction count, reducing the attacker's freedom to craft proofs for deep internal nodes.
3. **Document the trust boundary explicitly**: Consuming contracts must be warned that `verify_transaction_inclusion` does not validate that `tx_id` is a real transaction; they must use `verify_transaction_inclusion_v2` exclusively.

---

### Proof of Concept

Given a mainchain block `B` with Merkle root `R` and at least 4 transactions `[T0, T1, T2, T3]`:

```
Internal node N = double_sha256(T0 || T1)   # depth-1 internal node
Merkle proof for N at position 0 (even):
  proof[0] = double_sha256(T2 || T3)        # sibling of N at depth 1

compute_root_from_merkle_proof(N, 0, [double_sha256(T2||T3)])
  = double_sha256(N || double_sha256(T2||T3))
  = double_sha256(double_sha256(T0||T1) || double_sha256(T2||T3))
  = R   ✓
```

Call:
```
verify_transaction_inclusion({
  tx_id: N,                          // internal node, not a real transaction
  tx_block_blockhash: B,
  tx_index: 0,
  merkle_proof: [double_sha256(T2||T3)],
  confirmations: 1
})
→ true
```

The function returns `true` for a transaction that does not exist in the Bitcoin blockchain. [3](#0-2) [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
