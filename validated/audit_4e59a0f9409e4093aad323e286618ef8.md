### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Original User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap(...)` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the original user. If the pool admin allowlists the router (the only way to enable router-mediated swaps on an allowlisted pool), every unprivileged user can bypass the per-user swap allowlist by routing through the public, permissionless router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` performs its identity check as follows:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool via `_callExtensionsInOrder`), and `sender` is the first argument forwarded from the pool's `swap` function.

In `MetricOmmPool.swap`, the pool calls `_afterSwap(msg.sender, recipient, ...)` — and symmetrically calls `_beforeSwap(msg.sender, recipient, ...)` — where `msg.sender` is whoever called `pool.swap(...)`. [2](#0-1) 

`_beforeSwap` then ABI-encodes that `sender` value and dispatches it to every configured extension: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInput` / `exactOutput`, the router calls `pool.swap(...)`. At that point `msg.sender` to the pool is the **router address**, so `sender` arriving at the extension is the router, not the original user.

The pool admin now faces an impossible choice:

| Router allowlisted? | Effect |
|---|---|
| **Yes** | `allowedSwapper[pool][router] = true` → every user bypasses the per-user allowlist by routing through the public router |
| **No** | All router-mediated swaps revert for every user, including legitimately allowlisted ones |

There is no configuration that simultaneously enforces per-user allowlisting **and** allows the router to be used.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC-verified counterparties, whitelisted market makers) can have that restriction completely bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInput(...)` targeting the restricted pool. The extension sees `sender = router`, which is allowlisted, and the swap proceeds. The attacker trades on a pool they were explicitly excluded from, potentially draining LP value or executing trades the pool admin intended to prevent.

---

### Likelihood Explanation

The router is the canonical user-facing entry point for swaps. A pool admin who deploys a swap-allowlisted pool and also wants users to be able to use the standard router **must** allowlist the router address. This is a natural, non-malicious configuration step. Once done, the allowlist is silently voided for all router-mediated swaps. The trigger requires no special privilege from the attacker — any EOA can call the public router. [4](#0-3) 

---

### Recommendation

The extension must verify the **original user**, not the immediate caller. Two standard approaches:

1. **Pass the original user through the router**: Have the router encode the original `msg.sender` into `extensionData` and have the extension decode and check it. This requires a trust relationship between the router and the extension.

2. **Check `tx.origin` as a fallback** (not recommended for general use, but acceptable in restricted-pool contexts where the pool admin controls who can interact).

3. **Preferred — check both `sender` and a router-forwarded identity**: Extend the `beforeSwap` hook interface or use `extensionData` to carry the original caller, and have the extension verify that value against the allowlist instead of (or in addition to) `sender`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured in beforeSwap order.
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker (not allowlisted) calls MetricOmmSimpleRouter.exactInput(...)
    targeting the restricted pool.
  - Router calls pool.swap(attacker_recipient, ...).
  - Pool calls _beforeSwap(msg.sender=router, ...).
  - Extension checks allowedSwapper[pool][router] == true  → passes.
  - Swap executes. Attacker bypasses the allowlist.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
``` [5](#0-4) [6](#0-5)
