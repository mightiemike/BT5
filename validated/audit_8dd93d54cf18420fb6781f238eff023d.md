### Title
Fixed Bitcoin MTP Time Window Applied to Zcash Causes Valid Block Rejection - (`btc-types/src/network.rs`, `contract/src/zcash.rs`)

### Summary

A single hardcoded constant `MAX_FUTURE_BLOCK_TIME_MTP = 90 * 60` (90 minutes) is shared across all chain implementations. The Zcash `check_pow` function uses this constant to bound how far ahead of the median-time-past (MTP) a block timestamp may be. However, the code's own comment acknowledges that Zcash's correct value is 129600 seconds (36 hours) — 24× larger. Any valid Zcash block whose timestamp falls between 90 minutes and 36 hours ahead of MTP will be permanently rejected by the contract, causing the Zcash light client to diverge from the real Zcash chain.

### Finding Description

`btc-types/src/network.rs` defines two shared time-window constants:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;      // 5 400 s  (Bitcoin value)
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60; // 7 200 s
``` [1](#0-0) 

`contract/src/zcash.rs` imports and applies `MAX_FUTURE_BLOCK_TIME_MTP` directly in `check_pow`:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment is self-contradictory: it states the correct Zcash value is 129600 s, yet the constant used is 5400 s. The Zcash reference implementation (`nMaxFutureBlockTime`) uses 7200 s for the local-time check and a separate, larger MTP window. The Bitcoin/Litecoin/Dogecoin implementations do **not** import `MAX_FUTURE_BLOCK_TIME_MTP` at all — only the Zcash path does. [3](#0-2) [4](#0-3) 

### Impact Explanation

Any Zcash block header submitted via `submit_blocks` whose `time` field satisfies:

```
prev_block_median_time_past + 5400 < block_header.time
    <= prev_block_median_time_past + 129600
```

is a **valid Zcash block** by consensus rules but will be **permanently rejected** by `check_pow`. Because `submit_block_header` panics on this condition, the relayer cannot advance the Zcash light client past such a block. The contract's `mainchain_tip_blockhash` and `mainchain_height_to_header` mappings stall, making all downstream `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` calls return incorrect results for any Zcash transaction in or after the rejected block. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

Zcash has a 75-second target block time. The MTP of the previous 11 blocks lags the chain tip by roughly 5–8 minutes. A block timestamp 90 minutes ahead of MTP is ~82–85 minutes ahead of wall-clock time — unusual under normal mining but entirely within Zcash consensus rules and observable during network disruptions, miner clock drift, or deliberate timestamp selection. The Zcash protocol explicitly permits this range; the contract does not.

### Recommendation

Replace the single shared constant with a per-chain value. For the Zcash build, use the protocol-correct value:

```rust
// btc-types/src/network.rs  (or a zcash-specific module)
#[cfg(feature = "zcash")]
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 129_600; // 36 hours — Zcash consensus

#[cfg(not(feature = "zcash"))]
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60; // 90 minutes — Bitcoin/Litecoin
```

Alternatively, move the constant into each chain's `NetworkConfig` / `ZcashConfig` struct so it is always chain-specific and cannot be silently reused across builds.

### Proof of Concept

1. Deploy the contract with `feature = "zcash"` and `network = Mainnet`.
2. Construct a valid Zcash block header whose `time` = `prev_MTP + 6000` (100 minutes ahead of MTP — valid per Zcash consensus, invalid per the contract's 90-minute cap).
3. Call `submit_blocks` with this header as an unprivileged NEAR account.
4. The contract panics with `"time-too-far-ahead-of-mtp"` even though the block satisfies Zcash consensus rules.
5. The Zcash mainchain tip is permanently stuck; no subsequent block can be submitted because each references the rejected block as its parent, which is absent from `headers_pool`. [7](#0-6) [8](#0-7)
