### Title
Authz `GenericAuthorization` for `MsgMintNFT` Lets Grantee Freely Choose Recipient, Minting NFTs to Themselves Without Per-Transaction Grantor Consent — (`x/nft/keeper/msg_server.go`, `x/nft/types/msgs.go`)

---

### Summary

The NFT module's `msgServer.MintNFT` derives the denom-creator check from `msg.Sender` (a field in the message body), not from the actual transaction signer. Because `MsgMintNFT.GetSigners()` also returns `[msg.Sender]`, the Cosmos SDK authz module treats `msg.Sender` as the granter. A grantee holding a `GenericAuthorization` for `MsgMintNFT` can therefore craft an inner message with `Sender = grantor` and `Recipient = attacker`, wrap it in `MsgExec`, and cause the keeper to mint NFTs directly to the attacker's address — passing the `IsDenomCreator` guard because the grantor's address is used for that check.

---

### Finding Description

**`GetSigners` returns `msg.Sender`:** [1](#0-0) 

The authz module uses `GetSigners()` to identify the granter. Because `msg.Sender` is a plain string field in the message body (not derived from the actual signing key), a grantee can set it to the grantor's address.

**`msgServer.MintNFT` passes `msg.Sender` directly to the keeper's creator check:** [2](#0-1) 

**`keeper.MintNFT` calls `IsDenomCreator` with the `sender` argument (which is `msg.Sender`):** [3](#0-2) 

**`IsDenomCreator` only checks that the address equals the stored creator — it does not verify the actual transaction signer:** [4](#0-3) 

There is no `MintNFTAuthorization` type anywhere in the codebase that would restrict the `Recipient` field. Only `GenericAuthorization` is available, which imposes no constraints on message content beyond the message type URL.

---

### Impact Explanation

An attacker (bob) who holds a `GenericAuthorization` for `/chainmain.nft.v1.Msg/MintNFT` granted by alice (the denom creator) can:

1. Construct `MsgMintNFT{Sender: alice, DenomId: alice_denom, Id: <new_id>, Recipient: bob}`.
2. Wrap it in `MsgExec` signed by bob.
3. The authz module sees `GetSigners() = [alice]`, finds alice→bob grant for `MsgMintNFT`, and executes the inner message.
4. `msgServer.MintNFT` parses `msg.Sender = alice`, calls `IsDenomCreator(alice)` — passes.
5. NFT is minted to bob.

**Concrete state delta:** bob's NFT balance in `alice_denom` increases; the denom's supply counter increases; alice never signed a mint transaction for this specific token or recipient.

---

### Likelihood Explanation

The precondition is that alice has granted bob a `GenericAuthorization` for `MsgMintNFT`. This is a realistic scenario: a denom creator might delegate minting rights to an operator or partner. Once the grant exists, the exploit is trivially executable with standard CLI tooling (`chain-maind tx authz exec`). No privileged access, governance, or key compromise is required beyond the grant itself.

---

### Recommendation

Implement a `MintNFTAuthorization` type (analogous to Cosmos SDK's `SendAuthorization`) that restricts which `Recipient` addresses are permitted, or that requires `Recipient` to equal the granter. Register it with the authz module so that denom creators can grant scoped minting rights without surrendering control over the recipient. Alternatively, add a check in `msgServer.MintNFT` that verifies the actual transaction signer (from the SDK context) matches `msg.Sender` when not executing under authz, or that the recipient is explicitly approved by the grantor.

---

### Proof of Concept

```bash
# 1. Alice issues a denom
chain-maind tx nft issue alice-denom --from alice ...

# 2. Alice grants bob GenericAuthorization for MsgMintNFT
chain-maind tx authz grant bob generic \
  --msg-type /chainmain.nft.v1.Msg/MintNFT \
  --from alice ...

# 3. Bob generates a MintNFT tx with Sender=alice, Recipient=bob
chain-maind tx nft mint alice-denom token1 \
  --recipient bob_address \
  --from alice \
  --generate-only > mint_tx.json

# 4. Bob executes it via MsgExec
chain-maind tx authz exec mint_tx.json --from bob ...

# 5. Assert: bob now owns token1 in alice-denom; alice never signed this mint
chain-maind query nft owner alice-denom token1
# => bob_address
```

The `IsDenomCreator` check at [5](#0-4)  passes because `msg.Sender` is alice, not because the actual signer is alice. The `Recipient` field is never validated against the granter's intent anywhere in the execution path. [2](#0-1)

```go
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```
