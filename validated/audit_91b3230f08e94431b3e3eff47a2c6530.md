### Title
`slowModeFees` Accumulates But Has No Claim or Withdrawal Function — (`core/contracts/EndpointStorage.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The `slowModeFees` storage variable in `EndpointStorage` accumulates real ERC20 tokens charged to users who submit slow-mode transactions, but no function exists anywhere in the protocol to withdraw or distribute those tokens. The fees are permanently locked in the `Endpoint` contract.

---

### Finding Description

In `EndpointStorage.sol`, `slowModeFees` is declared as a protocol-level accounting variable: [1](#0-0) 

In `EndpointTx.submitSlowModeTransactionImpl`, every slow-mode transaction that is not owner-only and not a `DepositCollateral` charges the caller a `SLOW_MODE_FEE` ($1 in quote token) and increments the variable: [2](#0-1) 

The actual ERC20 transfer is performed by `chargeSlowModeFee`, which pulls tokens from the caller directly into `address(this)` (the `Endpoint` contract): [3](#0-2) 

The `DumpFees` transaction type — the only protocol-level fee-collection path — handles the per-product `sequencerFee` mapping and calls `clearinghouse.claimSequencerFees`, but it never touches `slowModeFees`: [4](#0-3) 

`slowModeFees` is written to in exactly one place and never read, decremented, or forwarded anywhere. There is no `claimSlowModeFees`, no `withdrawSlowModeFees`, and no path through `DumpFees` or any other sequencer transaction that drains it.

---

### Impact Explanation

Every slow-mode transaction submitted by an unprivileged user (e.g., `WithdrawCollateral`, `LinkSigner`) pays `SLOW_MODE_FEE` = $1 in quote token. Those tokens accumulate in the `Endpoint` contract with no recovery path. The protocol permanently loses all slow-mode fee revenue. The `slowModeFees` accounting variable becomes a dead counter that tracks an inaccessible balance.

---

### Likelihood Explanation

Likelihood is high. Slow-mode transactions are the censorship-resistance escape hatch available to any user at any time. Every such submission by a non-owner triggers the fee charge. The condition is met by normal protocol usage with no special setup required.

---

### Recommendation

Add a claim path for `slowModeFees` analogous to the existing `DumpFees` / `claimSequencerFees` flow. The simplest fix is to include `slowModeFees` in the `DumpFees` handler: transfer the accumulated quote tokens to `X_ACCOUNT` (or directly to the sequencer/treasury) and reset `slowModeFees` to zero. Alternatively, add a dedicated `ClaimSlowModeFees` transaction type processed by the sequencer.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(withdrawCollateralTx)`.
2. `EndpointTx.submitSlowModeTransactionImpl` is reached via `delegatecall`.
3. The branch at line 370 executes: `chargeSlowModeFee(_getQuote(), sender)` transfers 1 USDC from the user to `address(this)` (Endpoint).
4. `slowModeFees += SLOW_MODE_FEE` increments the counter.
5. Repeat for every slow-mode user submission.
6. Sequencer submits `DumpFees` — only `sequencerFee[productId]` values are forwarded to `claimSequencerFees`; `slowModeFees` is untouched.
7. The accumulated USDC remains locked in the Endpoint contract indefinitely with no on-chain path to recover it. [5](#0-4) [6](#0-5)
