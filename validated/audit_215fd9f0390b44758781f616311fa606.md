### Title
Hardcoded Decimal Assumption in `getSlowModeFee()` Causes Permanent Revert for Sub-6-Decimal Quote Tokens — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getSlowModeFee()` contains an unchecked subtraction `token.decimals() - 6` on a `uint8` value. Under Solidity 0.8+ checked arithmetic, any quote token with fewer than 6 decimals causes a panic revert, permanently breaking slow-mode transaction submission for that deployment.

---

### Finding Description

`getSlowModeFee()` scales the constant `SLOW_MODE_FEE` (defined as `1_000_000`, i.e. $1 in 6-decimal units) up to the quote token's native decimals:

```solidity
// Clearinghouse.sol line 759-766
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    int256 multiplier = int256(10**(token.decimals() - 6));
    return uint256(int256(SLOW_MODE_FEE) * multiplier);
}
```

`token.decimals()` returns `uint8`. The expression `token.decimals() - 6` is evaluated as `uint8` arithmetic. Under Solidity ≥0.8, if `token.decimals() < 6`, this subtraction underflows and the transaction reverts with a panic. There is no guard analogous to the `require(decimals <= MAX_DECIMALS)` check present in `depositCollateral` and `fastWithdrawalFeeAmount`. [1](#0-0) 

By contrast, `depositCollateral` and `fastWithdrawalFeeAmount` both guard their decimal arithmetic: [2](#0-1) [3](#0-2) 

`getSlowModeFee()` is referenced in `EndpointStorage.sol` (confirmed by grep), meaning it is called during slow-mode transaction processing — a state-changing path. A permanent revert there blocks all slow-mode submissions for the affected deployment. [4](#0-3) 

The constant `SLOW_MODE_FEE = 1_000_000` is defined assuming 6-decimal quote token precision: [5](#0-4) 

---

### Impact Explanation

Any deployment that configures a quote token with fewer than 6 decimals (e.g., a 2-decimal stablecoin) will have `getSlowModeFee()` permanently revert. Because `EndpointStorage.sol` calls this function during slow-mode transaction submission, **all slow-mode transactions are permanently blocked** for that deployment. Slow mode is the censorship-resistance fallback path — its permanent failure is a functional protocol limitation, not merely a view-layer inconvenience. No funds are directly stolen, but the deployment is rendered partially non-functional without any upgrade.

---

### Likelihood Explanation

The Nado protocol is designed to support multiple token configurations on Ink Chain. While USDC (6 decimals) is the current quote token, the code contains no enforcement that the quote token must have ≥ 6 decimals at registration time. A deployer or future upgrade that registers a sub-6-decimal quote token would silently introduce this revert. The bug is latent and configuration-triggered, matching the Frax analog exactly.

---

### Recommendation

Replace the bare subtraction with a guarded scaling that handles both directions:

```solidity
function getSlowModeFee() external view returns (uint256) {
    ISpotEngine spotEngine = _spotEngine();
    IERC20Base token = IERC20Base(
        spotEngine.getConfig(QUOTE_PRODUCT_ID).token
    );
    uint8 d = token.decimals();
    if (d >= 6) {
        return uint256(int256(SLOW_MODE_FEE) * int256(10**(d - 6)));
    } else {
        return uint256(int256(SLOW_MODE_FEE) / int256(10**(6 - d)));
    }
}
```

Alternatively, add a `require(token.decimals() >= 6)` guard at product registration time, consistent with the existing `require(decimals <= MAX_DECIMALS)` pattern.

---

### Proof of Concept

1. Deploy Nado with a quote token whose `decimals()` returns `2` (e.g., a hypothetical 2-decimal stablecoin).
2. Call `Clearinghouse.getSlowModeFee()`.
3. Solidity 0.8 evaluates `uint8(2) - uint8(6)` → underflow → panic revert.
4. Any call path in `EndpointStorage.sol` that invokes `getSlowModeFee()` to charge the slow-mode fee now permanently reverts.
5. All slow-mode transaction submissions fail for this deployment, disabling the censorship-resistance path. [6](#0-5)
