### Title
Missing `_tryCloseIsolatedSubaccount` Cleanup Before Health Check Enables Incorrect Liquidation of Healthy Subaccounts — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`liquidateSubaccountImpl` checks `isUnderMaintenance(txn.liquidatee)` without first closing the liquidatee's isolated subaccounts that have zero perp positions (expired or unfilled orders). Margin locked in those stale isolated subaccounts is excluded from the parent's health calculation, making the parent appear less healthy than it actually is. A liquidator can liquidate a user whose parent account would be above maintenance health if the stale isolated subaccounts were properly closed first.

---

### Finding Description

When a user creates an isolated subaccount via `createIsolatedSubaccount`, margin is immediately transferred from the parent's quote balance to the isolated subaccount's quote balance: [1](#0-0) 

If the isolated order expires without being filled, the isolated subaccount's perp position remains zero (`balance.amount == 0`), but the margin stays locked in the isolated subaccount. The function `_tryCloseIsolatedSubaccount` is designed to detect this condition and return the margin to the parent: [2](#0-1) 

However, `_tryCloseIsolatedSubaccount` is only invoked in two places:
1. In `delistProduct` for isolated subaccounts of a delisted perp.
2. In `liquidateSubaccountImpl` — but **only when the liquidatee itself is an isolated subaccount** (line 621–625), not when the liquidatee is a parent subaccount that owns stale isolated subaccounts. [3](#0-2) 

The health check `isUnderMaintenance(txn.liquidatee)` calls `getHealth(subaccount, MAINTENANCE)`: [4](#0-3) 

`getHealth` aggregates only the parent subaccount's own engine balances. Isolated subaccounts are separate `bytes32` identifiers; their balances are never included in the parent's health calculation. Margin locked in isolated subaccounts with zero perp positions is therefore invisible to the parent's health check.

The same omission exists in `withdrawCollateral`, which also performs a health check without closing stale isolated subaccounts first: [5](#0-4) 

---

### Impact Explanation

A parent subaccount that owns one or more isolated subaccounts with zero perp positions (expired/unfilled orders) will have its health understated by the amount of margin locked in those subaccounts. A liquidator can submit a `LiquidateSubaccount` transaction against this parent, and `isUnderMaintenance` will return `true` even though the parent would be above maintenance health if the stale isolated subaccounts were closed first. The user suffers an incorrect liquidation: their positions are seized at a discount and liquidation fees are charged, constituting a direct asset loss.

---

### Likelihood Explanation

This is realistically reachable. Any user who:
1. Creates isolated perp orders that expire without being filled (a normal trading scenario), and
2. Has other positions whose health drops near the maintenance threshold

is vulnerable. The liquidator needs no special privileges — `liquidateSubaccountImpl` is callable by any non-isolated subaccount via the sequencer-submitted `LiquidateSubaccount` transaction type. The stale isolated subaccounts persist indefinitely until explicitly closed, so the window of exposure grows over time.

---

### Recommendation

Before the `isUnderMaintenance(txn.liquidatee)` check in `liquidateSubaccountImpl`, iterate over the liquidatee's isolated subaccounts (via `isolatedSubaccounts[liquidatee][id]` for all `id` up to `MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS`) and call `_tryCloseIsolatedSubaccount` for each one. This mirrors the cleanup already performed when the liquidatee is itself an isolated subaccount (line 621–625), but extends it to the parent's owned isolated subaccounts.

Apply the same cleanup in `withdrawCollateral` before the `getHealth` check, consistent with the pattern used in `delistProduct`.

---

### Proof of Concept

1. User's parent subaccount holds 1000 USDC and a perp position worth 400 USDC in health contribution.
2. User creates two isolated subaccounts for perp orders, locking 300 USDC each (600 USDC total) as margin. Parent's quote balance drops to 400 USDC.
3. Both isolated orders expire without being filled. Perp positions in both isolated subaccounts are zero.
4. Market moves against the user's parent perp position; parent's maintenance health (computed over 400 USDC quote + perp position) drops below zero.
5. Liquidator calls `LiquidateSubaccount` targeting the parent.
6. `isUnderMaintenance(parent)` returns `true` — the 600 USDC locked in the two stale isolated subaccounts is not counted.
7. If `_tryCloseIsolatedSubaccount` were called for both isolated subaccounts first, 600 USDC would be returned to the parent (total 1000 USDC), and `isUnderMaintenance` would return `false`.
8. Liquidation proceeds incorrectly: the user's perp position is liquidated at a discount, and liquidation fees are deducted from the parent's balance. [6](#0-5) [2](#0-1) [7](#0-6)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }

        public
        returns (int128 health)
    {
        ISpotEngine spotEngine = _spotEngine();
        IPerpEngine perpEngine = _perpEngine();

        health = spotEngine.getHealthContribution(subaccount, healthType);
        // min health means that it is attempting to borrow a spot that exists outside
        // of the risk system -- return min health to error out this action
        if (health == -INF) {
            return health;
        }
        health += perpEngine.getHealthContribution(subaccount, healthType);
