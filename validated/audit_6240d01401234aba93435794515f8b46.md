### Title
Checkpoint Advancement Without Bonus Payment in `slashRedelegationPosition` When Pool Is Empty — (`x/tieredrewards/keeper/slash.go`, `claim_rewards.go`)

### Summary
When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` module account balance is zero (or insufficient), `processEventsAndClaimBonus` deletes all pending validator events and advances the position's `LastEventSeq`/`LastBonusAccrual` checkpoints in-memory before the pool balance check. The `ErrInsufficientBonusPool` error is then swallowed in `slashRedelegationPosition`, and `setPositionWithState` persists the advanced checkpoints. The position permanently loses all accrued bonus for those event segments with no compensation.

### Finding Description

In `processEventsAndClaimBonus` [1](#0-0) , the loop over pending events calls `decrementEventRefCount` (which deletes events with `ReferenceCount=1` from the store) and `pos.UpdateLastEventSeq` for each event **before** the pool balance check at line 230. [2](#0-1) 

`applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` also mutate the in-memory `pos` before the check. [3](#0-2) 

In `slashRedelegationPosition`, `processEventsAndClaimBonus` is called with `&pos` (pointer), so all in-memory mutations are visible to the caller. [4](#0-3) 

The `ErrInsufficientBonusPool` error is explicitly swallowed: [5](#0-4) 

Then `setPositionWithState` is called unconditionally with the mutated `pos`, persisting the advanced checkpoints: [6](#0-5) 

Meanwhile, `decrementEventRefCount` has already deleted the events from the store: [7](#0-6) 

### Impact Explanation
A position permanently loses all accrued bonus rewards for the N event segments that were processed. The events are deleted, the checkpoints are advanced, and no coins are transferred. This is a direct, unrecoverable economic loss to the position owner. The magnitude scales with the number of pending events and the position's delegation size/duration.

### Likelihood Explanation
The `RewardsPoolName` balance can reach zero through normal operation (pool drained by prior claims, not yet replenished). Any redelegation slash event — a normal chain operation triggered by double-signing or downtime — while the pool is empty will trigger this bug. No privileged access or attacker control is required; the pool being empty is a realistic transient state.

### Recommendation
Move the `sufficientBonusPoolBalance` check **before** the event-processing loop, or — if the pool is insufficient — skip the entire `processEventsAndClaimBonus` call without mutating any state. Alternatively, separate the event-deletion/checkpoint-advancement from the payment step so that checkpoints only advance after a successful transfer.

### Proof of Concept
1. Create a position with a delegation to validator V.
2. Trigger N validator events (e.g., UNBOND/BOND cycles) so the position has N pending events with `ReferenceCount=1`.
3. Drain `RewardsPoolName` to zero balance.
4. Trigger a redelegation slash on the position's unbonding ID, firing `BeforeRedelegationSlashed`.
5. Assert: `position.LastEventSeq` has advanced by N, all N events are deleted from the store, but the owner received zero bonus coins.
