### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Fast-Path — (File: `core/contracts/EndpointTx.sol`)

### Summary
The fast-path `LinkSigner` transaction handler in `EndpointTx.sol` passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **current linked signer** — not just the subaccount owner — can sign a `LinkSigner` transaction and overwrite `linkedSigners[subaccount]` with any address. This is a direct analog to the Identity.sol bug: a delegated entity (linked signer ≈ module) can mutate the privileged signer slot (linked signer ≈ owner) without the actual owner's authorization.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateSignature`, which calls `verifier.validateSignature(sender, getLinkedSigner(sender), digest, signature)`. The verifier accepts the signature if the recovered address equals **either** the subaccount owner address **or** the current linked signer address. [2](#0-1) [3](#0-2) [4](#0-3) 

Therefore, the current linked signer can sign a valid `LinkSigner` transaction that sets `linkedSigners[subaccount]` to any attacker-controlled address — without the subaccount owner's key ever being involved.

By contrast, the **slow-mode** `LinkSigner` path correctly restricts this action to the subaccount owner via `validateSender`, which requires `address(uint160(bytes20(txSender))) == msg.sender`: [5](#0-4) [6](#0-5) 

The `linkedSigners` mapping is the sole privileged-signer slot for a subaccount in the fast-path: [7](#0-6) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to an attacker-controlled address.
2. The sequencer processes it; `linkedSigners[victim_subaccount]` is now the attacker's address.
3. The attacker's new linked signer can immediately sign `WithdrawCollateral`, `TransferQuote`, or `LiquidateSubaccount` transactions on behalf of the victim subaccount, draining all collateral.

The victim subaccount owner can attempt to reclaim control by submitting their own `LinkSigner` transaction, but the attacker can race to re-set it again (both share the same nonce counter keyed on the owner address). The attacker who already controls the linked signer slot has a persistent, renewable foothold.

**Corrupted state:** `linkedSigners[subaccount]` — the address authorized to sign all fast-path transactions for the subaccount.

---

### Likelihood Explanation

Any user who has ever granted linked signer access to a third-party service (trading bot, API key, automated strategy) is exposed if that service is compromised or acts maliciously. Linked signer delegation is a core, advertised feature of the protocol. The attack requires only possession of the linked signer's private key — no admin access, no governance, no oracle manipulation.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` handler, so that only the subaccount owner's key can authorize a linked signer change — consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only subaccount owner may change linked signer
);
``` [8](#0-7) 

---

### Proof of Concept

**Preconditions:**
- Victim subaccount `V` has linked signer `LS` (e.g., a trading bot key).
- Attacker obtains `LS`'s private key (service breach, key leak, insider).

**Steps:**

1. Attacker constructs `SignedLinkSigner { tx: { sender: V, signer: ATTACKER_ADDR, nonce: current_nonce }, signature: sign_with_LS(...) }`.
2. Attacker submits this as a fast-path transaction to the sequencer.
3. Sequencer calls `processTransaction` → `EndpointTx.LinkSigner` branch.
4. `validateSignedTx(..., true)` passes because `ECDSA.recover(digest, sig) == LS == getLinkedSigner(V)`.
5. `linkedSigners[V] = ATTACKER_ADDR` is written.
6. Attacker immediately signs `WithdrawCollateral { sender: V, productId: QUOTE, amount: full_balance, nonce: next_nonce }` with `ATTACKER_ADDR`.
7. Sequencer processes withdrawal; victim's collateral is transferred to attacker's wallet. [1](#0-0) [9](#0-8)
