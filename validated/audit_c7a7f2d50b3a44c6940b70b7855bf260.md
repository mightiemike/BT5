### Title
`max_receipt_size` Cap Not Re-Enforced After `output_data_receivers` Are Appended to Already-Validated Receipts — (File: `runtime/runtime/src/lib.rs`)

---

### Summary

The `max_receipt_size` protocol limit is checked at receipt-creation time via `validate_receipt(..., ValidateReceiptMode::NewReceipt)`, but the runtime subsequently appends `output_data_receivers` entries to those already-validated receipts without re-checking the size. Any unprivileged user who deploys a contract can craft a receipt that passes the size gate at creation, then trigger the `promise_return` or `promise_then` code path to push the receipt's Borsh-encoded length above `max_receipt_size`. The resulting oversized receipt is stored in state and forwarded across shards. The codebase explicitly acknowledges this as a live bug (issue #12606) and introduces a second, more permissive validation mode (`ExistingReceipt`) specifically to tolerate these receipts at runtime.

---

### Finding Description

`max_receipt_size` (currently 4,194,304 bytes) is declared in `LimitConfig` and enforced in `validate_receipt` only when `mode == ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After that check passes, two distinct runtime code paths append additional `output_data_receivers` entries to the receipt without re-running the size gate:

**Path 1 — `promise_return` (`lib.rs`):** When a function call returns `ReturnData::ReceiptIndex(i)`, the runtime extends the `output_data_receivers` of the newly-created receipt `i` with the receivers from the *current* action receipt: [2](#0-1) 

**Path 2 — `promise_then` (`receipt_manager.rs`):** When `create_action_receipt` is called to chain a new receipt onto existing ones, it pushes a `DataReceiver` into each predecessor receipt's `output_data_receivers` vector: [3](#0-2) 

In both paths, the receipt was already validated at creation time. No re-validation occurs after the mutation. The `ValidateReceiptMode::ExistingReceipt` documentation explicitly records this as a known bug: [4](#0-3) 

The test suite confirms both paths produce oversized receipts that reach state: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

- **Protocol invariant broken:** `max_receipt_size` is a consensus-level limit. Receipts exceeding it are stored in the delayed-receipt queue and forwarded across shards, violating the invariant that all in-flight receipts are bounded by `max_receipt_size`.
- **State witness inflation:** Oversized receipts inflate `ChunkStateWitness` beyond the `main_storage_proof_size_soft_limit`, undermining stateless validation assumptions.
- **Cross-shard bandwidth:** Oversized receipts bypass the outgoing-receipt size budgets enforced by congestion control, which assume receipts respect `max_receipt_size`.
- **Forced permissive mode:** The runtime must permanently tolerate oversized receipts via `ExistingReceipt` mode, meaning the limit cannot be strictly enforced even after a fix without a migration.

**Scope: High** — broken protocol-level size invariant reachable by any contract deployer, with cross-shard propagation.

---

### Likelihood Explanation

Any account that can deploy a WASM contract can trigger this. The attack requires:
1. Deploy a contract that calls `generate_large_receipt` (or equivalent) to create a receipt at exactly `max_receipt_size`.
2. Use `promise_return` or `promise_then` to cause the runtime to append `output_data_receivers` to that receipt.

No privileged role is required. The test at `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates the exact steps. The bug is reachable on mainnet today.

---

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended in both mutation sites:

1. In `runtime/runtime/src/lib.rs` (the `promise_return` path, lines 1028–1035): after `extend_from_slice`, compute `borsh::object_length` of the mutated receipt and return a `NewReceiptValidationError::ReceiptSizeExceeded` if the limit is exceeded.

2. In `runtime/runtime/src/receipt_manager.rs::create_action_receipt` (line 122–123): after pushing the `DataReceiver`, check the updated size of the predecessor receipt and return `HostError::GuestPanic` or a new `VMLogicError` variant if the limit would be exceeded.

Both checks must use the same `max_receipt_size` from `LimitConfig` that `validate_receipt` uses, so the enforcement is consistent across all code paths.

---

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept: [7](#0-6) 

Steps:
1. Deploy `near_test_contracts::rs_contract()`.
2. Call `max_receipt_size_promise_return_method1` with `args_size` set so the created receipt is exactly `max_receipt_size` bytes.
3. The contract uses `promise_return`, causing the runtime to append `output_data_receivers` to the receipt at `lib.rs:1029–1031`.
4. The receipt's Borsh length now exceeds `max_receipt_size`.
5. `assert_oversized_receipt_occurred` confirms the oversized receipt appears in the chain's incoming-receipt proofs — it was stored in state and forwarded across shards without rejection.

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
