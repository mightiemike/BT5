Looking at the `check_and_update_nonce` function and `GlobalContractDistributionReceipt::V1` nonce behavior, I've found a concrete analog.### Title
`GlobalContractDistributionReceipt::V1` nonce=0 permanently bypasses `check_and_update_nonce` idempotency guard, allowing stale V1 distribution receipts to overwrite newer AccountId-mode global contract deployments — (`File: runtime/runtime/src/global_contracts.rs`)

---

### Summary

`check_and_update_nonce` uses the predicate `incoming_nonce < stored_nonce` to detect stale `GlobalContractDistribution` receipts. `GlobalContractDistributionReceipt::V1` always returns `nonce() == 0`. After a V1 receipt is applied, `set_nonce` writes `0` back to the trie, leaving `stored_nonce == 0`. A subsequent V1 receipt for the same identifier evaluates `0 < 0 == false` and is treated as fresh — identical to the "already filled order" pattern in the external report. Two V1 receipts for the same `AccountId`-mode global contract that arrive out of order on any shard will both be applied, with the later-arriving (older-code) receipt silently overwriting the newer deployment.

---

### Finding Description

`GlobalContractDistributionReceipt` has two variants:

```
pub enum GlobalContractDistributionReceipt {
    V1(GlobalContractDistributionReceiptV1) = 0,   // no nonce field
    V2(GlobalContractDistributionReceiptV2) = 1,   // carries nonce: u64
}

pub fn nonce(&self) -> u64 {
    match &self {
        Self::V1(_) => 0,          // hard-coded sentinel
        Self::V2(v2) => v2.nonce,
    }
}
```

The freshness gate in `apply_distribution_current_shard` is:

```rust
fn check_and_update_nonce(...) -> Result<bool, RuntimeError> {
    let stored_nonce = get_nonce(state_update, &nonce_key)?;  // 0 when absent
    let incoming_nonce = global_contract_data.nonce();        // always 0 for V1

    if incoming_nonce < stored_nonce {   // 0 < 0 == false → never stale for V1
        return Ok(false);
    }
    set_nonce(state_update, nonce_key, incoming_nonce);       // writes 0 back
    Ok(true)
}
```

State after processing a V1 receipt: `stored_nonce = 0`. The next V1 receipt for the same identifier evaluates `0 < 0 == false` and is unconditionally applied. The guard is permanently open for V1 receipts.

`forward_distribution_next_shard` is called unconditionally after `apply_distribution_current_shard` regardless of whether the current shard's application was stale:

```rust
pub(crate) fn apply_global_contract_distribution_receipt(...) {
    let compute =
        apply_distribution_current_shard(receipt, global_contract_data, apply_state, state_update)?;
    forward_distribution_next_shard(                          // always called
        receipt, global_contract_data, ...
    )?;
    Ok(compute)
}
```

`forward()` for V1 preserves the V1 variant, so every forwarded hop also carries `nonce == 0` and passes the same broken check on every downstream shard.

**Concrete attack path (upgrade-boundary interaction):**

1. Before the V2 nonce fix ships, a user deploys an `AccountId`-mode global contract twice in quick succession. Two `V1` distribution receipts `R1` (old code) and `R2` (new code) enter the delayed-receipt queue on some shard `S`.
2. Network load causes `R1` to be deferred while `R2` is processed immediately on `S`. `stored_nonce[S] = 0`.
3. `R2` is forwarded as a V1 receipt to shard `S+1`, `S+2`, … and applied on each. `stored_nonce` stays `0` everywhere.
4. `R1` is later dequeued on `S`. `incoming_nonce = 0`, `stored_nonce = 0` → `0 < 0 == false` → applied. Old code overwrites new code on `S`.
5. `R1` is forwarded and applied on every subsequent shard, overwriting the new code everywhere.

Result: all shards silently revert to the old contract code. Every account that called `UseGlobalContract` with the `AccountId` identifier now executes the old WASM.

---

### Impact Explanation

- **Scope:** `AccountId`-mode global contracts. `CodeHash`-mode is unaffected because the identifier encodes the hash of the code; two different deployments produce two different identifiers.
- **Effect:** Silent, permanent state corruption. The `GlobalContractCode` trie key for the identifier is overwritten with old bytecode. No error is surfaced to the deployer or to accounts using the contract.
- **Blast radius:** Every account on every shard that has called `UseGlobalContract` with the affected `AccountId` identifier now runs the wrong WASM. This includes all downstream function calls, cross-contract calls, and any logic that depends on the contract's behavior.
- **Severity: High.** The global contract invariant ("all shards converge to the same code for a given identifier") is broken without any on-chain signal.

---

### Likelihood Explanation

- V1 receipts are only created before the nonce fix ships. The fix is present in the current codebase (CHANGELOG: "Add nonce-based idempotency for global contract distribution receipts"), meaning V1 receipts can still exist in the delayed-receipt queue on nodes upgraded from a pre-fix binary.
- The trigger requires two `AccountId`-mode deployments of the same contract before the fix, with the first deployment's receipt delayed past the second's. Delayed receipts are common under normal network load.
- The `new_v1` constructor remains callable in the codebase; any code path that constructs a V1 receipt (e.g., deserialization of a pre-fix state snapshot, or a test harness) re-opens the window.
- No privileged role is required. Any unprivileged account can deploy a global contract.

---

### Recommendation

1. **Reject V1 receipts after the nonce fix epoch.** Add a protocol-version gate in `apply_distribution_current_shard`: if `protocol_version >= GlobalContractDistributionV2` and the receipt is `V1`, treat it as stale and return `Ok(0)`.

2. **Alternatively, tighten the nonce sentinel.** After processing a V1 receipt, write `stored_nonce = 1` (not `0`) so that a subsequent V1 receipt (`incoming = 0`) evaluates `0 < 1 == true` and is correctly dropped:

```rust
// After applying a V1 receipt, advance stored_nonce past 0
// so future V1 receipts (nonce=0) are treated as stale.
let effective_stored = stored_nonce.max(1);
set_nonce(state_update, nonce_key, effective_stored);
```

3. **Gate forwarding on freshness.** `forward_distribution_next_shard` should only forward if `apply_distribution_current_shard` returned a non-zero compute value (i.e., the receipt was fresh), preventing stale V1 receipts from propagating to downstream shards.

---

### Proof of Concept

```
// Borsh-encoded V1 GlobalContractDistributionReceipt (discriminant = 0x00)
// nonce() returns 0 unconditionally.

// State before: TrieKey::GlobalContractNonce{id} absent → get_nonce() = 0

// Receipt R2 (new code, V1) processed first:
//   incoming_nonce = 0, stored_nonce = 0
//   0 < 0 == false → fresh → new code written → set_nonce(0)

// Receipt R1 (old code, V1) processed second:
//   incoming_nonce = 0, stored_nonce = 0
//   0 < 0 == false → fresh → old code overwrites new code → set_nonce(0)

// Invariant violated: TrieKey::GlobalContractCode{id} now contains old WASM
// on every shard where R1 arrived after R2.
```

The exact divergent Borsh bytes are the `V1` discriminant (`0x00`) in `GlobalContractDistributionReceipt`, which causes `nonce()` to return the hard-coded value `0` — identical to the initial `stored_nonce` — making the predicate `incoming_nonce < stored_nonce` permanently false and the "already applied" guard permanently open. fn check_and_update_nonce(
    global_contract_data: &GlobalContractDistributionReceipt,
    identifier: &GlobalContractCodeIdentifier,
    state_update: &mut TrieUpdate,
) -> Result<bool, RuntimeError> {
    let nonce_key = TrieKey::GlobalContractNonce { identifier: identifier.clone() };
    let stored_nonce = get_nonce(state_update, &nonce_key)?;
    let incoming_nonce = global_contract_data.nonce();

    // Allow the same nonce since the nonce is updated immediately when
    // initiating distribution to prevent multiple distributions with the same
    // nonce from being initiated.
    if incoming_nonce < stored_nonce {
        return Ok(false);
    }

    set_nonce(state_update, nonce_key, incoming_nonce);
    Ok(true)
}

fn forward_distribution_next_shard(
    receipt: &Receipt,
    global_contract_data: &GlobalContractDistributionReceipt,
    apply_state: &ApplyState,
    epoch_info_provider: &dyn EpochInfoProvider,
    state_update: &mut TrieUpdate,
    receipt_sink: &mut ReceiptSink,
    receipt_to_tx: &mut Vec<(CryptoHash, ReceiptToTxInfo)>,
) -> Result<(), RuntimeError> {
    let shard_layout = epoch_info_provider.shard_layout(&apply_state.epoch_id)?;
    let already_delivered_shards = BTreeSet::from_iter(
        global_contract_data
            .already_delivered_shards()
            .iter()
            .cloned()
            .chain(std::iter::once(apply_state.shard_id)),
    );
    let Some(next_shard) = shard_layout
        .shard_ids()
        .filter(|shard_id| !already_delivered_shards.contains(&shard_id))
        .next()
    else {
        return Ok(());
    };
    let already_delivered_shards = Vec::from_iter(already_delivered_shards);
    let predecessor_id = receipt.predecessor_id().clone();
    let next_receipt = global_contract_data.forward(next_shard, already_delivered_shards);
    let mut next_receipt = Receipt::new_global_contract_distribution(predecessor_id, next_receipt);
    let receipt_id = apply_state.create_receipt_id(receipt.receipt_id(), 0);
    next_receipt.set_receipt_id(receipt_id);
    if apply_state.save_receipt_to_tx {
        receipt_to_tx.push((
            receipt_id,
            ReceiptToTxInfo::V1(ReceiptToTxInfoV1 {
                origin: ReceiptOrigin::FromReceipt(ReceiptOriginReceipt {
                    parent_receipt_id: *receipt.receipt_id(),
                    parent_predecessor_id: receipt.predecessor_id().clone(),
                }),
                receiver_account_id: next_receipt.receiver_id().clone(),
                shard_id: apply_state.shard_id,
            }),
        ));
    }
    receipt_sink.forward_or_buffer_receipt(next_receipt, apply_state, state_update)?;
    Ok(())
