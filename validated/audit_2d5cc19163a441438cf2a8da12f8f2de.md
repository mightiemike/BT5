### Title
Deprecated `verify_transaction_inclusion` Accepts Unvalidated Raw `tx_id` From Any Caller, Enabling Merkle Proof Forgery - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` remains callable by any unprivileged NEAR account. It accepts a raw, caller-supplied `tx_id` with no on-chain validation that the value is a leaf-level transaction hash rather than an internal Merkle tree node. An attacker can supply a crafted internal-node hash as `tx_id` together with a matching proof path and receive a `true` return value for a transaction that was never included in the block. Downstream contracts consuming this result are deceived.

### Finding Description
`verify_transaction_inclusion` is marked `#[deprecated]` but is not removed and carries no access restriction beyond the `#[pause]` gate (which is an operational toggle, not a caller restriction). [1](#0-0) 

The function's own inline warning acknowledges the root cause: [2](#0-1) 

The verification logic in `compute_root_from_merkle_proof` performs no leaf-vs-internal-node discrimination. It accepts any 32-byte value as the starting hash and walks the proof path: [3](#0-2) 

Because Bitcoin's Merkle tree uses the same `double_sha256` function at every level, an internal node hash at depth *d* is a valid starting point for a proof of depth *d*. An attacker who knows the internal structure of a block's Merkle tree can pick any internal node, construct a valid proof path to the root, and call `verify_transaction_inclusion` with that node as `tx_id` and the matching `tx_index`. The function computes the root, finds it equal to the stored `merkle_root`, and returns `true`.

The `ProofArgs` struct places no constraint on `tx_id`: [4](#0-3) 

`verify_transaction_inclusion_v2` mitigates this by requiring a coinbase proof of equal length (bounding tree depth), but it does so only for calls that go through the v2 entry point. The v1 entry point remains open: [5](#0-4) 

### Impact Explanation
Any recipient contract that calls `verify_transaction_inclusion` to gate an action (e.g., releasing bridged assets, confirming a cross-chain payment) can be made to observe a `true` result for a Bitcoin transaction that was never broadcast or confirmed. The corrupted proof result is the boolean return value of `verify_transaction_inclusion`. The canonical-chain state itself is not altered, but the verification output — the only signal downstream contracts have — is forged.

### Likelihood Explanation
The attack requires no privileged role. Any NEAR account can call `verify_transaction_inclusion` directly. The 64-byte Merkle forgery technique is publicly documented (BitMEX research, CVE-2017-12842) and tooling to compute valid internal-node proofs from public block data is readily available. The only prerequisite is knowledge of a real confirmed block's Merkle tree, which is public.

### Recommendation
Remove `verify_transaction_inclusion` entirely from the public ABI. Because it is already deprecated and `verify_transaction_inclusion_v2` exists as the safe replacement, the v1 method should be deleted or replaced with a hard panic (`env::panic_str("use verify_transaction_inclusion_v2")`). Keeping a deprecated but callable method that the codebase itself documents as forgeable is equivalent to the external report's pattern of accepting raw, unvalidated data from an untrusted caller without server-side signing.

### Proof of Concept

1. Identify any confirmed Bitcoin block whose Merkle tree has at least two transactions. Obtain the full transaction list from a public block explorer.
2. Compute the Merkle tree. Pick any internal node `N` at depth *d* (e.g., the hash of `tx[0]` and `tx[1]` combined).
3. Construct a proof path of length `tree_depth - d` from `N` to the root. This is a valid `merkle_proof` array.
4. Determine `tx_index` consistent with `N`'s position (e.g., index 0 for the left subtree root).
5. Call `verify_transaction_inclusion` on the NEAR contract:
   ```json
   {
     "tx_id": "<internal_node_N_as_hex>",
     "tx_block_blockhash": "<hash_of_the_block>",
     "tx_index": 0,
     "merkle_proof": ["<sibling_at_depth_d>", ...],
     "confirmations": 1
   }
   ```
6. The contract computes `compute_root_from_merkle_proof(N, 0, proof)`, arrives at the correct `merkle_root`, and returns `true` — despite `N` not being any real transaction hash. [6](#0-5)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
