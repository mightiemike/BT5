### Title
Missing Sanity Check on `gc_threshold` in `init` Corrupts SPV Verification and GC Invariants - (File: contract/src/lib.rs)

### Summary
The `init` function accepts `gc_threshold = 0` without any validation. This single misconfigured parameter silently breaks two independent protocol invariants: the garbage-collection loop becomes permanently over-aggressive, and the `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` SPV API rejects every call that requests any meaningful confirmation count, rendering the contract's core output useless to all downstream consumers.

### Finding Description
`init` stores `args.gc_threshold` directly into contract state with no lower-bound check. [1](#0-0) 

The documentation comment only *recommends* `gc_threshold = 52704`; it is never enforced. [2](#0-1) 

`gc_threshold` then governs two separate code paths:

**Path 1 – `run_mainchain_gc` (called on every `submit_blocks`):** [3](#0-2) 

With `gc_threshold = 0`, `amount_of_headers_we_store > 0` is always true after genesis, so `total_amount_to_remove = amount_of_headers_we_store - 0 = amount_of_headers_we_store`. Every `submit_blocks` call immediately attempts to erase up to `batch_size` of the oldest mainchain headers, draining the stored chain down toward zero blocks on each invocation.

**Path 2 – `verify_transaction_inclusion` (public SPV API):** [4](#0-3) 

With `gc_threshold = 0`, the guard `args.confirmations <= 0` passes only when `confirmations == 0`. Any caller requesting even a single confirmation (the minimum meaningful security threshold for Bitcoin SPV) panics with "The required number of confirmations exceeds the number of blocks stored in memory". `verify_transaction_inclusion_v2` delegates to the same function and is equally affected. [5](#0-4) 

### Impact Explanation
The contract's sole externally-visible security output is the boolean returned by `verify_transaction_inclusion` / `verify_transaction_inclusion_v2`. Recipient contracts and dApps on NEAR consume this result to gate asset releases or cross-chain actions. With `gc_threshold = 0`:

- Every call with `confirmations >= 1` panics unconditionally, so no legitimate SPV proof can ever be verified.
- Simultaneously, the GC loop erases mainchain headers on every block submission, so even a `confirmations = 0` call will eventually panic on a missing header lookup (`ERR_KEY_NOT_EXIST`) once the initial blocks are purged.

The contract is rendered permanently non-functional for its stated purpose without any on-chain upgrade path short of a full migration.

### Likelihood Explanation
`init` is `#[private]` (callable only by the contract account), so this requires an operator mistake rather than an external attacker. However, the parameter is a plain `u64` with no guard, the documentation only gives a recommendation, and the value `0` is the Rust default for the type. The original analogous report explicitly recognises that admin-protected initialisation functions are still realistic sources of undetected misconfiguration. A deployment script that omits the field or sets it to zero would silently produce a broken contract with no error at init time.

### Recommendation
Add an explicit lower-bound check inside `init` before storing `gc_threshold`:

```rust
require!(
    args.gc_threshold > 0,
    "gc_threshold must be greater than zero"
);
```

Optionally also enforce that `gc_threshold` is at least as large as `args.submit_blocks.len()` so that the blocks submitted during initialisation are not immediately eligible for GC on the very first `submit_blocks` call.

### Proof of Concept

1. Deploy the contract with `InitArgs { gc_threshold: 0, ... }` (all other fields valid).
2. `init` succeeds — no panic, no error.
3. Call `verify_transaction_inclusion` with `confirmations: 1` for any mainchain transaction.
4. The call panics: `"The required number of confirmations exceeds the number of blocks stored in memory"` because `1 <= 0` is false.
5. Call `submit_blocks` with one valid header. Inside `run_mainchain_gc`, `amount_of_headers_we_store > 0` triggers removal of the oldest stored header on every invocation, progressively destroying the mainchain state.
6. After enough `submit_blocks` calls, even `confirmations: 0` calls panic with `ERR_KEY_NOT_EXIST` as the initial block is purged.
