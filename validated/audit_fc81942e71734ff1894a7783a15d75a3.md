### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` for post-0.14.0 blocks, producing a wrong block hash — (File: `crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The Rust `gas_prices_to_hash` function carries an explicit TODO acknowledging that `l2_gas_consumed` and `next_l2_gas_price` must be added to the gas-prices sub-hash "after 0.14.0". The deployed Starknet version is already 0.14.4, yet no new `BlockHashVersion` variant has been introduced and those two fields are still absent from the hash preimage. The production `echonet/echo_center.py` component already passes both fields to the block-hash CLI for every block it processes. The result is a version-boundary mismatch: the Rust sequencer computes a block hash that omits fields the protocol now requires, while the echonet path (and any external verifier following the updated spec) includes them.

---

### Finding Description

`gas_prices_to_hash` is the sole function that serialises gas-price data into the block hash preimage. For `BlockHashVersion::V0_13_4` — which covers **all** Starknet versions ≥ 0.13.4, including 0.14.4 — it hashes exactly six felts: `l1_gas_price_{wei,fri}`, `l1_data_gas_price_{wei,fri}`, `l2_gas_price_{wei,fri}`.

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> {
    if block_hash_version >= &BlockHashVersion::V0_13_4 {
        vec![
            HashChain::new()
                .chain(&STARKNET_GAS_PRICES0)
                ...
                .chain(&l2_gas_price.price_in_wei.0.into())
                .chain(&l2_gas_price.price_in_fri.0.into())
                .get_poseidon_hash(),
        ]
    } ...
}
```

The `BlockHashVersion` enum has only two variants (`V0_13_2`, `V0_13_4`); there is no `V0_14_x` gate. The `TryFrom<StarknetVersion>` conversion maps every version ≥ 0.13.4 — including 0.14.4 — to `V0_13_4`, so the missing fields can never be activated by the current code.

Meanwhile, `StorageBlockHeader` already stores both fields:

```rust
pub struct StorageBlockHeader {
    ...
    pub l2_gas_consumed: GasAmount,
    pub next_l2_gas_price: GasPrice,
    ...
}
```

And the production `echonet/echo_center.py` passes them to the block-hash CLI for every block:

```python
"l2_gas_consumed": fee_market_info["l2_gas_consumed"],
"next_l2_gas_price": fee_market_info["next_l2_gas_price"],
```

The Rust `calculate_block_hash` — called from `commitment_manager_impl.rs` to finalise every sequenced block — therefore produces a different digest than the CLI and any external verifier that follows the updated spec. This is the direct analog of the DNS version mismatch: records (block hashes) are written under one version schema (V0_13_4, missing two fields) but the protocol now expects them under a newer schema (post-0.14.0, including `l2_gas_consumed` and `next_l2_gas_price`).

---

### Impact Explanation

Every block finalised by the Rust sequencer at Starknet version ≥ 0.14.x carries a block hash computed without `l2_gas_consumed` and `next_l2_gas_price`. Any component that independently recomputes the hash using the updated spec — including the echonet CLI, L1 verifiers, or state-sync peers — will derive a different value. This constitutes a **wrong committed state value** (the stored block hash) and breaks any downstream proof or cross-component verification that relies on hash consistency.

Matching impact: *Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

---

### Likelihood Explanation

The TODO is unambiguous ("add … after 0.14.0") and the current deployed version is 0.14.4. The `echonet/echo_center.py` production path already supplies the missing fields to the CLI. Any block produced at version ≥ 0.14.x triggers the divergence automatically, with no special attacker action required.

---

### Recommendation

1. Add a `BlockHashVersion::V0_14_x` variant (or the appropriate cutover version) to `BlockHashVersion`.
2. Extend `TryFrom<StarknetVersion> for BlockHashVersion` to map versions ≥ the cutover to the new variant.
3. In `gas_prices_to_hash`, add a branch for the new variant that appends `l2_gas_consumed` and `next_l2_gas_price` to the Poseidon hash chain.
4. Update `calculate_block_hash` callers (e.g., `commitment_manager_impl.rs`) to pass the two new fields.
5. Add a regression test analogous to `test_block_hash_regression` for the new version.

#[allow(non_camel_case_types)]
#[derive(Clone, Debug, PartialEq, Eq, PartialOrd)]
pub enum BlockHashVersion {
    V0_13_2,
    V0_13_4,
}

impl From<BlockHashVersion> for StarknetVersion {
    fn from(value: BlockHashVersion) -> Self {
        match value {
            BlockHashVersion::V0_13_2 => StarknetVersion::V0_13_2,
            BlockHashVersion::V0_13_4 => StarknetVersion::V0_13_4,
        }
    }
}

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
