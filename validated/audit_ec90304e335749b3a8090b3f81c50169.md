### Title
Deprecated `verify_transaction_inclusion` Remains a Live NEAR Entry Point, Enabling Merkle Proof Forgery Bypass — (File: contract/src/lib.rs)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a fully callable NEAR contract method despite being superseded by `verify_transaction_inclusion_v2`. Rust's `#[deprecated]` attribute is a compile-time hint only; it does not remove the function from the on-chain ABI. Any unprivileged NEAR caller can invoke it directly by name, bypassing the coinbase Merkle proof validation that v2 adds to mitigate the 64-byte transaction second-preimage attack. This creates a persistent insecure entry point that recipient contracts or integrators may unknowingly or deliberately target.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` carries `#[deprecated(since = "0.5.0")]` and a `# Warning` doc comment acknowledging the second-preimage vulnerability. However, `#[deprecated]` in Rust is a compiler lint only — it emits a warning to callers within the same Rust compilation unit. It has no effect on the NEAR contract ABI: the function is still exported as a public method and is callable by any NEAR account via RPC. [1](#0-0) 

The function performs no coinbase proof validation. It accepts a caller-supplied `tx_id` and a caller-supplied `merkle_proof`, computes the Merkle root, and returns `true` if it matches the stored block header's `merkle_root`: [2](#0-1) 

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by first verifying a coinbase proof (which anchors the proof depth) before delegating to v1: [3](#0-2) 

Because v1 remains callable directly, the coinbase-depth anchor in v2 is trivially bypassed: an attacker simply calls `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2`. The CLAUDE.md explicitly acknowledges the second-preimage risk but frames it as a caller responsibility, not a contract-level enforcement gap: [4](#0-3) 

The `ProofArgs` struct accepted by v1 is fully attacker-controlled — `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` are all supplied by the caller: [5](#0-4) 

---

### Impact Explanation

An attacker can supply an internal Merkle tree node hash as `tx_id` along with a valid Merkle path to the block's `merkle_root`. The function returns `true`, asserting that a non-existent transaction is included in a confirmed Bitcoin block. Any NEAR recipient contract that calls `verify_transaction_inclusion` by name — rather than v2 — receives a forged `true` result and may release funds, unlock state, or trigger cross-chain actions based on a fabricated Bitcoin transaction. The corrupted value is the SPV proof result: a `bool` that downstream contracts treat as a trust anchor for Bitcoin state.

---

### Likelihood Explanation

The 64-byte transaction / second-preimage attack against Bitcoin Merkle trees is publicly documented (referenced in the contract's own deprecation notice and at the BitMEX link in the doc comment). The attack does not require breaking any cryptographic primitive — it exploits the absence of transaction-count validation in block headers. Any Bitcoin block is a valid target. The attacker needs only to identify an internal node hash and construct the corresponding Merkle path, both of which are derivable from public block data. The entry point (`verify_transaction_inclusion`) is a public, permissionless NEAR view call requiring no deposit, no role, and no privileged key.

---

### Recommendation

Remove `verify_transaction_inclusion` from the on-chain ABI entirely, or gate it with `#[private]` so it is only callable by the contract itself (preserving the internal delegation from v2). If backward compatibility with existing integrators is required, add the coinbase proof validation directly into v1 and remove the separate v2 function. Do not rely on `#[deprecated]` as a security boundary — it is invisible at the NEAR protocol layer.

---

### Proof of Concept

1. Select any confirmed Bitcoin block already stored in the contract (e.g., block 277136, used in the relayer integration tests).
2. Fetch the full block and compute its Merkle tree. Identify any internal node hash `N` at depth `d`.
3. Construct a Merkle proof `P` of length `d` that leads from `N` to the block's `merkle_root`.
4. Call the NEAR contract view method `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash, not a real transaction)
   - `tx_block_blockhash` = the block hash of the chosen block
   - `tx_index` = the index corresponding to `N`'s position at depth `d`
   - `merkle_proof` = `P`
   - `confirmations` = `0`
5. The contract returns `true`, asserting that the non-existent transaction `N` is confirmed in the Bitcoin block.
6. A recipient contract consuming this result would treat the forged transaction as valid.
