### Title
Gateway Stateless Validator Uses Static Config `min_gas_price` Instead of Protocol-Versioned `VersionedConstants.min_gas_price` — (File: `crates/apollo_gateway_config/src/config.rs`)

---

### Summary

`StatelessTransactionValidator.validate_resource_bounds` enforces a minimum L2 gas price using `self.config.min_gas_price`, a static operator-set field in `StatelessTransactionValidatorConfig`. The orchestrator's `apollo_versioned_constants::VersionedConstants` carries a protocol-versioned `min_gas_price` that already changed between Starknet versions (3 Gwei in V0_14_0 → 8 Gwei in V0_14_1+). A TODO comment in the codebase explicitly acknowledges the divergence and calls for the static field to be removed in favour of the versioned value. When the two values diverge, the gateway admits transactions whose L2 gas price is below the protocol floor, which the stateful validator (checking against the actual base gas price) will then reject — matching the "High: gateway/RPC admission accepts invalid transactions" impact.

---

`StatelessTransactionValidatorConfig` carries its own `min_gas_price` field (default `8_000_000_000`, i.e. 8 Gwei): [1](#0-0) 

A TODO comment immediately above it reads: [2](#0-1) 

The stateless validator enforces this static value: [3](#0-2) 

Meanwhile, the orchestrator's `VersionedConstants` defines a protocol-versioned `min_gas_price` that already differs across deployed versions: [4](#0-3) [5](#0-4) 

The fee-market code that computes the next block's gas price uses `VersionedConstants::latest_constants().min_gas_price` as its floor: [6](#0-5) 

This guarantees the on-chain base gas price is always ≥ `VersionedConstants.min_gas_price`. The stateful validator then checks the transaction's `max_price_per_unit` against that base gas price (via `min_gas_price_percentage`). The stateless validator, however, checks only against the static config field — which can be stale or misconfigured relative to the versioned value.

---

### Impact Explanation

When `StatelessTransactionValidatorConfig.min_gas_price` < `VersionedConstants.min_gas_price` (e.g. after a protocol upgrade that raises the floor but before the operator updates the config), the gateway's stateless check passes for any transaction whose `l2_gas.max_price_per_unit` falls in the gap `[config_min, vc_min)`. Because the fee market guarantees `base_gas_price ≥ vc_min`, the stateful validator (100 % of base gas price) will reject those same transactions. The gateway therefore admits transactions that are protocol-invalid — matching **High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

---

### Likelihood Explanation

The versioned `min_gas_price` already changed once (V0_14_0 → V0_14_1). Any future protocol upgrade that raises it again will reproduce the gap unless the operator simultaneously updates the static config. The TODO comment confirms the design intent is to eliminate the static field entirely, meaning the current dual-source architecture is acknowledged as incorrect.

---

### Recommendation

Remove `min_gas_price` from `StatelessTransactionValidatorConfig` as the TODO directs, and replace the check in `validate_resource_bounds` with the value from `VersionedConstants::latest_constants().min_gas_price` (or the block-version-appropriate constant). This eliminates the possibility of the two values diverging across protocol upgrades.

---

### Proof of Concept

1. Deploy the sequencer with the default static config (`min_gas_price = 8_000_000_000`).
2. Apply a protocol upgrade that sets `VersionedConstants.min_gas_price` to `10_000_000_000` (10 Gwei) without updating the static config.
3. Submit an `InvokeV3` transaction with `resource_bounds.l2_gas.max_price_per_unit = 9_000_000_000` (9 Gwei).
4. `StatelessTransactionValidator.validate_resource_bounds` passes: `9e9 >= 8e9`. [7](#0-6) 
5. The fee market has already set `base_gas_price ≥ 10e9` (guaranteed by `get_min_gas_price_for_height` falling back to `VersionedConstants::latest_constants().min_gas_price`). [6](#0-5) 
6. The stateful validator rejects the transaction (`9e9 < 10e9`), but the gateway has already accepted and forwarded it — the admission invariant is broken.
