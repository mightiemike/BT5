### Title
Fee-on-Transfer Token Deposit Overcredits User Balance — (`core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers `amount` tokens from the caller and then encodes that same `amount` into a slow-mode `DepositCollateral` transaction. When the sequencer later processes that transaction, `Clearinghouse.depositCollateral` credits the user's internal balance for the full `amount`. If the collateral token charges a transfer fee (fee-on-transfer ERC-20), the clearinghouse receives fewer tokens than `amount`, but the user is credited for the full `amount`. This inflates `totalDepositsNormalized` beyond the actual token holdings, causing a solvency shortfall that falls on the last users to withdraw.

---

### Finding Description

In `Endpoint.depositCollateralWithReferral`, the deposit flow is:

1. Transfer `amount` tokens from the caller to the contract.
2. Enqueue a slow-mode transaction encoding `amount` verbatim. [1](#0-0) 

When the sequencer executes the slow-mode transaction, `Clearinghouse.depositCollateral` is called:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [2](#0-1) 

`txn.amount` is the original user-supplied value — not the actual tokens received. For a fee-on-transfer token with a 1% fee, a deposit of 100 tokens results in only 99 tokens held by the clearinghouse, but the user's balance is credited for 100. The 1-token discrepancy is never reconciled.

The `ERC20Helper.safeTransferFrom` helper performs no balance-before/after check: [3](#0-2) 

The same pattern applies to `DepositInsurance` in `EndpointTx.submitSlowModeTransactionImpl`, which also calls `handleDepositTransfer` with the raw `txn.amount`: [4](#0-3) 

---

### Impact Explanation

`SpotEngine.totalDepositsNormalized` grows by more than the actual tokens held. On withdrawal, `assertUtilization` enforces `totalDeposits >= totalBorrows`: [5](#0-4) 

Because the accounting balance exceeds the real token balance, the protocol will eventually be unable to satisfy all withdrawals. Early withdrawers are made whole; late withdrawers suffer losses proportional to the accumulated fee discrepancy. This is a direct solvency/accounting corruption with concrete asset loss for users.

---

### Likelihood Explanation

The likelihood depends on whether a fee-on-transfer token is ever listed as a collateral product. The protocol does not enforce any restriction against such tokens in `addOrUpdateProduct`: [6](#0-5) 

Any operator who lists a fee-on-transfer token (e.g., USDT on certain chains, or tokens with configurable fees) opens this attack path. An unprivileged user can then exploit it simply by depositing and withdrawing, with no special permissions required.

---

### Recommendation

Replace the static `amount` used for accounting with the actual received amount, measured via a balance-before/after check in `handleDepositTransfer`. The credited amount passed into the slow-mode transaction should reflect what the contract actually received, not what the caller specified.

---

### Proof of Concept

1. A fee-on-transfer token (1% fee) is listed as a collateral product.
2. Alice calls `Endpoint.depositCollateralWithReferral(subaccount, productId, 100e6, "")`.
   - `handleDepositTransfer` pulls 100e6 from Alice; the token's fee burns 1e6, so the clearinghouse receives 99e6.
   - The slow-mode tx encodes `amount = 100e6`.
3. Sequencer processes the slow-mode tx; `Clearinghouse.depositCollateral` credits Alice with `100e6 * multiplier`.
4. Alice withdraws 100e6 — the clearinghouse sends 100e6 out but only held 99e6 from Alice's deposit.
5. After many such cycles, `totalDepositsNormalized` exceeds the real token balance; the last user to withdraw finds the clearinghouse insolvent for their product. [7](#0-6) [8](#0-7)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
