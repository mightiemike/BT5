### Title
EIP-712 Domain Separator Uses `address(uint160(productId))` Instead of `address(this)`, Enabling Cross-Deployment Signature Replay ŌĆö (`core/contracts/OffchainExchange.sol`)

---

### Summary

`getDigest` constructs the EIP-712 domain separator with `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. This means the digest ŌĆö and therefore every order signature ŌĆö is not bound to the specific `OffchainExchange` contract instance. Any two deployments of `OffchainExchange` on the same chain sharing the same `productId` will produce identical digests for identical orders, making signatures fully portable across deployments.

---

### Finding Description

In `OffchainExchange.sol`, `getDigest` builds the domain separator as:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),       // "Nado"
        _EIP712VersionHash(),    // "0.0.1"
        block.chainid,
        address(uint160(productId))  // ŌåÉ NOT address(this)
    )
);
``` [1](#0-0) 

EIP-712 requires the `verifyingContract` to be the address of the contract that will verify the signature, so that a signature is cryptographically bound to one specific contract instance. Using `address(uint160(productId))` ŌĆö a deterministic, deployment-independent value ŌĆö breaks this binding entirely.

The `matchOrders` entry point calls `getDigest` to compute the digest for both taker and maker orders before passing them to `_validateOrder` ŌåÆ `_checkSignature`: [2](#0-1) 

`_checkSignature` recovers the signer from the digest and checks it against the subaccount owner or linked signer: [3](#0-2) 

Because the digest is identical across any two `OffchainExchange` instances with the same `productId`, `chainId`, name, and version, a valid signature on instance A is also a valid signature on instance B.

---

### Impact Explanation

If two `OffchainExchange` deployments coexist on the same chain (e.g., a v1 alongside a v2 upgrade, a mainnet fork, or a parallel deployment), an attacker can:

1. Observe a signed order for `productId=P` submitted to deployment A.
2. Submit the identical `MatchOrdersWithSigner` transaction to deployment B's Endpoint.
3. `getDigest` on B produces the same bytes32 as on A (same `productId`, same `chainId`, same name/version).
4. `_checkSignature` passes ŌĆö the recovered signer matches the subaccount owner.
5. The trade executes on B against a market the user never intended to trade on.

This constitutes unauthorized trade execution: the user's assets on deployment B are moved without their consent, directly satisfying the Critical scope of solvency/accounting failure that transfers value incorrectly.

---

### Likelihood Explanation

The `matchOrders` function is gated by `onlyEndpoint`: [4](#0-3) 

This means the attacker must route the replayed order through the second deployment's Endpoint. This is a realistic constraint ŌĆö Endpoints accept externally submitted transaction batches ŌĆö but it does require a second live deployment on the same chain. Protocol upgrades, parallel deployments, or forks are all realistic scenarios where this condition is met. The root cause (wrong `verifyingContract`) is unconditional and present in every deployment.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction inside `getDigest`:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // ŌåÉ correct: binds signature to this contract instance
    )
);
``` [1](#0-0) 

Alternatively, use the inherited `_domainSeparatorV4()` from `EIP712Upgradeable`, which already uses `address(this)` correctly and is initialized via `__EIP712_init("Nado", "0.0.1")`: [5](#0-4) 

Note that changing the domain separator is a breaking change for all existing outstanding signed orders; a migration plan is required.

---

### Proof of Concept

```solidity
// Deploy two OffchainExchange instances on the same chain, both with productId=1 registered.
// Instance A: address 0xAAAA...
// Instance B: address 0xBBBB...

// User signs an order for productId=1 on instance A.
IEndpoint.Order memory order = IEndpoint.Order({
    sender: userSubaccount,
    priceX18: 1e18,
    amount: 1e18,
    expiration: block.timestamp + 1 hours,
    nonce: 1,
    appendix: 1  // version=1
});

bytes32 digestA = offchainExchangeA.getDigest(1, order);
bytes32 digestB = offchainExchangeB.getDigest(1, order);

// Both digests are identical ŌĆö same productId, chainId, name, version.
assert(digestA == digestB);

// Signature valid on A is also valid on B.
(uint8 v, bytes32 r, bytes32 s) = vm.sign(userPrivKey, digestA);
bytes memory sig = abi.encodePacked(r, s, v);

// Submit to instance B's endpoint ŌĆö _checkSignature passes, trade executes.
endpointB.submitMatchOrders(IEndpoint.MatchOrdersWithSigner({
    matchOrders: IEndpoint.MatchOrders({ productId: 1, taker: IEndpoint.SignedOrder(order, sig), ... }),
    takerLinkedSigner: address(0),
    makerLinkedSigner: address(0)
}));
```

The `assert(digestA == digestB)` holds because `address(uint160(1))` is the same constant regardless of which `OffchainExchange` instance calls `getDigest`. [6](#0-5)
