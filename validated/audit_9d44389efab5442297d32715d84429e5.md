### Title
Slow-Mode `LinkSigner` Does Not Increment Nonce, Enabling Replay of Old Linked-Signer Signatures After Key Reuse — (`core/contracts/EndpointTx.sol`)

---

### Summary

The slow-mode path for `LinkSigner` in `EndpointTx.sol` updates `linkedSigners[sender]` without calling `validateNonce`. Because the per-address nonce is never incremented on a slow-mode signer change, any previously signed transaction that was not yet consumed remains valid at the same nonce value. If the linked signer is later restored to a previously used address — either by the user or by a sequencer replaying a queued `SignedLinkSigner` — every signature the old signer produced at the current nonce becomes executable again.

---

### Finding Description

`EndpointStorage.sol` stores two independent mappings:

```
mapping(address => uint64) internal nonces;          // keyed by first 20 bytes of subaccount
mapping(bytes32 => address) internal linkedSigners;  // keyed by full bytes32 subaccount
``` [1](#0-0) 

The sequencer path for `LinkSigner` calls `validateSignedTx`, which calls `validateNonce` and increments the nonce before writing the new signer:

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
``` [2](#0-1) 

The slow-mode path for the same `LinkSigner` transaction type does **not** call `validateNonce` at all:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
``` [3](#0-2) 

`validateNonce` is the only mechanism that advances the counter and thereby invalidates all prior signatures:

```solidity
function validateNonce(bytes32 sender, uint64 nonce) internal virtual {
    require(nonce == nonces[address(uint160(bytes20(sender)))]++, ERR_WRONG_NONCE);
}
``` [4](#0-3) 

Because the slow-mode `LinkSigner` path skips this call, the nonce counter is frozen at its current value after the signer is swapped. Any signature produced by the old linked signer at that nonce is still structurally valid the moment the old signer address is reinstated.

Signature validation accepts either the subaccount owner address or the current linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [5](#0-4) 

Any unprivileged user can submit a slow-mode `LinkSigner` by paying the slow-mode fee; the transaction type is not in the owner-only guard: [6](#0-5)

### Impact Explanation

An attacker (or the sequencer acting on a stale queue entry) who possesses a signed `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, or `BurnNlp` payload from a linked signer can replay it after the following sequence:

1. The user signed transaction T at nonce N with linked signer A.
2. The user submitted a slow-mode `LinkSigner` to switch to signer B — nonce stays at N.
3. The user (or the sequencer) later submitted a slow-mode `LinkSigner` to restore signer A — nonce still at N.
4. Transaction T, signed by A at nonce N, is now valid on-chain.

The most severe case is `WithdrawCollateral`: the replayed transaction drains the subaccount's collateral to the attacker-controlled recipient without any new authorization from the user. The user believed the transaction was cancelled when they changed their linked signer.

---

### Likelihood Explanation

- Any user can submit slow-mode `LinkSigner` transactions by paying the fee — no privilege required.
- Signed transaction payloads are submitted to the sequencer's public API; an observer (including the sequencer itself if it retains queued transactions) can hold them.
- Linked-signer rotation is a documented user flow. Reverting to a previously used signer address (e.g., switching back to a hot wallet after temporarily using a cold wallet) is a realistic, non-adversarial action that silently re-enables old signatures.
- The three-day slow-mode delay provides a window during which the user may not realize the risk.

---

### Recommendation

**Short term:** In `processSlowModeTransactionImpl`, call `validateNonce` (or an equivalent nonce-bump helper) when processing a `LinkSigner` transaction, mirroring the sequencer path. This ensures that every signer change — regardless of path — advances the nonce and permanently invalidates all prior signatures.

**Long term:** Add a blacklist of previously used linked-signer addresses per subaccount, or document explicitly that linked-signer reuse is prohibited and that slow-mode signer changes do not cancel pending sequencer-queued transactions.

---

### Proof of Concept

```
State:  subaccount S, linkedSigners[S] = A, nonces[addr(S)] = N

Step 1: User signs WithdrawCollateral{sender=S, amount=X, nonce=N} with key A → bytes sigT

Step 2: User calls submitSlowModeTransaction(
            abi.encodePacked(uint8(LinkSigner), abi.encode(LinkSigner{sender=S, signer=B, nonce:ignored}))
        )
        → After 3 days, processSlowModeTransactionImpl runs:
            linkedSigners[S] = B
            nonces[addr(S)] still = N   ← no increment

Step 3: User calls submitSlowModeTransaction(
            abi.encodePacked(uint8(LinkSigner), abi.encode(LinkSigner{sender=S, signer=A, nonce:ignored}))
        )
        → After 3 days:
            linkedSigners[S] = A
            nonces[addr(S)] still = N   ← still no increment

Step 4: Attacker submits sigT to sequencer API.
        Sequencer calls processTransactionImpl with WithdrawCollateral{sender=S, amount=X, nonce=N, sig=sigT}.
        validateNonce: N == N ✓  (increments to N+1)
        validateSignature: recover(sigT) == A == linkedSigners[S] ✓
        clearinghouse.withdrawCollateral executes → X tokens leave subaccount S.
```

The broken invariant: a slow-mode `LinkSigner` must advance the nonce so that all signatures produced under the previous linked signer are permanently invalidated, matching the guarantee provided by the sequencer-path `LinkSigner`.
