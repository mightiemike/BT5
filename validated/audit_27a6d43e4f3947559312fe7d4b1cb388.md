### Title
Cross-Contract State Desynchronization on `setWithdrawPool` Migration Enables Fast Withdrawal Replay — (`File: core/contracts/Clearinghouse.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary
When the owner calls `setWithdrawPool` to migrate to a new `WithdrawPool` contract, the new pool starts with a blank `markedIdxs` mapping and `minIdx = 0`. Any user who previously submitted a fast withdrawal against the old pool can replay the identical `(idx, transaction, signatures)` tuple against the new pool, receiving the same withdrawal amount a second time.

---

### Finding Description

`Clearinghouse.setWithdrawPool` replaces the active withdrawal pool address atomically: [1](#0-0) 

The new `WithdrawPool` is a freshly deployed contract. Its deduplication state — `markedIdxs` and `minIdx` — is entirely empty: [2](#0-1) 

`submitFastWithdrawal` enforces deduplication solely through these two fields: [3](#0-2) 

Because the new pool has `minIdx = 0` and no entries in `markedIdxs`, every `idx > 0` passes both guards. The Schnorr signature check (`v.requireValidTxSignatures`) verifies the sequencer's signature over `(transaction, idx)` — a tuple that is identical to the one used in the original fast withdrawal and therefore still valid: [4](#0-3) 

After passing both checks, the pool transfers tokens to `sendTo`: [5](#0-4) 

The old pool's processed-withdrawal history is never migrated. The new pool is permanently desynchronized from the historical record of fast withdrawals.

The `submitWithdrawal` path (called by the Clearinghouse sequencer path) also updates `minIdx` only for new sequencer-submitted withdrawals: [6](#0-5) 

This means `minIdx` in the new pool advances only as the sequencer processes new withdrawals, leaving a window where all historical fast-withdrawal `idx` values remain replayable.

---

### Impact Explanation

An attacker who previously submitted a fast withdrawal with `idx = N` from the old pool can call `submitFastWithdrawal(N, transaction, signatures)` on the new pool immediately after `setWithdrawPool` is called. Since `N > 0 = minIdx` and `markedIdxs[N] = false` in the new pool, both guards pass. The attacker receives the same token amount a second time, draining funds from the new pool. The corrupted state delta is: `WithdrawPool.token.balanceOf(newPool)` decreases by `transferAmount` for each replayed withdrawal, with no corresponding on-chain accounting debit in the Clearinghouse.

---

### Likelihood Explanation

`setWithdrawPool` is a planned administrative operation (e.g., during a contract upgrade cycle). Once called, the replay window opens immediately and persists until the sequencer has processed enough new withdrawals to advance `minIdx` past all historical fast-withdrawal `idx` values. Any user who ever submitted a fast withdrawal can exploit this without any privileged access — they already hold the valid `(idx, transaction, signatures)` tuple from their original submission. The new pool will hold real token balances as soon as the Clearinghouse routes new withdrawals to it.

---

### Recommendation

Before activating the new pool via `setWithdrawPool`, transfer the deduplication state from the old pool to the new one. At minimum, initialize the new pool's `minIdx` to the current `minIdx` of the old pool, and either copy all `markedIdxs` entries for `idx > newMinIdx` or enforce that `setWithdrawPool` can only be called when no fast withdrawals have been submitted since the last sequencer-submitted withdrawal (i.e., `minIdx` covers all outstanding fast-withdrawal `idx` values). Alternatively, add a `setMinIdx(uint64)` owner function to the new pool so the migration can atomically set a safe floor before the pool goes live.

---

### Proof of Concept

1. User calls `submitFastWithdrawal(50, txData, sigs)` on `OldPool`. Tokens are transferred. `OldPool.markedIdxs[50] = true`, `OldPool.minIdx = 49` (set by last sequencer withdrawal).
2. Owner calls `Clearinghouse.setWithdrawPool(NewPool)`. `NewPool.minIdx = 0`, `NewPool.markedIdxs` is empty.
3. Clearinghouse processes one new sequencer withdrawal to `NewPool`, funding it with tokens. `NewPool.minIdx = 1`.
4. User calls `submitFastWithdrawal(50, txData, sigs)` on `NewPool`.
   - `!markedIdxs[50]` → `true` (passes)
   - `50 > 1` → `true` (passes)
   - `requireValidTxSignatures(txData, 50, sigs)` → passes (same valid sequencer signature)
   - `NewPool` transfers tokens to user a second time.
5. User has received double the withdrawal amount; `NewPool` is drained by the replayed amount. [7](#0-6) [8](#0-7)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
