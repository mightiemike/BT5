### Title
Repeated Margin Drain via Distinct-Digest `createIsolatedSubaccount` Calls on Existing Isolated Subaccount — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount` uses `digestToSubaccount[digest] != bytes32(0)` as its sole replay guard. Because the digest commits to the full `Order` struct (including `nonce` and `priceX18`), N signed orders with distinct nonces or prices produce N distinct digests. Each call bypasses the early-return, finds the already-existing isolated subaccount in the loop, and unconditionally executes the margin transfer block — deducting `margin` from the parent and crediting the isolated subaccount N times. No health check is performed on the parent after the deduction, so its `QUOTE_PRODUCT_ID` balance can go arbitrarily negative.

---

### Finding Description

**Entry point** — `EndpointTx.sol` lines 619–631:

```solidity
} else if (txType == IEndpoint.TransactionType.CreateIsolatedSubaccount) {
    IEndpoint.CreateIsolatedSubaccount memory txn = abi.decode(...);
    bytes32 newIsolatedSubaccount = IOffchainExchange(offchainExchange)
        .createIsolatedSubaccount(txn, getLinkedSigner(txn.order.sender));
    _recordSubaccount(newIsolatedSubaccount);
}
```

Unlike every other signed transaction type (`LinkSigner`, `TransferQuote`, `WithdrawCollateral`, etc.), **`validateSignedTx` is never called here**, so the on-chain sequential-nonce counter (`nonces[sender]++`) is never enforced for this transaction type. [1](#0-0) [2](#0-1) 

**Replay guard** — `OffchainExchange.sol` line 1009:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];
}
```

This is the only guard against re-execution. It fires only when the *exact same digest* is resubmitted. [3](#0-2) 

**Digest construction** — `OffchainExchange.sol` lines 291–322:

```solidity
bytes32 structHash = keccak256(abi.encode(
    keccak256(bytes(structType)),
    order.sender, order.priceX18, order.amount,
    order.expiration, order.nonce, order.appendix
));
```

The digest commits to `nonce` and `priceX18`. Two orders that differ in either field produce two distinct digests, each of which passes the guard independently. [4](#0-3) 

**Existing-subaccount lookup + unconditional margin transfer** — `OffchainExchange.sol` lines 1025–1087:

```solidity
for (uint256 id = 0; (1 << id) <= mask; id += 1) {
    ...
    if (productId == txn.productId) {
        newIsolatedSubaccount = subaccount;   // found existing
        break;
    }
}
// newIsolatedSubaccount != 0, so the creation block is skipped
digestToSubaccount[digest] = newIsolatedSubaccount;   // line 1072

int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);   // parent drained
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin); // iso credited
}
```

When the loop finds an existing isolated subaccount for `txn.productId`, execution falls through to line 1072 and then to the margin transfer block. There is **no guard** that skips the transfer when the subaccount already existed. [5](#0-4) [6](#0-5) 

**No post-transfer health check** — `SpotEngine.updateBalance` (lines 207–225) applies the delta unconditionally; the parent's balance is never checked for non-negativity after the deduction inside `createIsolatedSubaccount`. [7](#0-6) 

---

### Impact Explanation

After N calls with distinct digests (same `productId`, same `sender`, different `nonce`):

- `spotEngine.getBalance(QUOTE_PRODUCT_ID, parent).amount == initial - N * margin`
- `spotEngine.getBalance(QUOTE_PRODUCT_ID, isolatedSubaccount).amount == N * margin`

The parent's balance goes deeply negative (bad debt). The isolated subaccount holds excess collateral that was never legitimately deposited, violating the invariant that total collateral is conserved. The isolated subaccount can then be used to open leveraged positions backed by phantom margin.

---

### Likelihood Explanation

A trader signs N `CreateIsolatedSubaccount` orders for the same `productId` with incrementing `nonce` values (or varying `priceX18`) and submits them to the sequencer. Each is a valid, well-formed signed transaction. The sequencer has no on-chain reason to reject any of them. The attack requires only the trader's own private key and the ability to submit multiple signed orders — no admin access, no oracle manipulation, no external dependency.

---

### Recommendation

Add a guard immediately after the existing-subaccount lookup that skips the margin transfer when the isolated subaccount already existed before this call:

```solidity
bool isNewSubaccount = (newIsolatedSubaccount == bytes32(0));

if (newIsolatedSubaccount == bytes32(0)) {
    // ... creation logic ...
}

digestToSubaccount[digest] = newIsolatedSubaccount;

// Only transfer margin when the subaccount is newly created by this digest
if (isNewSubaccount) {
    int128 margin = int128(_isolatedMargin(txn.order.appendix));
    if (margin > 0) {
        digestToMargin[digest] = margin;
        spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.order.sender, -margin);
        spotEngine.updateBalance(QUOTE_PRODUCT_ID, newIsolatedSubaccount, margin);
    }
}
```

Alternatively, enforce the sequential nonce counter (`validateSignedTx`) for `CreateIsolatedSubaccount` in `EndpointTx.sol`, consistent with all other signed transaction types.

### Proof of Concept

```solidity
// Pseudocode — runs on unmodified Hardhat fork
address parent = ...; // funded with 1000 USDC in spotEngine
uint32 productId = 1;
uint128 margin = 100e6; // 100 USDC per call

for (uint64 n = 0; n < 10; n++) {
    IEndpoint.Order memory order = IEndpoint.Order({
        sender: bytes32(bytes20(parent)),
        priceX18: 1e18,
        amount: 1e18,
        expiration: block.timestamp + 1 days,
        nonce: n,                          // distinct nonce → distinct digest
        appendix: buildAppendix(margin)    // isolated flag set, margin encoded
    });
    bytes memory sig = sign(order, parentKey);
    endpoint.submitTransaction(
        abi.encodePacked(
            uint8(TransactionType.CreateIsolatedSubaccount),
            abi.encode(IEndpoint.CreateIsolatedSubaccount({
                order: order, productId: productId, signature: sig
            }))
        )
    );
}

// After 10 calls:
assert(spotEngine.getBalance(QUOTE_PRODUCT_ID, parentSubaccount).amount == 1000e6 - 10 * 100e6); // -0 → deeply negative
assert(spotEngine.getBalance(QUOTE_PRODUCT_ID, isolatedSubaccount).amount == 10 * 100e6);
```

Each iteration produces a unique digest (different `nonce`), bypasses the `digestToSubaccount` guard, finds the existing isolated subaccount in the loop, and executes the margin transfer — draining the parent 10× and crediting the isolated subaccount 10×.
