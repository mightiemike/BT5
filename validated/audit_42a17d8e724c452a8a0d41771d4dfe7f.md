### Title
Linked Signer Self-Escalation via `LinkSigner` with `allowLinkedSigner=true` — (`core/contracts/EndpointTx.sol`)

### Summary

The `LinkSigner` transaction handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`. This allows the *current* linked signer to sign a new `LinkSigner` transaction, replacing themselves with an attacker-controlled address. The new address can then immediately sign a `WithdrawCollateral` transaction, draining the subaccount — all without the subaccount owner's involvement.

---

### Finding Description

In `EndpointTx.sol`, the `LinkSigner` branch of `processTransactionImpl` is:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // allowLinkedSigner = true  ← root cause
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes `getLinkedSigner(sender)` to `Verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if the recovered address equals either the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the current linked signer can sign a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any address they choose.

The `WithdrawCollateral` handler also uses `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true
);
...
clearinghouse.withdrawCollateral(...);
``` [4](#0-3) 

---

### Impact Explanation

A linked signer can escalate their own privileges by re-linking the subaccount to a second attacker-controlled address, then using that address to withdraw all collateral. The subaccount owner's private key is never needed after the initial link is established. This breaks the invariant that only the subaccount owner can change who the linked signer is, and enables complete collateral theft.

---

### Likelihood Explanation

Any party who has ever been granted linked-signer status for a subaccount can execute this attack unilaterally. No admin access, sequencer compromise, or leaked keys are required. The attack is executable in two sequenced transactions through the normal `processTransactionImpl` path.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type so that only the subaccount owner (the address embedded in the `sender` bytes32) can authorize a signer change:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only subaccount owner may re-link
);
``` [5](#0-4) 

---

### Proof of Concept

```
Precondition: linkedSigners[subaccount] = attacker

Step 1 — Re-link:
  attacker crafts SignedLinkSigner {
      tx.sender = subaccount,
      tx.signer = attacker2,
      tx.nonce  = N
  } signed by attacker

  → processTransactionImpl(LinkSigner)
  → validateSignedTx(..., allowLinkedSigner=true)
  → getLinkedSigner(subaccount) == attacker  ✓ signature accepted
  → linkedSigners[subaccount] = attacker2

Step 2 — Drain:
  attacker2 crafts SignedWithdrawCollateral {
      tx.sender    = subaccount,
      tx.productId = USDC,
      tx.amount    = full balance,
      tx.nonce     = N+1
  } signed by attacker2

  → processTransactionImpl(WithdrawCollateral)
  → validateSignedTx(..., allowLinkedSigner=true)
  → getLinkedSigner(subaccount) == attacker2  ✓ signature accepted
  → clearinghouse.withdrawCollateral drains subaccount

Assert: subaccount balance == 0, funds transferred to attacker2's address.
```
