### Title
No Per-Week Token Bucket Isolation Allows Cross-Week Token Drainage — (File: `core/contracts/Airdrop.sol`)

### Summary
`Airdrop.sol` does not track per-week token allocations. When a Merkle root is registered for a given week, no corresponding token amount is recorded or locked for that week. All weeks share a single token pool. If the off-chain Merkle tree for any week encodes a total claimable amount that exceeds what was intended for that week (e.g., due to a script bug), claimants for that week will drain tokens that belong to future weeks, permanently breaking those future claims.

### Finding Description
`registerMerkleRoot` stores only a `bytes32` root for a given week number. It neither accepts a deposit of tokens nor records the intended total claimable amount for that week. [1](#0-0) 

There is no state variable tracking per-week allocations. The contract's entire token balance is a single undifferentiated pool shared across all registered weeks. [2](#0-1) 

When `_claim` executes, it transfers directly from the contract's total token balance with no check against a per-week budget. [3](#0-2) 

The contract therefore has no on-chain invariant enforcing that `sum(claimed[week][*]) <= allocation[week]` for any week.

### Impact Explanation
If the off-chain Merkle tree generation script produces a tree for week `W` whose total claimable sum exceeds the tokens intended for week `W`, all claimants for week `W` with valid proofs can claim their (inflated) amounts. Because the contract holds a single shared pool, this directly consumes tokens that were deposited for weeks `W+1`, `W+2`, etc. When those future weeks open for claiming, the contract balance will be insufficient and legitimate claimants will receive nothing or the contract will revert on transfer, permanently breaking all subsequent week distributions.

### Likelihood Explanation
Likelihood is low but non-negligible. The Merkle tree is constructed off-chain, and any arithmetic bug, indexing error, or double-counting in the generation script would produce an inflated tree. The contract provides zero on-chain defense against this class of error. This matches the exact risk profile of the reference report: low probability, high impact.

### Recommendation
Introduce per-week token accounting. When `registerMerkleRoot` is called, require the caller to simultaneously deposit the exact total claimable amount for that week, and record it in a `mapping(uint32 => uint256) weeklyAllocation`. In `_claim`, deduct from `weeklyAllocation[week]` and revert if the deduction would underflow. This ensures each week's claims are strictly bounded by the tokens explicitly allocated for it, preventing any cross-week drainage.

### Proof of Concept
1. Owner calls `registerMerkleRoot(1, root1)` and separately sends 1,000 tokens to the contract (intended: 500 for week 1, 500 for week 2).
2. Due to a script bug, `root1` encodes a total claimable amount of 800 tokens across all week-1 recipients.
3. All week-1 recipients call `claim(...)` with valid proofs and collectively receive 800 tokens.
4. Owner calls `registerMerkleRoot(2, root2)` (no additional deposit needed since 500 were already sent).
5. Week-2 recipients attempt to claim their 500 tokens, but only 200 remain in the contract.
6. `SafeERC20.safeTransfer` reverts for any claimant whose amount exceeds the remaining 200-token balance, permanently locking out legitimate week-2 claimants. [4](#0-3)
