### Title
`run_mainchain_gc` Lacks Caller Restriction, Enabling Frontrun Attacks That Corrupt Canonical Chain State and Break Verification - (File: contract/src/lib.rs)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function callable by **any unprivileged NEAR account** when the contract is unpaused. An attacker can monitor the NEAR mempool for pending `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` calls and frontrun them by calling `run_mainchain_gc` with a large `batch_size`, permanently deleting the target block from the canonical chain mappings. The verification call then panics, causing the consumer protocol's cross-chain operation to fail irreversibly.

---

### Finding Description

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`: [1](#0-0) 

This decorator restricts access only when the contract is **paused**. During normal (unpaused) operation, the function is open to any caller with no role check. The function permanently removes block headers from three canonical mappings: [2](#0-1) 

Specifically, `remove_block_header` deletes entries from both `mainchain_header_to_height` and `headers_pool`, and `mainchain_height_to_header.remove` deletes the height→hash mapping. After removal, `mainchain_initial_blockhash` is advanced. These deletions are **permanent** — there is no way to re-insert a GC'd block into the canonical chain mappings.

The `verify_transaction_inclusion` function depends on both `mainchain_header_to_height` and `headers_pool` being populated for the target block: [3](#0-2) 

If the target block has been GC'd, the first lookup panics with `"block does not belong to the current main chain"` and the second with `"cannot find requested transaction block"`.

The `Role` enum defines `UnrestrictedRunGC` specifically to allow GC on a paused contract, which confirms the design intent that GC should be a controlled operation — yet no equivalent restriction exists for the unpaused path: [4](#0-3) 

---

### Impact Explanation

An attacker who frontruns a `verify_transaction_inclusion_v2` call targeting a block near the GC boundary causes that call to panic permanently. For consumer protocols (bridges, atomic swaps, cross-chain lending) that rely on a `true` return value to release funds or finalize state, the operation fails and cannot be retried — the block is gone from state and cannot be re-added to the canonical mappings. The corrupted state invariant is: `mainchain_header_to_height[target_blockhash]` and `headers_pool[target_blockhash]` are deleted before the verification read, turning a valid proof into an irrecoverable panic.

---

### Likelihood Explanation

NEAR Protocol has ~1-second block times and a public mempool. An attacker can monitor pending transactions, identify `verify_transaction_inclusion` calls targeting blocks within `gc_threshold` blocks of the current tail, and submit `run_mainchain_gc(u64::MAX)` ahead of them. The attack is cheap (no stake or special role required), repeatable, and targeted — the attacker can selectively block specific verification calls without disrupting others.

---

### Recommendation

Restrict direct external calls to `run_mainchain_gc` to a privileged role. The internal call from `submit_blocks` (line 181) can remain unrestricted since it is already gated by `#[trusted_relayer]`. One approach:

```rust
// Before:
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {

// After: require DAO or RelayerManager for external calls
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id())
            || env::predecessor_account_id() == env::current_account_id(),
        "Unauthorized: only DAO, RelayerManager, or self may call run_mainchain_gc"
    );
    // ... existing logic
}
```

The `env::current_account_id()` check allows the internal self-call path from `submit_blocks` to continue working.

---

### Proof of Concept

1. Contract is deployed with `gc_threshold = 52704`. The canonical chain has grown to 52705 blocks (one block above threshold), so `total_amount_to_remove = 1`.
2. A bridge contract submits `verify_transaction_inclusion_v2` for the block at `mainchain_initial_blockhash` (the oldest stored block, height H).
3. The attacker sees this transaction in the NEAR mempool and submits `run_mainchain_gc { batch_size: 1 }` with higher gas priority.
4. `run_mainchain_gc` executes first: it calls `remove_block_header(&blockhash_at_H)` (deleting from `mainchain_header_to_height` and `headers_pool`) and `mainchain_height_to_header.remove(&H)`, then advances `mainchain_initial_blockhash` to H+1.
5. The bridge's `verify_transaction_inclusion_v2` executes next. It calls `headers_pool.get(&args.tx_block_blockhash)` — the entry is gone — and panics with `"cannot find requested transaction block"`.
6. The bridge's cross-chain operation fails permanently. The block cannot be re-submitted to the canonical mappings because `submit_block_header_inner` only inserts into `mainchain_height_to_header` / `mainchain_header_to_height` for new tips or reorgs, not for re-insertion of GC'd historical blocks.     #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
