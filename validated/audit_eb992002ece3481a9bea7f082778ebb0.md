Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Un-Allowlisted Addresses to Bypass the Deposit Guard — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address actually calling `addLiquidity` and supplying tokens) and validates only `owner` (the LP position beneficiary) against the per-pool allowlist. Any un-allowlisted address can bypass the guard by nominating any allowlisted address as the LP position `owner`, injecting liquidity into a permissioned pool without approval.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` encodes and forwards both `sender` (the `addLiquidity` caller) and `owner` (LP position beneficiary) to every registered extension: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` receives both parameters but discards `sender` (unnamed first parameter) and checks only `owner`: [2](#0-1) 

The check at line 38 is `allowedDepositor[msg.sender][owner]` — `owner` is the position beneficiary, not the token supplier. The actual depositor (`sender`) is never validated.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly names and checks `sender`: [3](#0-2) 

Additionally, the override drops the `onlyPool` modifier present in the base class: [4](#0-3) 

The override is declared `view` with no `onlyPool` guard, meaning any address can call it directly — though this secondary issue is less impactful because `msg.sender` is used as the pool key and no allowlist entry would exist for an arbitrary caller.

## Impact Explanation

A pool configured with `DepositAllowlistExtension` is intended to be a permissioned liquidity venue (e.g., KYC-gated, institutional-only). Because `sender` is never checked:

1. **Allowlist bypass**: Any un-allowlisted address `B` calls `pool.addLiquidity(owner=A, ...)` where `A` is any allowlisted address. The extension sees `allowedDepositor[pool][A] == true` and passes. `B` successfully injects liquidity into a permissioned pool without approval — a direct admin-boundary break.
2. **Liquidity manipulation**: `B` can place tokens into specific bins, altering the pool's depth ladder and affecting oracle-anchored pricing for subsequent swaps — a fund-impacting consequence for existing LPs.
3. **Collusion path**: `B` deposits with `owner=A`; `A` later removes liquidity and returns tokens to `B` off-chain, giving `B` full economic participation in the pool with zero allowlist enforcement.

This meets the **admin-boundary break** allowed impact: the pool admin's configured access control is bypassed by an unprivileged path.

## Likelihood Explanation

- **Low effort**: The bypass requires only knowing one allowlisted address, trivially discoverable on-chain via `AllowedToDepositSet` events.
- **No special role needed**: Any EOA or contract can execute this.
- **Always applicable**: Any pool with at least one allowlisted depositor (i.e., any meaningfully configured pool) is vulnerable.

Likelihood: **Medium-High**.

## Recommendation

Change the `beforeAddLiquidity` check to validate `sender` (the actual depositor) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
// fixed:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the caller and the position owner, check both. Also restore the `onlyPool` modifier on the override to match the base class.

## Proof of Concept

```
Setup:
  Pool P has DepositAllowlistExtension E configured.
  Pool admin sets: allowedDepositor[P][Alice] = true
  Bob is NOT on the allowlist.

Attack:
  Bob calls pool.addLiquidity(
      owner  = Alice,   // allowlisted address
      salt   = 0,
      deltas = <Bob's tokens>,
      ...
  )

Extension check (DepositAllowlistExtension.sol line 38):
  msg.sender                    = P      (pool, correct)
  allowAllDepositors[P]         = false
  allowedDepositor[P][Alice]    = true   ← owner is checked, not sender
  → check passes, no revert

Result:
  Bob's tokens enter pool P.
  Alice holds the LP position (owner = Alice, salt = 0).
  Bob has bypassed the deposit allowlist entirely.
  Alice removes liquidity and returns tokens to Bob off-chain.
  Bob has effectively participated in a permissioned pool without approval.
```
