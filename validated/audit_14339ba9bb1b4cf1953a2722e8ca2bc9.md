### Title
Deposited Collateral Permanently Locked When Slow Mode Execution Fails Due to Explicitly Removed Fund-Return Mechanism — (`File: core/contracts/Endpoint.sol`)

---

### Summary

When a user deposits collateral via `depositCollateral()` or `depositCollateralWithReferral()`, tokens are transferred to the `Clearinghouse` immediately and atomically, but the on-chain credit to the user's subaccount is deferred via a slow mode transaction with a mandatory 3-day delay. If that slow mode transaction fails during execution (e.g., because the product was delisted in the intervening window), the catch block in `_executeSlowModeTransaction()` silently discards the failure. A code comment explicitly confirms that the fund-return mechanism was removed: `// try return funds now removed`. The user has no on-chain path to recover their tokens.

---

### Finding Description

`depositCollateralWithReferral()` in `Endpoint.sol` performs two sequential operations:

1. **Immediate token transfer**: `handleDepositTransfer()` moves tokens from the user → `Endpoint` → `Clearinghouse` atomically.
2. **Deferred credit**: A `DepositCollateral` slow mode transaction is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days). [1](#0-0) 

The slow mode transaction is later executed by `_executeSlowModeTransaction()`, which calls `this.processSlowModeTransaction()` inside a `try/catch`. If the inner call reverts for any reason, the catch block does nothing — and the comment at line 226 explicitly documents that the original fund-return path was removed: [2](#0-1) 

Inside `processSlowModeTransactionImpl()`, the `DepositCollateral` branch calls `clearinghouse.depositCollateral(txn)`. This call can revert if, for example, the product was delisted via a `DelistProduct` transaction that was processed in the 3-day window between deposit submission and slow mode execution. [3](#0-2) 

When this revert is caught and silently swallowed, the tokens already transferred to the `Clearinghouse` remain there, unattributed to any subaccount balance. There is no cancel function, no refund path, and no admin recovery function scoped to individual user deposits. [4](#0-3) 

---

### Impact Explanation

A user's deposited tokens become permanently locked inside the `Clearinghouse` contract with no on-chain mechanism to recover them. The `Clearinghouse` holds the tokens but never credits the user's subaccount balance, so the funds are effectively burned from the user's perspective. There is no `cancelSlowModeTx`, no `recoverFailedDeposit`, and no admin sweep function scoped to failed deposits. [5](#0-4) 

---

### Likelihood Explanation

The 3-day slow mode delay creates a concrete window during which a `DelistProduct` transaction can be submitted and executed by the sequencer, invalidating a pending deposit for that product. `DelistProduct` is a valid, documented transaction type processed by `processSlowModeTransactionImpl()`. [6](#0-5) 

Additionally, any other revert path inside `clearinghouse.depositCollateral()` — including future validation logic — would trigger the same silent-discard behavior. The likelihood is **medium**: it requires a product delisting to race with a pending deposit, which is a realistic operational event.

---

### Recommendation

Restore the fund-return mechanism in the `catch` block of `_executeSlowModeTransaction()`. When a `DepositCollateral` slow mode transaction fails, the contract should decode the transaction type, identify the deposited token and amount, and transfer the tokens back to the original sender. Alternatively, store a per-user claimable refund mapping that the user can withdraw after a failed slow mode execution. [7](#0-6) 

---

### Proof of Concept

1. Product `X` exists and is active.
2. User calls `depositCollateral(subaccountName, productIdX, amount)`.
   - `handleDepositTransfer` moves `amount` tokens: user → Endpoint → Clearinghouse.
   - A `DepositCollateral` slow mode tx is enqueued at `txCount` with `executableAt = now + 3 days`.
3. Within the 3-day window, the sequencer submits a `DelistProduct` transaction for product `X`, which is processed immediately via the fast path.
4. After 3 days, anyone calls `executeSlowModeTransaction()`.
   - `_executeSlowModeTransaction()` dequeues the `DepositCollateral` tx.
   - `this.processSlowModeTransaction()` is called; inside, `clearinghouse.depositCollateral(txn)` reverts because product `X` is delisted.
   - The `catch` block is entered; `gasleft()` checks pass; execution falls through to `// try return funds now removed`.
   - No tokens are returned. No event is emitted.
5. User's `amount` tokens remain in the `Clearinghouse`, unattributed. The user's subaccount balance for product `X` is zero. Funds are permanently lost. [8](#0-7) [9](#0-8)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
