### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

The contract exposes two transaction-inclusion verification functions. `verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. The original `verify_transaction_inclusion` was deprecated but **never removed and remains publicly callable** by any unprivileged NEAR account. Because the coinbase proof check — the security-critical step — is never invoked on the deprecated path, an attacker can supply a crafted internal Merkle tree node as `tx_id` and receive a `true` verification result for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion_v2` was added to fix a known Bitcoin Merkle tree vulnerability ([64-byte transaction forgery](https://www.bitmex.com/blog/64-Byte-Transactions)). Its fix is a coinbase merkle proof check that validates the tree structure before delegating to the original function: [1](#0-0) 

The coinbase check at lines 358–365 ensures the Merkle tree's leaf-node structure is sound. Only after that check passes does it call the deprecated function: [2](#0-1) 

The deprecated `verify_transaction_inclusion` is still a public, unpermissioned contract method: [3](#0-2) 

It has no coinbase proof requirement. It only computes a Merkle root from the supplied proof and compares it to the stored block header's `merkle_root`: [4](#0-3) 

**Attack path:**

1. Attacker identifies any block accepted into the contract's mainchain (publicly readable via `get_block_hash_by_height` / `get_last_block_header`).
2. Attacker obtains the block's full transaction list and computes an internal Merkle tree node (every internal node is the double-SHA256 of two 32-byte child hashes — a 64-byte input, indistinguishable from a transaction hash at the API level).
3. Attacker constructs a valid Merkle proof from that internal node up to the block's `merkle_root`.
4. Attacker calls `verify_transaction_inclusion` with the internal node as `tx_id`, a valid `tx_block_blockhash`, the crafted `merkle_proof`, and sufficient `confirmations`.
5. The function computes `compute_root_from_merkle_proof(tx_id, tx_index, &merkle_proof)` and finds it equals `header.block_header.merkle_root` — returning `true`.

The security-critical coinbase proof check (the step that would detect this forgery) is defined in `verify_transaction_inclusion_v2` but is **never invoked** on the deprecated call path.

---

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` to gate asset releases, cross-chain bridge withdrawals, or other value-bearing actions can be deceived into accepting a forged proof. The attacker does not need to break any cryptographic primitive — they exploit the structural ambiguity of Bitcoin's Merkle tree. The corrupted state is the verification result: the contract returns `true` for a `tx_id` that is not a real transaction leaf.

**Impact: High** — forged proof acceptance can directly enable unauthorized asset movement in any consumer of this API.

---

### Likelihood Explanation

**Likelihood: Medium** — the 64-byte transaction forgery technique is publicly documented and well understood. The attacker needs only: (a) a block in the contract's mainchain (trivially obtained), (b) knowledge of that block's transaction list (public Bitcoin data), and (c) the ability to call a public NEAR contract method. No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add an unconditional `panic!` / `env::panic_str` body so it can no longer be called. All callers must migrate to `verify_transaction_inclusion_v2`. The coinbase merkle proof validation must be a mandatory step on every externally reachable proof-verification path.

---

### Proof of Concept

```
# 1. Query any accepted mainchain block hash
near view <contract> get_block_hash_by_height '{"height": N}'
# → returns block_hash B

# 2. Fetch block B's full transaction list from a Bitcoin node
# Compute internal Merkle node I at any tree level (e.g., hash(tx0 || tx1))
# Build Merkle proof P from I to merkle_root

# 3. Call the deprecated function — no coinbase proof required
near call <contract> verify_transaction_inclusion \
  --args-borsh <borsh(ProofArgs { tx_id: I, tx_block_blockhash: B,
                                   tx_index: <index of I in subtree>,
                                   merkle_proof: P, confirmations: 1 })>
# → returns true  (forged proof accepted)

# 4. Calling verify_transaction_inclusion_v2 with the same tx_id
#    would panic at "Incorrect coinbase merkle proof" — proving the fix exists
#    but is unreachable via the deprecated entry point.
```

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
