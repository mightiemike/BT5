Audit Report

## Title
`SwapAllowlistExtension` Checks Router Identity Instead of End-User Identity, Allowing Any User to Bypass Per-Pool Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps using `allowedSwapper[msg.sender][sender]`, where `sender` is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the router becomes `msg.sender` to the pool and therefore `sender` in the extension — not the end user. A pool admin who allowlists the router to enable allowlisted users to trade through it simultaneously opens the pool to every user on the router, completely defeating the access-control invariant.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to every extension.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`, forwarding the direct caller's address verbatim as `sender` to all configured extensions. [1](#0-0) 

**Step 2 — `SwapAllowlistExtension` checks `sender`, not the end user.**

Inside `beforeSwap`, `msg.sender` is the pool (the extension's caller) and `sender` is the direct caller of `pool.swap()`. The guard is `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][directCaller]`. [2](#0-1) 

**Step 3 — The router calls `pool.swap()` directly, substituting its own address for the user's.**

`exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) all call `IMetricOmmPoolActions(params.pool).swap(...)` with the router as `msg.sender`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. [3](#0-2) 

**The dilemma this creates for the pool admin:**

| Admin action | Effect |
|---|---|
| Does **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlists the router** | Every user on the router bypasses the allowlist |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same. The `allowAllSwappers` escape hatch does not help — it only makes the problem worse by removing the check entirely. [4](#0-3) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` (e.g., a KYC-gated pool, a private institutional pool, or a pool restricted to specific market-makers) can be accessed by any unprivileged user simply by routing through `MetricOmmSimpleRouter`. Once inside, the unauthorized user can execute swaps against LP capital that was never intended to be exposed to them, cause adverse selection losses for LPs, and drain one-sided liquidity from bins the pool admin expected to be protected. This is a direct admin-boundary break: the pool admin's access-control configuration is bypassed by an unprivileged path (the public router). Severity: High — direct loss of LP assets and broken core access-control invariant. [5](#0-4) 

## Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless contract. No special privilege, token, or role is required to call it. The bypass requires only that the pool admin has allowlisted the router — a natural and expected operational step for any pool that wants allowlisted users to have a good UX. The attacker needs zero knowledge of the pool's internals; they only need the pool address and the router address. The condition is therefore highly likely to be met in any real deployment of this extension. [6](#0-5) 

## Recommendation

**Short term:** `SwapAllowlistExtension.beforeSwap()` should not rely solely on `sender` when `sender` is a known router. Require the router to embed the true end-user address in `extensionData` and have the extension verify it against a registry of trusted routers. When `sender` is a known router, extract and check the attested user address; when `sender` is not a known router, check `sender` directly as today.

**Long term:** Define a protocol-level convention for periphery contracts to attest the originating user's address in `extensionData` in a tamper-proof way (e.g., a signed payload or a transient-storage forwarding pattern), and update all identity-sensitive extensions to consume that convention instead of relying on `sender` alone. [4](#0-3) 

## Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Pool admin calls setAllowedToSwap(pool, alice, true)
   — only Alice is supposed to be able to swap.
3. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so Alice can use MetricOmmSimpleRouter.
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, ...})
5. Router calls IMetricOmmPoolActions(pool).swap(...) — msg.sender to pool = router.
6. Pool calls _beforeSwap(msg.sender=router, ...) → extension.beforeSwap(sender=router, ...)
7. Extension checks allowedSwapper[pool][router] → true → passes.
8. Bob's swap executes successfully against the "private" pool.
``` [1](#0-0) [2](#0-1) [3](#0-2)
