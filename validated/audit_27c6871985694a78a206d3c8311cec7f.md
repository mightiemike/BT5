### Title
Odd-leaf duplicate position allows phantom `tx_index` to pass Merkle proof verification — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` uses only `current_position % 2` (left/right parity) to traverse the tree. It never validates that `transaction_position` is within the actual leaf count of the block. In a Bitcoin Merkle tree with an odd number of transactions, the last leaf is duplicated. This makes the proof for the real last leaf (position N-1) and the phantom duplicate (position N) **mathematically identical**, so both pass the root-equality check. An unprivileged caller can supply `tx_index = N` for a block with N transactions and receive `true` from `verify_transaction_inclusion`, even though no transaction exists at that position.

---

### Finding Description

**Root cause in `compute_root_from_merkle_proof`** (`merkle-tools/src/lib.rs` lines 34–52):

```
current_position % 2  →  left/right
current_position /= 2 →  move up one level
```

No bound check on `transaction_position` against the actual leaf count is performed. [1](#0-0) 

**Concrete arithmetic for a 3-tx block [T0, T1, T2]**:

Bitcoin pads to [T0, T1, T2, T2]. The tree is:

```
Level 0:  T0    T1    T2    T2(dup)
Level 1:  H(T0,T1)   H(T2,T2)
Root:     H(H(T0,T1), H(T2,T2))
```

| Call | tx_id | tx_index | Step 1 (pos%2) | Step 2 (pos%2) | Result |
|------|-------|----------|----------------|----------------|--------|
| Real | T2 | 2 (even) | H(T2, **T2**) → pos=1 | H(**H(T0,T1)**, H(T2,T2)) = root | ✓ |
| Phantom | T2 | 3 (odd) | H(**T2**, T2) → pos=1 | H(**H(T0,T1)**, H(T2,T2)) = root | ✓ |

Both calls use the identical proof `[T2, H(T0,T1)]` and both return `true`.

**Entrypoint — `verify_transaction_inclusion`** (`contract/src/lib.rs` lines 288–323):

The function checks that `tx_block_blockhash` is on the mainchain and that confirmations are satisfied, then delegates directly to `compute_root_from_merkle_proof` with the caller-supplied `tx_index`. There is no guard on whether `tx_index` is within the real transaction count of the block. [2](#0-1) 

The contract stores only the 80-byte block header (which contains the Merkle root but not the transaction count), so it has no way to bound-check `tx_index` at verification time. [3](#0-2) 

**`verify_transaction_inclusion_v2` does not fix this** (`contract/src/lib.rs` lines 347–369):

The v2 function adds a coinbase proof check (to mitigate the 64-byte forgery), but it then calls the deprecated v1 function for the actual transaction proof. The phantom-position bypass applies to the inner v1 call. [4](#0-3) 

**`#[deprecated]` does not prevent calls**: The Rust attribute only emits a compiler warning; the function remains fully callable at runtime and is not gated by any access-control role. [5](#0-4) 

---

### Impact Explanation

A downstream bridge that uses `(tx_id, tx_index)` as a replay-prevention nonce (a standard design) can be double-spent:

1. Attacker observes a legitimate BTC deposit transaction T2 at position 2 in a 3-tx block. The bridge processes it and marks `(T2, 2)` as spent.
2. Attacker calls `verify_transaction_inclusion` with `tx_id=T2`, `tx_index=3`, and the same proof `[T2, H(T0,T1)]`. The contract returns `true`.
3. The bridge sees `(T2, 3)` — a new, unseen key — and processes the deposit again.

The phantom position is indistinguishable from a real one at the contract level, because the contract has no transaction-count information.

---

### Likelihood Explanation

- The function is public, unpermissioned (no `#[trusted_relayer]`, no role check), and callable by any NEAR account.
- The only prerequisite is a confirmed mainchain block with an odd transaction count — the overwhelming majority of Bitcoin blocks qualify.
- The attacker needs only the real last-leaf proof, which is publicly derivable from any Bitcoin block explorer.
- No privileged role, leaked key, or social engineering is required.

---

### Recommendation

1. **Reject phantom positions explicitly**: Require that `tx_index < (1 << merkle_proof.len())` AND that the proof length is consistent with the claimed position. More precisely, if the proof has depth `d`, the maximum valid leaf index is `2^d - 1`; any index that maps to the same internal node as another valid index via the duplicate-last-leaf rule must be rejected.

2. **Store or supply the transaction count**: Pass the number of transactions in the block as part of `ProofArgs` and require `tx_index < tx_count`. The Merkle root alone is insufficient to bound-check the index.

3. **Detect the duplicate-leaf ambiguity**: At each level where `current_position` is the last odd index (i.e., `current_position == proof_hash_count_at_level - 1` and the level has odd width), reject the proof if `current_hash == proof_hash` (i.e., the sibling equals the node itself), which is the signature of the duplicate-last-leaf padding.

### Proof of Concept

```rust
// Pseudocode — directly testable with the existing merkle-tools crate

let t0 = H256::from([0x01; 32]);
let t1 = H256::from([0x02; 32]);
let t2 = H256::from([0x03; 32]);

// Real proof for T2 at position 2 in a 3-tx block
let proof = merkle_proof_calculator(vec![t0.clone(), t1.clone(), t2.clone()], 2);
// proof == [t2, H(t0,t1)]

// Verify at real position 2 → true (expected)
let root_real = compute_root_from_merkle_proof(t2.clone(), 2, &proof);

// Verify at phantom position 3 → also true (BUG)
let root_phantom = compute_root_from_merkle_proof(t2.clone(), 3, &proof);

assert_eq!(root_real, root_phantom); // passes — same root, different claimed index
```

Both calls return the same root, which equals the block's `merkle_root`. `verify_transaction_inclusion` therefore returns `true` for `tx_index=3` in a 3-transaction block, confirming the existence of a phantom transaction position. [6](#0-5) [7](#0-6)

pub fn merkle_proof_calculator(tx_hashes: Vec<H256>, transaction_position: usize) -> Vec<H256> {
    let mut transaction_position = transaction_position;
    let mut merkle_proof = Vec::new();
    let mut current_hashes = tx_hashes;

    while current_hashes.len() > 1 {
        if current_hashes.len() % 2 == 1 {
            current_hashes.push(current_hashes[current_hashes.len() - 1].clone());
        }

        if transaction_position % 2 == 1 {
            merkle_proof.push(current_hashes[transaction_position - 1].clone());
        } else {
            merkle_proof.push(current_hashes[transaction_position + 1].clone());
        }

        let mut new_hashes = Vec::new();

        for i in (0..current_hashes.len() - 1).step_by(2) {
            new_hashes.push(compute_hash(&current_hashes[i], &current_hashes[i + 1]));
        }

        current_hashes = new_hashes;
        transaction_position /= 2;
    }

    merkle_proof
}

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
