### Title
Unvalidated User-Supplied `coinbase_tx_id` Defeats the 64-Byte Merkle Forgery Fix in `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle proof forgery vulnerability present in the deprecated `verify_transaction_inclusion`. The mitigation requires the caller to supply a coinbase Merkle proof of the same depth as the transaction proof, anchoring the expected tree depth. However, the `coinbase_tx_id` field is entirely user-controlled and is never validated against the actual coinbase transaction hash stored in the block. An unprivileged NEAR caller can supply an internal Merkle tree node as `coinbase_tx_id` with a one-level-shorter proof, pass the coinbase check, and then forge a proof that a non-existent transaction is included in the block. The mitigation is therefore ineffective.

---

### Finding Description

`verify_transaction_inclusion_v2` performs two checks before delegating to the deprecated v1 function:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — both proofs must have the same depth.
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof) == header.block_header.merkle_root` — the coinbase proof must reconstruct the stored Merkle root. [1](#0-0) 

Neither check validates that `coinbase_tx_id` is the actual coinbase transaction hash. The contract stores only the Merkle root inside `LightHeader`; it does not store individual transaction hashes. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure function that accepts any starting hash and any proof vector: [3](#0-2) 

Because the function only checks that the computed root matches the stored root, an internal node hash is indistinguishable from a leaf hash. The coinbase anchor check can therefore be satisfied with a forged internal-node "coinbase," and the subsequent transaction proof can be forged at the same (shorter) depth.

---

### Impact Explanation

Any NEAR contract or user that calls `verify_transaction_inclusion_v2` to gate a cross-chain action (e.g., releasing bridged funds, minting wrapped tokens, settling a payment) will receive `true` for a Bitcoin transaction that does not exist. The attacker does not need to mine a block or control any privileged role; all required inputs are computable from public Bitcoin blockchain data.

---

### Likelihood Explanation

The function is publicly callable with no access-control restriction beyond the `#[pause]` guard. [4](#0-3) 

All inputs needed for the attack (internal node hashes) are derivable from any Bitcoin block whose header is already accepted by the contract. The attack requires no special tooling beyond a standard Bitcoin block explorer and a NEAR transaction.

---

### Recommendation

Validate that `coinbase_tx_id` equals the actual coinbase transaction hash before using it as the depth anchor. Because the contract does not store per-block transaction lists, the recommended approach is to require the caller to supply the raw coinbase transaction bytes, hash them on-chain, and compare the result against `coinbase_tx_id` before proceeding. Alternatively, store the coinbase transaction hash alongside each accepted `ExtendedHeader` at submission time so it can be looked up and compared during proof verification.

---

### Proof of Concept

Consider a block with four transactions `[tx0 (coinbase), tx1, tx2, tx3]` and Merkle tree:

```
Level 0 (leaves): tx0,  tx1,  tx2,  tx3
Level 1:          H01 = hash(tx0,tx1),  H23 = hash(tx2,tx3)
Level 2 (root):   R   = hash(H01, H23)
```

**Step 1 — forge the coinbase proof (depth 1 instead of 2):**

```
coinbase_tx_id        = H01   ← internal node, not a real transaction
coinbase_merkle_proof = [H23]
compute_root_from_merkle_proof(H01, 0, [H23])
  = hash(H01, H23) = R  ✓  (coinbase check passes)
```

**Step 2 — forge the transaction proof at the same depth:**

```
tx_id        = H23   ← internal node, not a real transaction
tx_index     = 1
merkle_proof = [H01]
compute_root_from_merkle_proof(H23, 1, [H01])
  = hash(H01, H23) = R  ✓  (tx inclusion check passes)
```

Both `merkle_proof.len() == coinbase_merkle_proof.len() == 1` satisfies the length check. [5](#0-4) 

`verify_transaction_inclusion_v2` returns `true` for `tx_id = H23`, which is an internal Merkle tree node and not a real Bitcoin transaction. Any downstream contract relying on this result to authorise a cross-chain action will be deceived. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
