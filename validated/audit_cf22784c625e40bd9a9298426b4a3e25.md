### Title
Unbounded `NlpLockedBalanceQueue` Growth Permanently Locks User NLP Tokens — (`SpotEngine.sol`)

---

### Summary

The `NlpLockedBalanceQueue` for a subaccount grows by one entry for every `mintNlp` transaction executed at a distinct oracle timestamp. There is no cap on `balanceCount`. The `tryUnlockNlpBalance` function drains the queue with an unbounded `while` loop. If the queue accumulates enough entries, this loop exceeds the block gas limit, permanently preventing the user from calling `burnNlp` and redeeming their NLP tokens.

---

### Finding Description

In `SpotEngine.sol`, `handleNlpLockedBalance` is invoked on every NLP balance update with a positive delta. It first calls `tryUnlockNlpBalance` to drain expired entries, then conditionally appends a new entry: [1](#0-0) 

The consolidation guard at line 152–156 merges a new deposit into the last queue slot only when `getOracleTime() + NLP_LOCK_PERIOD` exactly matches the last entry's `unlockedAt`. Any deposit made at a different oracle second creates a new slot and increments `balanceCount` with no upper bound: [2](#0-1) 

The `NlpLockedBalanceQueue` struct stores entries in an unbounded `mapping(uint64 => NlpLockedBalance)` indexed by a monotonically increasing counter: [3](#0-2) 

`tryUnlockNlpBalance` drains the queue with a `while` loop that must process every expired entry in a single call: [4](#0-3) 

`NLP_LOCK_PERIOD` is 4 days: [5](#0-4) 

The critical path to locked funds runs through `burnNlp` in `Clearinghouse.sol`, which calls `spotEngine.getNlpUnlockedBalance` before allowing any redemption: [6](#0-5) 

`getNlpUnlockedBalance` calls `tryUnlockNlpBalance`: [7](#0-6) 

If the `while` loop in `tryUnlockNlpBalance` exceeds the block gas limit, the entire `burnNlp` transaction fails with an out-of-gas error. The user's NLP tokens remain locked in the protocol with no recovery path.

The attack window is the 4-day lock period. A user who makes `N` `mintNlp` transactions at distinct oracle timestamps within any 4-day window accumulates `N` queue entries. Those entries all expire simultaneously after 4 days. If the user then attempts `burnNlp` without having made any intervening `mintNlp` calls (which would have partially drained the queue), `tryUnlockNlpBalance` must iterate over all `N` entries in one transaction.

Each iteration performs multiple storage operations (SLOAD on `queue.balances[unlockedUpTo]`, SSTORE for `delete`, SSTORE for `unlockedBalanceSum`, SSTORE for `unlockedUpTo++`), costing roughly 10,000–15,000 gas per entry. With a 30M block gas limit, approximately 2,000–3,000 entries are sufficient to cause an out-of-gas revert.

---

### Impact Explanation

A user who accumulates a sufficiently large `NlpLockedBalanceQueue` cannot call `burnNlp`. The `getNlpUnlockedBalance` → `tryUnlockNlpBalance` call path runs out of gas before returning, causing the transaction to revert. The user's NLP token balance remains in the protocol with no mechanism to redeem it. This is a concrete, permanent loss of access to deposited funds — not merely a gas cost increase.

---

### Likelihood Explanation

Low-to-medium. The attacker must make approximately 2,000+ `mintNlp` transactions at distinct oracle timestamps within a 4-day window. Each `mintNlp` charges a `HEALTHCHECK_FEE` of $1: [8](#0-7) [9](#0-8) 

The cost to trigger the issue is therefore $2,000+, making a deliberate self-attack expensive. However, a legitimate power user making frequent small NLP deposits over time can reach the threshold organically. Additionally, the sequencer processes `mintNlp` transactions in batches, and if oracle time advances by one second per block on Ink (~2-second block times), each block produces a unique `unlockedAt` value, meaning one deposit per block is sufficient to grow the queue at maximum rate.

---

### Recommendation

1. **Cap `balanceCount`**: Enforce a maximum queue depth (e.g., 500 entries). Reject or consolidate new deposits when the cap is reached.
2. **Paginated unlock**: Modify `tryUnlockNlpBalance` to accept a `maxEntries` parameter and process only a bounded number of entries per call, requiring callers to drain the queue across multiple transactions before `burnNlp` is permitted.
3. **Consolidate by day/epoch**: Instead of consolidating only on exact-second matches, round `getOracleTime()` to a coarser granularity (e.g., hourly or daily buckets) before computing `unlockedAt`, drastically reducing the number of distinct queue entries.

---

### Proof of Concept

1. User calls `mintNlp` once per block for 4 days on Ink (~2-second blocks → ~172,800 blocks → ~172,800 queue entries, each at a distinct oracle second).
2. User stops minting for 4 days. All entries expire (`unlockedAt <= getOracleTime()`), but no `mintNlp` call is made to drain them.
3. User calls `burnNlp`. This triggers `getNlpUnlockedBalance` → `tryUnlockNlpBalance`, which enters the `while` loop and attempts to process all 172,800 entries.
4. The loop consumes ~172,800 × 12,000 gas ≈ 2.07 billion gas, far exceeding the block gas limit.
5. The transaction reverts with out-of-gas. The user's NLP tokens are permanently locked.

Even at a more modest rate of 500 deposits per day for 4 days (2,000 entries, cost ~$2,000), the loop consumes ~24M gas, approaching or exceeding the block gas limit and making `burnNlp` unreliable or impossible.
