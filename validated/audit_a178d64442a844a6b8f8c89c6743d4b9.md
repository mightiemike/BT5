### Title
Protobuf `ValidResourceBounds` zero-check misclassifies new V3 transactions as legacy `L1Gas` variant, producing a divergent transaction hash - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-`ValidResourceBounds` converter uses a zero-check on `l2_gas` and `l1_data_gas` to select between the `L1Gas` (pre-0.13.3, 2-element hash preimage) and `AllResources` (0.13.3+, 3-element hash preimage) variants. A new V3 transaction with `l2_gas=0, l1_data_gas=0` passes through this neutral state and is misclassified as the legacy `L1Gas` variant. Because `get_tip_resource_bounds_hash` produces structurally different hash preimages for the two variants, the hash computed by a P2P-syncing node diverges from the hash committed by the block producer, which always uses `AllResources` for new V3 transactions.

---

### Finding Description

**Step 1 — Two variants, two hash preimages.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` branches on the `ValidResourceBounds` variant:

```
L1Gas     → [tip, L1_GAS_concat, L2_GAS_concat]                          (2 resource elements)
AllResources → [tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]   (3 resource elements)
``` [1](#0-0) 

**Step 2 — The zero-check in the protobuf converter.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` selects the variant purely by checking whether the numeric values are zero:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

The same zero-check exists in the RPC `ResourceBoundsMapping` converter: [3](#0-2) 

**Step 3 — The RPC/gateway path always uses `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`). The conversion chain `RpcInvokeTransactionV3 → InternalRpcInvokeTransactionV3 → InvokeTransactionV3` always wraps the bounds in `ValidResourceBounds::AllResources(...)`: [4](#0-3) 

So the block producer always computes the hash with the 3-element preimage (including `L1_DATA_GAS`), even when `l2_gas=0` and `l1_data_gas=0`.

**Step 4 — The P2P sync path converts to `L1Gas` for the same values.**

When a syncing node receives the same transaction via protobuf, the converter at line 431 sees `l1_data_gas.is_zero() && l2_gas.is_zero()` and returns `ValidResourceBounds::L1Gas`. The subsequent call to `get_tip_resource_bounds_hash` produces a 2-element preimage, yielding a different Poseidon hash. [5](#0-4) 

**Step 5 — Gateway validation does not prevent the trigger.**

The stateless validator only requires `max_possible_fee(Tip::ZERO) != 0`. A transaction with `l1_gas = {max_amount: N, max_price_per_unit: P}` and `l2_gas = l1_data_gas = 0` satisfies this check (fee = N×P > 0) and is accepted: [6](#0-5) 

---

### Impact Explanation

A V3 invoke transaction with `l2_gas=0, l1_data_gas=0` and non-zero `l1_gas` is valid and passes gateway admission. The block producer commits it with hash H₁ (3-element `AllResources` preimage). A syncing node that receives the block via P2P converts the same resource-bounds protobuf to `ValidResourceBounds::L1Gas` and computes hash H₂ (2-element preimage). H₁ ≠ H₂. The syncing node either rejects the block (wrong committed state) or stores an incorrect transaction hash, causing divergent state between nodes. This matches: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input** and **Transaction conversion or signature/hash logic binds the wrong hash or executable payload**.

---

### Likelihood Explanation

Any unprivileged user can submit a standard V3 invoke transaction with `l2_gas=0, l1_data_gas=0` and a non-zero `l1_gas` bound. This is a structurally valid transaction that passes all gateway checks. No special privileges or malformed bytes are required. The divergence is triggered automatically whenever such a transaction is included in a block and synced via P2P.

---

### Recommendation

The variant selection must not rely solely on whether the numeric values are zero, because `AllResources { l2_gas: 0, l1_data_gas: 0 }` and `L1Gas` are semantically distinct (different hash preimage lengths) despite having identical field values. Options:

1. **Explicit version marker in protobuf**: Add a boolean or enum field to `ResourceBounds` that explicitly encodes whether the transaction uses the 2-element or 3-element hash format, independent of the numeric values.
2. **Always use `AllResources` for new V3 transactions**: Remove the `L1Gas` variant from the conversion path for transactions received via P2P that are known to be post-0.13.3 (e.g., gated by block number or transaction version).
3. **Canonical hash computation**: Make `get_tip_resource_bounds_hash` version-aware (e.g., pass the Starknet version or a flag) so that the hash preimage length is determined by the protocol version, not the runtime variant.

---

### Proof of Concept

```
1. Attacker submits via RPC:
   InvokeV3 {
     l1_gas: { max_amount: 1000, max_price_per_unit: 1 },
     l2_gas: { max_amount: 0, max_price_per_unit: 0 },
     l1_data_gas: { max_amount: 0, max_price_per_unit: 0 },
     ...
   }

2. Gateway converts to AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }.
   Hash H₁ = poseidon([INVOKE, 0x3, sender,
                        poseidon([tip, L1_GAS_concat(X), L2_GAS_concat(0), L1_DATA_GAS_concat(0)]),
                        ...])
   (3-element resource hash)

3. Transaction included in block with hash H₁.

4. Syncing node receives block via P2P. Protobuf converter sees:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas(X)

5. Syncing node computes:
   Hash H₂ = poseidon([INVOKE, 0x3, sender,
                        poseidon([tip, L1_GAS_concat(X), L2_GAS_concat(0)]),
                        ...])
   (2-element resource hash)

6. H₁ ≠ H₂ → hash validation fails → block rejected or wrong hash stored.
```
