### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Allowing Any User to Bypass Per-User Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool. The pool always passes `msg.sender` of the `swap()` call as `sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router to enable router-based swaps, every user—including those not on the allowlist—can bypass the per-user restriction by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap(...)` on the user's behalf. From the pool's perspective `msg.sender` is the router, so `sender` passed to the extension is the router address—not the end user. The extension therefore checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`.

The pool's `swap` interface confirms no user-identity field is threaded through; only `recipient` (output destination) and `msg.sender` (input payer / callback target) exist: [3](#0-2) 

The extension interface shows `sender` and `recipient` are distinct fields, but the production `SwapAllowlistExtension` only uses `sender` (the intermediary) and ignores `recipient` (the beneficiary): [4](#0-3) 

---

### Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` intends to restrict swaps to a curated set of addresses. To also support router-based swaps for those allowed users, the admin must allowlist the router contract. Once the router is allowlisted, **any** address can call `MetricOmmSimpleRouter` and reach the pool, because the extension sees only the router address and approves it. The per-user allowlist is completely neutralized for the router path. Unauthorized users can execute swaps against the curated pool, receiving output tokens and paying input tokens—a full economic interaction the allowlist was meant to prevent.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the documented, supported periphery entry point for swaps. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router to serve its permitted users hits this condition automatically. Both steps are normal, expected operational choices, making the precondition easy to satisfy without any privileged or malicious setup.

---

### Recommendation

The extension must authenticate the **end user**, not the intermediary. Two options:

1. **Pass the real user through `extensionData`**: Require callers (including the router) to encode the actual user address in `extensionData`, and have the extension decode and check it. The router must be trusted to supply the correct address.
2. **Check `recipient` instead of (or in addition to) `sender`**: For swap allowlists the economically relevant actor is the address receiving output tokens (`recipient`). Gating on `recipient` is router-agnostic and cannot be spoofed by an intermediary.

Option 2 is simpler and does not require router cooperation:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

---

### Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` as `extension1`, `beforeSwap` order set to extension 1.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, router, true)` — necessary to let any allowed user swap via the router.
3. Attacker (address not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(recipient=attacker, ...)` — pool's `msg.sender` is the router.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. Swap executes; attacker receives output tokens.
8. The per-user allowlist provided zero protection. [5](#0-4) [1](#0-0)
