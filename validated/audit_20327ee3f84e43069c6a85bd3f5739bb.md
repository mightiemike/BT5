### Title
Silent Failure of Slow Mode `WithdrawCollateral` Permanently Drops User Withdrawal With No Recovery — (`core/contracts/Endpoint.sol`)

---

### Summary

The `_executeSlowModeTransaction` function in `Endpoint.sol` deletes a slow mode transaction from the queue **before** attempting to execute it, then wraps execution in a `try/catch` that silently swallows all non-OOG reverts. A comment in the catch block reads `// try return funds now removed`, confirming a prior refund path was deliberately removed. If a slow mode `WithdrawCollateral` transaction reverts (e.g., health check failure after the 3-day delay), the withdrawal is permanently dropped, the user's `SLOW_MODE_FEE` ($1) is non-refundable, and — critically in a sequencer-censorship scenario — the user has no remaining on-chain escape path for their collateral.

---

### Finding Description

The slow mode system is Nado's censorship-resistance mechanism: users who cannot get the sequencer to process their transactions can submit them on-chain and execute them after a 3-day delay. The execution path is:

**`Endpoint._executeSlowModeTransaction`** (lines 185–228):

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // ← consumed before execution

// ...
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed          // ← no recovery
}
``` [1](#0-0) 

The transaction is deleted from `slowModeTxs` at line 194 before the `try` block. If `processSlowModeTransaction` reverts for any reason other than OOG, the catch block does nothing — the transaction is gone.

**`EndpointTx.processSlowModeTransactionImpl`** handles `WithdrawCollateral` at lines 217–229:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
}
``` [2](#0-1) 

`clearinghouse.withdrawCollateral` performs a health check after the balance reduction. If the user's subaccount health falls below the initial threshold (due to price movements over the 3-day delay, or because the requested `amount` is too large), it reverts with `ERR_SUBACCT_HEALTH`. This revert propagates up to the `try/catch`, is silently swallowed, and the withdrawal is permanently dropped.

**Submission path** (`EndpointTx.submitSlowModeTransactionImpl`, lines 369–371) charges the fee at submission time:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;
``` [3](#0-2) 

`SLOW_MODE_FEE` is $1 (1,000,000 quote units). There is no refund path for a failed execution. [4](#0-3) 

---

### Impact Explanation

Two distinct impacts:

1. **Fee loss**: Every failed slow mode `WithdrawCollateral` permanently burns the user's $1 fee. The user must pay another $1 and wait another 3 days to retry.

2. **Collateral lock in censorship scenario**: The slow mode path is the *only* on-chain escape hatch when the sequencer is censoring a user. If the withdrawal fails silently (e.g., health deteriorated over the 3-day window), the user's collateral remains locked in the clearinghouse with no remaining on-chain mechanism to recover it. The sequencer can continue to censor re-submissions indefinitely, and each attempt costs $1 and 3 days.

This directly mirrors the original report's impact: a user-initiated cross-system operation (bridge deposit / slow mode withdrawal) fails silently due to an insufficient or invalid parameter, and the user's funds are unrecoverable through the same path.

---

### Likelihood Explanation

**Medium.** The 3-day `SLOW_MODE_TX_DELAY` is a mandatory window during which market conditions can change:

- A user with a leveraged position submits a slow mode withdrawal to reduce exposure.
- Over 3 days, the price moves adversely, reducing their health.
- At execution, `clearinghouse.withdrawCollateral` reverts on the health check.
- The withdrawal is silently dropped. [5](#0-4) 

This is not a contrived edge case — it is the natural consequence of a 3-day delay on a leveraged trading protocol. The censorship scenario (where slow mode is most needed) is also the scenario where a hostile sequencer has the most incentive to delay execution until the user's health deteriorates.

---

### Recommendation

1. **Re-queue on failure**: If `processSlowModeTransaction` reverts for a non-OOG reason, re-insert the transaction at the front of the queue (or at a new index) rather than permanently dropping it.

2. **Refund the fee on failure**: Restore a refund path in the catch block — credit `SLOW_MODE_FEE` back to the user's quote balance when the transaction fails for a non-OOG reason.

3. **Snapshot health at submission**: Record the user's health at submission time and allow the withdrawal to proceed if health was valid at submission, regardless of the 3-day drift.

---

### Proof of Concept

1. User has a subaccount with 100 USDC collateral and an open perp position near initial health.
2. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` transaction for 50 USDC. $1 fee is deducted. The tx is queued.
3. Over the next 3 days, the perp position moves against the user, reducing health below the initial threshold.
4. After 3 days, anyone calls `executeSlowModeTransaction()`.
5. `_executeSlowModeTransaction` deletes the tx from `slowModeTxs` at line 194.
6. `processSlowModeTransaction` → `clearinghouse.withdrawCollateral` reverts with `ERR_SUBACCT_HEALTH`.
7. The catch block at line 207 fires; gas check passes (not OOG); execution falls through to `// try return funds now removed`.
8. The slow mode tx is gone. The $1 fee is gone. The user's collateral remains locked. If the sequencer is censoring this user, there is no remaining on-chain withdrawal path. [1](#0-0) [6](#0-5)
