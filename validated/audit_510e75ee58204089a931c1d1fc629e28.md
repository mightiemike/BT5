### Title
No Slippage Protection in `mintNlp` / `burnNlp` Allows Users to Receive Significantly Less Value Than Expected — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`mintNlp` and `burnNlp` in `Clearinghouse.sol` execute at an `oraclePriceX18` that is supplied by the sequencer at execution time and is **not** committed to in the user's EIP-712 signature. Neither function accepts a user-specified minimum output bound. A user who signs a `BurnNlp` or `MintNlp` transaction has no on-chain protection against receiving far less value than they expected when the transaction was signed.

---

### Finding Description

The `MintNlp` and `BurnNlp` structs that users sign contain only `{sender, quoteAmount/nlpAmount, nonce}`: [1](#0-0) 

The `oraclePriceX18` field lives in the outer `SignedMintNlp` / `SignedBurnNlp` wrapper and is **excluded** from the EIP-712 digest: [2](#0-1) 

At execution time, `mintNlp` computes `nlpAmount = quoteAmount / oraclePriceX18` and `burnNlp` computes `quoteAmount = nlpAmount * oraclePriceX18 - burnFee`, both using the sequencer-supplied price with no user-specified bound: [3](#0-2) [4](#0-3) 

The protocol already defines `ERR_SLIPPAGE_TOO_HIGH` but never applies it in either of these paths: [5](#0-4) 

---

### Impact Explanation

**`burnNlp`**: A user commits to burning a fixed `nlpAmount`. If `oraclePriceX18` is lower at execution than at signing time, the user receives fewer quote tokens than expected — equivalent to selling NLP at a worse price with no floor. The `burnFee` compounds the loss.

**`mintNlp`**: A user commits to spending a fixed `quoteAmount`. If `oraclePriceX18` is higher at execution, the user receives fewer NLP tokens than expected — equivalent to buying NLP at a worse price with no ceiling.

In both cases the user's collateral balance is updated with the unfavorable amount and there is no revert path. The corrupted state is the user's `QUOTE_PRODUCT_ID` balance (burnNlp) or `NLP_PRODUCT_ID` balance (mintNlp) in the SpotEngine. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

This does not require a malicious sequencer. Normal NLP price volatility between the moment a user signs a transaction and the moment the sequencer includes it in a batch is sufficient. In periods of high volatility the gap between expected and actual output can be substantial. Any user calling `mintNlp` or `burnNlp` is exposed on every transaction.

---

### Recommendation

Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp`, include them in the EIP-712 digest, and add a revert guard in `Clearinghouse.mintNlp` / `Clearinghouse.burnNlp`:

```solidity
// burnNlp
require(quoteAmount >= int128(txn.minQuoteAmount), ERR_SLIPPAGE_TOO_HIGH);

// mintNlp
require(nlpAmount >= int128(txn.minNlpAmount), ERR_SLIPPAGE_TOO_HIGH);
```

The `ERR_SLIPPAGE_TOO_HIGH` constant is already defined and available for this purpose.

---

### Proof of Concept

1. NLP oracle price is `$1.00`. User signs `BurnNlp { nlpAmount: 10_000e18, nonce: N }` expecting to receive `~9990 USDC` (after 0.1% burn fee).
2. Before the sequencer processes the transaction, the NLP oracle price drops to `$0.80`.
3. Sequencer submits the batch with `oraclePriceX18 = 0.80e18`.
4. `Clearinghouse.burnNlp` executes: `quoteAmount = 10_000e18 * 0.80e18 / 1e18 - burnFee = ~7992 USDC`.
5. User receives `~7992 USDC` instead of `~9990 USDC` — a `~20%` loss — with no revert and no recourse.
6. The same scenario applies to `mintNlp` in reverse: a rising oracle price means the user receives fewer NLP tokens for the same quote spend. [8](#0-7)

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
