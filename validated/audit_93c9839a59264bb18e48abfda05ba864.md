### Title
Fork Difficulty Calculated from Mainchain Ancestors Instead of Fork Ancestors, Enabling PoW Bypass on Fork Blocks — (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating a fork block's difficulty, both the Dogecoin and Bitcoin modules call `blocks_getter.get_header_by_height(height_first)` to retrieve the boundary block used in the retarget calculation. This function reads exclusively from the `mainchain_height_to_header` index — it returns the mainchain block at that height, not the fork's actual ancestor. For any fork block, the difficulty is therefore computed from the wrong chain's timestamps, breaking the invariant that each block's `bits` must match the difficulty derived from its own chain history. An unprivileged relayer can exploit this to submit fork blocks whose PoW satisfies a mainchain-derived (potentially lower) difficulty target rather than the difficulty the fork's own history would require.

---

### Finding Description

The `BlocksGetter` trait exposes two methods:

- `get_prev_header` — follows `prev_block_hash` through `headers_pool`, correctly traversing the actual chain (main or fork).
- `get_header_by_height` — reads from `mainchain_height_to_header`, always returning a mainchain block regardless of which chain is being validated. [1](#0-0) 

In the Dogecoin module, `get_next_work_required` is called for **every** fork block (Digishield adjusts per-block after height 145,000). It computes `height_first = prev_block_header.block_height - blocks_to_go_back` and then fetches the boundary block's timestamp via `get_header_by_height`: [2](#0-1) 

The code itself acknowledges the problem with an explicit TODO:

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor` [3](#0-2) 

The same pattern appears in the Bitcoin module at every 2016-block retarget boundary: [4](#0-3) 

When a fork diverges from the mainchain at height H, fork blocks at heights H+1, H+2, … have their difficulty computed using mainchain blocks at heights H, H+1, … respectively. Those mainchain blocks carry different timestamps than the fork's actual ancestors. The resulting `actual_time_taken` (or `modulated_timespan` for Dogecoin) is therefore wrong, producing an incorrect `new_target` and `expected_bits`. [5](#0-4) 

The difficulty check in `check_pow` then compares the block's `bits` field against this incorrectly computed `expected_bits`: [6](#0-5) 

---

### Impact Explanation

An attacker who controls a fork can choose fork ancestor timestamps (within the MTP lower bound and the 2-hour future upper bound) such that the fork's true retarget would demand a **higher** difficulty than what the mainchain timestamps at those heights produce. The contract accepts the block if `block.bits == expected_bits_from_mainchain`, so the attacker mines at the lower mainchain-derived difficulty. The accepted fork block carries a `chain_work` value computed from its (artificially easy) `bits`: [7](#0-6) 

If the accumulated fork `chain_work` exceeds the mainchain tip's `chain_work`, an automatic reorg is triggered: [8](#0-7) 

The result is that the contract's canonical chain pointer (`mainchain_tip_blockhash`) and the `mainchain_height_to_header` index are updated to a fork whose blocks did not satisfy the difficulty their own chain history requires. Downstream consumers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will then verify transactions against a fraudulently promoted chain. [9](#0-8) 

---

### Likelihood Explanation

The entry point is `submit_blocks`, callable by any account that satisfies the trusted-relayer check (or bypassed by `Role::UnrestrictedSubmitBlocks`). A relayer submitting a competing fork — a normal operational scenario — triggers this path automatically. No privileged key or social engineering is required. For Dogecoin the bug fires on every fork block (Digishield is per-block); for Bitcoin it fires at every 2016-block retarget boundary that a fork crosses. The divergence between mainchain and fork timestamps grows naturally over time, making the exploitable window wide.

---

### Recommendation

Replace `get_header_by_height` calls inside `get_next_work_required` with ancestor traversal via `get_prev_header`, walking back the required number of steps from `prev_block_header` along the actual chain being validated. This mirrors how `get_median_time_past` already correctly traverses the chain: [10](#0-9) 

Specifically, replace the `get_header_by_height(height_first)` call in both `dogecoin.rs` and `bitcoin.rs` with a loop that calls `get_prev_header` `blocks_to_go_back` times starting from `prev_block_header`. This ensures the boundary block's timestamp is always taken from the chain being validated, not from the mainchain index.

---

### Proof of Concept

1. Contract is initialized with a Dogecoin mainchain. Mainchain blocks at heights 145,001–145,010 have timestamps spaced 60 s apart (normal spacing → normal difficulty).
2. Attacker submits a fork block at height 145,001 whose `prev_block_hash` points to the mainchain block at 145,000. The fork block's timestamp is set to `mainchain[145000].time + 600` (10 minutes later).
3. `get_next_work_required` for the fork block at 145,002 calls `get_header_by_height(145,000)`, returning the **mainchain** block at 145,000 with its 60-second spacing. The computed `modulated_timespan` reflects mainchain timing, not the fork's 600-second gap.
4. The resulting `expected_bits` is lower (easier target) than what the fork's own 600-second gap would require.
5. The attacker mines the fork block to satisfy only the easier mainchain-derived target and submits it. `check_pow` passes because `block.bits == expected_bits_from_mainchain`.
6. Repeated over enough blocks, the fork accumulates `chain_work` sufficient to trigger `reorg_chain`, promoting the fraudulent fork to the canonical mainchain. [11](#0-10) [12](#0-11)

    block_header: ExtendedHeader,

) -> u32 {
    use btc_types::network::MEDIAN_TIME_SPAN;

    let mut median_time = [0u32; MEDIAN_TIME_SPAN];
    let mut current_header = block_header;

    for slot in &mut median_time {
        *slot = current_header.block_header.time;
        current_header = prev_block_getter.get_prev_header(&current_header.block_header);
    }

    median_time.sort_unstable();
    median_time[median_time.len() / 2]
}

fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
}

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
}
