Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates `owner` instead of `sender`, fully bypassing the deposit guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address (the token-payer). Its `beforeAddLiquidity` hook discards the `sender` argument and instead validates `owner` (the position recipient). Any non-allowlisted actor can bypass the guard by specifying any allowlisted address as `owner`, causing their own tokens to enter the pool while the allowlisted address receives the position shares — completely defeating the pool admin's configured access control.

## Finding Description
`MetricOmmPool.addLiquidity` invokes `_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData)` at line 191. `ExtensionCalling._beforeAddLiquidity` faithfully encodes both arguments and dispatches to the extension. However, `DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed (discarding `sender`) and checks only `allowedDepositor[msg.sender][owner]`. Because `msg.sender` in the extension is the pool (the caller of the hook), the check becomes `allowedDepositor[pool][owner]`. An attacker simply supplies any allowlisted address as `owner`; the guard passes. The pool then calls `IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)` where `msg.sender` is the original non-allowlisted caller, pulling that caller's tokens into the pool. The `allowedDepositor` mapping is `public`, so allowlisted addresses are trivially discoverable on-chain.

## Impact Explanation
This is an admin-boundary break: the pool admin's sole on-chain mechanism to restrict who may inject liquidity is completely bypassed by any unprivileged actor. Non-allowlisted actors' tokens enter the pool, violating the invariant the guard is meant to enforce. The pool admin loses the ability to control liquidity composition, which may have regulatory, risk, or protocol-design implications depending on deployment context.

## Likelihood Explanation
The bypass requires no special privileges. Any EOA or contract that implements `IMetricOmmModifyLiquidityCallback` can call `pool.addLiquidity` directly with an allowlisted address as `owner`. The only precondition is knowing one allowlisted address, which is readable from the public `allowedDepositor` mapping. The attack is repeatable and requires no elevated access.

## Recommendation
Replace the unnamed first parameter with `sender` and check it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the check with the documented intent ("gate by depositor address") and with the actual token-payer identity passed by the pool.

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` and calls `setAllowedToDeposit(pool, alice, true)`.
2. `bob` (not allowlisted) deploys a contract implementing `IMetricOmmModifyLiquidityCallback` that transfers tokens in the callback.
3. `bob` calls `pool.addLiquidity(alice /*owner*/, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
5. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. Pool invokes `bob.metricOmmModifyLiquidityCallback(amount0, amount1, ...)` — `bob`'s tokens are transferred to the pool.
7. `alice` receives the position shares; `bob`'s tokens are inside the pool.
8. The deposit allowlist has been bypassed: a non-allowlisted actor's funds entered the pool.
