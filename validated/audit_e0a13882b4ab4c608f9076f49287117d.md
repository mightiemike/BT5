Let me analyze the external report and search for nearcore analogs. The core issue is about premature state creation using a precomputed/predictable identifier that causes legitimate operations to be misdirected or blocked.

### Title
Wrong `receiver_id` Domain Used in `validate_delegate_action` Blocks All `DeterministicStateInit` Inside `DelegateAction` — (File: `runtime/runtime/src/action_validation.rs`)

---

### Summary

`validate_delegate_action` passes the **outer transaction's `receiver_id`** — not the `DelegateAction`'s own `receiver_id` — as the `inner_receiver` when validating inner actions. Because `validate_deterministic_state_init` enforces that `keccak256(borsh(state_init))[12:32]` must equal the receiver, every legitimate `DeterministicStateInit` wrapped in a `DelegateAction` (meta-transaction) is unconditionally rejected with `InvalidDeterministicStateInitReceiver`. Simultaneously, a crafted transaction whose outer receiver matches the state-init hash but whose delegate receiver targets a different account passes initial tx validation, leaking a domain-confusion window before receipt validation closes it.

---

### Finding Description

`validate_delegate_action` in `runtime/runtime/src/action_validation.rs` selects the receiver to pass down to inner-action validation:

```rust
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // outer tx receiver — wrong domain
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
``` [1](#0-0) 

For protocol versions below 85, `inner_receiver` is `receiver` — the outer transaction's `receiver_id`, which is the **relayer's target** (the meta-tx sender's account), not the deterministic account being initialized.

`validate_deterministic_state_init` then enforces the hash-binding invariant:

```rust
let derived_id = derive_near_deterministic_account_id(&action.state_init);
if derived_id != *receiver_id {
    return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
}
``` [2](#0-1) 

The deterministic account ID is `"0s" + keccak256(borsh(DeterministicAccountStateInit))[12:32].hex()`: [3](#0-2) 

**Legitimate path (always fails before fix):**

| Field | Value |
|---|---|
| outer tx `receiver_id` | `det_account_b` (meta-tx sender, relayer target) |
| `delegate_action.receiver_id` | `det_account_a` (the deterministic account to create) |
| inner action | `DeterministicStateInit { state_init_a }` |
| `derived_id` | `det_account_a` ✓ |
| `inner_receiver` (buggy) | `det_account_b` ✗ |
| result | `InvalidDeterministicStateInitReceiver` — tx rejected |

**Exploit path (passes initial validation, fails at receipt):**

| Field | Value |
|---|---|
| outer tx `receiver_id` | `det_account_b` |
| `delegate_action.receiver_id` | `det_account_a` (wrong target) |
| inner action | `DeterministicStateInit { state_init_b }` |
| `derived_id` | `det_account_b` ✓ |
| `inner_receiver` (buggy) | `det_account_b` ✓ — passes tx validation |
| receipt validation | `receipt.receiver = det_account_a ≠ det_account_b` → `InvalidDeterministicStateInitReceiver` |

The exploit path is caught at receipt validation, but the initial tx-validation gate is bypassed, creating a domain-confusion window. The comment in the code acknowledges both effects:

> *"This makes it impossible to initialize deterministic accounts from meta transactions. The bug cannot be abused, if someone crafts a state init that passes validation here, it will fail when it is checked as incoming receipt."* [4](#0-3) 

The fix is gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit` at protocol version 85: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Severity: High (within scope)**

Any node running protocol version < 85 is affected. The impact is twofold:

1. **Complete DoS on `DeterministicStateInit` via meta-transactions.** The entire `DelegateAction` + `DeterministicStateInit` feature combination is non-functional. Any relayer submitting a legitimate meta-tx to create a deterministic account receives `InvalidDeterministicStateInitReceiver` at tx validation, wasting gas and making the feature unusable through the meta-tx path.

2. **Tx-validation domain confusion.** A crafted transaction whose outer `receiver_id` matches `keccak256(borsh(state_init_b))` but whose `delegate_action.receiver_id` targets a different account passes the initial tx-validation gate. The receipt-validation layer closes the window, but the invariant that "a tx carrying a `DeterministicStateInit` must be rejected at tx validation if the receiver does not match the hash" is violated for the duration of the pre-fix protocol epoch.

---

### Likelihood Explanation

**Likelihood: High**

- Any unprivileged user can submit a `DelegateAction` wrapping a `DeterministicStateInit`. No special role is required.
- The broken path is triggered by every legitimate attempt to use `DeterministicStateInit` inside a meta-transaction — a standard relayer workflow.
- The exploit path requires only knowledge of the `derive_near_deterministic_account_id` formula (public, documented in NEP-616) and a pre-existing deterministic account to act as the meta-tx sender.

---

### Recommendation

Replace `receiver` with `delegate_action.receiver_id()` as the `inner_receiver` in `validate_delegate_action` unconditionally (or behind the protocol feature gate as already done). The fix is already present in the codebase behind `ProtocolFeature::FixDelegatedDeterministicStateInit`:

```rust
// Correct: use the delegate action's own receiver, not the outer tx receiver
let inner_receiver = delegate_action.receiver_id();
``` [1](#0-0)

### Proof of Concept

The integration test `try_meta_tx_deterministic_receiver_exploit` in `test-loop-tests/src/tests/deterministic_account_id.rs` demonstrates both effects:

- **Pre-fix (protocol version 84):** The exploit tx passes `try_execute_tx` (initial tx validation) and returns `Ok(outcome)` where `outcome.status` is `Failure(NewReceiptValidationError(ActionsValidation(InvalidDeterministicStateInitReceiver)))` — tx-validation gate bypassed, receipt gate catches it.
- **Post-fix (protocol version 85):** The exploit tx is rejected immediately at `try_execute_tx` with `Err(InvalidTxError::ActionsValidation(InvalidDeterministicStateInitReceiver))`. [7](#0-6) 

The exploit construction:

```rust
// outer tx receiver = det_account_b (matches state_init_b hash)
// delegate action receiver = det_account_a (wrong target)
// inner action = DeterministicStateInit { state_init: state_init_b }
let delegate_action = DelegateAction {
    sender_id: det_account_b.clone(),
    receiver_id: det_account_a,          // ← wrong target
    actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
    ...
};
let tx = SignedTransaction::from_actions(
    ..., relayer, det_account_b,          // ← outer receiver matches state_init_b
    &relayer_signer,
    vec![Action::Delegate(Box::new(signed_delegate_action))],
    ...
);
``` [8](#0-7)

pub fn derive_near_deterministic_account_id(
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
