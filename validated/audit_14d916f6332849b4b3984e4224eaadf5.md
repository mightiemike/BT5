## Verdict

### Title
SwapAllowlistExtension Checks Router Address Instead of Original Caller, Allowing Allowlist Bypass via Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension::beforeSwap` receives `sender = router_address` (the pool's immediate `msg.sender`) when a swap is routed through `MetricOmmSimpleRouter`. The hook checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][original_user]`. The original user's identity is permanently lost at the pool boundary.

### Finding Description
When a user calls any `exact*` function on `MetricOmmSimpleRouter`, the router calls `pool.swap(...)` directly. The pool's `msg.sender` is the router. `MetricOmmPool::swap` calls `_beforeSwap(msg.sender, ...)` passing the router address as `sender`. `ExtensionCalling::_beforeSwap` encodes that router address and calls the extension. `SwapAllowlistExtension::beforeSwap` then evaluates `allowedSwapper[pool][router_address]`.

This creates a binary failure:

1. **If the router is NOT allowlisted**: All allowlisted users are silently blocked from using the router. They must call the pool directly, defeating the purpose of the router.
2. **If the router IS allowlisted** (the only way to enable router-mediated swaps): Every user on the network can bypass the allowlist by routing through the router, since the hook sees the allowlisted router address as the swapper.

There is no mechanism in the hook, the pool, or the router to propagate the original caller's identity through the pool boundary.

### Impact Explanation
The `SwapAllowlistExtension` is designed to restrict swaps to a specific set of addresses per pool (e.g., institutional traders, whitelisted market makers). This access control is completely nullified for router-mediated swaps. Any unprivileged user can swap in a restricted pool by calling `MetricOmmSimpleRouter::exactInputSingle` (or any `exact*` variant), provided the router is allowlisted. This breaks the core functionality the extension was designed to enforce.

### Likelihood Explanation
The router is the standard, documented entrypoint for swaps. Pool admins who want their allowlisted users to use the router must allowlist the router, which opens the pool to all users. This is a near-certain misconfiguration path with no in-protocol warning or mitigation.

### Recommendation
The extension must verify the original initiator, not the immediate caller. Options:
- Pass `tx.origin` as an additional parameter (fragile, breaks contract-to-contract flows).
- Require the router to forward the original caller's address in `extensionData`, and have the extension decode and verify it — with the pool or factory enforcing that only trusted routers can set this field.
- Alternatively, document that pools using `SwapAllowlistExtension` must allowlist individual users and must NOT allowlist the router, and that router-mediated swaps are incompatible with this extension.

### Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router usage.
3. Unprivileged attacker (not in the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle` targeting the restricted pool.
4. The pool calls `_beforeSwap(router, ...)`. The extension checks `allowedSwapper[pool][router] == true`. The check passes.
5. The attacker's swap executes successfully despite not being allowlisted.
