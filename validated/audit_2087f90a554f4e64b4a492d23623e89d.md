### Title
Unprivileged Caller Can Process Arbitrary Sequencer Transactions Including Unauthorized Liquidations — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`submitTransactionsCheckedWithGasLimit` in `Endpoint.sol` is missing the `require(msg.sender == sequencer)` guard and the verifier signature check that its sibling function `submitTransactionsChecked` enforces. Any unprivileged address can call it to inject and execute arbitrary sequencer-pipeline transactions. The most severe reachable path is submitting a `LiquidateSubaccount` transaction with `sender = N_ACCOUNT`, which the transaction processor unconditionally skips signature validation for, enabling unauthorized liquidation of any subaccount.

---

### Finding Description

`submitTransactionsChecked` enforces two sequencer-only guards before processing transactions:

```solidity
// Endpoint.sol lines 278-287
require(msg.sender == sequencer);
// ...
verifier.requireValidSignature(digest, e, s, signerBitmask);
```

`submitTransactionsCheckedWithGasLimit` (lines 296–312) calls the same `processTransaction` pipeline but omits both checks entirely:

```solidity
function submitTransactionsCheckedWithGasLimit(
    uint64 idx,
    bytes[] calldata transactions,
    uint256 gasLimit
) external {
    uint256 initialGas = gasleft();
    validateSubmissionIdx(idx);          // only checks idx == nSubmissions
    for (uint256 i = 0; i < transactions.length; i++) {
        bytes calldata transaction = transactions[i];
        processTransaction(transaction); // no sequencer or verifier check
        ...
    }
    verifier.revertGasInfo(...);         // gas measurement only, not auth
}
```

Inside `processTransactionImpl`, the `LiquidateSubaccount` branch skips all signature validation when `sender == N_ACCOUNT`:

```solidity
// EndpointTx.sol lines 396-411
if (signedTx.tx.sender != N_ACCOUNT) {
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true
    );
    ...
}
clearinghouse.liquidateSubaccount(signedTx.tx);
```

`N_ACCOUNT` is a well-known constant (`bytes32(2)`). An attacker crafts a `LiquidateSubaccount` payload with `sender = N_ACCOUNT`, targeting any victim subaccount, and calls `submitTransactionsCheckedWithGasLimit`. No private key, no signature, no sequencer role is required.

---

### Impact Explanation

The attacker can liquidate any subaccount that passes the `isUnderMaintenance` check (or force-finalize one that is already under maintenance). In `_handleLiquidationPayment`, the `sender` field (set to `N_ACCOUNT`) receives the liquidated assets at a discount price, and the liquidation fees flow to the insurance fund. Because `N_ACCOUNT` is a protocol-internal account (not attacker-controlled), the attacker does not directly pocket the assets — but the victim's collateral is forcibly transferred out of their subaccount at a penalty price, causing direct, irreversible asset loss to the victim. Additionally, the attacker can process `ExecuteSlowMode` with `fromSequencer = true` (bypassing the 3-day delay), inject arbitrary `SpotTick`/`PerpTick` timestamps to manipulate interest/funding rate accrual, and increment `nSubmissions` to desynchronize the sequencer's submission index.

---

### Likelihood Explanation

The entry point is a public `external` function with no role check. The only precondition is knowing the current `nSubmissions` value, which is readable on-chain. Any EOA or contract can call this at any time. The `N_ACCOUNT` constant is hardcoded in `Constants.sol` and publicly visible. Exploitation requires only constructing a valid ABI-encoded `LiquidateSubaccount` struct — no privileged access, no leaked keys, no governance capture.

---

### Recommendation

Add the sequencer-only guard and verifier signature check to `submitTransactionsCheckedWithGasLimit`, mirroring `submitTransactionsChecked`:

```solidity
function submitTransactionsCheckedWithGasLimit(
    uint64 idx,
    bytes[] calldata transactions,
    uint256 gasLimit
) external {
    validateSubmissionIdx(idx);
    require(msg.sender == sequencer);  // ADD THIS
    // optionally add verifier.requireValidSignature(...) if batch integrity is needed
    ...
}
```

---

### Proof of Concept

1. Read `nSubmissions` from `Endpoint` storage (public or via `getNonce`-adjacent slot).
2. Construct a `LiquidateSubaccount` struct:
   - `sender = N_ACCOUNT` (= `bytes32(2)`)
   - `liquidatee = <victim subaccount>`
   - `productId`, `amount`, `isEncodedSpread` set to valid liquidation parameters for the victim
   - `signature` = any bytes (ignored because `sender == N_ACCOUNT`)
3. ABI-encode as `abi.encodePacked(uint8(TransactionType.LiquidateSubaccount), abi.encode(signedTx))`.
4. Call `endpoint.submitTransactionsCheckedWithGasLimit(nSubmissions, [encodedTx], type(uint256).max)`.
5. `processTransaction` → `processTransactionImpl` → signature check skipped → `clearinghouse.liquidateSubaccount(txn)` executes, transferring the victim's collateral at a liquidation discount. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        validateSubmissionIdx(idx);
