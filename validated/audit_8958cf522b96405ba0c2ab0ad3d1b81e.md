### Title
Per-Segment `TruncateInt()` in `processEventsAndClaimBonus` Causes Accumulated Precision Loss in Bonus Reward Payouts — (`x/tieredrewards/keeper/bonus_rewards.go` + `claim_rewards.go`)

---

### Summary

`computeSegmentBonus` truncates its result to `math.Int` before returning. `processEventsAndClaimBonus` calls it once per validator event and sums the already-truncated integers. Each truncation discards up to 1 basecoin of fractional remainder. With N validator events processed in a single claim, the user can receive up to N basecoins less than they are mathematically entitled to. The dust silently stays in the rewards pool.

---

### Finding Description

`computeSegmentBonus` computes:

```
shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear
```

and immediately calls `.TruncateInt()` before returning a `math.Int`:

```go
// x/tieredrewards/keeper/bonus_rewards.go lines 41-45
return tokens.
    Mul(tier.BonusApy).
    MulInt64(durationSeconds).
    QuoInt64(types.SecondsPerYear).
    TruncateInt()
```

`processEventsAndClaimBonus` then calls this function once per validator event and accumulates the already-truncated integers:

```go
// x/tieredrewards/keeper/claim_rewards.go lines 161, 178-179, 211-212
totalBonus := math.ZeroInt()
// ...
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
totalBonus = totalBonus.Add(bonus)   // truncated integer added
// ...
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
totalBonus = totalBonus.Add(bonus)   // truncated integer added again
```

Each call to `computeSegmentBonus` independently truncates the fractional part of that segment's bonus. The fractional remainders are never carried forward. The correct approach — matching the external report's recommendation — is to accumulate `math.LegacyDec` values across all segments and call `.TruncateInt()` exactly once on the final sum.

---

### Impact Explanation

A user claiming bonus rewards whose position has spanned N validator events (slashes, unbonds, bonds) receives up to N basecoins less than they are owed. The shortfall is not redistributed; it silently remains in the `RewardsPoolName` module account. For positions with long histories on active validators (frequent slashes or jailing/unjailing cycles), the accumulated underpayment grows proportionally to the number of events. The corrupted value is the `totalBonus` integer sent via `bankKeeper.SendCoinsFromModuleToAccount` to the position owner.

---

### Likelihood Explanation

Any delegator who calls `MsgClaimTierRewards` on a position that has accumulated more than one validator event since its last claim triggers this path. Validator slashes, jailing, and unjailing are routine on a live PoS chain. A position held for months on an active validator will routinely accumulate multiple events per claim cycle, making this a near-certain occurrence for long-term tier participants.

---

### Recommendation

Change `computeSegmentBonus` to return `math.LegacyDec` instead of `math.Int`, accumulate the decimal values in `processEventsAndClaimBonus`, and call `.TruncateInt()` exactly once on the final sum before constructing `bonusCoins`:

```go
// computeSegmentBonus returns math.LegacyDec (no truncation)
func (k Keeper) computeSegmentBonus(...) math.LegacyDec {
    ...
    return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear)
}

// processEventsAndClaimBonus accumulates Dec, truncates once
totalBonusDec := math.LegacyZeroDec()
...
totalBonusDec = totalBonusDec.Add(k.computeSegmentBonus(...))
...
totalBonus := totalBonusDec.TruncateInt()
```

---

### Proof of Concept

Consider a position with `shares = 1`, `tokensPerShare = 1.0`, `bonusApy = 0.10`, and three validator events creating three segments each of duration `(SecondsPerYear / 3) + 1` seconds (so each segment's exact bonus is `0.10 × (SecondsPerYear/3 + 1) / SecondsPerYear ≈ 0.0333...` tokens, which truncates to 0).

- Current code: `computeSegmentBonus` returns `0` for each segment → `totalBonus = 0`. User receives nothing.
- Correct code: accumulated decimal `≈ 0.1000...` → `TruncateInt()` → user receives `0` only if the full-year sum is still below 1, but for realistic larger positions the difference is concrete.

For a more concrete case: `shares = 1000`, `tokensPerShare = 2.0`, `bonusApy = 0.04`, three segments of `(SecondsPerYear / 3) + 1` seconds each:

- Each segment exact value: `1000 × 2.0 × 0.04 × (SecondsPerYear/3 + 1) / SecondsPerYear ≈ 26.666...` → truncates to `26`
- Current: `totalBonus = 26 + 26 + 26 = 78`
- Correct: `totalBonusDec ≈ 80.000...` → `TruncateInt() = 80`
- **User is underpaid by 2 basecoins** for this single claim with 3 events.

With N events per claim and many positions, the systematic underpayment is concrete and reachable via the standard `MsgClaimTierRewards` transaction path. [1](#0-0) [2](#0-1) [3](#0-2)
