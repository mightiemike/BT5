### Title
Stale Python Object Identity Reuse in `clvm_tree_to_lazy_node` Causes Incorrect CLVM Tree Construction — (`File: wheel/src/api.rs`)

### Summary
`clvm_tree_to_lazy_node` uses raw CPython object memory addresses (`obj.as_ptr() as usize`) as cache keys in `identity_map`. Because CPython immediately reclaims memory when an object's reference count drops to zero, a new Python object can be allocated at the same address as a previously-processed (and now-dropped) child node. The `identity_map` then silently returns the stale `NodePtr` for the new, structurally different object — the same class of identity-confusion bug as the referral-minter report: an entity's future identity is pre-occupied by a prior entity, and no validation catches the aliasing.

### Finding Description

In `clvm_tree_to_lazy_node` (`wheel/src/api.rs`), three maps are maintained:

```
identity_map: HashMap<usize, NodePtr>   // keyed by Python object address
atom_map:     HashMap<Vec<u8>, NodePtr>
pair_map:     HashMap<(NodePtr,NodePtr), NodePtr>
``` [1](#0-0) 

When a pair node is visited, the addresses of its children are captured immediately:

```rust
let left_id  = left.as_ptr()  as usize;
let right_id = right.as_ptr() as usize;
let left_done  = identity_map.contains_key(&left_id);
``` [2](#0-1) 

If either child is not yet processed, a deferred `WorkItem::BuildPair { id, left_id, right_id }` is pushed, and the child `Bound<'py, PyAny>` objects are pushed as `WorkItem::Visit(left/right)`: [3](#0-2) 

Once `WorkItem::Visit(left)` is popped and processed, the `Bound<'py, PyAny>` smart pointer is consumed and dropped. In CPython, this immediately decrements the Python object's reference count. If the caller's Python tree uses **lazy pair evaluation** (a `.pair` property that constructs fresh Python objects on each access rather than returning cached children), no other Python reference to `left` exists at that point, so CPython frees the object immediately.

A subsequent allocation — triggered by any Python object creation during the traversal of sibling or cousin nodes — can reuse that freed address. When the new object is later encountered as a child of another pair, the check:

```rust
let left_done = identity_map.contains_key(&left_id);  // left_id == freed address
```

returns `true`, and the lookup:

```rust
let l = identity_map[&left_id];
``` [4](#0-3) 

silently returns the `NodePtr` of the **old, unrelated atom or pair** that previously occupied that address. The new object is never visited; its actual content is never inserted into the allocator. The resulting `LazyNode` tree is structurally wrong.

### Impact Explanation

The corrupted `LazyNode` is then passed to serializers such as `ser_2026` or `ser_backrefs`: [5](#0-4) 

The serialized bytes will encode the wrong subtree. If those bytes are used as a CLVM program or puzzle reveal on the Chia blockchain, the resulting tree hash will differ from what the caller intended, causing:

- **Consensus divergence**: nodes that re-derive the tree hash from the serialized bytes will compute a different value than the caller expected.
- **Incorrect program execution**: the CLVM interpreter will run a different program than intended, potentially bypassing spend conditions.
- **API equivalence break**: `clvm_tree_to_lazy_node(t)` followed by `ser_2026(...)` will not produce the same bytes as the canonical serialization of `t`, violating the documented contract shown in the Python test suite. [6](#0-5) 

### Likelihood Explanation

The attack requires a Python caller to supply an object whose `.pair` attribute is a property (or `__getattr__`) that constructs fresh Python objects on each access. This is a realistic pattern in lazy CLVM tree implementations (e.g., streaming deserializers, proxy objects, or any wrapper that does not cache its children). CPython's reference-counting GC makes the reclamation deterministic and immediate — no GC cycle is needed. An attacker who controls the Python object passed to `clvm_tree_to_lazy_node` (the function is publicly exported in the `clvm_rs` Python module) can reliably trigger the aliasing. [7](#0-6) 

### Recommendation

Replace the raw-pointer identity key with a stable, unforgeable identity. Two concrete options:

1. **Hold all child `Bound<'py, PyAny>` objects alive for the duration of the traversal** (e.g., store them in a `Vec` alongside the `identity_map`) so their reference counts never drop to zero and their addresses cannot be reused.
2. **Remove `identity_map` entirely and rely solely on `atom_map` and `pair_map`**, which key on content rather than address. This is safe because `atom_map` deduplicates by byte content and `pair_map` deduplicates by `(NodePtr, NodePtr)` — neither is address-dependent. The `identity_map` is only needed to short-circuit re-traversal of shared Python objects; that optimization can be replaced by always traversing and letting the content maps handle deduplication.

### Proof of Concept

```python
import gc
from clvm_rs.clvm_rs import clvm_tree_to_lazy_node, ser_2026

class LazyAtom:
    def __init__(self, data: bytes):
        self._data = data
    @property
    def atom(self):
        return self._data
    @property
    def pair(self):
        return None

class LazyPair:
    """Returns brand-new Python objects on every .pair access — no caching."""
    def __init__(self, left_bytes, right_bytes):
        self._left_bytes  = left_bytes
        self._right_bytes = right_bytes
    @property
    def atom(self):
        return None
    @property
    def pair(self):
        # Fresh objects every call → old ones become immediately collectable
        return (LazyAtom(self._left_bytes), LazyAtom(self._right_bytes))

# Build a tree whose left child is b"AAAA" and right child is b"BBBB"
root = LazyPair(b"AAAA", b"BBBB")

# Force GC between visits to maximise address reuse
gc.collect()

lazy = clvm_tree_to_lazy_node(root)
result = ser_2026(lazy, level=0)

# Canonical serialization of (b"AAAA" . b"BBBB")
import clvmr
a = clvmr.Allocator()
left  = a.new_atom(b"AAAA")
right = a.new_atom(b"BBBB")
pair  = a.new_pair(left, right)
from clvm_rs.clvm_rs import ser_2026 as _ser
from clvm_rs.clvm_rs import deser_backrefs
canonical = _ser(deser_backrefs(bytes([0xff,0x84,0x41,0x41,0x41,0x41,
                                        0x84,0x42,0x42,0x42,0x42])), level=0)

# Under the bug, `result` may encode b"AAAA" in both positions
# because the freed address of the first LazyAtom(b"AAAA") was
# reused when the second LazyAtom(b"BBBB") was allocated.
assert result == canonical, f"Tree corrupted: {result.hex()} != {canonical.hex()}"
```
