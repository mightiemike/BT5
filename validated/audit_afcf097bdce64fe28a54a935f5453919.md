### Title
`ENABLE_SECP_OPS` Flag Gates 1-Byte Secp Opcodes (64, 65) But Not 4-Byte Secp Opcodes (0x13d61f00, 0x1c3a8f00) in `ChiaDialect::op()` — (`src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` contains two independent dispatch paths that both resolve to `op_secp256k1_verify` and `op_secp256r1_verify`. The 1-byte opcode path (opcodes 64 and 65) is correctly gated behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (0x13d61f00 and 0x1c3a8f00) dispatches to the same functions with no flag check at all. An attacker-controlled CLVM program using 4-byte secp opcodes can invoke secp verification in any execution context — including `MEMPOOL_MODE` — regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

`ChiaDialect::op()` handles two opcode-length branches. For 4-byte opcodes, the dispatch is:

```rust
// src/chia_dialect.rs lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No `ENABLE_SECP_OPS` check is present. [1](#0-0) 

For 1-byte opcodes, the same underlying functions are gated:

```rust
// src/chia_dialect.rs lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

`MEMPOOL_MODE` explicitly excludes `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [3](#0-2) 

In mempool mode, a 1-byte opcode 64 or 65 falls through to `unknown_operator`, which returns `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set — correctly rejecting the operation. [4](#0-3) 

But a 4-byte opcode 0x13d61f00 or 0x1c3a8f00 is explicitly matched and dispatched to the secp verification function with no flag check, executing successfully. [5](#0-4) 

The two secp operator implementations themselves accept `_flags: ClvmFlags` but ignore it entirely — all validation logic is internal to the function and does not re-check `ENABLE_SECP_OPS`. [6](#0-5) 

---

### Impact Explanation

The `ENABLE_SECP_OPS` flag is the designated gate for secp operations. When it is absent (e.g., in `MEMPOOL_MODE`, or on a node that has not yet activated the secp soft-fork), the intent is that secp verification cannot be invoked. The 4-byte opcode path breaks this invariant: any caller that constructs a CLVM program using 4-byte secp opcodes bypasses the flag entirely and executes full secp signature verification.

Concrete broken invariants:
- **Mempool policy bypass**: Mempool nodes reject 1-byte secp opcodes but silently accept 4-byte ones, allowing programs that should be filtered to pass mempool validation.
- **Consensus divergence**: If `ENABLE_SECP_OPS` is used as a soft-fork activation flag, nodes with the flag unset will reject 1-byte secp programs but accept 4-byte secp programs. Nodes with the flag set accept both. This produces divergent validation results for the same spend bundle, which is a consensus-splitting condition.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to a node is attacker-supplied bytes. Constructing a program that uses 4-byte opcode 0x13d61f00 instead of 1-byte opcode 64 requires only knowledge of the opcode table, which is public. No special privileges, social engineering, or compromised infrastructure are required. The 4-byte opcodes are documented in the cost formula comment in the source itself. [7](#0-6) 

---

### Recommendation

Add an `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the check already present for 1-byte opcodes 64 and 65:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This makes both dispatch paths consistent, exactly as the original report recommended adding the variant check to `pairTransferNFTFrom()` and `pairTransferERC1155From()` to match `pairTransferERC20From()`.

---

### Proof of Concept

A CLVM program that invokes secp256k1 verification via the 4-byte opcode in a context where `ENABLE_SECP_OPS` is not set (e.g., `MEMPOOL_MODE`):

1. Construct a CLVM atom of exactly 4 bytes: `[0x13, 0xd6, 0x1f, 0x00]` (the secp256k1 opcode).
2. Build a CLVM program that applies this atom as an operator to a valid `(pubkey msg sig)` argument list.
3. Run the program under `ChiaDialect::new(MEMPOOL_MODE)`.

Expected (correct) behavior: the program is rejected because `ENABLE_SECP_OPS` is absent.

Actual behavior: `ChiaDialect::op()` enters the `op_len == 4` branch at line 157, matches `0x13d61f00` at line 176, and calls `op_secp256k1_verify` without consulting `ENABLE_SECP_OPS`. The secp verification executes and returns a result, bypassing the flag gate entirely. [8](#0-7)
