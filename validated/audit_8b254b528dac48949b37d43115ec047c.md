### Title
Missing Parent-Block Hash-vs-Target Check in Dogecoin AuxPoW Validation - (File: `contract/src/dogecoin.rs`)

### Summary
The Dogecoin AuxPoW submission path in `submit_block_header` verifies the structural validity of the AuxPoW proof (merkle branch, chain IDs, version flags) but never checks that the parent block's hash is less than or equal to the Dogecoin difficulty target. Any unprivileged NEAR caller acting as a relayer can submit a structurally well-formed AuxPoW whose parent block hash exceeds the target, and the contract will accept it as a valid Dogecoin block.

### Finding Description

The non-AuxPoW Dogecoin path and all other chains (Bitcoin, Litecoin, Zcash) explicitly verify that the block's PoW hash satisfies the difficulty target:

```rust
// contract/src/lib.rs  lines 520-525  (Bitcoin / Litecoin / Zcash)
let pow_hash = header.block_hash_pow();
require!(
    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
    format!("block should have correct pow")
);
``` [1](#0-0) 

The Dogecoin `submit_block_header` branches on whether AuxPoW data is present. For the non-AuxPoW branch it performs the same hash-vs-target check. For the AuxPoW branch it calls only `check_aux` and then falls through — the hash-vs-target check is entirely absent:

```rust
// contract/src/dogecoin.rs  lines 176-188
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);

    if let Some(ref aux_data) = aux_data {
        self.check_aux(&block_header, aux_data);   // ← no hash-vs-target check
    } else {
        let pow_hash = block_header.block_hash_pow();
        require!(
            U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
            format!("block should have correct pow")
        );
    }
}
``` [2](#0-1) 

`check_aux` validates the AuxPoW structure — chain merkle proof length, the `BLOCK_VERSION_AUXPOW` flag, chain-ID consistency, and the merkle root computation — but none of these steps compare the parent block's hash against the Dogecoin difficulty target: [3](#0-2) 

`check_target` (called before the branch) validates only that `block_header.bits` equals the expected difficulty value; it does not evaluate the parent block's hash at all: [4](#0-3) 

In AuxPoW / merged-mining, the parent block's hash is the actual proof of work. The Dogecoin protocol requires `SHA256d(parent_block_header) <= target_from_bits(dogecoin_block.bits)`. Without this check the contract cannot distinguish a legitimately mined AuxPoW block from a fabricated one.

### Impact Explanation

An attacker submits a Dogecoin block whose `bits` field encodes a valid (but easy or even zero-difficulty) target, together with a structurally valid AuxPoW (correct chain IDs, a valid merkle branch, the `BLOCK_VERSION_AUXPOW` flag set). Because the parent block's hash is never compared to the target, the contract stores the block, advances `mainchain_tip_blockhash`, and accumulates chain-work computed from the attacker-chosen `bits` value. By repeating this with inflated `bits` (low difficulty → high `work_from_bits` value) the attacker can manufacture a fake chain with arbitrarily large chain-work, causing the contract to reorg onto the attacker's fabricated chain. Any downstream consumer that calls `verify_transaction_inclusion` against this corrupted chain will receive false confirmations.

### Likelihood Explanation

The `submit_blocks` entry point is callable by any NEAR account that satisfies the trusted-relayer check (or by anyone if that guard is absent or bypassed). Constructing a structurally valid AuxPoW with a fake parent block requires only standard Bitcoin/Dogecoin serialization knowledge — no cryptographic secret is needed. The missing check is a single comparison, so exploitation is straightforward and requires no special tooling.

### Recommendation

After calling `check_aux`, add the same hash-vs-target assertion that the non-AuxPoW branch already performs, using the parent block's hash and the Dogecoin block's target:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
    // Verify the parent block's hash satisfies the Dogecoin target
    let parent_pow_hash = aux_data.parent_block.block_hash_pow();
    require!(
        U256::from_le_bytes(&parent_pow_hash.0) <= target_from_bits(block_header.bits),
        "auxpow: parent block hash does not meet Dogecoin target"
    );
} else {
    ...
}
```

This mirrors the check already present for non-AuxPoW blocks and matches the reference Dogecoin implementation's `CAuxPow::Check`.

### Proof of Concept

1. Construct a Dogecoin block header with `version | BLOCK_VERSION_AUXPOW` and `bits` set to `0x207fffff` (minimum difficulty / maximum target).
2. Build a fake parent Bitcoin block header whose double-SHA256 hash is any value — it need not be below any target.
3. Place the Dogecoin block hash anywhere in the parent coinbase and construct a trivially valid merkle branch.
4. Call `submit_blocks([( dogecoin_header, Some(aux_data) )])` on the NEAR contract.
5. `check_target` passes (bits match the expected value for the chosen difficulty epoch or testnet).
6. `check_aux` passes (chain IDs are correct, merkle branch is valid, version flag is set).
7. The parent block's hash is never compared to `target_from_bits(block_header.bits)`.
8. The contract stores the block and updates `mainchain_tip_blockhash` to the attacker's block hash.
9. Repeat with increasing `chain_work` to trigger a reorg onto the fabricated chain. [5](#0-4)

    pub(crate) fn check_aux(&mut self, block_header: &Header, aux_data: &AuxData) {
        // The Dogecoin block must have the AuxPoW flag set (bit 8) when AuxPoW data is present.
        // https://github.com/dogecoin/dogecoin/blob/master/src/auxpow.h
        const BLOCK_VERSION_AUXPOW: i32 = 0x100;

        require!(
            aux_data.chain_merkle_proof.len() <= 30,
            "Aux POW chain merkle branch too long"
        );
        require!(
            block_header.version & BLOCK_VERSION_AUXPOW != 0,
            "Aux POW block does not have AuxPoW flag set in version"
        );

        let chain_id = self.get_config().aux_chain_id;

        require!(
            chain_id == block_header.get_chain_id(),
            format!(
                "block does not have our chain ID (got {}, expected {chain_id})",
                block_header.get_chain_id()
            )
        );

        require!(
            chain_id != aux_data.parent_block.get_chain_id(),
            "Aux POW parent has our chain ID"
        );

        let chain_root = merkle_tools::compute_root_from_merkle_proof(
            block_header.block_hash(),
            aux_data.chain_id,
            &aux_data.chain_merkle_proof,
        );

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
