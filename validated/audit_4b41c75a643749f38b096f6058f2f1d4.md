### Title
Hardcoded Non-Empty Merkle Proof Requirement Rejects Valid Single-Transaction Block Proofs — (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function contains a hardcoded check that unconditionally rejects empty merkle proofs. For a Bitcoin block containing only one transaction (the coinbase), the merkle proof is legitimately empty — the merkle root IS the transaction hash. This hardcoded threshold makes the protocol incompatible with valid proofs for single-transaction blocks, directly analogous to M-02's hardcoded minimum that made the protocol incompatible with lower-decimal tokens.

---

### Finding Description

In `contract/src/lib.rs` at line 315, inside `verify_transaction_inclusion`:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This check is unconditional. In a Bitcoin block containing exactly one transaction (the coinbase), the Merkle tree has a single leaf. The merkle root equals the coinbase transaction hash, and the merkle proof for that transaction is empty — no sibling hashes are needed. This is a valid, well-defined Bitcoin state.

The production-recommended function `verify_transaction_inclusion_v2` calls `verify_transaction_inclusion` internally:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

Any unprivileged NEAR caller invoking `verify_transaction_inclusion_v2` with a valid empty-proof for a single-transaction block will receive a panic rather than `true`. The proof verification result is corrupted: a valid inclusion proof is treated as an error.

The `verify_transaction_inclusion_v2` length-equality check passes for two empty proofs (`0 == 0`), and the coinbase sub-check also passes (an empty proof with index 0 returns the `coinbase_tx_id` itself as the computed root, which equals the block's `merkle_root` for a single-tx block). The bug is exclusively in the hardcoded non-empty guard. [3](#0-2) 

---

### Impact Explanation

**Medium.** Downstream contracts or bridge protocols that rely on `verify_transaction_inclusion_v2` to release funds upon proof of Bitcoin transaction inclusion cannot process transactions from single-transaction blocks. The function panics instead of returning `true`, so the caller's NEAR transaction fails entirely. For any bridge that gates fund release on a `true` return value, legitimate bridge operations are permanently blocked for affected transactions. The corrupted proof result — panic instead of `true` — is the concrete broken invariant.

---

### Likelihood Explanation

**Low.** Single-transaction blocks (containing only the coinbase) are rare in modern Bitcoin but are a valid, protocol-defined state. They occurred frequently in early Bitcoin history and can still occur during periods of extremely low mempool activity. The condition is not attacker-controlled; it is a natural Bitcoin state the protocol must handle correctly. Any relayer or user submitting headers from such a block and then attempting to verify the coinbase transaction will trigger the bug.

---

### Recommendation

Remove the hardcoded non-empty guard or replace it with a case-split:

- **Empty proof (`merkle_proof.is_empty()`):** Verify directly that `tx_id == header.block_header.merkle_root` and `tx_index == 0`. Return the boolean result without calling `compute_root_from_merkle_proof`.
- **Non-empty proof:** Proceed with the existing `compute_root_from_merkle_proof` path.

This mirrors the M-02 recommendation: replace the hardcoded threshold with logic that scales to all valid inputs.

---

### Proof of Concept

1. A Bitcoin block exists with only the coinbase transaction (genesis block or any single-tx block). Its `merkle_root == coinbase_tx_hash`.
2. The block header is submitted and stored by the relayer via `submit_blocks`.
3. An unprivileged NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_hash`
   - `tx_block_blockhash = block_hash` (stored in mainchain)
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = coinbase_tx_hash`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
4. Length check passes: `0 == 0`. [4](#0-3) 
5. Coinbase sub-check passes: `compute_root_from_merkle_proof(coinbase_tx_hash, 0, &[])` returns `coinbase_tx_hash == merkle_root`. [5](#0-4) 
6. `verify_transaction_inclusion` is called internally.
7. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` panics. [1](#0-0) 
8. The valid proof is rejected; the caller's transaction fails.

The root cause is structurally identical to M-02: a hardcoded threshold (`!is_empty()`) that does not account for all valid protocol inputs (single-transaction blocks), limiting the protocol's correctness for legitimate use cases.

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
