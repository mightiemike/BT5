### Title
Permissionless `submitFastWithdrawal` Allows Arbitrary Ordering of Fast Withdrawals, Enabling Front-Running and Pool Drain - (`File: core/contracts/BaseWithdrawPool.sol`)

### Summary
`BaseWithdrawPool.submitFastWithdrawal()` is a permissionless public function that accepts any `idx > minIdx` in any order. Because multiple pending fast withdrawals all satisfy this single constraint simultaneously, any caller can choose which withdrawal to process first. This is a direct analog to the Connext out-of-order message processing bug: an unprivileged actor can selectively sequence fast withdrawals to front-run other users or drain the pool before a target withdrawal executes.

### Finding Description
`submitFastWithdrawal` enforces only two ordering constraints:

```solidity
require(!markedIdxs[idx], "Withdrawal already submitted");
require(idx > minIdx, "idx too small");
```

`minIdx` is updated only when the sequencer processes a withdrawal via `submitWithdrawal` (called by the clearinghouse). Until that happens, every pending fast withdrawal with `idx > minIdx` is simultaneously eligible for processing in any order by any caller. [1](#0-0) 

The signed withdrawal transactions and verifier signatures required by `v.requireValidTxSignatures(transaction, idx, signatures)` are observable on-chain or via the protocol's subgraph, so any observer can reconstruct and submit them. [2](#0-1) 

By contrast, the slow-mode queue in `Endpoint._executeSlowModeTransaction` enforces strict FIFO via `txUpTo`, so no analogous ordering bypass exists there. [3](#0-2) 

### Impact Explanation
When the `WithdrawPool` holds limited liquidity (a realistic condition since it is funded by LP deposits, not the full protocol treasury), an attacker can:

1. Observe multiple pending fast withdrawals (idx1 < idx2 < idx3, all > `minIdx`) with their public signatures.
2. Call `submitFastWithdrawal(idx3, ...)` first, processing the highest-index withdrawal ahead of earlier ones.
3. Repeat for idx2, leaving idx1 (the earliest submitter) last.
4. If pool liquidity is exhausted, the earliest submitter's `handleWithdrawTransfer` call reverts or transfers zero, denying them their fast withdrawal.

Additionally, a user whose own withdrawal is at idx3 can front-run idx1 and idx2 to guarantee their funds are available before the pool is drained. The fee structure (`sendTo == msg.sender` deducts fee from amount; otherwise caller pays) means the attacker can profitably self-serve their own withdrawal while blocking others. [4](#0-3) 

### Likelihood Explanation
The attack requires only:
- Observing pending fast withdrawal transactions and their verifier signatures (fully public via subgraph or on-chain events).
- Calling `submitFastWithdrawal` with a chosen `idx` before others do.

No privileged access, leaked keys, or governance capture is needed. Any user with a pending fast withdrawal has a direct financial incentive to front-run others when pool liquidity is constrained. Likelihood is **medium**: the pool must have insufficient liquidity to cover all pending withdrawals simultaneously, but this is a realistic condition during high withdrawal volume.

### Recommendation
Enforce FIFO ordering for fast withdrawals by tracking the next expected index and requiring `idx == nextFastWithdrawalIdx` (incrementing after each successful submission), mirroring the FIFO discipline already applied to the slow-mode queue via `txUpTo`. Alternatively, document explicitly that fast withdrawals are processed on a first-caller-wins basis and that pool liquidity is not guaranteed to be available in submission order, so users should not rely on ordering fairness.

### Proof of Concept
1. Suppose `minIdx = 50`. Three users submit fast withdrawals at `idx = 51`, `52`, `53`. The pool holds enough liquidity for two withdrawals only.
2. Attacker (owner of `idx = 53`) calls `submitFastWithdrawal(53, txn53, sigs53)`. Passes `idx > minIdx` check. Pool pays out withdrawal 53.
3. Attacker calls `submitFastWithdrawal(52, txn52, sigs52)`. Pool pays out withdrawal 52.
4. User at `idx = 51` calls `submitFastWithdrawal(51, txn51, sigs51)`. Pool balance is now zero; `handleWithdrawTransfer` → `token.safeTransfer` reverts. User 51 cannot fast-withdraw despite having submitted first. [5](#0-4) [6](#0-5)

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
```

```text
    function submitWithdrawal(
        IERC20Base token,
        address sendTo,
        uint128 amount,
        uint64 idx
    ) public {
        require(msg.sender == clearinghouse);

        if (markedIdxs[idx]) {
            return;
        }
        markedIdxs[idx] = true;
        // set minIdx to most recent withdrawal submitted by sequencer
        minIdx = idx;

        handleWithdrawTransfer(token, sendTo, amount);
    }
```
