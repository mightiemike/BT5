### Title
`AllResources` V3 Transaction with Zero L2/L1DataGas Bounds Silently Downgraded to `L1Gas` via Protobuf Deserialization, Producing a Different Transaction Hash and Wrong Execution Mode — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — checking whether `l2_gas` and `l1_data_gas` are both zero — to decide which enum variant to reconstruct. An `AllResources` V3 transaction whose user legitimately set both of those bounds to zero is silently downgraded to `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` includes the L1DataGas field in the hash only for `AllResources`, the two variants produce **different transaction hashes** for the same numeric inputs. The same lossy heuristic is duplicated in the RPC `ResourceBoundsMapping` converter.

---

### Finding Description

**Root cause — protobuf deserializer (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 431–435):**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) faithfully encodes all three fields for both variants, so the wire bytes are identical for `AllResources{l2_gas=0, l1_data_gas=0}` and `L1Gas`. The deserializer then uses the zero-value heuristic to pick the variant, permanently losing the original type information.

The same heuristic is duplicated in the RPC layer:

```rust
if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
    Self::L1Gas(value.l1_gas)
} else { ... }
``` [2](#0-1) 

**Hash divergence — `get_tip_resource_bounds_hash` (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3-element hash
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4-element hash
    }
});
``` [3](#0-2) 

For `L1Gas` the poseidon input is `[tip, L1Gas_packed, L2Gas_packed]` (3 elements). For `AllResources` it is `[tip, L1Gas_packed, L2Gas_packed, L1DataGas_packed]` (4 elements). Even when `l2_gas = 0` and `l1_data_gas = 0`, the packed L1DataGas element is non-zero (it encodes the resource name `L1_DATA` in the upper bits), so the two hashes are numerically different.

**Execution-mode divergence — `valid_resource_bounds_as_felts` and `calculate_resource_bounds`:**

`ValidResourceBounds::L1Gas` causes `get_gas_vector_computation_mode()` to return `NoL2Gas` and `calculate_resource_bounds` to return only 2 resource-bound entries to the contract via `get_execution_info`. `AllResources` returns 3. The OS Cairo code asserts `n_resource_bounds = 3` for V3 transactions. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Transaction hash mismatch (Critical / High):**

The gateway computes hash H1 using `AllResources` (4-element poseidon input, includes L1DataGas). After the transaction is propagated over P2P via protobuf, the receiving node deserializes it as `L1Gas` and recomputes hash H2 (3-element poseidon input, excludes L1DataGas). H1 ≠ H2. The receiving node either rejects the transaction (valid transaction rejected — High admission impact) or stores it under the wrong hash (wrong receipt/event — Critical execution impact).

**Wrong execution mode (Critical):**

A node that accepted the transaction with the downgraded `L1Gas` type executes it under `GasVectorComputationMode::NoL2Gas`. L2 gas is not checked against the user's signed bounds. The `get_execution_info` syscall returns 2 resource-bound entries instead of 3, diverging from what the account contract signed and what the OS expects, potentially causing account `__validate__` to fail or to accept a transaction it should reject.

---

### Likelihood Explanation

Any V3 invoke transaction submitted with `AllResourceBounds` where both `l2_gas` and `l1_data_gas` are zero triggers this. The gateway's stateless validator accepts `AllResourceBounds` with any combination of zero bounds (it only rejects if *all* bounds are zero). A user who sets only `l1_gas` and leaves the other two at zero — a common pattern for L1-only fee transactions — will hit this on every P2P hop. [6](#0-5) 

---

### Recommendation

The protobuf deserializer must not infer the `ValidResourceBounds` variant from field values. The protobuf schema should carry an explicit discriminant (e.g., a boolean `is_all_resources` flag or a `oneof` wrapper). Until the schema is updated, the deserializer should default to `AllResources` whenever `l1_data_gas` is present in the wire message (even if zero), reserving `L1Gas` only for the legacy case where `l1_data_gas` is absent (`None`):

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if value.l1_data_gas.is_none() {   // <-- use presence, not value
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The same fix must be applied to `From<ResourceBoundsMapping> for ValidResourceBounds` in the RPC layer, where the `L1DataGas` key's *presence* in the map (not its zero value) should determine the variant. [7](#0-6) [2](#0-1)

### Proof of Concept

1. Construct a valid V3 invoke transaction at the gateway with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: N, max_price_per_unit: P },
       l2_gas:      { max_amount: 0, max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0, max_price_per_unit: 0 },
   }
   ```
2. The gateway calls `InternalRpcInvokeTransactionV3::resource_bounds()` which always returns `ValidResourceBounds::AllResources(...)`, so hash H1 is computed with the 4-element poseidon input (includes L1DataGas packed as `0x4c315f44415441_0000000000000000_00000000000000000000000000000000`).
3. Serialize the transaction to protobuf via `From<ValidResourceBounds> for protobuf::ResourceBounds` — all three fields are present and non-`None` in the wire message.
4. Deserialize on the receiving node: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
5. Recompute hash H2 with the 3-element poseidon input (L1DataGas excluded). H1 ≠ H2.
6. The receiving node's hash verification fails; the transaction is dropped or stored under the wrong hash, causing wrong receipts and breaking the user's transaction. [8](#0-7) [9](#0-8)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
