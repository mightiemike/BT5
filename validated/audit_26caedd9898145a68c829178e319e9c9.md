### Title
Linked Signer Can Self-Escalate Authority via `LinkSigner` to Hijack Subaccount and Drain Funds ŌĆö (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is processed with `allowLinkedSigner = true`, meaning the current linked signer can replace itself with any attacker-controlled address. Once replaced, the attacker's address inherits full linked-signer authority ŌĆö including the ability to sign `TransferQuote` ŌĆö enabling complete drainage of the victim subaccount's quote balance to an arbitrary recipient subaccount. This is the on-chain analog of the overlay attack: a malicious actor "overlays" a fake signer identity on top of the legitimate one, silently hijacking the subaccount's authorization surface before the owner can react.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is validated with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576ŌĆō590
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ŌåÉ allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` calls `validateSignature`, which calls `Verifier.validateSignature`:

```solidity
// Verifier.sol lines 291ŌĆō304
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),   // ŌåÉ linked signer accepted
    ERR_INVALID_SIGNATURE
);
```

Because the linked signer is accepted as a valid signer for `LinkSigner` transactions, the current linked signer can sign a `LinkSigner` payload that writes any new address into `linkedSigners[subaccount]`. The attacker-controlled replacement address then inherits all linked-signer privileges.

`TransferQuote` is also processed with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 598ŌĆō614
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true   // ŌåÉ allowLinkedSigner
);
...
clearinghouse.transferQuote(signedTx.tx);
```

The `TransferQuote` struct accepts an arbitrary `recipient` subaccount, so the attacker can transfer the entire quote balance to their own subaccount in the same sequencer batch.

The `linkedSigners` mapping is stored in `EndpointStorage`:

```solidity
// EndpointStorage.sol line 50
mapping(bytes32 => address) internal linkedSigners;
```

The state delta is: `linkedSigners[victim_subaccount]` is overwritten from the legitimate hot-wallet address to the attacker's address, followed by `spotEngine` quote balance of `victim_subaccount` being zeroed and credited to the attacker's subaccount.

---

### Impact Explanation

A compromised or malicious linked signer (hot wallet) can:

1. Sign a `LinkSigner` transaction pointing `signer` to an attacker-controlled address ŌĆö overwriting `linkedSigners[victim]`.
2. Immediately sign a `TransferQuote` transaction with the new key, draining the victim's quote balance to the attacker's subaccount.

Both steps can be submitted in a single sequencer batch, making the attack atomic. The subaccount owner (cold wallet) retains the ability to sign a corrective `LinkSigner` transaction, but the funds are already gone before any on-chain reaction is possible. The corrupted state is: `linkedSigners[victim]` permanently replaced, and the victim's quote balance zeroed.

**Impact: High** ŌĆö direct, irreversible asset loss from any subaccount that has ever linked a signer.

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets ŌĆö keys held in memory by trading bots, browser extensions, or API servers. These are materially more likely to be compromised than the cold-wallet owner key. Any key-leak, supply-chain compromise of a trading client, or phishing of a bot operator is sufficient to trigger this attack. No admin access, sequencer compromise, or governance capture is required. The attacker only needs the linked signer's private key and the ability to submit a sequencer transaction.

**Likelihood: Medium** ŌĆö realistic for any active trader using the linked-signer delegation feature.

---

### Recommendation

Disallow the linked signer from signing `LinkSigner` transactions. Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
// EndpointTx.sol ŌĆö LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner (cold wallet) may rotate the linked signer
);
```

This ensures that only the subaccount owner's key can mutate the `linkedSigners` mapping, preserving the invariant that the cold wallet always retains revocation authority regardless of the hot wallet's state.

### Proof of Concept

**Setup:**
- Victim subaccount: `alice_subaccount` (owner address `0xAlice`, linked signer `0xHotWallet`)
- Attacker controls `0xHotWallet` (e.g., via key leak)
- Attacker owns `attacker_subaccount` (address prefix `0xAttacker`)

**Step 1 ŌĆö Overlay the signer:**
Attacker signs a `LinkSigner` transaction with `0xHotWallet`'s key:
```
sender  = alice_subaccount
signer  = bytes32(uint256(uint160(0xAttacker)))  // attacker's subaccount prefix
nonce   = current nonce of alice_subaccount
```
Submitted via `submitTransactionsChecked`. Result: `linkedSigners[alice_subaccount] = 0xAttacker`.

**Step 2 ŌĆö Drain via TransferQuote:**
Attacker signs a `TransferQuote` transaction with `0xAttacker`'s key:
```
sender    = alice_subaccount
recipient = attacker_subaccount
amount    = alice_subaccount's full quote balance
nonce     = incremented nonce
```
Submitted in the same batch. Result: `alice_subaccount` quote balance ŌåÆ 0; `attacker_subaccount` quote balance increases by the full amount.

Both steps execute atomically within a single sequencer batch. The cold wallet (`0xAlice`) has no opportunity to intervene before the funds are transferred. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
