### Title
`promise_return` Appends `output_data_receivers` to a Receipt After Its Size Has Already Passed `max_receipt_size` Validation, Allowing Unprivileged Users to Produce Receipts That Exceed the Hard Protocol Size Limit — (`File: runtime/runtime/src/lib.rs`)

---

### Summary

When a contract executes `promise_return(C)` inside a receipt that itself has `output_data_receivers`, the runtime appends those receivers to receipt C **after** C has already been validated against `max_receipt_size`. The resulting receipt silently exceeds the 4 MiB hard limit, bypasses all subsequent size checks (which use `ValidateReceiptMode::ExistingReceipt`), and is forwarded cross-shard, inflating state-witness size beyond the protocol-mandated 17 MiB ceiling. The bug is acknowledged in the codebase (issue #12606) but has no protocol-level fix; only a local accounting workaround exists.

---

### Finding Description

`validate_receipt` enforces `max_receipt_size` only in `ValidateReceiptMode::NewReceipt`:

```rust
// runtime/runtime/src/verifier.rs  line 533-541
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)…;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { … });
    }
}
``` [1](#0-0) 

After the function-call action completes, `apply_action_receipt` checks whether the result is `ReturnData::ReceiptIndex` (i.e., `promise_return`). If so, it **mutates** the already-validated receipt C by appending the parent receipt's `output_data_receivers` to it:

```rust
// runtime/runtime/src/lib.rs  lines 1028-1035
ReceiptEnum::ActionV2(new_action_receipt)
| ReceiptEnum::PromiseYieldV2(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
``` [2](#0-1) 

No re-validation of the receipt's Borsh size is performed after this mutation. The receipt is then forwarded to the destination shard, where it is processed under `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check:

```rust
// runtime/runtime/src/verifier.rs  lines 578-585

///    Runtime has to handle them gracefully until the receipt size limit bug is fixed.
///    See https://github.com/near/nearcore/issues/12606 for details.
ExistingReceipt,
``` [3](#0-2) 

The only mitigation is a local accounting clamp in `congestion_control.rs` that pretends the receipt is at most `max_receipt_size` for outgoing-size-limit bookkeeping, but does **not** prevent the oversized receipt from being created or forwarded:

```rust
// runtime/runtime/src/congestion_control.rs  lines 413-427
// There is a bug which allows to create receipts that are above the size limit.
// Let's pretend that all receipts are at most `max_receipt_size` to avoid receipts getting stuck.
if size > max_receipt_size { size = max_receipt_size; }
``` [4](#0-3) 

The same two-step bypass also applies to `DataReceipt` produced by `promise_return` of a large value: the returned value is wrapped in a `DataReceipt` whose Borsh size can exceed `max_receipt_size` without triggering a rejection. [5](#0-4) 

---

### Impact Explanation

`max_receipt_size` (4,194,304 bytes) is a **hard protocol invariant** introduced to keep `ChunkStateWitness` under 17 MiB. An oversized receipt forwarded cross-shard contributes its full Borsh size to the witness, not the clamped accounting value. A single receipt can reach up to `outgoing_receipts_big_size_limit` = 4,718,592 bytes (4.5 MiB), pushing the witness over the ceiling. Chunk validators independently compute witness size; if their computed size diverges from the producer's clamped accounting, they may refuse to endorse the chunk, causing a liveness failure on the receiving shard. Even without a liveness failure, the invariant that "all receipts in the witness are ≤ 4 MiB" is broken, undermining the correctness of all downstream size-budget calculations.

**Severity: High**

---

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this path. The recipe is:
1. Create a promise DAG `[A -then-> B]`.
2. Inside A, create a large receipt C (args padded to `max_receipt_size` minus envelope overhead) and call `promise_return(C)`.
3. The runtime appends B's `DataReceiver` to C, pushing C over the limit.

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates this is reachable and reproducible on mainnet protocol parameters. [6](#0-5) 

**Likelihood: High** — no special privilege required; standard NEAR SDK promise APIs suffice.

---

### Recommendation

Re-validate the Borsh size of receipt C **after** appending `output_data_receivers` in `apply_action_receipt`. If the post-mutation size exceeds `max_receipt_size`, treat it as a `NewReceiptValidationError::ReceiptSizeExceeded` and abort the action (returning an error outcome rather than panicking), consistent with how oversized receipts generated directly by the VM are handled. The same post-mutation check should be applied to `DataReceipt` produced by value-return paths. [7](#0-6) 

---

### Proof of Concept

**Exact divergent value:** receipt C has Borsh size = `max_receipt_size` (4,194,304 bytes) before mutation; after `extend_from_slice` of one `DataReceiver` (~36 bytes), its size becomes ≈ 4,194,340 bytes — strictly greater than `limit_config.max_receipt_size`.

**Call path:**

```
SignedTransaction
  → Runtime::apply
    → process_incoming_receipts (ValidateReceiptMode::ExistingReceipt — no size check)
      → apply_action_receipt (receipt A)
        → execute_function_call → ReceiptManager creates receipt C at max_receipt_size
        → validate_receipt(C, NewReceipt) → PASSES (size == max_receipt_size)
        → [lines 1028-1035] C.output_data_receivers.extend(A.output_data_receivers)
          → C now exceeds max_receipt_size — NO re-validation
        → receipt_sink.forward(C)  ← oversized receipt forwarded cross-shard
          → congestion_control::try_forward clamps size for accounting only
    → process_incoming_receipts on destination shard
        → validate_receipt(C, ExistingReceipt) → size check SKIPPED
        → receipt C processed normally
``` [8](#0-7) [9](#0-8) [1](#0-0)

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
