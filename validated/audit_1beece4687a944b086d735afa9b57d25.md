### Title
Caller-Controlled `confirmations = 0` Bypasses Confirmation-Depth Guard in SPV Proof Verification — (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field with no lower-bound enforcement. Passing `0` trivially satisfies every confirmation-depth check, allowing any unprivileged NEAR caller to obtain a `true` SPV proof result for a transaction that sits at the chain tip with zero subsequent blocks — the exact condition under which a Bitcoin double-spend reorg is most likely to succeed.

---

### Finding Description

The confirmation-depth guard in `verify_transaction_inclusion` is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

Because `args.confirmations` is a `u64` and the left-hand side is also a `u64`, the expression `N >= 0` is **always true** in Rust when `confirmations = 0`. The only other guard is an upper-bound check:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

`0 <= gc_threshold` is also always true. There is **no lower-bound check** anywhere in the function. The `ProofArgs` struct simply accepts whatever `u64` the caller provides:

```rust
pub struct ProofArgs {
    ...
    pub confirmations: u64,
}
``` [3](#0-2) 

`verify_transaction_inclusion_v2` delegates directly to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

The repository's own test suite confirms this is accepted behavior — every passing test uses `confirmations: 0`: [5](#0-4) 

---

### Impact Explanation

The `confirmations` parameter is the contract's sole mechanism for enforcing Bitcoin's probabilistic finality guarantee. A downstream contract or bridge application that calls `verify_transaction_inclusion_v2` and receives `true` is expected to treat the transaction as final. With `confirmations = 0`, the verified block is the chain tip — the block most exposed to reorganization. An attacker who controls mining power (or exploits a natural reorg) can:

1. Broadcast a transaction to a merchant/bridge that calls this contract.
2. Call `verify_transaction_inclusion_v2` with `confirmations: 0` immediately after the block is submitted to the light client.
3. Receive a `true` result and trigger the downstream action (e.g., token mint, fund release).
4. Mine a competing chain that excludes the original transaction, causing a reorg that invalidates the "verified" transaction.

The corrupted invariant is the **proof result**: the contract returns `true` for a transaction that has not achieved any confirmation depth, directly enabling double-spend attacks against any consumer of this API.

---

### Likelihood Explanation

The entry path requires no privilege — any NEAR account can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` as a view or change call. The only prerequisite is that the target block has been submitted to the contract, which is the normal operating condition. The attack is trivially reproducible by any caller who simply omits a meaningful `confirmations` value.

---

### Recommendation

Enforce a hardcoded minimum confirmation count. Reject any proof request where `confirmations` is below that floor:

```rust
const MIN_CONFIRMATIONS: u64 = 1; // or a chain-appropriate value such as 6

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "confirmations must be at least MIN_CONFIRMATIONS"
);
```

Additionally, consider documenting the expected minimum per supported chain (Bitcoin: 6, Litecoin: 12, etc.) and enforcing it at the contract level rather than relying on callers to supply a safe value.

---

### Proof of Concept

The existing test `test_verify_transaction_inclusion_v2_valid` already demonstrates the bypass — it passes `confirmations: 0` against a block at the chain tip and receives `true`: [6](#0-5) 

A malicious caller replicates this exactly:

```rust
// Attacker submits a block (or waits for the relayer to do so), then immediately:
let result: bool = near_client
    .view(contract_id, "verify_transaction_inclusion_v2")
    .args_borsh(ProofArgsV2 {
        tx_id: attacker_tx_hash,
        tx_block_blockhash: tip_block_hash,   // just-submitted tip block
        tx_index: attacker_tx_index,
        merkle_proof: attacker_merkle_proof,
        coinbase_tx_id: coinbase_hash,
        coinbase_merkle_proof: coinbase_proof,
        confirmations: 0,                     // bypasses all depth checks
    })
    .await?
    .json()?;
// result == true, with zero confirmation depth
```

The contract returns `true` for a transaction with no burial depth, satisfying the downstream application's inclusion check while the underlying Bitcoin transaction remains fully reversible.
