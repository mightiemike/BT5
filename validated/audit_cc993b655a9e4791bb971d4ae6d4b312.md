### Title
Unbounded `while` Loop in `tryUnlockNlpBalance` Permanently Locks NLP Tokens for Affected Subaccounts - (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState.tryUnlockNlpBalance` contains an unbounded `while` loop that iterates over every matured entry in a per-subaccount `NlpLockedBalanceQueue`. A user who accumulates a sufficiently large queue (by making many NLP deposits at distinct timestamps) will cause every subsequent NLP balance operation for their subaccount to revert with out-of-gas, permanently locking their NLP tokens with no recovery path.

---

### Finding Description

`tryUnlockNlpBalance` iterates unconditionally over all matured queue entries:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [1](#0-0) 

There is no `maxIterations` parameter or batch-processing limit. The loop runs until either the queue is exhausted or all remaining entries are still locked.

The queue grows in `handleNlpLockedBalance`: every NLP deposit (`amountDelta > 0`) at a timestamp that differs from the last entry's `unlockedAt` appends a new `NlpLockedBalance` entry and increments `balanceCount`. [2](#0-1) 

`tryUnlockNlpBalance` is called unconditionally inside `handleNlpLockedBalance`, which is itself called on every NLP `updateBalance` invocation: [3](#0-2) [4](#0-3) 

This means: once the queue is large enough and all entries have matured past `NLP_LOCK_PERIOD`, every call to `updateBalance` for `NLP_PRODUCT_ID` on that subaccount will attempt to iterate the entire queue and run out of gas.

---

### Impact Explanation

When the loop exceeds the block gas limit, every NLP `updateBalance` call for the affected subaccount reverts. Since NLP withdrawals and balance adjustments route through `updateBalance` → `handleNlpLockedBalance` → `tryUnlockNlpBalance`, the subaccount's NLP tokens become permanently inaccessible. There is no emergency bypass or administrative recovery path visible in the contracts. The locked NLP balance is a concrete, irrecoverable asset loss for the user.

---

### Likelihood Explanation

A user making NLP deposits across many distinct blocks (each at a different oracle timestamp) naturally accumulates one queue entry per deposit. Over weeks or months of regular participation, a moderately active NLP provider can build a queue of hundreds of entries. Once all entries mature past the lock period simultaneously, the next NLP operation triggers the full unbounded loop. No adversarial intent is required; normal usage is sufficient to trigger the condition.

---

### Recommendation

Add a `maxIterations` parameter to `tryUnlockNlpBalance` so callers can process the queue in bounded batches:

```solidity
function tryUnlockNlpBalance(bytes32 subaccount, uint256 maxIterations)
    public
    returns (Balance memory)
{
    NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[subaccount];
    uint256 processed = 0;
    while (
        processed < maxIterations &&
        queue.unlockedUpTo < queue.balanceCount &&
        queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
    ) {
        queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
        delete queue.balances[queue.unlockedUpTo];
        queue.unlockedUpTo++;
        processed++;
    }
    return queue.unlockedBalanceSum;
}
```

Internal callers (e.g., `handleNlpLockedBalance`) should pass a safe default (e.g., 50), and a separate public entry point should allow users to drain the queue incrementally before performing NLP operations.

---

### Proof of Concept

1. Alice makes 500 NLP deposits, each in a separate block (distinct oracle timestamps). Each deposit appends one entry to her `NlpLockedBalanceQueue`; `balanceCount` reaches 500.
2. After `NLP_LOCK_PERIOD` elapses, all 500 entries satisfy `unlockedAt <= getOracleTime()`.
3. Alice attempts to withdraw her NLP tokens. The withdrawal triggers `updateBalance(NLP_PRODUCT_ID, alice, -amount)` → `handleNlpLockedBalance` → `tryUnlockNlpBalance`.
4. The `while` loop attempts 500 storage reads and deletes in a single call, exhausting the block gas limit.
5. The transaction reverts. Every subsequent NLP operation for Alice's subaccount also reverts. Her NLP tokens are permanently locked with no recovery mechanism. [5](#0-4) [6](#0-5)
