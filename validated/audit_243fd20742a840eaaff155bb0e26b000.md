### Title
`BuilderUpdate` Event Declared in Interface but Never Emitted on Builder State Changes — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`IOffchainExchange` declares `event BuilderUpdate(uint32 indexed builder, address indexed owner)` to signal when a builder's configuration changes. However, `OffchainExchange.updateBuilder` — the sole function that mutates the `builders` mapping — never emits it. Every other event declared in `IOffchainExchange` is properly emitted by its corresponding function. Off-chain indexers and frontends that subscribe to `BuilderUpdate` to track builder ownership and fee-rate changes will permanently hold stale state, directly mirroring the ERC-4906 analog where a contract advertises an event contract but silently omits it on every relevant state transition.

---

### Finding Description

`IOffchainExchange.sol` declares the event:

```solidity
event BuilderUpdate(uint32 indexed builder, address indexed owner);
``` [1](#0-0) 

`OffchainExchange.updateBuilder` is the only function that writes to the `builders` mapping. It overwrites all four fields — `owner`, `defaultFeeTier`, `lowestFeeRate`, `highestFeeRate` — but emits nothing:

```solidity
function updateBuilder(bytes calldata transaction) external onlyEndpoint {
    IEndpoint.UpdateBuilder memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.UpdateBuilder)
    );
    builders[txn.builderId] = Builder(
        txn.owner,
        txn.defaultFeeTier,
        txn.lowestFeeRate,
        txn.highestFeeRate
    );
}
``` [2](#0-1) 

Contrast this with every other event in the interface: `FillOrder` is emitted in `matchOrders`, `FeeTierUpdate` in `updateFeeTier`, `ClaimBuilderFee` in `claimBuilderFee`, `BuilderFeePayment` in `emitBuilderEvent`, and `CloseIsolatedSubaccount` in `_tryCloseIsolatedSubaccount`. [3](#0-2) [4](#0-3) 

`BuilderUpdate` is the sole declared event that is never emitted anywhere in the contract.

The `builders` mapping stores the fee-rate bounds that `getUserFeeRateWithBuilder` enforces on every matched order:

```solidity
if (
    builder.owner == address(0) ||
    builderFeeRate > builder.highestFeeRate ||
    builderFeeRate < builder.lowestFeeRate
) {
    revert(ERR_INVALID_BUILDER);
}
``` [5](#0-4) 

When a builder raises `highestFeeRate` or changes `owner`, the on-chain state is updated immediately, but no event is emitted to notify off-chain consumers.

---

### Impact Explanation

Off-chain indexers, frontends, and SDK integrations that subscribe to `BuilderUpdate` to reconstruct the current builder registry will hold permanently stale data after any `updateBuilder` call. Concretely:

- A builder raises `highestFeeRate`. Off-chain systems still display the old (lower) ceiling. Users routing orders through that builder expect fees within the old range; the on-chain `applyFee` path charges the new (higher) rate. The discrepancy is invisible until settlement.
- A builder transfers `owner`. Off-chain systems still show the old owner as the fee-claim beneficiary, misleading integrators about who controls the builder's accrued fees.

The on-chain accounting is correct; the harm is that off-chain consumers cannot detect the change without polling storage directly, which the declared event is designed to make unnecessary.

---

### Likelihood Explanation

`updateBuilder` is a normal sequencer-dispatched operation invoked whenever a builder configuration needs to change. It is not an edge-case path. Every invocation silently omits the event, so the desynchronization is guaranteed to occur on any live builder update.

---

### Recommendation

Add the missing emission at the end of `updateBuilder`:

```solidity
function updateBuilder(bytes calldata transaction) external onlyEndpoint {
    IEndpoint.UpdateBuilder memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.UpdateBuilder)
    );
    builders[txn.builderId] = Builder(
        txn.owner,
        txn.defaultFeeTier,
        txn.lowestFeeRate,
        txn.highestFeeRate
    );
    emit BuilderUpdate(txn.builderId, txn.owner); // add this
}
```

---

### Proof of Concept

1. Deploy the protocol and call `updateBuilder` via the sequencer to register builder ID `1` with `lowestFeeRate = 0`, `highestFeeRate = 50 bps`.
2. Observe: no `BuilderUpdate` event is emitted. Off-chain indexers have no record of builder `1`.
3. Call `updateBuilder` again to raise `highestFeeRate` to `200 bps`.
4. Observe: again no `BuilderUpdate` event. Off-chain systems still show `highestFeeRate = 50 bps` (or nothing at all).
5. A user submits an order with `builderFeeRate = 150 bps` (within the new on-chain range). The order is accepted and the fee is charged on-chain. The user's off-chain fee estimate, computed from stale builder data, showed this rate as invalid and predicted rejection — the actual charge is a surprise.
