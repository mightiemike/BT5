### Title
User-Controlled `confirmations` Parameter Allows Bypassing Finality Check in SPV Proof Verification — (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations: u64` field. The contract enforces only an upper bound (`<= gc_threshold`) but imposes no lower bound. Any unprivileged NEAR caller can set `confirmations = 0`, making the confirmation guard trivially true and receiving a `true` SPV result for a transaction in a block with zero confirmations — a block that may still be reorganized away.

---

### Finding Description

`ProofArgs` and `ProofArgsV2` both expose a `confirmations` field that is entirely caller-controlled: [1](#0-0) [2](#0-1) 

Inside `verify_transaction_inclusion`, the only bounds check is an upper cap: [3](#0-2) 

The actual finality guard is: [4](#0-3) 

Because `confirmations` is `u64`, setting it to `0` makes the expression `(tip_height - target_height + 1) >= 0` unconditionally true. The guard is completely neutralized. The same `confirmations` field flows through `verify_transaction_inclusion_v2` via the `From<ProofArgsV2> for ProofArgs` conversion: [5](#0-4) 

So both public entry points are affected.

---

### Impact Explanation

The `confirmations` parameter exists precisely to enforce finality: a transaction in a block with fewer than N confirmations is still at risk of being erased by a chain reorganization. By setting `confirmations = 0`, a caller receives `true` from the SPV verifier for a transaction whose block may not yet be final. Any downstream contract or bridge that gates asset releases, cross-chain messages, or state transitions on the result of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` can be deceived into acting on an unconfirmed — and potentially rolled-back — Bitcoin transaction.

---

### Likelihood Explanation

The entry point is fully public and requires no privileged role. Any NEAR account can call `verify_transaction_inclusion` with `confirmations = 0`. The manipulation requires no special knowledge beyond reading the public ABI. Callers who want to act on a freshly submitted block (e.g., to front-run a bridge settlement) have a direct incentive to do so.

---

### Recommendation

Enforce a protocol-defined minimum confirmation count. Reject calls where `args.confirmations` falls below that minimum:

```rust
const MIN_CONFIRMATIONS: u64 = 6; // or a contract-stored configurable value

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations below protocol minimum"
);
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

The minimum should not be caller-supplied; it should be a contract constant or an admin-configurable parameter, analogous to the external report's recommendation that the volatility parameter be set by an admin rather than derived from user input.

---

### Proof of Concept

1. Relayer submits block B at height H (the current chain tip, 0 confirmations beyond itself).
2. Attacker calls `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of block B
   - `confirmations = 0`
   - valid `tx_id`, `tx_index`, `merkle_proof` for a transaction in block B
3. The check at line 305 evaluates as `(H - H) + 1 >= 0` → `1 >= 0` → `true`.
4. The function returns `true`.
5. A bridge contract consuming this result releases funds for a Bitcoin transaction that has zero network confirmations and could be double-spent or reorganized away.

The only validation that fires is the upper-bound check (`0 <= gc_threshold`), which always passes. No lower-bound check exists anywhere in the call path. [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
