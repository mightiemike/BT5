### Title
Migration `exitVestedAccountsPositions` halts v8 upgrade when a vesting-owned position has an active redelegation — (`x/tieredrewards/migrations/v2/migrate.go`)

### Summary
The v8 upgrade migration unconditionally calls `ForceFullExitWithDelegation` for every vesting-owned position. `ForceFullExitWithDelegation` calls `transferDelegationFromPosition`, which contains a "Defensive" guard that returns `ErrActiveRedelegation` if the position's delegator address has any pending staking-layer redelegation. Because the migration propagates this error without handling it, a single vesting-owned position with an active redelegation causes `RunMigrations` to fail and the upgrade handler to abort, halting the chain.

### Finding Description

`exitVestedAccountsPositions` iterates all positions, identifies vesting-account owners, and calls `ForceFullExitWithDelegation` for each: [1](#0-0) 

`ForceFullExitWithDelegation` calls `transferDelegationFromPosition` and propagates any error: [2](#0-1) 

`transferDelegationFromPosition` contains a "Defensive" guard that queries the staking module for active redelegations on the position's delegator address and returns `ErrActiveRedelegation` if any exist: [3](#0-2) 

`isRedelegating` queries the live staking state directly: [4](#0-3) 

The precondition — a vesting account owning a position with an active redelegation — is reachable because:

- `validateNonVestingAccount` blocks new position creation by vesting accounts, but this check was added at some point; the migration itself proves vesting-owned positions exist in pre-v8 state.
- `validateRedelegatePosition` does **not** call `validateNonVestingAccount`, so a vesting account that already owns a position can call `TierRedelegate` freely: [5](#0-4) 

- `TierRedelegate` records the redelegation in the staking module and sets a `RedelegationMappings` entry, but the staking-layer redelegation entry persists until the unbonding period elapses: [6](#0-5) 

If the upgrade is executed while the redelegation's unbonding period has not elapsed, `isRedelegating` returns `true`, `transferDelegationFromPosition` returns `ErrActiveRedelegation`, and the error bubbles all the way up through `Migrate` → `RunMigrations` → upgrade handler → chain halt.

### Impact Explanation
The upgrade handler aborts with an error. The chain halts at the upgrade height. State is partially migrated: positions processed before the failing one have been exited (delegations transferred back to owners, positions deleted), while positions after it remain untouched. This leaves the module in an inconsistent state with orphaned delegations and mismatched position records. Recovery requires a coordinated emergency patch and re-upgrade, which is operationally severe for a production chain.

### Likelihood Explanation
Any vesting account that owned a position before the `validateNonVestingAccount` guard was enforced and subsequently called `TierRedelegate` within the staking unbonding period before the v8 upgrade block creates this condition. The unbonding period on Cronos POS Chain is 28 days. If any such account performed a redelegation within 28 days of the upgrade, the migration fails. This is a realistic production scenario requiring no privileged access — only a normal user transaction (`MsgTierRedelegate`) from a vesting account that already holds a position.

### Recommendation
In `ForceFullExitWithDelegation` (or in `exitVestedAccountsPositions`), detect the active-redelegation case and handle it explicitly rather than propagating the error. Options:

1. **Skip and log**: If `isRedelegating` returns true, log a warning and skip the position for now; schedule a follow-up cleanup hook (e.g., via `AfterRedelegationModified`).
2. **Wait for completion**: Defer the exit to an `EndBlock` hook that fires once the redelegation completes.
3. **Remove the guard in the migration path**: Since `ForceFullExitWithDelegation` is a one-time migration function (marked "Can be deleted after v8 upgrade"), it can bypass the `isRedelegating` defensive check and directly unbond the position's shares (the staking module allows unbonding a redelegating delegation — it just creates an unbonding delegation entry instead of an instant transfer).

func (k Keeper) validateRedelegatePosition(ctx context.Context, pos types.PositionState, owner, dstValidator string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if pos.Delegation.ValidatorAddress == dstValidator {
		return types.ErrRedelegationToSameValidator
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if pos.HasTriggeredExit() && pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationElapsed
	}

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return err
	}

	if tier.IsCloseOnly() {
		return types.ErrTierIsCloseOnly
	}

	isRedelegating, err := k.isRedelegating(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isRedelegating {
		return errorsmod.Wrapf(types.ErrActiveRedelegation, "position %d has an active redelegation", pos.Id)
	}

	return nil
}
