### Title
NLP Pool Subaccount Owner Cannot Sign Withdrawals or Transfers Due to Incomplete Signer Variant Handling — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

Nado has two signer variants for subaccounts: `linkedSigners` (regular linked signer) and `nlpSigners` (NLP pool owner signer). A unified getter `getLinkedSignerOrNlpSigner` exists and is correctly used for order matching. However, `validateSignature` and `validateCompactSignature` only call `getLinkedSigner`, which only checks `linkedSigners`. This means the NLP pool owner — whose authority is registered in `nlpSigners` — cannot authorize `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, or `LinkSigner` transactions for the NLP pool subaccount. Since NLP pool subaccounts accumulate collateral through rebalancing and trading, funds can become permanently stuck.

---

### Finding Description

When an NLP pool is created or updated, the pool owner's address is stored in `nlpSigners[poolSubaccount]`: [1](#0-0) [2](#0-1) 

The protocol defines a unified getter that correctly resolves either signer variant: [3](#0-2) 

This unified getter is used for `MatchOrders` and `MatchOrdersWithAmount`: [4](#0-3) 

However, `validateSignature` and `validateCompactSignature` — which gate all other signed transactions — only call `getLinkedSigner`, which only checks `linkedSigners` and never consults `nlpSigners`: [5](#0-4) [6](#0-5) 

These two functions are called by `validateSignedTx`, which gates:
- `WithdrawCollateral` (line 418)
- `WithdrawCollateralV2` (line 442)
- `TransferQuote` (line 599)
- `LinkSigner` (line 581)
- `MintNlp` / `BurnNlp` (lines 539, 559) [7](#0-6) 

NLP pool subaccounts are special addresses derived from `NLP_POOL_ACCOUNT_START = 0x...0100000000`: [8](#0-7) 

The address portion (first 20 bytes) of these subaccounts is `address(0)`, not the pool owner's address. Therefore the pool owner cannot sign as the subaccount owner directly. The slow-mode `LinkSigner` path also cannot help, because it uses `validateSender` which checks `address(uint160(bytes20(txSender))) == sender` — a check the pool owner's address will never satisfy for an NLP pool subaccount. [9](#0-8) 

NLP pool subaccounts accumulate collateral through `_applyNlpRebalance` in `Clearinghouse.sol`, which directly credits quote balances to pool subaccounts: [10](#0-9) 

---

### Impact Explanation

The NLP pool owner, whose authority is registered in `nlpSigners`, cannot sign `WithdrawCollateral` or `WithdrawCollateralV2` for the NLP pool subaccount. Since the pool subaccount accumulates quote collateral through rebalancing and trading, and there is no alternative path for the pool owner to authorize a withdrawal (the subaccount address is not user-controlled, and slow-mode `LinkSigner` is also blocked), collateral can become permanently stuck in the NLP pool subaccount. The corrupted state is the quote balance in `SpotEngine` for the NLP pool subaccount that cannot be extracted by its authorized owner.

---

### Likelihood Explanation

NLP pool subaccounts are active protocol participants — they receive collateral via `_applyNlpRebalance` and participate in order matching. Any residual quote balance after pool operations (e.g., after `deleteNlpPool` which calls `clearNlpPoolPosition` but does not withdraw remaining spot balances) would be irrecoverable. The pool owner has no privileged path to work around this. Likelihood is **medium**: it requires an NLP pool to have a non-zero spot balance outside of the NLP token product, which is a realistic operational state.

---

### Recommendation

Replace `getLinkedSigner` with `getLinkedSignerOrNlpSigner` in both `validateSignature` and `validateCompactSignature`:

```solidity
// EndpointTx.sol

function validateSignature(...) internal virtual {
    verifier.validateSignature(
        sender,
-       allowLinkedSigner ? getLinkedSigner(sender) : address(0),
+       allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
        digest,
        signature
    );
}

function validateCompactSignature(...) internal virtual {
    verifier.validateCompactSignature(
        sender,
-       allowLinkedSigner ? getLinkedSigner(sender) : address(0),
+       allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
        digest,
        signature
    );
}
```

This mirrors the fix applied in the Lombard report: merge the two signer-resolution paths into one unified function that checks both variants.

---

### Proof of Concept

1. Admin calls `AddNlpPool` slow-mode transaction, creating NLP pool with `poolId = 1`, `owner = alice`. This sets `nlpSigners[poolSubaccount] = alice`.
2. The sequencer submits `MatchOrders` transactions where the NLP pool subaccount acts as maker. Trading profits accumulate as quote balance in `SpotEngine` for `poolSubaccount`. This works because `MatchOrders` uses `getLinkedSignerOrNlpSigner`.
3. Alice attempts to withdraw the accumulated quote by submitting a `WithdrawCollateral` transaction signed with her key, with `sender = poolSubaccount`.
4. The sequencer calls `processTransactionImpl` → `validateSignedTx` → `validateSignature`.
5. `validateSignature` calls `getLinkedSigner(poolSubaccount)`, which returns `linkedSigners[poolSubaccount]` = `address(0)` (never set).
6. `verifier.validateSignature` receives `linkedSigner = address(0)` and checks `recovered == address(uint160(bytes20(poolSubaccount)))` (= `address(0)`) or `recovered == address(0)`. Both fail for Alice's valid signature.
7. Transaction reverts with `ERR_INVALID_SIGNATURE`. The quote balance remains stuck in `poolSubaccount` with no recovery path. [11](#0-10)

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
