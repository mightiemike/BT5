### Title
Caller-Supplied `tx_id` and `tx_index` Enable Merkle Proof Forgery in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function accepts `tx_id`, `tx_index`, and `merkle_proof` entirely from the caller at verification time, with no prior commitment to any of these values. An unprivileged NEAR caller can pass an internal Merkle tree node hash as `tx_id` with a crafted `tx_index` and `merkle_proof` to make the function return `true` for a Bitcoin transaction that does not exist in the block. This is a direct analog to the external report's commitment-scheme weakness: just as the market direction was revealed at exercise time rather than committed to at open time, the proof inputs here are revealed at verification time rather than committed to at block-submission time.

---

### Finding Description

When a block is submitted via `submit_blocks`, the contract stores the block's `merkle_root` inside the `ExtendedHeader` in `headers_pool`. This is the only commitment made at submission time. [1](#0-0) 

At verification time, `verify_transaction_inclusion` accepts `tx_id`, `tx_index`, and `merkle_proof` entirely from the caller: [2](#0-1) 

The function delegates to `compute_root_from_merkle_proof`, which uses `tx_index` to determine the left/right hashing direction at each Merkle tree level: [3](#0-2) 

Because `tx_id` is accepted as an arbitrary 32-byte hash with no validation that it corresponds to a real leaf (transaction), an attacker can supply an **internal Merkle node hash** as `tx_id`. By choosing `tx_index` and `merkle_proof` to match the internal node's position in the tree, the computed root will equal the stored `merkle_root`, and the function returns `true` for a non-existent transaction.

The contract itself documents this broken invariant but defers enforcement to callers: [4](#0-3) 

The `#[deprecated]` attribute only produces a Rust compiler warning; the function remains fully callable at runtime by any unprivileged NEAR account.

---

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` and trusts its `true` result as proof of a real Bitcoin transaction inclusion is vulnerable to forgery. An attacker can claim that an arbitrary internal Merkle node hash represents a confirmed Bitcoin transaction in a legitimately submitted block. Downstream contracts (e.g., bridge contracts, payment verifiers, cross-chain settlement systems) that act on this result — releasing funds, minting tokens, or updating state — would do so based on a fabricated proof.

The corrupted proof result is concrete: `verify_transaction_inclusion` returns `true` for a `tx_id` that is an internal Merkle node, not a valid transaction hash, against a legitimately stored block.

---

### Likelihood Explanation

- The function is reachable by **any unprivileged NEAR caller** — no role, stake, or permission is required.
- Bitcoin block Merkle trees are **publicly available** from any Bitcoin node; the attacker can compute all internal node hashes offline before crafting the call.
- The attack requires only that the target block has ≥4 transactions (virtually all real Bitcoin blocks do).
- The `ProofArgs` struct is JSON-serializable, making the call trivially constructable. [5](#0-4) 

---

### Recommendation

Remove `verify_transaction_inclusion` entirely from the public API rather than leaving it callable with a deprecation notice. All callers should be migrated to `verify_transaction_inclusion_v2`, which adds a coinbase Merkle proof check that substantially raises the bar for forgery. [6](#0-5) 

If removal is not immediately feasible, add a runtime `env::panic_str` or `require!(false, "removed")` guard inside `verify_transaction_inclusion` so that on-chain calls revert unconditionally, making the deprecation enforceable at the protocol level rather than only at compile time.

---

### Proof of Concept

Assume a legitimately submitted block contains 4 transactions with hashes `A`, `B`, `C`, `D`. The Merkle tree is:

```
Level 1:  H_AB = hash(A, B)     H_CD = hash(C, D)
Level 0:  root = hash(H_AB, H_CD)
```

The attacker calls `verify_transaction_inclusion` with:

```json
{
  "tx_id":              "<H_AB>",
  "tx_block_blockhash": "<block_hash>",
  "tx_index":           0,
  "merkle_proof":       ["<H_CD>"],
  "confirmations":      1
}
```

`compute_root_from_merkle_proof(H_AB, 0, [H_CD])` computes:
- position 0 is even → `hash(H_AB, H_CD)` = `root` ✓

The function returns `true`. `H_AB` is an internal node, not a real transaction. The broken invariant — that a `true` result certifies a valid transaction — is violated. [7](#0-6) [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
