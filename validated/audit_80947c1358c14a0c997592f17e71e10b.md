### Title
Duplicate Position IDs in `ClaimTierRewards` Enable Double Bonus Reward Drain - (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`MsgClaimTierRewards` loads all requested positions into an in-memory slice before processing. No deduplication is performed on the caller-supplied `PositionIds`. When the same position ID appears twice, the second iteration operates on a stale in-memory snapshot of the position while reading live validator-event state from the store. If the position's validator has a per-event reference count greater than one (i.e., at least one other position is also delegated to the same validator), the events still exist after the first iteration's decrements and are processed a second time, causing `bankKeeper.SendCoinsFromModuleToAccount` to transfer bonus coins from `RewardsPoolName` twice. The final `setPosition` call then persists the stale snapshot, corrupting the on-chain position state.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` builds a `positions` slice by iterating over the caller-supplied `msg.PositionIds` and loading each position from the store:

```go
// x/tieredrewards/keeper/msg_server.go  lines 434-446
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
```

There is no deduplication check. If `msg.PositionIds = [P, P]`, both `positions[0]` and `positions[1]` hold the same initial state snapshot loaded from the store.

`claimRewardsAndUpdatesPositions` then iterates over this slice:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 110-134
for i := range positions {
    pos := &positions[i]
    ...
    base, err := k.claimBaseRewards(ctx, *pos)
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
```

**First iteration (i=0):** `processEventsAndClaimBonus` reads validator events from the store since `pos.LastEventSeq`, sends bonus coins via `bankKeeper.SendCoinsFromModuleToAccount`, and calls `decrementEventRefCount` for each event. If the validator has N positions delegated to it, each event's reference count drops from N to N-1. When N ≥ 2, events survive. `setPosition` persists the updated `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded` to the store.

**Second iteration (i=1):** `pos` now points to `positions[1]`, which still holds the **original stale snapshot** (old `LastEventSeq`, old `LastBonusAccrual`). `processEventsAndClaimBonus` calls `getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)` using the old sequence number. Because the events still exist in the store (ref count N-1 ≥ 1), the same events are returned and processed again. `bankKeeper.SendCoinsFromModuleToAccount` fires a second time, draining additional bonus coins from `RewardsPoolName`. `setPosition` then writes the stale snapshot back to the store, overwriting the correctly updated state from the first iteration.

The exact coin transfer that fires twice:

```go
// x/tieredrewards/keeper/claim_rewards.go  line 239
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
```

The state that is corrupted by the second `setPosition`:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 215-217
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
pos.UpdateLastKnownBonded(bonded)
```

---

### Impact Explanation

An attacker who owns a position on any validator that has at least one other tiered-rewards position (a common condition on a live network) can drain double the owed bonus rewards from the `RewardsPoolName` module account in a single transaction. The final `setPosition` call also corrupts the on-chain position state, resetting `LastEventSeq` and `LastBonusAccrual` to stale values, which can enable repeated exploitation across future blocks until the event reference counts are exhausted.

The corrupted value is the `RewardsPoolName` module account balance and the persisted `PositionState` fields `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`.

---

### Likelihood Explanation

The trigger requires only a single `MsgClaimTierRewards` transaction with a repeated position ID, which any position owner can construct. The precondition—another position delegated to the same validator—is satisfied by default on any active network where multiple users participate in tiered rewards on the same validator. No privileged access, leaked keys, or social engineering is required.

---

### Recommendation

Deduplicate `msg.PositionIds` before building the `positions` slice in `ClaimTierRewards`. A simple approach is to use a `map[uint64]struct{}` to detect and reject (or skip) duplicate IDs:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

Alternatively, apply the check-effects-interactions ordering strictly: persist the updated position state to the store **before** calling `bankKeeper.SendCoinsFromModuleToAccount`, so that any re-use of the same position within the same call reads the already-updated state.

---

### Proof of Concept

1. Attacker owns position `P` (ID = 42) delegated to validator `V`. At least one other position `Q` is also delegated to `V` (ref count per event ≥ 2).
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42]}`.
3. `ClaimTierRewards` loads position 42 twice into `positions[0]` and `positions[1]`, both with `LastEventSeq = S`.
4. First loop iteration: events since `S` are fetched, bonus `B` coins sent to attacker, each event ref count decremented (2 → 1, events survive), position persisted with `LastEventSeq = S'`.
5. Second loop iteration: `positions[1]` still has `LastEventSeq = S`. Events since `S` are fetched again (ref count 1, still present), bonus `B` coins sent to attacker **again**, each event ref count decremented (1 → 0, events deleted), position persisted with stale `LastEventSeq = S` (overwriting the correct `S'`).
6. Attacker receives `2B` bonus coins instead of `B`, draining an extra `B` from `RewardsPoolName`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
