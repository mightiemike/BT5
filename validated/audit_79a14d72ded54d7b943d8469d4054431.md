### Title
SPV Proof Verification Permanently Blocked During Pause With No Bypass Role — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are decorated with `#[pause]` but have no `except(roles(...))` bypass. In contrast, every other pausable function in the contract (`submit_blocks`, `run_mainchain_gc`) was given an explicit privileged bypass role. When the contract is paused, all SPV proof verification is completely blocked for every caller — including `DAO` — while block submission and GC can still proceed for privileged accounts. Any downstream bridge or cross-chain application with a time-sensitive claim window that depends on `verify_transaction_inclusion_v2` will be permanently denied during a pause, causing irreversible loss of funds.

### Finding Description

The contract uses `near_plugins`' `#[pause]` macro to gate functions. The design explicitly provides bypass roles for operational functions:

- `submit_blocks` carries `#[pause]` but is also annotated `#[trusted_relayer]` with `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`, so privileged relayers can still submit headers while paused. [1](#0-0) 
- `run_mainchain_gc` uses `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, giving a dedicated bypass role. [2](#0-1) 

However, both verification functions use a bare `#[pause]` with no `except` clause and no bypass role:

```rust
#[pause]
pub fn verify_transaction_inclusion(&self, ...) -> bool { ... }

#[pause]
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool { ... }
``` [3](#0-2) [4](#0-3) 

This means when the contract is paused, the entire SPV proof verification API is hard-blocked for all callers with no escape hatch — not even `DAO`. The asymmetry is the root cause: the contract designers added bypass roles for `submit_blocks` and `run_mainchain_gc` but omitted them for the verification functions.

Additionally, `verify_transaction_inclusion_v2` internally calls `verify_transaction_inclusion` (which is itself `#[pause]`-gated), compounding the blockage:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [5](#0-4) 

### Impact Explanation

The BTC light client is the trust anchor for cross-chain Bitcoin SPV proofs on NEAR. Downstream bridge contracts and cross-chain applications call `verify_transaction_inclusion_v2` to confirm Bitcoin deposits before releasing funds. Many such protocols enforce time-bounded claim windows (e.g., HTLCs, atomic swaps, bridge escrows with expiry). If the light client is paused during such a window — even briefly for routine maintenance — the user's proof call reverts unconditionally, the window expires, and the funds are permanently unclaimable. The corrupted invariant is: **a valid, confirmed Bitcoin transaction with a valid Merkle proof returns a forced failure instead of `true`**, causing downstream contracts to treat a legitimate deposit as non-existent.

### Likelihood Explanation

The contract is explicitly designed to be pausable for maintenance and security incidents. The `PauseManager` role exists precisely for this purpose. [6](#0-5)  Any pause event — however brief — triggers the vulnerability for all in-flight proof verifications. The likelihood is directly proportional to how often the contract is paused, which is a normal operational event.

### Recommendation

Add an `except(roles(...))` bypass to both verification functions, consistent with the pattern already used for `run_mainchain_gc`. A dedicated `UnrestrictedVerify` role (or reuse `Role::DAO`) should be added:

```rust
// Before (vulnerable):
#[pause]
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool { ... }

// After (fixed):
#[pause(except(roles(Role::DAO, Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion_v2(&self, ...) -> bool { ... }
```

Alternatively, since `verify_transaction_inclusion_v2` is a read-only view function that does not mutate state, consider whether it needs to be paused at all. Pausing a pure verification function provides no security benefit (it cannot be used to attack the contract state) while actively harming downstream users.

### Proof of Concept

1. `PauseManager` pauses the contract (routine maintenance).
2. Alice has a Bitcoin deposit confirmed on-chain. She calls `verify_transaction_inclusion_v2` on the NEAR light client to claim her bridged funds.
3. The `#[pause]` macro fires unconditionally — no bypass role exists — and the call panics/reverts.
4. Alice's bridge escrow has a 24-hour claim window. The pause lasts 25 hours.
5. The escrow expires. Alice's funds are permanently locked.
6. Meanwhile, a privileged relayer with `UnrestrictedSubmitBlocks` continues submitting new Bitcoin headers successfully — demonstrating the asymmetry: block submission works while proof verification is completely dead.

The attacker-controlled entry path is simply calling `verify_transaction_inclusion_v2` as any unprivileged NEAR account while the contract is paused. No special role, no leaked key, no social engineering — the pause state alone is sufficient to trigger the denial.
