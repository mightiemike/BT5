### Title
Linked Signer Can Hijack Subaccount Signing Authority by Self-Issuing a `LinkSigner` Transaction — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In Nado's fast-mode transaction processing, a `LinkSigner` transaction is validated with `allowLinkedSigner = true`. This means the **current linked signer** of a subaccount can sign a `LinkSigner` transaction to replace themselves with any arbitrary address — including an attacker-controlled one — without the subaccount owner's knowledge or consent. This is a direct analog to the reported privilege escalation: a limited-privilege entity uses its existing authority to permanently redirect that authority to a more powerful or attacker-controlled identity.

---

### Finding Description

When the sequencer processes a fast-mode `LinkSigner` transaction in `EndpointTx::processTransactionImpl`, the call to `validateSignedTx` passes `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 577–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` calls `validateSignature` which, when `allowLinkedSigner = true`, passes the current linked signer as an accepted signer:

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

`Verifier::validateSignature` accepts the signature if it comes from either the subaccount owner **or** the linked signer:

```solidity
// Verifier.sol lines 297–303
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
```

This means a linked signer can sign a `LinkSigner` transaction pointing `signedTx.tx.signer` to any address — including an attacker-controlled one — and the protocol will accept it, overwriting `linkedSigners[victim_subaccount]` with the attacker's address.

By contrast, the **slow-mode** `LinkSigner` path is safe: it uses `validateSender`, which requires `msg.sender` to be the address embedded in the subaccount bytes (i.e., the actual owner):

```solidity
// EndpointTx.sol lines 232–239
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The fast-mode path has no equivalent owner-only guard.

---

### Impact Explanation

Once the attacker's address is installed as the linked signer, it has full signing authority over the victim's subaccount. It can sign:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain all collateral to an arbitrary `sendTo` address
- `TransferQuote` — transfer quote balance to any registered subaccount
- `LiquidateSubaccount` — force-liquidate the victim's positions
- `MintNlp` / `BurnNlp` — manipulate NLP positions

The subaccount owner has no on-chain mechanism to detect or prevent this before the attacker acts. The nonce is consumed by the malicious `LinkSigner` transaction, so the owner's own pending transactions may be invalidated.

---

### Likelihood Explanation

Linked signers are a core, documented feature of Nado used by API traders and automated strategies. Any user who has set a linked signer is exposed. A malicious linked signer (e.g., a compromised API key, a malicious third-party integration, or a deliberately crafted attack) can exploit this with a single fast-mode transaction submitted through the sequencer. No special privileges beyond being the current linked signer are required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-mode `LinkSigner` transaction processing, so that only the subaccount owner's key can authorize a change to the linked signer:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
```

This aligns the fast-mode path with the security model already enforced by the slow-mode path.

---

### Proof of Concept

1. Victim (`0xVICTIM`) has a subaccount `victimSubaccount` and sets linked signer to `0xMALICIOUS` via a legitimate `LinkSigner` transaction.
2. `0xMALICIOUS` constructs a `LinkSigner` transaction: `{ sender: victimSubaccount, signer: bytes32(uint256(uint160(0xATTACKER))), nonce: currentNonce }`.
3. `0xMALICIOUS` signs this transaction with their own key and submits it to the sequencer.
4. The sequencer includes it in a batch. `processTransactionImpl` calls `validateSignedTx(..., true)`. `Verifier.validateSignature` accepts the signature because `recovered == linkedSigner (0xMALICIOUS)`.
5. `linkedSigners[victimSubaccount]` is now set to `0xATTACKER`.
6. `0xATTACKER` signs a `WithdrawCollateralV2` transaction with `sendTo = 0xATTACKER`, draining the victim's collateral. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
