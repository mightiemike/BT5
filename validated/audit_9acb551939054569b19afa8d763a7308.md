### Title
Deprecated `verify_transaction_inclusion` Accepts Internal Merkle Node as `tx_id`, Enabling SPV Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains callable by any unprivileged NEAR account despite being deprecated. It does not verify that the caller-supplied `tx_id` is a leaf-level transaction hash rather than an internal Merkle tree node. An attacker can supply an internal node hash with a shortened proof path, causing the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` accepts caller-controlled `tx_id`, `tx_index`, and `merkle_proof`, then delegates directly to `merkle_tools::compute_root_from_merkle_proof` and compares the result to the stored `header.block_header.merkle_root`: [1](#0-0) 

The function performs no check that `tx_id` is a leaf node (a real transaction hash) rather than an internal node of the Merkle tree. Because Bitcoin block headers store only the Merkle root and not the transaction count, the contract cannot determine the expected proof depth: [2](#0-1) 

`compute_root_from_merkle_proof` simply iterates over the supplied proof elements and hashes upward — it has no concept of tree depth or leaf vs. internal node: [3](#0-2) 

The function is marked `#[deprecated(since = "0.5.0")]`, but Rust's `#[deprecated]` attribute only emits a compiler warning. It does **not** prevent the function from being called at runtime. The function is still exposed as a public NEAR contract method, gated only by `#[pause]` (which is off by default): [4](#0-3) 

The analog to the external report is direct: just as the Babylon CLI accepted unverified covenant signatures into the quorum without checking their correctness, this contract accepts an unverified `tx_id` into its proof computation without checking whether it is a real transaction — and in both cases the verification function produces a false affirmative result that downstream consumers trust.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` and trusts its `bool` result will accept forged transaction inclusion proofs. An attacker can claim that an arbitrary internal Merkle node represents a confirmed Bitcoin transaction, and the function returns `true`. This breaks the core security guarantee of the SPV light client: that a `true` result means a specific transaction was confirmed on Bitcoin. Contracts that gate fund releases, cross-chain state transitions, or other irreversible actions on this result are directly exploitable.

---

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_transaction_inclusion` directly. The attacker only needs public information: the Merkle tree structure of any Bitcoin block already stored in the contract. No private keys, special roles, or social engineering are required. The attack is fully deterministic and requires no brute force.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public interface entirely, or gate it behind a privileged role so unprivileged callers cannot reach it. All callers must be migrated to `verify_transaction_inclusion_v2`, which enforces a coinbase proof of equal depth to bound the tree level and mitigate the second-preimage attack: [5](#0-4) 

---

### Proof of Concept

Given a block with 4 transactions `[tx0, tx1, tx2, tx3]` already stored in the contract:

1. Compute internal node: `N01 = SHA256d(tx0 || tx1)`
2. Compute internal node: `N23 = SHA256d(tx2 || tx3)`
3. Verify: `SHA256d(N01 || N23) == merkle_root` — true by construction
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N01` (an internal node, not a real transaction)
   - `tx_index = 0`
   - `merkle_proof = [N23]` (one element — one level shorter than a real leaf proof)
   - `tx_block_blockhash = <any confirmed block hash in the contract>`
   - `confirmations = 0`
5. `compute_root_from_merkle_proof(N01, 0, [N23])` = `SHA256d(N01 || N23)` = `merkle_root`
6. The function returns `true`, falsely asserting that `N01` is a confirmed Bitcoin transaction. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
