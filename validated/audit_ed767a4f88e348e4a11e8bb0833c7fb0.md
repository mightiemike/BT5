### Title
Borrow Rate Exceeds Implicit Cap When Utilization Surpasses 100% — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState._updateState()` computes `borrowerRateX18` using a two-segment piecewise formula. The above-inflection segment multiplies `interestLargeCapX18` by a ratio that is only bounded between 0 and 1 when `utilizationRatioX18 ≤ ONE`. Because the 20% `INTEREST_FEE_FRACTION` causes `totalBorrows` to compound faster than `totalDeposits`, utilization can drift above 100% over time without any guard in `_updateState`, making the ratio exceed 1 and the computed borrow rate exceed the intended cap of `interestFloorX18 + interestSmallCapX18 + interestLargeCapX18`.

---

### Finding Description

In `_updateState()`, `utilizationRatioX18` is computed as:

```solidity
int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
```

The above-inflection branch then computes:

```solidity
borrowerRateX18 +=
    config.interestSmallCapX18 +
    config.interestLargeCapX18.mul(
        (utilizationRatioX18 - config.interestInflectionUtilX18)
            .div(ONE - config.interestInflectionUtilX18)
    );
```

The ratio `(utilizationRatioX18 − interestInflectionUtilX18) / (ONE − interestInflectionUtilX18)` is designed to range from 0 to 1 as utilization moves from `interestInflectionUtilX18` to `ONE` (100%). At exactly 100% utilization the rate equals the intended maximum: `interestFloorX18 + interestSmallCapX18 + interestLargeCapX18`.

However, there is no clamp on `utilizationRatioX18` before this formula is applied. The protocol charges borrowers at `borrowRateMultiplierX18` but credits depositors at a strictly lower `depositRateMultiplierX18` because `INTEREST_FEE_FRACTION = 20%` is deducted:

```solidity
int128 realizedDepositRateX18 = totalDepositRateX18.mul(ONE - INTEREST_FEE_FRACTION);
```

This means `totalBorrows` (in actual terms) compounds faster than `totalDeposits` every time `_updateState` runs. Over time, `totalBorrows > totalDeposits`, so `utilizationRatioX18 > ONE`. The only utilization guard in the contract is `assertUtilization()`, which is explicitly scoped to withdrawals only and is never called inside `_updateState`.

With the default QUOTE product configuration (`interestInflectionUtilX18 = 0.8e18`, `interestLargeCapX18 = ONE`), once `utilizationRatioX18 = 1.1e18`:

```
ratio = (1.1e18 − 0.8e18) / (1e18 − 0.8e18) = 0.3e18 / 0.2e18 = 1.5
borrowerRateX18 = 0.01 + 0.04 + 1.0 × 1.5 = 1.55  (155% APR)
```

instead of the intended cap of `0.01 + 0.04 + 1.0 = 1.05` (105% APR).

---

### Impact Explanation

An inflated `borrowerRateX18` feeds directly into `borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(dt)`, which is then applied to `cumulativeBorrowsMultiplierX18`. Every subsequent balance lookup via `balanceNormalizedToBalance` multiplies a borrower's `amountNormalized` by this inflated cumulative multiplier, overstating their outstanding debt. Health checks in the Clearinghouse consume these overstated balances, making borrowers appear undercollateralized when they are not, and enabling liquidators to liquidate positions that should be healthy under the intended rate cap. The corrupted `cumulativeBorrowsMultiplierX18` is a persistent on-chain state delta that compounds with every future `_updateState` call.

---

### Likelihood Explanation

The drift is deterministic and passive: it requires no attacker action. Any spot product running at high utilization (near 100%) will see `totalBorrows` outpace `totalDeposits` due to the 20% fee fraction. The QUOTE product is initialized with `interestLargeCapX18 = ONE` (100% APR at full utilization), so the fee-driven divergence is significant. The condition `utilizationRatioX18 > ONE` is reachable in normal protocol operation over weeks to months at sustained high utilization, and `_updateState` is triggered on every sequencer-submitted transaction batch.

---

### Recommendation

Cap `utilizationRatioX18` to `ONE` before it enters the rate formula in `_updateState`, or add an explicit clamp on `borrowerRateX18` after the piecewise computation:

```solidity
// Option A: clamp utilization input
int128 effectiveUtil = utilizationRatioX18 > ONE ? ONE : utilizationRatioX18;
// use effectiveUtil in the piecewise branches

// Option B: clamp output
int128 maxRate = config.interestFloorX18 + config.interestSmallCapX18 + config.interestLargeCapX18;
if (borrowerRateX18 > maxRate) borrowerRateX18 = maxRate;
```

Either approach ensures `borrowerRateX18` never exceeds the intended cap regardless of how far utilization drifts above 100%.

---

### Proof of Concept

1. A spot product is initialized with the default QUOTE config: `interestInflectionUtilX18 = 0.8e18`, `interestFloorX18 = 0.01e18`, `interestSmallCapX18 = 0.04e18`, `interestLargeCapX18 = 1e18`.
2. Utilization reaches ~99% through normal borrowing activity.
3. The sequencer submits transaction batches over time; each call to `_updateState` applies `borrowRateMultiplierX18 > depositRateMultiplierX18` (due to `INTEREST_FEE_FRACTION = 20%`), causing `totalBorrows` to grow faster than `totalDeposits`.
4. After sufficient time, `totalBorrows > totalDeposits`, so `utilizationRatioX18 > ONE`. `assertUtilization` is never called during this process — it is only invoked on withdrawals.
5. On the next `_updateState` call, the `else` branch executes with `utilizationRatioX18 = 1.1e18`: the ratio `(1.1e18 − 0.8e18) / (1e18 − 0.8e18) = 1.5`, yielding `borrowerRateX18 = 1.55` (155% APR) instead of the capped 105% APR.
6. `cumulativeBorrowsMultiplierX18` is inflated; all borrower balances resolved via `balanceNormalizedToBalance` are overstated; Clearinghouse health checks flag solvent borrowers as undercollateralized; liquidators can liquidate them at a penalty.
