### Title
Incorrect Non-Empty Guard Blocks Valid SPV Proof for Single-Transaction Blocks — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains a `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that unconditionally panics when the caller supplies an empty Merkle proof. For a Bitcoin block containing exactly one transaction (the coinbase), the Merkle proof is legitimately empty — the transaction hash *is* the Merkle root. The underlying `compute_root_from_merkle_proof` handles this correctly, but the guard fires first and aborts execution, making it impossible to verify any transaction in a single-transaction block. `verify_transaction_inclusion_v2` (the current recommended API) is also broken because it delegates to the deprecated function internally.

---

### Finding Description

In `contract/src/lib.rs`, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The underlying `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over the proof slice. When the slice is empty the loop body never executes and the function returns `current_hash` unchanged — i.e., it returns `transaction_hash` directly:

```rust
for proof_hash in merkle_proof {
    // ...
}
current_hash   // == transaction_hash when proof is empty
``` [2](#0-1) 

For a block with a single transaction, `merkle_root == transaction_hash` by Bitcoin consensus rules. Passing an empty proof and `tx_index = 0` would therefore produce the correct result — but the `require!` guard fires before `compute_root_from_merkle_proof` is ever called, causing an unconditional panic.

`verify_transaction_inclusion_v2` (the non-deprecated entry point) calls `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())` after its own coinbase-proof check, so it inherits the same defect: [3](#0-2) 

---

### Impact Explanation

Any downstream NEAR contract or user that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to prove a transaction in a single-transaction Bitcoin block will receive a panic (transaction abort) instead of a `true` result. The light client's core SPV verification function is therefore unavailable for a well-defined, consensus-valid class of Bitcoin blocks. Bridges or applications that rely on this proof path for single-transaction blocks cannot complete their verification flow.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are a normal part of Bitcoin's history and continue to occur during periods of low mempool activity. The condition is deterministic and reproducible: any caller who knows the block has one transaction and constructs the correct (empty) proof will always trigger the panic. No special privileges or adversarial chain data are required — an ordinary unprivileged NEAR caller invoking the public `verify_transaction_inclusion_v2` method is sufficient.

---

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. The `compute_root_from_merkle_proof` function already handles the empty-proof case correctly by returning the transaction hash unchanged, which will equal the Merkle root for a single-transaction block and produce the correct boolean result without any special-casing. [4](#0-3) 

---

### Proof of Concept

1. A Bitcoin block at any height contains exactly one transaction (the coinbase), so `merkle_root = coinbase_txid`.
2. Caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_txid`
   - `tx_block_blockhash` = hash of that block (already in the contract's `headers_pool`)
   - `tx_index = 0`
   - `merkle_proof = []` (empty — correct for a single-transaction block)
   - `coinbase_tx_id = coinbase_txid`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. The coinbase-proof check in `verify_transaction_inclusion_v2` passes: `compute_root_from_merkle_proof(coinbase_txid, 0, [])` returns `coinbase_txid == merkle_root`. ✓
4. Execution reaches `self.verify_transaction_inclusion(args.into())`.
5. Inside `verify_transaction_inclusion`, line 315 fires: `require!(!args.merkle_proof.is_empty())` → **panic**, transaction aborted.
6. Expected result: `true`. Actual result: contract panic. [5](#0-4) [6](#0-5)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,

        );

            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
