Audit Report

## Title
`SwapAllowlistExtension` gates on router address instead of originating user, enabling allowlist bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` at the time `pool.swap()` is called. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. A pool admin who allowlists the router to support normal UX inadvertently grants every public user the ability to bypass the allowlist guard entirely.

## Finding Description
In `MetricOmmPool.swap`, `msg.sender` is captured and forwarded as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` passes this value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router: [4](#0-3) 

The check resolves to `allowedSwapper[pool][router]`. If the router is allowlisted (required for any user to trade via the router), every non-allowlisted address can bypass the guard by calling `exactInputSingle` or `exactInput`. The router stores the real payer in transient storage via `_setNextCallbackContext` but this value is never surfaced to the extension.

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict trading (e.g., KYC'd counterparties, whitelisted market makers) loses its access control entirely once the router is allowlisted. Any unprivileged address can execute swaps against the pool's liquidity by routing through the public router, enabling unauthorized arbitrage or directional trading that drains LP value — a direct loss of user principal and a broken core pool access-control invariant.

## Likelihood Explanation
The bypass requires a single public call to the canonical periphery router with no special privileges, flash loans, or multi-step setup. The router is a documented, supported entrypoint. Any user who discovers the pool is allowlist-gated can immediately attempt the router path. The condition (router allowlisted) is the expected operational state for any pool that wants to support normal UX.

## Recommendation
The extension must gate on the originating user, not the intermediate caller. Options:

1. **Pass the true initiator through the hook.** The router already stores `msg.sender` as the payer in transient storage (`_setNextCallbackContext(..., msg.sender, ...)`). The pool or router should surface this as a dedicated `initiator` argument in the swap hook so extensions can gate on it.
2. **Check `recipient` as a proxy** if the pool design guarantees the recipient is always the originating user (not always true for multi-hop paths).
3. **Short-term mitigation:** Document that pools using `SwapAllowlistExtension` must not allowlist the router and must require all allowlisted users to call the pool directly — this preserves the guard at the cost of router UX.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, router, true)
    so legitimate users can trade via router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(...) — pool sees msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes — attacker trades on the curated pool without being allowlisted

Result: attacker bypasses the swap allowlist and executes an unauthorized swap.
```
