### Title
Silent Acceptance of Trailing Bytes in `node_from_bytes` Enables Non-Canonical Program Deserialization — (File: `src/serde/de.rs`)

---

### Summary

`node_from_bytes` parses exactly one CLVM object from a byte slice but never verifies that all bytes in the slice were consumed. Trailing bytes are silently discarded. This function is called directly in `run_serialized_chia_program` — the primary Python-facing consensus entry point — for both the `program` and `args` inputs, with no post-parse residue check. The codebase already contains `is_canonical_serialization`, which explicitly enforces the trailing-byte invariant, but that guard is never invoked in the deserialization path.

---

### Finding Description

`node_from_bytes` in `src/serde/de.rs` wraps `node_from_stream` without checking whether the cursor reached the end of the buffer:

```rust
// src/serde/de.rs  lines 44-47
pub fn node_from_bytes(allocator: &mut Allocator, b: &[u8]) -> Result<NodePtr> {
    let mut buffer = Cursor::new(b);
    node_from_stream(allocator, &mut buffer)
    // ← NO CHECK: buffer.position() == b.len() as u64
}
```

`node_from_stream` terminates as soon as one complete CLVM object has been parsed; any bytes that follow are left unread and silently dropped.

This function is called in `wheel/src/api.rs` for both inputs to the main execution entry point:

```rust
// wheel/src/api.rs  lines 55-56
let program = node_from_bytes(&mut allocator, program).map_err(eval_to_py)?;
let args    = node_from_bytes(&mut allocator, args).map_err(eval_to_py)?;
```

By contrast, `is_canonical_serialization` in `src/serde/tools.rs` explicitly enforces full consumption:

```rust
// src/serde/tools.rs  line 235
f.get_ref().len() as u64 == f.position()
```

That guard is never invoked in the deserialization path used by `run_serialized_chia_program`.

The analog to the external report is direct:

| External Report | This Repository |
|---|---|
| `tokenIn` passed to external swap | byte slice passed to `node_from_stream` |
| External swap may consume only part of `tokenIn` | `node_from_stream` stops after first valid object |
| No check that all `tokenIn` was spent | No check that `buffer.position() == b.len()` |
| Leftover `tokenIn` silently accumulates | Trailing bytes silently discarded |
| Downstream accounting uses only partial output | Downstream execution uses only parsed prefix |

---

### Impact Explanation

Two distinct byte sequences can produce the same parsed program. For example, `0x80` (NIL) and `0x80 0x00` (NIL followed by a zero byte) both deserialize to the same `NodePtr`. Because the tree hash is computed from the parsed tree structure — not the raw bytes — both blobs yield the same puzzle hash and execute identically under clvm_rs.

If the Chia full node passes puzzle-reveal bytes directly to `run_serialized_chia_program` without first validating canonical serialization, an attacker can submit a spend bundle whose puzzle reveal contains trailing garbage. clvm_rs accepts it; any validator that enforces strict byte consumption rejects it. This is a consensus-divergence vector: the network can fork over a spend that one class of nodes accepts and another rejects.

Additionally, the non-canonical acceptance breaks the bijection between wire bytes and program identity that the rest of the serialization infrastructure (e.g., `is_canonical_serialization`, `serialized_length_from_bytes`) is designed to maintain.

---

### Likelihood Explanation

Medium. The attacker-controlled entry path is direct: `run_serialized_chia_program` is a public Python API that accepts raw bytes. Crafting a blob with trailing bytes requires no special privilege. The actual consensus impact depends on whether the Chia full node pre-validates canonical serialization before invoking clvm_rs; if it does not, the divergence is immediately triggerable. The existence of `is_canonical_serialization` as a separate, uncalled utility suggests the validation gap is real rather than intentional.

---

### Recommendation

Add a trailing-byte check inside `node_from_bytes`:

```rust
pub fn node_from_bytes(allocator: &mut Allocator, b: &[u8]) -> Result<NodePtr> {
    let mut buffer = Cursor::new(b);
    let result = node_from_stream(allocator, &mut buffer)?;
    if buffer.position() != b.len() as u64 {
        return Err(EvalErr::SerializationError);
    }
    Ok(result)
}
```

Apply the same fix to `node_from_bytes_backrefs`. Alternatively, call `is_canonical_serialization` as a pre-flight check in `run_serialized_chia_program` before invoking `node_from_bytes`.

### Proof of Concept

```python
from clvm_rs import run_serialized_chia_program, NO_UNKNOWN_OPS

# 0x80 = NIL (valid, canonical)
# 0x80 0x00 = NIL followed by a trailing zero byte (non-canonical)
canonical     = bytes([0x80])
non_canonical = bytes([0x80, 0x00])   # trailing byte

# Both succeed; trailing byte is silently discarded
cost1, r1 = run_serialized_chia_program(canonical,     canonical,     10**9, NO_UNKNOWN_OPS)
cost2, r2 = run_serialized_chia_program(non_canonical, non_canonical, 10**9, NO_UNKNOWN_OPS)

# Identical results — non-canonical input accepted without error
assert cost1 == cost2
# A strict validator that checks buffer.position() == len would reject the second call.
# If the full node does not pre-validate, the two nodes diverge on the same spend bundle.
```

    let mut counter = 1;
    let mut b = [0; 1];
    while counter > 0 {
        counter -= 1;

            return false;
