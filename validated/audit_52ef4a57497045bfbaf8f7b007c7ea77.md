### Title
Missing `ProtocolFeature` gate for `DeployGlobalContract` / `UseGlobalContract` / `DeterministicStateInit` in `validate_action_with_mode` — (File: `runtime/runtime/src/action_validation.rs`)

---

### Summary

`validate_action_with_mode` dispatches three action variants — `DeployGlobalContract`, `UseGlobalContract`, and `DeterministicStateInit` — to their per-action validators **without calling `require_protocol_feature`**, while the immediately adjacent variants `DelegateV2`, `TransferToGasKey`, and `WithdrawFromGasKey` all call `require_protocol_feature` before proceeding. The function's own comment declares this path "load-bearing for actions emitted by contracts via host functions," making the absent gate a reachable protocol-boundary bypass.

---

### Finding Description

In `validate_action_with_mode`, the match arm for `DelegateV2` explicitly calls:

```rust
require_protocol_feature(ProtocolFeature::DelegateV2, "DelegateV2", current_protocol_version)?;
``` [1](#0-0) 

Similarly, `TransferToGasKey` and `WithdrawFromGasKey` both delegate to helpers that call `require_protocol_feature(ProtocolFeature::GasKeys, ...)`. [2](#0-1) 

By contrast, the three arms for `DeployGlobalContract`, `UseGlobalContract`, and `DeterministicStateInit` call only size/identifier/receiver-derivation validators — none of which check whether `ProtocolFeature::GlobalContracts` or `ProtocolFeature::DeterministicStateInit` is enabled:

```rust
Action::DeployGlobalContract(a) => validate_deploy_global_contract_action(limit_config, a),
Action::UseGlobalContract(a)    => validate_use_global_contract_action(a),
Action::DeterministicStateInit(a) => validate_deterministic_state_init(limit_config, a, receiver)
``` [3](#0-2) 

The comment at the top of `validate_actions_with_mode` is explicit about why this path is the critical gate:

> "those actions create new receipts that never go through tx admission, so on a pre-feature protocol they must be rejected here." [4](#0-3) 

There is no centralized bulk gate for `GlobalContracts` or `DeterministicStateInit` analogous to the `PostQuantumSignatures` gate at the top of `validate_actions_with_mode`. The only centralized gate covers `PostQuantumSignatures`. [5](#0-4) 

`ProtocolFeature::GlobalContracts` and `ProtocolFeature::DeterministicStateInit` are defined as distinct versioned features in `core/primitives-core/src/version.rs`, confirming that a pre-activation protocol version exists against which the gate should fire.

---

### Impact Explanation

A smart contract running on a node whose current protocol version is below the `GlobalContracts` / `DeterministicStateInit` activation version can emit `DeployGlobalContract`, `UseGlobalContract`, or `DeterministicStateInit` actions via host functions. Because `validate_actions_with_mode` is the sole gate for contract-emitted receipts (they bypass tx admission), these actions pass validation and are applied to state. This can:

- Write global-contract or deterministic-account state entries that the protocol has not yet agreed to support, producing divergent trie roots across nodes at different upgrade stages.
- Cause consensus failures at the upgrade boundary when validators disagree on whether such state is valid.
- Permanently corrupt the trie for affected shards if the state is accepted by a supermajority before the discrepancy is detected.

**Severity: High** — protocol-boundary state divergence is consensus-breaking.

---

### Likelihood Explanation

Any deployed smart contract can emit arbitrary action types via the `promise_batch_action_*` host functions. No special privilege is required. The window is the upgrade epoch during which some validators have advanced past the feature activation version and others have not — a normal, recurring condition on NEAR mainnet during every protocol upgrade.

---

### Recommendation

Add `require_protocol_feature` calls in `validate_action_with_mode` for the three ungated variants, mirroring the pattern used for `DelegateV2`:

```rust
Action::DeployGlobalContract(a) => {
    require_protocol_feature(
        ProtocolFeature::GlobalContracts,
        "GlobalContracts",
        current_protocol_version,
    )?;
    validate_deploy_global_contract_action(limit_config, a)
}
Action::UseGlobalContract(a) => {
    require_protocol_feature(
        ProtocolFeature::GlobalContracts,
        "GlobalContracts",
        current_protocol_version,
    )?;
    validate_use_global_contract_action(a)
}
Action::DeterministicStateInit(a) => {
    require_protocol_feature(
        ProtocolFeature::DeterministicStateInit,
        "DeterministicStateInit",
        current_protocol_version,
    )?;
    validate_deterministic_state_init(limit_config, a, receiver)
}
```

Alternatively, add a centralized bulk gate in `validate_actions_with_mode` analogous to the `PostQuantumSignatures` gate, covering all `GlobalContracts`-gated and `DeterministicStateInit`-gated action variants.

---

### Proof of Concept

1. Deploy a WASM contract that, when called, issues `promise_batch_action_deploy_global_contract` (or the equivalent host call for `UseGlobalContract`/`DeterministicStateInit`).
2. Submit a transaction calling that contract on a node running a protocol version **below** `ProtocolFeature::GlobalContracts.protocol_version()`.
3. Observe that `validate_action_with_mode` reaches `validate_deploy_global_contract_action`, which only checks `action.code.len() <= limit_config.max_contract_size` — no feature-version check fires.
4. The receipt is accepted and applied; a global-contract entry is written to the trie at a protocol version that has not activated the feature, producing a state root that diverges from nodes that correctly reject the action. [6](#0-5)

fn validate_deploy_global_contract_action(
    limit_config: &LimitConfig,
    action: &DeployGlobalContractAction,
) -> Result<(), ActionsValidationError> {
    if action.code.len() as u64 > limit_config.max_contract_size {
        return Err(ActionsValidationError::ContractSizeExceeded {
            size: action.code.len() as u64,
            limit: limit_config.max_contract_size,
        });
    }

    Ok(())
}
