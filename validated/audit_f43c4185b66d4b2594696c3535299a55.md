### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain Without USDC Payment - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value. If the USDC token at the hardcoded Ink-chain address returns `false` instead of reverting on failure, the function continues to pull usdcE from the victim's `DirectDepositV1` address and send it to the caller — with no USDC actually transferred in.

---

### Finding Description

`ContractOwner` declares `using ERC20Helper for IERC20Base` and uses `safeTransfer`/`safeTransferFrom` throughout the codebase. However, `replaceUsdcEWithUsdc` uses a raw, unchecked `transferFrom` on line 616:

```solidity
// ContractOwner.sol line 608–620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

The three-step sequence is:
1. Pull USDC from `msg.sender` into `directDepositV1` (unchecked).
2. Pull all usdcE out of `directDepositV1` into `ContractOwner` (`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the owner).
3. Send that usdcE to `msg.sender` (safe, checked).

If step 1 silently returns `false`, steps 2 and 3 still execute, giving the caller free usdcE.

The `ERC20Helper.safeTransferFrom` wrapper is available and used everywhere else:

```solidity
// ERC20Helper.sol line 23–42
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
}

### Impact Explanation

An attacker can drain the entire usdcE balance held in any `DirectDepositV1` address (belonging to any subaccount) without providing USDC in return. The stolen usdcE represents real user collateral that was deposited into the protocol. The asset delta is: attacker gains `balance` usdcE; the `directDepositV1` address loses `balance` usdcE; no USDC is deposited in exchange. [3](#0-2) 

---

### Likelihood Explanation

The function is externally callable by any unprivileged address with no access control beyond the chain-ID check (`block.chainid == 57073`). The USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` is a non-standard deployment on the Ink chain (chain ID 57073). Non-standard or bridged stablecoin deployments on newer chains frequently implement the non-reverting `false`-return pattern. Any caller who has zero USDC balance or zero allowance but calls this function would trigger the silent failure path. [4](#0-3) 

---

### Recommendation

Replace the raw `transferFrom` with the `safeTransferFrom` helper already imported and used throughout the contract:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. A `directDepositV1` address for some `subaccount` exists and holds `N` usdcE tokens (deposited by a legitimate user).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` on chain 57073 with zero USDC balance or zero allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` silently (no revert).
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes: transfers all `N` usdcE from `directDepositV1` to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes: attacker receives `N` usdcE.
6. Net result: attacker gains `N` usdcE; the subaccount's `directDepositV1` is drained; no USDC was deposited. [7](#0-6) [3](#0-2)
