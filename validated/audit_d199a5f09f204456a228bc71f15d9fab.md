Audit Report

## Title
`SwapAllowlistExtension` Checks Intermediary Caller Instead of Actual Swapper, Enabling Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool populates with `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the call, `sender` is the router address. If the router is allowlisted (required for router-based swaps to function), every user routing through it bypasses per-user swap restrictions entirely, rendering the allowlist ineffective.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← direct caller of pool.swap(), not the end user
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this as `sender` to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) calls `pool.swap()`, the pool's `msg.sender` is the router contract address. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. For the router to be usable at all, the pool admin must allowlist the router address. Once the router is allowlisted, any unpermissioned user can call the router and the check passes unconditionally.

The existing integration test in `FullMetricExtension.t.sol` confirms this broken identity model — it allowlists `callers[0]` (the `TestCaller` intermediary contract), not `users[0]` (the actual end user), and the swap succeeds:

```solidity
// metric-periphery/test/extensions/FullMetricExtension.t.sol L68-74
swapExtension.setAllowedToSwap(address(pool), address(callers[0]), true);
_swap(0, users[0], false, int128(1000), type(uint128).max);
```

This contrasts with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `owner` — the actual LP position owner passed explicitly as a parameter — rather than `sender` (the intermediary caller). The swap hook has no equivalent "beneficial user" parameter; it only has `sender` (intermediary) and `recipient` (output receiver), neither of which reliably identifies the actual swapper when a router is involved.

## Impact Explanation
A pool admin deploying `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, institutional market makers) achieves no protection when `MetricOmmSimpleRouter` is allowlisted. Any unpermissioned user can call the router, which is the only address the extension checks. The pool's LP assets are exposed to unauthorized swap execution by arbitrary users, directly threatening LP principal and fee accrual. This is a broken core pool functionality causing loss of funds — the allowlist check is bypassed through the router path, matching the allowlist path impact gate.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface in the periphery layer. Any production deployment enabling router-based swaps must allowlist the router, which simultaneously disables per-user enforcement. The trigger requires no privileged access — any user with a standard EOA can call the router. The misconfiguration is structural: the pool admin cannot simultaneously allow router-based swaps and enforce per-user restrictions. No special conditions, timing, or elevated privileges are required.

## Recommendation
The `beforeSwap` hook must identify the actual end user, not the intermediary. Two viable approaches:

1. **Pass the real user via `extensionData`**: Require the router to encode the actual user address into `extensionData`, and have `SwapAllowlistExtension` decode and verify it (with a trusted-router attestation or signature).
2. **Trusted-router guard with `recipient` fallback**: If `sender` is a known trusted router, fall back to checking `recipient` as the effective swapper, requiring a registry of trusted routers in the extension.
3. **Structural fix**: Add a `swapper` field to the `beforeSwap` hook signature that the pool populates from a verified source, mirroring how `owner` is passed in `beforeAddLiquidity`.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, router allowlisted
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// Alice is NOT individually allowlisted
// allowedSwapper[pool][alice] == false

// Alice calls through the router — check passes because sender == router
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: alice,
    zeroForOne: false,
    amountIn: 1000,
    ...
}));
// ✓ swap succeeds — allowedSwapper[pool][router] == true
// Alice bypassed the per-user allowlist entirely
```

The pool's `_beforeSwap` passes `msg.sender` (router) as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]` which is `true`, so the revert is never triggered.
