### Title
Fork Difficulty Inherits Main-Chain Timestamps at Retarget Boundary — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

### Summary

When validating a fork block that falls on a difficulty-retarget boundary, the contract fetches the start-of-window block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header` — the **main chain** index. If the fork diverged before that window boundary, the fork's actual ancestor at that height is a different block with a different timestamp. The difficulty for the fork's retarget block is therefore computed using a cross-tree timestamp (the main chain's block), not the fork's own ancestry. This is the direct analog of the Arbitrum timer-inheritance bug: a structural property (block height) is used to look up state across two different chain trees without verifying lineage.

The Dogecoin module even contains an explicit developer acknowledgment of this concern:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
```

### Finding Description

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

It always resolves height → hash via `mainchain_height_to_header`, which only contains main-chain blocks. [1](#0-0) 

This function is called during difficulty retarget validation in all three non-Zcash chain modules:

**Bitcoin** (`bitcoin.rs` line 81):
```rust

``` [2](#0-1)

By contrast, Zcash's difficulty algorithm walks ancestry exclusively via `get_prev_header` (which follows `prev_block_hash` through `headers_pool`), so it is **not** affected. [5](#0-4) 

The `get_prev_header` implementation correctly follows the actual chain lineage: [6](#0-5) 

**Concrete trigger**: A fork that diverges at or before block height `H - difficulty_adjustment_interval` (i.e., before the start of the retarget window) will have a different block at `first_block_height` than the main chain. When the fork's tip reaches the retarget boundary at height `H`, `get_header_by_height(first_block_height)` returns the **main chain's** block at that height, not the fork's ancestor. The `first_block_time` used in `calculate_next_work_required` is therefore wrong.

The difficulty formula is:
```
actual_time_taken = prev_fork_block.time − main_chain_block[first_block_height].time
new_target = target_from_bits(prev_bits) × actual_time_taken / pow_target_timespan
```

Because `main_chain_block[first_block_height].time` is substituted for the fork's actual ancestor's timestamp, `actual_time_taken` is computed incorrectly. The expected `bits` value returned to `check_pow` is wrong, and the fork block is validated against this incorrect target. [7](#0-6) 

### Impact Explanation

The broken invariant is: **the difficulty for a fork block at a retarget boundary must be computed solely from the fork's own ancestry**. Instead, it is computed from a mix of the fork's `prev_block_header` (fork tree) and the main chain's start-of-window block (main chain tree).

Concretely:

1. An attacker submits a fork that diverges before the retarget window. The fork's block at `first_block_height` has an attacker-controlled timestamp `T_fork`.
2. The main chain's block at `first_block_height` has timestamp `T_main`.
3. If `T_fork > T_main`, the cross-tree computation uses `T_main` (earlier), inflating `actual_time_taken`, and producing a **lower difficulty** (higher target) for the fork's retarget block than the fork's actual history requires.
4. The fork's retarget block is accepted with this incorrect `bits` value. All subsequent fork blocks inherit this incorrect difficulty via `return prev_block_header.block_header.bits`.
5. The fork's chainwork accumulation rate is now decoupled from its actual PoW history. The `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` functions rely on `mainchain_header_to_height` membership to confirm a block is on the canonical chain — if a fork with an incorrectly validated retarget block is later promoted via `reorg_chain`, transaction inclusion proofs against that fork become valid. [8](#0-7) 

### Likelihood Explanation

The entry path is fully unprivileged: `submit_blocks` is callable by any trusted relayer (or by accounts with `UnrestrictedSubmitBlocks` role), and fork submission is a normal operational path explicitly handled by `store_fork_header` and `reorg_chain`. [9](#0-8) 

The bug is triggered whenever a fork diverges before the start of a retarget window — a realistic scenario during any chain reorganization. The TODO comment in `dogecoin.rs` confirms the developers identified this as an open correctness question. The attacker only needs to submit valid headers (with valid PoW) for the fork blocks preceding the retarget boundary; the retarget block itself is then validated against the incorrect cross-tree difficulty.

### Recommendation

Replace `get_header_by_height(first_block_height)` in the retarget logic with an ancestry walk that follows `get_prev_header` from the fork's `prev_block_header` backward by `difficulty_adjustment_interval - 1` steps. This mirrors how Zcash's `zcash_get_next_work_required` correctly traverses ancestry via `get_prev_header` in a loop, ensuring the start-of-window block is always the fork's own ancestor rather than the main chain's block at that height. [10](#0-9) 

### Proof of Concept

1. Initialize the contract with Bitcoin mainnet genesis at a difficulty-adjustment boundary (height divisible by 2016).
2. Submit 2016 main-chain blocks (heights 0–2015) to establish the main chain through one full retarget window.
3. Submit a fork block at height 0 with `prev_block_hash` = genesis and a timestamp `T_fork` set to the maximum allowed value (MTP + 1 second, or near the future-time cap). This fork block diverges before the retarget window.
4. Submit 2015 more fork blocks (heights 1–2015) on top of the fork block, each inheriting the same `bits` as the main chain (since they are within the retarget window).
5. Attempt to submit the fork's retarget block at height 2016. The contract calls `get_header_by_height(0)`, which returns the **main chain's** genesis block (timestamp `T_main`), not the fork's block at height 0 (timestamp `T_fork`).
6. The expected `bits` is computed as `calculate_next_work_required(prev_fork_block, T_main)` instead of `calculate_next_work_required(prev_fork_block, T_fork)`.
7. Since `T_fork > T_main`, `actual_time_taken` is overestimated, and the expected `bits` corresponds to a lower difficulty than the fork's actual history requires.
8. A fork retarget block with `bits` matching this lower difficulty (and a valid PoW hash below the easier target) is accepted by the contract, even though it does not satisfy the difficulty the fork's own ancestry mandates.

fn calculate_next_work_required(
    config: &NetworkConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let prev_block_time: i64 = prev_block_header.block_header.time.into();

    let mut actual_time_taken: i64 = prev_block_time - first_block_time;
    if actual_time_taken < config.pow_target_timespan / 4 {
        actual_time_taken = config.pow_target_timespan / 4;
    }
    if actual_time_taken > config.pow_target_timespan * 4 {
        actual_time_taken = config.pow_target_timespan * 4;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(actual_time_taken).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target = new_target
        / U256::from(<i64 as TryInto<u64>>::try_into(config.pow_target_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
