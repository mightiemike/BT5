### Title
Sanctioned Address Can Be Registered as a Linked Signer, Bypassing Sanctions Enforcement — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler (both slow-mode and fast-path) validates that the subaccount exists via `requireSubaccount`, but never calls `requireUnsanctioned` on the **signer address being registered**. A sanctioned address can therefore be installed as a linked signer for any subaccount, after which it can sign and submit protocol transactions on behalf of that subaccount with no further sanctions gate.

---

### Finding Description

`EndpointStorage.sol` defines two guard functions:

- `requireSubaccount` — confirms the subaccount has been recorded (i.e., has deposited).
- `requireUnsanctioned` — confirms an address is not on the Chainalysis sanctions list. [1](#0-0) [2](#0-1) 

The slow-mode `LinkSigner` path in `processSlowModeTransactionImpl` calls only `requireSubaccount` on the sender and then unconditionally writes the signer address into `linkedSigners`:

```solidity
// EndpointTx.sol lines 232-239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));  // ← no requireUnsanctioned(signer)
}
``` [3](#0-2) 

The fast-path `LinkSigner` handler in `processTransactionImpl` has the same omission:

```solidity
// EndpointTx.sol lines 576-590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
    linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));  // ← no requireUnsanctioned(signer)
}
``` [4](#0-3) 

The `requireUnsanctioned` call that does exist in `submitSlowModeTransactionImpl` (line 375) checks only `msg.sender` — the submitter — not the signer address embedded in the transaction payload: [5](#0-4) 

Once a sanctioned address is stored in `linkedSigners`, `validateSignature` / `validateCompactSignature` will accept its signatures as valid for the linked subaccount, because the verifier accepts either the subaccount owner or the linked signer: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A sanctioned address registered as a linked signer can sign `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, and any other sequencer-dispatched transaction on behalf of the subaccount. The sequencer submits these transactions with itself as `msg.sender`, so the per-submission `requireUnsanctioned(msg.sender)` gate is never triggered. The sanctions enforcement that the protocol explicitly implements is fully bypassed for all sequencer-path operations.

Corrupted state: `linkedSigners[subaccount]` is set to a sanctioned address, granting it persistent signing authority over the subaccount's assets and positions.

---

### Likelihood Explanation

The attack requires only two steps reachable by any unprivileged user:

1. Call `submitSlowModeTransaction` with a `LinkSigner` payload naming a sanctioned address as the signer. The submitter is not sanctioned, so `requireUnsanctioned(msg.sender)` passes. The signer address is never checked.
2. Wait for the slow-mode transaction to be executed (or for the sequencer to process the fast-path equivalent). The sanctioned address is now the linked signer.

No privileged access, no leaked keys, and no governance action is required.

---

### Recommendation

Add `requireUnsanctioned` on the signer address in both `LinkSigner` handlers:

**Slow-mode path (`processSlowModeTransactionImpl`):**
```solidity
requireSubaccount(txn.sender);
requireUnsanctioned(address(uint160(bytes20(txn.signer)))); // add this
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

**Fast path (`processTransactionImpl`):**
```solidity
validateSignedTx(...);
requireUnsanctioned(address(uint160(bytes20(signedTx.tx.signer)))); // add this
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
```

---

### Proof of Concept

1. Attacker controls address `A` (not sanctioned) and wants sanctioned address `S` to control subaccount `A || "default\x00\x00\x00\x00"`.
2. `A` calls `Endpoint.submitSlowModeTransaction` with a `LinkSigner` payload: `{sender: subaccount_of_A, signer: bytes32(uint256(uint160(S)))}`.
3. `submitSlowModeTransactionImpl` runs: `requireUnsanctioned(A)` passes; the payload is queued.
4. The sequencer executes the slow-mode tx via `processSlowModeTransactionImpl`. `requireSubaccount` passes; `linkedSigners[subaccount_of_A] = S` is written. No sanctions check on `S`.
5. `S` signs a `WithdrawCollateral` transaction off-chain for `subaccount_of_A`.
6. The sequencer calls `submitTransactionsChecked`, which calls `processTransactionImpl`. `validateSignedTx` recovers `S` from the signature, calls `getLinkedSigner` which returns `S`, and `verifier.validateSignature` accepts it.
7. `clearinghouse.withdrawCollateral` executes, draining the subaccount's collateral under the control of the sanctioned address `S`. [3](#0-2) [4](#0-3) [2](#0-1)
