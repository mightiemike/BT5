### Title
Malicious Counterparty Chain Can Drain NFT Escrow via Crafted `ClassId` in `OnRecvPacket` — (`x/nft-transfer/keeper/packet.go`)

---

### Summary

`processReceivedPacket` determines whether to mint vouchers or unescrow NFTs solely by calling `IsAwayFromOrigin(packet.GetSourcePort(), packet.GetSourceChannel(), data.ClassId)`. The `data.ClassId` field is attacker-controlled packet payload — the IBC core layer validates packet routing but never validates the content of `data.ClassId`. A malicious counterparty chain can craft a `ClassId` that starts with its own source port/channel prefix, forcing `IsAwayFromOrigin` to return `false` and triggering the unescrow branch, transferring any NFT held in the victim chain's escrow account to an arbitrary receiver.

---

### Finding Description

**Root cause:** `IsAwayFromOrigin` is a pure string-prefix check on `data.ClassId`, which is fully attacker-controlled: [1](#0-0) 

In `processReceivedPacket`, the branch decision is made entirely on this check: [2](#0-1) 

`ValidateBasic` on `NonFungibleTokenPacketData` only checks that `ClassId` is non-empty, that token/URI counts match, and that addresses are valid — it performs **no validation** that `ClassId` is consistent with the packet's routing: [3](#0-2) 

**Exploit path (concrete):**

Assume:
- Chain A (victim): port `nft`, channel `channel-0` ↔ Chain B (attacker): port `nft`, channel `channel-1`
- Chain A has NFT `token1` under class `nativeClass`

**Step 1:** A legitimate user on Chain A sends `token1` to Chain B. `createOutgoingPacket` calls `IsAwayFromOrigin("nft", "channel-0", "nativeClass")` → `true`, so `token1` is escrowed to `GetEscrowAddress("nft", "channel-0")`. [4](#0-3) 

**Step 2:** Attacker (controlling Chain B) crafts a malicious `OnRecvPacket` with:
- `packet.SourcePort = "nft"`, `packet.SourceChannel = "channel-1"` (Chain B's real channel — IBC core enforces this)
- `data.ClassId = "nft/channel-1/nativeClass"` (crafted payload)
- `data.TokenIds = ["token1"]`
- `data.Receiver = attacker_address`

**Step 3:** On Chain A, `processReceivedPacket` evaluates:
```
IsAwayFromOrigin("nft", "channel-1", "nft/channel-1/nativeClass")
  → prefixClassID = "nft/channel-1/"
  → strings.HasPrefix("nft/channel-1/nativeClass", "nft/channel-1/") = true
  → returns false  ← unescrow branch triggered
```

**Step 4:** In the unescrow branch:
```go
unprefixedClassID = RemoveClassPrefix("nft", "channel-1", "nft/channel-1/nativeClass")
                  = "nativeClass"
voucherClassID    = ParseClassTrace("nativeClass").IBCClassID()
                  = "nativeClass"   // no path → returns base class
escrowAddress     = GetEscrowAddress("nft", "channel-0")  // victim's escrow
TransferOwner(ctx, "nativeClass", "token1", escrowAddress, attacker_address)
``` [5](#0-4) 

`TransferOwner` checks that `escrowAddress` is the current owner of `token1` under `nativeClass` — which it is, from Step 1. The transfer succeeds. The attacker receives `token1` without ever legitimately sending it back. [6](#0-5) 

---

### Impact Explanation

Any NFT escrowed in `GetEscrowAddress(destPort, destChannel)` on the victim chain — from any user who has ever sent an NFT to the counterparty chain via that channel — can be stolen by the counterparty chain operator. The attacker does not need to own or have previously held the NFT. This is an unauthorized unescrow: the invariant that "unescrow requires a prior matching escrow event from the receiver" is violated.

---

### Likelihood Explanation

This requires a malicious IBC counterparty chain (i.e., a chain whose validators/operators are adversarial). This is a realistic threat model for IBC: chains are explicitly untrusted counterparties. The attack requires no governance access, no private key compromise, and no special privileges beyond operating a connected chain. The crafted packet passes all existing validation.

---

### Recommendation

In `processReceivedPacket`, validate that `data.ClassId` does **not** start with the destination chain's own port/channel prefix (which would indicate a spoofed "return" path). Specifically, before calling `IsAwayFromOrigin`, verify that if `data.ClassId` has a `sourcePort/sourceChannel/` prefix, the corresponding class trace was legitimately registered by a prior outgoing transfer from this chain. The ICS-721 spec requires that the receiving chain only unescrow tokens it previously escrowed — this invariant must be enforced by checking the class trace registry or by rejecting any `ClassId` that begins with `sourcePort/sourceChannel/` unless a matching outgoing escrow record exists.

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
