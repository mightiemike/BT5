### Title
Linked Signer Can Silently Replace Itself via `LinkSigner` Without Subaccount Owner Consent — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` fast-mode transaction is validated with `allowLinkedSigner = true`. This means the currently-linked signer for a subaccount can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any arbitrary address — including an attacker-controlled one — without the subaccount owner's knowledge or consent. A compromised linked signer can silently escalate to permanent, full signing authority over the subaccount.

---

### Finding Description

`processTransactionImpl` in `EndpointTx.sol` handles the `LinkSigner` transaction type as follows:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

The `allowLinkedSigner = true` flag is passed to `validateSignedTx`, which in turn calls `validateSignature` with `getLinkedSigner(sender)` as the permitted alternate signer:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner address or the `linkedSigner`: [3](#0-2) 

This means the currently-linked signer is authorized to sign a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any new address. The subaccount owner is never consulted for this mutation.

The `LinkSigner` struct carries no scope restriction — `signer` is a free `bytes32` field: [4](#0-3) 

---

### Impact Explanation

Once a linked signer is compromised, the attacker can:

1. Sign a `LinkSigner` transaction setting `linkedSigners[victim_subaccount] = attacker_address`.
2. The attacker-controlled address now has full signing authority over the subaccount.
3. The attacker uses this authority to sign `TransferQuote` (also `allowLinkedSigner = true`, line 604) to move all quote balance to an attacker-controlled subaccount: [5](#0-4) 

4. Alternatively, the attacker signs `WithdrawCollateral` (also `allowLinkedSigner = true`) to initiate withdrawals: [6](#0-5) 

The corrupted state is `linkedSigners[victim_subaccount]` — a mapping that controls who can authorize every subsequent critical operation on the subaccount. The asset delta is the full collateral balance of the victim subaccount.

Additionally, because the attacker controls the linked signer, they can continuously re-sign new `LinkSigner` transactions to maintain control even if the victim attempts to revoke via slow-mode. The attacker can front-run any slow-mode revocation by submitting a new fast-mode `LinkSigner` override through the sequencer.

---

### Likelihood Explanation

The linked signer feature is explicitly designed for automated trading bots and third-party integrations. Any dApp or service that holds a linked signer key is a potential compromise vector (key leak, malicious insider, DNS/BGP attack on the dApp, injection). The attack requires only that the linked signer key be accessible to the attacker — no privileged protocol access is needed. The sequencer processes the malicious `LinkSigner` transaction identically to a legitimate one.

---

### Recommendation

The `LinkSigner` transaction type should not accept signatures from the currently-linked signer. It is a privileged account-management operation that changes who can act on behalf of the subaccount, and it must only be authorized by the subaccount owner's key. Change the `allowLinkedSigner` flag for `LinkSigner` processing from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [7](#0-6) 

This mirrors the slow-mode `LinkSigner` path, which already enforces `validateSender(txn.sender, sender)` — requiring the transaction to originate from the subaccount owner's address: [8](#0-7) 

---

### Proof of Concept

**Setup**: Victim subaccount `0xVICTIM000...0001` has linked signer `signerA` (key held by a trading bot). Attacker compromises `signerA`'s private key.

**Attack**:

1. Attacker constructs a `LinkSigner` transaction:
   ```
   sender  = 0xVICTIM000...0001
   signer  = bytes32(uint256(uint160(attacker_address)))
   nonce   = current nonce of victim
   ```
2. Attacker signs this with `signerA`'s key (ECDSA over the EIP-712 digest).
3. Attacker submits to the sequencer via `submitTransactionsChecked`.
4. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`. `validateSignature` recovers `signerA` from the signature, which equals `getLinkedSigner(victim_subaccount)` → passes.
5. `linkedSigners[0xVICTIM000...0001] = attacker_address` is written.
6. Attacker now signs a `TransferQuote` transaction:
   ```
   sender    = 0xVICTIM000...0001
   recipient = attacker_controlled_subaccount
   amount    = full quote balance
   ```
   signed with `attacker_address`'s key.
7. `validateSignedTx(..., true)` recovers `attacker_address` == `getLinkedSigner(victim_subaccount)` → passes.
8. `clearinghouse.transferQuote` moves all quote from victim to attacker.

**Corrupted invariant**: `linkedSigners[victim_subaccount]` is mutated without the subaccount owner's signature, violating the assumption that only the owner can delegate signing authority. [1](#0-0) [5](#0-4)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
