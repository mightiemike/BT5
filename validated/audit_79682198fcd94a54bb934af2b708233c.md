### Title
Missing Event Emission on `linkedSigners` Mutation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.sol` processes `LinkSigner` transactions in both the fast path (`processTransactionImpl`) and the slow-mode path (`processSlowModeTransactionImpl`). In both cases, the `linkedSigners` mapping is mutated without emitting any event. Because `linkedSigners` controls which external address may authorize all subsequent signed transactions on behalf of a subaccount, the silent mutation creates an undetectable on-chain footprint for a critical security-relevant state change.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the `LinkSigner` transaction type at lines 576–590:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

No `emit` statement follows the assignment. The same omission exists in `processSlowModeTransactionImpl` at line 239:

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

The `linkedSigners` mapping is consumed by `getLinkedSigner()`, which is called inside both `validateSignature()` and `validateCompactSignature()`. Any address stored in `linkedSigners[subaccount]` can sign and authorize every transaction type that passes `allowLinkedSigner = true`, including `WithdrawCollateral`, `MintNlp`, `BurnNlp`, `TransferQuote`, `LiquidateSubaccount`, and `MatchOrders`.

By contrast, other security-relevant state changes in the same codebase do emit events. `Verifier.assignPubKey` emits `AssignPubKey` and `deletePubkey` emits `DeletePubkey`. `Clearinghouse.depositCollateral` and `withdrawCollateral` emit `ModifyCollateral`. The `LinkSigner` path is the only user-controlled delegation mechanism that produces no on-chain trace.

---

### Impact Explanation

The `linkedSigners` mapping is the sole on-chain record of signing authority delegation for a subaccount. Without an event:

- Off-chain monitoring infrastructure (indexers, alert systems, user dashboards) cannot observe when a subaccount's signing authority changes.
- A user whose primary key is compromised and whose linked signer is silently replaced has no on-chain signal to detect the takeover before the attacker drains collateral via a signed `WithdrawCollateral` or `BurnNlp` transaction.
- Protocol-level audit trails are incomplete: the only way to reconstruct the current linked signer for a subaccount is to replay all sequencer-submitted transaction batches from genesis, which is operationally expensive and error-prone.

The corrupted state is `linkedSigners[subaccount]`, which directly gates authorization for all asset-moving operations on that subaccount.

---

### Likelihood Explanation

Every user who calls `LinkSigner` — a standard, documented user operation — triggers this path. The transaction is submitted through the normal sequencer flow (`submitTransactionsChecked`) or via `submitSlowModeTransaction`. No special privilege is required. The missing event affects every `LinkSigner` invocation unconditionally.

---

### Recommendation

Declare a `LinkedSignerSet` event and emit it in both `LinkSigner` handling branches:

```solidity
event LinkedSignerSet(bytes32 indexed subaccount, address indexed signer);
```

In `processTransactionImpl`:
```solidity
linkedSigners[signedTx.tx.sender] = address(uint160(bytes20(signedTx.tx.signer)));
emit LinkedSignerSet(signedTx.tx.sender, address(uint160(bytes20(signedTx.tx.signer))));
```

In `processSlowModeTransactionImpl`:
```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
emit LinkedSignerSet(txn.sender, address(uint160(bytes20(txn.signer))));
```

This is consistent with how `Verifier.sol` emits `AssignPubKey` and `DeletePubkey` on every signer-set mutation.

---

### Proof of Concept

1. User A holds subaccount `0xAAAA...0001`.
2. User A submits a `LinkSigner` transaction designating address `0xBBBB` as their linked signer. The sequencer includes it in a batch submitted to `submitTransactionsChecked`.
3. `processTransactionImpl` executes: `linkedSigners[0xAAAA...0001] = 0xBBBB`. No event is emitted.
4. An attacker who later compromises User A's key submits a second `LinkSigner` transaction replacing `0xBBBB` with `0xCCCC`. Again, no event is emitted.
5. The attacker, now controlling `0xCCCC`, submits a signed `WithdrawCollateral` transaction. `getLinkedSigner` returns `0xCCCC`, `validateSignature` passes, and collateral is drained.
6. No on-chain event exists at any step to alert User A, their wallet, or any monitoring system that the linked signer was changed.

The root cause is the unconditional absence of an `emit` after the `linkedSigners` assignment in both `processTransactionImpl` and `processSlowModeTransactionImpl` in `core/contracts/EndpointTx.sol`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
