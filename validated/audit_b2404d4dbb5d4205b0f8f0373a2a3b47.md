### Title
Duplicate-Last-Leaf Index Aliasing in `compute_root_from_merkle_proof` Allows Phantom `tx_index` Verification — (`merkle-tools/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` performs no bounds check on `transaction_position` against the actual number of transactions in the block. In any odd-width Bitcoin Merkle tree, the last real leaf is duplicated to form an even-width level. This causes the same `(tx_id, merkle_proof)` pair to produce an identical root for **two distinct `tx_index` values**: the real last-leaf index and its phantom duplicate index. `verify_transaction_inclusion` returns `true` for the phantom index even though no transaction exists there.

---

### Finding Description

**Root cause — `merkle-tools/src/lib.rs`, `compute_root_from_merkle_proof`:** [1](#0-0) 

The function iterates over the proof, choosing left/right placement by `current_position % 2`, then halves the position. There is no check that `transaction_position < actual_tx_count`.

**Concrete arithmetic for a 3-transaction block `[T0, T1, T2]`:**

Bitcoin duplicates the last leaf, so the effective leaf array is `[T0, T1, T2, T2]`.

```
Level 0:  T0(0)  T1(1)  T2(2)  T2(3←phantom)
Level 1:  H(T0,T1)=H01      H(T2,T2)=H22
Level 2:  H(H01, H22) = Root
```

The canonical proof for T2 at index 2 is `proof = [T2, H01]`.

**Verification at index 2 (real):**
1. pos=2 (even): `H(T2, T2)` = H22; pos→1
2. pos=1 (odd): `H(H01, H22)` = Root ✓

**Verification at index 3 (phantom) with the identical proof `[T2, H01]`:**
1. pos=3 (odd): `H(T2, T2)` = H22; pos→1
2. pos=1 (odd): `H(H01, H22)` = Root ✓

Both paths produce the same root. `verify_transaction_inclusion` returns `true` for `tx_index=3`, a position that does not correspond to any real transaction.

**Call site in `contract/src/lib.rs`:** [2](#0-1) 

The only guards applied before reaching `compute_root_from_merkle_proof` are:
- `tx_block_blockhash` must be in `mainchain_header_to_height` (canonical chain membership) [3](#0-2) 
- Confirmation depth check [4](#0-3) 
- `merkle_proof` must be non-empty [5](#0-4) 

None of these guards constrain `tx_index` to `[0, actual_tx_count)`.

---

### Impact Explanation

`verify_transaction_inclusion` returns `true` for a `(tx_id, tx_block_blockhash, tx_index=N+1)` tuple where `N` is the last real index in an odd-width block. Position `N+1` does not correspond to any real on-chain transaction. Any downstream contract that:

1. calls `verify_transaction_inclusion` to authorize a withdrawal or release, and
2. deduplicates processed claims by `(tx_id, tx_block_blockhash, tx_index)`

can be double-spent: the attacker first processes the real claim at index `N`, then replays the identical proof at phantom index `N+1` to unlock a second withdrawal for the same Bitcoin transaction.

The attacker cannot forge a completely fake transaction — `tx_id` must be a real transaction in the block. The attack is limited to index aliasing of the last leaf.

**The GC timing angle in the question is a red herring.** After `run_mainchain_gc` advances `mainchain_initial_blockhash`, GC'd blocks are removed from `mainchain_header_to_height`, causing the canonical-chain lookup to panic. GC makes old blocks *unverifiable*, not more vulnerable. The duplicate-index attack works on any currently-retained canonical block regardless of GC state. [6](#0-5) 

---

### Likelihood Explanation

- The function is public and callable by any unprivileged account (no `#[private]`, no `#[trusted_relayer]` on `verify_transaction_inclusion`). [7](#0-6) 
- Odd-width blocks (blocks with an odd number of transactions) are common on mainnet.
- The attacker only needs to observe a legitimate proof already submitted on-chain and replay it with `tx_index` incremented by 1.
- No privileged role, key compromise, or social engineering is required.

---

### Recommendation

In `compute_root_from_merkle_proof`, add a parameter for the total transaction count and reject any `transaction_position >= tx_count` before computing the root:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    tx_count: usize,          // <-- add this
    merkle_proof: &Vec<H256>,
) -> H256 {
    assert!(transaction_position < tx_count, "tx_index out of bounds");
    // ... existing logic
}
```

The `tx_count` must be committed to the block header or supplied alongside the proof and independently verified. Alternatively, enforce that `tx_index` is strictly less than `2^(proof.len()-1)` only when the tree is known to be full — but the safest fix is to pass and check the actual transaction count.

The deprecated `verify_transaction_inclusion` should be removed entirely in favour of `verify_transaction_inclusion_v2`, which at minimum adds coinbase-proof validation. [8](#0-7)

### Proof of Concept

```rust
// 3-transaction block: [T0, T1, T2]
// Merkle tree (Bitcoin odd-leaf duplication):
//   Level 0: T0  T1  T2  T2(dup)
//   Level 1: H(T0,T1)   H(T2,T2)
//   Level 2: H(H(T0,T1), H(T2,T2))  = Root

let proof_for_T2 = vec![T2.clone(), H(T0, T1)];

// Real call — index 2, returns true (expected)
verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(),
    tx_block_blockhash: block_hash,
    tx_index: 2,
    merkle_proof: proof_for_T2.clone(),
    confirmations: 1,
});

// Phantom call — index 3, ALSO returns true (bug)
// Position 3 does not correspond to any real transaction.
verify_transaction_inclusion(ProofArgs {
    tx_id: T2.clone(),
    tx_block_blockhash: block_hash,
    tx_index: 3,              // phantom index
    merkle_proof: proof_for_T2,  // identical proof
    confirmations: 1,
});
// Both calls return `true`.
// A downstream contract deduplicating by (tx_id, block_hash, tx_index)
// would process two withdrawals for the same Bitcoin transaction.
```

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
