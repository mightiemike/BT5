### Title
NFT Denom Creator Front-Running Allows Attacker to Seize Exclusive Minting Rights - (File: x/nft/keeper/msg_server.go, x/nft/keeper/denom.go)

### Summary
`MsgIssueDenom` is a first-come-first-served operation that permanently assigns the `Creator` role to whichever address submits the transaction first. An unprivileged attacker who monitors the mempool can copy a victim's pending `MsgIssueDenom` transaction, resubmit it with their own address as `Sender` and a higher gas fee, and become the permanent creator of the targeted denom ID. The victim's transaction then fails with a duplicate-ID error, and the attacker holds exclusive minting rights over that denom indefinitely.

### Finding Description
`IssueDenom` in `x/nft/keeper/msg_server.go` derives the creator solely from `msg.Sender`, which is the transaction signer: [1](#0-0) 

`SetDenom` in `x/nft/keeper/denom.go` enforces uniqueness on both `denomID` and `denomName`, and permanently records the first submitter as creator: [2](#0-1) 

Once registered, `IsDenomCreator` is the authorization gate for all subsequent privileged operations on that denom (minting NFTs): [3](#0-2) 

There is no commit-reveal scheme, no creator-address binding in the message itself, and no replay protection tied to the intended creator. The `denomID` and `denomName` are fully visible in the mempool before inclusion.

### Impact Explanation
The corrupted value is `denom.Creator` stored in the KV store under `KeyDenomID(id)`. After a successful front-run:

- The attacker's address is permanently recorded as `denom.Creator` instead of the legitimate user's address.
- The legitimate user's `MsgIssueDenom` fails with `ErrInvalidDenom: denomID X has already exists`.
- The attacker holds exclusive rights to mint NFTs under the stolen denom ID via the `IsDenomCreator` check.
- The legitimate user can never reclaim that specific `denomID` or `denomName` — both are blocked permanently by `HasDenomID` and `HasDenomNm`. [4](#0-3) 

The attacker can use the stolen denom to mint counterfeit NFTs impersonating the victim's intended collection, or simply hold the denom hostage.

### Likelihood Explanation
Any full node operator or anyone running a node with mempool access can observe pending `MsgIssueDenom` transactions. No privileged role is required — only a standard account with enough funds to pay a slightly higher gas fee. The attack is especially attractive for high-value or brand-named NFT collections where the `denomID` has commercial significance. The attack is deterministic and requires no special timing beyond standard mempool front-running.

### Recommendation
Bind the intended creator address into the uniqueness check so that a front-runner using a different `Sender` cannot claim the same `denomID`. One approach is to include the creator address as part of the denom key (e.g., `KeyDenomID(creatorAddr + "/" + id)`), so different creators can register the same logical name without collision. Alternatively, adopt a commit-reveal scheme: in phase 1 the user commits a hash of `(denomID, creatorAddr, nonce)`; in phase 2 they reveal and register, with the contract verifying the committed creator matches `msg.Sender`. This mirrors the fix applied in OpenZeppelin Governor v4.9.1 which bound the proposer address into the proposal ID.

### Proof of Concept

1. Victim signs and broadcasts:
   ```
   MsgIssueDenom{Id: "blue-chip", Name: "Blue Chip Collection", Sender: victim_addr, gas: 100000}
   ```
2. Attacker observes this transaction in the mempool.
3. Attacker broadcasts:
   ```
   MsgIssueDenom{Id: "blue-chip", Name: "Blue Chip Collection", Sender: attacker_addr, gas: 110000}
   ```
4. Attacker's transaction is included first. `SetDenom` stores `denom.Creator = attacker_addr` under `KeyDenomID("blue-chip")`. [5](#0-4) 

5. Victim's transaction executes next. `HasDenomID("blue-chip")` returns `true`, and `SetDenom` returns `ErrInvalidDenom: denomID blue-chip has already exists`. Victim's tx fails. [6](#0-5) 

6. Any subsequent `MsgMintNFT` by the victim under `"blue-chip"` fails at `IsDenomCreator` because `denom.Creator == attacker_addr != victim_addr`. [7](#0-6) 

7. The attacker now mints NFTs under `"blue-chip"` freely, permanently locking the victim out of their intended collection.

```go
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
	if k.HasDenomID(ctx, denom.Id) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
	}

	if k.HasDenomNm(ctx, denom.Name) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
	}

	store := ctx.KVStore(k.storeKey)
	bz := k.cdc.MustMarshal(&denom)
	store.Set(types.KeyDenomID(denom.Id), bz)
	store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
	return nil
```

```go
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
```
