### Title
CEI Violation in `withdrawCollateral` — Token Transfer Precedes Balance Deduction, Enabling Cross-Function Reentrancy - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs a token transfer to the recipient via `handleWithdrawTransfer` **before** deducting the sender's balance in the spot engine and before running the health check. This violates the Checks-Effects-Interactions pattern and opens a cross-function reentrancy window identical in class to the H-04 finding in TimeswapPair.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. `handleWithdrawTransfer(token, sendTo, amount, idx)` — transfers tokens out of the clearinghouse to `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which calls `token.safeTransfer(to, amount)` — **tokens reach the recipient here**
2. `spotEngine.updateBalance(productId, sender, amountRealized)` — balance deducted **after** transfer
3. `spotEngine.assertUtilization(productId)` — utilization checked **after** transfer
4. `require(getHealth(sender, healthType) >= 0, ...)` — health checked **after** transfer [1](#0-0) 

The internal helper that performs the external call: [2](#0-1) 

Inside `BaseWithdrawPool.submitWithdrawal`, the token is forwarded to the recipient via `handleWithdrawTransfer`: [3](#0-2) [4](#0-3) 

At the moment `token.safeTransfer(to, amount)` executes inside `BaseWithdrawPool.handleWithdrawTransfer`, the spot engine balance for `sender` has **not yet been decremented**. Any reentrant call that reads or acts on that balance observes a stale, inflated value.

There is no `nonReentrant` guard on `withdrawCollateral` or on the publicly callable `executeSlowModeTransaction`. [5](#0-4) 

---

### Impact Explanation

If a supported collateral token implements ERC-777 receive hooks (or any `tokensReceived`/`onERC20Received` callback), the recipient contract can reenter during the `token.safeTransfer(to, amount)` call. At that moment:

- The sender's spot engine balance still reflects the **pre-withdrawal** amount.
- The health check has not yet run.

The attacker can call `Endpoint.executeSlowModeTransaction` (publicly callable, no reentrancy guard) to process a second queued withdrawal for the same subaccount. The second `withdrawCollateral` invocation sees the full pre-deduction balance, passes the health check, and transfers a second batch of tokens. Both withdrawals then complete, but the balance is only decremented twice sequentially — meaning the attacker can drain more collateral than their balance permits, corrupting the solvency accounting of the clearinghouse.

The corrupted state delta is: `spotEngine` balance for `sender` is decremented twice while the actual on-chain token reserve is reduced by `2 × amount`, breaking the invariant that `sum(balances) == token.balanceOf(clearinghouse)`.

---

### Likelihood Explanation

The trigger requires a supported collateral token with a receive hook (ERC-777 or equivalent). Standard USDC/USDT do not have hooks, which limits immediate exploitability. However:

- The protocol is designed to support arbitrary ERC-20 tokens via `addOrUpdateProducts`.
- Any future listing of an ERC-777 token (e.g., certain wrapped assets or vault tokens) immediately activates this path.
- The attacker only needs to queue two slow-mode withdrawal transactions and control a recipient contract — no privileged access required.

Likelihood: **Medium** (conditional on token type; architectural risk is present today).

---

### Recommendation

Move `handleWithdrawTransfer` to **after** all state updates and health checks in `withdrawCollateral`, following the CEI pattern:

```solidity
function withdrawCollateral(...) public virtual onlyEndpoint {
    // Checks
    require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
    require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    ...

    // Effects
    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
    int128 amountRealized = -int128(amount) * int128(multiplier);
    spotEngine.updateBalance(productId, sender, amountRealized);
    spotEngine.assertUtilization(productId);
    require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

    // Interactions (last)
    handleWithdrawTransfer(token, sendTo, amount, idx);
    emit ModifyCollateral(amountRealized, sender, productId);
}
```

Additionally, add a `nonReentrant` modifier to `withdrawCollateral` and `executeSlowModeTransaction` as defense-in-depth.

---

### Proof of Concept

1. Attacker deploys a recipient contract `Exploit` that, on receiving ERC-777 tokens, calls `Endpoint.executeSlowModeTransaction()`.
2. Attacker queues two slow-mode `WithdrawCollateral` transactions for the same subaccount, each for amount `A`. Subaccount balance = `A`.
3. Attacker (or anyone) calls `Endpoint.executeSlowModeTransaction()` to process the first withdrawal.
4. Execution enters `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(Exploit, A)`.
5. ERC-777 hook fires on `Exploit`. At this point `spotEngine.balance[sender] == A` (not yet decremented).
6. `Exploit` calls `Endpoint.executeSlowModeTransaction()` to process the second withdrawal.
7. Second `withdrawCollateral` runs: balance is still `A`, health check passes, second `A` tokens are transferred out.
8. Control returns to the first `withdrawCollateral`: balance is decremented by `A` (now `0`), health check passes (balance is `0`, not negative).
9. Net result: `2A` tokens withdrawn against a balance of `A`. Clearinghouse is insolvent by `A`. [6](#0-5) [3](#0-2) [5](#0-4)

        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
