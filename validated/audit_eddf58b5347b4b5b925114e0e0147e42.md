### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable and Lacks Coinbase Depth-Binding, Enabling 64-Byte Internal-Node Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is annotated `#[deprecated]` but carries no `#[private]` guard. In Rust/NEAR, `#[deprecated]` is a **compiler lint only**; it does not restrict on-chain invocability. Any unprivileged NEAR account can call the function directly. Because the function passes the caller-supplied `tx_id` straight into `compute_root_from_merkle_proof` without any coinbase depth-binding check, an attacker can supply an internal Merkle-tree node hash as `tx_id` together with a crafted proof and receive `true` — a forged transaction-inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub` with only a `#[pause]` gate: [1](#0-0) 

The function's own doc-comment acknowledges the risk explicitly: [2](#0-1) 

The verification logic passes the caller-controlled `args.tx_id` directly to `compute_root_from_merkle_proof` with no leaf-level constraint: [3](#0-2) 

`compute_root_from_merkle_proof` is a pure hash-chain computation; it accepts any 32-byte value as the starting leaf: [4](#0-3) 

`verify_transaction_inclusion_v2` fixes this by first verifying a coinbase proof of **equal length**, which depth-binds the target proof to the leaf level: [5](#0-4) 

Because v1 has no equivalent guard, an attacker can:

1. Pick any real Bitcoin block whose header is stored in the contract.
2. Identify an internal Merkle node `N` at depth `d` from the root (i.e., `d` levels above the leaves).
3. Build a valid `d`-element Merkle proof from `N` up to the known `merkle_root`.
4. Call `verify_transaction_inclusion({ tx_id: N, tx_index: crafted, merkle_proof: proof, tx_block_blockhash: real_block, confirmations: 1 })`.
5. The function returns `true`.

---

### Impact Explanation

Any NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) to gate a financial action (e.g., cross-chain bridge unlock, swap settlement, custody release) can be deceived into treating a non-existent Bitcoin transaction as confirmed. The forged `tx_id` is a hash of two real child nodes — it is computationally indistinguishable from a leaf hash by the v1 API.

---

### Likelihood Explanation

- The function is unconditionally reachable by any NEAR account (no role check, no `#[private]`).
- The 64-byte forgery technique is publicly documented ([BitMEX research](https://www.bitmex.com/blog/64-Byte-Transactions)) and requires only knowledge of a block's Merkle tree structure, which is public.
- Downstream consumers that integrated before v2 existed, or that call v1 by name, are permanently exposed until v1 is removed or gated.

---

### Recommendation

1. **Add `#[private]`** to `verify_transaction_inclusion` to make it callable only by the contract itself (i.e., only via `verify_transaction_inclusion_v2`'s internal `#[allow(deprecated)]` call). This is the minimal fix.
2. Alternatively, **remove the public export** of v1 entirely and update `verify_transaction_inclusion_v2` to inline the logic.
3. Audit all known consumers of the v1 ABI and migrate them to v2.

---

### Proof of Concept

```rust
// Pseudocode for a NEAR SDK integration test
let block = real_block_in_contract(); // block with known merkle_root and ≥2 txs
// internal_node = double_sha256(tx0_hash || tx1_hash)  — level-1 internal node
let internal_node: H256 = compute_hash(&tx0, &tx1);
// proof from internal_node to root: just the sibling at level 1, etc.
let crafted_proof = vec![sibling_at_level_1, ...];

let result = contract.verify_transaction_inclusion(ProofArgs {
    tx_id: internal_node,
    tx_block_blockhash: block.hash,
    tx_index: 0,          // crafted to match parity
    merkle_proof: crafted_proof,
    confirmations: 1,
});
assert!(result); // returns true — forgery succeeds

// v2 rejects the same internal node because coinbase proof length
// depth-binds the proof to the actual leaf level:
let result_v2 = contract.verify_transaction_inclusion_v2(ProofArgsV2 {
    tx_id: internal_node,
    coinbase_tx_id: real_coinbase,
    coinbase_merkle_proof: full_depth_coinbase_proof,
    merkle_proof: crafted_proof, // length mismatch or coinbase check fails
    ...
});
// panics: "Incorrect coinbase merkle proof"
```

The root cause is that `#[deprecated]` provides zero runtime enforcement in NEAR contracts — only `#[private]` or role-gating prevents external calls.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
