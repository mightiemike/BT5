### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Proof Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, unrestricted public entry point on the `BtcLightClient` contract. `verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle-node forgery path by requiring a coinbase proof, but v1 remains callable by any unprivileged NEAR account and skips that check entirely. Any consumer contract or relayer-path user that calls v1 — deliberately or by mistake — receives a proof result that is not protected against the known forgery class.

---

### Finding Description

`verify_transaction_inclusion_v2` wraps v1 and adds a mandatory coinbase-proof gate before delegating:

```
verify_transaction_inclusion_v2
  → require coinbase_merkle_proof matches block merkle_root   ← forgery guard
  → self.verify_transaction_inclusion(args.into())            ← v1 called internally
``` [1](#0-0) 

v1 itself contains no coinbase check. Its own warning documents the exact attack surface:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

Despite the `#[deprecated]` annotation, v1 carries only `#[pause]` — no role guard, no `#[private]`. It is a fully reachable public method: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools` is a pure hash computation; it has no awareness of whether its input is a real transaction hash or an internal tree node: [4](#0-3) 

The `ProofArgs` struct accepted by v1 contains only `tx_id`, `tx_block_blockhash`, `tx_index`, `merkle_proof`, and `confirmations` — no coinbase fields exist to enforce the guard even optionally: [5](#0-4) 

---

### Impact Explanation

A 64-byte value that is structurally identical to an internal Merkle tree node can be submitted as `tx_id` to v1. Because `compute_root_from_merkle_proof` treats it as a leaf, the computed root can be made to match a real block's `merkle_root` without any corresponding on-chain transaction. v1 returns `true`. Any consumer contract (bridge, atomic-swap, cross-chain state verifier) that calls v1 to gate a fund release or state transition will accept this forged proof as valid, corrupting the proof result that is the contract's sole security guarantee.

The corrupted value is the boolean return of `verify_transaction_inclusion` — the canonical proof result that downstream contracts treat as authoritative.

---

### Likelihood Explanation

The entry point is public and requires no role. Any NEAR account can call it. Consumer contracts written against an older SDK version or documentation that predates v2 will call v1 by default. The 64-byte Merkle forgery is a documented, practical attack (referenced in the v2 doc comment itself). The attacker needs only a valid block hash present in the main chain and the ability to construct a 64-byte preimage whose double-SHA256 satisfies the Merkle path — a well-understood offline computation.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or gate it with `#[private]` so it is only callable as an internal helper from `verify_transaction_inclusion_v2`. If backward compatibility must be preserved, add the coinbase proof check directly inside v1 so both entry points enforce the same invariant. The `#[deprecated]` annotation alone does not prevent external calls on NEAR.

---

### Proof of Concept

1. Attacker identifies a real block hash `B` present in `mainchain_header_to_height` with a known `merkle_root R`.
2. Attacker constructs a 64-byte value `N` (an internal Merkle node) and a `merkle_proof P` such that `compute_root_from_merkle_proof(N, idx, P) == R`. This is feasible offline because the Merkle tree structure of Bitcoin blocks is public.
3. Attacker calls `verify_transaction_inclusion({tx_id: N, tx_block_blockhash: B, tx_index: idx, merkle_proof: P, confirmations: 1})` directly — bypassing `verify_transaction_inclusion_v2` entirely.
4. The contract checks `mainchain_header_to_height` (block is on main chain ✓), checks confirmations (✓), checks `merkle_proof` is non-empty (✓), and returns `true`.
5. A consumer bridge contract that called v1 now releases funds for a transaction that never existed on Bitcoin. [6](#0-5)

    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
