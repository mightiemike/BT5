### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` — No Claim Mechanism (`BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.sol` accumulates fast-withdrawal fees into a `fees` mapping on every call to `submitFastWithdrawal()`. No function ever reads from or decrements this mapping to transfer the collected tokens out. The fee tokens are permanently locked in the `WithdrawPool` contract.

---

### Finding Description

Every time a liquidity provider (or the withdrawing user) calls `submitFastWithdrawal()`, a fee is computed and the contract either retains it from the withdrawal amount or pulls it in via `safeTransferFrom`. The fee is then credited to the internal accounting variable:

```solidity
fees[productId] += fee;   // BaseWithdrawPool.sol line 111
``` [1](#0-0) [2](#0-1) 

The `fees` mapping is declared as:

```solidity
// collected withdrawal fees in native token decimals
mapping(uint32 => int128) public fees;
```

A search of the entire contract and its concrete implementation `WithdrawPool.sol` reveals that `fees[productId]` is **never decremented, never read for a transfer, and never passed to any beneficiary**. The only token-moving functions in the contract are:

- `submitWithdrawal()` — moves tokens to the withdrawing user, no fee accounting. [3](#0-2) 
- `removeLiquidity()` — `onlyOwner`, moves an arbitrary amount of tokens but does not reference `fees` at all. [4](#0-3) 
- `handleWithdrawTransfer()` — internal helper, also unrelated to `fees`. [5](#0-4) 

`WithdrawPool.sol` adds no new functions; it only calls `_initialize`. [6](#0-5) 

The result: every ERC-20 token unit recorded in `fees[productId]` is held by the contract with no reachable code path to extract it through the intended fee accounting.

---

### Impact Explanation

Fast-withdrawal fees are real ERC-20 collateral tokens (USDC or any supported spot token). They accumulate in `WithdrawPool` indefinitely. The protocol receives no revenue from these fees, and no party — owner, sequencer, or protocol treasury — can claim them through the `fees` accounting path. Over time, as fast withdrawals are processed, the locked amount grows proportionally to protocol usage.

**Corrupted invariant:** `fees[productId]` is supposed to represent claimable protocol revenue; instead it is an ever-increasing dead-end counter whose corresponding token balance is permanently stranded.

---

### Likelihood Explanation

`submitFastWithdrawal()` is a public, permissionless function callable by any liquidity provider or withdrawing user. [7](#0-6)  Every successful fast withdrawal triggers fee accumulation. The lock is not conditional — it occurs on every execution path through the function. Likelihood is **high** because the function is a core, actively used protocol feature.

---

### Recommendation

Add a privileged `claimFees(uint32 productId, address recipient)` function (or extend `removeLiquidity`) that:
1. Reads `fees[productId]`.
2. Resets `fees[productId] = 0`.
3. Calls `handleWithdrawTransfer(getToken(productId), recipient, uint128(fees[productId]))`.

Ensure only an authorized role (owner or protocol treasury) can invoke it, and emit an event for auditability.

---

### Proof of Concept

1. Alice calls `submitFastWithdrawal(idx, transaction, signatures)` where `sendTo == msg.sender`.
2. `transferAmount -= uint128(fee)` — Alice receives less; the fee stays in the contract.
3. `fees[productId] += fee` — the fee is recorded. [8](#0-7) 
4. Repeat for 1,000 fast withdrawals. `fees[productId]` grows to a significant USDC balance.
5. Inspect every function in `BaseWithdrawPool` and `WithdrawPool` — no function decrements `fees[productId]` or transfers its value to any address.
6. The accumulated fee tokens are permanently locked in the `WithdrawPool` contract address.
