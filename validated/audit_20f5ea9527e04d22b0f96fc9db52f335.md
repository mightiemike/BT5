Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unauthorized actor to bypass the deposit allowlist gate - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and instead validates the caller-controlled `owner` parameter against the allowlist. Because `owner` is a free argument supplied by the caller, any unauthorized actor can pass any allowlisted address as `owner` and sail through the gate. The pool admin's intent to restrict who may deposit is completely defeated.

## Finding Description

**Call chain:**

`MetricOmmPool.addLiquidity` (line 191) calls:
```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```
where `msg.sender` is the actual depositor and `owner` is a free parameter supplied by the caller.

`ExtensionCalling._beforeAddLiquidity` (lines 88–99) forwards both values verbatim to the extension:
```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` (lines 32–42) receives `(sender, owner, ...)` but **discards `sender`** (first parameter is unnamed) and checks only `owner`:
```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

**Exploit path:**
1. Pool admin configures the allowlist: `allowedDepositor[pool][alice] = true`, `allowedDepositor[pool][bob] = false`.
2. Unauthorized actor Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)` — setting `owner = alice`.
3. Pool calls `_beforeAddLiquidity(bob /*sender*/, alice /*owner*/, ...)`.
4. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's callback is invoked to supply tokens; Alice receives the LP shares.
6. The allowlist gate is fully bypassed; Bob deposited into the pool despite being unauthorized.

**Why existing guards fail:** The `onlyPool` modifier in `BaseMetricExtension` only verifies that `msg.sender` is a registered pool — it does not validate which address is the actual depositor. There is no secondary check on `sender` anywhere in the hook.

The analogous `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (first parameter, the actual swap caller) and discards `recipient` (second parameter). The deposit extension has the parameters reversed in its validation logic.

## Impact Explanation
The `DepositAllowlistExtension` is a core access-control primitive. Its bypass means the pool admin cannot enforce any deposit restriction: any address can add liquidity to a gated pool by nominating an allowlisted address as `owner`. This breaks the allowlist's core pool functionality. The unauthorized depositor supplies tokens (via callback) and the allowlisted address receives LP shares — the depositor cannot recover those tokens since `removeLiquidity` enforces `msg.sender == owner` (line 206 of `MetricOmmPool.sol`). This constitutes broken core pool functionality (allowlist gate rendered inoperative) and enables forced LP exposure on allowlisted addresses without their consent. Severity: **Medium**.

## Likelihood Explanation
Exploitation requires only that a pool has `DepositAllowlistExtension` attached with at least one allowlisted address (a standard configuration). No privileged access, flash loan, or special token behavior is needed. Any unprivileged EOA or contract can execute the bypass in a single transaction. The attack is repeatable at will.

## Recommendation
Replace the unnamed first parameter with a named `sender` and validate it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap` (line 37), which checks `sender` (the actual swap caller) and discards `recipient`.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Foundry unit test sketch
function test_depositAllowlistBypass() public {
    // Setup: pool with DepositAllowlistExtension; only `alice` is allowlisted
    address alice = makeAddr("alice");
    address bob   = makeAddr("bob");
    extension.setAllowedToDeposit(pool, alice, true);
    // bob is NOT allowlisted

    // Bob calls addLiquidity with owner = alice
    vm.startPrank(bob);
    // Bob must implement IMetricOmmSwapCallback to supply tokens in callback
    pool.addLiquidity(
        alice,          // owner — allowlisted address used as bypass key
        0,              // salt
        deltas,
        callbackData,
        ""
    );
    vm.stopPrank();

    // Assert: bob bypassed the gate; alice holds LP shares; bob is not allowlisted
    assertFalse(extension.isAllowedToDeposit(pool, bob));
    assertGt(pool.positionShares(alice, 0, binIdx), 0); // alice received shares
}
```
