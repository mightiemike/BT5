### Title
EIP-712 Domain Separator Uses Product ID as `verifyingContract` Instead of Actual Contract Address — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.getDigest` constructs an EIP-712 domain separator using `address(uint160(productId))` as the `verifyingContract` field instead of `address(this)`. This violates the EIP-712 standard, which mandates that `verifyingContract` be the address of the contract that will verify the signature. As a result, order signatures are not bound to the specific `OffchainExchange` contract instance, enabling cross-deployment replay attacks and causing wallets to display a misleading contract address to signers.

---

### Finding Description

In `OffchainExchange.getDigest`, the domain separator is manually constructed as follows:

```solidity
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(uint160(productId))   // ← product ID cast to address, NOT address(this)
    )
);
``` [1](#0-0) 

The EIP-712 specification states:

> `verifyingContract`: The Ethereum address of the contract that will verify the signature.

By substituting `address(uint160(productId))` (e.g., `address(0x1)` for product 1) for `address(this)`, the domain separator is identical for **any** `OffchainExchange` deployment on the same chain that shares the same name (`"Nado"`), version (`"0.0.1"`), and product ID. The actual contract address is entirely absent from the signed material.

The `_TYPE_HASH` constant used here is copied verbatim from `EIP712Upgradeable` and includes the `verifyingContract` slot:

```solidity
bytes32 private constant _TYPE_HASH =
    keccak256(
        "EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"
    );
``` [2](#0-1) 

The resulting digest is then used directly in `_validateOrder` → `_checkSignature` to authenticate every order matched on the exchange:

```solidity
function _checkSignature(
    bytes32 subaccount,
    bytes32 digest,
    address linkedSigner,
    bytes memory signature
) internal view virtual returns (bool) {
    address signer = ECDSA.recover(digest, signature);
    ...
}
``` [3](#0-2) 

---

### Impact Explanation

**Cross-deployment order replay.** The `filledAmounts` mapping that tracks how much of each order has been filled is stored per contract instance:

```solidity
mapping(bytes32 => int128) public filledAmounts;
``` [4](#0-3) 

If the protocol deploys a new `OffchainExchange` contract (e.g., a v2 upgrade), the new instance starts with an empty `filledAmounts` mapping. Because the domain separator does not include `address(this)`, any order signature that was valid on the old contract — including orders that were already partially or fully filled — produces an identical digest on the new contract. An attacker can replay those signatures on the new deployment within the order's `expiration` window, causing trades to execute at stale prices or re-executing orders the user intended only for the prior deployment.

**Wallet phishing / misleading signing prompt.** When a user signs an order, their EIP-712-aware wallet displays the `verifyingContract` field. With the current code, the wallet shows `0x0000000000000000000000000000000000000001` (for product 1) rather than the actual `OffchainExchange` address. This breaks the phishing-prevention guarantee that EIP-712 provides and could be exploited to trick users into signing orders they believe are destined for a different contract.

---

### Likelihood Explanation

The non-compliance is present in every call to `getDigest`, which is invoked for every order matched via `matchOrders`. The cross-deployment replay vector is triggered whenever the protocol deploys a new `OffchainExchange` instance (upgrade, migration, or multi-deployment), which is a realistic operational event for an upgradeable protocol. The wallet display issue affects every user who signs an order through an EIP-712-aware wallet.

---

### Recommendation

Replace `address(uint160(productId))` with `address(this)` in the domain separator construction. If per-product domain separation is desired for replay isolation between products, include `productId` in the struct type string instead:

```solidity
// Correct domain separator
bytes32 domainSeparator = keccak256(
    abi.encode(
        _TYPE_HASH,
        _EIP712NameHash(),
        _EIP712VersionHash(),
        block.chainid,
        address(this)   // actual contract address
    )
);

// Per-product isolation via struct type
string memory structType =
    "Order(bytes32 sender,int128 priceX18,int128 amount,uint64 expiration,uint64 nonce,uint128 appendix,uint32 productId)";
```

Alternatively, use the inherited `_hashTypedDataV4` helper (already used correctly in `EndpointTx.validateSignedTx`) rather than manually constructing the domain separator.

---

### Proof of Concept

1. User signs an order for product 1 on `OffchainExchange` v1 (address `0xAAAA`). The domain separator encodes `verifyingContract = address(1)`, not `0xAAAA`.
2. The order is partially filled; `filledAmounts[digest] = X` on v1.
3. Protocol upgrades: a new `OffchainExchange` v2 is deployed at `0xBBBB`. Its `filledAmounts` mapping is empty.
4. Attacker calls `matchOrders` on v2 with the same signed order. `getDigest` produces the identical digest (same name, version, chainId, `address(1)`, same struct hash).
5. `_checkSignature` recovers the original signer — the signature is accepted.
6. `filledAmounts[digest]` on v2 is zero, so the full order amount is matched again, executing a trade the user never intended on the new contract. [5](#0-4) [6](#0-5)
