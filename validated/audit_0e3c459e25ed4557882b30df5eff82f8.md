Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap()` velocity check silently skipped on every pool's first swap due to uninitialized `lastMidPriceX64` — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap()` gates its entire velocity check behind `if (prevMid != 0)`. Because `lastMidPriceX64` is a mapping slot that defaults to zero and the contract has no factory-called `initialize` function, the guard is completely skipped on the first swap of every pool that registers this extension. Any oracle price — including a manipulated one — reaches the pool unchecked on that first call.

## Finding Description

In `PriceVelocityGuardExtension.beforeSwap()` (lines 53–76), the function reads `prevMid = s.lastMidPriceX64` from storage, unconditionally writes the current mid-price back, then enters the velocity check only if `prevMid != 0`:

```solidity
// lines 53-60
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // reads 0 on first swap
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // state updated unconditionally
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // ← entire guard skipped when prevMid == 0
```

`priceVelocityState` is a `mapping(address pool => PriceVelocityState)` where `PriceVelocityState.lastMidPriceX64` is a `uint128` that EVM initializes to zero. The contract exposes only two admin setters — `setMaxChangePerBlock` and `setLastMidPrice` (both `onlyPoolAdmin`, lines 24–34) — and no `initialize` function callable by the factory. Contrast this with `OracleValueStopLossExtension`, which has an `onlyFactory initialize()` (lines 46–68) that seeds state at pool creation time and is enforced by the factory.

Because no factory-enforced path seeds `lastMidPriceX64` to a non-zero value before the first swap, the exploit path is:

1. Pool is deployed with `PriceVelocityGuardExtension` as a `beforeSwap` extension; pool admin calls `setMaxChangePerBlock(pool, cap)`. `lastMidPriceX64` remains `0`.
2. Attacker (any unprivileged address) pushes a signed Pyth price update in the same block, moving the oracle mid-price arbitrarily far from fair value.
3. Attacker calls `pool.swap(...)` — this is the first swap.
4. `beforeSwap` reads `prevMid = 0`, skips the `if (prevMid != 0)` branch entirely, writes the manipulated mid-price to storage, and returns `IMetricOmmExtensions.beforeSwap.selector` (line 78).
5. The swap executes at the manipulated oracle price; the pool's bin math settles at that price, draining LP token balances in the direction of the skewed quote.
6. From the second swap onward `prevMid` is non-zero and the guard operates normally — but the damage from step 5 is already settled on-chain.

The existing `setLastMidPrice` admin setter is not a sufficient guard: the factory does not call it, the pool admin is not prompted to call it before opening the pool, and the first-swap window is publicly observable and fully unprivileged.

## Impact Explanation

This is bad-price execution reaching a pool swap: the velocity guard — the sole on-chain protection against oracle manipulation between blocks — is entirely absent on the first swap. An attacker who can influence the oracle price in the same block (straightforward on Pyth-backed pools where anyone can push a signed price update) can execute a swap at an arbitrarily skewed mid-price. The pool's bin math settles at that price, causing direct loss of LP principal proportional to pool depth and achievable oracle skew. This matches the allowed impact category of bad-price execution and direct loss of user principal.

**Impact: Medium** — LP funds are at risk on every pool's first swap; magnitude scales with pool depth and oracle skew achievable in one block.

## Likelihood Explanation

Every pool that registers `PriceVelocityGuardExtension` as a `beforeSwap` hook is affected on its very first swap without exception. The condition is structural, not probabilistic. The first-swap window is predictable from public pool creation events. The trigger is fully unprivileged — any address can submit the first swap. On Pyth-backed pools the attacker can push a fresh signed price update in the same transaction, making the oracle skew fully controllable. No special role or privileged access is required.

**Likelihood: Medium** — condition is always present at pool launch; exploitation requires timing the first swap with an oracle push, which is straightforward on Pyth.

## Recommendation

Add a factory-called `initialize(address pool, bytes calldata data)` function to `PriceVelocityGuardExtension` (mirroring `OracleValueStopLossExtension` lines 46–68) that seeds `lastMidPriceX64` from the pool's live oracle price at pool creation time, gated by `onlyFactory`. Alternatively, treat `prevMid == 0` as a fail-closed sentinel that reverts the swap, forcing the pool admin to explicitly seed the price via `setLastMidPrice` before the first swap is permitted.

## Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` as a `beforeSwap` extension.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e16)` (1% per block cap). `lastMidPriceX64` remains `0`.
3. Attacker pushes a Pyth signed price update moving the oracle mid-price 50% from fair value in the same block.
4. Attacker calls `pool.swap(...)` — this is the first swap.
5. `beforeSwap` reads `prevMid = 0` (line 54), skips the `if (prevMid != 0)` branch (line 60), writes the manipulated mid-price to storage (line 57), and returns success (line 78).
6. The swap executes at the 50%-skewed price; the pool pays out tokens at the manipulated rate, draining LP value.
7. Foundry fork test: assert that `PriceVelocityExceeded` is NOT emitted on the first swap even when the oracle price is moved by an amount far exceeding `maxChangePerBlockE18`, then assert it IS emitted on the second swap with the same delta.
