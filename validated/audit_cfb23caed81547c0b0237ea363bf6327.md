### Title
Genesis `ibc/HASH` Denom NFTs Permanently Locked — No Normal Message Path Can Transfer or Burn Them - (`x/nft/types/msgs.go`, `x/nft/keeper/keeper.go`)

---

### Summary

`ValidateGenesis` accepts `ibc/HASH` denom IDs via `ValidateDenomIDWithIBC`, but every user-facing message handler (`MsgTransferNFT`, `MsgBurnNFT`, `MsgEditNFT`) calls the stricter `ValidateDenomID`, which rejects the `ibc/` prefix at the message-validation layer. Additionally, the keeper's `BurnNFT` enforces an `IsDenomCreator` check that only the nft-transfer module account can pass. The result is that any NFT minted under an `ibc/HASH` denom loaded at genesis is permanently immovable and indestructible through all normal user-signed message paths.

---

### Finding Description

**Genesis accepts `ibc/HASH` denom IDs:**

`ValidateGenesis` in `x/nft/types/genesis.go` calls `ValidateDenomIDWithIBC`: [1](#0-0) 

`ValidateDenomIDWithIBC` in `x/nft/types/validation.go` explicitly allows the `ibc/` prefix and validates the hex hash: [2](#0-1) 

`InitGenesis` in `x/nft/genesis.go` calls `ValidateGenesis`, then `k.SetDenom` and `k.SetGenesisCollection`, so an `ibc/HASH` denom with NFTs is fully committed to state: [3](#0-2) 

**Message layer rejects `ibc/HASH` denom IDs:**

`MsgTransferNFT.ValidateBasic` calls `ValidateDenomID` (not `ValidateDenomIDWithIBC`): [4](#0-3) 

`MsgBurnNFT.ValidateBasic` does the same: [5](#0-4) 

`ValidateDenomID` rejects `ibc/HASH` on two independent grounds: the `/` character fails `IsAlphaNumeric`, and the 68-character length exceeds `MaxDenomLen = 64`: [6](#0-5) 

Both checks fire before the message ever reaches the keeper, so the rejection is unconditional.

**Keeper `BurnNFT` adds a second independent lock:**

Even if the `ValidateBasic` barrier were somehow bypassed, `BurnNFT` in `x/nft/keeper/keeper.go` calls `IsDenomCreator`, which requires the caller to be the denom's registered creator — the nft-transfer module account for IBC voucher denoms: [7](#0-6) 

A regular NFT owner is never the denom creator, so `BurnNFT` always returns an error for them. `BurnNFTUnverified` (which skips this check) is only callable internally by the nft-transfer keeper, not via any user-signed message. [8](#0-7) 

---

### Impact Explanation

Any NFT minted under an `ibc/HASH` denom that was loaded at genesis is:

- **Non-transferable on-chain**: `MsgTransferNFT.ValidateBasic` rejects the denom ID before the keeper is reached.
- **Non-burnable by the owner**: `MsgBurnNFT.ValidateBasic` rejects the denom ID, and even if bypassed, `BurnNFT` requires `IsDenomCreator`.
- **Non-editable**: `MsgEditNFT.ValidateBasic` also calls `ValidateDenomID` (line 135 of msgs.go), blocking that path too.

The only theoretical escape is the IBC send path via the nft-transfer module's `MsgTransfer`, but that requires: (a) an active, open IBC channel, (b) the class trace registered in the nft-transfer module state, and (c) the counterparty chain to be live and responsive. If any of these conditions are absent — which is likely for a genesis-bootstrapped chain that has not yet established IBC channels — the NFTs are permanently locked with no recovery path.

---

### Likelihood Explanation

The `ValidateGenesis` comment in `x/nft/types/genesis.go` explicitly notes "IBC-aware (ibc/{hash} vouchers from x/nft-transfer)" as the rationale for using `ValidateDenomIDWithIBC`, confirming this is an intended and documented code path. A chain upgrade, state migration, or genesis export/import that includes IBC voucher NFTs will silently create permanently locked assets. The inconsistency is not guarded by any runtime check.

---

### Recommendation

Replace `ValidateDenomID` with `ValidateDenomIDWithIBC` in the `ValidateBasic` methods of `MsgTransferNFT`, `MsgBurnNFT`, and `MsgEditNFT` in `x/nft/types/msgs.go`. This aligns the message-layer validation with the genesis-layer validation and allows owners of IBC voucher NFTs to exercise their normal ownership rights.

---

### Proof of Concept

1. Construct a genesis state with a `Collection` whose `Denom.Id = "ibc/AABBCC...FF"` (valid 64-hex-char hash, total length 68).
2. Populate the collection with one NFT owned by a regular user address.
3. Start the chain — `InitGenesis` → `ValidateGenesis` (passes via `ValidateDenomIDWithIBC`) → `SetDenom` + `SetGenesisCollection` commits the state.
4. Submit `MsgTransferNFT{DenomId: "ibc/AABB...FF", ...}` signed by the NFT owner → rejected at `ValidateBasic` with `ErrInvalidDenom` (length > 64 and non-alphanumeric `/`).
5. Submit `MsgBurnNFT{DenomId: "ibc/AABB...FF", ...}` signed by the NFT owner → same rejection.
6. Confirm the NFT remains in state, owned by the user, with no message path available to move or destroy it.

func ValidateDenomID(denomID string) error {
	if len(denomID) < MinDenomLen || len(denomID) > MaxDenomLen {
		return sdkerrors.Wrapf(ErrInvalidDenom, "the length of denom(%s) only accepts value [%d, %d]", denomID, MinDenomLen, MaxDenomLen)

		return sdkerrors.Wrapf(ErrInvalidDenom, "the denom(%s) only accepts lowercase alphanumeric characters, and begin with an english letter", denomID)
	}
	return nil
}

func ValidateDenomIDWithIBC(denomID string) error {

			return sdkerrors.Wrapf(ErrInvalidDenom, "the hash of ibc denom(%s) must be valid hex", denomID)
		}

		return nil
	}

}

```go
func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}
```
