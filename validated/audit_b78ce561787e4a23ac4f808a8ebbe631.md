### Title
Irrevocable `deployer` Role Cannot Be Transferred or Revoked — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.sol` stores a privileged `deployer` address that is assigned once during `initialize()` and is never updatable. Unlike the `owner` role (which inherits OpenZeppelin's `transferOwnership`), there is no `updateDeployer`, `setDeployer`, or equivalent function anywhere in the codebase. The `deployer` controls several high-impact protocol operations, including queuing product risk-parameter changes and triggering product delisting. If the deployer key is compromised or governance needs to migrate to a multisig, there is no on-chain remediation path.

---

### Finding Description

`ContractOwner` inherits `OwnableUpgradeable` and correctly exposes `transferOwnership` for the `owner` role. However, it also defines a separate `deployer` state variable: [1](#0-0) 

Set once in `initialize()`: [2](#0-1) 

And enforced by a dedicated modifier: [3](#0-2) 

The `onlyDeployer` modifier gates five functions:

| Function | Lines | Impact |
|---|---|---|
| `submitSpotAddOrUpdateProductCall` | 91–115 | Queues spot product risk/config changes |
| `submitPerpAddOrUpdateProductCall` | 117–137 | Queues perp product risk/config changes |
| `clearSpotAddOrUpdateProductCalls` | 139–141 | Clears pending spot config queue |
| `clearPerpAddOrUpdateProductCalls` | 143–145 | Clears pending perp config queue |
| `delistProduct` | 361–380 | Submits a `DelistProduct` slow-mode transaction that force-settles all open positions at a caller-supplied price | [4](#0-3) 

A full search of the file confirms there is no `updateDeployer`, `setDeployer`, or `transferDeployer` function. The `deployer` is permanently bound to the address that called `initialize()`.

---

### Impact Explanation

The most severe consequence is through `delistProduct`. A compromised deployer can submit a `DelistProduct` slow-mode transaction with an arbitrary `priceX18`, force-settling all open perp positions for a given product at that manipulated price. This directly corrupts the `vQuoteBalance` of every affected subaccount in `PerpEngine`, causing irreversible accounting damage and potential fund loss for all traders holding that product. [4](#0-3) 

Secondary impact: a compromised deployer can queue malicious `SpotAddOrUpdateProductCall` or `PerpAddOrUpdateProductCall` entries with extreme `RiskStore` values (e.g., zero initial/maintenance weights), which, once approved by the owner via `addOrUpdateProducts`, would allow undercollateralized borrowing or block legitimate liquidations. [5](#0-4) 

Even absent active compromise, the protocol cannot progressively improve its security posture (e.g., migrate the deployer from an EOA to a multisig) because there is no transfer mechanism.

---

### Likelihood Explanation

The deployer is a long-lived EOA or hot wallet used for operational tasks (product listings, delisting). Long-lived operational keys are a well-known target. The inability to rotate the key means any future compromise is permanent and unrecoverable on-chain. Likelihood is medium: key compromise is not routine, but the permanent nature of the role and the absence of any rotation path elevates the risk materially above a contract with a rotatable role.

---

### Recommendation

Add an `updateDeployer` function restricted to `onlyOwner`:

```solidity
function updateDeployer(address newDeployer) external onlyOwner {
    require(newDeployer != address(0), "zero address");
    deployer = newDeployer;
}
```

This mirrors the pattern already used for `submitter` in `BaseProxyManager`: [6](#0-5) 

Additionally, consider adding a one-time-set guard on `initialize` to prevent the deployer from being set to a zero address, and emit an event on deployer changes for off-chain monitoring.

---

### Proof of Concept

1. Protocol deploys `ContractOwner` with `deployer = 0xAlice` (an EOA).
2. Alice's private key is compromised.
3. Attacker calls `delistProduct([productId], [manipulatedPriceX18], [subaccounts])` from Alice's address.
4. `_submitSlowModeTransaction(IEndpoint.TransactionType.DelistProduct, ...)` is submitted to the endpoint slow-mode queue.
5. The endpoint processes the transaction, calling `Clearinghouse.delistProduct`, which force-settles all listed subaccounts at `manipulatedPriceX18`.
6. Subaccount balances in `PerpEngine` are permanently corrupted.
7. The protocol owner has no on-chain mechanism to revoke Alice's `deployer` role before or after the attack. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
