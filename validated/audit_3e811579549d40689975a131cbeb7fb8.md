### Title
u64 Multiplication Overflow in `op_unknown` Cost Multiplier Bypasses Cost Limit - (File: src/more_ops.rs)

### Summary
In `op_unknown` (`src/more_ops.rs`), the final cost is computed by multiplying a pre-validated `cost` (u64) by `cost_multiplier + 1` (also u64, derived from up to 4 opcode bytes). In Rust release builds, this multiplication silently wraps on u64 overflow. The post-multiplication overflow check only rejects values `> u32::MAX`, so a wrapped result of 0 (or any value ≤ u32::MAX) passes and is returned as the operation's cost. An attacker can craft opcode bytes and arguments to trigger this wrap, causing the opcode to report cost 0 to the `run_program` accumulator — a consensus-breaking cost undercharge.

### Finding Description

`op_unknown` computes cost in three stages:

**Stage 1 — base cost (bounded by `max_cost`):** [1](#0-0) 

**Stage 2 — pre-multiplication guard:** [2](#0-1) 

**Stage 3 — post-multiplication guard:** [3](#0-2) 

The `cost_multiplier` is extracted from the opcode atom bytes via `u32_from_u8`, so it is at most `u32::MAX = 4,294,967,295`, making `cost_multiplier + 1` at most `2^32 = 4,294,967,296`. [4](#0-3) 

`Cost` is defined as `u64`: [5](#0-4) 

The pre-multiplication `check_cost` at line 260 only ensures `cost ≤ max_cost`. In Chia's consensus, `max_cost` can be up to ~11 billion (> `u32::MAX = 4,294,967,295`). Therefore `cost` can legally be, e.g., `2^32 = 4,294,967,296` before the multiplication.

**Concrete overflow:**
- `cost = 2^32`, `cost_multiplier + 1 = 2^32`
- `cost * (cost_multiplier + 1) = 2^64`
- In Rust release mode, `2^64 mod 2^64 = 0`
- `0 > u32::MAX` → **false** → `Ok(Reduction(0, nil))` is returned

The `assert!(cost > 0)` at line 258 fires on the **pre-multiplication** value and does not protect the post-multiplication result. [2](#0-1) 

### Impact Explanation

The cost returned by `op_unknown` is accumulated into the program's total cost by `run_program`. A wrapped cost of 0 means the opcode consumes no budget. An attacker can embed multiple such opcodes in a CLVM program, each with carefully chosen bytes and arguments, causing the program's total reported cost to be far below its true computational cost. This allows programs that should be rejected (cost exceeds block limit) to be accepted — a **consensus-breaking cost undercharge**. Nodes running release builds accept the program; nodes with correct cost accounting reject it, causing a chain split.

### Likelihood Explanation

The attacker controls the opcode bytes (the `cost_multiplier` field) and the argument list (which determines the base cost). Both are directly encoded in attacker-supplied CLVM bytes. Achieving `cost = 2^32` via cost function 1 requires:

`99 + n × 320 + byte_count × 3 = 4,294,967,296`

This is satisfiable with a specific number of arguments and total argument byte length. The opcode bytes encoding `cost_multiplier = u32::MAX` with `cost_function = 1` are straightforward to construct. The attack requires no special permissions, no compromised nodes, and no social engineering — only crafted CLVM bytes submitted to the network.

### Recommendation

Replace the bare multiplication with a checked or saturating multiply, and treat overflow as an error:

```rust
// Instead of:
cost *= cost_multiplier + 1;

// Use:
cost = cost.checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the fix applied in the referenced Solidity report (using `SafeMath` for the addition), and is consistent with how `op_add`'s fast path already uses `checked_add`: [6](#0-5) 

### Proof of Concept

Craft an opcode atom where:
- Bytes `[0..len-1]` encode `u32::MAX` (i.e., `[0xff, 0xff, 0xff, 0xff]`)
- Last byte has bits `[7:6] = 01` → `cost_function = 1`

This gives `cost_multiplier = 4,294,967,295`, `cost_multiplier + 1 = 2^32`.

Pass arguments whose total byte length satisfies:
`99 + n × 320 + byte_count × 3 = 4,294,967,296`

(e.g., `n = 0`, `byte_count = (4,294,967,296 - 99) / 3 ≈ 1,431,655,732` bytes of atom data)

After the loop, `cost = 2^32`. The `check_cost(cost, max_cost)` passes (2^32 < 11 billion). Then `cost *= 2^32` wraps to 0 in release mode. The check `0 > u32::MAX` is false, so `Ok(Reduction(0, nil))` is returned — the opcode costs nothing. [7](#0-6)
