### Title
`op_mod` Uses Wrong Cost Constants (`DIV_BASE_COST`/`DIV_COST_PER_BYTE` Instead of `DIVMOD_BASE_COST`/`DIVMOD_COST_PER_BYTE`), Causing Systematic Cost Undercharging - (File: `src/more_ops.rs`)

---

### Summary

`op_mod` in `src/more_ops.rs` charges execution cost using `DIV_BASE_COST` (988) and `DIV_COST_PER_BYTE` (4) — the constants defined for the cheaper `op_div` operator — instead of `DIVMOD_BASE_COST` (1116) and `DIVMOD_COST_PER_BYTE` (6), which are the correct constants for the equivalent computational work. Because `mod_floor` internally performs the same full division as `div_mod_floor`, the actual CPU cost of `op_mod` matches `op_divmod`, not `op_div`. This is a direct semantic mismatch: the wrong cost parameter set is applied to the wrong operation, analogous to the StarBaseDCA bug where the wrong amount variable (`inAmount` total instead of per-cycle) was passed to a critical function.

---

### Finding Description

In `src/more_ops.rs`, three related operators are defined:

- `op_div` — computes quotient only; uses `DIV_BASE_COST = 988`, `DIV_COST_PER_BYTE = 4`
- `op_divmod` — computes quotient **and** remainder; uses `DIVMOD_BASE_COST = 1116`, `DIVMOD_COST_PER_BYTE = 6`
- `op_mod` — computes remainder only; **incorrectly uses** `DIV_BASE_COST = 988`, `DIV_COST_PER_BYTE = 4`

The cost line in `op_mod` is:

```rust
let cost = DIV_BASE_COST + ((a0_len + a1_len) as Cost) * DIV_COST_PER_BYTE;
``` [1](#0-0) 

The correct constants for this operation are `DIVMOD_BASE_COST` and `DIVMOD_COST_PER_BYTE`, as defined alongside `DIV_BASE_COST`:

```rust
const DIVMOD_BASE_COST: Cost = 1116;
const DIVMOD_COST_PER_BYTE: Cost = 6;

const DIV_BASE_COST: Cost = 988;
const DIV_COST_PER_BYTE: Cost = 4;
``` [2](#0-1) 

The `op_mod` implementation calls `a0.mod_floor(&a1)`, which in `num_bigint` internally calls `div_mod_floor` and discards the quotient — performing the **identical** CPU work as `op_divmod`. The same mismatch is present in the Malachite path `op_mod_malachite`: [3](#0-2) 

`op_mod` is a fully registered, production operator wired into the dispatch table: [4](#0-3) 

---

### Impact Explanation

For maximum-size operands (`a0_len = 256`, `a1_len = 1024`, total = 1280 bytes):

- **Cost charged by `op_mod`**: `988 + 1280 × 4 = 6,108`
- **Cost that should be charged**: `1116 + 1280 × 6 = 8,796`
- **Undercharge per call**: **2,688 cost units (~44% below correct)**

An attacker who crafts a CLVM program with many `mod` calls on large operands will consume significantly more CPU than the cost budget permits. Because `check_cost` is called against the underestimated value, programs that should be rejected as exceeding `max_cost` will be accepted. This is a **consensus-critical cost undercharging** vulnerability: different nodes enforcing the same nominal cost limit will accept programs that consume more real CPU than intended, enabling resource exhaustion or consensus divergence between nodes with different cost-enforcement implementations. [5](#0-4) 

---

### Likelihood Explanation

`op_mod` is unconditionally available in the default CLVM dialect — no softfork flag or extension is required. Any CLVM program submitted to the Chia mempool or block can invoke it. An attacker needs only to submit a valid CLVM coin spend that chains many `mod` calls with large operands. The operand size limits (256 and 1024 bytes) are enforced, but within those limits the undercharge is maximized at 44%. This is straightforward to exploit with attacker-controlled CLVM bytes.

---

### Recommendation

Replace `DIV_BASE_COST` and `DIV_COST_PER_BYTE` with `DIVMOD_BASE_COST` and `DIVMOD_COST_PER_BYTE` in both `op_mod` and `op_mod_malachite`:

```rust
// op_mod (both paths)
let cost = DIVMOD_BASE_COST + ((a0_len + a1_len) as Cost) * DIVMOD_COST_PER_BYTE;
```

Add a cost-regression test that verifies `op_mod` charges at least as much as `op_divmod` for identical inputs, since `mod_floor` performs a strict superset of the work done by `div_floor`.

---

### Proof of Concept

The following attacker-controlled CLVM bytes trigger the undercharging. Using the operator test format from `op-tests/`:

```
; op_mod with max-size operands: a0 = 256-byte positive int, a1 = 1024-byte positive int
; Charged cost: 988 + (256+1024)*4 = 6108
; Correct cost: 1116 + (256+1024)*6 = 8796
; Undercharge: 2688 per invocation

mod <256-byte-atom> <1024-byte-atom> => <remainder> | 6108
```

A program chaining N such calls will be accepted at cost `N × 6108` but consume CPU equivalent to `N × 8796`. With a block cost limit of, e.g., 11,000,000,000, an attacker can include approximately `11e9 / 6108 ≈ 1,800,000` `mod` calls where only `11e9 / 8796 ≈ 1,250,000` should be permitted — a ~44% excess CPU consumption per block accepted by the cost checker.     if flags.contains(ClvmFlags::MALACHITE) {

    }

    let (a1, a1_len) = int_atom(a, v1, "mod")?;
    if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
        return Err(EvalErr::InvalidOpArg(input, "mod".to_string()));
    }
    if a0_len > 256 || a1_len > 1024 {
        return Err(EvalErr::InvalidOpArg(input, "mod".to_string()));

    Ok(Reduction(cost + c, q))
