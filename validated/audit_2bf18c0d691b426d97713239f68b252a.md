### Title
`DepositAllowlistExtension` checks `owner` but ignores `sender`, allowing unauthorized depositors to bypass the allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual on-chain caller who pays the tokens) and only validates the `owner` argument (the LP-position holder). Because `addLiquidity` lets the caller freely choose any `owner` address, an address that is **not** on the allowlist can deposit tokens into the pool by nominating an allowlisted address as `owner`, bypassing the guard entirely.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

```
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

`msg.sender` is the actual caller who will be required to pay tokens via the `metricOmmModifyLiquidityCallback`. `owner` is a caller-supplied parameter that determines who holds the resulting LP position.

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (the first `address` parameter is unnamed `address,`) and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

An unauthorized address `Bob` cannot call `addLiquidity(owner=Bob, …)` because `allowedDepositor[pool][Bob]` is false. However, `Bob` **can** call `addLiquidity(owner=Alice, …)` where `Alice` is allowlisted. The extension checks `allowedDepositor[pool][Alice]` → true → passes. Bob's callback pays the tokens; Alice receives the LP position. The allowlist is fully bypassed.

This is structurally identical to the MEME404 bug: the invariant (only authorized addresses may deposit) is enforced on one path (`owner == caller`) but silently skipped on the alternate path (`owner != caller`).

---

### Impact Explanation

The deposit allowlist is an admin-configured access-control boundary. Its bypass allows:

1. **Compliance violation**: A sanctioned or KYC-rejected address can deposit funds into a restricted pool by nominating any allowlisted address as `owner`. The pool receives liquidity from an unauthorized source.
2. **Allowlist rendered ineffective**: Any pool that relies on `DepositAllowlistExtension` to enforce a closed LP set (e.g., institutional pools, permissioned vaults) can be infiltrated by any external party willing to gift the LP position to an authorized address.
3. **Two-step full bypass with collusion**: If `Bob` and `Alice` collude, `Bob` deposits via `owner=Alice`, then `Alice` calls `removeLiquidity` and returns the proceeds to `Bob`. The allowlist is circumvented end-to-end with no residual restriction.

This is an admin-boundary break: a pool-admin-configured restriction is bypassed by an unprivileged path.

---

### Likelihood Explanation

- Authorized depositor addresses are visible on-chain (emitted via `AllowedToDepositSet` events).
- No special privilege is required; any EOA or contract can call `addLiquidity` with an arbitrary `owner`.
- The only friction is that the unauthorized depositor must trust the `owner` to cooperate for withdrawal, or accept the loss of principal (still achieves the deposit bypass for compliance purposes).

Likelihood: **Medium** (requires a cooperating allowlisted address for full fund recovery, but the deposit bypass itself is unconditional).

---

### Recommendation

Check `sender` (the actual payer) instead of, or in addition to, `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    address pool_ = msg.sender;
    if (!allowAllDepositors[pool_]
        && !allowedDepositor[pool_][sender]   // ← check the actual caller/payer
        && !allowedDepositor[pool_][owner])   // optionally also gate by owner
    {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate by LP-position holder only, document this explicitly and rename the mapping/event to `allowedOwner` to avoid confusion. If the intent is to gate by depositor (payer), `sender` must be checked.

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  allowedDepositor[pool][Bob]   = false   // Bob is NOT authorized

Step 1 (expected revert):
  Bob calls pool.addLiquidity(owner=Bob, ...)
  → extension checks allowedDepositor[pool][Bob] → false → reverts ✓

Step 2 (bypass):
  Bob calls pool.addLiquidity(owner=Alice, ...)
  → extension checks allowedDepositor[pool][Alice] → true → passes ✓
  → Bob's metricOmmModifyLiquidityCallback pays token0/token1
  → Alice receives the LP position
  → Bob has deposited into the restricted pool without being on the allowlist ✓

Step 3 (full bypass with collusion):
  Alice calls pool.removeLiquidity(owner=Alice, ...)
  → Alice receives token0/token1 back
  → Alice transfers proceeds to Bob off-chain
  → Net effect: Bob deposited and withdrew from a pool he was explicitly barred from
``` [2](#0-1) [3](#0-2) [4](#0-3)
