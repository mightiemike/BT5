### Title
`maxBuy + 1` Ceiling Rounding in Spot Liability Liquidation Allows Removing More Liability Than Available Quote + Insurance Can Cover — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `_assertLiquidationAmount`, when computing the maximum liquidatable amount for a **spot liability** (`txn.amount < 0`, `engine == address(spotEngine)`), the protocol applies a `+1` ceiling rounding to `maxBuy`. This allows the liquidation of one more unit of spot liability than `quoteBalance.amount + insurance` can actually cover at `liquidationPrice`. When `_handleLiquidationPayment` executes, the actual payment exceeds the total available funds. `updateQuoteFromInsurance` can only cover up to `insurance`, leaving a residual negative quote balance (bad debt) on the liquidatee — the spot liability was removed without full compensation.

---

### Finding Description

In `_assertLiquidationAmount`, for the spot liability path:

```solidity
(int128 liquidationPrice, ) = getLiqPriceX18(txn.productId, txn.amount);
ISpotEngine.Balance memory quoteBalance = spotEngine.getBalance(QUOTE_PRODUCT_ID, txn.liquidatee);
int128 maxBuy = (quoteBalance.amount + insurance).div(liquidationPrice);
maxBuy = MathHelper.max(maxBuy + 1, 0);          // <-- ceiling rounding
maxLiquidatable = MathHelper.max(spotAmount, -maxBuy);
``` [1](#0-0) 

`maxBuy` is computed as `floor((quoteBalance.amount + insurance) / liquidationPrice)`. The `+1` adjustment makes it a ceiling, allowing the liquidator to submit `txn.amount = -(maxBuy + 1)`.

By the property of floor division:
- `liquidationPrice * maxBuy <= quoteBalance.amount + insurance`
- `liquidationPrice * (maxBuy + 1) > quoteBalance.amount + insurance`

So the actual payment for `maxBuy + 1` units **strictly exceeds** `quoteBalance.amount + insurance`.

In `_handleLiquidationPayment`, the spot liability is fully removed and the liquidatee's quote is debited by `liquidationPayment = liquidationPriceX18 * txn.amount` (negative): [2](#0-1) 

Then `updateQuoteFromInsurance` is called to cover any resulting negative quote balance from insurance:

```solidity
function updateQuoteFromInsurance(bytes32 subaccount, int128 insurance) external returns (int128) {
    ...
    int128 topUpAmount = MathHelper.max(MathHelper.min(insurance, -balanceAmount), 0);
    insurance -= topUpAmount;
    ...
}
``` [3](#0-2) 

Since the deficit `= liquidationPrice * (maxBuy + 1) - quoteBalance.amount > insurance`, `updateQuoteFromInsurance` can only cover `insurance`, leaving a residual negative quote balance of `deficit - insurance > 0` on the liquidatee. The spot liability was fully removed, but the protocol could not fully compensate for it.

The identical `maxBuy + 1` pattern also exists in the spread basis path: [4](#0-3) 

---

### Impact Explanation

The protocol removes a spot liability of `maxBuy + 1` units but only receives compensation for `maxBuy` units worth. The residual negative quote balance — up to `liquidationPrice - 1` per liquidation call — constitutes unrecoverable bad debt. The insurance fund is fully drained in the process. Concretely:

- `quoteBalance = 100`, `insurance = 50`, `liquidationPrice = 60`
- `maxBuy = floor(150/60) = 2`, `maxBuy + 1 = 3`
- Payment required: `3 * 60 = 180`
- Available: `100 + 50 = 150`
- Residual bad debt: `30` (insurance fully consumed, liquidatee still owes 30)

The spot liability (3 units) is fully removed from the liquidatee's account, but the protocol absorbs a 30-unit quote shortfall with no recourse.

---

### Likelihood Explanation

Any liquidator interacting through `Endpoint.sol` → `Clearinghouse.liquidateSubaccount` → `ClearinghouseLiq.liquidateSubaccountImpl` can trigger this path whenever a subaccount holds a negative spot balance and its `quoteBalance + insurance` is not an exact multiple of `liquidationPrice`. This is the common case for any non-trivially priced asset. No special privileges are required — any unprivileged liquidator can submit a `LiquidateSubaccount` transaction with `txn.amount = -(maxBuy + 1)`. [5](#0-4) 

---

### Recommendation

Replace the ceiling rounding with a floor-based bound that strictly respects the available funds. Instead of `maxBuy + 1`, use `maxBuy` directly:

```solidity
int128 maxBuy = (quoteBalance.amount + insurance).div(liquidationPrice);
maxBuy = MathHelper.max(maxBuy, 0);  // no +1
maxLiquidatable = MathHelper.max(spotAmount, -maxBuy);
```

This ensures `liquidationPrice * maxBuy <= quoteBalance.amount + insurance`, so `updateQuoteFromInsurance` can always fully cover the resulting negative quote balance. Apply the same fix to the spread basis path at line 140.

---

### Proof of Concept

1. Liquidatee has: `spotBalance = -3` (liability), `quoteBalance = 100`, no perp positions.
2. `insurance = 50`, `liquidationPrice = 60` (oracle price × maintenance weight factor).
3. `_assertLiquidationAmount` computes:
   - `maxBuy = floor((100 + 50) / 60) = 2`
   - `maxBuy + 1 = 3`
   - `maxLiquidatable = max(-3, -3) = -3` → liquidator can submit `txn.amount = -3`
4. `_handleLiquidationPayment` executes:
   - `spotEngine.updateBalance(productId, liquidatee, +3)` → spot liability cleared
   - `spotEngine.updateBalance(QUOTE_PRODUCT_ID, liquidatee, -180)` → quote: `100 - 180 = -80`
5. `updateQuoteFromInsurance(liquidatee, 50)`:
   - Covers `min(50, 80) = 50` → quote: `-80 + 50 = -30`
   - Returns `insurance = 0`
6. `insurance = 0 + liquidationFees` (small)
7. **Result**: Spot liability fully removed, but liquidatee has `-30` residual quote (bad debt). Insurance fund drained. Protocol lost 30 units of quote value. [6](#0-5) [7](#0-6)
