### Title
Silent Swallow of Failed Slow-Mode Deposit Permanently Locks User Collateral in Clearinghouse - (File: `core/contracts/Endpoint.sol`)

---

### Summary

When a user deposits collateral via `depositCollateralWithReferral`, tokens are transferred to the `Clearinghouse` immediately and atomically. A `DepositCollateral` slow-mode transaction is then queued for later execution. If that slow-mode transaction fails during execution, the `_executeSlowModeTransaction` catch block silently discards the error — and the tokens remain in the `Clearinghouse` with no on-chain path for the user to recover them. The codebase itself documents the removed refund path with the comment `// try return funds now removed`.

---

### Finding Description

The deposit flow in `Endpoint.sol` is a two-phase operation:

**Phase 1 — Immediate token transfer (atomic, in `depositCollateralWithReferral`):**

`handleDepositTransfer` pulls tokens from the caller and pushes them directly to the `Clearinghouse` in the same call. [1](#0-0) 

**Phase 2 — Deferred balance credit (queued slow-mode tx):**

A `SlowModeTx` of type `DepositCollateral` is enqueued with a 3-day delay. Only when this tx is executed does `clearinghouse.depositCollateral` actually credit the user's internal balance. [2](#0-1) 

**The silent-catch problem:**

`_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch`. If the inner call reverts for any reason, the error is silently discarded. The comment on line 226 explicitly acknowledges that a fund-return path existed here and was removed: [3](#0-2) 

**The slow-mode deposit handler:**

When executed, `processSlowModeTransactionImpl` calls `clearinghouse.depositCollateral`, which can revert if, for example, `_decimals(txn.productId)` reverts (token address is zero for a delisted/misconfigured product), `spotEngine.updateBalance` reverts, or any overflow check fires. [4](#0-3) [5](#0-4) 

If any of these revert, the slow-mode tx is deleted from the queue (`delete slowModeTxs[_slowModeConfig.txUpTo++]`), the catch block swallows the error, and the tokens — already sitting in the `Clearinghouse` — are never credited to the user's subaccount. There is no user-callable or admin-callable recovery function.

---

### Impact Explanation

A user's deposited collateral is permanently locked inside the `Clearinghouse` contract. The user's on-chain subaccount balance is never incremented. The tokens cannot be recovered by the user, and there is no admin sweep or emergency withdrawal function scoped to this scenario. The asset delta is: user loses `amount` of `token` with zero protocol-side credit.

---

### Likelihood Explanation

The window between Phase 1 (immediate transfer) and Phase 2 (slow-mode execution, up to 3 days later) creates a realistic race condition. Concrete triggers include:

- A spot product's token address being set to `address(0)` or reconfigured between deposit and execution, causing `_decimals` to revert with `ERR_INVALID_PRODUCT`.
- An overflow in `int128(txn.amount) * int128(multiplier)` if `amount` is near `INT128_MAX` and `multiplier > 1`.
- Any revert inside `spotEngine.updateBalance` due to engine state corruption or a future upgrade.

The 3-day slow-mode delay window makes the race condition non-trivial to exploit but also non-trivial to avoid.

---

### Recommendation

1. **Restore the refund path**: When `processSlowModeTransaction` fails for a `DepositCollateral` tx, transfer the tokens back to the original depositor from the `Clearinghouse`. The removed code path should be reinstated.
2. **Alternatively, defer the token transfer**: Do not transfer tokens to the `Clearinghouse` until the slow-mode tx is successfully executed. Hold them in the `Endpoint` during the delay window and only forward on success.
3. **Add an admin recovery function**: At minimum, add an owner-callable function to refund a specific slow-mode deposit index if the corresponding tx failed.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
2. `handleDepositTransfer` immediately moves `amount` tokens: `user → Endpoint → Clearinghouse`. [6](#0-5) 
3. A `SlowModeTx` is queued at index `txCount`. [7](#0-6) 
4. Between deposit and execution, the spot product's token is reconfigured so `spotEngine.getConfig(productId).token == address(0)`.
5. After 3 days, user (or sequencer) calls `executeSlowModeTransaction()`.
6. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(...)` inside a `try`.
7. Inside, `clearinghouse.depositCollateral` calls `_decimals(productId)` → `require(address(token) != address(0))` → **reverts**. [8](#0-7) 
8. The `catch` block fires. `gasleft()` checks pass. Execution continues silently. [9](#0-8) 
9. The slow-mode tx is deleted. The user's subaccount balance is `0`. The `amount` tokens remain in the `Clearinghouse` with no recovery path.

            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {

                    }

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
