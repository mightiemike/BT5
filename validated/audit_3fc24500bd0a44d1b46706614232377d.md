### Title
Unlimited-Allocator Double-Copy in `ser_2026` / `to_bytes_2026()` Enables Memory Amplification Without Heap Cap — (`src/serde_2026/ser.rs`, `wheel/src/api.rs`)

---

### Summary

The `to_bytes_2026()` serialization path in the Python API chains two separate unlimited-allocator allocations on attacker-controlled CLVM data, with no heap cap enforced at either step. Combined with the backrefs deserialization format's ability to expand compact wire bytes into a large in-memory tree, this creates a multi-stage memory amplification path analogous to the JSON-RPC double-parse vulnerability.

---

### Finding Description

**Stage 1 — Backrefs deserialization (unlimited allocator):**

`deser_backrefs` and `deser_auto` both construct `Allocator::new()`, which sets `heap_limit = u32::MAX as usize` (≈ 4 GB): [1](#0-0) [2](#0-1) 

The backrefs format (`node_from_bytes_backrefs`) allows a compact serialized blob to expand into a much larger in-memory tree via back-references. A small wire payload can produce a tree that fills the 4 GB allocator limit.

**Stage 2 — `clvm_tree_to_lazy_node` (first unlimited copy):**

`to_bytes_2026()` in `Program` calls `clvm_tree_to_lazy_node(self)`, which allocates a fresh `Allocator::new()` (unlimited) and copies the entire Python CLVM tree into it: [3](#0-2) [4](#0-3) 

**Stage 3 — `ser_2026` → `intern_tree` (second unlimited copy):**

`ser_2026` calls `serialize_2026` → `SerializerState::new`, which unconditionally calls `intern_tree(allocator, node)`: [5](#0-4) 

`intern_tree` delegates to `intern_tree_limited(source, node, u32::MAX as usize)` — another 4 GB-capped allocator — creating a full second copy of the tree: [6](#0-5) 

The `intern_tree_limited` call inside `SerializerState::new` is never given a caller-supplied heap cap; the `intern_tree_limited` variant exists but is not used here: [7](#0-6) 

**Result:** A single `to_bytes_2026()` call on a large CLVM program holds three simultaneous allocations — the deserialized source tree, the `clvm_tree_to_lazy_node` copy, and the `intern_tree` copy — all in unlimited allocators, with no aggregate cap.

---

### Impact Explanation

An attacker submits a crafted CLVM blob using the backrefs format. A compact ≈1 MB wire payload can expand to a large in-memory tree. If any downstream caller (e.g., chia-blockchain's mempool or generator runner) invokes `to_bytes_2026()` on the deserialized program, three simultaneous unlimited allocations are held. The `intern_tree` step also allocates three `HashMap` structures (`node_to_interned`, `atom_to_interned`, `pair_to_interned`) proportional to tree size, adding further overhead beyond the raw node storage. [8](#0-7) 

The `run_serialized_chia_program` path does have an optional 500 MB heap limit gated on `ClvmFlags::LIMIT_HEAP`, but the `ser_2026` / `to_bytes_2026()` path has no equivalent guard: [9](#0-8) 

---

### Likelihood Explanation

The `to_bytes_2026()` method is a public Python API on `Program`, the primary user-facing type. Any Chia node component that deserializes a coin-spend program from the mempool and then re-serializes it to the 2026 format (e.g., for caching, forwarding, or storage) will traverse this path. The backrefs format is the default wire format for CLVM programs on Chia, making attacker-controlled input routine.

---

### Recommendation

1. Replace the `intern_tree` call inside `SerializerState::new` with `intern_tree_limited`, passing a caller-supplied or source-allocator-derived heap cap: [5](#0-4) 

2. Apply a heap limit to `clvm_tree_to_lazy_node`'s internal allocator, mirroring the `LIMIT_HEAP` pattern used in `run_serialized_chia_program`: [4](#0-3) 

3. Apply a heap limit to `deser_legacy` and `deser_backrefs` (currently both use `Allocator::new()` with no cap): [10](#0-9) 

---

### Proof of Concept

```python
from clvm_rs.clvm_rs import deser_backrefs
from clvm_rs.program import Program

# Craft a backrefs-compressed blob that expands to a large tree.
# Each 0xfe back-reference byte re-uses a previously parsed subtree,
# allowing O(1) wire bytes to produce O(2^n) in-memory nodes.
# A ~1 MB blob with deeply nested back-references can expand to
# hundreds of MB in the source allocator.

blob = craft_backrefs_bomb()          # attacker-controlled wire bytes
program = Program.from_bytes(blob)    # Stage 1: unlimited deser_backrefs allocator

# Stage 2 + 3: clvm_tree_to_lazy_node (unlimited) + intern_tree (unlimited)
# Three simultaneous unlimited allocations held in memory:
serialized = program.to_bytes_2026()
```

The concrete corrupted invariant is the missing heap cap on `intern_tree` inside `SerializerState::new` at `src/serde_2026/ser.rs:94`, which allows attacker-controlled CLVM bytes to drive unbounded RSS growth in the process hosting the Python wheel.
