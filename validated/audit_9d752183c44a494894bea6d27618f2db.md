### Title
`SwapAllowlistExtension` checks the router address instead of the original user, enabling allowlist bypass for any user routing through `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. The pool passes `msg.sender` of the `pool.swap()` call as `sender`. When a user routes through `MetricOmmSimpleRouter`, the direct caller of `pool.swap()` is the router contract, not the original user. The extension therefore checks the router's address, not the user's address. If the pool admin allowlists the router to support router-mediated swaps, every unpermissioned user can bypass the allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol line 230-240
_beforeSwap(
  msg.sender,   // direct caller of pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension:

```solidity
// ExtensionCalling.sol line 162-176
abi.encodeCall(
  IMetricOmmExtensions.beforeSwap,
  (sender, recipient, zeroForOne, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then checks `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol line 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` all call `pool.swap()` directly from the router contract:

```solidity
// MetricOmmSimpleRouter.sol line 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
  );
```

Because the router is `msg.sender` of `pool.swap()`, the extension receives `sender = address(router)`, not the original user's address. The allowlist lookup `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user]`.

This creates two mutually exclusive broken states:

1. **Router not allowlisted**: Allowlisted users cannot use the router at all. Every router-mediated swap reverts with `NotAllowedToSwap`, breaking the primary periphery entry point for curated pools.

2. **Router allowlisted** (to restore router functionality): Every user, regardless of allowlist status, can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The extension sees `sender = router`, which is allowlisted, and passes unconditionally.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool — only specific addresses (e.g., KYC-verified counterparties, whitelisted market participants) are intended to trade. If the pool admin allowlists the router to support standard periphery usage, any unpermissioned user can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) and execute swaps on the restricted pool. The allowlist guard is completely bypassed. This constitutes a direct policy failure with fund-impacting consequences: non-allowlisted users can drain liquidity, execute arbitrage, or trade on pools that were designed to exclude them.

---

### Likelihood Explanation

The scenario is reachable by any unprivileged user with no special setup. The only precondition is that the pool admin has allowlisted the router address — a natural and expected configuration for any curated pool that also wants to support the standard periphery. The router is a public, permissionless contract. No privileged access, malicious setup, or non-standard tokens are required.

---

### Recommendation

The `SwapAllowlistExtension` must check the economically relevant actor — the original user — not the direct caller of `pool.swap()`. Two complementary fixes:

1. **In the extension**: When `sender` is a known router or intermediary, the extension should require the original user's identity to be passed in `extensionData` and verify it. Alternatively, the extension can maintain a separate allowlist for trusted forwarders (routers) that are expected to forward the real user identity.

2. **In the router**: `MetricOmmSimpleRouter` should forward the original `msg.sender` as part of `extensionData` so that allowlist extensions can recover and check the real user. The extension would then decode the user address from `extensionData` and check `allowedSwapper[pool][decodedUser]` when `sender` is a recognized forwarder.

The simplest correct fix is to have the router encode `msg.sender` into `extensionData` for every hop, and update `SwapAllowlistExtension.beforeSwap` to decode and check that value when `sender` is a trusted forwarder address.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed to swap.
3. Pool admin calls `setAllowedToSwap(pool, address(router), true)` — router is allowlisted to support periphery usage.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. The pool calls `_beforeSwap(router, ...)`.
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true`.
8. Bob's swap executes successfully despite not being on the allowlist.

Bob has bypassed the curated pool's allowlist guard with no special privileges, using only the standard public router entry point. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
