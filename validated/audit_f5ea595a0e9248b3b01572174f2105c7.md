### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Path, Enabling Secp256k1/r1 Verify Without Hard-Fork Guard — (`src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op` implementation in `src/chia_dialect.rs` contains two separate dispatch paths for secp operations. The 1-byte opcode path (opcodes 64 and 65) correctly gates on `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches directly to `op_secp256k1_verify` / `op_secp256r1_verify` **with no flag check at all**. An attacker-controlled CLVM program using the 4-byte encoding can invoke secp signature verification on any node, regardless of whether `ENABLE_SECP_OPS` is set.

### Finding Description

In `src/chia_dialect.rs`, the `op_len == 4` branch: [1](#0-0) 

dispatches `0x13d61f00` → `op_secp256k1_verify` and `0x1c3a8f00` → `op_secp256r1_verify` unconditionally. There is no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard anywhere in this branch.

The 1-byte path, by contrast, correctly gates both secp ops: [2](#0-1) 

`ENABLE_SECP_OPS` is defined as the hard-fork flag that controls secp opcode availability: [3](#0-2) 

`op_secp256k1_verify` and `op_secp256r1_verify` accept `_flags: ClvmFlags` but ignore it entirely (note the underscore), so no internal flag check compensates for the missing guard: [4](#0-3) 

### Impact Explanation

`ENABLE_SECP_OPS` is a hard-fork flag. Before the fork activates, full nodes run `ChiaDialect` without this flag. The invariant is that secp operations must be rejected pre-fork. The 4-byte path breaks this invariant:

- **Pre-fork node** (`ClvmFlags::empty()`): 1-byte opcode 64 → `unknown_operator` (correct); 4-byte opcode `0x13d61f00` → `op_secp256k1_verify` executes (incorrect).
- **Post-fork node** (`ENABLE_SECP_OPS` set): both paths → `op_secp256k1_verify` executes.

A CLVM program using the 4-byte encoding of secp verify will be **accepted by pre-fork nodes and post-fork nodes alike**, while the same program using the 1-byte encoding will be **rejected by pre-fork nodes**. This is a concrete consensus split: two nodes running the same software version but with different flag configurations will disagree on the validity of the same spend.

### Likelihood Explanation

The attacker only needs to craft a CLVM program whose operator atom is the 4-byte literal `0x13d61f00` with a valid secp256k1 pubkey/message/signature triple. This is fully attacker-controlled input through the standard CLVM execution API. No special privileges or compromised infrastructure are required.

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte opcode branch, mirroring the 1-byte path:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};

### Proof of Concept

```rust
#[test]
fn secp_4byte_bypasses_enable_flag() {
    use crate::allocator::Allocator;
    use crate::chia_dialect::{ChiaDialect, ClvmFlags};
    use crate::dialect::{Dialect, OperatorSet};

    let mut allocator = Allocator::new();
    let dialect = ChiaDialect::new(ClvmFlags::empty()); // no ENABLE_SECP_OPS

    // 4-byte opcode for secp256k1_verify
    let op_4byte = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00]).unwrap();
    // 1-byte opcode 64
    let op_1byte = allocator.new_atom(&[64]).unwrap();

    let args = allocator.nil();

    // 1-byte path: correctly returns unknown_operator / error (not secp)
    let r1 = dialect.op(&mut allocator, op_1byte, args, u64::MAX, OperatorSet::Default);
    // r1 is NOT a secp256k1 invocation — it hits unknown_operator

    // 4-byte path: INCORRECTLY invokes op_secp256k1_verify despite no ENABLE_SECP_OPS
    let r4 = dialect.op(&mut allocator, op_4byte, args, u64::MAX, OperatorSet::Default);
    // r4 returns a secp256k1 argument-count error (not unknown_operator),
    // proving op_secp256k1_verify was reached without the flag.
    assert!(matches!(r4, Err(_))); // secp arg error, not unknown-op
}
```

The distinguishing observable: with `ClvmFlags::empty()`, the 1-byte path returns an `unknown_operator` result (or `Unimplemented` under `NO_UNKNOWN_OPS`), while the 4-byte path returns a secp-specific argument error — proving `op_secp256k1_verify` was entered without `ENABLE_SECP_OPS` being set.
