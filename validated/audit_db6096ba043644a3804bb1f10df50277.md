### Title
Silent Failure of Slow-Mode `DepositCollateral` Permanently Locks User Tokens in Endpoint — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` transfers user tokens into the `Endpoint` contract at submission time, then queues a `DepositCollateral` slow-mode transaction. If that slow-mode transaction fails during execution — due to any state change in the mandatory 3-day window — the `_executeSlowModeTransaction` catch block silently discards the failure. A comment in the code explicitly confirms the refund path was removed. The deposited tokens are permanently locked with no recovery mechanism.

---

### Finding Description

**Phase 1 — Tokens taken at submission:**

`depositCollateralWithReferral` calls `handleDepositTransfer` to pull tokens from `msg.sender` into the `Endpoint` contract, then enqueues a `DepositCollateral` slow-mode transaction with a hardcoded 3-day delay. [1](#0-0) 

**Phase 2 — Silent failure at execution:**

`_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch`. If the inner call reverts for any reason, the catch block checks only for out-of-gas and otherwise falls through silently. The comment at line 226 — `// try return funds now removed` — is an explicit admission that a refund path previously existed and was deliberately deleted. [2](#0-1) 

**Phase 2 execution path for `DepositCollateral`:**

`processSlowModeTransactionImpl` decodes the transaction and calls `clearinghouse.depositCollateral(txn)`. Any revert inside `clearinghouse.depositCollateral` propagates back to the catch block and is swallowed. [3](#0-2) 

**Concrete revert triggers within the 3-day window:**

- A product's token address is migrated (the codebase already contains `replaceUsdcEWithUsdc` in `ContractOwner.sol`, confirming token migrations occur in production). After migration, `clearinghouse.depositCollateral` for the old `productId` will revert because the token mapping no longer resolves.
- A perp product is delisted via `clearinghouse.delistProduct`, and the deposit references a product whose engine state is torn down.
- The oracle price for the product is updated to `0` or a negative value by the sequencer between submission and execution, causing an internal health or price check inside `depositCollateral` to revert. [4](#0-3) 

---

### Impact Explanation

User tokens transferred to `Endpoint` during `depositCollateralWithReferral` are permanently locked. There is no claimable mapping, no retry function, and no admin rescue path for the deposited collateral. The `Endpoint` contract holds the tokens but the user's on-chain balance in `SpotEngine` is never credited. The asset delta is: user loses `amount` tokens of the deposited product with zero recourse.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a mandatory window during which protocol state can change. Token migrations have already occurred on mainnet (the `replaceUsdcEWithUsdc` function is production code, not a test artifact). Any future token migration or product configuration change during this window silently destroys pending deposits. The `depositCollateralWithReferral` function is also called by `DirectDepositV1.creditDeposit`, meaning automated deposit flows are equally exposed. [5](#0-4) 

---

### Recommendation

Restore a refund mapping analogous to the removed "try return funds" path. When `processSlowModeTransaction` fails for a `DepositCollateral` transaction, record the `(subaccount, productId, amount)` in a `claimableDeposits` mapping and expose a `claimFailedDeposit()` function that returns the tokens to the original depositor. Alternatively, validate all conditions that `clearinghouse.depositCollateral` will check before taking custody of tokens in `depositCollateralWithReferral`, so the two phases cannot desynchronize.

---

### Proof of Concept

1. User calls `Endpoint.depositCollateralWithReferral(subaccount, productId, amount, ref)`.
2. `handleDepositTransfer` pulls `amount` tokens from `msg.sender` into `Endpoint`. [6](#0-5) 
3. A `DepositCollateral` slow-mode tx is queued with `executableAt = block.timestamp + 3 days`. [7](#0-6) 
4. Within the 3-day window, the protocol migrates the token for `productId` (as has happened on mainnet for USDC.e → USDC). The old token address is no longer registered in `SpotEngine`.
5. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
6. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)` inside a `try`. [8](#0-7) 
7. `processSlowModeTransactionImpl` calls `clearinghouse.depositCollateral(txn)`, which reverts because the product's token is no longer valid.
8. The `catch` block fires. `gasleft()` is well above the threshold, so execution continues past `// try return funds now removed`.
9. The slow-mode tx is consumed (`txUpTo` is incremented at line 194), `nSubmissions` is incremented, and the user's `amount` tokens remain permanently locked in `Endpoint` with no credit to their subaccount and no recovery path. [9](#0-8)
