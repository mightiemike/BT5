Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of that call. A pool admin who allowlists the router to enable router-mediated swaps for their permitted users inadvertently opens the allowlist to every address on the network, as any caller of the router will pass the check.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same pattern applies to `exactOutputSingle`, `exactInput`, and `exactOutput` — all router entry points call `pool.swap()` directly with the router as `msg.sender`. There is no mechanism by which the extension can recover the originating end user from the `sender` argument. The pool admin faces a binary choice: do not allowlist the router (breaking UX for approved users) or allowlist the router (collapsing per-user gating to per-contract gating, allowing any address to bypass the restriction).

## Impact Explanation

A pool admin deploys a restricted pool (e.g., for KYC'd counterparties or institutional market makers) and configures `SwapAllowlistExtension`. To let their approved users trade through the standard router, they call `setAllowedToSwap(pool, router, true)`. From that point, any address on the network can call `router.exactInputSingle(...)` and the `beforeSwap` hook passes because `sender == router` is allowlisted. Non-approved users can execute swaps in a pool whose access control is supposed to prevent it, draining arbitrage value from LP positions priced assuming only trusted counterparties would trade. This constitutes a direct loss of LP principal through unauthorized swap execution — a broken core pool functionality causing loss of funds, matching the allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point deployed by the protocol. Any pool admin who wants their allowlisted users to have a normal trading UX will allowlist the router — this is the natural and expected admin action. The bypass requires no special privilege: any EOA or contract can call the router. The trigger is a routine admin configuration step, not an exotic attack setup, making exploitation highly likely once a restricted pool is deployed.

## Recommendation

Pass the economically relevant actor — the end user — through the hook chain rather than the direct caller. The cleanest fix is for the router to store the originating `msg.sender` in transient storage (it already stores `payer` for the callback via `_setNextCallbackContext`) and expose it via a standard interface that the pool reads and forwards as `sender` to extensions. Alternatively, the pool could accept an explicit `swapper` parameter separate from `msg.sender` that the router populates with `msg.sender` before calling the pool, and `SwapAllowlistExtension` would gate on that value instead of `sender`.

## Proof of Concept

```solidity
// Pool admin sets up a restricted pool and allowlists the router
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// alice is NOT individually allowlisted
// alice calls the router — beforeSwap sees sender=router, which IS allowlisted → passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    recipient: alice,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// alice successfully swaps in a pool she was never allowlisted for
```

The `beforeSwap` check at `SwapAllowlistExtension.sol` line 37 evaluates `allowedSwapper[pool][router]` — true — and returns the success selector, bypassing the per-user gate entirely. [5](#0-4)
