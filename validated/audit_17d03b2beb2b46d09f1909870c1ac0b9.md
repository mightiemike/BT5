Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist - (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented to gate deposits by depositor address, but its enforcement hook checks the `owner` parameter (the position owner, a caller-supplied argument) against the allowlist instead of the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call). Because `owner` is freely chosen by the caller, any un-allowlisted address can bypass the guard by supplying any allowlisted address as `owner`. The sibling `SwapAllowlistExtension` correctly checks `sender`, confirming this is an implementation error.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` encodes both as distinct positional arguments to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first argument (`sender`) and checks the second (`owner`) against the allowlist: [3](#0-2) 

The contract's own NatSpec and admin API contradict this: the notice reads "Gates `addLiquidity` by depositor address, per pool", the setter is `setAllowedToDeposit(address pool_, address depositor, bool allowed)`, and the storage mapping is `allowedDepositor[pool][depositor]`: [4](#0-3) [5](#0-4) 

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` in the analogous position: [6](#0-5) 

Since `owner` is a free parameter in `addLiquidity` with no constraint tying it to `msg.sender` (unlike `removeLiquidity`, which enforces `msg.sender == owner`), any caller can set `owner` to any allowlisted address: [7](#0-6) 

## Impact Explanation

The pool admin's deposit access-control boundary is fully bypassed. Any address, regardless of allowlist status, can deposit into a pool the admin intended to restrict. The exact wrong value is the extension's boolean decision: `allowedDepositor[msg.sender][owner]` returns `true` for an allowlisted `owner` even when the actual depositing `sender` is not allowlisted, causing the hook to return `IMetricOmmExtensions.beforeAddLiquidity.selector` instead of reverting. This breaks the admin-boundary invariant for pools using this extension for KYC/compliance or LP-tier gating.

## Likelihood Explanation

No special privilege is required. The attacker only needs to know one allowlisted address, which is publicly observable via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attacker calls `pool.addLiquidity(owner=<allowlisted_address>, ...)` directly; no flash loan, oracle manipulation, or privileged role is needed. The bypass is repeatable and unconditional.

## Recommendation

Replace the `owner` check with `sender` in `beforeAddLiquidity`:

```diff
- function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
+ function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
  {
-   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
+   if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
      revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
```

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` in `BEFORE_ADD_LIQUIDITY_ORDER`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — only Alice is allowlisted.
3. Bob (not allowlisted) calls `pool.addLiquidity(owner=alice, salt=..., deltas=..., callbackData=..., extensionData=...)` directly, funding the callback from his own tokens.
4. The pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. The extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
6. Bob's deposit succeeds; Alice holds the resulting position shares while Bob's tokens enter the pool. The allowlist is fully bypassed.
