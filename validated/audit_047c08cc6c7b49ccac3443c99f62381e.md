### Title
Silent Token Transfer Success on Self-Destructed Token Contract Corrupts Deposit/Withdrawal Accounting — (`File: core/contracts/libraries/ERC20Helper.sol`)

---

### Summary

`ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` perform low-level `.call()` on the token address without verifying that the target address contains contract code. Per EVM specification, a low-level `call` to a non-existent (e.g., self-destructed) address returns `(true, "")`. The existing success check `success && (data.length == 0 || abi.decode(data, (bool)))` passes silently when `data.length == 0`, meaning no revert occurs and no actual token movement happens. This is the direct Nado analog of the MultiSig `external_call` bug.

---

### Finding Description

`ERC20Helper` is a custom low-level transfer library used throughout the Nado protocol for all token movements: [1](#0-0) 

```solidity
function safeTransfer(IERC20Base self, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
``` [2](#0-1) 

When `address(self)` has no code (self-destructed contract), the EVM returns `success = true` and `data = ""`. The guard condition evaluates as `true && (0 == 0)` → passes. No revert. No tokens move.

The same flaw exists in `safeTransferFrom`: [3](#0-2) 

`Clearinghouse.sol` imports and uses this library for all collateral movements: [4](#0-3) 

`EndpointStorage` uses `safeTransferFrom` for deposit intake and `safeTransfer` for outbound transfers: [5](#0-4) 

`BaseWithdrawPool` uses both for liquidity operations: [6](#0-5) 

The same pattern is independently present in `DirectDepositV1.safeTransfer`: [7](#0-6) 

---

### Impact Explanation

**Withdrawal path (direct user asset loss):**
A user with a balance in SpotEngine for product X (token T) submits `WithdrawCollateral`. Clearinghouse calls `ERC20Helper.safeTransfer(token, user, amount)`. If token T has been self-destructed, the call returns `(true, "")`, the check passes, the SpotEngine balance is decremented, but the user receives zero tokens. The user's on-chain collateral is permanently destroyed with no recourse.

**Deposit path (phantom collateral inflation):**
A user calls `depositCollateral` for a product whose token has been self-destructed. `safeTransferFrom` returns `(true, "")`, the check passes, and SpotEngine credits the user's balance. The user now holds uncollateralized credit they can use to open positions or borrow against, corrupting the protocol's solvency invariant.

The corrupted state delta is: `SpotEngine.balances[productId][subaccount]` is mutated (incremented or decremented) without any corresponding real ERC-20 token movement, breaking the invariant that internal balances are fully backed by real token custody. [8](#0-7) 

---

### Likelihood Explanation

Moderate. The trigger requires a registered token contract to be self-destructed. This is realistic for:
- Upgradeable proxy tokens where the implementation is self-destructed (leaving the proxy pointing to dead code)
- Tokens with admin-controlled `selfdestruct` (common in early DeFi tokens)
- Tokens that are deprecated/delisted but whose contracts are destroyed by their issuers

Once triggered, the vulnerability is exploitable by any unprivileged user interacting through the standard `depositCollateral` or `withdrawCollateral` entrypoints. No special privileges are required on the Nado side.

---

### Recommendation

**Short term:** Add a contract existence check in both `ERC20Helper.safeTransfer` and `ERC20Helper.safeTransferFrom` before the low-level call:

```solidity
require(address(self).code.length > 0, "ERC20Helper: token is not a contract");
```

Apply the same fix to `DirectDepositV1.safeTransfer`.

**Long term:** Consider replacing the custom `ERC20Helper` library with OpenZeppelin's `SafeERC20`, which includes a contract existence check via `Address.functionCall` internally. Monitor all registered token contracts for self-destruct events and implement a circuit breaker that halts deposits/withdrawals for a product if its token contract loses code.

---

### Proof of Concept

1. Token `T` is registered in `SpotEngine` via `addOrUpdateProduct` with `config.token = address(T)`.
2. Token `T`'s contract is self-destructed (e.g., by its issuer).
3. User calls `Endpoint.depositCollateral(subaccountName, productId, amount)`.
4. Execution reaches `EndpointStorage.handleDepositTransfer` → `safeTransferFrom(token, from, amount)`.
5. `address(token).call(abi.encodeWithSelector(transferFrom.selector, ...))` returns `(true, "")`.
6. `require(true && (0 == 0 || ...))` → passes.
7. `Clearinghouse.depositCollateral` credits `amount` to the user's subaccount in SpotEngine.
8. User now holds phantom collateral with no real token backing, usable for trading or borrowing. [9](#0-8) [10](#0-9)
