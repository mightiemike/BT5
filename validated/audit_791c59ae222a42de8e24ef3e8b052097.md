### Title
`block_casm_hash_v1_declares` Versioned Constant Enforced Only in Execution, Not in Gateway Admission — (`crates/blockifier/src/transaction/transactions.rs`)

### Summary

The `block_casm_hash_v1_declares` versioned constant, enabled starting in protocol version 0.14.1, requires that Declare V3 transactions supply a V2 (Blake) compiled class hash. However, this check is placed exclusively inside `run_execute` — the execution phase — and is never evaluated during gateway stateful validation. The gateway's `run_validate_entry_point` only runs the `__validate__` account entry point and `perform_pre_validation_stage`, neither of which checks `block_casm_hash_v1_declares`. As a result, the gateway admits Declare V3 transactions carrying a V1 (Poseidon) compiled class hash that will deterministically fail during batcher execution.

### Finding Description

`block_casm_hash_v1_declares` is a boolean field in `VersionedConstants` that flipped from `false` to `true` between blockifier versioned-constants files `0_14_0` and `0_14_1`:

```
~ /block_casm_hash_v1_declares: true
```

When `true`, every Declare V3 transaction must carry a V2 (Blake) compiled class hash. The enforcement is in `run_execute`:

```rust
// crates/blockifier/src/transaction/transactions.rs  lines 184-188
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
```

`check_compile_class_hash_v2_declaration` recomputes the Blake hash of the attached CASM and returns `StarknetApiError::DeclareTransactionCasmHashMissMatch` if the transaction's `compiled_class_hash` field does not match.

The gateway's stateful validation path is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 317-341
let mut versioned_constants = VersionedConstants::get_versioned_constants(
    self.config.versioned_constants_overrides.clone(),
);
versioned_constants.disable_casm_hash_migration();   // only this flag is adjusted
// ...
let mut blockifier_validator = StatefulValidator::create(state, block_context);
blockifier_validator.validate(account_tx)            // runs __validate__ only
```

`StatefulValidator::validate` calls `perform_pre_validation_stage` (nonce, fee, proof-facts checks) and then the account's `__validate__` entry point. Neither path invokes `run_execute`, so `block_casm_hash_v1_declares` is never consulted. The same omission exists in the `native_blockifier` Python wrapper `PyValidator::create` / `perform_validations`.

The identical asymmetry exists in the `native_blockifier` path:

```rust
// crates/native_blockifier/src/py_validator.rs  lines 46-50
let mut versioned_constants = VersionedConstants::get_versioned_constants(Some(
    py_versioned_constants_overrides.into(),
));
versioned_constants.disable_casm_hash_migration();
```

`disable_casm_hash_migration` is the only flag adjusted; `block_casm_hash_v1_declares` is left at its versioned value (`true` for ≥ 0.14.1) but is never evaluated during validation.

### Impact Explanation

A Declare V3 transaction whose `compiled_class_hash` field contains a V1 (Poseidon) hash passes all gateway checks and enters the mempool. When the batcher later calls `run_execute`, the transaction fails with `DeclareTransactionCasmHashMissMatch`. The gateway has admitted a transaction that is guaranteed to fail during sequencing — a direct instance of the "Mempool/gateway/RPC admission accepts invalid transactions before sequencing" impact class.

### Likelihood Explanation
