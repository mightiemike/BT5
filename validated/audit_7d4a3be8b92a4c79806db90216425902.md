Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Depositors to Bypass Curated-Pool Guard — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently ignores the `sender` parameter (the actual `msg.sender` of the `addLiquidity` call) and gates access solely on `owner` (a freely caller-supplied LP position address). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` and `MetricOmmPoolLiquidityAdder` explicitly supports depositing on behalf of a third-party owner, any non-allowlisted user can bypass the guard by nominating any allowlisted address as `owner`. This structurally defeats the deposit allowlist's stated purpose of gating deposits by depositor address.

## Finding Description

**Wrong actor checked in the hook.**

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and never reads it, instead gating on `owner`: [1](#0-0) 

**Pool passes `msg.sender` as `sender` and an arbitrary address as `owner`.**

`MetricOmmPool.addLiquidity` accepts `owner` as a free caller-supplied parameter and forwards `msg.sender` as `sender` to the extension: [2](#0-1) 

The extension dispatch in `ExtensionCalling._beforeAddLiquidity` correctly forwards both actors, confirming `sender` is available but unused by the hook: [3](#0-2) 

**Periphery router explicitly supports depositing on behalf of a third-party owner.**

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an explicit `owner` differing from `msg.sender` (the payer), and stores `msg.sender` as the token payer in transient context — not as `owner`: [4](#0-3) 

The test suite confirms this is a supported, tested flow where `alice` pays but `bob` receives the LP position: [5](#0-4) 

**Attack path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists address `A`: `extension.setAllowedToDeposit(pool, A, true)`.
2. Non-allowlisted user `B` calls `adder.addLiquidityExactShares(pool, A, salt, deltas, max0, max1, "")`.
3. The adder calls `pool.addLiquidity(A, salt, deltas, ...)` with `msg.sender = adder`.
4. The pool calls `extension.beforeAddLiquidity(adder_address, A, ...)`.
5. The extension evaluates `allowedDepositor[pool][A]` → `true` → passes.
6. `B`'s tokens are pulled (B is the payer in transient context); `A` receives the LP position.
7. `B`'s tokens have entered the curated pool without `B` ever being allowlisted.

The same bypass works when `B` calls `pool.addLiquidity(A, ...)` directly, since the pool imposes no restriction on who may nominate an `owner`.

**Existing guards are insufficient.** `_validateOwner` only rejects `address(0)`: [6](#0-5) 

There is no check that `owner == msg.sender`, and no restriction in the pool on who may supply an arbitrary `owner`.

## Impact Explanation

The deposit allowlist is structurally bypassed. Any non-allowlisted user who knows one allowlisted address (discoverable on-chain via `AllowedToDepositSet` events) can deposit tokens into the curated pool. The pool admin's curation policy — whether for regulatory (KYC/AML), risk, or access-control reasons — is not enforced. Additionally, non-allowlisted users can grief allowlisted LPs by depositing tokens into their positions without consent, causing unexpected LP exposure and position-size changes the LP did not authorize. This constitutes broken core pool functionality (the allowlist guard) causing unauthorized fund entry into a curated pool.

## Likelihood Explanation

`MetricOmmPoolLiquidityAdder` explicitly supports and tests the "deposit on behalf of another owner" flow. Allowlisted addresses are discoverable on-chain. No special privilege is required — any EOA or contract can call the adder with an arbitrary `owner`. The bypass is trivially repeatable with zero cost beyond gas.

## Recommendation

Check `sender` (the actual depositor) instead of `owner` (the LP position owner), mirroring the correct pattern already used in `SwapAllowlistExtension.beforeSwap`: [7](#0-6) 

Fix for `DepositAllowlistExtension`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

The existing test `test_exactShares_canAddOnBehalfOfAnotherOwner` already demonstrates the mechanics: [8](#0-7) 

Adding `DepositAllowlistExtension` to the pool in that test setup with only `bob` allowlisted, then calling as `alice` with `owner = bob`, will show the extension passes (checking `bob`) while `alice` (the actual depositor/payer) was never allowlisted — confirming the bypass.
