Audit Report

## Title
Unguarded `payable` on Non-WETH Swap and Liquidity Paths Allows Permanent ETH Loss to Any Caller — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

Every user-facing swap function in `MetricOmmSimpleRouter` and every liquidity entry point in `MetricOmmPoolLiquidityAdder` is marked `payable` but contains no guard rejecting `msg.value > 0` when the input token is not WETH. ETH sent alongside a non-WETH call is silently retained by the contract while the ERC-20 swap or liquidity add completes normally. Any subsequent caller of the public `refundETH()` function receives the full stranded balance, causing an irreversible loss of the original sender's ETH.

## Finding Description

All four swap entry points in `MetricOmmSimpleRouter` are `payable` with no `msg.value` validation: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

All liquidity entry points in `MetricOmmPoolLiquidityAdder` are likewise `payable` with no guard: [5](#0-4) [6](#0-5) [7](#0-6) 

The internal `pay()` helper in `PeripheryPayments` only consumes `address(this).balance` when `token == WETH`. For every other token it falls through to the plain `safeTransferFrom` branch, leaving any ETH sent via `msg.value` untouched in the contract: [8](#0-7) 

`refundETH()` sends the **entire** contract ETH balance to `msg.sender`, not to the original depositor: [9](#0-8) 

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses; it does not prevent ETH from entering via `msg.value` on a `payable` call: [10](#0-9) 

The `multicall` wrapper on both contracts is also `payable`, compounding the attack surface: [11](#0-10) [12](#0-11) 

## Impact Explanation

A user who calls any non-WETH swap or liquidity function with `msg.value > 0` loses that ETH permanently. The ERC-20 operation completes successfully via `safeTransferFrom`, so no revert signals the mistake. The stranded ETH is immediately claimable by any address calling `refundETH()`, constituting a **direct, irreversible loss of user principal** with no on-chain recourse. This meets the Critical/High direct loss of user principal threshold under the Metric OMM allowed impact gate.

## Likelihood Explanation

The `payable` modifier on non-ETH paths is an open invitation for this mistake. Users familiar with Uniswap v3-style routers routinely send ETH expecting auto-wrapping. Scripts and wallets may include a non-zero `value` field by default. The loss requires only a single mistaken transaction and is immediately exploitable by any mempool observer or MEV bot calling `refundETH()` in the same or next block. No privileged access is required; any unprivileged caller can drain the stranded ETH.

## Recommendation

Add an explicit guard at the top of each swap function rejecting non-zero `msg.value` when the input token is not WETH:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput:
if (msg.value != 0 && params.tokenIn != WETH) revert UnexpectedETH();
```

For liquidity functions where pool tokens are never WETH (or cannot be verified at entry), reject any non-zero `msg.value` unconditionally:

```solidity
// In addLiquidityExactShares, addLiquidityWeighted:
if (msg.value != 0) revert UnexpectedETH();
```

Alternatively, remove the `payable` modifier entirely from all non-ETH-input functions and keep it only on paths that explicitly handle native ETH (e.g., `refundETH`, `unwrapWETH9`).

## Proof of Concept

1. User calls `exactInputSingle` with `params.tokenIn = address(USDC)` and `msg.value = 1 ether`.
2. The function is `payable`; the 1 ETH is accepted by the contract with no revert.
3. `_setNextCallbackContext` and `pool.swap` execute normally; the swap callback calls `pay(USDC, user, pool, amount)` which hits the `else` branch (`safeTransferFrom`) — the 1 ETH is untouched.
4. The swap completes successfully; the user receives USDC output and observes no error.
5. An attacker (or MEV bot) calls `refundETH()` in the next transaction and receives the full 1 ETH.
6. The victim has no on-chain mechanism to recover their ETH.

Foundry test sketch:
```solidity
function test_ethLostOnNonWethSwap() public {
    vm.deal(user, 1 ether);
    vm.prank(user);
    router.exactInputSingle{value: 1 ether}(usdcParams);
    // ETH is now stranded in router
    assertEq(address(router).balance, 1 ether);

    vm.prank(attacker);
    router.refundETH();
    assertEq(attacker.balance, 1 ether);
    assertEq(user.balance, 0);
}
```

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
