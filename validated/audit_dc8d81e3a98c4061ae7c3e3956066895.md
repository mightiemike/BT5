### Title
Hardcoded `Fee(1)` placeholder in `convert_consensus_l1_handler_to_internal_l1_handler` bypasses L1 fee adequacy check and commits wrong `paid_fee_on_l1` to block data - (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

When the consensus layer converts a `ConsensusTransaction::L1Handler` to an executable `L1HandlerTransaction`, the `paid_fee_on_l1` field — which represents the actual ETH fee paid on L1 for the L1→L2 message — is hardcoded to `Fee(1)` instead of the real value. This is the direct sequencer analog of the zero-slippage bug: a minimum-bound protection parameter is set to a trivial placeholder value, bypassing the intended check and committing wrong data to the block.

---

### Finding Description

In `convert_consensus_l1_handler_to_internal_l1_handler`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The `ConsensusTransaction::L1Handler` type carries only the raw `transaction::L1HandlerTransaction` (contract address, selector, calldata, nonce) — it does not carry `paid_fee_on_l1`. When the converter builds the executable transaction, it supplies `Fee(1)` as a permanent placeholder.

The blockifier's fee adequacy check for L1 handler transactions is:

```rust
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [2](#0-1) 

Because `Fee(1) != Fee(0)`, this check always passes regardless of what fee was actually paid on L1. The actual fee paid on L1 is never verified in the consensus execution path.

Additionally, `paid_fee_on_l1` is serialized into the `CentralL1HandlerTransaction` struct that is sent to the Cende/prover:

```rust
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,  // always Fee(1) from consensus path
            ...
        }
    }
}
``` [3](#0-2) 

This means the central blob committed to the prover/Cende carries `paid_fee_on_l1 = 1` for every L1 handler transaction processed through the consensus path, regardless of the actual ETH paid on L1.

The `ConsensusTransaction::L1Handler` type itself confirms it carries no fee field:

```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
``` [4](#0-3) 

---

### Impact Explanation

**Wrong state/receipt committed to block data (Critical scope):** The `paid_fee_on_l1` field in the central objects sent to the prover is always `Fee(1)` for L1 handler transactions processed through consensus, regardless of the actual ETH paid on L1. This is a canonicalization invariant violation: the executable transaction object bound to the block carries a divergent value (`Fee(1)`) instead of the actual fee from the L1 event.

**Fee adequacy check bypassed (High scope):** The blockifier's `InsufficientFee` guard — which is supposed to reject L1 handler transactions where no fee was paid — is permanently bypassed for all L1 handler transactions in the consensus path, because `Fee(1)` always satisfies `paid_fee != Fee(0)`. [5](#0-4) 

---

### Likelihood Explanation

Every L1 handler transaction that passes through the consensus path (i.e., every L1→L2 message included in a block proposal and validated by non-proposer validators) triggers this code path. This is normal sequencer operation, not an edge case. The TODO comment confirms this is a known placeholder, not a deliberate design choice.

---

### Recommendation

The `paid_fee_on_l1` value must be carried through the consensus message. Options:

1. Add `paid_fee_on_l1: Fee` to the `ConsensusTransaction::L1Handler` variant so the proposer can include the actual fee from the L1 scraper event, and validators can verify it against their own L1 provider state.
2. During conversion on the validator side, look up the actual `paid_fee_on_l1` from the local L1 provider/transaction manager using the transaction's nonce and contract address before constructing the executable transaction.

---

### Proof of Concept

**Trigger path:**

1. L1 scraper detects a `LogMessageToL2` event on Ethereum with `paid_fee_on_l1 = X` (e.g., `X = 1_000_000_000_000_000` wei).
2. Proposer includes the L1 handler transaction in a block proposal as `InternalConsensusTransaction::L1Handler(tx)` — the `paid_fee_on_l1` field is present in the proposer's executable transaction but is **not** serialized into the consensus message.
3. Validator receives `ConsensusTransaction::L1Handler(raw_tx)` and calls `convert_consensus_l1_handler_to_internal_l1_handler`.
4. The converter calls `executable_transaction::L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))`. [6](#0-5) 

5. The resulting executable transaction has `paid_fee_on_l1 = Fee(1)` instead of `Fee(1_000_000_000_000_000)`.
6. The blockifier executes the transaction; the check `if paid_fee == Fee(0)` passes because `Fee(1) != Fee(0)`. [7](#0-6) 

7. The `CentralL1HandlerTransaction` serialized for Cende contains `paid_fee_on_l1: Fee(1)` — a value that does not match the actual ETH paid on L1 and that diverges from what the proposer committed. [8](#0-7)
