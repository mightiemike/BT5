Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Swapper, Enabling Allowlist Bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool binds to `msg.sender` of `pool.swap()`. When EOA users route through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, not the user. Because `pool.swap()` unconditionally invokes `metricOmmSwapCallback` on its caller, EOAs cannot call the pool directly and must use the router. A pool admin who allowlists the router address (the only viable path for EOA users) inadvertently grants every user on the network the ability to swap, defeating the per-address gate entirely.

## Finding Description

**Root cause — actor binding in the extension**

`SwapAllowlistExtension.beforeSwap` receives `sender` and checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the value the pool passes as the first argument to `_beforeSwap`.

**What the pool passes as `sender`**

`MetricOmmPool.swap` passes its own `msg.sender` (the immediate caller of `pool.swap()`) as `sender`:

```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

**Why EOAs cannot call the pool directly**

`MetricOmmPool.swap` unconditionally invokes `metricOmmSwapCallback` on `msg.sender` after executing the swap:

```solidity
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
``` [3](#0-2) 

An EOA has no code, so this call reverts. Every EOA user must route through `MetricOmmSimpleRouter` or another callback-capable contract.

**The router passes itself as the effective sender**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly; the pool sees `msg.sender = router`:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**The impossible configuration**

The pool admin faces a forced binary choice:

| Admin configuration | Allowlisted EOA users | Non-allowlisted EOA users |
|---|---|---|
| Allowlist individual EOA addresses only | **Blocked** (router not allowlisted, EOAs can't call pool directly) | Blocked |
| Allowlist the router | Allowed | **Also allowed — bypass** |

There is no configuration that simultaneously permits allowlisted EOA users to swap through the router and blocks non-allowlisted EOA users. The extension checks `allowedSwapper[pool][router]`, a single boolean that is either true for everyone or false for everyone who uses the router.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (the only practical way to let EOA users trade) grants every user on the network the ability to swap on that pool. Non-allowlisted users can execute swaps that the allowlist was specifically configured to prevent, directly impacting LP positions. This constitutes a broken core pool functionality (allowlist gate) causing direct loss of LP principal — a High severity impact under the allowed impact gate.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical user-facing entry point. Any pool admin who wants allowlisted EOA users to trade must allowlist the router address. The mistake is structurally forced by the extension's actor-binding design, not by an unusual admin error. The trigger is an unprivileged EOA calling the public router after the admin has made this natural and necessary configuration choice. No special privileges or unusual conditions are required.

## Recommendation

The `SwapAllowlistExtension` must gate the **economic initiator** of the swap, not the immediate `msg.sender` of `pool.swap()`. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated change to the router and extension, and the extension must verify the encoding comes from a trusted router.

2. **Dedicated router-aware extension**: Deploy a variant of `SwapAllowlistExtension` that, when `sender` is a known trusted router, reads the original initiator from `extensionData` and checks that address against the allowlist instead.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — the only way to let EOA users trade.
3. A non-allowlisted EOA (`attacker`) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
4. The router calls `pool.swap(...)` with `msg.sender = router`.
5. `_beforeSwap` passes `sender = router` to the extension.
6. The extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
7. The attacker successfully swaps on a pool they were never meant to access, bypassing the per-address allowlist.

A minimal Foundry test: deploy pool + extension, allowlist only the router, assert that a non-allowlisted EOA can successfully swap via the router while a direct pool call from the same EOA reverts.
