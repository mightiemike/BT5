### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` Permanently Blocks Finalization of the Last Participant in a Perp Product — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

When `_finalizeSubaccount` in `ClearinghouseLiq.sol` attempts to socialize a negative `vQuoteBalance` for a subaccount that is the last (or only) participant in a given perp product, `PerpEngine.socializeSubaccount` divides by `state.openInterest`, which is zero at that point. This causes an unrecoverable revert, permanently blocking finalization of the insolvent subaccount and leaving its debt irresolvable.

---

### Finding Description

`_finalizeSubaccount` in `ClearinghouseLiq.sol` enforces that all perp position amounts are zero before proceeding: [1](#0-0) 

Only after this check does it call `perpEngine.socializeSubaccount` to absorb any remaining negative `vQuoteBalance`: [2](#0-1) 

Inside `PerpEngine.socializeSubaccount`, when `balance.vQuoteBalance < 0` and insurance is insufficient, the loss is distributed via: [3](#0-2) 

The denominator `state.openInterest` represents the total open interest of the perp product. Because `_finalizeSubaccount` requires all position amounts to be zero before this call, if the liquidatee was the last (or only) participant in that perp product, their position closure already reduced `openInterest` to zero. Dividing by zero in Solidity triggers a panic revert, making the entire `_finalizeSubaccount` call fail unconditionally.

The negative `vQuoteBalance` can persist even after position closure because the negative-PnL settlement loop in `_finalizeSubaccount` only settles up to the available quote balance: [4](#0-3) 

If the liquidatee's quote balance is zero or negative, no negative PnL is settled, and `vQuoteBalance` remains negative when `socializeSubaccount` is reached.

---

### Impact Explanation

The insolvent subaccount can never be finalized. Its negative `vQuoteBalance` is permanently stuck in the perp product's state. The insurance fund cannot be used to cover it, and the loss cannot be socialized. The protocol is left with an irresolvable accounting deficit in that perp product. Any future participants in the same product inherit a corrupted `cumulativeFundingLongX18`/`cumulativeFundingShortX18` baseline if the product is ever re-used, or the product is effectively bricked.

**Corrupted state delta:** `state.openInterest == 0` while `balance.vQuoteBalance < 0` for the liquidatee — the division at `PerpEngine.sol:166` reverts, leaving `insurance` and `lastLiquidationFees` in `ClearinghouseStorage` unmodified and the subaccount permanently non-finalizable.

---

### Likelihood Explanation

This requires a perp product to reach a state where only one subaccount holds an open position. This is realistic for:
- Newly listed or low-liquidity perp products
- Products where all other participants have exited
- Adversarial scenarios where a user deliberately becomes the sole participant in a low-liquidity product before going insolvent

The sequencer submits `LiquidateSubaccount` transactions, but the revert is deterministic and cannot be bypassed by any sequencer action once the state is reached.

---

### Recommendation

In `PerpEngine.socializeSubaccount`, guard against zero `openInterest` before dividing:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize to; absorb via insurance or write off
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

Additionally, consider whether the insurance fund should absorb the residual loss when `openInterest == 0`, rather than silently writing it off.

---

### Proof of Concept

1. A perp product (e.g., `perpId = 2`) has exactly one participant: subaccount `A` with `amount = 100`, `vQuoteBalance = -50`.
2. The price drops; `A` becomes under-maintenance.
3. A liquidator calls `liquidateSubaccountImpl` with `productId = 2`, `amount = -100`, closing `A`'s position to zero. `openInterest` for product 2 is now `0`.
4. `A`'s `vQuoteBalance` remains `-50` (quote balance is also 0, so the negative-PnL settlement loop at lines 352–365 settles nothing).
5. A finalize call is submitted: `liquidateSubaccountImpl` with `productId = type(uint32).max`.
6. `_finalizeSubaccount` passes the `balance.amount == 0` check for all perps.
7. `perpEngine.socializeSubaccount(A, insurance)` is called.
8. For product 2: `balance.vQuoteBalance = -50 < 0`, `insuranceCover = min(insurance, 50)`. Assume `insurance = 0`.
9. Execution reaches line 166: `fundingPerShare = 50 / state.openInterest = 50 / 0` → **panic revert**.
10. Subaccount `A` can never be finalized. The `-50` vQuoteBalance is permanently stuck. [5](#0-4) [6](#0-5) [7](#0-6)

        external
        returns (int128)
    {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];
            (State memory state, Balance memory balance) = getStateAndBalance(
                productId,
                subaccount
            );
            if (balance.vQuoteBalance < 0) {
                int128 insuranceCover = MathHelper.min(
                    insurance,
                    -balance.vQuoteBalance
                );
                insurance -= insuranceCover;
                balance.vQuoteBalance += insuranceCover;
                state.availableSettle += insuranceCover;

                // actually socialize if still not enough
                if (balance.vQuoteBalance < 0) {
                    // socialize across all other participants
                    int128 fundingPerShare = -balance.vQuoteBalance.div(
                        state.openInterest
                    );
                    state.cumulativeFundingLongX18 += fundingPerShare;
                    state.cumulativeFundingShortX18 -= fundingPerShare;
                    balance.vQuoteBalance = 0;
                }
                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
        }
        return insurance;
    }
