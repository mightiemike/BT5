### Title
Litecoin PoW Verification Falls Back to SHA256d When `scrypt_hash` Feature Is Absent, Allowing Fake Block Acceptance — (File: `btc-types/src/btc_header.rs`)

---

### Summary

The `block_hash_pow()` function in `btc_header.rs` is the sole source of the proof-of-work hash used to gate block acceptance in `submit_block_header`. It is guarded by a **separate** compile-time feature flag (`scrypt_hash`) that is independent of the `litecoin` chain feature. When `scrypt_hash` is absent from the Litecoin build, the function silently falls back to SHA256d — the Bitcoin hash — instead of scrypt. An unprivileged proof-submitter can then craft a Litecoin header whose SHA256d output satisfies the target while its scrypt output does not, causing the contract to accept a fabricated block and corrupt the canonical chain state.

---

### Finding Description

`block_hash_pow()` in `btc-types/src/btc_header.rs` is compiled conditionally:

```rust
// btc-types/src/btc_header.rs  lines 38-53
pub fn block_hash_pow(&self) -> H256 {
    let block_header = self.get_block_header_vec();
    #[cfg(feature = "scrypt_hash")]
    {
        let params = scrypt::Params::new(10, 1, 1, 32).unwrap();
        let mut output = [0u8; 32];
        scrypt::scrypt(&block_header, &block_header, &params, &mut output).unwrap();
        H256::from(output)
    }
    #[cfg(not(feature = "scrypt_hash"))]
    {
        double_sha256(&block_header)   // ← wrong for Litecoin
    }
}
```

The result of this function is the **only** value checked against the PoW target in `submit_block_header` (non-dogecoin path):

```rust
// contract/src/lib.rs  lines 520-525
let pow_hash = header.block_hash_pow();
require!(
    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
    format!("block should have correct pow")
);
```

The `check_pow` function for Litecoin (`contract/src/litecoin.rs` lines 19-47) validates difficulty bits, MTP timestamp, and block version — but **never** validates the raw PoW hash itself. That responsibility falls entirely on the `block_hash_pow()` call above.

`scrypt_hash` is a distinct feature flag from `litecoin`. The system's multi-chain build model (one WASM per chain, driven by `--features <chain>`) means a Litecoin deployment compiled with only `--features litecoin` will silently use SHA256d for PoW verification. There is no compile-time or runtime guard that enforces `scrypt_hash` must accompany `litecoin`.

---

### Impact Explanation

If the Litecoin contract is compiled without `scrypt_hash`:

1. The PoW gate checks SHA256d(header) ≤ target instead of scrypt(header) ≤ target.
2. SHA256d is orders of magnitude cheaper to compute than scrypt (N=1024, r=1, p=1).
3. An attacker constructs a header with valid `bits`, valid `time` (MTP), valid `version ≥ 4`, and a nonce satisfying SHA256d(header) ≤ target. No valid scrypt solution is needed.
4. The contract stores this header as a legitimate mainchain block, updating `mainchain_tip_blockhash` and `chain_work`.
5. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` receive `true` for transactions that never existed on the real Litecoin chain.

The corrupted state is: `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `headers_pool` all reflect fabricated blocks. SPV proofs built against these blocks are forgeries.

---

### Likelihood Explanation

The `scrypt_hash` feature is architecturally decoupled from `litecoin`. The wiki and build system describe chain builds as `--features litecoin` with no mention of `scrypt_hash` as a mandatory companion. Any deployment that omits `scrypt_hash` — whether by oversight, CI misconfiguration, or a future re-deployment — activates this path. The attack requires only the ability to call `submit_blocks` with a crafted header, which is the normal relayer entry point open to any registered trusted relayer (or any caller if the staking guard is not enforced).

---

### Recommendation

Tie the scrypt hash selection to the `litecoin` feature flag directly, eliminating the separate `scrypt_hash` flag:

```rust
#[cfg(any(feature = "litecoin", feature = "scrypt_hash"))]
{ /* scrypt path */ }
#[cfg(not(any(feature = "litecoin", feature = "scrypt_hash")))]
{ /* sha256d path */ }
```

Or enforce at compile time that `litecoin` implies `scrypt_hash` via a `Cargo.toml` dependency:

```toml
[features]
litecoin = ["scrypt_hash"]
```

This mirrors the fix in the referenced report: replace the chain-incompatible default with the correct chain-specific primitive, rather than relying on a separate opt-in flag.

### Proof of Concept

1. Deploy the Litecoin contract compiled with `--features litecoin` (without `scrypt_hash`).
2. Obtain the current `expected_bits` from `get_last_block_header`.
3. Construct a `Header` with:
   - `version = 4`
   - `prev_block_hash` = current tip hash
   - `time` = MTP + 1 (passes timestamp check)
   - `bits` = expected_bits
   - Iterate `nonce` until `SHA256d(header_bytes) ≤ target_from_bits(bits)`
4. Call `submit_blocks([crafted_header])`.
5. The contract accepts the block: `mainchain_tip_blockhash` is updated to the fabricated hash.
6. Call `verify_transaction_inclusion` with a fabricated `tx_id` and a trivial single-element merkle proof whose root matches the fabricated `merkle_root` field — returns `true`. [1](#0-0) [2](#0-1) [3](#0-2)
