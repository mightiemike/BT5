### Title
u64 Overflow in `op_unknown` Cost Multiplier Produces Zero-Cost Execution — (`File: src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` multiplies the computed inner cost by `(cost_multiplier + 1)` using a plain `*=` on a `u64`. In Rust release builds, integer overflow wraps silently. An attacker can craft an unknown opcode whose `cost_multiplier` is chosen so that the product wraps to a value ≤ `u32::MAX`, passing the post-multiplication guard and returning `Ok(Reduction(0, nil))` — zero reported cost — for a program that actually consumed billions of cost units.

---

### Finding Description

`op_unknown` is the handler for any opcode not recognized by `ChiaDialect`. It is reachable whenever `NO_UNKNOWN_OPS` is absent from the dialect flags, which is the case in consensus (block-validation) mode.

The function decodes two fields from the opcode atom: [1](#0-0) 

`cost_multiplier` is at most `u32::MAX` (4 294 967 295) because `u32_from_u8` rejects buffers longer than 4 bytes.

After computing an inner cost bounded by `max_cost` via `check_cost`, the function applies the multiplier: [2](#0-1) 

`cost` and `cost_multiplier` are both `u64`. The multiplication `cost *= cost_multiplier + 1` is **not** checked for overflow. In a Rust release build, `u64` arithmetic wraps modulo 2⁶⁴. The subsequent guard `if cost > u32::MAX as u64` only catches the *wrapped* value; if the wrapped result is ≤ `u32::MAX`, the function returns `Ok` with that small (possibly zero) cost.

**Concrete trigger:**

Choose opcode bytes `[0x7f, 0xff, 0xff, 0xff, 0x40]`:
- `cost_multiplier = u32_from_u8([0x7f, 0xff, 0xff, 0xff]) = 0x7fff_ffff = 2 147 483 647`
- `cost_multiplier + 1 = 2^31`
- `cost_function = (0x40 >> 6) = 1` (ARITH-style cost)

The opcode does **not** start with `0xff 0xff`, so the reserved-opcode guard at line 197 is not triggered. [3](#0-2) 

With `cost_function = 1`, the inner cost formula is:

```
cost = ARITH_BASE_COST + n * ARITH_COST_PER_ARG + total_bytes * ARITH_COST_PER_BYTE
     = 99 + n * 320 + total_bytes * 3
``` [4](#0-3) 

The attacker supplies enough arguments/bytes so that the inner cost equals exactly `2^33 = 8 589 934 592`. This is below the Chia block cost limit of ~11 billion, so `check_cost` passes.

Then:

```
cost *= cost_multiplier + 1
     = 2^33 * 2^31
     = 2^64
     ≡ 0  (mod 2^64)   ← wraps to zero
```

`0 ≤ u32::MAX` → the guard passes → `Ok(Reduction(0, nil))` is returned.

The `Cost` type is `u64`: [5](#0-4) 

`check_cost` only compares; it never detects the post-multiplication wrap: [6](#0-5) 

---

### Impact Explanation

`run_program` accumulates the cost returned by each operator call. If `op_unknown` returns cost 0, the program's total cost is underreported by up to ~8.6 billion units per invocation. An attacker can include multiple such opcodes in a single CLVM program, each consuming close to the block cost limit while reporting zero cost. This allows a transaction to perform far more computation than the block cost limit permits, exhausting node CPU without paying the expected fee — an undercharged execution / consensus-divergence impact. Nodes running debug builds (where overflow panics) would reject the block, while release-build nodes accept it, causing a consensus split.

---

### Likelihood Explanation

- Unknown opcodes are explicitly allowed in consensus mode (`NO_UNKNOWN_OPS` absent).
- `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS`, so the mempool rejects such transactions, but block validators do not. [7](#0-6) [8](#0-7) 

- The attacker controls all opcode bytes and all argument atoms — both are ordinary CLVM bytes in a puzzle/solution.
- The required inner cost (2^33) is achievable with a moderate number of small atom arguments, well within allocator limits.
- No privileged access, social engineering, or dependency compromise is required.

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant and return an error on overflow:

```rust
// Before (vulnerable):
cost *= cost_multiplier + 1;

// After (safe):
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

Additionally, add a `check_cost` call **after** the multiplication to ensure the final cost does not exceed `max_cost`.

### Proof of Concept

```
Opcode atom : [0x7f, 0xff, 0xff, 0xff, 0x40]
  cost_multiplier = 0x7fff_ffff  (2^31 - 1)
  cost_function   = 1            (ARITH-style)

Arguments: craft N atoms whose total byte count T satisfies
  99 + N*320 + T*3 = 2^33 = 8_589_934_592
  e.g. N = 1, T = (8_589_934_592 - 99 - 320) / 3 = 2_863_311_391 bytes
  (split across multiple atoms, each within allocator limits)

Execution (release build):
  inner cost = 2^33                    (passes check_cost with max_cost=11e9)
  cost *= 2^31                         → 2^64 wraps to 0
  0 ≤ u32::MAX                         → guard passes
  return Ok(Reduction(0, nil))         ← zero cost reported
```

The CLVM program `(a (q . (unknown_op arg1 arg2 ...)) 1)` embedding this opcode and the crafted argument list is fully attacker-controlled CLVM bytes, reachable through any Chia coin puzzle/solution pair evaluated in consensus mode.
