### Title
`set_executable_class_hash_v2` Uses `upsert` Instead of `insert`, Violating the Documented Write-Once Invariant of the `stateless_compiled_class_hash_v2` Registry â€” (`File: crates/apollo_storage/src/class_hash.rs`)

### Summary

The `ClassHashStorageWriter` trait explicitly documents that `set_executable_class_hash_v2` must return an error if the class hash already exists, establishing a write-once (immutable) registry invariant for the `stateless_compiled_class_hash_v2` MDBX table. The implementation, however, calls `table.upsert(...)` which silently overwrites any existing entry. This is the direct Sequencer analog of the PermissionRegistry bug: a registry that is supposed to be immutable once written can be silently mutated by any holder of a write transaction, remapping a `ClassHash` to a different `CompiledClassHash` (v2/Blake2). Because `get_compiled_class_hash_v2` is the authoritative source for selecting which compiled class artifact to execute and for computing the CASM-hash migration state diff, a corrupted entry causes the wrong compiled class to be selected for execution and the wrong hash to be committed to the state diff.

### Finding Description

**Invariant stated in the trait (documentation contract):**

```rust
/// Inserts the executable class hash corresponding to the given class hash.
/// An error is returned if the class hash already exists.
fn set_executable_class_hash_v2(
    self,
    class_hash: &ClassHash,
    executable_class_hash_v2: CompiledClassHash,
) -> StorageResult<Self>;
``` [1](#0-0) 

**Actual implementation â€” uses `upsert`, not `insert`:**

```rust
impl<T: StorageTransaction<Mode = RW>> ClassHashStorageWriter for T {
    fn set_executable_class_hash_v2(
        self,
        class_hash: &ClassHash,
        executable_class_hash_v2: CompiledClassHash,
    ) -> StorageResult<Self> {
        let table = self.open_table(&self.tables().stateless_compiled_class_hash_v2)?;
        table.upsert(self.txn(), class_hash, &executable_class_hash_v2)?;
        Ok(self)
    }

``` [2](#0-1) 

The `SimpleTable::upsert` implementation calls `WriteFlags::UPSERT`, which unconditionally overwrites any existing value for the key: [3](#0-2) 

By contrast, `SimpleTable::insert` uses `WriteFlags::NO_OVERWRITE` and returns `DbError::KeyAlreadyExists` if the key is present: [4](#0-3) 

**The `stateless_compiled_class_hash_v2` table is explicitly described as a separate, carefully-scoped registry:**

> "This is a table separate from Papyrus storage; scope and version do not apply on it. Use carefully, only within class manager code." [5](#0-4) 

**Production call path through state sync (`add_class_and_executable_unsafe`):**

The state sync flow for non-backward-compatible classes (Starknet version < 0.12.0) calls `add_class_and_executable_unsafe` on the Class Manager, which bypasses compilation and writes the caller-supplied `executable_class_hash_v2` directly to storage: [6](#0-5) 

The comment in the type definition confirms this is the intended path:
> "This method should only be used through state sync. It acts as a writer to the class storage, and bypasses compilation - thus unsafe." [7](#0-6) 

The call chain is: `add_class_and_executable_unsafe` â†’ `CachedClassStorage::set_class` â†’ `FsClassStorage::set_class` â†’ `mark_class_id_as_existent` â†’ `ClassHashStorage::set_executable_class_hash_v2` â†’ `StorageTransaction::set_executable_class_hash_v2` â†’ `table.upsert(...)`. [8](#0-7) 

The upper-layer guard in `FsClassStorage::set_class` checks the **filesystem** (`contains_class`), not the MDBX table: [9](#0-8) 

The MDBX `stateless_compiled_class_hash_v2` table and the filesystem are two independent stores. A class can be absent from the filesystem (guard passes) while already having an entry in the MDBX table (e.g., after a partial crash-recovery or a reorg replay), causing `upsert` to silently overwrite the existing `CompiledClassHash` with a different value supplied by the caller.

**Direct call in `native_blockifier` (no upper-layer guard):**

`native_blockifier/src/storage.rs` calls `set_executable_class_hash_v2` directly on the storage transaction, with only a Papyrus-state `class_undeclared` check â€” which queries a **different** table than `stateless_compiled_class_hash_v2`: [10](#0-9) 

A class can be "undeclared" in the Papyrus state reader yet already have an entry in the `stateless_compiled_class_hash_v2` table, causing `upsert` to silently overwrite it.

**How the corrupted hash reaches execution:**

`get_compiled_class_hash_v2` in the RPC execution state reader reads directly from the `stateless_compiled_class_hash_v2` table (or via the class manager client which ultimately reads the same table): [11](#0-10) 

`StateReaderAndContractManager::get_compiled_class_hash_v2` (used by the blockifier during execution) reads from the class manager cache, falling back to the state reader which reads from this table: [12](#0-11) 

This hash is used for CASM-hash migration: it determines which `CompiledClassHash` is written into the state diff for executed classes, and it is the value returned to the OS for proof verification.

### Impact Explanation

If the `stateless_compiled_class_hash_v2` entry for a `ClassHash` is silently overwritten with a different `CompiledClassHash`, then:

1. `get_compiled_class_hash_v2` returns the wrong hash for that class during execution.
2. The CASM-hash migration logic writes the wrong hash into the state diff (`class_hash_to_compiled_class_hash`).
3. The state commitment (block hash) is computed over the wrong state diff, producing a divergent block hash.
4. The blockifier selects the wrong compiled class artifact for execution, producing wrong execution results, wrong receipts, and wrong events.

This matches the allowed impact: **Critical â€” Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution** and **Critical â€” Wrong state, receipt, event, or storage value from blockifier/execution logic**.

### Likelihood Explanation

The upper-layer guards (`CachedClassStorage` in-memory check and `FsClassStorage` filesystem check) prevent the overwrite in the common case. However, the storage layer's documented invariant is broken, and the overwrite is reachable when:

- The filesystem and MDBX table are out of sync (e.g., after a crash between `write_class_atomically` and `mark_class_id_as_existent`, followed by manual or automated recovery that repopulates the MDBX table with a different hash).
- The `native_blockifier` Python path is used (Echonet replay), where only the Papyrus-state `class_undeclared` check is applied, not the filesystem check.
- A reorg or re-sync scenario causes `add_class_and_executable_unsafe` to be called for a class whose filesystem entry was deleted but whose MDBX entry was not.

Likelihood is **Low** for the main production sequencer path, but the storage-layer invariant violation is unconditional and the fix is trivial.

### Recommendation

Replace `table.upsert(...)` with `table.insert(...)` in the implementation of `set_executable_class_hash_v2`:

```rust
impl<T: StorageTransaction<Mode = RW>> ClassHashStorageWriter for T {
    fn set_executable_class_hash_v2(
        self,
        class_hash: &ClassHash,
        executable_class_hash_v2: CompiledClassHash,
    ) -> StorageResult<Self> {
        let table = self.open_table(&self.tables().stateless_compiled_class_hash_v2)?;
        // Use insert (not upsert) to enforce the write-once invariant documented in the trait.
        table.insert(self.txn(), class_hash, &executable_class_hash_v2)?;
        Ok(self)
    }
}
```

All callers that need idempotent behavior (e.g., state sync replay) should check `get_executable_class_hash_v2` first and skip the write if the entry already exists, rather than relying on silent overwrite at the storage layer.

### Proof of Concept

1. Start a sequencer node and sync it to a block that declares a Cairo 1 class `C` with `class_hash = H` and `compiled_class_hash_v2 = V1`. The MDBX table now contains `H â†’ V1`.

2. Simulate a partial crash: delete the filesystem files for class `H` (so `FsClassStorage::contains_class(H)` returns `false`) but leave the MDBX entry `H â†’ V1` intact.

3. Trigger `add_class_and_executable_unsafe(H, sierra, V2, casm2)` where `V2 â‰  V1` (e.g., via the state sync path re-fetching the class from a Feeder Gateway that returns a different CASM, or via the `native_blockifier` Python API).

4. Because `FsClassStorage::contains_class(H)` returns `false`, the guard passes. `write_class_atomically` writes the new class files. `mark_class_id_as_existent` calls `set_executable_class_hash_v2(H, V2)`, which calls `table.upsert(H, V2)`, silently overwriting `V1` with `V2`.

5. Subsequent calls to `get_compiled_class_hash_v2(H)` return `V2` instead of `V1`. The blockifier uses `V2` for CASM-hash migration, writing `H â†’ V2` into the state diff. The state commitment diverges from the canonical chain.
