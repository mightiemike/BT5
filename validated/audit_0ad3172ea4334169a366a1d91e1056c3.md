Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap` — the direct pool caller. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router contract address, not the originating user. A pool admin who allowlists the router address (a natural action to enable router-mediated swaps) inadvertently grants every router user unrestricted swap access, rendering the allowlist ineffective for all router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to every configured extension. `SwapAllowlistExtension.beforeSwap` then performs its allowlist check keyed on that `sender`: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract: [3](#0-2) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. In all cases, the pool receives the router as `msg.sender`, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`. If the pool admin has allowlisted the router (the natural action to enable router-mediated swaps for their allowlisted users), the check passes for every user who routes through the router, regardless of individual allowlist status.

The asymmetry with `DepositAllowlistExtension` confirms this is a bug, not a design choice — that extension correctly checks `owner` (the actual position owner passed explicitly by the pool), not `sender`: [4](#0-3) 

No existing guard prevents this: `allowedSwapper[pool][router] = true` is a valid admin-set state, and the extension has no mechanism to distinguish "router acting on behalf of user X" from "user X calling directly."

## Impact Explanation
Any user can swap against a pool whose `SwapAllowlistExtension` is configured to restrict access to specific addresses, simply by routing through `MetricOmmSimpleRouter`. The allowlist — intended to gate who may trade against the pool's liquidity — is rendered entirely ineffective for all router-mediated swaps. Unauthorized users gain full swap access, enabling direct loss of LP principal through unrestricted arbitrage or manipulation against a pool designed to trade only with trusted counterparties. This constitutes broken core pool functionality causing loss of funds and an admin-boundary break where the extension's access control decision is corrupted.

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router address. This is a natural and expected administrative action: a pool admin who wants their allowlisted users to trade via the standard router will call `setAllowedToSwap(pool, router, true)`, not realizing this grants access to all router users. The router is a public, permissionless contract with no mechanism to restrict which users call it. The misunderstanding is structurally induced by the API: `setAllowedToSwap(pool, router, true)` reads as "allow the router to swap," but its actual effect is "allow everyone who uses the router to swap." The condition is reachable by any unprivileged user once the admin has taken this natural configuration step.

## Recommendation
The extension must check the originating user, not the direct pool caller. The most robust code-level fix is to have the router encode `msg.sender` into `extensionData` before calling the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known router. Alternatively, extend the hook convention so the pool or router passes the true originating user as a dedicated field. The `DepositAllowlistExtension` pattern — checking an explicitly passed `owner` rather than `sender` — should be the model. Documenting "do not allowlist the router" is insufficient; a code-level fix is required.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension configured (allowAllSwappers = false).
2. Pool admin calls: extension.setAllowedToSwap(pool, router, true)
   // Intent: "let the router facilitate swaps for my allowlisted users"
   // Actual effect: allowedSwapper[pool][router] = true

3. Non-allowlisted attacker calls:
   router.exactInputSingle(ExactInputSingleParams{
     pool: pool,
     zeroForOne: true,
     amountIn: X,
     ...
   })
   // router is msg.sender of pool.swap → sender = router

4. Pool calls: _beforeSwap(msg.sender=router, ...)
5. Extension evaluates: allowedSwapper[pool][router] → true ← bypass succeeds
6. Swap executes. allowedSwapper[pool][attacker] was never checked.
```
Reproducible as a Foundry unit test: deploy pool with `SwapAllowlistExtension`, set `allowedSwapper[pool][router] = true`, confirm a non-allowlisted EOA can swap successfully via the router but is correctly rejected when calling `pool.swap` directly.
