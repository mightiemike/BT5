### Title
Unchecked Return Value of `token.approve()` in `creditDeposit()` Silently Fails for Non-Standard ERC20 Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without checking the returned boolean. For ERC20 tokens that return `false` on failure instead of reverting, the approval silently fails, and the subsequent `endpoint.depositCollateralWithReferral()` call will revert when the endpoint attempts `transferFrom` without allowance. User funds sitting in the DDA contract cannot be credited into the protocol, and the caller receives no meaningful error about the root cause.

---

### Finding Description

`creditDeposit()` is a public, permissionless function callable by any address. It iterates over all spot product tokens, checks the DDA's balance, and for each non-zero balance calls `approve` followed by `depositCollateralWithReferral`: [1](#0-0) 

The `approve` call on line 92 is a direct call on the `IERC20Base` interface, which declares `approve` as returning `bool`: [2](#0-1) 

The return value is discarded. The `ERC20Helper` library provides `safeTransfer` and `safeTransferFrom` wrappers that check return values, but **no `safeApprove` equivalent exists**: [3](#0-2) 

The same pattern appears in `ContractOwner.wrapVaultAsset()`, also a permissionless external function, where two consecutive `approve` calls are made without checking their return values: [4](#0-3) 

---

### Impact Explanation

For any ERC20 collateral token that returns `false` on a failed `approve` (rather than reverting — a known pattern in non-standard tokens such as USDT on some chains), the approval silently fails. The endpoint then attempts `transferFrom` on the DDA with zero allowance, causing the deposit to revert. Funds deposited into the DDA by users are not credited to their subaccount and remain stranded in the DDA contract. Because `creditDeposit()` does not revert with a meaningful error at the `approve` step, the failure is opaque to callers and off-chain monitoring.

---

### Likelihood Explanation

`creditDeposit()` is callable by any unprivileged address with no access control. The Nado protocol supports multiple spot collateral tokens via `spotEngine.getProductIds()`. If any listed token follows the non-reverting `approve` pattern, every call to `creditDeposit()` for that token silently fails the approval step. The likelihood is medium: it depends on the specific tokens listed, but the code path is always reachable.

---

### Recommendation

Replace the bare `approve` calls with a safe wrapper that checks the return value, analogous to the existing `ERC20Helper.safeTransfer`:

```solidity
// In ERC20Helper.sol, add:
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Then replace the unchecked calls:

- `DirectDepositV1.sol` line 92: `token.approve(address(endpoint), balance)` → `token.safeApprove(address(endpoint), balance)`
- `ContractOwner.sol` lines 530–531: both `assetToken.approve(...)` calls → `assetToken.safeApprove(...)`
- `ContractOwner.sol` line 254: `quoteToken.approve(address(endpoint), uint256(amount))` → `quoteToken.safeApprove(...)`

---

### Proof of Concept

1. A non-standard ERC20 token (e.g., one that returns `false` on `approve` when a precondition fails) is listed as a spot collateral product.
2. A user sends tokens directly to the DDA contract address.
3. Any caller invokes `DirectDepositV1.creditDeposit()`.
4. `token.approve(address(endpoint), balance)` returns `false`; the return value is ignored.
5. `endpoint.depositCollateralWithReferral(...)` is called; the endpoint's internal `transferFrom` reverts because allowance is 0.
6. The entire `creditDeposit()` call reverts with a generic transfer error, not an approval error.
7. User funds remain stranded in the DDA with no protocol-level credit. [5](#0-4)

// @dev Adapted from https://github.com/Uniswap/v3-core/blob/main/contracts/libraries/TransferHelper.sol
library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
