### Title
USDT Deposits Permanently Bricked in `DirectDepositV1` After Below-Minimum Deposit Skipped by Endpoint - (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` unconditionally before calling `endpoint.depositCollateralWithReferral(...)`. The `Endpoint` silently returns without consuming the allowance when the deposit amount is below the minimum threshold. This leaves a non-zero residual allowance. On any subsequent call to `creditDeposit()`, USDT (and any other approval-reset-required token) reverts on the second `approve`, permanently bricking deposits for that DDA.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each product token with a non-zero balance, the contract first sets an allowance and then calls the endpoint:

```solidity
// DirectDepositV1.sol lines 91-98
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
``` [1](#0-0) 

In `Endpoint.depositCollateralWithReferral()`, there is an explicit early-return path when the deposit amount is below the minimum — **without calling `handleDepositTransfer`**, meaning the allowance is never consumed:

```solidity
// Endpoint.sol lines 137-142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

The `handleDepositTransfer` (which calls `transferFrom` and consumes the allowance) is only reached if `isValidDepositAmount` returns true: [3](#0-2) 

After the silent skip, the DDA holds a non-zero USDT allowance for the endpoint. On the next call to `creditDeposit()` — whether triggered by a new deposit or a retry — the `token.approve(address(endpoint), newBalance)` call reverts for USDT because USDT requires the allowance to be reset to 0 before setting a new non-zero value.

`creditDeposit()` has no access modifier and is callable by anyone: [4](#0-3) 

The `ContractOwner.wrapVaultAsset()` function, by contrast, already correctly performs approve-to-0 before approving a new amount, demonstrating the protocol is aware of this pattern in at least one place: [5](#0-4) 

---

### Impact Explanation

Once a USDT-based DDA has a non-zero residual allowance (from a skipped below-minimum deposit), every subsequent call to `creditDeposit()` reverts at the `approve` line. The DDA can never successfully deposit USDT into the protocol again. Any USDT sent to that DDA is permanently stuck — it cannot be credited to the subaccount via the normal deposit flow. The `withdraw` function (owner-only) can recover the tokens, but the deposit path is permanently broken for that DDA instance.

---

### Likelihood Explanation

The trigger is realistic: a user sends a small USDT amount to their DDA (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`), and `creditDeposit()` is called (by anyone, including bots or the user themselves). The endpoint silently skips the deposit, leaving the allowance set. The next `creditDeposit()` call — even with a valid amount — reverts. USDT is a primary collateral asset in DeFi protocols, making this a high-probability scenario. [6](#0-5) 

---

### Recommendation

In `DirectDepositV1.creditDeposit()`, reset the allowance to 0 before setting the new approval, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use OpenZeppelin's `SafeERC20.forceApprove()` which handles this atomically.

---

### Proof of Concept

1. A USDT-backed spot product exists in the protocol.
2. A DDA is created for subaccount `S` via `ContractOwner.createDirectDepositV1(S)`.
3. A user sends 1 wei of USDT to the DDA address (below `MIN_FIRST_DEPOSIT_AMOUNT`).
4. Anyone calls `DirectDepositV1(dda).creditDeposit()`.
   - `balance = 1`, so `token.approve(endpoint, 1)` is called — USDT allowance is now 1.
   - `endpoint.depositCollateralWithReferral(S, productId, 1, "-1")` is called.
   - Inside the endpoint, `isValidDepositAmount` returns `false` → early `return` without `transferFrom`.
   - Allowance remains at 1.
5. The user now sends 10,000 USDT to the DDA (a valid deposit amount).
6. Anyone calls `DirectDepositV1(dda).creditDeposit()` again.
   - `balance = 10000e6`, so `token.approve(endpoint, 10000e6)` is called.
   - **USDT reverts** because current allowance (1) is non-zero and new value is non-zero.
   - The entire `creditDeposit()` call reverts. The 10,000 USDT is stuck in the DDA and cannot be deposited. [4](#0-3) [7](#0-6)
