Audit Report

## Title
`AnchoredPriceProvider` L1-Only Staleness Check Causes Swap Failures on L2 Due to Sequencer Clock Skew - (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary

`AnchoredPriceProvider`, described as "the one standard provider for public pools," contains a three-argument `_isStale` function explicitly annotated `(L1)` that unconditionally classifies any oracle `refTime > block.timestamp` as stale. On L2 networks like Base, sequencer clock skew routinely causes oracle publication timestamps to be slightly ahead of `block.timestamp`, causing `_readLeg` to return `ok = false`, `getBidAndAskPrice()` to revert with `FeedStalled()`, and all swaps through affected pools to revert with `PriceProviderFailed`. No `AnchoredPriceProviderL2` variant exists in the codebase.

## Finding Description

The protocol explicitly separates L1 and L2 staleness logic. `PriceProviderL2` introduces a `FUTURE_TOLERANCE` immutable and a four-argument `_isStale` that tolerates oracle `refTime` up to `FUTURE_TOLERANCE` seconds ahead of `block.timestamp`:

```solidity
// PriceProviderL2.sol L135-150
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

`AnchoredPriceProvider` has no L2 variant. Its `_isStale` at line 228 hard-rejects any future `refTime`:

```solidity
// AnchoredPriceProvider.sol L221-230
/// @dev Pure staleness check (L1). Any future refTime is stale.
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;  // ← broken on L2
    return (nowTs - refTime) > maxDelta;
}
```

The exploit path is:
1. Oracle publishes a price with `refTime = block.timestamp + N` (N ≥ 1, normal Base sequencer skew).
2. `_readLeg` calls `_isStale(refTime, block.timestamp, MAX_REF_STALENESS)` → hits `if (refTime > nowTs) return true` → returns `(mid, spreadBps, refTime, false)`.
3. `_getBidAndAskPrice` receives `ok = false` → returns `(0, type(uint128).max)`.
4. `getBidAndAskPrice()` checks `bid == 0 || ask == type(uint128).max` → reverts `FeedStalled()`.
5. `MetricOmmPool._getBidAndAskPriceX64()` catches the revert → reverts `PriceProviderFailed(reason)`.
6. All `swap()` calls through any pool using `AnchoredPriceProvider` on Base fail for the duration of the skew window.

The glob search confirms no `AnchoredPriceProviderL2.sol` exists in the repository, while `PriceProviderFactoryL2.sol` and `PriceProviderL2.sol` do exist, confirming the intentional L1/L2 split was not extended to the anchored variant.

## Impact Explanation

Every swap through a pool whose price provider is an `AnchoredPriceProvider` fails during any period of sequencer clock skew on Base. `AnchoredPriceProvider` is the designated standard provider for public pools. The broken value is the `ok` boolean returned by `_readLeg`, which is incorrectly `false` for a valid, fresh oracle price. This matches the allowed impact: "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows." LPs cannot exit via swap-based routes and traders cannot execute during skew windows.

## Likelihood Explanation

Base is an explicitly supported L2 chain (evidenced by `PriceProviderFactoryL2.sol` and `PriceProviderL2.sol` in the codebase). Sequencer clock skew causing `refTime > block.timestamp` is a documented, recurring condition on optimistic rollups — it is precisely why `PriceProviderL2` was written with `FUTURE_TOLERANCE`. No privileged action, malicious setup, or non-standard token behavior is required. Any ordinary swap call during a skew window triggers the failure. The condition is unpredictable and repeating.

## Recommendation

Create an `AnchoredPriceProviderL2` variant that adds a `FUTURE_TOLERANCE` immutable and replaces the three-argument `_isStale` with a four-argument version mirroring `PriceProviderL2`:

```solidity
uint256 public immutable FUTURE_TOLERANCE;

function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal view returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > FUTURE_TOLERANCE;
    }
    return (nowTs - refTime) > maxDelta;
}
```

Alternatively, add `FUTURE_TOLERANCE` directly to `AnchoredPriceProvider` (defaulting to 0 for L1 deployments) and use it in the existing `_isStale`, consistent with how `PriceProviderL2` already handles this.

## Proof of Concept

1. Deploy `AnchoredPriceProvider` on Base with a Pyth Lazer or Chainlink Data Streams feed.
2. Oracle publishes a price update with `refTime = block.timestamp + 2` (two-second sequencer clock skew, within normal Base operating range).
3. Call `pool.swap(...)`.
4. `_getBidAndAskPriceX64` calls `provider.getBidAndAskPrice()`.
5. Inside `_readLeg`, `_isStale(T+2, T, MAX_REF_STALENESS)` hits `if (refTime > nowTs) return true` → stale.
6. `_readLeg` returns `ok = false`; `_getBidAndAskPrice` returns `(0, type(uint128).max)`.
7. `getBidAndAskPrice()` reverts `FeedStalled()`.
8. `_getBidAndAskPriceX64` catches → reverts `PriceProviderFailed(...)`.
9. `swap()` reverts. All swaps are blocked for the duration of the clock skew window.

A Foundry fork test on Base can reproduce this by warping `block.timestamp` one second behind the oracle's `refTime` and asserting that `pool.swap(...)` reverts with `PriceProviderFailed`.
