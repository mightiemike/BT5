### Title
Spread Health Bonus Uses `longWeight` for Short Legs, Inflating Subaccount Health Score — (`core/contracts/Clearinghouse.sol`, `core/contracts/libraries/RiskHelper.sol`)

---

### Summary

The spread health bonus calculation in `Clearinghouse.getHealth` and `RiskHelper._getSpreadWeightX18` always uses `longWeight` for both legs of a spread position, regardless of whether each leg is long or short. For the short leg, `shortWeight` (> 1) should be used. Using `longWeight` (< 1) for the short leg underestimates `existingWeight` and miscalculates `spreadWeight`, producing a systematically inflated health bonus for any spread position. This allows users to appear healthier than they are, enabling excess borrowing, over-leveraging, and liquidation avoidance.

---

### Finding Description

In `Clearinghouse.getHealth`, the spread health adjustment is:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price).mul(spreadWeight - existingWeight);
```

`existingWeight` always averages `longWeight` from both legs regardless of position direction. For a long-spot/short-perp spread, the short perp leg should contribute `perpCoreRisk.shortWeight` (> 1) to `existingWeight`, not `perpCoreRisk.longWeight` (< 1). The same asymmetry applies to the short-spot/long-perp case.

`RiskHelper._getSpreadWeightX18` compounds the error:

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
}
```

In both branches, `longWeight` is used for the short leg. For the short leg, `ONE - longWeight` is positive (e.g., `1 - 0.9 = 0.1`), yielding `spreadWeight ≈ 0.98`. If `shortWeight` were used instead, `ONE - shortWeight` would be negative (e.g., `1 - 1.1 = -0.1`), yielding `spreadWeight ≈ 1.02`, capped at `maxSpreadWeight` (0.99 or 0.994).

The `IProductEngine.CoreRisk` struct only exposes `longWeight`, so neither function can access `shortWeight` even if the intent were correct.

---

### Impact Explanation

For a typical spread with `longWeight = 0.9` and `shortWeight = 1.1`:

| | Current (buggy) | Correct |
|---|---|---|
| `existingWeight` | `(0.9 + 0.9) / 2 = 0.9` | `(0.9 + 1.1) / 2 = 1.0` |
| `spreadWeight` | `1 - 0.1/5 = 0.98` | `1 - (-0.1)/5 = 1.02` → capped at `0.99` |
| `spreadWeight - existingWeight` | `+0.08` | `-0.01` |
| Spread contribution | `basisAmount × 2price × 0.08` **(bonus)** | `basisAmount × 2price × (-0.01)` **(penalty)** |

The net overstatement is `basisAmount × 2price × 0.09`. For a $1 M notional spread position, this is **$180,000 of phantom health**. This allows a user to:

- Withdraw collateral that should be locked
- Open additional positions beyond their true margin capacity
- Avoid liquidation when their true health is below the maintenance threshold

The individual leg contributions in `getHealthContribution` are computed correctly (using `_getWeightX18` which selects long/short weight by sign of `amount`). The inflation is entirely in the spread bonus layer.
