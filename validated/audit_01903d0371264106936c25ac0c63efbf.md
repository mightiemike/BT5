### Summary

`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase merkle proof. However, the original `verify_transaction_inclusion` (v1) is still declared `pub` on the NEAR contract and is callable by any unprivileged account. An attacker can call v1 directly, bypassing the coinbase proof check entirely, and obtain a `true` result for a forged transaction inclusion claim.

---

### Finding Description

`verify_transaction_inclusion_v2` adds a mandatory coinbase proof validation step before delegating to v1: [1](#0-0) 

The coinbase check at lines 358â€“365 is the only guard against the 64-byte internal-node forgery. After it passes, v2 calls v1 internally with `#[allow(deprecated)]`.

However, v1 itself remains a `pub` NEAR method: [2](#0-1) 

On NEAR, `pub` on a contract method means it is a callable entry point for any account. The Rust `#[deprecated]` attribute only emits a compiler warning; it does not restrict on-chain invocations. The contract's own doc-comment acknowledges the risk: [3](#0-2) 

The full v1 body performs only a Merkle root comparison with no coinbase anchor: [4](#0-3) 

`compute_root_from_merkle_proof` in `merkle-tools` is a pure hash-chain computation with no structural constraints on the leaf: [5](#0-4) 

---

### Impact Explanation

The 64-byte forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions) works by supplying a 64-byte value that is actually an internal Merkle tree node as the `tx_id`. Because `compute_root_from_merkle_proof` treats the leaf as an opaque hash, it will compute a valid root for this internal node, and v1 will return `true` for a transaction that does not exist.

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate fund releases, cross-chain actions, or other security-critical decisions can be deceived into accepting a forged proof. The protection introduced by v2 is completely bypassed because v1 is still reachable as a first-class public method.

---

### Likelihood Explanation

The entry path requires no privileges. Any NEAR account can call `verify_transaction_inclusion` directly with a crafted `ProofArgs`. The attacker only needs:
1. A real, confirmed Bitcoin block whose header is stored in the contract's `headers_pool`.
2. A 64-byte value that is a valid internal Merkle node of that block's transaction tree, together with the corresponding sibling path that reconstructs the block's `merkle_root`.

Both are derivable from public Bitcoin chain data. No key material, relayer access, or role is required.

---

### Recommendation

Make `verify_transaction_inclusion` (v1) a private method (remove `pub`, or rename to `verify_transaction_inclusion_inner`) so it is no longer an on-chain entry point. Only `verify_transaction_inclusion_v2` should be externally callable. This is the direct analog of the recommended fix in the external report: making `slash()` internal so only the protected path (commit-reveal) is reachable. [6](#0-5) 

---

### Proof of Concept

1. Identify a Bitcoin block `B` whose header is stored in the contract (obtainable via `get_block_hash_by_height`).
2. From public Bitcoin data, obtain the full transaction list of `B` and compute its Merkle tree.
3. Pick any internal node `N` at depth 1 (i.e., `SHA256d(tx0 || tx1)`). `N` is 32 bytes; pad or use it directly as a 32-byte `H256`.
4. Construct a `ProofArgs` where:
   - `tx_id` = `N`
   - `tx_block_blockhash` = hash of `B`
   - `tx_index` = 1 (position of `N` in the next level up)
   - `merkle_proof` = the sibling path from `N`'s position to the root
   - `confirmations` = any valid value
5. Call `verify_transaction_inclusion` (v1) directly on the NEAR contract.
6. The contract returns `true` â€” confirming inclusion of a transaction that does not exist â€” because `compute_root_from_merkle_proof(N, 1, siblings)` reconstructs the real `merkle_root` of block `B`. [7](#0-6) 

`verify_transaction_inclusion_v2` would have rejected this call at the coinbase proof check (lines 358â€“365 of `lib.rs`), but v1 has no such guard and is directly reachable. [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
