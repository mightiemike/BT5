### Title
Softfork Cost Atom Non-Canonical Encoding Causes Consensus/Mempool Divergence — (`src/run_program.rs`, `src/op_utils.rs`)

### Summary

`apply_op` passes `self.dialect.flags()` into `uint_atom::<8>` when parsing the softfork cost argument. In `MEMPOOL_MODE`, those flags include `CANONICAL_INTS`, which causes `uint_atom` to **reject** any cost atom with an unnecessary leading zero byte. In consensus mode (`ClvmFlags::empty()`), the same function **silently strips** leading zeros and returns the numeric value. A CLVM program whose softfork cost is encoded as `0x0001` (non-canonical for `1`) is therefore accepted by consensus and rejected by mempool — a concrete, locally testable divergence.

### Finding Description

**`uint_atom` with `CANONICAL_INTS` (mempool path):** [1](#0-0) 

For input bytes `[0x00, 0x01]`:
- `buf[0] == 0` → true
- `buf.len() < 2` → false
- `buf[1] & 0x80 == 0` → true (0x01 has no high bit)
- Condition fires → `Err(InvalidOpArg, "softfork requires u64 arg with no leading zeros")`

**`uint_atom` without `CANONICAL_INTS` (consensus path):** [2](#0-1) 

For the same `[0x00, 0x01]`: leading zero is stripped, `buf = [0x01]`, returns `Ok(1)`.

**The call site passes dialect flags directly:** [3](#0-2) 

**`MEMPOOL_MODE` includes `CANONICAL_INTS`:** [4](#0-3) 

**The codebase itself documents that consensus mode accepts non-canonical cost atoms:** [5](#0-4) 

The test at line 1067 uses `ClvmFlags::empty()` and `0x00000397` (non-canonical for 919) and expects `result: Some("()")` — confirming consensus accepts it. No equivalent test exists for `MEMPOOL_MODE` with the same input, because it would fail.

### Impact Explanation

A transaction containing a CLVM generator that invokes `softfork` with a non-canonical cost atom (e.g., `0x0001` for cost=1) will:
- **Pass** full-node consensus validation (block accepted)
- **Fail** mempool validation with `InvalidOpArg`

This means the transaction can only reach the chain via a miner who bypasses the mempool entirely. Any node running mempool validation will reject it and refuse to relay it. This is a concrete consensus/mempool equivalence break: the same CLVM bytes produce different accept/reject outcomes depending solely on which mode the evaluator is called in.

### Likelihood Explanation

The attacker-controlled path is straightforward: craft a CLVM spend bundle whose generator contains `(softfork (q . 0x0001) ...)` and submit it directly to a miner's RPC endpoint. No special privileges are required. The non-canonical encoding is trivially producible by any CLVM serializer that does not enforce canonical integer encoding.

### Recommendation

In `apply_op`, strip leading zeros from the softfork cost atom unconditionally before the `uint_atom` call, or parse the cost with `ClvmFlags::empty()` regardless of dialect mode. The cost argument is a protocol-level field whose encoding should be normalized identically in both modes. The `CANONICAL_INTS` strictness is appropriate for operator arguments inside the softfork body, but applying it to the cost field itself creates this divergence.

### Proof of Concept

```rust
// Rust differential test
use clvm_rs::chia_dialect::{ChiaDialect, ClvmFlags, MEMPOOL_MODE};
use clvm_rs::run_program::run_program;
use clvm_rs::allocator::Allocator;

// CLVM: (softfork (q . 0x0001) (q . 0) (q . (q . 1)) (q . ()))
// cost atom is 0x0001 — non-canonical encoding of 1

let prg_bytes = /* serialized bytes of the above program */;

let mut a = Allocator::new();
let prg = a.deserialize(&prg_bytes).unwrap();
let env = a.nil();

let consensus = ChiaDialect::new(ClvmFlags::empty());
let mempool   = ChiaDialect::new(MEMPOOL_MODE);

let r_consensus = run_program(&mut a, &consensus, prg, env, 10_000);
let r_mempool   = run_program(&mut a, &mempool,   prg, env, 10_000);

// r_consensus → Ok(...)   (leading zero stripped, cost = 1)
// r_mempool   → Err(InvalidOpArg "softfork requires u64 arg with no leading zeros")
assert!(r_consensus.is_ok());
assert!(r_mempool.is_err());  // divergence confirmed
```
