### Title
Valid `InvokeTransactionV3` with zero L2/L1-data gas bounds silently rejected during P2P propagation due to `ValidResourceBounds` variant collapse in protobuf deserialization — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter unconditionally collapses any resource-bounds message where both `l2_gas` and `l1_data_gas` are zero into `ValidResourceBounds::L1Gas(l1_gas)`. Because the gateway accepts — and the hash domain commits to — `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` as a fully valid transaction type, the round-trip through protobuf changes the variant. The downstream `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion then hard-rejects any non-`AllResources` variant, causing every such transaction to be silently dropped during mempool and consensus P2P propagation.

### Impact Explanation

Any `InvokeTransactionV3` submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` (only L1 gas specified, which the gateway accepts) is permanently unpropagateable via P2P. The originating node accepts and stores it; every peer node that receives it via mempool or consensus P2P deserialization maps it to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and drops it. The transaction can never be sequenced by any node other than the one that received it directly from the user. This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Additionally, if the hash is ever recomputed from the deserialized form (e.g., during RPC simulation or fee estimation on a synced node), the computed hash diverges from the canonical hash stored in the block, producing an authoritative-looking wrong value. This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The gateway explicitly accepts and tests `AllResourceBounds` with zero L2 and L1-data gas. Any user who specifies only L1 gas bounds (a common pattern for pre-0.13.3 style transactions submitted through the new API) triggers this path. The TODO comment in the converter (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) confirms this is a known transitional code path that has not been cleaned up, making the collision window indefinitely open.

### Recommendation

Remove the variant-collapsing heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. When all three fields (`l1_gas`, `l2_gas`, `l1_data_gas`) are present in the protobuf message, always produce `ValidResourceBounds::AllResources(...)` regardless of whether `l2_gas` and `l1_data_gas` are zero. Reserve `L1Gas` only for messages that genuinely omit `l2_gas` (i.e., pre-0.13.3 wire format where `l2_gas` is absent, not merely zero). This preserves the hash domain invariant and prevents the downstream `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejection.

### Proof of Concept

1. Submit to the gateway an `InvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`. The gateway accepts it and computes hash `H_all` (Poseidon over 3 resource felts including `L1_DATA_GAS_concat_zero`).

2. The originating node serializes the transaction to `protobuf::MempoolTransaction` → `protobuf::InvokeV3WithProof` → `protobuf::InvokeV3` with `resource_bounds = { l1_gas: Some(X), l2_gas: Some(zero), l1_data_gas: Some(zero) }`.

3. A peer node receives the protobuf message. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(X)`.

4. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` matches `L1Gas` against the `_ =>` arm → returns `Err(StarknetApiError::OutOfRange)`.

5. `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` maps the error to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and returns `Err(...)`.

6. The peer node drops the transaction. It is never added to the peer's mempool and never sequenced by that node.

7. If the peer node later receives the same transaction in a block via state sync and recomputes the hash, it obtains `H_l1` (Poseidon over 2 resource felts, omitting `L1_DATA_GAS_concat_zero`) ≠ `H_all`, producing a hash mismatch for any hash-verification step.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
