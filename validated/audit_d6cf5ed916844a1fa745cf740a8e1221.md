### Title
Non-Canonical Multi-Atom Group Encoding Accepted by `strict` Mode — (`src/serde_2026/de.rs`)

### Summary

The `deserialize_2026_body_from_stream` function accepts a negative `length_val` with `count=1` in the atom table, producing a second valid encoding for any single atom. This is a concrete canonicality violation: two distinct byte sequences decode to the same CLVM tree, and the `strict=true` mode — which is supposed to enforce canonical encoding — does not catch it.

### Finding Description

The atom table parser in `de.rs` branches on the sign of `length_val`:

- **Positive** `length_val`: singleton group — `(length, count) = (length_val, 1)` [1](#0-0) 
- **Negative** `length_val`: multi-atom group — reads an explicit `count` varint from the stream [2](#0-1) 

The only guard after the branch is:

```rust
if length == 0 || count == 0 {
    return Err(EvalErr::SerializationError);
}
``` [3](#0-2) 

There is **no rejection of `count == 1` on the negative path**. This means:

| Encoding | `length_val` | `count` | Result |
|---|---|---|---|
| Canonical (singleton) | `+48` | implicit 1 | `(length=48, count=1)` |
| Non-canonical (multi-atom) | `-48` | explicit `1` | `(length=48, count=1)` |

Both are accepted and produce identical `NodePtr` output. The `strict` flag only rejects non-minimal varint byte encodings via `varint_size` comparison — it has no awareness of this semantic redundancy. [4](#0-3) 

The serializer itself enforces `count > 1` for negative-length groups (confirmed by the test assertion `assert!(len > 0 && count > 1)` at line 377 of `tests.rs`), but the deserializer does not mirror this invariant. [5](#0-4) 

### Impact Explanation

The concrete impact is a **broken `strict`-mode canonicality guarantee**: two distinct byte sequences are both accepted as valid serde_2026 blobs and decode to the same CLVM tree. If serde_2026 blob bytes are used as identifiers (e.g., hashed for transaction IDs or block commitments), two blobs with different bytes but identical CLVM content will produce different hashes, enabling a split between nodes that hash the blob vs. nodes that hash the decoded tree.

The specific BLS/`op_bls_verify` impact claimed in the question is **not proven**: both encodings produce the same atom bytes, so BLS verification behaves identically for both. There is no path where one encoding passes and the other fails BLS verification.

The consensus split scenario requires nodes that enforce strict canonicality (rejecting `count=1` for negative `length_val`) to coexist with nodes that do not. Currently, **no** node enforces this check — even `strict=true` accepts both forms — so there is no active consensus split. The risk is forward-looking: if a future version adds the missing check, old blobs with the non-canonical encoding would be rejected by upgraded nodes but accepted by legacy nodes.

### Likelihood Explanation

An attacker can craft a valid serde_2026 blob with `length_val=-48, count=1` followed by 48 bytes of a G1 pubkey. This blob is accepted by all current deserializer configurations (`strict=true` or `strict=false`). The crafting requires only knowledge of the varint encoding format, which is fully documented in `varint.rs`. [6](#0-5) 

### Recommendation

Add a guard on the negative-`length_val` path rejecting `count == 1`, since a single atom of a given length must use the positive (singleton) encoding:

```rust
if length == 0 || count == 0 || (length_val < 0 && count == 1) {
    return Err(EvalErr::SerializationError);
}
```

Apply the same guard in `serialized_length_serde_2026` at `de.rs` line 192 to keep the two functions consistent. [7](#0-6) 

### Proof of Concept

```rust
// Blob A: canonical singleton encoding of a 1-byte atom b"X"
// group_count=1, length_val=+1, atom=b"X", instruction_count=1, inst=2 (push atom[0])
let canonical: &[u8] = &[0x01, 0x01, b'X', 0x01, 0x02];

// Blob B: non-canonical multi-atom group with count=1 for the same atom
// group_count=1, length_val=-1 (varint 0x7f), count=1 (varint 0x01), atom=b"X",
// instruction_count=1, inst=2
let non_canonical: &[u8] = &[0x01, 0x7f, 0x01, b'X', 0x01, 0x02];

let mut a1 = Allocator::new();
let n1 = deserialize_2026_body_from_stream(&mut a1, &mut Cursor::new(canonical),
    usize::MAX, true).unwrap();

let mut a2 = Allocator::new();
let n2 = deserialize_2026_body_from_stream(&mut a2, &mut Cursor::new(non_canonical),
    usize::MAX, true).unwrap(); // accepted even with strict=true

assert_eq!(a1.atom(n1).as_ref(), b"X");
assert_eq!(a2.atom(n2).as_ref(), b"X"); // same content, different bytes
```

Both calls succeed. The `strict=true` flag does not distinguish them. [8](#0-7)
