### Title
Parent `ShardUId` Constructed with Next Layout's Version Instead of Current Layout's Version at Resharding Boundary - (File: chain/chain/src/resharding/event_type.rs)

### Summary
In `ReshardingEventType::from_shard_layout`, the parent shard's `ShardUId` is constructed using `next_shard_layout.version()` instead of the current (pre-split) layout's version. The code comment explicitly acknowledges this is "technically wrong" but claims it is "ok" because the shard layout version has been frozen since `ShardLayoutV2`. This assumption breaks when the first dynamic resharding transitions a `ShardLayoutV2` (version 2) base into a `ShardLayoutV3` (always version 3), producing a parent `ShardUId` with the wrong version byte. The mismatch causes the resharding to be silently skipped in `split_shard`, leaving the shard layout updated in epoch info while the underlying trie state is never actually split.

### Finding Description
`ReshardingEventType::from_shard_layout` receives the **new** (post-split) shard layout and must reconstruct the `ShardUId` of the parent shard that is being split. The parent shard no longer exists in the new layout, so its UID cannot be derived from it directly. The code constructs it as:

```rust
let parent_shard = ShardUId::new(next_shard_layout.version(), parent_id);
```

The inline comment reads:

> "Please note the use of the next shard layout version. Technically speaking the current shard layout version should be used for the parent. However since ShardLayoutV2 the version is frozen so it is ok."

`ShardUId` is the canonical database key for trie state: `version (4 bytes) || shard_id (4 bytes)`. The version must match the layout in which the shard was created.

The "frozen" claim holds only within the `ShardLayoutV2` family, where the `version` field is not incremented on each derivation. However, `ShardLayoutV3` hard-codes `VERSION = 3`:

```rust
const VERSION: ShardVersion = 3;
impl ShardLayoutV3 {
    pub fn version(&self) -> ShardVersion { VERSION }
}
```

When dynamic resharding (`ProtocolFeature::DynamicResharding`) is first enabled and `next_next_shard_layout()` calls `next_shard_layout.derive_v3(...)` on a `ShardLayoutV2` base with `version = 2`, the returned layout is `ShardLayout::V3` with `version() = 3`. At the epoch boundary, `from_shard_layout` is called with this V3 layout. The parent shard lived in the V2 layout (version 2), so its correct UID is `ShardUId { version: 2, shard_id: X }`. The code instead produces `ShardUId { version: 3, shard_id: X }`.

In `split_shard`, the first thing done is:

```rust
if split_shard_event.parent_shard != shard_uid {
    // shard does not match event parent shard, skipping
    return Ok(Default::default());
}
```

The `shard_uid` argument is derived from the current epoch's shard layout (version 2), so it is `{ version: 2, shard_id: X }`. The event's `parent_shard` is `{ version: 3, shard_id: X }`. They do not match. The resharding is silently dropped on every node.

### Impact Explanation
The epoch info for epoch N+2 records the new `ShardLayoutV3` (the split has been committed to the epoch store), but the trie state is never actually partitioned. Every node skips the split identically, so there is no consensus divergence, but the protocol invariant that "epoch N+2's shard layout matches the actual trie partition" is broken. Nodes that subsequently try to serve or validate chunks for the new child shards will find no trie root for them, causing chunk production failures and potential chain stall. The impact is **Critical** because it silently corrupts the shard-layout-to-state correspondence at an epoch boundary, affecting all validators uniformly.

### Likelihood Explanation
The bug is triggered automatically at the first dynamic resharding event on any network whose shard layout at that moment is `ShardLayoutV2` with `version = 2` (Simple Nightshade V2, introduced at protocol version 57). If `ProtocolFeature::DynamicResharding` is activated before the network has migrated to Simple Nightshade V3 (`ShardLayoutV2` with `version = 3`, introduced at protocol version 135), the condition is met without any special action by any party. No privileged role is required; the epoch boundary processing is fully automatic.

### Recommendation
Pass the **current** (pre-split) shard layout into `from_shard_layout` alongside the next layout, and use it to construct the parent `ShardUId`:

```rust
pub fn from_shard_layout(
    current_shard_layout: &ShardLayout,   // <-- add this
    next_shard_layout: &ShardLayout,
    resharding_block: BlockInfo,
) -> Result<Option<ReshardingEventType>, Error> {
    ...
    // Use current layout version for the parent, which belongs to it.
    let parent_shard = ShardUId::new(current_shard_layout.version(), parent_id);
    ...
}
```

All call sites already have access to both layouts (current epoch and next epoch), so the change is mechanical. The comment justifying the wrong version should be removed.

### Proof of Concept
Concrete version values on mainnet:

| Epoch | Layout struct | `version()` |
|-------|--------------|-------------|
| N (current) | `ShardLayoutV2` | 2 |
| N+2 (derived by `derive_v3`) | `ShardLayoutV3` | 3 |

`from_shard_layout` is called with the N+2 layout. For parent shard id `X`:

```
event.parent_shard  = ShardUId { version: 3, shard_id: X }   // wrong
actual parent UID   = ShardUId { version: 2, shard_id: X }   // correct (from epoch N layout)
```

In `split_shard`:
```rust
split_shard_event.parent_shard  // { version: 3, shard_id: X }
!=
shard_uid                        // { version: 2, shard_id: X }  ← derived from current epoch layout
```
