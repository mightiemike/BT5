### Title
Unprivileged Caller Can Permanently Prune Canonical Chain State via `run_mainchain_gc` - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function that permanently deletes canonical block headers from contract storage. It carries only a `#[pause]` decorator, which gates access only when the contract is paused. When the contract is running normally, any unprivileged NEAR account can call it with an arbitrary `batch_size`, immediately pruning the mainchain down to `gc_threshold` blocks and permanently invalidating SPV proofs for all pruned blocks.

### Finding Description
The `Role` enum defines `UnrestrictedRunGC` as a role that "allows to use `run_mainchain_gc` API on a paused contract." [1](#0-0) 

The function itself is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`: [2](#0-1) 

The `#[pause]` macro from `near-plugins` only enforces access when the contract is in a paused state — it does not restrict callers when the contract is running normally. There is no `#[private]`, no `#[trusted_relayer]`, and no explicit role check guarding this function during normal operation. Any NEAR account can call it.

The function permanently removes entries from `mainchain_height_to_header` and `headers_pool`, and advances `mainchain_initial_blockhash`: [3](#0-2) 

By contrast, `submit_blocks` — the only other entry point that calls `run_mainchain_gc` — is protected by `#[trusted_relayer]`, which enforces that the caller is a registered, active relayer: [4](#0-3) 

### Impact Explanation
An attacker calls `run_mainchain_gc(u64::MAX)` directly. The function computes `total_amount_to_remove = amount_of_headers_we_store - gc_threshold` and removes that many blocks in one call, pruning the mainchain down to exactly `gc_threshold` blocks. The deleted block headers are permanently gone from `headers_pool` and `mainchain_height_to_header`.

Concrete consequences:
1. **SPV proof permanent invalidation**: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the block hash in `mainchain_header_to_height`. For any pruned block, this lookup panics with `"block does not belong to the current main chain"`. Downstream contracts relying on SPV proofs for those blocks are permanently broken. [5](#0-4) 

2. **Reorg blocking**: The `reorg_chain` function walks the fork chain back to the common ancestor by checking `mainchain_header_to_height`. If the common ancestor has been pruned, the walk panics with `"previous fork block should be there"`, permanently preventing a legitimate chain reorganization. [6](#0-5) 

3. **Canonical state corruption**: `mainchain_initial_blockhash` is permanently advanced, altering the contract's authoritative view of the Bitcoin canonical chain. [7](#0-6) 

### Likelihood Explanation
The entry point is a standard public NEAR contract method callable by any account with no deposit or staking requirement. The attacker needs only a NEAR account and gas. The attack is a single transaction. No privileged access, leaked keys, or social engineering is required.

### Recommendation
Add an explicit role check at the top of `run_mainchain_gc` to restrict direct external calls to authorized accounts (e.g., `Role::DAO` or a dedicated `GCManager` role), mirroring the pattern used for `submit_blocks`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO.into(), &env::predecessor_account_id())
            || self.acl_has_role(Role::UnrestrictedRunGC.into(), &env::predecessor_account_id()),
        "Unauthorized: caller lacks GC role"
    );
    // ...
}
```

Alternatively, mark the function `#[private]` and expose a separate role-gated wrapper, or restrict it to the contract account itself so only `submit_blocks` (via internal call) can trigger it.

### Proof of Concept
1. Deploy the contract and initialize it with a `gc_threshold` of, say, 10 and submit 100 blocks via a trusted relayer.
2. From any unprivileged NEAR account (no roles granted), call:
   ```
   run_mainchain_gc({"batch_size": 18446744073709551615})
   ```
3. Observe that `get_mainchain_size()` returns 10 — 90 blocks have been permanently deleted.
4. Attempt `verify_transaction_inclusion` for any transaction in one of the pruned blocks — the call panics with `"block does not belong to the current main chain"`.
5. The test at `contract/tests/test_basics.rs` line 515–521 demonstrates that `run_mainchain_gc` succeeds when called by `user_account` (a non-privileged account), confirming the missing access control. [8](#0-7)
