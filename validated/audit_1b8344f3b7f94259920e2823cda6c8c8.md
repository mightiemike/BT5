### Title
Unhandled Return Value of `approve` in `creditDeposit` Silently Breaks Collateral Deposits — (`File: core/contracts/DirectDepositV1.sol`)

### Summary
`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without checking the boolean return value. If a listed spot token's `approve` silently returns `false` instead of reverting, the subsequent `depositCollateralWithReferral` call proceeds with zero allowance, causing the endpoint to fail to pull funds. User tokens remain stranded in the DDA contract and the subaccount is never credited.

### Finding Description
In `DirectDepositV1.creditDeposit()`, for every spot product token held by the contract, the code calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
```

The return value of `approve` is never inspected. The `IIERC20Base` interface declares `approve` as returning `bool`, but the call site discards it entirely. [1](#0-0) 

The `ERC20Helper` library — which the project already uses for `safeTransfer` and `safeTransferFrom` — has no `safeApprove` equivalent, and `DirectDepositV1` defines its own `safeTransfer` helper but similarly omits a safe-approve wrapper. [2](#0-1) 

The same pattern appears in `ContractOwner.wrapVaultAsset()`, which calls `assetToken.approve(tokenAddr, 0)` and `assetToken.approve(tokenAddr, assetBalance)` on a raw `IERC20Base` without checking return values, before calling `IERC4626Base(tokenAddr).deposit(...)`. [3](#0-2) 

### Impact Explanation
If a spot collateral token's `approve` returns `false` (non-reverting failure — a known behavior of tokens such as USDT on certain chains, or any future-listed non-standard ERC20), the endpoint's `transferFrom` inside `depositCollateralWithReferral` will fail because no allowance was granted. The user's tokens remain locked in the `DirectDepositV1` contract with no automatic retry, and the subaccount balance is never updated. This constitutes a **collateral lock**: real user funds are permanently inaccessible through the normal deposit flow until a privileged `withdraw` call is made. [4](#0-3) 

### Likelihood Explanation
`creditDeposit()` carries no access modifier — it is callable by any unprivileged external address. [5](#0-4) 
The Nado protocol supports multiple spot products and can list new tokens over time. Any listed token that implements a non-reverting `approve` (returning `false` on failure) triggers this path. The likelihood is **medium**: standard OpenZeppelin ERC20 tokens revert, but non-standard tokens are routinely listed on DEX/perp protocols, and the protocol's multi-token loop increases exposure surface.

### Recommendation
Add a `safeApprove` helper to `ERC20Helper` (or `DirectDepositV1`) mirroring the existing `safeTransfer` pattern — using a low-level `call` and asserting `success && (data.length == 0 || abi.decode(data, (bool)))`. Replace the bare `token.approve(...)` calls in `creditDeposit()` and `ContractOwner.wrapVaultAsset()` with this safe wrapper. [6](#0-5) 

### Proof of Concept
1. A spot token `T` is listed in `SpotEngine` with a non-reverting `approve` that returns `false` when the spender already has a non-zero allowance (e.g., USDT-style).
2. A user sends `T` to their `DirectDepositV1` address.
3. Any caller (including the user or a bot) calls `DirectDepositV1.creditDeposit()`.
4. The loop reaches product `T`: `token.approve(address(endpoint), balance)` returns `false`; the return value is silently discarded.
5. `endpoint.depositCollateralWithReferral(...)` is called; the endpoint attempts `transferFrom(dda, clearinghouse, balance)` but the allowance is 0 (or unchanged from a prior partial state), causing the endpoint call to revert or transfer 0.
6. The subaccount receives no credit. The tokens remain in the DDA contract. The user has no self-service recovery path — only the owner can call `withdraw`. [1](#0-0)

```text
library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
