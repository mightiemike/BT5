Audit Report

## Title
Swap Allowlist Bypassed by Any User via `MetricOmmSimpleRouter` When Router Is Allowlisted - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to its own `msg.sender` — the router — when a swap is routed through `MetricOmmSimpleRouter`. If the pool admin allowlists the router so that legitimate users can swap through it, every user including non-allowlisted ones can bypass the guard by calling the same public router. The end user's identity is stored only in transient storage for payment callback purposes and is never forwarded to the extension.

## Finding Description
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender` against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← becomes `sender` in the extension
    ...
``` [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool. The actual end user (`msg.sender` of `exactInputSingle`) is stored only in transient storage via `_setNextCallbackContext` for payment purposes and is never forwarded to the pool or extension:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
``` [3](#0-2) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses (Alice, Bob).
2. Alice and Bob need to use the router, so admin calls `setAllowedToSwap(pool, router, true)`.
3. Charlie (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. `beforeSwap` receives `sender = router` → `allowedSwapper[pool][router] == true` → passes.
6. Charlie's swap executes despite never being allowlisted.

The wrong value is the `sender` argument checked by the extension: it is the router address rather than the originating trader address, causing `allowedSwapper[pool][router]` to be evaluated instead of `allowedSwapper[pool][charlie]`.

## Impact Explanation
`SwapAllowlistExtension` is the production mechanism for restricting pool access to specific counterparties (e.g., KYC-gated or institutional-only pools). When bypassed, any unprivileged user can execute swaps against a pool intended to be restricted. This constitutes broken core pool access-control functionality with direct fund-impacting consequences: LP assets are exposed to unrestricted trading, defeating the admin's access-control intent and potentially exposing LPs to adversarial flow they explicitly opted out of.

## Likelihood Explanation
The bypass requires the admin to allowlist the router — a natural and expected operational step if any legitimate user needs to swap through the router rather than calling the pool directly. The router is the standard user-facing entry point in the periphery. Any pool using `SwapAllowlistExtension` that also wants router access for its allowlisted users will inevitably trigger this condition. The attacker needs no special privileges, no malicious setup, and no non-standard tokens — only the ability to call a public router function.

## Recommendation
The extension must check the end user, not the immediate caller. The cleanest fix is to require the router to encode the originating user address in `extensionData` and have the extension decode and verify it. Alternatively, the extension can recognize known router addresses and read the actual trader from a standardized field in `extensionData`. A weaker mitigation is to document and enforce that the router must never be added to the swap allowlist (e.g., reject allowlisting of known router addresses in `setAllowedToSwap`), but this is fragile.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension as beforeSwap hook
  admin calls setAllowedToSwap(pool, alice, true)   // Alice is KYC'd
  admin calls setAllowedToSwap(pool, router, true)  // so Alice can use the router

Attack:
  charlie (not allowlisted) calls:
    router.exactInputSingle({
      pool: pool,
      recipient: charlie,
      zeroForOne: true,
      amountIn: X,
      ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(...) [msg.sender = router]
        → _beforeSwap(router, charlie, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true → PASSES
        → swap executes, charlie receives output tokens

Result: Charlie swaps successfully despite not being on the allowlist.
  Direct pool call by Charlie would revert (allowedSwapper[pool][charlie] == false).
```
