### Title
`ENABLE_SECP_OPS` Flag Check Missing on 4-Byte Secp Opcode Path, Bypassing Intended Gate - (File: `src/chia_dialect.rs`)

### Summary

The `ChiaDialect::op` function in `src/chia_dialect.rs` dispatches secp signature-verification operators via two distinct code paths: a 1-byte opcode path (opcodes 64 and 65) and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`). The 1-byte path correctly guards dispatch behind `ClvmFlags::ENABLE_SECP_OPS`, but the 4-byte path dispatches to the identical operator functions with no flag check at all. An attacker-controlled CLVM program can invoke `op_secp256k1_verify` or `op_secp256r1_verify` unconditionally by using the 4-byte opcode form, bypassing the flag gate entirely.

### Finding Description

In `src/chia_dialect.rs`, the `op` method of `ChiaDialect` handles 4-byte operators first:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

The 1-byte opcode path, reached only after the 4-byte branch returns, guards the same two functions behind `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The flag itself is documented as gating secp operations:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The 4-byte path also bypasses the `unknown_operator` check (which enforces `NO_UNKNOWN_OPS`), because the match arm for `0x13d61f00`/`0x1c3a8f00` returns before reaching `unknown_operator`. [4](#0-3) 

`MEMPOOL_MODE` includes `NO_UNKNOWN_OPS` but does **not** include `ENABLE_SECP_OPS`, meaning the mempool is supposed to reject secp operations that are not yet activated. The 4-byte form defeats this. [5](#0-4) 

### Impact Explanation

Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` (including `MEMPOOL_MODE`) intends to reject secp signature verification. A CLVM program using the 4-byte opcode `0x13d61f00` or `0x1c3a8f00` instead of the 1-byte opcode 64 or 65 will execute `op_secp256k1_verify` / `op_secp256r1_verify` successfully, returning `nil` on a valid signature. The flag gate is silently bypassed. This produces a **consensus/mempool divergence**: nodes enforcing the flag via 1-byte opcodes reject the operation, while the same node accepts it when the 4-byte form is used. The corrupted result is a successful `Reduction(cost, nil)` where an `EvalErr::Unimplemented` was expected.

### Likelihood Explanation

The entry path is fully attacker-controlled: the operator atom in a CLVM program is arbitrary bytes supplied by the submitter. Encoding `0x13d61f00` as a 4-byte atom is trivial. Any mempool or pre-activation consensus context that relies on `ENABLE_SECP_OPS` being absent to block secp operations is reachable.

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte opcode branch before dispatching to the secp functions:

```rust
0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If the 4-byte forms are intentionally always available (as legacy operators predating the flag), the `ENABLE_SECP_OPS` documentation and `MEMPOOL_MODE` definition must be updated to reflect that the flag only gates the 1-byte aliases, and callers relying on the flag to suppress secp execution must be warned that it is insufficient.

### Proof of Concept

1. Construct a `ChiaDialect` with `ClvmFlags::NO_UNKNOWN_OPS` (i.e., `MEMPOOL_MODE`) but without `ENABLE_SECP_OPS`.
2. Build a CLVM program whose operator atom is the 4-byte sequence `[0x13, 0xd6, 0x1f, 0x00]` with valid secp256k1 pubkey, message-hash, and signature arguments.
3. Call `run_program`. The 4-byte branch in `ChiaDialect::op` matches `0x13d61f00`, dispatches directly to `op_secp256k1_verify`, and returns `Reduction(1300000, nil)`.
4. Repeat with 1-byte opcode `64` and the same arguments: execution fails with `EvalErr::Unimplemented` because `ENABLE_SECP_OPS` is absent.
5. The two forms of the same operator produce different outcomes under identical flags — the flag gate is bypassed by the 4-byte form.
