### Title
`isValidDepositAmount` Applies `MIN_FIRST_DEPOSIT_AMOUNT` to Subsequent Deposits During Slow Mode Delay — (`File: core/contracts/Endpoint.sol`)

---

### Summary

The `isValidDepositAmount` function in `Endpoint.sol` determines whether to enforce `MIN_FIRST_DEPOSIT_AMOUNT` ($5) or `MIN_DEPOSIT_AMOUNT` ($0.1) based solely on whether `subaccountIds[subaccount] == 0`. However, `subaccountIds` is only updated when a slow mode `DepositCollateral` transaction is **processed** — up to 3 days after it is queued. During this window, any subsequent deposit by the same user is incorrectly treated as a "first deposit" and must independently meet the $5 minimum, even though the user has already committed $5 in the pending slow mode queue.

---

### Finding Description

In `Endpoint.sol`, `isValidDepositAmount` selects the minimum threshold as follows:

```solidity
function isValidDepositAmount(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount
) internal returns (bool) {
    int256 minDepositAmount = MIN_DEPOSIT_AMOUNT;
    if (subaccount != X_ACCOUNT && (subaccountIds[subaccount] == 0)) {
        minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT;
    }
    return clearinghouse.checkMinDeposit(productId, amount, minDepositAmount);
}
``` [1](#0-0) 

`subaccountIds[subaccount]` is set to a non-zero value only inside `_recordSubaccount`:

```solidity
function _recordSubaccount(bytes32 subaccount) internal {
    if (subaccountIds[subaccount] == 0) {
        subaccountIds[subaccount] = ++numSubaccounts;
        subaccounts[numSubaccounts] = subaccount;
    }
}
``` [2](#0-1) 

`_recordSubaccount` is called only inside `processSlowModeTransactionImpl`, when the `DepositCollateral` slow mode transaction is **executed**:

```solidity
if (txType == IEndpoint.TransactionType.DepositCollateral) {
    ...
    _recordSubaccount(txn.sender);
    clearinghouse.depositCollateral(txn);
}
``` [3](#0-2) 

Slow mode transactions carry a hardcoded 3-day execution delay:

```solidity
executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
``` [4](#0-3) 

Where `SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60` (3 days). [5](#0-4) 

This means `subaccountIds[subaccount]` remains `0` for the entire 3-day window after the first deposit is queued. Any subsequent call to `depositCollateral` during this window will still see `subaccountIds == 0` and apply `MIN_FIRST_DEPOSIT_AMOUNT` ($5) to the new `amount` alone — without accounting for the already-committed pending deposit.

The check in `depositCollateral` reverts hard on failure:

```solidity
require(
    isValidDepositAmount(subaccount, productId, amount),
    ERR_DEPOSIT_TOO_SMALL
);
``` [6](#0-5) 

And `depositCollateralWithReferral` (used by `DirectDepositV1.creditDeposit`) silently skips the deposit:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    return;
}
``` [7](#0-6) 

This means a `DirectDepositV1` credit attempt for a new subaccount that has already queued a first deposit will silently drop any subsequent deposit below $5, leaving funds stranded in the `DirectDepositV1` contract. [8](#0-7) 

---

### Impact Explanation

A user who has already committed $5 in a pending slow mode deposit cannot make any subsequent deposit below $5 for the entire 3-day slow mode delay window. Their `depositCollateral` call reverts with `ERR_DEPOSIT_TOO_SMALL`. For the `DirectDepositV1` path, sub-$5 deposits are silently skipped and the tokens remain stranded in the `DirectDepositV1` contract until the owner manually withdraws them. The user is forced to deposit at least $5 again even though their total committed collateral already meets or exceeds the minimum.

---

### Likelihood Explanation

Medium. The 3-day slow mode delay is a protocol-level constant that applies to every first deposit. Any user who makes a second deposit within 3 days of their first — a common pattern for active traders topping up collateral — will hit this check. The `DirectDepositV1` path is particularly affected because it is the primary deposit mechanism for users who send tokens directly to their deposit address.

---

### Recommendation

Call `_recordSubaccount` at the time the deposit is **queued** (inside `depositCollateral` / `depositCollateralWithReferral`), not only when the slow mode transaction is processed. This mirrors the fix applied in the DegenLockToken report: account for already-committed state when evaluating the minimum threshold.

Alternatively, modify `isValidDepositAmount` to treat a subaccount as "existing" if it has any pending slow mode `DepositCollateral` transactions in the queue.

---

### Proof of Concept

1. Alice calls `Endpoint.depositCollateral("default", USDC_ID, 5e6)` — $5 USDC, meeting `MIN_FIRST_DEPOSIT_AMOUNT`.
2. `isValidDepositAmount`: `subaccountIds[alice_default] == 0` → `minDepositAmount = $5`. Check passes. Slow mode transaction queued with `executableAt = now + 3 days`.
3. One day later, Alice calls `Endpoint.depositCollateral("default", USDC_ID, 1e6)` — $1 USDC.
4. `isValidDepositAmount`: `subaccountIds[alice_default]` is still `0` (slow mode not yet processed). `minDepositAmount = $5`. `checkMinDeposit` returns `false` ($1 < $5).
5. `depositCollateral` reverts: `ERR_DEPOSIT_TOO_SMALL`.
6. Alice is forced to deposit $5 again, even though she has already committed $5 in the pending queue. Her total committed collateral ($5 pending + $1 new = $6) would have exceeded the minimum, but the check only evaluates the new `amount` in isolation.
