### Title
Mandatory Bonus-Reward Settlement Before Every Exit Path Permanently Locks Tier Positions When Pool Is Depleted — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

The `x/tieredrewards` module forces a full reward settlement — including bonus rewards drawn from the `RewardsPoolName` module account — as a prerequisite for every position-mutation message. When the bonus pool balance is insufficient to cover accrued bonus, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, causing `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgExitTierWithDelegation`, and `MsgClearPosition` to all fail atomically. Because every exit path requires this settlement to succeed first, a user with a delegated position has **no way to exit** until the pool is externally replenished.

---

### Finding Description

Every position-mutation handler in `msg_server.go` calls `claimRewards()` before performing its primary operation:

- `TierUndelegate` — line 166: `pos, _, _, err = ms.claimRewards(ctx, pos)`
- `TierRedelegate` — line 229: `pos, _, _, err = ms.claimRewards(ctx, pos)`
- `AddToTierPosition` — line 314: `pos, _, _, err = ms.claimRewards(ctx, pos)`
- `ClearPosition` — line 406: `pos, _, _, err = ms.claimRewards(ctx, pos)`
- `ExitTierWithDelegation` — line 532: `pos, _, _, err = ms.claimRewards(ctx, pos)` [1](#0-0) [2](#0-1) 

`claimRewards()` calls `processEventsAndClaimBonus()`, which calls `sufficientBonusPoolBalance()`. If the pool balance is less than the accrued bonus, it returns `ErrInsufficientBonusPool` and the entire transaction is rolled back: [3](#0-2) [4](#0-3) [5](#0-4) 

The only messages that do **not** require reward settlement are `MsgTriggerExitFromTier` and `MsgWithdrawFromTier`. However, `MsgWithdrawFromTier` requires the position to already be undelegated (validated by `validateWithdrawFromTier`), which requires `MsgTierUndelegate` to have succeeded first. Since `MsgTierUndelegate` itself calls `claimRewards()`, the user is blocked at the very first step of the exit path. [6](#0-5) 

The `ForceFullExitWithDelegation` governance path also calls `claimRewards()` and would fail identically: [7](#0-6) 

---

### Impact Explanation

**Impact: High.** A user who has locked tokens in a tier position and triggered exit has **no reachable exit path** when the bonus pool is empty:

- `MsgTierUndelegate` → fails (`ErrInsufficientBonusPool`)
- `MsgExitTierWithDelegation` → fails (`ErrInsufficientBonusPool`)
- `MsgTierRedelegate` → fails (`ErrInsufficientBonusPool`)
- `MsgClearPosition` → fails (`ErrInsufficientBonusPool`)
- `MsgWithdrawFromTier` → blocked (requires prior undelegation)

The user's principal (staked tokens) is locked in the module account indefinitely. The position continues to exist and accrue bonus debt, making the pool deficit grow over time, further delaying replenishment. The corrupted invariant is: **a user who has satisfied all exit preconditions (exit triggered, exit duration elapsed) must always be able to undelegate or exit with delegation**. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Low-Medium.** The bonus pool is funded by external deposits (governance or any account sending tokens to `RewardsPoolName`). It is not automatically replenished. As positions accrue bonus over time and users claim rewards, the pool balance decreases monotonically until it is topped up. A pool depletion event is a realistic steady-state condition, not a theoretical edge case. The ADR itself acknowledges this scenario:

> *Pool empty (user-driven) | Message fails atomically. No state change. Retry after pool replenished.* [9](#0-8) 

Additionally, a large staker can accelerate pool depletion by claiming a disproportionate share of bonus rewards, leaving other users unable to exit.

---

### Recommendation

Decouple reward settlement from exit operations. Specifically:

1. **For `MsgTierUndelegate` and `MsgExitTierWithDelegation`**: Do not require bonus reward payment to succeed before allowing the exit. Instead, record the accrued bonus as a pending claimable balance (a `pendingBonus` mapping keyed by position or owner) and advance the position checkpoints regardless of pool balance. Allow the user to claim the pending bonus separately once the pool is replenished.

2. **For `MsgClearPosition`**: Similarly, settle bonus into a pending balance rather than requiring an immediate transfer.

3. Alternatively, introduce a `MsgForceExitSkipBonus` path that forfeits unclaimed bonus (similar to how `BeforeRedelegationSlashed` already silently forfeits bonus to avoid chain halt) so that a user can always exit at the cost of forgoing accrued-but-unpayable bonus. [10](#0-9) 

---

### Proof of Concept

1. User calls `MsgLockTier` and locks tokens into a tier position.
2. Time passes; bonus accrues. The bonus pool is depleted (either naturally or by other users claiming rewards).
3. User calls `MsgTriggerExitFromTier` — succeeds (no pool check).
4. User waits for `ExitUnlockAt` to elapse.
5. User calls `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated).
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
   - Transaction reverts. Position state unchanged.
6. User calls `MsgExitTierWithDelegation`:
   - Same failure path at `claimRewards`.
7. User calls `MsgClearPosition` (to cancel exit and try again later):
   - Same failure path at `claimRewards`.
8. User is permanently blocked from all exit paths. Tokens remain locked in the tier module account until an external party replenishes the pool. [11](#0-10) [3](#0-2) [4](#0-3)

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil
}

func (ms msgServer) WithdrawFromTier(ctx context.Context, msg *types.MsgWithdrawFromTier) (*types.MsgWithdrawFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateWithdrawFromTier(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
