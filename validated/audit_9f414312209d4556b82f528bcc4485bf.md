The vulnerability is real. Here is the analysis:

### Title
Missing AuxPoW Flag Enforcement Allows AuxPoW-Flagged Block to Bypass All AuxPoW Validation — (`contract/src/dogecoin.rs`)

---

### Summary

`submit_block_header` in the Dogecoin build dispatches PoW validation based solely on whether `aux_data` is `Some` or `None`. There is no guard that rejects a block whose `version & 0x100 != 0` (AuxPoW flag set) when `aux_data` is `None`. A trusted relayer can therefore submit a header with the AuxPoW version bit set and no `AuxData`, causing the contract to skip every AuxPoW check and fall through to a bare scrypt PoW comparison against the target.

---

### Finding Description

`BlockHeader` in the Dogecoin feature is the type alias `(Header, Option<AuxData>)`. [1](#0-0) 

`submit_blocks` accepts `Vec<BlockHeader>`, so any caller can supply `None` for the `AuxData` field of any header. [2](#0-1) 

Inside `submit_block_header`, the PoW branch is chosen purely by the `Option`:

```
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);   // full AuxPoW validation
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        ...
    );
}
``` [3](#0-2) 

The `check_aux` function does verify that the version flag is set, but that check is only reachable when `aux_data` is already `Some`: [4](#0-3) 

There is no symmetric guard in the `else` branch — no `require!(block_header.version & BLOCK_VERSION_AUXPOW == 0, ...)` — so a header with `version | 0x100` and `AuxData = None` silently falls through to the scrypt hash check. All of the following AuxPoW invariants are skipped:

- Chain-ID check on the submitted block (`block_header.get_chain_id() == chain_id`) [5](#0-4) 
- Parent-block chain-ID exclusion (`chain_id != aux_data.parent_block.get_chain_id()`) [6](#0-5) 
- Coinbase-to-merkle-root commitment [7](#0-6) 
- Chain-merkle-root presence in the parent coinbase [8](#0-7) 
- Expected-index derivation from nonce and chain ID [9](#0-8) 
- Parent-block scrypt PoW against the Dogecoin target [10](#0-9) 

---

### Impact Explanation

The light client stores the submitted header as a valid canonical Dogecoin block. Any downstream consumer — including `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` — will treat that header's `merkle_root` as authoritative. A block that the real Dogecoin network would unconditionally reject (AuxPoW flag set, no AuxPoW data) becomes an accepted anchor for transaction-inclusion proofs, breaking the core security invariant of the light client. [11](#0-10) 

---

### Likelihood Explanation

The entry point is `submit_blocks`, which is gated by `#[trusted_relayer]`. A registered trusted relayer is the required actor. The relayer controls the serialized `Vec<BlockHeader>` payload and can freely set `AuxData = None` for any header. The only remaining barrier is producing a scrypt hash of the 80-byte header that satisfies `hash <= target_from_bits(bits)`. This is equivalent to solo-mining a Dogecoin block at the current network difficulty — non-trivial but well within reach for an adversary with dedicated scrypt hardware, and far cheaper than constructing a cryptographically valid AuxPoW chain (which additionally requires a cooperating Bitcoin miner). The invariant violation is therefore reachable by any sufficiently motivated trusted relayer.

---

### Recommendation

Add an explicit guard in `submit_block_header` that rejects any header whose version carries the AuxPoW flag when no `AuxData` is provided:

```rust
if !skip_pow_verification {
    self.check_target(&block_header, &prev_block_header);

    const BLOCK_VERSION_AUXPOW: i32 = 0x100;
    match aux_data {
        Some(ref aux) => {
            self.check_aux(&block_header, aux);
        }
        None => {
            // Reject headers that claim to be AuxPoW blocks but carry no proof.
            require!(
                block_header.version & BLOCK_VERSION_AUXPOW == 0,
                "AuxPoW version flag set but no AuxData provided"
            );
            let pow_hash = block_header.block_hash_pow();
            require!(
                U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                "block should have correct pow"
            );
        }
    }
}
``` [3](#0-2) 

---

### Proof of Concept

1. Register as a trusted relayer (or use an account with `Role::UnrestrictedSubmitBlocks`).
2. Construct an 80-byte Dogecoin header with `version = (any_valid_version | 0x100)` and valid `prev_block_hash`, `bits`, and `time` fields.
3. Mine the header's scrypt hash until `scrypt(header) <= target_from_bits(bits)`.
4. Call `submit_blocks` with `headers = [( header, None )]`.
5. Observe that the contract accepts the block without calling `check_aux`, stores it in `headers_pool`, and potentially promotes it to the canonical chain tip — despite the block being unconditionally invalid on the real Dogecoin network. [12](#0-11)

    pub(crate) fn submit_block_header(
        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
