### Title
Zero-Amount `safeTransferFrom` in `submitFastWithdrawal` Blocks Fast Withdrawal Path for Zero-Fee Tokens â€” (File: `core/contracts/BaseWithdrawPool.sol`)

### Summary

`BaseWithdrawPool.submitFastWithdrawal` calls `safeTransferFrom(token, msg.sender, uint128(fee))` without checking whether `fee > 0`. When the computed fee rounds down to zero â€” which is possible for small withdrawal amounts on tokens with `withdrawFeeX18 == 0` â€” any ERC20 token that reverts on zero-value transfers will cause every `submitFastWithdrawal` call in the `sendTo != msg.sender` branch to revert, permanently blocking the fast withdrawal path for that product.

### Finding Description

`submitFastWithdrawal` handles two cases based on whether the caller (`msg.sender`) is also the recipient (`sendTo`): [1](#0-0) 

When `sendTo != msg.sender` (a third-party fast-withdrawal provider paying the fee on behalf of the user), the fee is pulled from `msg.sender` via `safeTransferFrom`. The fee is computed by `fastWithdrawalFeeAmount`: [2](#0-1) 

The final return value is `feeX18 / int128(multiplier)`. For a 6-decimal token, `multiplier = 10^12`. If `withdrawFeeX18 == 0` and the withdrawal amount is small (e.g., 1 wei), `feeX18` can be less than `multiplier`, causing integer division to yield `fee = 0`. The call then becomes:

```solidity
safeTransferFrom(token, msg.sender, uint128(0));
```

`ERC20Helper.safeTransferFrom` propagates this zero-value call directly to the token contract: [3](#0-2) 

For tokens that revert on zero-value transfers (a well-documented ERC20 variant), this causes the entire `submitFastWithdrawal` transaction to revert. There is no zero-amount guard anywhere in this path.

### Impact Explanation

The fast withdrawal path (`submitFastWithdrawal`) is permanently unavailable for the `sendTo != msg.sender` branch when the computed fee is zero and the collateral token reverts on zero transfers. Every attempt by a third-party provider to process such a withdrawal will revert. Users are forced to fall back to the standard sequencer-driven withdrawal path, which carries a settlement delay that the fast withdrawal system is specifically designed to bypass. [4](#0-3) 

### Likelihood Explanation

The condition is reachable without any privileged access. `submitFastWithdrawal` is a public function. A product with `withdrawFeeX18 == 0` is a valid configuration (the field is set per-product in `SpotEngine.addOrUpdateProduct`). For 6-decimal tokens, any withdrawal amount below approximately 1000 wei will produce a zero fee via integer truncation. The combination of a zero-fee product config and a token that reverts on zero transfers is realistic and requires no malicious intent. [5](#0-4) 

### Recommendation

Add a zero-amount guard before the `safeTransferFrom` call in `submitFastWithdrawal`:

```solidity
} else {
    if (uint128(fee) > 0) {
        safeTransferFrom(token, msg.sender, uint128(fee));
    }
}
```

This mirrors the recommended fix in the original report: skip the transfer entirely when the amount is zero, rather than attempting it unconditionally.

### Proof of Concept

1. Deploy a collateral token that reverts on zero-value transfers.
2. Register a product with `withdrawFeeX18 == 0`.
3. A user signs a `WithdrawCollateral` transaction for a small amount (e.g., 1 wei).
4. A third-party provider calls `submitFastWithdrawal` with `sendTo` set to the user's address (not `msg.sender`).
5. `fastWithdrawalFeeAmount` returns `0` due to integer truncation.
6. `safeTransferFrom(token, msg.sender, 0)` is called; the token reverts.
7. The fast withdrawal fails. Every subsequent attempt with the same parameters also fails. The user must wait for the sequencer-driven settlement path. [6](#0-5)

```text
    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
