Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual user, allowing allowlist bypass via `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` inside the pool, so the extension checks the router's address — not the actual user's. A pool admin who allowlists the router to enable router-mediated swaps for their curated pool inadvertently opens the pool to every user of the public router, completely bypassing per-user access control.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension hook:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. Whoever calls `pool.swap()` becomes the `sender` argument forwarded to every configured extension.

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

The value is passed verbatim through `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` to every extension in `BEFORE_SWAP_ORDER`.

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`:**

At line 37, `msg.sender` is the pool (the extension's caller) and `sender` is whoever called `pool.swap()`. The check is: is this specific address allowlisted for this pool?

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly:**

At lines 72–80, the router calls `IMetricOmmPoolActions(params.pool).swap(...)` with no encoding of the original `msg.sender` into the call. The router stores the original caller only in transient storage for the payment callback (`_setNextCallbackContext`), which is never visible to the pool or extensions.

**Step 5 — The same pattern holds for `exactInput` (all hops) and `exactOutputSingle`/`exactOutput`:**

Every pool call in the router is a direct `pool.swap()` call where the router is `msg.sender`. The original user's identity is never forwarded.

**Result:** When a pool admin allowlists the router address (a natural step to enable router-mediated swaps for their approved users), `allowedSwapper[pool][router] == true`. Any unprivileged user calling `router.exactInputSingle()` targeting the restricted pool will pass the allowlist check because the extension sees `sender = router`, which is allowlisted. The per-user allowlist is completely bypassed.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to specific addresses. Once the router is allowlisted, any EOA can swap in the restricted pool by routing through the public router. Unauthorized traders can drain the pool's liquidity at oracle-quoted prices, directly harming LP principal in a pool that was configured to restrict access. This constitutes a direct loss of user principal and broken core pool functionality (access control), meeting Sherlock High/Critical thresholds.

## Likelihood Explanation

The `SwapAllowlistExtension` is documented as "Gates `swap` by swapper address, per pool." A pool admin who reads this and wants to support router-mediated swaps for their allowlisted users will naturally allowlist the router address, believing the extension will still gate by individual user identity. The router is a canonical, publicly deployed periphery contract. Allowlisting it is a foreseeable and reasonable configuration step. Once the router is allowlisted, the bypass requires no special privilege — any EOA can call the router with no preconditions.

## Recommendation

1. **Extension-side fix:** `SwapAllowlistExtension.beforeSwap` should decode the actual initiator from `extensionData` (supplied by the router) and verify it against the allowlist, rather than relying solely on the `sender` argument.

2. **Router-side fix:** `MetricOmmSimpleRouter` should encode the original `msg.sender` into the `extensionData` it forwards to the pool for every `pool.swap()` call, so allowlist extensions can extract and verify the real user. Currently `params.extensionData` is passed through unchanged with no user identity appended.

Until fixed, pool admins must not allowlist the router address on pools that intend per-user access control.

## Proof of Concept

```solidity
// Pool admin sets up a curated pool:
//   extension = SwapAllowlistExtension
//   allowedSwapper[pool][router] = true   ← admin allowlists router to support router swaps
//   allowedSwapper[pool][alice]  = false  ← alice is NOT individually allowlisted

// Alice (not allowlisted) bypasses the guard:
vm.prank(alice);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(restrictedPool),
        recipient:        alice,
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:    0,
        deadline:         block.timestamp,
        tokenIn:          token0,
        extensionData:    ""
    })
);
// MetricOmmPool.swap: _beforeSwap(msg.sender=router, ...)   [MetricOmmPool.sol L231]
// SwapAllowlistExtension.beforeSwap: sender = router        [SwapAllowlistExtension.sol L37]
// allowedSwapper[pool][router] == true  → no revert
// Alice swaps successfully despite not being individually allowlisted
```
