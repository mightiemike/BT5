### Title
Division-by-Zero in `socializeSubaccount` When `openInterest` Is Zero Permanently Blocks Liquidation Finalization — (File: `core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when spreading residual bad debt across market participants. If `openInterest` reaches zero for a product while a subaccount still carries a negative `vQuoteBalance`, the call reverts with a division-by-zero, permanently blocking the finalization step of any liquidation that reaches socialization for that product.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, when insurance is insufficient to cover a subaccount's negative `vQuoteBalance`, the protocol attempts to spread the loss across all open-interest holders:

```solidity
// PerpEngine.sol line 166
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest   // ← no zero-check
);
``` [1](#0-0) 

`state.openInterest` is the sum of absolute position sizes across all subaccounts for a given product. It is decremented in `_updateBalance` whenever a position is reduced or closed:

```solidity
// PerpEngineState.sol line 30
state.openInterest -= balance.amount.abs();
``` [2](#0-1) 

A subaccount can have `balance.amount == 0` (position fully closed) while retaining a negative `vQuoteBalance` from accumulated funding payments or realised losses. This is precisely the state that `_finalizeSubaccount` in `ClearinghouseLiq` enforces before calling `socializeSubaccount` — it requires every perp `balance.amount == 0` as a precondition:

```solidity
// ClearinghouseLiq.sol line 313-320
for (uint32 i = 0; i < v.perpIds.length; ++i) {
    uint32 perpId = v.perpIds[i];
    IPerpEngine.Balance memory balance = perpEngine.getBalance(perpId, txn.liquidatee);
    require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
}
``` [3](#0-2) 

After that check passes, `_finalizeSubaccount` calls:

```solidity
// ClearinghouseLiq.sol line 386-389
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    v.insurance
);
``` [4](#0-3) 

If, at the moment of finalization, every other participant in the product has also closed their position (making `openInterest == 0`), the division on line 166 of `PerpEngine.sol` reverts. There is no guard, no fallback, and no alternative code path.

---

### Impact Explanation

The revert occurs inside `liquidateSubaccountImpl` → `_finalizeSubaccount` → `perpEngine.socializeSubaccount`. Because the entire call is atomic, the finalization transaction reverts and cannot be retried with different parameters — the precondition (`balance.amount == 0`) is already satisfied, so the only remaining step is socialization, which is permanently broken for that product. The insolvent subaccount cannot be cleared, bad debt is never written off, and the insurance accounting (`insurance` state variable in `ClearinghouseStorage`) is never updated. This is a direct solvency/accounting corruption: unresolvable bad debt accumulates in the protocol with no recovery path.

---

### Likelihood Explanation

The condition `openInterest == 0` for a product where a subaccount has negative `vQuoteBalance` is reachable through normal, unprivileged trading activity:

1. Subaccount A opens a long position in product P; subaccount B opens the matching short. `openInterest = 2X`.
2. Funding payments accumulate, making A's `vQuoteBalance` negative.
3. B closes their position. `openInterest = X`.
4. A's position is liquidated in a prior call (transferred to liquidator C). `openInterest = X` (C holds it).
5. C closes their position through normal trading. `openInterest = 0`.
6. A still has `amount == 0` and `vQuoteBalance < 0`.
7. Any liquidator calls `liquidateSubaccountImpl` with `productId = type(uint32).max` to finalize A.
8. `_finalizeSubaccount` passes the `amount == 0` check, then calls `socializeSubaccount`.
9. Insurance is insufficient; the code reaches `state.openInterest.div(...)` → **division by zero → permanent revert**.

Steps 3–5 are entirely within the control of unprivileged traders. No admin action, governance capture, or key compromise is required. The scenario is especially likely in low-liquidity markets, newly listed products, or products approaching delisting.

---

### Recommendation

Before dividing by `state.openInterest`, add an explicit guard:

```solidity
if (state.openInterest == 0) {
    // No participants remain to absorb the loss.
    // Absorb entirely from insurance or write off.
    insurance = MathHelper.max(0, insurance + balance.vQuoteBalance);
    balance.vQuoteBalance = 0;
} else {
    int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
    state.cumulativeFundingLongX18 += fundingPerShare;
    state.cumulativeFundingShortX18 -= fundingPerShare;
    balance.vQuoteBalance = 0;
}
```

This mirrors the pattern already used for the insurance-cover branch immediately above it and ensures the socialization path never divides by zero.

---

### Proof of Concept

```
1. Deploy protocol; register perp product P.
2. Subaccount A: openLong(P, X)   → openInterest = X
3. Subaccount B: openShort(P, X)  → openInterest = 2X
4. Advance time; funding payments make A.vQuoteBalance < 0.
5. B: closeShort(P, X)            → openInterest = X
6. Liquidator C: liquidateSubaccountImpl(liquidatee=A, productId=P, amount=X)
   → A.amount = 0, C.amount = X  → openInterest = X
7. C: closeLong(P, X)             → openInterest = 0
8. Assert: A.amount == 0, A.vQuoteBalance < 0, openInterest[P] == 0
9. Liquidator D: liquidateSubaccountImpl(liquidatee=A, productId=type(uint32).max)
   → _finalizeSubaccount passes amount==0 check
   → perpEngine.socializeSubaccount(A, insurance) called
   → balance.vQuoteBalance < 0, insurance insufficient
   → fundingPerShare = -balance.vQuoteBalance.div(0)  ← REVERT (division by zero)
10. Liquidation is permanently blocked; bad debt is unresolvable.
```
