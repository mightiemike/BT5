### Title
Fast Withdrawal Fee Pulled from Caller Instead of Recipient When `sendTo != msg.sender` - (File: `core/contracts/BaseWithdrawPool.sol`)

### Summary

In `BaseWithdrawPool.submitFastWithdrawal()`, when the resolved `sendTo` address differs from `msg.sender`, the fast-withdrawal fee is pulled via `safeTransferFrom` from `msg.sender` rather than being deducted from the amount sent to `sendTo`. This is a direct analog to M-16: the fee is charged to the wrong party.

### Finding Description

`submitFastWithdrawal` resolves the recipient address (`sendTo`) from the signed withdrawal transaction. For `WithdrawCollateral` (V1), `sendTo` is always derived from the signer's `bytes32 sender` field. For `WithdrawCollateralV2`, `sendTo` can be an explicitly set third-party address: [1](#0-0) 

After resolving `sendTo`, the fee logic branches on whether `sendTo == msg.sender`: [2](#0-1) 

- **`sendTo == msg.sender`**: fee is deducted from `transferAmount` — the recipient bears the cost correctly.
- **`sendTo != msg.sender`**: fee is pulled from `msg.sender` via `safeTransferFrom`, while `sendTo` receives the full `transferAmount` untouched. [3](#0-2) 

The caller (`msg.sender`) pays the fee but receives nothing. The recipient (`sendTo`) gets the full amount for free.

### Impact Explanation

Any address that calls `submitFastWithdrawal` for a `WithdrawCollateralV2` transaction where `sendTo != msg.sender` will have the fee token pulled from their own balance (provided they have a prior approval to the pool). The recipient receives the full withdrawal amount without bearing any fee cost. This is an unexpected, direct loss of funds for the caller — identical in class to M-16.

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function callable by any address. `WithdrawCollateralV2` explicitly supports a custom `sendTo` field, making `sendTo != msg.sender` a supported and realistic scenario. Any caller who has previously approved the pool (e.g., for their own deposits or withdrawals) and submits such a transaction is silently drained of the fee amount.

### Recommendation

Remove the `sendTo == msg.sender` branch entirely. Always deduct the fee from `transferAmount` so the recipient bears the cost regardless of who submits the transaction:

```solidity
require(transferAmount > uint128(fee), "Fee larger than balance");
transferAmount -= uint128(fee);
fees[productId] += fee;
handleWithdrawTransfer(token, sendTo, transferAmount);
```

This ensures the fee is always borne by the beneficiary of the withdrawal, not the submitter.

### Proof of Concept

1. Alice signs a `WithdrawCollateralV2` transaction with `sendTo = Bob` and `amount = 1000 USDC`.
2. Charlie (who has previously approved the pool to spend his USDC) calls `submitFastWithdrawal` with Alice's signed transaction.
3. `resolveFastWithdrawal` returns `sendTo = Bob`.
4. Since `Bob != Charlie` (i.e., `sendTo != msg.sender`), the branch at line 108 executes: `safeTransferFrom(token, Charlie, fee)`.
5. Charlie loses `fee` USDC from his own balance.
6. Bob receives the full `1000 USDC` — paying nothing.
7. Charlie has no recourse; the fee is non-refundable and credited to `fees[productId]`. [4](#0-3)

```text
    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
```
