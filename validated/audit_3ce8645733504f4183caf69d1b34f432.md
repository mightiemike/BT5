### Title
Withdrawal Permanently Blocked by Paused or Blacklisted Collateral Token, Trapping User Funds in Protocol — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.handleWithdrawTransfer` calls `token.safeTransfer(to, amount)` directly on the collateral ERC-20 token with no fallback path. If the token is paused (e.g., USDC emergency pause) or the recipient address is blacklisted (e.g., USDC/USDT blacklist), the call reverts. Because `submitWithdrawal` is the only mechanism by which the Clearinghouse settles queued withdrawals, and there is no skip or rescue path, the affected user's funds are permanently unwithdrawable from the protocol.

---

### Finding Description

The withdrawal settlement path in Nado is:

```
Endpoint → Clearinghouse.withdrawCollateral
         → Clearinghouse.handleWithdrawTransfer
             → token.safeTransfer(withdrawPool, amount)          [step A]
             → BaseWithdrawPool.submitWithdrawal(token, to, amount, idx)
                 → BaseWithdrawPool.handleWithdrawTransfer
                     → token.safeTransfer(to, amount)            [step B]
```

`Clearinghouse.handleWithdrawTransfer` first pushes tokens to the `WithdrawPool` (step A), then calls `submitWithdrawal` which pushes them onward to the user (step B). [1](#0-0) 

`BaseWithdrawPool.handleWithdrawTransfer` performs the final user-facing transfer with a bare `safeTransfer`: [2](#0-1) 

`ERC20Helper.safeTransfer` propagates any revert from the token contract directly: [3](#0-2) 

If the collateral token (e.g., USDC) is paused, or if `sendTo` is on the token's blacklist, step B reverts. Because both steps A and B are in the same transaction, the EVM unwinds everything — but the withdrawal **cannot be retried with a different path**. `submitWithdrawal` has no alternative branch, no try/catch, and no admin rescue function. The user's on-chain balance in `SpotEngine` remains credited, but the actual ERC-20 tokens can never be extracted. [4](#0-3) 

The same revert propagates up through `Clearinghouse.withdrawCollateral`, which is called by the Endpoint sequencer. Because the Endpoint enforces strict sequential submission indices (`idx == nSubmissions`), a permanently failing withdrawal transaction stalls the entire sequencer queue, blocking all subsequent protocol operations for all users. [5](#0-4) 

---

### Impact Explanation

- **Affected user**: Any user whose withdrawal destination address is blacklisted by the collateral token contract can never receive their funds. Their `SpotEngine` balance is credited but the underlying tokens are unwithdrawable.
- **Protocol-wide**: If the collateral token is paused, **all** pending withdrawals revert. The sequencer queue stalls at the first failing withdrawal index, freezing the entire protocol — no trades, no deposits, no liquidations can be processed until the token is unpaused.
- **No recovery path**: `BaseWithdrawPool` has no admin function to skip a failed withdrawal, redirect funds to an alternate address, or mark an index as resolved without transferring. `removeLiquidity` (owner-only) also calls `handleWithdrawTransfer` and would fail identically. [6](#0-5) 

---

### Likelihood Explanation

USDC is the canonical quote asset in Nado (configured as `QUOTE_PRODUCT_ID`). USDC has both a global `pause` and a per-address `blacklist` enforced at the transfer level. Circle has historically blacklisted addresses in response to regulatory orders. A user depositing USDC who is subsequently blacklisted by Circle — or a protocol-wide USDC pause during a market event — directly triggers this path with no admin override available inside Nado's contracts.

---

### Recommendation

1. **Decouple settlement from transfer**: In `BaseWithdrawPool.submitWithdrawal`, wrap `handleWithdrawTransfer` in a try/catch (or low-level call). On failure, record the failed withdrawal in a mapping (e.g., `failedWithdrawals[idx]`) so the sequencer can advance `minIdx` and continue processing.
2. **Add a rescue function**: Allow the owner or the user to redirect a stuck withdrawal to an alternate address, or to claim the tokens via a pull-based pattern.
3. **Separate `minIdx` advancement from transfer success**: `minIdx = idx` should be committed even when the token transfer fails, so the sequencer queue is never permanently blocked by a single bad withdrawal.

---

### Proof of Concept

1. User deposits USDC collateral via `Endpoint.depositCollateral`.
2. User submits a `WithdrawCollateral` transaction; the sequencer queues it.
3. Circle blacklists the user's address (or pauses USDC globally).
4. Sequencer calls `Endpoint.submitTransactions` → `Clearinghouse.withdrawCollateral` → `Clearinghouse.handleWithdrawTransfer` → `BaseWithdrawPool.submitWithdrawal` → `BaseWithdrawPool.handleWithdrawTransfer` → `token.safeTransfer(to, amount)`.
5. `safeTransfer` reverts because `to` is blacklisted (or token is paused).
6. The entire transaction reverts; `nSubmissions` is not incremented.
7. The sequencer retries and always reverts at the same index.
8. The user's USDC is permanently unwithdrawable; the sequencer queue is frozen for all users.

```text
        handleWithdrawTransfer(token, sendTo, amount);
```

```text
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
