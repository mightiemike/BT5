### Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelock Immediately — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. When the pool admin sets a `timelock` value large enough that the sum exceeds `type(uint32).max`, the result wraps to a small non-zero integer. Every subsequent `_requireElapsed` check then passes immediately, letting the admin apply any parameter change (drawdown, decay, watermarks) with zero delay — defeating the entire LP-protection mechanism.

---

### Finding Description

`_afterTimelock` returns the timestamp after which a pending change may be executed:

```solidity
// OracleValueStopLossExtension.sol L297-L299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` and then **silently truncated** to `uint32`. If `block.timestamp + timelock > type(uint32).max` (≈ 4.29 × 10⁹), the low 32 bits are kept, producing a value that may be far in the past.

The elapsed check is:

```solidity
// OracleValueStopLossExtension.sol L301-L303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
```

If `executeAfter` wraps to any value ≤ current `block.timestamp`, the revert is never triggered.

`_afterTimelock` is called by every propose function:

```solidity
// L80, L106, L133
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingTimelockExecuteAfter = executeAfter;   // or pendingDrawdownExecuteAfter, etc.
```

There is **no validation** that `timelock` is bounded to prevent overflow. `_validateDrawdown` and `_validateDecay` exist, but no `_validateTimelock` does.

---

### Impact Explanation

The `OracleValueStopLossExtension` timelock is the sole mechanism that gives LPs advance notice before the pool admin can:

- Set `drawdownE6 = 0` — disabling stop-loss entirely, allowing unlimited value drain per swap.
- Set `decayPerSecondE8` to maximum — rapidly eroding all high-watermarks so the guard never triggers.
- Set per-bin `highWatermarks` to 0 — making the guard trigger on every swap, freezing the pool.

With the timelock bypassed, the admin can apply any of these changes atomically in the same block as a swap, giving LPs no opportunity to exit.

---

### Likelihood Explanation

The pool admin is a semi-trusted role (not the fully-trusted factory owner). The contest scope explicitly lists "pool admin bypasses timelocks" as an allowed impact. The attack requires only two sequential admin transactions and no external conditions. `block.timestamp` on all live EVM chains is currently ~1.75 × 10⁹, so any `timelock` value above `type(uint32).max − block.timestamp` (≈ 2.54 × 10⁹ seconds, ~80 years) triggers the overflow — a value that is syntactically valid as `uint32`.

---

### Recommendation

Validate that `block.timestamp + timelock` does not overflow `uint32` before storing, or store and compare `executeAfter` as `uint256`:

```solidity
// Option A: validate on propose
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 ts = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(ts <= type(uint32).max, "timelock overflow");
    return uint32(ts);
}

// Option B: store executeAfter as uint256 throughout
```

Additionally, add a `_validateTimelock` that caps `newTimelock` to a reasonable maximum (e.g., `type(uint32).max − block.timestamp` at deploy time, or a protocol-defined ceiling like 365 days).

---

### Proof of Concept

Assume `block.timestamp = 1_753_142_400` (July 2026).

**Step 1 — Install an overflow-inducing timelock (current timelock = 0, so proposal executes immediately):**

```solidity
// Choose X such that uint32(block.timestamp + X) == 100 (non-zero, in the past)
uint32 X = uint32(type(uint32).max - 1_753_142_400 + 101); // = 2_541_824_996

vm.prank(poolAdmin);
ext.proposeOracleStopLossTimelock(pool, X);
// executeAfter = uint32(1_753_142_400 + 0) = 1_753_142_400 (current ts, already elapsed)

vm.prank(poolAdmin);
ext.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock = X
```

**Step 2 — Propose drawdown = 0 (disable stop-loss):**

```solidity
vm.prank(poolAdmin);
ext.proposeOracleStopLossDrawdown(pool, 0);
// executeAfter = uint32(1_753_142_400 + 2_541_824_996)
//              = uint32(4_294_967_396)
//              = uint32(4_294_967_396 % 2^32)
//              = 100   ← in the past
```

**Step 3 — Execute immediately (no delay elapsed):**

```solidity
vm.prank(poolAdmin);
ext.executeOracleStopLossDrawdown(pool);
// _requireElapsed(100): block.timestamp (1.75B) < 100 → false → passes
// drawdownE6 = 0 — stop-loss disabled with zero timelock delay
```
