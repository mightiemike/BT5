### Title
Reentrancy in `withdrawCollateral` via Token Transfer Callback Before Balance Update — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens to the recipient (`sendTo`) **before** deducting the subaccount balance in `SpotEngine`. No `nonReentrant` guard exists anywhere in the codebase. An attacker whose `sendTo` is a contract with a token-transfer callback (e.g., ERC-777 `tokensReceived`) can re-enter the Endpoint via the publicly callable `executeSlowModeTransaction()`, processing a pre-staged slow-mode withdrawal while the balance is still stale, and double-drain the `WithdrawPool`.

---

### Finding Description

`Clearinghouse.withdrawCollateral` executes in this order:

1. `handleWithdrawTransfer(token, sendTo, amount, idx)` — external calls: `token.safeTransfer(withdrawPool, amount)` then `BaseWithdrawPool.submitWithdrawal(...)` → `token.safeTransfer(sendTo, amount)` [1](#0-0) 
2. `spotEngine.updateBalance(productId, sender, amountRealized)` — balance deducted [2](#0-1) 
3. `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)` — health check [3](#0-2) 

The token transfer to `sendTo` is the deepest external call, occurring inside `BaseWithdrawPool.handleWithdrawTransfer`:

```solidity
function handleWithdrawTransfer(IERC20Base token, address to, uint128 amount) internal virtual {
    token.safeTransfer(to, uint256(amount));   // ← re-entry point
}
``` [4](#0-3) 

At this moment, `spotEngine.updateBalance` has **not yet been called** — the subaccount balance is still at its pre-withdrawal value.

`executeSlowModeTransaction()` on the Endpoint is **publicly callable** with no access control:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    nSubmissions += 1;
    slowModeConfig = _slowModeConfig;
}
``` [5](#0-4) 

This path eventually dispatches a slow-mode `WithdrawCollateral` transaction back into `Clearinghouse.withdrawCollateral`. Because the outer call has not yet deducted the balance, the inner call sees the full original balance, passes its own health check, and sends a second batch of tokens to `sendTo`.

There is **no `nonReentrant` modifier** anywhere in the codebase — confirmed by a full-codebase search returning zero matches for `nonReentrant` or `ReentrancyGuard`. [6](#0-5) 

---

### Impact Explanation

The attacker receives two token transfers from `WithdrawPool` for a single on-chain balance. After re-entry completes, the outer `withdrawCollateral` deducts the balance a second time (driving it negative). If the attacker holds sufficient cross-margined collateral (e.g., perp positions) to satisfy the outer health check at the doubly-deducted balance, the transaction succeeds and the `WithdrawPool` is drained of twice the intended amount. The protocol's accounting is permanently desynchronised: `SpotEngine` records a negative balance while real tokens have left the pool twice.

---

### Likelihood Explanation

**Prerequisites:**
1. The withdrawal token must support a transfer callback (ERC-777 `tokensReceived`, or any token with a hook on `transfer`/`safeTransfer`). Standard USDC on Ink Chain does not, but the protocol's token registry is open to any configured token.
2. The attacker must have a slow-mode withdrawal queued ≥ 3 days prior (the `SLOW_MODE_TX_DELAY`).
3. The sequencer must be processing a withdrawal for the same subaccount in the same batch.
4. The attacker must hold enough other collateral (perp positions or other spot balances) to pass the outer health check after double deduction.

Given the 3-day staging requirement and the ERC-777 token dependency, likelihood is **Medium** for any token with transfer hooks, and **Low** for the current USDC-only deployment — but the structural absence of any reentrancy guard makes this a latent high-severity risk as the token set expands.

---

### Recommendation

1. **Add `nonReentrant`** (OpenZeppelin `ReentrancyGuardUpgradeable`) to `withdrawCollateral`, `withdrawInsurance`, and `submitFastWithdrawal`.
2. **Follow CEI strictly**: call `spotEngine.updateBalance` and `assertUtilization` **before** `handleWithdrawTransfer`, mirroring how `depositCollateral` updates state before any external interaction.

---

### Proof of Concept

```
Setup:
  - Token T is ERC-777 (has tokensReceived hook)
  - Attacker subaccount S has: 100 T spot balance + 100 T-equivalent perp position
  - 3+ days ago: attacker submitted SlowModeTx (WithdrawCollateral, 100 T) → queued at slowModeTxs[k]
  - Sequencer now submits a batch containing WithdrawCollateral(S, 100 T, sendTo=AttackerContract)

Execution:
  1. submitTransactionsChecked → processTransaction → Clearinghouse.withdrawCollateral(S, 100T, AttackerContract, idx)
  2.   handleWithdrawTransfer → token.safeTransfer(withdrawPool, 100T)
  3.   BaseWithdrawPool.submitWithdrawal → token.safeTransfer(AttackerContract, 100T)
         ↳ AttackerContract.tokensReceived() fires
  4.       AttackerContract calls Endpoint.executeSlowModeTransaction()
  5.         → Clearinghouse.withdrawCollateral(S, 100T, AttackerContract, idx_slow)
  6.           handleWithdrawTransfer → token.safeTransfer(withdrawPool, 100T)
  7.           BaseWithdrawPool.submitWithdrawal → token.safeTransfer(AttackerContract, 100T)
  8.           spotEngine.updateBalance(S, -100T)  // balance: 100 → 0
  9.           getHealth(S) = 0 (perp covers) → PASSES
  10.      executeSlowModeTransaction returns
  11.    tokensReceived returns
  12.  spotEngine.updateBalance(S, -100T)  // balance: 0 → -100
  13.  getHealth(S) = -100T spot + 100T perp = 0 → PASSES

Result: AttackerContract received 200 T; WithdrawPool drained 200 T; SpotEngine records -100 T for S.

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
```
