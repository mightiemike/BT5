Audit Report

## Title
`lastDecayTs` Advances Without Actual Decay, Permanently Locking Stop-Loss After Breach on Small-Metric Bins — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_checkAndUpdateWatermarks` unconditionally writes `hwmS.lastDecayTs = uint32(block.timestamp)` on every swap at line 284, even when `_decayed()` returns the watermark unchanged due to integer division truncation. After a stop-loss breach, swaps in the non-blocked direction continue to advance the decay clock without moving the watermark, making the breach condition permanent for bins where `hwm * ratePerSecondE8 < E8`.

## Finding Description

**Root cause — unconditional clock advance:**

In `_checkAndUpdateWatermarks` (lines 258–285), `lastDecayTs` is always written regardless of whether decay produced a non-zero step:

```solidity
// lines 280-284
hwmS.token0 = uint104(hwm0);
hwmS.token1 = uint104(hwm1);
hwmS.lastDecayTs = uint32(block.timestamp);   // ← always written
```

**Truncation in `_decayed` (lines 319–324):**

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;   // ← truncates to zero when hwm * factor < E8
}
```

When `hwm * factor < E8`, `(hwm * factor) / E8 = 0`, so `_decayed` returns `hwm` unchanged.

**`_applyWatermark` (lines 328–336)** then returns the same `hwm` when `metric < hwm` (breach state), so `hwmS.token0/token1` are written back with identical values — but `lastDecayTs` is still advanced to `block.timestamp`.

**Exploit flow:**

1. A swap drains a bin enough to trigger `OracleStopLossTriggered` — the pool is blocked in the breached direction.
2. Swaps in the still-allowed direction call `afterSwap` → `_afterSwapOracleStopLoss` → `_checkAndUpdateWatermarks` for every touched bin.
3. Inside `_checkAndUpdateWatermarks`: `breach0 && zeroForOne` is `true && false = false` (non-breached direction), so no revert. Execution reaches lines 281–284.
4. `_decayed(hwmS.token0, decayRate, 1)` returns `hwmS.token0` unchanged (truncation). `_applyWatermark` returns the same `hwm`. `hwmS.token0` is written with the same value. `lastDecayTs` advances.
5. On the next swap, `dt = 1` again, same truncation, same result. The watermark never falls.
6. The pool is permanently blocked in the breached direction.

**Concrete numbers (using the contract's own NatDoc rate):**

- `ratePerSecondE8 = 58` (≈5%/day, cited at line 129), `dt = 1 s` → `factor = 58`
- Truncation condition: `hwm * 58 < 1e8` → `hwm < 1,724,137`
- `METRIC_SCALE = 1e6`; a bin with `t0 = 1000` scaled units, `shares = 10,000` → `t0ps = mulDiv(1000, 1e6, 10000) = 100`
- `hwm = 100`: `(100 × 58) / 1e8 = 0` — no decay, clock resets

No existing guard prevents this: the only recovery paths are (a) the pool admin manually overriding watermarks via `executeOracleStopLossHighWatermarks` (timelocked, privileged) or (b) a swap that raises the metric above the watermark (impossible while the bin is drained).

## Impact Explanation

After a stop-loss breach on a small-metric bin, the pool's swap path in the breached direction is permanently disabled. No unprivileged actor can re-arm the stop-loss. The pool is effectively dead as a trading venue in one direction, matching the allowed impact: **broken core pool functionality causing unusable swap flows**. LPs can still exit via `removeLiquidity`, but the pool cannot function as a two-sided market.

## Likelihood Explanation

- No privileged actor is required; any public swap in the non-blocked direction advances the clock.
- Small per-share metrics (`hwm < 1,724,137`) are common in bins partially consumed by prior swaps or pools with high share counts relative to token balances.
- The decay rate of 58 is the value cited in the contract's own NatDoc comment (line 129) as a typical configuration.
- The condition is self-reinforcing: once the clock starts advancing without decay, every subsequent swap in the non-blocked direction perpetuates it.

## Recommendation

Only advance `lastDecayTs` when the decay computation actually produced a non-zero step. Cache the pre-decay watermark values and compare:

```solidity
function _checkAndUpdateWatermarks(...) private {
    BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
    uint256 dt = block.timestamp - hwmS.lastDecayTs;

    uint256 stored0 = hwmS.token0;
    uint256 stored1 = hwmS.token1;
    uint256 decayed0 = _decayed(stored0, decayRate, dt);
    uint256 decayed1 = _decayed(stored1, decayRate, dt);

    (uint256 hwm0, bool breach0) = _applyWatermark(metricT0, decayed0, floorMultiplier);
    if (breach0 && zeroForOne) {
        revert OracleStopLossTriggered(binIdx, true, metricT0, (hwm0 * floorMultiplier) / E6);
    }

    (uint256 hwm1, bool breach1) = _applyWatermark(metricT1, decayed1, floorMultiplier);
    if (breach1 && !zeroForOne) {
        revert OracleStopLossTriggered(binIdx, false, metricT1, (hwm1 * floorMultiplier) / E6);
    }

    hwmS.token0 = uint104(hwm0);
    hwmS.token1 = uint104(hwm1);
    // Only advance the clock if decay actually moved the stored watermark
    if (decayed0 != stored0 || decayed1 != stored1) {
        hwmS.lastDecayTs = uint32(block.timestamp);
    }
}
```

This ensures elapsed time accumulates in `dt` until `hwm * factor >= E8` and a non-zero decay step occurs.

## Proof of Concept

```
Setup:
  - Pool with OracleValueStopLossExtension
  - drawdownE6 = 50_000, decayPerSecondE8 = 58
  - Bin 0: t0 = 1000 scaled units, shares = 10_000
    → metricT0 ≈ 100 (per-share, METRIC_SCALE = 1e6)

Step 1: First swap sets hwmS.token0 = 100, hwmS.lastDecayTs = T0.

Step 2: Drain bin to t0 = 40 → metricT0 ≈ 40.
        40 < 100 * (1e6 - 50_000) / 1e6 = 95 → breach triggered.
        zeroForOne swap reverts with OracleStopLossTriggered.

Step 3: Every second, a swap in the non-blocked direction (zeroForOne=false) calls afterSwap:
        dt = 1, factor = 58 * 1 = 58
        (100 * 58) / 1e8 = 0 → _decayed returns 100 (unchanged)
        _applyWatermark(40, 100, ...) → (100, true) [breach, but zeroForOne=false → no revert]
        hwmS.token0 = 100 (unchanged), hwmS.lastDecayTs = block.timestamp ← clock advances

Step 4: After 86400 seconds (1 day), expected decay: hwm ≈ 95.
        Actual: hwm = 100 forever. Pool permanently blocked in zeroForOne direction.

Foundry test outline:
  1. Deploy pool + OracleValueStopLossExtension with above params.
  2. Perform initial swap to set watermark.
  3. Drain bin via swap to trigger breach.
  4. Loop: vm.warp(block.timestamp + 1); perform non-blocked swap.
  5. After 86400 iterations, assert currentHighWatermarks() still returns 100.
  6. Attempt zeroForOne swap; assert it still reverts with OracleStopLossTriggered.
```
