Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call) and performs its allowlist check exclusively on `owner` (the caller-supplied position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` with no `msg.sender == owner` guard, any non-allowlisted address can bypass the allowlist by supplying an allowlisted address as `owner`. This renders the deposit allowlist entirely ineffective and enables forced LP position griefing on allowlisted addresses.

## Finding Description
In `DepositAllowlistExtension.beforeAddLiquidity` (L32–42), the first parameter (`sender`) is unnamed and discarded; the check is performed on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` (L191) passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

Unlike `removeLiquidity` (L206), which enforces `if (msg.sender != owner) revert NotPositionOwner()`, `addLiquidity` has no such guard. An attacker Bob (not allowlisted) calls `pool.addLiquidity(alice, ...)` where Alice is allowlisted. The extension checks `allowedDepositor[pool][alice]` → passes. Bob's tokens are pulled via callback; Alice receives the LP position. The allowlist is completely defeated.

The same bypass is reachable through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` (L56–68), where `_validateOwner` only checks `owner != address(0)` and does not require `owner == msg.sender`.

## Impact Explanation
The deposit allowlist — the pool admin's primary access control mechanism for gating LP deposits — is completely ineffective. Any unprivileged address can deposit into a restricted pool by naming any allowlisted address as `owner`. Additionally, an attacker can force unwanted LP positions onto allowlisted addresses without their consent, exposing them to impermanent loss or price risk in pools they did not choose to enter. Pools configured for compliance or KYC purposes provide no actual restriction. This constitutes an admin-boundary break by an unprivileged path and potential direct financial loss to allowlisted users.

## Likelihood Explanation
`addLiquidity` is a public, permissionless function with no `msg.sender == owner` guard. The bypass requires only knowing one allowlisted address, which is publicly readable from the `allowedDepositor` mapping or emitted events. No special privileges, flash loans, or complex setup are required. The periphery's `addLiquidityExactShares(pool, owner, ...)` makes the attack trivially accessible to EOAs.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Additionally, consider whether `addLiquidity` should enforce `msg.sender == owner` (as `removeLiquidity` does) to prevent forced position griefing, or whether the allowlist should check both `sender` and `owner`.

## Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension` configured.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — Alice is allowlisted; Bob is not.
3. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(bob, alice, ...)` → extension checks `allowedDepositor[pool][alice]` = `true` → **passes**.
5. Bob's tokens are pulled via callback; Alice receives the LP position.
6. Bob has successfully deposited into a pool he is not allowlisted for; Alice has an unwanted position she did not consent to.

Reproducible as a Foundry unit test: deploy pool with extension, allowlist Alice, call `addLiquidity` as Bob with `owner = alice`, assert the call succeeds and Alice holds the position. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
