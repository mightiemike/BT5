Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is set to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender`, so the allowlist check evaluates the router's address rather than the actual user's address. If the pool admin allowlists the router (the only way to let allowlisted users reach the pool via the router), any non-allowlisted user can bypass the per-pool swap gate entirely by calling the router.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the hook dispatcher.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`. When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `IMetricOmmPoolActions(params.pool).swap(...)`, making the router `msg.sender` inside the pool. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` = pool and `sender` = router.** [3](#0-2) 

The allowlist mapping is keyed `allowedSwapper[pool][swapper]`. When the router calls `pool.swap()`, `sender` = router address, so the check becomes `allowedSwapper[pool][router]`. The actual user's address is never consulted.

**The router does not encode the original caller into `extensionData`.** In `exactInputSingle`, the router passes `params.extensionData` directly from user-supplied calldata — the user controls this field entirely, so no trusted user-identity can be derived from it. [4](#0-3) 

**Existing guards are insufficient:** The only check in `beforeSwap` is `allowedSwapper[msg.sender][sender]`. There is no secondary check on `extensionData`, no check on the swap callback payer, and no mechanism to recover the original caller's identity from the pool's perspective.

**Contrast with `DepositAllowlistExtension`:** `beforeAddLiquidity` checks `owner` (the LP position owner, passed explicitly by the caller), not `sender`. This correctly identifies the economic actor regardless of who calls `addLiquidity`. [5](#0-4) 

## Impact Explanation
A pool deploying `SwapAllowlistExtension` to restrict trading to a curated set of counterparties (e.g., institutional traders, KYC'd addresses) is fully bypassed. Any non-allowlisted user can call `MetricOmmSimpleRouter` targeting the pool and trade freely. LP funds on the curated pool are exposed to unauthorized traders, enabling adverse selection and direct LP principal loss. This matches **"Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path"** and **"Broken core pool functionality causing loss of funds."**

## Likelihood Explanation
The bypass is reachable whenever: (1) a pool is deployed with `SwapAllowlistExtension` active, and (2) the pool admin allowlists the router — the only configuration that allows allowlisted users to reach the pool via the router. Both conditions are expected in normal production use. No privileged attacker role is required; any public user can call the router.

## Recommendation
The extension must check the economic actor (the end user), not the transport layer (the router). Two viable approaches:

1. **Encode the original caller in `extensionData` at the router level:** The router encodes `msg.sender` (the user) into `extensionData` before forwarding to the pool. `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. The encoding must be done by the router itself (not user-supplied) to be trustworthy.

2. **Check both `sender` and a decoded user address:** If `extensionData` is non-empty and originates from a trusted router, decode and check the user; otherwise fall back to `sender`. This preserves backward compatibility for direct pool calls.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` and `setAllowedToSwap(pool, router, true)` (router allowlisted so Alice can use it).
3. Bob (non-allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
4. Router calls `pool.swap(recipient, ...)` — `msg.sender` inside the pool = router address.
5. Pool calls `_beforeSwap(router, ...)`.
6. `SwapAllowlistExtension` evaluates `allowedSwapper[pool][router]` → `true`.
7. Bob's swap executes. The allowlist never checked Bob's address.
