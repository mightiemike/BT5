### Title
Ghost Heap Counter Not Updated on SmallAtom `substr` — Paired Accounting Invariant Broken - (File: src/allocator.rs)

### Summary

In `Allocator::new_substr`, when a substring of a `SmallAtom` produces a result that also fits in a `SmallAtom`, `ghost_atoms` is incremented but `ghost_heap` is not. Every other SmallAtom allocation site (`new_atom`, `new_small_number`, `new_concat` single-node path) updates both counters together. The missing `ghost_heap` update causes `heap_size()` to underreport actual virtual heap usage and silently bypasses the heap-limit guard for this code path.

### Finding Description

The `Allocator` maintains two paired "ghost" counters — `ghost_atoms` and `ghost_heap` — to account for SmallAtoms (atoms stored inline in a `NodePtr` rather than in `u8_vec`). These counters exist so that consensus-critical limits (`MAX_NUM_ATOMS`, `heap_limit`) behave identically to a hypothetical implementation that always allocates atoms on the heap.

Every SmallAtom creation site updates both counters together:

**`new_atom` (small path):** [1](#0-0) 

**`new_small_number`:** [2](#0-1) 

**`new_concat` (single-node path):** [3](#0-2) 

But in `new_substr` for the `SmallAtom → SmallAtom` branch, only `ghost_atoms` is incremented; `ghost_heap` is never touched, and the heap-limit guard is never checked: [4](#0-3) 

Compare with the `SmallAtom → Bytes` branch in the same function, which does allocate on the real heap (and is therefore bounded by the existing `check_atom_limit` call at the top of `new_substr`). The SmallAtom fast-path has no equivalent heap-size guard at all.

The heap-limit check that all other allocation paths respect is: [5](#0-4) 

This check is absent for the SmallAtom substr path, and because `ghost_heap` is never incremented there, even a future addition of such a check would compute the wrong value.

### Impact Explanation

`heap_size()` returns `u8_vec.len() + ghost_heap`: [6](#0-5) 

Each attacker-controlled `(substr ...)` call on a small integer that produces a small-integer result silently widens the gap between the reported and actual virtual heap size. The heap-limit guard in `new_atom` / `new_small_number` / `new_concat` uses the underreported `ghost_heap`, so subsequent real-heap allocations are permitted beyond the configured `heap_limit`. In mempool mode (`MEMPOOL_MODE` includes `LIMIT_HEAP`): [7](#0-6) 

a caller that constructs the allocator with a finite `heap_limit` (via `Allocator::new_limited`) to enforce the `LIMIT_HEAP` policy can have that limit bypassed by a program that first inflates `ghost_atoms` without inflating `ghost_heap` through repeated SmallAtom `substr` calls, then allocates real heap bytes that would otherwise be rejected.

The corrupted value is `ghost_heap` (and therefore `heap_size()`). The broken invariant is: for every SmallAtom created, `ghost_heap` must be incremented by the atom's byte length.

### Likelihood Explanation

`op_substr` is a standard, always-available CLVM operator (opcode 12): [8](#0-7) 

Any attacker-supplied CLVM program can call `(substr <small-int> 0 1)` in a tight loop. The SmallAtom → SmallAtom path is taken whenever the source atom is a small positive integer (≤ 26-bit value, no leading zeros, non-negative) and the resulting slice also satisfies `fits_in_small_atom`. This is trivially achievable with inputs like `(q . 1)` or `(q . 255)`. No special dialect flags or softfork guards are required.

### Recommendation

In `new_substr`, inside the `ObjectType::SmallAtom` branch where the result fits in a SmallAtom, add the missing paired update and heap-limit guard, mirroring `new_atom` and `new_small_number`:

```rust
// After: if let Some(new_val) = fits_in_small_atom(substr) {
let substr_len = (end - start) as usize;
if self.u8_vec.len() + self.ghost_heap + substr_len > self.heap_limit {
    return Err(EvalErr::OutOfMemory);
}
self.ghost_atoms += 1;
self.ghost_heap += substr_len;   // ← missing update
Ok(self.mk_node(ObjectType::SmallAtom, new_val as usize))
```

### Proof of Concept

```
; CLVM program: repeatedly substr a small atom to exhaust ghost_heap
; without triggering the heap_limit guard
(a
  (q . (a 2 (c 2 (c 5 (c 11 ())))))
  (c
    (q . (i 11
           (a 2 (c 2 (c (substr 5 (q . 0) (q . 1)) (c (- 11 (q . 1)) ()))))
           (q . ())))
    (c (q . 0x01) (c (q . 10000) ()))))
```

Each iteration calls `(substr 0x01 0 1)` → produces SmallAtom `0x01`, incrementing `ghost_atoms` by 1 but leaving `ghost_heap` unchanged. After N iterations, `ghost_heap` is underreported by N bytes, allowing N additional bytes of real heap allocation beyond `heap_limit` in any caller using `Allocator::new_limited`.
