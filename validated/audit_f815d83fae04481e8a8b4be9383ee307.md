Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of real swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument passed by the pool, which is `msg.sender` of the `pool.swap()` call — the router address when a user routes through `MetricOmmSimpleRouter`. If a pool admin allowlists the router to enable periphery access for allowlisted users, every unprivileged user on the internet can bypass the allowlist by calling the router, because the extension only sees `sender = router` (allowlisted), never the real user.

## Finding Description

The call chain is confirmed in production code:

**Step 1:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` with `msg.sender = router`. [1](#0-0) 

**Step 2:** `MetricOmmPool.swap` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`. [2](#0-1) 

**Step 3:** `ExtensionCalling._beforeSwap` forwards `sender` verbatim to the extension. [3](#0-2) 

**Step 4:** `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (correct) and `sender` is the router (wrong — should be the user). [4](#0-3) 

Two broken outcomes result:

1. **Bypass (high impact):** Admin allowlists the router so allowlisted users can use periphery → `allowedSwapper[pool][router] = true` → any user calling `router.exactInputSingle(...)` passes the check, because the extension sees `sender = router`.
2. **Broken functionality:** If admin does not allowlist the router, allowlisted users cannot use `MetricOmmSimpleRouter` at all — the extension always sees `sender = router` (not allowlisted) and reverts.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the explicit position owner parameter), not `sender`, so it does not share this bug. [5](#0-4) 

The existing test suite only tests direct pool calls (no router intermediary), so the bypass is not caught. [6](#0-5) [7](#0-6) 

## Impact Explanation

A pool using `SwapAllowlistExtension` is a curated pool intended to restrict trading to specific counterparties (KYC'd users, whitelisted market makers, specific strategies). When the admin takes the natural operational step of allowlisting the router (`setAllowedToSwap(pool, router, true)`) to enable periphery access, the curation is completely defeated. Any unprivileged EOA can trade on the pool by routing through `MetricOmmSimpleRouter`. LP principals are exposed to counterparties the pool was explicitly designed to exclude, causing direct LP loss if the pool's profitability depends on counterparty selection. This constitutes broken core pool functionality causing loss of funds and a broken allowlist invariant — a High severity impact under Sherlock thresholds.

## Likelihood Explanation

The trigger is a single, natural admin action: `setAllowedToSwap(pool, router, true)`. Any pool admin who wants allowlisted users to use the in-scope periphery router will take this step. The exploit requires no special privileges — any EOA can call `MetricOmmSimpleRouter.exactInputSingle`. The router is a production, in-scope periphery contract. The precondition (router allowlisted) is the expected operational state for any curated pool that intends to support periphery access.

## Recommendation

The pool should pass the real user identity through the swap path. Options:

1. Have `MetricOmmPool.swap` accept an explicit `originator` parameter and pass it as `sender` to extensions, with the router forwarding `msg.sender` as originator.
2. Use `extensionData` to carry a signed or router-attested user identity that `SwapAllowlistExtension` reads and verifies.
3. Document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with router-mediated swaps, rejecting pool configurations that combine a swap allowlist with a non-zero `beforeSwapOrder` when the router is a known caller.

The cleanest fix is option 1: add an `originator` field to the swap call path so extensions always receive the true initiating address.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension, only alice is allowlisted
swapAllowlist.setAllowedToSwap(pool, alice, true);
// Admin allowlists the router so alice can use periphery
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attack: bob (not allowlisted) bypasses the guard via router
vm.prank(bob);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: token0,
        recipient: bob,
        zeroForOne: true,
        amountIn: 1000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Succeeds: extension saw sender=router (allowlisted), not bob (not allowlisted)
// bob traded on a curated pool he was explicitly excluded from
```

The call chain: `bob → router.exactInputSingle → pool.swap(msg.sender=router) → _beforeSwap(sender=router) → SwapAllowlistExtension.beforeSwap checks allowedSwapper[pool][router] = true → passes`.
