Audit Report

## Title
`addLiquidity()` and `removeLiquidity()` bypass the `whenNotPaused` guard, allowing LP fund extraction during a paused pool state — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool.swap()` is protected by the `whenNotPaused` modifier, which reverts when `pauseLevel != 0`, but `addLiquidity()` and `removeLiquidity()` carry no such guard. When the pool is paused via `setPause()`, LPs can still call `removeLiquidity()` directly to transfer tokens out of the pool, defeating the intended emergency freeze.

## Finding Description

The `whenNotPaused` modifier at [1](#0-0)  delegates to `_checkNotPaused()` at [2](#0-1) , which reverts with `PoolPaused()` whenever `pauseLevel != 0`. The factory can set `pauseLevel` to 1 or 2 via `setPause()` at [3](#0-2) .

`swap()` correctly applies this guard: [4](#0-3) 

However, `addLiquidity()` at [5](#0-4)  and `removeLiquidity()` at [6](#0-5)  only carry `nonReentrant` — no `whenNotPaused`. Both proceed to call `LiquidityLib.removeLiquidity` and transfer tokens out of the pool with no pause check at any point in the call path.

## Impact Explanation

When the pool is paused at level 1 or 2, the intended invariant is that no user-facing state-changing operation can proceed. Because `removeLiquidity()` is unguarded, any LP can call it directly on the pool contract and receive token transfers from a pool whose state may be compromised (e.g., bad oracle prices, corrupted bin accounting). This constitutes a direct loss of LP principal for remaining LPs: one LP can drain their position — potentially at inflated accounting values — before the issue is resolved, leaving other LPs unable to recover their fair share once the pause is lifted. This meets the threshold of direct loss of user principal and broken core pool functionality.

## Likelihood Explanation

- The two-level pause mechanism is an explicit operational feature, meaning pausing is an expected admin action during incidents.
- Any LP can call `removeLiquidity()` directly on the pool contract without any special privilege — only `msg.sender == owner` is required, which the LP trivially satisfies for their own position.
- No precondition beyond the pool being paused is needed; the exploit path is a single direct contract call.
- The condition is repeatable for every LP during any pause window.

## Recommendation

Add the `whenNotPaused` modifier to both functions:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

This mirrors the guard already applied to `swap()` and uniformly enforces the pause invariant across all user-facing state-changing operations.

## Proof of Concept

1. Deploy a pool with an oracle-backed price provider.
2. Call `factory.setPause(poolAddress, 1)`, setting `pauseLevel = 1`.
3. Confirm `swap()` reverts with `PoolPaused()`.
4. Call `pool.removeLiquidity(owner, salt, deltas, extensionData)` directly as the LP. The call succeeds: no `whenNotPaused` check exists, `LiquidityLib.removeLiquidity` executes, and tokens are transferred to the LP.
5. The LP successfully withdraws their full position from a pool the admin intended to be completely frozen.
