Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual User, Allowing Any User to Bypass Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of the pool call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract address, not the actual user. If the router is allowlisted—a natural operational choice for a curated pool that wants to support the official periphery—every user, including those explicitly excluded, can bypass the swap allowlist entirely by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension is the pool): [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [3](#0-2) 

At that point `msg.sender` inside the pool is the **router address**, so the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The router never forwards the real user's address as the `sender` parameter—it simply calls `pool.swap(params.recipient, ...)` where the pool's `msg.sender` is the router itself.

The same structural problem exists for multi-hop `exactInput` paths: [4](#0-3) 

Existing guards are insufficient: the `allowedSwapper` mapping is keyed by `[pool][sender]` where `sender` is always the immediate caller of `pool.swap`. There is no mechanism in the extension or the pool to recover the original end-user address. The `extensionData` field is user-supplied and unverified, so it cannot be trusted to carry the real user's address without additional authentication.

## Impact Explanation
A curated pool's swap allowlist is its primary access-control boundary. Any user who is explicitly excluded from the allowlist can bypass it entirely by calling through `MetricOmmSimpleRouter`. This is a direct policy bypass: the pool admin's intent to restrict trading to a specific set of addresses is defeated for all router-mediated swaps. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, or whitelist-only liquidity), this allows unauthorized parties to trade against LP assets, causing direct loss of LP principal through adverse selection or fee leakage to unintended counterparties. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical, documented periphery swap entry point. Pool admins who configure a `SwapAllowlistExtension` and want to support normal user flows will allowlist the router as a matter of course. The bypass requires no special privileges, no malicious setup, and no unusual token behavior—any user with tokens can call the router. The trigger is a standard public function call, making this trivially repeatable.

## Recommendation
The extension must gate on the **original user**, not the intermediary. The cleanest fix consistent with the existing architecture is for the router to encode the real user's address in `extensionData` and for `SwapAllowlistExtension` to decode and verify it, with the pool verifying the router is a trusted forwarder. Specifically:

1. `MetricOmmSimpleRouter` should encode `msg.sender` into `extensionData` before calling `pool.swap`.
2. `SwapAllowlistExtension.beforeSwap` should, when `sender` is a known trusted forwarder (e.g., the router), decode the real user from `extensionData` and check `allowedSwapper[pool][realUser]` instead.
3. Alternatively, maintain a separate `trustedForwarder` registry in the extension so the pool can verify the router's claim about the real user.

## Proof of Concept
```
Setup:
  1. Deploy pool with SwapAllowlistExtension.
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     — natural step to support the official periphery.
  3. Pool admin does NOT allowlist attacker (address A).

Attack:
  4. Attacker (address A) calls:
       MetricOmmSimpleRouter.exactInputSingle({
         pool: pool,
         recipient: A,
         zeroForOne: true,
         amountIn: X,
         ...
       })
  5. Router calls pool.swap(A, true, X, ..., extensionData)
     with msg.sender = router.
  6. Pool calls _beforeSwap(msg.sender=router, ...).
  7. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  8. Swap executes. Attacker receives output tokens.

Result:
  Attacker, who is not on the allowlist, successfully swaps against
  the curated pool. The allowlist invariant is broken.
```
