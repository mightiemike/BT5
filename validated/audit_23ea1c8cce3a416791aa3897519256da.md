### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash for Syncing Nodes - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` converter in the P2P sync path silently collapses `ValidResourceBounds::AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` into `ValidResourceBounds::L1Gas(X)`. Because `get_tip_resource_bounds_hash` hashes a different number of elements for each variant — three resource-bound felts for `AllResources` (L1_GAS, L2_GAS, L1_DATA_GAS) versus two for `L1Gas` (L1_GAS, L2_GAS) — the transaction hash computed by a syncing node diverges from the hash computed by the proposing node. Any V3 transaction submitted with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` triggers this divergence.

---

### Finding Description

**Step 1 — Submission and hash computation at the proposer.**

A user submits a V3 invoke transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it; `RpcInvokeTransactionV3.resource_bounds` is typed as `AllResourceBounds` and is always wrapped as `ValidResourceBounds::AllResources` when the hash is computed:

```rust
// crates/starknet_api/src/rpc_transaction.rs:637-638
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
```

`get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources`. That function includes `l1_data_gas` in the poseidon preimage:

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

The resulting hash covers **four** poseidon inputs: `tip | l1_gas_felt | l2_gas_felt | l1_data_gas_felt`.

**Step 2 — Protobuf serialization.**

When the block is propagated via P2P, the `Transaction::Invoke(InvokeTransaction::V3(...))` is serialized. `From<ValidResourceBounds> for protobuf::ResourceBounds` emits `l1_data_gas: Some(ResourceBounds::default().into())` (all-zero) for the `L1Gas` arm, and the actual value for `AllResources`. For this transaction, `l1_data_gas` is serialized as zero.

**Step 3 — Protobuf deserialization at the syncing node.**

The syncing node deserializes the protobuf message using `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs:426-435
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong variant selected
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Because `l2_gas = 0` and `l1_data_gas = 0`, the condition is true and the variant becomes `ValidResourceBounds::L1Gas`. The original `AllResources` variant is silently discarded.

**Step 4 — Hash recomputation at the syncing node.**

`get_tip_resource_bounds_hash` is called with `ValidResourceBounds::L1Gas`. The `l1_data_gas` element is **not** included:

```rust
ValidResourceBounds::L1Gas(_) => vec![],   // l1_data_gas dropped
```

The hash now covers only **three** poseidon inputs: `tip | l1_gas_felt | l2_gas_felt`. This is a different hash than the one computed by the proposer.

**Step 5 — Divergence.**

The syncing node's recomputed hash ≠ the hash stored in the block. `validate_transaction_hash` returns `false`, causing the syncing node to reject the block or store the transaction under the wrong hash. Either outcome produces state divergence between the proposer and the syncing node.

---

### Impact Explanation

This is a **Critical** impact matching: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

A syncing node that receives a block containing a V3 transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` will compute a different transaction hash than the proposer. This causes:
- Hash validation failure → block rejection → state divergence between proposer and syncing nodes.
- Or, if hash validation is skipped, the transaction is stored under the wrong hash, making all receipts, events, and RPC responses for that transaction reference an incorrect hash.

---

### Likelihood Explanation

**High.** Any unprivileged user can submit a V3 transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }`. The gateway imposes no restriction on zero `l2_gas` or `l1_data_gas`. The transaction will be included in a block by the proposer and then synced to all other nodes, triggering the divergence on every syncing node.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not use the zero-check heuristic to select the variant. Instead, preserve the original variant by requiring an explicit discriminant field in the protobuf message, or always deserialize as `AllResources` when all three fields are present (matching the behavior of `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs`):

```rust
// crates/apollo_protobuf/src/converters/rpc_transaction.rs:212-223
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?,
        })
    }
}
```

The `transaction.rs` converter should adopt the same approach: always produce `AllResources` when all three fields are present, and only produce `L1Gas` when the protobuf message explicitly signals the legacy format (e.g., `l1_data_gas` is absent, not merely zero).

---

### Proof of Concept

1. Submit a V3 invoke transaction with `AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.

2. The proposer computes the hash using `get_tip_resource_bounds_hash` with `ValidResourceBounds::AllResources`. The poseidon preimage is `[tip=0, l1_gas_felt, l2_gas_felt=0, l1_data_gas_felt=0]` — **4 elements**.

3. The transaction is included in a block and serialized to protobuf. `l1_data_gas` is serialized as `ResourceLimits { max_amount: 0, max_price_per_unit: 0 }`.

4. A syncing node deserializes the protobuf. `l1_data_gas.is_zero() && l2_gas.is_zero()` is `true`, so `ValidResourceBounds::L1Gas(l1_gas)` is produced.

5. The syncing node recomputes the hash using `get_tip_resource_bounds_hash` with `ValidResourceBounds::L1Gas`. The poseidon preimage is `[tip=0, l1_gas_felt, l2_gas_felt=0]` — **3 elements**.

6. `poseidon([0, l1_gas_felt, 0, 0]) ≠ poseidon([0, l1_gas_felt, 0])`. The hashes diverge. The syncing node rejects the block or stores the transaction under the wrong hash. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
