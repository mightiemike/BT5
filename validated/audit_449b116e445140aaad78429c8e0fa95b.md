Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the originating user, allowing any caller to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to its own `msg.sender`. When `MetricOmmSimpleRouter` calls `pool.swap()`, the router becomes `msg.sender` at the pool level, so the extension checks the router's allowance rather than the end user's. Any pool admin who allowlists the router to enable router-based swaps for their curated users simultaneously grants every address on-chain the ability to bypass the per-user gate by routing through the router.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool. The pool sets that argument unconditionally to its own `msg.sender`:

```solidity
_beforeSwap(
    msg.sender,   // pool's msg.sender → router when routed
    recipient, zeroForOne, ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly with no intermediary, making the router the pool's `msg.sender`:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., "", params.extensionData);
```

The router stores the real payer (`msg.sender`) in transient storage for payment purposes only; it is never forwarded to the extension. The `extensionData` passed to the pool is `""` (empty), so the extension has no way to recover the originating user's address. The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

When a pool admin calls `setAllowedToSwap(pool, router, true)` — a necessary step to allow any individually allowlisted user to use the router — the check `allowedSwapper[pool][router] == true` passes for every caller of the router, including addresses that were never individually allowlisted.

## Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for curated pools to restrict who may trade. Once the router is allowlisted (a routine admin action required to support router-based swaps for legitimate users), the allowlist provides zero protection: any address can swap by routing through `MetricOmmSimpleRouter`. This is a direct policy bypass on curated pools, enabling unauthorized users to extract liquidity at oracle-anchored prices from pools intended to be restricted. This matches the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" allowed impact categories. Severity: High.

## Likelihood Explanation

The bypass requires the pool admin to have allowlisted the router address. This is a natural and expected action — without it, even individually allowlisted users cannot use the router. Any pool that wants to support both direct and router-based swaps for its allowlisted users will trigger the vulnerability. The router is a first-party, factory-validated contract (`_requireFactoryPool` check in `MetricOmmSwapRouterBase`), so allowlisting it is not a suspicious or adversarial act. The bypass is repeatable by any address with no special privileges.

## Recommendation

The extension must check the actual economic actor, not the intermediary. The router already stores the real payer in transient storage (`_setNextCallbackContext(..., msg.sender, ...)`). The correct fix is to expose the originating user address to the extension — either by passing it through `extensionData` at the router level, or by adding a dedicated hook argument for the originating caller. The `beforeSwap` hook should verify the originating user address (the payer/initiator), not just the immediate `msg.sender` of the pool call.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, router, true)   // enable router for allowlisted users
  pool admin calls setAllowedToSwap(pool, alice, true)    // alice is individually allowlisted
  bob is NOT individually allowlisted

Attack:
  bob calls router.exactInputSingle({pool: pool, ...})
    → _setNextCallbackContext stores bob as payer in transient storage (not visible to extension)
    → router calls pool.swap(...)          // msg.sender = router
    → pool calls _beforeSwap(router, ...)
    → extension checks allowedSwapper[pool][router] == true  ✓
    → swap executes for bob

Result: bob successfully swaps on a pool that was supposed to deny him.
```

Foundry test plan: deploy a pool with `SwapAllowlistExtension`, allowlist the router, do NOT allowlist `bob`, call `router.exactInputSingle` from `bob`'s address, assert the swap succeeds and no `NotAllowedToSwap` revert occurs.
