### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Fix Enforced by `verify_transaction_inclusion_v2` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a mandatory coinbase Merkle proof check to close the 64-byte transaction Merkle forgery vulnerability. However, the deprecated function retains `pub` visibility and no runtime access restriction, so any unprivileged NEAR caller can invoke it directly, completely bypassing the coinbase proof guard. This is a direct analog of the reported improper-access-control class: a security boundary introduced by a newer code path is silently circumventable because the old, insecure path was never restricted.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated]` and carries an explicit warning in its own doc comment:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

The function is still declared `pub` and is gated only by `#[pause]`, which blocks calls only when the contract is administratively paused — not as an access-control mechanism against ordinary callers. [2](#0-1) 

`verify_transaction_inclusion_v2` was introduced specifically to fix this by requiring a coinbase Merkle proof that proves the coinbase transaction (index 0) hashes to the block's `merkle_root` before delegating to the v1 logic: [3](#0-2) 

Because `verify_transaction_inclusion` is still `pub`, any caller can skip `verify_transaction_inclusion_v2` entirely and call the insecure path directly. The `#[deprecated]` Rust attribute is a compile-time lint only; it imposes zero runtime restriction on NEAR contract calls.

---

### Impact Explanation

Any downstream contract or off-chain system that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., a bridge unlock, a payment settlement) can be fed a crafted `tx_id` that is actually an internal Merkle tree node. The function will return `true`, falsely certifying transaction inclusion. The corrupted proof result is the exact state value the contract is designed to protect: the boolean authorisation that a specific Bitcoin transaction was confirmed on-chain.

---

### Likelihood Explanation

The entry path is a direct, permissionless NEAR RPC call — no role, staked deposit, or privileged key is required. The 64-byte Merkle forgery technique is publicly documented (the contract itself links to the BitMEX writeup in the deprecation notice). The attacker only needs to know the `merkle_root` of any mainchain block, which is public on-chain state.

---

### Recommendation

Remove `pub` from `verify_transaction_inclusion` or add `#[private]` to restrict it to same-account calls only, so that external callers are forced through `verify_transaction_inclusion_v2` and its coinbase proof guard. The internal call from `verify_transaction_inclusion_v2` (line 368) will continue to work because it originates from the contract itself. [4](#0-3) 

---

### Proof of Concept

1. Attacker identifies any block hash `B` present in `mainchain_header_to_height` and reads its `merkle_root` from `headers_pool`.
2. Using the 64-byte Merkle forgery technique, attacker constructs a `tx_id` that is an internal Merkle tree node and a `merkle_proof` such that `compute_root_from_merkle_proof(tx_id, index, proof) == merkle_root`.
3. Attacker calls `verify_transaction_inclusion` directly (not v2) with `tx_block_blockhash = B`, the crafted `tx_id`, `tx_index`, and `merkle_proof`, and `confirmations = 1`.
4. The function returns `true` — falsely certifying that the crafted `tx_id` is a confirmed transaction in block `B`.
5. Any bridge or settlement contract that trusts this return value processes the fraudulent cross-chain claim. [5](#0-4)

        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,

            "Incorrect coinbase merkle proof"
        );
