### Title
`DeterministicStateInitAction` Accepts Mutable `GlobalContractIdentifier::AccountId`, Breaking the Fixed-Code Invariant of Deterministic Accounts - (File: `runtime/runtime/src/action_validation.rs`)

### Summary

The `DeterministicStateInitAction` validation does not reject `GlobalContractIdentifier::AccountId` as the code field. The deterministic account ID is derived by Borsh-encoding the `DeterministicAccountStateInit` struct (which commits to the `AccountId` string, not to the actual code bytes) and hashing it. Because a global contract deployed under `AccountId` mode can be silently replaced by its owner at any time via a new `DeployGlobalContractAction`, the same deterministic account ID can execute entirely different code before and after the update. This directly contradicts the protocol's stated invariant that "deterministic accounts have a fixed code and fixed initial state associated with it."

### Finding Description

`DeterministicAccountStateInit` embeds a `GlobalContractIdentifier`, which has two variants:

```rust
pub enum GlobalContractIdentifier {
    CodeHash(CryptoHash) = 0,   // immutable: commits to exact code bytes
    AccountId(AccountId) = 1,   // mutable: owner can redeploy at any time
}
``` [1](#0-0) 

The deterministic account ID is derived by Borsh-encoding the entire `DeterministicAccountStateInit` and hashing it:

```rust
pub fn derive_near_deterministic_account_id(state_init: &DeterministicAccountStateInit) -> AccountId {
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
``` [2](#0-1) 

When the `AccountId` variant is used, the Borsh encoding commits to the owner's account ID string (discriminant byte `1` + account ID bytes), **not** to the actual code bytes. The account ID is therefore stable across code updates.

The validation function for `DeterministicStateInitAction` only checks that the account ID string is syntactically valid — it does not require the `CodeHash` variant:

```rust
fn validate_global_contract_identifier(
    identifier: &GlobalContractIdentifier,
) -> Result<(), ActionsValidationError> {
    if let GlobalContractIdentifier::AccountId(account_id) = &identifier {
        validate_action_account_id(account_id)?;  // only format check
    }
    Ok(())
}
``` [3](#0-2) 

Meanwhile, `action_deploy_global_contract` allows the owner to overwrite the global contract stored under `GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(owner) }` at any time with no restriction: [4](#0-3) 

When `deploy_deterministic_account` runs, it calls `use_global_contract`, which looks up the current code under the `AccountId` key and stores `AccountContract::GlobalByAccount(id)` on the account: [5](#0-4) 

After the owner redeploys, every account holding `AccountContract::GlobalByAccount(owner)` — including all deterministic accounts created with that identifier — silently executes the new code. The deterministic account ID does not change.

The protocol documentation explicitly states the broken invariant:

> "Deterministic accounts have a **fixed code** and fixed initial state associated with it." [6](#0-5) 

### Impact Explanation

Any unprivileged user can:
1. Deploy a benign global contract under `AccountId` mode.
2. Create one or more deterministic accounts referencing their account ID.
3. Advertise those accounts as trustworthy because they are "deterministic."
4. Attract deposits or locked state from other users or protocols.
5. Redeploy a malicious global contract under the same account ID.
6. The malicious code now runs on all affected deterministic accounts, with full access to their storage and balance.

The deterministic account ID never changes, so on-chain verification of the account ID against the original `DeterministicAccountStateInit` passes even after the code swap. The attack costs only gas for the redeployment.

### Likelihood Explanation

The attacker must control the global contract owner account, which is a role any ordinary user can assume. No validator or privileged protocol role is required. The social-engineering surface is real: the `DeterministicStateInitAction` is explicitly designed for trustless, verifiable account initialization, so users and protocols are likely to rely on the deterministic ID as a code commitment. The `AccountId` variant is used in production test infrastructure and the params estimator, confirming it is a supported, reachable code path. [7](#0-6) 

### Recommendation

In `validate_deterministic_state_init`, reject `GlobalContractIdentifier::AccountId` outright:

```rust
fn validate_deterministic_state_init(...) -> Result<(), ActionsValidationError> {
    if matches!(action.state_init.code(), GlobalContractIdentifier::AccountId(_)) {
        return Err(ActionsValidationError::InvalidDeterministicStateInitCode);
    }
    // ... existing checks
}
```

Only `GlobalContractIdentifier::CodeHash` commits to a specific, immutable code version and is safe for use in deterministic account initialization. If mutable-code deterministic accounts are intentionally desired, the protocol documentation and the account-type invariant must be updated to reflect that the "fixed code" guarantee does not hold when `AccountId` is used, and callers must be warned accordingly.

### Proof of Concept

```
1. attacker.near deploys contract_v1 (benign) via:
      DeployGlobalContractAction { code: contract_v1, deploy_mode: AccountId }
   → stored at GlobalContractCode { identifier: AccountId("attacker.near") }

2. attacker.near sends DeterministicStateInitAction to receiver=0sXXXX... where:
      state_init = DeterministicAccountStateInit::V1 {
          code: GlobalContractIdentifier::AccountId("attacker.near"),
          data: { "owner": b"attacker.near" },
      }
   → account 0sXXXX... is created, runs contract_v1, ID commits to "attacker.near" string

3. Victim verifies: derive_near_deterministic_account_id(state_init) == 0sXXXX... ✓
   Victim verifies: GlobalContractCode[AccountId("attacker.near")] == contract_v1 ✓
   Victim sends 1000 NEAR to 0sXXXX...

4. attacker.near deploys contract_v2 (drains balance) via:
      DeployGlobalContractAction { code: contract_v2, deploy_mode: AccountId }
   → overwrites GlobalContractCode { identifier: AccountId("attacker.near") }

5. Anyone calls a method on 0sXXXX...
   → contract_v2 executes, transfers 1000 NEAR to attacker.near
   → 0sXXXX... account ID is unchanged; the state_init still validates correctly
```

The divergent Borsh value is the discriminant byte `1` (variant `AccountId`) in the `GlobalContractIdentifier` field of `DeterministicAccountStateInit`. Replacing it with discriminant `0` (variant `CodeHash`) would commit to the exact code bytes and close the attack surface.

    result: &mut ActionResult,

    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))

        GlobalContractDeployMode::AccountId => {

        Receipt::new_global_contract_distribution(account_id, distribution_receipt);

    result.new_receipts.push(distribution_receipts);
