Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the end user, enabling full allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter, which `MetricOmmPool.swap()` sets to `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router contract becomes `msg.sender` to the pool. A pool admin who allowlists the router address (the natural step to enable router-based trading for curated users) inadvertently grants every unprivileged user the ability to bypass the allowlist gate entirely.

## Finding Description

**Pool binds `msg.sender` as `sender`:**

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  recipient,
  ...
  extensionData
);
```

**Extension checks that bound value:**

`SwapAllowlistExtension.beforeSwap` receives `sender` (the pool's `msg.sender`) and checks `allowedSwapper[pool][sender]`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The `bytes calldata` extensionData parameter is received but **completely ignored** — there is no mechanism to decode a real user address from it.

**Router is `msg.sender` to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router contract the `msg.sender` seen by the pool and therefore the `sender` seen by the extension:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The same pattern applies to `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165).

**Exploit path:**

1. Pool admin deploys pool with `SwapAllowlistExtension` on `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — alice is the only intended user.
3. Admin calls `setAllowedToSwap(pool, address(router), true)` — natural step to let alice use the standard router.
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: bob, ...})`.
5. Router calls `pool.swap(bob, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(sender=router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully trades on the curated pool.

No existing guard prevents this. The extension has no fallback check on `recipient`, no extensionData decoding, and no transient-storage lookup for the originating EOA.

## Impact Explanation

Any unprivileged user can trade on a curated pool intended to be restricted to a specific set of addresses. The allowlist — the sole access-control mechanism for swap gating — is rendered completely ineffective the moment the router is allowlisted. This constitutes broken core pool functionality: the pool's curation invariant (only allowlisted addresses may swap) is violated, exposing LP assets to trades from counterparties the pool admin explicitly excluded (e.g., sanctioned addresses, unverified users, or competitors).

## Likelihood Explanation

The trigger condition — the pool admin allowlisting the router — is a natural and expected operational step for any curated pool that wants its approved users to access the standard periphery. Once the router is allowlisted, the bypass requires zero special privilege: any EOA calls `exactInputSingle` through the router. The combination of a predictable admin action and a zero-skill exploit path makes this high likelihood for any curated pool deployed with the router as a supported entry point.

## Recommendation

The extension must gate the **economic actor** (the end user), not the intermediary contract. The cleanest fix is for the router to store `msg.sender` in transient storage (analogous to how it already stores the payer via `_setNextCallbackContext`) and include it in `extensionData`. The extension then decodes and checks that address instead of `sender`. Alternatively, the pool interface could expose a `swapOnBehalfOf(address user, ...)` entry point that the router calls, with the pool passing `user` as `sender` to extensions.
