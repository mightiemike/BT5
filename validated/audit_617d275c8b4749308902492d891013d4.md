### Title
Zero-Value Fee Transfer in `submitFastWithdrawal` Blocks Fast Withdrawals for Tokens That Revert on Zero Transfers - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.submitFastWithdrawal`, when a third-party fast-withdrawal provider processes a withdrawal (`sendTo != msg.sender`), the computed fee is transferred from the provider via `safeTransferFrom(token, msg.sender, uint128(fee))` without a zero-amount guard. When `withdrawFeeX18 = 0` for a product and the withdrawal amount is small enough that the proportional fee truncates to zero via integer division, this results in a zero-value transfer. For tokens that revert on zero-value transfers, the entire fast withdrawal reverts, permanently blocking the fast-withdrawal path for those tokens.

---

### Finding Description

`fastWithdrawalFeeAmount` computes the fee as:

```
proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18)   // 0.1% of amount
minFeeX18          = 5 * spotEngine().getConfig(productId).withdrawFeeX18
feeX18             = max(proportionalFeeX18, minFeeX18)
fee                = feeX18 / multiplier                        // integer division
``` [1](#0-0) 

For a 6-decimal token (`multiplier = 10^12`), if `withdrawFeeX18 = 0` (the default uninitialized value) and `amount < 1000` raw units (i.e., < 0.001 tokens):

- `proportionalFeeX18 = amount * 10^12 / 1000 = amount * 10^9`
- `fee = (amount * 10^9) / 10^12 = 0` (integer truncation)
- `minFeeX18 = 5 * 0 = 0`

The result is `fee = 0`. This zero value is then passed unconditionally into:

```solidity
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // zero-value transfer
}
``` [2](#0-1) 

`ERC20Helper.safeTransferFrom` performs a raw `call` to `transferFrom` with no zero-amount guard: [3](#0-2) 

For any token whose `transferFrom` reverts on a zero amount, the entire `submitFastWithdrawal` call reverts.

---

### Impact Explanation

Fast withdrawals are permanently blocked for any product whose collateral token reverts on zero-value transfers, when `withdrawFeeX18 = 0` and the withdrawal amount is below the rounding threshold. The affected withdrawal cannot be fast-processed; the user is forced onto the 3-day slow-mode path. Because `withdrawFeeX18` defaults to zero for any product that has not had it explicitly configured, this condition is reachable without any privileged action.

---

### Likelihood Explanation

- `withdrawFeeX18` is uninitialized (zero) by default in Solidity storage; any product not explicitly configured has `withdrawFeeX18 = 0`.
- Withdrawal amounts below the rounding threshold (e.g., < 1000 raw units for a 6-decimal token) are valid and can be submitted by any user.
- The `sendTo != msg.sender` branch is the standard third-party fast-withdrawal provider path.
- Tokens that revert on zero transfers exist in production (e.g., certain wrapped or rebasing tokens).

All four conditions can be satisfied simultaneously without any privileged access, making this reachable by an unprivileged user submitting a small withdrawal.

---

### Recommendation

Add a zero-amount guard before the `safeTransferFrom` call, mirroring the fix applied in the referenced report:

```diff
} else {
-   safeTransferFrom(token, msg.sender, uint128(fee));
+   if (fee > 0) safeTransferFrom(token, msg.sender, uint128(fee));
}
``` [4](#0-3) 

---

### Proof of Concept

1. Deploy a token with 6 decimals whose `transferFrom` reverts on zero-value transfers.
2. Register a product with this token and leave `withdrawFeeX18 = 0` (default).
3. A user signs a `WithdrawCollateral` transaction for `amount = 999` (raw units, i.e., 0.000999 tokens).
4. A third-party fast-withdrawal provider calls `submitFastWithdrawal` with `sendTo` set to the user's address (≠ `msg.sender`).
5. `fastWithdrawalFeeAmount` returns `0` (integer truncation: `999 * 10^9 / 10^12 = 0`).
6. `safeTransferFrom(token, msg.sender, 0)` is called → token reverts → fast withdrawal is permanently blocked for this withdrawal index. [5](#0-4)

        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
