### Title
Fork Difficulty Calculated Against Mainchain Ancestors Instead of Fork Ancestors, Enabling Underpriced PoW Acceptance — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_next_work_required` in all three chain modules resolves the interval-boundary ancestor block via `get_header_by_height`, which is implemented to read exclusively from `mainchain_height_to_header`. When a fork block is being validated, this lookup silently returns the **mainchain** block at the target height rather than the fork's actual ancestor. The difficulty threshold computed from that wrong ancestor is then enforced against the submitted fork block. This is a direct cross-module desynchronization: the PoW check is evaluated against stale mainchain state rather than the fork's live chain state, mirroring the original report's pattern of a cached index diverging from the real-time value used for a critical decision.

---

### Finding Description

`get_header_by_height` is the sole implementation of the `BlocksGetter::get_header_by_height` trait method:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always mainchain index
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

`mainchain_height_to_header` is only written by `store_block_header` (mainchain path) and `reorg_chain`; fork blocks stored via `store_fork_header` are never inserted into it. [2](#0-1) 

All three chain modules call `get_header_by_height` to obtain the interval-boundary block timestamp used to compute the next required target:

**Dogecoin** (post-145 000, per-block Digishield — the most impactful case, and the one the codebase itself flags):

```rust
// contract/src/dogecoin.rs  lines 291-295
// TODO: check if it is correct to get block header by height from mainchain
//       without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)   // ← mainchain block, not fork ancestor
    .block_header.time;
``` [3](#0-2) 

**Bitcoin** (2016-block retarget boundary):

```rust
// contract/src/bitcoin.rs  line 81
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

**Litecoin** (2016-block retarget boundary):

```rust
// contract/src/litecoin.rs  line 88
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [5](#0-4) 

For Dogecoin post-145 000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for every non-genesis block, and `height_first = prev_block_header.block_height - 1`. [6](#0-5) 

This means: for every fork block at height H (where H > 145 001), the Digishield difficulty is computed using the **mainchain** block at H-2 as the interval tail, not the fork's actual ancestor at H-2. Whenever the fork diverges at or before H-2, the two timestamps differ, producing a wrong `modulated_timespan` and therefore a wrong `expected_bits`. [7](#0-6) 

The computed `expected_bits` is then enforced as the only accepted value:

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
``` [8](#0-7) 

If the mainchain ancestor at H-2 has an **earlier** timestamp than the fork's actual ancestor at H-2, `actual_timespan` is inflated, `modulated_timespan` grows, the computed target rises (easier difficulty), and `expected_bits` encodes a lower required work. A fork block that satisfies this lower threshold passes `check_pow` even though it would fail the correct fork-chain difficulty check.

---

### Impact Explanation

A fork block accepted with an artificially low difficulty carries less chain work per block than it should. Because `chain_work` is accumulated additively from `work_from_bits(header.bits)`, a sequence of such underpriced fork blocks can accumulate enough total work to trigger `reorg_chain`:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [9](#0-8) 

After a reorg, `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` all reflect the attacker's fork. Subsequent calls to `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` check membership in `mainchain_header_to_height`:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));
``` [10](#0-9) 

Transactions in the legitimate chain are no longer verifiable; transactions in the attacker's fork are falsely verified. Any consumer contract relying on `verify_transaction_inclusion` for cross-chain asset bridging or atomic swaps receives corrupted proof results.

---

### Likelihood Explanation

- The entry point is `submit_blocks`, callable by any registered relayer (or any account holding `Role::UnrestrictedSubmitBlocks`). The relayer model is explicitly designed for external parties to submit adversarial chain data.
- For Dogecoin post-145 000, the desynchronization occurs for **every** fork block after the first divergence point — no special timing or race condition is required.
- The TODO comment at `dogecoin.rs:291` confirms the developers identified this exact ambiguity but left it unresolved.
- The attacker controls the fork block timestamps (within the MTP and future-time bounds), giving direct control over the sign and magnitude of the timestamp delta that determines whether the computed difficulty is easier or harder than correct.

---

### Recommendation

Replace `get_header_by_height` (mainchain index lookup) with an ancestor traversal that walks the fork's own `prev_block_hash` chain back to the required height. Concretely, add a `get_ancestor_at_height` helper that starts from `prev_block_header` and follows `get_prev_header` links until the target height is reached, then use that in all three `get_next_work_required` implementations instead of `blocks_getter.get_header_by_height(height_first)`. This mirrors the reference implementations (Bitcoin Core's `GetAncestor`) and eliminates the mainchain/fork desynchronization.

---

### Proof of Concept

**Setup (Dogecoin mainnet, height > 145 001):**

1. Mainchain has blocks at heights H-2 (hash `M`) and H-1 (hash `M'`), with `M.time = T_m`.
2. Attacker pre-mines a fork block `F` at height H-2 with `F.time = T_f` where `T_f > T_m` (fork ancestor has a later timestamp). `F` is submitted and stored in `headers_pool` only (fork path).
3. Attacker mines fork block `F'` at height H-1 building on `F`, submitted and stored as a fork block.
4. Attacker mines fork block `G` at height H building on `F'`.

**Difficulty check for `G`:**

- `prev_block_header` = `F'` (height H-1, from `headers_pool`)
- `height_first = H - 2`
- `get_header_by_height(H-2)` → returns mainchain block `M` (timestamp `T_m`)
- `actual_timespan = F'.time - T_m` (uses mainchain `T_m`, not fork ancestor `F.time = T_f`)
- Since `T_m < T_f`, `actual_timespan` is inflated relative to the correct value `F'.time - T_f`
- `modulated_timespan` is larger → `new_target` is larger → `expected_bits` encodes easier difficulty
- `G` is accepted with `bits` matching this easier target, carrying less work than the correct fork difficulty would require

**Reorg:**

5. Attacker repeats for enough blocks until the fork's cumulative `chain_work` exceeds the mainchain's.
6. `reorg_chain` executes, replacing `mainchain_tip_blockhash` and all height-to-hash mappings with the attacker's fork.
7. `verify_transaction_inclusion` now returns `true` for attacker-fork transactions and panics for legitimate mainchain transactions.

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
