Audit Report

## Title
`addLiquidity` and `removeLiquidity` bypass the `whenNotPaused` guard, allowing deposits and withdrawals into paused pools — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool` implements a two-level pause mechanism via `pauseLevel` and the `whenNotPaused` modifier, but only applies the modifier to `swap`. Both `addLiquidity` and `removeLiquidity` are callable while the pool is paused, allowing users to deposit principal into a pool that was paused precisely because its price oracle may be compromised. When the pool is unpaused, that newly deposited liquidity is immediately exposed to swaps at potentially bad prices, resulting in direct loss of user principal.

## Finding Description

`MetricOmmPool` defines `pauseLevel` as a two-level pause flag:

```solidity
// line 71-72
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
uint8 internal pauseLevel;
```

The guard is implemented at lines 174–177 and 643–645:

```solidity
modifier whenNotPaused() { _checkNotPaused(); _; }
function _checkNotPaused() internal view { if (pauseLevel != 0) revert PoolPaused(); }
```

`swap` correctly applies the modifier (line 224):
```solidity
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
```

`addLiquidity` (line 188) and `removeLiquidity` (line 200–201) do **not**:
```solidity
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...)
// and
  nonReentrant(PoolActions.REMOVE_LIQUIDITY)
```

The `_beforeAddLiquidity` / `_afterAddLiquidity` hooks are optional per-pool extension configurations and are not a substitute for the core pause guard. A pool with no deposit-allowlist extension has zero protection on `addLiquidity` when paused.

## Impact Explanation

When a pool is paused due to a suspected oracle compromise, users (including those routed through `MetricOmmPoolLiquidityAdder`, which performs no pause check of its own) can still call `addLiquidity` and deposit tokens. Upon unpausing — even if the oracle issue is only partially resolved — those deposits are immediately exposed to `swap` calls executing at bad bid/ask prices. This constitutes a direct loss of user principal deposited during the pause window. Severity: High.

## Likelihood Explanation

- Pausing is a reachable, semi-trusted admin action (no attacker privilege required to trigger the vulnerable state).
- Any user or router calling `addLiquidity` during a pause window is affected; no special role or knowledge is needed.
- `MetricOmmPoolLiquidityAdder` calls `addLiquidity` directly with no pause check, making this reachable by ordinary users through the standard periphery path.
- The scenario (pause due to oracle suspicion → deposit during pause → unpause → drain) is a realistic operational sequence.

## Recommendation

Apply `whenNotPaused` to `addLiquidity` at minimum, and consider applying it to `removeLiquidity` as well (though allowing withdrawals during a pause is a common and defensible design choice):

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...
function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

This enforces the pause invariant at the entry point of every fund-moving operation, consistent with the intent of `pauseLevel`.

## Proof of Concept

1. Admin calls `factory.setPauseLevel(pool, 1)` — pool is paused (e.g., oracle suspected compromised).
2. `pool.swap(...)` reverts with `PoolPaused()`. ✓
3. `pool.addLiquidity(owner, salt, deltas, callbackData, extensionData)` — **succeeds**, depositing user tokens into the paused pool. ✗
4. Admin resolves (or believes they have resolved) the oracle issue and calls `factory.setPauseLevel(pool, 0)`.
5. An attacker immediately calls `pool.swap(...)` at the still-bad oracle price, draining the liquidity deposited in step 3.
6. The user who deposited in step 3 suffers a direct loss of principal.

Foundry test outline:
- Deploy pool, set `pauseLevel = 1` via factory.
- Call `addLiquidity` — assert it succeeds and tokens are transferred.
- Call `swap` — assert it reverts with `PoolPaused()`.
- Set `pauseLevel = 0`, call `swap` at a manipulated price — assert deposited tokens are drained.
