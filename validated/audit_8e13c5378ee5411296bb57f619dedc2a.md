### Title
Merkle Leaf/Internal-Node Domain Confusion Allows Fake Transaction Inclusion Proof — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) does not enforce that the supplied `tx_id` is a leaf-level node. Because `compute_root_from_merkle_proof` treats any 32-byte value as a valid starting point and walks up the tree, an attacker can supply an **internal Merkle tree node** as `tx_id` with a shortened proof path and receive a `true` result for a transaction that does not exist. This is the direct structural analog of the Meson hash-collision bug: the same 32-byte hash value is simultaneously valid in two distinct verification contexts (leaf vs. internal node), with no domain separator distinguishing them.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` computes:

```
current_hash = transaction_hash
for each proof_hash:
    current_hash = double_sha256(current_hash || proof_hash)  // or reversed
``` [1](#0-0) 

There is no check that `transaction_hash` is a leaf-level hash. An internal node at depth `d` from the root has the value `N = double_sha256(left || right)`. If an attacker supplies `tx_id = N` and a proof of length `d` (the siblings from depth `d` up to the root), `compute_root_from_merkle_proof` correctly reconstructs the Merkle root and the function returns `true`.

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR method:

```rust

#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [2](#0-1) 

`#[deprecated]` is a Rust compiler lint only — it does not restrict runtime access. Any unprivileged NEAR account can call this method when the contract is not paused.

The v2 function adds a coinbase proof check to mitigate this:

```rust
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        args.merkle_proof.len() == args.coinbase_merkle_proof.len(), ...
    );
    require!(
        merkle_tools::compute_root_from_merkle_proof(args.coinbase_tx_id.clone(), 0usize,
            &args.coinbase_merkle_proof) == header.block_header.merkle_root, ...
    );
    self.verify_transaction_inclusion(args.into())
}
``` [3](#0-2) 

But v1 remains callable directly, bypassing the coinbase guard entirely.

The `ProofArgs` struct accepts `tx_id`, `tx_index`, `merkle_proof`, and `confirmations` with no structural constraint on proof depth relative to the block's actual tree depth: [4](#0-3) 

---

### Impact Explanation

Any consumer contract or off-chain system that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action (atomic swap release, bridged asset mint, etc.) can be deceived into accepting a proof for a Bitcoin transaction that was never broadcast or confirmed. The contract returns `true` for a fabricated `tx_id` that is actually an internal Merkle tree node of a real, confirmed block. The corrupted proof result is the direct output of the public API.

---

### Likelihood Explanation

The attack requires zero computational work beyond reading the Merkle tree of any real Bitcoin block already stored in the contract's mainchain. The attacker does not need to find a preimage or collision — they only need to:

1. Identify any internal node `N` at depth `d` in a confirmed block's Merkle tree (publicly available from any Bitcoin node).
2. Compute the `d` sibling hashes forming the proof path from `N` to the root.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index` = the position of `N` at depth `d`, and `merkle_proof` = the `d`-length sibling path.

The v1 function is publicly accessible to any NEAR account. Any downstream consumer that has not independently migrated to v2 is immediately exploitable.

---

### Recommendation

Remove or gate `verify_transaction_inclusion` (v1) so it is no longer callable by unprivileged accounts. The simplest fix is to add `#[private]` to the v1 function or delete it entirely, forcing all callers to use v2. Additionally, `compute_root_from_merkle_proof` should accept an explicit expected proof length (tree depth) and reject proofs shorter than the full depth from leaf to root, enforcing that the supplied hash is treated only as a leaf-level value.

---

### Proof of Concept

1. Identify a real Bitcoin block `B` stored in the contract's mainchain (e.g., via `get_block_hash_by_height`).
2. Fetch `B`'s full transaction list from a Bitcoin node and compute its Merkle tree.
3. Pick any internal node `N = double_sha256(tx_A || tx_B)` at depth `d = 1` from the root (i.e., a direct child of the root).
4. The sibling of `N` at depth 1 is `S`. The proof path is `[S]` (length 1).
5. Call on NEAR:
   ```
   verify_transaction_inclusion({
     tx_id: N,
     tx_block_blockhash: B.hash,
     tx_index: 0,          // position of N at depth 1
     merkle_proof: [S],    // one-element proof, not full depth
     confirmations: 1
   })
   ```
6. `compute_root_from_merkle_proof(N, 0, [S])` computes `double_sha256(N || S)` = the real Merkle root of `B`.
7. The function returns `true` for a transaction `N` that does not exist on Bitcoin. [1](#0-0) [5](#0-4)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

            .get(&self.mainchain_tip_blockhash)

            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
