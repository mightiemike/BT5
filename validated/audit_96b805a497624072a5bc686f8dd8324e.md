### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash and Breaking RPC Serving of Synced Transactions — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently changes a post-0.13.3 `AllResources` transaction (with zero `l2_gas` and `l1_data_gas`) into the pre-0.13.3 `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** depending on the variant (3 for `L1Gas`, 4 for `AllResources`), the transaction hash computed at submission time diverges from the hash that would be computed from the deserialized form. Additionally, the downstream `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion hard-rejects the `L1Gas` variant, causing `starknet_getTransactionByHash` to return an error for any such transaction that was synced via P2P.

---

### Finding Description

**Step 1 — The silent variant downgrade in protobuf deserialization**

In `crates/apollo_protobuf/src/converters/transaction.rs`, the conversion from `protobuf::ResourceBounds` to `ValidResourceBounds` inspects the numeric values of the deserialized fields and silently selects the variant:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← pre-0.13.3 variant
} else {
    ValidResourceBounds::AllResources(...)       // ← post-0.13.3 variant
})
``` [1](#0-0) 

A valid post-0.13.3 `AllResources` transaction whose `l2_gas` and `l1_data_gas` happen to be zero (both amounts and prices) is therefore deserialized as `L1Gas`. The serialization direction (`From<ValidResourceBounds> for protobuf::ResourceBounds`) always emits all three fields, so the round-trip is **not idempotent** for this class of transactions. [2](#0-1) 

**Step 2 — The hash domain boundary violation**

`get_tip_resource_bounds_hash` branches on the variant and hashes a **different number of elements**:

- `L1Gas` → `[tip, l1_gas_packed, l2_gas_packed(0)]` — **3 elements**
- `AllResources` with zero l2/l1_data gas → `[tip, l1_gas_packed, l2_gas_packed(0), l1_data_gas_packed(0)]` — **4 elements**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 3 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 4 elements total
    }
});
``` [3](#0-2) 

The Poseidon hash of 3 elements is structurally different from the Poseidon hash of 4 elements. The transaction hash **H1** computed at gateway admission (using `AllResources`, 4 elements) is therefore a different value from **H2** that would be computed from the protobuf-deserialized form (using `L1Gas`, 3 elements).

**Step 3 — RPC serving failure after P2P sync**

The P2P sync client stores the deserialized `Transaction::Invoke(InvokeTransactionV3 { resource_bounds: ValidResourceBounds::L1Gas(...), ... })` in local storage. When `starknet_getTransactionByHash` is called, the RPC layer calls:

```rust
Ok(TransactionWithHash { transaction: transaction.try_into()?, transaction_hash })
``` [4](#0-3) 

This invokes `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`, which hard-rejects the `L1Gas` variant:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange { string: "resource_bounds".to_string() });
    }
},
``` [5](#0-4) 

The RPC returns an internal error for a transaction that is valid and was correctly included in a block.

**Step 4 — The TODO confirms hash validation is absent**

The P2P sync client explicitly skips hash revalidation:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [6](#0-5) 

This means the stored hash H1 and the hash H2 that would be computed from the stored transaction body are silently divergent. Any future code path that recomputes the hash from the stored `Transaction` object (e.g., for proving, re-execution, or block hash verification) will produce H2 ≠ H1.

---

### Impact Explanation

**High — Transaction conversion binds the wrong hash/type/executable payload.**

A valid V3 transaction with `AllResources` (zero `l2_gas`, zero `l1_data_gas`) is permanently misrepresented in the storage of any node that synced it via P2P. The stored `ValidResourceBounds` variant is `L1Gas` instead of `AllResources`, which:

1. Causes `starknet_getTransactionByHash` to return an error (authoritative-looking wrong RPC response).
2. Causes any hash recomputation from the stored transaction to produce a different value than the canonical hash stored alongside it.
3. Causes the OS hint processor to reject the transaction with `"Only transactions with 3 resource bounds are supported. Got 1 resource bounds."` if the stored transaction is fed to the prover. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** Any user can submit a V3 transaction via the RPC API with `AllResourceBounds` where `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`. This is a structurally valid `AllResources` transaction. The gateway accepts it, computes the hash as `AllResources`, and propagates it. Every P2P-syncing node then silently corrupts the stored representation.

---

### Recommendation

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion must not infer the variant from the numeric values of the fields. The protobuf message should carry an explicit discriminant, or the conversion should always produce `AllResources` when all three fields are present (regardless of whether they are zero):

```diff
-Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
-    ValidResourceBounds::L1Gas(l1_gas)
-} else {
-    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
-})
+// Always produce AllResources when all three fields are present.
+// L1Gas is only used for pre-0.13.3 transactions that never carry l1_data_gas.
+Ok(if value.l1_data_gas.is_none() && l2_gas.is_zero() {
+    ValidResourceBounds::L1Gas(l1_gas)
+} else {
+    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
+})
``` [8](#0-7) 

The existing TODO comment (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.`) already acknowledges that `l1_data_gas = None` is the correct discriminant for the pre-0.13.3 `L1Gas` case. The fix is to use `None`-ness of `l1_data_gas` as the discriminant rather than the zero-value of the deserialized field.

---

### Proof of Concept

1. Submit a V3 invoke transaction via `starknet_addInvokeTransaction` with:
   - `resource_bounds.l1_gas = { max_amount: N, max_price_per_unit: P }` (nonzero)
   - `resource_bounds.l2_gas = { max_amount: 0, max_price_per_unit: 0 }`
   - `resource_bounds.l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`

2. The gateway computes hash H1 via `get_tip_resource_bounds_hash` with `AllResources` (4 elements: tip, l1_gas, l2_gas(0), l1_data_gas(0)).

3. The transaction is included in a block and propagated via P2P sync.

4. A syncing node deserializes the protobuf: `l1_data_gas` is `Some(zero)`, `l2_gas` is `Some(zero)` → `l1_data_gas.is_zero() && l2_gas.is_zero()` is `true` → stored as `ValidResourceBounds::L1Gas`.

5. Call `starknet_getTransactionByHash(H1)` on the syncing node → `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `Err(OutOfRange)` → RPC returns an internal error.

6. Recompute the hash from the stored transaction using `get_tip_resource_bounds_hash` with `L1Gas` (3 elements: tip, l1_gas, l2_gas(0)) → produces H2 ≠ H1. [9](#0-8) [10](#0-9) [11](#0-10)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
