### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Allowing Bypass of 64-Byte Transaction Forgery Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR contract method despite being deprecated. Any unprivileged caller can invoke it directly to obtain a `true` proof-verification result without supplying the coinbase Merkle proof that `verify_transaction_inclusion_v2` mandates. This is a direct analog to the fee-bypass pattern: a secure path exists (`v2`), but the insecure alternative path (`v1`) was never closed off.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). It enforces an additional coinbase Merkle proof check before delegating to v1: [1](#0-0) 

The coinbase proof check anchors the Merkle tree depth, preventing an attacker from supplying a 64-byte internal node hash as a `tx_id` and having it accepted as a valid leaf.

However, `verify_transaction_inclusion` (v1) is still decorated with `#[pause]` and `pub`, making it a fully reachable NEAR contract method: [2](#0-1) 

The `#[deprecated]` attribute in Rust is a **compile-time lint only**. It emits no runtime restriction. Any NEAR caller — including an adversarial contract — can call `verify_transaction_inclusion` directly, skipping the coinbase proof requirement entirely.

The v1 function performs only a raw Merkle root comparison with no depth or leaf-type validation: [3](#0-2) 

The internal `#[allow(deprecated)]` annotation on the v2→v1 delegation confirms the team is aware v1 is deprecated, yet it remains exposed: [4](#0-3) 

---

### Impact Explanation

An attacker can forge a transaction inclusion proof for a transaction that does not exist in a Bitcoin block. By crafting a 64-byte value that is actually an internal Merkle tree node (two concatenated 32-byte child hashes), the attacker supplies it as `tx_id` to `verify_transaction_inclusion` (v1). The raw `compute_root_from_merkle_proof` call will compute a root that matches the block's real `merkle_root`, causing the function to return `true`.

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a cross-chain action (e.g., releasing bridged assets, confirming an atomic swap, crediting a deposit) will be deceived into treating a non-existent Bitcoin transaction as confirmed.

The corrupted value is the **proof verification result** (`bool`) returned to the consumer contract — the canonical invariant that "a `true` result means a real transaction was included in a known Bitcoin block" is broken.

---

### Likelihood Explanation

- The function is publicly callable by any NEAR account with no role or stake requirement.
- The 64-byte transaction forgery technique is well-documented and has known tooling.
- Consumer contracts that integrated before v2 was introduced, or that call v1 for simplicity, are directly exposed.
- No privileged access, key compromise, or social engineering is required.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` (v1) or restrict it to `pub(crate)` / internal use only, so it is no longer callable as a NEAR contract method. The v2 function already delegates to v1 internally, so no functionality is lost. Alternatively, add an explicit on-chain guard at the top of v1 that panics unconditionally when called as a top-level entry point (e.g., by checking that the caller is the contract itself via `env::predecessor_account_id() == env::current_account_id()`).

---

### Proof of Concept

1. Attacker identifies a real Bitcoin block `B` stored in the contract with known `merkle_root`.
2. Attacker selects any two adjacent 32-byte hashes from the Merkle tree at depth ≥ 1 (internal node `N = SHA256d(left || right)`).
3. Attacker constructs a `ProofArgs` with:
   - `tx_id = N` (the 64-byte internal node's hash, treated as a fake tx hash)
   - `tx_block_blockhash = B`
   - `merkle_proof` = the sibling path from `N` up to the root (one level shorter than a real leaf proof)
   - `tx_index` = the index corresponding to `N`'s position
   - `confirmations = 1`
4. Attacker calls `verify_transaction_inclusion(args)` directly on the NEAR contract.
5. `compute_root_from_merkle_proof(N, index, proof)` reconstructs the correct `merkle_root` because `N` is a genuine internal node.
6. The function returns `true`.
7. Any consumer contract gating on this result treats the forged transaction as confirmed.

The v1 function at lines 288–323 performs no coinbase depth check and no leaf-type validation — the raw Merkle root match is the only gate, and it passes. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
