### Title
Undocumented `DISABLE_OP` Flag in `MEMPOOL_MODE` Causes Consensus/Mempool Divergence for `op_modpow` (Opcode 60) - (File: `src/chia_dialect.rs`)

---

### Summary

`DISABLE_OP` (0x200) is silently included in `MEMPOOL_MODE` and causes opcode 60 (`op_modpow`) to return a hard `Unimplemented` error in mempool validation, while the same opcode executes successfully in consensus mode. This is a flag-wiring error: the check is too strict in mempool mode, blocking a fully implemented and legitimate operator. The result is a consensus/mempool divergence reachable by any attacker-controlled CLVM program.

---

### Finding Description

In `src/chia_dialect.rs`, the `ClvmFlags` bitfield defines `DISABLE_OP = 0x200` at line 56 — the only flag in the entire set with **no documentation comment**. Every other flag (`CANONICAL_INTS`, `NO_UNKNOWN_OPS`, `LIMIT_HEAP`, `RELAXED_BLS`, `LIMIT_SOFTFORK`, `ENABLE_GC`, `ENABLE_KECCAK_OPS_OUTSIDE_GUARD`, `ENABLE_SHA256_TREE`, `ENABLE_SECP_OPS`, `MALACHITE`) carries an explanatory doc comment. `DISABLE_OP` does not. [1](#0-0) 

This flag is unconditionally included in `MEMPOOL_MODE`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)   // ← silently disables op 60
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [2](#0-1) 

Inside `ChiaDialect::op()`, opcode 60 is dispatched as follows:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
``` [3](#0-2) 

When `DISABLE_OP` is absent (consensus mode), opcode 60 resolves to `op_modpow`, a fully implemented operator with its own test vectors, fuzzer target, and benchmark suite. [4](#0-3) 

The divergence is structural: the same CLVM bytes containing opcode 60 produce `Unimplemented` in mempool validation and a valid `Reduction` in consensus execution.

---

### Impact Explanation

Any CLVM program that invokes opcode 60 (`op_modpow`) will be **rejected by the mempool** but **accepted by consensus**. This breaks the fundamental invariant that the mempool must not reject programs that consensus accepts.

Concrete consequences:
1. **Mempool bypass by block producers**: A farmer/block producer can include a spend bundle containing `op_modpow` directly in a block without routing it through the mempool. The spend bundle passes consensus validation but would have been rejected by every full node's mempool. This allows a block producer to include transactions that bypass all mempool-level checks (fee rate enforcement, spend bundle limits, etc.).
2. **Denial of legitimate use**: Any smart coin whose puzzle uses `op_modpow` cannot be spent through the normal transaction submission path, even though the spend is valid on-chain.

The corrupted result is concrete: `EvalErr::Unimplemented(o)` is returned in mempool mode for a program that returns a valid `Reduction` in consensus mode. [3](#0-2) 

---

### Likelihood Explanation

The entry path requires only attacker-controlled CLVM bytes — specifically, a program containing opcode 60 as an operator. No special privileges are needed to craft such a program. The divergence is deterministic and reproducible with any valid `modpow` invocation. A block producer exploiting this needs only to be a farmer, which is a realistic role in the Chia network.

---

### Recommendation

1. **Document `DISABLE_OP`**: Add a doc comment explaining why `op_modpow` is disabled in mempool mode. If the intent is to prevent expensive modpow computations in the mempool (DoS protection), this should be stated explicitly.
2. **Evaluate whether the flag belongs in `MEMPOOL_MODE`**: If `op_modpow` is a valid consensus operator, it should either be allowed in the mempool (with appropriate cost enforcement) or the divergence should be explicitly acknowledged and documented as an intentional policy.
3. **Rename the flag**: `DISABLE_OP` is dangerously generic. If the flag is intentional, rename it to something like `DISABLE_MODPOW` to make the scope of its effect clear and prevent future misuse.

---

### Proof of Concept

```
; CLVM program: (modpow base exp mod)
; opcode 60 = modpow
; Craft a spend bundle whose puzzle evaluates (60 base exp mod)

; In consensus mode (DISABLE_OP not set):
;   → op_modpow executes → returns valid Reduction

; In mempool mode (MEMPOOL_MODE includes DISABLE_OP):
;   → Err(EvalErr::Unimplemented(op_60)) is returned
;   → spend bundle is rejected by every full node's mempool

; A farmer can include this spend bundle directly in a block.
; Consensus nodes accept it. Mempool nodes never saw it.
```

The divergence is confirmed by the dispatch table at `src/chia_dialect.rs` lines 239–244 and the `MEMPOOL_MODE` constant at lines 72–76. The `op_modpow` implementation at `src/more_ops.rs` lines 1250–1284 is fully functional and has no consensus-mode guard. [5](#0-4) [3](#0-2) [4](#0-3)

pub fn op_modpow(a: &mut Allocator, input: NodePtr, max_cost: Cost, flags: ClvmFlags) -> Response {
    if flags.contains(ClvmFlags::MALACHITE) {
        return op_modpow_malachite(a, input, max_cost, flags);
    }
    let [base, exponent, modulus] = get_args::<3>(a, input, "modpow")?;

    let mut cost = MODPOW_BASE_COST;
    let (base, bsize) = int_atom(a, base, "modpow")?;
    cost += bsize as Cost * MODPOW_COST_PER_BYTE_BASE_VALUE;
    let (exponent, esize) = int_atom(a, exponent, "modpow")?;
    cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;
    check_cost(cost, max_cost)?;
    let (modulus, msize) = int_atom(a, modulus, "modpow")?;
    cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;
    check_cost(cost, max_cost)?;

    if bsize > 256 || esize > 256 || msize > 256 {
        return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
    }

    if exponent.sign() == Sign::Minus {
        return Err(EvalErr::InvalidOpArg(
            input,
            "ModPow with Negative Exponent".to_string(),
        ));
    }

    if modulus.sign() == Sign::NoSign {
        return Err(EvalErr::DivisionByZero(input));
    }

    let ret = base.modpow(&exponent, &modulus);
    let ret = a.new_number(ret)?;
    Ok(malloc_cost(a, cost, ret))
}
