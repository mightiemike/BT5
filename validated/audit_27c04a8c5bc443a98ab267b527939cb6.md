### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `state.openInterest == 0` Permanently Blocks Bankrupt Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when insurance cannot fully cover a negative `vQuoteBalance`. If `state.openInterest == 0` at the time of finalization, `MathSD21x18.div` reverts with `ERR_DIV_BY_ZERO`, permanently blocking the liquidation finalization path for the bankrupt subaccount.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is partially applied, the remaining negative `vQuoteBalance` is socialized by computing a per-share funding adjustment: [1](#0-0) 

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← reverts if == 0
    );
```

`MathSD21x18.div` explicitly guards against zero denominators: [2](#0-1) 

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
```

`state.openInterest` is the protocol-wide sum of `|balance.amount|` across all subaccounts for a given product: [3](#0-2) 

The finalization path (`_finalizeSubaccount`) requires that the bankrupt subaccount has `balance.amount == 0` for every perp product before `socializeSubaccount` is called: [4](#0-3) 

Because the bankrupt subaccount contributes 0 to `openInterest`, if **all other subaccounts** have also closed their positions in that product, `state.openInterest == 0` is a valid on-chain state. A subaccount can simultaneously hold `balance.amount == 0` and `balance.vQuoteBalance < 0` — this is the normal result of closing a position at a realized loss.

The call chain is:

```
Endpoint → ClearinghouseLiq.liquidateSubaccountImpl (productId == type(uint32).max)
  → _finalizeSubaccount
    → perpEngine.socializeSubaccount   ← reverts with ERR_DIV_BY_ZERO
``` [5](#0-4) 

---

### Impact Explanation

The revert is unconditional and there is no recovery path: the state that triggers it (`openInterest == 0`, `vQuoteBalance < 0`, insurance insufficient) cannot be changed by any other protocol action once all positions are closed. The bankrupt subaccount is permanently stuck — its negative `vQuoteBalance` can never be cleared, the finalization transaction always reverts, and the unresolved liability remains in the protocol's accounting indefinitely. This constitutes an accounting failure in the liquidation path that leaves the protocol with an unresolvable bad-debt entry.

---

### Likelihood Explanation

The scenario is reachable without any privileged access:

1. A product has low participation — e.g., a single trader opens and closes a position at a loss, leaving `balance.amount == 0`, `vQuoteBalance < 0`, and `state.openInterest == 0`.
2. The trader's account becomes insolvent (e.g., due to losses across other products).
3. A liquidator calls `liquidateSubaccountImpl` with `productId == type(uint32).max` to finalize.
4. `socializeSubaccount` hits the division by zero and reverts.

This is more likely on newly listed or low-liquidity products where open interest can naturally reach zero.

---

### Recommendation

Add a guard before the division: if `state.openInterest == 0`, there is no one to socialize the loss to. The protocol should either absorb the loss directly (e.g., write it off against a reserve) or skip socialization for that product:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the loss
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

The exact write-off policy (e.g., charge against insurance, emit an event, etc.) should be decided by the protocol, but the division must be guarded.

---

### Proof of Concept

```solidity
// 1. Deploy protocol with one perp product (productId = 2)
// 2. Alice opens a long position, then closes it at a loss:
//    balance.amount = 0, balance.vQuoteBalance = -100e18
//    state.openInterest = 0 (no other participants)
// 3. Alice's account becomes insolvent (e.g., spot liabilities)
// 4. Bob (liquidator) calls:
perpEngine.liquidateSubaccountImpl(LiquidateSubaccount({
    sender: bob,
    liquidatee: alice,
    productId: type(uint32).max,  // triggers _finalizeSubaccount
    amount: 0,
    isEncodedSpread: false,
    nonce: ...
}));
// 5. Execution reaches socializeSubaccount → div(vQuoteBalance, 0) → revert "DBZ"
// 6. No alternative finalization path exists; alice's account is permanently stuck.
```
