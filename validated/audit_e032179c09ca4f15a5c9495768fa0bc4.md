### Title
Unbounded `while` loop in `reorg_chain` exhausts gas on long forks, permanently blocking canonical-chain advancement — (File: `contract/src/lib.rs`)

---

### Summary

The `reorg_chain` function contains two unbounded loops whose iteration count is proportional to the length of the fork being promoted. Because fork blocks accumulate across many separate `submit_blocks` calls, the reorg loop can be arbitrarily long by the time it is triggered. When the loop exhausts NEAR's gas limit the transaction is rolled back, the canonical chain is permanently stuck at the pre-reorg state, and `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` continue to attest transactions from the superseded chain as valid.

---

### Finding Description

`reorg_chain` is invoked from `submit_block_header_inner` whenever a fork's cumulative chainwork exceeds the current main-chain tip's chainwork. [1](#0-0) 

It contains two loops with no iteration cap or gas-budget check:

**Loop 1 — `for` over excess main-chain blocks:** [2](#0-1) 

Iterates `last_main_chain_block_height − fork_tip_height` times, removing one storage entry per iteration.

**Loop 2 — `while` walking the entire fork back to the fork point:** [3](#0-2) 

Iterates once per fork block, performing **four to five storage operations** per step: two `insert` calls into `mainchain_height_to_header` and `mainchain_header_to_height`, one conditional `remove_block_header` (which itself removes from two maps), and one `headers_pool.get` to advance the cursor. [4](#0-3) 

Fork blocks are stored individually across many prior `submit_blocks` calls via `store_fork_header`: [5](#0-4) 

There is no mechanism that limits how many fork blocks can accumulate before a reorg is triggered. The existing `batch_size` guard in `run_mainchain_gc` does **not** protect `reorg_chain`: [6](#0-5) 

---

### Impact Explanation

When the `while` loop exhausts NEAR's gas limit the entire `submit_blocks` transaction is rolled back atomically. The fork blocks remain in `headers_pool` but `mainchain_tip_blockhash` is never updated. From that point forward:

- Every subsequent attempt to submit the same reorg-triggering block will also exhaust gas and roll back, so the canonical chain is permanently frozen at the pre-reorg height.
- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` resolve block membership via `mainchain_header_to_height`: [7](#0-6) 

  Because that map is never updated, both functions continue to return `true` for transactions that belong to the now-superseded chain. Any bridge or application consuming these results will accept double-spent or reorganized-out transactions as confirmed.

---

### Likelihood Explanation

Fork blocks are submitted by the relayer one batch at a time over many transactions. A fork of N blocks requires N prior `submit_blocks` calls, each of which succeeds individually (each stores one block in `headers_pool`). The gas-exhausting reorg is triggered only on the final block that tips the chainwork balance. On chains with low difficulty (testnet, Dogecoin, Zcash testnet) or under a deliberate selfish-mining scenario, forks of dozens to hundreds of blocks are realistic. The relayer is the production entry point; no leaked key or social engineering is required.

---

### Recommendation

Apply the same batched-iteration pattern already used in `run_mainchain_gc`: accept a `reorg_batch_size` parameter (or reuse the existing `batch_size` from `submit_blocks`), process at most that many fork blocks per call, and persist a cursor in contract state so subsequent calls resume where the previous one stopped. Alternatively, enforce a protocol-level cap on the maximum fork depth the contract will accept, rejecting any fork block whose height is more than `MAX_REORG_DEPTH` below the current main-chain tip before it is stored.

---

### Proof of Concept

1. Relayer submits N fork blocks (e.g., N = 500) via repeated `submit_blocks` calls, each succeeding individually. Each call stores one block in `headers_pool` via `store_fork_header`.
2. Relayer submits one final fork block whose cumulative `chain_work` exceeds `mainchain_tip_blockhash.chain_work`.
3. `submit_block_header_inner` detects `current_header.chain_work > total_main_chain_chainwork` and calls `reorg_chain`.
4. The `while` loop at line 616 iterates 500 times, each performing 4–5 storage writes/reads. The transaction exhausts NEAR's ~300 TGas limit and is rolled back.
5. `mainchain_tip_blockhash` remains at the pre-reorg value. Every future attempt to submit the same reorg block repeats step 4.
6. A consumer calls `verify_transaction_inclusion` with a transaction from the old (now-invalid) chain; the function returns `true` because `mainchain_header_to_height` still maps the old block hash to a height. [8](#0-7)

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
