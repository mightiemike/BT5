### Title
Spread Health Adjustment Uses Hardcoded Long Weight for Short Leg, Inflating Subaccount Health — (`core/contracts/BaseEngine.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`getCoreRisk` in `BaseEngine.sol` always returns the **long weight** for any position by hardcoding `amount=1` in the `_getWeightX18` call. The spread health adjustment in `Clearinghouse.getHealth` then uses this `longWeight` field to compute both `existingWeight` and `spreadWeight` for both legs of the spread — including the short leg — producing a mixed-unit average analogous to the VADER oracle bug. The result is an inflated health score for any subaccount holding a spread (long spot + short perp), allowing excess collateral withdrawal and liquidation avoidance.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` hardcodes `amount=1`:**

```solidity
// BaseEngine.sol L179-192
function getCoreRisk(...) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return IProductEngine.CoreRisk(
        amount,
        risk.priceX18,
        RiskHelper._getWeightX18(risk, 1, healthType)  // ← always 1, always long weight
    );
}
```

`CoreRisk.longWeight` is therefore always `longWeightInitialX18` (or maintenance), regardless of whether the actual position is long or short.

**Propagation — `Clearinghouse.getHealth` spread adjustment:**

```solidity
// Clearinghouse.sol L125-135
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;          // ← both are long weights
int128 spreadWeight = RiskHelper._getSpreadWeightX18(
    perpCoreRisk, spotCoreRisk, healthType  // ← also uses longWeight internally
);
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
```

For a canonical spread (long spot, short perp), the **base health** computed in `_calculateProductHealth` uses:
- Spot leg: `amount > 0` → `longWeightInitialX18` (e.g. 0.90)
- Perp leg: `amount < 0` → `shortWeightInitialX18` (e.g. 1.10)

But `existingWeight` uses `(0.90 + 0.90) / 2 = 0.90` instead of the correct `(0.90 + 1.10) / 2 = 1.00`.

`_getSpreadWeightX18` also uses `perpCoreRisk.longWeight` (0.90) when `spotCoreRisk.amount > 0`:

```solidity
// RiskHelper.sol L66-67
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
    // = 1 - (1 - 0.90)/5 = 0.98
    // correct would be: 1 - (1 - 1.10)/5 = 1.02, capped at 0.99
}
```

**Numerical impact:**

| | Wrong (longWeight for perp) | Correct (shortWeight for perp) |
|---|---|---|
| `existingWeight` | 0.90 | 1.00 |
| `spreadWeight` | 0.98 | 0.99 (capped) |
| Adjustment multiplier | +0.08 | −0.01 |
| Direction | **Positive bonus** | Small penalty |

The health is inflated by approximately `0.09 × basisAmount × (spotPrice + perpPrice)` relative to the correct value.

---

### Impact Explanation

Any subaccount holding a spread (long spot + short perp for the same underlying) receives an inflated `getHealth` result. This directly affects:

1. **`withdrawCollateral`** — `require(getHealth(sender, INITIAL) >= 0)` passes when it should fail, allowing excess collateral withdrawal.
2. **`isUnderMaintenance` / `isUnderInitial`** in `ClearinghouseLiq` — liquidation eligibility checks use `getHealth`, so an undercollateralized spread position evades liquidation.
3. **`mintNlp` / `burnNlp`** — health checks after NLP operations are also bypassed.

The magnitude scales with position size: a large spread position can produce a health inflation of `0.09 × notional`, which at $1M notional is $90,000 of phantom health credit.

---

### Likelihood Explanation

The trigger requires only that a subaccount hold a registered spread pair (long spot + short perp). Spread pairs are a first-class protocol feature encoded in the `spreads` storage variable. Any unprivileged trader can open such a position through normal `Endpoint` transaction flow. No special permissions, governance actions, or admin keys are required. The bug is active on every `getHealth` call for any spread holder.

---

### Recommendation

In `BaseEngine.getCoreRisk`, pass the actual `amount` (not hardcoded `1`) to `_getWeightX18` so that `CoreRisk.longWeight` reflects the direction-correct weight:

```solidity
RiskHelper._getWeightX18(risk, amount, healthType)  // use actual amount
```

Additionally, rename the `CoreRisk.longWeight` field to `weight` to reflect that it now carries the direction-adjusted value, and update `_getSpreadWeightX18` to accept the actual short-leg weight separately rather than reading `longWeight` from the `CoreRisk` struct.

---

### Proof of Concept

1. Trader opens: long 100 ETH spot (productId = spotId), short 100 ETH perp (productId = perpId), where (spotId, perpId) is a registered spread pair.
2. Assume `longWeightInitialX18 = 0.9e18`, `shortWeightInitialX18 = 1.1e18`, ETH price = $2000.
3. Base health from `_calculateProductHealth`:
   - Spot: `+100 × 0.9 × 2000 = +$180,000`
   - Perp: `-100 × 1.1 × 2000 = -$220,000`
   - Net base: `-$40,000`
4. Spread adjustment (buggy): `existingWeight = 0.9`, `spreadWeight = 0.98`
   - `health += 100 × (2000 + 2000) × (0.98 - 0.90) = +$32,000`
   - Total health: `-$40,000 + $32,000 = -$8,000` (still negative, but less so)
5. Spread adjustment (correct): `existingWeight = 1.0`, `spreadWeight = 0.99`
   - `health += 100 × 4000 × (0.99 - 1.00) = -$4,000`
   - Total health: `-$40,000 - $4,000 = -$44,000`
6. The buggy path reports `-$8,000` vs the correct `-$44,000` — a $36,000 phantom health inflation.
7. Trader calls `withdrawCollateral` for collateral that would bring health to `-$7,999`; the check `getHealth >= 0` fails in both cases here, but with a smaller spread the trader can withdraw collateral that should be blocked.
8. More critically, a trader near the liquidation boundary avoids `isUnderMaintenance` returning `true` due to the inflated health, preventing the liquidator from acting. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
```
