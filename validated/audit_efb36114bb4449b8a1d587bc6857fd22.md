Audit Report

## Title
Missing Deadline Guard in `MetricOmmPoolLiquidityAdder` Allows Stale-Price Liquidity Deposits — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` and `addLiquidityWeighted` accept no `deadline` parameter and perform no timestamp check before executing liquidity deposits. `MetricOmmSimpleRouter` calls `_checkDeadline` at the entry of every swap function, but the liquidity-adder path has no equivalent guard. A transaction delayed in the mempool can execute after the oracle price has moved, forcing the LP to deposit at a price they never accepted and incurring immediate impermanent loss bounded only by their own `maxAmountToken0`/`maxAmountToken1` caps.

## Finding Description
`MetricOmmSwapRouterBase._checkDeadline` (line 91–94) is called at the top of every `MetricOmmSimpleRouter` entry point (`exactInputSingle` line 68, `exactInput` line 93, `exactOutputSingle` line 131, `exactOutput` line 155). None of the four `MetricOmmPoolLiquidityAdder` public entry points (`addLiquidityExactShares` with owner, `addLiquidityExactShares` without owner, `addLiquidityWeighted` with owner, `addLiquidityWeighted` without owner) accept or check a deadline.

**`addLiquidityExactShares` path (lines 56–81):** No cursor-position check and no deadline. The LP specifies exact per-bin shares; the token amounts required are computed from the pool state at execution time. If the transaction is delayed and the active bin shifts, the token composition changes. The callback enforces `amount0Delta <= max0 && amount1Delta <= max1` (line 165), but any amount within those caps is accepted regardless of how stale the price is.

**`addLiquidityWeighted` path (lines 88–149):** `_validateBinAndBinPosition` (lines 263–286) checks that the current cursor falls within `[minimalCurBin, maximalCurBin]`. This provides partial price-range protection — if the price moves outside the specified cursor range the transaction reverts — but it does not protect against execution at any price within that range after an arbitrary delay. An LP who specified a wide cursor range (or whose transaction executes just before the cursor exits the range) still deposits at a stale price.

**Exploit flow:**
1. LP calls `addLiquidityExactShares` or `addLiquidityWeighted` with current-market parameters and reasonable max caps.
2. Transaction sits in the mempool (network congestion, gas price war, or deliberate withholding by a searcher).
3. Oracle price moves; the pool cursor shifts within the LP's cursor bounds (or any shift for `addLiquidityExactShares`).
4. Transaction executes: the pool computes token amounts from the new price, the callback pulls up to the caps, and the LP receives shares priced at the stale rate.
5. LP immediately holds a position with impermanent loss relative to the price at which they intended to deposit.

## Impact Explanation
The LP suffers direct, quantifiable loss of deposited principal: they receive LP shares valued at the stale price while having paid tokens at current market value (or vice versa). The loss is bounded by `maxAmountToken0`/`maxAmountToken1` but can be the full cap amount in an extreme price move. This constitutes direct loss of user principal above Sherlock medium thresholds, matching the allowed impact "Critical/High/Medium direct loss of user principal."

## Likelihood Explanation
No special privilege is required; any LP calling these public functions is exposed. The condition (mempool delay + price movement) is routine on congested networks and is actively exploited by MEV searchers who can withhold or delay transactions. The attack is repeatable and requires no on-chain setup beyond submitting a normal liquidity transaction.

## Recommendation
Add a `uint256 deadline` parameter to all four public entry points and call `_checkDeadline(deadline)` as the first statement, mirroring the pattern already used in `MetricOmmSwapRouterBase._checkDeadline`. For `addLiquidityExactShares`, also consider adding an optional cursor-bounds check (already present in `addLiquidityWeighted`) so LPs can bound the price range at which their deposit executes.

## Proof of Concept
1. Deploy a pool and fund an LP account.
2. LP calls `addLiquidityExactShares` with current-price parameters and `maxAmountToken0 = X`, `maxAmountToken1 = Y`.
3. Simulate mempool delay: advance `block.timestamp` by several hours and move the oracle price (shift the pool cursor) within the same Foundry test using `vm.warp` and a price-push call.
4. Execute the LP's original transaction.
5. Assert that the transaction succeeds (no revert), that the LP received shares at the new (stale) price, and that the token amounts pulled differ from what the LP would have deposited at the original price — confirming impermanent loss with no deadline protection.
