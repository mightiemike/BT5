### Title
Fee-on-Transfer Token Integration Bug in `depositCollateralWithReferral` Inflates User Balances — (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` pulls `amount` tokens from the depositor via `handleDepositTransfer`, then enqueues a slow-mode `DepositCollateral` transaction recording the same caller-supplied `amount`. When the sequencer later processes that slow-mode entry, `Clearinghouse.depositCollateral` credits the full `amount` to the subaccount balance — without ever verifying how many tokens were actually received. For any fee-on-transfer collateral token, the contract holds fewer tokens than it credits, permanently underfunding the collateral pool.

---

### Finding Description

In `Endpoint.depositCollateralWithReferral`, the token pull and the slow-mode record use the same caller-supplied `amount`:

```
handleDepositTransfer(token, msg.sender, uint256(amount));   // actual receipt may be < amount
...
DepositCollateral({ sender: subaccount, productId: productId, amount: amount })  // original amount recorded
``` [1](#0-0) 

When the slow-mode transaction is executed, `Clearinghouse.depositCollateral` scales and credits `txn.amount` directly:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [2](#0-1) 

No balance snapshot is taken before or after `handleDepositTransfer` to reconcile the actual received amount. For a fee-on-transfer token with a 1 % transfer fee, a deposit of 1 000 units causes the contract to receive 990 units but credit 1 000 units to the subaccount.

The same pattern exists in the `DepositInsurance` slow-mode path inside `EndpointTx.submitSlowModeTransactionImpl`:

```solidity
handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
``` [3](#0-2) 

followed by `clearinghouse.depositInsurance` crediting `txn.amount` to the insurance fund. [4](#0-3) 

---

### Impact Explanation

Every deposit with a fee-on-transfer collateral token inflates the depositor's on-chain balance by the fee amount. The depositor can later withdraw the full credited amount, but the contract only holds the fee-reduced amount. Repeated deposits drain the collateral pool relative to the sum of all credited balances, eventually making the protocol insolvent for that product. This is a concrete, permanent accounting corruption — not rounding drift.

---

### Likelihood Explanation

`depositCollateralWithReferral` is `public` with no caller restriction beyond a sanctions check, so any user can trigger it. The vulnerability activates whenever a fee-on-transfer ERC-20 is registered as a supported collateral product in `SpotEngine`. The protocol imposes no explicit exclusion of such tokens. Several real-world tokens (e.g., STA, PAXG, certain rebasing tokens) carry transfer fees, and the protocol's generic `Config.token` field accepts any ERC-20 address.

---

### Recommendation

Capture the contract's token balance before and after `handleDepositTransfer` and use the delta as the credited amount:

```solidity
uint256 pre  = token.balanceOf(address(this));
handleDepositTransfer(token, msg.sender, uint256(amount));
uint256 post = token.balanceOf(address(this));
uint128 actualReceived = uint128(post - pre);
// use actualReceived in the DepositCollateral slow-mode record
```

This mirrors the mitigation suggested in the reference report (cache the actual post-call balance rather than the pre-call input amount).
