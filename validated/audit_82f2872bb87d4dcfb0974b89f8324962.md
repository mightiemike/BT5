### Title
`op_bls_g1_subtract` and `op_bls_g2_subtract` Return Uninitialized Identity Group Element When Called With Zero Arguments — (File: `src/bls_ops.rs`)

---

### Summary

`op_bls_g1_subtract` and `op_bls_g2_subtract` initialize their accumulators to the group identity element (`G1Element::default()` / `G2Element::default()`) and use an `is_first` sentinel flag to track whether the accumulator has been properly set from a real argument. When either operator is invoked with zero arguments via attacker-controlled CLVM bytes, the loop never executes, `is_first` remains `true`, and the function silently returns the identity (zero) group element with no error and no guard. This is a direct structural analog to the "zero price before initialization" class: the accumulator is the uninitialized state, the loop body is the initializer, and there is no `require`-equivalent check.

---

### Finding Description

In `src/bls_ops.rs`, `op_bls_g1_subtract` is structured as follows:

```rust
let mut total = G1Element::default();   // ← identity/"zero" G1 point
let mut is_first = true;                // ← "not yet initialized" sentinel
while let Some((arg, rest)) = a.next(input) {
    ...
    if is_first {
        total = point;                  // ← first real initialization
    } else {
        total -= &point;
    };
    is_first = false;
}
// ← no check: if is_first == true, total was never initialized
Ok(Reduction(cost + 48 * MALLOC_COST_PER_BYTE, a.new_g1(total)?))
``` [1](#0-0) 

When called with an empty argument list (`input` is `NodePtr::NIL`), `a.next(input)` returns `None` immediately, the loop body never runs, `is_first` stays `true`, and `total` remains `G1Element::default()` — the identity point of the G1 group. The function returns this uninitialized identity element as a valid `G1Element` with no error.

`op_bls_g2_subtract` has the identical pattern:

```rust
let mut total = G2Element::default();
let mut is_first = true;
while let Some((arg, rest)) = a.next(input) { ... }
// no guard on is_first
Ok(Reduction(cost + 96 * MALLOC_COST_PER_BYTE, a.new_g2(total)?))
``` [2](#0-1) 

`op_bls_g2_add` shares the same zero-arg identity-return behavior (no `is_first` flag, but same structural outcome): [3](#0-2) 

Both operators are wired into the live `ChiaDialect` dispatch table as opcodes 49 (`op_bls_g1_subtract`) and 53 (`op_bls_g2_subtract`), reachable from any attacker-supplied CLVM program: [4](#0-3) 

---

### Impact Explanation

The identity element of G1 (resp. G2) is the cryptographic "zero" of the group. Returning it silently for a zero-argument call creates two concrete problems:

1. **Operator semantic mismatch / consensus divergence**: Any CLVM puzzle that uses `g1_subtract` to derive a public key from solution-supplied arguments can be fed an empty argument list by an attacker, yielding the identity G1 point as the "public key." The identity public key in BLS is a degenerate key — it verifies the identity signature — which can bypass downstream signature checks that do not separately validate the key is non-identity.

2. **Incorrect result propagated silently**: Unlike arithmetic operators where returning 0 for no args is mathematically well-defined and expected, the `is_first` sentinel in `op_bls_g1_subtract` and `op_bls_g2_subtract` is explicit evidence that the design intent is "the first argument initializes the accumulator." Returning the pre-initialization value when no argument is ever provided violates this invariant without signaling an error, making the behavior invisible to callers.

The corrupted result is a concrete `NodePtr` holding the 48-byte compressed identity G1 point (or 96-byte G2 identity), returned as a valid operator output with a valid cost, indistinguishable from a legitimately computed point.

---

### Likelihood Explanation

The entry path is direct: an attacker crafts a CLVM program containing the byte `0x31` (opcode 49, `g1_subtract`) applied to an empty argument list `()`. This is valid CLVM syntax, passes all parsing, and reaches `op_bls_g1_subtract` with `input = NodePtr::NIL`. No special privileges, no compromised nodes, and no social engineering are required. Any CLVM puzzle that delegates the argument list to the solution (a common pattern) is vulnerable to this injection.

---

### Recommendation

Add a guard after the loop in both `op_bls_g1_subtract` and `op_bls_g2_subtract` that checks whether `is_first` is still `true`, and return an error if so:

```rust
// after the while loop in op_bls_g1_subtract:
if is_first {
    return Err(EvalErr::InvalidOpArg(
        input,
        "g1_subtract requires at least one argument".to_string(),
    ));
}
```

Apply the same fix to `op_bls_g2_subtract`. For `op_bls_g2_add`, consider whether returning the identity for zero args is intentional (analogous to `(+ )` returning 0) or should also be guarded.

---

### Proof of Concept

The following CLVM program, when run against any environment, triggers the bug and returns the 48-byte compressed identity G1 point as a valid operator result:

```
(g1_subtract)
```

In CLVM bytes: `ff 31 80` (opcode 49 applied to nil).

Calling `run_program` with this program and any environment will return `Ok(Reduction(cost, identity_g1_nodeptr))` instead of an error, where `identity_g1_nodeptr` holds the 48-byte BLS12-381 G1 identity point `0xc0 00 00 ... 00`. A downstream puzzle that feeds this result into `bls_verify` or `bls_pairing_identity` as a public key will operate on the degenerate identity key, potentially bypassing the intended signature check. pub fn op_bls_g1_subtract(
    a: &mut Allocator,
    mut input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_G1_SUBTRACT_BASE_COST;
    check_cost(cost, max_cost)?;
    let mut total = G1Element::default();
    let mut is_first = true;
    while let Some((arg, rest)) = a.next(input) {
        input = rest;
        let point = a.g1(arg)?;
        cost += BLS_G1_SUBTRACT_COST_PER_ARG;
        check_cost(cost, max_cost)?;
        if is_first {
            total = point;
        } else {
            total -= &point;
        };
        is_first = false;
    }
    Ok(Reduction(
        cost + 48 * MALLOC_COST_PER_BYTE,
        a.new_g1(total)?,
    ))
}
