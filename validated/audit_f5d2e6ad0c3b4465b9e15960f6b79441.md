Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool populates with its own `msg.sender` — the router contract address when a user enters through `MetricOmmSimpleRouter`. If the pool admin allowlists the router to enable router-mediated swaps for any allowlisted user, the per-user allowlist is completely bypassed for all users, including non-allowlisted ones. This breaks the core invariant of a curated pool and exposes LP positions to unrestricted trading.

## Finding Description

The call chain is confirmed by the production code:

**Step 1:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making the pool's `msg.sender` equal to the router address. [1](#0-0) 

**Step 2:** `MetricOmmPool.swap` passes its own `msg.sender` (the router) as the first argument to `_beforeSwap`. [2](#0-1) 

**Step 3:** `SwapAllowlistExtension.beforeSwap` receives `sender = router` and checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`, never touching the end user's address. [3](#0-2) 

This creates an irreconcilable dilemma for any pool admin deploying a curated pool with `SwapAllowlistExtension`:

- **Router NOT allowlisted:** All router-mediated swaps revert with `NotAllowedToSwap`, even for users whose individual addresses are allowlisted. The primary periphery path is broken for all allowlisted users.
- **Router IS allowlisted** (the only way to enable router-mediated swaps): The extension sees `sender = router`, which is allowlisted, and passes for every caller regardless of whether their individual address is on the allowlist. Any non-allowlisted user bypasses the gate by calling `router.exactInputSingle(...)`.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the explicit `owner` parameter (the actual depositor identity supplied by the caller), not `sender` (the pool's `msg.sender`), so it does not share this flaw. [4](#0-3) 

## Impact Explanation

A curated pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, institutional partners) can be fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. The non-allowlisted user executes swaps against the pool's liquidity at oracle prices, extracting value from LP positions that were intended to be protected behind the allowlist. This constitutes a direct loss of LP principal and a broken core pool invariant: the allowlist policy is not enforced on the supported periphery path. This matches the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" allowed impacts.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap interface for the protocol. Any user can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` against any pool address. No special privilege, role, or setup is required beyond holding the input tokens. The bypass is triggered by the normal, documented usage of the router on any pool that has `SwapAllowlistExtension` configured with the router allowlisted — which is the only configuration that allows allowlisted users to use the router at all.

## Recommendation

`SwapAllowlistExtension` must gate the economically relevant actor — the end user — not the intermediary router. Two approaches:

1. **Short term:** Pass the original caller identity through `extensionData`. The router should encode `msg.sender` into `extensionData` before calling `pool.swap`, and the extension should decode and check that address. Alternatively, check the `recipient` parameter as a proxy for the economically benefiting party.
2. **Long term:** Redesign the `beforeSwap` hook signature to include an explicit `originator` field that the pool populates from transient storage set at the outermost entry point, so extensions can always gate the true end user regardless of intermediary contracts in the call stack.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Admin must also allowlist the router for router-mediated swaps to work.
extension.setAllowedToSwap(address(pool), allowedUser, true);
extension.setAllowedToSwap(address(pool), address(router), true); // required for router path

// Attack: bannedUser bypasses the allowlist via the router.
vm.prank(bannedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 10_000,
        amountOutMinimum: 0,
        recipient: bannedUser,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Pool passes msg.sender (router) as `sender` to _beforeSwap.
// Extension checks allowedSwapper[pool][router] == true → swap succeeds.
// bannedUser has successfully traded on a curated pool that should have blocked them.
```
