Audit Report

## Title
`executeOracleStopLossDecay` Retroactively Applies New Decay Rate Over Full Elapsed Period, Collapsing Watermarks and Silently Disabling Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

When `executeOracleStopLossDecay` stores a new (higher) `decayPerSecondE8`, it writes no checkpoint to `lastDecayTs` or to the stored watermark values. The next swap's `_checkAndUpdateWatermarks` then applies the new rate over the entire `[lastDecayTs, now]` window, which can span the full timelock period plus any idle time. If the resulting `factor = newRate × dt ≥ 1e8`, `_decayed` returns 0, the watermark collapses, and `_applyWatermark` never detects a breach — permanently disabling the stop-loss guard for that bin.

## Finding Description

`executeOracleStopLossDecay` stores the new rate with no watermark checkpoint:

```solidity
// L139-147
oracleStopLossConfig[pool_].decayPerSecondE8 = decay;   // new rate stored
(sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
// lastDecayTs and highWatermarks[pool_][binIdx] are untouched
``` [1](#0-0) 

On the next swap, `_checkAndUpdateWatermarks` reads the post-change rate and the unmodified `lastDecayTs`:

```solidity
// L267-270
BinHighWatermarks storage hwmS = highWatermarks[pool_][binIdx];
uint256 dt = block.timestamp - hwmS.lastDecayTs;   // full elapsed window
(uint256 hwm0, bool breach0) = _applyWatermark(
    metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
``` [2](#0-1) 

`_decayed` floors at 0 when `factor ≥ 1e8`:

```solidity
// L319-324
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;
``` [3](#0-2) 

When `hwm = 0`, `_applyWatermark` always returns `(metric, false)` because `metric >= 0` is unconditionally true for `uint256`:

```solidity
// L333
if (metric >= hwm) return (metric, false);
``` [4](#0-3) 

The maximum permitted rate is `1e8` per second (`_validateDecay` at L309-311). At that rate, `factor ≥ 1e8` after just **1 second** of elapsed time, instantly zeroing any watermark. [5](#0-4) 

The timelock delays the rate change but does not prevent the retroactive collapse — the collapse occurs on the first swap after `executeOracleStopLossDecay`, not at proposal time. This is a direct bypass of the timelock's intended LP-protection invariant: the pool admin's rate-change power is supposed to be bounded by the timelock, but the retroactive application nullifies that bound.

## Impact Explanation

Direct loss of LP principal. The stop-loss guard's sole purpose is to revert swaps that push per-share bin value below `hwm × (1 − drawdown)`. With the watermark collapsed to zero, `_applyWatermark` never sets `breached = true`, so no swap is ever reverted regardless of how much value is extracted from the bin. Any LP funds in the affected bin are unprotected for all subsequent swaps until a new watermark is manually set via the timelocked `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks` flow.

## Likelihood Explanation

The pool admin is semi-trusted and raising the decay rate is a normal operational action. Any pool that experiences a quiet period (no swaps for hours or days) followed by a decay-rate increase is vulnerable — this is a realistic operational sequence. Even without adversarial intent, a well-meaning admin who raises the rate after a weekend of inactivity will silently disable the guard on the first post-change swap. The maximum allowed rate (`1e8`) collapses any watermark after just 1 second of elapsed time, making the window of vulnerability extremely wide.

## Recommendation

Before storing the new rate in `executeOracleStopLossDecay`, checkpoint all bins with non-zero watermarks: apply `_decayed(hwmS.tokenX, oldRate, block.timestamp - hwmS.lastDecayTs)` and write the result back along with `lastDecayTs = block.timestamp`. This ensures the new rate is only applied to time elapsed **after** the change. Alternatively, store a `rateChangedAt` timestamp alongside the new rate and split `dt` in `_decayed` into two segments: `[lastDecayTs, rateChangedAt]` at the old rate and `[rateChangedAt, now]` at the new rate.

## Proof of Concept

**Setup:** Pool with `drawdownE6 = 100_000` (10%), `decayPerSecondE8 = 58` (~5%/day), `timelock = 1 day`. Watermarks set at `hwm0 = hwm1 = 1_000` at `T0`; `lastDecayTs = T0`.

| Step | Time | Action |
|------|------|--------|
| 1 | T0 | Watermarks set; `lastDecayTs = T0`. |
| 2 | T0 | Admin proposes `decayPerSecondE8 = 580` (10× increase). |
| 3 | T0 + 1 day | Timelock elapses; admin calls `executeOracleStopLossDecay`. New rate stored; no checkpoint written. |
| 4 | T0 + 7 days | First swap arrives. `dt = 604_800 s`. |

**Current code:** `factor = 580 × 604_800 = 350_784_000 ≥ 1e8` → `_decayed` returns `0`. Watermark = **0**. Guard never fires. Swap drains the bin.

**With checkpoint:** Phase 1 (6 days at rate 58): `hwm → 699`. Phase 2 (1 day at rate 580): `hwm → 349`. Floor = `314`. Guard correctly blocks any swap pushing per-share value below 314.
