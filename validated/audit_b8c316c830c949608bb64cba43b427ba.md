### Title
`ENABLE_SECP_OPS` Flag Gate Bypassed via 4-Byte Opcode Dispatch — (`File: src/chia_dialect.rs`)

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two independent code paths. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches the same functions **unconditionally**, with no flag check. An attacker-controlled CLVM program using the 4-byte opcode form executes real secp signature verification regardless of whether the softfork flag is set, bypassing the intended activation gate and producing consensus divergence between nodes with different flag configurations.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method of `ChiaDialect` handles 4-byte opcodes first:

```rust
// lines 157–182
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

The 1-byte opcode path, reached later, correctly gates the same functions:

```rust
// lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is the softfork activation gate for secp operations. `MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`:

```rust
// lines 72–76
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

Consequence table:

| Opcode form | `ENABLE_SECP_OPS` absent | `ENABLE_SECP_OPS` present |
|---|---|---|
| 1-byte (64 / 65) | unknown op / rejected | secp verify executes |
| 4-byte (`0x13d61f00` / `0x1c3a8f00`) | **secp verify executes** | secp verify executes |

A CLVM program using the 4-byte opcode form executes real secp verification on any node, regardless of flag state. The broken invariant is: *secp operators must not execute unless `ENABLE_SECP_OPS` is set*.

### Impact Explanation

**Consensus divergence.** During any window where nodes disagree on whether `ENABLE_SECP_OPS` is active (e.g., during softfork rollout, or when a node is misconfigured), a program using the 4-byte opcode form will:
- Execute and succeed on nodes without `ENABLE_SECP_OPS` (via the 4-byte path).
- Be treated as an unknown no-op on nodes without `ENABLE_SECP_OPS` using the 1-byte form.

This produces split evaluation results for the same CLVM bytes across the network, which is a critical integrity failure in the consensus/state-transition model.

**Mempool inconsistency.** In `MEMPOOL_MODE` (which sets `NO_UNKNOWN_OPS` but not `ENABLE_SECP_OPS`): the 1-byte opcode 64/65 falls through to `unknown_operator`, which returns `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set. But the 4-byte form `0x13d61f00` is explicitly dispatched to `op_secp256k1_verify` and executes. A transaction that should be rejected by the mempool (secp not yet activated) is instead accepted and its signature verified.

### Likelihood Explanation

The 4-byte opcode values are documented in the benchmark tool (`tools/src/bin/benchmark-clvm-cost.rs` lines 830, 839) and in the code comment at line 172–174 of `src/chia_dialect.rs`. Any CLVM program author who reads the source or benchmarks can discover and use these opcodes. No privileged access is required — the attacker only needs to submit a CLVM program (e.g., a spend bundle) using the 4-byte opcode form. The entry path is any call to `run_program` with a `ChiaDialect` that lacks `ENABLE_SECP_OPS`.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    if flags.contains(ClvmFlags::ENABLE_SECP_OPS) {
        let f = match opcode {
            0x13d61f00 => op_secp256k1_verify,
            0x1c3a8f00 => op_secp256r1_verify,
            _ => return unknown_operator(allocator, o, argument_list, flags, max_cost),
        };
        return f(allocator, argument_list, max_cost, flags);
    }
    return unknown_operator(allocator, o, argument_list, flags, max_cost);
}
```

This ensures both dispatch paths enforce the same activation gate.

### Proof of Concept

Construct a CLVM program using opcode `0x13d61f00` with valid secp256k1 arguments. Run it with a `ChiaDialect` that does **not** include `ENABLE_SECP_OPS`:

```rust
let mut allocator = Allocator::new();
// dialect without ENABLE_SECP_OPS
let dialect = ChiaDialect::new(ClvmFlags::empty());

// Build: (0x13d61f00 pubkey msg sig)
// where pubkey/msg/sig are valid secp256k1 inputs
let program = /* ... assemble 4-byte opcode call ... */;
let result = run_program(&mut allocator, &dialect, program, env, Cost::MAX);
// result is Ok(Reduction(1300061, NIL)) — secp verify executed and succeeded
// Expected: unknown operator (no-op) or rejection, since ENABLE_SECP_OPS is absent
```

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
```
