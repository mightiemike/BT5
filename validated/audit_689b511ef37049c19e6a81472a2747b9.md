### Title
SwapAllowlistExtension Bypassed via MetricOmmSimpleRouter — Any User Can Swap on Allowlist-Restricted Pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. When a user swaps through `MetricOmmSimpleRouter`, the pool receives the **router's address** as `sender`, not the user's address. If the pool admin allowlists the router (the only way to support router-mediated swaps), every user on the internet can bypass the per-user allowlist by routing through the public router.

---

### Finding Description

In `MetricOmmPool.swap()`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (or any other router entry point), the router calls `pool.swap()` directly: [2](#0-1) 

So `msg.sender` of `pool.swap()` is the **router contract**, not the user. The pool forwards `router_address` as `sender` to the extension.

`SwapAllowlistExtension.beforeSwap` then evaluates: [3](#0-2) 

The check is `allowedSwapper[pool][router_address]`. The user's identity is completely lost. This creates a binary trap for pool admins:

- **Router NOT allowlisted:** No user can swap through the router (even allowlisted users), breaking the supported periphery path.
- **Router allowlisted:** Every user on the internet can bypass the per-user allowlist by calling through the router.

The same identity substitution occurs in `exactInput` (multi-hop), `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified users, whitelisted market makers, or institutional counterparties) is completely open to any user who routes through `MetricOmmSimpleRouter`. The attacker can trade at oracle-anchored prices on a pool that was never intended to be publicly accessible, extracting value from LP positions that were sized and priced for a restricted counterparty set. This is a direct loss of LP principal and protocol fee revenue, and a broken core pool invariant.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported user-facing swap interface. Any pool admin who deploys a curated pool and wants their allowlisted users to be able to use the standard router **must** allowlist the router address. This is the natural and expected operational setup. The bypass is therefore reachable in every realistic curated-pool deployment that supports router access, triggered by any unprivileged user with no special preconditions.

---

### Recommendation

The extension must gate on the **economic actor** (the end user), not the immediate `pool.swap()` caller. Two viable fixes:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a protocol-level convention.

2. **Check `recipient` instead of `sender` for exact-input swaps, or add a dedicated `originator` field to the extension interface:** The `IMetricOmmExtensions.beforeSwap` signature already receives `recipient`; a separate `originator` field (the true end user) would be the cleanest fix.

3. **Allowlist the router and require the router to forward user identity:** The router could be the only allowlisted address, and the extension could decode the true user from `extensionData` that the router is required to populate with `msg.sender`.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists router: setAllowedToSwap(pool, router, true)
  - Pool admin does NOT allowlist attacker: allowedSwapper[pool][attacker] = false

Attack:
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...) with msg.sender = router
  - pool calls extension.beforeSwap(router, ...) 
  - extension checks allowedSwapper[pool][router] == true  ✓ PASSES
  - attacker's swap executes on the restricted pool

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds — allowlist completely bypassed
```

The existing test suite confirms the allowlist works for direct calls (`test_blocksSwapWhenSwapperNotAllowed`) but contains no test for the router-mediated path, leaving this bypass undetected. [5](#0-4) [6](#0-5)
