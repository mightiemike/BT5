### Title
Spice Epoch Last-Block Endorsement Contributions Systematically Excluded from Reward/Kickout Stats — (File: `chain/chain/src/spice/core.rs`)

---

### Summary

In Spice epochs, the per-epoch endorsement accumulator embedded in the last block's header (`spice_chunk_endorsement_stats`) is computed from `prev_hash` (the second-to-last block of the epoch), permanently excluding the last block's own chunk-certification contributions. Validators whose endorsed chunks are certified in the last block of the epoch receive undercounted endorsement stats, which can flip the binary endorsement-cutoff ratio from 1 to 0, causing incorrect kickouts and reduced rewards.

---

### Finding Description

`spice_chunk_endorsement_stats_for_next_block` is called by the block producer to populate the `spice_chunk_endorsement_stats` field of the last block of an epoch (block N). It calls `compute_spice_endorsement_stats` with `prev_hash = hash(N-1)`: [1](#0-0) 

`compute_spice_endorsement_stats` accumulates contributions from blocks 1 through N-1 by reading the stored running accumulator for block N-1 and adding block N-1's own certification contribution: [2](#0-1) 

Block N (the last block of the epoch) can itself contain `ChunkExecutionResult` statements that certify chunks from earlier in the epoch, crediting validators with endorsements via `credit_chunk_endorsement_stats`: [3](#0-2) 

Because the stats are computed from `prev_hash = hash(N-1)`, block N's own certifications are never folded in. The accumulator resets at epoch boundaries, so these contributions are permanently lost — they cannot be recovered from the next epoch's accumulator.

When `collect_blocks_info` reads `last_block_info.spice_chunk_endorsement_stats()` to build the `spice_endorsement_tracker` for reward/kickout computation, it receives stats that are missing block N's contributions: [4](#0-3) 

This tracker is then passed directly to `compute_validators_to_reward_and_kickout`: [5](#0-4) 

Inside that function, the spice endorsement stats are added to each validator's `chunk_stats.endorsement_stats`: [6](#0-5) 

These stats then flow into `get_validator_online_ratio` and `get_endorsement_ratio`, where the binary `endorsement_cutoff_threshold` is applied: [7](#0-6) 

A validator with endorsement ratio just at or above the threshold (e.g., 80/100 = 80% when threshold is 80%) who endorsed a chunk certified in block N would, without that credit, appear as 79/100 = 79% — below the threshold — and receive a binary ratio of 0 instead of 1.

---

### Impact Explanation

- **Incorrect kickouts**: Validators whose endorsed chunks are certified in the last block of the epoch receive undercounted `produced` and `expected` endorsement values. The binary `endorsement_cutoff_threshold` check in `get_endorsement_ratio` can flip from ratio=1 to ratio=0, triggering `NotEnoughChunkEndorsements` kickout for validators who were actually performing correctly.
- **Reduced rewards**: Even without a kickout, the undercounted endorsement ratio lowers the computed uptime multiplier in `calculate_reward`, reducing the validator's epoch reward proportionally.
- **Systematic, every epoch**: Every Spice epoch's last block can contain chunk certifications. The bug is not conditional on adversarial behavior — it affects all validators whose endorsed chunks happen to be certified in the last block.

---

### Likelihood Explanation

The Spice protocol certifies chunks asynchronously: a block certifies chunks from earlier blocks. The last block of an epoch is a normal block that can contain `ChunkExecutionResult` statements. In a healthy network, certifications are expected to appear in every block, including the last. The probability that at least one certification appears in the last block of any given epoch is high. The impact is bounded by how many certifications appear in that block, but the binary cutoff means even a single missed credit can cause a kickout.

---

### Recommendation

In `spice_chunk_endorsement_stats_for_next_block`, after computing the accumulated stats up to `prev_hash`, also compute and fold in the contribution of block N's own body (the block being produced). Specifically, call `endorsement_contribution_of_block` for block N's hash and add it to the accumulated stats before embedding them in the header. This mirrors the fix in the external report: include the current "scale's" contribution alongside the accumulated prior ones.

---

### Proof of Concept

**Setup**: Spice epoch with blocks 1 … N. Validator V is assigned to endorse a chunk in block N-2. V sends its endorsement. Block N-1 does not certify that chunk (e.g., it certifies a different chunk). Block N (the last block of the epoch) contains a `ChunkExecutionResult` certifying the chunk from block N-2, with V's endorsement matching the certified result.

**Expected**: V's `spice_chunk_endorsement_stats.produced` includes the credit from block N's certification. V's endorsement ratio = (prior_produced + 1) / (prior_expected + 1).

**Actual**: `spice_chunk_endorsement_stats_for_next_block` computes stats from `prev_hash = hash(N-1)`. Block N's certification is not included. V's `spice_chunk_endorsement_stats.produced` = prior_produced, `expected` = prior_expected.

**Divergent value**: If prior_produced / prior_expected < `chunk_validator_only_kickout_threshold` but (prior_produced + 1) / (prior_expected + 1) ≥ threshold, then:
- Correct: `get_endorsement_ratio` returns `(1, 1)` → V is not kicked out
- Actual: `get_endorsement_ratio` returns `(0, 1)` → V's uptime ratio collapses → V is kicked out with `NotEnoughChunkEndorsements`

The exact divergent Borsh-serialized field is `BlockHeaderInnerRestV7::spice_chunk_endorsement_stats[validator_V_id].produced`, which is `N` in the correct computation but `N-1` (or more generally, missing block N's contribution count) in the actual computation. [8](#0-7) [9](#0-8)

    pub fn spice_chunk_endorsement_stats_for_next_block(
        &self,
        prev_header: &BlockHeader,
        height: BlockHeight,
    ) -> Result<Vec<SpiceChunkEndorsementStats>, Error> {
        let last_final_block = prev_header.last_final_block_for_height(height);

            prev_header.hash(),

        )?;
        if !is_last_block_in_epoch {
            return Ok(Vec::new());
        }
        let epoch_id = self.epoch_manager.get_epoch_id_from_prev_block(prev_header.hash())?;
        let mut stats = compute_spice_endorsement_stats(
            &self.chain_store,
            self.epoch_manager.as_ref(),
            &epoch_id,
            prev_header.hash(),
        )?;
        if stats.iter().all(|s| *s == SpiceChunkEndorsementStats::default()) {
            stats.clear();
        }
        Ok(stats)
    }

fn endorsement_contribution_of_block(
    chain_store: &ChainStoreAdapter,
    epoch_manager: &dyn EpochManagerAdapter,
    epoch_info: &EpochInfo,
    credited_block_hash: &CryptoHash,
) -> Result<Vec<SpiceChunkEndorsementStats>, Error> {
    let credited_block = chain_store.get_block(credited_block_hash)?;
    if credited_block.header().is_genesis() || !credited_block.is_spice_block() {
        return Ok(vec![]);
    }
    let prev_uncertified =
        get_uncertified_chunks(chain_store, credited_block.header().prev_hash())?;
    let statements = credited_block.spice_core_statements();
    // (chunk, validator) -> the result they endorsed, from prior blocks and this one.
    // `unchecked_to_stored` is safe here: signatures are verified in
    // `validate_core_statements_in_block` before stats are computed.
    let endorsed_hash: HashMap<(&SpiceChunkId, &AccountId), ChunkExecutionResultHash> =
        prev_uncertified
            .iter()
            .flat_map(|info| {
                info.present_endorsements.iter().map(|(account, endorsement)| {
                    ((&info.chunk_id, account), endorsement.execution_result_hash.clone())
                })
            })
            .chain(statements.iter_endorsements().map(|endorsement| {
                (
                    (endorsement.chunk_id(), endorsement.account_id()),
                    endorsement.unchecked_to_stored().execution_result_hash,
                )
            }))
            .collect();

    let mut stats = vec![SpiceChunkEndorsementStats::default(); epoch_info.validators_iter().len()];
    for (chunk_id, execution_result) in statements.iter_execution_results() {
        let certified_hash = execution_result.compute_hash();
        let chunk_block = epoch_manager.get_block_info(&chunk_id.block_hash)?;
        let assignments = epoch_manager.get_chunk_validator_assignments(
            chunk_block.epoch_id(),
            chunk_id.shard_id,
            chunk_block.height(),
        )?;
        credit_chunk_endorsement_stats(
            &mut stats,
            assignments.assignments().iter().map(|(account_id, _stake)| account_id),
            |account_id| epoch_info.get_validator_id(account_id).copied(),
            |account_id| endorsed_hash.get(&(chunk_id, account_id)) == Some(&certified_hash),
        );
    }
    Ok(stats)
}

pub fn compute_spice_endorsement_stats(
    chain_store: &ChainStoreAdapter,
    epoch_manager: &dyn EpochManagerAdapter,
    epoch_id: &EpochId,
    prev_hash: &CryptoHash,
) -> Result<Vec<SpiceChunkEndorsementStats>, Error> {
    let epoch_info = epoch_manager.get_epoch_info(epoch_id)?;
    let num_validators = epoch_info.validators_iter().len();
    let contribution =
        endorsement_contribution_of_block(chain_store, epoch_manager, &epoch_info, prev_hash)?;

    // Carry the previous block's accumulator only within the same epoch; at the
    // epoch boundary (and genesis) the per-epoch accumulator resets.
    let prev_header = chain_store.get_block_header(prev_hash)?;
    let prev_stats = if !prev_header.is_genesis() && prev_header.epoch_id() == epoch_id {
        Some(get_spice_endorsement_stats(chain_store, prev_hash)?)
    } else {
        None
    };
    fold_endorsement_stats(num_validators, prev_stats.as_deref(), &contribution)
}

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
