### Title
Duplicate Bonus Reward Claim via Repeated Position IDs in `ClaimTierRewards` — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` accepts a caller-supplied list of position IDs and loads each position from the store before entering the processing loop. There is no deduplication of IDs. When the same position ID appears more than once, the position is loaded with its pre-claim state for every occurrence. The processing loop sends bonus rewards and then persists the updated checkpoint, but the in-memory slice entry for the duplicate still holds the old checkpoint. The second iteration therefore re-computes and re-sends the same bonus, draining the `RewardsPoolName` module account.

---

### Finding Description

`ClaimTierRewards` in `msg_server.go` first collects all requested positions into a slice: [1](#0-0) 

Each position is fetched from the store at load time. No uniqueness check is performed on `msg.PositionIds`. The slice is then handed to `claimRewardsAndUpdatesPositions`: [2](#0-1) 

Inside `claimRewardsAndUpdatesPositions`, each element of the slice is processed independently: [3](#0-2) 

For each element the loop:
1. Calls `claimBaseRewards` (sends base rewards via `distributionKeeper.WithdrawDelegationRewards`).
2. Calls `processEventsAndClaimBonus`, which updates the position's `LastBonusAccrual` and `LastEventSeq` **in the in-memory struct** and then sends bonus coins from the module pool to the owner.
3. Calls `setPosition` to persist the updated struct to the store.

The critical ordering inside `processEventsAndClaimBonus`: [4](#0-3) 

`applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime` in memory (line 215), and `SendCoinsFromModuleToAccount` sends the bonus (line 239). The store is only updated when `setPosition` returns in the outer loop.

When position ID `X` appears twice in the input slice, `positions[0]` and `positions[1]` are both loaded from the store **before** the loop runs, so both carry the identical pre-claim `LastBonusAccrual`. After iteration 0 finishes, the store holds the advanced checkpoint. But iteration 1 operates on `positions[1]`, which still has the old `LastBonusAccrual`. `processEventsAndClaimBonus` therefore recomputes the current-segment bonus from the old `LastBonusAccrual` to `blockTime` — the same interval as iteration 0 — and issues a second `SendCoinsFromModuleToAccount` transfer to the owner.

The current-segment bonus path that is unconditionally re-executed: [5](#0-4) 

---

### Impact Explanation

The `RewardsPoolName` module account is drained by a factor equal to the number of duplicate occurrences of a position ID in a single `MsgClaimTierRewards` transaction. An owner with one position can multiply their bonus payout arbitrarily within a single transaction by repeating the same ID `N` times, extracting up to `N × bonus` from the shared pool. This directly corrupts the bonus pool balance and harms all other position holders who depend on it.

Corrupted value: the `RewardsPoolName` module account balance (bond denom coins).

---

### Likelihood Explanation

Any position owner can trigger this with a standard `MsgClaimTierRewards` transaction by supplying a repeated position ID. No special privilege, validator access, or social engineering is required. The message is a normal user-facing transaction reachable via CLI or gRPC. The only constraint is that the pool must have sufficient balance for the first payout; subsequent payouts in the same transaction consume additional pool funds.

---

### Recommendation

Add a deduplication check on `msg.PositionIds` before loading positions, either in `MsgClaimTierRewards.Validate()` or at the top of `ClaimTierRewards`. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, id := range msg.PositionIds {
    if _, dup := seen[id]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", id)
    }
    seen[id] = struct{}{}
}
```

Alternatively, apply the check-effects-interactions pattern strictly: read the position from the store inside the processing loop (after the previous iteration has already called `setPosition`) rather than pre-loading all positions before any processing begins.

---

### Proof of Concept

1. Owner creates a tiered position with ID `42` and accrues bonus rewards over time.
2. Owner submits `MsgClaimTierRewards{ Owner: owner, PositionIds: [42, 42, 42] }`.
3. `ClaimTierRewards` loads position `42` three times from the store into `positions[0]`, `positions[1]`, `positions[2]`, all with the same `LastBonusAccrual = T_old`.
4. Iteration 0: `processEventsAndClaimBonus` computes bonus for `[T_old, blockTime]`, sends `B` coins to owner, sets `positions[0].LastBonusAccrual = blockTime`, calls `setPosition` (store updated).
5. Iteration 1: `processEventsAndClaimBonus` is called with `positions[1]` where `LastBonusAccrual = T_old` (old value). Computes bonus for `[T_old, blockTime]` again, sends another `B` coins to owner.
6. Iteration 2: same as iteration 1, sends a third `B` coins.
7. Owner receives `3B` instead of `B`; the rewards pool is short by `2B`.
