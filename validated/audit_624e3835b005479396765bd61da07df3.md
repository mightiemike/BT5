### Title
Division by Zero in `PerpEngine.socializeSubaccount()` When Last Position Holder Is Liquidated — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount()` divides by `state.openInterest` to spread bad debt across remaining participants. However, `getStateAndBalance()` — called at the top of the loop — internally removes the insolvent subaccount's own contribution from `openInterest` before the division occurs. If the subaccount is the last (or only) open-position holder in a market, `openInterest` reaches zero, causing an unconditional revert with `"DBZ"`. The liquidation path is permanently blocked for that product, leaving bad debt unresolvable.

---

### Finding Description

`PerpEngine.socializeSubaccount()` is called during liquidation to spread an insolvent subaccount's residual negative `vQuoteBalance` across all remaining open-interest holders. The function first calls `getStateAndBalance()`, which internally calls `_updateBalance(state, balance, 0, 0)`.

Inside `_updateBalance`, the very first operation is:

```solidity
state.openInterest -= balance.amount.abs();
``` [1](#0-0) 

This removes the insolvent subaccount's own position from `openInterest` before returning the state to `socializeSubaccount`. If this subaccount is the only remaining participant in the market, `state.openInterest` is now `0`.

Back in `socializeSubaccount`, if insurance is insufficient to cover the full loss, the code reaches:

```solidity
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest
    );
``` [2](#0-1) 

`MathSD21x18.div()` enforces `require(y != 0, ERR_DIV_BY_ZERO)`: [3](#0-2) 

There is no guard on `state.openInterest == 0` before this division. The transaction reverts unconditionally.

Note that `PerpEngineState.updateStates()` already demonstrates the correct pattern — it skips processing when `openInterest == 0`:

```solidity
if (state.openInterest == 0) {
    continue;
}
``` [4](#0-3) 

No equivalent guard exists in `socializeSubaccount`.

---

### Impact Explanation

When the last open-position holder in a perp market becomes insolvent and insurance is insufficient:

1. The liquidation call to `socializeSubaccount` reverts with `"DBZ"`.
2. The subaccount cannot be liquidated — its negative `vQuoteBalance` remains permanently unresolved.
3. The bad debt is stuck in the protocol with no recovery path, since every subsequent liquidation attempt for this subaccount will hit the same revert.
4. The `availableSettle` pool for the product is also corrupted, as the insurance credit applied before the revert is never committed (the entire transaction reverts).

This is a concrete solvency/accounting corruption: bad debt that cannot be cleared and a liquidation path that is permanently bricked for the affected product.

---

### Likelihood Explanation

This is realistic in:

- **New or low-liquidity perp markets** where only one trader has an open position.
- **Late-stage market wind-downs** where most participants have closed positions and one insolvent trader remains.
- **Adversarial scenarios**: a trader opens a position in a thin market, the price moves against them, insurance is depleted, and the protocol cannot liquidate them.

No admin access, governance capture, or sequencer compromise is required. The trigger is ordinary market activity (a single insolvent position in a market with no other open interest).

---

### Recommendation

Add a guard in `socializeSubaccount` before dividing by `state.openInterest`. If `openInterest` is zero, there are no other participants to socialize the loss across; the remaining bad debt should either be absorbed entirely by the insurance fund, written off, or handled via a dedicated zero-OI path:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No other participants; absorb remaining loss or mark as irrecoverable
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

This mirrors the existing guard in `updateStates()` and prevents the permanent DoS on the liquidation path.

---

### Proof of Concept

1. Product `P` is a perp market. Alice is the **only** trader with an open long position of size `S`. `state.openInterest == S`.
2. Price crashes. Alice's `vQuoteBalance` goes deeply negative (e.g., `-1000`).
3. Insurance fund has only `400` available.
4. Clearinghouse calls `PerpEngine.socializeSubaccount(alice, 400)`.
5. Inside the loop for product `P`, `getStateAndBalance(P, alice)` is called.
6. `_updateBalance` executes `state.openInterest -= balance.amount.abs()` → `state.openInterest = S - S = 0`.
7. `balance.vQuoteBalance = -1000`. Insurance covers `400`: `balance.vQuoteBalance = -600`. Still negative.
8. Code reaches `(-(-600)).div(0)` → `MathSD21x18.div` reverts with `"DBZ"`.
9. The entire liquidation transaction reverts. Alice's bad debt of `600` is permanently stuck. No future call can liquidate her. [5](#0-4) [6](#0-5)
