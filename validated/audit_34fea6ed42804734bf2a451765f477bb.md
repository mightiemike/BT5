### Title
Unchecked `bool` Return Value of `equihash::is_valid_solution` Allows Invalid Equihash PoW Acceptance - (`contract/src/zcash.rs`)

### Summary
`equihash::is_valid_solution` returns `Result<bool, _>`. The Zcash `check_pow` implementation calls `.unwrap_or_else(|e| panic)` to handle the error variant, but the extracted `bool` is then silently discarded. A block header carrying a cryptographically invalid Equihash solution that does not trigger a library-level `Err` will return `Ok(false)`, which is dropped without any check, causing the contract to accept the header as if PoW were valid.

### Finding Description

In `contract/src/zcash.rs`, the Equihash validation is:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` (crate version 0.2.2) has the signature `fn is_valid_solution(n, k, input, nonce, solution) -> Result<bool, Error>`. The `.unwrap_or_else` call extracts the `bool` from the `Ok` variant or panics on `Err`. The statement ends with `;`, so the extracted `bool` is **never inspected**. When the library returns `Ok(false)` — the normal path for a well-formed but cryptographically invalid solution — execution continues as if the check passed.

This is called from `check_target` → `check_pow`, which is invoked inside `submit_block_header` for every non-genesis header when `skip_pow_verification` is `false`. [2](#0-1) 

The public entry point is `submit_blocks`, which is callable by any trusted relayer (or, depending on configuration, any account). [3](#0-2) 

### Impact Explanation

An attacker who can call `submit_blocks` can inject Zcash block headers with arbitrary, invalid Equihash solutions. These headers pass all other checks (difficulty bits, timestamps, version) and are stored in `headers_pool` and promoted to the mainchain. Once accepted, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will return `true` for transactions claimed to be in those fabricated blocks, because the Merkle proof is checked only against the `merkle_root` field of the stored (attacker-controlled) header. [4](#0-3) 

This constitutes **proof-verification forgery**: a recipient contract consuming the `true` result of `verify_transaction_inclusion` would incorrectly believe a Bitcoin/Zcash transaction was confirmed.

### Likelihood Explanation

Any account that satisfies the `#[trusted_relayer]` gate (or bypasses it via `Role::UnrestrictedSubmitBlocks`) can trigger this. The `equihash` crate's `is_valid_solution` is documented to return `Ok(false)` for invalid solutions and `Err` only for structural/parameter errors, so `Ok(false)` is the routine code path for a bad solution — not an exceptional one. No special tooling is needed beyond constructing a header with a random 1344-byte solution field.

### Recommendation

Check the returned `bool` explicitly:

```rust
let valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}")));
require!(valid, "Invalid Equihash solution");
```

### Proof of Concept

1. Build a Zcash header that satisfies difficulty (`bits`), timestamp, and version checks but contains a random 1344-byte `solution` field.
2. Call `submit_blocks([header])` on the deployed Zcash contract.
3. `check_pow` calls `equihash::is_valid_solution(...)`, which returns `Ok(false)` because the solution is invalid but structurally well-formed.
4. `.unwrap_or_else` extracts `false`; the trailing `;` discards it; no panic occurs.
5. `submit_block_header_inner` stores the header in `headers_pool` and updates `mainchain_tip_blockhash`.
6. Call `verify_transaction_inclusion_v2` with a `tx_id` and a `merkle_proof` that hashes to the attacker-chosen `merkle_root` in the fake header — the function returns `true`. [5](#0-4) [6](#0-5)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
