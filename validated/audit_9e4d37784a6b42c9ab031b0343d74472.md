### Title
`serialized_length_serde_2026` Returns `Ok` for Blobs That `deserialize_2026` Rejects — Instruction-Stream Validation Gap Enables Parser Confusion - (File: `src/serde_2026/de.rs`)

---

### Summary

`serialized_length_serde_2026` explicitly skips all instruction-stream semantic checks (stack underflow, out-of-bounds atom/pair index references, wrong final stack size). It reads each instruction varint but never evaluates it. `deserialize_2026_body_from_stream` enforces all of these checks. An attacker-controlled serde_2026 blob can be crafted so that `serialized_length_serde_2026` returns `Ok(len)` while `deserialize_2026` returns `Err`. Any caller that uses the length probe as a validity gate — which the function's own docstring encourages — will accept a blob that the actual deserializer rejects, producing a consensus-divergence split between validators.

---

### Finding Description

`serialized_length_serde_2026` in `src/serde_2026/de.rs` walks the wire format to compute a byte count. For the instruction stream it does:

```rust
for _ in 0..instruction_count {
    read_varint(&mut cursor, strict)?;   // reads the varint, discards the value
}
``` [1](#0-0) 

It never inspects the varint value. By contrast, `deserialize_2026_body_from_stream` enforces:

- **Stack underflow** on opcode `1` / `-1` (cons): requires `stack.len() >= 2`
- **Out-of-bounds atom index** on `n >= 2`: `atoms.get(ai).ok_or(SerializationError)?`
- **Out-of-bounds pair index** on `n <= -2`: `pairs.get(pi).ok_or(SerializationError)?`
- **Final stack invariant**: `stack.len() != 1` → `Err` [2](#0-1) 

The docstring for `serialized_length_serde_2026` acknowledges this gap:

> "instruction-stream stack invariants and index validity are still checked at deserialize time — those can't be byte-counted" [3](#0-2) 

But the same docstring also says:

> "so callers can use the length helper to gate before deserializing without observing Ok-then-Err mismatches" [4](#0-3) 

These two statements are contradictory. The function explicitly cannot prevent `Ok`-then-`Err` mismatches for instruction-stream errors, yet the docstring positions it as a pre-deserialization gate.

The test `test_serialized_length_rejects_what_deserialize_rejects` only covers three header-time cases (instruction_count=0, group_length=0, multi_atom_group_count=0) and does not test any instruction-stream rejection: [5](#0-4) 

The fuzz harness in `serde_2026_panic.rs` explicitly handles the divergence by only asserting length agreement when **both** functions succeed — it does not assert that `serialized_length_serde_2026` returning `Ok` implies `deserialize_2026` returning `Ok`: [6](#0-5) 

---

### Impact Explanation

A caller that uses `serialized_length_serde_2026` as a validity gate — as the docstring encourages — will accept blobs that `deserialize_2026` rejects. On the Chia blockchain, if one node pre-filters with the length probe and another validates with full deserialization, they will reach different conclusions about whether a serde_2026-encoded CLVM program is valid. This is a **consensus divergence** vector: a crafted blob can be accepted by one validator and rejected by another, splitting the network's view of a transaction or block.

The corrupted result is concrete: `serialized_length_serde_2026` returns a valid `u64` byte count; `deserialize_2026` returns `Err(SerializationError)`. The `NodePtr` that the deserializer would have produced is never created, but the caller that trusted the length probe has already committed to treating the blob as valid.

---

### Likelihood Explanation

The function is part of the public API exported from `src/serde_2026/mod.rs`: [7](#0-6) 

Its docstring actively encourages use as a pre-deserialization gate. Any downstream consumer of the `clvmr` crate that follows the documented pattern will exhibit the divergence. The trigger requires only a well-formed atom table followed by a syntactically valid but semantically invalid instruction stream (e.g., a cons opcode with only one item on the stack), which is trivially constructable.

---

### Recommendation

`serialized_length_serde_2026` must either:

1. **Validate instruction-stream semantics** — simulate the stack machine (tracking stack depth and atom/pair index bounds) without allocating CLVM nodes, so that `Ok(len)` truly implies `deserialize_2026` will succeed on the same blob; or
2. **Remove the misleading docstring claim** — replace "so callers can use the length helper to gate before deserializing without observing Ok-then-Err mismatches" with an explicit warning that instruction-stream errors will still cause `deserialize_2026` to fail after `serialized_length_serde_2026` returns `Ok`, and that the function must not be used as a standalone validity gate.

The test `test_serialized_length_rejects_what_deserialize_rejects` must be extended to cover instruction-stream rejection cases (stack underflow, out-of-bounds atom index, out-of-bounds pair index, wrong final stack size).

### Proof of Concept

Construct the following attacker-controlled serde_2026 blob (body only, before prepending the magic prefix):

```
group_count    = 0x01          (1 group)
atom_len       = 0x01          (1 byte per atom, positive varint → single-atom group)
atom_data      = 0x41          ('A', atom index 0)
instruction_count = 0x02       (2 instructions)
instruction[0] = 0x02          (push atom at index 0 → stack: [A])
instruction[1] = 0x01          (cons: requires stack.len() >= 2, but stack has only 1 item)
```

Full blob with magic prefix `fd ff 32 30 32 36`:

```
fd ff 32 30 32 36   ← magic prefix
01                  ← group_count = 1
01                  ← atom_len = 1
41                  ← atom bytes ('A')
02                  ← instruction_count = 2
02                  ← instruction: push atom[0]
01                  ← instruction: cons (stack underflow)
```

```rust

pub fn serialized_length_serde_2026(buf: &[u8], max_atom_len: usize, strict: bool) -> Result<u64> {
    if !buf.starts_with(&SERDE_2026_MAGIC_PREFIX) {
        return Err(EvalErr::SerializationError);
    }

    let data = &buf[SERDE_2026_MAGIC_PREFIX.len()..];
    let mut cursor = Cursor::new(data);

    let group_count = checked_usize(read_varint(&mut cursor, strict)?)?;
    for _ in 0..group_count {
        let length_val = read_varint(&mut cursor, strict)?;
        let skip = if length_val < 0 {
            if length_val == i64::MIN {
                return Err(EvalErr::SerializationError);
            }
            let atom_len = checked_bounded_usize(-length_val, max_atom_len)?;
            let count = checked_usize(read_varint(&mut cursor, strict)?)?;
            if atom_len == 0 || count == 0 {
                return Err(EvalErr::SerializationError);
            }
            (atom_len as u64)
                .checked_mul(count as u64)
                .ok_or(EvalErr::SerializationError)?
        } else {
            let atom_len = checked_bounded_usize(length_val, max_atom_len)?;
            if atom_len == 0 {
                return Err(EvalErr::SerializationError);
            }
            atom_len as u64
        };
        let new_pos = cursor
            .position()
            .checked_add(skip)
            .ok_or(EvalErr::SerializationError)?;
        if new_pos > data.len() as u64 {
            return Err(EvalErr::SerializationError);
        }
        cursor.set_position(new_pos);
    }

    let instruction_count = checked_usize(read_varint(&mut cursor, strict)?)?;
    // Mirror `deserialize_2026_body_from_stream`: instruction_count == 0
    // leaves the stack empty and is rejected there, so reject it here too.
    if instruction_count == 0 {
        return Err(EvalErr::SerializationError);
    }
    for _ in 0..instruction_count {
        read_varint(&mut cursor, strict)?;
    }

    Ok(SERDE_2026_MAGIC_PREFIX.len() as u64 + cursor.position())
}
