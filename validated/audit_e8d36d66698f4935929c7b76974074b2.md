### Title
Unprivileged Caller Can Permanently Delete Canonical Mainchain Headers via `run_mainchain_gc` — (`contract/src/lib.rs`)

### Summary

`BtcLightClient::run_mainchain_gc` is a public, state-mutating function that permanently deletes mainchain block headers and advances `mainchain_initial_blockhash`. Its only attribute, `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, restricts access only when the contract is **paused**; it imposes no caller restriction during normal operation. Any unprivileged NEAR account can call it with `batch_size = u64::MAX` to aggressively prune the canonical chain, making historical SPV proofs permanently unverifiable and corrupting the contract's canonical-chain invariant.

### Finding Description

The function is declared as:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [1](#0-0) 

The `Role::UnrestrictedRunGC` role comment reads: *"Allows to use `run_mainchain_gc` API on a **paused** contract."* [2](#0-1) 

This means the role is only required when the contract is paused. When the contract is running normally (the default production state), the `#[pause]` guard is a no-op and **any NEAR account** can call `run_mainchain_gc` with an arbitrary `batch_size`.

Inside the function, for every height in `start_removal_height..end_removal_height`, the contract calls `remove_block_header` (which deletes from both `mainchain_header_to_height` and `headers_pool`) and removes the entry from `mainchain_height_to_header`. It then advances `mainchain_initial_blockhash` to the new oldest block: [3](#0-2) 

These deletions are **permanent and irreversible**. Once a header is removed from `headers_pool` and `mainchain_header_to_height`, it cannot be re-submitted (the relayer would need to re-submit it as a new block, but the contract would reject it because its `prev_block_hash` points to a block below the new `mainchain_initial_blockhash`).

The analog to the GachaToken `burn` vulnerability is exact:
- GachaToken: `burn` calls `_burn(address(0), from, id, amount)` — the zero-address `by` bypasses the authorization check, letting anyone destroy tokens.
- BTC Light Client: `run_mainchain_gc` has no caller check during normal operation — the `#[pause]` attribute only guards against calls when paused, letting anyone destroy canonical chain state.

In both cases, a function that should require authorization to perform a destructive, irreversible state mutation is callable by any unprivileged actor.

### Impact Explanation

**Impact: High**

1. **SPV proof integrity broken**: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height`. If the block was GC'd, the call panics with `"block does not belong to the current main chain"`. Any downstream NEAR contract that relies on the light client to verify a Bitcoin transaction inclusion will receive a permanent false negative for any block older than the new `mainchain_initial_blockhash`. [4](#0-3) 

2. **Reorg safety broken**: The CLAUDE.md explicitly documents: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor."* [5](#0-4) 

An attacker who aggressively GCs the chain before a legitimate relayer submits a fork will cause the contract to panic on the next reorg, permanently bricking chain-tip advancement.

3. **Canonical chain state corrupted**: `mainchain_initial_blockhash` is advanced to an attacker-chosen point, permanently altering the contract's view of what the oldest verifiable block is. [6](#0-5) 

### Likelihood Explanation

**Likelihood: High**

- Zero preconditions: the attacker needs no role, no stake, no deposit, and no special knowledge beyond the contract's public ABI.
- The call is cheap: `run_mainchain_gc(u64::MAX)` is bounded internally by `total_amount_to_remove`, so it removes at most `amount_of_headers_we_store - gc_threshold` headers per call. The attacker can call it repeatedly (once per NEAR block) to keep the chain perpetually trimmed to exactly `gc_threshold` blocks, ensuring any proof for a block older than `gc_threshold` blocks always fails.
- The contract is deployed on NEAR mainnet and its ABI is public.

### Recommendation

Add an explicit caller authorization check to `run_mainchain_gc`. The simplest fix is to restrict it to the same roles that can call `submit_blocks`, or introduce a dedicated `GCManager` role:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    // Add: require caller holds an authorized role
    require!(
        self.acl_has_role(Role::UnrestrictedRunGC, &env::predecessor_account_id())
            || self.acl_has_role(Role::DAO, &env::predecessor_account_id()),
        "Unauthorized: caller cannot trigger GC"
    );
    // ... rest of function
}
```

Alternatively, make `run_mainchain_gc` private and only callable internally from `submit_blocks`, removing its public exposure entirely.

### Proof of Concept

```rust
// Attacker account: any NEAR account with no roles
// Precondition: contract is deployed and has > gc_threshold mainchain blocks

// Step 1: Call run_mainchain_gc with max batch_size
attacker_account
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({ "batch_size": u64::MAX }))
    .transact()
    .await?;
// Succeeds with no authorization error.

// Step 2: Attempt to verify a transaction in a now-GC'd block
let result = any_account
    .call(contract.id(), "verify_transaction_inclusion_v2")
    .args_borsh(ProofArgsV2 {
        tx_block_blockhash: old_block_hash, // a block that was just GC'd
        // ...
    })
    .transact()
    .await?;
// Panics: "block does not belong to the current main chain"
// The SPV proof is permanently unverifiable for this block.
```
