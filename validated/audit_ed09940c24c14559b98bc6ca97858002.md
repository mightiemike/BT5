### Title
Post-Validation Receipt Mutation Bypasses `max_receipt_size` Protocol Feature Boundary — (File: `runtime/runtime/src/lib.rs`)

### Summary

An unprivileged contract can craft an action receipt that passes the `NewReceipt` size gate at exactly `max_receipt_size` (4,194,304 bytes), then use `promise_return` or `value_return` to force the runtime to mutate the receipt **after** validation by appending `output_data_receivers`. The resulting oversized receipt is forwarded cross-shard and re-processed under the lenient `ExistingReceipt` mode, which skips the size check entirely. This is the nearcore analog of the return-bomb pattern: the attacker pays for a receipt that fits the limit, but the runtime absorbs the cost of processing a receipt that exceeds it.

### Finding Description

`validate_receipt()` in `runtime/runtime/src/verifier.rs` enforces the size limit only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

After that validation passes, the runtime mutates the receipt in `runtime/runtime/src/lib.rs` by extending `output_data_receivers` from the parent receipt into the newly-created child receipt:

```rust
ReceiptEnum::Action(new_action_receipt)
| ReceiptEnum::PromiseYield(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
``` [2](#0-1) 

No re-validation of the receipt size occurs after this mutation. The receipt is then forwarded cross-shard. When it arrives, `process_incoming_receipts` calls `validate_receipt` with `ValidateReceiptMode::ExistingReceipt`, which skips the size check: [3](#0-2) 

The `ExistingReceipt` mode is explicitly documented as tolerating oversized receipts due to this bug: [4](#0-3) 

A second path exists via `value_return`: a contract returns a value of exactly `max_length_returned_data` bytes. The runtime wraps it in a `DataReceipt`, adding Borsh framing overhead that pushes the total receipt size above `max_receipt_size`. `validate_data_receipt` only checks the payload length, not the total receipt size: [5](#0-4) 

The congestion-control layer in `runtime/runtime/src/congestion_control.rs` acknowledges the bug and works around it by clamping the reported size to `max_receipt_size` for bandwidth-accounting purposes, but the actual oversized receipt bytes are still forwarded and executed: [6](#0-5) 

### Impact Explanation

Every validator shard that receives the oversized receipt must deserialize and execute it. The Borsh deserialization of a receipt larger than `max_receipt_size` (up to `max_receipt_size` + the size of the appended `output_data_receivers` list) consumes memory and CPU proportional to the actual byte count, not the limit. Because the congestion-control system clamps the *accounting* size but not the *actual* receipt, the attacker pays gas for a `max_receipt_size`-byte receipt while validators absorb the cost of processing a larger one. Repeated across many shards or blocks this constitutes a sustained, low-cost resource-exhaustion attack against validators. The protocol invariant that "no receipt larger than `max_receipt_size` bytes shall be forwarded or executed" is broken.

### Likelihood Explanation

Any account that can deploy a contract and call it (i.e., any NEAR account with a small storage-staking deposit) can trigger both paths. No privileged role is required. The attack is deterministic and repeatable: the attacker simply sizes the function-call arguments so that the resulting receipt lands at exactly `max_receipt_size` before the `output_data_receivers` mutation, then chains a `promise_then` to force the mutation.

### Recommendation

After the `output_data_receivers` extension (and after any other post-creation mutations), re-measure the receipt with `borsh::object_length` and reject it with `ReceiptValidationError::ReceiptSizeExceeded` if it exceeds `limit_config.max_receipt_size`. Similarly, before emitting a `DataReceipt` from `value_return`, compute the full receipt size (payload + Borsh framing) and reject it if the total exceeds `max_receipt_size`, not just the raw payload length.

### Proof of Concept

**Path 1 — `promise_return` mutation:**

1. Deploy a contract with a method `attack()` that:
   - Calls `promise_create` targeting itself with a large `args` blob sized so the resulting action receipt is exactly `max_receipt_size` bytes.
   - Calls `promise_then` on that promise (creating a callback receipt B).
   - Returns `promise_return(inner_promise)` so the runtime will copy B's `output_data_receivers` into the inner receipt.
2. Submit a transaction calling `attack()`.
3. When the inner receipt executes, the runtime appends B's `DataReceiver` entries to it via the `extend_from_slice` path in `lib.rs:1029–1031`, pushing the receipt size above 4,194,304 bytes.
4. The oversized receipt is forwarded cross-shard. Incoming-receipt validation uses `ExistingReceipt` mode and skips the size check. Validators deserialize and execute the full oversized payload.

**Path 2 — `value_return` data receipt:**

1. Deploy a contract with a method `bomb()` that calls `value_return` with exactly `max_length_returned_data` (4,194,304) bytes.
2. Chain it with `promise_then` so the return value is delivered as a `DataReceipt`.
3. The `DataReceipt` Borsh encoding adds a fixed overhead (receipt envelope, `data_id`, `Option` tag) on top of the 4,194,304-byte payload, producing a receipt whose total serialized size exceeds `max_receipt_size`.
4. `validate_data_receipt` only checks `data_len <= max_length_returned_data` and passes; the total receipt size is never checked.

/// Validates given data receipt. Checks validity of the length of the returned data.
fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())
}
