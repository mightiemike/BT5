### Title
Odd-leaf duplicate position reuse allows ghost-index proof acceptance — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` never validates that `transaction_position` is within the actual transaction count of the block. In a Bitcoin Merkle tree with an odd number of transactions, the last leaf is duplicated to form a pair. This duplication causes the same proof path to produce the correct root for **two distinct index values** — the real last index `N-1` and the nonexistent ghost index `N`. An unprivileged caller can pass `tx_index = N` with the real last transaction's proof and receive `true` from `verify_transaction_inclusion_v2`, even though no transaction exists at that position.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof` (`merkle-tools/src/lib.rs:34-51`)**

The function iterates over the proof hashes, using `current_position % 2` to decide left/right placement, then halves the position. There is no check that `transaction_position < tree_leaf_count`: [1](#0-0) 

**Arithmetic proof of the collision**

Consider a 3-transaction block `[T0, T1, T2]`:

```
Level 0:  T0        T1        T2
Level 1:  H(T0,T1)  H(T2,T2)   ← T2 duplicated (odd count)
Root:     H(H(T0,T1), H(T2,T2))
```

The canonical proof for T2 at index **2** is `proof = [T2, H(T0,T1)]`.

| Step | index=2 (real) | index=3 (ghost) |
|------|----------------|-----------------|
| pos%2 | 2%2=0 → `hash(T2, T2)` | 3%2=1 → `hash(T2, T2)` |
| pos/2 | pos=1 | pos=1 |
| pos%2 | 1%2=1 → `hash(H(T0,T1), H(T2,T2))` | 1%2=1 → `hash(H(T0,T1), H(T2,T2))` |
| result | **root** ✓ | **root** ✓ |

Both produce the same root. The function returns `true` for `tx_id=T2` at the ghost index 3.

**Why the coinbase check in v2 does not block this**

`verify_transaction_inclusion_v2` verifies that `coinbase_tx_id` is at position `0` using a separate proof path. It does not constrain `tx_index` of the target transaction in any way: [2](#0-1) 

After the coinbase check passes, the function delegates to `verify_transaction_inclusion`, which passes the attacker-supplied `tx_index` directly to `compute_root_from_merkle_proof` with no bounds check: [3](#0-2) 

The attacker supplies a valid coinbase proof (coinbase at index 0 with a correct sibling path) alongside `tx_index = N` for the ghost position. Both checks pass independently.

---

### Impact Explanation

The contract returns `true` for a `(tx_id, tx_index)` pair where `tx_index` points to a position that does not exist in the block. Any downstream bridge, unlock, mint, or withdrawal contract that:

1. calls `verify_transaction_inclusion_v2` and trusts its `true` return, **and**
2. uses `(block_hash, tx_index)` as a replay-protection key (a standard pattern)

…can be replayed. The attacker first redeems the real transaction at index `N-1`, then replays it at the ghost index `N` with an identical proof, bypassing the replay guard because the key differs.

---

### Likelihood Explanation

- The block must have an odd transaction count — this is common in Bitcoin (roughly half of all blocks).
- The attacker needs only the last transaction's txid and its standard Merkle proof, both of which are public on-chain data.
- No privileged role, key, or social engineering is required.
- The call is a public `view`-style NEAR contract call; any account can invoke it.

---

### Recommendation

1. **Enforce `tx_index < 2^proof_depth` and parity consistency.** After computing the root, verify that `current_position == 0` (it always will be), but also verify that the supplied `tx_index` is consistent with the tree width implied by the proof length. Concretely, reject any proof where `tx_index >= (1 << merkle_proof.len())`.

2. **Stronger fix — require the proof's first sibling to differ from `tx_id` when `tx_index` is even.** In a valid odd tree, the last leaf's sibling is itself. If `proof[0] == tx_id` and `tx_index % 2 == 0`, the caller is at the last leaf. The only valid index in that case is `tx_index` where `tx_index` is even and `tx_index + 1` would be the duplicate — reject `tx_index + 1` (odd) for the same `tx_id` and same proof.

3. **Simplest production fix:** add a check `tx_index < (1usize << merkle_proof.len())` in `verify_transaction_inclusion` before calling `compute_root_from_merkle_proof`. This bounds the index to the maximum leaf count implied by the proof depth and eliminates the ghost-index slot.

### Proof of Concept

```rust
// Pseudocode — runs against an unmodified contract with a 3-tx block
// Block transactions: [coinbase, T1, T2]  (odd count = 3)
// Merkle tree:
//   Level 0: coinbase, T1, T2
//   Level 1: H(coinbase,T1),  H(T2,T2)
//   Root:    H(H(coinbase,T1), H(T2,T2))

let coinbase_proof_for_index_0 = vec![T1_hash, H_T2_T2];   // coinbase at 0
let t2_proof                   = vec![T2_hash, H_coinbase_T1]; // T2 at index 2

// Legitimate call — returns true (expected)
verify_transaction_inclusion_v2(T2_hash, block_hash, tx_index=2,
    merkle_proof=t2_proof, coinbase_tx_id=coinbase_hash,
    coinbase_merkle_proof=coinbase_proof_for_index_0, confirmations=0);

// Ghost-index call — also returns true (BUG)
verify_transaction_inclusion_v2(T2_hash, block_hash, tx_index=3,  // ← ghost
    merkle_proof=t2_proof, coinbase_tx_id=coinbase_hash,
    coinbase_merkle_proof=coinbase_proof_for_index_0, confirmations=0);
```

Both calls return `true`. A bridge keyed on `(block_hash, tx_index)` processes the same economic event twice.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
