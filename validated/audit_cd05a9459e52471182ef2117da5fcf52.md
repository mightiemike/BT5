### Title
Unprivileged Caller Can Permanently Prune Mainchain Headers via `run_mainchain_gc()` - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, state-mutating function that permanently deletes mainchain block headers from contract storage. It carries no caller access control: the only guard it has, `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, only restricts access when the contract is paused. When the contract is operating normally (unpaused), any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, immediately and irreversibly pruning the maximum number of mainchain headers allowed by `gc_threshold`. This corrupts the canonical chain state relied upon by SPV proof consumers and breaks future chain reorganizations.

---

### Finding Description

`run_mainchain_gc` is declared `pub` and is exposed as a NEAR contract method with no role check on the caller:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause(except(roles(...)))]` attribute from `near_plugins` only controls what happens when the contract is in a paused state — it allows `UnrestrictedRunGC` role holders to bypass the pause. It does **not** restrict who may call the function when the contract is unpaused. In normal operation, the function is open to any NEAR account.

When called, the function:
1. Computes how many headers exceed `gc_threshold`.
2. Deletes up to `batch_size` of the oldest mainchain headers from `headers_pool` and `mainchain_height_to_header`.
3. Permanently advances `mainchain_initial_blockhash` to the new oldest block.

All three mutations are irreversible — deleted storage entries cannot be recovered.

The `Role` enum defines `UnrestrictedRunGC` specifically to allow privileged callers to run GC while the contract is paused, which signals clear design intent that GC should be a controlled operation. However, this intent is not enforced when the contract is unpaused.

The integration test at `contract/tests/test_basics.rs` line 515–521 confirms that a regular user account (holding only `UnrestrictedSubmitBlocks`) successfully calls `run_mainchain_gc(100)` — demonstrating the open entry path.

---

### Impact Explanation

**Impact: High**

An attacker calling `run_mainchain_gc(u64::MAX)` forces the contract to prune the maximum number of mainchain headers in a single transaction (bounded only by `gc_threshold`). This causes:

1. **SPV proof verification failure**: `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the block hash in `mainchain_header_to_height`. After pruning, any transaction in a deleted block will cause the call to panic with `"block does not belong to the current main chain"`. Downstream contracts or bridges relying on these proofs will permanently lose the ability to verify historical transactions.

2. **Chain reorganization breakage**: As documented in `contract/CLAUDE.md` line 60, if mainchain blocks near a fork point are garbage collected, a reorg will panic with `PrevBlockNotFound`. An attacker can deliberately trigger GC to remove blocks near an active fork, permanently preventing the contract from accepting the heavier chain.

3. **Irreversibility**: Once `mainchain_initial_blockhash` is advanced and entries are removed from `headers_pool` and `mainchain_height_to_header`, the state cannot be restored without redeploying and re-syncing from genesis.

---

### Likelihood Explanation

**Likelihood: High**

- The function is unconditionally callable by any NEAR account when the contract is unpaused.
- No deposit, staking, or special credential is required.
- The call is cheap (bounded gas per batch) and can be repeated.
- The attacker needs only to know the contract account ID, which is public.

---

### Recommendation

Add an explicit role check to `run_mainchain_gc` so that only authorized accounts (e.g., `Role::DAO` or a dedicated `GCManager` role) can invoke it directly. The `#[pause]` guard should be supplemented with an access control check:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    near_sdk::require!(
        self.acl_has_role("DAO", &env::predecessor_account_id())
            || self.acl_has_role("UnrestrictedRunGC", &env::predecessor_account_id()),
        "Unauthorized: caller lacks GC permission"
    );
    // ... existing logic
}
```

Alternatively, make the function `pub(crate)` and expose it only as an internal call from `submit_blocks`, removing the standalone public entry point entirely.

---

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and sync it to a height well above the threshold (e.g., height 100,000).
2. From any unprivileged NEAR account (no roles granted), call:
   ```
   run_mainchain_gc({ "batch_size": 18446744073709551615 })
   ```
3. The contract prunes `100000 - 52704 = 47296` mainchain headers in one transaction, advancing `mainchain_initial_blockhash` to block 47296.
4. Any subsequent call to `verify_transaction_inclusion` for a transaction in blocks 0–47295 panics: `"block does not belong to the current main chain"`.
5. If a fork existed at height 47000, any attempt to submit the heavier fork chain now panics with `PrevBlockNotFound` because the common ancestor was deleted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
