### Title
Governance-mutable `ExitDuration` retroactively extends lock-up for all pre-existing positions that have not yet triggered exit — (`x/tieredrewards/keeper/msg_server_auth.go`)

### Summary

`MsgUpdateTier` can change a tier's `ExitDuration` via governance with immediate effect. Because `TriggerExitFromTier` reads `tier.ExitDuration` live at call time, every position that was created under the old duration but has not yet triggered exit will be subject to the new (potentially much longer) duration. The module applies a reward-settlement guard only when `BonusApy` changes, leaving `ExitDuration` changes completely unguarded. This is the direct analog of the external report: a mutable parameter controlled by a privileged actor that retroactively alters the terms of existing user commitments.

---

### Finding Description

`UpdateTier` in `msg_server_auth.go` contains an asymmetric guard:

```go
if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
    if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
        return nil, err
    }
}
// ← no analogous guard for ExitDuration changes
if err := ms.SetTier(ctx, msg.Tier); err != nil { ... }
``` [1](#0-0) 

When `BonusApy` changes, all positions are settled at the old rate before the new rate takes effect. No equivalent protection exists for `ExitDuration`. The new duration is written to the `Tiers` store immediately.

`TriggerExitFromTier` then reads the live tier at call time:

```go
tier, err := ms.getTier(ctx, pos.TierId)
...
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [2](#0-1) 

`ExitUnlockAt` is computed as `block_time + tier.ExitDuration` and stored on the position only at trigger time. [3](#0-2) 

A position created when `ExitDuration = 1 year` will receive `ExitUnlockAt = now + 5 years` if governance has since raised the duration to 5 years and the user has not yet called `TriggerExitFromTier`. The user's tokens remain locked for the full new duration with no recourse.

The `Tier` struct stores `ExitDuration` as a mutable field, and `MsgUpdateTier` is the sole governance message that writes it: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

A user who locks tokens into a tier with `ExitDuration = 1 year` has a reasonable expectation that they can begin the exit process and recover their principal within roughly one year. If governance passes `MsgUpdateTier` raising `ExitDuration` to, say, 5 years, every position that has not yet called `TriggerExitFromTier` is silently subjected to the new 5-year lock-up. The user's principal is inaccessible for 4 additional years beyond what they agreed to. Because `ExitUnlockAt` gates both `MsgTierUndelegate` and `MsgExitTierWithDelegation`, the user cannot exit until the new deadline elapses.

The corrupted invariant is: **the exit commitment duration a position owner accepted at lock time is no longer honoured**.

---

### Likelihood Explanation

Governance legitimately adjusts tier parameters over time — the integration tests explicitly exercise `MsgUpdateTier` changing `exit_duration` without any expectation of harm. [6](#0-5) 

A routine governance proposal to extend exit durations (e.g., to match a new staking unbonding period, or to increase protocol security) would silently impose the new duration on all existing positions. No malicious intent is required; the harm is structural.

---

### Recommendation

1. **Snapshot `ExitDuration` at position creation.** Store the exit duration on the `Position` proto so that `TriggerExitFromTier` uses the value the user agreed to, not the live tier value. Governance changes would then only affect new positions.
2. **Or, apply the same guard used for `BonusApy`.** Before writing a new `ExitDuration`, require all existing positions to have already triggered exit (i.e., `ExitTriggeredAt` is non-zero), or reject the update if any position has not yet triggered exit.
3. **Or, emit an on-chain notice period.** Require a time-lock between the governance proposal passing and the new `ExitDuration` taking effect, giving existing position holders time to trigger exit under the old terms.

---

### Proof of Concept

```
1. Governance creates Tier 1 with ExitDuration = 1 year, BonusApy = 4%.
2. Alice calls MsgLockTier(tier_id=1, amount=1_000_000, validator=V).
   → Position P created; Alice expects to trigger exit and recover funds within ~1 year.
3. Governance passes MsgUpdateTier(tier={id:1, exit_duration:5y, bonus_apy:0.04, ...}).
   → Because BonusApy is unchanged, claimRewardsAndUpdateTierPositions is NOT called.
   → Tier 1's ExitDuration is immediately overwritten to 5 years.
4. Alice calls MsgTriggerExitFromTier(position_id=P).
   → getTier reads ExitDuration = 5 years.
   → pos.ExitUnlockAt = now + 5 years.
5. Alice cannot call MsgTierUndelegate or MsgExitTierWithDelegation until 5 years
   from now, 4 years longer than she agreed to at lock time.
```

The root cause is the absence of any guard in `UpdateTier` for `ExitDuration` changes, contrasted with the explicit guard that exists for `BonusApy` changes at lines 67–71 of `msg_server_auth.go`. [7](#0-6)

func (ms msgServer) UpdateTier(ctx context.Context, msg *types.MsgUpdateTier) (*types.MsgUpdateTierResponse, error) {
	if err := ms.requireAuthority(msg.Authority); err != nil {
		return nil, err
	}

	oldTier, err := ms.getTier(ctx, msg.Tier.Id)
	if err != nil {
		return nil, err
	}

	if !oldTier.BonusApy.Equal(msg.Tier.BonusApy) {
		if err := ms.claimRewardsAndUpdateTierPositions(ctx, msg.Tier.Id); err != nil {
			return nil, err
		}
	}

	if err := ms.SetTier(ctx, msg.Tier); err != nil {
		return nil, err
	}

	if err := ms.emitTierChangedEvent(ctx, types.TierChangeAction_TIER_CHANGE_ACTION_UPDATE, msg.Tier); err != nil {
		return nil, err
	}

	return &types.MsgUpdateTierResponse{}, nil
}

def test_update_tier_non_apy_no_claim(cluster):
    """Changing exit_duration without changing APY does not trigger rewards claiming.

    Verified by checking that the owner balance is unchanged.
    """
    owner = cluster.address("signer1")

    # Confirm the signer1 position on Tier 3 exists
    result = query_command(cluster, MODULE, "positions-by-owner", owner)
    tier3_pos = next(
        p for p in result.get("positions", []) if int(p["tier_id"]) == TIER_3_ID
    )
    assert tier3_pos is not None
    balance_before = cluster.balance(owner, DENOM)

    rsp = submit_gov_proposal(
        cluster,
        "community",
        MSG_UPDATE_TIER,
        {
            "tier": {
                "id": TIER_3_ID,
                "exit_duration": "10s",
                "bonus_apy": "0.030000000000000000",
                "min_lock_amount": "2000000",
                "close_only": False,
            }
        },
        title="Update Tier 3 exit_duration",
        summary="Only exit_duration changes, no APY change",
    )
    approve_proposal(cluster, rsp, msg=f",{MSG_UPDATE_TIER}")

    # No rewards should have been paid out to the owner.
    assert (
        cluster.balance(owner, DENOM) == balance_before
    ), "owner balance should not change when APY is unchanged"

    # Verify exit_duration is updated
    result = query_tiers(cluster)
    tier3 = next(
        (t for t in result.get("tiers", []) if int(t["id"]) == TIER_3_ID), None
    )
    assert tier3["exit_duration"] == "10s"
