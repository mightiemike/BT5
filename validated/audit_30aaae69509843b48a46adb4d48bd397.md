### Title
Linked Signer Can Escalate Privileges by Replacing Itself With an Attacker-Controlled Address - (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction in the fast (sequencer-submitted) path is validated with `allowLinkedSigner = true`, meaning the **currently linked signer** can sign a `LinkSigner` transaction to replace itself with any arbitrary address. This allows a compromised or malicious session key to permanently hijack a subaccount's signing authority, enabling subsequent unauthorized withdrawals, liquidations, and transfers.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` case calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes the current linked signer to `Verifier.validateSignature`:

```solidity
// Verifier.sol lines 291–304
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // ← linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

This means the current linked signer can produce a valid `LinkSigner` signature pointing to any new address, and the contract will overwrite `linkedSigners[subaccount]` with that attacker-controlled address.

By contrast, the **slow mode** path for `LinkSigner` correctly restricts the operation to the subaccount owner via `validateSender(txn.sender, sender)`, which checks `address(uint160(bytes20(txn.sender))) == msg.sender`:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← owner-only in slow mode
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The inconsistency between the two paths confirms the fast-path behavior is unintended.

---

### Impact Explanation

Once the linked signer is replaced with an attacker-controlled address, that address can sign any transaction that accepts `allowLinkedSigner = true`, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — direct token drain from the subaccount
- `LiquidateSubaccount` — force-liquidate the victim's positions at a loss
- `TransferQuote` — transfer quote balance to an attacker-controlled subaccount
- `MintNlp` / `BurnNlp` — manipulate NLP positions [4](#0-3) 

The result is complete, permanent loss of signing authority over the subaccount and direct asset loss for the user.

---

### Likelihood Explanation

Linked signers are session keys intended for automated trading bots, browser wallets, or API integrations — environments where private key exposure is more likely than for a cold-wallet subaccount owner. A single key compromise is sufficient to trigger the attack. The sequencer must include the malicious `LinkSigner` transaction, but the sequencer has no basis to reject a validly signed transaction. Likelihood is **medium**: requires a compromised session key, but session keys are by design stored in higher-risk environments.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in the fast path, so that only the subaccount owner (the address embedded in `sender`) can authorize a signer change:

```solidity
// EndpointTx.sol — LinkSigner case in processTransactionImpl
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow mode path, which already enforces owner-only authorization via `validateSender`. [5](#0-4) 

---

### Proof of Concept

1. Alice owns subaccount `0xAlice000...` and links session key `signerA` via a `LinkSigner` transaction signed by her wallet.
2. `linkedSigners[0xAlice000...] = signerA` is stored on-chain.
3. `signerA` is compromised (e.g., leaked from a trading bot).
4. The attacker uses `signerA` to sign a new `LinkSigner` transaction: `{ sender: 0xAlice000..., signer: 0xAttacker..., nonce: currentNonce }`.
5. The sequencer includes this transaction in a batch submitted via `submitTransactionsChecked`.
6. `processTransactionImpl` validates the signature — `signerA == getLinkedSigner(0xAlice000...)` — and accepts it, writing `linkedSigners[0xAlice000...] = 0xAttacker`.
7. The attacker now signs a `WithdrawCollateral` transaction from `0xAlice000...` to drain all collateral. [1](#0-0) [6](#0-5)
