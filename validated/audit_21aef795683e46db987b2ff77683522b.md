### Title
Deprecated `verify_transaction_inclusion` Accepts Internal-Node Hashes as Valid Transaction IDs, Enabling Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a live, unpermissioned NEAR contract method that does not validate whether the supplied `tx_id` is a leaf node (a real transaction) or an internal node of the Bitcoin Merkle tree. An unprivileged NEAR caller can supply an internal node hash as `tx_id` with a truncated proof path and receive `true` for a transaction that was never included in the block. Any consuming contract that gates fund releases on this return value can be drained.

---

### Finding Description

**Vulnerability class:** Proof-verification forgery (direct analog to the Merk Hash-node child bypass).

**Root cause — `merkle-tools/src/lib.rs`, `compute_root_from_merkle_proof` (lines 34–52):**

`compute_root_from_merkle_proof` iteratively hashes the caller-supplied `transaction_hash` with each element of `merkle_proof`, using `transaction_position` only to decide left/right ordering. It has no concept of tree depth and cannot distinguish a leaf hash from an internal node hash. [1](#0-0) 

**Root cause — `contract/src/lib.rs`, `verify_transaction_inclusion` (lines 288–323):**

The only structural guard on the proof is `require!(!args.merkle_proof.is_empty(), ...)` at line 315. There is no check that `merkle_proof.len()` equals the expected tree depth, and no check that `tx_id` is a leaf. The function is marked `#[deprecated]` in Rust source, but `#[deprecated]` is a compiler lint only — it does **not** remove the method from the NEAR ABI. Any NEAR account can call it directly via a signed transaction. [2](#0-1) 

**Why `verify_transaction_inclusion_v2` does not help here:**

`verify_transaction_inclusion_v2` adds a coinbase proof of equal length, which correctly defeats the 64-byte attack against *v2*. However, it does not remove or gate `verify_transaction_inclusion`. The deprecated method remains independently callable. [3](#0-2) 

---

### Impact Explanation

Any contract that calls `verify_transaction_inclusion` and uses the `true` result to authorize a deposit credit or bridge withdrawal can be deceived. An attacker can claim inclusion of a fabricated "transaction" (an internal Merkle node) in a real, confirmed Bitcoin block without ever broadcasting that transaction to the Bitcoin network. This maps directly to the Merk report's scenario: a bridge or exchange light client is tricked into crediting coins that were never deposited.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. A real Bitcoin block already stored in the contract's mainchain (publicly available).
2. Knowledge of that block's Merkle tree (computable from public block data).
3. A single NEAR transaction calling `verify_transaction_inclusion` — no privileged role, no leaked key.

The only friction is that integrators must not have migrated to v2, but the deprecated method is still live and the contract's own `verify_transaction_inclusion_v2` internally calls the deprecated path after the coinbase check, so the deprecated method is clearly still considered functional.

---

### Recommendation

1. **Short term:** Add an access-control restriction to `verify_transaction_inclusion` (e.g., `#[private]` or a role check) so it cannot be called by arbitrary NEAR accounts. Alternatively, add a depth guard: require `args.merkle_proof.len()` to equal the coinbase proof length derived from a separately validated coinbase proof, mirroring the v2 approach.
2. **Long term:** Remove `verify_transaction_inclusion` from the public ABI entirely. Document that `verify_transaction_inclusion_v2` is the only supported entry point and that its `true` result still assumes the caller has validated that `tx_id` is a well-formed Bitcoin transaction (not an internal node).

---

### Proof of Concept

Take a real Bitcoin block at height H (already stored in the contract's mainchain) with 4 transactions `[tx0, tx1, tx2, tx3]`. Its Merkle tree is:

```
root = hash(hash(tx0,tx1), hash(tx2,tx3))
         I0 = hash(tx0,tx1)    I1 = hash(tx2,tx3)
```

`I0` is an internal node, not a transaction. Call `verify_transaction_inclusion` with:

```
tx_id              = I0                  // internal node, not a real tx
tx_block_blockhash = <block hash at H>
tx_index           = 0
merkle_proof       = [I1]               // one-element proof: sibling at depth 1
confirmations      = 1
```

Inside `compute_root_from_merkle_proof`:
- `current_position = 0`, `0 % 2 == 0` → `current_hash = hash(I0, I1) = root` ✓

The function returns `true`. [4](#0-3) 

A consuming bridge contract that calls `verify_transaction_inclusion` and credits the caller on `true` would now credit the attacker for a Bitcoin deposit that never occurred.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
