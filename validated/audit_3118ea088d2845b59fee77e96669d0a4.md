### Title
`ENABLE_SECP_OPS` Flag Bypassed via 4-Byte Opcode Aliases — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` unconditionally when invoked via their 4-byte opcodes (`0x13d61f00` and `0x1c3a8f00`), while the same functions are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS` when invoked via their 1-byte aliases (opcodes 64 and 65). An attacker-controlled CLVM program can invoke secp signature verification at any time, regardless of whether the `ENABLE_SECP_OPS` flag is set, by using the 4-byte opcode form.

---

### Finding Description

`ChiaDialect::op()` has two separate dispatch branches for secp operators:

**Branch 1 — 1-byte opcodes (correctly gated):**

```rust
// src/chia_dialect.rs lines 248-249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If `ENABLE_SECP_OPS` is absent, these fall through to `unknown_operator`, which either returns a no-op cost (consensus mode) or errors (mempool mode). [1](#0-0) 

**Branch 2 — 4-byte opcodes (no flag check):**

```rust
// src/chia_dialect.rs lines 175-182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

There is no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard here. The secp functions execute unconditionally. [2](#0-1) 

The `ENABLE_SECP_OPS` flag is documented as a hard-fork gate: "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify). Hard-fork; enable only when it activates." [3](#0-2) 

The 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are the cost-encoded forms of the same operations (the multiplier bytes encode the fixed costs of 1,300,000 and 1,850,000 respectively). [4](#0-3) 

The broken invariant is: **secp verification must not execute unless `ENABLE_SECP_OPS` is set**. The 4-byte path violates this invariant completely.

---

### Impact Explanation

**Consensus divergence.** A node running a dialect without `ENABLE_SECP_OPS` (i.e., before the hard-fork activates) is supposed to treat secp opcodes as unknown operators — returning nil with a cost in consensus mode, or erroring in mempool mode. Instead, via the 4-byte opcode path, the actual secp cryptographic verification runs. This means:

- A CLVM program using `0x13d61f00` with a valid secp256k1 signature succeeds (returns nil, costs 1,300,000) on a pre-fork node, identical to a post-fork node — but the pre-fork node should have returned nil as an unknown-op no-op, not as a verified signature.
- A CLVM program using `0x13d61f00` with an *invalid* secp256k1 signature raises `EvalErr::Secp256Failed` on a pre-fork node, while a post-fork node would also raise the same error — but a node that truly treated it as unknown would have returned nil with cost.

The concrete corrupted result: `EvalErr::Secp256Failed` is returned (or not) based on cryptographic validity of attacker-supplied bytes, when the correct result for a pre-fork node is always nil (unknown-op no-op). This is a flag/operator wiring error producing consensus divergence between pre- and post-fork nodes. [5](#0-4) 

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to a node is attacker-controlled bytes. Crafting a 4-byte opcode atom `0x13d61f00` or `0x1c3a8f00` in a CLVM program is trivial — it is just a quoted atom. No special privileges, no social engineering, and no dependency on local configuration are required. The attacker only needs to submit a transaction containing a CLVM puzzle that uses the 4-byte opcode form. This is realistic for any Chia coin spend. [6](#0-5) 

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte opcode dispatch branch, mirroring the 1-byte opcode branch:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that secp verification — regardless of which opcode encoding is used — is only reachable when the hard-fork flag is active, matching the documented intent of `ENABLE_SECP_OPS`. [2](#0-1)

### Proof of Concept

Construct a CLVM program that invokes secp256k1 verification via the 4-byte opcode on a dialect without `ENABLE_SECP_OPS`:

```rust
// Pseudocode
let dialect = ChiaDialect::new(ClvmFlags::empty()); // ENABLE_SECP_OPS not set

// Build opcode atom: 0x13d61f00 (4 bytes)
let op = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00])?;

// Build args: (pubkey msg sig) — any valid secp256k1 triple
let args = /* valid pubkey, 32-byte msg hash, valid sig */;

// Invoke directly
let result = dialect.op(&mut allocator, op, args, Cost::MAX, OperatorSet::Default);
// result: Ok(Reduction(1_300_000, nil))  — secp verification ran and succeeded
// expected: Ok(Reduction(<unknown_op_cost>, nil)) — unknown operator no-op
```

Without the flag guard, `op_secp256k1_verify` executes and returns based on cryptographic validity of the attacker-supplied arguments, not as an unknown operator. With an invalid signature, it returns `EvalErr::Secp256Failed` — a result that should be impossible on a pre-fork node. [3](#0-2) [7](#0-6)
