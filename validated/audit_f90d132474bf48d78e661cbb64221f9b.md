### Title
`SwapAllowlistExtension` gates the router contract address instead of the actual swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. The pool always sets `sender = msg.sender` of the `swap()` call. When a user swaps through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual user. If the pool admin allowlists the router to support router-mediated swaps, every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to every configured extension: [2](#0-1) 

**Step 2 — `SwapAllowlistExtension` checks `sender` (the pool's `msg.sender`) against the allowlist.**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is whatever address called `pool.swap()`. [3](#0-2) 

**Step 3 — `MetricOmmSimpleRouter` calls `pool.swap()` directly with no user-level checks.**

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router is `msg.sender` at the pool level. The actual end-user (`msg.sender` of the router call) is never forwarded to the pool or the extension. [4](#0-3) 

**Step 4 — The allowlist therefore gates the router address, not the user.**

When the pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for legitimate users, the mapping entry `allowedSwapper[pool][router] = true` is set. Because the extension sees `sender = router` for every router-originated swap, the check passes for **any** caller of the router, regardless of whether that caller is on the allowlist. [5](#0-4) 

**Contrast with `DepositAllowlistExtension`.**

The deposit allowlist correctly gates by `owner` (the position owner explicitly passed to `addLiquidity`), not by `sender` (the caller). The swap allowlist has no equivalent second identity argument — the only identity the pool exposes to the extension is `sender = msg.sender`, which collapses to the router when the router is used. [6](#0-5) 

---

### Impact Explanation

Any user who is **not** on the swap allowlist can execute swaps on a restricted pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput` / `exactOutputSingle`). The pool receives real token input and sends real token output; the bypass is economically complete. The pool admin's intent to restrict swap access to a curated set of addresses is fully defeated. This constitutes broken core pool functionality with direct loss of the access-control invariant and potential fund-impacting consequences (e.g., a pool restricted to KYC'd counterparties or whitelisted market-makers is opened to arbitrary traders).

---

### Likelihood Explanation

The trigger requires the pool admin to allowlist the router address. This is the natural, expected configuration for any pool that wants to support router-mediated swaps for its allowlisted users — the admin allowlists the router thinking it acts as a trusted forwarder. The router itself imposes no user-level checks, so the allowlist is silently voided. The attacker needs no special privilege: any EOA can call the public router.

---

### Recommendation

The extension must gate the **actual end-user**, not the intermediary. Two viable approaches:

1. **Encode the real user in `extensionData`**: have the router append `abi.encode(msg.sender)` to `extensionData` before forwarding to the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that address. The extension should fall back to checking `sender` when no user override is present (direct pool calls).

2. **Check `recipient` as a proxy**: for single-hop swaps the recipient is often the actual user, but this breaks for multi-hop paths where intermediate recipients are the router itself.

Option 1 is the cleanest fix. The extension should also verify that the encoded user address is not `address(0)` and that the `extensionData` prefix is well-formed before trusting it.

### Proof of Concept

```solidity
// Setup: pool admin deploys pool with SwapAllowlistExtension and allowlists the router
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// alice is NOT individually allowlisted
// allowedSwapper[pool][alice] == false

// Attack: alice calls the router directly
vm.prank(alice);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool:            address(pool),
    tokenIn:         address(token0),
    tokenOut:        address(token1),
    zeroForOne:      true,
    amountIn:        1_000,
    amountOutMinimum: 0,
    recipient:       alice,
    deadline:        block.timestamp + 1,
    priceLimitX64:   0,
    extensionData:   ""
}));
// router calls pool.swap(...) → msg.sender at pool = router
// extension checks allowedSwapper[pool][router] == true → passes
// alice receives token1 output despite not being allowlisted
```

The pool's `beforeSwap` hook receives `sender = address(router)`, which is allowlisted, so `NotAllowedToSwap` is never triggered. Alice's swap settles in full. [7](#0-6) [8](#0-7) [1](#0-0)
