### Title
Merkle Proof Malleability Allows Forged Transaction Inclusion Verification — (`File: contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` accepts an arbitrary 32-byte value as `tx_id` without enforcing that it is a leaf-level transaction hash. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` together with a shorter-than-expected proof path, causing the function to return `true` for a transaction that was never included in the block. This is the direct analog of ECDSA signature malleability: just as `ecrecover` accepts non-unique `(r, s, v)` tuples that map to the same signer, `compute_root_from_merkle_proof` accepts non-unique `(tx_id, proof)` pairs that map to the same Merkle root.

---

### Finding Description

`verify_transaction_inclusion` is a public, `#[pause]`-gated (but not role-gated) NEAR contract method. It calls `merkle_tools::compute_root_from_merkle_proof` and compares the result to the stored `merkle_root`: [1](#0-0) 

`compute_root_from_merkle_proof` iterates over the supplied proof hashes and combines them with the running hash using position-based left/right ordering: [2](#0-1) 

Neither function validates that `transaction_hash` is a leaf-level hash. The Bitcoin Merkle tree is constructed by hashing pairs of 32-byte values; an internal node at depth *d* is itself a valid 32-byte value that, when supplied as `tx_id` with a proof of length `(tree_depth − d)`, will compute to the correct Merkle root. The code's own warning acknowledges this: [3](#0-2) 

The function is still directly callable despite being marked `#[deprecated]`. In Rust, `#[deprecated]` is a compiler lint, not a runtime access restriction. Any NEAR account can invoke it: [4](#0-3) 

`verify_transaction_inclusion_v2` mitigates this by requiring that both the coinbase proof and the target proof have the same length, anchoring the expected tree depth: [5](#0-4) 

However, v1 remains independently callable and carries no such depth enforcement.

---

### Impact Explanation

Any recipient contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., a bridge unlock, a payment confirmation, a cross-chain settlement) will receive `true` for a `tx_id` that is an internal Merkle node — a transaction that was never broadcast or mined. The corrupted value is the boolean return of `verify_transaction_inclusion`: it transitions from `false` (correct) to `true` (forged) for a fabricated `tx_id`. This directly enables proof-verification forgery with the same class of impact as ECDSA signature malleability: an attacker bypasses a cryptographic membership check without possessing the original secret (the actual transaction).

---

### Likelihood Explanation

The 64-byte internal-node attack against Bitcoin Merkle proofs is well-documented (bitmex.com/blog/64-Byte-Transactions) and requires no special privileges. The attacker only needs: (1) a valid block hash present in the contract's `headers_pool`, (2) knowledge of any internal Merkle node hash for that block (derivable from public block data), and (3) the ability to call a public NEAR contract method. All three are trivially satisfied by any unprivileged caller.

---

### Recommendation

Remove or hard-restrict `verify_transaction_inclusion` (v1) so it is no longer callable by external accounts. The `#[deprecated]` attribute does not prevent runtime invocation. Options:
- Add `#[private]` to make it callable only by the contract itself (since v2 already calls it internally via `#[allow(deprecated)]`).
- Or inline the v1 logic into v2 and delete the v1 public method entirely.

Additionally, `compute_root_from_merkle_proof` should enforce that the proof length matches the expected tree depth derived from the coinbase proof, rather than relying on callers to enforce this invariant.

---

### Proof of Concept

1. Identify a block hash `B` stored in the contract's `headers_pool` (publicly readable via `get_block_hash_by_height`).
2. From public Bitcoin block data, compute the Merkle tree for block `B`. Select any internal node `N` at depth 1 (i.e., `N = SHA256d(tx0 || tx1)`). Its position index in the depth-1 layer is `p`.
3. Construct a proof of length 1: the sibling of `N` at depth 1.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash = B`
   - `tx_index = p` (position of `N` in the depth-1 layer, adjusted for the halved index space)
   - `merkle_proof = [sibling_of_N]`
   - `confirmations = 1`
5. The function computes `compute_root_from_merkle_proof(N, p, [sibling])`, which equals the block's `merkle_root`, and returns `true` — falsely asserting that the non-existent transaction `N` is included in block `B`. [2](#0-1) [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
