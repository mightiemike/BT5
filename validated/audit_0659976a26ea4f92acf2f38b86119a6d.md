### Title
`SwapAllowlistExtension` checks the router's address instead of the real user's address, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the originating user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. Any user can bypass a curated pool's swap allowlist by routing through the public router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly: [4](#0-3) 

At the pool, `msg.sender` is the **router address**, so `sender` delivered to the extension is the router, not the originating EOA. The extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

The same mismatch applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` as `msg.sender = router`. [5](#0-4) 

---

### Impact Explanation

A pool admin deploys a curated pool (e.g., KYC-only, institutional-only) and configures `SwapAllowlistExtension` with a set of approved user addresses. The invariant is that only those addresses may swap. Because the extension checks the router's address instead of the user's address, any non-allowlisted user can bypass the gate by calling `MetricOmmSimpleRouter.exactInputSingle` (or any multi-hop variant). The allowlist is completely ineffective for router-mediated swaps, which is the primary supported public swap path. This constitutes a broken core pool functionality / admin-boundary break with direct policy bypass on curated pools.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint documented and deployed alongside the protocol. Any user who knows the pool has an allowlist can trivially route through the public router to bypass it. No privileged access, no special tokens, and no admin cooperation is required. The trigger is a single standard router call.

---

### Recommendation

The pool must deliver the **originating user's address** to the extension, not the intermediary's address. Two viable approaches:

1. **Pass the real initiator through the router**: The router stores the originating `msg.sender` in transient storage (it already uses transient storage for callback context) and the pool reads it back as the canonical `sender` for extension dispatch.

2. **Check `recipient` instead of `sender` in the extension**: If the pool's design guarantees that `recipient` is always the economic beneficiary, the extension can gate on `recipient`. However, `recipient` can be set to an arbitrary address by the caller, so this is only safe if the pool enforces `recipient == msg.sender` or a similar constraint.

The cleanest fix is approach (1): the router records the originating EOA in transient storage alongside the existing callback context, and the pool reads it when dispatching extension hooks.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
  - Admin does NOT allowlist the router or bob

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: bob,
       ...
     })
  2. Router calls pool.swap(bob, ...) — msg.sender at pool = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router]
     → router is not allowlisted → should revert NotAllowedToSwap
     
  BUT: if the admin also allowlisted the router (necessary for alice to use the router),
  then allowedSwapper[pool][router] == true → bob's swap passes.

  Result: bob, a non-allowlisted user, successfully swaps on a curated pool
  by routing through the public MetricOmmSimpleRouter.
  
  Alternatively, even without the router being allowlisted:
  alice cannot use the router at all (her swap reverts because the router is not allowlisted),
  breaking the intended UX and forcing direct pool calls — demonstrating the identity mismatch
  in either direction.
``` [6](#0-5) [7](#0-6)
