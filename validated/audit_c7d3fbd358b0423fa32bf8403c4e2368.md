### Title
`ValidResourceBounds` Protobuf Deserialization Misclassifies `AllResources` Transactions as `L1Gas` When Both `l2_gas` and `l1_data_gas` Are Zero, Producing a Wrong Transaction Hash — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` (used in the sync/consensus transaction path) applies a "zero-balance" heuristic: if both `l2_gas` and `l1_data_gas` are zero after deserialization, the transaction is classified as `ValidResourceBounds::L1Gas`. However, a legitimately submitted `AllResources` transaction can carry zero values for both of those fields. Because `get_tip_resource_bounds_hash` hashes a different number of resource elements depending on the variant (`L1Gas` → 2 elements; `AllResources` → 3 elements), the round-trip through protobuf produces a structurally different hash preimage, yielding a divergent transaction hash.

---

### Finding Description

**Root cause — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`**

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 417-436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant for AllResources txs
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) always emits `l1_data_gas: Some(...)` for **both** variants — including `AllResources` with zero `l1_data_gas`. So the `unwrap_or_default()` path is not the trigger; the trigger is the classification condition `l1_data_gas.is_zero() && l2_gas.is_zero()`, which is true for any `AllResources` transaction whose user set only `l1_gas`.

**Hash divergence — `get_tip_resource_bounds_hash`**

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 188-211
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip) -> ... {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 elements
        ValidResourceBounds::AllResources(all) =>
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?],    // 3 elements
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`L1Gas` → `poseidon(tip, l1_gas_packed, l2_gas_packed)`  
`AllResources` → `poseidon(tip, l1_gas_packed, l2_gas_packed, l1_data_gas_packed)`

Even when `l1_data_gas_packed` encodes all-zero bounds, the Poseidon hash over 3 elements differs from the hash over 2 elements. The two variants therefore produce **distinct transaction hashes for identical numeric field values**.

**Contrast with the RPC path (not affected)**

The RPC/mempool protobuf converter (`crates/apollo_protobuf/src/converters/rpc_transaction.rs`) deserializes directly into `AllResourceBounds` and requires `l1_data_gas` to be present, so it never applies the zero-balance heuristic and is not affected.

---

### Impact Explanation

A `Transaction::Invoke(InvokeTransactionV3)` stored in a committed block with `ValidResourceBounds::AllResources` and zero `l2_gas`/`l1_data_gas` is serialized to protobuf with `l1_data_gas = Some(zero)`. Any peer that deserializes this block via the sync path reconstructs the transaction as `ValidResourceBounds::L1Gas`. When that peer recomputes the transaction hash (e.g., during block validation via `validate_transaction_hash`), it produces a hash that does not match the committed hash, causing the block to be rejected or the stored hash to be wrong. This maps to:

- **Critical**: Wrong state, receipt, or revert result from blockifier/execution logic for accepted input.
- **High**: Transaction conversion or hash logic binds the wrong hash or executable payload.

---

### Likelihood Explanation

The gateway accepts `AllResourceBounds` with only `l1_gas` set (the gateway validator test explicitly covers `AllResourceBounds { l1_gas: NON_EMPTY_RESOURCE_BOUNDS, ..Default::default() }`). Any user who submits such a transaction triggers the condition. The transaction is then stored in a block as `AllResources`, and every syncing peer will misclassify it as `L1Gas`, computing a divergent hash. No special privilege or malicious peer is required; a normal user transaction is sufficient.

---

### Recommendation

Replace the zero-balance heuristic with an explicit discriminant. The protobuf `ResourceBounds` message should carry a boolean or enum field (e.g., `is_all_resources`) that is set by the serializer and read by the deserializer, so the variant is preserved exactly. Alternatively, since the TODO already acknowledges that `l1_data_gas = None` should be forbidden once 0.13.2 support is dropped, enforce that assertion now and treat `l1_data_gas = Some(zero)` as `AllResources` unconditionally (matching the serializer's behavior), removing the `L1Gas` classification branch from the deserializer entirely for current protocol versions.

---

### Proof of Concept

1. User submits `InvokeV3` with `AllResourceBounds { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }`.
2. Gateway calls `calculate_transaction_hash` → `get_tip_resource_bounds_hash` with `AllResources` → hash includes `l1_data_gas_packed` (zero-encoded) → hash = `H_all`.
3. Block is committed; `Transaction::Invoke(InvokeTransactionV3 { resource_bounds: AllResources(...), ... })` is stored with hash `H_all`.
4. Syncing peer receives the block via protobuf. `From<ValidResourceBounds>` serializes `AllResources` as `l1_data_gas: Some(zero), l2_gas: Some(zero)`.
5. Peer's `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l1_data_gas.is_zero() && l2_gas.is_zero()` → returns `L1Gas(l1_gas)`.
6. Peer calls `calculate_transaction_hash` → `get_tip_resource_bounds_hash` with `L1Gas` → hash does **not** include `l1_data_gas_packed` → hash = `H_l1 ≠ H_all`.
7. `validate_transaction_hash` compares `H_l1` against the committed `H_all` → mismatch → block rejected or wrong hash stored. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
