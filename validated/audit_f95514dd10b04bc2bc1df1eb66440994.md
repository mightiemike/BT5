### Title
Missing Post-Transfer Health Check on Parent Subaccount in `createIsolatedSubaccount` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`OffchainExchange.createIsolatedSubaccount` transfers quote margin from the parent subaccount to a newly created isolated subaccount without performing any health check on the parent afterward. The analogous collateral-reducing operation, `Clearinghouse.transferQuote`, explicitly enforces `require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH)` after the same `spotEngine.updateBalance` call. The omission in `createIsolatedSubaccount` allows a user to drain their parent account's collateral below initial health in a single signed transaction, bypassing the protocol's core solvency invariant.

---

### Finding Description

**Entrypoint:** `EndpointTx.processTransactionImpl` handles `TransactionType.CreateIsolatedSubaccount` by decoding the transaction and calling `IOffchainExchange(offchainExchange).createIsolatedSubaccount(txn, getLinkedSigner(txn.order.sender))`. [1](#0-0) 

**Margin extraction:** The margin value is attacker-controlled. `_isolatedMargin` reads bits `[127:64]` of the order appendix and scales by `10**12`, meaning any 64-bit value can be encoded. [2](#0-1) 

**The vulnerable transfer block:** After signature validation, the function deducts `margin` from the parent and credits the isolated subaccount. No health assertion follows. [3](#0-2) 

**The enforced invariant in `transferQuote`:** The identical collateral-reducing pattern in `Clearinghouse.transferQuote` ends with `require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH)`. [4](#0-3) 

`_isAboveInitial` calls `getHealth(subaccount, HealthType.INITIAL) >= 0`. [5](#0-4) 

The `createIsolatedSubaccount` path contains no equivalent call anywhere — not in `OffchainExchange`, not in the `EndpointTx` handler, and not in any hook (`_onCreateIsolatedSubaccount` is a no-op virtual). [6](#0-5) 

---

### Impact Explanation

A user who controls parent subaccount P with quote balance Q and open leveraged positions can:

1. Craft an order with `appendix` bits `[127:64]` encoding `margin ≈ Q` (leaving parent near-zero collateral).
2. Sign the order with their own key (signature check passes trivially since they own P).
3. Submit `CreateIsolatedSubaccount` — `spotEngine.updateBalance(QUOTE_PRODUCT_ID, P, -margin)` executes, parent balance drops to ~0.
4. Parent P is now below initial health with open positions still attached.
5. Any third-party liquidator can immediately liquidate P at a discount, extracting value from the protocol's insurance fund if the position is large enough to create bad debt.

The isolated subaccount retains the drained margin, so the attacker preserves value while forcing the parent into a liquidatable state — a state the protocol explicitly prevents via `transferQuote`'s health check.

---

### Likelihood Explanation

The path is fully reachable through the normal signed-order flow. No admin access, sequencer compromise, or external dependency is required. The attacker only needs to own the parent subaccount and craft a valid EIP-712 signed order with a large margin value in the appendix. The `digestToSubaccount` replay guard does not help because each unique order digest (different nonce or amount) creates a fresh execution path.

---

### Recommendation

Add a post-transfer initial health assertion on the parent subaccount inside `createIsolatedSubaccount`, mirroring `transferQuote`:

```solidity
// After the spotEngine.updateBalance calls (line ~1087):
require(
    clearinghouse.getHealth(txn.order.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [3](#0-2) 

---

### Proof of Concept

```
1. Deploy protocol with parent subaccount P holding 1000 USDC and a perp position
   requiring 100 USDC initial margin (parent initial health = +900).
2. Craft CreateIsolatedSubaccount order:
     sender   = P
     productId = <perp product>
     appendix bits [127:64] = 950 / 10^12  (encodes margin = 950e18 USDC)
     appendix bit [8] = 1  (isIsolated flag)
3. Sign with P's private key → _checkSignature passes.
4. Submit via processTransactionImpl(CreateIsolatedSubaccount).
5. spotEngine.updateBalance(QUOTE_PRODUCT_ID, P, -950e18) executes.
6. Assert: clearinghouse.getHealth(P, INITIAL) < 0  ← invariant broken, no revert.
7. Call liquidateSubaccount targeting P → succeeds, liquidator receives discount.
```
