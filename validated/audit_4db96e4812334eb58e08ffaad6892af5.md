### Title
`From<ResourceBoundsMapping> for ValidResourceBounds` Silently Downgrades `AllResources` to `L1Gas`, Producing a Wrong Transaction Hash in RPC-Served Transactions - (File: crates/apollo_rpc/src/v0_8/transaction.rs)

### Summary

A V3 transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is stored and hashed under the `ValidResourceBounds::AllResources` variant, which includes the zero `l1_data_gas` field in the Poseidon hash preimage. When the RPC layer later serializes and re-deserializes the same transaction through `ResourceBoundsMapping`, the `From<ResourceBoundsMapping> for ValidResourceBounds` conversion silently downgrades the variant to `ValidResourceBounds::L1Gas`, which excludes `l1_data_gas` from the hash. Any consumer that recomputes the transaction hash from the RPC-served representation obtains a different value than the stored canonical hash.

### Finding Description

**Step 1 – Admission stores `AllResources`.**

`RpcTransaction` always carries `AllResourceBounds` (never `ValidResourceBounds::L1Gas`). The gateway converts it to `InternalRpcInvokeTransactionV3`, whose `InvokeTransactionV3Trait::resource_bounds()` unconditionally wraps the value in `ValidResourceBounds::AllResources`:

```rust
// crates/starknet_api/src/rpc_transaction.rs:637-638
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)
    }
```

The transaction hash is then computed by `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with this `AllResources` value.

**Step 2 – `get_tip_resource_bounds_hash` includes `l1_data_gas` only for `AllResources`.**

```rust
// crates/starknet_api/src/transaction_hash.rs:203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

For `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` the hash chain is `[tip, L1_GAS(X), L2_GAS(0), L1_DATA_GAS(0)]`.
For `L1Gas(X)` the hash chain is `[tip, L1_GAS(X), L2_GAS(0)]`.
These are **different Poseidon digests**.

**Step 3 – The RPC round-trip silently changes the variant.**

When the stored `InvokeTransactionV3` (holding `ValidResourceBounds::AllResources`) is serialized to the RPC wire format it becomes `ResourceBoundsMapping`. When any consumer converts it back:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs:188-199
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)   // ← variant changed
        } else {
            Self::AllResources(...)
        }
    }
}
```

Because `l1_data_gas` and `l2_gas` are zero, the result is `L1Gas(X)`, not `AllResources(X, 0, 0)`. Any hash recomputation from this point forward omits the `l1_data_gas` field and produces a hash that does not match the stored canonical hash.

**Step 4 – The gateway accepts such transactions.**

The stateless validator explicitly allows `AllResourceBounds` with only `l1_gas` non-zero (the `valid_l1_gas` test case confirms this). The transaction is therefore reachable without any privilege.

### Impact Explanation

The RPC layer returns a transaction representation that is internally inconsistent with the stored transaction hash. Any client, relay, or verifier that:
- reads the transaction via RPC,
- converts `ResourceBoundsMapping` back to `ValidResourceBounds` using the standard conversion, and
- recomputes the transaction hash

will obtain a hash that differs from the canonical one. This constitutes an authoritative-looking wrong value returned by the RPC, matching the **High** impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

Downstream effects include: signature verification failures in wallets/SDKs that re-derive the hash, incorrect mempool deduplication if any component re-hashes from the RPC representation, and broken proof-of-inclusion checks.

### Likelihood Explanation

Any V3 transaction with only `l1_gas` set (a common pattern for pre-0.13.3 style transactions submitted through the new API) triggers the bug. The gateway explicitly accepts this input. No special privileges or malformed bytes are required.

### Recommendation

Remove the heuristic downgrade in `From<ResourceBoundsMapping> for ValidResourceBounds`. The variant should be preserved explicitly, either by:
1. Adding a discriminator field to `ResourceBoundsMapping` (e.g., a `bounds_type` tag), or
2. Always deserializing to `AllResources` when all three fields are present in the wire format, regardless of whether they are zero.

The same fix must be applied to the identical pattern in `crates/apollo_protobuf/src/converters/transaction.rs` (lines 431–435).

### Proof of Concept

```
1. Submit via RPC:
   InvokeV3 {
     resource_bounds: AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     },
     ...
   }

2. Sequencer stores hash H_all = Poseidon(tip, L1_GAS(1000,1), L2_GAS(0,0), L1_DATA_GAS(0,0))

3. Client reads transaction via starknet_getTransactionByHash → receives ResourceBoundsMapping
   { l1_gas: (1000,1), l2_gas: (0,0), l1_data_gas: (0,0) }

4. Client calls From<ResourceBoundsMapping> for ValidResourceBounds
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → produces ValidResourceBounds::L1Gas((1000,1))

5. Client recomputes hash H_l1 = Poseidon(tip, L1_GAS(1000,1), L2_GAS(0,0))
   (l1_data_gas omitted)

6. H_all ≠ H_l1  →  hash verification fails / wrong hash reported

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
