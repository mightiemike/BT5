The call path is fully traceable and the vulnerability is real. Here is the complete analysis:

**Exact call chain:**

1. `MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`
2. `ExtensionCalling._beforeSwap()` encodes that `sender` and calls the extension
3. `SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool, `sender` = whoever called `pool.swap()`

When the router intermediates: `attacker → router.exactInputSingle()` → `pool.swap()` with `msg.sender = router` → extension receives `sender = router` → checks `allowedSwapper[pool][router] = true` → **passes**.

The original EOA is never checked.

### Title
SwapAllowlistExtension Per-User Gate Bypassed via Allowlisted Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool binds to `msg.sender` of `pool.swap()`. When any public router (e.g. `MetricOmmSimpleRouter`) intermediates, `msg.sender` of `pool.swap()` is the router, not the originating EOA. If the router is allowlisted, every user who calls through it bypasses the per-user swap gate entirely.

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

So when `allowedSwapper[pool][router] = true`, the check passes for **any** caller of the router, regardless of whether that caller is individually allowlisted.

### Impact Explanation
Any EOA or contract not in `allowedSwapper[pool]` can swap against a restricted pool by routing through an allowlisted router. The pool's token balances are directly accessible to unauthorized swappers, constituting unauthorized fund movement from the pool. The invariant that "only individually allowlisted addresses may swap" is broken.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is the canonical public swap interface. Pool admins who want to allow router-mediated swaps for their allowlisted users will naturally add the router to `allowedSwapper`. The bypass is then immediately available to every user of the router — no special setup or privileged access is required by the attacker.

### Recommendation
The extension must resolve the originating user, not the immediate caller of `pool.swap()`. Options:
- Pass the original `msg.sender` (the EOA/initiator) through the router as an explicit parameter and have the extension check that value.
- Require that the pool's `swap` function only accepts direct calls (no router intermediation) when the allowlist extension is active.
- Have the router forward the original caller's identity in `extensionData` and have the extension decode and verify it (with appropriate authentication so it cannot be spoofed).
