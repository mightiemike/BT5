Audit Report

## Title
Router-mediated swaps corrupt swapper identity in `SwapAllowlistExtension::beforeSwap`, breaking per-user allowlist enforcement — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the address that called `pool.swap()`. When `MetricOmmSimpleRouter` mediates the swap, `sender` resolves to the router's address rather than the original user's address. This makes the per-user allowlist check meaningless for all router-mediated swaps, either locking out legitimately allowlisted users or opening the gate to all users if the router is allowlisted.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged as `sender` to every configured extension. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool: [3](#0-2) 

The same applies to `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`. The router stores the original payer only in transient storage for the payment callback — not for the extension hook: [4](#0-3) 

There is no mechanism in the router to forward the original `msg.sender` as the `sender` argument to the pool. The extension has no router-awareness and no fallback check.

## Impact Explanation

**Broken core pool functionality (Medium)**:

- **Scenario A — allowlisted user locked out of router**: Pool admin allowlists `alice` but not the router. Alice calls `exactInputSingle` → hook sees `sender = router` → `allowedSwapper[pool][router] = false` → revert. Alice must bypass the standard router and call `pool.swap()` directly, breaking standard UX.
- **Scenario B — router allowlisting opens the gate to all**: Pool admin allowlists the router address to enable router-mediated swaps. Now any user can call `exactInputSingle` and the hook passes, defeating the per-user allowlist entirely.

Both scenarios corrupt the invariant that the allowlist gates the actual economic actor performing the swap.

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects users to interact via `MetricOmmSimpleRouter` is affected. The router is a public, factory-validated contract reachable by any user. No privileged access, special setup, or malicious configuration is required. The bug is triggered by normal router usage.

## Recommendation

Pass the original caller's identity through the extension data or add a dedicated `originalSender` field to the hook signature. Alternatively, the router should embed the original `msg.sender` in `extensionData` in a tamper-proof way (e.g., verified by the pool against a router registry), or `SwapAllowlistExtension` should check both `sender` and a router-registry-verified original sender extracted from `extensionData`.

## Proof of Concept

```solidity
// Pool configured with SwapAllowlistExtension
// Pool admin allowlists alice, NOT the router
extension.setAllowedToSwap(pool, alice, true);

// Alice tries to swap via router — REVERTS
// hook sees sender = address(router), not alice
vm.prank(alice);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: alice,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// → NotAllowedToSwap()

// Scenario B: admin allowlists the router to "fix" this
extension.setAllowedToSwap(pool, address(router), true);

// Now ANY user (bob, who is NOT allowlisted) can swap via router
vm.prank(bob);
router.exactInputSingle(...); // succeeds — allowlist bypassed
```
