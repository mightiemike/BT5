### Title
Synthetic-ratio precision loss in `AnchoredPriceProvider._getBidAndAskPrice` truncates mid to zero, permanently blocking swaps on low-ratio synthetic pools — (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

When a pool's `AnchoredPriceProvider` is configured in synthetic (two-feed) mode, the ratio `mid1 × ORACLE_DECIMALS / mid2` truncates to zero via integer division whenever the base-feed price is much smaller than the quote-feed price. The zero `mid` propagates through `_computeBidAsk`, which returns the stalled sentinel `(0, type(uint128).max)`, causing every subsequent swap on that pool to revert with `FeedStalled`. No privileged action is required; ordinary market price divergence is sufficient to trigger the condition.

---

### Finding Description

In `AnchoredPriceProvider._getBidAndAskPrice`, line 267:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
```

`ORACLE_DECIMALS` is `1e8`. When `mid * 1e8 < mid2`, integer division truncates the result to zero. Both individual `_readLeg` calls can succeed (both prices non-zero, non-stale, within guard bounds), so `ok` and `ok2` are both `true`. The zero `mid` is then passed to `_computeBidAsk`:

```solidity
uint256 refBid = _bandEdge(mid, BPS_BASE_U - half, Math.Rounding.Floor);
// _bandEdge(0, ...) = Math.mulDiv(0, ...) = 0
if (refBid == 0 || refAsk > type(uint128).max || refBid >= refAsk) {
    return (0, type(uint128).max);
}
```

`_computeBidAsk` returns the stalled sentinel. `getBidAndAskPrice` then reverts:

```solidity
if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
```

Every call to `pool.swap(...)` invokes `_getBidAndAskPriceX64()`, which calls `getBidAndAskPrice()`, so all swaps revert.

**Concrete example:**
- PEPE/USD feed: `mid1 = 1137` (price $0.00001137, scaled to 1e8)
- BTC/USD feed: `mid2 = 12_000_000_000_000` (price $120,000, scaled to 1e8)
- `Math.mulDiv(1137, 1e8, 12_000_000_000_000)` = `113_700_000_000 / 12_000_000_000_000` = **0** [1](#0-0) [2](#0-1) [3](#0-2) 

The pool's `_getBidAndAskPriceX64` wraps this call and propagates the revert: [4](#0-3) 

---

### Impact Explanation

All swaps on any pool whose `AnchoredPriceProvider` is in synthetic two-feed mode with a price ratio below `1e-8` are permanently blocked for the duration of the price condition. The pool's core swap functionality is unusable. LPs can still call `removeLiquidity` (which does not invoke the price provider), but the pool cannot serve its primary purpose. This matches the allowed impact: **"Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."**

---

### Likelihood Explanation

Synthetic pairs with extreme price ratios are realistic in production (any meme-coin/BTC or meme-coin/ETH pair). The condition requires no privileged action — ordinary market price movement can push the ratio below `1e-8`. Once triggered, the DoS persists until the ratio recovers above the threshold. The `_readLeg` staleness and guard checks pass normally; the truncation occurs only in the ratio step, making it invisible to per-leg validation. [5](#0-4) 

---

### Recommendation

Replace the 8-decimal intermediate with an 18-decimal scale to preserve precision across the full realistic price range, and update `_computeBidAsk` / `_bandEdge` to account for the new scale:

```solidity
// Before (line 267):
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // 1e8 precision — truncates to 0

// After:
mid = Math.mulDiv(mid, 1e18, mid2);              // 1e18 precision — survives ratios down to ~1e-18
```

All downstream consumers of `mid` in `_computeBidAsk` and `_bandEdge` (specifically `STEP_DENOM = ORACLE_DECIMALS * BPS_BASE_U`) must be updated to match the new scale. As a minimal defensive patch, add an explicit zero-check immediately after the ratio:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max);  // explicit fail-closed, not silent truncation
```

### Proof of Concept

1. Deploy `AnchoredPriceProvider` with `baseFeedId = PEPE/USD`, `quoteFeedId = BTC/USD`.
2. Configure the mock oracle so PEPE/USD returns `mid = 1137`, `spreadBps = 1`, valid `refTime`; BTC/USD returns `mid = 12_000_000_000_000`, `spreadBps = 1`, valid `refTime`. Both pass all `_readLeg` guards.
3. Call `getBidAndAskPrice()`.
4. Observe: `Math.mulDiv(1137, 1e8, 12_000_000_000_000) = 0`; `_computeBidAsk(0, 2)` returns `(0, type(uint128).max)`; call reverts with `FeedStalled`.
5. Any `pool.swap(...)` using this provider also reverts, permanently blocking all swaps on the pool.

    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }

    function _computeBidAsk(uint256 mid, uint256 spreadBps)
        internal view returns (uint128, uint128)
    {
        // Circuit breaker: extreme (combined) uncertainty means the feed is clearly broken.
        if (spreadBps > MAX_SPREAD_BPS) {
            return (0, type(uint128).max);
        }

        // Reference band: mid ± (spreadBps + minMargin), bid rounded down, ask rounded up.
