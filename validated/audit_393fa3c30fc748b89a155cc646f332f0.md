### Title
Checkpoint Advances Before Pool Balance Check in `processEventsAndClaimBonus` Causes Permanent Bonus Reward Loss on Redelegation Slash — (`x/tieredrewards/keeper/bonus_rewards.go`, `slash.go`, `claim_rewards.go`)

### Summary

In `processEventsAndClaimBonus`, the position's `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded` checkpoints are mutated **before** the `sufficientBonusPoolBalance` check. When `ErrInsufficientBonusPool` is returned, `slashRedelegationPosition` explicitly swallows the error and then calls `setPositionWithState`, persisting the advanced checkpoints. The pre-slash bonus window is permanently closed; no future claim can recover it.

### Finding Description

**Step 1 — Entry point:**
`BeforeRedelegationSlashed` (hooks.go:128) calls `slashRedelegationPosition` (slash.go:19).

**Step 2 — Bonus settlement attempt:**
`slashRedelegationPosition` calls `processEventsAndClaimBonus(ctx, &pos)` at slash.go:54.

**Step 3 — Checkpoint mutation before pool check in `processEventsAndClaimBonus`:**

Inside the event loop (claim_rewards.go:172–198), `pos.UpdateLastEventSeq(entry.Seq)` is called for every processed event. [1](#0-0) 

After the loop, **before** any pool balance check:
- `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `pos.LastBonusAccrual` to `blockTime` [2](#0-1) 
- `pos.UpdateLastKnownBonded(bonded)` updates the bonded state [3](#0-2) 

Only **after** these mutations does the pool check occur:
```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err  // returns ErrInsufficientBonusPool
}
``` [4](#0-3) 

When the pool is empty, the function returns `(nil, ErrInsufficientBonusPool)` with `pos` already mutated in memory.

**Step 4 — Error swallowed, mutated position persisted:**

Back in `slashRedelegationPosition`, the `ErrInsufficientBonusPool` branch is explicitly caught and logged but **not returned**: [5](#0-4) 

Execution then falls through to `setPositionWithState`, which persists the position with the already-advanced `LastBonusAccrual`, `LastEventSeq`, and `LastKnownBonded`: [6](#0-5) 

`setPositionWithState` unconditionally writes the position to the store: [7](#0-6) 

**Step 5 — Permanent loss:**

When the pool is later refilled and the owner calls `ClaimTierRewards`, `processEventsAndClaimBonus` reads `pos.LastBonusAccrual` (now = slash blockTime) and `pos.LastEventSeq` (now past all pre-slash events) as its starting point. The pre-slash accrual window is permanently closed. No bonus for that period can ever be claimed.

### Impact Explanation

Position owners permanently lose all bonus rewards accrued up to the slash block. The `LastBonusAccrual` checkpoint is advanced to `blockTime` without any coins being transferred, and the event sequence is consumed. This is an irreversible fund loss for the position owner — the rewards are neither paid nor recoverable.

### Likelihood Explanation

The `RewardsPoolName` balance being zero or below the computed bonus is a realistic operational condition (pool not yet funded, drained by prior claims, or intentionally empty at chain start). Validator redelegation slashes are normal chain events. The combination is reachable without any privileged access — a standard redelegation slash on a validator with tiered-reward positions is sufficient.

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the `sufficientBonusPoolBalance` check and the `SendCoinsFromModuleToAccount` call succeeds. Checkpoints must only advance when the corresponding coins are confirmed transferred. Alternatively, when `ErrInsufficientBonusPool` is caught in `slashRedelegationPosition`, the position should be persisted with its **original** (pre-call) checkpoint values, not the mutated ones.

### Proof of Concept

1. Create a position with an active delegation to a validator.
2. Drain `RewardsPoolName` to zero balance.
3. Trigger a redelegation slash via `BeforeRedelegationSlashed` (unbondingId mapped to the position).
4. Observe: `slashRedelegationPosition` logs "insufficient bonus pool" and continues; `setPositionWithState` persists `pos.LastBonusAccrual = blockTime`.
5. Refill `RewardsPoolName` with sufficient funds.
6. Call `ClaimTierRewards` for the position owner.
7. Assert: owner receives **zero** bonus for the pre-slash period. The `LastBonusAccrual` already points past it, so `computeSegmentBonus` computes zero for any segment before `blockTime`.
