### Title
Contracts with function bodies > 196,608 bytes deployed at protocol v83 (prepare_v2, no body-size check) become permanently unexecutable after upgrade to v84 (prepare_v3, `max_function_body_size = 196_608`) — (`runtime/near-vm-runner/src/prepare.rs`, `prepare_v3.rs`)

---

### Summary

The prepare-path switch from `prepare_v2` to `prepare_v3` is gated on `config.reftypes_bulk_memory`. At protocol v83 that flag is `false` (prepare_v2, no `max_function_body_size` check). At protocol v84 it becomes `true` (prepare_v3, `max_function_body_size = 196_608`). A contract with a single function body of 200,000 bytes is within the 4 MB `max_contract_size` at v83 and is accepted by prepare_v2, but is permanently rejected by prepare_v3 at v84 with `PrepareError::FunctionBodyTooLarge`. The contract becomes unexecutable for all callers after the upgrade.

### Impact Explanation

Any contract deployed at v83 whose largest function body exceeds 196,608 bytes is permanently unexecutable after the network upgrades to v84. Because NEAR re-prepares contracts at call time using the current epoch's `Config`, there is no path to execute the contract after the upgrade. All callers receive a preparation error. The contract's storage and balance are locked but inaccessible.

---

### Likelihood Explanation

The `max_contract_size` at v83 is 4,194,304 bytes (4 MB). A single function body of 200,000 bytes is well within that limit and is a realistic size for a hand-written or compiler-generated contract. The scenario requires only an ordinary `DeployContract` action — no privileged access. The upgrade from v83 to v84 is a supported, documented upgrade path.

---

### Recommendation

When introducing a new per-function limit in a new prepare path, either:
1. Grandfather existing contracts by checking the limit only for contracts deployed at or after the protocol version that introduced the limit (using the contract's stored deploy-time protocol version), or
2. Scan all deployed contracts during the migration and reject or warn about those that would fail the new limit before the upgrade activates, or
3. Set the initial value of `max_function_body_size` high enough (e.g., equal to `max_contract_size`) so that no contract previously accepted by prepare_v2 can be rejected by prepare_v3.

### Proof of Concept

```rust

// Deploy it at protocol v83 (reftypes_bulk_memory=false → prepare_v2).
// Confirm prepare_contract returns Ok(_).

// The same stored wasm bytes are re-prepared at call time.
let config_v84 = /* protocol 84 config: reftypes_bulk_memory=true, max_function_body_size=196_608 */;
let result = prepare_contract(&wasm, &config_v84, kind);
assert_matches!(result, Err(PrepareError::FunctionBodyTooLarge));
// Contract is now permanently unexecutable.
```
