### Title
Linked Signer Can Escalate Privileges by Signing `LinkSigner` Transactions to Hijack Subaccount Signing Authority — (`core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is processed with `allowLinkedSigner = true`. This means the **currently linked signer** can sign a `LinkSigner` transaction to replace the linked signer with any arbitrary address — including an attacker-controlled one — without the subaccount owner's consent. This is an unauthorized escalation of signing authority over another user's subaccount.

---

### Finding Description

When the sequencer submits a `LinkSigner` transaction via `processTransactionImpl`, the validation call passes `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to the verifier:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

The verifier accepts the signature if it comes from either the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the **current linked signer** can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any address.

By contrast, the slow-mode path for `LinkSigner` correctly restricts this to the subaccount owner only, using `validateSender` which checks `address(uint160(bytes20(txn.sender))) == sender` (i.e., the `msg.sender` must be the owner address):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast path has no equivalent owner-only restriction.

---

### Impact Explanation

Once the attacker becomes the new linked signer, they can sign any transaction type that accepts `allowLinkedSigner = true`, including:

- **`LiquidateSubaccount`** (line 402): Sign liquidation transactions as the victim-liquidator.
- **`MintNlp` / `BurnNlp`** (lines 539–573): Force the victim to mint or burn NLP tokens, incurring fees and locking/unlocking capital at attacker-chosen times.
- **`LinkSigner`** again (line 581): Maintain persistent access by re-setting the linked signer to themselves, preventing the owner from revoking access.
- **Order signing** (via `MatchOrders`): The attacker's address is passed as `takerLinkedSigner` / `makerLinkedSigner` in `MatchOrdersWithSigner`. The attacker can sign orders at unfavorable prices. If the attacker also controls a counterparty subaccount, they can drain the victim's quote balance through adversarial fills. [5](#0-4) 

The most severe impact is the order-matching vector: the attacker signs a sell order for the victim at a price far below market, matches it against an attacker-controlled buy order, and extracts the victim's collateral as quote profit on the counterparty side.

---

### Likelihood Explanation

Linked signers are explicitly designed for automated trading bots and API key delegation. A compromised API key, a malicious third-party trading service, or a rogue bot operator can exploit this without any on-chain precondition beyond already being set as a linked signer. The sequencer processes valid signed transactions without distinguishing whether the signer is the owner or a linked signer — it has no reason to reject a well-formed `LinkSigner` transaction signed by the current linked signer.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner should be permitted to change the linked signer, consistent with the slow-mode path behavior:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
-   true
+   false  // only the subaccount owner may change the linked signer
);
``` [6](#0-5) 

---

### Proof of Concept

1. Owner of subaccount `S` sets `linkedSigners[S] = botAddress` (e.g., a trading bot API key).
2. The bot (or an attacker who compromised the bot key) constructs a `SignedLinkSigner` transaction with `tx.sender = S`, `tx.signer = attackerAddress`, signed by `botAddress`.
3. The bot submits this to the sequencer. The sequencer calls `processTransactionImpl` → `validateSignedTx(..., true)` → signature validates because `recovered == botAddress == getLinkedSigner(S)`.
4. `linkedSigners[S]` is overwritten with `attackerAddress`.
5. The attacker now signs a `MatchOrders` taker order for `S` at a price far below market (e.g., selling 1 ETH for $1 USDC).
6. The attacker's own subaccount `S2` is the maker, buying 1 ETH for $1 USDC.
7. The sequencer processes the match. `S` loses the ETH; `S2` gains it at near-zero cost.
8. The attacker repeats step 5–7 until `S`'s collateral is drained.
