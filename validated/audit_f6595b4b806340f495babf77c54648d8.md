### Title
Bonus Accrual Checkpoint Advances Before Pool Check in `BeforeRedelegationSlashed` Hook, Permanently Erasing Accrued Rewards — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `processEventsAndClaimBonus`, `applyBonusAccrualCheckpoint` unconditionally advances `pos.LastBonusAccrual` to `blockTime` **before** the `sufficientBonusPoolBalance` check. When the pool is insufficient, `ErrInsufficientBonusPool` is returned with the checkpoint already mutated in the in-memory `pos` struct. In `slashRedelegationPosition`, that error is deliberately swallowed (chain-halt avoidance), and `setPositionWithState` then persists the advanced checkpoint. The entire accrual window `[original LastBonusAccrual, blockTime]` is permanently erased with no coins transferred to the owner and no way to recover them after the pool is refilled.

---

### Finding Description

**Step 1 — Checkpoint advances before pool check** [1](#0-0) 

`applyBonusAccrualCheckpoint` mutates `pos.Position.LastBonusAccrual` at line 215. The pool balance check (`sufficientBonusPoolBalance`) only runs at line 230. If the pool is short, `return nil, err` fires at line 231 — but the checkpoint mutation has already happened in the caller's `pos` struct (passed as `*types.PositionState`).

**Step 2 — Error swallowed, mutated position persisted** [2](#0-1) 

`slashRedelegationPosition` calls `processEventsAndClaimBonus(ctx, &pos)` at line 54. When `ErrInsufficientBonusPool` is returned, lines 56–63 log and swallow it. Execution falls through to line 77 (`setPositionWithState`), which persists `pos` — including the already-advanced `LastBonusAccrual`.

**Step 3 — Persistence confirms the overwrite** [3](#0-2) 

`setPositionWithState` calls `k.Positions.Set(ctx, pos.Id, pos)` at line 139, writing the advanced `LastBonusAccrual` to the store. On the next call to `processEventsAndClaimBonus`, `segmentStart` is initialized from the now-advanced `pos.LastBonusAccrual`, so the erased window is never recomputed.

**Step 4 — `applyBonusAccrualCheckpoint` definition** [4](#0-3) 

The function unconditionally calls `pos.UpdateLastBonusAccrual(accrualEnd)` with no guard for whether the corresponding coins were actually transferred.

---

### Impact Explanation

Any position that is in a redelegating state when `BeforeRedelegationSlashed` fires and the `RewardsPoolName` balance is below the computed bonus amount permanently loses all bonus rewards accrued since its last checkpoint. After the pool is replenished, a subsequent `MsgClaimTierRewards` call will compute zero bonus for the erased window because `segmentStart` now equals `blockTime` of the slash. The owner receives nothing for that period and has no recourse.

---

### Likelihood Explanation

The `RewardsPoolName` pool is a finite module account funded by governance or protocol allocation. Under sustained usage it can be partially or fully depleted. Validator slashing (double-sign, downtime) is a routine chain event. The intersection — a redelegating position exists when a slash fires and the pool is below the accrued bonus — is a realistic operational condition, not a contrived one. No attacker capability beyond holding a tier position on a validator that gets slashed is required.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` (and `pos.UpdateLastKnownBonded`) to **after** the successful `SendCoinsFromModuleToAccount` call, so the checkpoint only advances when coins are actually transferred. In the insufficient-pool branch of `slashRedelegationPosition`, do not call `setPositionWithState` with the mutated `pos`; instead reload the unmodified position from the store (or preserve a pre-mutation copy) before persisting.

---

### Proof of Concept

```
1. Create a tier position P and redelegate it (so a RedelegationMapping entry exists).
2. Advance block time by 30 days so bonus accrues.
3. Drain RewardsPoolName to zero (e.g., via repeated ClaimTierRewards on other positions).
4. Call Hooks().BeforeRedelegationSlashed(ctx, unbondingID, smallShares).
   → processEventsAndClaimBonus advances pos.LastBonusAccrual to blockTime (line 215),
     then returns ErrInsufficientBonusPool (line 231).
   → slashRedelegationPosition swallows the error (lines 56-63).
   → setPositionWithState persists the advanced LastBonusAccrual (line 77 → position.go:139).
5. Refund RewardsPoolName with sufficient tokens.
6. Call MsgClaimTierRewards for position P.
   → processEventsAndClaimBonus sets segmentStart = pos.LastBonusAccrual = blockTime of slash.
   → computeSegmentBonus([blockTime, blockTime]) = 0.
   → Owner receives zero bonus despite 30 days of accrual.
```

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {

	}
