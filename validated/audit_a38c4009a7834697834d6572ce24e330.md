### Title
`verify_transaction_inclusion` Lacks Coinbase Merkle Proof Validation Present in `verify_transaction_inclusion_v2` - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (v1) is still publicly callable and lacks the coinbase Merkle proof validation that `verify_transaction_inclusion_v2` introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability. Any unprivileged NEAR caller can invoke v1 directly, bypassing the protection entirely and obtaining a `true` result for a fabricated transaction inclusion claim.

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte Merkle proof forgery attack (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring the caller to also supply a coinbase Merkle proof that anchors the tree at index 0: [1](#0-0) 

The coinbase proof check forces the caller to demonstrate knowledge of the real coinbase transaction at position 0, which makes it impossible to substitute an internal Merkle node as a fake `tx_id`.

`verify_transaction_inclusion` (v1) performs no such check. It only verifies that the supplied `merkle_proof` is non-empty and that `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` equals the stored Merkle root: [2](#0-1) 

`compute_root_from_merkle_proof` in `merkle-tools` simply walks the proof path from the supplied hash upward: [3](#0-2) 

Because any internal Merkle node is itself a valid intermediate hash in the tree, an attacker can supply an internal node as `tx_id` together with a valid sub-path proof, and the function will compute the correct Merkle root and return `true`. The function's own documentation acknowledges this: [4](#0-3) 

Despite being marked `#[deprecated]`, v1 carries no access-control restriction. The `#[deprecated]` attribute is a Rust compiler hint only; it imposes no runtime barrier. The function remains `pub` and `#[pause]`-gated (pausable by the `PauseManager` role), meaning any unprivileged NEAR account can call it whenever the contract is not paused: [5](#0-4) 

The structural asymmetry is identical to the external report: `verify_transaction_inclusion_v2` has the protection; `verify_transaction_inclusion` does not; both are reachable from the same public surface.

### Impact Explanation

A dApp or bridge that calls `verify_transaction_inclusion` to gate a high-value action (e.g., releasing funds, minting tokens) can be deceived into believing an arbitrary non-existent transaction was confirmed in a real mainchain block. The attacker does not need to mine any block or control any privileged role. All required inputs (a real mainchain block hash, its Merkle tree structure, and an internal node) are publicly observable on-chain.

### Likelihood Explanation

The mainchain Merkle tree of every accepted block is public. Constructing the required internal-node proof requires only off-chain computation against publicly available block data. The call to v1 is a single NEAR transaction with no deposit or staking requirement. Any actor with a NEAR account can execute this at negligible cost.

### Recommendation

Remove the public accessibility of `verify_transaction_inclusion`. Either:
1. Change its visibility from `pub` to `pub(crate)` or remove it entirely, forcing all callers to use `verify_transaction_inclusion_v2`; or
2. Add the same coinbase Merkle proof validation to v1 before the existing Merkle root comparison.

The deprecation notice alone is insufficient because it does not prevent external callers from invoking the function.

### Proof of Concept

1. Identify any mainchain block `B` with at least two transactions. Its Merkle tree has at least one internal node `N = SHA256d(tx0 || tx1)`.
2. Observe that `N` sits at position 0 in the next level of the tree. Construct a Merkle proof `proof` from `N` up to the Merkle root (this is just the remaining sibling hashes in the tree, all publicly known).
3. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction hash)
   - `tx_block_blockhash = B`
   - `tx_index = 0` (position of `N` in its level)
   - `merkle_proof = proof`
   - `confirmations = 1`
4. `compute_root_from_merkle_proof(N, 0, proof)` walks the proof and produces the correct Merkle root of `B`.
5. The function returns `true`, falsely asserting that the non-existent "transaction" `N` is included in block `B`. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
