### Title
ML-DSA-65 `TransferToGasKey` / `WithdrawFromGasKey` send-fee uses trie-id length (33 B) instead of full pubkey length (1953 B), undercharging by ~59× - (File: `runtime/runtime/src/config.rs`)

---

### Summary

After `PostQuantumSignatures` activates, the send-fee charged for a `TransferToGasKey` (or `WithdrawFromGasKey`) action carrying an ML-DSA-65 public key is computed from the 33-byte on-trie hash identifier (`trie_id_len()`) rather than the 1953-byte borsh-encoded full public key that actually appears in the action. The exec-fee path in the same file correctly uses `trie_id_len()` (the trie key is the hash), but the send-fee path should use the full pubkey length because the send fee covers the cost of processing the action on the sender's shard, where the full 1953-byte key is present. The host-function path (`promise_batch_action_transfer_to_gas_key`) passes `public_key_len as usize` (1953 bytes) to the same `gas_key_transfer_send_fee` helper, so the two code paths diverge by a factor of ≈59× for ML-DSA-65 keys.

---

### Finding Description

`gas_key_transfer_send_fee` is documented as "Based on the public key length (what the sender sees)": [1](#0-0) 

The per-byte component is `gas_key_byte.send_fee × public_key_len`. For ed25519 and secp256k1, `trie_id_len()` equals the borsh length (33 and 65 bytes respectively), so the two representations agree. For ML-DSA-65, they diverge:

| representation | bytes |
|---|---|
| `PublicKey::MLDSA65.trie_id_len()` | 33 |
| `PublicKey::MLDSA65.len()` (borsh) | 1953 |

**Transaction path** — `total_send_fees` in `runtime/runtime/src/config.rs`:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
``` [2](#0-1) 

This passes **33 bytes** for ML-DSA-65.

**Host-function path** — `promise_batch_action_transfer_to_gas_key` in `runtime/near-vm-runner/src/logic/logic.rs`:

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
``` [3](#0-2) 

This passes **1953 bytes** for ML-DSA-65 (the borsh-encoded length the contract supplied).

The exec-fee path in `config.rs` correctly uses `trie_id_len()` because the receiver reads/writes the 33-byte hash in the trie: [4](#0-3) 

So the developer correctly applied `trie_id_len()` to the exec fee but also applied it to the send fee, where the full pubkey length is the right measure. The same pattern applies to `WithdrawFromGasKey`.

The `trie_id_len()` / `len()` split is explicitly documented as a new contract that all cost code must respect: [5](#0-4) 

The `access_key_storage_usage` function in `runtime/runtime/src/access_keys.rs` correctly uses `trie_id_len()` for storage-stake (the trie key is the hash), confirming that `trie_id_len()` is the right choice for trie-side costs and `len()` is the right choice for wire-side costs: [6](#0-5) 

---

### Impact Explanation

The send fee for a `TransferToGasKey` action with an ML-DSA-65 key is undercharged by a factor of 1953 / 33 ≈ **59×** when submitted as a direct transaction. Any account that holds an ML-DSA-65 gas key (possible once `PostQuantumSignatures` is active) can fund that key or withdraw from it at a tiny fraction of the intended gas cost. Because the send fee is burned at transaction-to-receipt conversion, the undercharge is a direct protocol-level loss: the sender pays ≈1.7% of the gas that should be burned to cover the sender-shard processing cost of the 1953-byte pubkey. At scale this is a cheap spam / DoS vector against the sender shard.

The exec fee is unaffected (correctly based on the 33-byte trie key). The host-function path has the opposite asymmetry (exec fee overcharged, send fee correct), so contracts calling `promise_batch_action_transfer_to_gas_key` are not exposed to the undercharge.

---

### Likelihood Explanation

The `PostQuantumSignatures` feature must be active (protocol version ≥ 85) and the signer must hold an ML-DSA-65 gas key. Both conditions are reachable by any unprivileged user after feature activation: `AddKey` with `GasKeyFullAccess` and an ML-DSA-65 public key is the only prerequisite. No validator or operator privilege is required.

---

### Recommendation

In `total_send_fees` (`runtime/runtime/src/config.rs`), replace `trie_id_len()` with `len()` for the send-fee argument of `gas_key_transfer_send_fee` for both `TransferToGasKey` and `WithdrawFromGasKey`:

```diff
 TransferToGasKey(action) => {
-    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
+    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.len())
         .total()
 }
 WithdrawFromGasKey(action) => {
-    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
+    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.len())
         .total()
 }
```

/// Send fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the public key length (what the sender sees).
pub fn gas_key_transfer_send_fee(
    cfg: &RuntimeFeesConfig,
    sender_is_receiver: bool,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).send_fee(sender_is_receiver);
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .send_fee(sender_is_receiver)
        .checked_mul(public_key_len as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

fn access_key_storage_usage(
    fee_config: &RuntimeFeesConfig,
    public_key: &PublicKey,
    access_key: &AccessKey,
) -> StorageUsage {
    let storage_usage_config = &fee_config.storage_usage_config;
    // Use the on-trie identifier length, not the borsh-serialized pubkey
    // length: ML-DSA-65 access keys live in the trie as a SHA3-256 hash
    // (33 bytes incl. type tag), not as a 1953-byte full pubkey.
    public_key.trie_id_len() as u64
        + borsh::object_length(access_key).unwrap() as u64
        + storage_usage_config.num_extra_bytes_record
}
