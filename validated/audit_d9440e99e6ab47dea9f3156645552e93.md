### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain from Any DDA — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw `.transferFrom()` call at line 616 whose boolean return value is never checked. Because the function is callable by any unprivileged user and the `ERC20Helper.safeTransferFrom` wrapper already exists in the codebase, a caller who provides zero USDC (or has no allowance set) can silently pass the inbound transfer step and still receive the usdcE held in any victim DDA.

---

### Finding Description

`replaceUsdcEWithUsdc` is a public, permissionless function (no `onlyOwner` modifier) that is intended to swap usdcE held in a `DirectDepositV1` (DDA) contract for real USDC supplied by the caller:

```
ContractOwner.sol line 608-620

function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ← unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

The three-step sequence is:
1. Pull `balance` USDC from `msg.sender` into the DDA.
2. Withdraw all usdcE from the DDA to `ContractOwner` (via `DirectDepositV1.withdraw`, which sends to `msg.sender` of that call — i.e., `ContractOwner`).
3. Forward the usdcE from `ContractOwner` to `msg.sender`.

Step 1 uses the raw interface call `IERC20Base(usdc).transferFrom(...)`. The `IERC20Base` interface declares `transferFrom` as returning `bool`, but the return value is discarded. If the USDC token at that address returns `false` on failure (insufficient balance, no allowance, or any other condition) rather than reverting, execution continues silently into steps 2 and 3, which do succeed — draining usdcE from the DDA to the attacker at no cost.

The project already has `ERC20Helper.safeTransferFrom` which wraps the low-level call and enforces the return value:

```
ERC20Helper.sol lines 23-42

function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.transferFrom.selector, from, to, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

This wrapper is used correctly everywhere else in the codebase (e.g., `EndpointStorage.safeTransferFrom`, `BaseWithdrawPool.safeTransferFrom`) but was omitted at this call site.

---

### Impact Explanation

Any user on chain 57073 (Ink) can call `replaceUsdcEWithUsdc` targeting any subaccount whose DDA holds usdcE. If the USDC token returns `false` on a failed transfer (a well-known non-reverting ERC-20 pattern), the attacker receives the full usdcE balance of the DDA without depositing any USDC. The usdcE belongs to the subaccount owner and represents real collateral value. The corrupted asset delta is: attacker gains `balance` usdcE; DDA loses `balance` usdcE; no USDC is deposited in exchange.

---

### Likelihood Explanation

The function is externally callable by any address with no privilege requirement. The only gate is `block.chainid == 57073`. The USDC token at the hardcoded address is a deployment-specific contract; if it follows the non-reverting pattern (returning `false` on failure) rather than the reverting pattern, the exploit is trivially executable with zero USDC balance or allowance. Even if the current deployment reverts, the unchecked pattern is a latent risk that activates if the token is ever upgraded or replaced.

---

### Recommendation

Replace the raw `.transferFrom()` call at line 616 with the project's existing `safeTransferFrom` wrapper from `ERC20Helper`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports `ERC20Helper` and applies `using ERC20Helper for IERC20Base` at line 24, so the fix is a one-word change.

---

### Proof of Concept

1. Attacker identifies any `subaccount` whose DDA holds usdcE (readable via `IERC20Base(usdcE).balanceOf(directDepositV1Address[subaccount])`).
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC balance and zero allowance set.
3. `IERC20Base(usdc).transferFrom(attacker, dda, balance)` returns `false` (no revert).
4. `DirectDepositV1(dda).withdraw(usdcE)` executes — usdcE moves from DDA → ContractOwner.
5. `IERC20Base(usdcE).safeTransfer(attacker, balance)` executes — usdcE moves from ContractOwner → attacker.
6. Attacker holds `balance` usdcE; DDA is drained; no USDC was ever deposited. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
