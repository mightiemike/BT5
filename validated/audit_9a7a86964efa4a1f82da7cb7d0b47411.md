### Title
`ValidResourceBounds` Variant Normalization in P2P Protobuf Deserialization Produces Divergent Transaction Hash Preimage — (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The P2P sync protobuf deserializer for `ValidResourceBounds` silently normalizes an `AllResources` variant with zero `l2_gas` and `l1_data_gas` into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` includes a different number of resource felts in the Poseidon preimage depending on the variant (3 for `L1Gas`, 4 for `AllResources`), a V3 transaction originally hashed under `AllResources` will produce a different transaction hash when its resource bounds are re-read from the P2P-deserialized form. Any component that recomputes the hash from the deserialized `Transaction` object — re-execution, proving, or hash-based RPC lookups — will bind the wrong hash, causing signature verification failure or wrong authoritative values.

---

### Finding Description

**Divergent hash preimage in `get_tip_resource_bounds_hash`**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the Poseidon preimage differently for the two `ValidResourceBounds` variants:

```
L1Gas variant      → [tip, L1_GAS_packed, L2_GAS_packed(0)]          // 3 elements
AllResources(0,0)  → [tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]  // 4 elements
``` [1](#0-0) 

The two variants therefore produce distinct Poseidon digests even when every numeric field is identical.

**Normalization in the P2P protobuf deserializer**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in the P2P sync converter collapses `AllResources` with zero `l2_gas` and `l1_data_gas` into `L1Gas`:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`) always emits a zero `l1_data_gas` field for the `L1Gas` variant: [3](#0-2) 

So a transaction that entered the system as `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` round-trips through protobuf as `L1Gas(l1_gas=X)`.

**Contrast with the mempool/consensus protobuf path**

The mempool/consensus converter (`rpc_transaction.rs`) uses a separate `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` that always produces `AllResourceBounds` and never normalizes: [4](#0-3) 

The two converters are therefore inconsistent: the same wire bytes produce `AllResources` on the consensus path and `L1Gas` on the P2P sync path.

**Hash recomputation sites**

`Transaction::calculate_transaction_hash` dispatches to `get_invoke_transaction_v3_hash`, which calls `get_tip_resource_bounds_hash` with the stored `ValidResourceBounds` variant: [5](#0-4) 

The `blockifier_reexecution` state reader explicitly recomputes the hash from the deserialized `Transaction`: [6](#0-5) 

Any proving or re-execution component that follows the same pattern will compute H2 ≠ H1.

---

### Impact Explanation

A V3 transaction with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` is accepted by the gateway and assigned hash H1 (4-element Poseidon preimage). After P2P sync deserialization the same transaction is represented as `L1Gas(l1_gas=X)`. Any component that recomputes the hash from this deserialized object produces H2 (3-element preimage), H2 ≠ H1. Consequences:

1. **Wrong hash bound to the executable transaction** — the blockifier passes `tx_hash` to the account's `__validate__` entry point. If H2 is used, the ECDSA check over H1 fails, causing the transaction to revert or the re-execution to produce a wrong receipt.
2. **Wrong authoritative hash in RPC responses** — a syncing node that stores or serves H2 returns an incorrect value for `starknet_getTransactionByHash`, matching the "RPC returns an authoritative-looking wrong value" impact.
3. **Proving failure** — the OS/prover re-executes the block using the deserialized transaction; the wrong hash propagates into the execution trace, invalidating the proof.

---

### Likelihood Explanation

The trigger is a V3 transaction with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0`. The gateway's stateless validator accepts such transactions (the test suite explicitly validates `AllResourceBounds` with only `l1_gas` non-zero): [7](#0-6) 

Any user who submits a V3 transaction without specifying L2 or data-gas bounds triggers the condition. The normalization is unconditional and requires no special privilege.

---

### Recommendation

Remove the `L1Gas` normalization from the P2P sync deserializer. When `l1_data_gas` is present in the protobuf message (even as zero), always deserialize as `AllResources`. The TODO comment at line 426 already acknowledges this:

```
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
```

Once legacy 0.13.2 support is dropped, the deserializer should unconditionally produce `AllResources` for any V3 transaction, matching the behavior of the mempool/consensus converter in `rpc_transaction.rs`. Until then, the deserializer should preserve the variant that was serialized rather than collapsing it based on zero-value fields.

---

### Proof of Concept

```
1. Submit starknet_addInvokeTransaction with:
     resource_bounds = { L1_GAS: {max_amount: 1000, max_price: 1},
                         L2_GAS: {max_amount: 0,    max_price: 0},
                         L1_DATA_GAS: {max_amount: 0, max_price: 0} }
   → Gateway stores ValidResourceBounds::AllResources(...)
   → Hash H1 = Poseidon([INVOKE, ver, sender,
                          Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)]),
                          ...])

2. Transaction is included in block B.

3. Peer node syncs block B via P2P protobuf.
   → TryFrom<protobuf::ResourceBounds> for ValidResourceBounds fires:
       l1_data_gas.is_zero() && l2_gas.is_zero()  →  L1Gas(l1_gas)

4. Peer node recomputes hash:
   H2 = Poseidon([INVOKE, ver, sender,
                   Poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)]),   // only 3 elements
                   ...])

5. H1 ≠ H2.

6. Peer node's blockifier passes H2 to __validate__; ECDSA check over H1 fails.
   Re-execution reverts / proof is invalid.
```

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
