### Title
`minDepositRateMultiplierX18` Incorrectly Applied to Borrow Accumulator, Overcharging All Borrowers — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

In `SpotEngineState._updateState()`, the `minDepositRateMultiplierX18` — a deposit-side floor rate — is incorrectly multiplied into `cumulativeBorrowsMultiplierX18`. This causes every borrower to pay the minimum deposit rate on top of their regular borrow rate, inflating borrow costs beyond what the two-slope interest rate model intends.

---

### Finding Description

`SpotEngineState._updateState()` implements a two-slope interest rate model. After computing the regular borrow rate and deposit rate, the function applies a minimum deposit rate floor to guarantee depositors a baseline yield: [1](#0-0) 

```solidity
// apply the min deposit rate
if (minDepositRateX18 != 0) {
    int128 minDepositRatePerSecondX18 = minDepositRateX18.div(...);
    int128 minDepositRateMultiplierX18 = (ONE + minDepositRatePerSecondX18).pow(int128(dt));

    state.cumulativeBorrowsMultiplierX18 = state   // ← WRONG: deposit-side rate applied to borrow accumulator
        .cumulativeBorrowsMultiplierX18
        .mul(minDepositRateMultiplierX18);

    state.cumulativeDepositsMultiplierX18 = state  // ← correct
        .cumulativeDepositsMultiplierX18
        .mul(minDepositRateMultiplierX18);
    ...
}
```

The parameter `minDepositRateX18` is defined in `ISpotEngine.Config` as a deposit-side floor: [2](#0-1) 

Its name and semantic role are unambiguous: it is a minimum rate for depositors, not a borrow rate. Applying it to `cumulativeBorrowsMultiplierX18` is the wrong variable — the borrow accumulator should only be compounded by the rate produced by the two-slope model (lines 70–98). [3](#0-2) 

The analog to the external report is direct: just as `primeRate` was used instead of `optimalRate` in the first slope, here `minDepositRateMultiplierX18` (a deposit-side quantity) is applied to the borrow-side accumulator, corrupting the borrow rate beyond the model's intent.

---

### Impact Explanation

Every user with a negative `amountNormalized` (i.e., a borrower) has their effective debt compounded by both the regular borrow rate **and** `minDepositRateX18`. The actual borrow cost becomes:

```
effectiveBorrowRate = regularBorrowRate + minDepositRate
```

This inflates `cumulativeBorrowsMultiplierX18` beyond the intended value, causing:
- All borrow balances (computed as `amountNormalized * cumulativeBorrowsMultiplierX18`) to be overstated
- Borrowers to repay more than the interest rate model specifies
- Health checks in `Clearinghouse.getHealth()` to use inflated borrow values, potentially triggering premature liquidations [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The bug is triggered on every `updateStates()` call for any product where `minDepositRateX18 != 0`. The sequencer calls this path continuously via `Endpoint.submitTransactions()` → `EndpointTx` → `spotEngine.updateStates()`. No special conditions are required — any borrower in any such product is affected on every state update. [6](#0-5) 

---

### Recommendation

Remove the application of `minDepositRateMultiplierX18` to `cumulativeBorrowsMultiplierX18`. The minimum deposit rate is a depositor-side guarantee and should only compound the deposit accumulator:

```diff
 if (minDepositRateX18 != 0) {
     int128 minDepositRatePerSecondX18 = minDepositRateX18.div(...);
     int128 minDepositRateMultiplierX18 = (ONE + minDepositRatePerSecondX18).pow(int128(dt));

-    state.cumulativeBorrowsMultiplierX18 = state
-        .cumulativeBorrowsMultiplierX18
-        .mul(minDepositRateMultiplierX18);

     state.cumulativeDepositsMultiplierX18 = state
         .cumulativeDepositsMultiplierX18
         .mul(minDepositRateMultiplierX18);

     depositRateMultiplierX18 = depositRateMultiplierX18.mul(minDepositRateMultiplierX18);
-    borrowRateMultiplierX18 = borrowRateMultiplierX18.mul(minDepositRateMultiplierX18);
 }
```

If the protocol intends to fund the minimum deposit rate from its own reserves rather than from borrowers, the shortfall should be sourced from the insurance fund or fee balance, not by inflating `cumulativeBorrowsMultiplierX18`.

---

### Proof of Concept

1. Deploy `SpotEngine` and configure a product with `minDepositRateX18 = 1e16` (1% annualized).
2. Have a user deposit and another user borrow at low utilization (e.g., 10%), where the regular borrow rate is ~0.5% annualized.
3. Call `updateStates(dt)` for one year (`dt = 31536000`).
4. Observe that `cumulativeBorrowsMultiplierX18` has compounded at ~1.5% (0.5% regular + 1% min deposit), not the intended ~0.5%.
5. The borrower's effective debt is 3× the intended interest cost — the minimum deposit rate is being charged to borrowers who have no relationship to the deposit floor guarantee. [7](#0-6)
