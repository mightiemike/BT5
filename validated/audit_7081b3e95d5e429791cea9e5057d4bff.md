### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed`, Making `overall_fee` Irreconcilable with Documented Formula for V3 Transactions - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is populated from the actual execution receipt and therefore **does** include L2 gas costs, but the documented formula (`overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`) does not account for L2 gas. For any V3 transaction that consumes non-zero L2 gas, the formula produces a value strictly lower than `overall_fee`, making the response internally inconsistent.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                   // price present, amount absent
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is the full fee charged by the blockifier, which for V3 (`AllResources`) transactions equals:

```
l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)
```

The struct carries `l2_gas_price` but never sets `l2_gas_consumed` (the field does not exist). The in-code TODO at line 104 acknowledges this:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

The OpenRPC spec at `crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` line 3650 codifies the broken formula as authoritative:

```json
"description": "The estimated fee for the transaction ... equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price"
```

This is the direct analog of the external bug: the aggregate (`overall_fee`) silently incorporates a component (L2 gas cost) that is not reflected in any of the individual breakdown fields returned to the caller, so the formula the spec advertises does not hold.

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` that reconstructs the expected fee from the breakdown fields will compute a value **lower** than `overall_fee` for every V3 transaction with non-zero L2 gas. Concretely:

- A wallet or SDK that verifies `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price == overall_fee` will always fail this check for modern V3 transactions, causing it to either reject a valid estimation or silently trust a value it cannot verify.
- A caller that derives resource bounds from the breakdown (rather than from `overall_fee`) will under-provision L2 gas, causing the submitted transaction to revert with an insufficient-resource-bounds error.
- The `l2_gas_price` field is present but orphaned — it cannot be used to compute anything without the missing `l2_gas_consumed` counterpart, making the field itself misleading.

This matches: **High — RPC fee estimation returns an authoritative-looking wrong value.**

### Likelihood Explanation

Every V3 (`AllResources`) transaction executed on Starknet 0.13.3+ consumes L2 gas. The discrepancy is therefore triggered by the normal, unprivileged use of `starknet_estimateFee` or `starknet_simulateTransactions` with any modern transaction type. No special permissions or adversarial input are required.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it in `tx_execution_output_to_fee_estimation`:

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,
    pub l1_data_gas_price: GasPrice,
    pub l2_gas_consumed: Felt,   // add this
    pub l2_gas_price: GasPrice,
    pub overall_fee: Fee,
    pub unit: PriceUnit,
}

// in tx_execution_output_to_fee_estimation:
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),  // add this
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

Update the OpenRPC spec description to:

```
"equals to gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*(l2_gas_price + tip)"
```

### Proof of Concept

1. Submit any V3 invoke transaction to `starknet_estimateFee`.
2. Receive a `FeeEstimation` response where `l2_gas_price > 0` and `overall_fee > 0`.
3. Compute `reconstructed = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`.
4. Observe `reconstructed < overall_fee` — the gap equals `l2_gas_consumed * (l2_gas_price + tip)`, a value the response provides no way to derive.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
