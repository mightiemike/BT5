### Title
`DISABLE_OP` Flag Check in `op_mod`, `op_div`, and `op_divmod` Is Dead Code, Leaving Mempool-Mode Restriction Unenforced — (`src/more_ops.rs` / `src/chia_dialect.rs`)

---

### Summary

The `DISABLE_OP` flag is part of `MEMPOOL_MODE` and is intended to restrict certain arithmetic operators when running in mempool (stricter) mode. For opcode 60 (`op_modpow`), the flag correctly and completely blocks the operator at the dispatch level in `chia_dialect.rs`. However, for opcodes 19/20/61 (`op_div`, `op_divmod`, `op_mod`), the `DISABLE_OP` check inside each operator function uses a threshold of `a0_len > 2048`, which is larger than the unconditional size limit of `a0_len > 256` that follows immediately after. This makes every `DISABLE_OP` check in those three functions permanently dead code: the flag has zero effect on their behavior, and they execute identically in mempool mode and consensus mode.

---

### Finding Description

**Dispatch-level gate for `op_modpow` (opcode 60) — correct:**

In `src/chia_dialect.rs`, `op_modpow` is completely blocked when `DISABLE_OP` is set:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;
    }
    op_modpow
}
61 => op_mod,   // dispatched unconditionally — no gate here
``` [1](#0-0) 

**Dead-code `DISABLE_OP` check inside `op_mod` (opcode 61):**

```rust
// threshold 2048 — never reached because the check below fires at 256
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "mod".to_string()));
}
// unconditional limit — always fires first for any a0_len > 256
if a0_len > 256 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "mod".to_string()));
}
``` [2](#0-1)

### Impact Explanation

The `DISABLE_OP` flag is structurally asymmetric: it completely gates `op_modpow` at the dispatch level, but the analogous checks inside `op_mod`, `op_div`, and `op_divmod` are unreachable dead code. Any CLVM program submitted to the mempool that uses `op_mod`, `op_div`, or `op_divmod` with inputs up to 256 bytes is accepted by the mempool under exactly the same rules as consensus mode. If the intent of `DISABLE_OP` was to impose a stricter mempool-only restriction on these operators (e.g., a lower size threshold, or a complete block like `op_modpow`), that restriction is silently absent. A crafted CLVM coin spend can exploit this to submit computations via `op_mod`/`op_div`/`op_divmod` that the mempool was supposed to reject, while `op_modpow` with equivalent computational load is correctly blocked.

---

### Likelihood Explanation

The entry path is direct and attacker-controlled: any CLVM program bytes submitted to a Chia full node's mempool via the standard spend-bundle API reach `ChiaDialect::op()` with `MEMPOOL_MODE` flags set. The attacker only needs to craft a spend bundle that uses opcode 61 (`mod`), 19 (`/`), or 20 (`divmod`) with inputs near the 256-byte limit. No special privileges, social engineering, or dependency compromise is required.

---

### Recommendation

1. **Remove the dead-code checks** from `op_mod`, `op_div`, `op_divmod` (and their Malachite variants) if `DISABLE_OP` is not intended to restrict them — the checks add confusion with no effect.
2. **Or fix the threshold**: if the intent is to impose a stricter mempool-only limit, the `DISABLE_OP` threshold must be set to a value strictly less than 256 (the unconditional limit) to have any effect.
3. **Or gate at dispatch**: if the intent is to completely block these operators in mempool mode (matching `op_modpow`'s treatment), add the `DISABLE_OP` guard at the `match op { ... }` arm in `ChiaDialect::op()`, not inside the operator functions.
4. **Add a documentation comment** to `DISABLE_OP` clarifying its intended scope.

---

### Proof of Concept

```
# In mempool mode (DISABLE_OP set), this program is accepted — DISABLE_OP has no effect:
(mod 256-byte-atom 1024-byte-divisor)

# In mempool mode, this equivalent-cost program is correctly rejected:
(modpow base exponent modulus)
```

A test demonstrating the dead code:

```rust
let mut a = Allocator::new();
// 257-byte atom — exceeds unconditional limit, DISABLE_OP check at 2048 never fires
let big = a.new_atom(&[0u8; 257]).unwrap();
let one  = a.new_atom(&[1u8]).unwrap();
let args = a.new_pair(big, a.new_pair(one, a.nil()).unwrap()).unwrap();
// With DISABLE_OP set (mempool mode), op_mod rejects at the unconditional 256-byte check,
// NOT at the DISABLE_OP check — proving the DISABLE_OP branch is dead.
let result = op_mod(&mut a, args, 1_000_000, ClvmFlags::DISABLE_OP);
assert!(result.is_err()); // fails at unconditional check, not DISABLE_OP
```
