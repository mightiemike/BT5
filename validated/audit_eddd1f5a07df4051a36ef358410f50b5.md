Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the actual end-user, allowing any caller to bypass a curated pool's swap gate via `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of the `pool.swap()` call ŌĆö the direct caller, not the economic actor. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract. If the pool admin allowlists the router (the natural operational step to let permitted users use the router), every user ŌĆö including non-allowlisted ones ŌĆö can bypass the curated pool's swap gate by calling the router instead of the pool directly.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ŌåÉ always the direct caller of pool.swap()
```

`SwapAllowlistExtension.beforeSwap` then checks that exact value against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The real user's identity (`msg.sender` of `exactInputSingle`) is stored in transient storage via `_setNextCallbackContext` solely for the payment callback ŌĆö it is never threaded into `extensionData` or any parameter visible to `beforeSwap`. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`.

`DepositAllowlistExtension.beforeAddLiquidity` avoids this problem by checking `owner` ŌĆö an address explicitly supplied by the caller and preserved through the liquidity-adder path ŌĆö rather than `sender`:

```solidity
// DepositAllowlistExtension.sol L32,38
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

No equivalent "actual swapper" identity is threaded through the swap path.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd or otherwise vetted addresses is fully bypassed. Any non-allowlisted user can trade by calling `MetricOmmSimpleRouter` instead of the pool directly, as long as the router itself is allowlisted. The pool receives real token input and delivers real token output to the attacker's chosen `recipient`, so the loss of curation is immediate and fund-impacting: LP providers are exposed to counterparties the pool admin explicitly intended to exclude. This constitutes a broken core pool functionality causing loss of funds and an admin-boundary break where an unprivileged path circumvents the pool admin's access control.

## Likelihood Explanation

The scenario is highly reachable. A pool admin who wants allowlisted users to be able to use the standard router will add the router to `allowedSwapper`. This is the expected operational pattern ŌĆö without it, allowlisted users cannot use the router at all. The moment the router is allowlisted, the bypass is open to everyone. No privileged access, no special token, and no unusual setup is required; any user with the router's ABI can exploit it.

## Recommendation

Thread the actual end-user identity through the swap path, mirroring how `DepositAllowlistExtension` uses `owner`. One approach: add an explicit `swapper` field to the extension payload that the router populates with `msg.sender` before calling the pool, and have `SwapAllowlistExtension` read that field from `extensionData`. Alternatively, gate on `tx.origin` inside the extension, accepting the trade-off that contract-initiated swaps are blocked. A third option is to have the router pass the real user address as the `recipient` and check `recipient` in the extension ŌĆö but this only works if recipient and payer are always the same actor.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` ŌĆö only Alice is permitted.
3. Admin calls `setAllowedToSwap(pool, router, true)` ŌĆö router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` with `pool` as the target.
5. The router calls `pool.swap(recipient=Bob, ...)`. Inside the pool, `msg.sender = router`.
6. The pool calls `_beforeSwap(sender=router, ...)` at [1](#0-0) .
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` ŌåÆ passes at [2](#0-1) .
8. Bob's swap executes at the oracle price, receiving real tokens from the curated pool.

Direct call by Bob (`pool.swap(...)`) would revert with `NotAllowedToSwap` because `allowedSwapper[pool][Bob] == false`. The router path silently bypasses the guard. The real user identity is stored only in transient storage for the payment callback at [3](#0-2)  and never reaches the extension.
