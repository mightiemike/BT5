Audit Report

## Title
Deposit Allowlist Bypassed via Attacker-Controlled `owner` Parameter — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor/payer) and gates only on `owner` (the LP share recipient). Because `owner` is a free caller-controlled parameter in `MetricOmmPool.addLiquidity`, any non-allowlisted address can deposit into a curated pool by nominating an allowlisted address as `owner`, fully bypassing the pool's deposit curation invariant.

## Finding Description

**Root cause:** `DepositAllowlistExtension.beforeAddLiquidity` declares its first parameter unnamed, silently discarding `sender`: [1](#0-0) 

The check `allowedDepositor[msg.sender][owner]` passes when `owner` is any allowlisted address, regardless of who actually called the pool.

**Call path:**

`MetricOmmPool.addLiquidity` accepts `owner` as a raw, unrestricted caller-supplied parameter: [2](#0-1) 

`_beforeAddLiquidity` faithfully forwards both `msg.sender` (the real caller) and `owner` (attacker-chosen) to the extension: [3](#0-2) 

But the extension drops `sender` and checks only `owner`: [4](#0-3) 

**Exploit flow:**
1. Attacker calls `pool.addLiquidity(owner=allowlistedAddr, ...)`
2. `_beforeAddLiquidity(sender=attacker, owner=allowlistedAddr, ...)` is invoked
3. Extension checks `allowedDepositor[pool][allowlistedAddr] == true` → passes
4. `LiquidityLib.addLiquidity` mints LP shares to `allowlistedAddr`
5. The `addLiquidity` callback is sent to `msg.sender` (attacker), who pays the tokens

No existing guard prevents this: `removeLiquidity` enforces `msg.sender == owner` at L206, but `addLiquidity` has no equivalent check. [5](#0-4) 

## Impact Explanation

The pool admin's core invariant — "only allowlisted actors may inject tokens into the pool" — is violated. Any unprivileged EOA or contract can alter the pool's bin composition without being allowlisted. The allowlisted address receives unsolicited LP shares it did not request or pay for. Pool token/share accounting remains internally consistent (no insolvency), but the attacker self-harms (pays tokens, receives no LP shares). This constitutes broken core pool functionality with a real but indirect economic consequence — **Medium** severity.

## Likelihood Explanation

Exploitable by any EOA with zero special privileges. The only prerequisite is knowing one allowlisted address for the target pool, which is publicly readable on-chain via `allowedDepositor`. [6](#0-5) 

## Recommendation

Check `sender` (the actual depositor/payer) instead of `owner` (the LP share recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, gate on both `sender` and `owner` if the intent is that the LP share recipient must also be allowlisted.

## Proof of Concept

```solidity
function test_allowlistBypass() public {
    address allowlisted = makeAddr("allowlisted");
    address attacker    = makeAddr("attacker");

    // Pool admin allowlists only `allowlisted`
    depositExtension.setAllowedToDeposit(address(pool), allowlisted, true);

    // Attacker is NOT allowlisted
    assertFalse(depositExtension.isAllowedToDeposit(address(pool), attacker));

    // Attacker calls addLiquidity with owner = allowlisted
    vm.startPrank(attacker);
    token0.approve(address(pool), type(uint256).max);
    token1.approve(address(pool), type(uint256).max);
    // Does NOT revert — extension passes because allowedDepositor[pool][allowlisted] == true
    pool.addLiquidity(allowlisted, 0, deltas, callbackData, "");
    vm.stopPrank();

    // LP shares minted to `allowlisted`, tokens paid by `attacker`
    // Allowlist invariant violated: attacker injected tokens without being allowlisted
}
```
