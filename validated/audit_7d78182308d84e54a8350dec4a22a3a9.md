### Title
Violation of Checks-Effects-Interactions in `withdrawCollateral` Enables Re-Entrancy Double-Withdrawal via ERC777 Token Hook — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral()` performs the token transfer to the recipient **before** decrementing the subaccount balance in `SpotEngine` and before running the health check. If a supported collateral token implements ERC777-style transfer hooks (or any callback-on-receive mechanism), a recipient contract can re-enter the withdrawal flow during the callback while the on-chain balance still reflects the full pre-withdrawal amount, allowing a double (or repeated) withdrawal of collateral.

---

### Finding Description

In `Clearinghouse.withdrawCollateral()`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` → triggers `token.safeTransfer(to, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer` (line 189), delivering tokens to `sendTo` and firing any ERC777 `tokensReceived` hook on the recipient.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` → balance is decremented **only after** the transfer.
3. **Line 419** — `require(getHealth(sender, healthType) >= 0, ...)` → health check runs **only after** the transfer. [1](#0-0) 

The internal call chain for the transfer is:

- `Clearinghouse.handleWithdrawTransfer` (line 383) calls `token.safeTransfer(withdrawPool, amount)`, then (line 384) calls `BaseWithdrawPool.submitWithdrawal`.
- `BaseWithdrawPool.submitWithdrawal` (line 131) calls `handleWithdrawTransfer(token, sendTo, amount)`.
- `BaseWithdrawPool.handleWithdrawTransfer` (line 189) calls `token.safeTransfer(to, amount)` — this is the point where an ERC777 `tokensReceived` hook fires on `sendTo`. [2](#0-1) [3](#0-2) 

At the moment the hook fires, `spotEngine.updateBalance` has **not yet been called**, so the subaccount's balance in `SpotEngine` still reflects the full pre-withdrawal amount. The attacker's re-entrant call therefore passes the health check with the stale (inflated) balance.

There is **no `nonReentrant` guard** anywhere in the production Solidity codebase — confirmed by a full-codebase search for `ReentrancyGuard`, `nonReentrant`, `_status`, and `_ENTERED`, all of which return zero matches. [4](#0-3) 

---

### Impact Explanation

An attacker who controls a recipient contract with an ERC777 `tokensReceived` hook can:

1. Pre-queue a second `WithdrawCollateral` slow-mode transaction via `Endpoint.submitSlowModeTransaction()` (publicly callable, no privileged access required).
2. Wait the mandatory 3-day slow-mode delay for it to become executable.
3. Trigger the first withdrawal (via sequencer or after the delay via `Endpoint.executeSlowModeTransaction()`).
4. Inside the ERC777 `tokensReceived` callback, call `Endpoint.executeSlowModeTransaction()` to process the second queued withdrawal.
5. Because `spotEngine.updateBalance` has not yet run, the subaccount balance is still at the pre-withdrawal level; the health check at line 419 passes for the second withdrawal as well.

**Corrupted state delta:** `SpotEngine.balances[productId][sender].amountNormalized` is decremented only once (or not at all if the outer call reverts after re-entry), while the protocol has paid out twice. The `totalDepositsNormalized` accounting in `SpotEngineState._updateBalanceNormalized` is correspondingly corrupted, breaking `assertUtilization`. [5](#0-4) [6](#0-5) 

**Asset delta:** Direct loss of collateral tokens from the `WithdrawPool` / `Clearinghouse` equal to the attacker's full collateral balance per re-entrant call, repeatable until the pool is drained.

---

### Likelihood Explanation

- **Requires ERC777 (or callback-on-receive) token** to be listed as a supported collateral product. ERC777 is a published standard; any governance-approved listing of such a token activates the vulnerability.
- **No privileged access required.** `submitSlowModeTransaction` and `executeSlowModeTransaction` are both publicly callable by any EOA or contract. [7](#0-6) [8](#0-7) 

- **Pre-queuing is the only setup cost**, and the 3-day delay is a known, predictable window — not a barrier.
- **Impact: High** (direct, repeatable collateral theft). **Likelihood: Medium** (conditional on ERC777 token listing, but zero protocol-side mitigations exist).

---

### Recommendation

1. **Adopt Checks-Effects-Interactions strictly in `withdrawCollateral`.** Move `spotEngine.updateBalance` and `getHealth` checks to execute **before** `handleWithdrawTransfer`:

```solidity
// EFFECTS first
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// INTERACTION last
handleWithdrawTransfer(token, sendTo, amount, idx);
```

2. **Add a `nonReentrant` modifier** (OpenZeppelin `ReentrancyGuardUpgradeable`) to `withdrawCollateral`, `submitFastWithdrawal`, and any other function that performs external token transfers.

3. **Document and enforce a token allowlist policy** that explicitly rejects ERC777 and any token with transfer hooks until the above fixes are in place.

---

### Proof of Concept

```
Attacker setup (3 days before attack):
  1. Deploy AttackerContract with tokensReceived() hook.
  2. Deposit N ERC777 tokens into Nado under subaccount controlled by AttackerContract.
  3. Call Endpoint.submitSlowModeTransaction(WithdrawCollateral{amount: N}) — queues tx #2.

Attack day (after 3-day delay):
  4. Sequencer (or attacker via executeSlowModeTransaction) processes tx #1:
       Endpoint → Clearinghouse.withdrawCollateral(sender, productId, N, AttackerContract, idx1)
         → handleWithdrawTransfer → BaseWithdrawPool.handleWithdrawTransfer
           → token.safeTransfer(AttackerContract, N)
             *** ERC777 tokensReceived hook fires ***
             AttackerContract calls Endpoint.executeSlowModeTransaction()
               → Clearinghouse.withdrawCollateral(sender, productId, N, AttackerContract, idx2)
               → spotEngine balance is still N (not yet decremented)
               → health check passes
               → AttackerContract receives second N tokens
             *** hook returns ***
         → spotEngine.updateBalance(-N)  ← only one decrement for two withdrawals
         → getHealth passes (balance now 0, but 2N tokens already sent)

Result: AttackerContract holds 2N tokens; SpotEngine records 0 balance.
        Protocol is short N tokens.
``` [9](#0-8) [3](#0-2)

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
