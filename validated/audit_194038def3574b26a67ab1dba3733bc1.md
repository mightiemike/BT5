### Title
Fee-on-Transfer Token Deposits Inflate Subaccount Collateral Beyond Actual Received Amount — (File: `core/contracts/EndpointStorage.sol`)

### Summary
`handleDepositTransfer` in `EndpointStorage.sol` transfers a caller-specified `amount` from the depositor to the endpoint, then forwards the same `amount` to the clearinghouse, without checking the actual balance received. For fee-on-transfer tokens, the clearinghouse receives `amount - fee` but the slow-mode transaction queued for the sequencer records the full `amount`. When the sequencer processes the deposit, `Clearinghouse.depositCollateral` credits the subaccount with the full `amount`, creating phantom collateral that exceeds the real token balance held by the protocol.

### Finding Description

The deposit path is:

1. `Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer(token, msg.sender, uint256(amount))`. [1](#0-0) 

2. `handleDepositTransfer` performs two transfers using the fixed caller-supplied `amount`: [2](#0-1) 

   - `safeTransferFrom(token, from, amount)` — endpoint receives `amount - fee` for a fee-on-transfer token.
   - `safeTransferTo(token, address(clearinghouse), amount)` — attempts to forward the original `amount`, not the actually-received `amount - fee`.

3. A slow-mode transaction is enqueued encoding the original `amount`: [3](#0-2) 

4. When the sequencer executes the slow-mode tx, `Clearinghouse.depositCollateral` credits `amountRealized = int128(txn.amount) * multiplier` to the subaccount's spot balance — using the original `amount`, not the actual received amount: [4](#0-3) 

The `ERC20Helper` library correctly handles non-bool-returning tokens via low-level calls: [5](#0-4) 

However, it performs no before/after balance check to detect fee-on-transfer tokens. The `IERC20Base` interface also does not restrict token types: [6](#0-5) 

### Impact Explanation

**Impact: Medium.**

For any fee-on-transfer token listed as a spot product, a depositor is credited with more collateral than the clearinghouse actually holds. The discrepancy (`fee * multiplier` per deposit) accumulates across all deposits of that token. This inflates subaccount health scores, allowing users to open larger positions or withdraw more than the protocol's real reserves support. The protocol becomes insolvent for that token as the sum of all credited balances exceeds the actual token balance held by the clearinghouse.

Additionally, the second transfer in `handleDepositTransfer` — `safeTransferTo(token, clearinghouse, amount)` — will revert if the endpoint holds no surplus balance of that token, since it only received `amount - fee` from the user. This means deposits of fee-on-transfer tokens either revert (DoS) or silently drain other users' deposited funds if the endpoint holds a surplus. [7](#0-6) 

### Likelihood Explanation

**Likelihood: Medium.**

Fee-on-transfer tokens (e.g., USDT with its dormant fee switch, STA, PAXG) are not uncommon. The protocol does not document which tokens are unsupported, and any operator listing such a token as a spot product exposes the protocol to this issue. No privileged access is required — any user can trigger this by calling `depositCollateral` or `depositCollateralWithReferral`. [8](#0-7) 

### Recommendation

In `handleDepositTransfer`, record the clearinghouse's token balance before and after the transfer and use the actual delta as the credited amount, rather than the caller-supplied `amount`. Alternatively, document explicitly that fee-on-transfer tokens are not supported as spot collateral products. [2](#0-1) 

### Proof of Concept

1. A fee-on-transfer token (1% fee per transfer) is listed as a spot product.
2. Alice calls `Endpoint.depositCollateralWithReferral(aliceSubaccount, productId, 1000e6)`.
3. `handleDepositTransfer` calls `safeTransferFrom(token, Alice, endpoint, 1000e6)` → endpoint receives `990e6`.
4. `handleDepositTransfer` calls `safeTransferTo(token, clearinghouse, 1000e6)` → clearinghouse receives `990e6` (another 1% fee), endpoint balance goes to `-10e6` (reverts) OR drains other users' funds if surplus exists.
5. The slow-mode tx is queued with `amount = 1000e6`.
6. Sequencer processes the tx: `Clearinghouse.depositCollateral` credits Alice with `1000e6 * multiplier` in spot balance.
7. Alice's on-chain balance reflects `1000e6` worth of collateral, but the clearinghouse holds at most `980.1e6` of the token.
8. Alice can trade or withdraw against the phantom `~19.9e6` of collateral, draining other users' funds. [2](#0-1) [9](#0-8)
