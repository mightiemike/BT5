### Title
`updateStates()` Not Called Before Balance Changes Triggered via Unprivileged Slow Mode Execution — (`core/contracts/SpotEngineState.sol`, `core/contracts/Endpoint.sol`)

---

### Summary

`SpotEngineState._updateState()` computes interest using the **current** utilization ratio at the moment `updateStates(dt)` is called. Any balance change that alters `totalDepositsNormalized` or `totalBorrowsNormalized` **before** `updateStates()` is called causes the interest for the entire elapsed period `dt` — including the time before the balance change — to be calculated at the wrong utilization ratio. The unprivileged entry point is `Endpoint.executeSlowModeTransaction()`, which is callable by anyone after the 3-day timeout and executes a pending deposit without `updateStates()` being called first.

---

### Finding Description

In `SpotEngineState.sol`, `_updateState()` computes the borrow rate from the current utilization ratio:

```
utilizationRatioX18 = totalBorrows / totalDeposits
``` [1](#0-0) 

The multipliers `cumulativeDepositsMultiplierX18` and `cumulativeBorrowsMultiplierX18` are then updated using this ratio and the full time delta `dt` passed in by the sequencer: [2](#0-1) 

`updateStates(uint128 dt)` is the only function that calls `_updateState()`, and it is gated to the Endpoint: [3](#0-2) 

`SpotEngine.updateBalance()` — which is called by `Clearinghouse.depositCollateral()` — directly mutates `totalDepositsNormalized` via `_updateBalanceNormalized()` **without calling `_updateState()` first**: [4](#0-3) [5](#0-4) 

The unprivileged trigger is `Endpoint.executeSlowModeTransaction()`, which has no access control and can be called by anyone once `txn.executableAt <= block.timestamp` (3 days after the deposit was submitted): [6](#0-5) [7](#0-6) 

When this executes a pending `DepositCollateral` slow-mode transaction, `totalDepositsNormalized` increases, lowering the utilization ratio. The sequencer subsequently calls `updateStates(dt)` where `dt` spans the entire period since the last update — including the time **before** the deposit. The interest for that pre-deposit period is then computed at the post-deposit (lower) utilization ratio, understating the true interest owed.

The symmetric case applies to withdrawals: if a withdrawal slow-mode transaction is executed by an unprivileged caller, `totalDepositsNormalized` decreases, raising utilization, and the subsequent `updateStates(dt)` call overstates interest for the pre-withdrawal period.

---

### Impact Explanation

- **Loss of yield to lenders**: A large deposit executed via slow mode lowers utilization. The sequencer's next `updateStates(dt)` call applies the lower rate retroactively to the entire elapsed period, so lenders accrue less interest than they earned during the high-utilization window before the deposit.
- **Overpaid interest for borrowers**: A large withdrawal executed via slow mode raises utilization. The next `updateStates(dt)` call applies the higher rate retroactively, causing borrowers to pay more interest than they owed during the low-utilization window before the withdrawal.

The magnitude scales with the size of the balance change relative to total pool liquidity and the length of the elapsed period `dt`.

---

### Likelihood Explanation

`executeSlowModeTransaction()` is permissionlessly callable by any address after the 3-day timeout. This timeout is reached whenever the sequencer is offline, congested, or censoring a specific user. The scenario is realistic and requires no privileged access, no leaked keys, and no governance capture. The attacker does not need to be the depositor — any third party can trigger the execution.

---

### Recommendation

`_updateState()` must be called before any mutation of `totalDepositsNormalized` or `totalBorrowsNormalized`. The cleanest fix mirrors the Sentiment recommendation: add an internal `_updateStates()` call at the top of `SpotEngine.updateBalance()` (both overloads), so that interest is always settled against the pre-change utilization ratio before the pool composition changes.

Alternatively, the slow-mode execution path should emit a sequencer-signed `updateStates` transaction atomically before executing any balance-changing slow-mode transaction.

---

### Proof of Concept

1. Pool state: 1000 USDC total deposits, 500 USDC total borrows. Utilization = 0.5. `updateStates()` was last called at `T=0`.
2. At `T=0`, Alice submits a `depositCollateral` slow-mode transaction for 1000 USDC. `executableAt = T + 3 days`.
3. The sequencer goes offline. No `updateStates()` is called.
4. At `T = 3 days`, Charlie (unprivileged) calls `Endpoint.executeSlowModeTransaction()`.
   - `Clearinghouse.depositCollateral()` → `SpotEngine.updateBalance()` → `totalDepositsNormalized` increases.
   - New pool: 2000 USDC deposits, 500 USDC borrows. Utilization = 0.25.
   - `updateStates()` is **not** called.
5. At `T = 3 days + ε`, the sequencer resumes and calls `updateStates(dt)` with `dt = 3 days`.
   - `_updateState()` computes the borrow rate using utilization = **0.25** (post-deposit).
   - This rate is applied to the full 3-day window, including the first 3 days when utilization was **0.5**.
   - Borrowers pay interest at the 0.25-utilization rate for the entire period instead of the 0.5-utilization rate.
   - Lenders lose the yield difference for the 3-day high-utilization window. [8](#0-7) [9](#0-8)

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
```

                continue;

            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }
