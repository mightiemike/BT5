### Title
Accumulated Builder Fees Permanently Locked When Builder Is Deregistered — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

When a builder is deregistered by setting its `owner` to `address(0)` via `updateBuilder()`, any previously accumulated `collectedBuilderFee[quoteId][builderId]` becomes permanently inaccessible. The `claimBuilderFee()` function enforces that only the registered owner can claim, and with `owner == address(0)` no real sender can ever satisfy that check. There is no recovery path.

---

### Finding Description

The `OffchainExchange` contract maintains a builder fee system. During order matching, a portion of the trader's quote delta is withheld as a builder fee and credited to an in-contract ledger:

```solidity
collectedBuilderFee[market.quoteId][feeInfo.builderId] += orderInfo.builderFee;
``` [1](#0-0) 

Builders are registered and updated via `updateBuilder()`, which unconditionally overwrites the `builders[builderId]` struct with whatever values are passed — including `owner = address(0)`:

```solidity
builders[txn.builderId] = Builder(
    txn.owner,
    txn.defaultFeeTier,
    txn.lowestFeeRate,
    txn.highestFeeRate
);
``` [2](#0-1) 

The only way to claim accumulated fees is through `claimBuilderFee()`, which requires:

```solidity
require(
    builders[builderId].owner == address(uint160(bytes20(sender))),
    ERR_UNAUTHORIZED
);
``` [3](#0-2) 

Once `owner` is set to `address(0)`, no real `sender` can ever satisfy `address(uint160(bytes20(sender))) == address(0)`, because `sender` is a `bytes32` subaccount whose first 20 bytes encode a real wallet address. The accumulated `collectedBuilderFee` mapping entries are never cleared by `updateBuilder()`, and no other function (`dumpFees()`, `claimSequencerFees()`, etc.) touches `collectedBuilderFee`. The fees are permanently stranded.

`dumpFees()` only handles `marketInfo[productId].collectedFees` (protocol fees), not builder fees: [4](#0-3) 

---

### Impact Explanation

The `collectedBuilderFee` ledger represents real economic value: it is the portion of quote delta deducted from traders during order matching. When `claimBuilderFee()` is eventually called, it credits the builder's spot engine balance via `spotEngine.updateBalance(productId, sender, collectedFee)`. If the builder is deregistered before claiming, those credits can never be issued. The deducted value has already been removed from traders' balances but can never be credited to anyone — it is permanently locked in the accounting system with no recovery mechanism. The magnitude scales with trading volume routed through the deregistered builder prior to deregistration.

---

### Likelihood Explanation

The trigger requires an admin `UpdateBuilder` transaction with `owner = address(0)` while the builder has a non-zero `collectedBuilderFee` balance. This is a realistic operational scenario: a builder may be removed for violating terms of service, or a builder may voluntarily deregister. The window between fee accumulation and claiming is unbounded — builders are not required to claim before deregistration. The `ContractOwner.updateBuilders()` helper makes this a routine admin action with no guard against unclaimed fees. [5](#0-4) 

---

### Recommendation

In `updateBuilder()`, before overwriting the builder record with `owner = address(0)`, check whether `collectedBuilderFee` is non-zero for any product and either:

1. Revert if unclaimed fees exist, forcing the builder to claim first; or
2. Automatically sweep the accumulated fees to a designated recovery address (e.g., `X_ACCOUNT` or the protocol treasury) before zeroing the builder record.

Analogously to the Velodrome fix, the deregistration path should not silently orphan accumulated value.

---

### Proof of Concept

1. Builder `B` (id = `42`) is registered with `owner = alice`.
2. Traders route orders through builder `42`; `collectedBuilderFee[QUOTE_PRODUCT_ID][42]` accumulates to `1000e18`.
3. Admin calls `updateBuilders([42], [address(0)], ...)` via `ContractOwner`, which submits an `UpdateBuilder` slow-mode transaction.
4. `updateBuilder()` executes: `builders[42].owner = address(0)`.
5. Alice attempts `ClaimBuilderFee { sender: alice_subaccount, builderId: 42 }`.
6. `claimBuilderFee()` evaluates: `builders[42].owner (= address(0)) == address(uint160(bytes20(alice_subaccount))) (= alice)` → `false` → revert `ERR_UNAUTHORIZED`.
7. No other code path can drain `collectedBuilderFee[QUOTE_PRODUCT_ID][42]`. The `1000e18` in quote value is permanently locked. [6](#0-5) [2](#0-1)

    function claimBuilderFee(bytes32 sender, uint32 builderId)
        external
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(
            builders[builderId].owner == address(uint160(bytes20(sender))),
            ERR_UNAUTHORIZED
        );
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            int128 collectedFee = collectedBuilderFee[productId][builderId];
            if (collectedFee == 0) {
                continue;
            }
            emit ClaimBuilderFee(builderId, productId, sender, collectedFee);
            spotEngine.updateBalance(productId, sender, collectedFee);
            collectedBuilderFee[productId][builderId] = 0;
        }
    }
