### Title
`SwapAllowlistExtension` gates on the router address instead of the actual swapper, allowing any user to bypass the per-pool swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When users route through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the end user. If the router is allowlisted (required for normal operation), every user — including those not individually permitted — can bypass the swap allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument (`sender`) to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap(), i.e. the router
    recipient,
    zeroForOne,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to the extension:

```solidity
// ExtensionCalling.sol L149+
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
``` [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` value is allowlisted:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [3](#0-2) 

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap()`, so `sender` = router address. The allowlist lookup becomes `allowedSwapper[pool][router]`. If the router is allowlisted (which is necessary for any user to swap through it), the check passes for **every caller of the router**, regardless of whether the individual user is permitted.

The asymmetry is visible by comparing with `DepositAllowlistExtension`, which correctly gates on `owner` (the end user / position owner), not `sender` (the LiquidityAdder contract):

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [4](#0-3) 

`DepositAllowlistExtension` ignores `sender` (the LiquidityAdder) and checks `owner` (the actual depositor). `SwapAllowlistExtension` does the opposite: it ignores `recipient` (the actual beneficiary) and checks `sender` (the router). The intended economic actor for a swap allowlist is the user initiating the trade, not the routing contract.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses is fully bypassed. Any unpermitted user who calls `MetricOmmSimpleRouter.exactInputSingle()` (or any other router path that calls `pool.swap()`) will pass the allowlist check as long as the router itself is allowlisted. The pool receives and settles the swap normally, so the user receives output tokens they should have been denied. This is a direct, fund-impacting curation failure on every pool using this extension.

---

### Likelihood Explanation

The bypass is reachable through the standard, documented periphery path (`MetricOmmSimpleRouter`). No special privileges, malicious setup, or non-standard tokens are required. Any user who knows the router address can trigger it. The router must be allowlisted for legitimate users to trade, so the precondition is always satisfied in any live curated pool.

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to gate on `recipient` (the actual swap beneficiary) instead of `sender` (the calling contract), mirroring the correct actor choice made in `DepositAllowlistExtension`:

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, if the intent is to gate on the initiating EOA, the pool would need to thread `tx.origin` or a verified caller identity through the extension data — but gating on `recipient` is the closest structural parallel to how `DepositAllowlistExtension` gates on `owner`.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin allowlists only `alice` via `setAllowedToSwap(pool, alice, true)`.
3. Admin also allowlists the router via `setAllowedToSwap(pool, router, true)` so that `alice` can trade through it.
4. `bob` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. The router calls `pool.swap(recipient=bob, ...)`.
6. The pool calls `extension.beforeSwap(sender=router, recipient=bob, ...)`.
7. The extension checks `allowedSwapper[pool][router]` → `true` → passes.
8. `bob` receives output tokens despite never being individually permitted.

The allowlist is completely defeated for any user routing through the shared router.
