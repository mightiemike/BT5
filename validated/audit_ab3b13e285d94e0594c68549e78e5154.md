### Title
`StatelessTransactionValidatorConfig` accepts `min_sierra_version > max_sierra_version` with no cross-field validation, silently making all declare transactions unreachable - (`File: crates/apollo_gateway_config/src/config.rs`)

### Summary

`StatelessTransactionValidatorConfig` derives `Validate` but carries no schema-level cross-field validation. An operator can set `min_sierra_version` greater than `max_sierra_version` and the config system accepts it without error. Once the gateway starts with this configuration, every declare transaction is permanently rejected at the stateless validation stage because the version-range check can never be satisfied.

### Finding Description

`StatelessTransactionValidatorConfig` holds two paired version bounds:

```rust
// crates/apollo_gateway_config/src/config.rs  lines 166-186
#[derive(Clone, Debug, Deserialize, PartialEq, Serialize, Validate)]
pub struct StatelessTransactionValidatorConfig {
    ...
    pub min_sierra_version: VersionId,
    pub max_sierra_version: VersionId,
    ...
}
``` [1](#0-0) 

The struct derives `Validate` but has **no** `#[validate(schema(function = "..."))]` attribute and no field-level range constraints on either version field. `GatewayStaticConfig` propagates validation via `#[validate(nested)]`:

```rust
// crates/apollo_gateway_config/src/config.rs  lines 40-58
#[derive(Clone, Debug, Deserialize, PartialEq, Serialize, Validate)]
pub struct GatewayStaticConfig {
    #[validate(nested)]
    pub stateless_tx_validator_config: StatelessTransactionValidatorConfig,
    ...
}
``` [2](#0-1) 

The nested call reaches `StatelessTransactionValidatorConfig::validate()`, but because no cross-field rule exists, the invariant `min_sierra_version <= max_sierra_version` is never checked. The full node config validation path (`validate_node_config` → `config_validate` → nested `Validate`) therefore passes silently even when the bounds are inverted. [3](#0-2) 

At runtime, every incoming declare transaction hits `validate_sierra_version`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs  lines 293-313
fn validate_sierra_version(&self, sierra_program: &[Felt]) -> ... {
    let mut max_sierra_version = self.config.max_sierra_version;
    max_sierra_version.0.patch = usize::MAX;

    let sierra_version = VersionId::from_sierra_program(sierra_program)?;
    if self.config.min_sierra_version <= sierra_version
        && sierra_version <= max_sierra_version
    {
        return Ok(());
    }
    Err(StatelessTransactionValidatorError::UnsupportedSierraVersion { ... })
}
``` [4](#0-3) 

When `min_sierra_version > max_sierra_version`, the conjunction `min <= version && version <= max` is logically unsatisfiable for every possible `version`. Every declare transaction is rejected with `UnsupportedSierraVersion` regardless of the actual Sierra version embedded in the contract.

### Impact Explanation

Once the gateway is running with inverted version bounds, **all declare transactions are rejected at stateless validation** before they reach the mempool or batcher. No new contract classes can be registered on the network. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The gateway returns an authoritative-looking `UnsupportedSierraVersion` error to callers, giving no indication that the configuration itself is broken.

### Likelihood Explanation

The trigger requires an operator to write a misconfigured `StatelessTransactionValidatorConfig` (e.g., via the deployment config files such as `replacer_gateway_config.json` where `max_contract_bytecode_size` is already a placeholder variable). Because no validation gate exists, the misconfiguration survives startup and is only discovered when declare transactions start failing in production. The likelihood is **Low** in isolation but the absence of any guard makes it a latent operational risk with no safety net. [5](#0-4) 

### Recommendation

Add a schema-level cross-field validation function to `StatelessTransactionValidatorConfig`, mirroring the pattern already used in `BatcherDynamicConfig` and `HttpServerConfig`:

```rust
// crates/apollo_gateway_config/src/config.rs
#[derive(Clone, Debug, Deserialize, PartialEq, Serialize, Validate)]
#[validate(schema(function = "validate_stateless_tx_validator_config"))]
pub struct StatelessTransactionValidatorConfig { ... }

fn validate_stateless_tx_validator_config(
    config: &StatelessTransactionValidatorConfig,
) -> Result<(), ValidationError> {
    if config.min_sierra_version > config.max_sierra_version {
        return Err(ValidationError::new(
            "min_sierra_version must be <= max_sierra_version",
        ));
    }
    Ok(())
}
``` [6](#0-5) [7](#0-6) 

### Proof of Concept

1. Set the gateway config with inverted Sierra version bounds (e.g., in `gateway_config.json`):
   ```json
   "gateway_config.static_config.stateless_tx_validator_config.min_sierra_version.major": 1,
   "gateway_config.static_config.stateless_tx_validator_config.min_sierra_version.minor": 9,
   "gateway_config.static_config.stateless_tx_validator_config.max_sierra_version.major": 1,
   "gateway_config.static_config.stateless_tx_validator_config.max_sierra_version.minor": 1
   ```
2. Start the sequencer node. `validate_node_config()` → `config_validate(self)` → nested `Validate` on `StatelessTransactionValidatorConfig` passes without error because no cross-field rule exists.
3. Submit any declare transaction with a valid Sierra contract (e.g., Sierra version 1.7.0, which is within the normal accepted range).
4. The gateway calls `validate_sierra_version`. The check `1.9.0 <= 1.7.0 && 1.7.0 <= 1.1.0` is false. The transaction is rejected with `UnsupportedSierraVersion { version: 1.7.0, min_version: 1.9.0, max_version: 1.1.0 }`.
5. Every declare transaction from every user is rejected. No new contract classes can be deployed. [8](#0-7) [4](#0-3)
