### Title
Unchecked `u64` Multiplication Overflow in `op_unknown` Cost Multiplier Allows Cost-Limit Bypass — (File: `src/more_ops.rs`)

---

### Summary

In `src/more_ops.rs`, the `op_unknown` function computes a final cost by multiplying an accumulated base cost by an attacker-controlled `cost_multiplier`. This multiplication is performed on a `u64` (`Cost`) without overflow protection. In Rust release builds (where `overflow-checks` is off by default), the multiplication silently wraps, producing a value far smaller than the true cost. The post-multiplication guard (`if cost > u32::MAX`) then passes on the wrapped value, and the function returns a near-zero cost to the evaluator — effectively allowing an arbitrarily expensive unknown opcode to execute for free.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` handles unknown opcodes in lenient/consensus mode. It computes a base cost (`cost`) from the opcode's `cost_function` field and the supplied arguments, then scales it:

```rust
// src/more_ops.rs lines 258–266
check_cost(cost, max_cost)?;          // ← ensures cost ≤ max_cost (u64)
cost *= cost_multiplier + 1;          // ← UNCHECKED u64 multiplication
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode bytes via `u32_from_u8`, so it can be up to `u32::MAX` = 4 294 967 295, making `cost_multiplier + 1` up to 2³². `cost` before the multiplication is bounded only by `max_cost` (up to ~11 × 10⁹ for Chia). The product can therefore exceed `u64::MAX` (~1.8 × 10¹⁹).

**Concrete overflow scenario (cost_function = 3, concat-like):**

| Variable | Value |
|---|---|
| `cost_multiplier` | `u32::MAX` = 4 294 967 295 |
| `cost_multiplier + 1` | 2³² = 4 294 967 296 |
| Attacker-chosen `cost` | 2³² = 4 294 967 296 (reachable under Chia's 11 × 10⁹ max_cost) |
| True product | 2³² × 2³² = 2⁶⁴ |
| Wrapped result (mod 2⁶⁴) | **0** |
| `0 > u32::MAX`? | **false** → accepted |
| Returned cost | **0** |

The attacker achieves a program that should cost 2⁶⁴ units but is charged **0**.

To reach `cost = 2³²` with `cost_function = 3`:
- `CONCAT_BASE_COST` (142) + N × `CONCAT_COST_PER_ARG` (135) + bytes × `CONCAT_COST_PER_BYTE` (3) = 4 294 967 296
- Approximately 31 814 571 zero-byte arguments suffice; well within the allocator's `MAX_NUM_PAIRS = 62 500 000` limit.

The `check_cost` call on line 260 only validates the pre-multiplication `cost` against `max_cost`; it does not protect against the multiplication itself overflowing.

---

### Impact Explanation

- **Consensus divergence / cost-limit bypass**: A node running a release build silently accepts the wrapped cost and continues execution. A node with overflow checks enabled panics (DoS). Both outcomes break consensus.
- **Undercharged execution**: The evaluator's `cost` accumulator in `run_program` receives the wrapped (near-zero) value, so the program's total cost is massively under-reported. The `CostExceeded` guard is never triggered.
- **Corrupted result**: The `Reduction` returned carries a false cost, propagating the error into the caller's cost accounting.

---

### Likelihood Explanation

- `op_unknown` is called whenever `dialect.allow_unknown_ops()` is true, which is the case in Chia's consensus/lenient mode for forward-compatibility with future opcodes.
- The attacker fully controls the opcode bytes (setting `cost_function` and `cost_multiplier`) and the argument list (controlling the pre-multiplication `cost`).
- No special privileges are required; any spend bundle submitted to the mempool can contain such a program.
- The overflow requires only that `max_cost ≥ 2³²` (~4.3 × 10⁹), which is satisfied by Chia's block cost limit of ~11 × 10⁹.

---

### Recommendation

Replace the bare multiplication with a checked or saturating variant, and treat overflow as an invalid opcode:

```rust
// src/more_ops.rs — replace lines 261–265
let Some(final_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o))?;
};
if final_cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(final_cost as Cost, allocator.nil()))
}
```

Additionally, enable `overflow-checks = true` in the release profile in `Cargo.toml` as a defence-in-depth measure for all cost arithmetic.
