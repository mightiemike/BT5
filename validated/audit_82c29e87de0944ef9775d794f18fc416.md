Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` Silently Bypasses Velocity Guard on First Swap Due to Zero-Initialized `lastMidPriceX64` — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` gates all velocity math behind `if (prevMid != 0)`. Because `lastMidPriceX64` is a plain storage mapping field that defaults to zero, and the extension does not override `initialize` to seed it at pool-creation time, every pool using this extension has its velocity guard silently bypassed on the first swap. LPs are exposed to the full magnitude of any oracle price movement that occurred between pool deployment and the first swap, which is the exact risk the guard is designed to prevent.

## Finding Description

In `beforeSwap`, the prior mid-price is read, the new mid-price is written unconditionally, and then the velocity check is gated:

```solidity
// PriceVelocityGuardExtension.sol L54-76
uint128 prevMid = s.lastMidPriceX64;   // reads 0 on first swap
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // state written unconditionally
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // FALSE on first swap → entire guard skipped
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity math and revert
    }
}
```

`lastMidPriceX64` is a field in `mapping(address pool => PriceVelocityState) public priceVelocityState` and defaults to zero in Solidity. The extension does not override `initialize`; the base class `BaseMetricExtension.initialize` is a no-op that simply returns the selector without writing any state. The only seeding path is the admin-only `setLastMidPrice` (L29–34), which is optional and not called automatically by any on-chain mechanism. There is no factory-enforced precondition requiring it to be called before the first swap.

By contrast, `OracleValueStopLossExtension` overrides `initialize` (L46–68) to write its config and set `initialized = true`, and its `afterSwap` calls `_requireInitialized` to enforce that the pool was properly set up before any hook logic runs. `PriceVelocityGuardExtension` has no equivalent guard.

Exploit path:
1. Pool is deployed with `PriceVelocityGuardExtension`; `maxChangePerBlockE18` is set to a non-zero value. Admin does not call `setLastMidPrice`.
2. Oracle price moves significantly (e.g., 30%) between deployment and block N.
3. Any unprivileged user calls `pool.swap(...)` at block N. Inside `beforeSwap`: `prevMid = priceVelocityState[pool].lastMidPriceX64` → **0**; `if (prevMid != 0)` is false; velocity math never executes; no revert.
4. The swap settles at the 30%-moved price. LPs absorb the full adverse selection.
5. From the second swap onward, `prevMid` is non-zero and the guard correctly enforces the velocity cap.

## Impact Explanation

The velocity guard's purpose is to protect LPs from rapid oracle-price movements that cause value leakage (LVR). Bypassing it on the first swap means LPs sell tokens at a worse-than-intended rate for the full magnitude of any price move that occurred since deployment. The loss is bounded by the price move magnitude and pool liquidity depth, but is material for volatile assets or pools with large TVL. This constitutes a direct loss of LP principal and qualifies as bad-price execution reaching a pool swap — both within the allowed impact gate.

## Likelihood Explanation

The condition is met for every pool deployed with this extension unless the admin manually calls `setLastMidPrice` before the first swap. There is no on-chain enforcement of this precondition. Any unprivileged user can be the first swapper; no special role, setup, or coordination is required beyond the pool being live. The window between pool deployment and the first swap is the only requirement, and oracle prices routinely move during that window.

## Recommendation

**Option A (fail closed):** At the start of `beforeSwap`, if `maxChangePerBlockE18 != 0` and `prevMid == 0`, revert with a dedicated error (e.g., `PriceVelocityNotInitialized`). This forces the admin to call `setLastMidPrice` before the pool accepts swaps.

**Option B (initialize at deployment):** Override `initialize` in `PriceVelocityGuardExtension` to accept and store an initial `lastMidPriceX64` and `lastUpdateBlock`, mirroring the pattern used by `OracleValueStopLossExtension.initialize` (L46–68). This provides a factory-enforced, atomic seeding path that eliminates the unenforceable off-chain precondition.

function test_firstSwapBypassesVelocityGuard() public {
    // Deploy pool with PriceVelocityGuardExtension
    // Set maxChangePerBlockE18 = 1e15 (≈ 0.1% per block)
    // Do NOT call setLastMidPrice

    // Simulate oracle price moving 30% before first swap
    // (e.g., via mock oracle returning a price 30% higher)

    // First swap: prevMid == 0 → guard skipped → no revert
    pool.swap(...); // succeeds despite 30% price move

    // Second swap in same block with 1% move → correctly reverts
    vm.expectRevert(PriceVelocityGuardExtension.PriceVelocityExceeded.selector);
    pool.swap(...);
}
