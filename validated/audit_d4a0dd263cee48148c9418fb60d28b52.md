### Title
Difficulty Retarget Calculation Reads Mainchain Height Index for Fork Blocks, Enabling Incorrect Difficulty Acceptance — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_header_by_height()` is backed by a single shared mapping, `mainchain_height_to_header`, that only tracks mainchain blocks. When a fork block at a retarget boundary is validated, the difficulty calculation fetches the start-of-interval ancestor from this mainchain-only index rather than walking the fork's actual ancestor chain. This is the direct analog of the external report: a single shared accumulator is used as if it correctly represents state for multiple distinct chains (mainchain and forks), causing information loss and incorrect difficulty computation.

---

### Finding Description

`BlocksGetter::get_header_by_height` is implemented in `lib.rs` as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

`mainchain_height_to_header` is a single `LookupMap<u64, H256>` that maps block height to the **mainchain** block hash at that height. [2](#0-1) 

This function is called during difficulty retarget validation for every supported chain. In `bitcoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [3](#0-2) 

In `litecoin.rs`:

```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
``` [4](#0-3) 

In `dogecoin.rs`:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [5](#0-4) 

The Dogecoin module even contains a developer `TODO` acknowledging this exact ambiguity.

When a fork block is submitted at a retarget height (e.g., Bitcoin height `N` where `N % 2016 == 0`), the contract calls `get_header_by_height(N - 2016)` to retrieve the timestamp of the block at the start of the retarget window. However, if the fork diverged from the mainchain at or before height `N - 2016`, the block at that height in `mainchain_height_to_header` is a **mainchain block**, not the fork's actual ancestor. The fork's true ancestor at that height is stored in `headers_pool` but is not indexed by height — it is only reachable by walking `prev_block_hash` links.

The result: the expected difficulty for the fork block is computed using the mainchain ancestor's timestamp, not the fork's ancestor's timestamp. These timestamps can differ arbitrarily.

---

### Impact Explanation

If the fork's ancestor at the retarget boundary has a later timestamp than the mainchain's block at the same height, the computed `actual_time_taken` is smaller, yielding a **harder** expected difficulty — causing a legitimately valid fork block to be rejected. Conversely, if the fork's ancestor has an earlier timestamp, `actual_time_taken` is larger, yielding an **easier** expected difficulty — allowing a fork block with an artificially low difficulty target to pass the `check_pow` / `check_target` gate.

The security-critical case is the latter: an attacker can craft a fork whose ancestor at the retarget boundary has a manipulated timestamp (within the MTP and future-time constraints), causing the contract to accept a fork block with a lower difficulty than the real Bitcoin network requires. This corrupts the `chain_work` comparison used for reorg decisions:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [6](#0-5) 

A fork accepted with an inflated difficulty target produces an inflated `chain_work`, potentially triggering a reorg to a fraudulent chain. Downstream consumers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` would then verify transactions against a fraudulent mainchain tip. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The attacker must be a registered trusted relayer (the `submit_blocks` entry point is gated by `#[trusted_relayer]`). [8](#0-7) 

However, the trusted relayer system is a staking mechanism open to any NEAR account that meets the stake requirement — it is not a closed privileged set. A malicious relayer is an explicitly modeled threat for a light client bridge. The attacker additionally needs to submit a fork that crosses a retarget boundary, which is a standard scenario in adversarial chain-reorg attacks. The Dogecoin path is the most exposed because its per-block Digishield retarget means every block is a retarget boundary after height 145,000. [9](#0-8) 

---

### Recommendation

Replace `get_header_by_height` (mainchain-only lookup) with an ancestor-walking function that follows `prev_block_hash` links from the fork's `prev_block_header` back to the required height. This ensures the retarget window uses the fork's actual ancestor timestamps, not the mainchain's. The `headers_pool` already stores all submitted headers (mainchain and forks), so the walk is possible without additional storage.

Concretely, add a helper such as:

```rust
fn get_ancestor_at_height(
    &self,
    start: &ExtendedHeader,
    target_height: u64,
) -> ExtendedHeader { ... }
```

and replace all `blocks_getter.get_header_by_height(...)` calls inside `get_next_work_required` with this ancestor walk.

---

### Proof of Concept

1. The mainchain is at height 2016 (Bitcoin retarget boundary). Mainchain block at height 0 has timestamp `T_main`.
2. Attacker submits a fork starting at height 0 whose genesis ancestor has timestamp `T_fork = T_main - X` (earlier, within valid range).
3. Attacker submits fork blocks up to height 2015 (all non-retarget, accepted using `prev_block_header.bits`).
4. Attacker submits fork block at height 2016. `check_pow` calls `get_next_work_required`, which calls `get_header_by_height(0)`.
5. `get_header_by_height(0)` returns the **mainchain** block at height 0 (timestamp `T_main`), not the fork's ancestor (timestamp `T_fork`).
6. `actual_time_taken = T_prev_fork - T_main` is larger than the true fork interval `T_prev_fork - T_fork`, yielding a lower (easier) expected difficulty.
7. The fork block passes `check_pow` with a bits value that would be rejected if the correct fork ancestor timestamp were used.
8. If the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the mainchain tip with the attacker's fraudulent chain.
9. Subsequent calls to `verify_transaction_inclusion` return results against the fraudulent chain. [10](#0-9) [11](#0-10)
