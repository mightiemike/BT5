### Title
NEAR Relayer Private Key Stored in Plaintext on Filesystem — (File: `relayer/src/near_client.rs`, `relayer/src/config.rs`)

---

### Summary

The relayer's NEAR private key — which authorizes all `submit_blocks` calls to the BTC light client contract as a trusted relayer — is stored in plaintext on the filesystem, either as a bare string in a TOML config file or as a JSON credentials file read without any encryption. An attacker with read access to the relayer host's filesystem can steal the key and submit adversarial block headers to the contract, directly corrupting the canonical chain state (`mainchain_tip_blockhash`, `mainchain_height_to_header`, `mainchain_header_to_height`).

---

### Finding Description

The `NearConfig` struct declares `private_key` as a plain `String`: [1](#0-0) 

When the credentials-file path is used instead, `NearClient::new` reads the file with `std::fs::read_to_string` and extracts the key as a raw JSON string — no decryption, no integrity check: [2](#0-1) 

The example config confirms the expected deployment pattern — a plaintext JSON file on disk: [3](#0-2) 

The key is then loaded into an `InMemorySigner` and used to sign every `submit_blocks` transaction sent to the contract: [4](#0-3) 

The contract's `submit_blocks` entry point is gated by `#[trusted_relayer]`, meaning the signing account must be a registered trusted relayer. The relayer account is, by definition, registered — so possession of its private key grants full trusted-relayer authority: [5](#0-4) 

---

### Impact Explanation

A successful key theft gives the attacker the ability to call `submit_blocks` as a trusted relayer. The contract will accept any well-formed header chain from a trusted relayer and update:

- `mainchain_tip_blockhash`
- `mainchain_height_to_header` / `mainchain_header_to_height`
- `headers_pool` [6](#0-5) 

This corrupts the canonical chain mapping that `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` rely on to confirm Bitcoin transaction proofs. Downstream contracts consuming these verification results would accept forged inclusion proofs for transactions that never occurred on the real Bitcoin chain.

---

### Likelihood Explanation

**Medium.** The relayer is a long-running service deployed in Docker (Dockerfile and compose files are present in the repo). Container escape vulnerabilities, shared-volume misconfigurations, CI/CD secret leakage, or a compromised host all provide realistic read access to the plaintext credentials file or TOML config. The file is not encrypted at rest and requires no special privilege to read once filesystem access is obtained — identical to the wallet.dat scenario in the reference report.

---

### Recommendation

1. **Never store the raw private key in a plaintext file.** Use a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager) and inject the key at runtime via a memory-only path.
2. If a credentials file must be used, encrypt it at rest (e.g., age, GPG) and decrypt only into a memory buffer, never writing the plaintext key to disk.
3. Restrict filesystem permissions on the credentials file to the relayer process user only (`chmod 600`).
4. Consider using a hardware security module (HSM) or a NEAR-native multi-sig scheme so that no single plaintext key can authorize header submissions.

---

### Proof of Concept

1. Attacker gains read access to the relayer host (container escape, shared volume, compromised CI secret, or insider).
2. Reads the plaintext credentials file:
   ```
   cat ./relayer-credentials.json
   # {"account_id":"relayer.near","private_key":"ed25519:AAAA..."}
   ```
   or reads `private_key` directly from the TOML config.
3. Constructs a `NearClient` using the stolen key and the registered relayer `account_id`.
4. Calls `submit_blocks` with a crafted sequence of headers (e.g., a fabricated fork with higher chainwork) — the contract's `#[trusted_relayer]` check passes because the signing account is the legitimate registered relayer.
5. The contract executes `reorg_chain`, replacing the canonical chain tip and height-to-hash mappings with attacker-controlled values.
6. Subsequent calls to `verify_transaction_inclusion_v2` by downstream contracts return `true` for transactions that do not exist on the real Bitcoin chain. [7](#0-6) [8](#0-7)

        if prev_block_header.block_hash == self.mainchain_tip_blockhash {

            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
