### Title
`max_receipt_size` Invariant Bypassed via Post-Validation Receipt Mutation — (`File: runtime/runtime/src/lib.rs`)

---

### Summary

The `max_receipt_size` protocol limit is checked on a receipt **before** the runtime applies two post-processing mutations that enlarge it. In the `promise_return` path, `output_data_receivers` are appended to an already-validated receipt. In the `value_return` path, a new `DataReceipt` is constructed from the full returned value without checking the final Borsh-serialized receipt size. Both paths allow any unprivileged user to produce on-chain receipts that exceed `max_receipt_size`, breaking the protocol invariant the limit is meant to enforce.

---

### Finding Description

`validate_receipt` in `runtime/runtime/src/verifier.rs` measures the Borsh-serialized size of a receipt and rejects it if it exceeds `limit_config.max_receipt_size`: [1](#0-0) 

This check runs on the receipt **as it exists at validation time**. After validation, `apply_action_receipt` in `runtime/runtime/src/lib.rs` performs two mutations that are never re-checked:

**Path 1 — `promise_return` (ReceiptIndex result):**

When a contract calls `promise_return(idx)`, the runtime takes the already-validated receipt at `receipt_index` and appends the parent receipt's `output_data_receivers` to it: [2](#0-1) 

A contract can craft receipt C to be exactly `max_receipt_size` bytes. It passes validation. Then the runtime appends one or more `DataReceiver` structs (each ~64 bytes: 32-byte `CryptoHash` + account ID) to `output_data_receivers`, pushing the final Borsh size above the limit with no re-check.

**Path 2 — `value_return` (Value result):**

When a contract returns a value, the runtime wraps it in a new `DataReceipt` inside a `Receipt::V0`: [3](#0-2) 

The only guard on the returned value is `validate_data_receipt`, which checks only the raw data length against `max_length_returned_data`: [4](#0-3) 

Since `max_length_returned_data == max_receipt_size == 4_194_304`, a value of exactly `max_receipt_size` bytes passes this check. But the final `Receipt::V0` wrapping it carries additional Borsh overhead: `predecessor_id` (account ID + length prefix), `receiver_id`, `receipt_id` (32 bytes), enum discriminant, `data_id` (32 bytes), and `Option` tag — approximately 200 extra bytes — pushing the serialized receipt to ~4,194,504 bytes, confirmed by the test: [5](#0-4) 

The codebase explicitly acknowledges this as a known bug. A dedicated `ValidateReceiptMode::ExistingReceipt` variant was introduced specifically to tolerate these oversized receipts at processing time: [6](#0-5) 

Two integration tests confirm both paths produce receipts above `max_receipt_size` in production: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Any unprivileged user who can deploy a contract can produce receipts that exceed `max_receipt_size`. These receipts are committed to on-chain state and forwarded to all validators. Systems that rely on the `max_receipt_size` invariant — including bandwidth scheduling (`core/primitives/src/bandwidth_scheduler.rs`) and congestion control (`runtime/runtime/src/congestion_control.rs`) — receive inputs outside their assumed bounds. The `ExistingReceipt` mode prevents crashes, but the protocol-level size invariant is broken, and the oversized receipts are stored and propagated across the network.

**Scope: High** — protocol invariant broken by unprivileged user action; oversized receipts propagate to all validators.

---

### Likelihood Explanation

Exploitation requires only a deployed contract and a standard function call. No privileged role, validator access, or special network position is needed. The test suite includes explicit end-to-end demonstrations of both paths succeeding. Likelihood is **High**.

---

### Recommendation

1. **Re-validate after mutation (Path 1):** After appending `output_data_receivers` to the receipt at `lib.rs:1028–1035`, re-measure the Borsh size and return an error if it exceeds `max_receipt_size`.

2. **Check full receipt size for data receipts (Path 2):** In `validate_data_receipt` (or at the construction site in `lib.rs:1044–1056`), compute the Borsh size of the complete `Receipt::V0` wrapper — not just `data.len()` — and reject it if it exceeds `max_receipt_size`. The envelope overhead (~200 bytes) must be included in the budget.

3. **Tighten `max_length_returned_data`:** Set `max_length_returned_data` to `max_receipt_size` minus the maximum possible `Receipt::V0` envelope overhead so that a value passing the data-length check is guaranteed to fit inside a receipt.

---

### Proof of Concept

**Path 2 (value_return):**

1. Deploy a contract with a method that calls `value_return` with a buffer of exactly `max_receipt_size` (4,194,304) bytes and a `promise_then` callback.
2. Submit a transaction calling that method.
3. The `value_return` host function checks `num_bytes (4194304) > max_length_returned_data (4194304)` → false, passes.
4. The runtime constructs `Receipt::V0 { ..., receipt: ReceiptEnum::Data(DataReceipt { data_id: [32 bytes], data: Some([4194304 bytes]) }) }`.
5. Borsh-serialized size = 4,194,304 (data) + ~200 (envelope) = **4,194,504 bytes > max_receipt_size**.
6. `validate_data_receipt` is called, not `validate_receipt` with the full size check, so no error is raised.
7. The oversized receipt is committed to state and forwarded to the next shard.

The test at `test-loop-tests/src/tests/max_receipt_size.rs:216` (`test_max_receipt_size_value_return`) executes exactly this scenario and asserts that an oversized receipt is found in the chain: [9](#0-8)

fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())
}
