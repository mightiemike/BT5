Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual swapper, allowing full allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives the `sender` parameter from the pool, which is set to `msg.sender` of the `pool.swap()` call. When any user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract. If the pool admin allowlists the router (required for any allowlisted user to use the router), every non-allowlisted user can bypass the gate by calling the router instead of the pool directly.

## Finding Description

**Step 1 — Pool passes `msg.sender` (the router) as `sender` to extensions:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that value unchanged:**

`_callExtensionsInOrder` is invoked with `sender` encoded directly into the call payload. [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`:**

`msg.sender` inside the extension is the pool; `sender` is whoever called `pool.swap()` — the router, not the end user. [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender`:** [4](#0-3) 

The same identity collapse occurs in `exactInput` (multi-hop, router is caller for every hop): [5](#0-4) 

And in `_exactOutputIterateCallback` (recursive exact-output hops, router calls pool from within the callback): [6](#0-5) 

**Why existing guards fail:** The extension has no mechanism to distinguish the router acting on behalf of an allowlisted user from the router acting on behalf of an arbitrary user. The check `allowedSwapper[msg.sender][sender]` evaluates `allowedSwapper[pool][router]`, which is `true` for any caller once the router is allowlisted. There is no `originator` field, no `extensionData` decoding of the real user, and no on-chain way for the extension to recover the actual end user from the router call.

The pool admin faces an impossible choice: not allowlisting the router blocks all router-mediated swaps (including for legitimately allowlisted users); allowlisting the router opens the gate to every address on-chain.

## Impact Explanation

Any unprivileged user can execute swaps on a pool whose admin intended to restrict swap access to specific addresses. The bypass moves the pool's bin cursor, consumes LP liquidity, and extracts output tokens — constituting direct loss of LP assets and complete breakage of the swap-gating core pool functionality. This meets the "Broken core pool functionality causing loss of funds" and "Admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation

The bypass is reachable by any address with no special privileges. The only prerequisite — the router being allowlisted — is the natural and expected configuration for any pool intended to be usable through the protocol's own periphery. A pool admin who does not allowlist the router renders the pool unusable via the router for all users, including their own allowlisted ones, creating strong incentive to allowlist the router. The exploit is a single direct call to `MetricOmmSimpleRouter.exactInputSingle` and is fully repeatable.

## Recommendation

The `sender` forwarded to extensions must represent the economic actor, not the intermediary:

1. **Router-side fix**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` (e.g., prepend `abi.encode(msg.sender)`) before passing it to `pool.swap`, so extensions can decode and verify the real user.
2. **Extension-side fix**: `SwapAllowlistExtension.beforeSwap` should decode the real originator from `extensionData` when `sender` is a known/trusted router, or the pool interface should carry an explicit `originator` field through the hook chain.

Until fixed, pool admins must be warned that allowlisting the router is functionally equivalent to calling `setAllowAllSwappers(pool, true)`.

## Proof of Concept

```
1. Deploy MetricOmmPool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
2. Pool admin calls:
     extension.setAllowedToSwap(pool, alice, true);   // allowlist Alice
     extension.setAllowedToSwap(pool, router, true);  // allow router so Alice can use it
3. Bob (not allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: bob, ...})
4. Inside pool.swap(), msg.sender == router (not Bob).
5. Extension evaluates: allowedSwapper[pool][router] == true → passes.
6. Bob's swap executes on the restricted pool, extracting output tokens.
7. Verification: Bob calling pool.swap() directly would revert:
     allowedSwapper[pool][bob] == false → NotAllowedToSwap.
```
