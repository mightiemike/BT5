### Title
`Verifier` Inherits `EIP712Upgradeable` But Never Initializes or Uses It — (File: `core/contracts/Verifier.sol`)

---

### Summary
`Verifier` inherits from OpenZeppelin's `EIP712Upgradeable` but never calls `__EIP712_init()` in its `initialize()` function, and no method in the contract ever invokes `_hashTypedDataV4`, `_domainSeparatorV4`, or any other EIP712 state. The inheritance is entirely dead, creating a false impression that `Verifier` applies EIP712 domain separation to its signature verification logic.

---

### Finding Description

`Verifier` is declared as:

```solidity
contract Verifier is EIP712Upgradeable, OwnableUpgradeable, IVerifier {
```

Its `initialize` function only calls `__Ownable_init()` and never calls `__EIP712_init()`:

```solidity
function initialize(Point[8] memory initialSet) external initializer {
    __Ownable_init();
    // __EIP712_init is never called
    ...
}
```

The two primary signature-verification entry points, `validateSignature` and `validateCompactSignature`, are both declared `pure` — they perform raw ECDSA recovery against a caller-supplied `digest` with no domain-separator contribution from `Verifier` itself:

```solidity
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure { ... }

function validateCompactSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    IEndpoint.CompactSignature memory signature
) public pure { ... }
```

Similarly, `computeDigest` is `pure` and returns only the raw struct hash, with no domain separator applied:

```solidity
function computeDigest(
    IEndpoint.TransactionType txType,
    bytes calldata transactionBody
) public pure returns (bytes32) { ... }
```

The actual EIP712 domain wrapping (`_hashTypedDataV4`) is applied in `EndpointTx`, not in `Verifier`. `Verifier`'s `EIP712Upgradeable` inheritance contributes nothing to the protocol's security model.

---

### Impact Explanation

**Low.** The EIP712 domain separation that protects user-signed transactions is correctly applied in `EndpointTx` via `_hashTypedDataV4`. No signature verification is actually broken. The harm is a false sense of security: a developer or auditor reading `Verifier` may assume it enforces EIP712 domain binding on the digests it validates, when in fact it does not. This could lead to incorrect reasoning about the trust boundary of `Verifier.validateSignature` if it is ever called with a digest that has not already been domain-wrapped by the caller.

---

### Likelihood Explanation

**High.** It is certain that `Verifier` never uses its inherited EIP712 state. `__EIP712_init` is never called, and no function in the contract references `_hashTypedDataV4` or `_domainSeparatorV4`. This is statically verifiable.

---

### Recommendation

Either remove `EIP712Upgradeable` from `Verifier`'s inheritance list entirely (since domain separation is handled by `EndpointTx`), or, if `Verifier` is intended to own the domain-separation step, move the `_hashTypedDataV4` call into `Verifier.computeDigest` and call `__EIP712_init("Nado", "0.0.1")` in `initialize`.

---
