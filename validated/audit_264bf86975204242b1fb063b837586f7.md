Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Originating User, Allowing Any Caller to Bypass Per-Pool Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When swaps are routed through `MetricOmmSimpleRouter`, the pool receives the router contract as `msg.sender`, so the extension checks `allowedSwapper[pool][router]` rather than the originating EOA. A pool admin who allowlists the router to enable router-mediated swaps for approved users inadvertently grants swap access to every caller of the router, completely nullifying the allowlist.

## Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the calling pool: [1](#0-0) 

`MetricOmmPool.swap` passes its own `msg.sender` (the direct caller of `pool.swap`) as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards this `sender` unchanged to the extension: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The originating EOA (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes and is never forwarded to the pool or extension as the swap initiator. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

This creates an irresolvable dilemma: if the admin does not allowlist the router, no allowlisted user can use it; if the admin does allowlist the router, every user who calls the router bypasses the allowlist. The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

## Impact Explanation

Any unprivileged user can bypass a `SwapAllowlistExtension`-gated pool by calling any of the four `MetricOmmSimpleRouter` swap entry points. The allowlist is a core access-control mechanism intended to restrict who may trade — for compliance, risk management, or exclusive-access pools. With the router allowlisted, the restriction is completely nullified: users who should be blocked can execute swaps against oracle-anchored prices that were only intended for approved counterparties, constituting a broken core pool functionality and potential direct loss of funds to pool LPs.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard publicly deployed periphery contract; any EOA can call it. A pool admin who wants allowlisted users to trade through the router **must** allowlist the router address — there is no alternative mechanism. The bypass requires no special privileges, no malicious setup, and no non-standard token behavior. Any EOA can trigger it repeatably.

## Recommendation

The extension must check the originating user, not the direct pool caller. The cleanest production fix is to have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known router address. Alternatively, the pool interface could be extended with a dedicated `originator` field that the router populates, allowing the extension to gate on the true initiator regardless of intermediary contracts.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap hook
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is approved
  - Pool admin calls setAllowedToSwap(pool, router, true)  // required for alice to use router

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(...) → pool sees msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - extension checks allowedSwapper[pool][router] → true
  - bob's swap executes successfully despite not being on the allowlist

Result:
  - bob trades against a restricted pool
  - The allowlist invariant is broken for all four router entry points
  - Any user with router access can trade, draining pool liquidity at oracle prices
    intended only for approved counterparties
```
