### Title
NLP Mint/Burn Oracle Price Not Committed to by User Signature, Enabling Sequencer-Controlled Slippage — (File: `core/contracts/Clearinghouse.sol`, `core/contracts/interfaces/IEndpoint.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` transaction structs that users sign do not include any oracle price field or price-limit constraint. The `oraclePriceX18` used to compute how many NLP tokens a user receives (or how much quote a user receives on burn) is provided exclusively by the sequencer in the outer `SignedMintNlp`/`SignedBurnNlp` wrapper and is never committed to by the user's EIP-712 signature. There is no on-chain check that the execution price falls within any user-acceptable range. The error constant `ERR_SLIPPAGE_TOO_HIGH` is defined but is not applied anywhere in the NLP mint/burn execution path.

---

### Finding Description

When a user mints NLP, they sign the inner `MintNlp` struct:

```solidity
struct MintNlp {
    bytes32 sender;
    uint128 quoteAmount;
    uint64 nonce;
}
``` [1](#0-0) 

The outer `SignedMintNlp` struct, which the sequencer constructs and submits, contains `oraclePriceX18` and `nlpPoolRebalanceX18` — fields that are **not** part of the inner struct the user signs:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;
    int128[] nlpPoolRebalanceX18;
}
``` [2](#0-1) 

In `EndpointTx.processTransactionImpl`, `validateSignedTx` is called with `signedTx.tx.sender` and `signedTx.tx.nonce`, meaning the EIP-712 digest is computed over the inner `MintNlp` struct only. The sequencer-supplied `oraclePriceX18` is then passed directly to `clearinghouse.mintNlp` without any user-signed constraint:

```solidity
validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, nlpPools, signedTx.nlpPoolRebalanceX18);
``` [3](#0-2) 

Inside `Clearinghouse.mintNlp`, the NLP amount the user receives is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [4](#0-3) 

And for `burnNlp`, the quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
``` [5](#0-4) 

Neither `mintNlp` nor `burnNlp` contains any check that `oraclePriceX18` falls within a user-specified acceptable range. The error constant `ERR_SLIPPAGE_TOO_HIGH` exists in the codebase but is never applied in this execution path:

```solidity
string constant ERR_SLIPPAGE_TOO_HIGH = "STH";
``` [6](#0-5) 

The identical pattern applies to `BurnNlp`: [7](#0-6) [8](#0-7)

### Impact Explanation

**For `mintNlp`:** A user commits to spending a fixed `quoteAmount`. The NLP tokens received are `quoteAmount / oraclePriceX18`. If the sequencer supplies an inflated `oraclePriceX18`, the user receives proportionally fewer NLP tokens for the same quote spent — a direct, quantifiable asset loss with no on-chain recourse.

**For `burnNlp`:** A user commits to burning a fixed `nlpAmount`. The quote returned is `nlpAmount * oraclePriceX18 - burnFee`. If the sequencer supplies a deflated `oraclePriceX18`, the user receives proportionally less quote — again a direct asset loss.

In both cases the corrupted state delta is the user's quote balance (mint: overpaid in quote per NLP unit received; burn: underpaid in quote per NLP unit burned). There is no health check or invariant that catches a price that is merely unfavorable rather than zero.

---

### Likelihood Explanation

The sequencer is the only party that can submit `processTransactionImpl` calls. Exploitation requires sequencer misbehavior or compromise. However, the protocol provides users with **zero on-chain defense-in-depth**: the `MintNlp`/`BurnNlp` structs have no price-limit field, the user's signature does not commit to any price, and no slippage guard is applied at execution. If the sequencer is compromised, goes rogue, or is subject to a key-extraction attack, every pending NLP mint/burn can be executed at an arbitrarily unfavorable price. The existence of `ERR_SLIPPAGE_TOO_HIGH` as an unused constant suggests this gap was recognized but not closed.

---

### Recommendation

Add `minNlpAmountOut` (for mint) and `minQuoteAmountOut` (for burn) fields to the inner `MintNlp` and `BurnNlp` structs so they are covered by the user's EIP-712 signature. In `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp`, assert the computed output meets the user's stated minimum before applying balance updates, using the existing `ERR_SLIPPAGE_TOO_HIGH` error.

---

### Proof of Concept

1. User signs `MintNlp { sender: alice, quoteAmount: 1000e18, nonce: 5 }` intending to mint NLP at the current fair price of 10 USDC/NLP (expecting ~100 NLP).
2. Sequencer constructs `SignedMintNlp` with `oraclePriceX18 = 100e18` (10× inflated) and submits to `Endpoint`.
3. `validateSignedTx` passes — the user's signature is valid over `(alice, 1000e18, 5)`.
4. `clearinghouse.mintNlp` computes `nlpAmount = 1000e18 / 100e18 = 10` NLP instead of 100 NLP.
5. Alice's quote balance decreases by 1000 USDC; she receives only 10 NLP instead of 100 — a 90% loss with no on-chain check to prevent it.
