### Title
`ValidResourceBounds::AllResources` with zero L2/L1DataGas silently collapses to `L1Gas` in protobuf deserialization, producing a divergent transaction hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary
The protobuf deserialization of `ValidResourceBounds` uses a value-based heuristic to distinguish between the `L1Gas` and `AllResources` variants. An `AllResources` transaction whose `l2_gas` and `l1_data_gas` fields are both zero is silently converted to `L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage for `AllResources` but omits it for `L1Gas`, the two variants produce different transaction hashes for the same logical transaction. This breaks the canonicalization invariant across the p2p sync boundary.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs`, the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation selects the variant purely by inspecting field values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

`ResourceBounds::is_zero()` returns `true` only when both `max_amount == 0` and `max_price_per_unit == 0`: [2](#0-1) 

Therefore any `AllResources` transaction whose `l2_gas` and `l1_data_gas` are `{amount:0, price:0}` is deserialized as `L1Gas` after a protobuf round-trip. The codebase itself acknowledges this in the consensus test:

```rust

The hash divergence is rooted in `get_tip_resource_bounds_hash`:

```rust

    ValidResourceBounds::L1Gas(_) => vec![],                          // ← omits l1_data_gas
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← includes it

``` [4](#0-3) 

Concretely, for the same `l1_gas` value and `tip`:

| Variant | Poseidon preimage |
|---|---|
| `L1Gas(l1_gas)` | `[tip, l1_gas_packed, l2_gas_packed(=0)]` |
| `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)` | `[tip, l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)]` |

These produce **different hashes H1 ≠ H2** because the `AllResources` path appends an extra `l1_data_gas_packed(=0)` element.

The p2p sync path uses `InvokeTransactionV3` (which carries `ValidResourceBounds`) serialized through this same protobuf converter: [5](#0-4) 

The `get_invoke_transaction_v3_hash` function calls `get_tip_resource_bounds_hash` and therefore inherits the divergence: [6](#0-5) 

### Impact Explanation

A transaction submitted with `AllResources(l2_gas={0,0}, l1_data_gas={0,0})` is assigned hash H1 at the gateway (using the `AllResources` path). After the transaction is propagated via p2p sync and deserialized on a peer node, the `ValidResourceBounds` variant silently becomes `L1Gas`. Any code path on the receiving node that recomputes the transaction hash from the deserialized struct — including `validate_transaction_hash`, transaction commitment computation, or RPC hash queries — will produce H2 ≠ H1. This causes the receiving node to bind the wrong hash to the transaction, matching the "transaction conversion or signature/hash logic binds the wrong hash or executable payload" impact category.

### Likelihood Explanation

The trigger requires an `AllResources` transaction where both `l2_gas.max_price_per_unit == 0` and `l1_data_gas.max_price_per_unit == 0` (in addition to zero amounts). This is a valid on-chain configuration (zero-price bounds are used in fee-free/bootstrap transactions). The trigger is unprivileged: any user can craft such a transaction. The codebase's own test workaround (`add_gas_values_to_transaction`) confirms the developers know this path is reachable.

### Recommendation

Replace the value-based heuristic with an explicit discriminant. Either:
1. Add a boolean tag field to the protobuf `ResourceBounds` message to encode the variant, or
2. Change the deserialization to always produce `AllResources` when all three gas fields are present in the wire message, regardless of their values — reserving `L1Gas` only for the legacy case where `l1_data_gas` is absent from the wire message (the `unwrap_or_default()` branch at line 427). use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::block::GasPrice;
use starknet_api::execution_resources::GasAmount;
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;

// Construct AllResources with zero l2_gas and l1_data_gas
let l1_gas = ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) };
let zero  = ResourceBounds { max_amount: GasAmount(0),   max_price_per_unit: GasPrice(0) };
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas, l2_gas: zero, l1_data_gas: zero,
});
let tip = Tip(0);

// H1: hash computed with AllResources (includes l1_data_gas in preimage)
let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();

// Simulate protobuf round-trip: AllResources → protobuf → L1Gas
// (because l2_gas.is_zero() && l1_data_gas.is_zero())
let l1_gas_only = ValidResourceBounds::L1Gas(l1_gas);

// H2: hash computed with L1Gas (omits l1_data_gas from preimage)
let h2 = get_tip_resource_bounds_hash(&l1_gas_only, &tip).unwrap();

assert_ne!(h1, h2); // ← divergent hashes for the same logical transaction

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}
