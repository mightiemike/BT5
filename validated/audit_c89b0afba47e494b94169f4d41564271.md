### Title
Asymmetric Taker Fee Gross-Up in `applyFee` Systematically Overcharges Buy-Side Taker Orders — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

The `applyFee` function in `OffchainExchange.sol` applies a **directionally asymmetric fee formula** to taker orders. When a taker is buying (negative `quoteDelta`), the fee is computed using a gross-up division (`meteredQuote.div(keepRateX18)`), yielding an effective fee rate of `feeRate / (1 - feeRate)`. When a taker is selling (positive `quoteDelta`), the fee is computed using standard multiplication (`meteredQuote.mul(keepRateX18)`), yielding an effective fee rate of exactly `feeRate`. This is the direct Nado analog of the BPool double-fee issue: two economically symmetric operations (buy vs. sell) produce materially different fee burdens, with the discrepancy scaling sharply at higher fee rates.

---

### Finding Description

In `OffchainExchange.applyFee`, after computing `meteredQuote`, the fee is calculated as:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [1](#0-0) 

**Sell-side taker** (`meteredQuote > 0`, receiving quote):
- `newMeteredQuote = meteredQuote × (1 − feeRate)`
- `fee = meteredQuote × feeRate`
- Effective fee rate = `feeRate` ✓

**Buy-side taker** (`meteredQuote < 0`, paying quote):
- `newMeteredQuote = meteredQuote / (1 − feeRate)`
- `fee = meteredQuote − meteredQuote / (1 − feeRate) = |meteredQuote| × feeRate / (1 − feeRate)`
- Effective fee rate = `feeRate / (1 − feeRate)` — **higher than stated**

The `builderFee` is then separately deducted on top of this, calculated symmetrically on `matchQuote.abs()`:

```solidity
orderInfo.builderFee = matchQuote.abs().mul(feeInfo.builderFeeRate);
orderInfo.quoteDelta = orderInfo.quoteDelta - orderInfo.fee - orderInfo.builderFee;
``` [2](#0-1) 

The `applyFee` function is called for both taker and maker orders during `matchOrders`: [3](#0-2) 

---

### Impact Explanation

At a 10% taker fee rate:
- A sell-side taker trading 100 USDC notional pays **10 USDC** in fees (10% effective rate), netting 90 USDC.
- A buy-side taker trading 100 USDC notional pays **11.11 USDC** in fees (11.11% effective rate), spending 111.11 USDC total.

The discrepancy is `feeRate / (1 − feeRate) − feeRate = feeRate² / (1 − feeRate)`. At 10% fee this is ~1.11%; at 20% fee it is 5%; at 30% fee it is ~12.86%. The extra fees are credited to `market.collectedFees` and ultimately flow to the protocol, not to any counterparty — but the buy-side taker is systematically overcharged relative to the stated fee rate. Any user who models their expected cost using the advertised `takerRateX18` will receive less `tokenOut` than expected when buying, with the magnitude of the shortfall scaling with fee rate. This is a concrete, quantifiable asset delta on every buy-side taker fill.

---

### Likelihood Explanation

This affects **every taker buy order** (negative `quoteDelta`) on every product, at all times. No special conditions are required. The effect is small at low fee tiers but becomes material at higher fee tiers. Any trader, liquidator, or NLP pool acting as a taker on the buy side is affected through the standard `matchOrders` entrypoint, which is the primary trading path in the protocol. [4](#0-3) 

---

### Recommendation

Apply the same fee formula for both positive and negative `meteredQuote`. The standard approach is to always use multiplication:

```solidity
// Symmetric: fee = |meteredQuote| * feeRate for both directions
int128 newMeteredQuote = meteredQuote.mul(keepRateX18);
```

If the gross-up behavior for buyers is intentional (e.g., to ensure the protocol always receives `feeRate` of the gross settlement value), this asymmetry must be explicitly documented in the fee rate disclosures so that users can correctly model their expected costs. The fee tier configuration and any user-facing documentation should reflect the true effective rate for buy-side takers.

---

### Proof of Concept

Concrete numeric example with `feeRate = 10%` (`keepRateX18 = 0.9`):

**Buy-side taker** (`meteredQuote = −100`):
```
newMeteredQuote = −100 / 0.9 = −111.11
fee             = −100 − (−111.11) = +11.11
quoteDelta      = −100 − 11.11 = −111.11   ← pays 111.11 USDC
effective rate  = 11.11 / 100 = 11.11%
```

**Sell-side taker** (`meteredQuote = +100`):
```
newMeteredQuote = 100 × 0.9 = 90
fee             = 100 − 90 = 10
quoteDelta      = 100 − 10 = 90            ← receives 90 USDC
effective rate  = 10 / 100 = 10.00%
```

For the same 100 USDC notional, the buy-side taker pays **1.11 USDC more** in fees than the sell-side taker. At a 30% fee tier, the buy-side taker pays **12.86 USDC more** per 100 USDC notional. This discrepancy is realized on every matched order where the taker is the buyer, through the `matchOrders` → `applyFee` path. [5](#0-4)

        uint32 productId,
        OrderInfo memory orderInfo,
        MarketInfo memory market,
        int128 alreadyMatched, // in quote
        uint128 appendix,
        bool taker
    ) internal {
        // X account is passthrough for trading and incurs
        // no fees
        if (orderInfo.sender == X_ACCOUNT) {
            return;
        }
        int128 matchQuote = orderInfo.quoteDelta;
        int128 meteredQuote = 0;
        if (taker) {
            // flat minimum fee
            if (alreadyMatched == 0) {
                meteredQuote += market.minSize;
                if (matchQuote < 0) {
                    meteredQuote = -meteredQuote;
                }
            }

            // exclude the portion on [0, self.min_size) for match_quote and
            // add to metered_quote
            // fee is only applied on [minSize, quote_amount)
            int128 feeApplied = MathHelper.abs(alreadyMatched + matchQuote) -
                market.minSize;
            feeApplied = MathHelper.min(feeApplied, matchQuote.abs());
            if (feeApplied > 0) {
                if (matchQuote < 0) {
                    feeApplied = -feeApplied;
                }
                meteredQuote += feeApplied;
            }
        } else {
            // for maker rebates things stay the same
            meteredQuote += matchQuote;
        }
        FeeInfo memory feeInfo = getUserFeeRateWithBuilder(
            orderInfo.sender,
            productId,
            appendix,
            taker
        );

        int128 keepRateX18 = ONE - feeInfo.feeRate;
        int128 newMeteredQuote = (meteredQuote > 0)
            ? meteredQuote.mul(keepRateX18)
            : meteredQuote.div(keepRateX18);
        orderInfo.fee = meteredQuote - newMeteredQuote;
        orderInfo.builderFee = matchQuote.abs().mul(feeInfo.builderFeeRate);
        orderInfo.quoteDelta =

            orderInfo.fee -
            orderInfo.builderFee;

            collectedBuilderFee[market.quoteId][feeInfo.builderId] += orderInfo
                .builderFee;
            emitBuilderEvent(orderInfo, feeInfo.builderId, productId);
        }
    }
