### Title
`SwapAllowlistExtension` gates the router address instead of the actual end-user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router contract**, not the actual end-user. The allowlist therefore gates the router's address, not the human trader. Any non-allowlisted user can bypass a pool's swap allowlist by calling through the public router.

---

### Finding Description

`ExtensionCalling._beforeSwap` forwards `msg.sender` of the pool's `swap` call as the `sender` argument to every registered extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInput*`, the router calls `pool.swap(recipient, ...)` directly. Inside the pool, `msg.sender` is the **router address**. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

This is structurally opposite to how `DepositAllowlistExtension` works: the deposit guard checks the `owner` argument (the position owner who receives LP shares — the economically relevant party), not the `sender` (the caller who pays): [3](#0-2) 

The swap allowlist has no equivalent "check the economically relevant actor" logic; it checks the intermediary.

---

### Impact Explanation

**Bypass path (Critical/High):** A pool admin deploys a swap allowlist to restrict trading to KYC'd or whitelisted addresses. The admin allowlists the `MetricOmmSimpleRouter` (or sets `allowAllSwappers = true` for it) so that allowlisted users can trade via the router. Because the check is on the router address, **every user** — including non-allowlisted ones — can call `MetricOmmSimpleRouter.exactInputSingle` and pass the guard. The allowlist is entirely ineffective.

**Blocking path (Medium):** If the admin does not allowlist the router, then individually allowlisted users who attempt to trade through the router are blocked, even though they are on the allowlist. Legitimate users are denied access to the pool's swap functionality.

Both outcomes break the core invariant that the swap allowlist gates the actual swapper.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary public entry point for swaps. Any user who calls it triggers the bypass. No special privileges, flash loans, or unusual token behavior are required. The only precondition is that the pool has a `SwapAllowlistExtension` configured — a standard production deployment scenario explicitly described in the periphery documentation.

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to accept and check a user-supplied identity from `extensionData`, or — more robustly — mirror the deposit allowlist pattern by checking the `recipient` (the economically relevant party who receives output tokens) rather than `sender`. Alternatively, the pool's `swap` function should pass the original caller through a dedicated field that the router populates (similar to how `addLiquidity` separates `sender` from `owner`).

A minimal fix:

```solidity
// Instead of checking sender (the router), check recipient (the actual beneficiary)
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Or, for full correctness, introduce an `originalSender` field in the swap parameters that the router populates with `msg.sender` before calling the pool.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as the `beforeSwap` hook.
2. Admin calls `setAllowedToSwap(pool, router, true)` so that router-mediated swaps are permitted for allowlisted users.
3. `alice` (non-allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. The router calls `pool.swap(alice, ...)`. Inside the pool, `msg.sender = router`.
5. `_beforeSwap(router, alice, ...)` is dispatched. The extension checks `allowedSwapper[pool][router]` → `true`.
6. The swap proceeds. `alice` — who was never allowlisted — successfully swaps in a restricted pool.

Conversely, if step 2 is omitted (router not allowlisted), an allowlisted user `bob` who calls through the router is blocked at step 5 because `allowedSwapper[pool][router]` is `false`, even though `allowedSwapper[pool][bob]` is `true`. [2](#0-1) [1](#0-0) [3](#0-2)
