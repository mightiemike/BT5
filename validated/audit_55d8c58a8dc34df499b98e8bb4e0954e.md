### Title
`state_diff_commitment` Omitted from `CommitmentParts` Consensus Message Allows Proposer to Bind Arbitrary State-Diff Hash in `PartialBlockHash` — (File: `crates/apollo_protobuf/src/converters/consensus.rs`)

---

### Summary

The `CommitmentParts` struct carried in every `ProposalFin` consensus message is missing the `state_diff_commitment` field. Because `state_diff_commitment` is a required input to `calculate_block_hash` (and therefore to `PartialBlockHash`), a receiving validator cannot reconstruct or verify the `state_diff_commitment` component of the `proposal_commitment` from the data it receives over the wire. A malicious proposer can therefore embed an arbitrary `state_diff_commitment` inside the `proposal_commitment` it broadcasts, and peers will vote on it without being able to detect the substitution.

---

### Finding Description

`BlockHeaderCommitments` has five fields that all feed into `calculate_block_hash`:

```
transaction_commitment, event_commitment, receipt_commitment,
state_diff_commitment, concatenated_counts (packed_lengths)
``` [1](#0-0) 

`calculate_block_hash` (Rust) chains all five into the Poseidon hash: [2](#0-1) 

`PartialBlockHash::from_partial_block_hash_components` calls `calculate_block_hash` with a zero state-root and zero parent-hash, so the `PartialBlockHash` that becomes `proposal_commitment` also depends on `state_diff_commitment`. [3](#0-2) 

The consensus `CommitmentParts` struct, however, only carries four of the five fields — `state_diff_commitment` is absent: [4](#0-3) 

The protobuf wire definition mirrors this omission: [5](#0-4) 

The conversion from `FinishedProposalInfo` to `CommitmentParts` explicitly maps only the four present fields and silently drops `state_diff_commitment`: [6](#0-5) 

The serializer (`From<CommitmentParts> for protobuf::CommitmentParts`) likewise never writes `state_diff_commitment`: [7](#0-6) 

The analog to the external bug is exact: just as `totalCoverTokens` was never decremented when a policy was removed (leaving a stale inflated value), `state_diff_commitment` is never included when `BlockHeaderCommitments` is serialised into `CommitmentParts` (leaving a permanently absent field that the receiver can never verify).

---

### Impact Explanation

A receiving validator reconstructs the `PartialBlockHash` from the `CommitmentParts` it receives in `ProposalFin` and compares it against the `proposal_commitment`. Because `state_diff_commitment` is absent from `CommitmentParts`, the receiver has no way to check whether the `state_diff_commitment` embedded in the proposer's `proposal_commitment` matches the actual state diff that was executed. A Byzantine proposer can:

1. Execute a valid block, obtaining the real `state_diff_commitment` `S`.
2. Substitute a different value `S'` when computing `PartialBlockHash`, producing a `proposal_commitment` that commits to `S'`.
3. Broadcast `CommitmentParts` (without `state_diff_commitment`) alongside this `proposal_commitment`.
4. Receiving validators cannot detect the substitution; they vote `Precommit` on a `proposal_commitment` that encodes the wrong state-diff hash.

The committed block hash therefore binds the wrong `state_diff_commitment`, breaking the integrity guarantee that the block hash canonically identifies the state transition. This matches the allowed impact: **"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result"** and **"High. Transaction conversion or signature/hash logic binds the wrong … hash … or executable payload."**

---

### Likelihood Explanation

Exploitability requires a Byzantine proposer — a node that has been elected as the current round's proposer and is willing to deviate from the protocol. In a BFT system with `f < n/3` Byzantine nodes this is a low-probability event in practice, but the protocol itself provides no cryptographic barrier once a Byzantine proposer is elected. Likelihood: **Low** (requires a malicious proposer), but the consequence is a permanently incorrect block hash on-chain.

---

### Recommendation

Add `state_diff_commitment` to `CommitmentParts` and to the protobuf `CommitmentParts` message (field 6), and populate it in both the `From<&FinishedProposalInfo> for CommitmentParts` conversion and the `From<CommitmentParts> for protobuf::CommitmentParts` serialiser. Receiving validators must then include the received `state_diff_commitment` when reconstructing `PartialBlockHash` to verify `proposal_commitment`.

---

### Proof of Concept

```
1. Proposer executes block B, obtains real state diff D with commitment S = hash(D).
2. Proposer picks arbitrary S' ≠ S.
3. Proposer computes PartialBlockHash using S' instead of S:
       proposal_commitment = calculate_block_hash(
           ..., state_diff_commitment = S', ...)
4. Proposer broadcasts ProposalFin {
       proposal_commitment,          // encodes S'
       fin_payload: CommitmentParts {
           transaction_commitment, event_commitment,
           receipt_commitment, concatenated_counts,
           // state_diff_commitment absent — field does not exist in struct
       }
   }
5. Each receiving validator:
       received_partial_hash = reconstruct_from(CommitmentParts)
       // reconstruction uses S=0 (default) because field is absent
       // received_partial_hash ≠ proposal_commitment, but validator
       // has no way to supply the correct S' to close the gap.
   Validator either (a) skips the check (accepting the wrong hash) or
   (b) always sees a mismatch and is forced to accept the proposer's
   assertion, giving the proposer full control over state_diff_commitment
   in the finalised block hash.
```
