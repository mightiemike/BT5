### Title
`burnNlp` gives caller less quote than implied by signed `nlpAmount` due to undisclosed `burnFee` and `HEALTHCHECK_FEE` deductions — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

When a user signs a `BurnNlp` transaction specifying `nlpAmount`, the EIP-712 digest commits only to `(sender, nlpAmount, nonce)`. The user expects to receive `nlpAmount × oraclePriceX18` in quote. However, two separate fee deductions silently reduce the actual quote received: a `burnFee` of `max($1, 0.1% of quoteAmount)` inside `Clearinghouse.burnNlp`, and a flat `HEALTHCHECK_FEE` of `$1` charged from the quote balance before the burn executes. Neither fee is committed to in the signed payload, and no minimum-output parameter exists in the struct.

---

### Finding Description

**Entry path — `EndpointTx.sol` (BurnNlp branch):**

```
chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);   // line 566 — deducts $1 from quote
clearinghouse.burnNlp(signedTx.tx, ...);           // line 568
``` [1](#0-0)

```solidity

int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);   // max($1, 0.1%)
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

``` [2](#0-1)

    signedTx.tx.sender,
    signedTx.tx.nlpAmount,
    signedTx.tx.nonce

``` [3](#0-2) 

**`BurnNlp` struct has no `minQuoteAmount` field:**

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [4](#0-3) 

**Fee constants:**

```solidity
int128 constant HEALTHCHECK_FEE = 1e18;  // $1
int128 constant ONE = 10**18;            // burnFee minimum = $1
``` [5](#0-4) 

The same pattern applies symmetrically to `mintNlp`: the user signs for `quoteAmount` but `HEALTHCHECK_FEE` is charged from the quote balance before `mintNlp` executes, so the user's total quote cost is `quoteAmount + $1` while receiving only `quoteAmount / oraclePriceX18` NLP — a higher effective price per NLP token than the oracle price implies. [6](#0-5) 

---

### Impact Explanation

Every user who calls `burnNlp` receives less quote than the full value of the NLP tokens they burned. For a burn of NLP worth $1,000:

- `burnFee = max($1, $1,000 × 0.1%) = $1`
- `HEALTHCHECK_FEE = $1`
- Actual quote received: **$998** instead of **$1,000**

The user's signed payload (`nlpAmount`) does not reflect this outcome. There is no on-chain slippage guard or minimum-output parameter, so the discrepancy is invisible to the user at signing time. The corrupted balance is the user's `QUOTE_PRODUCT_ID` balance, which is reduced by `burnFee + HEALTHCHECK_FEE` beyond what the signed `nlpAmount` implies.

---

### Likelihood Explanation

This affects every unprivileged user who submits a `BurnNlp` transaction through the normal sequencer path. No special conditions are required. Likelihood is high because it is a systematic, deterministic deduction applied to every burn.

---

### Recommendation

1. Add a `minQuoteAmount` field to the `BurnNlp` struct and include it in the EIP-712 digest, allowing users to commit to a minimum acceptable output on-chain.
2. Similarly, add a `maxTotalCost` or `maxFee` field to `MintNlp` so users can bound the total quote deducted (including `HEALTHCHECK_FEE`).
3. Update the DApp to clearly display the actual quote a user will receive after `burnFee` and `HEALTHCHECK_FEE` deductions before the user signs.

---

### Proof of Concept

1. User signs `BurnNlp { sender, nlpAmount = 1000e18, nonce }` — expecting to receive `1000 USDC` (assuming `oraclePriceX18 = 1e18`).
2. Sequencer submits the transaction to `EndpointTx`.
3. `chargeFee(sender, HEALTHCHECK_FEE)` deducts `$1` from the user's quote balance.
4. Inside `burnNlp`: `quoteAmount = 1000e18`, `burnFee = max(1e18, 1000e18/1000) = 1e18`, `quoteAmount = 999e18`.
5. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, sender, 999e18)` — user receives `$999`.
6. Net quote change for user: `+999 - 1 (HEALTHCHECK_FEE) = +$998` instead of the expected `+$1,000`.
7. The user's signed payload gave no indication of the `$2` shortfall, and no on-chain check enforces a minimum output.
