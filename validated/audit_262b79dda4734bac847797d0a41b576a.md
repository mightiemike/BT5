### Title
Unprotected `replaceUsdcEWithUsdc` with Hardcoded Token Addresses Allows Any Caller to Drain usdcE from Arbitrary User DDAs — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` carries no access-control modifier and embeds two hardcoded token addresses. Any unprivileged caller can invoke it against any user's `DirectDepositV1` (DDA) contract, atomically withdrawing all usdcE from that DDA and receiving it in exchange for USDC they supply. The subaccount owner never consents to the swap, and if usdcE trades at a premium to USDC the attacker extracts value directly from the DDA.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` (lines 608–620) is declared `external` with no `onlyOwner`, `onlyDeployer`, or any other guard:

```solidity
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;   // hardcoded
    address usdc  = 0x2D270e6886d130D724215A266106e6832161EAEd;   // hardcoded
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
``` [1](#0-0) 

The execution sequence is:

1. `ContractOwner` (as the DDA's `Ownable` owner) calls `DirectDepositV1.withdraw(usdcE)`, which transfers the entire usdcE balance from the DDA to `ContractOwner`. [2](#0-1) 
2. `ContractOwner` pulls `balance` USDC from the external caller via `transferFrom`.
3. `ContractOwner` forwards the usdcE it just received to the external caller via `safeTransfer`.

Because `DirectDepositV1` is created by `ContractOwner` (via `new DirectDepositV1{salt:…}(…)`), `ContractOwner` is permanently the DDA's owner and can call `withdraw` on behalf of any caller who reaches `replaceUsdcEWithUsdc`. [3](#0-2) 

The two token addresses are hardcoded literals with no registry or upgrade path. [4](#0-3) 

---

### Impact Explanation

**Concrete asset delta:** The DDA's entire usdcE balance is atomically removed and replaced with an equal *token-count* of USDC. Because usdcE (bridged USDC.e) and native USDC do not always trade 1:1, an attacker who calls this function when usdcE is at a premium (e.g., 1.005 USDC per usdcE) supplies `N` USDC worth `N × 1.000` and receives `N` usdcE worth `N × 1.005`, extracting `N × 0.005` from the DDA owner at zero risk.

Even at parity the subaccount owner suffers an unauthorized change to the token composition of their deposit address. If the protocol's spot engine lists usdcE but not USDC (or vice-versa), the subsequent `creditDeposit()` call will silently skip the unrecognised token, effectively freezing the deposited value in the DDA until the owner manually intervenes.

---

### Likelihood Explanation

The only preconditions for the attacker are:
- Hold any amount of USDC on chain 57073 (Ink mainnet).
- Approve `ContractOwner` for that USDC amount.
- Know (or enumerate) a `subaccount` whose DDA holds usdcE.

All three are trivially satisfiable by any on-chain participant. The chain-ID guard (`block.chainid == 57073`) limits exposure to Ink mainnet but does not restrict the caller identity in any way. [5](#0-4) 

---

### Recommendation

1. **Add access control.** Gate `replaceUsdcEWithUsdc` with `onlyOwner` or `onlyDeployer` so only the protocol operator can trigger the migration.
2. **Remove hardcoded addresses.** Store `usdcE` and `usdc` as upgradeable state variables (or use a `ContractRegistry`) so they can be corrected without a full redeployment if the addresses change or were set incorrectly.
3. **Validate the exchange.** If the function must remain permissionless, enforce that the caller receives no more value than they supply (e.g., require an oracle-checked 1:1 value equivalence before executing the swap).

### Proof of Concept

```
// Attacker EOA on chain 57073
// Precondition: attacker holds ≥ N usdcE-equivalent USDC

1. USDC.approve(address(contractOwner), N)

2. contractOwner.replaceUsdcEWithUsdc(victimSubaccount)
   // ContractOwner.withdraw() pulls N usdcE from victim's DDA → ContractOwner
   // ContractOwner.transferFrom() pulls N USDC from attacker → victim's DDA
   // ContractOwner.safeTransfer() sends N usdcE to attacker

// Result:
//   Victim DDA: usdcE balance = 0, USDC balance = N
//   Attacker:   usdcE balance += N, USDC balance -= N
//   If usdcE/USDC > 1.0, attacker profits the spread; victim's DDA
//   composition changed without consent.
```
