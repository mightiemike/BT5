### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downcasts `AllResources` to `L1Gas`, Causing Consensus P2P Transaction Rejection - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation silently converts an `AllResources` variant to `L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. A downstream conversion (`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`) then hard-rejects any `L1Gas` variant. The result is that a valid V3 invoke transaction whose `AllResourceBounds` has `l2_gas = 0` and `l1_data_gas = 0` is accepted by the gateway, stored in the mempool, included in a proposed block, serialized to protobuf, but then **irrecoverably rejected** by every validator that deserializes it over the consensus P2P channel.

---

### Finding Description

**Step 1 – Serialization (proposer side)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. When the proposer serializes a `ConsensusTransaction` to protobuf it goes through:

```
RpcInvokeTransactionV3 → InvokeTransactionV3 (AllResources) → protobuf::InvokeV3
```

The `From<ValidResourceBounds> for protobuf::ResourceBounds` serializer always emits all three fields (`l1_gas`, `l2_gas`, `l1_data_gas`), so a transaction with `l2_gas = 0` and `l1_data_gas = 0` is faithfully written to the wire with those zero values. [1](#0-0) 

**Step 2 – Deserialization (validator side)**

On receipt, `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` calls `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3`, which internally calls `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`. That conversion delegates to:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← downcast happens here
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

**Step 3 – Hard rejection**

The deserialized `InvokeTransactionV3` (now carrying `L1Gas`) is then converted back to `RpcInvokeTransactionV3`:

```rust
impl TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3 {
    fn try_from(value: InvokeTransactionV3) -> Result<Self, Self::Error> {
        Ok(Self {
            resource_bounds: match value.resource_bounds {
                ValidResourceBounds::AllResources(bounds) => bounds,
                _ => {
                    return Err(StarknetApiError::OutOfRange { ... }); // ← always fails for L1Gas
                }
            },
            ...
        })
    }
}
``` [3](#0-2) 

The caller in `rpc_transaction.rs` maps this error to `DEPRECATED_RESOURCE_BOUNDS_ERROR`:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [4](#0-3) 

**Step 4 – Hash domain divergence (secondary effect)**

Even if the rejection were bypassed, `get_tip_resource_bounds_hash` produces structurally different hash preimages for the two variants:

- `L1Gas` → 2 resource-bound felts hashed: `[L1_GAS, L2_GAS]`
- `AllResources` → 3 resource-bound felts hashed: `[L1_GAS, L2_GAS, L1_DATA_GAS]` [5](#0-4) 

A transaction whose hash was computed with `AllResources` (3 elements) would fail `validate_transaction_hash` after being deserialized as `L1Gas` (2 elements), producing a different Poseidon digest.

---

### Impact Explanation

**Impact: High** — Valid transactions are rejected by every validator that receives them over the consensus P2P channel. The proposer includes the transaction in a block, serializes it, and sends it to peers. Each peer's `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` returns an error, causing the entire proposal to be rejected. This matches the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

**Likelihood: Medium** — Any V3 invoke transaction submitted with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` triggers the bug. The gateway's stateless validator accepts such transactions (non-zero l1_gas satisfies the resource-bounds check). The scenario is realistic for users who only care about L1 gas and leave the other two bounds at zero.

---

### Recommendation

Remove the `AllResources → L1Gas` downcast from the protobuf deserializer. When all three fields are present on the wire (even if zero), reconstruct `AllResources` unconditionally:

```rust
// In TryFrom<protobuf::ResourceBounds> for ValidResourceBounds
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

The `L1Gas` variant should only be produced when deserializing legacy pre-0.13.3 transactions that genuinely lack `l2_gas` and `l1_data_gas` fields on the wire (i.e., when `value.l2_gas` is `None`, not when it is `Some(zero)`).

---

### Proof of Concept

1. Construct a valid V3 invoke transaction:
   ```
   AllResourceBounds { l1_gas: { max_amount: 1000, max_price: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }
   ```
2. Submit it to the gateway — it is accepted and stored in the mempool.
3. The proposer includes it in a `ConsensusTransaction` block proposal and serializes it to `protobuf::ConsensusTransaction::InvokeV3`.
4. A validator receives the protobuf and calls `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`.
5. Internally, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` fires the branch at line 431: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
6. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hits the `_ => Err(...)` arm and returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
7. The validator rejects the entire proposal. The transaction can never be sequenced despite being valid. [6](#0-5) [7](#0-6) [8](#0-7)

    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));

        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;

            ValidResourceBounds::L1Gas(l1_gas)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
