Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks LP position recipient (`owner`) instead of actual token payer (`sender`), allowing non-allowlisted addresses to deposit into curated pools — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently drops the `sender` argument and checks only `owner` (the LP position recipient). Because `MetricOmmPool.addLiquidity` is fully public and accepts any caller-supplied `owner`, any non-allowlisted address can call the pool directly with an allowlisted `owner`, pass the guard, and deposit tokens into the pool without appearing on the allowlist.

## Finding Description
`DepositAllowlistExtension.beforeAddLiquidity` ignores its first parameter (`sender`) and gates only on `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`MetricOmmPool.addLiquidity` is a fully public function with no caller restriction. It passes `msg.sender` (the actual payer) as `sender` and the caller-supplied `owner` argument as `owner` to the extension dispatcher:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both arguments to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
_callExtensionsInOrder(
    BEFORE_ADD_LIQUIDITY_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
);
```

The extension receives `sender` (the non-allowlisted payer) as its first argument but never reads it. It reads only `owner`. The contract's own NatSpec states it "Gates `addLiquidity` by depositor address" and the setter uses the parameter name `depositor` (`setAllowedToDeposit(address pool_, address depositor, bool allowed)`), confirming the intent is to gate the actual payer — not the LP position recipient. The implementation checks the wrong actor.

Existing guards are insufficient: `removeLiquidity` correctly enforces `msg.sender == owner`, but `addLiquidity` has no analogous restriction on who may supply the `owner` argument.

## Impact Explanation
`DepositAllowlistExtension` is the pool admin's mechanism to enforce a curated depositor set (e.g., KYC/compliance-gated pools). By checking `owner` instead of `sender`, the guard is rendered ineffective against any non-allowlisted address that supplies an allowlisted `owner`. The non-allowlisted address successfully deposits tokens into the pool — the core action the guard was meant to block — without appearing on the allowlist. This is an **admin-boundary break**: the pool admin's deposit restriction is bypassed by an unprivileged, permissionless path. Any pool relying on `DepositAllowlistExtension` for compliance or access control cannot enforce that restriction on the actual payer.

## Likelihood Explanation
The attack requires no special privileges. `pool.addLiquidity` is fully public with no `onlyFactory` or similar guard. The only prerequisite is knowing one allowlisted address to supply as `owner`. Allowlisted addresses are discoverable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit`. The attacker must supply tokens (they pay into the pool), but they lose nothing if they collude with the allowlisted `owner` who can later call `removeLiquidity` (which enforces `msg.sender == owner`) and return the proceeds. The attack is repeatable with zero on-chain friction.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual payer/depositor) instead of `owner` (the LP position recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the LP position recipient, both `sender` and `owner` should be checked.

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` as an active extension.
2. Pool admin calls `extension.setAllowedToDeposit(pool, A, true)` — address A is allowlisted; address B is not.
3. Address B (a contract implementing `metricOmmModifyLiquidityCallback`) calls `pool.addLiquidity(owner = A, salt, deltas, callbackData, extensionData)` directly.
4. Pool calls `_beforeAddLiquidity(msg.sender = B, owner = A, …)`.
5. Extension evaluates `allowedDepositor[pool][A]` → `true` → guard passes.
6. Pool mints LP shares to position `(A, salt)`.
7. Pool invokes `metricOmmModifyLiquidityCallback` on B; B pays `amount0` and `amount1` tokens.
8. B (non-allowlisted) has successfully deposited tokens into the curated pool. A holds the LP position and can later call `removeLiquidity` to return proceeds to B.
