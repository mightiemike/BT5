### Title
Validator-Manipulable `block.timestamp` Bypasses 3-Day Slow Mode Time-Lock — (`File: core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary
The slow mode time-lock in Nado relies exclusively on `block.timestamp` for both recording when a transaction becomes executable and for enforcing the delay at execution time. A block proposer (validator) who can skew `block.timestamp` can cause the `executableAt` guard to pass prematurely, allowing slow mode transactions — including `WithdrawCollateral` — to be executed before the intended 3-day safety window has elapsed.

---

### Finding Description
Two code paths stamp slow mode transactions with `executableAt = uint64(block.timestamp) + SLOW_MODE_TX_DELAY`:

- `Endpoint.sol` `depositCollateralWithReferral` [1](#0-0) 
- `EndpointTx.sol` `submitSlowModeTransactionImpl` [2](#0-1) 

`SLOW_MODE_TX_DELAY` is defined as exactly 3 days (259 200 seconds): [3](#0-2) 

The enforcement gate in `_executeSlowModeTransaction` is:

```solidity
require(
    fromSequencer || (txn.executableAt <= block.timestamp),
    ERR_SLOW_TX_TOO_RECENT
);
``` [4](#0-3) 

`executeSlowModeTransaction()` is permissionless — any address may call it — and always passes `fromSequencer = false`, so the `block.timestamp` branch is the only guard: [5](#0-4) 

The attack surface is two-sided:

1. **At submission time** — a validator sets `block.timestamp` as low as consensus allows (just above the previous block), minimising `executableAt`.
2. **At execution time** — a validator sets `block.timestamp` as high as consensus allows, maximising the apparent elapsed time.

The combined drift from both manipulations reduces the effective delay below 3 days. On chains where the block proposer has wider latitude over `block.timestamp` (e.g., some EVM-compatible L2s or sidechains), the reduction can be substantial.

---

### Impact Explanation
Slow mode `WithdrawCollateral` transactions processed via `processSlowModeTransactionImpl` transfer user collateral out of the protocol: [6](#0-5) 

The 3-day delay is the sole on-chain safety window that allows the protocol to detect and respond to suspicious withdrawals before funds leave. Bypassing it — even partially — shortens the response window and can result in direct collateral loss for users whose withdrawal was submitted by a malicious actor or whose account was compromised.

---

### Likelihood Explanation
On Ethereum mainnet, per-block timestamp drift is bounded to roughly ±15 seconds, making a full 3-day bypass infeasible. However:

- On EVM-compatible chains where the sequencer or block proposer controls `block.timestamp` more freely, the window can be compressed significantly.
- The `executeSlowModeTransaction()` entry point is permissionless, so no privileged access is required beyond block-proposal rights.
- The vulnerability is structural: there is no secondary time source, no commit-reveal, and no sequencer co-signature required for the permissionless execution path.

Likelihood is **low on Ethereum L1**, **medium on chains with looser timestamp rules**.

---

### Recommendation
1. Record `executableAt` using a monotonic, manipulation-resistant counter (e.g., block number with a known block time) rather than raw `block.timestamp`.
2. Alternatively, require a sequencer co-signature even for the permissionless `executeSlowModeTransaction()` path, or add a secondary block-number-based guard alongside the timestamp check.
3. If `block.timestamp` must be retained, document the minimum trusted timestamp drift for each supported chain and add an explicit assertion that the drift cannot compress the delay below an acceptable threshold.

---

### Proof of Concept

1. User (or attacker) calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload. A colluding validator includes this transaction in a block with `block.timestamp` set to the minimum allowed value `T_min`. The stored `executableAt = T_min + 259200`.

2. Three days minus `Δ` seconds later (where `Δ` is the total achievable timestamp skew), a colluding validator includes a call to `executeSlowModeTransaction()` in a block with `block.timestamp = T_min + 259200`. The check `txn.executableAt <= block.timestamp` passes.

3. `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral(...)` executes, transferring collateral before the true 3-day window has elapsed. [6](#0-5)
