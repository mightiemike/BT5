Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating EOA, enabling allowlist bypass for all router-mediated swaps - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool always sets to its own `msg.sender`. When `MetricOmmSimpleRouter` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the originating EOA. The extension therefore checks the router's allowlist status rather than the individual user's, creating two mutually exclusive failure modes: allowlisted users are silently blocked from using the router (if the router is not allowlisted), or any unprivileged user can bypass the curated gate entirely by routing through the router (if the router is allowlisted to restore access).

## Finding Description
**Root cause — pool always passes its own `msg.sender` as `sender`:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when the swap originates from the router: [3](#0-2) 

**Router never forwards the originating EOA to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` stores the EOA only in transient callback context for payment settlement (`_setNextCallbackContext(..., msg.sender, ...)`), but the `pool.swap(...)` call receives `msg.sender = router`: [4](#0-3) 

The same applies to `exactInput` multi-hop swaps: [5](#0-4) 

**Exploit path:**
```
EOA (bob, not allowlisted) → MetricOmmSimpleRouter.exactInputSingle()
  → pool.swap(recipient, ...) [msg.sender = router]
    → _beforeSwap(sender=router, ...)
      → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          checks: allowedSwapper[pool][router] == true → passes
          bob's swap executes despite not being allowlisted
```

No existing guard prevents this. The extension has no mechanism to read the originating EOA from `extensionData` or any other field — it only receives the `sender` argument which is structurally bound to the pool's `msg.sender`.

## Impact Explanation
**Failure mode A (router not allowlisted — default state):** Every allowlisted user is silently blocked from using `MetricOmmSimpleRouter`. The router is not in `allowedSwapper[pool]`, so every router-mediated call reverts with `NotAllowedToSwap`. This is broken core pool functionality — allowlisted users lose access to multi-hop routing, deadline enforcement, and slippage protection and must call the pool directly.

**Failure mode B (pool admin allowlists the router to restore router access):** Any unprivileged EOA can bypass the curated gate by calling `exactInputSingle` or `exactInput` on the router. The entry `allowedSwapper[pool][router] = true` passes for every EOA regardless of individual allowlist status. This is a direct allowlist bypass with fund-impacting consequences on pools that rely on the gate for access control (KYC pools, private liquidity pools, restricted counterparty pools).

## Likelihood Explanation
The router is the primary user-facing swap interface deployed alongside the pool. Pool admins who configure `SwapAllowlistExtension` and want allowlisted users to access the router will inevitably encounter failure mode A and attempt to resolve it by allowlisting the router, directly triggering failure mode B. The bypass requires no special privilege — any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with a valid pool address and a non-zero swap amount.

## Recommendation
The pool must forward the originating user's address to extensions, not its own `msg.sender`. Two complementary fixes:

1. **Router side:** `MetricOmmSimpleRouter` should encode the originating `msg.sender` into `extensionData` before calling `pool.swap(...)`, so extensions can read it.
2. **Extension side:** `SwapAllowlistExtension.beforeSwap` should decode the originating user from `extensionData` rather than relying on the `sender` argument.

Alternatively, extend the pool's `swap` signature with an explicit `originator` address that the router populates with `msg.sender`, and have `ExtensionCalling._beforeSwap` forward that field to extensions instead of the pool's `msg.sender`.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted to enable router access

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(...) with msg.sender = router
  - pool calls _beforeSwap(sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes
  - bob's swap executes successfully despite not being allowlisted

Result:
  - The curated pool's allowlist is completely bypassed for any user going through the router
  - alice's individual allowlist entry is irrelevant for router-mediated swaps
```
