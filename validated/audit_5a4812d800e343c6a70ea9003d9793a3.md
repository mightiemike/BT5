### Title
Missing Governance Update Path for `decay_epoch_start` in `x/inflation` Module - (File: `x/inflation/keeper/msg_server.go`)

### Summary

The `x/inflation` module stores two distinct categories of state: `Params` (MaxSupply, BurnedAddresses, DecayRate) and `decay_epoch_start`. `MsgUpdateParams` updates only `Params`. There is no governance message, CLI command, or any on-chain transaction path to update `decay_epoch_start` after it is set at genesis or by an upgrade handler. This is a direct analog to the reported missing-update-function class: one critical module state field is permanently frozen while all other fields are governable.

---

### Finding Description

The `x/inflation` module uses two separate store keys:

- `ParamsKey` — holds `Params{MaxSupply, BurnedAddresses, DecayRate}`, updatable via `MsgUpdateParams` through governance.
- `DecayEpochStartKey` — holds the block height at which inflation decay begins, used every block in `DeflationCalculationFn`. [1](#0-0) 

`SetDecayEpochStart` exists as a keeper method and is called from `InitGenesis` and upgrade handlers: [2](#0-1) 

However, `MsgUpdateParams` — the only governance-accessible message in the module — calls only `SetParams` and never touches `decay_epoch_start`: [3](#0-2) 

The ADR explicitly documents this gap: *"It is set by the upgrade handler (e.g., v7.0.0 sets it to the upgrade activation height) or via genesis. It is not changed by `MsgUpdateParams`."* [4](#0-3) 

The proto service definition for `x/inflation` exposes only `UpdateParams` — there is no `UpdateDecayEpochStart` RPC: [5](#0-4) 

`decay_epoch_start` is also not exposed by the query interface — only `Params` is queryable: [6](#0-5) 

---

### Impact Explanation

`decay_epoch_start` is a direct input to the inflation decay formula executed every block:

```
months_elapsed = (current_height - decay_epoch_start) / blocks_per_month
inflation_rate = base_rate × (1 - decay_rate) ^ months_elapsed
``` [7](#0-6) 

If `decay_epoch_start` is set to an incorrect value — or if governance needs to reset it (e.g., after a chain halt/restart, a misconfigured upgrade, or a deliberate policy change to restart the decay clock) — there is no on-chain mechanism to do so. The consequences are:

- If `decay_epoch_start` is set too far in the past, `months_elapsed` is inflated, causing the decay factor to be applied too aggressively, suppressing inflation below the intended schedule and minting fewer tokens than the economic model requires.
- If `decay_epoch_start` is set too far in the future (or to 0, meaning not set), decay is never applied even when `DecayRate > 0`, causing the chain to mint tokens at the full base rate indefinitely, potentially breaching the `MaxSupply` cap and triggering a chain halt. [8](#0-7) 

The corrupted value is the on-chain token supply trajectory: either excess minting (leading to a supply-cap panic halt) or suppressed minting (breaking the intended economic schedule). Both are irreversible without a software upgrade.

---

### Likelihood Explanation

The v7.0.0 upgrade sets `decay_epoch_start` to the upgrade activation height. Any future upgrade that resets or adjusts inflation decay parameters (e.g., changing `DecayRate` to a new value and wanting to restart the decay clock from the new upgrade height) would require a new software upgrade solely to update `decay_epoch_start`, because governance cannot do it via `MsgUpdateParams`. This is a realistic operational scenario given the chain's documented upgrade history. [9](#0-8) 

---

### Recommendation

Add a `MsgUpdateDecayEpochStart` governance message (or extend `MsgUpdateParams` to include `decay_epoch_start` as an optional field) so that governance can update the decay epoch start on-chain:

```go
// In proto/chainmain/inflation/v1/tx.proto
rpc UpdateDecayEpochStart(MsgUpdateDecayEpochStart) returns (MsgUpdateDecayEpochStartResponse);

message MsgUpdateDecayEpochStart {
  option (cosmos.msg.v1.signer) = "authority";
  string authority        = 1 [(cosmos_proto.scalar) = "cosmos.AddressString"];
  uint64 decay_epoch_start = 2;
}
```

And in the msg server:

```go
func (k msgServer) UpdateDecayEpochStart(ctx context.Context, msg *types.MsgUpdateDecayEpochStart) (*types.MsgUpdateDecayEpochStartResponse, error) {
    if k.authority != msg.Authority {
        return nil, errors.Wrapf(govtypes.ErrInvalidSigner, "invalid authority; expected %s, got %s", k.authority, msg.Authority)
    }
    if err := k.SetDecayEpochStart(ctx, msg.DecayEpochStart); err != nil {
        return nil, err
    }
    return &types.MsgUpdateDecayEpochStartResponse{}, nil
}
```

---

### Proof of Concept

1. Chain launches with `decay_epoch_start = H` (set by upgrade handler at height H).
2. Governance passes a proposal to change `DecayRate` to a new value via `MsgUpdateParams` — succeeds, `DecayRate` is updated.
3. Governance wants to reset `decay_epoch_start` to the current height so decay restarts from now — submits `MsgUpdateParams` with the new `DecayRate`.
4. `MsgUpdateParams` handler calls only `SetParams`, never `SetDecayEpochStart`.
5. `decay_epoch_start` remains at the original height H. `months_elapsed` continues to grow from H, not from the intended new epoch.
6. The inflation rate is computed against the wrong reference point permanently. No governance transaction can correct this. [10](#0-9) [11](#0-10)

func (k Keeper) SetParams(ctx context.Context, params types.Params) error {
	if err := params.Validate(); err != nil {
		return err
	}
	store := k.storeService.OpenKVStore(ctx)
	bz := k.cdc.MustMarshal(&params)
	return store.Set([]byte(types.ParamsKey), bz)
}

// SetDecayEpochStart persists the block height at which inflation decay begins (e.g. upgrade activation height).
func (k Keeper) SetDecayEpochStart(ctx context.Context, height uint64) error {
	store := k.storeService.OpenKVStore(ctx)
	bz := make([]byte, 8)
	binary.BigEndian.PutUint64(bz, height)
	return store.Set([]byte(types.DecayEpochStartKey), bz)
}

			// Power() only accepts uint64, so decompose the exponent:
			// x^months = x^n × (x^(1/m))^r, where months = n + r/m
			// Note: we compute (x^(1/m))^r, NOT (x^r)^(1/m), because x^r underflows
			// to 0 for large r when x < 1, whereas x^(1/m) stays close to 1.
