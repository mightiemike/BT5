### Title
`OracleValueStopLossExtension` Fully-Decayed Watermark (Zero) Causes Stop-Loss Guard to Fail Open — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

The `OracleValueStopLossExtension` tracks per-bin high watermarks that decay linearly over time. When a pool is inactive long enough for `decayPerSecondE8 × dt ≥ 1e8`, `_decayed()` returns `0`. At that point `_applyWatermark(metric, 0, floor)` unconditionally returns `(metric, false)` — no breach — because any `uint256 metric ≥ 0` is always true. The stop-loss guard that is supposed to protect LP value fails completely open for the first swap after the dormancy window, the exact structural analog of the TokenFaucet "long-dormant faucet allows instant full withdrawal after refill" bug.

---

### Finding Description

**`_decayed` floors to zero:** [1](#0-0) 

```solidity
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;          // ← watermark collapses to zero
return hwm - (hwm * factor) / E8;
```

With the example rate of `58` (≈5 %/day), `factor ≥ 1e8` after `1e8 / 58 ≈ 1 724 138 s ≈ 20 days` of inactivity.

**`_applyWatermark` with `hwm = 0` never fires a breach:** [2](#0-1) 

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // ← uint256 metric ≥ 0 always true
    breached = metric < (hwm * floorMultiplier) / E6;
    return (hwm, breached);
}
```

When `hwm = 0`, the first branch is always taken: `breached = false`, `newHwm = metric` (whatever the current, possibly catastrophically low, per-share value is).

**`_checkAndUpdateWatermarks` propagates the bypass:** [3](#0-2) 

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) {
    revert OracleStopLossTriggered(...);   // ← never reached when hwm=0
}
hwmS.token0 = uint104(hwm0);             // ← silently resets to current (low) value
hwmS.lastDecayTs = uint32(block.timestamp);
```

The revert is never reached; the watermark is silently reset to the current (potentially near-zero) metric, and the swap that drained LP value succeeds.

**Hook is called after the swap has already settled:** [4](#0-3) 

The `_afterSwap` hook fires after token transfers are complete, so a bypassed revert means the value-extracting swap is irreversible.

---

### Impact Explanation

The contract's own stated invariant is broken:

> "Guarantee: value per share at oracle marks cannot fall faster than drawdown (one-time) + decay × t (ongoing)."

When the watermark has fully decayed to zero, value per share can fall to any level — including zero — without the stop-loss triggering. LPs who deployed capital into a pool that was configured with this extension for protection receive none of that protection after the dormancy window. Any swap that extracts LP value (e.g., via a stale or manipulated oracle price) passes unchecked, causing direct loss of LP principal.

---

### Likelihood Explanation

- Any pool that experiences a gap in swap activity longer than `1e8 / decayPerSecondE8` seconds enters the vulnerable window. At the documented example rate of `58`, that is ~20 days — a realistic dormancy period for low-volume pools.
- No privileged action is required to trigger the bypass; the attacker only needs to wait.
- The first swap after the window resets the watermark to whatever the current (post-drain) metric is, so the bypass is self-concealing.

---

### Recommendation

When the decayed watermark reaches zero, the guard must **fail closed** rather than open. Two complementary fixes:

1. **Floor the decayed watermark at a sentinel non-zero value** (e.g., `1`) so that `_applyWatermark` always evaluates the breach condition rather than short-circuiting.
2. **Treat `hwm == 0` as an uninitialized / fully-expired state and revert** (or block the swap direction) until the admin explicitly re-anchors the watermark via `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks`.

Either approach ensures the guard fails closed on a zero watermark, consistent with the "stale, zero, inverted, or rounded values fail closed" requirement.

---

### Proof of Concept

1. Pool is deployed with `OracleValueStopLossExtension`, `drawdownE6 = 100_000` (10 %), `decayPerSecondE8 = 58`.
2. Normal activity sets `highWatermarks[pool][0].token0 = H` (some meaningful value), `lastDecayTs = T0`.
3. Pool goes dormant. After `t > 1e8 / 58 ≈ 1 724 138 s` (≈20 days), `_decayed(H, 58, t)` returns `0`.
4. Oracle price becomes stale or is manipulated to a level that allows value extraction from bin 0.
5. Attacker calls `pool.swap(…, zeroForOne=true, …)`.
6. Pool executes the swap, transferring token1 to the attacker at the bad price.
7. `_afterSwap` → `_afterSwapOracleStopLoss` → `_checkAndUpdateWatermarks`:
   - `_decayed(H, 58, t)` → `0`
   - `_applyWatermark(metricT0_low, 0, 900_000)` → `(metricT0_low, false)`
   - `breach0 = false` → no revert
8. Swap settles. LP value is drained. Watermark is silently reset to `metricT0_low`.
9. The stop-loss extension now "protects" from the new, already-drained baseline — the loss is permanent.
