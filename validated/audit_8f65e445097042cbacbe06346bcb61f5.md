### Title
`AllResources` → `L1Gas` Misclassification in Protobuf Deserialization Silently Drops Valid Transactions from P2P Propagation — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` classifies a transaction's resource bounds as `ValidResourceBounds::L1Gas` whenever the wire-encoded `l2_gas` and `l1_data_gas` fields are both zero. This is the exact condition that holds for any `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` transaction submitted through the RPC gateway. The gateway accepts and hashes such transactions using the `AllResources` path, but every P2P peer (both consensus and mempool) that deserializes the same transaction reconstructs it as `L1Gas`, causing the subsequent `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion to fail with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The transaction is silently dropped by all peers and never sequenced.

### Finding Description

**Step 1 — Gateway accepts the transaction.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. The gateway's stateless validator explicitly allows transactions where only `l1_gas` is non-zero: [1](#0-0) 

The hash is computed via `InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3`, which always wraps the bounds as `ValidResourceBounds::AllResources(self.resource_bounds)`: [2](#0-1) 

`get_invoke_transaction_v3_hash` then includes the `L1_DATA_GAS` resource felt in the hash because the variant is `AllResources`: [3](#0-2) 

**Step 2 — Serialization to protobuf preserves all three gas fields.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources` with all three gas bounds present (even if zero): [4](#0-3) 

**Step 3 — Deserialization misclassifies the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` inspects the decoded values and, when both `l2_gas` and `l1_data_gas` are zero, returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`: [5](#0-4) 

**Step 4 — Conversion to `RpcInvokeTransactionV3` fails.**

The consensus deserialization path goes `protobuf::InvokeV3WithProof` → `InvokeTransactionV3` (now carrying `L1Gas`) → `RpcInvokeTransactionV3`. The last step explicitly rejects anything that is not `AllResources`: [6](#0-5) 

The error returned is `DEPRECATED_RESOURCE_BOUNDS_ERROR`: [7](#0-6) 

This failure propagates through both the consensus path: [8](#0-7) 

and the mempool path: [9](#0-8) 

**Step 5 — The codebase already acknowledges the problem in tests.**

The consensus protobuf test explicitly works around this by forcing `l2_gas.max_amount = 1` before serialization, with the comment:

> "If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant." [10](#0-9) 

The workaround exists only in the test; production code has no equivalent guard.

### Impact Explanation

Any user who submits a valid `InvokeV3` transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` through the RPC gateway will have that transaction accepted locally but silently dropped by every P2P peer (consensus and mempool). The transaction is never sequenced. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger condition — `l2_gas = 0` and `l1_data_gas = 0` — is a normal, explicitly tested configuration. The gateway's stateless validator accepts it. Any user who sets only `l1_gas` (a common pattern for pre-0.13.3 style transactions submitted through the new RPC) hits this path. No privileged access is required.

### Recommendation

Replace the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` usage in the consensus and mempool deserialization paths with the already-existing `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` converter: [11](#0-10) 

This converter unconditionally produces `AllResourceBounds` regardless of whether the gas values are zero, which is the correct semantic for any transaction arriving through the RPC/consensus pipeline. The `ValidResourceBounds::L1Gas` variant should only be produced when deserializing legacy pre-0.13.3 transactions from the state-sync / block-history path, not from the live transaction propagation path.

### Proof of Concept

1. Submit via RPC: `InvokeV3 { resource_bounds: AllResourceBounds { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }, ... }`
2. Gateway accepts; hash `H` is computed using the `AllResources` branch of `get_tip_resource_bounds_hash` (three resource felts hashed).
3. The proposer serializes the transaction into a `ConsensusTransaction` protobuf and broadcasts it.
4. Every receiving peer calls `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`.
5. Inside, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas=0, l1_data_gas=0` and returns `ValidResourceBounds::L1Gas`.
6. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
7. The transaction is dropped by all peers; it never enters any peer's mempool or block.

impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value
                .l1_data_gas
                .ok_or(missing("ResourceBounds::l1_data_gas"))?
                .try_into()?,
        })
