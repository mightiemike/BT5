Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual end-user swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is documented as gating swaps "by swapper address," but it checks `sender` — the direct caller of `MetricOmmPool.swap` — against the allowlist. When `MetricOmmSimpleRouter` intermediates the swap, `sender` is the router contract address, not the end-user. If a pool admin allowlists the router (a natural action to enable router-based swaps for permitted users), every unprivileged user can bypass the per-user restriction by routing through it.

## Finding Description

**Root cause — wrong actor checked:**

`MetricOmmPool.swap` passes `msg.sender` (the direct pool caller) as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that same `sender` verbatim into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][direct_pool_caller]`: [3](#0-2) 

**Router as direct pool caller:**

`MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router `msg.sender` inside the pool: [4](#0-3) 

The end-user's address (`msg.sender` of `exactInputSingle`) is stored only in transient callback context for payment purposes and is never forwarded to the pool as `sender`.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a set of approved addresses.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps for approved users (a natural, expected action).
3. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` targeting that pool.
4. The pool receives `msg.sender = router`; the extension checks `allowedSwapper[pool][router] == true` and passes — the unauthorized user's swap executes.

**Contrast with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly ignores `sender` (the direct caller) and checks `owner` — the economically relevant actor: [5](#0-4) 

The swap extension has no equivalent second parameter carrying the true initiator's identity.

## Impact Explanation
The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may swap in a pool. A bypass allows any unprivileged user to trade in a pool the admin intended to gate, breaking the core access-control invariant. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, or whitelist-only liquidity), this constitutes broken core pool functionality and an admin-boundary break reachable by an unprivileged router caller — matching the contest's allowed impact criteria.

## Likelihood Explanation
The precondition — the router being allowlisted — is a natural and expected admin action: any pool admin who wants approved users to access the pool via the standard periphery router must allowlist it. Once the router is allowlisted, the bypass is trivially reachable by any address with no special capability, capital, or timing requirement, and is fully repeatable.

## Recommendation
Pass the true initiator's address through the swap path. The simplest fix mirrors `DepositAllowlistExtension`: add a dedicated `swapper` parameter (analogous to `owner`) that the pool populates with the economically relevant actor. Alternatively, `SwapAllowlistExtension.beforeSwap` should check `recipient` or require the pool to forward the original `msg.sender` of the outermost call — but the cleanest solution is to align the swap hook signature with the deposit hook pattern so the checked identity is always the end-user, not the intermediary.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Admin allowlists the router (intended to enable router swaps for approved users)
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 3. Unprivileged user (not individually allowlisted) calls the router
vm.prank(unauthorizedUser);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    recipient: unauthorizedUser,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp,
    tokenIn: address(token1),
    extensionData: ""
}));
// Swap succeeds — allowlist bypassed because extension sees sender=router, not unauthorizedUser
```
