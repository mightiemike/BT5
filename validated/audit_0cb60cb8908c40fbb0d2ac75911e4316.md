### Title
ETH Permanently Stuck in `DirectDepositV1` When Constructor-Time Native Wrap Fails and Recovery Path Is Broken â€” (`core/contracts/DirectDepositV1.sol`)

---

### Summary

In `DirectDepositV1.sol`, the constructor intentionally does not revert when the native-token wrapping call fails, leaving ETH in the DDA. The only recovery path â€” `ContractOwner.withdrawFromDirectDepositV1()` â†’ `DirectDepositV1.withdrawNative()` â€” is permanently broken because `ContractOwner` has no `receive()` or `fallback()` function and cannot accept ETH, making any stuck ETH irrecoverable.

---

### Finding Description

**Root cause â€” constructor silent failure:**

In the `DirectDepositV1` constructor, if `wrappedNative.call{value: balance}("")` fails, the ETH stays in the DDA and only an event is emitted: [1](#0-0) 

The comment reads: *"shouldn't revert even if the transfer fails, otherwise the funds will be stuck in the DDA forever."* The comment acknowledges the ETH is stuck â€” it only avoids making the deployment itself revert. The ETH remains in the DDA's balance.

**Root cause â€” broken recovery path:**

The intended recovery is `ContractOwner.withdrawFromDirectDepositV1()` with `token == address(0)`: [2](#0-1) 

This calls `DirectDepositV1.withdrawNative()`: [3](#0-2) 

`withdrawNative()` is `onlyOwner`. The owner of `DirectDepositV1` is `ContractOwner`, set at deployment via `Ownable` (`msg.sender` during `new DirectDepositV1{salt: ...}(...)`): [4](#0-3) 

So `withdrawNative()` can only be called from `ContractOwner`. When called, `msg.sender` is `ContractOwner`. The line `msg.sender.call{value: balance}("")` tries to push ETH into `ContractOwner`. But `ContractOwner` has **no `receive()` or `fallback()` function** â€” a codebase-wide search confirms only `DirectDepositV1` and `MockWrappedNative` define `receive()`: [5](#0-4) 

The call fails, `withdrawNative()` reverts with `"Failed to transfer native token to owner"`, and the ETH is permanently locked in the DDA.

---

### Impact Explanation

Any ETH that ends up in a `DirectDepositV1` contract and fails to wrap at constructor time is permanently irrecoverable. The only designated recovery function (`withdrawFromDirectDepositV1` with `token == address(0)`) always reverts because `ContractOwner` cannot receive ETH. The DDA's native ETH balance is frozen with no escape path.

---

### Likelihood Explanation

Low. Requires an attacker to precompute the CREATE2 address of the DDA for a target subaccount (deterministic: deployer = `ContractOwner`, salt = `bytes32(uint256(1))`, init code includes constructor args), send ETH to that address before deployment, and trigger deployment via the public `creditDepositV1()`: [6](#0-5) 

Additionally, the wrapping call must fail. In normal operation `wrappedNative` is a WETH-like contract that accepts ETH, so the wrapping succeeds. However, if `wrappedNative` is temporarily non-functional, gas-limited, or the DDA is deployed on a chain where the wrap reverts, the ETH is stuck with no recovery.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`. Alternatively, refactor `withdrawFromDirectDepositV1` to avoid routing ETH through `ContractOwner` â€” for example, pass `msg.sender` as the recipient directly into a modified `withdrawNative(address recipient)` call, bypassing the intermediate ETH hold in `ContractOwner`.

---

### Proof of Concept

1. Compute the CREATE2 address of the DDA for subaccount `X` using `ContractOwner` as deployer, salt `bytes32(uint256(1))`, and the DDA bytecode hash (verifiable via `getDirectDepositV1BytecodeHash()`).
2. Send ETH to that precomputed address from any EOA.
3. Call `ContractOwner.creditDepositV1(X)` â€” public, no access control.
4. The DDA constructor runs, finds `address(this).balance != 0`, calls `wrappedNative.call{value: balance}("")`.
5. If the call fails: `NativeTokenTransferFailed` is emitted, ETH stays in the DDA.
6. Owner calls `ContractOwner.withdrawFromDirectDepositV1(X, address(0))`.
7. Internally: `DirectDepositV1.withdrawNative()` executes `ContractOwner.call{value: balance}("")`.
8. `ContractOwner` has no `receive()` â†’ EVM reverts â†’ `require(success, "Failed to transfer native token to owner")` fires.
9. ETH is permanently stuck in the DDA with no callable recovery path.
