### Title
Unrestricted Caller Access on `run_mainchain_gc` Allows Any Account to Prematurely Purge Canonical Chain State - (`contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function that permanently deletes verified block headers from the canonical chain. It carries only a `#[pause]` guard, which restricts calls only when the contract is paused. When the contract is operating normally (unpaused), any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, immediately purging all GC-eligible mainchain headers. This breaks the SPV verification guarantee for any transaction whose block is deleted, corrupting the core security invariant of the light client.

### Finding Description
`run_mainchain_gc` is decorated with `#[pause(except(roles(Role::UnrestrictedRunGC)))]` and nothing else. [1](#0-0) 

The `#[pause]` macro only gates calls when the contract is paused. When unpaused — the normal production state — there is no caller check whatsoever. Any NEAR account can invoke this function directly. The function then permanently removes mainchain block headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and advances `mainchain_initial_blockhash` forward: [2](#0-1) 

The intended design is that GC runs automatically inside `submit_blocks` (called by the trusted relayer) with `batch_size` equal to the number of submitted headers, providing a bounded, relayer-controlled deletion rate. The public `run_mainchain_gc` entry point was intended as a manual maintenance call, but it has no authorization guard to enforce that intent. [3](#0-2) 

This is confirmed by the integration test, where `user_account` — which holds only `UnrestrictedSubmitBlocks`, not `UnrestrictedRunGC` — successfully calls `run_mainchain_gc` directly: [4](#0-3) 

### Impact Explanation
An attacker calls `run_mainchain_gc(u64::MAX)` at any time. The contract removes every mainchain header that is currently beyond `gc_threshold`, advancing `mainchain_initial_blockhash` to the new oldest block. Any subsequent call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a block that was just deleted panics with `"block does not belong to the current main chain"`: [5](#0-4) 

For any bridge, atomic swap, or cross-chain lending protocol that relies on this light client to confirm a Bitcoin transaction, the attacker can invalidate a pending SPV proof at will — after the transaction has been broadcast and the counterparty has acted on it — by purging the block before the consumer contract calls `verify_transaction_inclusion`. The deleted state is irrecoverable without a full re-submission of the purged headers by the relayer.

### Likelihood Explanation
The entry point is fully public and requires no stake, deposit, or role. Any NEAR account can call it in a single transaction. The attacker only needs to know that the chain has grown beyond `gc_threshold` (observable via `get_mainchain_size` and `get_last_block_height`, both public view functions). The attack is cheap, deterministic, and repeatable.

### Recommendation
Add a caller authorization check to `run_mainchain_gc` that mirrors the pattern used on `submit_blocks`. The simplest fix is to restrict direct external calls to accounts holding `Role::DAO`, `Role::RelayerManager`, or a new dedicated `Role::GCManager`, while keeping the internal call path from `submit_blocks` unrestricted:

```rust
// Option A: restrict to a role
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id()),
        "Unauthorized"
    );
    // ...
}

// Option B: make the public entry point private and expose a separate
// authorized wrapper, keeping the internal call from submit_blocks unchanged.
```

Alternatively, remove the public `run_mainchain_gc` entry point entirely and rely solely on the automatic GC triggered inside `submit_blocks`.

### Proof of Concept
```rust
// Any unprivileged NEAR account can execute this:
let attacker = sandbox.dev_create_account().await?; // no roles granted

// Observe current chain size (public view)
let size_before: u64 = attacker
    .view(contract.id(), "get_mainchain_size")
    .args_json(json!({}))
    .await?.json()?;

// Purge all GC-eligible headers in one call
let outcome = attacker
    .call(contract.id(), "run_mainchain_gc")
    .args_json(json!({"batch_size": u64::MAX}))
    .max_gas()
    .transact()
    .await?;
assert!(outcome.is_success()); // succeeds with no role

// verify_transaction_inclusion now panics for any purged block
let proof_result = attacker
    .call(contract.id(), "verify_transaction_inclusion")
    .args_borsh(ProofArgs { tx_block_blockhash: purged_block_hash, ... })
    .transact()
    .await?;
assert!(!proof_result.is_success()); // "block does not belong to the current main chain"
```
