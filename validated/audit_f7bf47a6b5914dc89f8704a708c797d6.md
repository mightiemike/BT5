### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Caller to Drain usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` performs a raw `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. The function has no access control beyond a chain ID check, meaning any unprivileged caller on Ink Chain mainnet can invoke it. If the USDC token at the hardcoded address returns `false` on failure rather than reverting, the caller receives the full `usdcE` balance held by the target `DirectDepositV1` contract without providing any USDC in exchange.

---

### Finding Description

`replaceUsdcEWithUsdc` is designed as a migration helper: it pulls USDC from the caller into a `DirectDepositV1` contract, then withdraws the equivalent `usdcE` balance back to the caller. The critical flaw is at line 616:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value of this call is silently discarded. The protocol's own `ERC20Helper.safeTransferFrom` wraps the call in a low-level `call` and asserts `success && (data.length == 0 || abi.decode(data, (bool)))`, but this function bypasses that wrapper entirely and calls `transferFrom` directly on the interface. [1](#0-0) 

The subsequent two operations — `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` and `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — execute unconditionally regardless of whether the USDC pull succeeded. [2](#0-1) 

The function carries no `onlyOwner` or `onlyDeployer` modifier; the sole gate is `require(block.chainid == 57073, ERR_UNAUTHORIZED)`, making it callable by any EOA or contract on Ink Chain mainnet. [3](#0-2) 

`DirectDepositV1.withdraw` is `onlyOwner`, and `ContractOwner` is the deployer/owner of every `DirectDepositV1` instance, so `ContractOwner` can always invoke it on behalf of any caller. [4](#0-3) 

The rest of the codebase consistently uses `ERC20Helper.safeTransfer` / `safeTransferFrom`, which perform the proper low-level call with return-value assertion. This one site is the sole deviation. [5](#0-4) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink Chain returns `false` on a failed `transferFrom` (e.g., insufficient allowance or balance) rather than reverting, the attacker:

1. Pays zero USDC.
2. Receives the full `usdcE` balance held by the targeted `DirectDepositV1` contract.

The `DirectDepositV1` contract loses its `usdcE` collateral, which was deposited by a legitimate user and is pending credit into their subaccount. The user's subaccount never receives the deposit credit, and the funds are permanently stolen. Every `DirectDepositV1` holding a non-zero `usdcE` balance is independently drainable in a single transaction.

---

### Likelihood Explanation

The function is permissionless on Ink Chain mainnet. The only precondition is that a `DirectDepositV1` for the target subaccount exists and holds a non-zero `usdcE` balance — a normal operational state during the migration window this function was designed for. Whether the specific USDC deployment at the hardcoded address returns `false` or reverts on failure determines exploitability; bridged or wrapped USDC variants on L2s do not always match Circle's canonical revert-on-failure behavior. The missing check is a latent vulnerability that activates the moment a non-reverting USDC variant is in use.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own `ERC20Helper.safeTransferFrom`, which is already imported and used throughout `ContractOwner.sol` via `using ERC20Helper for IERC20Base`:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This ensures the transaction reverts if the USDC pull fails for any reason, regardless of whether the token reverts or returns `false`. [6](#0-5) 

---

### Proof of Concept

1. A legitimate user sends `usdcE` to their `DirectDepositV1` address. The DDA now holds `balance > 0` usdcE.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, balance)` returns `false` (no revert) because attacker has no allowance.
4. Return value is ignored; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers all `usdcE` from the DDA to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, balance)` transfers all `usdcE` to the attacker.
7. Attacker holds `balance` usdcE. Victim's DDA is empty. Victim's subaccount receives no deposit credit. [1](#0-0)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
