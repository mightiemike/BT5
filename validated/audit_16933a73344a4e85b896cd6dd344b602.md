Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing unlisted callers to bypass the deposit allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller who provides tokens) and instead validates `owner` (the LP-position recipient) against the allowlist. Because `owner` is a free caller-supplied argument, any address not on the allowlist can bypass the gate by setting `owner` to any allowlisted address, depositing into a pool that is supposed to be restricted.

## Finding Description

**Call chain:**

1. Pool calls `ExtensionCalling._beforeAddLiquidity(sender, owner, ...)` where `sender = msg.sender` of the `addLiquidity` call (the actual token depositor) and `owner` is a caller-supplied LP-position recipient. [1](#0-0) 

2. This encodes and dispatches `IMetricOmmExtensions.beforeAddLiquidity(sender, owner, ...)` to the extension. [2](#0-1) 

3. `DepositAllowlistExtension.beforeAddLiquidity` receives both arguments but **ignores `sender`** (first parameter is unnamed) and only checks `owner`: [3](#0-2) 

   ```solidity
   function beforeAddLiquidity(address, address owner, ...)  // sender silently dropped
       ...
   {
       if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
           revert ...NotAllowedToDeposit();
       }
   ```

**Root cause:** The contract's own NatSpec says it "Gates `addLiquidity` by depositor address", but the depositor is `sender`, not `owner`. `owner` is a free parameter the caller can set to any address. [4](#0-3) 

**Exploit path:**
- Pool has `DepositAllowlistExtension` configured; `allowedDepositor[pool][alice] = true`, attacker `bob` is not listed.
- `bob` calls `pool.addLiquidity(owner=alice, ...)`.
- Pool calls `extension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
- Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
- `bob`'s tokens enter the pool; LP position is minted to `alice`.
- The allowlist gate is fully bypassed; `bob` deposited without authorization.

**Existing guards reviewed:** The `allowedDepositor` and `allowAllDepositors` mappings are correctly keyed by pool (`msg.sender` inside the extension = the pool). The only flaw is that the wrong actor address (`owner` vs `sender`) is looked up. [5](#0-4) 

## Impact Explanation
The `DepositAllowlistExtension` is a core access-control primitive. Its bypass allows any unprivileged address to deposit into a restricted pool, breaking the allowlist invariant entirely. Depending on pool configuration this can result in unauthorized liquidity provision, pool composition manipulation, or circumvention of KYC/compliance gating — all constituting broken core pool functionality caused by an unprivileged actor. Severity: **Medium** (broken core access-control functionality reachable by any unprivileged caller; direct fund loss to the attacker is self-inflicted, but the pool's intended restriction is nullified).

## Likelihood Explanation
Exploitable by any address with no special privileges. The only precondition is knowing one allowlisted address (trivially discoverable on-chain via `AllowedToDepositSet` events or direct mapping reads). Fully repeatable with no time constraints.

## Recommendation
Replace the `owner` check with a `sender` check in `beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns with the stated invariant ("Gates `addLiquidity` by depositor address") and matches the `sender` semantics used throughout the extension framework.
