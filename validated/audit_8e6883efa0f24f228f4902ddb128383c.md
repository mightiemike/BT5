### Title
Uncleared Allowance in `creditDeposit()` Causes Permanent DoS for USDT-like Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. `Endpoint.depositCollateralWithReferral()` contains an explicit early-return path that skips the `transferFrom` when the deposit amount is below the minimum threshold — leaving a non-zero allowance unconsumed. For any ERC20 token that reverts on a non-zero→non-zero `approve` (e.g., USDT), a subsequent call to `creditDeposit()` will permanently revert, bricking the DDA's deposit flow.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product tokens and, for each non-zero balance, calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The critical flaw is that `Endpoint.depositCollateralWithReferral()` contains an explicit early-return path:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    return;
}
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` is never called, so the allowance set by `token.approve(...)` is **never consumed**. The allowance remains non-zero.

On the next call to `creditDeposit()`, for any token that reverts when `approve` is called with a non-zero current allowance (USDT-like behavior), the `token.approve(address(endpoint), balance)` call reverts — causing the entire loop and the entire `creditDeposit()` call to revert.

By contrast, `ContractOwner.wrapVaultAsset()` correctly resets the allowance to zero before setting a new one:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

This inconsistency confirms the developers are aware of the pattern but did not apply it in `DirectDepositV1`.

---

### Impact Explanation

An attacker can permanently disable `creditDeposit()` for any DDA associated with a victim subaccount:

1. Send a dust amount of a USDT-like spot token to the victim's DDA (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`).
2. Call `creditDeposit()` — the approve fires, but `depositCollateralWithReferral` returns early, leaving a non-zero allowance.
3. Any subsequent call to `creditDeposit()` reverts at the `token.approve(...)` line for that token.
4. Because the loop iterates all product IDs and reverts on the first failure, **the entire `creditDeposit()` function is bricked** — not just for the attacked token.

Funds accumulating in the DDA (including legitimate large deposits) cannot be credited to the subaccount via the normal flow. The owner can still call `withdraw()` to rescue tokens, but the deposit mechanism is permanently disrupted until the allowance is manually cleared — which is not possible from within the current contract. [4](#0-3) 

---

### Likelihood Explanation

- `creditDeposit()` is `external` with no access control — any unprivileged caller can trigger it.
- The early-return path in `depositCollateralWithReferral` is explicitly designed to handle sub-minimum deposits, making the leftover-allowance condition reachable by design.
- An attacker only needs to send a dust amount of a USDT-like token to the DDA and call `creditDeposit()` once.
- If any USDT-like token (reverting on non-zero→non-zero approve) is listed as a spot product on Ink Chain, this is directly exploitable.

---

### Recommendation

Reset the allowance to zero before setting a new one in `creditDeposit()`, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), 0);      // reset first
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
}
``` [1](#0-0) 

---

### Proof of Concept

1. A USDT-like token `T` is listed as a spot product (productId `P`) in `SpotEngine`.
2. Attacker sends 1 wei of `T` to victim's DDA address (below `MIN_DEPOSIT_AMOUNT`).
3. Attacker calls `DirectDepositV1(dda).creditDeposit()`:
   - `balance = T.balanceOf(dda) = 1`
   - `T.approve(endpoint, 1)` — succeeds, allowance = 1
   - `endpoint.depositCollateralWithReferral(subaccount, P, 1, "-1")` — hits early return, no `transferFrom`, allowance stays at 1
4. Victim's user sends 10,000 `T` to the DDA (legitimate deposit).
5. Anyone calls `creditDeposit()` again:
   - `balance = 10001`
   - `T.approve(endpoint, 10001)` — **REVERTS** (current allowance = 1, non-zero)
   - Entire `creditDeposit()` reverts; 10,000 `T` is stuck in the DDA, uncreditable. [5](#0-4) [4](#0-3)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
