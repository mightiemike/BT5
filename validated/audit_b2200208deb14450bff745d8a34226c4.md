### Title
SwapAllowlistExtension gates on the router address instead of the actual user, allowing allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument, which equals `msg.sender` of the pool's `swap()` call. When users route through `MetricOmmSimpleRouter`, that value is the router's address, not the user's. A pool admin who allowlists the router to let approved users access the pool via the router inadvertently opens the pool to every caller, because any unprivileged user can route through the same public router and pass the check.

### Finding Description
`SwapAllowlistExtension.beforeSwap()` enforces:

```solidity
// SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by `ExtensionCalling._beforeSwap()`, which is the `msg.sender` of the pool's `swap()` call. [1](#0-0) 

The pool's `swap()` passes `msg.sender` as `sender` to the extension dispatcher: [2](#0-1) 

`ExtensionCalling._beforeSwap()` forwards it verbatim: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` with `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`. [4](#0-3) 

If the pool admin allowlists the router (a natural step to let approved users access the pool via the standard periphery), then **any** unprivileged user can bypass the per-address allowlist by routing through the public router contract. The extension has no mechanism to recover the original `tx.origin` or the user address embedded in the router's transient callback context.

Contrast this with `DepositAllowlistExtension.beforeAddLiquidity()`, which correctly gates on `owner` (the economic actor) rather than `sender` (the immediate caller): [5](#0-4) 

The asymmetry confirms the swap-side check is bound to the wrong identity.

### Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (e.g., a private market-maker pool or a KYC-gated pool) is fully open to any caller the moment the pool admin allowlists the router. The allowlist invariant — that only approved addresses may execute swaps — is silently broken. Unauthorized users gain access to oracle-priced liquidity on terms the pool admin never intended to offer, which can result in LP value extraction and breaks the core access-control guarantee of the extension. This constitutes a broken core pool functionality with Medium-severity fund impact.

### Likelihood Explanation
Medium. The pool admin must allowlist the router for the bypass to be reachable. However, this is a natural and expected operational step: approved users typically interact through the router rather than calling the pool directly. The `SwapAllowlistExtension` provides no documentation warning that allowlisting the router collapses the per-user gate to an all-or-nothing gate, making the misconfiguration likely in practice.

### Recommendation
The extension should gate on the actual economic actor, not the immediate pool caller. Two viable approaches:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a coordinated convention between router and extension.
2. **Check `tx.origin` as a fallback**: When `sender` is a known router, fall back to `tx.origin`. This is imperfect but closes the router-mediated bypass for EOA callers.

The `DepositAllowlistExtension` pattern (gating on `owner`, the explicit economic actor) should be the model: the swap extension should receive and check the true initiating user address rather than the immediate `msg.sender` of the pool call.

### Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is approved
  admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack:
  charlie (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      zeroForOne: true,
      amountIn: X,
      amountOutMinimum: 0,
      ...
    })

  → router calls pool.swap(recipient, true, X, limit, "", "")
      msg.sender to pool = router
  → pool calls extension.beforeSwap(router, ...)
      check: allowedSwapper[pool][router] == true  ✓
  → swap executes for charlie despite charlie not being allowlisted
```
