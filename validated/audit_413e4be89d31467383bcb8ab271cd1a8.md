### Title
Withdraw Fee Charged Before Balance Deduction Causes Full-Balance Withdrawal to Revert — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, for both `WithdrawCollateral` and `WithdrawCollateralV2` transaction types, the sequencer fee (`withdrawFeeX18`) is deducted from the sender's virtual product balance **before** `clearinghouse.withdrawCollateral` is called. Because `withdrawCollateral` then deducts the full user-specified `amount` from the same balance and enforces a health check, any user attempting to withdraw their entire balance will have the transaction revert — the fee deduction leaves the balance insufficient to absorb both the fee and the full withdrawal.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` branch executes:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId          // fee charged in the product being withdrawn
);
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,            // full user-specified amount, not fee-adjusted
    address(0),
    nSubmissions
);
``` [1](#0-0) 

`chargeFee` calls `spotEngine.updateBalance(productId, sender, -fee)`, reducing the sender's X18 balance by `withdrawFeeX18` in the **same product token** as the withdrawal. [2](#0-1) 

`clearinghouse.withdrawCollateral` then computes `amountRealized = -int128(amount) * int128(multiplier)` and calls `spotEngine.updateBalance(productId, sender, amountRealized)`, deducting the full withdrawal amount from the already-reduced balance. The function then enforces:

```solidity
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [3](#0-2) 

If the user's pre-transaction balance equals exactly `amount_x18`, the sequence is:

| Step | Balance (X18) |
|---|---|
| Before fee | `amount_x18` |
| After `chargeFee` | `amount_x18 − fee_x18` |
| After `withdrawCollateral` deduction | `−fee_x18` |
| Health check | **FAIL** (negative balance = liability) |

The identical pattern exists for `WithdrawCollateralV2`: [4](#0-3) 

---

### Impact Explanation

Any user who attempts to withdraw their entire balance of a product will have the transaction revert with `ERR_SUBACCT_HEALTH`. The actual token transfer (`handleWithdrawTransfer`) executes before the health check, meaning the tokens leave the clearinghouse but the virtual balance update and health check revert the entire transaction — the user's withdrawal is blocked. The user cannot withdraw their full balance without knowing to manually subtract the fee from the withdrawal amount, which is not enforced or communicated by the contract interface.

---

### Likelihood Explanation

High. This is triggered by any unprivileged user submitting a standard `WithdrawCollateral` or `WithdrawCollateralV2` transaction for their full product balance — a routine and expected user action. No special conditions, admin access, or adversarial setup is required.

---

### Recommendation

Charge the fee **after** `clearinghouse.withdrawCollateral` completes (so the health check already reflects the withdrawal), or adjust the withdrawal amount inside `withdrawCollateral` to be `amount − fee_adjusted` so the user receives the net amount and the balance settles to zero. This mirrors the resolution applied in BRT-1: account for the fee in the amount calculation rather than deducting it as a prior independent step.

---

### Proof of Concept

1. User holds 100 USDC (virtual balance: `100e18` in X18).
2. User signs and submits a `WithdrawCollateral` transaction for `amount = 100e6` (100 USDC in native decimals).
3. `chargeFee` deducts `withdrawFeeX18 = 1e18` (1 USDC) → virtual balance: `99e18`.
4. `clearinghouse.withdrawCollateral` converts `100e6` → `100e18` (X18) and deducts it → virtual balance: `−1e18`.
5. `getHealth(sender, INITIAL)` returns a negative value (user now has a net liability of 1 USDC with no offsetting assets).
6. `require(getHealth(...) >= 0, ERR_SUBACCT_HEALTH)` reverts.
7. The user's full-balance withdrawal is permanently blocked unless they re-submit for `amount = 99e6` or less.
