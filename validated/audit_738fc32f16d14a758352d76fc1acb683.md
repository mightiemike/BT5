Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the end user, allowing any caller to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the router is allowlisted â€” the only way to let any allowlisted user use the router â€” every unprivileged caller can bypass the curated pool's swap restriction by routing through the router. This fully defeats the access-control guarantee of any pool deploying `SwapAllowlistExtension`.

## Finding Description

**Step 1 â€” Pool passes `msg.sender` as `sender` to `_beforeSwap`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 231. When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router contract address. [1](#0-0) 

**Step 2 â€” `ExtensionCalling._beforeSwap` forwards `sender` verbatim to every extension:**

The value is passed unchanged into `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`. [2](#0-1) 

**Step 3 â€” `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`:**

`msg.sender` inside the extension is the pool; `sender` is whatever the pool forwarded â€” the router address when routing through `MetricOmmSimpleRouter`. The check `allowedSwapper[msg.sender][sender]` therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [3](#0-2) 

**Step 4 â€” `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly with no user identity propagation:**

The router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` with an empty string `""` as `extensionData`, so no caller identity is encoded for the extension to decode. [4](#0-3) 

**Root cause â€” irreconcilable dilemma for pool admins:**

| Admin choice | Consequence |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router â€” broken core functionality |
| **Allowlist the router** | Every unprivileged caller bypasses the allowlist via the router |

Once the router is allowlisted (the only viable operational choice), any address can call `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and the extension passes because `allowedSwapper[pool][router] == true`. The same flaw applies to `exactInput` (L103-112) and `exactOutputSingle` (L135-137). [5](#0-4) [6](#0-5) 

No existing guard compensates: `allowAllSwappers` is a separate opt-out that pool admins of curated pools would not set, and there is no mechanism in the pool or router to propagate the original caller's identity to the extension. [7](#0-6) 

## Impact Explanation

A curated pool (institutional-only, KYC-gated, or whitelist-restricted) that relies on `SwapAllowlistExtension` to enforce access control is fully bypassed for any caller who routes through `MetricOmmSimpleRouter`. The unauthorized swapper receives pool output tokens and the pool receives input tokens at oracle-anchored prices â€” a direct, real-money swap that the pool admin intended to block. LP composition and fee revenue are affected by trades that should never have executed. This is a **High** severity policy-bypass with direct fund impact: unauthorized parties extract value from a pool designed to exclude them.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public entry point for swaps in the periphery layer. Any user who discovers that a pool uses `SwapAllowlistExtension` can trivially route through the router with a single `exactInputSingle` call. No privileged access, no special setup, and no malicious token is required. The bypass is repeatable indefinitely. Likelihood is **High**.

## Recommendation

The extension must gate the economically relevant actor â€” the end user â€” not the intermediary contract. Two sound approaches:

1. **Pass the original caller through `extensionData`**: The router encodes `msg.sender` into `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes and checks it. This requires a convention between router and extension but no core changes.

2. **Explicit `swapper` parameter in `pool.swap`**: Modify `MetricOmmPool.swap` to accept an explicit `swapper` address (separate from `msg.sender`) that the router populates with the original caller, and forward that to extensions. This mirrors how `DepositAllowlistExtension` correctly gates `owner` (the position beneficiary), which is caller-supplied and pool-enforced. [8](#0-7) 

## Proof of Concept

```solidity
// Pool deployed with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin must allowlist the router so allowedUser can use it.
swapAllowlist.setAllowedToSwap(pool, address(router), true);  // router allowlisted

// Attacker (not allowlisted) calls the router directly.
router.exactInputSingle(ExactInputSingleParams({
    pool:             pool,
    recipient:        attacker,
    zeroForOne:       true,
    amountIn:         1e18,
    amountOutMinimum: 0,
    priceLimitX64:    0,
    deadline:         block.timestamp,
    tokenIn:          token0,
    extensionData:    ""
}));
// Router calls pool.swap(attacker, true, ..., "", "")
// Pool calls _beforeSwap(msg.sender=router, ...)
// Extension checks allowedSwapper[pool][router] == true â†’ passes
// Attacker receives token1 output from the curated pool.
// allowedSwapper[pool][attacker] was never set.
```
