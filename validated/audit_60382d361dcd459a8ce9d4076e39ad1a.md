### Title
Unchecked `approve` Return Value in `DirectDepositV1.creditDeposit()` Silently Bypasses Allowance Gate for Non-Standard Tokens — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` directly through the `IIERC20Base` interface without checking the boolean return value. The same contract defines a `safeTransfer` helper that correctly validates ERC20 return values via low-level call, but no equivalent `safeApprove` exists. For any non-standard token that returns `false` from `approve` without reverting, the allowance is never set, yet execution continues unconditionally into `endpoint.depositCollateralWithReferral`. This is the direct Nado analog to the external report's concern: unchecked return values from ERC20 external calls, with inconsistent use of safe wrappers creating a silent failure path.

---

### Finding Description

`creditDeposit()` is an unrestricted `external` function — no `onlyOwner` or similar modifier — callable by any address: [1](#0-0) 

At line 92, `token.approve(address(endpoint), balance)` is called through the typed interface. The `bool` return value is silently discarded: [2](#0-1) 

The same contract defines `safeTransfer` using a low-level `.call` and explicitly validates `success && (data.length == 0 || abi.decode(data, (bool)))`: [3](#0-2) 

No equivalent `safeApprove` exists. The `ERC20Helper` library used elsewhere in the protocol follows the same safe pattern for `transfer` and `transferFrom` but provides no `approve` wrapper: [4](#0-3) 

The same unchecked `approve` pattern appears in `ContractOwner.depositInsurance()` and `ContractOwner.wrapVaultAsset()`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Scenario A — Non-standard token returns `false` from `approve`, reverts from `transferFrom` on zero allowance (realistic):**

1. `creditDeposit()` iterates all `spotEngine.getProductIds()`.
2. For a non-standard token, `approve` returns `false`; the return value is discarded; allowance remains 0.
3. `endpoint.depositCollateralWithReferral` is called; internally `handleDepositTransfer` calls `ERC20Helper.safeTransferFrom`.
4. `safeTransferFrom` uses a low-level call; `transferFrom` reverts (zero allowance) → `success = false` → `safeTransferFrom` reverts.
5. The entire `creditDeposit` loop reverts. **All tokens held in the DDA for that subaccount cannot be deposited**, locking user funds in the DDA contract until the owner manually calls `withdraw`.

**Scenario B — Non-standard token returns `false` from `approve` and `true` from `transferFrom` regardless of allowance (less common but possible for certain fee-on-transfer or permissioned tokens):**

1. `approve` silently fails; allowance stays 0.
2. `transferFrom` returns `true` anyway; `safeTransferFrom` passes.
3. The deposit is credited to the subaccount on-chain without actual token movement. **Collateral accounting is corrupted**: the subaccount holds a positive balance backed by no real tokens.

The corrupted state delta in Scenario B is the subaccount's spot balance in `SpotEngine` being incremented without a corresponding token transfer into the clearinghouse.

---

### Likelihood Explanation

`creditDeposit()` has no access control and is callable by any address, including through `ContractOwner.creditDepositV1()`: [7](#0-6) 

Scenario A is realistic for any token that follows the "return false instead of revert" pattern (a known class of non-conforming ERC20s). Scenario B requires a more unusual token but is not theoretical — tokens with custom allowance logic exist. The protocol explicitly supports arbitrary spot tokens added via governance, increasing the surface area for non-standard token behavior.

---

### Recommendation

- **Short term:** Replace the direct `token.approve(...)` call in `creditDeposit()` with a low-level safe wrapper analogous to `safeTransfer`, checking `success && (data.length == 0 || abi.decode(data, (bool)))`. Apply the same fix to `ContractOwner.depositInsurance()` and `ContractOwner.wrapVaultAsset()`.
- **Long term:** Add a `safeApprove` function to `ERC20Helper` and use it consistently everywhere `approve` is called on an external token address, mirroring the existing `safeTransfer` / `safeTransferFrom` pattern already present in the library.

### Proof of Concept

```
Attacker/user triggers: ContractOwner.creditDepositV1(subaccount)
  → DirectDepositV1.creditDeposit()
      loop over productIds:
        token = IIERC20Base(nonStandardTokenAddr)
        balance = token.balanceOf(address(this))  // e.g. 1000e6
        token.approve(address(endpoint), balance)  // returns false, NOT checked
        endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")
          → handleDepositTransfer(token, msg.sender=DDA, amount=1000e6)
              → ERC20Helper.safeTransferFrom(token, DDA, clearinghouse, 1000e6)
                  low-level call to transferFrom:
                    [Scenario A] reverts → safeTransferFrom reverts → creditDeposit reverts
                                 → 1000e6 tokens stuck in DDA, no deposit credited
                    [Scenario B] returns true (ignores allowance) → deposit credited
                                 → subaccount balance += 1000e6 with no real token transfer
```

Root cause: `core/contracts/DirectDepositV1.sol` line 92 — `token.approve(address(endpoint), balance)` return value silently discarded, no `safeApprove` wrapper exists. [8](#0-7)

library ERC20Helper {
    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
