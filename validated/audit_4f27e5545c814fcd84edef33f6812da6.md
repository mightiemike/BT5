### Title
`verify_transaction_inclusion` Returns `true` for Internal Merkle Node Hashes Without Leaf-Node Validation — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated-but-still-callable `verify_transaction_inclusion` function does not validate that the supplied `tx_id` is an actual leaf-node transaction hash rather than an internal Merkle tree node. Any unprivileged NEAR caller can supply an internal-node hash as `tx_id` together with a valid Merkle proof for that node, causing the function to return `true` for a transaction that does not exist. This is the direct analog of `previewWithdraw` not accounting for withdrawal fees: a public verification interface returns a result that omits a critical factor, misleading every integrator that consumes it.

---

### Finding Description

`verify_transaction_inclusion` computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block header's `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The function performs no check that `tx_id` is a leaf node. In Bitcoin's Merkle tree every internal node is itself a 32-byte double-SHA256 hash, indistinguishable in format from a transaction ID. An attacker who knows the Merkle tree of any block can pick any internal node `N` at depth `d`, construct a valid `merkle_proof` of length `depth - d` that walks from `N` up to the root, and call `verify_transaction_inclusion` with `tx_id = N`. `compute_root_from_merkle_proof` will faithfully reproduce the root and the function returns `true`, asserting that a non-existent transaction is confirmed in that block. [2](#0-1) 

The function is annotated `#[deprecated]` and carries a docstring warning, but `#[deprecated]` in Rust is a compile-time lint for Rust callers only; it has no effect on NEAR cross-contract calls. The function remains fully reachable by any NEAR account. [3](#0-2) 

`verify_transaction_inclusion_v2` mitigates the attack by first validating a coinbase Merkle proof, then delegating to `verify_transaction_inclusion`. However, `verify_transaction_inclusion` is still independently callable, bypassing the coinbase guard entirely. [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged funds, minting wrapped tokens, settling a cross-chain swap) can be deceived into accepting a forged proof. The attacker does not need to control any privileged role; they only need knowledge of the Merkle tree of any block already stored in the light client, which is public Bitcoin data. The corrupted value is the proof result: `true` is returned for a transaction that was never broadcast or mined.

---

### Likelihood Explanation

The function is publicly callable on every deployed instance of the contract. Any integrator that has not yet migrated to `verify_transaction_inclusion_v2` is immediately exploitable. The required inputs (block Merkle tree, internal node hash, sibling path) are all derivable from public Bitcoin RPC data with no special access. Likelihood is high for any integration that still uses the v1 endpoint.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or gate it so it panics unconditionally (forcing callers to use v2). If removal is not immediately feasible, add an explicit leaf-node guard: require that the `merkle_proof` length equals the full tree depth and that `tx_index` is within the leaf range, or require a coinbase sibling proof identical to v2's approach.

---

### Proof of Concept

1. Identify any block stored in the light client (e.g., block at height `H` with known Merkle tree).
2. Select any internal node `N` at depth `d` in that block's Merkle tree (public data from Bitcoin RPC `getblock`).
3. Construct `merkle_proof`: the `depth - d` sibling hashes walking from `N` to the root.
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash)
   - `tx_block_blockhash` = the block hash at height `H`
   - `tx_index` = the leaf-range index corresponding to `N`'s subtree position
   - `merkle_proof` = the sibling path constructed above
   - `confirmations = 1`
5. `compute_root_from_merkle_proof(N, tx_index, proof)` reproduces the stored `merkle_root`; the function returns `true`, falsely asserting that `N` is a confirmed transaction in that block. [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
