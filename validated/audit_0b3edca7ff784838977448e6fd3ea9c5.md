Audit Report

## Title
`PriceVelocityGuardExtension` skips velocity check on every pool's first swap due to uninitialized `lastMidPriceX64` — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension` does not override `initialize()`, so the factory's initialization call is a no-op and `priceVelocityState[pool].lastMidPriceX64` remains `0` at pool creation. In `beforeSwap`, the entire velocity enforcement block is gated on `prevMid != 0`, so the check is unconditionally skipped on the first swap regardless of how large the oracle price deviation is. Any pool relying on this extension for oracle-manipulation protection is fully unprotected on its opening trade.

## Finding Description
`BaseMetricExtension.initialize()` is a virtual no-op: [1](#0-0) 

`PriceVelocityGuardExtension` never overrides it, so the factory's loop: [2](#0-1) 

…writes nothing to `priceVelocityState[pool]`. Both `lastMidPriceX64` and `maxChangePerBlockE18` remain `0`.

In `beforeSwap`, the guard reads `prevMid` and immediately writes the new mid price, then gates the entire enforcement block: [3](#0-2) 

On the first swap `prevMid == 0`, so `if (prevMid != 0)` is `false` and `PriceVelocityExceeded` can never fire. After that first swap `lastMidPriceX64` is set to the current oracle mid, and all subsequent swaps are checked normally (provided `maxChangePerBlockE18` has been set by the admin).

The admin-callable `setLastMidPrice()` exists but is not invoked by the factory and is not enforced before the first swap: [4](#0-3) 

By contrast, `OracleValueStopLossExtension` correctly overrides `initialize()` to atomically populate its per-pool config at factory time: [5](#0-4) 

## Impact Explanation
The velocity guard's sole purpose is to block swaps when the oracle mid price has moved faster than `maxChangePerBlockE18 * sqrt(1 + blockDiff)`. On the first swap that protection is entirely absent. A trader who can influence the oracle price (or who observes a large oracle move and races to be first) can execute a swap at an arbitrarily large price deviation without triggering `PriceVelocityExceeded`. The pool executes the swap at the manipulated oracle bid/ask, giving the attacker more output tokens than the legitimate price permits — a direct loss of LP principal. This constitutes bad-price execution and swap conservation failure, both within the allowed impact gate.

## Likelihood Explanation
Every pool that deploys `PriceVelocityGuardExtension` and has `maxChangePerBlockE18` set (via `setMaxChangePerBlock`) is affected on its very first swap. The window is exactly one transaction but is deterministic and requires no special privilege — any public caller can be the first swapper. The admin can close the window by calling `setLastMidPrice()` before the first swap, but the protocol does not enforce this and the `initialize()` path provides no mechanism to do so atomically at deployment.

## Recommendation
Override `initialize()` in `PriceVelocityGuardExtension` to decode `extensionInitData` and seed both `lastMidPriceX64` and `maxChangePerBlockE18` atomically at pool creation, mirroring the pattern used by `OracleValueStopLossExtension`:

```solidity
function initialize(address pool, bytes calldata data)
  external
  override
  onlyFactory
  returns (bytes4)
{
  (uint128 initialMidPriceX64, uint64 maxChangePerBlockE18_) = abi.decode(data, (uint128, uint64));
  PriceVelocityState storage s = priceVelocityState[pool];
  s.lastMidPriceX64      = initialMidPriceX64;
  s.lastUpdateBlock      = uint64(block.number);
  s.maxChangePerBlockE18 = maxChangePerBlockE18_;
  emit LastMidPriceUpdated(pool, initialMidPriceX64);
  emit MaxChangePerBlockSet(pool, maxChangePerBlockE18_);
  return IMetricOmmExtensions.initialize.selector;
}
```

This ensures the guard is active from the very first swap, exactly as the pool creator intends when they configure the extension.

## Proof of Concept
1. Deploy a pool with `PriceVelocityGuardExtension` as a `beforeSwap` hook.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e15)` (0.1% per block cap).
3. Admin does **not** call `setLastMidPrice()` (the factory never does it either).
4. Oracle price jumps 50% in one block (simulated manipulation).
5. Attacker calls `pool.swap(...)` — first swap on this pool.
6. Inside `beforeSwap`: `prevMid = priceVelocityState[pool].lastMidPriceX64 = 0` → `if (prevMid != 0)` is `false` → no velocity check → swap executes at the 50%-deviated oracle price.
7. `lastMidPriceX64` is now set to the manipulated price.
8. Attacker profits from the mispriced swap; LPs bear the loss.
9. All subsequent swaps are correctly velocity-checked — only the first swap is unprotected.
