### Title
`SwapAllowlistExtension.beforeSwap` checks the wrong actor (`sender` = router) instead of the actual swapper — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is a production extension that gates `swap` by swapper address, per pool. Its `beforeSwap` hook checks the `sender` parameter, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. The allowlist therefore checks the router's address rather than the actual swapper's address. This creates two mutually exclusive broken states: (1) allowlisted users cannot swap through the router because the router is not allowlisted, or (2) if the admin allowlists the router to fix (1), every user on the network can bypass the allowlist entirely.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

The pool populates `sender` with `msg.sender` of the `swap()` call:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
``` [3](#0-2) 

The actual user's address is stored only in transient storage as the payer for the callback; it is never forwarded to the pool or to the extension. The pool sees `msg.sender = router`, so the extension receives `sender = router` and checks `allowedSwapper[pool][router]`.

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the second parameter, the economic beneficiary of the position), not `sender`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [4](#0-3) 

No equivalent "actual swapper" field exists in the swap hook signature — the interface only exposes `sender` (immediate caller) and `recipient` (output destination):

```solidity
// IMetricOmmExtensions.sol L50-60

    address sender,
    address recipient,

### Impact Explanation

**State A — router not allowlisted (default):** Every allowlisted user who calls `MetricOmmSimpleRouter` is blocked by the extension because the router's address is not in `allowedSwapper`. The supported periphery path is completely unusable for curated pools. This is broken core pool functionality.

**State B — admin allowlists the router to fix State A:** `allowedSwapper[pool][router] = true`. Now `allowAllSwappers[pool]` is effectively true for every user on the network, because any address can call the router and the router's address is what the extension checks. The allowlist is fully bypassed. Any user — including those the admin explicitly excluded — can trade in the curated pool.

State B is the fund-impacting path: if the pool offers subsidized pricing, favorable oracle rates, or is restricted to counterparties for regulatory reasons, unauthorized users can extract value or violate the curation invariant.

---

### Likelihood Explanation

The likelihood is high. `MetricOmmSimpleRouter` is the canonical supported swap entrypoint. Any operator who deploys a curated pool with `SwapAllowlistExtension` and then tries to make it usable via the router will naturally allowlist the router, triggering State B. The misconfiguration is the only path to making the router work with the extension, so it is the expected operational outcome.

---

### Recommendation

The `beforeSwap` hook signature does not carry the originating user's address. Two options:

1. **Extend the hook interface** to include an explicit `swapper` field (analogous to `owner` in `beforeAddLiquidity`) populated by the pool with the address the router forwards via `extensionData` or a dedicated parameter. The router must be updated to pass `msg.sender` as the swapper identity.

2. **Short-term mitigation**: Document that `SwapAllowlistExtension` is incompatible with `MetricOmmSimpleRouter` and enforce this at the factory level (e.g., revert pool creation if both are configured together without an explicit override flag).

---

### Proof of Concept

1. Pool admin deploys a curated pool with `SwapAllowlistExtension` configured as `beforeSwap` extension.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool, recipient: bob, ...})`.
   - Router calls `pool.swap(bob, ...)` with `msg.sender = router`.
   - Extension checks `allowedSwapper[pool][router]` → false → reverts. ✓ Bob blocked.
4. Alice (allowlisted) calls the same router path.
   - Extension checks `allowedSwapper[pool][router]` → false → reverts. ✗ Alice also blocked — broken functionality.
5. Admin calls `setAllowedToSwap(pool, router, true)` to fix Alice's flow.
6. Bob calls the router again.
   - Extension checks `allowedSwapper[pool][router]` → true → passes. ✗ Bob bypasses the allowlist.
7. Bob successfully swaps in the curated pool, violating the curation invariant. [1](#0-0) [6](#0-5) [7](#0-6)
