### Title
Unbounded Iteration in `tryUnlockNlpBalance` Causes Permanent NLP Burn DoS — (`File: core/contracts/SpotEngineState.sol`)

---

### Summary

`tryUnlockNlpBalance` in `SpotEngineState.sol` iterates over every expired entry in a user's `NlpLockedBalanceQueue` without any cap. Because each NLP mint at a distinct oracle timestamp appends a new entry to the queue with no upper bound on `balanceCount`, a user who mints NLP across many distinct timestamps will accumulate a queue large enough to exhaust gas when `tryUnlockNlpBalance` is called, permanently preventing them from burning their NLP.

---

### Finding Description

`NlpLockedBalanceQueue` is a per-subaccount mapping-backed queue that records each NLP mint as a locked balance entry with a future `unlockedAt` timestamp. [1](#0-0) 

In `handleNlpLockedBalance`, a new entry is appended to the queue (`queue.balanceCount++`) whenever the most recent entry's `unlockedAt` timestamp does not match the current oracle time plus `NLP_LOCK_PERIOD`. This means every mint at a distinct oracle timestamp creates a distinct queue slot: [2](#0-1) 

There is no cap on `balanceCount`. The queue grows monotonically with each distinct-timestamp mint.

`tryUnlockNlpBalance` processes all expired entries in a single unbounded `while` loop: [3](#0-2) 

This function is called unconditionally inside `handleNlpLockedBalance`: [4](#0-3) 

`handleNlpLockedBalance` is invoked from `updateBalance` whenever `productId == NLP_PRODUCT_ID`: [5](#0-4) 

`updateBalance` for `NLP_PRODUCT_ID` is called by `Clearinghouse.burnNlp` (and `mintNlp`): [6](#0-5) 

---

### Impact Explanation

After the `NLP_LOCK_PERIOD` elapses, a user with a sufficiently large queue will have all entries become eligible for unlocking simultaneously. The next call to `burnNlp` (or any NLP balance update) triggers `tryUnlockNlpBalance`, which must iterate over every expired entry in one transaction. With no iteration cap, this loop exhausts the block gas limit, causing the transaction to revert. The user's NLP balance remains permanently locked — they cannot burn their NLP or exit their position. This constitutes a collateral lock: real user funds (NLP tokens with quote value) become permanently inaccessible.

The sequencer's slow-mode error handler silently drops out-of-gas transactions rather than reverting the batch: [7](#0-6) 

This means the failed `burnNlp` is silently discarded, and the user has no recovery path.

---

### Likelihood Explanation

Any user who mints NLP regularly — e.g., a liquidity provider who mints once per block or once per oracle update over weeks — will accumulate hundreds to thousands of queue entries. This is normal, expected behavior for an active NLP participant. No adversarial intent is required; the DoS is a natural consequence of routine protocol usage. The only mitigation that currently exists (merging entries with the same `unlockedAt`) only helps if two mints occur within the same oracle timestamp, which is unlikely for a user minting across many separate transactions.

---

### Recommendation

Introduce a hard cap on `balanceCount` (analogous to `MAX_STAKING_CONDITIONS_LIMIT` in the referenced report), or redesign `tryUnlockNlpBalance` to process at most `N` entries per call and allow partial unlocking across multiple transactions. Alternatively, expose a standalone `tryUnlockNlpBalance` entry point (already `public`) that users can call incrementally to drain their queue before attempting a burn, and document this as a required step when the queue is large.

---

### Proof of Concept

1. User calls `mintNlp` (via sequencer/slow-mode) `K` times, each at a distinct oracle timestamp. Each call appends one entry: `queue.balanceCount` grows to `K`. No merging occurs because `getOracleTime()` differs each time. [8](#0-7) 

2. After `NLP_LOCK_PERIOD` passes, all `K` entries satisfy `unlockedAt <= getOracleTime()`.

3. User submits `burnNlp`. The sequencer includes it in a batch. Execution reaches `handleNlpLockedBalance` → `tryUnlockNlpBalance`.

4. The `while` loop iterates `K` times, each iteration performing a storage read (`queue.balances[queue.unlockedUpTo]`), an addition, a `delete`, and an increment — approximately 5,000–10,000 gas per iteration. For `K = 3,000`, this exceeds 15–30M gas, surpassing typical block gas limits. [9](#0-8) 

5. The transaction reverts with out-of-gas. The sequencer's catch block silently drops it. The user's NLP is permanently unburnable.

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }
