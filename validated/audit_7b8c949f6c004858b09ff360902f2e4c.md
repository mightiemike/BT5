### Title
`skip_stateful_validations` hardcodes `Nonce(Felt::ONE)` instead of reading `max_nonce_for_validation_skip` from config, causing gateway to accept or reject transactions contrary to operator configuration - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` defines and serializes a `max_nonce_for_validation_skip` field intended to control the nonce threshold below which stateful `__validate__` is skipped for invoke transactions following a pending `deploy_account`. The free function `skip_stateful_validations` hardcodes `Nonce(Felt::ONE)` and never reads the config field, making the config value a dead parameter. When an operator sets `max_nonce_for_validation_skip` to `0` (disable skip) or to any value `> 1`, the gateway's admission behavior diverges from the declared configuration.

### Finding Description

`StatefulTransactionValidatorConfig` declares and serializes `max_nonce_for_validation_skip`: [1](#0-0) 

The field is documented as "Maximum nonce for which the validation is skipped." Its default is `Nonce(Felt::ONE)`.

The function that is supposed to enforce this threshold is `skip_stateful_validations`: [2](#0-1) 

This is a **free function** — it takes no `self` parameter and therefore has no access to `StatefulTransactionValidatorConfig`. It hardcodes the comparison `tx.nonce() == Nonce(Felt::ONE)` unconditionally.

The call site in `run_pre_validation_checks` has `self` (and thus the config) but does not pass the threshold to the free function: [3](#0-2) 

The analog to the external bug is exact: the config "stores" the intended skip threshold in `max_nonce_for_validation_skip` (analogous to tokens held in `RToken`), but `skip_stateful_validations` reads from a hardcoded constant `Nonce(Felt::ONE)` (analogous to `LendingPool` reading its own zero balance). The two sources diverge whenever the config is set to any non-default value.

### Impact Explanation

**Case 1 — `max_nonce_for_validation_skip = 0` (operator disables skip):**  
The code still skips `__validate__` for any invoke with `nonce == 1` when a `deploy_account` is pending in the mempool. The gateway admits a transaction that the config explicitly forbids skipping validation for. This is "accepts invalid transactions before sequencing."

**Case 2 — `max_nonce_for_validation_skip > 1` (operator widens the skip window):**  
Invoke transactions with nonces `2 … N` are sent through `__validate__` even though the account contract is not yet deployed. The `__validate__` call fails because the account class is not yet on-chain, causing the gateway to reject transactions that the config declares valid. This is "rejects valid transactions before sequencing."

### Likelihood Explanation

The config field is public, documented, and included in the serialized config schema. Any operator who reads the documentation and sets `max_nonce_for_validation_skip` to a non-default value will silently get the wrong admission behavior. The unprivileged trigger is a user submitting an invoke transaction with the relevant nonce; the root cause is the config/code divergence.

### Recommendation

Pass the config threshold into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded literal:

```rust
// In run_pre_validation_checks:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // pass config value
).await?;

// In skip_stateful_validations:
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
    max_nonce_for_validation_skip: Nonce,       // new parameter
) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() <= max_nonce_for_validation_skip   // was: == Nonce(Felt::ONE)
            && account_nonce == Nonce(Felt::ZERO) {
            ...
        }
    }
    Ok(false)
}
```

### Proof of Concept

1. Set `max_nonce_for_validation_skip = 0` in the gateway config (operator intent: never skip `__validate__`).
2. Submit a `deploy_account` transaction for address `A` to the mempool.
3. Submit an invoke transaction from address `A` with `nonce = 1`.
4. `skip_stateful_validations` evaluates `Nonce(1) == Nonce(Felt::ONE)` → `true`, ignoring the config value `0`.
5. `account_tx_in_pool_or_recent_block(A)` returns `true` (the deploy_account is pending).
6. The function returns `true` (skip validation), and the invoke is admitted without running `__validate__` — contrary to the operator's explicit configuration.

Conversely, set `max_nonce_for_validation_skip = 2`. Submit an invoke with `nonce = 2` while `deploy_account` is pending. The hardcoded `== Nonce(Felt::ONE)` check fails; `__validate__` runs against an undeployed account and the transaction is rejected, contrary to the operator's configuration. [2](#0-1) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
