### Title
AuxPoW Parent Block PoW Checked Against Dogecoin Target Instead of Parent Chain Target — (`contract/src/dogecoin.rs`)

### Summary

In `check_aux`, the parent block's proof-of-work hash is validated against `target_from_bits(block_header.bits)` — the **Dogecoin block's** target — rather than `target_from_bits(aux_data.parent_block.bits)` — the **parent chain's** target. This allows an attacker to mine the AuxPoW parent block to Dogecoin's difficulty instead of the parent chain's (e.g., Litecoin's) difficulty, breaking the core AuxPoW invariant.

### Finding Description

The vulnerable check is at lines 149–154: [1](#0-0) 

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
    format!("block should have correct pow")
);
```

`pow_hash` is the hash of `aux_data.parent_block` — the parent chain block. But the target it is compared against is `target_from_bits(block_header.bits)` — the Dogecoin block's bits, not the parent block's bits. The correct check must use `target_from_bits(aux_data.parent_block.bits)`.

The `check_pow` function does enforce that `block_header.bits` matches the expected Dogecoin difficulty: [2](#0-1) 

But this only validates the Dogecoin block's difficulty — it says nothing about whether the parent block satisfies the parent chain's difficulty rules. The parent block's `bits` field is never validated against any expected parent-chain difficulty.

### Impact Explanation

The AuxPoW protocol requires that the parent block (e.g., a Litecoin block) satisfies the parent chain's own difficulty. This is what makes merged mining costly: the miner must produce a block that satisfies the harder of the two chains' difficulties. By checking the parent block's hash against the Dogecoin target instead, the contract accepts a parent block that only satisfies Dogecoin's difficulty, even if Dogecoin's difficulty is significantly easier than the parent chain's.

An attacker can:
1. Set `parent_block.bits` to encode a harder target (lower numeric value) than Dogecoin's current `block_header.bits`.
2. Mine the parent block only to Dogecoin's difficulty (the actual check target).
3. Construct a valid coinbase commitment and chain merkle proof around this parent block.
4. Submit the Dogecoin block via the public `submit_block_header` entrypoint.

The block passes all checks and is accepted as canonical, despite the parent block not satisfying the parent chain's difficulty. This directly enables false canonical header acceptance at reduced mining cost.

### Likelihood Explanation

The entrypoint is fully public — `submit_block_header` is an externally callable NEAR contract method: [3](#0-2) 

No privileged role, DAO, or key compromise is required. The attacker only needs to construct a valid AuxPoW structure (coinbase tx, merkle proofs, chain root commitment) and mine the parent block to Dogecoin's difficulty. All other AuxPoW structural checks (chain merkle proof, coinbase commitment, chain ID, expected index) are correctly enforced and must be satisfied, but none of them prevent this attack.

### Recommendation

Replace `block_header.bits` with `aux_data.parent_block.bits` in the PoW check:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(aux_data.parent_block.bits),
    format!("block should have correct pow")
);
```

This matches the reference Dogecoin AuxPoW implementation, which checks the parent block's hash against the parent block's own declared target.

### Proof of Concept

1. Observe the current Dogecoin mainnet `bits` value, e.g. `0x1a0ffff0` (target T_doge).
2. Construct a parent block header with `bits = 0x1800ffff` (target T_parent < T_doge, i.e., harder).
3. Mine a nonce such that `block_hash_pow(parent_block) <= T_doge` (satisfies Dogecoin difficulty, but NOT T_parent).
4. Build a valid coinbase transaction embedding the Dogecoin block hash via the chain merkle root commitment.
5. Construct valid `merkle_proof` and `chain_merkle_proof` arrays.
6. Call `submit_block_header` with this Dogecoin header and the crafted `AuxData`.
7. The check at line 152 evaluates `pow_hash <= target_from_bits(block_header.bits)` = `pow_hash <= T_doge` — **passes**.
8. The Dogecoin block is accepted as canonical despite the parent block not satisfying the parent chain's difficulty. [1](#0-0)
