### Title
Withdrawal Permanently Blocked at 100% Utilization Due to Transfer-Before-Check Ordering — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers tokens out of the Clearinghouse **before** calling `spotEngine.assertUtilization`. When the SpotEngine's utilization reaches 100% (all deposited tokens are borrowed), the Clearinghouse holds zero free tokens. Any withdrawal attempt fails with a generic ERC20 revert at the transfer step, before the meaningful `ERR_MAX_UTILIZATION` guard can execute. No protocol mechanism exists to close borrowing positions or drain liquidity back to the Clearinghouse, permanently blocking depositors from accessing their funds.

---

### Finding Description

`Clearinghouse.withdrawCollateral` executes in this order:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` is called, which immediately executes `token.safeTransfer(withdrawPool, uint256(amount))` at **line 383**, moving tokens out of the Clearinghouse.
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` updates the accounting.
3. **Line 413** — `spotEngine.assertUtilization(productId)` checks that `totalDeposits >= totalBorrows`. [1](#0-0) 

`assertUtilization` in `SpotEngine` enforces the invariant:

```solidity
require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
``` [2](#0-1) 

The SpotEngine is a full lending/borrowing system. `SpotEngineState._updateState` computes a live `utilizationRatioX18 = totalBorrows.div(totalDeposits)` and accrues interest accordingly. [3](#0-2) 

Negative `amountNormalized` values represent borrowed positions. When `totalBorrows` approaches `totalDeposits`, the Clearinghouse's actual ERC-20 token balance approaches zero — all deposited tokens have been lent out through the protocol's accounting. [4](#0-3) 

At 100% utilization, `token.balanceOf(address(clearinghouse)) ≈ 0`. The `safeTransfer` at line 383 reverts with a generic ERC-20 insufficient-balance error before `assertUtilization` ever runs. The meaningful `ERR_MAX_UTILIZATION` revert path is never reached. [5](#0-4) 

There is no protocol function to close borrowing positions, force-repay borrowers, or otherwise drain liquidity back into the Clearinghouse to unblock depositors. `removeLiquidity` on `BaseWithdrawPool` is `onlyOwner` and operates on the WithdrawPool's own balance, not the Clearinghouse's. [6](#0-5) 

---

### Impact Explanation

Any user with a positive spot balance who attempts `withdrawCollateral` when utilization is at or near 100% receives a generic ERC-20 revert. Their funds are locked in the Clearinghouse with no on-chain recovery path. The corrupted invariant is: **a depositor with a positive SpotEngine balance cannot redeem their collateral even though the accounting confirms they are owed it**, because the actual token balance of the Clearinghouse is zero. This is a direct asset-lock on depositor funds.

---

### Likelihood Explanation

The SpotEngine's interest rate model is parameterized to allow utilization up to 100% — the `interestLargeCapX18` rate kicks in above `interestInflectionUtilX18` (configured at 0.8) but does not cap utilization at any value below 1.0. [7](#0-6) 

A large borrower (or coordinated borrowers) can push utilization to 100% through normal protocol usage (cross-margin borrowing, leveraged spot positions). Once reached, all depositor withdrawals are blocked. Likelihood is **medium**: requires high utilization, which is a realistic market condition during periods of high demand for leverage.

---

### Recommendation

Reorder the operations in `Clearinghouse.withdrawCollateral` so that `spotEngine.assertUtilization` (and the balance update) is called **before** `handleWithdrawTransfer`. This ensures the meaningful `ERR_MAX_UTILIZATION` error is surfaced and the transfer is never attempted when the Clearinghouse is underfunded:

```solidity
// 1. Update accounting first
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);

// 2. Check utilization before any transfer
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 3. Only then transfer
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, consider adding a mechanism (analogous to the Gearbox recommendation) that allows the protocol to close or partially unwind borrowing positions to restore liquidity when utilization is critically high.

---

### Proof of Concept

1. Alice deposits 1000 USDC into the Clearinghouse via `depositCollateral`. SpotEngine records `totalDepositsNormalized += 1000`.
2. Bob borrows 1000 USDC (via cross-margin trading), creating a negative balance. SpotEngine records `totalBorrowsNormalized += 1000`. Clearinghouse token balance = 0.
3. Alice calls `withdrawCollateral` for 100 USDC via the Endpoint.
4. `Clearinghouse.withdrawCollateral` is invoked. At line 408, `handleWithdrawTransfer` is called.
5. At line 383, `token.safeTransfer(withdrawPool, 100e6)` is executed. Since `token.balanceOf(clearinghouse) == 0`, this reverts with a generic ERC-20 error.
6. `assertUtilization` at line 413 is never reached. Alice receives no meaningful error and cannot recover her funds.
7. No on-chain function exists to force Bob to repay or to otherwise restore the Clearinghouse's token balance. [8](#0-7)

    ) public virtual onlyEndpoint {

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));

        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
