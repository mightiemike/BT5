### Title
Unconditional Trust of Caller-Supplied `_cached_sha256_treehash` in `Program.wrap()` Enables Permanent Wrong Puzzle-Hash Commitment — (File: `wheel/python/clvm_rs/program.py`)

---

### Summary

`Program.wrap()` blindly copies `_cached_sha256_treehash` from any `CLVMStorage`-conforming input object without verification. Once set, `Program.tree_hash()` returns this value permanently without recomputation. A caller-supplied object carrying a wrong `_cached_sha256_treehash` silently poisons all downstream puzzle-hash derivations, equality checks, and curry operations, with no recovery path — a direct analog to the "wrong rootHash committed with no recovery" class from the external report.

---

### Finding Description

**Root cause — `Program.wrap()` copies the hash blindly:** [1](#0-0) 

```python
@classmethod
def wrap(cls, v: CLVMStorage) -> Program:
    if isinstance(v, Program):
        return v
    o = cls()
    ...
    o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)  # line 142
    return o
```

`getattr(v, "_cached_sha256_treehash", None)` is accepted from any `CLVMStorage` object without any verification that the value is the correct SHA-256 tree hash of `v`'s actual content.

**Root cause — `Program.tree_hash()` never recomputes once set:** [2](#0-1) 

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
```

If `_cached_sha256_treehash` is non-`None` (copied from the input in `wrap()`), it is returned immediately. There is no API to invalidate or reset it.

**Root cause — `Treehasher.sha256_treehash()` trusts the cached value on any sub-object:** [3](#0-2) 

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
```

When computing the hash of a parent node, the traversal short-circuits on any child that already has `_cached_sha256_treehash` set. A wrong hash on a child propagates into every ancestor's hash.

**Entry path:**

`CLVMStorage` is a public duck-typing protocol. Any Python object with `atom` and `pair` attributes qualifies. `Program.to()` calls `to_clvm_object()`, which passes a `CLVMStorage` object through unchanged: [4](#0-3) 

```python
if is_clvm_storage(v):
    v = cast(CLVMStorage, v)
    did_convert.append(v)
    continue
```

Then `Program.wrap(v)` is called on it, copying the wrong hash. The `CLVMStorage` protocol is intentionally open — third-party libraries, plugin code, or any caller that constructs a `CLVMStorage` object and sets `_cached_sha256_treehash` to an incorrect value will have that value silently accepted.

---

### Impact Explanation

Once a `Program` object carries a wrong `_cached_sha256_treehash`:

1. **`Program.__eq__()`** uses `tree_hash()` for comparison — equality checks silently return wrong results. [5](#0-4) 

2. **`Program.curry_hash()`** calls `self.tree_hash()` to derive the puzzle hash — the wrong hash propagates into every curried puzzle hash derived from this program. [6](#0-5) 

3. **Parent nodes** computed via `Program.to((p, p))` or `Program.new_pair()` will also carry wrong hashes because `Treehasher` short-circuits on the child's cached value. [3](#0-2) 

In the Chia blockchain context, puzzle hashes are coin addresses. A wrong puzzle hash means coins are sent to an unspendable address or validation fails. There is no recovery mechanism — the wrong hash is permanent for the lifetime of the `Program` object.

---

### Likelihood Explanation

**Medium.** The `CLVMStorage` protocol is a documented, public Python duck-typing interface. Any third-party library, plugin, or application layer that:
- Implements `CLVMStorage` and sets `_cached_sha256_treehash` (e.g., for performance), or
- Has a bug in its own hash caching, or
- Deliberately sets a wrong hash

…and then passes that object to `Program.to()` or `Program.wrap()` will trigger this. The `Treehasher` itself writes `_cached_sha256_treehash` back onto sub-objects it traverses (line 66, 90), so any object that was previously hashed by a *different* `Treehasher` instance (with different `atom_prefix`/`pair_prefix`) and then passed to the standard `CHIA_TREEHASHER` path would also carry a wrong cached value that is silently trusted. [7](#0-6) 

---

### Recommendation

1. In `Program.wrap()`, do not copy `_cached_sha256_treehash` from the input object unless the input is already a `Program` (i.e., already produced by this library). Remove line 142 or guard it with `isinstance(v, Program)`.
2. In `Treehasher.sha256_treehash()`, only trust `_cached_sha256_treehash` on objects that are known to have been hashed by the same `Treehasher` instance (e.g., by checking a sentinel attribute set by this specific hasher).
3. Document clearly that `_cached_sha256_treehash` on externally-supplied `CLVMStorage` objects is trusted unconditionally, so callers understand the invariant they must uphold.

### Proof of Concept

```python
from clvm_rs.program import Program

class MaliciousStorage:
    """Minimal CLVMStorage with a wrong pre-set tree hash."""
    atom = b"foo"
    pair = None
    _cached_sha256_treehash = b"\xde\xad" * 16  # 32 bytes of garbage

p = Program.wrap(MaliciousStorage())

# tree_hash() returns the wrong cached value without recomputation
assert p.tree_hash() == b"\xde\xad" * 16  # wrong; correct is sha256(b"\x01foo")

# The wrong hash propagates into parent nodes
p2 = Program.to((p, p))
# p2.tree_hash() is also wrong because Treehasher short-circuits on p's cached value
wrong_parent = p2.tree_hash()

# Equality is now broken
p_correct = Program.from_bytes(bytes.fromhex("83666f6f"))  # atom "foo"
assert p != p_correct  # False — they compare as unequal despite same content

# curry_hash() derives a wrong puzzle hash
mod = Program.fromhex("ff02ff0480")
wrong_puzzle_hash = mod.curry_hash(p.tree_hash())
# wrong_puzzle_hash != mod.curry(p).tree_hash() — silent divergence
```
