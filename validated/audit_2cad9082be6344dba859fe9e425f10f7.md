### Title
Linked Signer Can Permanently Hijack Subaccount by Self-Replacing Its Own Authorization — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **currently active linked signer** — not just the subaccount owner — can sign a `LinkSigner` transaction to replace itself with any arbitrary address. Because `linkedSigners` is a persistent, never-expiring mapping, a compromised linked signer can permanently seize control of a subaccount and drain all collateral before the owner can react.

---

### Finding Description

`EndpointStorage.sol` stores a persistent mapping:

```solidity
mapping(bytes32 => address) internal linkedSigners;
``` [1](#0-0) 

This mapping is written by two code paths. In the **slow-mode path** (`processSlowModeTransactionImpl`), `LinkSigner` is gated by `validateSender`, which requires `msg.sender == address(uint160(bytes20(txn.sender)))` — i.e., only the subaccount owner's EOA can submit it:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [2](#0-1) 

In the **sequencer path** (`processTransactionImpl`), however, `LinkSigner` is validated with `allowLinkedSigner = true`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // <-- allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [3](#0-2) 

`validateSignedTx` forwards this flag into `validateSignature`:

```solidity
validateSignature(
    sender,
    _hashTypedDataV4(...),
    signature,
    allowLinkedSigner
);
``` [4](#0-3) 

And `Verifier.validateSignature` accepts the recovered address if it equals **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [5](#0-4) 

The result: the current linked signer can produce a valid `SignedLinkSigner` transaction pointing `signer` to any attacker-controlled address, and the sequencer will process it as legitimate.

---

### Impact Explanation

Once the linked signer is replaced with an attacker address, the attacker holds **persistent, indefinite signing authority** over the victim subaccount. The attacker can immediately sign `WithdrawCollateral` or `TransferQuote` transactions to drain all collateral. The original owner's only recourse is a slow-mode `LinkSigner` transaction, which carries a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window the attacker can fully drain the account. The corrupted state delta is: `linkedSigners[victim_subaccount]` is permanently overwritten to the attacker's address, and the victim's collateral balance is reduced to zero. [6](#0-5) 

---

### Likelihood Explanation

Linked signers are designed for hot wallets and automated trading bots — entities with weaker key-management hygiene than cold-wallet subaccount owners. Any key compromise, phishing event, or malicious upgrade to a contract-based linked signer is sufficient to trigger the attack. No admin access, sequencer compromise, or governance capture is required. The attacker only needs to produce one valid EIP-712 signature from the compromised linked signer key.

---

### Recommendation

Change the `allowLinkedSigner` flag to `false` for `LinkSigner` transactions in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to change the linked signer. The slow-mode path already enforces this correctly via `validateSender`.

```solidity
// EndpointTx.sol – processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // linked signer must NOT be able to change itself
);
``` [7](#0-6) 

---

### Proof of Concept

1. **Setup**: Alice owns subaccount `alice_sub`. She sets linked signer to `bot_address` via a valid `LinkSigner` slow-mode transaction. `linkedSigners[alice_sub] = bot_address`.

2. **Compromise**: Attacker obtains `bot_address` private key (key leak, phishing, or malicious contract upgrade).

3. **Attack**: Attacker crafts a `SignedLinkSigner` struct:
   - `tx.sender = alice_sub`
   - `tx.signer = attacker_address` (bytes32-padded)
   - `tx.nonce = nonces[alice_address]` (current nonce)
   - `signature` = EIP-712 signature over the above, signed by `bot_address`

4. **Sequencer processes**: `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`. `Verifier.validateSignature` recovers `bot_address`, which equals `linkedSigners[alice_sub]` → passes. `linkedSigners[alice_sub]` is overwritten to `attacker_address`.

5. **Drain**: Attacker signs a `SignedWithdrawCollateral` with `attacker_address` for `alice_sub`. `validateSignedTx(..., true)` recovers `attacker_address == linkedSigners[alice_sub]` → passes. Alice's collateral is transferred out.

6. **No recovery within delay**: Alice submits a slow-mode `LinkSigner` to reset the signer, but the 3-day `SLOW_MODE_TX_DELAY` means the drain completes before the reset takes effect. [8](#0-7) [1](#0-0)
