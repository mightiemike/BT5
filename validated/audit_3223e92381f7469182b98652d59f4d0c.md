### Title
`WithdrawCollateral` (V1) Charges Current `withdrawFeeX18` Instead of Fee Committed at Signing Time - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateral` (V1) transaction handler in `EndpointTx.sol` charges the user the **current** `withdrawFeeX18` from `SpotEngine` at execution time. Because the fee is not included in the signed transaction struct, the user cannot commit to a specific fee at signing time. If `withdrawFeeX18` is updated between when the user signs the withdrawal and when the sequencer processes it, the user is charged a different fee than was in effect when they signed.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `WithdrawCollateral` (V1) path reads the fee live from `spotEngine.getConfig()` at execution time:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,  // current fee, not signed fee
    signedTx.tx.productId
);
``` [1](#0-0) 

The `WithdrawCollateral` struct (V1) contains only `sender`, `productId`, `amount`, and `nonce` — no fee field:

```solidity
struct WithdrawCollateral {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
}
``` [2](#0-1) 

The `withdrawFeeX18` is part of the mutable `Config` struct stored in `SpotEngine`, and is overwritten wholesale whenever `addOrUpdateProduct` is called by the owner:

```solidity
configs[productId] = config;
``` [3](#0-2) 

The `chargeFee` function directly debits the user's spot balance:

```solidity
function chargeFee(bytes32 sender, int128 fee, uint32 productId) internal {
    spotEngine.updateBalance(productId, sender, -fee);
    sequencerFee[productId] += fee;
}
``` [4](#0-3) 

The protocol itself acknowledged this design flaw and fixed it in `WithdrawCollateralV2`, which includes `feeX18` in the signed struct and validates it against the current fee:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [5](#0-4) 

V1 has no equivalent protection.

---

### Impact Explanation

A user who signs a `WithdrawCollateral` (V1) transaction when `withdrawFeeX18 = X` may have their balance debited by `Y > X` if the fee is raised before the sequencer processes the transaction. The user's signed intent committed to withdrawing `amount` of collateral, but the actual net received is reduced by a fee they never agreed to. Conversely, if the fee decreases, the user pays less than the current rate, which is a protocol accounting discrepancy. The corrupted state is the user's spot balance for `productId`, which is decremented by an amount the user did not consent to at signing time.

---

### Likelihood Explanation

The owner can update `withdrawFeeX18` at any time via `addOrUpdateProduct`. The sequencer processes signed transactions with an inherent delay (off-chain ordering, batching). Any fee update that lands between a user's signing event and the sequencer's processing of that V1 withdrawal will silently charge the user the new fee. This is a normal operational scenario — fee adjustments are expected — and V1 withdrawals remain in use alongside V2.

---

### Recommendation

Either:
1. **Deprecate V1 withdrawals** and require all new withdrawals to use `WithdrawCollateralV2`, which already includes the user-committed `feeX18` field with the `require(signedTx.feeX18 <= currentFeeX18)` guard.
2. **Or**, for V1 compatibility, snapshot the `withdrawFeeX18` at the time the transaction is first queued (e.g., in slow-mode) and use that snapshot at execution time.

---

### Proof of Concept

1. `withdrawFeeX18` for `productId = 1` is `100` (in X18 units).
2. User signs a `WithdrawCollateral` V1 transaction: `{ sender, productId: 1, amount: 1000e18, nonce: N }`.
3. Before the sequencer processes it, the owner calls `SpotEngine.addOrUpdateProduct(1, ..., config_with_withdrawFeeX18=500, ...)`.
4. Sequencer processes the V1 withdrawal. `EndpointTx` reads `spotEngine.getConfig(1).withdrawFeeX18 = 500` and calls `chargeFee(sender, 500, 1)`.
5. User is charged `500` instead of the `100` that was in effect when they signed — a 5× overcharge with no recourse, as the nonce is consumed and the transaction cannot be replayed. [6](#0-5) [7](#0-6)
