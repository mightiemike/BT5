Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates caller-controlled `owner` instead of actual depositor `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead validates `owner`, a free caller-controlled argument. Because any caller can supply an allowlisted address as `owner`, the deposit allowlist is completely ineffective: any non-allowlisted address can deposit into a permissioned pool by naming an allowlisted address as the LP-position recipient. The sibling `SwapAllowlistExtension.beforeSwap` correctly validates `sender`, confirming the inconsistency is a defect, not a design choice.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (first parameter is unnamed) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`SwapAllowlistExtension.beforeSwap` correctly validates `sender`:

```solidity
// SwapAllowlistExtension.sol lines 31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

**Exploit path:**
1. Pool admin configures `DepositAllowlistExtension` with only address `A` allowlisted (`allowedDepositor[pool][A] = true`).
2. Non-allowlisted address `B` calls `pool.addLiquidity(owner = A, salt, deltas, callbackData, extensionData)`.
3. The hook evaluates `allowedDepositor[pool][A]` → `true` → passes without reverting.
4. `B` is never checked. `B` pays the tokens via the modify-liquidity callback; LP shares are minted to `A`.
5. If `B` and `A` collude (or `B` controls `A`), `A` calls `removeLiquidity` and returns proceeds to `B`, completing a full allowlist bypass.

Even without collusion, `B` can force liquidity into the pool under `A`'s position, violating the pool admin's access control intent.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who may interact with a permissioned pool (KYC/AML compliance, private LP pools). With this bug the guard is completely ineffective against the actual caller. Any address can call `addLiquidity` as long as it names an allowlisted address as `owner`. This constitutes a broken core pool functionality (the allowlist extension is rendered inoperative) and an admin-boundary break where an unprivileged path bypasses a configured pool access control. Severity: **High** — the entire deposit restriction mechanism is defeated with zero preconditions beyond knowing one allowlisted address (which is on-chain readable from `allowedDepositor` events or storage).

## Likelihood Explanation

No special privileges are required. The attacker only needs to identify one allowlisted address (trivially readable from `AllowedToDepositSet` events or direct storage reads) and call `addLiquidity` with that address as `owner`. The attack is repeatable on every block, requires no flash loans or complex setup, and works against any pool using `DepositAllowlistExtension`.

## Recommendation

Name and validate `sender` instead of `owner` in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins must then allowlist the actual depositor addresses (the `msg.sender` of `addLiquidity` calls), not LP-position owner addresses. Existing allowlist entries keyed on `owner` addresses must be migrated to the corresponding `sender` addresses.

## Proof of Concept

```solidity
// Foundry test sketch
function test_allowlistBypass() public {
    address allowlistedA = makeAddr("allowlistedA");
    address attackerB    = makeAddr("attackerB");

    // Admin allowlists only A
    vm.prank(poolAdmin);
    extension.setAllowedToDeposit(address(pool), allowlistedA, true);

    // B is not allowlisted — should revert, but does not
    assertFalse(extension.isAllowedToDeposit(address(pool), attackerB));

    // B calls addLiquidity naming A as owner
    vm.prank(attackerB);
    // Expect: revert NotAllowedToDeposit — Actual: succeeds
    pool.addLiquidity(allowlistedA, salt, deltas, callbackData, extensionData);

    // LP shares minted to A; B paid the tokens — allowlist fully bypassed
}
