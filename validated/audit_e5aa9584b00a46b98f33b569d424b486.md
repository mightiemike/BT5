### Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the end-user, allowing any unprivileged user to bypass per-user swap allowlists via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument it receives from the pool, which is the pool's `msg.sender` — the router contract — not the original end-user. When a pool admin allowlists the router to enable router-mediated swaps for their curated user set, every unprivileged user can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`.

---

### Finding Description

**Call chain that exposes the wrong actor:**

`MetricOmmPool.swap` captures `msg.sender` and forwards it verbatim as `sender` to `_beforeSwap`:

```
MetricOmmPool.swap (msg.sender = router)
  → _beforeSwap(sender = msg.sender = router, ...)
    → ExtensionCalling._callExtensionsInOrder
      → SwapAllowlistExtension.beforeSwap(sender = router, ...)
``` [1](#0-0) 

Inside `ExtensionCalling._beforeSwap`, the router address is encoded as the `sender` argument passed to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

where `msg.sender` = pool and `sender` = router. [3](#0-2) 

The check resolves to `allowedSwapper[pool][router]`. The original end-user's address is never consulted.

**The structural conflict:**

A pool admin who wants allowlisted users (Alice, Bob) to be able to swap through the router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` is `true` for every call that arrives through the router — regardless of who the actual end-user is. Charlie (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap`, the extension sees `sender = router`, the router is allowlisted, and the swap succeeds. [4](#0-3) 

The router never forwards the original caller's identity to the pool or to the extension. There is no mechanism in the current extension interface for the extension to recover the true end-user address. [5](#0-4) 

The result is a binary, irreconcilable conflict identical in structure to the external report's fixed-size array mismatch: the extension is designed to gate individual users, but the `sender` binding makes it structurally impossible to enforce per-user policy when the router is in the call path. Either the router is not allowlisted (no user can swap through the router, even allowlisted ones) or the router is allowlisted (every user can swap through the router, including non-allowlisted ones).

---

### Impact Explanation

Any user who is not individually allowlisted can bypass the `SwapAllowlistExtension` on a curated pool by routing through `MetricOmmSimpleRouter`. This breaks the core access-control invariant of the allowlist extension: "a curated pool must enforce the same allowlist policy regardless of which supported public entrypoint reaches it." Unauthorized users can execute swaps against restricted pools, extracting value (arbitrage, front-running) that the pool admin intended to reserve for a specific set of participants.

---

### Likelihood Explanation

High. The `MetricOmmSimpleRouter` is the primary user-facing swap entrypoint. Any pool admin who deploys a `SwapAllowlistExtension` and wants their allowlisted users to be able to use the router — a completely reasonable and expected operational requirement — must allowlist the router address. The moment they do, the bypass is open to all users. No malicious intent is required; the flaw is triggered by normal, good-faith pool administration.

---

### Recommendation

The `SwapAllowlistExtension` must gate the original end-user, not the intermediary. Two viable approaches:

1. **Encode the original caller in `extensionData`**: The router encodes `msg.sender` (the end-user) into the `extensionData` it forwards to the pool. The `SwapAllowlistExtension` decodes and checks that address. This requires a convention between the router and the extension.

2. **Check `recipient` instead of `sender` for router flows**: If the pool's design intent is that the economically relevant actor for a swap is the `recipient` (the address receiving output tokens), the extension could check `recipient`. However, this changes semantics for direct swaps where `sender != recipient`.

3. **Separate router-aware allowlist**: Introduce a router-aware variant of the extension that accepts a signed or encoded user identity from the router, verified against a known router address.

At minimum, the documentation for `SwapAllowlistExtension` must warn that allowlisting the router grants unrestricted swap access to all users.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  pool admin calls: setAllowedToSwap(pool, Alice, true)   // Alice is allowlisted
  pool admin calls: setAllowedToSwap(pool, router, true)  // router allowlisted so Alice can use it

Attack:
  Charlie (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: Charlie,
      ...
    })

  Router calls: pool.swap(Charlie, zeroForOne, amount, ..., extensionData)
    → pool passes msg.sender (= router) as sender to _beforeSwap
    → SwapAllowlistExtension.beforeSwap(sender = router, ...)
    → allowedSwapper[pool][router] == true  ✓
    → swap executes for Charlie despite Charlie not being allowlisted

Result:
  Charlie successfully swaps on a pool that was intended to restrict access to Alice only.
  The allowlist is fully bypassed for any user who routes through the router.
``` [6](#0-5) [7](#0-6) [8](#0-7)
