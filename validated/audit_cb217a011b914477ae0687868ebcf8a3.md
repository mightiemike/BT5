### Title
Silent Slow-Mode Transaction Drop via `abi.decode` Revert Permanently Loses User's Slow Mode Fee — (`core/contracts/Endpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.processSlowModeTransactionImpl` calls `abi.decode` on the raw `transaction[1:]` bytes supplied by the user. If those bytes are malformed or truncated, `abi.decode` reverts. `Endpoint._executeSlowModeTransaction` wraps the call in a `try/catch` that silently discards the revert — the inline comment `// try return funds now removed` confirms a prior fund-recovery path was deleted. Because `chargeSlowModeFee` already performed a real ERC-20 `safeTransferFrom` during submission, the user's slow-mode fee is permanently lost and the intended operation never executes.

---

### Finding Description

**Step 1 — Fee is charged on submission.**

`EndpointStorage.chargeSlowModeFee` performs a real token transfer from the caller: [1](#0-0) 

This is called inside `EndpointTx.submitSlowModeTransactionImpl` for every transaction type that is not in the owner-only list (i.e., `WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`, and others): [2](#0-1) 

The transaction bytes are then stored verbatim in `slowModeTxs` without any structural validation of the body.

**Step 2 — `abi.decode` is called on unvalidated user bytes during processing.**

When the stored transaction is later executed, `processSlowModeTransactionImpl` calls `abi.decode(transaction[1:], ...)` for each supported type. Representative examples: [3](#0-2) [4](#0-3) [5](#0-4) 

If `transaction[1:]` is empty, truncated, or otherwise ABI-malformed, `abi.decode` reverts unconditionally — there is no way to guard against this in Solidity without a try/catch around the decode itself.

**Step 3 — The outer try/catch silently discards the revert.**

`_executeSlowModeTransaction` deletes the queued transaction *before* attempting execution, then wraps the call in a catch that does nothing: [6](#0-5) 

The comment `// try return funds now removed` on line 226 explicitly documents that a prior refund path was deleted. The catch block now only checks for out-of-gas; any application-level revert (including an `abi.decode` panic) is silently swallowed.

---

### Impact Explanation

`chargeSlowModeFee` transfers the quote token (e.g., USDC) from the submitter to the `Endpoint` contract at submission time. When the corresponding slow-mode transaction later fails silently due to an `abi.decode` revert, those tokens remain in the contract with no mechanism to recover them. The intended operation (withdrawal, signer link, builder-fee claim, etc.) is also permanently dropped from the queue. The corrupted state delta is: `user quote balance -= slowModeFee` with no corresponding credit or refund.

---

### Likelihood Explanation

Any unprivileged user who calls `submitSlowModeTransaction` with a valid first byte (`txType`) but a malformed or zero-length body triggers this path. This can happen accidentally (wrong ABI encoding, off-by-one slice, client-side bug) or deliberately. The fee is a real ERC-20 amount; the loss is permanent and unrecoverable on-chain.

---

### Recommendation

1. **Validate body length before storing**: In `submitSlowModeTransactionImpl`, add a minimum-length check on `transaction[1:]` for each `txType` before charging the fee and enqueuing the transaction.
2. **Restore a refund path in the catch block**: Re-introduce the removed "try return funds" logic so that a failed slow-mode transaction refunds the slow-mode fee to the original sender.
3. **Wrap `abi.decode` in a try/catch at the decode site**: Although Solidity does not support `try/catch` around `abi.decode` directly, the call can be wrapped via a low-level `this.decodeXxx(...)` external call pattern to isolate the panic.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction` with `transaction = bytes([uint8(TransactionType.WithdrawCollateral)])` — a single-byte payload (valid type, empty body).
2. `submitSlowModeTransactionImpl` falls into the `else` branch (line 369–372), calls `chargeSlowModeFee`, and stores the transaction. The user's quote tokens are transferred to `Endpoint`.
3. Sequencer (or anyone after the delay) calls `executeSlowModeTransaction`.
4. `_executeSlowModeTransaction` deletes the entry from `slowModeTxs` and calls `this.processSlowModeTransaction`.
5. Inside `processSlowModeTransactionImpl`, `abi.decode(transaction[1:], (IEndpoint.WithdrawCollateral))` is called on a zero-length slice — this panics with an ABI decode error.
6. The `catch` block in `_executeSlowModeTransaction` fires; the gas check passes; execution continues silently.
7. The user's slow-mode fee is permanently held in `Endpoint` with no refund path. The withdrawal is never processed. [7](#0-6) [8](#0-7)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
