### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Ancestor Timestamp Instead of Fork's Own Ancestor — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When computing the required difficulty for a fork block at a difficulty-adjustment boundary, `get_next_work_required` fetches the timestamp of the first block in the difficulty period via `get_header_by_height`. This function always returns the **mainchain** block at that height, not the fork's actual ancestor. When the fork diverges before the difficulty-period boundary, the wrong timestamp is used, producing an incorrect expected-difficulty value. A developer-placed TODO comment in `dogecoin.rs` explicitly acknowledges this uncertainty.

---

### Finding Description

In all three chain implementations, difficulty retargeting fetches the "first block of the period" timestamp using `get_header_by_height`:

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [4](#0-3) 

This unconditionally reads from `mainchain_height_to_header`, which maps heights to the **mainchain** block hashes. It has no knowledge of which fork chain is currently being validated.

By contrast, `prev_block_header` is correctly fetched via `get_prev_header`, which follows `prev_block_hash` through `headers_pool` — correctly traversing the fork chain: [5](#0-4) 

The desynchronization: `prev_block_header` is the fork's true ancestor, but `get_header_by_height(height_first)` returns the **mainchain's** block at that height. When the fork diverges before `height_first`, these two blocks have different timestamps, and the difficulty calculation is computed from the wrong one.

The check happens inside `check_pow` → `get_next_work_required`, called from `submit_block_header` **before** the fork block is stored. At that moment, `mainchain_height_to_header` still reflects the mainchain, not the fork being validated. [6](#0-5) 

---

### Impact Explanation

**Incorrect difficulty accepted (under-difficulty fork accepted):** If the mainchain's block at `height_first` has a *later* timestamp than the fork's block at the same height, the computed `actual_time_taken` is smaller, the retargeted difficulty is *higher* than the fork's true required difficulty, and the `expected_bits == block_header.bits` check passes for a block that does not meet the fork's actual required difficulty. A relayer can submit a fork chain whose blocks carry insufficient PoW, have it accepted, and — if its cumulative chainwork exceeds the mainchain — trigger a reorg that installs a fraudulent canonical chain.

**Valid fork blocks incorrectly rejected (DoS):** If the mainchain's block at `height_first` has an *earlier* timestamp, the computed difficulty is lower than the fork's true required difficulty, and a legitimately mined fork block is rejected.

The first case is the security-critical one: it breaks the PoW invariant that every accepted block must meet the difficulty computed from its own ancestor chain.

---

### Likelihood Explanation

- **Dogecoin** (`difficulty_adjustment_interval = 1` after block 145,000): every block is a retarget boundary. `height_first` is always `prev_block_header.block_height - 1`. Any fork block at height ≥ fork-point + 2 is affected, because the mainchain block at `height_first` differs from the fork's block at that height. This is the highest-likelihood case. [7](#0-6) 

- **Bitcoin / Litecoin** (`difficulty_adjustment_interval = 2016`): the fork must diverge before the difficulty-period boundary. This requires a deeper fork but is still reachable by any relayer submitting a competing chain.

The developer's own TODO comment in `dogecoin.rs` line 291 confirms this was identified as an open correctness question and never resolved.

---

### Recommendation

Replace `get_header_by_height` in the difficulty calculation with a function that walks backward through the fork's ancestor chain (via `get_prev_header`) from `prev_block_header` until it reaches `height_first`. This mirrors how Bitcoin Core's `GetAncestor` works and ensures the timestamp used for retargeting belongs to the fork's own history, not the mainchain's.

---

### Proof of Concept

1. The contract is initialized with a mainchain at height H (a difficulty-adjustment boundary for Dogecoin, so every block retargets).
2. A fork diverges at height H: the fork's block at H has timestamp `T_fork`; the mainchain's block at H has timestamp `T_main`, where `T_fork ≠ T_main`.
3. A relayer submits fork block at height H+1 (prev = fork block at H). The retarget needs `height_first = H`.
4. `get_header_by_height(H)` returns the **mainchain** block with timestamp `T_main`.
5. `calculate_next_work_required` computes `actual_time_taken = T_prev_fork - T_main` instead of the correct `T_prev_fork - T_fork`.
6. If `T_main > T_fork`, `actual_time_taken` is smaller → computed target is harder → the check `expected_bits == block_header.bits` passes for a block whose bits encode a target easier than the true required target.
7. The under-difficulty fork block is accepted. If the fork accumulates enough such blocks to exceed mainchain chainwork, `reorg_chain` installs it as the new canonical chain. [8](#0-7) [9](#0-8)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
```
