### Title
Zero-Value `safeTransferFrom` in Fast Withdrawal Fee Collection Reverts for Tokens That Reject Zero Transfers — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.submitFastWithdrawal`, when the caller is a third-party fee payer (`sendTo != msg.sender`), the computed `fee` is transferred from `msg.sender` via `safeTransferFrom` with no zero-value guard. For small withdrawal amounts of low-decimal tokens, integer division causes `fee` to round to zero. If the collateral token reverts on zero-value transfers, the entire fast withdrawal reverts, permanently blocking that withdrawal path for the affected `idx`.

---

### Finding Description

`submitFastWithdrawal` supports two fee-payment modes:

1. **Self-pay** (`sendTo == msg.sender`): fee is deducted from `transferAmount`; a `require(transferAmount > uint128(fee))` guard prevents zero-amount transfers.
2. **Third-party pay** (`sendTo != msg.sender`): the caller pays the fee via `safeTransferFrom`, then the full `transferAmount` is sent to `sendTo`. [1](#0-0) 

In the third-party path, the fee is computed as:

```solidity
int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;
int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
return feeX18 / int128(multiplier);
``` [2](#0-1) 

For a 6-decimal token (e.g., USDC), `multiplier = 10^12`. For a withdrawal of `< 1000` native units (< 0.001 USDC), `feeX18 < 10^12`, so `fee = feeX18 / 10^12 = 0` due to integer division. When `fee == 0`, the code unconditionally executes:

```solidity
safeTransferFrom(token, msg.sender, uint128(fee)); // transfers 0
``` [3](#0-2) 

`safeTransferFrom` calls `token.transferFrom(msg.sender, address(this), 0)` with no zero-amount guard: [4](#0-3) 

If the token reverts on zero-value transfers (a known ERC-20 variant), the entire `submitFastWithdrawal` call reverts. Because `markedIdxs[idx]` is set to `true` at the start of the function **before** the fee transfer, the `idx` is permanently consumed: [5](#0-4) 

This means the withdrawal at that `idx` can never be re-submitted via the fast path.

---

### Impact Explanation

A fast withdrawal signed by a user for a small amount of a zero-transfer-reverting token is permanently bricked when a third-party fee payer attempts to execute it. The `idx` is marked as used before the revert-triggering transfer, so no retry is possible via `submitFastWithdrawal`. The user must fall back to the slow-mode withdrawal path (3-day delay), losing the fast-withdrawal guarantee. If the token also reverts on zero-value transfers in the slow path's `handleWithdrawTransfer`, funds could be further delayed.

---

### Likelihood Explanation

- The `sendTo != msg.sender` path is a supported and documented use case (third-party fee payment for fast withdrawals).
- Tokens that revert on zero-value transfers are a known ERC-20 variant (documented in the weird-ERC20 list).
- For 6-decimal tokens, any withdrawal below ~1000 native units causes the fee to round to zero, which is a realistic amount for dust or small positions.
- No minimum withdrawal amount is enforced in `submitFastWithdrawal`.

---

### Recommendation

Add a zero-value guard before the `safeTransferFrom` call in the third-party fee path:

```solidity
} else {
    if (uint128(fee) > 0) {
        safeTransferFrom(token, msg.sender, uint128(fee));
    }
}
```

Similarly, guard `handleWithdrawTransfer` against zero `transferAmount` to prevent a second zero-transfer revert on the payout side.

---

### Proof of Concept

1. A user signs a `WithdrawCollateral` transaction for `amount = 500` (native units) of a 6-decimal token whose `withdrawFeeX18 = 0`.
2. A third-party relayer calls `submitFastWithdrawal(idx, transaction, signatures)` with `msg.sender != sendTo`.
3. `markedIdxs[idx] = true` is set.
4. `fastWithdrawalFeeAmount` computes: `feeX18 = max(FAST_WITHDRAWAL_FEE_RATE * 500e12, 0)`. Even if `FAST_WITHDRAWAL_FEE_RATE` is 10 bps (1e14), `feeX18 = 1e14 * 500e12 / 1e18 = 5e10 < 1e12`, so `fee = 5e10 / 1e12 = 0`.
5. `safeTransferFrom(token, msg.sender, 0)` is called.
6. The token reverts on zero-value transfer → entire tx reverts.
7. `idx` is permanently marked; the fast withdrawal is bricked. [6](#0-5)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
