Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of End User, Enabling Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` — and therefore the `sender` forwarded to the extension — is the router contract, not the originating user. A pool admin who allowlists the router address to permit legitimate users to trade through it inadvertently grants every user on-chain the ability to bypass the curated pool's access control.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    zeroForOne,
    amountSpecified,
    priceLimitX64,
    packedSlot0Initial,
    bidPriceX64,
    askPriceX64,
    extensionData
);
```

**`SwapAllowlistExtension.beforeSwap` checks that `sender` argument:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (used as the mapping key) and `sender` is the value forwarded from the pool — the router's address.

**Router call path — `MetricOmmSimpleRouter.exactInputSingle`:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
IMetricOmmPoolActions(params.pool).swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
);
```

The router calls `pool.swap()` directly; inside that call `msg.sender` is the router. The originating user's address is never forwarded to the pool or the extension.

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses.
2. Admin calls `setAllowedToSwap(pool, routerAddress, true)` so that allowlisted users can trade conveniently through the public router.
3. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the restricted pool.
4. The pool receives `msg.sender = routerAddress`, passes it as `sender` to `_beforeSwap`, and the extension finds `allowedSwapper[pool][routerAddress] == true` — the check passes.
5. The non-allowlisted user's swap executes successfully, bypassing the intended access control.

**Existing guards are insufficient:** The `onlyPool` modifier on `beforeSwap` only verifies the caller is a registered pool; it does not recover the true originating user. There is no mechanism in the current architecture to thread the original `tx.origin` or a signed identity through the router into the extension check.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control primitive for curated pools. Its bypass allows any unprivileged user to trade on a pool the admin explicitly restricted. This breaks the pool's intended access model and can expose restricted liquidity to unauthorized counterparties, undermining the pool admin's ability to enforce curated-pool invariants. This constitutes broken core pool functionality (allowlist gate) reachable by any unprivileged trader.

## Likelihood Explanation
The precondition — admin allowlisting the router — is the natural and expected operational step for any pool that wants to support router-based trading while maintaining an allowlist. The bypass requires no special privilege: any Ethereum address can call the public router functions. The attack is repeatable on every swap and requires no capital beyond the swap input amount.

## Recommendation
Pass the true originating user through the call chain rather than relying on `msg.sender` of `pool.swap()`. Options include:
- Have the router encode the originating `msg.sender` in `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it (requires a trusted router registry or signature scheme to prevent spoofing).
- Introduce a dedicated `swapFor(address onBehalfOf, ...)` pool entry point that records the true user and passes it as `sender` to extensions, with the router calling that variant.
- Check `tx.origin` as a fallback identity in the extension (acceptable only if flash-loan and contract-caller risks are acceptable to the pool admin).

## Proof of Concept
```solidity
// Foundry test sketch
function test_allowlistBypassViaRouter() public {
    // 1. Pool admin allowlists the router so legitimate users can trade
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // 2. Non-allowlisted attacker calls the router
    address attacker = makeAddr("attacker");
    deal(address(token0), attacker, 1e18);
    vm.startPrank(attacker);
    token0.approve(address(router), type(uint256).max);

    // 3. Swap succeeds — attacker bypasses the allowlist
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: type(uint256).max,
        priceLimitX64: 0,
        extensionData: ""
    }));
    vm.stopPrank();
    // Assert: swap succeeded despite attacker not being in allowedSwapper
}
```
