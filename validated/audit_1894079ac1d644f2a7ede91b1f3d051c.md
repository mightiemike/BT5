### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Wrong Transaction Hash - (File: crates/apollo_protobuf/src/converters/transaction.rs)

### Summary
The protobuf deserializer for `ValidResourceBounds` silently converts an `AllResources` variant (with zero `l2_gas` and `l1_data_gas`) into the `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant, the transaction hash recomputed after P2P deserialization diverges from the hash the user signed and the hash stored in the block. This is a direct analog of the external "dual-effect" bug: one conversion step silently changes the accounting domain (hash preimage length), causing the downstream value (transaction hash) to be wrong.

### Finding Description

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` produces structurally different hashes depending on the `ValidResourceBounds` variant:

- `L1Gas(X)` → hashes `[tip, L1_GAS_packed, L2_GAS_packed]` (2 resource felts)
- `AllResources(X, 0, 0)` → hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` (3 resource felts, even when the third is zero) [1](#0-0) 

The protobuf deserializer at line 431 of `crates/apollo_protobuf/src/converters/transaction.rs` applies the following heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

A V3 transaction submitted via the gateway with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is stored internally as `ValidResourceBounds::AllResources`. Its hash H1 is computed with 3 resource felts. When this transaction is serialized to protobuf and deserialized on a peer (P2P sync path), the heuristic fires and produces `ValidResourceBounds::L1Gas(X)`. The hash H2 recomputed from this deserialized form uses only 2 resource felts, so H2 ≠ H1.

The same pattern exists in the RPC-layer conversion: [3](#0-2) 

The `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` is called after deserialization to produce the stored `tx_hash`: [4](#0-3) 

So the `InternalRpcTransaction` on the receiving node carries H2 as its `tx_hash`, while the user's signature covers H1.

### Impact Explanation

During block validation by a peer node (consensus or P2P sync), the blockifier executes the transaction using the `tx_hash` field from `InternalRpcTransaction`. The account contract's `__validate__` entry point receives H2 as the transaction hash and verifies the user's ECDSA signature against it. Because the user signed H1, signature verification fails and the transaction reverts. The proposer executed the same transaction successfully (using H1), so the two nodes produce divergent execution results for the same accepted input.

This matches the Critical impact scope: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

Additionally, if `validate_transaction_hash` is invoked during sync, the mismatch between the stored hash H1 and the recomputed hash H2 causes valid blocks to be rejected outright, matching the High impact scope: **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

### Likelihood Explanation

V3 transactions with zero `l2_gas` and `l1_data_gas` are common in practice (many real transactions on Starknet 0.13.x set only L1 gas bounds). Any such transaction propagated via P2P consensus or sync triggers the bug. No special privileges are required; any user submitting a standard V3 transaction with only L1 gas bounds is affected.

### Recommendation

Remove the `L1Gas`/`AllResources` heuristic from the protobuf deserializer. For V3 transactions, always produce `ValidResourceBounds::AllResources`. The `L1Gas` variant should only be used when deserializing pre-0.13.3 transactions that were originally signed under the two-resource-felt hash formula. A version gate (e.g., checking the transaction version field) is the correct discriminant, not the zero-value of optional fields.

```rust
// Correct: always AllResources for V3 transactions
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Apply the same fix to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`.

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: GasAmount(1000)/GasPrice(1), l2_gas: 0/0, l1_data_gas: 0/0 }`.
2. Compute H1 = `get_invoke_transaction_v3_hash(tx, chain_id, version)` — this hashes 3 resource felts.
3. Sign H1 with the account's private key.
4. Serialize the transaction to protobuf (`protobuf::ResourceBounds { l1_gas: ..., l2_gas: None/zero, l1_data_gas: None/zero }`).
5. Deserialize via `ValidResourceBounds::try_from(protobuf_resource_bounds)` — the heuristic fires, producing `L1Gas(l1_gas)`.
6. Compute H2 = `get_invoke_transaction_v3_hash(deserialized_tx, chain_id, version)` — this hashes only 2 resource felts.
7. Assert H1 ≠ H2. The receiving node stores H2; the account contract verifies the signature against H2 and rejects it, while the proposer accepted it under H1.

        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)
        } else {
