### Title
`getCoreRisk` Hardcodes `amount=1` in `_getWeightX18`, Causing Wrong Directional Weight for Short Positions in Spread Health Calculations — (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk` passes the literal `1` instead of the actual position `amount` to `RiskHelper._getWeightX18`. Because `1 >= 0` always evaluates true, the function unconditionally returns `longWeightInitialX18` (or `longWeightMaintenanceX18`) regardless of whether the subaccount holds a short position. `Clearinghouse.getHealth` then uses this wrong weight in both `existingWeight` and `_getSpreadWeightX18`, producing materially incorrect spread health adjustments.

---

### Finding Description

In `BaseEngine.getCoreRisk`:

```solidity
// core/contracts/BaseEngine.sol, line 190
RiskHelper._getWeightX18(risk, 1, healthType)   // ← hardcoded 1, should be `amount`
``` [1](#0-0) 

`_getWeightX18` branches on the sign of its second argument:

```solidity
if (amount >= 0) {          // 1 >= 0 → always true
    weight = longWeightInitialX18 / longWeightMaintenanceX18;
} else {
    weight = shortWeightInitialX18 / shortWeightMaintenanceX18;  // never reached
}
``` [2](#0-1) 

The product configuration enforces `longWeightMaintenance ≤ 10^9` (≤ 1) and `shortWeightMaintenance ≥ 10^9` (≥ 1): [3](#0-2) 

So for any short position, `getCoreRisk.longWeight` is always `longWeightInitialX18` (e.g. 0.9) when it should be `shortWeightInitialX18` (e.g. 1.1).

`Clearinghouse.getHealth` consumes this field in two places:

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight   = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price).mul(spreadWeight - existingWeight);
``` [4](#0-3) 

`_getSpreadWeightX18` also reads `perpCoreRisk.longWeight` and `spotCoreRisk.longWeight` directly: [5](#0-4) 

---

### Impact Explanation

**Case 1 — Long spot + Short perp (typical basis trade):**

| | Correct | Buggy |
|---|---|---|
| `perpCoreRisk.longWeight` | `shortWeightInitialX18` ≈ 1.1 | `longWeightInitialX18` ≈ 0.9 |
| `existingWeight` | (0.9 + 1.1)/2 = **1.0** | (0.9 + 0.9)/2 = **0.9** |
| `spreadWeight` | 1 − (1−1.1)/5 = 1.02 → capped **0.99** | 1 − (1−0.9)/5 = **0.98** |
| `spreadWeight − existingWeight` | 0.99 − 1.0 = **−0.01** | 0.98 − 0.9 = **+0.08** |
| Health delta (basisAmount > 0) | small **penalty** | large **bonus** |

The spread health bonus is inflated by ~9× the correct value. A user holding a long spot + short perp spread can withdraw collateral that should be locked, violating the withdrawal invariant.

**Case 2 — Short spot + Long perp:**

`basisAmount < 0`. The buggy `spreadWeight − existingWeight = +0.08` produces a negative health delta, while the correct value (`−0.01`) produces a positive one. Health is understated, enabling premature liquidation of a solvent account.

---

### Likelihood Explanation

- The spread loop in `getHealth` runs for every configured spread pair on every health check (withdrawal, liquidation, `_isAboveInitial`).
- Any subaccount holding a spread position (long spot + short perp or vice versa) is affected on every health evaluation.
- No special privileges or sequencer access are required; a normal user opening a basis trade triggers the bug automatically.
- The magnitude of the error scales with position size and the gap between `longWeightInitialX18` and `shortWeightInitialX18`.

---

### Recommendation

Replace the hardcoded `1` with the actual `amount` in `getCoreRisk`:

```solidity
// core/contracts/BaseEngine.sol
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
-   RiskHelper._getWeightX18(risk, 1, healthType)
+   RiskHelper._getWeightX18(risk, amount, healthType)
);
``` [6](#0-5) 

---

### Proof of Concept

1. Deploy the protocol with a spread pair `(spotId=1, perpId=2)`.
2. Open a long spot position (`amount = +100`) and a short perp position (`amount = −100`) for `subaccountA`.
3. Call `perpEngine.getCoreRisk(subaccountA, 2, INITIAL)`.
4. Observe: returned `longWeight == longWeightInitialX18` (e.g. 0.9×10^18), not `shortWeightInitialX18` (e.g. 1.1×10^18).
5. Call `clearinghouse.getHealth(subaccountA, INITIAL)`.
6. Observe: `existingWeight = 0.9` and `spreadWeight − existingWeight = +0.08`, producing a large positive health bonus.
7. Attempt `withdrawCollateral` for an amount that would be rejected under correct health accounting — the call succeeds.
8. Assert that the withdrawn amount exceeds what is permissible under correct weight semantics. [7](#0-6) [8](#0-7)

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
