### Title
Pending-Reorg Race Condition Allows False-Positive SPV Proof Result for Orphaned Transactions — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` evaluate mainchain membership and confirmation depth as a point-in-time snapshot. An attacker who controls (or observes) a pending `submit_blocks` transaction that will trigger a chain reorganization can order their verification call before the reorg executes, obtaining a `true` SPV result for a transaction whose containing block will be orphaned moments later.

---

### Finding Description

Both verification entry points delegate to the same core check in `verify_transaction_inclusion`:

```rust
let target_block_height = self
    .mainchain_header_to_height
    .get(&args.tx_block_blockhash)
    .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

Both checks — mainchain membership via `mainchain_header_to_height` and confirmation depth relative to `mainchain_tip_blockhash` — are evaluated against the state at the instant the call executes. There is no mechanism to account for a concurrently pending `submit_blocks` transaction that will alter that state.

When `submit_blocks` processes a fork with higher cumulative `chain_work`, `reorg_chain` is invoked:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [2](#0-1) 

`reorg_chain` removes the previously-canonical blocks from both `mainchain_header_to_height` and `headers_pool` via `remove_block_header`:

```rust
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash);
}
``` [3](#0-2) 

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [4](#0-3) 

The invariant that `verify_transaction_inclusion` is supposed to enforce — "this transaction is in a block that is, and will remain, canonical" — is broken because the function only checks the first half of that invariant.

`verify_transaction_inclusion_v2` is equally affected because it delegates to `verify_transaction_inclusion` after its coinbase check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [5](#0-4) 

---

### Impact Explanation

**Impact: Medium**

Any downstream NEAR contract that calls `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` to gate an irreversible action (e.g., releasing bridged assets, minting tokens, settling a payment) can be made to act on a `true` result for a transaction that is no longer in the canonical Bitcoin chain. The corrupted value is the boolean proof result returned by the SPV API — the primary security guarantee this contract provides to consumers.

---

### Likelihood Explanation

**Likelihood: Low-Medium**

The most realistic trigger is an attacker who controls both sides of the race:

1. The attacker pre-builds a competing Bitcoin fork with higher cumulative `chain_work` (valid PoW required, but the attacker is already doing this as part of a Bitcoin double-spend).
2. The attacker submits a NEAR cross-contract call to `verify_transaction_inclusion` for the target transaction (in the block about to be orphaned).
3. The attacker then submits `submit_blocks` with the competing fork headers.
4. Because both transactions originate from the attacker's account, NEAR processes them in submission order — verification first, reorg second.

A secondary path exists if the attacker merely observes a legitimate relayer's pending `submit_blocks` and races their own verification call ahead of it. NEAR has no gas-price auction, making this harder but not impossible within the same block's transaction ordering.

---

### Recommendation

1. **Require a post-submission finality window**: Record the NEAR block height at which each Bitcoin block was first accepted into the mainchain. `verify_transaction_inclusion` should require that `env::block_height() >= acceptance_height + NEAR_FINALITY_BUFFER` before returning `true`, ensuring the mainchain state has stabilized.

2. **Expose a "known competing fork" flag**: If `headers_pool` contains a fork whose tip `chain_work` is within a configurable threshold of the mainchain tip, treat the chain as "contested" and refuse verification until the contest resolves.

3. **Downstream guidance**: Document that callers must not treat a single `true` result as final; they should re-verify after a NEAR-level delay or use the `confirmations` parameter conservatively enough to exceed any plausible reorg depth.

---

### Proof of Concept

```
Bitcoin state (known to attacker):
  Mainchain:  A → B → C → D → E   (block B contains TX the attacker wants verified)
  Secret fork: A → B' → C' → D' → E' → F'  (higher chain_work than E)

NEAR contract state before attack:
  mainchain_tip_blockhash = hash(E)
  mainchain_header_to_height contains hash(B) → height(B)

Attack sequence (attacker controls transaction ordering):
  Step 1: Attacker submits NEAR tx₁:
            verify_transaction_inclusion_v2({ tx_id: TX, tx_block_blockhash: hash(B),
                                              confirmations: 4, ... })
          → Executes against current state
          → hash(B) found in mainchain_header_to_height ✓
          → height(E) - height(B) + 1 = 4 ≥ 4 ✓
          → Merkle proof valid ✓
          → Returns true  ← FALSE POSITIVE

  Step 2: Attacker submits NEAR tx₂:
            submit_blocks([B', C', D', E', F'])
          → chain_work(F') > chain_work(E) → reorg_chain() fires
          → hash(B) removed from mainchain_header_to_height and headers_pool
          → mainchain_tip_blockhash = hash(F')

  Result: Downstream contract received `true` for TX in block B,
          which is now orphaned and no longer part of the canonical chain.
```
