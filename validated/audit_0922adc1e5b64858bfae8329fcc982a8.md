### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Mitigation Introduced in `verify_transaction_inclusion_v2` — (`File: contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a coinbase-proof check to block the 64-byte transaction Merkle forgery attack. However, the old function is still declared `pub` and is fully reachable by any unprivileged NEAR caller. Rust's `#[deprecated]` attribute is a compiler-only hint; it does not restrict on-chain method dispatch. An attacker can call the old entry-point directly, skip the coinbase guard entirely, and obtain a `true` proof result for a fabricated transaction.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced to close the 64-byte Merkle forgery vector documented at https://www.bitmex.com/blog/64-Byte-Transactions. Its guard requires:

1. `coinbase_merkle_proof.len() == merkle_proof.len()` — same tree depth.
2. A valid coinbase proof that anchors the tree depth to a real coinbase transaction. [1](#0-0) 

Only after both checks pass does it delegate to the old function: [2](#0-1) 

The old function, however, is still `pub` and carries only a Rust-level `#[deprecated]` annotation: [3](#0-2) 

`#[deprecated]` emits a compiler warning to Rust callers; it has zero effect on NEAR's method-dispatch layer. Any NEAR account can submit a transaction that calls `verify_transaction_inclusion` directly. The function itself even documents the risk in its own warning comment: [4](#0-3) 

The Merkle proof computation in `compute_root_from_merkle_proof` is a pure hash-chain walk with no structural validation: [5](#0-4) 

Because Bitcoin's Merkle tree is built by double-SHA256-ing 64-byte concatenations of child hashes, an attacker can craft a 64-byte byte string that is simultaneously a valid serialisation of an internal node and a plausible "transaction ID". Feeding that value as `tx_id` with a proof that terminates one level above the real leaf layer causes `compute_root_from_merkle_proof` to return the correct `merkle_root`, making the function return `true` for a transaction that was never in the block.

---

### Impact Explanation

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged funds, minting tokens, settling a payment) can be deceived into accepting a forged inclusion proof. The corrupted value is the boolean proof result: the function returns `true` for a transaction that does not exist in the claimed block. This is a proof-verification forgery with direct financial impact on any consumer of the API.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` via a standard NEAR RPC `call` transaction. The 64-byte forgery technique is publicly documented and has known tooling. The only prerequisite is that the attacker knows a valid `tx_block_blockhash` that is on the current mainchain — information that is freely readable from `get_block_hash_by_height` or `get_last_n_blocks_hashes`. [6](#0-5) 

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or replace it with a `pub(crate)` / private modifier so it is no longer dispatchable as a NEAR contract method. Because `verify_transaction_inclusion_v2` already delegates to it internally, the internal call path can remain. Alternatively, add an explicit `env::panic_str` guard at the top of `verify_transaction_inclusion` that aborts any direct external call, preserving the internal delegation path only.

---

### Proof of Concept

**Analogy to the Sui report:**

| Sui | BTC Light Client |
|---|---|
| `Bar` without `key` → verifier skips check | `verify_transaction_inclusion` without coinbase guard → forgery check skipped |
| Upgrade adds `key` to `Bar` | `verify_transaction_inclusion_v2` adds coinbase guard |
| Old module version still callable | Old function still `pub` and dispatchable |
| Pass object from old version into new version | Call old entry-point directly, bypassing new guard |

**Attack steps:**

1. Identify any confirmed block hash `B` on the mainchain via `get_block_hash_by_height(h)`.
2. Construct a 64-byte value `fake_tx` that equals the concatenation of two real leaf hashes at depth `d-1` of the Merkle tree in block `B`. Its double-SHA256 equals the internal node at depth `d-1`.
3. Build a `merkle_proof` of length `d-1` (one level shorter than a real leaf proof) that walks from that internal node up to the `merkle_root`.
4. Call `verify_transaction_inclusion` directly with `tx_id = fake_tx`, `tx_block_blockhash = B`, `tx_index` chosen to match the position of the internal node, and the crafted `merkle_proof`.
5. `compute_root_from_merkle_proof` returns the correct `merkle_root`; the function returns `true`.
6. Any contract that trusted this result to release funds or confirm a cross-chain event is now deceived. [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
