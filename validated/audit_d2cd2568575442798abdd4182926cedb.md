Audit Report

## Title
Missing Deadline Guard in `addLiquidityExactShares` Allows Stale-Price Liquidity Deposits — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts no `deadline` parameter and performs no time-based expiry check before executing a liquidity deposit. `MetricOmmSimpleRouter` guards every swap entry point with `_checkDeadline` from `MetricOmmSwapRouterBase`, but the liquidity-adder path has no equivalent. A transaction that sits in the mempool and executes after the oracle price has moved forces the LP to deposit at a price they never accepted, causing immediate impermanent loss bounded only by the caller-supplied token caps.

## Finding Description

`MetricOmmSwapRouterBase._checkDeadline` (line 91–94) reverts when `block.timestamp > deadline`. Every swap entry in `MetricOmmSimpleRouter` — `exactInputSingle` (line 68), `exactInput` (line 93), `exactOutputSingle` (line 131), `exactOutput` (line 155) — calls this guard as its first statement.

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (lines 56–81) has no such guard. Its only pre-execution checks are:
- `_validateOwner` — rejects `address(0)` owner
- `_validateDeltas` — checks array length consistency
- `maxAmountToken0` / `maxAmountToken1` caps enforced inside the callback (line 165)

None of these checks are time-sensitive. The function immediately calls `_addLiquidity` → `IMetricOmmPoolActions(pool).addLiquidity(...)`, which settles token amounts based on the oracle price at execution time, not at submission time.

**Exploit flow:**
1. LP calls `addLiquidityExactShares` with shares targeting bins around the current oracle price P1, setting `maxAmountToken0` and `maxAmountToken1` generously to accommodate expected amounts.
2. Transaction is delayed in the mempool (network congestion, MEV, etc.).
3. Oracle price moves to P2 (e.g., token0 depreciates relative to token1).
4. Transaction executes: the pool computes token amounts at P2. The LP deposits more of the now-cheaper token0 than intended, receiving shares at the unfavorable price.
5. The callback check at line 165 only verifies `amount0Delta <= max0 && amount1Delta <= max1` — it does not check whether the price or composition is acceptable.

`addLiquidityWeighted` has partial mitigation via `_validateBinAndBinPosition` (lines 263–286), which reverts if the active cursor has moved outside caller-supplied bin bounds. However, `addLiquidityExactShares` has no equivalent cursor or time check.

## Impact Explanation
An LP depositing via `addLiquidityExactShares` can be forced to deposit at a stale oracle price, receiving shares at an unfavorable token composition. The immediate impermanent loss is bounded by the max token caps but can be substantial if caps are set generously (as is typical). This constitutes direct loss of LP principal through bad-price execution on a core liquidity-provision path.

## Likelihood Explanation
No attacker capability is required — the vulnerability is triggered by ordinary mempool delay or network congestion. Any LP using `addLiquidityExactShares` during volatile market conditions is at risk. The condition is repeatable and requires no privileged access.

## Recommendation
Add a `uint256 deadline` parameter to both `addLiquidityExactShares` overloads and call `_checkDeadline(deadline)` as the first statement, mirroring the pattern in `MetricOmmSwapRouterBase`. Alternatively, inherit from `MetricOmmSwapRouterBase` or extract `_checkDeadline` into a shared base used by both the router and the liquidity adder.
