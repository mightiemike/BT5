### Title
`DepositAllowlistExtension` checks position `owner` instead of actual token payer, allowing non-allowlisted users to deposit into curated pools — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` gates deposits by checking the `owner` (position recipient) against the allowlist. `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` explicitly separates the position `owner` from the token payer (`msg.sender`). A non-allowlisted user can therefore bypass the deposit guard entirely by setting `owner` to any allowlisted address while remaining the actual payer of tokens into the pool.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` receives two actor addresses from the pool: `sender` (the immediate caller of `pool.addLiquidity`, i.e. the LiquidityAdder contract) and `owner` (the position recipient). The hook silently discards `sender` and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol line 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the owner-explicit overload) accepts an arbitrary `owner` address and passes `msg.sender` as the separate `payer`:

The pool's `addLiquidity` then calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the LiquidityAdder contract and `owner` is the caller-supplied position recipient:

```solidity

``` [4](#0-3) 

`ExtensionCalling._beforeAddLiquidity` forwards both `sender` and `owner` to the extension:

```solidity
// ExtensionCalling.sol line 88-99
function _beforeAddLiquidity(address sender, address owner, ...) internal {
    _callExtensionsInOrder(
        BEFORE_ADD_LIQUIDITY_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
    );
}
``` [5](#0-4) 

The extension ignores `sender` entirely and checks only `owner`. Because `owner` is fully caller-controlled via the LiquidityAdder's explicit-owner overload, the guard is misbound to the wrong actor.

The callback then pulls tokens from the actual payer (`msg.sender` of the LiquidityAdder call), not from `owner`:

```solidity

The same flaw applies to `addLiquidityWeighted` with an explicit `owner`, which also passes `owner` to the probe and paying `addLiquidity` calls while `msg.sender` remains the payer. [7](#0-6) 

---

### Impact Explanation

The deposit allowlist is completely bypassed. Any non-allowlisted address can inject tokens into a curated pool by supplying an allowlisted address as `owner`. The LP shares are minted to the allowlisted address (a forced, unconsented deposit), while the attacker's tokens enter the pool. The pool admin's curation policy — the only mechanism preventing unauthorized liquidity from entering the pool — is rendered ineffective. If the attacker controls the allowlisted address (e.g., a compromised or colluding account), they can also recover the LP shares by calling `removeLiquidity` from that address, completing a full round-trip deposit-and-withdraw cycle through a pool they were explicitly barred from.

---

### Likelihood Explanation

The `addLiquidityExactShares(pool, owner, ...)` overload is a standard, documented, publicly callable function on the production periphery contract. No special privileges are required. The only prerequisite is knowing one allowlisted address for the target pool, which is discoverable from on-chain events (`AllowedToDepositSet`). The bypass requires a single transaction.

---

### Recommendation

The `beforeAddLiquidity` hook must gate the economically relevant actor — the entity whose tokens enter the pool — not the position recipient. Two complementary fixes:

1. **Pass the payer to the hook.** Extend the `beforeAddLiquidity` interface to include a `payer` parameter and have the pool (or LiquidityAdder via `callbackData`) supply it. The extension then checks `payer` instead of `owner`.

2. **Short-term mitigation.** In `DepositAllowlistExtension.beforeAddLiquidity`, check `sender` (the first parameter, currently ignored) in addition to or instead of `owner`. While `sender` is the LiquidityAdder address rather than the end user, this at minimum prevents the bypass through the LiquidityAdder and forces direct pool calls, where `msg.sender == owner` is the natural constraint.

---

### Proof of Concept

```
Setup:
  Pool deployed with DepositAllowlistExtension.
  Admin calls: depositExtension.setAllowedToDeposit(pool, alice, true)
  Bob (not allowlisted) holds token0 and token1 and has approved the LiquidityAdder.

Attack (single transaction):
  Bob calls:
    liquidityAdder.addLiquidityExactShares(
        pool,
        alice,      // owner = allowlisted address
        salt,
        deltas,
        max0, max1,
        ""
    )

Execution trace:
  1. LiquidityAdder._addLiquidity sets payContext: payer = Bob
  2. pool.addLiquidity(owner=alice, ...) is called
  3. pool._beforeAddLiquidity(sender=LiquidityAdder, owner=alice, ...)
  4. DepositAllowlistExtension checks allowedDepositor[pool][alice] → true → no revert
  5. LiquidityLib mints LP shares to alice
  6. pool calls LiquidityAdder.metricOmmModifyLiquidityCallback
  7. Callback pulls tokens from Bob (payer) into the pool

Result:
  Bob's tokens are in the pool. Alice holds LP shares she did not request.
  The deposit allowlist has been bypassed without Bob being allowlisted.
  If Bob controls alice, he calls pool.removeLiquidity(alice, ...) to recover tokens.
```
