Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` gates the router (intermediary) instead of the end user, enabling allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` at the pool, so the extension checks `allowedSwapper[pool][router]` rather than the identity of the end user. Any pool admin who allowlists the router (the natural configuration for permissioned pools that want to support router-mediated swaps) inadvertently grants every address on the network the ability to swap against the pool, defeating the allowlist entirely.

## Finding Description
`SwapAllowlistExtension.beforeSwap()` performs the following check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool — which is `msg.sender` of `pool.swap()`. [2](#0-1) 

`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap()`, and `_beforeSwap()` encodes it as the first argument to the extension call: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly with no mechanism to forward the originating user's identity: [4](#0-3) 

The pool therefore records `msg.sender = router` and passes it as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`, not the identity of the end user.

**The bypass:** A pool admin who wants allowlisted users to use the router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, `allowedSwapper[pool][router] == true` for every call arriving through the router — regardless of who the actual caller is. Any unprivileged address can call `router.exactInputSingle()` and the extension will pass.

**Contrast with `DepositAllowlistExtension`:** The deposit allowlist correctly gates the position `owner` (the economic beneficiary), not `sender` (the intermediary). The pool's `addLiquidity()` accepts `owner` as an explicit argument separate from `msg.sender`, so the extension always sees the correct identity: [5](#0-4) 

The swap interface has no equivalent "on-behalf-of" field — `pool.swap()` only exposes `recipient` (output destination) and `msg.sender` (the intermediary paying input tokens). There is no structural way for the router to forward the originating user's identity without an explicit protocol-level field or a trusted `extensionData` encoding convention.

The existing unit tests for `SwapAllowlistExtension` only test direct calls from the pool address and never exercise the router-mediated path: [6](#0-5) 

## Impact Explanation
A pool admin who deploys a permissioned pool (e.g., for institutional LPs, KYC'd counterparties, or a closed market-making arrangement) and allowlists the router loses all swap-side access control. Any address on the network can swap against the pool's liquidity at oracle-derived prices. This constitutes a direct loss of LP principal whenever the oracle price differs from the market price — which is the normal operating condition. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break" impact criteria.

## Likelihood Explanation
Allowlisting the router is the natural and expected configuration for any pool that wants allowlisted users to enjoy the router's slippage protection, deadline checks, and multi-hop routing. A pool admin who does not allowlist the router forces allowlisted users to call `pool.swap()` directly, losing all periphery conveniences. The bypass therefore activates under a common, well-motivated admin configuration requiring no special attacker capability — any EOA can call `router.exactInputSingle()`.

## Recommendation
The swap interface should be extended with an explicit "on-behalf-of" / originator field analogous to the `owner` field in `addLiquidity`, so the extension always sees the real actor regardless of intermediary. Alternatively, `SwapAllowlistExtension` should gate the `recipient` argument (the address receiving output tokens) rather than `sender`, since `recipient` is set by the end user and passed through the router unchanged. A third option is to require the router to encode the originating user's address in `extensionData` and have the extension decode and verify it — though this requires the extension to trust the router's claim and introduces a coupling between the extension and specific router implementations.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` attached to `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, userA, true)` and `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
3. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
4. Router calls `pool.swap(userB, zeroForOne, amount, ...)` — `msg.sender` at the pool is `router`.
5. Pool calls `_beforeSwap(sender=router, recipient=userB, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
6. `userB`'s swap executes at oracle prices against the pool's LP liquidity, bypassing the allowlist entirely.

A Foundry integration test can reproduce this by: deploying a pool with `SwapAllowlistExtension`, allowlisting only `userA` and the router, then pranking as `userB` to call `router.exactInputSingle()` and asserting the swap succeeds (no `NotAllowedToSwap` revert) and token balances change for `userB`.
