### Title
Variable-Length `peer_id` Concatenation Without Length Delimiter Creates Non-Canonical Signature Preimage in Identity Verification - (File: `crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary

`build_peer_identity_message_digest` concatenates a variable-length `peer_id.to_bytes()` field directly against a fixed-length `challenge` field with no length delimiter between them. Two distinct `(peer_id, challenge)` pairs whose raw byte concatenations are identical will produce the same `MessageDigest` and therefore the same ECDSA signing target. A signature produced for one pair is cryptographically valid for the other, breaking the one-to-one binding between a signed identity claim and its signer.

### Finding Description

In `crates/apollo_signature_manager/src/signature_manager.rs`, the function `build_peer_identity_message_digest` constructs the preimage for ECDSA identity signing as:

```
INIT_PEER_ID  ||  peer_id.to_bytes()  ||  challenge.0
``` [1](#0-0) 

`peer_id.to_bytes()` returns the libp2p multihash encoding of the peer's public key. This encoding is **not fixed-length**: an Ed25519 identity-multihash PeerId serialises to 39 bytes, while a SHA-256-hashed PeerId serialises to 34 bytes. No length prefix or field delimiter separates the `peer_id` bytes from the `challenge` bytes.

The codebase itself acknowledges this defect with an open TODO: [2](#0-1) 

Because the boundary between the two variable-length fields is implicit, the byte string

```
INIT_PEER_ID || peer_id_A(39 B) || challenge_A(32 B)
```

is identical to

```
INIT_PEER_ID || peer_id_B(43 B) || challenge_B(28 B)
```

whenever `peer_id_B.to_bytes() = peer_id_A.to_bytes() || challenge_A.0[0..4]` and `challenge_B.0 = challenge_A.0[4..]`. The resulting `MessageDigest` (a `Felt` produced by `blake2s_to_felt`) is the same for both inputs.

The signing path (`sign_identification`) and the verification path (`verify_identity`) both call `build_peer_identity_message_digest` and therefore share the same non-canonical preimage: [3](#0-2) [4](#0-3) 

### Impact Explanation

A valid ECDSA signature `S` produced by a sequencer node for `(peer_id_A, challenge_A)` is also a valid signature for any `(peer_id_B, challenge_B)` whose raw concatenation equals that of the original pair. An attacker who can observe a legitimate identity-verification exchange and who controls a PeerId whose byte encoding satisfies the boundary-shift equation can present the intercepted signature as their own identity proof, causing `verify_identity` to return `true` for the wrong signer. This binds the wrong peer identity to a verified signature, matching the "High – signature/hash logic binds the wrong signer" impact category.

### Likelihood Explanation

Exploitation requires the attacker to control a PeerId whose `to_bytes()` output satisfies a specific byte-boundary relationship with a target `(peer_id, challenge)` pair. In practice this means either (a) grinding key generation until the multihash prefix matches the required pattern, or (b) operating in a deployment where the challenge is predictable or attacker-influenced. The TODO comment confirms the developers consider this a real, unresolved canonicalization gap. Likelihood is moderate: the condition is non-trivial to satisfy but the code provides no structural defence against it.

### Recommendation

Replace the raw byte concatenation with a length-prefixed or structurally delimited encoding so that each field occupies an unambiguous, fixed-width slot in the preimage. For example, encode `peer_id` as a fixed 32-byte Starknet address (the `staker_address` the TODO already proposes as a replacement) or prepend a 4-byte big-endian length before the `peer_id` bytes. Alternatively, use a structured hash such as `Poseidon::hash_array(&[domain_tag, peer_id_felt, challenge_felt])` where every field is a canonical `Felt`, eliminating variable-length ambiguity entirely. The parallel `build_signed_payload` in `crates/apollo_propeller/src/signature.rs` uses fixed-width `[u8; 32]` fields for `MessageRoot` and `CommitteeId` and is not affected. [5](#0-4) 

### Proof of Concept

```
// Attacker controls two key pairs whose PeerId byte lengths differ by k bytes.
// Let peer_id_A.to_bytes() = [0x00, 0x25, ...] (39 bytes, Ed25519 identity multihash)
// Let challenge_A.0          = [c0, c1, ..., c31] (32 bytes)
//
// Attacker finds peer_id_B such that:
//   peer_id_B.to_bytes() = peer_id_A.to_bytes() ++ [c0, c1, c2, c3]  (43 bytes)
//   challenge_B.0        = [c4, c5, ..., c31]                         (28 bytes)
//
// Then:
//   INIT_PEER_ID || peer_id_A(39) || challenge_A(32)
// = INIT_PEER_ID || peer_id_B(43) || challenge_B(28)
//
// blake2s_to_felt produces the same Felt for both inputs.
// ecdsa_sign(sk_A, digest) is accepted by ecdsa_verify(pk_A, digest) AND
// by ecdsa_verify(pk_B, digest) if the attacker also holds sk_B — but more
// critically, the intercepted signature S for (peer_id_A, challenge_A) passes
// verify_identity(peer_id_B, challenge_B, S, pk_A) = true, binding pk_A's
// signature to peer_id_B's identity claim.
``` [1](#0-0) [6](#0-5)

fn build_peer_identity_message_digest(peer_id: PeerId, challenge: Challenge) -> MessageDigest {
    let challenge = &challenge.0;
    let peer_id = peer_id.to_bytes();
    let mut message = Vec::with_capacity(INIT_PEER_ID.len() + peer_id.len() + challenge.len());
    message.extend_from_slice(INIT_PEER_ID);
    message.extend_from_slice(&peer_id);
    message.extend_from_slice(challenge);

    MessageDigest(blake2s_to_felt(&message))
}
```
