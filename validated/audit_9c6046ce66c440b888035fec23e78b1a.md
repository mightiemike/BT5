### Title
Stale Mainchain Height Snapshot Used for Fork Difficulty Calculation Allows PoW Bypass — (`contract/src/dogecoin.rs`, `contract/src/litecoin.rs`, `contract/src/bitcoin.rs`)

---

### Summary

When validating a fork block's difficulty, `get_next_work_required` fetches the "first block in the adjustment interval" via `get_header_by_height`, which reads from the **current mainchain** height-to-hash mapping. For a fork that diverged before that height, the mainchain block at that height is a **different block** than the fork's actual ancestor. The timestamp from the wrong (stale mainchain) block is then used to compute the required difficulty, producing an incorrect target. An attacker who is a trusted relayer can exploit this to submit fork blocks that satisfy the incorrectly-computed (lower) difficulty but not the correct one, potentially corrupting the canonical chain tip.

---

### Finding Description

`get_header_by_height` is implemented in `contract/src/lib.rs`:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This always resolves a height to the **mainchain** block at that height, not the fork ancestor.

In `contract/src/dogecoin.rs`, `get_next_work_required` calls this function to obtain the timestamp of the first block in the difficulty window:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The same pattern appears in `contract/src/litecoin.rs` and `contract/src/bitcoin.rs`: [3](#0-2) [4](#0-3) 

The `calculate_next_work_required` function for Dogecoin computes:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [5](#0-4) 

If `first_block_time` is taken from the mainchain block at `height_first` rather than the fork's actual ancestor at that height, the computed `modulated_timespan` is wrong, producing an incorrect (potentially lower) required difficulty.

The developers themselves flagged this with a `TODO` comment at the exact call site, confirming awareness of the ambiguity.

---

### Impact Explanation

**Dogecoin (most severe):** Post-block 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. [6](#0-5) 

A fork diverging just **2 blocks back** causes `get_header_by_height(height_first)` to return the mainchain block at that height instead of the fork's ancestor. If the mainchain block at `height_first` has a timestamp `T_main` that is later than the fork block's timestamp `T_fork` at the same height, then:

- Correct `modulated_timespan = prev_time − T_fork` (larger)
- Actual `modulated_timespan = prev_time − T_main` (smaller → lower difficulty target)

The attacker mines fork blocks satisfying the incorrectly-computed lower difficulty. If the fork accumulates sufficient chainwork (possible because each block is easier to mine), `submit_block_header_inner` triggers `reorg_chain`, replacing `mainchain_tip_blockhash` with the attacker's fork tip. [7](#0-6) 

After the reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` verify transactions against the corrupted chain, returning `true` for transactions that never occurred on the real Dogecoin chain. [8](#0-7) 

**Bitcoin/Litecoin:** The same `get_header_by_height` call exists, but `difficulty_adjustment_interval = 2016`, so the fork must diverge more than 2016 blocks back for the stale snapshot to affect the boundary block's timestamp. This makes exploitation significantly harder but not impossible for a well-resourced attacker.

---

### Likelihood Explanation

`submit_blocks` is gated by `#[trusted_relayer]`, but the staking mechanism allows any account to become a trusted relayer by depositing stake. [9](#0-8) [10](#0-9) 

For Dogecoin, the attack requires only a 2-block fork divergence and mining blocks at a reduced difficulty — a realistic capability for any party already operating as a relayer. The `TODO` comment at the exact vulnerable call site confirms the developers identified this as an open correctness question, raising the likelihood that it is a genuine gap rather than an intentional design choice.

---

### Recommendation

Replace `get_header_by_height` (mainchain lookup) with an ancestor traversal that follows `prev_block_hash` links from the fork block being validated, analogous to how `zcash.rs` correctly uses `get_prev_header` in a loop: [11](#0-10) 

For Dogecoin and Litecoin/Bitcoin, walk back `blocks_to_go_back` steps from `prev_block_header` via `get_prev_header` to find the true fork ancestor at `height_first`, and use that block's timestamp for the difficulty calculation. This eliminates the stale mainchain snapshot and aligns the difficulty computation with the actual fork ancestry.

---

### Proof of Concept

1. Deploy the Dogecoin light client contract on NEAR testnet.
2. Stake to become a trusted relayer.
3. Initialize with a mainchain of blocks at heights K, K+1, K+2, … where the mainchain block at height K has timestamp `T_main = T_base + 120` (2 minutes after base).
4. Submit a fork block at height K+1 whose `prev_block_hash` points to the mainchain block at K (fork diverges here), with timestamp `T_fork_K1 = T_base + 60`.
5. Submit a fork block at height K+2 with timestamp `T_fork_K2 = T_base + 65`.
   - Correct `first_block_time` = fork block at K+1 = `T_base + 60`; correct `modulated_timespan = 5s` → very high difficulty.
   - Actual `first_block_time` = mainchain block at K+1 = `T_main = T_base + 120`; actual `modulated_timespan = T_base+65 − (T_base+120) = −55s` → clamped to `min_timespan` → minimum difficulty.
6. Mine the fork block at K+2 against the minimum difficulty (trivially easy).
7. The contract accepts it via `submit_block_header` because `check_pow` uses the stale mainchain timestamp.
8. Continue extending the fork; once chainwork exceeds the mainchain, `reorg_chain` fires and `mainchain_tip_blockhash` is set to the attacker's fork tip.
9. Call `verify_transaction_inclusion` with a fabricated transaction in the attacker's fork block — it returns `true`. [12](#0-11) [13](#0-12)

```rust
fn get_next_work_required(
    config: &DogecoinConfig,
    block_header: &Header,
    prev_block_header: &ExtendedHeader,
    blocks_getter: &impl BlocksGetter,
) -> u32 {
    // Dogecoin: Special rules for minimum difficulty blocks with Digishield
    if allow_min_difficulty_for_block(config, block_header, prev_block_header) {
        // Special difficulty rule for testnet:
        // If the new block's timestamp is more than 2* nTargetSpacing minutes
        // then allow mining of a min-difficulty block.
        return config.proof_of_work_limit_bits;
    }

    // Only change once per difficulty adjustment interval
    let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
    let difficulty_adjustment_interval = if new_difficulty_protocol {
        1
    } else {
        config.difficulty_adjustment_interval
    };

    if (prev_block_header.block_height + 1) % difficulty_adjustment_interval != 0 {
        if config.pow_allow_min_difficulty_blocks {
            // Special difficulty rule for testnet:
            // If the new block's timestamp is more than 2* 10 minutes
            // then allow mining of a min-difficulty block.
            if block_header.time
                > prev_block_header.block_header.time + config.pow_target_spacing * 2
            {
                return config.proof_of_work_limit_bits;
            }

            // Return the last non-special-min-difficulty-rules-block
            let mut current_block_header = prev_block_header.clone();

            while current_block_header.block_header.bits == config.proof_of_work_limit_bits
                && current_block_header.block_height % config.difficulty_adjustment_interval != 0
            {
                current_block_header =
                    blocks_getter.get_prev_header(&current_block_header.block_header);
            }

            return current_block_header.block_header.bits;
        }

        return prev_block_header.block_header.bits;
    }

    // Litecoin: This fixes an issue where a 51% attack can change difficulty at will.
    // Go back the full period unless it's the first retarget after genesis. Code courtesy of Art Forz
    let mut blocks_to_go_back = difficulty_adjustment_interval - 1;
    if prev_block_header.block_height + 1 != difficulty_adjustment_interval {
        blocks_to_go_back = difficulty_adjustment_interval;
    }

    // Go back by what we want to be 14 days worth of blocks
    let height_first = prev_block_header
        .block_height
        .checked_sub(blocks_to_go_back)
        .unwrap_or_else(|| env::panic_str("Height underflow when calculating first block height"));

    // TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
    let first_block_time = blocks_getter
        .get_header_by_height(height_first)
        .block_header
        .time;

    calculate_next_work_required(config, prev_block_header, i64::from(first_block_time))
