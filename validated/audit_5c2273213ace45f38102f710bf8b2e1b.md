### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (`contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (v1) is still a live, publicly callable NEAR entry point despite being deprecated. It lacks the coinbase Merkle proof check that `verify_transaction_inclusion_v2` introduced specifically to block the 64-byte transaction Merkle forgery attack. Any unprivileged NEAR caller can invoke v1 directly, bypassing the protection entirely and obtaining a `true` SPV result for a fabricated transaction.

### Finding Description
`verify_transaction_inclusion_v2` was introduced to close the 64-byte transaction Merkle forgery vulnerability (described at https://www.bitmex.com/blog/64-Byte-Transactions). It enforces a coinbase Merkle proof check before delegating to v1:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [1](#0-0) 

The v1 function, however, performs no such check. It computes a Merkle root directly from the caller-supplied `tx_id` and `merkle_proof` and returns `true` if it matches the stored block's `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

The `#[deprecated]` attribute in Rust is a compile-time lint only. It does not gate or remove the function from the NEAR ABI. The function carries `#[pause]` but no access-control restriction, so any unprivileged NEAR account can call it at runtime. [3](#0-2) 

The contract's own warning acknowledges the gap:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [4](#0-3) 

### Impact Explanation
An attacker who knows a valid Bitcoin block hash stored in the contract can craft a 64-byte input whose double-SHA256 hash equals an internal Merkle tree node. By supplying that hash as `tx_id` with a suitably crafted `merkle_proof`, they can make `verify_transaction_inclusion` return `true` for a transaction that was never included in the block. Any downstream NEAR contract that calls v1 to gate asset releases, bridge withdrawals, or other value-bearing operations will be deceived into accepting a forged proof.

### Likelihood Explanation
The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with arbitrary `ProofArgs`. The 64-byte preimage technique is publicly documented and has known tooling. Downstream integrators who have not yet migrated to v2 (or who call v1 programmatically) are directly exposed.

### Recommendation
Remove `verify_transaction_inclusion` from the public NEAR ABI entirely, or gate it with an access-control role that prevents unprivileged callers from invoking it. If backward compatibility must be preserved, at minimum add the same coinbase Merkle proof check that v2 performs, making v1 and v2 functionally equivalent and safe.

### Proof of Concept
1. Identify any block hash `B` stored in `mainchain_header_to_height` (publicly readable via `get_block_hash_by_height`).
2. Retrieve the block's `merkle_root` from `headers_pool`.
3. Construct a 64-byte blob whose double-SHA256 equals an internal Merkle node that, combined with a crafted `merkle_proof`, produces `merkle_root`.
4. Call `verify_transaction_inclusion` with `tx_id = <crafted hash>`, `tx_block_blockhash = B`, `tx_index = <matching index>`, `merkle_proof = <crafted proof>`, `confirmations = 1`.
5. The function returns `true` despite no real transaction existing at that position, because the coinbase anchor check present in v2 is absent here. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
