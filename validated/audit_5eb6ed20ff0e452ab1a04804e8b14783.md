### Title
`op_modpow` Cost Formula Uses Wrong Functional Form, Enabling Undercharged Execution — (`File: src/more_ops.rs`)

### Summary

The cost formula for `op_modpow` charges `esize² × 3 + msize² × 21` cost units, but the actual computational complexity of modular exponentiation is O(esize × msize²). At the maximum allowed operand sizes (256 bytes each), the formula undercharges by approximately 10×, allowing an attacker to consume significantly more CPU time than the cost limit intends to permit.

### Finding Description

In `src/more_ops.rs`, `op_modpow` computes cost as follows:

```rust
let (exponent, esize) = int_atom(a, exponent, "modpow")?;
cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;   // esize² × 3
check_cost(cost, max_cost)?;
let (modulus, msize) = int_atom(a, modulus, "modpow")?;
cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;        // msize² × 21
check_cost(cost, max_cost)?;

if bsize > 256 || esize > 256 || msize > 256 {
    return Err(EvalErr::InvalidOpArg(input, "modpow".to_string()));
}
```

The actual computational complexity of modular exponentiation using square-and-multiply (as used by `num_bigint`'s `modpow`) is **O(esize × msize²)**:
- O(esize) iterations — one per bit of the exponent
- Each iteration performs a Montgomery multiplication of numbers up to `msize` bytes — O(msize²) per operation

The formula uses the **sum of squares** (`esize² + msize²`) instead of the **product** (`esize × msize²`), omitting the cross-term entirely. This is the same class of error as the original report: using a metric that does not accurately represent the quantity being measured.

The benchmark tool confirms this mismatch. It measures the exponent cost with a fixed modulus (`number`, 32 bytes) and the modulus cost with a fixed exponent, so the cross-term `esize × msize²` is never captured:

```rust
Operator {
    opcode: 60,
    name: "modpow (exponent cost)",
    arg: Placeholder::ThreeArgs(Some(number), None, Some(number)),
    ...
},
```

At `esize = msize = 256` (the maximum allowed by the size guard):
- **Formula cost**: 256² × 3 + 256² × 21 = 1,572,864 cost units
- **Actual complexity**: O(256 × 256²) = O(16,777,216) — approximately **10× more expensive**

The identical formula is duplicated in `op_modpow_malachite`, so both execution paths are affected.

### Impact Explanation

The CLVM cost limit (typically 11 billion units) is the primary DoS protection for Chia nodes. Because the cost formula undercharges by ~10× at maximum operand sizes, an attacker can submit programs that consume ~10× more CPU time than the cost limit intends to allow. A node evaluating such a program would spend far more wall-clock time than the cost model predicts, degrading throughput or causing block validation delays. The corrupted invariant is the cost-to-CPU-time ratio that the cost model is supposed to maintain.

### Likelihood Explanation

`op_modpow` is reachable via the softfork guard from any submitted CLVM program. An attacker needs only to craft a program calling `modpow` with 256-byte exponent and 256-byte modulus operands. No privileged access is required. The attack is straightforward to construct and can be repeated across many transactions.

### Recommendation

Replace the independent quadratic terms with a formula that captures the cross-term:

```rust
// Replace:
cost += (esize * esize) as Cost * MODPOW_COST_PER_BYTE_EXPONENT;
cost += (msize * msize) as Cost * MODPOW_COST_PER_BYTE_MOD;

// With a formula proportional to esize × msize²:
cost += esize as Cost * (msize * msize) as Cost * MODPOW_COST_PER_BYTE_CROSS;
```

Re-calibrate `MODPOW_COST_PER_BYTE_CROSS` against benchmarks that vary both `esize` and `msize` simultaneously. Apply the same fix to `op_modpow_malachite`.

### Proof of Concept

A CLVM program submitted to a Chia node:

```
(softfork
  (q . 1600000)          ; declared cost — accepted by formula
  (q . 1)                ; softfork extension enabling modpow
  (q . (modpow
    <256-byte base>
    <256-byte exponent>  ; esize = 256
    <256-byte modulus>)) ; msize = 256
  (q . ()))
```

The formula charges ~1.6M cost units and the `check_cost` guard passes. The actual `num_bigint::modpow` call performs O(256 × 256²) ≈ 16.7M units of Montgomery multiplication work — approximately 10× more CPU time than the cost model accounts for. Repeating this pattern up to the 11B cost limit allows ~6,875 such calls, each performing the full 10× excess computation. [1](#0-0) [2](#0-1) [3](#0-2)
