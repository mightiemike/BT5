### Title
`ValidResourceBounds` Silently Downgraded from `AllResources` to `L1Gas` During P2P Protobuf Deserialization, Binding Wrong Transaction Hash Domain — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The P2P sync protobuf converter for `ValidResourceBounds` silently collapses an `AllResources` variant (which hashes three resource bounds: L1\_gas, L2\_gas, L1\_data\_gas) into an `L1Gas` variant (which hashes only two: L1\_gas, L2\_gas) whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` branches on the variant to decide how many fields enter the Poseidon preimage, the hash computed from the deserialized `Transaction` is structurally different from the hash the signer and gateway computed, breaking the canonicalization invariant that a transaction's hash is uniquely determined by its signed fields.

### Finding Description

**Root cause — protobuf deserialization silently changes the resource-bounds variant:** [1](#0-0) 

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← silently zero
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)                      // ← variant changed
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

When a peer sends a transaction whose `l1_data_gas` field is absent (old 0.13.2 wire format) or whose `l1_data_gas` and `l2_gas` are both zero, the receiver silently produces `L1Gas` even if the originating node computed the hash under `AllResources`.

**Hash domain branches on the variant:** [2](#0-1) 

```rust
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip) -> ... {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                    // 2-element preimage
        ValidResourceBounds::AllResources(all) =>
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?], // 3-element preimage
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

**Gateway always computes the hash under `AllResources`:** [3](#0-2) 

`InternalRpcInvokeTransactionV3::resource_bounds()` always returns `ValidResourceBounds::AllResources(self.resource_bounds)`, so the gateway hash is always the 3-element Poseidon hash.

**Concrete divergence for a transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`:**

| Side | Variant | Preimage elements | Hash |
|---|---|---|---|
| Gateway / signer | `AllResources` | `[tip, l1_gas, 0, 0]` | H₁ |
| P2P receiver | `L1Gas` (downgraded) | `[tip, l1_gas, 0]` | H₂ ≠ H₁ |

The extra zero felt for `l1_data_gas` in the `AllResources` path makes H₁ ≠ H₂ even though all numeric values are identical.

### Impact Explanation

Any component that recomputes the transaction hash from the deserialized `Transaction` struct after P2P sync — including `validate_transaction_hash`, re-execution, or proof-input construction — will produce H₂ while the stored/expected hash is H₁. This binds the wrong hash to the transaction, matching the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."* [4](#0-3) 

### Likelihood Explanation

Any unprivileged user can submit a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` through the normal gateway endpoint. The gateway accepts it (no check enforces non-zero L2/data gas). The transaction is then propagated via P2P mempool or included in a synced block. The downgrade fires deterministically on every receiving node running the P2P sync path. [5](#0-4) 

### Recommendation

Replace the heuristic variant-selection logic with an explicit, lossless encoding. The protobuf wire format should carry a boolean or enum flag indicating whether the transaction was signed under `AllResources` or `L1Gas`, independent of whether the numeric values happen to be zero. Until the 0.13.2 compatibility window is closed, the deserializer should default to `AllResources` (not `L1Gas`) when `l1_data_gas` is absent, because all new transactions use `AllResources` and the hash domain must be preserved:

```rust
// Safer default: preserve AllResources semantics for new transactions
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

Alternatively, store the variant tag explicitly in the protobuf message so the receiver never has to infer it from field values.

    type Error = ProtobufConversionError;

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
