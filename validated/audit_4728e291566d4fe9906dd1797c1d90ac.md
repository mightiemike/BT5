### Title
`verify_transaction_inclusion_v2` Pause State Desynchronized from Internal `verify_transaction_inclusion` Dependency — (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion_v2` carries its own `#[pause]` guard but unconditionally delegates to `verify_transaction_inclusion`, which carries a **separate, independently-controllable** `#[pause]` guard. Because `near-plugins` allows each feature to be paused by name, an operator can pause `verify_transaction_inclusion` while leaving `verify_transaction_inclusion_v2` unpaused. Any caller — including an integrated recipient contract — that queries the pause state of `verify_transaction_inclusion_v2`, finds it active, and then calls it will receive an unexpected panic from the inner pause check, not a `bool` result. The pause state of the outer function is therefore a misleading indicator of actual availability.

---

### Finding Description

`verify_transaction_inclusion_v2` is decorated with `#[pause]` and internally calls `verify_transaction_inclusion`, which is also decorated with `#[pause]`: [1](#0-0) 

The `near-plugins` `#[pause]` macro wraps the **function body** itself, not only the external NEAR-call dispatch layer. The generated code is roughly:

```rust
pub fn verify_transaction_inclusion(&self, args: ProofArgs) -> bool {
    if self.is_paused("verify_transaction_inclusion") { env::panic_str("Paused"); }
    // … original body …
}
```

Because the pause check is embedded in the Rust function, it fires on every call site — including the internal `self.verify_transaction_inclusion(args.into())` at line 368 inside `verify_transaction_inclusion_v2`. The two features are registered under different names and can be paused independently by any account holding `Role::PauseManager`.

A realistic operational scenario:

1. A maintainer discovers a correctness issue in the deprecated `verify_transaction_inclusion` (v1) and pauses it to block direct callers.
2. `verify_transaction_inclusion_v2` is intentionally left **unpaused** because it adds the coinbase-proof guard and is the recommended path.
3. Any caller — on-chain or off-chain — that checks the pause state of `verify_transaction_inclusion_v2` and finds it active will proceed to call it.
4. The call passes the v2 pause check, validates the coinbase proof, then panics inside the internal v1 call.

The result is that the pause state of `verify_transaction_inclusion_v2` does **not** accurately reflect whether the function will complete successfully.

---

### Impact Explanation

Recipient contracts that gate fund-release or cross-chain settlement on a `bool` result from `verify_transaction_inclusion_v2` will receive an unexpected panic instead of `true`/`false`. Depending on how the caller handles panics (e.g., treating a failed cross-contract call as `false`, or propagating the failure), this can:

- Silently deny a legitimately proven Bitcoin transaction (false negative), blocking fund release.
- Cause the calling contract to enter an inconsistent state if it does not handle the revert path.

The corrupted value is the **proof result** — the function is expected to return a `bool` but instead panics, which is an incorrect output for any consumer relying on SPV verification.

---

### Likelihood Explanation

Medium. The scenario requires a `PauseManager` to pause `verify_transaction_inclusion` independently of `verify_transaction_inclusion_v2`. This is a natural operational action (pausing the deprecated v1 while keeping v2 live), not an exotic configuration. The deprecation notice on v1 actively encourages operators to think of the two functions as independent, making the independent-pause scenario more likely, not less.

---

### Recommendation

Remove the `#[pause]` attribute from `verify_transaction_inclusion` (the deprecated v1 path) and instead gate it only through `verify_transaction_inclusion_v2`'s pause check, since v2 is the sole supported entry point. Alternatively, add an explicit runtime assertion inside `verify_transaction_inclusion_v2` that checks and surfaces the v1 pause state before delegating, so callers receive a clear, predictable error rather than an opaque panic from a hidden dependency.

---

### Proof of Concept

```
# 1. Operator pauses only the v1 feature by name:
near call <contract> pa_pause '{"feature": "verify_transaction_inclusion"}' \
    --accountId pause_manager

# 2. v2 pause state is NOT set — querying it returns "active":
near call <contract> pa_is_paused \
    '{"feature": "verify_transaction_inclusion_v2"}'
# → false  (i.e., not paused — misleading)

# 3. Recipient contract (or off-chain relayer) calls v2 with a valid proof:
near call <contract> verify_transaction_inclusion_v2 \
    --base64 <borsh-encoded ProofArgsV2> --accountId recipient

# → PANIC: "Paused"   (from the inner verify_transaction_inclusion pause check)
#   Expected: true or false
```

The panic originates at the `near-plugins`-generated pause guard inside `verify_transaction_inclusion` triggered by the internal call at: [2](#0-1) 

while the outer function's own pause check at: [3](#0-2) 

had already passed, making the pause state of `verify_transaction_inclusion_v2` a misleading indicator of actual reachability — the direct analog of the ERC4626 `maxDeposit`/`deposit` desynchronization reported in the reference finding.
