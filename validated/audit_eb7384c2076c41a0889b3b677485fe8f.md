Audit Report

## Title
SwapAllowlistExtension Checks Router Address as `sender` Instead of Actual User, Enabling Allowlist Bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against a per-pool allowlist. `MetricOmmPool.swap` passes `msg.sender` (the direct caller) as `sender` to `_beforeSwap`, so when swaps are routed through `MetricOmmSimpleRouter`, the extension receives the router's address as `sender` — not the actual user. This creates a binary failure: either the router is allowlisted (any user bypasses the restriction) or it is not (allowlisted users cannot swap through the router at all).

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the `sender` argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // <-- direct caller, i.e. the router
    recipient,
    ...
);
```

`_beforeSwap` in `ExtensionCalling.sol` forwards this value unchanged to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router — not the originating user. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly, making itself the pool's `msg.sender`:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The same pattern applies in `exactInput` (L103-112), `exactOutputSingle` (L135-137), and `exactOutput` (L165-181). There is no mechanism for the router to forward the originating user's identity to the pool or extension. The check `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][actual_user]`.

## Impact Explanation

**Bypass path (High):** A pool admin deploys `SwapAllowlistExtension` to restrict swaps to a curated set of addresses. To allow those users to swap through `MetricOmmSimpleRouter` (the standard periphery path), the admin must allowlist the router. Once `allowedSwapper[pool][router] = true`, every call through the router passes the check regardless of who the actual user is. Any non-allowlisted address bypasses the curated restriction by routing through `MetricOmmSimpleRouter`. This is a broken core access-control invariant with direct fund-flow impact: unauthorized users execute swaps on a pool intended to be restricted.

**Broken functionality path (Medium):** If the admin does not allowlist the router, allowlisted users cannot swap through the router — the extension reverts with `NotAllowedToSwap` because `sender = router` is not in the allowlist. This renders the standard swap entrypoint unusable for the intended users on any pool using `SwapAllowlistExtension`.

## Likelihood Explanation

`SwapAllowlistExtension` is a production periphery contract intended for curated pools. `MetricOmmSimpleRouter` is the standard user-facing swap entrypoint. Any pool that deploys both will encounter this mismatch on the first router-mediated swap. No privileged action beyond normal pool setup is required; any user can call the router. The bypass path is triggered the moment the admin allowlists the router to enable router-mediated swaps for legitimate users — a completely expected administrative action.

## Recommendation

The pool should pass the originating caller's identity separately from the direct `msg.sender`. The cleanest fix is to add an explicit `swapper` parameter to `pool.swap` that the router populates with `msg.sender` (the actual user), and have the pool forward that value as `sender` to the extension. Alternatively, `SwapAllowlistExtension` can be redesigned to check `recipient` when `sender` is a known router, but this requires a registry of trusted routers and is more fragile. Threading the real user address through the call stack ensures the extension always evaluates the economically relevant actor.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` as `extension1` in the `beforeSwap` order.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin does **not** allowlist `charlie` (`allowedSwapper[pool][charlie]` remains `false` by default).
4. `charlie` calls `router.exactInputSingle(...)` with `recipient = charlie`.
5. The router calls `pool.swap(charlie, ...)` — pool's `msg.sender = router`.
6. Pool calls `_beforeSwap(router, charlie, ...)` → extension receives `sender = router`.
7. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
8. `charlie`'s swap executes despite never being allowlisted.

The same flow applies to `exactInput`, `exactOutputSingle`, and `exactOutput` in `MetricOmmSimpleRouter`.
