### Title
Dogecoin Fork Difficulty Calculated Using Mainchain Timestamps Instead of Fork Ancestor Timestamps - (`contract/src/dogecoin.rs`)

### Summary

In the Dogecoin build of the BTC light client, `get_next_work_required` calls `get_header_by_height` to retrieve the boundary block for difficulty calculation. `get_header_by_height` always reads from `mainchain_height_to_header`, which only contains mainchain blocks. When a fork block is being validated, the boundary block at the required height is the fork's ancestor, not the mainchain block at that height. Using the mainchain block's timestamp instead of the fork ancestor's timestamp skews `actual_time_taken`, producing an incorrect expected difficulty for every Dogecoin fork block that is two or more blocks deep into the fork (post-height 145,000, where `difficulty_adjustment_interval = 1`).

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the difficulty for the next block by measuring the elapsed time between the previous block and the block one interval earlier: [1](#0-0) 

For Dogecoin post-height 145,000, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. The function then calls: [2](#0-1) 

`get_header_by_height` is implemented as: [3](#0-2) 

It reads exclusively from `mainchain_height_to_header`. Fork blocks are stored only in `headers_pool`, never in `mainchain_height_to_header`: [4](#0-3) 

So when validating a fork block at height `H+2` (where the fork diverged at height `H`), `get_header_by_height(H)` returns the **mainchain** block at height `H`, not the fork's block at height `H`. These two blocks have different timestamps. The developer explicitly flagged this uncertainty with a TODO comment: [5](#0-4) 

The skewed timestamp propagates into `calculate_next_work_required`: [6](#0-5) 

If the mainchain block at height `H` has a later timestamp than the fork's block at height `H` (which the attacker controls by setting an early timestamp on the fork block), `modulated_timespan` is inflated, producing a higher (easier) target. The contract then accepts the fork block's `bits` field matching this inflated target, and the PoW check uses those inflated bits: [7](#0-6) 

### Impact Explanation

An attacker can submit a Dogecoin fork chain where fork blocks carry timestamps set just above the Median Time Past floor. For fork blocks two or more deep, the contract computes difficulty using the mainchain block's timestamp at the boundary height instead of the fork ancestor's timestamp. If the mainchain block is timestamped later than the fork block (which the attacker ensures by setting early fork timestamps), the computed target is inflated. The contract accepts fork blocks with less proof-of-work than the real Dogecoin protocol requires. If the fork accumulates sufficient chainwork to trigger a reorg (`current_header.chain_work > total_main_chain_chainwork`), the contract's canonical chain diverges from the real Dogecoin network. Consumer contracts calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` then verify transaction inclusion against a canonical chain that the real Dogecoin network does not recognize, producing false positive SPV proofs. [8](#0-7) 

### Likelihood Explanation

The trigger is reachable by any unprivileged NEAR caller via `submit_blocks`. Post-height 145,000, every Dogecoin fork block beyond the first is affected because `difficulty_adjustment_interval = 1` means every block is a retarget boundary. The attacker controls fork block timestamps (within the MTP and future-time bounds), making the timestamp delta manipulation straightforward. No privileged role, leaked key, or social engineering is required.

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` (dogecoin.rs, and analogously in bitcoin.rs and litecoin.rs) with a traversal that walks the `prev_block_hash` chain from `prev_block_header` backward through `headers_pool` to find the true ancestor at `height_first`. This ensures the boundary block's timestamp is always the fork's own ancestor, not the mainchain block at the same height.

### Proof of Concept

1. Contract is initialized with Dogecoin mainchain at height 200,000 (post-145,000, so `difficulty_adjustment_interval = 1`). Mainchain block at height 200,001 has timestamp `T_main = 1_700_000_100`.
2. Attacker calls `submit_blocks` with a fork block at height 200,001 (`fork_block_1`) whose `prev_block_hash` points to the mainchain block at 200,000. Attacker sets `fork_block_1.time = T_main - 50` (50 seconds earlier, still above MTP). This block is accepted as a fork block.
3. Attacker calls `submit_blocks` with `fork_block_2` at height 200,002, `prev_block_hash = fork_block_1.hash`, timestamp `T_fork2`.
4. `get_next_work_required` computes `height_first = 200,001 - 1 = 200,000`... wait, let me re-check. `blocks_to_go_back = 1`, `height_first = prev_block_header.block_height - 1 = 200,001 - 1 = 200,000`. `get_header_by_height(200,000)` returns the mainchain block at 200,000 (correct here since fork diverged at 200,001).
5. For `fork_block_3` at height 200,003: `height_first = 200,002 - 1 = 200,001`. `get_header_by_height(200,001)` returns the **mainchain** block at 200,001 (timestamp `T_main = 1_700_000_100`), not `fork_block_1` (timestamp `T_main - 50 = 1_700_000_050`). The `modulated_timespan` is computed as `T_fork2 - T_main` instead of the correct `T_fork2 - fork_block_1.time`. The 50-second difference inflates the timespan, producing an easier target. The contract accepts `fork_block_3` with bits matching this inflated target, requiring less actual PoW than the real Dogecoin protocol mandates.
6. Repeating for subsequent fork blocks, the attacker builds a fork chain with systematically under-powered blocks. If the fork's cumulative chainwork exceeds the mainchain's, `reorg_chain` is triggered, and the contract's canonical chain becomes the attacker's fork. [9](#0-8) [10](#0-9)
