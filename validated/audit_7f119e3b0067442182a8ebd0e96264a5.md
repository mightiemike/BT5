### Title
Gas-Key Host Functions Pass Borsh Wire Length Instead of Trie-ID Length to Fee Calculators for ML-DSA-65 Keys — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The three gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`) pass the raw caller-supplied `public_key_len` (the borsh wire length of the key bytes) directly to the fee-calculation helpers `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee`. For ML-DSA-65 keys this value is **1953 bytes** (1 tag + 1952 raw pubkey bytes). The correct value — used by every transaction-path fee calculation — is `public_key.trie_id_len()`, which is **33 bytes** (1 tag + 32-byte SHA3-256 hash). The two diverge only for ML-DSA-65; for ed25519 and secp256k1 they are identical, so the bug is invisible in all existing tests.

---

### Finding Description

The protocol documentation explicitly states the invariant:

> "Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`."

The transaction-path fee calculation in `permission_exec_fees` correctly honours this:

```rust
// runtime/runtime/src/config.rs:389-394
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← correct: 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
``` [1](#0-0) 

The host-function path in both the interpreter-mode (`logic.rs`) and Wasmtime-mode (`wasmtime_runner/logic.rs`) implementations passes the raw wire length instead:

```rust
// runtime/near-vm-runner/src/logic/logic.rs:3155-3159
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← BUG: 1953 for ML-DSA-65, should be 33
    num_nonces,
);
``` [2](#0-1) 

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call`: [3](#0-2) 

And in `promise_batch_action_transfer_to_gas_key` (both `logic.rs` and `wasmtime_runner/logic.rs`): [4](#0-3) [5](#0-4) 

The fee helper `gas_key_add_key_exec_fee` uses `public_key_len` to compute the per-nonce trie key length:

```rust
// core/parameters/src/cost.rs:888-889
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
``` [6](#0-5) 

For ML-DSA-65 with a 10-byte account ID and 4 nonces:
- **Correct** `nonce_key_len`: `2 + 10 + 33 + 4 = 49` bytes
- **Actual** `nonce_key_len`: `2 + 10 + 1953 + 4 = 1969` bytes

The per-byte exec fee component is multiplied by `(nonce_key_len + NONCE_VALUE_LEN) × num_nonces`, producing a ~40× overcharge on that component.

The `trie_id_len()` contract is documented precisely: [7](#0-6) 

The existing fee-parity test `test_gas_key_fee_parity` only exercises ED25519 keys, for which `len()` and `trie_id_len()` are identical (both 33 bytes), so the divergence is never exercised: [8](#0-7) 

---

### Impact Explanation

Any on-chain contract that calls `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, or `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key is charged ~40–60× more gas on the per-byte component than the equivalent transaction-path operation. This breaks the documented fee-domain invariant (`trie_id_len()` must be used for all trie-byte-priced paths) and can cause legitimate contract calls to exhaust their gas budget when the same operation submitted as a transaction would succeed. The divergence is deterministic and reproducible from the borsh-encoded key length alone.

---

### Likelihood Explanation

Both `ProtocolFeature::GasKeys` and `ProtocolFeature::PostQuantumSignatures` are stabilised (protocol version 85). The host functions are exposed under the `#[gas_key_host_fns]` gate in `runtime/near-vm-runner/src/imports.rs`. Any contract deployed after protocol version 85 that programmatically manages ML-DSA-65 gas keys via the promise API will trigger the overcharge. The bug is invisible for ed25519/secp256k1 keys, so it has gone undetected by all existing tests. [9](#0-8) 

---

### Recommendation

Replace `public_key_len as usize` with `public_key.decode()?.trie_id_len()` (or decode the key first and use its `trie_id_len()`) in all three host-function implementations in both `runtime/near-vm-runner/src/logic/logic.rs` and `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`. Extend `test_gas_key_fee_parity` to cover `KeyType::MLDSA65` to prevent regression.

---

### Proof of Concept

The exact divergent byte values:

| Path | `public_key_len` passed to fee helper | Source |
|---|---|---|
| Transaction (`permission_exec_fees`) | `public_key.trie_id_len()` = **33** | `runtime/runtime/src/config.rs:392` |
| Host fn (`promise_batch_action_add_gas_key_with_full_access`) | `public_key_len as usize` = **1953** | `runtime/near-vm-runner/src/logic/logic.rs:3158` |

For `num_nonces = 4`, `account_id_len = 10`, `NONCE_VALUE_LEN = 8`:

- Correct per-byte exec gas: `gas_key_byte × (49 + 8) × 4 = gas_key_byte × 228`
- Actual per-byte exec gas: `gas_key_byte × (1969 + 8) × 4 = gas_key_byte × 7908`
- **Overcharge ratio: ~34.7×**

The transaction path correctly uses `trie_id_len()`: [10](#0-9) 

The host function path uses the raw wire length: [11](#0-10)

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
```
