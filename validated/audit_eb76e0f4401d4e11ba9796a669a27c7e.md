Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Controlled `owner` Instead of Actual Depositor `sender`, Enabling Full Allowlist Bypass — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor, i.e., `msg.sender` of the pool call) and instead checks the `owner` parameter against the allowlist. Because `owner` is a free caller-supplied argument to `addLiquidity`, any unprivileged actor can bypass the deposit allowlist by nominating any allowlisted address as `owner`, rendering the pool admin's access control gate entirely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)` where `msg.sender` is the actual depositor and `owner` is a caller-supplied LP-position recipient: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both as the first two positional arguments of the hook call: [2](#0-1) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter (`sender`) is unnamed and discarded; only `owner` is tested against the allowlist mapping: [3](#0-2) 

The allowlist mapping is keyed on `depositor`, making the intent clear — it is meant to gate the address supplying tokens, not the LP-position recipient: [4](#0-3) 

The sister extension `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it checks `sender` and discards `recipient` (the second parameter): [5](#0-4) 

`DepositAllowlistExtension` inverts this: it checks the wrong field. There are no other guards in the `addLiquidity` path that enforce the depositor allowlist.

## Impact Explanation

The deposit allowlist is a pool-admin-configured access control gate. Bypassing it allows any unprivileged actor to deposit tokens into a restricted pool. The attacker's tokens are consumed by the pool via the liquidity callback, while LP shares are credited to the allowlisted `owner` address. This constitutes broken core pool functionality (the allowlist access control is rendered inoperative), enables pool composition manipulation by unauthorized parties (shifting bin balances in a restricted pool, affecting oracle-anchored pricing), and can harm existing LPs through adverse bin selection. The wrong value is the `allowedDepositor` check result: it evaluates `allowedDepositor[pool][owner]` (always true for any allowlisted address) instead of `allowedDepositor[pool][sender]` (which would be false for the unauthorized depositor).

## Likelihood Explanation

Exploitation requires no special privilege. Any address can call `addLiquidity` on the pool with `owner` set to any allowlisted address. The allowlist of a live pool is fully observable on-chain via the public `allowedDepositor` and `allowAllDepositors` mappings. The attack is trivially repeatable with zero preconditions beyond knowing one allowlisted address.

## Recommendation

Name and check `sender` instead of `owner`, mirroring `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address /*owner*/, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only `Alice`: `allowedDepositor[pool][Alice] = true`.
2. `Bob` (not allowlisted) calls: `pool.addLiquidity(Alice /*owner*/, salt, deltas, callbackData, extensionData)`.
3. The pool calls `_beforeAddLiquidity(Bob /*sender*/, Alice /*owner*/, ...)`.
4. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender = Bob` (discarded, unnamed) and `owner = Alice`.
5. The check `allowedDepositor[pool][Alice]` is `true` → no revert.
6. Bob's tokens are pulled from his wallet via the liquidity callback and deposited into the pool; LP shares are credited to Alice.
7. Bob has deposited into a pool he was explicitly barred from, bypassing the pool admin's access control entirely.

A Foundry test can confirm this by: deploying the extension with a factory mock, setting `allowedDepositor[pool][Alice] = true` (Bob not set), calling `beforeAddLiquidity(Bob, Alice, ...)` directly on the extension with `msg.sender = pool`, and asserting no revert occurs — then calling `beforeAddLiquidity(Bob, Bob, ...)` and asserting it reverts with `NotAllowedToDeposit`.
