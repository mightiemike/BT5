Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, enabling full allowlist bypass via router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool, which is the pool's `msg.sender` — the router contract — not the original end user. When a pool admin allowlists the router address to permit router-mediated swaps, every unprivileged user can bypass the swap allowlist by routing through `MetricOmmSimpleRouter`, because the extension sees the router's address and passes the check regardless of who initiated the transaction.

## Finding Description

**Call path:**

1. Unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle()` (or `exactInput`, `exactOutputSingle`, `exactOutput`).
2. The router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)` — here `msg.sender` to the pool is the **router address**.
3. `MetricOmmPool.swap()` (line 230–240) calls `_beforeSwap(msg.sender, ...)` passing the router as `sender`.
4. `ExtensionCalling._beforeSwap` (line 160–176) ABI-encodes `sender = router` and dispatches to the configured extension.
5. `SwapAllowlistExtension.beforeSwap` (line 37) evaluates:
   ```solidity
   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
   ```
   where `msg.sender` = pool and `sender` = **router address**.

The check therefore asks: "is the router allowed to swap on this pool?" — not "is the original user allowed?"

**Root cause:** The pool has no mechanism to forward the original caller's identity through the router. The pool correctly passes its own `msg.sender` as `sender`, but that is the router, not the human trader. The extension has no access to the true originator.

**Exploit scenario:**
- Pool admin deploys a curated pool with `SwapAllowlistExtension` and allowlists only addresses `A` and `B`.
- To let `A` and `B` use the router, the admin must also call `setAllowedToSwap(pool, router, true)`.
- Once the router is allowlisted, any address `C` (not on the allowlist) calls `router.exactInputSingle(...)` and the extension sees `sender = router`, passes the check, and the swap executes — full allowlist bypass.
- Alternatively, if the admin never allowlists the router, even `A` and `B` cannot use the router, breaking the intended UX.

**Existing guards reviewed and shown insufficient:**
- `BaseMetricExtension.onlyPool` (line 19–24) only verifies that the caller of `beforeSwap` is a registered pool — it does not recover the original user identity.
- The `sender` parameter in `IMetricOmmExtensions.beforeSwap` is documented as the pool's `msg.sender`, which is structurally the router when the router is used.
- There is no transient-storage or callback mechanism that threads the original `msg.sender` from the router into the extension call.

## Impact Explanation
A curated pool using `SwapAllowlistExtension` to restrict trading to a specific set of addresses is rendered ineffective for any user who routes through `MetricOmmSimpleRouter`. Any unprivileged user can execute swaps on a pool that was designed to be permissioned, receiving output tokens they should not be entitled to trade for. This constitutes a broken core pool functionality (allowlist policy bypass) and a direct admin-boundary break where an unprivileged path circumvents a configured access control. Severity: **High**.

## Likelihood Explanation
The router is the primary supported periphery entrypoint for swaps. Any pool that enables router access (by allowlisting the router) is immediately exploitable by any address. No special privileges, flash loans, or multi-block setup are required — a single `exactInputSingle` call suffices. The condition (router allowlisted on a curated pool) is the normal operational state for any pool that wants to support both allowlist enforcement and router usage.

## Recommendation
The extension must gate on the **original user's identity**, not the intermediary's. Two viable approaches:

1. **Pass original caller through router:** Have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it. This requires a convention between router and extension.
2. **Check recipient instead of sender:** If the pool's design intent is to gate who *receives* output, check the `recipient` argument. If the intent is to gate who *initiates* the trade, the only sound fix is approach 1 or requiring direct pool calls (no router) for allowlisted pools.
3. **Dedicated router that preserves caller identity:** Extend the pool's `swap` interface or use a trusted-forwarder pattern so the original `msg.sender` is authenticated and forwarded to extensions.

## Proof of Concept

```solidity
// Foundry test sketch
function test_swapAllowlistBypassViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
    swapExtension.setAllowedToSwap(address(pool), address(router), true); // admin must do this for router to work
    // allowedUser is NOT set — only router is allowlisted

    address attacker = makeAddr("attacker");
    deal(address(tokenIn), attacker, 1_000e18);
    vm.startPrank(attacker);
    tokenIn.approve(address(router), type(uint256).max);

    // Direct pool swap should revert
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(attacker, true, 1000, type(uint128).max, "", "");

    // Router swap bypasses the allowlist — succeeds despite attacker not being allowlisted
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(tokenIn),
        tokenOut: address(tokenOut),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // attacker received output — allowlist bypassed
    vm.stopPrank();
}
```
