### Title
Unsigned `oraclePriceX18` in `MintNlp`/`BurnNlp` Enables Arbitrary NLP Price Manipulation via Slow Mode — (File: `core/contracts/Verifier.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The EIP-712 digest for `MintNlp` and `BurnNlp` transactions omits the `oraclePriceX18` field. Because the slow mode path allows any user to submit these transaction types with self-constructed calldata, an unprivileged attacker can set an arbitrary oracle price for NLP minting and burning. The only on-chain protection is a 0.1% burn fee, which is trivially overcome by any price deviation above that threshold. The attacker can mint NLP at a manipulated low price and burn at a manipulated high price, extracting value from NLP pool participants.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` each carry an `oraclePriceX18` field that is **not covered by the user's signature**. In `Verifier.sol`, the digest for `MintNlp` is computed over only `{MINT_NLP_SIGNATURE, sender, quoteAmount, nonce}`: [1](#0-0) 

And for `BurnNlp` over only `{BURN_NLP_SIGNATURE, sender, nlpAmount, nonce}`: [2](#0-1) 

The `oraclePriceX18` field is present in both structs but is never hashed into the digest: [3](#0-2) 

In the normal sequencer path, the sequencer appends `oraclePriceX18` to the transaction before submitting it on-chain. However, the slow mode path (`submitSlowModeTransactionImpl`) accepts `MintNlp` and `BurnNlp` from any caller without any restriction on the `oraclePriceX18` value: [4](#0-3) 

When the slow mode transaction is later executed, `processTransactionImpl` decodes the full `SignedMintNlp` struct and passes the attacker-supplied `oraclePriceX18` directly to `clearinghouse.mintNlp`: [5](#0-4) 

Inside `mintNlp`, the NLP token amount is computed as `nlpAmount = quoteAmount / oraclePriceX18`: [6](#0-5) 

Inside `burnNlp`, the quote payout is `quoteAmount = nlpAmount * oraclePriceX18 - burnFee`, where `burnFee` is only 0.1%: [7](#0-6) 

There is no on-chain validation that `oraclePriceX18` is within any reasonable range or matches any trusted price feed. The only constraint is `priceX18 > 0`.

---

### Impact Explanation

An attacker who submits a slow mode `MintNlp` with a manipulated low `oraclePriceX18` receives far more NLP tokens per unit of quote than the true NAV warrants. After the NLP lock period expires, the attacker submits a slow mode `BurnNlp` with a manipulated high `oraclePriceX18`, receiving far more quote per NLP token than the true NAV warrants. The difference, minus the 0.1% burn fee, is extracted directly from the NLP pool's quote reserves, diluting all other NLP holders. The corrupted state delta is:

- `spotEngine.balances[NLP_PRODUCT_ID][attacker]` — inflated NLP balance on mint
- `spotEngine.balances[QUOTE_PRODUCT_ID][attacker]` — inflated quote balance on burn
- `spotEngine.balances[QUOTE_PRODUCT_ID][nlpPool[i]]` — drained NLP pool quote reserves [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who:
1. Holds enough quote collateral to pay the slow mode fee and the `quoteAmount`
2. Can read `nlpPools` from `EndpointStorage` to construct a valid `nlpPoolRebalanceX18`
3. Waits the 3-day slow mode delay and the NLP lock period

No privileged access, governance capture, or sequencer compromise is required. The slow mode path is a public, permissionless entry point. The attack is deterministic and profitable for any price deviation above 0.1%. [10](#0-9) 

---

### Recommendation

Include `oraclePriceX18` in the EIP-712 digest for both `MintNlp` and `BurnNlp` in `Verifier.sol`. This forces the user to commit to a specific price at signing time, preventing post-signing manipulation. Additionally, consider adding an on-chain staleness or bounds check on `oraclePriceX18` against a trusted price reference (e.g., the last sequencer-submitted price stored in `priceX18[NLP_PRODUCT_ID]`) to bound the deviation allowed in any single mint or burn. [11](#0-10) [12](#0-11) 

---

### Proof of Concept

1. Attacker reads `nlpPools` from `EndpointStorage` to learn pool subaccounts and count.
2. Attacker signs a `MintNlp` message committing to `{sender=attacker, quoteAmount=1000e18, nonce=N}`. The signature does **not** cover `oraclePriceX18`.
3. Attacker constructs `SignedMintNlp{tx: {sender, quoteAmount=1000e18, nonce=N}, signature: sig, oraclePriceX18: 1 /* 1 wei */, nlpPoolRebalanceX18: [1000e18]}` and calls `submitSlowModeTransaction`.
4. After 3 days, attacker calls `executeSlowModeTransaction`. `mintNlp` executes: `nlpAmount = 1000e18 / 1 = 1000e36` NLP tokens are credited to the attacker. The NLP pool receives `1000e18` quote.
5. After the NLP lock period, attacker signs a `BurnNlp` message committing to `{sender=attacker, nlpAmount=1000e36, nonce=N+1}`.
6. Attacker constructs `SignedBurnNlp{..., oraclePriceX18: 1e36, nlpPoolRebalanceX18: [-quoteOut]}` and submits via slow mode.
7. After 3 days, `burnNlp` executes: `quoteAmount = 1000e36 * 1e36 - burnFee`. The attacker receives an astronomically large quote balance, draining the NLP pool.

The signature validation passes at step 4 and step 7 because `oraclePriceX18` is absent from the digest. [1](#0-0) [13](#0-12) [14](#0-13)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
