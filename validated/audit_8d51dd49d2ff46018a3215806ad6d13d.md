### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. The extension therefore checks whether the **router** is allowlisted, not whether the **user** is allowlisted. If the router is allowlisted (a natural admin action to enable router-mediated swaps), every user on the network can bypass the swap allowlist by routing through the router.

---

### Finding Description

**Call chain:**

```
User → MetricOmmSimpleRouter.exactInputSingle()
     → pool.swap(recipient, ..., extensionData)   // msg.sender = router
     → ExtensionCalling._beforeSwap(msg.sender=router, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever called `pool.swap()`: [3](#0-2) 

When the router calls `pool.swap()`, `sender = router`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The router itself passes no user identity into the pool call — it simply calls `pool.swap(params.recipient, ...)` with itself as `msg.sender`: [4](#0-3) 

---

### Impact Explanation

Two fund-impacting failure modes arise:

**Mode A — Complete allowlist bypass (High):** A pool admin who wants to allow router-mediated swaps will allowlist the router address. Once the router is allowlisted, `allowedSwapper[pool][router]` is `true`, and the check passes for every user who routes through the router — regardless of whether that user is individually allowlisted. The entire per-user access control is nullified for all router-mediated swaps. Unauthorized users gain full swap access to a pool that was designed to be restricted (e.g., a private institutional pool), enabling them to extract value through arbitrage or price impact against allowlisted LPs.

**Mode B — Broken core swap functionality for legitimate users (Medium):** If the router is not allowlisted, an individually allowlisted user who calls `router.exactInputSingle()` will be rejected because the extension sees `sender = router` and `allowedSwapper[pool][router] = false`. The primary user-facing swap interface is completely unusable for allowlisted users, forcing them to call `pool.swap()` directly (requiring them to implement the swap callback themselves), which breaks the intended UX and integration path.

---

### Likelihood Explanation

Mode A requires the pool admin to allowlist the router — a natural and expected configuration step for any pool that intends to support the periphery router. Mode B occurs unconditionally for every allowlisted user who uses the router without the router being allowlisted. Both modes are reachable through normal, unprivileged user actions (calling the public router) once the pool is deployed with the extension.

---

### Recommendation

The extension must check the **end user's identity**, not the intermediary's. Two options:

1. **Pass the original user through the router:** Have the router encode the original `msg.sender` in `extensionData` and have the extension decode and check it. This requires a protocol-level convention.

2. **Check `sender` only for direct pool calls; require the router to forward user identity:** Add a dedicated field to the extension interface or use a separate allowlist entry for router-mediated swaps that still validates the end user.

The simplest safe fix is to document that the `sender` field is the direct pool caller and require pool admins to allowlist only EOAs or contracts that themselves enforce user-level access control — but this is fragile. A structural fix in the extension interface is preferable.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Pool admin calls `setAllowedToSwap(pool, alice, true)` and does **not** allowlist `bob`.
4. `bob` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Router calls `pool.swap(...)` with `msg.sender = router`.
6. Extension checks `allowedSwapper[pool][router] == true` → passes.
7. `bob` successfully swaps in the restricted pool, bypassing the allowlist entirely.

Conversely, if step 2 is skipped (router not allowlisted):

4. `alice` (allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
5. Extension checks `allowedSwapper[pool][router] == false` → reverts with `NotAllowedToSwap`.
6. `alice` cannot use the router despite being individually allowlisted. [3](#0-2) [5](#0-4)
