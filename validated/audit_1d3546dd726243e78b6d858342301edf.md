### Title
`SwapAllowlistExtension` gates on the router address rather than the ultimate user, allowing any unprivileged address to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the actual user. If the pool admin allowlists the router (the natural step to enable router-mediated swaps), every unprivileged address can bypass the individual allowlist by calling any router entry-point.

---

### Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`, which forwards it verbatim to the extension as `sender`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  zeroForOne,
  amountSpecified,
  priceLimitX64,
  packedSlot0Initial,
  bidPriceX64,
  askPriceX64,
  extensionData
);
```

`SwapAllowlistExtension.beforeSwap` then checks that `sender` against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

So the extension sees `sender = address(router)`, not the actual user. The allowlist check becomes `allowedSwapper[pool][router]`. If the router is allowlisted — the natural configuration for a pool that wants to support the standard periphery — every address on the network can swap by calling the router, regardless of whether they are individually allowlisted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the second parameter, the actual beneficiary), not `sender` (the intermediary):

```solidity
// DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
```

The swap allowlist is architecturally inconsistent with the deposit allowlist and cannot enforce per-user restrictions when the router is in the call path.

---

### Impact Explanation

Any address can swap in a pool that the admin intended to restrict to a specific set of counterparties (e.g., KYC-verified addresses, whitelisted protocols, or institutional partners). The bypass is unconditional once the router is allowlisted: the attacker supplies any valid `amountIn`, the pool executes the swap at the oracle price, and the attacker receives output tokens. The pool's access-control invariant — that only allowlisted addresses may trade — is permanently broken for all router-mediated paths. This constitutes an admin-boundary break with direct fund-flow consequences (unauthorized parties extract value from a restricted pool).

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who wants to allow router-mediated swaps for their allowlisted users will naturally add the router to `allowedSwapper`. The admin has no on-chain signal that doing so opens the pool to all users; the `isAllowedToSwap` view function returns `true` for the router, giving no indication of the bypass. The trigger requires only a standard router call with no special permissions, making this reachable by any unprivileged user.

---

### Recommendation

Gate `SwapAllowlistExtension.beforeSwap` on the economically relevant actor — the address that pays tokens in — rather than the immediate pool caller. Two concrete approaches:

1. **Decode the payer from `extensionData`**: Require the router to encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and check that address. This is consistent with how `extensionData` is already forwarded through the full hook chain.

2. **Mirror the deposit pattern**: Introduce a `payer` field analogous to `owner` in the liquidity hooks, passed explicitly by the pool from the callback context, so the extension always sees the address that funds the swap.

Until fixed, document that allowlisting the router in `SwapAllowlistExtension` is equivalent to `setAllowAllSwappers(pool, true)`.

---

### Proof of Concept

```
Setup:
  1. Pool admin deploys MetricOmmPool with SwapAllowlistExtension as beforeSwap hook.
  2. Admin calls setAllowedToSwap(pool, alice, true)       // alice is the only allowed swapper
  3. Admin calls setAllowedToSwap(pool, router, true)      // enable router-mediated swaps for alice

Attack:
  4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
         pool:            <restricted pool>,
         tokenIn:         token1,
         zeroForOne:      false,
         amountIn:        X,
         amountOutMinimum: 0,
         recipient:       bob,
         ...
       })

  5. Router calls pool.swap(bob, false, X, limit, "", "")
     → pool calls _beforeSwap(msg.sender=router, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
     → allowedSwapper[pool][router] == true  → check passes
     → swap executes at oracle price
     → Bob receives token0 output

Result: Bob, who is not individually allowlisted, successfully swaps in a pool
        the admin intended to restrict to alice only.
        The allowlist guard is fully bypassed via the public router.
