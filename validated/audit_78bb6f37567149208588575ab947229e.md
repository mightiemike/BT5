### Title
Liquidator Can Finalize Subaccount With Insufficient Quote Balance, Causing Negative Quote — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`_finalizeSubaccount()` settles the liquidatee's positive perp PnL against the liquidator's quote balance via `_settlePnlAgainstLiquidator()`, but never checks that the liquidator's quote balance remains non-negative after the settlement. `liquidateSubaccountImpl()` returns immediately after `_finalizeSubaccount()` returns `true` with no health check on `txn.sender`. A liquidator with insufficient quote can trigger this path and end up with a negative quote balance, effectively printing quote and threatening protocol solvency.

---

### Finding Description

`liquidateSubaccountImpl()` dispatches to `_finalizeSubaccount()` when `txn.productId == type(uint32).max`. [1](#0-0) 

Inside `_finalizeSubaccount()`, for every perp product where the liquidatee holds positive `vQuoteBalance`, the function calls `_settlePnlAgainstLiquidator()`: [2](#0-1) 

`_settlePnlAgainstLiquidator()` unconditionally decreases the liquidator's spot quote balance by `pnl`: [3](#0-2) 

After all settlements complete, `_finalizeSubaccount()` returns `true` and `liquidateSubaccountImpl()` returns — **with no health or balance check on `txn.sender`**: [1](#0-0) 

Contrast this with `_handleLiquidationPayment()`, the other liquidation path, which explicitly guards the liquidator's health after every balance mutation: [4](#0-3) 

The finalization path has no equivalent guard.

---

### Impact Explanation

When the liquidatee has large positive `vQuoteBalance` across multiple perp products (realized PnL from closed positions not yet settled to spot), the liquidator absorbs all of it as a spot quote debit. If the liquidator's quote balance is smaller than the total positive PnL being settled, their quote balance goes negative. Because the protocol's accounting is purely internal (signed integers), this negative balance is accepted silently. The liquidatee's spot quote increases by the same amount, meaning quote has been created from nothing. Repeated exploitation across multiple finalization events can drain the insurance fund and socialize losses across all participants, threatening protocol solvency.

---

### Likelihood Explanation

The preconditions are realistic:
1. A liquidatee must be under maintenance health (`isUnderMaintenance` passes).
2. All perp position `amount` fields must be zero (positions closed), but `vQuoteBalance > 0` (realized PnL not yet settled).
3. All non-quote spot balances must be `<= 0`.
4. The liquidator calls with `productId == type(uint32).max`.

A liquidatee can reach this state naturally: a trader closes all perp positions (amount → 0) but retains positive `vQuoteBalance`, while their spot quote balance is sufficiently negative to remain under maintenance health. Any unprivileged address can act as liquidator and submit the finalization transaction through the sequencer-gated `Endpoint`.

---

### Recommendation

Add a health check on `txn.sender` after `_finalizeSubaccount()` returns `true`, mirroring the guard already present in `_handleLiquidationPayment()`:

```solidity
if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
    require(
        txn.sender == N_ACCOUNT || !isUnderInitial(txn.sender),
        ERR_SUBACCT_HEALTH
    );
    if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
        IOffchainExchange(
            IEndpoint(getEndpoint()).getOffchainExchange()
        ).tryCloseIsolatedSubaccount(txn.liquidatee);
    }
    return;
}
```

---

### Proof of Concept

1. Liquidatee holds zero perp position amounts but large positive `vQuoteBalance` (e.g., `+1,000,000e18` USDC equivalent) across several perp products, and a deeply negative spot quote balance (e.g., `-900,000e18`), keeping them under maintenance health.
2. Attacker controls a liquidator subaccount with only `100,000e18` quote balance.
3. Attacker submits `LiquidateSubaccount` with `productId = type(uint32).max` through the `Endpoint`.
4. `liquidateSubaccountImpl()` calls `_finalizeSubaccount()`.
5. The positive PnL settlement loop calls `_settlePnlAgainstLiquidator()` for each perp, debiting the liquidator's quote by `1,000,000e18` total.
6. Liquidator's quote balance becomes `100,000e18 - 1,000,000e18 = -900,000e18`.
7. `_finalizeSubaccount()` returns `true`; `liquidateSubaccountImpl()` returns with no health check.
8. The liquidator now holds a negative quote balance. The liquidatee's quote balance has been inflated. Net quote has been printed. [3](#0-2) [5](#0-4)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
