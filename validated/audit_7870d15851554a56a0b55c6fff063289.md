Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of end user, allowing any caller to bypass per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` gates pool swaps by checking the `sender` argument passed from `MetricOmmPool.swap()`, which is always `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates a swap, `sender` is the router contract address for every user. A pool admin who allowlists the router (required for any router-mediated swap to succeed) inadvertently grants every user of the router unrestricted swap access, collapsing the per-user allowlist to a per-contract gate.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` verbatim as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient,
    ...
    extensionData
);
```

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L149-177
function _beforeSwap(address sender, ...) internal {
    _callExtensionsInOrder(
        BEFORE_SWAP_ORDER,
        abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
    );
}
```

`SwapAllowlistExtension.beforeSwap` then checks that identity against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`MetricOmmSimpleRouter` calls `pool.swap()` directly in all four entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), making the router the `msg.sender` in every pool call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router provides no mechanism to forward the originating user's address into `sender`; `extensionData` is user-controlled and cannot be trusted for identity. For any router-mediated swap to succeed on an allowlisted pool, the admin must add `allowedSwapper[pool][router] = true`. Once set, the check `allowedSwapper[pool][router]` returns `true` for every user who routes through the router — the per-user gate collapses entirely.

## Impact Explanation
Any unprivileged user can bypass a pool's swap allowlist by routing through `MetricOmmSimpleRouter`. Once the router is allowlisted (the only way to enable router-mediated swaps on an allowlisted pool), the allowlist no longer gates individual users. Unauthorized users gain full swap access to a pool intended to be restricted, enabling them to consume LP capital and extract output tokens against a pool the admin explicitly prohibited them from accessing. This is a direct admin-boundary break with fund-impacting consequences: unauthorized swap settlement against restricted LP capital.

## Likelihood Explanation
The scenario is reachable by any unprivileged user with no special setup. The only precondition — the pool admin allowlisting the router — is a natural and expected operational action to support normal UX. No malicious initial setup, non-standard tokens, or privileged cooperation is required. The attack is repeatable by any user on any allowlisted pool that has enabled router access.

## Recommendation
The `sender` identity passed to extension hooks must reflect the economic actor, not the intermediary contract. Two complementary fixes:

1. **Router-level**: `MetricOmmSimpleRouter` should encode the originating `msg.sender` into `extensionData` using an authenticated field (e.g., ABI-encoded with a router-specific prefix), and `SwapAllowlistExtension` should decode and verify it when `sender` is a known trusted forwarder.
2. **Extension-level**: `SwapAllowlistExtension` should expose a separate allowlist for trusted forwarders and, when `sender` is a known forwarder, decode the real user from `extensionData` before performing the allowlist check.

The simplest safe default is to never allowlist the router address directly; instead, require that all allowlisted swaps arrive via direct `pool.swap()` calls, and document this constraint explicitly.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension (beforeSwap order set)
  allowedSwapper[pool][router] = true   ← admin enables router access
  allowedSwapper[pool][attacker] = false ← attacker is explicitly blocked

Attack:
  attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient=attacker, ...)  [msg.sender = router]
    → pool calls _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
        allowedSwapper[pool][router] == true  ✓ passes
    → swap executes, attacker receives output tokens

Result:
  Attacker swaps against a pool they are explicitly blocked from,
  receiving output tokens while the pool's LP capital is consumed
  without the access restriction the admin configured.
```

Foundry test outline:
1. Deploy pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. Call `setAllowedToSwap(pool, router, true)` as pool admin.
3. Call `setAllowedToSwap(pool, attacker, false)` as pool admin.
4. Impersonate attacker and call `MetricOmmSimpleRouter.exactInputSingle(...)`.
5. Assert swap succeeds and attacker receives output tokens despite being explicitly blocked.
