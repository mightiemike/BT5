### Title
`ENABLE_SECP_OPS` Flag Not Checked in 4-Byte Opcode Dispatch Path — (`File: src/chia_dialect.rs`)

### Summary

The `ENABLE_SECP_OPS` flag is defined to gate secp signature-verification operators, but it is only enforced on the 1-byte opcode aliases (opcodes 64 and 65). The separate 4-byte opcode dispatch path (`0x13d61f00` → `op_secp256k1_verify`, `0x1c3a8f00` → `op_secp256r1_verify`) executes the same operators unconditionally, with no flag check. An attacker-controlled CLVM program can invoke secp verification using the 4-byte encoding and bypass the intended gate entirely.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` function of `ChiaDialect` has two separate dispatch paths for secp operators.

**Path 1 — 1-byte opcodes (correctly gated):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

If `ENABLE_SECP_OPS` is absent, these fall through to `unknown_operator`, which either rejects (mempool mode) or no-ops (consensus mode).

**Path 2 — 4-byte opcodes (ungated):** [2](#0-1) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

There is **no** `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` guard here. The 4-byte path is evaluated first (before the 1-byte dispatch), so it is reached regardless of whether `ENABLE_SECP_OPS` is set.

The flag's own documentation confirms the intended scope: [3](#0-2) 

```
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The flag description names only the 1-byte aliases. The 4-byte originals are silently left ungated.

The `MEMPOOL_MODE` constant does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

In mempool mode, the 1-byte secp opcodes (64, 65) are rejected because `ENABLE_SECP_OPS` is absent and they fall to `unknown_operator` which raises under `NO_UNKNOWN_OPS`. But the 4-byte secp opcodes are **explicitly dispatched before** the unknown-operator path, so `NO_UNKNOWN_OPS` never applies to them. They execute unconditionally in both mempool and consensus mode.

---

### Impact Explanation

**Consensus divergence / soft-fork bypass.** Per the operator checklist, flags like `ENABLE_SECP_OPS` exist precisely so the chain can exist in a state before a soft-fork activates and behave consistently with older nodes: [5](#0-4) 

A node running new code but without `ENABLE_SECP_OPS` set (pre-activation state) will still execute secp verification via the 4-byte path. An older node (or a node that treats 4-byte secp opcodes as unknown) will treat the same program as a no-op with cost. The two nodes reach different outcomes for the same CLVM program — a consensus split.

**Mempool inconsistency.** In mempool mode, a program using 1-byte opcode 64 is rejected as unimplemented, while the identical operation encoded as `0x13d61f00` succeeds. An attacker can craft a program that passes mempool validation (via 4-byte encoding) but would be rejected if re-encoded with the 1-byte alias, or vice versa, depending on flag state.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM bytecode submitted to the VM can encode the 4-byte secp opcode. No special privilege is required. The attacker only needs to know the 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`), which are documented in the source comments. The trigger is deterministic and reproducible.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
// in the op_len == 4 branch:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both encodings of the secp operators are gated by the same flag, eliminating the bypass and restoring consistent behavior across mempool and consensus modes.

---

### Proof of Concept

1. Construct a CLVM program whose operator atom is the 4-byte atom `\x13\xd6\x1f\x00` (secp256k1_verify) with valid `pubkey msg sig` arguments.
2. Instantiate `ChiaDialect::new(ClvmFlags::empty())` — no `ENABLE_SECP_OPS`.
3. Call `run_program`. The 4-byte dispatch path fires, `op_secp256k1_verify` executes, and the program succeeds (or raises on bad sig) — **without** `ENABLE_SECP_OPS` being set.
4. Repeat with 1-byte opcode 64 under the same flags: the program falls to `unknown_operator` and is treated as a no-op (or rejected in mempool mode).

The two encodings of the same logical operator produce different outcomes under identical flags, confirming the wiring error.
