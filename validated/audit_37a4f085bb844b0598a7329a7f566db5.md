Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of originating user, allowing any caller to bypass per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the immediate caller of `pool.swap` — the router contract, not the originating user. When a pool admin allowlists the router address to enable router-mediated swaps for their curated users, every unprivileged address can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, because the extension sees `sender = router` and `allowedSwapper[pool][router] = true` for all callers indiscriminately.

## Finding Description
**Root cause:** `MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap` (L230–240). When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router contract. `ExtensionCalling._beforeSwap` encodes and forwards this `sender` value unchanged to `IMetricOmmExtensions.beforeSwap` (L162–176). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` = `allowedSwapper[pool][router]` (L37). The router stores the originating `msg.sender` only in transient callback context via `_setNextCallbackContext` for payment settlement (L71); it never encodes the originating user into `extensionData` or any other field visible to extensions. The `extensionData` forwarded to the pool is `params.extensionData`, which is user-supplied and not router-injected (L79).

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension`, allowlists `alice` and the router (`setAllowedToSwap(pool, router, true)`) to enable router-mediated swaps.
2. `charlie` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
3. Router calls `pool.swap(recipient, ...)` — `msg.sender` inside pool is `address(router)`.
4. Pool calls `_beforeSwap(address(router), ...)`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Swap executes for `charlie`, bypassing the per-user allowlist entirely.

**Why existing guards fail:** The only guard is `allowedSwapper[pool][sender]` in the extension. There is no mechanism in the pool, router, or extension to distinguish which EOA initiated the router call. The router's transient storage of the originating user is inaccessible to extensions.

## Impact Explanation
LPs on curated pools rely on `SwapAllowlistExtension` to ensure only trusted counterparties trade against their liquidity. Bypassing the allowlist exposes LP principal to adverse selection from arbitrary, unintended actors. This constitutes broken core pool functionality (the allowlist is a primary pool configuration feature) and direct loss of LP value through unintended trade exposure. The pool admin's intent — restricting swap counterparties — is completely defeated for any pool that supports router-mediated swaps.

## Likelihood Explanation
The only precondition is that the pool admin has allowlisted the router address, which is the natural and expected configuration for any curated pool intending to support the protocol's own periphery router. No malicious setup, non-standard tokens, privileged access, or special conditions are required. The trigger is a standard public call to `router.exactInputSingle`. The condition is met by any pool that has both `SwapAllowlistExtension` and router support enabled simultaneously, which is the intended production configuration.

## Recommendation
The `SwapAllowlistExtension` must not rely on `sender` (the immediate pool caller) to identify the economic actor when a router is involved. Two viable fixes:

1. **Extension-data forwarding (cleanest):** `MetricOmmSimpleRouter` encodes `msg.sender` into `extensionData` for each hop before calling `pool.swap`. `SwapAllowlistExtension.beforeSwap` decodes the originating user from `extensionData` when `sender` is a known router, and checks `allowedSwapper[pool][originatingUser]` instead.
2. **Recipient-based check:** Gate on `recipient` instead of `sender`, since `recipient` is the address receiving output tokens and is more directly tied to the economic beneficiary. This requires no router changes but changes the semantics of the allowlist.

The cleanest production fix is option 1: the router always appends the originating `msg.sender` to `extensionData` so extensions can verify the true initiator.

## Proof of Concept
```
// Setup
pool = deploy MetricOmmPool with SwapAllowlistExtension
admin calls setAllowedToSwap(pool, alice, true)   // intended allowlisted user
admin calls setAllowedToSwap(pool, router, true)  // to let alice use the router

// Attack
charlie (not allowlisted) calls:
  router.exactInputSingle({pool: pool, recipient: charlie, ...})

// Execution trace
router.exactInputSingle:
  _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, charlie, tokenIn)  // charlie stored for payment only
  pool.swap(charlie, ...)  // msg.sender = router

pool.swap:
  _beforeSwap(msg.sender=router, ...)

ExtensionCalling._beforeSwap:
  encodes IMetricOmmExtensions.beforeSwap(sender=router, ...)
  calls SwapAllowlistExtension.beforeSwap(sender=router, ...)

SwapAllowlistExtension.beforeSwap:
  allowedSwapper[pool][router] == true  ✓  // charlie bypasses allowlist
  returns selector — swap proceeds

// Result: charlie executes swap against curated LP pool with no allowlist check on charlie's address
```

A Foundry integration test can confirm this by: (1) deploying pool + extension, (2) allowlisting only `alice` and `router`, (3) calling `router.exactInputSingle` from `charlie`, and (4) asserting the swap succeeds (no `NotAllowedToSwap` revert).
