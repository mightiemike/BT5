### Title
AuxPoW Coinbase Position Off-by-One Allows Chain Root at Byte 20 — (`contract/src/dogecoin.rs`)

### Summary

`check_aux` uses `pos_chain_root <= 40` when the Dogecoin spec requires `pos_chain_root < 40`. Because `pos_chain_root` is a **hex-string offset** (2 chars per byte), position 40 corresponds to byte offset 20 — one byte past the 20-byte limit. A crafted coinbase with the chain merkle root starting at hex position 40 passes the guard and is accepted as valid.

### Finding Description

In `contract/src/dogecoin.rs`, `check_aux` searches for the chain merkle root in the parent coinbase `script_sig` as a hex string:

```rust
let script_sig = coinbase_tx.input.first().unwrap()
    .script_sig.to_hex_string();          // hex string: 2 chars per byte
let mut pos_chain_root = script_sig
    .find(&chain_root.to_string())
    .expect("Aux POW missing chain merkle root in parent coinbase");
```

When no merged-mining header is present, the position is validated as:

```rust
require!(pos_chain_root <= 40,
    "Aux POW chain merkle root must start in the first 20 bytes of the parent coinbase");
``` [1](#0-0) 

The Dogecoin reference implementation enforces `pc - script.begin() > 20` as the rejection condition, i.e. the root must start **strictly before** byte 20. In hex-string terms that is `pos_chain_root < 40`. The code uses `<= 40`, so `pos_chain_root == 40` (byte offset 20) passes when it must not.

### Impact Explanation

A coinbase whose chain merkle root begins at hex position 40 (byte 20) is accepted by the contract. This means the parent coinbase does not legitimately commit to the Dogecoin block hash in the required position, yet the block is stored as a valid canonical header. Any downstream consumer of the light client (e.g. a bridge verifying Dogecoin transaction inclusion) will treat a fraudulently constructed block as canonical, enabling false proof acceptance.

### Likelihood Explanation

The entry point is `submit_blocks`, which is guarded by `#[trusted_relayer]`. [2](#0-1) 

The `#[trusted_relayer]` macro is a staking-based system; `bypass_roles` (`DAO`, `UnrestrictedSubmitBlocks`) are not required — a staked relayer suffices. [3](#0-2) 

A malicious actor who stakes and becomes an active relayer can submit a crafted Dogecoin block with a specially positioned coinbase. The staking barrier raises the cost but does not prevent the exploit; the trusted-relayer submission path is an explicitly scoped entry point.

### Recommendation

Change the comparison from `<=` to `<`:

```rust
// Before (buggy):
require!(pos_chain_root <= 40, "...");

// After (correct):
require!(pos_chain_root < 40, "...");
```

This matches the Dogecoin spec: the chain merkle root must start **strictly within** the first 20 bytes (hex positions 0–39 inclusive).

### Proof of Concept

Construct a `script_sig` hex string where `chain_root` appears at exactly hex position 40 (byte 20), with no merged-mining header present. Call `check_aux` with this coinbase. With the current `<= 40` guard the `require!` passes; with the corrected `< 40` guard it panics.

```rust
// Pseudocode unit test
let padding = "00".repeat(20);          // 20 bytes = hex positions 0..39
let chain_root_hex = chain_root.to_string(); // 64 hex chars (32 bytes)
let suffix = "0100000000000000";        // n_size=1, n_nonce=0
let script_sig = format!("{padding}{chain_root_hex}{suffix}");
// pos_chain_root == 40  →  require!(40 <= 40) passes  ← BUG
// should be:            →  require!(40 < 40)  panics  ← correct
``` [4](#0-3)
