### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the calling pool and `sender` is the pool's `msg.sender` — the router contract — not the originating user. When a pool admin allowlists the router to support standard periphery swaps, every unpermissioned user can bypass the curated allowlist by routing through `MetricOmmSimpleRouter`.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension: [1](#0-0) 

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the end user: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

This creates an irreconcilable dilemma for the pool admin:

| Admin action | Effect |
|---|---|
| Allowlist individual users only | Allowlisted users **cannot** swap through the router (broken functionality) |
| Allowlist the router to support periphery path | **Every** user can bypass the allowlist through the router |

There is no configuration that simultaneously enforces per-user access control and supports the standard router path.

The `DepositAllowlistExtension` is not affected in the same way because it gates on `owner` (the position recipient), which the liquidity adder passes explicitly and which the pool enforces separately. [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd users, whitelisted market makers) is fully bypassed once the pool admin allowlists the router. Any unpermissioned user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps on the curated pool, draining LP value at oracle-derived prices without authorization. This is a direct loss of the access-control invariant the extension was deployed to enforce, with fund-impacting consequences for LPs who deposited under the assumption that only vetted counterparties could trade.

---

### Likelihood Explanation

Medium. The router is the canonical periphery swap path documented and deployed by the protocol. A pool admin who wants to support standard tooling (wallets, aggregators, front-ends) while maintaining an allowlist will naturally attempt to allowlist the router address. The admin has no on-chain signal that doing so opens the allowlist to all users; the `isAllowedToSwap` view function returns `true` for the router, giving no indication of the bypass. The scenario requires a deliberate admin action, but it is the expected action for any operator who wants both access control and router compatibility.

---

### Recommendation

Pass the originating user's address through the swap path so the extension can gate on the actual economic actor. Two complementary approaches:

1. **Router-side**: Add an optional `originSender` field to swap parameters; the router populates it with `msg.sender` and passes it in `extensionData`. The extension decodes and checks it.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should accept an encoded `originSender` from `extensionData` and fall back to `sender` only when no override is present, so router-mediated swaps carry the real user identity.

Either approach closes the gap between the actor the allowlist intends to gate and the actor the extension actually observes.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  pool admin calls setAllowedToSwap(pool, router, true)   // to support periphery path
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool: pool,
        recipient: attacker,
        zeroForOne: true,
        amountIn: X,
        ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient=attacker, ...)   // msg.sender = router
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes at oracle price
        → attacker receives token output

  Result: attacker swaps on a curated pool without being individually allowlisted.
``` [5](#0-4) [6](#0-5) [7](#0-6)

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
