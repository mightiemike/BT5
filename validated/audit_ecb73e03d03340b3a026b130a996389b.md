### Title
Unprotected `createDirectDepositV1` Allows Any Caller to Pre-Register Direct Deposit Addresses for Arbitrary Subaccounts - (File: core/contracts/ContractOwner.sol)

### Summary
`createDirectDepositV1` in `ContractOwner.sol` is declared `public` with no access control. Any unprivileged caller can invoke it for any `subaccount` value, deploying a `DirectDepositV1` contract via CREATE2 and permanently writing the result into `directDepositV1Address[subaccount]`. Because the CREATE2 salt is hardcoded to `bytes32(uint256(1))`, the deployment address is deterministic and the slot can only ever be filled once per subaccount. This permanently locks the DDA configuration for that subaccount before the protocol or the subaccount owner has any say.

### Finding Description
`createDirectDepositV1` at line 486 of `ContractOwner.sol` is `public` with no modifier:

```solidity
function createDirectDepositV1(bytes32 subaccount)
    public
    returns (address payable)
{
    require(
        getDirectDepositV1BytecodeHash() ==
            0x7974df41bdca2be1539fa7d01f41277f0d728823b20230a18a31e40c707874e7,
        "dda hash"
    );
    DirectDepositV1 directDepositV1 = new DirectDepositV1{
        salt: bytes32(uint256(1))
    }(address(endpoint), address(spotEngine), subaccount, wrappedNative);
    directDepositV1Address[subaccount] = payable(directDepositV1);
    return payable(directDepositV1);
}
``` [1](#0-0) 

The fixed salt `bytes32(uint256(1))` means the CREATE2 address for a given `subaccount` is fully deterministic and can only be deployed once. Once an attacker calls this for a target subaccount, `directDepositV1Address[subaccount]` is permanently set and the slot cannot be rewritten (a second CREATE2 with identical parameters reverts). [2](#0-1) 

Both `creditDepositV1` and `wrapVaultAsset` check `directDepositV1Address[subaccount] == address(0)` before calling `createDirectDepositV1`, so they will silently consume the attacker-pre-created DDA without any indication that it was externally forced: [3](#0-2) 

```solidity
function creditDepositV1(bytes32 subaccount) external {
    address payable directDepositV1 = directDepositV1Address[subaccount];
    if (directDepositV1 == address(0)) {
        directDepositV1 = createDirectDepositV1(subaccount);
    }
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [3](#0-2) 

### Impact Explanation
The `DirectDepositV1` constructor sets its `Ownable` owner to `msg.sender`, which is always `ContractOwner` (not the external attacker), so the attacker cannot call `withdraw` or `withdrawNative` on the DDA. [4](#0-3) 

However, the concrete protocol-level impact is:

1. **Permanent DDA configuration lock**: Because the salt is fixed, the CREATE2 address for a given `(ContractOwner, salt=1, bytecode(subaccount, endpoint, spotEngine, wrappedNative))` tuple is unique and immutable. If `endpoint` or `spotEngine` is later upgraded (both are upgradeable proxies), the pre-created DDA still points to the old addresses. The protocol cannot redeploy a fresh DDA for that subaccount — the CREATE2 call reverts and `directDepositV1Address[subaccount]` already holds the stale address. Any subsequent `creditDepositV1` or `wrapVaultAsset` call for that subaccount will use the stale DDA, which interacts with the old `endpoint`/`spotEngine` addresses. [5](#0-4) 

2. **Mapping pollution at scale**: An attacker can pre-register DDAs for every possible subaccount (or targeted high-value ones), permanently fixing the DDA configuration across the protocol before any legitimate user or operator action.

### Likelihood Explanation
The function is `public` with no guard. Any EOA or contract can call it with zero preconditions. The only cost is gas. The attack is trivially executable on mainnet at any time before a target subaccount's DDA is created.

### Recommendation
Add an access control modifier to `createDirectDepositV1`. The simplest fix matching the external report's recommendation is to make it `internal` and call it only from `creditDepositV1` and `wrapVaultAsset`, or restrict it to `onlyOwner`/`onlyDeployer`:

```solidity
// Option A: make internal
function createDirectDepositV1(bytes32 subaccount)
    internal   // was: public
    returns (address payable)

// Option B: restrict to deployer
function createDirectDepositV1(bytes32 subaccount)
    public
    onlyDeployer
    returns (address payable)
```

Additionally, consider using a per-subaccount salt (e.g., `salt = subaccount`) instead of the hardcoded `bytes32(uint256(1))` so that future redeployments with updated constructor arguments remain possible.

### Proof of Concept
```solidity
// Attacker pre-registers a DDA for victim's subaccount
bytes32 victimSubaccount = bytes32(abi.encodePacked(victimAddress, bytes12("default")));

// No access control — succeeds for any caller
address dda = ContractOwner(contractOwner).createDirectDepositV1(victimSubaccount);

// directDepositV1Address[victimSubaccount] is now permanently set.
// If endpoint/spotEngine are later upgraded, the protocol cannot create
// a fresh DDA for victimSubaccount — CREATE2 reverts (address already deployed),
// and creditDepositV1 / wrapVaultAsset silently use the stale DDA.
```
