### Title
Consensus `validate_proposal` path bypasses `StatelessTransactionValidator` DA-mode and resource-bound checks, allowing OS-unprovable transactions to be committed — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `StatelessTransactionValidator` enforces several invariants on transactions entering via the gateway (RPC path), including that `nonce_data_availability_mode` and `fee_data_availability_mode` must be `DataAvailabilityMode::L1`. The validator's own comments state these checks exist specifically "to prevent transactions from failing the OS." However, transactions entering via the consensus proposal path bypass `StatelessTransactionValidator` entirely. A malicious proposer can include transactions with `DataAvailabilityMode::L2` DA modes in a consensus proposal; the validator node accepts, executes, and commits them. Since the Starknet OS enforces L1 DA mode, the committed block becomes unprovable.

This is a direct structural analog to the `veRAACToken` bug: `lock()` (gateway path) enforces the cap; `increase()` (consensus path) does not.

---

### Finding Description

**Gateway path — check enforced:**

`StatelessTransactionValidator::validate()` is called for every `RpcTransaction` entering through the gateway: [1](#0-0) 

```rust
fn validate_nonce_data_availability_mode(&self, tx: &RpcTransaction) -> ... {
    let expected_da_mode = DataAvailabilityMode::L1;
    let da_mode = *tx.nonce_data_availability_mode();
    if da_mode != expected_da_mode {
        return Err(StatelessTransactionValidatorError::InvalidDataAvailabilityMode { ... });
    }
    Ok(())
}
```

The comment is explicit: *"The Starknet OS enforces that the nonce data availability mode is L1. We add this validation here in the gateway to prevent transactions from failing the OS."* [2](#0-1) 

**Consensus path — check absent:**

When a validator node receives a proposal from a peer, `handle_proposal_part()` converts each `ConsensusTransaction` directly: [3](#0-2) 

This calls `convert_consensus_tx_to_internal_consensus_tx()`: [4](#0-3) 

Which delegates to `convert_rpc_tx_to_internal()`: [5](#0-4) 

`convert_rpc_tx_to_internal()` only:
- Computes the transaction hash
- For `Declare`: validates `compiled_class_hash`
- For `Invoke`: extracts proof data

It does **not** call `StatelessTransactionValidator::validate()`. There is no check for `nonce_data_availability_mode`, `fee_data_availability_mode`, `max_l2_gas_amount`, `max_calldata_length`, `max_signature_length`, `validate_empty_paymaster_data`, or `validate_empty_account_deployment_data`.

**Blockifier does not fill the gap:**

The blockifier's `perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — not DA modes: [6](#0-5) 

The gateway comment confirms the blockifier does not check DA modes; only the OS does.

**The DA mode is included in the transaction hash:**

`nonce_data_availability_mode` and `fee_data_availability_mode` are committed into the transaction hash via `concat_data_availability_mode`: [7](#0-6) 

So a malicious proposer can craft a fully signed, hash-consistent transaction with `L2` DA mode, include it in a proposal, and the validator will accept it.

---

### Impact Explanation

A malicious proposer crafts an `RpcInvokeTransactionV3` with `nonce_data_availability_mode: DataAvailabilityMode::L2`, signs it correctly (the signature covers the hash, which includes the DA mode), and includes it in a consensus proposal. The validator's `handle_proposal_part()` accepts it, the blockifier executes it without checking DA mode, and the transaction is committed to the block. The Starknet OS then fails to prove the block because it enforces L1 DA mode. The committed block is unprovable — wrong state from blockifier/execution logic for accepted input.

This matches: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

The same bypass applies to `validate_empty_paymaster_data` and `validate_empty_account_deployment_data`, both of which are also OS-enforced invariants that the gateway adds "to prevent transactions from failing the OS."

---

### Likelihood Explanation

The trigger requires a malicious consensus proposer (a staked validator). Within the BFT threat model, up to `f` Byzantine validators are assumed possible. A single Byzantine proposer in their assigned round can inject one such transaction per proposal. No special privilege beyond being a scheduled proposer is required.

---

### Recommendation

Apply the equivalent of `StatelessTransactionValidator` checks (at minimum DA mode and empty-field invariants) inside `convert_rpc_tx_to_internal()` or as a dedicated validation step in `handle_proposal_part()` before forwarding transactions to the batcher. Alternatively, add DA mode validation to the blockifier's `perform_pre_validation_stage` so the invariant is enforced regardless of the entry path — mirroring how `validate_proof_facts` is enforced at the blockifier level for all paths.

---

### Proof of Concept

```
1. Attacker is a scheduled proposer for height H, round R.

2. Attacker crafts:
     RpcInvokeTransactionV3 {
         nonce_data_availability_mode: DataAvailabilityMode::L2,  // OS-invalid
         fee_data_availability_mode:   DataAvailabilityMode::L1,
         ... (all other fields valid)
     }
   Signs it with the correct account key (signature covers the hash,
   which includes the L2 DA mode — so the signature is valid for this tx).

3. Attacker wraps it as ConsensusTransaction::RpcTransaction and streams
   it in a ProposalPart::Transactions batch to all validators.

4. Each validator's handle_proposal_part() calls:
     transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
   → convert_rpc_tx_to_internal()
   → computes hash, no DA mode check, returns Ok(InternalRpcTransaction)

5. The internal tx is forwarded to the batcher via send_txs_for_proposal().
   The blockifier executes it; perform_pre_validation_stage() does not check
   DA mode. Execution succeeds.

6. The block is committed with this transaction.

7. The Starknet OS attempts to prove the block. It enforces L1 DA mode
   (see check_proof_facts / OS execution constraints). Proving fails.
   The committed block is unprovable — chain halts at height H.
``` [1](#0-0) [3](#0-2) [5](#0-4) [6](#0-5)

    async fn convert_rpc_tx_to_internal(
        &self,
        tx: RpcTransaction,
    ) -> TransactionConverterResult<(InternalRpcTransaction, Option<(ProofFacts, Proof)>)> {
        let (tx_without_hash, proof_data) = match tx {
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
                let proof_data = if tx.proof_facts.is_empty() {
                    None
                } else {
                    Some((tx.proof_facts.clone(), tx.proof.clone()))
                };
                (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
            }
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
                let ClassHashes { class_hash, executable_class_hash_v2 } =
                // TODO(Dori): Make this async and spawn a task to compile and add it to the class manager.
                    self.class_manager_client.add_class(tx.contract_class).await?;
                // TODO(Aviv): Ensure that we do not want to
                // allow declare with compiled class hash v1.
                if tx.compiled_class_hash != executable_class_hash_v2 {
                    return Err(TransactionConverterError::ValidateCompiledClassHashError(
                        ValidateCompiledClassHashError::CompiledClassHashMismatch {
                            computed_class_hash: executable_class_hash_v2,
                            supplied_class_hash: tx.compiled_class_hash,
                        },
                    ));
                }
                (
                    InternalRpcTransactionWithoutTxHash::Declare(InternalRpcDeclareTransactionV3 {
                        sender_address: tx.sender_address,
                        compiled_class_hash: tx.compiled_class_hash,
                        signature: tx.signature,
                        nonce: tx.nonce,
                        class_hash,
                        resource_bounds: tx.resource_bounds,
                        tip: tx.tip,
                        paymaster_data: tx.paymaster_data,
                        account_deployment_data: tx.account_deployment_data,
                        nonce_data_availability_mode: tx.nonce_data_availability_mode,
                        fee_data_availability_mode: tx.fee_data_availability_mode,
                    }),
                    None,
                )
            }
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                let contract_address = tx.calculate_contract_address()?;
                (
                    InternalRpcTransactionWithoutTxHash::DeployAccount(
                        InternalRpcDeployAccountTransaction {
                            tx: RpcDeployAccountTransaction::V3(tx),
                            contract_address,
                        },
                    ),
                    None,
                )
            }
        };
        let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
        Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
    }
