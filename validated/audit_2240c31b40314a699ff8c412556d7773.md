### Title
`ValidResourceBounds` Variant Lost in Protobuf Round-Trip Causes Consensus Proposal Rejection for Zero-Gas Transactions — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a zero-value heuristic to choose between the `L1Gas` and `AllResources` variants. When an `AllResources` transaction with `l2_gas = 0` and `l1_data_gas = 0` is serialized and sent over the consensus P2P channel, the receiving validator reconstructs it as `L1Gas`. The subsequent `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion then fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing the validator to reject the entire block proposal. Any unprivileged user can trigger this by submitting a gateway-accepted transaction with zero L2/data-gas bounds.

---

### Finding Description

**Root cause — variant-erasing protobuf deserialization**

`ValidResourceBounds` has two variants with different hash semantics:

```
ValidResourceBounds::L1Gas(_)          // pre-0.13.3
ValidResourceBounds::AllResources(_)   // 0.13.3+
```

The hash function `get_tip_resource_bounds_hash` produces structurally different outputs for the two variants even when the numeric values are identical: [1](#0-0) 

For `AllResources`, `L1_DATA_GAS` is appended to the hash input; for `L1Gas`, it is not. Two transactions that are byte-for-byte identical in their resource fields therefore hash to different values depending solely on which variant tag was chosen.

The protobuf deserializer reconstructs the variant from the wire values using a zero-value heuristic: [2](#0-1) 

If `l2_gas = 0` **and** `l1_data_gas = 0`, the result is `L1Gas`; otherwise `AllResources`. There is no explicit discriminator field in the protobuf schema. This means an `AllResources` transaction with both fields zeroed is **indistinguishable** from an `L1Gas` transaction on the wire.

**Broken conversion on the validator side**

`RpcInvokeTransactionV3` (the consensus-layer type) requires `AllResources`: [3](#0-2) 

The consensus protobuf deserializer calls this conversion: [4](#0-3) 

The comment on line 129 explicitly states: *"This conversion can fail only if the resource_bounds are not AllResources."* When the heuristic produces `L1Gas`, the `try_into()` call returns `DEPRECATED_RESOURCE_BOUNDS_ERROR` and the entire `ConsensusTransaction` deserialization fails.

**The test suite acknowledges the defect**

The consensus round-trip test works around the issue by forcibly setting `l2_gas.max_amount = GasAmount(1)`: [5](#0-4) 

The comment reads: *"If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant."* No equivalent guard exists in production code.

**End-to-end attack path**

1. User submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` to the gateway.
2. Gateway accepts it (no validation rejects zero L2/data-gas). `convert_rpc_tx_to_internal` computes the hash via the `AllResources` path — hash includes `L1_DATA_GAS` concatenation. [6](#0-5) 

3. Proposer calls `convert_internal_consensus_tx_to_consensus_tx`, reconstructing `RpcInvokeTransactionV3 { resource_bounds: AllResourceBounds { l2_gas: 0, l1_data_gas: 0 } }`. [7](#0-6) 

4. Protobuf serialization emits `ResourceBounds { l2_gas: 0, l1_data_gas: 0 }`.
5. Validator deserializes: `ValidResourceBounds::L1Gas` (heuristic fires). `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
6. `convert_consensus_tx_to_internal_consensus_tx` returns an error; the proposal is rejected. [8](#0-7) 

---

### Impact Explanation

A single unprivileged user can submit a gateway-valid transaction that, once included in a block proposal, causes every validator to reject that proposal. Repeated submissions prevent the proposer from ever committing a block, constituting a targeted consensus liveness failure. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger condition — `l2_gas = 0` and `l1_data_gas = 0` — is reachable by any user. No special privilege, key material, or network position is required. The gateway performs no validation that would reject such bounds. The test workaround (`GasAmount(1)`) demonstrates the condition is easily hit.

---

### Recommendation

1. **Add an explicit discriminator to the protobuf schema.** Introduce a boolean or enum field (e.g., `is_all_resources`) in `ResourceBounds` so the variant survives serialization without relying on zero-value inference.

2. **Normalize at the gateway boundary.** Before computing the transaction hash, convert any `AllResources` with `l2_gas = 0` and `l1_data_gas = 0` to `L1Gas`, so the variant is canonical and consistent across all nodes.

3. **Add gateway validation.** Reject `AllResources` transactions where both `l2_gas` and `l1_data_gas` are zero, forcing users to either use `L1Gas` semantics explicitly or supply non-zero bounds.

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
