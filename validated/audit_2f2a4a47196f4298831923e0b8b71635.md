### Title
Compromised Linked Signer Can Irrevocably Retain Subaccount Control via Self-Re-Delegation — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the fast-mode `LinkSigner` transaction handler calls `validateSignedTx` with `allowLinkedSigner = true`. This permits the **current linked signer** — not just the subaccount owner — to sign and submit a `LinkSigner` transaction that replaces themselves with any address they control. A compromised linked signer can therefore continuously re-delegate their privilege to a fresh address before the subaccount owner's removal transaction is processed, making revocation practically impossible and maintaining persistent unauthorized control over the subaccount.

---

### Finding Description

`EndpointStorage.sol` stores a `linkedSigners` mapping (`bytes32 subaccount → address signer`) that grants a delegated address the ability to sign sensitive transactions — including `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, and `BurnNlp` — on behalf of a subaccount. [1](#0-0) 

In `processTransactionImpl` (fast-mode path), the `LinkSigner` branch decodes a `SignedLinkSigner` and calls `validateSignedTx` with the fifth argument hardcoded to `true`: [2](#0-1) 

`validateSignedTx` with `allowLinkedSigner = true` passes `getLinkedSigner(sender)` as the accepted signer to `verifier.validateSignature`: [3](#0-2) 

`Verifier.validateSignature` accepts the recovered address if it equals **either** the subaccount owner address **or** the linked signer: [4](#0-3) 

Therefore, the current linked signer can produce a valid `SignedLinkSigner` that overwrites `linkedSigners[subaccount]` with any address they choose — including a fresh address they own — without the subaccount owner's knowledge or consent.

The slow-mode `LinkSigner` path is correctly restricted: it uses `validateSender`, which enforces `msg.sender == address(uint160(bytes20(txn.sender)))`, so only the actual subaccount owner can invoke it. [5](#0-4) 

However, the slow-mode path carries a hardcoded **3-day delay** before execution: [6](#0-5) 

This 3-day window gives a compromised linked signer ample time to re-delegate via the fast-mode path before the owner's slow-mode removal is processed.

---

### Impact Explanation

A compromised linked signer can:

1. Submit a fast-mode `LinkSigner` transaction (signed by themselves, valid because `allowLinkedSigner = true`) to replace themselves with attacker-controlled address B.
2. When the subaccount owner submits a fast-mode `LinkSigner` to remove the signer, both transactions compete for the same nonce slot (`nonces[address(uint160(bytes20(sender)))]`). If the attacker's transaction is sequenced first, the owner's transaction fails with `ERR_WRONG_NONCE`.
3. The owner increments to nonce N+1; the attacker (now using address B) submits another re-delegation to address C. This cycle repeats indefinitely.
4. Throughout this period, the attacker retains the ability to sign `WithdrawCollateral` / `WithdrawCollateralV2` / `TransferQuote` transactions, draining the subaccount's collateral.

The corrupted state is `linkedSigners[subaccount]` — it is continuously mutated to attacker-controlled addresses, preventing the legitimate owner from ever achieving a clean revocation in the fast-mode path. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

Linked signers are by design hot-wallet API keys used for high-frequency trading. Hot wallets are the most common target of key compromise. Once a linked signer key is leaked, the attacker has an immediate, on-chain mechanism to perpetuate their access indefinitely. The subaccount owner's only reliable revocation path (slow mode) is delayed by 3 days, during which the attacker can drain funds and re-delegate repeatedly. No sequencer collusion is required for the initial re-delegation — the attacker simply submits a valid signed transaction.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — linked signer must NOT be able to re-delegate itself
);
```

This restricts `LinkSigner` mutations to the subaccount owner's key only, matching the invariant already enforced in the slow-mode path via `validateSender`. A compromised linked signer would then be unable to prevent revocation. [9](#0-8) 

---

### Proof of Concept

1. Owner (`0xALICE`) links signer `0xSIGNER_A` to subaccount `ALICE_SUB` via a `LinkSigner` fast-mode transaction. `linkedSigners[ALICE_SUB] = 0xSIGNER_A`.
2. Attacker compromises `0xSIGNER_A`.
3. Attacker constructs `SignedLinkSigner { sender: ALICE_SUB, signer: 0xATTACKER_B, nonce: N }` and signs it with `0xSIGNER_A`'s key.
4. Attacker submits this to the sequencer. `validateSignedTx(..., true)` recovers `0xSIGNER_A` == `getLinkedSigner(ALICE_SUB)` → valid. `linkedSigners[ALICE_SUB]` is set to `0xATTACKER_B`.
5. Owner submits `SignedLinkSigner { sender: ALICE_SUB, signer: 0x0, nonce: N+1 }` signed by `0xALICE` to revoke.
6. Attacker (now using `0xATTACKER_B`) submits `SignedLinkSigner { sender: ALICE_SUB, signer: 0xATTACKER_C, nonce: N+1 }`. If sequenced first, owner's transaction fails.
7. Attacker simultaneously submits `WithdrawCollateral { sender: ALICE_SUB, ... }` signed by `0xATTACKER_B` to drain collateral before the owner can recover. [2](#0-1) [4](#0-3) [10](#0-9)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
