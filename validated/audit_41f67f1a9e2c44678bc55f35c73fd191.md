### Title
Child `ChunkExtra.proposed_split` Inherits Parent's Stale Split Proposal Across Resharding Boundary — (`File: chain/chain/src/resharding/manager.rs`)

### Summary

When a shard is split during dynamic resharding, `process_memtrie_resharding_storage_update` creates child `ChunkExtra` records by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — which encodes the parent shard's trie-split proposal — is **not reset to `None`** in the child `ChunkExtra`. This stale value is then used as the authoritative reference for validating the first chunk header produced by each child shard. Because the first real chunk of a child shard will compute `proposed_split = None` (the child is a fresh shard, well below any split threshold), the chunk header validation in `validate_chunk_with_chunk_extra_and_receipts_root` will reject every first chunk of every child shard as `InvalidChunkHeaderShardSplit`, halting block production on those shards after every dynamic resharding event.

### Finding Description

In `process_memtrie_resharding_storage_update` (`chain/chain/src/resharding/manager.rs`, lines 255–266), the child `ChunkExtra` is constructed by cloning the parent and selectively overwriting two fields:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
```

The `proposed_split` field of `ChunkExtraV5` is **not cleared**. It retains whatever `Option<TrieSplit>` the parent shard had at the resharding boundary block — which, by design, is `Some(TrieSplit { boundary_account, left_memory, right_memory })` because the parent's split proposal is what triggered the resharding in the first place.

This child `ChunkExtra` is saved to `DBCol::ChunkExtra` keyed by `(block_hash, child_shard_uid)` and becomes the `prev_chunk_extra` for the first chunk of the child shard in the next epoch.

When the first chunk of the child shard is produced and validated, `validate_chunk_with_chunk_extra_and_receipts_root` (`chain/chain/src/validate.rs`, lines 176–185) enforces:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
```

The chunk header's `proposed_split` will be `None` — the child shard is new, its memory usage is far below any split threshold, and `compute_proposed_split` returns `None`. But `prev_chunk_extra.proposed_split()` is `Some(parent_split)`. The mismatch causes `InvalidChunkHeaderShardSplit`, and the first chunk of every child shard is rejected.

The `proposed_split` field exists in `ChunkExtraV5` (`core/primitives/src/types.rs`, lines 879–900) and is the authoritative source for chunk header validation. The TODO comment at line 255 of `manager.rs` explicitly acknowledges that not all fields are set, but does not identify `proposed_split` as one of the missing ones.

### Impact Explanation

After every dynamic resharding event, both child shards are unable to produce their first valid chunk. The chunk producer for each child shard will produce a chunk header with `proposed_split = None`, which will be rejected by all validators as `InvalidChunkHeaderShardSplit`. This causes the child shards to stall at the epoch boundary — no chunks can be included, no transactions processed, and no state transitions occur on those shards. The network cannot recover without a protocol-level fix or a manual intervention to clear the stale `proposed_split` from the stored `ChunkExtra`. Since dynamic resharding is designed to trigger automatically based on memory thresholds, this is a reachable production path.

### Likelihood Explanation

The trigger is automatic: any time a shard's trie memory usage exceeds `memory_usage_threshold` and `DynamicResharding` is enabled, a split is proposed and eventually executed. The parent shard's final chunk will always have a non-`None` `proposed_split` (that is the value that caused the split). The child `ChunkExtra` always inherits it. The first chunk of each child shard always computes `None`. The collision is deterministic and 100% reproducible on every dynamic resharding event.

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra` and before saving it for the child, explicitly reset `proposed_split` to `None`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None;  // child starts fresh, no split pending
```

This requires adding a `proposed_split_mut()` accessor to `ChunkExtra` analogous to the existing `state_root_mut()` and `congestion_info_mut()`. The existing TODO comment at line 255 should be updated to enumerate all fields that require explicit initialization rather than inheritance from the parent.

### Proof of Concept

**Step 1 — Resharding boundary block (epoch N, last block):**
- Parent shard P has `memory_usage >= memory_usage_threshold`.
- `compute_proposed_split` returns `Some(TrieSplit { boundary_account: "mid", left_memory: X, right_memory: Y })`.
- This is stored in `ChunkExtra` for P at the boundary block and embedded in the chunk header.
- `get_upcoming_shard_split` selects P for splitting; `shard_split = Some((P_id, "mid"))` is embedded in the block header.

**Step 2 — `finalize_epoch` (epoch N → N+1 boundary):**
- `next_next_shard_layout` derives a new `ShardLayoutV3` splitting P into children L and R.
- `EpochInfoV5.last_resharding` is set to the new epoch height.

**Step 3 — `process_memtrie_resharding_storage_update` (epoch N+1, first block):**
- `child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra)` — inherits `proposed_split = Some(TrieSplit { "mid", X, Y })`.
- Only `state_root` and `congestion_info` are overwritten.
- `child_chunk_extra` with stale `proposed_split = Some(...)` is saved for both L and R.

**Step 4 — First chunk of child shard L (epoch N+2, first block):**
- Chunk producer runs `compute_proposed_split` for L: L is a new shard, `memory_usage << memory_usage_threshold`, returns `None`.
- Chunk header for L has `proposed_split = None`.
- `validate_chunk_with_chunk_extra_and_receipts_root` is called with `prev_chunk_extra` = the stale child `ChunkExtra` (from Step 3).
- `prev_chunk_extra.proposed_split() = Some(TrieSplit { "mid", X, Y })` ≠ `chunk_header.proposed_split() = None`.
- Returns `Err(Error::InvalidChunkHeaderShardSplit(...))`.
- All validators reject the chunk. Child shard L (and R) cannot produce their first chunk. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
