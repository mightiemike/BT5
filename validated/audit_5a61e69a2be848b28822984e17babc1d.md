### Title
Zero `l2_gas`/`l1_data_gas` in `AllResources` Silently Downgrades to `L1Gas` in Protobuf Conversion, Producing a Divergent Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-internal conversion for `ValidResourceBounds` uses a zero-value heuristic to decide between `L1Gas` and `AllResources`. When a post-0.13.3 `AllResources` transaction carries `l2_gas = 0` and `l1_data_gas = 0`, the converter silently produces `ValidResourceBounds::L1Gas`. Because `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage only for `AllResources`, the hash computed after P2P receipt diverges from the hash committed by the originating node. Any downstream consumer that re-derives the hash from the stored `InvokeTransactionV3` — including `validate_transaction_hash`, RPC trace/simulation re-execution, and external clients — observes a hash that does not match the authoritative stored value.

### Finding Description

**Conversion boundary — `crates/apollo_protobuf/src/converters/transaction.rs`**

```rust
// line 431
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources txs with zero fields
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The same zero-value downgrade exists in the RPC layer:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs line 190
if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
    Self::L1Gas(value.l1_gas)
}
```

**Hash domain — `crates/starknet_api/src/transaction_hash.rs`**

`get_tip_resource_bounds_hash` branches on the enum variant, not on the numeric values:

```rust
// line 203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2-element preimage
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3-element preimage
    }
});
```

For a transaction with `l2_gas = 0, l1_data_gas = 0`:

| Variant | Poseidon preimage |
|---|---|
| `AllResources` (originating node) | `[tip, l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)]` |
| `L1Gas` (after protobuf round-trip) | `[tip, l1_gas_packed, l2_gas_packed(=0)]` |

These produce **different hashes** (H1 ≠ H2) for identical numeric field values.

**Attack path**

1. A user submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`. The gateway accepts it; `InternalRpcInvokeTransactionV3` always wraps in `AllResources`.
2. The gateway computes and stores hash **H1** via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(AllResources, …)` — 3-element preimage.
3. The transaction is committed in a block and propagated via P2P as protobuf `InvokeV3`.
4. A syncing node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`; since `l2_gas.is_zero() && l1_data_gas.is_zero()`, it produces `ValidResourceBounds::L1Gas`.
5. The syncing node stores `InvokeTransactionV3 { resource_bounds: L1Gas(X), … }` alongside the peer-supplied hash **H1**.
6. Any call to `validate_transaction_hash` on the syncing node recomputes **H2** (2-element preimage) and returns `false` for a valid transaction.
7. The RPC converts `L1Gas` → `ResourceBoundsMapping { l2_gas: 0, l1_data_gas: 0 }` (via `From<ValidResourceBounds> for ResourceBoundsMapping`). A client that re-derives `ValidResourceBounds` from this mapping again gets `L1Gas` and computes **H2 ≠ H1**, making the authoritative stored hash appear invalid.

### Impact Explanation

This is a **High** impact finding matching: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

- `validate_transaction_hash` returns `false` for valid transactions stored on syncing nodes, breaking any integrity check that calls it.
- The RPC serves a `(transaction_data, hash)` pair where the hash is not reproducible from the data using the standard `ResourceBoundsMapping → ValidResourceBounds` path, making the node's responses untrustworthy for any client that verifies hashes.
- `starknet_traceTransaction` re-executes from the stored `InvokeTransactionV3`; with `L1Gas` instead of `AllResources`, the fee-bounds variant used during re-execution diverges from the original execution, potentially producing wrong trace/receipt values.

### Likelihood Explanation

Any unprivileged user can craft a valid `RpcInvokeTransactionV3` with `l2_gas = 0` and `l1_data_gas = 0` (both are zero by default in `AllResourceBounds::default()`). The gateway accepts such a transaction. Every syncing node that receives the committed block via P2P will silently misclassify it. No special privileges, malformed bytes, or peer compromise are required.

### Recommendation

Remove the zero-value heuristic from both converters. The `AllResources` / `L1Gas` distinction is a **semantic** property of the transaction version, not derivable from whether the numeric fields happen to be zero. Options:

1. **Always produce `AllResources`** in `TryFrom<protobuf::ResourceBounds>` when `l1_data_gas` is present in the protobuf message (even if zero), reserving `L1Gas` only for the legacy case where `l1_data_gas` is absent (`None`). The existing `unwrap_or_default()` already distinguishes these two cases:

```rust
// Proposed fix
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),   // legacy 0.13.2 wire format
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

2. Apply the same fix to `From<ResourceBoundsMapping> for ValidResourceBounds` in `crates/apollo_rpc/src/v0_8/transaction.rs`, or add an explicit version tag to `ResourceBoundsMapping` so the variant is not inferred from numeric values.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, GasAmount, GasPrice, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;

let l1_gas = ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) };
let tip = Tip(0);

// Hash as computed by the originating node (AllResources, zero l2/l1_data)
let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas,
    l2_gas: ResourceBounds::default(),      // zero
    l1_data_gas: ResourceBounds::default(), // zero
});
let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();

// Hash as computed after protobuf round-trip (L1Gas, because both zero fields trigger downgrade)
let l1_gas_variant = ValidResourceBounds::L1Gas(l1_gas);
let h2 = get_tip_resource_bounds_hash(&l1_gas_variant, &tip).unwrap();

// H1 ≠ H2 — same numeric values, different hash domain
assert_ne!(h1, h2, "hash diverges after protobuf downgrade");
```

The divergence is confirmed by the `get_tip_resource_bounds_hash` implementation: `AllResources` chains `l1_data_gas_packed` into the Poseidon state; `L1Gas` does not. [1](#0-0) [2](#0-1) [3](#0-2)
