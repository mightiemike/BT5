### Title
`SwapAllowlistExtension` gates by direct caller (`sender`) instead of the economic actor, allowing any user to bypass a curated pool's swap allowlist via the allowlisted router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router address, not the user. If the router is allowlisted (which is required for any user to use it), every user — including non-allowlisted ones — passes the guard. The `DepositAllowlistExtension` avoids this by checking `owner` (the economic actor), but `SwapAllowlistExtension` has no equivalent binding.

---

### Finding Description

**Pool passes `msg.sender` as `sender` to the extension:** [1](#0-0) 

```solidity
_beforeSwap(
    msg.sender,   // ← direct caller of pool.swap(), not the end-user
    recipient,
    ...
);
```

**Extension checks `sender` (the direct caller), not the end-user:** [2](#0-1) 

```solidity
function beforeSwap(address sender, address, ...)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
```

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap(...)`. The pool sets `sender = router`. The extension then evaluates `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for **every** user who routes through it, regardless of whether that user is individually allowlisted.

**Contrast with `DepositAllowlistExtension`, which correctly checks `owner` (the economic actor):** [3](#0-2) 

```solidity
function beforeAddLiquidity(address, address owner, ...)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```

`DepositAllowlistExtension` ignores `sender` and checks `owner`, which the `LiquidityAdder` passes as the user's address. `SwapAllowlistExtension` has no equivalent — it only sees the direct caller.

**The admin faces an impossible choice:**

| Router allowlisted? | Outcome |
|---|---|
| No | Legitimate allowlisted users cannot use the router at all |
| Yes | Every user bypasses the per-user allowlist via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users.

---

### Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) cannot enforce that restriction when the router is in use. Any unprivileged user can call `router.exactInputSingle()` and trade in the restricted pool, bypassing the intended access control. This breaks the core curation guarantee of the extension and allows unauthorized value extraction from pools that rely on restricted swap access for their economic model.

---

### Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool admin who wants their allowlisted users to be able to use the router must allowlist the router address. This is the expected operational configuration, making the bypass reachable in every realistic curated-pool deployment that uses the router.

---

### Recommendation

Change `SwapAllowlistExtension.beforeSwap` to check `recipient` (the address receiving the output tokens, which the router sets to the actual user) rather than `sender` (the direct caller):

```solidity
// Before (wrong actor):
function beforeSwap(address sender, address, ...)
    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {

// After (economic actor):
function beforeSwap(address, address recipient, ...)
    ...
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
```

This mirrors the pattern used by `DepositAllowlistExtension`, which correctly gates by the economic actor (`owner`) rather than the direct caller. Alternatively, the router could be extended to pass the originating user's address through `extensionData`, and the extension could decode and verify it — but checking `recipient` is the minimal, consistent fix.

---

### Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is allowed to swap.
3. Admin calls `setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. Bob (not allowlisted) calls `router.exactInputSingle({..., recipient: bob})`.
5. Router calls `pool.swap(recipient=bob, ...)`.
6. Pool calls `extension.beforeSwap(sender=router, recipient=bob, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` → `true` → swap proceeds.
8. Bob successfully swaps in the restricted pool, bypassing the per-user allowlist. [4](#0-3) [5](#0-4) [6](#0-5)
