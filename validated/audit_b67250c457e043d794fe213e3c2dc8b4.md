### Title
`getSlowModeFee()` Hardcodes 6-Decimal Assumption for Quote Token, Causing `slowModeFees` Accounting Corruption for Non-6-Decimal Tokens — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getSlowModeFee()` scales the slow-mode fee to the quote token's native decimals via `10**(token.decimals() - 6)`, but `EndpointTx.submitSlowModeTransactionImpl` always increments `slowModeFees` by the raw constant `SLOW_MODE_FEE` (1e6) regardless of the token's actual decimals. For any quote token with more than 6 decimals, the amount actually transferred from users is `10**(decimals-6)` times larger than what is recorded in `slowModeFees`, permanently corrupting the fee accounting ledger.

---

### Finding Description

`SLOW_MODE_FEE = 1000000` is defined as "$1" in 6-decimal terms. [1](#0-0) 

`getSlowModeFee()` dynamically scales this to the quote token's native decimals:

```solidity
int256 multiplier = int256(10**(token.decimals() - 6));
return uint256(int256(SLOW_MODE_FEE) * multiplier);
``` [2](#0-1) 

`chargeSlowModeFee()` in `EndpointStorage` transfers exactly `getSlowModeFee()` tokens from the user:

```solidity
token.safeTransferFrom(from, address(this), clearinghouse.getSlowModeFee());
``` [3](#0-2) 

But immediately after, `submitSlowModeTransactionImpl` increments the ledger by the raw constant, not the scaled amount:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;   // always 1_000_000, regardless of token decimals
``` [4](#0-3) 

For a quote token with 18 decimals (e.g., a DAI-style stablecoin):

| | Amount |
|---|---|
| Tokens transferred from user | `1_000_000 × 10¹²` = `10¹⁸` (1 full token) |
| Recorded in `slowModeFees` | `1_000_000` (0.000001 of a full token) |
| Mismatch factor | `10¹²` |

`slowModeFees` is subsequently read in `ContractOwner.sol` and `OffchainExchange.sol` (confirmed by grep) to account for or distribute collected fees. Because `slowModeFees` is in 6-decimal units while the Endpoint contract holds 18-decimal-unit tokens, any downstream transfer or accounting keyed to `slowModeFees` will process `10¹²` times fewer tokens than were actually collected, leaving the remainder permanently stranded in the Endpoint contract.

---

### Impact Explanation

The `slowModeFees` state variable is the corrupted balance. Every slow-mode transaction submitted by an unprivileged user causes the Endpoint contract to collect `getSlowModeFee()` tokens but record only `SLOW_MODE_FEE` (1e6). Over time, the gap between actual holdings and the tracked ledger grows proportionally to `10**(decimals-6)` per transaction. Fee distribution or accounting operations that consume `slowModeFees` will systematically under-account for collected funds, leaving the excess irrecoverably locked in the Endpoint contract.

---

### Likelihood Explanation

The protocol does not enforce that the quote token has exactly 6 decimals. `getSlowModeFee()` itself proves this by dynamically reading `token.decimals()`. Any deployment where the quote token has more than 6 decimals (e.g., 18-decimal stablecoins such as DAI, FRAX, or LUSD) triggers the mismatch on every single user-submitted slow-mode transaction. The entry path requires no privilege: any user calling `submitSlowModeTransaction` with a non-`DepositCollateral` transaction type reaches the vulnerable lines. [5](#0-4) 

---

### Recommendation

Replace the raw constant increment with the scaled value so `slowModeFees` is always denominated in the token's native units:

```solidity
// Before
slowModeFees += SLOW_MODE_FEE;

// After
slowModeFees += int128(uint128(clearinghouse.getSlowModeFee()));
```

Alternatively, define `slowModeFees` as always tracking in 18-decimal internal units (consistent with `MAX_DECIMALS`) and apply the same `10**(MAX_DECIMALS - decimals)` multiplier used everywhere else in the clearing system. [6](#0-5) 

---

### Proof of Concept

1. Protocol is deployed with a quote token that has 18 decimals (e.g., DAI).
2. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload.
3. `submitSlowModeTransactionImpl` reaches the `else` branch and calls `chargeSlowModeFee(_getQuote(), sender)`.
4. `chargeSlowModeFee` calls `clearinghouse.getSlowModeFee()` → returns `1_000_000 × 10¹² = 10¹⁸`.
5. `safeTransferFrom` moves `10¹⁸` DAI tokens from the user into the Endpoint contract.
6. `slowModeFees += SLOW_MODE_FEE` records only `1_000_000`.
7. After N transactions: Endpoint holds `N × 10¹⁸` DAI; `slowModeFees` = `N × 1_000_000`.
8. Any fee-distribution call keyed to `slowModeFees` processes `N × 1_000_000` token units — `10¹²× ` less than the actual balance — leaving `N × (10¹⁸ − 1_000_000) ≈ N × 10¹⁸` DAI permanently stranded.

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
