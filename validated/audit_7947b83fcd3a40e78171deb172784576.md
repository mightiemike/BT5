### Title
Stale `digestToSubaccount` Entry After Close Allows Zero-Margin Position Reopening — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the active-subaccount registries (`parentSubaccounts`, `isolatedSubaccounts`, `isolatedSubaccountsMask`) but never clears `digestToSubaccount[digest]`. The early-return guard in `createIsolatedSubaccount` treats a non-zero `digestToSubaccount[digest]` as "already created" and returns the old isolated subaccount address without transferring any margin from the parent. `matchOrders` then routes the order to that zombie subaccount — which holds zero collateral — and `isHealthy` unconditionally returns `true`, so the fill succeeds with no margin backing.

---

### Finding Description

**Step 1 — Normal creation (margin transferred):**

`createIsolatedSubaccount` computes `digest = getDigest(productId, order)`, finds `digestToSubaccount[digest] == bytes32(0)`, allocates `iso`, and transfers `margin` from parent to `iso`: [1](#0-0) 

**Step 2 — Close clears active registries but not the digest mapping:**

`_tryCloseIsolatedSubaccount` (called when `balance.amount == 0`) transfers the remaining quote balance back to parent, then clears: [2](#0-1) 

`digestToSubaccount[digest]` and `digestToMargin[digest]` are **never touched** anywhere in `_tryCloseIsolatedSubaccount`. [3](#0-2) 

**Step 3 — Re-submission hits the early-return, no margin transferred:**

When the same signed order is submitted again via `CreateIsolatedSubaccount`, the function finds `digestToSubaccount[digest] != bytes32(0)` and returns immediately: [4](#0-3) 

The margin transfer block (lines 1074–1087) is never reached. The returned `iso` address now has zero quote balance and `parentSubaccounts[iso] == bytes32(0)` (cleared in step 2).

**Step 4 — `matchOrders` routes to the zombie iso, health check is a no-op:**

`matchOrders` unconditionally redirects the taker/maker sender to `digestToSubaccount[digest]`: [5](#0-4) 

After balances are updated, the only guard is: [6](#0-5) 

But `isHealthy` is defined as: [7](#0-6) 

A `grep` across all `.sol` files confirms `isHealthy` is only defined in `OffchainExchange.sol` with no production override. The fill therefore succeeds against an iso with zero collateral.

**Signature check passes:** `_checkSignature` accepts either `address(uint160(bytes20(subaccount)))` or `linkedSigner`: [8](#0-7) 

`matchOrders` passes `getLinkedSignerOrNlpSigner(original_parent_sender)` as `linkedSigner`, so the parent's key satisfies the check even though `order.sender` is now the iso address.

---

### Impact Explanation

The isolated subaccount opens a leveraged perp position with zero quote collateral. Any adverse price movement produces a negative balance on the iso with `parentSubaccounts[iso] == bytes32(0)`, meaning the loss cannot be attributed to or recovered from the parent. The protocol absorbs the shortfall as bad debt. Additionally, if `tryCloseIsolatedSubaccount` is later called on the zombie iso (e.g., via liquidation path at `ClearinghouseLiq.sol:622-624`), it will attempt to transfer the negative quote balance to `bytes32(0)` — the zero-address subaccount — further corrupting accounting state. [9](#0-8) 

---

### Likelihood Explanation

The precondition is straightforward: a trader opens an isolated position, closes it (bringing perp `balance.amount` to zero), then resubmits the identical signed order. The on-chain code imposes no nonce or expiry check that would block reuse of the same EIP-712 digest after a close. The sequencer processes `CreateIsolatedSubaccount` transactions as submitted; there is no on-chain guard preventing the second submission.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, clear the digest mappings. Because the function does not have direct access to the digest, the cleanest fix is to add a reverse mapping `subaccountToDigest` populated in `createIsolatedSubaccount`, then delete both entries on close:

```solidity
// on create:
subaccountToDigest[newIsolatedSubaccount] = digest;

// on close:
bytes32 d = subaccountToDigest[subaccount];
if (d != bytes32(0)) {
    delete digestToSubaccount[d];
    delete digestToMargin[d];
    delete subaccountToDigest[subaccount];
}
```

Alternatively, add a check in `createIsolatedSubaccount` that the returned iso is still active (i.e., `parentSubaccounts[digestToSubaccount[digest]] != bytes32(0)`) before taking the early-return path, and re-execute the full creation + margin transfer if it is not.

### Proof of Concept

```
State before:
  spotEngine.balance(QUOTE, parent) = 1000
  spotEngine.balance(QUOTE, iso)    = 0
  digestToSubaccount[D]             = bytes32(0)
  parentSubaccounts[iso]            = bytes32(0)

Step 1 — createIsolatedSubaccount(order, margin=100):
  digestToSubaccount[D] = iso
  spotEngine.balance(QUOTE, parent) = 900
  spotEngine.balance(QUOTE, iso)    = 100

Step 2 — matchOrders fills order partially; then trader closes position:
  perpEngine.balance(productId, iso).amount = 0
  tryCloseIsolatedSubaccount(iso):
    spotEngine.balance(QUOTE, iso)    = 0   (100 returned to parent)
    spotEngine.balance(QUOTE, parent) = 1000
    parentSubaccounts[iso]            = bytes32(0)
    isolatedSubaccounts[parent][id]   = bytes32(0)
    digestToSubaccount[D]             = iso  ← NOT cleared

Step 3 — createIsolatedSubaccount(same order):
  digestToSubaccount[D] != bytes32(0) → early return iso
  spotEngine.balance(QUOTE, parent) = 1000  ← unchanged
  spotEngine.balance(QUOTE, iso)    = 0     ← no margin transferred

Step 4 — matchOrders with same order:
  taker.order.sender redirected to iso (digestToSubaccount[D] = iso)
  iso opens position with 0 collateral
  isHealthy(iso) → true
  Fill succeeds → iso has open perp position, 0 quote balance
  Protocol bad debt created on adverse price move
```
