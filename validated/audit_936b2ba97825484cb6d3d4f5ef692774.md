### Title
`ENABLE_SECP_OPS` Flag Check Bypassed by 4-Byte Opcode Form of secp256k1/secp256r1 Verify — (File: `src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` gates the 1-byte secp opcodes (64 = `secp256k1_verify`, 65 = `secp256r1_verify`) behind the `ENABLE_SECP_OPS` flag. However, the 4-byte opcode forms of the same operators (`0x13d61f00` and `0x1c3a8f00`) are dispatched directly to the same underlying functions **without any flag check**. This is a flag/operator wiring error: the capability gate is applied to one encoding of the operator but silently bypassed by an alternative encoding of the identical operation.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method has two separate dispatch paths for secp operators.

**Path 1 — 1-byte opcodes (correctly gated):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
_ => {
    return unknown_operator(allocator, o, argument_list, flags, max_cost);
}
```

When `ENABLE_SECP_OPS` is absent, opcodes 64 and 65 fall through to `unknown_operator`, which either returns a cost-only no-op (lenient mode) or `EvalErr::Unimplemented` (strict/mempool mode).

**Path 2 — 4-byte opcodes (ungated):** [2](#0-1) 

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

There is **no check for `ENABLE_SECP_OPS`** before dispatching to `op_secp256k1_verify` or `op_secp256r1_verify`. The actual cryptographic verification executes unconditionally.

The 4-byte opcodes encode the same fixed costs as the 1-byte forms (`SECP256K1_VERIFY_COST = 1300000`, `SECP256R1_VERIFY_COST = 1850000`): [3](#0-2) 

The `ENABLE_SECP_OPS` flag is defined as the hard-fork gate for these operators: [4](#0-3) 

`MEMPOOL_MODE` does not include `ENABLE_SECP_OPS`: [5](#0-4) 

---

### Impact Explanation

**Consensus divergence / hard-fork bypass.** The `ENABLE_SECP_OPS` flag is the hard-fork activation gate for secp operations. Before the hard fork activates:

- Programs using 1-byte opcodes 64/65: secp verification is a no-op (lenient) or rejected (strict/mempool). Outcome is deterministic and flag-controlled.
- Programs using 4-byte opcodes `0x13d61f00`/`0x1c3a8f00`: actual `op_secp256k1_verify` / `op_secp256r1_verify` executes regardless of flag state.

An attacker submitting CLVM bytes with the 4-byte opcode form can:
1. Execute real secp256k1/secp256r1 signature verification on any node, regardless of whether the hard fork has activated.
2. Cause a program to **fail** (via `EvalErr::Secp256Failed`) on all nodes via the 4-byte form, while the same logical program using the 1-byte form would **succeed** (no-op) on pre-fork nodes — a direct consensus split.
3. In mempool mode (`MEMPOOL_MODE`), the 1-byte secp opcodes are rejected (`Unimplemented`), but the 4-byte form executes the full cryptographic check, bypassing the mempool's stricter validation.

The corrupted result is the execution outcome (success vs. `Secp256Failed` vs. `Unimplemented`) and the associated cost, which diverges between the two encodings of the same operator under the same flag configuration.

---

### Likelihood Explanation

**High.** The 4-byte opcode bytes are attacker-controlled CLVM input. Any caller of `run_program` with a CLVM program containing a 4-byte atom `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` as an operator triggers the bypass. No special privileges, configuration, or social engineering are required. The bypass is structural and unconditional.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path in `src/chia_dialect.rs`, mirroring the guard applied to the 1-byte opcodes:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures both encodings of the secp operators are uniformly gated by the hard-fork activation flag.

### Proof of Concept

Construct a CLVM program whose operator atom is the 4-byte sequence `[0x13, 0xd6, 0x1f, 0x00]` (secp256k1_verify) with three atom arguments (pubkey, msg, sig). Run it with a `ChiaDialect` that does **not** have `ENABLE_SECP_OPS` set. The call reaches `op_secp256k1_verify` directly and either returns `Ok(Reduction(1300000, nil))` on valid inputs or `Err(EvalErr::Secp256Failed(...))` on invalid inputs — neither of which is the `unknown_operator` no-op or `Unimplemented` error that the 1-byte opcode 64 would produce under the same flag configuration. ```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
