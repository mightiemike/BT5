### Title
`ValidResourceBounds` Variant Collapse in Protobuf Round-Trip Produces Wrong Transaction Hash — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant. When a transaction carries `AllResources(l1_gas, l2_gas=0, l1_data_gas=0)`, the round-trip through protobuf silently collapses it to `L1Gas(l1_gas)`. Because `get_tip_resource_bounds_hash` hashes a different number of field elements for each variant, the transaction hash computed after deserialization diverges from the hash computed at ingestion, breaking the hash-binding invariant for any transaction that legitimately uses `AllResources` with zero L2 and L1-data-gas bounds.

---

### Finding Description

**Step 1 — User submits a valid `AllResources` transaction with zero L2/L1-data-gas.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. A user may submit:

```
AllResourceBounds { l1_gas: <non-zero>, l2_gas: 0, l1_data_gas: 0 }
```

The gateway's stateless validator explicitly accepts this (test cases `valid_l1_gas`, `valid_l2_gas`, etc. confirm any single non-zero resource is valid). [1](#0-0) 

**Step 2 — Gateway computes hash H₁ using the `AllResources` path.**

`convert_rpc_tx_to_internal` converts `RpcInvokeTransactionV3 → InternalRpcInvokeTransactionV3` (preserving `AllResourceBounds`), then calls `tx_without_hash.calculate_transaction_hash(&self.chain_id)`. [2](#0-1) 

`InternalRpcInvokeTransactionV3::resource_bounds()` returns `ValidResourceBounds::AllResources(self.resource_bounds)`, so `get_tip_resource_bounds_hash` is called with the `AllResources` variant. [3](#0-2) 

Inside `get_tip_resource_bounds_hash`, the `AllResources` branch appends a third felt — `get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)` — even when `l1_data_gas` is zero:

```
H₁ = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_GAS_packed(0)])
``` [4](#0-3) 

**Step 3 — Transaction is serialized to protobuf for P2P propagation.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits all three fields, including `l1_data_gas: Some(zero)`: [5](#0-4) 

**Step 4 — Receiving node deserializes and collapses the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // → zero
// l1_data_gas.is_zero() && l2_gas.is_zero() → true
ValidResourceBounds::L1Gas(l1_gas)                          // ← wrong variant
``` [6](#0-5) 

**Step 5 — Receiving node computes hash H₂ using the `L1Gas` path.**

With `L1Gas`, `get_tip_resource_bounds_hash` omits the L1_DATA_GAS felt entirely:

```
H₂ = poseidon([tip, L1_GAS_packed, L2_GAS_packed(0)])
``` [7](#0-6) 

**H₁ ≠ H₂** — the poseidon hash over three elements differs from the hash over two elements, even when the third element is zero, because poseidon is length-sensitive.

---

### Impact Explanation

The transaction hash is the canonical identifier used in:
- The block's transaction commitment (receipt commitment, event commitment).
- Signature verification — the account contract signs over the hash; a wrong hash means the signature check runs against the wrong preimage.
- Fee/resource-bound enforcement — the blockifier reads `resource_bounds` from the deserialized transaction; if the variant is `L1Gas` instead of `AllResources`, the fee-check path changes (`L1Gas` only checks L1 gas; `AllResources` checks all three resources).

A block containing such a transaction will either be rejected by syncing nodes (hash mismatch → chain-split / DoS) or, if the hash is trusted from the peer, the transaction will be executed with the wrong resource-bound variant, bypassing L2-gas and L1-data-gas upper-bound checks.

This matches: **High — Transaction conversion or signature/hash logic binds the wrong hash or executable payload.**

---

### Likelihood Explanation

Any user can trigger this by submitting an invoke V3 transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` — a configuration the gateway explicitly accepts. No privileged access is required. The condition is hit whenever such a transaction is included in a block and that block is propagated over the P2P layer.

---

### Recommendation

Replace the value-based heuristic with an explicit variant tag in the protobuf wire format, or add a dedicated boolean/enum field that records whether the transaction was signed under the `L1Gas` (pre-0.13.3) or `AllResources` (post-0.13.3) scheme. Until then, the deserializer should treat `l1_data_gas = None` as an error for any transaction whose `l2_gas` is non-zero (i.e., enforce the TODO comment: "Assert data gas is not none once we remove support for 0.13.2").

Concretely, the fix in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` should be:

```rust
// Instead of unwrap_or_default(), require l1_data_gas when l2_gas is present:
let l1_data_gas = value.l1_data_gas
    .ok_or(DEPRECATED_RESOURCE_BOUNDS_ERROR)?
    .try_into()?;
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
```

and handle the legacy `l2_gas = 0, l1_data_gas = None` case separately.

---

### Proof of Concept

1. Submit an invoke V3 transaction to the gateway with `resource_bounds = AllResourceBounds { l1_gas: 1000, l2_gas: 0, l1_data_gas: 0 }`.
2. Gateway computes H₁ = `get_invoke_transaction_v3_hash(...)` with `AllResources` → hash includes `L1_DATA_GAS_packed(0)` as the fourth poseidon input. [8](#0-7) 
3. Transaction is serialized to `protobuf::ResourceBounds` with `l1_data_gas = Some(ResourceLimits { max_amount: 0, max_price_per_unit: Some(0) })`. [5](#0-4) 
4. Receiving node deserializes: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`. [9](#0-8) 
5. Receiving node computes H₂ = `get_invoke_transaction_v3_hash(...)` with `L1Gas` → hash omits `L1_DATA_GAS_packed(0)`. [7](#0-6) 
6. H₁ ≠ H₂ — the block's transaction commitment built from H₁ will not match the hash H₂ recomputed by the syncing node, causing block rejection or wrong resource-bound enforcement.

    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
