### Title
`ContractOwner` Cannot Receive Native ETH, Permanently Locking Funds in `DirectDepositV1` Contracts — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` defines no `receive()` or `fallback()` function. The `withdrawFromDirectDepositV1` path for native ETH recovery calls `DirectDepositV1.withdrawNative()`, which pushes ETH back to `msg.sender` (i.e., `ContractOwner`) via a low-level call. Because `ContractOwner` cannot accept ETH, the push fails, the `require` reverts, and any native ETH stranded in a `DirectDepositV1` (DDA) contract is permanently unrecoverable.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` handles the `token == address(0)` (native ETH) branch as follows: [1](#0-0) 

It calls `DirectDepositV1.withdrawNative()`, which unconditionally pushes the DDA's entire ETH balance to `msg.sender` (the `ContractOwner` contract): [2](#0-1) 

`ContractOwner` is an upgradeable contract that inherits `EIP712Upgradeable` and `OwnableUpgradeable`. It defines no `receive()` or `fallback()` function anywhere in its body: [3](#0-2) 

Because no ETH-accepting hook exists on `ContractOwner`, the low-level `call{value: balance}("")` in `withdrawNative` returns `success = false`. The `require(success, "Failed to transfer native token to owner")` then reverts the entire call, leaving the ETH permanently locked inside the DDA.

The `DirectDepositV1` constructor already anticipates wrapping failures and emits `NativeTokenTransferFailed` rather than reverting, explicitly to avoid funds being stuck forever: [4](#0-3) 

However, the intended recovery path (`withdrawFromDirectDepositV1`) is itself broken for the same reason, negating the constructor's defensive design.

---

### Impact Explanation

Any native ETH that ends up in a DDA — whether from a failed constructor wrap, a future edge case, or a direct send that triggers the `receive()` but whose wrapping subsequently fails — is permanently unrecoverable. The `withdrawFromDirectDepositV1(subaccount, address(0))` code path is completely non-functional. The ETH balance of the DDA is frozen with no alternative recovery mechanism.

---

### Likelihood Explanation

The `DirectDepositV1` constructor explicitly handles the case where `wrappedNative.call{value: balance}("")` fails silently and emits `NativeTokenTransferFailed`. This is a documented, anticipated failure mode. When it occurs, the owner's only recovery tool (`withdrawFromDirectDepositV1`) also fails due to the missing `receive()` on `ContractOwner`, making the fund lock permanent. Likelihood is **low-medium**: the wrapping failure path is rare but explicitly anticipated by the codebase, and the recovery function is completely broken when it is needed.

---

### Recommendation

Add a `receive() external payable {}` function to `ContractOwner.sol`, exactly as the GMX team did for `Bank.sol`. This allows `DirectDepositV1.withdrawNative()` to successfully push ETH to `ContractOwner`, after which `ContractOwner` forwards it to `msg.sender` (the multisig owner) as intended on line 633.

---

### Proof of Concept

1. `ContractOwner` is deployed — no `receive()` function exists.
2. A DDA is created via `createDirectDepositV1(subaccount)`.
3. ETH is sent to the DDA. The `receive()` in `DirectDepositV1` calls `wrappedNative.call{value: msg.value}("")`. Suppose this call fails (e.g., the wrapped native contract is paused or non-standard). The `require(success, "Failed to wrap native token.")` reverts the receive, so ETH is not accepted via `receive()`. Alternatively, ETH arrives during construction and wrapping fails silently — `NativeTokenTransferFailed` is emitted and ETH remains in the DDA.
4. Owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
5. Internally, `DirectDepositV1.withdrawNative()` executes:
   ```solidity
   (bool success, ) = msg.sender.call{value: balance}(""); // msg.sender = ContractOwner
   require(success, "Failed to transfer native token to owner"); // REVERTS
   ```
6. `ContractOwner` has no `receive()` → the call returns `success = false` → `require` reverts.
7. The entire `withdrawFromDirectDepositV1` transaction reverts. ETH remains permanently locked in the DDA with no alternative recovery path. [5](#0-4) [2](#0-1)

    function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
        external
        onlyOwner
    {
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        if (token == address(0)) {
            uint256 preBalance = address(this).balance;
            DirectDepositV1(directDepositV1).withdrawNative();
            uint256 postBalance = address(this).balance;
            require(postBalance > preBalance, "empty");
            (bool success, ) = msg.sender.call{value: postBalance - preBalance}(
                ""
            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));
            require(postBalance > preBalance, "empty");
            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
            );
        }
    }
