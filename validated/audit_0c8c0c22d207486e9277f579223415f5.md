Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the end user. A pool admin who allowlists the router to support router-mediated swaps for allowlisted users simultaneously opens the pool to every user on the network, breaking the core invariant of the extension.

## Finding Description
`SwapAllowlistExtension.beforeSwap` enforces the allowlist as follows:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension is called by the pool), and `sender` is the value forwarded from `MetricOmmPool.swap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-L231
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    ...
``` [1](#0-0) 

This `msg.sender` is then passed as `sender` through `ExtensionCalling._beforeSwap` into the extension: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly with no originator forwarding:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-L80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
    );
``` [3](#0-2) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`. The pool admin faces an impossible choice:

- **Do not allowlist the router** → allowlisted users cannot use the router; they must call `pool.swap()` directly.
- **Allowlist the router** → every user on the network bypasses the allowlist by routing through the router.

The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap()` with `msg.sender = router`. [4](#0-3) 

## Impact Explanation
Any non-allowlisted user can execute swaps on a curated pool that is supposed to restrict trading to a specific set of addresses (e.g., KYC'd wallets, protocol-internal actors). The allowlist protection silently fails open for all router-mediated swaps. This is a direct policy bypass that breaks the core invariant of `SwapAllowlistExtension`: "only allowlisted addresses may swap on this pool." The impact is broken core pool functionality (access control) causing unrestricted swap access to a pool designed to be curated. [5](#0-4) 

## Likelihood Explanation
- `MetricOmmSimpleRouter` is the standard, publicly deployed periphery swap entry point, callable by anyone permissionlessly.
- Any user can call `exactInputSingle` or any other router function without any privileged access.
- A pool admin who deploys a curated pool and wants to support router-mediated swaps for allowlisted users has no choice but to allowlist the router, which opens the gate to everyone.
- No special setup, non-standard tokens, or privileged access is required. [6](#0-5) 

## Recommendation
Pass the original end user through the extension chain rather than the immediate `msg.sender` of `pool.swap()`. One approach: the router stores the originating user in transient storage (it already uses transient storage for callback context via `TransientCallbackPool`) and the pool reads it as an additional `originator` field forwarded to extensions. Alternatively, the extension can be redesigned to check `sender` only when `sender` is not a known/registered router, and check a router-forwarded user field otherwise. At minimum, document that allowlisting the router defeats the allowlist and provide a router variant that forwards the originating user. [7](#0-6) 

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack (Bob, not allowlisted):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(...) — msg.sender of pool.swap() is the router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension evaluates: allowedSwapper[pool][router] == true → passes
  5. Bob's swap executes on the curated pool despite Bob never being allowlisted

Result:
  - allowedSwapper[pool][bob] == false, yet Bob swapped successfully
  - The allowlist invariant is broken for all router-mediated swaps
```

A Foundry test can reproduce this by: deploying a pool with `SwapAllowlistExtension`, allowlisting only `alice` and the `router`, then calling `router.exactInputSingle` from `bob`'s address and asserting the swap succeeds (no `NotAllowedToSwap` revert). [8](#0-7)
