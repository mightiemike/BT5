### Title
`limbs_for_int` Underestimates Byte-Size of Negative Intermediate Products, Causing Undercharged Execution Cost in `op_multiply` - (File: src/more_ops.rs)

---

### Summary

`limbs_for_int` computes the byte-size of a `Number` using `v.bits().div_ceil(8)`, which counts bits in the **absolute value** only. For negative numbers whose absolute value has its highest bit set in the highest byte (e.g., `-129`, `-255`, `-32769`), the CLVM two's complement encoding requires one extra byte that `limbs_for_int` does not account for. In `op_multiply`, the running size tracker `l0` is updated via `limbs_for_int` after each multiplication step and is then fed directly into the cost formula and the `l0 > 1024` size guard. The underestimate causes the cost of subsequent multiplications to be undercharged and allows intermediate products up to 1025 bytes to bypass the 1024-byte limit.

---

### Finding Description

`limbs_for_int` is defined as:

```rust
fn limbs_for_int(v: &Number) -> usize {
    v.bits().div_ceil(8) as usize
}
```

`BigInt::bits()` returns the number of bits in the **absolute value**. For a negative number `-n`, the CLVM two's complement encoding needs an extra byte whenever the highest byte of `n` has its MSB set (i.e., `n` is in the range `(2^(8k-1), 2^(8k))` for some `k ≥ 1`):

| Value | Absolute value bits | `limbs_for_int` | Actual CLVM bytes |
|---|---|---|---|
| `-128` | 8 | 1 | 1 (`0x80`) ✓ |
| `-129` | 8 | 1 | 2 (`0xff 0x7f`) ✗ |
| `-255` | 8 | 1 | 2 (`0xff 0x01`) ✗ |
| `-32768` | 16 | 2 | 2 (`0x80 0x00`) ✓ |
| `-32769` | 16 | 2 | 3 (`0xff 0x7f 0xff`) ✗ |

In `op_multiply`, after each multiplication step, `l0` is updated with the underestimated value and used in the cost formula for the **next** step:

```rust
l0 = limbs_for_int(&total);   // may underestimate by 1 for negative products
if l0 > 1024 {
    return Err(EvalErr::InvalidOpArg(arg, "*".to_string()));
}
```

Then on the next iteration:

```rust
cost += (l0 as Cost + l1) * MUL_LINEAR_COST_PER_BYTE;
cost += (l0 as Cost * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
```

When `l0` is 1 less than the true byte-size, the cost is undercharged by `MUL_LINEAR_COST_PER_BYTE + l1 / MUL_SQUARE_COST_PER_BYTE_DIVIDER = 6 + l1/128` per affected step. Additionally, the `l0 > 1024` guard can be bypassed by 1 byte, allowing a 1025-byte intermediate product to proceed.

The existing test `test_limbs_for_int` only exercises positive numbers and numbers with a leading `0x00` byte. It never tests negative numbers like `[0xff, 0x7f]` (`-129`) or `[0xff, 0x01]` (`-255`), so the underestimate is undetected.

---

### Impact Explanation

An attacker who submits crafted CLVM bytecode can cause `op_multiply` to undercharge execution cost when intermediate products are negative numbers with absolute values in ranges like `[129, 255]`, `[32769, 65535]`, etc. This is an arithmetic semantic mismatch: the cost model assumes `limbs_for_int` correctly reflects the CLVM byte-size of the intermediate result, but it does not for these negative values. The undercharge per step is small (`≤ 14` cost units for `l1 ≤ 256`), but it accumulates across multiple multiplication steps and also allows a 1-byte bypass of the 1024-byte intermediate-size guard.

**Impact: Low** — the per-step undercharge is bounded and the size-guard bypass is at most 1 byte.

---

### Likelihood Explanation

Any attacker-controlled CLVM program using the `*` operator with operands crafted to produce negative intermediate products (e.g., multiplying a large positive number by `-1` or any negative factor) can trigger this path. No special privileges or configuration are required; the entry point is the standard `run_program` API exposed to all callers.

**Likelihood: High** — negative intermediate products are trivially constructible from attacker-controlled CLVM bytes.

---

### Recommendation

Replace `limbs_for_int` with a function that correctly computes the CLVM two's complement byte-length for both positive and negative numbers. For a negative number `-n`, the correct byte count is `ceil(bits(n) / 8)` only when `n` is a power of 2; otherwise it is `ceil(bits(n) / 8) + 1` when the highest byte of `n` has its MSB set. A correct implementation:

```rust
fn limbs_for_int(v: &Number) -> usize {
    if v.sign() == Sign::NoSign {
        return 0;
    }
    // For negative numbers, two's complement may need an extra byte
    // when the absolute value's highest bit is set in the top byte.
    let bits = v.bits() as usize;
    let byte_len = bits.div_ceil(8);
    if v.sign() == Sign::Minus && bits % 8 != 0 {
        // highest bit of absolute value is NOT in the sign position; no extra byte needed
        byte_len
    } else if v.sign() == Sign::Minus {
        // highest bit IS in the sign position; need extra byte for sign
        byte_len + 1
    } else {
        byte_len
    }
}
```

Also add test cases for negative numbers like `[0xff, 0x7f]` (`-129`) and `[0xff, 0x01]` (`-255`) to `test_limbs_for_int`.

### Proof of Concept

The root cause is at: [1](#0-0) 

`limbs_for_int` uses `v.bits().div_ceil(8)`, which counts bits in the absolute value only, missing the extra sign byte required for negative numbers whose absolute value has its MSB set in the highest byte. [2](#0-1) 

`l0` is updated with the underestimated value and used as the size of the running product in the cost formula and size guard. [3](#0-2) 

The cost formula uses `l0` directly; a 1-byte underestimate causes `6 + l1/128` cost units to be silently dropped per affected step.

The test suite at lines 118–153 never exercises negative numbers with absolute values in `[129, 255]` or `[32769, 65535]`, leaving the underestimate undetected: [4](#0-3) 

A concrete trigger: submit a CLVM program that multiplies `0x81` (= 129) by `0xff` (= -1), producing `-129`. `limbs_for_int(-129)` returns `1` instead of `2`. The next multiplication step uses `l0 = 1` instead of `l0 = 2`, undercharging cost by `6 + l1/128` units.
