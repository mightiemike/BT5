Audit Report

## Title
`SwapAllowlistExtension` checks `sender` (direct caller / router) instead of the actual economic actor, allowing any user to bypass the per-address swap gate via a whitelisted router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is the direct `msg.sender` of `pool.swap()`. When a router intermediary calls the pool, `sender` is the router address, not the actual trader. If the router is allowlisted, every user who routes through it bypasses the per-address swap restriction the pool admin intended to enforce. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly gates on `owner` (the economic actor / LP position owner), revealing a deliberate design inconsistency that leaves the swap allowlist ineffective in the router path.

## Finding Description

**Root cause — wrong identity checked in `SwapAllowlistExtension`:**

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (enforced by `BaseMetricExtension.onlyPool`). `sender` is the first argument forwarded by the pool — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` (or any other router) calls `pool.swap(recipient=user, ...)`, `sender` = router address, not the user.

**Contrast with `DepositAllowlistExtension`:**

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Here the first argument (`sender`) is explicitly ignored and `owner` (the LP position owner / economic actor) is checked. This is the correct pattern.

**Extension call path (pool → extension):**

`ExtensionCalling._beforeSwap` encodes and forwards `(sender, recipient, ...)` to the extension:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64,
         packedSlot0Initial, bidPriceX64, askPriceX64, extensionData)
    )
);
```

The actual trader's address is available as `recipient` (second argument) but `SwapAllowlistExtension` discards it (unnamed `address,` parameter) and checks `sender` instead.

**Exploit flow:**

1. Pool admin deploys pool with `SwapAllowlistExtension`, sets `allowAllSwappers[pool] = false`, and allowlists only specific trader addresses.
2. Pool admin also allowlists the `MetricOmmSimpleRouter` address as a swapper (a natural operational choice — the router is a trusted periphery contract).
3. Unprivileged trader (not individually allowlisted) calls `router.swap(pool, ...)`.
4. Router calls `pool.swap(recipient=trader, ...)` → pool calls `extension.beforeSwap(sender=router, recipient=trader, ...)`.
5. Extension evaluates `allowedSwapper[pool][router] == true` → passes. The non-allowlisted trader's swap executes.

## Impact Explanation
The pool admin's intent — restricting swaps to a curated set of addresses — is completely defeated for any user who routes through a whitelisted router. This is an admin-boundary break: an unprivileged trader bypasses an access-control gate the pool admin explicitly configured. Any pool relying on `SwapAllowlistExtension` for KYC/compliance or liquidity-protection gating is rendered ineffective. Severity: **Medium** (broken core access-control invariant reachable by any unprivileged user; no direct principal loss but the gate the admin paid to configure provides zero protection in the router path).

## Likelihood Explanation
The router is the standard entry point for swaps in the Metric OMM periphery. A pool admin who enables the swap allowlist and also permits the router (a common operational requirement) will unknowingly open the gate to all users. The condition is easy to reach, requires no special privileges, and is repeatable by any trader.

## Recommendation
Mirror the `DepositAllowlistExtension` pattern: check the economic actor, not the intermediary. For swaps the economic actor is the `recipient` (the address that receives output tokens and whose funds are at stake):

```solidity
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

Alternatively, if the intent is to gate on the initiating address, document clearly that the router must never be allowlisted and that each end-user must call the pool directly — but this breaks the standard periphery flow.
