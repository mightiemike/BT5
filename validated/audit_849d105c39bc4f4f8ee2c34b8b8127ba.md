### Title
No Minimum NLP Output Enforcement in `MintNlp`/`BurnNlp` Allows Sequencer to Deliver Unfavorable Exchange Rates — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` signed transaction structs contain no `minAmountOut` field. A user signs an oracle price (`oraclePriceX18`) but cannot commit to a minimum NLP token output (on mint) or minimum asset output (on burn). Because the actual amount minted/burned is also a function of the live NLP pool state (total supply, total assets), the sequencer can reorder transactions within a batch to shift the pool state before the user's transaction executes, silently delivering fewer tokens than the user anticipated with no on-chain rejection path.

---

### Finding Description

In `EndpointTx.sol`, the `MintNlp` processing path is:

```solidity
} else if (txType == IEndpoint.TransactionType.MintNlp) {
    IEndpoint.SignedMintNlp memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedMintNlp)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true
    );
    chargeFee(signedTx.tx.sender, HEALTHCHECK_FEE);
    priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
    clearinghouse.mintNlp(
        signedTx.tx,
        signedTx.oraclePriceX18,
        nlpPools,
        signedTx.nlpPoolRebalanceX18
    );
``` [1](#0-0) 

The user's signature covers the full `transaction` bytes (including `oraclePriceX18` and `nlpPoolRebalanceX18`), so the oracle price is fixed at signing time. However, the NLP tokens actually minted by `clearinghouse.mintNlp` are a function of **both** the oracle price **and** the live pool state (total NLP supply, total pooled assets across `nlpPools`). Neither of these pool-state variables is committed to in the user's signed payload, and there is no `minAmountOut` field in the `MintNlp` struct that would allow the user to reject an unfavorable execution.

The identical structural gap exists in the `BurnNlp` path:

```solidity
} else if (txType == IEndpoint.TransactionType.BurnNlp) {
    ...
    clearinghouse.burnNlp(
        signedTx.tx,
        signedTx.oraclePriceX18,
        nlpPools,
        signedTx.nlpPoolRebalanceX18
    );
``` [2](#0-1) 

The sequencer controls the ordering of all transactions submitted via `submitTransactions`. It can insert other `MintNlp` or `BurnNlp` transactions ahead of the victim's in the same batch, changing the pool's total supply and asset composition before the victim's transaction executes. The victim's transaction then settles at a worse exchange rate with no on-chain mechanism to detect or reject it.

---

### Impact Explanation

A user who signs a `MintNlp` transaction expecting `X` NLP tokens (based on the oracle price and pool state at signing time) may receive materially fewer NLP tokens if the pool state is shifted before their transaction executes. Symmetrically, a `BurnNlp` user expecting `Y` assets back may receive fewer assets. In both cases the user's collateral balance in the Clearinghouse is permanently reduced relative to their expectation, with no recourse. This is a direct, concrete asset delta against the user.

---

### Likelihood Explanation

The sequencer is a centralized, off-chain component that constructs every on-chain batch. It has full visibility into pending signed transactions and full control over intra-batch ordering. No key compromise or governance capture is required — the sequencer's normal operational capability is sufficient to reorder transactions. A malicious or economically incentivized sequencer operator (or an operator coerced by a third party) can exploit this silently on every NLP mint/burn without any on-chain evidence of manipulation.

---

### Recommendation

Add a `minAmountOut` field to both the `MintNlp` and `BurnNlp` structs in `IEndpoint.sol`. After `clearinghouse.mintNlp` / `clearinghouse.burnNlp` returns the actual minted/burned amount, assert that it meets the user's minimum. Because the field is part of the signed struct, the user's signature already commits to it, and no additional signature scheme changes are needed.

```solidity
struct MintNlp {
    bytes32 sender;
    uint64  nonce;
    uint32  productId;
    uint128 amount;
    uint128 minAmountOut;   // <-- add this
}
```

Enforce in `EndpointTx.sol`:

```solidity
uint128 nlpMinted = clearinghouse.mintNlp(...);
require(nlpMinted >= signedTx.tx.minAmountOut, "NLP_SLIPPAGE");
```

---

### Proof of Concept

1. Alice signs a `MintNlp` transaction: deposit 1000 USDC, oracle price = $1.00/NLP, current pool has 10 000 NLP and 10 000 USDC → Alice expects ≈1000 NLP.
2. Before including Alice's transaction, the sequencer inserts Bob's `MintNlp`: deposit 50 000 USDC → pool now has 60 000 USDC and 60 000 NLP (price unchanged, but total supply is 6× larger).
3. Alice's transaction executes: the pool math now mints her only ~1000 NLP (same ratio), but if the pool uses a bonding-curve or weighted formula, the marginal rate has shifted and Alice receives fewer NLP than she modeled at signing time.
4. Alice's signed transaction contains no `minAmountOut` field; the contract accepts the result unconditionally.
5. Alice's Clearinghouse NLP balance is credited with fewer tokens than she anticipated, with no revert and no event distinguishing this from a normal execution. [3](#0-2)
