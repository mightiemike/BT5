### Title
Fast Withdrawal Fee Charged to Relayer (`msg.sender`) Instead of Recipient (`sendTo`) — (`File: core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` charges the fast-withdrawal fee against `msg.sender`'s token balance whenever `sendTo != msg.sender`. Any third-party relayer or aggregator contract that submits a fast withdrawal on behalf of a user will have its own token balance drained, or the call will revert if it holds no balance, breaking composability.

---

### Finding Description

`submitFastWithdrawal` resolves the destination address (`sendTo`) from the signed withdrawal transaction and then applies the fee with a branch:

```solidity
if (sendTo == msg.sender) {
    require(transferAmount > uint128(fee), "Fee larger than balance");
    transferAmount -= uint128(fee);          // fee deducted from recipient's amount ✓
} else {
    safeTransferFrom(token, msg.sender, uint128(fee));  // fee pulled from caller ✗
}
``` [1](#0-0) 

When a relayer (any address that is not `sendTo`) calls `submitFastWithdrawal`, the `else` branch fires and `safeTransferFrom` pulls the fee from the relayer's own token balance. [2](#0-1) 

The `sendTo` value comes from the signed transaction payload — either the `sender` field cast to an address (V1), or an explicit `sendTo` field (V2). Neither field is the caller. [3](#0-2) 

The `WithdrawCollateralV2` struct explicitly supports a `sendTo` address distinct from the signer, making the third-party-relayer path a first-class, intended use case. [4](#0-3) 

---

### Impact Explanation

- A relayer contract that calls `submitFastWithdrawal` on behalf of users will have its token balance silently drained by the fee on every call.
- If the relayer holds no token balance, `safeTransferFrom` reverts, making fast withdrawals completely inaccessible through any third-party relayer.
- The full `transferAmount` is still sent to `sendTo` (line 113), so the recipient receives the undeducted amount while the relayer absorbs the fee — a direct, unintended asset transfer from the relayer to the protocol fee pool. [5](#0-4) 

---

### Likelihood Explanation

`submitFastWithdrawal` is a public, permissionless function. Any contract (aggregator, wallet abstraction layer, keeper bot, or cross-chain bridge adapter) that relays fast withdrawals is a realistic caller where `sendTo != msg.sender`. The `WithdrawCollateralV2` path with an explicit `sendTo` field makes this scenario an explicitly supported design intent, raising likelihood further. [6](#0-5) 

---

### Recommendation

Remove the `else` branch entirely. Always deduct the fee from `transferAmount` (the recipient's withdrawal amount), mirroring the already-correct `sendTo == msg.sender` branch:

```solidity
require(transferAmount > uint128(fee), "Fee larger than balance");
transferAmount -= uint128(fee);
fees[productId] += fee;
handleWithdrawTransfer(token, sendTo, transferAmount);
```

This ensures the fee is always borne by the withdrawal recipient, regardless of who submits the transaction.

---

### Proof of Concept

1. User signs a `WithdrawCollateralV2` transaction with `sendTo = userAddress`, `amount = 1000 USDC`.
2. A relayer contract calls `submitFastWithdrawal(idx, transaction, signatures)` — `msg.sender = relayerContract`, `sendTo = userAddress`.
3. `sendTo != msg.sender` → `else` branch executes: `safeTransferFrom(token, relayerContract, fee)`.
4. The relayer's USDC balance is reduced by `fee` (e.g., 5 USDC).
5. `handleWithdrawTransfer(token, userAddress, 1000 USDC)` sends the full undeducted amount to the user.
6. Net result: relayer loses `fee` tokens; user receives full amount; protocol collects fee from the wrong party. [7](#0-6)
