### Title
`BaseEngine.getCoreRisk` Hardcodes `amount=1`, Causing `Clearinghouse.getHealth` to Overestimate Spread Health for Long Spot / Short Perp Positions — (`core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk` always passes the literal `1` (positive) as the `amount` argument to `RiskHelper._getWeightX18`, so `CoreRisk.longWeight` always holds `longWeightInitialX18` regardless of the actual position direction. `Clearinghouse.getHealth` then uses this field for both `existingWeight` and `spreadWeight` in the spread adjustment. For a long-spot / short-perp spread the short perp's `shortWeightInitialX18` (> 1) is silently replaced by `longWeightInitialX18` (< 1), making `spreadWeight − existingWeight` larger than it should be and inflating the health contribution for the hedged portion.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` line 190:**

```solidity
RiskHelper._getWeightX18(risk, 1, healthType)   // hardcoded positive 1
```

`_getWeightX18` branches on the sign of `amount`:

```solidity
if (amount >= 0) {          // always true because amount == 1
    weight = healthType == INITIAL
        ? risk.longWeightInitialX18      // always returned
        : risk.longWeightMaintenanceX18;
} else {
    weight = ...shortWeightInitialX18;   // never reached via getCoreRisk
}
```

Compare with `_calculateProductHealth` line 167, which correctly passes the real balance:

```solidity
int128 weight = RiskHelper._getWeightX18(risk, amount, healthType);
```

**Downstream effect in `Clearinghouse.getHealth` lines 125–135:**

```solidity
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight   = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price)
                     .mul(spreadWeight - existingWeight);
```

`_getSpreadWeightX18` for `spotCoreRisk.amount > 0`:

```solidity
spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
```

Both `existingWeight` and `spreadWeight` consume `perpCoreRisk.longWeight`, which is always `perpLongWeightInitialX18`.

### Impact Explanation

Concrete numerical example with typical perp risk parameters:

| Parameter | Value |
|---|---|
| `spotLongWeightInitialX18` | 0.90 |
| `perpLongWeightInitialX18` | 0.90 |
| `perpShortWeightInitialX18` | 1.10 |
| `maxSpreadWeight` (INITIAL) | 0.99 |

**Buggy path (perpLongWeight = 0.90 used for short perp):**
- `existingWeight = (0.90 + 0.90) / 2 = 0.90`
- `spreadWeight = 1 − (1 − 0.90)/5 = 0.98` → capped at 0.99 → **0.98**
- `spreadWeight − existingWeight = +0.08`
- Spread adjustment = `+0.08 × basisAmount × (spotPrice + perpPrice)` **(positive, health inflated)**

**Correct path (perpShortWeight = 1.10 used for short perp):**
- `existingWeight = (0.90 + 1.10) / 2 = 1.00`
- `spreadWeight = 1 − (1 − 1.10)/5 = 1.02` → capped at 0.99 → **0.99**
- `spreadWeight − existingWeight = −0.01`
- Spread adjustment = `−0.01 × basisAmount × (spotPrice + perpPrice)` **(negative, correct penalty)**

The bug flips the sign of the spread adjustment and inflates health by `0.09 × basisAmount × (spotPrice + perpPrice)` per spread pair. A user holding a large long-spot / short-perp spread can pass `getHealth(INITIAL)` with materially less collateral than the protocol requires, leaving it undercollateralized.

---

### Likelihood Explanation

The path is fully reachable through normal trading: any user can open a long spot position and a short perp position on a registered spread pair. No special privileges, governance capture, or sequencer compromise is needed. The spread discount is applied automatically in every `getHealth` call whenever the spread pair is registered and the subaccount holds opposite-direction positions.

---

### Recommendation

In `BaseEngine.getCoreRisk`, pass the actual balance `amount` (not the literal `1`) to `_getWeightX18`:

```solidity
// Before (buggy):
RiskHelper._getWeightX18(risk, 1, healthType)

// After (correct):
RiskHelper._getWeightX18(risk, amount, healthType)
```

This aligns `getCoreRisk` with `_calculateProductHealth`, which already uses the real `amount`. The `CoreRisk.longWeight` field name should also be renamed to `weight` to reflect that it now carries the direction-appropriate weight.
