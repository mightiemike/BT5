### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (File: `contract/src/lib.rs`)

---

### Summary

The `verify_transaction_inclusion` function is explicitly marked `#[deprecated]` due to a known 64-byte transaction Merkle proof forgery vulnerability, yet it remains a fully public, unpermissioned NEAR contract method. Any caller — including an adversarial one — can invoke it directly, bypassing the coinbase-proof guard introduced in `verify_transaction_inclusion_v2`. Consumer contracts that rely on this endpoint can be deceived into accepting a forged transaction inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` is annotated with both `#[deprecated(since = "0.5.0", ...)]` and `pub`, making it a live, callable NEAR contract method with no access restriction. [1](#0-0) 

The function's own doc-comment acknowledges the vulnerability explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The fix — `verify_transaction_inclusion_v2` — adds a coinbase Merkle proof check that pins the proof depth, preventing an attacker from supplying an internal node as `tx_id`. [3](#0-2) 

However, the old function is never removed or access-gated. The `#[deprecated]` attribute in Rust is a **compiler lint only**; it does not prevent the method from being exported in the compiled WASM or called on-chain. Any NEAR account can call `verify_transaction_inclusion` directly, completely circumventing the v2 guard.

The underlying Merkle verification in `compute_root_from_merkle_proof` performs no check that the supplied `tx_id` is a leaf node rather than an internal node: [4](#0-3) 

`ProofArgs` accepts an arbitrary `tx_id: H256` with no structural constraint: [5](#0-4) 

---

### Impact Explanation

An attacker who knows the Merkle tree of a confirmed Bitcoin block can craft a `ProofArgs` where `tx_id` is the hash of an internal Merkle node (not a real transaction). `compute_root_from_merkle_proof` will compute the correct Merkle root from this internal node, and `verify_transaction_inclusion` will return `true`. Any consumer contract (bridge, atomic swap, lending protocol) that calls this method to gate a state transition will accept the forged proof and execute the protected action — e.g., releasing funds, minting tokens, or crediting a balance — for a transaction that does not exist.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no social engineering. The attacker only needs:
1. A confirmed Bitcoin block whose Merkle tree they can inspect (public data).
2. The ability to call a NEAR contract method (any NEAR account).

The 64-byte internal-node forgery technique is publicly documented (BitMEX research, CVE-2017-12842). The contract's own comments reference the exact attack vector. The deprecated function is live on every deployed chain variant (bitcoin, litecoin, dogecoin, zcash) because the `#[deprecated]` attribute has no runtime effect in compiled WASM.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely, or restrict it with an access-control role (e.g., `Role::DAO`) so it cannot be called by unprivileged accounts. The `#[deprecated]` Rust attribute is a compile-time lint and provides zero on-chain protection. `verify_transaction_inclusion_v2` should be the sole publicly callable proof-verification endpoint. [6](#0-5) 

---

### Proof of Concept

1. Identify a confirmed Bitcoin block `B` with at least 4 transactions. Its Merkle tree has internal nodes whose hashes are computable from public data.
2. Pick internal node `N` at depth 1 (hash of two leaf transactions). Compute the sibling path from `N` to the Merkle root — this is a valid Merkle proof of length `tree_depth - 1`.
3. Call `verify_transaction_inclusion` on the NEAR contract with:
   - `tx_id` = hash of internal node `N`
   - `tx_block_blockhash` = block hash of `B` (confirmed, in mainchain)
   - `tx_index` = position of `N` in the level above the leaves
   - `merkle_proof` = sibling path from `N` to root
   - `confirmations` = 1
4. `compute_root_from_merkle_proof(N, tx_index, proof)` reconstructs the correct Merkle root of `B`.
5. The function returns `true`.
6. A consumer contract that called `verify_transaction_inclusion` to authorize a withdrawal now releases funds for a transaction that was never broadcast. [7](#0-6) [4](#0-3)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
