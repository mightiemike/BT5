### Title
`coinbase_tx_id` Is Not Validated Against the Actual Coinbase Transaction in `verify_transaction_inclusion_v2`, Bypassing the 64-Byte Merkle Forgery Mitigation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` accepts a caller-supplied `coinbase_tx_id` and only checks that it hashes to the block's Merkle root via the provided `coinbase_merkle_proof`. It never validates that `coinbase_tx_id` is the actual coinbase transaction (the leaf at index 0). An unprivileged NEAR caller can supply an internal Merkle tree node as the "coinbase" and another internal node as the "transaction", satisfying every check in the function and causing it to return `true` for a transaction that does not exist in the block.

---

### Finding Description

The function `verify_transaction_inclusion_v2` was introduced to mitigate the [64-byte transaction Merkle proof forgery](https://www.bitmex.com/blog/64-Byte-Transactions). The mitigation relies on requiring a coinbase proof of the same length as the transaction proof, which is intended to fix the tree depth and prevent an internal node from being presented as a leaf.

The critical flaw is that `coinbase_tx_id` is a free caller-supplied parameter. The contract only verifies:

```
compute_root_from_merkle_proof(args.coinbase_tx_id, 0, &args.coinbase_merkle_proof)
    == header.block_header.merkle_root
``` [1](#0-0) 

This check passes for **any** value that hashes to the Merkle root with the given proof — including an internal Merkle tree node. Because `coinbase_tx_id` is never cross-checked against the block's actual coinbase transaction, the depth-fixing invariant the mitigation relies on is broken: an attacker can supply a pair of internal nodes (one as "coinbase", one as "transaction") with equal-length proofs, and both proofs will verify against the Merkle root.

The `compute_root_from_merkle_proof` function in `merkle-tools/src/lib.rs` is a pure hash-chain computation with no leaf/node distinction: [2](#0-1) 

It accepts any starting hash and any position, so it is equally valid for internal nodes as for leaf transactions.

---

### Impact Explanation

`verify_transaction_inclusion_v2` is the production-recommended SPV proof endpoint (the v1 is explicitly deprecated). Any bridge, atomic swap, or cross-chain lending protocol that calls this function to confirm a Bitcoin payment will receive a forged `true` result for a transaction that was never broadcast or mined. This enables an attacker to claim funds on NEAR without ever sending a real Bitcoin transaction.

---

### Likelihood Explanation

The attack requires only:
1. Any real Bitcoin block that is already in the contract's mainchain (public information).
2. Knowledge of that block's Merkle tree structure (fully public; derivable from any Bitcoin block explorer).
3. A standard NEAR function call — no privileged role, no staking, no key compromise.

All inputs are computable offline from public data. The call is open to any NEAR account.

---

### Recommendation

After verifying the coinbase proof, assert that `args.coinbase_tx_id` equals the coinbase transaction hash stored or derivable for that block, **or** require the caller to supply the raw coinbase transaction bytes and hash them on-chain to obtain the coinbase txid before running the proof. At minimum, document that `coinbase_tx_id` must be the actual leaf-0 transaction and add an on-chain check against a trusted source (e.g., a stored coinbase hash per block, or a separate coinbase-bytes parameter that is hashed in-contract).

---

### Proof of Concept

Consider a mainchain block with four transactions `[T0, T1, T2, T3]` and the following Merkle tree:

```
        Root
       /    \
     N01    N23
    /   \  /   \
   T0  T1 T2  T3

N01 = double_sha256(T0 || T1)
N23 = double_sha256(T2 || T3)
Root = double_sha256(N01 || N23)
```

The attacker calls `verify_transaction_inclusion_v2` with:

| Field | Value |
|---|---|
| `tx_block_blockhash` | hash of the real block (in mainchain) |
| `coinbase_tx_id` | **N01** (internal node, not the real coinbase T0) |
| `coinbase_merkle_proof` | `[N23]` (length 1) |
| `tx_id` | **N23** (internal node, not a real transaction) |
| `merkle_proof` | `[N01]` (length 1) |
| `tx_index` | 1 |
| `confirmations` | any valid value |

Step-by-step evaluation inside the contract:

1. **Length check** — `merkle_proof.len() == coinbase_merkle_proof.len()` → `1 == 1` ✓ [3](#0-2) 

2. **Coinbase proof check** — `compute_root_from_merkle_proof(N01, 0, [N23])`:
   - position 0 is even → `hash(N01, N23)` = `Root` ✓ [4](#0-3) 

3. **Transaction proof check** (inside `verify_transaction_inclusion`) — `compute_root_from_merkle_proof(N23, 1, [N01])`:
   - position 1 is odd → `hash(N01, N23)` = `Root` ✓ [5](#0-4) 

4. **Result**: function returns `true` for `tx_id = N23`, which is an internal Merkle node — not a real Bitcoin transaction.

The `coinbase_tx_id` parameter in `ProofArgsV2` is the unvalidated caller-supplied input that is the root cause: [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
