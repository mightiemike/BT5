### Title
Odd-Leaf Duplicate Position Ambiguity in `compute_root_from_merkle_proof` Allows Phantom `tx_index` to Verify Against Real Merkle Root â€” (`merkle-tools/src/lib.rs` + `contract/src/lib.rs`)

---

### Summary

`compute_root_from_merkle_proof` does not bind a proof to a unique leaf position. In any Bitcoin block whose transaction count is odd, the last leaf is duplicated during tree construction. This creates a mathematical collision: the same `(tx_id, merkle_proof)` tuple produces the correct Merkle root for **both** `tx_index = N-1` (the real last leaf) and `tx_index = N` (a phantom position that does not exist in the block). `verify_transaction_inclusion` passes the caller-supplied `tx_index` directly into this function with no bounds check, so it returns `true` for the phantom index. Any downstream bridge, unlock, or mint contract that uses `(tx_id, tx_index)` as its replay-protection key can be double-spent.

---

### Finding Description

**Root cause â€” `compute_root_from_merkle_proof`:** [1](#0-0) 

The function iterates over `merkle_proof`, at each step choosing left/right by `current_position % 2`, then halving `current_position`. There is no check that `transaction_position` is within the actual leaf count of the tree.

**Why odd trees create a collision:**

Consider a block with 3 transactions `[T0, T1, T2]`. Bitcoin's Merkle construction duplicates the last leaf:

```
Level 0 (padded): [T0,  T1,  T2,  T2]
Level 1:          [H(T0,T1), H(T2,T2)]
Root:              H(H(T0,T1), H(T2,T2))
```

The canonical proof for `tx_index=2` (real, even):
- Step 1: pos=2 (even) â†’ `H(T2, proof[0])` where `proof[0]=T2` â†’ `H(T2,T2)`; posâ†’1
- Step 2: pos=1 (odd) â†’ `H(proof[1], H(T2,T2))` where `proof[1]=H(T0,T1)` â†’ **Root** âś“

The same proof `[T2, H(T0,T1)]` with `tx_index=3` (phantom, odd):
- Step 1: pos=3 (odd) â†’ `H(proof[0], T2)` where `proof[0]=T2` â†’ `H(T2,T2)`; posâ†’1
- Step 2: pos=1 (odd) â†’ `H(proof[1], H(T2,T2))` where `proof[1]=H(T0,T1)` â†’ **Root** âś“

Both paths produce the identical root. The proof is not position-binding.

**The entrypoint â€” `verify_transaction_inclusion`:** [2](#0-1) 

The function is public (no `#[private]`, no `#[trusted_relayer]`), only gated by `#[pause]`. It passes the caller-supplied `args.tx_index` directly to `compute_root_from_merkle_proof` with no upper-bound validation: [3](#0-2) 

The block header stored on-chain contains only the Merkle root, not the transaction count: [4](#0-3) 

So the contract has no data available to reject an out-of-range index.

**`ProofArgs` â€” all fields are caller-controlled:** [5](#0-4) 

`tx_index` is a plain `u64` with no validation beyond the `usize` cast.

**`verify_transaction_inclusion_v2` inherits the bug:** [6](#0-5) 

The v2 function validates the coinbase proof (mitigating the 64-byte forgery), but then calls the deprecated v1 function unchanged. The phantom-index collision is unaffected by coinbase validation because the coinbase is at `tx_index=0`, a completely separate proof path.

---

### Impact Explanation

A downstream bridge that uses `(tx_id, tx_index)` as its replay-protection key (a common pattern when a single transaction can fund multiple outputs or when the bridge wants to distinguish "which slot was claimed") can be double-spent:

1. Attacker observes real transaction `T2` at position `N-1` in an odd-tx block.
2. First call: `verify_transaction_inclusion(T2, block, N-1, proof)` â†’ `true` â†’ bridge processes, marks `(T2, N-1)` as used.
3. Second call: `verify_transaction_inclusion(T2, block, N, proof)` â†’ `true` â†’ bridge does not find `(T2, N)` in its used-set â†’ processes again â†’ **double-spend / double-mint**.

The light client itself is the source of the incorrect `true` return for a position that does not exist in the block.

---

### Likelihood Explanation

- Roughly half of all Bitcoin blocks have an odd transaction count, making the precondition common.
- The attacker needs no special privileges â€” the function is callable by any NEAR account.
- The proof to supply is the canonical proof for the real last transaction; no cryptographic work is required.
- The only external dependency is a downstream bridge that keys replay protection on `(tx_id, tx_index)` rather than `tx_id` alone.

---

### Recommendation

1. **Short-term:** Require callers to supply the total transaction count (`tx_count`) alongside the proof, and validate `tx_index < tx_count`. Compute the expected proof length as `ceil(log2(tx_count))` and reject proofs whose length does not match.
2. **Longer-term:** Migrate all callers to `verify_transaction_inclusion_v2` and add the same `tx_count` guard there. The coinbase proof in v2 implicitly bounds the tree depth but does not prevent a phantom sibling at the same depth.
3. **Downstream bridges:** Until the contract is patched, bridges should key replay protection on `tx_id` alone (not `tx_index`), or independently verify the raw transaction data to confirm the index is within the real transaction list.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
