### Title
Missing `approve(0)` Before Re-approval in `DirectDepositV1.creditDeposit()` Permanently Blocks USDT Deposits - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For USDT and other non-standard ERC20 tokens that revert when `approve()` is called with a non-zero existing allowance, any residual allowance from a prior incomplete deposit cycle will permanently brick USDT deposits for the affected subaccount's DDA.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each spot product token held by the DDA contract, the code unconditionally calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The approval is set to `balance` (a `uint256`), but the deposit call passes `uint128(balance)`. If `balance > type(uint128).max`, the endpoint's `transferFrom` only consumes `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)`. On the next invocation of `creditDeposit()`, the call to `token.approve(address(endpoint), newBalance)` will revert for USDT because USDT requires the existing allowance to be zero before a new non-zero approval is set.

Even absent the `uint128` truncation path, any scenario in which the endpoint does not consume the full approved amount (e.g., internal deposit caps, fee deductions, or future endpoint logic changes) leaves a non-zero residual allowance that permanently blocks subsequent `creditDeposit()` calls for USDT on that DDA.

The same codebase demonstrates awareness of this requirement: `ContractOwner.wrapVaultAsset()` correctly calls `approve(0)` before `approve(assetBalance)`:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [2](#0-1) 

This pattern was applied in `wrapVaultAsset` but omitted in `creditDeposit()`.

---

### Impact Explanation

Once a non-zero USDT allowance is left on a DDA contract, every subsequent call to `creditDeposit()` for that DDA will revert at the `approve` step for USDT. The subaccount's DDA becomes permanently unable to deposit USDT collateral into the protocol. Funds sent to the DDA are not lost (they remain in the DDA contract), but they cannot be credited to the subaccount without an admin intervention (e.g., `withdrawFromDirectDepositV1`). This constitutes a token-transfer integration failure that blocks a supported collateral asset from being usable through the DDA flow.

---

### Likelihood Explanation

`creditDeposit()` is `external` with no access control modifier. [3](#0-2) 

`ContractOwner.creditDepositV1()` is also `external` with no access control, meaning any unprivileged caller can trigger the deposit flow for any subaccount. [4](#0-3) 

The `uint128` truncation trigger requires an astronomically large USDT balance and is not realistic. However, any endpoint behavior that does not consume the full approved `uint256` amount (including future endpoint upgrades) will trigger the bug. The missing `approve(0)` is a latent defect that becomes exploitable the moment any such condition is met.

---

### Recommendation

Prepend `token.approve(address(endpoint), 0)` before each `token.approve(address(endpoint), balance)` call in `creditDeposit()`, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`. Alternatively, use OpenZeppelin's `SafeERC20.forceApprove()` (or `safeIncreaseAllowance` / `safeDecreaseAllowance`) to handle non-standard ERC20 tokens safely.

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
```

---

### Proof of Concept

1. A subaccount's `DirectDepositV1` DDA receives USDT.
2. `creditDeposit()` is called (by anyone). It calls `token.approve(address(endpoint), balance)` setting USDT allowance to `balance`.
3. The endpoint's `depositCollateralWithReferral` consumes less than `balance` tokens (e.g., due to `uint128` truncation if `balance > type(uint128).max`, or any internal cap/fee logic), leaving a non-zero residual USDT allowance on the DDA.
4. More USDT is sent to the DDA.
5. `creditDeposit()` is called again. At step `token.approve(address(endpoint), newBalance)`, USDT reverts because the existing allowance is non-zero.
6. All future `creditDeposit()` calls for USDT on this DDA revert. The subaccount can no longer deposit USDT collateral via the DDA flow. [5](#0-4)
