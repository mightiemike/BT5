### Title
No Emergency Recovery Mechanism for Escrowed NFTs When IBC Channel Closes Permanently - (`x/nft-transfer/ibc_module.go`)

### Summary

The `x/nft-transfer` module locks NFTs in a per-channel escrow address during outbound IBC transfers. If the channel is permanently closed (e.g., counterparty-initiated closure after a frozen or expired IBC client), the `OnChanCloseConfirm` handler performs no recovery action, and no admin or governance mechanism exists to release the escrowed NFTs. The only recovery paths — `OnTimeoutPacket` and error-ack — are unavailable once the IBC client is frozen or expired, leaving escrowed NFTs permanently inaccessible.

---

### Finding Description

In `createOutgoingPacket`, when the sending chain is the source chain (`isAwayFromOrigin = true`), each NFT is transferred from the sender to the channel-specific escrow address:

```go
escrowAddress := types.GetEscrowAddress(sourcePort, sourceChannel)
if err := k.nftKeeper.TransferOwner(ctx, classID, tokenID, sender, escrowAddress); err != nil {
    return channeltypes.Packet{}, err
}
``` [1](#0-0) 

These escrowed NFTs are only released via two paths:

1. `OnTimeoutPacket` → `refundPacketToken` (unescrows back to sender)
2. `OnAcknowledgementPacket` with error → `refundPacketToken` (unescrows back to sender) [2](#0-1) 

However, `OnChanCloseConfirm` — triggered when the counterparty closes the channel — performs no recovery:

```go
func (im IBCModule) OnChanCloseConfirm(
    ctx sdk.Context,
    portID,
    channelID string,
) error {
    return nil
}
``` [3](#0-2) 

While IBC provides `MsgTimeoutOnClose` to time out in-flight packets after channel closure, this mechanism requires the IBC light client to be active. If the client is frozen (e.g., due to a misbehaviour submission) or expired at the time of channel closure, `MsgTimeoutOnClose` cannot be submitted. No admin message, governance proposal, or module-level escape hatch exists to release the escrowed NFTs in this scenario.

The module also explicitly blocks user-initiated channel closure:

```go
func (im IBCModule) OnChanCloseInit(...) error {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidRequest, "user cannot close channel")
}
``` [4](#0-3) 

This means the only channel-close path is counterparty-initiated, which is outside the local chain's control.

---

### Impact Explanation

NFTs locked in the channel escrow address (`GetEscrowAddress(portID, channelID)`) become permanently inaccessible to their original owners. The escrow address is a deterministic module account with no private key; only the IBC packet lifecycle callbacks can move NFTs out of it. If those callbacks are unreachable (frozen/expired client), the NFTs are unrecoverable. The corrupted invariant is: **every NFT transferred into escrow must eventually be either delivered to the destination or returned to the sender**. This invariant breaks when the IBC client is frozen and the channel is closed. [5](#0-4) 

---

### Likelihood Explanation

Low. The scenario requires two concurrent conditions: (1) an IBC channel used for NFT transfer is closed by the counterparty, and (2) the associated IBC light client is frozen or expired at the time of closure, preventing `MsgTimeoutOnClose`. IBC client freezing can occur via a governance `MsgSubmitMisbehaviour` or natural expiry. This is the same low-likelihood window described in the original report (remote path becomes permanently inaccessible after send but before receipt/timeout).

---

### Recommendation

Implement an admin or governance-gated recovery message (e.g., `MsgReclaimEscrowedNFT`) that allows releasing NFTs from a closed channel's escrow address back to their original owners, gated by proof that the channel is closed and the packet was never acknowledged. Alternatively, implement recovery logic inside `OnChanCloseConfirm` that iterates pending in-flight packets and refunds them, consistent with how some ICS-20 implementations handle this case.

---

### Proof of Concept

1. User calls `MsgTransfer` for NFT `tokenID` on `classID` via `nft/channel-0`. `createOutgoingPacket` transfers the NFT to `escrowAddress = GetEscrowAddress("nft", "channel-0")`. [1](#0-0) 

2. Before the packet is relayed, the counterparty chain's governance freezes the IBC client (e.g., via misbehaviour evidence).

3. The counterparty chain initiates `ChanCloseInit`. The local chain processes `ChanCloseConfirm`, which calls `OnChanCloseConfirm` — returning `nil` with no recovery action. [3](#0-2) 

4. The user attempts `MsgTimeoutOnClose` to recover the NFT. This fails because the IBC client is frozen and cannot verify the proof.

5. No admin message exists to release the NFT from `escrowAddress`. The NFT is permanently locked. [6](#0-5)

// GetEscrowAddress returns the escrow address for the specified channel.
// The escrow address follows the format as outlined in ADR 028:
// https://github.com/cosmos/cosmos-sdk/blob/master/docs/architecture/adr-028-public-key-addresses.md
func GetEscrowAddress(portID, channelID string) sdk.AccAddress {
	// a slash is used to create domain separation between port and channel identifiers to
	// prevent address collisions between escrow addresses created for different channels
	contents := fmt.Sprintf("%s/%s", portID, channelID)

	// ADR 028 AddressHash construction
	preImage := []byte(Version)
	preImage = append(preImage, 0)
	preImage = append(preImage, contents...)
	hash := sha256.Sum256(preImage)
	return hash[:20]
```
