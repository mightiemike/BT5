### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable On-Chain, Enabling 64-Byte Merkle Proof Forgery to Bypass Coinbase Proof Mitigation — (`contract/src/lib.rs`)

---

### Summary

The contract introduced `verify_transaction_inclusion_v2` to mitigate the well-documented 64-byte transaction Merkle proof forgery vulnerability. However, the original `verify_transaction_inclusion` function — explicitly marked `#[deprecated]` and carrying an in-code warning that it can return `true` for internal Merkle tree nodes — remains a fully public, unguarded `#[pause]` method callable by any unprivileged NEAR account. Any external contract consuming the verification result can be deceived into accepting a fraudulent Bitcoin transaction inclusion proof by calling v1 directly, bypassing v2's coinbase proof check entirely.

---

### Finding Description

`verify_transaction_inclusion` is annotated `#[deprecated(since = "0.5.0")]` and carries an explicit warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

Despite this, the function is declared `pub` with only `#[pause]` (no `#[private]`, no `#[trusted_relayer]`, no role restriction), making it callable by any NEAR account whenever the contract is not paused: [2](#0-1) 

The v2 function was introduced specifically to close this gap by requiring a coinbase Merkle proof that proves the coinbase transaction is at index 0, preventing an attacker from substituting an internal Merkle tree node as a "transaction": [3](#0-2) 

The `From<ProofArgsV2> for ProofArgs` conversion in `btc-types` drops `coinbase_tx_id` and `coinbase_merkle_proof` when v2 delegates to v1 internally: [4](#0-3) 

The cross-module desynchronization: the contract's security model was updated (v1 → v2) to require coinbase proof validation, but the old v1 entry point was never removed or access-restricted. Any caller — including an adversarial NEAR contract consuming verification results — can invoke v1 directly, bypassing the coinbase proof check entirely.

The actual proof check in v1 is: [5](#0-4) 

`compute_root_from_merkle_proof` in `merkle-tools` is a pure hash-chain computation with no validation that `transaction_hash` is a leaf node: [6](#0-5)

### Impact Explanation

An attacker can forge a Bitcoin transaction inclusion proof accepted as `true` by the on-chain contract. Specifically:

- An internal Merkle tree node `N` at position `i` in level `L` of a real Bitcoin block's Merkle tree is a valid 64-byte value.
- The attacker submits `tx_id = N`, `tx_index = i`, and `merkle_proof = [siblings at levels L+1 … root]`.
- `compute_root_from_merkle_proof(N, i, proof)` correctly reconstructs the block's `merkle_root`.
- `verify_transaction_inclusion` returns `true`.
- Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate asset releases, bridge withdrawals, or other state transitions accepts a fraudulent proof.

**Impact: High** — proof-verification forgery enabling fraudulent acceptance of non-existent Bitcoin transactions.

---

### Likelihood Explanation

**Likelihood: High.** The function is unconditionally public, requires no privileged role, and is callable by any NEAR account. The 64-byte Merkle tree attack is fully documented (referenced in the contract's own codebase at the v2 docstring). All inputs are attacker-controlled. No special conditions are required beyond the contract not being paused.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public contract interface entirely. Since `verify_transaction_inclusion_v2` delegates to it internally via `self.verify_transaction_inclusion(args.into())`, refactor the shared logic into a private helper function and make the v1 public entry point unreachable (e.g., add `env::panic_str("use verify_transaction_inclusion_v2")` at the top, or remove the `pub` qualifier and the `#[pause]` attribute).

---

### Proof of Concept

1. Identify any real Bitcoin block with ≥ 2 transactions (all mainnet blocks qualify). Let its `merkle_root` be `R`.
2. Obtain the full Merkle tree. Pick any internal node `N` at tree level `L`, position `i`.
3. Collect the sibling hashes at levels `L+1` through the root — this is the Merkle proof `P`.
4. Construct `ProofArgs { tx_id: N, tx_block_blockhash: <block_hash>, tx_index: i, merkle_proof: P, confirmations: 1 }`.
5. Call `verify_transaction_inclusion` on the NEAR contract with these args (Borsh-serialized).
6. `compute_root_from_merkle_proof(N, i, P)` returns `R == header.block_header.merkle_root` → function returns `true`.
7. The contract has accepted a proof for a "transaction" that does not exist. [7](#0-6) [8](#0-7)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
