### Title
`Verifier` Can Be Initialized With Zero Signers, Allowing Signature-Free Fast Withdrawals - (File: `core/contracts/Verifier.sol`, `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`Verifier.initialize()` does not enforce that at least one valid signer pubkey is registered. If initialized with all-zero points, `nSigner` remains `0`. `Verifier.requireValidTxSignatures()` enforces `require(nSignatures == nSigner)`, which trivially passes with an empty signatures array when `nSigner == 0`. `BaseWithdrawPool.submitFastWithdrawal()` is a public function that relies solely on this check for authorization, allowing any caller to drain collateral tokens from the `WithdrawPool` contract without any valid signature.

---

### Finding Description

`Verifier.initialize()` iterates over the provided `initialSet` and only calls `_assignPubkey()` (which increments `nSigner`) for non-zero points: [1](#0-0) 

If all 8 entries in `initialSet` are zero points (`x == 0 && y == 0`), `nSigner` is never incremented and remains `0` after initialization. There is no post-initialization check requiring `nSigner > 0`.

`requireValidTxSignatures()` enforces the quorum as: [2](#0-1) 

When `nSigner == 0`, passing `signatures = []` (empty array) results in `nSignatures = 0`, and `require(0 == 0)` passes unconditionally. No cryptographic verification occurs.

`BaseWithdrawPool.submitFastWithdrawal()` is `public` and its only authorization gate is this call: [3](#0-2) 

After the bypassed signature check, the function decodes an attacker-supplied `transaction` to extract `productId`, `sendTo`, and `transferAmount`, then calls `handleWithdrawTransfer(token, sendTo, transferAmount)` — transferring real ERC-20 collateral tokens out of the pool.

---

### Impact Explanation

An attacker can call `submitFastWithdrawal()` with an arbitrary crafted `transaction` encoding any `sendTo` address and `amount`, and an empty `signatures` array. The `WithdrawPool` contract holds real collateral tokens deposited by users awaiting withdrawal settlement. All tokens held in the pool can be drained to an arbitrary address. The corrupted asset delta is the full ERC-20 token balance of the `WithdrawPool` contract.

---

### Likelihood Explanation

`Verifier.initialize()` accepts an `initialSet` parameter with no on-chain enforcement that any entry is non-zero. A deployment error, misconfiguration, or a deployer passing a default-initialized `Point[8]` array results in `nSigner == 0`. This is directly analogous to the reported `AccountManage` issue where `signNum` was not set during `init()`. The entry path (`submitFastWithdrawal`) is public and requires no special role or privilege.

---

### Recommendation

In `Verifier.initialize()`, add a post-initialization check requiring `nSigner > 0`:

```solidity
function initialize(Point[8] memory initialSet) external initializer {
    __Ownable_init();
    for (uint256 i = 0; i < 8; ++i) {
        if (!isPointNone(initialSet[i])) {
            _assignPubkey(i, initialSet[i].x, initialSet[i].y);
        }
    }
    require(nSigner > 0, "Must initialize with at least one signer");
}
```

Additionally, `requireValidTxSignatures` should explicitly guard against the zero-signer state:

```solidity
require(nSigner > 0, "No signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

If the `Verifier` is already deployed with `nSigner == 0`, `assignPubkey()` must be called immediately by the owner to register at least one valid signer before any `WithdrawPool` liquidity is present.

---

### Proof of Concept

1. Deploy `Verifier` via `initialize(Point[8] memory initialSet)` where all 8 entries are `Point(0, 0)`. `nSigner` remains `0`.
2. `WithdrawPool` is initialized pointing to this `Verifier`. Collateral tokens accumulate in the pool as users withdraw.
3. Attacker calls:
   ```solidity
   withdrawPool.submitFastWithdrawal(
       validIdx,                          // any idx > minIdx not yet marked
       craftedTransaction,                // encodes sendTo=attacker, amount=poolBalance
       new bytes[](0)                     // empty signatures array
   );
   ```
4. Inside `submitFastWithdrawal`, `v.requireValidTxSignatures(transaction, idx, new bytes[](0))` executes. `nSignatures = 0`, `nSigner = 0`, `require(0 == 0)` passes.
5. `resolveFastWithdrawal` decodes `sendTo = attacker`, `transferAmount = poolBalance`.
6. `handleWithdrawTransfer(token, attacker, poolBalance)` transfers all tokens to the attacker. [4](#0-3) [5](#0-4)

```text
    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```
