### Title
`ResourceBoundsMapping` → `ValidResourceBounds` Heuristic Produces Wrong Hash Domain in RPC Simulation for V3 Transactions with Zero L2/L1DataGas Bounds - (File: crates/apollo_rpc/src/v0_8/transaction.rs)

### Summary
The RPC layer converts the wire-format `ResourceBoundsMapping` to the internal `ValidResourceBounds` enum using a value-based heuristic: if both `l1_data_gas` and `l2_gas` are zero, it selects the `L1Gas` variant; otherwise `AllResources`. The gateway, however, always uses `AllResources` for every V3 transaction because `RpcInvokeTransactionV3` holds `AllResourceBounds` directly. For a V3 transaction with `l1_gas = X`, `l2_gas = 0`, `l1_data_gas = 0`, the gateway computes a transaction hash that includes `l1_data_gas` in the Poseidon preimage (4-element hash), while the RPC simulation/estimation path computes a hash that omits it (3-element hash). These are cryptographically distinct values. When `starknet_simulateTransactions` is called without `SKIP_VALIDATE`, the account's `__validate__` function receives the wrong hash, causing signature verification to fail for a transaction the gateway would accept and execute successfully.
