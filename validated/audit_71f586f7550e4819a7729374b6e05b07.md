### Title
`gas_prices_to_hash` omits `l2_gas_consumed` and `next_l2_gas_price` from block hash preimage, causing sequencer block hash to diverge from echonet/CLI canonical hash — (File: `crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

The `gas_prices_to_hash` function, which feeds the gas-price slot of `calculate_block_hash`, permanently omits `l2_gas_consumed` and `next_l2_gas_price` from the Poseidon preimage. Both fields are first-class members of `BlockHeaderWithoutHash` and are already forwarded to the block-hash CLI by the production `echonet` component. The Rust sequencer therefore produces a block hash that does not commit to the fee-market outcome of the block, while the echonet CLI produces a different hash for the same block. Two blocks that differ only in `l2_gas_consumed` / `next_l2_gas_price` are hash-identical under the Rust path, and any block hash served by the sequencer's RPC is authoritative-looking but wrong relative to the protocol-canonical value.

---

### Finding Description

**The omission in `gas_prices_to_hash`**

`gas_prices_to_hash` accepts only three price arguments and hashes only six felts (for `BlockHashVersion::V0_13_4`):

```
STARKNET_GAS_PRICES0 | l1_gas_price_wei | l1_gas_price_fri
                     | l1_data_gas_price_wei | l1_data_gas_price_fri
                     | l2_gas_price_wei | l2_gas_price_fri
```

An explicit TODO acknowledges the gap:

```rust
// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.
pub fn gas_prices_to_hash(
    l1_gas_price: &GasPricePerToken,
    l1_data_gas_price: &GasPricePerToken,
    l2_gas_price: &GasPricePerToken,
    block_hash_version: &BlockHashVersion,
) -> Vec<Felt> { … }
```

The current Starknet version is ≥ 0.14.4, so the "after 0.14.0" gate has long passed.

**The fields exist in the header but not in the hash input struct**

`BlockHeaderWithoutHash` carries both fields:

```rust
pub struct BlockHeaderWithoutHash {
    …
    pub l2_gas_consumed: GasAmount,
    pub next_l2_gas_price: GasPrice,
    …
}
```

`PartialBlockHashComponents`, the sole input to `calculate_block_hash`, does not:

```rust
pub struct PartialBlockHashComponents {
    pub header_commitments: BlockHeaderCommitments,
    pub block_number: BlockNumber,
    pub l1_gas_price: GasPricePerToken,
    pub l1_data_gas_price: GasPricePerToken,
    pub l2_gas_price: GasPricePerToken,
    pub sequencer: SequencerContractAddress,
    pub timestamp: BlockTimestamp,
    pub starknet_version: StarknetVersion,
    // l2_gas_consumed and next_l2_gas_price are absent
}
```

Even if `gas_prices_to_hash` were patched, the data is structurally unreachable from `calculate_block_hash`.

**The echonet CLI already includes both fields**

The production `echonet/echo_center.py` `_compute_block_hash` method passes both fields to the block-hash CLI binary:

```python
result = self._run_block_hash_cli(
    "block-hash",
    {
        "header": {
            …
            "l2_gas_consumed": fee_market_info["l2_gas_consumed"],
            "next_l2_gas_price": fee_market_info["next_l2_gas_price"],
            …
        },
        …
    },
)
```

The CLI is the canonical hash reference. The Rust `calculate_block_hash` is not equivalent to it.

**The version gate does not protect against this**

`BlockHashVersion::try_from(StarknetVersion)` maps all versions ≥ 0.13.4 to `V0_13_4`. There is no `V0_14_x` variant that would trigger inclusion of the missing fields. The gate that should have been added "after 0.14.0" was never introduced.

---

### Impact Explanation

**Wrong block hash served by RPC (High)**  
`starknet_getBlockWithTxHashes` and `starknet_getBlockWithTxs` return the hash computed by `calculate_block_hash`. That hash does not commit to `l2_gas_consumed` or `next_l2_gas_price`. The value is authoritative-looking but diverges from the echonet/CLI canonical hash for every block produced on version ≥ 0.14.0.

**Hash collision across distinct fee-market states**  
Two blocks that are identical except for `l2_gas_consumed` or `next_l2_gas_price` produce the same Rust-side block hash. `next_l2_gas_price` is the EIP-1559-style base fee for the next block; a manipulated value that is hash-invisible can silently alter the gas price schedule without invalidating any stored hash.

**Sequencer–echonet hash mismatch in production**  
The echonet `_log_block_hash_comparison` already records mismatches between its CLI-computed hash and the mainnet hash. The Rust sequencer's hash is a third, distinct value for the same block, meaning any cross-component hash verification (consensus, proof, L1 settlement) operates on inconsistent values.

---

### Likelihood Explanation

The divergence is present in every block with `starknet_version ≥ 0.14.0`. The TODO has been in the codebase since at least the 0.14.0 era; the current version is 0.14.4. The echonet code already forwards the missing fields to the CLI, confirming the protocol intends them to be included. No version gate, canonicalization check, or fallback prevents the omission.

---

### Recommendation

1. Add `l2_gas_consumed: GasAmount` and `next_l2_gas_price: GasPrice` to `PartialBlockHashComponents`.
2. Introduce a new `BlockHashVersion::V0_14_0` (or appropriate version) variant in `BlockHashVersion` and extend `TryFrom<StarknetVersion>` to map versions ≥ 0.14.0 to it.
3. Update `gas_prices_to_hash` to chain `l2_gas_consumed` and `next_l2_gas_price` into the Poseidon preimage when `block_hash_version >= V0_14_0`.
4. Update `PartialBlockHashComponents::new` to populate the new fields from `BlockInfo` (or pass them explicitly).
5. Add a regression test that verifies the Rust hash matches the echonet CLI output for a real 0.14.x block.

---

### Proof of Concept

```
Given: two PartialBlockHashComponents structs A and B that are identical
       except A.l2_gas_consumed = 0, B.l2_gas_consumed = 1_000_000
       and   A.next_l2_gas_price = 1, B.next_l2_gas_price = 999_999_999

Step 1: call calculate_block_hash(&A, root, parent)
Step 2: call calculate_block_hash(&B, root, parent)

Result (current code): hash_A == hash_B
  — because PartialBlockHashComponents does not carry l2_gas_consumed or
    next_l2_gas_price, so gas_prices_to_hash receives identical inputs.

Step 3: call the echonet block-hash CLI with block A's full header
Step 4: call the echonet block-hash CLI with block B's full header

Result (CLI): cli_hash_A ≠ cli_hash_B  (fields are included)
              cli_hash_A ≠ hash_A       (sequencer diverges from CLI)

Consequence: the sequencer's RPC returns hash_A for block A, but the
             canonical protocol hash is cli_hash_A. Any verifier that
             recomputes the hash from the header fields (including
             l2_gas_consumed / next_l2_gas_price) will reject the
             sequencer's stored hash as invalid.
```

The analog to the flash-loan reward-inflation bug is exact: just as a deposit and withdrawal in the same block left the reward calibration timestamp uncommitted (allowing inflated rewards to persist), here `l2_gas_consumed` and `next_l2_gas_price` are uncommitted in the block hash (allowing fee-market state to be silently altered without hash invalidation). pub enum BlockHashVersion {
```rust
    V0_13_2,
    V0_13_4,
}

impl From<BlockHashVersion> for StarknetVersion {
    fn from(value: BlockHashVersion) -> Self {
        match value {
            BlockHashVersion::V0_13_2 => StarknetVersion::V0_13_2,
            BlockHashVersion::V0_13_4 => StarknetVersion::V0_13_4,
        }
    }
}

impl TryFrom<StarknetVersion> for BlockHashVersion {
    type Error = StarknetApiError;

    fn try_from(value: StarknetVersion) -> StarknetApiResult<Self> {
        if value < Self::V0_13_2.into() {
            Err(StarknetApiError::BlockHashVersion { version: value.to_string() })
        } else if value < Self::V0_13_4.into() {
            // Starknet versions 0.13.2 and 0.13.3 both have the same block hash mechanism.
            Ok(Self::V0_13_2)
        } else {
            Ok(Self::V0_13_4)
        }
    }
}

// The prefix constant for the block hash calculation.
type BlockHashConstant = Felt;

impl From<BlockHashVersion> for BlockHashConstant {

            BlockHashVersion::V0_13_2 => *STARKNET_BLOCK_HASH0,
            BlockHashVersion::V0_13_4 => *STARKNET_BLOCK_HASH1,
        }
    }
}
```

```rust
pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
```
