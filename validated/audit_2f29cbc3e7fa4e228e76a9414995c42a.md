### Title
Accrued Bonus Rewards Permanently Forfeited During Redelegation Slash When Pool Is Insufficient — (`x/tieredrewards/keeper/slash.go`)

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the error is silently swallowed. However, `processEventsAndClaimBonus` already advanced the position's bonus checkpoints (`LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`) in memory before the pool check. The subsequent `setPositionWithState` call persists those advanced checkpoints without any payment, permanently forfeiting the accrued bonus for that period.

### Finding Description

`processEventsAndClaimBonus` advances the position's checkpoints unconditionally before checking whether the pool can cover the computed bonus:

```go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // advances LastBonusAccrual
pos.UpdateLastKnownBonded(bonded)                        // advances LastKnownBonded
// ... also pos.UpdateLastEventSeq(entry.Seq) inside the event loop

if totalBonus.IsZero() {
    return sdk.NewCoins(), nil
}
// ...
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // returns error AFTER checkpoints already advanced in-memory
}
``` [1](#0-0) 

In user-driven message paths (`ClaimTierRewards`, `TierUndelegate`, etc.), this error propagates up and the Cosmos SDK transaction rolls back entirely, so the in-memory checkpoint advancement is discarded. The user can retry.

In the `BeforeRedelegationSlashed` hook path, however, `slashRedelegationPosition` explicitly catches and discards the `ErrInsufficientBonusPool` error and then calls `setPositionWithState` with the already-mutated `pos`:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // continues — no return
    } else {
        return err
    }
}
// pos.LastBonusAccrual, LastEventSeq, LastKnownBonded are now advanced in memory
...
return k.setPositionWithState(ctx, pos, ...)   // persists advanced checkpoints
``` [2](#0-1) 

The persisted `LastBonusAccrual` and `LastEventSeq` mark the period as "processed." On the next claim, `processEventsAndClaimBonus` starts from the advanced checkpoint, so the forfeited period is never revisited. The accrued bonus is permanently lost.

### Impact Explanation

A position owner who redelegated their tier position permanently loses all bonus rewards that accrued between their last claim and the moment of the redelegation slash, whenever the bonus pool is empty or insufficient at slash time. The corrupted value is the `LastBonusAccrual` timestamp and `LastEventSeq` counter stored in the `Position` record — both are advanced without the corresponding `SendCoinsFromModuleToAccount` transfer occurring. [3](#0-2) 

### Likelihood Explanation

Two independent conditions must coincide:

1. **Redelegation slash fires** — a validator that was the source of a tier position's redelegation is slashed within the redelegation window. This is a normal, production-reachable event triggered by validator downtime or double-signing.
2. **Bonus pool is empty or insufficient** — the `rewards_pool` module account holds less than the computed bonus. The pool is funded externally; it can run dry during high-demand periods or if governance fails to replenish it.

Neither condition requires privileged access or attacker control. The slash is triggered by the standard Cosmos SDK slashing path → `BeforeRedelegationSlashed` hook → `slashRedelegationPosition`. The pool being empty is a realistic operational state. [4](#0-3) 

### Recommendation

Move `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` to after the successful `SendCoinsFromModuleToAccount` call, so checkpoints only advance when payment actually occurs. Alternatively, in `slashRedelegationPosition`, when the pool is insufficient, do not call `setPositionWithState` with the mutated `pos`; instead preserve the original checkpoints so the user can claim the forfeited period once the pool is replenished.

### Proof of Concept

1. Alice locks tokens in a tier and redelegates to validator B.
2. The bonus pool is drained (e.g., by many concurrent claims or governance inaction).
3. Validator A (the source) is slashed within the redelegation window.
4. `BeforeRedelegationSlashed` fires → `slashRedelegationPosition` → `processEventsAndClaimBonus` computes a non-zero bonus, advances `LastBonusAccrual` to `blockTime`, then returns `ErrInsufficientBonusPool`.
5. `slashRedelegationPosition` logs the error and calls `setPositionWithState` with the mutated `pos`.
6. Alice's `LastBonusAccrual` is now `blockTime`; the accrual period is marked consumed.
7. Alice calls `MsgClaimTierRewards` after the pool is replenished — bonus computation starts from the advanced `LastBonusAccrual`, so the forfeited period yields zero. The bonus is permanently lost. [5](#0-4) [6](#0-5)
