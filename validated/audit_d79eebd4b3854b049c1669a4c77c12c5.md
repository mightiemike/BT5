### Title
`ENABLE_SECP_OPS` Flag Not Checked in 4-Byte Opcode Dispatch Path, Making Flag Protection Unreachable for Legacy Secp Opcodes — (`File: src/chia_dialect.rs`)

---

### Summary

`ClvmFlags::ENABLE_SECP_OPS` is declared as the gate controlling whether `secp256k1_verify` and `secp256r1_verify` are available. It is correctly checked for the 1-byte opcode dispatch path (opcodes 64 and 65). However, the 4-byte opcode dispatch path (`0x13d61f00` and `0x1c3a8f00`) dispatches directly to the same secp implementations **without ever checking `ENABLE_SECP_OPS`**. The flag's protection is only half-wired: any caller that omits `ENABLE_SECP_OPS` (including `MEMPOOL_MODE`) still executes live secp verification when a program uses the 4-byte opcode form.

---

### Finding Description

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function has two separate dispatch paths for secp operations.

**Path 1 — 4-byte opcodes (lines 157–183):** When `op_len == 4`, the function reads the raw 4-byte opcode and matches it:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

There is **no check** for `ClvmFlags::ENABLE_SECP_OPS` here.

**Path 2 — 1-byte opcodes (lines 248–249):** When `op_len == 1`, the match arm correctly gates the same functions:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The flag is defined at line 63 as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The docstring says "secp opcodes 64 and 65" — but the 4-byte opcode variants invoke the identical underlying functions and are not mentioned as exempt from the flag. `MEMPOOL_MODE` (lines 72–76) does **not** include `ENABLE_SECP_OPS`, meaning mempool-mode nodes that intend to reject secp ops will still execute them when the 4-byte opcode form is used.

---

### Impact Explanation

Any caller that constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — including the standard `MEMPOOL_MODE` — expects secp verification to be unavailable. A CLVM program that encodes the secp opcode as the 4-byte atom `0x13d61f00` or `0x1c3a8f00` bypasses the flag entirely and causes live cryptographic verification to run. The concrete corrupted result is:

- `op_secp256k1_verify` / `op_secp256r1_verify` executes and either returns `nil` (valid signature) or raises `Secp256Failed`, **instead of** being routed to `unknown_operator()` which would either charge the unknown-op cost and return `nil`, or raise `Unimplemented` if `NO_UNKNOWN_OPS` is set.
- In `MEMPOOL_MODE` (which sets `NO_UNKNOWN_OPS`), the 4-byte secp opcodes should be rejected as unknown operators, but instead they execute as real signature verifications. This is a consensus/mempool divergence: a node in mempool mode accepts or rejects a transaction based on secp signature validity rather than opcode legality.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the mempool can encode the 4-byte opcode `0x13d61f00` as a raw atom. No special privileges are required. The 4-byte opcode form is documented in the source code itself (lines 172–174), making it discoverable. Any node running `MEMPOOL_MODE` without `ENABLE_SECP_OPS` is affected.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

Alternatively, if the 4-byte opcodes are intentionally always available (as a legacy path predating the flag), the flag's documentation and name should be updated to clarify it only gates the 1-byte aliases, and `MEMPOOL_MODE` should be audited to confirm the intended behavior for the 4-byte path.

---

### Proof of Concept

1. Construct a `ChiaDialect` with `MEMPOOL_MODE` (which includes `NO_UNKNOWN_OPS` but excludes `ENABLE_SECP_OPS`).
2. Submit a CLVM program that invokes opcode `0x13d61f00` (4-byte secp256k1_verify) with a valid pubkey, message, and signature.
3. Observe: `op_secp256k1_verify` executes and returns `nil` (success), charging `SECP256K1_VERIFY_COST = 1300000`.
4. Expected: the opcode should be routed to `unknown_operator()`, which — because `NO_UNKNOWN_OPS` is set — raises `EvalErr::Unimplemented`.
5. The same program using 1-byte opcode `64` is correctly rejected because `ENABLE_SECP_OPS` is absent.
