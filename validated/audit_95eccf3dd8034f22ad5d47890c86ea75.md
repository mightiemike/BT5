### Title
`UnrestrictedSubmitBlocks` Role Bypasses Trusted Relayer Staking Requirement — (`contract/src/lib.rs`)

---

### Summary

The `UnrestrictedSubmitBlocks` role is documented as granting the ability to call the contract API even when it is paused. However, because it is listed in the `bypass_roles` of the `#[trusted_relayer]` macro, it also silently bypasses the trusted relayer staking/registration requirement on `submit_blocks`. This is an over-extension of the role's documented scope: an admin who grants `UnrestrictedSubmitBlocks` to an account (believing they are only granting pause-bypass capability) inadvertently grants that account the ability to submit block headers without being a staked, registered trusted relayer.

---

### Finding Description

In `contract/src/lib.rs`, the `#[trusted_relayer]` macro is applied at the `impl` level with the following configuration:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    manager_roles(Role::DAO, Role::RelayerManager),
    config_roles(Role::DAO)
)]
``` [1](#0-0) 

The `UnrestrictedSubmitBlocks` role is documented as:

```rust
/// Allows to use contract API even after contract is paused
UnrestrictedSubmitBlocks,
``` [2](#0-1) 

The `submit_blocks` function carries both `#[pause]` and `#[trusted_relayer]` attributes:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...)
``` [3](#0-2) 

The `bypass_roles` parameter of the `trusted_relayer` macro causes any account holding `Role::DAO` **or** `Role::UnrestrictedSubmitBlocks` to skip the trusted relayer guard entirely — including the staking and registration checks. This is confirmed by the test helper comment:

> "Grant the `UnrestrictedSubmitBlocks` role to an account so it passes the `#[trusted_relayer]` guard on `submit_blocks`." [4](#0-3) 

The role's documented purpose is narrowly scoped to pause-bypass. Its actual effect is broader: it also bypasses the economic security mechanism (staking) that the trusted relayer system enforces. An admin reading the role documentation has no indication that granting `UnrestrictedSubmitBlocks` removes the staking requirement.

This is structurally identical to the ENS finding: in ENS, approval on the wrapper contract (scoped to wrapped tokens) inadvertently extended to unwrapped tokens via an OR condition. Here, `UnrestrictedSubmitBlocks` (scoped to pause-bypass) inadvertently extends to full relayer capability via the OR in `bypass_roles`.

---

### Impact Explanation

An account holding `UnrestrictedSubmitBlocks` can call `submit_blocks` without satisfying the trusted relayer staking requirement. Inside `submit_blocks`, each submitted header is passed to `submit_block_header`, which calls `submit_block_header_inner`, which can trigger `reorg_chain`: [5](#0-4) 

A successful reorg overwrites `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height`: [6](#0-5) 

These are the exact fields queried by `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to determine whether a block belongs to the canonical chain and whether sufficient confirmations exist: [7](#0-6) 

Corrupting these mappings causes the proof verification functions to return false positives — confirming transactions in blocks that are not on the true canonical chain.

The attack is directly reachable when `skip_pow_verification` is `true` (a flag stored in contract state at init time and used unconditionally in `submit_blocks`): [8](#0-7) 

When PoW verification is enforced (`skip_pow_verification = false`), the attacker still bypasses the staking requirement but must supply headers with valid PoW, which is computationally expensive for Bitcoin/Litecoin/Zcash. For Dogecoin, AuxPoW headers can be crafted by a merge-mining pool operator, making the attack more accessible.

---

### Likelihood Explanation

The attack requires two conditions:

1. An admin grants `UnrestrictedSubmitBlocks` to an account — plausible because the role's documentation does not disclose that it also bypasses the staking requirement. An admin intending only to allow submission during a pause period would naturally grant this role.
2. Either `skip_pow_verification = true` (a deployment-time choice that the README notes is for testing but which is not enforced to be false in production) or the attacker has access to mining resources (lower bar for Dogecoin AuxPoW).

The combination of a misleading role description and a reachable code path makes this a medium-likelihood issue.

---

### Recommendation

Separate the pause-bypass capability from the trusted relayer bypass. The `bypass_roles` of the `trusted_relayer` macro should only include roles that are explicitly intended to bypass the staking/registration requirement (e.g., `Role::DAO`). The pause-bypass for `submit_blocks` should be handled via the `#[pause(except(roles(...)))]` pattern already used for `run_mainchain_gc`: [9](#0-8) 

Concretely:
- Remove `Role::UnrestrictedSubmitBlocks` from `bypass_roles` in the `trusted_relayer` macro.
- Add `#[pause(except(roles(Role::UnrestrictedSubmitBlocks)))]` to `submit_blocks` so the role only bypasses the pause guard, not the staking guard.
- Update the role documentation to accurately describe its actual effect.

---

### Proof of Concept

1. Admin grants `UnrestrictedSubmitBlocks` to account A, believing they are only enabling submission during a contract pause.
2. Account A calls `submit_blocks` with adversarial block headers. The `#[trusted_relayer]` guard is bypassed because `UnrestrictedSubmitBlocks` is in `bypass_roles`; no staking check is performed.
3. If `skip_pow_verification = true`, the headers are accepted without PoW validation. `submit_block_header_inner` is called for each header.
4. If the adversarial fork accumulates more `chain_work` than the current tip, `reorg_chain` is triggered, overwriting `mainchain_tip_blockhash` and the height-to-hash mappings.
5. A downstream consumer calling `verify_transaction_inclusion` with a transaction in the adversarial chain receives `true`, falsely confirming a transaction that never occurred on the real Bitcoin chain.

            // Validate chain

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
