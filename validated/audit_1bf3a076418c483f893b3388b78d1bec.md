### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing the 64-Byte Merkle Forgery Invariant Enforced by `verify_transaction_inclusion_v2` - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` was introduced to enforce a coinbase-proof invariant that mitigates the known 64-byte transaction Merkle forgery attack. However, the original `verify_transaction_inclusion` function remains `pub` and callable by any unprivileged NEAR account, providing a direct bypass of that invariant. Any caller — including consumer contracts and bridges — can invoke the deprecated path and receive a `true` result for a forged transaction inclusion proof.

---

### Finding Description

The contract exposes two public entry points for SPV proof verification:

**`verify_transaction_inclusion` (deprecated, lines 288–323):**
```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    // ...
    require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
    merkle_tools::compute_root_from_merkle_proof(
        args.tx_id,
        usize::try_from(args.tx_index).unwrap(),
        &args.merkle_proof,
    ) == header.block_header.merkle_root
}
```

**`verify_transaction_inclusion_v2` (lines 347–369):**
```rust
#[pause]
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
    require!(
        merkle_tools::compute_root_from_merkle_proof(
            args.coinbase_tx_id.clone(), 0usize, &args.coinbase_merkle_proof,
        ) == header.block_header.merkle_root,
        "Incorrect coinbase merkle proof"
    );
    #[allow(deprecated)]
    self.verify_transaction_inclusion(args.into())
}
```

The v2 function adds a mandatory coinbase Merkle proof check — the invariant that prevents the 64-byte transaction forgery attack (documented at https://www.bitmex.com/blog/64-Byte-Transactions). The attack works because `compute_root_from_merkle_proof` treats its first argument as a leaf hash without verifying it is a real transaction; a 64-byte internal Merkle node can be crafted to satisfy the proof, returning `true` for a non-existent transaction.

The `#[deprecated]` attribute on `verify_transaction_inclusion` is a Rust compiler hint only — it emits a warning at the call site inside the contract but imposes **no runtime access restriction**. The function remains `pub` with only `#[pause]` gating (which only applies when the contract is administratively paused). Any NEAR account can call `verify_transaction_inclusion` directly, completely skipping the coinbase proof check.

This is structurally identical to M-10: a protective invariant check (`takerInvariant` / coinbase proof) guards the "correct" entry point (`closeMakeFor` / `verify_transaction_inclusion_v2`), but an alternative entry point (`closeAll` via liquidation / `verify_transaction_inclusion` directly) achieves the same outcome without the check.

---

### Impact Explanation

A consumer contract or bridge that calls `verify_transaction_inclusion` — or an attacker who directly invokes it — can obtain a `true` result for a Bitcoin transaction that does not exist. Concretely:

- An attacker constructs a 64-byte value that, when treated as a leaf in `compute_root_from_merkle_proof`, produces the block's actual `merkle_root` via a valid sibling path.
- The attacker calls `verify_transaction_inclusion` with this fake `tx_id` and the crafted proof.
- The function returns `true`.
- Any bridge, atomic swap, or cross-chain lending protocol that trusts this result releases funds or executes state changes based on a non-existent Bitcoin transaction.

---

### Likelihood Explanation

The entry point is fully public and requires no special role or privileged key. The 64-byte Merkle forgery technique is well-documented and has known construction methods. Any NEAR caller can trigger this path in a single transaction. Consumer contracts that integrated before v2 was introduced, or that call the deprecated function by mistake, are permanently exposed.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` or add an explicit access-control guard (e.g., `#[private]`) so it can only be called internally by `verify_transaction_inclusion_v2`. Alternatively, delete the deprecated function entirely and inline its logic into v2. A Rust `#[deprecated]` attribute alone is insufficient to prevent external callers from bypassing the coinbase-proof invariant.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature, `skip_pow_verification = false`).
2. Submit a real block header with a known `merkle_root` `R`.
3. Craft a 64-byte blob `F` such that `double_sha256(concat(F, sibling))` = `R` for some chosen `sibling` value (standard 64-byte Merkle forgery construction).
4. Call `verify_transaction_inclusion` with `tx_id = F`, `tx_index = 0`, `merkle_proof = [sibling]`, `tx_block_blockhash = <that block>`, `confirmations = 1`.
5. The call returns `true` despite `F` not being any real transaction in the block.
6. Calling `verify_transaction_inclusion_v2` with the same `tx_id` and no valid `coinbase_tx_id` would panic with `"Incorrect coinbase merkle proof"`, confirming the invariant is only enforced on the v2 path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
