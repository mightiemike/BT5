### Title
No Minimum Output Protection in `burnNlp` Allows Users to Receive Fewer Quote Tokens Than Expected - (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `burnNlp` function in `Clearinghouse.sol` converts a user's NLP tokens into quote tokens using a sequencer-supplied `oraclePriceX18`. The user-signed `BurnNlp` struct contains only `sender`, `nlpAmount`, and `nonce` — no `minQuoteAmount` field. Because the oracle price is appended by the sequencer at execution time and is not part of the user's signed commitment, the user has no on-chain mechanism to enforce a minimum quote output. This is a direct structural analog to the SecuritizeVault `liquidate()` slippage vulnerability.

---

### Finding Description

In `Clearinghouse.sol`, `burnNlp` computes the quote output as:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [1](#0-0) 

The `oraclePriceX18` is not part of the user-signed `BurnNlp` struct:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
}
``` [2](#0-1) 

Instead, `oraclePriceX18` is a sequencer-appended field in `SignedBurnNlp`:

```solidity
struct SignedBurnNlp {
    BurnNlp tx;
    bytes signature;
    int128 oraclePriceX18;
    int128[] nlpPoolRebalanceX18;
}
``` [3](#0-2) 

In `EndpointTx.sol`, the sequencer decodes the transaction and passes `signedTx.oraclePriceX18` directly to `clearinghouse.burnNlp(...)`:

```solidity
clearinghouse.burnNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [4](#0-3) 

The user's signature covers only `BurnNlp` (sender, nlpAmount, nonce). The oracle price used to compute the quote output is entirely outside the user's signed commitment and is set by the sequencer at execution time. There is no `minQuoteAmount` check anywhere in the execution path.

The same structural issue exists in `mintNlp`: the user signs a fixed `quoteAmount`, but the NLP tokens received are `quoteAmount.div(oraclePriceX18)`, with no `minNlpAmount` protection. [5](#0-4) 

---

### Impact Explanation

A user who submits a `BurnNlp` transaction expecting to receive `X` quote tokens for their NLP may receive significantly fewer tokens if the oracle price used at execution time is lower than the price at signing time. The NLP token balance is irrevocably debited (`-nlpAmount`) while the quote credit is determined solely by the sequencer-provided price. There is no revert path if the output falls below the user's expectation. This constitutes a direct, measurable asset loss: the user permanently loses NLP tokens and receives fewer quote tokens than they agreed to. [6](#0-5) 

---

### Likelihood Explanation

The NLP oracle price is dynamic. Transactions submitted to the sequencer are queued and processed with latency. Even with an honest sequencer, the price at execution time can differ materially from the price at signing time, particularly during volatile market conditions. The user has no on-chain recourse: the signed transaction contains no price bound, so the sequencer is free to execute at any oracle price it provides. This is a realistic, non-theoretical risk for any user interacting with the `BurnNlp` flow. [7](#0-6) 

---

### Recommendation

1. Add a `minQuoteAmount` field to the `BurnNlp` struct so it is part of the user's signed commitment:

```solidity
struct BurnNlp {
    bytes32 sender;
    uint128 nlpAmount;
    uint64 nonce;
    uint128 minQuoteAmount; // slippage protection
}
```

2. In `burnNlp`, enforce the minimum after computing `quoteAmount`:

```solidity
require(uint128(quoteAmount) >= txn.minQuoteAmount, "Insufficient output amount");
```

3. Apply the same fix to `mintNlp` by adding a `minNlpAmount` field to the `MintNlp` struct.

4. Consider adding a deadline/expiry field to both structs to protect against long-pending transactions executing at stale prices. [8](#0-7) 

---

### Proof of Concept

1. User signs a `BurnNlp` transaction: `{ sender: alice, nlpAmount: 1000e18, nonce: 5 }` when NLP oracle price is `$10`, expecting to receive `~9990 USDC` (after 0.1% burn fee).
2. The transaction is queued in the sequencer. NLP price drops to `$8` before execution.
3. Sequencer constructs `SignedBurnNlp` with `oraclePriceX18 = 8e18` and submits it on-chain.
4. `burnNlp` executes: `quoteAmount = 1000e18 * 8e18 / 1e18 = 8000e18`, minus burn fee `= ~7992 USDC`.
5. Alice's NLP balance is debited by `1000e18` and she receives only `~7992 USDC` instead of the expected `~9990 USDC` — a loss of approximately `~$1998` with no revert or protection.
6. No `minQuoteAmount` check exists anywhere in `burnNlp` to prevent this outcome. [9](#0-8)
