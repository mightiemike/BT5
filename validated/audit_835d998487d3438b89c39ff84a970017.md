### Title
Malicious sync peer injects arbitrary `n_transactions` via `transactions.n_leaves` with no cross-check, storing wrong transaction count and producing divergent `concat_counts` Felt — (`crates/apollo_protobuf/src/converters/header.rs`)

---

### Summary

The `TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` conversion blindly trusts `value.transactions.n_leaves` as `n_transactions` with no cross-check. The p2p sync client performs no cryptographic block-hash verification before committing the header to storage. A malicious sync peer can send `transactions.n_leaves=999` for a block with 0 actual transactions; the value passes conversion, is written to the DB, and is later used by `concat_counts` to produce a divergent `concatenated_counts` Felt that is baked into every downstream block-hash recomputation.

---

### Finding Description

**Step 1 — Unchecked conversion.**
`n_transactions` is taken verbatim from the protobuf field:

```rust
// crates/apollo_protobuf/src/converters/header.rs  lines 85-91
let n_transactions = value
    .transactions
    .as_ref()
    .ok_or(missing("SignedBlockHeader::transactions"))?
    .n_leaves          // ← attacker-controlled
    .try_into()
    .expect("Failed converting u64 to usize");
```

No guard compares this value against an actual transaction list, a Merkle-tree leaf count, or the signed block hash. [1](#0-0) 

**Step 2 — No block-hash verification in the p2p sync client.**
`parse_data_for_block` checks only block-number ordering and signature-vector length (== 1). There is no call to `verify_block_signature` anywhere in `crates/apollo_p2p_sync/**` (confirmed by grep). The attacker-supplied `block_hash` field and all header fields are accepted as-is. [2](#0-1) 

**Step 3 — Direct storage write.**
`write_to_storage` calls `append_header` with the peer-supplied `block_header` (including the poisoned `n_transactions`) and commits immediately. [3](#0-2) 

**Step 4 — `concat_counts` uses the stored `n_transactions`.**
Whenever `Option<BlockHeaderCommitments>` is derived from a stored `BlockHeader`, `concat_counts` is called with `block_header.n_transactions`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 160-165
concatenated_counts: concat_counts(
    block_header.n_transactions,   // ← poisoned value
    block_header.n_events,
    state_diff_length,
    block_header.block_header_without_hash.l1_da_mode,
),
```

`concat_counts` packs `transaction_count` into the top 64 bits of a Felt. With `n_transactions=999` instead of 0, the resulting Felt is `0x00000000000003E7…` instead of `0x0000000000000000…`. [4](#0-3) [5](#0-4) 

**Step 5 — Divergent block hash.**
`calculate_block_hash` hashes `header_commitments.concatenated_counts` (the `packed_lengths` field) as a direct input. A wrong `concatenated_counts` produces a wrong block hash for every downstream recomputation. [6](#0-5) 

---

### Impact Explanation

| Artifact | Effect |
|---|---|
| `StorageBlockHeader.n_transactions` | Stores attacker-chosen value (e.g. 999) |
| RPC `starknet_getBlockWithTxHashes` / `starknet_getBlockWithTxs` | Serves wrong transaction count |
| `BlockHeaderCommitments.concatenated_counts` | Wrong Felt; bits 0-63 encode 999 instead of true count |
| Recomputed block hash | Diverges from the on-chain hash; any verifier that recomputes the hash from stored fields will disagree with L1 |
| `n_events` (same path via `events.n_leaves`) | Same unchecked copy; event count is equally poisonable |

The p2p sync client also stores the peer-supplied `block_hash` field without verifying it against the computed hash, so the attacker can simultaneously inject a fake `block_hash` that the RPC will serve authoritatively. [7](#0-6) 

---

### Likelihood Explanation

Any node that syncs headers over p2p is exposed. The attacker only needs to be a reachable sync peer — no cryptographic key material, no operator privilege. The `parse_data_for_block` path is the normal production sync path and is exercised for every block. The existing TODO comment in the codebase acknowledges the missing cross-check:

```rust
// TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
// concatenated_counts.
``` [8](#0-7) 

---

### Recommendation

1. **Verify the block hash before storing.** Recompute `BlockHeaderCommitments` from the received fields and verify that `calculate_block_hash(…)` matches the peer-supplied `block_hash`. Reject the peer on mismatch.
2. **Verify the block signature.** Call `verify_block_signature` (or the consensus equivalent) against a trusted sequencer public key before committing any header.
3. **Cross-check `n_transactions` / `n_events` against `concatenated_counts`.** If the peer also supplies a `concatenated_counts` (or it can be derived from the signed block hash), extract the transaction count from bits 0-63 and assert it equals `n_leaves`.

### Proof of Concept

```rust
// Pseudo-test (production path, no mocks needed for the conversion itself)
use apollo_protobuf::protobuf;
use apollo_protobuf::sync::SignedBlockHeader;

let proto = protobuf::SignedBlockHeader {
    block_hash:    Some(/* any valid felt */ ...),
    parent_hash:   Some(/* any valid felt */ ...),
    number:        0,
    time:          0,
    sequencer_address: Some(/* any */ ...),
    state_root:    Some(/* any */ ...),
    transactions:  Some(protobuf::Patricia { n_leaves: 999, root: None }),
    events:        Some(protobuf::Patricia { n_leaves: 0,   root: None }),
    state_diff_commitment: Some(protobuf::StateDiffCommitment {
        state_diff_length: 0, root: None,
    }),
    protocol_version: "0.13.4".to_string(),
    // ... fill required gas price fields ...
    signatures: vec![/* one dummy signature */],
    ..Default::default()
};

let header: SignedBlockHeader = proto.try_into().unwrap();
// Passes conversion with zero actual transactions:
assert_eq!(header.block_header.n_transactions, 999);

// concat_counts now encodes 999 in bits 0-63:
use starknet_api::block_hash::block_hash_calculator::concat_counts;
use starknet_api::data_availability::L1DataAvailabilityMode;
let cc = concat_counts(999, 0, 0, L1DataAvailabilityMode::Calldata);
// cc != concat_counts(0, 0, 0, L1DataAvailabilityMode::Calldata)
assert_ne!(cc, concat_counts(0, 0, 0, L1DataAvailabilityMode::Calldata));
```

```rust
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let l1_data_availability_byte: u8 = match l1_data_availability_mode {
        L1DataAvailabilityMode::Calldata => 0,
        L1DataAvailabilityMode::Blob => 0b10000000,
    };
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),
        &[l1_data_availability_byte],
        &[0_u8; 7], // zero padding
    ]
    .concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}
