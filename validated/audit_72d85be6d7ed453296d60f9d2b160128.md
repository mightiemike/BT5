### Title
`SwapAllowlistExtension.beforeSwap` gates the router's address instead of the actual swapper's identity, enabling any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` from the pool's perspective — the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the actual user's address. If the pool admin allowlists the router to enable router-mediated swaps, the allowlist is completely bypassed: any user, regardless of allowlist status, can swap on the curated pool by calling the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and uses it as the identity to check against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

`sender` is populated by `MetricOmmPool.swap()` as `msg.sender` — the direct caller of the pool:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    ...
);
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
``` [3](#0-2) 

So `msg.sender` inside `pool.swap()` is the **router address**, not the user. The extension therefore checks `allowedSwapper[pool][router]` — not the actual user's address.

Contrast this with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly gates the `owner` (the economic beneficiary) rather than `sender` (the operator/payer), making it bypass-resistant through `MetricOmmPoolLiquidityAdder`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [4](#0-3) 

The swap extension has no equivalent "economic actor" field to check — it only sees `sender` (the direct pool caller), which is the router in router-mediated flows.

---

### Impact Explanation

Two concrete impacts arise:

**Impact 1 — Allowlist bypass (High):** If the pool admin allowlists the router address to enable router-mediated swaps for their curated users, any user (including non-allowlisted ones) can bypass the restriction by calling `MetricOmmSimpleRouter.exactInputSingle/exactInput/exactOutputSingle/exactOutput`. The router is a public, permissionless contract. The extension sees `allowedSwapper[pool][router] == true` and passes, regardless of who the actual user is. The router then pulls tokens from the actual user via the swap callback. The curated pool's access control is completely defeated.

**Impact 2 — Broken core functionality (Medium):** If the pool admin does not allowlist the router (to avoid the bypass above), then allowlisted users cannot use the router at all — their swaps revert with `NotAllowedToSwap` because the router's address is not in the allowlist. This makes the router unusable for any curated pool with a swap allowlist, breaking a core supported periphery flow.

Both impacts are contest-relevant: the first is a direct policy bypass on a curated pool; the second is broken core pool functionality for a supported entrypoint.

---

### Likelihood Explanation

The bypass is reachable by any user who knows the pool uses a swap allowlist. The router is a public, deployed contract. The pool admin is semi-trusted and is expected to configure the allowlist correctly; the flaw is structural — there is no correct configuration that simultaneously (a) allows allowlisted users to use the router and (b) blocks non-allowlisted users from using the router. The pool admin cannot fix this through allowlist configuration alone.

---

### Recommendation

The `SwapAllowlistExtension` should gate the actual user rather than the direct pool caller. Two approaches:

1. **Preferred — check `recipient` as a proxy for the user:** For most swap flows, `recipient` is the user's address. The extension could check `allowedSwapper[pool][recipient]` instead of `allowedSwapper[pool][sender]`. This is consistent with how the deposit allowlist gates `owner`.

2. **Alternative — require direct pool calls for allowlisted pools:** Document that pools using `SwapAllowlistExtension` must not allowlist the router, and that allowlisted users must call the pool directly. This is a weaker mitigation and breaks the router UX.

The deeper fix is to pass the originating user's address through the extension data or as a dedicated field in the `beforeSwap` hook signature, analogous to how `addLiquidity` separates `sender` (operator) from `owner` (economic actor).

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin allowlists alice (EOA) and the router address
    (to allow alice to swap through the router)
  - Pool admin does NOT allowlist bob (EOA)

Attack:
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool: curated_pool,
       recipient: bob,
       ...
     })
  2. Router calls pool.swap(bob, ...) — msg.sender = router
  3. Pool calls _beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[pool][router] == true → passes
  5. Swap executes; router pulls tokens from bob via callback
  6. Bob successfully swaps on a pool he is not allowlisted for

Result:
  allowedSwapper[pool][bob] == false, yet bob's swap succeeds.
  The SwapAllowlistExtension guard is bypassed.
``` [5](#0-4) [6](#0-5) [2](#0-1)
