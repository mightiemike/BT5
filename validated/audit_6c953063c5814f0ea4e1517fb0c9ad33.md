### Title
Stale Sender Captured in `ordersInfo` Before Isolated Subaccount Redirect Causes Balance Updates to Wrong Account - (File: core/contracts/OffchainExchange.sol)

### Summary
In `OffchainExchange.sol::matchOrders`, the `ordersInfo.taker.sender` and `ordersInfo.maker.sender` fields are populated from `taker.order.sender` / `maker.order.sender` **before** those fields are overwritten with the resolved isolated subaccount from `digestToSubaccount`. Any downstream logic that uses `ordersInfo.taker.sender` / `ordersInfo.maker.sender` for balance updates, fee charging, or accounting will operate on the original (parent) subaccount rather than the intended isolated subaccount, breaking the isolation invariant.

### Finding Description
In `matchOrders` (lines 631–720 of `OffchainExchange.sol`), the execution sequence is:

1. **Lines 653–672**: `ordersInfo` is constructed, capturing `sender: taker.order.sender` and `sender: maker.order.sender` into `ordersInfo.taker.sender` and `ordersInfo.maker.sender` respectively.

2. **Lines 673–678**: Immediately after, `taker.order.sender` and `maker.order.sender` are conditionally overwritten with the isolated subaccount resolved from `digestToSubaccount`:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
    maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
}
```

Because `ordersInfo.taker.sender` was already assigned by value at step 1, it is **not** updated by the reassignment at step 2. It permanently holds the original (parent) subaccount address. The `taker.order.sender` / `maker.order.sender` fields now correctly point to the isolated subaccount, but `ordersInfo.taker.sender` / `ordersInfo.maker.sender` remain stale.

This is the direct analog to the SakeVault `openPosition` bug: a value is captured into a tracking struct, then the source variable is reassigned, leaving the struct field holding the pre-redirect (wrong) identity for all subsequent operations. [1](#0-0) 

### Impact Explanation
Any balance update, fee deduction, or accounting operation that consumes `ordersInfo.taker.sender` / `ordersInfo.maker.sender` will credit or debit the **parent** subaccount instead of the **isolated** subaccount. This:

- Bypasses the isolated subaccount's balance tracking entirely, allowing the isolated position to accumulate losses or gains that are silently absorbed by the parent.
- Breaks the core isolation invariant: the isolated subaccount's collateral is supposed to be the only capital at risk for that position.
- Enables a trader to open an isolated position whose P&L flows to the parent account, effectively defeating margin isolation and potentially allowing undercollateralized exposure.

### Likelihood Explanation
Any user who has a valid `digestToSubaccount` mapping entry (i.e., has previously called `createIsolatedSubaccount` and received a digest-to-isolated-subaccount mapping) and then submits a `MatchOrdersWithSigner` transaction will trigger this path. This is a normal, unprivileged trading flow. No special privileges, admin access, or key compromise is required. [2](#0-1) 

### Recommendation
Move the `ordersInfo` construction **after** the isolated subaccount redirect, so that `ordersInfo.taker.sender` and `ordersInfo.maker.sender` capture the final, resolved sender values:

```solidity
// First resolve isolated subaccount redirects
if (digestToSubaccount[getDigest(callState.productId, taker.order)] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[getDigest(callState.productId, taker.order)];
}
if (digestToSubaccount[getDigest(callState.productId, maker.order)] != bytes32(0)) {
    maker.order.sender = digestToSubaccount[getDigest(callState.productId, maker.order)];
}

// Then construct ordersInfo with the resolved senders
ordersInfo = OrdersInfo(
    OrderInfo({
        digest: getDigest(callState.productId, taker.order),
        sender: taker.order.sender,  // now correctly the isolated subaccount
        ...
    }),
    ...
);
```

Alternatively, update `ordersInfo.taker.sender` and `ordersInfo.maker.sender` explicitly after the redirect assignments.

### Proof of Concept

1. Alice calls `createIsolatedSubaccount` for product X, producing `isolatedSubaccount_A` and a `digest_A` → `isolatedSubaccount_A` mapping stored in `digestToSubaccount`.
2. Alice submits a `MatchOrdersWithSigner` transaction where `taker.order.sender = alice_parent` and the order digest equals `digest_A`.
3. Inside `matchOrders`:
   - Line 656: `ordersInfo.taker.sender = alice_parent` (stale capture).
   - Line 674: `taker.order.sender = isolatedSubaccount_A` (correct redirect).
4. `_validateOrder` runs against `isolatedSubaccount_A` (correct).
5. Balance/fee updates consume `ordersInfo.taker.sender = alice_parent` (wrong account).
6. `isolatedSubaccount_A`'s balance is never touched; `alice_parent`'s balance absorbs all P&L, defeating isolation. [3](#0-2)

```text
    function matchOrders(IEndpoint.MatchOrdersWithSigner calldata txn)
        external
        onlyEndpoint
    {
        CallState memory callState = _getCallState(txn.matchOrders.productId);

        OrdersInfo memory ordersInfo;

        MarketInfo memory market = getMarketInfo(callState.productId);
        IEndpoint.SignedOrder memory taker = txn.matchOrders.taker;
        IEndpoint.SignedOrder memory maker = txn.matchOrders.maker;

        // isolated subaccounts cannot be used as sender
        require(
            !RiskHelper.isIsolatedSubaccount(taker.order.sender),
            ERR_INVALID_TAKER
        );
        require(
            !RiskHelper.isIsolatedSubaccount(maker.order.sender),
            ERR_INVALID_MAKER
        );

        ordersInfo = OrdersInfo(
            OrderInfo({
                digest: getDigest(callState.productId, taker.order),
                sender: taker.order.sender,
                amount: taker.order.amount,
                fee: 0,
                builderFee: 0,
                quoteDelta: 0,
                amountDelta: 0
            }),
            OrderInfo({
                digest: getDigest(callState.productId, maker.order),
                sender: maker.order.sender,
                amount: maker.order.amount,
                fee: 0,
                builderFee: 0,
                quoteDelta: 0,
                amountDelta: 0
            })
        );
        if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
            taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
        }
        if (digestToSubaccount[ordersInfo.maker.digest] != bytes32(0)) {
            maker.order.sender = digestToSubaccount[ordersInfo.maker.digest];
        }

        require(
            _validateOrder(
                callState,
                market,
                taker,
                ordersInfo.taker.digest,
                true,
                txn.takerLinkedSigner
            ),
            ERR_INVALID_TAKER
        );
        require(
            _validateOrder(
                callState,
                market,
                maker,
                ordersInfo.maker.digest,
                false,
                txn.makerLinkedSigner
            ),
            ERR_INVALID_MAKER
```
