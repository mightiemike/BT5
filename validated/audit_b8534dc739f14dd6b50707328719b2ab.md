### Title
Permanent NFT Metadata Lock After Transfer Due to Conjunctive `IsOwner` + `IsDenomCreator` Guard in `EditNFT` — (`x/nft/keeper/keeper.go`)

### Summary

`EditNFT` requires the caller to simultaneously satisfy both `IsOwner` and `IsDenomCreator` using the **same address**. Once an NFT is transferred away from the denom creator, no address can ever satisfy both conditions at once, permanently locking the NFT's metadata.

### Finding Description

`EditNFT` in `x/nft/keeper/keeper.go` applies two sequential authorization checks against the single `owner` parameter: [1](#0-0) 

`IsOwner` checks that `owner` equals `nft.Owner` in the KV store: [2](#0-1) 

`IsDenomCreator` checks that `owner` equals `denom.Creator`, which is set at `IssueDenom` time and **never changes**: [3](#0-2) 

`TransferOwner` (called by `MsgTransferNFT`) updates `nft.Owner` to the recipient but leaves `denom.Creator` unchanged: [4](#0-3) 

After transfer:
- **Victim** (`new owner`): `IsOwner` passes, `IsDenomCreator` fails → `ErrUnauthorized`
- **Denom creator** (`old owner`): `IsOwner` fails → `ErrUnauthorized`

No address can satisfy both guards simultaneously. The NFT metadata is permanently frozen.

### Impact Explanation

Any NFT transferred away from its denom creator becomes permanently uneditable. This breaks the expected NFT lifecycle for any use case involving dynamic metadata (e.g., game items, certificates, tiered reward positions referenced in `x/tieredrewards/keeper/position_state.go`). The state change is irreversible — there is no admin override, no governance path, and no migration hook that re-enables editing. [5](#0-4) 

### Likelihood Explanation

This is triggered by the normal, expected operation of minting an NFT to a recipient other than the denom creator, or transferring a minted NFT. `MintNFT` already supports minting to an arbitrary `recipient` distinct from `sender`: [6](#0-5) 

Any such mint immediately produces an NFT that can never be edited. No special attacker privileges are required — any user who receives an NFT via `MsgTransferNFT` or `MsgMintNFT` (with `recipient != sender`) triggers this state.

### Recommendation

Decouple the two authorization checks. `EditNFT` should allow editing if the caller is **either** the current owner **or** the denom creator (or both), not require both simultaneously. Alternatively, restrict editing to the denom creator only (removing the `IsOwner` check), consistent with how `MintNFT` is gated solely on `IsDenomCreator`: [7](#0-6) 

The same conjunctive flaw exists in `BurnNFT`: [8](#0-7) 

Both should be reviewed and fixed together.

### Proof of Concept

```
1. Alice calls MsgIssueDenom(id="testdenom", sender=Alice)
   → denom.Creator = Alice

2. Alice calls MsgMintNFT(denomId="testdenom", id="token1", sender=Alice, recipient=Bob)
   → nft.Owner = Bob

3. Bob calls MsgEditNFT(denomId="testdenom", id="token1", sender=Bob)
   → IsOwner(Bob) ✓, IsDenomCreator(Bob) ✗ → ErrUnauthorized

4. Alice calls MsgEditNFT(denomId="testdenom", id="token1", sender=Alice)
   → IsOwner(Alice) ✗ → ErrUnauthorized

Result: token1 metadata is permanently locked. No party can edit it.
```

```go
func (k Keeper) EditNFT(
	ctx sdk.Context, denomID, tokenID, tokenNm,
	tokenURI, tokenData string, owner sdk.AccAddress,
) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	if types.Modified(tokenNm) {
		nft.Name = tokenNm
	}

	if types.Modified(tokenURI) {
		nft.URI = tokenURI
	}

	if types.Modified(tokenData) {
		nft.Data = tokenData
	}

	k.setNFT(ctx, denomID, nft)

	return nil
}
```
