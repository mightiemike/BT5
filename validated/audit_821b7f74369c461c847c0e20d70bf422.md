### Title
Phantom-Position Merkle Proof Accepted by `verify_transaction_inclusion_v2` — (`contract/src/lib.rs`, `merkle-tools/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion_v2` returns `true` for a real transaction hash (`tx_id = T2`) submitted at a phantom index (`tx_index = N`, the duplicate padding slot) in any odd-length block. The coinbase depth-equality guard does not prevent this because the phantom proof has the same depth as the real proof and independently hashes to the correct merkle root.

---

### Finding Description

**Relevant code paths:**

`verify_transaction_inclusion_v2` performs three checks:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — same depth
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, coinbase_merkle_proof) == merkle_root` — coinbase anchors depth
3. Delegates to `verify_transaction_inclusion`, which checks `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == merkle_root` [1](#0-0) 

Neither function validates that `tx_index` is within the range of actual transactions in the block. The contract stores only the merkle root in the header — no transaction count. [2](#0-1) 

`compute_root_from_merkle_proof` is a pure positional hash-chain computation with no bounds check: [3](#0-2) 

**Why the phantom proof hashes to the correct root:**

For a 3-tx block `[T0, T1, T2]`, Bitcoin's merkle tree duplicates the last leaf:

```
Padded leaves:  [T0,  T1,  T2,  T2]   (indices 0–3)
Level 1:        [H(T0,T1),  H(T2,T2)]
Root:           H(H(T0,T1), H(T2,T2))
```

Proof for **T2 at real index 2** = `[T2, H(T0,T1)]`
- pos 2 even → `H(T2, T2)`; pos 1 odd → `H(H(T0,T1), H(T2,T2))` = root ✓

Proof for **T2 at phantom index 3** = `[T2, H(T0,T1)]` (identical proof)
- pos 3 odd → `compute_hash(T2, T2)` = `H(T2,T2)`; pos 1 odd → `H(H(T0,T1), H(T2,T2))` = root ✓ [4](#0-3) 

The proof for the last real transaction at index `N-1` is **identical** to the proof for the phantom slot at index `N` in any odd-length tree, because the duplication step makes both positions hash to the same parent node `H(T2,T2)`.

The coinbase depth guard (`merkle_proof.len() == coinbase_merkle_proof.len()`) is satisfied because both proofs have depth 2. The coinbase proof correctly anchors the tree to depth 2, but this does not prevent the phantom index from also resolving to the root. [5](#0-4) 

---

### Impact Explanation

`verify_transaction_inclusion_v2` returns `true` for `(tx_id=T2, tx_index=3)` in a 3-tx block, where index 3 is a phantom padding slot — not a real transaction position. Any downstream system that calls this function to gate a state transition (e.g., token minting, bridge unlock, cross-chain message relay) will accept a proof for a phantom position as valid inclusion evidence. This is a **wrong-index** false positive, matching the critical scope target.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of any odd-length Bitcoin block (trivially public)
- The last real transaction hash in that block (trivially public from any block explorer)
- The real coinbase proof for that block (trivially derivable from the block's transaction list)

No privileged role, no relayer compromise, no key leak. Any unprivileged NEAR account can call `verify_transaction_inclusion_v2` directly.

---

### Recommendation

The contract stores only the merkle root per block header — it has no transaction count. The fix requires one of:

1. **Store `tx_count` in the block header** at submission time and add `require!(tx_index < tx_count)` in `verify_transaction_inclusion`.
2. **Require `tx_index != coinbase_index_of_sibling`** — more complex and fragile.
3. **Require that `tx_id != coinbase_tx_id` when `tx_index != 0`** — partial mitigation only.

Option 1 is the correct fix. The transaction count must be submitted alongside the block header and stored in the `headers_pool` entry so that `verify_transaction_inclusion` can bound-check the claimed index.

### Proof of Concept

For a 3-tx block `[T0, T1, T2]` with merkle root `R = H(H(T0,T1), H(T2,T2))`:

```
coinbase_tx_id          = T0
coinbase_merkle_proof   = [T1, H(T2,T2)]   // depth 2, resolves to R ✓

tx_id                   = T2               // real tx, but submitted at phantom index
tx_index                = 3               // phantom slot (does not exist in block)
merkle_proof            = [T2, H(T0,T1)]  // depth 2, resolves to R ✓
```

Step-by-step through `compute_root_from_merkle_proof(T2, 3, [T2, H(T0,T1)])`:

| Step | position | branch | result |
|------|----------|--------|--------|
| 1 | 3 (odd) | `H(proof=T2, current=T2)` | `H(T2,T2)` |
| 2 | 1 (odd) | `H(proof=H(T0,T1), current=H(T2,T2))` | `R` |

Both proofs have length 2 → length check passes.
Coinbase proof resolves to `R` → coinbase check passes.
Tx proof resolves to `R` → `verify_transaction_inclusion` returns `true`.

`verify_transaction_inclusion_v2` returns **`true`** for a transaction at a phantom position that does not exist in the block. [6](#0-5) [3](#0-2) [1](#0-0)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
