### Title
`BaseEngine._initialize()` Uses `initializer` Instead of `onlyInitializing`, Leaving Concrete Engine `initialize()` Functions Unprotected — (`File: core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine` is an `abstract` contract inherited by `SpotEngine` and `PerpEngine`. Its internal `_initialize()` function carries the `initializer` modifier — the wrong modifier for a base contract's init function. The concrete `initialize()` functions in `SpotEngine` and `PerpEngine` carry **no** `initializer` modifier at all, and neither contract has a `_disableInitializers()` constructor guard. This is the direct analog of H-03: the wrong modifier is placed on the base contract's init function, and the concrete contracts are left without the standard OZ initialization guard.

---

### Finding Description

`BaseEngine._initialize()` is declared `internal initializer`: [1](#0-0) 

The concrete `SpotEngine.initialize()` calls `_initialize()` but carries **no** `initializer` modifier: [2](#0-1) 

`PerpEngine.initialize()` is identical — no `initializer` modifier, just a bare call to `_initialize()`: [3](#0-2) 

Neither `SpotEngine` nor `PerpEngine` has a constructor calling `_disableInitializers()`, unlike `Clearinghouse`, `Verifier`, and `ContractOwner` which all do: [4](#0-3) 

The correct OZ pattern for a base contract's init function is `onlyInitializing` (not `initializer`), with `initializer` placed on the concrete contract's public `initialize()`. Because `_initialize` uses `initializer` instead of `onlyInitializing`, if `initializer` were ever correctly added to `SpotEngine.initialize()` or `PerpEngine.initialize()`, the call chain would revert — exactly the H-03 failure mode. In the current state, the protection is inverted: the base function is guarded but the public entry point is not.

---

### Impact Explanation

Because `SpotEngine.initialize()` and `PerpEngine.initialize()` carry no `initializer` modifier and no `_disableInitializers()` constructor guard exists:

1. **Implementation contract takeover**: An attacker can call `SpotEngine.initialize()` or `PerpEngine.initialize()` directly on the implementation contract (before or after proxy deployment), passing themselves as `_admin`. `_initialize` sets `__Ownable_init()` and `transferOwnership(_admin)`, making the attacker the owner of the implementation.

2. **Owner-gated functions exposed**: With ownership of the engine implementation, the attacker can call `updateRisk()` and `addOrUpdateProduct()` (both `onlyOwner`) on the implementation contract. In a delegatecall architecture (as used by `Endpoint._delegatecallEndpointTx()`), corrupted implementation state can propagate into proxy execution context.

3. **Broken upgrade safety**: The `initializer` on `_initialize` means the standard fix (adding `initializer` to the concrete `initialize()`) would cause a revert due to nested `initializer` calls — the exact H-03 failure mode — blocking any future safe re-initialization path.

---

### Likelihood Explanation

`SpotEngine` and `PerpEngine` are deployed as upgradeable proxies. The implementation contracts are publicly accessible on-chain. Any unprivileged caller can invoke `SpotEngine.initialize()` or `PerpEngine.initialize()` on the bare implementation address at any time. No special privilege, leaked key, or social engineering is required. The attack window is open from the moment the implementation is deployed until the proxy's `initialize()` is called.

---

### Recommendation

1. Change `BaseEngine._initialize()` from `initializer` to `onlyInitializing`:

```solidity
function _initialize(
    address _clearinghouseAddr,
    address _offchainExchangeAddr,
    address _endpointAddr,
    address _admin
) internal onlyInitializing {
    ...
}
```

2. Add `initializer` to the concrete `initialize()` functions in `SpotEngine` and `PerpEngine`:

```solidity
function initialize(...) external initializer {
    _initialize(...);
    ...
}
```

3. Add `_disableInitializers()` constructors to `SpotEngine` and `PerpEngine`, matching the pattern already used in `Clearinghouse`, `Verifier`, and `ContractOwner`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

---

### Proof of Concept

1. Deploy `SpotEngine` implementation contract (no proxy).
2. Call `SpotEngine.initialize(clearinghouse, offchainExchange, quote, endpoint, attacker)` directly on the implementation.
3. `_initialize` runs with `initializer` modifier — succeeds, sets `_initialized = true`, calls `transferOwnership(attacker)`.
4. Attacker is now owner of the implementation contract.
5. Attacker calls `updateRisk(productId, maliciousRiskStore)` on the implementation — accepted because `onlyOwner` passes.
6. Any subsequent `delegatecall` from `Endpoint` into this implementation executes in the proxy's storage context with attacker-controlled logic paths. [5](#0-4) [6](#0-5) [3](#0-2) [7](#0-6)
