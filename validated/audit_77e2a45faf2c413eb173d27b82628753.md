Audit Report

## Title
Router-mediated swaps pass the router address as `sender` to `SwapAllowlistExtension`, breaking per-user allowlist enforcement — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract address, not the end user. This makes per-user allowlist enforcement impossible for router-mediated swaps: allowlisting the router grants every user access, while not allowlisting it blocks all legitimate router users.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap` (metric-core/contracts/MetricOmmPool.sol L230–240). Every router entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap()` directly without forwarding the original caller — the pool's `msg.sender` is always the router address. `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. If the router is allowlisted (required for legitimate users to use it), any unprivileged user can call `router.exactInputSingle(...)` and pass the check, because the check resolves to `allowedSwapper[pool][router] == true`. The existing guard is structurally insufficient: it checks the correct mapping key format but receives the wrong address as input.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific addresses (e.g., KYC-verified counterparties, institutional participants) cannot enforce that restriction when users route through `MetricOmmSimpleRouter`. If the router is allowlisted, any unprivileged user can trade on the curated pool by routing through it, completely bypassing the per-user allowlist. This is a direct admin-boundary break: the pool admin's configured access control is bypassed by an unprivileged path through the supported periphery router.

## Likelihood Explanation
The router is the primary supported public swap entrypoint. No special privileges, flash loans, or multi-transaction sequences are required — a single `exactInputSingle` call suffices. Any user who discovers the router is allowlisted on a curated pool can immediately exploit this.

## Recommendation
The `sender` passed to extension hooks should represent the originating user, not the immediate caller of `pool.swap()`. Options:
1. **Pass the original user through the router:** Add a `payer`/`originator` field to the pool's `swap` interface (or a separate hook argument) that the router populates with `msg.sender` before calling the pool. The extension then checks this field.
2. **Router-aware extension variant:** Deploy a `SwapAllowlistExtension` variant that decodes the true originator from `extensionData` (forwarded by the router), and require the router to include `msg.sender` in `extensionData`.
3. **Document and enforce:** If the design intent is that the router must never be allowlisted, document this explicitly and provide a router-aware extension as the supported path for curated pools.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][alice] = true   (alice is the only allowed user)
  - allowedSwapper[pool][router] = true  (router must be allowlisted for alice to use it)

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  - Router calls pool.swap(...) — pool's msg.sender = router
  - pool passes msg.sender (router) as sender to _beforeSwap
  - beforeSwap receives sender = router
  - Check: allowedSwapper[pool][router] == true → PASSES
  - Bob's swap executes on the allowlisted pool

Contrast (direct call):
  - bob calls pool.swap() directly — msg.sender = bob
  - beforeSwap receives sender = bob
  - Check: allowedSwapper[pool][bob] == false → REVERTS (correctly)
```
