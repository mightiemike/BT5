Audit Report

## Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Oracle Mid Price, Miscalibrating the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean of bid and ask at line 218, while the protocol's canonical oracle mid price is the geometric mean, as defined by `SwapMath.midAndSpreadFeeX64FromBidAsk` and used by the sibling `PriceVelocityGuardExtension`. Because AM ≥ GM for all positive bid/ask pairs, the stop-loss always uses an inflated mid price whenever the oracle spread is non-zero, causing watermarks to be set at wrong levels and making the stop-loss guard for token1 outflow systematically less sensitive than the configured `drawdownE6` threshold.

## Finding Description

**Root cause — line 218:**

```solidity
// OracleValueStopLossExtension.sol:218
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
```

The protocol's canonical oracle mid price is the geometric mean, defined in `SwapMath.midAndSpreadFeeX64FromBidAsk`:

```solidity
// SwapMath.sol:70
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);  // geometric mean
```

The sibling extension `PriceVelocityGuardExtension` correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk` and even comments *"geometric mid of two uint128 bid/ask quotes"*:

```solidity
// PriceVelocityGuardExtension.sol:48-49
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
// casting to `uint128` is safe: geometric mid of two uint128 bid/ask quotes fits uint128 (same bound as pool)
```

**Exploit flow:**

1. Any swap through a pool with `OracleValueStopLossExtension` attached calls `afterSwap`, which calls `_afterSwapOracleStopLoss`.
2. At line 218, `midPriceX64` is computed as `(bid + ask) / 2` instead of `sqrt(bid * ask)`. By AM-GM, this value is strictly greater than the geometric mean whenever `bid ≠ ask`.
3. This inflated `midPriceX64` is passed to `_metrics` (lines 254–255):
   - `metricT0 = t0ps + (t1 * Q64 / midPriceX64) / shares` — t1 component divided by inflated mid → **deflated**
   - `metricT1 = (t0 * midPriceX64 / Q64) / shares + t1ps` — t0 component multiplied by inflated mid → **inflated**
4. `_checkAndUpdateWatermarks` (lines 270–284) sets `hwm0` to the deflated `metricT0` and `hwm1` to the inflated `metricT1`.
5. The drawdown floor for token1 outflow is `hwm0 * floorMultiplier / E6`, which is lower than it should be. The stop-loss for `zeroForOne == true` fires later than the configured `drawdownE6` threshold.
6. The drawdown floor for token0 outflow is `hwm1 * floorMultiplier / E6`, which is higher than it should be. The stop-loss for `zeroForOne == false` fires earlier than configured.

**No existing guard compensates for this.** The `_validateDrawdown` check only enforces `drawdownE6 <= E6`; it does not correct the mid price formula. The miscalibration is permanent and accumulates across every watermark update.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value drain. Its stated invariant is that value per share at oracle marks cannot fall faster than `drawdown + decay × t`. With the arithmetic mean, the effective drawdown threshold for token1 outflow is approximately `drawdownE6 + s² · E6`, where `s` is the oracle spread fraction. For a 5% configured drawdown and a 10% oracle spread, the guard fires ~0.25% later than intended. For a $10M pool, this represents ~$25,000 of additional LP principal that can drain before the stop-loss triggers. The discrepancy scales with `s²` and grows during high-volatility periods when oracle spreads widen — exactly when stop-loss protection is most critical. This constitutes a direct, quantifiable loss of LP principal above Sherlock thresholds.

## Likelihood Explanation

The bug is active on every swap through any pool with this extension attached, whenever `bidPriceX64 ≠ askPriceX64` (i.e., any non-zero oracle spread, which is the normal operating condition). No special attacker action is required. Any unprivileged trader executing an ordinary swap will trigger the miscalibrated watermark update. The miscalibration is permanent and compounds across every watermark update.

## Recommendation

Replace the arithmetic mean with `SwapMath.midAndSpreadFeeX64FromBidAsk`, consistent with the protocol's canonical oracle mid price and with `PriceVelocityGuardExtension`:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

This requires importing `SwapMath` into `OracleValueStopLossExtension`, which is already done in `PriceVelocityGuardExtension`.

## Proof of Concept

**Setup:** Pool with 1,000 token0 and 1,000 token1 per bin, 1,000 shares. Oracle: `bid = 0.9·Q64`, `ask = 1.1·Q64` (10% spread). Configured drawdown = 5% (`drawdownE6 = 50_000`).

**Step 1 — First swap sets watermarks:**
```
AM mid  = (0.9 + 1.1)/2 · Q64 = 1.0 · Q64
GM mid  = sqrt(0.9 · 1.1) · Q64 ≈ 0.9950 · Q64

metricT0 (AM) = 1000 + 1000·Q64/(1.0·Q64) = 2000   ← set too low
metricT0 (GM) = 1000 + 1000·Q64/(0.995·Q64) ≈ 2005  ← correct

hwm0 (AM) = 2000
hwm0 (GM) = 2005
```

**Step 2 — Token1 drains; stop-loss check:**
```
floor (AM) = 2000 · 0.95 = 1900
floor (GM) = 2005 · 0.95 = 1904.75
```

A `metricT0` of 1902 would trigger the GM-based guard but **not** the AM-based guard. The pool continues to allow token1 outflow beyond the 5% drawdown limit.

**Foundry test plan:** Deploy `OracleValueStopLossExtension` with `drawdownE6 = 50_000`. Set oracle `bid = 0.9·Q64`, `ask = 1.1·Q64`. Execute a swap to set watermarks. Drain token1 until `metricT0` falls between 1900 and 1904.75. Assert that the stop-loss does **not** revert (demonstrating the guard fires later than configured). Then replace line 218 with the geometric mean and assert the stop-loss **does** revert at the same drain level.
