### Title
Zero-Signer Bypass in `requireValidTxSignatures` Allows Unauthorized Fast Withdrawals During Key Rotation — (`core/contracts/Verifier.sol`)

---

### Summary

When `nSigner` reaches `0` (all pubkeys deleted), `requireValidTxSignatures` unconditionally passes with an empty `signatures` array, because its sole authorization check is `nSignatures == nSigner` — which evaluates to `0 == 0`. Any caller can then invoke `BaseWithdrawPool.submitFastWithdrawal` with a self-crafted withdrawal transaction and drain pool funds.

---

### Finding Description

`requireValidTxSignatures` counts non-empty entries in the caller-supplied `signatures` array and enforces:

```solidity
require(nSignatures == nSigner, "not enough signatures");
``` [1](#0-0) 

When `nSigner == 0`, passing `signatures = []` yields `nSignatures = 0`, so `0 == 0` is trivially satisfied and the function returns without any cryptographic check.

`nSigner` reaches `0` whenever the owner deletes all registered pubkeys:

```solidity
function deletePubkey(uint256 index) public onlyOwner {
    if (!isPointNone(pubkeys[index])) {
        nSigner -= 1;
        delete pubkeys[index];
    }
``` [2](#0-1) 

This state naturally arises during a key rotation if the operator removes old keys before registering new ones. The window may be brief, but it is observable on-chain and exploitable by any watcher.

`submitFastWithdrawal` has no other authorization guard — the only security check is the call to `requireValidTxSignatures`:

```solidity
v.requireValidTxSignatures(transaction, idx, signatures);
``` [3](#0-2) 

The `checkQuorum` function cited in the question is on a completely separate code path (`requireValidSignature`, used for Schnorr-based operations) and is **never called** by `submitFastWithdrawal`. The inconsistency between the two quorum thresholds is real but irrelevant to this exploit path.

---

### Impact Explanation

An attacker who observes `nSigner == 0` on-chain can:

1. Craft a well-formed `WithdrawCollateral` or `WithdrawCollateralV2` transaction bytes with their own address as `sendTo`/`sender`.
2. Call `submitFastWithdrawal(idx, transaction, [])` with an empty signatures array.
3. Pass all checks: `!markedIdxs[idx]`, `idx > minIdx`, and `requireValidTxSignatures` (trivially).
4. Receive `transferAmount` tokens from the pool via `handleWithdrawTransfer`. [4](#0-3) 

The pool's entire token balance for any supported `productId` is at risk. This is a direct, unauthorized asset transfer — a clear solvency violation.

---

### Likelihood Explanation

The attacker does not need any privileged access. They only need to:
- Monitor the `DeletePubkey` event or `nSigner` storage slot.
- Submit a transaction in the same block or the next block after `nSigner` drops to `0`.

Key rotation is a routine operational event. The code does not enforce "add before delete" ordering, making the zero-signer window a realistic occurrence.

---

### Recommendation

Add an explicit guard in `requireValidTxSignatures` rejecting the zero-signer state:

```solidity
require(nSigner > 0, "no signers registered");
require(nSignatures == nSigner, "not enough signatures");
```

Additionally, enforce in `deletePubkey` that at least one key must remain registered, or require that new keys are assigned before old ones are removed, eliminating the zero-signer window entirely.

---

### Proof of Concept

```solidity
// 1. Deploy Verifier with 2 pubkeys → nSigner == 2
// 2. Owner calls deletePubkey(0) → nSigner == 1
// 3. Owner calls deletePubkey(1) → nSigner == 0
// 4. Attacker crafts a WithdrawCollateral tx bytes with sendTo = attacker
// 5. Attacker calls submitFastWithdrawal(minIdx+1, craftedTx, [])
//    → requireValidTxSignatures: nSignatures=0, nSigner=0, 0==0 → passes
//    → resolveFastWithdrawal returns attacker address as sendTo
//    → handleWithdrawTransfer sends tokens to attacker
// Assert: attacker received pool funds with zero signatures
``` [5](#0-4) [6](#0-5)

        require(idx > minIdx, "idx too small");

        (

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
