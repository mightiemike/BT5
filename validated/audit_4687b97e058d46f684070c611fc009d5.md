Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Unauthorized Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call) and instead checks `owner` (a free caller-supplied argument designating the position recipient). Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any address not on the allowlist can pass an already-allowed address as `owner` and have the extension approve the call, completely defeating the deposit gate.

## Finding Description
`MetricOmmPool.addLiquidity` accepts `owner` as an arbitrary caller-supplied address and forwards `msg.sender` as `sender` to the extension hook:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

There is no `msg.sender == owner` guard on `addLiquidity` (only `removeLiquidity` enforces this at L206): [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` ignores `sender` entirely (unnamed first parameter) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly gates on `sender`: [4](#0-3) 

Exploit path:
1. Pool is deployed with `DepositAllowlistExtension`; `allowedDepositor[pool][alice] = true`; bob is NOT allowed.
2. Bob calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. Extension checks `allowedDepositor[pool][alice]` → `true` → hook passes.
4. Pool executes the liquidity callback, pulling tokens from bob.
5. Position is minted under alice's address.
6. Alice calls `removeLiquidity` (she satisfies `msg.sender == owner`) and recovers the tokens; proceeds split off-chain.

## Impact Explanation
This is an admin-boundary break: a pool admin who deploys a curated pool with `DepositAllowlistExtension` to restrict liquidity provision to KYC'd or trusted addresses finds the control is completely ineffective. Any unauthorized address can inject liquidity into the pool, bypassing the allowlist policy entirely. With collusion between the unauthorized depositor and an allowed address, the unauthorized depositor can also recover their tokens, effectively laundering unauthorized liquidity through the curated pool.

## Likelihood Explanation
The bypass requires only a single direct call to `pool.addLiquidity` with `owner` set to any known allowed address. The `allowedDepositor` mapping is public on-chain, so any motivated actor can trivially identify an allowed address. No special privileges, flash loans, or complex setup are needed. The attack is repeatable at will.

## Recommendation
Change `beforeAddLiquidity` to gate on `sender` (the actual depositor/payer) rather than `owner` (the position recipient), consistent with how `SwapAllowlistExtension` gates on `sender`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Also update `isAllowedToDeposit` and `setAllowedToDeposit` documentation to clarify that the controlled address is the depositor/payer, not the position owner.

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - allowedDepositor[pool][alice] = true
  - bob is NOT on the allowlist

Attack:
  1. bob calls pool.addLiquidity(alice, salt, deltas, callbackData, "")
  2. _beforeAddLiquidity(bob, alice, ...) is called on the extension
  3. Extension checks allowedDepositor[pool][alice] → true → passes
  4. Pool pulls tokens from bob via liquidity callback
  5. Position minted for alice
  6. alice calls pool.removeLiquidity(alice, salt, deltas, "") → receives tokens
  7. alice and bob split proceeds off-chain

Result: bob bypassed the deposit allowlist entirely with a single direct call.
```
