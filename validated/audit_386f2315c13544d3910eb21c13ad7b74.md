### Title
Misleading `ENABLE_SECP_OPS` Flag: 4-Byte Secp Opcode Path Bypasses Flag Gate — (`File: src/chia_dialect.rs`)

### Summary
The `ENABLE_SECP_OPS` flag in `ChiaDialect` is documented as the hard-fork activation gate for `secp256k1_verify` and `secp256r1_verify`. However, the 4-byte opcode dispatch path (`0x13d61f00`, `0x1c3a8f00`) routes directly to the same secp functions **without checking `ENABLE_SECP_OPS`**, while the 1-byte alias path (opcodes 64, 65) correctly requires the flag. This is a direct flag/operator wiring error: the flag is a misleading access control that does not actually gate the secp functionality it claims to control.

### Finding Description
In `src/chia_dialect.rs`, the `ChiaDialect::op()` method has two separate dispatch paths for secp operations:

**Path 1 — 4-byte opcodes (no flag check):** [1](#0-0) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

**Path 2 — 1-byte opcodes (flag-gated):** [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The `ENABLE_SECP_OPS` flag is defined as: [3](#0-2) 

`MEMPOOL_MODE` — the stricter mode used for transaction validation — does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

This means that in mempool mode, 1-byte secp opcodes (64, 65) fall through to `unknown_operator` and are rejected by `NO_UNKNOWN_OPS`. But the 4-byte opcodes `0x13d61f00` and `0x1c3a8f00` are matched and dispatched to the real secp functions before `unknown_operator` is ever reached — bypassing both the `ENABLE_SECP_OPS` check and the `NO_UNKNOWN_OPS` rejection.

### Impact Explanation
Any caller who constructs a `ChiaDialect` without `ENABLE_SECP_OPS` — including the standard `MEMPOOL_MODE` — operates under the false assumption that secp signature verification is disabled. An attacker-controlled CLVM program that encodes secp opcodes as 4-byte atoms (`0x13d61f00` or `0x1c3a8f00`) will have those operations fully executed regardless of the flag state. The concrete corrupted invariant is: **the `ENABLE_SECP_OPS` flag does not actually gate secp execution**. If this flag is used as a hard-fork activation boundary (enabling secp ops only after a specific block height), programs using 4-byte secp opcodes would be accepted by all nodes at all times, creating a consensus divergence between nodes that enforce the flag boundary and those that do not. Additionally, mempool-mode validation silently accepts secp verification via 4-byte opcodes despite `ENABLE_SECP_OPS` being absent from `MEMPOOL_MODE`.

### Likelihood Explanation
The entry path is fully attacker-controlled: any CLVM program submitted to the mempool or evaluated by the VM can use 4-byte secp opcodes. No special privileges are required. The 4-byte encoding is a valid atom that any CLVM serializer can produce. The likelihood is high for any scenario where `ENABLE_SECP_OPS` is intended as a hard-fork gate.

### Recommendation
The 4-byte secp opcode dispatch block should check `ENABLE_SECP_OPS` before routing to the secp functions, consistent with how the 1-byte aliases are gated:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that `ENABLE_SECP_OPS` is the single, authoritative gate for secp functionality regardless of opcode encoding length, eliminating the misleading dual-path access control.

### Proof of Concept
Craft a CLVM program using the 4-byte secp opcode atom `\x13\xd6\x1f\x00` as the operator with valid pubkey/msg/sig arguments. Run it under `ChiaDialect::new(MEMPOOL_MODE)` (which excludes `ENABLE_SECP_OPS`). The secp verification executes and returns `nil` on success — despite `ENABLE_SECP_OPS` being absent — while the equivalent program using 1-byte opcode `\x40` (64) would be rejected with `Unimplemented` under the same flags.
