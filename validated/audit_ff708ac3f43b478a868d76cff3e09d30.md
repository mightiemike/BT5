### Title
`burnNlp` Permitted When NLP Pool Subaccounts Are Undercollateralized, Worsening Recovery — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp()` drains quote tokens from NLP pool subaccounts but performs **no health check on those pool subaccounts** after the drain. When NLP pools are already undercollateralized (negative health due to accumulated trader losses), any NLP holder can still burn tokens and withdraw quote tokens, deepening the deficit and making recovery harder. This is the direct analog to the external report's pattern: an operation that should be restricted during a degraded collateral state is not restricted, worsening the system's ability to recover.

---

### Finding Description

`Clearinghouse.burnNlp()` executes the following state changes:

1. Reduces the sender's NLP balance and increases `N_ACCOUNT`'s NLP balance.
2. Increases the sender's quote balance by `quoteAmount`.
3. Calls `_applyNlpRebalance()` which **reduces the quote balance of each NLP pool subaccount** by the corresponding `nlpPoolRebalanceX18[i]` amount.

After these changes, the only post-condition checks are on the **sender's** state:

```solidity
require(
    spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
    ERR_SUBACCT_HEALTH
);
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
```

There is **no check on the health of the NLP pool subaccounts** after their quote balances are reduced. [1](#0-0) 

By contrast, `forceRebalanceNlpPool()` — which also redistributes quote tokens among NLP pool subaccounts — does enforce a post-rebalance health check on pool subaccounts (starting at index 1):

```solidity
for (uint128 i = 1; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
``` [2](#0-1) 

This asymmetry means `burnNlp` can push NLP pool subaccounts into (or deeper into) negative health with no on-chain enforcement to stop it.

The NLP pool subaccounts hold quote tokens that back trader positions. When traders incur losses that exceed available settlement funds, those losses are socialized against the NLP pools via `socializeSubaccount`, reducing their `cumulativeDepositsMultiplierX18` and potentially leaving them with negative effective quote balances. [3](#0-2) 

`nlpPools[0]` is initialized to `N_ACCOUNT` — the primary NLP pool — and is the subaccount that absorbs the largest share of rebalance flows. [4](#0-3) 

---

### Impact Explanation

**Impact: Medium**

When NLP pools are undercollateralized, each `burnNlp` call:
- Withdraws quote tokens from already-depleted pool subaccounts.
- Increases the total quote deficit that must be replenished before the pools return to health.
- Directly mirrors the external report's pattern: each burn increases the amount of quote tokens required to restore the system to a healthy state, prolonging undercollateralization.

Remaining NLP holders bear the cost: their redemption value is diluted as the pool deficit grows, and socialization losses are amplified.

---

### Likelihood Explanation

**Likelihood: Medium**

- NLP pool undercollateralization is a realistic outcome during extreme market conditions (large coordinated trader profits, oracle manipulation, or cascading liquidations).
- Any NLP holder — an unprivileged user — can submit a `BurnNlp` transaction. The sequencer processes it via `EndpointTx.processTransaction`. [5](#0-4) 
- Critically, users can also submit `BurnNlp` via the **slow-mode escape hatch** (`executeSlowModeTransaction`), bypassing any off-chain sequencer-level filtering the operator might attempt. [6](#0-5) 
- There is no on-chain gate preventing burns during pool undercollateralization.

---

### Recommendation

After `_applyNlpRebalance` in `burnNlp`, add a health check on each NLP pool subaccount, consistent with the pattern already used in `forceRebalanceNlpPool`:

```solidity
// After _applyNlpRebalance in burnNlp:
for (uint128 i = 0; i < nlpPools.length; i++) {
    require(
        getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
        ERR_SUBACCT_HEALTH
    );
}
```

This ensures that `burnNlp` cannot be executed when doing so would push any NLP pool subaccount into (or further into) negative health, directly preventing the degraded-state worsening described above.

---

### Proof of Concept

1. NLP pools accumulate losses from trader PnL settlement; `N_ACCOUNT` (pool 0) has a negative effective quote balance after socialization.
2. User holds unlocked NLP tokens (acquired before the loss event).
3. User signs and submits a `BurnNlp` transaction (or submits via slow mode after the timeout).
4. `EndpointTx` calls `clearinghouse.burnNlp(...)`. [7](#0-6) 
5. `_applyNlpRebalance` reduces `N_ACCOUNT`'s quote balance further. [8](#0-7) 
6. Post-burn checks only verify the **sender's** health — the NLP pool's worsened health is never checked. [1](#0-0) 
7. The pool deficit grows; remaining NLP holders face larger socialization losses and a longer recovery path.

    function socializeSubaccount(bytes32 subaccount) external {
        require(msg.sender == address(_clearinghouse), ERR_UNAUTHORIZED);

        uint32[] memory _productIds = getProductIds();
        for (uint128 i = 0; i < _productIds.length; ++i) {
            uint32 productId = _productIds[i];

            State memory state = states[productId];
            Balance memory balance = balanceNormalizedToBalance(
                state,
                balances[productId][subaccount]
            );
            if (balance.amount < 0) {
                int128 totalDeposited = state.totalDepositsNormalized.mul(
                    state.cumulativeDepositsMultiplierX18
                );

                state.cumulativeDepositsMultiplierX18 = (totalDeposited +
                    balance.amount).div(state.totalDepositsNormalized);

                require(state.cumulativeDepositsMultiplierX18 > 0);

                state.totalBorrowsNormalized += balance.amount.div(
                    state.cumulativeBorrowsMultiplierX18
                );

                _setBalanceAndUpdateBitmap(
                    productId,
                    subaccount,
                    BalanceNormalized({amountNormalized: 0})
                );
                _setState(productId, state);
            }
        }
    }
