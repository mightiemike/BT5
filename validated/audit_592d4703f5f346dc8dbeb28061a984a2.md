Audit Report

## Title
Pool Admin Bypasses Factory Fee Caps via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol` forwards raw `uint16` bin-level additional fees directly to the pool with no cap validation, while the analogous `setPoolAdminFees` correctly enforces `maxAdminSpreadFeeE6`. A pool admin can set `addFeeBuyE6 = addFeeSellE6 = 65535` (6.5535% in E6 units) on any bin, bypassing whatever ceiling the factory owner has configured, and extract a proportional share of the inflated surplus through `collectFees`.

## Finding Description

`setPoolAdminFees` enforces the factory cap before updating fees:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check and passes the raw values directly to the pool:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, not the fee magnitude: [3](#0-2) 

The stored `addFeeBuyE6`/`addFeeSellE6` are added directly to `baseFeeX64` on every swap through that bin:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) 

The inflated fee causes traders to pay more than the oracle price, increasing the pool's surplus (`balance - binTotals - notionalFees`). The admin then receives `adminSpreadFeeE6 / spreadSumE6` of that surplus on `collectFees`: [6](#0-5) 

The hard cap constants (`HARD_MAX_SPREAD_FEE_E6 = 200_000`) only constrain the factory owner's `setFeeCaps` call; they are never applied to `setPoolBinAdditionalFees`: [7](#0-6) 

## Impact Explanation

Traders swapping through affected bins pay up to 6.5535% (`uint16.max` in E6) additional fee per bin, far exceeding any `maxAdminSpreadFeeE6` the factory owner has configured. The pool admin receives a proportional share of the inflated surplus through `collectFees`, constituting a direct financial loss for traders. This is a confirmed admin-boundary break: the factory owner's fee governance cap is rendered ineffective for per-bin fees.

## Likelihood Explanation

The pool admin is a semi-trusted role explicitly bounded by factory caps. The bypass requires a single call to `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` with no preconditions, no timelock, and no special oracle state. Any pool admin can trigger this at any time on any active pool, immediately affecting the next swap.

## Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap configurable by the factory owner independently of the global admin spread cap.

## Proof of Concept

1. Factory owner calls `setFeeCaps` setting `maxAdminSpreadFeeE6 = 10_000` (1%).
2. Pool admin attempts `setPoolAdminFees(pool, 10_001, 0)` → reverts with `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → **succeeds**. Bin 0 now carries 6.5535% additional buy and sell fee.
4. Next trader swapping through bin 0 pays `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — 6.5535% above the oracle price, far exceeding the factory owner's intended 1% ceiling.
5. On the next `collectPoolFees` call, the admin receives their proportional share of the inflated surplus, extracting value from traders beyond what the factory's cap system was designed to permit.
