### Title
`ValidResourceBounds` Protobuf Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Wrong Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to choose between the `L1Gas` and `AllResources` variants. When both `l2_gas` and `l1_data_gas` are zero it silently produces `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound elements depending on the variant (3 for `L1Gas`, 4 for `AllResources`), any `AllResources` transaction whose l2/l1-data amounts happen to be zero will have its transaction hash recomputed to a different value after a P2P round-trip, binding the wrong hash to the executable payload.

### Finding Description

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `resource_bounds: AllResourceBounds` — the three-resource variant is the only type accepted at the RPC boundary. Their `InvokeTransactionV3Trait::resource_bounds()` implementation always returns `ValidResourceBounds::AllResources(self.resource_bounds)`. [1](#0-0) 

`get_tip_resource_bounds_hash` then hashes **four** elements — `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` — for `AllResources`: [2](#0-1) 

For `L1Gas` it hashes only **three** elements — `[tip, L1_GAS_packed, L2_GAS_packed]` — omitting `L1_DATA_GAS_packed`: [3](#0-2) 

When the transaction is propagated over P2P and deserialized from protobuf, the converter applies this heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [4](#0-3) 

An `AllResources` transaction whose `l2_gas` and `l1_data_gas` amounts are both zero (a valid and common configuration — e.g., a pure L1-gas transaction submitted via the new RPC format, or a client-side-proving transaction) is silently re-classified as `L1Gas`. The hash recomputed from the deserialized object omits the `L1_DATA_GAS_packed` element, producing a **different Poseidon digest** than the one the submitter signed and the one stored in the block.

The `invoke_v3_client_side_proving.json` test fixture demonstrates exactly this shape: `L2_GAS` amounts are zero and `L1_DATA_GAS` is absent (defaults to zero): [5](#0-4) 

### Impact Explanation

Any node that recomputes the transaction hash from the deserialized `InvokeTransactionV3` — for gateway admission, mempool deduplication, or block validation — will obtain a hash that does not match the one the account signed. This falls squarely under:

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

Concretely:
- A receiving gateway that re-derives the hash will reject a valid, correctly-signed transaction.
- A block-validating node will compute a mismatched hash and reject the block.
- If the stored hash is used for execution but the deserialized `resource_bounds` variant is `L1Gas`, fee-computation mode switches from `AllResources` (per-resource accounting) to `L1Gas` (discounted single-resource accounting), producing wrong fee charges.

### Likelihood Explanation

The trigger requires only that a user submit an `AllResources` Invoke V3 transaction with `l2_gas.max_amount == 0` and `l1_data_gas.max_amount == 0`. This is a normal configuration for any transaction that does not use Sierra gas or blob DA. The client-side-proving path (which sets `l2_gas` to zero) is an explicit production use-case already tested in the repository. No privileged access is required; any unprivileged user can craft such a transaction.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
