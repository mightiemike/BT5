Looking at the `x/tieredrewards` module, specifically `ClaimTierRewards` in `msg_server.go` and `claimRewardsAndUpdatesPositions` in `claim_rewards.go`, there is a concrete analog to the duplicate-claim vulnerability class.

---

### Title
Duplicate Position ID in `MsgClaimTierRewards` Enables Double Bonus Reward Drain from Rewards Pool - (`x/tieredrewards/keeper/msg_server.go`)

### Summary
`ClaimTierRewards` loads all positions from state into a slice **before** processing any of them. If a caller submits the same position ID multiple times in `msg.PositionIds`, the same stale position state (with the same `LastEventSeq` and `LastBonusAccrual`) is processed multiple times. Each iteration independently computes and transfers bonus rewards from the `RewardsPoolName` module account, resulting in N-fold overclaim for a single position.

### Finding Description

In `ClaimTierRewards` (`msg_server.go:429–468`), positions are loaded in a first loop and appended to a slice:

```go
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    // ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

There is no deduplication of `msg.PositionIds`. If the same ID appears N times, N identical copies of the position state (same `LastEventSeq`, same `LastBonusAccrual`) are appended to the slice.

`claimRewardsAndUpdatesPositions` then iterates over this slice:

```go
for i := range positions {
    pos := &positions[i]
    // ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    // ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

`setPosition` at the end of iteration `i=0` writes the updated position (with advanced `LastEventSeq` and `LastBonusAccrual`) to the KV store. However, `positions[1]` through `positions[N-1]` are **in-memory copies loaded before any writes occurred**. They all carry the original stale `LastEventSeq` and `LastBonusAccrual`.

Inside `processEventsAndClaimBonus`, the current-segment bonus is computed using `segmentStart = pos.LastBonusAccrual`:

```go
segmentStart := pos.LastBonusAccrual
// ...
if bonded && val.IsBonded() {
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
    totalBonus = totalBonus.Add(bonus)
}
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
``` [3](#0-2) 

Because each duplicate copy has the same old `LastBonusAccrual`, the segment `[old_LastBonusAccrual, blockTime]` is computed and paid on every iteration. The bonus is transferred via:

```go
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
    return nil, err
}
``` [4](#0-3) 

This `SendCoinsFromModuleToAccount` call executes once per duplicate entry, each time transferring the full bonus amount from the `RewardsPoolName` pool to the owner.

For historical validator events (the loop over `events`), the same events are re-fetched using the stale `pos.LastEventSeq` and their ref counts are decremented again via `decrementEventRefCount`. When the ref count for an event is already 1 (only this position references it), the second decrement would underflow or error, causing the transaction to revert for that path. However, the **current-segment bonus** (computed after the events loop, without any ref-count guard) is always re-paid as long as the validator is bonded — which is the normal operating condition. [5](#0-4) 

### Impact Explanation

The `RewardsPoolName` module account balance is drained at N times the legitimate rate. An attacker submitting `PositionIds = [id, id, id, ..., id]` (N repetitions) receives N times the bonus rewards they are entitled to in a single transaction. This directly corrupts the `RewardsPoolName` module account balance and constitutes unauthorized asset movement from the protocol's reward pool to the attacker.

### Likelihood Explanation

Any position owner can trigger this with a standard `MsgClaimTierRewards` transaction. No special privileges, validator access, or social engineering is required. The only precondition is holding at least one active, delegated tiered-rewards position with accrued bonus rewards — a normal user state. The entry path is a standard CLI-signed transaction.

### Recommendation

Deduplicate `msg.PositionIds` before the loading loop in `ClaimTierRewards`. A simple approach:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    // ... existing load logic
}
```

Alternatively, reload the position from the KV store at the start of each iteration inside `claimRewardsAndUpdatesPositions` rather than operating on pre-loaded copies, so that the updated `LastEventSeq` and `LastBonusAccrual` from a prior iteration are always visible.

### Proof of Concept

1. User creates a tiered-rewards position with ID `42`, delegated to a bonded validator. Bonus rewards accrue over time.
2. User constructs `MsgClaimTierRewards{Owner: user, PositionIds: [42, 42, 42, 42, 42]}`.
3. The first loop in `ClaimTierRewards` loads position `42` five times, producing five identical `PositionState` copies with the same `LastBonusAccrual = T_old`.
4. `claimRewardsAndUpdatesPositions` iterates over all five copies. Each call to `processEventsAndClaimBonus` computes `computeSegmentBonus` over `[T_old, T_now]` and calls `SendCoinsFromModuleToAccount` for the full bonus amount.
5. `setPosition` after each iteration writes the updated position to the KV store, but the remaining in-memory copies are unaffected.
6. The user receives 5× the legitimate bonus reward. The `RewardsPoolName` account is debited 5× the correct amount. [6](#0-5) [7](#0-6)

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventTierRewardsClaimed{
		Owner:        msg.Owner,
		PositionIds:  msg.PositionIds,
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClaimTierRewardsResponse{
		BaseRewards:  totalBase,
		BonusRewards: totalBonus,
		PositionIds:  msg.PositionIds,
	}, nil
}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
