## Title
False `OracleStopLossTriggered` DoS via `_metrics` `minShares` Floor When `binTotalShares < minimalMintableLiquidity` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

## Summary

`_metrics` floors the share denominator at `minimalMintableLiquidity` (`minShares`). When `binTotalShares` legitimately falls below `minShares` through normal multi-LP withdrawals, the computed per-share metrics are artificially deflated relative to the previously recorded high-watermark, causing `_checkAndUpdateWatermarks` to fire `OracleStopLossTriggered` on a legitimate swap and permanently blocking swaps in the affected direction until recovery.

---

## Finding Description

### The floor in `_metrics` [1](#0-0) 

```solidity
uint256 shares = totalShares < minShares ? minShares : totalShares;
```

When `totalShares < minShares`, the denominator is inflated to `minShares`, making `metricT0` and `metricT1` lower than the true per-share value.

### The per-position check does NOT protect `binTotalShares`

`removeLiquidity` enforces `minimalMintableLiquidity` only on the **individual position's remaining shares**, not on the bin's aggregate total: [2](#0-1) 

```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

A user removing **all** their shares (`newUserShares == 0`) passes this check unconditionally. With two LPs each holding shares ≥ `minShares` individually, one LP removing all their shares can leave `binTotalShares` in the range `(0, minShares)`.

### Watermark is not updated on breach [3](#0-2) 

When `breach0` or `breach1` is true, the function reverts **before** writing the new watermark. The watermark stays at the old high, so every subsequent swap in the affected direction also reverts.

---

## Impact Explanation

Swaps touching any bin where `0 < binTotalShares < minimalMintableLiquidity` are permanently blocked in the direction that checks the breached metric. This is broken core pool functionality: the swap path is unusable. LP principal remains accessible via `removeLiquidity` (which does not call `afterSwap`), but the pool's primary trading function is DoS'd until either:
- New liquidity is added to bring `binTotalShares ≥ minShares`, or
- The watermark decays to zero (only if `decayPerSecondE8 > 0`), or
- The pool admin resets the watermark (timelocked, not immediate).

---

## Likelihood Explanation

Any pool with `OracleValueStopLossExtension` and non-zero `drawdownE6` is affected. The condition is reachable through normal LP behavior: two or more LPs each holding ≥ `minShares` in the same bin, one of whom fully exits. No attacker privilege is required — the exiting LP is exercising a normal right. The condition is especially likely in low-liquidity bins or during market stress when LPs withdraw.

---

## Recommendation

Replace the `minShares` floor in `_metrics` with a skip: if `totalShares < minShares`, treat the bin as having no meaningful metric (skip watermark update, analogous to the `totalShares == 0` skip already present at line 238). Alternatively, enforce a bin-level minimum total shares invariant in `removeLiquidity` — but this would break LP exit rights and is a more invasive change. [4](#0-3) 

The existing `if (totalShares == 0) continue;` guard should be extended to `if (totalShares < minShares) continue;` so that bins with sub-floor share counts are simply not evaluated against the watermark.

---

  function _metrics(uint104 t0, uint104 t1, uint256 totalShares, uint256 minShares, uint256 midPriceX64)
    private
    pure
    returns (uint256 metricT0, uint256 metricT1)
  {
    uint256 shares = totalShares < minShares ? minShares : totalShares;
    uint256 t0ps = Math.mulDiv(uint256(t0), METRIC_SCALE, shares);
    uint256 t1ps = Math.mulDiv(uint256(t1), METRIC_SCALE, shares);
    metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
    metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
  }
