Audit Report

## Title
`SwapAllowlistExtension` checks the router address as the swapper instead of the originating user, allowing any user to bypass a curated pool's swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which is the router contract address when a user routes through `MetricOmmSimpleRouter`. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`, meaning it checks whether the **router** is allowlisted rather than the originating EOA. This renders the swap allowlist completely ineffective for router-mediated swaps, which is the primary public swap path.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // <-- whoever called pool.swap()
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this value unchanged to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))` (L149-177).

`SwapAllowlistExtension.beforeSwap` then gates on:
```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly:
```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    ...
  );
```

At the pool, `msg.sender` is the **router address**, so `sender` delivered to the extension is the router, not the originating EOA. The extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`.

The same mismatch applies to `exactInput` (L99-118), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188) — all router entry points call `pool.swap()` as `msg.sender = router`.

There is no existing guard that recovers the originating EOA. The router stores the originating `msg.sender` in transient storage for callback payment purposes (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` at L71), but this context is never read by the pool or extension during the `beforeSwap` hook.

## Impact Explanation
A pool admin deploys a curated pool (e.g., KYC-only, institutional-only) and configures `SwapAllowlistExtension` with a set of approved user addresses. The invariant is that only those addresses may swap. Because the extension checks the router's address instead of the user's address, the allowlist is completely ineffective for router-mediated swaps. This is an admin-boundary break: the pool admin's access control policy is bypassed by any unprivileged user making a standard router call. If the admin allowlists the router to enable legitimate users to swap via the router, then all non-allowlisted users can also swap freely through the router. If the admin does not allowlist the router, legitimate allowlisted users cannot use the router at all, breaking core swap functionality.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap entrypoint deployed alongside the protocol. Any user who knows the pool has an allowlist can trivially route through the public router to bypass it. No privileged access, no special tokens, and no admin cooperation is required. The trigger is a single standard router call, repeatable by any address.

## Recommendation
The pool must deliver the originating user's address to the extension, not the intermediary's address. The cleanest fix is for the router to record the originating EOA in transient storage alongside the existing callback context, and for the pool to read it when dispatching extension hooks. Alternatively, the extension could be redesigned to check `recipient` if the pool enforces `recipient == msg.sender`, but this is not currently enforced and `recipient` is caller-controlled.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Admin calls setAllowedToSwap(pool, alice, true)   // only alice is allowed
  - Admin calls setAllowedToSwap(pool, router, true)  // necessary for alice to use the router

Attack:
  1. bob (non-allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: bob,
       ...
     })
  2. Router calls pool.swap(bob, ...) — msg.sender at pool = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router]
     → router IS allowlisted (admin had to allowlist it for alice to use the router)
     → bob's swap passes

  Result: bob, a non-allowlisted user, successfully swaps on a curated pool
  by routing through the public MetricOmmSimpleRouter.

  Alternatively, if admin does NOT allowlist the router:
  - alice cannot use the router at all (her swap reverts because the router is not allowlisted)
  - demonstrating the identity mismatch breaks the intended UX in either direction
```
