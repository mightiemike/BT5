Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address as `sender` Instead of End-User, Enabling Full Allowlist Bypass — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the end-user. The allowlist check `allowedSwapper[pool][sender]` therefore evaluates the router's address. A pool admin who allowlists the router to permit any routed swap inadvertently grants every user — including those explicitly excluded — the ability to bypass the per-user allowlist.

## Finding Description

**Root cause — `sender` is the immediate caller of `pool.swap()`**

In `MetricOmmPool.swap()`:
```solidity
_beforeSwap(
    msg.sender,   // ← always the direct caller of pool.swap()
    recipient, ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` verbatim: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whatever the pool passed in: [3](#0-2) 

**Router path — `sender` becomes the router address**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router `msg.sender` of that call: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Exploit flow**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to a whitelist of addresses.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` to allow routed swaps for any permitted user.
3. An unprivileged user (not on the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`. The extension checks `allowedSwapper[pool][router]` → `true`. The swap proceeds.
5. The non-allowlisted user successfully swaps on a pool they were explicitly excluded from.

**Existing guards are insufficient**

The `onlyPool` modifier on `BaseMetricExtension.beforeSwap` only verifies the caller is a registered pool — it does not recover the original end-user identity. [6](#0-5) 

There is no mechanism in the current call path to pass the original `msg.sender` (the end-user) through the router into the pool's `sender` parameter.

## Impact Explanation
The `SwapAllowlistExtension` is rendered completely ineffective for router-mediated swaps. Any user can bypass a pool's swap access control by routing through `MetricOmmSimpleRouter`, violating the admin-boundary invariant that pool admins can restrict who may trade. This constitutes broken core pool functionality — the access control gate the pool admin configured does not protect the pool as intended.

## Likelihood Explanation
The condition is trivially reachable: any unprivileged user can call the public router functions. The only precondition is that the pool admin has allowlisted the router (a necessary step to allow any routed swap). No special privileges, flash loans, or complex setup are required. The bypass is repeatable on every router-mediated swap.

## Recommendation
Pass the original end-user address through the call chain. One approach: have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known router. A cleaner approach is to add a dedicated `swapper` field to the pool's swap interface that the router populates with `msg.sender` before calling `pool.swap()`, and have the pool pass that value (not its own `msg.sender`) as `sender` to `_beforeSwap`. Alternatively, the extension can maintain a registry of trusted routers and, when `sender` is a router, require the actual user identity to be supplied and verified via `extensionData`.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension
// 2. Pool admin allowlists the router: extension.setAllowedToSwap(pool, router, true)
// 3. Confirm attacker is NOT allowlisted: assertFalse(extension.isAllowedToSwap(pool, attacker))
// 4. Attacker calls router.exactInputSingle({pool: pool, ...}) — swap succeeds
// 5. Assert attacker received output tokens despite not being on the allowlist
```
A Foundry integration test deploying `MetricOmmPool` + `SwapAllowlistExtension` + `MetricOmmSimpleRouter`, allowlisting only the router, and confirming a non-allowlisted EOA can swap via the router, reproduces this in full.
