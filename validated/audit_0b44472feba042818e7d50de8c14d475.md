### Title
Zero-Margin Isolated Subaccount Registered as Active Without Minimum Collateral Validation — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`createIsolatedSubaccount()` in `OffchainExchange.sol` unconditionally registers a new isolated subaccount into the `isolatedSubaccounts` mapping regardless of whether any margin is deposited. `isIsolatedSubaccountActive()` then returns `true` for any registered subaccount without checking its collateral balance. This directly mirrors the reported bug: a state entity is marked "active" without validating a minimum collateral threshold.

---

### Finding Description

`_isolatedMargin()` extracts the margin from the upper 64 bits of the order `appendix`:

```solidity
function _isolatedMargin(uint128 appendix) internal pure returns (uint128) {
    return (appendix >> 64) * (10**12);
}
``` [1](#0-0) 

When those bits are zero, `_isolatedMargin` returns `0`. Inside `createIsolatedSubaccount()`, the subaccount is unconditionally written into `isolatedSubaccounts` at line 1065 before the margin check:

```solidity
isolatedSubaccounts[txn.order.sender][id] = newIsolatedSubaccount;
``` [2](#0-1) 

The margin transfer is then gated on `if (margin > 0)`:

```solidity
int128 margin = int128(_isolatedMargin(txn.order.appendix));
if (margin > 0) {
    digestToMargin[digest] = margin;
    spotEngine.updateBalance(...);
    spotEngine.updateBalance(...);
}
``` [3](#0-2) 

So when `margin == 0`, the subaccount is registered but receives no collateral. `isIsolatedSubaccountActive()` then returns `true` for it purely based on existence in the mapping — no balance check:

```solidity
function isIsolatedSubaccountActive(bytes32 parent, bytes32 subaccount)
    external view returns (bool)
{
    for (uint256 id = 0; id < MAX_ISOLATED_SUBACCOUNTS_PER_ADDRESS; id++) {
        if (subaccount == isolatedSubaccounts[parent][id]) {
            return true;
        }
    }
    return false;
}
``` [4](#0-3) 

`transferQuote()` in `Clearinghouse.sol` relies on `isIsolatedSubaccountActive` as the sole gate before crediting an isolated subaccount with quote tokens:

```solidity
} else if (RiskHelper.isIsolatedSubaccount(txn.recipient)) {
    require(
        IOffchainExchange(offchainExchange).isIsolatedSubaccountActive(
            txn.sender,
            txn.recipient
        ),
        ERR_UNAUTHORIZED
    );
}
``` [5](#0-4) 

A zero-margin isolated subaccount passes this gate and can receive quote transfers.

---

### Impact Explanation

A zero-collateral isolated subaccount is treated as fully "active" by the protocol. Any subsequent `TransferQuote` transaction targeting it will pass the `isIsolatedSubaccountActive` guard and credit it with quote tokens. The isolated subaccount's own balance is never checked for a minimum threshold at creation or at the `isIsolatedSubaccountActive` call site. This breaks the protocol invariant that an isolated subaccount must hold margin before it is considered active and eligible to receive funds, and it allows a user to route quote balances into a zero-collateral isolated context that bypasses the intended margin-gating logic. [6](#0-5) 

---

### Likelihood Explanation

The trigger is fully unprivileged. Any user with a valid subaccount can submit a `CreateIsolatedSubaccount` slow-mode transaction through `EndpointTx` with the upper 64 bits of `appendix` set to zero. No special role, leaked key, or governance action is required. The `appendix` field is entirely user-controlled. [7](#0-6) 

---

### Recommendation

Add a minimum margin check inside `createIsolatedSubaccount()` before registering the subaccount. Require `margin > 0` (or `margin >= MIN_ISOLATED_MARGIN`) before writing to `isolatedSubaccounts`. Alternatively, update `isIsolatedSubaccountActive()` to also verify that the isolated subaccount's quote balance in `spotEngine` is above zero (or a defined minimum), so that "active" status reflects actual collateral presence rather than mere registration. [8](#0-7) 

---

### Proof of Concept

1. User holds a valid parent subaccount with sufficient quote balance.
2. User crafts a `CreateIsolatedSubaccount` transaction where `appendix >> 64 == 0` (zero margin bits), with the isolated flag set (`(appendix >> 8) & 1 == 1`).
3. `EndpointTx` dispatches to `OffchainExchange.createIsolatedSubaccount()`.
4. Inside `createIsolatedSubaccount()`: `_isolatedMargin(appendix)` returns `0`; the subaccount is written to `isolatedSubaccounts[sender][id]` unconditionally; the `if (margin > 0)` branch is skipped — no collateral is transferred.
5. `isIsolatedSubaccountActive(parent, newIsolatedSubaccount)` now returns `true` despite zero balance.
6. User submits a `TransferQuote` transaction with `recipient = newIsolatedSubaccount`.
7. `Clearinghouse.transferQuote()` calls `isIsolatedSubaccountActive` → `true`; the guard passes; quote tokens are credited to the zero-collateral isolated subaccount.
8. The parent's `_isAboveInitial` health check is applied to the sender, but the isolated subaccount itself has no minimum collateral enforced — it is active with zero margin, contradicting the protocol's intended isolation invariant. [3](#0-2) [9](#0-8)
