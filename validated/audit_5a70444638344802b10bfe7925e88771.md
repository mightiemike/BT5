### Title
Protobuf `ValidResourceBounds` round-trip silently downgrades `AllResources` to `L1Gas` when l2_gas=0 and l1_data_gas=0, producing a divergent transaction hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic — `l1_data_gas.is_zero() && l2_gas.is_zero()` — to decide whether to reconstruct a `ValidResourceBounds::L1Gas` (pre-0.13.3) or `ValidResourceBounds::AllResources` (post-0.13.3) variant. A post-0.13.3 V3 transaction whose user-supplied `AllResourceBounds` has zero `l2_gas` and zero `l1_data_gas` is silently downgraded to `L1Gas` on the receiving P2P node. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant, the transaction hash computed after P2P deserialization diverges from the hash computed and stored at the originating gateway, breaking hash canonicalization across the network.

### Finding Description

**Serialization path (originating node)**

`RpcInvokeTransactionV3` carries `AllResourceBounds` (never `ValidResourceBounds`). When the gateway converts it to `InternalRpcInvokeTransactionV3`, the `resource_bounds` field is `AllResourceBounds`. The hash is computed via `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash`, which calls `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash`. For `ValidResourceBounds::AllResources` the function appends the `l1_data_gas` packed felt, producing a 4-element poseidon input:

```
poseidon([tip, l1_gas_packed, l2_gas_packed(=0), l1_data_gas_packed(=0)])
``` [1](#0-0) 

**Protobuf serialization** of `ValidResourceBounds::AllResources` with zero `l2_gas` and `l1_data_gas` emits all three fields as zero: [2](#0-1) 

**Protobuf deserialization (receiving P2P node)**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)   // ← wrong variant chosen
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

The receiving node now holds `ValidResourceBounds::L1Gas`. When it recomputes the hash, `get_tip_resource_bounds_hash` takes the `L1Gas` branch and omits `l1_data_gas`, producing a 3-element poseidon input:

```
poseidon([tip, l1_gas_packed, l2_gas_packed(=0)])
``` [4](#0-3) 

`poseidon([tip, A, 0, 0]) ≠ poseidon([tip, A, 0])` — the two hashes are structurally distinct.

**Trigger condition**

The gateway accepts any `AllResourceBounds` with zero `l2_gas` and `l1_data_gas` because no validation enforces non-zero L2 gas for V3 transactions. The gateway's DA-mode checks are unrelated: [5](#0-4) 

An unprivileged user submitting a V3 invoke/declare/deploy-account transaction with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` is sufficient to trigger the divergence.

### Impact Explanation

Two concrete impacts follow from the hash divergence:

1. **Wrong hash bound to the executable payload (Critical/High)**: The receiving node stores the transaction under the transmitted hash H1 but internally holds a `L1Gas` variant. Any subsequent hash recomputation (e.g., `validate_transaction_hash`) produces H2 ≠ H1, causing the block or transaction to be flagged as invalid. A single such transaction in a block can prevent all downstream nodes from syncing past that block.

2. **Wrong fee/resource accounting (High)**: With `L1Gas`, `get_gas_vector_computation_mode` returns `NoL2Gas`, so the blockifier skips L2-gas and L1-data-gas bounds checks entirely. A transaction that was signed and admitted under `AllResources` semantics is executed under `L1Gas` semantics, producing incorrect fee computation and resource enforcement. [6](#0-5) 

### Likelihood Explanation

Any user can craft a valid V3 transaction with zero `l2_gas` and `l1_data_gas`. The gateway performs no check preventing this. The transaction will be accepted, included in a block, and then cause every P2P-syncing node to compute a divergent hash. No privileged access is required; the only cost is a transaction fee.

### Recommendation

Replace the value-based heuristic with an explicit version/era tag. The protobuf message should carry a boolean or enum field indicating whether the bounds are `L1Gas` (pre-0.13.3) or `AllResources` (post-0.13.3). Until the wire format is updated, the deserializer must default to `AllResources` when `l1_data_gas` is present in the message (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (`None`):

```rust
let l1_data_gas_opt = value.l1_data_gas;
Ok(match l1_data_gas_opt {
    None => ValidResourceBounds::L1Gas(l1_gas),   // absent ↔ pre-0.13.3
    Some(ldg) => ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas,
        l2_gas,
        l1_data_gas: ldg.try_into()?,
    }),
})
```

The TODO comment in the code already anticipates this fix: [7](#0-6) 

### Proof of Concept

1. Submit a V3 invoke transaction via RPC with `resource_bounds = { l1_gas: {max_amount: 1000, max_price_per_unit: 1}, l2_gas: {max_amount: 0, max_price_per_unit: 0}, l1_data_gas: {max_amount: 0, max_price_per_unit: 0} }`.
2. The gateway computes hash H1 using `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash` with `AllResources` → 4-element poseidon.
3. The transaction is stored with H1 and propagated via P2P as `protobuf::InvokeV3` with `l2_gas = 0`, `l1_data_gas = 0`.
4. The receiving node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`; the condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is true → `ValidResourceBounds::L1Gas(l1_gas)`.
5. The receiving node recomputes the hash using `L1Gas` → 3-element poseidon → H2.
6. H1 ≠ H2. `validate_transaction_hash` returns `false`; the block containing this transaction is rejected, and the node cannot advance its chain tip past this block. [8](#0-7) [9](#0-8)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
