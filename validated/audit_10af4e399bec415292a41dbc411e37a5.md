### Title
NFT Metadata Permanently Frozen After Transfer Due to Conjunctive `IsOwner` + `IsDenomCreator` Guard in `EditNFT` — (`x/nft/keeper/keeper.go`)

### Summary
`EditNFT` requires the sender to simultaneously satisfy `IsOwner` (current NFT holder) **and** `IsDenomCreator` (original denom issuer). After any `TransferNFT`, these two conditions can never be satisfied by the same address, making NFT metadata permanently immutable.

### Finding Description
In `x/nft/keeper/keeper.go`, `EditNFT` (lines 85–118) calls `IsOwner` then `IsDenomCreator` on the same `owner` address, both with early-return on error. `denom.Creator` is set once at `IssueDenom` and never updated. `TransferOwner` only updates `nft.Owner` and the owner index. After a transfer:

- **Buyer calls `MsgEditNFT`**: `IsOwner` passes, `IsDenomCreator` fails → `ErrUnauthorized`
- **Creator calls `MsgEditNFT`**: `IsOwner` fails (creator no longer holds the token) → `ErrUnauthorized`

No code path allows either party to satisfy both checks simultaneously post-transfer. The same AND-logic applies to `BurnNFT` (lines 141–161), so the transferred NFT also cannot be burned by either party through the normal `MsgBurnNFT` path. [5](#0-4) 

### Impact Explanation
NFT metadata (`Name`, `URI`, `Data`) is permanently frozen the moment the NFT is transferred for the first time. Any marketplace, game, or protocol built on this module that expects post-sale metadata updates (e.g., dynamic NFTs, royalty URI updates, provenance data) is permanently broken for all transferred tokens. The state corruption is irreversible without a governance migration.

### Likelihood Explanation
This is triggered by the most basic NFT lifecycle: issue denom → mint → transfer → attempt edit. It requires no special privileges, no governance, and no external dependencies. Any user who receives an NFT via `MsgTransferNFT` immediately experiences this.

### Recommendation
Replace the conjunctive guard with an OR condition: allow edit if the sender is **either** the current owner **or** the denom creator (or both). Alternatively, define a separate `UpdateDenomMetadata` path for creator-level edits and allow the current owner to edit their own token's mutable fields.

```go
// Option: allow owner OR denom creator
isOwner := owner.Equals(nft.GetOwner())
_, creatorErr := k.IsDenomCreator(ctx, denomID, owner)
if !isOwner && creatorErr != nil {
    return sdkerrors.Wrapf(types.ErrUnauthorized, ...)
}
```

### Proof of Concept
```
1. IssueDenom(sender=Alice)          → denom.Creator = Alice
2. MintNFT(sender=Alice, recipient=Alice, tokenID="t1")
                                     → nft.Owner = Alice
3. TransferNFT(sender=Alice, recipient=Bob, tokenID="t1")
                                     → nft.Owner = Bob
4. EditNFT(sender=Bob, ...)
   → IsOwner(Bob) = PASS
   → IsDenomCreator(Bob) = FAIL  → ErrUnauthorized
5. EditNFT(sender=Alice, ...)
   → IsOwner(Alice) = FAIL       → ErrUnauthorized
```

Both step 4 and step 5 fail. The NFT metadata is now permanently frozen with no authorized editor. [6](#0-5)

func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
