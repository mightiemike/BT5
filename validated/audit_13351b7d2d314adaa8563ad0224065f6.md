### Title
Ignored Boolean Return Value of `equihash::is_valid_solution` Allows Invalid PoW Acceptance — (`contract/src/zcash.rs`)

### Summary

The Zcash PoW verification path in `check_pow` calls `equihash::is_valid_solution`, which returns `Result<bool, Error>`. The code handles only the `Err` variant (panicking on error) via `.unwrap_or_else`, but the inner `bool` — which signals whether the solution is cryptographically valid — is silently discarded. An `Ok(false)` result (invalid solution, no error) causes no panic and no rejection. Any unprivileged NEAR caller can submit a Zcash block header with a fabricated Equihash solution and have it accepted into the canonical chain.

### Finding Description

In `contract/src/zcash.rs`, the `check_pow` function performs Equihash solution validation:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

The `equihash` crate (version `0.2.2`, declared as an optional dependency for the `zcash` feature) exposes `is_valid_solution` with the signature `fn is_valid_solution(n, k, input, nonce, solution) -> Result<bool, Error>`. The semantics are:

- `Ok(true)` — solution is valid
- `Ok(false)` — solution is **invalid** (wrong answer, but well-formed input)
- `Err(e)` — malformed/unparseable input

`.unwrap_or_else(|e| { env::panic_str(...) })` only intercepts the `Err` arm. When the function returns `Ok(false)`, `.unwrap_or_else` unwraps to `false`, and that `bool` is immediately dropped — the statement is a no-op. No `require!`, no binding, no check. The block passes PoW validation unconditionally as long as the nonce and solution bytes are parseable. [2](#0-1) 

The `check_pow` function is called from `check_target`, which is called from `submit_block_header` (the non-dogecoin path) whenever `skip_pow_verification` is `false`: [3](#0-2) 

The public entry point is `submit_blocks`, which is `#[payable]` and `#[pause]` but not restricted to any privileged role beyond the `trusted_relayer` macro — and the `trusted_relayer` bypass roles (`DAO`, `UnrestrictedSubmitBlocks`) are not required for ordinary submission: [4](#0-3) 

### Impact Explanation

A successful exploit corrupts the canonical chain state of the Zcash light client. Specifically:

- `mainchain_tip_blockhash` is updated to a block hash backed by zero real PoW.
- `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height` are populated with fraudulent entries. [5](#0-4) 

Downstream contracts consuming `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` will receive `true` for transactions in attacker-fabricated blocks, enabling false proof-of-inclusion results for arbitrary Bitcoin-family transactions on the Zcash deployment. [6](#0-5) 

### Likelihood Explanation

Any NEAR account can call `submit_blocks` on the Zcash contract deployment. The attacker only needs to craft a block header with:
1. A `bits` field matching the expected difficulty (computable from on-chain state via `get_config`).
2. A `solution` field that is well-formed (parseable by the equihash crate) but cryptographically invalid — trivially achievable by supplying any byte sequence of the correct length.
3. A `prev_block_hash` pointing to the current tip.

No privileged access, no key material, and no social engineering is required.

### Recommendation

Bind the return value and assert it is `true`:

```rust
let valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
require!(valid, "Invalid Equihash solution: solution check returned false");
```

This mirrors the pattern used for every other PoW check in the codebase, which uses `require!` to enforce the boolean result.

### Proof of Concept

1. Deploy the Zcash variant of the contract (`zcash` feature flag).
2. Call `submit_blocks` with a `BlockHeader` where:
   - `prev_block_hash` = current `mainchain_tip_blockhash`
   - `bits` = value returned by `get_config().proof_of_work_limit_bits`
   - `nonce` = any 32-byte value accepted by the equihash parser
   - `solution` = a byte vector of the correct length (1344 bytes for n=200, k=9) filled with zeros — parseable but cryptographically invalid, causing `is_valid_solution` to return `Ok(false)`
3. Observe that the block is accepted: `get_last_block_header()` returns the fraudulent header and `get_last_block_height()` increments.
4. Call `verify_transaction_inclusion_v2` with a Merkle proof against the fraudulent block's `merkle_root` (attacker-chosen) and observe `true` returned for a fabricated transaction. [7](#0-6)
