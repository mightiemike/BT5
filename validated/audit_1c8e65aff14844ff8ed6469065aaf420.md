### Title
Fork Difficulty Validation Uses Main-Chain Boundary Block Instead of Fork Ancestor, Enabling Under-Difficulty Fork Acceptance — (File: `contract/src/bitcoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

When validating the difficulty of a fork block at a retarget boundary, `get_next_work_required` fetches the period's first block via `get_header_by_height`, which always returns the **main-chain** block at that height. If the fork diverged before the difficulty boundary, the fork's actual ancestor at that height is a different block with a different timestamp. Using the main-chain block's timestamp instead of the fork ancestor's timestamp produces an incorrect expected difficulty, which can be exploited to submit fork blocks with less PoW than the protocol requires.

---

### Finding Description

In `bitcoin.rs`, `get_next_work_required` computes the expected difficulty at a retarget boundary by fetching the first block of the 2016-block period:

```rust
// contract/src/bitcoin.rs:81-86
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs:677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This exclusively reads from `mainchain_height_to_header`, which only contains main-chain blocks. Fork blocks are stored only in `headers_pool` via `store_fork_header`:

```rust
// contract/src/lib.rs:665-667
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [3](#0-2) 

When a fork block is being validated at a retarget height, if the fork diverged before the difficulty boundary, the fork's actual ancestor at `first_block_height` is a different block than the main-chain block at that height. The difficulty calculation uses the **main-chain** block's timestamp as `first_block_time`, not the fork's ancestor's timestamp. The two data sources — the fork's `prev_block_header` (correctly taken from the fork chain) and `first_block_time` (incorrectly taken from the main chain) — are never cross-validated against each other.

The same issue exists in `dogecoin.rs`, where the developers themselves flagged the concern with a TODO comment:

```rust
// contract/src/dogecoin.rs:291-295
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [4](#0-3) 

The invariant that must hold is: **the expected difficulty for a fork block at a retarget boundary must be calculated using the fork's actual ancestor at the boundary height, not the main-chain block at that height.** No assertion or cross-check enforces this.

---

### Impact Explanation

If the fork's ancestor at the boundary height has a **later** timestamp than the main-chain block at that height, the `actual_time_taken` computed using the main-chain block's timestamp is artificially larger. A larger `actual_time_taken` produces a higher difficulty target (easier to mine). The attacker sets `header.bits` to the (incorrectly) lower difficulty; `check_target` accepts it because the expected bits match, and the PoW check passes because the block hash satisfies the inflated target.

```rust
// contract/src/bitcoin.rs:23-26
require!(
    expected_bits == block_header.bits,
    "bad-diffbits: incorrect proof of work"
);
