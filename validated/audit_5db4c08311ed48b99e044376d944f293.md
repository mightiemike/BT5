### Title
`SwapAllowlistExtension` checks router address instead of actual user when swaps are routed through `MetricOmmSimpleRouter` - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed by the pool, which is `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks the router's address against the allowlist instead of the real swapper's address. This either permanently breaks swap access for allowlisted users who route through the router, or allows any user to bypass the allowlist entirely by routing through the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool and checks it against `allowedSwapper[pool][sender]`: [1](#0-0) 

The pool passes `msg.sender` as `sender` to `_beforeSwap`, which forwards it unchanged to the extension: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant) executes a swap, it calls `pool.swap(...)` directly, making the router the pool's `msg.sender`: [3](#0-2) 

The extension therefore evaluates `allowedSwapper[pool][router_address]` instead of `allowedSwapper[pool][actual_user]`. The actual user's identity is never consulted.

---

### Impact Explanation

Two mutually exclusive failure modes arise depending on how the pool admin configures the allowlist:

**Mode A — Allowlisted users cannot use the router.**
If the admin allowlists specific user addresses (e.g. `alice`), those users can swap directly against the pool but cannot route through `MetricOmmSimpleRouter`. The router address is not in the allowlist, so every router-mediated swap reverts with `NotAllowedToSwap`. Core swap functionality is broken for the intended user set.

**Mode B — Any user bypasses the allowlist via the router.**
If the admin allowlists the router address to enable router-mediated swaps, the check `allowedSwapper[pool][router]` passes for every caller regardless of who initiated the transaction. Any non-allowlisted user can bypass the gate by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`). The allowlist provides no protection.

Both outcomes have direct fund impact: in Mode A, legitimate LPs cannot attract the intended swap flow; in Mode B, unauthorized actors can drain pool liquidity at oracle-quoted prices, violating the pool's intended access model.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is a standard periphery contract that any user can call. No special privilege is required. A pool admin who deploys `SwapAllowlistExtension` to restrict access to a curated set of market makers or institutional counterparties will encounter this issue the first time any allowlisted user attempts to route through the router, or the first time a non-allowlisted user discovers the bypass. The trigger is a normal public transaction.

---

### Recommendation

The `SwapAllowlistExtension` should gate on the **original transaction initiator** rather than the immediate pool caller. Two approaches:

1. **Pass the original initiator through the router.** The router already stores `msg.sender` in transient storage as the payer. The pool could expose an `initiator` field in the swap call, or the extension could read it from a trusted periphery registry.

2. **Check `tx.origin` as a fallback for router-mediated paths.** This is fragile and generally discouraged, but is sometimes used as a stopgap.

3. **Require direct pool interaction for allowlisted pools.** Document that pools using `SwapAllowlistExtension` must not allowlist the router address and must instruct users to call the pool directly. This is an operational workaround, not a code fix.

The cleanest fix is to thread the original user identity through the hook arguments so the extension can check the economically relevant actor, mirroring how `DepositAllowlistExtension` correctly checks `owner` (the position beneficiary) rather than `sender` (the caller/payer). [4](#0-3) 

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // alice is the only allowed swapper
  bob is NOT allowlisted

Attack (Mode B bypass):
  admin later adds setAllowedToSwap(pool, router, true)
  // to let alice use the router
  bob calls MetricOmmSimpleRouter.exactInputSingle({pool, ...})
  // pool.swap() is called with msg.sender = router
  // extension checks allowedSwapper[pool][router] → true
  // bob's swap succeeds despite not being allowlisted

Broken functionality (Mode A):
  admin does NOT allowlist the router
  alice calls MetricOmmSimpleRouter.exactInputSingle({pool, ...})
  // pool.swap() is called with msg.sender = router
  // extension checks allowedSwapper[pool][router] → false
  // revert NotAllowedToSwap — alice cannot use the router
``` [5](#0-4) [6](#0-5)
