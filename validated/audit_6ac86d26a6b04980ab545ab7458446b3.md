### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery - (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` is explicitly marked deprecated due to a known Merkle proof forgery vulnerability, yet it remains a live, publicly callable NEAR contract method. Any unprivileged caller can invoke it directly and receive a `true` result for a forged transaction inclusion proof, bypassing the security fix introduced in `verify_transaction_inclusion_v2`.

### Finding Description

The contract exposes two transaction inclusion verification endpoints. The original, `verify_transaction_inclusion`, is annotated with `#[deprecated(since = "0.5.0", note = "Use verify_transaction_inclusion_v2 instead.")]` because it is vulnerable to the 64-byte transaction Merkle proof forgery attack documented at https://www.bitmex.com/blog/64-Byte-Transactions. [1](#0-0) 

The function's own docstring acknowledges the broken invariant:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

Despite this, the function is still decorated with `#[pause]` (not `#[private]`) and is part of the public contract ABI. The Rust `#[deprecated]` attribute is a **compiler lint for Rust callers only** — it emits no runtime restriction and does not prevent NEAR RPC callers from invoking the method directly. [3](#0-2) 

The secure replacement, `verify_transaction_inclusion_v2`, adds a mandatory coinbase Merkle proof check before delegating to the deprecated function: [4](#0-3) 

The coinbase check anchors the Merkle tree at a known-valid leaf (the coinbase transaction at index 0), preventing an attacker from substituting an internal node hash as a `tx_id`. The deprecated path performs no such check: [5](#0-4) 

### Impact Explanation

An attacker can craft a 64-byte blob whose double-SHA256 hash equals an internal Merkle tree node of a real block. By supplying this hash as `tx_id` with a corresponding `merkle_proof` path, `verify_transaction_inclusion` returns `true` for a transaction that does not exist in the block. Any downstream contract or application that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing funds, minting tokens, confirming a deposit) will be deceived into accepting a forged proof. The corrupted state is the proof result itself: a `true` return value for a non-existent transaction.

### Likelihood Explanation

The attack requires no privileged role, no leaked keys, and no social engineering. Any NEAR account can call `verify_transaction_inclusion` directly via RPC with a crafted `ProofArgs`. The 64-byte Merkle forgery technique is publicly documented and has known tooling. The only prerequisite is identifying a real block whose Merkle tree has an internal node the attacker can preimage — a condition that is trivially satisfiable for any block with more than one transaction.

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with `#[private]` to prevent external calls. Since `verify_transaction_inclusion_v2` already delegates to it internally via `#[allow(deprecated)]`, the internal call path can be preserved while the external entry point is closed. Alternatively, replace the internal delegation in `verify_transaction_inclusion_v2` with a direct inline call so the deprecated function can be fully removed.

### Proof of Concept

1. Identify any real Bitcoin block `B` with ≥ 2 transactions. Its Merkle tree has internal nodes.
2. Compute the hash of any internal Merkle node `N` in block `B`.
3. Construct a `merkle_proof` path from `N`'s position in the tree up to the Merkle root (this is a standard Merkle sibling path, publicly derivable from the block).
4. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node hash, not a real txid)
   - `tx_block_blockhash` = hash of block `B` (which must be on the contract's main chain)
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the sibling path constructed in step 3
   - `confirmations = 1`
5. The function computes `compute_root_from_merkle_proof(N, index, proof)` which equals the block's `merkle_root`, and returns `true` — falsely asserting that the non-existent transaction `N` is included in block `B`. [5](#0-4)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
