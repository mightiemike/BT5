### Title
`getCoreRisk` Always Returns Long Weight, Causing `existingWeight` to Be Under-Estimated in Spread Health Calculation — (`Clearinghouse.sol` / `BaseEngine.sol`)

---

### Summary

In `Clearinghouse.getHealth`, the spread health adjustment computes `existingWeight` using `longWeight` from both `spotCoreRisk` and `perpCoreRisk`. However, `BaseEngine.getCoreRisk` always returns the **long** weight regardless of the actual position direction, because it hard-codes `1` (positive) as the `amount` argument to `_getWeightX18`. For a spread with a short leg (e.g., long spot + short perp), the short leg's weight should be `shortWeightInitialX18`, not `longWeightInitialX18`. Since `shortWeight ≥ 1 ≥ longWeight` (enforced by product config validation), `existingWeight` is systematically under-estimated, causing the spread health credit to be over-estimated.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` always passes `1` to `_getWeightX18`:** [1](#0-0) 

```solidity
function getCoreRisk(...) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return IProductEngine.CoreRisk(
        amount,
        risk.priceX18,
        RiskHelper._getWeightX18(risk, 1, healthType)  // ← always 1 (positive)
    );
}
```

`_getWeightX18` branches on the sign of `amount`: [2](#0-1) 

Because `1` is always positive, `getCoreRisk` always returns `longWeightInitialX18` or `longWeightMaintenanceX18` into `CoreRisk.longWeight`, regardless of whether the actual position is long or short.

**Incorrect `existingWeight` in spread health calculation:** [3](#0-2) 

```solidity
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(
    perpCoreRisk, spotCoreRisk, healthType
);
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
```

For a **long-spot / short-perp** spread (`spotCoreRisk.amount > 0`, `perpCoreRisk.amount < 0`):

| Field | Code uses | Should use |
|---|---|---|
| `spotCoreRisk.longWeight` | `spotLongWeightInitialX18` | `spotLongWeightInitialX18` ✓ |
| `perpCoreRisk.longWeight` | `perpLongWeightInitialX18` | `perpShortWeightInitialX18` ✗ |

Product config validation enforces `shortWeightMaintenance >= 10^9` and `longWeightMaintenance <= 10^9`: [4](#0-3) 

So `perpShortWeightInitialX18 ≥ 1 ≥ perpLongWeightInitialX18`. The actual `existingWeight` should be **higher** than what is computed. This makes `spreadWeight − existingWeight` **larger** than it should be, inflating the health credit added to the subaccount.

The over-estimation magnitude per unit of basis is:

```
Δ = (spotPrice + perpPrice) × 3(perpShortWeight − perpLongWeight) / 10
```

For typical parameters (e.g., `longWeight = 0.9`, `shortWeight = 1.1`), this is ~6% of the combined position value.

---

### Impact Explanation

Users holding spread positions (long spot + short perp, or short spot + long perp) receive more health credit than the protocol intends. This allows them to maintain positions with less collateral than required by the risk model. If prices move adversely, these positions may become under-collateralized before the protocol can liquidate them, threatening protocol solvency and the insurance fund.

---

### Likelihood Explanation

High. Spread trading (long spot + short perp) is a standard strategy in any cross-margin DEX. Any user who opens such a position automatically benefits from the inflated health credit. No special permissions or unusual conditions are required — the miscalculation is triggered by the ordinary `getHealth` call path used during every health check.

---

### Recommendation

In `BaseEngine.getCoreRisk`, pass the actual `amount` to `_getWeightX18` instead of the hard-coded `1`:

```diff
- RiskHelper._getWeightX18(risk, 1, healthType)
+ RiskHelper._getWeightX18(risk, amount, healthType)
```

This returns `shortWeightInitialX18` when the position is short, so `existingWeight` in `Clearinghouse.getHealth` will correctly reflect the weight already applied to each leg of the spread.

---

### Proof of Concept

1. User deposits collateral into a subaccount.
2. User buys spot token X (long spot, `spotAmount > 0`) and opens a short perp on X (`perpAmount < 0`), forming a spread.
3. `Clearinghouse.getHealth` is called (e.g., during a withdrawal or trade health check).
4. `getCoreRisk` returns `perpLongWeightInitialX18` for the short perp leg instead of `perpShortWeightInitialX18`.
5. `existingWeight = (spotLongWeight + perpLongWeight) / 2` is smaller than the correct `(spotLongWeight + perpShortWeight) / 2`.
6. `spreadWeight − existingWeight` is inflated; the subaccount receives excess health credit proportional to `basisAmount × (spotPrice + perpPrice) × 3(perpShortWeight − perpLongWeight) / 10`.
7. The user can withdraw more collateral or open larger positions than the risk model permits, leaving the protocol exposed to under-collateralized losses.

```text
    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
```
