### Title
Protocol Upgrade Vote Denominator Includes All Block Producers While Numerator Only Counts Those Who Produced Blocks, Skewing the 80% Threshold — (`chain/epoch-manager/src/lib.rs`)

---

### Summary

In `collect_blocks_info`, the protocol-upgrade stake threshold is evaluated as:

> `voted_stake_for_version > (total_block_producer_stake × 4/5)`

The **denominator** (`total_block_producer_stake`) is the sum of every unique block producer's stake for the epoch. The **numerator** (`voted_stake_for_version`) is accumulated only from validators who appear in `version_tracker`, which is populated exclusively when a validator actually produces a block. Block producers who produce **zero blocks** during the epoch are silently included in the denominator but contribute nothing to any version's numerator. This is the exact structural skew from the AuraLocker report: a "total supply" that counts all participants versus an "individual vote" that counts only active participants.

---

### Finding Description

**Denominator — `total_block_producer_stake`** (`chain/epoch-manager/src/lib.rs`, lines 590–598):

```rust
let total_block_producer_stake: Balance = epoch_info
    .block_producers_settlement()
    .iter()
    .copied()
    .collect::<HashSet<_>>()
    .iter()
    .fold(Balance::ZERO, |sum, &id| {
        sum.checked_add(epoch_info.validator_stake(id)).unwrap()
    });
```

This iterates over every unique `ValidatorId` in `block_producers_settlement` — the full set of validators assigned to produce blocks this epoch — and sums their stakes unconditionally.

**Numerator source — `version_tracker`** (`chain/epoch-manager/src/epoch_info_aggregator.rs`, lines 194–198):

```rust
// Step 3: update version tracker
let block_producer_id = epoch_info.sample_block_producer(block_info_height);
self.version_tracker
    .entry(block_producer_id)
    .or_insert_with(|| *block_info.latest_protocol_version());
```

`update_tail` is called once per block that is actually observed. The `entry(...).or_insert_with(...)` pattern means a validator is recorded in `version_tracker` **only when they produce at least one block**. A validator who is assigned slots but produces zero blocks during the epoch never appears in `version_tracker`.

**Threshold check** (`chain/epoch-manager/src/lib.rs`, lines 623–633):

```rust
let next_next_epoch_version = if let Some((version, stake)) =
    versions.into_iter().max_by_key(|&(_version, stake)| stake)
{
    let numer = *config.protocol_upgrade_stake_threshold.numer() as u128;
    let denom = *config.protocol_upgrade_stake_threshold.denom() as u128;
    let threshold = Balance::from_yoctonear(
        (U256::from(total_block_producer_stake.as_yoctonear()) * U256::from(numer)
            / U256::from(denom))
        .as_u128(),
    );
    if stake > threshold { version } else { protocol_version }
```

`stake` here is the sum of stakes of validators who voted for the winning version (only those in `version_tracker`). `threshold` is derived from `total_block_producer_stake`, which includes validators who never voted. The maximum achievable numerator is therefore strictly less than `total_block_producer_stake` whenever any block producer produces zero blocks.

**Exact divergent value**: Any block producer whose `ValidatorId` appears in `block_producers_settlement` but never appears as the return value of `epoch_info.sample_block_producer(h)` for any produced block height `h` in the epoch contributes its full stake to the denominator and zero to every version's numerator.

---

### Impact Explanation

The mainnet `protocol_upgrade_stake_threshold` is `4/5` (80%). If block producers holding collectively more than 20% of `total_block_producer_stake` produce zero blocks in an epoch, the maximum achievable `stake` for any version is less than 80% of `total_block_producer_stake`, and the upgrade is permanently blocked for that epoch regardless of how many other validators vote for it.

Concretely, with total stake = 100 and offline validators holding 25%:
- `total_block_producer_stake` = 100
- Maximum achievable `stake` for new version = 75
- `threshold` = 80
- `75 > 80` → false → upgrade does not proceed

The protocol upgrade is blocked even though 100% of online validators voted for the new version. This can delay a critical security upgrade by at least one full epoch (≈12 hours on mainnet).

---

### Likelihood Explanation

The divergence is structural and present in every epoch. It is triggered whenever any block producer produces zero blocks. This can occur:

1. **Organically**: A validator node crashes or loses connectivity for an entire epoch. With ~100 block producers on mainnet, the probability of at least one being fully offline is non-negligible.
2. **Adversarially**: A staker holding >20% of block-producer stake deliberately goes offline for one epoch to block a specific protocol upgrade, then returns. They are kicked out at epoch end but the upgrade is delayed. The cost is one epoch of missed rewards and a kickout, which may be acceptable to an adversary opposed to a particular upgrade.

The 80% threshold is already high; any non-trivial offline stake pushes the effective required fraction of online validators above 100%, making the upgrade impossible for that epoch.

---

### Recommendation

The denominator should be restricted to the same population as the numerator. Two equivalent fixes:

1. **Restrict the denominator to validators who actually voted**: Replace `total_block_producer_stake` with the sum of stakes of validators present in `version_tracker`:

```rust
let voted_stake: Balance = version_tracker
    .values()
    .map(|_| /* already iterated below */)
    ...
```

More precisely, compute `total_voted_stake` as the sum of `epoch_info.validator_stake(id)` for all `id` in `version_tracker.keys()`, and use that as the denominator.

2. **Alternatively, assign a default version to non-producing block producers**: Before the threshold check, for every `ValidatorId` in `block_producers_settlement` that is absent from `version_tracker`, insert the current `protocol_version` as their vote. This treats silence as a vote for the status quo (analogous to the AuraLocker mitigation of tracking only delegated balances).

---

### Proof of Concept

The divergence is directly readable from two code locations:

**Location 1** — `version_tracker` is populated only on block production: [1](#0-0) 

**Location 2** — `total_block_producer_stake` sums ALL block producers unconditionally: [2](#0-1) 

**Location 3** — threshold is evaluated against the inflated denominator: [3](#0-2) 

Scenario: epoch with 5 block producers, stakes `[30, 25, 20, 15, 10]`, threshold 4/5.
- `total_block_producer_stake` = 100.
- Producer with stake 25 is offline all epoch → absent from `version_tracker`.
- Remaining 4 producers all vote for new version → `stake` = 75.
- `threshold` = 80.
- `75 > 80` → false → upgrade blocked.
- If the denominator were restricted to voters: `threshold` = 75 × 4/5 = 60; `75 > 60` → true → upgrade proceeds.
