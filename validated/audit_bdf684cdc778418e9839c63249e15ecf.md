### Title
Stale `LastKnownBonded` Cache in Tiered Rewards Bonus Calculation Causes Overpayment When Validator Unbonds Before Any Position Exists — (`File: x/tieredrewards/keeper/position.go`)

---

### Summary

`createDelegatedPosition` unconditionally initialises `LastKnownBonded = true` for every new position, regardless of the validator's actual current bonded state. Because the staking hooks only record UNBOND/BOND events when at least one position already exists for a validator (`count > 0`), a validator that unbonded while it had zero positions leaves no event trail. A position created for that validator during its unbonded period therefore starts with a stale cached bonded flag of `true`. When the validator later rebonds, a BOND event is recorded and `processEventsAndClaimBonus` incorrectly treats the entire interval from position-creation to rebond as a bonded, bonus-accruing segment, overpaying bonus rewards from the `RewardsPoolName` module account.

---

### Finding Description

**Root cause 1 — hardcoded `LastKnownBonded = true` at position creation** [1](#0-0) 

`createDelegatedPosition` passes the literal `true` as the `lastKnownBonded` argument to `types.NewPosition` without querying the validator's actual bonded state. A validator that is currently unbonded (jailed, in the unbonding period) will produce a position whose persisted `LastKnownBonded` field is `true`.

**Root cause 2 — hooks skip event recording when no positions exist** [2](#0-1) [3](#0-2) 

`AfterValidatorBeginUnbonding` and `AfterValidatorBonded` both return early when `count == 0`. If a validator unbonds while it has zero tiered-rewards positions, no UNBOND event is appended. The event log for that validator therefore contains no record of the unbonded period that preceded the first position.

**Stale cache used in bonus computation** [4](#0-3) 

`processEventsAndClaimBonus` seeds the replay loop with `bonded := pos.LastKnownBonded`. Because this value is `true` and no UNBOND event exists in the log, the loop reaches the first recorded event — the BOND event emitted when the validator rebonds — and executes the bonus-accrual branch: [5](#0-4) 

This computes bonus for the entire `[position_creation_time, rebond_time]` window using the snapshot `tokensPerShare` captured at rebond time, even though the validator was unbonded throughout that window.

---

### Impact Explanation

The `RewardsPoolName` module account is drained by the spurious bonus payment: [6](#0-5) 

The overpayment magnitude is:

```
shares × tokensPerShare × tier.BonusApy × (rebond_time − creation_time) / SecondsPerYear
```

For a high-APY tier and a long unbonded window this can be a material fraction of the pool. Because the pool also funds legitimate base-reward top-ups for all stakers, draining it harms every delegator on the chain.

---

### Likelihood Explanation

The trigger sequence is realistic and requires no privileged access:

1. A validator is bonded, then jailed or begins unbonding while it has zero tiered-rewards positions (common early in a chain's life or after a validator loses all positions).
2. Any user submits a `MsgLock` (or equivalent commit-delegation message) targeting that validator while it is still unbonded. Cosmos SDK accepts delegations to unbonded validators.
3. The validator is unjailed and rebonds; the BOND hook fires and records one event.
4. The user calls `MsgClaimRewards`; `processEventsAndClaimBonus` overpays bonus for the entire unbonded window.

A sophisticated user can deliberately time step 2 to maximise the unbonded window and therefore the overpayment.

---

### Recommendation

1. **Query the validator's actual bonded state at position creation** and pass it as `lastKnownBonded` instead of the hardcoded `true`:

   ```go
   val, err := k.stakingKeeper.GetValidator(ctx, valAddr)
   if err != nil { return types.Position{}, err }
   lastKnownBonded := val.IsBonded()
   pos := types.NewPosition(..., lastKnownBonded, blockTime)
   ```

2. **Record a synthetic UNBOND event when the first position is created for a currently-unbonded validator**, so the event log accurately reflects the validator's state from the position's perspective.

---

### Proof of Concept

```
Block N:   Validator V bonds. No tiered-rewards positions exist.
Block N+k: Validator V is jailed → AfterValidatorBeginUnbonding fires,
           count == 0 → no UNBOND event recorded.
Block N+m: Alice calls MsgLock for validator V (V is still unbonded).
           createDelegatedPosition sets LastKnownBonded = true,
           lastEventSeq = current_seq (no events in log).
Block N+p: Validator V is unjailed → AfterValidatorBonded fires,
           count == 1 → BOND event appended at seq S with tokensPerShare R.
Block N+q: Alice calls MsgClaimRewards.
           processEventsAndClaimBonus:
             bonded      = true   (stale LastKnownBonded)
             segmentStart = block N+m time
             events       = [BOND at block N+p]
             → bonded==true: computeSegmentBonus(N+m .. N+p, R) → non-zero bonus PAID
             → bonded set true (BOND event)
             → segmentStart = N+p
           After loop: bonded && val.IsBonded() → computeSegmentBonus(N+p .. N+q) PAID
           Alice receives bonus for the entire [N+m, N+p] unbonded window — incorrect.
```
