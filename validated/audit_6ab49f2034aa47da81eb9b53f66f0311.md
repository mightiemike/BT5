### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Allows Draining usdcE from DirectDepositV1 — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw, unchecked `IERC20Base.transferFrom` to pull USDC from the caller before releasing usdcE from the `DirectDepositV1` contract. If the USDC token's `transferFrom` returns `false` instead of reverting, the function silently continues, transferring usdcE to the caller without receiving any USDC in return.

---

### Finding Description

`replaceUsdcEWithUsdc` is a permissionless function (no access modifier beyond a chain-ID gate) that is intended to perform a 1-for-1 swap: the caller provides native USDC and receives bridged USDC.e (usdcE) held in a `DirectDepositV1` contract.

The critical sequence is:

```solidity
// ContractOwner.sol line 616
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);   // ← raw, return value ignored
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));          // transfers usdcE → ContractOwner
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                    // transfers usdcE → caller
``` [1](#0-0) 

The outgoing leg (`safeTransfer` on line 618) correctly uses `ERC20Helper.safeTransfer`, which checks the return value and reverts on failure. [2](#0-1) 

The incoming leg (line 616) calls `IERC20Base.transferFrom` directly on the interface, whose return value is declared as `bool` but is never checked. [3](#0-2) 

`ERC20Helper.safeTransferFrom` exists precisely to handle this case, but is not used here. [4](#0-3) 

---

### Impact Explanation

If the USDC token at the hardcoded address (`0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain) returns `false` on a failed `transferFrom` (e.g., insufficient allowance, insufficient balance, or any non-reverting failure path), the function does not revert. Execution continues:

- `DirectDepositV1.withdraw(usdcE)` transfers the full usdcE balance from the `DirectDepositV1` contract to `ContractOwner`.
- `safeTransfer(usdcE)` then transfers that balance to the caller.

The caller receives usdcE without having provided any USDC. The `DirectDepositV1` subaccount's usdcE balance is fully drained. The intended USDC deposit into `directDepositV1` never occurs, so the subaccount also does not receive the replacement collateral.

**Corrupted asset delta**: usdcE balance of the `DirectDepositV1` contract is reduced to zero; the caller's usdcE balance increases by `balance`; no USDC enters `directDepositV1`.

---

### Likelihood Explanation

The function is callable by any unprivileged address on Ink chain (chainid 57073) — there is no `onlyOwner` or similar guard. [5](#0-4) 

The attacker-controlled entry path is direct: call `replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance granted to `ContractOwner`. Whether the specific USDC deployment at that address reverts or returns `false` on failure determines exploitability. Circle's USDC implementations have historically varied across chains; the absence of a return-value check means the contract is not robust to any non-reverting failure mode. The inconsistency with the `safeTransfer` used on the very next line confirms this is an oversight rather than a deliberate design choice.

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, consistent with how the rest of the codebase handles ERC20 transfers:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already defined in `ERC20Helper` and the `using ERC20Helper for IERC20Base` directive is already active in `ContractOwner`. [6](#0-5) 

---

### Proof of Concept

1. Identify a `DirectDepositV1` contract holding a non-zero usdcE balance (e.g., via `IERC20Base(usdcE).balanceOf(directDepositV1) > 0`).
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an address with **zero** USDC allowance granted to `ContractOwner` (or zero USDC balance).
3. If the USDC token returns `false` instead of reverting: `transferFrom` silently fails, `DirectDepositV1.withdraw(usdcE)` executes, and `safeTransfer(usdcE, msg.sender, balance)` sends the full usdcE balance to the attacker.
4. The `DirectDepositV1` subaccount loses its entire usdcE collateral; the attacker gains it for free.

        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

    function safeTransfer(
        IERC20Base self,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(IERC20Base.transfer.selector, to, amount)
        );
        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
