### Title
Wrong `MAX_FUTURE_BLOCK_TIME_MTP` Constant Causes Rejection of Valid Zcash Blocks - (`btc-types/src/network.rs`)

### Summary
`MAX_FUTURE_BLOCK_TIME_MTP` is defined as `90 * 60 = 5,400 seconds` (90 minutes), but the Zcash protocol requires this value to be `129,600 seconds` (36 hours). The code's own inline comment in `contract/src/zcash.rs` explicitly acknowledges the correct value, making this a direct analog of M-4: a constant whose numeric value is set to a unit-mismatched, far-too-small quantity.

### Finding Description

In `btc-types/src/network.rs`, two future-timestamp guard constants are defined:

```rust
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60;       // = 5,400 s  (90 min)
pub const MAX_FUTURE_BLOCK_TIME_LOCAL: u32 = 2 * 60 * 60; // = 7,200 s  (2 h)
``` [1](#0-0) 

`MAX_FUTURE_BLOCK_TIME_MTP` is consumed exclusively by the Zcash `check_pow` path in `contract/src/zcash.rs`:

```rust
// MAX_FUTURE_BLOCK_TIME_MTP is typically 129600 seconds (36 hours) in Zcash
require!(
    block_header.time
        <= next_work_result.prev_block_median_time_past + MAX_FUTURE_BLOCK_TIME_MTP,
    "time-too-far-ahead-of-mtp: block timestamp is too far ahead of median-time-past"
);
``` [2](#0-1) 

The comment is unambiguous: the correct Zcash value is **129,600 seconds (36 hours)**. The stored constant is **5,400 seconds (90 minutes)** — a factor of 24× too small. Bitcoin and Litecoin never import `MAX_FUTURE_BLOCK_TIME_MTP`; only the Zcash feature flag path uses it. [3](#0-2) [4](#0-3) 

### Impact Explanation

The Zcash consensus rules permit a block timestamp to be up to 36 hours ahead of the Median Time Past (MTP) of the previous block. With the constant set to 90 minutes, any valid Zcash block whose timestamp falls in the window `(MTP + 5400, MTP + 129600]` seconds is unconditionally rejected by the light client with `"time-too-far-ahead-of-mtp"`. This corrupts the contract's canonical chain mapping: the light client diverges from the real Zcash chain, making `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` return false negatives for transactions in those blocks. SPV proofs for legitimate Zcash transactions become permanently unverifiable on-chain. [5](#0-4) 

### Likelihood Explanation

The Zcash protocol explicitly allows timestamps up to 36 hours ahead of MTP. During periods of low hash rate, network disruption, or deliberate timestamp advancement by miners (all valid under consensus), blocks with timestamps more than 90 minutes ahead of MTP will appear on the canonical chain. The relayer will attempt to submit them via `submit_blocks()`, and the contract will reject them. No privileged access is required; any unprivileged relayer or proof submitter triggers the broken invariant. [6](#0-5) 

### Recommendation

Change the constant in `btc-types/src/network.rs` to match the Zcash protocol specification:

```rust
// Before (wrong):
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 90 * 60; // 5,400 s — Bitcoin value, not Zcash

// After (correct):
pub const MAX_FUTURE_BLOCK_TIME_MTP: u32 = 36 * 60 * 60; // 129,600 s — Zcash protocol value
``` [7](#0-6) 

Alternatively, define a separate `ZCASH_MAX_FUTURE_BLOCK_TIME_MTP` constant and use it only in the Zcash module to avoid any future confusion with the Bitcoin/Litecoin 90-minute value.

### Proof of Concept

1. Deploy the Zcash-feature build of the contract on NEAR testnet.
2. Construct a syntactically valid Zcash block header where `block_header.time = prev_block_median_time_past + 7200` (2 hours ahead of MTP — valid per Zcash consensus, invalid per this contract).
3. Call `submit_blocks()` with that header as an unprivileged relayer.
4. The contract panics with `"time-too-far-ahead-of-mtp"` even though the block is valid on the real Zcash network.
5. Subsequent calls to `verify_transaction_inclusion_v2` for any transaction in that block return false, demonstrating the broken canonical-chain mapping. [8](#0-7) [7](#0-6)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
