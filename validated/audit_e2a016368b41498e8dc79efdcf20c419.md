### Title
Linked Signer Can Replace Itself via `LinkSigner`, Enabling Persistent Subaccount Takeover — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction type is processed with `allowLinkedSigner = true`, meaning the currently active linked signer — a delegated hot-wallet key — can sign a transaction that replaces itself with any attacker-controlled address. This is the Solidity analog of the iOS Keychain bug: a weaker credential (`linkedSigner`, analogous to `AfterFirstUnlockThisDeviceOnly`) is permitted to perform a sensitive privileged mutation (changing the linked signer) that should require the stronger credential (the subaccount owner's key, analogous to `kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly`).

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted signer when `allowLinkedSigner` is `true`: [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if it recovers to either the subaccount owner address **or** the linked signer: [3](#0-2) 

The result is that the linked signer can sign a `LinkSigner` transaction to overwrite `linkedSigners[sender]` with any address it chooses: [4](#0-3) 

The linked signer is a hot-wallet key intended for delegated trading. It is not the subaccount owner. Allowing it to mutate the `linkedSigners` mapping is an over-permissive access control: the weaker credential can perform a privileged operation that should be restricted to the stronger credential (the owner key embedded in the subaccount's `bytes32` address).

---

### Impact Explanation

Once an attacker obtains a linked signer key (e.g., from a compromised trading bot or leaked API key), the attack chain is:

1. Attacker signs a `LinkSigner` transaction using the compromised linked signer key, setting `signer` to an attacker-controlled address.
2. The sequencer processes this transaction, overwriting `linkedSigners[victim]` with the attacker's address.
3. Even if the victim attempts to revoke by submitting their own `LinkSigner` from the owner key, the attacker can re-submit from the new linked signer, creating a persistent race.
4. The attacker's new linked signer signs `WithdrawCollateral` or `TransferQuote` transactions (both also accept `allowLinkedSigner = true`) to drain the subaccount's collateral. [5](#0-4) 

The corrupted state is `linkedSigners[subaccount]` in `EndpointStorage`, and the downstream asset delta is the full collateral balance of the victim subaccount across all spot products. [6](#0-5) 

---

### Likelihood Explanation

Linked signers are explicitly designed for use by automated trading systems and hot wallets. Key compromise of a hot wallet is a realistic and common threat. The attack requires no on-chain privileges, no admin access, and no sequencer compromise — only possession of the linked signer's private key. The nonce mechanism prevents replay but does not prevent a fresh `LinkSigner` transaction signed by the compromised key.

**Impact: 3 | Likelihood: 3**

---

### Recommendation

`LinkSigner` is a privileged operation that changes the signing authority of a subaccount. It should only be executable by the subaccount owner (the address embedded in the first 20 bytes of the `bytes32 sender`), never by the currently linked signer. Change the `allowLinkedSigner` argument to `false` for the `LinkSigner` transaction type:

```solidity
// In EndpointTx.processTransactionImpl, LinkSigner branch:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // was: true — owner key required to change linked signer
);
```

This mirrors the iOS recommendation: use the strictest access attribute (`kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly`) for the most sensitive operations, not a weaker one.

---

### Proof of Concept

1. Alice sets `linkedSigners[alice_subaccount] = hot_wallet` via a normal `LinkSigner` signed by her owner key.
2. Attacker steals `hot_wallet` private key.
3. Attacker constructs `LinkSigner { sender: alice_subaccount, signer: attacker_address, nonce: current_nonce }`.
4. Attacker signs with `hot_wallet` key — accepted because `allowLinkedSigner = true` in the `LinkSigner` branch.
5. Sequencer processes the transaction; `linkedSigners[alice_subaccount]` is now `attacker_address`.
6. Attacker signs `WithdrawCollateral { sender: alice_subaccount, productId: USDC, amount: full_balance }` with `attacker_address`.
7. Funds are transferred to `attacker_address`. [1](#0-0) [5](#0-4)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
