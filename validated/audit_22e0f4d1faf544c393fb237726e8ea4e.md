### Title
Liability Liquidation Can Be Blocked By Depositing Spot Tokens To Liquidatee's Account — (File: `core/contracts/Endpoint.sol`, `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` accepts an arbitrary `bytes32 subaccount` target with no approval required from the recipient. An attacker can use this to deposit a non-quote spot asset into any liquidatee's subaccount. Once the sequencer processes the resulting slow-mode transaction, the liquidatee holds a positive spot balance, causing `_assertCanLiquidateLiability` in `ClearinghouseLiq.sol` to revert and permanently blocking liability liquidation for that account.

---

### Finding Description

**Unauthorized deposit to arbitrary subaccount:**

`depositCollateralWithReferral` in `Endpoint.sol` takes a caller-supplied `bytes32 subaccount` as the deposit target. Tokens are pulled from `msg.sender`, not from the subaccount owner. The only guards are an isolated-subaccount check and a sanctions check — there is no requirement that `msg.sender` owns or has been approved by the target subaccount. [1](#0-0) 

This means any unprivileged caller can deposit any listed spot token into any non-isolated subaccount by simply holding the token and calling this function.

**The blocking check in `_assertCanLiquidateLiability`:**

When a liquidator attempts to liquidate a liability position (`txn.amount < 0`, spread or spot engine), `liquidateSubaccountImpl` calls `_assertCanLiquidateLiability`. This function iterates all spot products and requires that the liquidatee holds **no positive spot balances**: [2](#0-1) 

If the liquidatee has even a dust-level positive balance in any weighted spot product, the call reverts with `ERR_NOT_LIQUIDATABLE_LIABILITIES`.

**Call site in `liquidateSubaccountImpl`:** [3](#0-2) 

**Sequencer can execute slow-mode transactions immediately:**

Deposits via `depositCollateralWithReferral` are queued as slow-mode transactions with a nominal 3-day delay. However, the sequencer bypasses this delay by submitting an `ExecuteSlowMode` transaction type, which calls `_executeSlowModeTransaction` with `fromSequencer = true`, skipping the `executableAt` guard: [4](#0-3) [5](#0-4) 

The sequencer routinely includes `ExecuteSlowMode` transactions in its batches as part of normal operation, meaning the deposit can be credited to the liquidatee's account within the same sequencer batch window as the liquidation attempt.

---

### Impact Explanation

An attacker can permanently block liability liquidation for any targeted subaccount by depositing a minimal amount of any weighted spot token to it. The liquidatee remains under maintenance health (deeply insolvent) but cannot be liquidated via the liability path. This causes the protocol to accumulate bad debt, as the underwater account cannot be resolved. The attacker's cost is only the minimum deposit amount of a spot token.

---

### Likelihood Explanation

The attack requires:
1. The attacker holds any listed spot token (e.g., wBTC, wETH).
2. The attacker approves `Endpoint` and calls `depositCollateralWithReferral` targeting the liquidatee's `bytes32` subaccount (publicly observable on-chain).
3. The sequencer processes the slow-mode deposit in its next batch (standard behavior).

No privileged access, no leaked keys, no governance capture. The liquidatee's subaccount identifier is fully public. The attack is cheap (dust deposit), repeatable, and can be re-applied each time the liquidatee's spot balance is cleared.

---

### Recommendation

Require explicit approval from the subaccount owner before crediting a deposit to a third-party subaccount. Specifically, `depositCollateralWithReferral` should verify that `msg.sender` is the address encoded in the first 20 bytes of `subaccount`, or that the subaccount owner has pre-approved the deposit (analogous to the fix applied for Options assets in the referenced prior audit). This mirrors the fix recommended in the original report: extend the approval system to cover all deposit paths that can target arbitrary recipient accounts.

---

### Proof of Concept

1. Observe that `victim` subaccount is under maintenance health with a large negative USDC balance and no positive spot balances — eligible for liability liquidation.
2. Attacker calls:
   ```solidity
   endpoint.depositCollateralWithReferral(
       victim,          // bytes32 subaccount — no approval required
       wbtcProductId,   // any weighted spot product
       minDepositAmount,
       ""
   );
   ``` [6](#0-5) 
3. Sequencer includes an `ExecuteSlowMode` transaction in its next batch; the deposit is credited immediately, giving `victim` a positive wBTC balance.
4. Liquidator submits `LiquidateSubaccount` with `amount < 0` targeting `victim`'s USDC liability.
5. `liquidateSubaccountImpl` calls `_assertCanLiquidateLiability`, which iterates spot products and hits:
   ```solidity
   require(balance.amount <= 0, ERR_NOT_LIQUIDATABLE_LIABILITIES);
   ```
   for the wBTC product — reverts. [7](#0-6) 
6. Liquidation is blocked. Attacker repeats the deposit whenever the balance is cleared, indefinitely preventing resolution of the insolvent account.
