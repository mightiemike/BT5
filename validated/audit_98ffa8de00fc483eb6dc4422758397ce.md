### Title
`ValidResourceBounds` protobuf deserialization silently mutates `AllResources` to `L1Gas` when l2/l1_data gas are zero, producing a divergent transaction hash domain - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to select the enum variant. When a V3 transaction carries `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the deserializer silently returns `ValidResourceBounds::L1Gas(l1_gas)` instead of `ValidResourceBounds::AllResources(...)`. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant (2 for `L1Gas`, 3 for `AllResources`), the transaction hash computed after protobuf round-trip diverges from the hash computed at ingestion time and from the hash the Starknet OS always computes for V3 transactions.

---

### Finding Description

**The two hash domains**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` (lines 188–211) builds the fee-fields hash differently per variant:

```rust
// L1Gas → 2 resource felts: tip | L1_GAS | L2_GAS
// AllResources → 3 resource felts: tip | L1_GAS | L2_GAS | L1_DATA_GAS
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [1](#0-0) 

The Starknet OS Cairo program enforces exactly 3 resource bounds for every V3 transaction (`assert n_resource_bounds = 3`), so it always produces the `AllResources` hash. [2](#0-1) 

**The broken protobuf deserializer**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` (lines 417–436) selects the variant by inspecting field values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← 2-resource hash domain
} else {
    ValidResourceBounds::AllResources(...)       // ← 3-resource hash domain
})
``` [3](#0-2) 

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471–490) always emits all three fields, including `l1_data_gas = 0` for the `L1Gas` variant:

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),
    l1_data_gas: Some(ResourceBounds::default().into()),  // zero
},
``` [4](#0-3) 

A V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` therefore round-trips as `L1Gas(l1_gas)`. The codebase itself acknowledges this in the consensus test:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);  // workaround
}
``` [5](#0-4) 

The workaround is applied only in the test; production code has no such guard.

**Ingestion path always uses `AllResources`**

`RpcInvokeTransactionV3.resource_bounds` is typed as `AllResourceBounds` (not `ValidResourceBounds`), so the gateway always computes the hash with the 3-resource domain:

```rust
pub struct RpcInvokeTransactionV3 {
    pub resource_bounds: AllResourceBounds,  // always AllResources
    ...
}
``` [6](#0-5) 

`convert_rpc_tx_to_internal` computes `tx_hash` from `InternalRpcTransactionWithoutTxHash`, which wraps `AllResourceBounds` in `ValidResourceBounds::AllResources(...)` before hashing:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [7](#0-6) 

**State-sync path uses `ValidResourceBounds` and is affected**

The state-sync path propagates committed blocks as `Transaction::Invoke(InvokeTransaction::V3(InvokeTransactionV3))`, where `InvokeTransactionV3.resource_bounds: ValidResourceBounds`. Deserialization of this type from protobuf goes through the broken converter, silently changing the variant and therefore the hash domain. [8](#0-7) 

**Secondary effect: wrong `GasVectorComputationMode`**

`get_gas_vector_computation_mode` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. This changes which resource bounds are validated during fee pre-validation and which gas vector is used for fee computation:

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_) => GasVectorComputationMode::NoL2Gas,
    }
}
``` [9](#0-8) 

---

### Impact Explanation

A syncing node that receives a block containing a V3 transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` via the state-sync protobuf path will:

1. Deserialize the transaction as `ValidResourceBounds::L1Gas(l1_gas)`.
2. Recompute the transaction hash using the 2-resource domain → hash H_l1.
3. The stored/canonical hash is H_all (3-resource domain, as computed by the gateway and the OS).
4. H_l1 ≠ H_all: hash verification fails, or the wrong hash is used in receipts/events.
5. Re-execution in the prover uses `GasVectorComputationMode::NoL2Gas` instead of `All`, producing wrong fee accounting and potentially wrong execution results.

This matches the impact category: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input** (Critical) and **RPC execution, fee estimation, tracing, simulation returns an authoritative-looking wrong value** (High).

---

### Likelihood Explanation

- Any unprivileged user can submit a valid V3 transaction with only L1 gas bounds set (l2_gas = 0, l1_data_gas = 0). The gateway's stateless validator accepts this.
- State sync is a normal, continuous operation on every non-sequencer node.
- No special attacker capability is required beyond submitting a standard V3 transaction.

---

### Recommendation

Remove the value-based heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Options:

1. **Add a discriminant field** to the protobuf `ResourceBounds` message (e.g., `bool is_all_resources`) so the variant is encoded explicitly rather than inferred from values.
2. **Always return `AllResources`** for V3 transactions in the state-sync path, since V3 transactions always carry all three resource bounds by protocol definition.
3. **Align with the serde path**: the JSON/serde deserializer correctly preserves `AllResources` when `L1DataGas` key is present (even with zero value). The protobuf path should apply the same logic — presence of the `l1_data_gas` field (even zero) should select `AllResources`.

---

### Proof of Concept

```
1. Submit V3 invoke transaction to the gateway:
   resource_bounds = {
     l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
     l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
     l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 }
   }

2. Gateway accepts it. Hash H_all is computed via:
   get_tip_resource_bounds_hash(AllResources { l1_gas: 1000, l2_gas: 0, l1_data_gas: 0 }, tip)
   → poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0), concat(L1_DATA, 0, 0))
   → H_all

3. Block is committed. State-sync serializes the transaction to protobuf:
   ResourceBounds { l1_gas: (1000,1), l2_gas: (0,0), l1_data_gas: (0,0) }

4. Syncing node deserializes via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → returns ValidResourceBounds::L1Gas(ResourceBounds { max_amount: 1000, max_price_per_unit: 1 })

5. Syncing node recomputes hash:
   get_tip_resource_bounds_hash(L1Gas { max_amount: 1000, max_price_per_unit: 1 }, tip)
   → poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0))   ← only 2 resource felts
   → H_l1

6. H_l1 ≠ H_all: the syncing node holds the transaction with a wrong hash domain.
   The Starknet OS (which always uses 3 resource bounds) computes H_all,
   so any proof generated from the syncing node's state is invalid.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
