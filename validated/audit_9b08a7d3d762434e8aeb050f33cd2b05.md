### Title
Missing Post-Multiplication `check_cost` in `op_unknown` Allows Cost Budget Overrun — (`src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` validates the pre-multiplication cost against `max_cost`, then multiplies by `(cost_multiplier + 1)` derived from attacker-controlled opcode bytes, but never re-validates the final cost against `max_cost`. The function can return `Ok(Reduction(cost, nil))` with a `cost` that silently exceeds the caller-supplied budget, directly analogous to an ERC20 `transfer` returning `false` instead of reverting while the caller treats it as success.

### Finding Description

In `op_unknown` (`src/more_ops.rs`, lines 160–267), the cost enforcement sequence is:

```rust
// line 260 — checks pre-multiplication cost only
check_cost(cost, max_cost)?;
// line 261 — multiplies by attacker-controlled value (up to u32::MAX + 1)
cost *= cost_multiplier + 1;
// line 262-265 — only checks against u32::MAX, NOT against max_cost
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode atom bytes via `u32_from_u8(&op[0..op.len() - 1])`, which is entirely attacker-controlled CLVM input. [1](#0-0) 

The pre-multiplication `check_cost` at line 260 ensures `cost ≤ max_cost` at that moment. After the multiplication at line 261, `cost` can be anywhere in the range `(max_cost, u32::MAX]`. The only guard is `cost > u32::MAX`, which is a completely different bound. No `check_cost(cost, max_cost)` is performed after the multiplication. [2](#0-1) 

The `cost_function` field (bits 7–6 of the last opcode byte) selects one of four cost formulas (constant, add-like, mul-like, concat-like), each of which accumulates a pre-multiplication cost that is checked against `max_cost`. The multiplier then scales this cost by up to `u32::MAX + 1 = 4,294,967,296`. [3](#0-2) 

The returned `Reduction(cost, nil)` is consumed by `apply_op` in `src/run_program.rs`, which pushes the result and returns the cost to the main execution loop without any additional budget re-check at that layer: [4](#0-3) 

```rust
let r = self.dialect.op(
    self.allocator, operator, operand_list, max_cost, current_extensions,
)?;
self.push(r.1)?;
Ok(r.0)   // cost returned without re-checking against max_cost
```

### Impact Explanation

An attacker-crafted CLVM program running in consensus (lenient) mode — where `allow_unknown_ops()` returns `true` — can include an unknown opcode whose bytes encode a large `cost_multiplier`. The function returns a `Reduction` whose cost field exceeds `max_cost` without emitting `Err(EvalErr::CostExceeded)`. The run_program loop accumulates this over-budget cost into the running total. Depending on whether the loop re-checks the accumulated total (which `apply_op` does not do inline), execution may proceed past the declared budget. This enables:

- **Undercharged execution**: a program consumes more computational resources than the declared `max_cost` allows, breaking the fee/resource model.
- **Consensus divergence**: nodes enforcing different `max_cost` limits may disagree on whether a transaction is valid, since `op_unknown` silently absorbs the overage rather than returning a deterministic `CostExceeded` error.

### Likelihood Explanation

Unknown opcodes are reachable in any consensus-mode execution (non-mempool, `allow_unknown_ops() = true`). The attacker controls the full opcode byte sequence, making `cost_multiplier` directly attacker-controlled. Crafting a triggering input requires only choosing an opcode whose last byte encodes a non-zero `cost_function` (so the pre-multiplication cost is non-trivial) and whose leading bytes encode a multiplier large enough to push the final cost above `max_cost` but below `u32::MAX`. This is straightforward to construct from raw CLVM bytes.

### Recommendation

Add a final `check_cost` call after the multiplication, before returning:

```rust
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    check_cost(cost, max_cost)?;   // ← add this
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

This mirrors the pattern used by every other operator in `src/more_ops.rs`, which calls `check_cost` after each cost-accumulating step. [2](#0-1) 

### Proof of Concept

Construct a CLVM program in consensus mode with an unknown opcode whose bytes are, for example, `[0x00, 0x00, 0x01, 0x40]`:

- Last byte `0x40` → `cost_function = 1` (add-like), lower 6 bits ignored.
- Leading bytes `[0x00, 0x00, 0x01]` → `cost_multiplier = 1` (after `u32_from_u8`).

With `max_cost = 500`, the add-like formula accumulates a pre-multiplication cost of, say, 419 (base + per-arg). `check_cost(419, 500)` passes. After `cost *= 2`, cost becomes 838, which exceeds `max_cost = 500` but is far below `u32::MAX`. The function returns `Ok(Reduction(838, nil))` instead of `Err(CostExceeded)`. [5](#0-4)

pub fn op_unknown(
    allocator: &mut Allocator,
    o: NodePtr,
    mut args: NodePtr,
    max_cost: Cost,
) -> Response {
    // unknown opcode in lenient mode
    // unknown ops are reserved if they start with 0xffff
    // otherwise, unknown ops are no-ops, but they have costs. The cost is computed
    // like this:

    // byte index (reverse):
    // | 4 | 3 | 2 | 1 | 0          |
    // +---+---+---+---+------------+
    // | multiplier    |XX | XXXXXX |
    // +---+---+---+---+---+--------+
    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // 1 is always added to the multiplier before using it to multiply the cost, this
    // is since cost may not be 0.

    // cost_function is 2 bits and defines how cost is computed based on arguments:
    // 0: constant, cost is 1 * (multiplier + 1)
    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // this means that unknown ops where cost_function is 1, 2, or 3, may still be
    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();

    if op.is_empty() || (op.len() >= 2 && op[0] == 0xff && op[1] == 0xff) {
        Err(EvalErr::Reserved(o))?;
    }

    let cost_function = (op[op.len() - 1] & 0b11000000) >> 6;
    let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
        Some(v) => v as u64,
        None => {
            return Err(EvalErr::Invalid(o))?;
        }
    };

    let mut cost = match cost_function {
        0 => 1,
        1 => {
            let mut cost = ARITH_BASE_COST;
            let mut byte_count: u64 = 0;
            while let Some((arg, rest)) = allocator.next(args) {
                args = rest;
                cost += ARITH_COST_PER_ARG;
                let len = atom_len(allocator, arg, "unknown op")?;
                byte_count += len as u64;
                check_cost(cost + (byte_count as Cost * ARITH_COST_PER_BYTE), max_cost)?;
            }
            cost + (byte_count * ARITH_COST_PER_BYTE)
        }
        2 => {
            let mut cost = MUL_BASE_COST;
            let mut first_iter: bool = true;
            let mut l0: u64 = 0;
            while let Some((arg, rest)) = allocator.next(args) {
                args = rest;
                let len = atom_len(allocator, arg, "unknown op")?;
                if first_iter {
                    l0 = len as u64;
                    first_iter = false;
                    continue;
                }
                let l1 = len as u64;
                cost += MUL_COST_PER_OP;
                cost += (l0 + l1) * MUL_LINEAR_COST_PER_BYTE;
                cost += (l0 * l1) / MUL_SQUARE_COST_PER_BYTE_DIVIDER;
                l0 += l1;
                check_cost(cost, max_cost)?;
            }
            cost
        }
        3 => {
            let mut cost = CONCAT_BASE_COST;
            while let Some((arg, rest)) = allocator.next(args) {
                args = rest;
                let len = atom_len(allocator, arg, "unknown op")?;
                cost += CONCAT_COST_PER_ARG;
                cost += CONCAT_COST_PER_BYTE * (len as Cost);
                check_cost(cost, max_cost)?;
            }
            cost
        }
        _ => 1,
    };

    assert!(cost > 0);

    check_cost(cost, max_cost)?;
    cost *= cost_multiplier + 1;
    if cost > u32::MAX as u64 {
        Err(EvalErr::Invalid(o))?
    } else {
        Ok(Reduction(cost as Cost, allocator.nil()))
    }
}
