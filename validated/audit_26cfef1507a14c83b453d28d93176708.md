### Title
`deser_auto` silently drops `max_atom_len` enforcement for legacy/backrefs-format blobs — (File: `wheel/src/api.rs`)

---

### Summary

`deser_auto` accepts a `max_atom_len` parameter that callers rely on to cap atom sizes, but the limit is only wired into the serde_2026 deserialization path. When the input blob does not carry the serde_2026 magic prefix, the function silently falls through to `node_from_bytes_backrefs`, which receives no atom-length bound at all. An attacker who controls the serialized blob can therefore bypass the caller-specified constraint entirely by presenting a legacy or backrefs-format payload.

---

### Finding Description

`deser_auto` in `wheel/src/api.rs` is the recommended Python-facing auto-detecting deserializer. Its signature advertises `max_atom_len` (defaulting to `PY_DEFAULT_MAX_ATOM_LEN = 1 << 20`, i.e. 1 MiB) as a hard cap on any single atom:

```rust
#[pyfunction]
#[pyo3(signature = (blob, *, max_atom_len=PY_DEFAULT_MAX_ATOM_LEN, strict=true))]
fn deser_auto(blob: &[u8], max_atom_len: usize, strict: bool) -> PyResult<LazyNode> {
    let mut a = Allocator::new();
    let node = if let Some(body) = blob.strip_prefix(SERDE_2026_MAGIC_PREFIX.as_slice()) {
        deserialize_2026_body_from_stream(&mut a, &mut Cursor::new(body), max_atom_len, strict)
            .map_err(eval_to_py)?
    } else {
        node_from_bytes_backrefs(&mut a, blob).map_err(eval_to_py)?   // ← max_atom_len dropped
    };
    Ok(LazyNode::new(Rc::new(a), node))
}
``` [1](#0-0) 

`deserialize_2026_body_from_stream` enforces `max_atom_len` via `checked_bounded_usize`:

```rust
let length = checked_bounded_usize(-length_val, max_atom_len)?;
``` [2](#0-1) 

`node_from_bytes_backrefs`, by contrast, has no such parameter and delegates directly to `parse_atom` with no size bound:

```rust
pub fn node_from_bytes_backrefs(allocator: &mut Allocator, b: &[u8]) -> Result<NodePtr> {
    let mut buffer = Cursor::new(b);
    node_from_stream_backrefs(allocator, &mut buffer, |_node| {})
}
``` [3](#0-2) 

The only backstop is the raw `Allocator` heap limit, which `Allocator::new()` sets to `u32::MAX` (≈ 4 GiB):

```rust
pub fn new() -> Self {
    Self::new_limited(u32::MAX as usize)
}
``` [4](#0-3) 

The Python-level `deserialize()` wrapper in `serde.py` propagates `max_atom_len` only for `"2026"` and `"auto"` format strings, giving callers the false impression that the limit is always active:

```python
if fmt in ("2026", "auto"):
    kwargs: dict = {"strict": strict}
    if max_atom_len is not None:
        kwargs["max_atom_len"] = max_atom_len
    return fn(blob, **kwargs)
return fn(blob)
``` [5](#0-4) 

Even when `max_atom_len` is forwarded to `deser_auto`, the Rust function ignores it for any blob that lacks the serde_2026 magic prefix.

---

### Impact Explanation

A caller that invokes `deser_auto(blob, max_atom_len=X)` (or `deserialize(blob, fmt="auto", max_atom_len=X)`) believes it is protected against oversized atoms. An attacker who supplies a legacy or backrefs-encoded blob containing a single atom of arbitrary size (up to ~4 GiB) bypasses that protection entirely. The `Allocator` will attempt to satisfy the allocation, causing unbounded heap growth and potential process-level memory exhaustion. Because `deser_auto` is the recommended entry point for untrusted blobs, any downstream component that relies on `max_atom_len` for resource control is silently unprotected when the input is not serde_2026.

---

### Likelihood Explanation

High. The legacy and backrefs wire formats are the dominant formats in the existing Chia ecosystem. An attacker submitting a crafted blob to any service that calls `deser_auto` (or `deserialize(..., fmt="auto")`) with a `max_atom_len` guard needs only to omit the six-byte serde_2026 magic prefix. No cryptographic material, privileged access, or social engineering is required. The `PY_DEFAULT_MAX_ATOM_LEN` constant and the `max_atom_len` keyword argument are prominently documented, so callers will naturally assume the limit is universally enforced.

---

### Recommendation

Add a `max_atom_len` parameter to `node_from_bytes_backrefs` (and its underlying `node_from_stream_backrefs` / `parse_atom` chain) and thread it through from `deser_auto`'s `else` branch. The serde_2026 path already demonstrates the correct pattern: `checked_bounded_usize(length_val, max_atom_len)` at the point of atom allocation. Apply the same guard inside `parse_atom` so that both deserialization paths honour the caller's constraint uniformly.

---

### Proof of Concept

```python
from clvm_rs.serde import deser_auto

# Craft a legacy-format blob: 0x8f = atom of length 15 bytes, followed by 15 zero bytes.
# Replace with a multi-megabyte atom header to exceed max_atom_len=1024.
# Legacy atom encoding: leading byte 0x8f means 15-byte atom.
# For a 2 MiB atom: header bytes encode length 2*1024*1024, then 2 MiB of zeros.
import struct

atom_len = 2 * 1024 * 1024          # 2 MiB — well above max_atom_len=1024
# Legacy encoding: lengths 0x40..0x7ff use two bytes: 0x80|(hi), lo
header = bytes([0x80 | (atom_len >> 8), atom_len & 0xFF])  # simplified; real encoding varies
blob = header + b'\x00' * atom_len

# Caller expects max_atom_len=1024 to be enforced — it is NOT for this format
node = deser_auto(blob, max_atom_len=1024)   # succeeds; 2 MiB atom allocated
```

The call returns successfully with a 2 MiB atom despite `max_atom_len=1024`, because `node_from_bytes_backrefs` receives no bound. Scaling `atom_len` to hundreds of megabytes drives the process toward OOM.
