### Title
Publicly Callable `verify_transaction_inclusion` Accepts Internal Merkle Node as `tx_id`, Enabling 64-Byte Forgery Attack — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` remains a fully callable public NEAR contract method despite its `#[deprecated]` annotation. Any unprivileged account can call it directly with a `ProofArgs` where `tx_id` is an internal Merkle tree node hash, bypassing the coinbase proof validation that `verify_transaction_inclusion_v2` enforces. `compute_root_from_merkle_proof` treats the supplied hash as a leaf without any validation, so a crafted proof starting from an internal node walks correctly to the real Merkle root and the function returns `true`.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub fn` inside a `#[near]` impl block: [1](#0-0) 

The `#[deprecated]` Rust attribute is a **compile-time lint only**. It generates a compiler warning for callers in Rust source but does **not** remove the function from the compiled WASM export. On NEAR, every `pub fn` in a `#[near]` impl block is exported as a callable contract method regardless of deprecation status.

The only runtime gate is `#[pause]`, which blocks calls only when a `PauseManager` has explicitly paused the contract. There is no `#[private]`, no role check, and no `trusted_relayer` guard: [2](#0-1) 

Inside the function, `compute_root_from_merkle_proof` is called with the attacker-supplied `args.tx_id` as the starting hash: [3](#0-2) 

`compute_root_from_merkle_proof` performs no validation that the input is a leaf transaction — it simply iterates the proof and hashes upward: [4](#0-3) 

The v2 method enforces a coinbase proof check before delegating to v1: [5](#0-4) 

But this guard is entirely skippable because v1 is independently reachable.

**Concrete attack for a 4-transaction block `[T0, T1, T2, T3]`:**

```
Level 0 (leaves): H(T0), H(T1), H(T2), H(T3)
Level 1 (internal): N01 = SHA256d(H(T0)||H(T1)),  N23 = SHA256d(H(T2)||H(T3))
Level 2 (root):     R   = SHA256d(N01||N23)
```

Attacker calls `verify_transaction_inclusion` with:
- `tx_id = N01` (an internal node, not a real transaction)
- `tx_index = 0` (even, so it is placed on the left)
- `merkle_proof = [N23]`

`compute_root_from_merkle_proof` computes `SHA256d(N01 || N23) = R`, which equals `header.block_header.merkle_root`, so the function returns `true`.

The code's own documentation acknowledges this: [6](#0-5) 

---

### Impact Explanation

Any downstream system (bridge, DeFi protocol, cross-chain application) that calls `verify_transaction_inclusion` to confirm a Bitcoin transaction is included in a block can be deceived into accepting a fabricated "transaction" that is actually an internal Merkle node. This constitutes **false transaction inclusion verification** — the exact Critical impact in scope.

---

### Likelihood Explanation

- No special privileges required; any NEAR account can call the method.
- The block's Merkle tree structure is fully public (Bitcoin is a public chain).
- Internal node hashes are trivially computable from any block's transaction list.
- The attack requires only constructing a valid `ProofArgs` struct and submitting a NEAR transaction — no cryptographic breaking required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the WASM export entirely. Since it is only called internally by `verify_transaction_inclusion_v2` via `#[allow(deprecated)] self.verify_transaction_inclusion(args.into())`, refactor the shared logic into a `pub(crate)` helper function that is not exported as a NEAR contract method. This eliminates the bypass path while preserving the internal call chain.

---

### Proof of Concept

```rust
// Given a real block with transactions [T0, T1, T2, T3] already submitted to the light client:
// Compute N01 = SHA256d(H(T0) || H(T1))  -- this is a real internal node
// Compute N23 = SHA256d(H(T2) || H(T3))  -- this is the sibling

let fake_proof = ProofArgs {
    tx_id: N01,           // internal node, NOT a real transaction
    tx_block_blockhash: real_block_hash,
    tx_index: 0,          // even index → N01 goes left
    merkle_proof: vec![N23],
    confirmations: 1,
};

// Direct call to the deprecated-but-still-exported method:
let result = contract.verify_transaction_inclusion(fake_proof);
assert!(result); // returns true — false positive confirmed
```

`compute_root_from_merkle_proof(N01, 0, &[N23])` computes `SHA256d(N01 || N23) = R = merkle_root`, so the check at line 322 passes and the function returns `true` for a non-existent transaction. [7](#0-6) [8](#0-7)

```rust
#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
```
