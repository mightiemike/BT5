### Title
Permissionless `wrapVaultAsset` Allows Attacker to Force-Convert Victim's DDA Underlying to Vault Shares, Altering Collateral Type and Health Profile Without Consent — (`core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.wrapVaultAsset` carries no access control modifier. Any caller can invoke it against any victim's subaccount DDA, converting the victim's pending underlying tokens into ERC-4626 vault shares before `creditDepositV1` is called. Because vault shares carry their own `longWeightInitialX18` / `longWeightMaintenanceX18` risk parameters (which may differ from the underlying's), the victim's health is silently degraded without their consent.

---

### Finding Description

`wrapVaultAsset` is declared `external` with no `onlyOwner`, `onlyDeployer`, or subaccount-ownership check: [1](#0-0) 

The function:
1. Resolves (or creates) the DDA for the caller-supplied `subaccount`.
2. Calls `DirectDepositV1.withdraw(assetToken)` — which is `onlyOwner`, but the owner of every DDA is `ContractOwner` itself (set at deploy time), so this always succeeds when called from within `ContractOwner`. [2](#0-1) 

3. Approves the vault and calls `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)`, depositing the underlying and returning vault shares to the DDA. [3](#0-2) 

After wrapping, `creditDepositV1` (also permissionless, no modifier) calls `DirectDepositV1.creditDeposit()`: [4](#0-3) 

`creditDeposit()` is itself `external` with no access control, iterates all product IDs, and calls `endpoint.depositCollateralWithReferral(subaccount, vaultProductId, balance, "-1")` for any non-zero balance: [5](#0-4) 

The result: vault shares — not the underlying — are credited to the victim's subaccount.

---

### Impact Explanation

Vault shares and their underlying asset are registered as separate spot products with independent `RiskStore` entries. Health is computed using `longWeightInitialX18` / `longWeightMaintenanceX18` per product: [6](#0-5) 

If the vault product's `longWeightInitialX18` is lower than the underlying's, the victim's initial health decreases after the forced wrap. A sufficiently large position could push the victim below the liquidation threshold, enabling an attacker to liquidate them at a profit. The victim also becomes dependent on the vault's oracle price rather than the underlying's, and may face withdrawal restrictions imposed by the vault.

---

### Likelihood Explanation

The precondition is simply that a victim has sent underlying tokens to their DDA and `creditDepositV1` has not yet been called — a normal, expected state during the deposit flow. The attack requires two permissionless calls (`wrapVaultAsset` then `creditDepositV1`) and zero special privileges. Any EOA can execute it on mainnet.

---

### Recommendation

Add a subaccount-ownership check to `wrapVaultAsset` (and, for consistency, to `creditDepositV1`). The subaccount encodes the owner address in its high 20 bytes:

```solidity
require(
    address(bytes20(subaccount)) == msg.sender,
    "unauthorized"
);
```

This mirrors the pattern already enforced in `Endpoint.depositCollateral`, where `subaccount` is constructed from `msg.sender`: [7](#0-6) 

---

### Proof of Concept

```
Precondition:
  - vaultProduct registered with longWeightInitialX18 < underlyingProduct.longWeightInitialX18
  - victim sends 1000 USDC to their DDA (creditDepositV1 not yet called)

Attack:
  1. attacker calls ContractOwner.wrapVaultAsset(victimSubaccount, vaultProductId)
       → DDA's 1000 USDC is withdrawn to ContractOwner
       → ContractOwner deposits 1000 USDC into vault, vault shares returned to DDA
  2. attacker calls ContractOwner.creditDepositV1(victimSubaccount)
       → DDA.creditDeposit() iterates products, finds vault shares
       → calls endpoint.depositCollateralWithReferral(victimSubaccount, vaultProductId, shares, "-1")
       → vault shares credited to victim's subaccount

Assert:
  - victim's subaccount holds vault shares, not USDC
  - victim's initial health = shares * vaultPrice * vaultLongWeightInitial
                            < 1000 USDC * usdcPrice * usdcLongWeightInitial
  - if health < 0, victim is immediately liquidatable
```

    function _getWeightX18(
        Risk memory risk,
        int128 amount,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }

        int128 weight;
        if (amount >= 0) {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.longWeightInitialX18
                : risk.longWeightMaintenanceX18;
        } else {
            weight = healthType == IProductEngine.HealthType.INITIAL
                ? risk.shortWeightInitialX18
                : risk.shortWeightMaintenanceX18;
        }

        return weight;
    }
