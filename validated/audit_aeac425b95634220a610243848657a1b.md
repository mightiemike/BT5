Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` for Velocity Accounting, Rendering the Guard Ineffective on Fast L2s — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` measures elapsed time between price observations using `block.number`. On fast L2s (Arbitrum ~250 ms/block, Optimism ~2 s/block, MegaETH sub-100 ms/block), `blockDiff` accumulates far faster than on Ethereum mainnet (~12 s/block), causing `allowedSq` to scale proportionally larger per unit of real time. A pool admin who calibrates `maxChangePerBlockE18` for mainnet semantics inadvertently permits multiples of that price movement per 12-second window on Arbitrum, effectively disabling the velocity cap that protects LPs from rapid oracle-price manipulation.

## Finding Description

The guard stores `lastUpdateBlock` and computes `blockDiff = block.number - prevBlock`: [1](#0-0) 

It then enforces:

```
actualSq <= allowedSq = maxChange² × (1 + blockDiff)
``` [2](#0-1) 

On Arbitrum Nitro (~250 ms/block), 12 real seconds produce `blockDiff ≈ 48`, so `allowedSq` is `49×` larger than for `blockDiff = 1` on mainnet. The guard's real-time price-movement ceiling scales as `√(1 + blockDiff)`, meaning the same 12-second window permits `√49 ≈ 7×` more price movement on Arbitrum than on mainnet. The protocol is confirmed deployed on Arbitrum, Optimism, Base, MegaETH, Monad, and others — all with sub-mainnet block times: [3](#0-2) [4](#0-3) 

The existing guard check at line 72 (`if (actualSq > allowedSq) revert`) is structurally correct but operates on an `allowedSq` that is inflated by fast block production, so it does not fail closed as intended for rapid real-time price movements. [5](#0-4) 

## Impact Explanation

When the velocity guard is bypassed by fast block production, a manipulated or flash-crashed oracle price passes `beforeSwap`. The pool executes swaps at the bad price, paying out more of the LP reserve than the oracle-anchored curve permits. This constitutes **bad-price execution** (an unclamped oracle-derived bid/ask quote reaches a pool swap) and **swap conservation failure** (a trader receives more than the curve permits, or the pool fails to receive the owed input amount), both of which are accepted impact categories. LP principal (token0 or token1 reserves) is directly drained below what LP shares entitle holders to.

## Likelihood Explanation

The protocol is deployed by design on Arbitrum and other fast-block L2s. A single `maxChangePerBlockE18` value is set per pool by the pool admin, who is likely to calibrate it against mainnet or a reference chain — the contract comment says only "per block" with no mention of L2 block-time differences: [6](#0-5) 

No on-chain mechanism warns the admin that the parameter is chain-specific. Any oracle price movement that is legitimate over 12 s on mainnet but occurs in 0.25 s on Arbitrum will pass the guard silently. An unprivileged trader triggers the exploit simply by calling swap after a rapid oracle price movement — no special permissions required.

## Recommendation

Replace `block.number` with `block.timestamp` for velocity accounting and rename the parameter to `maxChangePerSecondE18`. The check becomes:

```solidity
uint256 secondsDiff = block.timestamp - s.lastUpdateTs;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + secondsDiff);
```

`block.timestamp` advances at wall-clock rate on all EVM-compatible L2s (Arbitrum, Optimism, zkSync all honour it), making the guard chain-agnostic.

## Proof of Concept

**Setup**: Pool deployed on Arbitrum Nitro. Admin sets `maxChangePerBlockE18 = 0.01e18` (1% per block, intended as 1% per 12 s on mainnet).

**Step 1** – Swap at block B, mid-price = 1.00 → stored as `lastMidPriceX64`, `lastUpdateBlock = B`.

**Step 2** – 12 real seconds pass; Arbitrum produces 48 L2 blocks. `block.number = B + 48`.

**Step 3** – Oracle price moves to 1.07 (7% increase in 12 s — a realistic flash event).

**Step 4** – Any user calls swap. Guard computes:
```
blockDiff  = 48
changeE18  = 0.07e18
actualSq   = (0.07e18)² = 4.9e33
allowedSq  = (0.01e18)² × (1 + 48) = 1e32 × 49 = 4.9e33
```

`actualSq == allowedSq` → guard passes. On mainnet with `blockDiff = 1`, `allowedSq = 2e32` and the swap reverts.

**Result**: A 7% price spike in 12 seconds — which the guard was configured to block — passes silently on Arbitrum. Swaps execute at the manipulated price, draining LP reserves.
