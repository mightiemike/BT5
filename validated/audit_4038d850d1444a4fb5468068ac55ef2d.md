### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the address that called `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router, so `sender` resolves to the router address â€” not the originating user. A pool admin who allowlists the router to support router-mediated swaps inadvertently opens the gate to every user, including those explicitly excluded from the allowlist.

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces the allowlist as follows:

```solidity
// SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct), and `sender` is the first argument forwarded by the pool from its own `msg.sender` â€” i.e., whoever called `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInput/exactOutput`, the router calls `pool.swap(...)`, so the pool's `msg.sender` is the router. The pool then passes the router address as `sender` to the extension hook.

The allowlist lookup therefore becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

Two broken outcomes follow:

1. **Allowlist bypass**: If the pool admin allowlists the router address (necessary for any router-mediated swap to work), every user â€” including explicitly blocked addresses â€” can bypass the allowlist by routing through `MetricOmmSimpleRouter`.
2. **Broken functionality**: If the pool admin does not allowlist the router, legitimate allowlisted users cannot use the router at all, breaking the supported periphery path.

The same structural flaw exists in `DepositAllowlistExtension.beforeAddLiquidity`, which checks `owner` but the economically relevant actor (the payer providing funds) may differ when routed through `MetricOmmPoolLiquidityAdder`. [1](#0-0) 

### Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties (e.g., KYC'd addresses, protocol-owned bots, or whitelisted market makers) loses that protection entirely for any user who routes through `MetricOmmSimpleRouter`. The disallowed user can execute swaps at oracle-derived prices, draining the pool's liquidity or extracting value in ways the pool admin explicitly intended to prevent. This is a direct loss of LP principal and a curation failure on a production pool. [2](#0-1) 

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary supported periphery entrypoint for swaps; most users will route through it.
- The pool admin must allowlist the router to make router-mediated swaps work at all, which is the expected operational setup.
- No special privileges or malicious setup are required â€” any user can call the router.
- The bypass is reachable on every production pool that uses `SwapAllowlistExtension` with the router allowlisted. [3](#0-2) 

### Recommendation

The pool must pass the **originating user** as `sender` to the extension, not the intermediate router. Two approaches:

1. **Pool-level fix**: In `MetricOmmPool.swap`, accept an explicit `sender` parameter from the caller and pass it through to `_beforeSwap`. The router would forward `msg.sender` (the actual user) as this parameter.
2. **Extension-level fix**: The extension should not trust the `sender` argument forwarded by the pool when the pool's `msg.sender` is a known router. Instead, the pool should always pass the original transaction initiator.

The `DepositAllowlistExtension` should be audited for the same pattern â€” verifying that `owner` (not `payer`) is the economically relevant actor the allowlist is intended to gate. [4](#0-3) 

### Proof of Concept

  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, router, true)   // to enable router swaps
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker calls MetricOmmSimpleRouter.exactInputSingle(pool, tokenIn, amountIn, ...)
  2. Router calls pool.swap(sender=router, ...)
  3. Pool calls extension.beforeSwap(sender=router, ...)

  5. Swap executes at oracle price for attacker
  6. Attacker receives tokenOut despite being explicitly excluded from the allowlist
