### Title
Stale `cumulativeBorrowsMultiplierX18` Understates Borrow Balances in Health Checks During Slow-Mode Withdrawals — (`core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState.getBalance()` computes a borrower's actual debt as `amountNormalized × cumulativeBorrowsMultiplierX18`, where `cumulativeBorrowsMultiplierX18` is a cached value that is only advanced when the sequencer calls `updateStates(dt)`. When the sequencer is unresponsive and users fall back to slow-mode withdrawals, this multiplier is stale. The health check executed inside `Clearinghouse.withdrawCollateral()` therefore understates the borrower's debt, allowing the borrower to withdraw more collateral than they are entitled to and leaving the protocol undercollateralized.

---

### Finding Description

**Cached multiplier, never refreshed on read.**

`SpotEngineState` stores interest accrual in two cumulative multipliers:

```
State.cumulativeDepositsMultiplierX18
State.cumulativeBorrowsMultiplierX18
```

These are advanced only inside `_updateState`, which is called exclusively from `updateStates(uint128 dt)`: [1](#0-0) 

`updateStates` is gated `onlyEndpoint`, meaning it can only be triggered by the sequencer through `submitTransactionsChecked`. No permissionless path exists to advance the multipliers.

**Balance reads use the raw stored state.**

`getBalance` and `getStateAndBalance` read `states[productId]` directly from storage and pass it to `balanceNormalizedToBalance` without first calling `_updateState`: [2](#0-1) 

`balanceNormalizedToBalance` then computes:

```
balance.amount = amountNormalized × cumulativeBorrowsMultiplierX18   (for borrowers)
``` [3](#0-2) 

If `cumulativeBorrowsMultiplierX18` has not been advanced recently, the returned `balance.amount` is less negative than the true debt — the borrower appears to owe less than they actually do.

**Health check in `withdrawCollateral` uses this stale balance.**

`Clearinghouse.withdrawCollateral` applies the balance delta and then gates the transfer on:

```solidity
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
``` [4](#0-3) 

`getHealth` calls `spotEngine.getHealthContribution` → `_calculateProductHealth` → `_getBalance` → `getBalance`, the entire chain using the stale `cumulativeBorrowsMultiplierX18`: [5](#0-4) [6](#0-5) 

**Slow-mode is the concrete trigger.**

Slow-mode withdrawals are the protocol's designed fallback when the sequencer is unresponsive. A user submits a `WithdrawCollateral` slow-mode transaction; after the 3-day timeout anyone can execute it via `executeSlowModeTransaction`: [7](#0-6) 

Because the sequencer is unresponsive, `updateStates` has not been called during the downtime window. The `cumulativeBorrowsMultiplierX18` is frozen at its last sequencer-submitted value. The health check passes on understated debt, and the withdrawal succeeds.

**Contrast with PerpEngine.**

`PerpEngineState.getStateAndBalance` calls `_updateBalance(state, balance, 0, 0)` before returning, which applies the current `cumulativeFundingLongX18` / `cumulativeFundingShortX18` from the stored state to the balance in memory: [8](#0-7) 

`SpotEngineState` has no equivalent step — it returns the raw normalized balance multiplied by the stale stored multiplier.

---

### Impact Explanation

A borrower with a negative spot balance (`amountNormalized < 0`) can execute a slow-mode collateral withdrawal during sequencer downtime. The health check uses a stale `cumulativeBorrowsMultiplierX18`, understating the borrower's debt. The borrower withdraws collateral that should be locked against the accrued-but-unrecorded interest, leaving the protocol undercollateralized. Depositors bear the resulting shortfall. The magnitude scales with the borrow rate and the duration of sequencer downtime; at a 10 % annual rate and 30 days of downtime the understatement is ~0.82 % of total borrows.

---

### Likelihood Explanation

The slow-mode path is a first-class, documented protocol feature explicitly designed for sequencer unresponsiveness. Any sequencer downtime exceeding the 3-day slow-mode timeout creates the vulnerability window. No privileged access, key compromise, or social engineering is required — the attacker is the borrower themselves, acting through the permissionless `executeSlowModeTransaction` entry point. The trigger is realistic and the entry path is fully on-chain.

---

### Recommendation

Before computing a balance for any health check, derive the current multiplier from elapsed time rather than reading the stale stored value. Concretely, `getBalance` and `getStateAndBalance` in `SpotEngineState` should call `_updateState` (or an equivalent view-only projection) using `block.timestamp` minus the last-updated timestamp before invoking `balanceNormalizedToBalance`. This mirrors how `PerpEngineState.getStateAndBalance` applies the current cumulative funding before returning a balance. Alternatively, store a `lastUpdatedAt` timestamp in `State` and compute the projected multiplier on every read path that feeds into a health check.

---

### Proof of Concept

1. At time `T0`, sequencer calls `updateStates(dt)` setting `cumulativeBorrowsMultiplierX18 = M`.
2. User borrows spot asset; `amountNormalized = -N` is stored.
3. Sequencer goes offline. No further `updateStates` calls occur.
4. At time `T0 + 3 days + ε`, user calls `executeSlowModeTransaction` for a `WithdrawCollateral` slow-mode tx they submitted earlier.
5. `withdrawCollateral` → `getHealth` → `getBalance` computes debt as `(-N) × M` (stale).
6. True debt at `T0 + 3 days` is `(-N) × M'` where `M' > M` (interest accrued).
7. Health check passes on the understated debt; collateral is transferred out.
8. Protocol is undercollateralized by `N × (M' - M)` units of the borrowed asset. [3](#0-2) [2](#0-1) [9](#0-8)

        State memory state,

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {

        } else {

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }
