Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` checks the router address as `sender` instead of the originating user, allowing any user to bypass per-user swap gating on curated pools via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's own `msg.sender` — the immediate caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. Any pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user — including those explicitly excluded — the ability to bypass the per-user gate by calling any of the router's `exact*` entry points.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender` to the extension.**

In `MetricOmmPool.swap()`, `_beforeSwap` is called with `msg.sender` as the first argument:

```solidity
_beforeSwap(
  msg.sender,   // <-- pool's msg.sender, i.e. whoever called pool.swap()
  recipient,
  ...
);
``` [1](#0-0) 

**Step 2 — `SwapAllowlistExtension.beforeSwap()` checks `sender` against the allowlist.**

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  return IMetricOmmExtensions.beforeSwap.selector;
}
``` [2](#0-1) 

Here `msg.sender` is the pool (correct), and `sender` is whoever called `pool.swap()` — which is the **router**, not the end user, for any router-mediated swap.

**Step 3 — The router calls `pool.swap()` directly, making itself the pool's `msg.sender`.**

`exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` from the router contract: [3](#0-2) 

`exactInput` does the same for each hop: [4](#0-3) 

In every case the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

**Step 4 — The bypass.**

A pool admin who wants to support both direct and router-mediated swaps must allowlist the router:
```
allowedSwapper[pool][router] = true
```
Once the router is allowlisted, every user — including those not individually allowlisted — can bypass the gate by calling any of the router's `exact*` entry points. The extension sees `sender = router` (allowlisted) and passes unconditionally. There is no secondary check on the originating user anywhere in the call path.

## Impact Explanation

`SwapAllowlistExtension` is the sole mechanism for curated pools to restrict who may trade. A complete bypass means unauthorized users can execute swaps on pools intended to be restricted (e.g., KYC-gated, institutional-only, or whitelist-only pools). LP funds in those pools are exposed to trades from actors the pool admin explicitly intended to exclude. This constitutes a broken core pool access-control invariant and an admin-boundary break reachable by any unprivileged user through the protocol's own supported public entrypoint.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who deploys a curated pool and also wants to support router-mediated swaps must allowlist the router — this is the expected operational configuration. No special privileges, malicious setup, or non-standard tokens are required. Any unprivileged user can trigger the bypass by calling `exactInputSingle`, `exactInput`, or `exactOutputSingle` on the router. Likelihood is **Medium-High**: the vulnerable configuration is the natural one for any production curated pool.

## Recommendation

The extension must gate on the **actual end user**, not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original `msg.sender` through the router.** The router already stores the original caller in transient storage for the callback context (`_setNextCallbackContext(..., msg.sender, ...)`). Extend this to expose the original initiator to the pool/extension via an additional field in the swap call or a dedicated transient slot the extension can read.

2. **Short-term mitigation:** Document explicitly that allowlisting the router address defeats per-user gating, and prohibit pool admins from using the router allowlist entry as a workaround for router support. Require separate `allowAllSwappers` flag for open-access pools.

## Proof of Concept

```solidity
// Setup: curated pool with SwapAllowlistExtension
// Pool admin allowlists alice for direct swaps AND the router for router-mediated swaps
allowlist.setAllowedToSwap(pool, alice, true);
allowlist.setAllowedToSwap(pool, address(router), true); // needed for router support

// Bob (not allowlisted) bypasses the gate via router
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    tokenIn: token0,
    recipient: bob,
    amountIn: 1000,
    amountOutMinimum: 0,
    zeroForOne: true,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// ✓ swap succeeds — extension saw sender=router (allowlisted), not bob (not allowlisted)
// Bob receives token1 from a pool he was explicitly excluded from
```
