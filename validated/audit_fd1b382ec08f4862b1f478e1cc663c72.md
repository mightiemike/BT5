### Title
`availableSettle` Not Decremented During Liquidation-Based PnL Settlement — (`File: core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`PerpEngine.state.availableSettle` is decremented in the normal `settlePnl()` path but is never decremented when PnL is settled during liquidation via `_settlePnlAgainstLiquidator()`. This causes `availableSettle` to be permanently overstated after any liquidation that settles positive PnL, allowing other traders to settle more PnL than the pool actually supports.

---

### Finding Description

The `availableSettle` field in `PerpEngine.State` is a global cap on how much virtual quote (`vQuoteBalance`) can be converted into real spot quote across all subaccounts. It is decremented in the normal settlement path:

**`PerpEngine.settlePnl()` (lines 94–95):**
```solidity
state.availableSettle -= canSettle;
balance.vQuoteBalance -= canSettle;
```

During liquidation, `ClearinghouseLiq._settlePositivePerpPnl()` and `_finalizeSubaccount()` both call `_settlePnlAgainstLiquidator()` to settle positive PnL from the liquidatee:

**`ClearinghouseLiq._settlePnlAgainstLiquidator()` (lines 259–270):**
```solidity
perpEngine.updateBalance(perpId, txn.liquidatee, 0, -pnl);
perpEngine.updateBalance(perpId, txn.sender, 0, pnl);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, pnl);
spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, -pnl);
```

This call reduces the liquidatee's `vQuoteBalance` by `pnl` and increases their spot quote by `pnl` — which is economically identical to `settlePnl()`. However, `perpEngine.updateBalance()` routes through `PerpEngineState._updateBalance()`, which does not touch `state.availableSettle`. The decrement `state.availableSettle -= pnl` is never performed.

This function is invoked in two liquidation sub-paths:

1. **`_settlePositivePerpPnl()`** (lines 415–437): called when liquidating a spot/spread liability, iterates all perp products and settles any positive PnL of the liquidatee.
2. **`_finalizeSubaccount()`** (lines 322–366): called when `txn.productId == type(uint32).max`, settles all positive and negative PnL of the liquidatee.

Both paths settle real economic value (vQuoteBalance → spot quote) without decrementing `availableSettle`.

---

### Impact Explanation

After each liquidation that settles positive PnL, `state.availableSettle` is overstated by the settled amount. Since `getSettlementState()` caps each trader's settleable amount at `min(positionPnl, state.availableSettle)`, an inflated `availableSettle` allows other traders to settle more PnL than the pool's actual capacity supports. Over time, with repeated liquidations, the overstatement compounds. Traders can drain the spot engine's quote reserves beyond what is backed by real losses covered by insurance or socialization, corrupting the solvency invariant that total settled PnL ≤ total covered losses.

---

### Likelihood Explanation

Liquidations are a routine, unprivileged operation triggered by any trader when a subaccount's maintenance health falls below zero. Any liquidation of a subaccount holding a perp position with positive PnL (a common scenario — e.g., a long position in a rising market that becomes undercollateralized on the quote side) will trigger `_settlePositivePerpPnl()` or `_finalizeSubaccount()`, causing the bug. No special access or configuration is required.

---

### Recommendation

Decrement `state.availableSettle` in `_settlePnlAgainstLiquidator()` when `pnl > 0` (positive PnL being settled from liquidatee to liquidator). Since `_settlePnlAgainstLiquidator()` calls `perpEngine.updateBalance()` rather than `perpEngine.settlePnl()`, the decrement must be added explicitly. One approach is to expose a dedicated settlement function in `IPerpEngine` that mirrors `settlePnl()` but accepts an explicit amount, or to add the decrement directly inside `_settlePnlAgainstLiquidator()` via a new engine call:

```solidity
// After settling positive pnl against liquidator, reduce availableSettle
if (pnl > 0) {
    perpEngine.decrementAvailableSettle(perpId, pnl);
}
```

Alternatively, route liquidation-based PnL settlement through the existing `settlePnl()` mechanism so the decrement is applied consistently.

---

### Proof of Concept

1. Subaccount `A` holds a long perp position with `vQuoteBalance = +1000` (positive PnL) and becomes undercollateralized on the quote side.
2. Liquidator calls `liquidateSubaccount` with `txn.productId` set to the spot liability product.
3. `liquidateSubaccountImpl()` calls `_assertCanLiquidateLiability()` then `_settlePositivePerpPnl()`.
4. `_settlePositivePerpPnl()` calls `_settlePnlAgainstLiquidator(txn, perpId, 1000, ...)`.
5. `perpEngine.updateBalance(perpId, A, 0, -1000)` reduces A's `vQuoteBalance` to 0.
6. `spotEngine.updateBalance(QUOTE_PRODUCT_ID, A, 1000)` increases A's spot quote by 1000.
7. `state.availableSettle` for `perpId` is **not decremented** — it remains at its prior value, say `S`.
8. Trader `B` now calls `settlePnl()` with their own positive PnL. `getSettlementState()` returns `min(B_pnl, S)` instead of `min(B_pnl, S - 1000)`.
9. Trader `B` settles 1000 more quote than the pool supports, draining the spot engine's quote reserves beyond the backed amount.

```text
    function settlePnl(bytes32 subaccount, uint256 productIds)
        external
        returns (int128)
    {
        _assertInternal();
        int128 totalSettled = 0;

        while (productIds != 0) {
            uint32 productId = uint32(productIds & ((1 << 32) - 1));
            // otherwise it means the product is a spot.
            if (productId % 2 == 0) {
                (
                    int128 canSettle,
                    State memory state,
                    Balance memory balance
                ) = getSettlementState(productId, subaccount);

                state.availableSettle -= canSettle;
                balance.vQuoteBalance -= canSettle;

                totalSettled += canSettle;

                _setState(productId, state);
                _setBalanceAndUpdateBitmap(productId, subaccount, balance);
            }
            productIds >>= 32;
        }
        return totalSettled;
```
