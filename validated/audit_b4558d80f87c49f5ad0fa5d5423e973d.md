### Title
Publicly Callable `verify_transaction_inclusion` (v1) Accepts Internal Merkle Tree Nodes as Valid Transaction IDs, Enabling Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains a live, publicly callable NEAR method. It only checks that a supplied hash reconstructs the block's Merkle root via a provided proof path, without confirming the hash is a leaf node (a real transaction). An unprivileged caller can supply an internal Merkle tree node as `tx_id` with a crafted proof that correctly reconstructs the `merkle_root`, causing the function to return `true` for a Bitcoin transaction that does not exist.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` in Rust, but this attribute is purely a compile-time warning — it does not remove the function from the deployed WASM binary or restrict on-chain access. Any NEAR account can call it directly. [1](#0-0) 

The function's sole proof check is: [2](#0-1) 

`compute_root_from_merkle_proof` traverses the supplied proof path starting from `tx_id` and returns whatever hash results: [3](#0-2) 

There is no check that `tx_id` is a leaf node. Any hash that appears at any level of the Merkle tree — including internal nodes — will produce a valid root reconstruction if paired with the correct sibling path. The contract's own documentation acknowledges this explicitly: [4](#0-3) 

`verify_transaction_inclusion_v2` fixes this by requiring a coinbase proof of equal length, which anchors the proof depth to the actual tree height: [5](#0-4) 

But v1 remains reachable and unprotected.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (releasing bridged funds, minting wrapped tokens, recording a cross-chain event) can be deceived into accepting a forged Bitcoin transaction proof. The attacker does not need to control any privileged role, submit any real Bitcoin transaction, or perform any on-chain Bitcoin operation. All required data (block Merkle tree structure) is publicly available from the Bitcoin network.

---

### Likelihood Explanation

High. The attack requires only:
1. A Bitcoin block already stored in the contract (any block with ≥2 transactions).
2. Knowledge of that block's transaction list (public Bitcoin data).
3. A direct NEAR function call to `verify_transaction_inclusion` — no special role, no deposit, no staking.

The `#[pause]` gate is the only runtime barrier, and the contract operates unpaused in normal production conditions.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public NEAR API entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. All callers should be migrated to `verify_transaction_inclusion_v2`, which correctly anchors proof depth via the coinbase proof length check. [6](#0-5) 

---

### Proof of Concept

Consider a Bitcoin block stored in the contract with four transactions: `tx0, tx1, tx2, tx3`.

The Merkle tree is:
```
h01  = double_sha256(tx0 || tx1)
h23  = double_sha256(tx2 || tx3)
root = double_sha256(h01 || h23)
```

An attacker constructs `ProofArgs`:
- `tx_id       = h01`  ← an internal node, not a real transaction
- `tx_index    = 0`
- `merkle_proof = [h23]`
- `tx_block_blockhash` = any block hash stored in the contract whose `merkle_root` equals `root`
- `confirmations` = any value ≤ `gc_threshold`

`compute_root_from_merkle_proof(h01, 0, [h23])`:
- position 0 is even → `current_hash = double_sha256(h01 || h23)` = `root` ✓ [7](#0-6) 

The function returns `true`. No Bitcoin transaction with hash `h01` exists; the attacker has forged a valid inclusion proof for a phantom transaction.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
