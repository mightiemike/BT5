Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, making the swap allowlist either permanently broken or trivially bypassable via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of the pool. When `MetricOmmSimpleRouter` is used, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This makes the allowlist either permanently broken (allowlisted users cannot swap through the router) or trivially bypassable (if the router is allowlisted, every user can trade on a curated pool).

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
// MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,   // ← whoever called pool.swap()
  ...
```

`ExtensionCalling._beforeSwap` forwards that value verbatim as `sender` in the ABI-encoded call to every configured extension (lines 149–176 of `ExtensionCalling.sol`).

**Step 2 — `SwapAllowlistExtension` keys the allowlist on that `sender`.**

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()` — the router, not the end-user.

**Step 3 — `MetricOmmSimpleRouter` is `msg.sender` to the pool.**

Every router entry-point calls `IMetricOmmPoolActions(params.pool).swap(...)` directly (e.g., `exactInputSingle` at lines 72–80 of `MetricOmmSimpleRouter.sol`), making the router the `msg.sender` the pool sees.

**Result:** The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. Pool admin configures `setAllowedToSwap(pool, alice, true)`. When Alice calls `router.exactInputSingle(...)`, the extension receives `sender = address(router)`, looks up `allowedSwapper[pool][router]` (false), and reverts `NotAllowedToSwap`.

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly checks the explicit `owner` parameter (the position owner), not `sender`, so it correctly identifies the economically relevant actor regardless of who `msg.sender` is (line 38 of `DepositAllowlistExtension.sol`).

## Impact Explanation

A pool configured with `SwapAllowlistExtension` as a `beforeSwap` hook cannot simultaneously (a) allow its allowlisted users to trade through the standard periphery router and (b) block non-allowlisted users from doing the same. The two goals are mutually exclusive:

- **Allowlist not including the router** → allowlisted users are blocked from the router; core swap functionality is broken for the intended audience.
- **Allowlist including the router** → every unprivileged user bypasses the curation gate and trades on a pool that should be restricted, causing direct violation of the pool's intended access-control invariant and enabling unauthorized trades.

This constitutes broken core pool functionality (the swap allowlist is the primary access-control mechanism for curated pools) and enables unauthorized users to execute swaps on restricted pools.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap interface for end-users. Any pool that activates `SwapAllowlistExtension` and expects users to interact through the router will immediately encounter this failure. No special privileges, flash loans, or unusual token behavior are required — a standard `exactInputSingle` call is sufficient. The failure is deterministic and reproducible on every router-mediated swap.

## Recommendation

Pass the original end-user identity through the swap path so the extension can check it:

1. **Preferred — encode the real user in `extensionData`**: The router encodes `msg.sender` (the real user) into `extensionData`, and the extension reads it from there after verifying the pool is the caller.

2. **Simpler — check `sender` only when the caller is not a known router**: The extension reads the pool's factory to determine whether `sender` is a registered periphery contract, and if so, decodes the real user from `extensionData`.

Either way, the extension must distinguish the intermediary router from the economic actor it is supposed to gate.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension as beforeSwap hook
  pool admin calls: setAllowedToSwap(pool, alice, true)
  pool admin does NOT allowlist the router

Attack path A — allowlisted user is blocked:
  alice calls router.exactInputSingle({pool: pool, ...})
  router calls pool.swap(recipient, ...) → msg.sender to pool = router
  pool calls _beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router] → false
  → revert NotAllowedToSwap   ← alice cannot use the router

Attack path B — bypass (pool admin allowlists router to fix A):
  pool admin calls: setAllowedToSwap(pool, router, true)
  eve (not allowlisted) calls router.exactInputSingle({pool: pool, ...})
  router calls pool.swap(...) → msg.sender = router
  extension checks allowedSwapper[pool][router] → true
  → swap succeeds   ← eve bypasses the allowlist entirely

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension as beforeSwap hook
  2. setAllowedToSwap(pool, alice, true)
  3. vm.prank(alice); router.exactInputSingle(...) → expect revert NotAllowedToSwap
  4. setAllowedToSwap(pool, router, true)
  5. vm.prank(eve); router.exactInputSingle(...) → succeeds (bypass confirmed)
```
