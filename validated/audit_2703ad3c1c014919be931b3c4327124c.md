### Title
Incorrect Shortfall Calculation in `topUpBaseRewards` Uses Fee Collector Balance That May Already Be Zero Due to BeginBlocker Ordering - (File: x/tieredrewards/keeper/abci.go)

---

### Summary

`topUpBaseRewards` computes the "default stakers reward per block" by reading the fee collector's current balance at BeginBlocker time. If the distribution module's `BeginBlocker` executes before the tiered-rewards module's `BeginBlocker` â€” which is the standard Cosmos SDK module ordering â€” the fee collector is already empty when `topUpBaseRewards` runs. The shortfall is therefore always computed as the full target amount, causing the rewards pool to be drained at the maximum configured rate every block regardless of actual fee revenue, and causing validators to receive both the normal fee distribution and the full top-up simultaneously.

---

### Finding Description

`topUpBaseRewards` in `x/tieredrewards/keeper/abci.go` is designed to ensure stakers receive at least `TargetBaseRewardsRate` APY. It does so by:

1. Computing the target per-block reward:
   ```
   targetStakersRewardPerBlock = totalBonded Ã— targetBaseRewardsRate / blocksPerYear
   ```
2. Computing the "default" per-block reward from fees:
   ```
   defaultStakersRewardPerBlock = feeCollectorBalance Ã— (1 âˆ’ communityTax)
   ```
3. Topping up the shortfall from the rewards pool.

The critical assumption is that `feeCollectorBalance` represents the previous block's fee revenue at the time this function runs. This is only true if the tiered-rewards `BeginBlocker` executes **before** the distribution module's `BeginBlocker`.

In the Cosmos SDK, `x/distribution`'s `BeginBlocker` calls `AllocateTokens`, which sweeps the entire fee collector balance and distributes it to validators. If distribution's `BeginBlocker` runs first (the standard SDK ordering), the fee collector is empty when `topUpBaseRewards` reads it:

```go
// x/tieredrewards/keeper/abci.go:73-77
feeCollectorBalance := k.bankKeeper.GetBalance(ctx, feeCollectorAddr, bondDenom)
defaultStakersRewardPerBlock := math.LegacyNewDecFromInt(feeCollectorBalance.Amount).
    MulTruncate(math.LegacyOneDec().Sub(communityTax))

shortFallAmount := targetStakersRewardPerBlock.Sub(defaultStakersRewardPerBlock).TruncateInt()
```

With `feeCollectorBalance = 0`, `defaultStakersRewardPerBlock = 0`, and `shortFallAmount = targetStakersRewardPerBlock` (always positive when `TargetBaseRewardsRate > 0`). The guard at line 79 (`if !shortFallAmount.IsPositive()`) never fires, so the module unconditionally sends the full target amount from `RewardsPoolName` to `distributiontypes.ModuleName` and calls `AllocateTokensToValidator` for every bonded validator. [1](#0-0) 

The result is that validators receive **two** reward streams in the same block:
- The normal fee distribution (already allocated by `x/distribution`'s `BeginBlocker`).
- The full top-up from the rewards pool (allocated by `topUpBaseRewards`).

The intended behavior â€” top up only the shortfall between fees and the target rate â€” is never achieved.

---

### Impact Explanation

The `RewardsPoolName` module account balance is drained at the full `targetStakersRewardPerBlock` rate every block, regardless of actual fee revenue. If the chain collects fees equal to or exceeding the target, the rewards pool is still fully consumed. The pool, which is a finite governance-funded resource, is depleted at the maximum possible rate. Validators receive more than the target base rewards rate, breaking the economic invariant the module is designed to enforce.

The corrupted value is the `RewardsPoolName` module account balance (drained at `targetStakersRewardPerBlock` per block instead of only the shortfall).

---

### Likelihood Explanation

The standard Cosmos SDK module manager registers `x/distribution` before custom application modules. If `chain-main`'s `app/app.go` follows this convention (which is the default for Cosmos SDK apps), the distribution `BeginBlocker` runs before the tiered-rewards `BeginBlocker` every single block. The issue would therefore be triggered on every block where `TargetBaseRewardsRate > 0` and the rewards pool is non-empty â€” i.e., continuously during normal chain operation. No special transaction, privileged key, or attacker action is required; the bug is exercised by the normal block lifecycle. [2](#0-1) 

---

### Recommendation

The tiered-rewards `BeginBlocker` must be registered **before** the distribution module's `BeginBlocker` in `app/app.go`'s `SetOrderBeginBlockers` call, so that `feeCollectorBalance` is read before distribution sweeps it. Alternatively, the shortfall calculation should be made independent of BeginBlocker ordering by reading the fee revenue from a source that is not cleared by distribution (e.g., a per-block fee snapshot written by a custom ante handler or a pre-distribution hook), or by restructuring the top-up to run as a pre-distribution step.

---

### Proof of Concept

1. Chain is configured with `TargetBaseRewardsRate > 0` and a funded `RewardsPoolName` account.
2. A block is produced with non-zero transaction fees (e.g., 1000 `basecro`).
3. Distribution's `BeginBlocker` runs first, sweeping the 1000 `basecro` from the fee collector and distributing it to validators.
4. Tiered-rewards `BeginBlocker` runs next. `feeCollectorBalance = 0`. `defaultStakersRewardPerBlock = 0`. `shortFallAmount = targetStakersRewardPerBlock` (e.g., 500 `basecro`).
5. The module sends 500 `basecro` from `RewardsPoolName` to `distributiontypes.ModuleName` and calls `AllocateTokensToValidator` for each validator.
6. Validators receive 1000 `basecro` (from step 3) + 500 `basecro` (from step 5) = 1500 `basecro`, instead of the intended max(1000, 500) = 1000 `basecro`.
7. The rewards pool is drained by 500 `basecro` even though fees already exceeded the target. [3](#0-2)
