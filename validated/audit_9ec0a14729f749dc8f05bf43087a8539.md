### Title
Fee-on-Transfer Token Accounting Inflation via Two-Hop Deposit Transfer — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` routes deposits through a two-hop transfer (user → Endpoint → Clearinghouse) without measuring the actual amount received at either hop. The slow-mode transaction queued immediately after records the caller-supplied `amount`, not the tokens that arrived. When `Clearinghouse.depositCollateral` later processes that slow-mode tx, it credits the subaccount with the original `amount`. For a fee-on-transfer collateral token, the Clearinghouse holds fewer tokens than the sum of all credited subaccount balances, creating a solvency gap that grows with every such deposit.

---

### Finding Description

`EndpointStorage.handleDepositTransfer` (lines 111–119) performs two sequential transfers using the same nominal `amount`:

```
safeTransferFrom(token, from, amount)      // hop 1: user → Endpoint
safeTransferTo(token, address(clearinghouse), amount)  // hop 2: Endpoint → Clearinghouse
``` [1](#0-0) 

No balance snapshot is taken before or after either hop.

`depositCollateralWithReferral` (lines 144–165) calls `handleDepositTransfer` and then immediately enqueues a slow-mode `DepositCollateral` transaction whose `amount` field is the original caller-supplied value, not the tokens that actually arrived in the Clearinghouse. [2](#0-1) 

When the sequencer (or anyone after the timeout) executes that slow-mode tx, `Clearinghouse.depositCollateral` computes `amountRealized = int128(txn.amount) * int128(multiplier)` and calls `spotEngine.updateBalance(txn.productId, txn.sender, amountRealized)` — crediting the full nominal amount with no reference to on-chain token balances. [3](#0-2) 

For a fee-on-transfer token the two hops each consume a fee:

- **Hop 1**: Endpoint receives `amount − fee₁` (not `amount`).
- **Hop 2**: Endpoint attempts to forward `amount`. This succeeds only if the Endpoint already holds at least `fee₁` of the token. Such a balance accumulates naturally: `chargeSlowModeFee` pulls the quote token directly into the Endpoint contract on every slow-mode submission. [4](#0-3) 

When hop 2 succeeds by consuming pre-existing Endpoint balance, the Clearinghouse receives `amount − fee₂` tokens, but the subaccount is credited `amount`. The `fee₁ + fee₂` shortfall is silently absorbed from the Endpoint's accumulated balance, and the Clearinghouse's real holdings fall below the sum of all subaccount balances.

---

### Impact Explanation

The SpotEngine's recorded subaccount balances exceed the Clearinghouse's actual token holdings by `fee₂` per exploited deposit. An attacker with an inflated balance can:

1. **Over-leverage**: Health checks (`getHealth`, `_isAboveInitial`) use SpotEngine balances, so the attacker passes health checks with less real collateral than required.
2. **Drain the pool**: On withdrawal, `withdrawCollateral` debits the subaccount by `amount` and transfers `amount` out of the Clearinghouse. Because the Clearinghouse was already short, the deficit compounds. Eventually `assertUtilization` will block legitimate withdrawals for other users.

The corrupted state variable is `SpotEngine.balances[productId][subaccount].amount`, which is set higher than the real token reserve backing it.

---

### Likelihood Explanation

The trigger requires two conditions:

1. A fee-on-transfer ERC-20 is listed as a supported collateral product (no code-level guard prevents this).
2. The Endpoint holds a non-zero balance of that token at deposit time — satisfied whenever any slow-mode fee has been charged in the same token, or whenever a prior deposit of that token left dust in the Endpoint.

Both conditions are realistic in a live deployment that supports multiple collateral assets. The entry point (`depositCollateral` / `depositCollateralWithReferral`) is fully public and requires no special role.

---

### Recommendation

Replace the fixed-`amount` two-hop pattern with a balance-delta measurement:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 received = token.balanceOf(address(clearinghouse)) - before;
    // return `received` to the caller so the slow-mode tx records the real amount
}
```

Pass `received` (not `amount`) as the `DepositCollateral.amount` field when enqueuing the slow-mode transaction in `depositCollateralWithReferral`.

---

### Proof of Concept

Assume token X charges a 1 % transfer fee and is listed as a Nado collateral product. The Endpoint holds 10 X from prior slow-mode fee charges.

1. Attacker calls `Endpoint.depositCollateral(name, productId, 1000)`.
2. `handleDepositTransfer` runs:
   - `safeTransferFrom(X, attacker, 1000)` → Endpoint receives **990 X** (1 % fee).
   - `safeTransferTo(X, clearinghouse, 1000)` → succeeds because Endpoint has 990 + 10 = 1000 X; Clearinghouse receives **990 X** (another 1 % fee).
3. Slow-mode tx is queued with `amount = 1000`.
4. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits attacker's subaccount with **1000 X** (normalized).
5. Clearinghouse holds **990 X** but the subaccount shows **1000 X** — a 10 X solvency gap.
6. Attacker withdraws 1000 X; Clearinghouse sends 1000 X to WithdrawPool, leaving it **10 X short** for other depositors. [1](#0-0) [2](#0-1) [5](#0-4)
