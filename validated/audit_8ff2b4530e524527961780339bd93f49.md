### Title
Missing `SetDecayEpochStart` Call in v8 Upgrade Handler Permanently Disables Inflation Decay — (`app/upgrades.go`, `x/inflation/keeper/mint.go`)

---

### Summary

The v8 upgrade handler never calls `SetDecayEpochStart`, so the `DecayEpochStartKey` store entry is never written. Every subsequent call to `DeflationCalculationFn` hits the `!ok` early-return path and returns `baseRate` unchanged for all future blocks. When `DecayRate > 0` is configured, the chain mints tokens at the full undecayed rate indefinitely, producing a supply that permanently exceeds the intended decayed inflation schedule.

---

### Finding Description

`registerV8UpgradeHandler` only calls `RunMigrations` and returns: [1](#0-0) 

`RunMigrations` calls `InitGenesis` for the new `inflation` module (ConsensusVersion = 1, so it has no prior version map entry) with `DefaultGenesis()`, which sets `DecayEpochStart = 0`: [2](#0-1) [3](#0-2) 

`InitGenesis` skips writing the key when the value is zero: [4](#0-3) 

`getDecayEpochStart` therefore returns `ok=false` on every call post-upgrade: [5](#0-4) 

`DeflationCalculationFn` returns `baseRate` unchanged when `ok=false`, bypassing all decay math: [6](#0-5) 

The store key comment itself documents the intended contract: *"set by the upgrade handler or genesis"* — but the upgrade handler never does so: [7](#0-6) 

`SetDecayEpochStart` is only called from `InitGenesis` (guarded by `!= 0`) and tests — never from the upgrade handler: [8](#0-7) 

---

### Impact Explanation

When `DecayRate > 0` (the entire purpose of the module), the chain mints at the full `baseRate` forever instead of the intended exponentially decaying schedule. The cumulative excess supply grows monotonically with every block. This is a concrete supply accounting flaw: the actual token supply diverges upward from the intended schedule, permanently inflating supply above the designed cap.

---

### Likelihood Explanation

The `x/inflation` module is introduced in v8 specifically to implement decaying inflation. Any deployment that sets `DecayRate > 0` — either in the initial genesis params or via a subsequent governance `MsgUpdateParams` — immediately triggers the broken path. The bug is deterministic and requires no attacker action: it fires on every `BeginBlock` after the upgrade.

---

### Recommendation

Add a `SetDecayEpochStart` call inside `registerV8UpgradeHandler`, immediately after `RunMigrations` succeeds, using the current block height:

```go
if err := app.InflationKeeper.SetDecayEpochStart(ctx, uint64(sdkCtx.BlockHeight())); err != nil {
    return nil, fmt.Errorf("v8: failed to set decay epoch start: %w", err)
}
``` [9](#0-8) 

---

### Proof of Concept

1. Start a chain with `DecayRate = 0.05` in inflation params.
2. Execute the v8 upgrade handler via `RunMigrations`.
3. Call `DeflationCalculationFn` at any post-upgrade height.
4. Observe the returned rate equals `baseRate` (no decay applied), because `getDecayEpochStart` returns `ok=false`.
5. Assert `returnedRate < baseRate` — this assertion fails, confirming decay is permanently disabled. [10](#0-9)

func (k Keeper) SetDecayEpochStart(ctx context.Context, height uint64) error {
	store := k.storeService.OpenKVStore(ctx)
	bz := make([]byte, 8)
	binary.BigEndian.PutUint64(bz, height)
	return store.Set([]byte(types.DecayEpochStartKey), bz)
}
```

func (k Keeper) getDecayEpochStart(ctx context.Context) (uint64, bool, error) {
	store := k.storeService.OpenKVStore(ctx)
	bz, err := store.Get([]byte(types.DecayEpochStartKey))
	if err != nil {
		return 0, false, err
	}
	if len(bz) == 0 {
		return 0, false, nil
	}
	if len(bz) != 8 {
		return 0, false, fmt.Errorf("invalid decay epoch start encoding: len=%d", len(bz))
	}
	return binary.BigEndian.Uint64(bz), true, nil
```
