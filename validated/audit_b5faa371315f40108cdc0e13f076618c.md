### Title
Lossy Protobuf Serialization of `ValidResourceBounds` Conflates `AllResources` (zero L2/data gas) with `L1Gas`, Producing a Wrong Transaction Hash on P2P Sync - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf round-trip for `ValidResourceBounds` is lossy. When a V3 transaction carries `AllResources` bounds with both `l2_gas` and `l1_data_gas` set to their zero default (`max_amount = 0, max_price_per_unit = 0`), the serializer emits the same wire bytes as a `L1Gas` transaction. The deserializer then reconstructs the wrong variant (`L1Gas` instead of `AllResources`). Because `get_tip_resource_bounds_hash` includes `l1_data_gas` in the Poseidon preimage only for `AllResources`, the reconstructed transaction produces a different hash than the one committed by the sequencer, breaking hash canonicalization across the P2P sync boundary.

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471–489):

```rust
ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
    l1_gas: Some(l1_gas.into()),
    l2_gas: Some(value.get_l2_bounds().into()),       // ResourceBounds::default() → zero
    l1_data_gas: Some(ResourceBounds::default().into()), // zero
},
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
```

When `l2_gas` and `l1_data_gas` are both zero, both arms emit identical wire bytes. The variant tag (`L1Gas` vs `AllResources`) is not encoded.

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

`ResourceBounds::is_zero()` is `max_amount == 0 && max_price_per_unit == 0`. Any `AllResources` transaction whose user-supplied `l2_gas` and `l1_data_gas` are both zero is silently downgraded to `L1Gas`.

**Hash divergence** (`get_tip_resource_bounds_hash`, lines 188–210):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
```

`Poseidon([tip, l1, l2]) ≠ Poseidon([tip, l1, l2, l1_data])` even when `l2 = 0` and `l1_data = 0`. The sequencer commits the 3-felt hash; the syncing peer recomputes the 2-felt hash.

### Impact Explanation

**Impact: High** — Transaction conversion binds the wrong type and executable payload. A syncing node that receives a block containing such a transaction via P2P will reconstruct a `ValidResourceBounds::L1Gas` object instead of `AllResources`. Any component that recomputes or verifies the transaction hash (e.g., block hash verification, receipt commitment, re-execution) will derive a different value than the one committed by the sequencer. This constitutes a wrong hash/type binding across the P2P sync boundary, matching the "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact.

### Likelihood Explanation

**Likelihood: Medium** — The trigger requires a V3 transaction where the user sets `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}`. This is the `Default` value for `ResourceBounds` and is a natural choice for a user who does not intend to consume L2 gas or L1 data gas. The gateway accepts `AllResourceBounds` directly (all `RpcTransaction` V3 types carry `AllResourceBounds`, not `ValidResourceBounds`), so no special privilege is required. The condition is reachable by any unprivileged user submitting a standard V3 transaction.

### Recommendation

Encode the variant discriminant explicitly in the protobuf message so the round-trip is lossless. One approach: add a boolean or enum field `resource_bounds_version` (or `is_all_resources`) to `protobuf::ResourceBounds` and set/check it during conversion. Alternatively, treat the presence of `l1_data_gas` as the discriminant only when the field is truly absent (i.e., `None`) rather than zero — but this requires the serializer to omit `l1_data_gas` for `L1Gas` transactions instead of writing a zero value.

The existing TODO comment at line 426 (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.`) acknowledges the transitional state; the fix should be applied before that TODO is resolved, since the current zero-sentinel heuristic is ambiguous.

### Proof of Concept

1. Construct a `ValidResourceBounds::AllResources` with `l2_gas = ResourceBounds::default()` and `l1_data_gas = ResourceBounds::default()` (both zero).
2. Serialize via `protobuf::ResourceBounds::from(bounds)`.
3. Deserialize via `ValidResourceBounds::try_from(proto_bounds)`.
4. Assert the result is `ValidResourceBounds::L1Gas(_)` — the variant has changed.
5. Compute `get_tip_resource_bounds_hash` on the original `AllResources` and on the deserialized `L1Gas` with the same `l1_gas` and `tip`.
6. Observe the two hashes differ: the `AllResources` path chains 3 resource felts (`l1_gas`, `l2_gas=0`, `l1_data_gas=0`) while the `L1Gas` path chains only 2 (`l1_gas`, `l2_gas=0`), producing a different Poseidon output.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
