Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Original User, Enabling Complete Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which is `msg.sender` as seen by the pool — the router contract when users swap via `MetricOmmSimpleRouter`. Any pool admin who allowlists the router to support router-mediated swaps for their users simultaneously grants swap access to every caller of the router, completely defeating the per-user allowlist.

## Finding Description
In `MetricOmmPool.swap()`, `msg.sender` (the router) is passed as `sender` to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap()` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router — not the original user: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly with no mechanism to forward the original `msg.sender` identity: [4](#0-3) 

The router stores the original caller only in transient callback context for payment purposes (`_setNextCallbackContext(..., msg.sender, ...)`), but this is never surfaced to the pool's extension hook path. [5](#0-4) 

## Impact Explanation
A pool admin who wants allowlisted users to swap via the standard periphery router must call `setAllowedToSwap(pool, router, true)`. Once set, `allowedSwapper[pool][router]` returns `true` for every caller of the router regardless of whether that caller is individually allowlisted. The allowlist is completely bypassed for all router-mediated swaps, allowing unauthorized users to execute swaps on a curated pool at oracle-derived prices the pool was configured to restrict to specific counterparties. This constitutes a broken core pool access-control mechanism causing potential loss of LP value — a High severity impact under Sherlock thresholds. [6](#0-5) 

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard documented periphery entrypoint. Any user who discovers a pool uses `SwapAllowlistExtension` can trivially route through the router with a single `exactInputSingle` call. No special privileges, flash loans, or multi-step setup are required. The precondition — router being allowlisted — is the expected operational state for any pool intending to support router users. [7](#0-6) 

## Recommendation
The extension must gate the original user, not the intermediate caller. The `beforeSwap` hook arguments (or a transient-storage context set by the router before calling the pool) should carry an `originator` field representing the original `msg.sender` at the router entry point. `SwapAllowlistExtension` should check that field instead of `sender`. As a short-term mitigation, document that pools using `SwapAllowlistExtension` must not allowlist the router address and that allowlisted users must call the pool directly — though this is a severe usability restriction and not a real fix. [8](#0-7) 

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension as beforeSwap extension.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to let allowlisted users swap via the router).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).

Attack:
  - attacker (not on allowlist) calls:
      MetricOmmSimpleRouter.exactInputSingle({
          pool: pool,
          recipient: attacker,
          zeroForOne: true,
          amountIn: X,
          ...
      })
  - Pool calls _beforeSwap(msg.sender=router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Swap executes successfully for the non-allowlisted attacker.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds — allowlist fully bypassed.
``` [9](#0-8)
