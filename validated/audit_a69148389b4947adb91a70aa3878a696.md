### Title
Softfork Nesting Depth Limit Enforced Only in Mempool Mode, Bypassable in Consensus Mode — (`src/run_program.rs`)

---

### Summary

The softfork nesting depth limit of 20 is gated behind `ClvmFlags::LIMIT_SOFTFORK`, which is only set in `MEMPOOL_MODE`. In consensus mode (without the flag), the limit is entirely absent. A malicious farmer can directly include a CLVM transaction with arbitrarily deep softfork nesting that the mempool would reject, and all consensus nodes will accept it. This is a direct structural analog to the sNote bug: a limit that is enforced in one context (mempool) but not enforced cumulatively/universally (consensus), allowing bypass by using the unprotected path.

---

### Finding Description

In `apply_op()`, the softfork nesting depth check reads:

```rust
if self.dialect.flags().contains(ClvmFlags::LIMIT_SOFTFORK)
    && self.softfork_stack.len() >= 20
{
    return Err(EvalErr::SoftforkStackDepthExceeded);
}
```

The entire guard is conditional on `ClvmFlags::LIMIT_SOFTFORK`. This flag is defined as part of `MEMPOOL_MODE` only:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

In consensus mode (the flag is absent), the check is never reached. Every call to `apply_op` that encounters the `softfork` opcode unconditionally pushes a new `SoftforkGuard` onto `self.softfork_stack` and an `Operation::ExitGuard` onto `self.op_stack`, with no depth ceiling:

```rust
self.softfork_stack.push(SoftforkGuard {
    expected_cost: current_cost + expected_cost,
    allocator_state: self.allocator.checkpoint(),
    operator_set: ext,
    ...
});
self.op_stack.push(Operation::ExitGuard);
```

Each `SoftforkGuard` holds an `allocator_state: Checkpoint` (a snapshot of the allocator's internal vector lengths) and an `expected_cost`. The stacks grow proportionally to nesting depth. The only bound on depth in consensus mode is `max_cost / min_cost_per_softfork_level` (approximately `max_cost / 241`).

The existing test explicitly confirms this behavior is reachable:

```rust
#[case::over_limit_without_flag(21, ClvmFlags::empty(), "")]
fn test_limit_softfork_stack(...)
```

Depth 21 succeeds with no error when `LIMIT_SOFTFORK` is absent.

---

### Impact Explanation

**Mempool/consensus discrepancy**: A CLVM program with more than 20 nested `softfork` invocations is rejected by the mempool (which uses `MEMPOOL_MODE` including `LIMIT_SOFTFORK`) but accepted by all consensus nodes (which do not set `LIMIT_SOFTFORK`). A malicious farmer can bypass the mempool entirely and include such a transaction directly in a block. All consensus nodes will process and accept it.

**Memory amplification**: Each nesting level adds one `SoftforkGuard` (~50 bytes including the `Checkpoint`) to `softfork_stack` and one `Operation::ExitGuard` to `op_stack`. These are not freed until the corresponding `ExitGuard` operations execute. With Chia's block cost limit (~11 billion), the theoretical maximum depth is ~45 million levels, consuming gigabytes of memory on consensus nodes — memory usage that is not accurately reflected in the cost model.

**Corrupted invariant**: The invariant "softfork nesting depth ≤ 20" is enforced only in mempool mode. In consensus mode, the limit does not exist. This is structurally identical to the sNote bug: a limit checked in one context (per-call / mempool) that is not enforced in the other context (cumulative / consensus).

---

### Likelihood Explanation

Exploitation requires a malicious farmer willing to pay the CLVM cost of deeply nested softforks and to bypass the mempool by constructing a block directly. This is a realistic threat model for a motivated adversary on the Chia network. The attacker-controlled entry path is: craft a CLVM program with >20 nested `softfork` invocations, submit it directly to a farmer (or be the farmer), and have it included in a block. All consensus nodes running `run_program` without `LIMIT_SOFTFORK` will process it without error.

---

### Recommendation

Remove the `ClvmFlags::LIMIT_SOFTFORK` gate from the depth check, or enforce an equivalent depth ceiling unconditionally in `apply_op`. If the limit is intentionally absent in consensus mode for backward compatibility, the cost model must be updated so that each softfork nesting level carries a cost proportional to the memory it consumes (i.e., `GUARD_COST` must be raised to make deep nesting prohibitively expensive within the block cost limit).
