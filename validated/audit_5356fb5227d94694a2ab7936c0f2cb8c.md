### Title
`CongestionInfoV1.allowed_shard: u16` Silently Truncates `ShardId` (u64) After Dynamic Resharding Produces IDs > 65535, Breaking Deadlock-Prevention Invariant - (File: `core/primitives/src/congestion_info.rs`)

---

### Summary

`CongestionInfoV1`, the Borsh-serialized congestion record embedded in every chunk header, stores the "allowed shard" field as `u16` (max 65 535). `ShardId` is `u64`. The `Into<u16> for ShardId` implementation performs a **silent truncating cast** (`self.0 as u16`). Dynamic resharding (`ShardLayoutV3::derive_impl`) allocates new shard IDs as `max_shard_id + 1` and `max_shard_id + 2`, so IDs grow monotonically with each split. Once any live shard ID exceeds 65 535, every call to `finalize_allowed_shard` writes a wrong value into the Borsh-serialized chunk header, and the comparison in `outgoing_gas_limit` / `outgoing_size_limit` can never match the real shard, permanently disabling the deadlock-prevention guarantee of congestion control.

---

### Finding Description

**Type narrowing — the exact analog of the external report**

The external report describes a `uint256 → uint64` narrowing of `calculatedPayout` that silently overflows when a high-decimal stablecoin is used. The nearcore analog is a `u64 → u16` narrowing of `allowed_shard` that silently truncates when dynamic resharding produces large shard IDs.

    fn into(self) -> u16 {
        self.0 as u16   // ← silent truncation, no bounds check

}

**`finalize_allowed_shard` silently truncates the selected shard ID**

**The broken comparison in `outgoing_gas_limit` and `outgoing_size_limit`**

```rust
// core/primitives/src/congestion_info.rs:83-89
if Self::is_fully_congested(congestion) {
    if sender_shard == ShardId::from(self.info.allowed_shard()) {
        self.config.allowed_shard_outgoing_gas
    } else {
        Gas::ZERO   // ← every shard hits this branch when allowed_shard is truncated
    }
}
``` [5](#0-4) 

`self.info.allowed_shard()` returns the truncated `u16`, which is widened back to `ShardId` via `From<u16>`. If the real shard ID was, say, `65537`, the stored value is `1` (65537 & 0xFFFF), and `ShardId::from(1u16)` is `ShardId(1)` — a completely different shard. The comparison never matches the actual allowed shard, so every sender receives `Gas::ZERO` when the receiver is fully congested.

**Dynamic resharding allocates monotonically growing shard IDs**

```rust
// core/primitives/src/shard_layout/v3.rs:271-273
let max_shard_id =
    *shard_ids.iter().max().expect("there should always be at least one shard");
let new_shards = vec![max_shard_id + 1, max_shard_id + 2];
``` [6](#0-5) 

Each split increments the maximum shard ID by 2. Starting from any initial layout, after enough splits the maximum shard ID exceeds `u16::MAX` (65 535). At that point every `finalize_allowed_shard` call writes a wrong value into the Borsh-serialized chunk header.

---

### Impact Explanation

When a shard reaches 100% congestion (`congestion_level == 1.0`), the protocol's deadlock-prevention guarantee requires exactly one "allowed shard" to be able to forward `allowed_shard_outgoing_gas` to the congested shard per chunk. This prevents the network from reaching a state where no shard can make progress.

With the truncated `allowed_shard` value in `CongestionInfoV1`, no sender shard ever matches the stored (wrong) value. Every sender receives `Gas::ZERO` from `outgoing_gas_limit`. The congested shard can never drain its delayed-receipt queue because no new gas can enter it. This is a **permanent, protocol-level DoS** for the congested shard: it is frozen in a fully-congested state with no path to recovery, and the broken value is Borsh-serialized into every subsequent chunk header, making the corruption self-perpetuating across the network.

---

### Likelihood Explanation

The trigger requires shard IDs to exceed 65 535. With `ShardLayoutV3::derive_impl` allocating `max + 1` and `max + 2` per split, and starting from mainnet's current small shard IDs, this requires approximately 32 767 splits. The `max_number_of_shards` config parameter bounds the total shard count but not the shard ID values — IDs grow monotonically regardless of how many shards are retired. If `max_number_of_shards` is set to a value larger than ~32 767, or if the initial shard IDs are already large, the threshold is reached sooner. The vulnerability is latent in the current codebase and will be triggered as dynamic resharding is used over time. There is no bounds check, no compile-time guard, and no runtime assertion anywhere in the `finalize_allowed_shard` → `set_allowed_shard` → Borsh-serialize path.

---

### Recommendation

1. **Widen `CongestionInfoV1.allowed_shard` to `u64`** (or at minimum `u32`). Because `CongestionInfoV1` is Borsh-serialized in the chunk header and is a `ProtocolSchema`-checked type, this requires introducing `CongestionInfoV2` with the wider field and a versioned migration, following the existing pattern for `CongestionInfo::V1`.

2. **Remove or restrict `Into<u16> for ShardId`** in `core/primitives-core/src/types.rs`. The only legitimate use is the `TrieKey::BufferedReceipt` trie-key encoding (which already has an explicit `assert!` guard). All other call sites should use the full `u64` representation.

3. **Add a checked conversion** wherever a `ShardId` must be narrowed, returning an error rather than silently truncating.

### Proof of Concept

**Step 1 — Confirm the truncating cast:**
```rust
let shard_id = ShardId::new(65537u64);  // 0x10001
let as_u16: u16 = shard_id.into();      // 0x0001 = 1  ← wrong
assert_eq!(as_u16, 1u16);
```

**Step 2 — Confirm the broken comparison:**
```rust
// Simulate what finalize_allowed_shard stores:
let real_allowed = ShardId::new(65537);
let stored: u16 = real_allowed.into();  // 1

// Simulate what outgoing_gas_limit checks:
let recovered = ShardId::from(stored);  // ShardId(1)
assert_ne!(real_allowed, recovered);    // 65537 ≠ 1 → Gas::ZERO for every sender
```

**Step 3 — Confirm shard IDs exceed u16::MAX after enough splits:**
```rust
// ShardLayoutV3::derive_impl allocates max+1, max+2 per split.
// Starting from max_shard_id = 5 (current mainnet):
// After 32,766 splits: max_shard_id = 5 + 2*32766 = 65537 > u16::MAX
let initial_max: u64 = 5;
let splits: u64 = 32_766;
let final_max = initial_max + 2 * splits;  // 65537
assert!(final_max > u16::MAX as u64);
```

At this point, `finalize_allowed_shard` writes `(65537u64 as u16) = 1` into `CongestionInfoV1.allowed_shard`, which is Borsh-serialized into the chunk header and broadcast to all peers. Every peer's `outgoing_gas_limit` check fails to match the real allowed shard, and the congested shard is permanently frozen.
