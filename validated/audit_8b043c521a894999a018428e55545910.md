### Title
Unrestricted `run_mainchain_gc` Allows Any Unprivileged Caller to Prune Mainchain Headers, Enabling Permanent Fork-Choice Corruption - (File: `contract/src/lib.rs`)

### Summary

`run_mainchain_gc` is a state-mutating public function protected only by a pause guard (`#[pause(except(roles(...)))]`). When the contract is in normal (unpaused) operation, **any unprivileged NEAR account** can call it with an arbitrary `batch_size`. This is the direct analog to the Solflare "insufficient origin validation" class: `submit_blocks` carries a `#[trusted_relayer]` caller identity check, but `run_mainchain_gc` carries no equivalent restriction, leaving a state-mutating entry point open to the world.

### Finding Description

`submit_blocks` is decorated with both `#[pause]` and `#[trusted_relayer]`, meaning only registered relayers (or accounts holding `Role::DAO` / `Role::UnrestrictedSubmitBlocks`) can call it. [1](#0-0) 

`run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`. The `#[pause]` decorator controls behavior when the contract is **paused**; it does not restrict who may call the function when the contract is in normal operation. [2](#0-1) 

The function removes mainchain headers from `headers_pool` and `mainchain_height_to_header`, then advances `mainchain_initial_blockhash` to the new oldest retained block: [3](#0-2) 

The `batch_size` parameter is attacker-controlled and is bounded only by `min(total_amount_to_remove, batch_size)`, where `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`. An attacker passing `u64::MAX` removes every header beyond `gc_threshold` in a single call. [4](#0-3) 

### Impact Explanation

The contract's own documentation explicitly identifies the consequence:

> "If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound` when it cannot walk the chain back to the common ancestor." [5](#0-4) 

The reorg walk in `reorg_chain` iterates backward through `headers_pool` until it finds the common ancestor: [6](#0-5) 

If an attacker calls `run_mainchain_gc(u64::MAX)` immediately after a fork branch is submitted but before the relayer submits the fork tip that would trigger a reorg, the common ancestor block is deleted from `headers_pool`. The subsequent `submit_blocks` call for the fork tip triggers `reorg_chain`, which panics at `PrevBlockNotFound`. Because the pruned blocks cannot be re-submitted (they are old mainchain blocks whose parent chain is now gone from storage), the reorg can **never succeed**. The contract is permanently stuck reporting a lighter chain as canonical, while the actual Bitcoin network has reorganized to the heavier fork. Every downstream SPV proof consumer that trusts `mainchain_header_to_height` or `verify_transaction_inclusion` now operates against a corrupted canonical chain mapping.

The corrupted state variable is `mainchain_tip_blockhash` (and the `mainchain_height_to_header` / `mainchain_header_to_height` indexes), which permanently point to the lighter chain. [7](#0-6) 

### Likelihood Explanation

The entry path requires no stake, no role, and no privileged key — any NEAR account can call `run_mainchain_gc`. The attack window is the period between a fork branch being stored in `headers_pool` (via `store_fork_header`) and the relayer submitting the fork tip. On a live chain with active relaying, this window is narrow but non-zero. An attacker monitoring the NEAR mempool for fork-branch submissions can front-run the fork-tip submission with a `run_mainchain_gc(u64::MAX)` call. The attack is cheap (one NEAR transaction, no deposit required) and irreversible.

### Recommendation

Add `#[trusted_relayer]` to `run_mainchain_gc`, or add an explicit caller check (`require!(env::predecessor_account_id() == env::current_account_id() || self.acl_has_role(...))`) so that only authorized relayers or DAO accounts can trigger manual GC. The internal call from `submit_blocks` (line 181) is already gated by `#[trusted_relayer]` on `submit_blocks` itself, so adding the guard to `run_mainchain_gc` does not break the internal call path. [8](#0-7) 

### Proof of Concept

1. Deploy the contract with `gc_threshold = 100` and submit 200 mainchain blocks (so 100 blocks are eligible for GC).
2. Submit a fork branch block (stored via `store_fork_header`) whose common ancestor with the mainchain is at height 50 (within the GC-eligible range).
3. Before submitting the fork tip, call `run_mainchain_gc(u64::MAX)` from any unprivileged account. This removes blocks at heights 0–99, advancing `mainchain_initial_blockhash` to height 100. The common ancestor at height 50 is now deleted from `headers_pool`.
4. Submit the fork tip. `reorg_chain` walks backward from the fork tip, reaches height 50, calls `self.headers_pool.get(&prev_block_hash)`, finds nothing, and panics with `PrevBlockNotFound`.
5. The fork can never be applied. The contract permanently reports the lighter chain as canonical. [9](#0-8)
