### Title
`ENABLE_SECP_OPS` Flag Guard Bypassed via 4-Byte Opcode Alias — (`File: src/chia_dialect.rs`)

### Summary

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths: a 4-byte opcode path (opcodes `0x13d61f00` / `0x1c3a8f00`) that carries **no flag check**, and a 1-byte opcode path (opcodes 64 / 65) that is gated by `ClvmFlags::ENABLE_SECP_OPS`. The flag guard on the 1-byte path is rendered dead code as a security control because the identical secp functions are always reachable via the 4-byte path, regardless of whether `ENABLE_SECP_OPS` is set.

### Finding Description

`ChiaDialect::op()` has two distinct dispatch branches:

**Branch 1 — 4-byte opcodes (lines 157–183):** When the operator atom is 4 bytes long, the code matches specific 32-bit values and dispatches directly to the secp functions with no flag check:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);
```

**Branch 2 — 1-byte opcodes (lines 248–249):** When the operator atom is 1 byte, opcodes 64 and 65 are gated by `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

Because Branch 1 is evaluated first (the `if op_len == 4` block returns early), and because it dispatches to the exact same `op_secp256k1_verify` / `op_secp256r1_verify` functions without any flag check, the `ENABLE_SECP_OPS` guard in Branch 2 is structurally bypassed. An attacker-controlled CLVM program that encodes the 4-byte opcode atoms `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` will invoke the secp verification functions unconditionally, regardless of the dialect's flag state.

### Impact Explanation

The `ENABLE_SECP_OPS` flag is documented as a hard-fork activation gate: "Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify)." Any caller or chain configuration that sets `ENABLE_SECP_OPS = false` to prevent secp operations from executing (e.g., before a soft-fork activates, or in a restricted execution context) will find that the gate is silently bypassed via the 4-byte opcode path. The secp functions execute at full cost (1,300,000 and 1,850,000 respectively) and perform real cryptographic verification. The flag check on opcodes 64/65 is dead code as a security control — it enforces nothing that the 4-byte path does not already allow unconditionally.

### Likelihood Explanation

The entry path requires only attacker-controlled CLVM bytes — specifically, encoding the 4-byte atom `\x13\xd6\x1f\x00` or `\x1c\x3a\x8f\x00` as an operator in a CLVM program. No privileged access, key material, or special configuration is required. Any external submitter of CLVM programs (mempool transactions, puzzle reveals) can trigger this path.

### Recommendation

The 4-byte secp opcode dispatch block should apply the same `ENABLE_SECP_OPS` flag check as the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

This ensures the flag is a consistent and complete gate for secp operations across both opcode encodings.
