### Title
Protocol Fee Revenue Permanently Lost Due to `memory` Parameter in `updateCollectedFees` — (`core/contracts/OffchainExchange.sol`)

---

### Summary

In `OffchainExchange.sol`, the `updateCollectedFees` function accepts `MarketInfo memory market` as a value-copy parameter. Any mutation to `market.collectedFees` inside the function is discarded when the function returns. As a result, `marketInfo[productId].collectedFees` is never incremented by trade fees, `dumpFees()` always observes a zero balance and never credits `X_ACCOUNT`, and all protocol fee revenue is permanently stranded.

---

### Finding Description

`matchOrders` loads the market state into a local `memory` variable and then calls `updateCollectedFees` twice — once for the taker fee and once for the maker fee: [1](#0-0) 

```solidity
function updateCollectedFees(
    uint32, /* productId */
    MarketInfo memory market,   // ← passed by value; mutations are invisible to caller
    bool, /* taker */
    int128 fee,
    int128 /* builder fee */
) internal virtual {
    market.collectedFees += fee; // ← written to a stack copy; lost on return
}
```

Because Solidity passes `memory` structs by value, the increment to `market.collectedFees` inside `updateCollectedFees` is applied to a temporary copy and is discarded the moment the function returns. The `market` variable in `matchOrders` is never modified.

After both calls the function writes the (still-original) value back to storage: [2](#0-1) 

```solidity
marketInfo[callState.productId].collectedFees = market.collectedFees; // saves original value
```

This line therefore either keeps `collectedFees` at zero (its initial value after a `dumpFees` reset) or, if `dumpFees` has never run, keeps it at whatever stale value it held before the trade. It never accumulates.

`dumpFees` skips any product whose `collectedFees` is zero: [3](#0-2) 

```solidity
MarketInfoStore memory market = marketInfo[productId];
if (market.collectedFees == 0) {
    continue;
}
spotEngine.updateBalance(quoteIds[productId], X_ACCOUNT, market.collectedFees);
market.collectedFees = 0;
marketInfo[productId] = market;
```

Because `collectedFees` is always zero, `dumpFees` never credits `X_ACCOUNT` and never resets anything. Every fee deducted from a trader's `quoteDelta` via `applyFee` is removed from the user's balance but credited to no one — the tokens remain locked in the contract with no accounting entry.

---

### Impact Explanation

Every matched order deducts a fee from the taker's (and potentially maker's) quote balance via `applyFee`, which reduces `orderInfo.quoteDelta` before it is applied to the subaccount balance. That fee amount is never credited to `X_ACCOUNT` because `collectedFees` is never updated. The cumulative effect is:

- All protocol trading fee revenue is permanently stranded — it is subtracted from user balances but credited to no account.
- `dumpFees` is rendered a no-op for every product, so the sequencer fee-distribution mechanism is completely broken.
- The discrepancy between on-chain token holdings and the sum of all tracked subaccount balances grows with every trade, creating an accounting inconsistency that can affect solvency assertions (`manualAssert`, `assertUtilization`).

---

### Likelihood Explanation

The bug is triggered unconditionally on every call to `matchOrders`. No special setup, privileged role, or unusual market condition is required. Any user submitting orders through the `Endpoint` causes the sequencer to invoke `matchOrders`, which silently drops the fee update. The bug has been present since deployment and affects 100% of trades.

---

### Recommendation

`updateCollectedFees` must not receive `market` by value if it is expected to mutate it. The simplest correct fix is to accumulate fees directly in the caller before the final storage write:

```diff
-    updateCollectedFees(callState.productId, market, true,
-        ordersInfo.taker.fee, ordersInfo.taker.builderFee);
-    updateCollectedFees(callState.productId, market, false,
-        ordersInfo.maker.fee, ordersInfo.maker.builderFee);
+    market.collectedFees += ordersInfo.taker.fee;
+    market.collectedFees += ordersInfo.maker.fee;
```

Alternatively, change the signature to pass `collectedFees` by reference (e.g., accept `int128 fee` and return the updated total, or operate directly on `marketInfo[productId].collectedFees` in storage).

---

### Proof of Concept

1. Deploy the protocol and register a spot product.
2. Submit two crossing orders through `Endpoint`; the sequencer calls `matchOrders`.
3. `applyFee` computes a non-zero taker fee and subtracts it from `ordersInfo.taker.quoteDelta`.
4. `updateCollectedFees` is called with the local `market` copy; `market.collectedFees` is incremented inside the function but the change is discarded on return.
5. `matchOrders` writes the original (zero) `market.collectedFees` back to `marketInfo[productId].collectedFees`.
6. Call `dumpFees`; observe that `marketInfo[productId].collectedFees == 0`, the `continue` branch is taken, and `X_ACCOUNT`'s balance is unchanged.
7. Repeat for N trades; `X_ACCOUNT` receives zero fees regardless of trading volume, while user balances have been reduced by the cumulative fee amount.
