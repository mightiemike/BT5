### Title
Missing Deadline Check on `MsgTriggerExitFromTier` Allows Users to Earn Excess Bonus Rewards via Delayed Transaction Execution - (File: x/tieredrewards/keeper/msg_server.go)

### Summary
`MsgTriggerExitFromTier` anchors `ExitUnlockAt` to `ctx.BlockTime()` at execution time with no deadline guard. A transaction that sits in the mempool and is included later than the user intended shifts `ExitUnlockAt` forward by the same delay, extending the bonus-accrual window and draining the rewards pool beyond the protocol's intended schedule.

### Finding Description

In `TriggerExitFromTier`, the handler unconditionally uses the block time at execution:

```go
sdkCtx := sdk.UnwrapSDKContext(ctx)
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
``` [1](#0-0) 

`TriggerExit` sets both timestamps from that single `blockTime`:

```go
func (p *Position) TriggerExit(blockTime time.Time, duration time.Duration) {
    p.ExitTriggeredAt = blockTime
    p.ExitUnlockAt = blockTime.Add(duration)
}
``` [2](#0-1) 

Bonus rewards accrue from `LastBonusAccrual` up to `ExitUnlockAt` (the cap enforced in `computeSegmentBonus`):

```go
if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
    segmentEnd = pos.ExitUnlockAt
}
``` [3](#0-2) 

`MsgTriggerExitFromTier` carries no `deadline` field:

```proto
message MsgTriggerExitFromTier {
  string owner       = 1;
  uint64 position_id = 2;
}
``` [4](#0-3) 

Because `ExitUnlockAt = blockTime_actual + tier.ExitDuration`, any delay D between the user's intended submission time T₁ and actual inclusion time T₁+D shifts `ExitUnlockAt` forward by D seconds. The bonus formula:

```go
durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
return tokens.Mul(tier.BonusApy).MulInt64(durationSeconds).QuoInt64(types.SecondsPerYear).TruncateInt()
``` [5](#0-4) 

…then pays out D × shares × tokensPerShare × BonusApy / SecondsPerYear extra tokens from the rewards pool that the protocol never intended to pay.

The same root cause applies to `MsgLockTier` with `TriggerExitImmediately = true`: [6](#0-5) 

### Impact Explanation

The concrete corrupted value is the **rewards pool balance** (`tieredrewards` module account `RewardsPoolName`). Every second of mempool delay on a `MsgTriggerExitFromTier` transaction translates directly into extra tokens transferred from the pool to the position owner when rewards are later claimed. For a position holding 1 000 000 tokens at BonusApy = 4 %:

| Delay | Extra bonus drained |
|-------|-------------------|
| 1 hour | ≈ 4.6 tokens |
| 1 day | ≈ 110 tokens |
| 1 week | ≈ 767 tokens |

Across many positions the cumulative drain can be material, shortening the pool's runway and reducing rewards available to other legitimate participants.

### Likelihood Explanation

Cosmos SDK mempools are not FIFO-guaranteed. A transaction submitted with a low fee, during a period of high network load, or when a validator set is partially offline can remain unconfirmed for minutes to hours. A user who observes their transaction is pending can also resubmit with a higher fee, causing the original to be dropped and re-submitted later — effectively controlling the delay. The entry path is a standard, unprivileged `MsgTriggerExitFromTier` signed by any position owner.

### Recommendation

Add an optional `deadline` (Unix nanoseconds or `google.protobuf.Timestamp`) to `MsgTriggerExitFromTier` and `MsgLockTier`. In the handler, reject the transaction if `ctx.BlockTime()` is after the deadline:

```go
if !msg.Deadline.IsZero() && sdkCtx.BlockTime().After(msg.Deadline) {
    return nil, errorsmod.Wrap(types.ErrDeadlineExceeded, "trigger-exit deadline passed")
}
```

A zero deadline means "no constraint", preserving backward compatibility. This mirrors the Uniswap `ensure(deadline)` modifier and the fix applied in Bond Protocol PR #54.

### Proof of Concept

1. Position owner has `LastBonusAccrual = T₀`, position holds S shares on a bonded validator.
2. Owner signs and broadcasts `MsgTriggerExitFromTier` at wall-clock time T₁, intending `ExitUnlockAt = T₁ + ExitDuration`.
3. Transaction is delayed (low fee / congestion) and included at block time T₁ + D.
4. `pos.TriggerExit(T₁+D, ExitDuration)` → `ExitUnlockAt = T₁ + D + ExitDuration`.
5. Owner later calls `MsgClaimTierRewards` at any time after `ExitUnlockAt`.
6. `computeSegmentBonus` integrates from T₀ to `T₁ + D + ExitDuration` instead of `T₁ + ExitDuration`.
7. Extra bonus paid = D × S × tokensPerShare × BonusApy / SecondsPerYear, transferred from the rewards pool to the owner — funds the protocol never intended to release.
