### Title
ML-DSA-65 Gas-Key Host Functions Use Wire-Format Key Length Instead of Trie-ID Length for Exec-Fee Calculation, Breaking Fee Parity with Transaction Path — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

When a contract adds or funds an ML-DSA-65 gas key via the host functions `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, or `promise_batch_action_transfer_to_gas_key`, the exec-fee and send-fee helpers receive `public_key_len as usize` — the raw borsh wire length of the key (1953 bytes for ML-DSA-65) — instead of the on-trie identifier length returned by `PublicKey::trie_id_len()` (33 bytes). The transaction path correctly calls `trie_id_len()`. The divergence is 1920 bytes per nonce slot, which can cause the prepaid-gas ceiling to be exceeded and the host-function call to fail, or silently overcharge burnt gas on the send side.

---

### Finding Description

**The two divergent call sites:**

**Transaction path** (`runtime/runtime/src/config.rs`, `permission_exec_fees`):
```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← 33 bytes for ML-DSA-65
    gas_key_info.num_nonces,
);
``` [1](#0-0) 

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`, `promise_batch_action_add_gas_key_with_full_access`):
```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← 1953 bytes for ML-DSA-65 (wire length)
    num_nonces,
);
``` [2](#0-1) 

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call`: [3](#0-2) 

And in the Wasmtime runner for both host functions: [4](#0-3) [5](#0-4) 

**Why the values diverge for ML-DSA-65:**

`PublicKey::len()` (wire) returns `1 + 1952 = 1953` bytes for ML-DSA-65, while `PublicKey::trie_id_len()` returns `1 + 32 = 33` bytes because the trie stores only the SHA3-256 hash of the pubkey, not the full key. [6](#0-5) 

`gas_key_add_key_exec_fee` uses `public_key_len` to compute the nonce trie-key length:
```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + size_of::<NonceIndex>();
``` [7](#0-6) 

With `public_key_len = 1953` instead of `33`, `nonce_key_len` is inflated by 1920 bytes per nonce slot. The per-byte exec fee is then multiplied by `(nonce_key_len + NONCE_VALUE_LEN) * num_nonces`, producing a prepaid-gas charge that is `1920 * gas_key_byte_exec_fee * num_nonces` larger than the transaction path charges.

The same mismatch applies to `promise_batch_action_transfer_to_gas_key`, which passes `public_key_len as usize` to both `gas_key_transfer_send_fee` and `gas_key_transfer_exec_fee`, while the transaction path passes `action.public_key.trie_id_len()`: [8](#0-7) [9](#0-8) 

**The fee-parity test does not cover ML-DSA-65:**

The existing parity test (`test_gas_key_fee_parity_full_access`) only exercises ED25519 keys, for which `len() == trie_id_len()`, so the divergence is invisible: [10](#0-9) 

**The design document explicitly flags this as a contract all callers must respect:**

> `PublicKey::trie_id_len()` is a new contract that all storage-cost code must respect. Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes. [11](#0-10) 

---

### Impact Explanation

For an ML-DSA-65 gas key with `N` nonces, the host-function path charges `~1920 × gas_key_byte_exec_fee × N` extra prepaid gas compared to the transaction path. This prepaid gas is refundable in principle, but:

1. If the inflated prepaid gas exceeds `max_total_prepaid_gas` (300 TGas on mainnet), the host-function call is rejected with `PrepaidGasExceeded`, making it **impossible** for contracts to add ML-DSA-65 gas keys with more than a small number of nonces via host functions.
2. For `TransferToGasKey` via host function, the send-fee overcharge is **burnt** (irreversible), causing contracts to permanently overpay for each ML-DSA-65 gas-key funding operation.
3. The fee parity invariant between transaction and host-function paths — explicitly tested and relied upon — is broken for ML-DSA-65 keys.

---

### Likelihood Explanation

Both `PostQuantumSignatures` (protocol version 85) and `GasKeys` must be active simultaneously. Once both are live, any deployed contract can trigger this path by calling the gas-key host functions with an ML-DSA-65 public key. No privileged role is required. The bug is latent until ML-DSA-65 gas keys see real use, but the root cause is present in the production code today.

---

### Recommendation

In all three host-function implementations, decode the public key first and call `trie_id_len()` on the decoded key before passing the length to the fee helpers:

```rust
// After: let public_key = self.get_public_key(public_key_ptr, public_key_len)?;
let decoded_pk = public_key.decode()?;
let trie_key_len = decoded_pk.trie_id_len();

let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    trie_key_len,   // ← use trie_id_len, not public_key_len
    num_nonces,
);
```

Apply the same fix to `promise_batch_action_transfer_to_gas_key` for both send and exec fee helpers. Extend `test_gas_key_fee_parity_full_access` and `test_gas_key_fee_parity_function_call` to also exercise ML-DSA-65 keys.

---

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key (1953 borsh bytes) and `num_nonces = 10`.
2. Submit the same `AddKey` action via a signed transaction.
3. Observe that `gas_burnt` and `tokens_burnt` differ between the two paths by approximately `1920 × gas_key_byte_exec_fee × 10` gas units.
4. With `num_nonces` large enough (e.g., ≥ 160 for typical `gas_key_byte` rates), the host-function call fails with `PrepaidGasExceeded` while the transaction path succeeds.

The exact divergent Borsh-level value is: host function passes `1953` to `gas_key_add_key_exec_fee`; transaction path passes `33`. The difference of `1920` bytes is the gap between `PublicKey::MLDSA65.len()` and `PublicKey::MLDSA65.trie_id_len()`.

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();

        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
