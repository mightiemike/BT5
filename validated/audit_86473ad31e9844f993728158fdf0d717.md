### Title
Wrong `VersionedConstants` Selected for Pending-Block Execution Due to Missing Version in Storage — (`crates/apollo_rpc_execution/src/lib.rs`)

### Summary

`create_block_context` always falls back to `StarknetVersion::LATEST` when building the execution context for a pending block, because the pending block number is not yet committed to storage and `get_starknet_version` returns `None` for it. If the pending block's actual protocol version differs from `LATEST` (e.g., during a protocol upgrade window), the wrong `VersionedConstants` are loaded and every RPC fee-estimation, simulation, and call against the pending block uses incorrect gas limits, syscall costs, and feature flags.

---

### Finding Description

In `create_block_context` (`crates/apollo_rpc_execution/src/lib.rs`, lines 370–373), the Starknet version is resolved as follows:

```rust
let starknet_version = storage_reader
    .begin_ro_txn()?
    .get_starknet_version(block_number)?
    .unwrap_or(StarknetVersion::LATEST);
``` [1](#0-0) 

When `maybe_pending_data` is `Some`, `block_number` is set to `block_context_number.unchecked_next()` — the pending block number, which has never been committed to storage:

```rust
Some(pending_data) => (
    block_context_number.unchecked_next(),   // ← pending block number
    pending_data.timestamp,
    ...
``` [2](#0-1) 

The storage test confirms that `get_starknet_version` returns `None` for any block number beyond the committed range:

```rust
let block_6_starknet_version =
    reader.begin_ro_txn().unwrap().get_starknet_version(BlockNumber(6)).unwrap();
assert!(block_6_starknet_version.is_none());
``` [3](#0-2) 

Because the pending block is never in storage, `get_starknet_version` always returns `None` for it, and the fallback `StarknetVersion::LATEST` is unconditionally used. The `VersionedConstants` are then selected from this fallback:

```rust
let versioned_constants = VersionedConstants::get(&starknet_version)?;
``` [4](#0-3) 

The execution `PendingData` struct carries `timestamp`, `l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`, `sequencer`, and `l1_da_mode`, but **no `starknet_version` field**. The client-side pending block (`PendingBlockOrDeprecated`) does carry a `starknet_version` string, but it is never propagated into the execution `PendingData`. [5](#0-4) 

The `VersionedConstants` differ materially across versions — for example, `invoke_tx_max_n_steps` is 3 000 000 for v0.13.0, 4 000 000 for v0.13.1, and 10 000 000 for v0.13.2+: [6](#0-5) 

---

### Impact Explanation

Any caller of `starknet_estimateFee`, `starknet_simulateTransactions`, or `starknet_call` with `block_id = "pending"` triggers `create_block_context` with `maybe_pending_data = Some(...)`. The resulting `BlockContext` is built with `VersionedConstants::LATEST` regardless of the pending block's actual protocol version. During a protocol-upgrade window — when the node binary has been updated to know about a newer version but the network's pending block is still on the older version, or vice versa — the wrong constants are used for every pending-block execution. This produces authoritative-looking but incorrect fee estimates, simulation traces, and call results.

---

### Likelihood Explanation

The divergence is triggered whenever the pending block's actual `starknet_version` differs from `StarknetVersion::LATEST`. This occurs during any protocol upgrade window in central-sync (feeder-gateway) mode: the node operator upgrades the binary (advancing `LATEST`) before the network's pending block advances its version, or the network advances first while the node binary lags. The RPC endpoint is publicly accessible and requires no special privileges.

---

### Recommendation

Propagate `starknet_version` from the client pending block through `client_pending_data_to_execution_pending_data` into the execution `PendingData` struct. In `create_block_context`, when `maybe_pending_data` is `Some`, read `starknet_version` from `pending_data` directly instead of querying storage with the uncommitted block number. The storage fallback (`unwrap_or(StarknetVersion::LATEST)`) should only apply to committed blocks whose version entry may be absent for historical reasons (pre-v0.9.1 blocks).

---

### Proof of Concept

1. Node binary is updated so `StarknetVersion::LATEST = V0_14_0`; the network's pending block is still on `V0_13_6`.
2. User calls `starknet_estimateFee([tx], [], "pending")`.
3. `create_block_context` sets `block_number = latest_committed + 1`.
4. `get_starknet_version(latest_committed + 1)` → `None` (block not in storage).
5. Fallback: `starknet_version = V0_14_0`; `VersionedConstants::get(&V0_14_0)` is loaded.
6. The actual pending block uses `V0_13_6` constants. The divergent `invoke_tx_max_n_steps`, syscall gas costs, and feature flags (e.g., `enable_stateful_compression`, `block_casm_hash_v1_declares`) cause the fee estimate to be computed under the wrong execution rules.
7. A transaction consuming, say, 5 000 000 Sierra gas steps would be estimated as valid under `V0_14_0` limits but would revert under the actual `V0_13_6` limits, or vice versa — the RPC returns an authoritative-looking wrong value.
