### Title
Double-Claim of Bonus Rewards via Duplicate Position IDs in `MsgClaimTierRewards` — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

The `ClaimTierRewards` message handler in `x/tieredrewards` loads all requested positions into an in-memory slice **before** processing any of them. If a caller submits duplicate position IDs, the second in-memory copy retains stale checkpoint state (`LastEventSeq`, `LastBonusAccrual`). When `claimRewardsAndUpdatesPositions` iterates over the slice, the second copy re-computes and re-pays the same bonus rewards that were already paid during the first iteration, draining the `RewardsPoolName` module account.

---

### Finding Description

`ClaimTierRewards` first loads every requested position from the store into a local slice, then hands the entire slice to `claimRewardsAndUpdatesPositions`:

```go
// x/tieredrewards/keeper/msg_server.go  lines 434-451
positions := make([]types.PositionState, 0, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

`claimRewardsAndUpdatesPositions` iterates the slice, calls `processEventsAndClaimBonus` (which mutates the element in place), and then persists the updated position to the store:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 110-134
for i := range positions {
    pos := &positions[i]
    ...
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    ...
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

Inside `processEventsAndClaimBonus`, the bonus for the **current open segment** is computed using `pos.LastBonusAccrual` as the segment start, and the checkpoint is advanced to `blockTime` only in the in-memory copy:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 165-215
segmentStart := pos.LastBonusAccrual
...
if bonded && val.IsBonded() {
    bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
    totalBonus = totalBonus.Add(bonus)
}
applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // advances checkpoint in-memory
``` [3](#0-2) 

The bonus is then transferred out of the module account:

```go
// x/tieredrewards/keeper/claim_rewards.go  lines 239-241
if err := k.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.RewardsPoolName, ownerAddr, bonusCoins); err != nil {
    return nil, err
}
``` [4](#0-3) 

**The broken invariant**: when `msg.PositionIds = [N, N]`, `positions[0]` and `positions[1]` are two independent copies of the same position loaded from the store at the same point in time. After the first iteration updates `positions[0]` and writes the advanced checkpoint to the store, `positions[1]` still holds the **old** `LastBonusAccrual`. The second iteration therefore recomputes the identical open-segment bonus and issues a second `SendCoinsFromModuleToAccount` call — a direct double-spend from `RewardsPoolName`.

There is no deduplication guard in the loading loop, and `validateClaimRewards` only checks ownership, not whether the same ID appears more than once. [5](#0-4) 

---

### Impact Explanation

An attacker with any active tiered-rewards position can drain the `RewardsPoolName` module account by submitting a single `MsgClaimTierRewards` transaction with the same position ID repeated `k` times. Each repetition re-pays the full open-segment bonus computed against the stale `LastBonusAccrual`. Because the bonus pool backs the enhanced APY for all stakers, exhausting it eliminates bonus rewards for every participant on the chain.

---

### Likelihood Explanation

The only prerequisite is owning a tiered-rewards position with any accrued bonus (i.e., at least one block has elapsed since the last claim). This is the normal operating state for any active position. No privileged role, governance action, or special configuration is required. The attack is a single, standard user-signed transaction.

---

### Recommendation

1. **Deduplicate position IDs** in `MsgClaimTierRewards.Validate()` (or at the top of `ClaimTierRewards`) before the loading loop, rejecting any message that contains the same ID more than once.
2. **Follow checks-effects-interactions**: instead of pre-loading all positions into a slice, load, process, and persist each position one at a time inside a single loop so the store is always authoritative before the next position is read.

---

### Proof of Concept

1. Attacker holds position ID `42` in a bonded tier with accrued bonus rewards.
2. Attacker submits `MsgClaimTierRewards{Owner: attacker, PositionIds: [42, 42, 42, ...]}` with the ID repeated `k` times.
3. The loading loop appends `k` identical copies of position `42` (all with the same stale `LastBonusAccrual = T0`) to the `positions` slice.
4. **Iteration 1**: `processEventsAndClaimBonus` computes bonus for segment `[T0, blockTime]`, sends `B` coins from `RewardsPoolName` to attacker, advances `LastBonusAccrual` to `blockTime` in `positions[0]`, writes to store.
5. **Iterations 2…k**: each uses a stale copy with `LastBonusAccrual = T0`, recomputes the same segment `[T0, blockTime]`, and sends another `B` coins from `RewardsPoolName`.
6. Attacker receives `k × B` coins; the `RewardsPoolName` account is reduced by `k × B` in a single transaction.
