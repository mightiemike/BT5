Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user — any user can bypass per-user swap allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument, which is `msg.sender` of the pool's `swap` call — the router, not the originating user. When a pool admin allowlists the router to support router-based swaps for legitimate users, every address can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, rendering the extension's access control entirely ineffective.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this `sender` value directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender = pool` and `sender = router`: [3](#0-2) 

When a pool admin calls `setAllowedToSwap(pool, router, true)` to permit router-based swaps, the check becomes `allowedSwapper[pool][router] == true` for every call through the router, regardless of who the originating user is. The actual user (`msg.sender` of the router call) is never consulted. `MetricOmmSimpleRouter.exactInput` and `exactInputSingle` both call `pool.swap(...)` directly with `msg.sender = router`: [4](#0-3) 

There is no mechanism in the router to stamp the originating user into `extensionData` or any other field that the extension could read. The existing unit tests for `SwapAllowlistExtension` only test direct pool calls (where `sender` is the actual user) and do not cover the router-mediated path: [5](#0-4) 

## Impact Explanation

`SwapAllowlistExtension` is the designated mechanism for pool curation — restricting which addresses may swap against a pool. Once the router is allowlisted (a necessary step for any curated pool that also supports router-based swaps for legitimate users), the extension's per-user gate is completely bypassed. Any unprivileged address can call `exactInputSingle` or `exactInput` through the router and execute swaps against a pool that is supposed to be restricted. This breaks the core pool curation functionality the extension is designed to provide, constituting broken core pool functionality per the allowed impact gate.

## Likelihood Explanation

Any pool that uses `SwapAllowlistExtension` to restrict swappers and also allowlists the router so that approved users can swap via the router is vulnerable. This is the natural and expected configuration for a curated pool with router integration. The pool admin has no way to simultaneously allowlist the router and enforce per-user restrictions using this extension. The bypass is unconditional, repeatable, and requires no special privileges — any address can exploit it.

## Recommendation

`SwapAllowlistExtension.beforeSwap` must check the actual initiating user, not the immediate `sender`. One approach: if `sender` is a known trusted router, read the actual user from a trusted field in `extensionData` that the router stamps with `msg.sender` before calling the pool. Alternatively, the router could expose a registry mapping `pool.swap` call context to the originating user that the extension can query. At minimum, documentation must clearly state that allowlisting the router grants all router users swap access, and a separate per-user enforcement mechanism must be provided for curated pools that use the router.

## Proof of Concept

```solidity
function testFuzz_swapAllowlistBypassViaRouter(bool zeroForOne) public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted, attacker NOT allowlisted
    vm.prank(admin);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // attacker is NOT in allowedSwapper[pool]

    // Attacker routes through router
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        zeroForOne: zeroForOne,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: attacker,
        tokenIn: zeroForOne ? token0 : token1,
        deadline: block.timestamp,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // Pool sees msg.sender = router, extension checks allowedSwapper[pool][router] = true
    // Attacker's swap executes successfully — allowlist bypassed in both directions
}
```
