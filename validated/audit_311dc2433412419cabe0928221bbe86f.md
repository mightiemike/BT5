### Title
Gateway stateless DA-mode and field-emptiness checks are absent from the consensus proposal conversion path, allowing a malicious proposer to inject OS-invalid transactions that bypass admission controls — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`, `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `StatelessTransactionValidator` in the gateway enforces several invariants that the Starknet OS itself requires — including that `nonce_data_availability_mode` and `fee_data_availability_mode` must be `DataAvailabilityMode::L1`, and that `paymaster_data` and `account_deployment_data` must be empty. These checks exist explicitly because the OS will fail any transaction that violates them. However, transactions arriving through the consensus proposal path (`validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx`) bypass every one of these checks. A malicious proposer can craft a `ConsensusTransaction` with `nonce_data_availability_mode = L2`, and every honest validator node will accept it, convert it, and forward it to the batcher for execution — with no stateless gate applied.

### Finding Description

**Gateway path (checked):**

`GenericGateway::add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)` before any conversion:

```
validate_contract_address
validate_empty_account_deployment_data   // "OS enforces this is empty"
validate_empty_paymaster_data            // "OS enforces this is empty"
validate_resource_bounds
validate_tx_size (calldata / signature / proof)
validate_nonce_data_availability_mode    // "OS enforces L1"
validate_fee_data_availability_mode      // "OS enforces L1"
validate_client_side_proving_allowed
validate_proof_facts_and_proof_consistency
validate_declare_tx (Sierra version, class size, entry-point ordering)
```

Only after all of these pass does the gateway call `convert_rpc_tx_to_internal_and_executable_txs`.

**Consensus path (unchecked):**

`handle_proposal_part` in `validate_proposal.rs` receives a `ConsensusTransaction` from the network and calls:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
```

`convert_consensus_tx_to_internal_consensus_tx` dispatches to `convert_rpc_tx_to_internal`, which computes the transaction hash and (for Declare) validates the compiled class hash — but **never calls `StatelessTransactionValidator::validate`**. All of the OS-required invariants listed above are silently skipped.

The gateway source code itself documents the invariant that is broken:

> "The Starknet OS enforces that the nonce data availability mode is L1. We add this validation here in the gateway to prevent transactions from failing the OS."

The same comment appears for `fee_data_availability_mode`, `paymaster_data`, and `account_deployment_data`. The intent is clear: these checks exist to prevent OS-failing transactions from entering the pipeline. The consensus path provides a second entry point that skips all of them.

**Exact divergent values:**

| Field | Gateway enforces | Consensus path enforces |
|---|---|---|
| `nonce_data_availability_mode` | must equal `DataAvailabilityMode::L1` | not checked |
| `fee_data_availability_mode` | must equal `DataAvailabilityMode::L1` | not checked |
| `paymaster_data` | must be empty | not checked |
| `account_deployment_data` | must be empty | not checked |
| `resource_bounds` | non-zero, ≥ `min_gas_price` | not checked |
| calldata + proof_facts length | ≤ `max_calldata_length` | not checked |
| signature length | ≤ `max_signature_length` | not checked |
| proof size | ≤ `max_proof_size` | not checked |
| `allow_client_side_proving` | enforced | not checked |

### Impact Explanation

A malicious proposer (Byzantine validator) constructs a `ConsensusTransaction::RpcTransaction` carrying `nonce_data_availability_mode = DataAvailabilityMode::L2`. Every honest validator node running `validate_proposal` will:

1. Receive the transaction in a `ProposalPart::Transactions` batch.
2. Call `convert_consensus_tx_to_internal_consensus_tx`, which computes the transaction hash binding the L2 DA mode into the hash preimage and returns an `InternalConsensusTransaction` — no error.
3. Forward the transaction to the batcher via `send_txs_for_proposal`.
4. The blockifier executes the transaction. The Starknet OS, which enforces `nonce_data_availability_mode = L1`, will fail the transaction during proof generation, producing a wrong revert result for an accepted input.

For the `allow_client_side_proving = false` case: the gateway config flag is a node-operator policy to disable client-side proving. A malicious proposer can force validator nodes to accept and execute proof-carrying transactions regardless of this flag, because `validate_client_side_proving_allowed` is never called on the consensus path.

This matches:
- **High**: Mempool/gateway/RPC admission accepts invalid transactions before sequencing.
- **Critical**: Wrong revert result from blockifier/execution logic for accepted input (OS-failing DA-mode transaction is committed to a block).

### Likelihood Explanation

The trigger requires a Byzantine proposer — a realistic threat in any BFT consensus deployment. The proposer role rotates among validators, so any single compromised or malicious validator can exploit this during its proposer turn. No special privilege beyond being a validator is required. The bypass is structural: the consensus conversion path has never had stateless validation wired into it.

### Recommendation

Apply the OS-required invariants at the conversion boundary that is shared by both paths. Concretely:

1. Move `validate_nonce_data_availability_mode`, `validate_fee_data_availability_mode`, `validate_empty_paymaster_data`, and `validate_empty_account_deployment_data` into `convert_rpc_tx_to_internal` (in `TransactionConverter`) so they are enforced regardless of which entry point is used.
2. Alternatively, add a call to `StatelessTransactionValidator::validate` inside `handle_proposal_part` before forwarding transactions to the batcher, mirroring the gateway flow.
3. The `allow_client_side_proving` policy check should also be applied in the consensus path if it is intended as a node-level control rather than a pure RPC-admission filter.

### Proof of Concept

```
// Malicious proposer constructs:
let tx = RpcTransaction::Invoke(RpcInvokeTransaction::V3(RpcInvokeTransactionV3 {
    nonce_data_availability_mode: DataAvailabilityMode::L2,  // OS-invalid
    fee_data_availability_mode:   DataAvailabilityMode::L2,  // OS-invalid
    paymaster_data: PaymasterData(vec![Felt::ONE]),           // OS-invalid
    // ... other fields valid ...
}));
let consensus_tx = ConsensusTransaction::RpcTransaction(tx);

// Proposer includes consensus_tx in a ProposalPart::Transactions batch.

// Honest validator node in handle_proposal_part:
//   convert_consensus_tx_to_internal_consensus_tx(consensus_tx)
//     -> convert_rpc_tx_to_internal(tx)
//        -> tx_without_hash.calculate_transaction_hash(&self.chain_id)  // hash binds L2 DA mode
//        -> returns Ok(InternalRpcTransaction { tx: ..., tx_hash })
//   NO StatelessTransactionValidator::validate() is called.
//   Transaction forwarded to batcher -> blockifier -> OS fails it at proof time.
```

The root cause is in `crates/apollo_transaction_converter/src/transaction_converter.rs` at `convert_rpc_tx_to_internal` (line 334–392), which is called by both the gateway (after stateless validation) and the consensus path (without stateless validation), and in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` at `handle_proposal_part` (line 598–646), which never invokes the stateless validator before forwarding transactions to the batcher.     /// The Starknet OS enforces that the nonce data availability mode is L1. We add this validation
    /// here in the gateway to prevent transactions from failing the OS.
    fn validate_nonce_data_availability_mode(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let expected_da_mode = DataAvailabilityMode::L1;
        let da_mode = *tx.nonce_data_availability_mode();
        if da_mode != expected_da_mode {
            return Err(StatelessTransactionValidatorError::InvalidDataAvailabilityMode {
                field_name: "nonce".to_string(),

        };

        &self,
        tx: ConsensusTransaction,
    ) -> TransactionConverterResult<(InternalConsensusTransaction, Option<VerifyAndStoreProofTask>)>
    {
        match tx {
            ConsensusTransaction::RpcTransaction(tx) => {
                let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
                let task = proof_data.map(|(proof_facts, proof)| {
                    self.spawn_verify_and_store_proof(proof_facts, proof)

                let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
                Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
            }
        }
    }
