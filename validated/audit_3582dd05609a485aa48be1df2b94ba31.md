### Title
Superfluous `Staking` Permission and Open-Receive Entry on Legacy `tieredrewards` Module Account — (`app/app.go`)

---

### Summary

The `maccPerms` map in `app/app.go` registers `tieredrewardstypes.ModuleName` with the `authtypes.Staking` permission, and `moduleAccsAllowedToReceiveExternalFunds` marks the same account as open to receive external funds via `MsgSend`. Both entries are explicitly annotated in the source as **"Legacy, not used anymore."** No code in the current `x/tieredrewards` module uses this module account or its `Staking` permission. The combination violates the principle of least privilege: the `Staking` permission grants the bank module authority to perform staking-related coin movements against this account, and the open-receive entry means any user can send tokens to the dead account address and lose them permanently.

---

### Finding Description

`app/app.go` declares two entries for the legacy `tieredrewards` module account:

```go
// Legacy, not used anymore. Created by the previous implementation which uses this module to staking on behalf of users.
tieredrewardstypes.ModuleName: {authtypes.Staking},
``` [1](#0-0) 

```go
// Legacy, not used anymore. Created by the previous implementation which uses this module to staking on behalf of users.
tieredrewardstypes.ModuleName: true,
``` [2](#0-1) 

The `BlockedAddrs()` function derives its blocked-address map directly from `maccPerms` and `moduleAccsAllowedToReceiveExternalFunds`:

```go
func (app *ChainApp) BlockedAddrs() map[string]bool {
    blockedAddrs := make(map[string]bool)
    for acc := range maccPerms {
        blockedAddrs[authtypes.NewModuleAddress(acc).String()] = !moduleAccsAllowedToReceiveExternalFunds[acc]
    }
    return blockedAddrs
}
``` [3](#0-2) 

Because `moduleAccsAllowedToReceiveExternalFunds[tieredrewardstypes.ModuleName]` is `true`, `BlockedAddrs()` returns `false` for `authtypes.NewModuleAddress("tieredrewards")`. The bank module therefore permits `MsgSend` to that address from any account.

The current `x/tieredrewards` keeper operates exclusively against `types.RewardsPoolName` and per-position delegator accounts derived at position creation time. No keeper method, message server, or BeginBlocker ever calls `SendCoinsFromModuleToAccount`, `SendCoinsFromModuleToModule`, `DelegateCoinsFromAccountToModule`, or `UndelegateCoinsFromModuleToAccount` with `tieredrewardstypes.ModuleName` as the source or destination. [4](#0-3) [5](#0-4) 

The `authtypes.Staking` permission is the sensitive capability: it authorises the bank module to accept `DelegateCoinsFromAccountToModule` and `UndelegateCoinsFromModuleToAccount` calls targeting this account. No current code path exercises these calls against `tieredrewardstypes.ModuleName`, making the permission entirely superfluous.

---

### Impact Explanation

**Permanent user fund loss via `MsgSend`.** Because `BlockedAddrs()` returns `false` for `authtypes.NewModuleAddress("tieredrewards")`, the bank module accepts any `MsgSend` to that address. Since no module code ever withdraws from or interacts with this account, any tokens deposited there are irrecoverably locked. The exact corrupted value is the full balance of any denomination sent to `authtypes.NewModuleAddress("tieredrewards")`.

**Superfluous `Staking` permission increases attack surface.** The `authtypes.Staking` permission means the bank module will silently permit `DelegateCoinsFromAccountToModule` and `UndelegateCoinsFromModuleToAccount` against this account if any future upgrade handler, migration, or module accidentally references `tieredrewardstypes.ModuleName` instead of `tieredrewardstypes.RewardsPoolName`. This is the direct analog to the `snap_getEntropy` finding: a sensitive capability granted to a component that does not need it.

---

### Likelihood Explanation

Low-to-medium. The `MsgSend` path is reachable by any unprivileged account submitting a standard Cosmos SDK transaction. A user who copies the module account address from a block explorer, a buggy frontend that resolves the wrong recipient, or an integrator who confuses the legacy module account with the active `rewards_pool` account can trigger permanent fund loss. The `Staking` permission exploit requires a future code change to materialise, but the open-receive path is exploitable today.

---

### Recommendation

1. Remove the `authtypes.Staking` permission from the legacy entry in `maccPerms`, replacing it with `nil` to retain the blocked-address registration without granting staking capability.
2. Remove `tieredrewardstypes.ModuleName` from `moduleAccsAllowedToReceiveExternalFunds` so that `BlockedAddrs()` returns `true` for the legacy account address, causing the bank module to reject any `MsgSend` to it.
3. If the legacy module account must remain registered (e.g., to preserve the blocked-address invariant for the existing on-chain account), keep it in `maccPerms` with `nil` permissions only.

---

### Proof of Concept

1. Derive the legacy module account address:
   ```
   addr = authtypes.NewModuleAddress("tieredrewards")
   ```
2. Submit a standard `MsgSend` from any funded account to `addr` for any amount of `basecro`.
3. Observe that the transaction succeeds (bank module does not block the send because `BlockedAddrs()["<addr>"] == false`).
4. Query the balance of `addr` — it now holds the sent tokens.
5. Confirm no message, keeper method, or BeginBlocker in `x/tieredrewards` can withdraw those tokens: the keeper only calls `SendCoinsFromModuleToAccount` with `types.RewardsPoolName`, never with `types.ModuleName`. [6](#0-5) [3](#0-2)
