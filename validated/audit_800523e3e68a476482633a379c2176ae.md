### Title
Fee-on-Transfer Token Accounting Mismatch in `handleDepositTransfer` Inflates Subaccount Credits, Causing Last Withdrawer to Fail — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential token transfers: user → Endpoint, then Endpoint → Clearinghouse. For Fee-on-Transfer (FoT) tokens, each hop deducts a fee, so the Clearinghouse receives less than `amount`. However, the slow mode transaction queued in `Endpoint.depositCollateralWithReferral` records the original `amount`, causing the subaccount to be credited more than what was actually deposited. Over time this depletes the Clearinghouse's real token balance, causing the last withdrawer to fail.

---

### Finding Description

**Root cause — two-hop transfer with no fee accounting:**

`EndpointStorage.handleDepositTransfer` (lines 111–119) performs:

```solidity
safeTransferFrom(token, from, amount);                    // hop 1: Endpoint receives amount − fee₁
safeTransferTo(token, address(clearinghouse), amount);    // hop 2: forwards the ORIGINAL amount, not amount − fee₁
``` [1](#0-0) 

For a FoT token with rate `r`:
- Hop 1: Endpoint receives `amount × (1 − r)`.
- Hop 2: Endpoint attempts to forward `amount` (the original parameter). If the Endpoint holds a pre-existing balance of this token ≥ `amount × r` (e.g., accumulated from slow-mode fee charges or prior deposits), the transfer succeeds and Clearinghouse receives `amount × (1 − r)` again (second FoT deduction). If no such buffer exists, hop 2 reverts, bricking deposits entirely for that token.

**Accounting record uses the pre-fee amount:**

`Endpoint.depositCollateralWithReferral` queues a slow-mode transaction encoding the caller-supplied `amount`, not the amount actually received by Clearinghouse:

```solidity
DepositCollateral({
    sender: subaccount,
    productId: productId,
    amount: amount          // ← original, not actual received
})
``` [2](#0-1) 

When the sequencer processes this slow-mode transaction, `Clearinghouse.depositCollateral` credits the subaccount with `amountRealized = txn.amount × multiplier`:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [3](#0-2) 

The subaccount balance in the SpotEngine is inflated by `fee₂ × multiplier` relative to what Clearinghouse actually holds.

**Withdrawal path confirms the shortfall:**

`Clearinghouse.handleWithdrawTransfer` pushes the full recorded `amount` to the WithdrawPool:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));
BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
``` [4](#0-3) 

When the Clearinghouse's real balance is exhausted by earlier withdrawals, this `safeTransfer` reverts for the last claimant.

---

### Impact Explanation

Each FoT deposit inflates the SpotEngine's internal accounting by `fee₂ × multiplier` tokens relative to the Clearinghouse's real balance. With N depositors, the Clearinghouse is short `N × fee₂` tokens. The last user(s) to call `withdrawCollateral` will receive a revert from `Clearinghouse.handleWithdrawTransfer` due to insufficient token balance — a direct loss of funds for those users. The protocol's solvency invariant (`manualAssert` checks) will also diverge from on-chain reality.

---

### Likelihood Explanation

Medium. The Nado protocol registers arbitrary ERC-20 tokens as spot products via `SpotEngine`. The `IERC20Base` interface imposes no restriction on FoT tokens. [5](#0-4) 

Any operator or governance action that lists a FoT token as a product immediately exposes every depositor of that token to this bug. The trigger is a standard `depositCollateral` or `depositCollateralWithReferral` call — no special permissions required.

---

### Recommendation

Measure the actual amount received by Clearinghouse using a balance snapshot, and pass that value into the slow-mode transaction instead of the caller-supplied `amount`:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferTo(token, address(clearinghouse), token.balanceOf(address(this)));
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then in `depositCollateralWithReferral`, use the returned `actualReceived` as the `amount` field in the queued `DepositCollateral` slow-mode transaction.

---

**Setup:** FoT token `FOT` with 1 % transfer fee is registered as product `42` in SpotEngine. Endpoint holds a small pre-existing `FOT` buffer (e.g., 100 tokens from a prior slow-mode fee charge in the same token).
