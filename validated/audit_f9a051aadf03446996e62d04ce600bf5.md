### Title
`allow_client_side_proving` Feature Flag Checked on RPC Path but Absent on Consensus/P2P Conversion Path — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

The `allow_client_side_proving` feature flag in `StatelessTransactionValidatorConfig` is enforced only in the gateway's RPC admission path. The consensus/P2P path — `TransactionConverter::convert_consensus_tx_to_internal_consensus_tx` — performs no equivalent check. When an operator sets `allow_client_side_proving = false` to disable client-side proving (e.g., during a security incident or protocol maintenance), any validator peer can still inject Invoke V3 transactions carrying non-empty `proof_facts` via the consensus channel. Those transactions bypass the feature gate, are hashed and admitted as `InternalConsensusTransaction`, and are forwarded to the batcher for execution under the blockifier's `VersionedConstants`-based `validate_proof_facts`, which is a separate, protocol-level check that may still permit the proof version.

---

### Finding Description

**RPC path — flag is checked:**

`StatelessTransactionValidator::validate` calls `validate_client_side_proving_allowed`:

```rust
fn validate_client_side_proving_allowed(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if self.config.allow_client_side_proving {
        return Ok(());
    }
    let RpcInvokeTransaction::V3(tx) = tx;
    let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
    if has_proof_data {
        return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
    }
    Ok(())
}
``` [1](#0-0) 

The config field and its description:

```rust
// If true, allows transactions with non-empty proof_facts or proof fields.
pub allow_client_side_proving: bool,
``` [2](#0-1) 

**Consensus/P2P path — flag is absent:**

`TransactionConverter::convert_consensus_tx_to_internal_consensus_tx` calls `convert_rpc_tx_to_internal` directly, with no reference to `allow_client_side_proving`:

```rust
async fn convert_consensus_tx_to_internal_consensus_tx(
    &self,
    tx: ConsensusTransaction,
) -> TransactionConverterResult<(InternalConsensusTransaction, Option<VerifyAndStoreProofTask>)>
{
    match tx {
        ConsensusTransaction::RpcTransaction(tx) => {
            let (internal_tx, proof_data) = self.convert_rpc_tx_to_internal(tx).await?;
            let task = proof_data.map(|(proof_facts, proof)| {
                self.spawn_verify_and_store_proof(proof_facts, proof)
            });
            Ok((InternalConsensusTransaction::RpcTransaction(internal_tx), task))
        }
``` [3](#0-2) 

`convert_rpc_tx_to_internal` computes the transaction hash and produces an `InternalRpcTransaction` without consulting `StatelessTransactionValidatorConfig`:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [4](#0-3) 

**What the blockifier checks instead:**

The blockifier's `validate_proof_facts` uses `VersionedConstants.os_constants.allowed_proof_versions` and `allowed_virtual_os_program_hashes` — a protocol-level allowlist, not the gateway's feature flag:

```rust
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt()) {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
``` [5](#0-4) 

In `blockifier_versioned_constants_0_14_4.json`, `allowed_proof_versions` is non-empty (`["0x50524f4f4631"]`) and `allowed_virtual_os_program_hashes` contains two hashes. So a well-formed client-side proving transaction passes the blockifier check even when `allow_client_side_proving = false` at the gateway. [6](#0-5) 

---

### Impact Explanation

When an operator sets `allow_client_side_proving = false` — the intended effect is that no transaction carrying `proof_facts` is admitted or executed by that node. The RPC path honors this. The consensus/P2P path does not. A validator peer can propose a block containing Invoke V3 transactions with non-empty `proof_facts`; the receiving node's `convert_consensus_tx_to_internal_consensus_tx` will accept, hash, and forward them to the batcher. The blockifier will then execute them if the proof version and program hash are in the current `VersionedConstants` allowlist. The feature flag is silently bypassed, and the node executes transactions it was configured to reject.

This matches the **High** impact category: *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing* — the admission gate (`allow_client_side_proving`) is present on one ingestion path and absent on the other, exactly mirroring the external report's asymmetric feature-check pattern.

---

### Likelihood Explanation

Moderate. Exploitation requires a co-operating or malicious validator peer to include client-side proving transactions in a block proposal. The `allow_client_side_proving` flag is currently `true` by default in production configs, so the bypass is only relevant when an operator explicitly disables the feature. However, the scenario where an operator disables the feature precisely because of a security concern (and then finds the consensus path still admits such transactions) is realistic and mirrors the original bug's maintenance-window scenario.

---

### Recommendation

1. **Add the feature-flag check to the consensus ingestion path.** Either pass `allow_client_side_proving` into `TransactionConverter` and check it inside `convert_rpc_tx_to_internal`, or add a separate validation step in `convert_consensus_tx_to_internal_consensus_tx` that mirrors `validate_client_side_proving_allowed`.

2. **Alternatively, document the scope explicitly.** If `allow_client_side_proving` is intentionally a gateway-only admission policy (not a node-wide execution policy), document this clearly and rename it to reflect its limited scope (e.g., `gateway_allow_client_side_proving`), so operators do not rely on it to prevent execution.

---

### Proof of Concept

1. Configure the gateway with `allow_client_side_proving = false`.
2. Attempt to submit an Invoke V3 transaction with non-empty `proof_facts` via the RPC gateway → rejected with `ClientSideProvingNotAllowed`.
3. A validator peer constructs the same transaction and includes it in a `ConsensusTransaction::RpcTransaction` block proposal.
4. The receiving node calls `convert_consensus_tx_to_internal_consensus_tx` → no `allow_client_side_proving` check is performed → `InternalConsensusTransaction` is produced with the `proof_facts` intact.
5. The batcher receives the transaction and calls `convert_internal_consensus_tx_to_executable_tx` → the blockifier's `validate_proof_facts` checks `VersionedConstants.allowed_proof_versions` (non-empty in v0.14.4) → transaction is executed.
6. The feature flag intended to disable client-side proving has been bypassed entirely via the consensus path. [1](#0-0) [3](#0-2) [7](#0-6)
