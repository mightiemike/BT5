Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Allowlist Bypass Through MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When `MetricOmmSimpleRouter` is the caller, `sender` is the router contract address, not the actual end-user. A pool admin who allowlists the router to enable router-based swaps inadvertently grants every user — including non-allowlisted ones — the ability to bypass the per-user curation policy.

## Finding Description

**Root cause — wrong actor checked:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. [1](#0-0) 

**What the pool passes as `sender`:**

`MetricOmmPool.swap` always passes `msg.sender` — the direct caller of `swap()` — as the first argument to `_beforeSwap`. [2](#0-1) 

**What the router passes to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly. The pool's `msg.sender` is therefore the **router contract**, not the end-user. The end-user's address is stored only in the router's transient callback context (`_setNextCallbackContext`) and is never forwarded to the pool or the extension. [3](#0-2) 

**Asymmetry with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the actual position owner passed explicitly by the caller), not `sender` (the liquidity adder contract). The swap-side check is bound to the wrong actor. [4](#0-3) 

**Two broken states:**

1. **Router unusable for curated pools.** If the admin allowlists specific users but not the router, those users' router transactions revert with `NotAllowedToSwap` because the extension sees the router address, which is not in the allowlist.
2. **Allowlist fully bypassed.** If the admin adds the router to the allowlist to enable router-based swaps, the check becomes `allowedSwapper[pool][router] == true`, which passes for **every** user who routes through `MetricOmmSimpleRouter`, including non-allowlisted ones.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, whitelisted market makers, or permissioned institutional traders) cannot simultaneously allow those users to trade via `MetricOmmSimpleRouter` and block non-allowlisted users from trading via the same path. If the admin resolves the usability problem by adding the router to the allowlist, the curation policy is entirely voided: any address can call `MetricOmmSimpleRouter.exactInputSingle` and trade on the pool. Non-allowlisted users gain access to pool liquidity they were explicitly excluded from, which can result in direct loss of LP assets or protocol fees if the allowlist was protecting against adverse selection, front-running, or regulatory exposure. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break" criteria.

## Likelihood Explanation

The trigger is a natural, expected administrative action: adding the router to the allowlist so that allowlisted users can use the standard periphery. Any pool admin who deploys `SwapAllowlistExtension` and also wants their users to use `MetricOmmSimpleRouter` will hit this configuration. The bypass requires no special privileges — any user with a standard EOA can call the router.

## Recommendation

The extension must check the **economic actor** (the end-user), not the intermediary. Two complementary fixes:

1. **Pass the original user through the router.** `MetricOmmSimpleRouter` should forward the originating user address in `extensionData` or as a dedicated parameter, and `SwapAllowlistExtension` should decode and check that address when `sender` is a known router.
2. **Alternatively, add an explicit `originator` parameter to `pool.swap()`.** Analogous to how `addLiquidity` separates `sender` from `owner`, the pool's `swap()` signature could accept an explicit `originator` address which the router sets to `msg.sender` before calling the pool, and which extensions check instead of `sender`.

Until fixed, pools relying on `SwapAllowlistExtension` for access control should not allowlist the router and should require users to call `pool.swap()` directly.

## Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, alice, true)  — only alice is allowed.
3. Pool admin calls setAllowedToSwap(pool, router, true) — to let alice use the router.
4. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. The router calls pool.swap(...); pool passes msg.sender = router to _beforeSwap.
6. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → passes.
7. Bob's swap executes successfully despite not being in the allowlist.
```
