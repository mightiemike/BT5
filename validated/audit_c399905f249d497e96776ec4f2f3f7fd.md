### Title
AuxPoW PoW Verification Uses Dogecoin Block's `bits` Instead of Parent Block's `bits` - (File: `contract/src/dogecoin.rs`)

### Summary
In `check_aux()`, the PoW hash of the AuxPoW parent block is validated against `block_header.bits` (the Dogecoin child block's difficulty target) instead of `aux_data.parent_block.bits` (the parent block's own difficulty target). This is a direct analog to the reported bug class: a function uses a value from the wrong context, causing incorrect verification behavior.

### Finding Description
In `check_aux()`, the final PoW check reads:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
    format!("block should have correct pow")
);
```

`pow_hash` is the hash of `aux_data.parent_block` — the Bitcoin/parent block that commits to the Dogecoin block in its coinbase. The PoW check must verify that this parent block's hash satisfies the **parent block's own difficulty target**, derived from `aux_data.parent_block.bits`. Instead, the code checks it against `block_header.bits`, which is the **Dogecoin child block's** difficulty target — a completely different value from a different context.

The non-AuxPoW path in the same function correctly uses `block_header.bits` for `block_header.block_hash_pow()`, making the inconsistency clear by contrast:

```rust
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        ...
    );
}

### Impact Explanation
When Dogecoin's difficulty (`block_header.bits`) is lower (easier) than the parent Bitcoin block's difficulty (`aux_data.parent_block.bits`), an attacker can submit a crafted parent block whose hash satisfies the Dogecoin difficulty threshold but does **not** satisfy the Bitcoin difficulty threshold. The contract accepts this as a valid AuxPoW proof, allowing a block backed by insufficient real PoW to be inserted into the canonical chain. This corrupts the canonical chain state and undermines the SPV security guarantee: downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against such a block receive a fraudulent confirmation.

**Impact: High** — canonical chain state is corrupted; fraudulent transaction inclusion proofs become possible.

### Likelihood Explanation
Dogecoin and Bitcoin operate at different difficulty levels that change independently. Whenever Dogecoin difficulty is lower than Bitcoin difficulty (a common real-world condition), the wrong-bits check is exploitable by any relayer-path participant who can submit a `(Header, Some(AuxData))` pair to `submit_blocks`. No privileged role or leaked key is required beyond relayer access, which is the normal operational entry point.

**Likelihood: High** — the condition (Dogecoin difficulty < Bitcoin difficulty) holds routinely, and the entry path is the standard relayer submission flow.

### Recommendation
Replace `block_header.bits` with `aux_data.parent_block.bits` in the AuxPoW PoW check so the parent block's hash is validated against the parent block's own declared difficulty target:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(aux_data.parent_block.bits),
    format!("block should have correct pow")
);
```

This mirrors the correct pattern used in the non-AuxPoW branch, where the hash of a block is always checked against that same block's `bits` field.

### Proof of Concept
1. Observe that Dogecoin mainnet currently has a lower difficulty than Bitcoin mainnet (i.e., `block_header.bits` encodes an easier target than `aux_data.parent_block.bits`).
2. Construct a Dogecoin block header with AuxPoW data where `aux_data.parent_block` is a synthetic Bitcoin-format block whose hash satisfies `target_from_bits(block_header.bits)` (Dogecoin difficulty) but **not** `target_from_bits(aux_data.parent_block.bits)` (Bitcoin difficulty).
3. Populate the coinbase and merkle proofs to pass all other `check_aux` checks (chain root, coinbase script, index).
4. Submit via `submit_blocks` through the relayer path.
5. The contract accepts the block at line 152 because `U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits)` passes, even though the parent block has no real Bitcoin-level PoW behind it.
6. The block is stored in `headers_pool` and may become the canonical tip, after which `verify_transaction_inclusion_v2` will confirm transactions in a block secured by insufficient PoW. [1](#0-0) [2](#0-1) [3](#0-2)
