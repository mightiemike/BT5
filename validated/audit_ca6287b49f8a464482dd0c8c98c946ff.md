### Title
`op_unknown` Cost Budget Check Applied at Pre-Multiplication Scale, Allowing Returned Cost to Exceed `max_cost` — (File: `src/more_ops.rs`)

---

### Summary

In `src/more_ops.rs`, the function `op_unknown` enforces the cost budget (`check_cost`) against the raw, pre-multiplied cost value, but the value actually returned to the `run_program` loop is `cost * (cost_multiplier + 1)`. This is a direct arithmetic semantic mismatch: the guard is applied at the wrong scale, exactly analogous to the reference report's `amount = _amount * PCT_BASE` being used where `_amount` should have been used directly.

---

### Finding Description

`op_unknown` is the handler for unknown opcodes in lenient (consensus) mode. It computes a base cost from the `cost_function` field (0–3), then checks it against `max_cost`, and only **afterwards** multiplies by `(cost_multiplier + 1)`:

```rust
// src/more_ops.rs lines 258–266
assert!(cost > 0);

check_cost(cost, max_cost)?;          // ← checks PRE-multiplied cost
cost *= cost_multiplier + 1;          // ← multiplies AFTER the check
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

The `cost_multiplier` is decoded from the opcode atom bytes via `u32_from_u8`, yielding a value up to `u32::MAX - 1`:

```rust
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    ...
};
``` [2](#0-1) 

For `cost_function = 0` (constant), the pre-multiplied cost is exactly `1`. The check `check_cost(1, max_cost)` passes for any realistic `max_cost`. The returned cost is then `1 * (cost_multiplier + 1)`, which can reach `u32::MAX = 4,294,967,295` — a value `(cost_multiplier + 1)` times larger than what was checked.

The only post-multiplication guard is an overflow check against `u32::MAX`, not against `max_cost`:

```rust
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [3](#0-2) 

This is the exact structural analog of the reference bug: a value is scaled up by a large factor **after** the validation check, so the check is applied at the wrong (too-small) scale.

---

### Impact Explanation

The `Reduction(cost, ...)` returned by `op_unknown` is consumed by `apply_op` in `run_program`, which accumulates it into `current_cost`:

```rust
let r = self.dialect.op(
    self.allocator,
    operator,
    operand_list,
    max_cost,
    current_extensions,
)?;
self.push(r.1)?;
Ok(r.0)   // ← post-multiplied cost returned to run_program loop
``` [4](#0-3) 

Because `op_unknown` passes its internal `check_cost` (checking the pre-multiplied value) and returns without error, the run_program loop receives a cost up to `u32::MAX` and adds it to `current_cost`. Two concrete consequences follow:

1. **Undercharged execution / budget bypass**: If the unknown opcode is the terminal operation in a program, the program completes successfully and reports a cost that can be orders of magnitude larger than `max_cost` — the budget check inside `op_unknown` was trivially satisfied (e.g., `check_cost(1, max_cost)`) while the actual returned cost is `~4.3 billion`.

2. **Consensus divergence**: Any implementation that correctly applies the cost check post-multiplication (e.g., a reference Python implementation or a future Rust version) will reject programs that this code accepts, or vice versa. Because the corrupted value is the `Cost` field of the `Reduction` — the canonical output of operator execution — this divergence propagates directly into block validation.

The corrupted artifact is the `Cost` component of `Reduction(cost as Cost, allocator.nil())` returned from `op_unknown`.

---

### Likelihood Explanation

This is reachable in production consensus mode. In non-mempool (lenient) mode, `NO_UNKNOWN_OPS` is not set, so `unknown_operator` dispatches to `op_unknown`:

```rust
fn unknown_operator(...) -> Response {
    if flags.contains(ClvmFlags::NO_UNKNOWN_OPS) {
        Err(EvalErr::Unimplemented(o))?
    } else {
        op_unknown(allocator, o, args, max_cost)
    }
}
``` [5](#0-4) 

The opcode bytes are fully attacker-controlled CLVM input. Crafting the trigger requires only choosing the correct byte layout for the opcode atom — no privileged access, no social engineering, no dependency on external state.

---

### Recommendation

Move `check_cost` to after the multiplication, so the budget is enforced against the value that is actually returned:

```rust
assert!(cost > 0);
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    check_cost(cost, max_cost)?;   // ← check POST-multiplied cost
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

### Proof of Concept

Craft a 5-byte unknown opcode atom with the following layout (per the documented format):

```
Byte index (reverse): | 4 | 3 | 2 | 1 | 0 |
Bytes:                [0x3F, 0xFF, 0xFF, 0xFF, 0x00]
```

- Bytes `[0..4]` = `[0x3F, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = 0x3FFFFFFF = 1,073,741,823`
- Last byte `0x00` → `cost_function = 0` (constant), 6 ignored bits = 0

Execution in `op_unknown`:
- Pre-multiplied cost = `1` (cost_function 0)
- `check_cost(1, max_cost)` → passes for any `max_cost ≥ 1`
- `cost *= 1,073,741,823 + 1` → `cost = 1,073,741,824`
- `1,073,741,824 ≤ u32::MAX` → no overflow error
- Returns `Reduction(1_073_741_824, nil)`

If `max_cost = 1_000` (a tight per-program budget), the internal check passes (`1 ≤ 1_000`), but the returned cost is `1,073,741,824` — over one million times the budget. The run_program loop accumulates this value, and the program either terminates with an inflated cost or triggers CostExceeded on the next operation, depending on whether a post-accumulation check exists — a behavior that diverges from any implementation that correctly checks the post-multiplied cost.
