### Title
Protobuf `ValidResourceBounds` Conversion Uses Zero as Sentinel, Causing Transaction Hash Divergence Across P2P Boundary — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf-to-internal conversion for `ValidResourceBounds` uses zero values of `l2_gas` and `l1_data_gas` as a sentinel to distinguish between the pre-0.13.3 `L1Gas` variant and the post-0.13.3 `AllResources` variant. This is the direct sequencer analog of the external report: the code cannot distinguish between "field not present / not set" and "field explicitly set to zero." A post-0.13.3 `AllResources` transaction with zero L2 and L1DataGas bounds is silently downgraded to `L1Gas`, producing a different transaction hash than the one the sender signed, breaking hash canonicalization across the P2P boundary.

### Finding Description

In `crates/apollo_protobuf/src/converters/transaction.rs`, the `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation first defaults the absent `l1_data_gas` field to zero (for 0.13.2 backward compatibility), then uses a zero-check on both `l2_gas` and `l1_data_gas` to decide which enum variant to produce:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The identical pattern appears in the RPC path:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)
        } else {
            Self::AllResources(...)
        }
    }
}
```

The two cases that produce identical zero-check results are:
- **Case A**: Pre-0.13.3 transaction — `l1_data_gas` absent from protobuf, `l2_gas` = 0 → correctly becomes `L1Gas`
- **Case B**: Post-0.13.3 transaction — `l1_data_gas` explicitly set to zero, `l2_gas` explicitly set to zero → incorrectly becomes `L1Gas`

The transaction hash computation in `get_invoke_transaction_v3_hash` calls `get_tip_resource_bounds_hash`, which internally calls `valid_resource_bounds_as_felts`. That function includes `L1DataGas` in the hash only for the `AllResources` variant:

```rust
if let ValidResourceBounds::AllResources(AllResourceBounds { l1_data_gas, .. }) =
    resource_bounds
{
    resource_bounds_vec.push(ResourceAsFelts::new(Resource::L1DataGas, l1_data_gas)?)
}
```

For `L1Gas`: hash input = `[tip, L1Gas, L2Gas(0)]` — 3 elements  
For `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)`: hash input = `[tip, L1Gas, L2Gas(0), L1DataGas(0)]` — 4 elements

These produce different Poseidon digests. The sender signs the `AllResources` hash; the receiving P2P node recomputes the `L1Gas` hash. The hashes diverge.

### Impact Explanation

A user submits a valid post-0.13.3 `InvokeTransactionV3` with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)`. The originating node computes `hash_A` (AllResources path, includes L1DataGas=0). The user signs `hash_A`. When the transaction is propagated via P2P, the receiving node deserializes the protobuf, applies the zero-check, produces `L1Gas`, and recomputes `hash_B` (L1Gas path, excludes L1DataGas). `hash_A ≠ hash_B`. The receiving node rejects the transaction due to hash mismatch, or stores it under the wrong hash, breaking mempool consistency and block inclusion across nodes.

This matches: **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

Any post-0.13.3 client that submits an `AllResources` transaction with zero L2 and L1DataGas bounds (e.g., an L2-gas-only transaction where L1 and L1DataGas are intentionally zeroed) triggers this path. The condition is reachable without any privileged access. The `TODO` comment in the code explicitly acknowledges the ambiguity and defers the fix.

### Recommendation

Replace the zero-value sentinel with an explicit discriminant. Options:
1. Add a boolean or enum field to the protobuf `ResourceBounds` message that explicitly encodes the variant (`L1Gas` vs `AllResources`), so the conversion does not rely on value inspection.
2. Once 0.13.2 support is dropped (as the TODO intends), treat an absent `l1_data_gas` as a hard error (`missing(...)`) rather than defaulting to zero, eliminating the ambiguity entirely.
3. In the interim, preserve the original protobuf `Option<ResourceLimits>` presence/absence as the discriminant: `None` → `L1Gas`, `Some(zero)` → `AllResources`.

### Proof of Concept

1. Construct `InvokeTransactionV3` with `resource_bounds = AllResources { l1_gas: ResourceBounds { max_amount: 100, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Compute `hash_A = get_invoke_transaction_v3_hash(tx, chain_id, version)` — this calls `valid_resource_bounds_as_felts` with `AllResources`, producing 3 resource elements including `L1DataGas(0)`.
3. Sign `hash_A`.
4. Serialize to protobuf: `l1_gas = Some(...)`, `l2_gas = Some(zero)`, `l1_data_gas = Some(zero)`.
5. On the receiving node, `TryFrom<protobuf::ResourceBounds>` fires: `l1_data_gas.unwrap_or_default()` → zero; `l2_gas` → zero; zero-check passes → `ValidResourceBounds::L1Gas(l1_gas)`.
6. Compute `hash_B = get_invoke_transaction_v3_hash(tx', chain_id, version)` — `valid_resource_bounds_as_felts` with `L1Gas` produces only 2 resource elements, omitting `L1DataGas`.
7. `hash_A ≠ hash_B`. The receiving node rejects the transaction or stores it under the wrong hash. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub fn valid_resource_bounds_as_felts(
    resource_bounds: &ValidResourceBounds,
    exclude_l1_data_gas: bool,
) -> Result<Vec<ResourceAsFelts>, StarknetApiError> {
    let mut resource_bounds_vec: Vec<_> = vec![
        ResourceAsFelts::new(Resource::L1Gas, &resource_bounds.get_l1_bounds())?,
        ResourceAsFelts::new(Resource::L2Gas, &resource_bounds.get_l2_bounds())?,
    ];
    if exclude_l1_data_gas {
        return Ok(resource_bounds_vec);
    }
    if let ValidResourceBounds::AllResources(AllResourceBounds { l1_data_gas, .. }) =
        resource_bounds
    {
        resource_bounds_vec.push(ResourceAsFelts::new(Resource::L1DataGas, l1_data_gas)?)
    }
    Ok(resource_bounds_vec)
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
}
