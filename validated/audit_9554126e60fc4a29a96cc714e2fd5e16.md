### Title
`revert_state_diff` Unconditionally Deletes `stateless_compiled_class_hash_v2` Entries for Migrated Classes, Permanently Corrupting V2 Hash Lookup After Any Block Revert — (`crates/apollo_storage/src/state/mod.rs`)

---

### Summary

The `stateless_compiled_class_hash_v2` table stores the Blake2 (V2) compiled class hash for each Cairo 1 class. It is written **once**, at class declaration time, by the class manager. When `enable_casm_hash_migration` is active, the blockifier also records migrated classes in the state diff's `class_hash_to_compiled_class_hash` map — but does **not** re-write their V2 hash to `stateless_compiled_class_hash_v2`. When `revert_state_diff` is called to undo a migration block, `delete_compiled_class_hashes_v2` iterates over every key in `class_hash_to_compiled_class_hash` and unconditionally deletes the corresponding row from `stateless_compiled_class_hash_v2`. For migrated classes this row was written in an earlier declaration block, not in the migration block. After the revert the class still exists in state, but its V2 hash is permanently gone. Any subsequent call to `get_compiled_class_hash_v2` for that class returns `StateError::MissingCompiledClassHashV2`, causing block finalization to fail and producing wrong execution results for all transactions that touch the affected class.

---

### Finding Description

**Two distinct write paths share one stateless table.**

The `stateless_compiled_class_hash_v2` table (`TableIdentifier<ClassHash, NoVersionValueWrapper<CompiledClassHash>, SimpleTable>`) has no block-number component in its key. It is written by the class manager at declaration time:

```rust
// native_blockifier/src/storage.rs  lines 210-214
for (class_hash, casm_contract_class, compiled_class_hash_v2) in undeclared_casm_contracts {
    append_txn = append_txn.append_casm(&class_hash, &casm_contract_class)?;
    append_txn =
        append_txn.set_executable_class_hash_v2(&class_hash, compiled_class_hash_v2)?;
}
``` [1](#0-0) 

When `enable_casm_hash_migration` is `true`, `finalize_block` collects classes whose on-chain compiled-class-hash is still V1 (Poseidon) and injects them into the state diff as if they were newly declared:

```rust
// native_blockifier/src/storage.rs  lines 224-229
for (class_hash, compiled_class_hash) in migrated_class_hash_to_compiled_class_hash {
    thin_state_diff
        .class_hash_to_compiled_class_hash
        .insert(class_hash, compiled_class_hash);
}
``` [2](#0-1) 

Crucially, `set_executable_class_hash_v2` is **not** called for migrated classes; their V2 hash already lives in `stateless_compiled_class_hash_v2` from the original declaration block.

**The revert path does not distinguish the two cases.**

`revert_state_diff` opens `stateless_compiled_class_hash_v2` and passes every key from `thin_state_diff.class_hash_to_compiled_class_hash` to `delete_compiled_class_hashes_v2`:

```rust
// crates/apollo_storage/src/state/mod.rs  lines 764-768
delete_compiled_class_hashes_v2(
    inner_txn,
    thin_state_diff.class_hash_to_compiled_class_hash.keys(),
    &compiled_class_hash_v2_table,
)?;
``` [3](#0-2) 

```rust

``` [4](#0-3) 

For a class that was **migrated** (not declared) in the reverted block, this deletes the row that was written in the original declaration block — a row that `revert_state_diff` has no business touching.

**Broken invariant after revert.**

After the revert:

- The class still exists in the `compiled_class_hash` table (keyed by `(ClassHash, BlockNumber)`) with its original V1 hash.
- `stateless_compiled_class_hash_v2[class_hash]` is gone.
- `ApolloReader::get_compiled_class_hash_v2` reads from this table and returns `StateError::MissingCompiledClassHashV2`:

```rust
// crates/apollo_state_reader/src/apollo_state.rs  lines 256-263
fn get_compiled_class_hash_v2(...) -> StateResult<CompiledClassHash> {
    self.read_compiled_class_hash_v2(class_hash)?
        .ok_or(StateError::MissingCompiledClassHashV2(class_hash))
}
``` [5](#0-4) 

`CasmHashMigrationData::from_state` calls `should_migrate` for every executed class hash during block finalization. `should_migrate` calls `get_compiled_class_hash_v2`. For any affected class this now returns an error, propagating through `finalize_block` and causing the block to fail:

---

### Impact Explanation

After any block revert that contained migration entries, the sequencer permanently loses the V2 compiled class hash for every migrated class. Subsequent block finalization that executes those classes fails with `MissingCompiledClassHashV2`, producing wrong execution results and wrong state/receipts for accepted transactions. This matches: **Critical — Wrong state, receipt, or revert result from blockifier/execution logic for accepted input.**

---

### Likelihood Explanation

Block reverts occur during normal re-org handling and during the `revert_blocks_and_eternal_pending` flow. Any node that reverts a block produced under a protocol version with `enable_casm_hash_migration = true` (0.14.1, 0.14.3, 0.14.4) and that block contained at least one migrated class will trigger the corruption. No privileged access is required; the trigger is ordinary network operation.

---

### Recommendation

In `revert_state_diff`, distinguish between classes that were **newly declared** in the reverted block and classes that were only **migrated**. Only delete from `stateless_compiled_class_hash_v2` for classes whose declaration block matches the reverted block number (i.e., those present in `declared_classes_block_table` with `block_number == reverted_block`). Migrated classes — whose declaration block is earlier — must not have their V2 hash deleted. Alternatively, record migration writes to `stateless_compiled_class_hash_v2` separately (with a block-scoped key or a separate migration table) so that revert can target only migration-time writes.

---

### Proof of Concept

1. Deploy a Cairo 1 contract class `C` in block `M` under a protocol version with `enable_casm_hash_migration = false`. The class manager writes `stateless_compiled_class_hash_v2[C] = blake2(C)`. The on-chain compiled class hash is V1 (Poseidon).

2. Upgrade to a protocol version with `enable_casm_hash_migration = true` (e.g., 0.14.3). Execute a transaction that calls class `C` in block `N`. `finalize_block` detects `C` needs migration and inserts `C → blake2(C)` into the state diff's `class_hash_to_compiled_class_hash`. `stateless_compiled_class_hash_v2[C]` is **not** re-written (it already exists from step 1).

3. Trigger a revert of block `N` (e.g., via re-org). `revert_state_diff` calls `delete_compiled_class_hashes_v2` with `C` as a key. `stateless_compiled_class_hash_v2[C]` is deleted.

4. Re-execute block `N` (or any subsequent block that executes class `C`). `CasmHashMigrationData::from_state` calls `should_migrate(state, C)`. `get_compiled_class_hash_v2(C)` reads `stateless_compiled_class_hash_v2[C]` → `None` → returns `StateError::MissingCompiledClassHashV2(C)`. `finalize_block` returns an error. The block cannot be committed, and any transaction touching class `C` produces a wrong result. [9](#0-8) [10](#0-9) [11](#0-10)
