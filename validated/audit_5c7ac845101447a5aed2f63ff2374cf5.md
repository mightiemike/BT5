### Title
Lack of Slippage Protection in NLP Mint and Burn — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The `mintNlp` and `burnNlp` functions in `Clearinghouse.sol` convert between quote tokens and NLP tokens using a sequencer-supplied `oraclePriceX18`. The user's signed transaction contains no minimum output amount. If the oracle price moves adversely between transaction submission and sequencer execution, users receive fewer tokens than expected and have no on-chain protection.

---

### Finding Description

The `MintNlp` struct that the user signs contains only `sender`, `quoteAmount`, and `nonce`: [1](#0-0) 

The `oraclePriceX18` is appended by the sequencer in `SignedMintNlp` and is **not** part of the user-signed payload: [2](#0-1) 

In `mintNlp`, the NLP amount the user receives is computed as:

```solidity
int128 nlpAmount = quoteAmount.div(oraclePriceX18);
``` [3](#0-2) 

There is no check that `nlpAmount` meets any user-specified minimum. The user signed only the quote amount to spend, not the minimum NLP they are willing to accept.

The same structural gap exists in `burnNlp`. The `BurnNlp` struct the user signs contains only `sender`, `nlpAmount`, and `nonce`: [4](#0-3) 

The quote returned is:

```solidity
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
quoteAmount = MathHelper.max(0, quoteAmount - burnFee);
``` [5](#0-4) 

No minimum quote output is enforced. The `ERR_SLIPPAGE_TOO_HIGH` error constant exists in the codebase but is never applied to NLP mint or burn paths: [6](#0-5) 

The sequencer sets `priceX18[NLP_PRODUCT_ID]` from `signedTx.oraclePriceX18` at execution time: [7](#0-6) 

---

### Impact Explanation

A user submitting a `MintNlp` transaction calculates the expected NLP amount based on the current oracle price. If the NLP oracle price rises before the sequencer processes the transaction, the user receives fewer NLP tokens than expected for the same `quoteAmount` spent. Symmetrically, a `BurnNlp` user expecting a certain quote return receives less if the oracle price falls. In both cases the transaction succeeds silently with a worse-than-expected outcome and no on-chain recourse. The asset delta is a direct reduction in the user's credited NLP or quote balance.

---

### Likelihood Explanation

NLP oracle price is updated per transaction by the sequencer. During volatile market conditions or periods of high sequencer queue depth, the price used at execution can differ materially from the price at submission time. Any user interacting with `mintNlp` or `burnNlp` is exposed without needing any adversarial actor — legitimate oracle price movement is sufficient.

---

### Recommendation

Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp`. Include these fields in the user-signed digest in `Verifier.sol`. Enforce them in `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp` with a revert using `ERR_SLIPPAGE_TOO_HIGH` if the computed output falls below the user's specified minimum.

---

### Proof of Concept

1. NLP oracle price is currently `1.00` USDC per NLP token.
2. User signs a `MintNlp` with `quoteAmount = 1000e18` (1000 USDC), expecting to receive 1000 NLP.
3. Before the sequencer processes the transaction, the oracle price updates to `1.10` USDC per NLP.
4. Sequencer executes `mintNlp` with `oraclePriceX18 = 1.10e18`.
5. `nlpAmount = 1000e18 / 1.10e18 ≈ 909` NLP — user receives ~91 fewer NLP than expected.
6. No revert occurs; the user's quote balance is debited the full 1000 USDC.

The signed `MintNlp` struct contains no `minNlpAmount` field, so there is no on-chain check that could protect the user from this outcome.
