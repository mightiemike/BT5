### Title
Unchecked `u64` Multiplication in `op_unknown` Cost Calculation Enables Cost Wrap-Around to Zero — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the final cost multiplication `cost *= cost_multiplier + 1` is an unchecked `u64` operation. In Rust release builds, integer overflow wraps silently. An attacker-controlled unknown opcode can be crafted so that the pre-multiplication `cost` is a precise multiple of `2^32` while `cost_multiplier + 1 = 2^32` (`cost_multiplier = u32::MAX`), causing the product to wrap to exactly `0`. The subsequent guard `if cost > u32::MAX` passes (0 ≤ u32::MAX), and the function returns `Reduction(0, nil)` — a cost of zero — for an operation that should have consumed billions of cost units. This is a direct analog of the "safe math not used" class: an unchecked arithmetic operation on an attacker-controlled value produces a semantically wrong result that bypasses the enforcement gate.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` handles unknown opcodes in non-mempool (block-validation) mode. It decodes two fields from the opcode bytes:

- `cost_multiplier: u64` — derived via `u32_from_u8`, so bounded to `[0, u32::MAX]`
- `cost_function: u8` — 2 bits selecting one of four cost formulas [1](#0-0) 

It then computes a base `cost` from the arguments (formulas 0–3), checks it against `max_cost`, and applies the multiplier: [2](#0-1) 

The critical sequence is:

```
check_cost(cost, max_cost)?;   // ensures cost ≤ max_cost — does NOT prevent overflow below
cost *= cost_multiplier + 1;   // UNCHECKED u64 multiplication
if cost > u32::MAX as u64 {    // only catches non-wrapped results
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` can be `u32::MAX = 4_294_967_295`, making `cost_multiplier + 1 = 2^32 = 4_294_967_296`. If `cost` (before multiplication) equals any multiple of `2^32`, then:

```
cost * 2^32  mod  2^64  =  0
```

The result is `0`, which satisfies `0 ≤ u32::MAX`, so the function returns `Ok(Reduction(0, nil))`.

**How to reach `cost = 2^32` with cost_function = 1 (arith-like):**

```
cost = ARITH_BASE_COST + N × ARITH_COST_PER_ARG + byte_count × ARITH_COST_PER_BYTE
     = 99 + N × 320 + byte_count × 3
``` [3](#0-2) 

With N 1-byte arguments (`byte_count = N`):

```
cost = 99 + N × 323
```

We need `99 + N × 323 ≡ 0 (mod 2^32)`. Since `gcd(323, 2^32) = 1` (323 = 17 × 19, both odd), a unique solution `N` exists modulo `2^32`. The smallest positive solution is approximately `N ≈ 13.3 million` 1-byte atoms — roughly 13 MB of serialized argument data, well within what a non-heap-limited block-validation run can allocate.

The `check_cost` guard at line 260 only ensures `cost ≤ max_cost` before the multiplication; it does not prevent the product from overflowing. The comment in `run_program.rs` itself acknowledges this concern: [4](#0-3) 

---

### Impact Explanation

An attacker submits a block containing a CLVM spend whose puzzle uses an unknown opcode crafted as described. The block validator calls `op_unknown` in non-mempool mode (block validation does not set `NO_UNKNOWN_OPS`): [5](#0-4) 

The returned cost is `0` instead of the correct value (which would be `2^32 × 2^32 = 2^64`, far exceeding any budget). The run-program loop accumulates `0` additional cost for this operator. The block's total cost is therefore underreported by billions of units. This enables:

1. **Undercharged execution**: The attacker's spend consumes real CPU/memory but is charged zero cost, allowing it to pass any cost-limit gate.
2. **Consensus divergence**: Any reference implementation or future clvm_rs build compiled in debug mode (where overflow panics) would reject the block or produce a different result, splitting consensus.
3. **Block cost limit bypass**: Multiple such spends in one block can collectively exceed the block cost limit while reporting zero cost, enabling a resource-exhaustion attack on validators.

---

### Likelihood Explanation

- `op_unknown` is reachable without any special flags in block-validation mode; `MEMPOOL_MODE` sets `NO_UNKNOWN_OPS` but block validation does not.
- The attacker controls the opcode bytes (setting `cost_multiplier = u32::MAX` and `cost_function = 1`) and the argument list length entirely through the serialized CLVM program.
- The required argument count (~13 million 1-byte atoms) is ~13 MB of data. Without `LIMIT_HEAP`, the allocator imposes no heap ceiling.
- No privileged access, compromised nodes, or social engineering is required — only the ability to submit a transaction to the network.
- The exact `N` is computable offline via modular inverse of 323 mod `2^32`.

---

### Recommendation

Replace the unchecked multiplication with a checked variant and treat overflow as an invalid opcode:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the pattern already used correctly in `op_add`'s fast path: [6](#0-5)

### Proof of Concept

**Opcode bytes** (5 bytes, `cost_function = 1`, `cost_multiplier = u32::MAX`):

```
[0xFF, 0xFF, 0xFF, 0xFF, 0x40]
 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^ cost_multiplier prefix = 0xFFFFFFFF = u32::MAX
                         ^^^^ last byte: bits[7:6] = 01 → cost_function = 1
                              bits[5:0] ignored
```

**Argument list**: A CLVM list of exactly `N` atoms each containing 1 byte, where `N` is the unique solution to `99 + 323N ≡ 0 (mod 2^32)`, i.e. `N = (2^32 - 99) × 323^{-1} mod 2^32`.

**Execution trace**:
1. `cost_multiplier = 4_294_967_295`, `cost_function = 1`
2. Loop over N 1-byte args: `cost = 99 + N×320 + N×3 = 99 + 323N = 2^32` (by construction)
3. `check_cost(2^32, max_cost)` passes (max_cost ≥ 2^32 in block-validation with typical budgets of ~11×10^9)
4. `cost *= 4_294_967_296` → `2^32 × 2^32 = 2^64` wraps to `0` in u64
5. `0 > u32::MAX` is false → returns `Ok(Reduction(0, nil))`
6. Run-program loop adds `0` to total block cost [2](#0-1) [1](#0-0) [7](#0-6)
