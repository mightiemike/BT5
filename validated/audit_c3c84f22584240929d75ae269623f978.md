### Title
Dogecoin Fork Difficulty Desynchronization: Mainchain Timestamps Used During Fork Block Validation — (File: `contract/src/dogecoin.rs`)

### Summary
When validating Dogecoin fork blocks, `get_next_work_required()` calls `blocks_getter.get_header_by_height(height_first)` to obtain the timestamp of the block one period ago. This function resolves heights exclusively through `mainchain_height_to_header`, which is not updated until after a reorg completes. For any fork block at depth ≥ 3 from the divergence point, the difficulty calculation silently uses the mainchain block's timestamp instead of the fork chain's block timestamp, producing an incorrect `expected_bits` value.

### Finding Description
In `contract/src/dogecoin.rs`, `get_next_work_required()` computes the DigiShield difficulty for each incoming block: [1](#0-0) 

For Dogecoin mainnet, `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for all non-genesis blocks, meaning `height_first = prev_block_header.block_height - 1`. The timestamp at that height is fetched via: [2](#0-1) 

The implementation of `get_header_by_height` resolves the height through `mainchain_height_to_header`: [3](#0-2) 

Fork blocks are stored in `headers_pool` via `store_fork_header`, but `mainchain_height_to_header` is only updated inside `reorg_chain`, which runs *after* all validation is complete: [4](#0-3) 

**Concrete desynchronization:** Suppose the fork diverges from the mainchain at height H. When validating a fork block at height H+k (k ≥ 3):
- `prev_block_header` is the fork block at H+k−1 (correctly fetched from `headers_pool` via `prev_block_hash`)
- `height_first = H+k−2`, which is inside the fork chain
- `get_header_by_height(H+k−2)` returns the **mainchain** block at that height, not the fork block

The developers themselves flagged this in a TODO comment at the exact call site: [2](#0-1) 

The DigiShield formula then computes: [5](#0-4) 

`actual_timespan` is computed as `fork_prev_block.time − mainchain_block_at_height_first.time`. Because the two timestamps come from different chains, this value is structurally incorrect and attacker-influenced.

### Impact Explanation
An attacker who submits a crafted fork chain can set fork block timestamps (bounded only by MTP) such that `actual_timespan` is minimized relative to the mainchain timestamp at `height_first`. The DigiShield damping formula clamps the adjustment to ±25% per block, but because Dogecoin adjusts every block, a sustained fork can compound this across many blocks. The result is that fork blocks are accepted with `expected_bits` that are easier than the protocol requires, reducing the hashpower needed to accumulate chainwork sufficient to trigger a reorg. A successful reorg corrupts `mainchain_tip_blockhash`, `mainchain_height_to_header`, and all SPV proofs derived from `verify_transaction_inclusion` / `verify_transaction_inclusion_v2`.

### Likelihood Explanation
The entry path is `submit_blocks`, callable by any staked relayer. Dogecoin's per-block retarget means the desynchronization activates on every fork block at depth ≥ 3, not just at epoch boundaries. The attacker controls fork block timestamps within MTP constraints and can choose the fork point to maximize the timestamp delta. The mainchain timestamps at `height_first` are public, so the attacker can pre-compute the exact timestamps needed. The 25%-per-block DigiShield cap limits the per-block gain but does not prevent accumulation over a long fork.

### Recommendation
Replace the `get_header_by_height` call in `get_next_work_required` (Dogecoin) with an ancestor traversal that walks the fork chain via `get_prev_header`, starting from `prev_block_header`, until reaching `height_first`. This mirrors the approach used in `zcash_get_next_work_required`, which correctly traverses the chain via `prev_block_getter.get_prev_header` rather than a height-indexed mainchain lookup.

### Proof of Concept
1. Deploy the contract with Dogecoin mainnet config (post-block 145,000, `difficulty_adjustment_interval = 1`).
2. Establish a mainchain of height H+5 with known timestamps T₀…T₅.
3. Submit a fork starting at H: fork blocks F₁, F₂ with valid PoW at the mainchain difficulty.
4. For fork block F₃ (height H+3), the contract calls `get_header_by_height(H+1)` and retrieves the **mainchain** block at H+1 with timestamp T₁.
5. Set F₂.time = T₁ + 1 (just above MTP, and very close to T₁). Then `actual_timespan = 1`, `modulated_timespan` collapses to `min_timespan = 45`, and `expected_bits` for F₃ is computed at 25% easier than the correct value.
6. Mine F₃ at this reduced difficulty. The contract accepts it because `expected_bits == block_header.bits` and the PoW hash satisfies the (easier) target.
7. Repeat for F₄, F₅, … accumulating chainwork at reduced cost until the fork's chainwork exceeds the mainchain, triggering `reorg_chain` and corrupting the canonical chain state.
