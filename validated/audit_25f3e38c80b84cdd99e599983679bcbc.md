### Title
Missing `ENABLE_SECP_OPS` Flag Guard in 4-Byte Opcode Dispatch Path Allows Secp Verification to Execute Unconditionally — (`src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` has two separate dispatch paths for secp signature-verification operators. The 1-byte opcode path (opcodes 64 / 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte opcode path (opcodes `0x13d61f00` / `0x1c3a8f00`) dispatches to the **same** `op_secp256k1_verify` / `op_secp256r1_verify` functions **without** checking `ENABLE_SECP_OPS`. Any caller that omits the flag — including the production `MEMPOOL_MODE` constant — still executes real secp cryptographic verification when the 4-byte encoding is used, defeating the guard entirely.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` contains two distinct dispatch branches for secp operators:

**4-byte path — no flag check:** [1](#0-0) 

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // ← no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // ← no ENABLE_SECP_OPS check
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**1-byte path — flag check present:** [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

Both paths call the identical underlying functions `op_secp256k1_verify` / `op_secp256r1_verify` defined in `src/secp_ops.rs`. [3](#0-2) 

The production `MEMPOOL_MODE` constant does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
    // ENABLE_SECP_OPS is absent
```

Therefore, in mempool mode:
- A program using **1-byte opcode 64** → falls through to `unknown_operator` → rejected by `NO_UNKNOWN_OPS`.
- A program using **4-byte opcode `0x13d61f00`** → dispatched directly to `op_secp256k1_verify` → real ECDSA verification executes unconditionally.

The flag `ENABLE_SECP_OPS` is documented as the gate for secp operations: [5](#0-4) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The guard exists in one path and is absent in the parallel path — a direct structural analog to the external report's missing deadline check.

---

### Impact Explanation

An attacker-controlled CLVM program submitted to a node running in `MEMPOOL_MODE` (or any configuration without `ENABLE_SECP_OPS`) can encode the secp operator as the 4-byte opcode `0x13d61f00` or `0x1c3a8f00`. The interpreter will:

1. Skip the `ENABLE_SECP_OPS` guard entirely.
2. Execute real `k256` / `p256` ECDSA verification (`op_secp256k1_verify` / `op_secp256r1_verify`).
3. Return success or `EvalErr::Secp256Failed` based on the actual cryptographic result.

This produces a **different execution outcome** from what the flag configuration mandates. A node with `ENABLE_SECP_OPS` unset is supposed to treat secp opcodes as unknown/rejected; instead it silently performs real signature verification. This is a **consensus-divergence** vector: two nodes with identical flag sets but different code versions (one that routes 4-byte opcodes through `op_unknown`, one that routes them through `op_secp256k1_verify`) will disagree on whether a coin spend is valid.

---

### Likelihood Explanation

The 4-byte encoding is a valid, parseable CLVM atom. Any attacker who can submit a CLVM program (e.g., via the mempool) can craft the 4-byte opcode. No special privileges are required. The bypass is deterministic and reproducible. Likelihood is **medium**: it requires knowledge of the opcode encoding but no other preconditions.

---

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures that regardless of which opcode encoding is used, secp verification is only reachable when the dialect explicitly enables it.

### Proof of Concept

```
; Program using 4-byte opcode 0x13d61f00 (secp256k1_verify)
; Run with MEMPOOL_MODE (ENABLE_SECP_OPS absent):
;
; 1-byte opcode 64 path:
;   (64 pubkey msg sig) → unknown_operator → EvalErr (NO_UNKNOWN_OPS)
;
; 4-byte opcode 0x13d61f00 path:
;   (0x13d61f00 pubkey msg sig) → op_secp256k1_verify → real ECDSA check
;   → succeeds if signature is valid, bypassing ENABLE_SECP_OPS guard
```

The attacker-controlled entry is the raw CLVM bytes submitted to `run_program`. The interpreter reaches `ChiaDialect::op()` at the `op_len == 4` branch, dispatches to `op_secp256k1_verify` without checking `ENABLE_SECP_OPS`, and the secp verification executes. The corrupted invariant is: "secp verification must not execute when `ENABLE_SECP_OPS` is unset." The exact corrupted result is `Ok(Reduction(1300000, nil))` (success) or `EvalErr::Secp256Failed` — neither of which should be reachable without the flag.

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
