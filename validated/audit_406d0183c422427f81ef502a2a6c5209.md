### Title
Irrevocable `deployer` Role in `ContractOwner` Enables Unauthorized Perpetual Product Delisting — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` stores a `deployer` address at initialization that retains permanent, irrevocable `onlyDeployer` privileges even after ownership is transferred to a multisig. There is no `updateDeployer()` function. The `deployer` can unilaterally call `delistProduct()`, which submits a `DelistProduct` slow-mode transaction directly to the `Endpoint`, force-settling all open perpetual positions at the current oracle price — without any multisig approval.

---

### Finding Description

In `ContractOwner.initialize()`, the caller is stored as `deployer` and ownership is immediately transferred to a separate `multisig` address: [1](#0-0) 

The `onlyDeployer` modifier permanently gates several functions to this address: [2](#0-1) 

Unlike `BaseProxyManager`, which exposes `updateSubmitter()` to allow the owner to rotate the analogous `submitter` role, `ContractOwner` has **no mechanism to revoke or transfer the `deployer` role**. The two roles (`owner` and `deployer`) are not programmatically connected: transferring or renouncing ownership has zero effect on `deployer` privileges.

The most impactful `onlyDeployer`-gated function is `delistProduct()`: [3](#0-2) 

This directly calls `_submitSlowModeTransaction` with `TransactionType.DelistProduct`. In `EndpointTx.submitSlowModeTransactionImpl`, the `DelistProduct` type is in the branch that requires `sender == owner()`: [4](#0-3) 

Since `ContractOwner` is the owner of `Endpoint`, the call succeeds. After the 3-day slow-mode delay, `Clearinghouse.delistProduct()` executes: [5](#0-4) 

This force-closes all positions in the targeted perp product by zeroing each subaccount's base balance and crediting/debiting quote at `txn.priceX18`. The deployer supplies both the `productId` and `pricesX18` array.

---

### Impact Explanation

The deployer can force-delist any perpetual product and settle all listed subaccounts' positions at an attacker-supplied price. This directly corrupts the quote balance of every affected subaccount — a concrete, irreversible asset delta. The `Clearinghouse.delistProduct` implementation confirms the price must match `_getPriceX18(txn.productId)` at execution time, but the deployer controls *when* to submit and can time the submission to a favorable oracle price. All users with open perp positions in the delisted product suffer forced settlement with no recourse.

---

### Likelihood Explanation

The deployer key is a single EOA that retains this privilege indefinitely. If the deployer key is ever compromised, rotated without revoking the stored address, or the deployer acts adversarially after handing governance to the multisig, the attack is directly executable. The 3-day slow-mode delay provides a window for detection but not prevention, since there is no on-chain mechanism to cancel a queued slow-mode transaction.

---

### Recommendation

1. Add an `updateDeployer(address newDeployer)` function gated by `onlyOwner`, mirroring `BaseProxyManager.updateSubmitter()`. [6](#0-5) 

2. After deployment is complete, the multisig should call `updateDeployer(address(0))` to permanently renounce the deployer role.
3. Consider moving `delistProduct` to `onlyOwner` since it is a high-impact, irreversible protocol action equivalent in severity to the other `onlyOwner` slow-mode submissions.

---

### Proof of Concept

1. Protocol is deployed; `ContractOwner.initialize(multisig, deployerEOA, ...)` is called.
2. `deployer = deployerEOA` is stored; ownership transfers to `multisig`. No `updateDeployer` exists.
3. `deployerEOA` calls `ContractOwner.delistProduct([perpProductId], [manipulatedPriceX18], [subaccounts])`.
4. `ContractOwner` (as `Endpoint` owner) successfully enqueues a `DelistProduct` slow-mode transaction.
5. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
6. `Clearinghouse.delistProduct` executes: every listed subaccount's perp position is zeroed and their quote balance is adjusted by `baseDelta * manipulatedPriceX18`, corrupting their collateral accounting. [7](#0-6)

    function delistProduct(bytes calldata transaction) external onlyEndpoint {
        IEndpoint.DelistProduct memory txn = abi.decode(
            transaction[1:],
            (IEndpoint.DelistProduct)
        );
        // only perp can be delisted
        require(
            productToEngine[txn.productId] == _perpEngine(),
            ERR_INVALID_PRODUCT
        );
        require(txn.priceX18 == _getPriceX18(txn.productId), ERR_INVALID_PRICE);
        IPerpEngine perpEngine = _perpEngine();
        for (uint256 i = 0; i < txn.subaccounts.length; i++) {
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                txn.productId,
                txn.subaccounts[i]
            );
            int128 baseDelta = -balance.amount;
            int128 quoteDelta = -baseDelta.mul(txn.priceX18);
            perpEngine.updateBalance(
                txn.productId,
                txn.subaccounts[i],
                baseDelta,
                quoteDelta
            );
            if (RiskHelper.isIsolatedSubaccount(txn.subaccounts[i])) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.subaccounts[i]);
            }
        }
    }
