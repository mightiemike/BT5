### Title
`RelayerManager` role cannot update relayer configuration despite documented intent - (File: `contract/src/lib.rs`)

### Summary
The `RelayerManager` role is explicitly documented as being able to "reject applications **and** update relayer config," but the `#[trusted_relayer]` macro's `config_roles` attribute excludes `Role::RelayerManager`, restricting relayer-config updates to `Role::DAO` only. This is a direct role/permission mismatch: the role designed for operational relayer management is silently unable to perform half of its stated purpose.

### Finding Description
In `contract/src/lib.rs`, the `Role` enum documents `RelayerManager` as:

> "May manage trusted relayer staking: reject applications and update relayer config." [1](#0-0) 

The `#[trusted_relayer]` macro is then applied to `impl BtcLightClient` with three distinct role sets:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    manager_roles(Role::DAO, Role::RelayerManager),
    config_roles(Role::DAO)
)]
``` [2](#0-1) 

- `manager_roles` governs functions for rejecting relayer applications — `RelayerManager` is included here.
- `config_roles` governs functions for updating the relayer configuration — `RelayerManager` is **absent**; only `Role::DAO` is listed.

The result is structurally identical to the RAACToken bug: a role (`RelayerManager` / `minter`) is granted to an operational account to manage a subsystem, but the underlying config-update functions are gated by a different, more-privileged role (`DAO` / `owner`). Any account holding only `RelayerManager` that attempts to call a `config_roles`-protected function will be rejected by the access-control guard.

### Impact Explanation
An account granted `RelayerManager` — the role explicitly designed for day-to-day relayer management — cannot update the relayer configuration. The only account that can do so is one holding `Role::DAO`, which is the contract's highest-privilege governance role. This means:

1. Relayer configuration updates are permanently coupled to the `DAO` role, defeating the purpose of the dedicated `RelayerManager` role.
2. Any operational workflow that grants `RelayerManager` to a hot key or automated manager account (the natural deployment pattern) leaves relayer config permanently frozen from that account's perspective.
3. The broken invariant is: `RelayerManager` ⊄ `config_roles`, contradicting the documented contract of the role.

### Likelihood Explanation
The `RelayerManager` role exists precisely to allow a non-DAO operational account to manage relayers. Any deployment that follows the documented role semantics and grants `RelayerManager` to an account for relayer management will immediately discover that config updates silently fail. The entry path is a standard NEAR contract call from the `RelayerManager`-holding account to the config-update function generated by `#[trusted_relayer]`.

### Recommendation
Add `Role::RelayerManager` to the `config_roles` attribute so that the role's documented capabilities match its actual permissions:

```diff
 #[trusted_relayer(
     bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
     manager_roles(Role::DAO, Role::RelayerManager),
-    config_roles(Role::DAO)
+    config_roles(Role::DAO, Role::RelayerManager)
 )]
``` [2](#0-1) 

### Proof of Concept
1. Deploy the contract and call `init`.
2. Grant `Role::RelayerManager` to account `alice` via `acl_grant_role`.
3. As `alice`, call the relayer-config update function generated by `#[trusted_relayer]` (the function gated by `config_roles`).
4. Observe the call panics with an access-control rejection, even though `alice` holds the role documented as authorised to "update relayer config."
5. Repeat step 3 as an account holding `Role::DAO` — it succeeds.

The mismatch between the role's documented contract (line 71–72) and the macro attribute (line 123) is the root cause, directly parallel to `RAACToken`'s `onlyOwner` vs. `onlyMinter` mismatch.
