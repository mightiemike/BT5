### Title
4-Byte Secp Opcode Aliases Bypass `ENABLE_SECP_OPS` Flag Gate — (`File: src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through **two independent branches** with different flag requirements. The 4-byte opcode branch (checked first) carries no flag guard at all, while the 1-byte opcode branch correctly requires `ENABLE_SECP_OPS`. Because the 4-byte branch is evaluated first and returns immediately, any caller using the 4-byte encodings `0x13d61f00` / `0x1c3a8f00` executes real secp signature verification regardless of whether `ENABLE_SECP_OPS` is set — including in `MEMPOOL_MODE`, where `ENABLE_SECP_OPS` is absent and `NO_UNKNOWN_OPS` would otherwise reject them.

---

### Finding Description

In `src/chia_dialect.rs`, `ChiaDialect::op()` contains two separate dispatch paths for the secp operators:

**Branch 1 — 4-byte opcodes, no flag check (lines 157–182):** [1](#0-0) 

```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(...);
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // no ENABLE_SECP_OPS guard
        0x1c3a8f00 => op_secp256r1_verify,   // no ENABLE_SECP_OPS guard
        _ => return unknown_operator(...),
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**Branch 2 — 1-byte opcodes, flag-gated (lines 248–249):** [2](#0-1) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The 4-byte branch is evaluated first and returns immediately. The `ENABLE_SECP_OPS` guard on opcodes 64/65 is therefore completely bypassed for any caller that uses the 4-byte encoding.

The 4-byte values are not arbitrary: the code comment confirms they were chosen so that the unknown-operator cost formula yields exactly `SECP256K1_VERIFY_COST = 1300000` and `SECP256R1_VERIFY_COST = 1850000`. [3](#0-2) 

The `MEMPOOL_MODE` constant does **not** include `ENABLE_SECP_OPS`: [4](#0-3) 

And `unknown_operator` — the function that would reject unknown ops in mempool mode — is only reached for 4-byte opcodes that do **not** match the two secp values: [5](#0-4) 

---

### Impact Explanation

An attacker-controlled CLVM program that encodes a secp call using the 4-byte opcode form (`0x13d61f00` or `0x1c3a8f00`) will:

1. **Bypass `ENABLE_SECP_OPS`** — the hard-fork flag intended to gate secp availability is silently skipped.
2. **Bypass `NO_UNKNOWN_OPS`** — in `MEMPOOL_MODE` the 4-byte path never reaches `unknown_operator`, so the program is accepted and executed rather than rejected.
3. **Execute real cryptographic verification** — `op_secp256k1_verify` / `op_secp256r1_verify` perform actual ECDSA verification; this is not a no-op. [6](#0-5) 

The concrete corrupted result is a **mempool/consensus divergence**: a node running in mempool mode with `MEMPOOL_MODE` flags accepts and executes a secp-bearing program that should be rejected, while a node using 1-byte opcode 64/65 for the same operation is correctly rejected. This divergence is reachable from any externally submitted CLVM program.

---

### Likelihood Explanation

Exploitation requires only choosing the 4-byte opcode encoding instead of the 1-byte encoding when constructing a CLVM program. No privileged access, social engineering, or dependency compromise is needed. The 4-byte values are documented in the source comment, making them trivially discoverable. Any wallet, generator, or puzzle that targets the pre-hard-fork window where `ENABLE_SECP_OPS` is absent is affected.

---

### Recommendation

Add the same `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch before calling the secp functions:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => return unknown_operator(allocator, o, argument_list, flags, max_cost),
};
```

This ensures both encodings of the secp operators are subject to the same flag gate, eliminating the divergence.

---

### Proof of Concept

Construct a CLVM program whose operator atom is the 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]` with three arguments (pubkey, msg-hash, sig). Run it under `ChiaDialect::new(MEMPOOL_MODE)`:

- **Expected**: rejected with `EvalErr::Unimplemented` (because `NO_UNKNOWN_OPS` is set and `ENABLE_SECP_OPS` is absent).
- **Actual**: `op_secp256k1_verify` executes, returns `Ok(Reduction(1300000, nil))` on a valid signature — accepted by the mempool.

The same program using 1-byte opcode `64` is correctly rejected. The two encodings of the identical operator produce opposite mempool outcomes, confirming the wiring error. [7](#0-6) [8](#0-7)

```rust
pub fn op_secp256k1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256K1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256k1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256k1_verify pubkey")?;
    let verifier = K1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256k1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is message
    let msg = atom(a, msg, "secp256k1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256k1_verify sig")?;
    let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256k1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
```
