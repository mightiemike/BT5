### Title
`ValidResourceBounds` silently downgraded from `AllResources` to `L1Gas` during P2P sync deserialization, causing execution-semantic divergence between proposer and synced nodes — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf converter `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` uses a **value-based heuristic** — checking whether `l2_gas` and `l1_data_gas` are zero — to decide whether to produce `ValidResourceBounds::L1Gas` or `ValidResourceBounds::AllResources`. This is the sequencer analog of the external report's hardcoded `MAX_TIME_WINDOW`: users (and the protocol) are forced to rely on a hardcoded implicit rule with no explicit type tag, and a valid V3 `AllResources` transaction with zero L2/data-gas bounds is silently re-typed to the legacy `L1Gas` variant after P2P sync deserialization. The proposer node executes the transaction under `AllResources` semantics; every synced node re-executes it under `L1Gas` semantics, producing divergent fee calculations, divergent `get_execution_info` syscall output (2 vs 3 resource-bound entries), and divergent RPC simulation/trace results.

---

### Finding Description

**Root cause — value-based type inference with no explicit discriminant:** [1](#0-0) 

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← defaults to zero if absent
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)                      // ← legacy variant
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
```

This converter is invoked by the P2P sync path when deserializing `InvokeTransactionV3`: [2](#0-1) 

A V3 transaction submitted through the gateway always carries `AllResourceBounds` (the `RpcInvokeTransactionV3.resource_bounds` field is typed `AllResourceBounds`): [3](#0-2) 

The gateway converts it to `InternalRpcInvokeTransactionV3`, which also stores `AllResourceBounds` and always hashes with `ValidResourceBounds::AllResources`: [4](#0-3) 

When the block is later synced via P2P, the transaction is deserialized through `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, which calls the value-based converter above. If the original transaction had `l2_gas = 0` and `l1_data_gas = 0` (a valid configuration — the gateway accepts any V3 transaction with at least one non-zero bound), the result is `ValidResourceBounds::L1Gas`, the legacy pre-0.13.3 variant.

**Divergent execution semantics:**

`ValidResourceBounds::L1Gas` and `ValidResourceBounds::AllResources` differ in three execution-critical ways:

1. **`get_execution_info` syscall output** — `valid_resource_bounds_as_felts` returns **2 entries** for `L1Gas` (L1Gas, L2Gas) vs **3 entries** for `AllResources` (L1Gas, L2Gas, L1DataGas): [5](#0-4) 

2. **Gas vector computation mode** — `L1Gas` → `GasVectorComputationMode::NoL2Gas`; `AllResources` → `GasVectorComputationMode::All`: [6](#0-5) 

3. **Resource bounds checking** — `check_resources_within_bounds` applies different logic for each variant: [7](#0-6) 

The proposer node executes the transaction with `AllResources` (3 resource-bound entries, `All` gas mode). Every synced node stores and re-executes it with `L1Gas` (2 resource-bound entries, `NoL2Gas` mode). The `get_execution_info` syscall returns a structurally different `TxInfo` to the contract on synced nodes.

---

### Impact Explanation

**High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

On any node that received the block via P2P sync:
- `starknet_simulateTransactions` and `starknet_estimateFee` re-execute the transaction with `L1Gas` semantics, returning fee estimates that differ from the actual charged fee.
- `starknet_traceTransaction` / `starknet_traceBlockTransactions` produce execution traces where the `get_execution_info` syscall returns only 2 resource-bound entries instead of 3, diverging from the canonical trace produced by the proposer.
- Any contract that reads `resource_bounds_end - resource_bounds_start` (the count of resource bounds) via `get_execution_info` will observe a different value on synced nodes, potentially causing divergent contract-level logic and wrong state in simulation.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a valid V3 invoke transaction with `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway accepts such transactions. Once included in a block and propagated via P2P sync, every synced node is affected for that transaction. No special privileges or malicious peer are required.

---

### Recommendation

Add an explicit type discriminant to the protobuf `ResourceBounds` message (e.g., a boolean `is_all_resources` field or a separate oneof), and use it in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` instead of the value-based heuristic. Alternatively, always deserialize to `AllResources` in the P2P sync path (matching the gateway's invariant that all new V3 transactions use `AllResources`), and only produce `L1Gas` when deserializing pre-0.13.3 legacy transactions identified by their block number or transaction version.

---

### Proof of Concept

1. Submit a V3 invoke transaction to the gateway with `resource_bounds = { l1_gas: { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: { max_amount: 0, max_price_per_unit: 0 }, l1_data_gas: { max_amount: 0, max_price_per_unit: 0 } }`. The gateway accepts it and assigns hash H with `AllResources` semantics.

2. The transaction is included in block B. The proposer executes it with `AllResources` (3 resource-bound entries in `get_execution_info`).

3. Block B is synced to a peer node via P2P. The peer deserializes the transaction through `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` → `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Since `l2_gas.is_zero() && l1_data_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas`.

4. Call `starknet_simulateTransactions` on the peer node for transaction H. The simulation uses `L1Gas` semantics: `valid_resource_bounds_as_felts` returns 2 entries, `get_gas_vector_computation_mode()` returns `NoL2Gas`, and `check_resources_within_bounds` applies L1-only logic. The returned fee estimate and execution trace diverge from the canonical result on the proposer node.

5. Specifically, if the contract under test calls `get_execution_info` and reads `resource_bounds_end - resource_bounds_start`, it observes `2` on the peer and `3` on the proposer — a structural divergence in the authoritative RPC response.

pub fn valid_resource_bounds_as_felts(
    resource_bounds: &ValidResourceBounds,
    exclude_l1_data_gas: bool,
) -> Result<Vec<ResourceAsFelts>, StarknetApiError> {
    let mut resource_bounds_vec: Vec<_> = vec![
        ResourceAsFelts::new(Resource::L1Gas, &resource_bounds.get_l1_bounds())?,
        ResourceAsFelts::new(Resource::L2Gas, &resource_bounds.get_l2_bounds())?,
    ];
    if exclude_l1_data_gas {
        return Ok(resource_bounds_vec);
    }
    if let ValidResourceBounds::AllResources(AllResourceBounds { l1_data_gas, .. }) =
        resource_bounds
    {
        resource_bounds_vec.push(ResourceAsFelts::new(Resource::L1DataGas, l1_data_gas)?)
    }
    Ok(resource_bounds_vec)
}
