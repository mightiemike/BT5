### Title
Checks-Effects-Interactions Violation in `withdrawCollateral` Enables Reentrancy-Based Double-Withdrawal via ERC-777 Token Callback — (File: `core/contracts/Clearinghouse.sol`)

### Summary
`Clearinghouse.withdrawCollateral` transfers tokens to the user-controlled `sendTo` address **before** debiting the corresponding balance in `SpotEngine`. If a listed token implements ERC-777 (or any transfer-hook mechanism), the recipient contract's `tokensReceived` callback fires while the SpotEngine balance is still at its pre-withdrawal value. The attacker can exploit this window by calling `Endpoint.depositCollateral` during the callback, re-crediting the balance before the debit lands, and then repeating the withdrawal — draining the `WithdrawPool` without a corresponding balance reduction.

### Finding Description

**Root cause — CEI violation in `Clearinghouse.withdrawCollateral`:** [1](#0-0) 

The execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` is called. This chains through to `BaseWithdrawPool.submitWithdrawal` → `BaseWithdrawPool.handleWithdrawTransfer` → `token.safeTransfer(sendTo, amount)`. Tokens physically leave the `WithdrawPool` and arrive at `sendTo`. [2](#0-1) [3](#0-2) 

2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)` is called **after** the transfer, debiting the subaccount balance. [4](#0-3) 

**The exploitable window** exists between steps 1 and 2. For any token that invokes a `tokensReceived` hook on the recipient (ERC-777 standard), the attacker's contract receives control at step 1 while the SpotEngine still records the full pre-withdrawal balance.

**Reentry path via `depositCollateral`:**

`DirectDepositV1.creditDeposit` demonstrates that `endpoint.depositCollateralWithReferral` is callable by any external contract without sequencer authorization: [5](#0-4) 

During the ERC-777 callback, the attacker calls `Endpoint.depositCollateral` (or `depositCollateralWithReferral`) for their own subaccount, depositing the just-received tokens back. Because no reentrancy guard exists anywhere in the call stack, this succeeds and credits the SpotEngine balance by `+amount`. When execution returns to `withdrawCollateral`, the debit of `-amount` is applied to a balance that has already been re-inflated, leaving the subaccount balance unchanged while the `WithdrawPool` has paid out `amount` tokens with no net accounting reduction.

**No reentrancy guard exists** anywhere in the observed contract stack (`Clearinghouse`, `BaseWithdrawPool`, `Endpoint`).

### Impact Explanation

An attacker who controls a subaccount and whose `sendTo` address is a contract with an ERC-777 `tokensReceived` hook can loop the withdraw → redeposit cycle to drain the entire `WithdrawPool` token balance for any affected product. The `WithdrawPool` holds real user collateral awaiting settlement; complete drainage constitutes direct theft of user funds. Severity: **High**.

### Likelihood Explanation

The precondition is that at least one listed spot product uses an ERC-777-compatible token (e.g., any token built on the ERC-777 standard, or a token with transfer hooks such as certain wrapped assets). The attacker requires no privileged keys — only a funded subaccount and a deployed receiver contract. The sequencer must process the initial withdrawal, which is a normal protocol obligation for any valid user request.

### Recommendation

1. Apply the **checks-effects-interactions** pattern in `Clearinghouse.withdrawCollateral`: call `spotEngine.updateBalance` and the health check **before** calling `handleWithdrawTransfer`.
2. Add an OpenZeppelin `ReentrancyGuard` (`nonReentrant`) modifier to `Clearinghouse.withdrawCollateral`, `BaseWithdrawPool.fastWithdrawal`, and the `Endpoint` deposit entry points.
3. Audit all token listings to flag ERC-777 or hook-bearing tokens as requiring additional guards.

### Proof of Concept

```
Initial state:
  SubaccountA (SpotEngine): 100 TOKX
  WithdrawPool: 1000 TOKX
  Attacker contract: 0 TOKX

Step 1: Attacker submits WithdrawCollateral(subaccountA, 100 TOKX, sendTo=AttackerContract)
Step 2: Sequencer processes → Clearinghouse.withdrawCollateral called
Step 3: handleWithdrawTransfer → token.safeTransfer(AttackerContract, 100)
        [ERC-777 tokensReceived fires on AttackerContract]
Step 4: Inside callback:
          AttackerContract calls Endpoint.depositCollateral(subaccountA, 100 TOKX)
          → SpotEngine.updateBalance(subaccountA, +100)
          → SubaccountA balance: 100 + 100 = 200 TOKX
Step 5: Callback returns; execution resumes in withdrawCollateral
          spotEngine.updateBalance(subaccountA, -100)
          → SubaccountA balance: 200 - 100 = 100 TOKX  ← unchanged
          getHealth check passes
Step 6: Net result after one cycle:
          SubaccountA (SpotEngine): 100 TOKX  (unchanged)
          WithdrawPool: 900 TOKX              (drained by 100)
          AttackerContract: 0 TOKX            (redeposited)
Step 7: Repeat until WithdrawPool is empty.
```

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
