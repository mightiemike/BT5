### Title
Linked Signer Can Overwrite Its Own Entry in `linkedSigners` Without Subaccount Owner Consent — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` handler inside `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner: true`. This means the currently registered linked signer — not the subaccount owner — can sign a valid `LinkSigner` transaction to replace itself with any arbitrary address. A compromised or malicious linked signer can therefore silently delegate full signing authority over a victim's subaccount to an attacker-controlled address, without the owner ever authorizing that address.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner: true` calls `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier:

```solidity
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

The result: the current linked signer can sign a `LinkSigner` transaction that writes any new address into `linkedSigners[subaccount]`. The subaccount owner's consent is never required for this mutation.

Contrast this with the **slow-mode** `LinkSigner` path, which correctly enforces `validateSender(txn.sender, sender)` — requiring the on-chain `msg.sender` to match the subaccount owner address:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The sequencer (fast) path has no equivalent ownership gate.

---

### Impact Explanation

Once the attacker's address is installed as the new linked signer it inherits all signing rights the original linked signer held. Concretely:

1. **Unauthorized withdrawal initiation.** `WithdrawCollateral` (V1) is processed with `allowLinkedSigner: true` and always passes `sendTo = address(0)`, so funds are sent to the subaccount owner's wallet — not the attacker's. This is disruptive (forced liquidation risk, position closure) but not direct theft. [5](#0-4) 

2. **Theft via adversarial order matching.** The attacker-controlled linked signer can sign `MatchOrders` / `MatchOrdersWithAmount` orders at prices far from market. If the attacker also controls the counterparty subaccount, the victim's subaccount pays above-market prices, transferring real quote value to the attacker. This is direct, quantifiable asset theft. [6](#0-5) 

3. **Persistent access.** Because the attacker can keep re-signing `LinkSigner` transactions faster than the owner can revoke them (both paths consume the owner's nonce), the attacker can maintain a window of control long enough to execute the above.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API keys in the Nado ecosystem. Any of the following realistic events gives an attacker a foothold:

- Leaked or phished API key / private key for the linked signer address.
- Compromised third-party trading bot that holds the linked signer key.
- Social-engineering of a user into setting the attacker's address as their linked signer.

Once the foothold exists, the escalation to a new linked signer (step 2 of the PoC) is a single signed transaction — no further privileges are needed.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the high 20 bytes of `sender`) should be permitted to change the linked signer. The linked signer is a delegated key, not an owner; it must not be able to re-delegate ownership-level state.

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← only owner may change linked signer
);
``` [7](#0-6) 

---

### Proof of Concept

1. Victim calls `submitSlowModeTransaction(LinkSigner{sender: victim_sub, signer: bot_address})` → `linkedSigners[victim_sub] = bot_address`.
2. Attacker compromises `bot_address` (leaked key, phishing, etc.).
3. Attacker uses `bot_address` to sign: `SignedLinkSigner{tx: {sender: victim_sub, signer: attacker_address, nonce: N}, signature: sig_from_bot}`.
4. Attacker submits the signed bytes to the sequencer (standard API call, no privilege required).
5. Sequencer includes the transaction; `processTransactionImpl` reaches the `LinkSigner` branch.
6. `validateSignedTx(..., allowLinkedSigner: true)` → verifier accepts `sig_from_bot` because `bot_address == linkedSigners[victim_sub]`.
7. `linkedSigners[victim_sub] = attacker_address` is written.
8. Attacker now signs `MatchOrders` placing victim as taker at an inflated price against attacker's own maker subaccount; sequencer matches the orders; quote balance transfers from victim to attacker. [1](#0-0) [8](#0-7)
