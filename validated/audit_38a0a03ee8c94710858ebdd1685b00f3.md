### Title
`WithdrawCollateralV2` Silently Unhandled in Slow-Mode Execution Path Causes Fee Loss and Emergency Withdrawal Failure — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`WithdrawCollateralV2` is a fully-defined transaction type in the `TransactionType` enum and is correctly dispatched in `processTransactionImpl`. However, `processSlowModeTransactionImpl` has no branch for it and falls through to `revert()`. Because `submitSlowModeTransactionImpl` does not block this type from being queued (it charges the slow-mode fee and enqueues it), any user who submits a `WithdrawCollateralV2` via the slow-mode emergency path loses their fee and has their withdrawal permanently fail.

---

### Finding Description

The `TransactionType` enum in `IEndpoint.sol` defines `WithdrawCollateralV2` as a first-class transaction type. [1](#0-0) 

`processTransactionImpl` in `EndpointTx.sol` handles it correctly, validating the compact signature, enforcing fee bounds, and calling `clearinghouse.withdrawCollateral` with the `sendTo` field. [2](#0-1) 

`processSlowModeTransactionImpl`, however, has no branch for `WithdrawCollateralV2`. After exhausting all handled types it reaches: [3](#0-2) 

Meanwhile, `submitSlowModeTransactionImpl` does not include `WithdrawCollateralV2` in the admin-only guard list and does not explicitly revert on it, so it falls through to the fee-charging branch: [4](#0-3) 

The transaction is therefore accepted, the slow-mode fee is deducted from the user, and the transaction is stored in `slowModeTxs`. When the sequencer (or the user after the 3-day delay) calls `processSlowModeTransaction`, execution reaches `revert()` unconditionally. [5](#0-4) 

---

### Impact Explanation

1. **Concrete fee loss**: The user pays `SLOW_MODE_FEE` in quote tokens at submission time. This is non-refundable; the fee is already credited to `slowModeFees` before the transaction is queued.
2. **Emergency withdrawal blocked**: The slow-mode path is the protocol's emergency withdrawal mechanism for when the sequencer is unresponsive. A user who specifically needs to withdraw to a custom `sendTo` address (the feature that distinguishes V2 from V1) cannot do so via slow mode — their only V2-capable path is the sequencer-gated fast path.
3. **Funds not at risk of theft**, but the user's ability to exercise their emergency withdrawal right using the V2 format is permanently broken for each submission attempt, with a fee penalty each time.

---

### Likelihood Explanation

The slow-mode path is documented as the user-facing emergency withdrawal route. `WithdrawCollateralV2` is the current withdrawal format (it adds `sendTo` and `appendix` fields). Any user or integration that constructs a V2 withdrawal and submits it via `submitSlowModeTransaction` — a public, permissionless function — will trigger this. No special privilege is required. The 3-day delay means the failure is not immediately visible, increasing the chance of repeated fee loss.

---

### Recommendation

Add a `WithdrawCollateralV2` branch to `processSlowModeTransactionImpl` that mirrors the logic in `processTransactionImpl` but uses the slow-mode sender validation (`validateSender`) instead of signature validation, consistent with how `WithdrawCollateral` (V1) is handled in the same function.

Alternatively, add an explicit `revert()` guard for `WithdrawCollateralV2` in `submitSlowModeTransactionImpl` (before the fee is charged) to prevent the fee-loss vector while the full handler is implemented.

---

### Proof of Concept

1. User constructs a `WithdrawCollateralV2` payload with a custom `sendTo` address.
2. User calls `Endpoint.submitSlowModeTransaction(payload)`.
3. `submitSlowModeTransactionImpl` is reached via `delegatecall`. The type is not `DepositCollateral`, not in the admin-only list → `chargeSlowModeFee` executes, deducting the fee. The transaction is stored in `slowModeTxs`.
4. After `SLOW_MODE_TX_DELAY` (3 days), user calls `executeSlowModeTransaction`.
5. `processSlowModeTransactionImpl` is invoked. No branch matches `WithdrawCollateralV2`. Execution reaches `else { revert(); }`.
6. The slow-mode transaction is consumed (deleted from `slowModeTxs`) but the withdrawal never executes. The fee is permanently lost. [6](#0-5) [7](#0-6)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
