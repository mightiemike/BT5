### Title
Stale Pending Data Fallback in `read_pending_data` Emits Zero `l1_data_gas_price`/`l2_gas_price` to `starknet_estimateFee` During Sync Gaps — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

The fallback path in `read_pending_data` always constructs a `DeprecatedPendingBlock`, which by design returns `GasPricePerToken::default()` (zero) for `l1_data_gas_price` and `l2_gas_price`. These zeros propagate through `client_pending_data_to_execution_pending_data` → `create_block_context` → `tx_execution_output_to_fee_estimation`, causing `starknet_estimateFee` with `block_id=pending` to return `NonzeroGasPrice::MIN` for both fields instead of the actual values from the latest committed block header.

### Finding Description

`read_pending_data` is triggered on every `starknet_estimateFee` call with `block_id=pending`. [1](#0-0) 

When the in-memory pending data is stale (its `parent_block_hash` does not match the latest committed block hash), the fallback constructs a `DeprecatedPendingBlock` and copies only `eth_l1_gas_price`/`strk_l1_gas_price` from the latest header. `l1_data_gas_price` and `l2_gas_price` are left at `Default::default()`. [2](#0-1) 

The `l1_data_gas_price()` and `l2_gas_price()` accessors on `PendingBlockOrDeprecated::Deprecated` unconditionally return zero: [3](#0-2) 

The conversion function `client_pending_data_to_execution_pending_data` calls these accessors, propagating zero into `ExecutionPendingData`: [4](#0-3) 

`create_block_context` receives zero and substitutes `NonzeroGasPrice::MIN` (not the actual header value): [5](#0-4) 

`tx_execution_output_to_fee_estimation` then reads these MIN-substituted prices and returns them to the caller as authoritative fee estimate fields: [6](#0-5) 

### Impact Explanation

Any unprivileged caller invoking `starknet_estimateFee` with `block_id=pending` during a sync gap receives a `FeeEstimation` response where `l1_data_gas_price` and `l2_gas_price` are `NonzeroGasPrice::MIN` (effectively 1) instead of the actual values from the latest committed block header. On mainnet, where these prices are orders of magnitude higher, the returned `overall_fee` will be severely underestimated. Callers who use this estimate to set resource bounds on V3 transactions will have their transactions rejected by the sequencer for insufficient fee.

Additionally, `l1_da_mode` is forced to `Calldata` for the `Deprecated` variant, which further distorts fee estimates for nodes operating in blob (KZG) mode. [7](#0-6) 

### Likelihood Explanation

The sync gap condition (`pending_data.block.parent_block_hash() != latest_header.block_hash`) occurs naturally and continuously during normal node operation — any time the feeder gateway has not yet delivered a new pending block after a block is committed. This is not an edge case; it is the steady-state between block commits. No special attacker action is required; any caller polling `starknet_estimateFee` on `block_id=pending` will intermittently receive wrong estimates.

### Recommendation

In the fallback branch of `read_pending_data`, construct a `PendingBlockOrDeprecated::Current(PendingBlock { ... })` instead of `DeprecatedPendingBlock`, copying `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode` from `latest_header.block_header_without_hash` alongside the already-copied fields. This ensures the synthetic pending block carries the same gas prices as the latest committed block header regardless of sync state.

### Proof of Concept

1. Store a block header with non-zero `l1_data_gas_price` (e.g., `price_in_wei = 1000`) and `l2_gas_price` (e.g., `price_in_wei = 500`) in storage.
2. Set the in-memory `pending_data` to a block whose `parent_block_hash` differs from the stored block's hash (simulating a sync gap).
3. Call `starknet_estimateFee` with `block_id=pending` via the RPC.
4. Observe that the returned `FeeEstimation.l1_data_gas_price` is `NonzeroGasPrice::MIN` (1) and `FeeEstimation.l2_gas_price` is `NonzeroGasPrice::MIN` (1), not 1000 and 500 respectively.

The existing test at `crates/apollo_rpc/src/v0_8/api/test.rs` lines 1042–1050 already exercises the stale-pending-data path for `get_block` but does not assert on `l1_data_gas_price` or `l2_gas_price`, and no equivalent test exists for `estimate_fee`. [8](#0-7)
