### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks whether the **router** is allowlisted, not the originating user. If the pool admin allowlists the router address (a natural step to enable router-mediated swaps on a curated pool), the allowlist is completely bypassed for every user.

---

### Finding Description

**Trace through the call stack:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle(params)`.
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ..., params.extensionData)` — here `msg.sender` to the pool is the **router contract**.
3. `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, forwarding the router as `sender`. [1](#0-0) 

4. `ExtensionCalling._beforeSwap` encodes and dispatches `sender` (the router) to the configured extension. [2](#0-1) 

5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is the **router**, not the originating user. [3](#0-2) 

**The wrong actor is checked.** The allowlist is keyed `allowedSwapper[pool][router]`. If the pool admin allowlists the router to enable router-mediated swaps, every user on-chain can bypass the allowlist by calling `exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` on the router.

**Contrast with `DepositAllowlistExtension`**, which correctly ignores `sender` and checks `owner` (the actual position owner), making it router-agnostic: [4](#0-3) 

The asymmetry between the two extensions means a pool admin who understands the deposit allowlist behavior will incorrectly assume the swap allowlist works the same way.

**Two concrete failure modes:**

| Scenario | Admin action | Result |
|---|---|---|
| **Bypass** | Admin allowlists the router to enable router-mediated swaps | Every user can swap through the router, allowlist is void |
| **DoS** | Admin allowlists specific EOAs only | Those EOAs cannot use the router at all; must call the pool directly |

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` is a curated pool — the admin intends to restrict swap access to specific counterparties (e.g., institutional users, KYC'd addresses, or protocol-controlled addresses). If the router is allowlisted (the natural step to enable router-mediated swaps), any unprivileged user can bypass the restriction and swap against the pool's liquidity. LP funds are exposed to unauthorized traders at the pool's oracle-anchored prices, which may be favorable to the attacker and unfavorable to LPs. This is a direct loss of LP principal through unauthorized swap execution.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap entry point. A pool admin who deploys a curated pool and wants allowlisted users to be able to use the router will naturally call `setAllowedToSwap(pool, router, true)`. The documentation and interface give no indication that this opens the pool to all users. The likelihood of this misconfiguration is high for any operator who deploys a curated pool and expects router compatibility. [5](#0-4) 

---

### Recommendation

The `beforeSwap` hook must gate the **originating user**, not the direct pool caller. Two options:

1. **Pass the real user via `extensionData`**: Require the router to encode the originating `msg.sender` into `extensionData` and have the extension decode and check it. The extension must also verify that the caller is the trusted router before trusting the encoded identity.

2. **Check `recipient` as a proxy**: For exact-input swaps the recipient is often the user, but this is not reliable for multi-hop or contract recipients.

The cleanest fix is option 1: the router encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address when `sender` is a known router, falling back to checking `sender` directly for non-router callers.

---

### Proof of Concept

```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists the router so that allowlisted users can use it.
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// Attacker (not allowlisted) calls the router directly.
// The extension sees sender == address(router), which IS allowlisted.
// The swap succeeds — allowlist is bypassed.
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: token0,
        recipient: attacker,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    })
);
// Attacker receives token1 from a pool they were never supposed to access.
