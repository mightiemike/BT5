### Title
Merkle Second-Preimage Attack via Internal Node Passed as Leaf in `verify_transaction_inclusion` - (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (v1) accepts any 32-byte `tx_id` and verifies it against the block's Merkle root using `compute_root_from_merkle_proof`. Because every internal Merkle node is also a 32-byte `double_sha256` digest — the same type as a leaf — an attacker can supply an internal node hash as `tx_id` with a proof path that is one level shorter than the real tree depth. The verification passes, causing the contract to attest that a non-existent transaction was included in a confirmed Bitcoin block.

### Finding Description

`compute_hash` in `merkle-tools/src/lib.rs` concatenates two 32-byte hashes and applies `double_sha256`:

```rust
fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);
    double_sha256(&concat_inputs)
}
``` [1](#0-0) 

The output is a 32-byte `H256`, identical in type and size to a leaf (transaction hash). `compute_root_from_merkle_proof` iterates over the supplied proof elements without any check on proof length relative to the expected tree depth:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;
    for proof_hash in merkle_proof {
        ...
        current_position /= 2;
    }
    current_hash
}
``` [2](#0-1) 

`verify_transaction_inclusion` (v1) passes the caller-supplied `tx_id` directly into this function with no depth or transaction-count validation:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

The function is still publicly callable (only `#[pause]`-gated, not access-controlled): [4](#0-3) 

### Impact Explanation

Any downstream NEAR contract or bridge that calls `verify_transaction_inclusion` (v1) and acts on a `true` result — e.g., releasing funds, minting tokens, or recording a cross-chain event — can be deceived into processing a fabricated Bitcoin transaction. The attacker does not need to forge a Bitcoin block; they only need to know the internal node hashes of a real block already stored in the contract.

### Likelihood Explanation

The attack requires only knowledge of a real Bitcoin block's transaction hashes (publicly available) to compute internal node values. No privileged access, private keys, or social engineering is needed. The function is callable by any unprivileged NEAR account when the contract is not paused. The contract's own documentation acknowledges the vulnerability:

> "This function is vulnerable to the standard Bitcoin merkle tree second-preimage attack — it may return `true` for an internal node hash rather than a real transaction hash." [5](#0-4) 

### Recommendation

Remove or hard-disable `verify_transaction_inclusion` (v1) so callers are forced to use `verify_transaction_inclusion_v2`, which mitigates the attack by requiring a same-depth coinbase proof. If v1 must remain for backward compatibility, add an access-control gate (e.g., require a trusted-relayer role) so unprivileged callers cannot invoke it.

### Proof of Concept

Consider a Bitcoin block with 4 transactions `[tx0, tx1, tx2, tx3]`:

```
n01  = double_sha256(tx0 || tx1)
n23  = double_sha256(tx2 || tx3)
root = double_sha256(n01 || n23)   ← stored in block header
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id      = n01`   (an internal node, not a real transaction)
- `tx_index   = 0`
- `merkle_proof = [n23]`  (one element, one level above the real leaf depth)

`compute_root_from_merkle_proof(n01, 0, [n23])` computes:
```
compute_hash(n01, n23) = double_sha256(n01 || n23) = root  ✓
```

The function returns `true`. The contract attests that `n01` — which is not a transaction ID — was included in the block. Any bridge or application consuming this result will act on a forged inclusion proof. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

    /// Verifies that a transaction is included in a block at a given block height
    ///
    /// # Deprecated
    /// Use [`verify_transaction_inclusion_v2`] instead, which includes coinbase merkle proof validation
    /// to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param `tx_id` transaction identifier
    /// @param `tx_block_blockhash` block hash at which transacton is supposedly included
    /// @param `tx_index` index of transaction in the block's tx merkle tree
    /// @param `merkle_proof` merkle tree path (concatenated LE sha256 hashes) (does not contain initial `transaction_hash` and `merkle_root`)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if `tx_id` is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
