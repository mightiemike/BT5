### Title
`slowModeFees` ERC20 Tokens Permanently Locked in Endpoint Contract with No Recovery Path - (File: `core/contracts/EndpointTx.sol`, `core/contracts/EndpointStorage.sol`)

---

### Summary

Every slow mode transaction submission transfers `SLOW_MODE_FEE` ($1 in quote tokens) from the user into the `Endpoint` contract itself. These tokens are tracked in the `slowModeFees` accounting variable but there is no function anywhere in the protocol that transfers them back out. The `DumpFees` mechanism only handles `sequencerFee` (off-chain accounting entries in the spot engine), not the actual ERC20 tokens sitting in the `Endpoint` contract. The accumulated `slowModeFees` are permanently locked.

---

### Finding Description

In `EndpointTx.submitSlowModeTransactionImpl()`, for all non-exempt transaction types, the following two lines execute:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;
```

`chargeSlowModeFee` is defined in `EndpointStorage`:

```solidity
function chargeSlowModeFee(IERC20Base token, address from) internal virtual {
    require(address(token) != address(0));
    token.safeTransferFrom(from, address(this), clearinghouse.getSlowModeFee());
}
```

The destination is `address(this)` — the `Endpoint` contract. The quote tokens are now held by `Endpoint`.

The `DumpFees` slow mode transaction handler in `processSlowModeTransactionImpl` only processes `sequencerFee[spotIds[i]]` — an accounting mapping that tracks fees charged via `chargeFee()` (which deducts from a user's spot engine balance). It does **not** touch `slowModeFees` or transfer any ERC20 tokens out of `Endpoint`:

```solidity
} else if (txType == IEndpoint.TransactionType.DumpFees) {
    IOffchainExchange(offchainExchange).dumpFees();
    uint32[] memory spotIds = spotEngine.getProductIds();
    int128[] memory fees = new int128[](spotIds.length);
    for (uint256 i = 0; i < spotIds.length; i++) {
        fees[i] = sequencerFee[spotIds[i]];
        sequencerFee[spotIds[i]] = 0;
    }
    requireSubaccount(X_ACCOUNT);
    clearinghouse.claimSequencerFees(fees);
```

A grep across all `.sol` files confirms `slowModeFees` is only written in two places (`EndpointStorage.sol` line 55 declaration, `EndpointTx.sol` line 371 increment) and never read for any transfer. There is no `claimSlowModeFees`, `withdrawSlowModeFees`, or any other function that moves these tokens out of `Endpoint`.

---

### Impact Explanation

Every call to `submitSlowModeTransaction` for a `WithdrawCollateral`, `LinkSigner`, `ClaimBuilderFee`, or any other non-exempt transaction type causes `SLOW_MODE_FEE` ($1 in quote tokens) to be permanently locked in the `Endpoint` contract. The protocol cannot recover this revenue. Over the lifetime of the protocol, with many users submitting slow mode transactions, this accumulates into a non-trivial permanently irrecoverable token balance.

**Impact: Medium** — protocol fee revenue is permanently locked; not user collateral, but a concrete asset delta with no recovery path.

---

### Likelihood Explanation

Slow mode transactions are a standard, documented user-facing entrypoint. Any user who submits a `WithdrawCollateral` via slow mode (the primary self-custody exit path), a `LinkSigner`, or a `ClaimBuilderFee` triggers this. This is not a rare scenario — it is the normal operation of the protocol's censorship-resistance mechanism.

**Likelihood: High** — triggered by routine protocol usage.

---

### Recommendation

Add a privileged `claimSlowModeFees(address recipient)` function to `Endpoint` (or expose it via `EndpointTx`) that transfers the accumulated quote tokens equal to `slowModeFees` out of the `Endpoint` contract to the protocol treasury, and resets `slowModeFees` to zero. Alternatively, route `chargeSlowModeFee` to transfer directly to the `Clearinghouse` (consistent with how `handleDepositTransfer` works) so the tokens are never stranded in `Endpoint`.

---

### Proof of Concept

1. User calls `Endpoint.submitSlowModeTransaction(withdrawCollateralTx)`.
2. `EndpointTx.submitSlowModeTransactionImpl` is reached via `delegatecall`.
3. The transaction type is `WithdrawCollateral` — not in the owner-only exempt list.
4. `chargeSlowModeFee(_getQuote(), sender)` executes: quote token `safeTransferFrom(user, address(Endpoint), $1)`.
5. `slowModeFees += SLOW_MODE_FEE` increments the counter.
6. The $1 in quote tokens now sits in the `Endpoint` contract.
7. Owner submits `DumpFees` slow mode transaction. `processSlowModeTransactionImpl` handles it: only `sequencerFee` mapping entries are processed and credited to `X_ACCOUNT` via `claimSequencerFees`. `slowModeFees` is never read; no ERC20 transfer out of `Endpoint` occurs.
8. Repeat for every slow mode submission. The balance in `Endpoint` grows monotonically with no drain path.
