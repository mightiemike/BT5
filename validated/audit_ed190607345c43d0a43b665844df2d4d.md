### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC-E Drain Without Providing USDC — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` is a permissionless migration function callable by any address on Ink chain (chain ID 57073). It is supposed to atomically swap USDC-E held in a user's `DirectDepositV1` (DDA) contract for native USDC. However, the `transferFrom` call that pulls USDC from the caller into the DDA is made without checking its return value. If the call returns `false` instead of reverting, the function continues, withdraws USDC-E from the DDA to `ContractOwner`, and forwards it to `msg.sender` — without the caller ever having provided USDC.

---

### Finding Description

The root cause is in `ContractOwner.replaceUsdcEWithUsdc()`:

```solidity
// ContractOwner.sol line 616
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [1](#0-0) 

The `transferFrom` at line 616 uses the raw `IERC20Base` interface call — **not** the `ERC20Helper.safeTransferFrom` wrapper used everywhere else in the protocol. `ERC20Helper.safeTransferFrom` explicitly checks `success && (data.length == 0 || abi.decode(data, (bool)))` and reverts on failure. [2](#0-1) 

The raw call discards the `bool` return value entirely. If `transferFrom` returns `false` (a valid ERC-20 failure mode for tokens that do not revert), execution continues:

1. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` — pulls all USDC-E from the DDA into `ContractOwner`.
2. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` — forwards that USDC-E to the attacker.

The attacker receives USDC-E without having deposited any USDC. The DDA is drained.

The function has no caller access control — only a chain ID check:

```solidity
require(block.chainid == 57073, ERR_UNAUTHORIZED);
``` [3](#0-2) 

Any address on Ink chain can call it for any `subaccount` that has a deployed DDA with a non-zero USDC-E balance.

The analog to the external report is direct: the price-feeder ignores a `429` error signal from Binance and continues operating as if the request succeeded. Here, the contract ignores a `false` return value from `transferFrom` and continues operating as if the token pull succeeded — draining the DDA in the process.

---

### Impact Explanation

An attacker calls `replaceUsdcEWithUsdc(victimSubaccount)` for any DDA holding USDC-E. If the native USDC token on Ink chain returns `false` on a failed `transferFrom` (e.g., attacker has zero USDC balance or allowance), the function drains the entire USDC-E balance from the victim's DDA and transfers it to the attacker. The victim's DDA is left empty; their subaccount collateral is not credited. This is a direct, concrete asset delta: USDC-E tokens are permanently removed from the DDA without the corresponding USDC deposit.

---

### Likelihood Explanation

The function is permissionless and targets any DDA on Ink chain. The exploitability depends on whether the native USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain returns `false` on failure rather than reverting. Circle's canonical USDC implementation reverts, which would prevent exploitation. However, bridged or chain-specific USDC deployments sometimes differ. The inconsistency with the rest of the codebase — which uniformly uses `ERC20Helper.safeTransferFrom` — indicates this is an unintentional omission rather than a deliberate design choice, making the bug real regardless of current token behavior. Likelihood is **medium**: the code path is reachable by any unprivileged caller, and the only barrier is the specific revert-vs-return behavior of the deployed USDC token.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own safe wrapper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

This is consistent with every other token pull in the protocol (`EndpointStorage.safeTransferFrom`, `ERC20Helper.safeTransferFrom`) and ensures the transaction reverts if the pull fails, preventing the subsequent USDC-E withdrawal from proceeding. [4](#0-3) 

---

### Proof of Concept

1. Identify a DDA (`directDepositV1Address[victimSubaccount]`) holding `N` USDC-E tokens.
2. Call `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` from an address with zero USDC balance and zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, dda, N)` returns `false` (insufficient balance/allowance); return value is not checked.
4. `DirectDepositV1(dda).withdraw(usdcE)` executes: `N` USDC-E moves from DDA → `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes: `N` USDC-E moves from `ContractOwner` → attacker.
6. Attacker holds `N` USDC-E; victim's DDA is empty; no USDC was ever deposited. [5](#0-4) [6](#0-5)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
