### Title
Linked Signer Revocation Race Condition Allows Authorized Signer to Drain Subaccount Before Revocation Takes Effect — (`EndpointTx.sol`)

---

### Summary

The `linkedSigners` mapping in Nado grants a designated address the ability to sign transactions on behalf of a subaccount. When a subaccount owner submits a `LinkSigner` revocation via the slow mode path, the revocation is queued with a mandatory 3-day delay. During this window, the old linked signer remains fully authorized and can submit fast-mode `WithdrawCollateral` or `TransferQuote` transactions to the sequencer, draining the subaccount before the revocation ever takes effect. This is a direct structural analog to the PSP22 `approve` double-spend: an absolute authorization value is overwritten without any mechanism to atomically invalidate the prior grantee's in-flight actions.

---

### Finding Description

The `linkedSigners` mapping [1](#0-0)  acts as the protocol's authorization grant — it is the exact analog of an ERC20 allowance. A subaccount owner sets it via `LinkSigner`, and the designated address can then sign privileged transactions on the subaccount's behalf.

When a subaccount owner wants to revoke a linked signer, the only on-chain self-service path is `submitSlowModeTransaction` with a `LinkSigner` payload. This enqueues the revocation with a hardcoded 3-day delay:

```solidity
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    ...
});
``` [2](#0-1) 

The slow mode `LinkSigner` handler then writes the new signer only when the transaction is eventually executed:

```solidity
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

During the entire 3-day window, `linkedSigners[subaccount]` still resolves to the old signer. Both `WithdrawCollateral` (fast mode) and `TransferQuote` pass `allowLinkedSigner = true` to signature validation:

```solidity
validateSignedTx(
    signedTx.tx.sender, signedTx.tx.nonce,
    transaction, signedTx.signature,
    true   // allowLinkedSigner
);
``` [4](#0-3) [5](#0-4) 

`validateSignature` / `validateCompactSignature` resolve the linked signer at call time:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest, signature
);
``` [6](#0-5) 

Because `getLinkedSigner` reads the live mapping value, any fast-mode transaction signed by the old linked signer and submitted to the sequencer during the 3-day window will pass validation and be executed.

---

### Impact Explanation

A malicious linked signer who observes a pending slow-mode revocation on-chain can:

1. Sign a `TransferQuote` transaction moving the subaccount's entire quote balance to an attacker-controlled subaccount.
2. Sign a `WithdrawCollateral` transaction for any collateral product.

Both actions are processed immediately by the sequencer while `linkedSigners[victim_subaccount]` still points to the attacker. The subaccount owner's revocation executes 3 days later against an already-drained account.

**Corrupted state delta:** `spotEngine` balance for the victim subaccount is reduced to zero (or near zero) before the intended revocation takes effect. The attacker's subaccount balance increases by the same amount, which they can then withdraw normally.

---

### Likelihood Explanation

- The slow-mode revocation is publicly visible on-chain the moment it is submitted (it is stored in `slowModeTxs` with a public index).
- The 3-day window is extremely large — sufficient for any monitoring linked signer to act.
- No sequencer compromise is required. The sequencer processes the linked signer's fast-mode transactions in the normal course of operation because they are cryptographically valid at the time of processing.
- The only prerequisite is that the victim previously linked a signer who later becomes adversarial — a realistic scenario (e.g., a compromised API key, a third-party trading bot, a delegated operator).

**Likelihood: Medium** — requires a prior `LinkSigner` grant and a malicious/compromised linked signer, but the exploitation window is 3 days and requires no special privilege beyond what was already granted.

---

### Recommendation

1. **Immediate freeze on revocation submission:** When a slow-mode `LinkSigner` revocation is submitted, record a `pendingRevocation[subaccount]` timestamp. In `getLinkedSigner`, return `address(0)` if a pending revocation exists for the subaccount. This atomically blocks the old signer from the moment the revocation is queued.

2. **Alternatively, allow fast-mode self-revocation:** Permit the subaccount owner to submit a `LinkSigner` (with `signer = address(0)`) as a fast-mode transaction signed by the owner's own key (not the linked signer), bypassing the slow-mode queue entirely. This mirrors the `decreaseAllowance` pattern from PSP22.

3. **Scope-limit linked signer permissions:** Consider restricting linked signers from signing `TransferQuote` (which moves funds to arbitrary recipients) and limiting `WithdrawCollateral` to the subaccount owner's address only — analogous to PSP22's `increaseAllowance`/`decreaseAllowance` approach of narrowing the attack surface rather than removing the feature.

---

### Proof of Concept

```
1. Alice (subaccount owner) calls submitSlowModeTransaction(LinkSigner{sender: alice_sub, signer: bob_addr})
   → linkedSigners[alice_sub] = bob_addr
   → Sequencer processes this fast-mode LinkSigner

2. Alice decides to revoke Bob. She calls:
   submitSlowModeTransaction(LinkSigner{sender: alice_sub, signer: address(0)})
   → slowModeTxs[N] = {executableAt: now + 3 days, tx: LinkSigner{signer: 0}}
   → This is publicly visible on-chain immediately

3. Bob observes the pending revocation in slowModeTxs[N].

4. Bob constructs and signs (with bob_addr key):
   TransferQuote{sender: alice_sub, recipient: bob_sub, amount: alice_full_balance, nonce: current_nonce}

5. Bob submits this to the sequencer as a fast-mode transaction.

6. Sequencer calls processTransactionImpl → validateSignedTx(alice_sub, ..., allowLinkedSigner=true)
   → getLinkedSigner(alice_sub) returns bob_addr  ← still valid, revocation not yet executed
   → Signature check passes
   → clearinghouse.transferQuote(alice_sub → bob_sub, full_balance) executes

7. Bob withdraws from bob_sub normally.

8. Three days later: executeSlowModeTransaction() processes Alice's LinkSigner revocation.
   → linkedSigners[alice_sub] = address(0)
   → Alice's account is already empty.
```

**Root cause line:** `EndpointTx.sol:239` — `linkedSigners` is only updated when the slow-mode transaction executes, not when it is submitted, leaving a 3-day window during which the old signer retains full authorization. [7](#0-6) [8](#0-7) [9](#0-8)
