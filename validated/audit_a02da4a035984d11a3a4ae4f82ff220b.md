Audit Report

## Title
SwapAllowlistExtension Checks Router Address as Swapper Identity, Allowing Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool, which is the pool's own `msg.sender` — the router — not the originating user. When any user routes through `MetricOmmSimpleRouter`, the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This makes per-user allowlist enforcement impossible when the router is involved: either the router is blocked entirely (breaking all router-mediated swaps), or it is allowlisted (granting every user unrestricted access to the restricted pool).

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at line 230–240, passing its own `msg.sender` as the `sender` argument: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the first argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls the pool directly with no mechanism to forward the original caller's identity to the pool's `swap` function. The original `msg.sender` is stored only in transient storage for payment callback purposes via `_setNextCallbackContext`, and is never passed to the pool as the swap initiator: [4](#0-3) 

For multihop `exactInput`, intermediate hops explicitly set the payer to `address(this)` (the router), compounding the issue: [5](#0-4) 

The same pattern applies to `exactOutputSingle` and `exactOutput`. In every router-mediated path, the extension sees `sender = router`, never the real user.

## Impact Explanation

Any pool configured with `SwapAllowlistExtension` for per-user access control (KYC-gated, institution-only, regulatory-restricted) is fully bypassable by any unprivileged EOA calling `MetricOmmSimpleRouter`. The attacker receives output tokens from a restricted pool without being individually allowlisted. This is a direct policy bypass with fund-flow consequences: unauthorized users trade on pools they should be excluded from. The dilemma is irreconcilable — no configuration simultaneously supports the router and enforces per-user policy.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary documented public swap entrypoint. Any pool admin who deploys `SwapAllowlistExtension` expecting users to use the router (the normal case) will either inadvertently open the allowlist to all users or break the router entirely. The bypass requires no special privilege, no flash loan, and no contract deployment — any EOA can call the router directly.

## Recommendation

The pool must forward the original caller's identity to extensions, not its own `msg.sender`. The cleanest fix is for `MetricOmmPool.swap` to accept an explicit `sender` address parameter (similar to Uniswap v4's unlock/callback chain), and for `MetricOmmSimpleRouter` to pass `msg.sender` in that field. Alternatively, the router can write the original caller into a trusted transient slot before calling the pool, and the pool reads that slot to derive the authoritative `sender` for extension dispatch. The router already stores `msg.sender` in transient storage via `_setNextCallbackContext`; extending this pattern to the pool's extension dispatch path would resolve the issue without breaking the existing callback payment flow.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps (required for the router to function at all on this pool).
3. Non-allowlisted attacker calls `router.exactInputSingle({pool: restrictedPool, ...})`.
4. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension receives `sender = router`.
5. Extension evaluates `allowedSwapper[pool][router] == true` → passes without checking the attacker's address.
6. Attacker receives output tokens from the restricted pool.

The same bypass applies to `exactInput` (all hops), `exactOutputSingle`, and `exactOutput`.
