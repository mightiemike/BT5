Audit Report

## Title
Missing upper bound on `timelock` in `OracleValueStopLossExtension` allows pool admin to permanently freeze all guard reconfiguration — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`initialize()` validates `drawdownE6` and `decayPerSecondE8` against upper bounds but applies no upper bound to `timelock`. `proposeOracleStopLossTimelock()` similarly accepts any `uint32` without validation. A pool admin can set `timelock = type(uint32).max` (≈136 years) either at pool creation or immediately post-creation when the current timelock is zero, permanently freezing all guard parameters and either bricking all swaps or permanently disabling stop-loss protection for LPs.

## Finding Description

`initialize()` decodes three parameters and validates only two:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // capped at E6
_validateDecay(decayPerSecondE8); // capped at E8
// timelock — no validation, stored directly
``` [1](#0-0) 

`proposeOracleStopLossTimelock()` also accepts any `uint32` with no validation:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    ...
}
``` [2](#0-1) 

Every subsequent admin action — `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, `proposeOracleStopLossHighWatermarks`, and `proposeOracleStopLossTimelock` itself — computes `executeAfter` via `_afterTimelock`, which adds the **current stored timelock** to `block.timestamp`: [3](#0-2) 

The timelock is therefore self-referential: reducing it requires waiting the current timelock first. `cancelOracleStopLossTimelock` only cancels a *pending* proposal; it cannot undo an already-committed timelock value: [4](#0-3) 

**Post-initialization exploit path (no malicious initial setup required):**

1. Pool is created with `timelock = 0` (immediate execution, a common default).
2. Pool admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`.
3. `_afterTimelock` returns `block.timestamp + 0 = block.timestamp`; `_requireElapsed` passes immediately.
4. Admin calls `executeOracleStopLossTimelock` in the same block.
5. `oracleStopLossConfig[pool_].timelock` is now `type(uint32).max` ≈ 136 years.
6. Every future `propose*` call sets `executeAfter = block.timestamp + 136 years`; no change can ever be executed.

The existing validators for `drawdownE6` and `decayPerSecondE8` confirm the protocol's intent to bound all three parameters, making the missing timelock bound a clear omission: [5](#0-4) 

## Impact Explanation

Two fund-impacting outcomes depending on the frozen `drawdownE6` value:

**Scenario A — Swaps permanently bricked (`drawdownE6 = 1`):** `floorMultiplier = E6 - 1 = 999_999`. Any swap causing even a 1-unit drop in per-share metric triggers `OracleStopLossTriggered`, reverting the swap. With `timelock = type(uint32).max`, `drawdownE6` can never be raised. All swaps revert permanently; the pool is unusable for trading. LPs cannot exit via swaps and are stuck with illiquid positions. This constitutes broken core pool functionality causing loss of funds and unusable swap flows. [6](#0-5) 

**Scenario B — Stop-loss permanently disabled (`drawdownE6 = E6`):** `floorMultiplier = 0`, so `metric < (hwm * 0) / E6` is never true for `uint256`; the guard never fires. With `timelock = type(uint32).max`, `drawdownE6` can never be lowered. LPs have no protection against value extraction through adversarial swaps draining bin reserves below fair value. [7](#0-6) 

This is an admin-boundary break: the pool admin exceeds the intended cap on `timelock` (which the protocol clearly intends to bound, given the analogous caps on the other two parameters), causing permanent, irreversible fund-impacting harm to LPs.

## Likelihood Explanation

The pool admin is semi-trusted. The trigger requires only two sequential admin calls when the current timelock is zero — a common initial configuration. Miscalculation (e.g., passing seconds instead of days, or passing `type(uint32).max` as a sentinel value) is a realistic error. The `proposeOracleStopLossTimelock` function accepts any `uint32` with no validation, making the misconfiguration easy to commit and impossible to recover from without a contract upgrade.

## Recommendation

Add an upper bound on `timelock` in both `initialize()` and `proposeOracleStopLossTimelock()`, analogous to the existing bounds on `drawdownE6` and `decayPerSecondE8`:

```solidity
uint32 internal constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock(timelock)` in `initialize()` alongside `_validateDrawdown` and `_validateDecay`, and apply it in `proposeOracleStopLossTimelock()` before storing `newTimelock`.

## Proof of Concept

```solidity
// Pool created with timelock = 0
extension.initialize(pool, abi.encode(uint32(1), uint32(0), uint32(0)));
// drawdownE6 = 1 (any metric drop triggers stop-loss), timelock = 0

// Admin immediately locks the timelock to max
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
// executeAfter = block.timestamp + 0 = block.timestamp → passes _requireElapsed

vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock = type(uint32).max ≈ 136 years

// Now try to fix drawdown — proposal is set 136 years in the future
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 500_000);
// executeAfter = block.timestamp + type(uint32).max → never executable

// All swaps now revert because drawdownE6 = 1 triggers on any metric drop
// Pool is permanently bricked for trading
```
