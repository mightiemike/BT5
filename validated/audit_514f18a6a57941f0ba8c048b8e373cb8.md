### Title
`verify_transaction_inclusion` Rejects Valid SPV Proofs for Single-Transaction Blocks via Unnecessary Non-Empty Proof Guard - (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains an overly strict `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that unconditionally rejects any SPV proof with an empty proof vector. This is not required by the Bitcoin SPV protocol: when a block contains exactly one transaction, the merkle root equals the transaction hash and an empty proof is the only mathematically correct proof. The underlying `compute_root_from_merkle_proof` already handles this case correctly, making the guard a spurious rejection of valid inputs — a direct structural analog to the ERC-165 gate in `HolographERC721.safeTransferFrom`.

---

### Finding Description

In `verify_transaction_inclusion`, after confirming the block is on the main chain and has sufficient confirmations, the contract executes:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [1](#0-0) 

The `compute_root_from_merkle_proof` function iterates over the proof vector and returns `transaction_hash` unchanged when the vector is empty:

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {   // no-op when empty
        ...
    }
    current_hash   // returns transaction_hash as-is
}
``` [2](#0-1) 

For a block containing exactly one transaction (the coinbase), the Bitcoin protocol defines `merkle_root = txid`. An empty proof is the only valid proof: `compute_root_from_merkle_proof(tx_id, 0, &[])` returns `tx_id`, which equals `merkle_root`. The `require!` at line 315 fires before this computation is reached, causing the call to panic with `"Merkle proof is empty"` even though the proof is correct.

The same defect propagates into `verify_transaction_inclusion_v2`, which passes through to `verify_transaction_inclusion` after converting `ProofArgsV2` into `ProofArgs`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

For a single-transaction block, `coinbase_merkle_proof` is also empty, so the length-equality check passes and the coinbase root check passes (empty proof correctly returns `coinbase_tx_id == merkle_root`), but the call then panics inside `verify_transaction_inclusion`. [4](#0-3) 

---

### Impact Explanation

Any unprivileged NEAR caller invoking `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with a cryptographically valid SPV proof for a single-transaction Bitcoin block receives a panic instead of `true`. The contract's cross-chain verification guarantee is broken for this class of blocks. Any downstream protocol (bridge, oracle, escrow) that relies on this contract to confirm transaction inclusion in single-transaction blocks will permanently fail to process those confirmations, even though the on-chain header data is correct and the proof is valid.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are a documented and recurring phenomenon on Bitcoin mainnet. They occur during low-mempool periods and when miners deliberately mine empty blocks. The condition is reachable by any unprivileged caller supplying real Bitcoin block data — no special privileges, leaked keys, or social engineering are required. The entry path is the public `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` methods. [5](#0-4) [6](#0-5) 

---

### Recommendation

Remove the spurious guard at line 315. The `compute_root_from_merkle_proof` function already handles the empty-proof case correctly and will return the transaction hash unchanged, which is then compared against `header.block_header.merkle_root`. No additional guard is needed:

```rust
// Remove this line:
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [7](#0-6) 

---

### Proof of Concept

1. A Bitcoin block at some height `H` contains exactly one transaction (the coinbase), with `txid = T` and `merkle_root = T`.
2. The relayer submits this block header; it is accepted and stored in `headers_pool` and `mainchain_height_to_header`.
3. A NEAR caller invokes `verify_transaction_inclusion` with:
   - `tx_id = T`
   - `tx_block_blockhash = block_hash(H)`
   - `tx_index = 0`
   - `merkle_proof = []` (the only valid proof)
   - `confirmations = 1`
4. The contract reaches line 315: `require!(!args.merkle_proof.is_empty(), ...)` — panics with `"Merkle proof is empty"`.
5. Had the guard been absent, `compute_root_from_merkle_proof(T, 0, &[])` would return `T`, which equals `header.block_header.merkle_root`, and the function would correctly return `true`. [8](#0-7) [9](#0-8)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
