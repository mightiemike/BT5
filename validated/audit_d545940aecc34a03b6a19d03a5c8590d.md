### Title
`Clearinghouse.setWithdrawPool` Replacement Desynchronizes `markedIdxs` State, Enabling Fast Withdrawal Replay Against New Pool — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` replaces the active `withdrawPool` address with no migration of the old pool's `markedIdxs` or `minIdx` state. The new pool starts with all indices unmarked and `minIdx = 0`. Any previously processed fast withdrawal transaction — whose sequencer signatures remain permanently valid on-chain — can be replayed against the new pool by any caller, draining LP liquidity deposited there.

---

### Finding Description

`Clearinghouse.setWithdrawPool` performs a bare address swap:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
``` [1](#0-0) 

The `BaseWithdrawPool` contract guards against double-processing fast withdrawals using two state variables:

- `markedIdxs[idx]` — set to `true` when a fast withdrawal at sequencer index `idx` is processed
- `minIdx` — updated to the most recent sequencer-submitted withdrawal index, blocking stale replays

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;
``` [2](#0-1) 

When `setWithdrawPool` is called, the new pool contract is freshly deployed: `markedIdxs` is empty for all indices and `minIdx = 0`. The old pool's accumulated replay-protection state is entirely abandoned. Every fast withdrawal that was legitimately processed against the old pool now appears unprocessed to the new pool.

The sequencer signatures used in `submitFastWithdrawal` are verified by `Verifier.requireValidTxSignatures` over the `(transaction, idx)` pair:

```solidity
Verifier v = Verifier(verifier);
v.requireValidTxSignatures(transaction, idx, signatures);
``` [3](#0-2) 

These signatures are immutable — they were broadcast on-chain as calldata and remain permanently valid. No replay protection exists at the `Verifier` level for the pool-replacement scenario.

The `submitWithdrawal` path (called by the clearinghouse for sequencer-driven withdrawals) also sets `minIdx`:

```solidity
minIdx = idx;
``` [4](#0-3) 

This value is pool-local. After replacement, the new pool's `minIdx` starts at `0`, so the `idx > minIdx` guard passes for every historical fast withdrawal index.

---

### Impact Explanation

**Impact: Medium**

After a pool upgrade, any on-chain observer can extract historical `submitFastWithdrawal` calldata (transaction bytes + sequencer signatures) and replay them against the new pool. Each replay passes both guards (`markedIdxs` and `minIdx`), re-executes the token transfer, and marks the index in the new pool. LP liquidity deposited into the new pool to service future fast withdrawals is drained proportionally to the volume of historical fast withdrawals. The corrupted state delta is: `token.balanceOf(newPool)` reduced to zero; `fees[productId]` inflated by replayed fee amounts.

---

### Likelihood Explanation

**Likelihood: Medium**

A pool upgrade is a realistic operational event (contract bug fix, fee model change, or liquidity restructuring). The attack requires no privileged access: the attacker only needs historical calldata, which is permanently available from any block explorer or archive node. The `submitFastWithdrawal` function is `public` with no caller restriction. The window of vulnerability opens the moment the new pool receives any LP deposit.

---

### Recommendation

1. **Seed `minIdx` on deployment**: When `setWithdrawPool` is called, read the old pool's current `minIdx` and initialize the new pool to the same value before activating it. This blocks all historical fast withdrawal indices from being replayed.

2. **Alternatively, add a no-existing-state guard**: Before allowing `setWithdrawPool` to proceed, verify that the old pool holds no token balances and that no fast withdrawals have been processed (i.e., `minIdx == 0`). This mirrors the M-05 recommendation of checking for zero deposited positions before allowing an update.

3. **Carry over `markedIdxs` for the active window**: Transfer the set of marked indices from the old pool to the new pool for all indices in the range `[minIdx, current_nSubmissions]`.

---

### Proof of Concept

1. Protocol is live; the old `WithdrawPool` has processed N fast withdrawals. Each call to `submitFastWithdrawal` emitted calldata with `(idx, transaction, signatures)` — all permanently on-chain.

2. Owner calls `ContractOwner.setWithdrawPool(newPool)` → `Clearinghouse.setWithdrawPool(newPool)`. [5](#0-4) 

3. LP deposits tokens into `newPool` to provide fast withdrawal liquidity.

4. Attacker collects all historical `submitFastWithdrawal` calldata from block history.

5. For each historical `(idx, transaction, signatures)`:
   - `newPool.markedIdxs[idx]` is `false` → passes first guard
   - `newPool.minIdx` is `0`, `idx > 0` → passes second guard
   - `Verifier.requireValidTxSignatures` accepts the original signatures → passes
   - Pool transfers tokens to the original withdrawal recipient again

6. All LP liquidity in `newPool` is drained. The `fees[productId]` mapping is also inflated by replayed fee credits, corrupting the pool's accounting.
