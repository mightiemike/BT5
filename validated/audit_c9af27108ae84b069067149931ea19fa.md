### Title
`Role::UnrestrictedSubmitBlocks` Cannot Bypass Pause Due to Missing `except` Clause on `submit_blocks` - (File: `contract/src/lib.rs`)

---

### Summary

`Role::UnrestrictedSubmitBlocks` is explicitly documented as allowing use of the contract API even when paused, but the `#[pause]` decorator on `submit_blocks` has no `except(roles(...))` clause. This makes the role permanently ineffective: any account holding it is still blocked by the pause check, identical in structure to the ZeroLendToken boolean-logic role-bypass failure.

---

### Finding Description

The `Role` enum in `contract/src/lib.rs` defines `UnrestrictedSubmitBlocks` with the comment:

> "Allows to use contract API even after contract is paused" [1](#0-0) 

The `run_mainchain_gc` function correctly implements this pattern using `near_plugins`'s `except` syntax:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
``` [2](#0-1) 

However, `submit_blocks` uses a bare `#[pause]` with no `except` clause:

```rust
#[payable]
#[pause]
#[trusted_relayer]
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
) -> PromiseOrValue<()> {
``` [3](#0-2) 

The `#[trusted_relayer]` macro at the impl level declares `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`:

```rust
#[trusted_relayer(
    bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks),
    manager_roles(Role::DAO, Role::RelayerManager),
    config_roles(Role::DAO)
)]
``` [4](#0-3) 

But `bypass_roles` in `#[trusted_relayer]` only bypasses the staking/trusted-relayer requirement — it does not affect the `#[pause]` guard, which is a separate macro that fires unconditionally at function entry. Because `#[pause]` is listed before `#[trusted_relayer]` in the attribute stack, the pause check executes first and panics before the trusted-relayer bypass logic is ever reached.

The root cause is structurally identical to the ZeroLendToken bug: a role intended to bypass a restriction is defined and documented, but the guard condition is written in a way that makes the role permanently inert.

---

### Impact Explanation

When the contract is paused (e.g., during an emergency or upgrade window), no new Bitcoin block headers can be submitted — including by accounts explicitly granted `Role::UnrestrictedSubmitBlocks` or `Role::DAO`. The light client's canonical chain state freezes. Any downstream NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to confirm Bitcoin transactions will receive stale or absent confirmation data for the duration of the pause, breaking SPV proof workflows that depend on a live chain tip.

---

### Likelihood Explanation

The contract is designed for production use with a `PauseManager` role and an explicit pause/unpause lifecycle. The `UnrestrictedSubmitBlocks` role was granted to specific accounts precisely so they could maintain chain liveness during pauses. Any operator who pauses the contract and then expects privileged relayers to continue operating will discover the bypass is silently non-functional. This is a realistic operational scenario, not a theoretical one.

---

### Recommendation

Apply the `except` clause to `submit_blocks`, mirroring the pattern already used on `run_mainchain_gc`:

```diff
-#[pause]
+#[pause(except(roles(Role::UnrestrictedSubmitBlocks, Role::DAO)))]
 #[trusted_relayer]
 pub fn submit_blocks(
     &mut self,
     #[serializer(borsh)] headers: Vec<BlockHeader>,
 ) -> PromiseOrValue<()> {
``` [5](#0-4) 

This aligns `submit_blocks` with the documented intent of `Role::UnrestrictedSubmitBlocks` and with the existing correct implementation on `run_mainchain_gc`.

---

### Proof of Concept

1. Deploy the contract with `skip_pow_verification = true`.
2. Grant `Role::PauseManager` to account A and `Role::UnrestrictedSubmitBlocks` to account B.
3. Account A calls `pa_pause` to pause the contract.
4. Account B (holding `UnrestrictedSubmitBlocks`) calls `submit_blocks` with a valid header batch.
5. **Observed:** The call panics with the `near_plugins` pause error, despite B holding the role documented to bypass the pause.
6. **Expected:** The call succeeds, because `Role::UnrestrictedSubmitBlocks` is explicitly defined to allow API use while paused.
7. Compare with `run_mainchain_gc`: an account holding `Role::UnrestrictedRunGC` can call it while paused, because `#[pause(except(roles(Role::UnrestrictedRunGC)))]` is correctly specified. [2](#0-1)
