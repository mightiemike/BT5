### Title
`validate_proof_size` Never Invoked in `StatelessTransactionValidator::validate` — Proof Size Limit Unenforced at Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
`StatelessTransactionValidator::validate` calls every invoke-specific check except `validate_proof_size`. The `max_proof_size` config and the `ProofTooLarge` error variant exist solely to enforce a proof byte-length ceiling, but the check is dead code. Any caller with `allow_client_side_proving = true` (the production default) can submit an `RpcInvokeTransactionV3` whose `proof` field exceeds the configured limit; the gateway accepts it, stores the oversized proof in the proof manager, and forwards the transaction to the mempool.

### Finding Description

`StatelessTransactionValidator::validate` dispatches invoke-specific checks as follows:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    // validate_proof_size is NEVER called
}
``` [1](#0-0) 

The function that enforces the limit is fully implemented and wired to a config field and a dedicated error variant:

```rust
fn validate_proof_size(
    &self,
    tx: &RpcInvokeTransaction,
) -> StatelessTransactionValidatorResult<()> {
    let RpcInvokeTransaction::V3(tx) = tx;
    let proof_size = tx.proof.0.len();
    if proof_size > self.config.max_proof_size {
        return Err(StatelessTransactionValidatorError::ProofTooLarge {
            proof_size,
            max_proof_size: self.config.max_proof_size,
        });
    }
    Ok(())
}
``` [2](#0-1) 

The config default sets `max_proof_size: 480_000` and the production schema publishes it as a tunable parameter: [3](#0-2) [4](#0-3) 

The `ProofTooLarge` error variant is declared but can never be returned: [5](#0-4) 

After the gateway accepts the transaction, `convert_rpc_tx_to_internal` extracts the proof and schedules it for archiving in the proof manager: [6](#0-5) 

The proof is then dropped from the internal representation (`// Note: proof field is dropped`), but the raw bytes have already been handed to the proof manager: [7](#0-6) 

`validate_proof_facts_and_proof_consistency` guarantees that a non-empty `proof_facts` implies a non-empty `proof`, so any transaction that carries proof facts will also carry the full proof blob and trigger proof-manager storage: [8](#0-7) 

### Impact Explanation

An unprivileged submitter can craft an `RpcInvokeTransactionV3` with `proof_facts` set to any valid non-empty value and `proof.0` set to an arbitrarily large byte vector (bounded only by the coarser overall transaction-size limit). The gateway admits the transaction, stores the oversized proof blob in the proof manager, and enqueues the transaction in the mempool. This bypasses the operator-configured resource ceiling on proof storage, enabling unbounded proof-manager growth and potential denial-of-service against the proof archiving subsystem.

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

`allow_client_side_proving` defaults to `true` in both `StatelessTransactionValidatorConfig::default()` and the production `config_schema.json`. No privileged role is required; any network-reachable submitter can exploit this. The only prerequisite is knowledge that the proof size check is absent, which is visible from the source.

### Recommendation

Add the missing call inside the invoke branch of `validate`:

```rust
if let RpcTransaction::Invoke(invoke_tx) = tx {
    self.validate_client_side_proving_allowed(invoke_tx)?;
    self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    self.validate_proof_size(invoke_tx)?;   // ← add this line
}
``` [9](#0-8) 

### Proof of Concept

1. Start a gateway node with `allow_client_side_proving = true` (the default).
2. Construct a valid `RpcInvokeTransactionV3` with:
   - `proof_facts` set to any non-empty `ProofFacts` value (satisfies the consistency check).
   - `proof.0` set to a `Vec<u8>` of length `480_001` (one byte over `max_proof_size`).
3. Submit the transaction via `POST /add_transaction` or the RPC gateway endpoint.
4. **Observed:** the gateway returns a success response and the transaction is forwarded to the mempool; no `ProofTooLarge` error is raised.
5. **Expected:** the gateway should reject the transaction with `StatelessTransactionValidatorError::ProofTooLarge { proof_size: 480001, max_proof_size: 480000 }`.

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }
