Looking at the actual code to trace the `lastLiquidationFees` storage variable through the liquidation flow.

### Title
Stale `lastLiquidationFees` in `_finalizeSubaccount` Causes Premature Socialization When Insurance Is Sufficient — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a global storage variable that is written by every non-finalize call to `_handleLiquidationPayment` and read, unmodified, by the subsequent finalize call to `_finalizeSubaccount`. When the most-recent non-finalize liquidation generated large fees, the finalize call subtracts those fees from `insurance` before computing `v.canLiquidateMore`, making available insurance appear smaller than it actually is. If the subtraction pushes the effective insurance below the remaining deficit, `v.canLiquidateMore` is set to `false` and `spotEngine.socializeSubaccount` is called even though the real insurance fund was sufficient to cover the deficit in full — creating unnecessary bad debt for all other participants.

---

### Finding Description

**Storage layout**

`lastLiquidationFees` is declared as a persistent `int128` in `ClearinghouseStorage`: [1](#0-0) 

**Write path — `_handleLiquidationPayment`**

Every non-finalize liquidation (i.e., `txn.productId != type(uint32).max`) ends with:

```solidity
insurance += v.liquidationFees;   // line 579
lastLiquidationFees = v.liquidationFees;  // line 586
``` [2](#0-1) 

The comment at lines 581–585 explains the intent: fees just added to `insurance` should not be counted toward socialization, to prevent the refill from blocking the socialization step.

**Read path — `_finalizeSubaccount`**

When `txn.productId == type(uint32).max`, `_handleLiquidationPayment` is **never called** (the function returns early at line 626). Therefore `lastLiquidationFees` is always the value written by the *previous* non-finalize call — potentially from a completely different subaccount's liquidation. The finalize path then does:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;          // line 369
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;  // line 370
``` [3](#0-2) 

If `lastLiquidationFees` is large, `v.insurance` can be far below the true insurance balance, making `v.canLiquidateMore` false when it should be true.

**Socialization branch**

When `v.canLiquidateMore` is false, the code proceeds to:

```solidity
v.insurance = perpEngine.socializeSubaccount(txn.liquidatee, v.insurance);
int128 insuranceCover = MathHelper.min(v.insurance, -quoteBalance.amount);
if (insuranceCover > 0) { v.insurance -= insuranceCover; ... }
if (v.insurance <= 0) {
    spotEngine.socializeSubaccount(txn.liquidatee);   // line 408
}
v.insurance += lastLiquidationFees;   // line 410 — fees restored
insurance = v.insurance;              // line 411
``` [4](#0-3) 

The fees are restored to `insurance` at line 410, but `spotEngine.socializeSubaccount` has already been called at line 408, distributing the remaining deficit as bad debt to other participants. The portion of the deficit that the real insurance fund could have covered (the `lastLiquidationFees` slice) is instead mutualized.

**Cross-subaccount contamination**

`lastLiquidationFees` is not scoped to a specific liquidatee. A liquidation of subaccount A sets the global variable; the very next finalize call for subaccount B reads it. A liquidator can therefore:

1. Liquidate a large position on any subaccount to maximize `lastLiquidationFees`.
2. Immediately submit a finalize call for a different (or the same) subaccount whose remaining deficit falls in the window `(insurance - lastLiquidationFees, insurance]`.

---

### Impact Explanation

- **Bad debt creation**: losses that the insurance fund could have absorbed are instead socialized across all participants, violating the protocol invariant that socialization only occurs when insurance is truly exhausted.
- **Insurance fund misaccounting**: after the call, `insurance` retains `lastLiquidationFees` (line 410–411), but the socialized amount has already been distributed as bad debt — the protocol ends up with more insurance than it should while other participants bear losses they should not.
- **Scope match**: asset theft / bad debt creation for other participants — Critical per the stated scope.

---

### Likelihood Explanation

The attack requires only two sequentially submitted liquidation transactions (both reachable through the normal endpoint-gated liquidation path). No privileged access, no oracle manipulation, and no exotic token behavior is needed. The window where `quoteBalance.amount + (insurance - lastLiquidationFees) ≤ 0 < quoteBalance.amount + insurance` is reachable whenever a large position is liquidated just before a finalize call, which is the normal end-of-liquidation sequence.

---

### Recommendation

Scope `lastLiquidationFees` to the current finalize call rather than persisting it across calls. The simplest fix is to compute the fee reserve inside `_finalizeSubaccount` itself — either by passing the current call's fees as a parameter, or by zeroing `lastLiquidationFees` at the start of `_finalizeSubaccount` and only applying the reserve when the finalize call is immediately preceded by a non-finalize call for the **same** liquidatee. Alternatively, replace the global variable with a per-subaccount mapping so cross-subaccount contamination is impossible.

---

### Proof of Concept

```
Setup:
  insurance = 100
  lastLiquidationFees = 80  (set by a prior large liquidation of any subaccount)
  liquidatee quote balance after PnL settlement = -50
  (all perp/spot positions already closed — finalize preconditions met)

Step 1 — submit finalize tx (productId = type(uint32).max):
  v.insurance = 100 - 80 = 20
  v.canLiquidateMore = (-50 + 20) > 0  →  false   ← WRONG (100 > 50)

  perpEngine.socializeSubaccount(liquidatee, 20) → returns 20 (no perp positions)
  insuranceCover = min(20, 50) = 20
  v.insurance = 0
  quoteBalance updated to -30
  v.insurance <= 0  →  spotEngine.socializeSubaccount(liquidatee) called
    → 30 units of bad debt distributed to other participants

  v.insurance += 80  →  insurance stored as 80

Correct outcome (if lastLiquidationFees were 0):
  v.insurance = 100
  v.canLiquidateMore = (-50 + 100) > 0  →  true
  insuranceCover = min(100, 50) = 50
  quoteBalance = 0
  v.insurance = 50  →  no socialization
  insurance stored as 50

Delta: 30 units incorrectly socialized; insurance overstated by 30.
```
