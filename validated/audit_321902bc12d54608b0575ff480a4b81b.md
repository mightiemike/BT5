### Title
Python Object-Identity (`id()`) Aliasing in `clvm_tree_to_lazy_node` Produces a Silently Wrong `NodePtr` Mapping — (`File: wheel/src/api.rs`)

---

### Summary

`clvm_tree_to_lazy_node` uses the raw CPython heap address of each visited Python object (`pyobj.as_ptr() as usize`) as the sole key in `identity_map`. CPython's reference-counting GC reclaims an object the instant its last Python reference is dropped. Because the function drops the `Bound<'py, PyAny>` wrapper for a pair node at the end of its match arm — before the corresponding `BuildPair` work item is processed — a caller-controlled Python tree whose `.pair` property does not cache its children can cause a node to be freed and its address reused by a later, structurally different node. When that later node is looked up in `identity_map`, the stale entry is returned, silently substituting the wrong `NodePtr` into the output `LazyNode`.

---

### Finding Description

**Root cause — `wheel/src/api.rs`, lines 196 and 215:**

```rust
let mut identity_map: HashMap<usize, NodePtr> = HashMap::new();
// ...
let id = pyobj.as_ptr() as usize;   // raw CPython heap address
if identity_map.contains_key(&id) {
    continue;                        // ← stale hit if address was reused
}
``` [1](#0-0) 

When a pair node is visited, the code:
1. Extracts `left` and `right` via `pyobj.getattr("pair")`.
2. Pushes `WorkItem::BuildPair { id, left_id, right_id }` and `WorkItem::Visit(left/right)` onto the stack.
3. Falls off the end of the `WorkItem::Visit` match arm — **dropping `pyobj`'s `Bound<'py, PyAny>` wrapper**, decrementing the Python refcount. [2](#0-1) 

If the caller's Python tree does **not** cache children (i.e., `.pair` creates fresh objects on every access), `pyobj` is the only live Python reference to that node. Dropping the `Bound` wrapper brings the refcount to zero; CPython immediately frees the object. The same sequence applies to `left` once `WorkItem::Visit(left)` is popped and processed: `left` is consumed, its `Bound` is dropped, and if nothing else holds it, its heap slot is freed.

CPython's small-object allocator (`obmalloc`) aggressively reuses freed slots. The very next Python allocation triggered by `right.getattr("pair")` — or by any property getter called during traversal — can land at the same address. When that new object is later encountered as a child of another node, `identity_map.contains_key(&new_id)` returns `true` (because `new_id == old_freed_id`), and the function silently returns the **old, wrong `NodePtr`** without visiting the new object at all.

**Concrete trigger sequence:**

```
Visit(root)
  → extracts left=A, right=B; pushes BuildPair(root,A,B), Visit(B), Visit(A)
  → root dropped (refcount → 0 if ephemeral) → freed

Visit(A)  [atom "foo"]
  → identity_map[A_addr] = atom_node_for_foo
  → A dropped (refcount → 0 if ephemeral) → freed; A_addr now available

Visit(B)  [pair]
  → B.pair() allocates child E at A_addr  ← CPython reuses freed slot
  → identity_map.contains_key(&E_addr) == true  ← stale hit!
  → E is silently mapped to atom_node_for_foo instead of its real NodePtr
```

---

### Impact Explanation

The `LazyNode` returned by `clvm_tree_to_lazy_node` contains a subtree that is structurally different from the input Python tree. Any downstream operation on that `LazyNode` is corrupted:

- **`ser_2026` / `ser_backrefs` / `ser_legacy`**: serializes the wrong bytes.
- **`sha256_treehash`**: produces the wrong puzzle hash.
- **`run_serialized_chia_program`**: if the wrong bytes are re-deserialized and executed, the program semantics differ from what the caller constructed.

`Program.to_bytes_2026()` calls `clvm_tree_to_lazy_node(self)` directly: [3](#0-2) 

A wrong serialization produced here and later submitted to the Chia mempool would be validated against a different puzzle hash than the one the wallet computed, breaking coin-spend authorization silently rather than with an explicit error.

---

### Likelihood Explanation

**Moderate-to-low** in the wild, but **high** for a targeted attacker:

- The standard `Program` class caches children in `self._pair`, so its objects are not freed mid-traversal under normal use. [4](#0-3) 

- However, `clvm_tree_to_lazy_node` is documented and exported to accept **any** Python object satisfying the `CLVMStorage` protocol (`.atom` / `.pair`). A caller using a lazy or streaming CLVM tree representation — common in wallet tooling that avoids materializing large trees — will naturally produce ephemeral child objects. [5](#0-4) 

- The function is re-exported from `clvm_rs.serde` as a first-class public API: [6](#0-5) 

- CPython's `obmalloc` reuses freed slots deterministically within the same size class, making address reuse highly predictable once the attacker controls the allocation pattern.

---

### Recommendation

Hold every visited Python object alive for the duration of the traversal by storing the `Bound<'py, PyAny>` handles in a side-list alongside the `identity_map`:

```rust
let mut live_objects: Vec<Bound<'_, PyAny>> = Vec::new();
// inside WorkItem::Visit:
live_objects.push(pyobj.clone());   // keep refcount > 0 until function returns
```

Alternatively, replace the raw-pointer key with a monotonically increasing integer assigned on first visit, stored as a Python-side attribute or in a parallel `Vec`. This severs the tie between object lifetime and map key validity entirely.

### Proof of Concept

```python
import gc
from clvm_rs import clvm_tree_to_lazy_node

class EphemeralAtom:
    def __init__(self, data: bytes):
        self._data = data
    @property
    def atom(self): return self._data
    @property
    def pair(self): return None

class EphemeralPair:
    """Children are created fresh on every .pair access — no caching."""
    def __init__(self, left_factory, right_factory):
        self._lf = left_factory
        self._rf = right_factory
    @property
    def atom(self): return None
    @property
    def pair(self):
        return (self._lf(), self._rf())

# Build a tree: root = pair(pair(atom_A, atom_B), pair(atom_C, atom_D))
# where every .pair call allocates fresh Python objects.
atom_A = lambda: EphemeralAtom(b"AAAA")
atom_B = lambda: EphemeralAtom(b"BBBB")
atom_C = lambda: EphemeralAtom(b"CCCC")
atom_D = lambda: EphemeralAtom(b"DDDD")

left_subtree  = lambda: EphemeralPair(atom_A, atom_B)
right_subtree = lambda: EphemeralPair(atom_C, atom_D)
root = EphemeralPair(left_subtree, right_subtree)

# Force GC to maximise address reuse
gc.collect()

lazy = clvm_tree_to_lazy_node(root)

# Serialize and re-deserialize to observe the corrupted structure
from clvm_rs import ser_legacy, deser_legacy
blob = ser_legacy(lazy)
# If aliasing occurred, blob encodes a tree where one subtree is
# silently replaced by a structurally unrelated atom or pair.
print(blob.hex())   # compare against a reference serialization
```

The exact collision depends on CPython's allocator state, but is reliably reproducible by controlling allocation order (e.g., pre-allocating and freeing same-sized objects before the call).
