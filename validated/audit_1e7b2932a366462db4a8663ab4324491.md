### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Originating User, Allowing Any Caller to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the originating user. If the pool admin allowlists the router address (the natural step to enable router-mediated swaps), every user of the public router bypasses the allowlist, executing swaps the pool was designed to restrict.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

`msg.sender` here is the pool (correct), but `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) is the caller, it calls `pool.swap()` directly: [4](#0-3) 

So `sender` arriving at the extension is the **router address**, not the originating EOA. The allowlist lookup becomes `allowedSwapper[pool][router]`.

A pool admin who wants legitimate users to be able to use the router must call `setAllowedToSwap(pool, router, true)`. Once the router is allowlisted, **every caller of the public router** passes the check — the allowlist is fully neutralised for router-mediated swaps. Conversely, if the admin allowlists individual EOAs instead, those EOAs are silently blocked from using the router even though they are on the allowlist, breaking the intended swap flow.

---

### Impact Explanation

- **Allowlist bypass (primary):** Any unprivileged user routes through `MetricOmmSimpleRouter` and executes swaps in a pool the admin intended to restrict. Unrestricted swaps drain LP token balances, causing direct loss of LP principal.
- **Broken allowlist for legitimate users (secondary):** If the admin allowlists individual EOAs, those EOAs cannot use the router, making the primary swap interface unusable for the very users the admin intended to serve.

Both consequences are fund-impacting: the first allows unauthorized extraction of LP assets; the second renders the pool's swap flow unusable for allowlisted participants.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface documented in the periphery. Any pool admin who enables the `SwapAllowlistExtension` and also wants users to access the router will allowlist the router address — this is the natural, expected configuration. The bypass is then reachable by any unprivileged caller with zero special access.

---

### Recommendation

Bind the allowlist check to the **originating user** rather than the direct pool caller. Two complementary approaches:

1. **Pass the originating user through the router.** Add a `swapper` field to the router's swap parameters and forward it in `extensionData`. The extension decodes and checks that address instead of `sender`.

2. **Check `sender` only when it is not a known router; otherwise decode the real user from `extensionData`.** This requires the extension to be router-aware.

The simplest safe fix is approach 1: the router encodes `msg.sender` into `extensionData` before calling `pool.swap`, and `SwapAllowlistExtension` decodes and gates that value. This mirrors the fix recommended in the external report (binding `msg.sender` into the salt) — here the fix is binding `msg.sender` of the router into the extension payload so the guard always sees the economically relevant actor.

---

### Proof of Concept

```
Setup
─────
1. Deploy pool with SwapAllowlistExtension configured on beforeSwap.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   (necessary so that any user can reach the pool via the router).
3. Attacker address is NOT individually allowlisted.

Attack
──────
4. Attacker calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
5. Router calls pool.swap(...) — msg.sender at pool = router.
6. Pool calls _beforeSwap(router, ...).
7. SwapAllowlistExtension.beforeSwap receives sender = router.
8. Check: allowedSwapper[pool][router] == true  →  PASSES.
9. Swap executes; attacker receives output tokens from LP reserves.

Expected (correct) behaviour: step 8 should check the originating user,
which is not allowlisted, and revert with NotAllowedToSwap.
```

The root cause is identical in structure to the external report: an identifier that should bind to the originating actor (`msg.sender` in the LeapFactory salt; the originating user in the allowlist check) is instead taken from an intermediate contract, allowing any caller who controls that intermediate to bypass the guard.
