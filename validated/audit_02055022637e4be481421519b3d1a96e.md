### Title
`DeployAccountTransaction V3` Silently Zeroes `account_deployment_data` in `create_tx_info`, Causing Wrong `TxInfo` Exposed to Account Contract During Execution - (File: `crates/blockifier/src/transaction/transactions.rs`)

---

### Summary

When a `DeployAccountTransaction V3` is executed, the `create_tx_info` method in the blockifier hardcodes `account_deployment_data: AccountDeploymentData::default()` instead of reading the field from the transaction. This means the account contract's constructor receives an empty `account_deployment_data` array in its `TxInfo` syscall response, regardless of what the user submitted. The field is part of the signed transaction hash preimage, so the hash is computed over the real value, but the value exposed to the executing contract is always empty.

---

### Finding Description

`DeployAccountTransactionV3` (defined in `crates/starknet_api/src/transaction.rs`) does **not** carry an `account_deployment_data` field at the struct level — the field is absent from the type definition:

```rust
pub struct DeployAccountTransactionV3 {
    pub resource_bounds: ValidResourceBounds,
    pub tip: Tip,
    pub signature: TransactionSignature,
    pub nonce: Nonce,
    pub class_hash: ClassHash,
    pub contract_address_salt: ContractAddressSalt,
    pub constructor_calldata: Calldata,
    pub nonce_data_availability_mode: DataAvailabilityMode,
    pub fee_data_availability_mode: DataAvailabilityMode,
    pub paymaster_data: PaymasterData,
    // account_deployment_data is ABSENT
}
``` [1](#0-0) 

However, `CurrentTransactionInfo` — the struct that is passed to the executing contract via the `get_execution_info` syscall — **does** include `account_deployment_data`:

```rust
pub struct CurrentTransactionInfo {
    pub common_fields: CommonAccountFields,
    pub resource_bounds: ValidResourceBounds,
    pub tip: Tip,
    pub nonce_data_availability_mode: DataAvailabilityMode,
    pub fee_data_availability_mode: DataAvailabilityMode,
    pub paymaster_data: PaymasterData,
    pub account_deployment_data: AccountDeploymentData,  // present
    pub proof_facts: ProofFacts,
}
``` [2](#0-1) 

In `TransactionInfoCreatorInner for DeployAccountTransaction`, the V3 branch explicitly fills this field with a hardcoded default:

```rust
starknet_api::transaction::DeployAccountTransaction::V3(tx) => {
    TransactionInfo::Current(CurrentTransactionInfo {
        ...
        paymaster_data: tx.paymaster_data.clone(),
        account_deployment_data: AccountDeploymentData::default(),  // ← hardcoded empty
        proof_facts: ProofFacts::default(),
    })
}
``` [3](#0-2) 

By contrast, `DeclareTransaction V3` correctly propagates the field from the transaction struct:

```rust
starknet_api::transaction::DeclareTransaction::V3(tx) => {
    TransactionInfo::Current(CurrentTransactionInfo {
        ...
        account_deployment_data: tx.account_deployment_data.clone(),  // ← correct
        proof_facts: ProofFacts::default(),
    })
}
``` [4](#0-3) 

The Starknet OS Cairo program also expects `account_deployment_data` to be present in the `TxInfo` struct passed to the account contract: [5](#0-4) 

---

### Impact Explanation

The `account_deployment_data` field is defined in the Starknet spec as data needed to deploy and initialize an account contract from which a transaction will be initiated. An account contract's constructor or `__validate_deploy__` entrypoint may read this field via the `get_execution_info` syscall to verify deployment parameters or perform initialization logic. Because the blockifier always passes an empty array, any account contract that reads `account_deployment_data` during a `DeployAccount V3` execution will receive wrong data. This can cause:

- **Wrong execution result**: A contract that validates `account_deployment_data` will see an empty array instead of the submitted data, potentially causing incorrect validation logic to pass or fail.
- **Wrong state**: If the constructor uses `account_deployment_data` to initialize storage, the storage will be set to wrong values, permanently corrupting the deployed account's state.
- **Divergence between OS and blockifier**: The Starknet OS Cairo program populates `account_deployment_data` from the actual transaction data. If the OS and blockifier disagree on the value, re-execution for proving will produce a different result, breaking the proof.

This matches the **Critical** impact category: wrong state or revert result from blockifier/syscall/execution logic for accepted input.

---

### Likelihood Explanation

Any `DeployAccount V3` transaction submitted with a non-empty `account_deployment_data` field will trigger this bug. The field is part of the public Starknet V3 transaction schema and is accepted by the gateway. An attacker or user can submit such a transaction through the standard RPC `starknet_addDeployAccountTransaction` endpoint. The `RpcDeployAccountTransactionV3` struct does not carry `account_deployment_data` either, so the field is structurally absent from the entire deploy-account path — meaning no existing deploy-account transaction can carry a non-empty value through the current code. However, the OS Cairo program and the `CurrentTransactionInfo` struct both define the field, creating a latent divergence that would manifest if the struct were extended or if the OS enforces the field's presence during proof verification. [6](#0-5) 

---

### Recommendation

1. Add `account_deployment_data` to `DeployAccountTransactionV3` in `crates/starknet_api/src/transaction.rs` (analogous to how `DeclareTransactionV3` carries it).
2. Propagate the field through `RpcDeployAccountTransactionV3` and all conversion paths (`From` impls, protobuf converters, client reader objects).
3. In `TransactionInfoCreatorInner for DeployAccountTransaction`, replace the hardcoded `AccountDeploymentData::default()` with `tx.account_deployment_data.clone()`.
4. Ensure the `DeployAccountTransactionV3Trait` used by `get_deploy_account_transaction_v3_hash` does not need updating (the hash function does not currently include `account_deployment_data` for deploy-account, consistent with the Starknet spec for this transaction type).

---

### Proof of Concept

1. Submit a `DeployAccount V3` transaction where the account contract's `__validate_deploy__` reads `get_execution_info().account_deployment_data` and asserts it equals a known non-empty value.
2. The blockifier will pass `account_deployment_data = []` to the contract.
3. The assertion fails (or the wrong branch executes), producing a different execution result than what the Starknet OS would produce when re-executing the same transaction for proof generation.
4. Concretely, in `create_tx_info` for `DeployAccountTransaction::V3`, the line `account_deployment_data: AccountDeploymentData::default()` at `crates/blockifier/src/transaction/transactions.rs:290` is the root cause. [7](#0-6)
