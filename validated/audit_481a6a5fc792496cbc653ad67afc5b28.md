Audit Report

## Title
Arithmetic Mid-Price Formula in `_afterSwapOracleStopLoss` Systematically Miscalibrates Stop-Loss Watermarks — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean `(bid + ask) / 2` at line 218, while the rest of the protocol uses the geometric mean `sqrt(bid * ask)` via `SwapMath.midAndSpreadFeeX64FromBidAsk`. By AM-GM inequality, the arithmetic mean is always ≥ the geometric mean, causing `metricT0` to be underestimated and `metricT1` to be overestimated. This sets the `zeroForOne` stop-loss watermark lower than the true value, allowing LPs to suffer a larger drawdown than the configured `drawdownE6` before the guard triggers.

## Finding Description
At [1](#0-0)  the mid price is computed as:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
```

This arithmetic mean is fed into `_metrics`, which computes: [2](#0-1) 

- `metricT0` divides by `midPriceX64` — a larger denominator (arithmetic ≥ geometric) yields a **smaller** `metricT0`.
- `metricT1` multiplies by `midPriceX64` — a larger factor yields a **larger** `metricT1`.

The underestimated `metricT0` is then stored as the high watermark `hwmS.token0`: [3](#0-2) 

The breach threshold for `zeroForOne` is `hwm0 * floorMultiplier / E6`: [4](#0-3) 

Because `hwm0` is set too low, the threshold is also too low, so the guard allows the actual per-share value to fall further than `drawdownE6` before reverting.

In contrast, `PriceVelocityGuardExtension` correctly uses the geometric mean: [5](#0-4) 

And `SwapMath.midAndSpreadFeeX64FromBidAsk` explicitly documents and implements the geometric mean: [6](#0-5) 

## Impact Explanation
The `OracleValueStopLossExtension` is the primary on-chain mechanism preventing LPs from suffering unbounded impermanent loss or oracle-manipulation-driven value drain. Its stated invariant is that value per share at oracle marks cannot fall faster than `drawdown + decay × t`. With the arithmetic mean, the `metricT0` watermark is set below the true value by a factor proportional to the oracle spread. For a 10% bid/ask spread the error is ≈0.45%; for a 50% spread it is ≈8.3%. This constitutes a direct, quantifiable loss of LP principal beyond the configured protection level — a medium-to-high severity direct loss of user principal.

## Likelihood Explanation
Every swap on a pool configured with `OracleValueStopLossExtension` triggers `afterSwap`, which calls `_afterSwapOracleStopLoss`. No privileged action is required — any unprivileged swapper causes the watermark to be set with the wrong mid price. The error is always in the same direction (arithmetic ≥ geometric) and accumulates monotonically as the watermark ratchets upward on wrong values. Pools with wider oracle spreads (volatile or illiquid pairs) are most affected.

## Recommendation
Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (wrong):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

## Proof of Concept
**Setup:** Oracle with 10% spread: `ask = 1.1 · Q64`, `bid = Q64 / 1.1 ≈ 0.9091 · Q64`. Bin: `t0 = 1000`, `t1 = 1000`, `shares = 1000`. `drawdownE6 = 100_000` (10% drawdown allowed).

**Geometric mid (correct):**
```
mid_geo = sqrt(0.9091 * 1.1) * Q64 = Q64
metricT0_correct = 1 + 1 = 2.0
threshold_correct = 2.0 * 0.9 = 1.8
```

**Arithmetic mid (current code):**
```
mid_arith = (0.9091 + 1.1) / 2 * Q64 = 1.00455 * Q64
metricT0_wrong = 1 + (1000 * Q64 / (1.00455 * Q64)) / 1000 = 1.99547
threshold_wrong = 1.99547 * 0.9 = 1.79592
```

**Result:** The stop-loss triggers at `metricT0 < 1.79592` instead of `< 1.8`, allowing an additional ≈0.23% of value loss beyond the configured 10% drawdown. For a 50% oracle spread, the additional unprotected drawdown is ≈8.3%.

  function midAndSpreadFeeX64FromBidAsk(uint256 bidPriceX64, uint256 askPriceX64)
    internal
    pure
    returns (uint256 midPriceX64, uint256 baseFeeX64)
  {
    midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
    baseFeeX64 = Math.mulDiv(askPriceX64, ONE_X64, midPriceX64, Math.Rounding.Ceil) - ONE_X64;
