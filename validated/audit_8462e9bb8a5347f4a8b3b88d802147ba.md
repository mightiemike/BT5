### Title
Receipts with gas exceeding `allowed_shard_outgoing_gas` are permanently stuck in the outgoing buffer when `ClampOutgoingGasAdmission` is not enabled — (`runtime/runtime/src/congestion_control.rs`)

### Summary

Before `ProtocolFeature::ClampOutgoingGasAdmission` (protocol version 85) is activated, `ReceiptSinkV2::try_forward` uses the receipt's full gas as the admission threshold when deciding whether to drain a buffered receipt. When a destination shard is fully congested, the `forward_limit.gas` for the "allowed shard" is exactly `allowed_shard_outgoing_gas` (1 PGAS). Any buffered receipt whose gas exceeds 1 PGAS will never satisfy `forward_limit.gas >= admission_gas`, causing it to be permanently stuck in the outgoing buffer — an exact structural analog to `YearnAdapter._protocolWithdraw` calling `yVault.withdraw` without specifying `maxLoss`, where the hardcoded default threshold causes all withdrawals to revert permanently.

### Finding Description

In `ReceiptSinkV2::try_forward`, the admission check is:

```rust
let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
    .enabled(apply_state.current_protocol_version)
{
    gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
} else {
    gas   // ← full receipt gas used without clamping
};

if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    // forward
} else {
    // not forwarded — stays in buffer forever
}
``` [1](#0-0) 

When `ClampOutgoingGasAdmission` is **not** enabled (protocol versions 68–84), `admission_gas = gas` (the full prepaid gas of the receipt). The `outgoing_limit` for a fully congested shard is populated in `ReceiptSink::new` as:

```rust
let gas_limit = if shard_id != apply_state.shard_id {
    other_congestion_control.outgoing_gas_limit(apply_state.shard_id)
} else {
    Gas::MAX
};
``` [2](#0-1) 

For a fully congested shard, `outgoing_gas_limit` returns `Gas::ZERO` for all non-allowed shards, and `allowed_shard_outgoing_gas` (1 PGAS = 1,000,000,000,000,000 gas) for the single "allowed shard". The allowed-shard mechanism is the deadlock-prevention guarantee: it must always be able to drain at least one receipt per chunk. [3](#0-2) 

The `allowed_shard_outgoing_gas` is set to 1 PGAS at protocol version 68: [4](#0-3) 

Without the clamp, the admission check for the allowed shard becomes `1 PGAS >= gas`. Any receipt with gas > 1 PGAS fails this check and is never forwarded. The buffer-draining loop in `forward_from_buffer_to_shard` breaks on the first non-forwarded receipt:

```rust
ReceiptForwarding::NotForwarded(_) => {
    break;
}
``` [5](#0-4) 

So a single large receipt at the head of the buffer permanently blocks all subsequent receipts from being drained, regardless of their size.

The fix introduced by `ClampOutgoingGasAdmission` (protocol version 85) clamps `admission_gas` to `min(gas, allowed_shard_outgoing_gas)`, so the check becomes `1 PGAS >= min(gas, 1 PGAS)`, which is always true, restoring the deadlock-freedom guarantee. [6](#0-5) 

### Impact Explanation

A receipt permanently stuck at the head of a shard's outgoing buffer:

1. **Blocks all subsequent receipts** to that shard from being forwarded (the loop breaks on the first non-forwarded receipt).
2. **Permanently locks any NEAR tokens** attached as deposits to the stuck receipt — they cannot be refunded because the receipt never executes.
3. **Breaks the congestion-control deadlock-freedom invariant**: the "allowed shard" is supposed to guarantee at least one receipt drains per chunk, but this guarantee fails for receipts with gas > 1 PGAS.
4. **Cross-contract calls** that land in the buffer never execute, silently breaking application logic.

This is High/Critical: user funds are permanently locked with no recovery path, and the congestion control system's core safety property is violated.

### Likelihood Explanation

- Congestion control was activated at protocol version 68; `ClampOutgoingGasAdmission` is at version 85. The window of exposure is protocol versions 68–84.
- `MIN_SUPPORTED_PROTOCOL_VERSION = 84`, meaning the minimum version a current binary supports is 84, which is inside the vulnerable window. [7](#0-6) 

- Any user submitting a function call with > 1 PGAS (1,000 TGAS) attached gas produces a receipt that, if buffered during full congestion, is permanently stuck. While 1 PGAS is large for a single call, cross-contract chains and batch receipts can accumulate gas in this range.
- The trigger condition (full congestion on a destination shard) is a normal operational state, not an attack.

### Recommendation

The fix is already deployed as `ProtocolFeature::ClampOutgoingGasAdmission` at protocol version 85. No further action is needed for the current network. For documentation and historical correctness:

1. Add a comment in the `else { gas }` branch explicitly stating it is the pre-v85 legacy path and that it can cause receipt stalls.
2. Consider adding a protocol-version-gated assertion or metric that fires when a receipt is not forwarded due to gas admission failure, to detect regressions if the feature gate is ever incorrectly applied.

### Proof of Concept

1. Network runs at protocol version 84 (congestion control active, `ClampOutgoingGasAdmission` not yet active).
2. Shard S becomes fully congested (`outgoing_gas_limit` returns 0 for all shards except the allowed shard, which gets `allowed_shard_outgoing_gas = 1 PGAS`).
3. A user submits a cross-shard function call with 2 PGAS attached gas targeting shard S. The receipt is buffered in the outgoing buffer.
4. Each chunk, `forward_from_buffer_to_shard` iterates the buffer. For the stuck receipt: `admission_gas = 2 PGAS`, `forward_limit.gas = 1 PGAS`. Check: `1 PGAS >= 2 PGAS` → false → `NotForwarded` → `break`.
5. The receipt never drains. The user's attached deposit is permanently locked. All subsequent receipts in the buffer are also blocked.
6. After upgrading to protocol version 85, `admission_gas = min(2 PGAS, 1 PGAS) = 1 PGAS`, check: `1 PGAS >= 1 PGAS` → true → receipt is forwarded and the buffer drains. [8](#0-7)

```rust
    fn try_forward(
        receipt: Receipt,
        gas: Gas,
        mut size: u64,
        shard: ShardId,
        outgoing_limit: &mut HashMap<ShardId, OutgoingLimit>,
        outgoing_receipts: &mut Vec<Receipt>,
        apply_state: &ApplyState,
        stats: &mut ReceiptSinkStats,
    ) -> Result<ReceiptForwarding, RuntimeError> {
        // There is a bug which allows to create receipts that are above the size limit. Receipts
        // above the size limit might not fit under the maximum outgoing size limit. Let's pretend
        // that all receipts are at most `max_receipt_size` to avoid receipts getting stuck.
        // See https://github.com/near/nearcore/issues/12606
        let max_receipt_size = apply_state.config.wasm_config.limit_config.max_receipt_size;
        if size > max_receipt_size {
            tracing::debug!(
                target: "runtime",
                receipt_id=?receipt.receipt_id(),
                size,
                max_receipt_size,
                "try_forward observed a receipt with size exceeding the size limit",
            );
            size = max_receipt_size;
        }

        // Default case set to `Gas::MAX`: If no outgoing limit was defined for the receiving
        // shard, this usually just means the feature is not enabled. Or, it
        // could be a special case during resharding events. Or even a bug. In
        // any case, if we cannot know a limit, treating it as literally "no
        // limit" is the safest approach to ensure availability.
        let default_gas_limit = Gas::MAX;

        // Since bandwidth scheduler, a shard is not allowed to send any receipts if it doesn't have a grant.
        let default_size_limit = 0;

        let default_outgoing_limit =
            OutgoingLimit { gas: default_gas_limit, size: default_size_limit };
        let forward_limit = outgoing_limit.entry(shard).or_insert(default_outgoing_limit);

        let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
            .enabled(apply_state.current_protocol_version)
        {
            gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
        } else {
            gas
        };

        if forward_limit.gas >= admission_gas && forward_limit.size >= size {
            tracing::trace!(target: "runtime", ?shard, receipt_id=?receipt.receipt_id(), "forwarding buffered receipt");
            outgoing_receipts.push(receipt);
            forward_limit.gas = forward_limit.gas.saturating_sub(gas);
            forward_limit.size -= size;
            stats.forwarded_receipts.entry(shard).or_default().add_receipt(size, gas);

            Ok(ReceiptForwarding::Forwarded)
        } else {
            tracing::trace!(target: "runtime", ?shard, receipt_id=?receipt.receipt_id(), "not forwarding buffered receipt");
            Ok(ReceiptForwarding::NotForwarded(receipt))
        }
    }
```
