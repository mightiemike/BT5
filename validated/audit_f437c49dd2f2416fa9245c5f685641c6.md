### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit guard — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces its allowlist against the `owner` argument (the LP position recipient) rather than the `sender` argument (the actual caller who triggers the deposit and settles tokens). Because `pool.addLiquidity` accepts an arbitrary `owner` address from any caller, an address that is not on the allowlist can deposit into a guarded pool simply by naming an allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension hook:

- `sender` = `msg.sender` — the contract or EOA that actually called `addLiquidity` and will settle tokens via `metricOmmModifyLiquidityCallback`
- `owner` = the explicit parameter — the address that will own the resulting LP position [1](#0-0) 

The extension hook receives both, but `DepositAllowlistExtension` silently discards `sender` (named `address` with no variable) and gates only on `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The contract's own NatSpec states the intent is to gate by **depositor** address. The `SwapAllowlistExtension` correctly gates on `sender` (the direct caller of `pool.swap`): [3](#0-2) 

The asymmetry is the root cause. Because `MetricOmmPoolLiquidityAdder` explicitly supports adding liquidity on behalf of a different `owner` (and the pool itself imposes no restriction on who may supply `owner`), any caller can name an allowlisted address as `owner` and the guard passes: [4](#0-3) 

The test suite confirms this path is reachable and intentionally supported by the periphery: [5](#0-4) 

---

### Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who may add liquidity to a guarded pool (e.g., KYC-gated, institutional, or whitelist-only pools). With this bug:

1. An address not on the allowlist can call `pool.addLiquidity(allowlistedAddress, salt, deltas, ...)` directly, or route through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, allowlistedAddress, ...)`.
2. The `beforeAddLiquidity` hook checks `owner = allowlistedAddress` → passes.
3. The unauthorized caller settles tokens via the callback; LP shares are credited to `allowlistedAddress`.
4. The pool admin's access control is fully bypassed without any privileged action.

The allowlisted address receives LP shares it did not request and cannot prevent. While it can later call `removeLiquidity` (which enforces `msg.sender == owner`), the admin's intent to restrict pool participation is broken. Additionally, an attacker can grief an allowlisted LP by repeatedly forcing unwanted positions into their account, complicating their position management and potentially exposing them to impermanent loss in bins they did not choose.

---

### Likelihood Explanation

- The bypass requires no special privilege — any EOA or contract can call `pool.addLiquidity` or `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with an arbitrary `owner`.
- The attacker only needs to know one allowlisted address (publicly readable from `allowedDepositor` mapping or from past events).
- The attacker bears the token cost, making this a griefing vector rather than a profit-seeking one, but the admin-boundary break is unconditional.

---

### Recommendation

Change `beforeAddLiquidity` to gate on `sender` (the actual depositor/payer) instead of `owner`, consistent with how `SwapAllowlistExtension` gates on `sender`:

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended semantic is to restrict LP ownership (not the payer), the NatSpec and event names should be updated to reflect that, and the allowlist key should be documented as the position owner rather than the depositor.

---

### Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - Admin calls setAllowedToDeposit(pool, Alice, true)
  - Bob is NOT on the allowlist

Attack:
  1. Bob deploys a contract implementing metricOmmModifyLiquidityCallback
     that transfers tokens to the pool on demand.
  2. Bob calls pool.addLiquidity(
         owner    = Alice,   // allowlisted → guard passes
         salt     = 99,
         deltas   = <valid bins>,
         callbackData = "",
         extensionData = ""
     )
  3. beforeAddLiquidity receives sender=Bob, owner=Alice.
     Check: allowedDepositor[pool][Alice] == true → no revert.
  4. Pool executes the deposit; Bob's callback pays the tokens.
  5. Alice now holds LP shares in bins she never chose.
  6. Bob has bypassed the deposit allowlist without being allowlisted.

Alternatively via MetricOmmPoolLiquidityAdder (no custom callback needed):
  Bob calls liquidityAdder.addLiquidityExactShares(
      pool, Alice, 99, deltas, maxAmount0, maxAmount1, ""
  )
  — same outcome; Bob is the payer, Alice is the owner, guard passes.
``` [2](#0-1) [6](#0-5) [4](#0-3)
