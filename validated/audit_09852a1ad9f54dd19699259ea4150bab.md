### Title
Linked Signer Can Self-Escalate by Re-Assigning the Linked Signer Role — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path transaction is validated with `allowLinkedSigner = true`, meaning the **currently registered linked signer** can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. This grants the linked signer account-management authority it should not possess — a direct analog to the email spoofing bug, where missing authentication policy allowed any party to impersonate a trusted identity and act on its behalf.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `LinkSigner` calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateCompactSignature`, which passes `getLinkedSigner(sender)` as the accepted alternate signer: [2](#0-1) 

`Verifier.validateCompactSignature` then accepts a signature from **either** the subaccount owner **or** the currently registered linked signer: [3](#0-2) 

This means the linked signer — an operational key intended for delegated trading — can sign a `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with any arbitrary address: [4](#0-3) 

The `linkedSigners` mapping is the sole source of truth for delegated authority across the protocol: [5](#0-4) 

Furthermore, `getLinkedSigner` propagates the parent's linked signer to **all isolated subaccounts** automatically: [6](#0-5) 

This means a single `LinkSigner` re-assignment on the parent subaccount compromises every isolated subaccount simultaneously.

**Attack path:**
1. Victim sets linked signer to address `L` (e.g., a trading bot API key).
2. Attacker obtains `L`'s signing capability (key exposure, phishing of the bot infrastructure, etc.).
3. Attacker uses `L` to sign a `LinkSigner` transaction pointing to attacker-controlled address `A`.
4. `linkedSigners[victim_subaccount] = A` is written on-chain.
5. Attacker now signs withdrawals, transfers, and liquidation transactions as the new linked signer.
6. Victim attempts revocation via the slow-mode `LinkSigner` path (which uses `validateSender` and requires `msg.sender` to match the subaccount owner). However, the attacker can re-submit a fast-path `LinkSigner` re-link before the slow-mode transaction is executed, maintaining persistent control. [7](#0-6) 

---

### Impact Explanation

A successful attacker gains full delegated authority over the victim's subaccount: they can sign `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, and order transactions. All collateral and positions in the subaccount — and all of its isolated subaccounts — are at risk of theft or forced liquidation. The corrupted state is `linkedSigners[subaccount]`, which is the root of all delegated signing authority in the protocol.

**Impact: 4 / 5**

---

### Likelihood Explanation

Linked signer keys are operational keys (API keys, trading bot keys, hot wallets) that are materially more likely to be exposed than a user's primary cold wallet. Any compromise of the trading infrastructure — server breach, leaked `.env`, phishing of the bot operator — is sufficient. No admin, sequencer, or governance compromise is required. The attacker needs only a single signed `LinkSigner` transaction submitted through the normal sequencer fast path.

**Likelihood: 3 / 5**

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in the fast path. Only the subaccount owner (the address embedded in the `bytes32` subaccount identifier) should be authorized to change the linked signer. The linked signer is a delegated trading key and must not have account-management authority over its own role. [8](#0-7) 

Change the final argument from `true` to `false`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
```

---

### Proof of Concept

```
1. Alice owns subaccount 0xAlice...0001 and sets linkedSigners[0xAlice...0001] = 0xBotKey.
2. Attacker compromises 0xBotKey.
3. Attacker constructs:
     LinkSigner { sender: 0xAlice...0001, signer: 0xAttacker, nonce: N }
   and signs it with 0xBotKey.
4. Attacker submits the signed transaction to the sequencer fast path.
5. EndpointTx.sol line 588 executes:
     linkedSigners[0xAlice...0001] = 0xAttacker
6. Attacker signs WithdrawCollateral { sender: 0xAlice...0001, amount: MAX }
   with 0xAttacker — accepted by Verifier.validateCompactSignature line 315.
7. All collateral is drained.
8. Alice submits slow-mode LinkSigner to revoke; attacker re-links via fast path
   before slow-mode executes, maintaining persistent control.
```

    function validateCompactSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        IEndpoint.CompactSignature memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature.r, signature.vs);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
