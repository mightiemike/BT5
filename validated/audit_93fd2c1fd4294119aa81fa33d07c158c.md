Audit Report

## Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Real Swapper, Allowing Any User to Bypass a Curated Pool's Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap`, so the extension checks the router address rather than the end user. If the router is allowlisted (the only way to permit router-based swaps), every user—including non-allowlisted ones—can bypass the curated allowlist by routing through the router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` value is allowlisted, using `msg.sender` (the pool) as the key and `sender` (the immediate caller of `pool.swap`) as the swapper: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender` of that call, not the end user: [4](#0-3) 

The same pattern holds for `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The result is a structural dilemma with no valid configuration: allowlisting the router permits every user to bypass the allowlist, while not allowlisting the router prevents allowlisted users from using the router at all. There is no existing guard in `SwapAllowlistExtension`, `ExtensionCalling`, or `MetricOmmPool` that propagates the originating user identity through the router call stack.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC-verified counterparties, institutional participants, or whitelisted market makers) is fully bypassed by any unprivileged user who calls `MetricOmmSimpleRouter.exactInputSingle` or any other router entry point. The non-allowlisted user receives pool output tokens and pays input tokens exactly as if they were allowlisted. The pool's curation invariant is broken, and LP exposure to non-approved counterparties is unlimited. This constitutes broken core pool functionality causing loss of funds and an admin-boundary break where the pool admin's allowlist configuration is bypassed by an unprivileged path.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface. Any pool admin who enables the router for their pool (the expected production configuration) and also deploys `SwapAllowlistExtension` will be in the vulnerable state. The bypass requires no special privileges, no flash loans, and no multi-transaction setup—a single `exactInputSingle` call suffices. The attack is repeatable and permissionless.

## Recommendation
The `sender` argument passed to `beforeSwap` must represent the end user, not the immediate caller of `pool.swap`. Two complementary fixes:

1. **Router-side**: Have the router encode the original `msg.sender` inside `extensionData` and pass it to the pool. The extension can then decode and check that address.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should accept and decode a user-address payload from `extensionData` when `sender` is a known router, falling back to `sender` for direct calls.

The cleanest long-term fix is for `MetricOmmPool.swap` to accept an explicit `originator` parameter (separate from `recipient`) that the router sets to `msg.sender` before forwarding to the pool, and for extensions to check `originator` rather than `sender`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap` order.
2. Call `extension.setAllowedToSwap(pool, alice, true)` — only Alice is allowlisted.
3. Call `extension.setAllowedToSwap(pool, router, true)` — router is allowlisted so Alice can use it.
4. As `bob` (not allowlisted), call `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
5. The pool calls `extension.beforeSwap(sender=router, ...)` because the router is `msg.sender` of `pool.swap`.
6. The check `allowedSwapper[pool][router]` is `true` → no revert.
7. Bob's swap executes successfully on the curated pool, bypassing the allowlist entirely.

The corrupted value is `sender` inside `beforeSwap`: it holds the router address instead of the actual swapper (`bob`), causing the allowlist lookup to return `true` for a non-allowlisted user. [6](#0-5) [7](#0-6) [8](#0-7)
