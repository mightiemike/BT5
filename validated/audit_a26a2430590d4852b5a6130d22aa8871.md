Audit Report

## Title
SwapAllowlistExtension checks router address instead of originating EOA, allowing any user to bypass per-user swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` equal to `msg.sender` as seen by the pool, which is the router contract address when a swap is routed through `MetricOmmSimpleRouter`. If a pool admin allowlists the router to support router-based swaps, every user of that router passes the allowlist check regardless of whether their own address is individually permitted, completely defeating the per-user access control the extension is designed to enforce.

## Finding Description
The call chain is confirmed in production code:

**Step 1 — Router calls pool directly:** `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` with no originator field. [1](#0-0) 

**Step 2 — Pool passes `msg.sender` (= router) as `sender` to `_beforeSwap`:** `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, so when the caller is the router, `sender` = router address. [2](#0-1) 

**Step 3 — `ExtensionCalling._beforeSwap` forwards that `sender` (= router) to the extension unchanged:** [3](#0-2) 

**Step 4 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][router]`:** [4](#0-3) 

The originating EOA is stored in transient storage by the router via `_setNextCallbackContext(..., msg.sender, ...)` for payment purposes only; it is never threaded into the extension call path and is inaccessible from within `beforeSwap`. [5](#0-4) [6](#0-5) 

There is no mechanism within the extension to recover the originating EOA. `tx.origin` is not used (correctly), and no originator field is passed through the hook interface.

## Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g. KYC'd addresses, whitelisted market makers) and also allowlists the router to support router-based swaps inadvertently grants unrestricted swap access to every user of that router. Any non-allowlisted EOA can execute swaps against the pool by routing through `MetricOmmSimpleRouter`, causing unauthorized token outflow from a pool whose admin explicitly intended to restrict access. This constitutes broken core pool functionality with direct fund impact — the exact wrong value is the extension's `allowedSwapper[pool][sender]` decision, which resolves `true` for the router when it should resolve `false` for the non-allowlisted EOA. [7](#0-6) 

## Likelihood Explanation
The scenario requires only that a pool uses `SwapAllowlistExtension` and also needs to support router-based swaps — a natural and expected configuration. The router is a public, permissionless contract. No privileged access is required by the attacker. Any EOA can call `exactInputSingle` on the router targeting such a pool. The condition is straightforward and repeatable. [8](#0-7) 

## Recommendation
Thread the originating caller through the swap path so the extension can check the true economic actor. One approach: have the router encode the originating EOA into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when present. Alternatively, gate on the payer stored in transient storage (`_getPayer()`) by exposing it through a router-side hook, or introduce a dedicated router-aware allowlist that validates the payer rather than the pool-visible `sender`. [6](#0-5) 

## Proof of Concept
```solidity
// Foundry integration test sketch
function test_routerBypassesSwapAllowlist() public {
    // Pool admin deploys extension, allowlists ONLY the router (not the EOA attacker)
    SwapAllowlistExtension ext = new SwapAllowlistExtension(factory);
    // ... deploy pool with ext as beforeSwap extension ...
    ext.setAllowedToSwap(pool, address(router), true); // router allowlisted
    // ext.setAllowedToSwap(pool, attacker, false);    // attacker NOT allowlisted

    // Non-allowlisted EOA routes through the router
    vm.prank(attacker);
    uint256 out = router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: pool,
            tokenIn: token0,
            recipient: attacker,
            zeroForOne: true,
            amountIn: 1e18,
            amountOutMinimum: 0,
            priceLimitX64: 0,
            deadline: block.timestamp,
            extensionData: ""
        })
    );
    // Swap succeeds: allowedSwapper[pool][router] == true, attacker's address never checked
    assertGt(out, 0);
}
```

The check at `SwapAllowlistExtension` L37 evaluates `allowedSwapper[pool][router]` which is `true`, so the revert at L38 is never triggered, and the non-allowlisted EOA's swap completes successfully. [9](#0-8)

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
