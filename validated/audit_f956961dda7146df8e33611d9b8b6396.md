Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual swapper, allowing full allowlist bypass via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is the pool's `msg.sender` — the direct caller of `pool.swap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. The extension therefore checks whether the router is allowlisted, not the real economic actor. Any non-allowlisted user can bypass a curated pool's access control with a single router call, breaking the pool admin's intended access boundary.

## Finding Description

**Root cause — `SwapAllowlistExtension.beforeSwap`:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is the first argument the pool forwarded. The extension assumes `sender` is the economic actor.

**How the pool populates `sender` — `MetricOmmPool.swap`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← pool's msg.sender, i.e. whoever called pool.swap
    ...
);
```

**How the router calls `pool.swap` — `MetricOmmSimpleRouter.exactInputSingle`:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The router is `msg.sender` to the pool. The pool passes `address(router)` as `sender` to `_beforeSwap`, which forwards it verbatim through `ExtensionCalling._beforeSwap` (L149-177) to the extension. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**Multi-hop path — `exactInput` intermediate hops:**

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L103
_setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
```

For hops after the first, the payer is `address(this)` (the router), so the extension again sees the router for every intermediate pool.

**The dilemma for pool admins:**
- Do **not** allowlist the router → all router-mediated swaps revert, even for legitimately allowlisted users who use the router.
- **Do** allowlist the router (required for any router-mediated swap to work) → every non-allowlisted user bypasses the allowlist via a single `exactInputSingle` call.

There is no mechanism in `ExtensionCalling._beforeSwap` or the extension itself to recover the original EOA from `extensionData`.

## Impact Explanation

A pool deploying `SwapAllowlistExtension` to restrict trading to KYC'd users, institutional partners, or protocol-controlled addresses is fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The pool admin's access-control boundary is broken by the standard, publicly accessible periphery path. This is a direct admin-boundary break: a factory/pool role check (the swap allowlist) is bypassed by an unprivileged path. The wrong value is the extension's boolean decision — it evaluates `allowedSwapper[pool][router] == true` and passes, when it should evaluate `allowedSwapper[pool][attacker] == false` and revert.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. The bypass requires no special privileges, no flash loans, and no multi-transaction setup — a single `exactInputSingle` call suffices. Any user who reads the periphery interface can execute this. Likelihood is **High**.

## Recommendation

The extension must identify the economic actor, not the immediate caller of `pool.swap`. Two viable approaches:

1. **Embed the real user in `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it, enforcing that only a factory-registered router may supply this field.
2. **Router-aware check:** The extension reads the pool's factory, checks whether `sender` is a registered router, and if so decodes the real user from `extensionData`.

The simplest interim mitigation is to prohibit allowlisting the router address and require all curated-pool users to call `pool.swap` directly, but this breaks UX. The correct long-term fix is option 1 or 2.

// - pool configured with SwapAllowlistExtension
// - allowedSwapper[pool][allowedUser] = true

// - allowedSwapper[pool][attacker] = false (never set)

vm.prank(attacker); // attacker is NOT in allowedSwapper[pool]
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: pool,
        tokenIn: token0,
        recipient: attacker,
        zeroForOne: true,
        amountIn: 1000e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        extensionData: "",
        deadline: block.timestamp
    })
);
// pool.swap is called with msg.sender == router
// _beforeSwap(router, ...) → extension checks allowedSwapper[pool][router] == true → passes
// Attacker swaps successfully on a pool they are not authorized to use.
