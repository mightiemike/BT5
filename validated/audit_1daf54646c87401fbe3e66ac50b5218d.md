### Title
Unenforceable Cross-Crate Invariant Between `BouncerWeights::receipt_l2_gas` and `VersionedConstants::max_block_size` Causes Incorrect EIP-1559 Fee Calculation - (File: `crates/blockifier/src/bouncer.rs`)

### Summary
`BouncerWeights::receipt_l2_gas` (blockifier, runtime config) and `VersionedConstants::max_block_size` (apollo_versioned_constants, versioned JSON) are documented with "Must stay in sync" comments but live in separate crates with no runtime enforcement. When a new Starknet version ships a changed `max_block_size` without a corresponding update to the node's `receipt_l2_gas` config, the EIP-1559 fee market and the block bouncer operate on divergent block-size assumptions, producing systematically wrong gas prices.

### Finding Description

Two separate values govern the same logical quantity — the maximum L2 gas a block may consume — but are owned by different subsystems with no cross-validation:

**Value 1 — `BouncerWeights::receipt_l2_gas`** (blockifier, runtime config):

```rust
// NOTE: Must stay in sync with orchestrator_versioned_constants' max_block_size.
pub receipt_l2_gas: GasAmount,
```

Default hardcoded to `GasAmount(5800000000)` in `BouncerWeights::default()`. This is the bouncer's hard cap; once accumulated `receipt_l2_gas` reaches this value the block is closed.

**Value 2 — `VersionedConstants::max_block_size`** (apollo_versioned_constants, per-version JSON):

```rust
/// The maximum block size in gas units.
// NOTE: Must stay in sync with BouncerWeights receipt_l2_gas.
// NOTE: When max_block_size is changed, update `gas_target` accordingly to maintain the ratio.
pub max_block_size: GasAmount,
```

Loaded from `orchestrator_versioned_constants_<version>.json` at compile time. Used by `calculate_next_base_gas_price` to assert `gas_target < max_block_size` and to calibrate the EIP-1559 price-adjustment denominator.

**The divergence path:**

1. A new Starknet version (e.g., `V0_14_5`) ships a new `orchestrator_versioned_constants_0_14_5.json` with an updated `max_block_size` (and a proportionally updated `gas_target`).
2. The binary is rebuilt; `VersionedConstants::max_block_size` and `gas_target` change automatically.
3. The node's runtime config (`config_schema.json` / operator-supplied config) still carries the old `receipt_l2_gas = 5800000000`.
4. No startup assertion or cross-validation catches the mismatch.

**No enforcement exists.** The only related test is `versioned_constants_gas_target_is_valid` in `crates/apollo_consensus_orchestrator/src/fee_market/test.rs`, which only checks `gas_target < max_block_size`; it does not check `max_block_size == receipt_l2_gas`. The config_schema.json description itself acknowledges the invariant informally: *"Should equal max_block_size."*

### Impact Explanation

`calculate_next_base_gas_price` computes the EIP-1559 price adjustment using `gas_target` (derived as a fixed ratio of `max_block_size`). The bouncer closes blocks at `receipt_l2_gas`. When these diverge:

- If `receipt_l2_gas < max_block_size`: blocks close before reaching the fee market's assumed capacity. Actual gas-per-block is bounded by `receipt_l2_gas`, which may be below `gas_target`. The fee market perpetually sees "underfull" blocks and drives the gas price down — users systematically underpay.
- If `receipt_l2_gas > max_block_size`: blocks can exceed the fee market's assumed capacity. The fee market perpetually sees "overfull" blocks and drives the gas price up — users systematically overpay.

Both cases produce **incorrect fee, gas, and L1 gas price effects with direct economic impact**, matching the Critical impact scope: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

### Likelihood Explanation

Every minor Starknet version bump that adjusts `max_block_size` in the versioned constants JSON silently invalidates the existing node config's `receipt_l2_gas`. Because the two values live in different crates and different configuration surfaces (compiled-in JSON vs. runtime config file), a developer updating one has no automated signal to update the other. The "Must stay in sync" comments are the only guard, and they are not machine-checked.

### Recommendation

Add a startup assertion (or a dedicated test) that reads `VersionedConstants::latest_constants().max_block_size` and compares it against the configured `BouncerConfig::block_max_capacity.receipt_l2_gas`, failing fast if they differ. For example, in `open_storage_internal` or the batcher's initialization path:

```rust
assert_eq!(
    versioned_constants.max_block_size,
    bouncer_config.block_max_capacity.receipt_l2_gas,
    "receipt_l2_gas must equal max_block_size; update the node config after a version upgrade."
);
```

Alternatively, derive `receipt_l2_gas` directly from `VersionedConstants::max_block_size` at block-building time rather than reading it from a separate config field, eliminating the two-value invariant entirely.

### Proof of Concept

1. Current state: `BouncerWeights::default().receipt_l2_gas = GasAmount(5800000000)` (hardcoded); `config_schema.json` `receipt_l2_gas = 5800000000`.
2. A new versioned constants file `orchestrator_versioned_constants_0_14_5.json` is added with `"max_block_size": 6000000000` and `"gas_target": 3600000000` (60 % ratio).
3. The binary is rebuilt. `VersionedConstants::latest_constants().max_block_size = 6000000000`, `gas_target = 3600000000`.
4. The operator does not update the node config; `receipt_l2_gas` remains `5800000000`.
5. The bouncer closes blocks at 5800000000 gas. The fee market's `gas_target` is 3600000000. Since `5800000000 > 3600000000`, blocks will frequently appear "overfull" to the fee market, driving the gas price upward beyond what the actual block capacity warrants.
