### Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` **to the pool**, not the original end-user. When a swap is routed through `MetricOmmSimpleRouter`, `sender` resolves to the router's address. If the pool admin allowlists the router (a natural action to enable router-mediated swaps for curated users), every unprivileged user can bypass the allowlist entirely by routing through the router.

---

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension dispatcher.**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with `msg.sender` as the `sender` argument: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards that same `sender` verbatim to every configured extension.** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`.**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (the pool calls the extension), and `sender` is the first parameter — which equals `msg.sender` **to the pool**.

**Step 4 — `MetricOmmSimpleRouter` is `msg.sender` to the pool, not the end-user.**

`exactInputSingle` calls `pool.swap(params.recipient, ...)` directly; the pool sees `msg.sender = router`: [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**The resulting invariant break:**

| Scenario | Extension checks | Result |
|---|---|---|
| Router NOT allowlisted, allowlisted user calls router | `allowedSwapper[pool][router]` = false | Allowlisted user **cannot** use the router |
| Router IS allowlisted, non-allowlisted user calls router | `allowedSwapper[pool][router]` = true | Non-allowlisted user **bypasses** the allowlist |

A pool admin who wants to enable router-mediated swaps for their curated users will naturally allowlist the router address. Doing so silently opens the pool to every user who routes through `MetricOmmSimpleRouter`, defeating the entire purpose of the allowlist.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to KYC'd, institutional, or otherwise curated addresses can be fully bypassed. Any unprivileged user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`) targeting the restricted pool. The extension sees the router's address, which is allowlisted, and permits the swap. The non-allowlisted user receives pool output tokens at oracle-anchored prices, draining LP value that was reserved for curated counterparties. This is a direct loss of LP principal and a complete failure of the pool's access-control invariant.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary supported periphery entry point. A pool admin who deploys a curated pool and wants their allowlisted users to be able to use the router has no choice but to allowlist the router address — there is no mechanism to pass the original user's identity through the current extension interface. The admin action is not malicious; it is the only available path to enable router support, and its side-effect (universal bypass) is non-obvious. Likelihood is **Medium** (requires admin to allowlist the router, which is the natural and expected configuration for any pool that supports periphery routing).

---

### Recommendation

The extension must gate on the **economically relevant actor**, not the intermediary. Two viable approaches:

1. **Pass original sender via `extensionData`**: The router encodes `msg.sender` (the end-user) into `extensionData`; the extension decodes and verifies it. The pool's `onlyPool` guard on the extension entry point ensures only a legitimate pool can invoke the check.

2. **Dedicated router allowlist**: Separate `allowedRouter` from `allowedSwapper`. When `sender` is a known router, decode the real user from `extensionData` and apply the swapper check against that address.

Either approach closes the gap between the actor the admin intends to gate and the actor the extension actually checks.
