### Title
`slowModeFees` Accounting Corrupted by Hardcoded Decimal Assumption — (`File: core/contracts/EndpointTx.sol` / `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.getSlowModeFee()` correctly scales the fee to the quote token's native decimals using a hardcoded base of `6`. However, `EndpointTx.submitSlowModeTransactionImpl()` always increments the internal `slowModeFees` accounting variable by the raw constant `SLOW_MODE_FEE = 1000000` (6-decimal units), regardless of the quote token's actual decimals. This creates a permanent mismatch between the tokens actually collected from users and the amount recorded in `slowModeFees`.

---

### Finding Description

`SLOW_MODE_FEE` is defined as `1000000` — $1 expressed in 6-decimal units (USDC/USDT scale). [1](#0-0) 

`getSlowModeFee()` scales this constant to the quote token's native decimals by hardcoding `6` as the base:

```solidity
int256 multiplier = int256(10**(token.decimals() - 6));
return uint256(int256(SLOW_MODE_FEE) * multiplier);
``` [2](#0-1) 

`chargeSlowModeFee` then transfers exactly `getSlowModeFee()` tokens from the user — the correct token-native amount: [3](#0-2) 

But immediately after charging the user, `submitSlowModeTransactionImpl` increments `slowModeFees` by the raw constant `SLOW_MODE_FEE`, not by the scaled amount:

```solidity
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += SLOW_MODE_FEE;   // always 1_000_000 (6-dec units)
``` [4](#0-3) 

For a quote token with 18 decimals (DAI, WETH, etc.):

| | Actual tokens collected | `slowModeFees` increment |
|---|---|---|
| Per slow-mode tx | `10^18` (1 token, 18-dec) | `1_000_000` (6-dec units) |
| After N txs | `N × 10^18` | `N × 10^6` |
| Ratio | — | **under-counted by `10^12`** |

The `slowModeFees` state variable is therefore permanently and systematically wrong for any quote token that is not a 6-decimal asset. [5](#0-4) 

---

### Impact Explanation

`slowModeFees` is the protocol's internal ledger of collected slow-mode fee revenue. Any downstream function that reads `slowModeFees` to determine how many quote tokens to transfer to the sequencer or fee recipient will use a value that is `10^12` times smaller than the actual tokens held. For an 18-decimal quote token, the sequencer/protocol would be credited with effectively zero slow-mode fee revenue despite users having paid the correct on-chain amounts. The collected tokens remain locked in the contract with no corresponding accounting entry.

Additionally, if the quote token has **fewer than 6 decimals**, `token.decimals() - 6` underflows (Solidity 0.8+ reverts on `uint8` underflow), causing `getSlowModeFee()` to always revert. This makes `chargeSlowModeFee` always revert, permanently bricking all non-owner slow-mode transaction submission for that deployment. [6](#0-5) 

---

### Likelihood Explanation

Any user calling `submitSlowModeTransaction` (e.g., to submit a `WithdrawCollateral`, `LinkSigner`, or similar slow-mode transaction) triggers the path. The trigger requires only that the quote token has decimals ≠ 6. This is a realistic deployment scenario — DAI (18 dec) or any non-USDC/USDT quote asset would reproduce the bug on every slow-mode submission. [7](#0-6) 

---

### Recommendation

Replace the hardcoded `SLOW_MODE_FEE` increment with the actual scaled fee amount returned by `getSlowModeFee()`:

```solidity
uint256 feeCharged = clearinghouse.getSlowModeFee();
chargeSlowModeFee(_getQuote(), sender);
slowModeFees += int128(int256(feeCharged));
```

Alternatively, store `slowModeFees` in a normalized 18-decimal unit and convert on both sides consistently, mirroring the `depositCollateral` pattern already used in `Clearinghouse.sol`: [8](#0-7) 

---

### Proof of Concept

1. Deploy with an 18-decimal quote token (e.g., DAI).
2. User calls `submitSlowModeTransaction` with a `WithdrawCollateral` payload.
3. `chargeSlowModeFee` transfers `getSlowModeFee() = 1_000_000 × 10^12 = 10^18` DAI from the user. ✓ correct token transfer.
4. `slowModeFees += SLOW_MODE_FEE` adds `1_000_000` to `slowModeFees`. ✗ should add `10^18`.
5. After 1000 slow-mode transactions: contract holds `1000 × 10^18` DAI in slow-mode fees, but `slowModeFees = 1_000_000_000` (only `10^-9` DAI equivalent in 18-dec terms).
6. Any fee-claim logic reading `slowModeFees` distributes `~0` DAI to the sequencer; `1000 × 10^18` DAI remains permanently stranded. [4](#0-3) [2](#0-1)
