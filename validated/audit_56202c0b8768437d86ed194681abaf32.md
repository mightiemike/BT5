### Title
Interactions-Before-Effects in `withdrawCollateral` Enables Double-Withdrawal via ERC777 Reentrancy — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` executes the outbound token transfer before updating the sender's balance in `SpotEngine` and before performing the health check. No `ReentrancyGuard` exists anywhere in the production contract suite. If the collateral token is ERC777-compliant, the recipient's `tokensReceived` hook fires during the transfer while the sender's on-chain balance is still at its pre-withdrawal level. An attacker can exploit this window by calling the unguarded public function `Endpoint.executeSlowModeTransaction()` to process a pre-staged second withdrawal, which passes the health check against the stale balance and drains double the entitled collateral.

---

### Finding Description

In `Clearinghouse.withdrawCollateral` the execution order is:

1. **Line 408 — Interaction**: `handleWithdrawTransfer(token, sendTo, amount, idx)` — tokens leave the protocol and reach `sendTo`.
2. **Line 412 — Effect**: `spotEngine.updateBalance(productId, sender, amountRealized)` — balance decremented.
3. **Line 413 — Effect**: `spotEngine.assertUtilization(productId)`.
4. **Line 419 — Check**: `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)`.
5. **Line 420 — Effect**: `emit ModifyCollateral(...)`. [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse.sol` first transfers tokens to `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which calls `BaseWithdrawPool.handleWithdrawTransfer`, which executes `token.safeTransfer(to, uint256(amount))`. [2](#0-1) [3](#0-2) 

If the collateral token is ERC777, `token.safeTransfer(to, amount)` triggers the `tokensReceived` hook on `to` (the attacker's contract). At that exact moment, `spotEngine.updateBalance` has not yet been called — the sender's balance in `SpotEngine` is still the full pre-withdrawal amount.

`Endpoint.executeSlowModeTransaction` is `external` with no reentrancy guard: [4](#0-3) 

It reads `slowModeConfig` from storage, processes the next queued slow-mode transaction, and writes back. When withdrawal A is processed by the sequencer via `submitTransactionsChecked` (a direct `WithdrawCollateral` transaction type, not an `ExecuteSlowMode` wrapper), `slowModeConfig` in storage is **not** modified during withdrawal A's execution. Therefore, when the re-entrant call to `executeSlowModeTransaction()` loads `slowModeConfig` from storage, it correctly finds `txUpTo` pointing to withdrawal B — a separately pre-staged slow-mode withdrawal for the same subaccount. Because the balance has not yet been decremented by withdrawal A, the health check for withdrawal B passes, and withdrawal B is paid out in full.

A grep across all production Solidity files confirms zero uses of `nonReentrant` or `ReentrancyGuard`: [5](#0-4) 

---

### Impact Explanation

An attacker holding balance `X` in a subaccount can withdraw `2X` (or more, with additional pre-staged slow-mode transactions) while only being entitled to `X`. This is a direct collateral theft from the protocol's token reserves. The `SpotEngine` balance is decremented only once (by withdrawal A completing), leaving the protocol short by `X` tokens per exploit cycle. The `ModifyCollateral` event is also emitted after the transfer, corrupting the event log ordering and causing off-chain accounting systems to observe an inconsistent state.

---

### Likelihood Explanation

The precondition is that the collateral token is ERC777-compliant (or any token with a receive-hook, such as tokens with `_afterTokenTransfer` callbacks). ERC777 tokens are a supported and documented token standard. The attacker needs only to: (a) hold a balance in the protocol, (b) deploy a receiver contract, (c) pre-stage a slow-mode withdrawal and wait for the delay, and (d) trigger the sequencer to process a direct withdrawal. Steps (c) and (d) are fully within an unprivileged user's control. No privileged access, governance capture, or key compromise is required.

---

### Recommendation

1. Apply OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to `Clearinghouse.withdrawCollateral`, `Endpoint.executeSlowModeTransaction`, and `Endpoint.depositCollateralWithReferral`.
2. Restructure `withdrawCollateral` to follow strict CEI: update `spotEngine` balance and perform the health check **before** calling `handleWithdrawTransfer`.

Corrected order for `withdrawCollateral`:
```solidity
// 1. CHECKS
require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
// ...

// 2. EFFECTS
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
emit ModifyCollateral(amountRealized, sender, productId);

// 3. INTERACTION
handleWithdrawTransfer(token, sendTo, amount, idx);
```

### Proof of Concept

1. Attacker deploys `AttackerContract` with an ERC777 `tokensReceived` hook.
2. Attacker deposits `X` tokens into subaccount `A` via `Endpoint.depositCollateral`.
3. Attacker calls `Endpoint.submitSlowModeTransaction` to queue a slow-mode `WithdrawCollateral` for amount `X` (withdrawal B). Waits for `SLOW_MODE_TX_DELAY` to pass so it becomes executable.
4. Attacker signals the sequencer (or waits for normal sequencer operation) to submit a direct `WithdrawCollateral` transaction for amount `X` with `sendTo = address(AttackerContract)` (withdrawal A).
5. Sequencer calls `Endpoint.submitTransactionsChecked([withdrawalA_tx])`. This calls `Clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `BaseWithdrawPool.handleWithdrawTransfer` → `token.safeTransfer(AttackerContract, X)`.
6. ERC777 `tokensReceived` fires on `AttackerContract`. At this moment, `spotEngine` balance for subaccount `A` is still `+X` (not yet decremented).
7. `AttackerContract.tokensReceived` calls `Endpoint.executeSlowModeTransaction()`. Storage `slowModeConfig.txUpTo` points to withdrawal B (unmodified by withdrawal A's sequencer path). Withdrawal B's `executableAt` has passed. `executeSlowModeTransaction` calls `Clearinghouse.withdrawCollateral` for amount `X` again. Balance is still `+X`, health check passes. `X` tokens are transferred to `AttackerContract`.
8. Re-entry returns. Withdrawal A completes: `spotEngine.updateBalance` decrements balance by `X` (to `0`). Health check passes (balance is now `0`, which is `>= 0`).
9. **Result**: Attacker received `2X` tokens. Protocol's token reserve is short by `X`. [6](#0-5) [7](#0-6) [4](#0-3)
