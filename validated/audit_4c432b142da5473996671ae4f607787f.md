### Title
Deprecated `verify_transaction_inclusion` Remains Externally Callable, Enabling 64-Byte Merkle Proof Forgery to Bypass SPV Security Fix — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was deprecated in favor of `verify_transaction_inclusion_v2` specifically to close the 64-byte transaction Merkle proof forgery vulnerability. However, v1 remains a fully public, externally callable NEAR contract method. Rust's `#[deprecated]` attribute only emits compile-time warnings for Rust callers — it does not prevent any external NEAR RPC caller from invoking the method by name. Any unprivileged caller can therefore bypass the coinbase-proof guard introduced in v2 and obtain a `true` verification result for a forged proof.

---

### Finding Description

`verify_transaction_inclusion` is marked deprecated at `contract/src/lib.rs:283–286`:

```rust
#[deprecated(
    since = "0.5.0",
    note = "Use `verify_transaction_inclusion_v2` instead."
)]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

The deprecation was introduced to mitigate the 64-byte transaction Merkle proof forgery vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). The fix was implemented in `verify_transaction_inclusion_v2` (`contract/src/lib.rs:347–369`), which first validates a coinbase Merkle proof before delegating to v1:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

The coinbase-proof check in v2 is what prevents an attacker from supplying an internal 64-byte Merkle node as `tx_id`. v1 has no such check and the contract itself documents the risk:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."

Because `verify_transaction_inclusion` is `pub` and compiled into the WASM binary, any external NEAR account can call it directly via RPC, completely bypassing v2's coinbase guard. The `#[deprecated]` attribute is invisible at the ABI level.

---

### Impact Explanation

The broken invariant is: *a `true` result from the contract's proof verification API guarantees that a real Bitcoin transaction was included in a confirmed mainchain block.*

An attacker who calls v1 directly can supply a crafted 64-byte internal Merkle node as `tx_id` along with a valid Merkle path to the block's `merkle_root`. v1 computes `compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof)` and compares it to `header.block_header.merkle_root` — this comparison succeeds for a forged internal node, returning `true`. Any dApp or bridge contract that gates fund releases on a `true` result from `verify_transaction_inclusion` (v1) will release funds for a Bitcoin transaction that never existed.

---

### Likelihood Explanation

Medium. The 64-byte forgery technique is publicly documented and well understood. The v1 method name is stable and discoverable from the contract ABI. Any dApp that integrated before v0.5.0 or that calls v1 by name (rather than v2) is a concrete target. No privileged access, key material, or social engineering is required — only a valid Bitcoin block and knowledge of its Merkle tree structure.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or replace its body with an unconditional panic:

```rust
pub fn verify_transaction_inclusion(&self, ...) -> bool {
    env::panic_str("verify_transaction_inclusion is removed; use verify_transaction_inclusion_v2");
}
```

Keeping the method callable while marking it deprecated creates the same class of desynchronization as the reported Starknet bug: the security fix exists on one path (v2) but the vulnerable path (v1) remains reachable.

---

### Proof of Concept

1. Identify any real Bitcoin block `B` with known `merkle_root` and at least two transactions.
2. Compute an internal Merkle node `N` (64 bytes = concatenation of two child hashes) that lies on the Merkle path to `merkle_root`.
3. Construct `ProofArgs { tx_id: N, tx_block_blockhash: B, tx_index: <index of N>, merkle_proof: <path from N to root>, confirmations: 1 }`.
4. Call `verify_transaction_inclusion(args)` directly via NEAR RPC.
5. The call returns `true` — no coinbase proof was required, no real transaction was referenced.
6. A dApp gating on this result releases funds. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
