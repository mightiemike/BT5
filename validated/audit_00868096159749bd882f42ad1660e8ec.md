### Title
Unsafe `transferFrom` in `replaceUsdcEWithUsdc` Allows Draining DDA usdcE Without Providing USDC — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` uses a raw, unchecked `IERC20Base(usdc).transferFrom(...)` to pull USDC from the caller before draining usdcE from a `DirectDepositV1` (DDA) contract. If the USDC `transferFrom` returns `false` instead of reverting (e.g., during a USDC pause), execution continues, the DDA's entire usdcE balance is withdrawn to `ContractOwner`, and then forwarded to the caller via a `safeTransfer`. The caller receives usdcE without ever providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an unrestricted `external` function (only gated by `block.chainid == 57073`). Its intended flow is:

1. Pull `balance` of USDC from `msg.sender` into the DDA.
2. Withdraw all usdcE from the DDA to `ContractOwner`.
3. Forward that usdcE to `msg.sender`.

The critical flaw is at line 616:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // return value ignored
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));       // drains DDA unconditionally
IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // pays caller unconditionally
``` [1](#0-0) 

The return value of `transferFrom` is never checked. Steps 2 and 3 execute regardless of whether step 1 succeeded. By contrast, the outbound usdcE transfer on line 618 correctly uses `safeTransfer`, making the asymmetry a clear oversight. [2](#0-1) 

`DirectDepositV1.withdraw` is `onlyOwner` (owner = `ContractOwner`), so it unconditionally transfers the full usdcE balance of the DDA to `ContractOwner`, which then forwards it to the caller. [3](#0-2) 

---

### Impact Explanation

An attacker targeting any DDA that holds usdcE can:

- Receive the full usdcE balance of the DDA without paying any USDC.
- The DDA is left with zero usdcE and zero USDC (the USDC transfer silently failed).
- Any user whose funds were held in that DDA pending a `creditDeposit` call loses their usdcE permanently — it cannot be recovered without governance intervention.

The corrupted asset delta is exact: the DDA loses `balance` usdcE tokens; the attacker gains `balance` usdcE tokens; the protocol receives zero USDC in exchange.

---

### Likelihood Explanation

The trigger condition — USDC's `transferFrom` returning `false` instead of reverting — is realistic and precedented:

- Circle's USDC contract on some chains implements a pausable transfer mechanism. During the March 2023 SVB crisis, USDC's peg broke and emergency pause scenarios were actively discussed.
- On Ink (chain ID 57073), the hardcoded USDC address `0x2D270e6886d130D724215A266106e6832161EAEd` is a bridged/wrapped variant whose exact pause and return-value behavior may differ from mainnet USDC.
- Any DDA that has accumulated usdcE is a valid target; the function is callable by any EOA or contract with no further preconditions. [4](#0-3) 

---

### Recommendation

Replace the raw `transferFrom` with the project's own `safeTransferFrom` wrapper (already present in `ERC20Helper`) and apply the Checks-Effects-Interactions pattern by verifying the inbound transfer before executing the outbound withdrawal:

```solidity
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
    address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance); // safe
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);
    }
}
```

`ERC20Helper.safeTransferFrom` already exists in the codebase and performs the necessary low-level call with return-value validation. [5](#0-4) 

---

### Proof of Concept

1. A DDA exists for some `subaccount` and holds `N` usdcE tokens (e.g., deposited by a user before the migration).
2. USDC on Ink is paused (or the attacker has zero USDC balance and the token returns `false` rather than reverting).
3. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)`.
4. `IERC20Base(usdc).transferFrom(attacker, dda, N)` returns `false`; no revert; no USDC moves.
5. `DirectDepositV1(dda).withdraw(usdcE)` executes: DDA's `N` usdcE is transferred to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes: attacker receives `N` usdcE.
7. Net result: attacker gains `N` usdcE; DDA loses `N` usdcE; protocol receives `0` USDC. [6](#0-5)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
