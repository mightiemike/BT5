### Title
Rebasing Token Rewards Permanently Stuck in Clearinghouse Due to Fixed-Amount Slow Mode Queue Accounting - (File: `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

The Nado protocol's deposit flow transfers tokens to `Clearinghouse` immediately but records the original `amount` in the slow mode queue for deferred accounting. If a rebasing token (e.g., stETH) is used as collateral, any positive rebase that occurs during the mandatory slow mode delay results in the rebased increment being permanently stuck in `Clearinghouse` with no accounting entry and no recovery path.

---

### Finding Description

When a user calls `Endpoint.depositCollateralWithReferral()`, two things happen in sequence:

**Step 1 — Immediate token transfer to Clearinghouse:**

`handleDepositTransfer()` in `EndpointStorage.sol` pulls `amount` tokens from the caller and forwards them directly to `Clearinghouse` in the same transaction. [1](#0-0) 

**Step 2 — Deferred accounting via slow mode queue:**

A `SlowModeTx` is enqueued with the original `amount` hardcoded into the payload. This transaction cannot be executed until `SLOW_MODE_TX_DELAY` (hardcoded to 3 days) has elapsed. [2](#0-1) 

**Step 3 — Accounting credit uses the stale amount:**

When the slow mode transaction is eventually processed, `Clearinghouse.depositCollateral()` credits the user's `SpotEngine` balance using `txn.amount` — the value frozen at deposit time, not the current token balance. [3](#0-2) 

**The gap:** During the 3-day window between Step 1 and Step 3, a rebasing token held by `Clearinghouse` increases in balance. The delta (`rebasedBalance - originalAmount`) has no corresponding accounting entry in `SpotEngine` and no owner. It cannot be withdrawn by any user or recovered by any protocol function, because `withdrawCollateral` debits from a subaccount's recorded balance, and no subaccount was ever credited the delta. [4](#0-3) 

The same structural issue applies to `WithdrawPool`, which transiently holds tokens between `Clearinghouse.handleWithdrawTransfer()` and `BaseWithdrawPool.submitWithdrawal()`. A rebase during that window would leave a delta in `WithdrawPool` with no accounting entry. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any rebasing token listed as a supported collateral product will silently accumulate unaccounted token balance in `Clearinghouse` over time. The rebased increment:

- Is never credited to any subaccount.
- Cannot be withdrawn via `withdrawCollateral` (no balance entry exists).
- Cannot be recovered via `removeLiquidity` on `WithdrawPool` (it sits in `Clearinghouse`, not `WithdrawPool`).
- Causes the protocol's actual token holdings to permanently diverge from its internal `SpotEngine` accounting.

The `manualAssert` mechanism checks internal state consistency (e.g., `insurance`, `spotStates`) but does not compare actual ERC-20 `balanceOf` against `totalDepositsNormalized`, so it will not detect or surface this divergence. [7](#0-6) 

---

### Likelihood Explanation

The trigger requires a rebasing token to be registered as a supported collateral product in `SpotEngine`. The protocol uses a generic `IERC20Base` interface with no explicit exclusion of rebasing tokens. If stETH or any other rebasing token is listed, every deposit during a positive rebase window (which for stETH occurs daily) will leave a small stuck delta. Over time and across many depositors, the stuck amount compounds. No attacker action is required — normal user deposits are sufficient. [8](#0-7) 

---

### Recommendation

- **Preferred:** Explicitly disallow rebasing tokens as collateral products. Add a check in `SpotEngine.addProduct()` or the product registration path that rejects tokens whose `balanceOf` is not share-based.
- **Alternative:** For any rebasing token integration, use the wrapped non-rebasing variant (e.g., `wstETH` instead of `stETH`), which has a stable balance and is safe to record by amount.
- **Partial mitigation:** Replace the fixed-amount slow mode payload with a balance-delta approach: record the Clearinghouse's token balance before and after the transfer, and credit the actual received amount rather than the user-supplied `amount`.

---

### Proof of Concept

1. A rebasing token `RToken` is registered as product `productId=2` in `SpotEngine`.
2. Alice calls `Endpoint.depositCollateralWithReferral(aliceSubaccount, 2, 10e18, "")`.
   - `handleDepositTransfer` moves `10e18` RToken from Alice → Endpoint → Clearinghouse.
   - `SlowModeTx` is queued: `{amount: 10e18, executableAt: now + 3 days}`.
3. 1 day later, RToken rebases +3%. Clearinghouse now holds `10.3e18` RToken.
4. After 3 days, the slow mode tx is executed.
   - `Clearinghouse.depositCollateral` credits Alice with `10e18` (the original amount).
   - `0.3e18` RToken remains in Clearinghouse with no accounting entry.
5. Alice withdraws her full balance of `10e18`. Clearinghouse still holds `0.3e18` RToken.
6. No function exists to recover the `0.3e18`. It is permanently locked. [9](#0-8) [3](#0-2)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
        // copy from submitSlowModeTransaction
        SlowModeConfig memory _slowModeConfig = slowModeConfig;

        slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: abi.encodePacked(
                uint8(TransactionType.DepositCollateral),
                abi.encode(
                    DepositCollateral({
                        sender: subaccount,
                        productId: productId,
                        amount: amount
                    })
                )
            )
        });
        slowModeConfig = _slowModeConfig;
    }

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
