### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Divergent Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` in the block-sync path silently reclassifies an `AllResources` transaction as `L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` field in the hash preimage only for `AllResources`, the same transaction data produces two distinct hash values depending on which side of the protobuf round-trip it is observed on. This breaks the canonicalization invariant that a transaction's hash must be stable across serialization boundaries.

---

### Finding Description

**Step 1 — The hash preimage is variant-dependent.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the fee-fields hash differently for the two `ValidResourceBounds` variants:

```
AllResources  →  poseidon(tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat)
L1Gas         →  poseidon(tip, L1_GAS_concat, L2_GAS_concat)
``` [1](#0-0) 

When `l2_gas = 0` and `l1_data_gas = 0`, the two variants produce **different hashes** even though the numeric field values are identical.

**Step 2 — The protobuf deserializer silently changes the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in the block-sync converter applies a zero-check to decide which variant to reconstruct:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [2](#0-1) 

A transaction that entered the system as `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is silently re-emitted as `L1Gas(X)` after one protobuf round-trip.

**Step 3 — The gateway always hashes as `AllResources`.**

All three RPC transaction types (`RpcInvokeTransactionV3`, `RpcDeclareTransactionV3`, `RpcDeployAccountTransactionV3`) carry `resource_bounds: AllResourceBounds` and are converted to `ValidResourceBounds::AllResources(...)` before hash computation: [3](#0-2) 

The `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` path therefore always produces an `AllResources` hash, including the `L1_DATA_GAS` term even when it is zero. [4](#0-3) 

**Step 4 — The divergence.**

| Stage | `ValidResourceBounds` variant | `L1_DATA_GAS` in hash preimage | Hash value |
|---|---|---|---|
| Gateway (original) | `AllResources` | ✓ included (zero) | `H_A` |
| After protobuf round-trip | `L1Gas` | ✗ absent | `H_B ≠ H_A` |

**Step 5 — The gateway accepts the triggering transaction.**

The stateless validator's resource-bounds check operates on `AllResourceBounds` directly and passes when `l1_gas` is non-zero, regardless of whether `l2_gas` and `l1_data_gas` are zero: [5](#0-4) 

The `min_gas_price` guard (`l2_gas.max_price_per_unit < min_gas_price`) is satisfied when `min_gas_price = 0`, which is the value used in the testing configuration and is a valid operational setting (e.g., during bootstrap when `validate_resource_bounds = false`). [6](#0-5) 

---

### Impact Explanation

Any syncing node that re-derives the transaction hash from the deserialized `InvokeTransactionV3` (e.g., via `validate_transaction_hash` or during receipt/event commitment) will compute `H_B` while the block records `H_A`. This breaks the hash-canonicalization invariant and can cause:

1. **Wrong hash bound to the executable payload** — the `ValidResourceBounds` type changes from `AllResources` to `L1Gas`, altering which fee-enforcement branch (`check_resources_within_bounds`) is taken during execution. An `AllResources` transaction that should check all three gas bounds is instead checked only against the L1-gas bound, potentially allowing fee overdraft on L2 or L1-data gas. [7](#0-6) 

2. **Hash mismatch on sync** — a node that re-computes the hash from the deserialized transaction will reject a legitimately sequenced block, causing a network split between nodes that hold the original transaction and nodes that re-derive it.

---

### Likelihood Explanation

The trigger is a standard V3 invoke/declare/deploy-account transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0`. This is accepted by the gateway whenever `min_gas_price = 0` (the default in the testing configuration) or `validate_resource_bounds = false` (the documented bootstrap setting). No privileged access is required; any unprivileged user can submit such a transaction.

---

### Recommendation

1. **Preserve the variant across serialization.** Add an explicit discriminant field to the protobuf `ResourceBounds` message (e.g., `bool is_all_resources`) so the deserializer can reconstruct the correct variant without relying on a zero-value heuristic.

2. **Alternatively, make the hash invariant to the variant.** Always include the `L1_DATA_GAS` term in `get_tip_resource_bounds_hash`, even for `L1Gas` transactions (using a zero value). This makes the hash stable regardless of which variant is reconstructed.

3. **Add a gateway guard.** Reject transactions where `l2_gas = 0` and `l1_data_gas = 0` when `validate_resource_bounds = true`, so the ambiguous case never enters the sequenced state.

---

### Proof of Concept

```
// 1. Craft a V3 invoke transaction with:
//    l1_gas = { max_amount: 1000, max_price_per_unit: 1 }
//    l2_gas = { max_amount: 0,    max_price_per_unit: 0 }
//    l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }
//
// 2. Submit to the gateway (accepted when min_gas_price=0 or validate_resource_bounds=false).
//    Gateway computes hash H_A using AllResources path (includes L1_DATA_GAS=0 in preimage).
//    Signature is over H_A.
//
// 3. Transaction is included in a block and propagated via P2P protobuf sync.
//
// 4. Receiving node deserializes ResourceBounds:
//    l1_data_gas.is_zero() && l2_gas.is_zero()  →  ValidResourceBounds::L1Gas(l1_gas)
//    (crates/apollo_protobuf/src/converters/transaction.rs:431)
//
// 5. Receiving node re-derives hash using L1Gas path (L1_DATA_GAS absent from preimage).
//    Computed hash H_B ≠ H_A.
//
// 6. validate_transaction_hash(tx, block_number, chain_id, H_A, options)
//    → possible_hashes = [H_B]  (only the re-derived hash)
//    → H_A ∉ possible_hashes  → returns false → block rejected.
//
// Additionally, if the block is accepted without hash re-verification,
// the executable transaction carries ValidResourceBounds::L1Gas instead of AllResources,
// causing check_resources_within_bounds to skip L2 and L1_DATA gas bound enforcement.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

    type Error = ProtobufConversionError;

        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

        let l1_gas: ResourceBounds = l1_gas.try_into()?;

        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
