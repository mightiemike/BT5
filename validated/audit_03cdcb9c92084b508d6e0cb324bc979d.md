### Title
Uninitialized `FeeRates` for New Products Under Non-Default Fee Tiers Causes Zero Taker Fees — (`File: core/contracts/OffchainExchange.sol`)

---

### Summary

`getTierFeeRateX18` returns an uninitialized `FeeRates` struct (both fields zero) for any product added after a tier's `nonDefaultFeeTierMask` bit was set. This causes users assigned to that tier to pay zero taker fees on newly listed products, corrupting protocol fee accounting.

---

### Finding Description

`updateTierFeeRates` populates `feeRates[tier][productId]` for all products that exist at call time when `txn.productId == QUOTE_PRODUCT_ID`. It then permanently sets the tier's bit in `nonDefaultFeeTierMask`:

```solidity
// core/contracts/OffchainExchange.sol:962-990
function updateTierFeeRates(IEndpoint.UpdateTierFeeRates memory txn) external onlyEndpoint {
    if (txn.productId == QUOTE_PRODUCT_ID) {
        // iterates only over currently existing products
        for (uint32 i = 0; i < spotProductIds.length; i++) { ... }
        for (uint32 i = 0; i < perpProductIds.length; i++) { ... }
    }
    nonDefaultFeeTierMask |= uint128(1) << txn.tier;  // bit set permanently
}
```

When a new product is added later, `feeRates[tier][newProductId]` is never written. `getTierFeeRateX18` checks only the global tier bit — not whether the specific `(tier, productId)` entry was ever initialized:

```solidity
// core/contracts/OffchainExchange.sol:933-946
function getTierFeeRateX18(uint32 tier, uint32 productId) public view returns (FeeRates memory) {
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];  // returns {0, 0} for new products
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 });
}
```

The returned `takerRateX18 = 0` propagates into `applyFee`:

```solidity
// core/contracts/OffchainExchange.sol:556-560
int128 keepRateX18 = ONE - feeInfo.feeRate;   // ONE - 0 = ONE
int128 newMeteredQuote = meteredQuote.mul(keepRateX18); // meteredQuote * 1 = meteredQuote
orderInfo.fee = meteredQuote - newMeteredQuote; // = 0
```

The fee is zero. The protocol collects nothing.

---

### Impact Explanation

Any user assigned a non-default fee tier (any tier `T` where `nonDefaultFeeTierMask & (1 << T) != 0`) pays **zero taker fees** on every product listed after the tier's fee rates were last configured. The `market.collectedFees` balance for those products is never incremented for these users, permanently reducing protocol fee revenue. The corrupted state is `market.collectedFees` and the `X_ACCOUNT` balance that would have received those fees via `dumpFees`.

---

### Likelihood Explanation

This is a normal operational sequence: fee tiers are configured once at launch or periodically, and new products are added over time. Any user with a non-default fee tier — including high-volume traders who are commonly assigned custom tiers — automatically benefits from zero fees on every new product without any special action. The trigger requires only that the user submit a valid signed order through the sequencer on a newly listed product, which is the standard trading path.

---

### Recommendation

In `getTierFeeRateX18`, fall back to the default fee schedule when the specific `(tier, productId)` entry has never been written, rather than relying solely on the global tier bit. One approach: track initialization per `(tier, productId)` pair, or fall back to the default rates when both fields are zero:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId) public view returns (FeeRates memory) {
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        FeeRates memory rates = feeRates[tier][productId];
        if (rates.takerRateX18 != 0 || rates.makerRateX18 != 0) {
            return rates;
        }
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 });
}
```

Alternatively, `updateTierFeeRates` should be called again for each new product at listing time, and this should be enforced in `addOrUpdateProduct`.

---

### Proof of Concept

1. Admin calls `updateTierFeeRates({tier: 1, productId: QUOTE_PRODUCT_ID, makerRateX18: -100_000_000_000_000, takerRateX18: 300_000_000_000_000})` — sets rates for all existing products under tier 1, sets bit 1 in `nonDefaultFeeTierMask`.
2. Admin lists a new spot product with `productId = 5` via `addOrUpdateProduct`.
3. `feeRates[1][5]` is never written; it remains `{makerRateX18: 0, takerRateX18: 0}`.
4. User with `feeTiers[user] = 1` submits a signed taker order on product 5 to the sequencer.
5. Sequencer calls `matchOrders` → `applyFee(..., taker=true)` → `getUserFeeRateWithBuilder` → `getTierFeeRateX18(1, 5)`.
6. `nonDefaultFeeTierMask & (1 << 1) != 0` is true → returns `feeRates[1][5]` = `{0, 0}`.
