### Title
Storage Deposits for GC-Evicted Blocks Are Permanently Locked in the Contract — (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` charges callers a NEAR storage deposit proportional to the **net** storage increase after both adding new headers and running GC. When GC evicts old blocks and frees storage, the NEAR tokens previously paid for those blocks are released back to the contract's balance but are never returned to anyone. There is no withdrawal or refund mechanism for freed storage deposits, so they accumulate permanently in the contract.

---

### Finding Description

In `submit_blocks`, the accounting logic is:

```rust
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();          // snapshot before

for header in headers {
    self.submit_block_header(header, ...);            // adds N blocks → storage ↑
}

self.run_mainchain_gc(num_of_headers);               // removes M old blocks → storage ↓

let diff_storage_usage = env::storage_usage()
    .saturating_sub(initial_storage);               // NET change only
let required_deposit = env::storage_byte_cost()
    .saturating_mul(diff_storage_usage.into());     // charge for net increase

let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id()).transfer(refund).into()
} else {
    PromiseOrValue::Value(())
}
``` [1](#0-0) 

The `diff_storage_usage` captures only the **net** storage delta for the current call. When `run_mainchain_gc` removes old blocks, the storage freed by those removals offsets the storage added by new headers. If GC removes as many or more bytes than were added, `saturating_sub` yields zero and the caller pays nothing — which is correct for the current call. [2](#0-1) 

However, the storage freed by GC was **paid for in earlier `submit_blocks` calls**. In NEAR Protocol, freeing storage releases the locked NEAR back into the contract's account balance. The contract has no mechanism to return these freed deposits to the original payers. The `run_mainchain_gc` function, which performs the actual eviction, handles no payment accounting at all: [3](#0-2) 

`remove_block_header` deletes entries from `mainchain_header_to_height` and `headers_pool`, freeing storage bytes, but no deposit is ever returned: [4](#0-3) 

There is no `withdraw`, `claim_storage_refund`, or equivalent function anywhere in the contract. The freed NEAR accumulates in the contract's balance with no recovery path.

---

### Impact Explanation

**Impact: High.** For every block evicted by GC, approximately `storage_byte_cost × bytes_per_ExtendedHeader` NEAR is permanently locked. With the recommended `gc_threshold = 52704` (one year of Bitcoin blocks), the contract reaches steady-state GC after the first year and then evicts one block for every block added — meaning every subsequent `submit_blocks` call permanently locks a portion of the deposit paid by an earlier call. Over the multi-year lifetime of a production deployment, the cumulative locked amount is material.

---

### Likelihood Explanation

**Likelihood: High.** GC is triggered automatically inside every `submit_blocks` call once the stored chain length exceeds `gc_threshold`. This is the normal, intended operating mode of the contract — not an edge case. Additionally, `run_mainchain_gc` is a public, permissionless function callable by any account, meaning GC can be triggered externally at any time without any privileged access. [5](#0-4) 

---

### Recommendation

Track the storage deposit paid per `submit_blocks` call (or per block) and return the proportional freed deposit when GC evicts those blocks. Alternatively, compute the storage delta **before** running GC (i.e., charge for the gross addition of new headers) and separately credit back the freed deposit from GC-evicted blocks to the original depositor or a designated treasury account. A minimal fix is to change the storage measurement to capture only the addition phase:

```rust
let initial_storage = env::storage_usage();
for header in headers {
    self.submit_block_header(header, self.skip_pow_verification);
}
let gross_storage_added = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost()
    .saturating_mul(gross_storage_added.into());

// Run GC after charging — freed deposit is credited to a tracked refund pool
self.run_mainchain_gc(num_of_headers);
```

---

### Proof of Concept

1. Relayer submits batches totalling `gc_threshold + 1` blocks, paying cumulative storage deposits of `D` NEAR across multiple `submit_blocks` calls.
2. On the `(gc_threshold + 1)`-th call, `run_mainchain_gc` evicts the oldest block. The storage for that block is freed; the corresponding deposit (originally paid in step 1) is released into the contract's balance.
3. The current caller receives a refund of only `amount - required_deposit` for the **net** storage change of the current call. The freed deposit from the evicted block is not returned to anyone.
4. Every subsequent `submit_blocks` call repeats step 2–3: one block added, one block evicted, one block's worth of deposit permanently locked.
5. After `K` calls past the GC threshold, `K × (storage_byte_cost × bytes_per_header)` NEAR is permanently locked in the contract with no recovery path.

    #[pause(except(roles(Role::UnrestrictedRunGC)))]
    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
```
