### Title
Linked Signer Can Bypass Owner-Only Guard to Permanently Entrench or Rotate Its Own Authority — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The fast-mode `LinkSigner` transaction in `EndpointTx.sol` is validated with `allowLinkedSigner = true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to update the linked signer mapping for a subaccount. This bypasses the intended guard that only the subaccount owner should control who holds signing authority over their account. A malicious or compromised linked signer can use this to entrench itself or rotate authority to a new attacker-controlled address, permanently locking the subaccount owner out of revocation.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is handled as follows: [1](#0-0) 

The call to `validateSignedTx` passes `allowLinkedSigner = true` as the final argument. This causes `validateCompactSignature` to accept a signature from either the subaccount owner **or** the current linked signer: [2](#0-1) 

The `getLinkedSigner` lookup resolves the current linked signer for the subaccount: [3](#0-2) 

Because the linked signer is accepted as a valid signer for `LinkSigner` transactions, it can sign a new `LinkSigner` payload setting `signedTx.tx.signer` to any address — including itself or a fresh attacker-controlled address — and the contract will write it unconditionally: [4](#0-3) 

By contrast, the slow-mode `LinkSigner` path enforces `validateSender`, which requires the transaction submitter to be the address embedded in the subaccount bytes32 (i.e., the owner), not the linked signer: [5](#0-4) 

The nonce is keyed by the subaccount owner's address, not the linked signer's address: [6](#0-5) 

The current nonce is publicly readable via `getNonce`, so the linked signer can always construct a valid nonce-bearing `LinkSigner` transaction.

---

### Impact Explanation

**Impact: Medium**

A malicious or compromised linked signer can:

1. **Entrench itself**: Sign a `LinkSigner` transaction pointing `signer` back to itself before the owner's revocation is processed, making revocation a persistent race condition.
2. **Rotate to a new attacker address**: Sign a `LinkSigner` transaction pointing `signer` to a fresh attacker-controlled address, silently transferring full signing authority.

Once the linked signer is re-set, it retains the ability to sign withdrawals, trades, and any other transaction type that accepts `allowLinkedSigner = true`. This constitutes unauthorized persistent control over the subaccount's assets and positions.

---

### Likelihood Explanation

**Likelihood: Medium**

The precondition is that a linked signer is malicious or becomes compromised (e.g., a leaked API key, a malicious third-party trading bot, or a compromised off-chain service). This is a realistic operational scenario for any protocol that encourages users to delegate signing to automated systems. Once the precondition is met, the exploit requires only a single signed transaction submitted to the sequencer — no privileged access, no governance capture, no oracle manipulation.

---

### Recommendation

Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the subaccount `bytes32`) should be permitted to update the linked signer mapping. The slow-mode path already enforces this correctly via `validateSender`; the fast-mode path should match that invariant.

```solidity
// In processTransactionImpl, change:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // ← was true; linked signer must NOT be able to update itself
);
```

### Proof of Concept

**Setup:**
- Alice owns subaccount `alice_addr:default`.
- Alice links a third-party trading bot (`botAddress`) as the linked signer.
- Alice later decides to revoke the bot and submits a slow-mode `LinkSigner` transaction setting `signer = address(0)`.

**Attack:**
1. The bot monitors the mempool/slow-mode queue and detects Alice's revocation.
2. The bot reads the current nonce for `alice_addr` via `getNonce(alice_addr)` — say it is `N`.
3. The bot signs a fast-mode `SignedLinkSigner` payload: `sender = alice_addr:default`, `nonce = N`, `signer = botAddress` (itself), signed by `botAddress`.
4. The bot submits this to the sequencer. Because `allowLinkedSigner = true`, the signature from `botAddress` is accepted.
5. The sequencer processes the bot's transaction first (nonce `N` consumed), writing `linkedSigners[alice_addr:default] = botAddress`.
6. Alice's slow-mode revocation now uses nonce `N+1` (or fails if it also used `N`), but the linked signer is already re-entrenched.
7. The bot retains full signing authority and can drain Alice's collateral via signed withdrawal transactions. [1](#0-0) [7](#0-6)

        address recovered = ECDSA.recover(digest, signature.r, signature.vs);
