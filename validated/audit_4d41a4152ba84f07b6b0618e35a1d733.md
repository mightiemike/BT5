### Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the originating swapper, allowing any user to bypass the curated-pool allowlist by routing through `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool sees `msg.sender = router`, so the extension gates on the router's address, not the originating user. A pool admin who allowlists the router to let their approved users access it inadvertently opens the gate to every user on the public router, completely defeating the allowlist.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to every configured extension:

```solidity
_beforeSwap(
    msg.sender,   // ← always the immediate caller of the pool
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks that exact value against the per-pool allowlist:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` inside the extension is the pool; `sender` is whoever called `pool.swap()`. When a user goes through `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput` / `exactOutput`), the router calls `pool.swap(...)` directly:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The pool receives `msg.sender = MetricOmmSimpleRouter`. The extension therefore evaluates `allowedSwapper[pool][MetricOmmSimpleRouter]`, not `allowedSwapper[pool][originalUser]`.

The extension provides no mechanism to recover the original user's identity from `extensionData`; it ignores that parameter entirely. The pool admin therefore faces an impossible choice:

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| Allowlist the router | **Every** user on the public router bypasses the allowlist |

There is no configuration that selectively permits allowlisted users to use the router while blocking non-allowlisted users. A pool admin who allowlists the router — a natural step to give approved users a better UX — silently opens the pool to the entire public.

The `DepositAllowlistExtension` avoids this problem by checking `owner` (the LP-share beneficiary) rather than `sender` (the payer), because the owner is the economically relevant actor for deposits. The `SwapAllowlistExtension` has no equivalent: it checks `sender`, which collapses to the router address on every router-mediated swap.

---

### Impact Explanation

A curated pool protected by `SwapAllowlistExtension` is fully bypassed by any user who routes through `MetricOmmSimpleRouter` once the pool admin allowlists the router. The allowlist's purpose — restricting swaps to KYC'd counterparties, institutional participants, or any other curated set — is defeated. LPs who deposited under the assumption that only approved counterparties would trade against them are exposed to the full public, which can include adversarial MEV bots, front-runners, or simply users the pool was designed to exclude. This is a direct loss of the protection the pool admin paid to configure and LPs relied upon.

---

### Likelihood Explanation

The trigger is a pool admin allowlisting the router — a natural, expected action for any operator who wants their approved users to enjoy the router's UX (slippage limits, multi-hop, deadline checks). The extension gives no warning that doing so opens the gate to everyone. The router is a public, permissionless contract. Once the router is allowlisted, any user can exploit the bypass without any further privileged action.

---

### Recommendation

The extension must gate on the originating user, not the immediate pool caller. Two viable approaches:

1. **Signed identity in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension reads and verifies it (e.g., via a trusted-forwarder pattern or a factory-registered router list).
2. **Router-aware allowlist**: Add a `mapping(address router => bool)` of trusted routers; when `sender` is a trusted router, decode the real user from `extensionData` and check that address instead.

The simplest safe fix is to reject any `sender` that is not itself in the allowlist and document that router-mediated swaps are unsupported on allowlisted pools until a forwarding mechanism is added.

### Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)   // Alice is the approved trader
3. Admin calls setAllowedToSwap(pool, router, true)  // Admin wants Alice to use the router

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <curated pool>,
           recipient: bob,
           ...
       })

5. Router calls pool.swap(bob, ...) with msg.sender = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true   ✓ (step 3)
   Extension does NOT revert.

8. Bob's swap executes on the curated pool.
   The allowlist is fully bypassed.
```
