### Title
`verify_transaction_inclusion` Omits Coinbase Depth-Anchor Check Present in `verify_transaction_inclusion_v2`, Enabling Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a still-callable public contract method that verifies transaction inclusion using only a raw Merkle path check. The correct implementation, `verify_transaction_inclusion_v2`, adds two additional guards: a proof-length equality check and a coinbase Merkle proof verification. Because `verify_transaction_inclusion` omits these guards, an unprivileged NEAR caller can supply a `tx_id` that is an internal Merkle tree node (not a real transaction leaf) and receive a `true` verification result. Consumer contracts that call the v1 entry point are exposed to the 64-byte transaction Merkle proof forgery attack.

---

### Finding Description

`verify_transaction_inclusion` performs exactly one cryptographic check:

```
compute_root_from_merkle_proof(tx_id, tx_index, merkle_proof) == header.block_header.merkle_root
``` [1](#0-0) 

This check is satisfied by **any** node in the Merkle tree — leaf or internal — as long as the caller supplies the correct sibling path. The function does not verify that `tx_id` is a leaf-level node.

`verify_transaction_inclusion_v2` closes this gap with two additional invariants before delegating to v1:

1. **Proof-length equality**: `merkle_proof.len() == coinbase_merkle_proof.len()`. Because the coinbase transaction is always at index 0 (a leaf), its proof length equals the full tree depth. Requiring the target proof to have the same length forces the target to also be at leaf depth.
2. **Coinbase proof verification**: the coinbase Merkle proof is independently verified against the block's `merkle_root`. [2](#0-1) 

`verify_transaction_inclusion` carries a `#[deprecated]` attribute and a `# Warning` doc comment acknowledging the forgery risk, but it remains a fully callable `pub` method gated only by `#[pause]` — no role restriction prevents an unprivileged caller from invoking it directly. [3](#0-2) 

The `ProofArgs` struct accepted by v1 contains no coinbase fields, so the missing check cannot be reconstructed from the call arguments. [4](#0-3) 

---

### Impact Explanation

A consumer contract (atomic swap, bridged-asset release, cross-chain state gate) that calls `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2` can be made to accept a forged proof. The attacker supplies:

- A real mainchain `tx_block_blockhash` (already stored in the contract).
- A `tx_id` equal to the hash of an internal Merkle node in that block.
- A valid sibling path from that internal node to the root.

`verify_transaction_inclusion` returns `true`. The consumer contract believes a Bitcoin transaction with hash `tx_id` was confirmed, and releases funds or updates state accordingly — with no corresponding real transaction on-chain.

The corrupted value is the **proof result** returned to the consumer: a `true` that should be `false`.

---

### Likelihood Explanation

The entry point is unrestricted (any NEAR account, no staking or role requirement). The only prerequisite is knowledge of an internal Merkle node hash for any mainchain block, which is derivable from public block data. Consumer contracts that integrated before `v2` was introduced, or that copied example code referencing the v1 method, are silently vulnerable. The deprecation attribute is a Rust compiler hint, not an on-chain enforcement mechanism.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with a role that no external caller holds. The `#[deprecated]` attribute does not prevent NEAR cross-contract calls. `verify_transaction_inclusion_v2` already delegates to v1 internally after applying the coinbase guards, so removing the public surface of v1 does not break v2.

If backward compatibility must be preserved, add the same two guards from v2 directly into v1:

```rust
// Inside verify_transaction_inclusion, before the Merkle root comparison:
require!(
    merkle_proof.len() == coinbase_merkle_proof.len(),
    "Proof length mismatch"
);
require!(
    compute_root_from_merkle_proof(coinbase_tx_id, 0, &coinbase_merkle_proof)
        == header.block_header.merkle_root,
    "Invalid coinbase proof"
);
```

---

### Proof of Concept

**Setup**: A Bitcoin mainchain block with 4 transactions is stored in the contract. Its Merkle tree has 3 levels. The internal node `N = hash(tx0, tx1)` is at depth 1 from the bottom; its Merkle proof to the root is `[hash(tx2, tx3)]` (length 1).

**Attack**:
```
verify_transaction_inclusion({
    tx_id:             N,                    // internal node, not a real tx
    tx_block_blockhash: <valid mainchain hash>,
    tx_index:          0,                    // even index → left child path
    merkle_proof:      [hash(tx2, tx3)],     // valid 1-element path to root
    confirmations:     1,
})
```

`compute_root_from_merkle_proof(N, 0, [hash(tx2,tx3)])` produces the correct `merkle_root`. The function returns `true`.

**Why v2 rejects it**: The coinbase proof for `tx0` has length 2 (two levels to the root). The forged proof has length 1. The check `merkle_proof.len() == coinbase_merkle_proof.len()` (1 ≠ 2) panics, rejecting the call. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
