### Title
Depleted `RewardsPoolName` permanently blocks position exits — (`x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

All exit paths in `x/tieredrewards` (`TierUndelegate`, `TierRedelegate`, `ExitTierWithDelegation`, `ClearPosition`, `AddToTierPosition`) unconditionally call `claimRewards`, which calls `processEventsAndClaimBonus`. When the shared `RewardsPoolName` module account cannot cover the computed bonus, `sufficientBonusPoolBalance` returns a hard error — not zero — causing the entire exit transaction to revert. Because `BeginBlocker`'s `topUpBaseRewards` drains the same pool every block when base-reward shortfalls exist, a user who has accrued bonus rewards can be permanently blocked from exiting a locked position until governance refills the pool.

---

### Finding Description

**Root cause — `sufficientBonusPoolBalance` returns an error, not zero:** [1](#0-0) 

When `poolBalance < bonus`, the function returns `ErrInsufficientBonusPool`. This error propagates through `processEventsAndClaimBonus`: [2](#0-1) 

…through `claimRewards`: [3](#0-2) 

…and surfaces as a transaction-level failure in every exit message handler:

- `TierUndelegate` — [4](#0-3) 
- `TierRedelegate` — [5](#0-4) 
- `ExitTierWithDelegation` — [6](#0-5) 
- `ClearPosition` — [7](#0-6) 
- `AddToTierPosition` — [8](#0-7) 

**Pool drain vector — `topUpBaseRewards` in `BeginBlocker`:**

Every block, `topUpBaseRewards` transfers from `RewardsPoolName` to the distribution module to cover base-reward shortfalls: [9](#0-8) 

When the pool balance is less than the shortfall, the entire remaining balance is consumed: [10](#0-9) 

After this drain, any user whose accrued bonus exceeds the remaining pool balance will have every exit transaction revert.

**Race condition between concurrent bonus claimants:**

Even without `topUpBaseRewards`, two users with accrued bonuses competing in the same block exhibit the same race: the first claim succeeds and reduces the pool; the second claim fails because `poolBalance < bonus`, and the second user's exit is reverted. Unlike the external report (where the contract silently sets bonus to zero and proceeds), here the entire operation is aborted, leaving the user's delegation locked with no way to exit.

---

### Impact Explanation

A user who has completed their exit lock duration submits `TierUndelegate` or `ExitTierWithDelegation`. If the `RewardsPoolName` pool has been drained — either by `BeginBlocker` or by a concurrent bonus claim — the transaction reverts with `ErrInsufficientBonusPool`. The user's delegation remains locked. There is no alternative exit path that bypasses `claimRewards`. The user is stuck until governance or an external actor refills the pool.

The corrupted invariant: *a position whose exit lock duration has elapsed must be exitable by its owner*. The pool depletion breaks this invariant unconditionally.

---

### Likelihood Explanation

`topUpBaseRewards` runs every block and actively drains `RewardsPoolName` whenever fee revenue falls short of `TargetBaseRewardsRate`. This is a normal, expected operational condition — not an edge case. Any period of low fee revenue combined with a non-zero `TargetBaseRewardsRate` will drain the pool, immediately blocking all exits for users with non-zero accrued bonuses. No attacker action is required; the BeginBlocker itself is the trigger.

---

### Recommendation

Decouple the exit path from mandatory bonus claiming. Two concrete options:

1. **Skip bonus on insufficient pool, do not error.** In `processEventsAndClaimBonus`, replace the hard error with a graceful skip (pay zero bonus, advance the checkpoint) when the pool is insufficient. This mirrors the external report's suggested fix of treating the bonus as optional rather than blocking.

2. **Separate the bonus pool from the base-rewards pool.** Use distinct module accounts for `topUpBaseRewards` and for bonus payouts so that BeginBlocker cannot drain the bonus pool.

---

### Proof of Concept

1. Chain is running with `TargetBaseRewardsRate > 0` and a finite `RewardsPoolName` balance.
2. User A creates a tiered position via `MsgLockTier`, accrues bonus over time.
3. `BeginBlocker` runs for several blocks with low fee revenue; `topUpBaseRewards` drains `RewardsPoolName` to zero.
4. User A triggers exit (`MsgTriggerExitFromTier`) and waits for the exit lock duration to elapse.
5. User A submits `MsgTierUndelegate`. The call chain is:
   - `TierUndelegate` → `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance`
   - `poolBalance (0) < bonus (N)` → `ErrInsufficientBonusPool` returned
   - Transaction reverts.
6. User A's delegation remains locked. Repeated attempts in subsequent blocks continue to fail as long as the pool remains empty. No alternative message exists to exit without claiming bonus.

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
