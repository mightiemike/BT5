### Title
Silent `i32`-to-`u64` sign-extension in `calculate_reward` corrupts epoch token minting and can panic the epoch manager — (File: `chain/epoch-manager/src/reward_calculator.rs`)

### Summary
`calculate_reward` casts `Rational32::numer()` and `Rational32::denom()` (both `&i32`) directly to `u64` via `as u64` before feeding them into `U256` arithmetic. In Rust, `as u64` on a negative `i32` is a **sign-extending** cast: `(-1i32) as u64 == 18446744073709551615`. No validation anywhere in the call chain rejects a negative numerator or denominator for `max_inflation_rate`, `protocol_reward_rate`, `online_min_threshold`, or `online_max_threshold`. A protocol-upgrade boundary that introduces any of these as a negative rational silently corrupts every subsequent epoch's reward calculation across all nodes simultaneously.

### Finding Description

In `calculate_reward`:

```rust
// chain/epoch-manager/src/reward_calculator.rs lines 69-77
let epoch_total_reward = Balance::from_yoctonear(
    (U256::from(*max_inflation_rate.numer() as u64)   // ← i32 → u64, silent sign extension
        * U256::from(total_supply.as_yoctonear())
        * U256::from(epoch_duration)
        / (U256::from(self.num_seconds_per_year)
            * U256::from(*max_inflation_rate.denom() as u64)  // ← same
            * U256::from(NUM_NS_IN_SECOND)))
    .as_u128(),
);
```

`Rational32::numer()` returns `&i32`. The `as u64` cast is **not** a checked or absolute-value cast; it reinterprets the two's-complement bit pattern as an unsigned 64-bit integer. For any negative `i32` value `n`, `n as u64 == u64::MAX - (-n as u64) + 1`, producing a value near `u64::MAX`.

The same pattern is repeated for `protocol_reward_rate` (lines 80–81) and for all four online-threshold components (lines 104–107, 120–123):

```rust
let online_min_numer =
    U256::from(*online_thresholds.online_min_threshold.numer() as u64);
let online_min_denom =
    U256::from(*online_thresholds.online_min_threshold.denom() as u64);
// ...
let online_max_numer =
    U256::from(*online_thresholds.online_max_threshold.numer() as u64);
let online_max_denom =
    U256::from(*online_thresholds.online_max_threshold.denom() as u64);
let online_numer =
    online_max_numer * online_min_denom - online_min_numer * online_max_denom;
```

If `online_min_threshold` has a negative numerator, `online_min_numer` becomes near-`u64::MAX` as a `U256`, making `online_min_numer * online_max_denom` astronomically large. The subsequent U256 subtraction `online_max_numer * online_min_denom - online_min_numer * online_max_denom` **underflows**, panicking in debug builds and wrapping to a near-`U256::MAX` value in release builds.

The `ParameterValue::Rational` conversion in `core/parameters/src/parameter_table.rs` (lines 135–148) performs **no sign validation**:

```rust
&ParameterValue::Rational { numerator, denominator } => {
    Ok(Rational32::new(numerator, denominator))
}
```

`Rational32::new` accepts any `i32` pair. There is no downstream guard in `EpochConfig` construction, `RewardCalculator::new`, or `calculate_reward` itself that asserts positivity.

### Impact Explanation

**Scenario A — negative `max_inflation_rate` numerator:** `epoch_total_reward` is computed as `(~u64::MAX * total_supply * epoch_duration) / (...)`. With a mainnet total supply of ~1 billion NEAR and a typical epoch duration, the numerator overflows `U256` capacity and the `.as_u128()` truncation produces a near-`u128::MAX` yoctoNEAR value. This is credited to the protocol treasury and validators, minting an unbounded number of tokens and breaking the total-supply invariant permanently for every subsequent epoch.

**Scenario B — negative `online_min_threshold` numerator:** The U256 subtraction `online_numer = online_max_numer * online_min_denom - online_min_numer * online_max_denom` underflows. In a release build this wraps to a near-`U256::MAX` value, making `uptime_denum` near-`U256::MAX`, and the final `U512` division produces zero reward for every validator. In a debug build it panics, halting epoch finalization and stalling the chain.

### Likelihood Explanation

`max_inflation_rate` and `online_thresholds` are fields of `EpochConfig`, which is updated at every protocol-version upgrade boundary via the `EpochConfigStore`. The `ParameterValue::Rational` deserialization path accepts any `i32` numerator without a positivity check. A protocol upgrade that introduces a negative rational — whether through a misconfigured parameter-table YAML entry or a deliberate proposal — activates the bug simultaneously on every node that applies the upgrade, with no per-node escape hatch. There is no existing validation layer between the parameter table and `calculate_reward` that would catch or reject a negative value.

### Recommendation

1. In `calculate_reward`, replace every `*foo.numer() as u64` and `*foo.denom() as u64` with an explicit non-negative assertion, e.g.:
   ```rust
   u64::try_from(*max_inflation_rate.numer())
       .expect("max_inflation_rate numerator must be non-negative")
   ```
2. Add a sign-validation step in the `TryFrom<&ParameterValue> for Rational32` implementation in `core/parameters/src/parameter_table.rs` that returns an error if `numerator < 0` or `denominator <= 0` for rate parameters.
3. Add an `EpochConfig::validate()` method that asserts all rate rationals are in `[0, 1]` and is called at config construction time and at every protocol-upgrade boundary.
