### Title
Storage Deposits Paid by Relayers Are Permanently Lost When GC Removes Their Block Headers - (`contract/src/lib.rs`)

### Summary

When `run_mainchain_gc` removes old mainchain block headers from storage, the NEAR storage deposits originally paid by relayers for those headers are never returned. The freed storage balance accumulates permanently in the contract account with no withdrawal mechanism, causing relayers to permanently lose the NEAR they deposited.

### Finding Description

`submit_blocks` is `#[payable]` and requires callers to attach a NEAR deposit proportional to the net storage increase their call causes. The accounting is done by measuring storage before and after the entire call (including the embedded GC run):

```rust
let amount = env::attached_deposit();
let initial_storage = env::storage_usage();

for header in headers { self.submit_block_header(...); }
self.run_mainchain_gc(num_of_headers);

let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());
require!(amount >= required_deposit, ...);
let refund = amount.saturating_sub(required_deposit);
``` [1](#0-0) 

When GC removes old headers, `env::storage_usage()` drops. Because `saturating_sub` is used, if GC frees more bytes than the new headers consume, `diff_storage_usage` becomes `0` and the current caller pays nothing and gets their full deposit back. The freed storage balance (previously paid by earlier relayers) is silently absorbed into the contract account.

`run_mainchain_gc` removes headers from both `mainchain_height_to_header` and `headers_pool` via `remove_block_header`, freeing the storage bytes those entries occupied:

```rust
self.remove_block_header(blockhash);
self.mainchain_height_to_header.remove(&height);
``` [2](#0-1) 

```rust
fn remove_block_header(&mut self, header_block_hash: &H256) {
    self.mainchain_header_to_height.remove(header_block_hash);
    self.headers_pool.remove(header_block_hash);
}
``` [3](#0-2) 

There is no per-depositor tracking, no refund logic in `run_mainchain_gc`, and no `withdraw` function anywhere in the contract. The freed NEAR is permanently locked in the contract account.

The same removal-without-refund pattern also occurs during chain reorganization in `reorg_chain`, where demoted mainchain blocks are deleted:

```rust
self.remove_block_header(&current_main_chain_blockhash);
self.mainchain_height_to_header.remove(&height);
``` [4](#0-3) 

### Impact Explanation

Relayers who paid storage deposits for block headers that are subsequently removed by GC or reorg permanently lose those deposits. The freed NEAR accumulates in the contract account with no mechanism to recover it. With a default `gc_threshold` of 52,704 blocks and continuous operation, the cumulative loss across all relayers grows unboundedly over time. The freed NEAR is not accessible to any party — not the original depositors, not the DAO, not any admin role.

### Likelihood Explanation

GC runs automatically on every `submit_blocks` call (with `batch_size = num_of_headers`). Additionally, `run_mainchain_gc` is a public, permissionless function callable by any NEAR account. Any unprivileged caller can trigger GC at any time, accelerating the removal of headers and the permanent capture of prior depositors' NEAR. Reorgs are also a normal part of Bitcoin chain operation and trigger the same removal path. [5](#0-4) 

### Recommendation

Track the storage deposit paid per `submit_blocks` call (e.g., keyed by block height range or caller) and return the proportional freed deposit to the original depositor when GC removes those headers. Alternatively, implement a contract-level withdrawal function that allows the DAO to redistribute accumulated freed storage balance back to relayers, or adopt a storage-staking model where freed storage is immediately refunded to the predecessor of the GC-triggering call.

### Proof of Concept

1. Relayer A calls `submit_blocks([h1, h2, ..., h100])` with a deposit of `D` NEAR covering 100 headers' storage cost. The contract stores all 100 headers.
2. Later, the mainchain grows past `gc_threshold`. Any caller invokes `run_mainchain_gc(100)`. The 100 headers submitted by Relayer A are removed from `headers_pool` and `mainchain_height_to_header`.
3. The freed storage bytes (worth `D` NEAR) are released back into the contract account's available balance.
4. Relayer A has no way to reclaim `D` NEAR. There is no refund path, no per-depositor record, and no withdrawal function.
5. The next `submit_blocks` caller benefits: their `diff_storage_usage` is reduced by the freed bytes (via `saturating_sub`), so they pay less or nothing — but Relayer A's `D` NEAR is gone permanently. [6](#0-5)
