### Title
`DepositAllowlistExtension` gates on position `owner` instead of the actual depositor `sender`, allowing any non-allowlisted address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." However, its `beforeAddLiquidity` hook silently ignores the `sender` argument (the actual caller who pays tokens) and instead checks the `owner` argument (the position recipient). Because `MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address that need not equal `msg.sender`, any non-allowlisted address can deposit tokens into an allowlisted address's position, fully bypassing the admin-configured allowlist.

---

### Finding Description

**Actor binding in the pool:**

`MetricOmmPool.addLiquidity` passes two distinct addresses to the extension:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

- `msg.sender` — the actual caller who will pay tokens via the modify-liquidity callback (the real "depositor")
- `owner` — an arbitrary address supplied by the caller as the position beneficiary [1](#0-0) 

**Wrong actor checked in the extension:**

`DepositAllowlistExtension.beforeAddLiquidity` discards `sender` (first parameter, written as `address,`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
``` [2](#0-1) 

**Bypass path:**

A non-allowlisted address `Bob` calls:

```
pool.addLiquidity(owner = Alice /*allowlisted*/, salt, deltas, callbackData, extensionData)
```

The extension receives `(sender=Bob, owner=Alice)`, ignores `Bob`, checks `Alice` → passes. Bob's tokens are transferred via the modify-liquidity callback; Alice's position is credited. Bob has interacted with the pool as a liquidity provider despite being explicitly excluded by the pool admin.

**`ExtensionCalling` confirms the argument order:** [3](#0-2) 

The first argument is `sender` (the caller), the second is `owner` (the beneficiary). The extension ignores the first and gates on the second.

---

### Impact Explanation

The pool admin deploys a curated pool with `DepositAllowlistExtension` to restrict which addresses may provide liquidity (e.g., KYC/compliance, protocol-controlled LP cohort). The allowlist invariant is broken: any address, regardless of allowlist status, can deposit tokens into the pool by nominating an allowlisted address as `owner`. The restricted address interacts with the pool as a real economic actor (paying tokens, affecting bin balances and fee accrual), which the admin-configured boundary was designed to prevent. This is an admin-boundary break where an unprivileged path bypasses a factory/pool-level access control.

---

### Likelihood Explanation

The trigger requires only a standard `addLiquidity` call with `owner` set to any allowlisted address. No special permissions, flash loans, or oracle manipulation are needed. Any address that knows an allowlisted LP's address can execute this. The only cost to the attacker is the tokens deposited (which accrue to the nominated `owner`'s position), so the action is economically neutral-to-negative for the attacker but still constitutes a policy violation.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor/caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This aligns the check with the documented invariant ("Gates `addLiquidity` by depositor address") and with the analogous `SwapAllowlistExtension`, which correctly checks `sender`. [4](#0-3)

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  Alice (0xAlice) → allowedDepositor[pool][Alice] = true
  Bob   (0xBob)   → allowedDepositor[pool][Bob]   = false (not allowlisted)

Attack:
  vm.prank(Bob);
  pool.addLiquidity(
      owner        = Alice,   // allowlisted — extension checks this and passes
      salt         = 0,
      deltas       = <valid liquidity delta>,
      callbackData = "",      // Bob transfers tokens in the callback
      extensionData = ""
  );

Result:
  - Extension checks allowedDepositor[pool][Alice] → true → no revert
  - Bob's tokens are pulled via metricOmmModifyLiquidityCallback
  - Alice's position is credited with the deposited shares
  - Bob (non-allowlisted) has successfully deposited into the pool,
    bypassing the admin-configured deposit allowlist
``` [2](#0-1) [5](#0-4)
