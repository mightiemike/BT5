Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (the only way to permit router-mediated swaps), every unprivileged user can bypass the individual-user allowlist by routing through the router, because the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][end_user]`.

## Finding Description
The call chain is confirmed in production code:

1. `MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

2. `ExtensionCalling._beforeSwap` forwards `sender` unchanged to every registered extension: [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

4. `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly with no mechanism to forward the originating user: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. There is no existing guard that recovers the original user identity from `extensionData` or any other channel. The extension has no awareness of whether the immediate caller is a trusted router acting on behalf of a user.

This creates an irreconcilable dilemma: if the router is not allowlisted, even allowlisted users cannot swap through it; if the router is allowlisted (the only viable configuration for router-mediated swaps), any unprivileged user can bypass the allowlist entirely.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. An unauthorized user can trade against LP positions that were intended to be protected, constituting a broken core pool access-control mechanism and an admin-boundary break where an unprivileged path circumvents a configured guard. The allowlist policy — the sole purpose of the extension — is rendered ineffective.

## Likelihood Explanation
Any pool that deploys `SwapAllowlistExtension` and also wants to support the official `MetricOmmSimpleRouter` must allowlist the router. Once the router is allowlisted, the bypass is trivially reachable by any user with no special privileges, capital requirements, or flash loans. The only precondition is that the pool admin has allowlisted the router, which is the natural and expected configuration for any pool intending to support router-mediated swaps.

## Recommendation
The extension must gate the economic actor, not the immediate caller. Options:

1. **Router forwards user identity in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes it when `sender` is a known trusted router. Requires coordinated changes to both the router and the extension.
2. **Dedicated router-aware extension variant**: Provide a separate extension that reads the originating user from `extensionData` when the immediate caller is a trusted router address, falling back to `sender` for direct pool calls.
3. **Documentation and incompatibility guard**: At minimum, document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and add an on-chain check that reverts if `sender` is a known router address, forcing direct pool interaction only.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Admin allowlists alice: allowedSwapper[pool][alice] = true
  - Admin allowlists the router: allowedSwapper[pool][router] = true
    (required so alice can use the router)

Attack (charlie, not allowlisted):
  1. charlie calls router.exactInputSingle({pool: pool, ...})
  2. router calls pool.swap(...) — pool sees msg.sender = router
  3. pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  5. Swap executes for charlie despite charlie not being allowlisted

Result:
  - charlie successfully swaps on a pool restricted to alice only
  - The allowlist policy is completely bypassed for any user routing through the router
```
