### Title
Caller-Supplied `hash_value` Accepted Without Recomputation in Python-to-Rust L1 Handler Conversion — (`crates/native_blockifier/src/py_l1_handler.rs`)

---

### Summary

`py_l1_handler()` constructs an executable `L1HandlerTransaction` by reading `hash_value` directly from the Python object without recomputing it from the transaction fields and the sequencer's canonical `chain_id`. The blockifier then uses this unverified hash as the authoritative `transaction_hash` in `get_execution_info()` syscall results, receipts, and events. This is the direct Sequencer analog of the Atlas `reconcile()` bug: a caller-supplied parameter is used in place of a canonically derived value, breaking the invariant that `tx_hash == calculate_transaction_hash(chain_id, version)`.

---

### Finding Description

In `crates/native_blockifier/src/py_l1_handler.rs`, the function `py_l1_handler` constructs an `L1HandlerTransaction` by reading `hash_value` directly from the Python object:

```rust
pub fn py_l1_handler(py_tx: &PyAny) -> NativeBlockifierResult<L1HandlerTransaction> {
    let py_l1_handler_tx: PyL1HandlerTransaction = py_tx.extract()?;
    let tx = starknet_api::transaction::L1HandlerTransaction::try_from(py_l1_handler_tx)?;
    let tx_hash = TransactionHash(py_attr::<PyFelt>(py_tx, "hash_value")?.0); // ← caller-supplied
    let paid_fee_on_l1 = Fee(py_attr::<u128>(py_tx, "paid_fee_on_l1")?);
    Ok(L1HandlerTransaction { tx, tx_hash, paid_fee_on_l1 })
}
``` [1](#0-0) 

The canonical construction path, `L1HandlerTransaction::create()`, recomputes the hash from the transaction fields and the node's `chain_id`:

```rust
pub fn create(raw_tx: ..., chain_id: &ChainId, paid_fee_on_l1: Fee) -> Result<...> {
    let tx_hash = raw_tx.calculate_transaction_hash(chain_id, &raw_tx.version)?;
    Ok(Self { tx: raw_tx, tx_hash, paid_fee_on_l1 })
}
``` [2](#0-1) 

`py_l1_handler` bypasses `create()` entirely and directly constructs the struct with the Python-supplied `hash_value`. No post-construction verification is performed in the blockifier execution path.

The `tx_hash` field is then propagated into `TransactionInfo` via `create_tx_info()`:

```rust
impl TransactionInfoCreator for L1HandlerTransaction {
    fn create_tx_info(&self) -> TransactionInfo {
        TransactionInfo::Deprecated(DeprecatedTransactionInfo {
            common_fields: CommonAccountFields {
                transaction_hash: self.tx_hash, // ← unverified Python-supplied value
                ...
            },
            ...
        })
    }
}
``` [3](#0-2) 

This `transaction_hash` is what the `get_execution_info()` syscall returns to executing contracts, and what is recorded in receipts and events.

The `echonet/os_input_builder.py` confirms the Python side passes `hash_value` from central objects directly:

```python
def _l1_handler_to_executable(central_tx: JsonObject) -> JsonObject:
    return {
        "tx": inner,
        "tx_hash": central_tx["hash_value"],  # ← from central, not recomputed
        ...
    }
``` [4](#0-3) 

The OS Cairo code does recompute and assert the hash (`%{ AssertTransactionHash %}`), but only in the proving path — not in the blockifier execution path used for sequencing. [5](#0-4) 

---

### Impact Explanation

If the Python sequencer passes a `hash_value` that diverges from the canonical Rust-computed hash (e.g., due to a chain_id mismatch, a Python-side hash computation bug, or a version gate difference), the blockifier will:

1. Return the wrong `transaction_hash` from the `get_execution_info()` syscall to any contract that reads it.
2. Record the wrong hash in transaction receipts and events.
3. Cause the L1 provider's state tracking (which keys on `tx_hash`) to operate on a hash that does not match what the OS/prover would compute, creating a divergence between the sequencer's execution record and the provable state.

This matches: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

---

### Likelihood Explanation

The trigger requires the Python-to-Rust boundary to supply a `hash_value` that differs from the canonical computation. This can occur due to:

- A Python-side hash computation bug (e.g., wrong chain_id encoding, wrong version).
- A misconfiguration where the Python sequencer and Rust blockifier are configured with different chain IDs.
- The `echonet/os_input_builder.py` path forwarding a `hash_value` from central objects that was computed under a different version or chain context.

The Python sequencer is a trusted component, so this is not directly triggerable by an external user. However, the absence of any verification at the conversion boundary means any divergence between Python and Rust hash computation silently produces wrong execution results with no error. Likelihood is **Medium** given the complexity of the Python/Rust hash computation boundary and the existence of multiple hash version paths (`L1HandlerVersions::AsInvoke`, `V0Deprecated`, `V0`).

---

### Recommendation

Replace the direct struct construction in `py_l1_handler` with `L1HandlerTransaction::create()`, which canonically recomputes the hash from the transaction fields and the node's `chain_id`:

```rust
pub fn py_l1_handler(py_tx: &PyAny) -> NativeBlockifierResult<L1HandlerTransaction> {
    let py_l1_handler_tx: PyL1HandlerTransaction = py_tx.extract()?;
    let tx = starknet_api::transaction::L1HandlerTransaction::try_from(py_l1_handler_tx)?;
    let paid_fee_on_l1 = Fee(py_attr::<u128>(py_tx, "paid_fee_on_l1")?);
    // Use create() to canonically compute tx_hash from fields + chain_id
    Ok(L1HandlerTransaction::create(tx, &chain_id, paid_fee_on_l1)?)
}
```

If the Python-supplied `hash_value` must be preserved for compatibility, add an assertion that it matches the recomputed value and return an error on mismatch, analogous to how `validate_transaction_hash` works for other transaction types. [6](#0-5) 

---

### Proof of Concept

1. Configure the Python sequencer with `chain_id = SN_MAIN` but pass an L1 handler transaction with `hash_value` computed for `chain_id = SN_SEPOLIA`.
2. Call `py_l1_handler(py_tx)` — it accepts the mismatched hash without error.
3. The blockifier executes the L1 handler. Any contract calling `get_execution_info()` receives the Sepolia hash as `transaction_hash`.
4. The receipt records the Sepolia hash. The L1 provider's state tracking marks the Sepolia hash as committed.
5. The OS/prover recomputes the hash using the node's actual `chain_id = SN_MAIN` and gets a different value, creating a divergence between the sequencer's committed state and the provable execution trace.
