### Title
Bonus Checkpoints Advanced Without Payment During Insufficient-Pool Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, the error is deliberately swallowed to prevent a chain halt. However, by the time that error is returned, `processEventsAndClaimBonus` has already advanced the position's bonus checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and decremented validator-event reference counts in the store. The caller then persists the modified position via `setPositionWithState`. The result is that the position's accounting state is permanently advanced past events for which no bonus was ever paid — a direct analog to the OracleAdapterV4 pattern of updating a safety-anchor value during a flagged/error condition.

---

### Finding Description

`processEventsAndClaimBonus` (`claim_rewards.go`) takes a `*types.PositionState` pointer and mutates it in-place as it walks validator events: [1](#0-0) 

For every event in the loop it:
1. Decrements the event's reference count in the store (`decrementEventRefCount`, line 196).
2. Advances `pos.LastEventSeq` (`pos.UpdateLastEventSeq`, line 193).

After the loop it applies the accrual checkpoint and persists the bonded state: [2](#0-1) 

Only *then* does it check whether the pool can cover the computed bonus: [3](#0-2) 

When the pool is insufficient, `ErrInsufficientBonusPool` is returned — but all checkpoint mutations and store-level reference-count decrements have already occurred.

Back in `slashRedelegationPosition`, this error is explicitly swallowed: [4](#0-3) 

For a **partial** slash the function then persists the already-mutated `pos` (with advanced checkpoints, no bonus paid): [5](#0-4) 

For a **full** slash `ClearBonusCheckpoints()` is called before persisting, so that path is unaffected. The partial-slash path is the vulnerable one.

---

### Impact Explanation

After the slash completes:

- `pos.LastEventSeq` has been advanced past all events that were processed.
- `pos.LastBonusAccrual` and `pos.LastKnownBonded` reflect the post-event state.
- The validator-event reference counts for those events have been decremented; once they reach zero the events are eligible for garbage collection.

On the next call to `processEventsAndClaimBonus` for this position (e.g., a user-initiated `ClaimRewards`), `getValidatorEventsSince` will return only events *after* the advanced `LastEventSeq`. The events that were consumed during the failed slash are gone. The position holder permanently loses all bonus rewards that accrued up to the slash block, with no recourse.

The corrupted value is: **`pos.LastEventSeq` / `pos.LastBonusAccrual` / `pos.LastKnownBonded`** — the exact accounting anchors that gate future bonus calculations.

---

### Likelihood Explanation

Two independently realistic conditions must coincide:

1. A validator that has tiered-reward positions is slashed for a redelegation (`BeforeRedelegationSlashed` hook fires). Validator slashing is a normal, unprivileged protocol event triggered by evidence submission or double-sign detection.
2. The `tieredrewards` bonus pool (`types.RewardsPoolName`) is depleted or underfunded at that block. Pool depletion is an expected operational state explicitly handled by the codebase (the `ErrInsufficientBonusPool` path exists precisely because it is anticipated).

No privileged access, key compromise, or social engineering is required. Any delegator whose position is on a slashed validator during a low-pool period is affected without any action on their part.

---

### Recommendation

- Move `applyBonusAccrualCheckpoint`, `pos.UpdateLastKnownBonded`, and all `decrementEventRefCount` calls to *after* the `sufficientBonusPoolBalance` check succeeds, so checkpoints are only advanced when the bonus is actually paid.
- Alternatively, when `ErrInsufficientBonusPool` is caught in `slashRedelegationPosition`, reset `pos` to its pre-call snapshot before calling `setPositionWithState`, preserving the original checkpoints.
- Consider not swallowing the error silently; instead, skip the bonus-claim step entirely (do not call `processEventsAndClaimBonus`) when the pool is known to be empty, leaving checkpoints untouched for a future claim.

---

### Proof of Concept

1. Delegator D creates a tiered-reward position on validator V and redelegates to validator V2.
2. The `tieredrewards` bonus pool is drained (e.g., by normal reward payouts over time).
3. V2 commits a double-sign; the staking module calls `SlashRedelegation`, which fires `BeforeRedelegationSlashed`.
4. `slashRedelegationPosition` calls `processEventsAndClaimBonus(&pos)`.
5. Inside `processEventsAndClaimBonus`: all pending validator events are walked, `pos.LastEventSeq` is advanced, `decrementEventRefCount` is called for each event, `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` update `pos`.
6. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`; the function returns without paying bonus.
7. Back in `slashRedelegationPosition`, the error is swallowed (lines 56–63 of `slash.go`).
8. Because `sharesToUnbond < pos.Delegation.Shares` (partial slash), `setPositionWithState(ctx, pos, nil)` persists the mutated position.
9. D later calls `ClaimRewards`; `getValidatorEventsSince` returns no events before the advanced `LastEventSeq`. D receives zero bonus for the entire pre-slash accrual period. The pool refill that happens afterward cannot retroactively recover the lost rewards.
