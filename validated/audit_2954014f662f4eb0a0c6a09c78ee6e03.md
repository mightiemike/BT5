Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, not the end user. This creates two symmetric failures: if the admin allowlists the router to enable router-based swaps, every non-allowlisted user bypasses the gate; if the admin does not allowlist the router, every individually-allowlisted user is silently blocked from the standard periphery path.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` to the pool: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — in every case the router is the direct caller of `pool.swap()`: [5](#0-4) 

The structural mismatch is confirmed by contrast with `DepositAllowlistExtension.beforeAddLiquidity`, which correctly checks `owner` (the real position owner), not `sender` (the adder contract): [6](#0-5) 

`SwapAllowlistExtension` has no equivalent "real user" argument — `recipient` is the output destination, not the payer/initiator — so the mismatch is structural and cannot be resolved without a protocol-level change.

## Impact Explanation

**Bypass path (primary):** A pool configured with `SwapAllowlistExtension` to restrict swaps to KYC'd or whitelisted counterparties is completely ineffective once the admin allowlists the router. Any non-allowlisted address calls `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point); the pool sees `sender = router`; the allowlist check passes; the swap executes. The allowlist invariant is broken for all router-mediated swaps — which is the standard user path. This constitutes unauthorized value extraction from a restricted pool.

**Blocking path (secondary):** If the admin does not allowlist the router, allowlisted users who call through the router receive `NotAllowedToSwap` even though they are individually permitted. Their only recourse is to call `pool.swap()` directly, bypassing the router's slippage protection, deadline enforcement, and multi-hop routing — or they cannot swap at all. This constitutes broken core swap functionality for allowlisted users.

Both outcomes are fund-impacting and meet Sherlock contest thresholds.

## Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool that deploys `SwapAllowlistExtension` and also expects users to use the router will encounter one of the two failure modes immediately upon deployment. The bypass requires no special privilege — any EOA can call the router. The blocking affects every allowlisted user who uses the router. Both conditions are reachable in normal protocol operation without any malicious setup.

## Recommendation

`SwapAllowlistExtension.beforeSwap` should gate the actual end user, not the direct caller of `pool.swap()`. Two options:

1. **Decode the real user from `extensionData`**: Require the router to encode `msg.sender` (the end user) into `extensionData` and have the extension decode and check that address. This requires a coordinated change in the router.

2. **Check `recipient` as a proxy**: If the pool's swap design guarantees that `recipient` is always the end user's address, gate on `recipient` instead of `sender`. This is only safe if `recipient` cannot be set to an arbitrary allowlisted address by an attacker.

The cleanest fix is option 1, mirroring how `DepositAllowlistExtension` correctly gates `owner` (the position owner) rather than `sender` (the adder contract).

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Admin also allowlists the router so users can swap via the standard periphery path.
swapAllowlist.setAllowedToSwap(address(pool), address(router), true);
// allowedUser is NOT separately allowlisted — admin assumes router allowlist covers them.

// Non-allowlisted attacker routes through the router:
vm.prank(attacker); // attacker not in allowedSwapper
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1_000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: address(token0),
    extensionData: "",
    deadline: block.timestamp
}));
// Swap succeeds — attacker bypassed the allowlist.
// pool.swap() sees msg.sender = router; router is allowlisted; check passes.

// Separately: allowedUser (individually allowlisted) tries to use the router
// when the router itself is NOT allowlisted:
swapAllowlist.setAllowedToSwap(address(pool), allowedUser, true);
vm.prank(allowedUser);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
router.exactInputSingle(...); // reverts — pool sees sender = router, not allowedUser
```

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
