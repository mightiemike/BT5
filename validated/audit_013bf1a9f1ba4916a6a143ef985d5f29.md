### Title
`op_ash` and `op_lsh` Bypass Cost Budget Enforcement Before Performing Expensive Bignum Shift — (`File: src/more_ops.rs`)

### Summary

Both `op_ash` (arithmetic shift) and `op_lsh` (logical shift) in `src/more_ops.rs` accept a `max_cost` parameter but never call `check_cost` before performing the expensive bignum shift and heap allocation. Cost is computed only after the work is done and the result is already allocated. This allows an attacker-controlled CLVM program to force validator nodes to perform bignum shift operations and allocate memory that exceeds the remaining cost budget, consuming real CPU and heap resources before the budget check fires in the next evaluation loop iteration.

### Finding Description

In `src/more_ops.rs`, both shift operators declare their `max_cost` parameter with a leading underscore, signaling it is intentionally unused:

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
    let [n0, n1] = get_args::<2>(a, input, "ash")?;
    let (i0, l0) = int_atom(a, n0, "ash")?;
    let a1 = i32_atom(a, n1, "ash")?;
    if !(-65535..=65535).contains(&a1) {
        return Err(EvalErr::ShiftTooLarge(n1));
    }
    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };  // expensive bignum op
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;                                      // heap allocation
    let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
    Ok(malloc_cost(a, cost, r))                                    // cost never validated
}
```

`op_lsh` (line 982) is identical in structure — `_max_cost` is unused, no `check_cost` call exists.

Contrast this with every other arithmetic operator in the same file:

- `op_div` calls `check_cost(cost, max_cost)?` at line 672 before the division.
- `op_multiply` calls `check_cost(cost, max_cost)?` at lines 617, 625, and 645 before each multiplication step.
- `op_modpow` calls `check_cost(cost, max_cost)?` at lines 1261 and 1264 before the modular exponentiation.

The evaluation engine in `src/run_program.rs` passes the remaining budget as `effective_max_cost - cost` to each operator via `apply_op` (line 523). For `op_ash`/`op_lsh`, this value is silently discarded. The returned cost is added to the running total and checked only at the top of the next loop iteration (line 514). By that point, the bignum shift and heap allocation have already executed.

### Impact Explanation

An attacker submits a CLVM program containing `(ash <large_atom> 32767)` or `(lsh <large_atom> 32767)`. With a 1-byte input `0x01` and a shift of 32767 bits, the result is a ~4096-byte bignum. The true cost of this single operation is:

```
ASHIFT_BASE_COST + (1 + 4096) * ASHIFT_COST_PER_BYTE
= 596 + 4097 * 3 = 12,887
```

plus malloc cost for the 4096-byte result atom. If the remaining cost budget at the point of execution is, say, 100, the operation still executes in full — allocating the bignum, writing it into the CLVM heap — before the budget overrun is detected. A program can chain multiple such calls (each within a sub-expression evaluated before the budget check fires) to amplify the resource consumption per rejected transaction. In mempool mode (`MEMPOOL_MODE` flags), where validators must evaluate every submitted transaction to determine validity, this allows an attacker to force disproportionate CPU and heap work per rejected spend bundle, degrading validator throughput.

### Likelihood Explanation

The entry path requires only the ability to submit a CLVM program to a Chia full node — no keys, no privileged access. The `ash` (opcode 22) and `lsh` (opcode 23) operators are part of the default operator set dispatched unconditionally by `ChiaDialect::op` (lines 212–213 of `src/chia_dialect.rs`). The shift-range guard (`-65535..=65535`) is the only input validation present, and it still permits shifts large enough to produce multi-kilobyte results. Any attacker who can broadcast transactions to the Chia network can trigger this path repeatedly.

### Recommendation

Add a `check_cost` call in both `op_ash` and `op_lsh` before performing the shift and heap allocation. Because the output size `l1` is not known until after the shift, a conservative pre-check using only the input-side cost should be inserted first, followed by a second check after `l1` is computed:

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, max_cost: Cost, _flags: ClvmFlags) -> Response {
    let [n0, n1] = get_args::<2>(a, input, "ash")?;
    let (i0, l0) = int_atom(a, n0, "ash")?;
    let a1 = i32_atom(a, n1, "ash")?;
    if !(-65535..=65535).contains(&a1) {
        return Err(EvalErr::ShiftTooLarge(n1));
    }
    // Pre-check with input-side cost before performing the expensive shift
    let pre_cost = ASHIFT_BASE_COST + (l0 as Cost) * ASHIFT_COST_PER_BYTE;
    check_cost(pre_cost, max_cost)?;

    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;
    let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
    check_cost(cost, max_cost)?;
    Ok(malloc_cost(a, cost, r))
}
```

Apply the same fix to `op_lsh`.

### Proof of Concept

**Trigger:** Submit the following CLVM program to a Chia full node with a low `max_cost`:

```
(ash (q . 0x01) (q . 32767))
```

**Expected (correct) behavior:** The operation should be rejected immediately when the pre-computed cost exceeds the remaining budget.

**Actual behavior:** The bignum `1 << 32767` (~4096 bytes) is computed and allocated in the CLVM heap. Only after `op_ash` returns does the engine detect the budget overrun and raise `CostExceeded`. The expensive work has already been performed.
