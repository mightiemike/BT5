### Title
Linked Signer Can Overwrite Its Own Delegation Entry, Hijacking Subaccount Authorization ŌĆö (File: `core/contracts/EndpointTx.sol`)

### Summary

In `EndpointTx.processTransactionImpl`, the fast-path `LinkSigner` handler passes `allowLinkedSigner = true` to `validateSignedTx`. This means the **current linked signer** ŌĆö not only the subaccount owner ŌĆö can produce a valid `SignedLinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any arbitrary address. The subaccount owner's consent is not required.

### Finding Description

The fast-path `LinkSigner` handler in `processTransactionImpl` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ŌåÉ allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which forwards `getLinkedSigner(sender)` to the verifier:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts a signature from **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Because the linked signer's signature satisfies this check, it can craft and sign a `SignedLinkSigner` payload where `signedTx.tx.signer` is any address ŌĆö including an attacker-controlled one ŌĆö and the write to `linkedSigners[signedTx.tx.sender]` will succeed.

By contrast, the **slow-mode** `LinkSigner` path correctly enforces ownership:

```solidity
validateSender(txn.sender, sender);   // msg.sender must be the subaccount owner
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast path has no equivalent ownership check, creating an asymmetry.

### Impact Explanation

`linkedSigners` is the authorization registry for subaccount signing. Once overwritten, the new linked signer address is accepted by `validateSignature` for **all** fast-path transaction types that pass `allowLinkedSigner = true`, including `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, and `BurnNlp`. [5](#0-4) 

A malicious linked signer can therefore:

1. Replace `linkedSigners[victim_subaccount]` with an attacker-controlled address.
2. Use that address to sign `WithdrawCollateral` / `WithdrawCollateralV2` transactions, draining the victim's collateral.

The victim can recover only via the slow-mode `LinkSigner` path, which carries a `SLOW_MODE_TX_DELAY` (hardcoded 3 days). During that window the attacker retains full signing authority. [6](#0-5) 

### Likelihood Explanation

Any address that has ever been granted linked-signer status for a subaccount is a potential attacker. The nonce required for the `SignedLinkSigner` payload is publicly readable via `Endpoint.getNonce(address)`. [7](#0-6) 

The sequencer processes all structurally valid signed transactions; no sequencer compromise is needed ŌĆö the on-chain signature check itself passes. The attack requires only: (a) possession of a linked-signer private key, and (b) knowledge of the current nonce.

### Recommendation

Change `allowLinkedSigner` to `false` for the fast-path `LinkSigner` handler, mirroring the slow-mode path's ownership enforcement:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may re-link
);
``` [8](#0-7) 

This ensures that `linkedSigners` entries can only be written by the address that owns the subaccount, matching the invariant enforced in the slow-mode path.

### Proof of Concept

1. Alice owns subaccount `alice_sub` and has set `linkedSigners[alice_sub] = bob`.
2. Bob (the linked signer) reads `nonce = endpoint.getNonce(address(bytes20(alice_sub)))`.
3. Bob constructs `SignedLinkSigner { tx: { sender: alice_sub, signer: attacker, nonce: nonce }, signature: bob_sig }`.
4. Bob submits this to the sequencer's off-chain API.
5. The sequencer calls `submitTransactionsChecked` ŌåÆ `processTransaction` ŌåÆ `processTransactionImpl`.
6. `validateSignedTx(..., true)` recovers `bob` from the signature; `bob == getLinkedSigner(alice_sub)` ŌåÆ passes.
7. `linkedSigners[alice_sub]` is overwritten with `attacker`.
8. Attacker signs `WithdrawCollateral { sender: alice_sub, productId: 0, amount: MAX, nonce: nonce+1 }` and submits it.
9. Alice's collateral is withdrawn to the attacker before Alice can submit a slow-mode `LinkSigner` to recover. [1](#0-0) [9](#0-8)
