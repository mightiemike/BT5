### Title
Stale `PromiseYield` trie keys survive `DeleteAccount`, enabling phantom callback execution on recreated accounts — (File: `core/store/src/utils/mod.rs`)

---

### Summary

`remove_account` does not remove `PromiseYieldReceipt`, `PromiseYieldStatus`, `YieldIdToDataId`, or `DataIdToYieldId` trie keys. When an account with a pending yield is deleted and a new account is created with the same `account_id`, the stale yield state causes the old callback receipt to be unexpectedly executed on the new account when the global timeout queue fires — a direct nearcore analog of the GMX phantom-decrease invariant violation.

---

### Finding Description

`remove_account` in `core/store/src/utils/mod.rs` removes `Account`, `ContractCode`, `AccessKey`, gas-key nonces, and `ContractData` trie keys: [1](#0-0) 

It does **not** remove any of the following keys, all of which are keyed by `receiver_id` (the account being deleted):

| Trie key | Written by |
|---|---|
| `TrieKey::PromiseYieldReceipt { receiver_id, data_id }` | `set_promise_yield_receipt` |
| `TrieKey::PromiseYieldStatus { receiver_id, data_id }` | `set_promise_yield_status` |
| `TrieKey::YieldIdToDataId { receiver_id, yield_id }` | `set_yield_id_mapping` |
| `TrieKey::DataIdToYieldId { receiver_id, data_id }` | `set_yield_id_mapping` | [2](#0-1) 

These keys are written when a contract calls `promise_yield_create_with_id` (protocol feature `YieldWithId`): [3](#0-2) 

A fifth artifact, `TrieKey::PromiseYieldTimeout { index: N }`, is a **global** shard-level queue entry that records `{ account_id, data_id, expires_at }`. It is also never removed by `remove_account` because it is not keyed by account. [4](#0-3) 

The normal cleanup path for all five artifacts only runs when a `PromiseResume` receipt is processed: [5](#0-4) 

If the account is deleted before the resume or timeout arrives, none of that cleanup ever runs.

---

### Impact Explanation

**Vector 1 — Phantom callback execution on the new account.**  
When the global timeout queue fires at `expires_at`, `resolve_promise_yield_timeouts` checks whether `TrieKey::PromiseYieldReceipt { receiver_id: "alice.near", data_id: X }` still exists. Because `remove_account` left it behind, the check succeeds and a `PromiseResume` receipt is emitted for the new `alice.near`. `process_receipt` then finds the stale `PromiseYieldReceipt`, removes it, and executes the old callback function call on the new account — which the new owner never created and never consented to. [6](#0-5) 

**Vector 2 — Permanent yield-id DoS on the new account.**  
`create_promise_yield_receipt_with_id` checks `has_yield_id_mapping` before creating a new yield. The stale `YieldIdToDataId` entry causes the check to return `true`, so `promise_yield_create_with_id` returns `u64::MAX` (the duplicate sentinel) for any call using the same `yield_id`. The new account is permanently blocked from using that yield-id until the stale mapping is eventually cleaned up by the phantom callback execution. [7](#0-6) 

**Vector 3 — Unauthorized `promise_yield_resume` succeeds.**  
`submit_promise_resume_data` accepts a resume if either `has_promise_yield_receipt` or `has_promise_yield_status` is true. The stale `PromiseYieldStatus = Yielded` entry satisfies the second condition, so any caller can invoke `promise_yield_resume` with the old `data_id` and have it accepted as a valid resume on the new account. [8](#0-7) 

---

### Likelihood Explanation

- Account deletion requires `storage_usage ≤ MAX_ACCOUNT_DELETION_STORAGE_USAGE` (10 KB). The four PromiseYield trie keys are each tens of bytes; they do not prevent deletion.
- Sub-account creation is permissionless: any holder of `near` can create `alice.near`.
- An attacker can deliberately "poison" an account name: create `alice.near`, call `promise_yield_create_with_id`, delete `alice.near`, and leave the stale state. A victim who later acquires `alice.near` inherits the phantom callback.
- The `YieldWithId` feature (`ProtocolFeature::YieldWithId`) is a nightly/upcoming feature, but the `PromiseYieldReceipt` and `PromiseYieldStatus` leaks apply to the base `promise_yield_create` path as well (protocol version 67+, stable).

---

### Recommendation

`remove_account` must iterate and remove all PromiseYield-related trie keys for the account being deleted, mirroring the existing iteration over `ContractData` keys:

- `TrieKey::PromiseYieldReceipt { receiver_id, data_id }` — iterate via the `PromiseYieldReceipt` prefix
- `TrieKey::PromiseYieldStatus { receiver_id, data_id }` — iterate via the `PromiseYieldStatus` prefix
- `TrieKey::YieldIdToDataId { receiver_id, yield_id }` — iterate via the `YieldIdToDataId` prefix
- `TrieKey::DataIdToYieldId { receiver_id, data_id }` — iterate via the `DataIdToYieldId` prefix

Additionally, `resolve_promise_yield_timeouts` should verify that the target account still exists before emitting a `PromiseResume` receipt, as a defense-in-depth measure.

### Proof of Concept

```
Block B1:
  alice.near calls promise_yield_create_with_id(yield_id=[0u8;32], callback="on_resume")
  → trie gains:
      PromiseYieldReceipt { receiver_id="alice.near", data_id=X } = <callback receipt>
      PromiseYieldStatus  { receiver_id="alice.near", data_id=X } = Yielded
      YieldIdToDataId     { receiver_id="alice.near", yield_id=[0;32] } = X
      DataIdToYieldId     { receiver_id="alice.near", data_id=X } = [0;32]
      PromiseYieldTimeout { index=N } = { account_id="alice.near", data_id=X, expires_at=B1+200 }

Block B2:
  alice.near sends DeleteAccountAction (storage_usage is small; deletion succeeds)
  → remove_account removes: Account, ContractCode, AccessKeys, ContractData
  → remove_account does NOT remove: the five keys above

Block B3:
  Victim creates alice.near via CreateAccountAction
  → new account exists; stale PromiseYield keys still present in trie

Block B1+200:
  resolve_promise_yield_timeouts fires for index=N:
    contains_key(PromiseYieldReceipt { "alice.near", X }) → true  (stale)
    emits PromiseResume { receiver_id="alice.near", data_id=X, data=None }

Block B1+201:
  process_receipt handles PromiseResume:
    get_promise_yield_receipt("alice.near", X) → Some(<old callback receipt>)  (stale)
    executes old callback "on_resume" on the new alice.near account
    new account owner never created this yield; callback executes unexpectedly
```

The exact divergent trie values are the four Borsh-serialized entries listed above, which survive `DeleteAccountAction` because `remove_account` in `core/store/src/utils/mod.rs` has no code path that touches `PromiseYieldReceipt`, `PromiseYieldStatus`, `YieldIdToDataId`, or `DataIdToYieldId` keys. /// Removes account, code and all access keys and gas keys associated to it.
pub fn remove_account(
    state_update: &mut TrieUpdate,
    account_id: &AccountId,
) -> Result<RemoveAccountResult, StorageError> {
    state_update.remove(TrieKey::Account { account_id: account_id.clone() });
    state_update.remove(TrieKey::ContractCode { account_id: account_id.clone() });

    let mut gas_key_nonce_count: usize = 0;
    let mut gas_key_nonce_total_key_bytes: usize = 0;

    // Removing access keys and gas key nonces
    let lock = state_update.trie().lock_for_iter();
    let mut keys_to_remove: Vec<TrieKey> = Vec::new();
    for raw_key in state_update
        .locked_iter(&trie_key_parsers::get_raw_prefix_for_access_keys(account_id), &lock)?
    {
        let raw_key = raw_key?;
        let key_handle = trie_key_parsers::parse_key_handle_from_access_key_key(
            &raw_key, account_id,
        )
        .map_err(|_e| {
            StorageError::StorageInconsistentState(
                "Can't parse key handle from raw key for AccessKey".to_string(),
            )
        })?;
        let nonce_index =
            trie_key_parsers::parse_nonce_index_from_gas_key_key(&raw_key, account_id, &key_handle)
                .map_err(|_e| {
                    StorageError::StorageInconsistentState(
                        "Can't parse nonce index from raw key for AccessKey".to_string(),
                    )
                })?;
        if let Some(index) = nonce_index {
            gas_key_nonce_count += 1;
            gas_key_nonce_total_key_bytes += raw_key.len();
            keys_to_remove.push(TrieKey::gas_key_nonce(
                account_id.clone(),
                key_handle.clone(),
                index,
            ));
        } else {
            keys_to_remove.push(TrieKey::access_key(account_id.clone(), key_handle.clone()));
        }
    }
    drop(lock);

    for trie_key in keys_to_remove {
        state_update.remove(trie_key);
    }

    // Removing contract data
    let lock = state_update.trie().lock_for_iter();
    let data_keys = state_update
        .locked_iter(&trie_key_parsers::get_raw_prefix_for_contract_data(account_id, &[]), &lock)?
        .map(|raw_key| {
            trie_key_parsers::parse_data_key_from_contract_data_key(&raw_key?, account_id)
                .map_err(|_e| {
                    StorageError::StorageInconsistentState(
                        "Can't parse data key from raw key for ContractData".to_string(),
                    )
                })
                .map(Vec::from)
        })
        .collect::<Result<Vec<_>, _>>()?;
    drop(lock);

    for key in data_keys {
        state_update.remove(TrieKey::ContractData { account_id: account_id.clone(), key });
    }
    Ok(RemoveAccountResult { gas_key_nonce_count, gas_key_nonce_total_key_bytes })

                &mut self.trie_update,
                &self.account_id,
                data_id,
                PromiseYieldStatus::ResumeInitiated,
            );
            return Ok(true);
        }

        Ok(false)
