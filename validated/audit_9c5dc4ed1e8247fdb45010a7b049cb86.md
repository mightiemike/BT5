Audit Report

## Title
`DepositAllowlistExtension` checks position `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool" and uses a mapping named `allowedDepositor`. However, its `beforeAddLiquidity` hook ignores the `sender` argument (the actual `msg.sender` of the pool's `addLiquidity` call and the token payer) and instead checks only `owner` (the position beneficiary). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` allows any caller to supply an arbitrary `owner` with no restriction beyond `owner != address(0)`, any unprivileged address can bypass the deposit allowlist on any curated pool by naming an already-allowed address as `owner`.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` forwards both values to every configured extension:

```solidity
// ExtensionCalling.sol lines 95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but declares it as an unnamed, unused `address`, then checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-38
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
```

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(address pool, address owner, ...)` accepts any caller-supplied `owner` with `_validateOwner` only rejecting `address(0)`:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

Exploit path:
1. Pool is configured with `DepositAllowlistExtension` as `beforeAddLiquidity` hook; `allowedDepositor[pool][alice] = true`; Bob is not in the allowlist.
2. Bob calls `liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`.
3. Inside `_addLiquidity`, the adder calls `pool.addLiquidity(alice, ...)` with `msg.sender = liquidityAdder`.
4. Pool calls `_beforeAddLiquidity(liquidityAdder, alice, ...)`.
5. Extension receives `sender = liquidityAdder`, `owner = alice`; checks `allowedDepositor[pool][alice] == true` → no revert.
6. Bob's tokens are pulled via callback; alice receives the LP position.
7. Bob has deposited into a pool he is not authorized to enter.

The wrong value is the extension's access-control decision: it returns `beforeAddLiquidity.selector` (allow) when it should revert with `NotAllowedToDeposit`, because it evaluates `allowedDepositor[pool][owner]` instead of `allowedDepositor[pool][sender]`.

## Impact Explanation
The pool admin's deposit allowlist is completely nullified. Any unprivileged address can supply liquidity to a curated pool by naming any already-allowlisted address as `owner`. The LP share distribution is altered without authorization; the allowlisted `owner` receives an unsolicited position they did not initiate. If the pool carries specific risk parameters or regulatory constraints tied to LP identity, those constraints are silently violated. This is a direct admin-boundary break: an unprivileged caller bypasses a pool-admin-configured access control that is the sole on-chain mechanism for restricting depositors.

## Likelihood Explanation
The attack requires only that at least one address is in the allowlist — a necessary precondition for the pool to be usable at all. The attacker needs no special role, no privileged key, and no cooperation from the allowlisted address. `MetricOmmPoolLiquidityAdder` is a supported periphery contract callable by any address. The attack is repeatable at any time a curated pool with this extension is live.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual token payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended policy is to gate both the payer and the position owner, both checks should be applied. The mapping name `allowedDepositor` and the NatSpec "Gates `addLiquidity` by depositor address" both indicate the payer is the intended subject of the guard.
