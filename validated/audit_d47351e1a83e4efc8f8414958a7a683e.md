### Title
Unchecked `u64` Multiplication Overflow in `op_unknown` Cost Accounting Allows Cost-Limit Bypass — (File: `src/more_ops.rs`)

---

### Summary

`op_unknown` in `src/more_ops.rs` performs a `u64` multiplication (`cost *= cost_multiplier + 1`) **after** the pre-multiplication cost has already passed `check_cost`. In Rust release builds, integer overflow wraps silently. An attacker can craft an unknown opcode whose arguments push the pre-multiplication cost to a specific value (e.g., `2^32`) and whose `cost_multiplier` is `u32::MAX`, causing the product to wrap to `0` modulo `2^64`. The subsequent guard `if cost > u32::MAX` passes, and the function returns `Ok(Reduction(0, ...))` — reporting zero cost for what should be an enormously expensive operation.

---

### Finding Description

In `op_unknown` (`src/more_ops.rs`), the cost computation proceeds as follows:

1. A base cost is computed from the opcode's `cost_function` field and its arguments.
2. `check_cost(cost, max_cost)?` is called — this verifies the pre-multiplication cost is within budget.
3. `cost *= cost_multiplier + 1` is executed — **no overflow check**.
4. `if cost > u32::MAX as u64 { Err(...) }` — this is the only post-multiplication guard. [1](#0-0) 

`cost` is `u64` (`Cost = u64`). [2](#0-1) 

`cost_multiplier` is derived from `u32_from_u8`, so it is at most `u32::MAX = 4_294_967_295`, making `cost_multiplier + 1` at most `2^32 = 4_294_967_296`. [3](#0-2) 

Chia's production `max_cost` is `11_000_000_000`, which is greater than `2^32`. Therefore, `cost` (after passing `check_cost`) can legally be `4_294_967_296` (= `2^32`). When multiplied by `cost_multiplier + 1 = 2^32`:

```
cost = 2^32 * 2^32 = 2^64 ≡ 0 (mod 2^64)
```

In Rust release builds, this wraps to `0`. The guard `0 > u32::MAX` is `false`, so the function returns `Ok(Reduction(0, allocator.nil()))` — **zero cost**.

---

### Impact Explanation

The returned `Reduction` cost is added directly to the running total in `run_program`: [4](#0-3) 

A zero (or near-zero) cost returned from `op_unknown` means the caller's cost accumulator is not incremented, allowing the program to execute what should be a budget-exhausting operation for free. This enables:

- **Cost-limit bypass**: Programs that should exceed `max_cost` complete successfully.
- **Consensus divergence**: Debug builds panic on overflow; release builds wrap silently. Nodes running different build profiles reach different outcomes for the same CLVM program — a direct consensus split.
- **DoS of full nodes**: An attacker can submit transactions containing programs that perform unbounded computation at near-zero declared cost, exhausting node CPU without being rate-limited by the cost meter.

---

### Likelihood Explanation

The attack is fully attacker-controlled. The opcode bytes and argument list are part of the CLVM program, which is embedded in a Chia transaction. Any node running in lenient/consensus mode (which calls `op_unknown` for unrecognized opcodes) is exposed. The `cost_multiplier` field is directly encoded in the opcode atom bytes, and the pre-multiplication cost is tunable via the number and size of arguments. No special privileges or configuration are required. [5](#0-4) 

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and place the overflow guard **before** returning:

```rust
// Replace:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    return Err(EvalErr::Invalid(o));
}
```

This ensures that any product exceeding `u64::MAX` is caught as an error rather than silently wrapping to a small value. [1](#0-0) 

---

### Proof of Concept

Craft an unknown opcode atom with:
- Last byte = `0x40` → `cost_function = 1` (arith-like), lower 6 bits ignored
- Preceding 4 bytes = `0xFF 0xFF 0xFF 0xFF` → `cost_multiplier = u32::MAX = 4_294_967_295`, so `cost_multiplier + 1 = 2^32`

Provide enough atom arguments (each of size ~1 byte) to accumulate:
```
cost = ARITH_BASE_COST + N * ARITH_COST_PER_ARG + total_bytes * ARITH_COST_PER_BYTE = 2^32
```
i.e., `99 + N*320 + total_bytes*3 = 4_294_967_296`.

With `N ≈ 13_421_772` single-byte arguments: `cost ≈ 2^32`.

After `check_cost(2^32, 11_000_000_000)` passes:
```
cost *= 2^32   →   2^64 mod 2^64 = 0
```
`0 > u32::MAX` is `false` → `Ok(Reduction(0, NIL))`.

The program's total cost is not incremented, bypassing the cost limit entirely. [6](#0-5)

pub fn op_unknown(
    allocator: &mut Allocator,
    o: NodePtr,
    mut args: NodePtr,
    max_cost: Cost,
) -> Response {
    // unknown opcode in lenient mode
    // unknown ops are reserved if they start with 0xffff
    // otherwise, unknown ops are no-ops, but they have costs. The cost is computed
    // like this:

    // byte index (reverse):
    // | 4 | 3 | 2 | 1 | 0          |
    // +---+---+---+---+------------+
    // | multiplier    |XX | XXXXXX |
    // +---+---+---+---+---+--------+
    //  ^               ^    ^
    //  |               |    + 6 bits ignored when computing cost
    // cost_multiplier  |
    // (up to 4 bytes)  + 2 bits
    //                    cost_function

    // 1 is always added to the multiplier before using it to multiply the cost, this
    // is since cost may not be 0.

    // cost_function is 2 bits and defines how cost is computed based on arguments:
    // 0: constant, cost is 1 * (multiplier + 1)
    // 1: computed like operator add, multiplied by (multiplier + 1)
    // 2: computed like operator mul, multiplied by (multiplier + 1)
    // 3: computed like operator concat, multiplied by (multiplier + 1)

    // this means that unknown ops where cost_function is 1, 2, or 3, may still be
    // fatal errors if the arguments passed are not atoms.

    let op_atom = allocator.atom(o);
    let op = op_atom.as_ref();

    if op.is_empty() || (op.len() >= 2 && op[0] == 0xff && op[1] == 0xff) {
        Err(EvalErr::Reserved(o))?;
    }

    let cost_function = (op[op.len() - 1] & 0b11000000) >> 6;
    let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
        Some(v) => v as u64,
        None => {
            return Err(EvalErr::Invalid(o))?;
        }
    };
