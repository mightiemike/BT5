### Title
GC-Induced Walk-Back Panic Causes Permanent Submission Lockout on Litecoin Testnet — (`contract/src/litecoin.rs`)

### Summary

On Litecoin testnet (`pow_allow_min_difficulty_blocks = true`), `get_next_work_required` contains a walk-back loop that traverses `headers_pool` via `get_prev_header` to find the last non-min-difficulty anchor block. `run_mainchain_gc` removes old blocks from `headers_pool` without any awareness of this walk-back dependency. Once GC prunes the anchor block, every subsequent `submit_blocks` call for a min-difficulty block panics with `"PrevBlockNotFound"`, permanently locking out block submission.

---

### Finding Description

The walk-back loop in `get_next_work_required` (`contract/src/litecoin.rs` lines 65–71):

```rust
let mut current_block_header = prev_block_header.clone();
while current_block_header.block_header.bits == config.proof_of_work_limit_bits
    && current_block_header.block_height % config.difficulty_adjustment_interval != 0
{
    current_block_header =
        blocks_getter.get_prev_header(&current_block_header.block_header);
}
```

terminates only when it finds a block with `bits != proof_of_work_limit_bits` OR a block at a retarget boundary (`height % 2016 == 0`). It walks back through `headers_pool` using `prev_block_hash` links. [1](#0-0) 

`get_prev_header` panics unconditionally if the target block is absent from `headers_pool`:

```rust
fn get_prev_header(&self, current_header: &LightHeader) -> ExtendedHeader {
    self.headers_pool
        .get(&current_header.prev_block_hash)
        .unwrap_or_else(|| env::panic_str("PrevBlockNotFound"))
}
``` [2](#0-1) 

`run_mainchain_gc` removes blocks from `headers_pool` via `remove_block_header`, which calls `self.headers_pool.remove(header_block_hash)`: [3](#0-2) 

GC is triggered automatically on every `submit_blocks` call with `batch_size = num_of_headers`: [4](#0-3) 

There is no guard ensuring that the walk-back anchor (the last non-min-difficulty block, or the nearest retarget-boundary block) is retained in `headers_pool` after GC runs.

**Concrete trigger sequence** (Litecoin testnet, `difficulty_adjustment_interval = 2016`):

1. Initialize with genesis at height 2015 (satisfies the Litecoin init constraint `(height+1) % 2016 == 0`) with normal difficulty bits. [5](#0-4) 
2. Submit 20 min-difficulty blocks (heights 2016–2035). Note height 2016 is a retarget boundary (`2016 % 2016 == 0`) but also has `bits == proof_of_work_limit_bits`.
3. With `gc_threshold = 5`, GC prunes heights 2015–2030 from `headers_pool` (21 stored − 5 threshold = 16 removed). Both the genesis anchor (height 2015, normal bits) and the retarget boundary block (height 2016) are deleted.
4. Submit block at height 2036. Walk-back starts at height 2035, iterates through 2034, 2033, 2032, 2031 — all min-difficulty, all `height % 2016 ≠ 0`. On the next iteration it calls `get_prev_header` on height 2031 to fetch height 2030, which is no longer in `headers_pool` → **panic: "PrevBlockNotFound"**.

Litecoin testnet config confirms `pow_allow_min_difficulty_blocks: true`: [6](#0-5) 

---

### Impact Explanation

Every `submit_blocks` call for any new min-difficulty block panics after GC prunes the walk-back anchor. Since testnet naturally produces long runs of min-difficulty blocks (slow blocks trigger the min-difficulty rule), and GC runs automatically on every `submit_blocks` call, this state is reached in normal operation. Once reached, the contract cannot accept any further min-difficulty blocks — a permanent submission lockout — until the contract is upgraded or redeployed.

---

### Likelihood Explanation

On Litecoin testnet, min-difficulty blocks are common whenever block times exceed `2 * pow_target_spacing` (300 seconds). A run of min-difficulty blocks longer than `gc_threshold` is routine. GC fires automatically on every `submit_blocks` call. No attacker action is required; this is a natural operational failure mode.

---

### Recommendation

The GC must preserve the walk-back anchor. Two options:

1. **Retain the nearest retarget-boundary block**: When computing the GC lower bound, ensure `mainchain_initial_blockhash` is never set to a block whose walk-back anchor (the nearest `height % 2016 == 0` block, or the nearest non-min-difficulty block) has been pruned.

2. **Cache the anchor**: Store the hash of the current walk-back anchor as a separate contract field and exclude it from GC deletion, similar to how `mainchain_initial_blockhash` is tracked.
