### Title
Proof-Verification Forgery via Publicly Callable Deprecated `verify_transaction_inclusion` Accepting Internal Merkle Tree Nodes as `tx_id` — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable by any unprivileged NEAR account. It accepts fully attacker-controlled inputs — including `tx_id` — and passes them directly into `compute_root_from_merkle_proof` without validating that `tx_id` is a leaf node (a real transaction hash) rather than an internal Merkle tree node. An attacker can supply an internal node hash as `tx_id` with a crafted sibling proof, causing the function to return `true` for a transaction that does not exist. Any downstream NEAR contract consuming this result is deceived into accepting a forged inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` and `#[pause]`, but carries no `#[private]` attribute. It is therefore reachable by any unprivileged NEAR caller. [1](#0-0) 

The function's own doc-comment acknowledges the flaw explicitly: [2](#0-1) 

The entire proof check reduces to a single comparison: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` treats its first argument as an opaque 32-byte value and hashes it upward through the tree without any constraint that it must be a leaf: [4](#0-3) 

Because internal Merkle nodes are also 32-byte SHA-256 digests, an attacker can substitute any internal node for `tx_id`, supply the correct sibling path, and reproduce the stored `merkle_root` exactly. The function returns `true`.

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a separate coinbase proof that anchors the tree at a known leaf position: [5](#0-4) 

However, the deprecated entry point was never gated with `#[private]`, so the fix is incomplete: the vulnerable path remains open in parallel.

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a privileged action (e.g., releasing bridged assets, minting tokens, or settling a payment) receives `true` for a fabricated transaction. The attacker does not need to produce a real Bitcoin transaction; they only need knowledge of any mainchain block's Merkle tree structure, which is public. The corrupted value is the boolean proof result returned to the caller — a direct analog to the `_swap` report's corrupted authorization state.

---

### Likelihood Explanation

- No privileged role is required; any NEAR account can call the function.
- All inputs needed for the attack (block Merkle tree, internal node hashes, sibling paths) are publicly available from any Bitcoin full node.
- The attacker only needs to pick a confirmed mainchain block already stored in the contract's `headers_pool`, compute its Merkle tree offline, and submit a single NEAR transaction.
- The function is not removed, only deprecated, so integrators who have not yet migrated remain permanently exposed.

---

### Recommendation

Apply `#[private]` to `verify_transaction_inclusion` immediately, or remove the function body and replace it with `env::panic_str("removed: use verify_transaction_inclusion_v2")`. Deprecation annotations are compile-time Rust warnings only; they have no effect on NEAR's runtime dispatch and do not prevent external callers from invoking the method.

---

### Proof of Concept

Given a mainchain block `B` with four transactions `[T0, T1, T2, T3]`:

```
Level 0 (leaves): T0,        T1,        T2,        T3
Level 1:          N01=H(T0,T1),          N23=H(T2,T3)
Level 2 (root):   R = H(N01, N23)
```

1. Compute `N01 = double_sha256(T0 || T1)` — a real internal node, not a transaction.
2. Call `verify_transaction_inclusion` with:
   - `tx_id          = N01`
   - `tx_block_blockhash = B`
   - `tx_index       = 0`   (even position → left child)
   - `merkle_proof   = [N23]`
   - `confirmations  = 1`
3. Inside `compute_root_from_merkle_proof`:
   - `current_position = 0` (even) → `compute_hash(N01, N23)` = `R`
4. `R == header.block_header.merkle_root` → function returns **`true`**.

`N01` is not a Bitcoin transaction. No such transaction exists. The contract has certified its inclusion. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
