### Title
Swap Allowlist Bypassed via Router: `sender` Identity Mismatch Lets Any User Swap on Restricted Pools — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` to the pool, so `sender` = router address. If the pool admin allowlists the router to enable router-mediated swaps, every user — including those the admin explicitly excluded — can bypass the allowlist by routing through the public router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the `msg.sender` of the `pool.swap()` call:

```solidity
// SwapAllowlistExtension.sol:31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The pool passes `msg.sender` as `sender` to the extension:

```solidity
// ExtensionCalling.sol:160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, zeroForOne, amountSpecified, priceLimitX64, ...)
    )
);
```

where `sender` = `msg.sender` of `pool.swap()`.

When a user calls `MetricOmmSimpleRouter.exactInput()`, the router calls `pool.swap()` directly. The pool's `msg.sender` is the router, so `sender` passed to the extension is the **router address**, not the end user.

**Attack path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., a whitelist of institutional traders).
2. To allow those whitelisted users to also use the router, the admin must allowlist the router address: `setAllowedToSwap(pool, routerAddress, true)`.
3. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInput()` → router calls `pool.swap()` → extension checks `allowedSwapper[pool][router]` → passes, because the router is allowlisted.
4. The disallowed user successfully swaps against the restricted pool.

Alternatively, if the admin does not allowlist the router, then even individually-allowlisted users cannot use the router — breaking expected UX.

---

### Impact Explanation

The swap allowlist is the primary access-control mechanism for restricting who can trade against a pool. A bypass allows unauthorized users to:
- Extract value from LP positions in pools intended for private/institutional use
- Arbitrage against pools whose LP terms assume a restricted counterparty set
- Undermine the pool admin's security model entirely

This is a direct admin-boundary break: an unprivileged path (the public `MetricOmmSimpleRouter`) bypasses a configured guard, causing LP funds to be exposed to unauthorized counterparties.

---

### Likelihood Explanation

- The `MetricOmmSimpleRouter` is a public, permissionless contract.
- Any user who discovers the allowlist restriction can trivially route through the router.
- The pool admin has no way to prevent this without also blocking all router-mediated swaps for legitimate users.
- The scenario where a pool admin allowlists the router (to support router UX) while also maintaining a user-level allowlist is the natural operational configuration, making this reachable in normal deployment.

---

### Recommendation

The extension should check the **end user's identity**, not the intermediary's. Two options:

1. **Pass the original user through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`, and the extension decodes and checks it. This requires router cooperation and is fragile.

2. **Check `recipient` instead of `sender`**: For swap allowlists, gate on `recipient` (the address receiving output tokens) rather than `sender` (the router). This is semantically closer to "who benefits from the swap."

3. **Preferred — use a dedicated allowlist that the router enforces**: Move the allowlist check into the router itself, or have the extension read the original user from a transient storage slot written by the router before calling `pool.swap()`.

The cleanest fix is to have `SwapAllowlistExtension` check the `recipient` parameter (the economic beneficiary of the swap) rather than `sender` (the mechanical caller), since `recipient` is always the end user regardless of routing path.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - allowedSwapper[pool][router] = true   (admin enables router for whitelisted users)
  - allowedSwapper[pool][alice] = true    (alice is whitelisted)
  - allowedSwapper[pool][bob] = false     (bob is NOT whitelisted)

Direct swap by bob:
  bob → pool.swap(...)
  extension checks allowedSwapper[pool][bob] → false → REVERT ✓

Router swap by bob:
  bob → router.exactInput(pool, ...)
  router → pool.swap(recipient=bob, ...)
  extension checks allowedSwapper[pool][router] → true → PASSES ✗

Result: bob bypasses the allowlist by routing through the public router.
