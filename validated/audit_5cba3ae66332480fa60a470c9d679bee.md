### Title
Consensus Vote Signature Never Validated in `handle_vote` — (`File: crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary

`handle_vote` in `SingleHeightConsensus` accepts any vote whose `voter` field appears in the committee list, but never verifies the cryptographic signature carried in `vote.signature`. The verification library (`verify_precommit_vote_signature`) exists and returns `Result<bool>` — the boolean being the safety flag analogous to `pricesWereSafe` in the external report — but the caller never invokes it. Any network peer can therefore forge votes attributed to any committee member.

The `Vote` struct carries a `signature: RawSignature` field. The node's own votes are emitted with `signature: RawSignature::default()` (a zero value), confirming that the signing half is also unimplemented: [2](#0-1) 

The verification counterpart exists as a public library function that returns `SignatureVerificationResult<bool>` — `Ok(true)` for a valid signature, `Ok(false)` for an invalid one — exactly the pattern the external report describes as "the caller's responsibility to check": [3](#0-2) [4](#0-3) 

The codebase itself acknowledges the consequence in the future-vote cache comment:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses…" [5](#0-4) 

### Impact Explanation

Because `voter` is only checked for committee membership and the signature is never verified, any reachable network peer can craft a `Vote` message with `voter` set to any committee member's address and `signature` set to arbitrary bytes. The state machine will count that vote toward quorum. With enough forged votes an attacker can drive the consensus state machine to a `DecisionReached` event for an attacker-chosen block commitment, causing the sequencer to commit wrong state. This maps to:

**Critical — Wrong state committed from blockifier/execution logic for accepted input.**

### Likelihood Explanation

The consensus network is reachable by any peer that can connect to the broadcast channel. The `handle_vote` path in `manager.rs` calls `continue_propagation` (re-broadcasts the vote to other peers) before any signature check, so a single forged vote is also amplified across the network. No privileged access is required; only the ability to send a well-formed protobuf `Vote` message with a valid committee member address.

### Recommendation

Call `verify_precommit_vote_signature` (or the equivalent per-vote verification path) inside `handle_vote` before the vote is passed to the state machine, and treat `Ok(false)` as a rejection (report the peer, drop the vote). Simultaneously, replace `RawSignature::default()` in `make_self_vote` with a real ECDSA signature over the canonical vote digest so that the verification path is exercised end-to-end.

### Proof of Concept

1. Connect to the consensus broadcast channel as any network peer.
2. Construct a `Vote` with:
   - `voter` = address of a known committee member (public information)
   - `proposal_commitment` = target block hash
   - `signature` = `[0u8; 64]` (the current default)
3. Send the vote. `handle_vote` checks `vote.voter` is in the committee ✓, skips signature verification, and forwards the vote to the state machine.
4. Repeat for enough distinct committee member addresses to satisfy the quorum threshold.
5. The state machine emits `SMRequest::DecisionReached` for the attacker-chosen commitment, causing the sequencer to finalize a block the attacker selected. [6](#0-5)
