### Title
Protobuf `ValidResourceBounds` Serialization Ambiguity Causes Consensus Proposal Rejection for Valid Transactions — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`ValidResourceBounds::AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` and `ValidResourceBounds::L1Gas(X)` produce **identical protobuf bytes** during serialization. On deserialization, the zero-check always reconstructs `L1Gas(X)`. Because `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` only accepts `AllResources`, the validator's conversion fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing it to mark the entire consensus proposal as `Failed`. The proposer, which never goes through the protobuf path, executes the transaction successfully. This creates a consensus split that an unprivileged attacker can trigger by submitting a single crafted transaction.

---

### Finding Description

**Root cause — non-injective serialization:**

`From<ValidResourceBounds> for protobuf::ResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` serializes both variants to the same bytes when l2_gas and l1_data_gas are zero:

```rust
// AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),       // Some(zero)
        l1_data_gas: Some(l1_data_gas.into()), // Some(zero)
    },

// L1Gas(X)
ValidResourceBounds::L1Gas(l1_gas) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(value.get_l2_bounds().into()),  // Some(zero)
        l1_data_gas: Some(ResourceBounds::default().into()), // Some(zero)
    },
``` [1](#0-0) 

**Root cause — zero-check deserialization picks the wrong variant:**

```rust

        let l1_data_gas = value.l1_data_gas.unwrap_or_default();

            ValidResourceBounds::L1Gas(l1_gas)   // ← always chosen when both are zero
        } else {
            ValidResourceBounds::AllResources(...)

When `AllResources { X, 0, 0 }` is deserialized, it becomes `L1Gas(X)`.

**Hash preimage divergence (secondary effect):**

`get_tip_resource_bounds_hash` produces different outputs for the two variants. `AllResources` hashes 3 elements (L1_GAS, L2_GAS, L1_DATA_GAS); `L1Gas` hashes only 2 (L1_GAS, L2_GAS). Even if the conversion did not fail, the validator would compute a different `tx_hash` than the proposer, causing signature verification to fail in the blockifier. [6](#0-5) 

**Gateway accepts the trigger transaction:**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. The stateless validator accepts transactions with only `l1_gas` non-zero (l2_gas and l1_data_gas zero). The gateway computes the hash using `ValidResourceBounds::AllResources(self.resource_bounds)`, which includes the L1_DATA_GAS element. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A single attacker-submitted transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` causes every validator to reject the proposer's block proposal. The proposer executes the transaction successfully (it never goes through the protobuf path), but all validators fail to convert it and mark the proposal `Failed`. This prevents consensus from finalizing any block that contains such a transaction, constituting a denial-of-service on block production.

Matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** and **Critical. Wrong state … or revert result from blockifier/syscall/execution logic for accepted input** (the validator and proposer would compute different `tx_hash` values, causing signature verification divergence if the conversion error were bypassed).

---

### Likelihood Explanation

The gateway explicitly accepts transactions with only `l1_gas` non-zero. No privileged access is required. The attacker only needs to submit one such transaction to the public RPC endpoint. The bug is deterministic and reproducible on every consensus round that includes such a transaction.

---

### Recommendation

1. **Add an explicit discriminant to the protobuf `ResourceBounds` message** (e.g., a boolean `is_all_resources` field) so the variant can be reconstructed unambiguously without relying on value-based zero-checks.

2. **Alternatively**, in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, always reconstruct as `AllResources` when all three fields are present (even if zero), and only use `L1Gas` when `l1_data_gas` is absent (`None`) and `l2_gas` is zero. This preserves backward compatibility with pre-0.13.3 messages while correctly handling the ambiguous case.

3. **Remove the `TODO(Shahak)` workaround** (`unwrap_or_default` for `l1_data_gas`) once 0.13.2 support is dropped, and enforce that `l1_data_gas` must be present for all new transactions.

---

### Proof of Concept

```
1. Attacker submits RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      ResourceBounds { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      ResourceBounds::default(),   // zero
         l1_data_gas: ResourceBounds::default(),   // zero
     }

2. Gateway accepts it. Hash H1 computed with AllResources (3 elements):
     poseidon(tip, concat(L1_GAS, X), concat(L2_GAS, 0), concat(L1_DATA_GAS, 0))

3. Transaction enters mempool as InternalRpcTransaction { resource_bounds: AllResourceBounds }.

4. Proposer includes it in block proposal. Serializes to protobuf:
     ResourceBounds { l1_gas: Some(X), l2_gas: Some(0), l1_data_gas: Some(0) }

5. Validator deserializes:
     l1_data_gas = Some(0).unwrap_or_default() = 0
     l2_gas = 0
     0.is_zero() && 0.is_zero() == true  →  ValidResourceBounds::L1Gas(X)

6. TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3:
     resource_bounds is L1Gas, not AllResources  →  DEPRECATED_RESOURCE_BOUNDS_ERROR

7. convert_consensus_tx_to_internal_consensus_tx returns Err(...)

8. validate_proposal returns HandledProposalPart::Failed(...)

9. Consensus cannot finalize the block. Repeat for every subsequent proposal
   that includes this transaction.
```

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
