Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end-user when swaps are routed through `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of the pool's `swap()` call. When a user swaps via `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract, not the end user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. This creates two concrete failure modes: (1) allowlisted EOAs cannot swap through the router even though they are explicitly permitted, and (2) if the pool admin allowlists the router address to enable router-based swaps, every unprivileged user bypasses the allowlist entirely.

## Finding Description

**Call path — direct swap:**
`user EOA → pool.swap()` → `MetricOmmPool.swap` sets `msg.sender = user` → `_beforeSwap(sender=user, ...)` → `SwapAllowlistExtension.beforeSwap(sender=user)` → checks `allowedSwapper[pool][user]` ✓

**Call path — router swap:**
`user EOA → MetricOmmSimpleRouter.exactInputSingle()` → `pool.swap(recipient, ...)` with `msg.sender = router` → `_beforeSwap(sender=router, ...)` → `SwapAllowlistExtension.beforeSwap(sender=router)` → checks `allowedSwapper[pool][router]` ✗

In `MetricOmmPool.swap`, the pool passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // <-- this is the router, not the end user
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension:

```solidity
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, recipient, ...)  // sender = router address
)
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is the router (wrong — should be the end user).

**Bypass exploit path:**
1. Pool admin deploys a curated pool with `SwapAllowlistExtension`, `allowAllSwappers = false`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps (a natural operational step).
3. Any unprivileged user — including addresses the pool admin never intended to allow — calls `router.exactInputSingle(...)`.
4. The extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and passes. The allowlist is completely bypassed for all router users.

**Broken-functionality path (no pool admin error required):**
1. Pool admin allowlists specific EOAs: `setAllowedToSwap(pool, alice, true)`.
2. Alice calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router]` — router is not allowlisted → `NotAllowedToSwap` revert.
4. Alice, a legitimately allowlisted user, cannot use the standard periphery swap path.

The existing `onlyPool` guard in `BaseMetricExtension` only verifies the caller is a registered pool; it does not fix the wrong-actor binding.

## Impact Explanation
The bypass path allows any unprivileged user to trade on a pool whose admin explicitly configured an allowlist to restrict access. This is a direct admin-boundary break: the allowlist policy is rendered ineffective for all router-mediated swaps. The broken-functionality path makes the standard swap entrypoint unusable for legitimately allowlisted users, constituting broken core pool functionality. Both outcomes fall within the contest's allowed impact gate (admin-boundary bypass by an unprivileged path; broken core swap flow).

## Likelihood Explanation
The `MetricOmmSimpleRouter` is the primary public swap interface. Any user who calls it against an allowlisted pool triggers the mismatch. The bypass variant requires the pool admin to allowlist the router address, which is a natural operational step when the admin wants to support router-based swaps. The broken-functionality variant requires no pool admin error at all and is triggered by any allowlisted user attempting to use the router. Both paths are repeatable and require no special privileges.

## Recommendation
The extension should gate on the end user, not the direct pool caller. One approach: pass the original `msg.sender` (the end user) through `extensionData` from the router, and have the extension decode it. A cleaner approach: add a `swapper` field to the extension interface that the pool populates from a trusted source (e.g., a transient-storage slot set by the router before calling the pool), or require direct pool calls for allowlisted pools. At minimum, document that `sender` in `beforeSwap` is the direct pool caller, so pool admins know to allowlist the router rather than individual EOAs — and understand the implication that doing so opens the pool to all router users.

## Proof of Concept

```solidity

// 2. Pool admin allowlists the router (to support router swaps)
swapExtension.setAllowedToSwap(address(pool), address(router), true);

// 3. Unprivileged attacker (not individually allowlisted) calls router
vm.prank(attacker); // attacker not in allowedSwapper[pool]
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    tokenIn: token0,
    tokenOut: token1,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: attacker,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Extension checks allowedSwapper[pool][router] == true → passes
// Attacker successfully swaps on a pool they were never meant to access
```
