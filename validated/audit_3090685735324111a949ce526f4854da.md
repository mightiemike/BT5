Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Enabling Allowlist Bypass or Breaking Router Swaps for Allowlisted Users - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`MetricOmmPool.swap` derives `sender` from `msg.sender` and forwards it to `_beforeSwap`, which passes it unchanged to `SwapAllowlistExtension.beforeSwap`. When `MetricOmmSimpleRouter` mediates a swap, the pool's `msg.sender` is the router, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][end_user]`. This produces two mutually exclusive failure modes: allowlist bypass (if the router is allowlisted) or broken router functionality for allowlisted users (if it is not).

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool forwarded: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` with no mechanism to pass the originating EOA: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The pool's `swap` interface has no explicit `sender` parameter; it always derives it from `msg.sender`. The extension has no visibility into the original EOA.

**Mode A – Allowlist bypass:** Pool admin allowlists the router so router-mediated swaps work. Because the check is `allowedSwapper[pool][router]`, every user who calls the router passes the check regardless of individual allowlist status. Any unprivileged user can trade on a curated pool by routing through `MetricOmmSimpleRouter`.

**Mode B – Broken router functionality:** Pool admin allowlists only individual user addresses and does not allowlist the router. Allowlisted users who call through the router have their swaps revert with `NotAllowedToSwap` because the router address is not in the allowlist.

Existing guards are insufficient: `allowAllSwappers` is a global bypass that defeats the purpose of per-user gating entirely, and there is no mechanism in the extension or pool interface to recover the originating EOA.

## Impact Explanation

Mode A is a direct admin-boundary break: an unprivileged user gains access to a curated pool explicitly configured to exclude them. Depending on pool configuration, this enables unauthorized extraction of LP value or violation of compliance constraints the pool admin intended to enforce. Mode B renders the router unusable for every allowlisted user on any pool using `SwapAllowlistExtension` without also allowlisting the router, constituting broken core swap functionality. Both modes meet the allowed impact gates.

## Likelihood Explanation

`SwapAllowlistExtension` is a first-party periphery extension documented as a reference implementation for access-controlled pools. The router is the primary user-facing swap surface. Any pool that deploys this extension and expects per-user gating will encounter one of the two failure modes the moment a user attempts a router-mediated swap. No special privileges or malicious setup are required — a standard `exactInputSingle` call through `MetricOmmSimpleRouter` is sufficient to trigger either mode.

## Recommendation

`MetricOmmPool.swap` should accept an explicit `sender` parameter (the originating user) rather than deriving it from `msg.sender`. The router would pass `msg.sender` (the EOA) as that parameter, and the pool would forward it to extensions. This mirrors the existing pattern used for `recipient`. Alternatively, `SwapAllowlistExtension` could require the router to embed the real user address in `extensionData` and verify it, but this is fragile and requires router cooperation. The cleanest fix is a first-class `sender` parameter on `pool.swap`.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — Alice is the only allowlisted user.
3. Pool admin also calls `setAllowedToSwap(pool, router, true)` so that router-mediated swaps are possible.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. The router calls `pool.swap(recipient=bob, ...)`. The pool's `msg.sender` is the router.
6. `_beforeSwap(router, bob, ...)` is called; the extension checks `allowedSwapper[pool][router] == true` → passes.
7. Bob's swap executes successfully on a pool he was explicitly excluded from.

Conversely, if step 3 is omitted (router not allowlisted):

4. Alice calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. The router calls `pool.swap(...)`. Pool's `msg.sender` is the router.
6. Extension checks `allowedSwapper[pool][router] == false` → reverts `NotAllowedToSwap`.
7. Alice cannot use the router despite being individually allowlisted.
