Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Caller-Controlled `owner` Instead of `sender`, Fully Bypassing the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter and gates on `owner`, which is a free caller-supplied field. Because `MetricOmmPoolLiquidityAdder` accepts an arbitrary `owner` address (validated only to be non-zero), any unprivileged address can pass the allowlist check by nominating any allowlisted address as `owner`. The pool admin's deposit restriction is completely defeated.

## Finding Description

`MetricOmmPool.addLiquidity` fires the hook with the true caller as `sender` and the caller-supplied position owner as `owner`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values faithfully to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then ignores `sender` entirely (first parameter is unnamed) and checks only `owner`: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` address, validates only that it is non-zero, and passes it straight through to the pool: [4](#0-3) 

The contrast with `SwapAllowlistExtension` makes the mismatch explicit — the swap guard correctly checks `sender`: [5](#0-4) 

## Impact Explanation

A pool configured with `DepositAllowlistExtension` intends to restrict liquidity provision to a curated set of addresses. Because the guard checks `owner` (caller-controlled) rather than `sender` (the actual depositing address), the allowlist invariant is completely broken. Any external actor can deposit into a restricted pool by supplying any allowlisted address as `owner`. This constitutes a broken core pool functionality (deposit access control rendered inoperative) and an admin-boundary break (pool admin's restriction is bypassed by an unprivileged path). Additionally, it enables griefing: an attacker can force liquidity into another address's position without that address's consent.

## Likelihood Explanation

Exploitation requires only knowledge of one allowlisted address and a call through the publicly available `MetricOmmPoolLiquidityAdder`. No privileged role, special token, or malicious setup is needed. The attacker bears only gas costs and the token amounts they choose to deposit. The attack is repeatable against every pool that has `DepositAllowlistExtension` configured.

## Recommendation

Replace the `owner` check with a `sender` check, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Pool admins using `MetricOmmPoolLiquidityAdder` should then allowlist the adder contract address as a trusted intermediary, consistent with how the swap allowlist handles the router.

## Proof of Concept

1. Pool `P` is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` extension.
2. Pool admin calls `setAllowedToDeposit(P, alice, true)`. Only `alice` is supposed to deposit.
3. Attacker `bob` (not allowlisted) calls:
   ```solidity
   liquidityAdder.addLiquidityExactShares(P, alice, salt, deltas, max0, max1, "");
   ```
4. `MetricOmmPoolLiquidityAdder._validateOwner(alice)` passes (non-zero check only).
5. Pool calls `_beforeAddLiquidity(liquidityAdder, alice, ...)`.
6. Extension evaluates `allowedDepositor[P][alice]` → `true` → no revert.
7. `bob`'s tokens are transferred into pool `P`; the LP position is minted to `alice`.
8. The deposit allowlist has been bypassed; `bob` has added liquidity to a pool that was supposed to exclude him.
