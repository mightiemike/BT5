### Title
`op_ash` and `op_lsh` Ignore `max_cost`, Performing Expensive Bignum Work Before Any Budget Check — (File: `src/more_ops.rs`)

### Summary

`op_ash` and `op_lsh` explicitly ignore the `max_cost` parameter (both are named `_max_cost`). They perform the full bignum shift and heap allocation before computing the cost, and never call `check_cost`. This is the direct analog of the external report's pattern: a "commit" phase (the interpreter's pre-operator cost check) is cheaper than the "execute" phase (the actual bignum shift + allocator write), allowing the operator to consume CPU and heap beyond the remaining budget before the interpreter can stop it.

### Finding Description

Both shift operators have the signature:

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response
pub fn op_lsh(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response
```

The `_max_cost` prefix signals the parameter is intentionally unused. Inside each function the execution order is:

1. Parse arguments
2. Perform the bignum shift: `let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };`
3. Allocate the result atom: `let r = a.new_number(v)?;`
4. **Only then** compute the cost: `let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;`
5. Return `malloc_cost(a, cost, r)` — which adds `atom_len(r) * MALLOC_COST_PER_BYTE`

The cost formula includes `l1` — the byte-length of the **output** — which is unknowable until after the shift executes. Because the output size drives the cost, no pre-check is possible with the current design, and none is attempted.

Compare with `op_sha256`, which calls `check_cost` incrementally inside its argument loop:

```rust
cost += blob.as_ref().len() as Cost * SHA256_COST_PER_BYTE;
check_cost(cost, max_cost)?;
```

`op_ash`/`op_lsh` have no equivalent guard.

The interpreter's main loop does check cost before dispatching each operator:

```rust
if cost > effective_max_cost {
    return Err(EvalErr::CostExceeded);
}
// ...
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
```

But this pre-check uses the **accumulated** cost, not the cost the operator is about to incur. When `cost == effective_max_cost` the check passes (`>` not `>=`), `apply_op` is called with `max_cost = 0`, and `op_ash`/`op_lsh` proceed to do the full shift and allocation with a zero budget — ignoring it entirely.

### Impact Explanation

An attacker-controlled CLVM program can invoke `(ash X 65535)` or `(lsh X 65535)` when the remaining budget is at or near zero. The operator will:

- Execute a 65535-bit bignum left-shift (CPU-intensive)
- Allocate a result atom up to ~8 192 bytes on the `Allocator` heap
- Return a cost of approximately `596 + (l0 + 8192) * 3 + 8192 * 10 ≈ 107 000`

All of this work is done before the interpreter can enforce the budget. The budget enforcement is post-hoc: the next loop iteration sees the cost exceeded and returns `CostExceeded`, but the CPU cycles and heap bytes have already been consumed.

Inside a `softfork` guard the `effective_max_cost` is the caller-declared `expected_cost`. A caller who declares a small `expected_cost` and then calls `ash`/`lsh` with a large shift forces the operator to do full work against a budget of zero.

The maximum over-budget work is bounded (~8 KB allocation, one bignum shift), so this is not catastrophic. However it is a concrete budget-enforcement bypass: the operator does real work that the cost model says should be rejected, and the rejection only arrives after the fact.

### Likelihood Explanation

Any attacker-controlled CLVM bytes submitted to `run_program` or `run_serialized_chia_program` can trigger this. The trigger is a single `ash` or `lsh` call with a large positive shift amount (up to 65535) on a non-trivial input, timed so that the accumulated cost equals `effective_max_cost` at the point of dispatch. This is straightforward to craft deliberately.

### Recommendation

Add a conservative pre-check in `op_ash` and `op_lsh` before performing the shift. The maximum output size is computable from the input size and shift amount:

- For a left shift of `a1` bits on an input of `l0` bytes, the output is at most `l0 + ceil(a1 / 8)` bytes.
- Compute the upper-bound cost from this estimate and call `check_cost` before executing the shift.

This mirrors the pattern used by `op_sha256` and the `sha256tree` operator (which the documentation explicitly notes must tally cost during runtime so it can fail immediately if `max_cost` is exceeded).

### Proof of Concept

```
; Remaining budget is exactly 0 when this operator is dispatched.
; op_ash ignores max_cost=0 and performs the full 65535-bit shift.
(ash (q . 500) (q . 65535))
```

Attacker submits CLVM bytes encoding a program that accumulates cost to exactly `max_cost` via cheap operations, then calls `(ash 500 65535)`. The operator executes the full bignum shift and allocates ~8 KB before the interpreter can enforce the budget.
