### Title
Deprecated `verify_transaction_inclusion` Remains Callable and Vulnerable to 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function in `contract/src/lib.rs` lacks validation that the caller-supplied `tx_id` is a Merkle tree leaf node rather than an internal node. Any unprivileged NEAR caller can supply a crafted internal-node hash as `tx_id` along with a valid sibling path to the root, causing the function to return `true` for a transaction that does not exist in the block. This is a direct analog to M-26's missing-validation class: just as `formPOL` accepted liquidity deposits without bounding the pool ratio, `verify_transaction_inclusion` accepts proof inputs without bounding what `tx_id` may represent.

---

### Finding Description

`verify_transaction_inclusion` is still a public, callable NEAR method. In Rust, `#[deprecated]` emits a compiler warning only; it does not restrict runtime invocation. Any NEAR account can call it today. [1](#0-0) 

The function delegates to `merkle_tools::compute_root_from_merkle_proof`, which iterates over the supplied `merkle_proof` sibling list, hashing left or right at each step according to `current_position % 2`, and returns whatever hash results: [2](#0-1) 

The only guard on `tx_id` is: [3](#0-2) 

There is **no check** that `tx_id` is a leaf-level hash. Bitcoin's Merkle tree internal nodes are the SHA256d of two concatenated 32-byte child hashes (64 bytes of input). An attacker who knows the Merkle tree of any mainchain block can:

1. Pick an internal node `N` at height `h` above the leaves (depth `D − h` from the root).
2. Observe that `N`'s position at its level is `p = leaf_index >> h`.
3. Collect the `D − h` sibling hashes from `N`'s level up to the root.
4. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = p`, `merkle_proof = [those D−h siblings]`.

`compute_root_from_merkle_proof` will faithfully walk from `N` to the root and produce the correct `merkle_root`. The comparison succeeds and the function returns `true`, even though `N` is not a transaction.

The `verify_transaction_inclusion_v2` replacement correctly mitigates this by requiring a coinbase proof of the same length as the tx proof, which forces the proof depth to equal the actual tree depth and prevents internal-node substitution: [4](#0-3) 

However, v1 remains reachable and unprotected.

---

### Impact Explanation

Any NEAR dApp or bridge contract that calls `verify_transaction_inclusion` (v1) to gate cross-chain actions — releasing funds, minting tokens, recording settlements — can be deceived into accepting a proof for a Bitcoin transaction that was never broadcast or confirmed. The corrupted protocol value is the **proof result**: the function returns `true` where the correct answer is `false`. Downstream contracts that trust this result will execute actions predicated on a non-existent Bitcoin event.

---

### Likelihood Explanation

Medium. The Merkle tree of every Bitcoin block is public. An attacker needs only to read the block's transaction list, compute the tree, select any internal node, and assemble the sibling path. No privileged access, key material, or social engineering is required. The function is callable by any NEAR account with no role restriction.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it behind an access-control role so that only trusted callers can invoke it. All consumers must be migrated to `verify_transaction_inclusion_v2`. If backward compatibility is required, add an explicit check that `args.merkle_proof.len()` equals the expected tree depth derived from a coinbase proof, mirroring the v2 logic.

---

### Proof of Concept

Given a mainchain block `B` with `merkle_root = R` and at least two transactions (tree depth `D ≥ 1`):

1. Compute the full Merkle tree for `B`.
2. Select the root's left child `N` (an internal node at height `D−1`). Its position at that level is `0`.
3. Its sibling is the root's right child `S`.
4. Call on NEAR:

```
verify_transaction_inclusion({
    tx_id:              N,          // internal node, not a real tx hash
    tx_block_blockhash: hash(B),
    tx_index:           0,          // position of N at its level
    merkle_proof:       [S],        // one sibling brings N → R
    confirmations:      1,
})
```

5. `compute_root_from_merkle_proof(N, 0, [S])` computes `SHA256d(N ‖ S) = R`.
6. `R == header.block_header.merkle_root` → function returns **`true`**.

No transaction with hash `N` exists in block `B`. The verification is forged. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}

fn compute_hash(first_tx_hash: &H256, second_tx_hash: &H256) -> H256 {
    let mut concat_inputs = Vec::with_capacity(64);
    concat_inputs.extend(first_tx_hash.0);
    concat_inputs.extend(second_tx_hash.0);

    double_sha256(&concat_inputs)
}
