### Title
`OracleValueStopLossExtension` Silently Disables Stop-Loss When `drawdownE6 == 0` Instead of Enforcing Strictest Protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` contains an inverted boundary guard: it returns early (no check) when `drawdownE6 == 0`, but `0` is the semantically strictest setting (zero drawdown allowed). A pool initialized with `drawdownE6 = 0` has its entire stop-loss bypassed, allowing any public swapper to drain LP value without triggering the configured protection.

---

### Finding Description

In `_afterSwapOracleStopLoss`, the first substantive line after loading config is:

```solidity
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;          // ← early exit, no watermark check
``` [1](#0-0) 

The `drawdownE6` parameter represents the maximum permitted fractional loss in E6 units. The floor multiplier used in the actual breach check is:

```solidity
uint256 floorMultiplier = E6 - drawdown;
``` [2](#0-1) 

When `drawdown == 0`, `floorMultiplier = 1e6`, meaning the floor equals 100% of the watermark — any metric drop below the watermark triggers a breach. This is the **strictest** possible configuration, not a "disabled" state.

The actual disabled state is `drawdown == E6` (100% drawdown allowed), which makes `floorMultiplier = 0` and the breach condition `metric < 0` — never true.

The `_validateDrawdown` function explicitly permits `0` as a valid input without any "disables the guard" annotation:

```solidity
function _validateDrawdown(uint256 drawdownE6) private pure {
    if (drawdownE6 > E6) revert OracleStopLossDrawdownTooLarge(drawdownE6);
}
``` [3](#0-2) 

Contrast this with the decay parameter, which **does** document that zero disables it:

```
/// @notice Linear watermark decay per second, E8 scale (58 ~= 5%/day). 0 disables decay.
``` [4](#0-3) 

No equivalent documentation exists for `drawdownE6 = 0`. A pool admin who sets `drawdownE6 = 0` intending the strictest stop-loss (no drawdown tolerated) receives zero protection.

The `_applyWatermark` and `_checkAndUpdateWatermarks` functions that would enforce the breach are never reached:

```solidity
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) {
    revert OracleStopLossTriggered(...);
}
``` [5](#0-4) 

---

### Impact Explanation

Any public swapper on a pool whose `OracleValueStopLossExtension` was initialized with `drawdownE6 = 0` can execute swaps that drain LP value without the stop-loss ever triggering. The watermarks are never updated and no revert is issued regardless of how far the per-share metric falls. LP principal is directly at risk.

---

### Likelihood Explanation

`drawdownE6 = 0` is a valid, accepted initialization value (passes `_validateDrawdown`). A pool admin intending the strictest protection — "no drawdown tolerated" — would naturally supply `0`. The absence of any "0 disables" documentation makes this a realistic misconfiguration. The trigger is any public `swap` call on such a pool; no special privilege is required.

---

### Recommendation

Remove the `if (drawdown == 0) return;` early exit, or replace it with the semantically correct disabled check:

```solidity
// Disabled state: 100% drawdown allowed → floor is always 0 → never breaches
if (drawdown >= E6) return;
```

Alternatively, document explicitly that `drawdownE6 = 0` disables the guard and add a validation that rejects 0 if the intent is "strictest protection". The decay parameter's pattern (`0 disables decay`) should be mirrored for drawdown if the intent is truly to use 0 as a sentinel.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` initialized via `abi.encode(uint32(0), uint32(58), uint32(0))` — `drawdownE6 = 0`, decay enabled, no timelock.
2. Add liquidity to the pool.
3. Execute a sequence of adversarial swaps that move the oracle mid-price sharply, causing per-share bin value to drop well below the initial watermark.
4. Observe: `afterSwap` is called, `_afterSwapOracleStopLoss` loads `drawdown = 0`, immediately returns, and no `OracleStopLossTriggered` revert is issued.
5. With `drawdownE6 = 50_000` (5%) under identical conditions, the same swap sequence reverts with `OracleStopLossTriggered`.

The corrupted value is `drawdown == 0` being treated as "guard disabled" when it should produce `floorMultiplier = 1e6` — the strictest possible floor — causing the stop-loss to fail open on every swap.
