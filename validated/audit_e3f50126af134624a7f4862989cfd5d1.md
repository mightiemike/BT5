### Title
Fork Retarget Reads Mainchain Timestamp Instead of Fork Ancestor — (`contract/src/litecoin.rs`, `contract/src/lib.rs`)

---

### Summary

`get_next_work_required` in the Litecoin module calls `blocks_getter.get_header_by_height(first_block_height)` to obtain the timestamp of the interval-start block for difficulty retargeting. The sole implementation of `get_header_by_height` unconditionally reads from `mainchain_height_to_header` — the mainchain height index — even when the block being validated belongs to a fork whose divergence point is below `first_block_height`. When the fork diverges before `first_block_height`, the mainchain block at that height is a different block (with a different timestamp) than the fork's true ancestor at that height. The contract therefore computes the expected `bits` using the wrong timestamp, accepts a fork header whose `bits` would be rejected by the real Litecoin network, and may store that fork as canonical after a reorg.

---

### Finding Description

**`get_header_by_height` is mainchain-only:** [1](#0-0) 

The function reads exclusively from `mainchain_height_to_header`. There is no fork-aware path.

**`get_next_work_required` calls it unconditionally at retarget boundaries:** [2](#0-1) 

`first_block_height = prev_block_header.block_height - blocks_to_go_back` (2016 blocks back for a normal retarget). The timestamp of the block at that height is fed directly into `calculate_next_work_required`.

**The invariant that breaks:** when a fork diverges at height `H_fork < first_block_height`, the fork's ancestor at `first_block_height` is a fork block (attacker-controlled timestamp), but the contract reads the mainchain block at that height (fixed, canonical timestamp). The two timestamps differ, so the computed `bits` differs from what the real Litecoin node would compute for the same fork.

**The same bug is acknowledged in the Dogecoin module with a TODO comment:** [3](#0-2) 

The comment reads: *"TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor"* — confirming the design is known to be incorrect.

**Contrast with `get_prev_header`**, which correctly traverses the fork lineage via `headers_pool` using `prev_block_hash` links: [4](#0-3) 

`get_header_by_height` has no equivalent fork-aware lookup.

---

### Impact Explanation

An attacker who controls a fork with scrypt-valid headers can:

1. Diverge the fork at height `H_fork`, where `H_fork < first_block_height` for the next retarget.
2. Build fork blocks at heights `H_fork+1` through `first_block_height` with a crafted timestamp `T_fork` at `first_block_height`.
3. At the retarget block, the contract reads the mainchain block's timestamp `T_main` (not `T_fork`), computes `bits_contract = f(prev_time, T_main)`.
4. The attacker sets the fork's retarget block `bits` to `bits_contract`. The contract's `check_pow` passes.
5. The real Litecoin network would compute `bits_real = f(prev_time, T_fork)` and reject the block.
6. If the fork accumulates more chainwork, `submit_block_header_inner` triggers a reorg, promoting the fork to mainchain. [5](#0-4) 

The contract's mainchain now contains headers with a difficulty that the source chain would never accept as canonical. Any downstream bridge using this light client's `verify_transaction_inclusion` to confirm finality treats those confirmations as valid, enabling theft or loss of bridged funds.

---

### Likelihood Explanation

- **Entry point**: `submit_blocks` is gated by `#[trusted_relayer]`, so the attacker must be a staked relayer or exploit the relayer path. This is explicitly listed as an in-scope entry point. [6](#0-5) 

- **PoW requirement**: The attacker must produce scrypt-valid headers for the entire fork (2016+ blocks). This requires meaningful hashrate but is achievable for a well-resourced attacker targeting a bridge with high TVL.
- **No other guard catches this**: `check_pow` only verifies that `header.bits == expected_bits` where `expected_bits` is computed by the buggy `get_next_work_required`. There is no cross-check against the fork's own ancestor chain. [7](#0-6) 

---

### Recommendation

Replace the height-based lookup in `get_next_work_required` with an ancestor walk using `get_prev_header`, which correctly follows `prev_block_hash` links through the fork's own lineage. Starting from `prev_block_header`, walk back `blocks_to_go_back` steps via `get_prev_header` to reach the true interval-start ancestor on the fork. This is O(2016) storage reads per retarget validation — acceptable given that retargets are infrequent — and matches what the reference Litecoin node does.

The `get_header_by_height` function should either be removed from the `BlocksGetter` trait or documented as mainchain-only and never called during fork validation.

---

### Proof of Concept

**Setup (Litecoin mainnet, `difficulty_adjustment_interval = 2016`):**

```
Mainchain:  [0] -> [1] -> ... -> [2014] -> [2015_main] -> [2016_main] -> ... -> [4031_main]
Fork:       [0] -> [1] -> ... -> [2014] -> [2015_fork] -> [2016_fork] -> ... -> [4031_fork] -> [4032_fork]
                                         ^ divergence point (H_fork = 2014)
```

**Retarget at fork block 4032:**
- `prev_block_height = 4031`, `blocks_to_go_back = 2016`
- `first_block_height = 4031 - 2016 = 2015`
- Fork diverged at 2014, so `first_block_height (2015) > H_fork (2014)` → mainchain block at 2015 is read

**Timestamp manipulation:**
- Set `T_fork_2015` (fork block at height 2015) = `T_main_2015 + Δ` (e.g., Δ = 3 days in seconds)
- Contract computes `bits_contract` using `T_main_2015` (mainchain timestamp)
- Real node computes `bits_real` using `T_fork_2015 = T_main_2015 + Δ` → different difficulty

**Submission:**
- Attacker submits fork blocks 2015 through 4032 via `submit_blocks`
- At block 4032, `check_pow` computes `expected_bits` using `T_main_2015` → matches attacker's crafted `bits_contract` → passes
- If fork chainwork > mainchain chainwork, reorg fires, promoting the fork

**Verification:** Run the same retarget formula against a reference Litecoin node using the fork's true ancestor at height 2015 (`T_fork_2015`). The node produces `bits_real ≠ bits_contract` and rejects block 4032 as `bad-diffbits`. The contract accepted it. [2](#0-1) [1](#0-0)
