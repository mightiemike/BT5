### Title
Missing Lower-Bound Validation on Caller-Supplied `confirmations` Allows Zero-Confirmation Proof Acceptance — (`contract/src/lib.rs`, `btc-types/src/contract_args.rs`)

---

### Summary

The `confirmations` field in `ProofArgs` and `ProofArgsV2` accepts any `u64` value, including `0`. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` enforce only an upper-bound check (`confirmations <= gc_threshold`) but no lower-bound check. Passing `confirmations = 0` trivially satisfies every guard in the function, causing the contract to return `true` for a transaction in the chain-tip block — a block with zero depth that is maximally susceptible to reorganization.

---

### Finding Description

`ProofArgs.confirmations` and `ProofArgsV2.confirmations` are declared as plain `u64` with no minimum enforced at the type or validation layer: [1](#0-0) [2](#0-1) 

Inside `verify_transaction_inclusion`, the two guards that are supposed to enforce confirmation depth are:

```rust
require!(
    args.confirmations <= self.gc_threshold,   // (1) upper-bound only
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
...
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,                 // (2) depth check
    "Not enough blocks confirmed"
);
``` [3](#0-2) 

With `confirmations = 0`:

- Guard (1): `0 <= gc_threshold` — always `true` for any `u64`.
- Guard (2): `(any_u64) >= 0` — always `true` because `u64` is unsigned; the left-hand side can never be negative.

Both guards are vacuously satisfied. The function proceeds to compute and return the Merkle proof result, yielding `true` for a transaction whose block is the current chain tip — zero blocks deep.

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its own coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and gates an action (e.g., releasing bridged funds, minting tokens) on a `true` result can be exploited. An attacker:

1. Broadcasts a Bitcoin transaction.
2. Waits only for the relayer to include the containing block in the light client (no additional confirmations needed).
3. Immediately calls `verify_transaction_inclusion_v2` with `confirmations = 0`.
4. Receives `true`.
5. Triggers the downstream action before the Bitcoin transaction has any confirmation depth, while the block is still trivially reorganizable.

The corrupted invariant is the proof-verification result: the contract asserts a transaction is "confirmed" when it has zero confirmation depth, directly falsifying the SPV guarantee the contract is designed to provide.

---

### Likelihood Explanation

The entry path requires no privilege — `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, unpermissioned NEAR calls. The only prerequisite is that the relayer has already submitted the block containing the target transaction, which is the normal operating condition. The exploit is a single call with a trivially crafted argument (`confirmations = 0`). Any actor who knows the `tx_id` and `tx_block_blockhash` can execute it immediately after block submission.

---

### Recommendation

Add an explicit lower-bound check at the start of `verify_transaction_inclusion` (and/or enforce it in `ProofArgs` construction):

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Additionally, consider enforcing a protocol-level minimum (e.g., `>= 6` for Bitcoin mainnet) as a constant, analogous to how the original report recommended capping `_FEE_RATE_` at a safe maximum. The minimum should be documented and enforced in `ProofArgs` / `ProofArgsV2` so that callers cannot accidentally or maliciously bypass it.

---

### Proof of Concept

```
// Attacker constructs ProofArgs with confirmations = 0
let args = ProofArgs {
    tx_id:              <valid tx hash in tip block>,
    tx_block_blockhash: <hash of current chain tip>,
    tx_index:           <correct index>,
    merkle_proof:       <valid merkle path>,
    confirmations:      0,   // ← no lower-bound check exists
};

// Call is public, no role required
let result = contract.verify_transaction_inclusion(args);
// result == true, despite 0 confirmation depth
```

Guard (1) at [5](#0-4)  passes because `0 <= gc_threshold`.
Guard (2) at [6](#0-5)  passes because `u64 >= 0` is always true.
The Merkle root comparison at [7](#0-6)  then determines the return value, returning `true` for a valid proof — with zero confirmation depth enforced.
