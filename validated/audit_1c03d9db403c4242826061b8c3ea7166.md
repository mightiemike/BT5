### Title
Silent Checkpoint Advancement on `ErrInsufficientBonusPool` During Redelegation Slash Permanently Destroys Accrued Bonus Rewards — (`x/tieredrewards/keeper/slash.go`, `claim_rewards.go`)

---

### Summary

When `BeforeRedelegationSlashed` fires and the `RewardsPoolName` module account has insufficient balance, `slashRedelegationPosition` deliberately swallows `ErrInsufficientBonusPool` to prevent a chain halt. However, by the time `processEventsAndClaimBonus` returns that error, it has **already mutated** the in-memory `pos` pointer — advancing `LastBonusAccrual`, `LastEventSeq`, and decrementing (potentially deleting) validator event reference counts. The mutated position is then persisted via `setPositionWithState`, permanently destroying the owner's accrued bonus with no payment and no recoverable state.

---

### Finding Description

**Step 1 — Entry point.**

`BeforeRedelegationSlashed` in `hooks.go` routes directly to `slashRedelegationPosition`: [1](#0-0) 

**Step 2 — Error swallowing.**

`slashRedelegationPosition` calls `processEventsAndClaimBonus(ctx, &pos)` passing a pointer. If the pool is empty, the returned `ErrInsufficientBonusPool` is caught, logged, and execution continues: [2](#0-1) 

**Step 3 — State mutations happen BEFORE the pool balance check inside `processEventsAndClaimBonus`.**

The function mutates the `pos` pointer in three ways before it ever checks the pool:

1. **`pos.UpdateLastEventSeq`** is called inside the event loop for every processed event: [3](#0-2) 

2. **`decrementEventRefCount`** is called for each event — permanently deleting events whose reference count reaches zero: [4](#0-3) 

3. **`applyBonusAccrualCheckpoint`** advances `pos.LastBonusAccrual` to the current block time: [5](#0-4) 

Only **after** all three mutations does the function check pool sufficiency and return `ErrInsufficientBonusPool`: [6](#0-5) 

**Step 4 — Mutated position is persisted.**

After swallowing the error, `slashRedelegationPosition` calls `setPositionWithState` with the already-mutated `pos`: [7](#0-6) 

This writes to the `Positions` store the advanced `LastBonusAccrual` and `LastEventSeq` values, with no bonus ever transferred to the owner.

**Step 5 — Loss is unrecoverable.**

Because `LastEventSeq` has advanced past the processed events, and those events' reference counts have been decremented (and possibly deleted), the owner can never re-claim the bonus for that period — even if the pool is later refilled. The `applyBonusAccrualCheckpoint` advancement means the accrual window is also closed.

---

### Impact Explanation

- **Permanent loss of accrued bonus rewards** for the position owner, with zero coins transferred.
- No on-chain event is emitted for the skipped payment (only a node log message).
- The position's checkpoints are advanced as if payment succeeded, making the loss unrecoverable.
- Affects any position whose redelegation is slashed while `RewardsPoolName` balance is insufficient.

---

### Likelihood Explanation

- The `RewardsPoolName` pool is a finite module account. It can be depleted by normal reward claims across many positions.
- Validator slashes are routine chain events requiring no special privileges.
- No governance, operator compromise, or privileged key is needed — the precondition (empty pool + slash) is reachable through normal chain operation.
- An adversary who wants to grief a specific victim could front-run a known slash by draining the pool via their own legitimate reward claims.

---

### Recommendation

The fix must ensure that **if the bonus cannot be paid, the position checkpoints are not advanced**. Two options:

1. **Reorder operations in `processEventsAndClaimBonus`**: check pool sufficiency *before* mutating `pos.LastBonusAccrual`, `pos.LastEventSeq`, and decrementing event reference counts. If the pool is insufficient, return the error without any state mutation.

2. **In `slashRedelegationPosition`**: if `ErrInsufficientBonusPool` is caught, explicitly roll back the checkpoint fields on `pos` to their pre-call values before calling `setPositionWithState`, and do not decrement event reference counts.

Option 1 is cleaner. The pool check at line 230 of `claim_rewards.go` should be moved to before the event loop, or the function should operate on a copy and only apply mutations after confirming the payment can succeed.

---

### Proof of Concept

```go
// Keeper test outline:
// 1. Create a tiered position with a redelegation mapping.
// 2. Append a validator SLASH event so processEventsAndClaimBonus has work to do.
// 3. Drain RewardsPoolName to zero.
// 4. Record pos.LastBonusAccrual and pos.LastEventSeq before the hook.
// 5. Call BeforeRedelegationSlashed (or slashRedelegationPosition directly).
// 6. Assert:
//    a. No error returned.
//    b. pos.LastBonusAccrual > pre-call value (checkpoint advanced).
//    c. pos.LastEventSeq > pre-call value (event seq advanced).
//    d. Owner's bank balance unchanged (no bonus paid).
//    e. RewardsPoolName balance still zero (no transfer attempted).
```

The test would confirm that the position's checkpoints advance while the owner receives nothing, violating the invariant that accrued bonus rewards are always paid before a slash settles.
