### Title
`finalize_epoch()` validator-selection fallback silently discards the computed `next_next_shard_layout`, creating a permanent divergence between the committed `BlockHeaderInnerRestV6.shard_split` and the stored `EpochInfoV5.shard_layout` for epoch T+2 — (`chain/epoch-manager/src/lib.rs`)

---

### Summary

`EpochManager::finalize_epoch()` first derives `next_next_shard_layout` (potentially a new `ShardLayoutV3` with a split boundary account) from the `shard_split` field already committed in the last block's header, then calls `proposals_to_epoch_info()` with that layout. When `proposals_to_epoch_info()` fails with `EpochError::ThresholdError` or `EpochError::NotEnoughValidators`, the fallback path clones `next_epoch_info` (epoch T+1's info) and only increments its epoch height — silently discarding both the computed `next_next_shard_layout` and the computed `last_resharding` value. The stored `EpochInfoV5.shard_layout` for epoch T+2 therefore contains the T+1 layout (no split), while the block header that triggered `finalize_epoch()` has already committed a non-`None` `shard_split` to the network. This is a permanent, deterministic protocol-state divergence: every node runs the same fallback and agrees on the wrong layout, but the layout is inconsistent with the block header's committed split.

---

### Finding Description

In `finalize_epoch()`, the normal path is:

1. Derive `next_next_shard_layout` via `next_next_shard_layout()` — this reads `block_info.shard_split()` from the last block of epoch T and, when dynamic resharding is active, calls `ShardLayout::derive_v3()` to produce a new `ShardLayoutV3` with the split boundary account.
2. Compute `last_resharding` from whether the layout changed.
3. Call `proposals_to_epoch_info(..., next_next_shard_layout.clone(), ..., last_resharding)` to produce the epoch T+2 `EpochInfo`.
4. Save the result as `EpochInfoV5` (because `ProtocolFeature::DynamicResharding` is enabled), which stores `shard_layout` and `last_resharding` as first-class fields.

The fallback path for `ThresholdError` and `NotEnoughValidators` is:

```rust
Err(EpochError::ThresholdError { .. }) => {
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);   // clones T+1 info
    *epoch_info.epoch_height_mut() += 1;
    epoch_info                                                  // next_next_shard_layout dropped
}
Err(EpochError::NotEnoughValidators { .. }) => {
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);   // clones T+1 info
    *epoch_info.epoch_height_mut() += 1;
    epoch_info                                                  // last_resharding dropped
}
```

`next_epoch_info` is epoch T+1's `EpochInfo`. Its `shard_layout()` (if V5) returns the T+1 layout — the layout **before** the split. The fallback stores this as the T+2 epoch info, so `get_shard_layout(&epoch_T2_id)` returns the pre-split layout. Meanwhile, the block header for the last block of epoch T already has `shard_split: Some((shard_id, boundary_account))` committed and broadcast to the network.

The exact divergent Borsh value: `EpochInfoV5.shard_layout` for epoch T+2 is the T+1 `ShardLayout` (e.g., a `ShardLayoutV2` or `ShardLayoutV3` without the new boundary account), instead of the `ShardLayoutV3` that `derive_v3()` would have produced with the new boundary account inserted and new shard IDs allocated.

The `can_reshard()` cooldown check also reads `last_resharding()` from the stored epoch info. Because the fallback clones T+1's info (which has `last_resharding` from T+1, not the newly scheduled T+2 resharding), the cooldown counter is also wrong: the next epoch boundary will see `last_resharding` as if no resharding was scheduled for T+2, potentially allowing a second split one epoch too early.

---

### Impact Explanation

**High.** The shard layout stored in `EpochInfoV5` for epoch T+2 is the authoritative source for all shard-layout-dependent operations: `get_shard_layout()`, `will_shard_layout_change()`, `is_resharding_boundary()`, `get_resharding_parent_shard_uid()`, and the resharding actor's trigger in `start_resharding()`. All of these will use the wrong (pre-split) layout. The resharding that was committed to in the block header is silently skipped. The `last_resharding` cooldown field is also wrong, potentially allowing a back-to-back resharding in the next epoch, which the code explicitly documents as unsafe (`min_epochs_between_resharding` must be `> 0` precisely to prevent this). The network agrees on the wrong state deterministically, so there is no fork — but the protocol invariant "the shard layout stored in epoch T+2's `EpochInfo` must match the layout derived from the `shard_split` committed in the last block of epoch T" is permanently broken for that epoch.

---

### Likelihood Explanation

**Low but reachable at a protocol upgrade boundary.** `ThresholdError` fires when `stake_sum < num_seats` (all or nearly all validators unstake simultaneously). `NotEnoughValidators` fires when there are fewer validator proposals than `minimum_validators_per_shard * num_shards`. Both are reachable without malicious intent on a new network, a testnet, or immediately after the `DynamicResharding` feature activates (when the shard count increases and the validator set may be temporarily undersized for the new shard count). The combination — a dynamic resharding split scheduled at the same epoch boundary where validator selection fails — is the specific trigger. The `DynamicResharding` feature activation is itself a public upgrade-boundary event, making this a protocol-upgrade-boundary interaction rather than a purely operational one.

---

### Recommendation

In the `ThresholdError` and `NotEnoughValidators` fallback arms of `finalize_epoch()`, the fallback epoch info must carry the correct `next_next_shard_layout` and `last_resharding`. The simplest fix is to construct a new `EpochInfo` using `EpochInfo::new()` with the already-computed `next_next_shard_layout` and `last_resharding`, copying the validator set from `next_epoch_info` unchanged (since validator selection failed). Alternatively, if the intent is to carry forward the T+1 validator assignment, the fallback should at minimum update the `shard_layout` and `last_resharding` fields of the cloned info to match the computed values:

```rust
Err(EpochError::ThresholdError { .. }) => {
    // carry forward T+1 validators but use the correct T+2 shard layout
    let mut epoch_info = EpochInfo::clone(&next_epoch_info);
    *epoch_info.epoch_height_mut() += 1;
    // TODO: update shard_layout and last_resharding to next_next_shard_layout / last_resharding
    epoch_info
}
```

Additionally, add an assertion or a test that verifies: when `finalize_epoch()` is called with a non-`None` `block_info.shard_split()`, the stored epoch T+2 info's `shard_layout()` differs from the T+1 layout regardless of whether `proposals_to_epoch_info()` succeeds or falls back.

---

### Proof of Concept

**Trigger path:**

1. Dynamic resharding is enabled (`ProtocolFeature::DynamicResharding` active).
2. At the last block of epoch T, the block producer embeds `shard_split: Some((shard_id, "boundary"))` in `BlockHeaderInnerRestV6`. This is broadcast and accepted by the network.
3. `finalize_epoch()` is called. `next_next_shard_layout()` correctly derives a new `ShardLayoutV3` with the boundary account inserted.
4. `proposals_to_epoch_info()` is called. Because all validators submitted unstake proposals (or the new shard count exceeds the validator count), it returns `EpochError::NotEnoughValidators`.
5. The fallback clones `next_epoch_info` (T+1's `EpochInfoV5`) and increments its epoch height. The `shard_layout` field of the clone is the T+1 layout — no split.
6. This is saved as the T+2 epoch info.

**Divergent Borsh bytes:** The `EpochInfoV5` Borsh encoding for epoch T+2 contains the T+1 `ShardLayout` variant tag and bytes (e.g., `ShardLayoutV2` or `ShardLayoutV3` without the new boundary account), instead of the `ShardLayoutV3` bytes that `derive_v3()` would have produced (new boundary account, two new shard IDs `max+1` and `max+2`, updated `shards_split_map` and `shards_ancestor_map`).

**Observable consequence:** `epoch_manager.get_shard_layout(&epoch_T2_id)` returns the pre-split layout. `will_shard_layout_change()` returns `false` for blocks in epoch T+1. The resharding actor is never triggered. The `can_reshard()` cooldown check for the next epoch sees `last_resharding` from T+1 (not T+2), potentially allowing a split one epoch too early.
