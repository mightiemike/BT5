Let me check the perp case and whether there's a production override of `isHealthy`, plus how `_tryCloseIsolatedSubaccount` interacts with the perp position check.

### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Closure Permanently Locks Funds in Orphaned Subaccount — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the three ownership mappings (`parentSubaccounts`, `isolatedSubaccounts`, `isolatedSubaccountsMask`) but never clears `digestToSubaccount[digest]`. Any subsequent `matchOrders` call carrying the same order digest will silently redirect the taker's sender to the now-closed isolated subaccount, credit it with base or quote balance, and leave those funds permanently unreachable because the parent-link is gone.

---

### Finding Description

**Root cause — missing `digestToSubaccount` cleanup in `_tryCloseIsolatedSubaccount`:**

`_tryCloseIsolatedSubaccount` performs the following cleanup:

```
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [1](#0-0) 

`digestToSubaccount[digest]` is **never touched**. After closure it still maps the original order digest to the now-orphaned isolated subaccount bytes32.

**How `matchOrders` consumes the stale entry:**

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [2](#0-1) 

The replacement is unconditional — there is no liveness check on the isolated subaccount.

**Why `_validateOrder` does not block the re-fill:**

`_validateOrder` checks version, signature, remaining `filledAmounts`, reduce-only logic, and expiration. It contains no guard of the form `parentSubaccounts[order.sender] != bytes32(0)`. [3](#0-2) 

**Why the signature check passes for the closed isolated subaccount:**

`_checkSignature` recovers the ECDSA signer and compares it to `address(uint160(bytes20(subaccount)))`. [4](#0-3) 

The isolated subaccount bytes32 is constructed as `(address << 96) | (productId << 32) | (id << 24) | 6910831`, so its first 20 bytes are identical to the parent's address. [5](#0-4) 

The original signature (signed by the parent's private key) therefore satisfies the check against the closed isolated subaccount — the recovered address is the same.

**Why `isHealthy` does not block the re-fill:**

The base `OffchainExchange.isHealthy` is a virtual stub that unconditionally returns `true`. No override exists in the production codebase. [6](#0-5) 

**Two concrete trigger paths:**

*Path A — Perp isolated subaccount liquidated while order is partially filled:*

1. `createIsolatedSubaccount` → `digestToSubaccount[digest]` = `iso`, perp position opened.
2. Order partially filled → `filledAmounts[digest]` < `order.amount`, perp position ≠ 0.
3. Isolated subaccount goes underwater → `liquidateSubaccountImpl` → `_finalizeSubaccount` zeroes the perp position → calls `tryCloseIsolatedSubaccount(iso)`.
4. `_tryCloseIsolatedSubaccount`: `perpEngine.getBalance(productId, iso).amount == 0` → closure proceeds, `parentSubaccounts[iso]` = 0, `digestToSubaccount[digest]` **unchanged**.
5. Sequencer processes remaining fill → `matchOrders` → `taker.order.sender` = `iso` → `_updateBalances` credits `iso` with perp base/quote → `filledAmounts[digest]` updated → funds stuck. [7](#0-6) 

*Path B — Spot isolated subaccount closed while order is partially filled:*

For a spot product, `perpEngine.getBalance(spotProductId, iso).amount` is always 0 (spot products are not tracked by the perp engine). Therefore `_tryCloseIsolatedSubaccount` will close the subaccount even when the spot base balance is non-zero from a partial fill. The closure only sweeps the USDC quote balance back to the parent; the spot base balance is already stranded. Any subsequent re-fill via the stale `digestToSubaccount` entry adds more base balance to the orphaned subaccount. [8](#0-7) 

**State after the exploit:**

| Storage variable | Value |
|---|---|
| `digestToSubaccount[digest]` | `iso` (stale, non-zero) |
| `parentSubaccounts[iso]` | `bytes32(0)` |
| `isolatedSubaccounts[parent][id]` | `bytes32(0)` |
| `spotEngine.balance[iso]` or `perpEngine.balance[iso]` | non-zero (locked) |

The parent has no pointer to `iso`, so the funds cannot be withdrawn, transferred, or recovered through any normal protocol path. A second call to `_tryCloseIsolatedSubaccount(iso)` would read `parent = parentSubaccounts[iso] = bytes32(0)` and transfer balances to the zero-address subaccount, not the real owner.

---

### Impact Explanation

User funds (spot base tokens or perp quote balance) are permanently locked in an orphaned isolated subaccount with no reachable parent. The invariant that every isolated subaccount balance is reachable by its parent is broken. There is no recovery path in the protocol.

---

### Likelihood Explanation

The sequencer processes orders from an order book and may not cancel remaining fills when an isolated subaccount is closed or liquidated. The liquidation path (Path A) is particularly realistic: liquidation is triggered by market conditions, not by the user, and the sequencer's order book may still hold the partially-filled order. Path B can be triggered by a user who closes their isolated subaccount believing the order is cancelled. Neither path requires any privileged access beyond normal user interaction.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, iterate over all digests associated with the subaccount and clear them, or maintain a reverse mapping from isolated subaccount to its digest(s) so they can be deleted on closure. At minimum, add a guard in `matchOrders` that rejects fills routed to an isolated subaccount whose `parentSubaccounts` entry is `bytes32(0)`:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    bytes32 iso = digestToSubaccount[ordersInfo.taker.digest];
    require(parentSubaccounts[iso] != bytes32(0), ERR_INVALID_TAKER);
    taker.order.sender = iso;
}
```

The same guard should be applied to the maker side.

### Proof of Concept

```solidity
// 1. Parent subaccount creates an isolated perp order
bytes32 digest = offchainExchange.getDigest(productId, order);
// createIsolatedSubaccount sets digestToSubaccount[digest] = iso

// 2. Partial fill: filledAmounts[digest] = order.amount / 2
// iso now holds a non-zero perp position

// 3. Liquidation zeroes the perp position and calls tryCloseIsolatedSubaccount(iso)
// parentSubaccounts[iso] == bytes32(0) after this call
// digestToSubaccount[digest] still == iso

// 4. Sequencer submits matchOrders with the same taker order
// matchOrders: taker.order.sender = digestToSubaccount[digest] = iso (closed!)
// _validateOrder: signature check passes (same address), amount remaining > 0
// _updateBalances: credits iso with base/quote delta
// isHealthy(iso): returns true (stub)
// filledAmounts[digest] updated

// Assert: iso has non-zero balance, parentSubaccounts[iso] == bytes32(0)
assert(spotEngine.getBalance(productId, iso).amount != 0);
assert(offchainExchange.getParentSubaccount(iso) == bytes32(0));
// Funds are permanently locked — no withdrawal or transfer path exists
```

        if ((signedOrder.order.appendix & 255) != orderVersion()) {

        }

            return true;

                return false;

                    linkedSigner,
                    signedOrder.signature
                )) &&

            (order.amount != 0) &&
