### Title
Caller Can Bypass Coinbase Merkle Proof Check by Invoking Deprecated `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function remains publicly callable by any unprivileged NEAR account. By invoking it directly instead of `verify_transaction_inclusion_v2`, a caller bypasses the coinbase Merkle proof check that guards against the 64-byte transaction Merkle proof forgery vulnerability. This is a direct analog to the external report: just as a dapp could set `displayMessage = false` to suppress the security check, any NEAR caller can select the unguarded entry point to suppress the coinbase verification step, causing the contract to return `true` for a forged proof.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability (documented in the function's own NatSpec comment referencing the BitMEX blog post). It does so by requiring a coinbase Merkle proof at position 0: [1](#0-0) 

The coinbase check enforces that the proof path anchors to a known-valid leaf (the coinbase transaction), preventing an attacker from substituting an internal Merkle node hash as `tx_id`.

However, the original `verify_transaction_inclusion` function — which contains **no coinbase check** — is still a live, public, unpausable-by-default contract method: [2](#0-1) 

The only guard in the deprecated path is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

There is no check that `tx_id` is a leaf node rather than an internal node. `compute_root_from_merkle_proof` is purely positional: [4](#0-3) 

An attacker who supplies an internal node hash as `tx_id` with a correctly-positioned sibling proof will produce the correct Merkle root, and the function returns `true`.

The caller's ability to choose between the two entry points is the toggle: calling `verify_transaction_inclusion` instead of `verify_transaction_inclusion_v2` is functionally equivalent to setting `displayMessage = false` in the external report — it suppresses the security check entirely.

---

### Impact Explanation

Any NEAR smart contract that calls `verify_transaction_inclusion` to gate a privileged action (releasing bridged funds, minting wrapped BTC, confirming a cross-chain swap) can be deceived into accepting a forged Bitcoin transaction inclusion proof. The attacker does not need to produce a real Bitcoin transaction; they only need to identify an internal Merkle node in a block already accepted by the light client and construct a valid sibling proof for it. The contract will return `true`, and the downstream contract will execute its guarded action against a transaction that never existed.

---

### Likelihood Explanation

All information required for the attack (block Merkle tree structure, internal node hashes, sibling paths) is publicly available from any Bitcoin block explorer. No privileged access, key material, or social engineering is required. The attacker only needs to call a public NEAR contract method with crafted arguments. Likelihood is **Medium-High** for any deployment where a downstream contract consumes `verify_transaction_inclusion` results.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely. Because `verify_transaction_inclusion_v2` already delegates to it internally after performing the coinbase check, the deprecated function serves no purpose that cannot be served by the v2 path. Removing it eliminates the caller-controlled bypass of the coinbase guard, mirroring the external report's recommendation to remove the `displayMessage` toggle and consistently enforce the security check.

---

### Proof of Concept

Assume a Bitcoin block already accepted by the contract contains four transactions: `T0, T1, T2, T3`.

```
Level 2 (root): merkle_root = hash(H01, H23)
Level 1:        H01 = hash(T0, T1),  H23 = hash(T2, T3)
Level 0 (leaf): T0, T1, T2, T3
```

**Attack call** (any unprivileged NEAR account):

```json
{
  "tx_id":             "<H01>",
  "tx_block_blockhash":"<accepted block hash>",
  "tx_index":          0,
  "merkle_proof":     ["<H23>"],
  "confirmations":     1
}
```

**Execution trace inside `compute_root_from_merkle_proof`:**

```
current_hash     = H01          (attacker-supplied internal node)
current_position = 0
proof_hash       = H23
  → position 0 is even → current_hash = hash(H01, H23) = merkle_root
```

`merkle_root == header.block_header.merkle_root` → function returns **`true`**.

`H01` is not a transaction; it is an internal Merkle node. The proof is forged. Any downstream contract acting on this `true` result is deceived. [5](#0-4) [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
