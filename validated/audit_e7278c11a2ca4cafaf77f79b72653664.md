### Title
`cumulativeFundingLongX18` Inflated to Overflow via Socialization Against Tiny `openInterest` — (`core/contracts/PerpEngine.sol`)

---

### Summary

When a perp subaccount is socialized during liquidation finalization, `fundingPerShare` is computed by dividing the bad-debt `vQuoteBalance` by `state.openInterest`. If `openInterest` is very small (e.g., 1 wei in X18 units), `fundingPerShare` becomes enormous and is permanently added to `cumulativeFundingLongX18` / `cumulativeFundingShortX18`. Any subsequent call to `_updateBalance` for a user with a non-trivial position will then overflow inside `diffX18.mul(balance.amount)`, permanently bricking balance updates for that perp market.

---

### Finding Description

**Root cause — `PerpEngine.socializeSubaccount`:** [1](#0-0) 

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest
);
state.cumulativeFundingLongX18 += fundingPerShare;
state.cumulativeFundingShortX18 -= fundingPerShare;
```

`MathSD21x18.div(x, y)` computes `(int256(x) * 1e18) / y`. When `state.openInterest = 1` (one wei of base asset in X18 representation), the result is `int256(vQuoteBalance) * 1e18`. For a bad-debt `vQuoteBalance` of even −10 units (−1e19 in X18), `fundingPerShare = 1e37`, which fits inside `int128` (max ≈ 1.7e38) and is silently accepted. [2](#0-1) 

**Overflow site — `PerpEngineState._updateBalance`:** [3](#0-2) 

```solidity
int128 cumulativeFundingAmountX18 = (balance.amount > 0)
    ? state.cumulativeFundingLongX18
    : state.cumulativeFundingShortX18;
int128 diffX18 = cumulativeFundingAmountX18 -
    balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
```

`diffX18.mul(balance.amount)` calls `MathSD21x18.mul`, which computes `(int256(diffX18) * balance.amount) / 1e18`. With `diffX18 = 1e37` and `balance.amount = 1e20` (100 units of base asset), the intermediate product is `1e57 / 1e18 = 1e39 > 1.7e38`, triggering the `ERR_OVERFLOW` revert. [4](#0-3) 

**Trigger path — `ClearinghouseLiq._finalizeSubaccount`:** [5](#0-4) 

`socializeSubaccount` is called only after the liquidatee's perp `amount` has been verified to be zero (lines 313–320), so the liquidatee no longer contributes to `openInterest`. If an attacker holds the only remaining position (1 wei), `openInterest = 1` at the moment of socialization. [6](#0-5) 

---

### Impact Explanation

Once `cumulativeFundingLongX18` is inflated, every call that reaches `_updateBalance` for any user with a position larger than roughly `1.7e38 / fundingPerShare` base-asset units will revert with `ERR_OVERFLOW`. This permanently prevents:

- Any trade or order fill (`updateBalance` in `PerpEngine`)
- Any liquidation of other accounts in that market
- Any PnL settlement (`settlePnl`)
- Any `getStateAndBalance` / `getBalance` view that internally calls `_updateBalance`

The entire perp market is bricked for all participants.

---

### Likelihood Explanation

An unprivileged attacker can engineer this condition:

1. Open a perp position of 1 wei in Account B (minimum possible, sets their contribution to `openInterest = 1`).
2. Open a large leveraged perp position in Account A (same or different wallet) and allow it to accumulate bad debt (e.g., via adverse price movement or in a low-liquidity market).
3. Ensure no other participants hold positions in that market at the time of finalization (feasible in a new or low-activity market).
4. Trigger liquidation and finalization of Account A (`productId = type(uint32).max`).
5. `socializeSubaccount` is called with `openInterest = 1`, inflating `cumulativeFundingLongX18`.

The attacker does not need any privileged access. The only cost is the capital to open the positions and absorb the bad debt, which is recoverable if the attacker controls both accounts.

---

### Recommendation

Mirror the Sophon fix: enforce a minimum `openInterest` floor before performing the division, or cap `fundingPerShare` to a safe maximum value. Concretely, in `PerpEngine.socializeSubaccount`:

```solidity
// Require openInterest to be above a meaningful minimum before socializing
require(state.openInterest >= MIN_OPEN_INTEREST, ERR_INSUFFICIENT_OPEN_INTEREST);
int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
```

Alternatively, clamp `fundingPerShare` so that `fundingPerShare * maxExpectedPosition` cannot exceed `int128` bounds, and add a corresponding invariant check in `_updateBalance`.

---

### Proof of Concept

```
Setup:
  - Perp market with openInterest = 0
  - Attacker Account B: opens position of 1 wei → openInterest = 1
  - Attacker Account A: opens large leveraged long, accumulates vQuoteBalance = -1e19 (bad debt)
  - No other participants in this perp market

Step 1: Liquidate Account A to zero perp amount (standard liquidation steps)
Step 2: Call liquidateSubaccount with productId = type(uint32).max (finalize)
  → _finalizeSubaccount verifies balance.amount == 0 ✓
  → perpEngine.socializeSubaccount(AccountA, insurance) is called
  → openInterest = 1 (only Account B's 1-wei position)
  → fundingPerShare = 1e19 * 1e18 / 1 = 1e37  (fits int128)
  → cumulativeFundingLongX18 += 1e37

Step 3: Any user with balance.amount >= ~17 units (1.7e19 in X18) calls any
        function that triggers _updateBalance:
  → diffX18 = 1e37
  → diffX18.mul(1.7e19) = (1e37 * 1.7e19) / 1e18 = 1.7e38 → borderline overflow
  → diffX18.mul(2e19) = 2e38 > 1.7e38 → ERR_OVERFLOW revert

Result: All balance updates, trades, liquidations, and PnL settlements
        for this perp market permanently revert.
```
