Audit Report

## Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`, `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no check against `maxAdminSpreadFeeE6`, while the sibling entrypoint `setPoolAdminFees` enforces that cap. A pool admin can set per-bin spread fees up to `type(uint16).max` (65 535 ≈ 6.5535%) on any bin, bypassing whatever lower cap the factory owner has configured. These values are added directly to `baseFeeX64` in every swap execution path, causing traders to pay excess fees beyond the intended policy limit.

## Finding Description
`setPoolAdminFees` enforces the cap at lines 414–415: [1](#0-0) 

The sibling entrypoint `setPoolBinAdditionalFees` at lines 450–457 performs no such check — it forwards values straight to the pool: [2](#0-1) 

`setBinAdditionalFees` in the pool at lines 464–474 only validates the bin index range, not the fee magnitude: [3](#0-2) 

The stored `addFeeBuyE6`/`addFeeSellE6` values are added directly to `baseFeeX64` in both swap directions: [4](#0-3) [5](#0-4) 

They also affect the quoted bid/ask prices returned by `getSellAndBuyPrices`: [6](#0-5) 

The hard cap constant is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%), but `maxAdminSpreadFeeE6` can be set lower by the factory owner: [7](#0-6) 

Since `addFeeBuyE6`/`addFeeSellE6` are `uint16`, their maximum is 65 535, which can exceed any `maxAdminSpreadFeeE6` value the factory owner sets below that threshold.

## Impact Explanation
This is a direct admin-boundary break: the pool admin exceeds the fee cap the factory owner is entitled to enforce. Traders swapping through an affected bin pay up to ≈6.5535% additional spread fee beyond the configured cap, resulting in direct loss of trader principal. The inflated fee is reflected consistently in `getSellAndBuyPrices`, so price-limit slippage protection does not prevent the swap from executing at the inflated rate.

## Likelihood Explanation
Any pool admin can call `setPoolBinAdditionalFees` at any time in a single transaction with no preconditions beyond holding the `poolAdmin[pool]` role. The factory owner's `maxAdminSpreadFeeE6` provides zero protection on this path. The bypass is unconditional and repeatable.

## Recommendation
Add the cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

## Proof of Concept
```solidity
function test_binFeeBypassesAdminCap() public {
    // Factory owner lowers the admin spread cap to 1% (10_000 in E6)
    factory.setFeeCaps(200_000, 10_000, 1_000_000, 1_000_000);

    // Pool admin sets per-bin fee to uint16.max (65535 ≈ 6.5535%)
    vm.prank(admin);
    factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, type(uint16).max);

    // Stored value exceeds maxAdminSpreadFeeE6
    assertGt(uint256(type(uint16).max), factory.maxAdminSpreadFeeE6()); // 65535 > 10000

    // Swap through bin 0 and measure effective fee exceeds the cap
    // ... swap call, compare amountIn vs amountOut vs oracle mid-price ...
}
```
