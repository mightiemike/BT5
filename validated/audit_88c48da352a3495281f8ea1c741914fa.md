### Title
NLP Holders Cannot Exit Their Position Without Sequencer Cooperation — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`BurnNlp` is absent from `processSlowModeTransactionImpl`, so NLP holders have no on-chain self-service exit path. The slow mode system silently accepts and charges for `BurnNlp` submissions but drops them on execution, permanently locking NLP holders into their positions unless the sequencer cooperates.

---

### Finding Description

The Nado protocol provides a slow mode queue (`submitSlowModeTransaction` → `executeSlowModeTransaction`) as an emergency, sequencer-independent path for users to perform critical operations. `WithdrawCollateral`, `LinkSigner`, and `ClaimBuilderFee` are all handled in `processSlowModeTransactionImpl`, giving users a trustless exit for those actions.

However, `BurnNlp` — the only mechanism for an NLP holder to redeem their position and recover underlying quote tokens — is **not handled** in `processSlowModeTransactionImpl`.

The bug has two compounding parts:

**Part 1 — Fee charged, transaction queued:**
In `submitSlowModeTransactionImpl`, `BurnNlp` is not in the admin-only guard list (lines 355–367), so it falls through to the `else` branch:

```solidity
} else {
    chargeSlowModeFee(_getQuote(), sender);
    slowModeFees += SLOW_MODE_FEE;
}
```

The slow mode fee is deducted from the user and the transaction is enqueued. [1](#0-0) 

**Part 2 — Silent failure on execution:**
When the queued transaction is later executed, `processSlowModeTransactionImpl` is called. `BurnNlp` matches none of the handled cases and hits the terminal `else { revert(); }`:

```solidity
} else {
    revert();
}
``` [2](#0-1) 

This revert is silently swallowed by the `try/catch` in `_executeSlowModeTransaction`:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch { ... }
``` [3](#0-2) 

The net result: the user's slow mode fee is permanently lost, the `BurnNlp` transaction is discarded, and the NLP position is unchanged.

By contrast, `BurnNlp` **is** handled in `processTransactionImpl` (the sequencer-only path): [4](#0-3) 

This means the only functional path to burn NLP is through the sequencer. There is no on-chain fallback.

---

### Impact Explanation

NLP holders are locked into their positions without sequencer cooperation. If the sequencer is unavailable, unresponsive, or censoring a specific user, that user has no on-chain mechanism to exit their NLP position and recover their underlying quote collateral. Any attempt to use the slow mode emergency path results in a concrete asset loss (slow mode fee) with zero effect on the NLP position.

The corrupted state delta is:
- `spotEngine` balance for `sender` is not updated (no NLP burn, no quote credit)
- `slowModeFees` is incremented (fee taken with no service rendered)

---

### Likelihood Explanation

Any NLP holder who attempts to use the slow mode path — the protocol's documented emergency exit mechanism — will trigger this. The scenario is realistic whenever the sequencer is slow, congested, or a user believes they are being censored. The user has no way to know in advance that `BurnNlp` will silently fail in slow mode; the submission step succeeds and charges a fee, giving false confidence.

---

### Recommendation

Add a `BurnNlp` handler inside `processSlowModeTransactionImpl` analogous to the existing `WithdrawCollateral` handler. The handler should decode the `BurnNlp` struct, validate that `txn.sender` matches `sender` via `validateSender`, and call `clearinghouse.burnNlp(...)` with an appropriate oracle price. Alternatively, if a trustless oracle price cannot be provided in slow mode, add `BurnNlp` to the admin-only guard in `submitSlowModeTransactionImpl` to prevent the fee from being charged for a transaction that will never execute. [5](#0-4) 

---

### Proof of Concept

1. Alice holds NLP tokens (has a non-zero NLP balance in `SpotEngine`).
2. The sequencer becomes unresponsive. Alice wants to exit.
3. Alice calls `Endpoint.submitSlowModeTransaction(burnNlpTx)` where `burnNlpTx[0] == uint8(TransactionType.BurnNlp)`.
4. In `submitSlowModeTransactionImpl`: `BurnNlp` is not in the admin-only list → `chargeSlowModeFee` deducts `SLOW_MODE_FEE` from Alice's quote balance → transaction is enqueued. [6](#0-5) 
5. After `SLOW_MODE_TX_DELAY` (3 days), Alice or anyone calls `executeSlowModeTransaction()`.
6. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(txn.sender, txn.tx)` inside a `try/catch`.
7. `processSlowModeTransactionImpl` receives the `BurnNlp` transaction, finds no matching branch, and executes `revert()`.
8. The `catch` block in `_executeSlowModeTransaction` silently absorbs the revert (gas check passes).
9. Alice's slow mode fee is gone. Her NLP position is unchanged. She remains locked in with no further recourse. [2](#0-1)
