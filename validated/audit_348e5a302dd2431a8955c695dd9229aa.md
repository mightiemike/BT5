### Title
Unsafe `transferFrom()` Without Return Value Check Enables Potential Token Drain in `replaceUsdcEWithUsdc` — (`core/contracts/ContractOwner.sol`)

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` uses a raw `IERC20Base(usdc).transferFrom()` call at line 616 without checking its return value. If the USDC token on Ink chain (chain ID 57073) returns `false` on failure rather than reverting, the function continues execution, withdrawing `usdcE` from the `directDepositV1` contract and sending it to the caller — without the caller having provided any USDC.

### Finding Description

The function `replaceUsdcEWithUsdc` is designed as a token-swap helper: it pulls USDC from `msg.sender` into a `directDepositV1` contract, then withdraws the `usdcE` held there and returns it to `msg.sender`. The critical step is:

```solidity
// ContractOwner.sol line 616
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

The `transferFrom` at line 616 is a raw call — its boolean return value is silently discarded. If it returns `false` (i.e., the caller has no USDC or insufficient allowance), execution does not revert. The subsequent `withdraw` and `safeTransfer` calls still execute, draining `usdcE` from `directDepositV1` and sending it to the caller for free.

This is in direct contrast to the rest of the codebase, which consistently uses `ERC20Helper.safeTransferFrom` — the protocol's own safe wrapper that checks both the call success and the decoded boolean return value. [1](#0-0) 

The `ERC20Helper` library, used everywhere else, properly handles non-reverting tokens: [2](#0-1) 

`EndpointStorage` and `BaseWithdrawPool` both use `safeTransferFrom` consistently: [3](#0-2) [4](#0-3) 

### Impact Explanation

If the hardcoded USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain (chain ID 57073) returns `false` on a failed transfer rather than reverting (as USDT and some bridged tokens do), any caller can:

1. Call `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose `directDepositV1` holds a non-zero `usdcE` balance.
2. The `transferFrom` silently fails (caller provides no USDC).
3. `DirectDepositV1.withdraw(usdcE)` executes, moving the full `usdcE` balance to `ContractOwner`.
4. `safeTransfer(msg.sender, balance)` sends the `usdcE` to the attacker.

The `directDepositV1` contract loses its entire `usdcE` balance. The corrupted asset delta is the full `usdcE` balance of the targeted `directDepositV1` contract.

### Likelihood Explanation

The function has no access control beyond `require(block.chainid == 57073)`, making it callable by any unprivileged user on Ink chain. The exploitability depends on whether the bridged USDC at the hardcoded address returns `false` on failure or reverts. Bridged/wrapped USDC variants frequently deviate from the standard, and the protocol's own `ERC20Helper` was written precisely to handle this class of token. The inconsistency — using safe wrappers everywhere except this one call — is a concrete root cause, not a theoretical one.

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`, so no additional imports are needed. [5](#0-4) 

### Proof of Concept

1. A subaccount `S` has a deployed `directDepositV1` at address `D` holding 1000 `usdcE`.
2. Attacker `A` (with zero USDC balance and zero allowance) calls `ContractOwner.replaceUsdcEWithUsdc(S)` on Ink chain.
3. `balance = IERC20Base(usdcE).balanceOf(D)` → 1000.
4. `IERC20Base(usdc).transferFrom(A, D, 1000)` → returns `false` (A has no USDC), but **no revert occurs**.
5. `DirectDepositV1(D).withdraw(usdcE)` → transfers 1000 `usdcE` from `D` to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(A, 1000)` → transfers 1000 `usdcE` to attacker `A`.
7. Attacker receives 1000 `usdcE` at zero cost; `D` is drained. [6](#0-5)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
