### Title
Linked Signer Can Reassign Itself, Preventing Revocation — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path transaction handler in `EndpointTx.sol` validates the transaction with `allowLinkedSigner: true`, meaning the **currently linked signer** is permitted to sign a new `LinkSigner` transaction. This gives the linked signer the ability to reassign itself to any address it controls, making it impossible for the subaccount owner to permanently revoke a compromised linked signer.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx(..., true)` resolves the valid signer as either the subaccount owner **or** the current linked signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

This means the linked signer itself can submit a valid `LinkSigner` transaction to overwrite `linkedSigners[subaccount]` with a new address it controls. The subaccount owner's attempt to revoke the linked signer (by setting `signer = address(0)`) can be front-run at the sequencer level: the compromised linked signer monitors the sequencer queue and submits its own `LinkSigner` transaction pointing to a fresh address before the owner's revocation is sequenced.

By contrast, the slow-mode `LinkSigner` path correctly uses `validateSender(txn.sender, sender)`, which enforces that only the on-chain `msg.sender` (the subaccount owner's wallet) can update the linked signer:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The fast-path and slow-path diverge in exactly this authorization check, and the fast-path is the one that is exploitable.

---

### Impact Explanation

A linked signer is a delegated key that can sign sensitive fast-path transactions on behalf of a subaccount (orders, transfers, liquidations, and — critically — further `LinkSigner` updates). If a linked signer key is compromised, the subaccount owner has no reliable way to revoke it: every revocation attempt can be countered by the compromised key re-linking to a fresh address it controls. The attacker retains indefinite signing authority over the subaccount and can drain it via signed `WithdrawCollateral`, `TransferQuote`, or order-placement transactions.

---

### Likelihood Explanation

Linked signers are explicitly designed for programmatic/API use (bots, market makers, automated strategies). API keys are a common compromise vector. Once a linked signer key leaks, the owner's only recourse — submitting a revocation through the sequencer — is raceable by the attacker using the same sequencer API. No privileged access beyond the already-granted linked signer role is required to sustain the attack.

---

### Recommendation

Change the `allowLinkedSigner` flag to `false` for `LinkSigner` transactions in the fast path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the slow-mode path's owner-only enforcement and ensures that only the subaccount owner's key can install or revoke a linked signer, eliminating the self-reassignment loop.

---

### Proof of Concept

1. Alice owns subaccount `S` and has linked signer `L` (an API key) via a prior `LinkSigner` transaction.
2. `L`'s private key is leaked. Alice submits a fast-path `LinkSigner` transaction signed by her owner key, setting `signer = address(0)` to revoke `L`.
3. Before the sequencer processes Alice's transaction, the attacker (holding `L`) submits their own fast-path `LinkSigner` transaction signed by `L`, setting `signer = L2` (another attacker-controlled address). Because `allowLinkedSigner: true`, this signature is accepted.
4. The sequencer processes the attacker's transaction first (or interleaved). `linkedSigners[S]` is now `L2`.
5. Alice's revocation transaction executes next, setting `linkedSigners[S] = address(0)`.
6. The attacker immediately repeats step 3 using `L2`, re-linking to `L3`. This loop continues indefinitely.
7. At any point during the loop, the attacker uses the active linked signer to sign a `WithdrawCollateral` or `TransferQuote` transaction, draining `S`. [1](#0-0) [4](#0-3)
