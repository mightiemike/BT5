Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to every configured extension. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end user. `SwapAllowlistExtension.beforeSwap` checks this forwarded `sender` against the per-pool allowlist, so it evaluates the router's address rather than the real economic actor, completely defeating the allowlist's intended access control.

## Finding Description
`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 230–240, passing its direct caller as `sender`. `ExtensionCalling._beforeSwap()` (lines 160–176) forwards this value unchanged via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`. `SwapAllowlistExtension.beforeSwap` (lines 37–38) then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the forwarded value.

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (line 72), the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly, making the router the pool's `msg.sender`. The extension therefore receives `sender = address(router)`, not the actual user.

**Bypass path (Impact A):** A pool admin who wants to support router-based swaps must allowlist the router address. Once `allowedSwapper[pool][router] == true`, any address — including addresses the admin explicitly excluded — can call `router.exactInputSingle()` and pass the `beforeSwap` guard, because the extension only sees the router. The curated pool's swap restriction is completely defeated.

**Lockout path (Impact B):** If the admin allowlists only individual user addresses (not the router), those users cannot swap through the router at all, because the extension sees the router address and reverts `NotAllowedToSwap`. This breaks the core swap flow for the intended participants.

No existing guard compensates for this: the extension has no mechanism to retrieve the original initiator, and the pool exposes no transient `swapInitiator()` getter. The `_setNextCallbackContext` in the router stores `msg.sender` for the payment callback only — it is not accessible to extensions.

## Impact Explanation
Impact A constitutes a broken core pool functionality and admin-boundary break: an unprivileged trader bypasses a pool admin's explicit access control and executes swaps in a curated pool from which they were excluded, receiving output tokens they are not entitled to. This is a direct fund-impacting outcome (the pool's liquidity is consumed by unauthorized parties) meeting the High severity threshold. Impact B renders the primary swap entrypoint unusable for allowlisted users, constituting broken core swap functionality at Medium severity.

## Likelihood Explanation
No special permissions, malicious setup, or non-standard tokens are required. Any pool that deploys `SwapAllowlistExtension` and expects users to swap through `MetricOmmSimpleRouter` will encounter this mismatch on the first router-based swap. The attack requires only a standard `exactInputSingle` call. The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` (lines 99–125, 130–147, 154–188). The condition is trivially reachable and repeatable.

## Recommendation
1. **Pool-side (primary fix):** Add an optional `initiator` parameter to `swap()` (defaulting to `msg.sender`) so the router can pass the real user. Alternatively, write the initiator to a transient storage slot before calling extensions, mirroring the existing `inSwap()` / `_currentAction()` pattern.
2. **Router-side:** `MetricOmmSimpleRouter` already stores the real `msg.sender` in `_setNextCallbackContext` for the payment callback. Expose this same value to extensions via a transient slot that `SwapAllowlistExtension` can query directly.
3. **Extension-side (short-term):** `SwapAllowlistExtension.beforeSwap` should query a pool-exposed `swapInitiator()` getter rather than relying on the `sender` argument, or maintain a separate allowlist for trusted routers that then enforces per-user checks off-chain.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, router, true)   // enable router path
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls router.exactInputSingle({pool: pool, ...})
    → router calls pool.swap(recipient, ...)           // router is pool's msg.sender
    → pool calls _beforeSwap(address(router), ...)
    → extension checks allowedSwapper[pool][router] == true  ✓
    → swap executes — attacker bypasses the allowlist

Expected:
  extension should check allowedSwapper[pool][attacker] == false → revert NotAllowedToSwap

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
  2. Pool admin calls setAllowedToSwap(pool, address(router), true).
  3. Assert attacker (not allowlisted) can call router.exactInputSingle() and receive output tokens.
  4. Assert attacker calling pool.swap() directly reverts NotAllowedToSwap.
```
