Audit Report

## Title
Router Identity Substitution Bypasses SwapAllowlistExtension Per-User Gate — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates, `sender` is always the router address, not the original user. A pool admin who allowlists the router to enable router-mediated swaps for their whitelist inadvertently opens the pool to every caller of the router.

## Finding Description
`MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap` (MetricOmmPool.sol L231). `ExtensionCalling._beforeSwap` forwards that value unchanged to the extension (ExtensionCalling.sol L165). `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]` (SwapAllowlistExtension.sol L37). The extension ignores the `bytes calldata` parameter entirely and never reads the original user identity. Both `exactInputSingle` and `exactInput` call `pool.swap()` directly from the router contract (MetricOmmSimpleRouter.sol L72–80, L104–112), so the pool always sees `msg.sender = router`. If the admin calls `setAllowedToSwap(pool, router, true)` to allow whitelisted users to reach the pool via the router, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, regardless of whether they are on the per-user whitelist.

## Impact Explanation
Any unprivileged user can swap in a pool whose admin intended to restrict swaps to a specific whitelist, by routing through `MetricOmmSimpleRouter`. The per-user allowlist invariant is broken: the extension cannot distinguish the original user from any other user once the router is allowlisted. This constitutes a broken core access-control / admin-boundary break under the contest rules.

## Likelihood Explanation
A pool admin who wants whitelisted users to be able to use the router will naturally call `setAllowedToSwap(pool, router, true)`. The code and documentation give no indication that doing so opens the pool to all users. The scenario requires no privileged attacker capability beyond calling the public router functions.

## Recommendation
The extension must verify the original user, not the intermediary. Options:
- Have the router encode the original `msg.sender` into `extensionData` and have the extension decode and verify it against a trusted-router registry.
- Store the original caller in transient storage inside the pool and expose it to extensions as a separate `originator` argument.
- Document clearly that allowlisting the router is equivalent to `setAllowAllSwappers(pool, true)`, so admins can make an informed choice.

## Proof of Concept
```
1. Pool deployed with SwapAllowlistExtension; allowedSwapper[pool][alice] = true.
2. Admin calls setAllowedToSwap(pool, router, true) to let Alice use the router.
3. Bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...}).
4. Router calls pool.swap(...); pool passes msg.sender=router to _beforeSwap.
5. Extension checks allowedSwapper[pool][router] == true → passes.
6. Bob's swap executes in a pool he was never meant to access.
```
