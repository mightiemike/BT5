### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently downgrades `AllResources` to `L1Gas` when `l2_gas=0` and `l1_data_gas=0`, producing a divergent transaction hash during P2P sync — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary
The protobuf-to-`ValidResourceBounds` conversion uses a value-based heuristic to distinguish pre-0.13.3 (`L1Gas`) from post-0.13.3 (`AllResources`) transactions. When a V3 transaction carries `AllResources` with zero `l2_gas` and zero `l1_data_gas`, the heuristic silently downgrades it to `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** for each variant (3 for `L1Gas`, 4 for `AllResources`), the recomputed hash diverges from the hash computed at submission time, breaking hash validation for any syncing node that processes such a transaction.

---

### Finding Description

**Root cause — protobuf conversion:**

In `crates/apollo_protobuf/src/converters/transaction.rs` lines 417–436, the conversion from `protobuf::ResourceBounds` to `ValidResourceBounds` uses a zero-value heuristic:

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        // ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)   // ← silently downgrades AllResources
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
``` [1](#0-0) 

The intent is to handle pre-0.13.3 transactions (which only carry L1 gas and serialize without `l1_data_gas`). However, the heuristic also fires for valid V3 (`AllResources`) transactions where the user legitimately sets `l2_gas=0` and `l1_data_gas=0` — for example, a transaction that only consumes L1 gas.

**Divergent hash — two different Poseidon preimages:**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` lines 188–210 produces structurally different outputs for the two variants:

```rust
// L1Gas path: 3 elements
resource_felts = [L1_GAS, L2_GAS=0]

// AllResources path (l2_gas=0, l1_data_gas=0): 4 elements
resource_felts = [L1_GAS, L2_GAS=0, L1_DATA_GAS=0]
``` [2](#0-1) 

The final hash is `poseidon(tip, ...resource_felts)`. A 3-element and a 4-element Poseidon chain produce cryptographically distinct values even when the extra element is zero.

**Submission path always uses `AllResources`:**

`InternalRpcInvokeTransactionV3` and `InternalRpcDeclareTransactionV3` both store `AllResourceBounds` (not `ValidResourceBounds`) and always wrap it as `ValidResourceBounds::AllResources` in their `TransactionHasher` implementations:

```rust
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)  // always AllResources
    }
``` [3](#0-2) 

So the gateway always computes the hash using the 4-element chain. The downgrade only occurs during protobuf deserialization in the P2P sync path, which uses the `Transaction` type (not `RpcTransaction`).

**Separate protobuf converter for RPC/consensus path is unaffected:**

The mempool/consensus path uses a different converter that always produces `AllResourceBounds`:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(...)?.try_into()?,
        })
    }
}
``` [4](#0-3) 

The vulnerable `ValidResourceBounds` converter is used exclusively in the `Transaction` (historical block sync) path.

---

### Impact Explanation

A V3 transaction submitted with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway and its hash H1 is computed using the 4-element Poseidon chain. When this transaction is later serialized to protobuf and deserialized by a syncing peer via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, it is reconstructed as `L1Gas`. The peer recomputes hash H2 using the 3-element chain. H1 ≠ H2, so hash validation fails and the peer rejects the block containing this transaction.

This matches: **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

Any user can submit a V3 transaction with only `l1_gas` non-zero (`l2_gas=0`, `l1_data_gas=0`). The gateway's stateless validator accepts such transactions — the test suite explicitly validates `valid_l1_gas` (only l1_gas non-zero) as a passing case. [5](#0-4) 

The divergence is deterministic and reproducible for every such transaction. No privileged access is required.

---

### Recommendation

The protobuf `ResourceBounds` message does not encode the `ValidResourceBounds` variant tag. Fix options:

1. **Add an explicit variant field** to the protobuf `ResourceBounds` message (e.g., `bool is_all_resources`) to distinguish `L1Gas` from `AllResources` without relying on zero-value heuristics.
2. **Derive the variant from the transaction version**: V3 transactions always use `AllResources`; pre-V3 transactions use `L1Gas`. Pass the transaction version into the converter.
3. **Remove the downgrade entirely**: Always deserialize as `AllResources` when `l1_data_gas` is present in the protobuf message (even if zero), and only use `L1Gas` when `l1_data_gas` is absent (the pre-0.13.2 case the TODO comment refers to).

---

### Proof of Concept

1. Submit a V3 invoke transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
2. Gateway computes:
   ```
   H1 = poseidon(INVOKE, version, sender,
                 poseidon(tip, L1_GAS_packed, L2_GAS_packed=0, L1_DATA_GAS_packed=0),
                 ...)
   ```
   (4-element resource hash, `AllResources` path)
3. Transaction is included in a block and committed.
4. A syncing peer deserializes the transaction from protobuf: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas`.
5. Peer recomputes:
   ```
   H2 = poseidon(INVOKE, version, sender,
                 poseidon(tip, L1_GAS_packed, L2_GAS_packed=0),
                 ...)
   ```
   (3-element resource hash, `L1Gas` path)
6. H1 ≠ H2 → `validate_transaction_hash` returns `false` → peer rejects the block. [6](#0-5) [1](#0-0) [2](#0-1)

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
