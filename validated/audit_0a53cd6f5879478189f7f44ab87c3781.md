### Title
Missing Expiration Timestamp on `WithdrawCollateral` and `WithdrawCollateralV2` Signed Transactions - (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateral` and `WithdrawCollateralV2` structs lack an `expiration` field. Unlike `Order`, which carries an explicit `expiration` validated on-chain via `_expired()`, user-signed withdrawal transactions carry no time bound. A signed withdrawal can be held by the sequencer and executed arbitrarily far in the future, with no on-chain mechanism for the user to enforce a deadline or cancel the request.

---

### Finding Description

The `Order` struct explicitly includes an `expiration` field:

```solidity
struct Order {
    bytes32 sender;
    int128 priceX18;
    int128 amount;
    uint64 expiration;   // <-- present
    uint64 nonce;
    uint128 appendix;
}
``` [1](#0-0) 

This expiration is enforced on-chain in `OffchainExchange._validateOrder()`:

```solidity
function _expired(uint64 expiration) internal view returns (bool) {
    return expiration <= getOracleTime();
}
``` [2](#0-1) 

The order is rejected if expired:

```solidity
!_expired(order.expiration);
``` [3](#0-2) 

By contrast, both withdrawal structs carry **no expiration field**:

```solidity
struct WithdrawCollateral {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;          // no expiration
}

struct WithdrawCollateralV2 {
    bytes32 sender;
    uint32 productId;
    uint128 amount;
    uint64 nonce;
    address sendTo;
    uint128 appendix;      // no expiration
}
``` [4](#0-3) 

When the sequencer processes a `WithdrawCollateral` or `WithdrawCollateralV2` via `processTransactionImpl`, it validates the nonce and signature but performs **no expiration check**:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.SignedWithdrawCollateral memory signedTx = abi.decode(...);
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, true);
    // no expiration check
    clearinghouse.withdrawCollateral(...);
``` [5](#0-4) 

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
    IEndpoint.SignedWithdrawCollateralV2 memory signedTx = abi.decode(...);
    validateSignedTx(signedTx.tx.sender, signedTx.tx.nonce, transaction, signedTx.signature, ...);
    // no expiration check
    clearinghouse.withdrawCollateral(...);
``` [6](#0-5) 

The same gap exists for `MintNlp` and `BurnNlp`, which also carry no expiration field:

```solidity
struct MintNlp { bytes32 sender; uint128 quoteAmount; uint64 nonce; }
struct BurnNlp  { bytes32 sender; uint128 nlpAmount;  uint64 nonce; }
``` [7](#0-6) 

For `MintNlp`/`BurnNlp`, the `oraclePriceX18` used to value the NLP position is **supplied by the sequencer at execution time**, not locked in by the user at signing time. Without an expiration, the sequencer can delay execution until a price that is unfavorable to the user. [8](#0-7) 

Additionally, the slow-mode path (`processSlowModeTransactionImpl`) also handles `WithdrawCollateral` without any expiration check:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    ...
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(...);
``` [9](#0-8) 

Slow-mode transactions have an `executableAt` lower bound (3-day delay) but **no upper expiration bound**, meaning they remain valid indefinitely once the window opens. [10](#0-9) 

---

### Impact Explanation

A user who signs a `WithdrawCollateral` or `WithdrawCollateralV2` transaction intends to withdraw at the time of signing. Without an expiration:

- The sequencer can delay execution indefinitely. During the delay, the user's collateral value may drop significantly (due to price movements), and the withdrawal executes at a time the user would not have chosen.
- Because nonces are strictly sequential (`nonce == nonces[address]++`), the user cannot submit any subsequent signed transaction (e.g., a cancel or a new withdrawal) until the pending one is processed. The user's account is effectively frozen.
- For `MintNlp`/`BurnNlp`, the oracle price is sequencer-supplied at execution time. A delayed execution means the user receives NLP tokens valued at a price they never agreed to. [11](#0-10) 

---

### Likelihood Explanation

The sequencer is the primary processor of these transactions. Sequencer delay (due to congestion, bugs, or deliberate ordering) is a realistic operational scenario. The slow-mode path is directly user-submitted and executable by anyone after `executableAt`, with no expiration, making the stale-execution path reachable without any privileged actor. The asymmetry with `Order` (which has expiration) shows this is an oversight rather than a design choice. [1](#0-0) 

---

### Recommendation

Add a `uint64 expiration` field to `WithdrawCollateral`, `WithdrawCollateralV2`, `MintNlp`, and `BurnNlp`. In `processTransactionImpl` and `processSlowModeTransactionImpl`, validate the expiration against `getOracleTime()` (or `block.timestamp`) using the same pattern already applied to `Order`:

```solidity
require(!_expired(signedTx.tx.expiration), ERR_EXPIRED);
```

This mirrors the existing `_expired` guard used for orders and gives users a trustless deadline guarantee. [2](#0-1) 

---

### Proof of Concept

1. User signs `WithdrawCollateralV2 { sender: userSubaccount, productId: 1, amount: 1000e18, nonce: 5, sendTo: userAddr, appendix: 0 }` at time T when their collateral is healthy.
2. User submits the signed transaction to the sequencer.
3. Sequencer queues the transaction but does not process it immediately (e.g., due to congestion).
4. At time T+N, the user's collateral value has dropped 40% due to market movement. The user wants to cancel the withdrawal and instead add more collateral.
5. The user cannot submit any new signed transaction (nonce 6 would fail; nonce 5 is already signed and pending).
6. The sequencer eventually processes the withdrawal at T+N, draining the user's collateral at the worst possible time.
7. No on-chain check rejects the transaction as stale. [6](#0-5)
