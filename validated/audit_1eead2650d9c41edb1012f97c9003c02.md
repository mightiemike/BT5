### Title
`verify_transaction_inclusion` Returns True for Internal Merkle Nodes Due to Absent tx_id Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is a publicly callable NEAR function that explicitly documents a critical missing check: it may return `true` when the supplied `tx_id` is an internal Merkle tree node rather than a real transaction hash. The function's own documentation states this validation is "performed at a higher level of verification," but no such higher-level check exists anywhere in the contract. The `#[deprecated]` Rust attribute is a compiler hint only — it imposes no runtime restriction. Any unprivileged NEAR caller can invoke this function directly and receive a fraudulent `true` result.

---

### Finding Description

`verify_transaction_inclusion` (lib.rs lines 288–323) is a public, `#[pause]`-gated function. Its own doc-comment contains an explicit warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification." [1](#0-0) 

The function's entire verification body reduces to a single Merkle root comparison:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

No check is performed to distinguish a real transaction leaf from an internal Merkle node. The only guard before this comparison is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

This guard does not prevent the 64-byte internal-node forgery. The `#[deprecated]` marker on the function is a Rust compiler warning; it does not prevent runtime invocation by any NEAR account. [4](#0-3) 

The v2 function (`verify_transaction_inclusion_v2`) was introduced to mitigate this by requiring a coinbase Merkle proof at index 0. However, it does so by calling the deprecated v1 function internally via `args.into()`, meaning the v1 path remains live and reachable independently. [5](#0-4) 

The documented "higher level of verification" is never implemented anywhere in the contract codebase. This is the direct analog to the external report's pattern: functionality is documented as existing but is absent from the code.

---

### Impact Explanation

Any downstream NEAR contract or application that calls `verify_transaction_inclusion` (v1) to gate an action — such as releasing bridged assets, minting tokens, or confirming a cross-chain payment — can be deceived into accepting a fabricated transaction inclusion proof. An attacker who supplies a 64-byte internal Merkle node as `tx_id` with a structurally valid Merkle path receives `true` from the contract, even though no such transaction exists on Bitcoin. This breaks the core security invariant of the SPV light client: that a `true` result guarantees the named transaction was confirmed on the source chain.

---

### Likelihood Explanation

The function is publicly callable with no role restriction beyond the global pause flag. The 64-byte Merkle forgery attack is a documented, well-understood Bitcoin SPV vulnerability (referenced in the contract's own v2 deprecation notice). An attacker needs only to: (a) identify a real Bitcoin block whose headers are stored in the contract, (b) locate or construct a 64-byte internal Merkle node within that block's tree, and (c) compute a valid Merkle path for it. The contract itself stores the block headers and exposes `get_block_hash_by_height` and `get_last_block_header`, giving the attacker all the information needed to construct the proof offline. No privileged access is required.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public ABI entirely, or add a runtime guard that panics unconditionally (making it truly uncallable). The documented "higher level of verification" must either be implemented inside the contract or the function must be made inaccessible. Relying on a Rust `#[deprecated]` attribute to prevent production use is insufficient.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature flag) with a real mainchain genesis block.
2. Submit a batch of real Bitcoin block headers via `submit_blocks` so the contract has a confirmed mainchain tip.
3. Select any stored block. Obtain its full transaction list and compute its Merkle tree offline.
4. Identify an internal Merkle node `N` at depth `d` whose 32-byte value, when paired with a sibling, produces the parent node. Construct a valid Merkle path from `N` to the Merkle root.
5. Call `verify_transaction_inclusion` with:
   - `tx_id = N` (the internal node, not a real transaction)
   - `tx_block_blockhash` = the block's hash (present in `mainchain_header_to_height`)
   - `tx_index` = the index corresponding to `N`'s position in the tree
   - `merkle_proof` = the computed path (non-empty, structurally valid)
   - `confirmations = 1`
6. The contract computes `compute_root_from_merkle_proof(N, index, path)`, which equals the block's `merkle_root`, and returns `true` — falsely asserting that the internal node `N` is a confirmed Bitcoin transaction. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
