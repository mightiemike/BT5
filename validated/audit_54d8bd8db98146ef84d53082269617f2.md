### Title
`ValidResourceBounds` Variant Reclassification in P2P Block-Sync Protobuf Converter Causes Transaction Hash Preimage Divergence — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation used in the block-sync path silently reclassifies an `AllResources` transaction as `L1Gas` whenever both `l2_gas` and `l1_data_gas` are zero. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for `L1Gas` (two resource entries: L1\_GAS, L2\_GAS) versus `AllResources` (three entries: L1\_GAS, L2\_GAS, L1\_DATA\_GAS), the transaction hash computed at gateway ingestion time diverges from the hash that would be computed from the same transaction after it is reconstructed from a P2P-synced block. This is the sequencer-native analog of the external report's MakerOrder/TakerOrder price mismatch: one representation (RPC/gateway path) is used for hash computation and signature binding, while a second representation (P2P sync reconstruction) silently produces a different hash for the same transaction bytes.

---

### Finding Description

**Step 1 — Submission path (RPC → Internal)**

`RpcInvokeTransactionV3.resource_bounds` is typed as `AllResourceBounds` (not `ValidResourceBounds`). When the gateway converts it to `InternalRpcInvokeTransactionV3`, the field is preserved as `AllResourceBounds`, and the transaction hash is computed via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash(ValidResourceBounds::AllResources(...))`. [1](#0-0) 

For `AllResources`, `get_tip_resource_bounds_hash` always appends the `l1_data_gas` packed field to the hash preimage, even when its value is zero: [2](#0-1) 

So the hash preimage for an `AllResources` transaction with `l2_gas=0, l1_data_gas=0` is:
`poseidon(tip, packed_l1_gas, packed_l2_gas=0, packed_l1_data_gas=0)`

**Step 2 — Serialization to protobuf**

When the transaction is serialized to `protobuf::ResourceBounds` for P2P transmission, `AllResources` with zero `l2_gas` and `l1_data_gas` serializes all three fields as present-but-zero: [3](#0-2) 

**Step 3 — Deserialization (P2P block-sync path)**

The block-sync converter (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`) uses `unwrap_or_default()` for `l1_data_gas` and then applies a zero-check to decide the variant: [4](#0-3) 

When both `l2_gas.is_zero()` and `l1_data_gas.is_zero()`, the result is `ValidResourceBounds::L1Gas(l1_gas)`. The `AllResources` variant is lost.

**Step 4 — Hash divergence**

For `L1Gas`, `get_tip_resource_bounds_hash` omits `l1_data_gas` from the preimage entirely: [5](#0-4) 

So the hash recomputed from the reconstructed transaction is:
`poseidon(tip, packed_l1_gas, packed_l2_gas=0)`

This is a **different value** from the hash computed at submission time, which was:
`poseidon(tip, packed_l1_gas, packed_l2_gas=0, packed_l1_data_gas=0)`

The protobuf round-trip is not idempotent: `AllResources{l2_gas=0, l1_data_gas=0}` → protobuf → `L1Gas`. The `ValidResourceBounds` variant is not preserved.

**Step 5 — Execution mode divergence**

Beyond the hash, `get_gas_vector_computation_mode()` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. This changes the gas accounting path in the blockifier during re-execution of synced blocks: [6](#0-5) 

With `AllResources` and `l2_gas.max_amount=0`, any L2 gas consumption causes a revert. With `L1Gas`, L2 gas is not checked. A transaction that was reverted on the proposer node (due to L2 gas exhaustion) would not be reverted on a syncing node that reconstructed it as `L1Gas`.

---

### Impact Explanation

**Hash binding (High):** The transaction hash is the signed commitment. The hash computed at submission time (using `AllResources`) differs from the hash that would be computed from the reconstructed transaction (using `L1Gas`) after P2P sync. Any component that recomputes the hash from the reconstructed `Transaction` object — including `validate_transaction_hash`, block commitment verification, or re-execution — will produce a wrong hash. This matches: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

**State divergence (Critical):** During re-execution of synced blocks (e.g., in `blockifier_reexecution` or proving), the reconstructed `L1Gas` variant changes the gas computation mode from `All` to `NoL2Gas`. A transaction that was reverted on the proposer (L2 gas exhausted with `max_amount=0`) would not be reverted on the re-executing node, producing a different state root, receipt, and revert result. This matches: *"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

---

### Likelihood Explanation

The attack requires a transaction with `l2_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0` (so both fields are zero and trigger reclassification). The stateless validator checks `l2_gas.max_price_per_unit >= min_gas_price`: [7](#0-6) 

This gate is bypassed when `min_gas_price = 0` (the default in test/bootstrap configurations) or when `validate_resource_bounds = false`. The stateful validator similarly checks the L2 gas price against the previous block's price, but this check is also skipped during bootstrap. There is no check on `l1_data_gas.max_price_per_unit` at all in the stateless validator. The `TODO` comment in the converter itself acknowledges the temporary nature of the `unwrap_or_default()` behavior: [8](#0-7) 

---

### Recommendation

1. **Preserve the `AllResources` variant through protobuf round-trips.** The deserialization in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` should not reclassify `AllResources` as `L1Gas` based on zero values. The variant should be encoded explicitly in the protobuf message (e.g., via a discriminant field or by requiring `l1_data_gas` to be present for all post-0.13.3 transactions).

2. **Enforce the `TODO`**: Remove the `unwrap_or_default()` fallback for `l1_data_gas` once 0.13.2 support is dropped, and assert that `l1_data_gas` is always present for `MempoolTransaction` and `ConsensusTransaction` (as the protobuf comment already states for 0.14.0+).

3. **Add a canonicalization invariant test**: Assert that `ValidResourceBounds::AllResources(AllResourceBounds { l1_gas: X, l2_gas: zero, l1_data_gas: zero })` round-trips through protobuf without changing variant, and that the hash is preserved.

### Proof of Concept

```rust
// Construct AllResources with zero l2_gas and l1_data_gas
let original = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas: ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) },
    l2_gas: ResourceBounds::default(),      // zero
    l1_data_gas: ResourceBounds::default(), // zero
});

// Compute hash at submission time (AllResources path: includes l1_data_gas=0)
let hash_at_submission = get_tip_resource_bounds_hash(&original, &Tip(0)).unwrap();
// Preimage: poseidon(0, packed_l1_gas, packed_l2_gas=0, packed_l1_data_gas=0)

// Serialize to protobuf (P2P transmission)
let proto: protobuf::ResourceBounds = original.into();
// proto.l1_gas = Some(...), proto.l2_gas = Some(zero), proto.l1_data_gas = Some(zero)

// Deserialize in block-sync path
let reconstructed = ValidResourceBounds::try_from(proto).unwrap();
// reconstructed = ValidResourceBounds::L1Gas(l1_gas)  ← variant changed!

// Compute hash from reconstructed transaction (L1Gas path: omits l1_data_gas)
let hash_after_sync = get_tip_resource_bounds_hash(&reconstructed, &Tip(0)).unwrap();
// Preimage: poseidon(0, packed_l1_gas, packed_l2_gas=0)

assert_ne!(hash_at_submission, hash_after_sync); // FAILS: hashes diverge
```

The two hashes differ because `AllResources` hashes 4 elements (tip + 3 resources) while `L1Gas` hashes 3 elements (tip + 2 resources). The `l1_data_gas=0` packed field is present in one preimage and absent in the other, producing a different Poseidon output.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
