### Title
Missing Cross-Validation Between `BouncerWeights::receipt_l2_gas` and `VersionedConstants::max_block_size` Causes Incorrect EIP-1559 Fee Market Gas Pricing - (`crates/blockifier/src/bouncer.rs` / `crates/apollo_versioned_constants/src/lib.rs`)

### Summary

The sequencer's EIP-1559 L2 gas price calculation (`calculate_next_base_gas_price`) uses `VersionedConstants::max_block_size` as the block capacity denominator, while the batcher's bouncer enforces `BouncerWeights::receipt_l2_gas` as the actual block-fill limit. These two values are configured independently in separate subsystems and must be equal for correct fee market operation. The code documents this with explicit "Must stay in sync" comments but provides no startup or runtime validation to enforce the invariant. An operator (or a misconfigured deployment) can set them to divergent values, causing every subsequent block to carry a systematically wrong next-block L2 gas price.

### Finding Description

Two independently-configured values govern the same logical quantity — the maximum L2 gas per block:

**Value 1 — Bouncer block capacity** (`receipt_l2_gas`): [1](#0-0) 

The default is `GasAmount(5800000000)` and the config-schema description explicitly states "Should equal max_block_size": [2](#0-1) 

**Value 2 — Fee market block capacity** (`max_block_size`): [3](#0-2) 

The comment on `max_block_size` explicitly states "Must stay in sync with BouncerWeights receipt_l2_gas." The current versioned-constants JSON files set `max_block_size = 5800000000`: [4](#0-3) 

**How they are used divergently:**

`receipt_l2_gas` is the bouncer's hard stop — the batcher closes a block when accumulated `receipt_l2_gas` reaches this limit: [5](#0-4) 

`max_block_size` is the denominator in the EIP-1559 price-adjustment formula inside `calculate_next_base_gas_price`: [6](#0-5) 

The formula computes `price_change = (price × |gas_used − gas_target|) / (gas_target × denominator)`. The `gas_target` is also read from `VersionedConstants` (not from `BouncerConfig`), so both the target and the capacity used for fee-market math come from the orchestrator versioned constants, while the actual block-fill limit comes from the batcher's `BouncerConfig`.

**No cross-validation exists.** The `validate_dynamic_config` function for `ContextDynamicConfig` only validates `min_l2_gas_price_per_height` ordering and minimum price floor: [7](#0-6) 

The `SequencerNodeConfig::cross_member_validations` checks component-config presence/absence but never compares `receipt_l2_gas` against `max_block_size`: [8](#0-7) 

The only guard is a runtime `assert!` inside `calculate_next_base_gas_price` that checks `gas_target < max_block_size` — but this does not check `receipt_l2_gas == max_block_size`, and it fires as a panic mid-execution rather than at startup: [9](#0-8) 

**Concrete divergence scenario:**

An operator upgrades the orchestrator versioned constants to a new Starknet version that raises `max_block_size` (e.g., from `5800000000` to `7000000000`) but forgets to update `receipt_l2_gas` in `config_schema.json` / `BouncerConfig`. The batcher still closes blocks at 5.8 B gas, but the fee market computes prices as if blocks can hold 7 B gas. Every block appears under-utilized to the EIP-1559 formula, so the next-block L2 gas price is continuously driven downward — users are systematically undercharged and the fee market signal is broken. The reverse (raising `receipt_l2_gas` without raising `max_block_size`) causes the fee market to see blocks as perpetually over-full and drives prices upward without bound.

### Impact Explanation

The `next_l2_gas_price` computed by `calculate_next_base_gas_price` is embedded in every committed block header and is the authoritative price used by the gateway's stateful validator to accept or reject transactions: [10](#0-9) 

A wrong `max_block_size` produces a wrong `next_l2_gas_price` in every block, causing incorrect fee accounting with direct economic impact on all users. This matches the "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" criterion.

### Likelihood Explanation

Both values are independently configurable through separate config files (`config_schema.json` for `receipt_l2_gas` and versioned-constants JSON for `max_block_size`). Any protocol upgrade that changes block capacity must update both files. The absence of any automated check means a single missed update silently corrupts the fee market for the entire network. The diff-regression files show that `max_block_size` has already changed across versions: [11](#0-10) 

### Recommendation

Add a startup cross-validation — either in `SequencerNodeConfig::cross_member_validations` or in a dedicated invariant check called at node boot — that asserts:

```
batcher_config.block_builder_config.bouncer_config.block_max_capacity.receipt_l2_gas
  == VersionedConstants::latest_constants().max_block_size
```

Return a `ConfigError` (not a panic) if they differ, so the node refuses to start with an inconsistent configuration rather than silently producing wrong fee-market prices.

Additionally, add a similar check that `gas_target < max_block_size` at startup (not only as a runtime `assert!` inside `calculate_next_base_gas_price`).

### Proof of Concept

1. In `crates/apollo_node/resources/config_schema.json`, change `batcher_config.static_config.block_builder_config.bouncer_config.block_max_capacity.receipt_l2_gas` from `5800000000` to `4000000000`.
2. Leave `orchestrator_versioned_constants_0_14_4.json` with `max_block_size = 5800000000`.
3. Start the node — it starts without error.
4. Produce blocks at ~50% fill (≈ 2 B gas per block, which is below both limits).
5. Observe `calculate_next_base_gas_price` using `gas_target = 1040000000` and `max_block_size = 5800000000`: the formula sees `gas_used (2B) > gas_target (1.04B)` and raises the price. But the batcher is actually closing blocks at 4 B capacity, so the true utilization ratio is 50%, not 192%. The fee market raises prices as if the network is congested when it is not.
6. Conversely, set `receipt_l2_gas = 7000000000` and leave `max_block_size = 5800000000`. The batcher accepts blocks up to 7 B gas, but the fee market treats any block above 5.8 B as "impossible" — the `gas_used > max_block_size` path is never reached in the formula, so the price never rises to reflect true congestion. ```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
