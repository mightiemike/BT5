### Title
`DepositAllowlistExtension` Checks LP Recipient (`owner`) Instead of Depositor (`sender`), Allowing Unauthorized Deposits — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual depositor who pays tokens) and instead gates on `owner` (the LP-position recipient). Any address not in the allowlist can bypass the deposit restriction by calling `addLiquidity` with an authorized owner's address.

---

### Finding Description

`MetricOmmPool.addLiquidity` invokes the extension hook as:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Inside `ExtensionCalling._beforeAddLiquidity`, this becomes:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

So the first argument (`sender`) is `msg.sender` of `addLiquidity` — the address that pays tokens via the swap callback — and the second argument (`owner`) is the LP-position recipient.

`DepositAllowlistExtension.beforeAddLiquidity` receives both but **discards `sender`** (unnamed `address`) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

The contract's own NatSpec states the intent: *"Gates `addLiquidity` by depositor address, per pool."* [4](#0-3) 

The depositor is the address that pays tokens (`msg.sender` of `addLiquidity` → `sender`). The implementation checks the wrong actor.

---

### Impact Explanation

Any address not in the allowlist can call:

```solidity
pool.addLiquidity(allowlisted_address, salt, deltas, callbackData, extensionData)
```

The extension sees `owner = allowlisted_address`, the check passes, the unauthorized caller pays the tokens via callback, and the allowlisted address receives LP shares it never requested. The pool admin's curation policy — restricting which depositors may contribute liquidity — is fully bypassed by any unprivileged caller. This is an **admin-boundary break**: a pool-admin-configured allowlist check is circumvented by an unprivileged path.

---

### Likelihood Explanation

Exploitation requires no special privileges, no flash loans, and no oracle manipulation. Any EOA or contract can call `addLiquidity` with an authorized owner's address. The bypass is deterministic and costs only gas plus the deposited tokens (which the attacker controls).

---

### Recommendation

Bind the allowlist check to `sender` (the actual depositor) rather than `owner` (the LP recipient):

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The analogous `SwapAllowlistExtension.beforeSwap` already does this correctly — it checks `sender` (the swapper) and discards `recipient`: [5](#0-4) 

`DepositAllowlistExtension` should follow the same pattern.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` configured; `alice` is allowlisted, `bob` is not.
2. `bob` calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
3. The extension receives `sender = bob` (discarded), `owner = alice`.
4. Check: `allowedDepositor[pool][alice]` → `true` → no revert.
5. `bob`'s callback pays the tokens; `alice` receives LP shares.
6. `bob` has deposited into a curated pool despite not being in the allowlist.

The pool admin's deposit restriction is silently bypassed on every call where the caller supplies any allowlisted address as `owner`.
