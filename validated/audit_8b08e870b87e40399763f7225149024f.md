### Title
`DelegateAction` / `DelegateActionV2` Signing Domain Lacks Chain Binding — Cross-Chain Signature Replay - (File: `core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

### Summary

The NEP-366 meta-transaction (`DelegateAction`) and its gas-key variant (`DelegateActionV2`) sign a payload that contains no chain-specific identifier. A signature produced for one NEAR network (e.g. testnet) is cryptographically valid on any other NEAR network (e.g. mainnet) where the same account and key exist, the nonce is still acceptable, and the block height is still below `max_block_height`. A relayer — or any party that observes a signed delegate action — can replay it on a different chain without the signer's consent.

### Finding Description

`DelegateAction.get_nep461_hash()` constructs the signed payload as:

```
SHA256( borsh(MessageDiscriminant) || borsh(DelegateAction) )
```

`MessageDiscriminant` is the constant `MIN_ON_CHAIN_DISCRIMINANT + NEP_366_META_TRANSACTIONS` = `0x4000_016E` (little-endian u32). `DelegateAction` contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`.

Neither the discriminant nor any field of `DelegateAction` encodes the genesis hash, chain ID, or any other chain-specific value. The same is true for `DelegateActionV2` / `VersionedDelegateActionPayload::get_nep461_hash()`, which uses discriminant `MIN_ON_CHAIN_DISCRIMINANT + NEP_611_GAS_KEYS`.

By contrast, a regular `Transaction` (V0 or V1) includes `block_hash` — a recent chain-specific block hash — in its signed body, which the runtime validates against `transaction_validity_period`. This makes ordinary transactions implicitly chain-bound. `DelegateAction` deliberately replaced `block_hash` with `max_block_height` (a plain integer) to allow offline signing without a live chain connection, but this removed the only chain-binding mechanism.

The exact divergent Borsh bytes signed for a `DelegateAction` on mainnet and testnet are **identical** for any user who has the same account, key, nonce state, and chooses a `max_block_height` that is valid on both chains (which is almost always the case since mainnet and testnet track similar heights).

Relevant code paths:

- `DelegateAction::get_nep461_hash` — builds `SignableMessage` with no chain field: [1](#0-0) 

- `VersionedDelegateActionPayload::get_nep461_hash` — same pattern for V2: [2](#0-1) 

- `SignableMessage` — only fields are `discriminant` (a fixed constant) and `msg` (the action body); no chain context: [3](#0-2) 

- `DelegateAction` struct — `max_block_height` is a plain `u64`, not a chain-specific hash: [4](#0-3)

- `TransactionV0` includes `block_hash` (chain-specific); `DelegateAction` does not: [8](#0-7) 

### Impact Explanation

An attacker (e.g. a malicious relayer, or any party who intercepts a signed `DelegateAction`) can submit the same signed payload on a different NEAR network. If the signer's account exists on both networks with the same access key and a valid nonce, the inner actions execute on the unintended chain. Inner actions can include `Transfer`, `FunctionCall`, `AddKey`, `DeleteKey`, `DeleteAccount`, and `DeployContract` — the full set of `NonDelegateAction` variants. This allows unauthorized fund transfers, key manipulation, or contract deployment on the victim's account on the replayed chain.

### Likelihood Explanation

Many NEAR users — especially developers and power users — maintain the same named account (e.g. `alice.near` / `alice.testnet` share the same key material in practice, and cross-chain account mirroring is common) on multiple NEAR networks with the same key pair. Relayers are the natural adversary: they receive signed `DelegateAction` payloads and are trusted only to submit them on the intended chain. A malicious relayer can trivially replay any received delegate action on mainnet if the conditions are met. The `max_block_height` bound provides no chain binding — it is a plain integer that is valid on any chain whose current height is below that value, which is almost always true since mainnet and testnet track similar block heights.

### Recommendation

Include a chain-specific identifier in the `DelegateAction` signed payload. The natural choice is the genesis block hash (`GenesisId::hash`), which is already defined in the codebase: [9](#0-8) 

Add a `genesis_hash: CryptoHash` field to `DelegateAction` and `DelegateActionV2`, populate it at signing time, validate it at execution time in `apply_delegate_action`, and gate the new field behind a protocol feature so the change is backward-compatible. Alternatively, encode the genesis hash into the `MessageDiscriminant` or into a new wrapper type around `SignableMessage` so that the chain binding is enforced at the signing-domain level without changing the action struct layout.

### Proof of Concept

1. User `alice.near` exists on both mainnet and testnet with the same ED25519 key pair and nonce `N` on both chains.
2. Alice signs a `DelegateAction` for testnet: `sender_id="alice.near"`, `receiver_id="bob.near"`, `actions=[Transfer{deposit=1_000_000}]`, `nonce=N+1`, `max_block_height=current_testnet_height+1000`.
3. The signed payload bytes are: `borsh(0x4000016E_u32) || borsh(DelegateAction{...})` — identical on both chains.
4. A malicious relayer, instead of submitting to testnet, wraps the `SignedDelegateAction` in an outer `Transaction` addressed to `alice.near` on **mainnet** and broadcasts it.
5. Mainnet runtime calls `apply_delegate_action` → `signed_delegate_action.verify()` → `DelegateAction::get_nep461_hash()` → same hash → signature verifies → `Transfer` of 1 NEAR executes on mainnet from Alice's account without her consent.

The nonce `N+1` is valid on mainnet (Alice's mainnet nonce is still `N`), and `max_block_height` is valid since mainnet height is similar to testnet height. The attack succeeds.
