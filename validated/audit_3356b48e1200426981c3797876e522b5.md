### Title
CEI Pattern Violation in `fastWithdraw` Enables Reentrancy to Drain Pool — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

The `fastWithdraw` function in `BaseWithdrawPool.sol` makes external token transfer calls without first marking the transaction index as consumed. This violates the Checks-Effects-Interactions pattern. Any ERC-20 token with a receive-hook (e.g., ERC-777) or a malicious token registered as a spot product can reenter `fastWithdraw` with the same signed transaction before the replay-prevention state is written, draining the pool.

---

### Finding Description

`submitWithdrawal` — the sequencer-driven withdrawal path — correctly follows CEI: it sets `markedIdxs[idx] = true` **before** calling `handleWithdrawTransfer`. [1](#0-0) 

`fastWithdraw` — the user-callable path — does not. The visible body of the function (lines 91–114) performs:

1. Signature validation (`v.requireValidTxSignatures`) — **no state written**
2. `safeTransferFrom` to collect the fee from `msg.sender` (line 108) — **external call**
3. `fees[productId] += fee` (line 111) — state update, but only for fee accounting
4. `handleWithdrawTransfer(token, sendTo, transferAmount)` (line 113) — **external call, principal transfer** [2](#0-1) 

Nowhere in this path is `markedIdxs[idx]` set to `true` before the external calls. The `handleWithdrawTransfer` implementation calls `token.safeTransfer(to, uint256(amount))`: [3](#0-2) 

If `token` is an ERC-777 or any token that invokes a callback on the recipient (`sendTo`), the recipient can reenter `fastWithdraw` with the identical signed transaction. Because `markedIdxs[idx]` has not been set yet, the signature check passes again, and the full `transferAmount` is sent a second (and further) time.

---

### Impact Explanation

Each reentrant call transfers `transferAmount` tokens from the pool to `sendTo`. The pool's liquidity for that product can be fully drained in a single transaction. The `fees[productId]` accounting is also corrupted because the fee is added multiple times while the actual fee token was only collected once (or not at all if `sendTo == msg.sender`).

---

### Likelihood Explanation

The attack requires a token registered as a spot product that supports transfer callbacks (ERC-777 or equivalent). The attacker must also obtain a valid signed fast-withdrawal transaction (their own, legitimately signed). No privileged role is needed beyond being a normal user with a valid fast-withdrawal signature. The entry point (`fastWithdraw`) is publicly callable. [4](#0-3) 

---

### Recommendation

Apply the CEI pattern to `fastWithdraw`: mark `markedIdxs[idx] = true` (and check it) **before** any external call, mirroring the pattern already used in `submitWithdrawal`.

```solidity
// At the top of fastWithdraw, after signature validation:
require(!markedIdxs[idx], "already processed");
markedIdxs[idx] = true;
// ... then proceed with fee collection and transfer
``` [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys a malicious ERC-777 token and gets it listed as a spot product (or targets an existing ERC-777 spot token).
2. Attacker obtains a valid fast-withdrawal signature for `transferAmount = X` to their contract address (`sendTo = attacker`).
3. Attacker calls `fastWithdraw(transaction, idx, signatures)`.
4. Execution reaches `handleWithdrawTransfer` → `token.safeTransfer(attacker, X)` at line 113.
5. The token's `tokensReceived` hook fires on the attacker contract before `markedIdxs[idx]` is set.
6. Attacker reenters `fastWithdraw` with the same `(transaction, idx, signatures)`.
7. Signature validation passes (idx not yet marked), and another `X` tokens are transferred.
8. Steps 5–7 repeat until the pool balance for that product is exhausted.
9. After the reentrant calls unwind, `markedIdxs[idx]` is never set (it is not set anywhere in `fastWithdraw`), so the attack can be repeated in a subsequent transaction as well. [2](#0-1)
