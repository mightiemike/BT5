Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Originating User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router when a user routes through `MetricOmmSimpleRouter`. If a pool admin allowlists the router (a natural action to enable periphery usage), every non-allowlisted user can bypass the curation gate by routing through the router, because the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. The originating user's address is never forwarded to the pool or extension.

## Finding Description
**Root cause — call chain:**

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` value is on the allowlist for the pool (`msg.sender` inside the extension is the pool): [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly, making itself `msg.sender` to the pool: [3](#0-2) 

The originating user's address is stored only in transient callback context for payment purposes and is never forwarded to the pool or extension: [4](#0-3) 

**Why existing checks fail:** There is no mechanism in the pool or extension interface to receive or verify the originating EOA. The extension is structurally incapable of distinguishing which EOA initiated the router call. The only configurable path is `setAllowedToSwap`, which operates on the `sender` value the pool provides — always the router when periphery is used: [5](#0-4) 

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (to let approved users use the standard periphery) inadvertently opens the gate to every user. Any address can call `MetricOmmSimpleRouter.exactInputSingle` and the extension will pass because `allowedSwapper[pool][router] == true`. The intended curation policy — restricting swaps to specific KYC'd or institutional addresses — is completely bypassed. LPs who deposited under the assumption that only approved counterparties would trade against them are exposed to unrestricted public flow, constituting broken core pool access-control functionality with direct economic impact on LP positions.

## Likelihood Explanation
The trigger is a reasonable, non-malicious admin action: allowlisting the router so that approved users can use the standard periphery. Any pool admin who wants approved users to access the pool through the router will hit this configuration. The exploit itself requires no privilege — any EOA calls the public router function `exactInputSingle`. The condition is repeatable on every swap attempt by any non-allowlisted user.

## Recommendation
The extension must receive and check the economically relevant actor, not the intermediary. The preferred fix is to have the router forward `msg.sender` as an additional field in `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires a coordinated interface change between the router and extension. Alternatively, document clearly that allowlisting the router opens the gate to all users, and require admins to allowlist individual EOAs who call the pool directly — but this makes the standard periphery unusable for curated pools.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only `approvedUser` is allowlisted.
// Admin also allowlists the router so approvedUser can use periphery.
swapExtension.setAllowedToSwap(address(pool), approvedUser, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // ← opens the gate

// Exploit: bannedUser bypasses the allowlist via the router.
vm.prank(bannedUser);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: bannedUser,
        tokenIn: address(token1),
        zeroForOne: false,
        amountIn: 1000,
        amountOutMinimum: 0,
        priceLimitX64: type(uint128).max,
        deadline: block.timestamp + 1,
        extensionData: ""
    })
);
// ✓ Swap succeeds — extension saw sender=router (allowlisted), not bannedUser (blocked).
// allowedSwapper[pool][bannedUser] == false was never consulted.
```

The `beforeSwap` check at line 37 resolves to `allowedSwapper[pool][router] == true` and passes, while `allowedSwapper[pool][bannedUser] == false` is never evaluated. [6](#0-5)
