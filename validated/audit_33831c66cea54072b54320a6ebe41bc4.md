### Title
Smart Contract Subaccounts Permanently Locked Out of `WithdrawCollateralV2` with Custom `sendTo` Due to ECDSA-Only Signature Verification - (File: `core/contracts/Verifier.sol`)

---

### Summary

`Verifier.validateSignature()` and `Verifier.validateCompactSignature()` use `ECDSA.recover()` exclusively with no EIP-1271 fallback. Smart contracts (multisigs, protocol vaults, DeFi integrations) that own subaccounts cannot produce ECDSA signatures. For `WithdrawCollateralV2` with `sendTo != address(0)`, the linked-signer bypass is explicitly disabled, and there is no slow-mode equivalent — permanently locking smart contract subaccount owners out of this operation.

---

### Finding Description

Subaccounts in Nado are `bytes32` values whose first 20 bytes are the owner's address. Any address — including a smart contract — can create a subaccount by calling `depositCollateral()` or `depositCollateralWithReferral()`: [1](#0-0) 

All fast-path signed transactions are validated through `EndpointTx.validateSignedTx()`, which calls `validateSignature()` → `Verifier.validateSignature()`: [2](#0-1) 

`ECDSA.recover()` is the sole verification mechanism. There is no EIP-1271 `isValidSignature()` branch. The same applies to the compact variant: [3](#0-2) 

For `WithdrawCollateralV2`, the `allowLinkedSigner` flag is set to `signedTx.tx.sendTo == address(0)`. When `sendTo` is a non-zero address, the linked signer is explicitly excluded: [4](#0-3) 

This means only the subaccount owner itself can authorize a `WithdrawCollateralV2` with a custom `sendTo`. If the owner is a smart contract, it cannot produce a valid ECDSA signature, and the operation is permanently inaccessible. There is no slow-mode handler for `WithdrawCollateralV2` — the slow-mode path only covers `WithdrawCollateral` (v1), which forces `sendTo = address(0)`: [5](#0-4) 

---

### Impact Explanation

A smart contract subaccount owner (e.g., a Gnosis Safe treasury, a DeFi vault, or any protocol integration) that deposits collateral can never execute `WithdrawCollateralV2` with a non-zero `sendTo`. The only available withdrawal path is the slow-mode `WithdrawCollateral` (v1), which forces funds back to the contract's own address and incurs a mandatory 3-day delay. The `sendTo` routing feature — which is the entire purpose of `WithdrawCollateralV2` — is permanently inaccessible to these users. Additionally, `MintNlp`, `BurnNlp`, and `TransferQuote` have no slow-mode equivalents at all, making those operations entirely unavailable to smart contract subaccount owners without first linking an EOA signer via slow mode.

---

### Likelihood Explanation

Smart contracts interacting with on-chain trading protocols is a standard pattern. Gnosis Safe multisigs are the dominant treasury management tool for DAOs and protocols. Any protocol that integrates Nado by depositing collateral from a contract address will encounter this limitation. The `depositCollateralWithReferral()` function accepts an arbitrary `bytes32 subaccount`, making it trivially easy for a contract to create a subaccount under its own address.

---

### Recommendation

In `Verifier.validateSignature()` and `Verifier.validateCompactSignature()`, after `ECDSA.recover()` fails to match the sender, add an EIP-1271 fallback: if `address(uint160(bytes20(sender)))` has contract code, call `IERC1271(senderAddr).isValidSignature(digest, signature)` and accept the result if it returns `bytes4(0x1626ba7e)` (the EIP-1271 magic value). This mirrors the fix recommended in the external report and is the standard pattern used by OpenZeppelin's `SignatureChecker` library.

---

### Proof of Concept

1. A Gnosis Safe at address `0xSafe` calls `endpoint.depositCollateral(bytes12("default"), productId, amount)`, creating subaccount `bytes32(abi.encodePacked(0xSafe, bytes12("default")))` with collateral.
2. The Safe wants to withdraw to a different address `0xRecipient` using `WithdrawCollateralV2` with `sendTo = 0xRecipient`.
3. The sequencer processes the transaction, calling `validateSignedTx(sender, nonce, tx, sig, false)` (since `sendTo != address(0)`, `allowLinkedSigner = false`).
4. `validateSignedTx` calls `validateSignature(sender, digest, sig, false)` → `verifier.validateSignature(sender, address(0), digest, sig)`.
5. `ECDSA.recover(digest, sig)` returns a garbage address (Gnosis Safe produces EIP-1271 signatures, not raw ECDSA).
6. The `require` at line 298–303 of `Verifier.sol` reverts with `ERR_INVALID_SIGNATURE`.
7. The Safe cannot use `WithdrawCollateralV2` with any `sendTo != address(0)`, ever. The only fallback is slow-mode `WithdrawCollateral` (v1) which forces `sendTo = address(0)` and has a 3-day delay.

        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
