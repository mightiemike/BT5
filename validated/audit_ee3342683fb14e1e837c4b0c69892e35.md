### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Internal-Node Forgery — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` but carries no `#[private]` guard and is declared `pub fn`, making it a live, externally callable NEAR contract method. Any unprivileged account can call it directly with a `ProofArgs` whose `tx_id` is an internal Merkle tree node rather than a leaf transaction hash. Because `compute_root_from_merkle_proof` is position-only and performs no leaf/internal-node distinction, the call returns `true`, falsely certifying that a fabricated transaction is included in a real block.

---

### Finding Description

`verify_transaction_inclusion` is exposed as a public NEAR contract method: [1](#0-0) 

The `#[deprecated]` Rust attribute emits a compiler warning only; it has no runtime effect and does not restrict external callers. The `#[pause]` attribute is a pausability gate, not an access-control gate. There is no `#[private]` annotation.

The function passes the caller-supplied `tx_id` directly into `compute_root_from_merkle_proof` without any check that it is a leaf-level transaction hash: [2](#0-1) 

`compute_root_from_merkle_proof` is purely positional — it concatenates and double-SHA256-hashes whatever 32-byte value it receives, with no leaf/internal-node distinction: [3](#0-2) 

`verify_transaction_inclusion_v2` closes this gap by first verifying a coinbase proof at index 0 against the block's `merkle_root`, which anchors the proof depth and prevents an attacker from presenting an internal node as a leaf: [4](#0-3) 

But that guard is entirely absent from the v1 path. The `From<ProofArgsV2> for ProofArgs` conversion that drops `coinbase_tx_id` and `coinbase_merkle_proof` is only used on the internal call from v2 to v1 — it is not a gate on the v1 entry point itself: [5](#0-4) 

The code's own documentation acknowledges the weakness explicitly: [6](#0-5) 

---

### Impact Explanation

An attacker who knows any block stored in the contract's `headers_pool` (public state) can:

1. Obtain the block's full transaction list and compute its Merkle tree.
2. Select any internal node `N` at tree depth `d` (where `d < tree_height`).
3. Construct `ProofArgs { tx_id: N, tx_index: <position of N at depth d>, merkle_proof: <remaining d' siblings up to root>, confirmations: 0 }`.
4. Call `verify_transaction_inclusion` directly.
5. Receive `true` — the contract certifies that `N` (a fabricated, non-existent transaction) is included in the block.

Any downstream protocol or bridge that relies on `verify_transaction_inclusion` returning `true` as proof of a real Bitcoin transaction is vulnerable to accepting a completely fabricated payment or event.

---

### Likelihood Explanation

The entry point is unrestricted — any NEAR account can call it with zero privileges. The required inputs (block hash, Merkle tree structure) are fully derivable from public Bitcoin and NEAR chain data. No key compromise, DAO vote, or social engineering is required. The attack is mechanically straightforward and locally reproducible.

---

### Recommendation

Add `#[private]` to `verify_transaction_inclusion` to restrict it to internal contract calls only (i.e., only reachable via `verify_transaction_inclusion_v2`):

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
#[private]   // ← add this
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
```

This ensures the only public entry point for proof verification is `verify_transaction_inclusion_v2`, which enforces the coinbase anchor check.

### Proof of Concept

Given a block with transactions `[T0 (coinbase), T1, T2, T3]` stored in the contract:

```
Level 2 (root):  R = H(M01, M23)
Level 1:         M01 = H(T0, T1),   M23 = H(T2, T3)
Level 0 (leaf):  T0, T1, T2, T3
```

Attacker uses internal node `M01` as `tx_id`:

```rust
// ProofArgs constructed by attacker
ProofArgs {
    tx_id: M01,          // internal node, NOT a real transaction
    tx_block_blockhash: <real block hash in headers_pool>,
    tx_index: 0,         // M01 is at position 0 at level 1
    merkle_proof: vec![M23],  // one sibling gets back to root R
    confirmations: 0,
}
```

`compute_root_from_merkle_proof(M01, 0, [M23])` computes `H(M01, M23) = R`, which equals `header.block_header.merkle_root`. The function returns `true` for a transaction that does not exist.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
