### Title
`tree_hash_from_stream` Silently Rejects All Back-Reference Streams That `node_from_bytes_backrefs` Accepts, Breaking Hash Invariant — (`src/serde/tools.rs`)

### Summary

`tree_hash_from_stream` has no branch for the `BACK_REFERENCE` byte (`0xfe`). Every valid backref-encoded stream accepted by `serialized_length_from_bytes` and `node_from_bytes_backrefs` causes `tree_hash_from_stream` to return `Err(SerializationError)`, while the other path returns a valid hash. The invariant "one canonical byte stream → one tree hash" is concretely broken.

### Finding Description

`tree_hash_from_stream` dispatches on the first byte of each node: [1](#0-0) 

The branches cover `0xff` (cons), `0x80` (nil), `<= 0x7f` (single-byte atom), and a catch-all `else` for multi-byte atoms. There is **no branch for `BACK_REFERENCE = 0xfe`**. When `0xfe` is read, execution falls into the `else` arm and calls `decode_size(f, 0xfe)`. [2](#0-1) 

Inside `decode_size_with_offset`, `0xfe` has 7 leading ones, so `atom_start_offset = 7` and `size_blob` is allocated with length 7: [3](#0-2) 

The guard `if size_blob.len() > 6` fires immediately (7 > 6) and returns `Err(SerializationError)`: [4](#0-3) 

By contrast, `serialized_length_from_bytes` and `node_from_bytes_backrefs` both have an explicit `BACK_REFERENCE` branch that correctly resolves the path: [5](#0-4) [6](#0-5) 

`tree_hash_from_stream` is a public export: [7](#0-6) 

### Impact Explanation

Any caller that uses `tree_hash_from_stream` as a fast-path hash check and `node_from_bytes_backrefs` for deserialization will disagree on both validity and hash for the same byte stream. If different nodes in a consensus network use different code paths, they will compute different puzzle hashes for the same serialized program, enabling consensus splits.

### Likelihood Explanation

The proof is mechanical and requires no special privileges. Any valid backref-encoded CLVM object (produced by the library's own `node_to_bytes_backrefs`) triggers the bug. The existing test suite already contains the exact triggering byte sequence and confirms both sides of the split:

- `serialized_length_from_bytes` + `node_from_bytes_backrefs` accept `ff86666f6f626172fe01`: [8](#0-7) 

- `node_from_bytes_backrefs` + `ObjectCache::treehash` return hash `9148834131750904...`: [9](#0-8) 

- `tree_hash_from_stream` on the same bytes returns `Err(SerializationError)` because `decode_size(f, 0xfe)` hits the `size_blob.len() > 6` guard.

### Recommendation

Add an explicit `BACK_REFERENCE` branch in `tree_hash_from_stream` that resolves the back-reference path against the already-computed hash stack, mirroring the logic in `serialized_length_from_bytes`. Alternatively, add an assertion or integration test that runs both paths on every valid backref-encoded stream and asserts identical results.

### Proof of Concept

```
buf = ff86666f6f626172fe01   // ("foobar" "foobar") with back-reference

serialized_length_from_bytes(buf)                          -> Ok(10)
node_from_bytes_backrefs(&mut alloc, buf)                  -> Ok(node)
ObjectCache::new(treehash).get_or_calculate(&alloc, &node) -> Some(9148834131750904c023598bed28db269bdb29012514579e723d63e27829bcba)
tree_hash_from_stream(&mut Cursor::new(buf))               -> Err(SerializationError)
```

The split is deterministic, locally reproducible, and requires only a byte string that the library itself produces via `node_to_bytes_backrefs`.
