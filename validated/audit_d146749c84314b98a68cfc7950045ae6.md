### Title
`starknet_estimateFee` Omits `l2_gas_consumed` from `FEE_ESTIMATE` Response, Making L2 Gas Resource Bounds Undeducible for v3 Transactions â€” (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `FeeEstimation` struct and `tx_execution_output_to_fee_estimation` function expose `l2_gas_price` but silently drop `gas_vector.l2_gas` (L2 gas consumed). The OpenRPC `FEE_ESTIMATE` schema description simultaneously states `overall_fee = gas_consumed*gas_price + data_gas_consumed*data_gas_price`, which is arithmetically wrong for any v3 (`AllResources`) transaction that consumes L2 gas. Clients relying on the returned components to reconstruct the fee or to set `l2_gas` resource bounds receive an authoritative-looking but incorrect picture.

---

### Finding Description

**Root cause â€” missing field in `tx_execution_output_to_fee_estimation`:**

`crates/apollo_rpc_execution/src/objects.rs` lines 161â€“182 build the `FeeEstimation` object:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed:      gas_vector.l1_gas.0.into(),      // âś“ exposed
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // âś“ exposed
    l1_data_gas_price,
    l2_gas_price,                                        // âś“ price exposed
    // gas_vector.l2_gas is NEVER exposed               // âś— amount silently dropped
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

The `GasVector` receipt has three components â€” `l1_gas`, `l1_data_gas`, and `l2_gas`. Only the first two are surfaced. The third is silently discarded. An internal TODO at line 104 acknowledges this gap:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

**Wrong schema description in the OpenRPC spec:**

`crates/apollo_rpc/resources/V0_8/starknet_api_openrpc.json` lines 3648â€“3651 describe `overall_fee` as:

```json
"description": "The estimated fee for the transaction (in wei or fri, depending on the tx version),
                equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price"
```

The actual fee computation in `GasVector::cost()` (`crates/starknet_api/src/execution_resources.rs` lines 156â€“186) is:

```
overall_fee = l1_gas * l1_gas_price
            + l1_data_gas * l1_data_gas_price
            + l2_gas * (l2_gas_price + tip)
```

For any v3 transaction with non-zero `l2_gas`, the schema description is arithmetically wrong. The `overall_fee` value in the response is correct (it comes from the actual receipt), but the description tells clients it equals a sum that omits the L2 gas term entirely.

**Consequence for resource-bound construction:**

For v3 (`AllResources`) transactions, the gateway pre-validates that `l2_gas` resource bounds are sufficient (`crates/blockifier/src/transaction/account_transaction.rs` lines 398â€“424). A client that calls `starknet_estimateFee` to learn what bounds to set:

- Receives `l2_gas_price` but no `l2_gas_consumed`.
- Cannot compute the correct `l2_gas` `max_amount` to place in the transaction.
- If it trusts the schema description and reconstructs the fee as `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`, it obtains a value strictly lower than `overall_fee` for any transaction with L2 gas consumption, creating a visible inconsistency.
- If it sets `l2_gas.max_amount = 0`, the transaction is rejected at pre-validation.

---

### Impact Explanation

**High â€” RPC fee estimation returns an authoritative-looking wrong value.**

The `FEE_ESTIMATE` schema description is wrong: it claims `overall_fee = gas_consumed*gas_price + data_gas_consumed*data_gas_price`, but for v3 transactions the actual `overall_fee` includes an additional `l2_gas_consumed * l2_gas_price` term. A client that follows the schema description to verify or reconstruct the fee will compute a value that is lower than the returned `overall_fee`, with no way to explain the discrepancy from the response alone. Additionally, the missing `l2_gas_consumed` field makes it impossible for clients to correctly populate `l2_gas` resource bounds for v3 transactions, which can cause valid transactions to be rejected at the gateway.

---

### Likelihood Explanation

Any v3 (`AllResources`) transaction that executes Cairo 1 code consumes L2 (Sierra) gas. This is the standard transaction type for Starknet 0.13.3+. The divergence between the schema description and the actual `overall_fee` is therefore triggered by every such transaction. Any client that calls `starknet_estimateFee` and attempts to verify or decompose the returned fee will observe the inconsistency.

---

### Recommendation

1. Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0.into()` in `tx_execution_output_to_fee_estimation`.
2. Add `l2_gas_consumed` and `l2_gas_price` to the `FEE_ESTIMATE` OpenRPC schema and mark them required.
3. Correct the `overall_fee` description to: `"equals to gas_consumed*l1_gas_price + data_gas_consumed*l1_data_gas_price + l2_gas_consumed*l2_gas_price"`.

        let tipped_l2_gas_price =

                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \

                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \

                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
