### Title
Depleted Bonus Pool Unconditionally Blocks All Position Exit Paths, Locking User Staked Tokens — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

Both exit paths for a delegated tier position — `TierUndelegate` and `ExitTierWithDelegation` — unconditionally call `claimRewards` before executing exit logic. `claimRewards` fails atomically when the bonus pool is insufficient. Because there is no user-controlled way to skip or forfeit the bonus claim, a depleted pool locks all staked tokens inside the module with no user-controlled workaround.

---

### Finding Description

In `msg_server.go`, `TierUndelegate` calls `claimRewards` at line 166 before any unbonding logic: [1](#0-0) 

`ExitTierWithDelegation` does the same at line 532: [2](#0-1) 

`claimRewards` calls `processEventsAndClaimBonus`, which calls `sufficientBonusPoolBalance`: [3](#0-2) 

If the pool balance is less than the accrued bonus, `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`, which propagates back through `claimRewards` and causes the entire exit transaction to revert. [4](#0-3) 

The two exit paths for a delegated position are:

1. `TierUndelegate` → wait for unbonding → `WithdrawFromTier`
2. `ExitTierWithDelegation` → instant delegation transfer back to owner

Both are blocked. `WithdrawFromTier` itself does not call `claimRewards`, but it requires the position to already be undelegated (enforced by `validateWithdrawFromTier`), which can only be reached via `TierUndelegate` — itself blocked. [5](#0-4) 

The ADR explicitly documents this behavior as intentional for user-driven paths:

> *Pool empty (user-driven): Message fails atomically. No state change. Retry after pool replenished.*

However, the design provides no user-controlled mechanism to forfeit the bonus and proceed with undelegation. The `BeforeRedelegationSlashed` hook does forfeit bonus silently to avoid chain halt, but no equivalent escape hatch exists for the user-facing exit messages. [6](#0-5) 

---

### Impact Explanation

A user's staked tokens — which may be substantial, given `MinLockAmount` requirements per tier — are locked inside the `x/tieredrewards` module with no user-controlled exit. The only resolution is for an operator or governance to replenish the `RewardsPoolName` module account. Until that happens, `TierUndelegate` and `ExitTierWithDelegation` both revert for every position that has accrued any non-zero bonus. The corrupted invariant is: *a user who has completed their exit commitment period must always be able to recover their principal*.

---

### Likelihood Explanation

The bonus pool is funded externally (e.g., via governance or an inflation allocation). It is a finite resource that depletes as users claim rewards. With multiple positions accruing bonus simultaneously, the pool can reach zero through normal operation — especially if the pool is not continuously topped up. Any position with even one second of accrued bonus will trigger the pool check and be blocked. This is a realistic, non-adversarial scenario.

---

### Recommendation

Decouple the bonus claim from the undelegation/exit logic. Two concrete options:

1. **Forfeit unpaid bonus on exit (preferred):** If `sufficientBonusPoolBalance` fails inside `TierUndelegate` or `ExitTierWithDelegation`, skip the bonus payment (advance the accrual checkpoint without sending coins) and allow the exit to proceed. This mirrors the existing behavior in `BeforeRedelegationSlashed`.

2. **Operator-settable minimum pool threshold:** Require governance to maintain a minimum pool balance before new positions can be created, reducing the likelihood of depletion while positions are still active.

Option 1 is preferred because it gives users a guaranteed exit path regardless of pool state, matching the security invariant that principal recovery must always be possible.

---

### Proof of Concept

1. User calls `MsgLockTier` and creates a delegated position with accrued bonus.
2. The bonus pool is depleted — either naturally (other users claim) or because it was never adequately funded.
3. User's exit commitment elapses; they call `MsgTierUndelegate`.
4. Handler reaches line 166: `pos, _, _, err = ms.claimRewards(ctx, pos)`.
5. `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` → returns `ErrInsufficientBonusPool`.
6. `TierUndelegate` returns the error; no state change occurs.
7. User tries `MsgExitTierWithDelegation` — same failure at line 532.
8. User's staked tokens remain locked in the module indefinitely until an operator replenishes the pool. [7](#0-6) [8](#0-7)

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil
}

func (ms msgServer) WithdrawFromTier(ctx context.Context, msg *types.MsgWithdrawFromTier) (*types.MsgWithdrawFromTierResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateWithdrawFromTier(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
