### Title
Missing Zero-Value Validation on `gc_threshold` in `init` Permanently Misconfigures GC and SPV Proof Verification — (`contract/src/lib.rs`)

---

### Summary

The `init` function accepts `gc_threshold` from `InitArgs` without validating it is non-zero. If initialized with `gc_threshold = 0`, the contract is permanently misconfigured: every `submit_blocks` call triggers aggressive GC that progressively destroys all stored mainchain headers, and `verify_transaction_inclusion` permanently rejects any SPV proof requiring more than zero confirmations. There is no setter for `gc_threshold`, making this irrecoverable without a contract upgrade.

---

### Finding Description

In `contract/src/lib.rs`, the `init` function stores `args.gc_threshold` directly into contract state with no zero-value guard:

```rust
// contract/src/lib.rs line 135-145
pub fn init(args: InitArgs) -> Self {
    let mut contract = Self {
        ...
        gc_threshold: args.gc_threshold,   // ← no require!(args.gc_threshold > 0)
        ...
    };
``` [1](#0-0) 

The `gc_threshold` field is defined in `InitArgs` as a plain `u64` with no constraints: [2](#0-1) 

With `gc_threshold = 0`, two invariants break permanently:

**1. Aggressive GC on every `submit_blocks` call**

`run_mainchain_gc` is called automatically from `submit_blocks` (line 181). Inside it:

```rust
// line 391-393
if amount_of_headers_we_store > self.gc_threshold {          // always true when gc_threshold=0
    let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold; // = all headers
    let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);
``` [3](#0-2) 

Since `amount_of_headers_we_store > 0` is always true (at least one block is stored), every `submit_blocks` call removes `batch_size` headers from the oldest end of the mainchain. Over successive calls the mainchain is stripped down to just the tip, destroying the historical index needed for reorg detection and proof lookups.

**2. SPV proof verification permanently broken**

`verify_transaction_inclusion` enforces:

```rust
// line 289-292
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [4](#0-3) 

With `gc_threshold = 0`, this becomes `require!(args.confirmations <= 0)`. Any caller requesting the standard 6-confirmation SPV proof is permanently rejected. `verify_transaction_inclusion_v2` delegates to this function and is equally broken. [5](#0-4) 

There is no administrative setter for `gc_threshold` anywhere in the contract. The only recovery path is a full contract upgrade via the `Upgradable` pattern.

---

### Impact Explanation

- **SPV proof verification is permanently disabled** for any `confirmations > 0` — the primary consumer-facing function of the contract returns a hard panic for all standard confirmation depths.
- **Mainchain storage is progressively destroyed** on every `submit_blocks` call, eventually leaving only the tip block. This breaks chain reorg detection (`reorg_chain` walks back through `headers_pool` and panics with `PrevBlockNotFound` when ancestors are missing) and all historical block lookups.
- **No recovery path** exists without a privileged contract upgrade, which itself requires the `DAO`/`CodeDeployer` role and a staging delay.

---

### Likelihood Explanation

- `init` is called exactly once during deployment by the contract account (privileged, analogous to Alice deploying Staking.sol).
- A deployer who passes an uninitialized or accidentally zeroed `gc_threshold` (e.g., default-constructed `InitArgs`, copy-paste error, scripting bug) permanently misconfigures the contract.
- The documentation recommends `gc_threshold = 52704` but this is advisory only — no on-chain enforcement exists.
- The scenario is a direct analog to the Staking.sol report: a legitimate privileged actor makes a one-time initialization mistake with no recovery.

---

### Recommendation

Add a non-zero guard in `init` before storing `gc_threshold`:

```rust
require!(args.gc_threshold > 0, "gc_threshold must be non-zero");
```

This should be placed in `init` in `contract/src/lib.rs` before the `Self { ... }` constructor, mirroring the pattern of the existing `require!` checks in `init_genesis`.

Long term, expand initialization tests to include zero-value inputs for all numeric `InitArgs` fields and document the minimum valid value for `gc_threshold`.

---

### Proof of Concept

1. Deploy the contract with `InitArgs { gc_threshold: 0, genesis_block_height: 0, ... }` — `init` succeeds with no error.
2. Call `submit_blocks([block1])` as a trusted relayer. Internally `run_mainchain_gc(1)` fires; since `amount_of_headers_we_store (12+1) > gc_threshold (0)`, it removes 1 block from the oldest end of the mainchain.
3. Repeat step 2 for each new block. After ~12 calls, all genesis-era headers are GC'd; `mainchain_initial_blockhash` now points to a block that may itself be removed on the next call, causing a panic in `run_mainchain_gc` at line 411–414.
4. Independently, call `verify_transaction_inclusion` with `confirmations: 6` at any point — it immediately panics: `"The required number of confirmations exceeds the number of blocks stored in memory"`. [6](#0-5)

    pub fn run_mainchain_gc(&mut self, batch_size: u64) {
        let initial_blockheader = self
            .headers_pool
            .get(&self.mainchain_initial_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let tip_blockheader = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

        let amount_of_headers_we_store =
            tip_blockheader.block_height - initial_blockheader.block_height + 1;

        if amount_of_headers_we_store > self.gc_threshold {
            let total_amount_to_remove = amount_of_headers_we_store - self.gc_threshold;
            let selected_amount_to_remove = std::cmp::min(total_amount_to_remove, batch_size);

            let start_removal_height = initial_blockheader.block_height;
            let end_removal_height = initial_blockheader.block_height + selected_amount_to_remove;
            env::log_str(&format!(
                "Num of blocks to remove {selected_amount_to_remove}"
            ));

            for height in start_removal_height..end_removal_height {
                let blockhash = &self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));

                self.remove_block_header(blockhash);
                self.mainchain_height_to_header.remove(&height);
            }

            self.mainchain_initial_blockhash = self
                .mainchain_height_to_header
                .get(&end_removal_height)
                .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        }
    }
