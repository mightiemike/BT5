### Title
Unbounded Validator Event Iteration in `processEventsAndClaimBonus` Can Permanently Lock Position Funds - (File: `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` iterates over every validator event accumulated since a position's `LastEventSeq` with no upper bound. Because every reward-settling message handler calls `claimRewards` before mutating or exiting a position, a position whose validator has accumulated a large number of events cannot be exited, undelegated, redelegated, or otherwise freed. The user's staked CRO locked in the position's delegator address becomes permanently inaccessible.

---

### Finding Description

`getValidatorEventsSince` in `x/tieredrewards/keeper/validator_events.go` performs an unbounded `Walk` over the `ValidatorEvents` collection, returning every event with sequence number greater than `startSeq`: [1](#0-0) 

The returned slice is consumed in full by `processEventsAndClaimBonus`, which iterates over every entry in a single transaction: [2](#0-1) 

There is no cap on the number of events that can accumulate per validator. `appendValidatorEvent` simply increments a sequence counter and stores the event: [3](#0-2) 

Events are reference-counted by the number of positions active on the validator at event time and are only garbage-collected when every such position has processed them. If many positions exist on a validator, events persist longer, allowing further accumulation.

Every reward-settling message handler calls `claimRewards` (which calls `processEventsAndClaimBonus`) before performing any state mutation:

- `MsgTierUndelegate` calls `claimRewards` before undelegating: [4](#0-3) 
- `MsgTierRedelegate` calls `claimRewards` before redelegating: [5](#0-4) 
- `MsgAddToTierPosition` calls `claimRewards` before adding: [6](#0-5) 
- `MsgClearPosition` calls `claimRewards` before clearing: [7](#0-6) 
- `MsgExitTierWithDelegation` calls `claimRewards` before transferring: [8](#0-7) 

`MsgWithdrawFromTier` (the final fund-recovery path) requires the position to already be undelegated, which requires `MsgTierUndelegate` to succeed first. `MsgTriggerExitFromTier` does not call `claimRewards` and can still be submitted, but it only sets the exit timestamp — the actual fund recovery still requires one of the above handlers to succeed.

There is no mechanism to process validator events for a position in batches across multiple transactions. `processEventsAndClaimBonus` processes the entire pending event list atomically in one call.

---

### Impact Explanation

If a validator accumulates enough events between two consecutive claims by a position (e.g., due to repeated jailing/unjailing cycles or frequent slashing), the transaction processing all those events will exceed the block gas limit. Because every exit path calls `claimRewards` first, the user's staked CRO locked in the position's per-position delegator address (`pos.DelegatorAddress`) becomes permanently inaccessible. The exact corrupted value is the staked CRO balance held by the position's delegator account, which can no longer be undelegated, redelegated, or transferred back to the owner.

---

### Likelihood Explanation

Validators are jailed and unjailed in normal protocol operation (double-sign, downtime). Each jail/unjail cycle produces an `UNBOND` and a `BOND` event. A validator that is jailed and unjailed repeatedly over months — while a position on it has not claimed rewards — accumulates O(jailing cycles) events. There is no protocol-level cap on how many times a validator can be jailed and unjailed. A position owner who is inactive (e.g., lost key access temporarily, or simply did not claim) and whose validator is frequently jailed faces this scenario without any malicious actor involvement. The likelihood is medium: it requires a combination of validator instability and position inactivity, both of which are realistic in production.

---

### Recommendation

Introduce a per-call cap on the number of validator events processed in a single `processEventsAndClaimBonus` invocation, analogous to the `MAX_TOKEN_GUILDBANK_COUNT` fix in the Moloch report. Store the position's `LastEventSeq` checkpoint after each partial processing so that subsequent calls resume from where the previous one stopped. This allows a position owner to drain the event backlog across multiple transactions before executing the desired exit operation. Alternatively, enforce a hard cap on the total number of events stored per validator (e.g., by rejecting new events once the count exceeds a threshold), which bounds the worst-case iteration cost.

---

### Proof of Concept

1. Alice creates a position on validator V via `MsgLockTier`. Her position's `LastEventSeq` is set to the current latest event sequence for V (e.g., `seq=0`).
2. Validator V is jailed and unjailed 10,000 times over the following months. Each cycle appends two events (`UNBOND` at seq 1, 3, 5, … and `BOND` at seq 2, 4, 6, …). Alice's position still has `LastEventSeq=0`.
3. Alice submits `MsgTierUndelegate` to recover her funds.
4. The handler calls `claimRewards` → `processEventsAndClaimBonus` → `getValidatorEventsSince(ctx, V, 0)`, which walks all 20,000 events in a single KV-store iteration.
5. The transaction exhausts the block gas limit and is rejected.
6. Alice retries with a higher gas limit — but the block gas limit is a hard ceiling she cannot exceed.
7. Alice cannot submit `MsgExitTierWithDelegation` (same `claimRewards` call), `MsgTierRedelegate` (same), or `MsgAddToTierPosition` (same). `MsgWithdrawFromTier` requires the position to be undelegated first. Her CRO is permanently locked. [9](#0-8) [1](#0-0)

func (k Keeper) getValidatorEventsSince(ctx context.Context, valAddr sdk.ValAddress, startSeq uint64) ([]EventEntry, error) {
	// Range from (valAddr, startSeq+1) to end of valAddr prefix.
	rng := collections.NewPrefixedPairRange[sdk.ValAddress, uint64](valAddr).
		StartExclusive(startSeq)

	var entries []EventEntry
	err := k.ValidatorEvents.Walk(ctx, rng, func(key collections.Pair[sdk.ValAddress, uint64], event types.ValidatorEvent) (bool, error) {
		entries = append(entries, EventEntry{Seq: key.K2(), Event: event})
		return false, nil
	})
	if err != nil {
		return nil, err
	}
	return entries, nil
}

func (k Keeper) processEventsAndClaimBonus(ctx context.Context, pos *types.PositionState) (sdk.Coins, error) {
	// Rewards should have been claimed before undelegation
	if !pos.IsDelegated() {
		return sdk.NewCoins(), nil
	}

	valAddr, err := sdk.ValAddressFromBech32(pos.Delegation.ValidatorAddress)
	if err != nil {
		return nil, err
	}

	events, err := k.getValidatorEventsSince(ctx, valAddr, pos.LastEventSeq)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	blockTime := sdkCtx.BlockTime()

	totalBonus := math.ZeroInt()
	// Use the persisted bonded state from the last replay, not a hardcoded default.
	// This prevents overpaying bonus for unbonded gaps between claims.
	bonded := pos.LastKnownBonded
	segmentStart := pos.LastBonusAccrual

	tier, err := k.getTier(ctx, pos.TierId)
	if err != nil {
		return nil, err
	}

	for _, entry := range events {
		evt := entry.Event

		if bonded {
			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
			bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
			totalBonus = totalBonus.Add(bonus)
		}

		// Update bonded state based on event type.
		switch evt.EventType {
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_UNBOND:
			bonded = false
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_BOND:
			bonded = true
		case types.ValidatorEventType_VALIDATOR_EVENT_TYPE_SLASH:
			// Slash doesn't change bonded state.
		}

		segmentStart = evt.Timestamp
		pos.UpdateLastEventSeq(entry.Seq)

		// Decrement reference count.
		if err := k.decrementEventRefCount(ctx, valAddr, entry.Seq); err != nil {
			return nil, err
		}
	}
