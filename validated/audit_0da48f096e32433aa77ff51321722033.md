### Title
`unlockedBalanceSum` Not Decremented on NLP Burn Allows NLP Lock Period Bypass — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

`NlpLockedBalanceQueue.unlockedBalanceSum` is a running accumulator that tracks how much NLP a subaccount has unlocked. It is only ever **incremented** when locked NLP matures, but is **never decremented** when NLP is burned via `burnNlp()`. Because `burnNlp()` uses this stale value as the sole gate for the lock-period check, a user who has previously unlocked and burned NLP retains a permanent "unlock credit" that can be used to immediately burn newly minted (still-locked) NLP, bypassing the lock period entirely.

---

### Finding Description

In `SpotEngineState.tryUnlockNlpBalance()`, each time a locked NLP batch matures, its amount is added to `queue.unlockedBalanceSum`:

```solidity
queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
``` [1](#0-0) 

This sum is returned by `getNlpUnlockedBalance()` and is the only check enforcing the lock period in `Clearinghouse.burnNlp()`:

```solidity
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [2](#0-1) 

After the check passes, `burnNlp()` reduces the actual NLP balance via `spotEngine.updateBalance()`:

```solidity
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
``` [3](#0-2) 

`updateBalance()` flows through `_updateBalanceNormalized()` in `SpotEngineState`, which updates `totalDepositsNormalized` / `totalBorrowsNormalized` — but **never touches `unlockedBalanceSum`**. [4](#0-3) 

The result: `unlockedBalanceSum` permanently retains the cumulative total of all NLP ever unlocked for a subaccount, regardless of how much has been burned. It is structurally identical to the `vaultCount` desync: a counter used in a critical gate check is not updated when the related state (actual NLP balance) changes.

---

### Impact Explanation

A user can bypass the NLP lock period and immediately liquidate newly minted (locked) NLP:

1. Mint NLP amount `X` → locked until time `T`.
2. Wait until `T` → `tryUnlockNlpBalance()` sets `unlockedBalanceSum = X`.
3. Burn all `X` NLP → actual balance = 0, but `unlockedBalanceSum` remains `X`.
4. Mint new NLP amount `Y` (where `Y ≤ X`) → locked until `T + lockPeriod`.
5. Immediately call `burnNlp(Y)`:
   - Lock check: `unlockedBalanceSum (X) >= Y` → **passes** (stale value).
   - Balance check: `getBalance(...).amount >= 0` → **passes** (balance was Y, now 0).
6. User has burned locked NLP, receiving quote tokens without waiting for the lock period.

The corrupted state is `unlockedBalanceSum`, which diverges from the true "currently available unlocked NLP." The concrete asset impact is that the attacker extracts quote tokens from the NLP pool ahead of the lock schedule, undermining the economic protection the lock period provides against short-term manipulation of the NLP price. [5](#0-4) 

---

### Likelihood Explanation

**Medium-High.** Any user who has previously completed a full mint-wait-burn cycle holds a permanent unlock credit. No privileged access, no oracle manipulation, and no external dependency is required. The only precondition is having previously unlocked NLP, which is the normal usage path. The attack is repeatable: each new mint of amount `≤ unlockedBalanceSum` can be immediately burned.

---

### Recommendation

Decrement `unlockedBalanceSum` by the burned amount inside `burnNlp()` (or inside `getNlpUnlockedBalance()` after the check). The invariant that must hold is:

```
unlockedBalanceSum == (total ever unlocked) − (total already burned from unlocked)
```

Concretely, after the `spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount)` call in `burnNlp()`, the corresponding `NlpLockedBalanceQueue.unlockedBalanceSum` for `txn.sender` must be reduced by `nlpAmount`. [6](#0-5) 

---

### Proof of Concept

```
State before attack:
  unlockedBalanceSum[alice] = 0
  NLP balance[alice]        = 0

Step 1: alice calls mintNlp(1000 USDC) → 1000 NLP locked until T
  NLP balance[alice] = 1000 (locked)

Step 2: time passes, T reached; tryUnlockNlpBalance() called
  unlockedBalanceSum[alice] = 1000

Step 3: alice calls burnNlp(1000)
  check: unlockedBalanceSum(1000) >= 1000 → PASS
  NLP balance[alice] = 0
  unlockedBalanceSum[alice] = 1000  ← NOT decremented (BUG)

Step 4: alice calls mintNlp(1000 USDC) → 1000 NLP locked until T + lockPeriod
  NLP balance[alice] = 1000 (locked, should not be burnable yet)

Step 5: alice immediately calls burnNlp(1000)
  check: unlockedBalanceSum(1000) >= 1000 → PASS  ← stale value bypasses lock
  balance check: 1000 >= 0 → PASS
  NLP balance[alice] = 0
  alice receives ~1000 USDC worth of quote tokens

Result: alice burned locked NLP without waiting for the lock period.
        unlockedBalanceSum[alice] = 1000 (still stale, attack is repeatable)
```

    function tryUnlockNlpBalance(bytes32 subaccount)
        public
        returns (Balance memory)
    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
            subaccount
        ];
        while (
            queue.unlockedUpTo < queue.balanceCount &&
            queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
        ) {
            // we can unlock this balance
            queue.unlockedBalanceSum.amount += queue
                .balances[queue.unlockedUpTo]
                .balance
                .amount;
            delete queue.balances[queue.unlockedUpTo];
            queue.unlockedUpTo++;
        }

        return queue.unlockedBalanceSum;
    }
