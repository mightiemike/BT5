Looking at the actual code carefully against each claim in the submission.

Audit Report

## Title
Pool Admin Bypasses Timelock Constraint via Zero Timelock at Initialization or Post-Deployment Reduction, Disabling Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`OracleValueStopLossExtension.initialize` decodes a `timelock` parameter from caller-supplied `data` but applies no minimum-value validation, silently accepting `timelock = 0`. With `timelock = 0`, the propose-then-execute pattern collapses to a single block because `_requireElapsed` evaluates `block.timestamp < block.timestamp`, which is always `false`. The same gap exists in `proposeOracleStopLossTimelock`, which also accepts `newTimelock = 0` without a lower-bound check, allowing a pool admin to reduce a previously non-zero timelock to zero after the original delay elapses. In both cases the pool admin can then set `drawdownE6 = 1e6` in one block, making `floorMultiplier = 0` and permanently disabling the stop-loss guard for all future swaps.

## Finding Description

**Root cause — missing timelock validation in `initialize`:**

```solidity
// OracleValueStopLossExtension.sol L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) =
    abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);    // enforced
_validateDecay(decayPerSecondE8); // enforced
// timelock: no _validateTimelock call; zero is silently stored
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8,
    timelock: timelock, initialized: true
});
```

**Root cause — missing lower-bound check in `proposeOracleStopLossTimelock`:**

```solidity
// L78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock)
    external onlyPoolAdmin(pool_)
{
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock; // newTimelock = 0 accepted
    ...
}
```

**Why `_requireElapsed` never reverts when `timelock = 0`:**

`_afterTimelock` returns `block.timestamp + 0 = block.timestamp`. `_requireElapsed` checks `block.timestamp < executeAfter`, i.e., `block.timestamp < block.timestamp`, which is always `false`, so the revert never fires. Propose and execute can be called in the same transaction.

**Why `drawdownE6 = 1e6` disables the guard:**

`_validateDrawdown` only rejects values strictly greater than `E6`; `1e6` passes. In `_afterSwapOracleStopLoss`:

```solidity
// L234
uint256 floorMultiplier = E6 - drawdown; // = 1e6 - 1e6 = 0
```

Inside `_applyWatermark`:

```solidity
// L334
breached = metric < (hwm * floorMultiplier) / E6;
// = metric < 0  (uint256) → always false
```

`_checkAndUpdateWatermarks` never reverts, so `afterSwap` never blocks any swap direction regardless of value drained from LP bins.

**Exploit path (post-deployment, bypassing existing timelock):**

1. Pool is deployed with a legitimate non-zero timelock (e.g., 24 h). LPs deposit.
2. Pool admin calls `proposeOracleStopLossTimelock(pool_, 0)`. `executeAfter = block.timestamp + 24h`.
3. After 24 h, pool admin calls `executeOracleStopLossTimelock(pool_)`. Timelock is now 0.
4. Pool admin calls `proposeOracleStopLossDrawdown(pool_, 1_000_000)`. `executeAfter = block.timestamp + 0 = block.timestamp`.
5. In the same transaction, pool admin calls `executeOracleStopLossDrawdown(pool_)`. `_requireElapsed(block.timestamp)` → no revert. `drawdownE6 = 1_000_000`.
6. All subsequent `afterSwap` calls compute `floorMultiplier = 0`; `breached` is always `false`; stop-loss never fires. LP bins can be drained without revert.

## Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism preventing LP principal from being drained through adversarial or oracle-manipulated swaps. Disabling it via `drawdownE6 = 1e6` means every subsequent swap executes without the stop-loss check, allowing LP token0 and token1 principal to be drained with no on-chain recourse. This is a direct loss of user principal and constitutes an admin-boundary break: the pool admin exceeds the constraint the timelock is specifically designed to impose.

## Likelihood Explanation

The post-deployment path requires only pool-admin privilege and patience equal to the current timelock duration — no external dependencies, no oracle manipulation, no special tokens. The initialization path requires no waiting at all. Both paths are reachable in production because no factory-level validation of the `timelock` field exists and `proposeOracleStopLossTimelock` imposes no lower bound on `newTimelock`. The pool admin is the semi-trusted role the timelock is explicitly designed to constrain; bypassing it is the exact admin-boundary break the contest rules flag as in-scope.

## Recommendation

Add a minimum-timelock guard applied in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 hours;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock < MIN_TIMELOCK)
        revert OracleStopLossTimelockTooShort(timelock);
}
```

Call `_validateTimelock(timelock)` alongside `_validateDrawdown` / `_validateDecay` in `initialize`, and call `_validateTimelock(newTimelock)` inside `proposeOracleStopLossTimelock` before storing `pendingTimelock`. Additionally, consider whether `_validateDrawdown` should reject `drawdownE6 == E6` (100 %), since that value makes `floorMultiplier = 0` and silently disables the guard even with a valid timelock.

## Proof of Concept

**Initialization path (single block):**

1. Factory deploys pool; pool admin supplies `abi.encode(500_000, 58, 0)` → `timelock = 0`. No revert.
2. LPs deposit.
3. Pool admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)`. `executeAfter = block.timestamp`.
4. Same transaction: `executeOracleStopLossDrawdown(pool)`. `_requireElapsed(block.timestamp)` → `false` → no revert. `drawdownE6 = 1_000_000`.
5. `floorMultiplier = 0`. All future `afterSwap` calls pass without stop-loss check. LP principal drainable.

**Post-deployment path (bypasses existing timelock):**

1. Pool starts with `timelock = 86400` (24 h). LPs deposit trusting the guard.
2. Pool admin calls `proposeOracleStopLossTimelock(pool, 0)`.
3. After 24 h: `executeOracleStopLossTimelock(pool)` → `timelock = 0`.
4. Pool admin calls `proposeOracleStopLossDrawdown(pool, 1_000_000)` and `executeOracleStopLossDrawdown(pool)` in the same block. No revert.
5. Stop-loss permanently disabled; LP principal exposed.
