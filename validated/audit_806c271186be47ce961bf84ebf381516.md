### Title
Mainchain-Only `get_header_by_height` Used for Fork Difficulty Calculation Allows Incorrect PoW Target Acceptance - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

`get_header_by_height` always resolves blocks from the mainchain index (`mainchain_height_to_header`). When a fork block is being validated at or after a difficulty-adjustment boundary, the retarget calculation fetches the wrong ancestor block — the mainchain block at that height instead of the fork's actual ancestor. This allows an attacker-controlled fork to pass difficulty verification with an incorrect (lower) PoW target, enabling chain reorganization with less cumulative work than the protocol requires.

---

### Finding Description

`get_header_by_height` in `contract/src/lib.rs` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

This unconditionally reads from `mainchain_height_to_header`. Fork blocks are stored via `store_fork_header`, which only inserts into `headers_pool` — never into `mainchain_height_to_header` or `mainchain_header_to_height`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [2](#0-1) 

All three chain implementations call `get_header_by_height` to fetch the retarget-boundary block during difficulty calculation:

**Dogecoin** (with an explicit TODO acknowledging the problem):
```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

**Bitcoin:**
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header, interval_tail_extend_header.block_header.time.into())
``` [4](#0-3) 

**Litecoin:**
```rust
let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(config, prev_block_header, interval_tail_extend_header.block_header.time.into())
``` [5](#0-4) 

The difficulty calculation uses the timestamp of the retarget-boundary block. For Bitcoin/Litecoin:

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
``` [6](#0-5) 

When a fork diverges before height `H` (a retarget boundary), the fork's block at `H` can have a different timestamp than the mainchain block at `H`. For all blocks in the fork at heights `> H`, the contract fetches the **mainchain** block at `H` as `first_block_time`, not the fork's actual ancestor at `H`. This produces an incorrect `actual_time_taken`, yielding a wrong expected `bits` value.

---

### Impact Explanation

If the fork's retarget-boundary block carries a **later** timestamp than the mainchain block at the same height, the contract computes a longer `actual_time_taken` → a higher target → a **lower expected difficulty** for all subsequent fork blocks. The attacker mines those blocks at this reduced difficulty. The blocks pass `check_pow` because `expected_bits == block_header.bits` is satisfied against the incorrectly computed value. If the fork accumulates sufficient (but artificially cheap) chainwork, `submit_block_header_inner` triggers `reorg_chain`, replacing the canonical chain with the attacker's fork. Downstream consumers of `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` then operate against a fraudulent chain tip. [7](#0-6) 

---

### Likelihood Explanation

The attack requires submitting a fork that diverges one block before a difficulty-adjustment boundary, then submitting a fork block at the boundary with a manipulated timestamp. The timestamp must still satisfy the MTP check and the future-time check, but both allow a window of several hours. The attacker then mines subsequent fork blocks at the reduced difficulty. This is reachable by any unprivileged caller of `submit_blocks` (no role required beyond the trusted-relayer stake, which is a financial barrier, not a cryptographic one). The `skip_pow_verification = false` production configuration is the only prerequisite. [8](#0-7) 

---

### Recommendation

Replace the height-based lookup with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from the fork tip, so the correct fork ancestor is always used regardless of what the mainchain index contains at that height. The Zcash implementation already does this correctly by using `get_prev_header` in a loop rather than `get_header_by_height`. [9](#0-8) 

---

### Proof of Concept

1. Initialize the contract with a mainchain at height `H` (a difficulty-adjustment boundary, e.g., `H % 2016 == 0` for Bitcoin).
2. Extend the mainchain to height `H + 2016` (the next retarget boundary). The mainchain block at `H` has timestamp `T_main`.
3. Submit a fork block at height `H` whose `prev_block_hash` points to the mainchain block at `H-1`. Set its timestamp to `T_fork > T_main` (within the MTP/future-time window). This block's difficulty is checked against the previous retarget interval (same as mainchain), so it passes.
4. Submit fork blocks at heights `H+1` through `H+2016`. For each, `get_next_work_required` calls `get_header_by_height(H)`, which returns the **mainchain** block at `H` with timestamp `T_main`, not the fork block with `T_fork`.
5. `actual_time_taken` is computed as `T_{H+2015} - T_main` instead of `T_{H+2015} - T_fork`. Since `T_main < T_fork`, the computed timespan is longer, yielding a higher target (lower difficulty).
6. The attacker mines fork blocks satisfying this lower difficulty. Each passes `check_pow`.
7. Once the fork's `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain.
