Audit Report

## Title
SwapAllowlistExtension Gates the Router Address Instead of the Actual Swapper, Allowing Any User to Bypass the Swap Allowlist via the Router - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` as the pool's `msg.sender`, which equals the router contract address when a user routes through `MetricOmmSimpleRouter`. If the pool admin allowlists the router address to permit router-mediated swaps, the check `allowedSwapper[pool][router]` passes for every caller of the router, completely neutralizing the allowlist for any user who routes through the public, permissionless router contract.

## Finding Description

**Root cause — `pool.swap()` passes `msg.sender` as `sender`:**

In `MetricOmmPool.swap()` (lines 230–240), `msg.sender` is passed directly as the `sender` argument to `_beforeSwap`. When `MetricOmmSimpleRouter.exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` (line 72–80 of `MetricOmmSimpleRouter.sol`), `msg.sender` inside the pool is the router contract, not the originating EOA. The router does not encode the original caller into `extensionData`.

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` (lines 149–177) passes `sender` verbatim into `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`. No transformation or enrichment occurs.

**The allowlist check operates on the wrong identity:**

`SwapAllowlistExtension.beforeSwap` (lines 37–38) evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender])
```
Here `msg.sender` = pool (correct key), `sender` = router (wrong identity). When the admin calls `setAllowedToSwap(pool, router, true)` to permit router-mediated swaps for legitimate users, the mapping `allowedSwapper[pool][router]` becomes `true`, and the check passes for **every** caller of the router regardless of their actual identity.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict trading.
2. Admin calls `setAllowedToSwap(pool, router, true)` so allowlisted users can use the router.
3. Unprivileged EOA (not in the allowlist) calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
4. Router calls `pool.swap(...)` → pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Unauthorized swap executes in full.

No existing guard prevents this: `allowAllSwappers` is a separate bypass flag, and there is no mechanism in the extension or the pool to recover the originating EOA from the call context.

## Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade in a pool. Any pool that (a) uses `SwapAllowlistExtension` and (b) allowlists the router — a natural and expected configuration — is fully exposed. Unauthorized actors can drain LP liquidity at oracle-anchored prices, causing direct loss of LP principal. This matches the allowed impact gate: **admin-boundary break — factory/oracle role checks bypassed by an unprivileged path**, and constitutes a High severity finding under Sherlock thresholds given the direct fund loss to LPs.

## Likelihood Explanation

The bypass requires no special privileges, no malicious setup, and no non-standard tokens. The router is a public, permissionless contract. Any pool using `SwapAllowlistExtension` that allowlists the router (the natural configuration for usability) is fully exposed to any EOA. The attack is repeatable and requires only a single `exactInputSingle` call.

## Recommendation

The extension must gate the **originating user**, not the immediate caller of `pool.swap()`. Preferred fix: require the router to encode the original `msg.sender` into `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that value when `sender` is a known trusted router. Alternatively, maintain a registry of trusted routers in the extension; if `sender` is a trusted router, decode the true initiator from `extensionData` and check `allowedSwapper[pool][initiator]`. Using `tx.origin` is only acceptable if the pool is never used in contract-to-contract flows. The `DepositAllowlistExtension` should be audited for the same pattern on the `beforeAddLiquidity` path.

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true).
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true).
  - Pool admin does NOT call setAllowAllSwappers(pool, true).

Attack:
  attacker (unlisted EOA) calls:
    MetricOmmSimpleRouter.exactInputSingle(ExactInputSingleParams{
        pool:    <restricted pool>,
        tokenIn: token0,
        ...
    });

  // MetricOmmSimpleRouter.sol L72-80:
  // Router calls pool.swap(recipient, zeroForOne, amount, priceLimit, "", extensionData)
  // pool.swap() → msg.sender = router → _beforeSwap(sender=router, ...)
  // SwapAllowlistExtension.beforeSwap(sender=router, ...)
  // checks: allowedSwapper[pool][router] == true → passes
  // Swap executes for the attacker.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds; attacker receives tokens from a pool they are not authorized to trade in.
```
