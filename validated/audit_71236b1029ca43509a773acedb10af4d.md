Looking at the `ClaimTierRewards` handler and `claimRewardsAndUpdatesPositions` in the tieredrewards module, I can identify a concrete analog.

### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claiming of Bonus Rewards - (File: `x/tieredrewards/keeper/msg_server.go`)

### Summary
The `ClaimTierRewards` message handler loads all requested positions into a slice before processing any of them, with no deduplication of `msg.PositionIds`. An owner can submit the same position ID twice in a single `MsgClaimTierRewards` transaction. The second iteration processes a stale in-memory copy of the position (with the old `LastEventSeq` checkpoint), causing `processEventsAndClaimBonus` to re-fetch and re-process the same validator events and send bonus coins a second time from the `RewardsPoolName` module account.

### Finding Description

`ClaimTierRewards` first loads all positions into a slice, then delegates to `claimRewardsAndUpdatesPositions`: [1](#0-0) 

The load loop appends a fresh copy of the position state for every entry in `msg.PositionIds`, including duplicates: [2](#0-1) 

`claimRewardsAndUpdatesPositions` then iterates over the slice. For the first occurrence of position P it calls `processEventsAndClaimBonus`, which advances `pos.LastEventSeq` in-memory and persists the updated checkpoint via `setPosition`: [3](#0-2) 

For the second occurrence of position P (a separate in-memory copy still holding the old `LastEventSeq`), `processEventsAndClaimBonus` calls `getValidatorEventsSince` with the stale checkpoint, re-fetches the same events, decrements their reference counts a second time, and issues a second `SendCoinsFromModuleToAccount` transfer of bonus coins: [4](#0-3) 

The `decrementEventRefCount` call inside the event loop will only fail (and revert the transaction) if the reference count for a given event has already reached zero — i.e., if the attacker's position is the *only* position that references those events. In any live network with multiple positions on the same validator, the reference count for shared events is ≥ 2, so both decrements succeed and the double-payment goes through. [5](#0-4) 

The root cause mirrors the external report exactly: the "already processed" state (`LastEventSeq` checkpoint) is written to the store only *after* the token transfer, and the pre-loaded stale copy bypasses that write for the duplicate entry — a checks-effects-interactions violation.

### Impact Explanation
An owner drains the `RewardsPoolName` module account by repeatedly submitting `MsgClaimTierRewards` with the same position ID listed N times per transaction. Each extra occurrence yields a full bonus-reward payment for the same accrual window. The corrupted values are:
- `RewardsPoolName` module account balance (drained faster than intended)
- `LastEventSeq` / `LastBonusAccrual` checkpoints (event reference counts decremented extra times, potentially corrupting the event GC accounting for other positions on the same validator)

### Likelihood Explanation
Any position owner can craft this transaction with zero privilege. The only precondition — that at least one other position exists on the same validator so event reference counts are ≥ 2 — is trivially satisfied on a live network. The attacker can also create a second position themselves to satisfy this condition before executing the exploit.

### Recommendation
Add a deduplication check on `msg.PositionIds` inside `MsgClaimTierRewards` (or inside `Validate()` in `msgs.go`) before the positions are loaded. A simple seen-set over position IDs is sufficient:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(types.ErrInvalidPositionID, "duplicate position id %d", posId)
    }
    seen[posId] = struct{}{}
    // ... existing load logic
}
```

Alternatively, follow the checks-effects-interactions pattern by persisting the updated position checkpoint *before* sending bonus coins in `processEventsAndClaimBonus`.

### Proof of Concept

1. Attacker owns position P (validator V, `LastEventSeq = A`). At least one other position Q also exists on V (so event ref counts ≥ 2).
2. Bonus rewards accrue; events A→B are recorded for V.
3. Attacker submits `MsgClaimTierRewards { owner: attacker, position_ids: [P, P] }`.
4. **Iteration 0** (`positions[0]`, `LastEventSeq = A`): `processEventsAndClaimBonus` fetches events A→B, decrements ref counts (2→1), sends bonus coins to attacker, `setPosition` persists `LastEventSeq = B`.
5. **Iteration 1** (`positions[1]`, stale `LastEventSeq = A`): `processEventsAndClaimBonus` fetches events A→B again (ref counts now 1→0), sends the same bonus coins to attacker a second time, `setPosition` persists `LastEventSeq = B` again (no revert).
6. Net result: attacker receives 2× the bonus rewards for a single accrual window; `RewardsPoolName` balance is reduced by 2× the correct amount.

func (ms msgServer) ClaimTierRewards(ctx context.Context, msg *types.MsgClaimTierRewards) (*types.MsgClaimTierRewardsResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	positions := make([]types.PositionState, 0, len(msg.PositionIds))
	for _, posId := range msg.PositionIds {
		pos, err := ms.getPositionState(ctx, posId)
		if err != nil {
			return nil, err
		}

		if err := ms.validateClaimRewards(pos.Position, msg.Owner); err != nil {
			return nil, err
		}

		positions = append(positions, pos)
	}

	totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
	if err != nil {
		return nil, err
	}

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
}
