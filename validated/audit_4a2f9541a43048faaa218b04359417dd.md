### Title
Filler Loses Execution Control After Fee Commitment in `submitFastWithdrawal`, Enabling Gas-Funded Exploitation by Recipient Contract — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

In `BaseWithdrawPool.submitFastWithdrawal`, the filler irrevocably pays a fee before tokens are transferred to `sendTo`. If the output token implements ERC-777 or any transfer hook, `sendTo` receives execution control after the filler has already committed their fee and cannot revert. The filler has no final checkpoint to abort. A malicious `sendTo` can perform arbitrary gas-intensive operations funded by the filler's gas budget, making the transaction unprofitable for the filler. Additionally, because `submitFastWithdrawal` has no reentrancy guard and `minIdx` is not updated within it, a reentrant call with a different pre-signed `idx` can drain other users' funds from the WithdrawPool.

---

### Finding Description

`submitFastWithdrawal` executes in this order:

1. **Line 88** — `markedIdxs[idx] = true` (replay guard for this specific `idx`)
2. **Line 108** — `safeTransferFrom(token, msg.sender, uint128(fee))` — fee irrevocably pulled from filler
3. **Line 111** — `fees[productId] += fee` — fee recorded
4. **Line 113** — `handleWithdrawTransfer(token, sendTo, transferAmount)` — tokens sent to `sendTo` [1](#0-0) 

`handleWithdrawTransfer` resolves to a bare `token.safeTransfer(to, uint256(amount))`: [2](#0-1) 

`safeTransfer` is implemented as a low-level `.call` with no reentrancy protection: [3](#0-2) 

If the token is ERC-777 (or any token with a `transfer` hook), `sendTo` receives a `tokensReceived` callback at step 4. At this point:

- The filler has already paid the fee (step 2) and **cannot revert**.
- There is **no reentrancy guard** on `submitFastWithdrawal`.
- `minIdx` is **not updated** inside `submitFastWithdrawal` — it is only updated in `submitWithdrawal` (line 129). So any pre-signed withdrawal with `idx > minIdx` remains valid for a reentrant call. [4](#0-3) 

A malicious `sendTo` contract can therefore:

1. During the callback, call `submitFastWithdrawal` again with a different pre-signed `idx` (e.g., `idx = N+1`).
2. This second call passes all guards (`!markedIdxs[N+1]`, `N+1 > minIdx`).
3. The WithdrawPool transfers a second batch of tokens to `sendTo` — drawn from **other users' pending withdrawal funds** held in the pool.
4. The filler of the original call has no control over this reentrant execution.

In `WithdrawCollateralV2`, the user controls `sendTo` directly: [5](#0-4) 

This makes deploying a malicious recipient trivial — the attacker signs a V2 withdrawal pointing `sendTo` at their hook contract.

---

### Impact Explanation

- **Financial loss for the filler**: The filler cannot accurately estimate gas before calling `submitFastWithdrawal` because `sendTo` can execute arbitrary logic during the callback. If the callback is gas-intensive, the filler's total gas cost exceeds the fee they collected, resulting in a net loss.
- **Theft from the WithdrawPool**: Via the reentrant `submitFastWithdrawal` call with a fresh `idx`, the attacker can withdraw a second tranche of tokens from the pool — tokens belonging to other users' pending withdrawals — without any corresponding balance deduction in `SpotEngine`.
- The corrupted state is: `WithdrawPool` token balance decreases by more than the attacker's legitimate entitlement; other users' pending withdrawals become underfunded.

---

### Likelihood Explanation

- **Precondition**: The registered token for the targeted `productId` must implement ERC-777 or a transfer hook. The protocol controls token registration, so standard ERC-20 tokens are safe. However, if any ERC-777-compatible token is ever listed, this path is immediately reachable.
- **Attacker capability**: Any user can sign a `WithdrawCollateralV2` transaction with `sendTo` pointing to a malicious contract. No privileged access is required.
- **Pre-signed second withdrawal**: The attacker only needs to have signed a second withdrawal transaction with a valid `idx > minIdx` before the attack — a normal user action.
- Likelihood is **medium-low** given the ERC-777 precondition, but the impact is severe enough to warrant mitigation.

---

### Recommendation

1. Add OpenZeppelin's `ReentrancyGuard` (`nonReentrant`) to `submitFastWithdrawal` in `BaseWithdrawPool`.
2. Follow the checks-effects-interactions pattern: update `fees[productId]` and all state before any external token call.
3. Consider adding a post-transfer callback to the filler (analogous to the UniswapX recommendation) so the filler can revert if the net outcome is unprofitable.
4. Alternatively, update `minIdx` inside `submitFastWithdrawal` (not just `submitWithdrawal`) to close the reentrant-idx window.

---

### Proof of Concept

```
1. Attacker deploys MaliciousRecipient implementing ERC-777 tokensReceived hook.
2. Attacker signs two WithdrawCollateralV2 transactions:
     - tx_A: productId=1, sendTo=MaliciousRecipient, amount=100, idx=10
     - tx_B: productId=1, sendTo=MaliciousRecipient, amount=50,  idx=11
   (idx=11 > minIdx, markedIdxs[11]=false at this point)
3. Filler calls submitFastWithdrawal(10, tx_A, sigs_A).
4. Line 108: fee pulled from filler — filler is now committed.
5. Line 113 → handleWithdrawTransfer → token.safeTransfer(MaliciousRecipient, 100).
6. ERC-777 hook fires inside MaliciousRecipient.tokensReceived():
     a. MaliciousRecipient approves WithdrawPool to spend its tokens (for fee).
     b. MaliciousRecipient calls submitFastWithdrawal(11, tx_B, sigs_B).
        - markedIdxs[11]=false ✓, idx=11 > minIdx ✓ → passes guards.
        - Fee pulled from MaliciousRecipient (negligible cost to attacker).
        - WithdrawPool transfers 50 tokens to MaliciousRecipient from pool reserves
          (other users' pending withdrawal funds).
7. Outer submitFastWithdrawal(10) resumes — filler cannot revert.
8. Net result: MaliciousRecipient holds 150 tokens; only 100 was legitimately owed.
   Other users' 50-token withdrawal is now underfunded in the WithdrawPool.

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
