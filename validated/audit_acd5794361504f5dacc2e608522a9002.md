### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the End-User, Making the Per-User Allowlist Either Permanently Broken or Fully Bypassable for Router-Mediated Swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of the pool's `swap` call. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` of the pool is the **router contract**, not the end-user. The extension therefore checks whether the **router** is allowlisted, not whether the **individual user** is allowlisted. This creates an irresolvable two-sided failure: if the router is not allowlisted, all allowlisted users are silently blocked from using the standard periphery; if the router is allowlisted to fix that, every non-allowlisted user can bypass the guard by routing through it.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. When `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) is used, it calls `pool.swap(...)` directly: [4](#0-3) 

So `sender` arriving at the extension is the **router's address**, not the end-user's address. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The pool admin has two choices, both broken:

| Admin action | Effect |
|---|---|
| Do **not** allowlist the router | Every allowlisted user is blocked from using the standard periphery; they must call `pool.swap()` directly |
| Allowlist the router | Every non-allowlisted user can bypass the guard by routing through `MetricOmmSimpleRouter` |

Neither option delivers the intended semantics ("only these specific addresses may swap").

---

### Impact Explanation

**Broken core swap flow (router path):** Allowlisted users who rely on `MetricOmmSimpleRouter` for exact-input or exact-output swaps receive `NotAllowedToSwap` even though they are individually permitted. The standard periphery entry point is unusable for any pool that deploys `SwapAllowlistExtension` without also allowlisting the router.

**Allowlist bypass (router allowlisted):** If the pool admin allowlists the router to restore router functionality, the guard is completely neutralised. Any address — including those the admin explicitly excluded — can call `MetricOmmSimpleRouter.exactInputSingle` and swap against the pool. The pool's LP providers are exposed to trades from counterparties the pool admin intended to block, which can include adversarial actors the allowlist was designed to exclude.

Both outcomes are direct consequences of the same root cause and satisfy the "broken core pool functionality" and "admin-boundary break" impact gates.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary user-facing swap entry point; virtually all non-technical users will route through it.
- Pool admins are semi-trusted and are not warned anywhere in the extension's NatSpec or interface that router-mediated swaps present the router address as the swapper identity.
- The `SwapAllowlistExtension` NatSpec states it "Gates `swap` by swapper address, per pool" with no caveat about intermediary contracts.
- The `IMetricOmmPoolActions` NatSpec for `NotAllowedToSwap` says "Only `swap` checks this when `SWAP_ALLOWLIST_PROVIDER` is set; `simulateSwapAndRevert` does not, so a passing simulation does not imply an allowed live swap" — it does not mention the router identity problem either. [5](#0-4) 

Any pool that deploys this extension and expects per-user swap gating will be misconfigured by default.

---

### Recommendation

The extension must receive the **economic actor** (the end-user), not the intermediary. Two complementary fixes:

1. **Router-side**: `MetricOmmSimpleRouter` should accept a `swapper` parameter (defaulting to `msg.sender`) and encode it into `extensionData` so extensions can decode the real user identity.

2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should decode a `swapper` address from `extensionData` when present and fall back to `sender` only when the field is absent. This preserves backward compatibility for direct pool calls while enabling correct gating through the router.

Alternatively, the pool can expose a dedicated allowlist check on `msg.sender` of the pool (i.e., the router) separately from the end-user check, but this requires the router to be a trusted, non-public forwarder — which it is not.

---

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is individually permitted
  allowedSwapper[pool][router] = false // router not allowlisted

Scenario A — legitimate user blocked:
  alice calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  → router calls pool.swap(recipient, ...)
  → pool calls _beforeSwap(msg.sender=router, ...)
  → extension checks allowedSwapper[pool][router] == false
  → revert NotAllowedToSwap
  alice cannot use the standard periphery despite being allowlisted.

Scenario B — guard bypass (admin allowlists router to fix A):
  admin sets allowedSwapper[pool][router] = true
  bob (not individually allowlisted) calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
  → router calls pool.swap(recipient, ...)
  → pool calls _beforeSwap(msg.sender=router, ...)
  → extension checks allowedSwapper[pool][router] == true
  → passes; bob swaps successfully despite not being allowlisted.
``` [3](#0-2) [6](#0-5) [7](#0-6)
