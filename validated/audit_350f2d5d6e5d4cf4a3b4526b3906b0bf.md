### Title
Upgradeable Collateral Token Introducing Fee-on-Transfer or Decimal Change Corrupts Protocol Solvency Accounting — (File: `core/contracts/Clearinghouse.sol`, `core/contracts/Endpoint.sol`)

---

### Summary

The Nado protocol registers collateral tokens by address with no check for whether the token is an upgradeable proxy. If a listed collateral token's implementation is upgraded to introduce fee-on-transfer behavior, or to change its `decimals()` return value, the protocol's internal accounting will diverge from actual token holdings. This corrupts solvency invariants, allows borrowing against phantom collateral, and can permanently lock depositor funds.

---

### Finding Description

**Token registration has no upgradeability guard.**

`SpotEngine.addOrUpdateProduct()` stores the token address in `configs[productId].token` with no check for whether the token is a proxy or has an upgradeable implementation. [1](#0-0) 

**Deposit credits user-specified `amount`, not actual received amount.**

`Endpoint.depositCollateralWithReferral()` calls `handleDepositTransfer()` with the user-supplied `amount`, then enqueues a `DepositCollateral` slow-mode transaction carrying that same `amount`. [2](#0-1) 

`Clearinghouse.depositCollateral()` then credits `amountRealized = int128(txn.amount) * int128(multiplier)` directly to the user's balance — using the user-specified `amount`, not the actual tokens received by the contract. [3](#0-2) 

If the token's implementation is upgraded to introduce a fee-on-transfer, the Clearinghouse receives `amount - fee` tokens but credits the full `amount` to the user. The gap is phantom collateral that does not exist on-chain.

**`_decimals()` is called live on every deposit and withdrawal.**

`Clearinghouse._decimals()` calls `token.decimals()` at runtime on every deposit, withdrawal, and minimum-deposit check — it is never cached at product registration time. [4](#0-3) 

If a proxy token's implementation is upgraded to change its `decimals()` return value (e.g., from 6 to 18), the `multiplier = 10^(MAX_DECIMALS - decimals)` used to normalize amounts changes immediately. Existing balances were stored under the old multiplier, but new deposits and withdrawals use the new multiplier, causing a permanent accounting mismatch.

**`assertUtilization` does not verify actual token holdings.**

The only on-chain solvency check, `SpotEngine.assertUtilization()`, verifies `totalDeposits >= totalBorrows` in internal normalized accounting — it does not compare internal accounting against the actual ERC20 balance held by the Clearinghouse. [5](#0-4) 

**`ERC20Helper.safeTransfer` / `safeTransferFrom` do not verify received amounts.**

Both helpers perform a low-level call and check the boolean return value, but do not measure the actual balance delta at the recipient. A fee-on-transfer upgrade silently passes these checks. [6](#0-5) 

---

### Impact Explanation

- **Phantom collateral inflation**: Every deposit after a fee-on-transfer upgrade credits more collateral than was received. The internal `totalDepositsNormalized` grows faster than actual token holdings.
- **Borrowing against phantom collateral**: Health calculations in `Clearinghouse.getHealth()` use inflated internal balances, allowing users to borrow or open positions against collateral that does not exist.
- **Locked funds / withdrawal failure**: When users attempt to withdraw, `Clearinghouse.withdrawCollateral()` calls `token.safeTransfer(withdrawPool, amount)`. Once actual token holdings are exhausted, all subsequent withdrawals revert, permanently locking remaining depositors' funds.
- **Broken liquidations**: Liquidators computing health against inflated balances cannot correctly identify insolvent subaccounts, leaving bad loans unresolvable. [7](#0-6) 

---

### Likelihood Explanation

Medium. The trigger requires two conditions: (1) a listed collateral token is an upgradeable proxy (USDC, USDT, and many other major stablecoins are), and (2) its implementation is upgraded to introduce fee-on-transfer or a decimal change. Condition (1) is already true for the most commonly used collateral assets in DeFi. Condition (2) is an external event outside Nado's control. No privileged Nado role is required to trigger the impact once the token upgrade occurs — any ordinary depositor calling `depositCollateral` after the upgrade is sufficient.

---

### Recommendation

1. **Cache decimals at product registration time.** Store `token.decimals()` in `Config` at the time `addOrUpdateProduct` is called and use the cached value in all subsequent computations. This prevents a decimal change from corrupting the multiplier mid-lifecycle.

2. **Use a balance-delta pattern for deposits.** In `handleDepositTransfer`, record `balanceBefore = token.balanceOf(address(this))`, perform the transfer, then compute `actualReceived = token.balanceOf(address(this)) - balanceBefore`. Pass `actualReceived` — not the user-supplied `amount` — to the slow-mode `DepositCollateral` transaction.

3. **Implement an upgrade-detection freeze.** Maintain a stored code hash of each collateral token's implementation (analogous to how `ProxyManager` tracks Nado's own contract hashes). Before any deposit or withdrawal, verify the token's implementation hash has not changed; if it has, freeze interactions with that product until governance reviews the upgrade. [8](#0-7)

### Proof of Concept

```
Setup:
  - USDC (upgradeable proxy, 6 decimals) is listed as productId=1 in SpotEngine.
  - Alice deposits 1000 USDC (1_000_000 raw units).
  - Clearinghouse receives 1_000_000 raw units.
  - Clearinghouse.depositCollateral credits Alice:
      multiplier = 10^(18-6) = 1e12
      amountRealized = 1_000_000 * 1e12 = 1e18  ✓ correct

Attack (fee-on-transfer upgrade):
  - USDC implementation is upgraded to charge a 1% transfer fee.
  - Bob deposits 1000 USDC (1_000_000 raw units).
  - ERC20Helper.safeTransferFrom succeeds (returns true).
  - Clearinghouse actually receives 990_000 raw units (1% fee deducted).
  - BUT: slow-mode DepositCollateral carries amount=1_000_000.
  - Clearinghouse.depositCollateral credits Bob:
      amountRealized = 1_000_000 * 1e12 = 1e18  ← phantom 10_000 units credited

Result:
  - Internal totalDepositsNormalized reflects 2000 USDC.
  - Actual token balance in Clearinghouse = 1_990_000 raw units = 1990 USDC.
  - 10 USDC of phantom collateral exists in the system.
  - Bob can borrow against the phantom 10 USDC.
  - After enough such deposits, the last withdrawers find the Clearinghouse
    balance insufficient and their withdrawals revert permanently.
  - assertUtilization() passes throughout because it only checks
    totalDeposits >= totalBorrows in internal accounting.
``` [9](#0-8) [10](#0-9)

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
