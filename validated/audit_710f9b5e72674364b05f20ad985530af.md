### Title
Fee-on-Transfer Token Deposit Credits User-Specified Amount Instead of Actual Received Amount, Corrupting Clearinghouse Solvency — (`File: core/contracts/EndpointStorage.sol`)

---

### Summary

`Endpoint.depositCollateralWithReferral` transfers tokens using the caller-supplied `amount` and enqueues a slow-mode `DepositCollateral` transaction recording that same `amount`. `Clearinghouse.depositCollateral` later credits the subaccount with the full user-specified `amount`. No step measures the actual tokens received. If a fee-on-transfer token is registered as a collateral product, the Clearinghouse's real token balance grows by `amount − fees` while every depositor's subaccount is credited the full `amount`, creating an ever-growing deficit that eventually makes the protocol insolvent.

---

### Finding Description

The deposit pipeline has three steps, none of which reconcile the user-specified amount against the actual received amount:

**Step 1 — `Endpoint.depositCollateralWithReferral` (Endpoint.sol:144–164)**

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)          // ← user-supplied, never re-measured
);
...
DepositCollateral({
    sender: subaccount,
    productId: productId,
    amount: amount           // ← same user-supplied value enqueued
})
``` [1](#0-0) 

**Step 2 — `EndpointStorage.handleDepositTransfer` (EndpointStorage.sol:111–119)**

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);              // pull from user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // push Endpoint → Clearinghouse
}
```

Both calls use the original `amount`. There is no balance snapshot before/after either transfer to verify what was actually received. [2](#0-1) 

`ERC20Helper.safeTransferFrom` only checks the boolean return value of `transferFrom`; it does not assert that the contract's balance increased by exactly `amount`. [3](#0-2) 

**Step 3 — `Clearinghouse.depositCollateral` (Clearinghouse.sol:193–208)**

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
```

`txn.amount` is the value that was serialised in Step 1 — the original user-supplied figure — not the tokens that physically arrived. [4](#0-3) 

---

### Impact Explanation

With a fee-on-transfer collateral token, every deposit inflates the credited subaccount balance relative to the Clearinghouse's real holdings. The invariant

> `Σ subaccount balances (product P)` ≤ `Clearinghouse.token_P.balanceOf(clearinghouse)`

is broken on every deposit. Accumulated over many deposits, the Clearinghouse becomes insolvent: it cannot honour all withdrawals, and later withdrawers suffer direct, permanent loss of funds. The `DirectDepositV1.creditDeposit` path is equally affected because it reads `balance = token.balanceOf(address(this))` and passes that as `amount` to `depositCollateralWithReferral`, which then executes the same two-hop transfer without re-measuring. [5](#0-4) 

---

### Likelihood Explanation

The protocol does not whitelist token behaviour; any ERC-20 can be registered as a collateral product. Fee-on-transfer tokens (e.g. tokens with a built-in transfer tax, or tokens whose fee switch is toggled post-listing) are a well-known token class. An unprivileged user needs only to call the public `depositCollateral` or `depositCollateralWithReferral` entrypoints with such a token — no special role or privileged access is required. [6](#0-5) 

---

### Recommendation

In `handleDepositTransfer`, measure the Clearinghouse's token balance before and after the two transfers and use the delta — not the caller-supplied `amount` — as the value forwarded to the slow-mode transaction:

```solidity
function handleDepositTransfer(IERC20Base token, address from, uint256 amount) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // return actualReceived so the caller enqueues the correct amount
}
```

The returned `actualReceived` value should replace `amount` in the `DepositCollateral` slow-mode payload. Additionally, consider explicitly disallowing fee-on-transfer tokens at product registration time.

---

### Proof of Concept

1. A fee-on-transfer token `FEE` (2 % fee per transfer) is registered as product `P`.
2. Alice calls `Endpoint.depositCollateral(subaccountName, P, 1000e18)`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(FEE, Alice, Endpoint, 1000e18)` → Endpoint receives `980e18` (2 % fee).
   - `safeTransferTo(FEE, Endpoint, Clearinghouse, 1000e18)` → Clearinghouse receives `980e18` (another 2 % fee on the second hop, but Endpoint only has `980e18`, so this call reverts **unless** the Endpoint already holds a residual balance from prior activity).
   - In the scenario where the Endpoint holds a residual balance (e.g., from slow-mode fee accumulation), the second transfer succeeds by drawing on that residual, and Clearinghouse receives `980e18` while Endpoint's residual is silently drained.
4. The slow-mode tx records `amount = 1000e18`.
5. When the sequencer executes the slow-mode tx, `Clearinghouse.depositCollateral` credits Alice's subaccount with `1000e18 × multiplier`.
6. Alice's subaccount shows `1000e18` but the Clearinghouse only holds `980e18` net of Alice's deposit, creating a `20e18`-token deficit per deposit cycle.
7. Repeated deposits drain the Clearinghouse's real reserves relative to its internal accounting, eventually preventing other users from withdrawing their full balances. [2](#0-1) [7](#0-6) [8](#0-7)
