### Title
SwapAllowlistExtension checks the router's address instead of the actual swapper, allowing any user to bypass the swap allowlist via the router - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument passed from the pool. The pool always passes `msg.sender` (the immediate caller) as `sender`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the router contract, not the end user. If the pool admin allowlists the router address to enable router-based swaps, every unprivileged user can bypass the allowlist by routing through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (used as the per-pool namespace key) and `sender` is the address that called `pool.swap()`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(...)` directly: [4](#0-3) 

At that point `msg.sender` to the pool is the router contract. The extension therefore evaluates `allowedSwapper[pool][router]` ŌĆö the router's allowlist entry ŌĆö rather than the actual end user's entry. If the pool admin has allowlisted the router address (the natural step to let allowlisted users trade conveniently through the supported periphery), every address in the system can bypass the allowlist by routing through the router.

The same mismatch applies to `exactOutputSingle`, `exactInput`, and `exactOutput`, and to the recursive `_exactOutputIterateCallback` path where intermediate hops call `pool.swap` with `msg.sender = router` as well: [5](#0-4) 

---

### Impact Explanation

A curated pool with `SwapAllowlistExtension` is designed to restrict trading to specific counterparties (e.g., KYC-verified addresses, institutional traders, or protocol-controlled addresses). Once the pool admin allowlists the router to let those users trade through the standard periphery, the allowlist is fully bypassed for every address. Unauthorized users can:

- Trade on a pool whose LP positions were sized and priced assuming only specific, trusted counterparties.
- Extract value from LPs through adverse selection or arbitrage that the allowlist was intended to prevent.
- Cause direct LP principal loss on pools whose risk model depends on counterparty restriction.

This meets the "Admin-boundary break" and "direct loss of LP assets" impact criteria.

---

### Likelihood Explanation

Likelihood is **high**:

1. `MetricOmmSimpleRouter` is the primary supported swap interface; pool admins are expected to allowlist it so that their allowlisted users can trade through the standard UX.
2. The bypass requires no special privileges, no flash loans, and no multi-step setup ŌĆö any address calls `exactInputSingle` on the router pointing at the curated pool.
3. The pool admin has no on-chain signal that allowlisting the router opens the gate to all users; the `isAllowedToSwap` view function returns `true` only for explicitly listed addresses, giving a false sense of security. [6](#0-5) 

---

### Recommendation

The extension must resolve the actual end user, not the immediate pool caller. Two sound approaches:

1. **Pass the original `msg.sender` through the router.** Add a `swapper` field to the router's swap parameters and forward it inside `extensionData`; the extension decodes it and verifies it against the allowlist. The pool's reentrancy guard and callback verification already authenticate the router, so the decoded address can be trusted within that call.

2. **Check `sender` against the allowlist and require the router to be the caller only when the sender is allowlisted.** The extension receives both `sender` (router) and the decoded real user from `extensionData`, then checks the real user. This keeps the extension stateless with respect to router identity.

The `DepositAllowlistExtension` does not share this flaw because it gates on the `owner` argument (the economic beneficiary of the position), which the pool passes through correctly regardless of who the `msg.sender` is: [7](#0-6) 

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin allowlists Alice (0xAlice) and the router (0xRouter):
      setAllowedToSwap(pool, 0xAlice, true)
      setAllowedToSwap(pool, 0xRouter, true)   ŌåÉ required for Alice to use the router
  - Eve (0xEve) is NOT allowlisted.

Attack:
  1. Eve calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(recipient, ...) ŌĆö msg.sender to pool = 0xRouter.
  3. Pool calls _beforeSwap(sender=0xRouter, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][0xRouter] ŌåÆ true.
  5. Swap executes. Eve has bypassed the allowlist.

Direct call (control):
  1. Eve calls pool.swap(...) directly.
  2. Pool calls _beforeSwap(sender=0xEve, ...).
  3. SwapAllowlistExtension checks allowedSwapper[pool][0xEve] ŌåÆ false ŌåÆ revert NotAllowedToSwap.
```

The bypass is gated entirely on whether the router is allowlisted, which is the expected operational state for any pool that wants its allowlisted users to access the standard periphery.
