### Title
State Updated After External Token Transfer in `withdrawCollateral` — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs an external token transfer via `handleWithdrawTransfer` **before** decrementing the subaccount balance in `spotEngine`. This violates the checks-effects-interactions pattern. No reentrancy guard exists anywhere in the codebase. During the external call window, the subaccount's on-chain balance in `SpotEngine` remains at its pre-withdrawal value, meaning any re-entrant read of that balance — including the health check on line 419 — operates on stale state.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **External call** — `handleWithdrawTransfer(token, sendTo, amount, idx)` at line 408, which internally calls `token.safeTransfer(withdrawPool, amount)` and then `BaseWithdrawPool.submitWithdrawal → token.safeTransfer(sendTo, amount)`.
2. **State update** — `spotEngine.updateBalance(productId, sender, amountRealized)` at line 412.
3. **Health check** — `getHealth(sender, healthType) >= 0` at line 419. [1](#0-0) 

The `handleWithdrawTransfer` in `Clearinghouse` routes tokens through the `WithdrawPool`: [2](#0-1) 

The final leg of that transfer is `token.safeTransfer(to, amount)` inside `BaseWithdrawPool.handleWithdrawTransfer`: [3](#0-2) 

If the token implements a transfer callback (e.g., ERC-777 `tokensReceived`, or any hook-bearing ERC-20 variant), the recipient contract at `sendTo` is invoked **before** `spotEngine.updateBalance` has been called. At that moment, `SpotEngine` still records the full pre-withdrawal balance for `sender`.

A grep across the entire repository confirms there is **no** `nonReentrant` modifier, `ReentrancyGuard` import, or equivalent lock anywhere: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

During the reentrancy window (after `token.safeTransfer` but before `spotEngine.updateBalance`), the subaccount's balance in `SpotEngine` is stale — it still reflects the full pre-withdrawal amount. Any protocol path that reads this balance during that window (health checks, liquidation eligibility, collateral queries) will operate on inflated collateral. Concretely:

- The post-withdrawal health check at line 419 uses the **not-yet-decremented** balance, so it cannot catch a health violation caused by the withdrawal itself until after the external call returns.
- A reentrant call that reads `spotEngine.getBalance` or `getHealth` for `sender` will see the full balance, potentially allowing a second action (e.g., another withdrawal queued via slow-mode, or a borrow) that would be rejected if the balance were already decremented. [6](#0-5) 

---

### Likelihood Explanation

The trigger requires a token registered as a supported collateral asset that implements a transfer callback. ERC-777 tokens and several DeFi-native tokens (e.g., USDC on some chains with hooks, cTokens, rebasing tokens) satisfy this. The `sendTo` address is attacker-controlled (it is the subaccount owner's address or an explicit `sendTo` field in `WithdrawCollateralV2`). The attacker deploys a contract at that address with a `tokensReceived` hook. The sequencer submits the withdrawal transaction in the normal course of operations — no privileged access is needed beyond having a funded subaccount. [7](#0-6) 

---

### Recommendation

Apply the checks-effects-interactions pattern: decrement `spotEngine` balance and perform the health check **before** calling `handleWithdrawTransfer`. The corrected order should be:

```solidity
// 1. Compute and apply state change first
int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
int128 amountRealized = -int128(amount) * int128(multiplier);
spotEngine.updateBalance(productId, sender, amountRealized);
spotEngine.assertUtilization(productId);

IProductEngine.HealthType healthType = sender == X_ACCOUNT
    ? IProductEngine.HealthType.PNL
    : IProductEngine.HealthType.INITIAL;
require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);

// 2. Then perform the external transfer
handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, add OpenZeppelin's `ReentrancyGuardUpgradeable` to `Clearinghouse` and apply `nonReentrant` to `withdrawCollateral`.

---

### Proof of Concept

1. Attacker deploys a contract `AttackerRecipient` at address `A` implementing ERC-777 `tokensReceived`.
2. Attacker opens a subaccount with `sender = bytes32(uint256(uint160(A)) << 96)` and deposits collateral.
3. Attacker signs a `WithdrawCollateralV2` transaction with `sendTo = A` and submits it to the sequencer.
4. Sequencer calls `Endpoint.submitTransactions` → `Clearinghouse.withdrawCollateral`.
5. `handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, amount)` then `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(A, amount)`.
6. ERC-777 hook fires on `A`. At this point `spotEngine` still shows the full pre-withdrawal balance for `sender`.
7. `AttackerRecipient.tokensReceived` calls any protocol function that reads `sender`'s balance (e.g., a slow-mode withdrawal or a borrow against the stale collateral). The health check passes because the balance has not yet been decremented.
8. Control returns; `spotEngine.updateBalance` is finally called, but the second action has already been committed against the inflated balance. [8](#0-7) [3](#0-2)

        uint128 amount,

        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
