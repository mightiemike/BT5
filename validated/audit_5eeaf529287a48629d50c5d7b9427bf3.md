### Title
Unbounded `while` Loop in `reorg_chain` Exhausts Gas on Long Forks, Permanently Locking the Contract on a Lower-Chainwork Chain — (File: `contract/src/lib.rs`)

---

### Summary

The `reorg_chain` function contains an unbounded `while` loop that iterates through every block in a fork chain from its tip back to the common ancestor with the main chain. A relayer who builds a sufficiently long fork chain can cause this loop to exhaust NEAR's gas limit when the reorg is triggered. Because the transaction is rolled back on failure, the contract is left permanently stuck with the lower-chainwork chain as its canonical tip, breaking the fundamental correctness guarantee of the light client.

---

### Finding Description

In `contract/src/lib.rs`, `reorg_chain` is invoked from `submit_block_header_inner` whenever a newly submitted fork block's accumulated `chain_work` exceeds the current main chain's `chain_work`. [1](#0-0) 

Inside `reorg_chain`, a `while` loop walks backward through the fork chain one block at a time, performing at least two `LookupMap` storage writes per iteration (one insert into `mainchain_height_to_header`, one insert into `mainchain_header_to_height`) and a conditional remove from `headers_pool`: [2](#0-1) 

There is **no bound** on the number of iterations. The loop terminates only when `fork_header_cursor.block_hash` is found in `mainchain_header_to_height`, i.e., when the common ancestor is reached. The number of iterations equals the fork depth.

Fork blocks are stored in `headers_pool` but are **never garbage-collected**. The GC mechanism (`run_mainchain_gc`) only removes mainchain blocks: [3](#0-2) 

This is explicitly documented: *"Only mainchain blocks are deleted; fork/sidechain blocks are not affected."* A relayer can therefore accumulate an arbitrarily long fork chain in `headers_pool` across many `submit_blocks` calls, each of which stores fork blocks without triggering a reorg (because chainwork < main chain). When the final block is submitted with sufficient chainwork to trigger a reorg, `reorg_chain` is called and the while loop must traverse all N accumulated fork blocks. If N is large enough, NEAR's gas limit is exceeded.

On gas exhaustion, NEAR rolls back the entire transaction. The result:
- All N fork blocks remain in `headers_pool` (they were committed in prior transactions).
- `mainchain_tip_blockhash` remains at the old, lower-chainwork block.
- Any subsequent attempt to submit the same fork tip will again call `reorg_chain` with the same N-block traversal and again exhaust gas.
- The contract is **permanently stuck** with the wrong canonical chain.

The same function also contains a second unbounded `for` loop that removes main-chain blocks when the main chain is longer than the fork: [4](#0-3) 

This loop is bounded by the height difference between the main chain tip and the fork tip, which is also uncapped.

---

### Impact Explanation

The contract's `mainchain_tip_blockhash` and `mainchain_header_to_height` index are permanently misaligned with the true highest-chainwork chain. Downstream consequences:

- `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` check `mainchain_header_to_height` to confirm a block belongs to the canonical chain. With the wrong tip recorded, transactions on the true Bitcoin main chain will fail verification, and transactions on the attacker's fork may pass verification. [5](#0-4) 

This breaks the core security invariant of the SPV light client: that the canonical chain it tracks reflects the highest-chainwork Bitcoin chain.

---

### Likelihood Explanation

`submit_blocks` is gated by `#[trusted_relayer]`. If the `omni_utils` trusted-relayer system is staking-based (permissionless entry), any unprivileged NEAR account can stake, become a relayer, and execute this attack. Even under a permissioned model, a single compromised or malicious relayer is sufficient. Additionally, a legitimate large natural reorg on the Bitcoin network (e.g., during a 51% attack) can trigger the same gas exhaustion without any adversarial intent, since fork blocks accumulate without bound and the reorg loop is uncapped. [6](#0-5) 

---

### Recommendation

1. **Cap fork depth before reorg**: Before calling `reorg_chain`, compute the fork depth (fork tip height minus common ancestor height). If it exceeds a configurable `max_reorg_depth`, reject the submission with a clear error. This forces multi-step reorgs to be handled off-chain or via a dedicated multi-transaction reorg protocol.

2. **Bound the while loop explicitly**: Add a counter inside the `while` loop and panic (or return an error) if the iteration count exceeds `max_reorg_depth`.

3. **GC fork blocks**: Implement a mechanism to prune fork blocks that are older than `gc_threshold` blocks, preventing unbounded accumulation in `headers_pool`.

---

### Proof of Concept

```
1. Deploy the contract with gc_threshold = 52704 (recommended default).

2. As a trusted relayer, submit N = 5000 fork blocks branching from the
   genesis block. Each block extends the previous fork block and has valid
   PoW. Because each fork block's chain_work < main chain's chain_work,
   no reorg is triggered; each block is stored in headers_pool via
   store_fork_header().

3. Submit the (N+1)th fork block whose cumulative chain_work exceeds the
   main chain's chain_work.

4. submit_block_header_inner detects chain_work > total_main_chain_chainwork
   and calls reorg_chain(fork_tip, last_main_chain_block_height).

5. The while loop at lib.rs:616 iterates 5001 times, performing 2–3
   LookupMap storage operations per iteration (~15,000 storage I/Os total).
   NEAR's 300 TGas limit is exhausted.

6. The transaction is rolled back. mainchain_tip_blockhash still points to
   the old lower-chainwork block.

7. Resubmitting the same fork tip block repeats step 4–6 indefinitely.
   The contract is permanently stuck on the wrong canonical chain.

8. verify_transaction_inclusion() now returns false for transactions on the
   true Bitcoin main chain and may return true for transactions on the
   attacker's fork.
``` [7](#0-6)

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
