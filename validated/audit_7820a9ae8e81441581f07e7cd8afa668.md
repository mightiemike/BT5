Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap`. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. Any non-allowlisted user can bypass a curated pool's swap allowlist by routing through the public router, provided the router itself is allowlisted — the operationally expected configuration for router-mediated swaps.

## Finding Description
In `MetricOmmPool.swap`, `_beforeSwap` is called with `msg.sender` as the `sender` argument:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← pool's msg.sender, which is the router when called via router
  recipient,
  ...
  extensionData
);
```

`SwapAllowlistExtension.beforeSwap` checks this `sender` against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

In `MetricOmmSimpleRouter.exactInputSingle`, the original caller's address is stored only in transient storage for the payment callback via `_setNextCallbackContext`, and is never forwarded to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData
  );
```

The pool therefore receives `msg.sender = router`. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The same wrong-actor binding applies to `exactInput` (L99-112), `exactOutputSingle` (L130-147), and `exactOutput` (L154-188) — all router entry points call `pool.swap` with `msg.sender = router`.

The `onlyPool` modifier in `BaseMetricExtension` only verifies the caller is a registered pool; it does not validate that `sender` is the actual end-user.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for approved users.
2. Non-allowlisted user Bob calls `router.exactInputSingle(...)`.
3. Router calls `pool.swap(bob_address_as_recipient, ...)` — pool's `msg.sender` is the router.
4. Extension checks `allowedSwapper[pool][router] = true` → passes.
5. Bob's swap executes on a pool intended to be restricted.
6. Direct call `pool.swap(...)` from Bob reverts because `allowedSwapper[pool][bob] = false`.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` (e.g., for KYC'd users or institutional counterparties) and allowlists the router to support router-mediated swaps for approved users inadvertently grants unrestricted swap access to any user. The allowlist provides zero protection against router-mediated swaps. Unauthorized users execute swaps on a pool designed to be restricted, draining LP value at prices the pool admin intended only for approved counterparties. This is a broken core pool access-control mechanism causing direct loss of LP value.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless contract. Any user can call `exactInputSingle` or `exactInput` with no preconditions beyond token approval. The bypass requires no privileged access, no special setup, and no malicious initial pool configuration. The only prerequisite — the router being allowlisted — is the natural operational step any pool admin must take to support router-mediated swaps for approved users.

## Recommendation
The extension must check the actual end-user identity, not the immediate pool caller. The preferred fix is to have the router encode `msg.sender` into `extensionData` before calling the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check it when `sender` is a known router. Alternatively, add a protocol-level convention where the router always encodes the originating caller in `extensionData`, and the extension decodes it. A partial mitigation is to document that `SwapAllowlistExtension` only enforces correctly on direct `pool.swap` calls and add a router-aware variant that reads the original caller from a signed or transient-storage-backed identity field.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, router allowlisted, Bob not allowlisted
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// Bob is NOT in allowedSwapper

// Bob bypasses the allowlist via the router:
vm.prank(bob);
token0.approve(address(router), type(uint256).max);

vm.prank(bob);
router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
    pool: address(pool),
    tokenIn: address(token0),
    tokenOut: address(token1),
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    recipient: bob,
    deadline: block.timestamp + 1,
    priceLimitX64: 0,
    extensionData: ""
}));
// Swap succeeds — extension checked allowedSwapper[pool][router] = true
// Bob (not allowlisted) successfully traded on a curated pool

// Confirm direct call is blocked:
vm.prank(bob);
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(bob, true, 1000, 0, "", "");
// Direct call reverts — allowedSwapper[pool][bob] = false
```
