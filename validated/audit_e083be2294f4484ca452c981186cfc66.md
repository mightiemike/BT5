### Title
Contract Subaccounts Blocked from Slow Mode Withdrawals via L1 Forced Inclusion Due to Missing Address Alias Check — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.validateSender` only compares the canonical address embedded in a transaction's `sender` field against the stored `msg.sender`. When a contract submits a slow mode transaction via L1 forced inclusion (bypassing the Ink Chain sequencer), the L2 records the aliased address as `msg.sender`. The stored aliased address never matches the canonical subaccount address, causing `validateSender` to revert and permanently blocking the operation for the duration of sequencer downtime.

---

### Finding Description

Nado is deployed on Ink Chain, an EVM-compatible L2. Like Optimism and Arbitrum, OP Stack chains apply **address aliasing** when a contract on L1 sends a transaction to L2 via forced inclusion: the L2 sees `msg.sender = contractAddress + 0x1111000000000000000000000000000000001111`.

The slow mode queue is Nado's censorship-resistance mechanism. When a user calls `Endpoint.submitSlowModeTransaction`, it delegates to `EndpointTx.submitSlowModeTransactionImpl`, which captures `msg.sender` as the `sender` stored in the `SlowModeTx` struct:

```solidity
// EndpointTx.sol line 341, 376-379
address sender = msg.sender;
// ...
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,   // <-- aliased address stored here
    tx: transaction
});
``` [1](#0-0) [2](#0-1) 

When the slow mode transaction is later executed, `processSlowModeTransactionImpl` calls `validateSender` for `WithdrawCollateral`, `LinkSigner`, and `ClaimBuilderFee`:

```solidity
// EndpointTx.sol line 217-222
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:], (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);  // <-- fails for aliased sender
``` [3](#0-2) 

`validateSender` only accepts the canonical address or `address(this)`:

```solidity
// EndpointTx.sol line 17-23
function validateSender(bytes32 txSender, address sender) internal view {
    require(
        address(uint160(bytes20(txSender))) == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [4](#0-3) 

The mismatch is structural:

| Value | Address |
|---|---|
| `address(uint160(bytes20(txn.sender)))` | `0xABCD...` (canonical) |
| `sender` (stored at submission) | `0xABCD... + 0x1111...1111` (aliased) |

These are never equal, so `validateSender` always reverts with `ERR_SLOW_MODE_WRONG_SENDER`.

The same check is applied to `LinkSigner` and `ClaimBuilderFee`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A contract that holds a Nado subaccount (e.g., a DeFi protocol using Nado for on-chain liquidity) cannot withdraw its collateral via the slow mode queue when the Ink Chain sequencer is down and it resorts to L1 forced inclusion. The collateral remains locked in the protocol for the entire duration of sequencer downtime. The `WithdrawCollateral` path in `processSlowModeTransactionImpl` is the only non-sequencer withdrawal route available to such a contract, and it is completely blocked. [7](#0-6) 

---

### Likelihood Explanation

The trigger requires: (1) the caller is a contract, not an EOA — realistic for any DeFi protocol integrating Nado as a liquidity venue; (2) the Ink Chain sequencer is down — a known operational risk for all L2s; (3) the contract attempts forced inclusion from L1 — the standard recovery path documented by OP Stack. All three conditions are independently plausible and can co-occur during a sequencer outage affecting a time-sensitive protocol.

---

### Recommendation

Extend `validateSender` to also accept the aliased form of the canonical address:

```solidity
function validateSender(bytes32 txSender, address sender) internal view {
    address canonical = address(uint160(bytes20(txSender)));
    // OP Stack alias: canonical + 0x1111000000000000000000000000000000001111
    address aliased = address(
        uint160(uint256(uint160(canonical)) +
            uint256(0x1111000000000000000000000000000000001111))
    );
    require(
        canonical == sender ||
            aliased == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [4](#0-3) 

---

### Proof of Concept

1. Protocol X deploys a contract at `0xABCD...` on L1 and the same address on Ink Chain (L2). It has a Nado subaccount `bytes32(abi.encodePacked(0xABCD..., subaccountName))` with deposited collateral.
2. The Ink Chain sequencer goes down. Protocol X needs to withdraw its collateral urgently.
3. Protocol X's L1 contract calls `Endpoint.submitSlowModeTransaction` via forced inclusion with a `WithdrawCollateral` payload where `txn.sender = bytes32(abi.encodePacked(0xABCD..., subaccountName))`.
4. On L2, `msg.sender = 0xABCD... + 0x1111...1111` (aliased). This aliased address is stored as `SlowModeTx.sender`.
5. After `SLOW_MODE_TX_DELAY`, Protocol X calls `executeSlowModeTransaction`. `processSlowModeTransactionImpl` calls `validateSender(txn.sender, sender)`.
6. `address(uint160(bytes20(txn.sender))) = 0xABCD...` ≠ `sender = 0xABCD... + 0x1111...1111` → reverts with `ERR_SLOW_MODE_WRONG_SENDER`.
7. The collateral withdrawal is blocked for the entire duration of sequencer downtime. The slow mode tx is consumed (deleted from the queue at line 194) but silently fails via the `try/catch` in `_executeSlowModeTransaction`, permanently losing the queued withdrawal attempt. [8](#0-7) [2](#0-1)
