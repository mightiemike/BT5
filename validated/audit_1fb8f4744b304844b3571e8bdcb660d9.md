### Title
Residual ERC20 Allowance After Silent Deposit Skip Permanently Breaks `creditDeposit()` for Non-Standard Tokens — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

In `DirectDepositV1.creditDeposit()`, `token.approve(address(endpoint), balance)` is called unconditionally before `endpoint.depositCollateralWithReferral()`. The endpoint function silently returns early — without pulling tokens — when the deposit amount is below the protocol minimum. This leaves a non-zero residual allowance on the endpoint. For tokens that revert on non-zero-to-non-zero allowance changes (e.g., USDT), any subsequent call to `creditDeposit()` permanently reverts, bricking the DDA deposit path for that token.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all registered spot products and, for each non-zero balance, executes:

```solidity
token.approve(address(endpoint), balance);          // line 92
endpoint.depositCollateralWithReferral(             // line 93–98
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
``` [1](#0-0) 

`depositCollateralWithReferral` in `Endpoint.sol` contains an explicit silent-return guard:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    return;   // no transfer, no allowance consumed
}
handleDepositTransfer(...);
``` [2](#0-1) 

When `isValidDepositAmount` returns `false` (balance below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`), the function returns without calling `handleDepositTransfer`. The allowance set on line 92 is **not consumed**. The endpoint retains a non-zero allowance for that token.

On the next invocation of `creditDeposit()` — after more tokens arrive in the DDA — the code again calls `token.approve(address(endpoint), newBalance)`. For tokens implementing the non-zero-to-non-zero protection (USDT reverts; some other tokens silently discard), this call either reverts permanently or silently sets the wrong allowance, causing the subsequent `transferFrom` to fail.

`creditDeposit()` carries no access control modifier — it is callable by any unprivileged address. [1](#0-0) 

The `IIERC20Base` interface used inside `DirectDepositV1` exposes only a bare `approve`; there is no `increaseAllowance`, `decreaseAllowance`, or reset-to-zero guard anywhere in the function. [3](#0-2) 

By contrast, `ContractOwner.wrapVaultAsset()` correctly resets to zero before re-approving, demonstrating the protocol is aware of this pattern elsewhere:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

This fix was not applied to `DirectDepositV1`.

---

### Impact Explanation

For any supported spot token that enforces the non-zero-to-non-zero allowance restriction:

- After one silent-skip event, the DDA's `creditDeposit()` permanently reverts on the `approve` call.
- All subsequent deposits through the DDA for that token are blocked.
- Funds already sitting in the DDA cannot be credited to the subaccount via the normal path.
- The only recovery is `ContractOwner.withdrawFromDirectDepositV1()` (owner-only), meaning user funds are effectively locked until admin intervention.

The corrupted state is the residual non-zero ERC20 allowance on the endpoint, which prevents the DDA from ever re-approving a different amount.

---

### Likelihood Explanation

- `creditDeposit()` is `external` with no access control; any address can trigger it at any time.
- The silent-return path in `depositCollateralWithReferral` is explicitly documented and intentional.
- A deposit below the minimum threshold is a normal operational condition (e.g., dust amounts, price fluctuations affecting USD-denominated minimums, or a first-deposit subaccount receiving a small initial transfer).
- If the protocol lists any token with non-zero-to-non-zero protection (USDT is the canonical example), the condition is reachable without any privileged access.

---

### Recommendation

In `DirectDepositV1.creditDeposit()`, reset the allowance to zero before setting the new value, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, check the current allowance and only call `approve` if it is zero, or use `increaseAllowance` if the token supports it.

---

### Proof of Concept

1. A `DirectDepositV1` DDA exists for subaccount `S` with USDT as the collateral token.
2. An attacker (or anyone) sends 1 wei of USDT to the DDA — an amount below `MIN_DEPOSIT_AMOUNT`.
3. Anyone calls `creditDeposit()`:
   - `token.approve(endpoint, 1)` — allowance set to 1.
   - `endpoint.depositCollateralWithReferral(S, productId, 1, "-1")` — `isValidDepositAmount` returns `false`; function returns silently. No `transferFrom` is called. Allowance of 1 remains.
4. The legitimate user later sends 1000 USDT to the DDA (above the minimum).
5. Anyone calls `creditDeposit()`:
   - `token.approve(endpoint, 1000)` — **REVERTS** because USDT disallows changing a non-zero allowance to another non-zero value.
6. The 1000 USDT is permanently stuck in the DDA, inaccessible via `creditDeposit()`, until the owner manually calls `withdrawFromDirectDepositV1`. [1](#0-0) [5](#0-4)

    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
