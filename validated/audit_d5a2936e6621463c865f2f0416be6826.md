### Title
Checkpoint Advance Before Pool Check in `processEventsAndClaimBonus` Causes Permanent Bonus Reward Loss on Insufficient Pool — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

When `BeforeRedelegationSlashed` fires and the rewards pool is depleted, `slashRedelegationPosition` calls `processEventsAndClaimBonus`, which mutates all position checkpoints and decrements all validator event reference counts **before** checking pool sufficiency. When `ErrInsufficientBonusPool` is returned, `slashRedelegationPosition` deliberately swallows it and persists the already-advanced position state. The result is that all accrued bonus rewards for the period are permanently destroyed: events are consumed, checkpoints are advanced, and no coins are paid.

---

### Finding Description

The call chain is:

```
BeforeRedelegationSlashed (hooks.go:128)
  → slashRedelegationPosition (slash.go:19)
    → processEventsAndClaimBonus (claim_rewards.go:142)
```

Inside `processEventsAndClaimBonus`, the event loop at lines 172–198 performs two **irreversible state mutations per event** before the pool balance check:

1. `pos.UpdateLastEventSeq(entry.Seq)` — advances the in-memory position checkpoint (line 193)
2. `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` — writes to on-chain state, potentially deleting the event if `ReferenceCount <= 1` (line 196)

After the loop, two more in-memory mutations occur:

3. `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` — advances `LastBonusAccrual` (line 215)
4. `pos.UpdateLastKnownBonded(bonded)` — updates bonded state (line 217)

Only **then** is the pool checked:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // line 230–232
}
``` [1](#0-0) [2](#0-1) 

When the pool is empty, `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`. At this point:
- All `decrementEventRefCount` writes are committed to the KV store (events deleted or ref-count decremented)
- The `pos` pointer carries advanced `LastEventSeq`, `LastBonusAccrual`, and `LastKnownBonded`

Back in `slashRedelegationPosition`, the error is explicitly swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [3](#0-2) 

Execution then falls through to `setPositionWithState(ctx, pos, ...)` with the already-mutated `pos`: [4](#0-3) 

This persists the advanced checkpoints. A subsequent `ClaimTierRewards` call will find `pos.LastEventSeq` already past all the events (which have been consumed), and `pos.LastBonusAccrual` already at `blockTime`, so it computes zero bonus and pays nothing.

The `decrementEventRefCount` function confirms that events with `ReferenceCount <= 1` are permanently deleted from the store: [5](#0-4) 

---

### Impact Explanation

The position owner permanently loses all bonus rewards accrued since their last claim. The invariant that "event ref counts must not be decremented and position checkpoints must not be advanced unless the corresponding bonus coins are paid" is violated. There is no recovery path: the events are gone from the store and the checkpoints are past them.

---

### Likelihood Explanation

The rewards pool can be depleted through normal operation (many users claiming, governance underfunding). The slash trigger (`BeforeRedelegationSlashed`) fires automatically whenever the staking module slashes a redelegation — no attacker action is required beyond having a position with a live redelegation mapping. Any position owner whose validator is slashed while the pool is empty loses their rewards. A targeted attacker could also time a redelegation to a validator they know will be slashed, then drain the pool (e.g., by front-running with their own claims) to guarantee the victim's loss.

---

### Recommendation

Move `sufficientBonusPoolBalance` to **before** the event loop, or restructure `processEventsAndClaimBonus` so that `decrementEventRefCount` and checkpoint advances only execute after a successful `SendCoinsFromModuleToAccount`. Alternatively, in the `ErrInsufficientBonusPool` branch of `slashRedelegationPosition`, restore the original `pos` checkpoints (snapshot them before calling `processEventsAndClaimBonus`) so that `setPositionWithState` persists the pre-call state, leaving the rewards claimable once the pool is refunded.

---

### Proof of Concept

1. Create a tier position and call `TierRedelegate`; record the `unbondingID` stored in `RedelegationMappings`.
2. Let the validator emit at least one event (e.g., a slash or bond/unbond transition) so `pos.LastEventSeq` lags behind.
3. Drain the `rewards_pool` module account to zero (e.g., via other positions claiming all available bonus).
4. Fire `BeforeRedelegationSlashed` with the recorded `unbondingID`.
5. Assert:
   - (a) The owner's balance did not increase.
   - (b) `pos.LastEventSeq` advanced past its pre-slash value (persisted by `setPositionWithState`).
   - (c) `ValidatorEvents` for the consumed seqs are deleted or ref-count decremented.
   - (d) A subsequent `ClaimTierRewards` for the same position pays zero bonus.

```go
func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}
