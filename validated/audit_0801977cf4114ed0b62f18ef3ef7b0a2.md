### Title
Silent Permanent Forfeit of Accrued Bonus Rewards During Redelegation Slash with Insufficient Pool - (`File: x/tieredrewards/keeper/slash.go`)

### Summary

In `x/tieredrewards/keeper/slash.go`, the `slashRedelegationPosition` function silently discards accrued bonus rewards when the bonus pool is insufficient during a `BeforeRedelegationSlashed` hook. Critically, `processEventsAndClaimBonus` advances the position's bonus checkpoints (`LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) and decrements event reference counts in the store **before** the pool-sufficiency check. When `ErrInsufficientBonusPool` is returned, the caller catches it, logs it, and then persists the position with the already-advanced checkpoints via `setPositionWithState`. The accrued bonus is permanently destroyed — the user can never reclaim it.

### Finding Description

`processEventsAndClaimBonus` walks pending validator events, updates `pos.LastEventSeq` and decrements event reference counts in the store (lines 193, 196), then calls `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` (lines 215, 217) — all **before** the pool balance check at line 230:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // checkpoints already advanced, ref counts already decremented
}
```

In `slashRedelegationPosition`, the `ErrInsufficientBonusPool` error is explicitly caught and swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // continues — pos already has advanced checkpoints
    } else {
        return err
    }
}
```

For a partial slash, execution falls through to `k.setPositionWithState(ctx, pos, nil)`, which persists the position with the advanced checkpoints. The consumed events (ref counts decremented, potentially garbage-collected) and the advanced `LastBonusAccrual` timestamp mean the accrual window is permanently closed. The user has no mechanism to retry or recover the forfeited bonus.

This is the direct analog to the YearnTokenAdapter's hardcoded `MAXIMUM_SLIPPAGE = 10000` (100%): a hardcoded behavior that allows 100% loss of accrued rewards with no user control, no minimum guarantee, and no recovery path.

### Impact Explanation

The exact corrupted value is the position owner's accrued bonus reward balance. After the hook fires:
- `pos.LastBonusAccrual` is advanced to `blockTime` (or `ExitUnlockAt`), closing the accrual window permanently.
- `pos.LastEventSeq` is advanced past all processed events, which have had their reference counts decremented and may be garbage-collected.
- The bonus coins that were computed but not sent are permanently lost — they are neither in the owner's account nor held in escrow.

The `RewardsPoolName` module account retains the coins, but the position can never claim them again because the checkpoints have moved past the accrual period.

### Likelihood Explanation

Two conditions must coincide:
1. A tier position undergoes a redelegation (via `MsgTierRedelegate`) and the destination validator is subsequently slashed during the redelegation's unbonding window, triggering `BeforeRedelegationSlashed`.
2. The bonus pool balance is insufficient to cover the accrued bonus at that moment.

Both conditions can occur naturally: the pool can run low if many users claim rewards concurrently, and validator slashing during redelegation periods is a documented Cosmos SDK scenario. An adversarial validator could also intentionally trigger downtime slashing after users redelegate to them, if the pool is known to be low.

### Recommendation

Move the pool-sufficiency check and the `SendCoinsFromModuleToAccount` call **before** advancing any position checkpoints or decrementing event reference counts. If the pool is insufficient, return the error without modifying `pos` state, so the accrued bonus remains claimable once the pool is replenished. Alternatively, store the owed bonus as a pending debt on the position to be settled when the pool recovers.

### Proof of Concept

1. User creates a tier position and calls `MsgTierRedelegate` — a `RedelegationMapping` entry is created.
2. The bonus pool is drained (e.g., many concurrent claims).
3. The destination validator is slashed during the redelegation's unbonding window; the staking module fires `BeforeRedelegationSlashed`.
4. `slashRedelegationPosition` is called → `processEventsAndClaimBonus` advances `LastEventSeq`, `LastBonusAccrual`, decrements event ref counts, then returns `ErrInsufficientBonusPool`.
5. The error is swallowed; `setPositionWithState` persists the position with advanced checkpoints.
6. The user later calls `MsgClaimTierRewards` or `MsgTierUndelegate` — `processEventsAndClaimBonus` finds no new events and `LastBonusAccrual` already at `blockTime`, returning zero bonus. The accrued rewards are permanently lost.
