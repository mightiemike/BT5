### Title
Gateway Stateful Validator Accepts Declare Transactions with V1 Compiled Class Hashes That Blockifier Rejects — (File: `crates/blockifier/src/blockifier_versioned_constants.rs`, `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`VersionedConstants::disable_casm_hash_migration()` only sets `enable_casm_hash_migration = false` but leaves `block_casm_hash_v1_declares = true`. The gateway's stateful validator calls this function before running the `__validate__` entry point, but the `block_casm_hash_v1_declares` enforcement lives exclusively in the blockifier's full-execution path (`account_tx.execute()`), not in the validate-only path (`StatefulValidator::validate`). The result is a version-boundary split: the gateway admits declare transactions carrying a V1 (Poseidon) compiled class hash, while the batcher rejects them with `DeclareTransactionCasmHashMissMatch` during block building.

---

### Finding Description

**Version-constant boundary.** Starting with Starknet 0.14.1, both `enable_casm_hash_migration` and `block_casm_hash_v1_declares` are set to `true` in the versioned-constants JSON files. [1](#0-0) 

The `VersionedConstants` struct carries both flags: [2](#0-1) 

**Incomplete disablement in the gateway.** The gateway's stateful validator calls `disable_casm_hash_migration()` before running the validate entry point: [3](#0-2) 

That helper only touches one flag: [4](#0-3) 

`block_casm_hash_v1_declares` is never cleared. The gateway therefore runs with `block_casm_hash_v1_declares = true` in its `VersionedConstants`, but the flag is never consulted during the validate-only path.

**Enforcement is execution-only.** The `block_casm_hash_v1_declares` check lives in `crates/blockifier/src/transaction/transactions.rs` and is exercised only when `account_tx.execute()` is called. The test confirms this: a declare with `HashVersion::V1` panics with `DeclareTransactionCasmHashMissMatch` only on the full-execution call, not on the validate call: [5](#0-4) 

The gateway's stateful validator calls only `blockifier_validator.validate(account_tx)`, which runs the `__validate_declare__` entry point but never reaches the compiled-class-hash check: [6](#0-5) 

The same pattern is present in the Python-wrapper validator: [7](#0-6) 

**Batcher enforces the flag.** During block building the batcher calls the full execution path with the latest `VersionedConstants` (both flags `true`). Any declare transaction carrying a V1 compiled class hash is rejected there, after having already been admitted by the gateway and placed in the mempool.

---

### Impact Explanation

This is a gateway/mempool admission divergence. An unprivileged user can submit a syntactically valid declare transaction whose `compiled_class_hash` field is the Poseidon (V1) hash of the Sierra class rather than the Blake (V2) hash. The gateway's stateless and stateful validators both pass it; the transaction enters the mempool. The batcher then discards it during execution. The gateway has accepted a transaction that will never be sequenced, matching the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

At scale this allows cost-free mempool pollution: an attacker can flood the mempool with structurally valid but permanently-unsequenceable declare transactions, degrading throughput and wasting batcher resources.
