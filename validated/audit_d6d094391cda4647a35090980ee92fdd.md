### Title
Missing `msg.value == 0` Guard on Non-WETH Swap and Liquidity Paths Allows Accidental ETH Loss to Any Caller — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

All swap and liquidity entry points in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are marked `payable` but never validate `msg.value == 0` when the input token is not WETH. Any ETH sent alongside a non-WETH operation is silently retained by the contract and can be claimed by any subsequent caller of `refundETH()`, causing a direct loss of the original sender's ETH.

---

### Finding Description

Every user-facing swap function in `MetricOmmSimpleRouter` is `payable`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Likewise, every liquidity entry point in `MetricOmmPoolLiquidityAdder` is `payable`: [5](#0-4) [6](#0-5) [7](#0-6) 

None of these functions check `msg.value == 0` before proceeding.

The internal `pay()` helper in `PeripheryPayments` only consumes `address(this).balance` when `token == WETH`. For every other token it falls through to the plain `safeTransferFrom` branch and ignores any ETH sitting in the contract: [8](#0-7) 

The `refundETH()` recovery function sends the **entire** contract ETH balance to `msg.sender` — not to the original depositor: [9](#0-8) 

The `receive()` guard only blocks direct ETH transfers; it does not prevent ETH from entering via `msg.value` in a `payable` call: [10](#0-9) 

---

### Impact Explanation

A user who calls any non-WETH swap or liquidity function with `msg.value > 0` (e.g., out of habit from ETH-native routers, or by mistake) has their ETH silently retained by the contract. The swap itself succeeds using `safeTransferFrom` for the ERC-20 token, so no revert signals the mistake. The stranded ETH is then claimable by any address that calls `refundETH()` in a subsequent transaction, resulting in a **direct, irreversible loss of user principal** with no on-chain recourse.

---

### Likelihood Explanation

The `payable` modifier on non-ETH paths is an open invitation for this mistake. Users familiar with Uniswap v3-style routers routinely send ETH expecting it to be auto-wrapped, or scripts/wallets may include a non-zero `value` field by default. The `multicall` wrapper is also `payable`, compounding the surface. The loss requires only a single mistaken transaction and is immediately exploitable by any observer of the mempool or block. [11](#0-10) [12](#0-11) 

---

### Recommendation

Add an explicit guard at the top of each swap and liquidity function that rejects non-zero `msg.value` when the input token is not WETH:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput:
if (msg.value != 0 && params.tokenIn != WETH) revert UnexpectedETH();

// In addLiquidityExactShares, addLiquidityWeighted:
// (pool token0/token1 are not WETH)
if (msg.value != 0) revert UnexpectedETH();
```

Alternatively, follow the 1inch pattern and add `require(msg.value == 0, "unexpected ETH")` before any non-native-token settlement path.

---

### Proof of Concept

1. User calls `exactInputSingle` with `params.tokenIn = address(USDC)` and `msg.value = 1 ether`.
2. The function is `payable`; the 1 ETH is accepted by the contract.
3. `_setNextCallbackContext` and `pool.swap` execute normally; the callback calls `pay(USDC, user, pool, amount)` which hits the `else` branch (`safeTransferFrom`) — the 1 ETH is untouched.
4. The swap completes successfully; the user receives USDC output and sees no error.
5. An attacker (or MEV bot) calls `refundETH()` in the next block and receives the 1 ETH.
6. The victim has no on-chain mechanism to recover their ETH. [13](#0-12) [14](#0-13) [9](#0-8)

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
