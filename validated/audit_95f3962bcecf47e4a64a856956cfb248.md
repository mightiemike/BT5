### Title
Rebasing Token Rewards Permanently Locked in Clearinghouse — (`core/contracts/Clearinghouse.sol`)

---

### Summary

The Nado `Clearinghouse` holds ERC20 collateral tokens on behalf of users. Deposit and withdrawal flows use user-specified amounts that are recorded in the `SpotEngine`'s internal normalized-balance accounting. If a rebasing ERC20 token (one whose `balanceOf()` increases over time for all holders, e.g., Aave aTokens) is configured as a spot collateral product, the rebase-accrued surplus accumulates in the `Clearinghouse` contract but is never credited to any user's internal balance and can never be withdrawn by anyone — it is permanently locked.

---

### Finding Description

**Deposit path** (`Clearinghouse.depositCollateral`): When a user deposits collateral, the Clearinghouse receives the tokens and credits the user's internal balance with exactly `amountRealized` — the amount transferred at deposit time. [1](#0-0) 

The `SpotEngine` records this as a normalized amount: `amountNormalized = amountRealized / cumulativeDepositsMultiplierX18`. [2](#0-1) 

**Withdrawal path** (`Clearinghouse.withdrawCollateral`): When a user withdraws, the `amount` parameter comes from the sequencer-submitted withdrawal transaction (bounded by the user's internal balance). The Clearinghouse transfers exactly `amount` tokens to the `WithdrawPool` and decrements the user's internal balance by the same value. [3](#0-2) 

**The gap**: The `SpotEngine`'s `cumulativeDepositsMultiplierX18` grows over time to reflect *protocol-internal* interest (from borrowers paying interest to depositors). This is entirely separate from the ERC20 token's own rebasing mechanism. If the underlying ERC20 token rebases — increasing `token.balanceOf(clearinghouse)` — that surplus is never credited to any internal balance and no code path exists to recover it.

**`assertUtilization` does not close the gap**: The only on-chain invariant check compares internal accounting totals against each other, not against the actual ERC20 balance held by the contract. [4](#0-3) 

There is no function in `Clearinghouse`, `SpotEngine`, or `WithdrawPool` that reads `token.balanceOf(address(this))` and distributes or recovers any surplus above the internally-tracked total deposits.

---

### Impact Explanation

For every rebasing token configured as a spot product, the delta between `token.balanceOf(clearinghouse)` and the internally-tracked `totalDepositsNormalized × cumulativeDepositsMultiplierX18 − totalBorrowsNormalized × cumulativeBorrowsMultiplierX18` grows monotonically and is irrecoverable. No user, no admin, and no protocol function can withdraw this surplus. The tokens are permanently locked in the `Clearinghouse`. [5](#0-4) 

---

### Likelihood Explanation

Likelihood is conditional on the protocol owner adding a rebasing token (e.g., an Aave aToken, stETH in rebase mode, or similar) via `SpotEngine.addOrUpdateProduct`. [6](#0-5) 

Once such a product exists, every deposit by any ordinary user triggers the accumulation. No privileged access or special attacker behavior is required beyond the standard `depositCollateralWithReferral` → `Clearinghouse.depositCollateral` flow. [7](#0-6) 

---

### Recommendation

1. For each spot product, track the total raw ERC20 amount deposited (not just the normalized internal balance).
2. Compute the rebase surplus as `token.balanceOf(clearinghouse) − trackedDeposits` and distribute it pro-rata to depositors by adjusting `cumulativeDepositsMultiplierX18`, or provide an admin sweep function that credits the surplus to a designated account.
3. Alternatively, explicitly document and enforce that rebasing tokens are not supported as spot collateral products, and add a runtime check (e.g., comparing pre/post `balanceOf` on deposit) to reject fee-on-transfer or rebasing tokens.

---

### Proof of Concept

1. Owner calls `SpotEngine.addOrUpdateProduct` with a rebasing token (e.g., aUSDC) as the `config.token`.
2. Alice calls `Endpoint.depositCollateralWithReferral(subaccount, productId, 1000e6, "")`. The Clearinghouse receives 1000 aUSDC; Alice's internal balance is credited 1000e18 (normalized).
3. One month passes. aUSDC rebases; the Clearinghouse now holds 1010 aUSDC. Alice's internal balance (via `cumulativeDepositsMultiplierX18`) may have grown slightly due to protocol interest, but the 10 aUSDC rebase surplus is not reflected anywhere in the SpotEngine state.
4. Alice submits a withdrawal for her full internal balance. `withdrawCollateral` transfers her internal-balance-equivalent amount (e.g., 1001 aUSDC) to the `WithdrawPool`. The remaining ~9 aUSDC rebase surplus stays in the Clearinghouse.
5. No function exists to recover these tokens. They are permanently locked. [8](#0-7)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }

    function assertUtilization(uint32 productId) external view {
        (State memory _state, ) = getStateAndBalance(productId, X_ACCOUNT);
        int128 totalDeposits = _state.totalDepositsNormalized.mul(
            _state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = _state.totalBorrowsNormalized.mul(
            _state.cumulativeBorrowsMultiplierX18
        );
        require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
    }
