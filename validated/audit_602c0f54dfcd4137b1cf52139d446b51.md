### Title
`PauseManager` Role Never Granted in `init`, Making `pa_pause`/`pa_unpause` Permanently Inaccessible — (`File: contract/src/lib.rs`)

### Summary

`BtcLightClient` derives `Pausable` from `near_plugins` and declares a `PauseManager` role intended to authorize pause/unpause operations. However, the `init` function never grants `PauseManager` (or any other role) to any external account. It sets the sole super-admin to `env::current_account_id()` — the contract itself — with no subsequent mechanism to delegate that authority. In a standard production deployment where the contract's full-access key is deleted after initialization, the `pa_pause()` and `pa_unpause()` entry points become permanently uncallable, rendering the entire pause subsystem dead code.

### Finding Description

`near_plugins`' `Pausable` derive macro generates two public entry points, `pa_pause()` and `pa_unpause()`, gated by a role check: the caller (`predecessor_account_id()`) must hold `Role::PauseManager`. The only way to grant that role is through `acl_grant_role`, which itself requires the caller to be a super-admin.

In `init`, the super-admin is set to `env::current_account_id()`:

```rust
near_sdk::require!(
    contract.acl_init_super_admin(env::current_account_id()),
    "Failed to initialize super admin",
);
``` [1](#0-0) 

`env::current_account_id()` is the contract's own account, not the deployer. After `init` returns, no external account holds super-admin status, and `init` grants zero roles to any external account — `PauseManager` included. [2](#0-1) 

The `PauseManager` role is declared and wired into the `Pausable` attribute:

```rust
#[pausable(manager_roles(Role::PauseManager))]
``` [3](#0-2) 

but it is never assigned to anyone. The three production entry points protected by `#[pause]` are:

- `submit_blocks` [4](#0-3) 
- `verify_transaction_inclusion` [5](#0-4) 
- `verify_transaction_inclusion_v2` [6](#0-5) 

The only path to grant `PauseManager` post-deployment is to call `acl_grant_role` signed by the contract account itself. In NEAR, that requires the contract account to retain a full-access key — a key that is routinely deleted after deployment as a standard security hardening step. The test harness works only because `near_workspaces::dev_deploy` retains the contract's full-access key in the sandbox:

```rust
// The contract itself is the super admin (set during `init`), so it can grant any role.
let outcome = contract
    .call("acl_grant_role")
    ...
``` [7](#0-6) 

This pattern works in tests but silently fails in production once the key is deleted.

### Impact Explanation

With no account holding `PauseManager`, `pa_pause()` will always revert. If a critical bug is found in `submit_blocks` (e.g., a header-validation bypass that corrupts the canonical chain) or in `verify_transaction_inclusion_v2` (e.g., a Merkle-proof forgery), operators have no on-chain lever to halt the contract. Downstream dApps consuming `verify_transaction_inclusion_v2` results would continue to receive potentially forged proofs with no ability to freeze the oracle. The pause subsystem — explicitly designed for this scenario — is permanently inoperative.

### Likelihood Explanation

Deleting a contract's full-access key after deployment is standard NEAR operational practice and is recommended in the NEAR documentation. The `init` function provides no `InitArgs` field for a pause-manager account and no in-code reminder to grant the role before key deletion. Any production deployment that follows standard key hygiene will silently ship with an unusable pause mechanism.

### Recommendation

Grant `PauseManager` (and other critical roles) to a designated admin account inside `init`, before returning. Add an `admin` (or `pause_manager`) field to `InitArgs`:

```rust
// In InitArgs
pub pause_manager: AccountId,

// In init()
contract.acl_grant_role("PauseManager".to_string(), args.pause_manager);
```

Alternatively, grant the role to `env::predecessor_account_id()` (the deployer) so that at least one external account can call `pa_pause()` immediately after deployment.

### Proof of Concept

1. Deploy the contract and call `init` (as the contract account, per `#[private]`).
2. Delete the contract account's full-access key (standard hardening).
3. Discover a critical bug in `submit_blocks`.
4. Attempt to call `pa_pause()` from any account — the call reverts because no account holds `PauseManager`.
5. `submit_blocks`, `verify_transaction_inclusion`, and `verify_transaction_inclusion_v2` remain live and exploitable indefinitely.
