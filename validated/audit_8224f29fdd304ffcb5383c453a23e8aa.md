### Title
Linked Signer Can Overwrite Its Own Authority via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The sequencer-path handler for `LinkSigner` transactions passes `allowLinkedSigner = true` to `validateSignedTx`. This means the **currently registered linked signer** — a limited-trust hot-wallet or API key — can sign a `LinkSigner` transaction to replace itself with any address, including an attacker-controlled one. Because `WithdrawCollateral`, `TransferQuote`, and other high-value transactions also accept linked-signer signatures, a single compromised linked signer can permanently escalate to full subaccount control and drain all collateral.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // ← allowLinkedSigner
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which in turn calls `verifier.validateSignature(sender, getLinkedSigner(sender), digest, signature)`. The verifier accepts a signature from **either** the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

Therefore, the linked signer can craft and sign a `SignedLinkSigner` transaction that sets `signedTx.tx.signer` to any address — including an attacker-controlled one — and the protocol will accept it and overwrite `linkedSigners[subaccount]`.

This is inconsistent with the **slow-mode** `LinkSigner` path, which correctly restricts the operation to the subaccount owner via `validateSender`:

```solidity
validateSender(txn.sender, sender);   // msg.sender must be the subaccount owner
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The slow-mode path correctly enforces owner-only authority for this operation; the sequencer path does not.

---

### Impact Explanation

Once the linked signer is replaced with an attacker-controlled address, the attacker inherits full signing authority over the subaccount. All high-value sequencer-path transactions accept `allowLinkedSigner = true`:

- `WithdrawCollateral` (V1) — attacker can drain all collateral to the default withdrawal address.
- `WithdrawCollateralV2` with `sendTo == address(0)` — same drain path.
- `TransferQuote` — attacker can transfer quote balance to any subaccount they control.
- `MintNlp` / `BurnNlp` — attacker can manipulate NLP positions.
- `LiquidateSubaccount` — attacker can liquidate the victim's subaccount. [4](#0-3) [5](#0-4) 

The corrupted state delta is `linkedSigners[subaccount]` being set to an attacker address, followed by complete collateral theft via a subsequent `WithdrawCollateral` signed by the attacker.

---

### Likelihood Explanation

Linked signers are explicitly designed to be hot wallets or API keys — entities with a lower security posture than the subaccount owner's cold wallet. Any user who has set a linked signer (a common, encouraged workflow for API trading) is exposed. A single leaked API key, phished session, or malicious API integration is sufficient to trigger the full escalation chain. No admin access, governance capture, or sequencer compromise is required — the attacker only needs to submit a valid EIP-712 signed transaction through the normal sequencer flow.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
``` [6](#0-5) 

This aligns the sequencer path with the slow-mode path, which already enforces owner-only authority for `LinkSigner`.

---

### Proof of Concept

1. Alice sets `linkedSigners[aliceSubaccount] = apiKeyAddress` via a normal `LinkSigner` slow-mode transaction.
2. Attacker compromises `apiKeyAddress` (leaked API key, phished session, etc.).
3. Attacker constructs a `SignedLinkSigner` transaction:
   - `tx.sender = aliceSubaccount`
   - `tx.signer = bytes32(uint256(uint160(attackerAddress)))`
   - `tx.nonce = currentNonce`
   - Signs with `apiKeyAddress` (the current linked signer).
4. Attacker submits the transaction to the sequencer. The sequencer processes it; `validateSignedTx(..., true)` accepts the `apiKeyAddress` signature.
5. `linkedSigners[aliceSubaccount]` is now `attackerAddress`.
6. Attacker constructs a `SignedWithdrawCollateral` transaction signed by `attackerAddress` (now the linked signer), withdrawing Alice's full collateral balance.
7. Sequencer processes the withdrawal; Alice's collateral is transferred out. [1](#0-0) [7](#0-6)
