### Title
Linked Signer Can Escalate to Arbitrary Address via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

---

### Summary

The `processTransactionImpl` handler for `LinkSigner` transactions calls `validateSignedTx` with `allowLinkedSigner=true`. Because `validateSignature` accepts either the subaccount owner **or** the current linked signer as a valid signer, a linked signer can sign and submit a new `LinkSigner` transaction that replaces themselves with any attacker-controlled address, without the subaccount owner's knowledge or consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch is:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx(..., true)` routes through `validateSignature`, which passes `getLinkedSigner(sender)` as the accepted alternate signer:

```solidity
// EndpointTx.sol lines 172–184
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if `recovered == owner` **OR** `recovered == linkedSigner`:

```solidity
// Verifier.sol lines 297–303
address recovered = ECDSA.recover(digest, signature);
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

There is **no guard** that restricts `LinkSigner` mutations to the subaccount owner only. The slow-mode path correctly enforces `validateSender(txn.sender, sender)` (i.e., `msg.sender` must be the owner), but the sequencer fast path has no equivalent restriction. [4](#0-3) 

---

### Impact Explanation

Once an attacker is the linked signer for a victim subaccount (via any prior legitimate or social-engineering authorization), they can:

1. Sign `LinkSigner(sender=victim_subaccount, signer=attacker2, nonce=N)` with their own key.
2. Submit it through the sequencer.
3. `validateSignedTx` passes because `recovered == getLinkedSigner(victim_subaccount) == attacker`.
4. `linkedSigners[victim_subaccount]` is overwritten with `attacker2`.
5. `attacker2` can now sign `WithdrawCollateral` (also `allowLinkedSigner=true`) to drain all collateral from the victim subaccount.

The victim retains the ability to sign directly as the owner and can submit a corrective `LinkSigner` via slow-mode, but the sequencer fast path creates a race condition window during which `attacker2` can execute withdrawals before the victim can react.

---

### Likelihood Explanation

Any party that has ever been granted linked-signer access (e.g., a trading bot, a delegated key, a compromised API key) can execute this attack unilaterally. No additional privileges, governance capture, or sequencer compromise are required. The sequencer is a passive relay and will process the transaction as long as the signature is valid.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`, so that only the subaccount owner (the address embedded in `sender`) can authorize changes to the linked signer relationship:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only owner may change linked signer
);
``` [5](#0-4) 

---

### Proof of Concept

```
1. victim_subaccount = bytes32(abi.encodePacked(victimAddress, subaccountName))
2. victim signs LinkSigner(sender=victim_subaccount, signer=attacker, nonce=0)
   → linkedSigners[victim_subaccount] = attacker
3. attacker signs LinkSigner(sender=victim_subaccount, signer=attacker2, nonce=1)
   → validateSignedTx passes: recovered == attacker == getLinkedSigner(victim_subaccount)
   → linkedSigners[victim_subaccount] = attacker2
4. attacker2 signs WithdrawCollateral(sender=victim_subaccount, amount=MAX, nonce=2)
   → validateSignedTx passes: recovered == attacker2 == getLinkedSigner(victim_subaccount)
   → victim's collateral is withdrawn to attacker2's address
5. Assert: linkedSigners[victim_subaccount] == attacker2
           victim collateral balance == 0
```
