### Title
`SwapAllowlistExtension.beforeSwap` checks the router address (`sender`) instead of the actual end user, allowing any unprivileged user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension` is intended to gate pool swaps by swapper identity. Its `beforeSwap` hook checks `sender`, which is the direct caller of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router address, not the actual end user. If the pool admin allowlists the router — a natural and expected action to enable standard router-mediated swaps — every unprivileged user can bypass the allowlist entirely. This is structurally inconsistent with `DepositAllowlistExtension`, which correctly checks `owner` (the actual position beneficiary) and therefore works correctly through the router.

---

### Finding Description

**Root cause — wrong identity checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-L41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`sender` is populated by `ExtensionCalling._beforeSwap`, which passes `msg.sender` of the `pool.swap()` call:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-L240
_beforeSwap(
    msg.sender,   // ← this becomes `sender` in the extension
    recipient,
    zeroForOne,
    ...
);
```

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-L176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(
        IMetricOmmExtensions.beforeSwap,
        (sender, recipient, ...)  // sender = msg.sender of pool.swap()
    )
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()`, so `msg.sender` inside the pool is the **router address**. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Structural inconsistency with `DepositAllowlistExtension`:**

`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` — the actual position beneficiary — not `sender`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-L42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Because `addLiquidity` passes the actual user as `owner`, the deposit allowlist works correctly through the router. The swap allowlist does not.

**Exploit path:**

1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to specific counterparties.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable standard router-mediated swaps (a natural, expected action — without this, no user can swap through the router at all).
3. Any unprivileged, non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The router calls `pool.swap(recipient=user, ...)` with `msg.sender` = router.
5. The extension evaluates `allowedSwapper[pool][router]` = `true` → swap proceeds.
6. The non-allowlisted user has fully bypassed the allowlist.

The pool admin faces an impossible choice: either allowlist the router (allowing all users to bypass the allowlist) or do not allowlist the router (making the router unusable for all users, including allowlisted ones).

---

### Impact Explanation

The swap allowlist is rendered completely ineffective whenever the router is allowlisted. Any user — regardless of allowlist status — can execute swaps in a pool intended to be restricted to specific counterparties. In pools designed for institutional or permissioned trading (e.g., with specific price agreements or regulatory requirements), this allows unauthorized parties to extract value at pool-set prices, directly harming LP principal and disrupting the pool's intended economic model. This is an admin-boundary break: an unprivileged path (router) bypasses a configured access control gate.

---

### Likelihood Explanation

The trigger condition — allowlisting the router — is the natural and expected configuration for any pool that wants to support standard user-facing swaps. A pool admin who configures `SwapAllowlistExtension` to restrict swaps to specific addresses will almost certainly also allowlist the router to enable normal operation, unaware that doing so voids the allowlist entirely. The inconsistency with `DepositAllowlistExtension` (which works correctly through the router) reinforces the expectation that the swap allowlist should behave the same way.

---

### Recommendation

Replace the `sender` check with a check on the actual end user. The `recipient` argument (second parameter, currently unnamed/ignored) is the address that receives swap output and is set to the actual user by the router. Alternatively, require the caller to pass the actual user identity in `extensionData` and verify it against the allowlist. The fix should mirror the pattern used by `DepositAllowlistExtension`, which checks `owner` (the actual beneficiary) rather than the direct caller:

```solidity
// Proposed fix: check recipient (actual end user) instead of sender
function beforeSwap(address, address recipient, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```
