### Title
`Clearinghouse::setWithdrawPool` Desynchronizes Fast-Withdrawal Replay Protection on Pool Replacement — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawPool` is a mutable global address set at initialization and used in every withdrawal flow. When the owner replaces it via `setWithdrawPool`, the new `WithdrawPool` contract starts with empty replay-protection state (`markedIdxs`, `minIdx = 0`). Any previously-settled fast-withdrawal transaction — which carries permanent verifier signatures — can be replayed against the new pool, draining its pre-funded liquidity.

---

### Finding Description

`Clearinghouse.withdrawPool` is assigned during `initialize`: [1](#0-0) 

It is the sole destination for every on-chain token transfer in `handleWithdrawTransfer`, which is called by both `withdrawCollateral` and `withdrawInsurance`: [2](#0-1) 

The owner can silently replace this address at any time: [3](#0-2) 

Fast-withdrawal replay protection lives entirely inside the `BaseWithdrawPool` contract as two storage variables: [4](#0-3) 

`submitFastWithdrawal` enforces replay protection using only these two fields: [5](#0-4) 

A freshly deployed `WithdrawPool` initialises with `markedIdxs` empty and `minIdx = 0`. Because `setWithdrawPool` performs no state migration, every `idx` that was already marked in the old pool is unknown to the new pool. The verifier signatures that authorised the original fast withdrawal are permanent and reusable — the same `(idx, transaction, signatures)` tuple passes `requireValidTxSignatures` on the new pool identically to how it passed on the old one. [6](#0-5) 

---

### Impact Explanation

Any user who previously submitted a fast withdrawal can replay the identical call against the new pool and receive a second payout. Because fast withdrawals are funded by pre-deposited liquidity (the pool must hold tokens before it can pay out), and because the admin must seed the new pool with liquidity for it to serve its purpose, the new pool's entire liquidity balance is at risk. There is no mechanism to recover these funds — the pool has no `recoverERC20`-style function, and `removeLiquidity` is owner-only but the funds have already left.

---

### Likelihood Explanation

The trigger is a legitimate administrative upgrade (`setWithdrawPool`), not a malicious action. The attack window opens the moment the new pool is deployed and funded, and closes only after `minIdx` is advanced past all previously-issued fast-withdrawal indices by processing slow-mode withdrawals. Any user who has ever used fast withdrawals holds a replayable `(idx, transaction, signatures)` tuple. The attack requires no special privilege, no front-running, and no new signed message — only the original transaction bytes that the attacker already possesses.

---

### Recommendation

When `setWithdrawPool` is called, initialise the new pool's `minIdx` to the current `nSubmissions` value from the `Endpoint`, so that all previously-issued fast-withdrawal indices are unconditionally rejected:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    // Advance minIdx so old fast-withdrawal transactions cannot be replayed
    BaseWithdrawPool(_withdrawPool).setMinIdx(
        IEndpoint(getEndpoint()).nSubmissions()
    );
    withdrawPool = _withdrawPool;
}
```

Alternatively, store the `withdrawPool` address inside each slow-mode withdrawal record at submission time (analogous to the recommended fix in the external report of storing `assetToken` inside the `Application` struct), so that execution always uses the pool that was active when the withdrawal was queued.

---

### Proof of Concept

1. User calls `submitFastWithdrawal(idx=500, transaction, signatures)` on `oldPool`. Pool pays out `amount` tokens. `oldPool.markedIdxs[500] = true`.
2. Owner calls `Clearinghouse.setWithdrawPool(newPool)`. `newPool.markedIdxs` is empty; `newPool.minIdx = 0`.
3. Owner seeds `newPool` with liquidity (required for fast withdrawals to function).
4. User calls `submitFastWithdrawal(idx=500, transaction, signatures)` on `newPool`.
   - `newPool.markedIdxs[500]` → `false` ✓
   - `500 > newPool.minIdx (0)` ✓
   - `requireValidTxSignatures` passes (same verifier, same signatures) ✓
5. `newPool` pays out `amount` tokens again to the user.
6. User has received `2 × amount` tokens; `newPool` liquidity is drained. [7](#0-6) [3](#0-2)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
