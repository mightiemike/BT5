### Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Enabling Allowlist Bypass or Blocking Allowlisted Users - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which the pool sets to `msg.sender` of `MetricOmmPool.swap`. When users route through `MetricOmmSimpleRouter`, `sender` is the **router address**, not the actual user. This creates a structural mismatch between the configured allowlist (keyed on user addresses) and the identity actually checked at hook time (the router). The result is a binary failure: either allowlisted users cannot use the router at all, or — if the pool admin allowlists the router to support router-mediated swaps — any unprivileged user can bypass the allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making itself `msg.sender` of that call: [4](#0-3) 

The same pattern holds for `exactInput` (all hops) and `exactOutput`: [5](#0-4) 

The extension ignores `extensionData` entirely and has no mechanism to recover the original caller's identity. There is no path through the router that delivers the actual user address as `sender` to the hook.

This produces two mutually exclusive failure modes for any pool that uses `SwapAllowlistExtension`:

**Mode A — Allowlist bypass:** The pool admin allowlists the router address (a natural choice to support router-mediated swaps for allowlisted users). Because `sender = router` for every router-originated call, `allowedSwapper[pool][router] == true` passes the check for **every** caller, regardless of whether that caller is on the intended allowlist. Any unprivileged user routes through `MetricOmmSimpleRouter` and trades on the curated pool.

**Mode B — Allowlisted users blocked:** The pool admin allowlists specific user addresses but does not allowlist the router. Allowlisted users who call through the router have `sender = router`, which is not in `allowedSwapper`, so their swaps revert with `NotAllowedToSwap`. The router — the primary supported periphery path — is permanently unusable for every allowlisted user on that pool.

The `extensionData` field is available in the hook signature but is unused by `SwapAllowlistExtension`, so there is no workaround short of calling `pool.swap()` directly.

---

### Impact Explanation

**Mode A (allowlist bypass):** Any unprivileged user can trade on a pool whose admin intended to restrict access to a curated set of counterparties. Depending on pool design, this can mean unauthorized parties drain LP-favorable pricing, violate compliance-gated pools, or front-run positions that the allowlist was meant to protect. This is a direct admin-boundary break reachable by any public user through the supported router path.

**Mode B (broken core functionality):** Every allowlisted user is permanently blocked from using `MetricOmmSimpleRouter` on the curated pool. The primary supported swap path is unusable, forcing users to interact directly with the pool contract — a broken core flow.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. Pool admins who deploy a `SwapAllowlistExtension` pool and want to support router-mediated swaps for their allowlisted users will naturally add the router to `allowedSwapper`, triggering Mode A. Alternatively, admins who do not add the router will immediately observe that their allowlisted users cannot swap via the router (Mode B). Both failure modes are reachable through normal, expected usage of the protocol's own periphery. No special privileges or malicious setup are required.

---

### Recommendation

The `beforeSwap` hook should check the **economically relevant actor** — the address that initiated the trade and will pay for it — rather than the immediate caller of `pool.swap`. Two complementary fixes:

1. **Pass the original caller through `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling `pool.swap`; `SwapAllowlistExtension` decodes and checks that address when `extensionData` is non-empty, falling back to `sender` for direct calls.

2. **Check `sender` only for direct calls; require routers to attest the real user:** Define a standard extension-data envelope that trusted routers populate with the real user address, and have `SwapAllowlistExtension` verify it. This mirrors how `DepositAllowlistExtension` correctly gates on `owner` (the position owner) rather than `sender` (the liquidity adder contract).

The deposit allowlist already demonstrates the correct pattern — it checks `owner`, the economically relevant actor, not `sender`: [6](#0-5) 

The swap allowlist should adopt the same principle.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for allowlisted users.
3. Attacker (address not in `allowedSwapper`) calls `MetricOmmSimpleRouter.exactInputSingle` targeting the pool.
4. Router calls `pool.swap(recipient, ...)` with `msg.sender = router`.
5. Pool calls `_beforeSwap(router, ...)`.
6. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes on the curated pool despite never being allowlisted.

Alternatively, for Mode B:

1. Pool admin calls `setAllowedToSwap(pool, alice, true)` but does **not** allowlist the router.
2. Alice calls `MetricOmmSimpleRouter.exactInputSingle`.
3. Router calls `pool.swap(...)` with `msg.sender = router`.
4. `SwapAllowlistExtension` checks `allowedSwapper[pool][router] == false` → reverts `NotAllowedToSwap`.
5. Alice, despite being allowlisted, cannot use the router.
