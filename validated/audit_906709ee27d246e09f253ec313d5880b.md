## Analysis

I traced the full path from protobuf receipt to storage to determine whether the described scenario is reachable and impactful.

### Deserialization path (converters/header.rs lines 93-115)

The converter maps `Patricia.root` (which is `optional` in the protobuf) directly to `Option<TransactionCommitment>` / `Option<EventCommitment>` with no guard on `n_leaves`: [1](#0-0) 

`n_transactions` is extracted from `n_leaves` unconditionally, while `transaction_commitment` is `None` when `root` is absent. The same applies to `event_commitment`. No error is returned; deserialization succeeds.

### Validation in parse_data_for_block

The only checks performed after deserialization are: [2](#0-1) 

- Block number ordering
- Signatures vector length == 1

There is **no** check that `transaction_commitment` is `Some` when `n_transactions > 0`, no block hash recomputation, and no signature verification against the block hash.

### Storage write

`write_to_storage` calls `append_header` directly with whatever is in the `BlockHeader`: [3](#0-2) 

`append_header` stores all fields verbatim, including `None` commitments: [4](#0-3) 

### The guard that exists — but is never reached in this path

`TryFrom<&BlockHeader> for Option<BlockHeaderCommitments>` does check for missing commitments on versions ≥ 0.13.2: [5](#0-4) 

However, this guard is only invoked when someone tries to **compute** the block hash from the stored header. The p2p sync client never calls it — it stores the attacker-supplied `block_hash` field directly without recomputing or verifying it.

### Signature verification — also absent

`verify_block_signature` exists: [6](#0-5) 

But it is never called in the p2p sync client header path. The attacker only needs to supply a signatures vector of length 1 with any content.

### Confirmed attack path

1. Attacker sends `protobuf::SignedBlockHeader` with `transactions = Patricia{n_leaves: 5, root: None}`, `events = Patricia{n_leaves: 3, root: None}`, `protocol_version = "0.13.2"`, and an arbitrary `block_hash`.
2. Deserialization succeeds: `n_transactions = 5`, `transaction_commitment = None`, `n_events = 3`, `event_commitment = None`.
3. `parse_data_for_block` passes (block number matches, signatures length == 1).
4. `append_header` stores the `BlockHeader` with `n_transactions = 5`, `transaction_commitment = None`, `n_events = 3`, `event_commitment = None`, and the attacker-supplied `block_hash`.

The existing test `block_header_without_commitments_to_bytes_and_back` in `header_test.rs` explicitly confirms that round-tripping `None` commitments is a supported and tested code path: [7](#0-6)

### Title
P2P sync stores `BlockHeader` with `transaction_commitment = None` / `event_commitment = None` and attacker-controlled `block_hash` when peer sends `Patricia{n_leaves > 0, root: None}` — (`crates/apollo_protobuf/src/converters/header.rs`)

### Summary
An unprivileged p2p peer can send a `SignedBlockHeader` protobuf with `transactions.root` and/or `events.root` absent while `n_leaves > 0`. The converter at lines 93–115 maps the absent root to `None` without checking `n_leaves`, producing a `BlockHeader` with `n_transactions > 0` but `transaction_commitment = None`. The p2p sync client performs no block-hash recomputation and no signature verification, so the header — including an attacker-chosen `block_hash` — is written to storage unconditionally.

### Finding Description
`TryFrom<protobuf::SignedBlockHeader> for SignedBlockHeader` extracts `n_transactions` from `n_leaves` and `transaction_commitment` from the optional `root` field independently. When `root` is absent, `transaction_commitment` is `None` regardless of `n_leaves`. The same applies to `event_commitment`. The p2p sync client's `parse_data_for_block` only validates block number ordering and signature-vector length; it never checks commitment completeness, recomputes the block hash, or verifies the signature against the block hash. `write_to_storage` then calls `append_header` which stores all fields verbatim.

The guard in `TryFrom<&BlockHeader> for Option<BlockHeaderCommitments>` that would catch missing commitments for versions ≥ 0.13.2 is never invoked during the p2p sync write path.

### Impact Explanation
The node stores a `BlockHeader` where `n_transactions > 0` but `transaction_commitment = None` (and/or `n_events > 0` but `event_commitment = None`), and the stored `block_hash` is whatever the attacker supplied. This data is subsequently:
- Served to RPC clients via `starknet_getBlockWithTxHashes` / `starknet_getBlockWithReceipts` as authoritative block metadata with a wrong block hash.
- Re-served to other p2p peers via `FetchBlockData for SignedBlockHeader`, propagating the corruption.
- If any downstream code path attempts to compute the block hash from the stored header for a version ≥ 0.13.2, it will receive `Err(MissingBlockHeaderCommitments)`, potentially causing node errors.

### Likelihood Explanation
Any peer that can establish a p2p connection and respond to a `BlockHeadersRequest` can trigger this. No special privileges are required. The attacker only needs to craft a valid-looking `SignedBlockHeader` protobuf with `root` fields omitted.

### Recommendation
In `parse_data_for_block` (or in the `TryFrom` converter itself), add a validation step that rejects a `SignedBlockHeader` where `n_leaves > 0` but `root` is absent. Additionally, for versions ≥ 0.13.2, verify that all four commitments (`transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`) are `Some` before accepting the header. Consider also recomputing and verifying the block hash against the stored commitments before writing to storage.

    block_hash: Some(fake_hash.into()),
    parent_hash: Some(parent_hash.into()),

    time: 0,

    state_root: Some(state_root.into()),

        root: None, // also None
    }),
    transactions: Some(protobuf::Patricia {
        n_leaves: 5,
        root: None, // <-- absent root with n_leaves > 0
    }),
    events: Some(protobuf::Patricia {
        n_leaves: 3,
        root: None, // <-- absent root with n_leaves > 0
    }),
    protocol_version: "0.13.2".to_string(),
