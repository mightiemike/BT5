### Title
Deposit to a Delisted Product Permanently Locks User Funds — (`File: core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` in `Endpoint.sol` immediately transfers ERC-20 tokens from the caller before the corresponding slow-mode transaction is processed on-chain. No check is made that the target `productId` is still active at deposit time. If the product is delisted between the deposit call and slow-mode execution, the slow-mode transaction fails silently and the tokens are permanently locked — the refund path that previously existed was explicitly removed (evidenced by the comment `// try return funds now removed`).

---

### Finding Description

**Step 1 — Immediate token transfer with no product-state guard**

`depositCollateralWithReferral` calls `handleDepositTransfer` at line 144 before any slow-mode queue entry is created: [1](#0-0) 

The only pre-transfer checks are sanctions screening and a minimum-amount guard. There is no assertion that `productId` is currently listed or active.

**Step 2 — Slow-mode transaction is queued with a 3-day delay**

Immediately after the transfer, a `DepositCollateral` slow-mode entry is pushed with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`: [2](#0-1) 

This creates a 3-day window during which the product's state can change.

**Step 3 — Products can be delisted via the sequencer**

`processSlowModeTransactionImpl` in `EndpointTx.sol` handles a `DelistProduct` transaction type that calls `clearinghouse.delistProduct`: [3](#0-2) 

A sequencer-submitted delist can be processed at any time, including during the 3-day window after a deposit slow-mode tx has been queued.

**Step 4 — Slow-mode execution silently swallows failures with no refund**

`_executeSlowModeTransaction` wraps `processSlowModeTransaction` in a `try/catch`. The catch block does nothing — the comment explicitly states the refund path was removed: [4](#0-3) 

If `clearinghouse.depositCollateral` reverts because the product is delisted, the exception is caught, the slow-mode entry is consumed, and the tokens already held by the contract are never credited to the user and cannot be recovered.

---

### Impact Explanation

A user's deposited ERC-20 tokens are transferred to the `Endpoint` contract at deposit time. If the corresponding slow-mode transaction fails (product delisted), those tokens remain in the contract with no on-chain mechanism to return them. The user suffers a permanent, total loss of the deposited amount. The `Endpoint` contract has no owner-callable rescue function for stuck deposit tokens.

---

### Likelihood Explanation

The slow-mode delay is hardcoded to 3 days (`SLOW_MODE_TX_DELAY`): [5](#0-4) 

Any product delist processed by the sequencer within that 3-day window after a deposit is queued triggers the loss. A user who deposits into a product that is being wound down (a realistic operational event) is at risk. The user has no way to cancel the queued slow-mode transaction once submitted.

---

### Recommendation

Add an active-product guard in `depositCollateralWithReferral` before calling `handleDepositTransfer`. Concretely, verify that `spotEngine.getToken(productId)` returns a non-zero address **and** that the product is present in `spotEngine.getProductIds()` (or expose a dedicated `isListedProduct` view). Alternatively, restore a refund path in `_executeSlowModeTransaction` for `DepositCollateral` failures so that tokens are returned to the original sender when the slow-mode tx cannot be applied.

---

### Proof of Concept

1. Product `X` is listed in `SpotEngine` with token `T`.
2. User calls `Endpoint.depositCollateralWithReferral(subaccount, X, amount, referral)`.
   - `handleDepositTransfer` pulls `amount` of token `T` from the user into `Endpoint`. [1](#0-0) 
   - A `DepositCollateral` slow-mode entry is queued, executable in 3 days. [2](#0-1) 
3. Within the 3-day window, the sequencer submits a `DelistProduct` transaction for product `X`. [3](#0-2) 
4. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`. `_executeSlowModeTransaction` calls `processSlowModeTransaction`, which calls `clearinghouse.depositCollateral` — this reverts because product `X` is delisted.
5. The `catch` block in `_executeSlowModeTransaction` fires. No refund is issued (`// try return funds now removed`). [6](#0-5) 
6. The user's `amount` of token `T` is permanently locked in the `Endpoint` contract with no recovery path.
