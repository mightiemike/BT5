### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain Without USDC Payment — (`File: core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` without checking its return value. If the USDC token returns `false` on failure instead of reverting, the function continues execution, withdraws usdcE from the `DirectDepositV1` vault, and transfers it to the caller — without the caller ever providing USDC.

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no access control beyond a chain ID check (`block.chainid == 57073`). Its purpose is to let any caller swap usdcE held in a `DirectDepositV1` address for USDC: the caller sends USDC in, and receives usdcE out.

The critical flaw is on line 616:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value of `transferFrom` is silently discarded. The `IERC20Base` interface declares `transferFrom` as returning `bool` (line 25–29 of `IERC20Base.sol`). If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` on failure (e.g., caller has no allowance or insufficient balance) rather than reverting, execution continues to:

```solidity
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));  // pulls usdcE to ContractOwner
IERC20Base(usdcE).safeTransfer(msg.sender, balance);             // sends usdcE to attacker
```

The inconsistency is stark: the outbound `safeTransfer` on line 618 uses the `ERC20Helper` safe wrapper (which checks the return value), but the inbound `transferFrom` on line 616 does not. The `using ERC20Helper for IERC20Base` directive is already declared at line 24, making `safeTransferFrom` directly available but unused here. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

If the USDC token on chain 57073 returns `false` on a failed `transferFrom` (caller has zero allowance or zero balance), an attacker can:

1. Call `replaceUsdcEWithUsdc(subaccount)` for any `subaccount` whose `DirectDepositV1` holds usdcE.
2. The silent `transferFrom` failure is ignored.
3. `DirectDepositV1.withdraw` transfers all usdcE from the vault to `ContractOwner`.
4. `safeTransfer` sends all usdcE to the attacker.

**Asset delta**: full usdcE balance of the targeted `DirectDepositV1` is drained to the attacker at zero cost. The `DirectDepositV1` vault loses its entire usdcE holding, and the corresponding subaccount's collateral is permanently lost. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

- The function is `external` with no `onlyOwner` or `onlyDeployer` guard — any unprivileged caller on chain 57073 can invoke it.
- The hardcoded USDC address (`0x2D270e6886d130D724215A266106e6832161EAEd`) is a chain-specific deployment whose exact behavior (revert vs. return-false) depends on the token implementation at that address on Ink chain. Non-Circle or bridged USDC variants commonly return `false` rather than revert.
- The `ERC20Helper.safeTransferFrom` wrapper is already present in the codebase and imported, making this a straightforward omission rather than a design gap. [6](#0-5) 

---

### Recommendation

Replace the raw `transferFrom` call on line 616 with the already-available `safeTransferFrom` from `ERC20Helper`:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` in `ERC20Helper` uses a low-level `call` and requires `success && (data.length == 0 || abi.decode(data, (bool)))`, which correctly handles both reverting and return-false token implementations. [3](#0-2) 

---

### Proof of Concept

1. A `DirectDepositV1` exists for `subaccount S` on chain 57073, holding 1000 usdcE.
2. Attacker EOA has 0 USDC and 0 USDC allowance granted to `ContractOwner`.
3. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(S)`.
4. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` → 1000.
5. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 1000)` → returns `false` (no allowance/balance); return value discarded, no revert.
6. `DirectDepositV1(directDepositV1).withdraw(usdcE)` → transfers 1000 usdcE to `ContractOwner`.
7. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` → attacker receives 1000 usdcE.
8. Attacker has drained the vault's usdcE at zero cost; subaccount S's collateral is permanently lost. [7](#0-6)

```text
    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
```
