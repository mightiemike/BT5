Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Originating User, Enabling Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the direct caller of the pool. When swaps are routed through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract, not the originating user. Any pool admin who allowlists the router address (a natural step to enable router-based swaps for vetted users) inadvertently opens the curated pool to every user who calls through the router, regardless of individual allowlist status.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

In `MetricOmmPool.swap` (lines 230–240), the pool calls `_beforeSwap(msg.sender, ...)`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the router contract address, not the end user. [1](#0-0) 

**`ExtensionCalling._beforeSwap` forwards `sender` unchanged:**

`_beforeSwap` encodes and forwards the `sender` value directly to every configured extension without modification. [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks the router, not the user:**

The extension checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the router. If `allowedSwapper[pool][router] == true`, the check passes for every call arriving through the router, regardless of who the originating user is. [3](#0-2) 

**`MetricOmmSimpleRouter` calls `pool.swap` directly, making itself `msg.sender`:**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, and the recursive `exactOutput` callback path all call `pool.swap(...)` directly from the router contract. [4](#0-3) [5](#0-4) [6](#0-5) 

**Exploit call chain:**
```
Alice (not allowlisted)
  → MetricOmmSimpleRouter.exactInputSingle(pool, ...)
      → pool.swap(recipient=Alice, ...)   [msg.sender = router]
          → _beforeSwap(sender=router, ...)
              → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                  → allowedSwapper[pool][router] == true  ✓ (bypass)
                  → swap executes
```

**Existing guards are insufficient:** The only guard is `allowedSwapper[pool][sender]`. There is no mechanism to recover the originating user's address from within the extension, and `extensionData` is caller-supplied and unauthenticated, so it cannot be trusted to carry the real user address.

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router (to enable router-based swaps for vetted users) inadvertently grants swap access to every address that calls through the router. Non-allowlisted users can execute swaps against LP positions that were provisioned under the assumption that only vetted counterparties would trade. This constitutes an admin-boundary break and broken core pool functionality: the allowlist invariant — that only explicitly approved addresses may swap — is violated, with direct fund impact on LPs whose risk assumptions are defeated.

Conversely, if the admin does not allowlist the router, individually allowlisted users cannot use the primary swap entry point at all, breaking the expected user flow.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap entry point. Any pool admin configuring a `SwapAllowlistExtension` who also wants router support will naturally allowlist the router, triggering the bypass. The attacker requires no special privilege — only the ability to call the public router. The condition is reachable by any unprivileged address and is repeatable without limit.

## Recommendation

Pass the originating user through the swap path rather than the direct caller. Two concrete options:

1. **Preferred — explicit `originator` field:** Extend the `swap` signature with an `originator` address that the router sets to `msg.sender` before calling the pool, and have the pool forward that value to extensions instead of (or in addition to) `msg.sender`. The pool should enforce that only trusted routers may set a non-zero `originator`.

2. **Alternative — authenticated `extensionData`:** Have `SwapAllowlistExtension` accept an `originatorData` field inside `extensionData` that the router populates with the user's address and signs (e.g., via EIP-712), and verify the signature inside the extension. This avoids changing the core pool interface but requires router-level signing support.

Until fixed, pool admins must be warned that allowlisting the router address opens the curated pool to all users.

## Proof of Concept

```
Setup:
  1. Deploy pool with SwapAllowlistExtension (beforeSwap order = extension 1).
  2. Pool admin calls setAllowedToSwap(pool, router, true)
     (natural step to enable router-based swaps for allowlisted users).
  3. Alice is NOT individually allowlisted.

Attack:
  1. Alice calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...}).
  2. Router calls pool.swap(recipient=Alice, ...) — msg.sender inside pool = router.
  3. Pool calls _beforeSwap(sender=router, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] → true.
  5. Swap executes. Alice trades on a pool she was never authorized to access.

Expected: revert NotAllowedToSwap.
Actual:   swap succeeds.

Foundry test outline:
  - deployPool(extensions=[swapAllowlist], beforeSwapOrder=[1])
  - swapAllowlist.setAllowedToSwap(pool, address(router), true)
  - vm.prank(alice); router.exactInputSingle(...)
  - assert swap succeeded (no revert)
```
