### Title
Sanctioned Address Can Be Registered as `linkedSigner` to Bypass Sanctions Enforcement - (File: `core/contracts/EndpointTx.sol`)

### Summary
The `LinkSigner` transaction handler in `EndpointTx.sol` stores an arbitrary address as the authorized signer for a subaccount without checking whether that address is sanctioned. A KYC-verified user can link a sanctioned address as their `linkedSigner`, granting that sanctioned entity full signing authority over the subaccount — including trades, withdrawals, and quote transfers — while the sanctions check is never applied to the linked signer.

### Finding Description
Nado's subaccount model allows any subaccount owner to designate a `linkedSigner` — an address whose ECDSA signatures are accepted as equivalent to the subaccount owner's own signature for all protocol operations.

The sanctions check in `depositCollateralWithReferral` is applied to `msg.sender` and the subaccount owner address extracted from the `subaccount` bytes32: [1](#0-0) 

However, when a `LinkSigner` transaction is processed — both in the fast-mode sequencer path and the slow-mode path — the signer address is written directly to `linkedSigners` with no `requireUnsanctioned` call on the signer:

**Fast-mode path** (`processTransactionImpl`): [2](#0-1) 

**Slow-mode path** (`processSlowModeTransactionImpl`): [3](#0-2) 

Once registered, the linked signer is retrieved and passed directly to `verifier.validateSignature` / `verifier.validateCompactSignature` without any sanctions check at use time: [4](#0-3) 

The `Verifier` accepts the signature if `recovered == linkedSigner`, with no further checks: [5](#0-4) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [6](#0-5) 

### Impact Explanation
A sanctioned entity gains full signing authority over a KYC-verified subaccount. It can sign and submit (via the sequencer) any transaction type that accepts a linked signer: spot/perp trades, withdrawals, quote transfers, NLP mint/burn, and isolated subaccount creation. The sanctioned entity never appears as `msg.sender` or as the subaccount owner, so the only existing sanctions gate — at deposit time — is never triggered. The sanctioned entity can profit from trades and extract funds via withdrawal, all while the on-chain sanctions enforcement is completely bypassed.

**Corrupted invariant**: The protocol's invariant that sanctioned addresses cannot interact with the exchange is broken. The `linkedSigner` address is a reachable, attacker-controlled state variable that is never validated against the sanctions list.

### Likelihood Explanation
Low/Medium. This requires a KYC-verified user to cooperate — either willingly (ToS violation) or through social engineering / coercion. Once the `LinkSigner` transaction is submitted and processed by the sequencer, the sanctioned entity can operate permissionlessly. The `linkedSigner` persists until explicitly revoked by the subaccount owner, so the window of exposure is open-ended.

### Recommendation
Apply `requireUnsanctioned` to the signer address in both `LinkSigner` processing paths before writing to `linkedSigners`:

```solidity
// In processSlowModeTransactionImpl (slow-mode LinkSigner)
requireUnsanctioned(address(uint160(bytes20(txn.signer))));
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));

// In processTransactionImpl (fast-mode LinkSigner)
address newSigner = address(uint160(bytes20(signedTx.tx.signer)));
requireUnsanctioned(newSigner);
linkedSigners[signedTx.tx.sender] = newSigner;
```

Additionally, consider adding a sanctions check on the `linkedSigner` at signature validation time in `getLinkedSigner`, so that a signer who becomes sanctioned after registration is automatically blocked.

### Proof of Concept

1. Sanctioned address `0xBAD` is on the Chainalysis sanctions list.
2. KYC-verified user `0xALICE` owns subaccount `bytes32(abi.encodePacked(0xALICE, bytes12("default")))`.
3. `0xALICE` submits a `LinkSigner` slow-mode transaction with `signer = bytes32(uint256(uint160(0xBAD)))`.
4. The sequencer processes it via `processSlowModeTransactionImpl`. The only check is `validateSender(txn.sender, sender)` — no sanctions check on `txn.signer`.
5. `linkedSigners[aliceSubaccount] = 0xBAD` is written.
6. `0xBAD` now signs a `WithdrawCollateral` transaction for `aliceSubaccount`. The sequencer submits it.
7. `validateSignedTx` calls `validateCompactSignature` → `verifier.validateCompactSignature(aliceSubaccount, 0xBAD, digest, sig)`.
8. `ECDSA.recover` returns `0xBAD`, which equals `linkedSigner` → signature accepted.
9. `clearinghouse.withdrawCollateral` executes, transferring funds to `0xBAD`'s chosen recipient.
10. The sanctions list was never consulted after step 2. [3](#0-2) [7](#0-6) [8](#0-7)
