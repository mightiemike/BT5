Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Any User to Bypass the Allowlist Guard — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the immediate `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks whether the **router** is allowlisted — not the actual end-user. Any pool that allowlists the router (the natural operational choice) becomes fully open to all users, silently defeating the allowlist guard.

## Finding Description

**Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap()` at lines 230–240 calls `_beforeSwap(msg.sender, recipient, ...)`. Whoever called `pool.swap()` becomes the `sender` argument forwarded to the extension. [1](#0-0) 

**Router calls `pool.swap()` directly, making itself `msg.sender`.**

`MetricOmmSimpleRouter.exactInputSingle()` at lines 71–80 calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. The router is `msg.sender` from the pool's perspective; the original user's address is stored only in transient callback context (`_setNextCallbackContext`) and is never forwarded to the pool or extension. [2](#0-1) 

The same pattern applies to `exactInput` (L103–112), `exactOutputSingle` (L135–137), and `exactOutput` (L165–181). [3](#0-2) 

**Extension checks the router address, not the user.**

`SwapAllowlistExtension.beforeSwap` at line 37 evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` = pool and `sender` = router. The actual end-user's address is never consulted. [4](#0-3) 

The allowlist mapping is keyed `(pool → swapper → bool)` and is documented as gating individual swappers, but the `sender` value it receives is structurally the immediate caller of `pool.swap()`, not the economic actor. [5](#0-4) 

**Existing guards are insufficient.** The `allowAllSwappers` flag and `allowedSwapper` mapping both operate on the same `sender` argument, so neither path can distinguish the router from the user. There is no mechanism in the extension, pool, or router to thread the original `msg.sender` through to the hook.

## Impact Explanation

A pool admin deploying a curated pool (KYC-gated, institutional-only, whitelist-only) with `SwapAllowlistExtension` must allowlist the router to permit approved users to trade via the standard periphery. Once the router is allowlisted, `allowedSwapper[pool][router]` returns `true` for every call, and every address on the network can trade on the pool by calling any router function. The allowlist guard is completely inoperative for all router-mediated swaps. Non-approved users receive identical execution to approved ones, violating the pool admin's explicit access-control policy and exposing LP assets to trades the admin prohibited. This is a direct admin-boundary break by an unprivileged path — any EOA — with no special permissions required.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point. Any pool admin who wants approved users to trade via the router must allowlist the router, which is the expected operational pattern. The bypass is therefore reachable on every production pool using `SwapAllowlistExtension` that accepts router-mediated swaps. No malicious setup, non-standard tokens, or elevated permissions are required — any EOA calling `exactInputSingle` on the router suffices.

## Recommendation

The extension must gate the economically relevant actor — the end-user — not the immediate caller of `pool.swap()`. The cleanest fix is to thread the original `msg.sender` from the router through to the extension as a distinct `originator` field: the router records `msg.sender` before calling the pool (it already does so in transient storage via `_setNextCallbackContext`), and the pool forwards it as a separate argument to the extension hooks. Alternatively, the extension can be documented as incompatible with router-mediated swaps and revert in `beforeSwap` when `sender` is a known router address, though this limits composability.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension (beforeSwap hook enabled)
  - Pool admin calls setAllowedToSwap(pool, router, true)   // allowlist the router
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker (non-allowlisted EOA) calls:
       router.exactInputSingle({pool: pool, recipient: attacker, ...})
  2. Router calls pool.swap(attacker, ...) — msg.sender to pool = router
  3. Pool calls _beforeSwap(msg.sender=router, recipient=attacker, ...)
  4. SwapAllowlistExtension.beforeSwap receives sender=router
  5. Checks allowedSwapper[pool][router] → true ✓
  6. Swap executes successfully for the non-allowlisted attacker

Result:
  - Non-allowlisted user completes a swap on a curated pool
  - SwapAllowlistExtension guard is fully bypassed
  - Pool admin's access-control policy is silently violated
```
