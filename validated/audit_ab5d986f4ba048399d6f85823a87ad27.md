### Title
Unspent Allowance Left by Silent Early Return in `depositCollateralWithReferral` Permanently Bricks USDT Deposits via `DirectDepositV1.creditDeposit()` - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` is a public, permissionless function that calls `token.approve(endpoint, balance)` before calling `endpoint.depositCollateralWithReferral(...)`. However, `depositCollateralWithReferral` contains a **silent early return** that skips the actual `transferFrom` when the deposit amount is below the minimum threshold — leaving the allowance set but unconsumed. For USDT-like tokens that revert on a non-zero `approve()` when a non-zero allowance already exists, any subsequent call to `creditDeposit()` will revert permanently, bricking the DDA for that token.

---

### Finding Description

`DirectDepositV1.creditDeposit()` is callable by anyone with no access control:

```solidity
function creditDeposit() external {
    ...
    if (balance != 0) {
        token.approve(address(endpoint), balance);          // allowance set
        endpoint.depositCollateralWithReferral(
            subaccount, productId, uint128(balance), "-1"  // may NOT consume it
        );
    }
}
``` [1](#0-0) 

Inside `Endpoint.depositCollateralWithReferral()`, there is a documented silent early return path:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    return;   // <-- exits WITHOUT calling handleDepositTransfer
}
handleDepositTransfer(...);  // transferFrom only reached if amount is valid
``` [2](#0-1) 

When this early return fires, `handleDepositTransfer` (which calls `transferFrom` and consumes the allowance) is never reached. The allowance set by `token.approve(address(endpoint), balance)` remains non-zero on the token contract.

The `approve()` call in `creditDeposit()` does not reset the allowance to zero before setting a new value: [3](#0-2) 

USDT (and similar tokens) implement a guard that reverts if `approve(spender, nonZeroAmount)` is called while `allowance(owner, spender) > 0`. Once a residual allowance exists, every future call to `creditDeposit()` for that token will revert at the `approve` line, permanently preventing deposits.

---

### Impact Explanation

A DDA (Direct Deposit Address) for a subaccount that holds USDT becomes permanently unable to process USDT deposits. `creditDeposit()` will revert on every future call for that token. The subaccount's USDT collateral path through the DDA is bricked. Funds sent to the DDA in USDT cannot be credited to the subaccount via the normal flow; they are effectively stuck until an admin calls `withdrawFromDirectDepositV1`, which requires owner access and is a manual recovery path.

---

### Likelihood Explanation

The trigger condition is realistic and cheap to induce:

1. A DDA receives a small USDT amount below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT` (e.g., dust sent by the attacker or a user).
2. Any unprivileged caller invokes `creditDeposit()`.
3. `depositCollateralWithReferral` silently returns without consuming the allowance.
4. The residual allowance is now set.
5. All future `creditDeposit()` calls for USDT revert.

The attacker only needs to send a dust USDT amount to the DDA and call the public `creditDeposit()`. No special privileges are required. [1](#0-0) 

---

### Recommendation

In `creditDeposit()`, reset the allowance to zero before setting a new value, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
``` [4](#0-3) 

Alternatively, use OpenZeppelin's `safeIncreaseAllowance` from a version that resets to zero before setting a new value, or use `forceApprove` from OZ v5+.

---

### Proof of Concept

1. Deploy or identify an existing DDA for a subaccount that uses USDT as a collateral token.
2. Send 1 wei of USDT to the DDA (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. Call `DirectDepositV1.creditDeposit()` from any EOA.
4. Observe: `token.approve(endpoint, 1)` executes; `depositCollateralWithReferral` silently returns early at the `isValidDepositAmount` check; allowance of 1 remains on USDT.
5. Send a legitimate USDT deposit (above minimum) to the DDA.
6. Call `creditDeposit()` again.
7. Observe: `token.approve(endpoint, legitimateAmount)` **reverts** because USDT disallows `approve` when allowance is already non-zero.
8. The DDA is permanently bricked for USDT deposits. [5](#0-4) [6](#0-5)
