### Title
`ValidResourceBounds` Protobuf Round-Trip Silently Collapses `AllResources` to `L1Gas` When L2 and L1_DATA Gas Are Zero, Producing a Divergent Transaction Hash During P2P Sync - (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf serializer for `ValidResourceBounds` is not injective: both `ValidResourceBounds::L1Gas(X)` and `ValidResourceBounds::AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` serialize to identical protobuf bytes. On deserialization the decoder always produces `L1Gas`. Because `get_tip_resource_bounds_hash` hashes a **different number of elements** for the two variants (2 vs 3), the transaction hash computed at submission time (always `AllResources`) diverges from the hash recomputed from the deserialized form (`L1Gas`) during p2p sync. Any syncing node that calls `validate_transaction_hash` on such a transaction will reject it, causing valid blocks to be refused and state divergence between the proposer and sync nodes.

---

### Finding Description

**Serialization collapse (root cause)**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both variants to the same wire bytes when L2 and L1_DATA gas are zero: [1](#0-0) 

`L1Gas` emits `l1_data_gas: Some(ResourceBounds::default().into())` (zero), and `AllResources(l2=0, l1_data=0)` emits `l1_data_gas: Some(zero)`. The wire representation is identical.

**Deserialization collapse**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (used in the p2p sync path for historical `Transaction` objects) applies the condition: [2](#0-1) 

When `l1_data_gas` is absent it defaults to zero (the `TODO` comment acknowledges this is a backward-compat workaround). If both `l2_gas` and `l1_data_gas` are zero the result is `L1Gas`, regardless of whether the original transaction was `AllResources`.

**Hash domain divergence**

`get_tip_resource_bounds_hash` hashes a different number of Poseidon elements depending on the variant: [3](#0-2) 

- `AllResources`: hashes `[tip, L1_GAS_packed, L2_GAS_packed, L1_DATA_GAS_packed]` — **4 elements**
- `L1Gas`: hashes `[tip, L1_GAS_packed, L2_GAS_packed]` — **3 elements** (no `L1_DATA_GAS`)

These produce different Poseidon digests. The tip-resource-bounds hash is a direct input to `get_invoke_transaction_v3_hash` and `get_declare_transaction_v3_hash`: [4](#0-3) [5](#0-4) 

**Submission always uses `AllResources`**

Every transaction entering through the gateway is typed as `RpcTransaction`, whose `resource_bounds` field is `AllResourceBounds` (never `L1Gas`): [6](#0-5) 

`convert_rpc_tx_to_internal` computes the canonical hash from `InternalRpcTransactionWithoutTxHash`, which always wraps bounds as `ValidResourceBounds::AllResources`: [7](#0-6) [8](#0-7) 

So the stored hash H₁ is always the 4-element Poseidon hash.

**Sync path recomputes H₂ ≠ H₁**

During p2p sync, `TryFrom<protobuf::TransactionInBlock>` reconstructs a `Transaction::Invoke(InvokeTransaction::V3(...))` or `Transaction::Declare(DeclareTransaction::V3(...))` using `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: [9](#0-8) 

For a transaction with `l2_gas=0, l1_data_gas=0`, this produces `L1Gas`. `validate_transaction_hash` then calls `get_transaction_hash` on this object, producing H₂ (3-element hash). Since H₁ ≠ H₂, validation returns `false`. [10](#0-9) 

**The consensus path is unaffected** because it uses `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` (strict, requires all three fields present): [11](#0-10) 

---

### Impact Explanation

Any syncing node that receives a block containing a V3 transaction with `AllResources(l1_gas>0, l2_gas=0, l1_data_gas=0)` will recompute a hash that does not match the stored hash. `validate_transaction_hash` returns `false`, causing the block to be rejected. The proposer and all sync nodes diverge on chain state. This is a **Critical** wrong-state result from accepted input, or at minimum a **High** admission-layer rejection of valid transactions.

---

### Likelihood Explanation

The gateway's stateless validator explicitly permits transactions where only `l1_gas` is non-zero: [12](#0-11) 

The check `max_possible_fee(Tip::ZERO) == Fee(0)` passes when `l1_gas.max_amount * l1_gas.max_price_per_unit > 0`, regardless of L2 and L1_DATA being zero. Any user can craft and submit such a transaction without special privileges. Once included in a block, every sync node is affected.

---

### Recommendation

The serialization must preserve the `AllResources` vs `L1Gas` distinction. Two options:

1. **Add a discriminant field** to `protobuf::ResourceBounds` (e.g., a boolean `is_all_resources`) so the deserializer can reconstruct the correct variant.
2. **Never collapse to `L1Gas` in the sync deserializer**: remove the `l1_data_gas.is_zero() && l2_gas.is_zero()` branch from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` and always produce `AllResources` when all three fields are present. Reserve `L1Gas` only for legacy pre-0.13.3 transactions that genuinely lack L2/L1_DATA fields in the wire format (i.e., when `l2_gas` is `None`).

---

### Proof of Concept

1. Submit a V3 invoke transaction with `resource_bounds = AllResourceBounds { l1_gas: {max_amount: 1000, max_price_per_unit: 1}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }` through the gateway. The gateway accepts it; the stored hash H₁ is computed via `get_invoke_transaction_v3_hash` with `ValidResourceBounds::AllResources(...)` → 4-element Poseidon hash.

2. The transaction is included in a block and broadcast over p2p.

3. A sync node receives the block. `TryFrom<protobuf::TransactionInBlock>` deserializes the transaction. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas=0, l1_data_gas=0` and returns `ValidResourceBounds::L1Gas(l1_gas)`.

4. `validate_transaction_hash` calls `get_transaction_hash` → `get_invoke_transaction_v3_hash` with `ValidResourceBounds::L1Gas(...)` → 3-element Poseidon hash H₂.

5. H₁ ≠ H₂. `validate_transaction_hash` returns `false`. The sync node rejects the block, diverging from the proposer.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
