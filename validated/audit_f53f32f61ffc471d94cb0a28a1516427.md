### Title
Stale Index Price in `updateStates()` Allows Incorrect Funding Rate Cap — (File: `core/contracts/PerpEngineState.sol`)

---

### Summary

`PerpEngineState.updateStates()` reads `_risk(productId).priceX18` to compute the maximum allowed daily funding payment, but performs no staleness check on that price. If the stored price has not been refreshed recently, the `MAX_DAILY_FUNDING_RATE` cap is computed against an outdated index, causing `cumulativeFundingLongX18` / `cumulativeFundingShortX18` to be updated with either an over-capped or under-capped payment. Because every position's `vQuoteBalance` is derived from these cumulative values, all open perp positions are exposed to corrupted accounting.

---

### Finding Description

`PerpEngineState.updateStates()` is the Nado analog of the Starknet `assets.funding_tick()`. It is invoked by the sequencer via a `PerpTick` transaction processed in `EndpointTx.processTransactionImpl()`:

```
// EndpointTx.sol line 476-485
} else if (txType == IEndpoint.TransactionType.PerpTick) {
    IEndpoint.PerpTick memory txn = abi.decode(transaction[1:], (IEndpoint.PerpTick));
    Times memory t = times;
    uint128 dt = t.perpTime == 0 ? 0 : txn.time - t.perpTime;
    perpEngine.updateStates(dt, txn.avgPriceDiffs);
    t.perpTime = txn.time;
    times = t;
}
```

Inside `updateStates`, the index price is read from storage at line 116 and immediately used to compute the funding cap at line 121:

```solidity
// PerpEngineState.sol lines 116-128
int128 indexPriceX18 = _risk(productId).priceX18;

int128 priceDiffX18 = avgPriceDiffs[i];
int128 maxPriceDiff = MAX_DAILY_FUNDING_RATE.mul(indexPriceX18);

if (priceDiffX18.abs() > maxPriceDiff) {
    priceDiffX18 = (priceDiffX18 > 0) ? maxPriceDiff : -maxPriceDiff;
}
```

`_risk(productId).priceX18` is stored in the shared `RISK_STORAGE` slot and is only updated when an `UpdatePrice` transaction is separately processed:


```solidity

```

`UpdatePrice` and `PerpTick` are entirely independent transaction types with no enforced ordering or co-submission requirement. There is no timestamp, block number, or sequence number attached to `_risk().value[productId].priceX18` to indicate when it was last set. `updateStates` reads it blindly.

The corrupted value flows directly into the cumulative funding accumulators:

```solidity
// PerpEngineState.sol lines 130-132
int128 paymentAmount = priceDiffX18.mul(dtX18).div(ONE_DAY_X18);
state.cumulativeFundingLongX18 += paymentAmount;
state.cumulativeFundingShortX18 += paymentAmount;
```

These accumulators are consumed in `_updateBalance()` every time a position is touched:

```solidity
// PerpEngineState.sol lines 31-36
int128 cumulativeFundingAmountX18 = (balance.amount > 0)
    ? state.cumulativeFundingLongX18
    : state.cumulativeFundingShortX18;
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
```

`deltaQuote` is added to `balance.vQuoteBalance`, which is the component of a position's PnL used in every health check (`getHealthContribution` → `_calculateProductHealth`).

---

### Impact Explanation

**Stale price too high (e.g., price crashed but `_risk.priceX18` not updated):**
`maxPriceDiff` is inflated → `avgPriceDiffs[i]` values that exceed the intended 2% daily cap relative to the real price pass through uncapped → `cumulativeFundingLongX18` / `cumulativeFundingShortX18` accumulate excessive payments → every position's `vQuoteBalance` is corrupted upward or downward depending on direction → positions that should be healthy are marked unhealthy (or vice versa), enabling incorrect liquidations or blocking valid withdrawals.

**Stale price too low (e.g., price surged but `_risk.priceX18` not updated):**
`maxPriceDiff` is deflated → valid funding payments are over-capped → the funding mechanism fails to balance long/short open interest, breaking the economic invariant the protocol relies on.

The corrupted `vQuoteBalance` is a direct asset-level impact: it is the realized PnL component of every perp position and is included in `getHealth()` calculations that gate withdrawals, liquidations, and trade health checks.

---

### Likelihood Explanation

`UpdatePrice` and `PerpTick` are separate sequencer-submitted transaction types. During periods of high volatility, network congestion, or any operational gap between price oracle updates and perp tick submissions, the stored price will lag the real market price. The longer the gap, the larger the potential miscalculation. This is a passive, always-present risk that does not require any malicious actor — it is a structural absence of a staleness guard in the protocol.

---

### Recommendation

Before using `_risk(productId).priceX18` in `updateStates`, validate that the price was updated within an acceptable recency window. One approach is to store a `lastPriceUpdateTime` alongside `priceX18` in `RiskStore` and require `block.timestamp - lastPriceUpdateTime <= MAX_PRICE_AGE` inside `updateStates`. Alternatively, require that `UpdatePrice` for all active perp products is submitted atomically in the same sequencer batch as `PerpTick`.

---

### Proof of Concept

1. At time T0, `UpdatePrice` sets `_risk(productId).priceX18 = 2000e18` for a perp product.
2. The market price drops to `500e18`. No `UpdatePrice` transaction is submitted for an extended period.
3. At time T1, the sequencer submits a `PerpTick` with `avgPriceDiffs[i] = 15e18` (which is 3% of the real price `500e18`, exceeding the intended `MAX_DAILY_FUNDING_RATE = 2%`).
4. `updateStates` computes `maxPriceDiff = 0.02 * 2000e18 = 40e18` using the stale price.
5. Since `15e18 < 40e18`, the check at line 123 does not cap the diff.
6. `paymentAmount = 15e18 * dt / ONE_DAY` is added to both `cumulativeFundingLongX18` and `cumulativeFundingShortX18`.
7. The actual intended cap at the real price would be `0.02 * 500e18 = 10e18`, so the payment is 50% larger than the protocol's `MAX_DAILY_FUNDING_RATE` invariant permits.
8. Every position's `vQuoteBalance` is updated with this inflated funding payment on the next `_updateBalance` call, corrupting realized PnL and health calculations across all open perp positions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
