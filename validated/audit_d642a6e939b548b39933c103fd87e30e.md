### Title
Dead `DISABLE_OP` Size-Limit Check in `op_div` / `op_mod` / `op_divmod` Due to Swapped Thresholds — (`src/more_ops.rs`)

---

### Summary

In `op_div`, `op_mod`, and `op_divmod`, the mempool-mode (`DISABLE_OP`) size check uses a threshold of **2048 bytes** for `a0_len`, while the unconditional check uses **256 bytes**. Because 256 < 2048, the `DISABLE_OP` check is permanently dead code: the unconditional check always fires first. The thresholds appear to be swapped relative to their intended roles, mirroring the flipped-comparison class of the reference report.

---

### Finding Description

In `src/more_ops.rs`, `op_div` contains:

```rust
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 2048 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
if a0_len > 256 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

`DISABLE_OP` is part of `MEMPOOL_MODE` and is intended to impose **stricter** limits in mempool mode than in consensus mode. For `op_modpow` (opcode 60), `DISABLE_OP` completely disables the operator. For `op_div`, `op_mod`, and `op_divmod`, the intent is to add a tighter size restriction in mempool mode.

However, the `DISABLE_OP` guard uses threshold **2048**, which is 8× larger than the unconditional threshold of **256**. The unconditional check therefore always fires before the `DISABLE_OP` check can ever be reached. The `DISABLE_OP` check is dead code.

The correct ordering should be:

```rust
// consensus-mode limit (permissive)
if a0_len > 2048 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
// mempool-mode limit (strict)
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

The same pattern appears identically in `op_mod` and `op_divmod` (and their `_malachite` variants).

---

### Impact Explanation

**Consensus divergence / valid-program rejection.** If the reference Python CLVM implementation permits `a0_len` up to 2048 bytes for `/`, `%`, and `divmod` in consensus mode, then the Rust implementation incorrectly rejects any program whose dividend exceeds 256 bytes. A CLVM coin spending a program that uses `/` with a 257-byte dividend would be accepted by Python nodes and rejected by Rust nodes, causing a chain split. Conversely, the mempool restriction that `DISABLE_OP` was meant to enforce is silently absent (the unconditional 256-byte cap happens to cover it, but only by accident).

---

### Likelihood Explanation

Any attacker-controlled CLVM byte sequence that encodes a `/`, `%`, or `divmod` call with a dividend atom longer than 256 bytes exercises this path. The entry point is fully attacker-reachable: CLVM bytes are parsed and executed by `run_program`, which dispatches opcode 19 (`op_div`), 20 (`op_divmod`), and 61 (`op_mod`) through `ChiaDialect::op`. No special flags or privileged access are required; the bug fires in the default consensus dialect.

---

### Recommendation

Swap the thresholds so the unconditional check uses the permissive consensus limit and the `DISABLE_OP` check uses the strict mempool limit:

```rust
// consensus-mode hard cap
if a0_len > 2048 || a1_len > 1024 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
// mempool-mode additional restriction
if flags.contains(ClvmFlags::DISABLE_OP) && a0_len > 256 {
    return Err(EvalErr::InvalidOpArg(input, "div".to_string()));
}
```

Apply the same fix to `op_mod`, `op_divmod`, `op_div_malachite`, and `op_mod_malachite`.

---

### Proof of Concept

Craft a CLVM program that divides a 257-byte atom by 1:

```
(/ <257-byte-atom> (q . 1))
```

Run it under `ChiaDialect::new(ClvmFlags::empty())` (consensus mode). The Rust VM returns `EvalErr::InvalidOpArg` because `a0_len = 257 > 256`.
