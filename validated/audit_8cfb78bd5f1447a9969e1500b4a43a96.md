### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed`, Making `overall_fee` Inconsistent with Its Described Formula — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_estimateMessageFee` exposes `l2_gas_price` but never populates or exposes `l2_gas_consumed`. The `overall_fee` field is computed from the full `GasVector` (L1 gas + L1 data gas + **L2 gas**), yet both the struct's inline documentation and the OpenAPI spec describe `overall_fee` as equaling only `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. For any V3 transaction with non-zero L2 gas consumption, the returned `overall_fee` is strictly greater than what the described formula produces from the other returned fields, making the response internally inconsistent and authoritative-looking wrong.

### Finding Description

`tx_execution_output_to_fee_estimation` in `crates/apollo_rpc_execution/src/objects.rs` builds the `FeeEstimation` response:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),      // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                  // price present …
    // l2_gas_consumed is NEVER set — field does not exist in the struct
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas cost
    unit: tx_execution_output.price_unit,
})
``` [1](#0-0) 

`receipt.fee` is produced by `GasVector::cost()`, which sums all three resources:

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,           Resource::L2Gas),
] { … }
``` [2](#0-1) 

The `FeeEstimation` struct's own documentation and the OpenAPI `FEE_ESTIMATE` schema both state:

> `overall_fee` — "equals to `gas_consumed*gas_price + data_gas_consumed*data_gas_price`" [3](#0-2) [4](#0-3) 

The L2 gas term (`l2_gas_consumed × l2_gas_price`) is silently included in `overall_fee` but has no corresponding `l2_gas_consumed` field in the response. A developer TODO acknowledges this gap:

> `// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` [5](#0-4) 

### Impact Explanation

**High — RPC fee estimation returns an authoritative-looking wrong value.**

Any caller of `starknet_estimateFee` or `starknet_estimateMessageFee` for a V3 transaction with non-zero L2 gas consumption receives a response where:

```
overall_fee  ≠  gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price
```

The gap equals `l2_gas_consumed × l2_gas_price`, which is not exposed. Concrete consequences:

1. **Callers who reconstruct the fee from the returned components** (as the spec says they can) compute a value lower than `overall_fee`. They will set `max_fee` or resource bounds too low, causing their transactions to be rejected at execution time.
2. **Callers who want to set `l2_gas.max_amount`** in their V3 resource bounds have no basis for doing so from the estimation response — the only guidance is the opaque `overall_fee` lump sum.
3. **The `l2_gas_price` field is present but useless** without `l2_gas_consumed`, making the response misleading.

### Likelihood Explanation

**High — always triggered for V3 transactions.**

All V3 transactions on Starknet ≥ 0.13.x use Sierra gas (L2 gas). The `l2_gas` component of `GasVector` is non-zero for any non-trivial transaction. The `starknet_estimateFee` endpoint is the primary tool users and wallets rely on to set resource bounds before submitting transactions.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`. Update the `overall_fee` description in both the struct and the OpenAPI schema to reflect the three-term formula:

```
overall_fee = gas_consumed × l1_gas_price
            + data_gas_consumed × l1_data_gas_price
            + l2_gas_consumed × l2_gas_price
```

Also update the `FEE_ESTIMATE` schema in `starknet_api_openrpc.json` to include `l2_gas_consumed` as a required field.

### Proof of Concept

Given a V3 invoke transaction on a block where `l2_gas_price = 8_000_000_000` fri:

1. Call `starknet_estimateFee` → response:
   ```json
   {
     "gas_consumed":      "0x0",
     "l1_gas_price":      "0xe8d4a51000",
     "data_gas_consumed": "0x80",
     "l1_data_gas_price": "0x3e8",
     "l2_gas_price":      "0x1dcd65000",
     "overall_fee":       "0x151eb86f3ed400",
     "unit": "FRI"
   }
   ```
   (Actual values from `test_execute_with_prefetch.json` fixture, where `l2_gas_consumed = 0xb56b6`.)

2. Reconstruct fee from returned fields per the spec formula:
   ```
   0x0 × 0xe8d4a51000 + 0x80 × 0x3e8 = 0x1f400
   ```

3. Actual `overall_fee` = `0x151eb86f3ed400` — **orders of magnitude larger** than the reconstructed value.

4. The missing term is `0xb56b6 × 0x1dcd65000 = 0x151eb86f1ff000`, which is `l2_gas_consumed × l2_gas_price` — a field that is never returned. [6](#0-5)

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
