### Title
Cost Multiplication Overflow in `op_unknown` Enables Cost Undercharge — (`src/more_ops.rs`)

---

### Summary

In `op_unknown`, the final cost multiplication `cost *= cost_multiplier + 1` is an unchecked `u64` multiplication. In Rust release builds (where integer overflow wraps silently), this can produce a wrapped value that is ≤ `u32::MAX`, bypassing the only post-multiplication guard and causing the function to return a fraudulently small cost for an attacker-crafted unknown opcode.

---

### Finding Description

`op_unknown` computes the cost of an unknown opcode in two stages:

1. **Base cost computation** (bounded by `max_cost` via `check_cost`): [1](#0-0) 

2. **Multiplication by the opcode-encoded multiplier**: [2](#0-1) 

```rust
check_cost(cost, max_cost)?;
cost *= cost_multiplier + 1;          // ← unchecked u64 multiplication
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is a `u64` derived from up to 4 opcode bytes, so its maximum value is `u32::MAX = 4_294_967_295`. `cost_multiplier + 1` is therefore at most `4_294_967_296 ≈ 2^32`. The base `cost` is bounded by `max_cost`, which in Chia's blockchain can be up to 11 billion (`11_000_000_000`).

**Overflow condition:**
```
cost × (cost_multiplier + 1) > u64::MAX (≈ 1.844 × 10^19)
```

With `cost = 8_589_934_592` (= 2^33) and `cost_multiplier = u32::MAX`:
```
2^33 × 2^32 = 2^65 ≡ 0 (mod 2^64)
```

In Rust release mode, the multiplication wraps to `0`. The guard `0 > u32::MAX` is `false`, so the function returns `Ok(Reduction(0, allocator.nil()))` — **zero cost** for an operation that should cost billions.

The `cost_multiplier` is extracted from the opcode bytes via `u32_from_u8`: [3](#0-2) 

`Cost` is defined as `u64`: [4](#0-3) 

The `check_cost` guard before the multiplication only ensures `cost ≤ max_cost`; it does not prevent the post-multiplication overflow: [5](#0-4) 

---

### Impact Explanation

`op_unknown` is called in lenient/consensus mode when an unknown opcode is encountered. The cost it returns is added directly to the running program cost in `run_program`: [6](#0-5) 

If `op_unknown` returns `0` (or a tiny wrapped value), the running cost does not increase. An attacker can craft a CLVM program containing many such unknown opcodes, each of which should consume billions of cost units but actually consumes zero. This allows:

1. **Cost limit bypass**: The attacker fills a block with computationally expensive unknown ops at negligible declared cost, violating the blockchain's resource accounting invariant.
2. **Consensus divergence**: Rust debug builds panic on overflow; release builds wrap silently. A block accepted by a release-mode node would be rejected (with a panic/error) by a debug-mode node, splitting consensus.

---

### Likelihood Explanation

The attacker controls the opcode bytes directly (they are part of the CLVM program bytes). To trigger the overflow:

- Set `cost_multiplier = u32::MAX` via opcode bytes `[0xfe, 0xff, 0xff, 0xff, 0x40]` (avoids the `0xff 0xff` reserved prefix; cost_function = 1).
- Provide enough atom arguments to push the base cost above `u64::MAX / (cost_multiplier + 1) ≈ 4.3 billion`.
- This requires `max_cost ≥ 4.3 billion`, which is within Chia's 11-billion block limit.

The attacker needs no special privileges — only the ability to submit a CLVM program, which is the normal entry point for all Chia coin spends.

---

### Recommendation

Replace the unchecked multiplication with a checked variant and return an error on overflow:

```rust
// Before:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
}

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .filter(|&c| c <= u32::MAX as u64)
    .ok_or(EvalErr::Invalid(o))?;
```

---

### Proof of Concept

**Opcode bytes**: `[0xfe, 0xff, 0xff, 0xff, 0x40]`
- `op[0..4] = [0xfe, 0xff, 0xff, 0xff]` → `cost_multiplier = 0xfeffffff = 4_278_190_079`
- `op[4] = 0x40` → `cost_function = 1` (add-like)
- Does not trigger the `0xff 0xff` reserved-prefix guard.

**Arguments**: Provide `n = 26_843_546` nil atoms as arguments.
- Base cost: `99 + 26_843_546 × 320 = 8_589_934_819`
- `check_cost(8_589_934_819, max_cost)` passes if `max_cost ≥ 8_589_934_819` (within 11B limit).

**Multiplication** (release mode):
```
8_589_934_819 × 4_278_190_080
= 36_757_...  (overflows u64)
≡ small_value (mod 2^64)
```

The wrapped value passes `cost > u32::MAX` check, and `op_unknown` returns a near-zero cost. The running program cost in `run_program` does not meaningfully increase, allowing the attacker to repeat this pattern across many opcodes within a single block.
