Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` ŌĆö the immediate caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, the router is that immediate caller, so the allowlist checks the router address rather than the actual end-user. If the router is allowlisted (the natural configuration to permit router-mediated swaps), every unprivileged caller can bypass the restriction by routing through the public router contract.

## Finding Description
`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the caller of the extension) and `sender` is the value passed by the pool from its own `msg.sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ŌåÉ becomes `sender` in the extension
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        ...
        params.extensionData
    );
```

The pool therefore sees `msg.sender = router`, passes `sender = router` to the extension, and the check becomes `allowedSwapper[pool][router]`. The actual end-user is never checked. The same path exists for `exactInput` (L104), `exactOutputSingle` (L136), and `exactOutput` (L165). In the multi-hop `exactOutput` path, intermediate hops are called from within `_exactOutputIterateCallback` (L220-228), where `msg.sender` is the prior pool ŌĆö also not the end-user.

No existing guard in the extension or pool recovers the original end-user identity; `extensionData` is passed through verbatim from the router caller without encoding the real `msg.sender`.

## Impact Explanation
The sole purpose of `SwapAllowlistExtension` is to restrict which addresses may trade on a pool. If the router is allowlisted, that invariant is completely broken: any unprivileged address can execute swaps on the restricted pool by calling the public router. This constitutes broken core pool functionality ŌĆö the allowlist gate ŌĆö causing unauthorized swap access. Severity: Medium.

## Likelihood Explanation
The bypass requires the pool admin to allowlist the router address. This is a realistic and natural configuration: an operator who wants to permit router-mediated swaps for approved users will allowlist the router, not realising that doing so grants access to every caller of the public router. The router is a well-known, publicly deployed contract requiring no special privilege to invoke. The condition is therefore likely to occur in production deployments.

## Recommendation
The extension must gate the economic actor (the end-user), not the intermediary. Viable approaches:

1. **Encode the real caller in `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`; the extension decodes and checks it. This requires a trusted-router convention and the extension must verify `msg.sender` (the pool) is using a known router.
2. **Conditional check**: If `sender` is a known router, decode the actual user from `extensionData`; otherwise check `sender` directly.
3. **Document the limitation explicitly**: If router-mediated allowlisting is intentionally unsupported, NatDoc and admin tooling must make clear that allowlisting the router grants access to all router callers, so operators do not do so expecting per-user gating.

## Proof of Concept
1. Admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin calls `setAllowedToSwap(pool, routerAddress, true)` ŌĆö intending to permit router-mediated swaps for approved users.
3. Non-allowlisted user `C` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(recipient, ...)` ŌĆö pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(sender = router, ...)` at `MetricOmmPool.sol` L230-231.
6. `SwapAllowlistExtension.beforeSwap(sender = router, ...)` evaluates `allowedSwapper[pool][router]` ŌåÆ `true`.
7. Swap executes successfully. User `C` has bypassed the allowlist without being individually approved.

**Foundry test sketch:**
```solidity

    // Admin allowlists router (natural config)
    vm.prank(poolAdmin);
