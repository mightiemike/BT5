### Title
Missing `ENABLE_SECP_OPS` Flag Guard on 4-Byte Secp Opcode Dispatch Path Allows Flag Bypass — (`File: src/chia_dialect.rs`)

### Summary
The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp operators via two separate code paths: a 1-byte opcode path (opcodes 64 and 65) that correctly checks `ClvmFlags::ENABLE_SECP_OPS` before dispatching, and a 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) that dispatches the identical `op_secp256k1_verify` / `op_secp256r1_verify` functions **without any flag check**. An attacker-controlled CLVM program can invoke secp signature verification unconditionally by using the 4-byte opcode encoding, bypassing the hard-fork activation guard entirely.

### Finding Description

In `src/chia_dialect.rs`, the `op()` method has two dispatch branches for secp operators.

**1-byte path (guarded):** [1](#0-0) 

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

**4-byte path (unguarded):** [2](#0-1) 

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => { return unknown_operator(...); }
};
return f(allocator, argument_list, max_cost, flags);
```

Both paths call the exact same operator functions (`op_secp256k1_verify`, `op_secp256r1_verify` in `src/secp_ops.rs`). [3](#0-2) [4](#0-3) 

The flag `ENABLE_SECP_OPS` is defined as the hard-fork activation guard for these operators: [5](#0-4) 

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

The 4-byte path is entered whenever the operator atom has length 4, which is fully attacker-controlled via the serialized CLVM program bytes. The 4-byte opcodes `0x13d61f00` (secp256k1) and `0x1c3a8f00` (secp256r1) are the "assigned-cost unknown operator" encodings of the same secp functions. Because the `op_len == 4` branch is checked **before** the `op_len == 1` branch, any CLVM program that encodes the operator as a 4-byte atom with those values reaches the unguarded dispatch unconditionally. [6](#0-5) 

This is structurally identical to the reported external vulnerability: a guard check (`if address(externalAssetVault) != address(0)`) exists in one function path but is absent in a parallel path that performs the same external call.

### Impact Explanation

**Impact: Medium**

The `ENABLE_SECP_OPS` flag is the designated hard-fork activation gate for secp signature verification. Without it set, secp ops should be unavailable. Because the 4-byte path bypasses this gate:

1. **Hard-fork guard is nullified**: Any CLVM program can invoke secp256k1 or secp256r1 signature verification regardless of whether `ENABLE_SECP_OPS` is set, defeating the purpose of the flag as a consensus-controlled activation mechanism.
2. **Consensus divergence risk**: Nodes or tooling that rely on `ENABLE_SECP_OPS` being unset to reject secp programs (e.g., pre-fork mempool validation, custom dialects, or integrators calling `ChiaDialect::new(flags)` without `ENABLE_SECP_OPS`) will silently accept secp programs submitted via 4-byte opcodes. This creates a split in accepted program sets between flag-aware and flag-unaware execution contexts.
3. **Signature verification bypass is not possible** (the secp functions themselves are correct), but the **activation invariant** — that secp ops are unavailable until the flag is set — is broken.

### Likelihood Explanation

**Likelihood: Medium**

The attacker entry point is any externally submitted CLVM program (puzzle or solution bytes). Encoding an operator as a 4-byte atom is trivially achievable by any CLVM serializer. The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source comments and derivable from the cost formula. No privileged access is required. The only constraint is that the attacker must know to use the 4-byte encoding rather than the 1-byte encoding — a low bar for any researcher or adversarial integrator examining the dialect source.

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte opcode dispatch path, mirroring the guard already present on the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both encoding paths are subject to the same activation guard, restoring the intended invariant.

### Proof of Concept

1. Construct a `ChiaDialect` with `ENABLE_SECP_OPS` **not** set:
   ```rust
   let dialect = ChiaDialect::new(ClvmFlags::empty());
   ```
2. Serialize a CLVM program whose operator atom is the 4-byte value `0x13d61f00` (secp256k1_verify) with valid pubkey/msg/sig arguments.
3. Call `run_program` with this program and the dialect above.
4. **Expected (correct) behavior**: The program should be rejected because `ENABLE_SECP_OPS` is not set.
5. **Actual behavior**: The program executes `op_secp256k1_verify` successfully, bypassing the flag guard entirely, because the `op_len == 4` branch at line 157 of `src/chia_dialect.rs` dispatches directly to the secp function without checking `ENABLE_SECP_OPS`. [7](#0-6) [8](#0-7)
