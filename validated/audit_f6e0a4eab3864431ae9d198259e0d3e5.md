### Title
`ENABLE_SECP_OPS` Flag Check Bypassed via 4-Byte Opcode Aliases for secp256k1/r1 Verify — (File: `src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through two separate code paths. The 1-byte opcode path (opcodes 64 and 65) is correctly gated behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (`0x13d61f00` and `0x1c3a8f00`) dispatches the identical functions with **no flag check at all**. An attacker-controlled CLVM program can use the 4-byte opcode forms to invoke secp signature verification even when `ENABLE_SECP_OPS` is not set — including in `MEMPOOL_MODE`.

---

### Finding Description

In `src/chia_dialect.rs`, the `op()` method handles two opcode length branches:

**Branch 1 — 4-byte opcodes (lines 175–182):** Secp opcodes are matched and dispatched unconditionally:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No flag check precedes this dispatch. [1](#0-0) 

**Branch 2 — 1-byte opcodes (lines 248–249):** The same functions are gated behind `ENABLE_SECP_OPS`:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `MEMPOOL_MODE` constant does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [3](#0-2) 

So in mempool mode:
- Opcode `0x40` (64) → falls to `unknown_operator` → `Err(Unimplemented)` because `NO_UNKNOWN_OPS` is set.
- Opcode `0x1c3a8f00` → **executes `op_secp256r1_verify` at full cost (1,850,000) with no flag check**.

The two opcode forms call the exact same Rust functions (`op_secp256k1_verify`, `op_secp256r1_verify` in `src/secp_ops.rs`) with identical semantics. [4](#0-3) 

---

### Impact Explanation

**Consensus / mempool divergence.** A node running in mempool mode rejects 1-byte secp opcodes (64, 65) but silently accepts and executes the 4-byte forms. Any caller that sets `ENABLE_SECP_OPS` to control whether secp verification is available is bypassed by using the 4-byte encoding. This creates an inconsistency in which programs are accepted or rejected depending on opcode encoding, not on the actual operation being performed. The corrupted result is the acceptance/rejection decision for a CLVM program: two programs that perform identical secp verification — one using opcode `0x40`, one using `0x1c3a8f00` — receive different treatment under the same dialect flags.

**Impact: Medium** — affects mempool validation consistency and flag-controlled feature gating.

---

### Likelihood Explanation

The 4-byte opcode forms are documented in the codebase (comments at lines 172–174 explain their cost encoding) and are referenced in fuzzing and benchmarking tooling. [5](#0-4)  An attacker who knows the opcode encoding can trivially craft a CLVM program using `0x13d61f00` or `0x1c3a8f00` as the operator atom. The entry path is direct: attacker-controlled CLVM bytes → `run_program` → `apply_op` → `dialect.op()` → 4-byte branch → secp execution without flag check.

**Likelihood: Medium** — requires knowledge of the 4-byte opcode encoding, which is publicly documented in the Chia protocol.

---

### Recommendation

Add the same `ENABLE_SECP_OPS` flag guard to the 4-byte opcode dispatch branch. In `src/chia_dialect.rs`, the match at lines 175–182 should be updated to:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures both opcode encodings for secp verification are subject to the same flag restriction, eliminating the inconsistency between the two dispatch paths.

### Proof of Concept

Construct a CLVM program using the 4-byte secp opcode form and run it under `MEMPOOL_MODE` (which excludes `ENABLE_SECP_OPS`):

```rust
use clvmr::chia_dialect::{ChiaDialect, MEMPOOL_MODE};
use clvmr::run_program::run_program;
use clvmr::allocator::Allocator;

let mut allocator = Allocator::new();
let dialect = ChiaDialect::new(MEMPOOL_MODE);

// Operator atom: 0x1c3a8f00 (secp256r1_verify, 4-byte form)
let op = allocator.new_atom(&[0x1c, 0x3a, 0x8f, 0x00]).unwrap();
// ... build valid pubkey/msg/sig args ...
// run_program executes op_secp256r1_verify despite ENABLE_SECP_OPS not being set.
// Contrast: using opcode 0x41 (65) under the same flags returns Err(Unimplemented).
```

The 1-byte opcode `0x41` is rejected by `unknown_operator` (because `NO_UNKNOWN_OPS` is in `MEMPOOL_MODE`), while `0x1c3a8f00` executes the identical secp verification function successfully — demonstrating the inconsistent restriction. [6](#0-5)

```rust
pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
