### Title
Linked Signer Can Unilaterally Replace Itself via `LinkSigner` Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` transaction handler in `EndpointTx.sol` invokes `validateSignedTx` with `allowLinkedSigner = true`, permitting the **current linked signer** to sign a `LinkSigner` transaction that replaces itself with any attacker-controlled address. Once replaced, the new address can authorize withdrawals, quote transfers, and other sensitive operations, fully draining the victim's subaccount.

---

### Finding Description

In `EndpointTx.sol`, the sequencer-path handler for `TransactionType.LinkSigner` is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` delegates to `validateCompactSignature`, which passes `getLinkedSigner(sender)` as the permitted signer to `Verifier.validateCompactSignature`:

```solidity
function validateCompactSignature(...) public pure {
    address recovered = ECDSA.recover(digest, signature.r, signature.vs);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
``` [2](#0-1) 

This means the **existing linked signer** for a subaccount is a valid signer for a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with any new address. The linked signer is a delegated hot-wallet key, not the subaccount owner. Allowing it to re-delegate authority to an arbitrary address is an authority escalation with no owner consent required.

The analog to the Jet v2 report is direct: just as the Jet `Configure` instruction could be invoked without a proper `ControlAuthority` check — allowing an attacker to change privileged settings — here the `LinkSigner` instruction can be invoked by the current linked signer to change the privileged signer, without requiring the subaccount owner's key.

---

### Impact Explanation

Once a linked signer replaces itself:

1. The new attacker-controlled address becomes the valid signer for the victim's subaccount.
2. The attacker can sign `WithdrawCollateral` transactions (sequencer path, `allowLinkedSigner = true`) to drain all spot collateral. [3](#0-2) 

3. The attacker can sign `TransferQuote` transactions to move quote balances to an attacker-controlled subaccount. [4](#0-3) 

4. The original subaccount owner cannot recover control until they submit a slow-mode `LinkSigner` transaction — but by then funds may already be gone.

**Asset delta**: Full loss of all collateral and quote balances in the victim's subaccount.

---

### Likelihood Explanation

Linked signers are commonly hot wallets, trading bots, or third-party API services. Any of the following realistic scenarios triggers the exploit:

- A trading bot's private key is leaked or compromised.
- A third-party service the user delegated to turns malicious.
- A user grants linked signer access to a smart contract that is later exploited.

The attack requires no admin access, no sequencer compromise, and no governance capture — only control of the currently registered linked signer key.

---

### Recommendation

`LinkSigner` is an authority-changing operation and must only be signable by the subaccount owner (the primary key embedded in the `sender` bytes32). Change `allowLinkedSigner` to `false` for this transaction type:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the Jet v2 remediation: use a `ControlAuthority`-equivalent check to ensure that only the canonical owner can invoke the authority-changing instruction.

---

### Proof of Concept

1. Alice has `alice_subaccount` with `linkedSigners[alice_subaccount] = bot_address`.
2. Attacker controls or compromises `bot_address`.
3. Attacker constructs a `SignedLinkSigner` with `tx.sender = alice_subaccount`, `tx.signer = attacker_address`, signed by `bot_address`.
4. Sequencer calls `processTransactionImpl`, reaches the `LinkSigner` branch.
5. `validateSignedTx(..., true)` → `getLinkedSigner(alice_subaccount)` returns `bot_address` → `ECDSA.recover` returns `bot_address` → passes. [5](#0-4) 

6. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`.
7. Attacker signs a `WithdrawCollateral` transaction using `attacker_address`; sequencer processes it; Alice's collateral is sent to the attacker.
