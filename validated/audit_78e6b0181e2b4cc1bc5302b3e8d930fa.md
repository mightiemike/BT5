Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of actual swapper, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is documented as gating swaps "by swapper address," but `beforeSwap` receives `sender` = `msg.sender` of the pool call, which is the router when `MetricOmmSimpleRouter` intermediates. The check `allowedSwapper[pool][sender]` therefore validates the router's address, not the end-user's. If the router is allowlisted for a pool, any unprivileged user can bypass per-user swap restrictions by routing through it.

## Finding Description
**Call chain:**

1. `MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

2. `ExtensionCalling._beforeSwap` encodes that `sender` into the extension call: [2](#0-1) 

3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][direct_pool_caller]`: [3](#0-2) 

4. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router is the direct pool caller — `msg.sender` inside `pool.swap` is the router, not the end-user: [4](#0-3) 

**Exploit path:** Pool admin configures `SwapAllowlistExtension` to restrict swaps to a set of approved addresses and allowlists the router (a natural operational choice to permit router-mediated swaps). Any unprivileged user then calls `MetricOmmSimpleRouter.exactInputSingle` targeting that pool. The extension receives `sender = router`, finds it allowlisted, and passes — the actual end-user's address is never checked.

**Contrast with correct pattern:** `DepositAllowlistExtension.beforeAddLiquidity` uses the `owner` parameter (the actual position owner), not `sender` (the caller), so router intermediation cannot bypass it: [5](#0-4) 

The existing `allowAllSwappers` escape hatch is irrelevant — when it is `true` there is no restriction at all; the bug manifests precisely in the restricted case.

## Impact Explanation
The `SwapAllowlistExtension` access control is fully defeated for any pool that allowlists the router. Pools intended to be permissioned (e.g., KYC-gated, institutional-only, or compliance-restricted pools) become open to any caller via the router. This breaks a core pool access-control invariant and constitutes broken core pool functionality. Severity: **Medium** — no direct token loss, but the intended restriction is completely bypassed by an unprivileged actor using a standard, in-scope periphery contract.

## Likelihood Explanation
The router is a standard, deployed periphery contract. A pool admin who wants to allow router-mediated swaps while still restricting individual swappers will naturally allowlist the router address. Once the router is allowlisted, any user can exploit this with a single `exactInputSingle` call — no special privileges, no malicious setup, fully repeatable.

## Recommendation
Pass the actual end-user through the call chain rather than the direct pool caller. Two options:

1. **Preferred:** Add a `swapper` field to `extensionData` that the router populates with `msg.sender` before calling `pool.swap`, and have `SwapAllowlistExtension` decode and check that field.
2. **Alternative:** Mirror `DepositAllowlistExtension`'s pattern — introduce a dedicated `swapper` parameter (analogous to `owner`) that the pool populates from a trusted source (e.g., the callback context or a separate argument), so the extension can check the true initiator.

## Proof of Concept
```solidity
// 1. Deploy pool with SwapAllowlistExtension configured
// 2. Pool admin allowlists the router:
//    extension.setAllowedToSwap(pool, address(router), true);
//    (does NOT allowlist attacker EOA)
// 3. Attacker calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    ...
}));
// Extension receives sender = address(router), which is allowlisted → passes.
// Attacker swaps successfully despite not being individually allowlisted.
```
A Foundry integration test can confirm by asserting the swap succeeds for a non-allowlisted EOA when the router is allowlisted, and fails when the router is not allowlisted.
