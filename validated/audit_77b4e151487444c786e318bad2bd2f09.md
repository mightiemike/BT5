### Title
Unprotected `initialize()` Functions Allow Front-Running of Proxy Initialization — (`File: core/contracts/Clearinghouse.sol`, `core/contracts/Endpoint.sol`, `core/contracts/OffchainExchange.sol`, `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`)

---

### Summary

Multiple core Nado contracts deployed as `TransparentUpgradeableProxy` instances expose `initialize()` functions with no caller access control and no `_disableInitializers()` guard in their implementation constructors. An attacker monitoring the mempool can front-run the deployer's initialization call and inject malicious constructor parameters — most critically a malicious `_withdrawPool` in `Clearinghouse`, a malicious `_sequencer` in `Endpoint`, or a malicious `_clearinghouse` in `OffchainExchange` — before the legitimate deployment completes.

---

### Finding Description

The following contracts are deployed as upgradeable proxies (tracked by `ProxyManager`) and have `initialize()` functions that accept privileged configuration parameters but impose no restriction on who may call them:

**`Clearinghouse.initialize()`** — accepts `_endpoint`, `_quote`, `_clearinghouseLiq`, `_spreads`, and `_withdrawPool`. The `_withdrawPool` address is the destination for every user withdrawal via `handleWithdrawTransfer()`. There is no constructor with `_disableInitializers()` in `Clearinghouse` or its parent `EndpointGated`. [1](#0-0) 

**`Endpoint.initialize()`** — accepts `_sequencer`, `_clearinghouse`, `_offchainExchange`, `_verifier`, and `_endpointTx`. The `sequencer` field gates all calls to `submitTransactionsChecked`. There is no constructor with `_disableInitializers()` in `Endpoint` or its parent `EndpointStorage`. [2](#0-1) 

**`OffchainExchange.initialize()`** — accepts `_clearinghouse` and `_endpoint`. These fields control which contracts can apply balance deltas and which engine addresses are trusted. No constructor guard exists. [3](#0-2) 

**`SpotEngine.initialize()` / `PerpEngine.initialize()`** — accept `_clearinghouse`, `_offchainExchange`, `_endpoint`, and `_admin`. The `canApplyDeltas` whitelist is populated from these parameters; a malicious `_clearinghouse` or `_offchainExchange` would be granted unrestricted balance-update authority. [4](#0-3) [5](#0-4) [6](#0-5) 

By contrast, `ContractOwner.initialize()` correctly enforces `require(_deployer == msg.sender, ...)`, and `Verifier`, `Airdrop`, and `BaseWithdrawPool` all call `_disableInitializers()` in their constructors — demonstrating that the pattern is known and selectively applied. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Loss of funds (critical path):** An attacker who front-runs `Clearinghouse.initialize()` and supplies their own address as `_withdrawPool` causes every subsequent call to `handleWithdrawTransfer()` — invoked on every `withdrawCollateral`, `withdrawInsurance`, and `rebalanceXWithdraw` — to transfer tokens to the attacker rather than the user. [9](#0-8) 

**Protocol takeover:** An attacker who front-runs `Endpoint.initialize()` and supplies their own address as `_sequencer` becomes the sole entity authorized to call `submitTransactionsChecked`, gaining full control over all on-chain state transitions. [10](#0-9) 

**Accounting corruption / deployment failure:** Front-running `SpotEngine.initialize()` or `PerpEngine.initialize()` with a malicious `_clearinghouse` grants that address unrestricted `canApplyDeltas` access, allowing arbitrary balance manipulation. Alternatively, the attacker's call consumes the `initializer` slot, causing the legitimate `Clearinghouse.addEngine()` call to revert and requiring a full redeploy. [11](#0-10) 

---

### Likelihood Explanation

The attack requires only mempool observation and a higher-gas transaction submitted between proxy deployment and the `initialize()` call. No special privileges, leaked keys, or social engineering are needed. The deployment sequence — deploy proxy, then call `initialize()` in a separate transaction — is the standard Hardhat/script pattern used throughout this codebase, and the window is reliably exploitable on any public network. The absence of `_disableInitializers()` in `Clearinghouse`, `Endpoint`, and `OffchainExchange` also means the bare implementation contracts are themselves initializable by anyone, compounding the risk.

---

### Recommendation

1. Add `/// @custom:oz-upgrades-unsafe-allow constructor` and call `_disableInitializers()` in the constructors of `Clearinghouse`, `Endpoint`, and `OffchainExchange` (matching the pattern already used in `Verifier`, `Airdrop`, and `BaseWithdrawPool`). [12](#0-11) 

2. Add a caller check to each `initialize()` function — e.g., `require(msg.sender == expectedDeployer)` — mirroring the guard already present in `ContractOwner.initialize()`. [13](#0-12) 

3. Where possible, pass initialization calldata directly to the `TransparentUpgradeableProxy` constructor so that proxy deployment and initialization are atomic in a single transaction, eliminating the front-running window entirely.

---

### Proof of Concept

```
1. Deployer broadcasts: deploy Clearinghouse proxy (no init data in constructor).
2. Attacker observes the proxy address in the mempool.
3. Attacker broadcasts (higher gas):
       clearinghouseProxy.initialize(
           legitimateEndpoint,
           legitimateQuote,
           legitimateClearinghouseLiq,
           legitimateSpreads,
           attackerAddress   // <-- malicious withdrawPool
       )
4. Attacker's tx mines first; Clearinghouse is now owned by attacker
   and withdrawPool == attackerAddress.
5. Deployer's initialize() tx reverts (already initialized).
   — OR —
   Deployer does not detect the failure and proceeds.
6. All subsequent user withdrawals via withdrawCollateral() call
   handleWithdrawTransfer(token, attackerAddress, amount, idx),
   transferring user funds to the attacker until detected.
``` [14](#0-13)

        uint64 idx,
        bytes[] calldata transactions,
        bytes32 e,
        bytes32 s,
        uint8 signerBitmask
    ) external {
        validateSubmissionIdx(idx);
        require(msg.sender == sequencer);
        // TODO: if one of these transactions fails this means the sequencer is in an error state
        // we should probably record this, and engage some sort of recovery mode

        bytes32 digest = keccak256(abi.encode(idx));
        for (uint256 i = 0; i < transactions.length; ++i) {
            digest = keccak256(abi.encodePacked(digest, transactions[i]));
        }
        verifier.requireValidSignature(digest, e, s, signerBitmask);

        for (uint256 i = 0; i < transactions.length; i++) {
            bytes calldata transaction = transactions[i];
            processTransaction(transaction);
            nSubmissions += 1;
        }
    }
