### Title
Sanctioned Address Can Be Registered as a Linked Signer Without Sanctions Check, Enabling Sanctioned Entity to Operate Through a Subaccount — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processSlowModeTransactionImpl`, the `LinkSigner` branch sets `linkedSigners[txn.sender]` to the caller-supplied signer address without verifying that the new signer is not sanctioned. Any user can submit a slow-mode `LinkSigner` transaction that installs a sanctioned address as the authorized signer for their subaccount. Once installed, the sanctioned entity can sign and submit `WithdrawCollateral`, `TransferQuote`, and trading transactions on behalf of that subaccount, bypassing the protocol's sanctions enforcement.

---

### Finding Description

`submitSlowModeTransactionImpl` (EndpointTx.sol line 375) calls `requireUnsanctioned(sender)` to block sanctioned users from queuing slow-mode transactions. However, it only checks the **submitter** (`msg.sender`), not the **signer address** embedded in the `LinkSigner` payload.

When the queued transaction is later executed via `processSlowModeTransactionImpl`, the `LinkSigner` branch (lines 232–239) performs:

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

There is no `requireUnsanctioned` call on `txn.signer`. The new signer address is written directly into `linkedSigners` regardless of its sanctions status. [1](#0-0) 

For comparison, `depositCollateralWithReferral` in `Endpoint.sol` explicitly checks both the depositor and the depositee:

```solidity
requireUnsanctioned(msg.sender);
requireUnsanctioned(sender);
``` [2](#0-1) 

No equivalent dual-party check exists for the `LinkSigner` slow-mode path.

The `requireUnsanctioned` helper is defined in `EndpointStorage.sol`:

```solidity
function requireUnsanctioned(address sender) internal view virtual {
    require(!sanctions.isSanctioned(sender), ERR_WALLET_SANCTIONED);
}
``` [3](#0-2) 

---

### Impact Explanation

Once a sanctioned address is installed as a linked signer for a subaccount, it can sign any transaction type that `allowLinkedSigner = true` permits, including:

- `WithdrawCollateral` / `WithdrawCollateralV2` — draining collateral to an arbitrary `sendTo` address
- `TransferQuote` — moving quote balance between subaccounts
- `MintNlp` / `BurnNlp` — interacting with the NLP pool

`validateSignedTx` accepts a linked signer's signature as equivalent to the subaccount owner's signature:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [4](#0-3) 

The corrupted state is `linkedSigners[subaccount] = <sanctioned address>`. The sanctioned entity gains full signing authority over the subaccount's assets and positions, defeating the protocol's sanctions controls without any on-chain barrier.

---

### Likelihood Explanation

The attack path is fully permissionless and on-chain:

1. `submitSlowModeTransaction` is callable by any non-sanctioned address with no special role.
2. `executeSlowModeTransaction` is callable by anyone after the 3-day delay.
3. No sequencer involvement is required; the slow-mode queue is the permissionless fallback path. [5](#0-4) 

A sanctioned entity that controls or coordinates with any non-sanctioned address can exploit this immediately. The 3-day delay is the only friction, and it is not a security control — it is a liveness mechanism.

---

### Recommendation

Add a `requireUnsanctioned` check on the signer address in the `LinkSigner` branch of `processSlowModeTransactionImpl`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(transaction[1:], (IEndpoint.LinkSigner));
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
+   requireUnsanctioned(address(uint160(bytes20(txn.signer))));
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

Apply the same fix to the fast-path `LinkSigner` branch in `processTransactionImpl` (lines 576–590), where no sanctions check exists on either the sender or the new signer. [6](#0-5) 

---

### Proof of Concept

1. Sanctioned address `S` coordinates with non-sanctioned address `A`.
2. `A` calls `submitSlowModeTransaction` with a `LinkSigner` payload: `{sender: A_subaccount, signer: S}`.
3. `submitSlowModeTransactionImpl` calls `requireUnsanctioned(A)` — passes, since `A` is not sanctioned. The transaction is queued.
4. After 3 days, anyone calls `executeSlowModeTransaction()`.
5. `processSlowModeTransactionImpl` reaches the `LinkSigner` branch and executes `linkedSigners[A_subaccount] = S` with no sanctions check on `S`.
6. `S` is now the linked signer for `A_subaccount`.
7. `S` signs a `WithdrawCollateralV2` transaction specifying `sendTo = S` and submits it to the sequencer.
8. `validateSignedTx` accepts `S`'s signature via `getLinkedSigner(A_subaccount) == S`.
9. `Clearinghouse.withdrawCollateral` transfers collateral to `S`, a sanctioned address, with no on-chain rejection. [1](#0-0) [7](#0-6)
