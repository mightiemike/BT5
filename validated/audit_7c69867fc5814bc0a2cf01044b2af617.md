### Title
Missing `max_l2_gas_amount` Upper Bound Check for Declare Transactions Allows Gateway Admission of Oversized Gas Bounds - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

The gateway's stateless validator enforces a configurable `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. A Declare transaction with `l2_gas.max_amount = u64::MAX` (or any value exceeding the configured limit of `1,210,000,000`) passes stateless validation and, when paired with `l2_gas.max_price_per_unit = 0`, also passes stateful balance validation, allowing it to be admitted to the mempool in violation of the intended admission policy.

### Finding Description

In `validate_resource_bounds`, the upper bound check on `l2_gas.max_amount` is gated behind a type check that silently skips Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // empty — no check performed
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1,210,000,000`: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms this gap is load-bearing: a Declare transaction with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`, while the same value would be rejected for Invoke or DeployAccount: [3](#0-2) 

The same test for Invoke/DeployAccount confirms the check fires correctly for those types: [4](#0-3) 

**Concrete admission path for `l2_gas.max_amount = u64::MAX`:**

1. Set `l2_gas.max_amount = u64::MAX`, `l2_gas.max_price_per_unit = 0`.
2. Set any non-zero `l1_gas` or `l1_data_gas` bound to satisfy the zero-fee guard:
   ```rust
   if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
       return Err(...ZeroResourceBounds...);
   }
   ``` [5](#0-4) 
3. Stateless validation passes (no upper bound check for Declare).
4. Stateful balance check uses `max_possible_fee`, which is computed with saturating arithmetic. Because `l2_gas.max_price_per_unit = 0`, the l2_gas term contributes zero to `max_possible_fee`, so the account's balance covers it. [6](#0-5) 
5. The transaction is admitted to the mempool.

During execution the OS reads `l2_gas.max_amount` as the initial gas budget:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [7](#0-6) 

The OS then caps this at `validate_max_sierra_gas` / `execute_max_sierra_gas` from versioned constants, so actual gas consumption is bounded. However, the gateway has already admitted a transaction that its own policy was designed to reject.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The `max_l2_gas_amount` config exists precisely to prevent transactions with unreasonably large gas bounds from entering the mempool. For Invoke and DeployAccount transactions this invariant is enforced. For Declare transactions it is not, creating an asymmetric admission policy. Any Declare transaction with `l2_gas.max_amount > 1,210,000,000` (up to `u64::MAX`) bypasses the intended gateway filter and is admitted to the mempool. This is the direct sequencer analog of the external report's missing upper bound check: the protocol enforces a bound for two of three transaction types but silently omits it for the third.

### Likelihood Explanation

**Low.** Exploiting this requires deliberately crafting a Declare transaction with an oversized `l2_gas.max_amount`. The transaction must still carry a valid Sierra class, pass signature verification, and satisfy the stateful nonce and balance checks. The OS caps actual gas consumption regardless, so there is no direct economic harm to the sequencer per transaction. The risk is primarily policy bypass and potential mempool resource accounting inconsistency.

### Recommendation

Apply the same `max_l2_gas_amount` upper bound check to Declare transactions. Remove the empty `if let RpcTransaction::Declare(_) = tx {}` branch and let the existing check apply uniformly:

```rust
// Remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Update the test `valid_l2_gas_amount_on_declare` to expect a `MaxGasAmountTooHigh` error when the amount exceeds the configured limit, consistent with `test_invalid_max_l2_gas_amount` for Invoke/DeployAccount.

### Proof of Concept

```rust
// Declare transaction that bypasses the max_l2_gas_amount check:
let config = StatelessTransactionValidatorConfig {
    validate_resource_bounds: true,
    max_l2_gas_amount: 1_210_000_000,  // production default
    min_gas_price: 0,
    ..Default::default()
};
let tx = rpc_tx_for_testing(
    TransactionType::Declare,
    RpcTransactionArgs {
        resource_bounds: AllResourceBounds {
            l2_gas: ResourceBounds {
                max_amount: GasAmount(u64::MAX),   // far exceeds max_l2_gas_amount
                max_price_per_unit: GasPrice(0),   // zero price → zero l2 fee contribution
            },
            l1_gas: ResourceBounds {               // non-zero to pass zero-fee guard
                max_amount: GasAmount(1),
                max_price_per_unit: GasPrice(1),
            },
            ..Default::default()
        },
        ..Default::default()
    },
);
let validator = StatelessTransactionValidator { config };
// Passes — no MaxGasAmountTooHigh error for Declare:
assert!(validator.validate(&tx).is_ok());
```

The equivalent Invoke transaction with identical resource bounds would return `Err(MaxGasAmountTooHigh { gas_amount: GasAmount(u64::MAX), max_gas_amount: 1_210_000_000 })`.
