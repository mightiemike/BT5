### Title
`ENABLE_SECP_OPS` Flag Guard Bypassed via 4-Byte Opcode Alias — (`File: src/chia_dialect.rs`)

### Summary

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function dispatches `op_secp256k1_verify` and `op_secp256r1_verify` through **two independent code paths**: a 4-byte opcode branch and a 1-byte opcode branch. The 1-byte branch (opcodes 64 and 65) correctly gates execution behind `ClvmFlags::ENABLE_SECP_OPS`. The 4-byte branch (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches to the **same operator functions unconditionally**, with no flag check. An attacker-controlled CLVM program can invoke secp signature verification using the 4-byte opcode form regardless of whether `ENABLE_SECP_OPS` is set, bypassing the flag guard entirely.

---

### Finding Description

`ChiaDialect::op()` first computes a merged `flags` value: [1](#0-0) 

Then it branches on opcode length. For 4-byte opcodes, it dispatches directly to the secp functions **without any flag check**: [2](#0-1) 

For 1-byte opcodes, the same functions are gated behind `ENABLE_SECP_OPS`: [3](#0-2) 

The two opcode forms call identical Rust functions: [4](#0-3) [5](#0-4) 

The `ENABLE_SECP_OPS` flag is defined as a distinct, non-overlapping bit: [6](#0-5) 

`MEMPOOL_MODE` — the stricter validation context — does **not** include `ENABLE_SECP_OPS`: [7](#0-6) 

This means: in mempool mode, a CLVM program using 1-byte opcode `0x40` (64) for secp256k1 is rejected as an unknown operator, but the same program rewritten to use 4-byte opcode `0x13d61f00` executes successfully and returns a valid signature-verification result.

---

### Impact Explanation

The concrete corrupted result is **operator dispatch divergence**: two syntactically distinct but semantically identical CLVM programs (one using the 1-byte alias, one using the 4-byte form) produce different outcomes under the same dialect flags. Specifically:

- Under `MEMPOOL_MODE` (no `ENABLE_SECP_OPS`): 1-byte secp opcode → `EvalErr::Unimplemented`; 4-byte secp opcode → successful secp verification result.
- Any node or validator that sets `ENABLE_SECP_OPS = false` expecting to disable all secp operations is silently wrong: the 4-byte form remains fully active.
- If `ENABLE_SECP_OPS` is intended as a hard-fork activation gate (analogous to `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` for keccak), the 4-byte path allows secp operations to execute before the fork activates, producing a consensus split between nodes that enforce the flag and those that don't.

The cost charged is identical in both paths (`SECP256K1_VERIFY_COST = 1300000`, `SECP256R1_VERIFY_COST = 1850000`), so there is no cost-accounting divergence — only execution-permission divergence. [8](#0-7) 

---

### Likelihood Explanation

The 4-byte opcode values `0x13d61f00` and `0x1c3a8f00` are documented in the source code itself (the comment at line 172–174 explains the cost-encoding formula). Any attacker reading the source can trivially construct a CLVM atom of those 4 bytes and embed it as an operator in a CLVM program. No mining, brute-forcing, or social engineering is required. The entry path is fully attacker-controlled CLVM bytes passed to `run_program`. [9](#0-8) 

---

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte opcode dispatch branch, mirroring the guard applied to the 1-byte aliases:

```rust
// in the op_len == 4 branch, before dispatching:
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode forms for secp operations are uniformly gated by the same flag, eliminating the bypass path. The fix is analogous to how `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` uniformly gates keccak at line 246. [10](#0-9)

### Proof of Concept

A CLVM program that invokes `secp256k1_verify` using the 4-byte opcode form:

```
; Opcode atom: 0x13d61f00 (4 bytes) — secp256k1_verify via the unguarded path
; Program: (0x13d61f00 pubkey msg sig)
; Run with ChiaDialect::new(MEMPOOL_MODE)  — ENABLE_SECP_OPS is NOT set

let opcode = allocator.new_atom(&[0x13, 0xd6, 0x1f, 0x00])?;  // 4-byte secp256k1 opcode
let args   = allocator.new_pair(pubkey, allocator.new_pair(msg, allocator.new_pair(sig, nil))?)?;
let prog   = allocator.new_pair(opcode, args)?;

// This succeeds even though ENABLE_SECP_OPS is absent from MEMPOOL_MODE:
let result = run_program(&mut allocator, &ChiaDialect::new(MEMPOOL_MODE), prog, nil, u64::MAX);
assert!(result.is_ok());  // secp verification ran and passed

// Contrast: the 1-byte alias (opcode 64 = 0x40) is correctly rejected:
let opcode_1byte = allocator.new_atom(&[64])?;
let prog2 = allocator.new_pair(opcode_1byte, args)?;
let result2 = run_program(&mut allocator, &ChiaDialect::new(MEMPOOL_MODE), prog2, nil, u64::MAX);
assert!(result2.is_err());  // EvalErr::Unimplemented — correctly blocked
```

The divergence between `result` (success) and `result2` (error) under identical flags demonstrates the bypass. The root cause is the missing `ENABLE_SECP_OPS` check at: [11](#0-10)
