### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Transaction Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in v0.5.0 specifically because it is vulnerable to the 64-byte transaction Merkle proof forgery attack. The fix — coinbase Merkle proof validation — was added only in `verify_transaction_inclusion_v2`. However, the deprecated function remains a live, unpermissioned public entry point callable by any NEAR account. An attacker can bypass the coinbase anchor check entirely by calling the old function directly, causing the contract to return `true` for a transaction that was never included in a Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion` performs SPV verification by computing a Merkle root from the supplied `tx_id`, `tx_index`, and `merkle_proof`, then comparing it to the stored `block_header.merkle_root`. [1](#0-0) 

The function contains no check that `tx_id` is a leaf node (a real transaction hash) rather than an internal Merkle tree node. The 64-byte attack exploits this: because Bitcoin's Merkle tree construction does not distinguish leaves from internal nodes, an attacker can supply an internal node hash as `tx_id` with a crafted proof path and obtain a valid root match, causing the function to return `true` for a non-existent transaction.

`verify_transaction_inclusion_v2` was introduced to close this gap by requiring a coinbase Merkle proof anchored at index 0, which proves the proof path is rooted at a real leaf: [2](#0-1) 

However, `verify_transaction_inclusion` is still decorated only with `#[deprecated]` and `#[pause]`. In Rust, `#[deprecated]` is a compiler lint — it does not restrict runtime callability. The function remains a fully reachable NEAR public method. The `#[allow(deprecated)]` annotation inside `verify_transaction_inclusion_v2` confirms the compiler knows the old function is still live: [3](#0-2) 

The deprecation notice and the `# Warning` comment in the doc block acknowledge the vulnerability but do not gate or remove the entry point: [4](#0-3) 

This is structurally identical to the H-01 pattern: an operation that should be restricted based on a known-bad state (the absence of coinbase proof validation) is not gated, so an attacker can invoke it directly to bypass the protection that was added in the successor function.

---

### Impact Explanation

**Impact: Medium**

Any downstream NEAR contract or off-chain application that calls `verify_transaction_inclusion` to confirm a Bitcoin transaction can be deceived. The contract returns `true` for a `tx_id` that is an internal Merkle tree node, not a real transaction. This corrupts the SPV proof result — the core security guarantee of the light client — and can be used to falsely prove that an arbitrary Bitcoin transaction occurred, enabling fraud in any protocol that gates actions on this verification (e.g., cross-chain bridges, peg-in/peg-out logic, conditional payments).

---

### Likelihood Explanation

**Likelihood: High**

The function is callable by any unprivileged NEAR account with no staking, role, or deposit requirement beyond the standard NEAR gas fee. The 64-byte Merkle proof forgery technique is publicly documented (https://www.bitmex.com/blog/64-Byte-Transactions) and requires only knowledge of a block's Merkle tree structure, which is public Bitcoin data. No private keys, privileged roles, or social engineering are needed.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or add a hard panic to prevent direct calls:

```rust
#[deprecated(since = "0.5.0", note = "Use `verify_transaction_inclusion_v2` instead.")]
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
+   env::panic_str("verify_transaction_inclusion is disabled; use verify_transaction_inclusion_v2");
    // ...
}
```

Alternatively, make the function `pub(crate)` so it is only callable internally by `verify_transaction_inclusion_v2`, which already performs the coinbase anchor check before delegating to it.

---

### Proof of Concept

1. Attacker identifies a Bitcoin block whose header is stored in the contract (any block in `mainchain_header_to_height`).
2. Attacker obtains the block's full Merkle tree from public Bitcoin data and selects an internal node `N` at depth `d`.
3. Attacker constructs a `merkle_proof` of length `d-1` that, when combined with `N` as `tx_id` at `tx_index`, reproduces the block's `merkle_root`.
4. Attacker calls `verify_transaction_inclusion` directly (not `_v2`) with `tx_id = N`, the crafted `merkle_proof`, and a valid `tx_block_blockhash`.
5. The function computes `compute_root_from_merkle_proof(N, tx_index, &proof) == merkle_root` → `true`.
6. The contract returns `true` for a transaction that does not exist. [5](#0-4)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
