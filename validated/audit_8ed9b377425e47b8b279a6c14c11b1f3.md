### Title
Tokens Permanently Locked in Clearinghouse When Slow-Mode Deposit Transaction Fails During Processing — (`core/contracts/Endpoint.sol`)

---

### Summary

When a user calls `depositCollateralWithReferral`, tokens are transferred to the `Clearinghouse` immediately and a slow-mode transaction is queued. If that slow-mode transaction later fails during processing, the catch block in `_executeSlowModeTransaction` silently discards the failure without returning the already-transferred tokens to the depositor. The protocol's own comment — `// try return funds now removed` — explicitly acknowledges the removal of the refund path, leaving deposited tokens permanently locked in the `Clearinghouse` with no user-accessible recovery mechanism.

---

### Finding Description

`Endpoint.depositCollateralWithReferral` performs two sequential operations:

1. Transfers tokens from `msg.sender` into the `Clearinghouse` via `handleDepositTransfer`.
2. Enqueues a `DepositCollateral` slow-mode transaction for later sequencer processing. [1](#0-0) 

When the sequencer (or a user after the timeout) later calls `executeSlowModeTransaction`, the inner call is wrapped in a `try/catch`:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

The comment `// try return funds now removed` is a direct admission that a refund path once existed and was deliberately deleted. If `processSlowModeTransaction` reverts for any reason — product delisted between deposit and processing, a state invariant violated by the time the 3-day delay elapses, or any other runtime failure — the catch block does nothing. The tokens that were already moved into the `Clearinghouse` at deposit time are now stranded there.

The `Clearinghouse` exposes no owner-callable token-rescue function analogous to `removeLiquidity` in `WithdrawPool`. The only withdrawal path is through a successful `withdrawCollateral` call, which requires the subaccount to have a credited balance — a balance that was never credited because the slow-mode deposit failed. [3](#0-2) 

A concrete trigger exists via `DirectDepositV1.creditDeposit()`, which is callable by anyone with no access control. It approves and forwards all token balances to `depositCollateralWithReferral` for a fixed `subaccount`. If any product is delisted (via `delistProduct`) during the 3-day slow-mode window, the corresponding deposit slow-mode transaction will fail on processing, and the tokens transferred from the DDA to the `Clearinghouse` are permanently locked. [4](#0-3) 

---

### Impact Explanation

User or DDA-held tokens transferred to the `Clearinghouse` during a deposit are permanently inaccessible if the corresponding slow-mode transaction fails. There is no on-chain mechanism for the depositor, the DDA owner, or the protocol owner to recover these tokens from the `Clearinghouse`. The asset delta is a complete loss of the deposited amount for the affected subaccount.

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` creates a meaningful window during which protocol state can change. Product delisting is an admin-callable operation (`delistProduct`) that is part of the normal protocol lifecycle. Any deposit made for a product that is subsequently delisted before the slow-mode transaction is processed will trigger this failure. Additionally, `creditDeposit()` on any deployed `DirectDepositV1` is callable permissionlessly, meaning an attacker can force a deposit into the queue at any time, increasing exposure.

---

### Recommendation

Restore a fund-return path in the catch block of `_executeSlowModeTransaction`. When a slow-mode deposit transaction fails, the `Clearinghouse` should refund the deposited tokens to the original depositor (recoverable from the slow-mode transaction payload). Alternatively, add an owner-callable token-rescue function to `Clearinghouse` analogous to `WithdrawPool.removeLiquidity`, so that stranded deposits can be returned to their rightful owners.

---

### Proof of Concept

1. Product `P` is listed; its token is `T`.
2. User (or anyone) calls `DirectDepositV1.creditDeposit()` — tokens `T` are transferred from the DDA to the `Clearinghouse`; a slow-mode `DepositCollateral` transaction is enqueued with a 3-day delay.
3. Within the 3-day window, the protocol owner calls `delistProduct([P], ...)`.
4. After the delay, the sequencer (or any caller) calls `executeSlowModeTransaction`.
5. `processSlowModeTransaction` → `clearinghouse.depositCollateral` fails because product `P` no longer exists.
6. The catch block at `Endpoint.sol:207` runs; `// try return funds now removed` — no refund is issued.
7. Tokens `T` remain in the `Clearinghouse` with no credited balance for the subaccount and no recovery path. [2](#0-1) [4](#0-3)
