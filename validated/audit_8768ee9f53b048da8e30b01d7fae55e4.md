Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Real User, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to its own `msg.sender`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract. If the pool admin allowlists the router (required for any user to swap via the router), every address — including those explicitly excluded — can bypass the gate by calling any router entry point.

## Finding Description

**Call chain — pool to extension:**

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

**The allowlist check:**

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

**The router never forwards the real user:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly — the pool sees `msg.sender = router`, not the EOA: [4](#0-3) 

The same applies to `exactInput` (line 104), `exactOutputSingle` (line 136), and `exactOutput` (line 165): [5](#0-4) 

**Why existing guards fail:**

The `allowedSwapper` mapping is keyed by `(pool, sender)`. When the router is the caller of `pool.swap()`, `sender` is always the router address regardless of which EOA initiated the transaction. There is no mechanism in the extension, the pool, or the router to propagate the original EOA. The `extensionData` field is forwarded by the router unchanged, but the extension ignores it entirely (the `bytes calldata` parameter is unnamed and unused): [6](#0-5) 

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that protection entirely once the router is allowlisted. Any unpermissioned address can execute swaps at the oracle-anchored bid/ask, draining LP-owned token reserves at will. Because the pool is oracle-priced, a disallowed actor can repeatedly extract value from bins at the live mid-price, causing direct loss of LP principal. This satisfies "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path."

## Likelihood Explanation

`MetricOmmSimpleRouter` is the canonical user-facing entry point. Any pool admin who wants allowlisted users to swap through the router must call `setAllowedToSwap(pool, router, true)`. This is a routine operational step, not an exotic configuration. The bypass is reachable on any production pool that combines `SwapAllowlistExtension` with the router, and the exploit requires no special privileges — any EOA can call the router.

## Recommendation

The extension must gate the economically relevant actor, not the intermediary. Two options:

1. **Decode the real user from `extensionData`**: The router already forwards `params.extensionData` to the pool unchanged. The extension can decode a user-supplied address from that payload and check it against the allowlist. Pool admins must document that router users must include their address in the extension payload.

2. **Document incompatibility**: Clearly state in the extension NatDoc that `SwapAllowlistExtension` cannot enforce per-user restrictions when the pool is accessed through any intermediary contract. Pool admins who need per-user gating must require direct `pool.swap()` calls and must not allowlist the router.

Minimal code-level fix for option 1:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata extensionData)
    external view override returns (bytes4)
{
    address actor = sender;
    if (extensionData.length >= 32) {
        address claimed = abi.decode(extensionData, (address));
        if (claimed != address(0)) actor = claimed;
    }
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][actor]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension; allowAllSwappers = false.
2. Pool admin calls setAllowedToSwap(pool, alice, true).
   // alice is the only allowed swapper.
3. Pool admin calls setAllowedToSwap(pool, router, true).
   // Necessary so alice can use the router.
4. Bob (not allowlisted) calls router.exactInputSingle({pool: pool, ...}).
   // router calls pool.swap(); pool sees msg.sender = router.
   // Extension checks allowedSwapper[pool][router] == true → passes.
   // Bob's swap executes despite being excluded from the allowlist.
5. Bob drains token reserves at oracle mid-price, causing LP losses.
```
