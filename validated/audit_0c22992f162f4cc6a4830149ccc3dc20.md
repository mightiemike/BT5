### Title
Equihash Solution Validity Return Value Ignored — Block Accepted Without Checking `is_valid_solution` Result - (File: `contract/src/zcash.rs`)

---

### Summary

In the Zcash build of the BTC light client, `check_pow` calls `equihash::is_valid_solution(...)` which returns `Result<bool, Error>`. The code handles the `Err` variant (panics), but silently drops the inner `bool` from the `Ok` variant. If the library returns `Ok(false)` — meaning the solution is structurally parseable but cryptographically invalid — the contract proceeds as if the solution is valid and accepts the block.

---

### Finding Description

In `contract/src/zcash.rs`, the `check_pow` function performs Equihash solution validation:

```rust
equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| {
        env::panic_str(&format!("Invalid Equihash solution: {e}"));
    });
``` [1](#0-0) 

`equihash::is_valid_solution` returns `Result<bool, Error>`. The `.unwrap_or_else(|e| { env::panic_str(...) })` call:
- On `Err(e)`: panics the contract — correct.
- On `Ok(bool)`: returns the `bool` — but the returned value is **immediately dropped** without any check.

If the library returns `Ok(false)` (the solution is well-formed but cryptographically invalid), the `bool` is silently discarded and execution continues. No `require!` or assertion follows to enforce that the returned value is `true`. The block is then stored into the canonical chain via `submit_block_header_inner`.

This is the direct analog of the ERC20 unchecked return value pattern: a function returns a boolean success indicator, the caller ignores it, and proceeds as if the operation succeeded.

The correct fix is:

```rust
let is_valid = equihash::is_valid_solution(n, k, &input, &block_header.nonce.0, &block_header.solution)
    .unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}")));
require!(is_valid, "Invalid Equihash solution");
```

---

### Impact Explanation

An attacker can craft a Zcash block header with an Equihash `solution` field that is structurally valid (correct byte length, parseable) but cryptographically invalid (does not satisfy the Equihash constraints). The `equihash::is_valid_solution` library function will return `Ok(false)` for such input. Because the `bool` is dropped, `check_pow` returns without panicking, and the block is accepted into the light client's `headers_pool` and potentially promoted to the mainchain tip.

Once an invalid block is accepted as the canonical tip, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will verify proofs against a fraudulent chain state. Any downstream NEAR contract consuming these verification results will receive a `true` result for transactions that were never confirmed in the real Zcash chain. [2](#0-1) 

---

### Likelihood Explanation

The entry path is fully reachable by any unprivileged NEAR account. `submit_blocks` is a `#[payable]` public method gated only by the `#[trusted_relayer]` macro and the `#[pause]` guard. The `#[trusted_relayer]` macro allows bypass via `Role::UnrestrictedSubmitBlocks` or `Role::DAO`, but even without bypass, a registered relayer (which is an unprivileged role in the staking model) can call it. The attacker only needs to supply a crafted `BlockHeader` with a well-formed but invalid Equihash solution — no privileged key or social engineering is required. [3](#0-2) 

---

### Recommendation

Capture and assert the boolean return value of `equihash::is_valid_solution`:

```rust
let solution_valid = equihash::is_valid_solution(
    n, k, &input, &block_header.nonce.0, &block_header.solution
)
.unwrap_or_else(|e| env::panic_str(&format!("Invalid Equihash solution: {e}")));

require!(solution_valid, "Invalid Equihash solution");
``` [4](#0-3) 

---

### Proof of Concept

1. Build the contract with `--features zcash`.
2. Initialize the contract with a valid Zcash genesis block.
3. Construct a `BlockHeader` for Zcash where:
   - All fields (version, prev_block_hash, bits, time) are valid and pass all `require!` checks in `check_pow`.
   - The `solution` field is a byte array of the correct length (1344 bytes for n=200, k=9) but does not satisfy the Equihash constraints (e.g., all zeros, or a randomly mutated valid solution).
4. Call `submit_blocks` with this header.
5. Observe that `equihash::is_valid_solution` returns `Ok(false)`, the `bool` is dropped, `check_pow` returns without panicking, and the block is inserted into `headers_pool` and set as `mainchain_tip_blockhash`.
6. Call `verify_transaction_inclusion_v2` with a fabricated Merkle proof against the fraudulent tip — it returns `true`. [5](#0-4) [6](#0-5)

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
