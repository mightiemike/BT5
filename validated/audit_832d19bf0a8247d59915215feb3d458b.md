Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any user to bypass the swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of the pool's `swap` call — the router contract address when swaps are routed through `MetricOmmSimpleRouter`. A pool admin who allowlists the router to support standard periphery usage for their curated users inadvertently grants swap access to every user on the network, since the router is a fully permissionless public contract with no access control of its own.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs the allowlist check at line 37:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool. In `MetricOmmPool.sol` at line 230–231, the pool calls `_beforeSwap(msg.sender, ...)`, so `sender` in the extension is `msg.sender` of the pool's `swap` call.

In `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80), the router calls `pool.swap(params.recipient, ...)` directly with no encoding of the originating user. Therefore, when any user routes through the router, the extension receives `sender = router_address` and evaluates `allowedSwapper[pool][router]`.

This creates an irresolvable dilemma:
- If the admin does **not** allowlist the router, individually allowlisted users cannot use the router at all.
- If the admin **does** allowlist the router, every user on the network can bypass the allowlist by calling through the router, since `allowedSwapper[pool][router] == true` for all callers.

The same issue applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all router entry points call `pool.swap(...)` directly without forwarding the originating user identity.

## Impact Explanation
A pool admin deploying a curated pool (e.g., KYC-gated, institution-only) with `SwapAllowlistExtension` who allowlists the router to support standard periphery usage inadvertently opens the pool to all users. Any non-allowlisted address can execute swaps against the pool by calling `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point). The allowlist guard is completely bypassed, allowing unauthorized parties to trade against LP capital deposited under the assumption of a restricted counterparty set. This constitutes a direct policy bypass with fund-impacting consequences for LPs.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the protocol's official swap periphery. Pool admins operating curated pools will naturally want their allowlisted users to be able to use it. The only on-chain path to enable this is `setAllowedToSwap(pool, router, true)` — a single call with no documentation or on-chain warning that it disables the allowlist for all users. The likelihood that at least one curated pool makes this configuration is high.

## Recommendation
The extension must check the economically relevant actor — the originating user — not the technical caller. Two approaches:

1. **Router-forwarded identity**: Have `MetricOmmSimpleRouter` encode the originating `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it, with a trust assumption on the router verified as a factory-registered contract.
2. **Router registry fallback**: In `beforeSwap`, if the direct caller (`sender`) is a factory-registered router, decode and check the originating user from `extensionData`; otherwise check `sender` directly.

The simplest safe fix is option 1: require that any allowlisted router forwards the original user address in `extensionData`, and reject calls from the router that do not carry a valid encoded user identity that passes the allowlist check.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension configured
  alice = allowlisted user  (allowedSwapper[pool][alice] = true)
  bob   = non-allowlisted user

Step 1 — Admin enables router support for alice:
  admin calls setAllowedToSwap(pool, router, true)
  // intends to let alice use the router
  // but this also allows bob

Step 2 — Bob bypasses the allowlist:
  bob calls MetricOmmSimpleRouter.exactInputSingle({
      pool: pool,
      recipient: bob,
      zeroForOne: true,
      amountIn: X,
      ...
  })

Step 3 — Extension check:
  pool.swap(msg.sender=router, ...)           // MetricOmmPool.sol L231
  → _beforeSwap(sender=router, ...)           // MetricOmmPool.sol L230
  → SwapAllowlistExtension.beforeSwap(sender=router, ...)
  → allowedSwapper[pool][router] == true  ✓   // SwapAllowlistExtension.sol L37
  → swap executes for bob with no revert

Result: bob swaps successfully against a pool intended to be restricted to alice only.
```
