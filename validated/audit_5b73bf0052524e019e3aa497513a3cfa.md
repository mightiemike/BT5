### Title
Stale `lastLiquidationFees` Cross-Liquidatee Contamination Causes Under-Coverage of Bad Debt — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a **global** storage variable that is unconditionally overwritten by every liquidation step, regardless of which subaccount is being liquidated. `_finalizeSubaccount` reads this global value to temporarily reduce the insurance pool before computing `insuranceCover`. When the finalization of subaccount B is preceded by a liquidation step on a **different** subaccount A, the stale `L_A` is used instead of B's own last-step fees, causing `insuranceCover` to be under-computed and leaving bad debt that should have been covered by insurance to be socialized instead.

---

### Finding Description

`lastLiquidationFees` is declared as a plain `int128` storage slot in `ClearinghouseStorage`: [1](#0-0) 

It is written unconditionally at the end of every non-finalization liquidation call, for any liquidatee: [2](#0-1) 

The design intent (per the inline comment) is to hold back the fees from the **most recent step of the current liquidatee** so that a small insurance refill does not block socialization. However, the variable carries no liquidatee identity.

In `_finalizeSubaccount`, the global value is subtracted from insurance before computing coverage, then added back unconditionally: [3](#0-2) 

If the sequence of sequencer-processed transactions is:

1. Liquidation step on subaccount **B** (last step) → `lastLiquidationFees = L_B` (small)
2. Liquidation step on subaccount **A** → `lastLiquidationFees = L_A` (large, overwrites)
3. Finalization of subaccount **B** (`productId = uint32.max`) → reads `L_A` instead of `L_B`

then at step 3:
- `v.insurance = insurance − L_A` (artificially low)
- `insuranceCover = min(v.insurance, −quoteBalance.amount)` is smaller than it should be
- The uncovered remainder of B's negative quote balance is passed to `spotEngine.socializeSubaccount`, spreading losses to other users
- At the end, `v.insurance += L_A` restores the insurance pool to a value **higher** than it should be, because the 300-unit gap was borne by socialization rather than insurance

The variable is never reset to zero after a finalization, so the contamination persists across any number of subsequent finalizations until another liquidation step overwrites it.

---

### Impact Explanation

Bad debt that the insurance pool has sufficient funds to cover is instead socialized across all other users' positions. The insurance pool retains excess funds it should have disbursed. This is a direct, quantifiable asset accounting error: other subaccounts absorb losses they should not bear, and the insurance balance is inflated relative to its true obligation. This matches the Critical scope: stale-state execution causing balance bypass.

---

### Likelihood Explanation

In any active market where multiple subaccounts are being liquidated concurrently, the sequencer will naturally interleave liquidation steps across different liquidatees. No special attacker capability is required — a liquidator simply needs to submit a liquidation step on any subaccount between B's last step and B's finalization. This is a normal operational condition, not a contrived edge case.

---

### Recommendation

Scope `lastLiquidationFees` to the specific liquidatee being finalized. The simplest fix is to change the storage variable to a `mapping(bytes32 => int128)` keyed by liquidatee:

```solidity
// ClearinghouseStorage.sol
mapping(bytes32 => int128) internal lastLiquidationFees;
```

Then in `_handleLiquidationPayment`:
```solidity
lastLiquidationFees[txn.liquidatee] = v.liquidationFees;
```

And in `_finalizeSubaccount`:
```solidity
int128 feesToHoldBack = lastLiquidationFees[txn.liquidatee];
v.insurance -= feesToHoldBack;
// ... (use feesToHoldBack in place of lastLiquidationFees throughout)
v.insurance += feesToHoldBack;
lastLiquidationFees[txn.liquidatee] = 0; // reset after finalization
insurance = v.insurance;
```

### Proof of Concept

```
Setup:
  insurance = 1000
  lastLiquidationFees = 0

Step 1: liquidateSubaccountImpl(liquidatee=B, productId=perpId, amount=X)
  → _handleLiquidationPayment sets lastLiquidationFees = 50 (L_B)
  → insurance = 1050

Step 2: liquidateSubaccountImpl(liquidatee=A, productId=spotId, amount=Y)
  → _handleLiquidationPayment sets lastLiquidationFees = 800 (L_A, overwrites L_B)
  → insurance = 1850

Step 3: liquidateSubaccountImpl(liquidatee=B, productId=uint32.max)
  → _finalizeSubaccount:
     v.insurance = 1850 − 800 = 1050   ← uses L_A, not L_B
     quoteBalance.amount = −1200
     insuranceCover = min(1050, 1200) = 1050
     B's quote: −1200 + 1050 = −150 → socialized
     v.insurance = 0 + 800 = 800
     insurance = 800

Expected (with L_B = 50):
     v.insurance = 1850 − 50 = 1800
     insuranceCover = min(1800, 1200) = 1200
     B's quote: −1200 + 1200 = 0 → fully covered, no socialization
     v.insurance = 600 + 50 = 650
     insurance = 650

Delta: 150 units of B's bad debt incorrectly socialized;
       insurance is 800 instead of 650 (150 units over-retained).
```

Assert: after step 3, `insurance == 650` and B's quote balance `== 0`. The unpatched code produces `insurance == 800` and B's quote balance `== −150` (socialized).
