### Title
`isValidDepositAmount` Applies `MIN_FIRST_DEPOSIT_AMOUNT` to Subsequent Deposits During 3-Day Slow Mode Processing Window - (`core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.isValidDepositAmount` determines whether a deposit is a "first deposit" by checking `subaccountIds[subaccount] == 0`. However, `subaccountIds` is only written by `_recordSubaccount`, which is called during slow mode transaction *processing* — not during submission. Because slow mode transactions have a mandatory 3-day delay before execution, any second deposit made within that window is incorrectly treated as a first deposit, forcing the user to meet the `MIN_FIRST_DEPOSIT_AMOUNT` ($5) threshold instead of the `MIN_DEPOSIT_AMOUNT` ($0.1) threshold.

---

### Finding Description

The deposit flow in Nado works as follows:

1. A user calls `Endpoint.depositCollateral()`, which calls `isValidDepositAmount()` to gate the deposit.
2. `isValidDepositAmount` checks `subaccountIds[subaccount] == 0` to decide which minimum applies: [1](#0-0) 

3. If the subaccount is new (`== 0`), `MIN_FIRST_DEPOSIT_AMOUNT` ($5) is required; otherwise `MIN_DEPOSIT_AMOUNT` ($0.1) applies. [2](#0-1) 

4. The deposit is queued as a slow mode transaction with a hardcoded 3-day delay: [3](#0-2) 

5. `_recordSubaccount` — the only function that sets `subaccountIds[subaccount]` — is called only when the slow mode transaction is *processed*, not when it is submitted: [4](#0-3) 

6. `_recordSubaccount` itself only writes the mapping if it is currently zero: [5](#0-4) 

**The gap:** Between the moment the user submits their first deposit (funds already transferred to the clearinghouse) and the moment the slow mode transaction is executed (up to 3 days later), `subaccountIds[subaccount]` remains `0`. Any call to `depositCollateral` during this window will re-evaluate `isValidDepositAmount` against `MIN_FIRST_DEPOSIT_AMOUNT` ($5), even though the user has already committed their first deposit.

---

### Impact Explanation

A user who has already made their first deposit and wants to add more collateral within the 3-day slow mode processing window is forced to deposit at least $5 worth of collateral per additional deposit, instead of the intended $0.1 minimum. This is a **50× overpayment requirement** for what should be a routine top-up. The user's funds from the first deposit are already held by the clearinghouse, but the on-chain `subaccountIds` state has not yet been updated, causing the protocol to misclassify the user as a new depositor.

---

### Likelihood Explanation

The 3-day slow mode delay is a protocol constant and applies to every first deposit. Any active user who deposits and then wants to add more collateral before the slow mode transaction is processed — a realistic scenario for traders responding to market conditions — will be affected. No special conditions or attacker involvement are required; the bug is triggered by normal user behavior.

---

### Recommendation

Update `subaccountIds` (via `_recordSubaccount`) at the time the deposit is *submitted* in `depositCollateral` / `depositCollateralWithReferral`, rather than waiting for slow mode processing. This ensures that subsequent deposits within the 3-day window are correctly evaluated against `MIN_DEPOSIT_AMOUNT` instead of `MIN_FIRST_DEPOSIT_AMOUNT`.

Alternatively, `isValidDepositAmount` could check for the existence of a pending slow mode deposit for the subaccount in addition to `subaccountIds[subaccount] == 0`.

---

### Proof of Concept

1. Alice calls `Endpoint.depositCollateral("default", productId, amount)` with `amount` worth $5 (meeting `MIN_FIRST_DEPOSIT_AMOUNT`). Funds are transferred to the clearinghouse. A slow mode tx is queued with `executableAt = block.timestamp + 3 days`.

2. At this point, `subaccountIds[Alice_subaccount] == 0` because `_recordSubaccount` has not yet been called.

3. One hour later, Alice wants to add $0.50 more collateral. She calls `depositCollateral` again with `amount` worth $0.50.

4. `isValidDepositAmount` evaluates: `subaccountIds[Alice_subaccount] == 0` → `minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT` ($5). The $0.50 deposit fails with `ERR_DEPOSIT_TOO_SMALL`.

5. Alice is forced to deposit at least $5 again, even though she has already established her account and her first deposit is pending in the clearinghouse.

6. Only after the sequencer processes the slow mode transaction (up to 3 days later) does `_recordSubaccount` set `subaccountIds[Alice_subaccount] != 0`, allowing deposits as small as $0.10.

    function isValidDepositAmount(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount
    ) internal returns (bool) {
        int256 minDepositAmount = MIN_DEPOSIT_AMOUNT;
        if (subaccount != X_ACCOUNT && (subaccountIds[subaccount] == 0)) {
            minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT;
        }
        return
            clearinghouse.checkMinDeposit(productId, amount, minDepositAmount);
    }
