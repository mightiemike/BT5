### Title
Deprecated `verify_transaction_inclusion` Remains Externally Callable and Lacks Coinbase Proof Check, Enabling 64-Byte Merkle Forgery — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is still a live, publicly callable NEAR contract method despite being marked `#[deprecated]`. Unlike `verify_transaction_inclusion_v2`, it performs no coinbase Merkle proof validation, leaving it fully exposed to the well-documented 64-byte transaction Merkle proof forgery attack. Any unprivileged NEAR caller can supply a crafted internal-node hash as `tx_id` and receive a `true` proof-of-inclusion result for a transaction that does not exist in the block.

---

### Finding Description

The contract exposes two parallel entry points for SPV proof verification:

**Old path — `verify_transaction_inclusion`** (lines 288–323): [1](#0-0) 

This function is annotated `#[deprecated(since = "0.5.0")]`, but in Rust that attribute is a **compiler lint only**. It does not prevent external NEAR RPC callers — other contracts or users — from invoking the method. The function is also gated only by `#[pause]`, meaning it is fully active unless an admin explicitly pauses the contract.

The function's own documentation warns:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [2](#0-1) 

The sole verification performed is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [3](#0-2) 

There is no check that `tx_id` is a leaf (real transaction) rather than an internal Merkle node.

**New path — `verify_transaction_inclusion_v2`** (lines 347–369): [4](#0-3) 

This function adds a coinbase proof check before delegating to v1:

```rust
require!(
    merkle_tools::compute_root_from_merkle_proof(
        args.coinbase_tx_id.clone(),
        0usize,
        &args.coinbase_merkle_proof,
    ) == header.block_header.merkle_root,
    "Incorrect coinbase merkle proof"
);
``` [5](#0-4) 

Because the coinbase transaction is always at index 0 and its hash is a real leaf, an attacker cannot simultaneously satisfy the coinbase proof and forge a second proof using an internal node — the two constraints are mutually exclusive. The v1 path has no such constraint.

The `ProofArgs` struct used by v1 carries no coinbase fields at all: [6](#0-5) 

---

### Impact Explanation

An attacker constructs a `ProofArgs` where `tx_id` is the hash of an internal Merkle tree node (exploiting the 64-byte transaction vulnerability documented at https://www.bitmex.com/blog/64-Byte-Transactions). They call `verify_transaction_inclusion` directly via NEAR RPC. The function computes a valid Merkle root from the internal node and returns `true`, falsely asserting that a non-existent transaction was included in a confirmed Bitcoin block.

Any downstream contract or off-chain system that consumes this boolean result — for example, a cross-chain bridge releasing funds upon SPV proof confirmation — is deceived into accepting a fabricated transaction. This corrupts the core security guarantee of the light client: the integrity of the proof-of-inclusion result.

---

### Likelihood Explanation

The attack path requires no privileged role, no leaked keys, and no social engineering. The function is callable by any NEAR account. The 64-byte Merkle forgery technique is publicly documented and well understood. The only friction is that an attacker must deliberately target the deprecated v1 endpoint rather than v2 — a trivial choice when the goal is to bypass the coinbase check.

---

### Recommendation

Remove `verify_transaction_inclusion` from the contract's public interface entirely, or enforce that it panics unconditionally (making it a true tombstone). The `#[deprecated]` Rust attribute provides zero enforcement for on-chain callers. As long as the method exists in the `#[near]` impl block, it is reachable via NEAR RPC regardless of the lint annotation. The v2 function is the correct and safe entry point and should be the only one available.

---

### Proof of Concept

1. Identify a confirmed Bitcoin block whose Merkle tree has depth ≥ 1 (any real block).
2. Select any internal node hash `N` at depth 1 (the hash of two concatenated leaf hashes).
3. Construct a `ProofArgs`:
   - `tx_id` = `N`
   - `tx_block_blockhash` = hash of the chosen block (already in the contract's mainchain)
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the sibling path from `N` up to the Merkle root
   - `confirmations` = 1
4. Call `verify_transaction_inclusion(args)` via NEAR RPC.
5. The function returns `true` — confirming inclusion of a transaction that does not exist — because it only checks that the computed root matches `header.block_header.merkle_root`, which it does when starting from an internal node with the correct sibling path. [7](#0-6)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
