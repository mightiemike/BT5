Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Trader, Allowing Any User to Bypass Per-User Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which is forwarded verbatim to `SwapAllowlistExtension.beforeSwap`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual trader. The allowlist therefore gates the router address rather than the individual user, making per-user swap restrictions unenforceable for any router-mediated swap.

## Finding Description
**Root cause — `MetricOmmPool.sol::swap` (line 231):**
```solidity
_beforeSwap(
    msg.sender,   // <-- always the direct caller; router when routed
    recipient,
    ...
);
```
When a user calls `MetricOmmSimpleRouter.exactInput()` (or any `exact*` entry point), the router calls `pool.swap(recipient, ...)` directly. From the pool's perspective `msg.sender` is the router contract address. `ExtensionCalling._beforeSwap` encodes this address as `sender` and passes it to every configured extension, including `SwapAllowlistExtension.beforeSwap`.

`SwapAllowlistExtension.beforeSwap` performs its allowlist lookup keyed on `(pool, sender)`. Because `sender` is the router, the check is:
- If the router address is **not** on the allowlist → every user's router swap is blocked (router is unusable for this pool).
- If the router address **is** on the allowlist → every user, including non-allowlisted ones, can bypass the per-user gate by routing through the router.

In either case the extension cannot distinguish between individual traders sharing the same router. The pool admin cannot simultaneously allow the router as a periphery entry point and enforce a per-user allowlist.

**Call path:**
```
User (non-allowlisted) 
  → MetricOmmSimpleRouter.exactInput(...)
    → MetricOmmPool.swap(recipient, ...) [msg.sender = router]
      → ExtensionCalling._beforeSwap(sender=router, ...)
        → SwapAllowlistExtension.beforeSwap(sender=router, ...)
          → allowedSwapper[pool][router] == true → passes
```

**Existing guards insufficient:** The `onlyPool` modifier on `BaseMetricExtension` only verifies the caller is the bound pool; it does not recover the original user identity. There is no mechanism in the hook interface or the pool to propagate the original EOA through the router.

## Impact Explanation
Any unprivileged, non-allowlisted trader can execute swaps on a pool that is intended to be restricted to a specific set of counterparties by routing through `MetricOmmSimpleRouter`. This breaks the core allowlist functionality — a pool admin's primary tool for restricting swap access — and allows unauthorized parties to trade against LP liquidity, potentially draining bins or executing swaps the pool designer explicitly intended to block. This is broken core pool functionality causing potential loss of funds to LPs and violation of the allowlist invariant.

**Severity: High** — allowlist bypass is unconditional once the router is allowlisted; no special conditions or timing required.

## Likelihood Explanation
Any user aware of the router contract can exploit this. The router is a public, permissionless periphery contract. Pool admins who configure `SwapAllowlistExtension` and also allowlist the router (a natural operational choice to enable normal trading UX) are fully exposed. The attack requires no privileged access, no special tokens, and is repeatable on every swap.

## Recommendation
The router must forward the original caller's identity to the pool, and the pool/extension must use that identity for allowlist checks. Two approaches:

1. **Pass original sender through `callbackData`/`extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension reads and verifies it (requires the extension to trust the router, which must itself be verified).
2. **Transient storage propagation:** The router writes the original `msg.sender` to a transient storage slot before calling the pool; the extension reads it from the pool's transient context. This is the cleanest approach given the protocol already uses EIP-1153 transient storage.
3. **Direct pool calls only:** Document that `SwapAllowlistExtension` pools must not allowlist the router and must require users to call the pool directly.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
// router address is also allowlisted (or allowAll=true for router)

// Attacker (not on allowlist) calls router:
router.exactInput(
    pool,
    zeroForOne,
    amountIn,
    minOut,
    attacker,   // recipient
    extensionData
);
// Pool sees msg.sender = router (allowlisted) → beforeSwap passes
// Attacker receives token output despite not being on the per-user allowlist
```
Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `alice` and the router, attempt swap from `bob` directly (expect revert), then attempt swap from `bob` via router (expect success — demonstrating bypass).
