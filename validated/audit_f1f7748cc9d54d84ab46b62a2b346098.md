### Title
Merkle Internal-Node Second-Preimage Forgery in `verify_transaction_inclusion` — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) accepts any 32-byte value as `tx_id` and passes it directly to `compute_root_from_merkle_proof` with no check that the value is a leaf node. An attacker who knows a block's Merkle tree structure can supply an **internal node hash** as `tx_id` with a correspondingly shorter proof path, causing the function to reconstruct the correct `merkle_root` and return `true` for a transaction that does not exist in the block.

---

### Finding Description

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` is a pure positional hash-chain computation: [1](#0-0) 

It applies no domain-separation between leaf and internal nodes. Every call is structurally identical regardless of whether `transaction_hash` is a real transaction ID or an intermediate tree node.

`verify_transaction_inclusion` feeds the caller-supplied `tx_id` directly into this function and compares the result to the stored `merkle_root`: [2](#0-1) 

The contract's own docstring acknowledges this at line 278: [3](#0-2) 

Despite being `#[deprecated]`, the function remains a live, publicly callable NEAR view/call method. The `#[deprecated]` Rust attribute only emits compiler warnings for Rust callers; it has no effect on NEAR RPC accessibility. The only runtime gate is `#[pause]`, which is inactive by default. [4](#0-3) 

**`verify_transaction_inclusion_v2` is not affected.** Its `merkle_proof.len() == coinbase_merkle_proof.len()` constraint forces the target proof to have the same depth as the coinbase (leaf-level) proof, which is always longer than any internal-node proof, blocking the attack: [5](#0-4) 

---

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) and acts on a `true` result — releasing funds, minting tokens, updating state — can be deceived into treating a non-existent transaction as confirmed. The attacker needs only public knowledge of the block's Merkle tree (all node hashes are derivable from the block's transaction list, which is public on-chain Bitcoin data).

---

### Likelihood Explanation

- The function is publicly callable with no access control beyond the default-inactive pause.
- All required inputs (block hash in `mainchain_header_to_height`, internal node hashes, sibling hashes) are derivable from public Bitcoin data.
- No privileged role, key leak, or social engineering is required.
- Any integrator that has not yet migrated from v1 to v2 is exposed.

---

### Recommendation

1. **Remove or hard-disable `verify_transaction_inclusion` (v1)** — replace the body with `env::panic_str("deprecated: use verify_transaction_inclusion_v2")` so it is unreachable at runtime, not merely discouraged at compile time.
2. If removal is not immediately possible, apply the same proof-length equality guard used in v2 to v1 as a stopgap.
3. Communicate urgently to all known integrators to migrate to `verify_transaction_inclusion_v2`.

---

### Proof of Concept

```
// 4-leaf tree: [A, B, C, D]
// Level 1: AB = hash(A,B),  CD = hash(C,D)
// Root    = hash(AB, CD)

// Legitimate proof for leaf A (index 0):
//   compute_root_from_merkle_proof(A, 0, [B, CD]) == Root  ✓

// Attack — internal node AB as "tx_id" (index 0, proof length 1):
//   compute_root_from_merkle_proof(AB, 0, [CD])
//   = compute_hash(AB, CD)
//   = Root  ✓  → verify_transaction_inclusion returns true
//                 for a transaction that does not exist
```

The proof path is one element shorter than a real leaf proof, which is exactly what `verify_transaction_inclusion_v2` catches via its length-equality check — but v1 has no such guard.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,

    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {

        } else {
