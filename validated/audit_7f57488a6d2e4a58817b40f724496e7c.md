### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` When L2/L1DataGas Are Zero, Causing Wrong Transaction Hash and P2P Rejection - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a **value-based heuristic** — checking whether `l2_gas` and `l1_data_gas` are both zero — to decide which enum variant to reconstruct. A V3 transaction whose `AllResources` bounds happen to carry zero L2 gas and zero L1 data gas is silently reconstructed as `ValidResourceBounds::L1Gas` after a protobuf round-trip. Because `get_tip_resource_bounds_hash` produces a structurally different hash for `L1Gas` vs `AllResources` (the former omits the L1DataGas felt entirely), the transaction hash computed after deserialization diverges from the hash computed at submission time. For Declare transactions in the consensus P2P path this causes an immediate hard error (`DEPRECATED_RESOURCE_BOUNDS_ERROR`); for all V3 transactions in the block-sync path it causes block-hash validation to fail, making a syncing node reject valid blocks.

### Finding Description

**Serialization side** (`From<ValidResourceBounds> for protobuf::ResourceBounds`, line 471–489):

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
```

All three fields are emitted, even when `l2_gas` and `l1_data_gas` are zero.

**Deserialization side** (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, lines 417–436):

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

There is no explicit type tag in the wire format; the variant is inferred purely from the numeric values of the fields. An `AllResources` transaction with zero L2 and L1DataGas bounds is therefore reconstructed as `L1Gas`.

**Hash divergence** (`get_tip_resource_bounds_hash`, lines 188–211 of `crates/starknet_api/src/transaction_hash.rs`):

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1DataGas omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // included
    }
});
```

Even when `l1_data_gas` is zero, `get_concat_resource` encodes the resource-name bytes (`L1_DATA_GAS = 0x…4c315f44415441`) into the felt, so the hash for `AllResources{l1_data_gas=0}` is **not equal** to the hash for `L1Gas`.

**Consensus Declare path hard failure** (`TryFrom<protobuf::DeclareV3WithClass> for RpcDeclareTransactionV3`, lines 169–191 of `crates/apollo_protobuf/src/converters/rpc_transaction.rs`):

```rust
resource_bounds: match common.resource_bounds {
    ValidResourceBounds::AllResources(resource_bounds) => resource_bounds,
    _ => { return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR); }   // ← triggered
},
```

The codebase itself acknowledges the issue in `crates/apollo_protobuf/src/converters/consensus_test.rs` lines 26–47:

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);  // workaround
}
```

The workaround is only applied in the test; production code has no such guard.

### Impact Explanation

Two concrete impacts:

1. **Consensus/P2P Declare rejection (High):** A valid V3 Declare transaction with `AllResources` bounds where `l2_gas = 0` and `l1_data_gas = 0` is propagated via consensus P2P. The receiving node deserializes it, reconstructs `L1Gas`, and the `RpcDeclareTransactionV3` conversion returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The transaction is dropped; the proposer's block is rejected by validators.

2. **Block-sync hash mismatch (High → Critical):** During P2P block sync, V3 transactions (Invoke, Declare, DeployAccount) with the same zero-bound pattern are deserialized with the wrong variant. `validate_transaction_hash` recomputes the hash using `L1Gas` semantics and gets a value that does not match the hash committed in the block header. The syncing node rejects the block as invalid, stalling synchronization.

Both map to: **High — Transaction conversion or signature/hash logic binds the wrong type or hash.**

### Likelihood Explanation

- Any V3 transaction where the user sets `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` triggers the bug. This is a natural configuration for users who only care about L1 gas bounds (e.g., pre-0.13.3 style usage on a post-0.13.3 network).
- No special privilege is required; any user submitting a transaction can trigger it.
- The test comment confirms the developers are aware the heuristic is fragile but have not fixed the production path.

### Recommendation

Add an explicit type discriminator to the protobuf `ResourceBounds` message (e.g., a boolean `is_all_resources` field or a `oneof` wrapper), and use it during deserialization instead of the value-based heuristic. Until the wire format is updated, the deserializer should default to `AllResources` whenever all three fields are present in the message, reserving `L1Gas` only for messages where `l2_gas` is absent (matching the `TODO(Shahak)` comment at line 426).

### Proof of Concept

1. Construct a V3 Invoke or Declare transaction with:
   ```
   resource_bounds = AllResources {
       l1_gas:      { max_amount: X, max_price_per_unit: Y },  // non-zero
       l2_gas:      { max_amount: 0, max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0, max_price_per_unit: 0 },
   }
   ```
2. Submit to the gateway. The gateway computes `tx_hash_A` using `AllResources` semantics (includes L1DataGas felt in the hash preimage).
3. The transaction is included in a block. The block's transaction commitment is computed using `tx_hash_A`.
4. A peer node receives the block via P2P. The transaction is deserialized from protobuf. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` returns `L1Gas` because `l2_gas.is_zero() && l1_data_gas.is_zero()`.
5. The peer recomputes `tx_hash_B` using `L1Gas` semantics (L1DataGas felt omitted). `tx_hash_B ≠ tx_hash_A`.
6. `validate_transaction_hash` returns `false`; the peer rejects the block.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
