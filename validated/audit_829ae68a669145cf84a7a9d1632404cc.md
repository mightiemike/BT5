### Title
Cost Computed After Execution Without `max_cost` Enforcement in `op_ash`/`op_lsh` — (File: `src/more_ops.rs`)

### Summary
Both `op_ash` and `op_lsh` ignore the `max_cost` parameter entirely. The shift computation and heap allocation are performed unconditionally, and the cost is computed only afterward based on the output size — which is determined by the attacker-controlled shift amount. This is a direct analog to the external report's class: a value (cost) is computed at execution time based on attacker-controlled state, with no pre-execution bound enforced.

### Finding Description

`op_ash` (line 918) and `op_lsh` (line 982) in `src/more_ops.rs` both accept `_max_cost: Cost` — the underscore prefix signals the parameter is intentionally unused: [1](#0-0) 

```rust
pub fn op_ash(a: &mut Allocator, input: NodePtr, _max_cost: Cost, _flags: ClvmFlags) -> Response {
    ...
    let v: Number = if a1 > 0 { i0 << a1 } else { i0 >> -a1 };
    let l1 = limbs_for_int(&v);
    let r = a.new_number(v)?;                                          // allocation happens here
    let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;  // cost computed after
    Ok(malloc_cost(a, cost, r))
}
``` [2](#0-1) 

The sequence is:
1. Shift is computed (`i0 << a1` or `i0 >> -a1`)
2. Result is allocated (`a.new_number(v)`)
3. Cost is computed based on `l0 + l1` (input bytes + output bytes)
4. No `check_cost(cost, max_cost)?` is ever called

Contrast this with every other variable-cost operator — `op_sha256`, `op_add`, `op_mul`, `op_concat` — which all call `check_cost(cost, max_cost)?` during iteration to fail early: [3](#0-2) [4](#0-3) 

The outer `run_program` loop passes the remaining budget as `max_cost` to each operator: [5](#0-4) 

`op_ash`/`op_lsh` receive this remaining budget and discard it. The outer loop only catches the exceeded cost on the *next* iteration, after the shift and allocation have already completed.

The cost of `ash`/`lsh` depends on `l1` — the output byte size — which is determined by the shift amount (attacker-controlled). With a maximum shift of 65535 bits and a large input atom, the output can be ~8192 bytes, yielding a cost of `ASHIFT_BASE_COST + (l0 + l1) * ASHIFT_COST_PER_BYTE + l1 * MALLOC_COST_PER_BYTE` — potentially tens of thousands of cost units — computed and charged only after the work is done. [6](#0-5) 

### Impact Explanation

An attacker-controlled CLVM program can invoke `ash` or `lsh` with a large input atom and a near-maximum shift (up to 65535 bits), causing the node to:
1. Perform the full bignum shift computation
2. Allocate up to ~8192 bytes in the `Allocator`

…even when the remaining cost budget is near zero. The cost is only checked by the outer loop after the operation returns. This breaks the invariant that `max_cost` bounds the work done per operation. Inside a softfork guard, where `expected_cost` must match exactly, this also makes the cost of any program using `ash`/`lsh` with solution-provided shift amounts impossible to predict statically, causing `SoftforkCostMismatch` errors. [7](#0-6) 

### Likelihood Explanation

Any CLVM puzzle that uses `ash` or `lsh` with a shift amount derived from the solution (attacker-supplied) is affected. The shift bound of 65535 is enforced, so the maximum extra work per invocation is bounded, but the enforcement gap is real and reachable via any attacker-submitted CLVM bytes passed to `run_program`. Likelihood is **medium** — the extra work per invocation is bounded by the allocator limits, but the cost model invariant is violated on every invocation of these operators when the budget is tight.

### Recommendation

Add a `check_cost` call before performing the shift, using an estimated upper-bound cost based on input size and maximum possible output size. After computing the actual output, perform a second `check_cost` with the precise cost. This mirrors the pattern used by `op_sha256`, `op_add`, and other operators:

```rust
// Before shift:
let max_possible_output = l0 + (a1.unsigned_abs() as usize).div_ceil(8);
let estimated_cost = ASHIFT_BASE_COST + ((l0 + max_possible_output) as Cost) * ASHIFT_COST_PER_BYTE;
check_cost(estimated_cost, max_cost)?;

// After shift:
let cost = ASHIFT_BASE_COST + ((l0 + l1) as Cost) * ASHIFT_COST_PER_BYTE;
check_cost(cost, max_cost)?;
```

### Proof of Concept

A CLVM program that triggers the issue with a near-exhausted budget:

```
; Program: (ash large_atom 65535)
; Run with max_cost = 1000 (well below the actual cost of ~50000+)
; op_ash executes the full shift and allocates ~8192 bytes before
; the outer loop catches CostExceeded on the next iteration.
```

Concretely, `lsh 500 65535` produces a ~8192-byte result atom (confirmed by the op-test corpus): [8](#0-7) 

The cost for this invocation is `277 + (2 + 8192) * 3 + 8192 * 10 ≈ 106,779` — all computed and allocated before any budget check, regardless of the `max_cost` passed in.
