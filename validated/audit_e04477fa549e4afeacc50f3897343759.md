Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address as Swapper, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps on the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool receives `msg.sender = router_address`. If the pool admin allowlists the router (a natural configuration to let permitted users enjoy router UX), every public user can bypass the per-user allowlist by calling the router, as the extension evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

## Finding Description
`SwapAllowlistExtension.beforeSwap` checks the `sender` parameter against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the pool calls the extension) and `sender` is the first argument the pool passes: [1](#0-0) 

`MetricOmmPool.swap` always passes `msg.sender` (the direct caller of `pool.swap()`) as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the pool see `msg.sender = router_address`: [3](#0-2) 

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd users.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` so permitted users can use router features (deadline checks, slippage bounds).
3. Unpermitted user calls `router.exactInputSingle(pool, ...)`.
4. Pool passes `sender = router_address` to the extension.
5. Extension finds `allowedSwapper[pool][router] == true` and passes.
6. Unpermitted user's swap executes — allowlist fully bypassed.

The same bypass applies to `exactInput`, `exactOutputSingle`, and `exactOutput` on the router, and to `simulateSwapAndRevert` on the pool (which also calls `_beforeSwap` with `msg.sender`): [4](#0-3) 

No existing guard prevents this: the extension has no mechanism to distinguish a router-mediated call from a direct call, and no way to simultaneously allowlist the router and restrict individual users through it.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade in a pool. When the router is allowlisted, the guard collapses to "anyone who can call the router" — i.e., the entire public. Restricted pools (permissioned institutional venues, KYC-gated pools, pools with regulatory constraints) lose their access control entirely. Unauthorized users can execute swaps, receive output tokens, and drain LP value through arbitrage or directional trading the allowlist was designed to prevent. This constitutes broken core pool functionality causing potential loss of funds and a broken access-control invariant.

## Likelihood Explanation
The scenario is highly realistic. A pool admin who wants permitted users to benefit from router features will naturally allowlist the router — this is the only way to enable router-mediated swaps for those users. The `SwapAllowlistExtension` provides no safe configuration that achieves both goals simultaneously, making the bypass reachable through ordinary, well-intentioned admin actions. Any unpermitted user can exploit this repeatedly at zero cost beyond gas.

## Recommendation
The extension must check the actual end user, not the intermediary. The cleanest fix is for the router to encode `msg.sender` into `extensionData` and for the extension to decode and verify it when the caller is a known router:

```solidity
// In SwapAllowlistExtension.beforeSwap:
address effectiveSender = sender;
if (isKnownRouter[sender] && extensionData.length >= 20) {
    effectiveSender = abi.decode(extensionData, (address));
}
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][effectiveSender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

The router must be updated to encode `msg.sender` into `extensionData` before calling `pool.swap()`. A registry of trusted routers should be maintained in the extension, settable only by the pool admin.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only permittedUser is allowlisted.
// Pool admin also allowlists the router so permittedUser can use it.
extension.setAllowedToSwap(pool, permittedUser, true);
extension.setAllowedToSwap(pool, address(router), true); // natural admin action

// Attack: unpermittedUser bypasses the allowlist via the router.
vm.startPrank(unpermittedUser);
token0.approve(address(router), type(uint256).max);

// Direct pool.swap() reverts: allowedSwapper[pool][unpermittedUser] == false
vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
pool.swap(unpermittedUser, true, 1000, type(uint128).max, "", "");

// Router call succeeds: allowedSwapper[pool][router] == true
// Extension sees sender = router, not unpermittedUser
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: unpermittedUser,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp + 1,
    extensionData: ""
}));
// swap succeeds — allowlist bypassed
vm.stopPrank();
```
