### Title
Dynamic Resharding Cooldown Invariant Bypassed at Static-to-Dynamic Upgrade Boundary — (`chain/epoch-manager/src/lib.rs`)

### Summary

The `can_reshard` function enforces a minimum-epochs-between-resharding cooldown by reading `EpochInfo::last_resharding()`. For all pre-V5 `EpochInfo` variants (V1–V4), this method unconditionally returns `None`. The cooldown check uses `is_none_or`, which treats `None` as "no resharding has ever occurred" and therefore always permits resharding. If a static resharding (protocol-version-driven shard layout change) occurs in the epoch immediately before `ProtocolFeature::DynamicResharding` activates, the first `EpochInfoV5` is created with `last_resharding = None`, silently erasing the cooldown record of the static resharding. A dynamic resharding can then be scheduled in the very next eligible epoch, violating the invariant that prevents back-to-back reshardings and triggering `InvalidChunkHeaderShardSplit` errors that can halt the chain.

---

### Finding Description

**The invariant.** `DynamicReshardingConfig::min_epochs_between_resharding` is a `NonZeroEpochHeight` (≥ 1). The code comment in `chain/epoch-manager/src/lib.rs` states: *"allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."*

**The cooldown check.** `can_reshard` reads `next_epoch_info.last_resharding()`:

```rust
let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
    next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
});
```

`EpochInfo::last_resharding()` is implemented as:

```rust
pub fn last_resharding(&self) -> Option<EpochHeight> {
    match self {
        Self::V1(_) | Self::V2(_) | Self::V3(_) | Self::V4(_) => None,
        Self::V5(v5) => v5.last_resharding,
    }
}
```

For any pre-V5 epoch, this returns `None`. `is_none_or(...)` evaluates to `true` when the option is `None`, so `can_reshard = true` unconditionally for any epoch whose `EpochInfo` is V1–V4.

**The upgrade-boundary gap.** `finalize_epoch` for epoch T computes `last_resharding` for the new `EpochInfoV5` (epoch T+2) as:

```rust
let has_same_shard_layout = next_next_shard_layout == next_shard_layout;
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
```

When dynamic resharding first activates in epoch T, `next_epoch_info` (for epoch T+1) is a V4 object, so `next_epoch_info.last_resharding() = None`. If no *dynamic* resharding occurs in epoch T (the transitional case), `has_same_shard_layout = true` and `last_resharding = None`. The first `EpochInfoV5` (for epoch T+2) therefore carries `last_resharding = None`.

**The exploitable scenario.** Suppose a static resharding (e.g., SimpleNightshade V2 → V3 protocol upgrade) takes effect in epoch T+1 (scheduled at end of epoch T-1). Dynamic resharding then activates in epoch T. During epoch T, `can_reshard` is called with `next_epoch_info` being the V4 epoch T+1 info → returns `None` → `can_reshard = true`. A dynamic resharding is scheduled for epoch T+2. The child shards from the static resharding became active in epoch T+1; a dynamic resharding of one of those child shards takes effect in epoch T+2 — only one epoch after the static resharding's children appeared. This is exactly the back-to-back scenario the cooldown is designed to prevent.

The code comment in `can_reshard` explicitly acknowledges this gap:

> *"It is theoretically possible that a static resharding was scheduled right before enabling dynamic resharding, but we assume this didn't happen."*

This assumption is an unguarded protocol invariant, not an enforced one.

---

### Impact Explanation

When a dynamic resharding is scheduled in epoch T+2 immediately after a static resharding whose children became active in epoch T+1:

- A child shard's first chunk in epoch T+1 has `ChunkExtra.proposed_split = None` (freshly computed, cooldown not yet elapsed from the child's perspective).
- Near the end of epoch T+1, `compute_proposed_split` is called for that child shard. Because `can_reshard` returns `true` (due to `last_resharding = None`), the child shard may emit `proposed_split = Some(...)` in its chunk header.
- The chunk header validator compares `chunk_header.proposed_split()` against `prev_chunk_extra.proposed_split()`. The mismatch produces `InvalidChunkHeaderShardSplit`.
- This is a consensus-level block/chunk rejection that can halt the chain.

**Severity: High.** A chain halt caused by a deterministic protocol invariant violation at a well-defined upgrade boundary.

---

### Likelihood Explanation

The trigger requires two protocol upgrades to be scheduled in adjacent epochs: a static resharding (e.g., a new `SimpleNightshade` layout) followed immediately by `ProtocolFeature::DynamicResharding`. Both are governance-level protocol upgrades voted on by validators. The NEAR upgrade schedule has historically bundled multiple protocol features into a single release. If `DynamicResharding` is stabilized and deployed in a release that also includes or closely follows a static shard layout change, the boundary condition is reachable without any malicious actor. The code itself flags this as a known assumption rather than an enforced invariant.

---

### Recommendation

In `finalize_epoch`, when creating the first `EpochInfoV5`, detect whether a static resharding occurred in the immediately preceding epochs and initialize `last_resharding` accordingly. Concretely, compare `next_shard_layout` against the shard layout of the epoch before `next_epoch_id`; if they differ, set `last_resharding = Some(next_epoch_info.epoch_height())` rather than inheriting `None` from the pre-V5 `next_epoch_info`. Alternatively, `get_shard_layout_history` already reconstructs the static resharding timeline — use it to populate `last_resharding` at the V4→V5 transition boundary instead of relying on the unguarded assumption.

---

### Proof of Concept

**Epoch sequence:**

| Epoch | Event |
|-------|-------|
| T-1 | Static resharding scheduled (e.g., protocol version bump to SimpleNightshade V3). `finalize_epoch(T-1)` creates `EpochInfoV4` for epoch T+1 with new shard layout. |
| T | `ProtocolFeature::DynamicResharding` activates. `finalize_epoch(T)` creates `EpochInfoV5` for epoch T+2. `next_epoch_info` (T+1) is V4 → `last_resharding() = None`. Transitional path: `next_next_shard_layout == next_shard_layout` → `has_same_shard_layout = true` → `last_resharding = None`. |
| T (during) | `get_upcoming_shard_split` calls `can_reshard(parent_hash, min_epochs_between_resharding)`. `next_epoch_info` (T+1) is V4 → `last_resharding() = None` → `is_none_or(...)` = `true` → `can_reshard = true`. A dynamic resharding of one of the new child shards is scheduled. |
| T+1 | Child shards from static resharding are active. Their first chunks compute `proposed_split = None`. Near epoch end, `compute_proposed_split` runs; `can_reshard` again returns `true` (V4 epoch info). Child shard emits `proposed_split = Some(...)` in chunk header. |
| T+2 | Chunk header validation: `chunk_header.proposed_split() = Some(...)` vs `prev_chunk_extra.proposed_split() = None` → `InvalidChunkHeaderShardSplit` → chain halt. |
