Audit Report

## Title
Caller-Supplied Prices in `simulateSwapAndRevert` Persistently Corrupt Guard Extension State, Enabling DoS of Legitimate Swaps — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary

`simulateSwapAndRevert` accepts caller-controlled `bidPriceX64` / `askPriceX64` and forwards them verbatim to `_beforeSwap` and `_afterSwap` extension hooks. The function terminates with a hard `revert` that rolls back pool-internal storage, but state written by external extension contracts (`PriceVelocityGuardExtension`, `OracleValueStopLossExtension`) during those hooks is not subject to the pool's revert and persists on-chain. An unprivileged caller can inject arbitrary prices to corrupt guard state, causing the guards to block all subsequent legitimate swaps.

## Finding Description

**Root cause — caller-supplied prices reach stateful extension hooks:**

In the real `swap()` path, bid/ask prices are fetched from the oracle before being forwarded to extensions: [1](#0-0) 

In `simulateSwapAndRevert`, the oracle is never called. The caller supplies `bidPriceX64` and `askPriceX64` directly, and the only validation is a `bid < ask` and `bid != 0` check: [2](#0-1) 

Both `_beforeSwap` and `_afterSwap` are called with these caller-controlled prices, then `_nonReentrantAfter()` clears the transient reentrancy lock before the revert: [3](#0-2) 

**Extension 1 — `PriceVelocityGuardExtension` state corruption:**

`beforeSwap` unconditionally writes the caller-supplied mid price to its own storage before performing the velocity check: [4](#0-3) 

These writes are to `priceVelocityState[pool_]` in the extension contract's own storage. When the pool reverts, these writes persist. A subsequent real `swap()` at the true oracle price computes velocity against the injected baseline; if the injected price is extreme (e.g., 99% below oracle), the velocity check at line 72 reverts the legitimate swap: [5](#0-4) 

**Extension 2 — `OracleValueStopLossExtension` watermark inflation:**

`afterSwap` uses the caller-supplied `bidPriceX64` / `askPriceX64` to compute `midPriceX64` and derive per-bin metrics, then writes the updated watermarks to extension storage: [6](#0-5) [7](#0-6) 

Injecting a very high mid price inflates `metricT1` (via `t0 * midPriceX64 / Q64`), causing `hwmS.token1` to ratchet up to an inflated value. On the next real swap at the true oracle price, the real `metricT1` is far below the inflated watermark, triggering `OracleStopLossTriggered` and blocking the swap.

**Why existing guards are insufficient:**

- The only price validation is `bid >= ask` and `bid == 0` — any extreme `bid < ask` pair passes.
- `simulateSwapAndRevert` has no `whenNotPaused` modifier (unlike `swap()`), so the attack works even when the pool is paused.
- `_nonReentrantAfter()` is called before the revert, clearing transient storage, so the attacker can call in a tight loop at zero token cost to walk guard state to any desired value. [8](#0-7) [9](#0-8) 

## Impact Explanation

Broken core pool functionality causing permanently unusable swap flows. Once `PriceVelocityGuardExtension.lastMidPriceX64` is poisoned with an extreme value, every subsequent real swap at the true oracle price exceeds the velocity threshold and reverts. Similarly, once `OracleValueStopLossExtension` watermarks are inflated, the stop-loss fires on every real swap. Both outcomes render the pool's swap function completely unusable until the pool admin manually resets guard state — a privileged, timelocked operation. LPs cannot exit via swaps and traders cannot execute, constituting a direct loss of core pool functionality.

## Likelihood Explanation

- **Unprivileged**: No access control, no `whenNotPaused` modifier on `simulateSwapAndRevert`.
- **Zero cost**: No tokens are transferred; the pool state reverts entirely. Only gas is consumed.
- **Repeatable**: The transient reentrancy lock is cleared before the revert, so the attacker can call in a loop within a single block to systematically drive guard state to any target value.
- **Permanent until admin intervention**: Recovery requires the pool admin to call `setLastMidPrice` (velocity guard) or `proposeOracleStopLossHighWatermarks` + wait for timelock + `executeOracleStopLossHighWatermarks` (stop-loss), which is a timelocked privileged operation.

## Recommendation

1. **Do not invoke extension hooks inside `simulateSwapAndRevert`**. The function's purpose is price quotation; guard extensions have no meaningful role in a simulation guaranteed to revert. Remove the `_beforeSwap` and `_afterSwap` calls from `simulateSwapAndRevert`.
2. If hooks must be called for simulation fidelity, **fetch prices from the oracle** (`_getBidAndAskPriceX64()`) instead of accepting caller-supplied values, matching the real `swap()` path.
3. As a defense-in-depth measure, guard extensions should check `IMetricOmmPool(msg.sender).inSwap() != address(0)` to confirm they are being called from a real swap context before writing state.

## Proof of Concept

```
Setup:
  - Pool configured with PriceVelocityGuardExtension at BEFORE_SWAP_ORDER slot 1.
  - maxChangePerBlockE18 = 0.05e18 (5% per block).
  - Current oracle price: 1.00 in Q64.64 units; lastMidPriceX64 = 1.00 * 2^64.

Attack:
  1. Attacker calls simulateSwapAndRevert(
         recipient = attacker,
         zeroForOne = true,
         amountSpecified = 1,
         priceLimitX64 = 0,
         bidPriceX64 = 1,          // near-zero price
         askPriceX64 = 2,
         extensionData = ""
     )
  2. _beforeSwap fires → PriceVelocityGuardExtension:
       s.lastMidPriceX64 = ~1 (near-zero injected mid)
       s.lastUpdateBlock = block.number
     Velocity check: prevMid was 1.00*2^64, new mid is ~1 → delta/prevMid ≈ 100% >> 5% → 
     BUT the check uses the NEW mid as prevMid for the NEXT call, not this one.
     (The write happens before the check; the check uses the OLD prevMid.)
     So the check passes (comparing old oracle price to injected price, but prevMid was already set).
     
     Actually re-reading: prevMid is read BEFORE the write. The write sets lastMidPriceX64 = injected.
     The check compares injected mid vs old prevMid. If injected is extreme, this check fires and 
     simulateSwapAndRevert reverts with PriceVelocityExceeded — but the storage write at line 57-58 
     already happened before the revert check at line 72.
     
     Wait — if the velocity check reverts inside the extension, the extension's storage write is also 
     rolled back (it's part of the same call). So the attacker needs to inject a price that:
     (a) passes the velocity check in simulateSwapAndRevert (so the write persists), AND
     (b) causes the velocity check to fail on the next real swap.
     
     This is achievable: inject a price just within the velocity threshold (e.g., 4.9% move),
     then the real oracle moves 5.1% from the injected baseline → guard blocks the real swap.
     Or: call simulateSwapAndRevert repeatedly in small steps to walk lastMidPriceX64 far from 
     the real oracle price, each step passing the velocity check individually.

  3. Pool state reverts; extension lastMidPriceX64 = injected value persists.
  4. Real swap() at oracle price → velocity check: |oracle - injected| / injected > threshold → REVERT.
  5. All legitimate swaps are permanently blocked.
```
