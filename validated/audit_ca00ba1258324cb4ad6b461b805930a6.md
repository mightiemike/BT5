### Title
Hard-Fork Dialect Flag `RELAXED_BLS` Implemented in Rust but Not Exported to Python API — (`wheel/src/api.rs`)

---

### Summary

`ClvmFlags::RELAXED_BLS` is a consensus-critical hard-fork flag defined in the Rust core that changes how `bls_g1_negate` and `bls_g2_negate` validate their point arguments. It is never exported to the Python module. Python callers of `run_serialized_chia_program` cannot reference it by name and must guess the raw bit value `0x0008`, or silently omit it — causing consensus divergence once the hard-fork activates.

---

### Finding Description

`ClvmFlags::RELAXED_BLS` is defined in `src/chia_dialect.rs`:

```rust
/// Make bls_g1_negate and bls_g2_negate accept invalid points, as long
/// as they at least have the right number of bytes in the atoms.
/// Hard-fork; enable only when it activates.
const RELAXED_BLS = 0x0008;
``` [1](#0-0) 

The flag is consumed inside `src/bls_ops.rs` by `op_bls_g1_negate` and `op_bls_g2_negate` to decide whether to accept structurally invalid G1/G2 points. Without the flag, those operators reject invalid points with an error; with it, they accept any 48-/96-byte atom. [2](#0-1) 

The Python module registration in `wheel/src/api.rs` exports seven named flag constants:

```rust
m.add("NO_UNKNOWN_OPS",    ClvmFlags::NO_UNKNOWN_OPS.bits())?;
m.add("LIMIT_HEAP",        ClvmFlags::LIMIT_HEAP.bits())?;
m.add("MEMPOOL_MODE",      MEMPOOL_MODE.bits())?;
m.add("ENABLE_SHA256_TREE",ClvmFlags::ENABLE_SHA256_TREE.bits())?;
m.add("ENABLE_SECP_OPS",   ClvmFlags::ENABLE_SECP_OPS.bits())?;
m.add("DISABLE_OP",        ClvmFlags::DISABLE_OP.bits())?;
m.add("CANONICAL_INTS",    ClvmFlags::CANONICAL_INTS.bits())?;
``` [3](#0-2) 

`RELAXED_BLS` (`0x0008`) is absent. The same gap exists for `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` (`0x0100`) and `MALACHITE` (`0x1000`), both of which also alter consensus-visible operator semantics. The Python stub confirms the omission: [4](#0-3) 

The project's own operator-addition checklist explicitly requires: *"Expose the new flag(s) to python in chia_rs."* The flag exists in Rust, is wired into operator dispatch, but the public Python surface has no named constant for it. [5](#0-4) 

---

### Impact Explanation

`run_serialized_chia_program` accepts a raw `u32` flags argument: [6](#0-5) 

Python callers that compose flags from the exported named constants will never include `RELAXED_BLS` because no such constant exists in the module. Once the corresponding hard-fork activates on-chain, any CLVM spend that calls `bls_g1_negate` or `bls_g2_negate` with a point that is structurally invalid (but correctly sized) will be accepted by a Rust node running with `RELAXED_BLS` set and rejected by a Python-driven node that omits it. This is a direct consensus divergence: two nodes evaluating the same attacker-crafted spend bundle reach different validity conclusions, which can be exploited to split the network or cause double-spend windows.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: an adversary submits a spend bundle whose puzzle calls `bls_g1_negate` (opcode 51) or `bls_g2_negate` (opcode 55) with a 48-/96-byte atom that is not a valid curve point. After the hard-fork activates, nodes with `RELAXED_BLS` accept it; nodes without it reject it. Python-based full nodes and wallets that build their flags word from the exported constants will silently omit the bit. The likelihood is moderate today (hard-fork not yet activated) but becomes high the moment activation occurs, because no Python caller can discover the flag from the public API alone.

---

### Recommendation

Add the missing flag constants to the Python module registration in `wheel/src/api.rs`:

```rust
m.add("RELAXED_BLS",                    ClvmFlags::RELAXED_BLS.bits())?;
m.add("ENABLE_KECCAK_OPS_OUTSIDE_GUARD",ClvmFlags::ENABLE_KECCAK_OPS_OUTSIDE_GUARD.bits())?;
m.add("MALACHITE",                      ClvmFlags::MALACHITE.bits())?;
```

Update `wheel/python/clvm_rs/clvm_rs.pyi` with the corresponding typed stubs. This mirrors the pattern already used for every other consensus-relevant flag and satisfies the project's own operator-addition checklist requirement.
