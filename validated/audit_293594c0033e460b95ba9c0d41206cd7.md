### Title
Unbounded `NlpLockedBalanceQueue` Iteration in `tryUnlockNlpBalance` Enables DoS of NLP Liquidations — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

A user can grow their `NlpLockedBalanceQueue` to an arbitrarily large size by minting NLP tokens many times at distinct oracle timestamps. Because `tryUnlockNlpBalance` is unconditionally invoked inside `handleNlpLockedBalance` — which is called by `SpotEngine.updateBalance` whenever `productId == NLP_PRODUCT_ID` — any liquidation that touches the liquidatee's NLP balance will iterate the entire queue of unlocked entries. A sufficiently large queue causes the liquidation transaction to revert due to block gas limit exhaustion, permanently blocking liquidation of the attacker's NLP collateral.

---

### Finding Description

`SpotEngineState.tryUnlockNlpBalance` processes all matured entries in a subaccount's `NlpLockedBalanceQueue` via an unbounded `while` loop:

```solidity
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [1](#0-0) 

A new queue entry is appended each time `handleNlpLockedBalance` is called with a positive `amountDelta` at an oracle timestamp that differs from the last entry's `unlockedAt`:

```solidity
} else {
    queue.balances[queue.balanceCount] = NlpLockedBalance({
        balance: Balance({amount: amountDelta}),
        unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
    });
    queue.balanceCount++;
}
``` [2](#0-1) 

`handleNlpLockedBalance` is triggered from `SpotEngine.updateBalance` whenever the product is `NLP_PRODUCT_ID`:

```solidity
if (productId == NLP_PRODUCT_ID) {
    handleNlpLockedBalance(subaccount, amountDelta);
}
``` [3](#0-2) 

Critically, `tryUnlockNlpBalance` is called **unconditionally at the top** of `handleNlpLockedBalance`, regardless of whether `amountDelta` is positive or negative:

```solidity

    _assertInternal();
    if (subaccount == N_ACCOUNT) return;
    tryUnlockNlpBalance(subaccount);   // <-- always executed
``` [4](#0-3)

---

### Impact Explanation

If the liquidatee has accumulated N unlocked entries in their `NlpLockedBalanceQueue`, every liquidation attempt against their NLP position will iterate N storage slots. Each iteration performs an `SLOAD` + `SSTORE` (delete). At N ≈ 3,000 entries, the loop alone consumes the Ethereum block gas limit (~30M gas), causing the liquidation transaction to revert unconditionally. Because `tryUnlockNlpBalance` is called regardless of the liquidation amount (even partial liquidations), there is no way to split the liquidation to avoid the loop. If NLP is the user's primary or sole collateral, the protocol cannot liquidate the under-collateralized subaccount, resulting in unrecoverable bad debt socialized across all depositors.

---

### Likelihood Explanation

The attacker submits many small `MintNlp` transactions through the sequencer at distinct oracle timestamps. Each mint costs only the sequencer fee and can use a dust NLP amount (no minimum is enforced). Over a period exceeding the 4-day `NLP_LOCK_PERIOD`, the attacker accumulates a large number of locked entries. After the lock period expires, all entries become unlockable simultaneously. The attacker then takes on a leveraged position that becomes liquidatable. The attack requires sustained effort over days but no privileged access, no oracle manipulation, and no external dependency — only repeated legitimate `MintNlp` calls. [6](#0-5) 

---

### Recommendation

1. **Cap queue growth**: Enforce a maximum `balanceCount` per subaccount (e.g., 256 entries). Reject or merge new NLP mints once the cap is reached.
2. **Batch-limit the unlock loop**: Process at most a fixed number of entries per call (e.g., 50), and track progress across calls so the loop never runs unboundedly in a single transaction.
3. **Decouple unlock from liquidation**: Do not call `tryUnlockNlpBalance` inside the liquidation hot path. Instead, require users to pre-process their queue before their NLP balance can be liquidated, or use a lazy accounting approach that does not require iteration.

---

### Proof of Concept

1. Attacker calls `MintNlp` (via the sequencer) once per oracle timestamp update over 5+ days, each time minting 1 wei of NLP. Each call at a new oracle timestamp appends one entry to `nlpLockedBalanceQueues[attacker]`. [2](#0-1) 

2. After 4 days (`NLP_LOCK_PERIOD`), all entries have `unlockedAt <= getOracleTime()`. [1](#0-0) 

3. Attacker opens a leveraged perp position that becomes under-collateralized. Their maintenance health drops below zero, making them liquidatable. [7](#0-6) 

4. Liquidator submits `LiquidateSubaccount` with `productId = NLP_PRODUCT_ID`. `_handleLiquidationPayment` calls `spotEngine.updateBalance(NLP_PRODUCT_ID, attacker, -amount)`. [5](#0-4) 

5. `handleNlpLockedBalance` is invoked, which calls `tryUnlockNlpBalance(attacker)`. The `while` loop iterates over all N unlocked entries, consuming O(N) gas. For N ≥ 3,000, the transaction reverts with out-of-gas. [4](#0-3) 

6. All liquidation attempts against the attacker's NLP collateral revert. The attacker's under-collateralized position persists indefinitely, accumulating bad debt.
