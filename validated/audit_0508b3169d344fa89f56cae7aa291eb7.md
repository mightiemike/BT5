Audit Report

## Title
Synthetic-ratio precision loss in `AnchoredPriceProvider._getBidAndAskPrice` truncates mid to zero, permanently blocking swaps on low-ratio synthetic pools — (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
In synthetic (two-feed) mode, `Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` at line 267 uses `ORACLE_DECIMALS = 1e8` as the intermediate scale. When the base-feed price is more than eight decimal orders of magnitude smaller than the quote-feed price, integer division truncates the ratio to zero. The zero `mid` propagates through `_computeBidAsk`, which returns the stalled sentinel `(0, type(uint128).max)`, causing `getBidAndAskPrice` to revert with `FeedStalled` and blocking every swap on the pool for the duration of the price condition.

## Finding Description
`_getBidAndAskPrice` (line 258–272) reads both legs via `_readLeg`, then computes the synthetic ratio at line 267:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // ORACLE_DECIMALS = 1e8
```

When `mid * 1e8 < mid2`, the result is zero. The only guard before this line is `mid2 == 0` (line 265); there is no post-ratio zero check. The zero `mid` is passed directly to `_computeBidAsk` (line 271).

Inside `_computeBidAsk` (line 299–349), `_bandEdge(0, BPS_BASE_U - half, Floor)` evaluates `Math.mulDiv(0, Q64 * edgeFactor, STEP_DENOM) = 0`, so `refBid == 0` at line 311 triggers the sentinel return `(0, type(uint128).max)`. `getBidAndAskPrice` (line 216) then reverts with `FeedStalled`. `_getBidAndAskPriceX64` in `MetricOmmPool` (line 804–813) wraps this in a `try/catch` that re-reverts with `PriceProviderFailed`, blocking every `pool.swap(...)` call.

Concrete arithmetic: PEPE/USD at $0.00001137 → `mid1 = 1137` (8-decimal); BTC/USD at $120,000 → `mid2 = 12_000_000_000_000` (8-decimal). `Math.mulDiv(1137, 1e8, 12_000_000_000_000) = 113_700_000_000 / 12_000_000_000_000 = 0`.

Per-leg `_readLeg` guards (staleness, `mid == 0`, `spreadBps >= ORACLE_BPS`, priceGuard bounds) all pass normally for both feeds; the truncation occurs only in the ratio step, making it invisible to per-leg validation.

## Impact Explanation
All swaps on any pool whose `AnchoredPriceProvider` is configured in synthetic two-feed mode with a price ratio below `1e-8` are permanently blocked for the duration of the price condition. The pool's core swap functionality is entirely unusable. This matches the allowed impact: **Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows.**

## Likelihood Explanation
Synthetic pairs with extreme price ratios are realistic in production — any meme-coin/BTC or meme-coin/ETH pair can satisfy the condition under ordinary market prices. No privileged action is required; the condition is triggered purely by the ratio of two valid, non-stale oracle prices. Once triggered, the DoS persists until the ratio recovers above `1e-8`. The condition is repeatable and requires no attacker — normal market movement is sufficient.

## Recommendation
Replace the 8-decimal intermediate with an 18-decimal scale to preserve precision across the full realistic price range:

```solidity
// Before (line 267):
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);   // 1e8 — truncates to 0 for ratios < 1e-8

// After:
mid = Math.mulDiv(mid, 1e18, mid2);              // 1e18 — survives ratios down to ~1e-18
```

All downstream consumers of `mid` in `_computeBidAsk` and `_bandEdge` that depend on `STEP_DENOM = ORACLE_DECIMALS * BPS_BASE_U` must be updated to match the new scale. As a minimal defensive patch, add an explicit zero-check immediately after the ratio computation:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max);  // explicit fail-closed, not silent truncation
```

## Proof of Concept
1. Deploy `AnchoredPriceProvider` with `baseFeedId = PEPE/USD`, `quoteFeedId = BTC/USD`.
2. Configure the mock oracle so PEPE/USD returns `mid = 1137`, `spreadBps = 1`, valid `refTime`; BTC/USD returns `mid = 12_000_000_000_000`, `spreadBps = 1`, valid `refTime`. Both pass all `_readLeg` guards.
3. Call `getBidAndAskPrice()`.
4. Observe: `Math.mulDiv(1137, 1e8, 12_000_000_000_000) = 0`; `_computeBidAsk(0, 2)` returns `(0, type(uint128).max)`; call reverts with `FeedStalled`.
5. Any `pool.swap(...)` using this provider also reverts with `PriceProviderFailed`, permanently blocking all swaps on the pool.
