### Title
`ValidResourceBounds` Variant Silently Changes Across Protobuf Round-Trip, Producing a Different Transaction Hash Preimage - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserialization of `ValidResourceBounds` uses a zero-value heuristic to decide between the `L1Gas` and `AllResources` variants. When a V3 invoke transaction is submitted with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}`, the Gateway computes its hash using the `AllResources` branch (which includes `l1_data_gas` in the hash preimage). After the block is synced via P2P and the transaction is deserialized from protobuf as `InvokeTransactionV3`, the same resource bounds are reconstructed as `ValidResourceBounds::L1Gas(X)` — a different variant — causing `get_tip_resource_bounds_hash` to produce a different hash value. The transaction hash computed at admission and the hash computed after P2P deserialization diverge.

### Finding Description

**Step 1 — Hash computed at Gateway (admission)**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. During `convert_rpc_tx_to_internal`, the field is wrapped as `ValidResourceBounds::AllResources(...)` and the hash is computed via `get_invoke_transaction_v3_hash` → `get_tip_resource_bounds_hash`. [1](#0-0) 

When the variant is `AllResources`, the function appends a third element — the packed `l1_data_gas` felt — to the poseidon hash input: [2](#0-1) 

Even when `l1_data_gas` is zero, `get_concat_resource` encodes the 7-byte resource name `b"L1_DATA"` into the felt, producing a non-zero value. The resulting hash H₁ therefore has **four** poseidon inputs: `[tip, l1_gas_packed, l2_gas_packed=0, l1_data_gas_packed≠0]`.

**Step 2 — Hash recomputed after P2P deserialization (sync)**

When a syncing node receives the block over P2P, `InvokeTransactionV3` (which carries `resource_bounds: ValidResourceBounds`) is reconstructed from protobuf via: [3](#0-2) 

The heuristic on line 431 — `if l1_data_gas.is_zero() && l2_gas.is_zero()` — silently converts the `AllResources` variant to `L1Gas`. The `InvokeTransactionV3` struct that holds `ValidResourceBounds`: [4](#0-3) 

now carries `ValidResourceBounds::L1Gas(X)`. When `get_tip_resource_bounds_hash` is called on this, the `L1Gas` branch omits `l1_data_gas` entirely: [5](#0-4) 

The recomputed hash H₂ has only **three** poseidon inputs: `[tip, l1_gas_packed, l2_gas_packed=0]`. H₁ ≠ H₂.

**The two-step divergence**

| Stage | Variant | Hash inputs |
|---|---|---|
| Gateway admission | `AllResources` | tip, L1_GAS, L2_GAS, L1_DATA |
| P2P sync deserialization | `L1Gas` | tip, L1_GAS, L2_GAS |

This is the direct analog of the external bug: a value (the resource-bounds variant) is determined twice at different pipeline stages and can differ between them, causing the hash preimage to change.

The `ValidResourceBounds` enum documents the distinction explicitly: [6](#0-5) 

`L1Gas` is the pre-0.13.3 format; `AllResources` is post-0.13.3. A transaction submitted with `AllResourceBounds` where l2/l1_data are zero is a valid post-0.13.3 transaction, but the protobuf round-trip silently downgrades it to the pre-0.13.3 representation.

The serialization path that encodes `ValidResourceBounds::L1Gas` to protobuf also writes a zero `l1_data_gas` field: [7](#0-6) 

So the wire bytes are identical for `L1Gas(X)` and `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}`, but the hash semantics differ.

### Impact Explanation

**High — Transaction conversion or signature/hash logic binds the wrong hash.**

A user signs H₁ (the `AllResources` hash). After P2P sync, the transaction is stored/served with hash H₂ (the `L1Gas` hash). Any component that recomputes the hash from the deserialized `InvokeTransactionV3` — including `validate_transaction_hash`, RPC `starknet_getTransactionByHash`, or the OS hash assertion `%{ AssertTransactionHash %}` — will see a mismatch. Concretely:

- A syncing node calling `validate_transaction_hash` on a synced block will reject a valid block (consensus/sync DoS).
- An RPC node serving `starknet_getTransactionByHash` will return an authoritative-looking wrong hash for the transaction.
- If the OS recomputes the hash during proof verification, the proof will fail for an otherwise valid transaction.

### Likelihood Explanation

Any user can trigger this by submitting a V3 invoke transaction with `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`. The gateway accepts such transactions (they pass all resource-bound checks). No privileged access is required. The condition is met by any pre-0.13.3-style transaction submitted through the post-0.13.3 RPC interface.

### Recommendation

Remove the zero-value heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Instead, encode the variant explicitly in the protobuf message (e.g., a boolean `is_all_resources` flag, or a separate oneof). Alternatively, always deserialize to `AllResources` when all three resource fields are present in the wire message, regardless of their values. The invariant that must hold is: **the `ValidResourceBounds` variant after deserialization must equal the variant that was used when the transaction hash was originally computed.**

### Proof of Concept

1. Submit an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: {max_amount: 1000, max_price_per_unit: 1}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }`.
2. The Gateway computes H₁ = poseidon("invoke", 3, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0), pack(L1_DATA,0,0)), ...).
3. The transaction is included in a block and propagated via P2P.
4. A syncing node deserializes the `InvokeTransactionV3` from protobuf. `l1_data_gas.is_zero() && l2_gas.is_zero()` → variant becomes `L1Gas`.
5. The syncing node computes H₂ = poseidon("invoke", 3, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0)), ...).
6. H₁ ≠ H₂ because `pack(L1_DATA, 0, 0)` = `0x4c315f44415441_0000000000000000_00000000000000000000000000000000` ≠ 0.
7. `validate_transaction_hash` returns `false` for the synced transaction; the block is rejected or the stored hash is wrong. [8](#0-7) [9](#0-8)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
