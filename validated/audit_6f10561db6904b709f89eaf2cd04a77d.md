### Title
Unconditional Trust of Attacker-Supplied `_cached_sha256_treehash` in `Program.wrap()` Enables Persistent Hash Cache Poisoning — (`wheel/python/clvm_rs/program.py`)

### Summary

`Program.wrap()` blindly copies `_cached_sha256_treehash` from any external `CLVMStorage` object without verification. Because `Program.tree_hash()` short-circuits on a non-`None` cached value and never re-validates it, an attacker who supplies a `CLVMStorage` object with a pre-set wrong hash permanently poisons the hash for the lifetime of that `Program` instance. Every downstream consumer — equality checks, puzzle-hash derivation, coin-ID computation — then operates on the attacker-chosen value.

### Finding Description

**Root cause — `Program.wrap()` copies the cache unconditionally:** [1](#0-0) 

```python
o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
```

Any `CLVMStorage` implementor may carry an arbitrary bytes value in `_cached_sha256_treehash`. `wrap()` copies it verbatim into the new `Program` object with no integrity check.

**Short-circuit in `Program.tree_hash()` — the poisoned value is never recomputed:** [2](#0-1) 

```python
def tree_hash(self) -> bytes:
    if self._cached_sha256_treehash is None:
        self._cached_sha256_treehash = sha256_treehash(self._unwrapped)
    return self._cached_sha256_treehash
```

Once the field is non-`None` (set from the external object), the branch that calls `sha256_treehash` is never taken. The wrong value is locked in for the object's lifetime — exactly the "once set per epoch, cannot be corrected" pattern from the reference report.

**`Treehasher.sha256_treehash()` also trusts the attribute unconditionally on sub-nodes:** [3](#0-2) 

```python
r = getattr(obj, "_cached_sha256_treehash", None)
if r is not None:
    self.cache_hits += 1
    hash_stack.append(r)
    return
```

Any sub-node of the tree that carries a pre-set wrong hash will also short-circuit here, propagating the corruption upward into parent-node hashes.

**`Program.__eq__()` uses `tree_hash()` for all equality decisions:** [4](#0-3) 

```python
def __eq__(self, other) -> bool:
    try:
        other_obj = self.to(other)
    except ValueError:
        return False
    return self.tree_hash() == other_obj.tree_hash()
```

A poisoned hash directly corrupts every equality comparison.

**Analog to the oracle manipulation pattern:**

| Panoptic oracle | clvm_rs `Program` |
|---|---|
| `pokeOracle()` sets oracle once per 64-s epoch; cannot be corrected | `Program.wrap()` copies `_cached_sha256_treehash` once; never re-validated |
| Attacker frontrunning with flash-loan tick | Attacker supplies `CLVMStorage` with wrong `_cached_sha256_treehash` |
| Manipulated value locked in for entire epoch | Manipulated value locked in for object lifetime |
| Downstream solvency checks use wrong price | Downstream puzzle-hash / equality checks use wrong hash |

### Impact Explanation

Any Python code that calls `Program.to(user_supplied_obj)` or `Program.wrap(user_supplied_obj)` where the caller controls `_cached_sha256_treehash` will produce a `Program` whose `tree_hash()` returns an attacker-chosen 32-byte value. Consequences include:

- **Puzzle-hash mismatch**: coin validation that compares `program.tree_hash()` against a stored puzzle hash can be bypassed or broken.
- **False equality / inequality**: `program_a == program_b` returns an attacker-controlled boolean, breaking any guard that relies on program identity.
- **Curry-hash derivation**: `Program.curry_hash()` calls `self.tree_hash()` internally; a poisoned base hash propagates into all derived puzzle hashes. [5](#0-4) 

### Likelihood Explanation

The `CLVMStorage` protocol is an open Python structural protocol — any object with `.atom` and `.pair` qualifies. `Program.to()` is the primary entry point for converting user-supplied data into `Program` objects and is called throughout Chia wallet and puzzle-validation code. An attacker who can pass a Python object (not just serialized bytes) to any code path that calls `Program.to()` or `Program.wrap()` can trigger this. The `_cached_sha256_treehash` attribute name is public and documented by the test suite. [6](#0-5) 

### Recommendation

In `Program.wrap()`, do not copy `_cached_sha256_treehash` from external objects unless the source is a trusted internal type (e.g., `CLVMTree` produced by the Rust deserializer). For all other `CLVMStorage` inputs, leave `_cached_sha256_treehash` as `None` so that `tree_hash()` computes it from the actual content on first access:

```python
# Only propagate the cache from types whose hash provenance is trusted
if isinstance(v, CLVMTree):
    o._cached_sha256_treehash = getattr(v, "_cached_sha256_treehash", None)
else:
    o._cached_sha256_treehash = None
```

Alternatively, add a verification step: after copying the cached value, assert it equals the freshly computed hash (at least in debug/test builds).

### Proof of Concept

```python
from clvm_rs.clvm_storage import CLVMStorage
from clvm_rs.program import Program

WRONG_HASH = b"\xff" * 32

class MaliciousStorage(CLVMStorage):
    atom = b"real_data"
    _pair = None
    _cached_sha256_treehash = WRONG_HASH  # pre-poisoned

    @property
    def pair(self):
        return self._pair

malicious = MaliciousStorage()
p = Program.to(malicious)

# Program.wrap() copies WRONG_HASH unconditionally
assert p._cached_sha256_treehash == WRONG_HASH

# tree_hash() short-circuits and returns the wrong value
assert p.tree_hash() == WRONG_HASH  # should be sha256(b"\x01" + b"real_data")

# Equality check is now attacker-controlled
legitimate = Program.new_atom(b"real_data")
# p and legitimate have the same content but different tree_hash() results
assert p != legitimate  # wrong: they should be equal
```

The poisoned hash is locked in for the lifetime of `p`; no subsequent call to `tree_hash()` will recompute it.
