### Title
Missing Zero-Address Validation for `_oracle` Constructor Parameter Permanently Breaks Pool Swap Functionality — (`File: smart-contracts-poc/contracts/PriceProvider.sol` and `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

Both `PriceProvider` and `AnchoredPriceProvider` accept an `_oracle` address in their constructors and assign it directly to the immutable `offchainOracle` without a zero-address check. If `address(0)` is passed, the deployed price provider silently stores a broken oracle binding. Because `offchainOracle` is immutable, the contract cannot be repaired; any pool that uses this provider will have all swap paths permanently reverted, while the factory's price-provider validation does not catch the defect.

---

### Finding Description

In `PriceProvider.sol` the constructor validates `_factory`, `_baseToken`, and `_quoteToken` for zero, but performs no such check on `_oracle`:

```solidity
// PriceProvider.sol lines 70-73
require(_factory != address(0));
factory = _factory;

offchainOracle = IOffchainOracle(_oracle);   // ← no zero-address guard
``` [1](#0-0) 

The identical pattern appears in `AnchoredPriceProvider.sol`:

```solidity
// AnchoredPriceProvider.sol lines 136-139
require(_factory != address(0));
factory = _factory;

offchainOracle = IOffchainOracle(_oracle);   // ← no zero-address guard
``` [2](#0-1) 

`offchainOracle` is declared `immutable` in both contracts, so it can never be updated after construction: [3](#0-2) [4](#0-3) 

When `getBidAndAskPrice()` is called on a provider whose `offchainOracle` is `address(0)`, the internal call:

```solidity
(uint256 mid, uint256 spread, , uint256 refTime) =
    IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);
``` [5](#0-4) 

issues a high-level call to `address(0)`. The EVM returns success with empty returndata; Solidity's ABI decoder then reverts because it cannot decode four `uint256` return values from zero bytes. The pool's `_getBidAndAskPriceX64()` catches this in its `try/catch` and re-reverts with `PriceProviderFailed`:

```solidity
try IPriceProvider(activePriceProvider).getBidAndAskPrice() returns (...) {
    ...
} catch (bytes memory reason) {
    revert PriceProviderFailed(reason);
}
``` [6](#0-5) 

The factory's `_validatePriceProvider` only checks that the provider is non-zero and that its `token0()`/`token1()` match the pool tokens — it never calls `getBidAndAskPrice()`, so a provider with `_oracle = address(0)` passes pool-creation validation:

```solidity
function _validatePriceProvider(address token0, address token1, address priceProvider) internal view {
    if (priceProvider == address(0)) revert InvalidPriceProvider();
    if (IPriceProvider(priceProvider).token0() != token0 || ...) revert PriceProviderTokenMismatch();
}
``` [7](#0-6) 

Because `token0()` and `token1()` are stored as immutables in the provider (not derived from the oracle), they return correctly even when `offchainOracle = address(0)`, allowing the broken provider to be registered for a live pool.

---

### Impact Explanation

Every call to `swap()` and `getSellAndBuyPrices()` on the affected pool permanently reverts with `PriceProviderFailed`. The swap path is the core revenue and utility function of the pool; its permanent failure renders the pool commercially dead. For pools deployed with `immutablePriceProvider = true` (`priceProviderTimelock == type(uint256).max`), there is no recovery path — the pool must be redeployed and all LP positions migrated. For mutable-provider pools the admin can propose a replacement, but only after the configured timelock elapses, during which the pool is fully non-functional for swaps.

---

### Likelihood Explanation

The `_oracle` parameter is the most operationally critical address in the provider constructor, yet it is the only address-type parameter that lacks a zero guard, while `_factory`, `_baseToken`, and `_quoteToken` are all explicitly checked. A deployer who accidentally passes `address(0)` (e.g., from a misconfigured deployment script or a missing environment variable) will receive no on-chain error at construction time. The factory's post-deployment validation does not catch the defect. The mistake is easy to make and silent.

---

### Recommendation

Add an explicit zero-address check for `_oracle` in both constructors, consistent with the existing guards for `_factory` and the token addresses:

**`PriceProvider.sol`** (after line 70):
```solidity
require(_oracle != address(0));
offchainOracle = IOffchainOracle(_oracle);
```

**`AnchoredPriceProvider.sol`** (after line 136):
```solidity
require(_oracle != address(0));
offchainOracle = IOffchainOracle(_oracle);
```

Optionally, extend `_validatePriceProvider` in the factory to perform a live `getBidAndAskPrice()` probe (or at minimum verify the oracle address is non-zero via the provider's public getter) so that a broken provider is rejected at pool-creation time rather than discovered only when the first swap is attempted.

---

### Proof of Concept

1. Deploy `PriceProvider` with `_oracle = address(0)`, valid `_factory`, `_baseToken = WETH`, `_quoteToken = USDC`, a valid `_offchainFeedId`, and any non-zero `_maxTimeDelta`. Construction succeeds; `offchainOracle` is permanently `address(0)`.

2. Call `MetricOmmPoolFactory.createPool(...)` with this provider as `params.priceProvider`. `_validatePriceProvider` calls `provider.token0()` → `WETH` ✓ and `provider.token1()` → `USDC` ✓. Pool is created successfully.

3. Any caller invokes `pool.swap(...)`. The pool calls `_getBidAndAskPriceX64()`, which calls `provider.getBidAndAskPrice()`. Internally, `IPricedOracle(address(0)).price(feedId, pool)` is issued; the EVM returns empty bytes; Solidity's ABI decoder reverts. The `catch` block fires and the pool reverts with `PriceProviderFailed`.

4. If the pool was created with `priceProviderTimelock == type(uint256).max`, `proposePoolPriceProvider` reverts with `PriceProviderImmutable`; the pool is permanently broken and must be redeployed. [8](#0-7)
