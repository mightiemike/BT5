### Title
u64 Multiplication Overflow in `op_unknown` Cost Calculation Enables Cost Undercharging — (`src/more_ops.rs`)

---

### Summary

In `op_unknown` (`src/more_ops.rs`, line 261), the final cost multiplication `cost *= cost_multiplier + 1` is performed as a plain `u64` operation with no overflow guard. In Rust release builds, integer overflow wraps silently (two's complement). An attacker who crafts a CLVM program with an unknown opcode whose byte encoding encodes a large `cost_multiplier` and passes arguments that drive the base cost above `u32::MAX` can cause the product to wrap to a value ≤ `u32::MAX`, bypassing the post-multiplication guard at line 262 and returning a fraudulently small (or zero) cost to the VM.

---

### Finding Description

`op_unknown` computes the cost of an unknown opcode in two phases:

**Phase 1 — base cost** (lines 209–256): one of four cost functions (constant, add-like, mul-like, concat-like) accumulates a `u64` cost from the arguments. Each inner loop calls `check_cost(…, max_cost)?`, so the base cost is bounded by `max_cost` on exit.

**Phase 2 — multiplier** (lines 260–266):

```rust
// src/more_ops.rs:260-265
check_cost(cost, max_cost)?;          // cost ≤ max_cost here
cost *= cost_multiplier + 1;          // ← plain u64 mul, no overflow check
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`cost_multiplier` is decoded from the opcode bytes via `u32_from_u8`, so it is bounded by `u32::MAX = 4_294_967_295`. Therefore `cost_multiplier + 1` is at most `2^32 = 4_294_967_296`.

With Chia's typical `max_cost = 11_000_000_000`:

```
11_000_000_000 × 4_294_967_296 ≈ 4.72 × 10^19  >  u64::MAX ≈ 1.84 × 10^19
```

The multiplication overflows. In a Rust **release** build the result wraps modulo `2^64`. A carefully chosen `(cost, cost_multiplier)` pair makes the wrapped result ≤ `u32::MAX`, so the guard at line 262 passes and the function returns a fraudulently small cost.

**Concrete overflow-to-zero example:**

| Variable | Value |
|---|---|
| `cost` (base, from cost_function=1) | `2^33 = 8_589_934_592` (≤ 11 × 10^9) |
| `cost_multiplier` | `2^31 − 1 = 2_147_483_647` (≤ u32::MAX) |
| `cost_multiplier + 1` | `2^31` |
| product | `2^33 × 2^31 = 2^64 ≡ 0 (mod 2^64)` |
| guard `0 > u32::MAX`? | **false** → returns `Ok(Reduction(0, nil))` |

The opcode bytes that encode `cost_multiplier = 2^31 − 1` with `cost_function = 1` are a 5-byte atom: `[0x7f, 0xff, 0xff, 0xff, 0x40]` (top 3 bytes = multiplier `0x7fffffff`, last byte bits 7-6 = `01` for cost_function 1).

The base cost `2^33` is reachable with cost_function = 1 by passing approximately 26.6 million 1-byte atoms (each costs `ARITH_COST_PER_ARG + ARITH_COST_PER_BYTE = 323`). In consensus mode the `LIMIT_HEAP` flag is absent, so no heap cap prevents this.

---

### Impact Explanation

**Cost undercharging / consensus divergence.** The VM returns a cost of 0 (or any small value) for an operation that should cost billions of units. This has two concrete consequences:

1. **Block cost limit bypass**: A malicious farmer can include a CLVM program whose true cost exceeds the block cost limit but whose reported cost is near zero, allowing the block to pass validation on release-build nodes.
2. **Debug/release consensus split**: Debug builds panic on the overflow at line 261; release builds silently accept the wrapped cost. Nodes running debug builds reject the block; release-build nodes accept it. This is a hard consensus divergence.

---

### Likelihood Explanation

`op_unknown` is reachable whenever `ClvmFlags::NO_UNKNOWN_OPS` is **not** set. `MEMPOOL_MODE` includes this flag, so mempool-mode validation is protected. However, consensus-mode block validation (the default `ChiaDialect` with no flags, or any dialect without `NO_UNKNOWN_OPS`) is fully exposed. A malicious farmer who constructs a block directly (bypassing the mempool) can exploit this without any special privileges beyond being a farmer.

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant and treat overflow as an error:

```rust
// src/more_ops.rs:261
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

This mirrors the pattern already used in `src/serde_2026/de.rs` where `checked_mul` is used to guard against overflow in atom-count calculations.

### Proof of Concept

**Opcode bytes** (5-byte atom, not starting with `0xffff`):
```
[0x7f, 0xff, 0xff, 0xff, 0x40]
 ^^^^^^^^^^^^^^^^^^^  ^^
 cost_multiplier       cost_function bits 7-6 = 01 (add-like)
 = 0x7fffffff = 2^31-1
```

**Arguments**: ~26.6 million 1-byte atoms (e.g., all `0x01`), driving base cost to `2^33`.

**Expected (correct) behavior**: cost = `2^33 × 2^31 = 2^64` → overflow → `EvalErr::Invalid`.

**Actual release-build behavior**: cost wraps to `0`, guard `0 > u32::MAX` is false, returns `Ok(Reduction(0, nil))`.
