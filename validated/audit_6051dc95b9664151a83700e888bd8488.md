Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded by the pool. When `MetricOmmSimpleRouter` calls `pool.swap()`, the pool's `msg.sender` is the router, which it passes as `sender` to the extension hook. The extension therefore checks whether the **router** is allowlisted rather than the actual end-user. Any pool admin who allowlists the router to enable router-mediated swaps for approved users inadvertently opens the pool to every address that calls the router.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes its own `msg.sender` as `sender` to hooks:**

In `MetricOmmPool.swap` (lines 230–240), the pool calls `_beforeSwap(msg.sender, ...)`. When the caller is `MetricOmmSimpleRouter`, `msg.sender` is the router address, not the original user.

`ExtensionCalling._beforeSwap` (lines 149–177) encodes that `sender` value and forwards it verbatim to every configured extension via `_callExtensionsInOrder`.

`SwapAllowlistExtension.beforeSwap` (lines 37–39) then evaluates:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is the router — so the check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Why the router cannot fix this:** `MetricOmmSimpleRouter.exactInputSingle` (lines 71–80) stores the real payer in transient storage for the swap callback (`_setNextCallbackContext(..., msg.sender, ...)`), but never passes the original `msg.sender` to `pool.swap` as a swap-initiator argument. The pool interface has no such parameter. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin allowlists `allowedUser` and also allowlists the router so that `allowedUser` can swap via the router: `setAllowedToSwap(pool, router, true)`.
3. Attacker (not allowlisted) calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router] == true` → passes.
6. Attacker receives output tokens; allowlist is fully bypassed.

**Existing guards are insufficient:** There is no mechanism in the current design for the extension to recover the original caller. The router stores the real payer only in transient storage for its own callback, and the extension has no access to that slot.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is intended to restrict swaps to a curated set of counterparties. Allowlisting the router — the only way to support router-mediated swaps — collapses the per-user check to a single bit (`allowedSwapper[pool][router]`) that is true for all users. Any unprivileged address can then execute swaps in the restricted pool, receiving output tokens at the oracle-anchored price. This constitutes a direct bypass of the pool's access-control policy with fund-impacting consequences: unauthorized users drain liquidity from a pool designed to serve only approved counterparties. This matches the allowed impact category of admin-boundary break via an unprivileged path and broken core pool functionality causing loss of funds.

## Likelihood Explanation
Likelihood is high. `MetricOmmSimpleRouter` is the primary user-facing entry point for the protocol. A pool admin who deploys `SwapAllowlistExtension` and wants approved users to swap via the router has no alternative but to allowlist the router address — there is no other mechanism. The moment the router is allowlisted, the guard is fully open to all users. The condition is reached through normal, documented usage of the protocol.

## Recommendation
The identity checked by `SwapAllowlistExtension` must be the address that economically initiates the swap, not the intermediate relay contract. Two concrete approaches:

1. **Router forwards caller:** Have `MetricOmmSimpleRouter` pass `msg.sender` as an explicit `initiator` parameter to `pool.swap`, and have the pool forward that value (rather than its own `msg.sender`) to extension hooks. This requires a pool interface change but is the cleanest fix.

2. **Extension reads transient payer:** The router already stores the real caller in transient storage before calling the pool (`_setNextCallbackContext`). The extension could read that well-known transient slot. This avoids a core interface change but couples the extension to the router's transient layout.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only allowedUser is allowlisted.
// Pool admin also allowlists the router so allowedUser can swap via router.
extension.setAllowedToSwap(address(pool), address(router), true);
extension.setAllowedToSwap(address(pool), allowedUser, true);

// Attack: attacker (not allowlisted) calls the router
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool:             address(pool),
    recipient:        attacker,
    zeroForOne:       true,
    amountIn:         1_000e18,
    amountOutMinimum: 0,
    priceLimitX64:    0,
    deadline:         block.timestamp,
    extensionData:    ""
}));
// Succeeds: extension checks allowedSwapper[pool][router] == true
// Attacker receives token1 output; allowlist is fully bypassed.
```
