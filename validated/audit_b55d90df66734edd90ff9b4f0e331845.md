### Title
NLP Pool Subaccount Health Not Checked After Negative Rebalance in `burnNlp` — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.burnNlp()` applies a user-controlled negative quote rebalance to NLP pool subaccounts via `_applyNlpRebalance`, but only checks the **sender's** health afterward. The health of the NLP pool subaccounts that absorb the negative rebalance is never verified. A user can concentrate the entire negative rebalance onto a single NLP pool subaccount, driving its quote balance deeply negative and its health below zero — making it liquidatable.

---

### Finding Description

In `burnNlp`, after state changes are applied, the post-state health check covers only `txn.sender`:

```solidity
// core/contracts/Clearinghouse.sol lines 511–529
spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

if (quoteAmount > 0) {
    spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
    _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18); // drains quote from pools
}

require(
    spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
    ERR_SUBACCT_HEALTH
);
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
// ← No health check for any NLP pool subaccount
``` [1](#0-0) 

`_applyNlpRebalance` updates the `QUOTE_PRODUCT_ID` balance of every NLP pool subaccount:

```solidity
// core/contracts/Clearinghouse.sol lines 439–451
function _applyNlpRebalance(...) internal {
    for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
        spotEngine.updateBalance(
            QUOTE_PRODUCT_ID,
            nlpPools[i].subaccount,
            nlpPoolRebalanceX18[i]
        );
    }
}
``` [2](#0-1) 

The on-chain validation of `nlpPoolRebalanceX18` only enforces:
1. The sum equals `-quoteAmount`
2. Each individual entry is `<= 0`

```solidity
// core/contracts/Clearinghouse.sol lines 506–509
_validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
    require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
}
``` [3](#0-2) 

This means a user can legally set `nlpPoolRebalanceX18[0] = -quoteAmount` and all other entries to `0`, concentrating the full drain on pool 0.

The `nlpPoolRebalanceX18` is part of `SignedBurnNlp` and is included in the signed transaction body passed to `validateSignedTx` via `transaction[1:]`, confirming the user controls this distribution:

```solidity
// core/contracts/EndpointTx.sol lines 568–573
clearinghouse.burnNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,
    nlpPools,
    signedTx.nlpPoolRebalanceX18  // user-signed
);
``` [4](#0-3) 

By contrast, `mintNlp` only applies **positive** rebalance to pools (their quote increases), so no pool health check is needed there. The asymmetry in `burnNlp` — where pools lose quote — is where the missing check matters. [5](#0-4) 

---

### Impact Explanation

NLP pool subaccounts hold `QUOTE_PRODUCT_ID` balances. If a pool's quote balance is driven negative by a concentrated rebalance and the deficit exceeds any other positive contributions to its health, the pool's health goes below zero. The liquidation guard in `liquidateSubaccountImpl` only protects `X_ACCOUNT` and `N_ACCOUNT`:

```solidity
// core/contracts/ClearinghouseLiq.sol lines 604–607
require(
    txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
    ERR_NOT_LIQUIDATABLE
);
``` [6](#0-5) 

NLP pool subaccounts (starting from `NLP_POOL_ACCOUNT_START`) are **not** protected. A liquidator can liquidate a pool subaccount driven unhealthy by this attack, seizing its assets at a discount and disrupting the protocol's liquidity provision.

---

### Likelihood Explanation

The entry path is a standard user-signed `BurnNlp` transaction. The user needs only a non-zero unlocked NLP balance. The malicious `nlpPoolRebalanceX18` distribution passes all on-chain validation. The sequencer processes validly-signed transactions without additional off-chain enforcement of pool health. No privileged access is required.

---

### Recommendation

After `_applyNlpRebalance` in `burnNlp`, add a health check for each NLP pool subaccount that received a non-zero (negative) rebalance, mirroring the pattern used in `forceRebalanceNlpPool`:

```diff
 if (quoteAmount > 0) {
     spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
     _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
 }

+for (uint128 i = 0; i < nlpPools.length; i++) {
+    if (nlpPoolRebalanceX18[i] < 0) {
+        require(
+            getHealth(nlpPools[i].subaccount, IProductEngine.HealthType.INITIAL) >= 0,
+            ERR_SUBACCT_HEALTH
+        );
+    }
+}

 require(
     spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
     ERR_SUBACCT_HEALTH
 );
```

---

### Proof of Concept

1. User holds `X` unlocked NLP tokens in their subaccount.
2. User constructs a `BurnNlp` transaction with `nlpAmount = X`, `nlpPoolRebalanceX18 = [-quoteAmount, 0, 0, ...]` (all drain concentrated on pool 0).
3. Sequencer processes the signed transaction; all on-chain checks pass (sum validation, sign validation, sender health check).
4. Pool 0's quote balance decreases by `quoteAmount`. If this exceeds pool 0's prior quote balance, pool 0's health goes negative.
5. A liquidator calls `liquidateSubaccount` targeting pool 0's subaccount, which is now under maintenance health, and seizes its remaining assets at a liquidation discount.
