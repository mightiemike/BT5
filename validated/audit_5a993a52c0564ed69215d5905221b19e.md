### Title
`getUserFeeRateWithBuilder` Assumes Monotonic Tier-Number-to-Fee-Rate Ordering, Which Is Never Enforced â€” (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`getUserFeeRateWithBuilder` uses a numeric comparison (`feeTier < builder.defaultFeeTier`) to enforce a minimum fee floor for users routing through a builder. This implicitly assumes that higher tier numbers always correspond to higher (or equal) fee rates. No such ordering is enforced anywhere in the codebase. When tier fee rates are configured non-monotonically â€” which is structurally possible and undetectable â€” the floor mechanism either silently undercharges users (protocol fee revenue loss) or overcharges them (user fee accounting corruption).

---

### Finding Description

In `getUserFeeRateWithBuilder`, the effective fee tier for a user is computed as:

```solidity
uint32 feeTier = feeTiers[address(uint160(bytes20(sender)))];
if (feeTier < builder.defaultFeeTier) {
    feeTier = builder.defaultFeeTier;
}
FeeRates memory userFeeRates = getTierFeeRateX18(feeTier, productId);
``` [1](#0-0) 

The intent is that `builder.defaultFeeTier` acts as a minimum fee tier floor â€” users with a "lower" tier (numerically) are bumped up to the builder's minimum. This only makes semantic sense if higher tier numbers always map to higher fee rates (i.e., the mapping is monotonic).

However, `getTierFeeRateX18` performs a pure lookup with no ordering guarantee:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId) public view returns (FeeRates memory) {
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];
    }
    return FeeRates({ makerRateX18: 0, takerRateX18: 200_000_000_000_000 });
}
``` [2](#0-1) 

And `updateTierFeeRates` sets fee rates per tier with no monotonicity check:

```solidity
feeRates[txn.tier][txn.productId] = FeeRates(txn.makerRateX18, txn.takerRateX18);
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
``` [3](#0-2) 

Similarly, `updateBuilder` stores `defaultFeeTier` with no validation that it maps to a fee rate higher than lower-numbered tiers:

```solidity
builders[txn.builderId] = Builder(txn.owner, txn.defaultFeeTier, txn.lowestFeeRate, txn.highestFeeRate);
``` [4](#0-3) 

The `Builder` struct and `UpdateBuilder` struct confirm `defaultFeeTier` is a raw `uint32` with no semantic constraint: [5](#0-4) [6](#0-5)

### Impact Explanation

Two concrete fee accounting corruptions arise:

**Scenario A â€” Protocol fee undercharge (revenue loss):**
- Tier 3 â†’ 5 bps (standard user tier)
- Tier 10 â†’ 1 bps (VIP tier, lower fee)
- User has `feeTier = 3`, builder has `defaultFeeTier = 10`
- `3 < 10` â†’ user is bumped to tier 10 â†’ pays 1 bps instead of 5 bps
- The builder's floor silently reduces the user's fee below their assigned rate; protocol collects less fee than intended

**Scenario B â€” User fee overcharge (accounting corruption):**
- Tier 3 â†’ 1 bps (VIP tier)
- Tier 10 â†’ 5 bps (standard tier)
- User has `feeTier = 3`, builder has `defaultFeeTier = 10`
- `3 < 10` â†’ user is bumped to tier 10 â†’ pays 5 bps instead of 1 bps
- The user's entitled fee rate is violated; their `quoteDelta` is reduced by more than it should be

In both cases, `orderInfo.fee` and `orderInfo.quoteDelta` are computed from the wrong `feeRate`, corrupting the settled quote balance for every affected trade. [7](#0-6) 

---

### Likelihood Explanation

Medium. The protocol owner configures both tier fee rates (via `updateTierFeeRates`) and builder `defaultFeeTier` values (via `updateBuilders` in `ContractOwner.sol`). These are independent operations with no cross-validation. As the number of tiers and builders grows, maintaining a consistent monotonic ordering manually becomes error-prone. A single tier reconfiguration (e.g., reassigning a VIP tier to a lower tier number) can silently break the invariant for all builders whose `defaultFeeTier` crosses the reordered tier. There is no on-chain guard to detect or prevent this. [8](#0-7) 

---

### Recommendation

Replace the numeric tier comparison with a direct fee-rate comparison. Instead of bumping the tier number to `builder.defaultFeeTier`, resolve both the user's fee rate and the builder's minimum fee rate independently, then take the maximum:

```solidity
FeeRates memory userFeeRates = getTierFeeRateX18(feeTiers[address(uint160(bytes20(sender)))], productId);
FeeRates memory builderMinRates = getTierFeeRateX18(builder.defaultFeeTier, productId);
int128 feeRate = taker
    ? MathHelper.max(userFeeRates.takerRateX18, builderMinRates.takerRateX18)
    : MathHelper.max(userFeeRates.makerRateX18, builderMinRates.makerRateX18);
```

This eliminates the implicit monotonicity assumption and makes the floor semantics correct regardless of how tier numbers are assigned.

### Proof of Concept

1. Admin calls `updateTierFeeRates` with `tier=3, takerRateX18=50_000_000_000_000` (5 bps) â€” sets tier 3 as a standard fee tier.
2. Admin calls `updateTierFeeRates` with `tier=10, takerRateX18=10_000_000_000_000` (1 bps) â€” sets tier 10 as a VIP tier.
3. Admin calls `updateBuilders` setting `builderId=1, defaultFeeTier=10`.
4. User A is assigned `feeTier=3` (5 bps standard).
5. User A submits a taker order with `appendix` encoding `builderId=1`.
6. In `getUserFeeRateWithBuilder`: `feeTier=3 < builder.defaultFeeTier=10` â†’ `feeTier` is set to 10.
7. `getTierFeeRateX18(10, productId)` returns 1 bps.
8. User A pays 1 bps instead of 5 bps â€” the builder's `defaultFeeTier` floor has silently reduced the fee below the user's assigned rate, causing protocol fee revenue loss on every such trade. [9](#0-8) [2](#0-1)
