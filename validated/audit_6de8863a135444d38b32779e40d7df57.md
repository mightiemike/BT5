Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` gates the router intermediary instead of the end user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` checks the `sender` argument, which resolves to `msg.sender` of `pool.swap()` — the router address when a user routes through `MetricOmmSimpleRouter`. Any pool admin who allowlists the router (the natural configuration for permissioned pools that want to support router-mediated swaps) inadvertently grants every caller of the router unrestricted swap access, completely defeating the per-pool swap allowlist.

## Finding Description

`SwapAllowlistExtension.beforeSwap()` performs the following check:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the first argument forwarded by the pool. In `MetricOmmPool.swap()`, the pool calls `_beforeSwap(msg.sender, ...)`, so `sender` is `msg.sender` of `pool.swap()`. When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap(params.recipient, ...)` directly, making `msg.sender` at the pool the router address. The extension therefore evaluates `allowedSwapper[pool][router]`, not the identity of the end user.

The complete call chain:
1. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, ...})`
2. Router calls `IMetricOmmPoolActions(params.pool).swap(params.recipient, ...)` — `msg.sender` at pool = router
3. Pool calls `_beforeSwap(msg.sender=router, recipient, ...)` → `ExtensionCalling._beforeSwap()` encodes `sender=router`
4. Extension evaluates `allowedSwapper[pool][router]` → `true` (router was allowlisted) → no revert
5. `userB`'s swap executes at oracle prices against the pool's LP liquidity

**Contrast with `DepositAllowlistExtension`:** The deposit allowlist correctly ignores `sender` (the intermediary, first argument discarded with `address,`) and gates `owner` (the economic beneficiary, second argument). No equivalent "on-behalf-of" field exists on the swap path, so the extension has no way to recover the real end user's identity.

## Impact Explanation

A pool admin who deploys a permissioned pool (e.g., for institutional LPs, KYC'd counterparties, or a closed market-making arrangement) and allowlists the router loses all swap-side access control. Any address on the network can swap against the pool's liquidity at oracle-derived prices. This constitutes a direct loss of LP principal whenever the oracle price differs from the market price (i.e., during normal operation), as the pool was explicitly configured to prevent such swaps. This is a broken core pool functionality causing loss of funds, meeting the Critical/High threshold.

## Likelihood Explanation

Allowlisting the router is the natural and expected configuration for any permissioned pool that wants allowlisted users to benefit from the router's slippage protection, deadline checks, and multi-hop routing. A pool admin who does not allowlist the router forces allowlisted users to call `pool.swap()` directly, losing all periphery conveniences. The bypass therefore activates under a common, well-motivated admin configuration and is repeatable by any unprivileged address with no special capability required.

## Recommendation

`SwapAllowlistExtension.beforeSwap()` should gate the `recipient` argument (the address that receives output tokens and is the economic beneficiary of the swap) rather than `sender` (the intermediary). Alternatively, the router should encode the originating user's address in `extensionData`, and the extension should decode and verify it — though this requires the extension to trust the router's encoding. The cleanest fix mirrors the deposit allowlist design: expose a dedicated "on-behalf-of" field in the swap interface so the extension always sees the real actor, analogous to how `addLiquidity` exposes `owner` separately from `sender`.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` attached to `BEFORE_SWAP_ORDER`.
2. Admin calls `setAllowedToSwap(pool, userA, true)` and `setAllowedToSwap(pool, router, true)` so that `userA` can use the router.
3. `userB` (not allowlisted) calls `router.exactInputSingle({pool: pool, recipient: userB, ...})`.
4. Router calls `pool.swap(userB, zeroForOne, amount, ...)` — `msg.sender` at the pool is `router`.
5. Pool calls `_beforeSwap(sender=router, ...)` → extension evaluates `allowedSwapper[pool][router]` → `true` → no revert.
6. `userB`'s swap executes at oracle prices against the pool's LP liquidity, bypassing the allowlist entirely.

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist only `userA` and `router`, call `router.exactInputSingle` from `userB`, assert no revert and tokens transferred — confirming the bypass.
