### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any caller to bypass the deposit allowlist with permanent loss of deposited funds — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

The `DepositAllowlistExtension.beforeAddLiquidity` hook ignores the `sender` parameter (the actual token payer / caller of `addLiquidity`) and instead gates on the caller-controlled `owner` parameter (the LP share recipient). Any unprivileged user can bypass the allowlist by setting `owner` to an allowlisted address. The pool then collects tokens from the unauthorized caller via the swap callback and mints LP shares to `owner`. The caller permanently loses their tokens with no recovery path, and the pool receives unauthorized liquidity in violation of the admin's intent.

---

### Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and checks only `owner`: [1](#0-0) 

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The `owner` parameter in `MetricOmmPool.addLiquidity` is fully caller-controlled — any address may be supplied: [2](#0-1) 

The pool passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to the extension: [3](#0-2) 

Because the extension ignores `sender` and checks only `owner`, an unauthorized caller can set `owner` to any allowlisted address and the guard passes. The pool then calls back on `msg.sender` (the unauthorized caller) to collect tokens, and mints LP shares to `owner`. The caller's tokens are irrecoverably transferred to the pool; the LP shares belong to `owner`, not the caller. `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

so the caller has no path to recover their principal.

---

### Impact Explanation

**Direct loss of user principal:** The unauthorized caller pays tokens via the `addLiquidity` callback but receives zero LP shares. Their tokens are permanently locked in the pool from their perspective — an exact structural analog to the GroupBuy M-03 bug where a contributor's ETH was accepted but never credited to `pendingBalances`.

**Allowlist invariant broken:** The pool admin's intent — restricting which addresses may deposit — is completely defeated. Any address can inject liquidity into a restricted pool by naming an allowlisted address as `owner`, altering pool depth and bin state without authorization.

**Severity:** Medium/High. The caller loses 100% of deposited principal. The allowlist extension provides no protection against unauthorized deposits.

---

### Likelihood Explanation

- Requires no special permissions; any EOA or contract can call `pool.addLiquidity` directly.
- Allowlisted addresses are publicly readable from `allowedDepositor` storage.
- A caller may trigger this accidentally (wrong `owner` address) or deliberately (griefing, pool manipulation).
- The `MetricOmmSimpleRouter` does not call `addLiquidity`, so the direct pool interface is the attack surface.

---

### Recommendation

Check `sender` (the actual token payer) instead of `owner` (the LP share recipient):

```solidity
// DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension`, which checks `sender` (the address that called `pool.swap`): [5](#0-4) 

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Admin calls `setAllowedToDeposit(pool, trustedLP, true)` — only `trustedLP` is authorized.
3. Unauthorized `attacker` reads `allowedDepositor[pool]` storage, finds `trustedLP`.
4. `attacker` calls `pool.addLiquidity(owner=trustedLP, salt=0, deltas=..., callbackData=..., extensionData=...)`.
5. Pool calls `_beforeAddLiquidity(sender=attacker, owner=trustedLP, ...)`.
6. Extension evaluates `allowedDepositor[pool][trustedLP]` → `true` → **no revert**.
7. `LiquidityLib.addLiquidity` runs: pool calls back on `attacker` to collect tokens; LP shares are minted to `trustedLP`.
8. `attacker` has paid tokens and holds zero LP shares. `removeLiquidity` requires `msg.sender == owner`, so `attacker` cannot recover funds.
9. `trustedLP` holds unexpected LP shares they did not request and may not redeem.

The allowlist is bypassed with zero privilege, and the caller suffers a 100% loss of deposited principal.
