### Title
`ENABLE_SECP_OPS` Flag Not Enforced on 4-Byte Opcode Dispatch Path, Allowing Secp Verification to Execute Without the Flag — (`src/chia_dialect.rs`)

---

### Summary

The `ENABLE_SECP_OPS` flag in `ChiaDialect` is intended to gate access to secp256k1 and secp256r1 signature-verification operators. However, the flag check is applied only to the 1-byte opcode path (opcodes 64 and 65). The 4-byte opcode path (`0x13d61f00` for secp256k1, `0x1c3a8f00` for secp256r1) dispatches to the same operator functions **unconditionally**, bypassing the flag entirely. An attacker-controlled CLVM program can use the 4-byte encoding to invoke secp verification even when `ENABLE_SECP_OPS` is absent, producing a consensus divergence between nodes that enforce the flag and those that do not.

---

### Finding Description

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function contains two independent dispatch paths for the secp operators.

**Path 1 — 4-byte opcodes (lines 157–183): no flag check**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());

    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,   // ← no ENABLE_SECP_OPS check
        0x1c3a8f00 => op_secp256r1_verify,   // ← no ENABLE_SECP_OPS check
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

**Path 2 — 1-byte opcodes (lines 248–249): flag check present**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

The flag is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
```

When `ENABLE_SECP_OPS` is **not** set, the two paths produce contradictory results for the same logical operator:

| Opcode | Encoding | Result without `ENABLE_SECP_OPS` |
|---|---|---|
| 64 | 1-byte | `unknown_operator` (no-op / error) |
| 65 | 1-byte | `unknown_operator` (no-op / error) |
| `0x13d61f00` | 4-byte | `op_secp256k1_verify` **executes** |
| `0x1c3a8f00` | 4-byte | `op_secp256r1_verify` **executes** |

The 4-byte opcodes are the "unknown-operator cost encoding" of the secp operators (the comment on line 172 confirms the cost derivation). They are not a separate, always-enabled feature; they are the same operators reached via a different byte-length path. The flag check that guards the 1-byte path is simply absent from the 4-byte path.

This is structurally identical to the external report's bug: a condition that should uniformly gate a code path is checked in one branch but silently omitted in a parallel branch that reaches the same downstream function.

---

### Impact Explanation

1. **Consensus divergence.** A node running without `ENABLE_SECP_OPS` will accept a CLVM spend that uses the 4-byte secp opcode and passes signature verification. A node that correctly enforces the flag for both paths would reject or no-op the same spend. This divergence can split chain validation.

2. **Pre-softfork activation bypass.** The `ENABLE_SECP_OPS` flag is a hard-fork gate. Before the hard-fork activates, the flag is absent. Any program using the 4-byte encoding can invoke secp verification unconditionally, defeating the purpose of the staged activation.

3. **Mempool-mode inconsistency.** In mempool mode (`NO_UNKNOWN_OPS`), opcode 64 without `ENABLE_SECP_OPS` raises `Unimplemented`. Opcode `0x13d61f00` without `ENABLE_SECP_OPS` silently executes secp verification. A malicious coin can craft a spend that passes mempool validation via the 4-byte path while the 1-byte path would be rejected.

---

### Likelihood Explanation

The 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`) are documented in the source comment and derivable from the cost formula. Any party who reads the source or the cost-formula documentation can construct a CLVM atom of length 4 with those bytes and embed it as an operator in a spend. No privileged access, social engineering, or dependency compromise is required. The entry path is fully attacker-controlled CLVM bytes passed to `run_program`.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte dispatch path, mirroring the guard already present on the 1-byte path:

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());

    let f = match opcode {
        0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
        0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
```

This ensures that both encodings of the secp operators are uniformly gated by the same flag, eliminating the divergence.

---

### Proof of Concept

Construct a CLVM atom of 4 bytes `[0x13, 0xd6, 0x1f, 0x00]` and use it as the operator in a program with valid secp256k1 arguments. Run the program with `ChiaDialect::new(ClvmFlags::empty())` (no `ENABLE_SECP_OPS`). The call reaches `op_secp256k1_verify` and performs real signature verification. Run the same program with the 1-byte opcode `[0x40]` (decimal 64) under the same flags: the call reaches `unknown_operator` and returns NIL. The two encodings of the same operator produce different results under the same flag configuration, confirming the wiring error.
