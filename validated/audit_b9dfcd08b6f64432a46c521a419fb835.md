Audit Report

## Title
Stop-Loss Mid-Price Uses Arithmetic Mean While Protocol Canonical Mid-Price Uses Geometric Mean, Causing Systematic Guard Miscalibration - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as the arithmetic mean `(bid + ask) / 2`, while the protocol's canonical mid-price function `SwapMath.midAndSpreadFeeX64FromBidAsk` and the sibling `PriceVelocityGuardExtension` both use the geometric mean `sqrt(bid * ask)`. By AM-GM, the arithmetic mean is always ≥ the geometric mean, so `metricT1` is systematically inflated, making `breach1` less likely to fire and allowing `zeroForOne == false` swaps to proceed past the configured drawdown floor.

## Finding Description

In `_afterSwapOracleStopLoss` at line 218:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;  // arithmetic mean
``` [1](#0-0) 

The protocol's canonical oracle mid-price function `SwapMath.midAndSpreadFeeX64FromBidAsk` (explicitly documented as "Geometric mid price") computes:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);  // geometric mean
``` [2](#0-1) 

`PriceVelocityGuardExtension.beforeSwap` correctly calls this canonical function:

```solidity
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
``` [3](#0-2) 

The inflated `midPriceX64` feeds directly into `_metrics`:

```solidity
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [4](#0-3) 

Because `mid_arithmetic ≥ mid_geometric`, `metricT1` is always inflated relative to the canonical mid price. The blocking logic:

```solidity
if (breach1 && !zeroForOne) {
    revert OracleStopLossTriggered(...);
}
``` [5](#0-4) 

...is therefore less likely to fire for `zeroForOne == false` swaps (token0 in, token1 out). The stop-loss compares an inflated `metricT1` against the watermark floor, so the actual per-share value in token1 terms (at the canonical geometric mid) can fall below the configured floor without triggering a revert.

## Impact Explanation

The contract's documented invariant is:

> "value per share at oracle marks cannot fall faster than drawdown (one-time) + decay × t (ongoing)" [6](#0-5) 

Because `metricT1` is evaluated at a price higher than the canonical oracle mid, the guard compares the watermark floor against an inflated metric. The actual per-share LP value in token1 terms can fall below the configured drawdown floor without triggering the stop-loss. The bias is `(AM − GM) / GM ≈ s²/8` where `s` is the fractional bid-ask spread: ≈ 0.125% for a 10% spread, ≈ 3.1% for a 50% spread. The discrepancy is largest precisely when the guard is most needed (volatile or manipulated markets with wide spreads), constituting a direct loss of LP principal past the configured protection threshold. This meets the Medium/High threshold for broken core pool functionality causing loss of LP assets.

## Likelihood Explanation

Every public swap on a pool with `OracleValueStopLossExtension` configured in its `afterSwap` slot triggers this path via `afterSwap` → `_afterSwapOracleStopLoss`. No special permissions are required; any unprivileged trader calling swap on such a pool triggers the miscalibrated check. The discrepancy is systematic and repeatable on every `zeroForOne == false` swap when the pool has a non-zero bid-ask spread. [7](#0-6) 

## Recommendation

Replace the arithmetic mean with the canonical geometric mean, consistent with `SwapMath.midAndSpreadFeeX64FromBidAsk` and `PriceVelocityGuardExtension`:

```solidity
// Before (arithmetic mean — inconsistent with protocol canonical mid):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (geometric mean — consistent with SwapMath and PriceVelocityGuardExtension):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64), uint256(askPriceX64)
);
``` [1](#0-0) [8](#0-7) 

## Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension` in the `afterSwap` slot, `drawdownE6 = 50_000` (5% floor), and a 10% oracle spread (e.g., `bid = 1.0e18`, `ask = 1.1e18` in Q64.64 units).
2. Seed the pool with token0 and token1 so that `metricT1` is just above the 5% floor at the geometric mean price (`sqrt(1.0 × 1.1) ≈ 1.0488`), i.e., the watermark is set such that `metricT1_geometric` is at the floor boundary.
3. Execute a `zeroForOne == false` swap. The extension computes `midPriceX64 = (1.0 + 1.1)/2 = 1.05`, which is ≈ 0.12% higher than the geometric mean `≈ 1.0488`.
4. The inflated `midPriceX64` raises `metricT1` by ≈ 0.06% (for a 50/50 bin), keeping it above the floor even though the actual value at the canonical geometric mid is below the floor.
5. The stop-loss does not revert; the swap completes and LP token1 value leaks past the configured drawdown limit. [9](#0-8)
