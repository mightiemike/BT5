### Title
Caller-Controlled `confirmations = 0` Bypasses Confirmation-Depth Guard in Unprotected SPV Proof Functions - (`contract/src/lib.rs`)

---

### Summary

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, unprotected NEAR entry points. The `confirmations` field inside `ProofArgs` / `ProofArgsV2` is entirely caller-supplied. The contract enforces only an upper-bound check (`confirmations <= gc_threshold`) and no lower-bound check. Passing `confirmations = 0` trivially satisfies every guard, causing the function to return `true` for a transaction in any mainchain block regardless of how recently that block was added — including the chain tip itself, which has zero confirmation depth and is fully reorg-able.

---

### Finding Description

`verify_transaction_inclusion` performs two checks on the caller-supplied `confirmations` value:

```
// upper-bound only — no lower-bound
require!(args.confirmations <= self.gc_threshold, ...);

// depth check
require!(
    (heaviest_block_header.block_height)
        .saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

When `confirmations = 0`:
- First check: `0 <= gc_threshold` → always passes.
- Second check: `(any u64 value) >= 0` → always passes because Rust `u64` arithmetic is unsigned; the left-hand side can never be negative.

The function then proceeds to verify the Merkle proof and returns `true` if it matches — with no confirmation depth enforced at all.

`verify_transaction_inclusion_v2` inherits the same flaw because it converts its `ProofArgsV2` directly into `ProofArgs` (preserving the caller-supplied `confirmations`) and delegates to `verify_transaction_inclusion`:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [2](#0-1) 

The `confirmations` field is defined as a plain `u64` with no documented minimum and no on-chain enforcement of a minimum: [3](#0-2) 

---

### Impact Explanation

Any consuming NEAR contract (bridge, cross-chain application, escrow) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` and gates an action on a `true` result can be deceived. An attacker who controls the call arguments can:

1. Broadcast a Bitcoin transaction and wait for it to appear in any mainchain block (even the tip, with 0 real confirmations).
2. Immediately call `verify_transaction_inclusion_v2` with `confirmations = 0` and a valid Merkle proof for that block.
3. Receive `true` from the contract.
4. Use that `true` result to trigger the consuming contract's action (e.g., release bridged funds) before the block has any meaningful confirmation depth.

Because the tip block is fully reorg-able, the attacker can then attempt to double-spend the underlying Bitcoin transaction on the source chain while the consuming contract has already acted on the forged "confirmed" proof. The corrupted value is the **proof result** (`true`) returned for a transaction with zero confirmation depth, which is the canonical output that downstream contracts consume.

---

### Likelihood Explanation

The entry point is fully public — no role, stake, or privileged key is required. The only cost is a valid Merkle proof for a real transaction in any mainchain block, which is trivially obtainable from any Bitcoin node. The attacker supplies `confirmations = 0` in the serialized `ProofArgs` struct. This is a realistic, low-effort attack path for any party interacting with a bridge or cross-chain protocol built on top of this light client.

---

### Recommendation

Enforce a protocol-level minimum confirmation count inside the contract, independent of the caller-supplied value:

```rust
const MIN_CONFIRMATIONS: u64 = 6; // or a configurable on-chain parameter

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    "Confirmations below protocol minimum"
);
```

Do not rely on callers to supply a safe `confirmations` value. The contract should enforce a floor just as it enforces a ceiling (`<= gc_threshold`).

---

### Proof of Concept

1. Deploy the contract with any valid genesis and a live mainchain tip at height `H`.
2. Obtain a real Bitcoin transaction `tx_id` included in the tip block (height `H`), along with its Merkle proof.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_block_blockhash` = hash of the tip block
   - `tx_id`, `tx_index`, `merkle_proof`, `coinbase_tx_id`, `coinbase_merkle_proof` = valid values for that block
   - `confirmations = 0`
4. The contract returns `true`.
5. The depth check `(H - H) + 1 = 1 >= 0` passes trivially; the coinbase and transaction Merkle proofs are valid; no other guard rejects the call.
6. A consuming contract acting on this `true` result releases funds or executes a cross-chain action for a transaction that has zero confirmation depth and is fully reorg-able. [4](#0-3) [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
