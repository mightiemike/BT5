### Title
SpotEngine Cumulative Interest Multipliers Not Updated Before Balance Changes, Enabling Utilization Rate Manipulation - (File: `core/contracts/SpotEngineState.sol`, `core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.updateBalance` modifies `totalDepositsNormalized` and `totalBorrowsNormalized` without first calling `_updateState` to snapshot accrued interest at the current utilization ratio. Because `_updateState` is only invoked via the sequencer-submitted `SpotTick` transaction, any deposit or withdrawal between two ticks silently shifts the utilization ratio used to price the entire elapsed period. An unprivileged depositor can time a large withdrawal immediately before a predictable `SpotTick` to spike the utilization ratio, forcing all borrowers to pay an inflated interest rate for the full inter-tick window, then re-deposit after the tick.

---

### Finding Description

`SpotEngineState._updateState` computes the borrow rate as a function of `utilizationRatioX18 = totalBorrows / totalDeposits` and applies it over the full elapsed time `dt`:

```
borrowRateMultiplierX18 = (1 + borrowerRateX18)^dt
``` [1](#0-0) 

This function is only ever called from `updateStates`, which is only triggered by a `SpotTick` transaction submitted by the sequencer:

```solidity
spotEngine.updateStates(dt);   // EndpointTx.sol line 473
``` [2](#0-1) 

However, both overloads of `SpotEngine.updateBalance` directly mutate `totalDepositsNormalized` and `totalBorrowsNormalized` via `_updateBalanceNormalized` **without first calling `_updateState`**:

```solidity
_updateBalanceNormalized(state, balance, amountDelta);   // mutates totalDepositsNormalized / totalBorrowsNormalized
...
_setState(productId, state);   // persists the new totals, no _updateState call
``` [3](#0-2) [4](#0-3) 

`updateBalance` is called on every user-facing balance mutation: `depositCollateral`, `withdrawCollateral`, trade settlement, NLP mint/burn, fee claims, and `transferQuote`. [5](#0-4) [6](#0-5) 

The consequence: when `SpotTick` eventually fires with elapsed time `dt`, `_updateState` reads the **current** (potentially manipulated) utilization ratio and applies the resulting rate retroactively over the entire period `dt`, rather than the utilization ratio that was actually in effect throughout that window.

---

### Impact Explanation

**Passive (always present):** Every deposit or withdrawal between two `SpotTick` submissions causes the interest rate for the entire inter-tick period to be computed at the wrong utilization ratio. All borrowers and depositors continuously receive incorrect interest accrual.

**Active (attacker-controlled):** An attacker holding a large deposit can:

1. Withdraw the deposit immediately before a predictable `SpotTick` → `totalDepositsNormalized` drops → utilization spikes (e.g., from 50% to 95%).
2. `SpotTick` fires; `_updateState` applies the high-utilization borrow rate over the full `dt` to all borrowers.
3. Attacker re-deposits after the tick.

The attacker's remaining deposits (held in a second subaccount) earn the inflated deposit multiplier for the full period while the attacker bore no deposit risk during that window. Borrowers pay the inflated borrow rate for the full period despite the actual utilization being normal for most of it.

The inverse attack (deposit before tick to suppress utilization) lets a borrower pay near-zero interest for the full period.

---

### Likelihood Explanation

`SpotTick` is submitted by the sequencer at regular, predictable intervals (standard for a high-frequency DEX). Any user with a meaningful deposit balance can observe the tick cadence on-chain and time a withdrawal/re-deposit around it. No privileged access, governance capture, or sequencer compromise is required — the attacker only needs to submit standard `WithdrawCollateral` and `DepositCollateral` transactions, both of which are permissionless user entry points. [7](#0-6) [8](#0-7) 

---

### Recommendation

Before any mutation of `totalDepositsNormalized` or `totalBorrowsNormalized`, call `_updateState` with the elapsed time since the last update to snapshot accrued interest at the current utilization ratio. Concretely, both `updateBalance` overloads in `SpotEngine` should call `_updateState(productId, state, dt)` (where `dt` is derived from a stored last-update timestamp) before invoking `_updateBalanceNormalized`. This mirrors the `accrueInterest()` pattern used in Compound/Aave and ensures the interest rate for each sub-period reflects the utilization that was actually in effect during that sub-period. [9](#0-8) [10](#0-9) 

---

### Proof of Concept

**Setup:** QUOTE product has `totalDeposits = 1000`, `totalBorrows = 500` (50% utilization). `SpotTick` fires every 60 seconds. Alice has borrowed 500 QUOTE. Bob has deposited 500 QUOTE (the other 500 is from other depositors).

**Attack:**

1. Bob monitors the chain and sees the last `SpotTick` was 59 seconds ago.
2. Bob calls `WithdrawCollateral` for his 500 QUOTE deposit.
   - `totalDepositsNormalized` drops → effective `totalDeposits` = 500, `totalBorrows` = 500.
   - Utilization = 100% (maximum rate tier).
3. One second later, the sequencer submits `SpotTick` with `dt = 60`.
   - `_updateState` reads utilization = 100%, computes maximum borrow rate, applies it over `dt = 60` seconds.
   - Alice's borrow multiplier increases as if she paid 100% utilization rate for the full 60-second window, even though utilization was only 50% for 59 of those 60 seconds.
4. Bob re-deposits 500 QUOTE. His new `amountNormalized` is computed at the now-higher `cumulativeDepositsMultiplierX18`, so he earns no interest for the period — but the remaining depositors (who held through the tick) earned the inflated deposit rate for the full 60 seconds.     function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }

    function _updateState(
        uint32 productId,
        State memory state,
        uint128 dt
    ) internal {
        int128 borrowRateMultiplierX18;
        int128 totalDeposits = state.totalDepositsNormalized.mul(
            state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = state.totalBorrowsNormalized.mul(
            state.cumulativeBorrowsMultiplierX18
        );
        int128 utilizationRatioX18 = totalBorrows.div(totalDeposits);
        int128 minDepositRateX18;
        {
            Config memory config = configs[productId];

            // annualized borrower rate
            int128 borrowerRateX18 = config.interestFloorX18;
            if (utilizationRatioX18 == 0) {
                // setting borrowerRateX18 to 0 here has the property that
                // adding a product at the beginning of time and not using it until time T
                // results in the same state as adding the product at time T
                borrowerRateX18 = 0;
            } else if (utilizationRatioX18 < config.interestInflectionUtilX18) {
                borrowerRateX18 += config
                    .interestSmallCapX18
                    .mul(utilizationRatioX18)
                    .div(config.interestInflectionUtilX18);
            } else {
                borrowerRateX18 +=
                    config.interestSmallCapX18 +
                    config.interestLargeCapX18.mul(
                        (
                            (utilizationRatioX18 -
                                config.interestInflectionUtilX18).div(
                                    ONE - config.interestInflectionUtilX18
                                )
                        )
                    );
            }

            // convert to per second
            borrowerRateX18 = borrowerRateX18.div(
                MathSD21x18.fromInt(31536000)
            );
            borrowRateMultiplierX18 = (ONE + borrowerRateX18).pow(int128(dt));
            minDepositRateX18 = config.minDepositRateX18;
        }
