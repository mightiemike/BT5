Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the first parameter (the actual `addLiquidity` caller) and instead validates the position `owner` against the allowlist. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` permits any caller to supply an arbitrary `owner`, any unprivileged address can bypass the deposit allowlist by naming an already-approved address as the position owner. This renders the pool admin's primary LP-access control ineffective.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` is declared as:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

The first parameter — the actual caller of `addLiquidity` on the pool — is unnamed and discarded. The guard checks `owner` (the position beneficiary), not the depositing address.

`MetricOmmPool.addLiquidity` calls the hook as `_beforeAddLiquidity(msg.sender, owner, ...)`, where `msg.sender` is the router/caller and `owner` is the position owner supplied by that caller: [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (the explicit-owner overload) allows any caller to supply an arbitrary `owner`; `_validateOwner` only rejects `address(0)`: [3](#0-2) 

In the callback, tokens are pulled from `payer` (the original `msg.sender`, i.e., the attacker), while the position is recorded under the supplied `owner`: [4](#0-3) 

The inconsistency with `SwapAllowlistExtension.beforeSwap`, which correctly checks the first parameter (`sender`), confirms the bug: [5](#0-4) 

**Exploit path:**
1. Pool admin deploys pool with `DepositAllowlistExtension`; adds Alice to the allowlist via `setAllowedToDeposit(pool, alice, true)`.
2. Bob (not on the allowlist) calls `addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, data)`.
3. Pool calls `_beforeAddLiquidity(LiquidityAdder, alice, ...)`.
4. Extension evaluates `allowedDepositor[pool][alice]` → `true` → no revert.
5. Bob's tokens are pulled in the callback; position is recorded under Alice.
6. Bob has deposited into a restricted pool without being on the allowlist.

The wrong value is the extension decision: `allowedDepositor[pool][alice]` is checked instead of `allowedDepositor[pool][sender]` (where `sender` = `LiquidityAdder`, representing Bob's call).

## Impact Explanation

The deposit allowlist — the pool admin's mechanism for restricting LP participation to approved addresses — is fully bypassed. Any unprivileged address can deposit into a restricted pool by naming any approved address as `owner`. This is an admin-boundary break: a pool-admin access control is circumvented by an unprivileged path, violating the configured access boundary.

## Likelihood Explanation

Exploitation requires only a standard call to `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with `owner` set to any address already on the allowlist. No special privileges, flash loans, or oracle manipulation are needed. Any on-chain observer can read the allowlist state and identify an approved address to use as `owner`. The attack is repeatable and permissionless.

## Recommendation

Change `beforeAddLiquidity` to check the first parameter (the actual depositor/caller) instead of `owner`, mirroring `SwapAllowlistExtension.beforeSwap`:

```solidity
// Correct:
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension`; call `setAllowedToDeposit(pool, alice, true)`.
2. From Bob's address (not on allowlist), call `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, data)`.
3. Observe: pool calls `_beforeAddLiquidity(LiquidityAdder, alice, ...)` → extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
4. Bob's tokens are transferred to the pool in the callback; position is minted under Alice.
5. Assert: Bob successfully deposited into a pool he is not authorized to access; the allowlist is bypassed.
