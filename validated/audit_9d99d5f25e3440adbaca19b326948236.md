### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Transaction Forgery Protection — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a mandatory coinbase Merkle proof check to prevent the 64-byte transaction forgery attack. However, the deprecated function retains full `pub` visibility and is callable by any unprivileged NEAR account. Because Rust's `#[deprecated]` attribute only emits a compiler warning for Rust callers and has zero effect on external NEAR RPC callers, the security fix introduced in v2 is entirely bypassable.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented at https://www.bitmex.com/blog/64-Byte-Transactions). It enforces a coinbase proof check before delegating to the old function: [1](#0-0) 

The coinbase proof check in v2 requires the caller to prove knowledge of the real coinbase transaction at index 0, which an attacker crafting a fake 64-byte internal Merkle node cannot satisfy: [2](#0-1) 

The deprecated v1 function, however, skips this check entirely and only verifies that the supplied `merkle_proof` is non-empty before computing the root: [3](#0-2) 

The function is still declared `pub` with only a `#[pause]` guard: [4](#0-3) 

Rust's `#[deprecated]` attribute is a compile-time lint. It has no runtime effect and is completely invisible to external NEAR RPC callers who invoke the method by name over JSON or Borsh. Any account can call `verify_transaction_inclusion` directly, bypassing the coinbase proof requirement entirely.

---

### Impact Explanation

A recipient contract that consumes the boolean result of `verify_transaction_inclusion` to gate asset releases, bridge withdrawals, or other state transitions can be deceived into accepting a forged transaction inclusion proof. The 64-byte forgery attack allows an attacker to construct a 64-byte blob that is simultaneously a valid serialised internal Merkle tree node and a plausible "transaction hash", producing a proof that passes the v1 root check against a real block header without the transaction ever existing in that block. The corrupted proof result (`true`) is indistinguishable from a legitimate one to the consuming contract.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` by name via a standard NEAR function-call transaction. The function is not gated by any role check. The only guard (`#[pause]`) is a liveness control, not a security boundary. The attack requires crafting a valid 64-byte forgery, which is a known, documented technique with public tooling.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion` so it becomes a private helper callable only by `verify_transaction_inclusion_v2` internally. Alternatively, add an explicit `#[private]` attribute (NEAR SDK) or an access-control role guard that prevents direct external invocation. The `#[deprecated]` annotation alone provides no protection against external callers.

---

### Proof of Concept

1. Deploy the contract (Bitcoin feature, `skip_pow_verification = true`).
2. Submit a chain of headers and record a block hash `B` whose `merkle_root` is known.
3. Craft a 64-byte blob `F` such that `double_sha256(F || sibling)` equals `merkle_root` of block `B` (standard 64-byte Merkle forgery).
4. Call `verify_transaction_inclusion` directly (not v2) with `tx_id = F`, `tx_index = 0`, `merkle_proof = [sibling]`, `tx_block_blockhash = B`, `confirmations = 1`.
5. The function returns `true` despite `F` not being a real transaction in block `B`.
6. Calling `verify_transaction_inclusion_v2` with the same forged `tx_id` and no valid coinbase proof panics with `"Incorrect coinbase merkle proof"`, confirming the v2 guard works but is bypassable by calling v1 directly.

            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"

            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

            "Incorrect coinbase merkle proof"
        );
