### Title
Stale Sequencer-Provided Prices Used in Slow Mode Withdrawal Health Checks — (File: `core/contracts/Clearinghouse.sol`)

### Summary

Nado's product engines store prices that are exclusively updated by the protocol's own sequencer. Slow mode `WithdrawCollateral` transactions are executable by anyone after a 3-day timeout without sequencer involvement. The health check performed during execution uses these sequencer-provided prices with no staleness guard. When the sequencer is down or delayed, prices can be arbitrarily stale, allowing a user to pass a health check that would fail at current market prices.

---

### Finding Description

Prices used for health calculations are stored in each engine's risk store and are updated only when the sequencer submits an `UpdatePrice` transaction through the normal batch submission path: [1](#0-0) 

This calls `engine.updatePrice(...)` which writes to `_risk().value[productId].priceX18`: [2](#0-1) 

`Clearinghouse.getHealth()` consumes these engine-stored prices via `spotEngine.getHealthContribution` and `perpEngine.getHealthContribution`: [3](#0-2) 

`withdrawCollateral` enforces health using this same stale-price path: [4](#0-3) 

The slow mode path allows any user to submit a `WithdrawCollateral` transaction that is queued on-chain: [5](#0-4) 

After `SLOW_MODE_TX_DELAY` (3 days), **anyone** can call `executeSlowModeTransaction()` to process it without sequencer involvement: [6](#0-5) [7](#0-6) 

The slow mode execution path for `WithdrawCollateral` directly calls `clearinghouse.withdrawCollateral(...)`: [8](#0-7) 

`Endpoint.getPriceX18()` — the only price accessor — performs no staleness check, only a non-zero guard: [9](#0-8) 

Similarly, `getTime()` tracks the last sequencer-submitted tick time but is never compared against `block.timestamp` to detect staleness: [10](#0-9) 

---

### Impact Explanation

If the sequencer is down for ≥ 3 days and collateral prices have dropped materially during that window, a user can execute a queued slow mode withdrawal whose health check passes against the last sequencer-submitted (stale, inflated) price. The withdrawal succeeds even though the account is undercollateralized at current market prices. This leaves the protocol holding a position that cannot be fully liquidated, directly corrupting the solvency invariant of the insurance fund and the clearinghouse.

Concrete state delta: `spotEngine` balance for `sender` is decremented by `amountRealized`; the corresponding token is transferred out via `WithdrawPool`; but `getHealth()` returns a non-negative value only because it used a stale price — at true market price the health would be negative.

---

### Likelihood Explanation

The Nado protocol is deployed on Ink Chain, a relatively new L2. Sequencer downtime events of hours to days are historically observed on L2 networks. The 3-day slow mode delay means a sequencer outage of that duration is required, which is uncommon but not implausible. The attacker does not need to cause the outage — they only need to observe it and have a pre-queued slow mode withdrawal. The attack is permissionless: `executeSlowModeTransaction()` has no caller restriction.

---

### Recommendation

Before executing a slow mode `WithdrawCollateral`, compare the last sequencer-submitted tick time against `block.timestamp`. If the gap exceeds a defined staleness threshold (e.g., 1 hour), revert rather than proceeding with potentially stale prices. Concretely, `getTime()` already returns the maximum of `spotTime` and `perpTime`; a guard such as:

```solidity
require(block.timestamp - getOracleTime() <= MAX_PRICE_STALENESS, ERR_STALE_PRICE);
```

inserted at the start of `withdrawCollateral` (or within `getHealth`) would prevent health checks from executing against stale sequencer data.

---

### Proof of Concept

1. Sequencer is live. User has 100 USDC of ETH collateral (ETH = $2000) and 80 USDC of quote debt. Health is positive.
2. User calls `submitSlowModeTransaction(WithdrawCollateral{sender, ETH_PRODUCT_ID, 10 USDC})`. A slow mode fee is charged; the tx is queued with `executableAt = block.timestamp + 3 days`.
3. Sequencer goes offline. No new `UpdatePrice` transactions are submitted. Engine-stored ETH price remains $2000.
4. ETH drops to $1200 on-chain. At true prices, user health = $1200 × weight − $80 < 0. Withdrawal should be blocked.
5. After 3 days, attacker (or user) calls `Endpoint.executeSlowModeTransaction()`.
6. `processSlowModeTransactionImpl` routes to `clearinghouse.withdrawCollateral(sender, ETH_PRODUCT_ID, 10 USDC, ...)`.
7. `getHealth()` computes health using stale engine price $2000: health = $2000 × weight − $80 − $10 > 0. Check passes.
8. 10 USDC worth of ETH is transferred out via `WithdrawPool`. The account is now undercollateralized at true market prices, and the protocol bears the shortfall.

```text
    function getHealth(bytes32 subaccount, IProductEngine.HealthType healthType)
        public
        returns (int128 health)
    {
        ISpotEngine spotEngine = _spotEngine();
        IPerpEngine perpEngine = _perpEngine();

        health = spotEngine.getHealthContribution(subaccount, healthType);
        // min health means that it is attempting to borrow a spot that exists outside
        // of the risk system -- return min health to error out this action
        if (health == -INF) {
            return health;
        }
        health += perpEngine.getHealthContribution(subaccount, healthType);
```

```text
    function updatePrice(bytes calldata transaction)
        external
        onlyEndpoint
        returns (uint32, int128)
    {
        IEndpoint.UpdatePrice memory txn = abi.decode(
            transaction[1:],
            (IEndpoint.UpdatePrice)
        );
        require(txn.priceX18 > 0, ERR_INVALID_PRICE);
        IProductEngine engine = productToEngine[txn.productId];
        if (address(engine) != address(0)) {
            engine.updatePrice(txn.productId, txn.priceX18);
            return (txn.productId, txn.priceX18);
        } else {
            return (0, 0);
        }
    }
```
