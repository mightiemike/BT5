### Title
`calculate_next_l2_gas_price_for_fin` and `calculate_next_base_gas_price` Always Bind `gas_target` and `gas_price_max_change_denominator` from `latest_constants()` Instead of the Block's Protocol-Version Constants, Producing a Wrong Committed Next-Block L2 Gas Price — (`crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

---

### Summary

`calculate_next_l2_gas_price_for_fin` reads `gas_target` unconditionally from `VersionedConstants::latest_constants()`, and `calculate_next_base_gas_price` reads both `max_block_size` and `gas_price_max_change_denominator` from the same source. The orchestrator's `VersionedConstants` type has five distinct versioned JSON files (V0_14_0 through V0_14_4), each of which can carry different values for these fields. Because neither function accepts nor consults the current block's `StarknetVersion`, the EIP-1559 adjustment always uses the *latest* constants even when the network is running at an older protocol version. The resulting `next_l2_gas_price` is committed to the block header with the wrong value, causing incorrect fee accounting for every transaction in the following block.

---

### Finding Description

`calculate_next_l2_gas_price_for_fin` accepts a `height: BlockNumber` parameter but uses it only to look up the operator-configured minimum gas price:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs  line 70
let gas_target = VersionedConstants::latest_constants().gas_target;
``` [1](#0-0) 

The `gas_target` value is then forwarded to `calculate_next_base_gas_price`, which independently calls `latest_constants()` again to obtain `max_block_size` (used in a correctness assertion) and `gas_price_max_change_denominator` (used directly in the price-change formula):

```rust
// line 92
let versioned_constants = VersionedConstants::latest_constants();
// ...
let denominator =
    gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
let price_change = (price_u256 * gas_delta) / denominator;
``` [2](#0-1) 

The orchestrator's `VersionedConstants` is defined with five distinct versioned JSON files:

```rust
define_versioned_constants!(
    VersionedConstants,
    VersionedConstantsError,
    StarknetVersion::V0_14_0,
    "resources/versioned_constants_diff_regression",
    (V0_14_0, "../resources/orchestrator_versioned_constants_0_14_0.json"),
    ...
    (V0_14_4, "../resources/orchestrator_versioned_constants_0_14_4.json"),
);
``` [3](#0-2) 

The struct fields `gas_price_max_change_denominator`, `max_block_size`, and `gas_target` are all first-class versioned fields: [4](#0-3) 

The `VersionedConstantsTrait::get()` method exists precisely to retrieve version-specific constants: [5](#0-4) 

Neither `calculate_next_l2_gas_price_for_fin` nor `calculate_next_base_gas_price` accepts a `StarknetVersion` argument, so neither can call `VersionedConstants::get(version)`. The Starknet version of the block being finalized is simply discarded at the call site.

The analog to the external report is exact: just as `_baseVestedAmount()` returns `_claim.amountWithdrawn` (a cumulative present-state value) for historical timestamps instead of the time-parameterized `vestAmt`, these functions return constants from the *latest* protocol version for a block that belongs to an *older* protocol version, contaminating the version-parameterized result with current state.

---

### Impact Explanation

The wrong `next_l2_gas_price` is written into the block header and propagated through consensus. Because all sequencer nodes run the same binary, they all compute the same wrong value, so consensus is not broken. However, the committed gas price deviates from what the protocol specifies for the current version. Every transaction in the subsequent block has its fee computed against the wrong price, causing over- or under-charging of users and incorrect balance effects. This matches **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The divergence is triggered whenever the sequencer binary is updated to a newer version (e.g., V0_14_4) while the network is still running at an older protocol version (e.g., V0_14_0 or V0_14_2). This is the normal state during any rolling protocol upgrade. With five distinct versioned-constants files already present, the probability that `gas_target` or `gas_price_max_change_denominator` differs across at least one version boundary is high. No privileged action is required; the bug fires automatically on every block finalization call after a binary upgrade.

---

### Recommendation

Pass the current block's `StarknetVersion` into both functions and replace every `VersionedConstants::latest_constants()` call with `VersionedConstants::get(&starknet_version)?`:

```rust
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    starknet_version: &StarknetVersion,   // <-- add
    l2_gas_used: GasAmount,
    ...
) -> GasPrice {
    ...
    let vc = VersionedConstants::get(starknet_version)
        .expect("block starknet version must be known");
    let gas_target = vc.gas_target;
    ...
    calculate_next_base_gas_price(
        current_l2_gas_price, l2_gas_used, gas_target, effective_min,
        starknet_version,   // <-- propagate
    )
}

### Proof of Concept

1. Network is running at protocol version V0_14_0. The sequencer binary is updated to V0_14_4 (which has a different `gas_target` or `gas_price_max_change_denominator`).
2. Block N is finalized. `calculate_next_l2_gas_price_for_fin` is called with `height = N`.
3. Line 70 executes: `gas_target = VersionedConstants::latest_constants().gas_target` — this returns V0_14_4's value, not V0_14_0's.
4. Line 92 executes: `versioned_constants = VersionedConstants::latest_constants()` — `gas_price_max_change_denominator` is again V0_14_4's value.
5. The denominator in the EIP-1559 formula is `gas_target_v0_14_4 * gas_price_max_change_denominator_v0_14_4` instead of the V0_14_0 values.
6. The resulting `next_l2_gas_price` differs from the protocol-correct value for V0_14_0.
7. This wrong price is committed to block N's header as `next_l2_gas_price` and used to charge fees for all transactions in block N+1. [6](#0-5) [7](#0-6)

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}

pub struct VersionedConstants {
    ///  This is used to calculate the base gas price for the next block according to EIP-1559 and
    /// serves as a sensitivity parameter that limits the maximum rate of change of the gas price
    /// between consecutive blocks.
    pub gas_price_max_change_denominator: u128,
    /// The minimum gas price in fri.
    pub min_gas_price: GasPrice,
    /// The maximum block size in gas units.
    // NOTE: Must stay in sync with BouncerWeights receipt_l2_gas.
    // NOTE: When max_block_size is changed, update `gas_target` accordingly to maintain the ratio.
    pub max_block_size: GasAmount,
    /// The target gas usage per block. Used by EIP-1559 to calculate the next block's gas price.
    // Target is 60% of max_block_size, making price adjustment more responsive to congestion.
    pub gas_target: GasAmount,
    /// The margin for the eth to fri rate disagreement, expressed as a percentage (parts per
    /// hundred).
    pub l1_gas_price_margin_percent: u32,
    /// Number of `fee_proposal` values used to compute `fee_actual` (sliding window).
    pub fee_proposal_window_size: u64,
    /// Maximum `fee_proposal` change per block in parts per thousand (e.g., `2` = 0.2%).
    pub fee_proposal_margin_ppt: u128,
}
