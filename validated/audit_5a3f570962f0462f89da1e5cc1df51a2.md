### Title
Unrestricted `creditDeposit()` Allows Any Caller to Force Deposits into DDA Subaccount — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1::creditDeposit()` has no access control. Any unprivileged caller can invoke it at any time to sweep all token balances held by the DDA contract into the hardcoded `subaccount`. A third party can front-run the DDA owner's intended `withdraw()` call by calling `creditDeposit()` first, forcing the owner's tokens into the protocol and subjecting them to a mandatory 3-day slow-mode withdrawal delay and withdrawal fees.

---

### Finding Description

`DirectDepositV1` is a deposit-address contract that holds a fixed `subaccount` set at construction. It exposes two relevant external functions:

```solidity
// core/contracts/DirectDepositV1.sol L83-101
function creditDeposit() external {          // ← no access control
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(
                subaccount, productId, uint128(balance), "-1"
            );
        }
    }
}
``` [1](#0-0) 

```solidity
// core/contracts/DirectDepositV1.sol L103-106
function withdraw(IIERC20Base token) external onlyOwner {   // ← owner-only
    uint256 balance = token.balanceOf(address(this));
    safeTransfer(token, msg.sender, balance);
}
``` [2](#0-1) 

The asymmetry is the root cause: `withdraw()` is `onlyOwner`, but `creditDeposit()` is unrestricted. The DDA owner may legitimately hold tokens in the DDA temporarily before deciding whether to deposit them or withdraw them directly. Because `creditDeposit()` is callable by anyone, an attacker can preempt the owner's `withdraw()` call by calling `creditDeposit()` first, atomically sweeping all DDA token balances into the protocol.

Once deposited, the tokens are no longer accessible via `withdraw()`. The owner must submit a `WithdrawCollateral` slow-mode transaction, which is subject to a hardcoded 3-day delay:

```solidity
// core/contracts/EndpointTx.sol L376-380
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
    sender: sender,
    tx: transaction
});
``` [3](#0-2) 

Additionally, the withdrawal path charges a `withdrawFeeX18` per product:

```solidity
// core/contracts/EndpointTx.sol L425-429
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
``` [4](#0-3) 

---

### Impact Explanation

The DDA owner loses discretion over when and whether to deposit tokens held in the DDA. A third party can:

1. Send a dust amount of any supported token to the DDA address (which is deterministic and publicly known).
2. Call `creditDeposit()` to sweep all DDA token balances — including any the owner intended to withdraw — into the protocol.
3. The owner's tokens are now locked in the protocol for a minimum of 3 days (slow-mode delay) and subject to per-product withdrawal fees.

This enables griefing / economic DoS: an attacker can repeatedly force deposits before each owner withdrawal attempt, continuously resetting the 3-day lock and extracting withdrawal fees from the owner.

---

### Likelihood Explanation

- `creditDeposit()` is a public, permissionless function — any EOA or contract can call it.
- The DDA address is deterministic (deployed via constructor with known parameters) and its purpose is publicly documented, making it a known target.
- The attacker's cost is only gas plus a dust token transfer; no protocol-level privilege is required.
- The attack is repeatable with no cooldown.

---

### Recommendation

Add `onlyOwner` to `creditDeposit()`, consistent with the access control already applied to `withdraw()`:

```solidity
function creditDeposit() external onlyOwner {
    ...
}
``` [5](#0-4) 

This ensures only the DDA owner can decide when to sweep balances into the protocol, preserving their discretion over deposit timing and preventing griefing via forced deposits.

---

### Proof of Concept

1. DDA owner holds 1000 USDC in the DDA, intending to call `withdraw(USDC)` to retrieve them freely.
2. Attacker observes the pending `withdraw()` call in the mempool (or simply monitors the DDA balance).
3. Attacker front-runs by calling `creditDeposit()`.
4. All 1000 USDC are deposited into the protocol under the DDA's `subaccount`.
5. Owner's `withdraw()` call now finds zero balance in the DDA and fails to recover anything.
6. Owner must submit a slow-mode `WithdrawCollateral` transaction, wait 3 days, and pay `withdrawFeeX18` to recover their 1000 USDC.
7. Attacker can repeat this every time the owner accumulates a balance in the DDA. [1](#0-0)
