### Title
Deposited Tokens Permanently Locked When Slow-Mode `DepositCollateral` Execution Fails Silently — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.sol` transfers user tokens to itself during `depositCollateralWithReferral`, then queues a slow-mode `DepositCollateral` transaction for later execution. The execution path wraps `processSlowModeTransaction` in a `try/catch` that silently discards any revert. If the inner call fails, the queued transaction is permanently consumed from the queue while the user's tokens remain locked in the Endpoint contract with no recovery path. The inline comment `// try return funds now removed` at line 226 is an explicit acknowledgment that a prior fund-return mechanism was deliberately removed, leaving no remediation path.

---

### Finding Description

The deposit flow proceeds in two decoupled steps:

**Step 1 — Token transfer (atomic, immediate):**
`depositCollateralWithReferral` transfers tokens from `msg.sender` to the Endpoint contract via `handleDepositTransfer`, then enqueues a `SlowModeTx` of type `DepositCollateral`. [1](#0-0) 

**Step 2 — Balance credit (deferred, fallible):**
When the slow-mode transaction is later executed, `_executeSlowModeTransaction` deletes the entry from the queue *before* attempting execution, then calls `processSlowModeTransaction` inside a `try/catch`. Any revert is silently swallowed: [2](#0-1) 

The critical lines are:

- **Line 194**: `delete slowModeTxs[_slowModeConfig.txUpTo++]` — the transaction is irrevocably removed from the queue before execution succeeds.
- **Lines 207–227**: The `try/catch` catches all reverts from `processSlowModeTransaction` and does nothing.
- **Line 226**: `// try return funds now removed` — the prior recovery mechanism was explicitly removed, leaving no fallback. [3](#0-2) 

**Concrete failure trigger in `processSlowModeTransactionImpl`:**

For a `DepositCollateral` slow-mode transaction, execution reaches `clearinghouse.depositCollateral(txn)`: [4](#0-3) 

Inside `Clearinghouse.depositCollateral`, the call to `_decimals(txn.productId)` will revert with `ERR_INVALID_PRODUCT` if the product's token address has been zeroed out or the product has been removed from the SpotEngine between the time of deposit and the time of execution (the slow-mode delay is hardcoded to 3 days): [5](#0-4) [6](#0-5) 

The 3-day window between deposit and execution is the relay window during which the desynchronization can occur: [7](#0-6) 

---

### Impact Explanation

**Impact: High.**

When `processSlowModeTransaction` reverts for a `DepositCollateral` slow-mode tx:

- The user's ERC-20 tokens have already been transferred to the Endpoint contract (`handleDepositTransfer` succeeded).
- The slow-mode queue entry has already been deleted (`delete slowModeTxs[...]` at line 194).
- No balance is credited in the SpotEngine.
- No refund is issued (the fund-return mechanism was removed).

The user's tokens are permanently locked in the Endpoint contract. There is no admin function, no retry path, and no rescue mechanism exposed to the user. The corrupted state is: `token.balanceOf(endpoint) > 0` for the deposited amount, with zero corresponding `spotEngine` balance for the depositor's subaccount.

---

### Likelihood Explanation

**Likelihood: Low.**

The trigger requires that `clearinghouse.depositCollateral` reverts during the 3-day slow-mode execution window. This can occur if the product's configuration is altered (token address zeroed, product removed from the engine) between deposit and execution. While this requires a protocol-level configuration change, the 3-day window is a meaningful exposure period. The structural vulnerability — silent failure with no recovery — is unconditional and affects every `DepositCollateral` slow-mode transaction regardless of trigger.

---

### Recommendation

Restore the fund-return mechanism that was previously removed. When `processSlowModeTransaction` fails for a `DepositCollateral` slow-mode transaction, the Endpoint should identify the deposited token and amount from the queued transaction bytes and return them to the original depositor. Alternatively, the token transfer should be deferred until after successful execution of the slow-mode transaction, eliminating the two-step desynchronization entirely.

---

### Proof of Concept

1. User calls `depositCollateral(subaccountName, productId, amount)` on `Endpoint.sol`.
2. `depositCollateralWithReferral` passes all pre-checks, calls `handleDepositTransfer` — tokens leave the user's wallet and enter the Endpoint contract.
3. A `SlowModeTx` of type `DepositCollateral` is enqueued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY`.
4. During the 3-day window, the product's token configuration is altered such that `_spotEngine().getConfig(productId).token == address(0)`.
5. `executeSlowModeTransaction()` is called (by user or sequencer).
6. `_executeSlowModeTransaction` deletes the queue entry at line 194, then calls `this.processSlowModeTransaction(...)` inside `try`.
7. Inside `processSlowModeTransactionImpl`, `clearinghouse.depositCollateral(txn)` calls `_decimals(txn.productId)`, which calls `_tokenAddress(productId)` returning `address(0)`, triggering `require(address(token) != address(0), ERR_INVALID_PRODUCT)` — revert.
8. The `catch` block at line 207 executes. The only code path is `// try return funds now removed` — nothing happens.
9. The user's tokens remain in the Endpoint contract. The queue entry is gone. No balance was credited. Funds are permanently lost. [8](#0-7) [9](#0-8) [10](#0-9)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }

    function _tokenAddress(uint32 productId) internal view returns (address) {
        return _spotEngine().getConfig(productId).token;
    }

    function _decimals(uint32 productId) internal virtual returns (uint8) {
        IERC20Base token = IERC20Base(_tokenAddress(productId));
        require(address(token) != address(0), ERR_INVALID_PRODUCT);
        return token.decimals();
    }

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
