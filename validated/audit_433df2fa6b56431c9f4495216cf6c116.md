### Title
4-Byte Secp Opcode Dispatch Bypasses `ENABLE_SECP_OPS` Flag Gate — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op()` function in `src/chia_dialect.rs` dispatches secp256k1 and secp256r1 verification through two separate code paths: a 4-byte opcode path and a 1-byte opcode path. The `ENABLE_SECP_OPS` flag gate was wired only into the 1-byte path (opcodes 64/65) but was never applied to the 4-byte path (opcodes `0x13d61f00` / `0x1c3a8f00`). An attacker supplying CLVM bytes that use the 4-byte encoding executes secp verification unconditionally, regardless of whether `ENABLE_SECP_OPS` is set.

---

### Finding Description

`ChiaDialect::op()` contains two distinct dispatch branches for secp operations.

**4-byte path — no flag check (lines 157–183):**

```rust
if op_len == 4 {
    let b = allocator.atom(o);
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => {
            return unknown_operator(allocator, o, argument_list, flags, max_cost);
        }
    };
    return f(allocator, argument_list, max_cost, flags);
}
``` [1](#0-0) 

**1-byte path — flag-gated (lines 248–249):**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

The flag was wired into the 1-byte dispatch but the 4-byte dispatch was never updated. The `MEMPOOL_MODE` constant does not include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

---

### Impact Explanation

Under `MEMPOOL_MODE` (or any configuration without `ENABLE_SECP_OPS`):

- A CLVM program using **1-byte opcode 64** falls through to `unknown_operator()`. Because `NO_UNKNOWN_OPS` is set in mempool mode, this returns `Err(EvalErr::Unimplemented(o))` — the program is **rejected**.
- A CLVM program using **4-byte opcode `0x13d61f00`** executes `op_secp256k1_verify` directly — the program is **accepted and the secp verification runs**.

The two encodings are semantically identical operations but receive opposite treatment under the same flag configuration. This is a concrete mempool/consensus divergence: a transaction spending a coin whose puzzle uses 4-byte secp opcodes is accepted by the mempool and validated by consensus, while an equivalent puzzle using 1-byte opcodes is rejected. Nodes or tooling that evaluate programs with different flag sets will disagree on validity.

The corrupted invariant is: *"secp operations are only available when `ENABLE_SECP_OPS` is set."* The 4-byte path silently violates this invariant for any caller that does not set the flag.

---

### Likelihood Explanation

The 4-byte secp opcodes (`0x13d61f00`, `0x1c3a8f00`) are documented in the cost formula comment and are referenced in the fuzzing harness (`clvm-fuzzing/src/make_tree.rs`) and the benchmark tool (`tools/src/bin/benchmark-clvm-cost.rs`), confirming they are reachable production paths. Any attacker who can submit a CLVM program — the standard entry point for Chia coin puzzles — can choose the 4-byte encoding. The encoding difference is a trivial substitution in the serialized CLVM bytes.

---

### Recommendation

Add the `ENABLE_SECP_OPS` flag check to the 4-byte dispatch path, mirroring the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

If the 4-byte opcodes are intended to be permanently available regardless of the flag (i.e., they predate the flag and are hardforked in), then the `ENABLE_SECP_OPS` flag documentation and the 1-byte gate should be revised to make the asymmetry explicit and intentional, and the flag's scope should be clearly documented as applying only to the 1-byte aliases.

---

### Proof of Concept

1. Construct a CLVM program that calls `op_secp256k1_verify` using the 4-byte opcode `0x13d61f00` with a valid pubkey, message digest, and signature.
2. Run it under `ChiaDialect::new(MEMPOOL_MODE)` (which excludes `ENABLE_SECP_OPS`).
3. Observe: the 4-byte path at line 176 matches, `op_secp256k1_verify` executes, and the program succeeds.
4. Repeat with the 1-byte opcode 64 under the same flags.
5. Observe: the match arm at line 248 does not fire (guard fails), execution falls to `unknown_operator()`, and `NO_UNKNOWN_OPS` causes `Err(EvalErr::Unimplemented)`.

The two programs — identical in semantics, differing only in opcode encoding — produce opposite results under the same dialect configuration, confirming the flag wiring gap. [5](#0-4) [6](#0-5)
