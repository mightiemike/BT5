Audit Report

## Title
`SwapAllowlistExtension` gates the router address instead of the end-user, allowing any unprivileged swapper to bypass the per-pool allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to its own `msg.sender` — the immediate caller of `pool.swap`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the end-user. If the router is allowlisted (the only way to enable router-based swaps), every user — including those the pool admin explicitly excluded — can bypass the guard by routing through the router. There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

## Finding Description

**Root cause — `beforeSwap` identity check** (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`, L37):

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

- `msg.sender` is the pool (the contract that called the extension).
- `sender` is the first argument forwarded by the pool, which is the pool's own `msg.sender`.

**Pool passes its own `msg.sender` as `sender`** (`metric-core/contracts/MetricOmmPool.sol`, L230-231):

```solidity
_beforeSwap(
    msg.sender,   // whoever called pool.swap becomes "sender" in the extension
    ...
);
```

**Router calls `pool.swap` directly** (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, L71-80):

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The router is the `msg.sender` of `pool.swap`, so the pool passes the router address as `sender` to the extension. The extension then checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`.

**Exploit call chain:**
```
end-user → router.exactInputSingle(...)
         → pool.swap(...)          [pool's msg.sender = router]
         → extension.beforeSwap(router, ...)
         → allowedSwapper[pool][router] == true  ← checked, NOT the end-user
```

The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` in `MetricOmmSimpleRouter`.

**Existing guards are insufficient**: The extension has no mechanism to distinguish the originating user from the intermediary router. The `extensionData` field passed by the router is `""` (empty string) for single-hop swaps, and user-supplied for multi-hop, so it cannot be trusted as an identity channel without additional verification.

## Impact Explanation

Any user excluded from the per-pool swap allowlist can bypass the guard by calling `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput`. The pool executes the swap at oracle-derived prices, so the attacker receives real tokens out of the pool. This is a direct admin-boundary break: an admin-configured guard is circumvented by an unprivileged path (the public router). If the allowlist was deployed to enforce regulatory compliance, risk limits, or LP-protection on a restricted pool, the bypass nullifies those controls entirely.

## Likelihood Explanation

The router is the primary user-facing interface deployed alongside the pool. A pool admin who wants allowlisted users to trade conveniently will allowlist the router — the only way to enable router-based swaps. Once the router is allowlisted, the bypass is trivially reachable by any address with no special privileges. The flaw is non-obvious because the extension's NatSpec states it "Gates `swap` by swapper address," which pool admins reasonably interpret as the end-user, not the intermediary contract.

## Recommendation

1. **Pass the originating user through a verified channel**: The router should encode the original `msg.sender` in `extensionData`, and the extension should verify it only when the pool's immediate caller is a known, trusted router that itself enforces the forwarded identity. This requires a registry of trusted routers in the extension.

2. **Alternatively, gate on `tx.origin`** inside the extension (acceptable only if the threat model excludes contract-based callers, which is a separate design decision).

3. **Document the limitation clearly**: If neither fix is applied, the extension's NatSpec must state that it gates the immediate caller of `pool.swap`, not the end-user, and that allowlisting the router opens the pool to all users.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true)    // to enable router swaps
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  - attacker calls router.exactInputSingle({pool: pool, ...})
  - router calls pool.swap(recipient, ...)
  - pool calls extension.beforeSwap(router, ...)
  - extension checks allowedSwapper[pool][router] == true  → passes
  - swap executes; attacker receives tokens from the restricted pool

Expected: revert NotAllowedToSwap
Actual:   swap succeeds
