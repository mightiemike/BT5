### Title
Missing Minimum-Amount-Out Guard in `removeLiquidity` Leaves LPs Exposed to Frontrunning Losses — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.removeLiquidity` accepts no `minAmount0Out` / `minAmount1Out` parameter and no cursor-bounds guard. A frontrunner can swap immediately before an LP's removal transaction, shifting the bin cursor and draining one token side of the active bin, so the LP receives a worse token mix than expected with no on-chain protection.

---

### Finding Description

`removeLiquidity` in `MetricOmmPool.sol` has the following signature:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

There is no `minAmount0Out`, `minAmount1Out`, or cursor-bounds argument. The function calls `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` extension hooks, but neither hook receives any user-supplied minimum threshold — the extension system has no standardised channel for the LP to express a slippage limit. [2](#0-1) 

By contrast, the **add-liquidity** path in the periphery (`addLiquidityWeighted`) explicitly protects the depositor with:

1. `maxAmountToken0` / `maxAmountToken1` caps enforced in the callback, and
2. `_validateBinAndBinPosition`, which reads the live cursor and reverts if it has been pushed outside the caller-specified `[minimalCurBin, maximalCurBin]` × `[minimalPosition, maximalPosition]` window. [3](#0-2) [4](#0-3) 

No equivalent periphery wrapper exists for `removeLiquidity`; LPs must call the pool directly and receive whatever the current bin state dictates.

The amount returned by `LiquidityLib.removeLiquidity` is a function of the live `curBinIdx`, `curPosInBin`, and per-bin token balances — all of which a frontrunner can shift with a single swap before the LP's transaction lands. [5](#0-4) 

---

### Impact Explanation

When a frontrunner swaps to move the active bin cursor (or to drain one token side of the current bin) before the LP's `removeLiquidity` call:

- The LP's shares are redeemed against a shifted bin state.
- The LP receives a worse token composition (e.g., more of the depreciating token, less of the appreciating token) than the price at the time they submitted the transaction.
- Because the pool is oracle-anchored and the spread fee bounds price movement, the maximum per-removal loss is bounded by the spread, but that bound can still represent a material percentage of principal (analogous to the 2 % cap in the HydraDX report).
- The loss is a direct reduction of LP principal with no recourse; the extension hooks (`_beforeRemoveLiquidity` / `_afterRemoveLiquidity`) fire but carry no user-supplied floor to revert on.

This satisfies the allowed-impact gate: **direct loss of user principal / owed LP assets**.

---

### Likelihood Explanation

- Any MEV searcher running a mempool monitor can detect a pending `removeLiquidity` call and sandwich it with a swap in the same block.
- The attack requires no special privilege — `swap` is permissionless and `removeLiquidity` enforces only `msg.sender == owner`.
- EVM chains targeted by Metric (Ethereum Mainnet, Base, HyperEVM) all have active MEV infrastructure capable of executing this sandwich.

---

### Recommendation

1. **Add slippage parameters to `removeLiquidity`**: introduce `uint256 minAmount0Out` and `uint256 minAmount1Out` and revert if the returned amounts fall below them.

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,   // <-- add
    uint256 minAmount1Out,   // <-- add
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
{
    ...
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
        revert InsufficientOutput(amount0Removed, amount1Removed);
    ...
}
```

2. **Add a periphery `removeLiquidity` wrapper** (analogous to `MetricOmmPoolLiquidityAdder`) that includes cursor-bounds validation before the removal, mirroring the protection already present for `addLiquidityWeighted`. [4](#0-3) 

---

### Proof of Concept

```
Setup
─────
Pool: token0 / token1, oracle mid-price = P, one active bin [P·(1-δ), P·(1+δ)]
LP deposits shares into the active bin.

Attack sequence (single block, three transactions ordered by MEV bot)
──────────────────────────────────────────────────────────────────────
Tx 1 (attacker): swap a large amount of token1 → token0
  Effect: cursor moves toward the upper edge of the bin; bin's token0 balance
          is partially drained, token1 balance increases.

Tx 2 (victim):   removeLiquidity(owner, salt, deltas, extensionData)
  Effect: LP redeems shares against the shifted bin state.
          Receives less token0 and more token1 than at the pre-attack price.
          No minAmount0Out check → transaction succeeds.

Tx 3 (attacker): swap token0 → token1 to restore cursor
  Effect: attacker profits from the round-trip spread; LP has permanently
          received a worse token mix.

Outcome
───────
LP started with X token0 worth of shares.
LP ends with X·(1 - slippage%) token0-equivalent value.
Slippage is bounded by the pool's spread fee but can reach several percent
for pools with wide spreads or large bin lengths, with no on-chain floor
to protect the LP.
```

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
