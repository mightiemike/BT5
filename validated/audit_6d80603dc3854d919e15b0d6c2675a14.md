### Title
SwapAllowlistExtension Gates the Router's Address Instead of the Actual Swapper When Trades Are Routed Through MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` which the pool sets to its own `msg.sender` — the immediate caller of `pool.swap()`. When a user routes through the public `MetricOmmSimpleRouter`, the router is `msg.sender` to the pool, so the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. The guard therefore evaluates an incomplete identity — the intermediary contract — while ignoring the economically relevant actor. This is the direct analog to the external bug where `canHedge()` evaluated only the current position and ignored the pending request, producing a wrong guard decision.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` is the per-pool swap gate:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct — it is the extension caller). `sender` is the address the pool passes as the first argument, which is the pool's own `msg.sender` — whoever called `pool.swap()`.

The `IMetricOmmPoolActions` interface confirms this binding:

> `NotAllowedToSwap` — "Swap allowlist rejected `msg.sender`."

When `MetricOmmSimpleRouter` calls `pool.swap()` on behalf of a user, the pool's `msg.sender` is the router contract. The pool therefore passes `sender = router` to `beforeSwap`. The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

The `DepositAllowlistExtension` does **not** share this flaw: `addLiquidity` accepts an explicit `owner` parameter (operator pattern), so the liquidity adder can pass the real user as `owner` and the extension correctly gates that address. The swap path has no equivalent explicit-sender parameter — it is always `msg.sender` of the `pool.swap()` call.

---

### Impact Explanation

Two fund-impacting outcomes follow directly:

**Path A — Allowlist bypass (High):**
A pool admin allowlists the router address (a natural configuration: "allow router-mediated swaps for my curated pool"). Any unpermissioned user can now bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`. The extension sees `sender = router`, which is allowlisted, and passes. The actual user's address is never checked. Non-KYC'd or otherwise excluded users trade freely on a curated pool, directly violating the pool's access policy and exposing LPs to unauthorized counterparties.

**Path B — Broken core functionality for allowlisted users (Medium/High):**
If the admin does not allowlist the router, every allowlisted user who routes through `MetricOmmSimpleRouter` has their swap rejected with `NotAllowedToSwap`, even though they are individually permitted. The router is a supported periphery path; blocking it breaks the core swap flow for legitimate users.

Both outcomes are reachable by any public user with no privileged access.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any user can call it.
- Pool admins configuring a swap allowlist will naturally want to support router-mediated swaps (multi-hop, slippage wrappers, etc.), making Path A a realistic misconfiguration.
- Even without router allowlisting, Path B is triggered the moment any allowlisted user attempts a router swap, which is a normal usage pattern.
- No special tokens, malicious setup, or privileged roles are required.

---

### Recommendation

The pool should pass the **originating user's address** as `sender` to the extension, not the immediate `msg.sender`. Two complementary fixes:

1. **Router-side**: `MetricOmmSimpleRouter` should pass the original `msg.sender` (the user) as an explicit `sender` argument to `pool.swap()`, and the pool interface should accept and forward it to extensions — analogous to how `addLiquidity` separates `owner` from `msg.sender`.

2. **Extension-side (interim)**: `SwapAllowlistExtension.beforeSwap` should accept and check an `extensionData`-encoded originator address when the immediate `sender` is a known router, or the pool should expose a `sender` override parameter in its swap interface.

The root fix is at the pool/router interface level: the swap path must carry the economic actor's identity through to the extension, just as the liquidity path does via the explicit `owner` parameter.

### Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, router, true)   // intends to allow router-mediated swaps
  admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls MetricOmmSimpleRouter.exactInput(pool, ...)
  router calls pool.swap(...)  →  pool's msg.sender = router
  pool calls extension.beforeSwap(sender=router, ...)
  extension checks allowedSwapper[pool][router]  →  true  →  passes
  attacker's swap executes on the curated pool

Result:
  attacker (not allowlisted) successfully swaps on a pool that was supposed to gate them
  the allowlist invariant is broken; LPs are exposed to unauthorized counterparties
```

Conversely, if the admin does not allowlist the router:
```
  allowlisted user calls MetricOmmSimpleRouter.exactInput(pool, ...)
  router calls pool.swap()  →  sender = router
  extension checks allowedSwapper[pool][router]  →  false  →  NotAllowedToSwap
  allowlisted user's swap is rejected through the supported periphery path
