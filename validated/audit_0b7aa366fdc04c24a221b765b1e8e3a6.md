Audit Report

## Title
Deposit Allowlist Checks LP Recipient Instead of Actual Depositor, Allowing Full Allowlist Bypass - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the address actually providing tokens) and validates `owner` (the caller-supplied LP recipient) instead. Because `MetricOmmPool.addLiquidity` allows any caller to freely specify an arbitrary `owner`, any address not on the allowlist can bypass the guard entirely by naming any allowlisted address as `owner`. The deposit allowlist is completely neutralised.

## Finding Description

`MetricOmmPool.addLiquidity` passes both the real caller and the caller-supplied recipient to the extension hook:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first positional parameter (`sender`) is unnamed and ignored. The allowlist check is performed exclusively against `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

The mapping and setter naming make the intended semantics unambiguous — the guard is supposed to restrict who **provides** tokens, not who **receives** shares:

```solidity
mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_)
``` [3](#0-2) 

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` and ignores `recipient`, confirming the asymmetry is a bug:

```solidity
function beforeSwap(address sender, address, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) { ... }
}
``` [4](#0-3) 

Exploit path:
1. Pool is deployed with `DepositAllowlistExtension` as its `beforeAddLiquidity` hook.
2. Pool admin allowlists `alice` via `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
3. Bob calls `pool.addLiquidity(owner = alice, ...)`.
4. Pool calls `beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. Check evaluates `allowedDepositor[pool][alice]` → `true` → passes.
6. Bob's tokens enter the pool; Alice receives LP shares. Bob has bypassed the allowlist.

No existing guard prevents this. The `addLiquidity` function imposes no restriction on who may specify an arbitrary `owner`. [5](#0-4) 

## Impact Explanation

The deposit allowlist is completely neutralised. Any address can deposit into a restricted pool by naming any known allowlisted address (e.g., the publicly readable pool admin from `poolAdmin[pool]`) as `owner`. This constitutes a broken core pool access-control mechanism: the pool admin's configuration is rendered ineffective, and an attacker can freely shift bin balances, alter the effective marginal price seen by subsequent swaps, and extract value from existing LPs through the oracle-anchored pricing mechanism.

## Likelihood Explanation

Exploitation requires no special privileges and no complex setup. Any caller needs only one allowlisted address, which is trivially obtainable from the factory's public `poolAdmin` mapping. The bypass is a single parameter substitution in a standard `addLiquidity` call, repeatable at will with zero cost beyond gas.

## Recommendation

Rename the first parameter and check `sender` instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

1. Deploy pool with `DepositAllowlistExtension` as `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is **not** allowlisted.
3. Bob calls:
   ```solidity
   pool.addLiquidity(
       owner        = alice,   // any allowlisted address
       salt         = 0,
       deltas       = <desired bins/shares>,
       callbackData = <token transfer data>,
       extensionData = ""
   );
   ```
4. Pool calls `DepositAllowlistExtension.beforeAddLiquidity(sender=bob, owner=alice, ...)`.
5. Check evaluates `allowedDepositor[pool][alice]` → `true` → **passes**.
6. Bob's tokens are deposited; Alice receives LP shares. Bob has bypassed the allowlist.
