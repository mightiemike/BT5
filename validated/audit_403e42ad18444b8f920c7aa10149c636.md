### Title
`OracleValueStopLossExtension` Uses Arithmetic Mean Instead of Geometric Mean for Mid-Price, Weakening the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid price as an **arithmetic mean** of bid and ask, while every other component in the protocol (the pool's swap engine, `PriceVelocityGuardExtension`, and the data-provider lens) uses the **geometric mean** (`sqrt(bid * ask)`). Because AM ≥ GM for any non-zero spread, the stop-loss metrics are systematically biased: `metricToken0` is underestimated and its high-watermark is set too low, so the guard requires a larger real value drop before it fires for `zeroForOne = true` swaps. The guard is weakened, not bypassed entirely, but the gap grows with spread width and can reach several percent for wide oracle quotes.

---

### Finding Description

In `_afterSwapOracleStopLoss`:

```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The pool's swap engine derives the mid price as:

```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);   // geometric mean
``` [2](#0-1) 

`PriceVelocityGuardExtension` also uses the geometric mean via `SwapMath.midAndSpreadFeeX64FromBidAsk`: [3](#0-2) 

The stop-loss metrics are defined as:

```
metricToken0 = t0*SCALE/shares + (t1 * Q64 / mid) * SCALE / shares
metricToken1 = (t0 * mid / Q64) * SCALE / shares + t1*SCALE/shares
``` [4](#0-3) 

By AM-GM inequality, `AM(bid, ask) ≥ GM(bid, ask)` for any non-zero spread. Therefore:

- `1/AM < 1/GM` → `metricToken0` is **underestimated** when AM is used.
- `AM > GM` → `metricToken1` is **overestimated** when AM is used.

The high-watermark ratchets up to the maximum observed metric. Because `metricToken0` is underestimated, `hwm0` is set to a lower value than it would be with the correct geometric mean. The stop-loss floor for token0 is:

```
floor0 = hwm0 * (E6 - drawdown) / E6
``` [5](#0-4) 

A lower `hwm0` produces a lower `floor0`, meaning the guard requires a **larger actual value drop** before it fires for `zeroForOne = true` swaps (which drain token1 from the pool and are checked against `metricToken0`). [5](#0-4) 

---

### Impact Explanation

The stop-loss guard for `zeroForOne = true` swaps is less sensitive than the pool admin configured. The gap between the intended floor and the actual floor is proportional to the oracle spread:

| Oracle spread | AM − GM | `metricToken0` underestimation |
|---|---|---|
| 1% | ~0.0025% | ~0.0025% of t1 component |
| 10% | ~0.5% | ~0.5% of t1 component |
| 50% | ~13.4% | ~13.4% of t1 component |

For a pool with significant token1 inventory and a 10% oracle spread, the effective drawdown threshold is ~0.5% wider than configured. On a $1M pool this represents ~$5,000 of additional extractable value before the stop-loss fires. The guard is not completely bypassed — it still fires eventually — but the invariant "value per share cannot fall faster than drawdown" is violated by the AM/GM gap.

---

### Likelihood Explanation

- Any swap through a pool with `OracleValueStopLossExtension` configured triggers `afterSwap` and the biased metric computation. No special privilege is required.
- The bias is always present whenever `bid ≠ ask` (i.e., whenever the oracle provides a spread, which is the normal operating mode).
- The effect is larger during high-volatility periods when oracle spreads widen, which is precisely when the stop-loss protection is most needed.

---

### Recommendation

Replace the arithmetic mean with the geometric mean, consistent with the rest of the protocol:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (correct, matches pool swap engine):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(
    uint256(bidPriceX64),
    uint256(askPriceX64)
);
```

This aligns the stop-loss metric with the price the pool actually uses for swap settlement, ensuring the guard fires at the threshold the pool admin intended.

---

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 100_000` (10%), oracle bid = 0.9 Q64, ask = 1.1 Q64.
2. Add liquidity: bin holds `t0 = 0`, `t1 = 1000` shares = 1000.
3. First swap (`zeroForOne = false`) sets the watermark:
   - AM mid = 1.0 Q64 → `metricToken0 = 0 + 1000/1.0 = 1000`
   - GM mid = 0.995 Q64 → correct `metricToken0 = 0 + 1000/0.995 ≈ 1005`
   - `hwm0` is set to 1000 (AM) instead of 1005 (GM).
4. Floor with AM: `1000 * 0.9 = 900`.
5. Floor with GM: `1005 * 0.9 = 904.5`.
6. A subsequent `zeroForOne = true` swap that reduces `metricToken0` to 901 would:
   - **Pass** the AM-based guard (901 > 900).
   - **Fail** the GM-based guard (901 < 904.5).
7. The pool admin configured a 10% drawdown but the guard effectively allows ~10.4% drawdown due to the formula mismatch.
