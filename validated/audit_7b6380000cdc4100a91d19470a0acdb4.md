### Title
Bonus Accrual Checkpoint Advanced Before Pool Balance Check Causes Permanent Reward Loss on Redelegation Slash — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `processEventsAndClaimBonus`, `applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime` **before** the pool balance check. When `slashRedelegationPosition` calls this function and the pool is insufficient, it swallows `ErrInsufficientBonusPool` and then persists the position with the already-advanced checkpoint. The accrued bonus window is permanently erased — no retry, no deferral.

### Impact Explanation

A position owner permanently loses all bonus rewards accrued since their last `LastBonusAccrual` timestamp up to the block of the redelegation slash. There is no retry path: the next call to `processEventsAndClaimBonus` for this position will start accrual from `blockTime` of the slash block, treating the prior window as already settled. This is a direct, unrecoverable loss of legitimately earned funds from the rewards pool that the owner was entitled to.

---

### Likelihood Explanation

The rewards pool is a module account funded by governance/admin deposits. It can be drained by normal reward claims across many positions. A redelegation slash (double-sign or downtime on the destination validator during the redelegation lock period) is an ordinary chain event. No attacker action is required against the victim — the victim's position is slashed as a side-effect of validator misbehavior. The two conditions (low pool balance + redelegation slash) can coincide naturally in production.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, so the checkpoint is only advanced when the bonus is actually paid. In the `ErrInsufficientBonusPool` branch of `slashRedelegationPosition`, do **not** persist the mutated position checkpoints — either skip `setPositionWithState` for the checkpoint fields or restore them before saving, so the owner can claim the owed bonus once the pool is refilled.

### Proof of Concept

1. Open a position on a tier with a bonded validator; let time pass so bonus accrues.
2. Drain the `RewardsPoolName` module account to zero (e.g., via repeated claims from other positions or direct governance drain).
3. Trigger `BeforeRedelegationSlashed` for the position's unbonding ID (simulate a redelegation slash via the staking keeper's `SlashRedelegation`).
4. Assert: `pos.LastBonusAccrual == blockTime` of the slash block (checkpoint advanced).
5. Assert: owner's balance did not increase (no bonus paid).
6. Refill the rewards pool.
7. Call `claimRewards` for the position.
8. Assert: the bonus for the window `[pre-slash LastBonusAccrual, slash blockTime]` is **not** paid — it starts accruing only from the slash block onward, confirming permanent loss.

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}

}
