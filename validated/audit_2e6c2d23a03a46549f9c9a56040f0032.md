### Title
`RuntimeDialect::op` bypasses `DISABLE_OP` guard for opcode 60 (`op_modpow`), diverging from `ChiaDialect` mempool behavior — (`src/runtime_dialect.rs`)

### Summary
`RuntimeDialect::op` dispatches opcode 60 directly through `f_lookup` without checking `ClvmFlags::DISABLE_OP`, while `ChiaDialect::op` explicitly gates opcode 60 behind that flag. Any caller using `RuntimeDialect` with `MEMPOOL_MODE` flags (which include `DISABLE_OP`) and `op_modpow` registered at index 60 will successfully execute `op_modpow`, whereas the same program run through `ChiaDialect` with identical flags returns `Unimplemented`.

### Finding Description
`MEMPOOL_MODE` is defined as: [1](#0-0) 

It includes `DISABLE_OP` specifically to disable `op_modpow` (opcode 60) in mempool-strict mode. `ChiaDialect::op` enforces this: [2](#0-1) 

`RuntimeDialect::op` has no equivalent guard. It resolves opcode 60 from `f_lookup` and calls it directly: [3](#0-2) 

`op_modpow` itself does not check `DISABLE_OP`: [4](#0-3) 

The `f_lookup_for_hashmap` function populates `f_lookup[60]` with `op_modpow` whenever `"op_modpow"` is mapped to `[60]` in the operator map: [5](#0-4) 

### Impact Explanation
A caller (wallet, mempool validator, Python-wheel consumer) that uses `RuntimeDialect` with `MEMPOOL_MODE` flags and the standard operator map (which includes `op_modpow → 60`) will accept a `(modpow ...)` program that `ChiaDialect` under the same flags rejects with `Unimplemented`. This is a direct mempool/consensus equivalence violation: the mempool-mode check that `DISABLE_OP` is supposed to enforce is silently skipped, so a program that should be rejected in strict mode passes validation.

### Likelihood Explanation
`RuntimeDialect` is a public API exposed through both the Rust crate and the Python wheel. Any downstream tool that constructs a `RuntimeDialect` with `MEMPOOL_MODE` (or any flags including `DISABLE_OP`) and the standard operator map — a natural and documented usage — is affected. The standard operator map includes `op_modpow` at index 60 via `opcode_by_name`: [6](#0-5) 

### Recommendation
Add an explicit `DISABLE_OP` check in `RuntimeDialect::op` before dispatching through `f_lookup`, mirroring the guard in `ChiaDialect::op`. Alternatively, strip `DISABLE_OP`-gated entries from `f_lookup` at construction time in `f_lookup_for_hashmap` when the flag is set, so the table itself reflects the active flag set.

use clvmr::allocator::Allocator;
use clvmr::chia_dialect::{ClvmFlags, MEMPOOL_MODE};
use clvmr::runtime_dialect::RuntimeDialect;
use clvmr::chia_dialect::ChiaDialect;
use clvmr::dialect::Dialect;

let mut alloc = Allocator::new();

// Build (modpow 3 10 7) = 4
let nil = alloc.nil();
let m = alloc.new_atom(&[7]).unwrap();
let args = alloc.new_pair(m, nil).unwrap();
let e = alloc.new_atom(&[10]).unwrap();
let args = alloc.new_pair(e, args).unwrap();
let b = alloc.new_atom(&[3]).unwrap();
let argument_list = alloc.new_pair(b, args).unwrap();
let op60 = alloc.new_atom(&[60]).unwrap();

// RuntimeDialect with MEMPOOL_MODE + op_modpow at index 60
let mut op_map = HashMap::new();
op_map.insert("op_modpow".to_string(), vec![60u8]);
let rd = RuntimeDialect::new(op_map, vec![1], vec![2], MEMPOOL_MODE);

// ^ succeeds — DISABLE_OP guard is bypassed

let cd = ChiaDialect::new(MEMPOOL_MODE);
let result2 = cd.op(&mut alloc, op60, argument_list, u64::MAX, Default::default());
assert!(result2.is_err(), "ChiaDialect: expected Unimplemented, got {:?}", result2);
// ^ fails with Unimplemented — guard is enforced
