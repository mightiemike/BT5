Audit Report

## Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` is intended to gate `addLiquidity` calls by the depositor's address. However, it checks the `owner` parameter (the position beneficiary) rather than the `sender` parameter (the actual `msg.sender` of `addLiquidity`). Any unprivileged, non-allowlisted actor can bypass the allowlist entirely by supplying any allowlisted address as `owner`, causing the pool admin's access-control intent to be completely nullified.

## Finding Description

**Call path:**

1. `MetricOmmPool.addLiquidity(owner, salt, deltas, callbackData, extensionData)` is called by an arbitrary `msg.sender`.
2. The pool calls `_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData)` — passing the real caller as `sender` and the caller-supplied `owner` as a separate argument.
3. `ExtensionCalling._beforeAddLiquidity` encodes both and calls `IMetricOmmExtensions.beforeAddLiquidity(sender, owner, ...)` on the extension.
4. `DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first (unnamed) argument and `owner` as its second, but **ignores `sender` entirely** and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

**Exploit flow:**

- Pool admin configures `DepositAllowlistExtension` and allowlists only `trustedLP`.
- Attacker (not allowlisted) calls `pool.addLiquidity(trustedLP, salt, deltas, callbackData, "")`.
- Extension evaluates `allowedDepositor[pool][trustedLP]` → `true`. Check passes.
- Attacker pays tokens via the swap callback; LP shares are minted to `trustedLP`.
- The allowlist is fully bypassed; the attacker has deposited into a restricted pool.

**Why existing guards fail:**

The `onlyPool` modifier in `BaseMetricExtension` only verifies that `msg.sender` is a registered pool — it does not fix the wrong field being checked inside the function body. The `removeLiquidity` guard (`msg.sender != owner` revert) is irrelevant here; it only prevents the attacker from withdrawing, not from depositing.

## Impact Explanation

The deposit allowlist is a pool admin access-control mechanism intended to restrict which addresses may provide liquidity (e.g., KYC/AML compliance, institutional-only pools). Because the check is on `owner` rather than `sender`, any unprivileged actor can add liquidity to a pool that is supposed to be restricted, violating the admin-boundary invariant. This constitutes broken core pool functionality (the allowlist extension does not enforce its stated gate) and an admin-boundary break where an unprivileged path bypasses a pool admin control.

## Likelihood Explanation

Exploitation requires no special privileges, no tokens at risk for the protocol, and no complex setup. Any EOA or contract can call `addLiquidity` with a known allowlisted address as `owner`. The condition is trivially reachable on any pool that deploys `DepositAllowlistExtension` with a non-empty allowlist. It is repeatable at will.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor / `msg.sender` of `addLiquidity`) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```
