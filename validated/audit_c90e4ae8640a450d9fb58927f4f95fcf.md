### Title
`WithdrawCollateralV2` Transfers Tokens to Sanctioned `sendTo` Address Without Recipient Sanctions Check — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateralV2` transaction path allows a user to specify an arbitrary `sendTo` recipient address. While the protocol enforces sanctions checks on both parties during deposits, the `sendTo` address in `WithdrawCollateralV2` is never validated against the sanctions list before tokens are transferred out of the protocol.

---

### Finding Description

The protocol maintains a `requireUnsanctioned` guard (defined in `EndpointStorage.sol`) and applies it symmetrically during deposits: `depositCollateralWithReferral` calls `requireUnsanctioned(msg.sender)` and `requireUnsanctioned(sender)` — checking both the depositor and the subaccount owner. [1](#0-0) 

However, in `EndpointTx.processTransactionImpl`, the `WithdrawCollateralV2` branch validates only the sender's signature via `validateSignedTx` and then passes the user-controlled `signedTx.tx.sendTo` directly to `clearinghouse.withdrawCollateral` with no sanctions check on the recipient: [2](#0-1) 

`validateSignedTx` calls `validateNonce`, `validateSignature`, and `requireSubaccount` — none of which invoke `requireUnsanctioned`: [3](#0-2) 

Inside `Clearinghouse.withdrawCollateral`, the `sendTo` address is used directly to route the token transfer with no sanctions gate: [4](#0-3) 

The `requireUnsanctioned` helper is only called in the deposit path and in `submitSlowModeTransactionImpl` (for the slow-mode submitter), never for any withdrawal recipient: [5](#0-4) 

---

### Impact Explanation

A user can sign a `WithdrawCollateralV2` transaction designating a sanctioned address as `sendTo`. The sequencer, which is designed to faithfully process user-signed transactions, submits it on-chain. The contract performs no on-chain sanctions check on `sendTo`, so real ERC-20 tokens are transferred to the sanctioned address. This directly bypasses the protocol's compliance system and delivers funds to an address the protocol explicitly intends to block. If the underlying token (e.g., USDC) also maintains its own blocklist, the transfer may revert, causing the withdrawal to fail and the user's collateral balance to be decremented while no tokens are received — locking funds in an unrecoverable state.

---

### Likelihood Explanation

Medium. The user controls the `sendTo` field in their signed `WithdrawCollateralV2` transaction. The sequencer processes user-signed transactions as a core protocol function. There is no off-chain enforcement guarantee encoded on-chain, and the asymmetry with the deposit path (which checks both parties) makes this an exploitable gap for any user who wishes to route a withdrawal to a sanctioned address.

---

### Recommendation

Add a `requireUnsanctioned(sendTo)` call in the `WithdrawCollateralV2` branch of `processTransactionImpl` before invoking `clearinghouse.withdrawCollateral`, mirroring the symmetric check already present in `depositCollateralWithReferral`. Alternatively, enforce the check inside `Clearinghouse.withdrawCollateral` after `sendTo` is resolved from `address(0)`.

```solidity
// In EndpointTx.processTransactionImpl, WithdrawCollateralV2 branch:
address resolvedSendTo = signedTx.tx.sendTo;
if (resolvedSendTo != address(0)) {
    requireUnsanctioned(resolvedSendTo);
}
clearinghouse.withdrawCollateral(
    signedTx.tx.sender,
    signedTx.tx.productId,
    signedTx.tx.amount,
    resolvedSendTo,
    nSubmissions
);
```

---

### Proof of Concept

1. Alice holds collateral in a Nado subaccount.
2. Alice signs a `WithdrawCollateralV2` transaction with `sendTo = SANCTIONED_ADDRESS` (an address on the OFAC SDN list that is registered in the `ISanctionsList` contract).
3. Alice submits the signed transaction to the sequencer.
4. The sequencer calls `submitTransactionsChecked`, which routes to `processTransactionImpl`.
5. `validateSignedTx` passes — it only checks Alice's nonce and signature, not `sendTo`.
6. `clearinghouse.withdrawCollateral(..., SANCTIONED_ADDRESS, ...)` is called.
7. `handleWithdrawTransfer` transfers tokens to `SANCTIONED_ADDRESS` with no revert.
8. The sanctioned address receives protocol tokens. The protocol's `requireUnsanctioned` system is fully bypassed for the withdrawal recipient. [6](#0-5) [7](#0-6)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
