### Title
`SpotEngine#addOrUpdateProduct()` Unconditionally Overwrites Token Address for Existing Products, Causing User Fund Loss on Withdrawal — (`File: core/contracts/SpotEngine.sol`)

---

### Summary

`SpotEngine.addOrUpdateProduct()` unconditionally overwrites `configs[productId]`, including the `token` address field, for both new and existing products. Because `BaseWithdrawPool.getToken()` reads `spotEngine().getConfig(productId).token` at withdrawal execution time, changing the token address for a product that already holds user deposits desynchronizes the recorded balance from the actual withdrawable asset, causing users to receive the wrong token or have their withdrawal permanently fail.

---

### Finding Description

In `SpotEngine.addOrUpdateProduct()`, the assignment `configs[productId] = config` on line 83 is unconditional — it executes for both new and existing products. The `isNewProduct` guard only gates the `_setState` initialization block (lines 85–96); it does not protect the `configs` write. [1](#0-0) 

The `Config` struct contains the `token` address field. There is no check preventing the owner from supplying a different `token` address when calling `addOrUpdateProduct` on a product that already has live user deposits.

The withdrawal path in `BaseWithdrawPool` resolves the token to transfer at execution time by calling `getToken(productId)`, which reads `spotEngine().getConfig(productId).token` live: [2](#0-1) 

This live read is used directly in `submitFastWithdrawal`: [3](#0-2) 

And in `Clearinghouse.withdrawInsurance`, the same pattern applies — the token is resolved from `spotEngine.getConfig(QUOTE_PRODUCT_ID).token` at the moment of transfer: [4](#0-3) 

---

### Impact Explanation

**Corrupted asset delta:** A user who deposited Token A for product X has their balance faithfully recorded in `SpotEngine.balances[X][subaccount]`. The `WithdrawPool` holds Token A. After the owner calls `addOrUpdateProduct(X, ..., Config{token: TokenB, ...}, ...)`, `configs[X].token` becomes Token B. Any subsequent withdrawal for product X — whether via `submitFastWithdrawal` or the sequencer-driven `submitWithdrawal` path — will attempt to transfer Token B. If the pool holds no Token B, the `safeTransfer` reverts, permanently blocking the user's withdrawal. If Token B is a different live token, the user receives the wrong asset while their original Token A deposit remains stranded in the pool.

The corrupted invariant: `configs[productId].token` must always match the token that was physically deposited and is held by the `WithdrawPool` for that product. `addOrUpdateProduct` breaks this invariant with no guard.

---

### Likelihood Explanation

The owner of `SpotEngine` is the `ContractOwner` multisig, which exposes `addOrUpdateProduct` calls via `rawSpotAddOrUpdateProductCalls`. [5](#0-4) 

A legitimate token migration scenario (e.g., USDC → USDC.e, or upgrading a vault token) would naturally involve calling `addOrUpdateProduct` with a new `token` address on an existing product. The loss is not contingent on malicious intent — it is a structural gap that fires whenever a token address update races with in-flight or queued user withdrawals. Likelihood is **Low-Medium**: the action requires admin execution, but the scenario is realistic during any token migration.

---

### Recommendation

Add a guard in `addOrUpdateProduct` that prevents changing the `token` address for an existing product that already has deposits:

```solidity
function addOrUpdateProduct(
    uint32 productId,
    uint32 quoteId,
    int128 sizeIncrement,
    int128 minSize,
    Config calldata config,
    RiskHelper.RiskStore calldata riskStore
) public onlyOwner {
    bool isNewProduct = _addOrUpdateProduct(
        productId, quoteId, sizeIncrement, minSize, riskStore
    );
    // Prevent token address change for existing products with deposits
    if (!isNewProduct) {
        require(
            configs[productId].token == config.token,
            "token address immutable after product creation"
        );
    }
    configs[productId] = config;
    ...
}
```

Alternatively, if token migration must be supported, add a post-update balance check analogous to the mitigation in the referenced report — verifying that the `WithdrawPool`'s balance of the new token is at least equal to the total deposits recorded in `SpotEngine` for that product before the update takes effect.

---

### Proof of Concept

1. Alice deposits `1,000,000 USDC` into the protocol for `productId = 1` (Token A = USDC). `SpotEngine.balances[1][Alice]` is credited; `WithdrawPool` holds 1,000,000 USDC.
2. Owner calls `SpotEngine.addOrUpdateProduct(1, ..., Config{token: USDCv2, ...}, ...)`. Line 83 executes: `configs[1].token = USDCv2`.
3. Alice submits a fast withdrawal. `BaseWithdrawPool.submitFastWithdrawal` calls `getToken(1)` → returns `USDCv2`. The pool holds zero USDCv2. `token.safeTransfer(Alice, amount)` reverts.
4. Alice's 1,000,000 USDC is permanently stranded in the `WithdrawPool` with no recovery path available to her. [6](#0-5) [7](#0-6)
