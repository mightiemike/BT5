### Title
Unchecked `transferFrom()` Return Value in `replaceUsdcEWithUsdc()` Allows Silent Failure Leading to usdcE Drain - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom()` directly without checking its return value, while the same contract already imports and uses `ERC20Helper.safeTransferFrom()` everywhere else. If the USDC token returns `false` instead of reverting on a failed transfer, the function silently continues, withdraws usdcE from the `DirectDepositV1` contract, and sends it to the caller â€” without the caller ever having paid USDC.

---

### Finding Description

`replaceUsdcEWithUsdc()` is an `external` function with no access control modifier (only a chain-ID guard). It is callable by any unprivileged address on chain 57073 (Ink). [1](#0-0) 

The critical sequence inside the `if (balance > 0)` block is:

1. **Line 616** â€” raw `transferFrom()` with unchecked return value:
   ```solidity
   IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
   ```
2. **Line 617** â€” usdcE is withdrawn from `directDepositV1` to `ContractOwner`.
3. **Line 618** â€” usdcE is sent to `msg.sender` via `safeTransfer`. [2](#0-1) 

The `ERC20Helper` library â€” already imported and applied via `using ERC20Helper for IERC20Base` â€” provides `safeTransferFrom()`, which wraps the call and requires either a revert or a `true` return value: [3](#0-2) 

Every other token pull in the codebase (e.g., `EndpointStorage.safeTransferFrom`, `chargeSlowModeFee`) uses this safe wrapper: [4](#0-3) 

Line 616 is the sole location in the production codebase that calls raw `.transferFrom()` on a user-supplied flow path. [5](#0-4) 

---

### Impact Explanation

If the USDC token at the hardcoded address returns `false` on a failed `transferFrom` (rather than reverting), the function does not revert. It proceeds to:
- Pull usdcE out of the victim `DirectDepositV1` contract (step 2).
- Transfer that usdcE to `msg.sender` (step 3).

The attacker pays nothing and receives the full usdcE balance held by any `directDepositV1` address that has been registered for a subaccount. The corrupted asset delta is: usdcE balance of `directDepositV1[subaccount]` â†’ drained to attacker.

---

### Likelihood Explanation

The hardcoded USDC token (`0x2D270e6886d130D724215A266106e6832161EAEd` on Ink chain) is a Circle-issued contract that currently reverts on failure, making exploitation unlikely under normal conditions. However:
- The function is permissionless â€” any address can call it for any registered subaccount.
- The inconsistency is a latent defect: if the token address is ever repointed (upgrade, migration, or chain reorg of the deployment), or if a non-reverting token is used in a future deployment, the path becomes immediately exploitable.
- The same contract already uses `safeTransfer` on line 618 for the outbound leg, making the omission on line 616 an unambiguous oversight rather than an intentional design choice.

---

### Recommendation

Replace the raw `transferFrom` on line 616 with the `safeTransferFrom` wrapper already available via `ERC20Helper`:

```solidity
// Before (line 616)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [3](#0-2) 

---

### Proof of Concept

1. A `DirectDepositV1` contract exists for some `subaccount` (registered via `createDirectDepositV1`) and holds a non-zero usdcE balance.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on chain 57073.
3. `balance = IERC20Base(usdcE).balanceOf(directDepositV1)` â†’ non-zero.
4. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` is called. If USDC returns `false` (no revert), execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE))` pulls all usdcE into `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` sends all usdcE to the attacker.
7. Attacker has received `balance` usdcE without paying any USDC. The `directDepositV1` contract is drained. [1](#0-0)

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
