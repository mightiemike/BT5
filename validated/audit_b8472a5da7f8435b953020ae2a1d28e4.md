### Title
Public `run_mainchain_gc` Permanently Destroys Fork-Point Ancestor, Making Any In-Progress Reorg Unrecoverable — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is an unrestricted public entry point that erases mainchain block headers from `headers_pool`. `reorg_chain` assumes every ancestor of a fork tip is present in `headers_pool`. When an attacker calls `run_mainchain_gc` while a fork is partially submitted and the fork-point block sits exactly at the retention boundary, the fork-point block is permanently deleted from `headers_pool`. Every subsequent attempt to promote the fork — or even to re-submit its blocks — panics unconditionally. The heavier chain can never become canonical, and any cross-chain settlement that depends on it is permanently frozen.

---

### Finding Description

**`run_mainchain_gc` has no access control and destroys `headers_pool` entries** [1](#0-0) 

The only gate is the `#[pause]` decorator; any unprivileged account can call it at any time. Inside the loop it calls `remove_block_header`, which deletes the block from **both** `mainchain_header_to_height` and `headers_pool`: [2](#0-1) [3](#0-2) 

**`store_fork_header` stores fork blocks only in `headers_pool`**

Fork blocks are never inserted into `mainchain_header_to_height`: [4](#0-3) 

**`reorg_chain` walks back through `headers_pool` and panics if any ancestor is missing**

The termination condition checks `mainchain_header_to_height`; the cursor advance fetches from `headers_pool` with an unconditional `unwrap`: [5](#0-4) 

**The broken invariant:** `reorg_chain` assumes the entire fork ancestry back to the common ancestor is present in `headers_pool`. GC can silently violate this assumption because it removes mainchain blocks from `headers_pool` without checking whether any pending fork headers reference them as ancestors.

---

### Impact Explanation

Once the fork-point block is GC'd:

1. `reorg_chain` panics at line 642 ("previous fork block should be there") every time the fork tip is submitted. The transaction reverts; the fork tip is never stored.
2. Re-submitting any fork block whose `prev_block_hash` points to the deleted fork-point block causes `get_prev_header` to panic with "PrevBlockNotFound" at line 674.
3. The fork-point block cannot be re-inserted: the contract has no API to add an arbitrary historical mainchain block.
4. The heavier chain is permanently unrecoverable. Any cross-chain proof or settlement that requires the heavier fork to be canonical is permanently frozen. [6](#0-5) 

---

### Likelihood Explanation

- `run_mainchain_gc` requires zero privileges and is callable by any NEAR account.
- The attacker only needs to observe two public getters (`get_last_block_height` and `get_block_hash_by_height`) to determine when the fork-point block is exactly at the retention boundary.
- Deep-but-realistic Bitcoin forks (2–6 blocks) occur during network partitions and mining pool incidents; a fork point near the retention boundary is reachable after the contract has been running for `gc_threshold` blocks (~1 year at default settings).
- The attacker does not need to control any Bitcoin mining power; they only need to time a single cheap NEAR transaction.

---

### Recommendation

1. **Protect `run_mainchain_gc` with a role check** (e.g., `Role::RelayerManager` or `Role::DAO`) so unprivileged accounts cannot trigger pruning.
2. **Alternatively, track the minimum fork-point height**: before removing a block, verify that no `headers_pool` entry whose `prev_block_hash` chain leads to it exists below the current `mainchain_initial_blockhash`. A simpler proxy is to refuse GC if any fork header in `headers_pool` has a height ≤ `end_removal_height`.
3. **`reorg_chain` should return an error instead of panicking** when a `headers_pool` lookup fails, so a corrupted state does not permanently brick the reorg path.

---

### Proof of Concept

```
State: gc_threshold = 3, mainchain heights 0..3 (4 blocks stored), tip at height 3.
Fork point: block at height 0 (mainchain_initial_blockhash).

Step 1 – Relayer submits fork blocks f1 (height 1) and f2 (height 2), both branching
         from height 0. Both are stored via store_fork_header (headers_pool only).

Step 2 – Attacker calls run_mainchain_gc(1).
         amount_of_headers_we_store = 4, gc_threshold = 3 → removes height 0.
         remove_block_header(hash_0):
           mainchain_header_to_height.remove(hash_0)  ← fork-point gone from height map
           headers_pool.remove(hash_0)                ← fork-point gone from pool

Step 3 – Relayer submits fork block f3 (height 3, chainwork > mainchain tip).
         submit_block_header_inner → store_fork_header(f3) → reorg_chain(f3, 3).

Step 4 – reorg_chain walks back: f3 → f2 → f1.
         f1.prev_block_hash = hash_0.
         headers_pool.get(hash_0) → None → panic("previous fork block should be there").
         Transaction reverts. f3 is never stored.

Step 5 – Relayer retries submitting f1.
         get_prev_header(f1) → headers_pool.get(hash_0) → None
         → panic("PrevBlockNotFound").
         Fork is permanently unrecoverable.
```

The sequence is reproducible in a workspace test by initializing with `skip_pow_verification: true`, submitting fork headers up to the penultimate block, calling `run_mainchain_gc` directly, then asserting that the final fork submission panics and that `get_block_hash_by_height` for the fork-point height returns `None`.
