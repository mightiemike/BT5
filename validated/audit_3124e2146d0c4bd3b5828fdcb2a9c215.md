### Title
Unprivileged Caller Can Force Token Withdrawal from Any DirectDepositV1 via Missing `onlyOwner` on `replaceUsdcEWithUsdc` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` lacks the `onlyOwner` modifier, allowing any caller on Ink chain to invoke `DirectDepositV1.withdraw()` — a function that is itself `onlyOwner` — on any registered DDA. The intended protected path for withdrawing tokens from a DDA is `withdrawFromDirectDepositV1()`, which is correctly gated by `onlyOwner`. The unprotected sibling function bypasses that gate entirely, mirroring the exact bug class of the reference report.

---

### Finding Description

`DirectDepositV1.withdraw()` is declared `onlyOwner`: [1](#0-0) 

`ContractOwner` is the owner of every DDA it deploys. The intended privileged path for the protocol team to recover tokens from a DDA is: [2](#0-1) 

This function is correctly gated with `onlyOwner`.

However, `replaceUsdcEWithUsdc()` — a migration helper — also calls `DirectDepositV1.withdraw()` on an arbitrary DDA, but carries **no `onlyOwner` modifier**: [3](#0-2) 

The only guard is `require(block.chainid == 57073, ERR_UNAUTHORIZED)`, which is satisfied by any caller on Ink Mainnet. The function:

1. Reads the usdcE balance of the target DDA.
2. Pulls `balance` USDC from `msg.sender` into the DDA via `transferFrom`.
3. Calls `DirectDepositV1(directDepositV1).withdraw(usdcE)` — withdrawing the DDA's usdcE to `ContractOwner`.
4. Forwards that usdcE to `msg.sender`.

The same pattern exists in `wrapVaultAsset()`, which also calls `DirectDepositV1.withdraw()` without any access control: [4](#0-3) 

---

### Impact Explanation

Any unprivileged caller can force a token swap on any DDA that holds usdcE, replacing the DDA's usdcE with USDC at a 1:1 nominal ratio. The DDA owner (subaccount owner) did not authorize this operation.

**Concrete asset delta**: The DDA's usdcE balance is zeroed and replaced with USDC. If usdcE trades at a premium to USDC (a realistic condition during a depeg or liquidity crunch), the DDA owner suffers a direct financial loss equal to `balance × (price(usdcE) − price(USDC))`. The attacker profits by the same amount.

Even at parity, the forced swap changes the token type that will be credited to the subaccount when `creditDeposit()` is later called, which may violate the subaccount owner's intent and interact unexpectedly with product-specific deposit logic.

The `wrapVaultAsset()` variant additionally exposes DDA asset balances to vault deposit fees or slippage, with no compensation to the DDA owner.

---

### Likelihood Explanation

- The Ink Mainnet chain ID check (`57073`) is trivially satisfied by any on-chain caller.
- DDAs accumulate usdcE passively; no special setup is required by the attacker.
- The attacker must hold USDC to execute the swap, but this is a low barrier.
- The function is callable permissionlessly at any time a DDA has a non-zero usdcE balance.

Likelihood: **Medium** — requires a DDA with usdcE balance and an attacker with USDC, both of which are routine conditions on Ink Mainnet.

---

### Recommendation

Add `onlyOwner` to `replaceUsdcEWithUsdc()` and `wrapVaultAsset()` in `ContractOwner.sol`, consistent with the access control applied to `withdrawFromDirectDepositV1()`:

```solidity
// Before
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
}

// After
function replaceUsdcEWithUsdc(bytes32 subaccount) external onlyOwner {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
}
```

### Proof of Concept

1. A subaccount owner sends usdcE to their DDA address. The DDA now holds `N` usdcE.
2. Attacker approves `ContractOwner` to spend `N` USDC via `IERC20(usdc).approve(contractOwner, N)`.
3. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` on Ink Mainnet.
4. Execution path:
   - `balance = usdcE.balanceOf(dda)` → `N`
   - `usdc.transferFrom(attacker, dda, N)` → DDA now holds `N` USDC, `N` usdcE
   - `dda.withdraw(usdcE)` → `N` usdcE transferred to `ContractOwner`
   - `usdcE.safeTransfer(attacker, N)` → attacker receives `N` usdcE
5. DDA now holds `N` USDC instead of `N` usdcE. If usdcE > USDC in market value, the subaccount owner has lost the premium; the attacker has captured it.
