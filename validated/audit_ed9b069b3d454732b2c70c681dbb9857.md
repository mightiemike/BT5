### Title
Unnecessary `merkle_proof.is_empty()` Check in `verify_transaction_inclusion` Blocks Valid Single-Transaction Block Proofs — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` contains a `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard that serves no cryptographic purpose and permanently rejects valid SPV proofs for coinbase transactions in single-transaction Bitcoin blocks. Because `verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` internally, both public entrypoints are affected.

---

### Finding Description

In `contract/src/lib.rs` at line 315, the deprecated-but-still-callable `verify_transaction_inclusion` function contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

Immediately after this guard, the function computes the Merkle root and compares it to the stored header's `merkle_root`:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

`compute_root_from_merkle_proof` iterates over the proof slice. When the slice is empty, the loop body never executes and the function simply returns `transaction_hash` unchanged:

```rust
for proof_hash in merkle_proof {
    // ...
}
current_hash  // returned as-is when merkle_proof is empty
``` [3](#0-2) 

For a Bitcoin block containing exactly one transaction (the coinbase), the Bitcoin protocol defines `merkle_root = coinbase_tx_id`. An empty proof is therefore the **correct and complete** proof: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id`, which equals `merkle_root`. The `require!` guard fires before this computation is ever reached, unconditionally panicking on a cryptographically valid input.

`verify_transaction_inclusion_v2`, the recommended replacement, calls `verify_transaction_inclusion` via `self.verify_transaction_inclusion(args.into())` after its own coinbase-proof check passes:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

So both public entrypoints are blocked for single-transaction blocks.

---

### Impact Explanation

Any NEAR consumer contract (bridge, atomic swap, cross-chain lending protocol) that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` to confirm a coinbase transaction in a single-transaction Bitcoin block will receive a panic instead of a `true` result. The proof is valid; the contract rejects it anyway. This permanently prevents verification of a class of legitimate Bitcoin transactions, breaking downstream protocols that depend on the light client's SPV guarantee.

---

### Likelihood Explanation

Single-transaction blocks (containing only the coinbase) are uncommon but entirely valid and occur on mainnet. Any relayer or consumer that submits such a block and then attempts to verify its coinbase transaction hits this path deterministically. No special attacker capability is required — any unprivileged NEAR caller supplying a valid proof for such a block triggers the bug.

---

### Recommendation

Remove the unnecessary guard:

```rust
// DELETE this line:
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
```

`compute_root_from_merkle_proof` already handles the empty-proof case correctly by returning the transaction hash itself, which is then compared against the stored `merkle_root`. No additional guard is needed. [1](#0-0) 

---

### Proof of Concept

1. A Bitcoin block is mined containing only the coinbase transaction. Its `merkle_root` equals `coinbase_tx_id`.
2. The relayer submits this block header via `submit_blocks`; it is accepted and stored.
3. A consumer contract calls `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_id`
   - `tx_index = 0`
   - `merkle_proof = []` (correct empty proof)
   - `coinbase_tx_id = coinbase_tx_id`
   - `coinbase_merkle_proof = []`
4. The length-equality check passes (`0 == 0`). [5](#0-4) 
5. The coinbase proof check passes: `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id == merkle_root`. [6](#0-5) 
6. `verify_transaction_inclusion` is called. It immediately panics at `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")`. [1](#0-0) 
7. The consumer contract receives a panic instead of `true`, blocking the cross-chain operation permanently for this block.
