### Title
`env::block_timestamp_ms()` Used as Real-Time Proxy for Bitcoin Future-Timestamp Validation — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`, `contract/src/zcash.rs`)

### Summary

All four chain-specific `check_pow` implementations use `env::block_timestamp_ms()` — the NEAR block timestamp — as a proxy for real wall-clock time to enforce the Bitcoin consensus rule that a submitted block header's timestamp must not exceed local time by more than `MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 seconds / 2 hours). NEAR block timestamps are set by block producers and do not reliably track real wall-clock time: they can be stale (behind real time) during network congestion, or slightly ahead of real time when validators advance them. This is the direct analog of the zkSync `SystemContext.sol` issue, where consecutive blocks can share the same timestamp, causing time-sensitive contracts to behave incorrectly.

### Finding Description

In every chain variant, `check_pow` contains:

```rust
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

`env::block_timestamp_ms()` returns the timestamp embedded in the current NEAR block by its block producer. NEAR Protocol guarantees only that this value is monotonically increasing and within a loose bound relative to the previous block; it does not guarantee that it equals real wall-clock time. In particular:

- **Stale direction**: During NEAR network congestion or validator delays, `env::block_timestamp_ms()` can lag real time by tens of seconds to minutes. A Bitcoin header with a timestamp equal to real current time satisfies Bitcoin consensus but may fail `block_header.time <= current_timestamp + 7200` if `current_timestamp` is sufficiently behind real time.
- **Ahead direction**: NEAR block producers can set timestamps slightly ahead of real time (within NEAR's own protocol bounds). This widens the acceptance window beyond the intended 2-hour ceiling, allowing a relayer to submit a Bitcoin header whose timestamp exceeds real current time by more than 2 hours — a header the Bitcoin network itself would reject as "time-too-new".

The broken invariant is: the "time-too-new" guard is supposed to enforce Bitcoin's consensus rule using real current time, but `env::block_timestamp_ms()` is not real current time.

### Impact Explanation

**Security direction (invalid header acceptance)**: If NEAR block timestamps are ahead of real time by Δ seconds, the effective acceptance window becomes `real_time + Δ + 7200`. A relayer can submit a Bitcoin header with a timestamp up to `Δ + 7200` seconds ahead of real time. Such a header would be rejected by the Bitcoin network but accepted by the light client. The light client's `mainchain_tip_blockhash` and `mainchain_height_to_header` would be updated to reflect a header that is invalid under Bitcoin consensus, corrupting the canonical chain state and causing `verify_transaction_inclusion` / `verify_transaction_inclusion_v2` to return results based on an invalid chain tip.

**Liveness direction (valid header rejection)**: If NEAR block timestamps are stale by Δ seconds, a Bitcoin header with timestamp `T` where `real_time - Δ < T ≤ real_time` fails the check even though it is valid under Bitcoin consensus. The relayer cannot advance the light client's tip, breaking SPV proof verification for recent transactions.

### Likelihood Explanation

NEAR block timestamps are not guaranteed to equal real time. During periods of NEAR network stress, validator rotation, or block production delays, the timestamp can deviate from real time by seconds to minutes. The stale direction is the more common case and directly prevents the relayer from submitting valid recent Bitcoin headers. The ahead direction is bounded by NEAR's own protocol rules but is still possible within those bounds. Any unprivileged relayer calling `submit_blocks` triggers this path on every header submission.

### Recommendation

Replace `env::block_timestamp_ms()` with a more reliable time source. The standard approach used by Bitcoin light clients on other chains is to use the **Median Time Past (MTP)** of the submitted chain itself as the reference for the future-timestamp check, rather than the host chain's block timestamp. Alternatively, document explicitly that `env::block_timestamp_ms()` is used as the time reference and that the 2-hour window is approximate, subject to NEAR block timestamp accuracy. If a stricter bound is required, consider storing a trusted timestamp at initialization and advancing it based on submitted Bitcoin header timestamps.

### Proof of Concept

1. NEAR network experiences a 5-minute block production delay; `env::block_timestamp_ms()` is 300 seconds behind real time.
2. Relayer fetches a valid Bitcoin block header with `time = real_current_time` (e.g., Unix timestamp 1_700_000_000).
3. Relayer calls `submit_blocks([header])`.
4. Inside `check_pow` (e.g., `contract/src/bitcoin.rs` line 35–38):
   - `current_timestamp = (real_current_time - 300)` (stale NEAR timestamp)
   - Check: `1_700_000_000 <= (1_700_000_000 - 300) + 7200` → `1_700_000_000 <= 1_700_006_900` → passes (this case is fine)
5. Now consider the ahead direction: NEAR validator sets timestamp 100 seconds ahead. Relayer submits a Bitcoin header with `time = real_current_time + 7250` (250 seconds beyond the intended 2-hour window).
   - `current_timestamp = real_current_time + 100`
   - Check: `real_current_time + 7250 <= (real_current_time + 100) + 7200` → `real_current_time + 7250 <= real_current_time + 7300` → passes
   - This header would be rejected by the Bitcoin network as "time-too-new" but is accepted by the light client, corrupting `mainchain_tip_blockhash`.
