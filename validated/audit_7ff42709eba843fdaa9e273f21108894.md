The function:
1. Requires exactly one G2 atom as the first argument (the signature)
2. Iterates over the remaining args to build `items` — but if there are none, `items` stays `Vec::new()`
3. Calls `aggregate_verify(&signature, items)` with no guard against an empty list

There is **no minimum-pair check** anywhere in the function.

The behavior of `aggregate_verify` (from `chia_bls = "0.38.2"`, blst-backed) with an empty items list follows standard BLS aggregate-verify math:

- The check reduces to: `e(G1_generator, signature) == identity_Gt`
- This holds **if and only if** `signature` is the G2 identity element (point at infinity)
- A random non-identity G2 point would **not** pass — the question's claim that "any G2 atom" passes is incorrect

So the concrete reachable case is:

- Input: `(bls_verify <G2_identity_96_bytes>)` — one argument, no pairs
- `a.g2(first(...))` succeeds (identity is a valid G2 point)
- `items = Vec::new()`
- `aggregate_verify(&identity_sig, [])` → `true`
- Returns `Ok(Reduction(cost, nil))` — **success**

The identity G2 element is the well-known 96-byte compressed infinity point (`0xc0` followed by 95 zero bytes).

### Title
`op_bls_verify` accepts the G2 identity signature with zero (pk, msg) pairs, bypassing signature verification — (`src/bls_ops.rs`)

### Summary
`op_bls_verify` has no guard requiring at least one `(pk, msg)` pair. When called with only a G2 identity signature and no pairs, `aggregate_verify` is invoked with an empty items list and returns `true`, causing the operator to return `Ok(nil)` — success — without verifying any real signature.

### Finding Description
In `src/bls_ops.rs`, `op_bls_verify` parses the first argument as a G2 signature, then collects subsequent `(G1, msg)` pairs into `items`. If no pairs follow the signature, `items` remains empty and `aggregate_verify(&signature, items)` is called unconditionally. [2](#0-1) 

With an empty items list, BLS aggregate-verify reduces to checking `e(G1_generator, sig) == identity_Gt`, which is satisfied exactly when `sig` is the G2 identity element. No minimum-pair enforcement exists anywhere in the function.

### Impact Explanation
Any CLVM puzzle that uses `bls_verify` to gate a spend can be bypassed by providing the G2 identity element as the signature and omitting all `(pk, msg)` pairs. The operator returns `nil` (truthy success in CLVM), satisfying the puzzle without a real cryptographic signature. This is a consensus-relevant path: `op_bls_verify` is registered as opcode `59` in `ChiaDialect` and is active in both consensus and mempool modes. [3](#0-2) 

### Likelihood Explanation
The G2 identity element is a publicly known, fixed 96-byte value. Any attacker who can craft CLVM spend bundles can exploit this against puzzles that use `bls_verify` without hardcoding the number of pairs. The exploit requires no special privileges, no compromised nodes, and no social engineering.

### Recommendation
Add an explicit guard at the start of the pair-collection loop or immediately before calling `aggregate_verify`:

```rust
if items.is_empty() {
    return Err(EvalErr::BLSVerifyFailed(input))?;
}
```

This enforces the invariant that `bls_verify` must verify at least one `(pk, msg)` pair.

pub fn op_bls_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let mut cost = BLS_PAIRING_BASE_COST;
    check_cost(cost, max_cost)?;

    let mut args = input;

    // the first argument is the signature
    let signature = a.g2(first(a, args)?)?;

    // followed by a variable number of (G1, msg)-pairs (as a flat list)
    args = rest(a, args)?;

    let mut items = Vec::<(PublicKey, Atom)>::new();
    while !nilp(a, args) {
        let pk = a.g1(first(a, args)?)?;
        args = rest(a, args)?;
        let msg = atom(a, first(a, args)?, "bls_verify message")?;
        args = rest(a, args)?;

        cost += BLS_PAIRING_COST_PER_ARG;
        cost += msg.as_ref().len() as Cost * BLS_MAP_TO_G2_COST_PER_BYTE;
        cost += DST_G2.len() as Cost * BLS_MAP_TO_G2_COST_PER_DST_BYTE;
        check_cost(cost, max_cost)?;

        items.push((pk, msg));
    }

    if !aggregate_verify(&signature, items) {
        Err(EvalErr::BLSVerifyFailed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
}
