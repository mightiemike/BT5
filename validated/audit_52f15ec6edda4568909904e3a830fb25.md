### Title
Genesis `chain_work` Initialized to Single-Block Work Instead of Accumulated Chainwork — (`contract/src/lib.rs`)

### Summary

In `init_genesis()`, the genesis block's `chain_work` is set to only the work of that single block (`work_from_bits(block_header.bits)`), not the actual accumulated chainwork at the supplied `genesis_block_height`. When the contract is initialized at a non-zero height (the normal production case, e.g., height 685,440), the stored `chain_work` is understated by a factor of ~`genesis_block_height`, collapsing the fork-selection security guarantee from requiring an attacker to outwork the entire prior chain to requiring only enough work to outpace the handful of blocks submitted during initialization.

### Finding Description

`ExtendedHeader.chain_work` is documented as *"Accumulated chainwork at this position for this block"*: [1](#0-0) 

In `init_genesis()`, the genesis block's `chain_work` is assigned as:

```rust
let chain_work = work_from_bits(block_header.bits);   // single block only
``` [2](#0-1) 

`InitArgs` provides no field for the caller to supply the real accumulated chainwork at the genesis height: [3](#0-2) 

Every subsequent block submitted during init adds its own single-block work on top of this understated base, so after the full initialization sequence the mainchain tip's `chain_work` equals only `N × work_per_block` (where N is the number of init blocks, typically 12), instead of `(genesis_height + N) × work_per_block`.

The real production deployment initializes at height 685,440: [4](#0-3) 

### Impact Explanation

Fork selection is decided solely by `chain_work`:

```rust
if current_header.chain_work > total_main_chain_chainwork {
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [5](#0-4) 

With the correct accumulated chainwork at height 685,440, an attacker starting a fork from the genesis block would need to mine approximately 685,453 valid blocks to outpace the main chain. With the understated genesis `chain_work` (1 block's worth), the attacker only needs to mine approximately 13 blocks — a reduction of roughly 52,000×. The invariant that the light client inherits Bitcoin's full accumulated-work security is broken: the contract accepts a reorg that real Bitcoin nodes would reject outright.

Additionally, `chain_work` is exposed publicly via `get_last_block_header()`: [6](#0-5) 

Any downstream NEAR dApp that reads `chain_work` to assess chain security (e.g., "has this chain accumulated at least X total work?") receives a value that is ~52,000× smaller than reality, leading to incorrect security assessments.

### Likelihood Explanation

**High.** Every production deployment that initializes at a non-zero block height (the standard operational pattern) is affected. The test suite confirms the real deployment uses height 685,440. The bug is triggered unconditionally by the initialization path — no special attacker action is needed to introduce the incorrect state; it is baked in at deploy time.

### Recommendation

Add an `initial_chain_work: U256` field to `InitArgs` and use it as the genesis block's `chain_work` in `init_genesis()`:

```diff
// btc-types/src/contract_args.rs
pub struct InitArgs {
    pub genesis_block_hash: H256,
    pub genesis_block_height: u64,
+   pub initial_chain_work: U256,
    pub skip_pow_verification: bool,
    pub gc_threshold: u64,
    pub network: Network,
    pub submit_blocks: Vec<Header>,
}
```

```diff
// contract/src/lib.rs — init_genesis()
- let chain_work = work_from_bits(block_header.bits);
+ let chain_work = args.initial_chain_work;
```

The deployer must supply the real accumulated chainwork at the genesis height, which can be read from any Bitcoin full node (`getblockheader <hash>` → `chainwork` field).

### Proof of Concept

1. Deploy the contract with `genesis_block_height = 685_440` and 12 init blocks (as in the production test fixture).
2. After init, call `get_last_block_header()`. Observe `chain_work ≈ 12 × work_from_bits(386_752_379)` — approximately `12 × 2^32 / target`.
3. The real Bitcoin accumulated chainwork at height 685,452 is approximately `685,452 × 2^32 / target` — roughly 57,121× larger.
4. A fork attacker who can submit blocks via the trusted-relayer path needs only to build a fork chain from the genesis block whose cumulative work exceeds `12 × work_per_block` (13 valid blocks) to trigger `reorg_chain()`, whereas the correct threshold would require outmining ~685,453 blocks.
