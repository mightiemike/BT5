### Title
Unchecked Integer Overflow in `op_unknown` Cost Multiplication Allows Undercharged Execution — (`File: src/more_ops.rs`)

---

### Summary

In `op_unknown`, the computed cost is validated against `max_cost` **before** a final multiplication by `(cost_multiplier + 1)`. That multiplication is not overflow-checked and, in a release build, silently wraps modulo 2⁶⁴. A crafted unknown opcode can cause the post-multiplication cost to wrap to `0`, bypassing the `u32::MAX` guard and returning `Reduction(0, nil)` — effectively free execution.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` handles unknown opcodes in lenient/consensus mode. It computes a base cost from the opcode's cost-function bits and argument sizes, then scales it by a multiplier encoded in the opcode bytes:

```
check_cost(cost, max_cost)?;      // line 260 — cost ≤ max_cost here
cost *= cost_multiplier + 1;      // line 261 — UNCHECKED u64 multiplication
if cost > u32::MAX as u64 {       // line 262 — guard runs on wrapped value
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
``` [1](#0-0) 

`cost_multiplier` is typed `u64` and is capped at `u32::MAX` (4 294 967 295) by `u32_from_u8`: [2](#0-1) 

So `cost_multiplier + 1` is at most `2³²`. If `cost` (the pre-multiplication value) is any positive multiple of `2³²`, then:

```
cost × 2³² mod 2⁶⁴  =  0
```

The wrapped result `0` passes the `> u32::MAX` guard (false), and `Reduction(0, nil)` is returned — a cost of zero for the entire unknown-op invocation.

The `[profile.release]` section in `Cargo.toml` sets only `lto = "thin"` and does **not** enable `overflow-checks = true`, so Rust's default release-mode wrapping applies: [3](#0-2) 

The `assert!(cost > 0)` at line 258 fires only on the pre-multiplication value and provides no protection after the multiply: [4](#0-3) 

---

### Impact Explanation

An attacker submits a CLVM program containing a crafted unknown opcode. The opcode:

1. Encodes `cost_multiplier = 0xFFFFFFFF` (u32::MAX) in its leading bytes.
2. Uses cost-function bits `01` (ARITH-like) or `11` (CONCAT-like) so the base cost is computed from attacker-supplied arguments.
3. Supplies enough arguments to drive the pre-multiplication cost to exactly `k × 2³²` for some integer `k ≥ 1` (minimum: `4 294 967 296`, well below Chia's consensus `max_cost` of ~11 billion).

The result: the opcode is accepted with a reported cost of `0` instead of `k × 2³² × (0xFFFFFFFF + 1)` ≈ `k × 2⁶⁴`. The running cost total is not incremented, so the attacker can repeat this pattern to execute arbitrarily expensive unknown-op sequences while consuming zero of the block's cost budget — a **consensus-level undercharged execution** vulnerability.

---

### Likelihood Explanation

- `op_unknown` is reachable whenever `allow_unknown_ops()` returns true (lenient/consensus mode), which is the normal path for Chia full nodes evaluating generator programs.
- The attacker controls the opcode bytes and argument list entirely through the serialized CLVM program — a fully attacker-controlled entry path.
- The required pre-multiplication cost (`4 294 967 296`) is below the standard Chia `max_cost` of `11 000 000 000`, so `check_cost` does not block it.
- No special privileges, social engineering, or dependency compromise is required.

---

### Recommendation

Re-check the cost **after** the multiplication using a saturating or checked multiply, and reject if the result exceeds `u32::MAX` or if overflow occurred:

```rust
check_cost(cost, max_cost)?;
let Some(scaled) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o));
};
cost = scaled;
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

Alternatively, enable `overflow-checks = true` in `[profile.release]` in `Cargo.toml` as a defence-in-depth measure, though a `checked_mul` guard is the correct targeted fix.

### Proof of Concept

Craft an opcode atom with:

| Field | Value | Encoding |
|---|---|---|
| Bytes 0–3 (multiplier) | `0xFF 0xFF 0xFF 0xFF` | `u32::MAX = 4 294 967 295` |
| Byte 4 (cost-function) | `0x40` | bits 7–6 = `01` → ARITH cost function |

`cost_multiplier = 0xFFFFFFFF`, `cost_function = 1`.

Supply arguments (atoms) whose total byte count drives the ARITH cost to exactly `4 294 967 296`:

```
cost = ARITH_BASE_COST + n × ARITH_COST_PER_ARG + B × ARITH_COST_PER_BYTE
     = 99 + n × 320 + B × 3  =  4 294 967 296
``` [5](#0-4) 

One solution: `n = 13 421 771`, `B = 19` → `99 + 4 294 966 720 + 57 = 4 294 966 876` — adjust `B` to hit the exact multiple. The `check_cost` at line 260 passes (cost ≤ 11 billion). The multiplication at line 261 produces `4 294 967 296 × 4 294 967 296 = 2⁶⁴ ≡ 0 (mod 2⁶⁴)`. The guard at line 262 passes (`0 > u32::MAX` is false). The function returns `Ok(Reduction(0, nil))`. [6](#0-5)
