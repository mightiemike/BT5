### Title
Residual Non-Zero Allowance in `creditDeposit()` Blocks Future Deposits for Non-Standard ERC20 Tokens - (File: `core/contracts/DirectDepositV1.sol`)

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. `Endpoint.depositCollateralWithReferral()` contains an explicit early-return path that skips the `transferFrom` when the deposit amount is below the minimum threshold, leaving the allowance non-zero. On any subsequent call to `creditDeposit()`, tokens that revert on a non-zero-to-non-zero `approve()` (e.g., USDT-style tokens) will cause the entire function to revert, permanently blocking deposits into the subaccount.

### Finding Description

In `DirectDepositV1.creditDeposit()`, for each product token with a non-zero balance, the contract sets an allowance and then calls the endpoint:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(
    subaccount,
    productId,
    uint128(balance),
    "-1"
);
``` [1](#0-0) 

The critical assumption is that `depositCollateralWithReferral` always consumes the full allowance. However, in `Endpoint.depositCollateralWithReferral()`, there is an explicit early-return path:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
handleDepositTransfer(...);
``` [2](#0-1) 

When this early-return fires, `handleDepositTransfer` is never called, so no `transferFrom` occurs and the allowance set by `approve()` remains fully intact at `balance`. The DDA contract now holds a non-zero allowance for the endpoint.

On the next invocation of `creditDeposit()` (after more tokens arrive), the code again calls `token.approve(address(endpoint), newBalance)`. For any token that enforces the USDT-style restriction — reverting when changing an allowance from a non-zero value to another non-zero value — this `approve()` call reverts, causing the entire `creditDeposit()` loop to revert. Deposits are permanently blocked until the allowance is manually cleared, which `DirectDepositV1` provides no mechanism to do.

Contrast this with `ContractOwner.wrapVaultAsset()`, which correctly uses the two-step reset pattern:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [3](#0-2) 

`creditDeposit()` applies no such reset.

### Impact Explanation

Any subaccount whose DDA holds a USDT-style token and has experienced at least one below-minimum deposit will have its `creditDeposit()` permanently reverting. Tokens sent to the DDA accumulate but cannot be credited to the subaccount. The `withdraw()` function is `onlyOwner`, so the affected user cannot self-rescue their funds. [4](#0-3) 

### Likelihood Explanation

`creditDeposit()` has no access control — any caller can invoke it. [5](#0-4) 

An attacker can deliberately trigger the residual-allowance state by sending a dust amount (below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`) of a qualifying token to the DDA and then calling `creditDeposit()`. This is a low-cost, permissionless griefing vector. Even without a deliberate attacker, organic dust deposits (e.g., from rounding in an upstream protocol) can trigger the same state.

### Recommendation

Reset the allowance to zero before setting a new value, mirroring the pattern already used in `ContractOwner.wrapVaultAsset()`:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, use `safeIncreaseAllowance` from OpenZeppelin's `SafeERC20` library, or check and clear any residual allowance before setting a new one.

### Proof of Concept

1. A DDA is created for `subaccount` via `ContractOwner.createDirectDepositV1()`.
2. Attacker sends 1 wei of a USDT-style token to the DDA address (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. Attacker calls `DirectDepositV1.creditDeposit()`.
   - `token.approve(endpoint, 1)` succeeds — allowance is now 1.
   - `endpoint.depositCollateralWithReferral(subaccount, productId, 1, "-1")` hits the `isValidDepositAmount` check, returns early without calling `handleDepositTransfer`. Allowance remains 1.
4. Legitimate user sends 1000 USDT to the same DDA.
5. Anyone calls `creditDeposit()`.
   - `token.approve(endpoint, 1000e6)` — USDT reverts because existing allowance (1) is non-zero.
   - Entire transaction reverts. The 1000 USDT is stuck in the DDA.
6. The user cannot call `withdraw()` (it is `onlyOwner`). Funds are inaccessible until the protocol owner intervenes. [6](#0-5) [7](#0-6)
