### Title
Linked Signer Can Reassign Its Own Privilege to Persist Access After Owner Revocation — (`core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` transaction handler in `EndpointTx.sol` validates the transaction with `allowLinkedSigner=true`, meaning the **current linked signer** can sign a `LinkSigner` transaction to replace itself with a new address it controls. This is a direct analog to the reported vulnerability: a delegated role holder can propagate its own privilege before the owner revokes it.

---

### Finding Description

In `EndpointTx.sol`, the fast-path handler for `TransactionType.LinkSigner` calls `validateSignedTx` with the fifth argument set to `true` (i.e., `allowLinkedSigner=true`): [1](#0-0) 

This means the signature check inside `validateSignedTx` → `validateCompactSignature` → `verifier.validateCompactSignature` accepts a signature from either the subaccount owner **or** the currently registered linked signer. [2](#0-1) 

The linked signer is stored in `linkedSigners[subaccount]` in `EndpointStorage`: [3](#0-2) 

By contrast, the **slow-mode** `LinkSigner` path uses `validateSender`, which enforces that the on-chain `msg.sender` matches the subaccount owner address — it does **not** allow the linked signer to act: [4](#0-3) 

The asymmetry is the root cause: the fast-path grants the linked signer the ability to overwrite `linkedSigners[sender]` with any address, including a new colluding address.

---

### Impact Explanation

A malicious linked signer can front-run the subaccount owner's revocation transaction by submitting its own `LinkSigner` transaction (signed by itself) to the sequencer, pointing `signer` to a fresh address it controls. Because nonces are consumed sequentially per subaccount address, the owner's pending revocation transaction (carrying the old nonce) will fail once the linked signer's transaction is processed first. The linked signer thereby maintains persistent signing authority over the subaccount and can continue to submit privileged operations — including `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp` — on behalf of the victim subaccount.

**Corrupted state:** `linkedSigners[subaccount]` is set to an attacker-controlled address instead of being cleared, and the subaccount nonce is consumed, invalidating the owner's revocation.

---

### Likelihood Explanation

**Low.** Requires a malicious linked signer who monitors the sequencer submission queue and races to submit a `LinkSigner` transaction before the owner's revocation is sequenced. The sequencer is a centralized component, so the race is off-chain rather than on-chain mempool front-running, making it harder but not impossible — especially if the linked signer is a compromised or adversarial third-party service.

---

### Recommendation

Restrict the fast-path `LinkSigner` handler to only accept signatures from the subaccount owner (i.e., pass `false` for `allowLinkedSigner`), consistent with the slow-mode path's `validateSender` enforcement. A linked signer should never be permitted to reassign the linked signer role for a subaccount it does not own.

```solidity
// EndpointTx.sol — fast-path LinkSigner handler
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // <-- change from true to false
);
```

---

### Proof of Concept

1. Alice owns subaccount `S`. She links `signerA` via a `LinkSigner` transaction.
2. `signerA` turns malicious. Alice submits a fast-path `LinkSigner` transaction (nonce = N) to the sequencer to set `signer = address(0)`.
3. Before the sequencer processes Alice's transaction, `signerA` submits its own fast-path `LinkSigner` transaction for subaccount `S` (nonce = N, signed by `signerA`), setting `signer = signerB` (a fresh address `signerA` controls).
4. The sequencer processes `signerA`'s transaction first: `linkedSigners[S] = signerB`, nonce incremented to N+1.
5. Alice's revocation transaction arrives with nonce = N, which now fails (`ERR_WRONG_NONCE`).
6. `signerB` retains full signing authority over `S` and can call `WithdrawCollateral` or `TransferQuote` to drain Alice's funds.
