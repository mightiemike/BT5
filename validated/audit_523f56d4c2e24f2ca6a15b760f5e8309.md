### Title
Outgoing-Receipt Shard-Routing Invariant Broken Under Consecutive Dynamic Resharding Events — (`File: chain/chain/src/store/mod.rs`)

### Summary

`get_outgoing_receipts_for_shard_from_store` contains a self-documented, load-bearing assumption that **at most one shard layout change** can occur between a chunk's `last_included_height` and the current `prev_block_hash`. With dynamic resharding now stabilized (NEAR 2.13.0, `ProtocolFeature::DynamicResharding`), shard layouts can change every epoch (`min_epochs_between_resharding ≥ 1`). When a shard has a missing chunk spanning two consecutive resharding epochs, the function computes the wrong ancestor shard ID, causing all outgoing receipts generated by the grandparent shard to be silently dropped — permanently losing cross-shard receipt delivery.

### Finding Description

`get_outgoing_receipts_for_shard_from_store` retrieves the outgoing receipts that a child shard must forward after a resharding event. It does so by:

1. Reading `shard_layout` (current layout at `prev_block_hash`)
2. Walking back to `last_included_height` and reading `receipts_shard_layout` (old layout)
3. If they differ, calling `shard_layout.get_parent_shard_id(shard_id)` to find the parent shard that originally generated the receipts
4. Fetching `get_outgoing_receipts(receipts_block_hash, receipts_shard_id)` [1](#0-0) 

The code itself documents the broken invariant:

> "Note, the current way of implementation assumes that at least one chunk is generated before shard layout are changed twice." [1](#0-0) 

When two layout changes occur between `last_included_height` and `prev_block_hash`:

- `shard_id` is a **grandchild** shard (two generations newer than the shard at `last_included_height`)
- `shard_layout.get_parent_shard_id(shard_id)` returns the **intermediate parent** (one generation back), not the grandparent
- The code then fetches `get_outgoing_receipts(receipts_block_hash, intermediate_parent_id)` — but the intermediate parent did not exist at `receipts_block_hash` (it was created in the first resharding, which happened after `last_included_height`)
- `get_outgoing_receipts` returns `None` / empty, so the receipts vector is empty
- `reassign_outgoing_receipts_for_resharding` then either returns empty receipts for the left child or clears them for the right child [2](#0-1) [3](#0-2) 

The grandparent's outgoing receipts — which were correctly generated and stored at `last_included_height` — are permanently lost. They will never be delivered to their destination shards.

### Impact Explanation

Cross-shard receipts generated before two consecutive resharding events, during a period of missing chunks, are permanently dropped. Since receipts carry token transfers and contract calls, this results in:

- Funds transferred cross-shard being permanently inaccessible (locked in the sending shard's state, never credited to the receiver)
- Contract calls silently failing with no error surfaced to the user
- No recovery path: the receipts are not queued, not retried, and not refunded

This is a **Critical** impact: permanent, silent loss of user funds and state.

### Likelihood Explanation

Dynamic resharding is now stabilized and active on mainnet (NEAR 2.13.0). The `min_epochs_between_resharding` parameter is configurable with a minimum of 1, meaning two resharding events can occur in consecutive epochs. Missing chunks spanning 2+ epochs is unusual under normal conditions but is reachable during:

- Chunk producer downtime (offline validator)
- Network partitions or high-latency periods
- State sync catchup scenarios where `last_included_height` lags significantly

The combination is **Medium** likelihood: individually each condition is uncommon, but both are reachable in production without any privileged action.

### Recommendation

Replace the single-step parent lookup with a multi-generation ancestor walk. When `shard_layout != receipts_shard_layout`, iteratively resolve the ancestor shard ID by walking the split history until the layout at `last_included_height` is reached, rather than assuming a single `get_parent_shard_id` call suffices. The `ShardLayoutV3` ancestor map (`ancestor_uids()`) already supports O(1) multi-generation ancestor lookup and should be used here. The existing GitHub issue [#4877](https://github.com/near/nearcore/issues/4877) tracks this.

### Proof of Concept

**Setup**: Dynamic resharding enabled, `min_epochs_between_resharding = 1`, epoch length = 10 blocks.

**Sequence**:

1. **Epoch N, block H**: Account `alice.near` (on shard S) sends a cross-shard transfer to `bob.near` (on shard T). The outgoing receipt is stored at `OutgoingReceipts(H, S)`. Shard T's chunk at height H is the `last_included_height` for shard T.

2. **Epoch N → N+1 boundary**: Shard T splits into T_left and T_right (first resharding). Shard T's chunk producer goes offline; T_left and T_right produce no chunks for the rest of epoch N+1.

3. **Epoch N+1 → N+2 boundary**: T_left splits into T_left_a and T_left_b (second resharding, permitted since `min_epochs_between_resharding = 1`).

4. **Epoch N+2, first block**: `validate_chunk_with_chunk_extra` is called for T_left_a. It calls `get_outgoing_receipts_for_shard(prev_block_hash, T_left_a_id, H)`.

5. **Inside `get_outgoing_receipts_for_shard_from_store`**:
   - `shard_layout` = layout at epoch N+2 (T_left_a exists)
   - `receipts_shard_layout` = layout at epoch N (only S and T exist)
   - `shard_layout != receipts_shard_layout` → true
   - `shard_layout.get_parent_shard_id(T_left_a_id)` → returns `T_left_id` (intermediate parent, epoch N+1)
   - `get_outgoing_receipts(H, T_left_id)` → **returns empty** (T_left didn't exist at height H)
   - Receipts vector is empty; `reassign_outgoing_receipts_for_resharding` returns empty

6. **Result**: The outgoing receipt from `alice.near` to `bob.near` is permanently lost. `bob.near` never receives the transfer. [4](#0-3) [5](#0-4)

    pub fn get_outgoing_receipts_for_shard_from_store(
        chain_store: &ChainStoreAdapter,
        epoch_manager: &dyn EpochManagerAdapter,
        prev_block_hash: CryptoHash,
        shard_id: ShardId,
        last_included_height: BlockHeight,
    ) -> Result<Vec<Receipt>, Error> {
        let shard_layout = epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?;
        let mut receipts_block_hash = prev_block_hash;
        loop {
            let block_header = chain_store.get_block_header(&receipts_block_hash)?;

            if block_header.height() != last_included_height {
                receipts_block_hash = *block_header.prev_hash();
                continue;
            }
            let receipts_shard_layout = epoch_manager.get_shard_layout(block_header.epoch_id())?;

            // get the shard from which the outgoing receipt were generated
            let receipts_shard_id = if shard_layout != receipts_shard_layout {
                shard_layout.get_parent_shard_id(shard_id)?
            } else {
                shard_id
            };

            let mut receipts = chain_store
                .get_outgoing_receipts(&receipts_block_hash, receipts_shard_id)
                .map(|v| v.to_vec())
                .unwrap_or_default();

            if shard_layout != receipts_shard_layout {
                // the shard layout has changed so we need to reassign the outgoing receipts
                Self::reassign_outgoing_receipts_for_resharding(
                    &mut receipts,
                    &shard_layout,
                    shard_id,
                    receipts_shard_id,
                )?;
            }

            return Ok(receipts);
        }
    }

    pub fn reassign_outgoing_receipts_for_resharding(
        receipts: &mut Vec<Receipt>,
        shard_layout: &ShardLayout,
        shard_id: ShardId,
        receipts_shard_id: ShardId,
    ) -> Result<(), Error> {
        tracing::debug!(target: "resharding", %shard_id, ?receipts_shard_id, "reassign_outgoing_receipts_for_resharding");

        let split_shard_ids = shard_layout.get_children_shards_ids(receipts_shard_id);
        let split_shard_ids =
            split_shard_ids.ok_or(Error::InvalidSplitShardsIds(shard_id, receipts_shard_id))?;

        // The target shard id is the split shard with the lowest shard id.
        let target_shard_id = split_shard_ids.iter().min();
        let target_shard_id =
            *target_shard_id.ok_or(Error::InvalidSplitShardsIds(shard_id, receipts_shard_id))?;

        if shard_id == target_shard_id {
            // This shard_id is the lowest index child, it gets all the receipts.
            Ok(())
        } else {
            // This shard_id is not the lowest index child, it gets no receipts.
            receipts.clear();
            Ok(())
        }
    }

/// Validate that all next chunk information matches previous chunk extra.
pub fn validate_chunk_with_chunk_extra(
    chain_store: &ChainStore,
    epoch_manager: &dyn EpochManagerAdapter,
    prev_block_hash: &CryptoHash,
    prev_chunk_extra: &ChunkExtra,
    prev_chunk_height_included: BlockHeight,
    chunk_header: &ShardChunkHeader,
) -> Result<Vec<Receipt>, Error> {
    let outgoing_receipts = chain_store.get_outgoing_receipts_for_shard(
        epoch_manager,
        *prev_block_hash,
        chunk_header.shard_id(),
        prev_chunk_height_included,
    )?;
    let outgoing_receipts_hashes = {
        let shard_layout = epoch_manager.get_shard_layout_from_prev_block(prev_block_hash)?;
        Chain::build_receipts_hashes(&outgoing_receipts, &shard_layout)?
    };
    let (outgoing_receipts_root, _) = merklize(&outgoing_receipts_hashes);

    validate_chunk_with_chunk_extra_and_receipts_root(
        prev_chunk_extra,
        chunk_header,
        &outgoing_receipts_root,
    )?;

    Ok(outgoing_receipts)
}
