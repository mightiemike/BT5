### Title
Protobuf `ValidResourceBounds` Round-Trip Collapses `AllResources(X,0,0)` to `L1Gas(X)`, Producing a Different Transaction Hash - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` silently changes the variant from `AllResources` to `L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant (2 for `L1Gas`, 3 for `AllResources`), a V3 transaction whose hash was computed with `AllResources` at the gateway arrives at a peer node with a structurally different `ValidResourceBounds`, causing the peer to compute a different hash or to reject the transaction outright with `DEPRECATED_RESOURCE_BOUNDS_ERROR`.

### Finding Description

**Step 1 – Gateway accepts the transaction with `AllResources`.**

`RpcInvokeTransactionV3` carries `resource_bounds: AllResourceBounds`. The gateway's `convert_rpc_tx_to_internal` converts it to `InvokeTransactionV3` via:

```rust
// From<RpcInvokeTransactionV3> for InvokeTransactionV3
resource_bounds: ValidResourceBounds::AllResources(tx.resource_bounds),
``` [1](#0-0) 

The hash is then computed by `get_tip_resource_bounds_hash` with the `AllResources` branch, which appends **three** resource felts (`l1_gas`, `l2_gas`, `l1_data_gas`) to the hash chain:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts total
    }
});
``` [2](#0-1) 

**Step 2 – Serialization to protobuf preserves all three fields.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits `l1_data_gas: Some(...)`, so `AllResources(X, 0, 0)` becomes `{ l1_gas: Some(X), l2_gas: Some(0), l1_data_gas: Some(0) }` on the wire. [3](#0-2) 

**Step 3 – Deserialization silently changes the variant.**

On the receiving peer, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` checks:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // = 0
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // ← variant changed!
} else {
    ValidResourceBounds::AllResources(...)
})
``` [4](#0-3) 

`AllResources(X, 0, 0)` becomes `L1Gas(X)`. The round-trip is not identity-preserving.

**Step 4 – Two concrete failure modes.**

*4a – Hash divergence.* If the peer recomputes the hash (e.g., in the consensus `convert_consensus_tx_to_internal_consensus_tx` path), `get_tip_resource_bounds_hash` now hashes only **two** resource felts instead of three, producing `H2 ≠ H1`. The peer stores or validates the transaction under the wrong hash.

*4b – Hard rejection.* In the consensus P2P path, `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [5](#0-4) 

Because `snapi_invoke.resource_bounds` is now `L1Gas`, the `try_into()` to `RpcInvokeTransactionV3` (which requires `AllResourceBounds`) fails, and the peer returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`, rejecting a valid transaction.

### Impact Explanation

A V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway (the stateless validator explicitly allows this pattern, as shown in `valid_l1_gas` test case) and assigned hash H1. When the proposer broadcasts this transaction to validators via P2P consensus, every validator deserializes the resource bounds as `L1Gas(X)` and either rejects the transaction with `DEPRECATED_RESOURCE_BOUNDS_ERROR` or computes hash H2 ≠ H1. The proposer's block is rejected by all validators, causing a consensus-level denial of service for any block containing such a transaction. [6](#0-5) 

### Likelihood Explanation

Any user who submits a V3 invoke transaction specifying only L1 gas (setting `l2_gas` and `l1_data_gas` to zero) triggers this path. This is a natural usage pattern for pre-0.13.3 style transactions submitted through the new V3 interface. No special privileges are required; a single such transaction in a proposed block is sufficient to cause the block to be rejected by all validators.

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not infer the variant from the field values. Instead, use an explicit discriminant field in the protobuf schema, or always deserialize to `AllResources` when all three fields are present (even if zero), preserving the original variant:

```rust
// Always produce AllResources when l1_data_gas is present in the message,
// regardless of whether its value is zero.
Ok(match value.l1_data_gas {
    Some(l1_data_gas_proto) => {
        let l1_data_gas: ResourceBounds = l1_data_gas_proto.try_into()?;
        ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
    }
    None => ValidResourceBounds::L1Gas(l1_gas),
})
```

This preserves the round-trip identity: `AllResources(X, 0, 0)` serializes with `l1_data_gas: Some(0)` and deserializes back to `AllResources(X, 0, 0)`, keeping the hash stable.

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: ResourceBounds { max_amount: 1000, max_price_per_unit: 1 }, l2_gas: ResourceBounds::default(), l1_data_gas: ResourceBounds::default() }`.
2. Compute its hash H1 via `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` — this uses `ValidResourceBounds::AllResources` and hashes 3 resource felts.
3. Serialize the transaction's `ValidResourceBounds::AllResources(...)` to `protobuf::ResourceBounds` via `From<ValidResourceBounds>` — produces `{ l1_gas: Some(X), l2_gas: Some(0), l1_data_gas: Some(0) }`.
4. Deserialize back via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` — produces `ValidResourceBounds::L1Gas(X)` because `l1_data_gas.is_zero() && l2_gas.is_zero()`.
5. Compute hash H2 from the deserialized `ValidResourceBounds::L1Gas(X)` — this hashes only 2 resource felts.
6. Assert `H1 ≠ H2` — the hash has changed after a single protobuf round-trip, demonstrating that the peer node will compute a different transaction hash than the one stored by the gateway, causing the transaction to be rejected during consensus validation.

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
