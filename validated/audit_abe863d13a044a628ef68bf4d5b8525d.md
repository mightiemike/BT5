### Title
Missing `#[trusted_relayer]` Guard on `run_mainchain_gc` Allows Unprivileged Callers to Aggressively Prune the Canonical Chain — (`contract/src/lib.rs`)

### Summary

`run_mainchain_gc` is a public, state-mutating NEAR contract method protected only by a pause gate. Any unprivileged NEAR account can call it directly with an arbitrarily large `batch_size`, forcing the contract to remove all blocks currently eligible for GC in a single transaction. This advances `mainchain_initial_blockhash` to the maximum extent permitted by the GC logic, immediately invalidating all pending `verify_transaction_inclusion` calls for the pruned blocks.

### Finding Description

`submit_blocks` carries both `#[pause]` and `#[trusted_relayer]`, restricting block submission to staked, authorized relayers: [1](#0-0) 

`run_mainchain_gc`, by contrast, carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]` — no `#[trusted_relayer]` attribute: [2](#0-1) 

Inside the function, the number of blocks actually removed is bounded by:

```
selected_amount_to_remove = min(amount_of_headers_we_store - gc_threshold, batch_size)
``` [3](#0-2) 

When a caller supplies `batch_size >= (amount_of_headers_we_store - gc_threshold)`, every block currently eligible for removal is deleted in one call. `mainchain_initial_blockhash` is then advanced to the new oldest block: [4](#0-3) 

Under normal operation (via `submit_blocks`), GC is gradual — `batch_size` equals the number of headers submitted in that call, so removal is incremental. The public entrypoint removes this throttle entirely.

### Impact Explanation

After aggressive GC, `mainchain_header_to_height` no longer contains the pruned block hashes. Any call to `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a pruned block panics at:

```
.unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"))
``` [5](#0-4) 

All SPV proofs for blocks below the new `mainchain_initial_blockhash` are permanently invalidated. Contracts or users relying on historical inclusion proofs (e.g., cross-chain bridges, payment verification) will receive hard panics instead of valid results.

### Likelihood Explanation

The attack requires no special role, no staked deposit, and no prior relationship with the contract. Any NEAR account can call `run_mainchain_gc` with `batch_size = u64::MAX` whenever the contract is unpaused and the mainchain has grown beyond `gc_threshold`. The existing test suite even calls `run_mainchain_gc` from an unprivileged `user_account` and asserts success: [6](#0-5) 

### Recommendation

Add `#[trusted_relayer]` to `run_mainchain_gc`, mirroring `submit_blocks`. If permitting DAO-authorized accounts to trigger manual GC is desired, add `Role::UnrestrictedRunGC` as a bypass role in the `#[trusted_relayer]` attribute, consistent with how `Role::UnrestrictedSubmitBlocks` is used for `submit_blocks`: [7](#0-6) 

### Proof of Concept

1. Deploy the contract with `gc_threshold = 52704` and submit enough blocks so `amount_of_headers_we_store > gc_threshold`.
2. From any unprivileged NEAR account (no role granted), call:
   ```json
   { "batch_size": 52704 }
   ```
   on `run_mainchain_gc`.
3. Assert the call succeeds and `get_mainchain_size()` returns exactly `gc_threshold`.
4. Call `verify_transaction_inclusion` for any block that was at a height below the new `mainchain_initial_blockhash`; observe it panics with `"block does not belong to the current main chain"`.

The existing test at `contract/tests/test_basics.rs:515–521` already demonstrates step 2 succeeding from an unprivileged account. [6](#0-5)
