Let me look at the OffchainExchange health checks and the RiskHelper to understand the INITIAL vs MAINTENANCE health weight difference.

### Title
`burnNlp` Enforces Weaker MAINTENANCE Health Check While `mintNlp` Enforces INITIAL Health Check, Allowing Bypass of Initial Margin Requirement — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`mintNlp` enforces an `INITIAL` health check after execution, but `burnNlp` only enforces a `MAINTENANCE` health check. Because `INITIAL` uses more conservative risk weights (`longWeightInitialX18`, `shortWeightInitialX18`) than `MAINTENANCE` (`longWeightMaintenanceX18`, `shortWeightMaintenanceX18`), a user can mint NLP, take on additional leveraged risk to push their INITIAL health negative while keeping MAINTENANCE health positive, then burn NLP to receive quote — bypassing the INITIAL margin requirement that would otherwise block any value-extracting operation.

---

### Finding Description

In `Clearinghouse.sol`, the two NLP operations apply asymmetric post-state health checks:

**`mintNlp`** (user pays quote, receives NLP): [1](#0-0) 

**`burnNlp`** (user burns NLP, receives quote): [2](#0-1) 

The `INITIAL` health type applies `longWeightInitialX18` / `shortWeightInitialX18` from `RiskHelper.Risk`, which are more conservative (lower leverage ceiling). The `MAINTENANCE` health type applies `longWeightMaintenanceX18` / `shortWeightMaintenanceX18`, which are less conservative (higher leverage ceiling): [3](#0-2) 

This means there exists a band of account states where `INITIAL` health is negative but `MAINTENANCE` health is positive. In this band, `withdrawCollateral` is blocked (it requires `INITIAL` health ≥ 0): [4](#0-3) 

But `burnNlp` is **not** blocked, because it only requires `MAINTENANCE` health ≥ 0. The burn credits the user's quote balance: [5](#0-4) 

---

### Impact Explanation

A user can receive quote (real value) from the NLP system while their INITIAL health is negative — a state that should prevent any value-extracting operation. The received quote improves the user's quote balance, which in turn improves their INITIAL health. Once INITIAL health is restored to ≥ 0, the user can call `withdrawCollateral` to extract the quote as real tokens. This allows a user to effectively bypass the initial margin requirement to extract collateral that the protocol's risk model says they should not be able to access.

The corrupted invariant: **a subaccount with negative INITIAL health must not be able to receive quote from the protocol**. `burnNlp` violates this invariant.

---

### Likelihood Explanation

Any user who has minted NLP and subsequently opened leveraged positions (spot borrows or perp positions) that push their INITIAL health negative while keeping MAINTENANCE health positive can trigger this. This is a realistic account state — the gap between INITIAL and MAINTENANCE weights is a designed feature of the risk system, and users with leveraged perp positions near the INITIAL margin boundary can reach it. No privileged access, governance capture, or external dependency is required. The user submits a signed `BurnNlp` transaction through the standard `Endpoint` flow.

---

### Recommendation

Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with `mintNlp`:

```solidity
// In burnNlp, replace:
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);

// With:
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

The comment in the code ("Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal") describes a valid concern, but it is fully addressed by an `INITIAL` check — if the burn would leave INITIAL health negative, it reverts. The weaker `MAINTENANCE` check is not necessary to satisfy this concern and introduces the bypass described above. [6](#0-5) 

---

### Proof of Concept

1. Alice has 1000 USDC collateral and opens a leveraged perp position such that her INITIAL health = +5 (barely positive) and MAINTENANCE health = +50.
2. Alice calls `mintNlp` with `quoteAmount = 5`. INITIAL health check passes (health = 0 after mint). Alice now holds NLP worth 5 USDC.
3. Alice opens an additional small perp position, pushing her INITIAL health to −3 while MAINTENANCE health remains +45.
4. Alice submits a signed `BurnNlp` transaction. `burnNlp` executes:
   - Alice's NLP balance decreases.
   - Alice's quote balance increases by ~5 USDC (minus burn fee).
   - MAINTENANCE health check passes (health = +45 + improvement from quote ≥ 0). ✓
   - INITIAL health check is **not performed**.
5. Alice's quote balance is now +5 USDC and her INITIAL health has improved from −3 to ~+2.
6. Alice calls `withdrawCollateral` for the quote. INITIAL health ≥ 0 passes. Alice withdraws ~5 USDC.

Alice has extracted ~5 USDC from the NLP pool while her account was in a state (negative INITIAL health) that should have blocked all value-extracting operations. [7](#0-6)

        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
