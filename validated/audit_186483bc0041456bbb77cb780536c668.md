### Title
Undersized Bitmask in `getIsolatedProductId` Silently Truncates Product ID to 16 Bits Despite `uint32` Return Type — (`File: core/contracts/libraries/RiskHelper.sol`)

---

### Summary

`RiskHelper.getIsolatedProductId` declares a `uint32` return type but applies a `0xFFFF` (16-bit) mask when extracting the product ID from the packed `bytes32` subaccount. For any product with ID > 65535, the function silently returns a wrong, truncated value. All callers that rely on this value to query or update perp engine balances operate on the wrong product, corrupting isolated subaccount closure and reuse logic.

---

### Finding Description

The isolated subaccount `bytes32` is constructed in `OffchainExchange.createIsolatedSubaccount` as:

```
// |  address | reserved | productId |   id   |  'iso'  |
// | 20 bytes |  6 bytes |  2 bytes  | 1 byte | 3 bytes |
newIsolatedSubaccount = bytes32(
    (uint256(uint160(senderAddress)) << 96) |
    (uint256(txn.productId) << 32)          |  // txn.productId is uint32
    (uint256(id) << 24)                     |
    6910831
);
``` [1](#0-0) 

`txn.productId` is `uint32` (32 bits), but the layout comment allocates only 2 bytes (16 bits) for the productId field (bits 32–47). When `txn.productId > 0xFFFF`, the upper 16 bits overflow into the "reserved" field (bits 48–63), corrupting the subaccount encoding.

The read-back function is:

```solidity
function getIsolatedProductId(bytes32 subaccount)
    internal pure returns (uint32)
{
    if (!isIsolatedSubaccount(subaccount)) { return 0; }
    return uint32((uint256(subaccount) >> 32) & 0xFFFF);  // mask is 16-bit only
}
``` [2](#0-1) 

The return type promises `uint32` (32 bits), but the mask `0xFFFF` captures only 16 bits. The upper 16 bits of any product ID are always zeroed out. This is structurally identical to the reported `slice4` bug: the declared output width (`uint32`) is wider than the actual extraction width (`0xFFFF` = 16 bits), so the returned value is silently wrong for any product ID > 65535.

---

### Impact Explanation

**`_tryCloseIsolatedSubaccount`** calls `getIsolatedProductId` and uses the result directly to query and mutate perp engine state:

```solidity
uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
IPerpEngine.Balance memory balance = perpEngine.getBalance(productId, subaccount);
if (balance.amount == 0) {
    // transfers vQuoteBalance and quoteBalance to parent
    perpEngine.updateBalance(productId, subaccount, 0, -balance.vQuoteBalance);
    perpEngine.updateBalance(productId, parent, 0, balance.vQuoteBalance);
    ...
}
``` [3](#0-2) 

If `productId` is truncated (e.g., actual product 65537 → returned as 1), the balance check queries the wrong product. If that wrong product shows `amount == 0`, the closure proceeds: `vQuoteBalance` is transferred between the isolated subaccount and parent under the wrong product ID, corrupting both accounts' perp balances. The actual position in product 65537 is never settled.

**`createIsolatedSubaccount`** uses `getIsolatedProductId` to check whether an existing isolated subaccount already covers the requested product:

```solidity
uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
if (productId == txn.productId) {
    newIsolatedSubaccount = subaccount;
    break;
}
``` [4](#0-3) 

For `txn.productId > 0xFFFF`, the comparison always fails (truncated value ≠ full value), so a new isolated subaccount is created instead of reusing the existing one. This bypasses the deduplication guard and allows a user to accumulate multiple isolated subaccounts for the same product, violating the one-subaccount-per-product invariant.

---

### Likelihood Explanation

The trigger requires a product ID > 65535 to exist in the protocol. `txn.productId` is `uint32` throughout all interfaces, and no on-chain enforcement caps it at 65535. As the protocol scales or if a product is registered with a high numeric ID, this becomes directly exploitable by any user submitting a `CreateIsolatedSubaccount` transaction — an unprivileged, externally reachable entry point. The attacker does not need any special role; they only need to interact with a product whose ID exceeds 16 bits.

---

### Recommendation

Change the mask in `getIsolatedProductId` from `0xFFFF` to `0xFFFFFFFF` to match the `uint32` return type:

```solidity
return uint32((uint256(subaccount) >> 32) & 0xFFFFFFFF);
```

Additionally, update the subaccount layout comment and the encoding in `createIsolatedSubaccount` to allocate 4 bytes (not 2) for the productId field, shifting the reserved region accordingly, so the encoding and decoding are consistent for the full `uint32` range.

---

### Proof of Concept

1. A product with ID `0x10001` (65537) is registered in the protocol.
2. A user calls `createIsolatedSubaccount` with `txn.productId = 65537`.
3. The subaccount is encoded as:
   ```
   (uint256(65537) << 32) = 0x0000_0001_0001_0000_0000_0000
   ```
   Bits 48–63 (the "reserved" field) now have bit 48 set — the encoding is corrupted.
4. Later, `_tryCloseIsolatedSubaccount` is called on this subaccount.
5. `getIsolatedProductId` returns `uint32(0x10001 & 0xFFFF) = 1` instead of `65537`.
6. `perpEngine.getBalance(1, subaccount)` is called — querying product 1's balance for this subaccount, which is 0.
7. Since `balance.amount == 0`, the closure proceeds: `vQuoteBalance` is moved between accounts under product 1, not product 65537.
8. The user's actual open position in product 65537 is never closed, and the parent account receives a spurious vQuote credit under product 1. [2](#0-1) [5](#0-4)
