### Title
`run_mainchain_gc()` Is Unrestricted Public, Allowing Any Caller to Permanently Delete Mainchain Block Headers - (`contract/src/lib.rs`)

### Summary
`BtcLightClient::run_mainchain_gc()` is a state-mutating NEAR public method with no caller restriction. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, immediately purging the maximum number of removable mainchain block headers in a single transaction. This permanently deletes those headers from on-chain storage and advances `mainchain_initial_blockhash`, making SPV proofs for the purged blocks permanently unverifiable.

### Finding Description
`run_mainchain_gc` is declared `pub` and carries only `#[pause(except(roles(Role::UnrestrictedRunGC)))]`: [1](#0-0) 

The `#[pause]` macro only gates the function when the contract is **paused**; when the contract is live (the normal operating state), any NEAR account can call it freely. There is no `#[trusted_relayer]`, no `#[private]`, and no role check on the caller.

The intended call path is internal: `submit_blocks` calls `run_mainchain_gc(num_of_headers)` where `num_of_headers` is the count of headers just submitted (typically a small number), so GC is amortized and gradual: [2](#0-1) 

An attacker bypasses this by calling `run_mainchain_gc` directly with `batch_size = u64::MAX`. Inside the function, `selected_amount_to_remove` is clamped to `total_amount_to_remove = amount_of_headers_we_store - gc_threshold`: [3](#0-2) 

This removes every block header that is eligible for GC in a single call. Each removed header is permanently erased from `headers_pool`, `mainchain_header_to_height`, and `mainchain_height_to_header`, and `mainchain_initial_blockhash` is advanced to the new oldest block: [4](#0-3) 

The deletion is irreversible. `submit_block_header` requires the previous block to already be in `headers_pool`; once a block is GC'd its predecessor is also gone, so the removed headers cannot be resubmitted.

### Impact Explanation
Any block header removed by the attacker-triggered GC is permanently absent from `mainchain_header_to_height`. The SPV verification path in `verify_transaction_inclusion` looks up the target block in that map and panics if it is missing: [5](#0-4) 

`verify_transaction_inclusion_v2` delegates to the same path: [6](#0-5) 

Any downstream contract or bridge that relies on the light client to confirm Bitcoin transaction inclusion will receive a hard panic (not a `false` return) for any block that the attacker has purged. Because the deletion is permanent and the blocks cannot be resubmitted, the impact is a permanent, irreversible corruption of the verifiable block range — not merely a transient denial of service.

### Likelihood Explanation
The attack requires no special role, no staked funds, and no privileged key. Any NEAR account can call `run_mainchain_gc` with `batch_size = u64::MAX` at any time the contract is unpaused. The only precondition is that the stored chain length exceeds `gc_threshold` (the recommended value is 52 704 blocks, roughly one year of Bitcoin history), which is the normal steady-state for a production deployment. The attack is a single cheap NEAR transaction.

### Recommendation
Remove the `pub` visibility from `run_mainchain_gc` so it can only be called internally by `submit_blocks`. If an externally callable GC endpoint is genuinely needed (e.g., for operational maintenance), gate it with an explicit role check (e.g., `Role::DAO` or a dedicated `GCManager` role) using `#[access_control_any(roles(Role::DAO))]` in addition to the existing `#[pause]` guard.

```rust
// Before (vulnerable)
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }

// After (fixed — internal only)
fn run_mainchain_gc(&mut self, batch_size: u64) { ... }
```

### Proof of Concept
1. Deploy the contract with `gc_threshold = 52704` and sync it until `amount_of_headers_we_store > gc_threshold` (e.g., 52 705 headers stored).
2. A user prepares a valid SPV proof for a transaction in the oldest stored block (height `H`).
3. Attacker calls:
   ```
   run_mainchain_gc { batch_size: 18446744073709551615 }
   ```
   This removes `52705 − 52704 = 1` block (block at height `H`) from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, and advances `mainchain_initial_blockhash` to height `H+1`.
4. User calls `verify_transaction_inclusion_v2` with the proof for block `H`. The contract panics: `"block does not belong to the current main chain"` because `mainchain_header_to_height.get(&H)` returns `None`.
5. The block at height `H` can never be resubmitted because its predecessor (height `H−1`) is also absent from `headers_pool`.

The test suite already demonstrates that an unprivileged `user_account` can call `run_mainchain_gc` directly: [7](#0-6)
