Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Actual Swapper, Permanently Breaking Per-User Allowlist Enforcement - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the pool's `msg.sender` ‚Äî the router contract ‚Äî when users route through `MetricOmmSimpleRouter`. This makes it impossible to simultaneously allow router-mediated swaps and restrict access to specific users: either the router is allowlisted (opening the pool to all users) or it is not (blocking all router users, including individually allowlisted ones). The deposit allowlist correctly uses the `owner` parameter (the actual position owner), but the swap allowlist has no equivalent identity anchor.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value directly to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`: [3](#0-2) 

When `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)`, the pool's `msg.sender` is the router contract, not the end user: [4](#0-3) 

So `allowedSwapper[pool][sender]` resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter (the actual position owner), which is always the real user regardless of who calls `addLiquidity`: [5](#0-4) 

No equivalent identity anchor exists in the swap path ‚Äî the `recipient` (second argument, currently ignored) is caller-controlled and not a reliable proxy for the actual swapper.

## Impact Explanation

Two mutually exclusive failure modes arise for any pool using `SwapAllowlistExtension` with `MetricOmmSimpleRouter`:

1. **Allowlist bypass**: If the pool admin allowlists the router address to enable router-mediated swaps, every unpermissioned user can also swap through the router ‚Äî the per-user restriction is completely defeated. Any user can call `exactInputSingle` and the extension checks `allowedSwapper[pool][router]` ‚Üí `true` ‚Üí swap proceeds regardless of the caller's identity.
2. **Broken core functionality**: If the pool admin does not allowlist the router, all router-mediated swaps revert for every user, including those individually allowlisted ‚Äî the supported periphery path is entirely unusable.

This constitutes broken core pool functionality: the swap allowlist ‚Äî a primary access-control mechanism for curated pools ‚Äî is rendered ineffective for the canonical swap path. Pools intended to restrict access to specific market makers or permissioned users are either fully open or fully closed to router users, with no valid intermediate configuration.

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects users to interact via `MetricOmmSimpleRouter` hits this immediately. Allowlisting the router is the natural and expected administrative action to enable router support, making the bypass scenario the default outcome. No special attacker capability is required ‚Äî a standard `exactInputSingle` call suffices.

## Recommendation

Pass the original end-user address through the swap call so the extension can check it. Options include:

- Add a `swapper` field to swap parameters that the router populates with `msg.sender` before calling the pool, and have the pool forward it as a distinct argument to the extension (separate from `sender`).
- Have `SwapAllowlistExtension` decode the actual user address from `extensionData`, with the router encoding `msg.sender` into `extensionData` before calling the pool.
- Gate on `recipient` only if the protocol enforces that `recipient` must equal the economic beneficiary (the actual user), mirroring the deposit allowlist's `owner` pattern ‚Äî but this requires a protocol-level invariant that `recipient` cannot be set to an arbitrary address.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to allow router-mediated swaps.
3. Unpermissioned user `Charlie` (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)`.
4. The pool calls `_beforeSwap(router, ...)` ‚Üí extension checks `allowedSwapper[pool][router]` ‚Üí `true` ‚Üí swap proceeds.
5. Charlie successfully swaps in a pool that was supposed to restrict him, because the router address ‚Äî not Charlie's address ‚Äî is what the allowlist checks.
