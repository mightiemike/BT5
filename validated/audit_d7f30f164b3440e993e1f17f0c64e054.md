### Title
`verify_transaction_inclusion` Returns `true` for Forged Internal-Node Merkle Proofs Without Reverting — (`contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function remains publicly callable and returns `bool` rather than reverting on invalid proofs. An unprivileged NEAR caller can supply an internal Merkle tree node hash as `tx_id` with a shortened proof path, causing the function to return `true` for a transaction that was never included in the block. Any downstream NEAR contract that calls v1 and trusts the result will silently accept a forged transaction inclusion proof — the direct analog of an ERC20 token returning `success = true` while the actual transfer did not occur as expected.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` computes a Merkle root from the caller-supplied `tx_id`, `tx_index`, and `merkle_proof`, then compares it to the stored `merkle_root` in the block header. [1](#0-0) 

The function does not validate that `tx_id` corresponds to a leaf-level transaction — it only checks structural Merkle proof validity. The underlying `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` does not validate that the proof length is consistent with `tx_index`: [2](#0-1) 

This means an attacker can treat any internal Merkle tree node as if it were a leaf. For a block with transactions `[A, B, C, D]`:

```
        root
       /    \
     AB      CD
    /  \    /  \
   A    B  C    D
```

The attacker provides:
- `tx_id` = `AB` (= `double_sha256(A || B)`, an internal node)
- `tx_index` = `0`
- `merkle_proof` = `[CD]`

`compute_root_from_merkle_proof(AB, 0, [CD])` computes `compute_hash(AB, CD)` = `root`, which matches the stored `merkle_root`. The function returns `true`, falsely asserting that `AB` is an included transaction.

The function's own docstring acknowledges this: [3](#0-2) 

`verify_transaction_inclusion_v2` was introduced to mitigate this via a coinbase proof requirement: [4](#0-3) 

However, v1 remains directly callable by any unprivileged NEAR caller, bypassing the coinbase check entirely.

---

### Impact Explanation

The corrupted value is the proof result: `verify_transaction_inclusion` returns `true` for a forged proof. Downstream NEAR contracts that call v1 and trust the `bool` result will silently accept forged transaction inclusion proofs. If such contracts gate fund releases on proof of a Bitcoin payment, an attacker can claim a payment occurred when it did not, causing direct financial loss. This is the exact analog of the ERC20 issue: the function returns a "success" signal that callers trust, but the underlying operation (proof validity) did not hold.

---

### Likelihood Explanation

**Medium.** The 64-byte transaction Merkle forgery is a well-documented Bitcoin vulnerability (https://www.bitmex.com/blog/64-Byte-Transactions). Every real Bitcoin block with more than one transaction contains exploitable internal nodes. The attacker needs only the block's public Merkle tree structure — no privileged access, no key material, no social engineering. The only barrier is that downstream contracts must be calling v1 rather than v2.

---

### Recommendation

1. **Remove or restrict v1**: Apply `#[private]` to `verify_transaction_inclusion` so only the contract itself can call it, or remove it entirely.
2. **Revert instead of returning `false`**: If v1 must remain, make it `panic!` on proof failure rather than returning `false`, so callers cannot silently accept invalid proofs — matching the behavior expected by the analog fix (checking actual state change rather than trusting a return value).
3. **Enforce v2 at the API boundary**: Document and enforce that all external callers must use `verify_transaction_inclusion_v2`.

---

### Proof of Concept

Given a confirmed block with transactions `[A, B, C, D]` and known Merkle tree:

```
root = compute_hash(AB, CD)
AB   = compute_hash(A, B)
CD   = compute_hash(C, D)
```

An unprivileged NEAR caller submits:

```json
{
  "tx_id":              "<AB hash>",
  "tx_block_blockhash": "<block hash>",
  "tx_index":           0,
  "merkle_proof":       ["<CD hash>"],
  "confirmations":      1
}
```

`compute_root_from_merkle_proof(AB, 0, [CD])`:
- `current_position = 0`, `0 % 2 == 0` → `current_hash = compute_hash(AB, CD)` = `root` [5](#0-4) 

Result equals `header.block_header.merkle_root` → function returns `true`. [6](#0-5) 

The forged proof is accepted. Any downstream contract gating on this `true` result is deceived.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
