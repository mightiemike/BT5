### Title
SwapAllowlistExtension gates the router address instead of the actual end user, allowing any user to bypass the swap allowlist on curated pools — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the actual user's address. A pool admin who allowlists the router (required for router-based swaps to function) inadvertently opens the allowlist to every user on the network.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool (the caller of the extension) and `sender` is whoever called `pool.swap`. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards the pool's own `msg.sender` as the `sender` argument: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router calls `pool.swap(...)` directly. At that point the pool's `msg.sender` is the **router**, so `sender` delivered to the extension is the router address, not the end user. [3](#0-2) 

For a curated pool to support router-based swaps at all, the pool admin **must** add the router to the allowlist:

```
extension.setAllowedToSwap(pool, address(router), true);
```

Once that entry exists, the check `allowedSwapper[pool][router]` returns `true` for every user who routes through the router, regardless of whether that user is individually permitted. The allowlist is silently reduced to a binary "router allowed / router not allowed" flag, defeating its per-user curation purpose entirely. [4](#0-3) 

The protocol's own audit target documentation acknowledges this exact risk:

> *"Because public users may enter through the router, the hook must gate the same actor the pool designers thought they were allowlisting."* [5](#0-4) 

---

### Impact Explanation

**High.** Any user — including those explicitly excluded from the allowlist — can swap on a curated pool by routing through `MetricOmmSimpleRouter`. The allowlist extension is the sole access-control mechanism for swap gating on such pools. Bypassing it breaks the core pool invariant that only approved counterparties may trade, and can expose LPs to trades they explicitly opted out of (e.g., regulatory, KYC, or risk-management constraints). This maps directly to the allowed impact gate: *broken core pool functionality causing loss of funds or unusable swap flows* and *admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path*.

---

### Likelihood Explanation

**Medium.** The bypass is only reachable when the pool admin has allowlisted the router. However, allowlisting the router is the **expected operational step** for any curated pool that wants to support the standard periphery UX. Pools that skip this step cannot use the router at all, so in practice most curated pools that are live and usable will have the router allowlisted, making the bypass universally available to any user.

---

### Recommendation

The extension must resolve the actual end user, not the immediate pool caller. Two sound approaches:

1. **Pass the real user through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires the extension to trust the router, so the router address itself must be verified (e.g., via the factory).

2. **Check `recipient` instead of `sender` for swap allowlisting**: If the pool's design always routes output to the real user as `recipient`, gating on `recipient` avoids the router-indirection problem. Verify this holds for all router call patterns before adopting.

3. **Dedicated router-aware allowlist**: Extend the extension to maintain a separate `trustedRouter` set; when `sender` is a trusted router, extract the real user from `extensionData` and apply the per-user check. [1](#0-0)

─────
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls extension.setAllowedToSwap(pool, alice, true).
   → alice is the only permitted swapper.
3. Pool admin calls extension.setAllowedToSwap(pool, router, true).
   → Required so alice can also use the router UI.

Attack
──────
4. Bob (not on allowlist) calls:
       router.exactInputSingle({
           tokenIn: token0,
           tokenOut: token1,
           pool: pool,
           recipient: bob,
           amountIn: X,
           amountOutMinimum: 0,
           priceLimit: ...,
           extensionData: ""
       })

5. Router calls pool.swap(recipient=bob, ...).
   pool.msg.sender = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. Extension evaluates:
       allowedSwapper[pool][router] == true  ✓

8. Swap executes. Bob receives token1 output.
   Bob was never individually allowlisted.
