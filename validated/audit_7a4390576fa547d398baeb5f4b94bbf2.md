### Title
Absent `l1_data_gas` in P2P Protobuf Silently Zeroed, Producing Wrong Transaction Hash for `AllResources` V3 Transactions - (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion silently defaults `l1_data_gas` to zero when the field is absent in a P2P protobuf message. For any V3 transaction with non-zero `l2_gas` **and** non-zero `l1_data_gas`, this produces `ValidResourceBounds::AllResources` with `l1_data_gas = 0`. The downstream hash function `get_tip_resource_bounds_hash` then includes the zeroed `l1_data_gas` in the Poseidon hash, yielding a transaction hash that diverges from the one the user signed. A malicious P2P peer can exploit this to cause the receiving sequencer node to bind the wrong hash to a valid transaction.

### Finding Description

**Partial initialization analog.** The external bug's root cause is that `_reduceAccountDebt` initializes only one slot of a two-element `pr[2]` array, while `updateAccountSecondaryDebt` reads both slots, treating the unset slot as zero. The sequencer analog is structurally identical: the protobuf converter initializes only the fields that are present in the wire message, while the hash function reads all three resource-bound fields, treating the absent one as zero.

**Root cause — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`**

```
// crates/apollo_protobuf/src/converters/transaction.rs  lines 417-436
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else { return Err(missing(...)); };
        let Some(l2_gas) = value.l2_gas else { return Err(missing(...)); };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // ← silent zero
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }
}
```

When a peer sends a protobuf `ResourceBounds` with `l1_data_gas = None` for a transaction that originally had `l1_data_gas = X > 0` and `l2_gas > 0`, the converter produces `AllResources { l1_gas, l2_gas, l1_data_gas: 0 }`. The `L1Gas` branch is not taken because `l2_gas` is non-zero, so the zeroed `l1_data_gas` is silently embedded in the `AllResources` variant.

**Downstream hash divergence — `get_tip_resource_bounds_hash`**

```
// crates/starknet_api/src/transaction_hash.rs  lines 188-211
pub fn get_tip_resource_bounds_hash(resource_bounds: &ValidResourceBounds, tip: &Tip) -> ... {
    let mut resource_felts = vec![
        get_concat_resource(&resource_bounds.get_l1_bounds(), L1_GAS)?,
        get_concat_resource(&resource_bounds.get_l2_bounds(), L2_GAS)?,
    ];
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all) => {
            vec![get_concat_resource(&all.l1_data_gas, L1_DATA_GAS)?]  // ← reads l1_data_gas
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

For `AllResources`, the hash commits to `l1_data_gas`. With the zeroed value the hash is:

```
poseidon(tip, pack(l1_gas), pack(l2_gas), pack(l1_data_gas=0))
```

whereas the user signed over:

```
poseidon(tip, pack(l1_gas), pack(l2_gas), pack(l1_data_gas=X))
```

These are distinct Felt values.

**Call path that binds the wrong hash**

`convert_rpc_tx_to_internal` (line 391 of `crates/apollo_transaction_converter/src/transaction_converter.rs`) recomputes the hash from the (now corrupted) transaction data and stores it in `InternalRpcTransaction.tx_hash`:

```
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
```

This is the same path used for both the gateway RPC flow and the consensus/mempool P2P propagation flow. In the mempool P2P path the `RpcTransaction` is deserialized from the peer's protobuf bytes and then fed directly into `convert_rpc_tx_to_internal`, so the wrong hash is computed and stored without any cross-check against the user's signature.

**Asymmetry between serializer and deserializer**

The serializer (`From<ValidResourceBounds> for protobuf::ResourceBounds`, lines 471-489) always emits `l1_data_gas: Some(...)`. The deserializer silently accepts `None`. This asymmetry means a well-behaved node never produces the malformed message, but a malicious peer can.

### Impact Explanation

A malicious P2P peer sends a mempool-propagated `AllResources` V3 transaction with `l1_data_gas` omitted from the protobuf `ResourceBounds` message. The receiving sequencer node:

1. Reconstructs `AllResources` with `l1_data_gas = 0`.
2. Computes a transaction hash that does not match the hash the user signed.
3. Stores `InternalRpcTransaction { tx_hash: wrong_hash, ... }`.

Downstream consequences:
- **Signature validation failure**: any component that verifies the ECDSA/Schnorr signature against `tx_hash` will reject a valid transaction.
- **Wrong executable payload**: the blockifier receives an `AccountTransaction` whose `tx_hash` does not match the signed preimage, breaking account-contract validation and potentially causing incorrect fee accounting (the `l1_data_gas` bound used for fee enforcement is now zero instead of the user's actual bound).
- **Consensus divergence**: nodes that received the transaction via the original RPC path hold the correct hash; nodes that received it via the tampered P2P path hold the wrong hash. This can cause disagreement on the transaction commitment in the block header.

This matches the impact category: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

### Likelihood Explanation

The trigger requires only that a peer participate in the mempool P2P network and send a single protobuf message with `l1_data_gas` omitted. No privileged access, no cryptographic material, and no special protocol state are required. The protobuf field is optional at the wire level (proto3 default), so any peer can craft such a message trivially. The affected code path is exercised for every `AllResources` V3 transaction with non-zero `l1_data_gas` that is received over P2P.

### Recommendation

Replace the silent `unwrap_or_default()` with a hard error for the P2P deserialization path, as the existing TODO comment already anticipates:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  line 427
// Before (vulnerable):
let l1_data_gas = value.l1_data_gas.unwrap_or_default();

// After (fixed):
let l1_data_gas = value.l1_data_gas
    .ok_or(missing("ResourceBounds::l1_data_gas"))?;
```

This mirrors the treatment of `l1_gas` and `l2_gas` (lines 420-425), which already return hard errors when absent. The TODO comment acknowledges this is the intended end-state once 0.13.2 backward-compatibility is dropped; the fix should be applied immediately for the P2P path because 0.13.2 transactions use `ValidResourceBounds::L1Gas` (zero `l2_gas`), which is classified before `l1_data_gas` is ever consulted. A 0.13.2 peer will never send a message with non-zero `l2_gas`, so requiring `l1_data_gas` when `l2_gas` is non-zero is safe today.

### Proof of Concept

```
1. Craft a valid V3 invoke transaction with:
     resource_bounds = AllResources { l1_gas: X, l2_gas: Y (>0), l1_data_gas: Z (>0) }
   The user signs hash H = poseidon(INVOKE, 3, sender, tip_rb_hash(X,Y,Z), ...).

2. Serialize to protobuf::ResourceBounds normally (l1_data_gas = Some(Z)).

3. As a malicious P2P peer, mutate the serialized bytes to set l1_data_gas = None
   (omit the field entirely — valid proto3).

4. Send the mutated RpcTransactionBatch to a victim sequencer node.

5. The victim's TryFrom<protobuf::ResourceBounds> for ValidResourceBounds fires:
     l1_data_gas = None.unwrap_or_default() = ResourceBounds { max_amount: 0, max_price: 0 }
     l2_gas is non-zero → AllResources { l1_gas: X, l2_gas: Y, l1_data_gas: 0 }

6. convert_rpc_tx_to_internal recomputes:
     H' = poseidon(INVOKE, 3, sender, tip_rb_hash(X,Y,0), ...)
     H' ≠ H

7. InternalRpcTransaction { tx_hash: H', ... } is stored and forwarded.
   Any signature check against H' fails; the valid transaction is dropped or
   executed under the wrong hash, corrupting fee accounting and block commitment.
