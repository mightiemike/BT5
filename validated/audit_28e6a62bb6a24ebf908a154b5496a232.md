Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypassed via Router — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which the pool sets to `msg.sender` — the immediate caller of `pool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, the router contract becomes the pool's `msg.sender`, not the end user. A pool admin who allowlists the router (a necessary step for allowlisted users to use the standard periphery) inadvertently grants every public user access to the restricted pool, because the extension only ever sees the router address as `sender`.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the pool's `msg.sender`. The original end-user address is stored only in transient storage for the payment callback and is never forwarded to the extension: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. The end-user's address (`msg.sender` at the router level) is captured only via `_setNextCallbackContext(..., msg.sender, ...)` for payment purposes and never reaches the extension.

A pool admin who wants allowlisted users to reach the pool via the router must call `setAllowedToSwap(pool, router, true)`. Once that entry exists, `allowedSwapper[pool][router] == true`, and every public user who calls any router entry point passes the extension check — because the check resolves to the router address, not the end user.

`DepositAllowlistExtension` does not share this flaw: it gates on `owner` (the position owner), which is explicitly passed by the caller and is independent of the intermediate caller: [5](#0-4) 

## Impact Explanation
Any user can trade on a pool the admin intended to restrict to a curated set of swappers. LP funds are directly exposed to swap flows the pool admin explicitly prohibited. Depending on the pool's purpose (institutional-only, KYC-gated, rate-limited), this enables unauthorized price impact, extraction of LP value by excluded actors, and violation of regulatory or contractual restrictions. This constitutes a direct loss of LP assets and an admin-boundary break reachable by any unprivileged caller.

## Likelihood Explanation
The trigger requires the pool admin to allowlist the router — a natural and expected action for any allowlisted pool intended to be usable through the standard periphery. If the admin does not allowlist the router, even allowlisted users cannot use `MetricOmmSimpleRouter`, making the allowlist effectively unusable with the canonical swap interface. The admin is therefore forced into the vulnerable configuration to provide normal UX. The attack requires no special privileges: any EOA can call `MetricOmmSimpleRouter.exactInputSingle` with the restricted pool as the target.

## Recommendation
Pass the original end-user address through the extension mechanism rather than the immediate pool caller. Two concrete options:

1. **Extension-data forwarding**: Require the router to encode the originating user address in `extensionData` and have `SwapAllowlistExtension` decode and verify it. The pool's `_beforeSwap` already forwards `extensionData` verbatim to extensions.
2. **Dedicated originator field**: Add an explicit `originator` field to the swap call signature (analogous to how `addLiquidity` separates `sender` from `owner`) so the pool can pass the true end-user to extensions independently of `msg.sender`.

Until fixed, pool admins must not allowlist the router on pools with `SwapAllowlistExtension`; they must require direct `pool.swap` calls from allowlisted EOAs only.

## Proof of Concept
```
Setup
─────
1. Deploy pool with SwapAllowlistExtension (beforeSwap order set).
2. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
3. Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:          <restricted pool>,
           recipient:     bob,
           zeroForOne:    true,
           amountIn:      X,
           ...
       })

5. Router calls pool.swap(...) — pool's msg.sender = router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true  →  passes.
8. Swap executes; Bob receives output tokens from the restricted pool.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```
