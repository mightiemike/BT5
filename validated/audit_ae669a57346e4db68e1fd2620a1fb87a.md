Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor, `msg.sender` of `addLiquidity`) and evaluates only `owner` (a free caller-supplied parameter). Because `owner` is unconstrained, any address can bypass the allowlist by naming an already-allowlisted address as `owner`. The unauthorized depositor's tokens are then permanently locked under `owner`'s position with no recovery path.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP-position beneficiary to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both addresses to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then discards the first argument (`sender`) with an unnamed parameter and checks only `owner`: [3](#0-2) 

Since `owner` is freely chosen by the caller, any address passes the guard by setting `owner` to any allowlisted address. The actual depositing party (`sender`) is never evaluated. After the deposit executes, `removeLiquidity` enforces `msg.sender == owner`, making the depositor's principal irrecoverable: [4](#0-3) 

The analogous `SwapAllowlistExtension.beforeSwap` correctly checks `sender`, confirming this is an inconsistency and not an intentional design: [5](#0-4) 

## Impact Explanation
Any pool configured with `DepositAllowlistExtension` (e.g., KYC-gated, institutional-only) accepts deposits from completely unauthorized addresses. The unauthorized depositor suffers permanent loss of principal: their tokens are pulled via the modify-liquidity callback into the pool, credited to `owner`'s position, and the depositor has no `removeLiquidity` access. This constitutes both an allowlist bypass (broken core pool access-control functionality) and direct, irrecoverable loss of user principal — meeting the Critical/High impact threshold.

## Likelihood Explanation
Exploitation requires only knowing one allowlisted address, which is trivially observable on-chain via `AllowedToDepositSet` events or direct `allowedDepositor` mapping reads. No special role, flash loan, or oracle manipulation is needed — a single plain `addLiquidity` call suffices. Every pool deploying `DepositAllowlistExtension` with a non-empty allowlist is affected.

## Recommendation
Check `sender` (the first parameter) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  bob is NOT allowlisted

Attack:
  bob calls pool.addLiquidity(
      owner  = alice,   // allowlisted — guard passes
      salt   = 0,
      deltas = { binIdxs: [0], shares: [1e18] },
      callbackData = "",
      extensionData = ""
  )

  Inside beforeAddLiquidity:
    sender (bob) is silently discarded (unnamed parameter)
    owner  (alice) passes allowedDepositor[pool][alice] == true

  LiquidityLib.addLiquidity executes:
    - positionBinShares[keccak(alice, 0, 0)] += 1e18
    - callback fires, pulling bob's tokens into the pool

Result:
  - bob's tokens are in the pool
  - alice holds the LP position
  - bob calls removeLiquidity(owner=alice, ...) → reverts NotPositionOwner
  - pool's deposit restriction is silently bypassed
  - bob's principal is permanently lost
```
