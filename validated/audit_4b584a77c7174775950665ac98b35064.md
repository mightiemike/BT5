### Title
SwapAllowlistExtension Checks Direct Caller Instead of Original User, Enabling Full Allowlist Bypass via Router — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is the direct `msg.sender` of the pool's `swap` call. When users route through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the original user. A pool admin who allowlists the router to enable router-mediated swaps for their curated users inadvertently opens the pool to every unprivileged caller of the same public router.

---

### Finding Description

**Root cause — wrong actor bound in the allowlist check**

In `MetricOmmPool.swap`, `msg.sender` is forwarded as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that same `sender` value and dispatches it to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of the pool's `swap`: [3](#0-2) 

**How the router breaks the identity**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly: [4](#0-3) 

Because the router is the immediate caller of `pool.swap`, `msg.sender` inside the pool equals the router's address. The `sender` forwarded to the extension is therefore the router, not the original user. The allowlist check becomes `allowedSwapper[pool][router_address]`.

**The forced configuration trap**

A pool admin who wants their allowlisted users to be able to use the router must add the router to the allowlist. Once the router is allowlisted, the check `allowedSwapper[pool][router_address]` passes for every call that arrives through the router — regardless of who the original caller is. The allowlist no longer distinguishes between `alice` (allowlisted) and `charlie` (not allowlisted) when both route through the same public contract.

The same structural problem applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

---

### Impact Explanation

Any unprivileged user can bypass the swap allowlist on a curated pool by routing through `MetricOmmSimpleRouter`. Curated pools designed for specific users (KYC'd participants, institutional LPs, whitelisted protocols) become accessible to all callers. Unauthorized swaps drain LP inventory at oracle prices, causing direct loss of LP principal and breaking the core access-control invariant the extension was deployed to enforce.

---

### Likelihood Explanation

High. `MetricOmmSimpleRouter` is the canonical, publicly documented swap entry point. A pool admin who deploys a swap-allowlisted pool and wants their allowlisted users to use the router has no choice but to allowlist the router address. This is the natural and expected operational step. Once taken, the bypass is trivially reachable by any user with no special privileges, no flash loan, and no admin interaction.

---

### Recommendation

The extension must check the original economic actor, not the intermediary. Concrete options:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires a coordinated convention between router and extension.
2. **Check `recipient` when it equals the economic beneficiary**: Only viable if the pool's swap design guarantees recipient == original user, which it does not in multi-hop paths.
3. **Factory-level incompatibility guard**: The factory rejects pool configurations that pair `SwapAllowlistExtension` with a known router address in the allowlist, or documents that the extension is incompatible with router-mediated flows and enforces direct-call-only usage.

---

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls:
     swapExtension.setAllowedToSwap(pool, alice, true);
     swapExtension.setAllowedToSwap(pool, address(router), true);
   (second call is required so alice can use the router)
3. charlie (not allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: charlie, ...})
4. Router calls pool.swap(...) — msg.sender inside pool = router.
5. SwapAllowlistExtension.beforeSwap receives sender = router.
6. Check: allowedSwapper[pool][router] == true → passes.
7. Charlie's swap executes on the curated pool despite never being allowlisted.
```

The bypass requires zero privileged access from charlie. The only precondition — the router being allowlisted — is the natural consequence of the pool admin trying to make the allowlist work with the supported periphery. [6](#0-5) [7](#0-6) [4](#0-3)
