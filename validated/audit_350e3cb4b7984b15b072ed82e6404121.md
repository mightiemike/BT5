Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Position `owner` Instead of Transaction `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` is documented as gating `addLiquidity` by depositor address, but its implementation checks the caller-supplied `owner` parameter (the LP position beneficiary) rather than `sender` (the actual `msg.sender` who pays the tokens). Because `owner` is a free argument to `addLiquidity` with no enforced equality to `msg.sender`, any un-allowlisted address can bypass the guard by naming an allowlisted address as `owner`. The pool admin's access-control configuration is rendered completely ineffective.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address and passes `msg.sender` as `sender` and the free `owner` argument to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both values verbatim to every configured extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently discards the first parameter (`sender`) and performs its allowlist check exclusively against `owner`: [3](#0-2) 

The allowlist mappings are keyed `allowedDepositor[pool][owner]`, so any caller who supplies an allowlisted address as `owner` passes the guard unconditionally: [4](#0-3) 

There is no `msg.sender == owner` check in `addLiquidity`. The only place that enforces caller-equals-owner is `removeLiquidity`: [5](#0-4) 

This asymmetry means the deposit path is fully open to any caller who knows one allowlisted address.

## Impact Explanation

A pool admin deploys `DepositAllowlistExtension` to restrict liquidity provision to a curated set of addresses (e.g., KYC'd LPs). The guard is silently inert against any caller who passes an allowlisted address as `owner`. The un-allowlisted caller pays the tokens via the swap callback; the allowlisted `owner` receives LP shares without having initiated or consented to the deposit. The pool admin's restriction has no effect on who can actually deposit. This constitutes broken core pool access-control functionality — the allowlist invariant stated in the audit pivots (*"deposit/swap allowlist checks must cover the exact actor/action intended and cannot be bypassed through … owner/salt separation"*) is violated unconditionally. **Severity: Medium.**

## Likelihood Explanation

Exploitation requires only knowledge of one allowlisted address (readable from the public `allowedDepositor` mapping or on-chain events) and the ability to call `addLiquidity` directly or through any router. No special role, privileged access, or admin cooperation is needed. Any EOA or contract can perform this in a single transaction, repeatably.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the first parameter, which is `msg.sender` of the pool's `addLiquidity` call) rather than `owner`:

```solidity
// fixed — check the actual caller
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to restrict both the caller and the position owner, both fields should be checked.

## Proof of Concept

```
Setup:
  pool  = MetricOmmPool with DepositAllowlistExtension configured
  Alice = allowedDepositor[pool][Alice] = true
  Bob   = allowedDepositor[pool][Bob]   = false  (not allowlisted)

Attack (single tx, no special role):
  Bob calls pool.addLiquidity(
      owner        = Alice,   // allowlisted address — passes the guard
      salt         = 0,
      deltas       = <desired liquidity>,
      callbackData = <Bob pays tokens in metricOmmSwapCallback>,
      extensionData = ""
  )

Result:
  - beforeAddLiquidity checks allowedDepositor[pool][Alice] → true → no revert
  - Bob's tokens are transferred to the pool via callback
  - LP shares are credited to Alice's position (owner=Alice, salt=0)
  - Bob has bypassed the deposit allowlist entirely
  - Pool admin's restriction on Bob is silently ineffective
```
