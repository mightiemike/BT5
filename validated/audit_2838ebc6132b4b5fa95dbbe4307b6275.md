### Title
Peer-Injected `fee_proposal_fri` Accepted Without Validation in P2P Sync, Corrupting `fee_proposals_window` and Skewing `next_l2_gas_price` - (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a peer-supplied `SignedBlockHeader` and stores it verbatim, including protobuf field 22 (`optional Uint128 fee_proposal_fri`), without any range or plausibility check. Because `fee_proposal_fri` is explicitly excluded from the block hash, a valid block signature does not cover it. A malicious p2p peer can therefore serve a structurally valid header (correct block number, correct signature count, valid block hash) with an arbitrary `fee_proposal_fri` value. On a sequencer node that falls behind and calls `try_sync`, the injected value is fed directly into `fee_proposals_window` via `record_fee_proposal`, corrupting the median used by `compute_fee_actual`, which in turn drives `calculate_next_l2_gas_price` and the `l2_gas_price` of the next produced block — the price used by RPC fee estimation.

---

### Finding Description

**Deserialization — no validation of `fee_proposal_fri`:**

In `crates/apollo_protobuf/src/converters/header.rs`, the converter reads the optional field and stores it unconditionally:

```rust
let fee_proposal_fri = value.fee_proposal_fri.map(|v| GasPrice(u128::from(v)));
``` [1](#0-0) 

The resulting value is placed directly into `BlockHeaderWithoutHash::fee_proposal_fri` with no bounds check. [2](#0-1) 

**`parse_data_for_block` — only checks block number and signature count:**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two semantic checks before returning `Ok(Some(signed_block_header))`:

1. Block number ordering.
2. `signatures.len() == ALLOWED_SIGNATURES_LENGTH`.

There is no check on `fee_proposal_fri`, no block-hash recomputation, and no signature verification over the header fields. [3](#0-2) 

**`fee_proposal_fri` is not part of the block hash:**

The proto definition carries an explicit warning:

```
// WARNING: this field is currently not part of the block hash, so the value must be trusted.
optional Uint128 fee_proposal_fri = 22;
``` [4](#0-3) 

The Rust struct mirrors this with a TODO:

```rust
// TODO(AndrewL): Add this field into the block hash.
pub fee_proposal_fri: Option<GasPrice>,
``` [5](#0-4) 

Consequently, even if signature verification were performed, it would not detect a modified `fee_proposal_fri`.

**Storage — value written as-is:**

`write_to_storage` calls `append_header` with the full `BlockHeader`, including the peer-supplied `fee_proposal_fri`, with no sanitization. [6](#0-5) 

The `StorageBlockHeader` persists the field: [7](#0-6) 

**`try_sync` — injected value enters `fee_proposals_window` without validation:**

When a sequencer node falls behind and calls `try_sync`, it reads the synced block from storage and calls `record_fee_proposal` with the stored (potentially attacker-controlled) value:

```rust
self.record_fee_proposal(height, sync_block.block_header_without_hash.fee_proposal_fri);
``` [8](#0-7) 

There is no validation of `fee_proposal_fri` at this point — contrast with the consensus path (`is_proposal_init_valid`), which enforces a margin bound: [9](#0-8) 

**`compute_fee_actual` → `next_l2_gas_price` → fee estimation:**

`fee_proposals_window` is the sole input to `compute_fee_actual`, which computes the median and returns `fee_actual`. This value drives `calculate_next_l2_gas_price`: [10](#0-9) 

The resulting `next_l2_gas_price` becomes the `l2_gas_price` of the next block, which is the price used by `exec_estimate_fee` for RPC fee estimation calls against that block. [11](#0-10) 

---

### Impact Explanation

A malicious p2p peer that serves `window_size / 2 + 1` blocks with `fee_proposal_fri = u128::MAX` (or any extreme value) during a sync window will shift the median returned by `compute_fee_actual` to that extreme. This propagates to `next_l2_gas_price` for the next produced block. RPC `starknet_estimateFee` and `starknet_estimateMessageFee` calls against that block will return fee estimates computed against the manipulated `l2_gas_price`, producing authoritative-looking wrong values. The impact is **High** under the "RPC fee estimation returns an authoritative-looking wrong value" criterion.

For pure sync/RPC-only nodes (not running consensus), `fee_proposal_fri` is stored but not consumed by any fee-estimation path, so the impact is limited to the stored field being wrong.

---

### Likelihood Explanation

Any unprivileged node on the p2p network can serve block headers. The attacker does not need operator, validator, or admin privileges. The only barrier is controlling enough blocks in the `fee_proposal_window_size` window to shift the median — feasible during a long sync gap or a targeted eclipse attack. The absence of block-hash coverage for this field means no cryptographic protection exists regardless of whether signature verification is added later.

---

### Recommendation

1. **Validate `fee_proposal_fri` in `parse_data_for_block`**: For blocks with `starknet_version >= V0_14_3`, reject headers where `fee_proposal_fri` is `None`; for older blocks, reject headers where it is `Some`. Apply the same margin bounds used in `is_proposal_init_valid` against the locally-known `fee_actual` (or at minimum a sanity cap).
2. **Include `fee_proposal_fri` in the block hash**: The TODO at `starknet_api/src/block.rs:245` and the proto warning both acknowledge this is deferred. Until it is done, the field cannot be cryptographically authenticated.
3. **Apply the margin check in `try_sync`**: Mirror the `fee_proposal_bounds` check from `is_proposal_init_valid` before calling `record_fee_proposal`.

```rust
pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}
