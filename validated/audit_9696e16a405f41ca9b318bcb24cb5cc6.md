### Title
Insufficient TokenID Validation in IBC NFT Packet Processing Allows Minting of Invariant-Violating NFTs - (File: x/nft-transfer/keeper/packet.go)

### Summary

`NonFungibleTokenPacketData.ValidateBasic()` only checks that `TokenIds` entries are non-empty, but never validates their format against the chain's own `ValidateTokenID` rule (`^[a-z][a-z0-9]*$`, 3–64 chars). When an IBC NFT packet is received, `processReceivedPacket` calls `k.nftKeeper.MintNFT` with the raw, unvalidated tokenIDs from the packet. This allows a counterparty chain to mint NFTs on the Cronos POS chain whose tokenIDs contain special characters, slashes, uppercase letters, or other content that violates the chain's own invariant. The resulting NFTs are permanently locked: `MsgBurnNFT`, `MsgEditNFT`, and `MsgTransferNFT` all call `ValidateTokenID` in their `ValidateBasic()`, so none of these operations can be applied to the malformed NFT.

### Finding Description

**Root cause — missing tokenID format check in packet validation:**

`NonFungibleTokenPacketData.ValidateBasic()` validates addresses and list lengths but never calls `ValidateTokenID` on any element of `TokenIds`:

```go
// x/nft-transfer/types/packet.go:41-71
func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
    if strings.TrimSpace(nftpd.ClassId) == "" { ... }
    if len(nftpd.TokenIds) == 0 { ... }
    if len(nftpd.TokenIds) != len(nftpd.TokenUris) { ... }
    // sender/receiver address checks only
    // ← no ValidateTokenID call on any TokenIds element
    return nil
}
``` [1](#0-0) 

**The chain's own tokenID invariant** is enforced only in the user-facing message path:

```go
// x/nft/types/validation.go:66-73
func ValidateTokenID(tokenID string) error {
    if len(tokenID) < MinDenomLen || len(tokenID) > MaxDenomLen { ... }
    if !IsBeginWithAlpha(tokenID) || !IsAlphaNumeric(tokenID) { ... }
    return nil
}
``` [2](#0-1) 

`IsAlphaNumeric` is `^[a-z0-9]+$` and `IsBeginWithAlpha` is `^[a-z].*`. Any tokenID containing `/`, `\`, uppercase letters, spaces, or other special characters is rejected by this function. [3](#0-2) 

**The IBC receive path bypasses this check entirely:**

```go
// x/nft-transfer/keeper/packet.go:181-184
for i, tokenID := range data.TokenIds {
    if err := k.nftKeeper.MintNFT(ctx, voucherClassID, tokenID, "", data.TokenUris[i], "", escrowAddress, receiver); err != nil {
        return err
    }
}
``` [4](#0-3) 

`MintNFT` in the keeper calls `IsDenomCreator` then `MintNFTUnverified`, neither of which validates the tokenID format: [5](#0-4) 

`MintNFTUnverified` only checks denom existence and duplicate tokenID, then writes the NFT directly to state: [6](#0-5) 

**The resulting NFT is permanently locked.** All three mutation messages call `ValidateTokenID` in `ValidateBasic()`:

- `MsgBurnNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)` [7](#0-6) 
- `MsgEditNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)` [8](#0-7) 
- `MsgTransferNFT.ValidateBasic()` → `ValidateTokenID(msg.Id)` (implied by the same pattern) [9](#0-8) 

Any NFT minted with a tokenID that fails `ValidateTokenID` cannot be burned, edited, or transferred via any normal on-chain transaction.

### Impact Explanation

A counterparty chain that permits arbitrary-string tokenIDs (which is common — IBC does not mandate the Cronos POS chain's `[a-z0-9]` constraint) can send an IBC-721 packet whose `TokenIds` contain special characters (e.g., `abc/def`, `ABC`, `abc!`, null bytes). Upon receipt, the Cronos POS chain mints these NFTs into the receiver's account. The minted NFTs:

1. **Violate the chain's own tokenID invariant** — state is permanently inconsistent with the rules enforced on every user-submitted transaction.
2. **Are permanently frozen** — the owner cannot burn, edit, or transfer them via `MsgBurnNFT`, `MsgEditNFT`, or `MsgTransferNFT` because all three reject the tokenID at `ValidateBasic()`.
3. **Pollute the NFT collection** — the supply counter for the IBC voucher denom is incremented for NFTs that can never be removed through normal operations.

The corrupted value is the NFT ownership record and supply count for the IBC voucher denom: an NFT is permanently credited to the receiver's account with no on-chain path to remove it.

### Likelihood Explanation

IBC is explicitly designed to interoperate with chains that have different validation rules. Many Cosmos chains (e.g., those using the Cosmos SDK native `x/nft` module) allow arbitrary UTF-8 strings as token IDs. A normal, unprivileged user on such a chain can create an NFT with a tokenID like `My Token #1` or `token/sub` and initiate an IBC transfer to the Cronos POS chain. No privileged access, key compromise, or social engineering is required. The entry path is a standard, supported IBC NFT transfer transaction.

### Recommendation

Add tokenID format validation inside `NonFungibleTokenPacketData.ValidateBasic()`, or alternatively inside `processReceivedPacket` before calling `MintNFT`. The check should mirror the existing `ValidateTokenID` logic, or a relaxed variant that at minimum rejects tokenIDs containing characters that could corrupt key parsing (e.g., null bytes). The same gap exists in `refundPacketToken` (line 44–48 of `packet.go`), which also calls `MintNFT` with unvalidated tokenIDs from packet data. [10](#0-9) 

### Proof of Concept

1. On a counterparty chain that allows arbitrary tokenIDs, create an NFT with tokenID `"My Token/001"` (contains `/` and uppercase, fails `ValidateTokenID`).
2. Submit an IBC NFT transfer to the Cronos POS chain via `MsgTransfer`.
3. The relayer submits the packet. `OnRecvPacket` calls `data.ValidateBasic()` — passes (tokenID is non-empty, lengths match).
4. `processReceivedPacket` calls `k.nftKeeper.MintNFT(ctx, voucherClassID, "My Token/001", ...)`.
5. `MintNFT` → `MintNFTUnverified` — no tokenID format check — NFT is written to state.
6. On the Cronos POS chain, attempt `MsgBurnNFT{Id: "My Token/001", ...}` — `ValidateBasic()` calls `ValidateTokenID("My Token/001")` → `ErrInvalidTokenID` — transaction rejected.
7. Attempt `MsgTransferNFT{Id: "My Token/001", ...}` — same rejection.
8. The NFT is permanently locked in the receiver's account with no on-chain removal path.

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil

// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
```
