### Title
`HttpServerDynamicConfig.max_sierra_program_size` Dynamic Update Bypasses Startup Cross-Validation Against `max_request_body_size` - (File: `crates/apollo_http_server_config/src/config.rs`)

### Summary
At node startup, `HttpServerConfig` enforces a cross-field invariant: `max_request_body_size > max_sierra_program_size`. However, when `max_sierra_program_size` is updated at runtime via the dynamic config path (`ConfigManager::set_node_dynamic_config`), only `HttpServerDynamicConfig` is validated in isolation — the cross-constraint against the immutable `max_request_body_size` is never re-checked. An operator-controlled dynamic config update can therefore set `max_sierra_program_size ≥ max_request_body_size`, causing the HTTP layer to reject with HTTP 413 every declare transaction whose Sierra program exceeds `max_request_body_size`, even though the gateway's own limit now permits it.

### Finding Description

**Startup path (constraint enforced):**

`HttpServerConfig` carries a schema-level validator:

```rust
#[derive(Clone, Debug, Default, Serialize, Deserialize, Validate, PartialEq)]
#[validate(schema(function = "max_size_validations"))]
pub struct HttpServerConfig {
    pub dynamic_config: HttpServerDynamicConfig,
    pub static_config: HttpServerStaticConfig,
}
``` [1](#0-0) 

The schema function enforces:

```rust
fn max_size_validations(http_server_config: &HttpServerConfig) -> Result<(), ValidationError> {
    let max_request_body_size = http_server_config.static_config.max_request_body_size;
    let max_sierra_program_size = http_server_config.dynamic_config.max_sierra_program_size;
    if max_request_body_size <= max_sierra_program_size {
        return Err(ValidationError::new(
            "max_request_body_size must be greater than max_sierra_program_size",
        ));
    }
    Ok(())
}
``` [2](#0-1) 

**Dynamic update path (constraint absent):**

`NodeDynamicConfig` holds only the dynamic half:

```rust
pub struct NodeDynamicConfig {
    ...
    #[validate(nested)]
    pub http_server_dynamic_config: Option<HttpServerDynamicConfig>,
    ...
}
``` [3](#0-2) 

`ConfigManager::set_node_dynamic_config` validates only `NodeDynamicConfig`:

```rust
pub(crate) async fn set_node_dynamic_config(
    &self,
    node_dynamic_config: NodeDynamicConfig,
) -> ConfigManagerResult<()> {
    if let Err(errors) = node_dynamic_config.validate() {
        ...
        return Err(ConfigManagerError::InvalidConfig(errors.to_string()));
    }
    ...
}
``` [4](#0-3) 

`NodeDynamicConfig::validate()` calls `HttpServerDynamicConfig::validate()` in isolation. `HttpServerDynamicConfig` has no cross-field constraint:

```rust
#[derive(Clone, Debug, Serialize, Deserialize, Validate, PartialEq)]
pub struct HttpServerDynamicConfig {
    pub accept_new_txs: bool,
    pub max_sierra_program_size: usize,
}
``` [5](#0-4) 

`max_request_body_size` lives in `HttpServerStaticConfig` and is never present in `NodeDynamicConfig`, so the cross-check is structurally impossible in the update path.

The `validate_node_dynamic_config` schema function only checks the consensus/context timeout relationship and is unrelated to HTTP sizing: [6](#0-5) 

**Defaults (showing the gap):**
- `max_request_body_size` = 5 MB (static, set at startup)
- `max_sierra_program_size` = 4 MB (dynamic, updatable at runtime) [7](#0-6) 

A dynamic update setting `max_sierra_program_size = 6 MB` passes all runtime validation but violates the invariant silently.

### Impact Explanation

The HTTP server applies `max_request_body_size` as a hard axum `RequestBodyLimitLayer`:

```rust
.layer(RequestBodyLimitLayer::new(self.config.static_config.max_request_body_size))
``` [8](#0-7) 

After the invalid dynamic update, any declare transaction whose serialized body exceeds `max_request_body_size` (e.g., 5 MB) is rejected with HTTP 413 at the transport layer — before the gateway's stateless validator, which now permits programs up to the new `max_sierra_program_size` (e.g., 6 MB). Valid transactions in the range `(max_request_body_size, max_sierra_program_size]` are permanently rejected by the RPC admission layer.

**Matched impact:** *High. Mempool/gateway/RPC admission rejects valid transactions before sequencing.*

### Likelihood Explanation

The `ConfigManagerRunner` polls a config file on disk and calls `set_node_dynamic_config` whenever the file changes: [9](#0-8) 

Any operator or deployment pipeline that increases `max_sierra_program_size` in the dynamic config file without a corresponding increase to the static `max_request_body_size` triggers the condition. Because the update is accepted without error, the operator receives no feedback that the invariant is broken. The likelihood is **medium** — it requires a config change, but the silent acceptance makes it easy to miss.

### Recommendation

Add a cross-validation check inside `ConfigManager::set_node_dynamic_config` that compares the incoming `max_sierra_program_size` against the static `max_request_body_size` stored at construction time:

```rust
pub struct ConfigManager {
    _config: ConfigManagerConfig,
    latest_node_dynamic_config: Arc<RwLock<NodeDynamicConfig>>,
+   http_server_static_config: Option<HttpServerStaticConfig>, // stored at init
}

pub(crate) async fn set_node_dynamic_config(
    &self,
    node_dynamic_config: NodeDynamicConfig,
) -> ConfigManagerResult<()> {
+   if let (Some(http_dyn), Some(http_static)) = (
+       &node_dynamic_config.http_server_dynamic_config,
+       &self.http_server_static_config,
+   ) {
+       if http_static.max_request_body_size <= http_dyn.max_sierra_program_size {
+           return Err(ConfigManagerError::InvalidConfig(
+               "max_request_body_size must be greater than max_sierra_program_size".into()
+           ));
+       }
+   }
    if let Err(errors) = node_dynamic_config.validate() { ... }
    ...
}
```

Alternatively, move the cross-validation into `NodeDynamicConfig` by storing the static bound as a field, or add a `Validate` impl on `HttpServerDynamicConfig` that accepts the static bound as a parameter.

### Proof of Concept

1. Start the node with default config: `max_request_body_size = 5 MB`, `max_sierra_program_size = 4 MB`. Startup validation passes.
2. Update the dynamic config file to set `max_sierra_program_size = 6 MB`.
3. `ConfigManagerRunner::update_config` reads the file, calls `set_node_dynamic_config` with the new `NodeDynamicConfig`.
4. `node_dynamic_config.validate()` succeeds — `HttpServerDynamicConfig` has no cross-field constraint.
5. The new `max_sierra_program_size = 6 MB` is committed.
6. Submit a declare transaction with a Sierra program of size 5.5 MB. The gateway's stateless validator accepts it (5.5 MB < 6 MB). The HTTP layer rejects it with HTTP 413 (5.5 MB > 5 MB `max_request_body_size`).
7. The transaction is permanently rejected despite being within the gateway's advertised limit.

    pub(crate) async fn update_config(
        &mut self,
    ) -> Result<NodeDynamicConfig, Box<dyn std::error::Error + Send + Sync>> {
        let config = load_and_validate_config(self.cli_args.clone(), false).map_err(|e| {
            CONFIG_MANAGER_UPDATE_ERRORS.increment(1);
            error!("ConfigManagerRunner: failed to update config: {e}");
            e
        })?;
        let node_dynamic_config = NodeDynamicConfig::from(&config);

        // Compare the previous and the newly read node dynamic config.
        if self.latest_node_dynamic_config == node_dynamic_config {
            // No change, so no action is needed.
            Ok(node_dynamic_config)
        } else {
            // Log the diff between the latest and the new node dynamic config.
            self.log_config_diff(&self.latest_node_dynamic_config, &node_dynamic_config);
            // Update the latest node dynamic config.
            self.latest_node_dynamic_config = node_dynamic_config.clone();
            match self
                .config_manager_client
                .set_node_dynamic_config(node_dynamic_config.clone())
                .await

                    info!("Successfully updated dynamic config");

                }
