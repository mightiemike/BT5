### Title
`get_tip_resource_bounds_hash` Silently Drops `l1_data_gas` from Transaction Hash When Protobuf Converter Downgrades `AllResources` to `L1Gas` - (File: crates/starknet_api/src/transaction_hash.rs)

### Summary

The transaction hash function `get_tip_resource_bounds_hash` conditionally excludes the `l1_data_gas` resource bound from the hash preimage when the resource bounds variant is `ValidResourceBounds::L1Gas`. The protobuf converter silently downgrades any `AllResources` transaction whose `l2_gas` and `l1_data_gas` fields are both zero to `ValidResourceBounds::L1Gas`. A transaction submitted via the RPC/gateway path is always hashed as `AllResources` (including the zero `l1_data_gas` field), while the same transaction received over P2P and converted through protobuf is hashed as `L1Gas` (excluding `l1_data_gas`). The two hash values are different, producing a canonical hash divergence across the two ingestion paths.

### Finding Description

**Hash computation — conditional exclusion of `l1_data_gas`**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` builds the fee-fields hash that is committed to in every V3 transaction hash:

```rust
// For new V3 txs, need to also hash the data gas bounds.
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],          // ← l1_data_gas ABSENT
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
```

When the variant is `L1Gas`, the `l1_data_gas` packed felt is not appended to `resource_felts`. Because `get_concat_resource` encodes the resource name (`b"L1_DATA"`) into the felt, even a zero-amount/zero-price `l1_data_gas` entry produces a non-zero felt. The two poseidon hashes are therefore structurally different.

**Protobuf converter — silent downgrade from `AllResources` to `L1Gas`**

The P2P protobuf converter in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following rule:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← downgrade
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Any transaction whose `l2_gas` and `l1_data_gas` fields are both zero is silently reclassified as `L1Gas`, regardless of whether the sender originally constructed and signed it as an `AllResources` V3 transaction.

**Divergent hash paths**

The RPC/gateway path always receives transactions as `RpcTransaction` / `InternalRpcTransactionWithoutTxHash`, which carry `AllResourceBounds` (never `ValidResourceBounds::L1Gas`). The hash is computed via `calculate_transaction_hash` → `get_tip_resource_bounds_hash` with `AllResources`, so `l1_data_gas` is always included.

The P2P/consensus path converts the wire protobuf to `ValidResourceBounds`. For a transaction with `l2_gas = 0` and `l1_data_gas = 0`, the converter produces `L1Gas`, and any subsequent hash computation (e.g., during storage read-back, hash validation, or re-derivation in the blockifier) uses the shorter preimage that omits `l1_data_gas`.

The two poseidon digests differ:

| Path | Preimage elements |
|---|---|
| RPC / gateway | `tip ‖ L1_GAS_packed ‖ L2_GAS_packed(0) ‖ L1_DATA_packed(0)` |
| P2P after protobuf | `tip ‖ L1_GAS_packed ‖ L2_GAS_packed(0)` |

### Impact Explanation

A V3 transaction submitted via the gateway with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` is signed over the longer hash. When the same transaction is propagated to a peer node via P2P, the protobuf converter downgrades it to `L1Gas`, and any hash re-derivation on the receiving side produces the shorter hash. This breaks the canonical hash invariant: the same transaction has two different hashes depending on which ingestion path is used. Consequences include:

- **Valid transactions rejected**: a peer node that re-derives the hash for deduplication or signature verification will compute a different value and reject the transaction.
- **Wrong hash bound to executable payload**: if the P2P-derived hash is stored and later used for receipt lookup, block commitment, or L1 message binding, the wrong hash is committed.

This matches the allowed impact scope: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"* and *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

Any V3 transaction that sets `l2_gas = 0` and `l1_data_gas = 0` (a valid configuration for a sender who only wants to pay in L1 gas, or for a transaction that writes no storage) triggers the downgrade. The protobuf path is exercised for every transaction propagated between sequencer nodes. No special privilege is required; a normal user submitting such a transaction via the public RPC endpoint is sufficient.

### Recommendation

1. **Remove the downgrade heuristic in the protobuf converter.** The `l1_data_gas.is_zero() && l2_gas.is_zero()` check was introduced for backward compatibility with pre-0.13.3 peers (see the `TODO` comment). Once that support window closes, the check should be removed and `AllResources` should always be preserved. In the interim, the downgrade should only apply when the transaction version is explicitly pre-0.13.3.

2. **Canonicalize the hash variant before storage.** The hash stored in `InternalRpcTransaction.tx_hash` should be the authoritative value computed at ingestion time and must not be recomputed from a potentially downgraded `ValidResourceBounds` variant.

3. **Add a cross-path hash consistency test** that submits a transaction with `l2_gas = 0` and `l1_data_gas = 0` via RPC, serializes it to protobuf, deserializes it, and asserts that the hash computed from the deserialized form equals the original.

### Proof of Concept

```
1. Construct a V3 InvokeTransaction with:
     resource_bounds = AllResourceBounds {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Submit via RPC gateway.
   → gateway calls get_tip_resource_bounds_hash(AllResources{...})
   → preimage = poseidon(tip, L1_GAS_packed, L2_GAS_packed(0), L1_DATA_packed(0))
   → hash_rpc = H1

3. Serialize the transaction to protobuf (as done in P2P propagation).
   → protobuf ResourceBounds { l1_gas: Some(...), l2_gas: Some(zero), l1_data_gas: Some(zero) }

4. Deserialize via TryFrom<protobuf::ResourceBounds> for ValidResourceBounds.
   → l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → result = ValidResourceBounds::L1Gas(l1_gas)   ← downgraded

5. Compute hash from deserialized form.
   → get_tip_resource_bounds_hash(L1Gas{...})
   → preimage = poseidon(tip, L1_GAS_packed, L2_GAS_packed(0))   ← l1_data_gas absent
   → hash_p2p = H2

6. Assert H1 != H2.  ← divergence confirmed
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
