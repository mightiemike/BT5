Audit Report

## Title
Pool Admin Can Zero the Timelock and Atomically Disable the Stop-Loss Guard — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`proposeOracleStopLossTimelock` accepts `newTimelock = 0` with no minimum-value check. Once executed, `_afterTimelock` returns `block.timestamp`, and `_requireElapsed` passes immediately in the same block (`block.timestamp < block.timestamp` is false). The pool admin can then propose and execute `drawdownE6 = 0` atomically, causing `_afterSwapOracleStopLoss` to return early unconditionally and disabling the entire stop-loss guard with zero LP reaction window for all subsequent parameter changes.

## Finding Description

**Root cause — no minimum timelock validation:**

`proposeOracleStopLossTimelock` at line 78 accepts any `uint32` value including 0 with no floor check: [1](#0-0) 

**Step 1 — Reduce timelock to zero (must wait current timelock T):**

The pool admin calls `proposeOracleStopLossTimelock(pool_, 0)`, waits T seconds, then calls `executeOracleStopLossTimelock`. This writes `oracleStopLossConfig[pool_].timelock = 0`: [2](#0-1) 

**Step 2 — Same-block propose + execute `drawdownE6 = 0`:**

With `timelock = 0`, `_afterTimelock` returns `uint32(block.timestamp + 0)` = `block.timestamp`: [3](#0-2) 

`_requireElapsed` checks `block.timestamp < executeAfter`. When `executeAfter == block.timestamp`, this evaluates to `block.timestamp < block.timestamp` → **false** → no revert: [4](#0-3) 

`_validateDrawdown(0)` only checks `drawdownE6 > E6`, so 0 passes: [5](#0-4) 

**Step 3 — Stop-loss guard fully disabled:**

`_afterSwapOracleStopLoss` short-circuits at `drawdown == 0`, skipping all watermark checks for every subsequent swap: [6](#0-5) 

The same zero-delay pattern applies identically to `proposeOracleStopLossDecay`/`executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks`/`executeOracleStopLossHighWatermarks`, since all three use `_afterTimelock` and `_requireElapsed` with the same logic.

## Impact Explanation

Once `drawdownE6 = 0`, every call to `afterSwap` returns early at line 217 without performing any watermark comparison or stop-loss check. The per-share value floor that prevents LP principal from being drained through adversarial swaps is completely inoperative. LPs are exposed to unlimited value leakage with no on-chain protection and no reaction window after the timelock is zeroed. This is a direct admin-boundary break: the pool admin exceeds the constraint imposed by the timelock mechanism that the NatDoc explicitly states exists to protect LPs ("Drawdown and decay changes are timelocked so LPs can react"). Severity: **High** — broken core pool functionality causing potential loss of LP principal.

## Likelihood Explanation

The pool admin is a single EOA or contract returned by `IMetricOmmPoolFactory(FACTORY).poolAdmin(pool_)` and is explicitly semi-trusted — the timelock exists precisely to limit their power. The attack requires only two phases: wait the current timelock (observable on-chain, but LPs may not monitor), then atomically disable protections in a single block. No external oracle manipulation, non-standard tokens, or factory owner involvement is needed. The attack is fully repeatable and requires no special conditions beyond pool admin key control.

## Recommendation

1. **Enforce a minimum timelock floor** in `proposeOracleStopLossTimelock`: add `require(newTimelock >= MIN_TIMELOCK)` with a protocol-defined constant (e.g., 1 hour or 1 day).
2. **Use strict inequality** in `_requireElapsed`: change `block.timestamp < executeAfter` to `block.timestamp <= executeAfter` so same-block execution is always rejected regardless of timelock value.
3. **Validate timelock at initialization**: `initialize` calls `_validateDrawdown` and `_validateDecay` but has no `_validateTimelock`, allowing a zero timelock to be set from pool creation: [7](#0-6) 

## Proof of Concept

```solidity
function test_timelockZeroDisablesStopLoss() public {
    // Pool deployed with timelock = 1 days, drawdownE6 = 50_000 (5%)

    // Phase 1: propose timelock = 0, wait current timelock, execute
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossTimelock(pool, 0);
    vm.warp(block.timestamp + 1 days);
    vm.prank(poolAdmin);
    ext.executeOracleStopLossTimelock(pool);
    // oracleStopLossConfig[pool].timelock == 0

    // Phase 2: same block — propose drawdown = 0 (executeAfter = block.timestamp)
    vm.prank(poolAdmin);
    ext.proposeOracleStopLossDrawdown(pool, 0);
    // Same block — execute drawdown = 0 (block.timestamp < block.timestamp == false, passes)
    vm.prank(poolAdmin);
    ext.executeOracleStopLossDrawdown(pool);

    (uint32 drawdown,,,) = ext.oracleStopLossConfig(pool);
    assertEq(drawdown, 0);

    // Phase 3: value-draining swap now succeeds — _afterSwapOracleStopLoss returns at drawdown == 0
    _executeValueDrainingSwap(pool); // no OracleStopLossTriggered revert
}
```
