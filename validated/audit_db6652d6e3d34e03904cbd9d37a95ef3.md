### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` — (`core/contracts/PerpEngine.sol`)

---

### Summary

When `PerpEngine.socializeSubaccount` is called during finalization and a product's `state.openInterest == 0` while `balance.vQuoteBalance < 0` (after insurance is exhausted), the call to `MathSD21x18.div(-balance.vQuoteBalance, state.openInterest)` reverts with `"DBZ"`. There is no guard for this case. The result is a permanently non-finalizable subaccount.

---

### Finding Description

**The division with no zero-guard:**

In `PerpEngine.socializeSubaccount`, after insurance is applied, if `balance.vQuoteBalance` is still negative the code attempts to spread the loss across all open-interest holders: [1](#0-0) 

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // ← can be 0
    );
```

`MathSD21x18.div` unconditionally requires the denominator to be non-zero: [2](#0-1) 

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
```

There is no `if (state.openInterest == 0) { ... }` branch before the division.

**How `openInterest` reaches zero:**

`openInterest` is the sum of absolute position sizes across all participants. In `_updateBalance`: [3](#0-2) 

```solidity
state.openInterest -= balance.amount.abs();   // remove old contribution
// ...
if (balance.amount > 0) {
    state.openInterest += balance.amount;
} else {
    state.openInterest -= balance.amount;     // abs of short
}
```

When every participant's `balance.amount == 0`, `openInterest == 0`.

**The finalization path that triggers this:**

`_finalizeSubaccount` (entered when `txn.productId == type(uint32).max`) first enforces that the liquidatee has no open perp positions: [4](#0-3) 

```solidity
for (uint32 i = 0; i < v.perpIds.length; ++i) {
    uint32 perpId = v.perpIds[i];
    IPerpEngine.Balance memory balance = perpEngine.getBalance(perpId, txn.liquidatee);
    require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
}
```

Then it calls: [5](#0-4) 

```solidity
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    v.insurance
);
```

If, at the moment of finalization, all other market participants have also closed their positions (making `openInterest == 0` globally for that product), and the liquidatee still carries a negative `vQuoteBalance` that exceeds available insurance, the `div` call reverts.

---

### Impact Explanation

- The entire `liquidateSubaccountImpl` transaction reverts with `"DBZ"`.
- Because the revert is deterministic (the on-chain state — `openInterest == 0`, `vQuoteBalance < 0`, insufficient insurance — does not self-resolve), every future attempt to finalize this subaccount will also revert.
- The subaccount is permanently stuck: its negative `vQuoteBalance` cannot be socialized, and the finalization path is permanently blocked.
- Since the transaction reverts atomically, the `insurance` state variable is **not** debited (the write at `insurance = v.insurance` on line 411 never executes). However, the subaccount remains in a non-finalizable limbo state. [6](#0-5) 

---

### Likelihood Explanation

The precondition requires:
1. `balance.amount == 0` for the liquidatee on a given perp product (enforced by the finalization check itself).
2. All other participants in that same product also have `balance.amount == 0` (so `openInterest == 0`).
3. The liquidatee's `vQuoteBalance < 0` after insurance.

This is an edge case but is reachable in production: low-liquidity perp markets, markets being wound down, or markets where the liquidatee was the last active participant. The liquidatee's negative `vQuoteBalance` can arise from accumulated funding payments even after their position size reaches zero via prior liquidation steps.

---

### Recommendation

Add a zero-check for `state.openInterest` before the division in `PerpEngine.socializeSubaccount`. If `openInterest == 0` and `vQuoteBalance < 0` after insurance, there are no other participants to socialize against; the loss should either be absorbed entirely by insurance (even if it drives insurance negative) or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; absorb into insurance or write off
        insurance += balance.vQuoteBalance; // insurance absorbs the residual loss
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [7](#0-6) 

---

### Proof of Concept

```
1. Deploy the protocol with a single perp product (productId = 2).
2. Have subaccount A open a long position; subaccount B open a short position of equal size.
3. Advance time so funding payments accumulate, driving A's vQuoteBalance negative.
4. Close B's position entirely (B.amount = 0).
5. Liquidate A's position size to 0 via normal liquidation steps (A.amount = 0, openInterest = 0).
   A.vQuoteBalance remains negative.
6. Drain insurance below |A.vQuoteBalance| (e.g., via prior socializations or fees).
7. Submit LiquidateSubaccount with productId = type(uint32).max (finalization trigger).
8. Observe revert with reason "DBZ" from MathSD21x18.div at PerpEngine.sol line 166.
9. Repeat step 7 indefinitely — the revert is permanent.
``` [8](#0-7) [9](#0-8)
