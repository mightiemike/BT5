### Title
Incorrect `existingWeight` in `Clearinghouse.getHealth()` Severely Understates Spread Health, Causing DoS for Legitimate Spread Positions — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

In `Clearinghouse.getHealth()`, the spread health adjustment uses `existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2`. Because `getCoreRisk` in `BaseEngine` always passes the literal `1` (positive) to `_getWeightX18`, `CoreRisk.longWeight` is **always the long weight** regardless of the actual position direction. For a long-spot / short-perp spread, the perp's actual health contribution uses `perpShortWeight` (≥ 1), not `perpLongWeight` (≤ 1). This causes `existingWeight` to be massively overstated, the spread credit to be massively understated, and the total health of the spread position to be severely negative when it should be positive.

---

### Finding Description

**Root cause — `BaseEngine.getCoreRisk` always returns long weight:**

`getCoreRisk` passes the literal `1` (positive) to `_getWeightX18`, so `CoreRisk.longWeight` is always `longWeightInitialX18` or `longWeightMaintenanceX18`, regardless of whether the actual position is long or short. [1](#0-0) 

**Root cause — `_calculateProductHealth` uses the actual position direction:**

In contrast, `_calculateProductHealth` passes the real `amount` to `_getWeightX18`, so a short perp position correctly uses `shortWeightInitialX18` (≥ 1) in the individual health contribution. [2](#0-1) 

**The broken formula in `Clearinghouse.getHealth()`:**

```solidity
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
...
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
``` [3](#0-2) 

The `existingWeight` is supposed to represent the effective weight already applied to the basis amount by the individual spot and perp health contributions, so that the spread adjustment correctly converts those contributions to the spread weight. For a long-spot / short-perp spread, the correct `existingWeight` is:

```
existingWeight = (spotLongWeight × spotPrice − perpShortWeight × perpPrice)
                 ─────────────────────────────────────────────────────────
                              (spotPrice + perpPrice)
```

Because `perpShortWeight > 1 > perpLongWeight`, the correct `existingWeight` is **negative** (or at minimum much smaller than the long-weight average). The code instead computes a positive value `(spotLongWeight + perpLongWeight) / 2 ≈ 0.9e18`, which is wrong in both sign and magnitude.
