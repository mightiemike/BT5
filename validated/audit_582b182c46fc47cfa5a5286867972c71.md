### Title
Unchecked `transferFrom()` Return Value in `replaceUsdcEWithUsdc()` Allows usdcE Drain Without Providing USDC — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` at line 616 without checking the boolean return value. The function has no access control. If the USDC token returns `false` on a failed transfer (e.g., insufficient allowance or balance), execution continues, withdrawing usdcE from the `DirectDepositV1` contract and forwarding it to the caller — with no USDC deposited in exchange.

---

### Finding Description

`replaceUsdcEWithUsdc()` is designed as a migration helper: it reads the usdcE balance held in a `DirectDepositV1` (DDA) contract, pulls an equivalent amount of USDC from the caller into the DDA, then withdraws the usdcE back to the caller. The intended flow is:

1. Pull `balance` USDC from `msg.sender` → `directDepositV1` (line 616)
2. Withdraw usdcE from `directDepositV1` → `ContractOwner` (line 617)
3. Forward usdcE from `ContractOwner` → `msg.sender` (line 618)

The critical flaw is at step 1:

```solidity
// core/contracts/ContractOwner.sol, line 616
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
```

The return value (`bool`) is silently discarded. If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (Ink chain, chainId 57073) returns `false` instead of reverting on a failed transfer — a behaviour permitted by the ERC20 standard and exhibited by several non-standard or bridged tokens — the transaction does not revert. Steps 2 and 3 execute unconditionally, draining usdcE from the DDA and delivering it to the caller.

The function carries no `onlyOwner` or `onlyDeployer` modifier, making it reachable by any unprivileged external caller. [1](#0-0) 

Contrast with the safe pattern used elsewhere in the same contract and in `ERC20Helper`: [2](#0-1) 

`ERC20Helper.safeTransferFrom` wraps the low-level call and `require`s that `success && (data.length == 0 || abi.decode(data, (bool)))`. The raw call at line 616 performs none of this.

---

### Impact Explanation

An attacker who calls `replaceUsdcEWithUsdc(subaccount)` for any subaccount whose DDA holds usdcE, while providing zero USDC allowance, can receive the full usdcE balance of that DDA for free. The corrupted asset delta is: usdcE leaves `directDepositV1` and is credited to the attacker; no USDC enters `directDepositV1`. The DDA's usdcE balance is zeroed, and the protocol receives nothing in return. [3](#0-2) 

---

### Likelihood Explanation

- The function is `external` with no access control — any EOA or contract can call it.
- The only precondition is that a DDA for the target subaccount exists and holds a non-zero usdcE balance (`balance > 0`).
- Exploitability depends on whether the USDC token at the hardcoded address returns `false` on failure rather than reverting. Bridged or wrapped USDC variants on non-mainnet chains (chainId 57073 = Ink) do not always match Circle's mainnet USDC revert behaviour. Even if the current deployment reverts, the code is structurally unsafe and a future token upgrade or redeployment could silently enable the drain. [4](#0-3) 

---

### Recommendation

Replace the raw `transferFrom` call with `ERC20Helper.safeTransferFrom`, which is already imported and used throughout the contract via `using ERC20Helper for IERC20Base`:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

Additionally, consider adding an `onlyOwner` modifier to `replaceUsdcEWithUsdc()` to restrict the migration path to trusted callers. [5](#0-4) 

---

### Proof of Concept

1. A DDA exists for `subaccount` with `usdcE.balanceOf(directDepositV1) = X > 0`.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(subaccount)` with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, X)` returns `false` (token does not revert on failure); return value is ignored.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes: usdcE balance `X` is transferred from `directDepositV1` to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, X)` executes: `X` usdcE is transferred to the attacker.
6. Net result: attacker receives `X` usdcE; `directDepositV1` receives 0 USDC. The subaccount's DDA is drained. [6](#0-5)

            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
