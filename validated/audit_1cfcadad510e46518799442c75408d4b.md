### Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the actual user, allowing any caller to bypass the swap allowlist via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` bound to `msg.sender` of `pool.swap()`. When a user swaps through `MetricOmmSimpleRouter`, `msg.sender` in the pool is the **router contract**, not the original user. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`. If the pool admin allowlists the router address (the natural step to enable router-mediated swaps), every unpermissioned user can bypass the curated pool's allowlist entirely.

### Finding Description

**Call chain when using the router:**

```
user → MetricOmmSimpleRouter.exactInputSingle()
         → pool.swap(recipient, ...) [msg.sender = router]
              → _beforeSwap(msg.sender=router, ...)
                   → SwapAllowlistExtension.beforeSwap(sender=router, ...)
                        → allowedSwapper[pool][router]  ← wrong actor checked
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` = pool and `sender` = the router: [3](#0-2) 

When the router calls `pool.swap`, it does not forward the original user identity — it passes itself as `msg.sender`: [4](#0-3) 

The same wrong-actor binding applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

**Two concrete consequences:**

1. **Allowlist bypass (primary impact):** A pool admin who allowlists the router address — the natural step to enable router-mediated swaps on a curated pool — inadvertently opens the pool to every user. Any address can call `router.exactInputSingle()` and the extension passes because `allowedSwapper[pool][router] == true`.

2. **Broken swap path for allowlisted users (secondary impact):** A pool admin who allowlists individual user addresses (the intended use case) finds that those users cannot swap through the router at all, because the extension sees `sender = router` (not allowlisted) and reverts `NotAllowedToSwap`. The standard periphery path is unusable for the pool's intended participants.

The unit tests for `SwapAllowlistExtension` only simulate the pool calling the extension directly (`vm.prank(address(pool))`), and the integration test uses `setAllowAllSwappers` rather than individual user allowlisting, so neither test surface catches this misbinding: [6](#0-5) [7](#0-6) 

### Impact Explanation

**High.** A pool admin who allowlists the router to support the standard periphery path loses the entire allowlist guarantee: every unpermissioned address can execute swaps on the curated pool. This is a direct bypass of the access-control guard with no additional privilege required beyond calling the public router. It also constitutes broken core pool functionality: allowlisted users cannot use the supported periphery swap path.

### Likelihood Explanation

**High.** The router is the canonical swap entry point documented and deployed alongside the pool. A pool admin who wants to support both direct and router-mediated swaps will allowlist the router. The documentation contains no warning that doing so opens the pool to all users. The misbinding is invisible from the admin's perspective because `isAllowedToSwap(pool, router)` returns `true` as intended, yet the actual enforcement is wrong.

### Recommendation

The extension must check the original user, not the intermediary. Two options:

**Option A — pass the original user through the router.** The router stores the original `msg.sender` in transient storage (already done for the payer context) and the pool exposes it as a separate `originator` argument to `beforeSwap`. This requires a core interface change.

**Option B — check `recipient` instead of `sender` in `SwapAllowlistExtension`.** The `recipient` is the address that receives output tokens and is the economically relevant actor for a swap allowlist. The router passes `params.recipient` (the actual user) as the `recipient` argument to `pool.swap`, so the extension already receives the correct address — it just needs to gate on `recipient` rather than `sender`:

```diff
- function beforeSwap(address sender, address, bool, ...)
+ function beforeSwap(address, address recipient, bool, ...)
    external view override returns (bytes4)
  {
-   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
+   if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][recipient]) {
      revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [3](#0-2) 

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (intending to enable router-mediated swaps)
  - Pool admin does NOT allowlist attacker address

Attack:
  - attacker calls router.exactInputSingle({pool, recipient: attacker, ...})
  - router calls pool.swap(attacker, ...)  [msg.sender = router]
  - pool calls _beforeSwap(router, ...)
  - SwapAllowlistExtension.beforeSwap(sender=router, ...)
  - allowedSwapper[pool][router] == true  → passes
  - attacker executes swap on curated pool without being individually allowlisted

Result:
  - Swap allowlist is completely bypassed
  - Any user can trade on the curated pool via the router
```

Alternatively, to demonstrate the broken-path scenario:

```
Setup:
  - Pool admin calls setAllowedToSwap(pool, alice, true)
  - Router is NOT allowlisted

Attack surface:
  - alice calls router.exactInputSingle({pool, recipient: alice, ...})
  - router calls pool.swap(alice, ...)  [msg.sender = router]
  - SwapAllowlistExtension.beforeSwap(sender=router, ...)
  - allowedSwapper[pool][router] == false → reverts NotAllowedToSwap
  - alice (a legitimately allowlisted user) cannot use the standard periphery path
```
