### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Actual User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` of `pool.swap()` is the router contract, not the end user. If the router is allowlisted (the only way to let allowlisted users use the router), every user — including non-allowlisted ones — passes the gate.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router is the direct caller of `pool.swap()`: [4](#0-3) 

So `sender` arriving at the extension is `address(router)`, not the end user. The extension has no visibility into who originally called the router.

The pool admin faces an impossible choice:

| Router allowlisted? | Allowlisted user via router | Non-allowlisted user via router |
|---|---|---|
| No | **Reverts** (router not in list) | Reverts |
| Yes | Succeeds | **Also succeeds — bypass** |

There is no configuration that simultaneously lets allowlisted users use the router and blocks non-allowlisted users.

The `DepositAllowlistExtension` does not share this flaw: it ignores `sender` and checks `owner` (the position owner), which the `LiquidityAdder` passes through unchanged from the caller's parameter: [5](#0-4) 

The swap allowlist has no equivalent identity-preserving mechanism.

---

### Impact Explanation

Any user can bypass a `SwapAllowlistExtension`-protected pool by routing through `MetricOmmSimpleRouter`. If the pool admin allowlists the router (the only way to make the router usable for legitimate participants), the allowlist is rendered completely ineffective. Unauthorized users can execute swaps in pools intended to be restricted (e.g., KYC/AML-gated, institutional-only, or whitelist-only pools), directly extracting value from LP positions that were priced and sized under the assumption of a controlled participant set.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the production-grade, publicly deployed periphery entry point. Pool admins who configure a `SwapAllowlistExtension` will naturally want their allowlisted users to be able to use the router. Allowlisting the router address is the obvious and only available mechanism to achieve this, making the bypass trivially reachable by any unprivileged actor. The project's own audit target list explicitly flags this exact identity-preservation question: [6](#0-5) 

---

### Recommendation

Preserve the original caller's identity through the router so the extension can check the actual economic actor. Two standard approaches:

1. **Pass the original caller in `callbackData`**: The router stores `msg.sender` in transient storage (it already does this for the payer). Extend the `extensionData` forwarded to the pool to include the original caller, and have the extension decode and check that address instead of `sender`.

2. **Check `sender` in the extension against a router-aware identity**: Introduce a `IMetricOmmSwapCallback`-style mechanism where the router attests the original caller inside `extensionData`, and the extension verifies the attestation signature or trusted-router flag before accepting the delegated identity.

The `DepositAllowlistExtension` pattern (checking `owner`, which is caller-supplied and passed through unchanged) is the correct model to follow.

---

### Proof of Concept

```
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)       // alice is KYC'd
3. Admin calls setAllowedToSwap(pool, router, true)      // needed so alice can use the router
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})
5. Router calls pool.swap(bob, ...) — msg.sender = router
6. Pool calls extension.beforeSwap(sender=router, ...)
7. Extension checks allowedSwapper[pool][router] == true → passes
8. Bob's swap executes successfully despite not being allowlisted.
``` [3](#0-2) [7](#0-6) [8](#0-7)
