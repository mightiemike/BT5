Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the actual user, allowing allowlist bypass or DoS on authorized users via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` at the time `swap()` is called. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap()`, so the extension checks `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actualUser]`. This either allows every user to bypass the allowlist if the router is allowlisted (Mode A), or silently blocks individually allowlisted users from using the router (Mode B).

## Finding Description

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
  msg.sender,   // <-- this is the router when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap()` forwards this value unchanged to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router address. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly without forwarding the original `msg.sender`:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The actual user's address is stored only in transient storage for the callback context (`_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)`) and is never passed to `pool.swap()` as a parameter the extension can inspect.

**Mode A — Bypass:** If the pool admin allowlists the router (to enable router-based swaps for any allowlisted user), every caller of `exactInputSingle` passes the guard regardless of individual allowlist status, because `allowedSwapper[pool][router] == true`.

**Mode B — DoS:** If the pool admin allowlists only specific EOAs and not the router, those EOAs are blocked when using the router because the extension sees `sender = router`, which is not on the list.

## Impact Explanation

This is a direct admin-boundary break. The pool admin's primary access-control mechanism — the swap allowlist — is rendered ineffective for any user routing through the public `MetricOmmSimpleRouter`. In Mode A, an unprivileged user trades against restricted liquidity (KYC-gated, compliance-restricted, or partner-only pools) without being individually allowlisted. In Mode B, authorized users permanently cannot use the supported router, breaking core swap functionality for intended participants. Both outcomes corrupt the extension's access-control decision (`allowedSwapper[pool][sender]` resolves to the wrong identity), which is the exact wrong value.

## Likelihood Explanation

High. `MetricOmmSimpleRouter` is a public, permissionless contract requiring no special privileges. Any user can call `exactInputSingle`. The pool admin faces a forced choice: allowlist the router (enabling Mode A bypass for all users) or not (causing Mode B DoS for individually allowlisted users). Both failure modes are reachable through normal, expected usage of the supported periphery contract.

## Recommendation

The pool should forward the original transaction initiator — not the immediate `msg.sender` — as `sender` to the extension. Concretely:

- The router should pass the actual user's address as a dedicated parameter to `pool.swap()`, and the pool should forward that value to the extension rather than its own `msg.sender`.
- Alternatively, the extension can require the router to attest to the real user's address inside `extensionData`, and the extension verifies the attestation against a trusted-router registry.
- A simpler mitigation is to document that the allowlist gates the immediate pool caller and require users of curated pools to call the pool directly — but this breaks the intended router UX.

## Proof of Concept

**Mode A — Allowlist bypass:**

1. Pool admin deploys a pool with `SwapAllowlistExtension`. Allowlists `alice` and `router` (to enable router-based swaps for `alice`).
2. `bob` (not allowlisted) calls `router.exactInputSingle({pool: curated_pool, tokenIn: USDC, ...})`.
3. Router calls `pool.swap(...)` — pool receives `msg.sender = router`.
4. Pool calls `extension.beforeSwap(router, ...)`.
5. Extension evaluates `allowedSwapper[pool][router]` → `true`.
6. `bob`'s swap executes on the restricted pool despite never being allowlisted.

**Mode B — Authorized user DoS:**

1. Pool admin allowlists only `alice` (not the router).
2. `alice` calls `router.exactInputSingle({pool: curated_pool, ...})`.
3. Extension evaluates `allowedSwapper[pool][router]` → `false`.
4. Revert: `NotAllowedToSwap` — `alice` is blocked despite being explicitly permitted.

Foundry test: deploy pool with `SwapAllowlistExtension`, allowlist the router, call `exactInputSingle` from a non-allowlisted EOA, assert the swap succeeds (demonstrating bypass).
