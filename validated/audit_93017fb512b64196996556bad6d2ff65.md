The call chain is fully traceable from the code. Here is the exact path:

**Step 1 — Router entry:** EOA calls `MetricOmmSimpleRouter.exactInputSingle`. Inside, the router calls `pool.swap(...)` directly. [1](#0-0) 

**Step 2 — Pool passes `msg.sender` (= router) as `sender` to `_beforeSwap`:** [2](#0-1) 

**Step 3 — `ExtensionCalling._beforeSwap` forwards that `sender` (= router) to the extension:** [3](#0-2) 

**Step 4 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][router]`:** [4](#0-3) 

The `sender` the extension sees is always the **router address**, never the originating EOA. If the pool admin allowlists the router (a natural choice to enable router-based swaps), every user routing through `MetricOmmSimpleRouter` passes the check regardless of whether their own address is in `allowedSwapper[pool]`. The per-user invariant the extension is designed to enforce — "Gates `swap` by swapper address, per pool" — is structurally broken for all router-mediated paths.

The Smart Audit Pivots explicitly flag this: *"allowlist checks must cover the exact actor/action intended and cannot be bypassed through router."*

### Title
SwapAllowlistExtension checks router address instead of originating user, allowing any EOA to bypass per-user swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives `sender = msg.sender` as seen by the pool, which is the router contract, not the originating EOA. Allowlisting the router grants unrestricted swap access to every user of that router, defeating the per-user curation the extension is designed to enforce.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`. [5](#0-4) 

When the call originates from `MetricOmmSimpleRouter.exactInputSingle`, `msg.sender` at the pool is the router contract address. [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` — i.e. `allowedSwapper[pool][router]`. [4](#0-3) 

There is no mechanism to recover the original EOA from within the extension. The extension has no access to `tx.origin` (and using it would be insecure), and no originator field is threaded through the call.

### Impact Explanation
A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to a curated set of counterparties (e.g. KYC'd addresses, whitelisted market makers) and also allowlists the router to support router-based swaps inadvertently grants unrestricted swap access to every user of that router. Any non-allowlisted EOA can drain pool reserves by routing through `MetricOmmSimpleRouter`, causing unauthorized token outflow from a pool whose admin explicitly intended to restrict access. This is broken core pool functionality with direct fund impact.

### Likelihood Explanation
The scenario is straightforward: any pool using `SwapAllowlistExtension` that also needs to support router-based swaps must allowlist the router, at which point the per-user gate is fully open. The router is a public, permissionless contract. No privileged access is required by the attacker.

### Recommendation
Thread the originating caller through the swap path so the extension can check the true economic actor. One approach: have the router pass the originating EOA as part of `extensionData`, and have `SwapAllowlistExtension.beforeSwap` decode and check that address when present. Alternatively, gate on `recipient` rather than `sender` for router flows, or introduce a dedicated router-aware allowlist that validates the payer stored in transient storage (`_getPayer()`) rather than the pool-visible `sender`.
