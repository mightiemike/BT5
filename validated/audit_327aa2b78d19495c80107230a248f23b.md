### Title
Fee-on-Transfer Token Accounting Corruption in `handleDepositTransfer` — (File: `core/contracts/EndpointStorage.sol`)

---

### Summary

`EndpointStorage.handleDepositTransfer` performs two sequential ERC-20 transfers using the caller-supplied nominal `amount` without measuring what was actually received. When a fee-on-transfer token (e.g., USDT with its fee switch enabled) is used as a spot collateral asset, the Clearinghouse receives less than `amount`, but the slow-mode deposit transaction records the full `amount`, causing `Clearinghouse.depositCollateral` to credit the user with more than the protocol actually holds. This creates a cumulative solvency deficit that grows with every deposit.

---

### Finding Description

`EndpointStorage.handleDepositTransfer` (lines 111–119) executes two back-to-back transfers using the same nominal `amount`:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);               // (1)
    safeTransferTo(token, address(clearinghouse), amount); // (2)
}
``` [1](#0-0) 

For a fee-on-transfer token with fee rate `f`:

- **Transfer (1)**: Endpoint receives `amount × (1 − f)`, not `amount`.
- **Transfer (2)**: Endpoint attempts to forward the full `amount` to the Clearinghouse. Because the Endpoint only holds `amount × (1 − f)` from this deposit, the call either:
  - **Reverts** if the Endpoint has no prior token balance (DoS path), or
  - **Succeeds by consuming accumulated Endpoint balance** (e.g., slow-mode fees collected in the quote token via `chargeSlowModeFee`), silently draining that reserve.

In either success path, the Clearinghouse receives `amount × (1 − f)` (a second fee is deducted on transfer (2)). [2](#0-1) 

Immediately after `handleDepositTransfer`, `depositCollateralWithReferral` enqueues a slow-mode transaction that records the **original** `amount`:

```solidity
DepositCollateral({
    sender: subaccount,
    productId: productId,
    amount: amount          // nominal, not actual received
})
``` [3](#0-2) 

When the sequencer later processes this slow-mode transaction, `Clearinghouse.depositCollateral` credits the user with the full nominal `amount`:

```solidity
int128 amountRealized = int128(txn.amount) * int128(multiplier);
spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
``` [4](#0-3) 

The invariant `sum(user balances) == Clearinghouse token balance` is broken: the user is credited `amount` but the Clearinghouse holds `amount × (1 − f)`. The deficit compounds with every deposit.

---

### Impact Explanation

Each deposit of a fee-on-transfer token inflates the SpotEngine's recorded user balance by `amount × f` relative to the Clearinghouse's actual token holdings. Over many deposits the cumulative shortfall grows unboundedly. When users attempt to withdraw, `withdrawCollateral` calls `handleWithdrawTransfer` which calls `token.safeTransfer(withdrawPool, amount)`: [5](#0-4) 

The Clearinghouse will eventually lack sufficient token balance to honor all withdrawals. The last withdrawers suffer direct, permanent loss of funds — a classic solvency corruption. Additionally, the Endpoint's accumulated slow-mode fee reserve (quote token) is silently drained to subsidize the forwarding shortfall, corrupting the fee accounting tracked in `sequencerFee` / `slowModeFees`.

---

### Likelihood Explanation

USDT's fee mechanism is currently set to zero but is a live, owner-controlled switch. If USDT is listed as the quote or any spot collateral product on Nado and the fee is activated, every subsequent deposit triggers the bug with no further attacker action required. Any unprivileged user calling `depositCollateral` or `depositCollateralWithReferral` is the entry path — no special role is needed. [6](#0-5) [7](#0-6) 

---

### Recommendation

Replace the fixed-`amount` two-transfer pattern with a balance-snapshot approach that measures what was actually received, and pass only the realized amount downstream:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), token.balanceOf(address(this)));
    uint256 actualReceived = token.balanceOf(address(clearinghouse)) - before;
    // use actualReceived when constructing the DepositCollateral slow-mode tx
}
```

The `DepositCollateral.amount` field stored in the slow-mode queue must be set to `actualReceived` (converted back to token decimals), so `Clearinghouse.depositCollateral` credits only what the protocol truly holds.

---

### Proof of Concept

1. USDT fee is enabled at rate `f = 0.1%` (10 bps).
2. Attacker/user calls `depositCollateral(subaccountName, USDT_PRODUCT_ID, 1_000_000 /* 1 USDT */)`.
3. `handleDepositTransfer` runs:
   - Transfer (1): Endpoint receives `999_000` (1_000 fee deducted).
   - Transfer (2): Endpoint forwards `1_000_000` to Clearinghouse. Endpoint has `999_000` from this deposit plus any prior slow-mode fee balance. If prior balance ≥ 1_000, the call succeeds; Clearinghouse receives `999_000` (another 1_000 fee deducted).
4. Slow-mode queue records `amount = 1_000_000`.
5. Sequencer processes: `spotEngine.updateBalance(USDT_PRODUCT_ID, sender, 1_000_000 × multiplier)` — user credited `1_000_000`.
6. Clearinghouse actual balance: `999_000`. Recorded user balance: `1_000_000`. **Deficit: 1_000 per deposit.**
7. After `N` such deposits the Clearinghouse is short `N × 1_000` USDT; the final withdrawers cannot redeem their balances. [1](#0-0) [8](#0-7)
