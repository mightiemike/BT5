### Title
Deposited Collateral Permanently Locked When Slow-Mode `DepositCollateral` Transaction Silently Fails — (`core/contracts/Endpoint.sol`)

---

### Summary

In `Endpoint.sol`, `depositCollateralWithReferral` transfers user tokens to the contract **before** queuing a slow-mode transaction. When that slow-mode transaction is later executed, it is wrapped in a `try {} catch {}` block. If the inner call reverts for any reason, the catch silently discards the failure. The user's tokens are already held by the contract, but `spotEngine.updateBalance` is never called, so the user's on-chain balance is never credited. A developer comment in the catch block — `// try return funds now removed` — explicitly confirms that the original fund-return mechanism was deliberately removed, leaving no recovery path.

---

### Finding Description

**Step 1 — Funds transferred before slow-mode queue**

In `Endpoint.depositCollateralWithReferral`, `handleDepositTransfer` pulls tokens from the caller into the contract, then a `SlowModeTx` of type `DepositCollateral` is appended to the queue: [1](#0-0) 

**Step 2 — Silent catch discards the revert**

In `_executeSlowModeTransaction`, the actual processing is wrapped in a bare `try {} catch {}`. If `processSlowModeTransaction` reverts for any reason, execution continues without crediting the user: [2](#0-1) 

The comment on line 226 — `// try return funds now removed` — is the explicit acknowledgement that the refund path was removed, leaving no recovery for failed deposits.

**Step 3 — Balance update is the skipped state change**

Inside `processSlowModeTransactionImpl`, a `DepositCollateral` slow-mode tx calls `clearinghouse.depositCollateral`, which calls `spotEngine.updateBalance`: [3](#0-2) [4](#0-3) 

If this call reverts, the `spotEngine` balance is never updated, but the ERC-20 tokens are already in the contract.

**Concrete revert triggers (no admin required):**

- A product is delisted between deposit time and execution time (slow-mode delay is hardcoded to 3 days per line 153). After delisting, `_decimals(txn.productId)` reverts because `token == address(0)`.
- `txn.amount > INT128_MAX` causes an overflow revert.
- `RiskHelper.isIsolatedSubaccount(txn.sender)` returns true for a sender that was later converted to an isolated subaccount. [5](#0-4) 

---

### Impact Explanation

Any user whose `DepositCollateral` slow-mode transaction reverts during execution permanently loses their deposited tokens. The tokens remain in the `Endpoint` contract with no accounting entry and no refund path. This is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

The 3-day slow-mode delay creates a realistic window for a product to be delisted between deposit and execution. The `executeSlowModeTransaction()` function is callable by anyone with no access control, meaning an attacker can trigger execution at a moment they know will fail (e.g., immediately after a product is delisted). The depositing user has no way to cancel the queued transaction. [6](#0-5) 

---

### Recommendation

Restore the fund-return logic inside the catch block. When a `DepositCollateral` slow-mode transaction fails, the contract should refund the deposited tokens to the original sender. Alternatively, decode the transaction type before executing and, for `DepositCollateral`, either revert the entire outer transaction on failure or store the amount in a claimable mapping so the user can recover their funds.

---

### Proof of Concept

1. User calls `depositCollateralWithReferral` for product `P` with amount `A`. Tokens are transferred; slow-mode tx is queued with `executableAt = now + 3 days`.
2. Within 3 days, product `P` is delisted via `DelistProduct` slow-mode tx (owner action).
3. After 3 days, anyone calls `executeSlowModeTransaction()`.
4. `_executeSlowModeTransaction` calls `this.processSlowModeTransaction(sender, tx)`.
5. Inside, `clearinghouse.depositCollateral` calls `_decimals(P)`, which calls `IERC20Base(address(0)).decimals()` and reverts.
6. The `catch {}` block is entered; `gasleft()` checks pass; execution continues silently.
7. `// try return funds now removed` — no refund is issued.
8. User's `A` tokens are permanently locked in `Endpoint`. Their `spotEngine` balance for product `P` is `0`. [7](#0-6) [8](#0-7)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }

    function _decimals(uint32 productId) internal virtual returns (uint8) {
        IERC20Base token = IERC20Base(_tokenAddress(productId));
        require(address(token) != address(0), ERR_INVALID_PRODUCT);
        return token.decimals();

    function depositCollateral(IEndpoint.DepositCollateral calldata txn)
        external
        virtual
        onlyEndpoint
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        uint8 decimals = _decimals(txn.productId);

        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - decimals));
        int128 amountRealized = int128(txn.amount) * int128(multiplier);

        spotEngine.updateBalance(txn.productId, txn.sender, amountRealized);
        emit ModifyCollateral(amountRealized, txn.sender, txn.productId);
    }
