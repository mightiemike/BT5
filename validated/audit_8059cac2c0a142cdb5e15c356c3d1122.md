### Title
`Role::UnrestrictedSubmitBlocks` Cannot Bypass Pause on `submit_blocks` Due to Missing `except` Clause — (File: `contract/src/lib.rs`)

---

### Summary

`Role::UnrestrictedSubmitBlocks` is defined and documented as the role that allows calling `submit_blocks` even when the contract is paused. However, the `#[pause]` attribute on `submit_blocks` carries no `except(roles(...))` clause, so the pause gate unconditionally rejects every caller when the contract is paused — including accounts that hold `UnrestrictedSubmitBlocks`. The role is completely inert with respect to the pause check. The parallel function `run_mainchain_gc` correctly wires its counterpart role with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, making the omission on `submit_blocks` a concrete, asymmetric implementation gap.

---

### Finding Description

`Role::UnrestrictedSubmitBlocks` is declared at line 44 with the explicit comment *"Allows to use contract API even after contract is paused"*: [1](#0-0) 

The `trusted_relayer` macro at lines 120–124 lists this role (and `Role::DAO`) as `bypass_roles`: [2](#0-1) 

That `bypass_roles` parameter only controls the **trusted-relayer whitelist gate** — it has no effect on the **pause gate**. The two are independent middleware layers applied in sequence.

`submit_blocks` is decorated with a bare `#[pause]` at line 167, with no `except` clause: [3](#0-2) 

In `near-plugins`, `#[pause]` without `except(roles(...))` means the pause check has **zero bypass paths**. When the contract is paused, every call to `submit_blocks` panics at the pause gate before the trusted-relayer check is even reached.

Contrast this with `run_mainchain_gc`, which correctly wires its counterpart role: [4](#0-3) 

The asymmetry is exact: `UnrestrictedRunGC` → `run_mainchain_gc` works; `UnrestrictedSubmitBlocks` → `submit_blocks` is silently broken.

---

### Impact Explanation

When the contract is paused (an emergency circuit-breaker action), the light client's canonical chain tip freezes. No new Bitcoin headers can be accepted. Downstream callers of `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will fail for any transaction in a block mined after the pause, because those blocks are never added to `mainchain_height_to_header` or `headers_pool`. [5](#0-4) 

The `UnrestrictedSubmitBlocks` role was designed precisely to prevent this: privileged relayers should be able to keep the chain live even during a pause. Because the `except` clause is absent, that design intent is entirely defeated. The `DAO` role — the highest-privilege role — is equally blocked, since `DAO` is not listed in any `except` clause on `submit_blocks` either.

---

### Likelihood Explanation

The contract must be paused for this to trigger, which requires a `PauseManager` action. However, the `UnrestrictedSubmitBlocks` role exists **only** for the paused-contract scenario; it has no other use. Any operator granted this role expecting to keep the light client synchronized during an emergency will be silently rejected. The likelihood is moderate: the bug is latent until a pause occurs, but the pause is the exact scenario the role was built for.

---

### Recommendation

Add the missing `except` clause to `submit_blocks` to match the pattern already used on `run_mainchain_gc`:

```rust
// Before (broken):
#[pause]
#[trusted_relayer]
pub fn submit_blocks(...) { ... }

// After (correct):
#[pause(except(roles(Role::UnrestrictedSubmitBlocks, Role::DAO)))]
#[trusted_relayer]
pub fn submit_blocks(...) { ... }
```

Including `Role::DAO` is consistent with the intent that the highest-privilege role can always operate the contract.

---

### Proof of Concept

1. Deploy the contract and grant `Role::UnrestrictedSubmitBlocks` to account `alice.near`.
2. Register `alice.near` as a trusted relayer (or grant `Role::DAO` to bypass the relayer check).
3. As `PauseManager`, call `pause` to pause the contract.
4. As `alice.near`, call `submit_blocks` with a valid batch of headers.
5. **Observed**: the call panics at the `#[pause]` gate — `alice.near` is rejected despite holding `UnrestrictedSubmitBlocks`.
6. **Expected**: the call succeeds, because `UnrestrictedSubmitBlocks` is documented to allow exactly this.
7. For comparison, repeat steps 3–5 with `run_mainchain_gc` and `Role::UnrestrictedRunGC` — that call succeeds, confirming the `except` clause works when present. [6](#0-5) [3](#0-2) [4](#0-3)
