### Title
`reorg_chain` Panics When GC Removes the Fork Divergence-Point Block — (`contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` removes old mainchain blocks from both `headers_pool` and `mainchain_header_to_height`. The `reorg_chain` while loop terminates only when it finds a fork-cursor block hash in `mainchain_header_to_height`. If the divergence-point mainchain block has been GC'd, it is absent from both maps, so the loop walks past all fork blocks, then calls `headers_pool.get(&prev_block_hash)` on the GC'd block and panics unconditionally.

---

### Finding Description

**`remove_block_header` deletes from both maps:** [1](#0-0) 

`run_mainchain_gc` calls this for every height in the removal window: [2](#0-1) 

After GC, mainchain block **M** at height **H** is gone from both `mainchain_header_to_height` and `headers_pool`.

**`reorg_chain` while-loop termination condition:** [3](#0-2) 

The loop terminates only when `fork_header_cursor.block_hash` is found in `mainchain_header_to_height`. Fork blocks are stored only in `headers_pool` (via `store_fork_header`), never in `mainchain_header_to_height`, so the loop walks all the way back to the deepest fork block **F1**, whose `prev_block_hash` is **M**.

**The panic site:** [4](#0-3) 

At this point `prev_block_hash` == M's hash, M is absent from `headers_pool`, `get` returns `None`, and `unwrap_or_else` fires `env::panic_str("previous fork block should be there")`.

There is no guard in `reorg_chain` that bounds the walk to `mainchain_initial_blockhash` or that handles a missing ancestor gracefully.

---

### Impact Explanation

The NEAR contract panics mid-execution. NEAR reverts all state changes made within the panicking call, so `mainchain_tip_blockhash` and `mainchain_initial_blockhash` remain in their pre-reorg state. The fork that legitimately outweighs the mainchain can never be promoted: every subsequent attempt to submit the winning fork tip will re-enter `reorg_chain` and panic again, permanently blocking fork promotion for that fork chain.

---

### Likelihood Explanation

`submit_blocks` is gated by `#[trusted_relayer]`, so the submitter must be a registered trusted relayer. However:

- The scenario requires no malicious intent — a legitimate relayer naturally submits a long fork that spans the GC window.
- `run_mainchain_gc` is publicly callable by anyone (no `trusted_relayer` restriction), so any account can advance GC to remove the divergence point independently of block submission.
- With `gc_threshold` set to the recommended 52,704 blocks (~1 year), a fork anchored more than a year back is the trigger; with smaller thresholds (e.g., test/staging deployments) the window is much shorter. [5](#0-4) 

---

### Recommendation

In `reorg_chain`, before the `unwrap_or_else` panic, check whether `prev_block_hash` equals or is older than `mainchain_initial_blockhash`. If the ancestor has been GC'd, the reorg cannot be completed safely; the function should return an error (or use `require!`/`env::panic_str` with a descriptive message that is handled at the call site) rather than silently panicking with a misleading message. Additionally, `store_fork_header` should refuse to store a fork block whose ancestor chain reaches below `mainchain_initial_blockhash`, preventing the inconsistent state from being created in the first place.

---

### Proof of Concept

```
State: gc_threshold = 5, skip_pow_verification = true

1. init() with genesis at height 0
2. submit_blocks([M1..M10])          // mainchain heights 1-10
3. run_mainchain_gc(6)               // removes M0..M5 from headers_pool
                                     // and mainchain_header_to_height;
                                     // mainchain_initial_blockhash = M6
4. submit_blocks([F1])               // fork block at height 1, parent = M0
   → get_prev_header panics "PrevBlockNotFound" because M0 is GC'd

   *** Revised sequence (fork submitted BEFORE GC) ***

1. init() with genesis at height 0
2. submit_blocks([M1..M10])          // mainchain heights 1-10
3. submit_blocks([F1, F2, F3])       // fork blocks at heights 1-3,
                                     // parent of F1 = genesis (height 0)
                                     // stored only in headers_pool
4. run_mainchain_gc(6)               // removes genesis..M5 from headers_pool
                                     // genesis (divergence point) is now gone
5. submit_blocks([F4..FN])           // fork blocks with cumulative chainwork
                                     // exceeding mainchain tip
   → submit_block_header_inner detects chain_work > mainchain
   → reorg_chain(FN, 10) called
   → while loop: FN → F(N-1) → ... → F1
   → F1.prev_block_hash = genesis hash
   → mainchain_header_to_height.contains_key(F1.hash) == false
   → headers_pool.get(genesis_hash) == None
   → PANIC: "previous fork block should be there"
```
