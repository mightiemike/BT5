Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which resolves to `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. If a pool admin allowlists the router to enable router-mediated swaps for permitted users, every unpermitted user can bypass the per-user allowlist by routing through the same public router.

## Finding Description
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool (the extension is called by the pool) and `sender` is whatever address the pool received as its own `msg.sender`: [1](#0-0) 

`MetricOmmPool.swap` passes its own `msg.sender` directly as the `sender` argument to `_beforeSwap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router contract the `msg.sender` at the pool level. The original end-user's address is stored only in transient callback context and is never forwarded into `extensionData`: [3](#0-2) 

The router stores `msg.sender` (the end-user) only as the payer for the callback via `_setNextCallbackContext`, but this is never encoded into `extensionData` passed to the pool or extension. The extension therefore checks `allowedSwapper[pool][router_address]` instead of `allowedSwapper[pool][end_user]`.

The same flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [4](#0-3) 

## Impact Explanation
When a pool admin allowlists the router address to enable router-mediated swaps for their permitted users, the per-user allowlist is completely nullified. Any unpermitted address can call `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) targeting the curated pool, and `beforeSwap` will pass because it sees the allowlisted router as the swapper. This constitutes broken core pool functionality â€” the access control mechanism deployed by the pool admin is fully bypassed, allowing unauthorized users to trade against LP capital that was deployed under the assumption of a restricted swapper set. This is a direct loss of the intended access control invariant with fund-impacting consequences for LPs in curated pools.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical production swap entry point. Pool admins who want permitted users to have normal UX (slippage protection, multi-hop, deadline checks) must allowlist the router. The bypass requires no special privileges â€” any unpermitted user can call the public router targeting any pool that has allowlisted the router. The attack is repeatable with zero setup cost.

## Recommendation
Pass the original end-user through the swap path so the extension can gate on the economically relevant actor. The preferred approach requires no core changes: the router encodes `msg.sender` into `extensionData` before calling the pool, and `SwapAllowlistExtension.beforeSwap` decodes and checks that address instead of `sender`. Specifically:

1. In `MetricOmmSimpleRouter`, encode `msg.sender` into `extensionData` (or prepend it) before passing to `pool.swap()`.
2. In `SwapAllowlistExtension.beforeSwap`, decode the originating user from `extensionData` (verifying the caller is a trusted router) and check `allowedSwapper[pool][originatingUser]` instead of `allowedSwapper[pool][sender]`.

Alternatively, add a trusted router registry so the extension can verify the relay and extract the originator from `extensionData` only when called via a registered router.

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)       // alice is permitted
  - Pool admin calls setAllowedToSwap(pool, router, true)      // router allowlisted for UX
  - bob is NOT in the allowlist

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  2. Router calls pool.swap(...) â€” msg.sender at pool = router address
  3. Pool calls _beforeSwap(sender=router, ...)
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] â†’ true
  5. Swap executes successfully for bob despite bob not being allowlisted

Result:
  bob swaps against the curated pool's LP capital, bypassing the intended access control.
  The pool admin's per-user allowlist is completely ineffective for any user who routes
  through the public MetricOmmSimpleRouter.
```
