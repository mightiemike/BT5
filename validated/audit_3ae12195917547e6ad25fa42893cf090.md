### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `openInterest == 0` Permanently Blocks Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when a subaccount's `vQuoteBalance` is negative after insurance coverage. When the market has no other open positions (`openInterest == 0`), this triggers `MathSD21x18.div`'s `ERR_DIV_BY_ZERO` revert, permanently preventing finalization of any insolvent subaccount that closed all its perp positions before becoming insolvent.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after attempting to cover a negative `vQuoteBalance` with insurance, the code socializes any remaining deficit across all open-interest holders:

```solidity
if (balance.vQuoteBalance < 0) {
    // socialize across all other participants
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest   // <-- can be 0
    );
    ...
}
``` [1](#0-0) 

`MathSD21x18.div` enforces a hard `require(y != 0, ERR_DIV_BY_ZERO)`: [2](#0-1) 

`state.openInterest` is the sum of `abs(balance.amount)` across **all** subaccounts. The value returned by `getStateAndBalance` for the victim subaccount already has the victim's own contribution removed and re-added (net zero change when `balance.amount == 0`): [3](#0-2) 

Therefore, when the victim is the only participant and has `balance.amount == 0`, `state.openInterest == 0` exactly.

`_finalizeSubaccount` **requires** `balance.amount == 0` for every perp product before calling `socializeSubaccount`: [4](#0-3) 

Then it calls `perpEngine.socializeSubaccount`: [5](#0-4) 

The combination is contradictory: `_finalizeSubaccount` enforces `balance.amount == 0` (which means the victim contributes 0 to `openInterest`), yet `socializeSubaccount` assumes `openInterest > 0` when `vQuoteBalance < 0`. There is no guard for the `openInterest == 0` case.

---

### Impact Explanation

- `_finalizeSubaccount` always reverts for any insolvent subaccount that closed all perp positions while retaining a negative `vQuoteBalance`.
- The insolvent subaccount is permanently un-finalizable: the liquidation transaction always reverts at `socializeSubaccount`.
- Insurance funds allocated to cover this loss (`v.insurance -= lastLiquidationFees`) are effectively locked — the accounting update that would consume or return them never completes.
- The negative `vQuoteBalance` remains in the system indefinitely, creating a persistent bad-debt accounting discrepancy.

---

### Likelihood Explanation

This is reachable through normal protocol usage with no special privileges:

1. A subaccount opens a perp position (it is the only participant in that market, or all others close before finalization).
2. The subaccount closes its position at a loss (`balance.amount = 0`, `vQuoteBalance < 0`).
3. The subaccount has no other assets to cover the loss → maintenance health < 0.
4. A liquidator submits `LiquidateSubaccount` with `productId == type(uint32).max` to trigger finalization.
5. `_finalizeSubaccount` passes the `balance.amount == 0` check for all perps.
6. `socializeSubaccount` is called; `state.openInterest == 0`; `balance.vQuoteBalance < 0` → `DBZ` revert.

The scenario is especially likely in low-liquidity or newly-launched perp markets where the insolvent account is the sole or last remaining participant.

---

### Recommendation

Add an explicit guard for the zero open-interest case inside `socializeSubaccount`. When `state.openInterest == 0` there are no counterparties to socialize the loss to; the deficit should either be absorbed entirely by insurance (already attempted above) or written off:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off residual bad debt
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

This mirrors the existing `openInterest == 0` guard already present in `updateStates`: [6](#0-5) 

---

### Proof of Concept

```
State: single perp market, productId = 2
  Alice: balance.amount = 0, vQuoteBalance = -100e18
  state.openInterest = 0  (Alice is the only participant, position closed)
  insurance = 50e18  (not enough to cover full loss)

Call: liquidateSubaccountImpl({ liquidatee: Alice, productId: type(uint32).max, ... })

Trace:
  _finalizeSubaccount:
    balance.amount == 0 for all perps  ✓  (passes require)
    v.insurance = 50e18 - lastLiquidationFees
    perpEngine.socializeSubaccount(Alice, ~50e18):
      balance.vQuoteBalance = -100e18 < 0
      insuranceCover = min(50e18, 100e18) = 50e18
      balance.vQuoteBalance = -100e18 + 50e18 = -50e18  (still < 0)
      fundingPerShare = 50e18.div(0)  →  require(0 != 0, "DBZ")  →  REVERT

Result: liquidation permanently reverts; Alice's subaccount is never finalized.
```
