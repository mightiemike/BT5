### Title
`int_atom` skips `CANONICAL_INTS` check, allowing non-canonical signed integer atoms in mempool mode for arithmetic operators — (`src/op_utils.rs`)

### Summary
`int_atom` has no `ClvmFlags` parameter and performs no canonical-form validation. `uint_atom` correctly rejects leading zeros when `CANONICAL_INTS` is set. Because `MEMPOOL_MODE` includes `CANONICAL_INTS`, and because `op_add`, `op_subtract`, `op_multiply`, and `op_gr` all call `int_atom`, those operators silently accept non-canonical signed integer atoms (e.g., `[0x00, 0x01]` = 1) in mempool mode.

### Finding Description
- `number_from_u8` in `src/number.rs` (lines 9–16) accepts any byte sequence via `BigInt::from_signed_bytes_be`, including redundant leading zeros.
- `int_atom` in `src/op_utils.rs` (lines 248–256) calls `a.number(args)` → `number_from_u8` with no canonical check.
- `uint_atom` in `src/op_utils.rs` (lines 67–86) explicitly checks `flags.contains(ClvmFlags::CANONICAL_INTS)` and rejects non-canonical atoms.
- `MEMPOOL_MODE` in `src/chia_dialect.rs` (lines 72–76) sets `CANONICAL_INTS`.
- `more_ops.rs` contains zero references to `CANONICAL_INTS`; the flags parameter received by `op_add`/`op_subtract`/`op_multiply`/`op_gr` is never forwarded to `int_atom`.

### Impact Explanation
An attacker can submit CLVM programs to the mempool that use arithmetic operators with non-canonical integer atoms (e.g., `[0x00, 0x01]` instead of `[0x01]`). These pass mempool validation despite `MEMPOOL_MODE` being intended to enforce canonical representation. The mempool's strictness invariant is violated: it accepts programs it is supposed to reject, potentially allowing non-canonical transaction data to be relayed and included.

### Likelihood Explanation
Trivially reachable. Any attacker-controlled CLVM program submitted to the mempool can include an `op_add` with a non-canonical argument. No special privileges required.

### Recommendation
Add a `flags: ClvmFlags` parameter to `int_atom` (mirroring `uint_atom`) and insert a canonical-form check analogous to the one in `uint_atom` lines 67–86. Update all callers in `more_ops.rs` to pass the `flags` argument they already receive.
