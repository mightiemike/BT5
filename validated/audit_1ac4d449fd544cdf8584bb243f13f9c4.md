### Title
Dogecoin Fork-Block Difficulty Retarget Uses Main-Chain Ancestor Instead of True Fork Ancestor — (`contract/src/dogecoin.rs`)

---

### Summary

`get_next_work_required` in the Dogecoin module fetches the retarget interval's first-block timestamp via `get_header_by_height(height_first)`, which unconditionally reads from `mainchain_height_to_header`. When the block being validated builds on a fork that diverged two or more blocks below the current height, `mainchain_height_to_header[height_first]` returns the **main-chain** block at that height, not the true fork-chain ancestor. The developers themselves flagged this with a `TODO` comment at the exact call site. The result is that `calculate_next_work_required` receives a wrong `first_block_time`, producing an incorrect `bits` target that the submitted block is then validated against.

---

### Finding Description

**Call chain:**
`submit_blocks` → `submit_block_header` → `check_target` → `check_pow` → `get_next_work_required` → `get_header_by_height(height_first)`

After height 145,000, Dogecoin's Digishield sets `difficulty_adjustment_interval = 1`, so every block is a retarget boundary and `blocks_to_go_back = 1`. Therefore `height_first = prev_block_header.block_height - 1`. [1](#0-0) 

The retarget lookup: [2](#0-1) 

`get_header_by_height` is implemented as: [3](#0-2) 

It reads exclusively from `mainchain_height_to_header`, which only tracks the **current canonical chain**.

**Concrete scenario (Digishield, post-145k):**

```
Main chain:  ... → A(h=N-1) → B(h=N) → C(h=N+1)  [tip]
Fork:        ... → A(h=N-1) → D(h=N) → E(h=N+1) → F(h=N+2)
```

When the attacker submits fork block **F** (building on E, height N+2):
- `prev_block_header` = E (height N+1), fetched correctly from `headers_pool` via `get_prev_header`
- `height_first` = N
- `get_header_by_height(N)` → returns **B** (main-chain block at height N)
- **Correct** value should be **D** (fork-chain ancestor of E at height N) [4](#0-3) 

`get_prev_header` walks the chain correctly via `prev_block_hash` links in `headers_pool`, but `get_header_by_height` ignores those links entirely and uses the height index.

The developers acknowledged this exact ambiguity with the comment at line 291:

> `// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor` [5](#0-4) 

---

### Impact Explanation

The attacker controls D's timestamp (within MTP and 2-hour-future constraints). By setting `D.time` significantly larger than `B.time`, the computed timespan `E.time − B.time` exceeds the correct `E.time − D.time`. Digishield clamps the modulated timespan to `[retarget_timespan − retarget_timespan/4, retarget_timespan + retarget_timespan/2]`: [6](#0-5) 

This allows up to a **+50% target increase (easier difficulty) per block** on every fork block at depth ≥ 2. The attacker can sustain this across the entire fork, progressively lowering the PoW requirement. If the fork accumulates sufficient chain work (easier with reduced difficulty), `reorg_chain` promotes it to the canonical chain, causing **false canonical header acceptance** with an incorrectly computed difficulty target. [7](#0-6) 

---

### Likelihood Explanation

- Requires the relayer role, which is explicitly listed as an in-scope attack path ("trusted-relayer submission path with attacker-relevant data").
- The Digishield interval-1 rule means the bug triggers on **every** fork block at depth ≥ 2 — no special timing is needed.
- The TODO comment confirms the developers identified this as an open correctness question, not a deliberate design choice.
- The attacker must mine valid PoW for the initial fork blocks (D, E) at the correct difficulty, but subsequent blocks (F onward) benefit from the reduced target.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from `prev_block_header`, going back `blocks_to_go_back` steps. This mirrors how `get_prev_header` already traverses the chain correctly and is the same pattern used by Zcash's `zcash_get_next_work_required`: [8](#0-7) 

The fix resolves the TODO at line 291 and ensures the retarget always uses the true fork-chain ancestor's timestamp regardless of what the main-chain index contains.

---

### Proof of Concept

1. Initialize the Dogecoin contract at height ≥ 145,001 with blocks A (h=N-1), B (h=N), C (h=N+1) on the main chain.
2. Submit fork block D building on A (h=N), with `D.time = B.time + 7000` (just under the 2-hour future limit). D's difficulty check uses `get_header_by_height(N-2)` — the common ancestor — so it passes correctly.
3. Submit fork block E building on D (h=N+1). E's difficulty check uses `get_header_by_height(N-1)` = A — correct.
4. Submit fork block F building on E (h=N+2). F's difficulty check calls `get_header_by_height(N)` = **B** (main chain), not D. The timespan `E.time − B.time` is inflated by ~7000 seconds, pushing the modulated timespan to `max_timespan`, yielding a ~50% easier target.
5. Assert that `F.bits` encodes the easier target and that the contract accepts F without panic.
6. Continue submitting fork blocks G, H, … each benefiting from the same inflation. Once fork chain work exceeds main chain work, `reorg_chain` promotes the fork, and `get_last_block_header` returns a block whose difficulty was computed against wrong ancestors throughout.
