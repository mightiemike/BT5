Looking at the inflation and tiered rewards modules for an analog to the "wrong oracle value used in financial calculation" vulnerability class.

The key analog is in `x/tieredrewards/keeper/bonus_rewards.go`: `computeSegmentBonus` uses the position's **current** delegation shares (`pos.Delegation.Shares`) when computing bonus for **all historical time segments**, not the shares that were actually held during those segments.

---

### Title
Current Delegation Shares Used for All Historical Bonus Segments Allows Retroactive Reward Inflation - (File: `x/tieredrewards/keeper/bonus_rewards.go`)

### Summary
`computeSegmentBonus` uses `pos.Delegation.Shares` — the position's share count at claim time — as the principal for every historical time segment. A user who calls `MsgAddToPosition` to increase their stake before claiming rewards causes all prior segments (back to `LastBonusAccrual`) to be computed with the inflated share count, extracting more bonus from the `RewardsPoolName` module account than was legitimately earned.

### Finding Description
In `processEventsAndClaimBonus`, the keeper walks every validator event since `pos.LastEventSeq` and calls `computeSegmentBonus` for each bonded segment:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, evt.Timestamp, evt.TokensPerShare)
``` [1](#0-0) 

Inside `computeSegmentBonus`, the token principal is computed as:

```go
tokens := pos.Delegation.Shares.Mul(tokensPerShare)
``` [2](#0-1) 

`tokensPerShare` is correctly snapshotted at the event boundary. However, `pos.Delegation.Shares` is the **live share count at claim time**, not the shares held during the segment. Validator events (bond/unbond/slash) are the only boundaries that advance `segmentStart`; a `MsgAddToPosition` call does not emit a validator event and does not advance `LastBonusAccrual` or `LastEventSeq`. Therefore, after an addition, all pending historical segments are replayed using the post-addition share count. [3](#0-2) 

The `applyBonusAccrualCheckpoint` that advances `LastBonusAccrual` is only called at the end of `processEventsAndClaimBonus` (i.e., during a reward claim), not during a position addition: [4](#0-3) 

### Impact Explanation
An attacker can drain the `RewardsPoolName` module account by:
1. Opening a position with a minimal amount (e.g., 1 basecro).
2. Waiting for validator bond/unbond/slash events to accumulate, building up a long un-claimed historical window.
3. Calling `MsgAddToPosition` with a large amount (e.g., 10,000,000 basecro), inflating `pos.Delegation.Shares`.
4. Calling `MsgClaimRewards` — every historical segment since `LastBonusAccrual` is now computed using the large share count, yielding bonus rewards orders of magnitude larger than legitimately earned.

The corrupted value is the `RewardsPoolName` module account balance: bonus tokens are transferred out via `bankKeeper.SendCoinsFromModuleToAccount` based on the inflated calculation. [5](#0-4) 

### Likelihood Explanation
The entry path requires only two standard user-signed transactions (`MsgAddToPosition` then `MsgClaimRewards`) with no privileged access. The longer the attacker waits between opening the position and adding funds, the larger the over-payment. Any user with sufficient funds to add to a position can execute this.

### Recommendation
Snapshot the delegation share count at each validator event (analogous to how `tokensPerShare` is already snapshotted in `ValidatorEvent`). Store `sharesAtEvent` alongside `tokensPerShare` in the event record and pass the snapshotted shares to `computeSegmentBonus` instead of `pos.Delegation.Shares`. Alternatively, force a reward claim (advancing `LastBonusAccrual` and `LastEventSeq` to the current block) inside `MsgAddToPosition` before updating the delegation shares.

### Proof of Concept
1. Alice opens a position with 1 basecro at block time `T0`. `LastBonusAccrual = T0`, `Shares ≈ 1`.
2. At `T1`, the validator emits a bond event (seq=1) with `tokensPerShare = 1.0`.
3. At `T2 = T0 + 1 year`, Alice calls `MsgAddToPosition` with 10,000,000 basecro. `Shares` becomes `≈ 10,000,001`. No reward claim occurs; `LastBonusAccrual` remains `T0`, `LastEventSeq` remains 0.
4. Alice calls `MsgClaimRewards`. `processEventsAndClaimBonus` replays from seq=0:
   - Segment `[T0, T1]`: `computeSegmentBonus` uses `Shares ≈ 10,000,001`, `tokensPerShare = 1.0`, `duration ≈ 1 year` → bonus ≈ `10,000,001 * bonusApy`.
   - Correct bonus for that segment: `1 * bonusApy`.
5. Alice receives ~10,000,000× the legitimate bonus for the pre-addition period, draining the bonus pool. [6](#0-5) [7](#0-6)

// computeSegmentBonus computes bonus for a time segment using a snapshot rate.
// Formula: shares * tokensPerShare * tier.BonusApy * durationSeconds / SecondsPerYear
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
