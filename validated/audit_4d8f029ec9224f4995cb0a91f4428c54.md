### Title
`GasPrice::wei_to_fri` Silently Returns Zero, Bypassing Oracle Fallback and Halting Block Production — (`crates/starknet_api/src/block.rs`)

### Summary

`GasPrice::wei_to_fri` performs integer division `(price_in_wei × eth_to_fri_rate) / WEI_PER_ETH` and returns `Ok(GasPrice(0))` without error when the numerator is smaller than `WEI_PER_ETH = 10^18`. The caller `get_l1_prices_in_fri_and_wei_and_conversion_rate` only falls back to previous-block data or configured defaults when the conversion returns `Err(…)`. A silent zero slips through as a successful conversion, so the fallback is never triggered. Downstream, `convert_to_sn_api_block_info` rejects the zero price via `NonzeroGasPrice::new`, causing every proposal build and validation to fail and halting block production.

---

### Finding Description

**Root cause — `GasPrice::wei_to_fri` (`crates/starknet_api/src/block.rs`, lines 429–442)**

```rust
pub fn wei_to_fri(self, eth_to_fri_rate: u128) -> Result<GasPrice, StarknetApiError> {
    // We use integer division since wei * eth_to_fri_rate is expected to be
    // high enough to not cause too much precision loss.
    Ok(self
        .checked_mul_u128(eth_to_fri_rate)
        .ok_or_else(|| { ... })?
        .checked_div(WEI_PER_ETH)          // ← integer division, no zero-check
        .expect("WEI_PER_ETH must be non-zero"))
}
```

The function returns `Ok(GasPrice(0))` whenever `price_in_wei × eth_to_fri_rate < 10^18`. The code comment even acknowledges the risk: *"expected to be high enough to not cause too much precision loss"* — but there is no guard that turns a zero result into an `Err`.

The condition is reachable. The test-fixture files already show `"price_in_fri": "0x0"` for historical blocks, and the constant comment in the same file states explicitly:

```rust
// These prices are in WEI. If we don't set them high enough the gas price when
// converted to FRI will be 0 (not allowed).
pub const TEMP_ETH_GAS_FEE_IN_WEI: u128 = u128::pow(10, 9); // 1 GWei.
```

**Fallback bypass — `get_l1_prices_in_fri_and_wei_and_conversion_rate` (`crates/apollo_consensus_orchestrator/src/utils.rs`, lines 159–182)**

```rust
let l1_gas_prices_fri_result =
    L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
// If conversion fails, leave return_value=None to try backup methods.
if let Ok(prices_in_fri) = l1_gas_prices_fri_result {
    return (prices_in_fri, prices_in_wei, eth_to_fri_rate); // ← zero accepted here
} else {
    warn!(...);
}
```

Because `wei_to_fri` returns `Ok(GasPrice(0))`, the `if let Ok(…)` branch is taken and the zero FRI price is returned as a valid result. The fallback to previous-block data (lines 186–208) and the fallback to configured defaults (lines 211–222) are never reached.

**Downstream rejection — `convert_to_sn_api_block_info` (`crates/apollo_consensus_orchestrator/src/utils.rs`, lines 301–347)**

```rust
if init.l1_gas_price_fri.0 == 0 ... {
    warn!("Zero gas price detected in block info: {:?}", init);
}
let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?; // ← Err(ZeroGasPrice)
```

`NonzeroGasPrice::new(GasPrice(0))` returns `Err(StarknetApiError::ZeroGasPrice)`. This propagates up through `initiate_validation` and causes both the proposer's `build_proposal` and the validator's `validate_proposal` to fail.

**Margin check does not save the situation — `within_margin` (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, lines 427–438)**

```rust
const GAS_PRICE_ABS_DIFF_MARGIN: u128 = 1;

fn within_margin(proposed: GasPrice, reference: GasPrice, margin_percent: u128) -> bool {
    if proposed.0.abs_diff(reference.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;  // ← 0 vs 0 passes here
    }
    ...
}
```

When both proposer and validator independently compute zero FRI prices from the same oracle data, `abs_diff(0, 0) = 0 ≤ 1` passes the margin check. The proposal is not rejected at the gas-price validation step; it proceeds to `convert_to_sn_api_block_info` where it fails.

---

### Impact Explanation

When the oracle-derived `eth_to_fri_rate` is low enough that `price_in_wei × eth_to_fri_rate < 10^18`:

1. `wei_to_fri` returns `Ok(GasPrice(0))` — no error, no fallback.
2. Both proposer and validator embed zero FRI gas prices in `ProposalInit`.
3. `convert_to_sn_api_block_info` fails with `ZeroGasPrice` on both sides.
4. Every proposal build and validation fails; no block is committed.
5. Block production halts entirely until the oracle rate recovers.

The fallback mechanism — which is explicitly designed to keep block production alive during oracle degradation — is rendered inoperative by the silent zero return. This matches the **High** impact category: the sequencer cannot sequence transactions, and the RPC cannot produce authoritative pending-block state.

---

### Likelihood Explanation

The condition `price_in_wei × eth_to_fri_rate < 10^18` is reachable under normal market conditions:

- `TEMP_ETH_GAS_FEE_IN_WEI = 10^9` (1 GWei) combined with `eth_to_fri_rate < 10^9` (i.e., 1 STRK costs more than 1 ETH) produces a zero result.
- The test fixture `block_post_0_13_1.json` already records `"price_in_fri": "0x0"` for a real historical block, confirming the condition has occurred on-chain.
- The `min_l1_gas_price_wei` config floor only helps if it is set high enough; a misconfigured or default-low minimum does not prevent the zero conversion.

---

### Recommendation

Add a zero-result guard inside `GasPrice::wei_to_fri`, mirroring the existing zero-rate guard in `calculate_eth_to_fri_rate`:

```diff
 pub fn wei_to_fri(self, eth_to_fri_rate: u128) -> Result<GasPrice, StarknetApiError> {
-    Ok(self
+    let result = self
         .checked_mul_u128(eth_to_fri_rate)
         .ok_or_else(|| {
             StarknetApiError::GasPriceConversionError(format!(
                 "Gas price is too high: {:?}, eth to fri rate: {:?}",
                 self, eth_to_fri_rate
             ))
         })?
         .checked_div(WEI_PER_ETH)
-        .expect("WEI_PER_ETH must be non-zero"))
+        .expect("WEI_PER_ETH must be non-zero");
+    if result.0 == 0 {
+        return Err(StarknetApiError::GasPriceConversionError(format!(
+            "wei_to_fri result is zero: price_in_wei={:?}, eth_to_fri_rate={:?}",
+            self, eth_to_fri_rate
+        )));
+    }
+    Ok(result)
 }
```

This causes `L1PricesInFri::convert_from_wei` to return `Err(…)`, which triggers the fallback chain in `get_l1_prices_in_fri_and_wei_and_conversion_rate` (previous-block data → configured defaults), preserving block-production liveness during low-rate oracle conditions.

```
Computation:
  10^9 × 5×10^8 = 5×10^17 < 10^18 = WEI_PER_ETH
```

```rust
fn within_margin(proposed: GasPrice, reference: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if proposed.0.abs_diff(reference.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    // Saturate: `reference.0 * margin_percent` can overflow u128 on large WEI prices.
    let margin = reference.0.saturating_mul(margin_percent) / 100;
    proposed.0.abs_diff(reference.0) <= margin
}
```
