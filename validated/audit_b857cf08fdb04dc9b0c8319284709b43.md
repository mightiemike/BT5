### Title
SwapAllowlistExtension Checks Router Address Instead of Actual EOA, Allowing Any User to Bypass Per-User Swap Allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` receives the `sender` argument that the pool passes as `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter.exactInputSingle`, the pool's `msg.sender` is the **router contract**, not the originating EOA. If the pool admin allowlists the router address, every EOA on the network can bypass the per-user allowlist by calling through the router.

### Finding Description

**Step 1 — Pool binds `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`. [1](#0-0) 

**Step 2 — Router is `msg.sender` to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router contract the pool's `msg.sender`. The originating EOA is only stored in transient callback context for payment purposes — it is never forwarded to the pool as the swap identity. [2](#0-1) 

**Step 3 — Extension checks router address, not EOA:**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the router address. The actual EOA is never visible to the extension. [3](#0-2) 

**Bypass path:**

A pool admin who wants to allow router-mediated swaps adds the router to the allowlist via `setAllowedToSwap(pool, router, true)`. From that point, any EOA — including ones the admin never intended to permit — can call `router.exactInputSingle` and the extension check passes because `allowedSwapper[pool][router] == true`. The per-user allowlist is completely neutralized. [4](#0-3) 

### Impact Explanation

The `SwapAllowlistExtension` is the sole on-chain mechanism for restricting which addresses may swap in a gated pool. When the router is allowlisted, the gate is open to the entire public. Any non-allowlisted user can execute swaps in a pool that was configured to be restricted, breaking the core access-control invariant the extension is designed to enforce. Depending on the pool's purpose (e.g., institutional-only, KYC-gated, whitelist-only liquidity), this enables unauthorized fund flows through the pool.

### Likelihood Explanation

Allowlisting the official router is the natural and expected configuration for any pool that wants to support the standard periphery swap UX. A pool admin has no on-chain signal that doing so opens the pool to all users. The `isAllowedToSwap` view function also returns `true` for any address when the router is allowlisted, giving no warning. [5](#0-4) 

### Recommendation

The extension must receive and check the **originating caller identity**, not the intermediary. Two complementary fixes:

1. **Router forwarding:** `MetricOmmSimpleRouter` should encode the originating `msg.sender` into `extensionData` (or a dedicated field) so the extension can decode and verify the real user.
2. **Extension-side decoding:** `SwapAllowlistExtension.beforeSwap` should, when `sender` is a known router, decode the real caller from `extensionData` and check that address against the allowlist instead.

Alternatively, the extension can maintain a separate `trustedRouter` mapping and require that the real caller be passed and verified through a signed or authenticated channel.
