### Title
Permissionless `MsgIssueDenom` Allows Attacker to Hijack IBC Voucher Denom, Permanently Blocking NFT Receipt and Enabling Unauthorized Voucher Minting — (`x/nft-transfer/keeper/packet.go`, `x/nft/keeper/keeper.go`)

### Summary

`MsgIssueDenom` is permissionless: any account can register any denom ID. The IBC voucher class ID (`ibc/{sha256(destPort/destChannel/classID)}`) is fully deterministic. An attacker who pre-registers this denom with their own address as creator causes `processReceivedPacket` to skip `IssueDenom` (denom already exists) but then fail at `MintNFT` (escrow address is not the creator). This produces an error acknowledgement for every subsequent packet carrying that classID on that channel, permanently blocking IBC NFT receipt. As a secondary impact, the attacker — now the denom creator — can freely mint fake IBC voucher NFTs.

### Finding Description

`processReceivedPacket` guards denom creation with `HasDenomID` but then unconditionally calls `MintNFT` with `escrowAddress` as the `sender` (creator check argument): [1](#0-0) 

`MintNFT` enforces `IsDenomCreator` against that sender: [2](#0-1) 

`MsgIssueDenom` has no authority guard — the caller's address becomes the denom creator: [3](#0-2) 

The `NFTKeeper` interface used by the IBC module exposes `MintNFT` (creator-checked), not `MintNFTUnverified`: [4](#0-3) 

`MintNFTUnverified` exists precisely for IBC/genesis use cases (no creator check), but is never wired into the IBC receive path: [5](#0-4) 

### Impact Explanation

**Impact 1 — Permanent blocking of IBC NFT receipt:** Once the attacker's denom is registered, every `OnRecvPacket` for that `(classID, destPort, destChannel)` triple returns an error ack. The sender chain's `OnAcknowledgementPacket` calls `refundPacketToken`, which correctly unescrows the NFT back to the sender (no permanent fund loss for the sender), but the transfer is permanently blocked for all future attempts on that channel. [6](#0-5) 

**Impact 2 — Unauthorized voucher minting:** The attacker, now the denom creator for `voucherClassID`, can call `MsgMintNFT` with themselves as sender and mint arbitrary NFTs under the IBC voucher class ID. These fake vouchers carry the canonical `ibc/{hash}` class ID and are indistinguishable on-chain from legitimate IBC-transferred NFTs, enabling fraud against users who receive or purchase them.

### Likelihood Explanation

- The `voucherClassID` is fully deterministic from public on-chain data (`destPort`, `destChannel`, `classID`).
- `MsgIssueDenom` requires only a valid sender and a unique ID — no governance, no special role.
- The attacker can front-run any observed `SendTransfer` transaction or pre-register the denom speculatively for any channel.
- No existing guard prevents a user from registering an `ibc/…` prefixed denom ID.

### Recommendation

Replace `MintNFT` with `MintNFTUnverified` in `processReceivedPacket`, mirroring the pattern already used for `BurnNFTUnverified` in `createOutgoingPacket`: [7](#0-6) 

Additionally, consider restricting `MsgIssueDenom` from accepting IDs that match the `ibc/` prefix, or validating in `processReceivedPacket` that if the denom already exists its creator is the expected escrow address before proceeding to mint.

// MintNFTUnverified mints an NFT without verifying if the owner is the creator of denom
// Needed during genesis initialization
func (k Keeper) MintNFTUnverified(ctx sdk.Context, denomID, tokenID, tokenNm, tokenURI, tokenData string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	if k.HasNFT(ctx, denomID, tokenID) {
		return sdkerrors.Wrapf(types.ErrNFTAlreadyExists, "NFT %s already exists in collection %s", tokenID, denomID)
	}

	k.setNFT(
		ctx, denomID,
		types.NewBaseNFT(
			tokenID,
			tokenNm,
			owner,
			tokenURI,
			tokenData,
		),
	)
	k.setOwner(ctx, denomID, tokenID, owner)
	k.increaseSupply(ctx, denomID)

	return nil
}

func (m msgServer) IssueDenom(goCtx context.Context, msg *types.MsgIssueDenom) (*types.MsgIssueDenomResponse, error) {
	sender, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return nil, err
	}

	ctx := sdk.UnwrapSDKContext(goCtx)
	if err := m.Keeper.IssueDenom(ctx, msg.Id, msg.Name, msg.Schema, msg.Uri, sender); err != nil {
		return nil, err
	}

	ctx.EventManager().EmitEvents(sdk.Events{
		sdk.NewEvent(
			types.EventTypeIssueDenom,
			sdk.NewAttribute(types.AttributeKeyDenomID, msg.Id),
			sdk.NewAttribute(types.AttributeKeyDenomName, msg.Name),
			sdk.NewAttribute(types.AttributeKeyCreator, msg.Sender),
		),
		sdk.NewEvent(
			sdk.EventTypeMessage,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.AttributeValueCategory),
			sdk.NewAttribute(sdk.AttributeKeySender, msg.Sender),
		),
	})

	return &types.MsgIssueDenomResponse{}, nil
}
