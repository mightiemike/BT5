Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of initiating user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of the pool. When any user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` to the pool, so the extension evaluates the router's allowlist status rather than the actual initiating user's. This makes the per-user swap allowlist silently bypassable by any unprivileged user who calls through the official periphery router.

## Finding Description
`MetricOmmPool.swap` invokes `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` receives that value as `sender` and checks it against the per-pool allowlist: [2](#0-1) 

When `MetricOmmSimpleRouter.exactInputSingle` (or any other router swap entry point) calls `IMetricOmmPoolActions(params.pool).swap(...)`, the router is `msg.sender` to the pool: [3](#0-2) 

This creates an irreconcilable actor mismatch. The check `allowedSwapper[pool][router]` is evaluated instead of `allowedSwapper[pool][user]`. If the pool admin allowlists the router to enable periphery access, every user bypasses the individual allowlist. If the admin does not allowlist the router, individually allowlisted users cannot trade through the router at all.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter — the actual position beneficiary — not the intermediary caller: [4](#0-3) 

The asymmetry between the two extensions confirms the swap path has the wrong actor bound.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., KYC'd addresses, whitelisted institutions) can be freely traded against by any unprivileged user by calling through `MetricOmmSimpleRouter`. The allowlist guard is silently bypassed, exposing LP funds to trades the pool admin explicitly intended to block. This constitutes broken core pool functionality and an admin-boundary break via an unprivileged path, matching the allowed impact gate.

**Severity: High**

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, publicly documented periphery entry point requiring no special privilege. Any user aware of the router can exploit this without any on-chain precondition beyond the pool admin having allowlisted the router (the natural action to enable periphery use). The pool admin has no on-chain mechanism to prevent router-mediated bypass while still allowing allowlisted users to trade through the router. The exploit is repeatable on every swap.

**Likelihood: High**

## Recommendation
Gate on the actual initiating user rather than the immediate pool caller. The preferred approach is to extend the `beforeSwap` hook to carry the true originating address — either via a dedicated `initiator` field in `extensionData` populated by the router, or by checking `recipient` if pool semantics guarantee it equals the economic beneficiary. The deposit allowlist's pattern of checking `owner` (the actual beneficiary, not `msg.sender`) should be mirrored in the swap allowlist. [5](#0-4) 

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured on `beforeSwap`.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allow the official `MetricOmmSimpleRouter` (required for any allowlisted user to trade through the periphery).
3. Attacker (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool.
4. The router calls `IMetricOmmPoolActions(pool).swap(recipient, ...)` — router is now `msg.sender` to the pool.
5. Pool calls `_beforeSwap(msg.sender=router, recipient, ...)`.
6. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router]` → `true` → no revert.
7. The attacker's swap executes successfully despite not being on the allowlist. [6](#0-5) [7](#0-6)
