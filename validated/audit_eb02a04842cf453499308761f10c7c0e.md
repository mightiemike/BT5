### Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Allowlist Bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` from the pool's perspective. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router to support router-mediated swaps for legitimate users, any non-allowlisted user can bypass the gate by calling the router — the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
``` [1](#0-0) 

The pool's `swap` function passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol
_beforeSwap(
    msg.sender,   // <-- whoever called pool.swap()
    recipient,
    ...
);
``` [2](#0-1) 

`ExtensionCalling._beforeSwap` then encodes this `sender` and calls the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
``` [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle(...)` or `exactInput(...)`, the router is the direct caller of `pool.swap()`. The pool therefore passes `address(router)` as `sender` to the extension. The extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

This creates an irresolvable dilemma for any pool admin who wants to use the allowlist with the router:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all — broken UX |
| **Allowlist the router** | Any non-allowlisted user bypasses the gate by calling the router |

The second branch is the exploitable path. The router is a public, permissionless contract. Allowlisting it is the only way to support router-mediated swaps, but doing so opens the gate to everyone.

The `DepositAllowlistExtension` does not share this flaw because it checks `owner` (the LP position owner), which the pool passes directly from the `addLiquidity` call parameter — not from `msg.sender`. The swap path has no equivalent owner-level argument; the only identity available is `sender = msg.sender`. [4](#0-3) 

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or whitelisted market makers) can have that restriction completely bypassed. Any non-allowlisted user calls `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting the restricted pool. The router forwards the swap to the pool; the extension sees `sender = router`; if the router is allowlisted, the swap executes. The non-allowlisted user receives pool output tokens, draining LP assets at oracle-derived prices without authorization. This is a direct loss of LP principal and a complete failure of the pool's access-control invariant.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing swap interface documented in the periphery. Any pool admin who deploys a swap-allowlisted pool and also wants to support the standard router path must allowlist the router — the protocol provides no other mechanism. The bypass is therefore reachable on any production pool that uses both `SwapAllowlistExtension` and the router, which is the expected deployment pattern. No special privileges, flash loans, or non-standard tokens are required; a single public `exactInputSingle` call suffices.

---

### Recommendation

1. **Pass the originating user through `extensionData`**: Have `MetricOmmSimpleRouter` encode `msg.sender` into the `extensionData` bytes it forwards to the pool. Update `SwapAllowlistExtension.beforeSwap` to decode and check this address when `sender` is a known router, falling back to `sender` for direct calls.

2. **Alternatively, add a `swapFor` / `originator` field to the pool's swap interface**: The pool could accept an explicit `originator` address (verified via a trusted-router registry) and pass it as `sender` to extensions, preserving the allowlist semantics regardless of routing path.

3. **At minimum, document the incompatibility**: If neither fix is applied, the extension's NatSpec must warn that allowlisting the router defeats the allowlist for all users, and that router-mediated swaps are unsupported on allowlisted pools.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls swapExtension.setAllowedToSwap(pool, router, true)
    (to support router-mediated swaps for legitimate users)
  - Pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  1. attacker (not allowlisted) calls:
       router.exactInputSingle(ExactInputSingleParams({
           pool: restrictedPool,
           tokenIn: token0,
           ...
           recipient: attacker,
           extensionData: ""
       }))

  2. Router calls pool.swap(attacker_as_recipient, ...)
     with msg.sender = router

  3. Pool calls _beforeSwap(sender=router, ...)

  4. ExtensionCalling calls extension.beforeSwap(sender=router, ...)

  5. Extension checks allowedSwapper[pool][router] == true  ✓
     → does NOT revert

  6. Swap executes; attacker receives output tokens from the pool

Result:
  - attacker bypassed the swap allowlist
  - LP assets transferred to a non-allowlisted address
  - SwapAllowlistExtension invariant broken
```

The root cause is structurally identical to the TermAuctionBidLocker analog: the guard validates one identity (the configured allowlist entries for real users) but the enforcement is applied against a different identity (the router), so the guard is silently bypassed for the entire class of router-mediated swaps.
