### Title
Wrong Weight Used in `_getSpreadWeightX18` for Short-Perp Leg of Spread — (`core/contracts/libraries/RiskHelper.sol`)

---

### Summary

`_getSpreadWeightX18` always reads `perpCoreRisk.longWeight` for the long-spot/short-perp branch. Because `getCoreRisk` hardcodes `amount = 1` when constructing `CoreRisk`, the single `longWeight` field is **always** the product's long weight regardless of the actual position direction. For a short perp the correct weight is `shortWeight ≥ 1`; using `longWeight ≤ 1` instead produces a **lower** spread weight, which for a negative (short) amount yields a **less-negative** health contribution — overstating INITIAL health.

---

### Finding Description

**Root cause 1 — `getCoreRisk` always returns long weight** [1](#0-0) 

```solidity
return IProductEngine.CoreRisk(
    amount,
    risk.priceX18,
    RiskHelper._getWeightX18(risk, 1, healthType)  // ← hardcoded 1, always longWeight
);
```

`_getWeightX18` branches on `amount >= 0`; passing the literal `1` means it always returns `longWeightInitialX18` / `longWeightMaintenanceX18`, never `shortWeightInitialX18`. [2](#0-1) 

**Root cause 2 — `_getSpreadWeightX18` uses `perpCoreRisk.longWeight` for the short-perp leg** [3](#0-2) 

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;   // ← wrong for short perp
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;   // ← wrong for short spot
}
```

**Protocol invariant on weights** (enforced in `_addOrUpdateProduct`): [4](#0-3) 

```
longWeightMaintenance  ≤ 1e9   →  longWeight  ≤ 1.0
shortWeightMaintenance ≥ 1e9   →  shortWeight ≥ 1.0
```

So `longWeight ≤ 1 ≤ shortWeight` always. The question's premise that `longWeight > shortWeight` is **factually wrong**, but the vulnerability is still real — the direction of the error is simply the opposite of what the question states.

**Numeric example (INITIAL health, volatile asset):**

| Parameter | Value |
|---|---|
| `longWeightInitial` | 0.80 |
| `shortWeightInitial` | 1.20 |
| Short perp amount | −100 |
| Price | 1.0 |

With **correct** `shortWeight`:
`spreadWeight = 1 − (1 − 1.20)/5 = 1.04` → capped at **0.99**
Health contribution = −100 × 0.99 = **−99**

With **buggy** `longWeight`:
`spreadWeight = 1 − (1 − 0.80)/5 = 0.96`
Health contribution = −100 × 0.96 = **−96**

INITIAL health is overstated by **+3** per 100 units of notional. The trader can open additional positions or withdraw collateral that the true risk parameters do not permit.

---

### Impact Explanation

A trader holding a long-spot/short-perp spread has their INITIAL health overstated. They can:
- Open additional leveraged positions beyond the true risk limit.
- Withdraw collateral that should be locked as margin.

This directly matches the scoped critical impact: *unauthorized mutation of protected trading state* (health accounting, withdrawal rights).

---

### Likelihood Explanation

- Reachable through any normal spread position (public order flow via `Endpoint`).
- No special privileges required.
- The effect scales with the spread between `longWeight` and `shortWeight`, which is larger for volatile assets — making it more impactful precisely where risk controls matter most.
- The `CoreRisk` struct has only one weight field (`longWeight`), so there is no way for `_getSpreadWeightX18` to access `shortWeight` through the current interface; the bug is structural. [5](#0-4) 

---

### Recommendation

1. Add a `shortWeight` field to `CoreRisk`:
   ```solidity
   struct CoreRisk {
       int128 amount;
       int128 price;
       int128 longWeight;
       int128 shortWeight;   // ← add
   }
   ```
2. In `getCoreRisk`, populate both fields using the actual `amount` for `longWeight` and `-1` for `shortWeight`:
   ```solidity
   IProductEngine.CoreRisk(
       amount,
       risk.priceX18,
       RiskHelper._getWeightX18(risk, 1, healthType),
       RiskHelper._getWeightX18(risk, -1, healthType)
   );
   ```
3. In `_getSpreadWeightX18`, use the directionally correct weight:
   ```solidity
   if (spotCoreRisk.amount > 0) {
       // long spot / short perp: use perp's shortWeight
       spreadWeight = ONE - (ONE - perpCoreRisk.shortWeight) / 5;
   } else {
       // short spot / long perp: use spot's shortWeight
       spreadWeight = ONE - (ONE - spotCoreRisk.shortWeight) / 5;
   }
   ```

---

### Proof of Concept

```solidity
// Fuzz: for any (longW, shortW) with longW < 1e18 < shortW
// show that _getSpreadWeightX18 uses longW for the short-perp leg

CoreRisk memory perpCoreRisk = CoreRisk({
    amount: -100e18,   // short perp
    price:  1e18,
    longWeight: 0.8e18  // getCoreRisk always returns this
});
CoreRisk memory spotCoreRisk = CoreRisk({
    amount: 100e18,    // long spot
    price:  1e18,
    longWeight: 0.8e18
});

int128 sw = RiskHelper._getSpreadWeightX18(
    perpCoreRisk, spotCoreRisk, HealthType.INITIAL
);
// sw = 1e18 - (1e18 - 0.8e18)/5 = 0.96e18
// Correct would use shortWeight=1.2e18 → capped at 0.99e18
// Health delta per 100 notional: (-100 * 0.96) - (-100 * 0.99) = +3 (overstated)
assert(sw < 0.99e18); // passes — confirms lower weight used, health overstated for short
```
