### Title
`SwapAllowlistExtension` Checks the Router Address Instead of the Actual Swapper, Enabling Full Allowlist Bypass Through the Supported Router Path — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap` call. When `MetricOmmSimpleRouter` intermediates, `sender` resolves to the **router address**, not the actual user. A pool admin who allowlists the router (the natural operational choice to enable router-based swaps) inadvertently grants every user on-chain access to the curated pool, completely defeating the allowlist.

---

### Finding Description

**Call chain when a user swaps through the router:**

1. User calls `router.exactInputSingle(params)` — `msg.sender` in the router = user.
2. Router calls `pool.swap(params.recipient, ...)` — `msg.sender` in the pool = **router address**.
3. Pool calls `_beforeSwap(msg.sender, ...)` = `_beforeSwap(router_address, ...)`.
4. Extension `beforeSwap` receives `sender = router_address`.
5. Extension evaluates `allowedSwapper[pool][router_address]`, **not** `allowedSwapper[pool][user]`. [1](#0-0) [2](#0-1) [3](#0-2) 

The pool passes `msg.sender` (the direct caller) as `sender` to the extension. The router calls the pool directly, so the extension always sees the router address as the identity to check.

**Contrast with `DepositAllowlistExtension`:** The deposit guard correctly checks `owner` — the explicit position-owner address passed as a separate parameter — not `sender` (the direct caller of `addLiquidity`). This means the deposit allowlist correctly identifies the economic actor regardless of which intermediary calls the pool. [4](#0-3) 

The swap allowlist has no equivalent "actual user" parameter to check because the `beforeSwap` interface only receives `sender` (the direct pool caller) and `recipient` (the output destination). Neither is the originating user when the router intermediates. [5](#0-4) 

---

### Impact Explanation

**Scenario A — Full allowlist bypass (High):**
A pool admin deploys a curated pool with `SwapAllowlistExtension` and calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps for their allowlisted users. Because the check resolves to `allowedSwapper[pool][router]`, every user who calls `router.exactInputSingle` or `router.exactInput` passes the guard. Unauthorized users trade against LP positions on a pool that was intended to be restricted, draining LP value at oracle-anchored prices without the LP's consent.

**Scenario B — Allowlisted users locked out (High/Medium):**
A pool admin allowlists individual addresses (e.g., `allowedSwapper[pool][alice] = true`) but does not allowlist the router. When Alice uses the router, the extension checks `allowedSwapper[pool][router]` = `false` and reverts `NotAllowedToSwap`. Alice cannot use the primary supported periphery path even though she is explicitly permitted. Core swap functionality is broken for all allowlisted users who rely on the router.

---

### Likelihood Explanation

Likelihood is **High**. The `MetricOmmSimpleRouter` is the primary and only supported periphery swap path. Pool admins who configure a swap allowlist will naturally need to enable router-based access. Allowlisting the router address is the obvious operational step, and it silently opens the pool to all users. The bug is invisible during normal testing because the test suite for `SwapAllowlistExtension` calls the extension directly from a mock pool address, never exercising the router path. [6](#0-5) 

---

### Recommendation

The `beforeSwap` hook signature must carry the originating user's address as a distinct parameter, separate from `sender` (the direct pool caller). Until the interface is extended, `SwapAllowlistExtension` should document that `sender` equals the direct caller of `pool.swap` and that router-mediated swaps will present the router address. The correct fix is one of:

1. **Extend the hook interface** to pass an explicit `origin` (the original EOA or initiating contract) alongside `sender`, and check `allowedSwapper[pool][origin]`.
2. **Check `recipient` as a proxy** only if the protocol guarantees recipient == user (it does not in the current design).
3. **Require users to call `pool.swap` directly** and document that the router is incompatible with `SwapAllowlistExtension` — but this breaks the intended UX.

Option 1 is the only correct fix.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; admin allowlists the router
swapExtension.setAllowedToSwap(address(pool), address(router), true);
// alice is NOT individually allowlisted
// allowedSwapper[pool][alice] == false

// Alice calls the router — extension sees sender = router, passes
vm.prank(alice);
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:            address(pool),
        tokenIn:         address(token0),
        tokenOut:        address(token1),
        zeroForOne:      true,
        amountIn:        1_000,
        amountOutMinimum: 0,
        recipient:       alice,
        deadline:        block.timestamp + 1,
        priceLimitX64:   0,
        extensionData:   ""
    })
);
// ✓ swap succeeds — alice bypassed the allowlist
```

The pool's `_beforeSwap` receives `sender = address(router)`. [7](#0-6) 

The extension evaluates `allowedSwapper[pool][router]` = `true` and does not revert, even though `allowedSwapper[pool][alice]` = `false`. [8](#0-7)
