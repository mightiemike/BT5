### Title
Reentrancy via Token Transfer Hook in `withdrawCollateral` Allows Over-Withdrawal Before Balance Update — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` transfers collateral tokens to the recipient **before** decrementing the subaccount's balance in `SpotEngine` and before performing the health check. If a listed collateral token implements transfer hooks (e.g., ERC-777 `tokensReceived`), an attacker can re-enter the protocol via the public, unguarded `Endpoint.executeSlowModeTransaction()` during the callback, processing a second withdrawal while the first withdrawal's balance decrement has not yet occurred. With perp-position health covering the resulting negative spot balance, both health checks pass and the attacker withdraws more collateral than they deposited, leaving the protocol undercollateralized.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)` — tokens are transferred out  
2. **Line 412** — `spotEngine.updateBalance(...)` — balance is decremented  
3. **Line 419** — `require(getHealth(...) >= 0)` — health is checked [1](#0-0) 

`handleWithdrawTransfer` in `Clearinghouse` first moves tokens to the `withdrawPool`, then calls `BaseWithdrawPool.submitWithdrawal`, which calls `handleWithdrawTransfer(token, sendTo, amount)` — a `token.safeTransfer(to, amount)` call directly to the user-controlled `sendTo` address. [2](#0-1) [3](#0-2) [4](#0-3) 

`ERC20Helper.safeTransfer` uses a low-level `.call()`, which will invoke any `tokensReceived` hook on the recipient if the token is ERC-777 compliant. [5](#0-4) 

`Endpoint.executeSlowModeTransaction()` is a public function with no reentrancy guard. It processes the next pending slow-mode transaction from the queue, which can be a `WithdrawCollateral` transaction, ultimately calling `clearinghouse.withdrawCollateral` again. [6](#0-5) 

A grep across all production contracts confirms **no `nonReentrant` modifier or `ReentrancyGuard` is used anywhere** in the codebase.

During the re-entrant inner call, `spotEngine.updateBalance` has not yet been called for the outer withdrawal. The inner call therefore reads the subaccount's full pre-withdrawal balance, passes its own health check, and completes. When control returns to the outer call, `updateBalance` is called a second time, decrementing the balance again. If the attacker holds open perp positions that provide sufficient health to cover the resulting negative spot balance, the outer health check also passes.

---

### Impact Explanation

An attacker can withdraw more collateral than their spot balance permits. The shortfall is masked by perp-position health at the time of the outer health check. The clearinghouse pays out real tokens from the shared collateral pool, leaving the protocol undercollateralized. Other users' ability to withdraw their own collateral is directly impaired — an exact analog to the DAT scenario where investors could not redeem FAIR for currency.

---

### Likelihood Explanation

**Medium.** Two conditions are required:

1. A collateral token with transfer hooks (ERC-777 or equivalent) must be listed. ERC-777 tokens exist in production DeFi and the protocol's `IERC20Base` interface does not exclude them.
2. The attacker must hold open perp positions providing enough health to cover the double-decrement. This is a normal protocol usage pattern.

The attacker controls the `sendTo` address (via `WithdrawCollateralV2.sendTo`) and can deploy a contract to receive the transfer and trigger the callback. The slow-mode queue is permissionless after the 3-day timeout, so no sequencer cooperation is needed. [7](#0-6) 

---

### Recommendation

Apply the **checks-effects-interactions** pattern: call `spotEngine.updateBalance` and `require(getHealth(...) >= 0)` **before** calling `handleWithdrawTransfer`. Alternatively, add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to `withdrawCollateral` and `executeSlowModeTransaction`.

---

### Proof of Concept

1. Attacker deploys `MaliciousReceiver` implementing ERC-777 `tokensReceived`.
2. Attacker deposits 100 USDC and opens a perp position providing +50 health units.
3. Attacker submits two slow-mode `WithdrawCollateral` transactions of 75 USDC each, with `sendTo = address(MaliciousReceiver)`, at least 3 days in advance.
4. After the timeout, attacker calls `endpoint.executeSlowModeTransaction()` (withdrawal 1).
5. Execution path: `_executeSlowModeTransaction` → `processSlowModeTransaction` → `clearinghouse.withdrawCollateral` → `handleWithdrawTransfer` → `withdrawPool.submitWithdrawal` → `token.safeTransfer(MaliciousReceiver, 75)`.
6. `MaliciousReceiver.tokensReceived()` fires. Inside, attacker calls `endpoint.executeSlowModeTransaction()` (withdrawal 2).
   - Inner `withdrawCollateral`: SpotEngine balance still reads 100 USDC (outer decrement not yet applied). Transfers 75 USDC. `updateBalance`: 100 → 25. Health = 25 + 50 = 75 ≥ 0 ✓.
7. Outer `withdrawCollateral` resumes. `updateBalance`: 25 → −50. Health = −50 + 50 = 0 ≥ 0 ✓.
8. **Result**: Attacker received 150 USDC, deposited 100 USDC. Protocol is undercollateralized by 50 USDC at the expense of other depositors. [8](#0-7) [6](#0-5)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
