### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Accepts Arbitrary `tx_id` Without Leaf-Node Validation — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully public, unpermissioned NEAR entry point. It accepts any 32-byte hash as `tx_id` without validating that the value is a leaf-level transaction hash rather than an internal Merkle tree node. This is the direct analog to M-03's missing input validation: just as `redeem()` accepted any ERC20 token address instead of only wBTC/renBTC, this function accepts any `tx_id` instead of only valid leaf transactions. The result is that an unprivileged NEAR caller can obtain a `true` SPV inclusion proof for a non-existent Bitcoin transaction.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub` with only a `#[pause]` attribute and a Rust `#[deprecated]` annotation: [1](#0-0) 

Rust's `#[deprecated]` is a compile-time lint warning only. It does **not** restrict external callers from invoking the function via NEAR's RPC API. Any unprivileged NEAR account can call it directly, bypassing the intent to migrate callers to v2.

Inside the function, the only checks performed are:

1. `confirmations <= gc_threshold`
2. `tx_block_blockhash` is on the mainchain (via `mainchain_header_to_height`)
3. Enough confirmations exist
4. `merkle_proof` is non-empty [2](#0-1) 

There is **no check** that `tx_id` is a leaf-level transaction hash. The function then computes: [3](#0-2) 

`compute_root_from_merkle_proof` in `merkle-tools` uses only `current_position % 2` and `current_position /= 2` to determine left/right placement at each level: [4](#0-3) 

This means an internal Merkle tree node at depth `d` can be supplied as `tx_id` with a valid proof of length `(tree_depth - d)`, and the computed root will match the block's `merkle_root`, causing the function to return `true`.

The v2 function mitigates this by requiring `merkle_proof.len() == coinbase_merkle_proof.len()`, which forces the proof depth to match the full tree depth (since the coinbase is always at index 0 with a full-depth proof): [5](#0-4) 

But v1 has no such constraint and remains callable.

---

### Impact Explanation

Any downstream contract or application that calls `verify_transaction_inclusion` and acts on a `true` result — for example, a cross-chain bridge releasing funds upon confirmed Bitcoin payment — can be deceived. An attacker can forge a proof that a non-existent Bitcoin transaction is included in a real, mainchain block. The corrupted proof result is: `true` returned for a `tx_id` that is an internal Merkle node, not a real transaction. This directly enables theft from bridge contracts or other SPV-dependent applications.

---

### Likelihood Explanation

Medium. Bitcoin block data is fully public. The 64-byte transaction Merkle forgery technique is well-documented (referenced in the contract's own deprecation notice and the bitmex blog post cited in the code). No special permissions are required — any NEAR account can call `verify_transaction_inclusion` directly. The only barrier is attacker knowledge of the technique.

---

### Recommendation

1. **Remove public access to v1**: Make `verify_transaction_inclusion` `pub(crate)` or add an explicit access-control role guard so it cannot be called externally.
2. **Enforce proof-length parity in v1**: Add `require!(args.merkle_proof.len() == expected_depth)` based on `tx_index`, matching the mitigation already present in v2.
3. **Direct all callers to v2**: The deprecation notice is insufficient — enforce migration at the contract level.

---

### Proof of Concept

1. Identify a real Bitcoin mainchain block `B` with at least 4 transactions: `T0` (coinbase), `T1`, `T2`, `T3`.
2. Compute the internal Merkle node `N = double_sha256(T2 || T3)`. `N` is a 32-byte internal node, not a real transaction.
3. Compute a valid Merkle proof for `N`'s position in the tree. The proof has length `tree_depth - 1` (one level shorter than a leaf proof).
4. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node)
   - `tx_block_blockhash` = `B`'s block hash (a real mainchain block)
   - `tx_index` = the index corresponding to `N`'s position
   - `merkle_proof` = the valid proof for `N`
   - `confirmations` = 1
5. `compute_root_from_merkle_proof(N, index, proof)` reconstructs the correct `merkle_root` of block `B`, so the function returns `true` — falsely asserting that the non-existent transaction `N` is included in block `B`. [6](#0-5)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
