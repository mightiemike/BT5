### Title
Unsigned `feeX18` in `WithdrawCollateralV2` EIP-712 Typehash Allows Sequencer to Charge Arbitrary Withdrawal Fees Without User Authorization — (`File: core/contracts/interfaces/IEndpoint.sol`, `core/contracts/EndpointTx.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `SignedWithdrawCollateralV2` struct carries a `feeX18` field that is **outside** the EIP-712 signed payload. The `WITHDRAW_COLLATERAL_V2_SIGNATURE` typehash does not include `feeX18`, so the user's compact signature never commits to the fee amount. The sequencer can freely set `feeX18` to any value from `0` up to the protocol-configured `currentFeeX18` when submitting a `WithdrawCollateralV2` transaction, and the contract will deduct that fee from the user's subaccount balance without the user having authorized that specific amount.

---

### Finding Description

**Unsigned field in the signed struct:**

`SignedWithdrawCollateralV2` is defined as:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;      // signed payload
    CompactSignature signature;
    int128 feeX18;                // NOT in the signed payload
}
``` [1](#0-0) 

The EIP-712 typehash used to verify the user's signature is:

```
"WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix)"
``` [2](#0-1) 

`feeX18` is absent from this string. The digest computed in `Verifier.computeDigest` for `WithdrawCollateralV2` encodes only the six fields of the inner `WithdrawCollateralV2` struct: [3](#0-2) 

**How `feeX18` is consumed:**

In `EndpointTx.processTransactionImpl`, after signature validation succeeds, the code reads `signedTx.feeX18` directly from the caller-supplied transaction bytes and charges it against the user's subaccount:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [4](#0-3) 

`chargeFee` deducts the fee from the user's on-chain spot balance and credits `sequencerFee`:

```solidity
function chargeFee(bytes32 sender, int128 fee, uint32 productId) internal {
    spotEngine.updateBalance(productId, sender, -fee);
    sequencerFee[productId] += fee;
}
``` [5](#0-4) 

The user signed a withdrawal for a specific `amount` but never authorized any particular fee value. The sequencer can set `feeX18` to the maximum allowed value (`currentFeeX18`) on every submission without the user's knowledge or consent.

**Contrast with `WithdrawCollateral` (V1):**

The V1 path charges the fee using the on-chain configured value directly — `spotEngine.getConfig(...).withdrawFeeX18` — with no caller-supplied fee field at all: [6](#0-5) 

V2 introduced `feeX18` as a caller-supplied, unsigned parameter, creating the vulnerability.

---

### Impact Explanation

The sequencer can charge the maximum configured withdrawal fee on every `WithdrawCollateralV2` transaction without the user having signed off on that specific fee amount. The deduction is applied to the user's spot collateral balance (`spotEngine.updateBalance`), reducing the net assets they receive from the withdrawal. Because `feeX18` is not part of the EIP-712 digest, no amount of careful signing by the user can constrain the fee to a value they explicitly agreed to. The impact is bounded by `currentFeeX18` (the protocol-configured maximum), but within that bound the user has zero on-chain protection.

---

### Likelihood Explanation

The sequencer is a privileged, trusted entity in the Nado architecture. Exploitation requires the sequencer to act against user interests. This is a weaker trigger than the original report (where any mempool observer could frontrun), but the root cause — an unsigned fee parameter in the EIP-712 typehash — is identical. The design flaw means the protocol provides no cryptographic guarantee to users about the fee they will pay, regardless of sequencer behavior. Any future sequencer rotation, compromise, or upgrade that introduces a fee-maximizing policy would immediately exploit this gap with no on-chain recourse for users.

---

### Recommendation

Add `feeX18` to the `WITHDRAW_COLLATERAL_V2_SIGNATURE` typehash and to the digest computation in `Verifier.computeDigest`:

```solidity
string internal constant WITHDRAW_COLLATERAL_V2_SIGNATURE =
    "WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)";
```

Update the `abi.encode` call in `Verifier.computeDigest` for `WithdrawCollateralV2` to include `signedTx.feeX18` as the seventh field. This ensures the user's compact signature cryptographically commits to the exact fee they are willing to pay, preventing any party from substituting a different value post-signing.

---

### Proof of Concept

1. Alice signs a `WithdrawCollateralV2` intent: `sender=Alice`, `productId=1`, `amount=1000e6`, `nonce=5`, `sendTo=Alice`, `appendix=0`. Her signature covers only these six fields. She expects to pay a small fee.

2. The sequencer (or a future malicious operator) constructs the transaction bytes with `feeX18 = currentFeeX18` (the maximum configured value).

3. `EndpointTx.processTransactionImpl` decodes the transaction, calls `validateSignedTx` — which verifies Alice's signature against the digest of the six-field typehash — and the check passes because `feeX18` is not in the digest.

4. The code then executes:
   ```solidity
   require(signedTx.feeX18 >= 0);           // passes
   require(signedTx.feeX18 <= currentFeeX18); // passes (set to max)
   chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
   ```
   Alice's subaccount balance is reduced by `currentFeeX18` — the maximum fee — even though she never authorized that amount.

5. Alice receives her withdrawal minus the maximum fee, with no on-chain mechanism to have enforced a lower fee at signing time. [1](#0-0) [2](#0-1) [7](#0-6)
