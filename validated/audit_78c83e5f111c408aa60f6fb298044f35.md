### Title
Silent `u64` Overflow in `op_unknown` Cost Multiplier Produces Zero-Cost Unknown Ops - (File: `src/more_ops.rs`)

### Summary
In `op_unknown`, the final cost is computed with an unchecked `u64` multiplication (`cost *= cost_multiplier + 1`). In Rust release mode this wraps silently. A crafted CLVM program can make the wrapped result fall at or below `u32::MAX`, bypassing the post-multiplication guard and returning a cost of 0 (or a tiny value) for an unknown op that should be prohibitively expensive.

### Finding Description

`Cost` is a `u64` type alias. [1](#0-0) 

Inside `op_unknown`, after computing a base cost through one of four cost-function branches, the code multiplies by `cost_multiplier + 1` and then applies a post-multiplication guard: [2](#0-1) 

`cost_multiplier` is decoded from the opcode bytes as a `u64` whose value is bounded by `u32::MAX` (derived via `u32_from_u8`): [3](#0-2) 

So `cost_multiplier + 1` is at most `2^32`. The pre-multiplication `check_cost` at line 260 guarantees only that `cost ≤ max_cost`. In Chia's consensus the block cost limit is ~11 billion (`> 2^32 − 1 ≈ 4.3 × 10^9`), so `cost` can legally reach values where `cost × (cost_multiplier + 1)` exceeds `u64::MAX`.

The `cost_function = 2` branch (mul-like cost) is the easiest trigger. Unlike `op_multiply`, which hard-limits each operand to 256 bytes, `op_unknown`'s cost_function=2 branch imposes **no size limit** on atoms: [4](#0-3) 

Compare with `op_multiply`'s explicit guard: [5](#0-4) 

**Concrete overflow path:**

With two atoms each of length `L ≈ 740 000` bytes, the cost_function=2 branch produces:

```
cost ≈ MUL_BASE_COST + MUL_COST_PER_OP + 2L×6 + L²/128
     ≈ 92 + 885 + 8 880 000 + 4 278 125 000
     ≈ 2^32
```

With `cost_multiplier = 2^32 − 1` (maximum encodable value), `cost_multiplier + 1 = 2^32`, so:

```
cost × (cost_multiplier + 1) = 2^32 × 2^32 = 2^64 ≡ 0  (mod 2^64)
```

The guard `if cost > u32::MAX as u64` evaluates to `0 > 4 294 967 295` → **false**, so the function returns `Ok(Reduction(0, nil))` — a cost of **zero**.

### Impact Explanation

An attacker-controlled CLVM program can include an unknown op (reachable in Chia's consensus lenient mode) whose true cost should be enormous but whose reported cost is 0 after the wrap. The running cost accumulator in `run_program` adds this 0: [6](#0-5) 

The program therefore passes the `max_cost` budget check while consuming far more computational work than the declared cost. This is a **cost-metering bypass** with direct consensus impact: a block containing such a program would be accepted by nodes running in release mode but would be correctly rejected (or produce a different cost) by any implementation that uses checked arithmetic, causing a **consensus divergence**.

### Likelihood Explanation

- `op_unknown` is reachable via any CLVM program that uses an opcode not in the current operator set, which is the normal forward-compatibility path in Chia consensus.
- The attacker controls the opcode bytes (setting `cost_function = 2` and `cost_multiplier = 2^32 − 1`) and the argument atoms (sized to drive the base cost to exactly `2^32`).
- No special privileges are required; any coin puzzle submitted to the mempool can trigger this.
- Chia's block cost limit (~11 billion) is well above the `2^32` threshold needed for the overflow.

### Recommendation

Replace the unchecked multiplication with a checked variant and treat overflow as an invalid op:

```rust
// Before (line 261):
cost *= cost_multiplier + 1;

// After:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
```

Additionally, consider adding per-atom size limits inside the `cost_function = 2` branch of `op_unknown` (mirroring the 256-byte cap in `op_multiply`) to prevent the base cost from reaching values that make overflow feasible.

### Proof of Concept

```
; Opcode bytes: last byte = 0b10_000000 (cost_function=2),
;               preceding 4 bytes = 0xFF 0xFF 0xFF 0xFF (cost_multiplier = 2^32-1)
; Two atom arguments each ~740 000 bytes long.
;
; Base cost (cost_function=2, two equal-length atoms L=740000):
;   cost ≈ 92 + 885 + 2*740000*6 + (740000^2)/128 ≈ 2^32
;
; After check_cost passes (max_cost = 11e9 > 2^32):
;   cost *= (2^32-1)+1 = 2^32
;   2^32 * 2^32 = 2^64 ≡ 0 (mod 2^64)
;
; Guard: 0 > u32::MAX → false → returns Ok(Reduction(0, nil))
; Actual cost charged to the program: 0
```

The attacker encodes this as a serialized CLVM program and submits it in a block. Nodes running in release mode accept it with cost 0; any node using checked arithmetic rejects it or computes a different cost, producing a consensus split.
