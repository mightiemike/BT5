### Title
Immediate-Effect Risk Weight Updates Enable Owner Front-Running of Liquidations and Withdrawals — (File: `core/contracts/BaseEngine.sol`)

### Summary
`BaseEngine.updateRisk` allows the owner to overwrite collateralization weights for any product in a single transaction with no time lock. Because every health-gated operation in the protocol reads these weights at execution time, a malicious owner can lower maintenance weights to trigger liquidations of previously healthy accounts, or lower initial weights to silently block pending user withdrawals, with no advance notice.

### Finding Description
`BaseEngine.updateRisk` writes a new `RiskStore` directly to `_risk().value[productId]` with immediate effect:

```solidity
// core/contracts/BaseEngine.sol lines 278-290
function updateRisk(uint32 productId, RiskHelper.RiskStore memory riskStore)
    external
    onlyOwner
{
    ...
    _risk().value[productId] = riskStore;
}
``` [1](#0-0) 

The stored fields — `longWeightInitial`, `shortWeightInitial`, `longWeightMaintenance`, `shortWeightMaintenance` — are consumed immediately by `_calculateProductHealth`, which multiplies each position's amount by the weight and the oracle price to produce a health contribution:

```solidity
health += amount.mul(weight).mul(risk.priceX18);
``` [2](#0-1) 

This health value gates every critical user action in the protocol:

- **Withdrawals** — `Clearinghouse.withdrawCollateral` line 419: `require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH)`
- **Liquidations** — `Clearinghouse.liquidateSubaccount` delegates to `ClearinghouseLiq` which checks maintenance health
- **Quote transfers** — `Clearinghouse.transferQuote` line 249: `require(_isAboveInitial(txn.sender), ERR_SUBACCT_HEALTH)`
- **NLP minting** — `Clearinghouse.mintNlp` line 479: `require(getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0, ERR_SUBACCT_HEALTH)` [3](#0-2) [4](#0-3) 

There is no pending/commit pattern, no `pendingRisk` mapping, and no minimum notice period. The same immediate-write pattern is present in `addOrUpdateProduct` in `SpotEngine.sol`, which also calls `_addOrUpdateProduct` → `_risk().value[productId] = riskStore` for existing products. [5](#0-4) 

### Impact Explanation
A malicious owner can:

1. **Trigger mass liquidations**: Lower `longWeightMaintenance` for a widely-held product. Accounts that were healthy under the old weights become undercollateralized instantly. A liquidator (or the owner acting as liquidator) can call `liquidateSubaccount` in the very next sequencer batch to seize collateral at a discount.

2. **Block withdrawals silently**: Lower `longWeightInitial` just before a user's `WithdrawCollateral` transaction is processed. The health check at `Clearinghouse.withdrawCollateral` line 419 fails. Because the sequencer's `_executeSlowModeTransaction` wraps execution in a `try/catch` that swallows reverts, the withdrawal is silently dropped with no refund path.

3. **Craft self-favorable conditions**: Raise weights before the owner's own deposits or trades, then lower them afterward, extracting value at users' expense.

The corrupted state delta is `_risk().value[productId]`, which determines the dollar value of health attributed to every position in that product across all subaccounts simultaneously. [6](#0-5) 

### Likelihood Explanation
The owner is a single privileged address (or multisig) that can call `updateRisk` in one transaction. The sequencer processes transactions in ordered batches via `submitTransactionsChecked`. The owner can submit an `updateRisk` call in the same block as a sequencer batch that includes user withdrawals or trades; the parameter change takes effect before the user's transaction is executed within that batch. No additional access beyond the owner key is required, and no on-chain signal is emitted that would give users time to react before the change is finalized. [7](#0-6) 

### Recommendation
Implement a two-step time lock for `updateRisk` (and `addOrUpdateProduct` for existing products):

1. **Propose**: Owner submits new risk parameters; they are stored as `pendingRisk[productId]` with a `pendingAt` timestamp. Emit an event so users can observe the pending change.
2. **Commit**: After a mandatory delay (e.g., 24–48 hours), a separate `applyRisk(productId)` function finalizes the update.

This mirrors the `pendingImpls` / `migrateAll` two-step pattern already used in `BaseProxyManager` for contract upgrades, and gives users advance notice to adjust positions before new collateralization thresholds take effect. [8](#0-7) [9](#0-8) 

### Proof of Concept
1. User A holds a long position in product X. With `longWeightMaintenance = 0.9e9`, their maintenance health is `+500 USD` (healthy).
2. Owner calls `updateRisk(X, {longWeightMaintenance: 0.5e9, ...})` — takes effect in the same block.
3. User A's maintenance health is now `-200 USD` (undercollateralized).
4. In the next sequencer batch, a liquidator calls `liquidateSubaccount` against User A.
5. User A's collateral is seized at a liquidation discount; the owner (or a coordinated liquidator) profits. User A had no opportunity to add margin or close their position.
