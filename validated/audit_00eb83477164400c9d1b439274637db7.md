### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the direct caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the end user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently opens the gate to every user on the internet, defeating the per-user allowlist entirely.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks the allowlist keyed on that `sender`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [4](#0-3) 

So `sender` seen by the extension is the **router address**, not the end user. The allowlist check becomes `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

This creates an irreconcilable dilemma for any pool admin:

| Admin action | Effect |
|---|---|
| Allowlist individual users, **not** the router | Allowlisted users **cannot** swap through the router (reverts) |
| Allowlist the router to enable router-mediated swaps | **Every** user on the internet can bypass the per-user allowlist |

There is no configuration that simultaneously allows specific users to swap through the router while blocking others.

---

### Impact Explanation

A pool protected by `SwapAllowlistExtension` is intended to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, institutional traders). If the pool admin allowlists the router — the natural step to let those users access the pool through the standard periphery — the allowlist becomes a no-op for all router-mediated swaps. Any unprivileged user can then:

1. Call `MetricOmmSimpleRouter.exactInputSingle` targeting the restricted pool.
2. The extension sees `sender = router`, which is allowlisted, and passes.
3. The unauthorized user executes a swap at oracle prices, draining LP funds.

This is a **direct loss of LP principal**: LPs deposited into a restricted pool expecting only specific counterparties; unauthorized swaps expose them to adverse selection and token outflows they did not consent to.

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary user-facing swap interface; most users will route through it.
- Any pool admin who deploys `SwapAllowlistExtension` and wants their allowlisted users to use the router **must** allowlist the router, triggering the bypass.
- The bypass requires no special privileges, no flash loans, and no exotic token behavior — a single `exactInputSingle` call suffices.
- The flaw is structural (wrong identity checked), not a misconfiguration edge case.

---

### Recommendation

The extension must identify the **end user**, not the intermediary. Two complementary fixes:

1. **Pass the original `msg.sender` through the router as a dedicated field.** The router already stores the payer in transient storage (`_getPayer()`). Encode the true initiator in `extensionData` and have the extension decode and verify it — but this requires the extension to trust the router, which reintroduces a trust assumption.

2. **Preferred — check `sender` only when `sender` is not a known router; otherwise require the true user to be encoded in `extensionData` and verified by the extension.** Alternatively, redesign the allowlist to gate on `recipient` (the economic beneficiary of the swap) rather than `sender` (the mechanical caller), since `recipient` is always the address that receives output tokens and is harder to spoof.

3. **Simplest safe fix:** Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the factory level by reverting pool creation if both the allowlist extension and a known router are configured together.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension.
  - Pool admin calls setAllowedToSwap(pool, router, true)
    (to let allowlisted users access the pool via the router).
  - Pool admin calls setAllowedToSwap(pool, alice, true).
  - Bob (not allowlisted) wants to swap.

Attack:
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool: restrictedPool,
       recipient: bob,
       zeroForOne: true,
       amountIn: X,
       ...
     });
  2. Router calls pool.swap(bob, true, X, ...).
     → msg.sender of pool.swap() = router.
  3. Pool calls _beforeSwap(sender=router, ...).
  4. SwapAllowlistExtension checks allowedSwapper[pool][router] == true → PASSES.
  5. Bob's swap executes. Bob receives token1 from the restricted pool.

Result:
  Bob, who is not on the allowlist, successfully swaps in a pool
  that was configured to restrict access to specific addresses.
  LP funds flow to an unauthorized counterparty.
