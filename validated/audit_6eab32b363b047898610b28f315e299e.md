### Title
Duplicate Position IDs in `MsgClaimTierRewards` Enable Double-Claim of Bonus Rewards — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`ClaimTierRewards` loads all requested positions into an in-memory slice before processing any of them. Because there is no deduplication of `PositionIds`, a caller who supplies the same position ID twice causes the same stale in-memory copy to be processed twice. The second pass reuses the old `LastBonusAccrual` checkpoint to compute and pay out bonus rewards for a time window that was already paid in the first pass, and then overwrites the store with the stale state, resetting the checkpoint for future exploitation.

---

### Finding Description

In `ClaimTierRewards` (`msg_server.go` lines 429–468), positions are loaded into a slice before any reward processing begins:

```go
for _, posId := range msg.PositionIds {
    pos, err := ms.getPositionState(ctx, posId)
    ...
    positions = append(positions, pos)
}
totalBase, totalBonus, err := ms.claimRewardsAndUpdatesPositions(ctx, positions)
``` [1](#0-0) 

There is no deduplication of `posId` values. If `msg.PositionIds = [X, X]`, both `positions[0]` and `positions[1]` are loaded with the same initial state, including the same `LastBonusAccrual` timestamp and `LastEventSeq`.

`claimRewardsAndUpdatesPositions` then iterates over the slice:

```go
for i := range positions {
    pos := &positions[i]
    base, err := k.claimBaseRewards(ctx, *pos)
    bonus, err := k.processEventsAndClaimBonus(ctx, pos)
    if err := k.setPosition(ctx, pos.Position, nil); err != nil { ... }
}
``` [2](#0-1) 

Inside `processEventsAndClaimBonus`, the current-segment bonus is computed using `pos.LastBonusAccrual` as the segment start, then `applyBonusAccrualCheckpoint` advances it to `blockTime`:

```go
bonus := k.computeSegmentBonus(*pos, tier, segmentStart, blockTime, currentRate)
...
applyBonusAccrualCheckpoint(&pos.Position, blockTime)
``` [3](#0-2) 

**First pass** (index 0): computes bonus for `[old_LastBonusAccrual, blockTime]`, pays it, advances `LastBonusAccrual = blockTime`, saves updated position to store.

**Second pass** (index 1): the in-memory copy still has `LastBonusAccrual = old_LastBonusAccrual`. `processEventsAndClaimBonus` computes the same segment `[old_LastBonusAccrual, blockTime]` again, pays the same bonus a second time, then calls `setPosition` with the stale copy — **overwriting the store with the old checkpoint**, resetting `LastBonusAccrual` back to `old_LastBonusAccrual`.

The base-reward double-claim is neutralised by the distribution module (rewards are zeroed after the first `WithdrawDelegationRewards` call), but the bonus-reward double-claim succeeds because it is computed purely from the in-memory `LastBonusAccrual` field, not from a distribution-module counter. [4](#0-3) 

---

### Impact Explanation

- **Corrupted value**: `RewardsPoolName` module account balance is drained by 2× the legitimate bonus for the duplicated position(s) per transaction.
- **Corrupted state**: `LastBonusAccrual` in the persisted position is reset to the pre-claim value after each exploit transaction, so the same window is claimable again in the next block.
- **Scope**: Any position owner can drain the bonus pool proportional to their accrued bonus, limited only by the pool balance and the `sufficientBonusPoolBalance` guard. [5](#0-4) 

---

### Likelihood Explanation

The entry path is a standard, unprivileged `MsgClaimTierRewards` transaction signed by any position owner. No special role, leaked key, or social engineering is required. The only prerequisite is holding at least one delegated tier position. The `msg.Validate()` call and the per-position `validateClaimRewards` ownership check do not deduplicate `PositionIds`. [6](#0-5) 

---

### Recommendation

Add a deduplication check for `PositionIds` before the load loop in `ClaimTierRewards`, or alternatively enforce uniqueness inside `msg.Validate()`. A simple seen-set suffices:

```go
seen := make(map[uint64]struct{}, len(msg.PositionIds))
for _, posId := range msg.PositionIds {
    if _, dup := seen[posId]; dup {
        return nil, errorsmod.Wrapf(sdkerrors.ErrInvalidRequest, "duplicate position id: %d", posId)
    }
    seen[posId] = struct{}{}
    ...
}
```

This mirrors the fix applied in the referenced report (adding a guard that prevents the same logical unit from being processed twice within one execution).

---

### Proof of Concept

1. Alice holds tier position with ID `42`, currently bonded, with `LastBonusAccrual = T0` and accrued bonus `B` for the period `[T0, now]`.
2. Alice submits `MsgClaimTierRewards{Owner: alice, PositionIds: [42, 42]}`.
3. `ClaimTierRewards` loads two identical copies of position 42 into `positions[0]` and `positions[1]`, both with `LastBonusAccrual = T0`.
4. First iteration: `processEventsAndClaimBonus` pays `B` from `RewardsPoolName` to Alice, sets `positions[0].LastBonusAccrual = now`, writes updated position to store.
5. Second iteration: `positions[1]` still has `LastBonusAccrual = T0`; `processEventsAndClaimBonus` computes the same segment `[T0, now]`, pays another `B` to Alice, then `setPosition` writes the stale copy back to the store, resetting `LastBonusAccrual = T0`.
6. Net result: Alice received `2B` bonus instead of `B`, and the store shows `LastBonusAccrual = T0`, so the attack is repeatable next block. [7](#0-6) [8](#0-7)

func (k Keeper) claimRewardsAndUpdatesPositions(ctx context.Context, positions []types.PositionState) (sdk.Coins, sdk.Coins, error) {
	totalBase := sdk.NewCoins()
	totalBonus := sdk.NewCoins()

	for i := range positions {
		pos := &positions[i]

		if !pos.IsDelegated() {
			continue
		}

		base, err := k.claimBaseRewards(ctx, *pos)
		if err != nil {
			return nil, nil, err
		}
		totalBase = totalBase.Add(base...)

		bonus, err := k.processEventsAndClaimBonus(ctx, pos)
		if err != nil {
			return nil, nil, err
		}
		totalBonus = totalBonus.Add(bonus...)

		if err := k.setPosition(ctx, pos.Position, nil); err != nil {
			return nil, nil, err
		}
	}

	return totalBase, totalBonus, nil
```
