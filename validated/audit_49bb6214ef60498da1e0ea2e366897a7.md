### Title
Wrong Risk Asset Used in Spread Liquidation Price Penalty for Long Spread Positions — (`core/contracts/ClearinghouseStorage.sol`)

---

### Summary

`getSpreadLiqPriceX18` computes the spread liquidation price penalty using `perpRisk` (with a positive `amount`) when `amount >= 0`, but for a long spread the perp leg is **short** — the opposite sign. The `amount < 0` branch correctly uses `spotRisk` with the negative amount (short spot weight), but the `amount >= 0` branch uses `perpRisk` with the positive amount (long perp weight) instead of `spotRisk` with the positive amount (long spot weight). This mirrors the Panoptic bug class exactly: a multi-value financial calculation silently picks the wrong component, corrupting the liquidation price used in `_handleLiquidationPayment`.

---

### Finding Description

`getSpreadLiqPriceX18` in `ClearinghouseStorage.sol` computes a penalty for the spread liquidation price:

```solidity
// ClearinghouseStorage.sol lines 82–99
if (amount >= 0) {
    penaltyX18 =
        (ONE -
            RiskHelper._getWeightX18(
                perpRisk,          // ← uses PERP risk
                amount,            // ← positive amount → longWeightMaintenanceX18
                IProductEngine.HealthType.MAINTENANCE
            )) /
        10;
} else {
    penaltyX18 =
        (RiskHelper._getWeightX18(
            spotRisk,              // ← uses SPOT risk
            amount,                // ← negative amount → shortWeightMaintenanceX18
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) /
        10;
}
```

For `amount >= 0` (long spread = long spot + short perp):
- The perp leg is **short**, so the correct weight is `perpRisk.shortWeightMaintenanceX18`.
- But `_getWeightX18(perpRisk, amount, MAINTENANCE)` with `amount >= 0` returns `perpRisk.longWeightMaintenanceX18` — the wrong weight.
- The symmetric correct fix is to use `spotRisk` with `amount >= 0` (long spot weight), matching how the `amount < 0` branch uses `spotRisk` with `amount < 0` (short spot weight).

The `amount < 0` branch is correct: `_getWeightX18(spotRisk, amount, MAINTENANCE)` with `amount < 0` returns `spotRisk.shortWeightMaintenanceX18`, which is the right weight for the short spot leg of a short spread.

The resulting liquidation price is then used directly in `_handleLiquidationPayment`:

```solidity
// ClearinghouseLiq.sol lines 460–470
(
    v.liquidationPriceX18,
    v.oraclePriceX18,
    v.oraclePriceX18Perp
) = getSpreadLiqPriceX18(spotId, perpId, txn.amount);

v.liquidationPayment = v.liquidationPriceX18.mul(txn.amount);

v.liquidationFees = (v.oraclePriceX18 - v.liquidationPriceX18)
    .mul(LIQUIDATION_FEE_FRACTION)
    .mul(txn.amount);
```

---

### Impact Explanation

When `txn.amount > 0` (long spread liquidation), `v.liquidationPriceX18` is computed with `perpRisk.longWeightMaintenanceX18` instead of `spotRisk.longWeightMaintenanceX18`. These weights can differ materially across products. Concretely:

- **Liquidation payment** (`liquidationPriceX18 * txn.amount`) paid to the liquidatee is wrong — either too high (overpaying the liquidatee, draining the liquidator) or too low (underpaying the liquidatee, constituting a partial theft of collateral).
- **Liquidation fees** (`(oraclePriceX18 - liquidationPriceX18) * fee * amount`) are miscalculated — if the wrong weight makes the penalty larger, fees are inflated and the liquidator is overcharged, discouraging liquidations of long spread positions and allowing insolvent accounts to persist.
- **Solvency invariant broken**: if the inflated fee makes long-spread liquidations unprofitable, liquidators will avoid them, leaving undercollateralized spread positions open and accumulating bad debt.

---

### Likelihood Explanation

Any liquidation of a long spread position (`txn.isEncodedSpread == true` and `txn.amount > 0`) triggers the bug. This is a standard liquidation path reachable by any unprivileged liquidator calling `liquidateSubaccountImpl`. No special privileges or unusual conditions are required. The bug is deterministic and fires on every such call.

---

### Recommendation

Replace `perpRisk` with `spotRisk` in the `amount >= 0` branch to match the symmetric design of the `amount < 0` branch:

```solidity
// ClearinghouseStorage.sol
if (amount >= 0) {
    penaltyX18 =
        (ONE -
            RiskHelper._getWeightX18(
                spotRisk,   // ← fix: use spot risk for long spot leg
                amount,
                IProductEngine.HealthType.MAINTENANCE
            )) /
        10;
} else {
    penaltyX18 =
        (RiskHelper._getWeightX18(
            spotRisk,
            amount,
            IProductEngine.HealthType.MAINTENANCE
        ) - ONE) /
        10;
}
```

Add a regression test asserting that for a long spread liquidation, the penalty uses `spotRisk.longWeightMaintenanceX18` and not `perpRisk.longWeightMaintenanceX18`.

---

### Proof of Concept

1. Deploy two products: spot (id=1, `longWeightMaintenanceX18 = 0.95e18`) and perp (id=2, `longWeightMaintenanceX18 = 0.80e18`).
2. Create a subaccount with a long spread (long spot, short perp) that is undercollateralized at maintenance.
3. Call `liquidateSubaccountImpl` with `isEncodedSpread = true`, `amount = +N`.
4. Observe `v.liquidationPriceX18 = spotPrice * (1 - (1 - 0.80)/10) = spotPrice * 0.98` (uses perp weight 0.80).
5. Expected: `v.liquidationPriceX18 = spotPrice * (1 - (1 - 0.95)/10) = spotPrice * 0.995` (uses spot weight 0.95).
6. The 1.5% price difference on a large position results in a materially wrong liquidation payment and inflated fees, demonstrable as a concrete asset delta. [1](#0-0) [2](#0-1) [3](#0-2)

```text
    function getSpreadLiqPriceX18(
        uint32 spotId,
        uint32 perpId,
        int128 amount
    )
        internal
        returns (
            int128,
            int128,
            int128
        )
    {
        RiskHelper.Risk memory spotRisk = IProductEngine(
            productToEngine[spotId]
        ).getRisk(spotId);
        RiskHelper.Risk memory perpRisk = IProductEngine(
            productToEngine[perpId]
        ).getRisk(perpId);

        int128 penaltyX18;
        if (amount >= 0) {
            penaltyX18 =
                (ONE -
                    RiskHelper._getWeightX18(
                        perpRisk,
                        amount,
                        IProductEngine.HealthType.MAINTENANCE
                    )) /
                10;
        } else {
            penaltyX18 =
                (RiskHelper._getWeightX18(
                    spotRisk,
                    amount,
                    IProductEngine.HealthType.MAINTENANCE
                ) - ONE) /
                10;
        }
        if (penaltyX18.abs() < MIN_SPREAD_LIQ_PENALTY_X18) {
            if (penaltyX18 < 0) {
                penaltyX18 = -MIN_SPREAD_LIQ_PENALTY_X18;
            } else {
                penaltyX18 = MIN_SPREAD_LIQ_PENALTY_X18;
            }
        }

        if (amount > 0) {
            return (
                spotRisk.priceX18.mul(ONE - penaltyX18),
                spotRisk.priceX18,
                perpRisk.priceX18
            );
        } else {
            return (
                spotRisk.priceX18.mul(ONE + penaltyX18),
                spotRisk.priceX18,
                perpRisk.priceX18
            );
        }
    }
```
