### Title
Missing `_disableInitializers()` in Core Upgradeable Implementation Contracts — (File: `core/contracts/Endpoint.sol`, `core/contracts/Clearinghouse.sol`, `core/contracts/SpotEngine.sol`, `core/contracts/PerpEngine.sol`, `core/contracts/OffchainExchange.sol`)

---

### Summary
Multiple core Nado protocol contracts deployed as upgradeable proxies are missing `_disableInitializers()` in their constructors. Any unprivileged caller can directly call `initialize()` on the unprotected implementation contracts, taking ownership and configuring critical protocol addresses to attacker-controlled values.

---

### Finding Description

The following production contracts inherit from OpenZeppelin's `Initializable` (via `OwnableUpgradeable`, `EIP712Upgradeable`, or `EndpointGated`) and expose an `initialize()` function, but define **no constructor** and therefore never call `_disableInitializers()`:

| Contract | `initialize()` present | Constructor with `_disableInitializers()` |
|---|---|---|
| `Endpoint.sol` | Yes (`initializer`) | **No** |
| `Clearinghouse.sol` | Yes (`initializer`) | **No** |
| `SpotEngine.sol` | Yes (calls `_initialize`) | **No** |
| `PerpEngine.sol` | Yes (calls `_initialize`) | **No** |
| `OffchainExchange.sol` | Yes (`initializer`) | **No** |

This is in direct contrast to the contracts in the same codebase that **correctly** protect their implementations:

- `BaseProxyManager.sol` — has `_disableInitializers()` [1](#0-0) 
- `ContractOwner.sol` — has `_disableInitializers()` [2](#0-1) 
- `Verifier.sol` — has `_disableInitializers()` [3](#0-2) 
- `Airdrop.sol` — has `_disableInitializers()` [4](#0-3) 
- `BaseWithdrawPool.sol` — has `_disableInitializers()` [5](#0-4) 

The unprotected `Endpoint.sol` `initialize()` sets `sequencer`, `clearinghouse`, `offchainExchange`, `verifier`, and `endpointTx` — all critical protocol addresses — with no guard against direct invocation on the implementation: [6](#0-5) 

The unprotected `Clearinghouse.sol` `initialize()` sets `clearinghouseLiq` (the delegatecall target for liquidations) and `withdrawPool` (the token custodian): [7](#0-6) 

The unprotected `OffchainExchange.sol` `initialize()` sets `clearinghouse`, `spotEngine`, and `perpEngine`: [8](#0-7) 

`SpotEngine` and `PerpEngine` both delegate to `BaseEngine._initialize()`, which calls `__Ownable_init()` and sets `canApplyDeltas` for the endpoint, clearinghouse, and offchain exchange — none of which are locked on the implementation: [9](#0-8) 

---

### Impact Explanation

An attacker who calls `initialize()` directly on any of these implementation contracts:

1. **Becomes the owner** of the implementation contract (`__Ownable_init()` runs with `msg.sender` as owner).
2. **Sets all critical addresses** (sequencer, clearinghouse, verifier, withdrawPool, clearinghouseLiq, etc.) to attacker-controlled values in the implementation's own storage.
3. **Can call all `onlyOwner` functions** on the implementation: `addOrUpdateProduct`, `updateRisk`, `setSpreads`, `setWithdrawPool`, `addEngine`, etc.
4. For `Endpoint` implementation: can set themselves as `sequencer` and call `submitTransactionsChecked` on the implementation, corrupting the implementation's `nSubmissions`, `priceX18`, and `linkedSigners` state.

While the proxy's storage is separate from the implementation's storage (so the live proxy state is not directly overwritten), the **ownership assumption of the implementation contract is concretely corrupted**. This is the exact class of vulnerability described in the external report. Additionally, this creates a foothold for further exploitation if any upgrade path, delegatecall chain, or cross-contract interaction reads from or trusts the implementation's own storage or owner.

---

### Likelihood Explanation

**High.** The `initialize()` functions on all affected contracts are `external` with no access control beyond the `initializer` modifier. Since the implementation contracts are never self-initialized at deployment (no constructor calls `_disableInitializers()`), any unprivileged caller can invoke `initialize()` on the implementation at any time after deployment. No special permissions, leaked keys, or governance capture are required.

---

### Recommendation

Add a constructor with `_disableInitializers()` to each affected contract, following the pattern already established in `BaseProxyManager`, `ContractOwner`, `Verifier`, `Airdrop`, and `BaseWithdrawPool`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

This should be added to: `Endpoint.sol`, `Clearinghouse.sol`, `SpotEngine.sol` (or `SpotEngineState`/`BaseEngine`), `PerpEngine.sol` (or `PerpEngineState`), and `OffchainExchange.sol`. Also consider `EndpointTx.sol` and `ClearinghouseLiq.sol`, which are delegatecall targets and similarly lack this protection.

---

### Proof of Concept

1. Deploy the Nado protocol normally (proxy + implementation).
2. Obtain the address of the `Endpoint` **implementation** contract (not the proxy).
3. Call directly on the implementation:
   ```solidity
   Endpoint(implementationAddress).initialize(
       attacker,   // _sanctions
       attacker,   // _sequencer
       attacker,   // _offchainExchange
       IClearinghouse(attacker), // _clearinghouse
       attacker,   // _verifier
       attacker    // _endpointTx
   );
   ```
4. The attacker is now `owner()` of the `Endpoint` implementation, `sequencer` is set to the attacker, and all critical addresses point to attacker-controlled contracts.
5. The attacker can call `submitTransactionsChecked()` on the implementation (as the self-designated sequencer), corrupting the implementation's `nSubmissions`, `priceX18`, and `linkedSigners` mappings.
6. Repeat for `Clearinghouse`, `SpotEngine`, `PerpEngine`, and `OffchainExchange` implementations.
