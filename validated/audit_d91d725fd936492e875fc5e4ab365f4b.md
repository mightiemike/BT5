Audit Report

## Title
SwapAllowlistExtension checks router address instead of actual user, allowing any user to bypass per-user swap allowlist via router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `pool.swap()`. When a swap is routed through `MetricOmmSimpleRouter`, `msg.sender` to the pool is always the router contract address. The extension therefore checks whether the **router** is allowlisted, not whether the **actual end user** is allowlisted. Any user can bypass a per-user allowlist by routing through the router if the router address is in the allowlist.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap` at [1](#0-0) , which forwards it verbatim to every configured extension via `ExtensionCalling._beforeSwap` at [2](#0-1) .

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — at [3](#0-2) .

`exactInputSingle` calls `IMetricOmmPoolActions(params.pool).swap(...)` as a regular external `CALL` at [4](#0-3) , so the pool always sees `msg.sender = router`, regardless of who called the router. The actual end user's address is never checked. The `multicall` delegatecall at [5](#0-4)  is irrelevant — the bypass is identical with a single direct `router.exactInputSingle(restrictedPool)` call.

A pool admin who allowlists the router to permit router-mediated swaps for legitimate users inadvertently opens the pool to all users. There is no configuration that simultaneously allows router-mediated swaps for allowlisted users while blocking non-allowlisted users.

## Impact Explanation
The `SwapAllowlistExtension`'s per-user access control is completely ineffective for router-mediated swaps. Any non-allowlisted user can swap on a restricted pool by calling `router.exactInputSingle(restrictedPool)`, provided the router is allowlisted. This breaks the pool curation invariant the extension is designed to enforce, allowing unauthorized participants to trade on pools intended to be private or restricted. This constitutes broken core pool functionality — the access control gate is rendered non-functional for the standard swap path.

## Likelihood Explanation
Any pool that (a) deploys `SwapAllowlistExtension` for per-user access control and (b) allowlists the router to support normal UX is fully exposed. This is the expected operational configuration — a pool admin who wants to restrict swaps to specific users while still supporting the standard router would naturally allowlist the router. The bypass requires no special privileges, no flash loans, and no unusual token behavior. It is reachable by any unprivileged caller.

## Recommendation
The extension must check the actual end user, not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the real user through the router:** Have `exactInputSingle` (and all router swap functions) encode `msg.sender` into `extensionData` and have the extension decode and verify it. The extension would then check the decoded user address rather than the `sender` argument.
2. **Check both router and user:** Require the pool admin to allowlist individual users (not the router), and have the router forward the real user address in a verifiable way (e.g., signed or via a trusted forwarder pattern).

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension, only router is allowlisted
extension.setAllowedToSwap(restrictedPool, address(router), true);
// attacker is NOT allowlisted:
assertFalse(extension.isAllowedToSwap(restrictedPool, attacker));

// Bypass: attacker routes through the router
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool: restrictedPool,
    ...
}));
// Swap succeeds — extension saw sender=router (allowlisted), not sender=attacker
```
