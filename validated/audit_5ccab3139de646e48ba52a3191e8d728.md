### Title
Post-Trade Solvency Check Permanently Bypassed via No-Op `isHealthy()` — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

The `isHealthy()` function in `OffchainExchange.sol` unconditionally returns `true`, making the post-trade health assertions at lines 826–827 of `matchOrders()` completely inert. Any order match that leaves a subaccount insolvent succeeds on-chain without reversion. This is the direct Nado analog to the UMA insolvent-initial-position class: a position that should be rejected by a solvency gate passes through because the gate is a stub.

---

### Finding Description

After computing balance deltas for both taker and maker in `matchOrders()`, the protocol calls:

```solidity
require(isHealthy(taker.order.sender), ERR_INVALID_TAKER);
require(isHealthy(maker.order.sender), ERR_INVALID_MAKER);
``` [1](#0-0) 

These two `require` statements are the sole on-chain enforcement that a trade does not leave either party insolvent. However, the function they call is:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [2](#0-1) 

The function ignores its argument entirely and always returns `true`. No override of `isHealthy` exists in the production `OffchainExchange` contract. The `virtual` modifier signals the function was designed to be overridden, but the override is absent, leaving a permanently open gate.

The balance updates that precede the check are real and committed:

```solidity
_updateBalances(callState, market.quoteId, taker.order.sender,
    ordersInfo.taker.amountDelta, ordersInfo.taker.quoteDelta);
_updateBalances(callState, market.quoteId, maker.order.sender,
    ordersInfo.maker.amountDelta, ordersInfo.maker.quoteDelta);
``` [3](#0-2) 

So balances are mutated, but the health gate that should reject the mutation if it produces insolvency is a no-op.

---

### Impact Explanation

An insolvent subaccount created through `matchOrders` has the following concrete consequences, mirroring the UMA report exactly:

1. **Liquidation is non-economic.** `liquidateSubaccountImpl` in `ClearinghouseLiq.sol` requires `isUnderMaintenance(txn.liquidatee)` to be true. A deeply insolvent subaccount with no assets to seize offers no profit to a liquidator, so it will not be liquidated voluntarily. [4](#0-3) 

2. **Insurance fund is drained on finalization.** `_finalizeSubaccount` covers the residual negative quote balance from `insurance`:

   ```solidity
   int128 insuranceCover = MathHelper.min(v.insurance, -quoteBalance.amount);
   if (insuranceCover > 0) {
       v.insurance -= insuranceCover;
       spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.liquidatee, insuranceCover);
   }
   ``` [5](#0-4) 

3. **Losses are socialized to all other users.** If insurance is insufficient, `spotEngine.socializeSubaccount(txn.liquidatee)` is called, spreading the deficit across all other subaccounts' balances — directly harming innocent depositors. [6](#0-5) 

The corrupted state delta is: `spotEngine` or `perpEngine` balance for the insolvent subaccount goes negative beyond what the insurance fund can cover, with the shortfall borne by all other users.

---

### Likelihood Explanation

The sequencer submits `matchOrders` transactions. Because the on-chain health gate is a no-op, the sequencer is the only enforcement layer. A sequencer that is compromised, misconfigured, or acting adversarially can submit any order pair — including one that deliberately leaves a party insolvent — and the contract will accept it. Users have no on-chain guarantee that their counterparty in any matched trade is solvent after the fill. The broken invariant is permanent until the contract is upgraded.

---

### Recommendation

Replace the stub with a real health check that calls `clearinghouse.getHealth()`:

```solidity
function isHealthy(bytes32 subaccount) internal virtual returns (bool) {
    return clearinghouse.getHealth(
        subaccount,
        IProductEngine.HealthType.INITIAL
    ) >= 0;
}
```

This mirrors the pattern already used in `Clearinghouse.withdrawCollateral` and `Clearinghouse.transferQuote`: [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Sequencer submits a `MatchOrders` transaction pairing taker `A` (long perp, zero collateral) against maker `B`.
2. `matchOrders` executes: `_updateBalances` credits `A` with a large long perp position and debits quote, leaving `A`'s health deeply negative.
3. `require(isHealthy(A.sender), ERR_INVALID_TAKER)` is reached — `isHealthy` returns `true` unconditionally.
4. Transaction succeeds. `A` now holds an insolvent perp position on-chain.
5. No liquidator will touch `A` (no profit). When `A` is eventually finalized via `liquidateSubaccountImpl` with `productId == type(uint32).max`, `_finalizeSubaccount` drains `insurance` to cover `A`'s deficit, then calls `socializeSubaccount`, spreading remaining losses to all depositors.
