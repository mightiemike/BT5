### Title
`AllResources`→`L1Gas` silent downgrade in protobuf conversion produces a divergent `get_tip_resource_bounds_hash`, breaking transaction hash canonicalization — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` is variant-gated: it hashes `[tip, L1_GAS, L2_GAS, L1_DATA_GAS]` for `AllResources` but only `[tip, L1_GAS, L2_GAS]` for `L1Gas`. The protobuf deserializer at `crates/apollo_protobuf/src/converters/transaction.rs:431` silently converts `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` → `L1Gas(X)` whenever both zero-valued fields are absent or zero. The same logical transaction therefore produces two distinct transaction hashes depending on which code path processes it. A transaction signed and hashed under `AllResources` arrives at a peer or RPC consumer with a different hash after protobuf round-trip, causing signature verification to bind the wrong preimage.

### Finding Description

**Step 1 — Hash is variant-gated.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` conditionally includes `L1_DATA_GAS` based on the enum variant:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // L1_DATA_GAS omitted
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // included
    }
});
```

For `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` the hash input is `[tip, packed(L1_GAS,X), packed(L2_GAS,0), packed(L1_DATA_GAS,0)]`. For `L1Gas(X)` it is `[tip, packed(L1_GAS,X), packed(L2_GAS,0)]`. These are different Poseidon inputs and produce different field elements.

**Step 2 — Protobuf deserializer silently downgrades the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs`:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // absent → zero
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)                      // downgrade
} else {
    ValidResourceBounds::AllResources(...)
})
```

A transaction originally carrying `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` — a valid, user-submittable configuration — is silently converted to `L1Gas(X)` after any protobuf round-trip. The same downgrade logic is duplicated in the RPC v0.8 layer at `crates/apollo_rpc/src/v0_8/transaction.rs:188-199`.

**Step 3 — Hash is recomputed after the downgrade.**

In the block-propagation path, `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(value.resource_bounds...)`, producing the downgraded `L1Gas` variant. Any subsequent call to `calculate_transaction_hash` on the deserialized `InvokeTransactionV3` — for block-hash construction, transaction-hash verification, or RPC simulation — computes hash H2 ≠ H1 (the hash the submitter and the originating gateway computed). The `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash` path used by `convert_rpc_tx_to_internal` at `crates/apollo_transaction_converter/src/transaction_converter.rs:391` is the canonical hash-assignment point; any peer that re-derives the hash from a protobuf-deserialized `InvokeTransactionV3` diverges from it.

**Step 4 — Signature preimage mismatch.**

The account contract's `__validate__` entry point receives the transaction hash as part of the execution context. If the hash delivered to the blockifier is H1 (original) but a verifying component recomputes H2 (post-downgrade), the signature — which was computed over H1 — fails verification against H2, or vice-versa. The same divergence affects the `proof_facts` conditional branch in `get_invoke_transaction_v3_hash` (lines 399–403), which appends a `proof_facts_hash` only when non-empty; a transaction carrying proof facts that is deserialized with a cleared `proof_facts` field would also lose that hash component.

### Impact Explanation

A transaction submitted with `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` — a structurally valid V3 invoke — is assigned hash H1 at the originating gateway. After protobuf serialization and deserialization (P2P block propagation, consensus message passing, or RPC v0.8 retrieval), the resource bounds are silently converted to `L1Gas(X)` and any hash recomputation yields H2 ≠ H1. This causes:

- **Wrong transaction hash in block hash computation**: the block hash commits to H2 while the receipt and storage record H1, producing an inconsistent chain state.
- **Signature verification failure**: the account's `__validate__` receives H2 but the ECDSA signature covers H1, causing a valid transaction to be rejected or an invalid one to be accepted depending on which hash reaches the VM.
- **Wrong RPC simulation result**: `starknet_estimateFee` / `starknet_simulateTransactions` recompute the hash from the deserialized fields; the simulation runs with H2 while the user's wallet signed H1, returning an authoritative-looking wrong result.

### Likelihood Explanation

The trigger condition — `AllResources` with `l2_gas=0` and `l1_data_gas=0` — is reachable by any user submitting a V3 transaction that sets only L1 gas bounds (a common pattern for wallets that have not yet adopted full three-resource bounds). The protobuf round-trip occurs on every P2P block propagation and every consensus message exchange, making the downgrade automatic and unprivileged. No special permissions or malformed bytes are required.

### Recommendation

1. **Remove the variant-collapsing logic** from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Always deserialize into `AllResources` when all three resource fields are present (even if zero), preserving the original variant and hash preimage.
2. **Make `get_tip_resource_bounds_hash` variant-independent** for the zero case: always include `L1_DATA_GAS` in the hash input when the transaction version is ≥ 3, regardless of the `ValidResourceBounds` variant. This eliminates the state-dependent hash switch that is the direct analog of the `_scaleFactor`/`assetShareRatio` switch in the original report.
3. **Add a round-trip hash test**: serialize an `AllResources{l1_gas:X, l2_gas:0, l1_data_gas:0}` transaction to protobuf, deserialize it, recompute the hash, and assert it equals the pre-serialization hash.

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;
use starknet_api::block::GasAmount;
use starknet_api::block::GasPrice;

fn main() {
    let l1_gas = ResourceBounds {
        max_amount: GasAmount(1000),
        max_price_per_unit: GasPrice(1),
    };
    let tip = Tip(0);

    // Hash computed by the submitter / originating gateway (AllResources variant)
    let all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas,
        l2_gas: ResourceBounds::default(),   // zero
        l1_data_gas: ResourceBounds::default(), // zero
    });
    let h1 = get_tip_resource_bounds_hash(&all_resources, &tip).unwrap();

    // Hash computed after protobuf round-trip (L1Gas variant — silent downgrade)
    let l1_gas_only = ValidResourceBounds::L1Gas(l1_gas);
    let h2 = get_tip_resource_bounds_hash(&l1_gas_only, &tip).unwrap();

    // h1 != h2 — same logical transaction, two different hash preimages
    assert_ne!(h1, h2, "hash canonicalization broken: AllResources and L1Gas produce different hashes");
}
```

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
