### Title
Protobuf `ValidResourceBounds` Deserialization Silently Converts `AllResources` to `L1Gas` When L2/L1-Data Gas Bounds Are Zero, Producing a Wrong Transaction Hash and Rejecting Valid Transactions in the Consensus Path - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a zero-check heuristic to decide between the `L1Gas` and `AllResources` enum variants. When a V3 transaction carries `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`, the round-trip through protobuf silently produces `ValidResourceBounds::L1Gas(X)` instead of `ValidResourceBounds::AllResources(...)`. This is the direct sequencer analog of the "missing minimum-return check" pattern: just as `amountMin = 0` removes slippage protection in the DeFi report, zero L2/L1-data gas bounds remove the canonicalization invariant that the resource-bounds variant must be preserved across serialization boundaries. The consequence is a divergent transaction hash and an outright rejection of the transaction by every peer that receives it through the consensus P2P path.

---

### Finding Description

**Root cause — protobuf deserialization heuristic (`transaction.rs` lines 431–435):**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

The condition collapses any `AllResources` value whose L2 and L1-data gas sub-fields are zero into the `L1Gas` variant. The codebase itself documents this hazard in the consensus round-trip test:

> "If all the fields of `AllResources` are 0 upon serialization, then the deserialized value will be interpreted as the `L1Gas` variant." [2](#0-1) 

The workaround applied in tests is to forcibly set `l2_gas.max_amount = GasAmount(1)` before serialization. No such guard exists in production code paths.

**Hash divergence — `get_tip_resource_bounds_hash` (`transaction_hash.rs` lines 188–210):**

For `ValidResourceBounds::L1Gas`, the hash chain contains exactly two resource felts (L1 gas + L2 gas zero). For `ValidResourceBounds::AllResources`, it contains three (L1 gas + L2 gas zero + L1-data gas zero). These produce **different Poseidon digests** even when the numeric values of all three resource bounds are identical. [3](#0-2) 

**Gateway accepts the transaction with `AllResources`:**

`InternalRpcDeclareTransactionV3` stores `resource_bounds: AllResourceBounds` and its `DeclareTransactionV3Trait` implementation always wraps it as `ValidResourceBounds::AllResources(self.resource_bounds)`. The hash stored in `InternalRpcTransaction.tx_hash` is therefore computed with the three-element chain. [4](#0-3) 

**Consensus peer rejects the transaction with `L1Gas`:**

When the proposer serialises the `ConsensusTransaction` to protobuf and a peer deserialises it, the heuristic fires and produces `L1Gas`. The declare-specific converter then hard-rejects it:

```rust
resource_bounds: match common.resource_bounds {
    ValidResourceBounds::AllResources(resource_bounds) => resource_bounds,
    _ => {
        return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR);
    }
},
``` [5](#0-4) 

For invoke and deploy-account transactions the same deserialization path applies because their `RpcInvokeTransactionV3` / `RpcDeployAccountTransactionV3` structs also carry `resource_bounds: AllResourceBounds` and the converter must extract `AllResources` from the deserialized `ValidResourceBounds`.

**Hash recomputation in `convert_rpc_tx_to_internal` (`transaction_converter.rs` line 391):**

Even if the peer does not immediately reject the transaction, the hash is recomputed from the deserialized `RpcTransaction`:

```rust
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [6](#0-5) 

Because the deserialized `InternalRpcDeclareTransactionV3` now wraps `AllResources` around the `L1Gas`-derived bounds, the recomputed hash differs from the one the account signed, breaking signature verification for any transaction that survives the earlier rejection.

**Trigger reachability:**

The stateless validator checks `l2_gas.max_price_per_unit >= min_gas_price` unconditionally: [7](#0-6) 

If `ResourceBounds::is_zero()` tests only `max_amount` (consistent with the test helper setting only `max_amount = GasAmount(1)` to suppress the bug), then a transaction with `l2_gas = { max_amount: 0, max_price_per_unit: ≥ min_gas_price }` passes the stateless validator while still satisfying `l2_gas.is_zero() == true`, making the trigger unprivileged. When `validate_resource_bounds = false` (bootstrap mode), the trigger is unconditional regardless of the `is_zero()` semantics. [8](#0-7) 

---

### Impact Explanation

Two distinct impacts apply:

1. **High — admission inconsistency.** A transaction accepted and hashed by the gateway is rejected by every consensus peer that deserialises it from protobuf (`DEPRECATED_RESOURCE_BOUNDS_ERROR` for declares; analogous extraction failure for invoke/deploy-account). The transaction is permanently lost from the block-building pipeline despite having been admitted.

2. **High — wrong hash bound to the executable payload.** For any transaction that survives the rejection, `convert_rpc_tx_to_internal` recomputes a hash using the `L1Gas` variant, producing a value that differs from the hash the account signed. The blockifier then executes the transaction under a hash the signer never authorised.

---

### Likelihood Explanation

The issue is acknowledged in production test code with an explicit comment and a manual workaround. The conversion heuristic has existed since the `AllResources` variant was introduced. Any V3 transaction whose L2 and L1-data gas `max_amount` fields are both zero (a valid configuration when a user intends to pay only in L1 gas) will reproduce the bug on every consensus P2P round-trip. The `validate_resource_bounds = false` bootstrap configuration makes the trigger unconditional for all such transactions.

---

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
