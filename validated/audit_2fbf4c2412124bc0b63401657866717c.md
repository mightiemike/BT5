### Title
AuxPoW Version Flag Enforced Only on the AuxPoW Submission Path, Not on the Direct-PoW Path — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin build of the BTC light client, `submit_block_header` branches on whether `AuxData` is present. The `BLOCK_VERSION_AUXPOW` flag (`0x100`) in the block version is verified **only** when AuxPoW data is supplied (inside `check_aux`). When no AuxPoW data is supplied, the code falls through to a direct-PoW check with **no assertion that the AuxPoW flag is absent**. A block whose version has the AuxPoW flag set can therefore be accepted by the contract using direct-PoW validation alone, bypassing the entire AuxPoW proof chain — a block the real Dogecoin network would unconditionally reject.

---

### Finding Description

`submit_block_header` in `contract/src/dogecoin.rs` dispatches on `aux_data`:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);   // ← enforces BLOCK_VERSION_AUXPOW flag
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
    // ← NO check that (block_header.version & BLOCK_VERSION_AUXPOW) == 0
}
``` [1](#0-0) 

Inside `check_aux`, the flag is explicitly required:

```rust
const BLOCK_VERSION_AUXPOW: i32 = 0x100;
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

The flag is therefore checked in exactly one direction: *"if AuxPoW data is present, the flag must be set."* The symmetric invariant — *"if the flag is set, AuxPoW data must be present"* — is never enforced. A caller can submit `(header_with_auxpow_flag_set, None)` and the contract will validate it purely via `block_hash_pow() ≤ target_from_bits(bits)`.

---

### Impact Explanation

On the real Dogecoin network, every block whose version has bit 8 set (`version & 0x100 != 0`) is required to carry a valid AuxPoW proof; without it the block is unconditionally invalid. The contract's missing check creates a divergence: a block that Dogecoin full nodes reject can be accepted by the on-chain light client.

If such a block is accepted, it is stored in `headers_pool` and can become the `mainchain_tip_blockhash` (or trigger a reorg via `submit_block_header_inner`). [3](#0-2) 

Once a fabricated block is on the contract's main chain, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will treat it as a valid anchor, allowing an attacker to forge SPV proofs for transactions that never existed on the real Dogecoin chain. [4](#0-3) 

The corrupted canonical mapping is: `mainchain_height_to_header[h]` and `mainchain_header_to_height[fake_hash]` point to a block hash that does not exist on the real Dogecoin chain.

---

### Likelihood Explanation

The entry point is `submit_blocks`, which is gated by `#[trusted_relayer]`. [5](#0-4) 

The `trusted_relayer` macro is an economic staking mechanism, not a cryptographic one — any account that stakes the required collateral becomes eligible. A well-funded attacker can stake, become a trusted relayer, and then submit the malformed block. Additionally, accounts holding `Role::UnrestrictedSubmitBlocks` bypass the trusted-relayer check entirely. [6](#0-5) 

The attacker must also produce a block whose direct scrypt/SHA256d hash satisfies `block_hash_pow() ≤ target_from_bits(bits)` — equivalent to mining a legitimate Dogecoin block. This is a significant but not insurmountable cost for a motivated adversary (e.g., a mining pool operator). Likelihood is **low-medium**.

---

### Recommendation

In the `else` branch of `submit_block_header` (the direct-PoW path), add an explicit assertion that the AuxPoW version flag is absent:

```rust
} else {
    require!(
        block_header.version & 0x100 == 0,
        "Non-AuxPoW submission must not have BLOCK_VERSION_AUXPOW set"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

This mirrors the symmetric check already present in `check_aux` and closes the inconsistency.

---

### Proof of Concept

1. Attacker stakes collateral and registers as a trusted relayer.
2. Attacker constructs a `Header` with `version = 0x00620100` (Dogecoin chain-id `0x62` in bits 31–16, AuxPoW flag `0x100` in bits 15–8) and a valid `prev_block_hash` pointing to the current contract tip.
3. Attacker mines until `scrypt(header_bytes) ≤ target_from_bits(bits)` — i.e., performs standard Dogecoin-difficulty PoW.
4. Attacker calls `submit_blocks` with `[(header, None)]` (no `AuxData`).
5. Inside `submit_block_header`:
   - `check_target` passes (difficulty is correct). [7](#0-6) 
   - `aux_data.is_none()` → `else` branch executes; direct-PoW check passes. [8](#0-7) 
   - `BLOCK_VERSION_AUXPOW` flag is never checked.
6. `submit_block_header_inner` stores the block in `headers_pool` and potentially promotes it to `mainchain_tip_blockhash`. [9](#0-8) 
7. The real Dogecoin network rejects this block; the contract's chain state now diverges from reality.
8. Attacker constructs a Merkle proof for a fabricated transaction anchored to the accepted fake block and calls `verify_transaction_inclusion_v2`, which returns `true`.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }
    }
