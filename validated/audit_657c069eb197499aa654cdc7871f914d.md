Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address as `sender` instead of actual end-user, nullifying per-user swap access control — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which `MetricOmmPool.swap` sets to `msg.sender` — the immediate caller of the pool. When `MetricOmmSimpleRouter` is used, `sender` is the router contract, not the end-user. This creates two mutually exclusive failure modes: either the router is allowlisted (granting every router user implicit access) or individual EOAs are allowlisted (locking them out when they route through the router).

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of the pool call. The actual end-user is stored only in transient callback context for payment, never forwarded to the pool or extension: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks `owner` (the position owner passed explicitly by the caller), which is a stable identity unaffected by intermediary routing: [6](#0-5) 

The swap extension has no equivalent stable identity — `sender` is always the intermediary when the router is used.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to specific counterparties (e.g., KYC-verified addresses, institutional participants) provides no effective restriction when the router is the entry point. Any unprivileged user can call `MetricOmmSimpleRouter.exactInputSingle` and swap against the restricted pool, causing unauthorized token outflows from the pool to arbitrary recipients and complete nullification of the pool admin's intended access boundary. This constitutes broken core pool functionality (access control) causing direct loss of funds to LPs who deposited under the assumption that the allowlist was enforced.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary production swap interface. Any pool deploying `SwapAllowlistExtension` expecting per-user gating is misconfigured by default. The bypass requires no special privileges — any EOA calling the router triggers it. The lockout of legitimately allowlisted users is equally automatic.

## Recommendation
The extension must identify the actual end-user, not the intermediary. Three viable approaches:

1. **Pass actual user via `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Acceptable since the router is a known, trusted periphery contract.
2. **Check `recipient` instead of `sender`**: For single-hop swaps the recipient is often the actual user. Imperfect for multi-hop paths where intermediate recipients are the router itself.
3. **Allowlist at the router level**: Add a router-level allowlist gating `exactInputSingle`/`exactInput` by `msg.sender` before calling the pool. The pool-level extension then only needs to allowlist the router as a trusted intermediary.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  pool admin calls setAllowedToSwap(pool, allowedUser, true)
  pool admin does NOT allowlist router or maliciousUser

Attack path A (lockout of legitimate user):
  allowedUser calls router.exactInputSingle(pool, ...)
  → router calls pool.swap(...)  [msg.sender = router]
  → pool calls _beforeSwap(sender=router, ...)
  → extension checks allowedSwapper[pool][router] → false
  → revert NotAllowedToSwap
  allowedUser is locked out despite being individually allowlisted.

Attack path B (bypass — pool admin allowlists router to unblock router users):
  pool admin calls setAllowedToSwap(pool, router, true)
  maliciousUser calls router.exactInputSingle(pool, ...)
  → router calls pool.swap(...)  [msg.sender = router]
  → pool calls _beforeSwap(sender=router, ...)
  → extension checks allowedSwapper[pool][router] → true
  → swap proceeds
  maliciousUser swaps successfully in a pool intended to be restricted.
```
