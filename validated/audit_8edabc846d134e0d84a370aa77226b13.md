### Title
Unsafe `transferFrom` Without Return Value Check in `replaceUsdcEWithUsdc` Allows Silent Failure and Potential usdcE Drain — (File: `core/contracts/ContractOwner.sol`)

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw `IERC20Base(usdc).transferFrom(...)` call without checking its return value, while the rest of the codebase consistently uses `ERC20Helper.safeTransferFrom` (which wraps the call and enforces the return value). If the usdc token at the hardcoded address returns `false` on failure rather than reverting, the inbound usdc leg silently fails while the outbound usdcE leg still executes, allowing any caller to drain usdcE from a victim's `DirectDepositV1` without providing usdc.

### Finding Description

`ContractOwner` imports and applies `ERC20Helper` as a library (`using ERC20Helper for IERC20Base`), which provides `safeTransfer` and `safeTransferFrom` wrappers that low-level-call the token and require `success && (data.length == 0 || abi.decode(data, (bool)))`. Every other token transfer in the contract uses these safe wrappers.

However, `replaceUsdcEWithUsdc` deviates:

```solidity
// ContractOwner.sol line 616 — unsafe, return value ignored
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// ContractOwner.sol line 618 — safe wrapper used for the outbound leg
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
```

The function has no access-control modifier — any address may call it for any registered `subaccount`. The execution sequence is:

1. Read `usdcE` balance of `directDepositV1`.
2. **Unsafe** `transferFrom`: pull `balance` usdc from `msg.sender` into `directDepositV1`.
3. `DirectDepositV1(directDepositV1).withdraw(usdcE)` — sends all usdcE to `ContractOwner`.
4. **Safe** `safeTransfer`: push `balance` usdcE from `ContractOwner` to `msg.sender`.

If step 2 returns `false` without reverting, steps 3 and 4 still execute. The caller receives `balance` usdcE while contributing zero usdc. The `directDepositV1` (belonging to an arbitrary victim subaccount) is drained.

### Impact Explanation

A caller with zero usdc balance (or zero allowance) can invoke `replaceUsdcEWithUsdc` for any subaccount whose `directDepositV1` holds usdcE. If the usdc token silently returns `false`, the caller receives the full usdcE balance of that `directDepositV1` at no cost. The victim subaccount permanently loses its usdcE collateral that was staged for deposit.

### Likelihood Explanation

The function is permissionless and chain-gated to ID 57073 (Ink mainnet). The exploitability depends on whether the specific usdc token at `0x2D270e6886d130D724215A266106e6832161EAEd` on that chain returns `false` on failure rather than reverting. Many ERC-20 tokens (including some USDC variants and bridged tokens) do return `false` rather than revert. The inconsistency with the rest of the codebase — which always uses safe wrappers — indicates this is an unintentional omission rather than a deliberate design choice, making it a realistic integration risk.

### Recommendation

Replace the raw `transferFrom` call with the `ERC20Helper`-backed safe variant, consistent with every other transfer in the contract:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe, using the already-imported ERC20Helper library):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

### Proof of Concept

1. Deploy or identify a subaccount on chain 57073 whose `directDepositV1` holds `N` usdcE.
2. As an attacker with 0 usdc allowance granted to `ContractOwner`, call `replaceUsdcEWithUsdc(subaccount)`.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` (no revert).
4. `DirectDepositV1.withdraw(usdcE)` executes — `N` usdcE moves to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, N)` executes — attacker receives `N` usdcE.
6. Victim's `directDepositV1` is empty; attacker gained `N` usdcE for free.

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
