### Title
Unknown-Op Cost Multiplier Integer Overflow Allows Cost Undercharge — (`src/more_ops.rs`)

---

### Summary

In `op_unknown` (`src/more_ops.rs`), the cost guard (`check_cost`) is applied to the **pre-multiplication** base cost, but the value actually returned to the interpreter is `cost * (cost_multiplier + 1)`. Because both operands are `u64` and Rust release builds use wrapping arithmetic on overflow, a crafted unknown opcode can cause the post-multiplication cost to wrap to zero (or any small value ≤ `u32::MAX`), bypassing the cost limit entirely. This is the direct analog of the staking report's root cause: a check is performed before the decisive scaling step, so the check does not guard the final allocated value.

---

### Finding Description

`op_unknown` in `src/more_ops.rs` computes a base cost from the opcode's arguments, then multiplies it by `(cost_multiplier + 1)`:

```
check_cost(cost, max_cost)?;      // line 260 — guards pre-multiplication cost
cost *= cost_multiplier + 1;      // line 261 — wraps silently in release mode
if cost > u32::MAX as u64 {       // line 262 — overflow can make this false
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))   // line 265 — returns wrapped cost
}
``` [1](#0-0) 

`cost_multiplier` is decoded from the opcode bytes as a `u32` cast to `u64`, so `cost_multiplier + 1` is at most `2^32 = 4,294,967,296`. [2](#0-1) 

`check_cost` only rejects if `cost > max_cost`: [3](#0-2) 

Chia's block cost limit is 11,000,000,000, so `cost` can legally reach values well above `u32::MAX = 4,294,967,295` before the multiplication. When `cost > 2^32` and `cost_multiplier = u32::MAX`, the product `cost * 2^32` exceeds `u64::MAX` and wraps. The concrete minimal case:

- `cost = 2^32 = 4,294,967,296` (passes `check_cost` since `4.3B < 11B`)
- `cost_multiplier = u32::MAX` → `cost_multiplier + 1 = 2^32`
- `cost * (cost_multiplier + 1) = 2^64 ≡ 0 (mod 2^64)` in wrapping arithmetic
- `0 > u32::MAX` is **false** → the function returns `Reduction(0, nil)` — **zero cost charged**

For `cost_function = 1` (add-like, the most common path), the base cost formula is:

```
ARITH_BASE_COST + n * ARITH_COST_PER_ARG + total_bytes * ARITH_COST_PER_BYTE
= 99 + n * 320 + total_bytes * 3
``` [4](#0-3) 

An attacker can choose `n` and `total_bytes` to make this sum exactly `2^32` (or any multiple of `2^32 / gcd(2^32, cost_multiplier+1)`). The inner `check_cost` inside the loop also passes because the running cost stays below `max_cost` throughout argument enumeration. [5](#0-4) 

---

### Impact Explanation

The returned cost of 0 (or any wrapped small value) is added to the interpreter's running `cost` accumulator in `run_program`:

```rust
cost += match op {
    Operation::Apply => self.apply_op(cost, effective_max_cost - cost)?,
    ...
};
``` [6](#0-5) 

Adding 0 means the block cost budget is not consumed. An attacker can embed this unknown op repeatedly in a CLVM program, each invocation costing 0, allowing a program to execute far more operations than the block cost limit permits. The concrete impacts are:

1. **Cost limit bypass**: Programs that should be rejected as too expensive are accepted by Rust nodes.
2. **Consensus divergence**: The reference Python `clvm` implementation uses arbitrary-precision integers that never overflow. Python would compute the true large cost (exceeding `u32::MAX`) and return `EvalErr::Invalid`, while the Rust implementation returns success with cost 0. The two implementations disagree on transaction validity — a chain-split vector.

---

### Likelihood Explanation

The attack requires only attacker-controlled CLVM bytes submitted as a standard transaction. Unknown opcodes are accepted in lenient/consensus mode (`allow_unknown_ops() = true`), which is the normal path for generator programs on the Chia blockchain. The opcode bytes that trigger the overflow are fully attacker-controlled: the `cost_multiplier` field is decoded directly from the opcode atom bytes, and the argument list controls the base cost. No privileged access, social engineering, or dependency compromise is required.

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and move the final cost guard to **after** the multiplication:

```rust
// Replace:
cost *= cost_multiplier + 1;
if cost > u32::MAX as u64 { ... }

// With:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
if cost > u32::MAX as u64 {
    return Err(EvalErr::Invalid(o))?;
}
```

This ensures the post-multiplication value is always checked before being returned, eliminating both the overflow and the misplaced guard.

---

### Proof of Concept

Craft an unknown opcode atom with:
- Last byte = `0b01_000000` → `cost_function = 1` (add-like), lower 6 bits = 0
- Preceding bytes = `[0xFF, 0xFF, 0xFF, 0xFF]` → `cost_multiplier = u32::MAX = 4,294,967,295`

Pass arguments totalling exactly `2^32 - 99 = 4,294,967,197` cost units (e.g., ~13.3 million 1-byte atoms, each costing `320 + 3 = 323` cost units).

Execution trace:
1. `check_cost(4_294_967_296, 11_000_000_000)` → passes (4.3B < 11B)
2. `cost *= 4_294_967_296` → `4_294_967_296 * 4_294_967_296 = 2^64 ≡ 0` (wrapping u64)
3. `0 > u32::MAX` → false
4. Returns `Reduction(0, nil)` — zero cost charged

The interpreter adds 0 to the running total; the block cost budget is unaffected. Repeating this op N times costs 0 total, while a Python node would reject each invocation with `EvalErr::Invalid`.
