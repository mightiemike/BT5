### Title
Insufficient Bonus Pool Blocks All Exit Paths, Permanently Locking User Principal - (`x/tieredrewards/keeper/msg_server.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`MsgTierUndelegate` and `MsgExitTierWithDelegation` — the only two exit paths available to a user after their exit commitment elapses — unconditionally call `claimRewards`, which calls `processEventsAndClaimBonus`, which hard-fails with `ErrInsufficientBonusPool` when the rewards pool cannot cover accrued bonus. If the bonus pool is empty and never replenished, users with any accrued bonus cannot undelegate or exit with delegation, permanently locking their principal tokens inside the module.

---

### Finding Description

`MsgTierUndelegate` calls `ms.claimRewards(ctx, pos)` before performing the unbond:

```go

if err != nil {
    return nil, err
}
```

    return types.PositionState{}, nil, nil, err
}
```

```

```

The same pattern applies to `MsgExitTierWithDelegation` at `msg_server.go:532`, `MsgTierRedelegate` at `msg_server.go:229`, `MsgAddToTierPosition` at `msg_server.go:314`, and `MsgClearPosition` at `msg_server.go:406`.

`MsgWithdrawFromTier` does **not** call `claimRewards` and is safe, but it requires the position to already be undelegated — which requires `MsgTierUndelegate` to have succeeded first. If `TierUndelegate` is blocked, `WithdrawFromTier` is unreachable.

The `BeforeRedelegationSlashed` hook already demonstrates the correct pattern for this situation: it forfeits bonus silently when the pool is insufficient to avoid a chain halt. No equivalent forfeit path exists for user-driven exit messages.

---

### Impact Explanation

A user who has accrued any non-zero bonus rewards (i.e., has been delegated to a bonded validator for any duration) cannot call `MsgTierUndelegate` or `MsgExitTierWithDelegation` if the bonus pool balance is less than their accrued bonus. Their principal tokens — locked in the position's per-position delegator address — are inaccessible until the pool is externally replenished. If the pool is never replenished (e.g., governance inaction, pool permanently drained), the principal is permanently stuck. The corrupted invariant is: **user principal tokens held at `pos.DelegatorAddress` are unreachable despite the exit commitment having elapsed**.

---

### Likelihood Explanation

The bonus pool is funded externally (governance or admin). The pool can be drained to zero by many users claiming rewards concurrently. Any position that has accrued even 1 token of bonus will trigger the failure. A user who triggers exit, waits the full exit duration (e.g., 1 year), and then attempts to undelegate during a period of pool exhaustion is completely blocked. This is a realistic operational scenario requiring no attacker — it can occur through normal usage patterns.

---

### Recommendation

For the two principal-recovery exit paths (`MsgTierUndelegate` and `MsgExitTierWithDelegation`), adopt the same forfeit-on-insufficient-pool pattern already used in `BeforeRedelegationSlashed` (`slash.go:54-63`): if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, log the forfeiture and continue with the exit rather than aborting the transaction. Base rewards (which route through `x/distribution` and are not pool-dependent) should still be claimed normally. This ensures user principal is always recoverable once the exit commitment elapses, regardless of pool state.

---

### Proof of Concept

1. User calls `MsgLockTier` and delegates to a bonded validator.
2. Time passes; bonus accrues (non-zero `totalBonus` in `processEventsAndClaimBonus`).
3. User calls `MsgTriggerExitFromTier` and waits for `ExitUnlockAt`.
4. The bonus pool is drained to zero (e.g., other users claim rewards, or pool is never funded).
5. User calls `MsgTierUndelegate` — transaction fails with `ErrInsufficientBonusPool` at `claim_rewards.go:230`.
6. User calls `MsgExitTierWithDelegation` — same failure at `msg_server.go:532` → `claim_rewards.go:97` → `claim_rewards.go:230`.
7. `MsgWithdrawFromTier` is unreachable because the position is still delegated.

	}

	}

		return nil, err
	}

	ownerAddr, err := sdk.AccAddressFromBech32(pos.Owner)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid owner address")

	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	balances := ms.bankKeeper.SpendableCoins(ctx, delAddr)
	if !balances.IsZero() {
		if err := ms.bankKeeper.SendCoins(ctx, delAddr, ownerAddr, balances); err != nil {
			return nil, err
		}
	}

	if err := ms.deletePosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionWithdrawn{
		Position: pos.Position,
		Amount:   balances,
	}); err != nil {
		return nil, err
	}

	return &types.MsgWithdrawFromTierResponse{
		Amount: balances,
	}, nil
}

func (k Keeper) claimRewards(ctx context.Context, pos types.PositionState) (types.PositionState, sdk.Coins, sdk.Coins, error) {
	if !pos.IsDelegated() {
		return pos, sdk.NewCoins(), sdk.NewCoins(), nil
	}

	base, err := k.claimBaseRewards(ctx, pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
```
