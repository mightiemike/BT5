Looking at the HSG vulnerability class — **cross-contract desynchronization in authorization checks**, where two components apply different checks to the same data, allowing an attacker to satisfy both independently — I need to find where Nado has two code paths that enforce different authorization rules for the same state-changing operation.

The most direct analog is in `EndpointTx.sol`.

---

### Title
Linked Signer Can Escalate Privileges by Signing `LinkSigner` Transactions — (`core/contracts/EndpointTx.sol`)

### Summary

The fast-mode path for `LinkSigner` in `EndpointTx.processTransactionImpl` passes `allowLinkedSigner = true`, meaning the current linked signer can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. The slow-mode path for the same operation enforces the opposite: only the subaccount owner (`msg.sender`) may change the linked signer. This desynchronization between the two execution paths allows a compromised linked signer to permanently take over a subaccount.

### Finding Description

**Fast-mode path** (`processTransactionImpl`, `EndpointTx.sol` lines 576–590):

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to `verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it recovers to either the subaccount owner **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

**Slow-mode path** (`processSlowModeTransactionImpl`, `EndpointTx.sol` lines 232–239):

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.LinkSigner)
    );
    validateSender(txn.sender, sender);   // ← only msg.sender == owner passes
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

`validateSender` requires `msg.sender` to be the subaccount owner address:

```solidity
function validateSender(bytes32 txSender, address sender) internal view {
    require(
        address(uint160(bytes20(txSender))) == sender ||
            sender == address(this),
        ERR_SLOW_MODE_WRONG_SENDER
    );
}
``` [5](#0-4) 

The two paths enforce **different authorization sets** for the same write to `linkedSigners`:

| Path | Who may authorize a `LinkSigner` change |
|---|---|
| Slow mode | Subaccount owner only (`msg.sender == owner`) |
| Fast mode | Subaccount owner **or** current linked signer |

This is the direct analog of the HSG bug: the Safe checked the first `threshold` signatures (owner-only), while the HSG checked all signatures (including invalid ones). Here, slow mode checks owner-only, while fast mode also accepts the linked signer — two components applying different checks to the same authorization decision.

### Impact Explanation

A compromised linked signer (e.g., a session key stored in a browser wallet or hot wallet) can:

1. Sign a `LinkSigner` transaction naming an attacker-controlled address as the new signer.
2. Submit it to the sequencer for fast-mode processing.
3. `processTransactionImpl` accepts it because `allowLinkedSigner = true`.
4. `linkedSigners[subaccount]` is overwritten with the attacker's address.
5. The attacker can now sign `WithdrawCollateral`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, and any other transaction that accepts `allowLinkedSigner = true`.

The subaccount owner loses control of their subaccount and all collateral within it. The `linkedSigners` mapping is the root state corrupted. [6](#0-5) 

### Likelihood Explanation

Linked signers are explicitly designed as session keys — lower-security keys stored in hot wallets, browser extensions, or automated trading bots. They are materially more likely to be compromised than the primary subaccount owner key. Any user who has ever called `LinkSigner` is exposed. The attack requires only that the linked signer submit one transaction to the sequencer, which is the normal operational path.

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transactions in the fast-mode path, matching the slow-mode path's restriction to the subaccount owner only:

```diff
- validateSignedTx(
-     signedTx.tx.sender,
-     signedTx.tx.nonce,
-     transaction,
-     signedTx.signature,
-     true
- );
+ validateSignedTx(
+     signedTx.tx.sender,
+     signedTx.tx.nonce,
+     transaction,
+     signedTx.signature,
+     false   // only the subaccount owner may change the linked signer
+ );
``` [7](#0-6) 

This aligns the fast-mode and slow-mode authorization checks for `LinkSigner`, eliminating the desynchronization.

### Proof of Concept

1. Alice calls `LinkSigner` (slow mode) to set Bob as her linked signer. `linkedSigners[alice_subaccount] = Bob`.
2. Bob's key is compromised. Attacker constructs a `SignedLinkSigner` transaction: `{sender: alice_subaccount, signer: attacker_address, nonce: current_nonce}`, signed by Bob.
3. Attacker submits this to the sequencer. The sequencer includes it in a `submitTransactions` batch.
4. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`.
5. `validateSignature` calls `getLinkedSigner(alice_subaccount)` → returns `Bob`. `verifier.validateSignature` recovers `Bob` from the signature, which equals `linkedSigner`. Check passes.
6. `linkedSigners[alice_subaccount] = attacker_address`.
7. Attacker signs a `WithdrawCollateral` transaction for Alice's subaccount. It passes `validateSignature` because `recovered == linkedSigner == attacker_address`. Alice's collateral is drained.
