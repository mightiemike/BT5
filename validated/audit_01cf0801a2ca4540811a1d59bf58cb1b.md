### Title
Unbounded Iteration Over `NlpLockedBalanceQueue` Causes Permanent NLP Token Lock — (`File: core/contracts/SpotEngine.sol`)

---

### Summary

Every NLP mint or burn operation for a subaccount unconditionally calls `tryUnlockNlpBalance`, which iterates over the subaccount's `NlpLockedBalanceQueue`. Because a new queue entry is appended each time a user mints NLP at a distinct oracle timestamp, the queue grows without bound. Once the queue is large enough, `tryUnlockNlpBalance` exhausts the block gas limit, permanently bricking all future `mintNlp` and `burnNlp` calls for that subaccount and making the locked NLP balance irrecoverable.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` is invoked inside `SpotEngine.updateBalance` whenever `productId == NLP_PRODUCT_ID`. Its first action, regardless of whether the caller is minting or burning, is to call `tryUnlockNlpBalance(subaccount)`. [1](#0-0) 

Each time a user mints NLP at a timestamp that differs from the last queue entry's `unlockedAt`, a brand-new `NlpLockedBalance` slot is appended and `queue.balanceCount` is incremented: [2](#0-1) 

The queue is backed by a `mapping(uint256 => NlpLockedBalance)` with no upper-bound cap and no compaction mechanism. `tryUnlockNlpBalance` must scan from the beginning of the queue on every invocation to compute `unlockedBalanceSum`. Because there is no persistent "head pointer" that advances past already-processed entries, the iteration cost grows linearly with `balanceCount`.

`burnNlp` in `Clearinghouse` calls `spotEngine.getNlpUnlockedBalance`, which also calls `tryUnlockNlpBalance` before the health check: [3](#0-2) [4](#0-3) 

There is no alternative code path that bypasses `tryUnlockNlpBalance` for a subaccount that has accumulated a large queue. No batch-processing entry point exists to drain the queue in smaller chunks.

---

### Impact Explanation

Once `tryUnlockNlpBalance` for a given subaccount exceeds the block gas limit:

- Every `mintNlp` call for that subaccount reverts.
- Every `burnNlp` call for that subaccount reverts (both the `getNlpUnlockedBalance` check and the `updateBalance` call inside it).
- The subaccount's NLP tokens are permanently locked: they cannot be redeemed for quote collateral, and the underlying quote value is irrecoverable.

This is a **collateral lock** — a concrete, permanent asset impact — not merely a gas-cost degradation.

---

### Likelihood Explanation

A user who mints NLP across many distinct oracle-time slots (e.g., one mint per block over an extended period) will steadily grow their queue. At typical EVM gas costs, a queue of several thousand entries is sufficient to exceed the 30 M gas block limit. This is reachable through normal protocol usage without any privileged access. The user may not be aware of the accumulation until the queue is already too large to process.

---

### Recommendation

1. **Introduce a persistent `headIndex`** in `NlpLockedBalanceQueue` so that `tryUnlockNlpBalance` advances the head past already-unlocked entries and never re-scans them. This bounds each call's iteration to newly-unlocked entries only.
2. **Cap `balanceCount`** or merge adjacent entries aggressively (not only when the timestamp is identical) to prevent unbounded growth.
3. **Expose a standalone `processNlpQueue(subaccount, maxEntries)`** function so that a subaccount with a large queue can drain it in batches before attempting a burn.

---

### Proof of Concept

1. Attacker/user calls `MintNlp` once per block (or at any distinct oracle timestamp) over an extended period. Each call appends one entry to `nlpLockedBalanceQueues[sender]` via `handleNlpLockedBalance` → `queue.balanceCount++`. [5](#0-4) 

2. After `N` distinct-timestamp mints, `queue.balanceCount == N`.

3. User submits a `BurnNlp` transaction. `Clearinghouse.burnNlp` calls `spotEngine.getNlpUnlockedBalance(txn.sender)`: [4](#0-3) 

4. `getNlpUnlockedBalance` calls `tryUnlockNlpBalance(subaccount)`, which iterates over all `N` queue entries. [3](#0-2) 

5. When `N` is large enough, the iteration exhausts the block gas limit. The transaction reverts.

6. The same revert occurs for every subsequent `mintNlp` and `burnNlp` call, because both paths invoke `handleNlpLockedBalance` → `tryUnlockNlpBalance`. The user's NLP balance is permanently frozen with no recovery path. [6](#0-5)

```text
    function handleNlpLockedBalance(bytes32 subaccount, int128 amountDelta)
        internal
    {
        _assertInternal();

        // N_ACCOUNT is not limited by lock period
        if (subaccount == N_ACCOUNT) return;

        tryUnlockNlpBalance(subaccount);
        if (amountDelta > 0) {
            NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
                subaccount
            ];
            if (
                queue.balanceCount > 0 &&
                queue.balances[queue.balanceCount - 1].unlockedAt ==
                getOracleTime() + NLP_LOCK_PERIOD
            ) {
                queue
                    .balances[queue.balanceCount - 1]
                    .balance
                    .amount += amountDelta;
            } else {
                queue.balances[queue.balanceCount] = NlpLockedBalance({
                    balance: Balance({amount: amountDelta}),
                    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
                });
                queue.balanceCount++;
            }
        } else if (amountDelta < 0) {
            Balance memory balanceSum = nlpLockedBalanceQueues[subaccount]
                .unlockedBalanceSum;
            balanceSum.amount += amountDelta;
            nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
        }
    }
```
