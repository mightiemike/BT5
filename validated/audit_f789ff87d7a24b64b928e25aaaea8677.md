### Title
Global `lastLiquidationFees` Overwrite Causes Cross-Liquidatee Insurance Accounting Corruption — (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

`lastLiquidationFees` is a single, unscoped global storage slot in `ClearinghouseStorage`. Any non-finalization liquidation of **any** subaccount overwrites it. When a finalization call (`productId == type(uint32).max`) is processed for subaccount A after a non-finalization liquidation of subaccount B has overwritten the slot, `_finalizeSubaccount` uses B's fee value to gate A's insurance socialization, producing incorrect insurance fund accounting and potential bad debt.

---

### Finding Description

`lastLiquidationFees` is declared as a bare `int128` storage variable with no per-liquidatee key: [1](#0-0) 

It is written unconditionally at the end of every non-finalization liquidation: [2](#0-1) 

It is then consumed in `_finalizeSubaccount` to compute the effective insurance available for socialization and cover: [3](#0-2) 

The execution path in `liquidateSubaccountImpl` is: [4](#0-3) 

Because `_finalizeSubaccount` returns `false` immediately when `productId != type(uint32).max`, and `_handleLiquidationPayment` is only reached on the non-finalization branch, the two operations are always in separate transactions. There is no mechanism that ties the stored `lastLiquidationFees` to a specific `liquidatee`.

**Interleaving sequence that breaks the invariant:**

| Step | Transaction | Effect on storage |
|------|-------------|-------------------|
| 1 | `liquidateSubaccountImpl(liquidatee=A, productId=X)` | `lastLiquidationFees = A_fees` |
| 2 | `liquidateSubaccountImpl(liquidatee=B, productId=X)` | `lastLiquidationFees = B_fees` ← overwrites |
| 3 | `liquidateSubaccountImpl(liquidatee=A, productId=uint32.max)` | reads `lastLiquidationFees = B_fees` — wrong |

At step 3, `_finalizeSubaccount` computes:

```
v.insurance = insurance;
v.insurance -= B_fees;          // should be A_fees
v.canLiquidateMore = (quoteBalance.amount + v.insurance) > 0;
v.insurance = perpEngine.socializeSubaccount(A, v.insurance);  // wrong budget
insuranceCover = min(v.insurance, -quoteBalance.amount);       // wrong cover
v.insurance += B_fees;          // adds back wrong amount
insurance = v.insurance;        // persists corrupted value
```

---

### Impact Explanation

**Case B_fees > A_fees:** `v.insurance` is understated. Less insurance is offered to `perpEngine.socializeSubaccount` and to `insuranceCover`, so losses that the insurance fund could have absorbed are instead socialized across all counterparties — incorrect bad debt distribution.

**Case B_fees < A_fees:** `v.insurance` is overstated. More insurance is consumed for A's cover than the protocol intended (the "held-back" buffer is smaller than it should be), depleting the insurance fund faster than designed.

In both cases the final `insurance` storage value written at line 411 is incorrect, and the error compounds across subsequent liquidations and finalizations.

---

### Likelihood Explanation

The attack requires only that a liquidator submits, in order:
1. A non-finalization liquidation of any second subaccount B (which must be under maintenance health — a normal market condition).
2. A finalization of subaccount A.

Both calls go through `liquidateSubaccount` → `delegatecall` → `liquidateSubaccountImpl`, gated by `onlyEndpoint`. The sequencer processes signed liquidation transactions in submission order. A liquidator controlling the submission order of their own signed transactions can trivially produce this sequence whenever two subaccounts are simultaneously liquidatable, which is common during market stress — exactly when the insurance fund matters most.

No privileged access, no sequencer compromise, and no impossible token behavior is required.

---

### Recommendation

Scope `lastLiquidationFees` to the specific liquidatee being processed. The simplest fix is to replace the global slot with a `mapping(bytes32 => int128)` keyed by `liquidatee`, or to pass the fee value as a return value from `_handleLiquidationPayment` and store it in a transient/local variable rather than in persistent storage. Alternatively, clear `lastLiquidationFees` to zero at the start of each `liquidateSubaccountImpl` call and only set it when the non-finalization branch executes, then verify in `_finalizeSubaccount` that the value belongs to the current `txn.liquidatee` (e.g., by storing a `(liquidatee, fees)` pair).

---

### Proof of Concept

```solidity
// Pseudocode — two sequential endpoint-submitted transactions

// Tx 1: partial liquidation of Alice (sets lastLiquidationFees = 50e18)
endpoint.submitTransaction(LiquidateSubaccount({
    sender: attacker, liquidatee: alice, productId: BTC_PERP, amount: -1e18
}));

// Tx 2: partial liquidation of Bob (overwrites lastLiquidationFees = 200e18)
endpoint.submitTransaction(LiquidateSubaccount({
    sender: attacker, liquidatee: bob, productId: BTC_PERP, amount: -4e18
}));

// Tx 3: finalize Alice — _finalizeSubaccount reads lastLiquidationFees = 200e18
//        instead of 50e18; effective insurance is understated by 150e18;
//        Alice's deficit is socialized when insurance could have covered it.
endpoint.submitTransaction(LiquidateSubaccount({
    sender: attacker, liquidatee: alice, productId: type(uint32).max, amount: 0
}));

// Assert: insurance after Tx3 != expected_insurance
// Assert: socialized bad debt > 0 when it should be 0
```
