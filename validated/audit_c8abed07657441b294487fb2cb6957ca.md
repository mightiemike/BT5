### Title
Zero-Confirmation Bypass in `verify_transaction_inclusion` Allows SPV Proof to Pass for Unconfirmed Transactions — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`, which delegates to it) accepts `confirmations = 0` from any unprivileged NEAR caller. When `confirmations` is zero, the depth guard `(tip_height - target_height + 1) >= confirmations` reduces to `N >= 0`, which is trivially true for every `u64`. The function then proceeds to validate the Merkle proof and returns `true`, producing a valid-looking SPV proof result for a transaction that has zero confirmation depth. Any recipient contract that consumes this result and releases funds or updates state based on it is exposed to a double-spend or race-condition attack.

---

### Finding Description

The confirmation-depth guard in `verify_transaction_inclusion` is:

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

When `args.confirmations = 0`, the left-hand side is some `u64 ≥ 0`, so the `require!` never fires regardless of how shallow the target block is. The only preceding guard is:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [2](#0-1) 

`0 <= gc_threshold` is always true, so this guard is also bypassed. The function then verifies the Merkle proof and returns `true`.

`verify_transaction_inclusion_v2` — the recommended replacement — calls `verify_transaction_inclusion` directly after its coinbase-proof check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

So the zero-confirmation bypass is inherited by the v2 path as well.

The structural parallel to H-03 is exact:

| H-03 (MarginTrading) | This repository |
|---|---|
| `debtWithAccruedInterest == 0` → `ratio = 0` | `confirmations = 0` |
| `0 <= redCoeff` always passes | `N >= 0` always passes |
| Unauthorized liquidation of debt-free account | Unauthorized SPV proof for unconfirmed transaction |

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion` (or v2) and releases funds, mints tokens, or updates on-chain state upon receiving `true` is vulnerable. An attacker can:

1. Broadcast a Bitcoin transaction and wait for it to appear in the tip block (one block, zero confirmations below tip).
2. Immediately call `verify_transaction_inclusion` with `confirmations = 0` and a valid Merkle proof.
3. The function returns `true`.
4. The recipient contract acts on the result.
5. The attacker then attempts to double-spend the Bitcoin transaction before it accumulates real confirmations.

Because the tip block can still be orphaned, the "verified" transaction may never be finalized on Bitcoin, yet the NEAR-side action has already been executed.

**Impact: Medium** — financial loss for any recipient contract that does not independently enforce a minimum confirmation count.

---

### Likelihood Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are public, unpermissioned NEAR view/call methods reachable by any account. [4](#0-3) [5](#0-4) 

The `ProofArgs` struct is fully attacker-controlled; `confirmations` is a plain integer field with no lower-bound validation anywhere in the contract. Any caller can supply `confirmations = 0` without any privilege. **Likelihood: High.**

---

### Recommendation

Add an explicit lower-bound check at the top of `verify_transaction_inclusion`:

```rust
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
    require!(args.confirmations >= 1, "Confirmations must be at least 1");
    require!(
        args.confirmations <= self.gc_threshold,
        "The required number of confirmations exceeds the number of blocks stored in memory"
    );
    // ... rest unchanged
}
```

Because `verify_transaction_inclusion_v2` delegates to v1, this single fix covers both entry points.

---

### Proof of Concept

1. Deploy the BTC light client contract (Bitcoin feature, `skip_pow_verification = false`).
2. Submit enough real headers so the main chain has at least one block beyond genesis.
3. Obtain a valid Merkle proof for a transaction `tx_id` in the tip block (`tx_block_blockhash = mainchain_tip_blockhash`).
4. Call:
   ```
   verify_transaction_inclusion({
       tx_id,
       tx_block_blockhash,   // tip block — zero confirmations below tip
       tx_index,
       merkle_proof,
       confirmations: 0      // zero-confirmation bypass
   })
   ```
5. Observe the function returns `true` despite the transaction having zero confirmation depth.
6. A recipient contract acting on this result would process the transaction before it is finalized on Bitcoin. [6](#0-5)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
