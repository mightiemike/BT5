### Title
Zero `confirmations` Bypasses Confirmation Security in `verify_transaction_inclusion` - (File: `contract/src/lib.rs`)

### Summary
`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) does not enforce a minimum confirmation count. When a caller supplies `confirmations = 0`, the confirmation depth check becomes trivially true for any u64 arithmetic, allowing any unprivileged NEAR caller to obtain a `true` verification result for any transaction in any mainchain block regardless of how recently it was added.

### Finding Description
The function `verify_transaction_inclusion` performs two checks related to confirmations:

```rust
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

and

```rust
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [1](#0-0) 

When `args.confirmations = 0`:
- The first check becomes `0 <= gc_threshold`, which is always true since `gc_threshold` is `u64`.
- The second check becomes `(u64_value) >= 0`, which is always true in Rust's unsigned arithmetic.

There is no `require!(args.confirmations >= 1, ...)` or any lower-bound enforcement anywhere in the function. The same issue propagates through `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` internally after its coinbase proof check. [2](#0-1) 

The `confirmations` field is caller-supplied via `ProofArgs` with no contract-side floor: [3](#0-2) 

### Impact Explanation
Any unprivileged NEAR caller can call `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` with `confirmations = 0` and a valid Merkle proof for any transaction in any mainchain block. The function returns `true` unconditionally with respect to confirmation depth. A downstream recipient contract that gates fund releases or cross-chain actions on this verification result would accept 0-confirmation Bitcoin transactions, making it directly vulnerable to double-spend attacks: an attacker broadcasts a transaction, immediately calls the NEAR contract with `confirmations = 0`, and claims the funds before the Bitcoin transaction is confirmed or even before a competing spend is detected.

### Likelihood Explanation
High. The entry path requires no privilege, no staked tokens, and no special role. Any NEAR account can call the public `verify_transaction_inclusion_v2` endpoint with `confirmations = 0` and a structurally valid proof. The `confirmations` field is a plain `u64` with no on-chain validation of its lower bound.

### Recommendation
Add an explicit lower-bound check at the start of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "Confirmations must be at least 1");
```

Alternatively, define a protocol-level constant (e.g., `MIN_CONFIRMATIONS = 6`) and enforce it:

```rust
require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
```

This mirrors the fix described in the reference report (adding `minimumTokensLockedForProposalCreation` as a hard lower bound), applied here as a hard lower bound on the confirmation count.

### Proof of Concept

1. Deploy the contract with any valid genesis block and `gc_threshold = 1000`.
2. Submit one block header containing a target transaction.
3. Call `verify_transaction_inclusion_v2` with:
   - `tx_id`: the target transaction hash
   - `tx_block_blockhash`: the block hash from step 2
   - `tx_index`: correct index
   - `merkle_proof`: valid Merkle path
   - `coinbase_tx_id` + `coinbase_merkle_proof`: valid coinbase proof
   - **`confirmations: 0`**
4. The function returns `true` immediately, with the block having zero confirmations beyond itself.
5. The confirmation depth check `(tip_height - target_height + 1) >= 0` evaluates to `1 >= 0` → `true`, bypassing all confirmation security. [4](#0-3)
