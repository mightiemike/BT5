### Title
Accrued Bonus Rewards Permanently Lost When Bonus Pool Is Insufficient During Redelegation Slash — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`slashRedelegationPosition` calls `processEventsAndClaimBonus`, which unconditionally advances the position's reward checkpoints and decrements event reference counts in the store **before** checking pool sufficiency. When the pool is insufficient, `ErrInsufficientBonusPool` is returned, but `slashRedelegationPosition` swallows that error and then persists the already-advanced position state. The bonus for the entire pre-slash accrual period is permanently destroyed.

---

### Finding Description

`processEventsAndClaimBonus` executes in this order:

1. **Loop over events** (lines 172–198): for each event, calls `pos.UpdateLastEventSeq(entry.Seq)` (in-memory) and `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` — a **direct store write** that removes or decrements the event.
2. **After the loop** (line 215): calls `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` — advances `pos.LastBonusAccrual` to `blockTime` in-memory.
3. **After the loop** (line 217): calls `pos.UpdateLastKnownBonded(bonded)` — in-memory.
4. **Only then** (line 230): calls `sufficientBonusPoolBalance` — if the pool is drained, returns `ErrInsufficientBonusPool`. [1](#0-0) 

Back in `slashRedelegationPosition`, the error is caught and swallowed:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
    } else {
        return err
    }
}
``` [2](#0-1) 

Execution then falls through to `setPositionWithState(ctx, pos, ...)`, which persists the `pos` struct — now carrying the advanced `LastEventSeq` and `LastBonusAccrual` — to the store. [3](#0-2) 

The result is a broken invariant: the position's checkpoints say "bonus has been settled up to `blockTime`", but no bonus was ever paid. The store-level event ref-count decrements are also already committed, so those events cannot be replayed. Any subsequent call to `processEventsAndClaimBonus` (e.g., via `ClaimTierRewards`) starts from the advanced `LastEventSeq` and `LastBonusAccrual`, skipping the entire pre-slash period entirely. [4](#0-3) 

---

### Impact Explanation

A position owner permanently loses all accrued bonus rewards for the period from their last claim up to the slash event. The loss is proportional to position size, tier `BonusApy`, and the length of the unclaimed accrual window. There is no recovery path: the events are consumed, the checkpoint is advanced, and the pool can be refilled afterward without any effect on the already-lost claim.

---

### Likelihood Explanation

The preconditions are realistic and reachable on mainnet:
1. A position redelegates, creating a redelegation entry tracked by `RedelegationMappings`.
2. Other positions (or the ABCI reward distribution) drain the bonus pool before the slash fires.
3. The validator is slashed (double-sign or downtime), triggering `BeforeRedelegationSlashed` → `slashRedelegationPosition`.

Bonus pool depletion is a normal operational condition (many concurrent claims, or a temporarily underfunded pool). Validator slashing is a standard chain event. The combination is not exotic.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastEventSeq` / `decrementEventRefCount` to **after** the successful pool transfer, or — if the intent is to skip bonus on pool exhaustion — do not advance the checkpoints and do not decrement event ref counts when returning `ErrInsufficientBonusPool`. The position should be left in its pre-call state so the accrual can be claimed once the pool is refilled.

Concretely, the fix in `processEventsAndClaimBonus` should:
- Compute `totalBonus` and collect which events were processed (in-memory only).
- Check pool sufficiency **before** calling `decrementEventRefCount` or `applyBonusAccrualCheckpoint`.
- Only advance checkpoints and decrement ref counts after a successful `SendCoinsFromModuleToAccount`.

---

### Proof of Concept

```
1. Create a tier with BonusApy > 0.
2. Create position P, delegate to validator V, wait N blocks to accrue bonus.
3. Drain the RewardsPool module account to zero (e.g., via other positions claiming).
4. Redelegate P from V to V2 (creates a redelegation entry + RedelegationMapping).
5. Slash validator V for the redelegation (triggers BeforeRedelegationSlashed →
   slashRedelegationPosition → processEventsAndClaimBonus).
   - processEventsAndClaimBonus computes totalBonus > 0, decrements event refs,
     advances LastBonusAccrual to blockTime, then returns ErrInsufficientBonusPool.
   - slashRedelegationPosition logs the error and calls setPositionWithState,
     persisting the advanced checkpoints.
6. Refill the RewardsPool.
7. Call MsgClaimTierRewards for position P.
   - processEventsAndClaimBonus starts from the already-advanced LastEventSeq /
     LastBonusAccrual; the pre-slash accrual window is gone.
   - Assert: bonus received == 0 for the pre-slash period. ✓
``` [5](#0-4) [6](#0-5)

func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
	accrualEnd := blockTime
	if pos.CompletedExitLockDuration(blockTime) {
		accrualEnd = pos.ExitUnlockAt
	}
	pos.UpdateLastBonusAccrual(accrualEnd)
}
