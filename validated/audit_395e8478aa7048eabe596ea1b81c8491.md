### Title
Contract with >1024 WASM type-section entries deployed at protocol v83 becomes permanently unexecutable after upgrade to v84 — (`runtime/near-vm-runner/src/prepare/prepare_v2.rs`)

### Summary
`max_types_per_contract` was absent (`None`) at protocol version 83 and introduced as `Some(1024)` at protocol version 84 via `84.yaml`. A contract with 1025+ type entries passes preparation at v83 (effective limit = `u64::MAX`) but fails re-preparation at v84 with `PrepareError::TooManyTypes` after the compiled-contract cache is invalidated at the epoch boundary. The contract becomes permanently unexecutable.

### Finding Description
In `PrepareContext::new()`:

```rust
type_limit: limits.max_types_per_contract.unwrap_or(u64::MAX),
``` [1](#0-0) 

At v83, `max_types_per_contract = None` → `type_limit = u64::MAX`. At v84, `max_types_per_contract = Some(1024)` → `type_limit = 1024`.

The type-section check in `run()`:

```rust
self.type_limit = self
    .type_limit
    .checked_sub(u64::from(reader.count()))
    .ok_or(PrepareError::TooManyTypes)?;
``` [2](#0-1) 

The same check exists identically in `prepare_v3.rs`: [3](#0-2) 

The `84.yaml` config diff introduces the limit as a **new** parameter (no `old` key = was `None` before): [4](#0-3) 

The `LimitConfig` field is `Option<u64>`: [5](#0-4) 

At the epoch boundary, `chain/chain/src/runtime/mod.rs` detects the wasm_config change via `cache_keys_differ()` and invalidates the compiled-contract cache: [6](#0-5) 

After invalidation, any function call to the contract triggers re-preparation under v84's config. For a contract with 1025 type entries, this returns `Err(PrepareError::TooManyTypes)`, which is stored and surfaced as a function-call failure. There is no grandfathering mechanism for contracts deployed before the limit existed.

### Impact Explanation
A contract deployed at v83 with >1024 WASM type-section entries becomes permanently unexecutable after the protocol upgrades to v84. All function calls to the contract fail. The contract's on-chain state and any locked funds remain inaccessible through the contract's own logic. This is a concrete, protocol-visible divergence: `Ok(prepared_bytes)` at v83 vs `Err(PrepareError::TooManyTypes)` at v84 for the same stored bytecode.

### Likelihood Explanation
Low-to-moderate. A WASM type section with >1024 entries is unusual for hand-written contracts but plausible for compiler-generated code (e.g., Rust contracts with many distinct closure or trait-object function signatures, or contracts using heavy generics). The v83→v84 upgrade is a supported sequential upgrade path. Any contract deployed on mainnet at v83 with this property would be silently broken by the upgrade.

### Recommendation
Apply `max_types_per_contract` only to contracts **deployed at or after** protocol version 84. One approach: during re-preparation triggered by cache invalidation, skip the type-count check for contracts whose deploy receipt was processed under a protocol version where the limit did not exist. Alternatively, set the initial limit high enough (e.g., `u64::MAX` or the WebAssembly spec's own limit of 2^32−1) to grandfather all existing contracts, and only enforce a tighter limit on new deployments.

### Proof of Concept
Using the existing `contract_with_n_types` helper already present in the test suite: [7](#0-6) 

1. Build a WASM module with 1025 type entries via `contract_with_n_types(1025)`.
2. Deploy it under a `RuntimeConfig` for protocol version 83 (where `max_types_per_contract = None`). Preparation succeeds.
3. Simulate the epoch boundary: switch to a `RuntimeConfig` for protocol version 84 (where `max_types_per_contract = Some(1024)`), invalidating the cache.
4. Attempt to call the contract. Re-preparation runs under v84's config and returns `Err(PrepareError::TooManyTypes)`.
5. The call fails permanently — the contract is unexecutable for the remainder of the chain's lifetime.
