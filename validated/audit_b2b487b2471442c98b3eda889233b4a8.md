### Title
`ValidResourceBounds` Protobuf Round-Trip Failure Silently Downgrades `AllResources` to `L1Gas`, Causing Validators to Reject Valid Block Proposals — (`crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` deserialization uses a value-based heuristic to decide between the `L1Gas` and `AllResources` variants. Any `AllResources` transaction whose `l2_gas` and `l1_data_gas` bounds are both zero is silently re-classified as `L1Gas` on the receiving side. Because `RpcInvokeTransactionV3` requires `AllResourceBounds`, the subsequent `InvokeTransactionV3 → RpcInvokeTransactionV3` conversion fails with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. A validator that receives such a transaction inside a consensus block proposal marks the entire proposal as `Failed`, breaking consensus progress for any block that contains such a transaction.

### Finding Description

**Serialization side** — `From<ValidResourceBounds> for protobuf::ResourceBounds` always emits all three fields:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [1](#0-0) 

**Deserialization side** — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` uses a lossy heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
} else {
    ValidResourceBounds::AllResources(...)
})
``` [2](#0-1) 

When `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is serialized and then deserialized, the result is `L1Gas(X)` — a different variant. The round-trip is broken.

**Downstream failure** — `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()` to extract `AllResourceBounds` from the deserialized `InvokeTransactionV3`. The comment in the code itself acknowledges this can only fail when the variant is not `AllResources`:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [3](#0-2) 

**Consensus rejection** — `validate_proposal` collects conversion results and, on any error, returns `HandledProposalPart::Failed`:

```rust
let conversion_results = match conversion_results {
    Ok(results) => results,
    Err(e) => {
        return HandledProposalPart::Failed(format!(
            "Failed to convert transactions. Stopping the build of the current proposal. {e:?}"
        ));
    }
};
``` [4](#0-3) 

**Gateway accepts the triggering transaction** — The stateless validator explicitly marks `AllResourceBounds { l1_gas: NON_EMPTY, l2_gas: 0, l1_data_gas: 0 }` as a valid positive case: [5](#0-4) 

**Secondary hash divergence** — `get_tip_resource_bounds_hash` includes `L1_DATA_GAS` in the Poseidon preimage only for `AllResources`, not for `L1Gas`. The proposer computes the hash with `AllResources` (three resource felts); if the validator were to recompute it after the downgrade it would use `L1Gas` (two resource felts), producing a different `tx_hash`. [6](#0-5) 

### Impact Explanation

A single unprivileged user transaction with `AllResourceBounds { l1_gas > 0, l2_gas = 0, l1_data_gas = 0 }` is accepted by the gateway, propagated to the mempool, and included by the proposer in a block proposal. Every validator that receives the proposal via P2P fails to deserialize the transaction and rejects the entire proposal. If the proposer consistently proposes blocks containing such transactions, consensus cannot finalize any block. This matches the **High** impact category: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

### Likelihood Explanation

The triggering condition is trivially reachable: submit any Invoke V3 transaction with only `l1_gas` non-zero and both `l2_gas` and `l1_data_gas` at their default zero values. No privileged access, no malformed bytes, and no special network position is required. The gateway's own test suite confirms this is a valid, accepted input.

### Recommendation

Remove the value-based heuristic. The protobuf schema already carries all three resource-bound fields; the deserializer should always produce `AllResources` when all three fields are present, regardless of whether their values are zero. The `L1Gas` variant should only be produced when the protobuf message genuinely omits `l2_gas` and `l1_data_gas` (i.e., `value.l2_gas.is_none() && value.l1_data_gas.is_none()`), which is the actual pre-0.13.3 wire format. The existing TODO comment (`// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2`) should be resolved by tying the variant selection to field presence, not field value. [7](#0-6) 

### Proof of Concept

```
1. Submit RpcInvokeTransactionV3 with:
       resource_bounds = AllResourceBounds {
           l1_gas:      ResourceBounds { max_amount: 1, max_price_per_unit: 1 },
           l2_gas:      ResourceBounds::default(),   // 0, 0
           l1_data_gas: ResourceBounds::default(),   // 0, 0
       }
   → Gateway accepts (matches the `valid_l1_gas` positive test case).

2. Proposer converts InternalRpcInvokeTransactionV3 → RpcInvokeTransactionV3
   → AllResourceBounds preserved; tx_hash computed with AllResources
     (three resource felts: L1_GAS, L2_GAS=0, L1_DATA_GAS=0).

3. Proposer serializes to protobuf::InvokeV3WithProof:
   From<ValidResourceBounds::AllResources> → protobuf::ResourceBounds {
       l1_gas: Some(1,1), l2_gas: Some(0,0), l1_data_gas: Some(0,0)
   }

4. Validator deserializes protobuf::ResourceBounds:
   l1_data_gas.is_zero() && l2_gas.is_zero()  →  ValidResourceBounds::L1Gas(l1_gas)
   ← WRONG VARIANT

5. snapi_invoke.try_into() (InvokeTransactionV3 → RpcInvokeTransactionV3) fails:
   resource_bounds is L1Gas, not AllResources → DEPRECATED_RESOURCE_BOUNDS_ERROR

6. validate_proposal returns HandledProposalPart::Failed(...)
   → Entire block proposal rejected by every validator.
```

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
