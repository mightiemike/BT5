### Title
AuxPoW Parent-Block PoW Validated Against Dogecoin Target Instead of Parent-Block Target — (File: `contract/src/dogecoin.rs`)

---

### Summary

`check_aux` validates the parent block's proof-of-work hash against the **Dogecoin block's** difficulty target (`block_header.bits`) rather than the **parent block's own** difficulty target (`aux_data.parent_block.bits`). Because Dogecoin's target is orders of magnitude easier than Bitcoin's, an attacker can forge a structurally valid AuxPoW submission backed by a parent block that never met Bitcoin's actual difficulty.

---

### Finding Description

In `check_aux`, the final PoW gate is:

```rust
let pow_hash = aux_data.parent_block.block_hash_pow();
require!(
    self.skip_pow_verification
        || U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
    format!("block should have correct pow")
);
``` [1](#0-0) 

`block_header` is the **Dogecoin** block being submitted; `block_header.bits` encodes the Dogecoin network's current difficulty. `aux_data.parent_block` is the Bitcoin (or other parent-chain) block whose hash must satisfy the parent chain's own, far-harder difficulty. The correct operand is `target_from_bits(aux_data.parent_block.bits)`.

Every other structural check in `check_aux` is performed correctly:

- The coinbase merkle proof is verified against `aux_data.parent_block.merkle_root`. [2](#0-1) 
- The chain merkle proof places the Dogecoin block hash inside the coinbase. [3](#0-2) 
- The merged-mining header position and `n_size`/`n_nonce` fields are validated. <cite repo="Oyahkilomeikhide/btc-light-client-contract--003" path="contract/src/dogecoin.rs" start="
