### Title
Missing Upper-Bound on `l2_gas.max_amount` for Declare Transactions Allows Unbounded Gas Bound Through Gateway Admission - (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces a `max_l2_gas_amount` ceiling on all incoming `RpcTransaction` types **except** `Declare`. A `Declare` transaction with `l2_gas.max_amount = u64::MAX` passes gateway admission unchanged, violating the invariant that the gateway rejects transactions whose declared resource bounds exceed the configured limit.

### Finding Description

In `validate_resource_bounds`, the upper-bound check on `l2_gas.max_amount` is explicitly skipped for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production config sets `max_l2_gas_amount = 1_210_000_000`: [2](#0-1) 

For `Invoke` and `DeployAccount` transactions the check fires correctly. For `Declare` transactions the branch is an empty no-op, so any value of `l2_gas.max_amount` — including `u64::MAX` — is accepted. The test `valid_l2_gas_amount_on_declare` explicitly confirms this bypass is reachable: [3](#0-2) 

The `l2_gas.max_amount` field is part of the transaction hash preimage via `get_tip_resource_bounds_hash`, so the admitted transaction carries the oversized value all the way into the signed hash and into the block: [4](#0-3) 

### Impact Explanation

Any unprivileged user can submit a `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = u64::MAX`. The gateway's `StatelessTransactionValidator` admits it without error. The transaction enters the mempool, is forwarded to the batcher, and is included in a block. The oversized bound is committed into the transaction hash and the block state diff. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

Trivially reachable: any caller of the public `add_transaction` HTTP endpoint can craft a `Declare` transaction with an arbitrarily large `l2_gas.max_amount`. No special privilege or knowledge is required. The gap is acknowledged in the source with a `TODO` comment, confirming it is a known unimplemented check rather than an intentional design decision.

### Recommendation

Apply the same upper-bound check to `Declare` transactions. Remove the early-return branch and let the existing check run unconditionally:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> … {
    …
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
            gas_amount: resource_bounds.l2_gas.max_amount,
            max_gas_amount: self.config.max_l2_gas_amount,
        });
    }
    Ok(())
}
```

This mirrors the existing enforcement for `Invoke` and `DeployAccount` and closes the admission-control gap for `Declare`.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and any non-zero `l2_gas.max_price_per_unit`.
2. Submit it to the gateway's `add_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for `Declare`.
4. The transaction is admitted to the mempool with `l2_gas.max_amount = 18_446_744_073_709_551_615`, far exceeding the configured limit of `1_210_000_000`.

The existing test already demonstrates step 3 with `max_amount: GasAmount(200)` against `max_l2_gas_amount: 100`: [3](#0-2)
