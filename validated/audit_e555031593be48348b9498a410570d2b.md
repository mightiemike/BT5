### Title
Off-by-One Cost Check in `op_keccak256` Allows Last Argument's Byte Cost to Bypass the Per-Operator Limit — (File: `src/keccak256_ops.rs`)

---

### Summary

`op_keccak256` accumulates `byte_count` **after** calling `check_cost`, so the cost check for each iteration uses the byte total from the **previous** iteration. The final argument's byte cost is added to `cost` after the loop exits, but no `check_cost` call follows. `new_atom_and_cost` also returns without checking against `max_cost`. An attacker can craft a keccak256 call whose last argument is arbitrarily large, forcing the node to hash that data before the cost limit is enforced.

---

### Finding Description

In `src/keccak256_ops.rs`, the loop body is:

```rust
while let Some((arg, rest)) = a.next(input) {
    input = rest;
    cost += KECCAK256_COST_PER_ARG;
    check_cost(
        cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,  // ← uses PREVIOUS byte_count
        max_cost,
    )?;
    let blob = atom(a, arg, "keccak256")?;
    byte_count += blob.as_ref().len();   // ← updated AFTER the check
    hasher.update(blob);
}
cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;
new_atom_and_cost(a, cost, &hasher.finalize())
``` [1](#0-0) 

On iteration N (the last argument), `check_cost` is called with `byte_count` = sum of bytes from arguments 1 … N-1. Argument N's bytes are then hashed and added to `byte_count`. After the loop, `cost += byte_count * KECCAK256_COST_PER_BYTE` incorporates argument N's bytes, but there is no subsequent `check_cost`. `new_atom_and_cost` simply returns `Ok(Reduction(cost + malloc_cost, node))` without any limit check. [2](#0-1) 

Compare the correct pattern used by `op_concat`, where the byte cost for the **current** argument is added to `cost` before `check_cost` is called: [3](#0-2) 

**Attacker-controlled entry path:**

- Keccak256 is reachable as opcode 62 when `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` is set (hard-fork flag), and inside the softfork guard via extension 1 (`OperatorSet::Keccak`) in all consensus modes. [4](#0-3) [5](#0-4) 

Any unprivileged user can submit a crafted CLVM blob to the Chia mempool. The trigger is a keccak256 call with N-1 small arguments (keeping the running cost just below `max_cost`) followed by one large argument whose byte cost is never checked inside the operator.

---

### Impact Explanation

The `Reduction` cost returned by `op_keccak256` can exceed `max_cost` by up to `last_arg_len × KECCAK256_COST_PER_BYTE` (= 2 cost units per byte). `run_program` will detect the overage when it adds the returned cost to the running total and will ultimately reject the transaction. However, the node has already performed the full SHA-3 hash of the oversized last argument before the limit is enforced. This is a bounded but real computational DoS: an attacker can force every validating node to hash up to one maximum-size atom (the allocator's atom-size ceiling) beyond the declared cost budget per keccak256 invocation, with no fee or cost credit for that work. [6](#0-5) 

---

### Likelihood Explanation

The trigger requires only a valid CLVM program submitted to the mempool — no privileged keys, no special roles. Keccak256 is available in consensus mode via softfork extension 1 and directly when the hard-fork flag is active. Constructing the exploit blob is trivial: encode a keccak256 call with a large trailing atom using the standard CLVM serializer.

---

### Recommendation

Move `byte_count += blob.as_ref().len()` **before** `check_cost` so the check always includes the current argument's bytes:

```rust
while let Some((arg, rest)) = a.next(input) {
    input = rest;
    cost += KECCAK256_COST_PER_ARG;
    let blob = atom(a, arg, "keccak256")?;
    byte_count += blob.as_ref().len();          // ← update first
    check_cost(
        cost + byte_count as Cost * KECCAK256_COST_PER_BYTE,
        max_cost,
    )?;
    hasher.update(blob);
}
cost += byte_count as Cost * KECCAK256_COST_PER_BYTE;
new_atom_and_cost(a, cost, &hasher.finalize())
```

Alternatively, add `check_cost(cost, max_cost)?` immediately after the loop, before `new_atom_and_cost`. Also audit `op_sha256` in `src/more_ops.rs` for the same pattern, since it uses identical `SHA256_COST_PER_ARG` / `SHA256_COST_PER_BYTE` constants. [7](#0-6)

### Proof of Concept

```
; CLVM pseudocode
(keccak256
  0x00          ; arg 1: 1 byte  — cost check passes (byte_count=0)
  0x00          ; arg 2: 1 byte  — cost check passes (byte_count=1)
  ...           ; arg N-1: 1 byte — cost check passes (byte_count=N-2)
  <1_048_576-byte atom>   ; arg N: 1 MiB — check uses byte_count=N-1, NOT 1 MiB
                           ; node hashes 1 MiB; 2 097 152 cost units added post-loop
                           ; run_program detects overage only after hashing completes
)
```

Serialise with the legacy or 2026 format, submit to the Chia mempool. Every full node evaluating this spend hashes 1 MiB of attacker-supplied data before the cost limit is enforced, with zero cost charged to the attacker for that work.
