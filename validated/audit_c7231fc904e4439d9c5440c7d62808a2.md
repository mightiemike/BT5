### Title
Unreset Token Allowance in `DirectDepositV1.creditDeposit()` After Early Return in `depositCollateralWithReferral` - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` before invoking `endpoint.depositCollateralWithReferral()`. Inside `depositCollateralWithReferral`, there is a documented early-return path that skips the actual `transferFrom` when the deposit amount is below the minimum threshold. When this early return fires, the allowance granted to the endpoint is never consumed and is never reset to zero. The `DirectDepositV1` contract retains its full token balance while the endpoint holds a live, non-zero allowance over those tokens.

---

### Finding Description

In `DirectDepositV1.creditDeposit()`:

```solidity
// DirectDepositV1.sol lines 91-99
if (balance != 0) {
    token.approve(address(endpoint), balance);          // allowance set
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),                               // only uint128 passed
        "-1"
    );
    // ← no token.approve(address(endpoint), 0) here
}
```

Inside `Endpoint.depositCollateralWithReferral()`:

```solidity
// Endpoint.sol lines 137-142
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked...
    return;   // ← early return, handleDepositTransfer is never called
}
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
```

When `isValidDepositAmount` returns `false` (deposit below minimum, or oracle price ≤ 0), `handleDepositTransfer` is skipped entirely. The tokens remain in the `DirectDepositV1` contract, but the endpoint now holds an allowance of `balance` over those tokens with no code path in `creditDeposit()` to reset it to zero.

A second, independent trigger exists: `balance` is approved as a `uint256` but only `uint128(balance)` is deposited. If `balance > type(uint128).max`, the endpoint pulls only the truncated amount and the residual allowance (`balance - uint128(balance)`) is never cleared.

---

### Impact Explanation

The `DirectDepositV1` contract is a per-subaccount escrow that holds user collateral tokens in transit. After the early-return path fires, the endpoint retains a non-zero ERC-20 allowance over the DDA's full token balance. Any future exploit of the endpoint contract (reentrancy, arbitrary-call gadget, malicious slow-mode transaction, or any other endpoint-level vulnerability) can use this dangling allowance to call `transferFrom(DDA, attacker, balance)` and drain the DDA's tokens — tokens that belong to the user whose subaccount the DDA was created for. Because `creditDeposit()` is permissionlessly callable by anyone, an attacker can deliberately manufacture the early-return condition (by sending a dust amount below the minimum deposit threshold) to plant the dangling allowance before a larger deposit arrives.

---

### Likelihood Explanation

- `creditDeposit()` has no access control (`external`, no modifier); any address can call it at any time.
- `ContractOwner.creditDepositV1()` is also `external` with no access control and calls `creditDeposit()` on the DDA.
- The early-return condition in `depositCollateralWithReferral` is explicitly designed to be reachable (the comment says "we cannot revert here") and fires whenever the deposit amount is below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`, or when the oracle price is ≤ 0.
- An attacker can reliably trigger this by sending a sub-minimum token amount to the DDA and calling `creditDeposit()`, leaving a dangling allowance before the legitimate user's larger deposit arrives.

---

### Recommendation

Reset the allowance to zero after `depositCollateralWithReferral` returns, regardless of whether the deposit succeeded:

```solidity
if (balance != 0) {
    token.approve(address(endpoint), balance);
    endpoint.depositCollateralWithReferral(
        subaccount,
        productId,
        uint128(balance),
        "-1"
    );
    token.approve(address(endpoint), 0);   // ← add this
}
```

Alternatively, mirror the fix from the referenced report and only set the allowance after confirming the deposit will proceed (i.e., after an explicit `isValidDepositAmount` pre-check inside `creditDeposit()` itself), so the allowance is never granted when the early-return path would be taken.

---

### Proof of Concept

1. Attacker identifies a `DirectDepositV1` DDA for a target subaccount (address is public via `ContractOwner.directDepositV1Address`).
2. Attacker sends 1 wei of a supported ERC-20 token to the DDA (below `MIN_DEPOSIT_AMOUNT`).
3. Attacker calls `ContractOwner.creditDepositV1(targetSubaccount)` (no access control).
4. `creditDeposit()` executes: `token.approve(endpoint, 1)` is called; `depositCollateralWithReferral` fires the early return at line 141 of `Endpoint.sol`; no `transferFrom` occurs; no allowance reset occurs.
5. The endpoint now holds `allowance(DDA, endpoint) == 1`.
6. The legitimate user later sends a large deposit to the DDA and calls `creditDeposit()` again — this overwrites the allowance with the new balance and the deposit succeeds. **However**, between steps 4 and 6, if the endpoint is exploited in any way that allows an arbitrary `transferFrom` call, the attacker can drain the DDA's tokens using the live allowance.
7. For the `uint128` truncation variant: if `balance > 2^128 - 1`, the endpoint is approved for the full `uint256 balance` but only pulls `uint128(balance)` tokens; the residual allowance `balance - uint128(balance)` is never cleared and persists indefinitely.

```text
    function depositCollateralWithReferral(
        bytes32 subaccount,
        uint32 productId,
        uint128 amount,
        string memory
    ) public {
        require(!RiskHelper.isIsolatedSubaccount(subaccount), ERR_UNAUTHORIZED);

        address sender = address(bytes20(subaccount));

        // depositor / depositee need to be unsanctioned
        requireUnsanctioned(msg.sender);
        requireUnsanctioned(sender);

        if (!isValidDepositAmount(subaccount, productId, amount)) {
            // we cannot revert here, otherwise direct deposit could be blocked when there are
            // multiple assets awaiting credit but one of them is below the minimum deposit amount.
            // we can just skip the deposit and continue with the next asset.
            return;
        }

        handleDepositTransfer(
            IERC20Base(spotEngine.getToken(productId)),
            msg.sender,
            uint256(amount)
        );
```
