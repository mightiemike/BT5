### Title
Hardcoded `softfork_kw` in `RuntimeDialect` Prevents Configurable Operator Wiring - (File: `src/runtime_dialect.rs`)

### Summary
`RuntimeDialect::new()` accepts `quote_kw` and `apply_kw` as caller-supplied parameters but hardcodes `softfork_kw` to `vec![36]`. Because `run_program.rs` intercepts opcode 36 before dispatching to `dialect.op()`, any attacker-controlled CLVM program that emits opcode 36 will always trigger the softfork execution path — even when a caller has mapped opcode 36 to a custom function in `op_map`. The custom operator is silently bypassed, and the softfork argument parser runs instead.

### Finding Description
`RuntimeDialect` is the configurable dialect implementation in `src/runtime_dialect.rs`. Its constructor signature is:

```rust
pub fn new(
    op_map: HashMap<String, Vec<u8>>,
    quote_kw: Vec<u8>,
    apply_kw: Vec<u8>,
    flags: ClvmFlags,
) -> RuntimeDialect {
    RuntimeDialect {
        f_lookup: f_lookup_for_hashmap(op_map),
        quote_kw,
        apply_kw,
        softfork_kw: vec![36], // softfork opcode  ← hardcoded
        flags,
    }
}
``` [1](#0-0) 

`quote_kw` and `apply_kw` are parameterized, but `softfork_kw` is hardcoded to `36`. The `Dialect` trait exposes all three as overridable methods: [2](#0-1) 

In `run_program.rs`, the interpreter checks `softfork_kw()` before dispatching to `dialect.op()`:

```rust
if op_atom == Some(self.dialect.apply_kw()) {
    ...
} else if op_atom == Some(self.dialect.softfork_kw()) {
    // softfork path — intercepts opcode 36 unconditionally
    ...
} else {
    let r = self.dialect.op(...)?;  // custom op never reached for opcode 36
    ...
}
``` [3](#0-2) 

Because `RuntimeDialect::softfork_kw()` always returns `36`, any CLVM program byte `0x24` (decimal 36) will be routed to the softfork handler regardless of what the caller placed in `op_map` for that slot. The `RuntimeDialect::op()` method is never reached for opcode 36. [4](#0-3) 

### Impact Explanation
The corrupted result is **operator dispatch**: a caller who registers opcode 36 in `op_map` will have that function silently bypassed. Instead, `parse_softfork_arguments` runs, expecting exactly 4 arguments `(cost, extension, program, env)`. If the custom operator has a different arity or argument types, the program either raises `InvalidOperatorArg: softfork takes exactly 4 argument(s)` or, in consensus (lenient) mode, silently returns nil and charges the declared softfork cost — producing a wrong result with no error. This is an **operator wiring error** causing API-equivalence divergence between `RuntimeDialect`-based and `ChiaDialect`-based execution of the same CLVM bytes. [5](#0-4) 

### Likelihood Explanation
`RuntimeDialect` is a production module exposed in `src/lib.rs` and used by Python bindings and tools. Any caller who builds a custom dialect via `RuntimeDialect::new()` and places an operator at opcode 36 in `op_map` will encounter this silently. The inconsistency is subtle: `quote_kw` and `apply_kw` are correctly parameterized, so a developer reasonably expects `softfork_kw` to be similarly configurable. Attacker-controlled CLVM bytes containing `0x24` as an opcode are the direct trigger.

### Recommendation
Add `softfork_kw` as a parameter to `RuntimeDialect::new()`, mirroring the treatment of `quote_kw` and `apply_kw`:

```rust
pub fn new(
    op_map: HashMap<String, Vec<u8>>,
    quote_kw: Vec<u8>,
    apply_kw: Vec<u8>,
    softfork_kw: Vec<u8>,   // ← add this
    flags: ClvmFlags,
) -> RuntimeDialect {
    RuntimeDialect {
        f_lookup: f_lookup_for_hashmap(op_map),
        quote_kw,
        apply_kw,
        softfork_kw,          // ← use parameter
        flags,
    }
}
```

Callers that want the standard Chia softfork opcode pass `vec![36]`; callers building alternative dialects can supply a different value or an empty vec to disable softfork interception entirely. [6](#0-5) 

### Proof of Concept
```rust
use std::collections::HashMap;
use clvmr::runtime_dialect::RuntimeDialect;
use clvmr::chia_dialect::ClvmFlags;
use clvmr::allocator::Allocator;
use clvmr::run_program::run_program;
use clvmr::serde::node_from_bytes;

// Register a custom no-op at opcode 36
let mut op_map = HashMap::new();
op_map.insert("custom_op".to_string(), vec![36u8]);

let dialect = RuntimeDialect::new(
    op_map,
    vec![1],  // quote
    vec![2],  // apply
    ClvmFlags::empty(),
);

let mut a = Allocator::new();
// CLVM program: (36 (q . 0x1234))  — invoke opcode 36 with one atom arg
// Expected: custom_op runs, returns some value
// Actual: softfork parser runs, fails with "softfork takes exactly 4 argument(s)"
//         or silently returns nil in lenient mode
let program_bytes = hex::decode("ff24ffff0182123480").unwrap();
let program = node_from_bytes(&mut a, &program_bytes).unwrap();
let result = run_program(&mut a, &dialect, program, a.nil(), 1_000_000);
// result is Err("InvalidOperatorArg: softfork takes exactly 4 argument(s)")
// instead of the custom operator's output
```
