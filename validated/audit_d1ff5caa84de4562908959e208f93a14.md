Audit Report

## Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Unprivileged Address to Bypass the Deposit Guard ‚Äî (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner` ‚Äî a free caller-supplied parameter. Because any caller can set `owner` to any allowlisted address, the deposit allowlist provides zero restriction on who can actually deposit tokens into the pool.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as separate arguments to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension unchanged: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops the first parameter (`sender`) and checks only `owner`: [3](#0-2) 

The contract's own NatSpec states it *"Gates `addLiquidity` by depositor address, per pool"* ‚Äî the depositor is `sender` (`msg.sender` of the pool call), not `owner`. The `owner` parameter is a free input that any caller can set to any address, including any known allowlisted address visible on-chain.

The correct pattern is already implemented in `SwapAllowlistExtension.beforeSwap`, which checks `sender` (the actual swapper) and ignores `recipient`: [4](#0-3) 

Additionally, `removeLiquidity` enforces `msg.sender == owner` at line 206, meaning the position shares credited to `owner` can only be withdrawn by `owner` themselves ‚Äî confirming that `owner` is a position-ownership parameter, not an identity check on the depositor. [5](#0-4) 

## Impact Explanation
The deposit allowlist is completely ineffective. Any non-allowlisted address can call `pool.addLiquidity(owner = <any_allowlisted_address>, ...)`, pass the guard, and have their tokens pulled via the `metricOmmModifyLiquidityCallback`. If the allowlist enforces KYC, compliance, or LP-set membership, all such controls are silently bypassed. The attacker can also manipulate pool state (shift `curBinIdx`, affect fee accrual, dilute existing LPs) without being on the allowlist. This constitutes broken core pool functionality and a bypass of an admin-configured access control boundary. [6](#0-5) 

## Likelihood Explanation
Exploitation requires only a single `addLiquidity` call with `owner` set to any known allowlisted address (e.g., the pool admin, a known LP, or any address visible on-chain via past events). No special privileges, flash loans, or multi-step setup are needed. Any address can execute this at any time the pool is active with the extension configured.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual depositor) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [3](#0-2) 

## Proof of Concept
1. Pool is deployed with `DepositAllowlistExtension` configured on `beforeAddLiquidity`.
2. Pool admin allowlists only `Bob` via `setAllowedToDeposit(pool, Bob, true)`.
3. `Alice` (not allowlisted) calls `pool.addLiquidity(owner = Bob, salt, deltas, callbackData, extensionData)`.
4. Pool calls `_beforeAddLiquidity(msg.sender=Alice, owner=Bob, ...)`.
5. Extension receives `(sender=Alice, owner=Bob)`, ignores `Alice`, checks `allowedDepositor[pool][Bob]` ‚Üí `true`.
6. Guard passes. Alice's tokens are pulled via `metricOmmModifyLiquidityCallback`. Bob receives the position shares.
7. Alice has deposited into a restricted pool without being on the allowlist.
8. If Alice and Bob cooperate (e.g., Bob is Alice's second address), Bob calls `removeLiquidity` to return the tokens, completing a full round-trip through a pool restricted to allowlisted depositors only. [7](#0-6)
