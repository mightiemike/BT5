### Title
Inaccurate Spread Health Calculation Due to Hardcoded Long Weight in `getCoreRisk` - (File: `core/contracts/BaseEngine.sol`)

---

### Summary

`BaseEngine.getCoreRisk()` always computes the position weight using a hardcoded positive sentinel value (`1`) instead of the actual position `amount`. This causes the function to always return the **long weight** regardless of whether the position is long or short. The returned `CoreRisk.longWeight` is then consumed by `Clearinghouse.getHealth()` and `RiskHelper._getSpreadWeightX18()` to compute the spread health adjustment. Because the short leg of a spread always receives the long weight instead of the short weight, the spread health bonus is materially overstated, allowing spread traders to carry more leverage than the protocol's risk parameters permit.

---

### Finding Description

In `BaseEngine.getCoreRisk()`, the weight field of the returned `CoreRisk` struct is computed by calling `RiskHelper._getWeightX18(risk, 1, healthType)` with a hardcoded `1` instead of the actual `amount`:

```solidity
// BaseEngine.sol lines 179-192
function getCoreRisk(
    bytes32 subaccount,
    uint32 productId,
    IProductEngine.HealthType healthType
) external returns (IProductEngine.CoreRisk memory) {
    RiskHelper.Risk memory risk = _risk(productId);
    (int128 amount, ) = _getBalance(productId, subaccount);
    return
        IProductEngine.CoreRisk(
            amount,
            risk.priceX18,
            RiskHelper._getWeightX18(risk, 1, healthType)  // ← always 1, never `amount`
        );
}
```

`_getWeightX18` branches on the sign of its second argument:

```solidity
// RiskHelper.sol lines 34-55
function _getWeightX18(Risk memory risk, int128 amount, ...) internal pure returns (int128) {
    int128 weight;
    if (amount >= 0) {
        weight = healthType == INITIAL ? risk.longWeightInitialX18 : risk.longWeightMaintenanceX18;
    } else {
        weight = healthType == INITIAL ? risk.shortWeightInitialX18 : risk.shortWeightMaintenanceX18;
    }
    return weight;
}
```

Because `1 >= 0` is always true, `getCoreRisk` always returns the **long weight**, even when `amount < 0` (a short position).

`Clearinghouse.getHealth()` calls `getCoreRisk` for both legs of every registered spread pair and feeds the results into the spread health adjustment:

```solidity
// Clearinghouse.sol lines 93-135
IProductEngine.CoreRisk memory perpCoreRisk = perpEngine.getCoreRisk(subaccount, _perpId, healthType);
IProductEngine.CoreRisk memory spotCoreRisk = spotEngine.getCoreRisk(subaccount, _spotId, healthType);
...
int128 existingWeight = (spotCoreRisk.longWeight + perpCoreRisk.longWeight) / 2;
int128 spreadWeight = RiskHelper._getSpreadWeightX18(perpCoreRisk, spotCoreRisk, healthType);
health += basisAmount.mul(spotCoreRisk.price + perpCoreRisk.price).mul(spreadWeight - existingWeight);
```

`_getSpreadWeightX18` also reads `perpCoreRisk.longWeight` for the short perp leg:

```solidity
// RiskHelper.sol lines 57-81
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;  // ← uses long weight for short perp
} else {
    spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;  // ← uses long weight for short spot
}
```

**Concrete numerical example** (long spot + short perp spread, typical risk params):

| Parameter | Correct (short weight for perp) | Actual (long weight for perp) |
|---|---|---|
| `perpWeight` | `1.1e18` (short) | `0.9e18` (long) |
| `existingWeight` | `(0.9 + 1.1)/2 = 1.0` | `(0.9 + 0.9)/2 = 0.9` |
| `spreadWeight` | `1 - (1 - 1.1)/5 = 1.02` | `1 - (1 - 0.9)/5 = 0.98` |
| `spreadWeight - existingWeight` | `0.02` | `0.08` |
| **Health bonus multiplier** | **1×** | **4×** |

The spread health bonus is inflated by **4×** relative to the protocol's intended risk parameters.

---

### Impact Explanation

The `getHealth()` function is the central solvency gate for every user action: withdrawals, transfers, and liquidation eligibility all depend on it. An inflated spread health bonus means:

1. **Excessive leverage**: A spread trader can withdraw collateral or open additional positions that would be blocked under correct health accounting, because their health reads as higher than it truly is.
2. **Liquidation bypass**: A spread position that is genuinely under-maintenance health may pass the `isUnderMaintenance` check and avoid liquidation, leaving the protocol exposed to a growing bad debt.
3. **Insurance fund drain**: When the position eventually becomes insolvent, the shortfall is socialized across other participants or drawn from the insurance fund, directly corrupting protocol solvency accounting.

The corrupted state delta is `health` in `Clearinghouse.getHealth()`, which propagates to every downstream check (`withdrawCollateral`, `transferQuote`, `liquidateSubaccountImpl`, `mintNlp`, `burnNlp`).

---

### Likelihood Explanation

Any user who holds a registered spread pair (long spot + short perp, or short spot + long perp) triggers this path on every health check. Spread trading is a core, documented feature of the protocol. No special permissions or privileged access are required. The entry path is:

1. Trader opens a spread position via `submitTransactions` → `OffchainExchange` → `PerpEngine`/`SpotEngine`.
2. Any subsequent action that calls `Clearinghouse.getHealth()` (withdrawal, transfer, liquidation check) will use the inflated spread weight.
3. A sophisticated trader can deliberately size the spread to maximize the health bonus and extract collateral beyond what the risk parameters allow.

---

### Recommendation

In `BaseEngine.getCoreRisk()`, pass the actual `amount` to `_getWeightX18` instead of the hardcoded `1`:

```solidity
// Before (incorrect):
RiskHelper._getWeightX18(risk, 1, healthType)

// After (correct):
RiskHelper._getWeightX18(risk, amount, healthType)
```

This ensures that short positions receive `shortWeightInitialX18` / `shortWeightMaintenanceX18`, matching the intent of the risk parameter system and producing accurate spread health adjustments.

---

### Proof of Concept

1. Deploy the protocol with a registered spread pair (e.g., spot product ID `1`, perp product ID `2`) with `longWeightInitialX18 = 0.9e18` and `shortWeightInitialX18 = 1.1e18`.
2. Open a spread: deposit collateral, go long spot product `1` and short perp product `2` with `basisAmount = 100e18` at price `1e18` each.
3. Call `Clearinghouse.getHealth(subaccount, INITIAL)`.
4. Observe: `existingWeight = 0.9`, `spreadWeight = 0.98`, delta = `0.08`. Health bonus = `100 * 2 * 0.08 = 16`.
5. Correct calculation: `existingWeight = 1.0`, `spreadWeight = 1.02`, delta = `0.02`. Health bonus should be `100 * 2 * 0.02 = 4`.
6. The inflated health of `+12` allows the trader to withdraw `12` units of quote collateral that should be locked, or to open additional positions beyond the protocol's intended leverage limits.

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
