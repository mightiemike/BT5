### Title
`verify_transaction_inclusion` (v1) Returns `true` for Internal Node Hash via Second-Preimage Attack — (`contract/src/lib.rs`)

### Summary
The deprecated-but-still-callable `verify_transaction_inclusion` function passes attacker-controlled `tx_id` and `tx_index` directly to `compute_root_from_merkle_proof` with no validation that `tx_id` is a leaf hash or that `tx_index` is within tree bounds. An attacker can supply an internal Merkle node hash as `tx_id` with a shortened proof and make the function return `true` for a transaction that does not exist.

### Finding Description
`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` iterates over proof hashes using only `current_position % 2` for left/right placement and `current_position /= 2` to advance levels. [1](#0-0) 

It performs no validation that:
1. `transaction_hash` is a leaf-level hash
2. `transaction_position` is within the valid range for the tree

`verify_transaction_inclusion` passes attacker-supplied `args.tx_id` and `args.tx_index` directly to this function: [2](#0-1) 

The only guard against an empty proof is `require!(!args.merkle_proof.is_empty())`: [3](#0-2) 

For a 3-tx block (leaves `[tx0, tx1, tx2]`, root = `H(H(tx0,tx1), H(tx2,tx2))`), the attacker submits:
- `tx_id = H(tx2,tx2)` (level-1 internal node)
- `tx_index = 3` (out of bounds; aliases to level-1 position 1 after one `/= 2` step)
- `proof = [H(tx0,tx1)]`

`compute_root_from_merkle_proof` computes `compute_hash(H(tx0,tx1), H(tx2,tx2))` = root, and the function returns `true`.

The same result is achieved with `tx_index = 1` (the correct level-1 position), confirming the root cause is the missing leaf-hash validation, not the out-of-bounds index alone.

The function is `pub` and not `#[private]`, so it is directly callable by any NEAR account: [4](#0-3) 

The codebase itself acknowledges this in `CLAUDE.md`: [5](#0-4) 

And in the function's own docstring: [6](#0-5) 

The function is deprecated since v0.5.0 but has not been removed or restricted to `#[private]`.

### Impact Explanation
Any external caller still invoking `verify_transaction_inclusion` (v1) can be presented with a forged inclusion proof for a non-existent transaction. The function returns `true`, enabling the 64-byte transaction forgery attack that `verify_transaction_inclusion_v2` was designed to prevent. Downstream systems that rely on the v1 return value to authorize actions (e.g., bridge unlocks, payment confirmations) can be deceived into accepting a fabricated transaction.

### Likelihood Explanation
The attack requires only public blockchain data: the transaction hashes of any block already stored in the contract's `headers_pool`. No privileged access, key compromise, or social engineering is needed. The v1 function remains callable by any NEAR account. The construction is straightforward and well-documented in Bitcoin security literature.

### Recommendation
- Remove `verify_transaction_inclusion` (v1) entirely from the contract, or add `#[private]` to prevent external calls.
- All callers must migrate to `verify_transaction_inclusion_v2`, which enforces a coinbase merkle proof of equal depth, preventing proof-shortening attacks. [7](#0-6) 

### Proof of Concept
```rust
// 3-tx tree
let tx0 = H256::from(...);
let tx1 = H256::from(...);
let tx2 = H256::from(...);

let internal_01 = compute_hash(&tx0, &tx1);       // H(tx0,tx1)
let internal_22 = compute_hash(&tx2, &tx2);       // H(tx2,tx2)  ← attacker's tx_id
let root        = compute_hash(&internal_01, &internal_22);

// Index-aliasing variant
let result = compute_root_from_merkle_proof(
    internal_22,        // internal node, not a leaf
    3,                  // out-of-bounds index (2k+1 for k=1)
    &vec![internal_01], // proof of length 1
);
assert_eq!(result, root); // passes — verify_transaction_inclusion returns true

// Standard second-preimage (no aliasing needed, same outcome)
let result2 = compute_root_from_merkle_proof(internal_22, 1, &vec![internal_01]);
assert_eq!(result2, root); // also passes
```

Both variants return `true` from `verify_transaction_inclusion` for `tx_id = H(tx2,tx2)`, which is not a real transaction hash.
