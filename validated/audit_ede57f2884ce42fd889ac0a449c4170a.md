### Title
Cross-Prefix Address Acceptance Permanently Locks Position Funds — (`x/tieredrewards/types/msgs.go`, `x/tieredrewards/types/position.go`)

---

### Summary

`MsgLockTier.Validate()` calls `sdk.AccAddressFromBech32` which accepts **any** bech32 prefix (including `cosmos1`), not just the chain's configured `cro1` prefix. The ante handler's `GetSigners` extracts the same raw bytes regardless of prefix, so signature verification passes. The keeper then stores the verbatim `cosmos1`-prefixed string as `Position.Owner`. All subsequent owner-gated operations (`TierUndelegate`, `TriggerExitFromTier`, `WithdrawFromTier`, `ClaimTierRewards`, etc.) call `IsOwner`, which is a **pure string equality check** — `p.Owner == address`. A user who submits `MsgLockTier` with a `cosmos1`-prefixed owner (same raw bytes as their `cro1` key) will have their funds permanently locked with no recovery path.

---

### Finding Description

**Step 1 — Validate() accepts any prefix** [1](#0-0) 

`sdk.AccAddressFromBech32` decodes any valid bech32 string regardless of the human-readable part. On a chain configured with `cro` prefix, `cosmos1abc...` and `cro1abc...` encoding the same 20 raw bytes both pass this check without error.

**Step 2 — GetSigners returns raw bytes (prefix-agnostic)**

The proto annotation `option (cosmos.msg.v1.signer) = "owner"` causes the SDK's message router to call `sdk.AccAddressFromBech32(msg.Owner)` to derive the required signer bytes. Since both `cosmos1abc...` and `cro1abc...` decode to the same bytes, the ante handler accepts a transaction signed by the `cro1abc...` key even when `msg.Owner = "cosmos1abc..."`.

**Step 3 — Owner string stored verbatim** [2](#0-1) 

`createDelegatedPosition` passes `owner` (the raw `msg.Owner` string) directly to `NewPosition`. The `cosmos1`-prefixed string is persisted as `Position.Owner`.

**Step 4 — IsOwner is a string comparison** [3](#0-2) 

Every owner-gated keeper validation calls `pos.IsOwner(owner)`: [4](#0-3) [5](#0-4) 

When the user later submits `MsgTierUndelegate` with `owner = "cro1abc..."`, the comparison `"cosmos1abc..." == "cro1abc..."` is `false`, returning `ErrNotPositionOwner`. The same failure applies to every other position operation.

**Step 5 — PositionsByOwner index uses raw bytes (inconsistency)** [6](#0-5) 

The secondary index is keyed by raw bytes (via `sdk.AccAddressFromBech32`), so the position IS discoverable by the correct owner via queries — but all mutating operations are blocked by the string-equality `IsOwner` guard.

**Chain prefix is `cro`:** [7](#0-6) 

---

### Impact Explanation

Any tokens locked via a `MsgLockTier` with a wrong-prefix owner string are permanently frozen:
- The delegation exists in `x/staking` under the position's delegator account
- The position record exists in state
- No message (`TierUndelegate`, `TriggerExitFromTier`, `WithdrawFromTier`, `ClaimTierRewards`, `AddToTierPosition`, `ClearPosition`) can be executed by the real key owner because all paths go through `IsOwner` string comparison
- The funds cannot be recovered without a governance-level force-exit or chain upgrade

The concrete state delta: locked tokens remain delegated under the position's derived delegator address indefinitely, with no authorized party able to initiate exit.

---

### Likelihood Explanation

The normal CLI path (`clientCtx.GetFromAddress().String()`) produces the chain-native `cro1` prefix and is safe: [8](#0-7) 

However, the vulnerability is reachable via:
- Direct gRPC/REST API calls with a manually constructed message (e.g., from a cosmos-ecosystem wallet or SDK that defaults to `cosmos1` prefix)
- Programmatic usage of the Go SDK without calling `config.SetConfig()` first (the SDK defaults to `cosmos` prefix before `SetConfig` is called)
- Cross-chain tooling or scripts that use the generic `cosmos1` prefix

The risk is elevated because `sdk.AccAddressFromBech32` accepting any prefix is a well-known footgun in the Cosmos ecosystem, and the chain's own `config.SetConfig()` must be explicitly called to enforce the `cro` prefix globally.

---

### Recommendation

1. **Enforce prefix in `Validate()`**: After decoding, re-encode the address and compare to the input string, or use a helper that checks the human-readable part matches the chain's configured prefix.
2. **Normalize owner on write**: In `createDelegatedPosition`, convert the decoded `sdk.AccAddress` back to string using `.String()` (which uses the chain's configured prefix) before storing, ensuring the stored string always uses the canonical prefix.
3. **Fix `IsOwner` to compare bytes**: Change `IsOwner` to decode both strings and compare raw bytes, making it prefix-agnostic:
   ```go
   func (p Position) IsOwner(addr string) bool {
       a1, err1 := sdk.AccAddressFromBech32(p.Owner)
       a2, err2 := sdk.AccAddressFromBech32(addr)
       return err1 == nil && err2 == nil && a1.Equals(a2)
   }
   ```

### Proof of Concept

```go
// 1. Generate a key; its cro1 address is cro1<bytes>
privKey := secp256k1.GenPrivKey()
rawAddr := sdk.AccAddress(privKey.PubKey().Address())

// 2. Encode with cosmos prefix instead of cro prefix
cosmosAddr, _ := bech32.ConvertAndEncode("cosmos", rawAddr)

// 3. Fund the rawAddr (cro1<bytes>) and submit MsgLockTier with cosmos1 owner
msg := &types.MsgLockTier{
    Owner:            cosmosAddr,   // "cosmos1<bytes>"
    Id:               1,
    Amount:           sdkmath.NewInt(1000),
    ValidatorAddress: valAddr.String(),
}
// msg.Validate() passes — sdk.AccAddressFromBech32 accepts cosmos1 prefix
// GetSigners returns rawAddr bytes — tx signed by cro1<bytes> key passes ante

resp, err := msgServer.LockTier(ctx, msg)
// err == nil, position created with Owner = "cosmos1<bytes>"

// 4. Attempt to undelegate using the correct cro1 address
_, err = msgServer.TierUndelegate(ctx, &types.MsgTierUndelegate{
    Owner:      rawAddr.String(),   // "cro1<bytes>"
    PositionId: resp.PositionId,
})
// err == types.ErrNotPositionOwner  ← funds permanently locked
```
