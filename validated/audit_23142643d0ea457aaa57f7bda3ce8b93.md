Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Allowing Any User to Bypass Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the pool's `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's. A pool admin who allowlists the router (required for router-mediated swaps to function) inadvertently grants every user — including non-allowlisted ones — the ability to bypass the swap gate by calling through the router.

## Finding Description

**Step 1 — Pool passes its own `msg.sender` as `sender`:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap(msg.sender, ...)`. When the router calls `pool.swap(...)`, `msg.sender` inside the pool is the router's address. [1](#0-0) 

**Step 2 — ExtensionCalling forwards `sender` unchanged:**

`ExtensionCalling._beforeSwap` encodes and forwards the `sender` value to every configured extension without modification. [2](#0-1) 

**Step 3 — SwapAllowlistExtension checks `sender` (the router), not the end user:**

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — the router, not the end user. The `extensionData` parameter is unnamed and entirely ignored. [3](#0-2) 

**Step 4 — Router stores the actual user in transient storage, never exposes it to extensions:**

`MetricOmmSimpleRouter.exactInputSingle` stores `msg.sender` (the real user) as the payer in transient storage via `_setNextCallbackContext`, but passes `params.extensionData` (user-supplied, not the real user's address) to the pool. The extension layer never receives the actual end user's identity. [4](#0-3) 

**Why existing checks fail:** The `allowAllSwappers` flag and `allowedSwapper` mapping both operate on the `sender` argument. There is no mechanism in `SwapAllowlistExtension` to decode the real user from `extensionData`, and the router never encodes it there. A direct call to `pool.swap()` by an unauthorized user correctly reverts, but the same user routed through the router passes the check because the router is allowlisted.

## Impact Explanation

A pool admin deploying a curated pool (KYC-only, institutional-only, etc.) with `SwapAllowlistExtension` must allowlist the router for their allowlisted users to use the standard periphery. This single operational step silently opens the pool to all users. Any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` or `exactInput` targeting the restricted pool and execute swaps against LP positions that were provisioned under the assumption of a restricted counterparty set. LP funds are directly at risk from counterparties the pool was designed to exclude — a broken core access-control invariant with direct fund impact.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard, documented periphery entry point. Allowlisting the router is the natural and expected operational step for any pool admin who wants their allowlisted users to use the router. The bypass requires no special privileges, no flash loans, and no multi-step setup — any user simply calls the public router with the restricted pool address. The precondition (router allowlisted) is triggered by normal, non-adversarial pool configuration.

## Recommendation

The extension must gate on the economically relevant actor, not the immediate caller of `pool.swap`. The cleanest fix is to have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` under a known prefix/selector, and have `SwapAllowlistExtension.beforeSwap` decode and check it when present, falling back to `sender` for direct pool calls. This requires a convention between router and extension but does not require any core pool changes.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][alice] = true   // alice is KYC'd
  allowedSwapper[pool][router] = true  // router allowlisted so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({
      pool: restrictedPool,
      tokenIn: token0,
      tokenOut: token1,
      ...
    })

  MetricOmmPool.swap(msg.sender=router) fires _beforeSwap(sender=router)
  SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  Bob's swap executes successfully despite not being on the allowlist.

Direct call check (for comparison):
  bob calls pool.swap() directly
  _beforeSwap(sender=bob)
  allowedSwapper[pool][bob] → false → revert NotAllowedToSwap ✓

Conclusion: the allowlist is enforced for direct calls but silently bypassed
for all router-mediated calls once the router is allowlisted.
```
