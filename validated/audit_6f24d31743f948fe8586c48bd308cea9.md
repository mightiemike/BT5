Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the end-user, enabling any unprivileged caller to bypass the swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the end-user. Any unprivileged user can therefore bypass the allowlist by calling the public router, provided the router itself is allowlisted — which is the only configuration that allows allowlisted users to use the router at all.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← router address when called via router
    ...
```

`ExtensionCalling._beforeSwap` forwards that value unchanged via `abi.encodeCall` to every configured extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L162-165
abi.encodeCall(
    IMetricOmmExtensions.beforeSwap,
    (sender, ...)
)
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` is used, the router calls `pool.swap` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The pool receives `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

Two mutually exclusive failure modes result:

| Scenario | Outcome |
|---|---|
| Pool admin does **not** allowlist the router | Allowlisted users cannot trade through the router (broken core flow) |
| Pool admin **does** allowlist the router | Any unprivileged user bypasses the allowlist by routing through the router |

No configuration simultaneously allows allowlisted users to use the router and blocks non-allowlisted users. Existing guards are insufficient because the extension has no access to the original `msg.sender` of the router call — the pool architecture does not propagate it.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled accounts) can be bypassed by any public user calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. The bypass is unconditional once the router is allowlisted, requires no special privileges, and allows unauthorized users to trade on a pool that was designed to be access-controlled. This constitutes broken core pool functionality and a direct allowlist bypass enabling unauthorized fund flows — matching the "Allowlist path" smart audit pivot and the "Broken core pool functionality causing loss of funds or unusable swap flows" allowed impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public periphery contract callable by anyone with no preconditions. Any pool using `SwapAllowlistExtension` that also needs to support router-mediated swaps (the standard user path) is affected. The bypass requires only a single public transaction. The only precondition — the router being allowlisted — is exactly the state the pool admin must set to make the router usable for allowlisted users, making the vulnerable configuration the expected production configuration.

## Recommendation
The extension must check the economically relevant actor — the end-user — not the intermediary. Two concrete approaches:

1. **Pass the original user through `extensionData`**: The router encodes the original `msg.sender` into `extensionData`; the extension decodes and verifies it. Requires a trusted encoding convention between router and extension.
2. **Router registry in the extension**: Maintain a registry of trusted routers; when `sender` is a trusted router, decode the real user from `extensionData` and check that address instead.
3. **Architectural fix**: The pool should pass both `msg.sender` (direct caller) and an optional `originSender` (end-user forwarded by the router) to extension hooks, so guards can key on the correct actor.

## Proof of Concept
```solidity
// Pool is configured with SwapAllowlistExtension.
// Admin allowlists only trustedUser and the router (to allow trustedUser to use the router).
swapExtension.setAllowedToSwap(address(pool), address(router), true);
swapExtension.setAllowedToSwap(address(pool), trustedUser, true);

// attacker (not allowlisted) calls the router directly
vm.prank(attacker);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1000e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    extensionData: "",
    deadline: block.timestamp
}));
// Succeeds: extension checks allowedSwapper[pool][router] == true
// Attacker swaps on a pool they were never authorized to access
```
