### Title
Unprivileged Caller Can Permanently Delete Canonical Chain State via `run_mainchain_gc` - (File: contract/src/lib.rs)

### Summary
`run_mainchain_gc` is a state-mutating public function that carries no caller access control when the contract is unpaused. Any unprivileged NEAR account can call it with an arbitrarily large `batch_size`, permanently deleting mainchain block headers from the canonical chain state ahead of schedule. This causes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for blocks that were previously verifiable, and can permanently prevent chain reorganizations from completing.

### Finding Description
The function `run_mainchain_gc` is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`:

```rust
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
```

The `#[pause]` attribute from `near-plugins` restricts the function only when the contract is **paused**. During normal (unpaused) operation, there is no role check and no caller restriction — any NEAR account can invoke this `&mut self` function. The `batch_size` parameter is fully attacker-controlled. [1](#0-0) 

Inside the function, when `amount_of_headers_we_store > gc_threshold`, it permanently removes block headers from `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`, then advances `mainchain_initial_blockhash`: [2](#0-1) 

The automatic GC triggered inside `submit_blocks` uses `batch_size = num_of_headers` (the count of headers in that single submission), so it removes at most as many blocks as were just submitted in one call. An attacker calling `run_mainchain_gc(u64::MAX)` removes **all** blocks currently beyond `gc_threshold` in a single transaction — far more aggressively than the automatic path. [3](#0-2) 

The test suite confirms that an account holding only `UnrestrictedSubmitBlocks` (not `UnrestrictedRunGC`) can successfully call `run_mainchain_gc` on an unpaused contract: [4](#0-3) 

### Impact Explanation
`verify_transaction_inclusion` looks up the target block in `mainchain_header_to_height`. If the block has been GC'd, it panics: [5](#0-4) 

An attacker can front-run a pending SPV proof call by first calling `run_mainchain_gc(u64::MAX)`, causing the proof to fail for a block that was legitimately within the verifiable window moments before. The deletion is **permanent and irreversible** — once a block header is removed from `headers_pool` and the height maps, it cannot be re-inserted.

Additionally, as documented in `contract/CLAUDE.md`: *"If mainchain blocks near the fork point have been garbage collected, reorg will fail — the contract panics with `PrevBlockNotFound`."* An attacker can trigger premature GC to permanently disable the chain's ability to reorganize around a fork, locking the canonical chain to a potentially weaker branch. [6](#0-5) 

### Likelihood Explanation
The entry path requires no privileges, no staking, and no special role. Any NEAR account can call `run_mainchain_gc` with a single transaction. The function is publicly documented and named. The attack is cheap (only gas cost) and the damage is irreversible. Likelihood is **high**.

### Recommendation
Add an explicit role guard to `run_mainchain_gc` so that only authorized accounts (e.g., `Role::DAO`, `Role::RelayerManager`, or a dedicated `GCManager` role) can invoke it externally. The internal call from `submit_blocks` should be routed through a private helper that bypasses the role check. For example:

```rust
// Public entry: role-gated
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id()),
        "Unauthorized"
    );
    self.run_mainchain_gc_internal(batch_size);
}

// Private helper called from submit_blocks
fn run_mainchain_gc_internal(&mut self, batch_size: u64) { ... }
```

### Proof of Concept
1. Deploy the contract and initialize it with `gc_threshold = 52704` and a live relayer submitting blocks.
2. Wait until the mainchain grows to `gc_threshold + N` blocks (N > 0), so GC has pending work.
3. From any unprivileged NEAR account (no roles), call:
   ```
   run_mainchain_gc({ "batch_size": 18446744073709551615 })
   ```
4. The contract removes all N blocks beyond `gc_threshold` in one shot, advancing `mainchain_initial_blockhash`.
5. Any downstream contract that now calls `verify_transaction_inclusion` for one of those N deleted blocks receives a panic: `"block does not belong to the current main chain"` — a false negative for a legitimately confirmed Bitcoin transaction.
6. If any of the deleted blocks were near an active fork's common ancestor, subsequent `submit_blocks` calls for the fork will panic with `"PrevBlockNotFound"`, permanently preventing the reorg. [7](#0-6)
