### Title
Undercharged Execution in `op_substr` — Missing Allocation Cost Charge — (File: `src/more_ops.rs`)

### Summary

`op_substr` (opcode 12) returns a fixed cost of `1` regardless of the size of the atom it creates. Every other operator that allocates a new atom applies `malloc_cost()`, which adds `MALLOC_COST_PER_BYTE` (10) per byte of the result. `op_substr` skips this entirely, allowing an attacker to produce arbitrarily large atoms at a flat cost of `1` per call, bypassing the cost model's allocation accounting.

### Finding Description

The CLVM cost model charges for every byte allocated via `MALLOC_COST_PER_BYTE = 10`. This is enforced consistently across all allocating operators:

- `op_sha256` → `new_atom_and_cost(a, cost, &hasher.finalize())` which adds `buf.len() * MALLOC_COST_PER_BYTE` [1](#0-0) 
- `op_concat` → `cost += len as Cost * (CONCAT_COST_PER_BYTE + MALLOC_COST_PER_BYTE)` [2](#0-1) 
- `op_strlen` → `Ok(malloc_cost(a, cost, size_node))` [3](#0-2) 
- `op_add`, `op_subtract`, `op_multiply`, `op_ash`, `op_lsh`, `op_lognot` → all call `malloc_cost(a, cost, r)` [4](#0-3) 

`op_substr`, however, does this:

```rust
let r = a.new_substr(a0, start as u32, end as u32)?;
let cost: Cost = 1;
Ok(Reduction(cost, r))
``` [5](#0-4) 

The cost is hardcoded to `1` with no scaling by output size and no `malloc_cost` call. The `_max_cost` parameter is also unused (underscore prefix), meaning no intermediate cost check is performed either. [6](#0-5) 

`malloc_cost` is defined as:

```rust
fn malloc_cost(a: &Allocator, cost: Cost, ptr: NodePtr) -> Reduction {
    let c = a.atom_len(ptr) as Cost * MALLOC_COST_PER_BYTE;
    Reduction(cost + c, ptr)
}
``` [4](#0-3) 

`MALLOC_COST_PER_BYTE` is 10. [7](#0-6) 

For a 1 MB substring, the correct charge would be `10 * 1_048_576 = 10_485_760` cost units. Instead, the charge is `1`.

### Impact Explanation

The CLVM cost budget is the sole mechanism preventing resource exhaustion in puzzle evaluation. If an attacker can produce large atoms at cost `1` each, they can:

1. Create one large atom (paying full cost once, e.g., via `concat` or environment injection).
2. Call `substr` in a loop to produce many large atoms at cost `1` per call.
3. Exhaust the allocator heap (atom-byte limit enforced by `LIMIT_HEAP`) or atom-count limit without paying proportional cost.

This breaks the consensus invariant that cost is a faithful proxy for resource consumption. A program that should be rejected as too expensive (cost > `max_cost`) may instead succeed, or two nodes with different `max_cost` settings may diverge on whether to accept a spend bundle. [8](#0-7) 

### Likelihood Explanation

Opcode 12 (`substr`) is a standard, always-available CLVM operator dispatched directly by `ChiaDialect::op()`. [9](#0-8) 

No special flags, softfork guards, or permissions are required. Any attacker who can submit a spend bundle to the mempool can craft CLVM bytes that invoke `substr` with attacker-controlled start/end indices on a large atom. The entry path is fully attacker-controlled and requires no privileged access.

### Recommendation

Apply `malloc_cost` to the result of `op_substr`, consistent with every other allocating operator:

```rust
// Replace:
let cost: Cost = 1;
Ok(Reduction(cost, r))

// With:
let cost: Cost = 1;
Ok(malloc_cost(a, cost, r))
```

Additionally, add a `check_cost` call using the `max_cost` parameter (currently ignored) to allow early termination if the budget is already exceeded before the call.

### Proof of Concept

CLVM program (pseudocode):
```
; Create a 65535-byte atom via concat, then extract it 10000 times via substr
; Each substr call costs 1 instead of 655,350 (65535 * 10)
; Total undercharge: ~6.5 billion cost units for 10000 calls
(concat <65535 bytes>) -> large_atom
(substr large_atom 0 65535)  ; cost: 1 (should be 655,351)
(substr large_atom 0 65535)  ; cost: 1 (should be 655,351)
... repeated 10000 times
```

In hex CLVM bytes, opcode `0x0c` (12 = `substr`) with a large atom argument and indices `0` and `N` will produce an N-byte atom at cost `1`. A program budget of `11_000_000` (the Chia block cost limit) would allow approximately 11 million such calls, each producing a large atom, exhausting the allocator heap at a fraction of the intended cost.
