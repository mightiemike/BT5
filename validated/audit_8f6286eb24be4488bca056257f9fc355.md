### Title
Router-Mediated Swap Bypasses `SwapAllowlistExtension` Per-User Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` keys its authorization check on the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the end user. The hook therefore checks whether the router is allowlisted, not whether the individual user is allowlisted. Any pool that allowlists the router to support router-mediated swaps simultaneously grants unrestricted swap access to every user who routes through it.

### Finding Description

**Call chain — direct swap (correct):**

```
User → MetricOmmPool.swap()
         msg.sender = User
         _beforeSwap(msg.sender=User, ...)          ← ExtensionCalling.sol:230
         → beforeSwap(sender=User, ...)
         → allowedSwapper[pool][User]               ← correct actor
```

**Call chain — router swap (broken):**

```
User → MetricOmmSimpleRouter.exactInputSingle()
         → IMetricOmmPoolActions(pool).swap(...)    ← SimpleRouter.sol:72-80
              msg.sender = Router
              _beforeSwap(msg.sender=Router, ...)   ← ExtensionCalling.sol:230
              → beforeSwap(sender=Router, ...)
              → allowedSwapper[pool][Router]        ← wrong actor
```

The pool passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as `sender` to the extension: [2](#0-1) 

The extension then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

The router calls `pool.swap()` directly with no mechanism to forward the original user's identity: [4](#0-3) 

### Impact Explanation

A pool admin who wants to restrict swaps to a curated set of users AND support the official router faces an impossible choice:

- **Do not allowlist the router**: allowlisted users cannot use the router at all (DoS on the intended UX).
- **Allowlist the router**: the per-user gate is completely bypassed — every user on the network can swap through the router regardless of their individual allowlist status.

The second scenario is the exploit path. Any unprivileged user routes through `MetricOmmSimpleRouter` and the hook passes because `allowedSwapper[pool][router] == true`. The allowlist provides zero protection against router-mediated swaps.

### Likelihood Explanation

The router is the primary intended entry point for end users. Pool admins who deploy a `SwapAllowlistExtension`-gated pool and want users to access it via the router will naturally allowlist the router address. The bypass requires no special privileges, no flash loans, and no oracle manipulation — only a standard router call.

### Recommendation

The extension must receive the **original initiating user**, not the immediate caller of `pool.swap()`. Options:

1. Have the router encode the original `msg.sender` into `extensionData` and have the extension decode it (requires trust that the router is the caller, verifiable via a router registry).
2. Pass the original user through a dedicated field in the hook signature rather than reusing `sender`.
3. Document that `SwapAllowlistExtension` is incompatible with router-mediated swaps and enforce this at the factory level.

### Proof of Concept

```solidity
// Setup: pool uses SwapAllowlistExtension; admin allowlists router
allowedSwapper[pool][router] = true;

// Attacker (not individually allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    ...
}));
// → pool.swap(msg.sender=router) → beforeSwap(sender=router)
// → allowedSwapper[pool][router] == true → PASSES
// Attacker swaps on a pool they were never individually authorized to access.
```

The "alternating swap directions across consecutive blocks" framing in the question is a red herring — the wrong-actor binding is structural and present on every single router-mediated swap regardless of direction or timing.
