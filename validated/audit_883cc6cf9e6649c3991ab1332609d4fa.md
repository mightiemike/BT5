Audit Report

## Title
`SwapAllowlistExtension` checks the router address as `sender`, allowing any caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` ŌĆö the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, the pool's `msg.sender` is the router contract, not the end user. If the pool admin allowlists the router (required for any allowlisted user to perform router-mediated swaps), every unpermissioned address can bypass the allowlist by calling the router instead of the pool directly.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ŌåÉ direct caller of pool.swap(), i.e. the router
    recipient,
    ...
);
``` [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards this value unchanged as the `sender` argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` is the pool (correct) and `sender` is the router address, not the end user.

`MetricOmmSimpleRouter.exactInputSingle` stores the real end-user address only in transient storage for the payment callback and calls `pool.swap()` directly ŌĆö the router becomes `msg.sender` from the pool's perspective:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

The allowlist check therefore resolves to `allowedSwapper[pool][router]`. The admin faces two losing options: (1) do not allowlist the router ŌĆö allowlisted users cannot use the router at all; (2) allowlist the router ŌĆö every address on the network can bypass the allowlist. No configuration simultaneously allows legitimate users to use the router and blocks unauthorized users.

## Impact Explanation

A pool protected by `SwapAllowlistExtension` is deployed to restrict swaps to a curated set of counterparties (KYC enforcement, LP exposure control, arbitrage prevention). Once the admin allowlists the router ŌĆö a necessary operational step for any allowlisted user to use multi-hop or exact-output swaps ŌĆö any unpermissioned address can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps against the pool. The allowlist invariant is silently violated: the extension is called, returns normally, and no on-chain signal indicates the bypass. Unauthorized arbitrageurs can drain LP value at oracle-fair prices. This constitutes broken core pool functionality (the allowlist access control) causing direct loss of LP assets.

## Likelihood Explanation

The router is the primary user-facing entry point for swaps. Any pool admin who wants allowlisted users to perform router-mediated swaps must allowlist the router. The bypass is then reachable by any address with no special privilege, no flash loan, and no unusual token behavior. The trigger is a standard `exactInputSingle` call. The condition is met in every realistic allowlisted-pool deployment that supports router usage.

## Recommendation

The pool should forward the original caller's identity to extensions as a distinct field separate from the direct `msg.sender`. Concretely: add an explicit `originator` parameter to the `beforeSwap` hook signature and have the router encode `msg.sender` into `extensionData` so the extension can decode and verify it. Alternatively, `SwapAllowlistExtension` should read the originator from a standardized field in `extensionData` when `sender` is a known router, or the router should be redesigned to pass the end-user address through a mechanism the extension can trust (e.g., a signed permit or a dedicated originator slot in the call). At minimum, the extension's NatSpec must document clearly that it gates the **direct pool caller**, not the end user, so admins understand that allowlisting the router opens the gate to all users.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured.
2. Admin: setAllowedToSwap(pool, alice, true)   // alice is the only intended swapper
3. Admin: setAllowedToSwap(pool, router, true)  // required so alice can use the router
4. Bob (not allowlisted) calls:
       router.exactInputSingle({pool: pool, recipient: bob, ...})
5. Pool.swap() is called with msg.sender = router.
6. _beforeSwap(sender=router, ...) is forwarded to SwapAllowlistExtension.
7. Extension checks: allowedSwapper[pool][router] == true ŌåÆ passes.
8. Bob's swap executes; allowlist is bypassed.
9. Bob repeats arbitrarily, extracting LP value.
```

The wrong value is `sender` in `allowedSwapper[pool][sender]` ŌĆö it resolves to the router address rather than the end user's address, making the allowlist check meaningless whenever the router is allowlisted.
