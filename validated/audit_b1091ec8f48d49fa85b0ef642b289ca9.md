### Title
Stale Tier APY Used in Lazy Bonus Reward Calculation Causes Incorrect Reward Amounts - (File: `x/tieredrewards/keeper/bonus_rewards.go`)

---

### Summary

The `x/tieredrewards` module calculates bonus rewards lazily using an event-sourcing model. When a user claims rewards, `ProcessEventsAndClaimBonus` replays `ValidatorEvent` entries and calls `ComputeSegmentBonus` for each segment. The `BonusAPY` used in that formula is read from the **current** `Tier` parameters at claim time. Because governance can update a tier's `BonusAPY` via `MsgUpdateTier` at any point, and because tier APY changes are never recorded as `ValidatorEvent` entries, the lazy calculation retroactively applies the post-update APY to all historical segments — including those that accrued before the governance change. This is the direct analog of the Rubicon bug: a rate that is a function of external, time-varying state is used as a constant over the entire accrual window.

---

### Finding Description

The `ComputeSegmentBonus` function computes bonus rewards for a time segment using the formula:

```
Bonus = Shares × (Tokens/Share) × BonusAPY × (Δt / SecondsPerYear)
``` [1](#0-0) 

The `BonusAPY` value is sourced from the live `Tier` object fetched at claim time, not from a historical snapshot stored at position creation or at each governance change. [2](#0-1) 

The event-sourcing system records only three event types for lazy replay: `VALIDATOR_EVENT_TYPE_SLASH`, `VALIDATOR_EVENT_TYPE_BOND`, and `VALIDATOR_EVENT_TYPE_UNBOND`. [3](#0-2) 

There is no `TIER_APY_CHANGE` (or equivalent) event type. Governance is permitted to update a tier's `BonusAPY` via `MsgUpdateTier`: [4](#0-3) 

When `ProcessEventsAndClaimBonus` is called, it replays all `ValidatorEvent` entries since the position's `LastEventSeq` and calls `ComputeSegmentBonus` for each segment using the **current** tier APY. Because no APY-change event was ever appended, the replay has no mechanism to use the APY that was in effect during each historical segment. [5](#0-4) 

---

### Impact Explanation

- If governance **increases** `BonusAPY`, any user who has not yet claimed rewards will receive a retroactive windfall: the higher APY is applied to all past segments, draining the rewards pool faster than intended.
- If governance **decreases** `BonusAPY`, users who have not yet claimed are silently underpaid for the period before the reduction, with no recourse.

In both cases the exact corrupted value is the `sdk.Coins` amount minted/transferred to the position owner during `ClaimRewards`, which diverges from the amount that was legitimately earned under the APY schedule that was in effect at each point in time. [6](#0-5) 

---

### Likelihood Explanation

Governance parameter updates to tiers are a supported, documented operation (`MsgUpdateTier`). Any governance proposal that adjusts `BonusAPY` — a routine act in a live protocol — triggers this miscalculation for every position that has not claimed since the last update. The entry path requires no privileged key: any delegator with an open position is affected automatically upon the next governance-approved tier update. [7](#0-6) 

---

### Recommendation

Record a `TIER_APY_CHANGE` validator event (or a dedicated tier-level event) whenever `MsgUpdateTier` modifies `BonusAPY`. The lazy replay in `ProcessEventsAndClaimBonus` must then use the APY that was in effect during each segment boundary, not the APY current at claim time. Alternatively, snapshot the `BonusAPY` into the `Position` struct at creation time and update it (with a forced partial claim) whenever the tier APY changes. [3](#0-2) [8](#0-7) 

---

### Proof of Concept

1. Alice opens a tiered position at block T=0 with `BonusAPY = 5%` via `MsgLockTier`. She does not claim rewards.
2. At block T=100, governance passes `MsgUpdateTier` raising `BonusAPY` to `20%`. No `ValidatorEvent` is appended.
3. At block T=200, Alice calls `MsgClaimRewards`. `ProcessEventsAndClaimBonus` replays events since `LastEventSeq=0`. It finds only bond/slash events; it fetches the current tier and reads `BonusAPY = 20%`. It calls `ComputeSegmentBonus` for the entire window T=0→200 using `20%`.
4. Alice receives bonus rewards calculated at `20%` for the full 200-block window, even though she was only entitled to `5%` for the first 100 blocks — a `3×` overpayment for that segment, drawn from the shared rewards pool. [1](#0-0) [9](#0-8)
