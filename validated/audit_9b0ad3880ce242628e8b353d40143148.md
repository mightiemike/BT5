### Title
Protobuf `ValidResourceBounds` Deserialization Silently Misclassifies `AllResources` as `L1Gas` When L2/Data Gas Are Zero, Producing a Wrong Transaction Hash Preimage - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` conversion uses a value-based heuristic to reconstruct the `ValidResourceBounds` variant after protobuf deserialization. When an `AllResources` transaction carries zero `l2_gas` and zero `l1_data_gas` bounds, the heuristic silently returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of elements for each variant, the transaction hash computed after P2P round-trip diverges from the hash computed at submission time, binding the wrong hash to the executable payload.

---

### Finding Description

**Serialization loses the variant tag.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes both `L1Gas` and `AllResources`-with-zero-l2/data-gas to an identical wire encoding: all three `ResourceLimits` fields present but with zero values for `l2_gas` and `l1_data_gas`. [1](#0-0) 

**Deserialization applies a wrong heuristic.**

On the receiving side, the converter inspects the decoded values and returns `L1Gas` whenever both `l2_gas` and `l1_data_gas` are zero, regardless of whether the sender intended `AllResources`: [2](#0-1) 

The test suite explicitly documents this invariant break: [3](#0-2) 

**The two variants produce different hash preimages.**

`get_tip_resource_bounds_hash` chains a different number of elements depending on the variant:

- `L1Gas` → `poseidon([tip, L1_GAS_concat, L2_GAS_concat])` — **3 elements**
- `AllResources` → `poseidon([tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat])` — **4 elements** [4](#0-3) 

A transaction originally hashed as `AllResources` (4-element poseidon) is re-hashed as `L1Gas` (3-element poseidon) after a P2P round-trip, yielding a different `TransactionHash`. The Starknet OS Cairo code independently always enforces exactly 3 resource-bound entries in `hash_fee_fields`, so the Rust-computed hash after misclassification also diverges from the OS-computed hash used in proofs. [5](#0-4) 

**Secondary effect: wrong `GasVectorComputationMode`.**

`ValidResourceBounds::get_gas_vector_computation_mode()` returns `NoL2Gas` for `L1Gas` and `All` for `AllResources`. The misclassification causes the blockifier to apply the wrong fee-validation and gas-accounting path for the re-executed transaction. [6](#0-5) 

---

### Impact Explanation

When a node receives a block over P2P containing an `AllResources` V3 transaction whose `l2_gas` and `l1_data_gas` bounds are both zero, the protobuf converter silently downgrades the variant to `L1Gas`. Any subsequent hash computation (e.g., for receipt verification, re-execution, or proof reconciliation) uses the 3-element poseidon preimage instead of the 4-element one, producing a `TransactionHash` that does not match the hash committed to by the original submitter or the Starknet OS. This constitutes **transaction conversion binding the wrong hash to the executable payload** — matching the High impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The trigger requires a V3 transaction with `AllResources` bounds where both `l2_gas.max_amount == 0` and `l1_data_gas.max_amount == 0`. Such transactions are structurally valid per the Starknet spec (the OS enforces non-zero `l2_gas` at execution time, but the hash and admission layers do not). The bug is reachable by any unprivileged peer that propagates such a transaction. The test workaround (`add_gas_values_to_transaction` forcibly sets `l2_gas.max_amount = 1`) confirms the developers are aware the production path is broken for zero-value bounds.

---

### Recommendation

**Short term:** Remove the value-based heuristic. The protobuf `ResourceBounds` message should carry an explicit discriminant field (e.g., a boolean `is_all_resources` or a `oneof` wrapper) so the variant can be faithfully round-tripped without inspecting field values.

**Long term:** Add a property-based test asserting that `ValidResourceBounds → protobuf → ValidResourceBounds` is an identity for all inputs, including `AllResources` with zero sub-fields. Enforce this invariant in CI to prevent regression.

---

### Proof of Concept

```rust
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use apollo_protobuf::protobuf;

// 1. Construct AllResources with zero l2_gas and l1_data_gas.
let original = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas: ResourceBounds { max_amount: 100.into(), max_price_per_unit: 1.into() },
    l2_gas: ResourceBounds::default(),   // zero
    l1_data_gas: ResourceBounds::default(), // zero
});

// 2. Serialize to protobuf (both L1Gas and AllResources-with-zeros produce identical bytes).
let proto: protobuf::ResourceBounds = original.into();

// 3. Deserialize — heuristic fires, returns L1Gas.
let roundtripped = ValidResourceBounds::try_from(proto).unwrap();
assert!(matches!(roundtripped, ValidResourceBounds::L1Gas(_))); // NOT AllResources

// 4. Hash diverges.
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;
let tip = &Tip(0);
let hash_original    = get_tip_resource_bounds_hash(&original,    tip).unwrap();
let hash_roundtripped = get_tip_resource_bounds_hash(&roundtripped, tip).unwrap();
assert_ne!(hash_original, hash_roundtripped); // different poseidon outputs
```

The divergence arises because `hash_original` chains 4 elements (tip + L1 + L2 + L1_DATA) while `hash_roundtripped` chains only 3 (tip + L1 + L2), matching the `L1Gas` branch of `get_tip_resource_bounds_hash`. [7](#0-6)

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));

        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
