### Title
`OracleValueStopLossExtension` Uses Arithmetic Mid-Price Instead of Geometric Mid-Price, Miscalibrating the Stop-Loss Guard - (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`OracleValueStopLossExtension._afterSwapOracleStopLoss` computes the oracle mid-price as the **arithmetic mean** of bid and ask, while every other pricing path in the protocol — including the pool's own swap engine and `PriceVelocityGuardExtension` — uses the **geometric mean** (`sqrt(bid * ask)`). Because the arithmetic mean is always ≥ the geometric mean (AM-GM inequality), the per-bin value metrics fed into the watermark comparison are systematically biased: `metricT0` is understated and `metricT1` is overstated relative to the pool's actual pricing. This causes the stop-loss guard to be miscalibrated: it allows more value extraction in the `zeroForOne` direction than the configured `drawdownE6` threshold intends.

### Finding Description

**Inconsistent mid-price formula in the stop-loss guard:**

In `OracleValueStopLossExtension._afterSwapOracleStopLoss` (line 218):
```solidity
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;
``` [1](#0-0) 

The pool's swap engine and `PriceVelocityGuardExtension` both use `SwapMath.midAndSpreadFeeX64FromBidAsk`, which computes the **geometric mean**:
```solidity
midPriceX64 = Math.sqrt(bidPriceX64 * askPriceX64);
``` [2](#0-1) 

`PriceVelocityGuardExtension` correctly calls `SwapMath.midAndSpreadFeeX64FromBidAsk`: [3](#0-2) 

The stop-loss guard then feeds this arithmetic mid into `_metrics`:
```solidity
metricT0 = _clampMetric(t0ps + Math.mulDiv(Math.mulDiv(uint256(t1), Q64, midPriceX64), METRIC_SCALE, shares));
metricT1 = _clampMetric(Math.mulDiv(Math.mulDiv(uint256(t0), midPriceX64, Q64), METRIC_SCALE, shares) + t1ps);
``` [4](#0-3) 

**Effect of AM > GM:**

Let `bid = m(1-s)`, `ask = m(1+s)` for spread `s`. Then:
- Arithmetic mean = `m`
- Geometric mean = `m·sqrt(1-s²) ≈ m·(1 - s²/2)`

So arithmetic mean exceeds geometric mean by `≈ m·s²/2`.

With the inflated arithmetic mid:
- `metricT0 = t0/shares + t1/(mid·shares)` → **understated** (dividing by a larger mid)
- `metricT1 = t0·mid/shares + t1/shares` → **overstated** (multiplying by a larger mid)

**Watermark miscalibration:**

The high watermark for token0 (`hwm0`) is ratcheted up to `metricT0` when `metricT0 >= hwm0`. Because `metricT0` is understated, `hwm0` is set to a lower value than the pool's actual token0-denominated value per share. The breach floor is `hwm0 * floorMultiplier / E6`. When the pool subsequently loses value through `zeroForOne` swaps (token1 outflow), the guard checks:
```solidity
if (breach0 && zeroForOne) {
    revert OracleStopLossTriggered(...);
}
``` [5](#0-4) 

Because both the watermark and the current metric are computed with the same inflated mid, the ratio `metricT0 / hwm0` is approximately preserved when the spread is constant. However, when the oracle spread widens between the watermark-setting swap and the value-extraction swap (a realistic scenario during volatility), the arithmetic mean diverges from the geometric mean by a different amount at each point in time. The guard's metric no longer tracks the pool's actual value consistently, causing the breach threshold to be hit at a different actual drawdown than `drawdownE6` specifies.

Concretely: if the spread widens from 0.5% to 5% between the watermark-setting event and the extraction event, the arithmetic-vs-geometric discrepancy changes from ~0.001% to ~0.125% of mid. The guard's `metricT0` at extraction time is understated by a larger margin than the stored `hwm0`, so the apparent drawdown is smaller than the real drawdown. The guard fails to trigger at the configured threshold, allowing LPs to lose more than `drawdownE6` permits.

### Impact Explanation

LPs who rely on `OracleValueStopLossExtension` to cap their drawdown at `drawdownE6` can suffer losses exceeding that threshold in the `zeroForOne` direction when oracle spreads widen between the watermark-setting swap and the value-extraction swap. The stop-loss guard — the primary on-chain protection for LP principal — fails to trigger at the correct drawdown boundary. This is a direct loss of LP principal above the configured protection level.

### Likelihood Explanation

The condition requires:
1. A pool configured with `OracleValueStopLossExtension` (a supported, documented extension).
2. Oracle spread that changes between the watermark-setting swap and the extraction swap — a normal occurrence during market volatility.
3. A `zeroForOne` swap that extracts value from the pool.

All three conditions are routine in production. No privileged access or malicious setup is required; any unprivileged swapper can trigger the extraction once the guard is miscalibrated.

### Recommendation

Replace the arithmetic mean in `_afterSwapOracleStopLoss` with the same geometric mean formula used by the pool's swap engine:

```solidity
// Before (incorrect):
uint256 midPriceX64 = (uint256(bidPriceX64) + uint256(askPriceX64)) / 2;

// After (consistent with pool pricing):
(uint256 midPriceX64,) = SwapMath.midAndSpreadFeeX64FromBidAsk(uint256(bidPriceX64), uint256(askPriceX64));
```

This aligns the stop-loss guard's value metric with the pool's actual pricing, ensuring the configured `drawdownE6` threshold is enforced accurately regardless of spread changes.

### Proof of Concept

1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 100_000` (10%), and an oracle with a 0.5% spread.
2. Add liquidity; the first swap sets `hwm0` using arithmetic mid ≈ geometric mid (spread is small, discrepancy negligible).
3. Oracle spread widens to 10% (e.g., during a volatile period). Arithmetic mid now exceeds geometric mid by ~0.5%.
4. An attacker executes a `zeroForOne` swap that extracts value. The guard computes `metricT0` using the inflated arithmetic mid, which understates the actual value drop. The apparent drawdown is ~0.5% less than the real drawdown.
5. The guard does not revert even though the pool's actual value (measured at the geometric mid the pool uses for pricing) has dropped by `drawdownE6 + 0.5%`.
6. LPs lose 0.5% more than the configured protection allows.
