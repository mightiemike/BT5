### Title
Merkle Proof Forgery via Unvalidated `tx_id` Accepts Internal Tree Nodes as Transactions — (File: `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` performs no validation that the caller-supplied `tx_id` is a leaf node (a real transaction) rather than an internal Merkle tree node. An unprivileged NEAR caller can supply an internal node hash as `tx_id` together with a crafted sibling proof and receive `true` for a transaction that never existed on Bitcoin. This is the direct analog of the reported "infinite limits" pattern: just as `swapForGem` blindly accepts any exchange rate, `verify_transaction_inclusion` blindly accepts any hash as a transaction identifier.

### Finding Description

`verify_transaction_inclusion` delegates all proof work to `compute_root_from_merkle_proof`: [1](#0-0) 

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` simply walks the proof array, hashing left or right based on the position bit at each level, with no constraint that the starting hash is a leaf: [2](#0-1) 

Consider a mainchain block with four transactions `[T0, T1, T2, T3]`. The Merkle tree is:

```
Level 0 (leaves): T0   T1   T2   T3
Level 1:          H01=H(T0,T1)  H23=H(T2,T3)
Level 2 (root):   R = H(H01, H23)
```

An attacker calls `verify_transaction_inclusion` with:
- `tx_id = H01` (an internal node, not a real transaction)
- `tx_index = 0`
- `merkle_proof = [H23]`

`compute_root_from_merkle_proof(H01, 0, [H23])` computes:
- position 0 → even → `H(H01, H23) = R`

Returns `R == header.block_header.merkle_root` → **`true`**.

The function returns `true` for a transaction that does not exist. The only guard present is `require!(!args.merkle_proof.is_empty())`, which the attacker trivially satisfies with a one-element proof.

The function is still fully callable on-chain despite the Rust `#[deprecated]` attribute: [3](#0-2) 

The `#[deprecated]` annotation produces a Rust compiler warning for Rust callers only; NEAR JSON-RPC callers receive no such signal and can invoke the method freely.

`verify_transaction_inclusion_v2` was introduced to fix this by requiring a coinbase proof at position 0: [4](#0-3) 

However, the original vulnerable entry point remains live and unguarded.

### Impact Explanation

Any dApp (bridge, escrow, oracle) that calls `verify_transaction_inclusion` to gate fund release on Bitcoin transaction inclusion can be exploited. The attacker supplies an internal Merkle node as `tx_id`, receives `true`, and the dApp releases funds for a Bitcoin transaction that was never broadcast or confirmed. Impact is equivalent to the reported "theft of user rewards up to 100%": the attacker fabricates proof of a payment and drains the dApp.

### Likelihood Explanation

- The function is publicly callable with no role restriction and no pause guard that would block it by default.
- All inputs (`tx_id`, `tx_index`, `merkle_proof`) are fully attacker-controlled.
- The required block must be on the current mainchain, but any real mainchain block with ≥2 transactions suffices; the attacker reads the Merkle tree from public Bitcoin data.
- No mining power, privileged key, or social engineering is required.
- The attack is deterministic and requires only one NEAR transaction.

### Recommendation

1. **Remove `verify_transaction_inclusion` entirely** from the contract, or replace its body with `env::panic_str("use verify_transaction_inclusion_v2")` so on-chain callers cannot invoke it.
2. If the function must remain for backward compatibility, add the same coinbase-proof guard that `verify_transaction_inclusion_v2` uses before computing the transaction proof.
3. Enforce a minimum `merkle_proof` length consistent with `tx_index` (i.e., `proof.len() >= ceil(log2(tx_index + 1))`) to prevent depth-shortcutting attacks.

### Proof of Concept

```
# Precondition: block B is on the mainchain with transactions [T0, T1, T2, T3]
# Compute from public Bitcoin data:
H01 = SHA256d(T0 || T1)
H23 = SHA256d(T2 || T3)

# NEAR call (unprivileged):

    "merkle_proof":      ["<H23 as hex>"],
    "confirmations":     1
  }
}' --accountId attacker.near

# Returns: true
# H01 is an internal node; no such transaction exists on Bitcoin.
```

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
