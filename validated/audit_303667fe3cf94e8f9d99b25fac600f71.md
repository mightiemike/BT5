### Title
AuxPoW Chain Merkle Root Byte-Order Mismatch Causes All Dogecoin AuxPoW Block Submissions to Panic — (File: `contract/src/dogecoin.rs`)

---

### Summary

In `check_aux`, the computed chain merkle root is searched inside the raw coinbase script hex using `chain_root.to_string()`. The `H256::Display` implementation **reverses** the bytes to produce a human-readable (big-endian display) hex string. However, the AuxPoW coinbase script stores the chain merkle root as **raw bytes in natural (little-endian) byte order**, matching the SHA256d output directly. The search always fails, causing every Dogecoin AuxPoW block submission to panic.

---

### Finding Description

**Step 1 — `H256::Display` reverses bytes** [1](#0-0) 

```rust
impl fmt::Display for H256 {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let reversed: Vec<u8> = self.0.into_iter().rev().collect();
        write!(f, "{}", hex::encode(reversed))
    }
}
```

`H256` stores bytes in natural (SHA256d output) order. `to_string()` reverses them to produce the block-explorer display format.

**Step 2 — `block_hash()` stores bytes in natural order** [2](#0-1) 

`block_header.block_hash()` calls `double_sha256(...)` and wraps the raw output bytes directly into `H256`. No reversal occurs here.

**Step 3 — `check_aux` searches for the display-order hex in the raw script** [3](#0-2) 

```rust
let chain_root = merkle_tools::compute_root_from_merkle_proof(
    block_header.block_hash(),   // H256 in natural byte order
    aux_data.chain_id,
    &aux_data.chain_merkle_proof,
);
// ...
let mut pos_chain_root = script_sig
    .find(&chain_root.to_string())   // ← reversed (display) hex
    .expect("Aux POW missing chain merkle root in parent coinbase");
```

`script_sig` is obtained from: [4](#0-3) 

```rust
let script_sig = coinbase_tx
    .input
    .first()
    .unwrap()
    .script_sig
    .to_hex_string();   // raw bytes of the script as hex
```

`Script::to_hex_string()` from the `bitcoin` crate returns the raw bytes of the script as a hex string. The AuxPoW coinbase script stores the chain merkle root as 32 raw bytes in natural byte order (consistent with Bitcoin's wire-format serialization of hashes). The `MERGED_MINING_HEADER` constant `"fabe6d6d"` is also searched in this same raw hex, confirming the search domain is raw bytes. [5](#0-4) 

**The mismatch:**

| Location | Byte order | Hex example (32-byte hash `[0x01…0x20]`) |
|---|---|---|
| Coinbase script (raw) | Natural (SHA256d output) | `"010203…20"` |
| `chain_root.to_string()` | Reversed (display) | `"201f1e…01"` |

`script_sig.find("201f1e…01")` will never match `"010203…20"` in the raw script, so `.expect(…)` always panics.

---

### Impact Explanation

Every Dogecoin AuxPoW block submission (merged-mined with Litecoin) that reaches `check_aux` panics with:

```
Aux POW missing chain merkle root in parent coinbase
```

This completely disables the Dogecoin AuxPoW verification path. Since AuxPoW is the dominant mining method for Dogecoin mainnet, the contract cannot accept any real Dogecoin mainnet blocks when `skip_pow_verification = false`. The canonical chain state is frozen at genesis; no new headers can be accepted. [6](#0-5) 

---

### Likelihood Explanation

This triggers on every AuxPoW block submission with `skip_pow_verification = false` (the production setting). No special attacker input is required — any legitimate relayer submitting a real Dogecoin mainnet block will hit this panic. The entry point is the public `submit_blocks` function callable by any NEAR account. [7](#0-6) 

---

### Recommendation

Replace `chain_root.to_string()` with `hex::encode(chain_root.0)` to search for the raw (natural byte order) hex in the coinbase script:

```rust
// Before (wrong — display/reversed order):
let mut pos_chain_root = script_sig
    .find(&chain_root.to_string())
    .expect("Aux POW missing chain merkle root in parent coinbase");

// After (correct — natural/raw byte order):
let mut pos_chain_root = script_sig
    .find(&hex::encode(chain_root.0))
    .expect("Aux POW missing chain merkle root in parent coinbase");

// Also fix the length calculation below:
pos_chain_root += hex::encode(chain_root.0).len(); // still 64, but consistent
```

---

### Proof of Concept

1. Deploy the contract with `feature = "dogecoin"` and `skip_pow_verification = false`.
2. Submit any real Dogecoin mainnet AuxPoW block (one where `aux_data` is `Some`).
3. `check_aux` is called. `chain_root.to_string()` produces the reversed hex. `script_sig.find(…)` returns `None`. `.expect("Aux POW missing chain merkle root in parent coinbase")` panics.
4. The transaction reverts. No Dogecoin AuxPoW block can ever be accepted. [8](#0-7)
