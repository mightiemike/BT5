### Title
Post-0.13.3 V3 Transactions with Zero L2/Data-Gas Bounds Silently Downgraded to Pre-0.13.3 `L1Gas` Variant During Protobuf Deserialization, Producing a Wrong Transaction Hash - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-`ValidResourceBounds` converter uses a value-based heuristic (`l2_gas == 0 && l1_data_gas == 0`) to distinguish pre-0.13.3 (`L1Gas`) from post-0.13.3 (`AllResources`) transactions. A legitimate post-0.13.3 V3 transaction whose user-supplied bounds happen to be zero for both L2 gas and L1 data gas is silently downgraded to the `L1Gas` variant. Because `get_tip_resource_bounds_hash` hashes a **different number of resource elements** for the two variants, any subsequent hash recomputation from the deserialized object produces a value that diverges from the hash the signer committed to, breaking signature-domain integrity.

---

### Finding Description

**The rounding-down analog.** In the vault bug, `assets → shares` floor-division silently produces `shares = 0` while `assets > 0`, letting a non-zero transfer proceed without burning any shares. Here, `AllResources { l2_gas: 0, l1_data_gas: 0 }` is silently "rounded down" to `L1Gas`, discarding the third resource element from the hash preimage while the original signature was computed over three elements.

**Exact location – the discriminator:**

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 417-436

    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {

        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // silently zero
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant for post-0.13.3 tx
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
``` [1](#0-0) 

**The hash-domain divergence.** `get_tip_resource_bounds_hash` produces structurally different Poseidon inputs for the two variants:

```
L1Gas      → poseidon(tip, pack(L1_GAS, l1), pack(L2_GAS, 0))          // 2 resource felts
AllResources → poseidon(tip, pack(L1_GAS, l1), pack(L2_GAS, 0),
                              pack(L1_DATA_GAS, 0))                      // 3 resource felts
``` [2](#0-1) 

The extra `pack(L1_DATA_GAS, 0)` element changes the Poseidon digest even when its value is zero, because Poseidon is not length-prefix-free.

**How a post-0.13.3 transaction reaches this path.** Every RPC-ingested V3 transaction is stored internally as `AllResourceBounds` (the `InternalRpcInvokeTransactionV3` struct). When the block is propagated over P2P or written to storage, it is serialized via `From<ValidResourceBounds> for protobuf::ResourceBounds`, which faithfully emits all three resource fields. On the receiving side, the `TryFrom` above is invoked. If the user set `l2_gas = 0` and `l1_data_gas = 0` (a valid choice for a pure-L1-gas transaction under the post-0.13.3 fee model), the converter silently returns `L1Gas`. [3](#0-2) 

**Downstream hash recomputation.** `validate_transaction_hash` recomputes the hash from the deserialized `Transaction` object and compares it to the stored hash: [4](#0-3) 

For the affected transaction the recomputed hash (2-element `L1Gas` path) differs from the stored hash (3-element `AllResources` path), so `validate_transaction_hash` returns `false` for a perfectly valid transaction.

Additionally, `calculate_resource_bounds` (used by the native blockifier to pass bounds to the Cairo VM via `get_execution_info`) returns only two elements for `L1Gas`, omitting `L1_DATA_GAS`: [5](#0-4) 

This means the resource-bounds array seen by the account contract during re-execution differs from what was signed, which can cause account-contract-level signature verification to fail.

---

### Impact Explanation

| Layer | Effect |
|---|---|
| Hash validation (`validate_transaction_hash`) | Returns `false` for a valid block transaction → syncing nodes reject valid blocks |
| RPC simulation / tracing | Recomputes wrong hash → `starknet_simulateTransactions`, `starknet_traceTransaction` return authoritative-looking wrong values |
| Native blockifier re-execution | `get_execution_info` returns 2-element resource-bounds array instead of 3-element → account contract sees wrong signed data |

The primary impact category is **High – RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, with a secondary path to **Critical – Wrong state/receipt/revert result from blockifier/syscall/execution logic for accepted input** if re-execution is triggered (e.g., during proving or RPC simulation).

---

### Likelihood Explanation

Any user who submits a V3 transaction and sets `l2_gas.max_amount = 0` and `l1_data_gas.max_amount = 0` (relying solely on `l1_gas` for fee payment, which is a documented valid configuration) triggers this path. The gateway accepts such transactions because it uses `AllResourceBounds` throughout. The downgrade only occurs during protobuf deserialization on the receiving side. No privileged access is required; a normal unprivileged user submitting a standard V3 transaction is sufficient.

---

### Recommendation

Replace the value-based heuristic with an explicit version or wire-format discriminator. The simplest fix is to preserve the `AllResources` variant whenever the protobuf message was produced from a post-0.13.3 source (e.g., by adding a boolean `is_all_resources` field to the protobuf, or by checking the transaction version). At minimum, remove the `unwrap_or_default()` fallback and require `l1_data_gas` to be present for all post-0.13.3 transactions, as the existing TODO comment already anticipates:

```rust
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.ok_or(DEPRECATED_RESOURCE_BOUNDS_ERROR)?;
// Then always produce AllResources for V3 transactions.
Ok(ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }))
``` [6](#0-5)

### Proof of Concept

1. Craft a V3 invoke transaction with `AllResourceBounds { l1_gas: { max_amount: 1000, max_price: 1 }, l2_gas: { 0, 0 }, l1_data_gas: { 0, 0 } }`.
2. Compute the transaction hash using the **3-element** `AllResources` path: `poseidon("invoke", 3, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0), pack(L1_DATA_GAS,0,0)), ...)`. Sign it.
3. Submit to the gateway. The gateway computes the same 3-element hash, stores `tx_hash = H3`.
4. The block is serialized to protobuf. The `From<ValidResourceBounds>` serializer emits all three resource fields (l1_gas, l2_gas=0, l1_data_gas=0).
5. A syncing node deserializes via `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Since `l2_gas.is_zero() && l1_data_gas.is_zero()`, it returns `ValidResourceBounds::L1Gas(l1_gas)`.
6. Call `validate_transaction_hash` on the deserialized transaction. It recomputes the **2-element** hash `H2 = poseidon("invoke", 3, sender, poseidon(tip, pack(L1_GAS,1000,1), pack(L2_GAS,0,0)), ...)`.
7. `H2 ≠ H3` → `validate_transaction_hash` returns `false` for a valid transaction.
8. For RPC simulation: `starknet_simulateTransactions` re-executes the transaction; `get_execution_info` returns a 2-element resource-bounds array; the account contract's hash computation diverges from the signed hash.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
