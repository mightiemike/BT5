### Title
Malicious IBC Counterparty Can Mint Unbacked NFT Vouchers via Double-Prefixed ClassId — (`x/nft-transfer/keeper/packet.go`)

### Summary
`processReceivedPacket` unconditionally prepends the dest port/channel prefix to `data.ClassId` when `isAwayFromOrigin = true`, without checking whether `data.ClassId` already starts with that prefix. A malicious counterparty chain can craft a packet whose `ClassId` already contains the dest prefix, causing the victim chain to mint NFT vouchers under a double-prefixed class ID that has no corresponding escrow on any chain.

### Finding Description
In `processReceivedPacket` (`x/nft-transfer/keeper/packet.go`, lines 155–165), when `isAwayFromOrigin` is `true`, the code builds:

```go
classPrefix     := types.GetClassPrefix(packet.GetDestPort(), packet.GetDestChannel())
prefixedClassID := classPrefix + data.ClassId
```

`IsAwayFromOrigin` only checks whether `data.ClassId` starts with the *source* port/channel prefix. It does not check whether `data.ClassId` already starts with the *dest* port/channel prefix. `ValidateBasic()` performs no structural validation of `ClassId` content. Therefore, an attacker who controls a counterparty chain can send a packet with `data.ClassId = "nft/channel-0/nativeClass"` (where `nft/channel-0` is the victim's dest port/channel), causing the victim to compute `prefixedClassID = "nft/channel-0/nft/channel-0/nativeClass"` and mint vouchers under `ibc/{sha256("nft/channel-0/nft/channel-0/nativeClass")}`.

### Impact Explanation
The attacker mints NFT vouchers on the victim chain that are not backed by any escrowed NFT. These vouchers represent ownership claims that cannot be redeemed (no corresponding escrow exists on any chain). This breaks the core ICS-721 invariant: every voucher must correspond to exactly one escrowed NFT. The attacker can mint arbitrary quantities of unbacked NFT vouchers for any token ID they specify in the packet.

### Likelihood Explanation
Any chain that has established a legitimate IBC channel with the victim chain can exploit this. The attacker only needs to commit a crafted packet on their chain (which they fully control) and have a relayer submit it. No privileged access, governance, or key compromise is required. The IBC core layer validates only the packet envelope, not the application-layer `ClassId` content.

### Recommendation
In `processReceivedPacket`, before prepending the dest prefix, check whether `data.ClassId` already starts with the dest prefix and reject (or strip) it:

```go
classPrefix := types.GetClassPrefix(packet.GetDestPort(), packet.GetDestChannel())
if strings.HasPrefix(data.ClassId, classPrefix) {
    return fmt.Errorf("classId already contains dest prefix: %s", data.ClassId)
}
prefixedClassID := classPrefix + data.ClassId
```

Alternatively, add a structural validation in `ValidateBasic()` or `OnRecvPacket` that rejects any `ClassId` whose prefix matches the receiving channel's port/channel pair.

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
}
