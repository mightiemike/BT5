### Title
`AllResources`→`L1Gas` Variant Misclassification in Protobuf Deserialization Produces Wrong Transaction Hash for Syncing Nodes — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` infers the variant (`L1Gas` vs `AllResources`) by inspecting whether the wire values for `l2_gas` and `l1_data_gas` are zero. An `AllResources` transaction with zero `l2_gas` and zero `l1_data_gas` is silently reclassified as `L1Gas` during P2P state-sync deserialization. Because `get_tip_resource_bounds_hash` produces a structurally different hash preimage for the two variants (2 vs 3 resource-bound elements), the transaction hash computed by a syncing node diverges from the hash computed by the sequencer that originally accepted and signed the transaction.

### Finding Description

**Originating type definition** — `ValidResourceBounds` is a two-variant enum:

```
L1Gas(ResourceBounds)          // pre-0.13.3, 2-element hash preimage
AllResources(AllResourceBounds) // 0.13.3+,   3-element hash preimage
``` [1](#0-0) 

**Hash preimage divergence** — `get_tip_resource_bounds_hash` branches on the variant:

- `L1Gas` → hashes `[tip, L1_GAS_concat, L2_GAS_concat]` (2 resource elements)
- `AllResources` → hashes `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` (3 resource elements) [2](#0-1) 

Even when `l2_gas` and `l1_data_gas` are zero, `get_concat_resource(&ResourceBounds::default(), L1_DATA_GAS)` produces a non-zero felt (it encodes the `L1_DATA_GAS` resource identifier), so the two hash values are numerically distinct.

**The flawed deserializer** — `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in the P2P state-sync converter decides the variant purely from the wire values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong for AllResources txs
} else {
    ValidResourceBounds::AllResources(...)
})
``` [3](#0-2) 

This converter is used when syncing historical `Transaction` objects (the `InvokeTransactionV3` storage type carries `ValidResourceBounds`). It is distinct from the consensus-path converter (`TryFrom<protobuf::ResourceBounds> for AllResourceBounds` in `rpc_transaction.rs`) which always produces `AllResources` and is not affected. [4](#0-3) 

**Origination path** — The gateway accepts `RpcInvokeTransactionV3` / `RpcDeclareTransactionV3`, both of which carry `AllResourceBounds` directly (no variant ambiguity). The converter computes the hash using `ValidResourceBounds::AllResources(...)` → 3-element preimage → hash `H_A`. [5](#0-4) 

When the block is later synced via P2P, `InvokeTransactionV3` is deserialized from protobuf. If `l2_gas = 0` and `l1_data_gas = 0`, the deserializer produces `L1Gas` → 2-element preimage → hash `H_B ≠ H_A`. [6](#0-5) 

**The RPC path** also uses the same `ResourceBoundsMapping → ValidResourceBounds` conversion with the identical zero-check heuristic:

```rust
if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
    Self::L1Gas(value.l1_gas)
``` [7](#0-6) 

### Impact Explanation

A syncing node that receives a block containing an `AllResources` invoke/declare/deploy-account transaction with `l2_gas = 0` and `l1_data_gas = 0` will:

1. Deserialize the transaction as `L1Gas` variant.
2. Recompute the transaction hash using the 2-element preimage → wrong hash `H_B`.
3. Store and serve `H_B` via RPC (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`).
4. Fail block-hash verification if transaction hashes are committed into the block hash.

This matches the impact scope: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** and **"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

### Likelihood Explanation

Any user can submit a valid `AllResources` transaction with zero `l2_gas` and zero `l1_data_gas` (the gateway accepts it — the stateless validator only checks that at least one resource bound is non-zero in amount, and `l1_gas` alone satisfies that). The trigger is therefore **unprivileged and requires no special access**. The condition is reachable on mainnet today for any pre-0.13.3-style transaction submitted through the new gateway path.

### Recommendation

The protobuf wire format must carry an explicit variant discriminant rather than inferring it from value content. Options:

1. **Add a boolean/enum field** to the protobuf `ResourceBounds` message (e.g., `bool all_resources = 4`) that is set by the serializer and read by the deserializer, making the variant unambiguous regardless of the numeric values.

2. **Alternatively**, always serialize and deserialize as `AllResources` for the P2P state-sync path (matching the consensus path), and only apply the zero-check heuristic when reading legacy on-disk data that predates the `AllResources` variant.

The serializer side already encodes the information correctly (it emits `l1_data_gas` as a zero `ResourceLimits` for `L1Gas` transactions), so the fix is purely on the deserializer side. [8](#0-7) 

### Proof of Concept

1. Submit an invoke V3 transaction with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
2. The gateway computes `tx_hash_A` via `get_invoke_transaction_v3_hash` with `AllResources` → 3-element preimage including `L1_DATA_GAS_concat`.
3. The transaction is included in a block. The sequencer stores `tx_hash_A`.
4. A syncing node receives the block via P2P state sync. `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` → produces `L1Gas(l1_gas)`.
5. The syncing node computes `tx_hash_B` via `get_invoke_transaction_v3_hash` with `L1Gas` → 2-element preimage (no `L1_DATA_GAS_concat`).
6. `tx_hash_A ≠ tx_hash_B`. The syncing node's RPC returns `tx_hash_B` for this transaction, diverging from the canonical hash. [9](#0-8) [10](#0-9)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
