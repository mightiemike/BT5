Audit Report

## Title
SwapAllowlistExtension Gates on Router Address Instead of Originating EOA, Enabling Full Allowlist Bypass â€” (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist, but `sender` is bound to `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` intermediates the call, `sender` is the router address, not the originating EOA. A pool admin who allowlists the router to support standard periphery swaps inadvertently grants every user who can call the router the ability to bypass the per-user allowlist entirely.

## Finding Description

**Extension check:** `SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`. [1](#0-0) 

**Pool binding:** `MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, so `sender` is always the immediate caller of `pool.swap()`, not the originating EOA. [2](#0-1) 

**Router call:** `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly with no forwarding of the original `msg.sender` to the pool or extension. The router is `msg.sender` to the pool. [3](#0-2) 

The original EOA's address is stored only in transient storage via `_setNextCallbackContext` for the payment callback, and is never forwarded to the pool or extension. [4](#0-3) 

The same structural mismatch applies to `exactInput` (multi-hop) and `exactOutput`/`exactOutputSingle`, where the router is also `msg.sender` to every pool it calls. [5](#0-4) 

**Exploit flow:**
1. Pool admin deploys pool with `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` â€” alice is the only intended swapper.
3. Admin calls `setAllowedToSwap(pool, router, true)` â€” to allow alice to use the standard router.
4. Bob (not allowlisted) calls `router.exactInputSingle(...)`.
5. Router calls `pool.swap(...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, ...)`.
7. Extension checks `allowedSwapper[pool][router] == true` â†’ passes.
8. Bob's swap executes; alice's per-user entry is irrelevant.

**Existing guards are insufficient:** There is no mechanism in the extension, pool, or router to propagate the originating EOA to the extension. The `allowAllSwappers` flag is a separate bypass path and does not mitigate this issue.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of counterparties (e.g., KYC-verified addresses, institutional partners) loses that restriction entirely once the router is allowlisted. Any unprivileged user can execute swaps at the oracle mid-price through the router, extracting LP value at the configured spread with no recourse until the admin removes the router from the allowlist. This constitutes a direct loss of LP principal on pools whose economic model depends on counterparty curation, meeting the "broken core pool functionality causing loss of funds" and "admin-boundary break" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical production swap entry point. Pool admins operating curated pools will routinely allowlist it to avoid breaking the standard user experience. The mismatch between the admin's intent (allow the router as a trusted intermediary for allowlisted users) and the actual effect (allow all users) is non-obvious from the extension's interface. The trigger is a single `setAllowedToSwap(pool, router, true)` call, which is a normal operational action. The condition is reachable by any unprivileged user with no special capability.

## Recommendation
The extension must gate on the economically relevant actor â€” the originating user â€” not the immediate caller of the pool. Two sound approaches:

1. **Pass the original sender explicitly:** Extend the pool's `swap` signature or the extension interface to carry the original sender (e.g., a user-supplied `onBehalfOf` address validated by the router via a signed permit), and check that address in the extension.
2. **Router-level enforcement:** Have the router verify allowlist membership before calling the pool, and have the extension reject router-mediated calls unless the router attests the user via a trusted mechanism (e.g., a signed message or a dedicated router allowlist check).
3. **Check `tx.origin` as a fallback:** For the allowlist extension specifically, check `tx.origin` when `sender` is a known trusted router, so the extension sees the EOA that initiated the transaction. (Note: this approach has its own tradeoffs with smart contract wallets.)
