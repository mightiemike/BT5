### Title
`SwapAllowlistExtension` Gates the Router Address Instead of the Actual Swapper, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which the pool sets to `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` inside the pool is the **router contract**, not the user. The allowlist therefore gates the router's address, not the actual economic actor. Any pool admin who allowlists the router to support router-mediated swaps simultaneously opens the pool to every user on-chain, defeating the curation entirely.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first positional argument to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the value forwarded above: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap`: [4](#0-3) 

So the extension receives `sender = address(router)` and evaluates `allowedSwapper[pool][router]`. The actual user who initiated the trade is never inspected.

There are two immediate consequences:

1. **Allowlist bypass**: If the pool admin allowlists the router address (the natural step to let allowlisted users trade via the router), every user on-chain can bypass the allowlist by routing through `MetricOmmSimpleRouter`.
2. **Allowlisted users locked out of the router**: If the admin allowlists only individual user addresses, those users cannot trade through the router at all, because the extension sees the router address and reverts.

The same structural problem applies to the `exactInput` multi-hop path and `exactOutputSingle`/`exactOutput` paths, all of which call `pool.swap` with `msg.sender = router`. [5](#0-4) 

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` is designed to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, institutional LPs, or protocol-controlled addresses). Once the router is allowlisted, any unpermissioned user can execute swaps at oracle-anchored prices, extracting value from LP positions through adverse selection or simply trading in a pool that was never intended to be public. LP principal is directly at risk because the pool's spread and stop-loss parameters were calibrated for a controlled counterparty set.

---

### Likelihood Explanation

Any pool admin who deploys a curated pool and also wants to support the standard periphery router will allowlist the router address. This is the expected operational pattern — the router is the primary user-facing entry point. The bypass is therefore triggered by normal, correct-seeming admin configuration, not by an exotic edge case.

---

### Recommendation

The pool must pass the original user's address to the extension, not the intermediary's address. Two complementary fixes:

1. **Router-side**: `MetricOmmSimpleRouter` should pass the originating user (`msg.sender` at the router level) as a verified field inside `extensionData`, and the extension should decode and verify it. However, this is forgeable unless the pool enforces it.

2. **Pool-side (preferred)**: Add a `swapper` parameter to `IMetricOmmPoolActions.swap` that the pool passes through to `_beforeSwap` as the identity to gate. The router sets this to `msg.sender` (the user); direct callers set it to themselves. The extension then checks `allowedSwapper[pool][swapper]` instead of `allowedSwapper[pool][sender]`.

Alternatively, the extension can be redesigned to check `recipient` (the address receiving output tokens) if that is the intended economic actor, but the cleanest fix is an explicit `swapper` field that the pool validates equals `msg.sender` for direct calls and is set by the router for mediated calls.

---

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls swapExtension.setAllowedToSwap(pool, address(router), true)
    (to allow alice, who is allowlisted, to use the router)
  - Pool admin calls swapExtension.setAllowedToSwap(pool, alice, true)

Attack:
  - Bob (not allowlisted) calls:
      router.exactInputSingle({pool: pool, recipient: bob, ...})
  - Router calls pool.swap(bob, ...) — msg.sender inside pool = router
  - Pool calls _beforeSwap(router, bob, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  - Bob's swap executes successfully despite not being on the allowlist

Result:
  - Bob trades at oracle-anchored prices in a pool intended only for allowlisted counterparties
  - LP funds are exposed to uncontrolled adverse selection
```

The existing test `test_allowedSwapSucceeds` in `FullMetricExtensionTest` allowlists `address(callers[0])` (the direct pool caller, a `TestCaller` contract), which masks the bug because the test never routes through `MetricOmmSimpleRouter`. [6](#0-5)
