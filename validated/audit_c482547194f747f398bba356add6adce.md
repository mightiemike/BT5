### Title
Freed Storage Deposits Permanently Locked in Contract After GC — (`contract/src/lib.rs`)

---

### Summary

The `submit_blocks` function requires callers to attach NEAR tokens to cover on-chain storage costs for new block headers. When `run_mainchain_gc` removes old headers (freeing that storage), the NEAR tokens originally deposited to pay for those headers are not returned to anyone. They accumulate in the contract's balance permanently, because the contract exposes no `withdraw`, `rescue`, or `reclaim_storage_deposit` function.

---

### Finding Description

`submit_blocks` is marked `#[payable]` and implements a storage-deposit pattern:

1. It records `initial_storage = env::storage_usage()` before processing.
2. It inserts all submitted headers into `headers_pool`, `mainchain_height_to_header`, and `mainchain_header_to_height`.
3. It immediately calls `self.run_mainchain_gc(num_of_headers)`, which **removes** old headers once `amount_of_headers_we_store > gc_threshold`.
4. It computes `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` — the **net** storage delta after both insertions and deletions.
5. It charges the caller only for the net positive delta and refunds any excess. [1](#0-0) 

The critical invariant broken here is that the **gross** storage freed by GC is never returned to the depositors who originally paid for it. When GC removes a block header, three entries are deleted:

- `headers_pool.remove(hash)` — removes the `ExtendedHeader`
- `mainchain_header_to_height.remove(hash)` — removes the height mapping
- `mainchain_height_to_header.remove(height)` — removes the hash mapping [2](#0-1) [3](#0-2) 

Each deletion frees storage bytes, converting the previously staked NEAR tokens into liquid contract balance. But because `diff_storage_usage` uses `saturating_sub`, a net-negative delta (GC freed more than was added) collapses to zero — the current caller pays nothing and receives nothing. The freed tokens simply remain in the contract account.

A grep across all of `contract/src/` for `withdraw`, `rescue`, `reclaim`, and `refund_storage` returns **zero matches**. There is no function in the contract that can move this accumulated balance out.

`run_mainchain_gc` is also exposed as a standalone public method callable by any unprivileged NEAR account (it is only gated by the `#[pause]` macro, with `UnrestrictedRunGC` as a bypass role): [4](#0-3) 

Any caller can invoke `run_mainchain_gc` directly, triggering storage deletion and locking the freed NEAR tokens without any refund path.

---

### Impact Explanation

Every block header removed by GC represents NEAR tokens that were deposited by a relayer and are now permanently unrecoverable. With the recommended `gc_threshold = 52704` (approximately one year of Bitcoin blocks) and a storage deposit on the order of ~500 milliNEAR per block (as used in the integration tests), the contract will accumulate tens of thousands of NEAR tokens over its operational lifetime with no mechanism to recover them. The only theoretical escape is a DAO-approved contract upgrade that adds a withdrawal function — a privileged, non-trivial operation that is not guaranteed to happen.

---

### Likelihood Explanation

This is not a corner case. GC is triggered automatically inside every `submit_blocks` call once the stored chain exceeds `gc_threshold`. In steady-state operation (the relayer submitting ~144 blocks/day), GC fires on every single submission batch. The locked-token accumulation is therefore continuous and grows monotonically from the moment the chain reaches the GC threshold.

---

### Recommendation

Add a privileged withdrawal function (restricted to `Role::DAO`) that transfers the contract's excess liquid balance (i.e., `env::account_balance() - env::storage_byte_cost() * env::storage_usage()`) to a designated treasury account. This mirrors the fix applied in the referenced M06 pull request, which added a recovery path for stuck assets.

---

### Proof of Concept

1. Relayer submits 52,705 blocks over time, each batch with a NEAR deposit. The contract stores exactly `gc_threshold + 1` headers.
2. On the next `submit_blocks` call (batch of N headers), `run_mainchain_gc(N)` removes N old headers, freeing their storage.
3. `diff_storage_usage = env::storage_usage().saturating_sub(initial_storage)` — if GC freed ≥ the bytes added by the new N headers, this is 0.
4. `required_deposit = 0`, so the caller pays nothing and receives nothing.
5. The NEAR tokens that originally paid for the N removed headers are now liquid in the contract's balance.
6. No function exists to withdraw them. They are permanently locked. [5](#0-4) [6](#0-5)
