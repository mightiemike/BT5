### Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router contract**, not the end user. If the router is allowlisted (so that legitimate users can use it), every user — including those the pool admin explicitly excluded — can bypass the allowlist by routing through the router.

---

### Finding Description

**Call path for a direct swap:**
```
User → pool.swap(...)
  msg.sender = User
  _beforeSwap(sender=User, ...)
  SwapAllowlistExtension.beforeSwap(sender=User, ...)
  → checks allowedSwapper[pool][User]   ✓ correct actor
```

**Call path via MetricOmmSimpleRouter:**
```
User → MetricOmmSimpleRouter.exactInputSingle(params)
  → pool.swap(params.recipient, ...)    // router is msg.sender to pool
    msg.sender = Router
    _beforeSwap(sender=Router, ...)
    SwapAllowlistExtension.beforeSwap(sender=Router, ...)
    → checks allowedSwapper[pool][Router]  ✗ wrong actor
```

In `MetricOmmPool.swap`, the pool passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, i.e., `allowedSwapper[pool][router]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making itself `msg.sender` to the pool: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`. [5](#0-4) 

This creates an irresolvable dilemma for the pool admin:

| Router allowlist state | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all |
| Router **allowlisted** | Every user (including blocked ones) can bypass the allowlist via the router |

There is no configuration that simultaneously allows allowlisted users to use the router and blocks non-allowlisted users from doing the same.

---

### Impact Explanation

On a curated pool where `SwapAllowlistExtension` is configured to restrict swaps to specific addresses (e.g., KYC'd counterparties, institutional traders, or whitelisted market makers), any unprivileged user can execute swaps by routing through the public `MetricOmmSimpleRouter`. This:

- Directly violates the pool's access-control invariant
- Allows unauthorized parties to trade against LP liquidity, potentially extracting value through unfavorable price execution that the allowlist was designed to prevent
- Constitutes a broken core pool functionality: the configured guard fails open for the entire public router path

---

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any user can call it
- No special setup is required; the attacker simply calls `exactInputSingle` or `exactInput` targeting the allowlisted pool
- The bypass is unconditional once the router is allowlisted (which is the only way legitimate users can use the router)
- Likelihood: **High**

---

### Recommendation

The extension must gate the **end user**, not the intermediate caller. Options:

1. **Preferred:** Have the router encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and check it when the immediate caller is a known router. This requires a trusted router registry or a signed-sender pattern.

2. **Alternative:** Add a `recipient`-based check in addition to `sender`, or require the pool to be called directly (no router support for allowlisted pools), and document this restriction explicitly.

3. **Minimum:** Document that pools using `SwapAllowlistExtension` must not allowlist the router address, and that allowlisted users must call the pool directly — but this removes router usability for curated pools entirely.

---
