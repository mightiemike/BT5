### Title
Bonus Rewards Permanently Lost When Bonus Pool Is Insufficient During Redelegation Slash - (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

In `slashRedelegationPosition`, when the bonus pool is insufficient, the `ErrInsufficientBonusPool` error from `processEventsAndClaimBonus` is deliberately swallowed to prevent a chain halt. However, `processEventsAndClaimBonus` has already advanced the position's `LastBonusAccrual` checkpoint and `LastEventSeq` in-memory **before** returning the error. The caller then persists this modified state via `setPositionWithState`. The result is that the position owner's accrued bonus rewards for the entire period since their last claim are permanently erased without payment — an exact structural analog to the GMX fee-receiver-not-paid bug.

### Finding Description

`processEventsAndClaimBonus` modifies the `pos *types.PositionState` pointer in-place throughout its execution. Critically, `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` are called **before** the `sufficientBonusPoolBalance` check: [1](#0-0) 

When `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`, the function returns `nil, err` — but `pos.Position.LastBonusAccrual` has already been advanced to `blockTime` (or `ExitUnlockAt`), and `pos.LastEventSeq` has been incremented for every processed event.

In `slashRedelegationPosition`, this error is explicitly caught and swallowed: [2](#0-1) 

Execution then continues with the already-mutated `pos`, and `setPositionWithState` is called unconditionally in both the full-slash and partial-slash branches: [3](#0-2) 

This persists the advanced checkpoint to the KVStore. On the next claim, `segmentStart = pos.LastBonusAccrual` will be set to the already-advanced time, so the skipped period is never re-computed. The bonus rewards for the entire accrual window are permanently lost.

### Impact Explanation

A position owner whose validator is slashed during an active redelegation, at a time when the `RewardsPoolName` module account balance is insufficient, permanently loses all accrued bonus rewards since their last claim. The `LastBonusAccrual` checkpoint advances as if payment occurred, but no coins are transferred. The base staking rewards (via `x/distribution`) are unaffected; only the tiered bonus APY component is lost.

The corrupted value is: `position.LastBonusAccrual` (advanced without payment) and `position.LastEventSeq` (consumed without reward settlement), stored in the tieredrewards KVStore.

### Likelihood Explanation

The trigger requires two concurrent conditions:
1. A position has an active redelegation entry (normal user behavior — `MsgTierRedelegate` is a supported message).
2. The `RewardsPoolName` module account is depleted at the time of the slash.

The bonus pool can be depleted through normal high-volume claiming activity. Validator slashing during redelegation is a standard Cosmos SDK protocol event. Neither condition requires privileged access. An attacker who can influence pool depletion timing (e.g., by front-running a known slash with mass claims) can reliably trigger this for targeted positions.

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the `sufficientBonusPoolBalance` check succeeds and coins are actually sent. This ensures the checkpoint only advances when payment is confirmed. Alternatively, if the pool is insufficient, the error should propagate without mutating the position state, so the accrual window is preserved for a future claim.

### Proof of Concept

1. User creates a tiered position delegating to validator V1 via `MsgLockTier`.
2. User calls `MsgTierRedelegate` to move the delegation from V1 to V2, creating an active redelegation entry tracked in `redelegation_mapping`.
3. The `RewardsPoolName` module account is depleted (e.g., by other users claiming large bonuses).
4. V1 is slashed during the redelegation window. The staking module fires `BeforeRedelegationSlashed`, which calls `slashRedelegationPosition`.
5. Inside `slashRedelegationPosition`, `processEventsAndClaimBonus` is called. It computes `totalBonus > 0`, advances `pos.LastBonusAccrual` to `blockTime`, then hits `sufficientBonusPoolBalance` and returns `ErrInsufficientBonusPool`.
6. The error is swallowed at [4](#0-3) 
7. `setPositionWithState` persists the mutated `pos` with the advanced checkpoint. [5](#0-4) 
8. When the user later calls `MsgClaimTierRewards`, `segmentStart = pos.LastBonusAccrual` is already at `blockTime` from step 5. The entire accrual window is gone. The user receives zero bonus for that period despite having earned it.
