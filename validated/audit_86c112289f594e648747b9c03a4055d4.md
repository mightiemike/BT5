### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing 64-Byte Merkle Forgery Protection — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` was introduced specifically to mitigate the 64-byte transaction Merkle proof forgery vulnerability by requiring a coinbase proof of equal depth. However, the original `verify_transaction_inclusion` (v1) remains a live, publicly callable NEAR function. Any unprivileged caller can invoke v1 directly, bypassing the coinbase proof guard entirely and obtaining a `true` result for a fraudulent internal-node "transaction."

### Finding Description

`verify_transaction_inclusion_v2` enforces two guards before delegating to v1:

1. `merkle_proof.len() == coinbase_merkle_proof.len()` — ensures the claimed tx is at the same tree depth as the coinbase.
2. A coinbase Merkle proof check against the stored `merkle_root` — confirms the coinbase is a real leaf at that depth. [1](#0-0) 

After both guards pass, v2 delegates to v1 via `self.verify_transaction_inclusion(args.into())`. [2](#0-1) 

However, v1 is still decorated only with `#[deprecated]` and `#[pause]` — neither of which restricts runtime access by unprivileged callers. [3](#0-2) 

The `#[deprecated]` attribute in Rust emits a compiler warning only; it has no effect on NEAR's runtime dispatch. Any NEAR account can call `verify_transaction_inclusion` directly, skipping the coinbase proof check entirely.

v1's only proof check is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [4](#0-3) 

`compute_root_from_merkle_proof` is purely positional — it does not distinguish between a leaf hash and an internal-node hash. [5](#0-4) 

An attacker who supplies a 64-byte internal Merkle node as `tx_id`, with the correct `tx_index` (its position at that tree level) and the corresponding sibling path as `merkle_proof`, will produce a root that matches `header.block_header.merkle_root`, causing v1 to return `true` for a transaction that was never broadcast or mined.

### Impact Explanation

Any downstream NEAR contract that calls `verify_transaction_inclusion` (v1) — or that an attacker tricks into calling v1 — will accept a fraudulent Bitcoin transaction inclusion proof. The light client is the canonical on-chain source of truth for Bitcoin SPV verification; a forged `true` result breaks the core security guarantee of the system and can enable theft or unauthorized actions in any protocol that gates value on Bitcoin transaction confirmation.

### Likelihood Explanation

Bitcoin blocks are fully public. An attacker can inspect any block's transaction list, compute the Merkle tree, identify any internal 64-byte node, and derive its sibling path. No privileged access, leaked keys, or social engineering is required. The call to v1 is a standard NEAR function call available to any account.

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the public interface entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. The `#[deprecated]` annotation is insufficient — it must be replaced with either deletion or a hard `require!(false, "use verify_transaction_inclusion_v2")` guard, or an ACL role check, to prevent direct bypass of the coinbase proof mitigation.

### Proof of Concept

1. Identify a real Bitcoin block already submitted to the light client (e.g., height 685452, whose header is stored on-chain).
2. Obtain the block's full transaction list and compute its Merkle tree.
3. Select any internal node `N` at depth `d` (where `d < tree_height`). `N` is the double-SHA256 of two child hashes concatenated — exactly 64 bytes of input, satisfying the 64-byte attack precondition.
4. Compute the sibling path from `N` up to the root (length = `tree_height - d` hashes).
5. Call `verify_transaction_inclusion` with:
   - `tx_id` = `N` (the internal node hash)
   - `tx_block_blockhash` = the real block hash
   - `tx_index` = the position of `N` at level `d`
   - `merkle_proof` = the sibling path computed in step 4
   - `confirmations` = any value ≤ `gc_threshold`
6. `compute_root_from_merkle_proof(N, tx_index, sibling_path)` reproduces the stored `merkle_root` exactly, so the function returns `true` — confirming a transaction that does not exist. [6](#0-5) [5](#0-4)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
