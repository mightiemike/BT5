### Title
Transaction Displacement Enables Acceptance of Future-Timestamped Bitcoin Block Headers - (File: `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/zcash.rs`)

### Summary
The `check_pow()` function validates that a submitted Bitcoin block header's timestamp is not more than `MAX_FUTURE_BLOCK_TIME_LOCAL` (7200 seconds / 2 hours) ahead of the NEAR execution block's timestamp via `env::block_timestamp_ms()`. Because this reference time is sampled at transaction execution rather than at submission, a transaction displacement attack can delay a `submit_blocks()` call long enough for a block header whose timestamp exceeds the 2-hour window at submission time to pass the check at execution time.

### Finding Description
In `contract/src/bitcoin.rs` (and identically in `contract/src/litecoin.rs` and `contract/src/zcash.rs`), the future-timestamp guard reads:

```rust
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
``` [1](#0-0) 

`MAX_FUTURE_BLOCK_TIME_LOCAL` is defined as `2 * 60 * 60 = 7200` seconds. [2](#0-1) 

`env::block_timestamp_ms()` returns the timestamp of the NEAR block in which the transaction is **executed**, not the timestamp at which the transaction was **submitted**. If a `submit_blocks()` transaction sits in the NEAR mempool and is displaced (delayed) by `Δ` seconds before inclusion, the effective allowed future window expands to `2 hours + Δ`. A block header with `block_header.time = T_submission + 2h + Δ` would be rejected if executed immediately but accepted after the delay.

The same pattern is present in all three chain-specific `check_pow()` implementations: [3](#0-2) [4](#0-3) 

The corrupted header, once accepted, is stored permanently in `headers_pool` and promoted to the mainchain if its `chain_work` exceeds the current tip. [5](#0-4) 

Downstream callers of `verify_transaction_inclusion()` or `verify_transaction_inclusion_v2()` then operate against a chain state that includes a header Bitcoin full nodes would reject. [6](#0-5) 

### Impact Explanation
A successfully displaced transaction causes the light client to permanently record a Bitcoin block header whose timestamp violates Bitcoin's `time-too-new` consensus rule. Any SPV proof verified against that header — or any descendant block built on top of it — returns `true` from `verify_transaction_inclusion_v2()` for transactions in a chain segment that Bitcoin full nodes do not recognise as canonical. Recipient contracts consuming the verification result would act on a forged inclusion proof.

### Likelihood Explanation
`submit_blocks()` is gated by `#[trusted_relayer]`, which restricts callers to staked, approved relayers or accounts holding `Role::UnrestrictedSubmitBlocks` / `Role::DAO`. [7](#0-6) [8](#0-7) 

This limits the realistic attacker to a compromised or malicious trusted relayer. Such a relayer can craft a header with `time = env::block_timestamp_ms()/1000 + 7200 + Δ`, submit the transaction, and simply wait `Δ` seconds for a later NEAR block to include it — no external displacement is even required; the relayer controls submission timing. Likelihood is therefore **medium-low**: the trusted-relayer barrier is real, but a single compromised relayer is sufficient and no additional on-chain privileges are needed beyond that role.

### Recommendation
Add a `deadline` parameter to `submit_blocks()` (a NEAR block timestamp in milliseconds) and revert at the start of execution if `env::block_timestamp_ms() > deadline`. This mirrors the fix recommended in the external report and ensures the future-timestamp window is anchored to the moment the relayer decided to submit, not to whenever the NEAR network happens to include the transaction.

```rust
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
    deadline_ms: u64,   // <-- new parameter
) -> PromiseOrValue<()> {
    require!(
        env::block_timestamp_ms() <= deadline_ms,
        "Transaction executed after deadline"
    );
    // ... existing logic
}
```

### Proof of Concept
1. At NEAR block time `T` (ms), a trusted relayer constructs a Bitcoin header with `time = T/1000 + 7201` (one second beyond the 2-hour window).
2. The relayer submits `submit_blocks([header])` to NEAR.
3. The relayer deliberately delays re-submission or waits for natural inclusion lag of ≥ 1 second (trivially achievable).
4. The transaction is included in a NEAR block at time `T + 1001 ms` (i.e., `current_timestamp = T/1000 + 1`).
5. The check evaluates: `7201 <= (T/1000 + 1) + 7200` → `7201 <= 7201` → **passes**.
6. The header is stored via `store_block_header()` / `store_fork_header()` and, if its `chain_work` is highest, promoted to the mainchain tip.
7. A subsequent call to `verify_transaction_inclusion_v2()` referencing this header returns `true` for any transaction whose Merkle proof the attacker controls, producing a forged SPV result accepted by any downstream contract.

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
