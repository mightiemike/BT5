### Title
`AddMod`, `MulMod`, `RangeCheck96` Builtin Counts Silently Zeroed in P2P Protobuf Receipt Round-Trip — (`crates/apollo_protobuf/src/converters/receipt.rs`)

### Summary

The protobuf `BuiltinCounter` message and its Rust converters omit `AddMod`, `MulMod`, and `RangeCheck96`. Any node syncing via P2P stores receipts with those three builtin counts permanently zeroed, and subsequently serves authoritative-looking wrong `execution_resources` via `starknet_getTransactionReceipt` for every transaction that used those builtins.

### Finding Description

**Proto schema gap.** The `BuiltinCounter` message in `receipt.proto` defines only 8 fields (`bitwise`, `ecdsa`, `ec_op`, `pedersen`, `range_check`, `poseidon`, `keccak`, `output`) and carries an explicit developer acknowledgement:

```
// TODO(alonl): add the missing builtins
``` [1](#0-0) 

**Serialization side drops the three builtins.** `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` only writes 7 builtins and hardcodes `output: 0`. `AddMod`, `MulMod`, and `RangeCheck96` are never read from the map: [2](#0-1) 

**Deserialization side never inserts them.** `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` inserts only 7 builtins plus `SegmentArena=0`. `AddMod`, `MulMod`, and `RangeCheck96` are absent: [3](#0-2) 

**The P2P sync test explicitly acknowledges the breakage.** The transaction sync test has two commented-out assertions with the note:

```rust
// TODO(alonl): Uncomment this once we fix protobuf conversion for receipt builtins.
// assert_eq!(actual_transaction_outputs, transaction_outputs);
``` [4](#0-3) 

**The receipt commitment does NOT cover builtin counts.** `calculate_receipt_hash` chains only `transaction_hash`, `actual_fee`, `messages_sent` hash, `revert_reason` hash, and `gas_consumed` (l1/l2/data gas vectors). Individual builtin counts are not hashed: [5](#0-4) 

This means the receipt commitment stored in the block header will still match after the lossy round-trip, so no integrity check catches the corruption. The wrong data is stored silently.

**The `ExecutionResources` struct in `starknet_api` fully supports all three builtins**, and the storage serializer assigns them discriminants 8, 9, 10: [6](#0-5) 

### Impact Explanation

A node syncing via P2P will store `AddMod=0`, `MulMod=0`, `RangeCheck96=0` in every receipt for transactions that actually used those builtins. When that node serves `starknet_getTransactionReceipt` (or `starknet_getBlockWithReceipts`), it returns an authoritative-looking but wrong `execution_resources.builtin_instance_counter`.

**Important correction to the question's framing:** The actual fee charged is computed from `GasVector` (l1_gas, l1_data_gas, l2_gas) by the blockifier at execution time — not from individual builtin counts in the stored receipt. Therefore, fee/gas accounting is **not** affected. The impact is confined to wrong RPC receipt data. [7](#0-6) 

### Likelihood Explanation

- `add_mod`, `mul_mod`, and `range_check96` are production builtins used by Cairo programs compiled with recent toolchains (they appear in the OS builtins struct and bouncer config).
- Any node running in P2P sync mode (not central/feeder-gateway sync) is affected for every such transaction.
- No privilege is required: any user can submit a transaction that exercises these builtins.
- The bug is already acknowledged by developers in both the proto file and the test file.

### Recommendation

1. Add `add_mod`, `mul_mod`, and `range_check96` fields to the `BuiltinCounter` proto message in `receipt.proto`.
2. Add corresponding mappings in both `From<HashMap<Builtin, u64>> for ProtobufBuiltinCounter` and `TryFrom<ProtobufBuiltinCounter> for HashMap<Builtin, u64>` in `receipt.rs`.
3. Uncomment the disabled assertions in `transaction_test.rs` once the fix is in place.

### Proof of Concept

```rust
use std::collections::HashMap;
use starknet_api::execution_resources::Builtin;
// type alias as in receipt.rs
type ProtobufBuiltinCounter = protobuf::receipt::execution_resources::BuiltinCounter;

let mut original = HashMap::new();
original.insert(Builtin::AddMod, 5u64);
original.insert(Builtin::MulMod, 3u64);
original.insert(Builtin::RangeCheck96, 7u64);

// Serialize to protobuf
let proto: ProtobufBuiltinCounter = original.clone().into();
// proto has no add_mod/mul_mod/range_check96 fields → all default to 0

// Deserialize back
let reconstructed = HashMap::<Builtin, u64>::try_from(proto).unwrap();

// These assertions PASS, demonstrating the silent data loss:
assert_eq!(reconstructed.get(&Builtin::AddMod).copied().unwrap_or(0), 0);   // was 5
assert_eq!(reconstructed.get(&Builtin::MulMod).copied().unwrap_or(0), 0);   // was 3
assert_eq!(reconstructed.get(&Builtin::RangeCheck96).copied().unwrap_or(0), 0); // was 7
```
