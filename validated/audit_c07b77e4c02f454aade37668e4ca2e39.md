### Title
`CongestionInfo::remove_buffered_receipt_gas` underflow when plain `Receipt` congestion gas is recomputed with post-upgrade `RuntimeConfig` at forwarding time — (`runtime/runtime/src/congestion_control.rs`)

---

### Summary

When a receipt is buffered in the outgoing buffer as a plain `Receipt` (i.e., before `use_state_stored_receipt` was enabled by a protocol upgrade), its congestion gas is computed with the `RuntimeConfig` at buffer time and added to `own_congestion_info.buffered_receipts_gas`. When the receipt is later forwarded after a protocol upgrade that raises fee parameters, `receipt_congestion_gas` recomputes the gas using the **current** `RuntimeConfig`, yielding a larger value. The subsequent `remove_buffered_receipt_gas` call then fails with `RuntimeError::UnexpectedIntegerOverflow` because the stored aggregate is smaller than the recomputed removal amount, propagating a fatal error through chunk application.

---

### Finding Description

**Push path — gas computed with config at buffer time:**

In `buffer_receipt`, when `use_state_stored_receipt = false`, the receipt is stored as a plain `Receipt` (not `StateStoredReceipt::V1`): [1](#0-0) 

The gas `G1 = compute_receipt_congestion_gas(receipt, config_old)` is added to `own_congestion_info.buffered_receipts_gas`. No metadata is persisted alongside the receipt.

**Pop path — gas recomputed with current config:**

In `forward_from_buffer_to_shard`, `receipt_congestion_gas` is called on the buffered receipt: [2](#0-1) 

For a plain `Receipt`, this dispatches to `compute_receipt_congestion_gas(receipt, config)` using the **current** `RuntimeConfig`: [3](#0-2) 

The result `G2` is then used to decrement the stored aggregate: [4](#0-3) 

**The invariant break:**

`remove_buffered_receipt_gas` uses `checked_sub` and returns `UnexpectedIntegerOverflow` if the subtraction underflows: [5](#0-4) 

If a protocol upgrade between buffer time and forward time raises any fee parameter consumed by `action_receipt_congestion_gas` (e.g., `ActionCosts::new_action_receipt` exec fee, or any per-action send/exec fee), then `G2 > G1`, and `buffered_receipts_gas - G2` underflows.

**Why `StateStoredReceipt::V1` is immune but plain `Receipt` is not:**

`StateStoredReceipt::V1` stores `congestion_gas` in its `StateStoredReceiptMetadata` at push time and reads it back at pop time — the config is never consulted again: [6](#0-5) 

Plain `Receipt` has no such metadata. `receipt_congestion_gas` always recomputes from the live config. The `should_update_outgoing_metadatas()` predicate is `false` for plain `Receipt`, so the `ReceiptGroupsQueue` is also not updated, leaving no secondary record of the original gas: [7](#0-6) 

---

### Impact Explanation

`forward_from_buffer_to_shard` returns `Result<(), RuntimeError>`. A returned `RuntimeError::UnexpectedIntegerOverflow` propagates through `forward_from_buffer` and out of `apply`. In nearcore, a `RuntimeError` from `apply` marks the chunk as invalid. Because all validators apply the same receipts against the same upgraded `RuntimeConfig`, every validator encounters the identical error on the same chunk, causing the affected shard to halt chunk production. The `CongestionInfo` is also left in a partially-mutated state (`receipt_bytes` already decremented, `buffered_receipts_gas` not decremented) before the error is returned.

**Severity: High** — shard-level chunk application halt, deterministic across all validators.

---

### Likelihood Explanation

Three conditions must coincide:

1. **Receipts buffered as plain `Receipt`**: This occurs during any epoch where `use_state_stored_receipt = false` in the `RuntimeConfig`. Receipts buffered in that window remain as plain `Receipt` objects in the trie indefinitely.
2. **A subsequent protocol upgrade raises fee parameters**: NEAR protocol has historically adjusted action fees across protocol versions. Any increase to `ActionCosts::new_action_receipt`, per-action exec fees, or send fees is sufficient.
3. **The buffer is not fully drained before the upgrade**: A congested receiving shard can hold receipts in the outgoing buffer for many consecutive chunks, easily spanning a protocol upgrade epoch boundary.

All three conditions are reachable through normal protocol operation without any privileged or adversarial action beyond submitting ordinary transactions to a congested shard.

---

### Recommendation

The gas value used to decrement `buffered_receipts_gas` must be the same value that was used to increment it — i.e., the gas computed at buffer time with the config that was active then. Two approaches:

1. **Preferred**: Require that all receipts entering the outgoing buffer are stored as `StateStoredReceipt::V1` (with `congestion_gas` in metadata) before any protocol version that changes fee parameters. Enforce this by gating the fee-change protocol version behind the `use_state_stored_receipt` protocol version, ensuring the buffer is fully drained of plain `Receipt` objects first.

2. **Defensive fallback**: In `forward_from_buffer_to_shard`, when the receipt is a plain `Receipt`, cap the removal amount at the current `buffered_receipts_gas` rather than using the recomputed value, and emit a warning metric for the discrepancy. This prevents the underflow while accepting a small accounting inaccuracy during the transition window.

---

### Proof of Concept

```
Epoch E  (protocol version N, use_state_stored_receipt = false):
  - User submits Tx → creates ActionReceipt R targeting congested shard S
  - R is buffered as plain Receipt (no metadata)
  - G1 = compute_receipt_congestion_gas(R, config_N)   // e.g. 500 Tgas
  - own_congestion_info.buffered_receipts_gas += G1     // = 500 Tgas

Epoch E+k (protocol version N+m, fee for new_action_receipt increased):
  - Shard S still congested; R still in outgoing buffer
  - forward_from_buffer_to_shard called
  - G2 = receipt_congestion_gas(R, config_{N+m})        // e.g. 600 Tgas (fees raised)
  - own_congestion_info.remove_buffered_receipt_gas(G2)
      → checked_sub(600, 500) = None
      → Err(RuntimeError::UnexpectedIntegerOverflow("remove_buffered_receipt_gas"))
  - apply() returns RuntimeError
  - All validators reject the chunk → shard S halts
```

The exact divergent Borsh-level value is the `delayed_receipts_gas` / `buffered_receipts_gas` field of the `CongestionInfoV1` struct stored in the chunk header, which encodes `G1` but is decremented by `G2 > G1`. [8](#0-7)

pub(crate) fn compute_receipt_congestion_gas(
    receipt: &Receipt,
    config: &RuntimeConfig,
) -> Result<Gas, IntegerOverflowError> {
    match receipt.versioned_receipt() {
        VersionedReceiptEnum::Action(action_receipt) => {
            // account for gas guaranteed to be used for executing the receipts
            action_receipt_congestion_gas(receipt, config, action_receipt.into())
        }
        VersionedReceiptEnum::Data(_data_receipt) => {
            // Data receipts themselves don't cost gas to execute, their cost is
            // burnt at creation. What we should count, is the gas of the
            // postponed action receipt. But looking that up would require
            // reading the postponed receipt from the trie.
            // Thus, the congestion control MVP does not account for data
            // receipts or postponed receipts.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::PromiseYield(_) => {
            // The congestion control MVP does not account for yielding a
            // promise. Yielded promises are confined to a single account, hence
            // they never cross the shard boundaries. This makes it irrelevant
            // for the congestion MVP, which only counts gas in the outgoing
            // buffers and delayed receipts queue.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::PromiseResume(_) => {
            // The congestion control MVP does not account for resuming a promise.
            // Unlike `PromiseYield`, it is possible that a promise-resume ends
            // up in the delayed receipts queue.
            // But similar to a data receipt, it would be difficult to find the cost
            // of it without expensive state lookups.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::GlobalContractDistribution(_) => Ok(Gas::ZERO),
    }
}

    pub fn remove_buffered_receipt_gas(&mut self, gas: u128) -> Result<(), RuntimeError> {
        match self {
            CongestionInfo::V1(inner) => {
                inner.buffered_receipts_gas =
                    inner.buffered_receipts_gas.checked_sub(gas).ok_or_else(|| {
                        RuntimeError::UnexpectedIntegerOverflow(
                            "remove_buffered_receipt_gas".into(),
                        )
                    })?;
            }
        }
        Ok(())
    }
