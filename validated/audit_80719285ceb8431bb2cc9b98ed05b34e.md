Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual User, Enabling Complete Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the direct caller of `pool.swap()` — the router address when swaps are routed through `MetricOmmSimpleRouter`. When a pool admin allowlists the router (the natural step to enable router-mediated swaps for curated users), every unprivileged user can bypass the per-user allowlist by routing through the router, because the extension checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actualUser]`. This completely nullifies the allowlist's curation policy and allows non-allowlisted users to execute swaps against the pool.

## Finding Description
**Root cause — wrong identity checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
`msg.sender` here is the pool (the caller of the extension), and `sender` is whatever `MetricOmmPool.swap` passed as the first argument to `_beforeSwap`.

**Pool passes `msg.sender` (the router) as `sender`:**

In `MetricOmmPool.swap`, the call is:
```solidity
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient, ...
);
```
`ExtensionCalling._beforeSwap` forwards this `sender` value unchanged into `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

**Router never forwards the original user's address:**

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` only in transient callback storage for payment settlement, then calls `pool.swap()` directly:
```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
IMetricOmmPoolActions(params.pool).swap(params.recipient, ..., params.extensionData);
```
The pool's `msg.sender` is always the router contract. The original user's address is never visible to the pool or the extension.

**The invariant break:**

The pool admin cannot simultaneously (1) allow router-mediated swaps by allowlisting the router and (2) enforce per-user allowlist restrictions. Once `allowedSwapper[pool][router] == true`, every user — including non-allowlisted ones — passes the check by routing through the router. The extension never evaluates `allowedSwapper[pool][actualUser]`.

**Existing guards are insufficient:**

There is no mechanism in the extension, pool, or router to recover the original user's identity. The `extensionData` field is passed through unchanged from router to pool to extension, but the router does not encode the original caller into it.

## Impact Explanation
A curated pool configured with `SwapAllowlistExtension` to restrict swaps to specific users (e.g., KYC-verified, institutional, or whitelisted counterparties) is completely bypassable by any unprivileged user routing through `MetricOmmSimpleRouter`. Non-allowlisted users can execute swaps against the pool at oracle-anchored prices, draining LP assets and violating the pool's curation policy. This constitutes direct loss of LP principal and broken core pool functionality (allowlist enforcement), meeting the contest's Critical/High threshold for broken core pool functionality causing loss of funds.

## Likelihood Explanation
The admin would naturally allowlist the router to allow allowlisted users to use the standard periphery — this is the expected and documented usage path. No special privileges or setup are required for the exploit beyond the admin's reasonable configuration. Any user can call `router.exactInputSingle()` targeting the curated pool and succeed. The exploit is repeatable and requires no special knowledge beyond knowing the pool uses `SwapAllowlistExtension` and the router is allowlisted.

## Recommendation
The extension must check the actual user identity, not the direct caller. Two approaches:

1. **`extensionData` forwarding (preferred):** The router encodes the original `msg.sender` into `extensionData` before calling `pool.swap()`; the extension decodes and checks it. The pool already forwards `extensionData` unchanged to the extension, so no pool changes are required.

2. **Documentation boundary:** Explicitly document that `SwapAllowlistExtension` cannot be used with `MetricOmmSimpleRouter` and that direct `pool.swap()` calls are required for allowlist enforcement to be meaningful. This is a weaker mitigation as it relies on admin awareness.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists userA (KYC'd):
     extension.setAllowedToSwap(pool, userA, true)
3. Admin allowlists the router (to let userA use the standard periphery):
     extension.setAllowedToSwap(pool, router, true)
4. userB (NOT allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: userB, ...})
5. Router calls pool.swap() — msg.sender at pool = router.
6. Pool calls _beforeSwap(sender=router, ...) → extension.beforeSwap(sender=router, ...).
7. Extension checks: allowedSwapper[pool][router] == true → passes.
8. userB's swap executes successfully, bypassing the allowlist entirely.
```
The state that should have been checked — `allowedSwapper[pool][userB]` — is never consulted. The extension evaluates `allowedSwapper[pool][router]` instead, which is `true` by the admin's legitimate configuration.
