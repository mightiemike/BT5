### Title
`ConsensusTransaction::L1Handler` Structurally Cannot Carry `paid_fee_on_l1`, Causing Hardcoded Fee Bypass in Validator Execution Path - (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field. When a validator converts a received consensus L1Handler transaction into an executable one, `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `Fee(1)` as the paid fee. The blockifier's post-execution fee guard (`if paid_fee == Fee(0) → Err`) is therefore always bypassed on the validator side, regardless of what was actually paid on L1. This is the direct Sequencer analog of the external report: just as `AxelarBridgedGovernor` cannot receive or forward ETH value because the type lacks `receive`/`fallback` and `execute` is not `payable`, `ConsensusTransaction::L1Handler` cannot carry `paid_fee_on_l1` because the inner type lacks the field, and the conversion compensates with a hardcoded sentinel.

---

### Finding Description

`ConsensusTransaction` is defined as:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction),   // ← no paid_fee_on_l1
}
```

`transaction::L1HandlerTransaction` has no `paid_fee_on_l1` field. By contrast, `executable_transaction::L1HandlerTransaction` (used in `InternalConsensusTransaction`) does carry it.

When the proposer broadcasts an L1Handler transaction through consensus, `convert_internal_consensus_tx_to_consensus_tx` strips the executable wrapper:

```rust
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))   // tx.tx drops paid_fee_on_l1
}
```

When a validator receives that consensus message and converts it back to an executable transaction, the missing field is filled with a hardcoded constant:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
```

The blockifier's `execute_raw` for `L1HandlerTransaction` then runs:

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(
        Box::new(TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee }),
    ));
}
```

Because `paid_fee_on_l1` is always `Fee(1)` on the validator side, this guard is permanently bypassed. The validator never sees the actual fee paid on L1.

---

### Impact Explanation

**Divergent execution between proposer and validator.** The proposer uses the real `paid_fee_on_l1` sourced from the L1 event. If that value is `Fee(0)` (e.g., due to an edge case in the L1 core contract, a test/staging environment, or a future contract upgrade), the proposer's blockifier returns a hard `Err(InsufficientFee)` and the transaction fails. The validator, holding `Fee(1)`, passes the guard and commits the transaction successfully. This produces divergent state roots and receipt commitments between proposer and validator, breaking consensus correctness.

Even when `paid_fee_on_l1 > 0`, the validator's executable object carries a semantically wrong value (`1` instead of the real amount), meaning any future logic that reads `paid_fee_on_l1` for accounting, receipts, or L1 message verification will operate on a fabricated value rather than the canonical one.

Matching impact: **High — Transaction conversion binds the wrong executable payload** (`paid_fee_on_l1` field of the executable `L1HandlerTransaction`), and potentially **Critical — Wrong receipt/revert result from blockifier execution logic** when proposer and validator diverge on the fee guard outcome.

---

### Likelihood Explanation

The L1 Starknet core contract currently enforces a minimum non-zero fee, making `paid_fee_on_l1 = 0` unlikely in production. However:
- The structural capability gap is unconditional and permanent until the TODO is resolved.
- Any environment (testnet, devnet, contract upgrade, or future fee-model change) where `paid_fee_on_l1 = 0` is possible immediately triggers the divergence.
- The hardcoded `Fee(1)` also silently corrupts the `paid_fee_on_l1` field in all validator-side receipts and execution infos, regardless of the actual L1 fee.

Likelihood: **Low** for the divergence trigger in production; **Certain** for the structural field corruption on every validator-processed L1Handler transaction.

---

### Recommendation

Add `paid_fee_on_l1` to `ConsensusTransaction::L1Handler` (or to a wrapper type) so the field is transmitted through the consensus wire format and is available to validators without fabrication:

```rust
pub enum ConsensusTransaction {
    RpcTransaction(RpcTransaction),
    L1Handler(transaction::L1HandlerTransaction, Fee),  // carry paid_fee_on_l1
}
```

Update `convert_internal_consensus_tx_to_consensus_tx` to preserve the field, and update `convert_consensus_l1_handler_to_internal_l1_handler` to use the transmitted value instead of the hardcoded `Fee(1)`. Until then, the TODO comment should be escalated to a tracked issue with a clear description of the divergence risk.

### Proof of Concept

**Step 1 – Proposer path (real fee):**
L1 event arrives with `paid_fee_on_l1 = Fee(0)` (edge case / test environment).
Proposer blockifier reaches:
```rust
if paid_fee == Fee(0) { return Err(InsufficientFee { ... }); }
```
→ Transaction execution returns `Err`; proposer marks the transaction as failed.

**Step 2 – Consensus broadcast (fee dropped):**
`convert_internal_consensus_tx_to_consensus_tx`:
```rust
InternalConsensusTransaction::L1Handler(tx) => Ok(ConsensusTransaction::L1Handler(tx.tx))
```
`tx.tx` is `transaction::L1HandlerTransaction`; `paid_fee_on_l1` is silently dropped.

**Step 3 – Validator path (hardcoded fee):**
`convert_consensus_l1_handler_to_internal_l1_handler`:
```rust
Fee(1)   // hardcoded, ignores the actual value
```
Validator blockifier reaches:
```rust
if paid_fee == Fee(0) { ... }   // Fee(1) != Fee(0) → guard passes
```
→ Transaction executes successfully; validator commits state changes.

**Result:** Proposer and validator produce different execution outcomes for the same L1Handler transaction, causing a state-root divergence and breaking consensus.
