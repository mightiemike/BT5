### Title
Bare `approve()` Call in `DirectDepositV1.creditDeposit()` Reverts for Non-Standard ERC20 Tokens (e.g., USDT) - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` using the `IIERC20Base` interface, which declares `approve` as returning `bool`. Tokens like USDT do not return a value from `approve()`, causing the ABI decoder to revert when it expects return data but receives none. This permanently blocks collateral deposits for any spot product whose underlying token is a non-standard ERC20.

---

### Finding Description

In `DirectDepositV1.sol`, the `creditDeposit()` function iterates over all spot product token addresses and calls bare `approve()` on each:

```solidity
token.approve(address(endpoint), balance);
``` [1](#0-0) 

The `IIERC20Base` interface declares `approve` with a `bool` return type:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [2](#0-1) 

When Solidity calls a function declared with a return type, the ABI decoder expects return data. USDT's `approve()` emits no return value. The EVM therefore reverts the call with an ABI decoding error, even though the approval itself would have succeeded on-chain.

The `ERC20Helper` library provides `safeTransfer` and `safeTransferFrom` (which tolerate missing return data via low-level `call`), but **no `safeApprove` equivalent** is defined or used here: [3](#0-2) 

The same bare `approve()` pattern also appears in `ContractOwner.wrapVaultAsset()`:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [4](#0-3) 

---

### Impact Explanation

If USDT (or any non-standard ERC20 that omits a return value from `approve()`) is registered as a spot product token, every call to `creditDeposit()` will revert at the `approve` step. This means:

- All collateral deposits routed through `DirectDepositV1` for that product are permanently blocked.
- Funds already held in the `DirectDepositV1` contract for that subaccount cannot be credited to the protocol.
- The `wrapVaultAsset()` path is similarly broken for vault assets whose underlying token is non-standard.

The corrupted state is: the subaccount's on-chain balance in the `DirectDepositV1` contract is non-zero but can never be forwarded to `Endpoint.depositCollateralWithReferral()`, effectively locking user funds.

---

### Likelihood Explanation

USDT is one of the highest-volume stablecoins and a natural candidate for a spot collateral product. Any operator listing USDT (or a USDT-like token) as a spot product triggers this bug for every user who sends funds to their `DirectDepositV1` address. The entry path requires no special privileges — `ContractOwner.creditDepositV1()` is a public function callable by anyone:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    ...
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [5](#0-4) 

---

### Recommendation

Replace the bare `token.approve(...)` call in `DirectDepositV1.creditDeposit()` with a safe approve pattern that uses a low-level `call` and tolerates a missing return value, analogous to the existing `safeTransfer` in `ERC20Helper`:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Apply the same fix to the two `assetToken.approve(...)` calls in `ContractOwner.wrapVaultAsset()`. [6](#0-5) 

---

### Proof of Concept

1. A spot product is listed with USDT as its token address.
2. A user sends USDT to their `DirectDepositV1` address (or it arrives via any transfer).
3. Anyone calls `ContractOwner.creditDepositV1(subaccount)`.
4. This calls `DirectDepositV1.creditDeposit()`, which reaches `token.approve(address(endpoint), balance)` for the USDT product.
5. USDT's `approve()` executes successfully on-chain but returns no data.
6. Solidity's ABI decoder, expecting a `bool`, reverts the entire transaction.
7. The USDT balance remains stuck in the `DirectDepositV1` contract; the subaccount never receives credit; the deposit is permanently blocked. [7](#0-6)
