### Title
Immutable `network` Parameter Set at Initialization Cannot Be Corrected, Enabling Permanently Incorrect PoW Validation — (File: `contract/src/lib.rs`)

---

### Summary

The `network: Network` field (Mainnet/Testnet) is written once during `init` and drives every consensus parameter used in PoW verification. No admin endpoint exists to update it. If the wrong variant is supplied at initialization, all subsequent block submissions are validated against incorrect parameters, and the mistake cannot be corrected without a full contract redeployment.

---

### Finding Description

`BtcLightClient` stores `network: Network` in its on-chain state, assigned directly from `args.network` in `init`: [1](#0-0) 

Every chain-specific module reads this field through `get_config()` to derive all consensus parameters used in PoW validation:

- **Bitcoin / Litecoin** (`bitcoin.rs`, `litecoin.rs`): `get_config()` calls `get_bitcoin_config(self.network)` / `get_litecoin_config(self.network)`, returning `pow_allow_min_difficulty_blocks`, `proof_of_work_limit_bits`, `pow_limit`, `pow_target_timespan`, `difficulty_adjustment_interval`, and `pow_target_spacing`. [2](#0-1) 

- **Zcash** (`zcash.rs`): `get_config()` calls `get_zcash_config(self.network)`, additionally returning `pow_allow_min_difficulty_blocks_after_height` (mainnet: `None`; testnet: `Some(299187)`), `proof_of_work_limit_bits` (mainnet: `0x1f07ffff`; testnet: `0x2007ffff`), and `pow_limit` (testnet is ~1024× weaker). [3](#0-2) 

These parameters are consumed directly inside `check_pow()`, which is called for every block submitted via `submit_blocks`: [4](#0-3) 

Searching the entire contract, there is no admin function to update `self.network` after initialization. The `migrate()` function carries the field over from old state unchanged, providing no correction path: [5](#0-4) 

The `InitArgs` struct exposes `network` as a plain caller-supplied field with no cross-validation against the actual submitted block headers: [6](#0-5) 

---

### Impact Explanation

**Zcash build (most severe):** If `Network::Testnet` is supplied for a mainnet deployment:

1. `proof_of_work_limit_bits` becomes `0x2007ffff` instead of `0x1f07ffff` — a ~1024× weaker PoW ceiling.
2. `pow_limit` becomes `0x07ff_ffff…` instead of `0x0007_ffff…` — 1024× weaker, so the clamp in `zcash_calculate_next_work_required` never triggers at mainnet difficulty levels. [7](#0-6) 

3. `pow_allow_min_difficulty_blocks_after_height` becomes `Some(299187)` instead of `None`. The special testnet rule in `zcash_get_next_work_required` then fires for any block whose timestamp is more than 6 × 75 = 450 seconds ahead of the previous block, returning `proof_of_work_limit_bits` as the required difficulty: [8](#0-7) 

An adversarial relayer can craft Zcash block headers with a timestamp 451 seconds ahead of the previous block and submit them with only `0x2007ffff`-level PoW — trivially achievable — and they will be accepted as valid mainchain blocks. The corrupted chain state then causes `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to return `true` for transactions in fraudulent blocks, breaking the SPV guarantee for every consuming contract.

**Bitcoin / Litecoin build:** If `Network::Testnet` is supplied for a mainnet deployment, `pow_allow_min_difficulty_blocks` becomes `true`. The `get_next_work_required` logic then allows any block whose timestamp is more than `2 × pow_target_spacing` seconds ahead of the previous block to be accepted at minimum difficulty: [9](#0-8) 

---

### Likelihood Explanation

Medium. The deployer must supply the wrong `Network` variant at `init` time. This is a realistic operational mistake given that the contract supports four chains and two network types, and the `network` field is a plain enum with no validation against the actual block data. The absence of any correction mechanism means a single init mistake permanently compromises the contract for its entire lifetime.

---

### Recommendation

Add a DAO-restricted admin endpoint to update `self.network`, analogous to the fix recommended in the reference report for `tokenPrecisionMultipliers`. For example:

```rust
#[near]
impl BtcLightClient {
    /// Corrects the network type if an incorrect value was supplied at init.
    /// Restricted to DAO role only.
    pub fn set_network(&mut self, network: Network) {
        // require DAO role
        self.network = network;
    }
}
```

Alternatively, document that the only recovery path is a contract redeployment via the existing `Upgradable` mechanism, and add an on-chain assertion during `init_genesis` that cross-checks the supplied `network` against a known property of the genesis block (e.g., the `proof_of_work_limit_bits` field of the genesis header's `bits`).

---

### Proof of Concept

1. Deploy the Zcash build of the contract.
2. Call `init` with `network: Network::Testnet` (mistake) and a valid Zcash mainnet genesis block.
3. Confirm `get_config()` returns `proof_of_work_limit_bits = 0x2007ffff` and `pow_allow_min_difficulty_blocks_after_height = Some(299187)`.
4. Once the tracked chain height exceeds 299187, craft a block header with `time = prev_block.time + 451` and any hash satisfying `0x2007ffff`-level PoW (trivially achievable on commodity hardware in seconds).
5. Call `submit_blocks` with this header — it is accepted and promoted to the mainchain tip.
6. Call `verify_transaction_inclusion_v2` with a fabricated Merkle proof for a non-existent transaction in this fraudulent block — it returns `true`.
7. Attempt to call any admin function to correct `self.network` — no such function exists; the contract must be fully redeployed.
