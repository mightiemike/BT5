Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Recipient `owner` Instead of Token-Providing `sender`, Allowing Allowlist Bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` who provides tokens via callback) and gates access solely on `owner` (the LP position recipient). Any address not on the allowlist can bypass the deposit restriction by calling `addLiquidity` with an allowlisted address as `owner`, injecting unauthorized liquidity into a permissioned pool.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the LP recipient to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both values to the extension: [2](#0-1) 

However, `DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as anonymous, discarding it entirely, and checks only `owner`: [3](#0-2) 

**Exploit path:**
1. Pool admin deploys a permissioned pool with `DepositAllowlistExtension`, allowlisting only address `A`.
2. Unprivileged attacker (address `B`, not allowlisted) calls `pool.addLiquidity(owner = A, ...)`.
3. The hook fires: `sender = B` (discarded), `owner = A` (checked) → `allowedDepositor[pool][A]` is `true` → check passes.
4. `B` satisfies the token callback, injecting liquidity into the pool; the LP position is minted to `A`.
5. `B` has successfully added liquidity to a pool it is not authorized to touch.

The NatDoc on the contract itself states it "Gates `addLiquidity` by depositor address" — the implementation contradicts this intent. [4](#0-3) 

No other guard in the core call path (`nonReentrant`, `LiquidityLib.addLiquidity`) checks the allowlist; the extension hook is the sole enforcement point.

## Impact Explanation
The deposit allowlist is the primary access-control mechanism for permissioned pools. Bypassing it allows any unprivileged address to inject liquidity at arbitrary bins, corrupting bin totals, bin share accounting, and pool-level token balances in ways the pool admin did not authorize. This breaks a core pool invariant (controlled liquidity composition) and constitutes broken core pool functionality. Severity: **Medium** — no direct theft of existing LP assets, but unauthorized state mutation of a permissioned pool with potential downstream impact on pricing, oracle feeds, and LP share dilution.

## Likelihood Explanation
Trivially reachable by any external caller with no special privileges. The attacker only needs to know one allowlisted address (publicly readable from `allowedDepositor` mapping) and supply tokens. Fully repeatable on every permissioned pool using this extension.

## Recommendation
Replace the anonymous first parameter with the named `sender` and gate on it instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// 1. Deploy pool with DepositAllowlistExtension; allowlist only `alice`.
// 2. As `bob` (not allowlisted):
pool.addLiquidity(
    alice,   // owner — passes the allowlist check
    salt,
    deltas,
    callbackData,
    extensionData
);
// Bob's callback provides tokens; alice receives LP shares.
// Assert: call succeeds despite bob not being allowlisted.
// Assert: pool bin totals updated with bob's injected liquidity.
```
