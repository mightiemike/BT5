### Title
Accrued Bonus Rewards Permanently Lost When Bonus Pool Is Insufficient During Redelegation Slash — (`File: x/tieredrewards/keeper/slash.go`)

---

### Summary

In `x/tieredrewards`, when a redelegation slash fires (`BeforeRedelegationSlashed` → `slashRedelegationPosition`), the code calls `processEventsAndClaimBonus` to settle bonus before the slash. Inside `processEventsAndClaimBonus`, the position's `LastBonusAccrual` checkpoint is advanced **before** the pool-balance check. If the pool is insufficient, `ErrInsufficientBonusPool` is returned, but `slashRedelegationPosition` deliberately swallows that error to avoid a chain halt. The modified position — with its checkpoint already advanced but no bonus paid — is then persisted. The user permanently loses all bonus that accrued up to the slash block; there is no retry path because the checkpoint has moved forward.

---

### Finding Description

**Step 1 — Checkpoint advanced before pool check.**

In `processEventsAndClaimBonus` (`claim_rewards.go`):

```
line 215: applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // checkpoint advanced
line 217: pos.UpdateLastKnownBonded(bonded)                        // bonded state updated
...
line 230: if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
line 231:     return nil, err                                       // ErrInsufficientBonusPool
line 232: }
```

The in-memory `pos` already carries the new `LastBonusAccrual = blockTime` when the error is returned. [1](#0-0) 

**Step 2 — Error swallowed in the slash hook.**

In `slashRedelegationPosition` (`slash.go`):

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error is swallowed — execution continues with modified pos
    } else {
        return err
    }
}
``` [2](#0-1) 

**Step 3 — Modified position (advanced checkpoint, no payment) is persisted.**

Execution falls through to `setPositionWithState`, which writes the position with the advanced `LastBonusAccrual` to the store: [3](#0-2) 

The bonus that accrued from the old `LastBonusAccrual` to `blockTime` is computed, not paid, and the window is closed forever.

---

### Impact Explanation

The exact corrupted value is `pos.LastBonusAccrual`. It is advanced to `blockTime` without the corresponding `SendCoinsFromModuleToAccount` transfer executing. The bonus amount — `shares × tokensPerShare × tier.BonusApy × durationSeconds / SecondsPerYear` for the elapsed period — is permanently forfeited by the user. No retry is possible: on the next `ClaimTierRewards` call, `processEventsAndClaimBonus` starts from the already-advanced checkpoint and computes zero for the lost window. [4](#0-3) 

This is a direct, permanent loss of earned bonus rewards for any delegator who holds a redelegating tier position when the pool is empty.

---

### Likelihood Explanation

The bonus pool (`RewardsPoolName`) is funded by external bank sends; it is not automatically replenished and drains continuously via both user bonus claims and the `BeginBlocker` base-rewards top-up. [5](#0-4) 

A pool-empty window is a realistic operational scenario (governance delay in replenishment, or the pool being intentionally drained by the BeginBlocker). Validator slashes (double-sign, downtime) are normal protocol events that any validator can trigger. The combination — pool empty at the moment a redelegation slash fires — is a realistic, non-contrived condition requiring no privileged access.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, so the checkpoint is only advanced when the bonus is actually paid. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** persist the position with the advanced checkpoint — restore the original checkpoint values before calling `setPositionWithState`, so the user can claim the accrued bonus once the pool is replenished. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

1. User calls `MsgLockTier` → position created, `LastBonusAccrual = T0`.
2. User calls `MsgTierRedelegate` → redelegation entry created, `RedelegationMappings[unbondingId] = positionId`.
3. Bonus pool is empty (or drained to zero by `BeginBlocker`).
4. Validator is slashed (double-sign evidence submitted) → SDK calls `BeforeRedelegationSlashed(ctx, unbondingId, sharesToUnbond)`.
5. `slashRedelegationPosition` is entered; `processEventsAndClaimBonus` is called on the position.
6. Inside `processEventsAndClaimBonus`: `applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = blockTime` (line 215); `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` (line 230–231).
7. Back in `slashRedelegationPosition`: error is swallowed (lines 56–63); `setPositionWithState` persists `pos` with `LastBonusAccrual = blockTime`.
8. User later calls `MsgClaimTierRewards`: `processEventsAndClaimBonus` starts from `LastBonusAccrual = blockTime`, computes zero bonus for the already-elapsed window. The bonus accrued from `T0` to `blockTime` is permanently lost.

func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
