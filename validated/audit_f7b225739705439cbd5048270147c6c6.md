Audit Report

## Title
`SwapAllowlistExtension.beforeSwap()` checks router address instead of actual swapper, enabling allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` parameter, which is `msg.sender` of the pool's `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` becomes the router's address rather than the actual user. A pool admin who allowlists the router to enable router-based swaps inadvertently opens the pool to every user, fully bypassing the intended access control.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap()` passes its own `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap()` forwards that value unchanged to every configured extension: [2](#0-1) 

**Step 2 — Router calls `pool.swap()` as `msg.sender`.**

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly. At that call site, `msg.sender` of the pool is the **router contract**, not the end user. The actual user (`msg.sender` of the router call) is stored only in transient callback context for payment purposes and is never forwarded to the pool or its extensions: [3](#0-2) 

**Step 3 — Extension checks the router address, not the actual user.**

`SwapAllowlistExtension.beforeSwap()` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when the swap came through the router: [4](#0-3) 

**Step 4 — Contrast with `DepositAllowlistExtension`.**

`DepositAllowlistExtension.beforeAddLiquidity()` correctly checks `owner` (the LP position owner — the actual economic actor), not `sender` (the caller). The protocol already understands the distinction between caller and economic actor for deposits but does not apply the same logic to swaps: [5](#0-4) 

**Step 5 — The impossible configuration trap.**

A pool admin who wants to restrict swaps to specific users (e.g., KYC-only) while also allowing those users to use the router faces an impossible choice:
- **Do not allowlist the router** → allowlisted users cannot use the router at all (their swaps are blocked because the router address is not in the allowlist).
- **Allowlist the router** → every user who calls through the router bypasses the allowlist, because the extension sees `allowedSwapper[pool][router] == true` regardless of who the actual user is.

## Impact Explanation

Any pool configured with `SwapAllowlistExtension` that allowlists the router (a reasonable admin action to support router-based swaps for its allowlisted users) becomes accessible to **all** users. Non-allowlisted users can bypass the access control by routing through `MetricOmmSimpleRouter`. In a curated pool (e.g., institutional-only, KYC-gated, or counterparty-restricted), this allows unauthorized traders to extract value from LPs who deposited under the assumption that only vetted counterparties could trade against them. This is a direct LP-fund-impacting consequence of the misconfigured guard, meeting the "broken core pool functionality causing loss of funds" impact criterion.

## Likelihood Explanation

Medium. The bypass requires the pool admin to allowlist the router address. This is a natural and expected configuration step for any admin who wants their allowlisted users to be able to use the standard periphery router. The admin has no indication from the contract or documentation that doing so opens the pool to everyone. The misconfiguration is silent — no revert, no event, no warning.

## Recommendation

The `SwapAllowlistExtension` must gate the actual economic actor, not the intermediary. Two viable approaches:

1. **Extension-data forwarding**: Require the router to encode the actual user's address in `extensionData` and have the extension decode and check it. The pool admin would configure the extension to trust the router's forwarded identity.
2. **Documentation + invariant enforcement**: Clearly document that the allowlist gates the direct pool caller, not the end user, and that the router is incompatible with allowlisted pools. Add a check in the extension that reverts if `msg.sender` (the pool) has the router registered as an allowed swapper while `allowAllSwappers` is false, to prevent the misconfiguration silently.

## Proof of Concept

```
1. Deploy MetricOmmPool with SwapAllowlistExtension configured.
2. Pool admin calls setAllowedToSwap(pool, allowedUser, true).
3. Pool admin calls setAllowedToSwap(pool, address(router), true)
   — intending to let allowedUser use the router.
4. Non-allowlisted attacker calls:
     router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
5. Router calls pool.swap(recipient, ...) — msg.sender = router.
6. Pool calls _beforeSwap(router, ...).
7. Extension evaluates: allowedSwapper[pool][router] == true → no revert.
8. Attacker's swap executes in the restricted pool.
```

The attacker never appears in the allowlist; the check passes solely because the router is allowlisted. The actual user identity (`msg.sender` of the router call) is never consulted by the extension. [6](#0-5) [7](#0-6)

        priceLimitX64,
        "",
