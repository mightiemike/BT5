Audit Report

## Title
`SwapAllowlistExtension` checks router address as `sender` instead of originating user, enabling full allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `sender`, which is the `msg.sender` of the pool's `swap` call. When `MetricOmmSimpleRouter` is used, `sender` is the router address, not the originating user. If the pool admin allowlists the router (the natural step to enable router-mediated swaps for allowlisted users), every non-allowlisted address can bypass the guard by routing through the router.

## Finding Description
**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

`MetricOmmPool.swap` calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
  msg.sender,   // router address when called via MetricOmmSimpleRouter
  recipient,
  ...
);
``` [1](#0-0) 

**Step 2 — Router calls `pool.swap` directly; `msg.sender` inside the pool is the router.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` with no forwarding of the originating user:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [2](#0-1) 

The same pattern applies to `exactInput` (L103–112), `exactOutputSingle` (L136–137), and `exactOutput` (L165–181). The router stores `msg.sender` only in transient callback context for payment purposes — it is never encoded into `extensionData` passed to the pool. [3](#0-2) 

**Step 3 — The allowlist check is keyed on `sender`, which is the router.**

```solidity
function beforeSwap(address sender, ...)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  ...
}
``` [4](#0-3) 

`msg.sender` here is the pool (correct). `sender` is the router (wrong). The effective check becomes `allowedSwapper[pool][router]`. Once the router is allowlisted, the check passes for every caller regardless of identity.

The router does not inject the originating user into `extensionData`; `params.extensionData` is passed through verbatim from the caller, so the extension has no on-chain path to recover the true initiator.

## Impact Explanation
Once the pool admin allowlists the router, any non-allowlisted address can call `MetricOmmSimpleRouter.exactInputSingle` (or any other router entry point) targeting the curated pool. The `beforeSwap` hook sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and permits the swap. The curation mechanism is fully and silently bypassed. For pools designed to restrict swaps to trusted market makers or KYC-verified counterparties, this exposes LP capital to adversarial or unintended swap flow — a direct loss-of-funds risk for LPs whose positions are priced assuming a controlled counterparty set.

## Likelihood Explanation
The admin must explicitly call `setAllowedToSwap(pool, router, true)`, but this is the natural and expected operational step when deploying a curated pool intended to be used with the protocol's own periphery router. No special privilege or unusual setup is required beyond that single admin call. Any unprivileged address can then exploit the bypass immediately and repeatedly.

## Recommendation
The extension must check the economically responsible actor — the address that initiated the trade — not the intermediate caller. Two viable fixes:

1. **Pass the originating user through `extensionData`**: The router encodes `msg.sender` into `extensionData` before calling `pool.swap`; the extension decodes and checks it. This requires a coordinated change to the router and extension, and the extension must reject calls where `extensionData` is absent or malformed.

2. **Document and enforce incompatibility**: Clearly state in `SwapAllowlistExtension` NatSpec that it is incompatible with `MetricOmmSimpleRouter`. Enforce this at the factory level by rejecting pool configurations that combine the swap allowlist extension with the router as an allowed caller.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension.
2. Admin: setAllowedToSwap(pool, user1, true)       // allowlist user1
3. Admin: setAllowedToSwap(pool, router, true)       // allowlist router so user1 can use it
4. user2 (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
5. Router calls pool.swap(...); pool calls _beforeSwap(router, ...)
6. Extension checks allowedSwapper[pool][router] == true → passes
7. user2 swaps successfully — allowlist fully bypassed.
```

The root cause is confirmed at:
- `MetricOmmPool.sol` L231: `msg.sender` (router) is passed as `sender` to `_beforeSwap`
- `SwapAllowlistExtension.sol` L37: the check `allowedSwapper[msg.sender][sender]` resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user2]` [5](#0-4) [6](#0-5)
