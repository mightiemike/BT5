### Title
Existing Linked Signer Can Overwrite Itself via Fast-Mode `LinkSigner`, Permanently Hijacking Subaccount Signing Authority — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The slow-mode `LinkSigner` path enforces that only the subaccount owner (the EOA whose address is embedded in the subaccount identifier) can set the linked signer. The fast-mode `LinkSigner` path passes `allowLinkedSigner = true` to `validateSignedTx`, meaning the **existing linked signer** is accepted as a valid signer for a transaction that changes the linked signer. A compromised linked signer can therefore replace itself with an attacker-controlled address, permanently hijacking the subaccount's signing authority and enabling full collateral theft.

---

### Finding Description

Two separate code paths handle `LinkSigner` transactions in `EndpointTx.sol`.

**Slow-mode path** (`processSlowModeTransactionImpl`, lines 232–239):

```solidity
validateSender(txn.sender, sender);
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
```

`validateSender` enforces:

```solidity
require(
    address(uint160(bytes20(txSender))) == sender || sender == address(this),
    ERR_SLOW_MODE_WRONG_SENDER
);
```

Here `sender` is `msg.sender` — the actual on-chain caller. Because a subaccount is `bytes32(abi.encodePacked(ownerAddress, subaccountName))`, `address(uint160(bytes20(txSender)))` is the owner's address. Only the owner's EOA can submit this slow-mode transaction.

**Fast-mode path** (`processTransactionImpl`, lines 576–590):

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // <-- allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
```

`validateSignedTx` with `allowLinkedSigner = true` resolves to:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
```

`Verifier.validateSignature` accepts a signature from either `address(uint160(bytes20(sender)))` **or** `linkedSigner`. Therefore the existing linked signer is a valid signer for a `LinkSigner` transaction — the transaction that changes the linked signer. The linked signer can sign a `LinkSigner` payload pointing `signedTx.tx.signer` to any arbitrary address, and the contract will write it unconditionally.

This is structurally identical to the `FanToArtistStaking` bug: a restriction placed on one entry point (`delegate()` / slow-mode `LinkSigner`) is silently absent from a secondary entry point (`delegateBySig()` / fast-mode `LinkSigner`) because the secondary path calls a lower-level primitive that does not re-apply the restriction.

---

### Impact Explanation

Once a linked signer key is compromised (e.g., leaked hot-wallet key), the attacker can:

1. Sign a fast-mode `LinkSigner` transaction pointing `signer` to an attacker-controlled address and submit it to the sequencer.
2. The sequencer includes it; `linkedSigners[subaccount]` is now the attacker's address.
3. The attacker immediately signs `WithdrawCollateral` (fast-mode, `allowLinkedSigner = true`, lines 418–424) to drain all collateral from the subaccount.

The owner cannot race to revoke: the slow-mode path has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). By the time the owner's revocation executes, the attacker has already changed the linked signer to a fresh address and drained funds. The attacker can repeat the linked-signer rotation indefinitely to stay ahead of any revocation attempt.

**Corrupted state**: `linkedSigners[subaccount]` is set to an attacker-controlled address. **Asset delta**: full collateral balance of the victim subaccount is withdrawable by the attacker.

---

### Likelihood Explanation

Linked signers are explicitly designed as hot-wallet keys used in automated trading systems. Hot-wallet key compromise is a realistic, well-documented threat. Any user who has ever set a linked signer is exposed. No privileged access, governance capture, or social engineering is required — only possession of the linked signer's private key, which is the exact threat model linked signers are supposed to be resilient against (i.e., a compromised linked signer should not be able to do more than the owner intended).

---

### Recommendation

Pass `false` for `allowLinkedSigner` when processing fast-mode `LinkSigner` transactions, so that only the subaccount owner's key is accepted as a valid signer for linked-signer mutations:

```solidity
// In processTransactionImpl, LinkSigner branch:
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This mirrors the invariant already enforced by the slow-mode path and ensures that the linked signer cannot escalate its own authority.

### Proof of Concept

**Setup**: Victim subaccount `V` has owner `owner` (address `0xALICE`) and linked signer `hotWallet` (address `0xHOT`). Attacker has compromised `hotWallet`.

**Step 1 — Attacker constructs a `SignedLinkSigner` payload**:
```
tx.sender = V                    // victim subaccount
tx.signer = bytes32(attacker)    // attacker-controlled address
tx.nonce  = currentNonce(V)
signature = sign(hotWallet, EIP712_digest(LinkSigner{V, attacker, nonce}))
```

**Step 2 — Sequencer includes the transaction**. `processTransactionImpl` reaches the `LinkSigner` branch:
- `validateSignedTx(V, nonce, txn, sig, true)` → `validateSignature` accepts `hotWallet` as signer because `getLinkedSigner(V) == hotWallet`. Passes.
- `linkedSigners[V] = attacker`. [1](#0-0) 

**Step 3 — Attacker drains collateral**. Attacker signs a `WithdrawCollateral` transaction for subaccount `V`:
- `validateSignedTx(V, nonce+1, txn, sig, true)` → `getLinkedSigner(V) == attacker`. Passes.
- `clearinghouse.withdrawCollateral(V, productId, fullBalance, address(0), ...)` executes. [2](#0-1) 

**Step 4 — Owner's revocation is too late**. Owner submits slow-mode `LinkSigner` to set signer to `address(0)`. It enters the queue with a 3-day delay. Before it executes, attacker rotates the linked signer again using the same technique. Funds are already gone. [3](#0-2) 

The root cause is the asymmetry between the two `LinkSigner` paths: the slow-mode path enforces owner-only via `validateSender` [4](#0-3) , while the fast-mode path passes `allowLinkedSigner = true`, silently granting the linked signer the power to mutate its own entry. [5](#0-4)
