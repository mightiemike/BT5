### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Free USDC-E Drain — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` uses a raw, unchecked `IERC20Base.transferFrom` to pull USDC from the caller before withdrawing USDC-E from a `DirectDepositV1` contract and forwarding it to the caller. If the `transferFrom` silently returns `false`, the function continues execution, transferring USDC-E to the caller without receiving any USDC in return.

---

### Finding Description

`replaceUsdcEWithUsdc` is a public, permissionless function (restricted only by `block.chainid == 57073`) that is intended to perform a 1:1 swap: the caller provides USDC, and receives an equivalent amount of USDC-E held in a `DirectDepositV1` (DDA) contract.

The critical sequence is:

```solidity
// Line 616 — raw transferFrom, return value silently discarded
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// Line 617 — withdraws USDC-E from DDA to ContractOwner
DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));

// Line 618 — safeTransfer sends USDC-E to caller
IERC20Base(usdcE).safeTransfer(msg.sender, balance);
``` [1](#0-0) 

The `transferFrom` at line 616 returns a `bool` that is never inspected. The protocol's own `ERC20Helper.safeTransferFrom` wrapper — which enforces the return value — is available and used throughout the codebase (e.g., `EndpointStorage.chargeSlowModeFee`, `EndpointStorage.safeTransferFrom`), but was not applied here. [2](#0-1) [3](#0-2) 

The asymmetry is explicit: `safeTransfer` is correctly used on line 618 for the USDC-E leg, but the USDC leg on line 616 is unguarded. [4](#0-3) 

`DirectDepositV1.withdraw` internally uses its own `safeTransfer` wrapper and will always succeed or revert — it does not depend on the USDC leg having succeeded. [5](#0-4) 

---

### Impact Explanation

If `transferFrom` returns `false` (caller has zero allowance, zero balance, or the token implementation returns false on failure rather than reverting):

1. `directDepositV1` receives **no USDC**.
2. `DirectDepositV1.withdraw` still transfers all USDC-E held in the DDA to `ContractOwner`.
3. `safeTransfer` still forwards that USDC-E to the attacker.

The attacker drains the full USDC-E balance of any targeted DDA contract at zero cost. The corrupted asset delta is: USDC-E balance of `directDepositV1[subaccount]` → attacker, with no USDC deposited in exchange.

---

### Likelihood Explanation

The function is callable by any address on chain 57073 (Ink mainnet) with no access control. The attacker only needs to call `replaceUsdcEWithUsdc` for any subaccount whose DDA holds a non-zero USDC-E balance, without having approved or holding any USDC. Whether the specific USDC deployment at `0x2D270e6886d130D724215A266106e6832161EAEd` returns `false` or reverts on failure determines exploitability; if it reverts, the bug is latent but still violates the safe-transfer invariant the rest of the codebase enforces. [6](#0-5) 

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ERC20Helper` is already imported and `using ERC20Helper for IERC20Base` is already declared in `ContractOwner`, so no additional imports are needed. [7](#0-6) 

---

### Proof of Concept

1. Identify a subaccount whose `directDepositV1Address[subaccount]` DDA holds a non-zero USDC-E balance.
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with **zero USDC balance and zero USDC allowance**.
3. `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` returns `false` (or is a no-op); execution continues.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers the full USDC-E balance to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(msg.sender, balance)` sends the full USDC-E amount to the attacker.
6. Attacker receives `balance` USDC-E; DDA receives 0 USDC. Net: free drain of the DDA's USDC-E holdings.

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
