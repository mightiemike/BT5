### Title
`verify_transaction_inclusion` Does Not Check Whether the Light Client Chain Tip Is Stale — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` measure confirmation depth relative to the stored `mainchain_tip_blockhash` but never verify that the tip's own timestamp is recent. If the relayer stops submitting headers, the contract silently continues to confirm transactions against an arbitrarily old chain snapshot, producing false-positive inclusion proofs for transactions that may have been reorganized or double-spent on the real Bitcoin network.

---

### Finding Description

Both verification entry points resolve the chain tip the same way:

```rust
// contract/src/lib.rs:294-297
let heaviest_block_header = self
    .headers_pool
    .get(&self.mainchain_tip_blockhash)
    .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
```

They then count confirmations relative to that tip:

```rust
// contract/src/lib.rs:304-308
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
```

Neither function checks whether `heaviest_block_header.block_header.time` is within an acceptable window of the current NEAR block time. The `time` field is present in every stored `ExtendedHeader` (via `LightHeader`), and `env::block_timestamp_ms()` is already used in `check_pow` for the future-time guard:

```rust
// contract/src/bitcoin.rs:35-39
let current_timestamp = u32::try_from(env::block_timestamp_ms() / 1000).unwrap();
require!(
    block_header.time <= current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL,
    "time-too-new: block timestamp too far in the future"
);
```

The same clock is available during verification but is never consulted. `verify_transaction_inclusion_v2` delegates entirely to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same gap:

```rust
// contract/src/lib.rs:367-368
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
```

---

### Impact Explanation

A consumer contract (bridge, exchange, escrow) calls `verify_transaction_inclusion_v2` and receives `true`. It then releases funds or settles a trade. If the light client's chain tip is hours or days old, the "confirmed" transaction may have been reorganized out of the real Bitcoin chain, or a double-spend may have succeeded on-chain after the relayer went silent. The consumer has no on-chain signal that the proof is based on stale state.

---

### Likelihood Explanation

The relayer is a single off-chain process. It can stop due to infrastructure failure, operator error, or deliberate griefing. There is no on-chain liveness enforcement: `submit_blocks` is permissioned to authorized relayers, but nothing forces them to submit within any time bound. A gap of even a few hours on a chain like Bitcoin (where 6-block reorgs are rare but possible) is enough to invalidate a "confirmed" transaction. The verification functions are public view calls, so any NEAR account can invoke them at any time without restriction.

---

### Recommendation

Add a staleness guard at the top of `verify_transaction_inclusion` using the already-available `block_header.time` and `env::block_timestamp_ms()`. Define a protocol-level constant (e.g., `MAX_TIP_AGE_SECONDS`) and reject proofs when the tip is older than that threshold:

```rust
let tip_time = u64::from(heaviest_block_header.block_header.time);
let now_secs = env::block_timestamp_ms() / 1000;
require!(
    now_secs.saturating_sub(tip_time) <= MAX_TIP_AGE_SECONDS,
    "light client tip is stale; relayer must submit recent headers before verification"
);
```

A reasonable default for Bitcoin is 7200 seconds (two hours, matching the existing `MAX_FUTURE_BLOCK_TIME_LOCAL` window). Chain-specific values should be chosen to match each network's expected block interval and acceptable lag.

---

### Proof of Concept

1. Deploy the contract and initialize it with a genesis block at height N.
2. Submit enough headers to satisfy a `confirmations = 6` check for some transaction T in block B.
3. Stop the relayer. The contract's `mainchain_tip_blockhash` is now frozen.
4. Wait 24 hours. On the real Bitcoin network, a reorg or double-spend of T occurs.
5. Call `verify_transaction_inclusion_v2` with T's proof and `confirmations = 6`.
6. The contract returns `true` — the confirmation count relative to the frozen tip still satisfies the requirement — even though T no longer exists in the real Bitcoin canonical chain.
7. A consumer contract acting on this result releases funds based on a transaction that no longer exists on Bitcoin.

The root cause is exclusively in `verify_transaction_inclusion` at [1](#0-0)  — the function reads `mainchain_tip_blockhash` at [2](#0-1)  and checks confirmation depth at [3](#0-2)  with no timestamp freshness check, while `verify_transaction_inclusion_v2` at [4](#0-3)  inherits the same gap by delegating to it. The `block_header.time` field is available in every stored `ExtendedHeader` [5](#0-4)  and `env::block_timestamp_ms()` is already used for the analogous future-time guard in `check_pow` [6](#0-5)  but is absent from the verification path.

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```
