### Title
`is_canonical_serialization` Accepts Semantically Invalid Back-References That `serialized_length_from_bytes` Rejects — (`src/serde/tools.rs`)

---

### Summary

`is_canonical_serialization` deliberately skips back-reference resolution, checking only that the path is a syntactically valid canonical atom. `serialized_length_from_bytes` fully resolves every back-reference via `traverse_path`, which returns `Err(PathIntoAtom)` when a path traverses into an atom. A crafted blob can pass `is_canonical_serialization` while causing `serialized_length_from_bytes` to return `Err`, breaking the implied API invariant that acceptance by the former implies acceptance by the latter.

---

### Finding Description

The divergence is structural and explicit. In `is_canonical_serialization`, the `BACK_REFERENCE` branch reads the path byte and calls `is_canonical_atom` to verify the path is a well-formed atom, then moves on: [1](#0-0) 

The comment at line 220 states the design intent: *"We don't actually need to resolve it, just parse the path and move on."* No semantic validation of the path is performed.

In contrast, `serialized_length_from_bytes` calls `traverse_path` on every back-reference: [2](#0-1) 

`traverse_path` returns `Err(EvalErr::PathIntoAtom)` whenever a path bit requires descending into an atom node: [3](#0-2) 

Both functions are exported as public API: [4](#0-3) 

---

### Proof of Concept

The question's stated blob `[0xff, 0x01, 0xfe, 0x02, 0x80]` does **not** reproduce the bug — `is_canonical_serialization` returns `false` because the trailing `0x80` byte causes the final position check (`f.get_ref().len() as u64 == f.position()`) to fail at line 235. Path `0x02` also does not cause `PathIntoAtom` because it resolves to the left child of the parse stack pair, which is a valid atom node.

The correct minimal PoC is **`[0xff, 0x01, 0xfe, 0x04]`**:

| Byte | Meaning |
|------|---------|
| `0xff` | CONS_BOX_MARKER |
| `0x01` | atom (value 1) — left child |
| `0xfe` | BACK_REFERENCE |
| `0x04` | path `0b100` — go left twice |

**`is_canonical_serialization([0xff, 0x01, 0xfe, 0x04])`:**
- Reads `0xff` → counter=2
- Reads `0x01` → `is_canonical_atom` → `true`, counter=1
- Reads `0xfe` → reads `0x04`, `is_canonical_atom(f, 0x04)` → `true` (≤ `MAX_SINGLE_BYTE`), counter=0
- Final check: position=4, len=4 → **returns `true`**

**`serialized_length_from_bytes([0xff, 0x01, 0xfe, 0x04])`:**
- After parsing `0x01`, parse stack `values = pair(nil, nil_orig)`
- Reads `0xfe`, parses path `[0x04]` = `0b00000100`
- `traverse_path` with path `0x04` on `pair(nil, nil_orig)`:
  - bit 0 (value 0) → go left → `arg_list = nil` (atom)
  - bit 1 (value 0) → go left from atom → **`Err(PathIntoAtom)`**
- **returns `Err(PathIntoAtom)`**

The two public API functions disagree on the same input.

---

### Impact Explanation

Any caller that uses `is_canonical_serialization` as a pre-filter before calling `serialized_length_from_bytes` or the deserializer (`node_from_bytes_backrefs`) will accept a blob that the downstream function rejects. This produces an inconsistent accept/reject decision. In a mempool or consensus context where one code path gates on `is_canonical_serialization` and another validates with `serialized_length_from_bytes`, nodes can diverge on whether a given serialized program is valid, which is an API-equivalence violation with potential consensus impact.

The existing fuzz harness `fuzz/fuzz_targets/serialized_length.rs` verifies agreement between `serialized_length_from_bytes` and `node_from_bytes_backrefs`, but no harness verifies agreement between `is_canonical_serialization` and `serialized_length_from_bytes` on backref-containing blobs. The harness `canonical_serialization.rs` only exercises blobs that successfully parse through `node_from_stream`, which does not handle back-references. [5](#0-4) 

---

### Likelihood Explanation

The blob is trivially constructable (4 bytes). The divergence is deterministic and reproducible. Any caller relying on `is_canonical_serialization` as a complete validity gate for backref-containing serializations is affected.

---

### Recommendation

`is_canonical_serialization` must resolve back-references using the same parse-stack tracking logic as `serialized_length_from_bytes`, or its documentation must be updated to explicitly state it does not validate back-reference semantics and callers must not use it as a complete validity gate. A fuzz harness asserting `is_canonical_serialization(b) == serialized_length_from_bytes(b).is_ok()` for all backref-containing inputs should be added.
