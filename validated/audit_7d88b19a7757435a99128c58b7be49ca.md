### Title
Difficulty Calculation for Fork Blocks Uses Mainchain Ancestor Timestamps Instead of Fork Ancestor Timestamps - (File: `contract/src/dogecoin.rs`)

### Summary

In the Dogecoin difficulty adjustment logic, `get_next_work_required` fetches the boundary ancestor block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. When the block being validated belongs to a fork chain that diverged before the boundary height, this returns the **mainchain** block at that height rather than the fork's actual ancestor. The difficulty target is therefore computed from the wrong timestamp, allowing an attacker to craft fork blocks whose required PoW is lower than the protocol mandates.

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes the Digishield difficulty target using the time difference between the previous block and an ancestor block `blocks_to_go_back` heights earlier:

```rust
// contract/src/dogecoin.rs  lines 286-297
let height_first = prev_block_header
    .block_height
    .checked_sub(blocks_to_go_back)
    ...;

// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`blocks_getter` is the contract itself. Its `get_header_by_height` implementation always reads from `mainchain_height_to_header`:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`, until a reorg promotes them: [3](#0-2) 

For Dogecoin with the new difficulty protocol active at height ≥ 145,000, `difficulty_adjustment_interval` is set to 1, so every single block triggers a retarget and `blocks_to_go_back = 1`: [4](#0-3) 

This means that for every fork block at height H, the difficulty is computed as:

```
modulated_timespan = time(fork_block[H-1]) - time(mainchain_block[H-2])
```

instead of the correct:

```
modulated_timespan = time(fork_block[H-1]) - time(fork_block[H-2])
```

The developers themselves flagged this with a `TODO` comment at line 291, acknowledging the concern but leaving it unresolved. [5](#0-4) 

The same structural defect exists in the Bitcoin difficulty path: [6](#0-5) 

### Impact Explanation

An attacker who submits a fork chain diverging at height H-3 controls the timestamps of fork blocks at H-2 and H-1. By setting the fork block at H-2 to a timestamp significantly later than the mainchain block at H-2, the attacker makes `time(fork[H-1]) - time(mainchain[H-2])` smaller than `time(fork[H-1]) - time(fork[H-2])`. The Digishield formula then computes a lower (easier) difficulty target for the next fork block. Because every block recalculates difficulty in the new Dogecoin protocol, this effect compounds across the entire fork chain. A fork chain accepted with systematically under-required PoW can accumulate enough `chain_work` to trigger `reorg_chain`, replacing the honest mainchain. Downstream consumers calling `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` against the reorged chain will receive incorrect SPV proofs for transactions that were never in the true Bitcoin/Dogecoin canonical chain. [7](#0-6) 

### Likelihood Explanation

The entry point is the public, unprivileged `submit_blocks` call (gated only by the optional trusted-relayer stake, which is bypassable via `Role::UnrestrictedSubmitBlocks` or simply by staking). The attacker needs to submit a fork chain with valid but under-difficulty PoW — feasible for Dogecoin whose current difficulty is orders of magnitude lower than Bitcoin's. The desynchronization is triggered automatically whenever the fork diverges before `height_first`, which is guaranteed for any fork longer than one block in the new difficulty protocol. No privileged access, leaked keys, or social engineering is required. [8](#0-7) 

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor traversal that walks the fork chain backwards through `headers_pool` using `prev_block_hash` links, rather than looking up by height in the mainchain index. This mirrors how Bitcoin Core resolves the retarget boundary block: by following the `pprev` pointer chain from the current tip, not by a height-indexed lookup into the active chain.

### Proof of Concept

1. The contract is initialized with Dogecoin mainchain at height H ≥ 145,000. Mainchain block at H-2 has timestamp `T_main = 1_000_000`.

2. Attacker calls `submit_blocks` with a fork starting at H-3:
   - Fork block at H-2: timestamp `T_fork_H2 = 1_001_000` (1000 s later than mainchain H-2).
   - Fork block at H-1: timestamp `T_fork_H1 = 1_001_060` (60 s after fork H-2, normal spacing).

3. When the attacker submits fork block at H, `check_pow` calls `get_next_work_required`:
   - `height_first = H-2`
   - `get_header_by_height(H-2)` returns **mainchain** block with `time = 1_000_000`.
   - `modulated_timespan = 1_001_060 - 1_000_000 = 1060` (instead of correct `1_001_060 - 1_001_000 = 60`).
   - Digishield clamps and computes a significantly lower difficulty target than the honest 60-second spacing would produce.

4. The attacker mines fork block at H against this reduced target, passes `check_pow`, and the block is accepted.

5. Repeating across subsequent blocks compounds the difficulty reduction. Once `fork.chain_work > mainchain.chain_work`, `reorg_chain` is triggered and the attacker's fork becomes the canonical chain. [9](#0-8)

// source https://github.com/dogecoin/dogecoin/blob/2c513d0172e8bc86fe9a337693b26f2fdf68a013/src/dogecoin.cpp#L41
fn calculate_next_work_required(
    config: &DogecoinConfig,
    prev_block_header: &ExtendedHeader,
    first_block_time: i64,
) -> u32 {
    let retarget_timespan = config.pow_target_timespan;
    let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;

    let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;

    let min_timespan = retarget_timespan - (retarget_timespan / 4);
    let max_timespan = retarget_timespan + (retarget_timespan / 2);

    if modulated_timespan < min_timespan {
        modulated_timespan = min_timespan;
    } else if modulated_timespan > max_timespan {
        modulated_timespan = max_timespan;
    }

    let new_target = target_from_bits(prev_block_header.block_header.bits);

    let (mut new_target, new_target_overflow) =
        new_target.overflowing_mul(<i64 as TryInto<u64>>::try_into(modulated_timespan).unwrap());
    require!(!new_target_overflow, "new target overflow");
    new_target =
        new_target / U256::from(<i64 as TryInto<u64>>::try_into(retarget_timespan).unwrap());

    if new_target > config.pow_limit {
        new_target = config.pow_limit;
    }

    new_target.target_to_bits()
}
```

    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
