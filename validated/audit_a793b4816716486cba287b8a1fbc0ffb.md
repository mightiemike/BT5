### Title
Outgoing Gas Limit Over-Consumed Because Admission Check Uses Clamped `admission_gas` But Deduction Subtracts Full `gas` — (`runtime/runtime/src/congestion_control.rs`)

---

### Summary

In `ReceiptSinkV2::try_forward`, the `ClampOutgoingGasAdmission` protocol feature (stable at version 85) introduces `admission_gas = gas.min(allowed_shard_outgoing_gas)` for the gate check, but the subsequent deduction from `forward_limit.gas` still uses the unclamped `gas`. This is the exact nearcore analog of the ETHCrowdfundBase M-03 pattern: a validation check is performed on a reduced value, but the downstream accounting uses the original (larger) value, causing the limit to be over-consumed and breaking the congestion-control invariant.

---

### Finding Description

`try_forward` in `runtime/runtime/src/congestion_control.rs` computes a clamped admission value:

```rust
let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
    .enabled(apply_state.current_protocol_version)
{
    gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
} else {
    gas
};

if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(gas);   // ← uses full `gas`, not `admission_gas`
    forward_limit.size -= size;
    ...
}
``` [1](#0-0) 

When `ClampOutgoingGasAdmission` is active, `admission_gas` is capped at `allowed_shard_outgoing_gas` (1 PGAS in production). A receipt whose actual `gas` exceeds that cap passes the gate check (`forward_limit.gas >= admission_gas`), but the deduction `forward_limit.gas.saturating_sub(gas)` subtracts the full, unclamped `gas`. Because `gas` can be up to 300 PGAS (the maximum prepaid gas), a single such receipt drives `forward_limit.gas` to zero via saturating subtraction, exhausting the entire per-chunk forwarding budget for that shard in one step.

The `allowed_shard_outgoing_gas` limit exists precisely to guarantee that a fully-congested shard always accepts at least that much work from its designated "allowed shard" each round — the deadlock-prevention invariant of the congestion-control protocol. [2](#0-1) 

The `ClampOutgoingGasAdmission` feature is assigned to stable protocol version 85: [3](#0-2) 

`allowed_shard_outgoing_gas` is 1 PGAS in the production runtime config: [4](#0-3) 

---

### Impact Explanation

**High.** The congestion-control system's deadlock-prevention guarantee is that the designated allowed shard can always drain at least `allowed_shard_outgoing_gas` of buffered receipts per chunk. With this bug, a single buffered receipt whose `gas > allowed_shard_outgoing_gas` passes the admission gate but over-consumes the limit, leaving `forward_limit.gas = 0`. All subsequent receipts in the same chunk's forwarding loop are blocked (`forward_limit.gas >= admission_gas` fails for every following receipt). If the outgoing buffer contains many such large receipts, only one drains per chunk instead of the expected quota, causing the buffer to grow faster than it drains and potentially producing a sustained liveness failure (receipts stuck indefinitely) on the congested shard.

---

### Likelihood Explanation

**Medium.** The condition requires:
1. A shard to be fully congested (`congestion_level() == 1.0`), activating the allowed-shard path.
2. At least one buffered receipt with `gas > allowed_shard_outgoing_gas` (> 1 PGAS). Any function call attaching more than 1 PGAS — the normal case for cross-contract calls — satisfies this.
3. Protocol version ≥ 85 (stable mainnet/testnet).

All three conditions are routinely met in production under load.

---

### Recommendation

Deduct `admission_gas` from `forward_limit.gas` instead of the unclamped `gas`, so the gate check and the accounting use the same value:

```rust
if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(admission_gas); // use clamped value
    forward_limit.size -= size;
    ...
}
```

This preserves the invariant that the limit is consumed by exactly the value that was tested in the gate, preventing over-consumption while still allowing large receipts to be forwarded.

---

### Proof of Concept

**Setup:** Shard A is fully congested. Shard B is the designated allowed shard. `forward_limit.gas` for shard A (from shard B's perspective) is set to `allowed_shard_outgoing_gas = 1 PGAS`. Shard B's outgoing buffer contains two receipts, each with `gas = 300 PGAS`.

**Execution of `try_forward` for receipt 1:**
- `admission_gas = min(300 PGAS, 1 PGAS) = 1 PGAS`
- Gate check: `1 PGAS >= 1 PGAS` → **passes**
- Deduction: `forward_limit.gas = 1 PGAS - 300 PGAS = 0` (saturating)
- Receipt 1 is forwarded.

**Execution of `try_forward` for receipt 2:**
- `admission_gas = min(300 PGAS, 1 PGAS) = 1 PGAS`
- Gate check: `0 PGAS >= 1 PGAS` → **fails**
- Receipt 2 stays in the buffer.

**Result:** Only one receipt drains per chunk instead of the expected quota. The buffer grows unboundedly under sustained load, breaking the deadlock-prevention guarantee of the congestion-control protocol. [5](#0-4)

    fn try_forward(
        receipt: Receipt,
        gas: Gas,
        mut size: u64,
        shard: ShardId,
        outgoing_limit: &mut HashMap<ShardId, OutgoingLimit>,
        outgoing_receipts: &mut Vec<Receipt>,
        apply_state: &ApplyState,
        stats: &mut ReceiptSinkStats,
    ) -> Result<ReceiptForwarding, RuntimeError> {
        // There is a bug which allows to create receipts that are above the size limit. Receipts
        // above the size limit might not fit under the maximum outgoing size limit. Let's pretend
        // that all receipts are at most `max_receipt_size` to avoid receipts getting stuck.
        // See https://github.com/near/nearcore/issues/12606
        let max_receipt_size = apply_state.config.wasm_config.limit_config.max_receipt_size;
        if size > max_receipt_size {
            tracing::debug!(
                target: "runtime",
                receipt_id=?receipt.receipt_id(),
                size,
                max_receipt_size,
                "try_forward observed a receipt with size exceeding the size limit",
            );
            size = max_receipt_size;
        }

        // Default case set to `Gas::MAX`: If no outgoing limit was defined for the receiving
        // shard, this usually just means the feature is not enabled. Or, it
        // could be a special case during resharding events. Or even a bug. In
        // any case, if we cannot know a limit, treating it as literally "no
        // limit" is the safest approach to ensure availability.
        let default_gas_limit = Gas::MAX;

        // Since bandwidth scheduler, a shard is not allowed to send any receipts if it doesn't have a grant.
        let default_size_limit = 0;

        let default_outgoing_limit =
            OutgoingLimit { gas: default_gas_limit, size: default_size_limit };
        let forward_limit = outgoing_limit.entry(shard).or_insert(default_outgoing_limit);

        let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
            .enabled(apply_state.current_protocol_version)
        {
            gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
        } else {
            gas
        };

        if forward_limit.gas >= admission_gas && forward_limit.size >= size {
            tracing::trace!(target: "runtime", ?shard, receipt_id=?receipt.receipt_id(), "forwarding buffered receipt");
            outgoing_receipts.push(receipt);
            forward_limit.gas = forward_limit.gas.saturating_sub(gas);
            forward_limit.size -= size;
            stats.forwarded_receipts.entry(shard).or_default().add_receipt(size, gas);

            Ok(ReceiptForwarding::Forwarded)
        } else {
            tracing::trace!(target: "runtime", ?shard, receipt_id=?receipt.receipt_id(), "not forwarding buffered receipt");
            Ok(ReceiptForwarding::NotForwarded(receipt))
        }
    }
