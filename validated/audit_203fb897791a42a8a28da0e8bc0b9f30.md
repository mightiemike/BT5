### Title
Fixed Liquidation Penalty Based on Maintenance Weight Causes Unnecessary Insurance Fund Drain for Technically Solvent Spot Liability Positions — (File: `core/contracts/ClearinghouseStorage.sol`)

---

### Summary

The liquidation price for spot liability positions is computed using a fixed penalty derived from the maintenance weight, with a minimum floor of 0.5% (`MIN_NON_SPREAD_LIQ_PENALTY_X18`). When a position is technically solvent (quote balance ≥ oracle price × |liability|) but under maintenance health, the penalty-inflated liquidation payment can exceed the available quote balance. The insurance fund then covers the shortfall via `updateQuoteFromInsurance`, draining it unnecessarily — an exact structural analog to M-04.

---

### Finding Description

**Root cause — `getLiqPriceX18` in `ClearinghouseStorage.sol`:** [1](#0-0) 

For a spot liability (`amount < 0`), the maintenance weight is `shortWeightMaintenance > 1`, so:

```
penaltyX18 = (shortWeightMaintenance - ONE) / 5   // e.g. (1.1 - 1) / 5 = 2%
// floor enforced:
if penaltyX18.abs() < MIN_NON_SPREAD_LIQ_PENALTY_X18 (0.5%) → use 0.5%
liquidationPrice = oraclePrice * (1 + penaltyX18)  // e.g. 1.02 * oraclePrice
```

The penalty is fixed by the risk parameter, not by the actual available quote balance of the liquidatee.

**Execution path — `_handleLiquidationPayment` in `ClearinghouseLiq.sol`:** [2](#0-1) 

For a spot liability liquidation (`amount < 0`):
1. Liquidatee's spot balance is reduced (liability cleared).
2. Liquidatee's quote balance is reduced by `liquidationPrice × |amount|` — the penalty-inflated price.
3. If the resulting quote balance is negative, `updateQuoteFromInsurance` is called: [3](#0-2) 

This tops up the liquidatee's negative quote from the insurance fund. If the insurance fund is insufficient, `socializeSubaccount` distributes the loss to all depositors. [4](#0-3) 

**The vulnerable range:**

A position is technically solvent when `quote_balance ≥ oracle_price × |spot_liability|`. It is liquidatable when maintenance health < 0 (i.e., `shortWeightMaintenance × oracle_price × |liability| > quote_balance + other_assets`). These two conditions can coexist. When they do, the liquidation payment at `liquidationPrice = oracle_price × (1 + penaltyX18)` exceeds the available quote, and the insurance fund absorbs the difference — even though the position could have been fully settled at oracle price with no insurance involvement.

The minimum floor `MIN_NON_SPREAD_LIQ_PENALTY_X18 = 0.5%` amplifies this: even positions with a natural penalty below 0.5% are charged the floor, widening the vulnerable range. [5](#0-4) 

The same structural issue applies to spread liquidations via `getSpreadLiqPriceX18` with `MIN_SPREAD_LIQ_PENALTY_X18 = 0.25%`. [6](#0-5) 

---

### Impact Explanation

**Impact: High.**

The insurance fund is drained for positions that are technically solvent. Each such liquidation transfers a portion of the insurance fund to the liquidator (via the fee split) without economic justification. If the insurance fund is depleted, `socializeSubaccount` is triggered, which reduces the `cumulativeDepositsMultiplierX18` for all depositors of the affected product — a direct, irreversible loss to all spot depositors. [7](#0-6) 

The `LIQUIDATION_FEE_FRACTION = 50%` means half of every penalty unit goes to the insurance fund and half to the liquidator. For a 2% penalty on a $100,000 position, the insurance fund is drained by up to $1,000 per liquidation event. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Low.**

The trigger requires a position to be in the specific range: `oracle_price × |liability| ≤ quote_balance < liquidation_price × |liability|`. This range is `penaltyX18 × oracle_price × |liability|` wide (e.g., 2% of position value for a 10% short maintenance weight). This scenario arises naturally during rapid price drops (flash crashes) where collateral value falls into this band — the same scenario described in M-04. It is not a common steady-state condition but is realistic during market stress.

---

### Recommendation

Before applying the penalty-inflated liquidation payment, check whether the liquidatee's quote balance is sufficient to cover the payment at oracle price. If the position is technically solvent at oracle price (i.e., `quote_balance ≥ oracle_price × |amount|`), cap the liquidation payment to `min(liquidation_price × |amount|, quote_balance)` to avoid unnecessary insurance fund drain. This mirrors the M-04 recommendation of capping the incentive to available collateral.

---

### Proof of Concept

**Setup:**
- Subaccount holds: `-100 ETH` (spot liability) + `101 USDC` (quote balance)
- ETH oracle price: `1 USDC`
- `shortWeightMaintenance = 1.10`

**Step 1 — Confirm liquidatable:**
Maintenance health = `101 - 1.10 × 100 = -9` → under maintenance, liquidatable. ✓

**Step 2 — Confirm technically solvent:**
Net value at oracle = `101 - 100 × 1 = +1 USDC` → solvent. ✓

**Step 3 — Compute liquidation price:**
`penaltyX18 = (1.10 - 1.0) / 5 = 0.02` (2%)
`liquidationPrice = 1.0 × 1.02 = 1.02 USDC/ETH`

**Step 4 — Apply liquidation payment:**
`liquidationPayment = 1.02 × (−100) = −102 USDC`
Liquidatee quote after: `101 − 102 = −1 USDC`

**Step 5 — Insurance drain:**
`updateQuoteFromInsurance` tops up `1 USDC` from the insurance fund.
`liquidationFees = (1.0 − 1.02) × 0.5 × (−100) = +1 USDC` → goes to insurance.

Net insurance impact: `−1 USDC` (drain) `+1 USDC` (fee) = `0` in this exact case. However, if the liquidatee's quote is lower (e.g., `100.5 USDC`), the drain is `1.5 USDC` while the fee is still `1 USDC`, resulting in a net `−0.5 USDC` insurance loss per liquidation. The drain scales with position size and the gap between available quote and liquidation payment. [2](#0-1) [1](#0-0) [9](#0-8)

    function updateQuoteFromInsurance(bytes32 subaccount, int128 insurance)
        external
        returns (int128)
    {
        _assertInternal();
        State memory state = states[QUOTE_PRODUCT_ID];
        BalanceNormalized memory balanceNormalized = balances[QUOTE_PRODUCT_ID][
            subaccount
        ];
        int128 balanceAmount = balanceNormalizedToBalance(
            state,
            balanceNormalized
        ).amount;
        if (balanceAmount < 0) {
            int128 topUpAmount = MathHelper.max(
                MathHelper.min(insurance, -balanceAmount),
                0
            );
            insurance -= topUpAmount;
            _updateBalanceNormalized(state, balanceNormalized, topUpAmount);
        }
        _setState(QUOTE_PRODUCT_ID, state);
        _setBalanceAndUpdateBitmap(
            QUOTE_PRODUCT_ID,
            subaccount,
            balanceNormalized
        );
        return insurance;
    }
