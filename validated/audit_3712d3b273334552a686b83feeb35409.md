Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-supplied `owner` instead of actual depositor `sender`, allowing full allowlist bypass — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address, but its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender`) and instead validates the caller-supplied `owner` parameter. Because `owner` is a free parameter any caller can set to any address, any unprivileged address can bypass the allowlist by naming an already-allowed address as `owner`, injecting liquidity into a restricted pool while paying tokens themselves via the callback.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully encodes both into the extension call: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then silently drops `sender` (first parameter is unnamed `address,`) and checks only `owner`: [3](#0-2) 

Since `owner` is freely chosen by the caller, an attacker picks any address already on the allowlist as `owner`. The extension evaluates `allowedDepositor[pool][allowedLP]` → `true` and does not revert. The pool then pulls tokens from the **actual caller** (the unauthorized address) via the liquidity callback, and the position is recorded under `allowedLP`. The attacker permanently loses the deposited tokens because `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), not `recipient`: [5](#0-4) 

The asymmetry confirms the deposit extension is checking the wrong actor.

## Impact Explanation
The allowlist guard is fully bypassed for deposits. Any address regardless of allowlist status can inject liquidity into a restricted pool. Concrete impacts:
1. **Compliance/KYC bypass**: Pools configured for permissioned LP sets accept liquidity from unauthorized parties.
2. **Pool state manipulation**: The unauthorized depositor controls which bins receive liquidity, moving `curBinIdx`/`curPosInBin` and altering bin balances, affecting swap pricing and potentially manipulating per-bin value metrics used by `OracleValueStopLossExtension`.
3. **LP dilution**: Unauthorized liquidity added at favorable prices dilutes existing LPs' share of future fee income.

This constitutes a broken core pool access-control mechanism with direct fund-impacting pool-state consequences, meeting the threshold for a High severity finding.

## Likelihood Explanation
- Requires no special role or privilege — any EOA or contract can call `pool.addLiquidity` directly.
- The bypass is trivially constructed: pick any address already on the allowlist as `owner`.
- The pool does not verify that `msg.sender == owner` for `addLiquidity` (only `removeLiquidity` enforces this).
- Fully repeatable with no cooldown or rate limit.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```solidity
// Setup: pool with DepositAllowlistExtension; only `allowedLP` is on the allowlist.
// `attacker` is NOT on the allowlist.

// Attacker calls addLiquidity directly, naming allowedLP as owner.
vm.prank(attacker);
pool.addLiquidity(
    allowedLP,       // owner — checked by extension → passes
    salt,
    deltas,
    callbackData,    // attacker's callback pays tokens
    extensionData
);
// Extension checks allowedDepositor[pool][allowedLP] == true → no revert.
// Pool pulls tokens from attacker via callback.
// Position recorded under allowedLP.
// Allowlist guard fully bypassed; attacker injected liquidity into a restricted pool.
```
