Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass Per-User Swap Gating via an Allowlisted Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is always `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is used, the router is `msg.sender` to the pool, so the extension checks whether the **router** is allowlisted — not the end user. Any address not on the allowlist can bypass per-user swap gating by routing through an allowlisted router, rendering the extension's access control completely ineffective.

## Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the extension:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards `sender` verbatim to the extension:** [2](#0-1) 

**Step 3 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`:**

`msg.sender` here is the pool (correct), and `sender` is whoever called `pool.swap()` — which is the **router**, not the end user: [3](#0-2) 

**Step 4 — `MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router `msg.sender` to the pool:** [4](#0-3) 

The original `msg.sender` (end user) is stored only in transient callback context for payment purposes via `_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn)` — it is never forwarded to the pool's `swap()` call as the `sender` argument. [5](#0-4) 

**Step 5 — The pool's `swap()` interface has no explicit `swapper` parameter; it always uses `msg.sender` internally:** [6](#0-5) 

**Existing test confirms the pattern:** `test_allowedSwapSucceeds` allowlists `callers[0]` (the intermediate caller, analogous to the router) and calls `_swap(0, users[0], ...)`. The test passes because the pool sees `callers[0]` as `msg.sender`, not `users[0]` — confirming the extension gates the intermediate caller, not the end user: [7](#0-6) 

## Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` intends to restrict swaps to a specific set of addresses. If the router is allowlisted — a natural operational choice since `MetricOmmSimpleRouter` is the canonical public swap entry point — every address on the network can bypass the per-user gate by calling `router.exactInputSingle()`. The allowlist is rendered completely ineffective for any pool that uses the router as an allowlisted swapper. This constitutes a broken access-control invariant with direct fund-flow impact: unauthorized swaps execute against the pool, violating the core functionality of the extension.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary public swap interface; pool admins will routinely allowlist it as the canonical entry point.
- No special knowledge or privilege is required — any EOA can call the router.
- The bypass is deterministic and requires zero preconditions beyond the router being allowlisted.
- The same bypass applies to `exactOutputSingle`, `exactInput`, and `exactOutput` router entry points, all of which call `pool.swap()` with the router as `msg.sender`.

## Recommendation

The extension must check the **originating user**, not the intermediate router. Two viable approaches:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a convention between router and extension.
2. **Add a `swapper` field to the pool's `swap()` interface**: The pool accepts an explicit `swapper` address (defaulting to `msg.sender` for direct calls), and the router passes the original `msg.sender`. This is the cleaner fix but requires a core interface change.
3. **Allowlist only EOAs / disallow router allowlisting**: Document that allowlisting a router defeats per-user gating. This is a documentation mitigation, not a code fix.

## Proof of Concept

```solidity
// Foundry integration test sketch
function test_unlisted_user_bypasses_allowlist_via_router() public {
    // Setup: pool with SwapAllowlistExtension, router allowlisted, user NOT allowlisted
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // allowedSwapper[pool][unlistedUser] is false (default)

    // Unlisted user routes through the allowlisted router
    vm.prank(unlistedUser);
    // Expect: should revert with NotAllowedToSwap, but it SUCCEEDS
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1000,
        amountOutMinimum: 0,
        recipient: unlistedUser,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    }));
    // Swap succeeds — allowlist bypassed
}
```

The existing `test_allowedSwapSucceeds` test in `FullMetricExtension.t.sol` (L68–74) confirms the design flaw: it allowlists `callers[0]` (the `TestCaller` contract, analogous to the router) and calls `_swap(0, users[0], ...)` — the test passes because the pool sees `callers[0]` as `msg.sender`, not `users[0]`. This confirms the extension gates the intermediate caller (router/TestCaller), not the end user, and any user behind that intermediate caller is implicitly permitted. [7](#0-6)
