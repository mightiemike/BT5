### Title
`DepositAllowlistExtension` checks LP position `owner` instead of transaction `sender`, allowing any address to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor) and gates on `owner` (a free caller-supplied parameter). Because `owner` is arbitrary input to `MetricOmmPool.addLiquidity`, any unprivileged address can bypass the deposit allowlist by naming an allowlisted address as `owner`.

### Finding Description
The contract's own NatSpec and mapping name confirm the intent:

> "Gates `addLiquidity` by depositor address, per pool."
> `mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;` [1](#0-0) 

Yet the hook implementation discards `sender` (first parameter, unnamed) and checks `owner` instead:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [2](#0-1) 

In `MetricOmmPool.addLiquidity`, `owner` is a free parameter supplied by the caller, and `msg.sender` is forwarded as `sender` to the extension:

```solidity
function addLiquidity(
    address owner,          // ← caller-controlled
    ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ... {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [3](#0-2) 

`ExtensionCalling._beforeAddLiquidity` encodes both arguments and passes them to the extension in order `(sender, owner, ...)`: [4](#0-3) 

The extension receives `sender` as its first argument but ignores it entirely, checking only `owner`. An attacker sets `owner` to any allowlisted address; the guard approves; the attacker's tokens are pulled via the callback; the LP position is minted to the named `owner`. The actual depositor is never validated.

### Impact Explanation
The deposit allowlist is rendered completely ineffective. Any unprivileged address can add liquidity to a pool the admin intended to restrict. Concrete consequences:

- **Admin-boundary break**: the pool admin's access-control configuration is circumvented without any privilege.
- **Pool-state manipulation**: the attacker can inject liquidity into arbitrary bins, shifting the pool's price distribution and per-share metrics consumed by `OracleValueStopLossExtension` watermark logic.
- **Regulatory / risk-management violation**: pools using the allowlist for KYC or counterparty-risk reasons are silently open to all depositors.

### Likelihood Explanation
Exploitation requires only one publicly observable allowlisted address (e.g., the pool admin's own address, which is readable from the factory) and a single `addLiquidity` call. No special privileges, flash loans, or multi-step setup are needed.

### Recommendation
Use `sender` (the actual depositor) instead of `owner` in the guard:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept
1. Pool admin deploys a pool with `DepositAllowlistExtension`; only `allowedLP` is permitted to deposit (`allowedDepositor[pool][allowedLP] = true`).
2. Attacker (any EOA or contract) calls `pool.addLiquidity(allowedLP, salt, deltas, callbackData, "")`.
3. `DepositAllowlistExtension.beforeAddLiquidity` evaluates `allowedDepositor[pool][allowedLP]` → `true` → no revert.
4. `LiquidityLib.addLiquidity` pulls tokens from the attacker via the swap callback and mints the LP position to `allowedLP`.
5. The attacker has deposited into a restricted pool without appearing on the allowlist; the actual depositor (`msg.sender`) was never checked.
