### Title
`verify_transaction_inclusion` Unconditionally Rejects Valid Empty Merkle Proofs for Single-Transaction Blocks — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (and by extension `verify_transaction_inclusion_v2`) contains a guard that unconditionally panics when `merkle_proof` is empty. For Bitcoin blocks containing only a single transaction (the coinbase), an empty merkle proof is the only mathematically correct proof — the merkle root equals the transaction hash directly. The underlying `compute_root_from_merkle_proof` function handles this case correctly, but the guard prevents it from ever being reached, causing all SPV verification calls for single-transaction blocks to revert.

---

### Finding Description

In `verify_transaction_inclusion`, the following guard is applied unconditionally before the merkle computation:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This guard then feeds into:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
``` [2](#0-1) 

The underlying `compute_root_from_merkle_proof` in `merkle-tools` correctly handles an empty proof — it simply returns the input `transaction_hash` unchanged, because the `for` loop over an empty slice never executes:

```rust
for proof_hash in merkle_proof {
    // never executes for empty proof
}
current_hash  // returns transaction_hash unchanged
``` [3](#0-2) 

For a Bitcoin block containing exactly one transaction, the merkle tree has a single leaf. The merkle root is defined as that leaf's hash — no sibling nodes exist, so the proof is legitimately empty. The correct verification is simply `tx_id == merkle_root`. The computation function would produce this result correctly, but the `require!` guard panics before it is reached.

`verify_transaction_inclusion_v2` is also affected because it calls the deprecated v1 function internally after its own coinbase check:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [4](#0-3) 

For a single-transaction block, `coinbase_merkle_proof` would also be empty, so the length-equality check in v2 passes (`0 == 0`), and the coinbase root check also passes (since `compute_root_from_merkle_proof` with an empty proof returns `coinbase_tx_id`, which equals `merkle_root` for a single-tx block). The call then falls through to v1 and panics.

Both functions are publicly callable by any unprivileged NEAR account — neither carries `#[private]` nor `#[trusted_relayer]`:

```rust
#[pause]
pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
``` [5](#0-4) 

```rust
#[pause]
pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
``` [6](#0-5) 

---

### Impact Explanation

**High.** The primary public API of the BTC light client is SPV transaction inclusion verification. Single-transaction blocks (containing only the coinbase) are a well-defined, valid class of Bitcoin blocks. Any downstream contract or application that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction in such a block will receive a panic/revert instead of a boolean result. This permanently breaks the light client's core function for this block class. Coinbase transaction verification — a common use case for bridge and cross-chain protocols — is entirely non-functional.

---

### Likelihood Explanation

**Medium.** Single-transaction blocks occur on Bitcoin mainnet during low-fee or low-activity periods, and were common in Bitcoin's early history. Any relayer or consumer application that needs to prove coinbase inclusion (e.g., for miner reward verification or early-block anchoring) will trigger this bug. The trigger requires no special privileges — any NEAR caller supplying a valid single-tx block proof hits it deterministically.

---

### Recommendation

Remove the unconditional `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. Instead, handle the empty-proof case explicitly:

```rust
if args.merkle_proof.is_empty() {
    return args.tx_id == header.block_header.merkle_root;
}
```

This matches the mathematical definition of a single-leaf merkle tree and is consistent with what `compute_root_from_merkle_proof` would compute if allowed to run.

---

### Proof of Concept

1. A Bitcoin block exists with exactly one transaction (the coinbase). Its `merkle_root` equals `coinbase_tx_hash`.
2. A NEAR caller submits `verify_transaction_inclusion` with:
   - `tx_id = coinbase_tx_hash`
   - `tx_block_blockhash = <valid block hash in mainchain>`
   - `tx_index = 0`
   - `merkle_proof = []` (empty — the only correct proof)
   - `confirmations = 1`
3. Execution reaches `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")`.
4. The call panics with `"Merkle proof is empty"` and reverts.
5. The proof is mathematically valid: `compute_root_from_merkle_proof(coinbase_tx_hash, 0, &[])` would return `coinbase_tx_hash == merkle_root`, but this line is never reached.

The same sequence applies to `verify_transaction_inclusion_v2` with `coinbase_merkle_proof = []` and `coinbase_tx_id = coinbase_tx_hash`.
