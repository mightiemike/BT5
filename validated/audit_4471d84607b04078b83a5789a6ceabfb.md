### Title
`Clearinghouse.depositInsurance` Never Transfers Tokens to Contract, Causing `withdrawInsurance` to Drain User Deposit Collateral — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.depositInsurance` only increments the `insurance` accounting variable without performing any actual token transfer into the contract. When `withdrawInsurance` is subsequently called, it executes `token.safeTransfer(withdrawPool, txn.amount)` from `Clearinghouse`'s own balance — which is funded exclusively by user collateral deposits. This means every insurance withdrawal silently drains user deposit tokens rather than insurance-specific funds, breaking the solvency invariant.

---

### Finding Description

`Clearinghouse.depositInsurance` is the only mechanism for crediting the protocol insurance fund. Its implementation is:

```solidity
function depositInsurance(bytes calldata transaction) external virtual onlyEndpoint {
    IEndpoint.DepositInsurance memory txn = abi.decode(transaction[1:], (IEndpoint.DepositInsurance));
    require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(QUOTE_PRODUCT_ID)));
    int128 amount = int128(txn.amount) * int128(multiplier);
    insurance += amount;  // @audit: only accounting update, no safeTransferFrom
}
``` [1](#0-0) 

There is no `safeTransferFrom` call. The `insurance` variable is incremented as a pure accounting entry with no corresponding token custody.

`Clearinghouse` holds real tokens because user collateral deposits are transferred into it via `handleDepositTransfer` in the `Endpoint` deposit flow, and `handleWithdrawTransfer` confirms this by transferring tokens **from** `Clearinghouse`:

```solidity
function handleWithdrawTransfer(IERC20Base token, address to, uint128 amount, uint64 idx) internal virtual {
    token.safeTransfer(withdrawPool, uint256(amount));  // transfers FROM Clearinghouse
    BaseWithdrawPool(withdrawPool).submitWithdrawal(token, to, amount, idx);
}
``` [2](#0-1) 

When `withdrawInsurance` is called, it passes the accounting check (`amount <= insurance`) and then calls `handleWithdrawTransfer`:

```solidity
require(amount <= insurance, ERR_NO_INSURANCE);
insurance -= amount;
...
handleWithdrawTransfer(token, txn.sendTo, txn.amount, idx);
``` [3](#0-2) 

Because `Clearinghouse` holds only user deposit tokens (not separately custodied insurance tokens), `handleWithdrawTransfer` transfers user collateral out of the contract to satisfy the insurance withdrawal. The `insurance` accounting variable and the actual token balance are permanently desynchronized.

The `_balanceOf` helper confirms `Clearinghouse` is expected to hold tokens directly: [4](#0-3) 

---

### Impact Explanation

Every call to `withdrawInsurance` drains tokens from the pool of user-deposited collateral. After insurance is withdrawn, users who attempt to withdraw their own collateral via `withdrawCollateral` → `handleWithdrawTransfer` will receive less than they are owed, or the call will revert entirely once the pool is exhausted. This directly breaks the core solvency invariant: user collateral balances recorded in `SpotEngine` no longer correspond to tokens held in `Clearinghouse`. The protocol accrues bad debt proportional to the total insurance ever withdrawn.

---

### Likelihood Explanation

`withdrawInsurance` is callable by the sequencer via `onlyEndpoint` as a routine operational action. The sequencer is not required to be malicious — it is expected to call `withdrawInsurance` to recover insurance funds. The bug triggers on every legitimate insurance withdrawal. No special conditions, attacker-controlled inputs, or compromised keys are required beyond the sequencer performing its normal duties.

---

### Recommendation

`depositInsurance` must perform an actual `safeTransferFrom` to transfer the quote token into `Clearinghouse` before incrementing the `insurance` variable, mirroring the pattern used for user collateral deposits. Alternatively, maintain a separate token custody contract for insurance funds so that insurance withdrawals cannot draw from the user collateral pool.

```solidity
function depositInsurance(bytes calldata transaction) external virtual onlyEndpoint {
    IEndpoint.DepositInsurance memory txn = abi.decode(transaction[1:], (IEndpoint.DepositInsurance));
    require(txn.amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
    // Transfer actual tokens into Clearinghouse before updating accounting
    IERC20Base token = IERC20Base(_spotEngine().getConfig(QUOTE_PRODUCT_ID).token);
    token.safeTransferFrom(msg.sender, address(this), txn.amount); // ADD THIS
    int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(QUOTE_PRODUCT_ID)));
    int128 amount = int128(txn.amount) * int128(multiplier);
    insurance += amount;
}
```

### Proof of Concept

1. Protocol deploys `Clearinghouse` and `SpotEngine`.
2. User A deposits 1,000 USDC collateral via `Endpoint.depositCollateral`. Tokens are transferred to `Clearinghouse`. `SpotEngine` records User A's balance as 1,000 USDC.
3. Sequencer submits a `DepositInsurance` transaction for 500 USDC. `Clearinghouse.depositInsurance` increments `insurance` to 500e18. **No tokens are transferred to `Clearinghouse`.** `Clearinghouse` still holds only 1,000 USDC (User A's deposit).
4. Sequencer submits a `WithdrawInsurance` transaction for 500 USDC. `withdrawInsurance` passes the check `500e18 <= insurance`, decrements `insurance`, and calls `handleWithdrawTransfer` which executes `token.safeTransfer(withdrawPool, 500)`. **500 USDC is transferred out of `Clearinghouse` from User A's collateral.**
5. `Clearinghouse` now holds only 500 USDC, but `SpotEngine` still records User A's balance as 1,000 USDC.
6. User A calls `withdrawCollateral` for 1,000 USDC. `handleWithdrawTransfer` attempts `token.safeTransfer(withdrawPool, 1000)` but `Clearinghouse` only holds 500 USDC. The transaction reverts. User A cannot recover their full collateral. [1](#0-0) [5](#0-4) [2](#0-1)
