### Title
Sanctions Check Only at Slow-Mode Submission, Not at Execution — Sanctioned User Can Pre-Queue Collateral Withdrawal That Executes After Sanctioning - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`submitSlowModeTransactionImpl` enforces `requireUnsanctioned(sender)` only at the moment a slow-mode transaction is **queued**. When the queued transaction is later **executed** via `processSlowModeTransactionImpl`, no sanction check is performed. A user who submits a `WithdrawCollateral` slow-mode transaction before being added to the sanctions list will have their withdrawal processed three days later, fully bypassing the sanctions enforcement mechanism.

---

### Finding Description

**Submission path — sanction check present:**

In `EndpointTx.submitSlowModeTransactionImpl`, after fee handling, the contract checks:

```solidity
requireUnsanctioned(sender);
slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: transaction
});
``` [1](#0-0) 

The transaction is queued with a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`).

**Execution path — sanction check absent:**

When the queued transaction is executed, `Endpoint._executeSlowModeTransaction` calls `processSlowModeTransaction`, which delegates to `processSlowModeTransactionImpl`. The `WithdrawCollateral` branch there is:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:], (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    clearinghouse.withdrawCollateral(
        txn.sender, txn.productId, txn.amount, address(0), nSubmissions
    );
``` [2](#0-1) 

There is no `requireUnsanctioned` call here. `clearinghouse.withdrawCollateral` also contains no sanction check: [3](#0-2) 

`requireUnsanctioned` is defined in `EndpointStorage` and only queries the external `ISanctionsList`: [4](#0-3) 

**Exploit flow:**

1. User (soon-to-be sanctioned) calls `Endpoint.submitSlowModeTransaction(WithdrawCollateral{...})`.
2. `requireUnsanctioned` passes — user is not yet on the list.
3. Transaction is stored with `executableAt = now + 3 days`.
4. User is added to the OFAC/sanctions list.
5. After 3 days, anyone calls `Endpoint.executeSlowModeTransaction()`.
6. `_executeSlowModeTransaction` → `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral` — no sanction check at any step.
7. Collateral is transferred to the sanctioned address via `WithdrawPool`. [5](#0-4) 

---

### Impact Explanation

A sanctioned user successfully withdraws all collateral from the protocol. The sanctions enforcement mechanism — the only on-chain control preventing sanctioned addresses from extracting value — is fully defeated. The asset delta is the entire collateral balance of the sanctioned subaccount. This is a direct regulatory compliance failure and a concrete fund-extraction path for a sanctioned party.

---

### Likelihood Explanation

OFAC sanctions list additions are public and often telegraphed (e.g., via blockchain analytics firms). A sophisticated user who anticipates sanctioning — or who monitors the mempool for a governance/admin transaction that would add them — can trivially submit a slow-mode withdrawal in advance. The 3-day delay window is generous: the user does not need to react in the same block, only before the sanctioning takes effect. No special privilege is required; `submitSlowModeTransaction` is an open external function. [6](#0-5) 

---

### Recommendation

Add `requireUnsanctioned` at execution time inside `processSlowModeTransactionImpl` for the `WithdrawCollateral` branch, checking both the `sender` (the queuing address) and the resolved `sendTo` recipient address. This mirrors the pattern already applied at deposit time:

```solidity
// In processSlowModeTransactionImpl, WithdrawCollateral branch:
requireUnsanctioned(sender);
requireUnsanctioned(address(uint160(bytes20(txn.sender))));
clearinghouse.withdrawCollateral(...);
```

This ensures that even a pre-queued withdrawal is blocked if the user is sanctioned by the time it executes.

---

### Proof of Concept

1. Alice holds 100,000 USDC collateral in subaccount `alice_subaccount`.
2. Alice learns she will be added to the sanctions list.
3. Alice calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateral), abi.encode(WithdrawCollateral{sender: alice_subaccount, productId: QUOTE_PRODUCT_ID, amount: 100000e6, nonce: N})))`.
4. `requireUnsanctioned(alice)` passes. Tx queued at index `k`, `executableAt = T+3days`.
5. At time `T+1day`, Alice is added to the OFAC sanctions list.
6. At time `T+3days+1`, Bob calls `Endpoint.executeSlowModeTransaction()`.
7. `processSlowModeTransactionImpl` executes the `WithdrawCollateral` branch with no sanction check.
8. `clearinghouse.withdrawCollateral` transfers 100,000 USDC to Alice's address.
9. Alice, a sanctioned party, has successfully extracted all collateral. [7](#0-6) [8](#0-7)

    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
    }
