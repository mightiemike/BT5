### Title
Linked Signer Can Overwrite Its Own Delegation — Unauthorized Subaccount Mutation via `LinkSigner` - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The fast-path `LinkSigner` handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`, permitting the **current linked signer** of a subaccount to sign and submit a new `LinkSigner` transaction. This allows a linked signer to replace itself (or any other linked signer) with an attacker-controlled address, permanently hijacking the subaccount's delegated signing authority without the subaccount owner's consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is handled as follows:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true   // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` calls `validateSignature`, which passes the current linked signer to `verifier.validateSignature`:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts the signature if it was produced by **either** the subaccount owner address **or** the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the current linked signer can produce a valid `SignedLinkSigner` transaction that sets `linkedSigners[subaccount]` to any arbitrary address, including an attacker-controlled one.

By contrast, the **slow-mode path** for `LinkSigner` correctly enforces owner-only authorization via `validateSender`, which checks that `msg.sender == address(uint160(bytes20(txn.sender)))`:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // owner-only check
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The fast-path and slow-path are inconsistent: the slow-mode path correctly restricts `LinkSigner` to the subaccount owner, but the fast-path (sequencer-submitted) allows the linked signer to perform this privileged mutation.

---

### Impact Explanation

The `linkedSigners` mapping is the sole authorization gate for all fast-path user operations — order placement (`MatchOrders`), withdrawals (`WithdrawCollateral`/`WithdrawCollateralV2`), NLP minting/burning, quote transfers, and liquidation initiation. A linked signer that overwrites itself with an attacker-controlled address grants the attacker full, persistent control over the victim subaccount's trading and withdrawal operations. The victim's only recourse is the slow-mode `LinkSigner` path, which has a mandatory 3-day delay (`SLOW_MODE_TX_DELAY`), during which the attacker retains full control and can drain the subaccount.

---

### Likelihood Explanation

A linked signer is explicitly designed to be a delegated, lower-trust key (e.g., an API key held by a trading bot or third-party service). Any party that obtains a linked signer key — through API key compromise, a malicious API provider, or a phishing attack — can immediately escalate to permanent subaccount control. The attack requires only the ability to submit a signed transaction to the sequencer, which is the normal user flow. No privileged access is needed.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`. Only the subaccount owner (the address embedded in the `sender` bytes32) should be permitted to modify the linked signer mapping. This aligns the fast-path with the already-correct slow-mode path.

```diff
 validateSignedTx(
     signedTx.tx.sender,
     signedTx.tx.nonce,
     transaction,
     signedTx.signature,
-    true
+    false
 );
``` [5](#0-4) 

---

### Proof of Concept

1. Victim subaccount `0xAlice...` has linked signer `0xApiKey` set via a prior `LinkSigner` transaction.
2. Attacker obtains `0xApiKey`'s private key (e.g., via API key leak).
3. Attacker constructs `SignedLinkSigner { tx: { sender: 0xAlice..., signer: 0xAttacker..., nonce: N }, signature: sig_by_0xApiKey }`.
4. Attacker submits this to the sequencer. The sequencer includes it in a batch.
5. `processTransactionImpl` decodes the transaction, calls `validateSignedTx(..., true)`.
6. `validateSignature` passes because `recovered == 0xApiKey == getLinkedSigner(0xAlice...)`.
7. `linkedSigners[0xAlice...] = 0xAttacker` is written.
8. Attacker now signs `WithdrawCollateral` transactions as the linked signer for `0xAlice...`, draining all collateral.
9. Victim's only recourse is the slow-mode `LinkSigner`, which takes 3 days to execute — during which the attacker retains full control. [1](#0-0) [4](#0-3)
