### Title
Missing Chain-ID in `SignableMessage` Domain Separator Enables Cross-Chain Replay of Meta-Transaction Signatures - (File: `core/primitives/src/signable_message.rs`)

---

### Summary

The `SignableMessage` domain separator used for `DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) meta-transaction signatures contains only a NEP-number-derived `u32` discriminant. It does not include the chain ID or any other chain-specific context. A `DelegateAction` signed by a user on one NEAR network (e.g., testnet) produces an identical hash on any other NEAR network (e.g., mainnet) when the field values match. A malicious relayer can therefore take a user's signed `DelegateAction` from one chain and submit it on another chain, executing the inner actions under the user's identity without their consent.

---

### Finding Description

`DelegateAction` (NEP-366) and `DelegateActionV2` (NEP-611) are the meta-transaction primitives in NEAR. A user signs the action off-chain; a relayer wraps it in an outer `SignedTransaction` and submits it on-chain. The user's signature is verified inside `apply_delegate_action()` by calling `signed_delegate_action.verify()`, which internally calls `get_nep461_hash()`.

`get_nep461_hash()` constructs a `SignableMessage` and hashes it:

```rust
// core/primitives/src/action/delegate.rs
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` is:

```rust
// core/primitives/src/signable_message.rs
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
```

`MessageDiscriminant` is a plain `u32` derived solely from the NEP number:

```rust
// core/primitives/src/signable_message.rs
impl From<SignableMessageType> for MessageDiscriminant {
    fn from(ty: SignableMessageType) -> Self {
        match ty {
            SignableMessageType::DelegateAction =>
                MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap(),
            SignableMessageType::DelegateActionV2 =>
                MessageDiscriminant::new_on_chain(NEP_611_GAS_KEYS).unwrap(),
        }
    }
}
```

`new_on_chain(nep)` simply computes `(1 << 30) + nep`. There is no chain ID, genesis hash, or any other network-specific context in the signed payload.

The `DelegateAction` struct itself also contains no chain-specific field:

```rust
// core/primitives/src/action/delegate.rs
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
```

The outer `SignedTransaction` that wraps the `DelegateAction` does include a `block_hash` (chain-specific), but that outer transaction is signed by the **relayer**, not the user. The user's signature covers only the `DelegateAction` payload, which is chain-agnostic.

The verification path in the runtime is:

```rust
// runtime/runtime/src/actions.rs
pub(crate) fn apply_delegate_action(...) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    ...
}
```

`verify()` calls `get_nep461_hash()` and checks the signature against it — with no chain-ID gate anywhere in the path.

---

### Impact Explanation

A malicious relayer who receives a user's `SignedDelegateAction` from testnet (or any NEAR-compatible chain) can wrap it in a new outer transaction on mainnet and submit it. If the following conditions hold on mainnet:

1. The same `sender_id` account exists.
2. The same `public_key` is registered as an access key.
3. The `nonce` in the `DelegateAction` is exactly `current_nonce + 1` for that key.
4. The current block height is below `max_block_height`.

…then the inner actions execute on mainnet under the user's identity. Inner actions can include `Transfer`, `FunctionCall`, `AddKey`, `DeleteKey`, etc. A user who signs a testnet `DelegateAction` to transfer tokens to a test address could have those same tokens drained on mainnet if the relayer is malicious and the nonce aligns.

The nonce condition is the primary practical barrier, but it is not a security guarantee: new accounts start at nonce 0, and a testnet `DelegateAction` with `nonce = 1` is immediately valid on a fresh mainnet account. Users who share keys across networks (a common practice) are directly exposed.

---

### Likelihood Explanation

- NEAR account IDs are not network-scoped; the same human-readable name (e.g., `alice.near`) can exist on mainnet and testnet simultaneously.
- Many developers and power users deliberately use the same key pair across networks for convenience.
- Relayers are third-party services; a compromised or malicious relayer is an explicit threat model for meta-transactions.
- The `max_block_height` field provides a time window but not a chain binding; on a fast chain the window can be hours or days.
- The attack requires no special privilege: any entity that receives the off-chain `SignedDelegateAction` (the relayer) can execute it.

---

### Recommendation

Include the chain ID in the signed payload. The minimal fix is to add the chain ID string to `SignableMessage` or to `DelegateAction` itself, so the Borsh-serialized bytes that are hashed differ across networks. Concretely:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub chain_id: String,   // e.g. "mainnet", "testnet"
    pub msg: &'a T,
}
```

This is the NEAR analog of EIP-712's `domainSeparator` which binds a signature to `chainId` and `verifyingContract`. Because adding `chain_id` changes the hash, this is a protocol-breaking change and must be gated behind a new protocol version, with old `DelegateAction` signatures (without chain_id) rejected after the upgrade boundary.

---

### Proof of Concept

**Setup**: Alice has account `alice.near` on both testnet and mainnet, using the same ED25519 key pair. Her mainnet key nonce is currently 0.

**Step 1** – Alice signs a testnet `DelegateAction` (nonce = 1, max_block_height = 9999999):

```rust
let delegate_action = DelegateAction {
    sender_id: "alice.near".parse().unwrap(),
    receiver_id: "bob.near".parse().unwrap(),
    actions: vec![transfer_action(10_000_000_000_000_000_000_000_000)], // 10 NEAR
    nonce: 1,
    max_block_height: 9_999_999,
    public_key: alice_key.public_key(),
};
let signed = SignedDelegateAction::sign(&alice_signer, delegate_action);
// Alice sends `signed` to the testnet relayer off-chain.
```

**Step 2** – Malicious relayer ignores testnet and submits on mainnet:

```rust
// Relayer wraps the SAME signed object in a mainnet outer transaction.
let outer_tx = SignedTransaction::from_actions(
    relayer_nonce,
    relayer_id.clone(),
    "alice.near".parse().unwrap(),  // outer receiver = DelegateAction sender
    &relayer_signer,
    vec![Action::Delegate(Box::new(signed))],
    mainnet_block_hash,             // valid mainnet block hash
);
// Submit to mainnet RPC.
```

**Step 3** – Mainnet runtime calls `apply_delegate_action`:

```rust
// signed_delegate_action.verify() calls:
//   hash(borsh([discriminant=(1<<30)+366, sender_id, receiver_id, actions, nonce=1, max_block_height, pubkey]))
// This hash is IDENTICAL to what Alice signed on testnet.
// Signature verifies. 10 NEAR is transferred from alice.near to bob.near on mainnet.
```

Alice's mainnet funds are transferred without her consent. The root cause is that `get_nep461_hash()` produces the same digest on every NEAR network for the same field values, because `SignableMessage` contains no chain-specific context. impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}
