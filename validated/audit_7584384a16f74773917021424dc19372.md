### Title
`ValidResourceBounds` Variant Misclassification in Protobuf Deserialization Produces Wrong Transaction Hash — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the Rust enum variant. An `AllResources` transaction whose `l2_gas` and `l1_data_gas` fields are both zero is silently re-classified as `L1Gas` on the receiving side. Because `get_tip_resource_bounds_hash` hashes a **different number of resource felts** for the two variants (2 for `L1Gas`, 3 for `AllResources`), the transaction hash computed after the round-trip does not match the hash the signer committed to. The codebase's own test file acknowledges the misclassification but only works around it in tests, leaving production code exposed.

---

### Finding Description

**Serialization** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 471–489):

```rust
impl From<ValidResourceBounds> for protobuf::ResourceBounds {
    fn from(value: ValidResourceBounds) -> Self {
        match value {
            ValidResourceBounds::L1Gas(l1_gas) => protobuf::ResourceBounds {
                l1_gas: Some(l1_gas.into()),
                l2_gas: Some(value.get_l2_bounds().into()),   // → zero ResourceBounds
                l1_data_gas: Some(ResourceBounds::default().into()), // → zero
            },
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
                protobuf::ResourceBounds {
                    l1_gas: Some(l1_gas.into()),
                    l2_gas: Some(l2_gas.into()),
                    l1_data_gas: Some(l1_data_gas.into()),
                },
        }
    }
}
```

When an `AllResources` transaction carries `l2_gas = 0` and `l1_data_gas = 0` (valid: `l1_gas` is non-zero), the wire bytes are **identical** to those produced by an `L1Gas` transaction.

**Deserialization** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–437):

```rust
impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        ...
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
        } else {
            ValidResourceBounds::AllResources(...)
        })
    }
}
```

The variant is inferred purely from field values, with no explicit discriminator. An `AllResources` transaction with zero `l2_gas`/`l1_data_gas` is reconstructed as `L1Gas`.

**Hash domain divergence** (`crates/starknet_api/src/transaction_hash.rs`, lines 188–211):

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];
    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts
        }
    });
    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```

`L1Gas` → Poseidon over `[tip, L1_GAS_felt, L2_GAS_felt]`
`AllResources` → Poseidon over `[tip, L1_GAS_felt, L2_GAS_felt, L1_DATA_GAS_felt]`

These are **structurally different hash preimages**. Even when `l1_data_gas = 0`, the `AllResources` path appends an extra zero felt, producing a different digest.

**Acknowledgement in test code** (`crates/apollo_protobuf/src/converters/consensus_test.rs`, lines 26–47):

```rust
// If all the fields of `AllResources` are 0 upon serialization,
// then the deserialized value will be interpreted as the `L1Gas` variant.
fn add_gas_values_to_transaction(transactions: &mut [ConsensusTransaction]) {
    ...
    resource_bounds.l2_gas.max_amount = GasAmount(1);  // workaround
    ...
}
```

The test patches `l2_gas.max_amount` to `1` to avoid the misclassification. No equivalent guard exists in production conversion paths.

**Downstream rejection** (`crates/starknet_api/src/rpc_transaction.rs`, lines 586–611):

```rust
impl TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3 {
    fn try_from(value: InvokeTransactionV3) -> Result<Self, Self::Error> {
        Ok(Self {
            resource_bounds: match value.resource_bounds {
                ValidResourceBounds::AllResources(bounds) => bounds,
                _ => return Err(StarknetApiError::OutOfRange { ... }),  // rejects L1Gas
            },
            ...
        })
    }
}
```

When the misclassified `L1Gas` variant reaches this conversion (used in the consensus-to-internal path), the transaction is hard-rejected with an error, even though it was a valid `AllResources` transaction accepted by the gateway.

---

### Impact Explanation

Two concrete impacts:

1. **Wrong hash bound to transaction (Critical/High — hash logic binds wrong hash):** In the P2P sync path, `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` reconstructs the transaction as `L1Gas`. Any subsequent call to `calculate_transaction_hash` on this object computes a hash over 2 resource felts instead of 3, producing a value that does not match the hash the account signed. A validator that re-derives the hash from the deserialized bytes will see a mismatch, causing incorrect receipt/state or a consensus split.

2. **Valid transaction rejected (High — valid transaction rejected before sequencing):** The conversion `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` returns `OutOfRange` for the `L1Gas` variant. A valid `AllResources` transaction with zero `l2_gas`/`l1_data_gas` that was accepted by the gateway is permanently dropped by any peer that receives it over the consensus P2P channel.

---

### Likelihood Explanation

Any user can submit a V3 invoke transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0`. This is a structurally valid `AllResources` transaction (non-zero fee via `l1_gas`). The gateway's `validate_resource_bounds` check passes because `max_possible_fee > 0`. No special privilege is required. The trigger is a normal user transaction submission.

---

### Recommendation

1. **Add an explicit variant discriminator** to the protobuf `ResourceBounds` message (e.g., a boolean `is_all_resources` flag or a separate oneof), so the deserializer does not rely on value inspection to recover the variant.

2. **Short-term**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, treat a present-but-zero `l1_data_gas` field differently from an absent one. The existing `TODO(Shahak)` comment already flags this: once 0.13.2 support is dropped, assert `l1_data_gas` is never `None` and always reconstruct as `AllResources` for post-0.13.3 messages.

3. **Remove the test workaround** `add_gas_values_to_transaction` and replace it with a proper fix so the round-trip is identity-preserving for all valid inputs.

---

### Proof of Concept

```
// Construct AllResources transaction with zero l2/l1_data gas
let original_bounds = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas:      ResourceBounds { max_amount: GasAmount(1000), max_price_per_unit: GasPrice(1) },
    l2_gas:      ResourceBounds::default(),   // zero
    l1_data_gas: ResourceBounds::default(),   // zero
});

// Hash computed by gateway / signer (AllResources path: 3 resource felts)
let hash_original = get_tip_resource_bounds_hash(&original_bounds, &Tip(0)).unwrap();
// Poseidon([0, concat(L1_GAS,1000,1), concat(L2_GAS,0,0), concat(L1_DATA,0,0)])

// Serialize to protobuf
let proto: protobuf::ResourceBounds = original_bounds.into();
// proto.l2_gas = Some({max_amount:0, max_price:0})
// proto.l1_data_gas = Some({max_amount:0, max_price:0})

// Deserialize on receiving node
let recovered = ValidResourceBounds::try_from(proto).unwrap();
// l1_data_gas.is_zero() && l2_gas.is_zero() → L1Gas variant ← WRONG

// Hash recomputed on receiving node (L1Gas path: 2 resource felts)
let hash_recovered = get_tip_resource_bounds_hash(&recovered, &Tip(0)).unwrap();
// Poseidon([0, concat(L1_GAS,1000,1), concat(L2_GAS,0,0)])

assert_ne!(hash_original, hash_recovered);  // ← divergence confirmed
// hash_original includes the extra L1_DATA_GAS zero felt; hash_recovered does not.

// Downstream: conversion to RpcInvokeTransactionV3 fails
let invoke_v3 = InvokeTransactionV3 { resource_bounds: recovered, ... };
let rpc: Result<RpcInvokeTransactionV3, _> = invoke_v3.try_into();
assert!(rpc.is_err());  // OutOfRange — valid transaction rejected
