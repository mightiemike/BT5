### Title
Stale `LinkSigner` Signature Remains Valid After Slow-Mode Override, Allowing Sequencer to Re-Link a Revoked Signer — (`core/contracts/EndpointTx.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `LINK_SIGNER_SIGNATURE` EIP-712 typehash does not commit to the **current** linked signer. There are two independent paths to mutate `linkedSigners[sender]`: a fast-mode signed path (consumes the nonce) and a slow-mode direct path (does **not** consume the nonce). Because the slow-mode path leaves the nonce untouched, any previously signed fast-mode `LinkSigner` transaction remains cryptographically valid after the slow-mode override executes. A malicious sequencer can hold a user's signed transaction, wait for the user's slow-mode revocation to land, and then replay the stale signature to re-link the revoked signer.

---

### Finding Description

**Two paths to mutate `linkedSigners`:**

**Path 1 — Fast mode (nonce consumed):**
In `processTransactionImpl()`, a `SignedLinkSigner` transaction is validated with `validateSignedTx()`, which calls `validateNonce()` and increments the sender's nonce before writing `linkedSigners[signedTx.tx.sender]`. [1](#0-0) 

**Path 2 — Slow mode (nonce NOT consumed):**
In `processSlowModeTransactionImpl()`, a `LinkSigner` transaction is decoded and applied directly to `linkedSigners[txn.sender]` after only a `validateSender` check. There is no call to `validateNonce()`. [2](#0-1) 

**The typehash does not commit to the current linked signer:**

```
"LinkSigner(bytes32 sender,bytes32 signer,uint64 nonce)"
``` [3](#0-2) 

The digest only covers `sender`, the new `signer`, and `nonce`. It does not include the value of `linkedSigners[sender]` at signing time. [4](#0-3) 

**Consequence:** After a slow-mode `LinkSigner` executes and changes `linkedSigners[Alice]` to a new address (e.g., `address(0)` to unlink), the nonce is unchanged. Any previously signed fast-mode `LinkSigner` transaction for that same nonce is still a valid signature and can be submitted by the sequencer to override the slow-mode result.

---

### Impact Explanation

A linked signer has broad authority: it can sign `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and order-matching transactions on behalf of the subaccount. [5](#0-4) 

If a user's intended slow-mode revocation is silently overridden, the revoked signer retains full signing authority. This enables unauthorized withdrawals, trades, or liquidations from the victim's subaccount — a direct asset-loss impact.

---

### Likelihood Explanation

The slow-mode mechanism exists precisely as a user-controlled safety valve against a misbehaving sequencer (3-day delay, direct on-chain submission). The attack requires the sequencer to:
1. Receive a user's signed `LinkSigner` transaction.
2. Withhold it while the user's slow-mode revocation executes.
3. Submit the stale transaction afterward.

This is entirely within the sequencer's normal operational capabilities (no key compromise required — only selective ordering/delay of already-held transactions). The protocol's own design acknowledges the sequencer as a semi-trusted party that the slow-mode path is meant to protect against.

---

### Recommendation

Include the **current** linked signer in the `LINK_SIGNER_SIGNATURE` typehash, mirroring the fix applied in the referenced IdRegistry report:

```solidity
string internal constant LINK_SIGNER_SIGNATURE =
    "LinkSigner(bytes32 sender,bytes32 signer,address currentLinkedSigner,uint64 nonce)";
```

When computing the digest, add `address(linkedSigners[signedTx.tx.sender])` as a field. This binds the signature to the state at signing time: if the slow-mode path changes `linkedSigners[sender]`, the old signature's digest no longer matches and the sequencer cannot replay it. [4](#0-3) 

---

### Proof of Concept

1. Alice signs `SignedLinkSigner(sender=Alice, signer=Bob, nonce=5)` and submits it to the sequencer (e.g., to grant Bob trading authority).
2. Alice discovers Bob is malicious. She calls `submitSlowModeTransaction()` with a `LinkSigner` payload setting `signer = address(0)` to revoke Bob. [2](#0-1) 
3. After the 3-day `SLOW_MODE_TX_DELAY`, the slow-mode transaction executes: `linkedSigners[Alice] = address(0)`. Alice's nonce is still `5`. [6](#0-5) 
4. The sequencer now submits Alice's original signed transaction (`nonce=5`, `signer=Bob`). `validateNonce` passes (nonce is still 5), the ECDSA signature is valid (the digest has not changed), and `linkedSigners[Alice]` is set back to Bob. [7](#0-6) 
5. Bob retains full signing authority over Alice's subaccount and can submit withdrawals or trades.
