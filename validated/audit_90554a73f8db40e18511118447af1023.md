### Title
Silent Skip in `wrapVaultAsset` Allows Donation-Inflated ERC4626 Vault to Permanently Strand User Funds in `DirectDepositV1` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

The `wrapVaultAsset` function in `ContractOwner.sol` uses `previewDeposit(assetBalance) != 0` as a gate before wrapping underlying asset tokens into ERC4626 vault shares. If an attacker inflates the vault's share price via a donation attack — causing `previewDeposit` to return `0` for a victim's balance — the function silently skips the deposit. The victim's underlying asset tokens remain permanently stranded inside the `DirectDepositV1` contract, inaccessible to the user, with no on-chain recovery path.

---

### Finding Description

`ContractOwner.wrapVaultAsset` is a publicly callable (`external`, no `onlyOwner`) function that wraps a user's underlying asset tokens (e.g., stETH) held in their `DirectDepositV1` address into the corresponding ERC4626 vault token (e.g., wstETH) registered as a Nado collateral product.

The critical guard at line 525 is:

```solidity
if (IERC4626Base(tokenAddr).previewDeposit(assetBalance) != 0) {
    DirectDepositV1(directDepositV1).withdraw(IIERC20Base(assetTokenAddr));
    ...
    IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
}
```

If `previewDeposit` returns `0`, the entire deposit block is skipped silently — no revert, no event, no fallback. The underlying asset tokens remain in `DirectDepositV1`.

An attacker can force `previewDeposit` to return `0` for a victim's balance by executing a donation attack on the ERC4626 vault: transferring a large amount of the underlying asset directly to the vault contract before any shares are minted. This inflates the share price (assets-per-share ratio) so that small deposit amounts round down to zero shares under the vault's `previewDeposit` implementation.

Once the silent skip occurs, the victim's underlying asset tokens are locked in `DirectDepositV1`. The only recovery function is:

```solidity
function withdraw(IIERC20Base token) external onlyOwner {
```

This is `onlyOwner` — the owner being `ContractOwner` itself. The victim has no self-service recovery path. Funds are stuck until an admin manually intervenes.

The `isWrapVaultAssetReady` helper at line 548 also returns `false` when `previewDeposit == 0`, meaning off-chain tooling would report the deposit as "not ready" and never retry, compounding the stuck-funds scenario.

---

### Impact Explanation

A victim who sends underlying asset tokens to their `DirectDepositV1` address expecting them to be wrapped and credited to their Nado subaccount will instead have those tokens permanently stranded. The tokens are not credited to any subaccount, cannot be used as collateral, and cannot be recovered by the user. The only recovery is admin-level intervention via `ContractOwner.withdrawFromDirectDepositV1`, which is `onlyOwner`. This constitutes a direct, concrete loss of user funds with no self-service remedy.

---

### Likelihood Explanation

The attacker must donate enough underlying asset to the ERC4626 vault to inflate the share price above the victim's deposit amount. This has a direct cost to the attacker. However:
- `wrapVaultAsset` is publicly callable by anyone, so the attacker can trigger the silent skip at will after the donation.
- The attacker does not need to interact with the Nado protocol itself to set up the attack — only with the external ERC4626 vault.
- The attack is permanent: once the silent skip occurs, the victim's funds are stuck until admin action.
- The `isWrapVaultAssetReady` check returning `false` means automated off-chain systems will not retry, making the stuck state persistent.

Likelihood: **Medium** — requires upfront capital for the donation but is permissionless to trigger and causes permanent fund loss.

---

### Recommendation

1. **Revert instead of silently skip**: Replace the silent `if` guard with an explicit revert when `previewDeposit` returns `0` for a non-zero `assetBalance`. This prevents funds from being silently stranded.

2. **Guard against zero-share minting**: Before calling `IERC4626Base.deposit`, verify that the returned share amount is non-zero. If it is zero, revert with a descriptive error rather than proceeding or skipping.

3. **Minimum deposit threshold**: Enforce a minimum `assetBalance` threshold (analogous to `MIN_DEPOSIT_AMOUNT`) before attempting the wrap, ensuring the deposit is economically meaningful relative to the vault's current share price.

---

### Proof of Concept

**Setup:**
- `tokenAddr` = ERC4626 vault token registered as a Nado collateral product (e.g., wstETH)
- `assetTokenAddr` = underlying asset (e.g., stETH)
- `victim` = user who has sent `1e15` stETH to their `DirectDepositV1` address

**Attack Steps:**

1. Attacker calls `stETH.transfer(address(wstETH_vault), 1e18)` — donating `1e18` stETH directly to the vault, inflating the share price to `>1e18` assets per share.

2. Attacker calls `ContractOwner.wrapVaultAsset(victimSubaccount, productId)`.

3. Inside `wrapVaultAsset`:
   - `assetBalance = stETH.balanceOf(directDepositV1)` → `1e15`
   - `IERC4626Base(wstETH).previewDeposit(1e15)` → `0` (rounds down due to inflated share price)
   - The `if` condition is `false` → entire deposit block is skipped.

4. Victim's `1e15` stETH remains in `DirectDepositV1`, never wrapped, never credited to their Nado subaccount.

5. Victim calls `creditDeposit()` on their `DirectDepositV1` — this iterates over Nado product tokens (vault shares), not the underlying asset, so the stETH balance is ignored and nothing is credited.

6. Victim has no further on-chain recourse. Funds are stuck until admin calls `withdrawFromDirectDepositV1`.
