### Title
`SwapAllowlistExtension` checks router address instead of actual swapper — allowlist fully bypassed via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` — the direct caller of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` is the router contract address, not the actual user. Any user can bypass a per-user swap allowlist by calling the pool through the router.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards this value verbatim to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` value is allowlisted: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter`, the router calls `pool.swap(...)`. Inside the pool, `msg.sender` is the router contract. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

The pool admin who configures the allowlist intends to restrict which **users** may trade. The extension stores entries keyed by swapper address (`allowedSwapper[pool][swapper]`), confirming the per-user intent: [4](#0-3) 

But the check is applied to the wrong address — the router — not the user.

The same structural mismatch exists for `addLiquidity`: the pool passes `msg.sender` (the `MetricOmmPoolLiquidityAdder`) as `sender` while the actual depositor is `owner`: [5](#0-4) 

If `DepositAllowlistExtension` follows the same pattern and checks `sender` instead of `owner`, the deposit allowlist is equally bypassed through the liquidity adder periphery.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a specific set of addresses (e.g., KYC'd counterparties, whitelisted market makers) provides **no effective restriction** when users route through `MetricOmmSimpleRouter`. Any address can trade in the restricted pool by calling the router. This breaks the core access-control invariant of the extension framework and constitutes an admin-boundary break: an unprivileged, non-allowlisted user can execute swaps that the pool admin explicitly prohibited.

---

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any user who discovers the allowlist restriction on direct pool calls will naturally try the router. No special privileges, flash loans, or unusual token behavior are required — a standard router call suffices.

---

### Recommendation

The pool should pass the **originating user identity** to extensions, not the intermediary caller. Two options:

1. **Preferred**: Add an explicit `swapper` field to `extensionData` that the router populates with `msg.sender` before calling the pool, and have the extension decode it. The extension must verify `msg.sender` (the pool) is trusted before accepting the embedded identity.

2. **Alternative**: For `addLiquidity`, change `_beforeAddLiquidity` to pass `owner` (the actual depositor) as the identity argument instead of `msg.sender`. For swaps, the router should pass the actual user address as a dedicated parameter rather than relying on `msg.sender` propagation.

---

### Proof of Concept

1. Pool admin deploys a pool and attaches `SwapAllowlistExtension`.
2. Admin calls `setAllowedToSwap(pool, alice, true)` — only Alice is permitted.
3. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
4. Router calls `pool.swap(recipient, ...)`. Inside the pool, `msg.sender = router`.
5. `_beforeSwap(router, recipient, ...)` is called; extension checks `allowedSwapper[pool][router]`.
6. If the router is allowlisted (or `allowAllSwappers[pool] == true`), Bob's swap succeeds despite not being on the allowlist.
7. Alternatively, if the router is not allowlisted, Alice's swap also fails through the router — the allowlist is broken in both directions.
