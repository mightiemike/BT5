Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives the `sender` argument from the pool, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the actual end-user. If the pool admin allowlists the router (the only way to permit allowlisted users to swap via the router), the guard is fully neutralized and any unprivileged address can swap freely against the restricted pool.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes its own `msg.sender` as `sender`:**

In `MetricOmmPool.swap`, the pool calls `_beforeSwap` with `msg.sender` as the first argument:

```solidity
_beforeSwap(
  msg.sender,   // ← router address when called via MetricOmmSimpleRouter
  recipient,
  ...
);
``` [1](#0-0) 

**`SwapAllowlistExtension.beforeSwap` checks that forwarded `sender`:**

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` = pool, `sender` = router. The check is `allowedSwapper[pool][router]`, never `allowedSwapper[pool][actualUser]`. [2](#0-1) 

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` with no originator forwarding:**

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
```

The original `msg.sender` (the end-user) is stored only in transient callback context for payment purposes; it is never placed in the `sender` slot seen by extensions. [3](#0-2) 

The same flaw applies to `exactInput` (multi-hop) and `exactOutputSingle`: [4](#0-3) [5](#0-4) 

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER` to restrict swapping to KYC'd addresses.
2. Admin calls `setAllowedToSwap(pool, address(router), true)` so that allowlisted users can swap via the standard router.
3. Any unprivileged EOA calls `router.exactInputSingle(...)`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Extension checks `allowedSwapper[pool][router]` → `true` → passes.
6. Unauthorized user receives output tokens; allowlist guard is fully bypassed.

**Existing guards are insufficient:** The `onlyPool` modifier on `BaseMetricExtension` only confirms that `msg.sender` to the extension is a registered pool — it does not help recover the actual end-user identity. [6](#0-5) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` is intended to restrict swapping to a curated set of addresses (e.g., KYC'd counterparties, whitelisted market makers). Once the router is allowlisted, the restriction is completely neutralized: any address can swap against the pool's LP reserves at oracle-derived prices. LPs who deposited under the assumption of restricted access have their capital exposed to unauthorized traders, constituting a direct loss of LP principal through adverse, unintended trades. This meets the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact criteria.

## Likelihood Explanation
Both required conditions — a pool with `SwapAllowlistExtension` and the router being allowlisted — are part of the natural production deployment pattern. There is no privileged attacker role required; any EOA can call the public router. The scenario is repeatable on every block as long as the router remains allowlisted. The `generate_scanned_questions.py` pivot target explicitly identifies this call path as a validation focus, confirming it is an intended production configuration.

## Recommendation
The pool must convey the original caller's identity to the extension. Two viable approaches:

1. **Router-forwarded identity via `extensionData`**: Define a convention where the router ABI-encodes `msg.sender` into `extensionData`, and `SwapAllowlistExtension.beforeSwap` decodes and checks that value when `sender` is a known trusted router. The extension must verify the router is trusted before accepting the encoded originator.

2. **Dedicated `originator` parameter**: Add an explicit `originator` field to the pool's `swap` signature that the router populates with its own `msg.sender`. The extension checks `originator` instead of `sender`. This is the cleanest fix and mirrors how Uniswap v4 separates `sender` from `hookData`-encoded originators.

## Proof of Concept
```solidity
// Setup: pool configured with SwapAllowlistExtension.
// Admin allowlists the router so allowlisted users can swap via router:
extension.setAllowedToSwap(address(pool), address(router), true);

// Attacker (not individually allowlisted) calls the public router:
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        recipient:        attacker,
        tokenIn:          token0,
        zeroForOne:       true,
        amountIn:         1_000e18,
        amountOutMinimum: 0,
        priceLimitX64:    0,
        deadline:         block.timestamp,
        extensionData:    ""
    })
);
// pool.swap is called with msg.sender = router.
// SwapAllowlistExtension checks allowedSwapper[pool][router] → true → passes.
// Attacker receives token1 output; allowlist guard is fully bypassed.

// Verify: assert extension.isAllowedToSwap(pool, attacker) == false
// but the swap succeeded anyway.
```

A Foundry integration test should:
1. Deploy pool with `SwapAllowlistExtension` in `BEFORE_SWAP_ORDER`.
2. Allowlist only the router (`setAllowedToSwap(pool, router, true)`).
3. Call `router.exactInputSingle` from an address that is NOT individually allowlisted.
4. Assert the call succeeds and the attacker receives output tokens, demonstrating the bypass.
