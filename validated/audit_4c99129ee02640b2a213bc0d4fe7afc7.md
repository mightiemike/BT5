### Title
Fork Difficulty Calculation Uses Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

### Summary

The `get_next_work_required` function in `dogecoin.rs` (and `bitcoin.rs`, `litecoin.rs`) retrieves the boundary block for difficulty calculation via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. When the block being validated belongs to a fork that diverged before `height_first`, the mainchain block at `height_first` is a different block than the fork's actual ancestor at that height. The difficulty calculation therefore uses the wrong timestamp, producing an incorrect expected target. For Dogecoin (per-block retarget after height 145,000), a fork of just two blocks is sufficient to trigger the discrepancy. A developer-authored `TODO` comment in the code explicitly flags this uncertainty.

### Finding Description

In `contract/src/dogecoin.rs`, `get_next_work_required` computes `height_first` and then fetches the boundary block:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

It always resolves through `mainchain_height_to_header`, which maps only canonical-chain heights. For a fork block whose chain diverged at height `D`, any `height_first > D` will resolve to the mainchain block at that height, not the fork's actual ancestor.

For Dogecoin mainnet after block 145,000, `difficulty_adjustment_interval` is effectively 1 (every block retargets):

```rust
let new_difficulty_protocol = prev_block_header.block_height >= 145_000;
let difficulty_adjustment_interval = if new_difficulty_protocol { 1 } else { ... };
``` [3](#0-2) 

With `blocks_to_go_back = 1`, `height_first = prev_block_header.block_height - 1`. For a fork block at height `H+1` whose chain diverged at height `H-1` (i.e., the fork is two blocks deep), `height_first = H-1 = D`, so the mainchain block at `H-1` is the same as the fork's ancestor — no bug. But for a fork block at height `H+2` (three blocks deep, diverged at `H-1`), `height_first = H` which is inside the diverged range: the mainchain block at `H` is `M_H`, while the fork's actual ancestor at `H` is `F_H`. The difficulty calculation uses `M_H.time` instead of `F_H.time`.

The same structural bug exists in `bitcoin.rs` and `litecoin.rs` (no TODO comment there), but with a 2016-block adjustment interval the fork would need to be implausibly long to matter. [4](#0-3) [5](#0-4) 

### Impact Explanation

The Dogecoin difficulty formula computes:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [6](#0-5) 

If an attacker sets `F_{H}.time` (the fork block at height `H`) to be significantly later than `M_{H-1}.time` (the mainchain block at `H-1`, which is what the bug uses as `first_block_time`), the computed timespan is inflated relative to the correct value `F_H.time - F_{H-1}.time`. A larger timespan produces a larger (easier) target. The clamping limits the per-block adjustment to +50% of `retarget_timespan`, but since Dogecoin retargets every block, the attacker can chain multiple fork blocks each benefiting from the inflated timespan.

The corrupted `expected_bits` is then compared against `block_header.bits`:

```rust
require!(
    expected_bits == block_header.bits,
    ...
);
``` [7](#0-6) 

If the attacker crafts `block_header.bits` to match the incorrectly computed `expected_bits`, the PoW check passes with a lower difficulty target than the fork's actual history requires. If the fork's cumulative `chain_work` then exceeds the mainchain's, `reorg_chain` is triggered, promoting the attacker's low-work fork to the canonical chain. Downstream callers of `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` would then receive `true` for transactions in the attacker's fork blocks.

### Likelihood Explanation

For Dogecoin mainnet after block 145,000, the bug is triggered by any fork three or more blocks deep — a routine occurrence during normal chain operation. The attacker needs only to:

1. Submit a fork block `F_{H-1}` with a timestamp set close to the 2-hour future-time ceiling (`MAX_FUTURE_BLOCK_TIME_LOCAL = 7200 s`), making it later than `M_{H-1}.time`.
2. Submit `F_H` building on `F_{H-1}` with a short inter-block time.
3. Submit `F_{H+1}` — the difficulty check uses `M_{H-1}.time` (mainchain) instead of `F_{H-1}.time` (fork), inflating the timespan and lowering the required target. [8](#0-7) 

No privileged role is required. `submit_blocks` is callable by any trusted relayer, and the `trusted_relayer` macro's staking requirement is the only gate — a motivated attacker can meet it.

### Recommendation

Replace the `get_header_by_height` call in `get_next_work_required` with an ancestor-walk that follows `prev_block_hash` links from the current fork block back to `height_first`, rather than resolving through the mainchain index. This mirrors how `get_median_time_past` already correctly walks ancestors via `get_prev_header`. The same fix should be applied consistently across `dogecoin.rs`, `bitcoin.rs`, and `litecoin.rs`.

### Proof of Concept

```
Mainchain: ... → [M_{H-2}] → [M_{H-1}] → [M_H] → [M_{H+1}] (tip)
                                 time=T

Fork (diverges at H-1):
              ... → [M_{H-2}] → [F_{H-1}] → [F_H] → [F_{H+1}]
                                  time=T+7000   time=T+7010

Validating F_{H+1} (height = H+1, prev = F_H):
  height_first = H
  get_header_by_height(H) → M_H  (mainchain block, time = T+600 approx)
  
  Bug timespan  = F_H.time - M_{H-1}.time = (T+7010) - T       = 7010  → clamped to max (90s)
  Correct span  = F_H.time - F_{H-1}.time = (T+7010) - (T+7000) = 10   → clamped to min (45s)

  Bug expected_bits  → target derived from 90s timespan (easier)
  Correct expected_bits → target derived from 45s timespan (harder)

Attacker crafts F_{H+1}.bits to match the bug's expected_bits (easier target),
finds a valid PoW hash meeting that easier target, and submits.
Contract accepts F_{H+1}. If chain_work(F_{H+1}) > chain_work(M_{H+1}),
reorg_chain() promotes the fork, corrupting the canonical chain.
```
