Audit Report

## Title
SwapAllowlistExtension Checks Router Address as Swapper, Allowing Any User to Bypass the Swap Allowlist via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `swap()` call. When swaps are routed through `MetricOmmSimpleRouter`, `sender` equals the router's address, not the end user. A pool admin who allowlists the router to enable router-mediated swaps inadvertently grants every user on the network the ability to bypass the allowlist, exposing LP funds to trades from actors the admin explicitly intended to exclude.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // = router address when called via MetricOmmSimpleRouter
  recipient, ...
);
```

`ExtensionCalling._beforeSwap` encodes and forwards this value unchanged to the extension:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks whether this `sender` is allowlisted per pool:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension's caller) and `sender` is the router when the swap originates from `MetricOmmSimpleRouter.exactInputSingle`:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router calls `pool.swap()` directly, so the pool sees `msg.sender = router`. The extension therefore evaluates `allowedSwapper[pool][router]`. If the pool admin allowlists the router address to permit router-mediated swaps — the natural operational step — the check passes for every user who routes through it, regardless of whether that user is individually allowlisted. The real end user's identity (`msg.sender` of the router call) is stored in transient storage for payment purposes only and is never forwarded to the pool or extension.

## Impact Explanation
A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of addresses (e.g., KYC'd counterparties, whitelisted market makers) is completely bypassed. Any unpermissioned user can execute swaps against the pool's liquidity by calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`. LP funds are exposed to trades from actors the pool admin explicitly intended to exclude. This constitutes a broken core pool invariant (allowlist policy) with direct fund-impact consequences for LPs — meeting the "Broken core pool functionality causing loss of funds" allowed impact criterion at High severity.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical public swap entrypoint. The pool admin must allowlist the router to enable any router-mediated swaps; without doing so, no user can swap through the router at all. This makes the bypass condition the default live state for any allowlisted pool that supports router access. Any user who discovers the allowlist can trivially route through the router with no special privileges or capital requirements.

## Recommendation
The extension must gate on the original end user, not the immediate caller of `pool.swap()`. The cleanest fix is for `MetricOmmSimpleRouter` to encode the real `msg.sender` in `extensionData` and for `SwapAllowlistExtension.beforeSwap` to decode and check it when `sender` is a known router. Alternatively, the pool admin should never allowlist the router address and should only allowlist individual EOAs directly calling `pool.swap()` — but this must be explicitly documented and enforced, as the current design provides no warning. A protocol-level convention passing the original initiator as a separate field through the pool's `swap()` interface would be the most robust fix.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps.
3. Unpermissioned user Bob (not individually allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router]` → `true`.
7. Bob's swap executes against the pool's LP funds despite Bob never being allowlisted.
