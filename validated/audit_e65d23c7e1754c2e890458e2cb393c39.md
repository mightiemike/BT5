Audit Report

## Title
Unbounded `newTimelock` in `proposeOracleStopLossTimelock` Causes uint32 Wrap, Bypassing the Stop-Loss Timelock Guard - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

## Summary

`proposeOracleStopLossTimelock` accepts any `uint32 newTimelock` without an upper-bound check. Setting it to `type(uint32).max` causes `_afterTimelock` to silently wrap the `uint256` sum back to a timestamp in the past, making every subsequent proposal immediately executable. The pool admin can then atomically propose and execute `drawdownE6 = E6`, collapsing `floorMultiplier` to zero and permanently disabling the stop-loss guard for all LP positions in the pool.

## Finding Description

`proposeOracleStopLossDrawdown` and `proposeOracleStopLossDecay` both call `_validateDrawdown` / `_validateDecay` before storing their pending values, but `proposeOracleStopLossTimelock` stores `newTimelock` with no validation: [1](#0-0) 

`_afterTimelock` computes the deadline as a `uint256` addition then truncates to `uint32`: [2](#0-1) 

With `block.timestamp ≈ 1_753_000_000` (July 2026) and `timelock = type(uint32).max = 4_294_967_295`:

```
uint32(1_753_000_000 + 4_294_967_295)
= uint32(6_047_967_295)
= 6_047_967_295 % 4_294_967_296
= 1_752_999_999   ← one second in the past
```

`_requireElapsed` then passes immediately for every subsequent proposal: [3](#0-2) 

The same absence of validation exists in `initialize`, where `timelock` is decoded and stored without any check: [4](#0-3) 

By contrast, `_validateDrawdown` and `_validateDecay` both enforce hard ceilings, but no equivalent `_validateTimelock` exists: [5](#0-4) 

**Exploit flow:**

1. Pool is initialized with `timelock = 0` (permitted by the constructor, confirmed in tests).
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. `_afterTimelock` returns `uint32(block.timestamp + 0) = block.timestamp`. `executeOracleStopLossTimelock` passes `_requireElapsed` immediately (same block). `oracleStopLossConfig[pool].timelock` is now `type(uint32).max`.
3. Admin calls `proposeOracleStopLossDrawdown(pool, 1e6)`. `_validateDrawdown(1e6)` passes (`1e6 <= E6`). `_afterTimelock` returns the wrapped past timestamp. `executeOracleStopLossDrawdown` passes `_requireElapsed` immediately. `drawdownE6` is now `1e6`.
4. In `_afterSwapOracleStopLoss`, `floorMultiplier = E6 - drawdown = 1e6 - 1e6 = 0`. [6](#0-5) 

5. Inside `_applyWatermark`, the breach condition becomes `metric < (hwm * 0) / E6`, which is always false: [7](#0-6) 

The stop-loss guard is permanently silenced. Note: the early-exit guard `if (drawdown == 0) return` at line 217 does **not** trigger because `drawdownE6 = 1e6 ≠ 0`. [8](#0-7) 

## Impact Explanation

With `floorMultiplier = 0`, `afterSwap` never reverts regardless of how far the per-share bin value drops. Any oracle-price manipulation or value extraction through swaps will not be blocked, exposing all LP principal in every bin the extension monitors. This is a direct loss of LP principal and constitutes an admin-boundary break: the pool admin has bypassed the only on-chain constraint (the timelock) that bounds their ability to make instant LP-adverse parameter changes.

## Likelihood Explanation

The attack requires only the pool admin role. The initial timelock can legitimately be zero (as shown in the test helper), making the entire attack executable atomically in a single block with no prior waiting period. Even if the initial timelock is non-zero, the admin need only wait out that one period to set `type(uint32).max`, after which all future proposals bypass the timelock permanently. The path is fully on-chain, requires no off-chain coordination, and is repeatable across any pool using this extension.

## Recommendation

Add a `_validateTimelock` function mirroring the existing validators and call it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days;

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply it in `initialize` alongside the existing validators: [9](#0-8) 

And at the top of `proposeOracleStopLossTimelock` before any storage write: [1](#0-0) 

## Proof of Concept

```solidity
// Pool initialized with timelock = 0
extension.initialize(pool, abi.encode(uint32(50_000), uint32(58), uint32(0)));

// Step 1: Set timelock to type(uint32).max — no validation, passes silently
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
// executeAfter = uint32(block.timestamp + 0) = block.timestamp → passes immediately
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == type(uint32).max

// Step 2: Propose drawdown = E6 (100%)
// _afterTimelock returns uint32(block.timestamp + type(uint32).max) = past timestamp
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 1e6);
// executeAfter is in the past → _requireElapsed passes immediately
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 == 1e6 → floorMultiplier == 0 → stop-loss permanently disabled

// Step 3: Verify afterSwap no longer reverts even on 100% value loss
// (breach condition is always false; LP principal fully exposed)
```
