### Title
Protobuf `ValidResourceBounds` Deserialization Uses Value-Based Heuristic That Misclassifies `AllResources` as `L1Gas`, Producing a Different Transaction Hash After Round-Trip — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` determines the variant (`L1Gas` vs `AllResources`) by inspecting the **runtime values** of the decoded fields rather than preserving the original variant. Because `get_tip_resource_bounds_hash` hashes a different number of resource-bound elements depending on the variant, a transaction whose `resource_bounds` is `AllResources` with `l2_gas = 0` and `l1_data_gas = 0` will produce one hash before protobuf serialization and a **different hash** after deserialization. Any component that recomputes the hash from the deserialized object (e.g., P2P state-sync hash validation) will see a mismatch, causing valid transactions to be rejected or stored under the wrong hash.

---

### Finding Description

**Step 1 – The value-based heuristic in the protobuf deserializer.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` decides the variant purely from the decoded field values:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

There is no structural marker in the protobuf wire format that distinguishes `L1Gas` from `AllResources`; the decision is made entirely by the zero-check. The serializer for `L1Gas` emits `l2_gas = 0` and `l1_data_gas = 0`, which is byte-for-byte identical to an `AllResources` transaction whose user-supplied bounds happen to be zero for those two fields. [2](#0-1) 

**Step 2 – The hash function produces a different number of elements per variant.**

`get_tip_resource_bounds_hash` always emits `[L1_GAS, L2_GAS]` for both variants, but appends `L1_DATA_GAS` **only** for `AllResources`:

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
``` [3](#0-2) 

- `AllResources` with `l2_gas = 0`, `l1_data_gas = 0` → hash over **3** elements: `[tip, L1_GAS_concat, L2_GAS_concat(=0), L1_DATA_GAS_concat(=0)]`
- `L1Gas` with the same l1_gas → hash over **2** elements: `[tip, L1_GAS_concat, L2_GAS_concat(=0)]`

These are distinct Poseidon digests even though all numeric values are identical.

**Step 3 – The gateway always creates `AllResources` for new transactions.**

`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both carry `AllResourceBounds` (never `ValidResourceBounds`). Their `InvokeTransactionV3Trait::resource_bounds()` unconditionally returns `ValidResourceBounds::AllResources(...)`:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [4](#0-3) 

The hash is therefore computed with the `AllResources` path (3 elements). The transaction is stored in the DB as `InvokeTransactionV3` with `ValidResourceBounds::AllResources(...)`. [5](#0-4) 

**Step 4 – The P2P state-sync path uses the broken `ValidResourceBounds` converter.**

When a block is transmitted via P2P, `InvokeTransactionV3` is serialized to protobuf using `From<ValidResourceBounds> for protobuf::ResourceBounds`. The receiving node deserializes it with `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. For a transaction where `l2_gas = 0` and `l1_data_gas = 0`, the receiving node reconstructs `L1Gas` instead of `AllResources`. Any subsequent call to `calculate_transaction_hash` on the deserialized object produces a hash that does not match the hash stored in the block. [6](#0-5)

---

### Impact Explanation

A transaction submitted via RPC with `AllResourceBounds` where `l2_gas = 0` and `l1_data_gas = 0` (a configuration the gateway explicitly accepts, as shown by the `valid_l1_gas` test case) is assigned a hash computed over 3 resource-bound elements. After protobuf round-trip through the P2P state-sync path, the same transaction is reconstructed with the `L1Gas` variant and its hash recomputed over 2 elements. The two hashes diverge. Any node that validates transaction hashes after P2P deserialization will reject the transaction as having an invalid hash, causing state-sync failures for blocks containing such transactions. This matches the **High** impact: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

---

### Likelihood Explanation

The trigger is an unprivileged user submitting a V3 invoke transaction with only `l1_gas` non-zero (l2_gas = 0, l1_data_gas = 0). The gateway's stateless validator accepts this configuration. The bug is latent in every P2P state-sync round-trip for any such transaction. No special permissions or malicious peers are required; a normal user transaction is sufficient.

---

### Recommendation

Replace the value-based heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit structural discriminant. The simplest fix is to add a boolean field (e.g., `is_all_resources`) to the protobuf `ResourceBounds` message and set it during serialization. Alternatively, use two separate protobuf message types for `L1Gas` and `AllResources`. The deserializer must never infer the variant from the numeric values of the bounds.

---

### Proof of Concept

1. Submit an `RpcInvokeTransactionV3` with `resource_bounds = AllResourceBounds { l1_gas: {max_amount: 100, max_price_per_unit: 1}, l2_gas: {0, 0}, l1_data_gas: {0, 0} }`.
2. The gateway computes `hash_A = poseidon(INVOKE, version, sender, tip_resource_bounds_hash_3_elements, ...)` where `tip_resource_bounds_hash_3_elements` covers `[tip, L1_GAS_concat, L2_GAS_concat(=0), L1_DATA_GAS_concat(=0)]`.
3. The transaction is included in a block and stored in the DB with `ValidResourceBounds::AllResources(...)`.
4. The block is transmitted via P2P state sync. The receiving node deserializes the transaction: `l1_data_gas.is_zero() && l2_gas.is_zero()` → `ValidResourceBounds::L1Gas(l1_gas)`.
5. The receiving node computes `hash_B = poseidon(INVOKE, version, sender, tip_resource_bounds_hash_2_elements, ...)` where `tip_resource_bounds_hash_2_elements` covers only `[tip, L1_GAS_concat, L2_GAS_concat(=0)]`.
6. `hash_A ≠ hash_B`. `validate_transaction_hash` returns `false`. The transaction (and its containing block) is rejected by the receiving node. [8](#0-7) [9](#0-8)

// An implementation of the SNIP: https://github.com/EvyatarO/SNIPs/blob/snip-8/SNIPS/snip-8.md
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

pub(crate) fn get_invoke_transaction_v3_hash<T: InvokeTransactionV3Trait>(
    transaction: &T,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    let tip_resource_bounds_hash =
        get_tip_resource_bounds_hash(&transaction.resource_bounds(), transaction.tip())?;
    let paymaster_data_hash =
        HashChain::new().chain_iter(transaction.paymaster_data().0.iter()).get_poseidon_hash();
    let data_availability_mode = concat_data_availability_mode(
        transaction.nonce_data_availability_mode(),
        transaction.fee_data_availability_mode(),
    );
    let account_deployment_data_hash = HashChain::new()
        .chain_iter(transaction.account_deployment_data().0.iter())
        .get_poseidon_hash();
    let calldata_hash =
        HashChain::new().chain_iter(transaction.calldata().0.iter()).get_poseidon_hash();
    let mut hash_chain = HashChain::new()
        .chain(&INVOKE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address().0.key())
        .chain(&tip_resource_bounds_hash)
        .chain(&paymaster_data_hash)
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce().0)
        .chain(&data_availability_mode)
        .chain(&account_deployment_data_hash)
        .chain(&calldata_hash);
    if !transaction.proof_facts().0.is_empty() {
        let proof_facts_hash =
            HashChain::new().chain_iter(transaction.proof_facts().0.iter()).get_poseidon_hash();
        hash_chain = hash_chain.chain(&proof_facts_hash);
    }
    Ok(TransactionHash(hash_chain.get_poseidon_hash()))
