### Title
`derive_near_deterministic_account_id` omits `chain_id` from the hash domain, enabling cross-network account-ID collision - (File: `core/primitives/src/utils.rs`)

### Summary

`derive_near_deterministic_account_id` computes a deterministic account ID as `"0s" + keccak256(borsh(DeterministicAccountStateInit))[12:32]`. The `DeterministicAccountStateInit` struct contains only `code: GlobalContractIdentifier` and `data: BTreeMap<Vec<u8>, Vec<u8>>` — no `chain_id`, no genesis hash, no network tag. The identical Borsh bytes therefore produce the identical `0s…` account ID on mainnet, testnet, and any future fork of NEAR, exactly mirroring the missing `chain.id` in `deployLPToken`.

### Finding Description

`derive_near_deterministic_account_id` in `core/primitives/src/utils.rs`:

```rust
pub fn derive_near_deterministic_account_id(
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
``` [1](#0-0) 

The only inputs to the hash are the fields of `DeterministicAccountStateInitV1`:

```rust
pub struct DeterministicAccountStateInitV1 {
    pub code: GlobalContractIdentifier,
    pub data: BTreeMap<Vec<u8>, Vec<u8>>,
}
``` [2](#0-1) 

No `chain_id`, genesis hash, or network tag is mixed into the preimage. The validation gate in `validate_deterministic_state_init` only checks that `derive_near_deterministic_account_id(&action.state_init) == receiver_id`; it has no awareness of which network is executing the check. [3](#0-2) 

NEAR already exposes `chain_id` as a first-class runtime value (the `chain_id` host function, `GenesisId.chain_id`, etc.), so the omission is not a design constraint — it is a gap in the hash domain. [4](#0-3) 

### Impact Explanation

**Pre-funding / front-run window (single network):** A user who sends a `Transfer` to `0s<hash>` before submitting `DeterministicStateInitAction` creates an uninitialized account holding funds. Any party who knows the `state_init` can race to submit `DeterministicStateInitAction` and become the first to initialize the account. Because the account ID is fully determined by the `state_init` hash (no chain-specific salt), the same race exists on every network that shares the same `state_init`.

**Cross-network / fork scenario (direct analog to M-14):** NEAR runs on mainnet and testnet today; the same `state_init` yields the same `0s…` account ID on both. If NEAR forks (as Ethereum did), every deterministic account ID is identical on both forks. Funds pre-sent to `0s<hash>` on one fork are claimable by whoever initializes the account on the other fork first. When `GlobalContractIdentifier::CodeHash` is used, the deployed code is also identical on both forks, so the contract logic provides no additional protection. When `GlobalContractIdentifier::AccountId` is used, the code may differ across networks, potentially giving an attacker a different (attacker-controlled) code path at the same address.

**Scope:** `DeterministicStateInitAction` is a live protocol feature introduced in v2.10.0 and callable by any unprivileged user. [5](#0-4) 

### Likelihood Explanation

NEAR mainnet and testnet already constitute two live networks where the collision is present today. Any user who pre-funds a `0s…` address on one network and then submits `DeterministicStateInitAction` on the other (or who is front-run on the same network) is exposed. A hard fork — historically not unprecedented for major blockchains — would immediately activate the full Wintermute-style scenario for all existing deterministic accounts.

### Recommendation

Include the network's `chain_id` (or the full `GenesisId`, which also carries the genesis block hash) in the Borsh preimage before hashing:

```rust
pub fn derive_near_deterministic_account_id(
    chain_id: &str,          // e.g. "mainnet", "testnet"
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    // domain-separate by chain_id length-prefix then state_init bytes
    let mut hasher = sha3::Keccak256::new();
    hasher.update((chain_id.len() as u32).to_le_bytes());
    hasher.update(chain_id.as_bytes());
    hasher.update(borsh::to_vec(state_init).expect("borsh must not fail"));
    let hash = hasher.finalize();
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
```

Because the account ID is part of the protocol (it is validated on-chain by `validate_deterministic_state_init`), this change requires a protocol version gate. Existing deterministic accounts created before the gate would retain their old IDs; new accounts created after the gate would use chain-scoped IDs. [6](#0-5) 

### Proof of Concept

```
// Same state_init on mainnet and testnet:
let state_init = DeterministicAccountStateInit::V1(DeterministicAccountStateInitV1 {
    code: GlobalContractIdentifier::CodeHash(some_hash),
    data: BTreeMap::from([(b"owner".to_vec(), b"alice.near".to_vec())]),
});

// On mainnet:
let id_mainnet = derive_near_deterministic_account_id(&state_init);
// On testnet (or any fork):
let id_testnet = derive_near_deterministic_account_id(&state_init);

assert_eq!(id_mainnet, id_testnet);  // PASSES — identical 0s… address on both networks
``` [1](#0-0) 

An attacker who observes a `Transfer` to `0s<hash>` on mainnet (pre-funding before `DeterministicStateInitAction`) can immediately submit `DeterministicStateInitAction` with the same `state_init` on mainnet to race the legitimate initializer, or — after a fork — initialize the account on the fork chain and control the funds deposited there. The `validate_deterministic_state_init` check passes on every network because it only verifies `derive_near_deterministic_account_id(&action.state_init) == receiver_id`, with no chain-scoped binding.
