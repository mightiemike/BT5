Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, enabling allowlist bypass and breaking the core swap flow - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates on `sender`, which is `msg.sender` of the pool's `swap()` call. When `MetricOmmSimpleRouter` intermediates a swap, `msg.sender` at the pool level is the router contract address, not the end user. The extension therefore evaluates `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, producing two fund-impacting failure modes: a complete allowlist bypass when the router is allowlisted, and permanent denial of router-mediated swaps for individually allowlisted users.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap()` forwards this value unchanged to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks this value against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

Inside the pool, `msg.sender` is the router contract address. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

**Failure Mode A — bypass:** A pool admin who calls `setAllowedToSwap(pool, router, true)` (a natural configuration to enable all router-mediated swaps) inadvertently opens the pool to every address. Any unprivileged user can call `MetricOmmSimpleRouter` and the extension passes because `allowedSwapper[pool][router] == true`.

**Failure Mode B — broken core flow:** A pool admin calls `setAllowedToSwap(pool, user1, true)`. `user1` calls `MetricOmmSimpleRouter.exactInputSingle()`. The router calls `pool.swap()`; `msg.sender` inside the pool is the router. The extension checks `allowedSwapper[pool][router] == false` and reverts `NotAllowedToSwap`. The allowlisted user is permanently blocked from using the router.

`DepositAllowlistExtension` does not share this flaw because it checks `owner` (the second argument to `beforeAddLiquidity`), which is the user-supplied position recipient and is not overwritten by any router intermediary. The swap path has no equivalent user-identity forwarding mechanism.

## Impact Explanation

**Failure Mode A** is a complete allowlist bypass on a curated pool. Any unprivileged user can trade on a pool the admin intended to restrict to vetted counterparties, potentially extracting value at oracle-anchored prices reserved for specific actors. This satisfies the "Admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path" criterion.

**Failure Mode B** renders the core swap flow permanently unusable for allowlisted users. `MetricOmmSimpleRouter` is the only supported periphery swap entry point; users who cannot implement `IMetricOmmSwapCallback` themselves are locked out entirely despite being explicitly allowlisted. This satisfies "Broken core pool functionality causing loss of funds or unusable swap flows."

## Likelihood Explanation

Any pool that deploys `SwapAllowlistExtension` and expects users to swap through `MetricOmmSimpleRouter` immediately hits Failure Mode B — no special privileges, flash loans, or exotic token behavior are required. Failure Mode A is reachable whenever an admin allowlists the router address, which is a natural and expected configuration step. Both modes are triggered by normal, unprivileged user interactions with the primary periphery entry point.

## Recommendation

The extension must check the original user's address, not the intermediary router's address. Two viable approaches:

1. **Pass the real user through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. Requires coordinated changes in the router and extension.

2. **Check `recipient` instead of `sender`**: The `beforeSwap` hook already receives `recipient` as its second argument. Gating on `recipient` is router-transparent because the router passes the user-supplied recipient directly (`params.recipient`). This is the lower-complexity fix.

## Proof of Concept

**Failure Mode A (bypass):**
```
1. Deploy pool with SwapAllowlistExtension on beforeSwap.
2. Pool admin calls setAllowedToSwap(pool, router, true).
3. Attacker (not in allowedSwapper[pool][attacker]) calls
   MetricOmmSimpleRouter.exactInputSingle(...) targeting the pool.
4. Router calls pool.swap(recipient, ...); msg.sender = router.
5. _beforeSwap(router, ...) → allowedSwapper[pool][router] == true → passes.
6. Attacker's swap executes on the curated pool despite never being allowlisted.
```

**Failure Mode B (broken flow):**
```
1. Pool admin calls setAllowedToSwap(pool, user1, true).
2. user1 calls MetricOmmSimpleRouter.exactInputSingle(...).
3. Router calls pool.swap(...); msg.sender = router.
4. _beforeSwap(router, ...) → allowedSwapper[pool][router] == false → reverts NotAllowedToSwap.
5. user1 is permanently blocked from using the router despite being allowlisted.
```
