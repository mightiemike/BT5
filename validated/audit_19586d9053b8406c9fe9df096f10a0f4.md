### Title
`paid_fee_on_l1` Silently Hardcoded to `Fee(1)` in Consensus-to-Internal L1Handler Conversion, Binding Wrong Executable Payload on Validator Nodes — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

When a validator node receives a `ConsensusTransaction::L1Handler` from the proposer over the P2P consensus channel, `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally substitutes `paid_fee_on_l1 = Fee(1)` for the actual L1-sourced fee value. The consensus protobuf message (`L1HandlerV0`) carries no `paid_fee_on_l1` field, so the field is never transmitted, and every validator silently reconstructs the executable transaction with a fabricated constant. The proposer, which sourced the transaction from the L1 events provider, holds the real value; validators do not. This is an asymmetric conversion: the `tx_hash` field is re-derived canonically from the transaction body, but `paid_fee_on_l1` — the other security-relevant field of the executable — is not.

---

### Finding Description

**Root cause — hardcoded constant in conversion:**

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

This function is called unconditionally for every `ConsensusTransaction::L1Handler` received during proposal validation:

```rust
// lines 197-199
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
``` [2](#0-1) 

**Why the field is absent from the wire format:**

The consensus protobuf `L1HandlerV0` message carries only `nonce`, `address`, `entry_point_selector`, and `calldata`. There is no `paid_fee_on_l1` field:

```protobuf
// crates/apollo_protobuf/src/proto/p2p/proto/consensus/consensus.proto
message ConsensusTransaction {
    oneof txn {
        ...
        L1HandlerV0 l1_handler = 4;
    }
    Hash transaction_hash = 5;
}
``` [3](#0-2) 

The Rust deserializer confirms the field is absent:

```rust
// crates/apollo_protobuf/src/converters/transaction.rs  lines 964-984
impl TryFrom<protobuf::L1HandlerV0> for L1HandlerTransaction {
    fn try_from(value: protobuf::L1HandlerV0) -> Result<Self, Self::Error> {
        // nonce, address, entry_point_selector, calldata only — no paid_fee_on_l1
        Ok(Self { version, nonce, contract_address, entry_point_selector, calldata })
    }
}
``` [4](#0-3) 

**Contrast with RPC transaction conversion — the asymmetry:**

For every `ConsensusTransaction::RpcTransaction`, the converter re-derives `tx_hash` from the full transaction body using the node's own `chain_id`, providing a canonical binding:

```rust
// lines 391-392
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
``` [5](#0-4) 

No equivalent re-derivation or validation exists for `paid_fee_on_l1` on the L1Handler path. The field is simply invented.

**Downstream propagation — central blob:**

The `paid_fee_on_l1` field is serialized verbatim into the `CentralL1HandlerTransaction` blob that is written to the central system for proof generation:

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs  lines 383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            hash_value: tx.tx_hash,
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // Fee(1) on every validator node
        }
    }
}
``` [6](#0-5) 

Every validator node that writes the previous-height blob will emit `paid_fee_on_l1 = 0x1` regardless of the actual L1 fee, diverging from the proposer's blob which carries the real value.

**Fee-check bypass on validator nodes:**

The blockifier's L1 handler execution enforces only `paid_fee != Fee(0)`:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs  lines 103-113
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [7](#0-6) 

Because validators always supply `Fee(1)`, this guard is permanently bypassed on every validator node. If the actual `paid_fee_on_l1` sourced from L1 were `Fee(0)` (e.g., due to an L1-side edge case or a future protocol change), the proposer would reject the transaction while validators would accept it, producing a consensus split.

---

### Impact Explanation

Validator nodes bind the wrong executable payload for every `L1HandlerTransaction` processed via consensus: the `paid_fee_on_l1` field is `Fee(1)` instead of the actual L1-sourced value. This matches the **High** impact criterion: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

Concrete consequences:
1. **Wrong central blob on validator nodes.** Every validator that writes the previous-height blob emits `paid_fee_on_l1 = 0x1` for all L1 handler transactions, diverging from the proposer's authoritative blob. If the central/proving system uses this field for fee verification or proof input, validator-written blobs are incorrect.
2. **Fee-check bypass.** The `paid_fee == Fee(0)` guard is permanently neutralized on validators, meaning any L1 handler transaction — including one with a zero actual fee — passes execution on validators even if the proposer would reject it.
3. **Proposer/validator divergence.** The proposer's `InternalConsensusTransaction::L1Handler` carries the real `paid_fee_on_l1`; every validator's copy carries `Fee(1)`. These are structurally different executable objects representing the same consensus transaction.

---

### Likelihood Explanation

The trigger is unconditional: **every** `L1HandlerTransaction` received by a validator via the consensus stream is affected. No special conditions are required. The fee-check bypass consequence requires `paid_fee_on_l1 = Fee(0)` on L1 to produce a visible consensus split, which is currently prevented by the L1 contract. The wrong-blob consequence is active on every block containing an L1 handler transaction.

---

### Recommendation

1. **Add `paid_fee_on_l1` to the `ConsensusTransaction` / `L1HandlerV0` protobuf message** so the proposer transmits the actual value and validators can use it directly.
2. **Remove the hardcoded `Fee(1)` constant** in `convert_consensus_l1_handler_to_internal_l1_handler` and replace it with the deserialized field value.
3. **Add a canonicalization check** analogous to the `tx_hash` re-derivation for RPC transactions: after deserializing, assert that the received `paid_fee_on_l1` is non-zero before constructing the executable transaction.

---

### Proof of Concept

The divergence is directly observable by comparing the two conversion paths in the same `TransactionConverter`:

```rust
// RPC path — tx_hash is re-derived canonically from body + chain_id
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
// → tx_hash is always correct regardless of what the proposer sent

// L1Handler path — paid_fee_on_l1 is invented, not derived
Ok(executable_transaction::L1HandlerTransaction::create(
    tx,
    &self.chain_id,
    Fee(1),   // ← always Fee(1), never the actual L1 value
)?)
```

A test demonstrating the divergence:

```rust
#[tokio::test]
async fn test_l1_handler_paid_fee_is_wrong_on_validator() {
    // Proposer creates L1Handler with actual paid_fee_on_l1 = Fee(1000)
    let raw_tx = L1HandlerTransaction { /* ... */ };
    let proposer_internal = L1HandlerTransaction::create(raw_tx.clone(), &chain_id, Fee(1000)).unwrap();
    assert_eq!(proposer_internal.paid_fee_on_l1, Fee(1000));

    // Validator receives via consensus (no paid_fee_on_l1 in protobuf)
    let consensus_tx = ConsensusTransaction::L1Handler(raw_tx);
    let (validator_internal, _) = converter
        .convert_consensus_tx_to_internal_consensus_tx(consensus_tx)
        .await
        .unwrap();
    let InternalConsensusTransaction::L1Handler(l1_tx) = validator_internal else { panic!() };

    // Divergence: validator has Fee(1), proposer had Fee(1000)
    assert_eq!(l1_tx.paid_fee_on_l1, Fee(1));   // always Fee(1)
    assert_ne!(l1_tx.paid_fee_on_l1, Fee(1000)); // not the actual value
}
```

    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::L1HandlerV0) -> Result<Self, Self::Error> {
        let version = L1HandlerTransaction::VERSION;

        let nonce = Nonce(value.nonce.ok_or(missing("L1HandlerV0::nonce"))?.try_into()?);

        let contract_address = value.address.ok_or(missing("L1HandlerV0::address"))?.try_into()?;

        let entry_point_selector = EntryPointSelector(entry_point_selector_felt);

        let calldata =
            value.calldata.into_iter().map(Felt::try_from).collect::<Result<Vec<_>, _>>()?;

        let calldata = Calldata(calldata.into());

        Ok(Self { version, nonce, contract_address, entry_point_selector, calldata })
    }
