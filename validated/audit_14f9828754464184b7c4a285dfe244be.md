### Title
`get_split_parent_shard_uids` Constructs Parent-Shard `ShardUId`s with the Wrong Layout Version — (`core/primitives/src/shard_layout/mod.rs`)

### Summary

`ShardLayout::get_split_parent_shard_uids` is documented to return the `ShardUId`s of shards from the **previous** layout that were split. It constructs each `ShardUId` using `self.version()` — the version of the **new** (current) layout — instead of the version of the **previous** layout. Because `ShardUId` is the 8-byte DB key prefix for all trie state in `DBCol::State`, this produces UIDs that do not match any real DB entries, causing callers in garbage collection and cold-storage archival to silently operate on the wrong key space.

### Finding Description

`ShardUId` encodes `(version: ShardVersion, shard_id: u32)` as an 8-byte little-endian pair and is used as the prefix for every trie node stored in the database. [1](#0-0) 

The function in question:

```rust
pub fn get_split_parent_shard_uids(&self) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(self.version(), shard_id))
        .collect()
}
``` [2](#0-1) 

`self` is the **new** shard layout. The parent shards belong to the **previous** layout. Their canonical `ShardUId` is `{version: prev_layout.version(), shard_id}`. The function instead uses `self.version()` (the new layout's version), producing `{version: new_layout.version(), shard_id}`.

For mainnet's resharding, the epoch config at protocol version 69 shows `ShardLayoutV1` with `"version": 3`. [3](#0-2) 

The previous layout had a lower version number. Any parent shard's trie state is stored in the DB under the **old** version prefix. The function returns UIDs with the **new** version prefix, which matches no DB entry.

The three production callers confirmed by grep are:
- `chain/chain/src/garbage_collection.rs`
- `core/store/src/archive/cold_storage.rs`
- `chain/epoch-manager/src/adapter.rs`

### Impact Explanation

**Garbage collection** uses these UIDs to identify and delete parent-shard state after resharding. With wrong UIDs, GC silently skips the actual parent-shard DB key range, leaving orphaned trie nodes permanently in `DBCol::State` — unbounded storage bloat proportional to the parent shard's state size.

**Cold-storage archival** uses these UIDs to copy parent-shard state to cold storage. With wrong UIDs, archival nodes silently miss copying the parent shard's historical trie nodes, causing silent data loss for archive nodes that rely on cold storage for pre-resharding history.

### Likelihood Explanation

Resharding has already occurred on mainnet (the `ShardLayoutV1` version field increments with each resharding event). Both GC and cold-storage archival run automatically after every resharding epoch boundary. The bug is therefore already triggered on every node that has processed a resharding epoch.

### Recommendation

The function must use the **previous** layout's version, not `self.version()`. The simplest fix is to accept the previous `ShardLayout` as a parameter:

```rust
pub fn get_split_parent_shard_uids(&self, prev_layout: &ShardLayout) -> BTreeSet<ShardUId> {
    let parent_shard_ids = self.get_split_parent_shard_ids();
    parent_shard_ids
        .into_iter()
        .map(|shard_id| ShardUId::new(prev_layout.version(), shard_id))
        .collect()
}
```

All three call sites must be updated to supply the previous layout.

### Proof of Concept

```rust
// Mainnet: old layout V1 version 2, new layout V1 version 3
// Parent shard (e.g. shard_id=2) has DB keys prefixed with {version:2, shard_id:2}
let new_layout: ShardLayout = /* V1 version 3 */;
let parent_uids = new_layout.get_split_parent_shard_uids();
// Returns: {ShardUId { version: 3, shard_id: 2 }}  ← WRONG
// Expected: {ShardUId { version: 2, shard_id: 2 }}  ← correct DB prefix

// GC calls delete_range(DBCol::State, wrong_uid.to_bytes(), ...)
// → no DB entries match → parent shard state never deleted → storage bloat
// Cold storage calls get(wrong_uid, hash) → MissingTrieValue → silent skip

pub struct ShardUId {
    pub version: ShardVersion,
    pub shard_id: u32,
}

impl ShardUId {
    pub fn new(version: ShardVersion, shard_id: ShardId) -> Self {
        Self { version, shard_id: shard_id.into() }
    }

    /// Returns the only shard uid in the ShardLayout::single_shard layout.
    /// It is not suitable for use with any other shard layout.
    #[cfg(feature = "test_utils")]
    pub fn single_shard() -> Self {
        ShardLayout::single_shard().shard_uids().next().unwrap()
    }

    /// Byte representation of the shard uid
    pub fn to_bytes(&self) -> [u8; 8] {
        let mut res = [0; 8];
        res[0..4].copy_from_slice(&u32::to_le_bytes(self.version));
        res[4..].copy_from_slice(&u32::to_le_bytes(self.shard_id));
        res
    }
