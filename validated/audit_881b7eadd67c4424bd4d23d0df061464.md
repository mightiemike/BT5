### Title
`get_tip_resource_bounds_hash` omits `L1_DATA_GAS` from hash preimage for V3 transactions with zero `l2_gas`/`l1_data_gas` in RPC execution path — (`crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion in the RPC layer silently downgrades a V3 transaction whose `l2_gas` and `l1_data_gas` are both zero to the `ValidResourceBounds::L1Gas` variant. This causes `get_tip_resource_bounds_hash` to hash only two resource elements (L1\_GAS, L2\_GAS) instead of three (L1\_GAS, L2\_GAS, L1\_DATA\_GAS). The gateway path, which always constructs `AllResourceBounds` → `ValidResourceBounds::AllResources`, always hashes three elements. The same transaction data therefore produces two different `tip_resource_bounds_hash` values depending on which path processes it, yielding two different `TransactionHash` values.

---

### Finding Description

**Analog to the external bug.** In `MlumStaking::addToPosition`, the multiplier was assigned from the *initial* lock duration instead of the *new* computed lock duration, creating a stale-value inconsistency. Here, the resource-bounds *variant* (which controls the hash preimage length) is derived from the *value* of the fields rather than from the canonical type of the transaction, creating the same stale-value inconsistency across two code paths that must agree on the hash.

**Gateway path (always `AllResources`).**
`RpcInvokeTransactionV3` and `InternalRpcInvokeTransactionV3` both store `resource_bounds: AllResourceBounds`. Their `InvokeTransactionV3Trait::resource_bounds()` implementation unconditionally wraps the value in `ValidResourceBounds::AllResources`:

```rust
// crates/starknet_api/src/rpc_transaction.rs  line 637-639
impl InvokeTransactionV3Trait for InternalRpcInvokeTransactionV3 {
    fn resource_bounds(&self) -> ValidResourceBounds {
        ValidResourceBounds::AllResources(self.resource_bounds)
    }
``` [1](#0-0) 

`get_tip_resource_bounds_hash` therefore always enters the `AllResources` branch and appends the `L1_DATA_GAS` element to the hash chain:

```rust
// crates/starknet_api/src/transaction_hash.rs  lines 203-208
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [2](#0-1) 

**RPC execution path (can produce `L1Gas`).**
The RPC layer's `ResourceBoundsMapping` struct carries all three fields as plain values. Its conversion to `ValidResourceBounds` uses the *value* of the fields to decide the variant:

```rust
// crates/apollo_rpc/src/v0_8/transaction.rs  lines 188-199
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← L1_DATA_GAS dropped from hash
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
``` [3](#0-2) 

When a V3 transaction carries `l1_data_gas = 0` and `l2_gas = 0` (but non-zero `l1_gas`), this conversion produces `ValidResourceBounds::L1Gas`. `get_tip_resource_bounds_hash` then enters the `L1Gas` branch and emits only two elements:

```
H_rpc  = poseidon(tip, packed(L1_GAS), packed(L2_GAS=0))          // 2 elements
H_gw   = poseidon(tip, packed(L1_GAS), packed(L2_GAS=0), packed(L1_DATA_GAS=0))  // 3 elements
H_rpc ≠ H_gw
```

The `TransactionHash` stored in `InternalRpcTransaction.tx_hash` (computed at the gateway) therefore differs from the hash computed by `calc_tx_hashes` in the RPC execution layer for the same transaction data. [4](#0-3) 

The gateway's stateless validator accepts transactions where at least one resource bound is non-zero, so a transaction with `l1_gas > 0`, `l2_gas = 0`, `l1_data_gas = 0` passes admission:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs  lines 64-68
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(...ZeroResourceBounds...);
}
``` [5](#0-4) 

---

### Impact Explanation

`starknet_simulateTransactions` and `starknet_estimateFee` call `calc_tx_hashes` which recomputes the hash from the `ExecutableTransactionInput`. For a V3 transaction with zero `l2_gas` and `l1_data_gas`, the hash returned inside `execution_info` (via the `get_execution_info` syscall) is `H_rpc`, while the gateway stores and uses `H_gw`. Any account contract that reads `get_execution_info().transaction_hash` during `__validate__` (the standard pattern for ECDSA accounts) will see `H_gw` on-chain but `H_rpc` in simulation. A user who signs `H_rpc` based on simulation output will have their transaction rejected on-chain. Conversely, a user who signs `H_gw` will see a simulation that appears to fail validation. This is a **High** impact: the RPC simulation and fee estimation return an authoritative-looking wrong `transaction_hash`.

---

### Likelihood Explanation

Any V3 transaction that specifies only L1 gas (the common pattern for users who do not need L2 gas or data gas) triggers this path. The condition `l1_data_gas.is_zero() && l2_gas.is_zero()` is satisfied by a large fraction of real-world V3 transactions. The trigger requires no special privilege and is reachable by any unprivileged user calling `starknet_simulateTransactions` or `starknet_estimateFee`.

---

### Recommendation

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion should not use the *value* of `l1_data_gas` and `l2_gas` to decide the variant for transactions arriving through the V3 RPC path. Since `ResourceBoundsMapping` always carries all three fields explicitly, the conversion should always produce `AllResources` when used for V3 transactions:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        // Always AllResources for V3 transactions; L1Gas is only for
        // pre-0.13.3 transactions that arrive without an l1_data_gas field.
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

Alternatively, the RPC layer should use a separate type for pre-0.13.3 transactions that genuinely lack `l1_data_gas`, rather than inferring the variant from a zero value.

---

### Proof of Concept

1. Construct a V3 invoke transaction: `l1_gas = { max_amount: 1000, max_price_per_unit: 1 }`, `l2_gas = { 0, 0 }`, `l1_data_gas = { 0, 0 }`, `tip = 0`.

2. Call `starknet_simulateTransactions`. The RPC layer converts `ResourceBoundsMapping` → `ValidResourceBounds::L1Gas(l1_gas)`. `get_tip_resource_bounds_hash` hashes `[tip, packed(L1_GAS), packed(L2_GAS=0)]` → hash `H_rpc`.

3. Submit the same transaction to the gateway. `InternalRpcInvokeTransactionV3::resource_bounds()` returns `ValidResourceBounds::AllResources(...)`. `get_tip_resource_bounds_hash` hashes `[tip, packed(L1_GAS), packed(L2_GAS=0), packed(L1_DATA_GAS=0)]` → hash `H_gw`.

4. Assert `H_rpc ≠ H_gw`. The simulation reported `H_rpc` as the transaction hash; the gateway committed `H_gw`. An account that signs `H_rpc` (as shown by simulation) will fail `__validate__` on-chain because the OS passes `H_gw`.
