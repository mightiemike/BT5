### Title
Unchecked `transferFrom` Return Value Enables usdcE Drain Without USDC Payment - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` directly without checking its boolean return value and without using the protocol's own `ERC20Helper.safeTransferFrom` wrapper. If the USDC token on chain 57073 (Ink) returns `false` on failure rather than reverting, an attacker can invoke this function with zero USDC balance or allowance, receive the usdcE held in any victim DDA for free, and leave the DDA drained of usdcE with no USDC deposited in exchange.

---

### Finding Description

`ContractOwner.replaceUsdcEWithUsdc` is an unrestricted `external` function (no `onlyOwner` or similar modifier) that is intended to atomically swap usdcE held in a `DirectDepositV1` (DDA) contract for USDC:

1. Read `balance = usdcE.balanceOf(directDepositV1)`.
2. Pull `balance` USDC from `msg.sender` into the DDA via `transferFrom`.
3. Withdraw usdcE from the DDA to `ContractOwner` via `DirectDepositV1.withdraw`.
4. Send usdcE to `msg.sender` via `safeTransfer`.

Step 2 uses a raw, unchecked call:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [1](#0-0) 

The return value (`bool`) is silently discarded. Steps 3 and 4 execute unconditionally regardless of whether the USDC pull succeeded. The rest of the protocol consistently uses `ERC20Helper.safeTransferFrom` which low-level-calls the token and `require`s both `success` and a truthy decoded return:

```solidity
require(
    success && (data.length == 0 || abi.decode(data, (bool))),
    ERR_TRANSFER_FAILED
);
``` [2](#0-1) 

`replaceUsdcEWithUsdc` bypasses this wrapper entirely.

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on Ink (chain 57073) returns `false` on an insufficient-allowance or insufficient-balance condition instead of reverting:

- An attacker with zero USDC calls `replaceUsdcEWithUsdc(victimSubaccount)`.
- The `transferFrom` silently returns `false`; no USDC moves.
- `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes, pulling all usdcE out of the DDA into `ContractOwner`.
- `usdcE.safeTransfer(msg.sender, balance)` sends the full usdcE balance to the attacker.

The DDA loses its entire usdcE balance. The subaccount owner's pending collateral (usdcE awaiting `creditDeposit`) is stolen. The attacker pays nothing. [3](#0-2) 

---

### Likelihood Explanation

The function is callable by any address on chain 57073 with no access control beyond the `chainid` check. Any DDA that holds usdcE (i.e., any subaccount that received usdcE before `creditDeposit` was called) is a valid target. The exploitability depends on whether the specific USDC deployment on Ink returns `false` vs. reverts on failure — a non-standard but ERC-20-spec-compliant behavior. Given the function is public and the asset delta is concrete (full usdcE balance of any DDA), likelihood is medium-high if the token does not revert.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's existing `ERC20Helper.safeTransferFrom`:

```solidity
// Before (vulnerable):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [4](#0-3) 

This ensures the transaction reverts if the USDC pull fails for any reason, preventing the subsequent usdcE withdrawal and transfer from executing.

---

### Proof of Concept

1. Identify a `subaccount` whose DDA (`directDepositV1Address[subaccount]`) holds a non-zero usdcE balance (e.g., a user who sent usdcE to their DDA but `creditDeposit` has not yet been called).
2. Call `ContractOwner.replaceUsdcEWithUsdc(subaccount)` from an EOA with zero USDC balance and zero USDC allowance granted to `ContractOwner`.
3. If USDC returns `false` on the failed `transferFrom` (rather than reverting):
   - `DirectDepositV1.withdraw(usdcE)` executes — usdcE moves from DDA → `ContractOwner`.
   - `usdcE.safeTransfer(msg.sender, balance)` executes — usdcE moves from `ContractOwner` → attacker.
4. Attacker receives `balance` usdcE; DDA is drained; no USDC was paid. [5](#0-4)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }
```

**File:** core/contracts/libraries/ERC20Helper.sol (L23-42)
```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
