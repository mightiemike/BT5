### Title
Fast Withdrawal Double-Spend via `setWithdrawPool` Replacing Pool With Fresh State — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.setWithdrawPool` allows the owner to swap the `withdrawPool` reference to a new `WithdrawPool` contract. The new pool starts with `markedIdxs` entirely empty and `minIdx == 0`, erasing all memory of previously processed fast withdrawals. Any user who previously submitted a fast withdrawal can replay the same signed transaction against the new pool and receive funds a second time, because the verifier signatures do not commit to the pool address and the new pool has no record of the prior submission.

---

### Finding Description

**Root cause — mutable pool reference with no state migration:**

`Clearinghouse.setWithdrawPool` unconditionally replaces the pool pointer:

```solidity
// Clearinghouse.sol L750-753
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    withdrawPool = _withdrawPool;
}
```

The new `WithdrawPool` is a freshly initialized contract. Its two replay-prevention fields start at zero:

```solidity
// BaseWithdrawPool.sol L37-42
mapping(uint64 => bool) public markedIdxs;   // all false
uint64 public minIdx;                          // 0
```

**Double-spend path in `submitFastWithdrawal`:**

Any caller can invoke `submitFastWithdrawal` directly on the pool contract:

```solidity
// BaseWithdrawPool.sol L81-114
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");  // passes: new pool has no history
    require(idx > minIdx, "idx too small");                     // passes: minIdx == 0
    markedIdxs[idx] = true;
    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);   // passes: same sigs still valid
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);      // funds sent again
}
```

**Verifier signatures are pool-agnostic:**

`requireValidTxSignatures` hashes only `(chainid, idx, txn)`:

```solidity
// Verifier.sol L267-271
bytes32 data = keccak256(
    abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
);
```

The pool address is not included. Signatures collected for a fast withdrawal on the old pool are cryptographically identical and fully valid on the new pool.

---

### Impact Explanation

An attacker who previously submitted a fast withdrawal for index `X` holds a valid `(transaction, signatures)` tuple. After `setWithdrawPool` is called with a funded replacement pool, the attacker calls `submitFastWithdrawal(X, transaction, signatures)` on the new pool. Both replay guards (`markedIdxs[X]` and `idx > minIdx`) pass because the new pool has no history. The pool transfers `transferAmount` to `sendTo` a second time.

- **Concrete asset delta:** The new `WithdrawPool`'s token balance is drained by the replayed withdrawal amount.
- **Broken invariant:** Each withdrawal index must be settled at most once across the lifetime of the protocol, not just within a single pool instance.
- **Severity:** Medium — requires an admin pool-replacement event as a prerequisite, but once that event occurs the exploit is permissionless and requires only on-chain data (the old transaction and signatures) that is already public.

---

### Likelihood Explanation

The admin must call `setWithdrawPool` and the replacement pool must hold liquidity (e.g., seeded for fast-withdrawal service). Both conditions are realistic operational scenarios (pool upgrade, bug fix, liquidity migration). Once they occur, any user who ever submitted a fast withdrawal can replay it with zero additional privilege. The signed transactions and verifier signatures are permanently on-chain and trivially retrievable.

---

### Recommendation

When deploying a replacement pool, initialize `minIdx` to the current `nSubmissions` value from the `Endpoint` contract before activating it. This ensures all previously sequenced indices are below the floor and cannot be replayed:

```solidity
function setWithdrawPool(address _withdrawPool) external onlyOwner {
    require(_withdrawPool != address(0));
    // Prevent replay of all previously processed indices
    BaseWithdrawPool(_withdrawPool).setMinIdx(
        IEndpoint(getEndpoint()).nSubmissions()
    );
    withdrawPool = _withdrawPool;
}
```

Alternatively, require the new pool to be initialized with the full `markedIdxs` bitmap migrated from the old pool before the pointer is switched.

---

### Proof of Concept

1. User signs a `WithdrawCollateral` transaction `T` for `productId=1, amount=1000`.
2. User calls `oldPool.submitFastWithdrawal(idx=500, T, verifierSigs)` → pool sends 1000 tokens to user. `oldPool.markedIdxs[500] = true`.
3. Admin calls `clearinghouse.setWithdrawPool(newPool)`. `newPool.markedIdxs[500] == false`, `newPool.minIdx == 0`.
4. User calls `newPool.submitFastWithdrawal(500, T, verifierSigs)`:
   - `!markedIdxs[500]` → `true` ✓
   - `500 > minIdx (0)` → `true` ✓
   - `requireValidTxSignatures` → passes (same hash, same sigs) ✓
   - Pool sends 1000 tokens to user **again**.
5. User has received 2000 tokens for a single on-chain withdrawal entitlement of 1000. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
