### Title
No Minimum Confirmation Bound Enforced in `verify_transaction_inclusion` — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept a caller-supplied `confirmations` value of `0` with no lower-bound enforcement. Any unprivileged NEAR caller can pass `confirmations: 0` and receive `true` for any transaction in any mainchain block, including one submitted seconds ago by a relayer, before the Bitcoin network has confirmed it at all.

---

### Finding Description

Both proof-verification entry points accept `ProofArgs` / `ProofArgsV2` from any NEAR caller. The only guard on the `confirmations` field is an **upper-bound** check:

```rust
// contract/src/lib.rs  L289-L291
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
``` [1](#0-0) 

The subsequent confirmation-depth check is:

```rust
// contract/src/lib.rs  L304-L308
require!(
    (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
        >= args.confirmations,
    "Not enough blocks confirmed"
);
``` [2](#0-1) 

When `args.confirmations = 0`:
- **Check 1**: `0 <= gc_threshold` → always `true`.
- **Check 2**: `(any u64) >= 0` → always `true`.

The function then proceeds to verify the Merkle proof and returns `true` for any transaction whose block exists anywhere in the current mainchain, regardless of depth. There is no lower bound — not even `>= 1` — enforced by the contract.

The same path is reachable through `verify_transaction_inclusion_v2`, which calls `verify_transaction_inclusion` internally after the coinbase proof check:

```rust
// contract/src/lib.rs  L367-L368
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `ProofArgs` struct exposes `confirmations` as a plain `u64` with no validation annotation:

```rust
// btc-types/src/contract_args.rs  L18-L24
pub struct ProofArgs {
    pub tx_id: H256,
    pub tx_block_blockhash: H256,
    pub tx_index: u64,
    pub merkle_proof: Vec<H256>,
    pub confirmations: u64,
}
``` [4](#0-3) 

---

### Impact Explanation

A downstream NEAR contract (e.g., a bridge, DEX, or custody protocol) that calls `verify_transaction_inclusion` with `confirmations: 0` — or that allows its own users to supply the confirmation count — will receive `true` for a Bitcoin transaction that has zero on-chain confirmations. Because Bitcoin reorganizations are common at shallow depths (1–6 blocks), the "verified" transaction may subsequently disappear from the canonical chain, allowing a double-spend or fraudulent cross-chain action to be finalized on NEAR while the corresponding Bitcoin payment is reversed.

The corrupted invariant is: **the contract's `true` return value no longer guarantees that the referenced transaction is buried under any minimum amount of proof-of-work**, which is the entire security premise of an SPV light client.

---

### Likelihood Explanation

The entry point is fully public and callable by any unprivileged NEAR account. No role, stake, or special permission is required. The attacker only needs to supply a valid Merkle proof for a transaction in a freshly submitted block and set `confirmations: 0`. The relayer already submits blocks continuously, so fresh, shallow blocks are always available.

---

### Recommendation

Enforce a **minimum confirmation count** inside `verify_transaction_inclusion`, rejecting calls where `args.confirmations` is below a protocol-defined floor (e.g., `1` as an absolute minimum, `6` as a sensible Bitcoin default):

```rust
const MIN_CONFIRMATIONS: u64 = 6;

require!(
    args.confirmations >= MIN_CONFIRMATIONS,
    format!("Confirmations must be at least {}", MIN_CONFIRMATIONS)
);
require!(
    args.confirmations <= self.gc_threshold,
    "The required number of confirmations exceeds the number of blocks stored in memory"
);
```

Additionally, expose the minimum as a configurable contract parameter so it can be adjusted per chain (Bitcoin vs. Litecoin vs. Dogecoin have different reorganization risk profiles), while always overriding any caller-supplied value that falls below the floor.

---

### Proof of Concept

1. Relayer submits a single new Bitcoin block header via `submit_blocks`. The mainchain tip advances by 1.
2. Attacker (any NEAR account) calls `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the just-submitted block
   - a valid Merkle proof for a transaction in that block
   - `confirmations: 0`
3. **Check 1** (`0 <= gc_threshold`): passes.
4. **Check 2** (`tip_height - tip_height + 1 = 1 >= 0`): passes.
5. Merkle proof verifies correctly.
6. Function returns `true` — the transaction is declared "verified" with zero confirmations.
7. Any downstream contract acting on this result (e.g., releasing bridged funds) does so for an unconfirmed Bitcoin transaction that may be double-spent or reorganized away.
