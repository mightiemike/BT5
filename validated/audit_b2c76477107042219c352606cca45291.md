### Title
Native ETH Permanently Locked in `DirectDepositV1` Due to Missing `receive()` in `ContractOwner` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `payable fallback()` function. Its `withdrawFromDirectDepositV1` function, when called with `token == address(0)`, invokes `DirectDepositV1.withdrawNative()`, which attempts to push ETH to `address(this)` (i.e., `ContractOwner`). Because `ContractOwner` cannot accept ETH, this call always reverts with `"Failed to transfer native token to owner"`, permanently locking any native ETH that resides in a DDA contract.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` is the sole on-chain recovery path for native ETH held in a `DirectDepositV1` (DDA) contract:

```solidity
// ContractOwner.sol lines 622–636
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external
    onlyOwner
{
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    if (token == address(0)) {
        uint256 preBalance = address(this).balance;
        DirectDepositV1(directDepositV1).withdrawNative();   // <-- sends ETH to ContractOwner
        uint256 postBalance = address(this).balance;
        require(postBalance > preBalance, "empty");          // <-- always fails
        (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
        require(success, "xfer");
    }
    ...
}
```

`DirectDepositV1.withdrawNative()` pushes ETH to its caller (`ContractOwner`) via a low-level call:

```solidity
// DirectDepositV1.sol lines 108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`ContractOwner` inherits only from `EIP712Upgradeable` and `OwnableUpgradeable`, neither of which defines a `receive()` or `payable fallback()`. The ETH push therefore reverts unconditionally, and the `require(success, ...)` in `withdrawNative()` propagates the revert up through `withdrawFromDirectDepositV1`.

**How ETH enters a DDA without being wrapped:**

1. **Pre-deployment funding via CREATE2**: `createDirectDepositV1` deploys every DDA with a fixed salt of `bytes32(uint256(1))`. The DDA address is therefore deterministic and publicly computable. Anyone can send ETH to that address before the DDA is deployed. The DDA constructor attempts to wrap it, but if the `wrappedNative` call fails it silently emits `NativeTokenTransferFailed` and leaves the ETH in the contract:

```solidity
// DirectDepositV1.sol lines 52–60
uint256 balance = address(this).balance;
if (balance != 0) {
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);   // does NOT revert
    }
}
```

2. **`selfdestruct` force-send**: Any contract can `selfdestruct` and designate the DDA as the ETH recipient. This bypasses the DDA's `receive()` function entirely and deposits ETH directly into the DDA's balance.

In both cases the only recovery path — `withdrawFromDirectDepositV1(subaccount, address(0))` — always reverts because `ContractOwner` cannot receive ETH.

---

### Impact Explanation

Native ETH that reaches a DDA through either of the two paths above is permanently locked. There is no alternative on-chain mechanism to extract it: `withdrawNative()` is `onlyOwner` (owner = `ContractOwner`), and `ContractOwner` is the only contract that calls it. The ETH balance of the DDA becomes irrecoverable.

---

### Likelihood Explanation

Medium-low. The DDA CREATE2 address is deterministic (fixed salt `bytes32(uint256(1))`), making pre-deployment funding trivially executable by any actor who computes the address. A `selfdestruct` force-send is similarly available to any deployed contract. The wrapping failure path (constructor silent failure) is less likely under normal conditions but is explicitly handled in the code, confirming the protocol designers anticipated it. Once ETH is in the DDA, the broken recovery path is 100% reliable in its failure.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the fix applied in the referenced H02 report: the executing contract must be able to receive ETH before it can forward it.

---

### Proof of Concept

1. Compute the deterministic DDA address for any `subaccount` using the fixed salt `bytes32(uint256(1))` and the `DirectDepositV1` creation code hash.
2. Send 1 ETH to that address before `createDirectDepositV1(subaccount)` is called (or `selfdestruct` ETH to it after deployment).
3. If sent before deployment: the constructor's wrapping call may fail (e.g., if `wrappedNative` is not yet live), emitting `NativeTokenTransferFailed` and leaving 1 ETH in the DDA.
4. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
5. Execution path: `ContractOwner` → `DirectDepositV1.withdrawNative()` → `ContractOwner.call{value: 1 ether}("")` → **reverts** (no `receive()`).
