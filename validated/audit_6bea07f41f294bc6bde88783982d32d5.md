Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the immediate pool caller (`sender`) instead of the real user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()` — the immediate pool caller. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address. A pool admin who allowlists the router so their curated users can access the pool through the standard periphery inadvertently grants every unprivileged user the same access, fully defeating the allowlist.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // immediate caller of pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap()` forwards this value verbatim to every configured extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`. `SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the immediate caller of `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
```

The router stores the real user's address only in transient callback context (for payment purposes), but never encodes it into `extensionData`. At the pool level `msg.sender = router`, so the extension receives `sender = router`. The check becomes `allowedSwapper[pool][router]`.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension`
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — alice is allowlisted
3. Pool admin calls `setAllowedToSwap(pool, router, true)` — router allowlisted so alice can use standard periphery
4. Bob (not allowlisted) calls `router.exactInputSingle({pool: curated_pool, ...})`
5. Router calls `pool.swap(...)` with `msg.sender = router`
6. Pool calls `_beforeSwap(router, ...)`
7. Extension checks `allowedSwapper[pool][router] == true` → passes
8. Bob's swap executes — allowlist fully bypassed

The same bypass applies to all four router entry points: `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.

The `DepositAllowlistExtension` does not share this flaw because it gates on `owner` (the position owner explicitly passed and preserved by the caller), not on `sender` (the immediate pool caller that the router replaces with itself).

## Impact Explanation

A pool admin who deploys a curated pool with `SwapAllowlistExtension` cannot simultaneously allow their allowlisted users to swap through `MetricOmmSimpleRouter` and block non-allowlisted users from doing the same. If the router is allowlisted — the natural and expected configuration — every user can bypass the allowlist with a single public router call. LPs who deposited into a curated pool expecting restricted counterparty flow are exposed to unrestricted toxic flow, leading to direct LP principal loss through adverse selection. This constitutes broken core pool functionality causing loss of funds.

## Likelihood Explanation

Medium-high. Allowlisting the router is the natural action for any pool admin who wants their curated users to access the pool through the standard periphery. The `MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who does not allowlist the router forces their allowlisted users to call the pool directly — an unusual and friction-heavy requirement. The bypass is reachable by any unprivileged user with a single public router call, requires no special privileges, and is repeatable indefinitely.

## Recommendation

The extension must gate on the economically relevant actor, not the immediate pool caller. Two viable approaches:

1. **Pass the real user through `extensionData`**: Require the router to encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and check that address. The extension must also verify that `sender` is a trusted router before trusting the encoded identity.

2. **Maintain a router registry in the extension**: Allow the pool admin to register trusted routers; when `sender` is a registered router, require the actual user identity to be passed in `extensionData` and verified against the allowlist.

The `DepositAllowlistExtension` pattern — gating on `owner`, which is explicitly passed and preserved — is the correct model to follow.

## Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is allowlisted
  - Pool admin calls setAllowedToSwap(pool, router, true)  // router allowlisted so alice can use it

Attack:
  - bob (not allowlisted) calls router.exactInputSingle({pool: curated_pool, ...})
  - Router calls pool.swap(...) with msg.sender = router
  - Pool calls _beforeSwap(router, ...)
  - SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
  - Swap proceeds — bob has bypassed the allowlist
```

Root cause locations:
- `SwapAllowlistExtension.beforeSwap` L37 checks `sender` (immediate pool caller), not the real user
- `MetricOmmPool.swap()` L231 passes `msg.sender` (the router) as `sender` to the extension
- `MetricOmmSimpleRouter.exactInputSingle()` L72-80 calls `pool.swap()` directly without encoding the real user into `extensionData`
