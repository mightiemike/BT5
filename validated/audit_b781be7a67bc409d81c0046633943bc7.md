### Title
Asymmetric `maxSpreadWeight` caps in `_getSpreadWeightX18` invert the INITIAL/MAINTENANCE health invariant, enabling liquidation of withdrawable positions — (`core/contracts/libraries/RiskHelper.sol`)

---

### Summary

`_getSpreadWeightX18` applies a tighter cap to INITIAL health (`0.99e18`) than to MAINTENANCE health (`0.994e18`). Because `existingWeight` in `Clearinghouse.getHealth` is also health-type-dependent (it uses the long weights returned by `getCoreRisk`, which differ between INITIAL and MAINTENANCE), the INITIAL spread benefit can exceed the MAINTENANCE spread benefit. When this overcomes the base-health difference, total INITIAL health exceeds total MAINTENANCE health — a direct inversion of the required invariant. A liquidator can then liquidate a subaccount whose INITIAL health is positive (i.e., the user could have withdrawn), causing unauthorized loss of the user's position.

---

### Finding Description

**Root cause — `RiskHelper._getSpreadWeightX18`** [1](#0-0) 

```
INITIAL  maxSpreadWeight = 1 - 1/100   = 0.990e18
MAINTENANCE maxSpreadWeight = 1 - 6/1000 = 0.994e18
```

The raw `spreadWeight` is derived from `perpCoreRisk.longWeight` (or `spotCoreRisk.longWeight`), which is the health-type-specific long weight returned by `getCoreRisk`. [2](#0-1) 

`getCoreRisk` always passes `1` to `_getWeightX18`, so `CoreRisk.longWeight` is always `longWeightInitialX18` for INITIAL and `longWeightMaintenanceX18` for MAINTENANCE. [3](#0-2) 

`updateRisk` enforces `longWeightInitial ≤ longWeightMaintenance`, so `existingWeight_M > existingWeight_I` always. [4](#0-3) 

**The health contribution** in `Clearinghouse.getHealth` is: [5](#0-4) 

```
health += basisAmount * (spotPrice + perpPrice) * (spreadWeight - existingWeight)
```

The **spread benefit** for each health type is `spreadWeight - existingWeight`. When both raw weights exceed their respective caps:

| | INITIAL | MAINTENANCE |
|---|---|---|
| `spreadWeight` (capped) | `0.990` | `0.994` |
| `existingWeight` | `(wI_spot + wI_perp)/2` | `(wM_spot + wM_perp)/2` |
| **benefit** | `0.990 − existingWeight_I` | `0.994 − existingWeight_M` |

If `existingWeight_M − existingWeight_I > 0.004`, then `benefit_I > benefit_M`. The INITIAL spread adjustment exceeds the MAINTENANCE spread adjustment. If this difference overcomes the base-health gap (which is always positive in the MAINTENANCE direction), total INITIAL health > total MAINTENANCE health.

---

### Impact Explanation

With INITIAL health > MAINTENANCE health, a subaccount can simultaneously satisfy:
- `getHealth(INITIAL) > 0` → passes withdrawal check (`isAboveInitial`) [6](#0-5) 
- `getHealth(MAINTENANCE) < 0` → passes liquidation eligibility check (`isUnderMaintenance`) [7](#0-6) 

A liquidator can call `liquidateSubaccount` on a position the user could have withdrawn, causing unauthorized mutation of the user's subaccount balances and positions. This matches the Critical scope.

---

### Likelihood Explanation

The condition requires `existingWeight_M − existingWeight_I > 0.004` (i.e., a 0.4% gap between INITIAL and MAINTENANCE long weights) and raw spread weights above 0.99. These are plausible for low-risk, high-weight assets (e.g., stablecoin-adjacent pairs). No attacker privilege is needed — any liquidator can trigger this once the protocol is configured with qualifying weights.

---

### Recommendation

Unify the spread weight cap across health types, or derive `maxSpreadWeight` from the underlying product weights rather than using hardcoded asymmetric constants. At minimum, add an invariant assertion: `assert(getHealth(INITIAL) <= getHealth(MAINTENANCE))` in tests for all spread-eligible products.

---

### Proof of Concept

**Setup (Hardhat, unmodified):**

```
perpLongWeightInitial    = 0.96e9   perpLongWeightMaintenance  = 0.98e9
spotLongWeightInitial    = 0.96e9   spotLongWeightMaintenance  = 0.97e9
perpShortWeightInitial   = 1.04e9   perpShortWeightMaintenance = 1.035e9
spotShortWeightInitial   = 1.04e9   spotShortWeightMaintenance = 1.035e9
```

These pass the `updateRisk` guard (`longWeightInitial ≤ longWeightMaintenance`, `shortWeightInitial ≥ shortWeightMaintenance`). [4](#0-3) 

**Position:** long 1 spot + short 1 perp, price = 1000, quote balance = 25.

**Arithmetic:**

```
raw_I  = 1 − (1−0.96)/5 = 0.992  → capped → spreadWeight_I  = 0.990
raw_M  = 1 − (1−0.98)/5 = 0.996  → capped → spreadWeight_M  = 0.994

existingWeight_I = (0.96+0.96)/2 = 0.960
existingWeight_M = (0.97+0.98)/2 = 0.975

spread_adj_I = 1 × 2000 × (0.990−0.960) = 60
spread_adj_M = 1 × 2000 × (0.994−0.975) = 38

base_health_I = (0.96×1000) + (−1.04×1000) = −80
base_health_M = (0.97×1000) + (−1.035×1000) = −65

total_I = −80 + 60 + 25 = +5   ← INITIAL health > 0  (withdrawable)
total_M = −65 + 38 + 25 = −2   ← MAINTENANCE health < 0 (liquidatable)
```

**Assert:**
```javascript
assert(await clearinghouse.getHealth(subaccount, INITIAL) > 0);    // passes
assert(await clearinghouse.getHealth(subaccount, MAINTENANCE) < 0); // passes
// Liquidator calls liquidateSubaccount → succeeds, draining a withdrawable position
``` [8](#0-7) [9](#0-8)

    function _getSpreadWeightX18(
        IProductEngine.CoreRisk memory perpCoreRisk,
        IProductEngine.CoreRisk memory spotCoreRisk,
        IProductEngine.HealthType healthType
    ) internal pure returns (int128) {
        if (healthType == IProductEngine.HealthType.PNL) {
            return ONE;
        }
        int128 spreadWeight;
        if (spotCoreRisk.amount > 0) {
            spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
        } else {
            spreadWeight = ONE - (ONE - spotCoreRisk.longWeight) / 5;
        }
        int128 maxSpreadWeight;
        if (healthType == IProductEngine.HealthType.INITIAL) {
            maxSpreadWeight = ONE - ONE / 100; // 0.99
        } else {
            maxSpreadWeight = ONE - (ONE * 6) / 1000; // 0.994
        }
        if (spreadWeight > maxSpreadWeight) {
            spreadWeight = maxSpreadWeight;
        }
        return spreadWeight;
    }
```
