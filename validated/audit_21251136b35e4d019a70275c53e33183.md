### Title
Per-Bin Additional Fee Has No Cap, Allowing Pool Admin to Exceed Factory Fee Limits - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation, while the analogous `setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps. A pool admin can set per-bin additional fees to `type(uint16).max` (65 535 in E6 = 6.5535%) on any bin, bypassing the factory's cap system and extracting excess fees from swappers in that bin.

### Finding Description
`setPoolAdminFees` enforces caps before writing:

```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` has no equivalent check and forwards the raw caller-supplied values straight to the pool:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-side handler also performs no cap check:

```solidity
// MetricOmmPool.sol lines 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

The only validation is the bin-index range check; the fee values themselves are written unconditionally. Because `addFeeBuyE6` and `addFeeSellE6` are `uint16`, the pool admin can supply `65535` (6.5535 % in E6 units) for any bin, on top of whatever base spread fee is already configured. The factory's cap machinery (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) is entirely bypassed for this code path.

### Impact Explanation
Swappers whose trades route through the targeted bin pay the base spread fee **plus** the uncapped per-bin additional fee. At `uint16.max` the additional fee is 6.5535 %, which can be front-run onto a pending swap in the same way described in the reference report: the pool admin monitors the mempool, calls `setPoolBinAdditionalFees` with the maximum value immediately before the victim's swap lands, and the victim's output is reduced by the inflated fee. The excess fee accrues to the admin fee destination, constituting a direct loss of user principal. This is a Medium-severity direct loss of user funds and an admin-boundary break (pool admin exceeds the cap system the factory is designed to enforce).

### Likelihood Explanation
Low — requires a malicious or compromised pool admin. However, the inconsistency is structural: `setPoolAdminFees` is capped while `setPoolBinAdditionalFees` is not, so the cap system the factory exposes to LPs and swappers as a safety guarantee does not actually cover all fee-setting paths.

### Recommendation
Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that can be tuned independently.

### Proof of Concept
1. Pool is deployed with `spreadFeeE6 = 10 000` (1 %) and `maxAdminSpreadFeeE6 = 200 000` (20 %).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Factory forwards the call; pool writes `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0 with no revert.
4. A victim's swap that routes through bin 0 now pays 1 % (base) + 6.5535 % (per-bin) = 7.5535 % total spread fee.
5. The excess 6.5535 % flows to `adminFeeDestination`, not to the LP or the swapper — a direct loss of principal to the swapper that the factory's published cap system was supposed to prevent. [1](#0-0) [2](#0-1) [3](#0-2)
