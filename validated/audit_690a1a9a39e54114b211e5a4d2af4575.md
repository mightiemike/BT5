### Title
Bonus Pool Depletion Blocks `TierUndelegate` and `ExitTierWithDelegation`, Locking User Tokens Indefinitely Beyond the Exit Commitment Period — (`x/tieredrewards/keeper/msg_server.go`)

---

### Summary

Both exit paths from a tiered-rewards position — `MsgTierUndelegate` and `MsgExitTierWithDelegation` — unconditionally call `claimRewards` before executing the actual delegation transfer. `claimRewards` calls `processEventsAndClaimBonus`, which atomically reverts the entire transaction if the bonus rewards pool balance is insufficient to cover accrued bonus. When the pool is empty, users who have already completed their exit lock duration (e.g., 1 year) have **no alternative path** to recover their locked tokens. The tokens remain locked in the tier module indefinitely until an external party replenishes the pool.

---

### Finding Description

`MsgTierUndelegate` in `msg_server.go` calls `claimRewards` at line 166 before undelegating:

```go
pos, _, _, err = ms.claimRewards(ctx, pos)
if err != nil {
    return nil, err
}
``` [1](#0-0) 

`claimRewards` calls `processEventsAndClaimBonus`, which checks pool sufficiency and returns `ErrInsufficientBonusPool` if the pool cannot cover the accrued bonus:

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err
}
``` [2](#0-1) 

`sufficientBonusPoolBalance` checks the module account balance and returns an error if it cannot cover the bonus: [3](#0-2) 

The same `claimRewards` call exists in `ExitTierWithDelegation` at line 532, blocking the only other exit path: [4](#0-3) 

The ADR explicitly documents this behavior as intentional for user-driven paths:

> **User-driven paths** (ClaimTierRewards, AddToPosition, Undelegate, Redelegate, ClearPosition): fail atomically. User retries after pool is replenished. [5](#0-4) 

The validation for `WithdrawFromTier` requires the position to be **not delegated** and **not unbonding** — meaning `TierUndelegate` must succeed first before `WithdrawFromTier` can be called: [6](#0-5) 

This creates a complete lock: if the bonus pool is empty, `TierUndelegate` reverts → the position stays delegated → `WithdrawFromTier` is also blocked (position is still delegated). There is no exit path that bypasses reward settlement.

---

### Impact Explanation

A user who has:
1. Locked tokens in a tier (e.g., 1-year tier)
2. Triggered exit
3. Waited the full exit lock duration

...is still unable to recover their tokens if the bonus pool is empty at the time they attempt to undelegate. The corrupted invariant is: **a user who has completed the exit commitment period must be able to exit**. The corrupted value is the user's locked token balance — it remains trapped in the tier module's per-position delegator address with no accessible exit path.

The pool is funded externally (by governance or protocol actors). If the pool is drained by high bonus demand, not replenished promptly, or governance fails to act, users face an indefinite lock on their principal — not just their rewards.

---

### Likelihood Explanation

The bonus pool is a shared resource consumed by all positions across all tiers. A high-APY tier with many large positions can drain the pool rapidly. The pool has no automatic replenishment mechanism — it depends on external funding. A realistic scenario: the pool is drained during a period of high claim activity; a user who has waited 1 year to exit their position now finds both `TierUndelegate` and `ExitTierWithDelegation` reverting. The entry path is a standard `MsgTierUndelegate` transaction submitted by any position owner after their exit lock has elapsed.

---

### Recommendation

Separate reward settlement from the exit path. Specifically:

1. In `TierUndelegate` and `ExitTierWithDelegation`, if `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool`, **skip the bonus payment** (forfeit or defer it) rather than reverting the entire transaction. The user should always be able to exit with their principal after the exit commitment elapses.
2. Alternatively, record the owed bonus as a claimable debt and allow the undelegate to proceed, paying the bonus when the pool is replenished.
3. At minimum, add a governance-controlled emergency exit that bypasses reward settlement for positions past their exit commitment.

---

### Proof of Concept

1. User calls `MsgLockTier` with `trigger_exit_immediately = true` on a tier with `ExitDuration = 365 days`.
2. User waits 365 days for exit lock to elapse.
3. Bonus pool is drained to zero (by other claimants or simply not replenished).
4. User submits `MsgTierUndelegate`:
   - `validateUndelegatePosition` passes (exit triggered, elapsed, delegated). [7](#0-6) 
   - `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`. [8](#0-7) 
   - Transaction reverts. Position remains delegated.
5. User submits `MsgExitTierWithDelegation`:
   - Same `claimRewards` call at line 532 reverts for the same reason. [4](#0-3) 
6. `MsgWithdrawFromTier` is also blocked because `validateWithdrawFromTier` requires `!pos.IsDelegated()`. [9](#0-8) 
7. User's principal is locked indefinitely. The unit test `TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient` confirms this revert behavior. [10](#0-9)

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil

func (k Keeper) validateUndelegatePosition(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.IsDelegated() {
		return types.ErrPositionNotDelegated
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	// skip check for zero amount as we want those positions to be able to close their position properly

	return nil

func (k Keeper) validateWithdrawFromTier(ctx context.Context, pos types.PositionState, owner string) error {
	if !pos.IsOwner(owner) {
		return types.ErrNotPositionOwner
	}

	if !pos.HasTriggeredExit() {
		return types.ErrExitNotTriggered
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if !pos.CompletedExitLockDuration(sdkCtx.BlockTime()) {
		return types.ErrExitLockDurationNotReached
	}

	if pos.IsDelegated() {
		return types.ErrPositionDelegated
	}

	isUnbonding, err := k.isUnbonding(ctx, pos.DelegatorAddress)
	if err != nil {
		return err
	}
	if isUnbonding {
		return types.ErrPositionUnbonding
	}

	return nil
```

// TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient verifies that ClaimTierRewards
// returns ErrInsufficientBonusPool when accrued bonus cannot be paid, so the tx rolls
// back and the user can retry later without losing base rewards to a partial claim.
func (s *KeeperSuite) TestMsgClaimTierRewards_FailsWhenBonusPoolInsufficient() {
	lockAmount := sdkmath.NewInt(sdk.DefaultPowerReduction.Int64())
	pos := s.setupNewTierPosition(lockAmount, false)
	delAddr := sdk.MustAccAddressFromBech32(pos.Owner)
	valAddr := sdk.MustValAddressFromBech32(pos.Delegation.ValidatorAddress)
	_, bondDenom := s.getStakingData()
	msgServer := keeper.NewMsgServerImpl(s.keeper)

	s.setValidatorCommission(valAddr, sdkmath.LegacyZeroDec())

	// Advance time and allocate base rewards, but intentionally leave bonus pool empty.
	s.ctx = s.ctx.WithBlockHeight(s.ctx.BlockHeight() + 1)
	s.ctx = s.ctx.WithBlockTime(s.ctx.BlockTime().Add(time.Hour * 24 * 365))
	s.allocateRewardsToValidator(valAddr, sdkmath.NewInt(100), bondDenom)

	// Bonus pool remains at 0 — bonus accrued but pool cannot cover it.
	balBefore := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)

	// Use a branched context so a failed message does not persist state (matches DeliverTx rollback).
	cacheCtx, _ := s.ctx.CacheContext()
	resp, err := msgServer.ClaimTierRewards(cacheCtx, &types.MsgClaimTierRewards{
		Owner:       delAddr.String(),
		PositionIds: []uint64{pos.Id},
	})
	s.Require().Error(err)
	s.Require().True(errors.Is(err, types.ErrInsufficientBonusPool))
	s.Require().Nil(resp)

	balAfter := s.app.BankKeeper.GetBalance(s.ctx, delAddr, bondDenom)
	s.Require().True(balAfter.Amount.Equal(balBefore.Amount), "failed claim must not transfer rewards")
}
