### Title
`last_resharding = None` Bypasses Resharding Cooldown at Static-to-Dynamic Resharding Upgrade Boundary — (`chain/epoch-manager/src/lib.rs`)

### Summary

`EpochInfoV5::last_resharding` is `None` for all pre-V5 `EpochInfo` variants and for the first V5 epoch created when dynamic resharding is enabled. The `can_reshard` cooldown gate treats `None` as "no resharding ever happened" and unconditionally returns `true`. This means that if a static resharding occurred in the epoch immediately before dynamic resharding was enabled, the mandatory inter-resharding cooldown is silently bypassed, allowing a dynamic resharding to be scheduled one epoch earlier than the invariant permits.

### Finding Description

`EpochInfoV5` stores `last_resharding: Option<EpochHeight>` to track when the most recent resharding occurred. [1](#0-0) 

For all pre-V5 variants (`V1`–`V4`), `last_resharding()` unconditionally returns `None`: [2](#0-1) 

The cooldown gate in `can_reshard` uses `is_none_or`, which evaluates to `true` whenever `last_resharding()` is `None`: [3](#0-2) 

The code comment explicitly acknowledges the broken invariant:

> "It is theoretically possible that a static resharding was scheduled right before enabling dynamic resharding, but we assume this didn't happen."

In `finalize_epoch`, `last_resharding` is propagated forward only from the dynamic-resharding pipeline: [4](#0-3) 

Static resharding epoch heights are never written into `last_resharding`. The first V5 `EpochInfo` created at the static→dynamic upgrade boundary therefore carries `last_resharding = None`, regardless of how recently a static resharding occurred.

The cooldown check is exercised in two places:

1. `compute_proposed_split` (chunk application): [5](#0-4) 

2. `get_upcoming_shard_split` (block production and validation): [6](#0-5) 

Both call `can_reshard` with the same `block_hash`, and both will receive `true` in the first dynamic-resharding epoch even if a static resharding just occurred.

### Impact Explanation

The documentation explicitly states why back-to-back reshardings are unsafe:

> "allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`." [7](#0-6) 

If a static resharding occurs at epoch H and dynamic resharding is enabled at epoch H (same protocol upgrade), `can_reshard` returns `true` during epoch H. A dynamic resharding is then proposed at epoch H and takes effect at epoch H+2. The cooldown check — `(H+1) − H = 1 ≥ 2` — would have blocked this if `last_resharding` were properly set to H, but since it is `None`, the check is skipped. The resulting `InvalidChunkHeaderShardSplit` validation failure causes a consensus-level chain halt.

### Likelihood Explanation

Dynamic resharding (`ProtocolFeature::DynamicResharding`) is currently a nightly feature: [8](#0-7) 

The scenario requires a static resharding to coincide with the epoch in which dynamic resharding is first enabled. This is a real upgrade-boundary condition: the production `EpochConfig` for protocol version 85 already ships a `dynamic_resharding_config` block: [9](#0-8) 

When dynamic resharding is stabilized, any network that has undergone a static resharding within `min_epochs_between_resharding` epochs of the activation point will silently bypass the cooldown on the first dynamic-resharding epoch.

### Recommendation

When creating the first `EpochInfoV5` at the static→dynamic resharding transition, initialize `last_resharding` to the epoch height of the most recent static resharding rather than `None`. This can be computed by inspecting the shard layout history: if the shard layout changed between the previous epoch and the current one, record the current epoch height as `last_resharding`. Concretely, in `finalize_epoch`, the `has_same_shard_layout` flag already detects layout changes: [4](#0-3) 

The same logic should be applied when the previous epoch's `EpochInfo` is pre-V5 (i.e., `last_resharding()` returns `None` but the shard layout actually changed), so that the first V5 epoch carries the correct `last_resharding` value instead of `None`.

### Proof of Concept

1. Configure a network with `min_epochs_between_resharding = 2` and a static shard layout change at protocol version P.
2. In the same protocol version P, enable `ShardLayoutConfig::Dynamic` with `force_split_shards` containing a shard that exists in the new layout.
3. At epoch H (the first epoch running protocol version P), `finalize_epoch` creates the first `EpochInfoV5` with `last_resharding = None`.
4. `can_reshard` is called during epoch H; `next_epoch_info.last_resharding()` returns `None`; `is_none_or` evaluates to `true`; `can_reshard` returns `true`.
5. `get_upcoming_shard_split` selects the forced shard; the split is embedded in the last block of epoch H.
6. `finalize_epoch` derives a new `ShardLayoutV3` for epoch H+2.
7. The child shard created by the static resharding at H has its first chunk at epoch H+1 with `proposed_split = None`. The dynamic resharding pipeline at epoch H+2 encounters a `proposed_split` mismatch, triggering `InvalidChunkHeaderShardSplit` and halting the chain. [10](#0-9) [2](#0-1)

    fn can_reshard(
        &self,
        block_hash: &CryptoHash,
        min_epochs_between_resharding: NonZeroEpochHeight,
    ) -> Result<bool, EpochError> {
        let block_info = self.get_block_info(block_hash)?;
        let next_epoch_id = self.get_next_epoch_id_from_info(&block_info)?;
        let next_epoch_info = self.get_epoch_info(&next_epoch_id)?;

        // last_resharding() returns `None` if no resharding happened since dynamic resharding
        // has been enabled. It is theoretically possible that a static resharding was scheduled
        // right before enabling dynamic resharding, but we assume this didn't happen.
        let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
            next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
        });
        Ok(can_reshard)
    }
