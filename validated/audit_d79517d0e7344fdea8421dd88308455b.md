Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating EOA, allowing any unprivileged caller to bypass per-pool swap allowlists via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's own `msg.sender` — the router contract — not the EOA that initiated the trade. When a pool admin allowlists the router address so that legitimate users can access the pool through `MetricOmmSimpleRouter`, every unprivileged EOA on the network gains the same access, because the extension has no visibility into the original caller. The allowlist is structurally nullified for any pool that uses the standard periphery.

## Finding Description

**Root cause:** `MetricOmmPool.swap` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ← router address when called via MetricOmmSimpleRouter
  recipient,
  ...
  extensionData
);
```

`ExtensionCalling._beforeSwap` faithfully forwards this value to every registered extension. `SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The extension has no access to the original EOA.

`MetricOmmSimpleRouter.exactInputSingle` stores the real payer only in transient storage for payment settlement — it is never forwarded to the pool's `swap` call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    ...
    params.extensionData   // ← original msg.sender never included
  );
```

**Inescapable dilemma for the pool admin:**

| Admin choice | Effect |
|---|---|
| Allowlist the router | Every EOA can swap — allowlist nullified |
| Do not allowlist the router | Allowlisted users cannot use the standard interface |

No configuration simultaneously allows allowlisted users to use `MetricOmmSimpleRouter` and blocks non-allowlisted users.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to a curated set of addresses (KYC-verified counterparties, institutional LPs, closed beta) loses that restriction entirely the moment the admin allowlists the router for legitimate users. Any unprivileged attacker can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router and execute swaps against the pool. This constitutes broken core pool functionality: the access-control guard that is supposed to fail closed instead fails open for the entire public, enabling unauthorized extraction of token0/token1 from the pool and directly reducing LP principal.

## Likelihood Explanation

The scenario requires the admin to allowlist the router — a natural and expected step when deploying a pool intended to be accessible through the protocol's standard periphery. The `FullMetricExtension` integration test confirms the intended pattern: it allowlists a `TestCaller` wrapper (the direct pool caller) rather than the router, which works only because the test bypasses the router entirely. Any real deployment that wants allowlisted users to use the router will hit this path. The `generate_scanned_questions.py` audit target explicitly flags this exact concern. The condition is trivially reachable by any EOA with no special privileges.

## Recommendation

The `sender` argument passed to `beforeSwap` must represent the economically relevant actor — the EOA initiating the trade — not the intermediate contract. Two viable fixes:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the original `msg.sender` into `extensionData` and have `SwapAllowlistExtension` decode and verify it. This requires a trust assumption that only an allowlisted router can forge this field, enforced by checking `msg.sender` (the pool) and validating the router identity on the extension side.

2. **Pool-side**: Add a separate `originator` field to the `swap` signature that the pool passes through to extensions, allowing the router to supply the real EOA. Extensions gate on `originator` instead of `sender`.

Either approach must ensure the router cannot be used to spoof an arbitrary allowlisted address.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only userA is allowlisted.
// Admin also allowlists the router so userA can use the standard interface.
swapExtension.setAllowedToSwap(address(pool), userA, true);
swapExtension.setAllowedToSwap(address(pool), address(router), true); // ← necessary for userA to use router

// Attacker (not allowlisted) calls the router directly.
vm.prank(attacker); // attacker != userA, not in allowlist
router.exactInputSingle(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: token0,
        tokenOut: token1,
        zeroForOne: true,
        amountIn: 1_000e18,
        amountOutMinimum: 0,
        recipient: attacker,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Swap succeeds: extension checked allowedSwapper[pool][router] == true,
// never inspecting `attacker`. Allowlist is bypassed.
```

The pool's `_beforeSwap` receives `sender = address(router)` at `MetricOmmPool.sol` L230-231. The extension approves it because `allowedSwapper[pool][router] == true` at `SwapAllowlistExtension.sol` L37. The router stores the real payer only in transient callback context at `MetricOmmSimpleRouter.sol` L71, never forwarding it to the pool.
