### Title
Missing Chain Differentiation in `draft-EIP712Upgradeable` Domain Separator Enables Cross-Fork Signature Replay for Collateral Withdrawals and Sensitive Operations — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx` inherits `EIP712Upgradeable` and initializes the domain separator via `__EIP712_init("Nado", "0.0.1")`. The `draft-EIP712Upgradeable` implementation caches the domain separator at initialization time and only rebuilds it when `block.chainid` differs from the cached value. On a post-deployment chain fork that retains the same `chainId`, the cached domain separator is identical on both chains. All user-signed transactions validated through `validateSignedTx` — including `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, and `LinkSigner` — produce the same EIP-712 digest on both chains, enabling full signature replay.

---

### Finding Description

`EndpointTx.validateSignedTx` computes the final digest as:

```solidity
_hashTypedDataV4(
    computeDigest(
        IEndpoint.TransactionType(uint8(transaction[0])),
        transaction[1:]
    )
)
``` [1](#0-0) 

`_hashTypedDataV4` is inherited from `draft-EIP712Upgradeable` and wraps the struct hash with the cached domain separator:

```solidity
__EIP712_init("Nado", "0.0.1");
``` [2](#0-1) 

The `draft-EIP712Upgradeable` implementation caches the domain separator at `__EIP712_init` time. Its `_domainSeparatorV4()` returns the cached value when `block.chainid == _CACHED_CHAIN_ID`. On a fork that retains the same `chainId`, this condition is always true, so the domain separator is identical on both chains.

Compounding this, `Verifier.computeDigest` is declared `public pure`:

```solidity
function computeDigest(
    IEndpoint.TransactionType txType,
    bytes calldata transactionBody
) public pure returns (bytes32) {
``` [3](#0-2) 

Being `pure`, it cannot read `block.chainid` or any chain-specific state. The struct hashes it produces for all transaction types — `WithdrawCollateral`, `WithdrawCollateralV2`, `LiquidateSubaccount`, `TransferQuote`, `MintNlp`, `BurnNlp`, `LinkSigner` — contain zero chain binding: [4](#0-3) [5](#0-4) 

The entire chain-binding responsibility falls on `_hashTypedDataV4`, which fails to distinguish chains when the fork retains the same `chainId`.

---

### Impact Explanation

On a post-deployment chain fork with the same `chainId` (e.g., a contentious hard fork where both chains retain the original `chainId`):

- A user's signed `WithdrawCollateral` or `WithdrawCollateralV2` transaction is cryptographically valid on both chains. If the fork occurs before the transaction is sequenced, the nonce state on the forked chain is identical to the original chain at the fork point. The sequencer operating on both chains (normal post-fork behavior) processes the same signed transaction on both chains, draining the user's collateral on both.
- A signed `LinkSigner` transaction replayed on the forked chain installs an attacker-controlled linked signer, granting persistent signing authority over the subaccount on that chain.
- A signed `LiquidateSubaccount` transaction replayed on the forked chain forces an unauthorized liquidation, destroying the victim's position.
- A signed `TransferQuote` transaction replayed on the forked chain transfers quote tokens to an unintended recipient.

The corrupted state delta is: `spotEngine` balance for the victim subaccount is decremented on the forked chain without the user's intent, and `linkedSigners[sender]` may be overwritten. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

Chain forks that retain the same `chainId` have occurred historically (e.g., BSC/BNB chain forks, Ethereum Classic). Nado is deployed on EVM-compatible chains where this risk is non-negligible. After a fork, the sequencer naturally continues operating on both chains — this is not a compromise scenario but standard operational behavior. Any signed transaction submitted to the sequencer before the fork is replayable on the forked chain because the nonce state is identical at the fork point and the domain separator is unchanged.

---

### Recommendation

**Short-term:** Override `_hashTypedDataV4` or `_domainSeparatorV4` in `EndpointTx` to always recompute the domain separator using the live `block.chainid` rather than relying on the cached value from `draft-EIP712Upgradeable`. Alternatively, include `block.chainid` directly in each struct hash inside `Verifier.computeDigest` (removing the `pure` mutability).

**Long-term:** Migrate from `draft-EIP712Upgradeable` to a version that unconditionally recomputes the domain separator on every call, or document and implement a post-fork migration procedure that re-initializes the domain separator with the new chain's identity.

---

### Proof of Concept

1. Alice signs a `WithdrawCollateral` transaction (nonce = N) on the original chain and submits it to the sequencer.
2. Before the sequencer processes it, the chain forks into chain A and chain A′, both with the same `chainId`. Both chains have Alice's nonce at N.
3. The sequencer processes Alice's transaction on chain A: nonce advances N → N+1, collateral is withdrawn.
4. The sequencer (or any party operating the sequencer role on chain A′) submits the same signed transaction on chain A′.
5. `validateSignedTx` on chain A′ computes `_hashTypedDataV4(computeDigest(...))`. Because `block.chainid == _CACHED_CHAIN_ID`, the cached domain separator is returned — identical to chain A. The struct hash from `computeDigest` (pure, no chain binding) is also identical. The digest matches Alice's signature.
6. Alice's nonce on chain A′ is still N, so `validateNonce` passes.
7. Alice's collateral is withdrawn on chain A′ without her intent. [8](#0-7) [9](#0-8) [10](#0-9)
