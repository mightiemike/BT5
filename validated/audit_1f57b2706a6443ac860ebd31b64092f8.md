### Title
Hardcoded `ReservedDataAvailabilityMode::Reserved` in RPC-to-client-transaction conversion silently overwrites actual DA mode, corrupting the hash-preimage fields returned by the RPC layer - (`crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

Three `From` conversions in `crates/apollo_rpc/src/v0_8/transaction.rs` unconditionally replace the caller-supplied `nonce_data_availability_mode` and `fee_data_availability_mode` with the hardcoded constant `client_transaction::ReservedDataAvailabilityMode::Reserved` (integer `0`, meaning L1). Because both DA-mode fields are committed into the V3 transaction hash preimage via `concat_data_availability_mode`, any transaction submitted with `DataAvailabilityMode::L2` will have its hash computed over the real value `1`, but the representation returned by the RPC/FGW layer will carry `0`. A client that re-derives the hash from the returned object will obtain a different digest than the one stored and broadcast by the sequencer.

---

    client_transaction::ReservedDataAvailabilityMode::Reserved,
``` [3](#0-2) 

The writer-side `ReservedDataAvailabilityMode` enum has only one variant (`Reserved = 0`), so the type system cannot represent L2 DA mode at all in this path. [4](#0-3) 

**Why DA mode is hash-critical**

`concat_data_availability_mode` packs both modes into a single felt that is chained into every V3 hash:

```rust
fn concat_data_availability_mode(nonce_mode, fee_mode) -> Felt {
    (data_availability_mode_index(fee_mode)
        + (data_availability_mode_index(nonce_mode) << DATA_AVAILABILITY_MODE_BITS))
        .into()
}
``` [5](#0-4) 

`DataAvailabilityMode::L1 → 0`, `DataAvailabilityMode::L2 → 1`. A transaction submitted with `nonce_data_availability_mode = L2` produces a hash over `1`; the RPC response carries `0`; re-deriving the hash from the response yields a different digest.

**Ingestion path uses the real value; RPC response path does not**

`TransactionConverter::convert_rpc_tx_to_internal` calls `InternalRpcTransactionWithoutTxHash::calculate_transaction_hash`, which dispatches to the trait implementations that read the actual `nonce_data_availability_mode` / `fee_data_availability_mode` fields from the `InternalRpcDeclareTransactionV3` / `InternalRpcInvokeTransactionV3` structs. [6](#0-5) 

The hash stored in `InternalRpcTransaction.tx_hash` is therefore correct. The divergence occurs only when the stored transaction is serialised back through the `client_transaction` conversion for the RPC/FGW response.

**Gateway accepts L2 DA mode**

`RpcTransaction` and all three `InternalRpc*` structs carry `DataAvailabilityMode` (not `ReservedDataAvailabilityMode`), so the gateway imposes no admission gate on the DA mode value. [7](#0-6) 

The OS Cairo code does assert `nonce_data_availability_mode = 0` and `fee_data_availability_mode = 0` (with a TODO to relax this once volition is supported), but that check occurs at proving time, not at gateway admission or blockifier execution time. [8](#0-7) 

---

### Impact Explanation

**High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

A client that calls `starknet_getTransactionByHash` (or any equivalent FGW endpoint) for a V3 transaction submitted with `nonce_data_availability_mode = L2` receives a response whose DA-mode fields are `0` (L1) instead of `1` (L2). Re-computing the transaction hash from the returned object yields a digest that does not match the one stored by the sequencer and broadcast to peers. This breaks:

- Client-side hash verification / receipt matching.
- Any downstream component (explorer, wallet, bridge) that reconstructs the hash from the returned fields.
- Signature-domain binding: the account contract signed over the hash that includes the real DA mode; the hash derivable from the RPC response is different, so the signature appears invalid against the returned data.

---

### Likelihood Explanation

The gateway accepts V3 transactions with any `DataAvailabilityMode`. No admission check rejects `L2`. The blockifier executes such transactions without error. The OS proving assertion (`assert … = 0`) is a future-compatibility guard with a TODO to remove it once volition is activated. As soon as volition is enabled (or if a node is run without the OS proving step), L2-DA-mode transactions will be accepted end-to-end while the RPC layer silently returns the wrong DA mode.

---

### Recommendation

Replace the three hardcoded `ReservedDataAvailabilityMode::Reserved` assignments with conversions that preserve the actual field value from the source transaction:

```rust
// Instead of:
nonce_data_availability_mode: client_transaction::ReservedDataAvailabilityMode::Reserved,
fee_data_availability_mode:   client_transaction::ReservedDataAvailabilityMode::Reserved,

// Use:
nonce_data_availability_mode: deploy_account_tx.nonce_data_availability_mode.into(),
fee_data_availability_mode:   deploy_account_tx.fee_data_availability_mode.into(),
```

The `From<DataAvailabilityMode> for ReservedDataAvailabilityMode` conversion already exists in the reader objects and correctly maps `L1 → Reserved` and `L2 → L2`. [9](#0-8) 

The writer-side `ReservedDataAvailabilityMode` enum must also be extended with an `L2 = 1` variant to match the reader, or the two types should be unified.

Add round-trip tests that submit a V3 transaction with `nonce_data_availability_mode = L2`, retrieve it via the RPC/FGW path, and assert that the returned DA modes match the submitted values and that the hash re-derived from the response equals the stored hash.

---

### Proof of Concept

1. Construct a V3 invoke transaction with `nonce_data_availability_mode = DataAvailabilityMode::L2`.
2. Submit it via the gateway (`/gateway/add_rpc_transaction`). The gateway accepts it; `convert_rpc_tx_to_internal` computes `tx_hash` using `concat_data_availability_mode(L2, L1)` → the felt encodes `nonce_mode = 1`.
3. Retrieve the transaction via `starknet_getTransactionByHash`. The response is produced through `From<InvokeTransactionV3> for client_transaction::InvokeTransaction`, which sets `nonce_data_availability_mode = Reserved` (0).
4. Re-derive the hash from the returned object using `concat_data_availability_mode(L1, L1)` → the felt encodes `nonce_mode = 0`.
5. Observe that the re-derived hash ≠ the hash returned in the response, demonstrating that the RPC layer returns an authoritative-looking but internally inconsistent transaction object. [10](#0-9) [11](#0-10)
