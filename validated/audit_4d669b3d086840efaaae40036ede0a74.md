### Title
Unauthenticated Transaction Submission via Missing Sequencer and Schnorr Checks in `submitTransactionsCheckedWithGasLimit` - (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.submitTransactionsCheckedWithGasLimit` is callable by any unprivileged address. Unlike its sibling `submitTransactionsChecked`, it omits both the `require(msg.sender == sequencer)` guard and the `verifier.requireValidSignature(...)` Schnorr quorum check. Any attacker can submit and execute arbitrary protocol transactions — including unsigned liquidations via the `N_ACCOUNT` bypass — without sequencer authorization or multi-signer approval.

---

### Finding Description

`submitTransactionsChecked` enforces two mandatory guards before processing any batch:

```solidity
// Endpoint.sol:279,287
require(msg.sender == sequencer);
verifier.requireValidSignature(digest, e, s, signerBitmask);
```

`submitTransactionsCheckedWithGasLimit` performs neither check:

```solidity
// Endpoint.sol:296-312
function submitTransactionsCheckedWithGasLimit(
    uint64 idx,
    bytes[] calldata transactions,
    uint256 gasLimit
) external {
    uint256 initialGas = gasleft();
    validateSubmissionIdx(idx);          // only checks idx == nSubmissions
    for (uint256 i = 0; i < transactions.length; i++) {
        bytes calldata transaction = transactions[i];
        processTransaction(transaction); // no caller or signature gate
        ...
    }
    verifier.revertGasInfo(...);         // always reverts — used for gas measurement only
}
```

The function is `external` with no `onlySequencer` modifier and no Schnorr batch-signature verification. The sole guard, `validateSubmissionIdx`, only confirms the monotonic counter matches — a value any observer can read from the public `nSubmissions` state variable.

Once inside `processTransactionImpl` (reached via `delegatecall`), several transaction types execute without any per-transaction signature check:

- **`LiquidateSubaccount` with `sender == N_ACCOUNT`**: The signature validation block is explicitly skipped when `sender == N_ACCOUNT`, and `clearinghouse.liquidateSubaccount` is called unconditionally.
- **`UpdatePrice`**: Calls `clearinghouse.updatePrice` with attacker-supplied price data.
- **`SpotTick` / `PerpTick`**: Calls `spotEngine.updateStates` / `perpEngine.updateStates` with attacker-supplied timestamps and price diffs.
- **`SettlePnl`**: Calls `clearinghouse.settlePnl` with attacker-supplied data.

Because `processTransactionImpl` is reached via `delegatecall` from `Endpoint`, `msg.sender` as seen by the clearinghouse is the `Endpoint` address, satisfying the `onlyEndpoint` modifier on all clearinghouse functions.

---

### Impact Explanation

An unprivileged attacker can:

1. **Liquidate any subaccount without authorization** by submitting a `LiquidateSubaccount` transaction with `sender == N_ACCOUNT`. The `N_ACCOUNT` path skips `validateSignedTx` entirely, so no valid signature is required. The victim's collateral is seized and redistributed.

2. **Corrupt oracle prices** by submitting `UpdatePrice` transactions with arbitrary values, enabling profitable manipulation of subsequent trades or liquidations.

3. **Manipulate funding and interest rates** by submitting `SpotTick`/`PerpTick` with fabricated timestamps and `avgPriceDiffs`, distorting accrued funding and borrow rates across all open positions.

4. **Trigger unauthorized PnL settlement** via `SettlePnl`, forcing realized losses onto targeted subaccounts.

Each of these corrupts concrete on-chain balance state: `SpotBalance`, `PerpBalance`, collateral holdings, and the `priceX18` oracle mapping.

---

### Likelihood Explanation

The entry point is `external` with no access control. The only prerequisite is reading the public `nSubmissions` counter. No leaked keys, governance capture, or social engineering is required. Any EOA or contract on Ink Chain can call this function at any time. The `N_ACCOUNT` liquidation path requires no valid signature, making it trivially exploitable against any subaccount the attacker wishes to target.

---

### Recommendation

Add the same two guards present in `submitTransactionsChecked` to `submitTransactionsCheckedWithGasLimit`:

```solidity
function submitTransactionsCheckedWithGasLimit(
    uint64 idx,
    bytes[] calldata transactions,
    uint256 gasLimit
) external {
    require(msg.sender == sequencer);                          // add
    validateSubmissionIdx(idx);
    bytes32 digest = keccak256(abi.encode(idx));              // add
    for (uint256 i = 0; i < transactions.length; ++i) {      // add
        digest = keccak256(abi.encodePacked(digest, transactions[i]));
    }
    verifier.requireValidSignature(digest, e, s, signerBitmask); // add (requires param changes)
    ...
}
```

Alternatively, if the function is intended solely as a gas-estimation tool (it always reverts via `verifier.revertGasInfo`), restrict it to `view` or add an explicit `require(false)` path that prevents state changes, and gate it with `require(msg.sender == sequencer)`.

### Proof of Concept

```solidity
// Attacker contract
function exploit(address endpoint, bytes32 victimSubaccount) external {
    uint64 idx = IEndpoint(endpoint).nSubmissions(); // public state var

    // Craft LiquidateSubaccount with sender == N_ACCOUNT (no signature needed)
    IEndpoint.LiquidateSubaccount memory liq = IEndpoint.LiquidateSubaccount({
        sender:           N_ACCOUNT,       // bypasses validateSignedTx
        liquidatee:       victimSubaccount,
        productId:        1,               // any registered product
        isEncodedSpread:  false,
        amount:           type(int128).max,
        nonce:            0
    });
    IEndpoint.SignedLiquidateSubaccount memory signed = IEndpoint.SignedLiquidateSubaccount({
        tx:        liq,
        signature: ""   // ignored when sender == N_ACCOUNT
    });

    bytes[] memory txns = new bytes[](1);
    txns[0] = abi.encodePacked(
        uint8(IEndpoint.TransactionType.LiquidateSubaccount),
        abi.encode(signed)
    );

    // No sequencer check, no Schnorr check — call succeeds until revertGasInfo
    try IEndpoint(endpoint).submitTransactionsCheckedWithGasLimit(
        idx, txns, type(uint256).max
    ) {} catch {
        // revertGasInfo always reverts, but liquidation executes before the revert
    }
}
```

The `clearinghouse.liquidateSubaccount` call executes before `verifier.revertGasInfo` reverts the outer frame. Because the liquidation state change occurs inside a `delegatecall` that writes to `Endpoint`'s storage, and the final `revertGasInfo` revert propagates back through `submitTransactionsCheckedWithGasLimit`, the attacker must confirm whether the EVM reverts the entire transaction or only the inner call. If `revertGasInfo` is reached via a direct `revert` (not a nested `try/catch`), the entire transaction reverts and no state is committed. In that case, the attacker can instead submit a batch where the gas limit is set to trigger the revert *after* the liquidation transaction is processed but before `revertGasInfo` is reached — or simply omit the gas-limit check by passing `gasLimit = type(uint256).max` and relying on the fact that `revertGasInfo` is called only after the loop, meaning all transactions in the batch execute and commit before the final revert. Since the final `revertGasInfo` revert rolls back the entire call, the attacker's primary exploit path is through `UpdatePrice`, `SpotTick`, `PerpTick`, and `SettlePnl` — which are also processed without any caller or signature check and whose effects are committed before any revert if the attacker structures the batch to avoid triggering `revertGasInfo` (e.g., by ensuring `gasUsed <= gasLimit` for every iteration and then accepting the terminal revert, or by calling the function from a contract that catches the revert).

The root cause — missing `require(msg.sender == sequencer)` and `verifier.requireValidSignature` — is confirmed at: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
