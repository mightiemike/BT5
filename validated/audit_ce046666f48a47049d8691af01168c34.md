Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Gates on Router Address Instead of Ultimate Swapper, Enabling Full Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is `msg.sender` of `MetricOmmPool.swap` — the direct caller of the pool, not the end user. When swaps are routed through `MetricOmmSimpleRouter`, `sender` is the router contract address. If the pool admin allowlists the router (a necessary step to permit router-mediated swaps for legitimate users), every unprivileged user can bypass the per-pool swap allowlist by calling the router, defeating the access control entirely.

## Finding Description
**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

```solidity
// metric-core/contracts/MetricOmmPool.sol line 230-240
_beforeSwap(
    msg.sender,   // ← router address when called via MetricOmmSimpleRouter
    recipient,
    ...
);
```

**`ExtensionCalling._beforeSwap` forwards it unchanged:**

```solidity
// metric-core/contracts/ExtensionCalling.sol line 160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

**`SwapAllowlistExtension.beforeSwap` checks that forwarded address:**

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol line 37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
// msg.sender = pool, sender = router (not the end user)
```

**Exploit flow:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
2. Pool admin allowlists the router address (`allowedSwapper[pool][router] = true`) so that legitimate users can use the router for multi-hop or slippage-protected swaps.
3. Any non-allowlisted attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
4. The router calls `pool.swap(...)` — `msg.sender` to the pool is the router.
5. The pool calls `extension.beforeSwap(router, ...)` — the extension sees `sender = router`, which is allowlisted, and passes.
6. The attacker's swap executes on the restricted pool with no allowlist enforcement.

**Existing guards are insufficient:** The `SwapAllowlistExtension` has no mechanism to unwrap the router intermediary and inspect the original `msg.sender`. The `MetricOmmSimpleRouter` does not forward the end user's identity to the pool or extension.

## Impact Explanation
The swap allowlist is a core access-control mechanism. Bypassing it allows any unprivileged user to trade on pools intended to be restricted (e.g., institutional-only, KYC-gated, or compliance-restricted pools). This constitutes broken core pool functionality and an admin-boundary break: the pool admin's configured access control is circumvented by an unprivileged path through a public router contract. Unauthorized swaps can drain pool liquidity reserved for specific counterparties or violate regulatory/compliance constraints enforced by the allowlist.

## Likelihood Explanation
The bypass requires the router to be allowlisted on the pool. This is a realistic and expected operational step: any pool admin who wants allowlisted users to benefit from router features (multi-hop, slippage protection, deadline checks) must allowlist the router. Once the router is allowlisted, the bypass is trivially repeatable by any user with no special privileges, capital requirements, or timing constraints.

## Recommendation
The extension must check the ultimate end user, not the intermediate caller. Two viable approaches:

1. **Pass end-user identity via `extensionData`:** Have the router encode `msg.sender` (the end user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it — combined with a check that `sender` (the pool's direct caller) is a trusted router registered in the factory or extension config.

2. **Check `recipient` instead of `sender`:** If the pool's design guarantees that `recipient` is always the end user (which it is for `exactInputSingle`), the extension could gate on `recipient`. However, this breaks for multi-hop paths where intermediate recipients are the router itself.

The cleanest fix is approach (1): require the router to attest the real user in `extensionData`, and have the extension verify both that the caller is a trusted router and that the attested user is allowlisted.
