### Title
Missing `ENABLE_SECP_OPS` Guard on 4-Byte Opcode Path Allows Secp Ops Without Flag — (`src/chia_dialect.rs`)

### Summary
`ChiaDialect::op` dispatches `op_secp256k1_verify` (0x13d61f00) and `op_secp256r1_verify` (0x1c3a8f00) via the 4-byte opcode branch unconditionally, while the 1-byte opcode branch (opcodes 64 and 65) requires `ENABLE_SECP_OPS`. A node without `ENABLE_SECP_OPS` rejects the 1-byte form but silently executes the 4-byte form, producing a consensus split.

### Finding Description

In `ChiaDialect::op`, the 4-byte branch matches the secp opcodes and calls the operator functions directly: [1](#0-0) 

No `ENABLE_SECP_OPS` check is present. By contrast, the 1-byte branch is properly guarded: [2](#0-1) 

Neither `op_secp256k1_verify` nor `op_secp256r1_verify` performs an internal flag check — both accept `_flags` (unused): [3](#0-2) [4](#0-3) 

There is no secondary guard anywhere in the call chain.

### Impact Explanation

Two nodes that differ only in whether `ENABLE_SECP_OPS` is set will reach different outcomes for the same CLVM program if that program uses the 4-byte opcode encoding:

- Node A (`ENABLE_SECP_OPS` not set): 1-byte opcode 64 → `unknown_operator`; 4-byte opcode 0x13d61f00 → **executes and succeeds**.
- Node B (`ENABLE_SECP_OPS` set): both forms execute and succeed.

This is a consensus split: the two nodes disagree on whether a spend is valid, which is a Critical impact for a blockchain VM.

### Likelihood Explanation

The 4-byte encoding is valid CLVM serialization. Any attacker who knows the opcode values (they are documented in the comment at line 172–173) can craft a program using the 4-byte form. No special privileges are required — only the ability to submit a CLVM program. [5](#0-4) 

### Recommendation

Add the `ENABLE_SECP_OPS` guard to the 4-byte branch, mirroring the 1-byte branch:

```rust
0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

Without this, the 4-byte path must fall through to `unknown_operator` when the flag is absent, exactly as the 1-byte path does.

### Proof of Concept

```rust
// Without ENABLE_SECP_OPS:
let dialect = ChiaDialect::new(ClvmFlags::empty());

// 1-byte opcode 64 → unknown_operator (rejected or no-op)
// 4-byte opcode 0x13d61f00 → op_secp256k1_verify executes → succeeds or returns crypto error
// The two paths produce different results for the same logical operator.
```

A concrete test: run a program `(0x13d61f00 pubkey msg sig)` with valid secp256k1 inputs against a `ChiaDialect` with no flags set. It will return `Ok(Reduction(1300000, nil))`. Run the same program with opcode `\x40` (1-byte 64) — it will return an unknown-operator result. The divergence is directly observable. [6](#0-5)
