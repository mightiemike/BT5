### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` — No Claim/Distribution Mechanism Exists - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool` collects real ERC20 tokens as fast-withdrawal fees and tracks them in a `fees[productId]` mapping, but no function exists anywhere in the contract (or its concrete child `WithdrawPool`) to distribute or claim those accumulated fees to any recipient. The accounting variable is a dead write: it grows monotonically and is never decremented.

---

### Finding Description

In `submitFastWithdrawal()`, a fee is computed and collected in two ways depending on the caller:

- **Self-withdrawal** (`sendTo == msg.sender`): `transferAmount` is reduced by `fee`, so the fee portion of the ERC20 tokens already held by the contract simply stays there.
- **Third-party withdrawal** (`sendTo != msg.sender`): `safeTransferFrom(token, msg.sender, uint128(fee))` pulls the fee directly from the caller into the contract.

In both branches the accounting line `fees[productId] += fee` is executed. [1](#0-0) [2](#0-1) 

After this point, neither `BaseWithdrawPool` nor `WithdrawPool` contains any function that reads `fees[productId]` for the purpose of transferring those tokens out to a protocol treasury, fee recipient, or any other address. The only token-exit path is `removeLiquidity()`, which is `onlyOwner`, does not reference `fees` at all, and is semantically a liquidity-management escape hatch rather than a fee-distribution mechanism. [3](#0-2) [4](#0-3) 

The `fees` mapping is therefore dead accounting: it is written on every fast withdrawal but never read for any distribution logic.

---

### Impact Explanation

Every fast withdrawal that goes through `submitFastWithdrawal()` causes real ERC20 tokens to accumulate inside `WithdrawPool` with no on-chain path to recover them as protocol revenue. The fee rate is non-trivial: `fastWithdrawalFeeAmount` takes the maximum of a proportional rate (`FAST_WITHDRAWAL_FEE_RATE`) and five times the per-product `withdrawFeeX18`. [5](#0-4) 

Over time, the sum of all fast-withdrawal fees is permanently locked in the contract. The protocol loses all intended fee revenue from this path. This is a direct asset-accounting corruption: the `fees` mapping records a liability that can never be settled.

---

### Likelihood Explanation

`submitFastWithdrawal()` is a public, permissionless function callable by any address that holds a valid sequencer-signed withdrawal transaction. Every invocation silently locks fee tokens. No special role, governance action, or unusual condition is required to trigger the accumulation. The function is a core fast-path for users who do not want to wait for the sequencer to process their withdrawal, so it is expected to be called frequently in normal protocol operation. [6](#0-5) 

---

### Recommendation

Add a privileged `claimFees(uint32 productId, address recipient)` function (or equivalent) to `BaseWithdrawPool` that:

1. Reads `fees[productId]`.
2. Resets `fees[productId]` to zero.
3. Calls `handleWithdrawTransfer(getToken(productId), recipient, uint128(fees[productId]))` to send the accumulated ERC20 tokens to the designated protocol treasury.

This mirrors the pattern used by `dumpFees()` in `OffchainExchange`, which correctly sweeps `collectedFees` to `X_ACCOUNT`. [7](#0-6) 

---

### Proof of Concept

1. Liquidity provider deposits 1 000 000 USDC into `WithdrawPool` via `removeLiquidity`'s inverse (direct transfer or owner deposit).
2. User calls `submitFastWithdrawal(idx, tx, sigs)` for a 100 000 USDC withdrawal where `sendTo == msg.sender`.
   - `fee = fastWithdrawalFeeAmount(...)` returns, say, 50 USDC.
   - `transferAmount` becomes 99 950 USDC; user receives 99 950 USDC.
   - `fees[productId] += 50` — 50 USDC stays in the contract.
3. Repeat for N users. `fees[productId]` grows to `N * 50` USDC.
4. Inspect every function in `BaseWithdrawPool` and `WithdrawPool`: none reads `fees[productId]` to transfer tokens out.
5. The only exit is `removeLiquidity()` (owner-only, no `fees` accounting update), confirming the fee tokens are permanently locked with no proper distribution path. [8](#0-7)

```text
    function fastWithdrawalFeeAmount(
        IERC20Base token,
        uint32 productId,
        uint128 amount
    ) public view returns (int128) {
        uint8 decimals = token.decimals();
        require(decimals <= MAX_DECIMALS);
        int256 multiplier = int256(10**(MAX_DECIMALS - uint8(decimals)));
        int128 amountX18 = int128(amount) * int128(multiplier);

        int128 proportionalFeeX18 = FAST_WITHDRAWAL_FEE_RATE.mul(amountX18);
        int128 minFeeX18 = 5 * spotEngine().getConfig(productId).withdrawFeeX18;

        int128 feeX18 = MathHelper.max(proportionalFeeX18, minFeeX18);
        return feeX18 / int128(multiplier);
    }
```
