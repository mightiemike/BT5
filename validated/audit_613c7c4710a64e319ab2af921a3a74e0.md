### Title
Protobuf `l1_data_gas=absent` Silently Downcasts `AllResources` to `L1Gas`, Producing Wrong Transaction Hash via SQMR Sync Path ‚ÄĒ (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` silently converts an `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` transaction into `ValidResourceBounds::L1Gas(l1_gas=X)` whenever the proto3 optional field `l1_data_gas` is absent. Because `get_tip_resource_bounds_hash` includes the `L1_DATA_GAS` packed felt in the poseidon preimage only for the `AllResources` variant, the two variants produce distinct hashes. The SQMR sync client stores the peer-supplied `transaction_hash` verbatim without recomputing it, so a malicious peer can store a transaction body whose hash does not match the stored hash, causing the RPC to serve an authoritative-looking wrong value.

---

### Finding Description

**Step 1 ‚ÄĒ The conversion** [1](#0-0) 

```
// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
let l1_data_gas = value.l1_data_gas.unwrap_or_default();   // absent ‚Üí zero
...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ‚Üź wrong variant
} else {
    ValidResourceBounds::AllResources(...)
})
```

When a peer omits `l1_data_gas` (proto3 optional, field 2) and sends `l2_gas=zero`, the converter unconditionally produces `L1Gas` regardless of whether the original transaction was an `AllResources` Starknet ‚Č• 0.13.3 transaction.

**Step 2 ‚ÄĒ The hash divergence** [2](#0-1) 

`get_tip_resource_bounds_hash` appends the `L1_DATA_GAS` packed felt to the poseidon preimage **only** for `AllResources`:

```
ValidResourceBounds::L1Gas(_) => vec![],          // L1_DATA_GAS omitted
ValidResourceBounds::AllResources(all_resources) =>
    vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
```

Therefore `hash(AllResources(X,0,0)) ‚Č† hash(L1Gas(X))` ‚ÄĒ the preimage differs by one felt.

**Step 3 ‚ÄĒ The sync client stores the hash without validation** [3](#0-2) 

```rust
block_body.transactions.push(transaction);
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);   // peer-supplied, unverified
```

The `transaction_hash` field is read directly from the protobuf: [4](#0-3) 

and stored verbatim. No recomputation from the deserialized body occurs.

**Step 4 ‚ÄĒ The serializer always emits `l1_data_gas=Some(zero)` for `L1Gas`** [5](#0-4) 

A legitimate node always sets `l1_data_gas = Some(zero)` when serializing `L1Gas`, so the absent-field path is only reachable from a peer that deliberately omits the field.

---

### Impact Explanation

A malicious peer serving the SQMR transaction sync stream can:

1. Take a legitimate on-chain `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` transaction whose canonical hash is `H_AllResources`.
2. Re-encode it with `l1_data_gas` absent in the protobuf, keeping `transaction_hash = H_AllResources`.
3. The sync client deserializes the body as `L1Gas(l1_gas=X)` and stores `(body=L1Gas, hash=H_AllResources)`.
4. Block-hash validation passes because it is computed over the stored transaction hashes (which are correct), not over the transaction bodies.
5. `starknet_getTransactionByHash(H_AllResources)` returns a transaction body with `L1Gas` resource bounds ‚ÄĒ the wrong variant. Any client that recomputes the hash from the returned body gets `H_L1Gas ‚Č† H_AllResources`.

This fits the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload"** and **"High. RPC execution ‚Ä¶ returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

- Any unprivileged peer in the P2P network can connect and serve the SQMR transaction stream.
- The only precondition is that a real `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` transaction exists on-chain, which is valid per the protocol.
- The TODO comment explicitly acknowledges the missing hash validation, confirming the gap is known but unguarded.

---

### Recommendation

1. **Short-term**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, treat `l1_data_gas=absent` as `AllResources` (not `L1Gas`) when the Starknet version is ‚Č• 0.13.3, or reject the message entirely once 0.13.2 support is dropped (as the TODO already notes).
2. **Short-term**: In the sync client's `parse_data_for_block`, recompute `get_transaction_hash` from the deserialized transaction body and compare it against the peer-supplied hash; disconnect the peer on mismatch (the TODO at line 88 should be resolved).
3. **Long-term**: The `unwrap_or_default()` fallback for `l1_data_gas` should be removed once 0.13.2 backward-compatibility is no longer required, as the TODO at line 426 already states.

---

### Proof of Concept

```rust
// Demonstrates the hash divergence
use starknet_api::transaction::fields::{
    AllResourceBounds, ResourceBounds, ValidResourceBounds,
};
use starknet_api::transaction_hash::get_tip_resource_bounds_hash;
use starknet_api::transaction::fields::Tip;
use starknet_api::execution_resources::GasAmount;
use starknet_api::block::GasPrice;

let l1_gas = ResourceBounds { max_amount: GasAmount(100), max_price_per_unit: GasPrice(1) };
let zero   = ResourceBounds::default();

// Path A: sender serialized as AllResources
let all = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas, l2_gas: zero, l1_data_gas: zero,
});
let hash_all = get_tip_resource_bounds_hash(&all, &Tip(0)).unwrap();

// Path B: receiver deserialized with l1_data_gas=absent ‚Üí L1Gas
let l1  = ValidResourceBounds::L1Gas(l1_gas);
let hash_l1  = get_tip_resource_bounds_hash(&l1, &Tip(0)).unwrap();

assert_ne!(hash_all, hash_l1, "hashes must differ ‚ÄĒ this is the bug");
```

The assert passes, confirming the canonicalization invariant is violated by the protobuf conversion.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
```
