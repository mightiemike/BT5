### Title
`op_modpow` (opcode 60) unconditionally blocked in mempool mode via `DISABLE_OP` flag, creating mempool-consensus divergence — (`File: src/chia_dialect.rs`)

---

### Summary

The `MEMPOOL_MODE` constant in `src/chia_dialect.rs` includes the `DISABLE_OP` flag, which causes `op_modpow` (opcode 60) to be rejected with `Unimplemented` whenever the mempool evaluates a CLVM program. Consensus mode does not set this flag, so the same program executes successfully there. Any coin whose puzzle uses `modpow` is therefore unspendable through the mempool — an overly strict gate that mirrors the original report's "only the host can trigger delivery" pattern, with the mempool acting as the single point of failure.

---

### Finding Description

In `src/chia_dialect.rs`, `MEMPOOL_MODE` is assembled as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)          // ← includes DISABLE_OP
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
```

Inside `ChiaDialect::op()`, the only place `DISABLE_OP` has any effect is the dispatch arm for opcode 60:

```rust
60 => {
    if flags.contains(ClvmFlags::DISABLE_OP) {
        return Err(EvalErr::Unimplemented(o))?;   // mempool path
    }
    op_modpow                                      // consensus path
}
```

No other opcode arm checks `DISABLE_OP`. The flag name is generic and carries no documentation explaining why `modpow` specifically must be suppressed in mempool mode. Meanwhile, opcode 60 appears in the `gc_candidate` match arm alongside every other production operator, confirming it is treated as a first-class consensus operator:

```rust
NodeVisitor::U32(
    ... | 59 | 60 | 61 | 62 | 63,
) => true,
```

The Python wheel exports both `DISABLE_OP` and `MEMPOOL_MODE` as public constants, so every Python caller that passes `MEMPOOL_MODE` (the documented "stricter mode" for mempool validation) silently inherits the `modpow` ban.

---

### Impact Explanation

A CLVM coin whose puzzle contains `modpow` (opcode 60) will be rejected by any node running mempool validation (`MEMPOOL_MODE`) with error `Unimplemented`. The same spend is accepted by consensus (which does not set `DISABLE_OP`). Concretely:

- **Mempool-consensus divergence**: the mempool refuses a spend that consensus would commit. Nodes that validate blocks in consensus mode will accept a block containing such a spend, but the mempool will never propagate it.
- **Coin lock via mempool**: the coin cannot be spent through the normal peer-to-peer mempool path. Only a farmer who constructs a block directly (bypassing mempool validation) can include the spend — an unreliable and non-standard path for ordinary users.
- **Attacker-crafted lock**: an adversary can deliberately create a coin whose puzzle uses `modpow`, knowing that the owner will be unable to spend it through the mempool, effectively locking the coin from the perspective of every standard full node.

---

### Likelihood Explanation

The entry path requires only attacker-controlled CLVM bytes. Any program that encodes opcode 60 (`0x3c`) as an operator atom and is submitted to a node running `MEMPOOL_MODE` triggers the rejection. The `run_serialized_chia_program` Python binding accepts raw bytes and a caller-supplied flags integer; passing `MEMPOOL_MODE` (the documented production value) is the normal usage. No special privileges or configuration are needed.

---

### Recommendation

Either:

1. **Remove `DISABLE_OP` from `MEMPOOL_MODE`** if `modpow` is a fully activated consensus operator (which the `gc_candidate` list and the consensus dispatch arm both imply it is). The mempool should not reject programs that consensus accepts.
2. **If the suppression is intentional** (e.g., `modpow` is pending a hard-fork activation), rename the flag to something self-documenting (e.g., `MEMPOOL_DISABLE_MODPOW`), add a comment explaining the activation timeline, and ensure the `gc_candidate` list is updated to exclude opcode 60 until activation.

---

### Proof of Concept

```
# CLVM program: (modpow base exp mod) — opcode 60 = 0x3c
# Serialized minimal form: (60 . args) where 60 is atom 0x3c

# In mempool mode (DISABLE_OP set via MEMPOOL_MODE):
run_serialized_chia_program(program_bytes, args_bytes, max_cost, MEMPOOL_MODE)
# → EvalErr::Unimplemented  (rejected)

# In consensus mode (DISABLE_OP absent):
run_serialized_chia_program(program_bytes, args_bytes, max_cost, 0)
# → Ok(cost, result)  (accepted)
```

The divergence is triggered by any attacker-supplied CLVM blob that uses opcode `0x3c` as an operator. The root cause is the unconditional inclusion of `DISABLE_OP` in `MEMPOOL_MODE` at `src/chia_dialect.rs` line 72–76, combined with the exclusive `DISABLE_OP` check at line 240–242.
