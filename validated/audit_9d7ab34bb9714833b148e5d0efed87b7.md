### Title
Missing Zero-Value Guard on `confirmations` Bypasses Confirmation-Depth Check — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` accept `confirmations = 0` without rejection. When zero is supplied, the only confirmation-depth guard in the function is trivially satisfied for every block in the mainchain, making the function return `true` for a transaction that has zero blocks built on top of it — i.e., a block that could still be reorganized away.

---

### Finding Description

`verify_transaction_inclusion` performs two checks before computing the Merkle proof:

**Check 1** — upper-bound guard: [1](#0-0) 

This passes for `confirmations = 0` because `0 <= gc_threshold` is always true.

**Check 2** — depth guard: [2](#0-1) 

The left-hand side evaluates to `(tip_height - target_height) + 1`, which is at minimum `1` (when the target block is the tip). With `confirmations = 0`, the condition becomes `1 >= 0`, which is unconditionally true for every block in the mainchain. The depth check is completely bypassed.

There is no lower-bound guard — no `require!(args.confirmations >= 1, ...)` — anywhere in either `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`. [3](#0-2) 

`verify_transaction_inclusion_v2` delegates to `verify_transaction_inclusion` after its coinbase-proof check, so it inherits the same flaw: [4](#0-3) 

The `confirmations` field is a plain `u64` in `ProofArgs` / `ProofArgsV2` with no enforced minimum: [5](#0-4) 

---

### Impact Explanation

The `confirmations` parameter exists precisely to enforce finality: a recipient contract is supposed to be able to demand that a Bitcoin transaction is buried under at least N blocks before it releases funds or updates state. With `confirmations = 0` accepted silently, the entire finality guarantee collapses. A transaction in a block that was submitted in the same NEAR transaction — or one block ago — passes verification identically to one buried under 100 blocks. Any downstream contract that calls `verify_transaction_inclusion` and trusts the boolean result is exposed to double-spend via a Bitcoin reorg: the attacker submits a valid block, immediately calls the verifier with `confirmations = 0`, receives `true`, triggers the recipient contract's payout, and the Bitcoin block is later reorganized away.

---

### Likelihood Explanation

The entry path requires no privilege. `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are `pub` functions gated only by `#[pause]`; any unprivileged NEAR account can call them. Supplying `confirmations = 0` is a single-field change to the `ProofArgs` struct. The attacker does not need to forge PoW or manipulate contract roles — they only need a real Bitcoin transaction in any block already accepted by the contract.

---

### Recommendation

Add an explicit lower-bound check at the top of `verify_transaction_inclusion`:

```rust
require!(args.confirmations >= 1, "confirmations must be at least 1");
```

This mirrors the upper-bound check already present and closes the zero-value bypass symmetrically. The same guard should be applied before the delegation call in `verify_transaction_inclusion_v2`, or enforced once in the shared path.

---

### Proof of Concept

1. Deploy the contract with any valid genesis block and `gc_threshold > 0`.
2. Submit one valid Bitcoin block header via `submit_blocks`.
3. Call `verify_transaction_inclusion` with:
   - `tx_block_blockhash` = hash of the block just submitted (tip of mainchain)
   - `confirmations = 0`
   - a valid Merkle proof for a transaction in that block
4. **Observed**: the function returns `true`.
5. **Expected**: the function should reject the call because `confirmations = 0` provides no finality guarantee.

The depth check at line 304–308 evaluates to `(tip_height - tip_height) + 1 >= 0` → `1 >= 0` → `true`, so the guard is bypassed and the Merkle proof computation proceeds normally. [2](#0-1)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
