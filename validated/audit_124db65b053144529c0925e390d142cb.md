### Title
Unsigned `feeX18` Field in `WithdrawCollateralV2` Allows Sequencer to Charge Maximum Fee Without User Consent — (`core/contracts/EndpointTx.sol`, `core/contracts/Verifier.sol`)

---

### Summary

The `SignedWithdrawCollateralV2` struct contains a `feeX18` field that is **not included in the EIP-712 digest** the user signs. The sequencer constructs the full `SignedWithdrawCollateralV2` payload and can freely set `feeX18` to any value up to `currentFeeX18`, charging the user the maximum configured withdrawal fee without their knowledge or consent.

---

### Finding Description

`SignedWithdrawCollateralV2` is defined as:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;          // ← outside the signed struct
}
``` [1](#0-0) 

The EIP-712 type string and digest computation in `Verifier.computeDigest` cover only the inner `WithdrawCollateralV2` fields (`sender`, `productId`, `amount`, `nonce`, `sendTo`, `appendix`):

```solidity
string internal constant WITHDRAW_COLLATERAL_V2_SIGNATURE =
    "WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,uint64 nonce,address sendTo,uint128 appendix)";
``` [2](#0-1) 

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(WITHDRAW_COLLATERAL_V2_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.productId,
        signedTx.tx.amount,
        signedTx.tx.nonce,
        signedTx.tx.sendTo,
        signedTx.tx.appendix
        // feeX18 is absent
    )
);
``` [3](#0-2) 

After signature validation passes, `processTransactionImpl` enforces only a range check on `feeX18` and then charges it directly:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [4](#0-3) 

The user's signature covers the withdrawal parameters but **not the fee they will be charged**. The sequencer, which constructs and submits the `SignedWithdrawCollateralV2` payload via `submitTransactions`, can set `feeX18` to `currentFeeX18` on every transaction.

This is structurally identical to the reported Solana bug: a binding data structure (`receipt` PDA / EIP-712 digest) is missing a key context field (`sale` key / `feeX18`), allowing a privileged actor to substitute a different value for that field without invalidating the user's authorization.

---

### Impact Explanation

The sequencer can silently charge every `WithdrawCollateralV2` caller the maximum configured withdrawal fee (`currentFeeX18`) regardless of any off-chain fee agreement or user expectation. The fee is debited from the user's subaccount balance via `chargeFee → spotEngine.updateBalance`, constituting a direct, concrete asset delta against the user. There is no on-chain mechanism for the user to commit to or verify the fee before their signature is consumed.

---

### Likelihood Explanation

`WithdrawCollateralV2` (`TransactionType` enum value 32) is the current withdrawal path and is not processable via the slow-mode queue (`processSlowModeTransactionImpl` handles only `WithdrawCollateral` V1). Every V2 withdrawal must pass through the sequencer, which sets `feeX18` unconditionally. A sequencer operator acting in self-interest can maximize fee extraction on every withdrawal with zero additional effort.

---

### Recommendation

Include `feeX18` in the EIP-712 type string and digest so the user explicitly authorizes the fee amount they will be charged:

```solidity
string internal constant WITHDRAW_COLLATERAL_V2_SIGNATURE =
    "WithdrawCollateralV2(bytes32 sender,uint32 productId,uint128 amount,"
    "uint64 nonce,address sendTo,uint128 appendix,int128 feeX18)";
```

Move `feeX18` into the `WithdrawCollateralV2` inner struct so it is covered by the `CompactSignature`. The sequencer should present the fee to the user before signing, and the contract should reject any transaction where the on-chain fee exceeds the signed value.

---

### Proof of Concept

1. User signs `WithdrawCollateralV2{sender=U, productId=P, amount=A, nonce=N, sendTo=0, appendix=0}` — `feeX18` is not part of this signed payload.
2. Sequencer receives the signed order and constructs `SignedWithdrawCollateralV2{tx=<above>, signature=<user sig>, feeX18=currentFeeX18}`.
3. Sequencer submits the batch via `Endpoint.submitTransactions`.
4. `processTransactionImpl` calls `validateSignedTx` — signature is valid because `feeX18` is absent from the digest.
5. `require(signedTx.feeX18 <= currentFeeX18)` passes.
6. `chargeFee(U, currentFeeX18, P)` debits the maximum fee from user `U`'s spot balance.
7. User receives `A` tokens but loses `currentFeeX18` in fees, which they never agreed to on-chain. [5](#0-4) [6](#0-5)
