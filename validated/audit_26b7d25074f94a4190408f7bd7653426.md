### Title
Host-function gas-key fee path passes wire-encoded key length (1953 B) instead of trie-id length (33 B) for ML-DSA-65 keys, overcharging by ~59× — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

When a WASM contract calls `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, or `promise_batch_action_add_gas_key_with_function_call` with an ML-DSA-65 public key, the host-function path forwards the raw `public_key_len` argument (the borsh wire length, 1953 bytes for ML-DSA-65) to `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. The transaction path for the same actions calls those same fee helpers with `public_key.trie_id_len()` (33 bytes for ML-DSA-65). The two representations diverge by a factor of ≈59×, producing a systematic gas-fee overcharge for every contract-initiated ML-DSA-65 gas-key operation.

---

### Finding Description

`PublicKey::trie_id_len()` and `PublicKey::len()` return identical values for ED25519 (33 B) and SECP256K1 (65 B), but diverge for ML-DSA-65:

- `len()` → 1 + 1952 = **1953 bytes** (borsh wire form)
- `trie_id_len()` → 1 + 32 = **33 bytes** (SHA3-256 hash stored in the trie)

The design document explicitly states: *"Every storage-stake and trie-byte-priced fee path was updated to call `trie_id_len()`."*

**Transaction path** (`runtime/runtime/src/config.rs`) correctly uses `trie_id_len()`:

```rust
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
        .total()
}
// ...
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [1](#0-0) [2](#0-1) 

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`) passes the raw `public_key_len` argument from WASM memory instead:

```rust
let send = gas_key_transfer_send_fee(&self.fees_config, sir, public_key_len as usize);
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie-id length

``` [3](#0-2) 

The same pattern appears in `promise_batch_action_add_gas_key_with_full_access` and `promise_batch_action_add_gas_key_with_function_call`:

```rust

    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← wire length, not trie-id length
    num_nonces,
);
``` [4](#0-3) 

The identical bug is present in the wasmtime backend: [5](#0-4) [6](#0-5) 

The fee functions themselves are correct — they are designed to receive the trie-id length:

```rust

    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,          // expected: trie_id_len()

    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    // per_byte = gas_key_byte * (trie_key_len + value_len)
``` [7](#0-6) 

When `public_key_len` is 1953 instead of 33, `trie_key_len` is inflated by 1920 bytes, and every `gas_key_byte` unit is multiplied by that inflated length.

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys, where `len() == trie_id_len()`, so the divergence is invisible to the test suite: [8](#0-7) 

---

### Impact Explanation

For every contract-initiated ML-DSA-65 gas-key action (fund, add, withdraw), the `gas_key_byte`-priced component of the fee is computed against 1953 bytes instead of 33 bytes — a ≈59× overcharge on that component. Concretely:

- `gas_key_transfer_exec_fee`: `trie_key_len` is inflated by 1920 bytes per call.
- `gas_key_add_key_exec_fee`: `nonce_key_len` is inflated by 1920 bytes, multiplied by `num_nonces`.
- `gas_key_transfer_send_fee`: send-side per-byte fee is multiplied by 1953 instead of 33.

Contracts that budget gas for these operations based on the documented trie-id semantics (or by observing the transaction-path cost) will run out of gas. The fee parity invariant between the transaction path and the host-function path — explicitly required by the design — is broken for ML-DSA-65 keys. [9](#0-8) 

---

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature is currently nightly-only (not enabled at stable protocol version 86). Once stabilized, any contract that manages ML-DSA-65 gas keys via host functions will trigger the overcharge on every fund/add/withdraw call. The call path is fully unprivileged: any deployed contract can invoke these host functions with a caller-supplied ML-DSA-65 key. [10](#0-9) [11](#0-10) 

---

### Recommendation

In both `runtime/near-vm-runner/src/logic/logic.rs` and `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`, replace `public_key_len as usize` with `public_key.trie_id_len()` (after the key is decoded) when calling `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. The decoded `public_key` is already available at those call sites. Add a fee-parity test that exercises ML-DSA-65 keys alongside the existing ED25519 test.

---

### Proof of Concept

1. Enable the `nightly` feature (which activates `PostQuantumSignatures`).
2. Deploy a contract that calls `promise_batch_action_transfer_to_gas_key` with a 1953-byte ML-DSA-65 borsh-encoded key and records `gas_burnt`.
3. Submit the equivalent `TransferToGasKey` action via a signed transaction with the same ML-DSA-65 key and record `gas_burnt`.
4. Observe that step 2 burns ≈59× more gas in the `gas_key_byte` cost category than step 3, matching the ratio 1953 / 33.

The divergence is exact and deterministic: `public_key_len = 1953` (wire) vs `trie_id_len() = 33` (hash), a difference of 1920 bytes multiplied by `gas_key_byte` per call. [12](#0-11) [13](#0-12)

pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}

impl PublicKey {
    /// Length of this public key's borsh encoding, in bytes - that is,
    /// the on-the-wire size of the raw key bytes plus a 1-byte borsh
    /// discriminant tag (the leading `+ 1` in each arm).
    ///
    /// For storage-fee accounting use [`PublicKey::trie_id_len`] instead;
    /// for ML-DSA-65 those two diverge (1953 wire vs 33 on-trie).
    // `is_empty` always returns false, so there is no point in adding it
    #[allow(clippy::len_without_is_empty)]
    pub fn len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_PUBLIC_KEY_LENGTH,
        }
    }

fn access_key_storage_usage(
    fee_config: &RuntimeFeesConfig,
    public_key: &PublicKey,
    access_key: &AccessKey,
) -> StorageUsage {
    let storage_usage_config = &fee_config.storage_usage_config;
    // Use the on-trie identifier length, not the borsh-serialized pubkey
    // length: ML-DSA-65 access keys live in the trie as a SHA3-256 hash
    // (33 bytes incl. type tag), not as a 1953-byte full pubkey.
    public_key.trie_id_len() as u64
        + borsh::object_length(access_key).unwrap() as u64
        + storage_usage_config.num_extra_bytes_record
}
