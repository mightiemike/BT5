### Title
Gas-key host-function fee computation uses raw wire-length (`public_key_len`) instead of on-trie hash length for ML-DSA-65 keys, causing a ~1920-byte fee overcharge per gas-key operation - (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_withdraw_from_gas_key`) pass the caller-supplied `public_key_len` (the raw Wasm-memory wire length of the borsh-encoded public key) directly into `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee`. For ML-DSA-65 keys this value is **1953** (1 tag byte + 1952 raw key bytes), but the actual on-trie identifier is only **33** bytes (1 tag byte + 32-byte SHA3-256 hash). The runtime's own action-level fee path (`exec_fee` in `runtime/runtime/src/config.rs`) and storage-usage path (`access_key_storage_usage` in `runtime/runtime/src/access_keys.rs`) both correctly call `public_key.trie_id_len()` to get 33. The host-function path never decodes the key before computing the fee, so it permanently uses the wrong (1953-byte) length, overcharging gas by a factor of ~59× on the per-byte component for every gas-key operation involving an ML-DSA-65 key.

---

### Finding Description

**The divergent value.** `PublicKey::trie_id_len()` returns 33 for `MLDSA65` (1 tag + 32-byte SHA3-256 hash). `PublicKey::len()` returns 1953 (1 tag + 1952 raw bytes). The two are identical for ed25519 (33) and secp256k1 (65), so the bug is invisible for classical keys.

**The correct path (action-level).** When a gas-key `AddKey` or `TransferToGasKey` action arrives as a regular transaction, `exec_fee` in `runtime/runtime/src/config.rs` calls `public_key.trie_id_len()`:

```rust
// runtime/runtime/src/config.rs:347-354
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
// config.rs:392
public_key.trie_id_len(),
```

**The broken path (host-function).** When a contract calls the same action via host function, the logic layer passes `public_key_len as usize` — the raw byte count of the borsh blob the contract wrote into Wasm memory — directly to the same fee helpers, without ever decoding the key first:

```rust
// runtime/near-vm-runner/src/logic/logic.rs:3155-3159
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,   // ← raw wire length, not trie_id_len
    num_nonces,
);
```

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call` (line 3229) and `promise_batch_action_transfer_to_gas_key` (line 3095), and is mirrored identically in the Wasmtime runner at `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` lines 3398, 3502, and 3325.

**The fee helper.** `gas_key_add_key_exec_fee` in `core/parameters/src/cost.rs` uses `public_key_len` to compute the nonce trie-key length:

```rust
// core/parameters/src/cost.rs:888-889
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
```

For an ML-DSA-65 key, `public_key_len` from the host-function path is 1953; the correct value is 33. The per-byte exec fee is multiplied by `(nonce_key_len + NONCE_VALUE_LEN) * num_nonces`, so the overcharge scales with both the key size error (~1920 extra bytes) and the number of nonces.

**Attacker-controlled representation.** Any unprivileged user who deploys a contract and calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 key controls `public_key_len`. The contract passes the borsh-encoded key (1953 bytes) to the host function, which uses that length for fee computation. The user cannot reduce the fee below the overcharged amount (the key must be valid to pass `public_key.decode()?`), but the overcharge is deterministic and guaranteed for every ML-DSA-65 gas-key host-function call.

**Parity break.** The existing test `test_gas_key_fee_parity_full_access` / `test_gas_key_fee_parity_function_call` only exercises ed25519 keys (where `len() == trie_id_len()`), so the divergence for ML-DSA-65 is not caught.

---

### Impact Explanation

Any contract that manages ML-DSA-65 gas keys via host functions will be charged ~59× the correct per-byte exec fee on every `AddKey` (gas key) and `TransferToGasKey`/`WithdrawFromGasKey` call. With `num_nonces = 1024` (the maximum), the overcharge on the per-byte component alone is approximately `1920 bytes × 1024 nonces × gas_key_byte_exec_fee`. At the current `gas_key_byte` exec fee this translates to a large excess gas burn that makes ML-DSA-65 gas keys economically unusable from contracts. The fee discrepancy also breaks determinism between the transaction path and the host-function path for the same logical operation, violating the protocol invariant that both paths charge identically (the invariant the `test_gas_key_fee_parity` tests enforce for ed25519).

---

### Likelihood Explanation

`PostQuantumSignatures` is stabilized at protocol version 85 and shipped in release 2.13.0. Any contract that creates or funds ML-DSA-65 gas keys via host functions after that protocol version activates will trigger the overcharge. The bug is reachable by any unprivileged user who deploys such a contract. The classical-key tests pass, so the bug is not caught by the existing test suite.

---

### Recommendation

In each gas-key host function, after decoding the public key (`public_key.decode()?`), use the decoded key's `trie_id_len()` instead of the raw `public_key_len` when calling the fee helpers. Concretely, decode the key before the fee computation and pass `decoded_key.trie_id_len()`:

```rust
let decoded_key = public_key.decode()?;
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_key.trie_id_len(),  // ← correct on-trie length
    num_nonces,
);
```

Apply the same fix to `promise_batch_action_add_gas_key_with_function_call` and `promise_batch_action_transfer_to_gas_key` (and their Wasmtime counterparts). Add a `test_gas_key_fee_parity` variant that exercises ML-DSA-65 keys to prevent regression.

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();

        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
