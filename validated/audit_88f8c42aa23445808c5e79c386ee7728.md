### Title
4-Byte Secp Opcode Aliases Bypass `ENABLE_SECP_OPS` Flag Gate — (`File: src/chia_dialect.rs`)

---

### Summary

The `ChiaDialect::op` function in `src/chia_dialect.rs` dispatches secp signature-verification operators through two separate opcode paths. The 1-byte opcodes (64 = `secp256k1_verify`, 65 = `secp256r1_verify`) are correctly gated behind `ClvmFlags::ENABLE_SECP_OPS`. However, the 4-byte alias opcodes (`0x13d61f00` and `0x1c3a8f00`) dispatch to the same underlying functions **without any flag check**, allowing an attacker-controlled CLVM program to invoke live secp signature verification even when `ENABLE_SECP_OPS` is absent from the dialect flags.

---

### Finding Description

`ChiaDialect::op` handles two opcode lengths before reaching the main 1-byte dispatch table.

**4-byte path (lines 157–182) — no flag check:**

```rust
if op_len == 4 {
    let opcode = u32::from_be_bytes(b.as_ref().try_into().unwrap());
    let f = match opcode {
        0x13d61f00 => op_secp256k1_verify,
        0x1c3a8f00 => op_secp256r1_verify,
        _ => { return unknown_operator(...); }
    };
    return f(allocator, argument_list, max_cost, flags);  // ← no ENABLE_SECP_OPS check
}
``` [1](#0-0) 

**1-byte path (lines 248–249) — correctly gated:**

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

The `ENABLE_SECP_OPS` flag is defined as the exclusive gate for secp opcodes:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [3](#0-2) 

`MEMPOOL_MODE` — the stricter validation context — does **not** include `ENABLE_SECP_OPS`:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [4](#0-3) 

The `extension` parameter passed into `op` only ever injects `ENABLE_KECCAK_OPS_OUTSIDE_GUARD` (for `OperatorSet::Keccak`); it never injects `ENABLE_SECP_OPS`, so no softfork guard path can legitimise the 4-byte opcodes either. [5](#0-4) 

---

### Impact Explanation

When a CLVM program encodes a secp operator using the 4-byte opcode form (`0x13d61f00` or `0x1c3a8f00`), `ChiaDialect::op` executes `op_secp256k1_verify` / `op_secp256r1_verify` unconditionally. Both functions perform real cryptographic verification and return `nil` on success or raise on failure. [6](#0-5) [7](#0-6) 

**Concrete broken invariant — consensus/mempool divergence:**

| Context | 1-byte opcode 64/65 | 4-byte opcode 0x13d61f00/0x1c3a8f00 |
|---|---|---|
| `ENABLE_SECP_OPS` not set (pre-softfork / mempool) | `Unimplemented` error (with `NO_UNKNOWN_OPS`) or `op_unknown` nil (without) | **Executes live secp verification** — returns nil on valid sig, raises on invalid |
| `ENABLE_SECP_OPS` set | Executes live secp verification | Executes live secp verification |

A puzzle that uses the 4-byte encoding will be **accepted and verified** by a node running without `ENABLE_SECP_OPS`, while a node that treats the same 4-byte opcode as a true unknown operator (e.g., an older implementation) would return nil unconditionally. This is a direct consensus-divergence vector: the same spend bundle evaluates to different outcomes on different nodes.

In mempool mode specifically, the 1-byte secp opcodes are rejected (`Unimplemented`), but the 4-byte aliases silently succeed, meaning the mempool's pre-softfork secp exclusion is completely ineffective against the 4-byte encoding.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: any CLVM program submitted to the mempool or evaluated by a consensus node can embed the 4-byte opcode bytes directly. No special privilege, social engineering, or dependency compromise is required. The attacker only needs to know the 4-byte opcode values (`0x13d61f00`, `0x1c3a8f00`), which are derivable from the cost formula documented in the source comment at line 172. The bypass is unconditional — it fires on every invocation regardless of flags. [8](#0-7) 

---

### Recommendation

Add the same `ENABLE_SECP_OPS` guard to the 4-byte dispatch path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => { return unknown_operator(allocator, o, argument_list, flags, max_cost); }
};
```

This mirrors the guard already applied to the 1-byte aliases and ensures both encodings are subject to the same activation-flag invariant.

### Proof of Concept

Construct a CLVM program that uses the 4-byte secp256k1 opcode directly:

```
; opcode bytes: 0x13 0xd6 0x1f 0x00  (secp256k1_verify, 4-byte form)
; Run with ChiaDialect::new(MEMPOOL_MODE)  — ENABLE_SECP_OPS is absent
(0x13d61f00
  (q . <33-byte-pubkey>)
  (q . <32-byte-msg-hash>)
  (q . <64-byte-sig>))
```

Expected (correct) behaviour: `Unimplemented` error, because `ENABLE_SECP_OPS` is not set and `NO_UNKNOWN_OPS` is set in `MEMPOOL_MODE`.

Actual behaviour: `op_secp256k1_verify` executes. With a valid `(pubkey, msg, sig)` triple it returns `nil` and charges 1 300 000 cost; with an invalid triple it raises `Secp256Failed`. In either case the secp gate is bypassed, producing a result that diverges from any node that correctly treats `0x13d61f00` as an unknown operator when `ENABLE_SECP_OPS` is absent.

pub fn op_secp256r1_verify(
    a: &mut Allocator,
    input: NodePtr,
    max_cost: Cost,
    _flags: ClvmFlags,
) -> Response {
    let cost = SECP256R1_VERIFY_COST;
    check_cost(cost, max_cost)?;

    let [pubkey, msg, sig] = get_args::<3>(a, input, "secp256r1_verify")?;

    // first argument is sec1 encoded pubkey
    let pubkey = atom(a, pubkey, "secp256r1_verify pubkey")?;
    let verifier = P1VerifyingKey::from_sec1_bytes(pubkey.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(input, "secp256r1_verify: pubkey is not valid".to_string())
    })?;

    // second arg is sha256 hash of message
    let msg = atom(a, msg, "secp256r1_verify msg")?;
    if msg.as_ref().len() != 32 {
        Err(EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: message digest is not 32 bytes".to_string(),
        ))?;
    }

    // third arg is a fixed-size signature
    let sig = atom(a, sig, "secp256r1_verify sig")?;
    let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| {
        EvalErr::InvalidOpArg(
            input,
            "secp256r1_verify: signature is not valid".to_string(),
        )
    })?;

    // verify signature
    let result = verifier.verify_prehash(msg.as_ref(), &sig);

    if result.is_err() {
        Err(EvalErr::Secp256Failed(input))?
    } else {
        Ok(Reduction(cost, a.nil()))
    }
