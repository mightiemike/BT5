### Title
Missing `sendTo` Recipient Validation in `WithdrawCollateralV2` Allows Permanent Fund Loss — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

The `WithdrawCollateralV2` withdrawal path allows users to specify an arbitrary `sendTo` address. `Clearinghouse.withdrawCollateral` only guards against `sendTo == address(0)` (falling back to the sender), but performs no check against protocol-internal contract addresses such as `address(clearinghouse)` or `address(withdrawPool)`. If a user signs a `WithdrawCollateralV2` transaction with `sendTo` set to one of these addresses, their withdrawn tokens are permanently locked with no user-accessible recovery path.

---

### Finding Description

`WithdrawCollateralV2` includes a user-controlled `sendTo` field: [1](#0-0) 

In `EndpointTx.processTransactionImpl`, this field is passed directly to `clearinghouse.withdrawCollateral` without any additional sanitization: [2](#0-1) 

Inside `Clearinghouse.withdrawCollateral`, the only guard is a fallback for `address(0)`. No check exists for `sendTo == address(this)` (the clearinghouse) or `sendTo == withdrawPool`: [3](#0-2) 

`handleWithdrawTransfer` then forwards the tokens to `withdrawPool` and calls `submitWithdrawal` with the unvalidated `sendTo`: [4](#0-3) 

`BaseWithdrawPool.submitWithdrawal` calls `handleWithdrawTransfer` which executes `token.safeTransfer(to, amount)` with no further validation: [5](#0-4) 

The same gap exists in the fast-withdrawal path. `resolveFastWithdrawal` for `WithdrawCollateralV2` falls back to sender only when `sendTo == address(0)`, but passes any other address through unchanged — including `address(this)` (the `WithdrawPool` itself): [6](#0-5) 

`submitFastWithdrawal` then calls `handleWithdrawTransfer(token, sendTo, transferAmount)` directly: [7](#0-6) 

---

### Impact Explanation

If `sendTo` is set to `address(clearinghouse)`, the token transfer chain ends with `token.safeTransfer(address(clearinghouse), amount)`. The clearinghouse has no mechanism to return arbitrary ERC20 tokens to a user — its accounting system tracks balances internally and the on-chain token balance is already debited from the user's subaccount before the transfer. The tokens are permanently locked.

If `sendTo` is set to `address(withdrawPool)`, the tokens are silently absorbed into the pool's token balance. They are not attributed to any subaccount and are only recoverable by the owner via `removeLiquidity` — not by the affected user.

In both cases, the user's on-chain collateral balance is decremented (`spotEngine.updateBalance` with a negative delta) while the tokens go to an unrecoverable address, resulting in a complete and permanent loss of the withdrawn amount.

---

### Likelihood Explanation

The `sendTo` field is embedded in the EIP-712-signed `WithdrawCollateralV2` message. A user could set it to a protocol contract address due to:

- A copy-paste error (e.g., copying the clearinghouse address instead of their wallet address)
- A malicious or buggy frontend that populates `sendTo` incorrectly
- A linked signer acting on behalf of the subaccount owner

The `submitFastWithdrawal` path is a public function callable by any address, making it directly reachable by any unprivileged caller who holds a valid sequencer-signed `WithdrawCollateralV2` transaction with a bad `sendTo`.

---

### Recommendation

In `Clearinghouse.withdrawCollateral`, extend the `sendTo` validation to reject protocol-internal addresses:

```solidity
if (sendTo == address(0)) {
    sendTo = address(uint160(bytes20(sender)));
} else {
    require(sendTo != address(this), "Cannot withdraw to clearinghouse");
    require(sendTo != withdrawPool, "Cannot withdraw to withdraw pool");
}
```

Apply the same guard in `BaseWithdrawPool.handleWithdrawTransfer` or at the top of `submitFastWithdrawal` after `resolveFastWithdrawal` returns:

```solidity
require(sendTo != address(this), "Cannot withdraw to pool");
```

---

### Proof of Concept

1. User signs a `WithdrawCollateralV2` transaction with `sendTo = address(clearinghouse)` and a valid nonce.
2. Sequencer processes it via `EndpointTx.processTransactionImpl` → `WithdrawCollateralV2` branch.
3. `clearinghouse.withdrawCollateral(sender, productId, amount, address(clearinghouse), idx)` is called.
4. `sendTo != address(0)`, so the fallback is skipped; `sendTo` remains `address(clearinghouse)`.
5. `Clearinghouse.handleWithdrawTransfer` sends tokens to `withdrawPool` and calls `withdrawPool.submitWithdrawal(token, address(clearinghouse), amount, idx)`.
6. `BaseWithdrawPool.submitWithdrawal` calls `handleWithdrawTransfer(token, address(clearinghouse), amount)` → `token.safeTransfer(address(clearinghouse), amount)`.
7. Tokens arrive at the clearinghouse contract. The user's subaccount balance has already been decremented by `spotEngine.updateBalance`. Tokens are permanently locked with no user recovery path. [8](#0-7)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
