Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the end-user, allowing allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which `MetricOmmPool.swap()` always sets to `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router's address, not the end-user. A pool admin who allowlists the router to enable routing simultaneously grants every user of the router the ability to bypass the per-user swap allowlist, voiding the curated-pool access control entirely.

## Finding Description

**Root cause — wrong actor bound in `beforeSwap`:**

`MetricOmmPool.swap()` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value as `sender` to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

So the extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end-user]`.

**Contrast with the correct deposit pattern:**

`DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter — the actual position beneficiary — not the caller: [5](#0-4) 

The pool passes `owner` as a distinct second argument to `_beforeAddLiquidity`: [6](#0-5) 

The swap hook has no analogous distinct "initiating user" parameter — only `sender` (the direct pool caller) and `recipient` (the output beneficiary), neither of which is the end-user when a router is involved.

**Multi-hop compounding:** In `exactInput`, intermediate hops use `address(this)` (the router) as the payer: [7](#0-6) 

The router address is checked at every hop, so the bypass applies to every pool in a multi-hop path that has allowlisted the router.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` in `beforeSwap` to restrict swaps to KYC-verified addresses.
2. Admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` to enable routing (required for any user to route through this pool).
3. Attacker (`0xBAD`, not individually allowlisted) calls `router.exactInputSingle({pool: pool, ...})`.
4. Router calls `pool.swap(...)` with `msg.sender = router`.
5. Pool calls `extension.beforeSwap(sender=router, ...)`.
6. Extension evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker's swap executes against the curated pool without individual authorization.

## Impact Explanation

This is an admin-boundary break: an unprivileged path (the public, permissionless router) bypasses the per-user access control the pool admin configured. The pool admin has no way to simultaneously allow routing and enforce per-user restrictions under the current design — allowlisting the router collapses the individual gate to a shared one. LP assets in the curated pool are exposed to swaps from actors the admin explicitly did not authorize, directly contradicting the pool's intended access model.

## Likelihood Explanation

The trigger condition — allowlisting the router — is the natural and expected operational step for any pool that wants to support routing. Without it, no user can route through the router to this pool. The admin is structurally induced into the mistake: there is no way to allow routing while preserving per-user enforcement. Any pool using `SwapAllowlistExtension` with routing enabled is affected. The bypass is repeatable by any user who can call the router.

## Recommendation

Mirror the deposit allowlist pattern by binding the check to the actual initiating user rather than the direct pool caller. The cleanest fix is to add an explicit `swapper` parameter to the `swap` signature (analogous to `owner` in `addLiquidity`), have the pool forward it to extensions, and have the router set it to `msg.sender` before calling `pool.swap()`. Alternatively, the router can encode `msg.sender` into `extensionData` and the extension can decode and verify it — though this requires the extension to trust the router's encoding.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension in beforeSwap
// 2. Pool admin allowlists the router (required for routing to work):
swapExtension.setAllowedToSwap(pool, address(router), true);

// 3. Attacker (not individually allowlisted) calls the router:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: 1e18,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    tokenIn: token0,
    deadline: block.timestamp,
    extensionData: ""
}));

// 4. router calls pool.swap() → msg.sender = router
// 5. pool calls extension.beforeSwap(sender=router, ...)
// 6. allowedSwapper[pool][router] == true → passes
// 7. Attacker's swap executes — allowlist bypassed
```
