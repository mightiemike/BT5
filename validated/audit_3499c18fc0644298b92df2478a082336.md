### Title
Negative `canSettle` in `PerpEngine.settlePnl` Inflates `availableSettle` and Drains Spot Quote — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.settlePnl` calls `getSettlementState`, which computes `canSettle = min(positionPnl, state.availableSettle)` with no non-negativity guard. When `positionPnl < 0` and `state.availableSettle > 0`, `canSettle` is negative. The subsequent arithmetic then **increases** `availableSettle` (subtracting a negative), **increases** the user's perp `vQuoteBalance` (reducing their unrealized loss), and returns a negative `totalSettled` to `Clearinghouse._settlePnl`, which calls `spotEngine.updateBalance(QUOTE_PRODUCT_ID, subaccount, negative_amount)`, **decreasing** the user's spot quote balance. The net effect on the individual user is zero (perp improves, spot worsens by the same amount), but `availableSettle` is permanently inflated without any real backing, allowing other profitable traders to over-settle and extract unbacked spot quote from the protocol.

---

### Finding Description

**Step 1 — `getSettlementState` returns a negative value** [1](#0-0) 

```solidity
availableSettle = MathHelper.min(
    calculatePositionPnl(balance, productId),
    state.availableSettle
);
```

If `positionPnl = -100e18` and `state.availableSettle = 50e18`, then `canSettle = min(-100e18, 50e18) = -100e18`. There is no `require(availableSettle >= 0)` or early-return guard.

**Step 2 — `settlePnl` applies the negative `canSettle` without validation** [2](#0-1) 

```solidity
(int128 canSettle, State memory state, Balance memory balance) =
    getSettlementState(productId, subaccount);

state.availableSettle -= canSettle;      // 50e18 - (-100e18) = 150e18  ← INCREASES
balance.vQuoteBalance -= canSettle;      // vQuote + 100e18             ← loss reduced
totalSettled += canSettle;               // totalSettled = -100e18
```

**Step 3 — `Clearinghouse._settlePnl` propagates the negative amount to spot** [3](#0-2) 

```solidity
int128 amountSettled = perpEngine.settlePnl(subaccount, productIds);
_spotEngine().updateBalance(QUOTE_PRODUCT_ID, subaccount, amountSettled);
// amountSettled = -100e18 → spot quote decreases by 100e18
```

There is no health check or non-negativity assertion after `_settlePnl` in `Clearinghouse.settlePnl`. [4](#0-3) 

---

### Impact Explanation

| State variable | Before | After | Direction |
|---|---|---|---|
| `state.availableSettle` | 50e18 | 150e18 | **+100e18 (inflated, unbacked)** |
| user perp `vQuoteBalance` | -100e18 | 0 | +100e18 (loss erased) |
| user spot quote balance | X | X - 100e18 | -100e18 |

The user's net wealth is unchanged, but `availableSettle` is now 100e18 larger than it should be. Every subsequent call to `settlePnl` for **other** profitable subaccounts draws against this inflated pool. Those users receive real spot quote credit (`spotEngine.updateBalance(QUOTE_PRODUCT_ID, ..., +amount)`) backed by nothing, leaving the protocol undercollateralized by the inflated amount. Repeated invocations compound the inflation linearly.

---

### Likelihood Explanation

`Clearinghouse.settlePnl` is gated by `onlyEndpoint`, and the endpoint processes sequencer-submitted `SettlePnl` transactions. In Vertex-style architectures the sequencer is expected to process all valid user settlement requests. A user with a losing perp position can request settlement; the sequencer has no on-chain reason to reject it since the contract itself imposes no guard. The path is reachable through the normal `SettlePnl` transaction flow without any privileged access.

---

### Recommendation

Add a non-negativity guard in `PerpEngine.settlePnl` before applying `canSettle`:

```solidity
if (canSettle <= 0) {
    productIds >>= 32;
    continue;
}
```

Alternatively, enforce it inside `getSettlementState`:

```solidity
availableSettle = MathHelper.max(
    0,
    MathHelper.min(calculatePositionPnl(balance, productId), state.availableSettle)
);
```

This ensures settlement only ever transfers non-negative amounts from `availableSettle` to spot quote, preserving the invariant that settlement moves realized positive PnL and never inflates the pool.
