### Title
`delistProduct` Does Not Clear `productToEngine` Mapping or Remove Product from Engine's `productIds` Array, Permanently Blocking Finalization and Liability Liquidation — (`File: core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.delistProduct` settles open balances for explicitly provided subaccounts but never clears `productToEngine[txn.productId]` or removes the delisted `productId` from `PerpEngine.productIds[]`. This stale state is consumed by `ClearinghouseLiq._finalizeSubaccount` and `ClearinghouseLiq._assertCanLiquidateLiability`, which iterate the full `perpEngine.getProductIds()` list. Any subaccount with a residual non-zero balance in the delisted product (missed from `txn.subaccounts`) is permanently blocked from finalization and liability liquidation, creating an irrecoverable solvency gap.

---

### Finding Description

`Clearinghouse.delistProduct` (lines 294–325) performs three actions: it validates the product is a perp, validates the settlement price, and zeroes out balances for each subaccount in `txn.subaccounts`. It does **not**:

1. Delete or zero `productToEngine[txn.productId]`
2. Remove `txn.productId` from `perpEngine.productIds[]`
3. Clear `perpEngine.states[txn.productId]` [1](#0-0) 

After the call, `productToEngine[txn.productId]` still resolves to the perp engine: [2](#0-1) 

And `perpEngine.getProductIds()` still returns the delisted `productId` because `productIds` is append-only in `BaseEngine._addOrUpdateProduct` — there is no removal path: [3](#0-2) 

`ClearinghouseLiq._finalizeSubaccount` calls `perpEngine.getProductIds()` and requires every returned `perpId` to have `balance.amount == 0` for the liquidatee: [4](#0-3) 

`ClearinghouseLiq._assertCanLiquidateLiability` does the same: [5](#0-4) 

Both functions iterate the full `perpIds` list, which permanently includes the delisted product. Any subaccount whose balance in the delisted product was not zeroed by `delistProduct` (because it was omitted from `txn.subaccounts`) will cause both checks to revert indefinitely.

Additionally, `ClearinghouseLiq._settlePositivePerpPnl` iterates all `perpEngine.getProductIds()` and settles PnL for each, including the delisted product: [6](#0-5) 

This means the delisted product's stale `vQuoteBalance` can be settled against a liquidator's quote balance, corrupting accounting.

Finally, `PerpEngineState.updateStates` iterates `productIds[i]` for `i < avgPriceDiffs.length`. Since the delisted product remains in `productIds`, the sequencer must continue supplying a funding rate entry for it or all subsequent funding rate updates will be applied to the wrong products (index shift): [7](#0-6) 

---

### Impact Explanation

- **Permanent liquidation/finalization block**: Any subaccount with a residual `balance.amount != 0` in the delisted product cannot be finalized (`ERR_NOT_FINALIZABLE_SUBACCOUNT`) or have its liabilities liquidated (`ERR_NOT_LIQUIDATABLE_LIABILITIES`). The subaccount is permanently stuck in an unhealthy state, and the insurance fund cannot recover from it.
- **Accounting corruption via stale PnL settlement**: `_settlePositivePerpPnl` will attempt to settle `vQuoteBalance` for the delisted product against the liquidator, transferring quote value based on stale/zeroed state.
- **Funding rate index desynchronization**: `updateStates` index alignment breaks if the sequencer stops sending entries for the delisted product, misapplying funding rates to all subsequent products.

---

### Likelihood Explanation

`delistProduct` accepts a caller-supplied `subaccounts` array. The sequencer must enumerate every subaccount with an open position. In practice, this list can be incomplete due to off-chain indexing lag, gas limits, or race conditions where new positions are opened between the sequencer's snapshot and the on-chain execution. The likelihood of at least one subaccount being missed is non-trivial for any product with significant open interest at delist time.

---

### Recommendation

In `Clearinghouse.delistProduct`, after settling all subaccount balances:

1. Clear `productToEngine[txn.productId] = IProductEngine(address(0))`.
2. Call a new function on `PerpEngine` (e.g., `removeProduct(uint32 productId)`) that removes `productId` from the `productIds` array and zeroes `states[productId]`.
3. Alternatively, add a `delistedProducts` mapping and gate `_finalizeSubaccount`, `_assertCanLiquidateLiability`, and `updateStates` to skip delisted product IDs.

---

### Proof of Concept

1. Sequencer calls `delistProduct` for `productId = 3` with `subaccounts = [A, B]`, but subaccount `C` also has `balance.amount = 500` in product 3 (missed due to indexing lag).
2. After the call: `productToEngine[3]` still points to perp engine; `perpEngine.productIds()` still includes `3`; `perpEngine.balances[3][C].amount = 500`.
3. Subaccount `C` becomes undercollateralized. A liquidator submits `LiquidateSubaccount` with `productId = type(uint32).max` (finalize path).
4. `_finalizeSubaccount` calls `perpEngine.getProductIds()` → returns `[..., 3, ...]`.
5. For `perpId = 3`: `perpEngine.getBalance(3, C).amount = 500 != 0` → `require(balance.amount == 0)` reverts with `ERR_NOT_FINALIZABLE_SUBACCOUNT`.
6. The liquidator retries with a non-finalize liquidation. `_assertCanLiquidateLiability` is called, iterates all `perpIds` including `3`, finds `balance.amount = 500 != 0` → reverts with `ERR_NOT_LIQUIDATABLE_LIABILITIES`.
7. Subaccount `C` is permanently unliquidatable. Its bad debt is unrecoverable by the insurance fund. [1](#0-0) [4](#0-3) [5](#0-4) [8](#0-7)

    function delistProduct(bytes calldata transaction) external onlyEndpoint {
        IEndpoint.DelistProduct memory txn = abi.decode(
            transaction[1:],
            (IEndpoint.DelistProduct)
        );
        // only perp can be delisted
        require(
            productToEngine[txn.productId] == _perpEngine(),
            ERR_INVALID_PRODUCT
        );
        require(txn.priceX18 == _getPriceX18(txn.productId), ERR_INVALID_PRICE);
        IPerpEngine perpEngine = _perpEngine();
        for (uint256 i = 0; i < txn.subaccounts.length; i++) {
            IPerpEngine.Balance memory balance = perpEngine.getBalance(
                txn.productId,
                txn.subaccounts[i]
            );
            int128 baseDelta = -balance.amount;
            int128 quoteDelta = -baseDelta.mul(txn.priceX18);
            perpEngine.updateBalance(
                txn.productId,
                txn.subaccounts[i],
                baseDelta,
                quoteDelta
            );
            if (RiskHelper.isIsolatedSubaccount(txn.subaccounts[i])) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.subaccounts[i]);
            }
        }
    }
