### Title
Fork-Block Retarget Uses Mainchain Ancestor Timestamp Instead of Fork Ancestor Timestamp, Producing Incorrect Difficulty Target — (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`, `contract/src/dogecoin.rs`)

---

### Summary

`get_next_work_required` fetches the first block of the retarget interval via `get_header_by_height`, which always resolves against `mainchain_height_to_header`. When the block under validation is a fork block whose lineage diverges before that interval boundary, the fork's actual ancestor at that height is a different block (stored only in `headers_pool`) with a potentially different timestamp. The code silently uses the mainchain block's timestamp instead, producing an incorrect difficulty target — an exact structural analog of using `balanceOf` (the inflated, wrong source) instead of `earningOf` (the correct, settled source).

---

### Finding Description

`check_pow` is invoked for every submitted block — both mainchain extensions and fork candidates — before the mainchain/fork distinction is made in `submit_block_header_inner`. [1](#0-0) 

At a retarget boundary, `get_next_work_required` (Bitcoin and Litecoin variants) computes the expected difficulty using the timestamp of the first block in the retarget interval: [2](#0-1) 

`get_header_by_height` is implemented to look up the block exclusively from `mainchain_height_to_header`: [3](#0-2) 

Fork blocks are stored only in `headers_pool` via `store_fork_header`, never in `mainchain_height_to_header`: [4](#0-3) 

When a fork diverges before `first_block_height`, the fork's ancestor at that height is a distinct block with an attacker-controlled timestamp. The code ignores it and uses the mainchain block's timestamp instead. The Dogecoin implementation carries an explicit developer acknowledgment of this exact concern: [5](#0-4) 

The same structural flaw is present in the Litecoin path: [6](#0-5) 

---

### Impact Explanation

The difficulty target for a fork block at a retarget boundary is computed from the wrong timestamp. An attacker who sets the fork's ancestor at `first_block_height` to a timestamp **later** than the mainchain block at that height causes the code to use the mainchain's earlier timestamp. This produces a larger `actual_time_taken` value in `calculate_next_work_required`, yielding a higher (easier) target than the fork's lineage actually warrants: [7](#0-6) 

With an artificially relaxed difficulty, the attacker mines the retarget block with less PoW than required. If the fork's accumulated `chain_work` then exceeds the mainchain tip's `chain_work`, `reorg_chain` is triggered: [8](#0-7) 

This corrupts `mainchain_height_to_header`, `mainchain_header_to_height`, and `mainchain_tip_blockhash` — the state that `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` rely on to confirm transaction finality. A reorganization to an attacker-controlled chain causes those functions to return results based on a fraudulent canonical chain.

---

### Likelihood Explanation

The entry path is `submit_blocks`, callable by any unprivileged NEAR account (the `#[trusted_relayer]` guard can be bypassed by accounts granted `UnrestrictedSubmitBlocks`, and the relayer itself is the normal submission path). The attacker must:

1. Mine fork blocks from a divergence point before `first_block_height` — at the same difficulty as the mainchain, since no retarget has occurred yet.
2. Set the fork's ancestor at `first_block_height` to a timestamp later than the mainchain block there (within MTP and future-time constraints, both of which are checked against the fork's own chain, not the mainchain's).
3. Mine the retarget block at the reduced difficulty produced by the bug.

The maximum difficulty reduction is bounded by the 4× clamp in `calculate_next_work_required`, but even a modest reduction meaningfully lowers the mining cost for the retarget block. The attack is realistic for a well-resourced adversary targeting a chain with a predictable retarget schedule.

---

### Recommendation

Replace the `get_header_by_height` call with an ancestor traversal that walks the fork's own lineage backward from `prev_block_header` to `first_block_height`. This mirrors the correct approach used in Bitcoin Core, which traverses `pindexFirst` by following `pprev` pointers rather than looking up by height in the active chain. The corrected logic should be:

```rust
// Walk back through the fork's own ancestor chain to first_block_height
let mut cursor = prev_block_header.clone();
while cursor.block_height > first_block_height {
    cursor = blocks_getter.get_prev_header(&cursor.block_header);
}
// cursor is now the fork's actual ancestor at first_block_height
calculate_next_work_required(config, prev_block_header, cursor.block_header.time.into())
```

This ensures the timestamp used for difficulty calculation always belongs to the block's own lineage, not the mainchain's — eliminating the desynchronization between the two sources.

---

### Proof of Concept

1. The mainchain has blocks at heights 0–2016 (one full retarget interval). The mainchain block at height 0 (`first_block_height` for the next retarget) has timestamp `T_main = 1_000_000`.

2. An attacker submits a fork diverging at height 1, with its own block at height 0 having timestamp `T_fork = 1_100_000` (100,000 seconds later than the mainchain block, within future-time limits relative to the fork's own chain).

3. When the attacker submits the fork block at height 2016 (the retarget boundary), `get_next_work_required` calls `get_header_by_height(0)`, which returns the **mainchain** block with `T_main = 1_000_000`.

4. `actual_time_taken = prev_block_time - T_main` is 100,000 seconds larger than `prev_block_time - T_fork` would be, producing a proportionally easier difficulty target (up to the 4× cap).

5. The attacker mines the height-2016 fork block at this reduced difficulty. If the fork's total `chain_work` exceeds the mainchain's, `reorg_chain` fires, replacing the canonical chain with the attacker's fork and corrupting all subsequent `verify_transaction_inclusion` results.
