### Title
ECDSA Signature Malleability in `op_secp256k1_verify` and `op_secp256r1_verify` — (File: `src/secp_ops.rs`)

---

### Summary

`op_secp256k1_verify` and `op_secp256r1_verify` in `src/secp_ops.rs` parse ECDSA signatures with `from_slice` and verify them with `verify_prehash` without enforcing low-s normalization. For any valid signature `(r, s)`, the malleable form `(r, n − s)` (where `n` is the curve order) also passes verification. Any CLVM puzzle that uses these operators for replay protection by marking the raw signature bytes as "used" can be bypassed by an attacker who submits the alternate form of a previously accepted signature.

---

### Finding Description

In `src/secp_ops.rs`, both verification operators follow the same pattern:

```rust
// op_secp256k1_verify (lines 88-97)
let sig = K1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

```rust
// op_secp256r1_verify (lines 43-51)
let sig = P1Signature::from_slice(sig.as_ref()).map_err(|_| { ... })?;
let result = verifier.verify_prehash(msg.as_ref(), &sig);
```

`K1Signature::from_slice` (k256 `0.14.0-rc`, `Cargo.toml` line 91) and `P1Signature::from_slice` (p256 `0.14.0-rc`, line 92) parse the raw 64-byte `(r ‖ s)` encoding and accept any `s` in `[1, n−1]` — including high-s values above `n/2`. Neither function normalizes `s`. The subsequent `verify_prehash` call (k256 `VerifyingKey::verify_prehash`; p256 `hazmat::PrehashVerifier::verify_prehash`) performs the mathematical ECDSA check without a low-s guard. No `normalize_s()` call appears anywhere in `secp_ops.rs`, and the test suite (`op-tests/test-secp-verify.txt`, `op-tests/test-secp256k1.txt`) contains no test case that supplies a high-s signature and expects rejection — consistent with both forms being accepted.

The `Cargo.toml` dependency block confirms the exact library versions in use:

```toml
k256 = { version = "0.14.0-rc", features = ["ecdsa"] }
p256 = { version = "0.14.0-rc", features = ["ecdsa"] }
```

---

### Impact Explanation

A CLVM puzzle that calls `secp256k1_verify` (or `secp256r1_verify`) and implements replay protection by recording the raw signature atom as "spent" is vulnerable. An attacker who observes a valid spend containing signature `(r, s)` can construct the alternate signature `(r, n − s)` — different bytes, same signer, same message — and submit it in a second spend. The operator returns success for both forms, so the puzzle's byte-level "already used" check does not fire.

This is the direct on-chain analog of the OpenZeppelin ECDSA malleability: the library-level primitive accepts two byte representations of the same authorization, undermining any higher-level uniqueness assumption built on top of it.

Impact: **consensus / wallet** — a coin protected by a secp256k1/secp256r1 puzzle with signature-byte replay protection can be double-spent.

---

### Likelihood Explanation

`secp256k1_verify` and `secp256r1_verify` are softfork extensions gated behind `ENABLE_SECP_OPS` (`ClvmFlags`). They are not used in Chia's standard BLS-based puzzles. However, they are production operators exposed to any caller that sets the flag, and the Chia ecosystem explicitly supports custom puzzles. A puzzle author who follows the natural pattern of "record the signature to prevent replay" — as is common in Ethereum-style designs — would produce a vulnerable puzzle. The malleability computation (`s' = n − s mod n`) is trivial for any attacker who can observe a valid spend on-chain.

Likelihood: **Medium** (operator is real and reachable; exploitation requires a puzzle that uses signature-byte replay protection, which is a natural but non-default design choice).

---

### Recommendation

Add explicit low-s enforcement in both operators before calling `verify_prehash`. For k256, call `sig.normalize_s()` and use the normalized form; for p256, apply the equivalent normalization. Alternatively, reject signatures where `s > n/2` with an `InvalidOpArg` error, matching the behavior Bitcoin enforced via BIP 62 / BIP 146. Add test vectors that supply high-s signatures and assert `FAIL` for both `secp256k1_verify` and `secp256r1_verify`.

---

### Proof of Concept

Given a valid secp256k1 spend with signature `(r, s)` where `s ≤ n/2`:

1. Compute `s' = n − s mod n` (trivial arithmetic on the secp256k1 curve order `n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141`).
2. Construct the 64-byte atom `r ‖ s'`.
3. Submit a second CLVM program invoking `(secp256k1_verify pubkey msg (q . <r ‖ s'>))`.
4. `K1Signature::from_slice` parses `(r, s')` without error (both `r` and `s'` are in `[1, n−1]`).
5. `verifier.verify_prehash(msg, &sig)` returns `Ok(())` because `(r, s')` satisfies the ECDSA verification equation for the same `(pubkey, msg)` pair.
6. The operator returns `Reduction(1300000, nil)` — success — despite the signature bytes being different from the first spend.

Any puzzle that checks `(= (f solution) cached-sig-atom)` or equivalent to prevent replay will not match `r ‖ s'` against the stored `r ‖ s`, and the second spend succeeds. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
