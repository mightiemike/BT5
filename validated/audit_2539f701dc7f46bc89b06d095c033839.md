### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

Two coexisting transaction-inclusion verification functions are exposed as public NEAR contract methods. The deprecated `verify_transaction_inclusion` (v1) lacks coinbase Merkle proof validation and is explicitly documented as vulnerable to the 64-byte transaction forgery attack. Despite being superseded by `verify_transaction_inclusion_v2`, v1 remains a live, unprivileged-callable endpoint. Any NEAR caller can invoke it directly, bypassing the security fix introduced in v2, and obtain a `true` result for a non-existent transaction.

---

### Finding Description

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` coexist as public methods on `BtcLightClient`. [1](#0-0) 

The v1 function is marked `#[deprecated]` and carries an explicit `# Warning` in its doc comment acknowledging the forgery risk, but `#[deprecated]` in Rust is a compile-time lint only — it imposes no runtime restriction on callers. [2](#0-1) 

The v1 proof check is a bare Merkle root comparison with no constraint on what `tx_id` represents: [3](#0-2) 

The v2 function was introduced specifically to close this gap by first verifying a coinbase proof (which pins the tree depth), then delegating to v1: [4](#0-3) 

Because v1 is still `pub` and decorated only with `#[pause]` (not paused by default), any unprivileged NEAR account can call `verify_transaction_inclusion` directly, skipping the coinbase check entirely.

The 64-byte attack works as follows: Bitcoin's Merkle tree is built by double-SHA256-hashing pairs of 32-byte child hashes, producing a 64-byte input. An internal node at depth `d-1` is itself a valid 32-byte hash. An attacker who supplies that internal node as `tx_id` with a proof path one level shorter than the real tree depth can still compute the correct Merkle root, causing v1 to return `true` for a transaction that does not exist in the block. [5](#0-4) 

---

### Impact Explanation

A recipient NEAR contract that calls `verify_transaction_inclusion` (v1) to gate a privileged action — releasing funds, minting tokens, updating state — receives a `true` result for a fabricated transaction. The corrupted proof result is the direct output of a public contract method, not an internal computation. The attacker controls `tx_id`, `tx_index`, and `merkle_proof` entirely; the only external constraint is that the target block must be on the confirmed mainchain, which is public information.

---

### Likelihood Explanation

The attack requires knowledge of a real Bitcoin block's Merkle tree structure, which is fully public. The BitMEX blog post referenced in the contract's own source code provides a worked example. The v1 method is callable by any NEAR account with no deposit or role requirement beyond the contract not being paused. Likelihood is **medium**: the attacker needs to identify a downstream consumer of v1 and construct a valid internal-node proof, but no privileged access is required.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely, or change its visibility to `pub(crate)` / private so it is only reachable via `verify_transaction_inclusion_v2`. If backward compatibility must be preserved, add a runtime `env::panic_str` at the top of v1 to make it unconditionally revert when called directly. The `#[deprecated]` attribute alone is insufficient because it is erased at runtime.

---

### Proof of Concept

1. Deploy the contract and submit a real Bitcoin block (e.g., height 685452) to the mainchain.
2. Obtain the block's full Merkle tree from a Bitcoin node. Identify an internal node `N` at depth `d-1` whose left and right children are `C_L` and `C_R` (so `N = SHA256d(C_L || C_R)`).
3. Construct `ProofArgs`:
   - `tx_id` = `N` (the internal node, treated as a fake leaf)
   - `tx_block_blockhash` = the confirmed block hash
   - `tx_index` = the index corresponding to `N`'s position at depth `d-1`
   - `merkle_proof` = the sibling path from depth `d-1` up to the root (one element shorter than a real leaf proof)
   - `confirmations` = 1
4. Call `verify_transaction_inclusion` directly on the NEAR contract with the above args (Borsh-serialized).
5. The function computes `compute_root_from_merkle_proof(N, tx_index, proof)`, which equals the real Merkle root, and returns `true` — falsely asserting that a non-existent transaction is included in the block. [6](#0-5) [7](#0-6)

    /// # Warning
    /// This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash.
    /// We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification.
    ///
    /// # Panics
    /// Multiple cases
    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
