### Title
NLP Signer Not Checked in `validateSignature`, Blocking NLP Pool Subaccount Transactions — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The protocol defines two tiers of signer authorization for subaccounts: `linkedSigners` (user-set) and `nlpSigners` (protocol-set, assigned to the NLP pool owner). For `MatchOrders` and `MatchOrdersWithAmount`, the code correctly falls through from tier 1 to tier 2 via `getLinkedSignerOrNlpSigner`. However, `validateSignature` — used for all `validateSignedTx`-gated transactions — only consults `getLinkedSigner` (tier 1). When no linked signer is set, it passes `address(0)` to the verifier and never checks `nlpSigners`. This mirrors the M-03 pattern exactly: a two-tier authorization system that fails to fall through to the second tier.

---

### Finding Description

The protocol stores two distinct signer mappings:

- `linkedSigners[subaccount]` — set by the user via `LinkSigner`
- `nlpSigners[subaccount]` — set by the protocol to the NLP pool owner when a pool is created or updated [1](#0-0) 

For `MatchOrders` and `MatchOrdersWithAmount`, the code correctly uses `getLinkedSignerOrNlpSigner`, which falls through from `linkedSigners` to `nlpSigners` when the former is unset: [2](#0-1) 

However, `validateSignature` — invoked by every `validateSignedTx` call — only passes `getLinkedSigner(sender)` to the verifier: [3](#0-2) 

`getLinkedSigner` only reads from `linkedSigners`, never from `nlpSigners`: [4](#0-3) 

NLP pool subaccounts have `nlpSigners[subaccount] = owner` set at pool creation, but `linkedSigners[subaccount]` is never populated by the protocol: [5](#0-4) 

This means that for every transaction validated via `validateSignedTx` — including `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LinkSigner`, `MintNlp`, `BurnNlp` — the NLP pool owner's signature is silently rejected. The verifier receives `address(0)` as the authorized signer and only accepts a signature from the raw address bytes of the NLP pool subaccount, which is a protocol-controlled address the pool owner does not control. [6](#0-5) 

This also creates a self-referential lock: the NLP pool owner cannot sign a `LinkSigner` transaction to register themselves as a `linkedSigner` for the pool subaccount, because `validateSignature` already requires a `linkedSigner` to be set. The slow-mode `LinkSigner` path uses `validateSender`, which requires `msg.sender` to equal the address embedded in the subaccount bytes — again a protocol-controlled address: [7](#0-6) 

---

### Impact Explanation

NLP pool owners are permanently unable to sign `validateSignedTx`-gated transactions for their NLP pool subaccounts. Concretely:

- They cannot sign `WithdrawCollateral` to withdraw accumulated balances from the NLP pool subaccount.
- They cannot sign `TransferQuote` to move quote out of the NLP pool subaccount.
- They cannot sign `LinkSigner` to bootstrap a linked signer for the pool subaccount, because that transaction itself requires a pre-existing linked signer or direct ownership of the subaccount address.

Any balance that accumulates in an NLP pool subaccount (e.g., from fee accrual or rebalancing) is inaccessible to the pool owner through the normal signed-transaction path. The only recovery path is through privileged sequencer or admin operations, which are not available to the pool owner.

---

### Likelihood Explanation

Every NLP pool owner who attempts to sign a `WithdrawCollateral` or `TransferQuote` for their pool subaccount will encounter this failure. The `nlpSigners` mapping is populated for every NLP pool at creation time, so the precondition is always satisfied. The trigger is any NLP pool owner attempting to exercise control over their pool subaccount's balance outside of `MatchOrders`.

---

### Recommendation

Replace `getLinkedSigner` with `getLinkedSignerOrNlpSigner` inside `validateSignature` and `validateCompactSignature`:

```solidity
function validateSignature(
    bytes32 sender,
    bytes32 digest,
    bytes memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
        digest,
        signature
    );
}

function validateCompactSignature(
    bytes32 sender,
    bytes32 digest,
    IEndpoint.CompactSignature memory signature,
    bool allowLinkedSigner
) internal virtual {
    verifier.validateCompactSignature(
        sender,
        allowLinkedSigner ? getLinkedSignerOrNlpSigner(sender) : address(0),
        digest,
        signature
    );
}
```

This mirrors the existing correct behavior in the `MatchOrders` path and ensures the two-tier authorization model is applied consistently across all transaction types.

---

### Proof of Concept

1. NLP pool is created via `addNlpPool`. `nlpSigners[poolSubaccount] = Alice` (pool owner). `linkedSigners[poolSubaccount]` is never set (remains `address(0)`).
2. The NLP pool subaccount accumulates quote balance through protocol operations.
3. Alice constructs a `WithdrawCollateral` transaction with `sender = poolSubaccount` and signs it with her key.
4. The sequencer submits it via `processTransactionImpl` → `validateSignedTx` → `validateSignature`.
5. `validateSignature` calls `getLinkedSigner(poolSubaccount)` → returns `address(0)` (no linked signer set).
6. `verifier.validateSignature(poolSubaccount, address(0), digest, aliceSignature)` is called.
7. The verifier checks only whether the signature matches the raw address of `poolSubaccount` — not Alice's address.
8. The transaction reverts. Alice cannot withdraw, cannot set a linked signer, and has no recovery path.
9. The same Alice signature would succeed for a `MatchOrders` transaction because that path uses `getLinkedSignerOrNlpSigner`, which returns `Alice` as the authorized signer. [8](#0-7) [3](#0-2)
