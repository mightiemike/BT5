### Title
`ENABLE_SECP_OPS` Hard-Fork Gate Bypassed via 4-Byte Opcode Aliases — (`File: src/chia_dialect.rs`)

---

### Summary

The `ENABLE_SECP_OPS` flag is declared as the hard-fork gate for secp256k1 and secp256r1 signature verification. It correctly guards the 1-byte opcode slots (64 and 65). However, the same two operations are also reachable via 4-byte opcode aliases (`0x13d61f00` and `0x1c3a8f00`) in a separate dispatch branch that performs **no flag check at all**. Any caller that omits `ENABLE_SECP_OPS` — including pre-fork consensus nodes and mempool validators — will silently execute full secp signature verification when those 4-byte opcodes appear in attacker-controlled CLVM bytes.

---

### Finding Description

In `src/chia_dialect.rs`, the `ChiaDialect::op()` function contains two independent dispatch paths.

**Path A — 1-byte opcodes (correctly gated):**

```rust
// lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
```

Without `ENABLE_SECP_OPS`, opcodes 64 and 65 fall through to `unknown_operator`, which either returns a no-op cost or errors under `NO_UNKNOWN_OPS`.

**Path B — 4-byte opcodes (ungated):**

```rust
// lines 175–182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

There is no `flags.contains(ClvmFlags::ENABLE_SECP_OPS)` check here. The function pointer is resolved and called unconditionally. The `flags` value passed into `f` still carries whatever the caller set, but neither `op_secp256k1_verify` nor `op_secp256r1_verify` inspects `ENABLE_SECP_OPS` — both functions ignore their `_flags` parameter entirely.

The two secp functions are identical in behavior regardless of which path invokes them:

```rust
// secp_ops.rs lines 15–19, 61–65
pub fn op_secp256r1_verify(a: &mut Allocator, input: NodePtr, max_cost: Cost, _flags: ClvmFlags) -> Response { ... }
pub fn op_secp256k1_verify(a: &mut Allocator, input: NodePtr, max_cost: Cost, _flags: ClvmFlags) -> Response { ... }
```

The `_flags` parameter is explicitly unused in both implementations.

---

### Impact Explanation

**Vulnerability class: flag/operator wiring error — hard-fork gate not applied to all dispatch paths.**

The analog to the original report is direct: just as `ReachDistributionV2`'s `onlyOwner` guard was wired to the wrong owner (the factory) making the guard ineffective, `ENABLE_SECP_OPS` is wired only to the 1-byte opcode slots while the 4-byte aliases are left completely unguarded — making the flag effectively ownerless for those paths.

Concrete corrupted result: a node running **without** `ENABLE_SECP_OPS` (pre-fork consensus mode or mempool mode without the flag) will:

1. **Reject** `(64 pubkey msg sig)` — 1-byte opcode, correctly blocked.
2. **Accept and execute** `(0x13d61f00 pubkey msg sig)` — 4-byte alias, gate bypassed.

This produces a **consensus divergence**: two nodes with different `ENABLE_SECP_OPS` settings will disagree on the validity of a transaction that uses the 4-byte opcode form. A pre-fork node accepts it; a node that has not yet activated the flag also accepts it — but both should reject it identically until the fork activates. The flag's entire purpose is to create a clean activation boundary; the 4-byte path destroys that boundary.

Additionally, under `MEMPOOL_MODE` (which sets `NO_UNKNOWN_OPS` but does **not** set `ENABLE_SECP_OPS`), the 4-byte secp opcodes are accepted and executed rather than rejected, meaning mempool policy diverges from the intended pre-fork rejection behavior.

---

### Likelihood Explanation

**Medium.** The 4-byte opcode encoding is part of the public CLVM serialization format. Any attacker who reads the dispatch table in `chia_dialect.rs` can craft a CLVM program using `0x13d61f00` or `0x1c3a8f00` as the operator atom. The program is valid CLVM bytes, passes serialization, and is submitted as a normal transaction. No privileged access, social engineering, or dependency compromise is required. The only prerequisite is knowledge of the opcode table, which is public source code.

---

### Recommendation

Apply the same `ENABLE_SECP_OPS` guard to the 4-byte dispatch branch. The simplest fix is to check the flag before resolving the function pointer:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This mirrors the guard pattern already used correctly in the 1-byte branch and ensures the hard-fork activation boundary is enforced uniformly across all opcode encodings.

### Proof of Concept

Attacker constructs a CLVM program with operator atom `\x13\xd6\x1f\x00` (4 bytes) and valid secp256k1 pubkey/msg/sig arguments. The program is submitted to a node running `ChiaDialect::new(ClvmFlags::NO_UNKNOWN_OPS)` (no `ENABLE_SECP_OPS`).

**Expected (correct) behavior:** rejected — secp ops not yet activated.

**Actual behavior:** `op_secp256k1_verify` executes, charges 1,300,000 cost, and returns `nil` on success — the hard-fork gate is silently bypassed.
