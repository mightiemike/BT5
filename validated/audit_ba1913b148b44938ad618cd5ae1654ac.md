### Title
`vm_resources_to_execution_resources` returns `UnknownBuiltin` error for valid Cairo 1 contracts using `range_check96`/`add_mod`/`mul_mod` builtins, causing `starknet_simulateTransactions` to fail on valid transactions — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `vm_resources_to_execution_resources` function in `crates/apollo_rpc_execution/src/objects.rs` does not handle the `range_check96`, `add_mod`, and `mul_mod` `BuiltinName` variants. These are valid Cairo 1 builtins available since StarknetVersion >= 0.13.2. When a contract using any of these builtins is simulated via the public `starknet_simulateTransactions` RPC endpoint, the conversion path terminates with `Err(ExecutionError::UnknownBuiltin)`, causing the RPC to return an error for an otherwise valid transaction.

---

### Finding Description

In `vm_resources_to_execution_resources` (lines 363–401), the match over `BuiltinName` handles eight known builtins. Three builtins introduced in StarknetVersion >= 0.13.2 — `range_check96`, `add_mod`, and `mul_mod` — are explicitly called out in a TODO comment but left unimplemented: [1](#0-0) 

The wildcard arm unconditionally returns an error: [2](#0-1) 

This function is called from `FunctionInvocation::try_from((CallInfo, GasVector))`: [3](#0-2) 

Which is called from all four transaction trace converters (`InvokeTransactionTrace`, `DeclareTransactionTrace`, `DeployAccountTransactionTrace`, `L1HandlerTransactionTrace`): [4](#0-3) 

The full public call chain is:

```
starknet_simulateTransactions (RPC)
  → simulate_transactions (api_impl.rs:1067)
    → exec_simulate_transactions
      → *TransactionTrace::try_from(TransactionExecutionInfo)
        → FunctionInvocation::try_from((CallInfo, GasVector))
          → vm_resources_to_execution_resources
            → Err(ExecutionError::UnknownBuiltin)  ← for range_check96/add_mod/mul_mod
```

The error propagates through `.map_err(execution_error_to_error_object_owned)?` at the RPC layer: [5](#0-4) 

---

### Impact Explanation

Any Cairo 1 contract that uses `range_check96`, `add_mod`, or `mul_mod` builtins (e.g., contracts using modular arithmetic or 96-bit range checks, which are standard in Sierra/CASM since 0.13.2) will cause `starknet_simulateTransactions`, `starknet_traceTransaction`, and `starknet_traceBlockTransactions` to return a `TRANSACTION_EXECUTION_ERROR` for a transaction that would execute successfully on-chain. This is a concrete wrong value returned by the RPC simulation path for valid inputs, matching the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

`range_check96` is emitted by the Sierra compiler for common integer operations in Cairo 1 contracts targeting StarknetVersion >= 0.13.2. Any deployed contract using these operations will trigger this path when simulated. The path is fully reachable by any unprivileged caller of the public RPC endpoint with no special privileges required.

---

### Recommendation

Add explicit match arms for `BuiltinName::range_check96`, `BuiltinName::add_mod`, and `BuiltinName::mul_mod` in `vm_resources_to_execution_resources`. The RPC `Builtin` enum (or `ComputationResources`) must be extended to carry these variants, or they must be mapped to existing fields if the RPC spec permits. The TODO comment at lines 385–388 already identifies this gap. [6](#0-5) 

---

### Proof of Concept

```rust
use cairo_vm::types::builtin_name::BuiltinName;
use cairo_vm::vm::runners::cairo_runner::ExecutionResources as VmExecutionResources;
use std::collections::HashMap;

// Construct a CallInfo whose vm_resources contain range_check96 with count > 0.
let mut counter = HashMap::new();
counter.insert(BuiltinName::range_check96, 1usize);
let vm_resources = VmExecutionResources {
    n_steps: 100,
    n_memory_holes: 0,
    builtin_instance_counter: counter,
};

// Call the function under test (exposed via FunctionInvocation::try_from path).
let result = vm_resources_to_execution_resources(vm_resources, GasVector::default());

// Asserts the bug: a valid builtin causes an error.
assert!(matches!(result, Err(ExecutionError::UnknownBuiltin { builtin_name })
    if builtin_name == BuiltinName::range_check96));
```

This directly confirms the TODO at lines 385–388 and the wildcard error at line 390. [1](#0-0)
