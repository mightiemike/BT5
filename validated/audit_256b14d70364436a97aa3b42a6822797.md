### Title
Unclaimed Bonus Rewards Permanently Lost When `ClearExit` Advances `LastBonusAccrual` Without Prior Claim - (File: `x/tieredrewards/types/position_state.go`)

---

### Summary

The `ClearExit` function in the `x/tieredrewards` module directly advances the `LastBonusAccrual` checkpoint to `blockTime` without first claiming the bonus rewards that have accrued since the position's last claim. This permanently destroys any unclaimed bonus rewards for the position owner, because the segment-bonus formula uses `LastBonusAccrual` as the start of the next accrual window.

---

### Finding Description

Bonus rewards in the `tieredrewards` module are computed lazily via `processEventsAndClaimBonus`. The formula accumulates rewards over time segments bounded by `[pos.LastBonusAccrual, eventTimestamp]` and `[segmentStart, blockTime]`. After paying out rewards, the checkpoint is correctly advanced:

```go
// x/tieredrewards/keeper/bonus_rewards.go
func applyBonusAccrualCheckpoint(pos *types.Position, blockTime time.Time) {
    accrualEnd := blockTime
    if pos.CompletedExitLockDuration(blockTime) {
        accrualEnd = pos.ExitUnlockAt
    }
    pos.UpdateLastBonusAccrual(accrualEnd)
}
``` [1](#0-0) 

This is called at the **end** of `processEventsAndClaimBonus`, after all rewards have been computed and transferred to the owner. [2](#0-1) 

However, `ClearExit` — which a position owner calls to cancel a pending exit — directly overwrites `LastBonusAccrual` with `blockTime` **without** going through `processEventsAndClaimBonus`:

```go
// x/tieredrewards/types/position_state.go
func (p *PositionState) ClearExit(blockTime time.Time) {
    p.ExitTriggeredAt = time.Time{}
    p.ExitUnlockAt = time.Time{}
    if p.IsDelegated() {
        // required so that positions who clear exit after exit lock duration won't have extra bonus accrued
        p.LastBonusAccrual = blockTime
    }
}
``` [3](#0-2) 

The developer comment reveals the intent: prevent future accrual past the exit lock duration. But the side effect is that all bonus rewards accrued since the last claim — from the old `LastBonusAccrual` up to `ExitUnlockAt` (or `blockTime` if exit lock has not yet completed) — are silently discarded. The next call to `processEventsAndClaimBonus` will use the new `blockTime` as `segmentStart`, skipping the entire unclaimed window.

The `computeSegmentBonus` function caps `segmentEnd` at `ExitUnlockAt` when it is non-zero:

```go
if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
    segmentEnd = pos.ExitUnlockAt
}
``` [4](#0-3) 

After `ClearExit`, `ExitUnlockAt` is zeroed out, so this cap no longer applies. The rewards that would have been bounded by `[oldLastBonusAccrual, ExitUnlockAt]` are permanently unreachable.

---

### Impact Explanation

Any position owner who has accrued bonus rewards and then calls `ClearExit` (to cancel a pending exit) permanently loses all bonus rewards accrued since their last `ClaimTierRewards` call. The `LastBonusAccrual` checkpoint is advanced past the accrual window without a corresponding payout. The lost tokens remain in the `RewardsPool` module account but are no longer attributable to the affected user. This is a direct, irreversible economic loss for the position owner.

---

### Likelihood Explanation

The trigger is a standard user transaction (`MsgClearExit` or equivalent) available to any position owner. No privileged role is required. The loss occurs whenever a user has unclaimed bonus rewards at the time they call `ClearExit`, which is the common case for any active position that has not claimed rewards in the same block. The longer the gap since the last claim, the larger the loss.

---

### Recommendation

Before advancing `LastBonusAccrual` in `ClearExit`, the keeper's message handler for `ClearExit` must first call `processEventsAndClaimBonus` (or the full `claimRewards` flow) to pay out all accrued bonus rewards. Only after the rewards are settled should the checkpoint be advanced. Alternatively, `ClearExit` should not modify `LastBonusAccrual` at all when the exit lock duration has not yet completed; the existing `applyBonusAccrualCheckpoint` logic in `processEventsAndClaimBonus` already handles the cap at `ExitUnlockAt` correctly.

---

### Proof of Concept

1. Alice creates a position in tier T and delegates. `LastBonusAccrual = T0`.
2. Alice triggers exit. `ExitTriggeredAt = T1`, `ExitUnlockAt = T1 + ExitDuration`.
3. Time advances to `T2` where `T2 > ExitUnlockAt`. Alice has accrued bonus rewards over `[T0, ExitUnlockAt]` but has not yet called `ClaimTierRewards`.
4. Alice changes her mind and calls `ClearExit` at block time `T2`.
5. `ClearExit` executes: `ExitTriggeredAt = zero`, `ExitUnlockAt = zero`, `LastBonusAccrual = T2`.
6. Alice's accrued rewards for `[T0, ExitUnlockAt]` are now permanently lost. The next `processEventsAndClaimBonus` call starts from `segmentStart = T2`, skipping the entire accrual window.
7. Alice receives zero bonus for the period she was bonded and exiting, despite being entitled to those rewards. [3](#0-2) [5](#0-4) [6](#0-5)

// computeSegmentBonus computes bonus for a time segment using a snapshot rate.
// Formula: shares * tokensPerShare * tier.BonusApy * durationSeconds / SecondsPerYear
func (k Keeper) computeSegmentBonus(pos types.PositionState, tier types.Tier, segmentStart, segmentEnd time.Time, tokensPerShare math.LegacyDec) math.Int {
	if !pos.ExitUnlockAt.IsZero() && segmentEnd.After(pos.ExitUnlockAt) {
		segmentEnd = pos.ExitUnlockAt
	}

	if !segmentEnd.After(segmentStart) {
		return math.ZeroInt()
	}

	durationSeconds := int64(segmentEnd.Sub(segmentStart) / time.Second)
	if durationSeconds <= 0 {
		return math.ZeroInt()
	}

	tokens := pos.Delegation.Shares.Mul(tokensPerShare)

	return tokens.
		Mul(tier.BonusApy).
		MulInt64(durationSeconds).
		QuoInt64(types.SecondsPerYear).
		TruncateInt()
}
