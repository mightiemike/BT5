### Title
`burnNlp` Applies Weaker Health Check Than All Other Balance-Reducing Operations, Allowing Subaccounts to Enter an Unliquidatable-but-Unhealthy State ‚Äî (`core/contracts/Clearinghouse.sol`)

---

### Summary

`burnNlp` enforces only a `MAINTENANCE` health check after reducing a subaccount's NLP balance, while every other balance-reducing operation in the Clearinghouse enforces the stricter `INITIAL` health check. This inconsistency allows a user to deliberately burn NLP and land in a state where their `INITIAL` health is negative but `MAINTENANCE` health is non-negative ‚Äî a gray zone that no other operation permits, and that the liquidation engine cannot resolve.

---

### Finding Description

In `Clearinghouse.sol`, the following operations all enforce an `INITIAL` health check after modifying a subaccount's balances:

- `withdrawCollateral` ‚Äî [1](#0-0) 
- `transferQuote` ‚Äî [2](#0-1) 
- `mintNlp` ‚Äî [3](#0-2) 
- `nlpProfitShare` ‚Äî [4](#0-3) 

`burnNlp`, however, only checks `MAINTENANCE` health:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
    ERR_SUBACCT_HEALTH
);
``` [5](#0-4) 

The code comment even acknowledges the issue: *"Burning NLP can decrease health if the burn fee exceeds the health improvement from the withdrawal."* The burn fee is `max(1, quoteAmount / 1000)` (‚â•0.1%), which is deducted from the quote returned to the user. [6](#0-5) 

A user whose `INITIAL` health is marginally positive can burn NLP such that the burn fee tips their `INITIAL` health below zero while keeping `MAINTENANCE` health non-negative. The comment claims this check "prevents malicious actors from deliberately creating unhealthy subaccounts," but it only prevents `MAINTENANCE` health from going negative ‚Äî it does not prevent `INITIAL` health from going negative.

`burnNlp` is a user-initiated, signature-validated transaction type (`TransactionType.BurnNlp`) submitted through the sequencer pipeline: [7](#0-6) 

---

### Impact Explanation

A subaccount placed in the `INITIAL < 0, MAINTENANCE >= 0` state is:

1. **Unable to withdraw collateral** ‚Äî `withdrawCollateral` requires `INITIAL >= 0`. [1](#0-0) 
2. **Unable to transfer quote** ‚Äî `transferQuote` requires `_isAboveInitial`. [2](#0-1) 
3. **Unable to mint more NLP** ‚Äî `mintNlp` requires `INITIAL >= 0`. [3](#0-2) 
4. **Not liquidatable** ‚Äî `liquidateSubaccountImpl` requires `MAINTENANCE < 0`. [8](#0-7) 

If the subaccount also carries spot liabilities (borrows), interest accrues via `SpotTick` updates. The subaccount cannot repay through normal withdrawal paths, and the growing liability will eventually push `MAINTENANCE` health negative ‚Äî at which point the account becomes liquidatable with potentially insufficient collateral to cover the debt, creating bad debt absorbed by the insurance fund.

---

### Likelihood Explanation

The trigger requires a user whose `INITIAL` health is close to zero to sign and submit a `BurnNlp` transaction. The 0.1% burn fee is small but deterministic. A user can calculate the exact NLP amount to burn that will tip `INITIAL` health below zero while keeping `MAINTENANCE` health non-negative. This is a deliberate, low-cost, user-controlled action with no external dependencies. The sequencer submits it as a normal fast-mode transaction. [7](#0-6) 

---

### Recommendation

Replace the `MAINTENANCE` health check in `burnNlp` with an `INITIAL` health check, consistent with all other balance-reducing operations:

```solidity
require(
    getHealth(txn.sender, IProductEngine.HealthType.INITIAL) >= 0,
    ERR_SUBACCT_HEALTH
);
```

This aligns `burnNlp` with `withdrawCollateral`, `transferQuote`, `mintNlp`, and `nlpProfitShare`, closing the inconsistency. [9](#0-8) 

---

### Proof of Concept

1. User opens a subaccount with spot liabilities (borrows) and NLP holdings such that `INITIAL` health is just above zero and `MAINTENANCE` health is comfortably positive.
2. User signs a `BurnNlp` transaction for an NLP amount whose burn fee (`max(1, quoteAmount/1000)`) is large enough to push `INITIAL` health below zero.
3. Sequencer submits the transaction via `processTransactionImpl` ‚Üí `clearinghouse.burnNlp`.
4. `burnNlp` deducts NLP, credits quote minus burn fee, and passes the `MAINTENANCE >= 0` check. [10](#0-9) 
5. Post-burn: `INITIAL health < 0`, `MAINTENANCE health >= 0`.
6. `withdrawCollateral` reverts with `ERR_SUBACCT_HEALTH` ‚Äî user cannot exit. [11](#0-10) 
7. `liquidateSubaccountImpl` reverts with `ERR_NOT_LIQUIDATABLE` ‚Äî account cannot be liquidated. [8](#0-7) 
8. Spot interest accrues on the borrow via `SpotTick`. Over time `MAINTENANCE` health crosses zero, the account becomes liquidatable with a deficit, and the insurance fund absorbs the bad debt.

    function burnNlp(
        IEndpoint.BurnNlp calldata txn,
        int128 oraclePriceX18,
        IEndpoint.NlpPool[] calldata nlpPools,
        int128[] calldata nlpPoolRebalanceX18
    ) external onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);

        ISpotEngine spotEngine = _spotEngine();
        spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);

        require(txn.nlpAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        int128 nlpAmount = int128(txn.nlpAmount);
        require(
            spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
            ERR_UNLOCKED_NLP_INSUFFICIENT
        );
        int128 quoteAmount = nlpAmount.mul(oraclePriceX18);
        int128 burnFee = MathHelper.max(ONE, quoteAmount / 1000);
        quoteAmount = MathHelper.max(0, quoteAmount - burnFee);

        _validateNlpRebalance(nlpPools, nlpPoolRebalanceX18, -quoteAmount);
        for (uint128 i = 0; i < nlpPoolRebalanceX18.length; i++) {
            require(nlpPoolRebalanceX18[i] <= 0, ERR_INVALID_NLP_REBALANCE);
        }

        spotEngine.updateBalance(NLP_PRODUCT_ID, txn.sender, -nlpAmount);
        spotEngine.updateBalance(NLP_PRODUCT_ID, N_ACCOUNT, nlpAmount);

        if (quoteAmount > 0) {
            spotEngine.updateBalance(QUOTE_PRODUCT_ID, txn.sender, quoteAmount);
            _applyNlpRebalance(spotEngine, nlpPools, nlpPoolRebalanceX18);
        }

        require(
            spotEngine.getBalance(NLP_PRODUCT_ID, txn.sender).amount >= 0,
            ERR_SUBACCT_HEALTH
        );
        // Burning NLP can decrease health if the burn fee exceeds the health improvement
        // from the withdrawal. This check prevents malicious actors from deliberately
        // creating unhealthy subaccounts through NLP burns.
        require(
            getHealth(txn.sender, IProductEngine.HealthType.MAINTENANCE) >= 0,
            ERR_SUBACCT_HEALTH
        );
    }
