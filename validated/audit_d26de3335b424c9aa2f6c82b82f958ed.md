### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Certifies Attacker-Controlled Internal Merkle Nodes as Valid Transactions - (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a live, publicly callable NEAR contract method. It accepts a fully attacker-controlled `tx_id` that can be an internal Merkle tree node rather than a real transaction hash, and returns `true`, falsely certifying a non-existent Bitcoin transaction as included in a block. Any downstream contract (bridge, atomic swap, cross-chain finance) that consumes this `true` result as proof of a Bitcoin payment is deceived — the same class of bypass as the original report, where a trusted intermediary forwards attacker-controlled data that appears authoritative while skipping the critical validity check.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` carries `#[deprecated]` but that Rust attribute is a compiler-only hint; it does **not** remove the method from the NEAR ABI. The function remains tagged `#[pause]` (callable unless the contract is explicitly paused) and is reachable by any unprivileged NEAR account.

The function's own inline documentation states the broken invariant explicitly:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash. We assume that validation of whether the `tx_id` corresponds to a valid transaction hash is performed at a higher level of verification."

The verification logic is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

`compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs:34-52` simply walks the proof array, hashing left/right at each step, and returns the final hash. It has no concept of tree depth or leaf vs. internal node. If the attacker supplies an internal node hash `I` at depth `d` with a valid proof of length `tree_depth − d` (shorter than the full leaf-to-root path), the computed root equals the real `merkle_root` and the function returns `true`.

The `verify_transaction_inclusion_v2` function (`lib.rs:347`) was introduced specifically to close this gap by requiring a coinbase proof of the same length as the tx proof, forcing the proof to span the full tree depth. But `v1` was never removed from the public interface, leaving the unmitigated path open. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

The light client is the trusted on-chain authority for Bitcoin transaction inclusion. Any bridge, atomic swap, or cross-chain finance contract on NEAR that calls `verify_transaction_inclusion` and acts on a `true` return value (e.g., releasing funds, minting wrapped tokens, settling a swap) can be exploited. An attacker can claim a Bitcoin payment occurred — pointing to a real block and a real `merkle_root` — without ever broadcasting a Bitcoin transaction. The corrupted value is the boolean proof result: `true` is returned for a `tx_id` that does not correspond to any real transaction. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` directly. The attacker needs only publicly available Bitcoin block data (Merkle tree structure), which is trivially obtained from any Bitcoin node or block explorer. The 64-byte transaction Merkle forgery technique is well-documented (referenced in the contract's own comments at `lib.rs:267-268`). The only constraint is that the target block must already be in the light client's mainchain, which is the normal operating state. [4](#0-3) 

---

### Recommendation

- **Short term:** Add `#[private]` to `verify_transaction_inclusion` to prevent any external NEAR account from calling it, or remove it from the public ABI entirely. The `#[deprecated]` Rust attribute provides zero on-chain access control.
- **Long term:** Ensure all consumers of SPV proof results use `verify_transaction_inclusion_v2` exclusively. Audit any downstream contract that may have integrated against the v1 interface. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Identify any Bitcoin block already accepted into the light client's mainchain (e.g., height `H`, block hash `B`, with `N ≥ 4` transactions).
2. Fetch the block's transaction list and compute its Merkle tree. Pick an internal node `I` at depth `d = 1` (the parent of two leaf transactions `T0` and `T1`). `I = double_sha256(T0 || T1)`.
3. The proof from `I` to the root has length `tree_depth − 1` (one step shorter than a leaf proof). Compute this proof using the sibling subtree hashes.
4. Call `verify_transaction_inclusion` on the NEAR contract:
   ```
   tx_id             = I          // internal node, not a real txid
   tx_block_blockhash = B
   tx_index          = 0          // position of I at depth d=1
   merkle_proof      = [sibling_of_I, ..., root_sibling]
   confirmations     = 0
   ```
5. `compute_root_from_merkle_proof(I, 0, proof)` walks the proof and produces the real `merkle_root`. The equality check passes and the function returns **`true`**.
6. A downstream bridge contract that calls `verify_transaction_inclusion` and releases funds on a `true` result is now exploited: no Bitcoin was ever sent. [2](#0-1) [7](#0-6) [8](#0-7)

```rust
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
```

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
```
