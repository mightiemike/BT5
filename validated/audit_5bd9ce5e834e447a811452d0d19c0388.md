### Title
Fee-on-Transfer Token Accounting Corruption in `depositCollateralWithReferral` — (`File: core/contracts/EndpointStorage.sol`, `core/contracts/Endpoint.sol`)

---

### Summary

`handleDepositTransfer` performs two sequential token transfers using the raw `amount` parameter without measuring the actual balance received. The slow-mode deposit transaction queued on-chain also records the original `amount`. When the sequencer later processes the deposit, the user is credited with `amount` regardless of how many tokens the clearinghouse actually received. For fee-on-transfer tokens, this inflates the user's internal collateral balance beyond the real token holdings of the clearinghouse.

---

### Finding Description

The deposit flow in Nado proceeds in two phases:

**Phase 1 — on-chain transfer (user-triggered):**

`Endpoint.depositCollateralWithReferral` calls `handleDepositTransfer`, which performs two sequential transfers using the raw `amount` parameter:

```solidity
// EndpointStorage.sol:111-118
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    safeTransferFrom(token, from, amount);              // user → Endpoint
    safeTransferTo(token, address(clearinghouse), amount); // Endpoint → Clearinghouse
}
```

No balance snapshot is taken before or after either call. The actual tokens received by the clearinghouse are never measured.

**Phase 2 — slow-mode credit (sequencer-triggered):**

Immediately after the transfer, the same raw `amount` is encoded into the slow-mode queue:

```solidity
// Endpoint.sol:152-165
slowModeTxs[_slowModeConfig.txCount++] = SlowModeTx({
    executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY,
    sender: sender,
    tx: abi.encodePacked(
        uint8(TransactionType.DepositCollateral),
        abi.encode(
            DepositCollateral({
                sender: subaccount,
                productId: productId,
                amount: amount          // ← original parameter, not actual received
            })
        )
    )
});
```

When the sequencer processes this slow-mode transaction, `Clearinghouse.depositCollateral` credits the user's spot balance with `txn.amount` — the original `amount`, not the actual tokens the clearinghouse holds.

**Fee-on-transfer trigger:**

For a token that deducts a fee from the recipient on the second leg (`transfer` from Endpoint to Clearinghouse), or for any token that applies a fee asymmetrically (e.g., only on `transfer`, not `transferFrom`, or only when the recipient is a specific contract):

- `safeTransferFrom(token, from, amount)` → Endpoint receives `amount` (no fee on this leg)
- `safeTransferTo(token, clearinghouse, amount)` → Clearinghouse receives `amount − fee`
- Slow-mode tx records `amount`
- User is credited with `amount`

**Result:** `spotEngine.balance[productId][subaccount] = amount`, but clearinghouse holds only `amount − fee`. The protocol's internal accounting is inflated relative to actual token custody.

---

### Impact Explanation

The user's credited collateral balance exceeds the actual tokens held by the clearinghouse. This is a solvency/accounting corruption: the sum of all user balances for a given product can exceed the real token reserve. A user can then withdraw or trade against phantom collateral, draining real funds from other depositors. The corrupted state is `spotEngine` balance for the deposited product, and the clearinghouse's token reserve.

---

### Likelihood Explanation

Any ERC-20 token registered as a spot product that implements a transfer fee (e.g., USDT with fees enabled, tokens with protocol-level transfer taxes, or rebasing tokens) triggers this path. The entry point is fully permissionless — any user can call `depositCollateralWithReferral` or `depositCollateral`. No privileged access is required. The protocol explicitly supports multiple spot tokens via `spotEngine.getToken(productId)`, making the attack surface broad.

---

### Recommendation

Replace the fixed-`amount` double-transfer pattern in `handleDepositTransfer` with a balance-difference approach. Measure the clearinghouse's token balance before and after the transfer, and use the difference as the credited amount in the slow-mode transaction:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then use `actualReceived` (not `amount`) when encoding the `DepositCollateral` slow-mode transaction in `depositCollateralWithReferral`.

---

### Proof of Concept

1. A fee-on-transfer token `T` (fee rate 1%) is registered as a spot product with `productId = P`.
2. Attacker calls `Endpoint.depositCollateralWithReferral(subaccount, P, 1000e6, "")`.
3. `handleDepositTransfer` executes:
   - `safeTransferFrom(T, attacker, 1000e6)` → Endpoint receives `1000e6` (no fee on this leg).
   - `safeTransferTo(T, clearinghouse, 1000e6)` → Clearinghouse receives `990e6` (1% fee deducted).
4. Slow-mode tx is queued with `DepositCollateral{amount: 1000e6}`.
5. Sequencer processes the slow-mode tx; `Clearinghouse.depositCollateral` credits attacker's subaccount with `1000e6`.
6. Attacker's internal balance = `1000e6`; clearinghouse actual holdings = `990e6`.
7. Attacker withdraws `1000e6`, forcing the clearinghouse to pay `10e6` from other depositors' funds.
8. Repeated deposits compound the deficit, draining the pool. [1](#0-0) [2](#0-1) [3](#0-2)
