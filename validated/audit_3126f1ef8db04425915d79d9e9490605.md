### Title
`next_l2_gas_price` and `l2_gas_consumed` Omitted from Block Hash Preimage Despite Protocol Requiring Them Post-0.14.0 — (`File: crates/starknet_api/src/block_hash/block_hash_calculator.rs`)

---

### Summary

`gas_prices_to_hash` in `block_hash_calculator.rs` carries an explicit TODO to include `l2_gas_consumed` and `next_l2_gas_price` in the block hash preimage "after 0.14.0". The current latest deployed version is **0.14.4**, yet the implementation still omits both fields. Because `next_l2_gas_price` is the sole input that seeds the EIP-1559 base-fee for the following block, a proposer can freely forge this value in the block header without invalidating the block hash. Every syncing or validating node then adopts the forged price as the next block's L2 gas price, producing incorrect fee accounting with direct economic impact.

---

### Finding Description

**Incomplete hash preimage — the TODO that was never resolved**

`gas_prices_to_hash` gates on `block_hash_version >= V0_13_4` and hashes only six gas-price scalars:

```
STARKNET_GAS_PRICES0 || l1_gas_price_wei || l1_gas_price_fri
                     || l1_data_gas_price_wei || l1_data_gas_price_fri
                     || l2_gas_price_wei || l2_gas_price_fri
```

The comment immediately above the function body reads:

> `// TODO(Ayelet): add l2_gas_consumed, next_l2_gas_price after 0.14.0.`

The latest versioned-constants file registered in the codebase is `blockifier_versioned_constants_0_14_4.json` / `orchestrator_versioned_constants_0_14_4.json`, confirming the network is already running at 0.14.4. The TODO condition has been satisfied for every block produced since 0.14.1, yet neither `l2_gas_consumed` nor `next_l2_gas_price` has been added to the hash chain.

**The echonet already expects these fields**

The OS-verification service (`echonet/echo_center.py`) passes both fields to the block-hash CLI:

```python
"l2_gas_consumed": fee_market_info["l2_gas_consumed"],
"next_l2_gas_price": fee_market_info["next_l2_gas_price"],
```

This confirms the protocol specification requires them in the hash for current versions, while the Rust implementation silently drops them.

**`next_l2_gas_price` is the EIP-1559 seed for the next block**

`next_l2_gas_price` is stored in `BlockHeaderWithoutHash` and `StorageBlockHeader`. In `try_sync`, a node that catches up by syncing a committed block unconditionally adopts the value from the header:

```rust
self.l2_gas_price = max(
    sync_block.block_header_without_hash.next_l2_gas_price,
    VersionedConstants::latest_constants().min_gas_price,
);
```

The proposer writes `next_l2_gas_price` into the header via `update_state_sync_with_new_block`:

```rust
next_l2_gas_price: self.l2_gas_price,
```

Because the field is not committed to the block hash, the proposer can substitute any value without changing the hash that validators verify. Validators and syncing nodes will accept the block and seed the next block's gas price from the forged value.

---

### Impact Explanation

**Incorrect fee/gas accounting with economic impact (Critical).**

A malicious proposer can:

1. Set `next_l2_gas_price` to an arbitrarily low value (e.g., `min_gas_price`) — all transactions in the next block pay far below the market rate, effectively subsidising the proposer's own transactions or those of colluding parties.
2. Set `next_l2_gas_price` to an arbitrarily high value — all transactions in the next block pay inflated fees; the proposer captures the surplus.
3. Suppress `l2_gas_consumed` to zero — the EIP-1559 formula in `calculate_next_base_gas_price` will compute a lower `next_l2_gas_price` than the actual block utilisation warrants, persistently deflating fees across subsequent blocks.

Because the block hash does not bind these fields, no validator or light client can detect the manipulation by verifying the hash alone.

---

### Likelihood Explanation

Any node that wins a consensus round becomes the proposer for that height. In a permissionless or semi-permissionless validator set this is reachable by any participant. The manipulation requires only writing a different value into the `BlockHeaderWithoutHash` before broadcasting the proposal — no special cryptographic capability is needed. The attack is silent: the forged header passes all existing hash-verification checks.

---

### Recommendation

1. **Implement the TODO immediately.** Add `l2_gas_consumed` and `next_l2_gas_price` to the `gas_prices_to_hash` hash chain for `block_hash_version >= V0_14_1` (or whichever version the protocol specification designates), matching the fields already passed by the echonet CLI.

2. **Add a validator-side check.** Before adopting `next_l2_gas_price` from a synced or proposed block, independently recompute the expected value from `l2_gas_consumed` and the previous block's gas price using `calculate_next_base_gas_price`, and reject the block if the header value diverges beyond the allowed margin.

3. **Regression test.** Extend `test_block_hash_regression` to cover a version ≥ 0.14.1 block that includes non-zero `l2_gas_consumed` and `next_l2_gas_price`, asserting the hash changes when either field changes.

---

### Proof of Concept

**Step 1 — Forge the header.** A proposer node patches `update_state_sync_with_new_block` to write:

```rust
next_l2_gas_price: GasPrice(VersionedConstants::latest_constants().min_gas_price.0),
l2_gas_consumed: GasAmount(0),
```

regardless of actual block utilisation.

**Step 2 — Broadcast the proposal.** The `ProposalFin` commitment is computed from `calculate_block_hash`, which calls `gas_prices_to_hash`. Because neither `next_l2_gas_price` nor `l2_gas_consumed` enters the hash chain, the commitment is identical to what an honest proposer would produce for the same transactions.

**Step 3 — Validators accept.** All validators verify the commitment against the hash of the received header fields that *are* hashed (gas prices, sequencer, timestamp, etc.). The forged fields are invisible to this check. The block reaches quorum and is committed.

**Step 4 — Syncing nodes adopt the forged price.** Any node that calls `try_sync` for this height reads `next_l2_gas_price` from the committed `BlockHeaderWithoutHash` and sets `self.l2_gas_price` to the forged minimum. The next block is built and validated at the artificially suppressed gas price, causing incorrect fee accounting for all transactions in that block.
