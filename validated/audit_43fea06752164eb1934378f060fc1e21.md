### Title
Insufficient Bonus Pool Reverts Block All Position Exit Paths, Locking User Funds — (File: `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

`processEventsAndClaimBonus` reverts with `ErrInsufficientBonusPool` when the rewards pool balance is below the computed bonus. Because every mutation handler (`TierUndelegate`, `ExitTierWithDelegation`, `TierRedelegate`, `AddToTierPosition`, `ClearPosition`) calls `claimRewards` — which calls `processEventsAndClaimBonus` — **before** performing its primary state change, an empty pool atomically blocks all of these operations. Users are unable to undelegate or exit their positions until the pool is externally replenished, locking their staked funds.

---

### Finding Description

`processEventsAndClaimBonus` computes accrued bonus and, before paying it out, checks pool sufficiency: [1](#0-0) 

If the pool balance is below the computed bonus, the function returns `ErrInsufficientBonusPool` and the entire call stack unwinds. This error propagates through `claimRewards`: [2](#0-1) 

Every mutation handler invokes `claimRewards` as a mandatory pre-step before its primary action:

- `TierUndelegate` — calls `claimRewards` at line 166 before `undelegate`
- `TierRedelegate` — calls `claimRewards` at line 229 before `redelegate`
- `AddToTierPosition` — calls `claimRewards` at line 314 before `delegate`
- `ClearPosition` — calls `claimRewards` at line 406 before resetting exit timestamps
- `ExitTierWithDelegation` — calls `claimRewards` at line 532 before `transferDelegationFromPosition` [3](#0-2) [4](#0-3) 

The ADR documents this as intentional for user-driven paths ("fail atomically, user retries after pool is replenished"), but it does not account for the scenario where the pool is not replenished in a timely manner or is deliberately drained: [5](#0-4) 

Contrast this with the `BeforeRedelegationSlashed` hook path, which explicitly swallows `ErrInsufficientBonusPool` to avoid a chain halt: [6](#0-5) 

The same graceful handling is absent from all user-facing exit paths.

---

### Impact Explanation

When the bonus pool is empty or insufficient:

1. `MsgTierUndelegate` fails — the user cannot start the unbonding period and cannot recover their staked tokens.
2. `MsgExitTierWithDelegation` fails — the user cannot transfer their delegation back and cannot exit the tier.
3. `MsgClearPosition` fails — the user cannot cancel a triggered exit, leaving them unable to re-lock or change state.
4. `MsgTierRedelegate` fails — the user cannot move to a different validator.

The corrupted invariant is: **a delegated position's owner must always be able to initiate exit from the tier**. An empty pool breaks this invariant and locks the user's staked tokens inside the module's per-position delegator address indefinitely.

---

### Likelihood Explanation

The rewards pool is a finite module account funded by governance or external deposits. It is consumed by every bonus claim across all positions. As positions accumulate accrued bonus over time, a single large `MsgClaimTierRewards` call (or a sequence of normal claims) can drain the pool to zero. Once empty, every subsequent mutation on any position fails until governance acts. No privileged access is required to trigger the condition — any position owner claiming their legitimate rewards can exhaust the pool. The pool top-up via `BeginBlocker` only covers base rewards, not the bonus pool: [7](#0-6) 

---

### Recommendation

Apply the same graceful handling used in `slashRedelegationPosition` to all user-facing exit paths. When `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` inside `TierUndelegate` and `ExitTierWithDelegation`, the handler should log the shortfall, skip the bonus payout (or record a pending debt), and allow the primary operation (undelegation, exit) to proceed. This mirrors the fix applied in the Synthetix analog: handle the blocking condition without reverting the outer call.

---

### Proof of Concept

1. A set of positions accumulates large accrued bonus over time.
2. One or more owners call `MsgClaimTierRewards`, draining the bonus pool to zero.
3. A different user with a delegated position calls `MsgTierUndelegate` (or `MsgExitTierWithDelegation`).
4. The handler calls `claimRewards` → `processEventsAndClaimBonus`.
5. `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` at `claim_rewards.go:230`.
6. The error propagates; `TierUndelegate` returns an error; the transaction is rejected.
7. The user's staked tokens remain locked inside the position's delegator address with no exit path until governance replenishes the pool. [1](#0-0) [8](#0-7)

	}

	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	bonus, err := k.processEventsAndClaimBonus(ctx, &pos)
	if err != nil {
		return types.PositionState{}, nil, nil, err
	}

	return pos, base, bonus, nil
}

func (ms msgServer) TierUndelegate(ctx context.Context, msg *types.MsgTierUndelegate) (*types.MsgTierUndelegateResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateUndelegatePosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	srcValidator := pos.Delegation.ValidatorAddress
	valAddr, err := sdk.ValAddressFromBech32(srcValidator)
	if err != nil {
		return nil, err
	}

	delAddr, err := sdk.AccAddressFromBech32(pos.DelegatorAddress)
	if err != nil {
		return nil, errorsmod.Wrap(sdkerrors.ErrInvalidAddress, "invalid delegator address")
	}

	completionTime, _, err := ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)
	if err != nil {
		return nil, err
	}

	pos.ClearBonusCheckpoints()

	if err := ms.setPosition(ctx, pos.Position, &ValidatorTransition{PreviousAddress: srcValidator}); err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventPositionUndelegated{
		PositionId:     pos.Id,
		TierId:         pos.TierId,
		Owner:          pos.Owner,
		Validator:      srcValidator,
		CompletionTime: completionTime,
	}); err != nil {
		return nil, err
	}

	return &types.MsgTierUndelegateResponse{
		CompletionTime: completionTime,
		PositionId:     pos.Id,
	}, nil

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)
	poolBalance := k.bankKeeper.GetAllBalances(ctx, poolAddr)
	if !poolBalance.IsAllGTE(bonus) {
		return errorsmod.Wrapf(types.ErrInsufficientBonusPool,
			"bonus: %s, pool balance: %s",
			bonus.String(), poolBalance.String())
	}

	return nil
