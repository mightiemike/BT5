### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery — (`contract/src/lib.rs`)

---

### Summary

The contract explicitly documents the 64-byte transaction Merkle proof forgery vulnerability and introduces `verify_transaction_inclusion_v2` as the fix. However, the original `verify_transaction_inclusion` function remains `pub` and is callable by any unprivileged NEAR account. The only in-path check that appears to validate the proof — `require!(!args.merkle_proof.is_empty())` — is logically insufficient to prevent the attack. Any caller can bypass the coinbase-proof mitigation in v2 by invoking the deprecated function directly, violating the invariant that "transaction inclusion proofs are only valid for actual leaf transactions."

---

### Finding Description

`verify_transaction_inclusion` is marked `#[deprecated]` and carries an explicit warning:

> "This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash." [1](#0-0) 

Despite this, the function is still declared `pub` and gated only by `#[pause]`, meaning any unprivileged NEAR account can call it whenever the contract is not paused. [2](#0-1) 

The only check inside the function that appears to guard proof validity is:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [3](#0-2) 

This check is analogous to the external report's `if (minUsdAmountOut < toUsdAmount(boostAmount)) minUsdAmountOut = toUsdAmount(boostAmount)`: it appears to enforce a correctness invariant but is logically insufficient. It only rejects empty proofs; it does not prevent an attacker from supplying an internal merkle tree node as `tx_id` with a valid non-empty proof path.

The intended fix, `verify_transaction_inclusion_v2`, enforces the invariant by:
1. Requiring `merkle_proof.len() == coinbase_merkle_proof.len()` (same tree depth).
2. Verifying the coinbase transaction is at index 0 with a valid proof. [4](#0-3) 

This ensures the target `tx_id` must be at the same depth as the coinbase (i.e., a leaf), preventing internal-node forgery. But because v1 is still `pub`, any caller can skip v2 entirely and call v1 directly.

---

### Impact Explanation

Any downstream NEAR contract or off-chain system that calls `verify_transaction_inclusion` for security-critical decisions — cross-chain bridges, conditional payment release, SPV-based authorization — can be deceived into accepting a forged transaction inclusion proof. The contract returns `true` for a `tx_id` that is an internal merkle tree node, not a real transaction. The invariant explicitly stated in the code's own warning is violated by a direct, unprivileged call.

---

### Likelihood Explanation

The entry path requires no privileges: any NEAR account can call `verify_transaction_inclusion` when the contract is unpaused. The 64-byte attack is a well-documented technique (referenced in the contract's own deprecation comment via the BitMEX blog post). An attacker who is a miner can craft a 64-byte transaction to produce a usable internal node; even without mining, historical Bitcoin blocks contain internal nodes that can be used. The attack is realistic and the entry path is fully open.

---

### Recommendation

Remove the `pub` visibility from `verify_transaction_inclusion`, or make it `pub(crate)` / private. If backward compatibility is required, add the same coinbase-proof guard from v2 directly into v1, or unconditionally redirect v1 to v2 internally. The deprecation notice alone does not close the attack surface.

---

### Proof of Concept

1. Identify a Bitcoin block `B` where an internal merkle tree node `N` is 64 bytes (or craft one as a miner).
2. Compute a valid merkle proof `P` for `N` at its position `idx` in the tree (the proof is valid because `N` is a real node in the tree).
3. Call `verify_transaction_inclusion` on the NEAR contract:
   - `tx_id = N` (the internal node, claimed as a transaction)
   - `tx_block_blockhash = hash(B)`
   - `tx_index = idx`
   - `merkle_proof = P` (non-empty, so the `require!` passes)
   - `confirmations = 1`
4. The function computes `compute_root_from_merkle_proof(N, idx, P)`, which equals `B.merkle_root` because `N` is genuinely in the tree.
5. The function returns `true` for a transaction that does not exist.

The coinbase check in `verify_transaction_inclusion_v2` is never reached because the caller invoked v1 directly. [5](#0-4)

    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
