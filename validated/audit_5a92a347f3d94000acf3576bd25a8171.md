### Title
`PerpEngineState.balances` Public Mapping Returns Stale `vQuoteBalance` Inconsistent with `getBalance()` — (File: `core/contracts/PerpEngineState.sol`)

---

### Summary

`PerpEngineState` declares its `balances` mapping as `public`, which auto-generates a Solidity getter that returns the raw stored `Balance` struct without applying pending funding payment adjustments. The protocol's canonical read path, `getBalance()`, applies `_updateBalance()` to account for accrued funding before returning. Any external caller reading `balances(productId, subaccount)` directly receives a stale `vQuoteBalance` that diverges from the value the protocol itself uses for health, liquidation, and settlement accounting.

---

### Finding Description

`PerpEngineState` stores perp balances in a `public` mapping: [1](#0-0) 

The canonical read function `getBalance()` loads the stored balance and then calls `_updateBalance(state, balance, 0, 0)` before returning: [2](#0-1) 

Inside `_updateBalance`, the key adjustment is:

```solidity
int128 diffX18 = cumulativeFundingAmountX18 - balance.lastCumulativeFundingX18;
int128 deltaQuote = vQuoteDelta - diffX18.mul(balance.amount);
balance.vQuoteBalance += deltaQuote;
``` [3](#0-2) 

This adjusts `vQuoteBalance` by the funding payments that have accrued since the last time the balance was written to storage. The stored `balances[productId][subaccount]` is only updated when `_setBalanceAndUpdateBitmap` is called: [4](#0-3) 

Between writes, the stored `vQuoteBalance` is stale. The auto-generated public getter `balances(productId, subaccount)` returns this stale value directly, while `getBalance()` returns the funding-adjusted value. The two diverge by exactly `-diffX18.mul(balance.amount)` — which grows with position size and elapsed funding time.

The internal `_getBalance()` override correctly routes through `getBalance()`: [5](#0-4) 

So the protocol's own health and liquidation logic (`BaseEngine._calculateProductHealth`, `Clearinghouse.getHealth`) always uses the funding-adjusted value. The inconsistency is exposed only through the public mapping getter.

---

### Impact Explanation

External protocols or off-chain systems that integrate with Nado by reading `balances(productId, subaccount)` directly — a natural choice given the mapping is `public` — will receive a `vQuoteBalance` that does not reflect pending funding payments. This can cause:

- **Incorrect PnL assessment**: `vQuoteBalance` is the unrealized quote component of a perp position. Stale values overstate or understate a trader's actual PnL.
- **Incorrect collateral/health calculations** in any external protocol that uses Nado perp balances as collateral inputs.
- **Incorrect liquidation decisions** by external liquidation bots or aggregators that read raw storage instead of calling `getBalance()`.

The discrepancy is not bounded by rounding — it scales linearly with `balance.amount` and the elapsed funding rate differential, and can be arbitrarily large for large positions over long periods.

---

### Likelihood Explanation

The `balances` mapping is `public`, making its auto-generated getter part of the contract's ABI. Any developer integrating with Nado who inspects the ABI or reads the source will see `balances` as a callable view function and may use it instead of `getBalance()`. This is the same integration pitfall as the Hats `balanceOfBatch` issue: the "wrong" path is equally visible and callable as the "right" one.

---

### Recommendation

Change the `balances` mapping visibility from `public` to `internal` in `PerpEngineState`:

```solidity
// Before
mapping(uint32 => mapping(bytes32 => Balance)) public balances;

// After
mapping(uint32 => mapping(bytes32 => Balance)) internal balances;
``` [1](#0-0) 

This forces all callers — both internal and external — to use `getBalance()`, which correctly applies the funding adjustment via `_updateBalance()`. If read access to the raw normalized storage is needed for tooling, a dedicated view function that explicitly documents the "stale, unadjusted" semantics should be provided instead.

---

### Proof of Concept

1. User opens a long perp position. Storage: `balances[productId][subaccount] = {amount: 100e18, vQuoteBalance: 0, lastCumulativeFundingX18: F0}`.
2. Time passes; the sequencer submits a `PerpTick`. `state.cumulativeFundingLongX18` increases to `F0 + 0.01e18` (1% funding accrued). The user's balance in storage is **not** updated.
3. External integrator calls `balances(productId, subaccount)` → receives `vQuoteBalance = 0` (stale).
4. External integrator calls `getBalance(productId, subaccount)` → `_updateBalance` computes `diffX18 = 0.01e18`, `deltaQuote = -0.01e18 * 100e18 / 1e18 = -1e18`, returns `vQuoteBalance = -1e18` (correct: user owes 1 unit of quote in funding).
5. The discrepancy is `1e18` quote units. For a position of 10,000 units at 1% funding, the discrepancy is `100e18` — a material misstatement of the user's financial position.
