### Title
Linked Signer Can Hijack Signing Authority via `LinkSigner` Transaction — (`core/contracts/EndpointTx.sol`)

### Summary

In `processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner=true`. This allows the current linked signer — not just the subaccount owner — to sign and submit a new `LinkSigner` transaction that replaces the linked signer with an attacker-controlled address. The subaccount owner is then locked out of the fast path and cannot revoke the new linked signer before their collateral is drained.

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner=true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` to the verifier when `allowLinkedSigner=true`: [2](#0-1) 

`Verifier.validateSignature` accepts a signature from **either** the subaccount owner address **or** the linked signer address: [3](#0-2) 

After validation passes, `linkedSigners[signedTx.tx.sender]` is unconditionally overwritten with the new signer: [4](#0-3) 

By contrast, the slow-mode path for `LinkSigner` correctly uses `validateSender`, which checks `msg.sender == address(uint160(bytes20(txn.sender)))` — restricting the operation to the actual on-chain owner: [5](#0-4) 

The fast path has no equivalent owner-only guard.

### Impact Explanation

An attacker who is already a linked signer for a victim subaccount can:

1. Sign `LinkSigner(sender=victim_subaccount, signer=attacker2, nonce=N)` using their own key.
2. Submit it via the sequencer through `processTransactionImpl`.
3. `validateSignedTx` passes because `recovered == getLinkedSigner(victim_subaccount) == attacker`.
4. `linkedSigners[victim_subaccount]` is set to `attacker2`.

The victim can only recover via the slow-mode path, which has a hardcoded 3-day delay (`SLOW_MODE_TX_DELAY`). During that window, `attacker2` — also a linked signer — can sign `WithdrawCollateral` (also validated with `allowLinkedSigner=true`) to drain all collateral before the revocation executes. [6](#0-5) 

The broken invariant: **only the subaccount owner should be able to modify the linked signer relationship**. A linked signer is a delegated trading key, not an account-management key.

### Likelihood Explanation

The precondition — that the attacker is already a linked signer — is a normal, supported protocol state. Users routinely link hot-wallet keys as signers for trading bots or API integrations. Any such linked key can execute this attack without any additional privileges, leaked secrets, or governance capture.

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the subaccount owner may change the linked signer
);
```

This aligns the fast path with the slow-mode path, which already enforces owner-only authorization via `validateSender`.

### Proof of Concept

```solidity
// 1. Victim links attacker as signer (legitimate prior action)
//    linkedSigners[victim_subaccount] = attacker

// 2. Attacker constructs and signs a new LinkSigner tx
IEndpoint.LinkSigner memory inner = IEndpoint.LinkSigner({
    sender: victim_subaccount,
    signer: bytes32(uint256(uint160(attacker2))),
    nonce: currentNonce
});
bytes memory sig = sign(attacker_key, EIP712digest(inner));

// 3. Sequencer submits via processTransactionImpl
endpoint.processTransactionImpl(abi.encodePacked(
    uint8(IEndpoint.TransactionType.LinkSigner),
    abi.encode(IEndpoint.SignedLinkSigner({ tx: inner, signature: sig }))
));

// 4. Assert: linkedSigners[victim_subaccount] == attacker2
// 5. attacker2 signs WithdrawCollateral to drain all funds
//    before victim's 3-day slow-mode revocation executes
```

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
