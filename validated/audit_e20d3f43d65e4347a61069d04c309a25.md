### Title
SwapAllowlistExtension checks router address instead of originating trader, allowing any non-allowlisted trader to bypass the swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` gates swaps using the `sender` argument supplied by the pool, which is always `msg.sender` of the `pool.swap()` call. When a trader routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating trader. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][trader]`. If the router is allowlisted (a natural configuration for a pool that wants to support router-mediated swaps), every trader — regardless of their own allowlist status — can bypass the restriction.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-240
_beforeSwap(
    msg.sender,   // ← router address when called via router
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, completely ignoring the originating trader's address.

A pool admin who allowlists the router (to permit router-mediated swaps) inadvertently opens the pool to all traders, because the router address passes the check for every swap it intermediates.

### Impact Explanation
Any trader who is **not** individually allowlisted can execute swaps on a restricted pool and receive output tokens, as long as the router is allowlisted. The allowlist — the pool's primary access-control mechanism — is rendered ineffective for all router-mediated swaps. This is a direct bypass of a core pool protection, allowing unauthorized fund flows out of a pool that was configured to be restricted.

### Likelihood Explanation
The scenario requires `allowedSwapper[pool][router]=true`. A pool admin who wants to support both router-mediated swaps and individual trader restrictions would naturally allowlist the router, not realizing that doing so grants unrestricted access to all router users. The configuration is reasonable and the flaw is non-obvious, making accidental misconfiguration likely in production deployments.

### Recommendation
The extension must resolve the originating trader identity rather than the immediate `pool.swap()` caller. Options:

1. **Pass the originating trader through `extensionData`**: the router encodes `msg.sender` (the trader) into `extensionData`, and the extension decodes and checks it. This requires a trusted encoding convention.
2. **Check `recipient` instead of `sender`**: if the pool's design guarantees that `recipient` is the economic beneficiary, gate on that. However, `recipient` can be a third party, so this may not match intent.
3. **Require direct pool calls for allowlisted pools**: document that `SwapAllowlistExtension` is incompatible with router intermediation and enforce this at the pool configuration level.

The cleanest fix is option 1: the router encodes `msg.sender` into `extensionData` and the extension verifies it, so the originating trader is always the checked identity.

### Proof of Concept
```solidity
// Foundry integration test sketch
function test_routerBypassesSwapAllowlist() public {
    // Admin allowlists only the router (not the trader)
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // Non-allowlisted EOA trader routes through the router
    vm.prank(trader); // trader is NOT in allowedSwapper[pool]
    uint256 amountOut = router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(token0),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1000,
            amountOutMinimum: 0,
            recipient: trader,
            deadline: type(uint256).max,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    // Swap succeeds and trader receives output tokens despite not being allowlisted

    assertGt(token1.balanceOf(trader), 0);
}
```

The swap succeeds because `beforeSwap` receives `sender=router`, `allowedSwapper[pool][router]=true` passes, and the pool transfers output tokens to the non-allowlisted trader.
