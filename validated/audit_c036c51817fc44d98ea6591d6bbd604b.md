Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, enabling allowlist bypass via `MetricOmmPoolLiquidityAdder` — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller of `pool.addLiquidity`) and gates only on `owner` (the position holder). Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a caller-supplied `owner` with no requirement that `owner == msg.sender`, any unprivileged address can bypass the deposit allowlist by naming an already-approved address as `owner` while paying with its own tokens.

## Finding Description

The `beforeAddLiquidity` hook in `DepositAllowlistExtension` has the following signature:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
```

The first `address` parameter — `sender` — is unnamed and completely ignored. The check at line 38 evaluates only `owner`:

```solidity
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

The pool correctly passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner` to `_beforeAddLiquidity`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` faithfully encodes and forwards both arguments to the extension: [3](#0-2) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts a free `owner` parameter and only validates it is non-zero: [4](#0-3) 

`_validateOwner` only rejects `address(0)`: [5](#0-4) 

**Full exploit path:**
1. Pool is deployed with `DepositAllowlistExtension` in `BEFORE_ADD_LIQUIDITY_ORDER`. Alice is allowlisted; Bob is not.
2. Bob calls `adder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`.
3. The adder calls `pool.addLiquidity(alice, salt, deltas, abi.encode(KIND_PAY), "")`.
4. The pool calls `_beforeAddLiquidity(adder_address, alice, ...)`.
5. The extension checks `allowedDepositor[pool][alice]` → **passes**.
6. Bob's tokens are pulled in the callback (`payer = msg.sender = bob`), minted into Alice's position.

Bob — never approved — has deposited into a restricted pool. The same path works via `addLiquidityWeighted(pool, alice, ...)` since the probe call also passes Alice's address through the allowlist check. [6](#0-5) 

## Impact Explanation

The deposit allowlist is the primary access-control mechanism for restricted pools (KYC-gated, regulatory-compliant, or curated LP pools). Any unprivileged address can bypass it entirely by routing through `MetricOmmPoolLiquidityAdder` and naming any already-approved address as `owner`. The pool receives liquidity from an unauthorized depositor, breaking the core invariant the extension was deployed to enforce. This constitutes an **admin-boundary break: factory/oracle role checks are bypassed by an unprivileged path** and **broken core pool functionality**. [7](#0-6) 

## Likelihood Explanation

- `MetricOmmPoolLiquidityAdder` is a public, permissionless contract with no access controls on `addLiquidityExactShares`.
- The only precondition is knowing one allowlisted address, which is trivially discoverable on-chain from `AllowedToDepositSet` events.
- No privileged role, special token, or flash-loan is required.
- The bypass works on every pool that uses `DepositAllowlistExtension` with a non-`allowAll` configuration. [4](#0-3) 

## Recommendation

Gate on `sender` (the actual transaction initiator) rather than `owner` in `DepositAllowlistExtension.beforeAddLiquidity`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Note: when the adder is used, `sender` will be the adder contract address, not the end user. If per-end-user gating through the adder is also required, the adder must forward the original `msg.sender` as an additional argument, or the allowlist must be checked at the adder level before the pool call. [1](#0-0) 

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.35;

// Setup: pool deployed with DepositAllowlistExtension in BEFORE_ADD_LIQUIDITY_ORDER.
// Alice is allowlisted (allowedDepositor[pool][alice] = true). Bob is NOT allowlisted.

contract DepositAllowlistBypassTest is Test {
    function test_bypass() public {
        // Bob calls as msg.sender, supplies alice as owner.
        // _validateOwner: alice != address(0) → passes.
        // beforeAddLiquidity receives (adder, alice, ...) and checks allowedDepositor[pool][alice] → true.
        // Bob's tokens are pulled in the callback; Alice's position is credited.
        vm.prank(bob);
        adder.addLiquidityExactShares(
            pool,
            alice,   // authorized owner used to pass the allowlist
            0,
            deltas,
            type(uint256).max,
            type(uint256).max,
            ""
        );
        // Bob has deposited into a pool he was never approved for.
        // Verify: Alice's position has shares; Bob's token balance decreased.
    }
}
