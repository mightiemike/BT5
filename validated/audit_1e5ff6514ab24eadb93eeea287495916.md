### Title
Sanctions Check Applied to Slow-Mode Withdrawal Transactions Permanently Locks Deposited Collateral for Sanctioned Users — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx.submitSlowModeTransactionImpl` applies `requireUnsanctioned(sender)` unconditionally to every slow-mode transaction, including `WithdrawCollateral`. A user who is added to the sanctions list after depositing collateral cannot use the slow-mode path — the protocol's only censorship-resistant, user-initiated fallback — to recover their funds. If the sequencer simultaneously refuses to process their withdrawal (the expected behavior for sanctioned addresses), the deposited collateral is permanently locked inside the clearinghouse.

---

### Finding Description

`EndpointStorage.requireUnsanctioned` reverts if the Chainalysis oracle marks an address as sanctioned: [1](#0-0) 

In `Endpoint.depositCollateralWithReferral`, this check is applied to both `msg.sender` and the subaccount owner before accepting a deposit: [2](#0-1) 

In `EndpointTx.submitSlowModeTransactionImpl`, the same check is applied to `sender` for **every** slow-mode transaction type before the transaction is enqueued: [3](#0-2) 

The slow-mode transaction types that can be submitted by a regular user include `WithdrawCollateral`: [4](#0-3) 

Because the `requireUnsanctioned` gate sits before the transaction is enqueued — not just before deposits — a sanctioned user is blocked from submitting a `WithdrawCollateral` slow-mode transaction entirely. The slow-mode path is the only on-chain, user-controlled withdrawal fallback that does not depend on sequencer cooperation. The sequencer-driven fast path (`processTransactionImpl`, lines 413–436) has no sanctions check of its own, but a sequencer operating in compliance with sanctions policy will refuse to include withdrawal transactions for sanctioned addresses. The result is that both paths are closed simultaneously, mirroring the `whenNotPaused` pattern in the reference report.

---

### Impact Explanation

A user who deposits collateral and is subsequently added to the Chainalysis sanctions list loses access to their deposited funds:

- The slow-mode `WithdrawCollateral` submission reverts at `requireUnsanctioned` in `submitSlowModeTransactionImpl`.
- The sequencer-processed fast path is not available to a sanctioned user in a compliant deployment.
- No other on-chain path exists for a non-privileged user to recover collateral from the clearinghouse.

The exact corrupted asset state is the user's collateral balance recorded in `SpotEngine`, which can never be transferred back to the user's wallet. This is a permanent collateral lock, not a temporary denial of service.

---

### Likelihood Explanation

The Chainalysis oracle is a live, externally-operated contract. Addresses are added to it regularly and without prior notice. Any Nado user who deposits collateral before being sanctioned — or whose address is incorrectly flagged — faces this condition. The sequencer's compliance-driven refusal to process withdrawals for sanctioned addresses is the expected operational posture, making the combined lock realistic rather than theoretical.

---

### Recommendation

Remove `requireUnsanctioned` from the withdrawal branch of `submitSlowModeTransactionImpl`. The sanctions check is appropriate for deposits (preventing new funds from entering the protocol from a sanctioned source) but must not be applied to `WithdrawCollateral` or any other transaction type whose purpose is to return already-deposited funds to the user. Specifically, in `submitSlowModeTransactionImpl`, gate the `requireUnsanctioned` call so it only executes when the transaction type is `DepositCollateral` or `DepositInsurance`, and skip it for `WithdrawCollateral`.

---

### Proof of Concept

1. User `Alice` calls `depositCollateralWithReferral` and deposits 10,000 USDC. The deposit succeeds because Alice is not yet sanctioned.
2. Alice is subsequently added to the Chainalysis sanctions oracle.
3. Alice attempts to recover her funds by calling `submitSlowModeTransaction` with a `WithdrawCollateral` payload.
4. Execution reaches `submitSlowModeTransactionImpl` → line 375: `requireUnsanctioned(sender)` → `sanctions.isSanctioned(Alice)` returns `true` → revert with `ERR_WALLET_SANCTIONED`.
5. Alice submits the same transaction to the sequencer; the sequencer refuses to include it in compliance with sanctions policy.
6. Alice has no remaining on-chain path to withdraw her 10,000 USDC. The collateral remains locked in the clearinghouse indefinitely. [5](#0-4) [1](#0-0)

```text
    function submitSlowModeTransactionImpl(bytes calldata transaction) public {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );

        // special case for DepositCollateral because upon
        // slow mode submission we must take custody of the
        // actual funds

        address sender = msg.sender;

        if (txType == IEndpoint.TransactionType.DepositCollateral) {
            revert();
        } else if (txType == IEndpoint.TransactionType.DepositInsurance) {
            IEndpoint.DepositInsurance memory txn = abi.decode(
                transaction[1:],
                (IEndpoint.DepositInsurance)
            );
            require(
                txn.amount >= uint128(SLOW_MODE_FEE),
                ERR_DEPOSIT_TOO_SMALL
            );
            handleDepositTransfer(_getQuote(), sender, uint256(txn.amount));
        } else if (
            txType == IEndpoint.TransactionType.WithdrawInsurance ||
            txType == IEndpoint.TransactionType.DelistProduct ||
            txType == IEndpoint.TransactionType.DumpFees ||
            txType == IEndpoint.TransactionType.RebalanceXWithdraw ||
            txType == IEndpoint.TransactionType.UpdateTierFeeRates ||
            txType == IEndpoint.TransactionType.AddNlpPool ||
            txType == IEndpoint.TransactionType.UpdateNlpPool ||
            txType == IEndpoint.TransactionType.DeleteNlpPool ||
            txType == IEndpoint.TransactionType.ForceRebalanceNlpPool ||
            txType == IEndpoint.TransactionType.NlpProfitShare ||
            txType == IEndpoint.TransactionType.UpdateBuilder
        ) {
            require(sender == owner());
        } else {
            chargeSlowModeFee(_getQuote(), sender);
            slowModeFees += SLOW_MODE_FEE;
        }

        IEndpoint.SlowModeConfig memory _slowModeConfig = slowModeConfig;
        requireUnsanctioned(sender);
        slowModeTxs[_slowModeConfig.txCount++] = IEndpoint.SlowModeTx({
            executableAt: uint64(block.timestamp) + SLOW_MODE_TX_DELAY, // hardcoded to three days
            sender: sender,
            tx: transaction
        });
        // TODO: to save on costs we could potentially just emit something
        // for now, we can just create a separate loop in the engine that queries the remote
        // sequencer for slow mode transactions, and ignore the possibility of a reorgy attack
        slowModeConfig = _slowModeConfig;
    }
```
