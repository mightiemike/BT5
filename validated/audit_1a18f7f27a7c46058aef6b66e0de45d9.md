### Title
P2P Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Producing a Different Transaction Hash Preimage - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation in the P2P sync path uses a zero-value heuristic to classify resource bounds. For a transaction submitted via RPC as `AllResources { l1_gas: non_zero, l2_gas: 0, l1_data_gas: 0 }`, the original hash is computed with three resource elements in the preimage. After protobuf round-trip, the same transaction is silently reclassified as `L1Gas`, and any subsequent hash recomputation uses only two resource elements, producing a divergent hash.

---

### Finding Description

**Serialization boundary — `crates/apollo_protobuf/src/converters/transaction.rs`, lines 417–436:**

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← 2-element hash preimage
} else {
    ValidResourceBounds::AllResources(...)       // ← 3-element hash preimage
})
``` [1](#0-0) 

The heuristic is: if both `l1_data_gas` and `l2_gas` are zero after deserialization, the variant is `L1Gas`. This is intended to distinguish pre-0.13.3 transactions from post-0.13.3 ones. However, it also fires for a post-0.13.3 `AllResources` transaction where the user legitimately set `l2_gas = 0` and `l1_data_gas = 0` (only `l1_gas` non-zero). The gateway's stateless validator explicitly accepts this case: [2](#0-1) 

**Hash preimage divergence — `crates/starknet_api/src/transaction_hash.rs`, lines 188–211:**

`get_tip_resource_bounds_hash` produces structurally different outputs for the two variants:

- `ValidResourceBounds::L1Gas(_)` → hashes `[tip, L1_GAS_concat, L2_GAS_concat]` (3 Poseidon inputs)
- `ValidResourceBounds::AllResources(...)` → hashes `[tip, L1_GAS_concat, L2_GAS_concat, L1_DATA_GAS_concat]` (4 Poseidon inputs) [3](#0-2) 

Even when `l2_gas` and `l1_data_gas` are numerically zero, the Poseidon hash over 3 elements differs from the hash over 4 elements. The `tip_resource_bounds_hash` is a direct input to `get_invoke_transaction_v3_hash`: [4](#0-3) 

**Round-trip failure path:**

1. User submits `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }`.
2. Gateway converts to `InternalRpcInvokeTransactionV3` (always `AllResourceBounds`), computes hash H₁ using `AllResources` (4-element Poseidon input).
3. Transaction is included in a block and propagated via P2P as `protobuf::InvokeV3`.
4. Receiving peer deserializes via `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`: `l1_data_gas = 0` and `l2_gas = 0` → classified as `ValidResourceBounds::L1Gas`.
5. Any hash recomputation on the receiving peer produces H₂ ≠ H₁ (3-element Poseidon input). [5](#0-4) 

The `InternalRpcInvokeTransactionV3` and `RpcInvokeTransactionV3` structs both use `AllResourceBounds` directly (never `L1Gas`), so the gateway-side hash is always computed with the 4-element preimage: [6](#0-5) 

The `InvokeTransactionV3` used in the P2P sync path uses `ValidResourceBounds`, which is where the downgrade occurs: [7](#0-6) 

---

### Impact Explanation

**High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

The `tip_resource_bounds_hash` is a committed field in the Starknet transaction hash. A transaction whose resource bounds are `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` has a canonical hash that includes the `l1_data_gas` element. After protobuf round-trip through the P2P sync path, the deserialized `InvokeTransactionV3` carries `L1Gas` resource bounds, and any call to `calculate_transaction_hash` or `validate_transaction_hash` on the receiving side produces a divergent hash. This breaks the hash-binding invariant for this class of transactions across the P2P boundary.

---

### Likelihood Explanation

**Low–Medium.** The trigger condition — `AllResources` with `l2_gas = 0` and `l1_data_gas = 0` — is an edge case but is explicitly accepted by the gateway's stateless validator (`valid_l1_gas` test case). No privilege is required; any user can submit such a transaction. The conversion is deterministic and silent (no error is returned).

---

### Recommendation

Replace the zero-value heuristic with an explicit version/era marker. The protobuf `ResourceBounds` message should carry a discriminant field (e.g., a boolean `is_all_resources` or a version tag) so that `AllResources` with zero `l2_gas`/`l1_data_gas` round-trips correctly. Alternatively, the `TryFrom` implementation should default to `AllResources` when `l1_data_gas` is absent (the pre-0.13.3 legacy case) rather than when it is zero, since zero is a valid post-0.13.3 value. The existing TODO comment at line 426 acknowledges this is a temporary workaround: [8](#0-7) 

---

### Proof of Concept

```
1. Craft RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds {
         l1_gas:      { max_amount: 1000, max_price_per_unit: 1 },
         l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
         l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
     }

2. Submit to gateway → accepted (matches `valid_l1_gas` test case).

3. Gateway computes H₁ = get_invoke_transaction_v3_hash(AllResources{...})
   → tip_resource_bounds_hash uses 4-element Poseidon:
      poseidon(tip, L1_GAS_packed, L2_GAS_packed_zero, L1_DATA_GAS_packed_zero)

4. Transaction propagated via P2P as protobuf::InvokeV3.
   Serialized ResourceBounds: l1_gas=1000/1, l2_gas=0/0, l1_data_gas=0/0.

5. Receiving peer deserializes:
   l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas(l1_gas)

6. Receiving peer computes H₂ = get_invoke_transaction_v3_hash(L1Gas{...})
   → tip_resource_bounds_hash uses 3-element Poseidon:
      poseidon(tip, L1_GAS_packed, L2_GAS_packed_zero)

7. H₁ ≠ H₂  (different Poseidon arity, same numeric inputs)
   → hash validation fails on receiving peer.
```

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
