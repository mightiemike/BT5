Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any caller to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` inside `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the user. If the pool admin allowlists the router to enable router-mediated swaps for legitimate users, every address on the network can bypass the allowlist by calling the router, rendering the access control ineffective.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol lines 230-240
_beforeSwap(
    msg.sender,   // whoever called pool.swap()
    recipient,
    ...
    extensionData
);
```

`SwapAllowlistExtension.beforeSwap` checks that identity against the per-pool allowlist:

```solidity
// SwapAllowlistExtension.sol lines 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (correct) and `sender` is whoever called `pool.swap()`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, passing `""` as `callbackData` and `params.extensionData` (user-supplied) as `extensionData`. No user-identity information is injected into the call:

```solidity
// MetricOmmSimpleRouter.sol lines 71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The `_setNextCallbackContext` stores the real user (`msg.sender`) only for the payment callback — it is never forwarded to the pool's `beforeSwap` hook. The pool therefore sees `msg.sender = router`, and the extension checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`.

The same pattern applies to `exactInput` (line 103-112), `exactOutputSingle` (line 135-137), and `exactOutput` (line 165-181) — all call `pool.swap()` with the router as `msg.sender`.

The admin faces an impossible choice:
- **Do not allowlist the router**: all router-mediated swaps fail, even for allowlisted users.
- **Allowlist the router**: every address on the network can swap via the router, defeating the allowlist entirely.

No existing guard compensates for this. The `onlyPool` modifier in `BaseMetricExtension` only verifies that the caller of `beforeSwap` is a registered pool — it does not validate the `sender` argument passed into the hook.

## Impact Explanation
Any user not on the pool's swap allowlist can bypass the restriction by calling `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) on a pool whose admin has allowlisted the router. Pools that rely on `SwapAllowlistExtension` to restrict access to specific counterparties (e.g., KYC'd traders, institutional participants, or whitelisted strategies) are exposed to unrestricted public swapping. This is an admin-boundary break: an unprivileged path (`MetricOmmSimpleRouter`) bypasses a pool-admin-configured access control, allowing unauthorized users to execute swaps and receive pool output tokens they are not entitled to.

## Likelihood Explanation
Allowlisting the router is the natural and expected step for any pool admin who wants their allowlisted users to use the standard periphery. The router is the primary user-facing entry point. A pool admin who allowlists the router to enable normal UX inadvertently opens the pool to all users. The bypass requires no special privileges, no flash loans, and no unusual token behavior — only a standard router call available to any address.

## Recommendation
The extension must gate the ultimate user, not the immediate caller of `pool.swap()`. Two complementary fixes:

1. **Router-side**: Have `MetricOmmSimpleRouter` encode the originating `msg.sender` into `extensionData` for each hop (e.g., as a standard ABI-encoded "originator" prefix), so extensions can read the real user identity from `extensionData`.

2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should decode the real user from `extensionData` when `sender` is a known router, or the protocol should define a standard "originator" field in `extensionData` that all periphery contracts populate and the extension validates.

Alternatively, document explicitly that `SwapAllowlistExtension` only gates direct pool callers and that router-mediated swaps must be handled by a separate mechanism (e.g., a router-level allowlist).

## Proof of Concept
```
Setup:
  pool P configured with SwapAllowlistExtension E
  allowedSwapper[P][alice] = true          // alice is the only allowed swapper
  allowedSwapper[P][router] = true         // admin allowlists router so alice can use it

Attack:
  bob (not allowlisted) calls:
    router.exactInputSingle({pool: P, recipient: bob, ...})

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient=bob, ...)      // msg.sender = router inside pool
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[P][router] == true  ✓  (no revert)
        → swap executes, bob receives tokens

  bob successfully swaps on a pool he is not allowlisted for.

Direct swap check (for comparison):
  bob calls pool.swap() directly:
    → _beforeSwap(sender=bob, ...)
      → allowedSwapper[P][bob] == false → revert NotAllowedToSwap  ✓
