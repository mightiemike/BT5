Audit Report

## Title
SwapAllowlistExtension Gates Router Address Instead of Actual Trader, Enabling Full Allowlist Bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks `sender`, which is `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` resolves to the router's address. If the pool admin allowlists the router — a necessary step for allowlisted users to reach the pool through the standard periphery — every user of that router, including non-allowlisted ones, bypasses the swap gate entirely. The allowlist provides zero protection for all router-mediated swaps.

## Finding Description

**Root cause — wrong identity checked in the hook**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and gates on it: [1](#0-0) 

`sender` is populated by `MetricOmmPool.swap` as `msg.sender` of the pool call: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards it verbatim to the extension: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is `msg.sender` of `pool.swap()`, so `sender = router`: [4](#0-3) 

The extension never sees the actual user's address.

**The forced dilemma**

If the admin allowlists alice individually (`allowedSwapper[pool][alice] = true`) but does NOT allowlist the router, alice calling the router produces `sender = router` → check fails → reverts. Allowlisted users cannot use the router at all unless the router itself is allowlisted. Once the router is allowlisted, the check `allowedSwapper[pool][router] == true` passes for every caller of the router, including non-allowlisted users.

**Contrast with `DepositAllowlistExtension`**

The deposit allowlist correctly gates on `owner` (the position beneficiary), which is invariant to who pays or calls: [5](#0-4) 

The swap allowlist has no equivalent stable identity — it gates the intermediary, not the trader.

**No existing guard compensates**

The router passes user-supplied `extensionData` directly to the pool without encoding the actual caller's address. There is no mechanism in `MetricOmmSimpleRouter` to automatically inject the real user identity into `extensionData` for the extension to decode and verify. [6](#0-5) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to known counterparties (e.g., a permissioned institutional pool) becomes open to any user of the public router once the router is allowlisted. Non-allowlisted users can execute swaps, extract LP value at oracle-anchored prices, and drain one-sided bins. This is a direct loss of LP principal and a broken core pool invariant (the allowlist guard). Severity: High — direct loss of user principal through a broken access control mechanism on a core pool action.

## Likelihood Explanation

The scenario requires the pool admin to allowlist the router, which is a necessary and expected action for allowlisted users to reach the pool through the standard periphery. `MetricOmmSimpleRouter` is a public, permissionless contract. No privileged access, no special setup, and no non-standard tokens are required. Any user can call `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` on the router to exploit this. The condition is easily and repeatably triggered.

## Recommendation

Gate on the actual user identity rather than the direct caller. Two options:

1. **Check `recipient`** — the address that receives output tokens is the economically relevant party for a swap. Change the check to `allowedSwapper[msg.sender][recipient]`. This mirrors the `DepositAllowlistExtension` pattern of gating on the beneficiary.
2. **Pass the originating user through `extensionData`** — the router encodes the real user address in `extensionData`, and the extension decodes and checks it. This requires a coordinated router + extension design and must be authenticated (e.g., the extension trusts only a known router to supply this data).

Option 1 is simpler and consistent with the existing `DepositAllowlistExtension` design pattern already present in the codebase.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension
  allowedSwapper[pool][router] = true   // admin enables router access for allowlisted users
  allowedSwapper[pool][alice]  = true   // alice is individually whitelisted
  allowedSwapper[pool][bob]    = false  // bob is NOT whitelisted

Attack:
  bob calls MetricOmmSimpleRouter.exactInputSingle(pool, ...)
    → router calls pool.swap(recipient=bob, ...)
    → MetricOmmPool._beforeSwap(sender=router, ...)
    → SwapAllowlistExtension.beforeSwap(sender=router, ...)
    → allowedSwapper[pool][router] == true → passes
    → swap executes, bob receives tokens

Result:
  bob, a non-allowlisted user, successfully swaps on a restricted pool.
  SwapAllowlistExtension provided zero protection.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension.
  2. setAllowedToSwap(pool, router, true).
  3. Prank as bob (not allowlisted).
  4. Call router.exactInputSingle with pool as target.
  5. Assert swap succeeds and bob receives output tokens.
  6. Assert allowedSwapper[pool][bob] == false (confirming bypass).
```

    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(

        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
