### Title
`DelegateAction` NEP-461 signed payload omits chain ID, enabling cross-network replay by an unprivileged relayer â€” (`core/primitives/src/action/delegate.rs`)

### Summary

`DelegateAction::get_nep461_hash()` produces a signing digest that commits to message type (NEP discriminant) and action content but contains no chain identifier. A malicious relayer that receives a user-signed `DelegateAction` intended for mainnet can submit it verbatim on testnet (or any other NEAR network) and, if the nonce and `max_block_height` are still valid there, the runtime will accept the signature and execute the inner actions against the user's account on the wrong network.

### Finding Description

`DelegateAction::get_nep461_hash()` builds the signed digest as:

```rust
let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
hash(&bytes)
```

`SignableMessage` serialises to `(MessageDiscriminant, &T)` where `MessageDiscriminant` is a 4-byte NEP number:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,   // NEP-366 = 0x4000016E
    pub msg: &'a T,
}
```

The `DelegateAction` payload contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`. None of these fields are network-specific. The `MessageDiscriminant` provides message-type separation (preventing a `DelegateAction` signature from being misread as a raw transaction) but provides **no chain separation**. There is no genesis hash, chain ID string, or any other network-scoped value anywhere in the signed bytes.

The runtime's `apply_delegate_action` verifies only:
1. Signature matches `get_nep461_hash()` â€” passes on any network
2. Nonce is greater than the stored access-key nonce â€” passes if the target network's nonce is lower
3. Current block height â‰¤ `max_block_height` â€” passes if the target network's height is lower

No chain ID check exists anywhere in the delegate-action execution path.

Contrast with validator-layer messages, which are implicitly chain-scoped because they commit to a `chunk_hash` or `block_hash` that is chain-specific. `DelegateAction` has no such anchor.

### Impact Explanation

A malicious relayer can:
1. Receive a user's signed `DelegateAction` (e.g., a token transfer or `AddKey`) intended for mainnet.
2. Submit it unchanged on testnet (or any other NEAR network where the same account exists).
3. If the user's testnet nonce is lower than the signed nonce and the testnet block height is below `max_block_height`, the runtime accepts the signature and executes the inner actions.

Consequences include unintended token transfers, unintended access-key additions or deletions, and unintended contract calls â€” all on a network the user never authorised. Because `max_block_height` is a block count (not a timestamp), and mainnet and testnet heights diverge, a `DelegateAction` signed with `max_block_height = current_mainnet_height + 100` may remain valid on testnet for an arbitrarily long window.

### Likelihood Explanation

The relayer role is explicitly untrusted in the meta-transaction design â€” the entire point of NEP-366 is that the user does not trust the relayer with their key, only with submission. Many NEAR users maintain the same account ID and access keys on both mainnet and testnet. Nonces on a freshly-used testnet account are often lower than on mainnet. The attack requires no special privilege: any entity that receives a signed `DelegateAction` (e.g., a public relayer service) can attempt the replay.

### Recommendation

Introduce a network-scoped domain separator into `SignableMessage` or into `DelegateAction` itself. The minimal fix is to add the chain ID (available as a string from `genesis_config.chain_id`) to the signed payload, analogous to EIP-712's `chainId` field. A cleaner approach is to extend `MessageDiscriminant` or `SignableMessage` with an optional `chain_id: Option<String>` field that is required for on-chain delegate actions, so that a signature produced for `"mainnet"` cannot verify on `"testnet"`.

### Proof of Concept

1. On mainnet, user signs:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "bob.near",
     actions: [Transfer { deposit: 1_000_000 }],
     nonce: 5,
     max_block_height: 120_000_000,
     public_key: <alice's key>,
   }
   ```
   Digest = `sha256(borsh(NEP_366_discriminant || delegate_action_bytes))` â€” no chain ID.

2. Malicious relayer wraps the same `SignedDelegateAction` in a testnet transaction and submits it.

4. Testnet runtime checks nonce (5 > alice's testnet nonce of 3 âś“) and block height (testnet at 80M < 120M âś“).

5. Transfer executes on testnet against alice's testnet balance without her authorisation.
