### Title
`validate_contract_code_response` accepts unauthenticated `ContractCodeResponse::V2` messages pre-`SignedContractCodeResponse` feature activation — (`chain/client/src/stateless_validation/validate.rs`)

### Summary

`validate_contract_code_response` gates signature enforcement on `ProtocolFeature::SignedContractCodeResponse` being enabled, but never rejects the post-feature wire variant (`ContractCodeResponse::V2`) when the feature is not yet active. Any network peer can craft a `ContractCodeResponse::V2` carrying arbitrary contract bytes and have it accepted and stored by a chunk validator without any authentication, because the variant-vs-protocol-version consistency check is absent.

### Finding Description

`ContractCodeResponse` is a versioned enum with two Borsh-discriminated variants:

- `V1` (discriminant `0`): unsigned, no `responder` field — the pre-`SignedContractCodeResponse` wire format
- `V2` (discriminant `1`): carries a `responder: AccountId` and a `signature: Signature` — the post-feature wire format

The sender side correctly gates which variant is emitted:

```rust
// core/primitives/src/stateless_validation/contract_distribution.rs
pub fn encode(..., protocol_version: ProtocolVersion) -> std::io::Result<Self> {
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        ContractCodeResponseV2::encode(...).map(Self::V2)
    } else {
        ContractCodeResponseV1::encode(...).map(Self::V1)
    }
}
```

The receiver side, however, only enforces the signature when the feature is enabled and does nothing else when it is not:

```rust
// chain/client/src/stateless_validation/validate.rs
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)   // ← V2 accepted unauthenticated pre-feature
}
```

Post-feature, `validate_witness_contract_code_response_signature` correctly rejects V1 because `response.responder()` returns `None` for V1 and the function immediately errors:

```rust
// chain/client/src/stateless_validation/validate.rs
fn validate_witness_contract_code_response_signature(...) -> Result<(), Error> {
    let Some(responder) = response.responder() else {
        return Err(Error::Other(
            "Unsigned contract code response in epoch where signature is required".to_owned(),
        ));
    };
    ...
}
```

The symmetry is broken: post-feature V1 is rejected, but pre-feature V2 is silently accepted. Because `ContractCodeResponse::V2` is a valid Borsh-deserializable type (discriminant `1`), any network peer can craft and send one. The `validate_chunk_relevant` check only verifies that the embedded `ChunkProductionKey` falls within the current height window — public information available from block headers — and imposes no sender-identity constraint.

After validation passes, the contracts are stored unconditionally:

```rust
// chain/client/src/stateless_validation/partial_witness/partial_witness_actor.rs
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    if !validate_contract_code_response(...)?.is_relevant() { return Ok(()); }
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
```

### Impact Explanation

A malicious network peer (no validator stake required) can, during any epoch whose protocol version is below 85 (pre-`SignedContractCodeResponse`):

1. Observe a live `ChunkProductionKey` from block headers.
2. Craft `ContractCodeResponse::V2` with an arbitrary `responder`, arbitrary compressed contract bytes, and a garbage `signature`.
3. Send it to one or more chunk validators before the legitimate response arrives.
4. The validator stores the injected bytes under the target `ChunkProductionKey`.
5. When the validator attempts to execute the chunk witness using the cached contracts, it operates on attacker-controlled code, causing witness validation to fail and the endorsement to be withheld.

Targeting a quorum of validators for the same shard simultaneously can prevent the chunk from accumulating sufficient endorsements, stalling block production for that shard.

### Likelihood Explanation

`SignedContractCodeResponse` activates at protocol version 85; the current stable binary is at version 86 (`STABLE_PROTOCOL_VERSION = 86`) and `MIN_SUPPORTED_PROTOCOL_VERSION = 84`. The window is the epoch(s) during which the network transitions from version 84 to 85. Any peer-level attacker with knowledge of the upgrade schedule and the ability to send network messages can race the legitimate response. No privileged role is required.

### Recommendation

Add a symmetric variant-vs-version check in `validate_contract_code_response`:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    } else if matches!(response, ContractCodeResponse::V2(_)) {
        // V2 is a post-feature format; reject it before the feature activates
        return Err(Error::Other(
            "ContractCodeResponse::V2 received before SignedContractCodeResponse activation".to_owned(),
        ));
    }
    Ok(ChunkRelevance::Relevant)
}
```

This mirrors the existing post-feature rejection of V1 (via the missing `responder`) and closes the pre-feature injection path for V2.

```rust
pub enum ContractCodeResponse {
    V1(ContractCodeResponseV1) = 0,
    V2(ContractCodeResponseV2) = 1,
}

impl ContractCodeResponse {
    pub fn encode(
        next_chunk: ChunkProductionKey,
        contracts: &Vec<CodeBytes>,
        signer: &ValidatorSigner,
        protocol_version: ProtocolVersion,
    ) -> std::io::Result<Self> {
        if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
            ContractCodeResponseV2::encode(next_chunk, contracts, signer).map(Self::V2)
        } else {
            ContractCodeResponseV1::encode(next_chunk, contracts).map(Self::V1)
        }
    }

    pub fn chunk_production_key(&self) -> &ChunkProductionKey {
        match self {
            Self::V1(v1) => &v1.next_chunk,
            Self::V2(v2) => &v2.inner.next_chunk,
        }
    }

    /// Account that produced this response. Available only for signed variants.
    pub fn responder(&self) -> Option<&AccountId> {
        match self {
            Self::V1(_) => None,
            Self::V2(v2) => Some(&v2.inner.responder),
        }
    }

    pub fn decompress_contracts(&self) -> std::io::Result<Vec<CodeBytes>> {
        let compressed_contracts = match self {
            Self::V1(v1) => &v1.compressed_contracts,
            Self::V2(v2) => &v2.inner.compressed_contracts,
        };
        compressed_contracts.decode().map(|(data, _size)| data)
    }

    /// Verifies the signature for signed variants. Returns `false` for
    /// unsigned variants since there is nothing to verify.
    pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
        match self {
            Self::V1(_) => false,
            Self::V2(v2) => v2.verify_signature(public_key),
        }
    }
