### Title
Cross-Chain Merkle Proof Replay in `_verifyProof` â€” (`core/contracts/Airdrop.sol`)

---

### Summary

The Merkle leaf in `_verifyProof` is constructed from only `(sender, totalAmount)`. It does not commit to `block.chainid` or the contract address. If the owner registers the same Merkle root on two or more chains, a proof generated for chain A is cryptographically identical and valid on chain B, allowing an attacker to claim the same weekly allocation on every chain where that root is registered.

---

### Finding Description

The leaf is built as:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(keccak256(abi.encode(sender, totalAmount)))
);
``` [1](#0-0) 

Neither `block.chainid` nor `address(this)` is included. The `claimed` mapping that guards against double-claiming is stored per-contract instance:

```solidity
mapping(uint32 => mapping(address => uint256)) internal claimed;
``` [2](#0-1) 

So `claimed[week][sender]` on chain A is entirely independent of `claimed[week][sender]` on chain B. The only check that could differ across chains is `merkleRoots[week]`, but if the owner registers the same root on both chains (a natural operational choice for a uniform multi-chain airdrop), the proof passes on both.

The full verification path is:

1. `claim()` â†’ `_claim()` â†’ `_verifyProof()` [3](#0-2) 
2. `_verifyProof` checks `claimed[week][sender] == 0`, verifies the chain-agnostic leaf, sets `claimed[week][sender] = totalAmount` [4](#0-3) 
3. `_claim` transfers `totalAmount` tokens [5](#0-4) 

---

### Impact Explanation

An attacker with a valid proof for week W on chain A can submit the identical `(week, totalAmount, proof)` tuple to the `claim()` function on chain B. Both calls succeed independently. The attacker receives `totalAmount` tokens on every chain where the same root is registered, multiplying their allocation by the number of chains. This is a direct, unauthorized token extraction â€” a Critical asset-loss impact.

---

### Likelihood Explanation

Nado is a multi-chain protocol. Registering the same weekly Merkle root across all deployed chains is the operationally straightforward approach for a uniform airdrop. There is no on-chain mechanism that prevents the owner from doing so, and no warning in the contract. The precondition is realistic and likely under normal operations.

---

### Recommendation

Bind the leaf to the chain and contract instance by including `block.chainid` and `address(this)` in the encoded data:

```solidity
bytes32 leaf = keccak256(
    bytes.concat(
        keccak256(abi.encode(block.chainid, address(this), sender, totalAmount))
    )
);
```

This makes every leaf unique per deployment, so a proof generated for one chain is invalid on any other chain even when the same Merkle root is registered.

---

### Proof of Concept

1. Deploy two `Airdrop` instances (same bytecode, different chains or forked environments) with the same `token` and `merkleRoot` for week 1.
2. Call `registerMerkleRoot(1, root)` on both instances.
3. Generate a valid Merkle proof for `(attacker, 1000e18)` against `root`.
4. Call `claim([{week:1, totalAmount:1000e18, proof:proof}])` on instance A â†’ succeeds, `claimed[1][attacker] = 1000e18` on A, attacker receives 1000 tokens.
5. Call `claim([{week:1, totalAmount:1000e18, proof:proof}])` on instance B â†’ succeeds, `claimed[1][attacker] = 1000e18` on B, attacker receives another 1000 tokens.
6. Assert both transfers completed â€” attacker holds 2000 tokens instead of 1000.
