Audit Report

## Title
SwapAllowlistExtension Bypass via MetricOmmSimpleRouter: Router Address Checked Instead of Actual Swapper — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When swaps are routed through `MetricOmmSimpleRouter`, the pool and extension observe the router's address as `sender`, not the originating user. A pool admin who allowlists the router address — a necessary operational step for permitted users to reach the pool via the standard router — inadvertently grants every caller of the public router the ability to bypass the per-user allowlist.

## Finding Description
`MetricOmmPool.swap()` captures `msg.sender` and passes it as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` — i.e., `allowedSwapper[pool][router]` — when the call originates from the router: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and all other router entry points) calls `pool.swap()` directly, making the router the identity the pool and extension observe: [4](#0-3) 

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the explicit `owner` parameter — the actual position owner — not the immediate `msg.sender` of `addLiquidity`. The swap path has no equivalent explicit-user parameter: [5](#0-4) 

No existing guard in the extension, pool, or router recovers the originating user's identity. The `extensionData` field is passed through but never populated by the router with caller identity, and the extension does not inspect it.

## Impact Explanation
`SwapAllowlistExtension` is the sole on-chain mechanism for restricting who may trade against a pool. Once bypassed, any unpermitted address can execute swaps against a pool whose admin believed it was access-controlled. This constitutes a broken core pool functionality causing direct loss of LP principal: unpermitted swappers can execute one-sided trades at oracle prices, drain bins, or front-run permitted users while the allowlist appears correctly configured on-chain.

## Likelihood Explanation
The bypass requires the pool admin to allowlist the router address. This is a natural and expected operational step: without it, permitted users cannot use the standard router at all. The admin has no in-code warning that allowlisting the router is equivalent to disabling the per-user allowlist for all router callers. The condition is therefore a realistic production configuration, not a contrived or adversarial setup.

## Recommendation
1. **Pass the originating user through `extensionData`**: The router should encode `msg.sender` (the actual user) into `extensionData` before calling `pool.swap()`. `SwapAllowlistExtension.beforeSwap` should decode and check that address when `extensionData` is non-empty, falling back to `sender` for direct pool calls.
2. **Add an explicit `swapper` parameter to `swap()`**: Analogous to the `owner` parameter in `addLiquidity`, an explicit `swapper` field would let the extension gate on the economically relevant actor rather than the immediate caller, aligning the swap and deposit allowlist patterns.
3. **Document the identity semantics**: Clearly state in `SwapAllowlistExtension` that `sender` is `msg.sender` of `pool.swap()`, so pool admins understand that allowlisting any intermediary contract grants that contract's entire caller set access.

## Proof of Concept
```
Setup:
  pool configured with SwapAllowlistExtension
  allowAllSwappers[pool] = false
  allowedSwapper[pool][permittedUser] = true
  allowedSwapper[pool][router]        = true   ← admin adds this to enable router access for permitted users

Attack:
  unpermittedUser calls MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})
  → router calls pool.swap(...)          msg.sender = router
  → pool calls _beforeSwap(router, ...)
  → extension checks allowedSwapper[pool][router] == true  ✓
  → swap executes; unpermittedUser receives output tokens

Result:
  unpermittedUser bypasses the per-user allowlist and trades against the restricted pool.
  The pool admin's access-control invariant is silently broken.

Foundry test outline:
  1. Deploy pool with SwapAllowlistExtension.
  2. setAllowAllSwappers(pool, false).
  3. setAllowedToSwap(pool, permittedUser, true).
  4. setAllowedToSwap(pool, address(router), true).
  5. vm.prank(unpermittedUser); router.exactInputSingle({pool: pool, ...}).
  6. Assert swap succeeds (no NotAllowedToSwap revert) and unpermittedUser receives output tokens.
```
