### Title
`mainchain_header_to_height` Not Cleaned Up During Chain Reorganization Causes Stale Index Entries and Premature Common-Ancestor Detection — (`contract/src/lib.rs`)

### Summary

During a chain reorganization in `reorg_chain()`, demoted main-chain blocks are removed from `mainchain_height_to_header` and `headers_pool`, but their entries are **never removed** from `mainchain_header_to_height`. This is the exact same desynchronization class as the `allocatedAmt` bug: one accounting structure is updated while a parallel one is left stale. A subsequent reorg whose fork chain passes through a previously-demoted block will incorrectly identify that block as the common ancestor, causing the promotion loop to terminate early and leaving the canonical-chain index in a corrupted state.

### Finding Description

`reorg_chain()` performs two distinct removal passes.

**Pass 1 — trim excess main-chain tip blocks** (lines 586–593):

```rust
for height in (fork_tip_height + 1)..=last_main_chain_block_height {
    let current_main_chain_blockhash = self
        .mainchain_height_to_header
        .get(&height)
        .unwrap_or_else(|| env::panic_str("cannot get a block"));
    self.remove_block_header(&current_main_chain_blockhash);   // removes from headers_pool
    self.mainchain_height_to_header.remove(&height);           // removes height→hash
    // ❌ mainchain_header_to_height NOT updated
}
```

**Pass 2 — promote fork blocks, evict displaced main-chain blocks** (lines 626–636):

```rust
let main_chain_block = self
    .mainchain_height_to_header
    .insert(&current_height, &current_block_hash);
self.mainchain_header_to_height
    .insert(&current_block_hash, &current_height);   // adds fork block

if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash);   // removes from headers_pool
    // ❌ mainchain_header_to_height NOT updated for evicted block
}
```

After either pass, every demoted block's hash remains as a live key in `mainchain_header_to_height`.

The common-ancestor detection loop in the same function reads that map:

```rust
while !self
    .mainchain_header_to_height
    .contains_key(&fork_header_cursor.block_hash)   // ← stale entries cause false positives
{
    ...
}
```

**Concrete attack scenario:**

1. Main chain: `A → B → C → D` (D is tip, height 3).
2. Attacker submits fork `A → B → E → F → G` with more cumulative work. Reorg fires:
   - `C` and `D` are removed from `mainchain_height_to_header` and `headers_pool`.
   - `C` and `D` **remain** in `mainchain_header_to_height` (stale).
   - `E`, `F`, `G` are inserted into both maps. New tip: `G`.
3. Attacker submits fork `A → B → C → D → H → I` with even more work. Reorg fires again:
   - Walk starts at `I`, steps to `H`, then to `D`, then to `C`.
   - `mainchain_header_to_height.contains_key(C)` → **true** (stale entry).
   - Loop exits. Contract treats `C` as the common ancestor.
   - Only `D`, `H`, `I` are promoted; `E`, `F`, `G` are **not** demoted.
   - `mainchain_tip_blockhash` is set to `I`, but heights 2–4 in `mainchain_height_to_header` still point to `E`, `F`, `G`.
   - The canonical-chain index is now permanently split: the tip hash says one chain, the height index says another.

The same stale-entry accumulation occurs in `run_mainchain_gc()` (lines 401–414), which also removes from `mainchain_height_to_header` without touching `mainchain_header_to_height`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

After the corrupted reorg, `mainchain_tip_blockhash` points to the attacker's chosen tip while `mainchain_height_to_header` maps heights to a different, older fork. Any call to `verify_transaction_inclusion()` that resolves a block by height will silently verify against the wrong block, returning `true` for transactions that are not in the canonical chain. Downstream contracts or bridges that consume SPV proofs will accept fraudulent inclusion proofs. The canonical-chain invariant — that `mainchain_tip_blockhash`, `mainchain_height_to_header`, and `mainchain_header_to_height` all agree on the same linear chain — is permanently broken until a privileged migration or redeployment.

### Likelihood Explanation

The trigger requires two sequential reorgs where the second fork's ancestry passes through a block demoted by the first reorg. This is a realistic Bitcoin network condition (competing mining pools, selfish mining, or a deliberate attacker controlling enough hashrate to produce valid PoW headers). The `submit_blocks()` entry point is callable by any unprivileged NEAR account, so no privileged access is needed. The attacker only needs to supply valid PoW headers, which is the normal relayer input.

### Recommendation

In `reorg_chain()`, whenever a block is removed from `mainchain_height_to_header` (both in the trim pass and in the promotion loop), also remove its hash from `mainchain_header_to_height`:

```rust
// Pass 1
self.mainchain_height_to_header.remove(&height);
self.mainchain_header_to_height.remove(&current_main_chain_blockhash); // add this

// Pass 2 (inside the promotion loop)
if let Some(current_main_chain_blockhash) = main_chain_block {
    self.remove_block_header(&current_main_chain_blockhash);
    self.mainchain_header_to_height.remove(&current_main_chain_blockhash); // add this
}
```

Apply the same fix in `run_mainchain_gc()` for every block removed from `mainchain_height_to_header`.

### Proof of Concept

```
State before attack:
  headers_pool:              {A, B, C, D, E, F, G}
  mainchain_height_to_header: {0→A, 1→B, 2→C, 3→D}
  mainchain_header_to_height: {A→0, B→1, C→2, D→3}
  mainchain_tip_blockhash:   D

Step 1 — submit fork [E,F,G] (more work than D):
  reorg_chain() removes C,D from height_to_header; promotes E,F,G
  height_to_header: {0→A, 1→B, 2→E, 3→F, 4→G}
  header_to_height: {A→0, B→1, C→2, D→3, E→2, F→3, G→4}  ← C,D still present
  tip: G

Step 2 — submit fork [C,D,H,I] (more work than G):
  reorg_chain() walk: I → H → D → C
  contains_key(C) == true  ← stale hit, loop exits
  Promotes only D,H,I; never demotes E,F,G
  height_to_header: {0→A, 1→B, 2→E, 3→D, 4→H, 5→I}  ← E still at height 2
  tip: I

verify_transaction_inclusion(tx, block_at_height=2) now resolves to E (old fork block),
not C (the block the caller expects), silently returning a wrong proof result.
```
