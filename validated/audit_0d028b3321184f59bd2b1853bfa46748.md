### Title
Censorable Airdrop Token Permanently Locks Rewards for Blacklisted Users — (File: `core/contracts/Airdrop.sol`)

---

### Summary

The `claim()` function in `Airdrop.sol` calls `SafeERC20.safeTransfer` to push airdrop rewards directly to `msg.sender`. If the configured airdrop token is a censorable ERC-20 (e.g., USDC or USDT) and the claimant's address is blacklisted by that token, every claim attempt reverts. Because there is no fallback path, the user's accrued rewards are permanently unclaimable and remain locked in the contract.

---

### Finding Description

`Airdrop._claim` is the internal function executed for every entry in the `claim()` loop:

```solidity
// core/contracts/Airdrop.sol  lines 65-73
function _claim(
    uint32 week,
    uint256 totalAmount,
    bytes32[] calldata proof
) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);  // ← hard revert
    emit Claim(msg.sender, week, totalAmount);
}
``` [1](#0-0) 

`claim()` iterates over every `ClaimProof` supplied by the caller and invokes `_claim` for each:

```solidity
// core/contracts/Airdrop.sol  lines 75-83
function claim(ClaimProof[] calldata claimProofs) external {
    for (uint32 i = 0; i < claimProofs.length; i++) {
        _claim(claimProofs[i].week, claimProofs[i].totalAmount, claimProofs[i].proof);
    }
}
``` [2](#0-1) 

`_verifyProof` writes `claimed[week][sender] = totalAmount` **before** the transfer executes:

```solidity
// core/contracts/Airdrop.sol  lines 42-63
claimed[week][sender] = totalAmount;   // written before safeTransfer
``` [3](#0-2) 

If `safeTransfer` reverts, the whole transaction reverts (including the `claimed` write), so the user can retry — but every retry will also revert. The contract provides no alternative delivery path (e.g., a pull-pattern, a delegated-recipient override, or a try/catch skip). The `token` address is fixed at initialization and cannot be changed:

```solidity
// core/contracts/Airdrop.sol  lines 24-31
function initialize(address _token, address _sanctions) external initializer {
    ...
    token = _token;
``` [4](#0-3) 

The existing sanctions guard only checks the Chainalysis list (`ISanctionsList.isSanctioned`); it does not cover token-level blacklists:

```solidity
// core/contracts/Airdrop.sol  lines 53-56
require(
    !ISanctionsList(sanctions).isSanctioned(sender),
    "address is sanctioned."
);
``` [5](#0-4) 

A user can pass the sanctions check and still be blacklisted at the token level (USDC/USDT maintain independent blacklists). Once blacklisted, every call to `claim()` reverts at `safeTransfer`, and the user's rewards are permanently stranded.

---

### Impact Explanation

The user's entire unclaimed airdrop balance across all registered weeks becomes permanently unclaimable. The tokens remain locked in the `Airdrop` contract with no recovery mechanism. This is a direct, irreversible asset loss for the affected user.

---

### Likelihood Explanation

USDC and USDT are the most common stablecoin choices for reward programs. Both maintain on-chain blacklists that can be updated at any time by their issuers. A user can be blacklisted after the merkle root is registered (i.e., after their entitlement is already committed on-chain), making the timing window realistic. No admin action within Nado is required to trigger the condition — only the normal use of a censorable token as the airdrop asset.

---

### Recommendation

Replace the push-transfer pattern with a pull-pattern, or wrap the transfer in a try/catch that skips failed transfers and emits an event, allowing the owner to arrange an alternative delivery:

```solidity
function _claim(uint32 week, uint256 totalAmount, bytes32[] calldata proof) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    try IERC20(token).transfer(msg.sender, totalAmount) returns (bool ok) {
        require(ok, "Transfer returned false");
        emit Claim(msg.sender, week, totalAmount);
    } catch {
        // revert claimed state so user can retry or admin can assist
        claimed[week][msg.sender] = 0;
        emit ClaimFailed(msg.sender, week, totalAmount);
    }
}
```

Alternatively, allow users to specify a `recipient` address distinct from `msg.sender`, so a blacklisted address can direct rewards to a non-blacklisted wallet.

---

### Proof of Concept

1. Owner deploys and initializes `Airdrop` with USDC as `token`.
2. Owner calls `registerMerkleRoot(1, root)` committing week-1 allocations.
3. Alice earns 1 000 USDC in week 1; her leaf is included in `root`.
4. USDC issuer blacklists Alice's address (e.g., due to a compliance action).
5. Alice calls `claim([{week:1, totalAmount:1000e6, proof:[...]}])`.
6. `_verifyProof` passes (Alice is not on the Chainalysis list).
7. `SafeERC20.safeTransfer(USDC, alice, 1000e6)` reverts — USDC blacklist check fails.
8. Entire transaction reverts; `claimed[1][alice]` remains 0.
9. Alice retries — same revert every time.
10. Alice's 1 000 USDC is permanently locked in the `Airdrop` contract.
