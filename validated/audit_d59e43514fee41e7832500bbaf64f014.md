### Title
Oracle Price Not Validated for Zero in `mintNlp`/`burnNlp` — (`core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.mintNlp` and `Clearinghouse.burnNlp` accept an `oraclePriceX18` parameter and immediately write it to the spot engine's price store and use it in NLP/quote calculations — without any check that the value is positive. The `updatePrice` path in the same contract explicitly enforces `require(txn.priceX18 > 0, ERR_INVALID_PRICE)`, making this an inconsistency with a concrete financial impact.

---

### Finding Description

`Clearinghouse.updatePrice` correctly rejects a zero or negative price:

```solidity
// Clearinghouse.sol line 367
require(txn.priceX18 > 0, ERR_INVALID_PRICE);
``` [1](#0-0) 

However, `mintNlp` and `burnNlp` accept `oraclePriceX18` without any equivalent guard:

```solidity
// Clearinghouse.sol line 462, 466
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);   // writes unvalidated price
int128 nlpAmount = quoteAmount.div(oraclePriceX18);        // divides by unvalidated price
``` [2](#0-1) 

```solidity
// Clearinghouse.sol line 494, 502
spotEngine.updatePrice(NLP_PRODUCT_ID, oraclePriceX18);   // writes unvalidated price
int128 quoteAmount = nlpAmount.mul(oraclePriceX18);        // multiplies by unvalidated price
``` [3](#0-2) 

The `oraclePriceX18` field lives in the outer `SignedMintNlp` / `SignedBurnNlp` structs, separate from the inner user-signed `MintNlp` / `BurnNlp` structs:

```solidity
struct SignedMintNlp {
    MintNlp tx;
    bytes signature;
    int128 oraclePriceX18;      // not part of the inner tx struct
    int128[] nlpPoolRebalanceX18;
}
``` [4](#0-3) 

`EndpointTx` passes this value directly to `clearinghouse.mintNlp` / `clearinghouse.burnNlp` and also writes it to `priceX18[NLP_PRODUCT_ID]` without any on-chain validation:

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(signedTx.tx, signedTx.oraclePriceX18, ...);
``` [5](#0-4) 

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.burnNlp(signedTx.tx, signedTx.oraclePriceX18, ...);
``` [6](#0-5) 

---

### Impact Explanation

**`burnNlp` with `oraclePriceX18 = 0`:**
- `quoteAmount = nlpAmount.mul(0) = 0`
- `burnFee = MathHelper.max(ONE, 0) = ONE` (minimum fee applied)
- `quoteAmount = MathHelper.max(0, 0 - ONE) = 0`
- The user's NLP balance is burned (`-nlpAmount`) and they receive **zero quote** in return — a total loss of the burned NLP value.
- Simultaneously, `spotEngine.updatePrice(NLP_PRODUCT_ID, 0)` corrupts the stored NLP price to zero, causing all subsequent health calculations for every NLP holder to treat their NLP as worthless. [7](#0-6) 

**`mintNlp` with `oraclePriceX18 = 0`:** triggers a division-by-zero revert in `quoteAmount.div(oraclePriceX18)`, which is a DoS on the mint path. [8](#0-7) 

The price corruption written to `spotEngine` and to `priceX18[NLP_PRODUCT_ID]` propagates to every downstream health check, liquidation threshold, and collateral valuation that reads the NLP price.

---

### Likelihood Explanation

The `oraclePriceX18` field is supplied at transaction-submission time and is not covered by any on-chain positive-value guard. The `updatePrice` path already demonstrates the protocol's intent to reject non-positive prices; the absence of the same guard in `mintNlp`/`burnNlp` is an oversight. Any path that results in a zero value reaching these functions — whether through a sequencer bug, a crafted submission, or an off-chain oracle failure — will corrupt protocol state with no on-chain safety net.

---

### Recommendation

Add the same guard used in `updatePrice` to both `mintNlp` and `burnNlp`:

```diff
function mintNlp(
    IEndpoint.MintNlp calldata txn,
    int128 oraclePriceX18,
    ...
) external onlyEndpoint {
+   require(oraclePriceX18 > 0, ERR_INVALID_PRICE);
    ...
}

function burnNlp(
    IEndpoint.BurnNlp calldata txn,
    int128 oraclePriceX18,
    ...
) external onlyEndpoint {
+   require(oraclePriceX18 > 0, ERR_INVALID_PRICE);
    ...
}
``` [9](#0-8) [10](#0-9) 

---

### Proof of Concept

1. A `BurnNlp` transaction is submitted with `oraclePriceX18 = 0`.
2. `EndpointTx` writes `priceX18[NLP_PRODUCT_ID] = 0` and calls `clearinghouse.burnNlp(..., 0, ...)`.
3. Inside `burnNlp`: `spotEngine.updatePrice(NLP_PRODUCT_ID, 0)` — NLP price in the spot engine is now zero.
4. `quoteAmount = nlpAmount.mul(0) = 0`; after fee logic, `quoteAmount` remains 0.
5. The sender's NLP balance is decremented by `nlpAmount`; no quote is credited.
6. The health check at line 526 passes because the NLP balance is still ≥ 0 (it was just reduced, not made negative).
7. All subsequent calls to `getHealth` for any NLP holder now value their NLP at price 0, potentially enabling illegitimate liquidations or blocking legitimate withdrawals. [11](#0-10)
