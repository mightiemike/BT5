### Title
Phantom-Index Merkle Proof Acceptance in Odd-Length Blocks — (`merkle-tools/src/lib.rs`, `contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion` (and `verify_transaction_inclusion_v2`) will return `true` when given the hash of the last real transaction in an odd-length block paired with a phantom out-of-bounds index equal to the block's transaction count. No bounds check on `tx_index` exists anywhere in the call path.

### Finding Description

Bitcoin's Merkle tree pads odd-length levels by duplicating the last leaf. In `merkle_proof_calculator`, when the list length is odd, the last element is appended to itself before pairing: [1](#0-0) 

For a 3-transaction block `[A, B, C]`, this produces `[A, B, C, C]` at level 0. The sibling selected for position 2 (even → `current_hashes[3]` = C) and for phantom position 3 (odd → `current_hashes[2]` = C) are **identical**: [2](#0-1) 

So `merkle_proof_calculator([A,B,C], 2) == merkle_proof_calculator([A,B,C], 3)` — both return `[C, hash(A,B)]`.

In `compute_root_from_merkle_proof`, the same cancellation occurs at level 0:
- Position 2 (even): `compute_hash(C, C)` = `hash(C,C)`
- Position 3 (odd): `compute_hash(C, C)` = `hash(C,C)`

Both produce the same intermediate hash, so both paths reconstruct the correct merkle root: [3](#0-2) 

`verify_transaction_inclusion` performs no bounds check on `tx_index` — it only checks that the proof is non-empty and that the computed root matches the stored header's merkle root: [4](#0-3) 

The block header stored in `headers_pool` contains only the merkle root, not the transaction count, so there is no way for the verifier to detect that index 3 is out of bounds for a 3-transaction block.

### Impact Explanation

Any caller — unprivileged, no special role required — can call `verify_transaction_inclusion` with:
- `tx_id = C` (the last real transaction hash, publicly known from the Bitcoin chain)
- `tx_index = N` (phantom, one past the end of an N-transaction odd-length block)
- `merkle_proof = proof_for_N-1` (the valid proof for the last real transaction)
- `tx_block_blockhash = B` (any canonical block with an odd transaction count)

The function returns `true`. The invariant "returns true only for a position that exists in the block" is broken.

Downstream consumers that use `(tx_hash, block_hash, tx_index)` as a unique key — e.g., a bridge contract preventing double-redemption — can be bypassed: the same transaction C can be "proven" at both index N-1 and phantom index N, yielding two distinct accepted proofs for the same on-chain event.

`verify_transaction_inclusion_v2` is equally affected: its coinbase pre-check uses a fixed index of 0 and does not constrain the target `tx_index`: [5](#0-4) 

### Likelihood Explanation

- Requires only a canonical block with an odd transaction count (extremely common on Bitcoin — the majority of blocks have an odd number of transactions).
- The attacker needs only the last transaction's hash and its standard Merkle proof, both publicly available from any Bitcoin node or block explorer.
- No privileged role, no key compromise, no social engineering.
- The call is a public view function with no access control.

### Recommendation

Add an explicit upper-bound check on `tx_index` before computing the proof. The block header alone does not carry the transaction count, so the fix requires one of:

1. Store the transaction count alongside the block header at submission time and reject `tx_index >= tx_count`.
2. Require the caller to supply the transaction count and a proof of it (e.g., via the coinbase path length), and validate `tx_index < tx_count`.
3. At minimum, document and enforce that `tx_index` must be strictly less than `2^(proof_length - 1)` for the last-leaf case, and reject proofs where `tx_index` is odd and the sibling in the proof equals `tx_id` (self-sibling at the last position).

### Proof of Concept

Concrete Rust unit test (no network, no NEAR runtime needed):

```rust
// In merkle-tools/src/lib.rs tests
#[test]
fn test_phantom_index_accepts_nonexistent_position() {
    // 3-transaction block: [A, B, C]
    let a = H256([0x01; 32]);
    let b = H256([0x02; 32]);
    let c = H256([0x03; 32]);
    let txs = vec![a.clone(), b.clone(), c.clone()];

    // Proof for the real last tx at index 2
    let proof_2 = merkle_proof_calculator(txs.clone(), 2);
    // Proof for phantom index 3 (block only has 3 txs, indices 0-2)
    let proof_3 = merkle_proof_calculator(txs.clone(), 3);

    // Both proofs are identical due to duplicate-leaf padding
    assert_eq!(proof_2, proof_3);

    // Compute the real merkle root
    let real_root = compute_root_from_merkle_proof(c.clone(), 2, &proof_2);

    // Phantom index 3 also reconstructs the same root with the same tx hash
    let phantom_root = compute_root_from_merkle_proof(c.clone(), 3, &proof_2);

    // This assertion passes — verify_transaction_inclusion returns true for index 3
    assert_eq!(real_root, phantom_root,
        "Phantom index 3 accepted: verify_transaction_inclusion returns true \
         for a position that does not exist in the block");
}
```

The test passes on the unmodified codebase, confirming that `compute_root_from_merkle_proof(C, 3, proof_for_2) == merkle_root` and therefore `verify_transaction_inclusion({tx_id: C, tx_index: 3, ...})` returns `true` for a 3-transaction block.
