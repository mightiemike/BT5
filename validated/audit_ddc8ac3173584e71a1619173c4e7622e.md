### Title
`allow_client_side_proving` Feature Flag Enforced on Gateway Path but Absent from Consensus Proposal Ingestion Path — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The gateway's `StatelessTransactionValidator` checks the operator-controlled `allow_client_side_proving` flag before accepting any Invoke V3 transaction that carries non-empty `proof_facts` or `proof`. The consensus proposal validation path (`convert_consensus_tx_to_internal_consensus_tx`) skips this check entirely. A malicious proposer can therefore inject client-side-proving Invoke V3 transactions into a block even when the operator has explicitly disabled the feature, bypassing the intended admission gate.

### Finding Description

**Gateway path (checked):**

`Gateway::add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)` before any conversion. [1](#0-0) 

Inside `StatelessTransactionValidator::validate`, for every `RpcTransaction::Invoke`, the method calls `validate_client_side_proving_allowed`: [2](#0-1) 

`validate_client_side_proving_allowed` reads `self.config.allow_client_side_proving` and rejects any Invoke V3 transaction whose `proof_facts` or `proof` fields are non-empty when the flag is `false`: [3](#0-2) 

**Consensus path (unchecked):**

When a validating node receives a `ProposalPart::Transactions` batch, `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx` for every transaction: [4](#0-3) 

`convert_consensus_tx_to_internal_consensus_tx` delegates to the private `convert_rpc_tx_to_internal`: [5](#0-4) 

`convert_rpc_tx_to_internal` extracts `proof_data` from the Invoke V3 branch and proceeds to hash and store the transaction with no call to `validate_client_side_proving_allowed` or any equivalent check: [6](#0-5) 

When `proof_data` is present, `spawn_verify_and_store_proof` is called, which verifies the proof cryptographically and stores it in the proof manager — all without consulting the `allow_client_side_proving` flag: [7](#0-6) 

The transaction then reaches the batcher and is executed by the blockifier. The blockifier's `validate_proof_facts` validates proof content (program hash, block hash, config hash) but has no knowledge of the gateway-level feature flag: [8](#0-7) 

### Impact Explanation

When `allow_client_side_proving = false`, the operator's intent is that no client-side-proving transactions enter the chain. A malicious proposer holding a valid proof can craft an Invoke V3 transaction with non-empty `proof_facts`/`proof`, include it in a consensus proposal, and have it accepted, verified, stored, and executed by every validating node — bypassing the feature gate entirely. This matches the "High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing" impact class: the consensus admission path accepts a transaction class that the operator's configuration explicitly forbids.

### Likelihood Explanation

Any consensus participant that becomes a proposer (a normal, non-privileged role in BFT consensus) can trigger this. The attacker must possess a cryptographically valid proof, which is a meaningful but not insurmountable barrier. The `allow_client_side_proving` flag is most likely `false` during the feature's rollout period, making the window of exposure real.

### Recommendation

Apply the same `allow_client_side_proving` check inside `convert_rpc_tx_to_internal` (or at the `convert_consensus_tx_to_internal_consensus_tx` boundary) before extracting or acting on `proof_data`. Concretely, if `proof_facts` is non-empty and the flag is `false`, the conversion should return an error, causing `handle_proposal_part` to mark the proposal as `Failed` and abort it.

### Proof of Concept

1. Configure the sequencer with `allow_client_side_proving = false`.
2. As a consensus proposer, craft an `RpcInvokeTransactionV3` with valid, non-empty `proof_facts` and `proof` fields.
3. Wrap it in a `ConsensusTransaction::RpcTransaction` and include it in a `ProposalPart::Transactions` batch.
4. Broadcast the proposal to validating peers.
5. Each peer's `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx`, which calls `convert_rpc_tx_to_internal`. No `allow_client_side_proving` check is performed.
6. `spawn_verify_and_store_proof` verifies and stores the proof; the transaction is forwarded to the batcher and executed by the blockifier — despite the feature being disabled.

    fn validate_client_side_proving_allowed(
        &self,
        tx: &RpcInvokeTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if self.config.allow_client_side_proving {
            return Ok(());
        }

        // Reject V3 transactions with proofs when client-side proving is disabled.
        let RpcInvokeTransaction::V3(tx) = tx;
        let has_proof_data = !tx.proof_facts.is_empty() || !tx.proof.is_empty();
        if has_proof_data {
            return Err(StatelessTransactionValidatorError::ClientSideProvingNotAllowed);
        }

        Ok(())
    }

            }
        }

    fn spawn_verify_and_store_proof(
        &self,
        proof_facts: ProofFacts,
        proof: Proof,
    ) -> VerifyAndStoreProofTask {
        let pmc = self.proof_manager_client.clone();
        let proof_facts_hash = proof_facts.hash();
        tokio::spawn(async move {
            let verified =
                Self::run_proof_verification(proof_facts.clone(), proof.clone(), pmc.clone())
                    .await?;

            if !verified {
                return Ok(());
            }

            let start = Instant::now();
            pmc.set_proof(proof_facts, proof).await?;
            let duration = start.elapsed();
            CONSENSUS_PROOF_MANAGER_STORE_LATENCY.record(duration.as_secs_f64());
            info!(
                "Proof manager store took: {duration:?} for proof facts hash: {proof_facts_hash:?}"
            );
            Ok(())
        })
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
