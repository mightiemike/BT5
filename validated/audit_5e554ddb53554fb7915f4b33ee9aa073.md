### Title
Unprotected `Verifier.initialize()` Allows Front-Running to Register Attacker-Controlled Signing Keys, Enabling Arbitrary `WithdrawPool` Drainage — (File: `core/contracts/Verifier.sol`)

---

### Summary

`Verifier.initialize()` has no caller access control. Any external account can front-run the legitimate deployment transaction and register arbitrary secp256k1 public keys as the protocol's canonical signing keys. Because `BaseWithdrawPool.submitFastWithdrawal()` is a public function that relies exclusively on `Verifier.requireValidTxSignatures()` for authorization, an attacker who controls the registered keys can craft and sign arbitrary fast-withdrawal transactions, directing the full token balance of the `WithdrawPool` to themselves.

---

### Finding Description

`Verifier.initialize()` is declared `external initializer` with no `msg.sender` check:

```solidity
function initialize(Point[8] memory initialSet) external initializer {
    __Ownable_init();
    for (uint256 i = 0; i < 8; ++i) {
        if (!isPointNone(initialSet[i])) {
            _assignPubkey(i, initialSet[i].x, initialSet[i].y);
        }
    }
}
``` [1](#0-0) 

The `initializer` modifier from OpenZeppelin prevents re-initialization but does not restrict *who* calls it first. The constructor correctly calls `_disableInitializers()` to protect the implementation contract, but the proxy's `initialize()` remains open to any caller. [2](#0-1) 

The keys registered during `initialize()` are stored in `pubkeys[i]` and are the sole trust root used by `requireValidTxSignatures()`:

```solidity
function requireValidTxSignatures(...) public view {
    ...
    require(nSignatures == nSigner, "not enough signatures");
}
``` [3](#0-2) 

`checkIndividualSignature` recovers the signer address from `getPubkeyAddress(signerIndex)`, which is derived directly from the stored `pubkeys` array:

```solidity
function getPubkeyAddress(uint8 index) public view returns (address) {
    Point memory p = getPubkey(index);
    return address(uint160(uint256(keccak256(abi.encode(p.x, p.y)))));
}
``` [4](#0-3) 

`BaseWithdrawPool.submitFastWithdrawal()` is a public function with no `onlyEndpoint` or `onlyOwner` guard. Its only authorization gate is `v.requireValidTxSignatures(transaction, idx, signatures)`:

```solidity
function submitFastWithdrawal(
    uint64 idx,
    bytes calldata transaction,
    bytes[] calldata signatures
) public {
    require(!markedIdxs[idx], "Withdrawal already submitted");
    require(idx > minIdx, "idx too small");
    markedIdxs[idx] = true;

    Verifier v = Verifier(verifier);
    v.requireValidTxSignatures(transaction, idx, signatures);
    ...
    handleWithdrawTransfer(token, sendTo, transferAmount);
}
``` [5](#0-4) 

`resolveFastWithdrawal` extracts `sendTo` and `transferAmount` directly from the attacker-supplied `transaction` bytes, with no independent on-chain validation of the withdrawal's legitimacy: [6](#0-5) 

---

### Impact Explanation

An attacker who front-runs `Verifier.initialize()` and registers their own secp256k1 key pair gains the ability to produce signatures that pass `requireValidTxSignatures()`. They can then call `submitFastWithdrawal()` with a crafted `WithdrawCollateralV2` transaction specifying `sendTo = attacker_address` and `amount = pool_balance`, draining all ERC-20 tokens held in the `WithdrawPool`. This is a direct, irreversible asset loss for all users who have pending or future withdrawals processed through the pool.

---

### Likelihood Explanation

The attack window is the block gap between proxy deployment and the legitimate `initialize()` call. This is a well-known, mechanically straightforward front-running attack requiring only mempool monitoring and a higher gas bid. No special privileges, leaked keys, or social engineering are needed. The attacker's key registration persists permanently since `initialize()` can only be called once.

---

### Recommendation

Add a deployer-only guard to `Verifier.initialize()`, consistent with the pattern used in `ContractOwner.initialize()`:

```solidity
function initialize(address _expectedDeployer, Point[8] memory initialSet)
    external
    initializer
{
    require(msg.sender == _expectedDeployer, "unauthorized initializer");
    __Ownable_init();
    ...
}
```

Alternatively, use a factory or `CREATE2`-based deployment that atomically deploys and initializes the proxy in a single transaction, eliminating the front-running window entirely.

---

### Proof of Concept

1. Deployer broadcasts `Verifier.initialize(legitimateKeys)` to the mempool.
2. Attacker generates a secp256k1 key pair `(privKey, pubPoint)`.
3. Attacker front-runs with `Verifier.initialize([pubPoint, zero, zero, ...])` at higher gas.
4. `Verifier` is now initialized with `pubkeys[0] = pubPoint`, `nSigner = 1`.
5. Protocol deployment continues; `WithdrawPool` is initialized pointing to this `Verifier`.
6. Users deposit collateral; tokens accumulate in `WithdrawPool`.
7. Attacker constructs a `WithdrawCollateralV2` transaction: `sender = attacker_bytes32`, `sendTo = attacker_address`, `productId = QUOTE_PRODUCT_ID`, `amount = pool_balance`.
8. Attacker computes `data = keccak256(abi.encodePacked(chainid, idx, transaction))` and signs `keccak256("\x19Ethereum Signed Message:\n32" || data)` with `privKey`.
9. Attacker calls `WithdrawPool.submitFastWithdrawal(idx=1, transaction, [signature])`.
10. `requireValidTxSignatures` passes: `nSignatures(1) == nSigner(1)`, ECDSA recovery matches `getPubkeyAddress(0)`.
11. `handleWithdrawTransfer` sends `pool_balance` tokens to `attacker_address`. [1](#0-0) [5](#0-4) [3](#0-2)

    function requireValidTxSignatures(
        bytes calldata txn,
        uint64 idx,
        bytes[] calldata signatures
    ) public view {
        require(signatures.length <= 256, "too many signatures");
        bytes32 data = keccak256(
            abi.encodePacked(uint256(block.chainid), uint256(idx), txn)
        );
        bytes32 hashedMsg = keccak256(
            abi.encodePacked("\x19Ethereum Signed Message:\n32", data)
        );

        uint256 nSignatures = 0;
        for (uint256 i = 0; i < signatures.length; i++) {
            if (signatures[i].length > 0) {
                nSignatures += 1;
                require(
                    checkIndividualSignature(
                        hashedMsg,
                        signatures[i],
                        uint8(i)
                    ),
                    "invalid signature"
                );
            }
        }
        require(nSignatures == nSigner, "not enough signatures");
    }
```

    function resolveFastWithdrawal(bytes calldata transaction)
        internal
        pure
        returns (
            uint32 productId,
            address sendTo,
            uint128 amount
        )
    {
        IEndpoint.TransactionType txType = IEndpoint.TransactionType(
            uint8(transaction[0])
        );
        if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
            IEndpoint.SignedWithdrawCollateral memory signedTx = abi.decode(
                transaction[1:],
                (IEndpoint.SignedWithdrawCollateral)
            );
            return (
                signedTx.tx.productId,
                address(uint160(bytes20(signedTx.tx.sender))),
                signedTx.tx.amount
            );
        }
        if (txType == IEndpoint.TransactionType.WithdrawCollateralV2) {
            IEndpoint.SignedWithdrawCollateralV2 memory signedTx = abi.decode(
                transaction[1:],
                (IEndpoint.SignedWithdrawCollateralV2)
            );
            // V2 appendix is intentionally ignored until fast-withdraw features use it.
            address resolvedSendTo = signedTx.tx.sendTo == address(0)
                ? address(uint160(bytes20(signedTx.tx.sender)))
                : signedTx.tx.sendTo;
            return (signedTx.tx.productId, resolvedSendTo, signedTx.tx.amount);
        }
        revert("Invalid withdrawal tx type");
    }

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
