### Title
Unbounded `NlpLockedBalanceQueue` Growth Per Subaccount Causes Permanent NLP Fund Lock via Gas Exhaustion in `tryUnlockNlpBalance` — (File: `core/contracts/SpotEngine.sol`)

---

### Summary

`handleNlpLockedBalance` appends a new entry to a per-subaccount `NlpLockedBalanceQueue` on every `MintNlp` call that occurs at a distinct oracle timestamp, with no cap on the queue length. `tryUnlockNlpBalance` later iterates the entire matured portion of that queue in an unbounded `while` loop. A user who mints NLP across enough distinct oracle timestamps will eventually grow their queue to a size where `tryUnlockNlpBalance` exceeds the block gas limit, permanently bricking their ability to call `burnNlp` and locking their NLP tokens in the protocol forever.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` is the internal function called on every positive NLP balance update (i.e., every `mintNlp`). It consolidates consecutive mints that share the same `unlockedAt` timestamp into a single queue entry, but creates a **new** entry for every mint that arrives at a different oracle time:

```solidity
// SpotEngine.sol lines 162-166
queue.balances[queue.balanceCount] = NlpLockedBalance({
    balance: Balance({amount: amountDelta}),
    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
});
queue.balanceCount++;
```

There is no upper bound on `queue.balanceCount`. The `NlpLockedBalanceQueue` struct itself uses a `mapping(uint64 => NlpLockedBalance)` for storage, so Solidity imposes no array-length revert — the counter simply grows without limit.

`tryUnlockNlpBalance` (called by both `handleNlpLockedBalance` and `getNlpUnlockedBalance`) drains matured entries via an unbounded `while` loop:

```solidity
// SpotEngineState.sol lines 292-303
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
```

Each iteration performs multiple cold/warm SLOADs and SSTOREs. Once the number of matured entries is large enough to exhaust the block gas limit inside this loop, every subsequent call that touches the subaccount's NLP balance reverts with out-of-gas.

`burnNlp` in `Clearinghouse.sol` is the critical path:

```solidity
// Clearinghouse.sol line 499
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
```

`getNlpUnlockedBalance` calls `tryUnlockNlpBalance` unconditionally. If that call OOGs, `burnNlp` always reverts, and the user's NLP balance is permanently unburnable.

---

### Impact Explanation

A user who has minted NLP across a sufficient number of distinct oracle timestamps will have their NLP tokens permanently locked. They cannot call `burnNlp` (the only redemption path), because every execution path through `burnNlp → getNlpUnlockedBalance → tryUnlockNlpBalance` will exceed the block gas limit. The locked NLP represents real quote-token value that the user deposited and can never recover. This is a permanent, irreversible asset loss, not merely a temporary denial of service.

**Corrupted state:** `nlpLockedBalanceQueues[subaccount].balanceCount` grows without bound; once the matured portion of the queue exceeds the per-block gas budget, the subaccount's NLP balance is effectively frozen at the protocol level.

---

### Likelihood Explanation

`MintNlp` is a standard user-facing operation submitted via the sequencer. A user who participates in NLP liquidity provision over an extended period — minting at different oracle timestamps across many blocks — will naturally accumulate queue entries. The oracle time used in `getOracleTime() + NLP_LOCK_PERIOD` changes with each block (or oracle update), so each mint at a new timestamp creates a new entry. No special privileges are required; any NLP depositor is exposed. The threshold for triggering the OOG condition depends on the block gas limit and per-iteration gas cost, but is reachable through normal long-term usage without any adversarial intent.

---

### Recommendation

Enforce a hard cap on `queue.balanceCount` per subaccount, analogous to how Geth enforces a maximum slot count per address in its mempool. A reasonable bound (e.g., 256 or 1024 entries) should be chosen based on the expected NLP lock period and oracle update frequency. When the cap is reached, either reject new mints or consolidate the oldest unlocked entry into `unlockedBalanceSum` before appending. Additionally, consider processing only a bounded number of entries per `tryUnlockNlpBalance` call (a "lazy drain" with a configurable batch size) to prevent any single call from consuming unbounded gas.

---

### Proof of Concept

1. User signs and submits `N` `MintNlp` transactions, each at a distinct oracle timestamp (e.g., one per block over many blocks).
2. The sequencer processes each transaction, calling `Clearinghouse.mintNlp` → `SpotEngine.updateBalance(NLP_PRODUCT_ID, sender, nlpAmount)` → `handleNlpLockedBalance`.
3. Because each mint arrives at a different `getOracleTime()`, the consolidation branch (`queue.balances[queue.balanceCount - 1].unlockedAt == getOracleTime() + NLP_LOCK_PERIOD`) is never taken, and `queue.balanceCount` increments to `N`.
4. After `NLP_LOCK_PERIOD` elapses, all `N` entries mature.
5. User submits a `BurnNlp` transaction. The sequencer calls `Clearinghouse.burnNlp` → `spotEngine.getNlpUnlockedBalance(sender)` → `tryUnlockNlpBalance`.
6. The `while` loop iterates `N` times. For sufficiently large `N`, the cumulative gas cost of `N` SLOADs + SSTOREs + `delete`s exceeds the block gas limit.
7. The transaction reverts with OOG. Every subsequent `BurnNlp` attempt also reverts. The user's NLP tokens are permanently locked.
