Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of End-User, Allowing Any User to Bypass the Swap Allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is the immediate `msg.sender` of the pool's `swap()` call. When users route through `MetricOmmSimpleRouter`, the router becomes the `msg.sender` of `pool.swap()`, so the extension checks the router's address against the allowlist rather than the actual end-user. Any pool that allowlists the router — required for router-mediated swaps to function — effectively opens the allowlist to every user on the network, allowing unauthorized addresses to extract value from LP reserves.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to every configured extension via `_callExtensionsInOrder`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making the router the `msg.sender` of that call: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

Because the router is a single shared contract, a pool admin who wants router-mediated swaps to work at all must add the router to `allowedSwapper[pool]`. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for every caller of the router, regardless of who that caller is. The per-user allowlist is completely bypassed for the router path. There is no existing guard that recovers the original end-user identity — `extensionData` is passed through unchanged from the router call and is not used by the extension to verify the caller.

## Impact Explanation
The `SwapAllowlistExtension` is the protocol's mechanism for restricting pool access to a defined set of counterparties (e.g., KYC'd institutions, whitelisted market makers). When the guard is bypassed, any unpermissioned address can execute swaps against the pool's LP reserves at the oracle-quoted price. Each such swap extracts value from LPs who believed the pool was access-controlled. This constitutes a direct loss of LP principal and a broken admin-boundary: the pool admin's allowlist configuration is rendered ineffective for all router-mediated flows. The bypass is permanent for the lifetime of the pool because the router address is immutable and the allowlist check is structural.

## Likelihood Explanation
The bypass is reachable by any user who calls the public `MetricOmmSimpleRouter` functions. No special privileges, flash loans, or oracle manipulation are required. The only precondition is that the pool admin has allowlisted the router — which is the expected operational state for any pool that intends to support standard swap UX. The trigger is a normal, valid swap call through the router, making this trivially exploitable by any network participant.

## Recommendation
The extension must check the economic actor, not the immediate caller. Two sound approaches:

1. **Pass the original user through the router**: Have the router encode the original `msg.sender` in `extensionData` and have the extension decode and check it. This requires a coordinated convention between the router and the extension.
2. **Check `recipient` instead of `sender`**: For the swap allowlist use-case, gating on the `recipient` (the address that receives output tokens) rather than `sender` (the address that initiated the call) may better reflect the intended access control, since the recipient is always the end-user even in router-mediated flows.

The simplest production fix is option 2 if the intent is to gate who receives tokens, or a signed/encoded user-identity pattern in `extensionData` if the intent is to gate who initiates the trade.

## Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` configured as a `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps (required for normal UX).
3. Pool admin does **not** add `alice` (an unauthorized user) to the allowlist.
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the restricted pool.
5. The router calls `pool.swap(...)` with `msg.sender = router`.
6. `_beforeSwap` passes `sender = router` to `SwapAllowlistExtension.beforeSwap`.
7. The check `allowedSwapper[pool][router]` returns `true` → the guard passes.
8. Alice's swap executes against LP reserves, extracting value from LPs who believed the pool was access-controlled.
