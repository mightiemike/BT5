### Title
Overly Restrictive Empty-Proof Guard in `verify_transaction_inclusion` Blocks Valid Single-Transaction Block Proofs — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` unconditionally panics when `merkle_proof` is empty. For a Bitcoin block containing exactly one transaction, an empty proof is mathematically correct — the Merkle root equals the transaction hash directly and no sibling hashes exist. The same guard is inherited by the current production entry point `verify_transaction_inclusion_v2`, which delegates to `verify_transaction_inclusion` after its own coinbase check passes. Any unprivileged NEAR caller attempting to verify a transaction in a single-transaction block will receive a panic instead of `true`.

---

### Finding Description

In `contract/src/lib.rs` at line 315, `verify_transaction_inclusion` contains:

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");
``` [1](#0-0) 

This guard fires unconditionally before the Merkle computation. However, `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` already handles the empty-proof case correctly: when the proof slice is empty, the `for` loop does not execute and the function returns the input `transaction_hash` unchanged. [2](#0-1) 

For a Bitcoin block containing exactly one transaction (e.g., a coinbase-only block), the Merkle tree has a single leaf. The Merkle root is defined as that leaf's hash, and no sibling hashes are needed — the proof is legitimately and correctly empty. Passing an empty `merkle_proof` with `tx_id == merkle_root` would produce a correct result from `compute_root_from_merkle_proof`, but the `require!` guard prevents the computation from ever running.

`verify_transaction_inclusion_v2` — the current recommended production API — delegates to `verify_transaction_inclusion` via:

```rust
#[allow(deprecated)]
self.verify_transaction_inclusion(args.into())
``` [3](#0-2) 

The `v2` coinbase check at lines 358–365 correctly passes for a single-transaction block (empty `coinbase_merkle_proof` returns `coinbase_tx_id`, which equals `merkle_root`), but the subsequent call into `verify_transaction_inclusion` then panics on the empty `merkle_proof`. [4](#0-3) 

The `v2` function also enforces `args.merkle_proof.len() == args.coinbase_merkle_proof.len()`, which is satisfied when both are empty (both length 0), so that guard does not protect against this path. [5](#0-4) 

---

### Impact Explanation

Any NEAR consumer contract or account that calls `verify_transaction_inclusion_v2` (or the deprecated `verify_transaction_inclusion`) to verify a transaction in a single-transaction block will receive a NEAR panic rather than a `true` return value. Consumer contracts built on top of this light client — atomic swaps, bridged asset release gates, cross-chain state verification — cannot process valid Bitcoin transactions from such blocks. If a release condition is gated on a successful `verify_transaction_inclusion_v2` call, the funds or state transition are permanently blocked for that transaction, with no recourse available to the user.

---

### Likelihood Explanation

Single-transaction Bitcoin blocks (coinbase only) exist on mainnet — all early Bitcoin blocks and occasional low-activity blocks fit this profile. The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion_v2` with a valid `ProofArgsV2` constructed from a real single-transaction block. The block header must already be submitted to the contract (via the relayer), which is the normal operating condition. No adversarial setup is required; the caller simply supplies correct, valid proof data and receives a panic.

---

### Recommendation

Remove the blanket `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. The downstream call to `compute_root_from_merkle_proof` already handles the empty case correctly and will return the `tx_id` unchanged, which is then compared against `header.block_header.merkle_root`. If the intent is to prevent a specific abuse class, scope the guard to cases where the proof length is inconsistent with the claimed `tx_index` (e.g., `tx_index > 0` with an empty proof), rather than rejecting all empty proofs unconditionally.

---

### Proof of Concept

1. A Bitcoin block **B** contains exactly one transaction **T** (coinbase). Its `merkle_root == T.tx_id`.
2. The relayer submits block **B** to the contract via `submit_blocks`. The header is stored in `headers_pool` and `mainchain_header_to_height`.
3. An unprivileged NEAR caller invokes `verify_transaction_inclusion_v2` with:
   - `tx_id = T.tx_id`
   - `tx_block_blockhash = B.hash`
   - `tx_index = 0`
   - `merkle_proof = []` ← valid for a single-tx block
   - `coinbase_tx_id = T.tx_id`
   - `coinbase_merkle_proof = []` ← valid for a single-tx block
   - `confirmations = 1`
4. The length-equality check passes: `0 == 0`. ✓
5. The coinbase proof check: `compute_root_from_merkle_proof(T.tx_id, 0, &[])` returns `T.tx_id`, which equals `merkle_root`. ✓
6. `verify_transaction_inclusion(args.into())` is called with `merkle_proof = []`.
7. `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` **panics** — a valid, correct proof is rejected. [6](#0-5) [7](#0-6)

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
