Audit Report

## Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` by Substituting Router Address for Actual Swapper Identity - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against its per-pool allowlist, but when a user routes through `MetricOmmSimpleRouter`, the pool receives the router contract as `msg.sender` and passes it as `sender` to the extension. The extension therefore checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][user]`, making the identity check structurally wrong for every router-mediated swap. This produces two high-severity failure modes: allowlist bypass for excluded users, and complete blockage of allowlisted users from the supported periphery path.

## Finding Description
**Root cause — pool unconditionally passes `msg.sender` as `sender`:**

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` at line 231. [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value verbatim to every configured extension. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool passed — the router address when called via router. [3](#0-2) 

**Router never forwards the original user address to the pool:**

`MetricOmmSimpleRouter.exactInputSingle` stores `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), then calls `pool.swap(params.recipient, ...)` with no user identity argument. The router is `msg.sender` at the pool. [4](#0-3) 

The same applies to `exactInput` (line 103), `exactOutputSingle` (line 135), and `exactOutput` (line 165). [5](#0-4) 

**Call path — direct swap (correct):**
```
user → MetricOmmPool.swap()
  _beforeSwap(msg.sender=user, ...)
    SwapAllowlistExtension.beforeSwap(sender=user)
      allowedSwapper[pool][user]  ✓
```

**Call path — router swap (broken):**
```
user → MetricOmmSimpleRouter.exactInputSingle()
  pool.swap(...)   // router is msg.sender to pool
    _beforeSwap(msg.sender=router, ...)
      SwapAllowlistExtension.beforeSwap(sender=router)
        allowedSwapper[pool][router]  ✗  (user identity lost)
```

No existing guard recovers the original user. The `allowAllSwappers` escape hatch only bypasses the check entirely; it does not restore per-user identity. [6](#0-5) 

## Impact Explanation
**Mode A — Allowlist bypass (High):** A pool admin who wants to support router-based swaps must add the router to the allowlist (`setAllowedToSwap(pool, router, true)`). Once done, every user — including those explicitly excluded — can bypass the allowlist by routing through `MetricOmmSimpleRouter`. The curated pool becomes effectively public to any caller. This is an admin-boundary break where an unprivileged path bypasses a configured access control.

**Mode B — Broken core functionality (High):** If the admin does not add the router, every allowlisted user is silently blocked from using the first-party periphery path. `SwapAllowlistExtension` is a first-party periphery contract; its structural incompatibility with the first-party router breaks the core swap flow for any pool that deploys it.

Both outcomes satisfy the contest's required impact gate: broken core pool functionality causing unusable swap flows, and admin-boundary break where an unprivileged path bypasses configured access control.

## Likelihood Explanation
The `SwapAllowlistExtension` is the primary access-control primitive for curated pools. Any pool that deploys it and expects to support router-based trading will encounter this issue immediately on the first router call. No special preconditions, oracle manipulation, or non-standard tokens are required. The attacker only needs to call `MetricOmmSimpleRouter.exactInputSingle` targeting a pool whose allowlist includes the router address. The condition is reachable by any unprivileged trader.

## Recommendation
The extension must receive the true originating user, not the immediate caller. Two options:

1. **Pass the original user through the pool's `swap` signature** — add a `swapper` parameter distinct from `msg.sender` so the router can supply `msg.sender` (the end user) explicitly. The pool still uses `msg.sender` for the callback, but passes the explicit `swapper` to extensions.

2. **Encode the original user in `extensionData`** — the router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This is only safe if the pool validates the caller is a known router, introducing a trust assumption.

Option 1 is cleaner and avoids trust assumptions.

## Proof of Concept
```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
swapExtension.setAllowedToSwap(pool, allowedUser, true);
// Admin also adds router so router-based swaps work at all:
swapExtension.setAllowedToSwap(pool, address(router), true);

// Attack: disallowedUser routes through the router.
vm.startPrank(disallowedUser);
token1.approve(address(router), type(uint256).max);
// This succeeds — the hook sees sender=router, which is allowlisted.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: disallowedUser,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    tokenIn: address(token1),
    deadline: block.timestamp,
    extensionData: ""
}));
// disallowedUser successfully swapped on a pool they were explicitly excluded from.
// beforeSwap resolves allowedSwapper[pool][router] == true → passes.
```

The `beforeSwap` check resolves to `allowedSwapper[pool][router] == true`, so it passes regardless of who the actual user is. [7](#0-6)
