### Title
Stale `digestToSubaccount` Entry After Isolated Subaccount Closure Allows Order Matching Against Closed, Empty Subaccount — (File: `core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears the parent/child linkage mappings but never clears `digestToSubaccount[orderDigest]`. After closure, `matchOrders` still reads the stale entry and redirects the order to the now-empty isolated subaccount. Because `isHealthy` in `OffchainExchange` unconditionally returns `true`, the trade settles against a subaccount that holds no margin, creating an undercollateralized position and bad debt.

---

### Finding Description

When a user creates an isolated order, `createIsolatedSubaccount` stores the mapping:

```
digestToSubaccount[orderDigest] = isolatedSubaccount
``` [1](#0-0) 

When the isolated subaccount is later closed via `_tryCloseIsolatedSubaccount`, the function transfers all balances back to the parent and clears three storage slots:

```solidity
isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
isolatedSubaccounts[parent][id] = bytes32(0);
parentSubaccounts[subaccount] = bytes32(0);
``` [2](#0-1) 

`digestToSubaccount[orderDigest]` is **never cleared**. It permanently retains the address of the now-closed isolated subaccount.

Subsequently, when `matchOrders` is called for the same order digest, it reads the stale entry and silently overwrites the order sender:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
``` [3](#0-2) 

The trade then executes against the closed isolated subaccount, which holds zero quote and zero perp balance. The post-trade health check cannot catch this because `isHealthy` in `OffchainExchange` is a stub that always returns `true`:

```solidity
function isHealthy(
    bytes32 /* subaccount */
) internal view virtual returns (bool) {
    return true;
}
``` [4](#0-3) 

The `require(isHealthy(...))` guards after balance updates therefore never revert: [5](#0-4) 

---

### Impact Explanation

The parent subaccount already received its full margin back when the isolated subaccount was closed (quote and vQuote balances transferred at lines 191–200). [6](#0-5) 

After the stale-digest match executes, the closed isolated subaccount holds a live perp position with a negative quote balance and zero collateral. This is an undercollateralized position that the protocol must absorb as bad debt. The parent has already extracted its margin, so the loss is borne by the insurance fund or other participants — directly analogous to the double-withdrawal drain in H-4.

---

### Likelihood Explanation

The sequencer processes `CloseIsolatedSubaccount` and `MatchOrders` as sequential transactions via `processTransactionImpl`. [7](#0-6) 

In a high-throughput environment the sequencer may have already queued a `MatchOrders` for a digest before it processes the corresponding `CloseIsolatedSubaccount`. Because the contract imposes no active-subaccount guard inside `matchOrders`, the stale redirect executes silently. A sophisticated user can also deliberately sequence these two transactions to guarantee the race condition.

---

### Recommendation

Inside `matchOrders`, before redirecting `order.sender` via `digestToSubaccount`, verify that the target isolated subaccount is still active:

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    bytes32 iso = digestToSubaccount[ordersInfo.taker.digest];
    require(isIsolatedSubaccountActive(taker.order.sender, iso), ERR_INVALID_TAKER);
    taker.order.sender = iso;
}
```

Alternatively, store the reverse mapping `isolatedSubaccountToDigest` and clear `digestToSubaccount` inside `_tryCloseIsolatedSubaccount`.

---

### Proof of Concept

1. User submits `CreateIsolatedSubaccount` with a long perp order for product P. The sequencer processes it: `digestToSubaccount[D] = isoAcct`, margin M is deposited to `isoAcct`.
2. User submits `CloseIsolatedSubaccount` for `isoAcct`. The sequencer processes it: margin M is returned to parent, `parentSubaccounts[isoAcct] = bytes32(0)`, but `digestToSubaccount[D]` remains `isoAcct`.
3. Parent subaccount now holds its full margin M back.
4. Sequencer processes a queued `MatchOrders` for digest D. `matchOrders` reads `digestToSubaccount[D] = isoAcct` and sets `taker.order.sender = isoAcct`.
5. The trade settles: `isoAcct` receives a long perp position and a negative quote balance. `isHealthy(isoAcct)` returns `true` unconditionally — no revert.
6. `isoAcct` now holds an undercollateralized position. Parent already holds M. Protocol absorbs the shortfall as bad debt. [8](#0-7) [9](#0-8) [10](#0-9)

    function _tryCloseIsolatedSubaccount(bytes32 subaccount) internal {
        uint32 productId = RiskHelper.getIsolatedProductId(subaccount);
        if (productId == 0) {
            return;
        }
        IPerpEngine.Balance memory balance = perpEngine.getBalance(
            productId,
            subaccount
        );
        if (balance.amount == 0) {
            uint8 id = RiskHelper.getIsolatedId(subaccount);
            address addr = address(uint160(bytes20(subaccount)));
            bytes32 parent = parentSubaccounts[subaccount];
            if (balance.vQuoteBalance != 0) {
                perpEngine.updateBalance(
                    productId,
                    subaccount,
                    0,
                    -balance.vQuoteBalance
                );
                perpEngine.updateBalance(
                    productId,
                    parent,
                    0,
                    balance.vQuoteBalance
                );
            }
            int128 quoteBalance = spotEngine
                .getBalance(QUOTE_PRODUCT_ID, subaccount)
                .amount;
            if (quoteBalance != 0) {
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    subaccount,
                    -quoteBalance
                );
                spotEngine.updateBalance(
                    QUOTE_PRODUCT_ID,
                    parent,
                    quoteBalance
                );
            }
            isolatedSubaccountsMask[addr] &= ~uint256(0) ^ (1 << id);
            isolatedSubaccounts[parent][id] = bytes32(0);
            parentSubaccounts[subaccount] = bytes32(0);

            emit CloseIsolatedSubaccount(subaccount, parent);
        }
    }
