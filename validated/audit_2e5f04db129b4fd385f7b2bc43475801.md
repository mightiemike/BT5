### Title
`DepositAllowlistExtension::beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address. However, its `beforeAddLiquidity` hook checks the `owner` parameter (position recipient) rather than the `sender` parameter (actual caller). Because `MetricOmmPool::addLiquidity` imposes no `msg.sender == owner` constraint, any non-allowlisted address can bypass the guard by nominating any allowlisted address as `owner`.

### Finding Description

`DepositAllowlistExtension::beforeAddLiquidity` receives two address arguments: the unnamed first parameter is `sender` (the actual `msg.sender` of the pool call) and the second is `owner` (the position recipient). The guard reads only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool::addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient, with no requirement that they match:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(address owner, ...) external nonReentrant(...) {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, ...
    );
    ...
}
``` [2](#0-1) 

The sibling `SwapAllowlistExtension` correctly checks `sender` (the actual caller), confirming the intended pattern:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-40
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    ...
}
``` [3](#0-2) 

**Exploit path:**
1. Pool admin deploys a pool with `DepositAllowlistExtension` and allowlists only address `A`.
2. Attacker (address `B`, not allowlisted) calls `pool.addLiquidity(A, salt, deltas, callbackData, extensionData)`.
3. The extension evaluates `allowedDepositor[pool][A]` → `true` → check passes.
4. `LiquidityLib.addLiquidity` mints shares credited to `A`; attacker pays tokens via the swap callback.
5. The allowlist is fully bypassed. `B` has added liquidity to a restricted pool.

Note: `removeLiquidity` does enforce `msg.sender == owner`, so the attacker cannot reclaim the deposited tokens — but the allowlist invariant is broken regardless. [4](#0-3) 

### Impact Explanation
The deposit allowlist is the pool admin's primary mechanism for restricting who may provide liquidity (e.g., for regulatory compliance, private pools, or curated LP sets). With this bug the guard is entirely ineffective: any unprivileged address can add liquidity to a restricted pool by naming any allowlisted address as `owner`. This is an admin-boundary break — the pool admin's configured access control is bypassed by a valid, unprivileged call path. Downstream consequences include unauthorized alteration of bin balances and pool state in pools that were intended to be access-controlled.

### Likelihood Explanation
Trivially reachable. No special privileges, tokens, or timing are required beyond knowing one allowlisted address (which is on-chain readable via `allowedDepositor`). Any external actor can execute this in a single transaction.

### Recommendation
Replace the `owner` check with `sender` (the first, currently unnamed parameter), mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

### Proof of Concept

```solidity
// Assume: pool configured with DepositAllowlistExtension
//         allowedDepositor[pool][alice] = true
//         allowedDepositor[pool][attacker] = false

vm.prank(attacker);
// Attacker specifies alice as owner — extension checks alice (allowed) and passes
pool.addLiquidity(
    alice,          // owner: allowlisted → guard passes
    0,              // salt
    deltas,         // bins/shares to add
    callbackData,   // attacker pays tokens here
    extensionData
);

// Result: liquidity added to restricted pool by non-allowlisted attacker.
// alice holds the position shares; attacker has bypassed the allowlist.
assertGt(pool.positionShares(alice, 0, binIdx), 0);
```
