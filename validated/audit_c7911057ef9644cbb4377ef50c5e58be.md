Audit Report

## Title
SwapAllowlistExtension Allowlist Bypassed via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates pool swaps by checking the `sender` argument, which is the direct `msg.sender` of `MetricOmmPool.swap`. When swaps are routed through `MetricOmmSimpleRouter`, the router becomes that `sender`. Any pool admin who allowlists the router address to enable router-mediated access for allowlisted users simultaneously grants unrestricted access to every user of the public router, because the extension cannot distinguish individual callers behind the router.

## Finding Description

**Pool side — sender binding**

`MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol lines 230-240
_beforeSwap(
    msg.sender,   // direct caller of pool.swap()
    recipient, zeroForOne, amountSpecified,
    priceLimitX64, packedSlot0Initial, bidPriceX64, askPriceX64, extensionData
);
```

`ExtensionCalling._beforeSwap` (lines 149–177) forwards this value unchanged via `abi.encodeCall` to every configured extension.

**Extension side — guard logic**

`SwapAllowlistExtension.beforeSwap` evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol lines 37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

**Router path — the bypass**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol lines 72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

The pool therefore receives `address(router)` as `msg.sender`, and passes it as `sender` to the extension. The extension evaluates `allowedSwapper[pool][router]`.

For allowlisted users to reach the pool via the router, the pool admin must call `setAllowedToSwap(pool, router, true)`. The moment that entry is set, every user — allowlisted or not — can bypass the gate by routing through the public `MetricOmmSimpleRouter`. The extension has no mechanism to recover the original EOA from behind the router. The same structural flaw applies to `exactInput`, `exactOutputSingle`, and `exactOutput` paths in the router.

**Existing guards are insufficient**: The `onlyPool` modifier on `BaseMetricExtension.beforeSwap` only verifies the caller is a registered pool — it does not verify the identity of the end user. There is no secondary check on the original `msg.sender` of the router call.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (KYC-gated, institutional-only, whitelist-only launch) loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. Unauthorized users can execute swaps in a pool intended to be access-controlled, draining liquidity reserved for allowlisted participants and interacting with pools whose oracle pricing or fee structure was calibrated for a specific counterparty set. This is a direct admin-boundary break: the pool admin's configured access boundary is bypassed by an unprivileged path (the public router), matching the allowed impact gate.

## Likelihood Explanation

Likelihood is medium. The trigger requires the pool admin to have allowlisted the router address — but this is the natural and necessary operational step for any allowlist-gated pool that also wants to support router-based swaps. Once the router is allowlisted (a single `setAllowedToSwap` call), the bypass is permanently open to all users and is trivially repeatable with no special attacker capability required.

## Recommendation

The extension must gate on the true end-user identity, not the intermediary. Two viable approaches:

1. **Pass-through identity in the router**: Have the router encode the original `msg.sender` into `extensionData` and have the extension decode and check that value. This requires a trusted router registry so the extension only accepts the decoded identity when the pool caller is a known router.
2. **Transient-storage forwarding**: The router writes the original `msg.sender` into a transient storage slot before calling `pool.swap`; the extension reads it from the router's storage via a trusted interface. This requires the extension to know and trust the router address.
3. **Do not allowlist the router**: Document that allowlisting the router address defeats the allowlist, and require pool admins to only allowlist individual EOAs or contracts that themselves enforce access control.

## Proof of Concept

```solidity
// Foundry test sketch
function test_routerBypassesAllowlist() public {
    // Setup: pool with SwapAllowlistExtension, only `allowedUser` is allowlisted
    vm.prank(poolAdmin);
    swapAllowlist.setAllowedToSwap(address(pool), address(router), true);
    // allowedUser is NOT separately allowlisted

    address attacker = makeAddr("attacker");
    deal(tokenIn, attacker, 1e18);

    vm.startPrank(attacker);
    IERC20(tokenIn).approve(address(router), 1e18);
    // Attacker routes through the public router — extension sees sender=router, passes
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        recipient: attacker,
        tokenIn: tokenIn,
        zeroForOne: true,
        amountIn: 1e18,
        amountOutMinimum: 0,
        priceLimitX64: 0,
        deadline: block.timestamp,
        extensionData: ""
    }));
    vm.stopPrank();
    // Attacker successfully swapped despite not being on the allowlist
}
```
