### Title
Dead Branch in `makerAccruesTakerFee` Due to Hardcoded `feeTiers[address(0)]` Makes Taker-Fee Accrual for Makers Permanently Unreachable - (File: `core/contracts/OffchainExchange.sol`)

### Summary
`makerAccruesTakerFee` in `OffchainExchange.sol` contains a condition that is always `false` in practice, making the taker-fee-accrual branch inside `matchOrders` permanently dead code. The root cause is that the third sub-condition hardcodes `feeTiers[address(0)]` instead of looking up the actual maker's fee tier, so the rate check `0 <= -3bps` is always `false`. This is a direct structural analog to the reported M-01 bug: a wrong comparison makes an intended execution branch unreachable.

### Finding Description

`makerAccruesTakerFee` is called inside `matchOrders` to decide whether a maker should accrue the taker fee (receive it as a rebate) rather than pay a standard maker fee:

```solidity
// apply the maker fee
if (makerAccruesTakerFee(maker.order.sender, callState.productId)) {
    ordersInfo.maker.fee = -ordersInfo.taker.fee;
    ordersInfo.maker.quoteDelta =
        ordersInfo.maker.quoteDelta +
        ordersInfo.taker.fee;
} else {
    applyFee(...);
}
```

The function itself is:

```solidity
function makerAccruesTakerFee(bytes32 subaccount, uint32 productId)
    internal
    view
    returns (bool)
{
    return
        subaccount != X_ACCOUNT &&
        address(uint160(bytes20(subaccount))) == address(0) &&
        getTierFeeRateX18(feeTiers[address(0)], productId).makerRateX18 <=
        TAKER_FEE_ACCRUAL_RATE_X18;
}
```

There are two compounding problems:

**Condition 2** — `address(uint160(bytes20(subaccount))) == address(0)`:
In Nado, a `bytes32` subaccount is encoded as `address (first 20 bytes) || name (last 12 bytes)`. For any real user, the first 20 bytes are their non-zero wallet address, so this condition is always `false` for regular traders. It is only `true` for special system accounts whose address portion is zero: `N_ACCOUNT` (`bytes32(2)`), `FEES_ACCOUNT` (`bytes32(0)`), and NLP pool accounts (starting at `NLP_POOL_ACCOUNT_START`).

**Condition 3** — `getTierFeeRateX18(feeTiers[address(0)], productId).makerRateX18 <= TAKER_FEE_ACCRUAL_RATE_X18`:
Even for the system accounts that pass condition 2, this condition uses `feeTiers[address(0)]` hardcoded. `feeTiers` is a mapping with a default value of `0`, so `feeTiers[address(0)] == 0` unless the protocol admin explicitly calls `updateFeeTier(address(0), ...)`. The default tier 0 returns `makerRateX18 = 0`. The threshold is `TAKER_FEE_ACCRUAL_RATE_X18 = -300_000_000_000_000` (−3 bps). Therefore `0 <= -3bps` is always `false`.

The combined result: `makerAccruesTakerFee` **always returns `false`**, making the `if` branch at line 780 permanently dead code. Every order match unconditionally falls into the `else` branch and calls `applyFee` for the maker.

The correct third condition should use the maker's actual address:
```solidity
getTierFeeRateX18(feeTiers[address(uint160(bytes20(subaccount)))], productId).makerRateX18 <=
    TAKER_FEE_ACCRUAL_RATE_X18
```
And condition 2 should be removed, since it unnecessarily restricts the check to system accounts.

### Impact Explanation

Any maker whose fee tier has `makerRateX18 <= -3bps` (i.e., a sufficiently large rebate) should have their `quoteDelta` increased by the taker fee (`ordersInfo.maker.quoteDelta + ordersInfo.taker.fee`) and their `fee` set to `-ordersInfo.taker.fee`. Instead, they always go through `applyFee(..., taker=false)`, which applies the standard maker rebate calculation. The taker-fee-accrual incentive mechanism — a distinct and intentional fee accounting path — is completely broken. Makers who qualify for this treatment receive incorrect (lower) quote deltas on every fill.

### Likelihood Explanation

The bug is triggered on every single call to `matchOrders` where the maker qualifies for taker-fee accrual. The sequencer submits `MatchOrders` and `MatchOrdersWithAmount` transactions continuously as part of normal protocol operation. No special attacker action is needed — the broken branch is simply never taken, silently miscalculating maker balances on every qualifying fill.

### Recommendation

Remove condition 2 and fix condition 3 to use the maker's actual address:

```diff
 function makerAccruesTakerFee(bytes32 subaccount, uint32 productId)
     internal
     view
     returns (bool)
 {
+    address makerAddr = address(uint160(bytes20(subaccount)));
     return
         subaccount != X_ACCOUNT &&
-        address(uint160(bytes20(subaccount))) == address(0) &&
-        getTierFeeRateX18(feeTiers[address(0)], productId).makerRateX18 <=
+        getTierFeeRateX18(feeTiers[makerAddr], productId).makerRateX18 <=
         TAKER_FEE_ACCRUAL_RATE_X18;
 }
```

### Proof of Concept

1. A maker subaccount `0xAlice000...000` (non-zero address) is assigned fee tier 5 via `updateFeeTier`, where tier 5 has `makerRateX18 = -500_000_000_000_000` (−5 bps, below the −3 bps threshold).
2. The sequencer submits a `MatchOrders` transaction pairing Alice as maker.
3. `matchOrders` calls `makerAccruesTakerFee(aliceSubaccount, productId)`.
4. Condition 2: `address(uint160(bytes20(aliceSubaccount))) == address(0)` → `aliceAddress == address(0)` → **false**.
5. Function returns `false` immediately; the `if` branch is skipped.
6. `applyFee(..., taker=false)` is called instead, applying the standard maker rebate.
7. Alice's `quoteDelta` is not augmented by the taker fee; the intended taker-fee-accrual accounting never occurs.
