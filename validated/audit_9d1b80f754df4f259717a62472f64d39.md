### Title
Linked Signer Can Overwrite Its Own Authorization — Unauthorized Subaccount Mutation via `LinkSigner` with `allowLinkedSigner = true` - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction is validated with `allowLinkedSigner = true`. This permits the **current linked signer** — a delegated hot-wallet with limited trust — to sign and submit a `LinkSigner` transaction that replaces itself (or any address) as the subaccount's authorized signer. The subaccount owner's private key is never required. A compromised or malicious linked signer can silently redirect all future signing authority to an attacker-controlled address, after which the attacker can drain collateral, transfer quote, or perform liquidations before the owner can recover via the 3-day slow mode path.

---

### Finding Description

`EndpointTx.processTransactionImpl` handles the sequencer-submitted `LinkSigner` transaction type at lines 576–590:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
```

`validateSignedTx` (lines 86–106) calls `validateSignature` (lines 172–184), which in turn calls `verifier.validateSignature` (lines 291–304 of `Verifier.sol`):

```solidity
function validateSignature(
    bytes32 sender,
    address linkedSigner,
    bytes32 digest,
    bytes memory signature
) public pure {
    address recovered = ECDSA.recover(digest, signature);
    require(
        (recovered != address(0)) &&
            ((recovered == address(uint160(bytes20(sender)))) ||
                (recovered == linkedSigner)),   // ← linked signer accepted
        ERR_INVALID_SIGNATURE
    );
}
```

Because `allowLinkedSigner = true` is passed for `LinkSigner`, the verifier accepts a signature from **either** the subaccount owner address **or** the current linked signer. The mutation target is `linkedSigners[signedTx.tx.sender]` — the very authorization credential that grants the linked signer its power. The linked signer can therefore sign a `LinkSigner` transaction pointing `signer` to any attacker-controlled address, and the contract will accept it and overwrite the mapping.

The `LinkSigner` struct carries a `nonce` field, so the nonce is consumed, but the nonce belongs to the **subaccount owner's address** (`nonces[address(uint160(bytes20(sender)))]`). The linked signer does not have its own nonce; it simply satisfies the signature check. After the nonce is consumed, the linked signer can craft a fresh `LinkSigner` with the next nonce value (which it can read from `getNonce()`) and repeat.

The slow-mode `LinkSigner` path (lines 232–239 of `EndpointTx.sol`) does not require a signature at all — it only checks `msg.sender == address(uint160(bytes20(txn.sender)))`. This means the owner can eventually recover, but only after the 3-day `SLOW_MODE_TX_DELAY` (defined in `Constants.sol` line 50). During that window the attacker holds full signing authority.

---

### Impact Explanation

Once the attacker controls the linked signer slot, they can sign any transaction that accepts `allowLinkedSigner = true`:

- `WithdrawCollateral` / `WithdrawCollateralV2` — drain all collateral to an arbitrary `sendTo` address.
- `TransferQuote` — move quote balance to an attacker subaccount.
- `LiquidateSubaccount` — force-liquidate the victim's positions at a penalty.
- `MintNlp` / `BurnNlp` — manipulate NLP positions.

All of these accept `allowLinkedSigner = true` in their respective `validateSignedTx` calls. The corrupted state is `linkedSigners[victim_subaccount]` → `attacker_address`, and the resulting asset delta is full collateral loss for the victim subaccount.

**Impact: High** — complete collateral theft is achievable in a single sequencer batch after the linked signer is overwritten.

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated trading bots and API keys in the Nado protocol. Any user who has ever called `LinkSigner` (either via sequencer or slow mode) is exposed. The attack requires only that the linked signer key be compromised (e.g., leaked API key, compromised trading bot, insider at a market-making firm). No admin access, no governance capture, and no sequencer compromise is needed — the attacker submits a normal sequencer transaction signed by the linked signer key.

**Likelihood: Medium** — linked signer keys are high-value targets with a realistic compromise surface (hot wallets, API servers, trading bots).

---

### Recommendation

Pass `allowLinkedSigner = false` when validating `LinkSigner` transactions in `processTransactionImpl`. Changing the linked signer is a privileged operation that must require the subaccount owner's own private key, not a delegated key. The linked signer should be able to trade on behalf of the subaccount but must not be able to mutate its own authorization or replace itself with another address.

```solidity
// EndpointTx.sol — processTransactionImpl, LinkSigner branch
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← require subaccount owner signature, not linked signer
);
```

### Proof of Concept

1. Alice owns subaccount `alice_addr || "default\x00\x00\x00\x00"` and has linked Bob's address as her linked signer via a prior `LinkSigner` transaction.
2. Bob (compromised) constructs a `SignedLinkSigner` with:
   - `tx.sender = alice_subaccount`
   - `tx.signer = attacker_address` (padded to `bytes32`)
   - `tx.nonce = nonces[alice_addr]` (read from `Endpoint.getNonce(alice_addr)`)
   - `signature` = Bob's ECDSA signature over the EIP-712 digest of the above
3. Bob submits this to the sequencer (standard API call, no special access).
4. The sequencer includes it in a `submitTransactionsChecked` batch.
5. `processTransactionImpl` decodes the `SignedLinkSigner`, calls `validateSignedTx(..., true)`, which calls `verifier.validateSignature(alice_subaccount, bob_address, digest, sig)`. Bob's signature recovers to `bob_address == linkedSigner`, so the check passes.
6. `linkedSigners[alice_subaccount]` is overwritten to `attacker_address`.
7. Attacker immediately signs a `WithdrawCollateralV2` with `sendTo = attacker_eoa`, draining Alice's collateral before she can submit a slow-mode recovery transaction. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
