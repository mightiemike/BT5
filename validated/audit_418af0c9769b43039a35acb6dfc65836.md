### Title
Merkle Proof Forgery via Internal Node Accepted as Transaction ID — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` does not validate that the caller-supplied `tx_id` is a leaf-level hash (an actual transaction) rather than an intermediate Merkle tree node hash. An unprivileged NEAR caller can supply an internal node hash as `tx_id` with a crafted proof that correctly computes to the stored `merkle_root`, causing the function to return `true` for a transaction that does not exist in the block.

---

### Finding Description

`verify_transaction_inclusion` in `contract/src/lib.rs` accepts `tx_id`, `tx_index`, and `merkle_proof` from any caller and delegates root computation to `merkle_tools::compute_root_from_merkle_proof`. The only completeness checks performed are: [1](#0-0) 

```rust
require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

There is no check that `tx_id` is a leaf-level hash. `compute_root_from_merkle_proof` in `merkle-tools/src/lib.rs` simply iterates the proof elements and hashes left/right based on position parity — it is agnostic to tree depth or node level: [2](#0-1) 

This is the direct analog of the Chainlink finding: `latestRoundData()` returns `(roundId, answer, startedAt, updatedAt, answeredInRound)` but only `answer > 0` is checked, leaving `updatedAt` and `answeredInRound` unvalidated. Here, the proof inputs `(tx_id, tx_index, merkle_proof)` are accepted but only the final computed root is compared — the completeness condition (is `tx_id` a leaf?) is never checked.

**Concrete attack for a block with 4 transactions T1, T2, T3, T4:**

| Symbol | Value |
|--------|-------|
| H12 | `double_sha256(T1 \|\| T2)` — internal node |
| H34 | `double_sha256(T3 \|\| T4)` — internal node |
| R | `double_sha256(H12 \|\| H34)` — merkle root |

Attacker calls `verify_transaction_inclusion` with `tx_id = H12`, `tx_index = 0`, `merkle_proof = [H34]`.

`compute_root_from_merkle_proof(H12, 0, [H34])`:
- position 0 is even → `hash(H12, H34)` = R
- Returns R

`R == header.block_header.merkle_root` → **`true`**

No transaction with ID H12 exists. The function has been deceived by an incomplete data check, exactly as `getExchangeRate` is deceived by an incomplete round check in the Chainlink report.

---

### Impact Explanation

Any NEAR contract that gates state transitions on `verify_transaction_inclusion` returning `true` — e.g., releasing bridged funds, recording cross-chain events, or authorizing withdrawals — can be deceived into accepting a forged proof. The attacker does not need any privileged role. The corrupted value is the **proof result**: a `true` return for a non-existent transaction. [3](#0-2) 

---

### Likelihood Explanation

The function is publicly callable by any NEAR account with no role restriction beyond the `#[pause]` gate (which is inactive by default). The attack requires only the block's transaction hashes — publicly available from any Bitcoin RPC node — to compute internal node hashes. No cryptographic preimage search is needed. The deprecated function remains live and callable despite `verify_transaction_inclusion_v2` existing as a fix.

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
