Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address instead of the originating user, allowing any user to bypass the per-user swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument it receives from the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool's `swap()` is called by the router, so `msg.sender` inside the pool is the router address, which is then forwarded as `sender` to the extension. If the pool admin allowlists the router to permit router-mediated swaps, every unpermissioned user can bypass the per-user allowlist by routing through `MetricOmmSimpleRouter`, completely defeating the allowlist's purpose.

## Finding Description
**Step 1 — Pool passes `msg.sender` (the router) as `sender` to the extension.**

In `MetricOmmPool.swap()`, the pool calls `_beforeSwap` with `msg.sender` as the first argument: [1](#0-0) 

When the call originates from `MetricOmmSimpleRouter`, `msg.sender` here is the router contract address.

**Step 2 — The extension checks `allowedSwapper[pool][sender]` where `sender` is the router.** [2](#0-1) 

`msg.sender` in the extension is the pool, and `sender` is the router. The check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`.

**Step 3 — The router calls `pool.swap()` directly, never forwarding the originating user.** [3](#0-2) 

The actual user's address (`msg.sender` of `exactInputSingle`) is stored in transient storage only for the payment callback context, but is never passed to `pool.swap()` or to the extension. The extension has no way to identify the real user.

**Existing guards are insufficient:** The only guard in `beforeSwap` is `allowedSwapper[msg.sender][sender]`. There is no secondary check on `extensionData` or any other field that could carry the originating user's identity.

## Impact Explanation
The `SwapAllowlistExtension` is the primary mechanism for curated pools to restrict who may trade. A complete bypass of this guard means unpermissioned users can execute swaps on pools intended to be restricted (e.g., institutional pools, KYC-gated pools). LP providers deposited under the assumption that only vetted counterparties would trade against their liquidity; allowing arbitrary swappers violates that assumption and can expose LPs to adverse selection or regulatory risk. This is broken core pool functionality: the allowlist guard silently fails open for the primary public swap entrypoint (`exactInputSingle`), matching the "Broken core pool functionality" allowed impact.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, documented user-facing swap entrypoint. Any pool admin who wants users to swap via the router must allowlist the router address, which immediately opens the bypass to all users. No special privilege, flash loan, or multi-step setup is required — a single `exactInputSingle` call suffices. The bypass is reachable by any unpermissioned user with no preconditions beyond the router being allowlisted, which is the natural and expected configuration for any pool that supports router-based UX.

## Recommendation
The extension must check the originating user, not the immediate pool caller. Two complementary fixes:

1. **Router forwards the originating user**: `MetricOmmSimpleRouter` should encode `msg.sender` (the actual user) into `extensionData` before forwarding it to `pool.swap()`.

2. **Extension reads the originating user from `extensionData`**: `SwapAllowlistExtension.beforeSwap` should decode the real user from `extensionData` when `sender` is a known intermediary, and gate on that address instead:

```solidity
// In SwapAllowlistExtension.beforeSwap:
address effectiveSender = extensionData.length >= 32
    ? abi.decode(extensionData, (address))
    : sender;
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][effectiveSender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

This requires the router to prepend `abi.encode(msg.sender)` to `extensionData` before forwarding it to the pool.

## Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; alice is allowlisted, bob is not.
// Pool admin allowlists the router so that alice can use it.
swapAllowlist.setAllowedToSwap(address(pool), address(router), true); // router allowlisted
swapAllowlist.setAllowedToSwap(address(pool), alice, true);
// bob is NOT allowlisted

// Bob bypasses the allowlist by routing through MetricOmmSimpleRouter:
vm.prank(bob);
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: bob,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp,
    extensionData: ""
}));
// Succeeds — extension saw sender=router (allowlisted), not bob (not allowlisted)
```

The extension receives `sender = address(router)`, checks `allowedSwapper[pool][router] == true`, and passes. Bob's swap executes despite being explicitly excluded from the allowlist.
