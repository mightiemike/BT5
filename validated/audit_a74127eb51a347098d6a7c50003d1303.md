### Title
`DirectDepositV1` Deployed with `ContractOwner` as Owner, Blocking Recovery of Native Tokens — (`core/contracts/DirectDepositV1.sol`, `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.createDirectDepositV1()` deploys `DirectDepositV1` via `new DirectDepositV1{salt: ...}(...)`. Because `DirectDepositV1` inherits OpenZeppelin's non-upgradeable `Ownable`, `msg.sender` at construction time — which is `ContractOwner`, the intermediary — becomes the permanent owner. The multisig (the actual human authority, owner of `ContractOwner`) is never the owner of any `DirectDepositV1` instance. `ContractOwner` exposes only one path to call `DirectDepositV1.withdraw()` (through `wrapVaultAsset()`), and has **no function at all** to call `DirectDepositV1.withdrawNative()`. Native tokens that become stranded in a `DirectDepositV1` instance are permanently unrecoverable.

---

### Finding Description

`ContractOwner.createDirectDepositV1()` is the factory entry point:

```solidity
// ContractOwner.sol lines 495–499
DirectDepositV1 directDepositV1 = new DirectDepositV1{
    salt: bytes32(uint256(1))
}(address(endpoint), address(spotEngine), subaccount, wrappedNative);
directDepositV1Address[subaccount] = payable(directDepositV1);
``` [1](#0-0) 

`DirectDepositV1` inherits `Ownable` and its constructor never calls `transferOwnership`. OpenZeppelin's `Ownable` constructor sets `msg.sender` — here `ContractOwner` — as the permanent owner:

```solidity
// DirectDepositV1.sol line 29
contract DirectDepositV1 is Ownable {
``` [2](#0-1) 

The two `onlyOwner` functions in `DirectDepositV1` are:

```solidity
// DirectDepositV1.sol lines 103–112
function withdraw(IIERC20Base token) external onlyOwner {
    uint256 balance = token.balanceOf(address(this));
    safeTransfer(token, msg.sender, balance);
}

function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
``` [3](#0-2) 

`ContractOwner` exposes `withdraw()` only through `wrapVaultAsset()`, which calls it exclusively to pull a vault's underlying asset token for re-wrapping:

```solidity
// ContractOwner.sol lines 526–528
DirectDepositV1(directDepositV1).withdraw(
    IIERC20Base(assetTokenAddr)
);
``` [4](#0-3) 

There is **no function in `ContractOwner` that calls `withdrawNative()`**. The multisig (owner of `ContractOwner`) is not the owner of `DirectDepositV1` and cannot call either `onlyOwner` function directly.

The constructor of `DirectDepositV1` attempts to forward any pre-existing native balance to `wrappedNative`, but silently emits an event on failure rather than reverting:

```solidity
// DirectDepositV1.sol lines 52–60
uint256 balance = address(this).balance;
if (balance != 0) {
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);
    }
}
``` [5](#0-4) 

If this silent failure occurs, the native tokens are permanently locked: the multisig cannot call `withdrawNative()` (not the owner), and `ContractOwner` has no passthrough for it.

---

### Impact Explanation

Native tokens stranded in a `DirectDepositV1` instance — for example, pre-sent to the deterministic CREATE2 address before deployment, or left over from a failed constructor wrap — are permanently unrecoverable. Neither the multisig nor `ContractOwner` can invoke `withdrawNative()`. The exact corrupted asset delta is the full native token balance of the affected `DirectDepositV1` contract.

---

### Likelihood Explanation

`createDirectDepositV1` is `public` with no access control, callable by any address. [6](#0-5) 
The CREATE2 address for each subaccount is deterministic and predictable. An attacker or user can pre-fund the address with native tokens before deployment. The constructor's silent-failure path (`NativeTokenTransferFailed`) then leaves those tokens stranded with no recovery path. Likelihood is low-to-medium: it requires deliberate pre-funding or a `wrappedNative` call failure, but the entry path is fully permissionless.

---

### Recommendation

In `createDirectDepositV1`, after deploying the instance, call `transferOwnership` to hand ownership to the multisig (i.e., `owner()` of `ContractOwner`):

```solidity
DirectDepositV1 directDepositV1 = new DirectDepositV1{salt: bytes32(uint256(1))}(
    address(endpoint), address(spotEngine), subaccount, wrappedNative
);
directDepositV1.transferOwnership(owner()); // transfer to multisig
```

Alternatively, add a `withdrawNative(bytes32 subaccount)` passthrough in `ContractOwner` that forwards the call to the correct `DirectDepositV1` instance, restricted to `onlyOwner`.

---

### Proof of Concept

1. Compute the CREATE2 address for a target `subaccount` using deployer = `ContractOwner`, salt = `bytes32(uint256(1))`, and the known `DirectDepositV1` init code hash.
2. Send native tokens to that address before `createDirectDepositV1(subaccount)` is called.
3. Call `ContractOwner.createDirectDepositV1(subaccount)` (permissionless, `public`).
4. The constructor attempts `wrappedNative.call{value: balance}("")`. If `wrappedNative` is paused or rejects the call, the transfer fails silently and native tokens remain in `DirectDepositV1`.
5. The multisig attempts `DirectDepositV1(addr).withdrawNative()` → reverts: `"Ownable: caller is not the owner"`.
6. `ContractOwner` has no function to call `withdrawNative()` → tokens are permanently locked.
