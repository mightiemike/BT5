### Title
Checkpoint-before-pool-check ordering in `processEventsAndClaimBonus` causes permanent bonus reward loss when `ErrInsufficientBonusPool` is swallowed by `slashRedelegationPosition` — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary
`processEventsAndClaimBonus` mutates `pos.LastBonusAccrual`, `pos.LastEventSeq`, and `pos.LastKnownBonded` in-memory before it checks whether the bonus pool has sufficient balance. When the pool is insufficient, it returns `ErrInsufficientBonusPool`. `slashRedelegationPosition` explicitly swallows this error and then calls `setPositionWithState`, which persists the already-advanced checkpoints. The position owner permanently loses all bonus rewards that accrued up to the slash event.

### Finding Description

The ordering bug is in `processEventsAndClaimBonus`: [1](#0-0) 

`pos.UpdateLastEventSeq` is called inside the event loop (line 193), and `applyBonusAccrualCheckpoint` / `pos.UpdateLastKnownBonded` are called unconditionally at lines 215–217 — all before the pool sufficiency check at line 230: [2](#0-1) 

When `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool`, the function returns with the in-memory `pos` already carrying advanced checkpoints and no coins sent.

Back in `slashRedelegationPosition`, the error is explicitly swallowed: [3](#0-2) 

Execution then falls through to `setPositionWithState`, which persists the mutated position: [4](#0-3) 

`setPositionWithState` writes the position to the store unconditionally: [5](#0-4) 

### Impact Explanation

The invariant that `LastBonusAccrual` must only advance when the corresponding bonus coins are transferred to the owner is violated. After the slash, the position's reward checkpoint is set to the slash block time. All bonus that accrued between the previous checkpoint and the slash block is permanently unclaimable — the next `processEventsAndClaimBonus` call will start from the advanced checkpoint and compute zero for the already-elapsed period.

For a partial slash, the position remains live and the owner continues to accrue future rewards, but the historical bonus is gone. For a full slash, the position is zeroed out and `ClearBonusCheckpoints` is called, but the bonus is equally unrecoverable. [6](#0-5) 

### Likelihood Explanation

The preconditions are:
1. A position with an active redelegation (normal user operation via `MsgRedelegate`)
2. A validator slash that triggers `BeforeRedelegationSlashed` (normal chain event when a validator double-signs or is jailed)
3. The `RewardsPoolName` module account balance is below the owed bonus at the time of the slash

Condition 3 is the binding constraint. The bonus pool can be depleted by normal reward claims from other positions, or by an adversary who holds large positions and claims rewards to drain the pool before a known slash event. The comment at line 55 ("Deliberately forgo bonus rewards if pool is insufficient to prevent chain halt") confirms this path is intentionally reachable — the design decision to avoid a chain halt is correct, but the implementation incorrectly advances checkpoints before confirming payment. [7](#0-6) 

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful `SendCoinsFromModuleToAccount` call, or save and restore the pre-call checkpoint values when `ErrInsufficientBonusPool` is returned. Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly call `pos.ClearBonusCheckpoints()` (or reset `LastBonusAccrual` / `LastEventSeq` to their pre-call values) before invoking `setPositionWithState`, so that the next legitimate claim can retry the full accrual period.

### Proof of Concept

1. Create a tier position and redelegate to a second validator (this creates a `RedelegationMappings` entry with a non-zero `unbondingID`).
2. Advance block time by 30 days so bonus accrues on the destination validator.
3. Drain the `RewardsPoolName` module account to zero (e.g., by claiming rewards from other large positions, or by direct test manipulation).
4. Trigger `BeforeRedelegationSlashed` with a partial `sharesToUnbond` (simulating a validator slash).
5. Assert:
   - The owner's balance did **not** increase (no bonus paid).
   - The persisted `LastBonusAccrual` equals the slash block time (checkpoint advanced).
   - A second call to `processEventsAndClaimBonus` with a funded pool returns zero bonus (the accrual period is gone).

The existing test `TestSlashRedelegationPosition_ClaimsBonusRewardsUpToSlash` in `slash_test.go` funds the pool before the slash; removing that funding step and re-running the assertion at line 89–91 would demonstrate the failure. [8](#0-7)
