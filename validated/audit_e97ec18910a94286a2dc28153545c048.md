Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks Position Owner Instead of Actual Payer, Allowing Unauthorized Deposits via Operator Pattern — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and gates on `owner` (the position holder) instead. Because `MetricOmmPoolLiquidityAdder` exposes a public `addLiquidityExactShares(pool, owner, ...)` overload that accepts an arbitrary `owner` while always using `msg.sender` as the payer, any unauthorized user can bypass the deposit allowlist by nominating any allowlisted address as `owner`. The pool admin's curation policy is fully circumvented without any special privileges.

## Finding Description

`DepositAllowlistExtension.beforeAddLiquidity` ignores its first parameter (`sender`) and checks only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` (the actual caller/payer) as the first argument to `_beforeAddLiquidity`, and `owner` (the position holder) as the second:

```solidity
// metric-core/contracts/MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [2](#0-1) 

`MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` accepts an arbitrary `owner` (validated only to be non-zero) while always recording `msg.sender` as the payer in transient storage:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L56-68
function addLiquidityExactShares(address pool, address owner, ...) external payable override {
    _validateOwner(owner);  // only checks owner != address(0)
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, ...);
}
``` [3](#0-2) 

The callback then pulls tokens from the stored payer (Bob), not from `owner` (Alice):

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L172-177
if (amount0Delta > 0) {
    pay(token0, payer, msg.sender, amount0Delta);
}
if (amount1Delta > 0) {
    pay(token1, payer, msg.sender, amount1Delta);
}
``` [4](#0-3) 

The contract's own naming confirms the intended gate is the depositing economic actor: the mapping is `allowedDepositor`, the setter is `setAllowedToDeposit`, and the NatSpec reads "Gates `addLiquidity` by depositor address, per pool." [5](#0-4) 

The implementation checks the wrong actor: `owner` (position holder) instead of `sender` (payer/depositor).

## Impact Explanation

This is an admin-boundary break. A pool admin configuring `DepositAllowlistExtension` to enforce KYC/AML compliance or curated LP access has that policy fully bypassed by any unprivileged user. The unauthorized user (Bob) deposits real tokens into the pool; the allowlist extension does not block him. The exact wrong value is the extension's boolean gate decision: `allowedDepositor[pool][alice]` is checked when `allowedDepositor[pool][bob]` should be checked, causing the extension to return `beforeAddLiquidity.selector` (allow) instead of reverting with `NotAllowedToDeposit`.

## Likelihood Explanation

The `addLiquidityExactShares(pool, owner, ...)` overload is a documented, public, production entrypoint requiring no special privileges. Any user who knows one allowlisted address can execute the bypass in a single transaction with no malicious setup, non-standard tokens, or off-chain coordination. The attack is repeatable indefinitely.

## Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is instead to gate by position owner, rename the mapping and setter to `allowedOwner`/`setAllowedOwner` and update NatSpec accordingly so pool admins are not misled about what the allowlist enforces.

## Proof of Concept

```
Setup:
  - Pool configured with DepositAllowlistExtension
  - Alice (0xAlice) is allowlisted: allowedDepositor[pool][alice] = true
  - Bob (0xBob) is NOT allowlisted

Attack:
  1. Bob approves LiquidityAdder for token0/token1
  2. Bob calls:
       LiquidityAdder.addLiquidityExactShares(
           pool,
           alice,   // owner = allowlisted address
           salt,
           deltas,
           maxAmount0,
           maxAmount1,
           ""
       )
  3. LiquidityAdder calls pool.addLiquidity(alice, ...)
     with msg.sender = LiquidityAdder
  4. Pool calls _beforeAddLiquidity(LiquidityAdder, alice, ...)
  5. Extension checks allowedDepositor[pool][alice] → true → no revert
  6. Callback pulls tokens from Bob (payer in transient context)
  7. Alice receives position shares; Bob's funds are in the pool

Result:
  - Bob (unauthorized) has deposited into a curated pool
  - DepositAllowlistExtension did not block him
  - Pool admin's curation policy is violated
```
