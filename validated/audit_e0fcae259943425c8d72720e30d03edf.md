### Title
Migration upgrade halt via `ErrInsufficientBonusPool` in `ForceFullExitWithDelegation` — (`x/tieredrewards/keeper/force_exit.go`)

### Summary
The v8 upgrade migration calls `ForceFullExitWithDelegation` for every vesting-account-owned tier position. This function calls `claimRewards` → `processEventsAndClaimBonus` → `sufficientBonusPoolBalance`. If the rewards pool balance is insufficient to pay the accrued bonus for any vesting-owned position, `ErrInsufficientBonusPool` propagates unguarded through the entire call stack, causing `RunMigrations` to return an error and the upgrade handler to abort. The chain halts at the upgrade height in a partially-migrated state.

### Finding Description
`ForceFullExitWithDelegation` in `x/tieredrewards/keeper/force_exit.go` calls `claimRewards` at line 37 and returns any error unconditionally at line 39. There is no special handling for `ErrInsufficientBonusPool`. In contrast, `slashRedelegationPosition` in `x/tieredrewards/keeper/slash.go` explicitly catches `ErrInsufficientBonusPool` at lines 56–63 and logs it instead of returning it, with the comment "Deliberately forgo bonus rewards if pool is insufficient to prevent chain halt." The migration path lacks this same protection.

The error propagation chain is:
`sufficientBonusPoolBalance` → `processEventsAndClaimBonus` → `claimRewards` → `ForceFullExitWithDelegation` → `exitVestedAccountsPositions` → `Migrate` → `Migrate1to2` → `RunMigrations` → upgrade handler returns error.

### Impact Explanation
The upgrade halts at the upgrade height. The chain cannot produce new blocks until the upgrade handler succeeds. Some vesting-owned positions may have already been force-exited (partial migration), while others remain. The chain is stuck in an inconsistent intermediate state. Recovery requires a coordinated emergency patch and re-upgrade, which is operationally severe.

### Likelihood Explanation
The rewards pool is a shared module account consumed by both BeginBlocker base-reward top-ups and user bonus claims. On a live chain with active positions and a non-trivially funded pool, the pool balance at any given block is not guaranteed to cover the sum of all accrued bonus for all vesting-owned positions. The longer the chain runs before the upgrade, the larger the accrued bonus. This is a passive condition requiring no attacker — it can arise from normal chain operation.

### Recommendation
Apply the same "chain-halt avoidance" pattern used in `slashRedelegationPosition` to `ForceFullExitWithDelegation`. Catch `ErrInsufficientBonusPool` from `claimRewards`, log it, and continue the exit without paying the bonus. The position's delegation must still be returned to the owner; only the bonus payment is skipped. This matches the documented invariant that the migration must complete atomically regardless of pool balance.

```go
posState, baseRewards, bonusRewards, err := k.claimRewards(ctx, posState)
if err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        logger.Error("force-exit: insufficient bonus pool, forgoing bonus rewards",
            "position_id", posID, "error", err)
        // Re-fetch posState without bonus claim so the exit can proceed.
        posState, err = k.getPositionState(ctx, posID)
        if err != nil {
            return fmt.Errorf("re-fetch position %d after bonus skip: %w", posID, err)
        }
        baseRewards, bonusRewards = sdk.NewCoins(), sdk.NewCoins()
    } else {
        return fmt.Errorf("claim rewards for position %d: %w", posID, err)
    }
}
```

### Proof of Concept
1. Create a vesting account owner with a tier position (commit or lock origin).
2. Advance block time so bonus accrues.
3. Drain the rewards pool to zero (or below the accrued bonus amount).
4. Call `migrator.Migrate1to2(ctx)`.
5. Assert: the call returns an error wrapping `ErrInsufficientBonusPool`, and the position is not deleted (partial migration state).

func (k Keeper) sufficientBonusPoolBalance(ctx context.Context, bonus sdk.Coins) error {
	if bonus.IsZero() {
		return nil
	}

	poolAddr := k.accountKeeper.GetModuleAddress(types.RewardsPoolName)

			"bonus: %s, pool balance: %s",
