### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus-to-Internal L1Handler Conversion Drops Actual L1 Fee — (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

When a validator node converts a `ConsensusTransaction::L1Handler` back into an `InternalConsensusTransaction::L1Handler` (executable form), the `paid_fee_on_l1` field is unconditionally hardcoded to `Fee(1)` instead of being forwarded from the original L1 event. This is the direct sequencer analog of the LayerZero refund-address bug: a value that must be threaded through a conversion boundary is instead replaced by a fixed intermediary-chosen constant, producing a wrong executable object and divergent fee accounting between the proposer and validator.

---

### Finding Description

**Conversion boundary where the value is lost**

`ConsensusTransaction::L1Handler` wraps `transaction::L1HandlerTransaction`, which carries no `paid_fee_on_l1` field: [1](#0-0) 

When the proposer converts its `InternalConsensusTransaction::L1Handler` (which holds the real fee from the L1 scraper) into a `ConsensusTransaction` for broadcast, the fee is silently dropped: [2](#0-1) 

**Hardcoded replacement on the validator side**

When the validator receives the `ConsensusTransaction::L1Handler` and converts it back to an executable form, `convert_consensus_l1_handler_to_internal_l1_handler` is called: [3](#0-2) 

`Fee(1)` is unconditionally substituted for the actual fee. The `TODO` comment acknowledges this is a placeholder, not the correct value.

**What the correct value should be**

The proposer's path preserves the real fee: the L1 events provider returns `executable_transaction::L1HandlerTransaction` objects with the actual `paid_fee_on_l1` scraped from the L1 event, and `ProposeTransactionProvider` wraps them directly: [4](#0-3) 

The `executable_transaction::L1HandlerTransaction::create` constructor that the validator calls computes the tx hash correctly from `chain_id`, but accepts whatever `paid_fee_on_l1` is passed in: [5](#0-4) 

**How `paid_fee_on_l1` is used in execution**

The blockifier's `execute_raw` for L1 handler transactions checks the field directly: [6](#0-5) 

The check is `paid_fee == Fee(0)` → error. Because the validator always injects `Fee(1)`, this guard is permanently bypassed on the validator side regardless of what was actually paid on L1.

---

### Impact Explanation

**Divergent execution between proposer and validator**

If an L1 message arrives with `paid_fee_on_l1 = Fee(0)` (possible if the L1 contract or a test/staging environment permits it, or if the scraper receives a zero-fee event):

- **Proposer**: executes with real `Fee(0)` → `InsufficientFee` error → transaction is reverted or excluded.
- **Validator**: executes with hardcoded `Fee(1)` → check passes → transaction succeeds and state is committed.

This is a proposer/validator execution divergence: the validator commits state that the proposer did not, producing a wrong state root and breaking consensus.

**Wrong fee value in execution context and central blob**

Even when `paid_fee_on_l1 > 0`, the validator's `TransactionExecutionInfo` and any downstream serialization (e.g., `CentralL1HandlerTransaction`) carry `Fee(1)` instead of the real fee: [7](#0-6) 

The central blob written by the proposer uses the real fee, but the validator's internal execution object carries the wrong value — a broken canonicalization invariant across the conversion boundary.

---

### Likelihood Explanation

The trigger requires an L1 handler transaction with `paid_fee_on_l1 = Fee(0)` to reach the sequencer. In production the Starknet core contract enforces a non-zero ETH value, making this unlikely on mainnet. However:

- Testnets, staging environments, or future contract upgrades may relax this constraint.
- The hardcoded `Fee(1)` is an acknowledged placeholder (`TODO(Gilad)`), meaning the invariant is known to be broken and is expected to be tightened — at which point any non-trivial fee check will immediately expose the divergence for all L1 handler transactions, not just zero-fee ones.

---

### Recommendation

`ConsensusTransaction::L1Handler` must carry `paid_fee_on_l1` so the value survives the round-trip through the consensus wire format. Options:

1. Add `paid_fee_on_l1: Fee` to `ConsensusTransaction::L1Handler` (and the protobuf `L1HandlerV0` message), so `convert_consensus_l1_handler_to_internal_l1_handler` can forward the real value.
2. Alternatively, pass the `paid_fee_on_l1` out-of-band alongside the consensus transaction (similar to how `InternalRpcDeployAccountTransaction` carries `contract_address`).

Remove the `Fee(1)` placeholder and the `TODO` once the real value is threaded through.

---

### Proof of Concept

1. Scrape an L1 `LogMessageToL2` event with `fee = 0` (or inject one in a test environment).
2. The L1 events provider stores `executable_transaction::L1HandlerTransaction { paid_fee_on_l1: Fee(0), … }`.
3. The proposer's `ProposeTransactionProvider` wraps it as `InternalConsensusTransaction::L1Handler` with `Fee(0)` and attempts execution → blockifier returns `InsufficientFee` → transaction is not committed.
4. The proposer broadcasts `ConsensusTransaction::L1Handler(transaction::L1HandlerTransaction { … })` (no fee field).
5. The validator calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))`.
6. The validator's blockifier executes with `paid_fee_on_l1 = Fee(1)` → check passes → state is committed.
7. Proposer and validator have different state roots → consensus failure / wrong committed state.
