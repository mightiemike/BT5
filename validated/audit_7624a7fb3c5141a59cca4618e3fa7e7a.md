### Title
Deactivated Perp Product Can Be Liquidated Due to Missing Active-Product Guard in `_assertLiquidationAmount` — (File: `core/contracts/ClearinghouseLiq.sol`)

---

### Summary

In `ClearinghouseLiq.sol`, the `_assertLiquidationAmount` function validates that a spot or spread product is "active" (`longWeightInitialX18 != 0`) before allowing liquidation. However, the perp-only branch of the same function contains no equivalent guard. A liquidator can therefore force balance mutations on a deactivated perp product, violating the protocol invariant that deactivated products must not be modified.

---

### Finding Description

In Nado, a product is considered deactivated when its `longWeightInitialX18` is set to `0`. This sentinel value is used consistently throughout the codebase to skip or reject operations on inactive products. For example:

- `_assertCanLiquidateLiability` skips spot products with `longWeightInitialX18 == 0` [1](#0-0) 
- `_finalizeSubaccount` skips spot products with `longWeightInitialX18 == 0` [2](#0-1) 

Inside `_assertLiquidationAmount`, the **spread** and **spot** branches both enforce this guard before proceeding:

```solidity
// spread branch (lines 154-157)
require(
    spotEngine.getRisk(spotId).longWeightInitialX18 != 0,
    ERR_INVALID_PRODUCT
);

// spot branch (lines 160-163)
require(
    spotEngine.getRisk(spotId).longWeightInitialX18 != 0,
    ERR_INVALID_PRODUCT
);
``` [3](#0-2) 

However, the **perp-only** `else` branch has no such check:

```solidity
} else {
    require(
        txn.amount % perpSizeIncrement == 0,
        ERR_INVALID_LIQUIDATION_AMOUNT
    );
    maxLiquidatable = perpAmount;  // no active-product guard
}
``` [4](#0-3) 

After `_assertLiquidationAmount` passes, `_handleLiquidationPayment` unconditionally calls `perpEngine.updateBalance` on `txn.productId`, mutating balances on the deactivated product. [5](#0-4) 

---

### Impact Explanation

If a perp product is deactivated after positions have been opened (its `longWeightInitialX18` is zeroed out), existing balances in that product persist on-chain. Because a deactivated perp contributes zero weight to health calculations, a subaccount holding only a deactivated perp position will appear healthy. However, if that subaccount is unhealthy for any other reason, a liquidator can target the deactivated perp product as `txn.productId`. The liquidation will pass `_assertLiquidationAmount` without rejection and proceed to `_handleLiquidationPayment`, which will:

1. Transfer the deactivated perp position from the liquidatee to the liquidator.
2. Adjust the quote balance of both parties based on a liquidation price derived from the deactivated product's stale oracle price.

This corrupts the accounting state of a product that is supposed to be frozen, and can result in the liquidator acquiring a position in a deactivated market at a potentially favorable price, or the liquidatee losing collateral to a liquidation that should not have been permitted.

**Corrupted state delta:** `perpEngine` balance for `txn.productId` (deactivated) is modified for both `txn.liquidatee` and `txn.sender`; `spotEngine` quote balance is modified for both parties. [6](#0-5) 

---

### Likelihood Explanation

- Requires a perp product to have been deactivated while open positions exist — a realistic operational scenario (e.g., a market is wound down).
- Requires the liquidatee to be under maintenance health for any reason — a normal liquidation precondition.
- The liquidator is an unprivileged caller with no special access requirements.
- The entry path is the standard `liquidateSubaccountImpl` function, reachable via `Endpoint.submitTransaction`. [7](#0-6) 

---

### Recommendation

Add the same active-product guard to the perp branch of `_assertLiquidationAmount` that already exists for the spot and spread branches:

```solidity
} else {
    require(
        txn.amount % perpSizeIncrement == 0,
        ERR_INVALID_LIQUIDATION_AMOUNT
    );
    require(
        perpEngine.getRisk(txn.productId).longWeightInitialX18 != 0,
        ERR_INVALID_PRODUCT
    );
    maxLiquidatable = perpAmount;
}
```

This mirrors the existing pattern used for spot and spread liquidations. [8](#0-7) 

---

### Proof of Concept

1. Admin sets `longWeightInitialX18 = 0` on perp product `P` (deactivating it). Existing positions in `P` remain on-chain.
2. Subaccount `V` holds a position in `P` and is also unhealthy due to another position (e.g., an underwater spot liability).
3. Liquidator `L` submits a `LiquidateSubaccount` transaction with `productId = P`, `isEncodedSpread = false`, targeting `V`.
4. `liquidateSubaccountImpl` passes: `isUnderMaintenance(V)` is true. [9](#0-8) 
5. `_assertLiquidationAmount` is reached. `productToEngine[P]` resolves to `perpEngine`, so the `else` branch executes. No `longWeightInitialX18 != 0` check is performed; `maxLiquidatable = perpAmount` passes. [4](#0-3) 
6. `_handleLiquidationPayment` calls `perpEngine.updateBalance(P, V, -amount, ...)` and `perpEngine.updateBalance(P, L, amount, ...)`, mutating balances on the deactivated product. [5](#0-4) 
7. `L` acquires a position in the deactivated market; `V` loses it along with quote collateral — all on a product that should have been immutable.
