### Title
Dogecoin Fork Difficulty Retarget Uses Mainchain Ancestor Timestamp Instead of Fork Ancestor — (`contract/src/dogecoin.rs`)

---

### Summary

`get_next_work_required` in the Dogecoin module calls `blocks_getter.get_header_by_height(height_first)` to obtain the retarget window's opening block time. The `BlocksGetter` implementation resolves this exclusively from `mainchain_height_to_header`, so for any fork block at depth ≥ 3 from the fork point (under the per-block Digishield protocol active since height 145,000), the function reads the **mainchain** block's timestamp instead of the **fork ancestor's** timestamp. The code itself flags this with a `TODO` comment at the exact line.

---

### Finding Description

**Root cause — `get_header_by_height` is mainchain-only:** [1](#0-0) 

The function unconditionally indexes `mainchain_height_to_header`. There is no fork-aware path.

**Where the wrong value is consumed:** [2](#0-1) 

With `new_difficulty_protocol` active (`prev_block_header.block_height >= 145_000`), `difficulty_adjustment_interval = 1`, so `blocks_to_go_back = 1` for every block after genesis. For a fork block at height `H+k`, `height_first = H+k-2`. [3](#0-2) 

**When the desynchronization first occurs:**

Let the fork diverge at height `H` (i.e., `F[H+1]` is the first fork-only block, child of the shared `M[H]`):

| Fork block | `height_first` | `get_header_by_height` returns | Correct fork ancestor |
|---|---|---|---|
| `F[H+1]` | `H-1` | `M[H-1]` | `M[H-1]` ✓ |
| `F[H+2]` | `H` | `M[H]` | `M[H]` ✓ |
| `F[H+3]` | `H+1` | `M[H+1]` | `F[H+1]` **✗** |
| `F[H+4]` | `H+2` | `M[H+2]` | `F[H+2]` **✗** |

Starting from `F[H+3]`, every retarget uses `M[H+1].time` instead of `F[H+1].time`. Because the attacker controls `F[H+1].time` (subject only to MTP and the 2-hour future-time cap), they can engineer a systematic divergence.

**How `calculate_next_work_required` is affected:** [4](#0-3) 

`modulated_timespan = prev_block_header.time − first_block_time`. If the attacker sets `F[H+1].time` significantly earlier than `M[H+1].time`, then `F[H+2].time − M[H+1].time` is larger than `F[H+2].time − F[H+1].time`, pushing `modulated_timespan` toward `max_timespan` and producing a **lower difficulty** (easier target) than the fork's actual history warrants. Conversely, setting `F[H+1].time` later than `M[H+1].time` produces a harder target, causing valid fork blocks to be rejected.

---

### Impact Explanation

The contract's difficulty enforcement for Dogecoin fork blocks (height ≥ 145,001, depth ≥ 3 from fork point) is computed against the wrong ancestor. An attacker can:

1. **Accept under-difficulty fork blocks**: craft `F[H+1].time` so that the mainchain timestamp produces an easier `expected_bits` than the fork's real history requires. The contract accepts blocks mined at lower difficulty, meaning each fork block represents less actual work than the `bits` field implies.
2. **Reject valid fork blocks**: craft `F[H+1].time` so the contract computes a harder target than the fork's real history requires, causing legitimate relayer submissions to be rejected (denial of fork tracking).

For impact (1): because `chain_work` is accumulated from `bits`, accepting easier bits means less work per block. However, the attacker can compensate by submitting more blocks, and the critical invariant broken is that the contract can no longer reliably enforce that fork blocks represent the claimed proof-of-work, undermining the SPV security guarantee for any consumer of the light client that relies on fork/reorg correctness.

---

### Likelihood Explanation

- The bug is reachable through the public `submit_blocks` call with no privileged role required.
- It activates on mainnet Dogecoin (height >> 145,000) for any fork submission of depth ≥ 3.
- The `TODO` comment at line 291 confirms the developers identified this uncertainty but left it unresolved.
- No existing guard compensates: `check_pow` calls `check_target` which calls `get_next_work_required` with the same broken `get_header_by_height`. [5](#0-4) 

---

### Recommendation

Replace the `get_header_by_height` call with a chain-traversal that walks `get_prev_header` from `prev_block_header` backward by `blocks_to_go_back` steps. This is already how the `get_prev_header` path works for the min-difficulty walk (lines 263–270) and is the correct approach for fork-aware ancestor lookup. The `get_header_by_height` shortcut is only safe when the block being validated is on the mainchain.

---

### Proof of Concept

```
State: Dogecoin mainnet, height ~145,002+
Fork diverges at height H = 145,001 (new_difficulty_protocol active)

Mainchain: M[145,000] → M[145,001] → M[145,002] → ...
Fork:      M[145,000] → F[145,001] → F[145,002] → F[145,003]

Set F[145,001].time = M[145,001].time − 600   (60 seconds earlier, within MTP)

When validating F[145,003]:
  prev = F[145,002], height_first = 145,001
  get_header_by_height(145,001) → M[145,001]   ← mainchain block
  first_block_time = M[145,001].time

  Contract computes:
    modulated_timespan = F[145,002].time − M[145,001].time
                       = F[145,002].time − (F[145,001].time + 600)
                       = (correct_timespan) − 600

  Correct computation would use:
    modulated_timespan = F[145,002].time − F[145,001].time
                       = correct_timespan

  The contract accepts F[145,003] with bits matching the incorrect (easier) target.
  assert(expected_bits_from_mainchain_ancestor ≠ expected_bits_from_fork_ancestor)
  → invariant violated; difficulty validation is incorrect for fork depth ≥ 3.
```
