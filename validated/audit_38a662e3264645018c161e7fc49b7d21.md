### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas` Variant, Causing Consensus Transaction Rejection and Hash Domain Collision - (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` applies a content-based heuristic to select the enum variant: if both `l2_gas` and `l1_data_gas` are zero it emits `ValidResourceBounds::L1Gas`, otherwise `ValidResourceBounds::AllResources`. A transaction submitted via RPC with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway, serialized as `AllResources`, but deserialized on the receiving consensus node as `L1Gas`. This causes two distinct failures: (1) the receiving node's `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` conversion hard-rejects the message with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, dropping the transaction before sequencing; and (2) if the transaction were to survive, `get_tip_resource_bounds_hash` would produce a different Poseidon hash for `L1Gas` (2-element preimage) versus `AllResources` (3-element preimage), binding the wrong hash to the transaction.

### Finding Description

**Step 1 – Gateway accepts the transaction.**

`RpcInvokeTransactionV3.resource_bounds` is typed as `AllResourceBounds` (not `ValidResourceBounds`), so any invoke submitted via RPC is always `AllResources`. A transaction with `AllResourceBounds { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` passes the stateless validator (the test suite explicitly marks `valid_l1_gas` with only `l1_gas` non-zero as valid). [1](#0-0) 

**Step 2 – Hash is computed over the `AllResources` preimage.**

`get_tip_resource_bounds_hash` branches on the variant. For `AllResources` it appends the `L1_DATA_GAS` packed felt (zero value) to the hash chain; for `L1Gas` it does not. The two paths produce different Poseidon digests even when `l1_data_gas = 0`. [2](#0-1) 

**Step 3 – Protobuf serialization preserves all three fields.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` always emits `l1_data_gas: Some(zero)` for the `AllResources` arm. [3](#0-2) 

**Step 4 – Protobuf deserialization silently changes the variant.**

On the receiving consensus node, `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the heuristic: `if l1_data_gas.is_zero() && l2_gas.is_zero() → L1Gas`. A wire message with `l2_gas = 0, l1_data_gas = 0` (both legitimately zero) is therefore decoded as `ValidResourceBounds::L1Gas`, not `AllResources`. [4](#0-3) 

**Step 5 – Conversion to `RpcInvokeTransactionV3` hard-fails.**

`TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` requires `AllResources`; if the variant is `L1Gas` it returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The comment in the caller explicitly acknowledges this failure mode. [5](#0-4) 

This error propagates through `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction`, causing the receiving node to reject the entire consensus message. [6](#0-5) 

**Step 6 – Hash domain collision (secondary).**

Even if the conversion were to succeed via an alternate path, the hash computed from `L1Gas` (2-element resource preimage) differs from the hash computed from `AllResources` (3-element resource preimage). The Starknet OS Cairo code enforces exactly 3 resource bounds (`assert n_resource_bounds = 3`), so the `L1Gas` hash would also be rejected at proof verification. [7](#0-6) 

### Impact Explanation

A user submitting a valid V3 invoke transaction with only `l1_gas` bounds set (l2_gas = 0, l1_data_gas = 0) — a configuration explicitly accepted by the gateway's stateless validator — will have that transaction silently dropped at the consensus layer. The proposer node includes the transaction in a block proposal; every validator node fails to deserialize the protobuf message and rejects it. This constitutes a gateway/RPC admission accepting a valid transaction that is then rejected before sequencing, matching the **High** impact scope.

### Likelihood Explanation

The trigger is any V3 invoke transaction with `AllResourceBounds` where both `l2_gas` and `l1_data_gas` are zero. This is a normal, user-reachable configuration (e.g., a transaction that only pays in L1 gas). No special privileges are required. The gateway explicitly accepts such transactions. The bug is deterministic: every such transaction will be rejected by every receiving consensus node.

### Recommendation

Replace the content-based variant heuristic in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with an explicit discriminator field in the protobuf schema (e.g., a `bounds_type` enum field), or always deserialize to `AllResources` when all three fields are present in the wire message (regardless of whether their values are zero). The TODO comment `// TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2` already signals the intent to remove the legacy fallback; the fix should be applied now. [8](#0-7) 

### Proof of Concept

1. Submit a V3 invoke via RPC with `resource_bounds = { L1_GAS: { max_amount: 1000, max_price_per_unit: 1 }, L2_GAS: { max_amount: 0, max_price_per_unit: 0 }, L1_DATA_GAS: { max_amount: 0, max_price_per_unit: 0 } }`.
2. Gateway accepts it; `convert_rpc_tx_to_internal` wraps it as `ValidResourceBounds::AllResources` and computes hash H₁ (3-element Poseidon preimage including `L1_DATA_GAS = 0`).
3. Transaction is propagated to a peer via `From<ConsensusTransaction> for protobuf::ConsensusTransaction` → `From<RpcInvokeTransactionV3> for protobuf::InvokeV3WithProof` → `From<ValidResourceBounds> for protobuf::ResourceBounds` (emits `l1_data_gas: Some(zero)`).
4. Peer deserializes: `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` → returns `ValidResourceBounds::L1Gas(l1_gas)`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` matches `L1Gas` arm → returns `DEPRECATED_RESOURCE_BOUNDS_ERROR`.
6. `TryFrom<protobuf::ConsensusTransaction> for ConsensusTransaction` propagates the error → peer rejects the consensus message.
7. Independently: if hash were recomputed from `L1Gas`, it would produce H₂ ≠ H₁ (2-element preimage, no `L1_DATA_GAS` term), breaking the signature binding.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}

            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })

                ConsensusTransaction::L1Handler(txn.try_into()?)
            }
