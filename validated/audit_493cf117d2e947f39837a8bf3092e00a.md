### Title
Missing `max_l2_gas_amount` Upper-Bound Check for Declare Transactions at Gateway Admission — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds()` enforces a configurable `max_l2_gas_amount` ceiling for Invoke and DeployAccount transactions but explicitly skips the identical check for Declare transactions via an empty match arm. The production ceiling is 1,210,000,000 gas units. Any user can submit a Declare transaction with `max_l2_gas_amount = u64::MAX` (18,446,744,073,709,551,615) and it will pass gateway admission, violating the sequencer's own admission invariant.

### Finding Description

In `validate_resource_bounds()` the check reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← empty: check is intentionally skipped
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The empty `Declare` arm is the direct analog of the external report's missing health-factor gate: a bound that is enforced at one point in the pipeline (Invoke/DeployAccount admission) is absent for a structurally identical transaction type (Declare). The TODO comment confirms the developers are aware of the gap.

The production config sets `max_l2_gas_amount = 1_210_000_000` for the stateless validator. A Declare transaction carrying `l2_gas.max_amount = u64::MAX` passes `validate_resource_bounds`, is forwarded to the mempool, and is placed in the priority queue (the mempool's own `validate_resource_bounds` flag gates only on `max_l2_gas_price`, not `max_l2_gas_amount`). The transaction then reaches the blockifier.

Inside the blockifier, `validate_tx` caps the initial gas counter at `min(max_l2_gas_amount, os_constants.execute_max_sierra_gas)`. Because `u64::MAX > execute_max_sierra_gas`, the effective cap is `execute_max_sierra_gas`, so execution itself is bounded. However, the **bouncer** uses the transaction's declared resource bounds to account for block capacity before execution. If the bouncer reads `max_l2_gas_amount` directly from the transaction to decide whether the transaction fits in the current block, a Declare with `max_l2_gas_amount = u64::MAX` will cause the bouncer to see an astronomically large gas claim, potentially causing it to reject the block or to miscount remaining block capacity, leading to incorrect block-level gas accounting.

The admission-time invariant "every admitted transaction satisfies `l2_gas.max_amount ≤ config.max_l2_gas_amount`" is broken for Declare transactions. Invoke and DeployAccount transactions with the same `max_l2_gas_amount` value are correctly rejected at the gateway.

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway's own policy (`max_l2_gas_amount = 1_210_000_000`) is not applied to Declare transactions. Any unprivileged user can submit a Declare with `max_l2_gas_amount = u64::MAX` and have it admitted, queued, and forwarded to the batcher. This violates the admission invariant that is consistently enforced for all other transaction types. Downstream, if the bouncer consults the declared `max_l2_gas_amount` for block-capacity accounting, the oversized value can cause incorrect gas/bouncer accounting (Critical scope: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact").

### Likelihood Explanation

**High.** The trigger requires no privilege: any user can craft a V3 Declare transaction with `l2_gas.max_amount = u64::MAX`. The gap is unconditional — it applies regardless of network state, block height, or configuration. The TODO comment confirms the gap is known but unresolved.

### Recommendation

Remove the empty `Declare` arm and apply the same `max_l2_gas_amount` ceiling to Declare transactions:

```rust
fn validate_resource_bounds(
    &self,
    tx: &RpcTransaction,
) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }

    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }

    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { … });
    }

    // Apply max_l2_gas_amount to ALL transaction types, including Declare.
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
    }

    Ok(())
}
```

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a valid Sierra contract class.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised.
4. Submit an equivalent `RpcInvokeTransactionV3` with the same `max_amount`. Observe that it is rejected with `MaxGasAmountTooHigh { gas_amount: u64::MAX, max_gas_amount: 1_210_000_000 }`.

The divergence is confirmed by the existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` lines 191–201, which explicitly asserts that a Declare transaction with `max_amount = 200` (exceeding the configured limit of `100`) passes validation — a test that would fail for any other transaction type.
