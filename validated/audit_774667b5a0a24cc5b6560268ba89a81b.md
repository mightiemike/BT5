### Title
Native ETH Permanently Locked in `DirectDepositV1` — Recovery Path Broken Due to Missing `receive()` in `ContractOwner` - (File: `core/contracts/ContractOwner.sol`, `core/contracts/DirectDepositV1.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `fallback()` function. `DirectDepositV1.withdrawNative()` is `onlyOwner` and sends ETH back to `msg.sender` via a low-level call. Because the owner of every DDA is `ContractOwner`, the ETH transfer always fails. The intended recovery function `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` is therefore permanently broken, and any native ETH that ends up in a DDA is irrecoverable.

---

### Finding Description

`DirectDepositV1` is deployed via CREATE2 with a fixed salt (`bytes32(uint256(1))`) inside `ContractOwner.createDirectDepositV1`. Its constructor explicitly handles the case where ETH is already present at the deployment address (e.g., pre-funded via `selfdestruct` to the deterministic address):

```solidity
// DirectDepositV1.sol lines 52–59
uint256 balance = address(this).balance;
if (balance != 0) {
    // shouldn't revert even if the transfer fails, otherwise the funds
    // will be stuck in the DDA forever.
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);
    }
}
``` [1](#0-0) 

The comment itself acknowledges that a failed wrap leaves ETH stuck forever. The protocol provides a recovery path via `ContractOwner.withdrawFromDirectDepositV1`:

```solidity
// ContractOwner.sol lines 628–636
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    ...
}
``` [2](#0-1) 

This calls `withdrawNative()` on the DDA:

```solidity
// DirectDepositV1.sol lines 108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
``` [3](#0-2) 

`onlyOwner` restricts this to `ContractOwner` (the deployer, set as owner at construction). When `ContractOwner` calls `withdrawNative()`, `msg.sender` is `ContractOwner`. The low-level ETH transfer to `ContractOwner` then fails because **`ContractOwner` has no `receive()` or `fallback()` function** — confirmed by a full-file search returning zero matches. The `require(success, ...)` then reverts the entire transaction. [4](#0-3) 

---

### Impact Explanation

Any native ETH that reaches a `DirectDepositV1` contract and is not successfully wrapped (constructor path: `NativeTokenTransferFailed` emitted) is permanently locked. The only designated recovery function (`withdrawFromDirectDepositV1` with `token == address(0)`) always reverts. There is no alternative ETH withdrawal path in either contract.

---

### Likelihood Explanation

The trigger requires pre-funding the deterministic CREATE2 address before DDA deployment. The salt is fixed (`bytes32(uint256(1))`), and all constructor arguments (`endpoint`, `spotEngine`, `subaccount`, `wrappedNative`) are observable on-chain or derivable. An attacker can compute the DDA address for any target subaccount and `selfdestruct`-fund it. If `wrappedNative.call` then fails (e.g., due to gas stipend exhaustion, a non-standard WETH, or a future upgrade), ETH is locked with no recovery. Even without an active attacker, the broken recovery function is a latent permanent loss of funds risk for any ETH that reaches a DDA by any means.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, refactor `withdrawNative()` to accept an explicit `recipient` parameter instead of using `msg.sender`, and pass the final owner address directly from `withdrawFromDirectDepositV1`.

---

### Proof of Concept

1. Attacker computes the CREATE2 address for a target subaccount's DDA using the fixed salt `bytes32(uint256(1))` and known constructor args.
2. Attacker deploys a contract, funds it with ETH, and calls `selfdestruct(ddaAddress)` — ETH is now at the DDA address before deployment.
3. Anyone calls `ContractOwner.creditDepositV1(subaccount)` or `createDirectDepositV1(subaccount)`.
4. DDA constructor runs: `address(this).balance != 0`, attempts `wrappedNative.call{value: balance}("")`.
5. If the call fails (any reason), `NativeTokenTransferFailed` is emitted and ETH is stuck.
6. Owner calls `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))`.
7. This calls `DirectDepositV1.withdrawNative()` → `ContractOwner.call{value: balance}("")` → fails (no `receive()`).
8. `require(success, "Failed to transfer native token to owner")` reverts.
9. ETH is permanently locked in the DDA with no recovery path. [5](#0-4) [6](#0-5)
