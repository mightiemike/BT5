Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the actual user, allowing any unprivileged caller to bypass the per-pool swap allowlist via `MetricOmmSimpleRouter` — (`File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which resolves to `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the actual user. If the pool admin allowlists the router — the natural step to enable router-mediated swaps for their allowlisted users — every unprivileged address on the network can bypass the individual allowlist by calling any router entry-point. This is architecturally inconsistent with `DepositAllowlistExtension`, which correctly gates on `owner` (the actual beneficiary), not `sender` (the intermediary).

## Finding Description

**Root cause — wrong actor checked in `beforeSwap`:**

`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and checks it against the per-pool allowlist:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension's caller), and `sender` is the first argument forwarded by the pool. `MetricOmmPool.swap` populates that argument with its own `msg.sender`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // whoever called pool.swap()
  recipient,
  ...
);
``` [2](#0-1) 

**Router call path — router becomes `msg.sender`:**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly. The actual user's address is stored in transient storage via `_setNextCallbackContext` for the payment callback, but is never forwarded to the pool's `swap()` call:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData
  );
``` [3](#0-2) 

So the extension sees `sender = address(router)`. The allowlist check becomes `allowedSwapper[pool][router]`. Once the router is allowlisted, every caller of the router passes the check unconditionally.

**Contrast with `DepositAllowlistExtension` — correct pattern:**

`DepositAllowlistExtension.beforeAddLiquidity` gates on `owner` (the second parameter, the actual beneficiary), not `sender` (the intermediary):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [4](#0-3) 

The swap allowlist has no equivalent `recipient` or payer field to gate on; it only receives `sender`, which is the intermediary when a router is in the call path.

**Existing guards are insufficient:** The `isAllowedToSwap` view function returns `true` for the router address, giving the admin no on-chain signal that allowlisting the router opens the pool to all users. There is no mechanism in the current extension interface to distinguish the router from the actual user.

## Impact Explanation

Any address can swap in a pool the admin intended to restrict to a specific set of counterparties (e.g., KYC-verified addresses, whitelisted protocols, or institutional partners). The bypass is unconditional once the router is allowlisted: the attacker supplies any valid `amountIn`, the pool executes the swap at the oracle price, and the attacker receives output tokens. The pool's access-control invariant — that only allowlisted addresses may trade — is permanently broken for all router-mediated paths. This constitutes an admin-boundary break with direct fund-flow consequences: unauthorized parties extract value from a restricted pool.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing swap interface. A pool admin who wants to allow router-mediated swaps for their allowlisted users will naturally add the router to `allowedSwapper`. The admin has no on-chain signal that doing so opens the pool to all users. The trigger requires only a standard router call with no special permissions, making this reachable by any unprivileged user with no preconditions beyond the router being allowlisted.

## Recommendation

Gate `SwapAllowlistExtension.beforeSwap` on the economically relevant actor — the address that pays tokens in — rather than the immediate pool caller. Two concrete approaches:

1. **Decode the payer from `extensionData`**: Require the router to encode `msg.sender` (the actual user) into `extensionData` and have the extension decode and check that address. This is consistent with how `extensionData` is already forwarded through the full hook chain.

2. **Mirror the deposit pattern**: Introduce a `payer` field analogous to `owner` in the liquidity hooks, passed explicitly by the pool from the callback context, so the extension always sees the address that funds the swap.

Until fixed, document that allowlisting the router in `SwapAllowlistExtension` is equivalent to calling `setAllowAllSwappers(pool, true)`.

## Proof of Concept

```
Setup:
  1. Pool admin deploys MetricOmmPool with SwapAllowlistExtension as beforeSwap hook.
  2. Admin calls setAllowedToSwap(pool, alice, true)       // alice is the only allowed swapper
  3. Admin calls setAllowedToSwap(pool, router, true)      // enable router-mediated swaps for alice

Attack:
  4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
         pool:             <restricted pool>,
         tokenIn:          token1,
         zeroForOne:       false,
         amountIn:         X,
         amountOutMinimum: 0,
         recipient:        bob,
         ...
       })

  5. Router calls pool.swap(bob, false, X, limit, "", "")
     → pool calls _beforeSwap(msg.sender=router, ...)
     → SwapAllowlistExtension.beforeSwap(sender=router, ...)
     → allowedSwapper[pool][router] == true  → check passes
     → swap executes at oracle price
     → Bob receives token0 output

Result: Bob, who is not individually allowlisted, successfully swaps in a pool
        the admin intended to restrict to alice only.
        The allowlist guard is fully bypassed via the public router.
```
