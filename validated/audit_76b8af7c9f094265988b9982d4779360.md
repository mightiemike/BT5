Audit Report

## Title
SwapAllowlistExtension checks router address instead of original user, enabling full allowlist bypass via MetricOmmSimpleRouter - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the original user. If the pool admin allowlists the router address (required for any router-mediated swap to succeed on an allowlisted pool), every unprivileged user can bypass the allowlist by calling the router instead of the pool directly.

## Finding Description
**Call path:**

1. `MetricOmmPool.swap` passes `msg.sender` as `sender` to `ExtensionCalling._beforeSwap` (MetricOmmPool.sol line 230–240).
2. `_beforeSwap` encodes `sender` as the first argument and calls the extension (ExtensionCalling.sol lines 160–176).
3. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is whoever called `pool.swap()` (SwapAllowlistExtension.sol lines 37–39).
4. `MetricOmmSimpleRouter.exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)` directly, so the pool sees `msg.sender = router` (MetricOmmSimpleRouter.sol lines 72–80, 104–112, 136–137, 165–181).

**Root cause:** The extension receives the router's address as `sender`, not the original EOA. The pool has no mechanism to forward the original caller's identity through the router.

**Bypass scenario:**
- Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd addresses.
- To allow allowlisted users to use the router, the admin calls `setAllowedToSwap(pool, router, true)`.
- Any non-allowlisted user calls `router.exactInputSingle(...)`. The pool sees `msg.sender = router`, the extension sees `sender = router`, finds it allowlisted, and permits the swap.
- The non-allowlisted user successfully trades on a pool that was supposed to be restricted.

**Existing guards are insufficient:** The `onlyPool` modifier on the extension only verifies the caller is a registered pool — it does not recover the original user. The router's `_requireExpectedCallbackCaller` only validates the callback caller is the expected pool; it does not affect what `sender` the pool forwards to the extension.

## Impact Explanation
Any unprivileged user can bypass a curated pool's swap allowlist by routing through `MetricOmmSimpleRouter`. This is a direct curation failure: the pool admin's intent to restrict trading to specific addresses is completely defeated. Disallowed users can trade against LP positions, extracting value from pools that were designed to be permissioned (e.g., institutional pools, KYC-gated pools). This meets the "admin-boundary break by an unprivileged path" and "broken core pool functionality" criteria at High severity.

## Likelihood Explanation
The bypass requires only that the pool admin has allowlisted the router address — a necessary step for any allowlisted user to use the router. Any attacker who observes this configuration (fully on-chain) can immediately exploit it with a standard router call. No special privileges, flash loans, or timing are required. The attack is repeatable on every block.

## Recommendation
Pass the original user's address through the router to the extension. One approach: include the original `msg.sender` in the `extensionData` field and have the extension decode and verify it, authenticated by the pool (since only the pool calls the extension). A cleaner approach is to add an originator field to the swap call path so the pool can forward the true initiator to extensions, similar to how Uniswap v4 hooks receive `tx.origin` or a signed permit. Alternatively, document that the router must never be allowlisted and that allowlisted users must call the pool directly — but this breaks the router's utility for curated pools.

## Proof of Concept

// 1. Deploy pool with SwapAllowlistExtension
// 2. Admin allowlists the router: swapExtension.setAllowedToSwap(pool, address(router), true)
// 3. Non-allowlisted attacker calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: address(pool),
    recipient: attacker,
    tokenIn: token0,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// pool.swap() is called with msg.sender = router
// SwapAllowlistExtension sees sender = router (allowlisted) → swap succeeds
// Attacker bypasses allowlist
