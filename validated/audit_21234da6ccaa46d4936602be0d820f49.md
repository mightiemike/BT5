### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Callable, Bypassing Coinbase Merkle Proof Validation — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) was deprecated in favour of `verify_transaction_inclusion_v2`, which adds a coinbase merkle proof check to prevent the 64-byte transaction Merkle proof forgery attack. However, v1 is still a live, publicly callable NEAR entry-point. Any unprivileged caller can invoke it directly, bypassing the coinbase validation entirely, and receive a `true` proof result for a transaction that was never included in a Bitcoin block.

---

### Finding Description

`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle proof forgery vulnerability (documented in the function's own NatSpec comment referencing the BitMEX disclosure). It enforces an additional coinbase merkle proof check before delegating to v1: [1](#0-0) 

The coinbase check requires that the coinbase transaction (always at index 0, always a real transaction) can be proven to belong to the same block's Merkle root. This prevents an attacker from supplying a 64-byte internal Merkle node as a fake `tx_id` and constructing a valid-looking proof path.

However, v1 is still decorated only with `#[deprecated]` and `#[pause]` — it carries **no access-control restriction** and remains a fully callable public NEAR method: [2](#0-1) 

The `#[deprecated]` attribute is a Rust compiler hint; it has no effect on the compiled WASM or on NEAR's method dispatch. Any account can call `verify_transaction_inclusion` directly, receiving a boolean result that is computed **without** the coinbase proof check.

This is structurally identical to the reported FeeCollector pattern: a security mechanism (`coinbase_merkle_proof` validation) is set up in one path (v2) but the old path (v1) remains reachable and silently ignores it.

---

### Impact Explanation

A recipient smart contract or off-chain relayer that calls `verify_transaction_inclusion` (v1) can be fed a forged SPV proof. The attacker supplies a 64-byte value as `tx_id` that is actually an internal Merkle tree node, constructs a valid Merkle path to the block's `merkle_root`, and the function returns `true`. The block header itself is genuine and on-chain; only the claimed transaction is fabricated. Any bridge, escrow, or cross-chain application that trusts this return value to release funds or update state is vulnerable to theft.

---

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_transaction_inclusion` with a crafted `ProofArgs`. The 64-byte forgery technique is publicly documented and the contract's own comments reference it. The only precondition is that the attacker knows a valid block hash stored in the contract's mainchain map, which is public state.

---

### Recommendation

Remove `verify_transaction_inclusion` (v1) from the contract entirely, or gate it with an access-control role so it cannot be called by unprivileged accounts. The v2 path already calls v1 internally via `#[allow(deprecated)]`, so removing the public entry-point does not break v2.

```rust
// Remove or restrict the public method:
// pub fn verify_transaction_inclusion(...) -> bool { ... }

// If backward compatibility is required, gate it:
#[access_control_any(roles(Role::DAO))]
pub fn verify_transaction_inclusion(...) -> bool { ... }
```

---

### Proof of Concept

1. Attacker identifies any valid `tx_block_blockhash` stored in `mainchain_header_to_height` (public state).
2. Attacker retrieves the block's `merkle_root` from `headers_pool`.
3. Attacker constructs a 64-byte `tx_id` that is an internal Merkle node and a `merkle_proof` path such that `compute_root_from_merkle_proof(tx_id, tx_index, proof) == merkle_root`.
4. Attacker calls `verify_transaction_inclusion` (v1) directly — **not** v2 — with `confirmations = 1`.
5. v1 skips the coinbase check entirely and evaluates only: [3](#0-2) 

6. The function returns `true` for a transaction that does not exist on Bitcoin.
7. Any downstream contract consuming this result releases funds or updates state based on a forged proof.

    #[deprecated(
        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
