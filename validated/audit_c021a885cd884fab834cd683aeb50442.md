Audit Report

## Title
`SwapAllowlistExtension` checks the immediate caller (`sender`) instead of the originating user, allowing full allowlist bypass via `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps on `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of `pool.swap()`. When `MetricOmmSimpleRouter` is the caller, `sender` is the router address, not the originating user. If the router is allowlisted, every user — including non-allowlisted ones — can bypass the per-user restriction. If the router is not allowlisted, individually allowlisted users cannot swap through the primary periphery entry point at all.

## Finding Description

**Root cause — identity mismatch in `beforeSwap`:**

`MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap()`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards `sender` unchanged to every configured extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, i.e. `allowedSwapper[pool][caller_of_pool_swap]`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly — the router is `msg.sender` to the pool, not the originating user: [4](#0-3) 

The pool therefore passes `sender = router_address` to the extension. The allowlist check becomes `allowedSwapper[pool][router]`, completely ignoring the actual end-user.

**Contrast with `DepositAllowlistExtension`**, which correctly checks `owner` (the economically relevant actor, the second parameter), not `sender` (the immediate caller): [5](#0-4) 

The two allowlist extensions are structurally inconsistent: deposit gating is correct; swap gating is not.

## Impact Explanation

**Bypass path (High):** A pool admin who allowlists the router — the natural configuration to enable normal UX — inadvertently opens the pool to every user. Any non-allowlisted address calls `router.exactInputSingle(pool, ...)` and the extension sees `allowedSwapper[pool][router] = true`, granting the swap. The per-user restriction is completely nullified, allowing unauthorized parties to drain favorable pricing from restricted pools.

**Broken-functionality path (Medium):** If the router is *not* allowlisted, individually allowlisted users cannot swap through the router at all, because the extension sees `allowedSwapper[pool][router] = false` and reverts with `NotAllowedToSwap`. Since the router is the primary user-facing entry point, this renders the allowlist incompatible with normal usage and breaks core swap functionality for legitimate users.

Both paths have direct fund-impact consequences meeting Sherlock thresholds.

## Likelihood Explanation

- `MetricOmmSimpleRouter` is the primary user-facing swap entry point in the protocol.
- Pool admins configuring a restricted pool are likely to allowlist the router to enable normal UX, without realizing this collapses per-user gating.
- No special privilege or malicious setup is required: any EOA can call the public router.
- The flaw is not obvious from reading `SwapAllowlistExtension` in isolation, since `DepositAllowlistExtension` works correctly and the inconsistency only appears on cross-extension comparison.

## Recommendation

**Short term:** In `SwapAllowlistExtension.beforeSwap`, check `recipient` (the second parameter, the address receiving swap output) instead of `sender`, or — more robustly — require the router to encode the originating user address in `extensionData` and have the extension decode and check that value.

**Long term:** Define a protocol-level convention for "the economically responsible actor" on each hook type and enforce it consistently across all extensions. For swaps, the router should forward the originating user address in `extensionData` so guards can gate the correct identity regardless of call depth.

## Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension configured in BEFORE_SWAP_ORDER.
2. Pool admin calls:
       extension.setAllowedToSwap(pool, router, true)
   (allowlists the router, intending to permit router-mediated swaps)
3. Non-allowlisted user `alice` calls:
       router.exactInputSingle({pool: pool, recipient: alice, ...})
4. Router executes pool.swap(alice, zeroForOne, amount, limit, "", extensionData)
   with msg.sender = router.
5. Pool calls _beforeSwap(sender=router, recipient=alice, ...).
6. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true
   → swap proceeds.
7. Alice, who was never individually allowlisted, successfully swaps on the
   restricted pool. Any non-allowlisted user can repeat this.
```
