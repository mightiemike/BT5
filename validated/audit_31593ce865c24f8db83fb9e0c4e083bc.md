### Title
Zero Fee Rates on Newly Added Products for Non-Default Fee Tiers Allow Fee-Free Trading — (`core/contracts/OffchainExchange.sol`)

---

### Summary

When a new product is registered in `SpotEngine` or `PerpEngine`, the `feeRates[tier][productId]` mapping in `OffchainExchange` is uninitialized (zero) for every non-default fee tier. Because `nonDefaultFeeTierMask` is set permanently once any tier is configured and is never cleared, any user already holding a non-default fee tier can trade on the new product at **zero taker and maker fees** until the admin explicitly calls `updateTierFeeRates` for the new product. This is a direct analog to H-03: a new market is added with zero rates by default, and the window before rates are atomically set is exploitable.

---

### Finding Description

`OffchainExchange.getTierFeeRateX18` decides which fee schedule to apply:

```solidity
function getTierFeeRateX18(uint32 tier, uint32 productId)
    public view returns (FeeRates memory)
{
    if (nonDefaultFeeTierMask & (1 << tier) != 0) {
        return feeRates[tier][productId];   // ← returns {0,0} for any new product
    }
    return FeeRates({
        makerRateX18: 0,
        takerRateX18: 200_000_000_000_000   // 2 bps default
    });
}
``` [1](#0-0) 

The branch `nonDefaultFeeTierMask & (1 << tier) != 0` is taken whenever **any** prior call to `updateTierFeeRates` has touched that tier. `nonDefaultFeeTierMask` is only ever ORed into, never cleared:

```solidity
nonDefaultFeeTierMask |= uint128(1) << txn.tier;
``` [2](#0-1) 

When `updateTierFeeRates` is called with `productId == QUOTE_PRODUCT_ID`, it iterates over all **currently registered** products and writes their fee rates:

```solidity
for (uint32 i = 0; i < spotProductIds.length; i++) {
    if (spotProductIds[i] == QUOTE_PRODUCT_ID) { continue; }
    feeRates[txn.tier][spotProductIds[i]] = FeeRates(txn.makerRateX18, txn.takerRateX18);
}
``` [3](#0-2) 

Any product added **after** this call is absent from the loop. Its `feeRates[tier][newProductId]` slot remains at the Solidity zero-value `{makerRateX18: 0, takerRateX18: 0}`. Because `nonDefaultFeeTierMask` already has the tier's bit set, `getTierFeeRateX18` returns `{0, 0}` for the new product — bypassing the 2 bps default entirely.

Product registration itself sets no fee rates. `SpotEngine.addOrUpdateProduct` and `PerpEngine.addOrUpdateProduct` only initialize engine-side state (cumulative multipliers, funding accumulators) and call `_exchange().updateMarket(...)`, which writes only `minSize` and `sizeIncrement` into `marketInfo`:

```solidity
function updateMarket(uint32 productId, uint32 quoteId,
    int128 sizeIncrement, int128 minSize) external {
    requireEngine();
    if (quoteId != type(uint32).max) { quoteIds[productId] = quoteId; }
    marketInfo[productId].minSize = int64(minSize / 1e9);
    marketInfo[productId].sizeIncrement = int64(sizeIncrement / 1e9);
}
``` [4](#0-3) 

No fee rate initialization occurs at product creation time.

---

### Impact Explanation

Every trade on the new product by a non-default-tier user goes through `applyFee`, which calls `getUserFeeRateWithBuilder` → `getTierFeeRateX18`. With `feeRate = 0`, the `keepRateX18 = ONE`, so `orderInfo.fee = 0`. The protocol collects nothing from these trades:

```solidity
int128 keepRateX18 = ONE - feeInfo.feeRate;
int128 newMeteredQuote = (meteredQuote > 0)
    ? meteredQuote.mul(keepRateX18)
    : meteredQuote.div(keepRateX18);
orderInfo.fee = meteredQuote - newMeteredQuote;
``` [5](#0-4) 

The protocol loses all taker fee revenue on the new product for every non-default-tier user during the uninitialized window. This can be done at scale (large notional volume, many users with non-default tiers), and it recurs on every new product listing — directly mirroring the H-03 pattern of per-token revenue loss.

---

### Likelihood Explanation

- Non-default fee tiers are assigned by the sequencer/admin via `updateFeeTier`; VIP users, market makers, and builder-affiliated accounts routinely hold them.
- New products are added periodically as the exchange expands.
- The window between `addOrUpdateProduct` and a subsequent `updateTierFeeRates` call for the new product is observable on-chain; a sophisticated user can front-run or simply trade immediately after the product listing transaction is confirmed.
- No special permissions are required for the exploiting user beyond already holding a non-default fee tier.

---

### Recommendation

Atomically initialize fee rates for all configured tiers when a new product is added. One approach: in `updateMarket` (or in `addOrUpdateProduct`), iterate over all bits set in `nonDefaultFeeTierMask` and write a sensible default (e.g., the same rates as `QUOTE_PRODUCT_ID` for that tier) into `feeRates[tier][productId]`. Alternatively, enforce that `updateTierFeeRates` with `QUOTE_PRODUCT_ID` is called in the same sequencer batch as the product addition before any `matchOrders` transaction for the new product is accepted.

---

### Proof of Concept

1. Admin calls `updateTierFeeRates({tier: 1, productId: QUOTE_PRODUCT_ID, takerRateX18: 5e14, makerRateX18: -1e14})` for all existing products. This sets `nonDefaultFeeTierMask |= (1 << 1)` and writes `feeRates[1][existingProductId]` for every currently registered product.

2. Admin calls `SpotEngine.addOrUpdateProduct(productId=100, ...)`. `feeRates[1][100]` is never written — it remains `{0, 0}`.

3. Alice holds `feeTiers[alice] = 1` (assigned previously by the sequencer).

4. Alice submits a `MatchOrders` transaction on `productId=100` for a large notional amount.

5. Inside `matchOrders` → `applyFee` → `getUserFeeRateWithBuilder` → `getTierFeeRateX18(1, 100)`:
   - `nonDefaultFeeTierMask & (1 << 1) != 0` → **true**
   - returns `feeRates[1][100]` = `{makerRateX18: 0, takerRateX18: 0}`

6. `orderInfo.fee = 0`. Alice pays zero taker fees. The protocol collects nothing.

7. This repeats for every non-default-tier user until the admin separately calls `updateTierFeeRates` for `productId=100`.
