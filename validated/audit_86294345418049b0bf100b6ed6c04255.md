### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Bypassing Coinbase Merkle Proof Forgery Guard — (File: `contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` (v1) is marked `#[deprecated]` but remains a fully public, unguarded NEAR entry point. Any unprivileged caller can invoke it directly, bypassing the coinbase Merkle proof validation that was introduced in `verify_transaction_inclusion_v2` specifically to block the 64-byte transaction Merkle proof forgery attack. The `#[deprecated]` Rust attribute is a compile-time lint only — it has zero effect on on-chain callability.

---

### Finding Description

`verify_transaction_inclusion_v2` enforces a coinbase Merkle proof check before delegating to v1: [1](#0-0) 

This check is the sole mitigation for the [64-byte transaction Merkle proof forgery vulnerability](https://www.bitmex.com/blog/64-Byte-Transactions), which allows an attacker to supply a hash of an internal Merkle tree node as a `tx_id` and have it accepted as a valid transaction hash. The contract's own documentation acknowledges this risk explicitly: [2](#0-1) 

However, `verify_transaction_inclusion` (v1) is declared `pub` with only a `#[pause]` attribute and no `#[private]` guard: [3](#0-2) 

The `#[deprecated]` macro is a Rust compiler hint — it does not remove the function from the compiled WASM ABI. Any NEAR account can call `verify_transaction_inclusion` directly, receiving a `bool` result without the coinbase proof ever being checked.

The v1 function only verifies that the supplied `tx_id` hashes to the block's `merkle_root` via the provided `merkle_proof`: [4](#0-3) 

Without the coinbase anchor, an attacker can craft a 64-byte value that is a valid internal Merkle node and present it as a `tx_id`. The function will return `true` for a transaction that was never broadcast or confirmed.

---

### Impact Explanation

**Impact: High**

Any downstream NEAR contract that calls `verify_transaction_inclusion` to gate a cross-chain action (e.g., releasing bridged assets, minting tokens, settling a payment) can be deceived into accepting a forged proof. The attacker receives a `true` result for a non-existent Bitcoin transaction, corrupting the proof-verification result that is the contract's sole security guarantee.

---

### Likelihood Explanation

**Likelihood: High**

The entry point requires no role, no stake, and no privileged key — any NEAR account can call it. The function name is stable, documented in the relayer client at: [5](#0-4) 

Existing integrators who have not migrated to v2 are already using this path. A malicious actor only needs to know the block hash of any confirmed Bitcoin block and construct a 64-byte internal node that collides with the Merkle root path.

---

### Recommendation

Remove `verify_transaction_inclusion` from the public ABI entirely by adding `#[private]` to the function, or replace its body with an unconditional `env::panic_str("use verify_transaction_inclusion_v2")`. A `#[deprecated]` annotation alone is insufficient to prevent on-chain invocation. The security invariant — that all SPV proofs must be anchored by a coinbase Merkle proof — must be enforced at the contract level, not left to caller discretion.

---

### Proof of Concept

1. Identify any confirmed Bitcoin block hash `B` already stored in the contract's `mainchain_header_to_height` map.
2. Compute a 64-byte value `N` that is an internal node of block `B`'s Merkle tree (i.e., the concatenation of two child hashes at any level).
3. Construct a `merkle_proof` path from `N` up to the Merkle root of block `B`.
4. Call `verify_transaction_inclusion` directly (not v2) with `tx_id = N`, `tx_block_blockhash = B`, and the crafted `merkle_proof`.
5. The function at lines 318–322 computes `compute_root_from_merkle_proof(N, index, proof)` and compares it to `header.block_header.merkle_root`. Because `N` is a genuine internal node, the root matches and the function returns `true` — falsely asserting that a non-existent transaction was confirmed in block `B`. [6](#0-5)

    #[deprecated(
        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."
    )]
    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
