### Title
Missing range check in `TryFrom<protobuf::Felt252> for Felt` allows malicious p2p peer to store non-canonical receipt field elements, causing RPC to return wrong `actual_fee` and L2→L1 message payload — (`crates/apollo_protobuf/src/converters/common.rs`)

### Summary

`Felt::from_bytes_be` silently reduces any 32-byte big-endian integer modulo the Stark field prime P. The p2p sync transaction path deserializes receipt fields (`actual_fee`, L2→L1 `payload` felts) through this unchecked converter, stores the silently-reduced values in the node's DB, and serves them via `starknet_getTransactionReceipt` with no receipt commitment verification anywhere in the pipeline.

### Finding Description

`TryFrom<protobuf::Felt252> for Felt` only checks that `value.elements.len() == 32`, then calls `Felt::from_bytes_be(&felt)` unconditionally. The commented-out range-checking code and the `TODO(Shahak): use from_bytes_checked once it's available` confirm the guard is intentionally absent pending an upstream API addition.

The p2p sync client's `TransactionStreamFactory::parse_data_for_block` receives `FullTransaction` objects from network peers. Each `FullTransaction` contains a `TransactionOutput` deserialized from a protobuf `Receipt`, which in turn deserializes `actual_fee` and `MessageToL1::payload` felts through the unchecked `TryFrom<protobuf::Felt252> for Felt` path. The resulting `BlockBody` is written to storage via `append_body` with no verification against the block header's `receipt_commitment`. The stored outputs are then served verbatim by `get_transaction_receipt`.

A malicious peer can craft a `TransactionWithReceipt` where any `Felt252` field encodes a value `v ≥ P`. The receiver stores `v mod P` — a different value — as the canonical field element.

**Concrete example:**
- Honest `actual_fee` = `Fee(5000)` → honest peer sends `Felt252 { elements: 5000_u128.to_be_bytes_padded() }`
- Malicious peer sends `Felt252 { elements: (P + 5000).to_bytes_be() }` → `from_bytes_be` reduces to `5000` (same value, no harm here)
- Malicious peer sends `Felt252 { elements: P.to_bytes_be() }` → reduces to `Felt::ZERO` → `actual_fee = Fee(0)` stored and served

For `MessageToL1::payload` felts (no secondary u128 range check), any `Felt252` encoding `P + k` is stored as `Felt(k)` regardless of the intended value.

### Impact Explanation

A syncing full node that connects to a malicious p2p peer will store wrong receipt data and serve it authoritatively via `starknet_getTransactionReceipt`. Clients relying on this RPC for fee accounting, L2→L1 message verification, or receipt proofs receive a wrong value with no indication of corruption. This matches: **High — RPC returns an authoritative-looking wrong value**.

### Likelihood Explanation

Any node in the p2p network can act as a sync peer. No special privilege is required. The attack requires crafting a single protobuf message with a non-canonical `Felt252` encoding. The absence of receipt commitment verification in the sync path means the corruption is never detected locally.

### Recommendation

1. **Immediate**: Add a range check in `TryFrom<protobuf::Felt252> for Felt`. The commented-out code already shows the intended logic — enable it (or use `from_bytes_be_slice` with a prior comparison against P's byte representation):
   ```rust
   // In common.rs, replace:
   Ok(Self::from_bytes_be(&felt))
   // With (once from_bytes_checked is available, or inline the check):
   if felt[0] > 0x08 || (felt[0] == 0x08 && felt > P_BYTES) {
       return Err(ProtobufConversionError::OutOfRangeValue { ... });
   }
   Ok(Self::from_bytes_be(&felt))
   ```
2. **Defense-in-depth**: Add receipt commitment verification in `TransactionStreamFactory::parse_data_for_block` — compute the receipt commitment from the received outputs and compare against the stored block header's `receipt_commitment` before calling `write_to_storage`.

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }
