### Title
`DepositAllowlistExtension` gates on caller-supplied `owner` instead of the actual depositor `sender`, allowing any non-allowlisted address to bypass the deposit guard — (`File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the pool's `msg.sender`, i.e., the real depositor) and instead checks the caller-supplied `owner` address against the allowlist. Because `owner` is a free parameter in `pool.addLiquidity`, any non-allowlisted address can pass the guard by nominating any allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and passes both `msg.sender` (as `sender`) and `owner` to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

The extension hook signature receives both values:

```solidity
function beforeAddLiquidity(
    address sender,   // ← actual depositor (msg.sender at pool level)
    address owner,    // ← caller-supplied, arbitrary
    ...
``` [2](#0-1) 

`DepositAllowlistExtension` drops `sender` entirely (unnamed first parameter) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Compare with `SwapAllowlistExtension`, which correctly checks `sender`:

```solidity
function beforeSwap(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
``` [4](#0-3) 

`_validateOwner` in `MetricOmmPoolLiquidityAdder` only rejects `address(0)`, placing no restriction on which `owner` a caller may nominate:

```solidity
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [5](#0-4) 

The test suite even documents the intended "add on behalf of another owner" pattern without any allowlist awareness: [6](#0-5) 

---

### Impact Explanation

The deposit allowlist is the pool admin's primary access-control gate for `addLiquidity`. Because the guard checks the wrong identity (`owner` instead of `sender`), it is completely ineffective: any non-allowlisted address can call `pool.addLiquidity(allowlistedAddress, salt, deltas, ...)` and the guard passes. The position is credited to the allowlisted address, but the non-allowlisted actor has successfully interacted with the pool in a way the admin explicitly intended to prevent. This is an admin-boundary break: an unprivileged path bypasses a configured access-control extension.

---

### Likelihood Explanation

Exploitation requires only knowledge of one allowlisted address (readable from `allowedDepositor` events or storage) and the ability to call `pool.addLiquidity` or `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` with that address as `owner`. No special privileges, flash loans, or oracle manipulation are needed. Any actor excluded from the allowlist can trigger this immediately.

---

### Recommendation

Replace the unnamed first parameter with `sender` and check it instead of (or in addition to) `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

This mirrors the correct pattern already used in `SwapAllowlistExtension.beforeSwap`. [3](#0-2) 

---

### Proof of Concept

```solidity
// Setup: pool has DepositAllowlistExtension; only `alice` is allowlisted.
// Attacker is not allowlisted.

address allowedAddress = alice;  // on allowlist
address attacker = makeAddr("attacker");

token0.mint(attacker, 1_000_000e18);
token1.mint(attacker, 1_000_000e18);
vm.startPrank(attacker);
token0.approve(address(pool), type(uint256).max);
token1.approve(address(pool), type(uint256).max);

// Attacker calls addLiquidity with allowedAddress as owner.
// The extension checks allowedDepositor[pool][allowedAddress] → true → passes.
// Attacker pays tokens; position is credited to allowedAddress.
pool.addLiquidity(
    allowedAddress,   // owner = allowlisted address (not the attacker)
    salt,
    deltas,
    callbackData,
    extensionData
);
// ✓ No revert — allowlist bypassed.
// allowedAddress now has a position funded by attacker.
```

The guard never inspects `attacker` (the real `msg.sender`/`sender`). The `NotAllowedToDeposit` revert is never reached. [3](#0-2) [7](#0-6)
