### Title
Native ETH Balance Counted in `_isDepositReady()` But Not Deposited in `DirectDepositV1.creditDeposit()` — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`ContractOwner._isDepositReady()` explicitly accounts for native ETH balance when determining whether a Direct Deposit Address (DDA) has sufficient funds to trigger a deposit. However, `DirectDepositV1.creditDeposit()` — the function that actually executes the deposit — only iterates over ERC20 token balances and contains no branch to wrap or deposit native ETH. This asymmetry means that if native ETH accumulates in the DDA (via force-send through `selfdestruct`, or a constructor-time wrapping failure), the readiness check returns a false positive while the deposit execution silently skips the native ETH, leaving it permanently stuck.

---

### Finding Description

**Root cause — `ContractOwner._isDepositReady()` (lines 576–578):**

```solidity
if (tokenAddr == wrappedNative) {
    balance += recipient.balance;  // native ETH counted here
}
```

When the product token is the wrapped native token, the readiness check adds the DDA's raw ETH balance (`recipient.balance`) to the ERC20 balance before evaluating the minimum deposit threshold. This is the "whitelist-like" branch that is aware of native ETH.

**Missing branch — `DirectDepositV1.creditDeposit()` (lines 83–101):**

```solidity
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));  // ERC20 only
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(...);
        }
    }
}
```

There is no corresponding branch to: (1) detect `address(this).balance > 0`, (2) wrap it via `wrappedNative.call{value: balance}("")`, and (3) deposit the resulting wrapped token. The "blacklist-like" execution path is entirely absent for native ETH.

**How native ETH accumulates in the DDA:**

1. **Force-send via `selfdestruct`**: Any attacker can deploy a contract holding ETH and `selfdestruct` it targeting the DDA address. ETH sent via `selfdestruct` bypasses the `receive()` hook entirely and lands directly in `address(this).balance` without triggering the wrapping logic.

2. **Constructor-time wrapping failure**: The constructor attempts to wrap any pre-existing ETH balance but does not revert on failure — it only emits `NativeTokenTransferFailed`. If the `wrappedNative` contract is unavailable at deployment time (e.g., CREATE2 pre-funding before the wrapped native contract is live), the ETH is silently stranded.

```solidity
// DirectDepositV1.sol constructor (lines 52–60)
uint256 balance = address(this).balance;
if (balance != 0) {
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);  // no revert — ETH stuck
    }
}
```

---

### Impact Explanation

Once native ETH is stranded in the DDA:

- `_isDepositReady()` returns `true` (it counts the native ETH toward the minimum deposit threshold for the wrapped native product).
- Any caller invoking `creditDeposit()` receives no error — the loop completes normally — but the native ETH is never wrapped or deposited.
- The subaccount's on-chain balance in the protocol is never credited for the stranded ETH.
- The ETH is irrecoverable by the subaccount owner; only the protocol owner can rescue it via `withdrawFromDirectDepositV1(subaccount, address(0))`.

The corrupted state delta: native ETH balance in the DDA is non-zero, `_isDepositReady()` signals readiness, but the subaccount's collateral balance in `SpotEngine` is never updated to reflect it.

---

### Likelihood Explanation

**Medium-low.** The force-send vector via `selfdestruct` is attacker-controlled and costs only the ETH being sent (the attacker loses their own funds). The constructor-time failure requires a specific deployment ordering. In both cases the trigger is externally reachable without any privileged access. The `receive()` hook does not protect against `selfdestruct`-based force-sends, making the accumulation path realistic on any live deployment.

---

### Recommendation

Add a native ETH handling branch inside `creditDeposit()`, mirroring the logic already present in `_isDepositReady()`:

```solidity
function creditDeposit() external {
    // Handle native ETH first: wrap it so it becomes depositable
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance != 0) {
        (bool success, ) = wrappedNative.call{value: nativeBalance}("");
        require(success, "Failed to wrap native token.");
    }

    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        // ... existing ERC20 deposit logic unchanged
    }
}
```

This closes the asymmetry between the readiness check and the deposit execution path, consistent with the pattern already established in `_isDepositReady()`.

---

### Proof of Concept

1. Attacker deploys a contract `Bomber` holding 1 ETH, with `selfdestruct(ddaAddress)` in its constructor, targeting a live DDA.
2. Native ETH lands in `DDA.balance` — `receive()` is never called.
3. Call `ContractOwner.isDepositReady(subaccount, false)` → returns `true` because `_isDepositReady()` adds `recipient.balance` (1 ETH) to the wrapped native balance check at line 577.
4. Call `DirectDepositV1(dda).creditDeposit()` → loop runs, `token.balanceOf(dda)` for the wrapped native product is 0 (ETH was never wrapped), `balance == 0`, deposit skipped silently.
5. Subaccount collateral in `SpotEngine` is unchanged. The 1 ETH is permanently stuck in the DDA, unrecoverable except by the protocol owner. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
