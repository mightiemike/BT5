Audit Report

## Title
Retroactive Decay Rate Application in `_checkAndUpdateWatermarks` Collapses All Bin Watermarks to Zero on Rate Update, Permanently Disabling Stop-Loss Guard — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary
`executeOracleStopLossDecay` writes a new global `decayPerSecondE8` without resetting any bin's `lastDecayTs`. On the next swap, `_checkAndUpdateWatermarks` applies the new rate retroactively over the full dormant interval `dt = block.timestamp − hwmS.lastDecayTs`. When `newRate * dt ≥ 1e8`, `_decayed` returns 0 for every bin watermark, and `_applyWatermark` treats a zero watermark as "no breach" (since `metric >= 0` is always true), silently passing the stop-loss check and permanently resetting watermarks to the current (potentially depressed) metric value.

## Finding Description

**Root cause — global rate write with no per-bin timestamp reset:**

`executeOracleStopLossDecay` atomically writes the new rate to the pool-wide config but leaves every bin's `lastDecayTs` untouched:

```solidity
// line 144
oracleStopLossConfig[pool_].decayPerSecondE8 = decay;   // no per-bin lastDecayTs reset
``` [1](#0-0) 

**Retroactive application in `_checkAndUpdateWatermarks`:**

On the next swap, `dt` spans the entire dormant period (from the last swap, not from the rate change):

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;   // full dormant interval
(uint256 hwm0, bool breach0) =
    _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
``` [2](#0-1) 

**`_decayed` floors to zero:**

```solidity
uint256 factor = ratePerSecondE8 * dt;
if (factor >= E8) return 0;
``` [3](#0-2) 

**`_applyWatermark` treats zero watermark as no breach:**

```solidity
if (metric >= hwm) return (metric, false);   // hwm == 0 → always true → no breach
``` [4](#0-3) 

**Watermark permanently reset to current (depressed) metric:**

After the silent pass, lines 281–284 write `hwm0 = metric` and `lastDecayTs = block.timestamp`, permanently anchoring the watermark at the current (potentially loss-impacted) value: [5](#0-4) 

**Existing guards are insufficient:** The timelock on `proposeOracleStopLossDecay` / `executeOracleStopLossDecay` only delays the rate write; it does not prevent the retroactive collapse. The `_validateDecay` cap (`decayPerSecondE8 ≤ 1e8`) is respected yet still permits rates that zero watermarks in seconds. [6](#0-5) 

## Impact Explanation
The NatDoc invariant — *"value per share at oracle marks cannot fall faster than drawdown (one-time) + decay × t (ongoing)"* — is broken for the entire pool on the swap immediately following the rate execution. Any swap that `OracleStopLossTriggered` would have reverted now executes freely. LPs suffer direct, unbounded principal loss through toxic-flow swaps. The guard remains disabled until the pool admin manually re-proposes and re-executes high-watermark values through the full timelock cycle. This is a fund-impacting cap failure: the stop-loss protection cap is bypassed by a pool admin action, which is within the allowed impact gate for semi-trusted pool admin roles.

## Likelihood Explanation
The trigger is the pool admin executing a legitimate, timelocked decay-rate increase — no exploit contract or external attacker is required. The condition `rate * dt ≥ 1e8` is easy to satisfy for any pool with a quiet period between the last swap and the rate-change execution:

- Rate = 100 (≈ 0.0001%/s): zeroes watermarks after `dt ≥ 1,000,000 s ≈ 11.6 days`
- Rate = 1000: zeroes after `dt ≥ 100,000 s ≈ 27.8 hours`
- Rate = 1e8 (maximum): zeroes after `dt ≥ 1 s`

Any pool that experiences a quiet period between the last swap and the rate-change execution is vulnerable. The timelock window (during which LPs could exit) does not prevent the retroactive collapse.

## Recommendation
Cache the decay rate inside each `BinHighWatermarks` record so that the rate used for a given interval is always the rate active when `lastDecayTs` was last written:

1. Add a `uint32 decayRateE8` field to `BinHighWatermarks`.
2. In `_checkAndUpdateWatermarks`, decay using `hwmS.decayRateE8` (the cached per-bin rate), then write the current global rate into `hwmS.decayRateE8` alongside the updated `lastDecayTs`.
3. In `executeOracleStopLossHighWatermarks`, also write the current global rate into `hwmS.decayRateE8` when setting watermarks.

This ensures a rate change only affects intervals that begin after the change, not the dormant period that preceded it.
