### Title
NFT Owners Who Are Not the Denom Creator Cannot Burn Their Own NFTs via `MsgBurnNFT` - (File: `x/nft/keeper/keeper.go`)

---

### Summary

The `MsgBurnNFT` message type is fully registered and exposed to users, but the underlying `BurnNFT` keeper function enforces an undocumented dual-authorization requirement: the caller must be **both** the NFT owner **and** the denom creator. Any NFT holder who is not the denom creator will have their `MsgBurnNFT` transaction rejected, making the burn entrypoint effectively inaccessible to the majority of NFT owners.

---

### Finding Description

In `x/nft/keeper/keeper.go`, the `BurnNFT` function performs two sequential authorization checks:

1. `k.IsOwner` — verifies the caller owns the specific token.
2. `k.IsDenomCreator` — verifies the caller is the original creator of the denom/collection. [1](#0-0) 

Both checks must pass. If the NFT was minted by the denom creator and then transferred to another user, that user owns the NFT but is not the denom creator. Their `MsgBurnNFT` transaction will fail at the `IsDenomCreator` check.

By contrast, the `BurnNFTUnverified` function — used internally by the IBC transfer path — only checks ownership and does **not** require denom creator status: [2](#0-1) 

This asymmetry means IBC mechanics can burn an NFT on behalf of any owner, but the user-facing `MsgBurnNFT` message cannot. The message type is registered and callable: [3](#0-2) 

And the `BurnNFT` msg server handler dispatches directly to the restricted keeper function: [4](#0-3) 

---

### Impact Explanation

Any NFT holder who is not the denom creator is permanently blocked from burning their own NFTs via the standard `MsgBurnNFT` transaction. The NFT owner field is correctly set and the message is accepted at `ValidateBasic`, but execution always fails with an authorization error. This corrupts the expected NFT ownership invariant: owning a token should grant the right to destroy it, but here ownership alone is insufficient. The NFT supply for a given denom can never be reduced by non-creator holders.

---

### Likelihood Explanation

This is triggered by any normal user who:
1. Receives an NFT minted by a denom creator (a standard workflow — denom creator mints, transfers to buyer/recipient).
2. Submits a `MsgBurnNFT` transaction signed by their own key.

This is a fully unprivileged, standard Cosmos SDK transaction path. No special access is required to hit the bug; it affects every NFT holder who is not the denom creator, which is the common case in any NFT marketplace or distribution scenario.

---

### Recommendation

Remove the `IsDenomCreator` check from `BurnNFT` in `x/nft/keeper/keeper.go`. Burning should only require that the caller is the current owner of the token. The `IsDenomCreator` guard is appropriate for minting and editing (where collection integrity must be preserved by the creator), but not for burning, where the owner's right to destroy their own asset is the only relevant authorization.

The corrected `BurnNFT` should mirror `BurnNFTUnverified` in its authorization logic — ownership check only — while retaining the denom existence check.

---

### Proof of Concept

1. Alice issues a denom (`denomID = "testdenom"`) — she is the denom creator.
2. Alice mints `tokenID = "token1"` and sets recipient to Bob's address.
3. Bob now owns `token1` in `testdenom`.
4. Bob submits `MsgBurnNFT{Sender: bob, Id: "token1", DenomId: "testdenom"}`.
5. `BurnNFT` is called: `IsOwner(ctx, "testdenom", "token1", bob)` passes.
6. `IsDenomCreator(ctx, "testdenom", bob)` fails — Bob is not the creator.
7. Transaction is rejected. Bob cannot burn his own NFT.

Meanwhile, if Bob initiates an IBC transfer of `token1` to another chain (sink-chain direction), `BurnNFTUnverified` is called internally and succeeds — demonstrating that the ownership check alone is sufficient for destruction, and the `IsDenomCreator` requirement in the user-facing path is an unintended restriction. func (k Keeper) BurnNFT(ctx sdk.Context, denomID, tokenID string, owner sdk.AccAddress) error {
	if !k.HasDenomID(ctx, denomID) {
		return sdkerrors.Wrapf(types.ErrInvalidDenom, "denom ID %s not exists", denomID)
	}

	nft, err := k.IsOwner(ctx, denomID, tokenID, owner)
	if err != nil {
		return err
	}

	_, err = k.IsDenomCreator(ctx, denomID, owner)
	if err != nil {
		return err
	}

	k.deleteNFT(ctx, denomID, nft)
	k.deleteOwner(ctx, denomID, tokenID, owner)
	k.decreaseSupply(ctx, denomID)

	return nil
}

func (m msgServer) BurnNFT(goCtx context.Context, msg *types.MsgBurnNFT) (*types.MsgBurnNFTResponse, error) {
	sender, err := sdk.AccAddressFromBech32(msg.Sender)
	if err != nil {
		return nil, err
	}

	ctx := sdk.UnwrapSDKContext(goCtx)
	if err := m.Keeper.BurnNFT(ctx, msg.DenomId, msg.Id, sender); err != nil {
		return nil, err
	}

	ctx.EventManager().EmitEvents(sdk.Events{
		sdk.NewEvent(
			types.EventTypeBurnNFT,
			sdk.NewAttribute(types.AttributeKeyDenomID, msg.DenomId),
			sdk.NewAttribute(types.AttributeKeyTokenID, msg.Id),
			sdk.NewAttribute(types.AttributeKeyOwner, msg.Sender),
		),
		sdk.NewEvent(
			sdk.EventTypeMessage,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.AttributeValueCategory),
			sdk.NewAttribute(sdk.AttributeKeySender, msg.Sender),
		),
	})

	return &types.MsgBurnNFTResponse{}, nil
