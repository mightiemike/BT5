### Title
Withdrawal Permanently Blocked at Maximum Utilization Due to Insufficient Clearinghouse Token Balance — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

When pool utilization reaches 100% (all deposited tokens are borrowed by subaccounts with negative balances), the Clearinghouse holds zero tokens for the affected product. Any call to `withdrawCollateral` will revert at the `safeTransfer` call inside `handleWithdrawTransfer` — before the `assertUtilization` guard can execute — permanently blocking all withdrawals for that product until utilization decreases organically.

---

### Finding Description

In `Clearinghouse.withdrawCollateral()`, the token transfer is unconditionally executed **before** the utilization check:

```
Line 408: handleWithdrawTransfer(token, sendTo, amount, idx);   // transfer OUT first
Line 412: spotEngine.updateBalance(productId, sender, amountRealized);
Line 413: spotEngine.assertUtilization(productId);              // check runs after
```

`handleWithdrawTransfer` (line 383) calls:

```solidity
token.safeTransfer(withdrawPool, uint256(amount));
```

This transfers tokens **from the Clearinghouse**. The Clearinghouse's actual token balance equals `totalDeposits − totalBorrows`. When utilization is at 100% (`totalBorrows == totalDeposits`), the Clearinghouse holds zero tokens for the product. The `safeTransfer` reverts with an ERC20 insufficient-balance error before `assertUtilization` (line 413) ever runs.

The `SpotEngine` permits subaccounts to carry negative balances (borrowing). There is no borrow cap or utilization ceiling enforced at borrow time — only `assertUtilization` on the withdrawal path. If enough subaccounts borrow a product, utilization reaches 100% and the Clearinghouse is drained. Since only **unhealthy** accounts can be liquidated via `ClearinghouseLiq`, healthy borrowers cannot be forced to repay, leaving the pool permanently stuck.

---

### Impact Explanation

Users with positive balances for a product cannot withdraw their funds when utilization is at 100%. Their collateral is locked inside the Clearinghouse with no on-chain recovery path. The slow-mode emergency withdrawal path (`executeSlowModeTransaction`) routes through the same `withdrawCollateral` → `handleWithdrawTransfer` → `safeTransfer` chain and fails identically. Funds remain inaccessible until borrowers voluntarily repay or are liquidated, which requires them to become unhealthy — an event that may never occur if prices are stable.

---

### Likelihood Explanation

Medium. The SpotEngine supports cross-margin borrowing of spot assets. A product with high demand (e.g., the quote token) can reach 100% utilization through normal trading activity or coordinated borrowing. No privileged access is required; any trader can borrow spot assets up to their health limit.

---

### Recommendation

1. **Enforce a utilization cap at borrow time** — reject any state update that would push `totalBorrows > MAX_UTILIZATION_RATE × totalDeposits` (e.g., 95%).
2. **Move `assertUtilization` before `handleWithdrawTransfer`** — check the post-withdrawal utilization in accounting space first; only execute the token transfer if the check passes.
3. **Add a forced-repayment or account-closing mechanism** — analogous to the Gearbox recommendation, allow the protocol to close or incentivize repayment of borrowing positions when utilization exceeds a critical threshold, even if the account is still healthy.

---

### Proof of Concept

1. Multiple subaccounts borrow a spot product (e.g., USDC) through normal trading until `totalBorrowsNormalized × cumulativeBorrowsMultiplierX18 == totalDepositsNormalized × cumulativeDepositsMultiplierX18`.
2. The Clearinghouse now holds 0 USDC.
3. A user with a positive USDC balance submits a `WithdrawCollateral` transaction via the sequencer (or slow-mode path).
4. The sequencer calls `Endpoint.submitTransactionsChecked` → `processTransaction` → `Clearinghouse.withdrawCollateral`.
5. At line 408, `handleWithdrawTransfer` is called; at line 383, `token.safeTransfer(withdrawPool, amount)` reverts with `ERC20: transfer amount exceeds balance`.
6. `assertUtilization` at line 413 never executes. The user's withdrawal is permanently blocked.

    function assertUtilization(uint32 productId) external view {
        (State memory _state, ) = getStateAndBalance(productId, X_ACCOUNT);
        int128 totalDeposits = _state.totalDepositsNormalized.mul(
            _state.cumulativeDepositsMultiplierX18
        );
        int128 totalBorrows = _state.totalBorrowsNormalized.mul(
            _state.cumulativeBorrowsMultiplierX18
        );
        require(totalDeposits >= totalBorrows, ERR_MAX_UTILIZATION);
    }
