### Title
Missing `receive()` in `ContractOwner` Permanently Breaks Native ETH Recovery from `DirectDepositV1` — (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.withdrawFromDirectDepositV1` with `token == address(0)` will always revert because `ContractOwner` declares no `receive()` or `fallback()` function, yet the recovery path routes ETH through it. Any native ETH that ends up stranded in a `DirectDepositV1` instance is permanently unrecoverable.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` is the designated admin path for recovering assets from a `DirectDepositV1` (DDA) contract. When called with `token == address(0)`, it invokes `DirectDepositV1.withdrawNative()`:

```solidity
// ContractOwner.sol lines 628–636
if (token == address(0)) {
    uint256 preBalance = address(this).balance;
    DirectDepositV1(directDepositV1).withdrawNative();   // <-- calls into DDA
    uint256 postBalance = address(this).balance;
    require(postBalance > preBalance, "empty");
    (bool success, ) = msg.sender.call{value: postBalance - preBalance}("");
    require(success, "xfer");
}
``` [1](#0-0) 

Inside `DirectDepositV1`, `withdrawNative()` pushes ETH back to its caller (`ContractOwner`) via a low-level call:

```solidity
// DirectDepositV1.sol lines 108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
``` [2](#0-1) 

`msg.sender` here is `ContractOwner`. A grep across all production Solidity files confirms that **only** `DirectDepositV1` and `MockWrappedNative` define a `receive()` function — `ContractOwner` does not: [3](#0-2) 

Because `ContractOwner` has no `receive()` or `fallback()`, the low-level ETH push from `withdrawNative()` fails unconditionally. The `require(success, ...)` inside `withdrawNative()` then reverts the entire call chain. The `postBalance > preBalance` check in `ContractOwner` is never reached.

**How ETH ends up stranded in a DDA:**

`DirectDepositV1` is deployed via CREATE2 with a fixed salt, making its address deterministic and pre-computable:

```solidity
// ContractOwner.sol lines 495–498
DirectDepositV1 directDepositV1 = new DirectDepositV1{
    salt: bytes32(uint256(1))
}(address(endpoint), address(spotEngine), subaccount, wrappedNative);
``` [4](#0-3) 

An attacker can `selfdestruct`-force ETH to that pre-computed address before deployment. The constructor then attempts to wrap it:

```solidity
// DirectDepositV1.sol lines 52–60
uint256 balance = address(this).balance;
if (balance != 0) {
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);
    }
}
``` [5](#0-4) 

Critically, the constructor **does not revert** on wrapping failure — it only emits an event. If `wrappedNative.call` fails (e.g., the wrapped-native contract is paused, upgraded, or otherwise non-functional at deployment time), the raw ETH remains in the DDA with no working recovery path.

The `receive()` function in `DirectDepositV1` also wraps ETH immediately and reverts on failure, so it cannot be used to bypass the issue:

```solidity
// DirectDepositV1.sol lines 64–67
receive() external payable {
    (bool success, ) = wrappedNative.call{value: msg.value}("");
    require(success, "Failed to wrap native token.");
}
``` [6](#0-5) 

---

### Impact Explanation

Any native ETH stranded in a `DirectDepositV1` instance is permanently locked. The only designated recovery function (`ContractOwner.withdrawFromDirectDepositV1` with `token == address(0)`) is provably broken: it will always revert because `ContractOwner` cannot receive ETH. There is no alternative recovery path in the codebase.

**Corrupted asset delta:** native ETH balance of `DirectDepositV1` — permanently unrecoverable.

---

### Likelihood Explanation

Low. The precondition requires:
1. An attacker pre-computing the CREATE2 address and selfdestructing ETH to it before deployment.
2. The `wrappedNative.call` in the constructor failing at deployment time.

Both conditions must hold simultaneously. In normal operation, the wrapping succeeds and no ETH remains. However, the recovery path being unconditionally broken is a latent, code-level defect independent of likelihood — any future scenario that leaves ETH in a DDA (including chain-level edge cases or a future wrapped-native upgrade) will hit this wall.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded from `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

Alternatively, restructure `withdrawFromDirectDepositV1` so that `DirectDepositV1.withdrawNative()` sends ETH directly to `msg.sender` (the multisig) rather than routing it through `ContractOwner`.

---

### Proof of Concept

1. Compute the deterministic CREATE2 address of `DirectDepositV1` for a target `subaccount` (salt = `bytes32(uint256(1))`).
2. Deploy a contract that `selfdestruct`s, sending ETH to that pre-computed address.
3. Call `ContractOwner.createDirectDepositV1(subaccount)` (or `creditDepositV1`). The constructor runs with `address(this).balance > 0`. If `wrappedNative.call` fails, ETH stays in the DDA and only `NativeTokenTransferFailed` is emitted — no revert.
4. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
5. Execution reaches `DirectDepositV1.withdrawNative()`, which calls `ContractOwner.call{value: balance}("")`.
6. `ContractOwner` has no `receive()` — the call returns `success = false`.
7. `require(success, "Failed to transfer native token to owner")` reverts.
8. ETH is permanently locked in the DDA.
