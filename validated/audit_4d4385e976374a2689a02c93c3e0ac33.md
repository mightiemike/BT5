### Title
Native Token Recovery from `DirectDepositV1` Permanently Blocked Due to Missing `receive()` in `ContractOwner` — (`ContractOwner.sol`)

---

### Summary

`ContractOwner` has no `receive()` or `fallback()` function, yet `DirectDepositV1.withdrawNative()` unconditionally sends ETH to `msg.sender` (i.e., `ContractOwner`) via a bare `call{value: balance}("")`. This causes `withdrawFromDirectDepositV1` to always revert when `token == address(0)`, permanently locking any native tokens stranded in a `DirectDepositV1` (DDA) contract.

---

### Finding Description

`ContractOwner.withdrawFromDirectDepositV1` is the designated recovery path for native tokens held in a DDA: [1](#0-0) 

```solidity
// ContractOwner.sol L622–636
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external
    onlyOwner
{
    address payable directDepositV1 = directDepositV1Address[subaccount];
    require(directDepositV1 != address(0), "no dda");
    if (token == address(0)) {
        uint256 preBalance = address(this).balance;
        DirectDepositV1(directDepositV1).withdrawNative();   // <-- sends ETH to ContractOwner
        uint256 postBalance = address(this).balance;
        require(postBalance > preBalance, "empty");
        ...
    }
}
```

`DirectDepositV1.withdrawNative()` sends the DDA's entire ETH balance to `msg.sender` (which is `ContractOwner`) and hard-requires the transfer to succeed: [2](#0-1) 

```solidity
// DirectDepositV1.sol L108–112
function withdrawNative() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool success, ) = msg.sender.call{value: balance}("");
    require(success, "Failed to transfer native token to owner");
}
```

`ContractOwner` inherits only from `EIP712Upgradeable` and `OwnableUpgradeable` and defines no `receive()` or `fallback()` function: [3](#0-2) 

Neither parent contract implements `receive()`. Consequently, the `call{value: balance}("")` into `ContractOwner` always fails, `require(success, ...)` reverts, and the entire `withdrawFromDirectDepositV1` call reverts.

**How native tokens end up stranded in a DDA:**

The DDA constructor explicitly handles the case where ETH is already present at the CREATE2 address at deployment time. If the wrapping call to `wrappedNative` fails, the ETH is silently left in the DDA: [4](#0-3) 

```solidity
// DirectDepositV1.sol L52–60
uint256 balance = address(this).balance;
if (balance != 0) {
    // shouldn't revert even if the transfer fails, otherwise the funds
    // will be stuck in the DDA forever.
    (bool success, ) = wrappedNative.call{value: balance}("");
    if (!success) {
        emit NativeTokenTransferFailed(balance);
    }
}
```

The comment itself acknowledges the risk of permanent lock. The `withdrawNative()` function is the intended escape hatch for exactly this scenario — but it is broken because `ContractOwner` cannot receive ETH.

The DDA is deployed with a fixed salt (`bytes32(uint256(1))`), making its address fully deterministic and predictable before deployment: [5](#0-4) 

Any party can front-run the DDA deployment by sending ETH to its pre-computed address. If `wrappedNative` is unavailable or reverts at that moment, the ETH is stranded and unrecoverable.

---

### Impact Explanation

Native tokens stranded in a `DirectDepositV1` contract (due to failed wrapping at construction) are permanently locked. The only designated recovery function, `withdrawFromDirectDepositV1(subaccount, address(0))`, always reverts because `ContractOwner` cannot receive ETH. The funds cannot be recovered through any other on-chain path in the protocol.

---

### Likelihood Explanation

The DDA's CREATE2 address is deterministic and publicly computable. A griever (or accidental sender) can deposit ETH to the address before deployment. If `wrappedNative` is temporarily non-functional at deployment time (e.g., during an upgrade or incident), the constructor silently leaves the ETH in the DDA and emits `NativeTokenTransferFailed`. From that point, the ETH is permanently unrecoverable. The likelihood is low but non-zero, and the constructor's own comment acknowledges the risk.

---

### Recommendation

Add a `receive()` function to `ContractOwner` so it can accept ETH forwarded by `DirectDepositV1.withdrawNative()`:

```solidity
receive() external payable {}
```

This mirrors the exact mitigation recommended in the referenced M-09 report and unblocks the `withdrawFromDirectDepositV1` recovery path.

---

### Proof of Concept

1. Compute the deterministic CREATE2 address for a DDA for a given `subaccount` (salt = `bytes32(uint256(1))`).
2. Send 1 ETH to that address before `createDirectDepositV1` is called.
3. Arrange for `wrappedNative.call{value: 1 ether}("")` to fail at construction time (e.g., mock `wrappedNative` that reverts).
4. Call `ContractOwner.createDirectDepositV1(subaccount)` — constructor runs, wrapping fails, `NativeTokenTransferFailed(1 ether)` is emitted, 1 ETH remains in the DDA.
5. Call `ContractOwner.withdrawFromDirectDepositV1(subaccount, address(0))` as the owner.
6. Observe revert: `DirectDepositV1.withdrawNative()` calls `ContractOwner.call{value: 1 ether}("")`, which fails because `ContractOwner` has no `receive()`, triggering `require(success, "Failed to transfer native token to owner")`.
7. The 1 ETH is permanently locked in the DDA with no recovery path.
