Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates on the router address instead of the originating user, allowing any caller to bypass per-user swap allowlists via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the pool's `msg.sender` — the router contract — not the originating user. When a pool admin allowlists the router to support standard periphery swaps, every unpermissioned user can bypass the curated allowlist by routing through `MetricOmmSimpleRouter`. There is no configuration that simultaneously enforces per-user access control and supports the standard router path.

## Finding Description

`MetricOmmPool.swap` passes its own `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // caller of pool.swap() — the router, not the end user
  recipient,
  ...
);
```

When `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`) calls `pool.swap(...)`, the pool's `msg.sender` is the router contract address, not the end user:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(
    params.recipient,
    params.zeroForOne,
    MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
    priceLimitX64,
    "",
    params.extensionData   // extensionData comes from caller; extension ignores it
  );
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37-39
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`. The `bytes calldata` (extensionData) parameter is present in the function signature but is never decoded or used to recover the originating user.

This creates an irreconcilable dilemma for the pool admin:

| Admin action | Effect |
|---|---|
| Allowlist individual users only | Allowlisted users **cannot** swap through the router (broken functionality) |
| Allowlist the router to support periphery path | **Every** user can bypass the allowlist through the router |

The `DepositAllowlistExtension` is not affected in the same way because it gates on `owner` (the position recipient), which the liquidity adder passes explicitly and which the pool enforces separately.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd users, whitelisted market makers) is fully bypassed once the pool admin allowlists the router. Any unpermissioned user can call `MetricOmmSimpleRouter.exactInputSingle` and execute swaps on the curated pool, trading at oracle-derived prices without authorization. This is a direct, demonstrable break of the access-control invariant the extension was deployed to enforce, with fund-impacting consequences for LPs who deposited under the assumption that only vetted counterparties could trade against their positions.

## Likelihood Explanation

Medium. The router is the canonical periphery swap path documented and deployed by the protocol. A pool admin who wants to support standard tooling (wallets, aggregators, front-ends) while maintaining an allowlist will naturally attempt to allowlist the router address. The admin has no on-chain signal that doing so opens the allowlist to all users; the `isAllowedToSwap` view function returns `true` for the router, giving no indication of the bypass. The scenario requires a deliberate admin action, but it is the expected action for any operator who wants both access control and router compatibility.

## Recommendation

Pass the originating user's address through the swap path so the extension can gate on the actual economic actor:

1. **Router-side**: Add an optional `originSender` field to swap parameters; the router populates it with `msg.sender` and passes it encoded in `extensionData`. The extension decodes and checks it.
2. **Extension-side**: `SwapAllowlistExtension.beforeSwap` should accept an encoded `originSender` from `extensionData` and fall back to `sender` only when no override is present, so router-mediated swaps carry the real user identity.

Either approach closes the gap between the actor the allowlist intends to gate and the actor the extension actually observes.

## Proof of Concept

```
Setup:
  pool configured with SwapAllowlistExtension (BEFORE_SWAP_ORDER = extension 1)
  pool admin calls setAllowedToSwap(pool, router, true)   // to support periphery path
  pool admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker (not allowlisted) calls:
    MetricOmmSimpleRouter.exactInputSingle({
        pool: pool,
        recipient: attacker,
        zeroForOne: true,
        amountIn: X,
        ...
    })

  Execution trace:
    router.exactInputSingle()
      → pool.swap(recipient=attacker, ...)   // msg.sender = router
        → _beforeSwap(sender=router, ...)
          → SwapAllowlistExtension.beforeSwap(sender=router, ...)
            → allowedSwapper[pool][router] == true  ✓  (no revert)
        → swap executes at oracle price
        → attacker receives token output

  Result: attacker swaps on a curated pool without being individually allowlisted.
```
