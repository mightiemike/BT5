### Title
SwapAllowlistExtension Gates the Router Address Instead of the End User, Allowing Any User to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]` where `sender` is the immediate caller of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `sender` equals the router address, not the actual end user. If the router is allowlisted (the only way to permit router-mediated swaps for allowlisted users), every user — including those explicitly excluded — can bypass the allowlist by routing through the router.

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks that `sender` is in the per-pool allowlist: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router is the direct caller of `pool.swap()`: [3](#0-2) 

So `sender` received by the extension is the **router address**, not the actual end user (`msg.sender` of the router call). The extension has no visibility into who called the router.

This creates an inescapable dilemma for the pool admin:

- **Do not allowlist the router** → even legitimately allowlisted users cannot use the router.
- **Allowlist the router** → every user, including those explicitly blocked, can bypass the allowlist by routing through the router.

The same bypass applies to multi-hop `exactInput` and `exactOutput` calls, and to intermediate hops where `sender` becomes `address(this)` (the router itself): [4](#0-3) 

### Impact Explanation

The `SwapAllowlistExtension` is the protocol's primary mechanism for restricting which addresses may trade against a pool. A complete bypass of this guard means:

- Any user can swap against a pool that was configured to be permissioned (e.g., KYC-gated, institutional-only, or whitelist-only pools).
- LP providers who deposited under the assumption that only vetted counterparties would trade against their positions are exposed to unrestricted counterparty flow, which can include adversarial or regulatory-non-compliant actors.
- The pool admin's configured access boundary is silently voided for all router-mediated swaps.

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool admin who enables the `SwapAllowlistExtension` and also allowlists the router (the natural and expected configuration) is immediately vulnerable. The attacker needs no special privileges — a standard `exactInputSingle` call suffices.

### Recommendation

The extension must verify the actual end user, not the intermediate router. Two approaches:

1. **Check `msg.sender` directly in the extension for direct pool calls, and require the router to forward the real user identity** via `extensionData`. The extension would decode the real user from `extensionData` when `sender` is a known router, and verify it against the allowlist.

2. **Redesign the allowlist to gate by `recipient` instead of (or in addition to) `sender`**, since `recipient` is the address that receives swap output and is harder to spoof without economic loss to the attacker.

3. **Alternatively, remove router allowlisting entirely** and require all allowlisted users to call `pool.swap()` directly, accepting that the router cannot be used with this extension.

### Proof of Concept

```solidity
// Setup: pool with SwapAllowlistExtension; only `allowedUser` is allowlisted.
// Pool admin also allowlists the router so allowedUser can use it.
swapAllowlist.setAllowedToSwap(pool, address(router), true);

// Attacker (not in allowlist) calls the router directly.
// The extension sees sender = address(router), which IS allowlisted → swap succeeds.
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: attacker,
    tokenIn: token1,
    zeroForOne: false,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: type(uint128).max,
    deadline: block.timestamp,
    extensionData: ""
}));
// Attacker successfully swaps despite being explicitly excluded from the allowlist.

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
