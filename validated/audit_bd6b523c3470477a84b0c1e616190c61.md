### Title
Unknown-Op Cost Multiplication Uses `u32::MAX` Cap on a `u64` Cost Type, Enabling Consensus Divergence via Undercharged Execution — (File: `src/more_ops.rs`)

---

### Summary

In `op_unknown`, the final cost of an unknown opcode is computed as `cost *= cost_multiplier + 1` (both `u64`), then checked against `u32::MAX` — a 32-bit limit applied to a 64-bit cost type. This is the direct structural analog of the reported `strconv.ParseInt(…, 32)` bug: a narrower integer width is used to gate a value that legitimately belongs to a wider type. Programs whose unknown-op cost exceeds `u32::MAX` but remains within the block's `max_cost` (`u64`) are rejected as invalid, causing consensus divergence. Additionally, in Rust release mode the multiplication silently wraps on `u64` overflow, allowing the wrapped result to pass the `u32::MAX` check and return an undercharged cost.

---

```

`Cost` is `u64` throughout the codebase. The block cost limit (`max_cost`) is also `u64` (Chia's limit is `11_000_000_000`). The `u32::MAX` threshold (`4_294_967_295`) is less than half of `max_cost`, so any unknown op whose `cost × (cost_multiplier + 1)` lands in the range `(u32::MAX, max_cost]` is rejected as `EvalErr::Invalid` even though it is within the block budget.

**Overflow path (undercharged execution):** In Rust release mode, `u64` multiplication wraps on overflow. If `cost × (cost_multiplier + 1) ≡ small_value (mod 2⁶⁴)` where `small_value ≤ u32::MAX`, the wrapped result passes the check and is returned as the final cost — far below the true cost.

---

### Impact Explanation

**Consensus divergence (primary):** A CLVM program using an unknown opcode with `cost_function = 2` (mul-like) and two ~512 KB atom arguments produces a base cost of approximately `2_153_775_081`. With `cost_multiplier = 1` (opcode bytes `[0x01, 0x80]`), the final cost is `≈ 4_307_550_162 > u32::MAX` → rejected as `Invalid`. Yet `4_307_550_162 ≤ max_cost = 11_000_000_000`, so the program is within the block budget and should succeed. Nodes running this code reject it; nodes with a corrected limit accept it — a chain split.

**Undercharged execution (secondary):** With `cost_multiplier = 0x7fffffff` (opcode `[0x7f, 0xff, 0xff, 0xff, X]`, not reserved) and base cost exactly `8_589_934_592` (a multiple of `2³³`): `8_589_934_592 × 2³¹ = 2⁶⁴ ≡ 0 (mod 2⁶⁴)`. The wrapped cost is `0`, which passes `if cost > u32::MAX` and returns `Reduction(0, nil)` — zero cost for an operation whose true cost is `8.5 billion`.

---

### Likelihood Explanation

**Consensus divergence:** Medium. Two ~512 KB atoms in a CLVM program is feasible within Chia's transaction size limits. The opcode `[0x01, 0x80]` is a valid non-reserved unknown op. An attacker or a legitimate future protocol extension could trigger this path.

**Undercharged execution via overflow:** Low. Requires a base cost of exactly `8_589_934_592`, which for `cost_function = 1` demands ~2.86 GB of argument data — impractical for a mempool transaction. For `cost_function = 2`, the quadratic cost growth makes it more feasible (~512 KB per atom), but the exact cost alignment is still difficult to achieve without precise control.

---

### Recommendation

Replace the `u32::MAX` cap with a check against `max_cost` (the `u64` block limit), and guard the multiplication against overflow:

```rust
// Before multiplication, check that it won't overflow or exceed max_cost
let Some(final_cost) = cost.checked_mul(cost_multiplier + 1) else {
    return Err(EvalErr::Invalid(o))?;
};
check_cost(final_cost, max_cost)?;
Ok(Reduction(final_cost, allocator.nil()))
```

This removes the 32-bit cap, prevents silent `u64` wrap-around, and aligns unknown-op cost enforcement with the rest of the cost model.

fn u32_from_u8_impl(buf: &[u8], signed: bool) -> Option<u32> {
    if buf.is_empty() {
        return Some(0);
    }

    // too many bytes for u32
    if buf.len() > 4 {
        return None;
    }

    let sign_extend = (buf[0] & 0x80) != 0;
    let mut ret: u32 = if signed && sign_extend { 0xffffffff } else { 0 };
    for b in buf {
        ret <<= 8;
        ret |= *b as u32;
    }
    Some(ret)
}

pub fn u32_from_u8(buf: &[u8]) -> Option<u32> {
    u32_from_u8_impl(buf, false)
}
