### Title
`PauseManager` Silently Disables All SPV Proof Verification With No Bypass — (`contract/src/lib.rs`)

---

### Summary

The `PauseManager` role is documented as controlling the ability to pause and unpause contract features. However, when it pauses the contract, it also completely disables `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` for **every caller with no bypass path** — including `DAO`. This is an undocumented side effect that goes beyond the role's stated purpose and mirrors the external report's class: a privileged role silently acquiring power over a second, unrelated system component.

---

### Finding Description

The contract defines three pause-gated public functions with different bypass configurations:

**`submit_blocks`** — carries both `#[pause]` and `#[trusted_relayer]`. The `trusted_relayer` macro injects `bypass_roles(Role::DAO, Role::UnrestrictedSubmitBlocks)`, so those accounts can still submit headers even when paused. [1](#0-0) [2](#0-1) 

**`run_mainchain_gc`** — uses `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, giving `UnrestrictedRunGC` holders a bypass. [3](#0-2) 

**`verify_transaction_inclusion` and `verify_transaction_inclusion_v2`** — both carry a bare `#[pause]` with **no `except` clause and no bypass role of any kind**. [4](#0-3) [5](#0-4) 

When `PauseManager` pauses the contract, the `near-plugins` `Pausable` macro causes every call to these two functions to panic unconditionally. No role — not `DAO`, not `UnrestrictedSubmitBlocks`, not any other — can bypass this. The SPV verification surface is entirely dark.

The `Role` enum documents `PauseManager` only as: *"May pause and unpause features."* [6](#0-5) 

There is no documentation that pausing also kills the verification API used by every downstream consumer.

---

### Impact Explanation

`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` are the contract's primary consumer-facing outputs. Bridges, atomic-swap contracts, and cross-chain lending protocols call these functions to confirm that a Bitcoin transaction is included in a canonical block before releasing funds or updating state. [7](#0-6) [8](#0-7) 

When `PauseManager` pauses the contract:
- Every cross-chain protocol that calls `verify_transaction_inclusion` receives a panic and cannot proceed.
- Funds locked pending a Bitcoin proof are frozen indefinitely.
- The canonical chain state continues to be updated (trusted relayers and `DAO` can still call `submit_blocks`), but no consumer can read a proof result from it.

This is the direct analog of the external report's *Example 2*: FreezerAddress calling `disableEscapeHatch()` and rendering `RescuerAddress` useless. Here, `PauseManager` calling `pause()` renders the entire SPV verification layer useless, with no escape path for any caller.

---

### Likelihood Explanation

`PauseManager` is an operational role expected to be exercised during incidents. Any account holding that role — whether acting in good faith during an emergency or acting adversarially — triggers this side effect automatically. No additional capability, key compromise, or social engineering is required beyond the role itself. The role is explicitly designed to be used, making accidental triggering of this side effect realistic.

---

### Recommendation

1. Add an explicit bypass role to both verification functions, mirroring the pattern already used for `run_mainchain_gc`:

```rust
#[pause(except(roles(Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion(...) -> bool { ... }

#[pause(except(roles(Role::UnrestrictedVerify)))]
pub fn verify_transaction_inclusion_v2(...) -> bool { ... }
```

2. Define and document `Role::UnrestrictedVerify` in the `Role` enum alongside the other bypass roles.

3. If the intent is that verification must also halt during a pause (e.g., because the canonical chain state is suspect), document this explicitly and emit an on-chain event when the pause is activated so that downstream consumers can detect and react to the state change.

---

### Proof of Concept

1. `PauseManager` account calls the `near-plugins`-generated `pa_pause(feature: "ALL")` (or the feature name covering `verify_transaction_inclusion`).
2. Any NEAR account — including `DAO` — calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2`.
3. The `Pausable` macro fires `env::panic_str("Pausable: method is paused")` before any contract logic executes.
4. Every consumer contract that cross-calls either function receives a failed promise; no SPV proof result is ever returned.
5. `submit_blocks` continues to succeed for `DAO` and trusted relayers (bypass roles are present), so the chain tip advances — but no consumer can verify anything against it.

The asymmetry between the bypass-equipped write path (`submit_blocks`) and the bypass-free read path (`verify_transaction_inclusion`) is the concrete broken invariant: the contract can ingest new Bitcoin headers but cannot serve proof results to any caller. [2](#0-1) [4](#0-3) [5](#0-4)

    /// Verifies that a transaction is included in a block at a given block height,
    /// with an additional coinbase merkle proof validation.
    /// This is needed to mitigate the 64-byte transaction Merkle proof forgery vulnerability:
    /// https://www.bitmex.com/blog/64-Byte-Transactions
    ///
    /// @param tx_id transaction identifier
    /// @param tx_block_blockhash block hash at which transaction is supposedly included
    /// @param tx_index index of transaction in the block's tx merkle tree
    /// @param merkle_proof merkle tree path (concatenated LE sha256 hashes) (does not contain initial transaction_hash and merkle_root)
    /// @param coinbase_tx_id coinbase transaction hash
    /// @param coinbase_merkle_proof merkle proof for the coinbase transaction (must have the same length as merkle_proof)
    /// @param confirmations how many confirmed blocks we want to have before the transaction is valid
    /// @return True if tx_id is at the claimed position in the block at the given blockhash, False otherwise
    ///
    /// # Panics
    /// - If `merkle_proof` and `coinbase_merkle_proof` have different lengths
    /// - If `tx_block_blockhash` is not found in the headers pool
    /// - If coinbase merkle proof does not match the block's merkle root
    /// - If the required number of confirmations exceeds the number of stored blocks
    /// - If the block does not belong to the current main chain
    /// - If there are not enough confirmed blocks
    #[pause]
    pub fn verify_transaction_inclusion_v2(&self, #[serializer(borsh)] args: ProofArgsV2) -> bool {
        require!(
            args.merkle_proof.len() == args.coinbase_merkle_proof.len(),
            "Coinbase merkle proof and transaction merkle proof should have the same length"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(
            merkle_tools::compute_root_from_merkle_proof(
                args.coinbase_tx_id.clone(),
                0usize,
                &args.coinbase_merkle_proof,
            ) == header.block_header.merkle_root,
            "Incorrect coinbase merkle proof"
        );

        #[allow(deprecated)]
        self.verify_transaction_inclusion(args.into())
    }
