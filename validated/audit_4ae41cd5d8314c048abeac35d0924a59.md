### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1Handler Conversion Bypasses the Only Blockifier Fee Guard, Allowing a Byzantine Proposer to Execute Fabricated L1→L2 Transactions - (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

When a `ConsensusTransaction::L1Handler` is converted to an executable transaction during proposal validation, the `paid_fee_on_l1` field is unconditionally hardcoded to `Fee(1)`. The protobuf `L1HandlerV0` wire message carries no `paid_fee_on_l1` field at all, so the actual fee paid on L1 is structurally absent from the consensus path. The blockifier's sole guard against a zero-fee L1Handler is `if paid_fee == Fee(0)`, which always passes with `Fee(1)`. A Byzantine proposer can therefore inject arbitrary L1Handler transactions that were never sent on L1, causing every validating peer to execute them and commit wrong state.

### Finding Description

**Root cause — wrong parameter in conversion:**

`convert_consensus_l1_handler_to_internal_l1_handler` in `crates/apollo_transaction_converter/src/transaction_converter.rs` always passes `Fee(1)` as `paid_fee_on_l1`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

**Wire format — `paid_fee_on_l1` is absent from the protobuf message:**

The protobuf `L1HandlerV0` → `L1HandlerTransaction` converter reconstructs only `version`, `nonce`, `contract_address`, `entry_point_selector`, and `calldata`. There is no `paid_fee_on_l1` field in the message, so the actual fee paid on L1 is structurally unrecoverable at the receiving validator:

```rust
impl TryFrom<protobuf::L1HandlerV0> for L1HandlerTransaction {
    fn try_from(value: protobuf::L1HandlerV0) -> Result<Self, Self::Error> {
        ...
        Ok(Self { version, nonce, contract_address, entry_point_selector, calldata })
    }
}
``` [2](#0-1) 

**Blockifier guard — only rejects `Fee(0)`:**

The blockifier's only enforcement is:

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...));
}
``` [3](#0-2) 

`Fee(1)` is never `Fee(0)`, so this check is permanently bypassed for every L1Handler arriving through the consensus path.

**Conversion call site — consensus proposal validation:**

During proposal validation, every `ConsensusTransaction::L1Handler` received from a peer is routed through this path:

```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
``` [4](#0-3) 

The resulting `InternalConsensusTransaction::L1Handler` is then forwarded directly to the batcher for execution with no further L1-origin check. [5](#0-4) 

### Impact Explanation

A Byzantine proposer can craft a `ConsensusTransaction::L1Handler` for a contract call that was never submitted on L1 (e.g., a token mint, a bridge deposit credit, or any `l1_handler` entry point). Every honest validator will:

1. Deserialize the protobuf `L1HandlerV0` (no fee field present).
2. Call `convert_consensus_l1_handler_to_internal_l1_handler`, injecting `paid_fee_on_l1 = Fee(1)`.
3. Pass the blockifier's `Fee(0)` guard.
4. Execute the fabricated L1Handler, committing the resulting state diff.

This produces **wrong state, wrong events, and wrong L1 message receipts** for an L1→L2 message that never existed on L1 — matching the Critical impact scope: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

### Likelihood Explanation

The trigger is a Byzantine proposer — any validator whose turn it is to propose a block. No special privilege beyond normal validator participation is required. The fabricated transaction passes all existing checks without any modification to the honest validator code.

### Recommendation

1. **Add `paid_fee_on_l1` to the `L1HandlerV0` protobuf message** so the actual fee is transmitted and can be verified by validators.
2. **Remove the hardcoded `Fee(1)`** in `convert_consensus_l1_handler_to_internal_l1_handler` and use the fee from the wire message.
3. **Strengthen the blockifier guard** to verify that `paid_fee_on_l1` meets the actual execution cost, not merely that it is non-zero.
4. **Add a cross-reference check** in the proposal-validation path: each L1Handler transaction in a proposal should be verified against the node's own L1 scraper state before being forwarded to the batcher.

        let contract_address = value.address.ok_or(missing("L1HandlerV0::address"))?.try_into()?;

        Ok(Self { version, nonce, contract_address, entry_point_selector, calldata })
    }
