Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` Silently Bypasses Velocity Check When `lastMidPriceX64` Is Uninitialized — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` gates the entire velocity check behind `if (prevMid != 0)`, but `lastMidPriceX64` is zero-initialized for every new pool and there is no `initialize` hook to seed it. On the first swap of any pool using this extension, the guard is silently skipped and the current oracle mid-price — regardless of magnitude — is committed as the new baseline, constituting unconstrained bad-price execution reaching a live pool swap with direct LP principal loss.

## Finding Description
`priceVelocityState` is a `mapping(address pool => PriceVelocityState)` whose fields default to zero. The only seeding mechanism is `setLastMidPrice`, an admin-only function with no enforcement that it is called before the first swap.

In `beforeSwap` (lines 53–76):
```solidity
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // 0 on first swap
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;          // written unconditionally
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                    // entire check skipped
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity enforcement
    }
}
``` [1](#0-0) 

The state write at lines 57–58 is unconditional, so the extreme oracle price is committed as the new baseline before the guard even has a chance to run. The `initialize` hook in `BaseMetricExtension` is a no-op that `PriceVelocityGuardExtension` does not override, so there is no factory-triggered seeding path. [2](#0-1) 

Additionally, the override drops the `onlyPool` modifier present on the base class declaration, meaning any address can call `beforeSwap` directly on the extension contract (though this only corrupts state keyed to `msg.sender`, not to a real pool). [3](#0-2) [4](#0-3) 

## Impact Explanation
The velocity guard is the protocol's primary on-chain defense against rapid oracle price movement reaching live swaps. When `prevMid == 0`, the guard fails open: the first swap executes at whatever oracle price the price provider returns, with no velocity constraint applied. If the oracle price has moved significantly since pool deployment (genuine market move, brief oracle spike, or a Pyth update with a large confidence interval that passes staleness checks), LPs are exposed to a swap at a price the guard was explicitly configured to block. After this first swap, the extreme price is committed as the new baseline, potentially allowing a second swap that is within the velocity envelope relative to the manipulated baseline but still far from the true market price. This is bad-price execution reaching a live pool swap with direct LP principal loss, matching the allowed impact gate.

## Likelihood Explanation
This is the default state for every pool that deploys `PriceVelocityGuardExtension`. The extension has no `initialize` hook, so the zero state is the starting condition for every pool. The pool admin must call `setLastMidPrice` before the first swap, but there is no on-chain enforcement of this ordering. The first public swapper — any unprivileged user — triggers the bypass. No special permissions, oracle manipulation, or adversarial setup is required; the bypass is structural and reachable on every newly configured pool. [5](#0-4) 

## Recommendation
Fail closed when no prior observation exists. Either:

1. **Revert if `prevMid == 0`** — require the admin to seed the price before the first swap is permitted:
   ```solidity
   if (prevMid == 0) revert NoPriorObservation();
   ```
2. **Override `initialize`** — seed `lastMidPriceX64` from the pool's live oracle price at pool creation via the factory-triggered `initialize` hook, analogous to requiring a minimum cardinality before a pool is eligible for guarded operations.
3. **Treat `prevMid == 0` as a full block** — block the swap entirely rather than allowing unconstrained execution when no prior observation exists.

## Proof of Concept
1. Deploy a pool with `PriceVelocityGuardExtension` configured (`maxChangePerBlockE18 = 1e16`, i.e., 1% per block). Do **not** call `setLastMidPrice`.
2. Oracle price at pool deployment: `P0 = 1000 USD`. Oracle price at block N: `P1 = 2000 USD` (100% move, far exceeding the 1% cap).
3. Any user calls `pool.swap(...)` at block N. The pool fetches `bid/ask` from the oracle (P1-based). `PriceVelocityGuardExtension.beforeSwap` is called.
4. `prevMid = s.lastMidPriceX64 = 0`. The `if (prevMid != 0)` branch is skipped. The swap executes at P1 with no velocity check.
5. `s.lastMidPriceX64` is now set to `sqrt(bid1 * ask1)` ≈ P1.
6. LPs suffer losses from a swap at 2× the price the guard was configured to block.
7. Contrast: if `setLastMidPrice(pool, P0_encoded)` had been called before step 3, the swap at block N would revert with `PriceVelocityExceeded` because `(P1 - P0)^2 / P0^2 = 1.0 >> maxChange^2 * (1 + blockDiff)`.
