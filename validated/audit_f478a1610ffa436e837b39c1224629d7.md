### Title
Exit Lock Period Unpredictably Extended Due to Block-Time-Dependent `ExitUnlockAt` Without Deadline — (File: x/tieredrewards/keeper/msg_server.go)

---

### Summary

`MsgTriggerExitFromTier` (and `MsgLockTier` / `MsgCommitDelegationToTier` with `trigger_exit_immediately = true`) computes `ExitUnlockAt` from the block time at **execution**, not at submission. No deadline parameter exists. A transaction that sits in the mempool longer than expected executes at a later block time, silently extending the user's exit lock period beyond what they intended — locking their delegated tokens in the module escrow for longer than consented.

---

### Finding Description

In `MsgTriggerExitFromTier`, the exit commitment is anchored to the block time at execution:

```go
// x/tieredrewards/keeper/msg_server.go:366-367
sdkCtx := sdk.UnwrapSDKContext(ctx)
pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)
```

`TriggerExit` sets `ExitTriggeredAt = blockTime` and `ExitUnlockAt = blockTime + tier.ExitDuration`. The identical pattern appears in `createDelegatedPosition` when `trigger_exit_immediately = true`:

```go
// x/tieredrewards/keeper/position.go:52-53, 67-68
sdkCtx := sdk.UnwrapSDKContext(ctx)
blockTime := sdkCtx.BlockTime()
// ...
if triggerExitImmediately {
    pos.TriggerExit(blockTime, tier.ExitDuration)
}
```

Neither `MsgTriggerExitFromTier`, `MsgLockTier`, nor `MsgCommitDelegationToTier` carries a deadline field. The proto definitions confirm this:

```proto
// proto/chainmain/tieredrewards/v1/tx.proto:119-143
message MsgLockTier {
  string owner               = 1;
  uint32 id                  = 2;
  string amount              = 3;
  string validator_address   = 4;
  bool   trigger_exit_immediately = 5;
  // no deadline
}
```

If the transaction is delayed in the mempool by Δ (due to congestion, low gas price, or network issues), it executes at `T + Δ` instead of the user's intended time `T`. The result:

```
ExitUnlockAt = (T + Δ) + tier.ExitDuration
             = T + tier.ExitDuration + Δ
```

The user's delegated tokens remain locked in the position's per-position delegator address (module-controlled escrow) for `tier.ExitDuration + Δ` instead of `tier.ExitDuration`.

---

### Impact Explanation

The exact corrupted value is `ExitUnlockAt` on the `Position` record stored in the `Positions` collection. The user's tokens are held in the position's deterministic delegator address (created by `createPositionDelegatorAccount`) and cannot be withdrawn or transferred until `ExitUnlockAt` elapses. The user has no recourse: `MsgClearPosition` resets the exit state entirely, forcing the user to restart the full `ExitDuration` commitment from scratch — it does not restore the originally intended `ExitUnlockAt`. The user is therefore forced to choose between waiting the extended period or losing all elapsed exit-commitment progress.

For tiers with long `ExitDuration` values (e.g., 1 year or 5 years), even a delay of hours to days is a meaningful, user-visible extension of the lock on real token value.

---

### Likelihood Explanation

Mempool congestion is a normal, non-adversarial condition on any public Cosmos chain during periods of high activity. A user who submits `MsgTriggerExitFromTier` with a low gas price, or during a congested period, will have their transaction delayed by an unpredictable amount. No privileged actor is required; the delay arises from ordinary network conditions. The scenario is directly analogous to the original report's "the transaction can be executed much later for some reason."

---

### Recommendation

Add an optional `deadline` (Unix timestamp) field to `MsgTriggerExitFromTier`, `MsgLockTier`, and `MsgCommitDelegationToTier`. In the message handler, before computing `ExitUnlockAt`, reject the transaction if `sdkCtx.BlockTime().After(deadline)`:

```go
if !msg.Deadline.IsZero() && sdkCtx.BlockTime().After(msg.Deadline) {
    return nil, errorsmod.Wrapf(types.ErrDeadlineExceeded,
        "block time %s is after deadline %s", sdkCtx.BlockTime(), msg.Deadline)
}
```

This mirrors the standard slippage-deadline pattern used in AMMs and insurance protocols and gives users a hard guarantee on the maximum `ExitUnlockAt` they will receive.

---

### Proof of Concept

1. Tier has `ExitDuration = 365 days`.
2. User submits `MsgTriggerExitFromTier` at block time `T = 2025-01-01 00:00:00 UTC`, expecting `ExitUnlockAt = 2026-01-01 00:00:00 UTC`.
3. Network is congested; transaction is included 48 hours later at block time `T + 2d`.
4. `pos.TriggerExit(sdkCtx.BlockTime(), tier.ExitDuration)` sets `ExitUnlockAt = 2026-01-03 00:00:00 UTC`.
5. User's tokens are locked for 367 days instead of 365 days.
6. User calls `MsgClearPosition` to cancel, but this resets `ExitTriggeredAt` and `ExitUnlockAt` to zero — the user must re-trigger and wait the full 365 days again from the new block time.
7. The user has no way to recover the originally intended `ExitUnlockAt`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

func (ms msgServer) ClearPosition(ctx context.Context, msg *types.MsgClearPosition) (*types.MsgClearPositionResponse, error) {
	if err := msg.Validate(); err != nil {
		return nil, err
	}

	pos, err := ms.getPositionState(ctx, msg.PositionId)
	if err != nil {
		return nil, err
	}

	if err := ms.validateClearPosition(ctx, pos, msg.Owner); err != nil {
		return nil, err
	}

	if !pos.HasTriggeredExit() {
		return &types.MsgClearPositionResponse{PositionId: pos.Id}, nil
	}

	pos, _, _, err = ms.claimRewards(ctx, pos)
	if err != nil {
		return nil, err
	}

	sdkCtx := sdk.UnwrapSDKContext(ctx)
	pos.ClearExit(sdkCtx.BlockTime())

	if err := ms.setPosition(ctx, pos.Position, nil); err != nil {
		return nil, err
	}

	if err := sdkCtx.EventManager().EmitTypedEvent(&types.EventExitCleared{
		PositionId: pos.Id,
		TierId:     pos.TierId,
		Owner:      pos.Owner,
	}); err != nil {
		return nil, err
	}

	return &types.MsgClearPositionResponse{PositionId: pos.Id}, nil
}
