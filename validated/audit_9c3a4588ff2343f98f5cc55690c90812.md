### Title
Non-ERC20-Compliant Token `approve()` Breaks `creditDeposit()` for USDT and Similar Tokens - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve()` via the `IIERC20Base` interface, which declares `approve` as returning `bool`. For non-ERC20-compliant tokens like USDT whose `approve()` emits no return value, Solidity 0.8.x's ABI decoder reverts on the missing return data. This permanently breaks the deposit flow for any supported product whose token is USDT or similarly non-compliant, locking deposited tokens in the DDA contract until an owner manually recovers them.

---

### Finding Description

`DirectDepositV1.sol` defines a local interface `IIERC20Base` with:

```solidity
function approve(address spender, uint256 amount) external returns (bool);
``` [1](#0-0) 

Inside `creditDeposit()`, for every supported spot product, the contract calls:

```solidity
token.approve(address(endpoint), balance);
``` [2](#0-1) 

This is a high-level Solidity interface call. In Solidity ≥0.8.0, when the callee returns no data but the interface declares `returns (bool)`, the ABI decoder reverts. USDT's `approve()` on Ethereum mainnet returns no value, so this call reverts unconditionally for any product backed by USDT.

Notably, the same file already implements a correct `safeTransfer` helper using low-level `.call()` to handle the missing-return-value case for `transfer`:

```solidity
function safeTransfer(IIERC20Base self, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.transfer.selector, to, amount)
    );
    require(success && (data.length == 0 || abi.decode(data, (bool))), "Transfer failed");
}
``` [3](#0-2) 

No equivalent `safeApprove` wrapper exists. The `approve` call is left as a raw interface call, creating the same class of bug the `safeTransfer` wrapper was designed to prevent.

`creditDeposit()` is also the target of `ContractOwner.creditDepositV1()`, which is itself `external` with no access control:

```solidity
function creditDepositV1(bytes32 subaccount) external {
    ...
    DirectDepositV1(directDepositV1).creditDeposit();
}
``` [4](#0-3) 

Both entry points are reachable by any unprivileged caller.

The rest of the protocol correctly uses `ERC20Helper.safeTransfer` / `ERC20Helper.safeTransferFrom` (low-level `.call()` wrappers) for all token movements: [5](#0-4) 

The `approve` path in `DirectDepositV1` is the sole gap.

---

### Impact Explanation

Any user who sends USDT (or another non-ERC20-compliant token that is a supported spot product) to their DDA contract will find `creditDeposit()` permanently reverting. The deposited tokens are stuck in the DDA contract. While `ContractOwner.withdrawFromDirectDepositV1()` (`onlyOwner`) can recover them, the deposit flow is completely broken for affected tokens — users cannot credit collateral to their subaccount through the DDA path. [6](#0-5) 

---

### Likelihood Explanation

USDT is one of the most widely used collateral assets in DeFi. If Nado lists USDT as a supported spot product (product ID registered in `SpotEngine`), every DDA-based deposit of USDT will fail. The trigger requires no special privileges — any user calling `creditDeposit()` or `creditDepositV1()` on a DDA holding USDT will reproduce the revert.

---

### Recommendation

Replace the raw `token.approve(...)` call in `creditDeposit()` with a low-level `safeApprove` helper analogous to the existing `safeTransfer`:

```solidity
function safeApprove(IIERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IIERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        "Approve failed"
    );
}
```

Then replace line 92 in `creditDeposit()`:

```solidity
// Before
token.approve(address(endpoint), balance);

// After
safeApprove(token, address(endpoint), balance);
``` [7](#0-6) 

---

### Proof of Concept

1. USDT is listed as a supported spot product in Nado's `SpotEngine`.
2. A user sends USDT to their `DirectDepositV1` DDA contract address.
3. The user (or anyone) calls `DirectDepositV1.creditDeposit()` or `ContractOwner.creditDepositV1(subaccount)`.
4. Inside `creditDeposit()`, the loop reaches the USDT product. `token.balanceOf(address(this))` returns a non-zero balance.
5. `token.approve(address(endpoint), balance)` is called. USDT's `approve()` executes successfully on-chain but returns no data.
6. Solidity 0.8.x's ABI decoder attempts to decode the empty return data as `bool` and reverts.
7. The entire `creditDeposit()` call reverts. The USDT remains stuck in the DDA contract.
8. Every subsequent call to `creditDeposit()` for this DDA reverts identically — the deposit flow is permanently broken for USDT. [7](#0-6)
