### Title
`ValidResourceBounds::AllResources` Silently Collapses to `L1Gas` on Protobuf Round-Trip, Breaking Consensus Deserialization for Valid Transactions - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf deserializer for `ValidResourceBounds` maps any `AllResources` payload whose `l2_gas` and `l1_data_gas` are both zero to `ValidResourceBounds::L1Gas`. A valid `RpcInvokeTransactionV3` carrying `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` is accepted by the gateway, serialized to protobuf by the proposer, and then irrecoverably rejected by every validator because the reverse conversion from `InvokeTransactionV3` back to `RpcInvokeTransactionV3` hard-errors on any variant that is not `AllResources`. The result is a consensus-level griefing path: one cheap, fee-paying transaction permanently breaks the proposer's block for all validators.

### Finding Description

**Step 1 – Gateway accepts the transaction.**

`RpcInvokeTransactionV3` carries `AllResourceBounds` (always three explicit fields). The gateway has no check that prevents `l2_gas = 0` and `l1_data_gas = 0` while `l1_gas > 0`. The transaction passes all stateless and stateful validations and is stored as `InternalRpcTransaction` with a correctly computed hash (using the `AllResources` branch of `get_tip_resource_bounds_hash`, which appends the l1_data_gas packed field). [1](#0-0) 

**Step 2 – Proposer serializes to protobuf.**

`From<ConsensusTransaction> for protobuf::ConsensusTransaction` calls `txn.into()` on the `RpcInvokeTransactionV3`. This goes through `From<RpcInvokeTransactionV3> for InvokeTransactionV3`, which always wraps the bounds as `ValidResourceBounds::AllResources(tx.resource_bounds)`, and then `From<InvokeTransactionV3> for protobuf::InvokeV3`, which emits all three resource-bound fields (l1_gas, l2_gas=0, l1_data_gas=0) into the wire message. [2](#0-1) 

**Step 3 – Validator deserializes: type collapse.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` applies the following decision:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

Because the wire values for l2_gas and l1_data_gas are zero, the deserializer returns `L1Gas` instead of `AllResources`. [3](#0-2) 

**Step 4 – Conversion back to `RpcInvokeTransactionV3` hard-errors.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` calls `snapi_invoke.try_into()`, which invokes `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3`. That conversion explicitly rejects any variant that is not `AllResources`:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => {
        return Err(StarknetApiError::OutOfRange {
            string: "resource_bounds".to_string(),
        });
    }
},
```

The error is mapped to `DEPRECATED_RESOURCE_BOUNDS_ERROR` and propagated up, causing the entire consensus proposal to be rejected. [4](#0-3) [5](#0-4) 

**Broken invariant (the analog to the external dust bug).**

The external bug: a value (dust) is silently included in `positionOpenUSDValueE36` when it should be excluded, corrupting the `netPnL` calculation. The sequencer analog: a type tag (`AllResources`) is silently replaced by a different type tag (`L1Gas`) during protobuf deserialization when the numeric values happen to be zero, corrupting the conversion domain and causing a hard rejection. In both cases, a legitimate value that passes an initial admission check is later treated as something different by a downstream component, with no error at the boundary where the meaning changes.

### Impact Explanation

Any validator that receives a consensus proposal containing an `AllResources` invoke transaction with zero l2_gas and l1_data_gas will fail to deserialize the transaction and reject the entire proposal. Because the proposer's block is valid from the proposer's perspective (the transaction was accepted by the gateway and the hash was computed correctly), the proposer will keep re-proposing the same block, and every validator will keep rejecting it. This stalls consensus for the duration that the transaction remains in the proposer's mempool.

Impact category: **High – Transaction conversion logic binds the wrong type, causing valid proposals to be rejected by all validators.**

### Likelihood Explanation

The trigger requires submitting a V3 invoke with `AllResourceBounds` where `l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`, and `l1_gas` non-zero. This is a structurally valid transaction that passes all gateway checks. No special privilege is required. A single such transaction per proposer rotation is sufficient to stall consensus.

### Recommendation

Fix `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` to preserve the `AllResources` variant whenever the wire message explicitly carries all three resource-bound fields, regardless of whether their numeric values are zero. One approach: add a boolean flag or a sentinel in the protobuf message to distinguish "old L1Gas-only transaction" from "new AllResources transaction with zero L2/data gas". Alternatively, always deserialize to `AllResources` when all three fields are present in the protobuf message, and only fall back to `L1Gas` when `l2_gas` is absent (i.e., `value.l2_gas` is `None`), which already signals the pre-0.13.3 wire format per the existing `TODO` comment. [6](#0-5) 

### Proof of Concept

1. Construct a `RpcInvokeTransactionV3` with:
   ```
   resource_bounds = AllResourceBounds {
       l1_gas:      { max_amount: 1000, max_price_per_unit: <current_l1_gas_price> },
       l2_gas:      { max_amount: 0,    max_price_per_unit: 0 },
       l1_data_gas: { max_amount: 0,    max_price_per_unit: 0 },
   }
   ```
2. Submit via `starknet_addInvokeTransaction`. The gateway accepts it; the hash is computed using the `AllResources` branch (includes the l1_data_gas packed field).
3. The proposer pulls the transaction from the mempool, wraps it in a `ConsensusTransaction`, and serializes to `protobuf::ConsensusTransaction` → `InvokeV3WithProof` → `InvokeV3`. The wire message contains `l2_gas = {max_amount:0, max_price_per_unit:0}` and `l1_data_gas = {max_amount:0, max_price_per_unit:0}`.
4. The validator receives the protobuf. `ValidResourceBounds::try_from` sees `l2_gas.is_zero() && l1_data_gas.is_zero()` → returns `L1Gas`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` hits the `_ => Err(OutOfRange)` arm → `DEPRECATED_RESOURCE_BOUNDS_ERROR` is returned.
6. The validator rejects the proposal. The proposer re-proposes; all validators reject again. Consensus stalls.

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }
}
