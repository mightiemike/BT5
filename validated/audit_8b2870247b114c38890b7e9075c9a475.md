### Title
`op_bls_pairing_identity` and `op_bls_verify` Return Verification Success With Zero Pairs, Performing No Actual Cryptographic Check — (File: `src/bls_ops.rs`)

---

### Summary

Both `op_bls_pairing_identity` and `op_bls_verify` in `src/bls_ops.rs` lack a guard requiring at least one cryptographic pair before claiming verification success. When invoked with zero (G1, G2) or (pubkey, msg) pairs respectively, both operators call their underlying BLS library functions with an empty list and — because the BLS aggregate of zero elements is the Gt identity — receive `true` back, then unconditionally return `Ok(Reduction(cost, a.nil()))`. No pairing or signature check is actually performed. This is the direct analog of H-02: a function returns "success" without verifying that the intended cryptographic effect occurred.

---

### Finding Description

**`op_bls_pairing_identity`** (lines 332–358):

```rust
let mut items = Vec::<(G1Element, G2Element)>::new();
let mut args = input;
while !nilp(a, args) {          // ← skipped entirely when args == nil
    ...
    items.push((g1, g2));
}
if !aggregate_pairing(items) { // ← called with empty Vec
    Err(EvalErr::BLSPairingIdentityFailed(input))?
} else {
    Ok(Reduction(cost, a.nil()))  // ← always reached when items is empty
}
```

When `input` is the nil terminator (no arguments), the `while` loop body never executes, `items` stays empty, and `aggregate_pairing([])` is called. In BLS12-381 the product of zero pairings is the Gt identity element, which is exactly what the identity check tests for, so the function returns `true`. The operator returns `Ok(Reduction(cost, a.nil()))` — success — without computing a single pairing.

**`op_bls_verify`** (lines 364–401):

```rust
let signature = a.g2(first(a, args)?)?;   // consumes first arg
args = rest(a, args)?;
let mut items = Vec::<(PublicKey, Atom)>::new();
while !nilp(a, args) {                     // ← skipped when no pairs follow
    ...
    items.push((pk, msg));
}
if !aggregate_verify(&signature, items) {  // ← called with empty Vec
    Err(EvalErr::BLSVerifyFailed(input))?
} else {
    Ok(Reduction(cost, a.nil()))           // ← reached when items is empty
}
```

When the attacker supplies only the G2 identity point and no (pubkey, msg) pairs, `items` is empty. `aggregate_verify(&identity_sig, [])` returns `true` by the same BLS convention (the aggregate of zero signatures is the G2 identity, which matches). The operator returns success without verifying any signature.

Neither function contains a guard of the form:
```rust
if items.is_empty() {
    return Err(EvalErr::InvalidOpArg(input, "...requires at least one pair".into()));
}
```

---

### Impact Explanation

Any CLVM puzzle that delegates signature enforcement to `bls_pairing_identity` or `bls_verify` and allows the solution to supply the argument list can be bypassed. An attacker crafts a solution that feeds an empty pair list (or, for `bls_verify`, the G2 identity point followed by nil). The operator returns nil (truthy success in CLVM), the puzzle's condition branch is satisfied, and the coin is spent without a valid signature ever being checked. Because CLVM programs are consensus-evaluated on every full node, a bypass here is a consensus-level coin-theft primitive, not merely a local misconfiguration.

---

### Likelihood Explanation

The entry path is fully attacker-controlled: the attacker supplies arbitrary CLVM bytes as the coin solution. Any puzzle that constructs the `bls_pairing_identity` or `bls_verify` argument list from solution data — a common pattern for flexible multi-sig or threshold puzzles — is vulnerable. The G2 identity point is a well-known, publicly documented constant. No privileged access, social engineering, or dependency compromise is required.

---

### Recommendation

Add an explicit non-empty check in both operators before dispatching to the BLS library:

```rust
// in op_bls_pairing_identity, before calling aggregate_pairing:
if items.is_empty() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_pairing_identity requires at least one (G1, G2) pair".to_string(),
    ));
}

// in op_bls_verify, before calling aggregate_verify:
if items.is_empty() {
    return Err(EvalErr::InvalidOpArg(
        input,
        "bls_verify requires at least one (pubkey, message) pair".to_string(),
    ));
}
```

This mirrors the recommended fix in H-02: require evidence that the intended effect (a real pairing / a real signature check) was actually requested before returning success.

### Proof of Concept

**`bls_pairing_identity` zero-pair bypass:**

```clvm
; CLVM program — always succeeds, no pairing computed
(bls_pairing_identity)
```

Attacker-controlled CLVM bytes encoding `(58)` (opcode 58 = `bls_pairing_identity`) with a nil argument list. `op_bls_pairing_identity` is called, `items` is empty, `aggregate_pairing([])` returns `true`, operator returns nil.

**`bls_verify` zero-pair bypass:**

```clvm
; supply the G2 identity point (48 zero bytes with compressed-infinity flag)
; followed by nil — no pubkey/msg pairs
(bls_verify <G2_identity_bytes>)
```

`op_bls_verify` parses the identity G2 point as the signature, finds no subsequent pairs, calls `aggregate_verify(&identity, [])` which returns `true`, and returns nil — verification "passed" with no key or message ever checked.
