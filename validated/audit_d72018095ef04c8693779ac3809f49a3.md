### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables USDC-E Theft from Direct Deposit Accounts ŌĆö (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc()` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the returned `bool`. If the USDC token on Ink Chain returns `false` on a failed transfer instead of reverting, the function silently proceeds to withdraw USDC-E from the DDA and deliver it to the caller ŌĆö with no USDC ever received in exchange.

---

### Finding Description

`replaceUsdcEWithUsdc` is a permissionless migration helper (no access-control modifier) that is supposed to atomically swap USDC-E held in a Direct Deposit Account (DDA) for USDC supplied by the caller:

```solidity
// ContractOwner.sol lines 608-620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // ŌåÉ return value ignored
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));       // USDC-E ŌåÆ ContractOwner
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);                 // USDC-E ŌåÆ caller
    }
}
``` [1](#0-0) 

The raw `transferFrom` is used here instead of the protocol's own `ERC20Helper.safeTransferFrom`, which explicitly checks both the low-level call success and the decoded `bool` return value:

```solidity
// ERC20Helper.sol lines 23-42
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
}
``` [2](#0-1) 

The `IERC20Base` interface is treated throughout the codebase as returning `bool` from `transferFrom`. Ignoring that return value is the exact same class of bug as the Allora `ResetChurnableTopics` issue: a critical operation's failure signal is discarded, and execution continues as if it succeeded.

---

### Impact Explanation

If `IERC20Base(usdc).transferFrom` returns `false` without reverting (a valid ERC-20 behaviour for tokens such as USDT-style implementations):

1. No USDC is transferred from the attacker to the DDA.
2. `DirectDepositV1.withdraw(usdcE)` still executes ŌĆö it transfers the full USDC-E balance from the DDA to `ContractOwner`.
3. `safeTransfer(msg.sender, balance)` still executes ŌĆö it transfers that USDC-E from `ContractOwner` to the attacker.

The attacker drains USDC-E from any DDA on Ink Chain (chain 57073) without supplying any USDC. The victim subaccount loses its USDC-E collateral with no compensation. Because the function is permissionless and accepts any `subaccount` argument, every DDA holding USDC-E is at risk.

---

### Likelihood Explanation

The function is restricted to chain 57073 (Ink Chain). The USDC token address hardcoded is `0x2D270e6886d130D724215A266106e6832161EAEd`. If that deployment follows a non-reverting ERC-20 pattern (returning `false` on failure), the vulnerability is directly exploitable by any external caller with zero USDC balance or zero allowance granted to `ContractOwner`. The function carries no access control, so no privilege is required.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's own `safeTransferFrom` wrapper, consistent with every other token transfer in the codebase:

```solidity
// Before (vulnerable)
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe)
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

This ensures that a `false` return value or a low-level call failure causes an immediate revert before any USDC-E is withdrawn from the DDA.

---

### Proof of Concept

1. A DDA for `subaccount` holds 10,000 USDC-E.
2. Attacker (with 0 USDC or 0 allowance to `ContractOwner`) calls `replaceUsdcEWithUsdc(subaccount)` on Ink Chain.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, 10000)` returns `false` ŌĆö no USDC moves.
4. `DirectDepositV1(directDepositV1).withdraw(usdcE)` executes ŌĆö 10,000 USDC-E moves from DDA ŌåÆ `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 10000)` executes ŌĆö 10,000 USDC-E moves from `ContractOwner` ŌåÆ attacker.
6. Attacker holds 10,000 USDC-E; the DDA and its subaccount hold nothing. [1](#0-0)

    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
