### Title
Linked Signer Can Overwrite Its Own Authorization — (`File: core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This allows an existing linked signer to sign and submit a `LinkSigner` transaction through the sequencer, replacing the subaccount's linked signer with any address it chooses — an account-management operation that should be restricted exclusively to the primary account owner.

---

### Finding Description

The `processTransactionImpl` function handles the sequencer-submitted `LinkSigner` transaction as follows: [1](#0-0) 

`validateSignedTx` is called with `allowLinkedSigner = true`. This flag is forwarded to `validateSignature` in `Verifier.sol`: [2](#0-1) 

When `allowLinkedSigner = true`, the verifier accepts a signature from either the primary account owner **or** the currently registered linked signer. Because `LinkSigner` itself is passed `true`, the linked signer's signature is sufficient to overwrite `linkedSigners[sender]` with any new address.

The slow-mode path for the same transaction type correctly restricts this operation to the primary account owner by using `validateSender(txn.sender, sender)`, which checks `msg.sender == address(uint160(bytes20(txn.sender)))`: [3](#0-2) 

The fast (sequencer) path lacks this equivalent restriction, creating an asymmetry: the slow-mode path enforces owner-only access, but the fast path does not.

---

### Impact Explanation

A linked signer can sign a `LinkSigner` transaction naming any address as the new linked signer. Once submitted through the sequencer, `linkedSigners[subaccount]` is overwritten without the primary owner's consent. The owner's only recourse is the slow-mode `LinkSigner`, which carries a hardcoded 3-day delay: [4](#0-3) 

During those 3 days, the attacker-controlled linked signer retains full authority to:
- Submit orders on the subaccount (trading losses)
- Sign `WithdrawCollateral` transactions (funds are sent to the owner's address, but the attacker can drain the account's health via bad trades)
- Sign `MintNlp` / `BurnNlp` operations
- Re-issue another `LinkSigner` to rotate the linked signer again before the owner's slow-mode revocation executes, creating an indefinite persistence loop

The broken invariant: **only the primary account owner should be able to mutate the linked signer**. A linked signer is a delegated, lower-privilege role intended for automated trading, not account management.

---

### Likelihood Explanation

Any subaccount that has registered a linked signer is exposed. Linked signers are commonly used for programmatic/bot trading. A compromised API key, a malicious third-party trading service, or a rogue bot can trigger this path. The attacker needs only to craft and submit a valid `LinkSigner` EIP-712 message signed by the current linked signer key — a straightforward operation requiring no special on-chain state.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false  // only the primary owner may change the linked signer
);
```

This mirrors the restriction already enforced in the slow-mode path and ensures that linked signer mutation is always an owner-only operation regardless of submission path.

---

### Proof of Concept

1. Alice registers Bob's address as her linked signer via a legitimate `LinkSigner` slow-mode transaction.
2. Bob (the linked signer) constructs a `SignedLinkSigner` struct with `sender = Alice's subaccount`, `signer = Carol` (an address Bob controls), and a valid nonce.
3. Bob signs the EIP-712 digest for this `LinkSigner` transaction with his own key.
4. Bob (or the sequencer on Bob's behalf) submits this transaction via `submitTransactions`.
5. `processTransactionImpl` reaches the `LinkSigner` branch, calls `validateSignedTx(..., true)`, which accepts Bob's signature because `recovered == linkedSigner`.
6. `linkedSigners[Alice's subaccount]` is now set to Carol.
7. Alice submits a slow-mode `LinkSigner` to revoke, but it is delayed 3 days.
8. Before the slow-mode executes, Carol signs another `LinkSigner` rotating to a fresh address, resetting the 3-day clock indefinitely.
9. Bob/Carol retain persistent linked-signer authority over Alice's subaccount. [1](#0-0) [5](#0-4) [2](#0-1)

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
