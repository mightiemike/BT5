### Title
Linked Signer Can Overwrite Its Own Delegation — Unauthorized Subaccount Signer Mutation - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the currently active linked signer — a delegated, lower-privileged role — can sign a `LinkSigner` transaction to replace the linked signer for any subaccount it controls with an arbitrary address. Only the subaccount owner (the primary key) should be permitted to update this delegation.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch reads:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:], (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

`validateSignedTx(..., true)` calls `validateSignature(sender, digest, sig, true)`, which in turn calls `verifier.validateSignature(sender, getLinkedSigner(sender), digest, sig)`. [2](#0-1) 

Inside `Verifier.validateSignature`, the check is:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

This means the signature is accepted from **either** the subaccount owner **or** the currently registered linked signer. Because the `LinkSigner` transaction writes directly to `linkedSigners[signedTx.tx.sender]`, the linked signer can overwrite the delegation mapping for the subaccount it was granted access to — pointing it to any address, including an attacker-controlled one. [4](#0-3) 

The `linkedSigners` mapping is the sole source of truth consulted by `getLinkedSigner`, which is used to authorize all subsequent signed transactions (withdrawals, liquidations, transfers, NLP operations). [5](#0-4) 

---

### Impact Explanation

A compromised or malicious linked signer can:

1. Replace `linkedSigners[victim_subaccount]` with an attacker-controlled address.
2. The attacker's address is then accepted as a valid signer for `WithdrawCollateral`, `WithdrawCollateralV2`, `TransferQuote`, `LiquidateSubaccount`, `MintNlp`, `BurnNlp`, and future `LinkSigner` transactions — all of which accept linked signers. [6](#0-5) 

The subaccount owner loses control of their account's signing delegation without any on-chain action on their part. All collateral in the subaccount is at risk of unauthorized withdrawal or transfer.

---

### Likelihood Explanation

Linked signers are explicitly designed for automated trading (hot wallets, API keys). Any user who has set a linked signer is exposed. A linked signer key compromise — a realistic operational event — immediately escalates to full account takeover rather than being limited to the delegated trading scope. The attack requires only a valid EIP-712 signature from the linked signer key, which is a normal sequencer-submitted transaction.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may update the linked signer
);
``` [7](#0-6) 

Additionally, consider adopting a two-step delegation pattern (propose + confirm) for `LinkSigner` to prevent a single compromised key from silently hijacking signing authority, analogous to the two-step ownership transfer recommended in the reference report.

---

### Proof of Concept

1. Alice owns `alice_subaccount` and sets Bob (`bob_key`) as her linked signer via a legitimate `LinkSigner` transaction.
2. Bob's key is compromised. The attacker crafts:
   ```
   SignedLinkSigner {
     tx: LinkSigner {
       sender: alice_subaccount,
       signer: bytes32(attacker_address),
       nonce: current_nonce
     },
     signature: sign(bob_key, EIP712_digest)
   }
   ```
3. The sequencer submits this as a `processTransactionImpl` call. `validateSignedTx` accepts Bob's signature because `allowLinkedSigner = true` and `getLinkedSigner(alice_subaccount) == bob_key`.
4. `linkedSigners[alice_subaccount]` is overwritten with `attacker_address`.
5. The attacker now signs `WithdrawCollateral` transactions on behalf of `alice_subaccount`, draining all collateral. [1](#0-0) [8](#0-7)
