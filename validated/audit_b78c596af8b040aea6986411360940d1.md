### Title
Unconditional `LastKnownBonded = true` After Redelegation to Unbonded Validator Enables Unearned Bonus Reward Drain — (File: `x/tieredrewards/keeper/msg_server.go`)

---

### Summary

`TierRedelegate` unconditionally sets `LastKnownBonded = true` on the position after redelegation, regardless of the destination validator's actual bonded status. When a user redelegates to an unbonded validator, the position's bonded-state checkpoint is wrong. The next time `processEventsAndClaimBonus` runs, it treats the entire period from redelegation to the validator's next BOND event as a bonded segment and pays out bonus rewards the user was never entitled to, draining the shared rewards pool.

---

### Finding Description

The bonus reward formula is:

```
shares × tokensPerShare × bonusApy × durationSeconds / SecondsPerYear
```

Bonus is only supposed to accrue while the validator is bonded. `processEventsAndClaimBonus` initialises its local `bonded` flag from `pos.LastKnownBonded` and then replays validator events forward from `pos.LastEventSeq`: [1](#0-0) 

For each event, if `bonded == true` at the start of the segment, bonus is computed for that segment. The `BOND` event type sets `bonded = true`; `UNBOND` sets it to `false`.

In `TierRedelegate`, after the staking redelegation is executed, the position's bonus checkpoints are advanced with a hardcoded `true`: [2](#0-1) 

`latestSeq` is the most recent event sequence number on the **destination** validator at the moment of redelegation. `pos.UpdateBonusCheckpoints(latestSeq, blockTime, true)` sets:

- `LastEventSeq = latestSeq` — future event replay starts after this point
- `LastBonusAccrual = blockTime` — segment start for the next claim
- `LastKnownBonded = true` — **unconditionally, regardless of whether the destination validator is actually bonded**

If the destination validator is currently unbonded (e.g., jailed), the UNBOND event that caused it to become unbonded has sequence ≤ `latestSeq` and will never be replayed by this position. The position therefore enters the next claim cycle believing the validator is bonded when it is not.

When the validator is later unjailed and bonds, a BOND event is appended at `latestSeq + 1`. On the next call to `processEventsAndClaimBonus`:

1. `bonded = true` (from the incorrect `LastKnownBonded`)
2. The BOND event is the first event processed
3. Because `bonded == true`, bonus is computed for the segment `[LastBonusAccrual, BOND_event_timestamp]` — the entire period the validator was unbonded
4. `bonded` is then set to `true` again (redundantly) by the BOND event
5. The final live segment also accrues bonus

The user receives bonus for the full unbonded gap, which they should not. [3](#0-2) 

---

### Impact Explanation

The attacker receives bonus rewards from the shared `RewardsPoolName` module account for a time window during which their validator was unbonded. The bonus pool balance is directly reduced by the unearned payout: [4](#0-3) 

The corrupted value is the `RewardsPoolName` module account balance. Repeated exploitation across multiple positions or multiple unbond/rebond cycles compounds the drain. Other legitimate position holders may be unable to claim their earned bonus if the pool is exhausted.

---

### Likelihood Explanation

The entry path is a normal, unprivileged `MsgTierRedelegate` transaction. Cosmos SDK's `BeginRedelegation` does not require the destination validator to be bonded. Jailed validators are a routine occurrence on any live chain; an attacker can monitor for jailed validators, redelegate to one, and wait for unjailing. The ADR confirms the only redelegation block is "after exit commitment elapses" — there is no check on the destination validator's bonded status: [5](#0-4) 

---

### Recommendation

Replace the hardcoded `true` with the destination validator's actual bonded status at the time of redelegation:

```go
dstVal, err := ms.stakingKeeper.GetValidator(ctx, dstValAddr)
if err != nil {
    return nil, err
}
pos.UpdateBonusCheckpoints(latestSeq, sdkCtx.BlockTime(), dstVal.IsBonded())
```

This ensures `LastKnownBonded` accurately reflects whether the destination validator was bonded at the moment the position moved to it, so the event-replay logic in `processEventsAndClaimBonus` starts from the correct bonded state.

---

### Proof of Concept

1. Validator **V1** is bonded. User calls `MsgLockTier` delegating to V1. Position P is created with `LastKnownBonded = true`.
2. Validator **V2** is jailed (unbonded). Its latest event seq is **N** (an UNBOND event at time `T_unbond`).
3. User calls `MsgTierRedelegate` moving P to V2. `claimRewards` settles any pending rewards. Then:
   ```
   pos.UpdateBonusCheckpoints(N, T_redelegate, true)
   ```
   Position P now has `LastEventSeq = N`, `LastBonusAccrual = T_redelegate`, `LastKnownBonded = true`.
4. V2 is unjailed at time `T_bond`. A BOND event is appended at seq **N+1** with `TokensPerShare = R`.
5. User calls `MsgClaimTierRewards` for P. `processEventsAndClaimBonus` runs:
   - `bonded = true` (from `LastKnownBonded`)
   - Processes BOND event at seq N+1: computes bonus for `[T_redelegate, T_bond]` with `bonded = true` and rate `R`
   - Bonus = `shares × R × bonusApy × (T_bond − T_redelegate) / SecondsPerYear`
   - Sends this amount from the rewards pool to the user
6. User receives bonus for the entire jailed period `[T_redelegate, T_bond]`, during which the validator was unbonded and no bonus should have accrued. [2](#0-1) [6](#0-5)

			// Compute bonus for the bonded segment [segmentStart, eventTime]
			// using the snapshot rate at the event.
