### Title
Linked Signer Can Self-Escalate by Signing `LinkSigner` Transaction — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `LinkSigner` fast-path handler in `EndpointTx.sol` calls `validateSignedTx` with `allowLinkedSigner = true`, permitting the currently registered linked signer to sign a new `LinkSigner` transaction. This allows a compromised or malicious linked signer to overwrite `linkedSigners[subaccount]` with an attacker-controlled address, escalating from delegated signing authority to full account control.

---

### Finding Description

In `EndpointTx.sol`, the fast-path `LinkSigner` handler is:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true          // <-- allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
``` [1](#0-0) 

The fifth argument `true` flows into `validateSignature` / `validateCompactSignature` in `Verifier.sol`, which accepts a signature from **either** the subaccount owner address **or** the registered `linkedSigner`:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [2](#0-1) 

This means the existing linked signer can produce a valid signature for a `LinkSigner` transaction, causing `linkedSigners[subaccount]` to be overwritten with any address the linked signer chooses.

The slow-mode `LinkSigner` path does **not** have this problem — it uses `validateSender(txn.sender, sender)`, which enforces that only the actual on-chain subaccount owner can set the linked signer:

```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    ...
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [3](#0-2) 

The missing validation is: no check that the signer of a `LinkSigner` transaction is the subaccount owner and not the current linked signer. This is the direct analog to the Amp `authorizeOperatorByPartition` / `revokeOperatorByPartition` missing `require(_operator != msg.sender)` — a delegated identity (linked signer / operator) can mutate the authorization state it was granted by.

---

### Impact Explanation

A linked signer (e.g., a trading bot or hot wallet) can sign a `LinkSigner` transaction to set `linkedSigners[victim_subaccount] = attacker_address`. The attacker then holds full signing authority over the victim subaccount and can sign any transaction that accepts linked signers — including `WithdrawCollateral`, `TransferQuote`, and order placement — draining all collateral from the subaccount. The corrupted state is `linkedSigners[subaccount]`, and the resulting asset delta is total loss of the subaccount's collateral balance.

---

### Likelihood Explanation

Linked signers are explicitly designed to be lower-trust keys (hot wallets, automated bots). Key compromise of a hot wallet is materially more likely than compromise of a cold primary key. Once the linked signer key is compromised, the attacker can permanently escalate to full account control in a single sequencer-submitted transaction before the owner can react via the slow-mode path.

---

### Recommendation

Change `allowLinkedSigner` to `false` for `LinkSigner` transaction validation in the fast path, so only the subaccount owner can set the linked signer. This aligns the fast path with the slow-mode path behavior and closes the privilege escalation:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only owner may set linked signer
);
``` [4](#0-3) 

---

### Proof of Concept

1. Alice has subaccount `alice_sub` with linked signer `hot_wallet`.
2. Attacker compromises `hot_wallet`'s private key.
3. Attacker constructs `LinkSigner { sender: alice_sub, signer: attacker_address, nonce: N }` and signs it with `hot_wallet`.
4. Attacker submits the transaction to the sequencer.
5. `validateSignedTx` accepts the signature because `allowLinkedSigner = true` and `hot_wallet == getLinkedSigner(alice_sub)`.
6. `linkedSigners[alice_sub]` is overwritten with `attacker_address`.
7. Attacker signs a `WithdrawCollateral` transaction using `attacker_address`'s key.
8. Alice's entire collateral balance is drained.
