### Title
Perp Bad Debt Socialization Reverts on Zero Open Interest, Permanently Blocking Subaccount Finalization — (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` divides the residual negative `vQuoteBalance` by `state.openInterest` without a zero-guard. When a perp market has no remaining open interest at the time of finalization, this division by zero causes a revert, permanently blocking `_finalizeSubaccount` and leaving bad debt unresolved in the protocol.

---

### Finding Description

During subaccount finalization in `ClearinghouseLiq._finalizeSubaccount`, after all perp positions are confirmed closed and insurance is applied, `perpEngine.socializeSubaccount` is called to redistribute any remaining negative `vQuoteBalance` across other market participants. [1](#0-0) 

Inside `PerpEngine.socializeSubaccount`, if the insurance fund is insufficient to cover the full negative `vQuoteBalance`, the remainder is socialized by computing a per-share funding adjustment: [2](#0-1) 

The division `.div(state.openInterest)` at line 166 has no zero-check. If `state.openInterest == 0` at the time of the call, the function reverts with a division-by-zero panic.

**Trigger path:**

1. A subaccount accumulates a deeply negative `vQuoteBalance` in a perp product (e.g., from adverse funding payments or a large unrealized loss).
2. The subaccount's perp position is liquidated via `_handleLiquidationPayment`, setting `balance.amount` to zero but leaving `vQuoteBalance` still negative (the liquidation payment only partially offsets the loss).
3. All other participants in that specific perp market close their positions, driving `state.openInterest` to zero.
4. A liquidator attempts to finalize the insolvent subaccount by calling `liquidateSubaccountImpl` with `txn.productId == type(uint32).max`.
5. `_finalizeSubaccount` passes the `balance.amount == 0` check for all perps, then calls `perpEngine.socializeSubaccount`.
6. Insurance is insufficient → the inner `if (balance.vQuoteBalance < 0)` branch is entered → division by `state.openInterest == 0` → **revert**.
7. Finalization is permanently blocked. The bad debt is never cleared.

The `_finalizeSubaccount` path is the **only** on-chain mechanism that calls `spotEngine.socializeSubaccount` or `perpEngine.socializeSubaccount`. There is no alternative path to clear the residual negative `vQuoteBalance` without admin intervention. [3](#0-2) 

---

### Impact Explanation

The unresolved negative `vQuoteBalance` remains permanently in the perp state. More critically, the insolvent subaccount's negative QUOTE balance (in `SpotEngine`) also cannot be cleared because `spotEngine.socializeSubaccount` is only called after `perpEngine.socializeSubaccount` succeeds: [4](#0-3) 

With the negative QUOTE balance uncleared, `SpotEngine.assertUtilization` — called on every `withdrawCollateral` — will revert with `ERR_MAX_UTILIZATION` once `totalBorrows > totalDeposits`: [5](#0-4) 

This blocks all withdrawals of the affected token for all users, not just the insolvent subaccount. The protocol becomes insolvent with no on-chain recovery path.

---

### Likelihood Explanation

The condition `state.openInterest == 0` with a residual negative `vQuoteBalance` is reachable in realistic market conditions:

- A low-liquidity perp market where the insolvent subaccount is the last active participant (or the last remaining after a market crash).
- A market being wound down where all other traders exit before the insolvent subaccount is finalized.
- A scenario where the counterparty to the insolvent position also closes or is liquidated before finalization is triggered.

The finalization transaction (`productId == type(uint32).max`) is permissionless — any unprivileged liquidator can trigger it — but the revert means no liquidator can ever succeed, making the block permanent. [6](#0-5) 

---

### Recommendation

Add a zero-check for `state.openInterest` before the division in `PerpEngine.socializeSubaccount`. If there is no open interest to socialize to, the remaining loss should either be absorbed by the insurance fund (if any remains), written off against a protocol reserve, or the `vQuoteBalance` should be zeroed out with the loss recorded separately:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No participants to socialize to; zero out the balance
        // and record the unrecoverable loss (e.g., emit an event or
        // deduct from a protocol reserve).
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

Additionally, document the expected behavior when the insurance fund is exhausted and no open interest exists, so the protocol's bad-debt handling policy is explicit in the codebase.

### Proof of Concept

1. Alice opens a 1 ETH long perp position in market X at $2000. Her `vQuoteBalance` accumulates to `-$200` from adverse funding.
2. ETH price drops to $1500. Alice is under maintenance margin. A liquidator calls `liquidateSubaccountImpl` for market X, closing Alice's position (`balance.amount = 0`). The liquidation payment is `$1350` (discounted). Alice's `vQuoteBalance` becomes `-$200 + $1350 - $1500 = -$350` (net loss).
3. Bob (the only other participant, who held the short side) closes his position. `state.openInterest` for market X drops to `0`.
4. A liquidator calls `liquidateSubaccountImpl` with `txn.productId = type(uint32).max` to finalize Alice.
5. `_finalizeSubaccount` confirms `balance.amount == 0` for all perps. It calls `perpEngine.socializeSubaccount(alice, insurance)`.
6. Insurance covers `$100`, leaving `vQuoteBalance = -$250`.
7. `balance.vQuoteBalance < 0` → enters socialization branch → `state.openInterest == 0` → **division by zero → revert**.
8. Alice's subaccount is never finalized. Her `-$250` QUOTE liability remains. `assertUtilization` begins reverting for all QUOTE withdrawals once borrows exceed deposits. [7](#0-6)
