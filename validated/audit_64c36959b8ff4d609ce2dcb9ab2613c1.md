### Title
Accumulated Fast Withdrawal Fees Permanently Locked With No Withdrawal Mechanism — (`core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` collects ERC-20 fee tokens from callers and credits them to the `fees[productId]` accounting mapping, but no function in `BaseWithdrawPool` or its concrete child `WithdrawPool` ever decrements `fees[productId]` or transfers those fee tokens out through a dedicated path. The accumulated fees are permanently locked in the contract with no protocol-level mechanism to retrieve or redistribute them.

---

### Finding Description

Every call to `submitFastWithdrawal()` computes a fee and either deducts it from the withdrawal amount or pulls it from `msg.sender`, then records it:

```solidity
fees[productId] += fee;          // line 111 — only write to fees[]
handleWithdrawTransfer(token, sendTo, transferAmount);
```

The `fees` mapping is declared as:

```solidity
mapping(uint32 => int128) public fees;   // line 40
```

Searching the entire contract (and its only concrete child `WithdrawPool.sol`, which adds nothing) reveals:

- `fees[productId]` is **written** in exactly one place: line 111 of `submitFastWithdrawal`.
- `fees[productId]` is **never read** in any transfer-triggering path.
- No function named `withdrawFees`, `claimFees`, `collectFees`, or similar exists anywhere in the codebase.

The only token-exit function that could theoretically be used is `removeLiquidity`:

```solidity
function removeLiquidity(uint32 productId, uint128 amount, address sendTo)
    external onlyOwner {
    handleWithdrawTransfer(getToken(productId), sendTo, amount);   // line 156
}
```

`removeLiquidity` transfers raw token balances but **never touches `fees[productId]`**, so:
1. The `fees` accounting ledger grows monotonically and is never cleared.
2. If the owner uses `removeLiquidity` to extract tokens, the `fees` mapping permanently overstates the fee balance, making it useless as an accounting primitive.
3. There is no protocol-sanctioned path for fee extraction that keeps the `fees` mapping consistent.

---

### Impact Explanation

Fast withdrawal fees paid by every caller of `submitFastWithdrawal()` accumulate in the `WithdrawPool` contract with no dedicated withdrawal path. The protocol cannot access, redistribute, or account for these fees through any supported contract entrypoint. Over time, as fast withdrawals are processed at scale, the locked fee amount grows proportionally to withdrawal volume. The `fees` mapping — the only on-chain record of how much has been collected — is permanently desynchronized from any actual extraction, breaking fee accounting invariants.

**Severity:** Medium. Real ERC-20 tokens are locked; the accounting state is permanently corrupted once any workaround (e.g., `removeLiquidity`) is used.

---

### Likelihood Explanation

`submitFastWithdrawal()` is a public, permissionless function callable by any user who holds a valid signed withdrawal transaction. Every invocation accumulates fees. The condition is triggered on every fast withdrawal processed by the protocol — this is a routine, high-frequency operation, not an edge case.

---

### Recommendation

Add a dedicated fee withdrawal function to `BaseWithdrawPool` that both transfers the accumulated tokens and resets the accounting:

```solidity
function withdrawFees(uint32 productId, address sendTo) external onlyOwner {
    int128 amount = fees[productId];
    require(amount > 0, "No fees to withdraw");
    fees[productId] = 0;
    handleWithdrawTransfer(getToken(productId), sendTo, uint128(amount));
}
```

This mirrors the pattern used in `removeLiquidity` but correctly clears the `fees` ledger entry, keeping on-chain accounting consistent.

---

### Proof of Concept

1. Alice calls `submitFastWithdrawal(idx, transaction, signatures)` with a valid signed withdrawal for 1,000 USDC.
2. `fastWithdrawalFeeAmount()` returns a fee of, say, 1 USDC.
3. `fees[productId] += 1e6` (line 111) — 1 USDC is now tracked as a fee and held in the contract.
4. Steps 1–3 repeat across thousands of fast withdrawals; `fees[productId]` grows to, say, 10,000 USDC.
5. The protocol operator searches `BaseWithdrawPool` and `WithdrawPool` for a fee withdrawal function — none exists.
6. The only exit is `removeLiquidity`, which transfers tokens but leaves `fees[productId]` unchanged at 10,000 USDC, permanently corrupting the accounting.
7. The 10,000 USDC in fees is effectively inaccessible through any protocol-sanctioned path.

**Root cause:** [1](#0-0)  — `fees` mapping declared but never decremented. [2](#0-1)  — sole write site, no corresponding read-and-transfer path. [3](#0-2)  — `removeLiquidity` is the only token-exit path and does not interact with `fees`. [4](#0-3)  — `WithdrawPool` adds no fee withdrawal logic.
