### Title
`StatelessTransactionValidator::validate_resource_bounds` Skips `max_l2_gas_amount` Admission Check for Declare Transactions - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces a per-transaction `max_l2_gas_amount` ceiling for `Invoke` and `DeployAccount` transactions but explicitly skips this check for `Declare` transactions. Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount` set to an arbitrarily large value (up to `u64::MAX`) and have it admitted through the gateway into the mempool, violating the gateway's own admission invariant.

---

### Finding Description

In `validate_resource_bounds`, the guard reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` branch is an intentional no-op: it matches Declare transactions and does nothing, falling through without executing the `max_l2_gas_amount` comparison. The developer TODO comment confirms the gap is known but unresolved.

For `Invoke` and `DeployAccount` transactions the check fires correctly: if `l2_gas.max_amount > config.max_l2_gas_amount` the transaction is rejected with `MaxGasAmountTooHigh`. For `Declare` transactions the check is never reached, so any value of `l2_gas.max_amount` — including `GasAmount(u64::MAX)` — passes stateless validation and is forwarded to the mempool.

The existing negative test (`test_invalid_max_l2_gas_amount`) only exercises `TransactionType::DeployAccount` and `TransactionType::Invoke`, confirming that the `Declare` path is untested and unguarded.

The analog to the original report is direct: `investors_supply` ↔ `config.max_l2_gas_amount`; `Allocated_Amount` ↔ the per-transaction gas claim; `addInvestor()` ↔ `validate_resource_bounds()`. Just as `addInvestor()` omits the `require(_amount <= investors_supply - Allocated_Amount)` guard for the caller-supplied amount, `validate_resource_bounds()` omits the `max_amount <= max_l2_gas_amount` guard for the Declare-supplied gas amount.

---

### Impact Explanation

A Declare transaction carrying `l2_gas.max_amount = u64::MAX` passes the gateway's stateless validator and enters the mempool. The `max_l2_gas_amount` configuration parameter exists precisely to prevent transactions from claiming more L2 gas than any block can accommodate; bypassing it for Declare transactions breaks the gateway's admission invariant. The transaction is then handed to the batcher and blockifier with an uncapped gas budget for the `__validate_declare__` entry point (before the OS-level `cap_remaining_gas(VALIDATE_MAX_SIERRA_GAS)` applies). This constitutes the gateway accepting a transaction it is configured to reject.

Impact classification: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The trigger requires no privilege. Any user who can submit an RPC transaction can craft a `Declare` transaction with an arbitrarily large `l2_gas.max_amount`. The code path is unconditionally reachable on every Declare submission. The TODO comment confirms the gap is present in the production code path.

---

### Recommendation

Remove the Declare-specific no-op branch and apply the same `max_l2_gas_amount` ceiling uniformly:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> … {
    …
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
            gas_amount: resource_bounds.l2_gas.max_amount,
            max_gas_amount: self.config.max_l2_gas_amount,
        });
    }
    Ok(())
}
```

If Declare transactions legitimately require a higher gas ceiling (e.g., for large Sierra programs), introduce a separate `max_l2_gas_amount_declare` configuration field and enforce it explicitly rather than skipping the check entirely.

---

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with:
   ```
   resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)
   resource_bounds.l2_gas.max_price_per_unit = <any value >= min_gas_price>
   ```
2. Submit it to the gateway's `add_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions regardless of `max_l2_gas_amount`.
4. The transaction enters the mempool with an uncapped L2 gas claim, violating the gateway's configured admission bound.
