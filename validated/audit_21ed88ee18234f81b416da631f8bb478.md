### Title
Case-Insensitive Bech32 Duplicate Bypass in `validateBurnedAddresses` Allows Double-Subtraction and MaxSupply Cap Evasion - (File: `x/inflation/types/params.go`)

---

### Summary

The `validateBurnedAddresses` function in `x/inflation/types/params.go` checks for duplicate burned addresses using a raw-string map key. Because bech32 is case-insensitive, the same address encoded in different cases (e.g., `cosmos1abc...` and `COSMOS1ABC...`) decodes to identical bytes but compares as different strings. A governance proposal can therefore register the same address twice under different cases, causing `BeginBlocker` to subtract that address's balance **twice** from the circulating supply, permanently understating it and bypassing the `MaxSupply` cap.

---

### Finding Description

`validateBurnedAddresses` iterates the supplied list, calls `sdk.AccAddressFromBech32(addr)` to confirm validity, then inserts the raw string `addr` into a `map[string]bool` to detect duplicates:

```go
addressMap := make(map[string]bool)
for i, addr := range burnedAddresses {
    if _, err := sdk.AccAddressFromBech32(addr); err != nil { ... }
    if addressMap[addr] {
        return fmt.Errorf("duplicate burned address found: %s", addr)
    }
    addressMap[addr] = true
}
``` [1](#0-0) 

The decoded `sdk.AccAddress` (the canonical bytes) is discarded; only the raw string is stored. Bech32 is case-insensitive, so `cosmos1abc...` and `COSMOS1ABC...` decode to the same 20-byte address but are different strings. The map key comparison therefore treats them as distinct entries, and the duplicate check passes.

The `BeginBlocker` then iterates `params.BurnedAddresses` and accumulates balances:

```go
for _, ba := range params.BurnedAddresses {
    balance, err := k.GetAddressBalance(ctx, ba, denom)
    burned = burned.Add(balance)
}
totalsupply = totalsupply.Sub(burned)
``` [2](#0-1) 

`GetAddressBalance` resolves both `cosmos1abc...` and `COSMOS1ABC...` to the same account and returns the same balance. The balance is therefore added to `burned` twice, and `totalsupply` is understated by exactly one copy of that balance.

---

### Impact Explanation

The `MaxSupply` cap is the chain's hard invariant against unbounded minting:

```go
if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
    panic("the total supply has exceeded the maximum supply: ...")
}
``` [3](#0-2) 

With a double-counted burned address holding balance `B`, the computed circulating supply is `real_circulating - B` instead of `real_circulating`. The chain will not halt even when `real_circulating` exceeds `MaxSupply` by up to `B`. Since the burn address on mainnet holds a large balance (the zero-address `cro1qqq...` accumulates all burned tokens), `B` can be very large, making the effective cap bypass proportionally large. The corrupted value is the `totalsupply` variable used in the cap comparison every block.

**Impact: 4** — The `MaxSupply` supply-cap invariant is silently broken; the chain mints tokens beyond the configured hard cap without halting.

---

### Likelihood Explanation

The entry path is a governance `MsgUpdateParams` transaction. Any token holder can submit a governance proposal; it passes with a simple majority of voting power. The mixed-case duplicate is subtle and unlikely to be caught by casual review of a proposal's JSON payload. The Cosmos SDK CLI and REST endpoints accept uppercase bech32 addresses as valid. No special key or role beyond governance participation is required.

**Likelihood: 2** — Requires a governance proposal to pass, which is a meaningful barrier, but the bypass is invisible to standard duplicate-detection tooling and easy to embed in a large parameter-update proposal.

---

### Recommendation

Replace the raw-string map key with the canonical byte representation returned by `sdk.AccAddressFromBech32`:

```go
addrBytes, err := sdk.AccAddressFromBech32(addr)
if err != nil { ... }
key := addrBytes.String()   // or hex.EncodeToString(addrBytes)
if addressMap[key] {
    return fmt.Errorf("duplicate burned address found: %s", addr)
}
addressMap[key] = true
```

This ensures that `cosmos1abc...` and `COSMOS1ABC...` are recognized as the same address regardless of the case used in the proposal. Additionally, normalize each address to its canonical lowercase form before storing it in `BurnedAddresses` so that `BeginBlocker` always resolves a unique set of accounts.

---

### Proof of Concept

1. Obtain governance authority (submit a `MsgUpdateParams` proposal through normal governance).
2. Set `BurnedAddresses` to `["cosmos1burnaddr...", "COSMOS1BURNADDR..."]` — the same address in two cases.
3. `validateBurnedAddresses` calls `sdk.AccAddressFromBech32` on both (both succeed), inserts `"cosmos1burnaddr..."` then `"COSMOS1BURNADDR..."` as distinct map keys — no duplicate error is returned.
4. The proposal passes and params are stored with both entries.
5. Every block, `BeginBlocker` calls `GetAddressBalance` for each entry. Both resolve to the same account; the balance `B` is added to `burned` twice.
6. `totalsupply` is understated by `B` every block. If `real_circulating = MaxSupply + X` where `X ≤ B`, the cap check `totalsupply.GT(maxsupply)` evaluates to `(MaxSupply + X - 2B).GT(MaxSupply)` which is false for any `X ≤ 2B`, so the chain never halts despite exceeding the cap.
