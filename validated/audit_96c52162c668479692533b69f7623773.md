### Title
Approval Race Condition in `DirectDepositV1.creditDeposit()` Permanently Bricks USDT Deposits - (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` calls `token.approve(address(endpoint), balance)` without first resetting the allowance to zero. For tokens like USDT that enforce approval race protection (requiring allowance == 0 before a new `approve`), any residual allowance after a deposit causes all subsequent `creditDeposit()` calls for that token to revert permanently. The contract has no recovery path to reset allowances.

---

### Finding Description

In `creditDeposit()`, for each supported spot product token with a non-zero balance, the contract calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The `approve` amount is `balance` (a `uint256`), but the deposit amount is `uint128(balance)`. If `balance > type(uint128).max`, the endpoint only pulls `uint128(balance)` tokens, leaving a residual allowance of `balance - uint128(balance)`. Even without overflow, any scenario where the endpoint does not consume the full approved amount (partial fill, fee deduction, internal cap) leaves a non-zero allowance.

On the next call to `creditDeposit()`, USDT's `approve` will revert because its implementation requires the current allowance to be zero before setting a new value. Since `DirectDepositV1` exposes no function to reset allowances — only `withdraw()` (transfer) and `withdrawNative()` — there is no recovery path.

By contrast, `ContractOwner.wrapVaultAsset()` already applies the correct reset-then-set pattern:

```solidity
assetToken.approve(tokenAddr, 0);
assetToken.approve(tokenAddr, assetBalance);
``` [2](#0-1) 

This fix was not applied to `DirectDepositV1`. The `ERC20Helper` library used elsewhere in the protocol provides only `safeTransfer` and `safeTransferFrom` — no `safeApprove` or `forceApprove` equivalent. [3](#0-2) 

---

### Impact Explanation

USDT deposits via `DirectDepositV1` are permanently bricked once a residual allowance accumulates. The subaccount's DDA contract can no longer process USDT collateral deposits. Funds sent to the DDA in USDT are effectively stuck (they can only be recovered via `withdraw()` by the owner, not deposited into the protocol). This constitutes a collateral lock for any subaccount whose DDA has a non-zero USDT allowance to the endpoint.

---

### Likelihood Explanation

`creditDeposit()` is an unrestricted `external` function — any caller can invoke it. [4](#0-3) 

The `uint128` truncation trigger is realistic: a user or contract sending more than `type(uint128).max` (~3.4 × 10^38 raw units) of a low-decimal token to the DDA would trigger it. Additionally, any endpoint-side behavior that does not consume the full approved amount (e.g., internal caps, rounding, or future upgrades) would also trigger it. Once triggered, the condition is permanent and self-reinforcing on every subsequent call.

---

### Recommendation

Replace the bare `approve` call in `creditDeposit()` with a reset-then-approve pattern, mirroring what `ContractOwner.wrapVaultAsset()` already does:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Alternatively, implement a `forceApprove` helper (as OpenZeppelin's `SafeERC20` does) in `ERC20Helper` and use it throughout the protocol, including in `DirectDepositV1`.

---

### Proof of Concept

1. USDT is listed as a supported spot product in `SpotEngine`.
2. A user sends `balance > type(uint128).max` USDT to their DDA contract, or the endpoint consumes less than the approved amount for any reason.
3. `creditDeposit()` is called (by anyone): `token.approve(endpoint, balance)` succeeds; `depositCollateralWithReferral(..., uint128(balance), ...)` pulls only `uint128(balance)`, leaving allowance = `balance - uint128(balance) > 0`.
4. On the next call to `creditDeposit()` with any USDT balance: `token.approve(endpoint, newBalance)` reverts because USDT's `approve` requires current allowance == 0.
5. All future USDT deposits via this DDA are permanently blocked. The subaccount cannot receive USDT collateral through the direct deposit path. The only exit is `withdraw()` (owner-only), which transfers tokens out but does not reset the allowance, so the deposit path remains broken indefinitely. [4](#0-3)

        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {

            address tokenAddr = spotEngine.getToken(productId);
