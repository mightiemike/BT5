### Title
Stale `digestToSubaccount` Mapping After Close Enables Zero-Margin Isolated Trading — (`core/contracts/OffchainExchange.sol`)

---

### Summary

`_tryCloseIsolatedSubaccount` clears `parentSubaccounts` and `isolatedSubaccounts` but never clears `digestToSubaccount[digest]`. After a close, resubmitting the same `CreateIsolatedSubaccount` transaction returns the old isolated subaccount address with no margin transfer. Subsequent `matchOrders` calls route trades to that 0-margin isolated subaccount, allowing the attacker to trade with zero collateral while the parent has already reclaimed its margin.

---

### Finding Description

**Step 1 — First `CreateIsolatedSubaccount` (digest D, margin M):**

`createIsolatedSubaccount` passes the early-return guard at line 1009 (mapping is zero), creates `isolatedSubaccount_A`, and writes:

```
digestToSubaccount[D]              = isolatedSubaccount_A
parentSubaccounts[isolatedSubaccount_A] = parent
isolatedSubaccounts[parent][id]    = isolatedSubaccount_A
spotEngine: parent -= M, isolatedSubaccount_A += M
``` [1](#0-0) 

**Step 2 — `CloseIsolatedSubaccount`:**

`_tryCloseIsolatedSubaccount` transfers M back to parent and clears the subaccount registrations:

```
isolatedSubaccounts[parent][id]         = bytes32(0)   ← cleared
parentSubaccounts[isolatedSubaccount_A] = bytes32(0)   ← cleared
spotEngine: isolatedSubaccount_A -= M, parent += M
```

But `digestToSubaccount[D]` is **never touched** — it still equals `isolatedSubaccount_A`. [2](#0-1) 

**Step 3 — Second `CreateIsolatedSubaccount` (same digest D):**

The early-return guard fires immediately:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    return digestToSubaccount[digest];   // returns isolatedSubaccount_A, no margin transfer
}
```

The parent loses no margin. `parentSubaccounts` and `isolatedSubaccounts` are not re-populated. `isolatedSubaccount_A` is returned as the "active" isolated subaccount with a zero spot balance. [3](#0-2) 

**Step 4 — `matchOrders` routes to the 0-margin subaccount:**

```solidity
if (digestToSubaccount[ordersInfo.taker.digest] != bytes32(0)) {
    taker.order.sender = digestToSubaccount[ordersInfo.taker.digest];
}
```

`digestToSubaccount[D]` is still `isolatedSubaccount_A`, so the trade executes against a subaccount with zero margin and no registered parent. [4](#0-3) 

**Why the second `CreateIsolatedSubaccount` can be submitted:**

The `CreateIsolatedSubaccount` transaction type is processed in `EndpointTx` without calling `validateSignedTx` (no endpoint-level nonce check). The signature check inside `createIsolatedSubaccount` is bypassed by the early return at line 1009. The same transaction bytes can be re-queued and processed by the sequencer. [5](#0-4) 

---

### Impact Explanation

The attacker reclaims margin M from the parent (via close), then reactivates the same isolated subaccount with 0 margin. `matchOrders` routes trades to it without any collateral check. Losses on those trades are absorbed by the protocol insurance fund. The attacker effectively trades with free leverage: profits go to the isolated subaccount (which the attacker controls via the parent address embedded in its `bytes32` identifier), losses are socialized.

---

### Likelihood Explanation

The attack requires:
1. Submitting a `CreateIsolatedSubaccount` transaction — a standard supported flow.
2. Submitting a `CloseIsolatedSubaccount` transaction — also a standard supported flow.
3. Resubmitting the same `CreateIsolatedSubaccount` bytes — no on-chain nonce prevents this for this transaction type.
4. Submitting a `matchOrders` transaction — the normal trading flow.

All four steps are reachable through the standard sequencer queue. No admin access, key compromise, or external dependency is required.

---

### Recommendation

In `_tryCloseIsolatedSubaccount`, clear `digestToSubaccount` for all digests that map to the closing subaccount. Because the current mapping is unidirectional (`digest → subaccount`), the fix requires either:

- **Option A:** Add a reverse mapping `subaccount → digest[]` populated in `createIsolatedSubaccount`, and iterate it during close to delete each `digestToSubaccount` entry.
- **Option B:** In `matchOrders`, after resolving `digestToSubaccount[digest]`, verify the resolved subaccount is still active via `isIsolatedSubaccountActive` before substituting the sender.
- **Option C:** In `createIsolatedSubaccount`, when the early-return path is taken, verify the returned subaccount is still active (i.e., `parentSubaccounts[digestToSubaccount[digest]] != bytes32(0)`); if not, treat the digest as unset and proceed with full creation and margin transfer.

Option C is the minimal on-chain fix with no new storage:

```solidity
if (digestToSubaccount[digest] != bytes32(0)) {
    bytes32 existing = digestToSubaccount[digest];
    if (parentSubaccounts[existing] != bytes32(0)) {
        return existing;  // still active, safe early return
    }
    // subaccount was closed; fall through to re-create and re-transfer margin
    delete digestToSubaccount[digest];
}
```
