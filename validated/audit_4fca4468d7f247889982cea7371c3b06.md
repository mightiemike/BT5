### Title
Blind `TryFrom<protobuf::ConsensusSignature>` Conversion Allows Malicious Sync Peer to Persist and Serve a Wrong Block Signature via RPC — (`crates/apollo_protobuf/src/converters/header.rs`)

---

### Summary

The `TryFrom<protobuf::ConsensusSignature> for BlockSignature` conversion performs a blind field-to-field mapping with no cryptographic validation. The p2p sync client pipeline also performs no signature verification before writing to storage. A malicious sync peer can send a `ConsensusSignature` with `r` and `s` swapped; the node will accept, persist, and serve the corrupted signature as authoritative block data.

---

### Finding Description

The conversion at lines 301–309 of `header.rs` maps `protobuf::ConsensusSignature.r → Signature.r` and `.s → Signature.s` with no ordering check, domain check, or cryptographic validation:

```rust
// crates/apollo_protobuf/src/converters/header.rs:301-309
impl TryFrom<protobuf::ConsensusSignature> for starknet_api::block::BlockSignature {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ConsensusSignature) -> Result<Self, Self::Error> {
        Ok(Self(Signature {
            r: value.r.ok_or(missing("SignedBlockHeader::r"))?.try_into()?,
            s: value.s.ok_or(missing("SignedBlockHeader::s"))?.try_into()?,
        }))
    }
}
``` [1](#0-0) 

The p2p sync client's `parse_data_for_block` in `apollo_p2p_sync/src/client/header.rs` only validates block number ordering and signature vector length — it performs **no cryptographic verification** of the signature content: [2](#0-1) 

The `write_to_storage` implementation then calls `append_block_signature` directly, persisting whatever `BlockSignature` was decoded: [3](#0-2) 

`append_block_signature` in storage performs only a block-number-existence check before inserting: [4](#0-3) 

The function `verify_block_signature` exists in `starknet_api/src/block.rs` and is capable of verifying the Poseidon-hashed ECDSA signature: [5](#0-4) 

However, a codebase-wide search confirms `verify_block_signature` is **only called in tests** (`block_test.rs`), never in the production sync pipeline. The stored signature is later re-served by the p2p sync server and by RPC without re-verification: [6](#0-5) 

---

### Impact Explanation

A malicious sync peer sends a `SignedBlockHeader` whose `ConsensusSignature` has `r_bytes` placed in the `s` field and `s_bytes` in the `r` field. The conversion succeeds, the corrupted `BlockSignature(s_bytes, r_bytes)` is written to the `block_signatures` DB table, and is subsequently:

1. Returned by the p2p sync server to any downstream node that syncs from this node (propagating the corruption).
2. Served by the RPC layer as the authoritative block signature.

Clients calling `starknet_getBlockWithTxHashes` or equivalent endpoints receive an authoritative-looking but cryptographically invalid signature. Any client or downstream system that calls `verify_block_signature` on the returned data will observe a verification failure for a block that is otherwise valid, breaking trust in the node's data. The corruption is **silent** — no error is logged or returned during ingestion.

Note: the specific sub-claim in the question that "downstream signature verifiers **accept** the swapped signature" is incorrect. A swapped ECDSA `(r, s) → (s, r)` is not a valid signature for the same message, so verifiers will **reject** it. The actual harm is that the node stores and advertises a wrong value, causing verification failures rather than false acceptances.

---

### Likelihood Explanation

Any node participating in the p2p sync network can act as a sync peer. No special privilege is required. The attack requires only constructing a valid-looking `SignedBlockHeader` protobuf message with swapped `r`/`s` bytes in the `ConsensusSignature` field — a trivial wire-level manipulation. The absence of any signature verification in the sync ingestion path means the attack succeeds deterministically.

---

### Recommendation

Call `verify_block_signature` (or an equivalent check) inside `parse_data_for_block` in `crates/apollo_p2p_sync/src/client/header.rs` before accepting a `SignedBlockHeader` from a peer. The sequencer public key required for verification is already available in the node configuration. A peer that sends an invalid signature should be penalized via the existing `BadPeerError` mechanism.

---

### Proof of Concept

```rust
// Construct a ConsensusSignature with r and s swapped.
let r_bytes = Felt::from(0xAAu64);
let s_bytes = Felt::from(0xBBu64);

let swapped = protobuf::ConsensusSignature {
    r: Some(s_bytes.into()), // s_bytes placed in r field
    s: Some(r_bytes.into()), // r_bytes placed in s field
};

let block_sig = starknet_api::block::BlockSignature::try_from(swapped)
    .expect("conversion must succeed");

// Assert the stored BlockSignature has r and s swapped relative to the original.
assert_eq!(block_sig.0.r, s_bytes);
assert_eq!(block_sig.0.s, r_bytes);
// No error is returned; the corrupted value is accepted and would be written to storage.
```

    fn append_block_signature(
        self,
        block_number: BlockNumber,
        block_signature: &BlockSignature,
    ) -> StorageResult<Self> {
        let current_header_marker = self.get_header_marker()?;
        if block_number >= current_header_marker {
            return Err(StorageError::BlockSignatureForNonExistingBlock {
                block_number,
                block_signature: *block_signature,
            });
        }

        let block_signatures_table = self.open_table(&self.tables().block_signatures)?;
        block_signatures_table.insert(self.txn(), &block_number, block_signature)?;
        Ok(self)
    }
