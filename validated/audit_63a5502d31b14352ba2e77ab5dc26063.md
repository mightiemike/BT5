Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual Swapper, Enabling Full Allowlist Bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` parameter, which the pool binds to `msg.sender` of the `pool.swap()` call. When `MetricOmmSimpleRouter` is the caller, `msg.sender` to the pool is the router contract address, not the originating user. If the pool admin allowlists the router to enable standard periphery usage, every unprivileged address can bypass the per-user swap allowlist by routing through the router.

## Finding Description

**Root cause — wrong actor checked in the extension:**
`SwapAllowlistExtension.beforeSwap` receives `sender` as its first argument and evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`.

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

**Pool binds `sender` to `msg.sender` of the `swap()` call:**
`MetricOmmPool.swap` unconditionally passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    ...
```

`ExtensionCalling._beforeSwap` forwards this value directly to the extension with no transformation.

**Router is `msg.sender` to the pool:**
`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(params.recipient, ...)` directly. There is no mechanism to forward the original `msg.sender` (the actual user) to the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

**Result:** The extension evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

**Exploit path:**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to a curated set.
2. Admin calls `setAllowedToSwap(pool, router, true)` to allow curated users to use the standard periphery.
3. Alice (not individually allowlisted) calls `router.exactInputSingle({pool: curated_pool, recipient: alice, ...})`.
4. Router calls `pool.swap(alice, ...)` — pool sees `msg.sender = router`.
5. Pool calls `_beforeSwap(sender=router, ...)`.
6. Extension checks `allowedSwapper[pool][router] == true` → passes. Alice swaps successfully on the curated pool.

**Second failure mode:** If the admin does NOT allowlist the router, individually allowlisted users cannot use `MetricOmmSimpleRouter` at all — the extension sees `sender=router` (not allowlisted) and reverts `NotAllowedToSwap`, making the standard periphery completely unusable for curated pools.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control extension. Its fundamental purpose — restricting swaps to a curated set of addresses — is completely nullified when the router is allowlisted. Any unprivileged EOA can execute swaps on a pool that the admin intended to restrict. This constitutes broken core pool functionality (the allowlist extension does not enforce its stated policy through the supported public periphery entrypoint) and an admin-boundary break (an unprivileged path bypasses the access control the admin configured). Severity: High.

## Likelihood Explanation
- Pool admins who deploy `SwapAllowlistExtension` and want curated users to use `MetricOmmSimpleRouter` will naturally allowlist the router — it is the expected operational pattern for using the standard periphery.
- The router is a factory-validated, trusted periphery contract; allowlisting it carries no obvious red flag.
- No special attacker capability is required: any EOA can call `router.exactInputSingle` with the curated pool address.
- The trigger is fully unprivileged once the router is allowlisted.

## Recommendation
The `beforeSwap` hook must gate on the economic actor, not the intermediary. The cleanest fix is to have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check it when present, falling back to `sender` for direct pool callers. A more robust protocol-level fix is to add an explicit `originator` field to the swap interface that the pool passes to extensions as a distinct parameter, separate from the callback-payer `sender`.

## Proof of Concept

```
Setup:
  pool deployed with SwapAllowlistExtension
  admin: setAllowedToSwap(pool, router, true)
  alice: address NOT in allowlist

Attack:
  1. alice calls router.exactInputSingle({
       pool: curated_pool, recipient: alice, amountIn: X, ...
     })
  2. router calls pool.swap(alice, zeroForOne, X, ...)
     → pool's msg.sender = router
  3. pool calls _beforeSwap(sender=router, recipient=alice, ...)
  4. SwapAllowlistExtension.beforeSwap(sender=router, ...)
     checks: allowedSwapper[pool][router] == true  ✓
     → does NOT revert
  5. Swap executes. Alice (not individually allowlisted)
     successfully swaps on the curated pool.

Foundry test outline:
  - Deploy SwapAllowlistExtension + pool with extension configured
  - vm.prank(admin); extension.setAllowedToSwap(pool, router, true)
  - vm.prank(alice); router.exactInputSingle(...)
  - Assert swap succeeds despite alice not being in allowlist
  - Assert extension.isAllowedToSwap(pool, alice) == false
```
