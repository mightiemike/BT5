### Title
Linked Signer Can Re-Delegate Subaccount Signing Authority to an Arbitrary Address â€” (`core/contracts/EndpointTx.sol`)

### Summary
`processTransactionImpl` processes `LinkSigner` transactions with `allowLinkedSigner = true`, meaning the current linked signer's signature is accepted as valid authorization to overwrite `linkedSigners[subaccount]`. A linked signer can therefore point the subaccount's signing authority at any address they control, permanently hijacking it without the subaccount owner's consent.

---

### Finding Description

In `processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` forwards that flag to `validateSignature`: [2](#0-1) 

`validateSignature` passes `getLinkedSigner(sender)` â€” the **current** linked signer â€” as the accepted alternate signer to the verifier: [3](#0-2) 

`Verifier.validateSignature` then accepts the signature if it was produced by **either** the subaccount owner **or** the linked signer: [4](#0-3) 

After validation passes, the storage write unconditionally overwrites the linked signer: [5](#0-4) 

There is no guard that restricts who may sign a `LinkSigner` transaction to the subaccount owner only. The slow-mode path correctly enforces this via `validateSender`, which requires `msg.sender == address(uint160(bytes20(txn.sender)))`: [6](#0-5) 

The fast (sequencer) path has no equivalent restriction.

---

### Impact Explanation

A linked signer can:
1. Sign a new `LinkSigner` transaction naming any address (e.g., `attacker2`) as the new signer for the victim subaccount.
2. Submit it through the sequencer path.
3. After processing, `linkedSigners[victim] = attacker2`.

From that point, `attacker2` can sign withdrawals, trades, liquidations, and further re-delegations on behalf of the victim. The subaccount owner loses signing authority without any on-chain action on their part. This satisfies the Critical scope: **unauthorized transaction execution on behalf of a user without valid consent or signer ownership**.

---

### Likelihood Explanation

Any party who has ever been granted linked-signer status â€” including a party the victim trusted temporarily â€” can execute this attack. No leaked keys, admin access, or governance capture is required. The attacker only needs their own private key and the ability to submit a signed transaction to the sequencer, which is the normal protocol interaction path.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
// EndpointTx.sol, LinkSigner branch in processTransactionImpl
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // <-- must be false; only the subaccount owner may re-link
);
```

This mirrors the invariant already enforced in the slow-mode path, where `validateSender` ensures only the address derived from the subaccount bytes32 can submit a `LinkSigner` transaction.

### Proof of Concept

```solidity
// Precondition: linkedSigners[victim] == attacker (set legitimately or via any prior path)

// Step 1: attacker constructs a LinkSigner tx pointing to attacker2

    sender: victim,          // victim subaccount bytes32
    signer: bytes32(uint256(uint160(attacker2))),
    nonce: nonces[address(uint160(bytes20(victim)))]
});
IEndpoint.SignedLinkSigner memory signedTx = IEndpoint.SignedLinkSigner({
    tx: inner,
    signature: sign(attacker_privkey, eip712Digest(inner))  // signed by attacker, NOT victim owner
});

// Step 2: sequencer processes the transaction
endpoint.processTransactionImpl(abi.encodePacked(
    uint8(IEndpoint.TransactionType.LinkSigner),
    abi.encode(signedTx)
));

// Step 3: assert takeover
assert(endpoint.getLinkedSigner(victim) == attacker2);
// attacker2 can now sign withdrawals, trades, and further re-delegations for victim
```
