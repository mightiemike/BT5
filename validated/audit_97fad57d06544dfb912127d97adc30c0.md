### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables Caller to Drain usdcE Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(...)` at line 616 without checking the return value, while the same function uses the safe wrapper (`safeTransfer`) for the outbound leg at line 618. The protocol's own `ERC20Helper` library provides `safeTransferFrom` for exactly this purpose, but it is not used here. If the USDC `transferFrom` silently returns `false`, the function continues, withdraws usdcE from the `DirectDepositV1` contract, and sends it to the caller — who provided nothing.

---

### Finding Description

`replaceUsdcEWithUsdc` is an `external` function with no `onlyOwner` guard, restricted only by `block.chainid == 57073`:

```solidity
// ContractOwner.sol L608–L620
function replaceUsdcEWithUsdc(bytes32 subaccount) external {
    require(block.chainid == 57073, ERR_UNAUTHORIZED);
    ...
    uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
    if (balance > 0) {
        IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance); // L616 — raw, unchecked
        DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));        // L617
        IERC20Base(usdcE).safeTransfer(msg.sender, balance);                  // L618 — safe
    }
}
``` [1](#0-0) 

The `ERC20Helper` library — already imported and bound via `using ERC20Helper for IERC20Base` — provides `safeTransferFrom` which low-level calls `transferFrom` and requires both `success == true` and a truthy decoded return value:

```solidity
// ERC20Helper.sol L23–L42
function safeTransferFrom(IERC20Base self, address from, address to, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(...);
    require(success && (data.length == 0 || abi.decode(data, (bool))), ERR_TRANSFER_FAILED);
}
``` [2](#0-1) 

This safe variant is not used at line 616. The raw `IERC20Base.transferFrom` returns a `bool` that is silently discarded. [3](#0-2) 

---

### Impact Explanation

If `IERC20Base(usdc).transferFrom` returns `false` (rather than reverting) — which is valid ERC-20 behavior for tokens that do not revert on failure — execution continues past line 616. The subsequent `DirectDepositV1.withdraw` call pulls all usdcE held in the `directDepositV1` contract to `ContractOwner`, and `safeTransfer` then sends that usdcE to `msg.sender`. The caller receives real usdcE tokens without having transferred any USDC. The `directDepositV1` contract's usdcE balance is fully drained. [4](#0-3) 

---

### Likelihood Explanation

The function is callable by any unprivileged address on chain 57073 (Ink). The only precondition is that a `directDepositV1` address for the target `subaccount` holds a nonzero usdcE balance. The attacker needs zero USDC allowance — they simply call the function without approving USDC. Whether this is exploitable in practice depends on whether the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` on chain 57073 returns `false` or reverts on insufficient allowance. Standard Circle USDC reverts, but the contract makes no assumption about this and provides no defense. The inconsistency with the rest of the contract (which uses `safeTransfer` on the very next line) confirms this is an unintentional omission. [5](#0-4) 

---

### Recommendation

Replace the raw `transferFrom` at line 616 with `ERC20Helper.safeTransferFrom`, consistent with how the rest of the contract handles outbound transfers:

```solidity
// Before (line 616):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After:
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`safeTransferFrom` is already available via the `using ERC20Helper for IERC20Base` directive declared at the top of the contract. [6](#0-5) 

---

### Proof of Concept

1. A `directDepositV1` for some `subaccount` holds `N` usdcE tokens.
2. Attacker calls `replaceUsdcEWithUsdc(subaccount)` on chain 57073 with zero USDC allowance.
3. `IERC20Base(usdc).transferFrom(attacker, directDepositV1, N)` returns `false` (no revert).
4. Return value is discarded; execution continues.
5. `DirectDepositV1(directDepositV1).withdraw(usdcE)` transfers all `N` usdcE to `ContractOwner`.
6. `IERC20Base(usdcE).safeTransfer(attacker, N)` sends `N` usdcE to the attacker.
7. Attacker holds `N` usdcE; `directDepositV1` holds nothing; no USDC was ever transferred. [4](#0-3) [7](#0-6)
