### Title
Missing Upper-Bound Constraint on `min_gas_price_percentage` Causes Gateway to Reject All Valid `AllResources` Transactions — (`crates/apollo_gateway_config/src/config.rs`)

---

### Summary

`StatefulTransactionValidatorConfig::min_gas_price_percentage` is a `u8` field (range 0–255) that is semantically a percentage (0–100). No `#[validate(range(max = 100))]` attribute is present on the field, so the `Validate`-derived config validation silently accepts values above 100. When such a value is loaded, `validate_tx_l2_gas_price_within_threshold` computes a threshold that exceeds the previous block's L2 gas price, causing the gateway to reject every valid `AllResources` transaction at admission time.

---

### Finding Description

`StatefulTransactionValidatorConfig` derives `Validate` and is nested-validated by `GatewayStaticConfig`, but `min_gas_price_percentage` carries no range constraint:

```rust
// crates/apollo_gateway_config/src/config.rs  line 286
pub min_gas_price_percentage: u8, // E.g., 80 to require 80% of threshold.
``` [1](#0-0) 

Compare with `fee_escalation_percentage` in the mempool config, which correctly carries `#[validate(range(min = 1, max = 100))]`:

```rust
// crates/apollo_mempool_config/src/config.rs  line 65-66
#[validate(range(min = 1, max = 100))]
pub fee_escalation_percentage: u8,
``` [2](#0-1) 

The field is consumed in `validate_tx_l2_gas_price_within_threshold`:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...)
}
``` [3](#0-2) 

If `min_gas_price_percentage` is set to, say, `150`, the threshold becomes `1.5 × previous_block_l2_gas_price`. Any transaction whose `max_price_per_unit` equals the previous block's gas price (the normal, valid case) will be below the threshold and rejected with `GAS_PRICE_TOO_LOW`.

The config is loaded from a JSON file and propagated through `GatewayStaticConfig` → `StatefulTransactionValidatorConfig` with `#[validate(nested)]`, but because no `range` constraint exists on the field, `validate()` returns `Ok` for any `u8` value. [4](#0-3) 

---

### Impact Explanation

**High. Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

With `min_gas_price_percentage > 100`, every `AllResources` transaction whose `l2_gas.max_price_per_unit` is at or below the previous block's L2 gas price is rejected at the stateful-validation stage with a `GAS_PRICE_TOO_LOW` error. Because the previous block's gas price is the natural reference point for users setting their gas price, this effectively blocks all normal `AllResources` transactions from entering the mempool. `L1Gas` (legacy) transactions are unaffected because the branch is skipped. [5](#0-4) 

---

### Likelihood Explanation

**Low.** Exploitation requires a misconfigured or maliciously set `min_gas_price_percentage` value in the node's config file. However, the absence of a validation constraint means the config framework provides no safety net: an operator typo (e.g., entering `150` instead of `15`) or a malicious config change would silently pass all validation checks and immediately cause the gateway to reject valid transactions.

---

### Recommendation

Add `#[validate(range(max = 100))]` to `min_gas_price_percentage` in `StatefulTransactionValidatorConfig`, mirroring the existing constraint on `fee_escalation_percentage`:

```rust
#[validate(range(max = 100))]
pub min_gas_price_percentage: u8,
``` [6](#0-5) 

---

### Proof of Concept

1. Set `gateway_config.static_config.stateful_tx_validator_config.min_gas_price_percentage` to `150` in the node config JSON.
2. Start the gateway. `GatewayStaticConfig::validate()` calls `StatefulTransactionValidatorConfig::validate()`, which returns `Ok(())` because no range constraint exists on the field.
3. Submit any `InvokeV3` or `DeclareV3` transaction with `AllResourceBounds` where `l2_gas.max_price_per_unit` equals the previous block's L2 gas price (e.g., `100`).
4. `validate_tx_l2_gas_price_within_threshold` computes `threshold = (150/100) * 100 = 150`. Since `100 < 150`, the transaction is rejected with:
   ```
   Transaction L2 gas price 100 is below the required threshold 150.
   ```
5. All `AllResources` transactions are rejected until the config is corrected and the node restarted. The existing test suite confirms the threshold arithmetic at lines 230–286 of `stateful_transaction_validator_test.rs` but never tests `min_gas_price_percentage > 100`. [7](#0-6)
