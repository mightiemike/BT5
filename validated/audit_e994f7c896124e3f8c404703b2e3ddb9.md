### Title
User-Controlled `feeX18` in `WithdrawCollateralV2` Allows Complete Withdrawal Fee Bypass — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

The `WithdrawCollateralV2` transaction type accepts a user-supplied `feeX18` field with no minimum enforcement. A user can sign a withdrawal with `feeX18 = 0`, causing the protocol to charge zero withdrawal fee. The original `WithdrawCollateral` (V1) always charges the full configured `withdrawFeeX18`; V2 breaks this invariant by delegating fee selection to the user.

---

### Finding Description

`SignedWithdrawCollateralV2` carries a `feeX18` field that is embedded in the user-signed transaction struct:

```solidity
struct SignedWithdrawCollateralV2 {
    WithdrawCollateralV2 tx;
    CompactSignature signature;
    int128 feeX18;          // ← user-controlled
}
``` [1](#0-0) 

When the sequencer processes this transaction type, the only fee validation performed is:

```solidity
require(signedTx.feeX18 >= 0);
require(signedTx.feeX18 <= currentFeeX18);
chargeFee(signedTx.tx.sender, signedTx.feeX18, signedTx.tx.productId);
``` [2](#0-1) 

The lower bound is `>= 0`, meaning `feeX18 = 0` is explicitly permitted by the contract. There is no floor enforcing a minimum fee. By contrast, the V1 path always charges the full configured fee:

```solidity
chargeFee(
    signedTx.tx.sender,
    spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18,
    signedTx.tx.productId
);
``` [3](#0-2) 

The `feeX18` value is part of the signed payload, so the user chooses it at signing time. Nothing in the contract prevents a user from signing with `feeX18 = 0` and submitting that transaction.

---

### Impact Explanation

Any user withdrawing via `WithdrawCollateralV2` can set `feeX18 = 0` and pay no withdrawal fee. The protocol's `sequencerFee` accounting for the relevant `productId` receives nothing for that withdrawal. Accumulated over many withdrawals, this drains the protocol's fee revenue entirely for the V2 path. The corrupted state delta is `sequencerFee[productId]` — it receives `0` instead of `withdrawFeeX18` per withdrawal. [4](#0-3) 

---

### Likelihood Explanation

The entry path is direct: any user who knows the V2 transaction format can sign a `WithdrawCollateralV2` with `feeX18 = 0`. No special privilege, admin access, or external dependency is required. The user controls the `feeX18` field at signing time, and the contract enforces no minimum. Every withdrawal via V2 is an opportunity to exploit this.

---

### Recommendation

Enforce a minimum fee equal to the configured `withdrawFeeX18` for the product, mirroring the V1 behavior:

```solidity
int128 currentFeeX18 = spotEngine.getConfig(signedTx.tx.productId).withdrawFeeX18;
require(signedTx.feeX18 >= currentFeeX18, "Fee below minimum");
require(signedTx.feeX18 <= currentFeeX18, "Fee above maximum");
```

If V2 is intentionally designed to allow fee discounts (e.g., for privileged users), the discount eligibility must be validated on-chain against a stored allowlist or role, not left to the user's discretion.

---

### Proof of Concept

1. User constructs a `WithdrawCollateralV2` struct with `feeX18 = 0` and signs it with their private key.
2. The signed transaction is submitted to the sequencer (or directly to `processTransactionImpl` if accessible).
3. The contract executes:
   - `require(0 >= 0)` — passes.
   - `require(0 <= currentFeeX18)` — passes (assuming `currentFeeX18 > 0`).
   - `chargeFee(sender, 0, productId)` — charges zero fee.
4. `clearinghouse.withdrawCollateral` executes the full withdrawal amount with no fee deducted.
5. `sequencerFee[productId]` is incremented by `0` instead of `withdrawFeeX18`. [5](#0-4)
