Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` validates position recipient (`owner`) instead of actual token payer, enabling full allowlist bypass - (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` gates deposits by checking `allowedDepositor[pool][owner]`, where `owner` is the LP-position recipient supplied by the caller — not the address that actually provides tokens. Because `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accepts an arbitrary `owner` with no requirement that `owner == msg.sender`, any unprivileged caller can name an allowlisted address as `owner`, pass the extension check, and have their own tokens pulled into the restricted pool while the allowlisted address receives the position.

## Finding Description

**Root cause — extension ignores `sender`, checks `owner`:**

`DepositAllowlistExtension.beforeAddLiquidity` discards the `sender` argument (unnamed, position 0) and validates only `owner`:

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol L38-40
if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
}
``` [1](#0-0) 

`msg.sender` here is the pool; `owner` is the position recipient passed by the original caller.

**`sender` is always the adder contract — useless for per-user gating:**

`ExtensionCalling._beforeAddLiquidity` passes `sender` (the direct pool caller, i.e., the adder contract) and `owner` (the position recipient) to the extension:

```solidity
// metric-core/contracts/ExtensionCalling.sol L95-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

`sender` is always the adder contract address — identical for every user — so it cannot distinguish callers.

**`owner` is caller-controlled; payer is stored separately in transient storage:**

`addLiquidityExactShares` (owner-explicit overload) only validates `owner != address(0)`, then stores `msg.sender` as the payer in transient storage, completely separate from `owner`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L65-67
_validateOwner(owner);   // only checks != address(0)
...
return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
``` [3](#0-2) 

The callback then pulls tokens from `payer` (the transient-stored `msg.sender`), not from `owner`:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L162-173
(address expectedPool, address payer, uint256 max0, uint256 max1) = _loadPayContext();
...
pay(token0, payer, msg.sender, amount0Delta);
``` [4](#0-3) 

**Complete exploit path:**

1. Pool admin deploys pool with `DepositAllowlistExtension`; only `alice` is allowlisted.
2. Unauthorized `bob` calls `addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "")`.
3. `_validateOwner(alice)` passes (non-zero).
4. Pool calls `beforeAddLiquidity(adderContract, alice, ...)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → no revert.
6. Callback pulls tokens from `bob` (transient payer).
7. `alice` receives the LP position; `bob`'s tokens enter the restricted pool.

The allowlist is fully bypassed. The extension has no access to the real payer at any point in the call chain.

## Impact Explanation

The deposit allowlist is the pool admin's primary mechanism for restricting who can provide liquidity (e.g., KYC-gated or compliance-restricted pools). The bypass allows any unprivileged address to inject funds into a restricted pool without being allowlisted. This constitutes an admin-boundary break: an unprivileged path circumvents a pool-admin-configured access control, altering pool composition and potentially affecting existing LPs' value. The allowlisted address (`alice`) receives an unsolicited position; while she can remove it, the pool's state is already modified during the window before removal.

## Likelihood Explanation

The `owner`-explicit overload of `addLiquidityExactShares` is a documented, publicly callable function on the standard periphery entry point for EOAs. Any actor who knows a single allowlisted address — observable on-chain via `AllowedToDepositSet` events emitted by `setAllowedToDeposit` — can execute the bypass. No special privilege is required; the only cost is the tokens the attacker voluntarily deposits. [5](#0-4) 

## Recommendation

1. **In `DepositAllowlistExtension.beforeAddLiquidity`**: validate `sender` (the direct pool caller) rather than `owner`. This requires the adder to enforce `owner == msg.sender` so the payer and position recipient are always the same address, making the `owner` check equivalent to a payer check.

2. **In `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` (owner-explicit overload)**: enforce `require(owner == msg.sender)` so the payer and position recipient are always the same address when going through the standard adder.

3. **Alternatively**: thread the real payer address through `extensionData` so the extension can validate it directly, though this requires the adder to be trusted to populate `extensionData` correctly.

## Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension; only alice is allowlisted.
// bob is NOT allowlisted.

// bob approves the adder for his tokens
token0.approve(address(adder), type(uint256).max);
token1.approve(address(adder), type(uint256).max);

// bob calls with alice as owner — allowlist check passes on alice, bob pays
vm.prank(bob);
adder.addLiquidityExactShares(
    address(pool),
    alice,          // owner: allowlisted → check passes
    uint80(1),
    deltas,
    1_000e18,
    1_000e18,
    ""
);

// Result: bob's tokens are in the pool, alice holds the LP position
// The deposit allowlist did not block bob
assertGt(pool.positionBinShares(alice, 1, binIdx), 0);
```
