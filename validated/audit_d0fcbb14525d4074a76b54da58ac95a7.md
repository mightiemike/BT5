Audit Report

## Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` before updating the pool admin's global spread fee. `setPoolBinAdditionalFees`, also gated by `onlyPoolAdmin`, forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap check, allowing a pool admin to set per-bin fees up to `uint16.max` (65 535, i.e. 6.5535% in E6 terms) regardless of the factory-configured cap. Every swap through an affected bin pays the uncapped fee, causing direct loss of trader principal above the protocol-permitted maximum.

## Finding Description

`setPoolAdminFees` guards the global spread fee path: [1](#0-0) 

`setPoolBinAdditionalFees`, also callable by the pool admin, contains no equivalent guard and passes values straight through: [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index range and stores the values unconditionally: [3](#0-2) 

The stored values are then added directly to `baseFeeX64` in all four swap paths: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The same uncapped values propagate into the periphery data provider: [9](#0-8) 

**Exploit path:** Pool admin calls `factory.setPoolBinAdditionalFees(pool, bin, 65535, 65535)` → factory forwards to `pool.setBinAdditionalFees` with no cap check → `_binStates[bin].addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` stored → every subsequent swap through that bin computes `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, adding 6.5535% on top of the oracle-derived base spread → trader receives fewer tokens than the protocol cap permits.

## Impact Explanation

The factory owner sets `maxAdminSpreadFeeE6` to bound the maximum fee the pool admin may impose on traders. That bound is enforced for the global spread fee path but is entirely absent from the per-bin fee path. A pool admin can set `addFeeBuyE6 = addFeeSellE6 = 65535` on any bin, adding up to 6.5535% on top of the oracle-derived base spread for every swap through that bin. Traders pay more than the protocol-configured maximum — a direct loss of user principal. This is a confirmed admin-boundary break: the pool admin exceeds the factory-configured cap.

## Likelihood Explanation

The pool admin is a per-pool role (`poolAdmin[pool]`) set at pool creation and transferable. Any pool admin — including a malicious or compromised one — can call `setPoolBinAdditionalFees` at any time with no timelock. The call requires no special pool state and succeeds immediately. The per-bin fee path is a distinct, independently reachable setter that the existing cap logic does not cover.

## Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (uint256(addFeeBuyE6)  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `setBinAdditionalFees` on the pool by reading the factory's `maxAdminSpreadFeeE6` at call time.

import "forge-std/Test.sol";

contract BinFeeCapBypassTest is Test {
    function test_binFeeExceedsAdminCap() public {
        // 1. Deploy factory with maxAdminSpreadFeeE6 = 10_000 (1%)
        // 2. Create pool; pool admin is address(this)
        // 3. Pool admin sets per-bin fee to uint16.max = 65535 (6.5535%)
        vm.prank(poolAdmin);
        factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, type(uint16).max);

        assertEq(s.addFeeSellE6, 65535);

    }
