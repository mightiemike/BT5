### Title
Overly Restrictive Bech32 Address Validation in `NonFungibleTokenPacketData.ValidateBasic` Blocks ICS-721 Packet Receipt from Non-Cosmos Chains — (`x/nft-transfer/types/packet.go`)

---

### Summary

`NonFungibleTokenPacketData.ValidateBasic()` validates both the `Sender` and `Receiver` fields as Cosmos Bech32 addresses. The ICS-721 specification explicitly states that address formats must **not** be validated on receipt because sender and receiver chains may use entirely different address encodings. The code's own comment acknowledges this, yet the implementation contradicts it. When Cronos POS chain receives an IBC NFT packet whose `Sender` field is a non-Bech32 address (e.g., from an EVM chain using hex addresses), `OnRecvPacket` calls `ValidateBasic()`, which rejects the packet with an error acknowledgement. The cross-chain NFT transfer is permanently blocked for any non-Cosmos counterparty chain.

---

### Finding Description

In `x/nft-transfer/types/packet.go`, the `ValidateBasic` function for `NonFungibleTokenPacketData` contains the following checks:

```go
// decode the sender address
if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
}
...
// decode the receiver address
if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
    return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
}
```

The function's own leading comment states:

> NOTE: The addresses formats are not validated as the sender and recipient can have different formats defined by their corresponding chains that are not known to IBC.

This comment is a direct contradiction of the implementation. The ICS-721 spec mirrors ICS-20's design: the receiver address is an opaque string whose format is defined by the destination chain, and the sender address is an opaque string from the source chain. Neither should be validated as Bech32 in a cross-chain context.

The call path that triggers this on receipt is:

1. A relayer submits a `MsgRecvPacket` to Cronos POS chain.
2. `IBCModule.OnRecvPacket` unmarshals the packet data and calls `im.keeper.OnRecvPacket(...)`.
3. `Keeper.OnRecvPacket` immediately calls `data.ValidateBasic()`.
4. If the source chain uses non-Bech32 addresses (e.g., `0xabc...` hex), `sdk.AccAddressFromBech32(nftpd.Sender)` fails.
5. `OnRecvPacket` returns an error; `IBCModule.OnRecvPacket` writes an error acknowledgement.
6. The source chain's `OnAcknowledgementPacket` refunds the sender — the transfer is permanently blocked. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any non-Cosmos chain (EVM, Solana, etc.) that attempts to send NFTs to Cronos POS via ICS-721 will have every inbound packet rejected at the `ValidateBasic` step. The error acknowledgement is written synchronously, so the source chain's `OnAcknowledgementPacket` triggers a refund. The NFT transfer is silently blocked with no recourse for the user other than using a different receiver chain. This makes the ICS-721 module on Cronos POS incompatible with any non-Cosmos counterparty, directly analogous to the TempleGold `msg.sender != _to` restriction that blocks multisig wallet users from cross-chain transfers.

The corrupted invariant is: **the ICS-721 escrow/mint flow on the destination chain is never reached for packets originating from non-Bech32 chains**, even though the channel is open and the packet is otherwise valid. [4](#0-3) 

---

### Likelihood Explanation

ICS-721 is an open standard designed for cross-ecosystem NFT transfers. As EVM-compatible chains (including Cronos EVM) increasingly adopt IBC, packets from chains using hex addresses are a realistic and growing use case. Any relayer can submit such a packet; no privileged access is required. The trigger is a standard `MsgRecvPacket` transaction submitted by an unprivileged relayer. [5](#0-4) 

---

### Recommendation

Remove the Bech32 address validation for both `Sender` and `Receiver` from `NonFungibleTokenPacketData.ValidateBasic()`, consistent with the function's own comment and the ICS-721 specification. Address validation should only occur at the point of actual use (e.g., when minting/transferring to the receiver on the local chain), where the local chain's address format is known.

```go
// Remove these blocks from ValidateBasic:
// if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil { ... }
// if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil { ... }
```

The receiver address is already validated as Bech32 in `processReceivedPacket` at line 143 (`sdk.AccAddressFromBech32(data.Receiver)`), which is the correct place — it is the local chain's responsibility to validate that the receiver is a valid local address, not `ValidateBasic`. [6](#0-5) 

---

### Proof of Concept

1. Chain B (EVM chain, hex addresses) has an ICS-721 channel open to Cronos POS chain.
2. User on Chain B calls the ICS-721 transfer with `sender = "0xAbCd..."` and `receiver = "cro1..."` (valid Cronos Bech32).
3. Chain B escrows/burns the NFT and sends the packet. The packet's `NonFungibleTokenPacketData.Sender` is `"0xAbCd..."`.
4. Relayer submits `MsgRecvPacket` to Cronos POS.
5. `IBCModule.OnRecvPacket` → `Keeper.OnRecvPacket` → `data.ValidateBasic()` → `sdk.AccAddressFromBech32("0xAbCd...")` → error: `"invalid sender address"`.
6. Error acknowledgement is written. Chain B's `OnAcknowledgementPacket` refunds the sender.
7. The NFT transfer is permanently blocked for all users on Chain B, regardless of how many times they retry. [7](#0-6) [2](#0-1)

// ValidateBasic is used for validating the nft transfer.
// NOTE: The addresses formats are not validated as the sender and recipient can have different
// formats defined by their corresponding chains that are not known to IBC.
func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil

func (im IBCModule) OnRecvPacket(
	ctx sdk.Context,
	channelVersion string,
	packet channeltypes.Packet,
	relayer sdk.AccAddress,
) ibcexported.Acknowledgement {
	ack := channeltypes.NewResultAcknowledgement([]byte{byte(1)})

	var data types.NonFungibleTokenPacketData
	if err := types.ModuleCdc.UnmarshalJSON(packet.GetData(), &data); err != nil {
		ack = channeltypes.NewErrorAcknowledgement(fmt.Errorf("cannot unmarshal ICS-721 nft-transfer packet data"))
	}

	// only attempt the application logic if the packet data
	// was successfully decoded
	if ack.Success() {
		if err := im.keeper.OnRecvPacket(ctx, channelVersion, packet, data); err != nil {
			ack = types.NewErrorAcknowledgement(err)
		}
	}
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	sdkCtx.EventManager().EmitEvent(
		sdk.NewEvent(
			types.EventTypePacket,
			sdk.NewAttribute(sdk.AttributeKeyModule, types.ModuleName),
			sdk.NewAttribute(sdk.AttributeKeySender, data.Sender),
			sdk.NewAttribute(types.AttributeKeyReceiver, data.Receiver),
			sdk.NewAttribute(types.AttributeKeyClassID, data.ClassId),
			sdk.NewAttribute(types.AttributeKeyTokenIDs, strings.Join(data.TokenIds, ",")),
			sdk.NewAttribute(types.AttributeKeyAckSuccess, fmt.Sprintf("%t", ack.Success())),
		),
	)

	// NOTE: acknowledgement will be written synchronously during IBC handler execution.
	return ack
}
