### Title
Wrong Weight Used in Spread Health Adjustment for Long-Spot / Short-Perp Basis Positions — (`core/contracts/libraries/RiskHelper.sol`, `core/contracts/Clearinghouse.sol`)

---

### Summary

`IProductEngine.CoreRisk` carries only a single weight field named `longWeight`. Both `_getSpreadWeightX18` and the `existingWeight` calculation in `Clearinghouse.getHealth` unconditionally read `perpCoreRisk.longWeight` for the perp leg, even when that leg is a **short** position whose correct margin weight is `shortWeightInitialX18` (≥ ONE). This causes the spread health adjustment to be computed with the wrong baseline, inflating INITIAL health for long-spot / short-perp basis trades.

---

### Finding Description

**`CoreRisk` struct — only one weight field** [1](#0-0) 

```solidity
struct CoreRisk {
    int128 amount;
    int128 price;
    int128 longWeight;   // ← only the long-side weight is stored
}
```

There is no `shortWeight` field. Whatever `getCoreRisk` returns, the caller can only read the long weight.

**`existingWeight` always uses `longWeight` for both legs** [2](#0-1) 

```solidity
int128 existingWeight = (spotCoreRisk.longWeight +
    perpCoreRisk.longWeight) / 2;
```

For a long-spot / short-perp basis trade the perp leg's correct weight is `shortWeightInitialX18` (≥ ONE). Using `longWeight` (< ONE) instead understates `existingWeight`.

**`_getSpreadWeightX18` also uses `perpCoreRisk.longWeight` for the short-perp leg** [3](#0-2) 

```solidity
if (spotCoreRisk.amount > 0) {
    spreadWeight = ONE - (ONE - perpCoreRisk.longWeight) / 5;
```

When `spotCoreRisk.amount > 0` (long spot), the perp is short. The formula should use the short weight of the perp leg, but it reads `longWeight`.

**Health adjustment formula** [4](#0-3) 

```solidity
health += basisAmount
    .mul(spotCoreRisk.price + perpCoreRisk.price)
    .mul(spreadWeight - existingWeight);
```

The adjustment is `basisAmount × price × (spreadWeight − existingWeight)`. Because `existingWeight` is too low (uses long weight instead of short weight for the perp), `spreadWeight − existingWeight` is too large, producing an inflated positive health contribution.

---

### Impact Explanation

Assume typical INITIAL weights: `longWeightInitialX18 = 0.9`, `shortWeightInitialX18 = 1.1`.

| | Wrong (current) | Correct |
|---|---|---|
| `existingWeight` | `(0.9 + 0.9) / 2 = 0.90` | `(0.9 + 1.1) / 2 = 1.00` |
| `spreadWeight` | `1 − (1 − 0.9)/5 = 0.98` | `1 − (1 − 1.1)/5 = 1.02` → capped at `0.99` |
| `spreadWeight − existingWeight` | **+0.08** | **−0.01** |

The current code adds `+8 %` of the basis notional to health; the correct code would subtract `−1 %`. The error is approximately `(shortWeight − longWeight) / 2 × basisNotional`, which for a $1 M basis position at these weights is **~$100 k of phantom health**. A trader can exploit this to open a long-spot / short-perp basis position and then borrow or trade against the inflated health, leaving the protocol under-collateralised if prices move.

---

### Likelihood Explanation

The path is fully permissionless: any trader can open a long spot position and a short perp position on the same underlying through the normal order flow. `getHealth(INITIAL)` is called on every withdrawal, transfer, and order that checks `_isAboveInitial`. No special privileges are required.

---

### Recommendation

1. Add a second weight field to `CoreRisk` (e.g., `shortWeight`) and populate it from `shortWeightInitialX18` / `shortWeightMaintenanceX18` in `getCoreRisk`.
2. In `Clearinghouse.getHealth`, compute `existingWeight` using the position-appropriate weight for each leg:
   ```solidity
   int128 existingWeight = (
       (spotCoreRisk.amount > 0 ? spotCoreRisk.longWeight : spotCoreRisk.shortWeight) +
       (perpCoreRisk.amount > 0 ? perpCoreRisk.longWeight : perpCoreRisk.shortWeight)
   ) / 2;
   ```
3. Pass the position direction into `_getSpreadWeightX18` (or pass both weights) so it can select the correct weight for the short leg.

---

### Proof of Concept

```solidity
// Fuzz: spotAmount > 0, perpAmount < 0, perpShortWeightX18 > ONE
// 1. Open long spot (spotAmount = X) and short perp (perpAmount = -X) on the same product.
// 2. Call Clearinghouse.getHealth(subaccount, INITIAL).
// 3. Manually compute correct health using shortWeightInitialX18 for the perp leg.
// 4. Assert: getHealth() > correctHealth  ← inflated by ~(shortW - longW)/2 * basisNotional
```

The assertion will hold for any product where `shortWeightInitialX18 > longWeightInitialX18` (all normal products), confirming the invariant is broken.
