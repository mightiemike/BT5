### Title
Unauthenticated `ContractCodeResponse::V1` Accepted Without Responder Identity Check Before `SignedContractCodeResponse` Activation — (File: `chain/client/src/stateless_validation/validate.rs`)

### Summary
`validate_contract_code_response` gates the entire responder-identity check on a single protocol-version comparison. Before `ProtocolFeature::SignedContractCodeResponse` activates (protocol version 85), the only validation performed is a chunk-relevance bounds check. `ContractCodeResponse::V1` carries no `responder` field and no signature, so any peer that can deliver a message with a valid `ChunkProductionKey` can inject arbitrary contract bytes into a chunk validator's pending-witness cache without authentication — an exact analog of the Sudoswap `isPair()` partial-check bug, where only the first 54 bytes (the proxy prefix) were verified while the critical identity parameters were ignored.

### Finding Description

`validate_contract_code_response` in `chain/client/src/stateless_validation/validate.rs` (lines 369–382) gates the entire responder-identity check on a single protocol-version comparison:

```rust
pub fn validate_contract_code_response(...) -> Result<ChunkRelevance, Error> {
    let key = response.chunk_production_key();
    require_relevant!(validate_chunk_relevant(epoch_manager, key, store)?);
    let protocol_version = epoch_manager.get_epoch_protocol_version(&key.epoch_id)?;
    if ProtocolFeature::SignedContractCodeResponse.enabled(protocol_version) {
        validate_witness_contract_code_response_signature(epoch_manager, response)?;
    }
    Ok(ChunkRelevance::Relevant)
}

Before activation the only check is `validate_chunk_relevant`, which verifies height/epoch bounds but says nothing about who sent the response. `ContractCodeResponse::V1` has no `responder` field and no signature:

```rust

    compressed_contracts: CompressedContractCode,

``` [2](#0-1) 

`ContractCodeResponse::verify_signature` returns `false` for V1 (not an error — just `false`), so even if a caller tried to verify, it would silently pass:

```rust
pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
    match self {
        Self::V1(_) => false,   // ← silent non-error for unsigned variant
        Self::V2(v2) => v2.verify_signature(public_key),
    }
}
``` [3](#0-2) 

After validation, `handle_contract_code_response` unconditionally stores whatever contracts were received:

```rust
fn handle_contract_code_response(&self, response: ContractCodeResponse) -> Result<(), Error> {
    // validate_contract_code_response passes with no identity check pre-v85
    let key = response.chunk_production_key().clone();
    let contracts = response.decompress_contracts()?;
    self.partial_witness_tracker.store_accessed_contract_codes(key, contracts)
}
``` [4](#0-3) 

The `ChunkProductionKey` (shard_id, epoch_id, height_created) is public information broadcast with every chunk header. Any peer that can send a `ContractCodeResponse` message with a valid key can overwrite the contract-code cache entry for that chunk without any identity check.

The signed V2 path (`validate_witness_contract_code_response_signature`) correctly verifies that the `responder` is a chunk producer for the shard and that the signature over the inner bytes is valid: [5](#0-4) 

But this path is entirely unreachable for pre-activation epochs.

`SignedContractCodeResponse` is assigned to protocol version 85; `MIN_SUPPORTED_PROTOCOL_VERSION` is 84, so the unauthenticated window is the 84→85 transition and any archival replay of pre-85 epochs: [6](#0-5) [7](#0-6) 

### Impact Explanation

A peer that races or replaces the legitimate chunk producer's response can store incorrect or empty contract bytes in the chunk validator's cache. When the validator assembles the state witness, it uses the stored bytes. If the bytes do not match the `CodeHash` values expected by the witness, the validator fails to endorse the chunk. If enough validators are targeted, the chunk loses its endorsement quorum, stalling block production for that shard. This is a targeted, shard-specific liveness DoS. Because the `ChunkProductionKey` is public and the attack requires only a single well-timed message, it is low-effort for a connected peer.

### Likelihood Explanation

The vulnerability is reachable whenever an epoch's protocol version is below 85. The 84→85 transition window is a real production boundary (`MIN_SUPPORTED_PROTOCOL_VERSION = 84`, `STABLE_PROTOCOL_VERSION = 86`). Archival nodes replaying pre-85 epochs also hit this path. The `ChunkProductionKey` is public. The attack requires sending a message before or after the legitimate response, which is feasible for any peer with a direct connection to the target validator.

### Recommendation

1. **Remove the protocol-version gate from the responder-identity check.** The check should be unconditional: if the response is V1 (unsigned), reject it outright; if V2, verify the signature. The gate was appropriate during the rollout window but should not remain as a permanent bypass.
2. **In `store_accessed_contract_codes`, verify that each received contract's `hash(bytes)` matches the `CodeHash` that was requested**, so that even if an unauthenticated response is accepted, the content cannot be silently substituted.
3. **Add a version-independent minimum check** that rejects `ContractCodeResponse::V1` entirely once `SignedContractCodeResponse` is active, preventing downgrade injection.

### Proof of Concept

1. Network epoch protocol version is 84 (pre-`SignedContractCodeResponse`).
2. Chunk validator V sends `ContractCodeRequest` for `ChunkProductionKey K` to chunk producer P.
3. Attacker A (any peer connected to V) constructs `ContractCodeResponse::V1 { next_chunk: K, compressed_contracts: <malicious/empty> }`.
4. A sends the response to V before P's legitimate response arrives.
5. V calls `validate_contract_code_response`: `validate_chunk_relevant` passes (K is valid); `ProtocolFeature::SignedContractCodeResponse.enabled(84)` is `false`, so the signature check branch is skipped entirely.
6. V stores the malicious/empty contracts via `store_accessed_contract_codes(K, ...)`.
7. V assembles the state witness using the injected contracts; the hash check fails; V does not endorse the chunk.
8. If enough validators are targeted, the chunk loses its endorsement quorum, stalling the shard.

fn validate_witness_contract_code_response_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    response: &ContractCodeResponse,
) -> Result<(), Error> {
    let Some(responder) = response.responder() else {
        return Err(Error::Other(
            "Unsigned contract code response in epoch where signature is required".to_owned(),
        ));
    };
    let key = response.chunk_production_key();
    let chunk_producers =
        epoch_manager.get_epoch_chunk_producers_for_shard(&key.epoch_id, key.shard_id)?;
    if !chunk_producers.contains(responder) {
        return Err(Error::Other(format!(
            "Contract code response responder {responder} is not a chunk producer for shard {} in epoch {:?}",
            key.shard_id, key.epoch_id,
        )));
    }
    let validator = epoch_manager.get_validator_by_account_id(&key.epoch_id, responder)?;
    if !response.verify_signature(validator.public_key()) {
        return Err(Error::Other("Invalid witness contract code response signature".to_owned()));
    }
    Ok(())
