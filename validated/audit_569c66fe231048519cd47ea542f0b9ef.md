### Title
Unchecked `transferFrom` Return Value in `replaceUsdcEWithUsdc` Enables usdcE Drain Without Providing USDC - (File: `core/contracts/ContractOwner.sol`)

---

### Summary

`ContractOwner.replaceUsdcEWithUsdc` calls `IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance)` without checking the return value. If the USDC token at the hardcoded address returns `false` on failure instead of reverting, an attacker can drain usdcE from any user's `DirectDepositV1` contract without providing any USDC in exchange. The rest of the protocol consistently uses `ERC20Helper.safeTransferFrom`, making this a clear deviation from the established safe-transfer pattern.

---

### Finding Description

`replaceUsdcEWithUsdc` is a publicly callable function (no access control beyond a chain ID check) that performs a three-step token swap:

1. Pull USDC from `msg.sender` into `directDepositV1` — **unchecked raw `transferFrom`**
2. Drain usdcE from `directDepositV1` to `ContractOwner` via `DirectDepositV1.withdraw`
3. Send usdcE from `ContractOwner` to `msg.sender` via `safeTransfer` [1](#0-0) 

The critical line is:

```solidity
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
``` [2](#0-1) 

The return value of this call is silently discarded. If the USDC token returns `false` instead of reverting (as non-standard tokens like USDT do), the function continues executing steps 2 and 3 regardless, draining usdcE from the victim's `DirectDepositV1` and transferring it to the attacker.

The protocol defines and uses `ERC20Helper.safeTransferFrom` everywhere else, which performs a low-level call and requires `success && (data.length == 0 || abi.decode(data, (bool)))`: [3](#0-2) 

`ContractOwner` itself imports and uses `ERC20Helper` via `using ERC20Helper for IERC20Base`, and even uses `safeTransfer` on the very next line (line 618) in the same function: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

If the USDC token at `0x2D270e6886d130D724215A266106e6832161EAEd` (chain 57073) returns `false` on a failed transfer rather than reverting:

- An attacker calls `replaceUsdcEWithUsdc(victimSubaccount)` without having approved any USDC.
- Step 1 silently fails — `directDepositV1` receives no USDC.
- Step 2 executes: `DirectDepositV1(directDepositV1).withdraw(usdcE)` drains the victim's usdcE balance to `ContractOwner`.
- Step 3 executes: `safeTransfer` sends the full usdcE balance to the attacker.

The victim's `DirectDepositV1` loses its entire usdcE balance with no compensation. The attacker-controlled state delta is the full `usdcE.balanceOf(directDepositV1)` amount. There is no recovery path — the tokens are gone.

---

### Likelihood Explanation

The function is callable by any unprivileged address on chain 57073 (Ink mainnet). The only precondition is that a target subaccount has a deployed `DirectDepositV1` with a nonzero usdcE balance. The exploitability depends on whether the specific USDC deployment at the hardcoded address returns `false` on failure rather than reverting. Many bridged or wrapped stablecoin implementations on newer chains do not follow the strict revert-on-failure pattern. The inconsistency with the rest of the codebase (which always uses `safeTransferFrom`) makes this a concrete, reachable risk rather than a theoretical one.

---

### Recommendation

Replace the raw `transferFrom` call with the protocol's established `safeTransferFrom` helper:

```solidity
// Before (unsafe):
IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);

// After (safe):
IERC20Base(usdc).safeTransferFrom(msg.sender, directDepositV1, balance);
```

`ContractOwner` already has `using ERC20Helper for IERC20Base` in scope, so `safeTransferFrom` is directly available. [4](#0-3) 

---

### Proof of Concept

1. Victim's subaccount has a deployed `DirectDepositV1` at address `dda` holding 1000 usdcE.
2. Attacker calls `ContractOwner.replaceUsdcEWithUsdc(victimSubaccount)` on chain 57073 with zero USDC approval.
3. `IERC20Base(usdc).transferFrom(attacker, dda, 1000)` returns `false` (no revert) — `dda` receives 0 USDC.
4. `DirectDepositV1(dda).withdraw(usdcE)` transfers 1000 usdcE from `dda` to `ContractOwner`.
5. `IERC20Base(usdcE).safeTransfer(attacker, 1000)` sends 1000 usdcE to the attacker.
6. Victim's `dda` is now empty; attacker gained 1000 usdcE at zero cost. [1](#0-0)

    function replaceUsdcEWithUsdc(bytes32 subaccount) external {
        require(block.chainid == 57073, ERR_UNAUTHORIZED);
        address payable directDepositV1 = directDepositV1Address[subaccount];
        require(directDepositV1 != address(0), "no dda");
        address usdcE = 0xF1815bd50389c46847f0Bda824eC8da914045D14;
        address usdc = 0x2D270e6886d130D724215A266106e6832161EAEd;
        uint256 balance = IERC20Base(usdcE).balanceOf(directDepositV1);
        if (balance > 0) {
            IERC20Base(usdc).transferFrom(msg.sender, directDepositV1, balance);
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(usdcE));
            IERC20Base(usdcE).safeTransfer(msg.sender, balance);
        }
    }

```text
    function safeTransferFrom(
        IERC20Base self,
        address from,
        address to,
        uint256 amount
    ) internal {
        (bool success, bytes memory data) = address(self).call(
            abi.encodeWithSelector(
                IERC20Base.transferFrom.selector,
                from,
                to,
                amount
            )
        );

        require(
            success && (data.length == 0 || abi.decode(data, (bool))),
            ERR_TRANSFER_FAILED
        );
    }
```
