### Title
Snapshot Utilization Rate in `_updateState` Causes Unpredictable Interest Accrual Discrepancy — (File: `core/contracts/SpotEngineState.sol`)

---

### Summary

`SpotEngineState._updateState` computes the borrow interest rate using the **current** (end-of-period) utilization ratio and applies it retroactively over the entire elapsed `dt` window. Because user deposits and borrows can shift utilization between `SpotTick` calls, the effective interest rate applied to the full period is not the time-weighted average rate that actually prevailed. A borrower who deposits a large amount immediately before a `SpotTick` is processed can suppress the utilization ratio used for the entire preceding period, causing all borrowers to underpay interest and all depositors to receive less than they are owed.

---

### Finding Description

`SpotEngineState._updateState` is the sole function that advances the cumulative interest multipliers for every spot product. It is invoked exclusively through `SpotEngine.updateStates`, which is called by the sequencer via the `SpotTick` transaction type in `EndpointTx`. [1](#0-0) 

Inside `_updateState`, the utilization ratio is sampled from the **live** state at the moment the function executes: [2](#0-1) 

That snapshot ratio is then fed into the compound-interest formula to produce a multiplier that is applied over the entire `dt` seconds since the last tick: [3](#0-2) 

The multiplier is then committed to the cumulative state: [4](#0-3) 

Between two consecutive `SpotTick` calls, users may freely deposit, withdraw, or borrow through the normal `Endpoint` transaction pipeline. Each such action changes `totalDepositsNormalized` or `totalBorrowsNormalized`, and therefore changes the utilization ratio that will be read by the next `_updateState` call. The interest for the **entire** preceding window is then computed at that shifted rate, not at the rate that actually prevailed during the window.

The maximum allowed `dt` is 7 days (`require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME)`), so a single tick can cover a substantial period. [5](#0-4) 

---

### Impact Explanation

**Accounting corruption in the SpotEngine.** A borrower who deposits a large amount of collateral immediately before a `SpotTick` lowers the utilization ratio read by `_updateState`. The compound multiplier computed at that lower rate is applied to the full preceding period, so:

- Every borrower in the system underpays interest for that period.
- Every depositor receives less interest than they are owed for that period.

The magnitude scales with the interest rate and `dt`. At the configured maximum `interestLargeCapX18 = ONE` (100 % APR) and a 7-day `dt`, the per-period borrow multiplier at full utilization is approximately `e^(1 × 7/365) ≈ 1.019`. A large deposit that halves utilization before the tick can reduce the multiplier to near 1.000, erasing ~1.9 % of interest for the entire period across all borrowers. For a protocol with significant TVL this is a non-trivial asset delta. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The `SpotTick` is submitted by the sequencer on a predictable schedule. An observer can infer the approximate timing from on-chain history. A borrower with sufficient capital can deposit just before the expected tick, wait for the tick to be included, then withdraw. No privileged access is required; the only entry points needed are the standard `DepositCollateral` and `WithdrawCollateral` transaction types processed by `Endpoint`. The attack is capital-intensive but not capital-destructive: the attacker's deposit earns (reduced) interest during the brief window and is fully recoverable.

---

### Recommendation

1. **Time-weighted utilization**: Record the utilization ratio and timestamp at every state-changing event (deposit, borrow, withdraw). In `_updateState`, compute a time-weighted average utilization over `dt` and derive the interest rate from that average rather than from the current snapshot.
2. **Continuous-rate index**: Alternatively, maintain a per-second interest index that is updated on every balance-changing call (similar to Aave V2's `liquidityIndex`), so that the rate applied always reflects the utilization at the time each second elapsed.
3. **Bounded tick interval**: Enforce a tighter maximum `dt` (e.g., 1 hour instead of 7 days) to limit the worst-case discrepancy until a structural fix is in place.

---

### Proof of Concept

1. At time `T0`, total deposits = 1000, total borrows = 800 → utilization = 80 % → borrow APR ≈ floor + smallCap + largeCap × (0.8 − 0.8)/(1 − 0.8) = floor + smallCap (e.g., 5 %).
2. No `SpotTick` is submitted for 7 days (`dt = 604800 s`).
3. At time `T1 = T0 + 604800 − ε`, the attacker (who is also a borrower) deposits 4000 units, making total deposits = 5000, borrows = 800 → utilization = 16 % → borrow APR ≈ floor + smallCap × 0.16/0.8 ≈ 1.8 %.
4. The sequencer submits `SpotTick`. `_updateState` reads utilization = 16 %, computes `borrowRateMultiplierX18 = (1 + 1.8%/yr / 31536000).pow(604800)` ≈ 1.000344 instead of the correct ≈ 1.000958 (at 5 % APR).
5. The attacker withdraws their 4000 deposit.
6. All borrowers paid ~0.034 % interest for the 7-day period instead of ~0.096 %. The depositors' `cumulativeDepositsMultiplierX18` grew proportionally less, permanently reducing their accrued yield. [7](#0-6)

    function updateStates(uint128 dt) external onlyEndpoint {
        State memory quoteState;
        require(dt < 7 * SECONDS_PER_DAY, ERR_INVALID_TIME);
        for (uint32 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            if (productId == NLP_PRODUCT_ID) {
                continue;
            }
            State memory state = states[productId];
            if (productId == QUOTE_PRODUCT_ID) {
                quoteState = state;
            }
            if (state.totalDepositsNormalized == 0) {
                continue;
            }
            _updateState(productId, state, dt);
            _setState(productId, state);
        }
    }
