### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable, Enabling 64-Byte Merkle Proof Forgery by Any Unprivileged Caller — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` function is still a live, publicly callable NEAR contract method. It carries no caller authentication or role restriction beyond the `#[pause]` macro. The function's own inline documentation acknowledges it can return `true` for an internal Merkle tree node rather than a real transaction hash — the well-known 64-byte Merkle proof forgery. Any unprivileged NEAR account can invoke it with a crafted `tx_id` and receive a fraudulent `true` result. A recipient contract that consumes this result to gate fund releases or cross-chain operations is directly deceivable.

---

### Finding Description

`verify_transaction_inclusion` at line 288 is decorated only with `#[deprecated]` and `#[pause]`. [1](#0-0) 

`#[deprecated]` is a Rust compiler hint — it emits a warning at compile time but imposes zero runtime restriction. The function remains a fully callable public NEAR method. `#[pause]` only blocks calls when the contract is administratively paused; it does not restrict who may call the function when the contract is live.

The function's own warning states: [2](#0-1) 

The verification logic computes the Merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored block header's `merkle_root`: [3](#0-2) 

There is no check that `tx_id` is a leaf-level transaction hash rather than an internal Merkle node. The 64-byte forgery technique (documented at the BitMEX link in the source) allows an attacker to construct a 64-byte blob that is simultaneously a valid internal node of the real Merkle tree and a plausible "transaction ID." Supplying this blob as `tx_id` with a valid sibling path causes `compute_root_from_merkle_proof` to reproduce the real `merkle_root`, and the function returns `true` for a transaction that does not exist.

`verify_transaction_inclusion_v2` was introduced specifically to close this gap by requiring a coinbase Merkle proof that anchors the tree at index 0: [4](#0-3) 

But v1 was never removed or access-gated, so the fix is bypassable by any caller who simply calls v1 directly.

The `ProofArgs` struct accepted by v1 is fully attacker-controlled: [5](#0-4) 

---

### Impact Explanation

A recipient contract that calls `verify_transaction_inclusion` (v1) to authorize a fund release, unlock a cross-chain bridge transfer, or confirm a payment receives a `true` result for a transaction that was never broadcast or mined. The attacker does not need any privileged role, staked collateral, or special account — only the ability to send a NEAR transaction. The corrupted value is the contract's proof result (`bool`), which downstream logic treats as a canonical confirmation of Bitcoin transaction inclusion.

---

### Likelihood Explanation

The function is live on every non-paused deployment. The 64-byte forgery technique is publicly documented and has known tooling. The only prerequisite is knowledge of a real block hash stored in the contract's `headers_pool` (readable via `get_block_hash_by_height` or `get_last_block_header`, both unrestricted view methods). Any downstream contract that has not migrated from v1 to v2 is immediately exploitable. Given that v1 was the only interface before version 0.5.0, legacy integrations are a realistic attack surface.

---

### Recommendation

1. **Remove or hard-restrict v1**: Apply `#[private]` to `verify_transaction_inclusion` so it is only callable by the contract itself (as it is used internally by v2). This eliminates the external attack surface without breaking v2's internal delegation.
2. **Audit downstream consumers**: Identify all contracts that call `verify_transaction_inclusion` directly and migrate them to `verify_transaction_inclusion_v2`.
3. **Emit a runtime guard**: If removal is not immediately feasible, add a `require!(false, "use verify_transaction_inclusion_v2")` at the top of v1's public body to make it unconditionally revert for external callers while the internal `#[allow(deprecated)]` call from v2 is restructured to bypass it.

---

### Proof of Concept

**Setup**: A real Bitcoin block at height H is stored in the contract's `headers_pool`. Its Merkle root is `R`. The block has at least two transactions, so its Merkle tree has internal nodes.

**Attacker steps**:

1. Call `get_block_hash_by_height(H)` → obtain `block_hash`.
2. Fetch the block's raw Merkle tree from a public Bitcoin node. Identify an internal node `N` at depth D whose sibling path, when hashed upward, produces `R`.
3. Construct `ProofArgs`:
   - `tx_id` = `N` (the 64-byte internal node, treated as a 32-byte hash)
   - `tx_block_blockhash` = `block_hash`
   - `tx_index` = index consistent with `N`'s position
   - `merkle_proof` = the sibling hashes from `N` up to (but not including) `R`
   - `confirmations` = 1
4. Call `verify_transaction_inclusion(args)` → returns `true`.
5. Present this `true` result to a downstream contract that gates a fund release on this call.

The downstream contract releases funds for a Bitcoin transaction that was never broadcast. [6](#0-5) [7](#0-6)

        since = "0.5.0",
        note = "Use `verify_transaction_inclusion_v2` instead."

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }
