### Title
AuxPoW Flag / `aux_data` Presence Mismatch Bypasses AuxPoW Validation for Dogecoin Blocks — (`contract/src/dogecoin.rs`)

---

### Summary

In the Dogecoin build of the BTC light client, `submit_block_header` accepts `(Header, Option<AuxData>)`. The code branches on whether `aux_data` is `Some` or `None` to decide which PoW path to execute. However, it never validates that the block header's `BLOCK_VERSION_AUXPOW` flag is consistent with the caller-supplied `aux_data` presence. A block whose `version` has the AuxPoW bit set can be submitted with `aux_data = None`, causing the contract to validate it via the block's own scrypt hash instead of the required parent-block AuxPoW path. This is a direct analog to M-10: a caller-supplied type indicator (`aux_data` presence) is not cross-checked against the actual object's declared type (`BLOCK_VERSION_AUXPOW` in `version`), and the inconsistency is silently accepted.

---

### Finding Description

In `contract/src/dogecoin.rs`, `submit_block_header` dispatches PoW validation based solely on whether the caller provided `aux_data`:

```rust
if let Some(ref aux_data) = aux_data {
    self.check_aux(&block_header, aux_data);
} else {
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        format!("block should have correct pow")
    );
}
``` [1](#0-0) 

`check_aux` enforces that the block header carries the AuxPoW flag:

```rust
require!(
    block_header.version & BLOCK_VERSION_AUXPOW != 0,
    "Aux POW block does not have AuxPoW flag set in version"
);
``` [2](#0-1) 

The inverse check is absent: when `aux_data = None`, there is **no** `require!(block_header.version & BLOCK_VERSION_AUXPOW == 0, ...)`. A block header with `version & 0x100 != 0` (AuxPoW flag set) can therefore be submitted with `aux_data = None` and will be routed to the direct scrypt-hash path, completely skipping `check_aux`.

The `BlockHeader` type alias for Dogecoin is `(Header, Option<AuxData>)`, so the caller fully controls whether `aux_data` is `Some` or `None` on every call to `submit_blocks`: [3](#0-2) 

The public entry point passes the caller-supplied tuple directly: [4](#0-3) 

---

### Impact Explanation

In Dogecoin's AuxPoW scheme, a block whose `version` has `BLOCK_VERSION_AUXPOW` set is **not** required to have its own scrypt hash below the target; only the parent Bitcoin block's hash must satisfy the target. The Dogecoin block's own hash is unconstrained.

When a relayer submits such a block with `aux_data = None`, the contract checks the Dogecoin block's own scrypt hash against the target. Because the block's own hash is unconstrained for AuxPoW blocks, an attacker who can mine at Dogecoin difficulty (far lower than Bitcoin difficulty) can craft a block with `BLOCK_VERSION_AUXPOW` set whose own scrypt hash happens to be below the target, and submit it without any AuxPoW data. The contract accepts and stores this block as canonical, even though it is invalid on the real Dogecoin network (AuxPoW-flagged blocks without AuxPoW data are rejected by Dogecoin nodes). This corrupts the light client's canonical chain state and can cause `verify_transaction_inclusion` to return results anchored to a fabricated chain.

---

### Likelihood Explanation

The `submit_blocks` entry point is gated by `#[trusted_relayer]`. [5](#0-4) 

If the trusted-relayer staking mechanism allows any account to register by staking (with the `RelayerManager` only able to reject applications), then any account with sufficient stake and Dogecoin-level mining power can trigger this path. Dogecoin's PoW difficulty is orders of magnitude lower than Bitcoin's, making direct block mining feasible for a motivated attacker. The mismatch requires no special knowledge beyond the block format.

---

### Recommendation

Add an explicit consistency check in the `else` branch of `submit_block_header` to reject any block whose `version` carries `BLOCK_VERSION_AUXPOW` when no `aux_data` is provided:

```rust
} else {
    require!(
        block_header.version & BLOCK_VERSION_AUXPOW == 0,
        "Block has AuxPoW flag set but no AuxPoW data was provided"
    );
    let pow_hash = block_header.block_hash_pow();
    require!(
        U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
        "block should have correct pow"
    );
}
```

This mirrors the existing check inside `check_aux` and closes the type-consistency gap. [6](#0-5) 

---

### Proof of Concept

1. Attacker registers as a trusted relayer (stakes the required amount).
2. Attacker constructs a `Header` with `version = 0x00010001` (AuxPoW flag `0x100` set, chain-id bits set to a valid value).
3. Attacker mines the header at Dogecoin difficulty: iterates `nonce` until `scrypt(header_bytes) <= target_from_bits(header.bits)`.
4. Attacker calls `submit_blocks` with `headers = vec![(crafted_header, None)]`.
5. `submit_block_header` enters the `else` branch (no `aux_data`), checks only the scrypt hash, and accepts the block.
6. The block is stored in `headers_pool` and promoted to the mainchain if its chainwork exceeds the current tip.
7. Subsequent calls to `verify_transaction_inclusion` operate against a chain that includes this fabricated, AuxPoW-less block.

The root cause — caller-controlled `aux_data` presence is not validated against the block header's own `BLOCK_VERSION_AUXPOW` flag — is the direct analog of M-10's missing cross-check between `tokenType` and the actual token's interface. [7](#0-6)

        &mut self,
        header: (Header, Option<AuxData>),
        skip_pow_verification: bool,
    ) {
        let (block_header, aux_data) = header;

        let prev_block_header = self.get_prev_header(&block_header);
        let current_block_hash = block_header.block_hash();

        if !skip_pow_verification {
            self.check_target(&block_header, &prev_block_header);

            if let Some(ref aux_data) = aux_data {
                self.check_aux(&block_header, aux_data);
            } else {
                let pow_hash = block_header.block_hash_pow();
                // Check if the block hash is less than or equal to the target
                require!(
                    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(block_header.bits),
                    format!("block should have correct pow")
                );
            }
        }

        let (current_block_computed_chain_work, overflow) = prev_block_header
            .chain_work
            .overflowing_add(work_from_bits(block_header.bits));
        require!(!overflow, "Addition of U256 values overflowed");

        let current_header = ExtendedHeader {
            block_header: block_header.clone().into_light(),
            block_hash: current_block_hash,
            chain_work: current_block_computed_chain_work,
            block_height: 1 + prev_block_header.block_height,
        };

        self.submit_block_header_inner(current_header, &prev_block_header);
    }
