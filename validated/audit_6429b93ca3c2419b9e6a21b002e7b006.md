### Title
`get_sortable_validator_online_ratio` Ignores `endorsement_cutoff_threshold`, Causing Incorrect Validator Exemption Ordering — (File: `chain/epoch-manager/src/validator_stats.rs`)

### Summary

`get_sortable_validator_online_ratio` always calls `get_validator_online_ratio(stats, None)`, permanently discarding the `endorsement_cutoff_threshold`. The exemption-from-kickout sort therefore ranks validators by their **raw** endorsement ratio, while the reward calculation and the kickout eligibility check both use the **cutoff-adjusted** (binary 0/1) ratio. A chunk-validator-only validator whose endorsement ratio sits just below the configured threshold is sorted higher than it should be, can be incorrectly exempted from kickout, and continues as an active validator without satisfying the protocol's endorsement requirement.

---

### Finding Description

**Two code paths diverge on the same input.**

**Path 1 – reward calculation** (`finalize_epoch`, `chain/epoch-manager/src/lib.rs` lines 898–913):

```rust
let online_thresholds = ValidatorOnlineThresholds {
    online_min_threshold: epoch_config.online_min_threshold,
    online_max_threshold: epoch_config.online_max_threshold,
    endorsement_cutoff_threshold: Some(
        epoch_config.chunk_validator_only_kickout_threshold,   // ← cutoff applied
    ),
};
self.reward_calculator.calculate_reward(
    validator_block_chunk_stats, ..., online_thresholds, ...
)
```

Inside `calculate_reward`, `get_validator_online_ratio(stats, Some(threshold))` is called. `get_endorsement_ratio` converts the raw endorsement ratio to **0** if `produced/expected < threshold`, or **1** otherwise.

**Path 2 – exemption sort** (`compute_validators_to_reward_and_kickout`, `chain/epoch-manager/src/lib.rs` lines 500–506):

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
```

`get_sortable_validator_online_ratio` is defined as:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);   // ← cutoff NEVER applied
    ...
}
```

The `None` is hard-coded; the function has no parameter for the threshold. The `config` object (which carries `chunk_validator_only_kickout_threshold`) is available in `compute_validators_to_reward_and_kickout` but is never forwarded to the sort key.

**Consequence of the divergence.**

Consider two chunk-validator-only validators with `chunk_validator_only_kickout_threshold = 70` and `validator_max_kickout_stake_perc = 50` (only one of the two can be exempted):

| Validator | Endorsements | Raw ratio | Cutoff-adjusted ratio |
|-----------|-------------|-----------|----------------------|
| A | 69 / 100 | 0.69 | **0** (below threshold) |
| B | 65 / 100 | 0.65 | **0** (below threshold) |

*Current behaviour (raw sort):* A ranks above B → A is exempted → B is kicked out.  
*Correct behaviour (cutoff-adjusted sort):* both rank at 0 → tie broken by stake / account-id, not by how close each validator came to the threshold.

A validator can deliberately maintain an endorsement ratio just below the cutoff (e.g., 69 % when the threshold is 70 %) to rank above peers who are equally non-compliant, thereby exploiting the exemption mechanism to avoid kickout while still failing the protocol's liveness requirement.

---

### Impact Explanation

A chunk-validator-only validator that does not meet `chunk_validator_only_kickout_threshold` can remain in the active validator set across epoch boundaries. The protocol's invariant — that validators below the endorsement threshold are removed — is violated. The validator receives no rewards (the reward path correctly applies the cutoff), but it retains its slot, preventing a compliant replacement from entering. Under sustained network stress, when many validators fall below the threshold, the incorrect sort systematically favours the least-bad non-compliant validators over compliant ones for exemption, compounding the liveness impact.

---

### Likelihood Explanation

`chunk_validator_only_kickout_threshold` is set to a non-zero value in every production epoch config. The bug is therefore always active once stateless validation (chunk-validator-only roles) is live. Any validator can observe the endorsement stats of peers on-chain and tune their own endorsement ratio to sit just below the threshold, making the exploit deliberate and repeatable without any privileged access.

---

### Recommendation

Add an `endorsement_cutoff_threshold: Option<u8>` parameter to `get_sortable_validator_online_ratio` and forward it to `get_validator_online_ratio`:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // ← add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

In `compute_validators_to_reward_and_kickout`, pass `Some(config.chunk_validator_only_kickout_threshold)`:

```rust
.map(|(account, stats)| (
    get_sortable_validator_online_ratio(
        stats,
        Some(config.chunk_validator_only_kickout_threshold),
    ),
    account,
))
```

This makes the exemption sort use the same metric as the reward calculation, eliminating the divergence.

### Proof of Concept

**Setup** (mirrors `test_chunk_validators_with_different_endorsement_ratio`):

- `chunk_validator_only_kickout_threshold = 70`
- `validator_max_kickout_stake_perc = 30` (≈ one of the two chunk-only validators can be exempted)
- Validator A (test2): endorsement 69/100, stake 500
- Validator B (test3): endorsement 60/100, stake 500

**Current code path:**

1. `get_sortable_validator_online_ratio` for A → `get_validator_online_ratio(stats_A, None)` → raw ratio ≈ 0.69
2. `get_sortable_validator_online_ratio` for B → raw ratio ≈ 0.60
3. Sort: A > B → A is exempted
4. Kickout check: B fails `less_than(70)` → B is kicked out; A survives

**Expected code path (with fix):**

1. `get_validator_online_ratio(stats_A, Some(70))` → endorsement below threshold → endorsement component = 0 → A's ratio = 0
2. `get_validator_online_ratio(stats_B, Some(70))` → same → B's ratio = 0
3. Tie broken by stake (equal) then account-id: "test2" < "test3" → test3 is exempted, test2 is kicked out (or vice-versa, deterministically by account-id ordering, not by proximity to the threshold)

The divergence is directly observable by running the existing test `test_chunk_validators_with_different_endorsement_ratio` and checking which validator is exempted versus which one should be exempted under the cutoff-adjusted metric. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
