### Title
`paid_fee_on_l1` Silently Dropped and Hardcoded in Consensus L1Handler Conversion Path — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `paid_fee_on_l1` field of `executable_transaction::L1HandlerTransaction` is accepted and stored in the proposer's internal representation, but is silently discarded when the transaction is serialized into a `ConsensusTransaction` for P2P broadcast. When a validating node reconstructs the `InternalConsensusTransaction::L1Handler` from the received `ConsensusTransaction`, it unconditionally substitutes `Fee(1)` for the actual fee. This is the direct sequencer analog of the external `adjustMaker` bug: a semantically meaningful parameter is accepted in one representation but ignored in the conversion path, with a hardcoded substitute used instead.

### Finding Description

**Proposer path — field is present and used:**

The L1 provider supplies `executable_transaction::L1HandlerTransaction` objects with the actual `paid_fee_on_l1` scraped from the Ethereum event log. The `ProposeTransactionProvider` wraps these directly as `InternalConsensusTransaction::L1Handler(tx)`, preserving the real fee value. [1](#0-0) 

The proposer's blockifier then executes the transaction using the real `paid_fee_on_l1`: [2](#0-1) 

**Conversion to `ConsensusTransaction` — field is dropped:**

When the proposer broadcasts the block proposal, `convert_internal_consensus_tx_to_consensus_tx` strips the `executable_transaction::L1HandlerTransaction` down to only its inner `transaction::L1HandlerTransaction`, discarding both `tx_hash` and `paid_fee_on_l1`: [3](#0-2) 

**Validator path — field is hardcoded:**

When a validator receives the `ConsensusTransaction::L1Handler` and calls `convert_consensus_tx_to_internal_consensus_tx`, it routes through `convert_consensus_l1_handler_to_internal_l1_handler`, which unconditionally substitutes `Fee(1)` regardless of the actual fee paid on L1: [4](#0-3) 

The `TODO(Gilad)` comment confirms this is a known placeholder, not an intentional design.

**Structural type showing the dropped field:** [5](#0-4) 

### Impact Explanation

The blockifier's fee check for L1 handler transactions is:

```rust
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...InsufficientFee...));
}
```

This means:

- **Proposer** executes with the real `paid_fee_on_l1` from the L1 event. If that value is `Fee(0)` (e.g., due to an L1 scraper bug, a zero-fee message, or a future protocol change), the proposer returns a hard `Err`, causing the transaction to be rejected from the block.
- **Validator** always executes with `Fee(1)`, which always passes the `!= 0` check, and would accept the same transaction.

This creates a **proposer/validator execution divergence**: the proposer rejects the transaction (hard error, not a revert), while the validator accepts it and commits state changes. The resulting state roots diverge, breaking consensus.

Additionally, validators never verify the actual fee paid on L1 — they unconditionally trust `Fee(1)`. This means a malicious proposer could include an L1 handler transaction for which no fee was actually paid on L1, and validators would execute it without objection.

The `paid_fee_on_l1` is also serialized into the `CentralL1HandlerTransaction` cende blob by the proposer with the real value, while any future validator-side cende path would use `Fee(1)`, creating a further serialization/accounting mismatch. [6](#0-5) 

### Likelihood Explanation

The divergence is triggered whenever `paid_fee_on_l1 == 0` reaches the blockifier on the proposer side. While the Starknet core contract on L1 requires a non-zero fee for `LogMessageToL2` events, the L1 scraper could set `paid_fee_on_l1 = 0` due to a parsing bug, a future protocol change, or a crafted L1 event. The hardcoded `Fee(1)` on the validator side means validators provide no independent check — they will always accept any L1 handler transaction regardless of what fee was actually paid.

### Recommendation

Pass the actual `paid_fee_on_l1` through the consensus serialization boundary. The `ConsensusTransaction::L1Handler` variant (or the protobuf `L1Handler` message) should carry `paid_fee_on_l1` so that `convert_consensus_l1_handler_to_internal_l1_handler` can use the real value instead of the `Fee(1)` placeholder. Until the TODO is resolved, at minimum the validator should query its own L1 provider for the `paid_fee_on_l1` associated with the transaction hash (which it already validates via `l1_events_provider_client.validate`).

### Proof of Concept

1. L1 scraper creates an `executable_transaction::L1HandlerTransaction` with `paid_fee_on_l1 = Fee(0)` (e.g., due to a parsing bug or a zero-fee L1 message).
2. Proposer's `ProposeTransactionProvider::get_l1_handler_txs` wraps it as `InternalConsensusTransaction::L1Handler(tx)` with `paid_fee_on_l1 = Fee(0)`.
3. Proposer's blockifier calls `execute_raw`; the fee check `if paid_fee == Fee(0)` fires and returns `Err(InsufficientFee)`. The proposer rejects the transaction.
4. If the proposer nonetheless includes the transaction in the proposal (or if the fee becomes 0 after the proposer's execution path), the `ConsensusTransaction::L1Handler(tx.tx)` broadcast drops `paid_fee_on_l1`.
5. Validator's `convert_consensus_l1_handler_to_internal_l1_handler` reconstructs with `Fee(1)`.
6. Validator's blockifier executes: `Fee(1) != Fee(0)` → passes the fee check → commits state changes.
7. Proposer state root ≠ Validator state root → consensus failure.
