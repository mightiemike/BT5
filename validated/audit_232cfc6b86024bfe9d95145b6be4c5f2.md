### Title
`ClearExit` Resets `LastBonusAccrual` to Current Block Time Without Settling Accrued Bonus Rewards - (File: `x/tieredrewards/types/position_state.go`)

---

### Summary

In the `x/tieredrewards` module, when a position owner calls `ClearExit` to cancel a pending exit and remain in their tier, the `ClearExit` function unconditionally resets `LastBonusAccrual` to the current block time. Because `LastBonusAccrual` is the exclusive start timestamp for all future bonus reward calculations, any bonus rewards that accrued between the owner's last claim and the moment `ClearExit` is called are permanently and silently discarded — without ever being paid out.

---

### Finding Description

`PositionState.ClearExit` is defined as:

```go
// x/tieredrewards/types/position_state.go:56-63
func (p *PositionState) ClearExit(blockTime time.Time) {
    p.ExitTriggeredAt = time.Time{}
    p.ExitUnlockAt   = time.Time{}
    if p.IsDelegated() {
        // required so that positions who clear exit after exit lock duration won't have extra bonus accrued
        p.LastBonusAccrual = blockTime   // ← overwrites accrual start unconditionally
    }
}
``` [1](#0-0) 

`LastBonusAccrual` is the field that marks the beginning of the next bonus-reward segment. The lazy bonus calculation in `processEventsAndClaimBonus` / `ComputeSegmentBonus` uses it as the segment start time; any time before `LastBonusAccrual` is invisible to the reward engine. [2](#0-1) 

The lifecycle that triggers the loss:

1. Owner creates a position at `T0`; `LastBonusAccrual = T0`.
2. Owner calls `TriggerExit` at `T1`; `ExitTriggeredAt = T1`, `ExitUnlockAt = T1 + ExitDuration`.
3. Owner has not claimed bonus rewards since `T0` (or since their last claim `Tlast`).
4. Owner calls `ClearExit` at `T3` (before or after `ExitUnlockAt`).
5. `ClearExit` sets `LastBonusAccrual = T3`.
6. The entire bonus reward segment `[Tlast, T3)` is now unreachable — the reward engine will never compute it.

The code comment acknowledges only the sub-case where `T3 > ExitUnlockAt` ("positions who clear exit after exit lock duration"), but the reset is applied unconditionally: it fires even when `T3 < ExitUnlockAt`, and it always discards the full `[Tlast, T3)` window rather than only the post-unlock portion `[ExitUnlockAt, T3)`. [3](#0-2) 

The `PositionState.Validate()` invariant enforces that `LastBonusAccrual` must be non-zero whenever a position is delegated, so the field is always live and always used as the reward-segment start. [2](#0-1) 

---

### Impact Explanation

Every position owner who calls `ClearExit` loses all bonus rewards that accrued since their last `ClaimTierRewards` call. The loss scales with:

- **Time since last claim**: a user who has not claimed for weeks or months before calling `ClearExit` loses the entire accumulated bonus.
- **Position size**: larger delegations accrue proportionally larger bonuses per unit time.
- **Tier reward rate**: higher-tier positions lose more per unit time.

The lost tokens remain in the `RewardsPool` module account and are never distributed to the rightful owner. This is a direct, permanent financial loss for the position owner with no recovery path.

---

### Likelihood Explanation

`ClearExit` is a standard, documented user action — it is the normal way to "stay in a tier" after second-guessing an exit. Any position owner who:

1. Has an active position with unclaimed bonus rewards, and
2. Calls `TriggerExit` followed by `ClearExit`

will silently lose those rewards. No special timing, privileged access, or adversarial setup is required. The trigger is a normal, unprivileged `MsgClearExit` transaction signed by the position owner themselves.

---

### Recommendation

Before resetting `LastBonusAccrual`, the keeper should settle all accrued bonus rewards up to the appropriate checkpoint. Two equivalent approaches:

**Option A — Claim rewards before clearing:**
In the message-server handler for `ClearExit`, call `processEventsAndClaimBonus` (or the equivalent internal claim helper) for the position before invoking `ClearExit`. This pays out `[Tlast, now)` and then the reset to `blockTime` is a no-op loss.

**Option B — Partial reset to `ExitUnlockAt`:**
If the intent is only to forfeit rewards earned after the exit lock expired (the "extra bonus" the comment refers to), set `LastBonusAccrual = ExitUnlockAt` instead of `blockTime` when `T3 > ExitUnlockAt`. For the case `T3 <= ExitUnlockAt`, do not reset `LastBonusAccrual` at all.

```go
func (p *PositionState) ClearExit(blockTime time.Time) {
    p.ExitTriggeredAt = time.Time{}
    p.ExitUnlockAt   = time.Time{}
    // Do NOT reset LastBonusAccrual here; caller must settle rewards first.
}
```

### Proof of Concept

```
T0  : Alice creates a 1,000,000 CRO position in Tier-2 (10% bonus APY).
      LastBonusAccrual = T0.

T0+90d : Alice calls TriggerExit.
         ExitTriggeredAt = T0+90d, ExitUnlockAt = T0+90d+ExitDuration.
         Alice has NOT called ClaimTierRewards since T0.
         Accrued bonus ≈ 1,000,000 × 10% × (90/365) ≈ 24,657 CRO.

T0+95d : Alice changes her mind and calls ClearExit.
         ClearExit sets LastBonusAccrual = T0+95d.
         The segment [T0, T0+95d) is now invisible to the reward engine.
         Alice's 24,657 CRO of accrued bonus is permanently lost.

T0+95d+1s : Alice calls ClaimTierRewards.
            Reward engine computes segment [T0+95d, T0+95d+1s) ≈ 0 CRO.
            Alice receives nothing for 95 days of staking.
```

The root cause is the unconditional `p.LastBonusAccrual = blockTime` at [4](#0-3)  inside `ClearExit`, which is structurally identical to the TempleGold bug where `t = 0` was set unconditionally when `_newBalance == 0`, resetting the accrual timestamp and erasing accumulated weight.
