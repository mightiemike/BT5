### Title
Burn-on-Transfer Token Two-Hop Deposit Inflates SpotEngine Balances, Breaking Withdrawal Solvency — (`core/contracts/EndpointStorage.sol`)

---

### Summary

`handleDepositTransfer` in `EndpointStorage.sol` performs a two-hop token transfer (user → Endpoint → Clearinghouse) using the caller-supplied `amount` for both hops. For burn-on-transfer tokens, the Clearinghouse receives less than `amount`, but the slow mode transaction queued in `depositCollateralWithReferral` encodes the original `amount`. When the sequencer processes the slow mode tx, `Clearinghouse.depositCollateral` credits the user's SpotEngine balance with the full `amount`, creating an inflated balance that exceeds actual token holdings. Subsequent withdrawals will fail or consume other users' funds.

---

### Finding Description

The deposit flow in Nado is:

**Step 1 — `Endpoint.depositCollateralWithReferral`** calls `handleDepositTransfer` with the user-supplied `amount`, then encodes that same `amount` into a slow mode transaction: [1](#0-0) 

**Step 2 — `EndpointStorage.handleDepositTransfer`** performs two sequential transfers, both using the original `amount`: [2](#0-1) 

- `safeTransferFrom(token, from, amount)` → Endpoint receives `amount − burn₁`
- `safeTransferTo(token, address(clearinghouse), amount)` → Endpoint attempts to forward the full `amount`, but only holds `amount − burn₁`

If the Endpoint contract has accumulated prior token balance (e.g., from slow-mode fees charged in the same token, or from other in-flight deposits), the second transfer succeeds by consuming those reserves. The Clearinghouse then receives `amount − burn₂` (a second burn on the outbound hop), while the slow mode tx still records `amount`.

**Step 3 — `Clearinghouse.depositCollateral`** credits the user using `txn.amount` — the original, unburned value — not the actual received amount: [3](#0-2) 

**Step 4 — `Clearinghouse.withdrawCollateral`** later attempts to transfer the full credited `amount` out: [4](#0-3) 

`handleWithdrawTransfer` calls `token.safeTransfer(withdrawPool, uint256(amount))` using the SpotEngine-recorded amount, which exceeds actual holdings: [5](#0-4) 

---

### Impact Explanation

The SpotEngine balance for a depositor of a burn-on-transfer token is inflated by `burn₁ + burn₂` (two hops). The Clearinghouse's actual token reserve is correspondingly short. As the deficit accumulates across multiple deposits, the Clearinghouse becomes insolvent for that token: later withdrawals revert (DoS for honest users) or, if the Clearinghouse holds surplus from other sources, silently consume other users' collateral. The `assertUtilization` check on withdrawal only guards against borrow/deposit ratio, not against the raw token shortfall caused by burns. [6](#0-5) 

---

### Likelihood Explanation

Likelihood is **low-medium**. It requires a burn-on-transfer ERC-20 to be listed as a supported collateral token in the SpotEngine. If any such token is ever added (e.g., a rebasing or fee-on-transfer stablecoin), every deposit by any unprivileged user through the standard `depositCollateral` / `depositCollateralWithReferral` entrypoints triggers the accounting inflation. No special role or privileged access is needed beyond a valid deposit call.

---

### Recommendation

In `handleDepositTransfer`, measure the Clearinghouse's actual token balance before and after the transfer chain and use the delta — not the caller-supplied `amount` — as the value encoded into the slow mode `DepositCollateral` transaction:

```solidity
function handleDepositTransfer(
    IERC20Base token,
    address from,
    uint256 amount
) internal returns (uint256 actualReceived) {
    require(address(token) != address(0), ERR_INVALID_PRODUCT);
    uint256 before = token.balanceOf(address(clearinghouse));
    safeTransferFrom(token, from, amount);
    safeTransferTo(token, address(clearinghouse), amount);
    actualReceived = token.balanceOf(address(clearinghouse)) - before;
}
```

Then pass `actualReceived` (cast to `uint128`) into the `DepositCollateral` struct instead of `amount`. Alternatively, document that burn-on-transfer tokens are explicitly unsupported and enforce this at token listing time.

---

### Proof of Concept

1. A burn-on-transfer token `BurnToken` (1% burn per transfer) is listed as a supported collateral (`productId = X`).
2. The Endpoint holds a prior `BurnToken` balance of `B ≥ amount` (e.g., from accumulated slow-mode fees in the same token).
3. Alice calls `depositCollateralWithReferral(aliceSubaccount, X, 1000e18, "")`.
4. `handleDepositTransfer` executes:
   - `safeTransferFrom(BurnToken, Alice, 1000e18)` → Endpoint receives `990e18` (1% burn).
   - `safeTransferTo(BurnToken, clearinghouse, 1000e18)` → succeeds (Endpoint has `B + 990e18`); Clearinghouse receives `990e18` (another 1% burn).
5. Slow mode tx is queued with `amount = 1000e18`.
6. Sequencer processes the tx; `Clearinghouse.depositCollateral` credits Alice's SpotEngine balance with `1000e18` (normalized).
7. Alice's SpotEngine balance: `1000e18`. Clearinghouse actual holdings: `990e18`. Deficit: `10e18` per deposit.
8. After N such deposits, the Clearinghouse is short `N × 10e18` tokens.
9. Bob (honest depositor) calls `withdrawCollateral`; `token.safeTransfer(withdrawPool, amount)` reverts because the Clearinghouse lacks sufficient `BurnToken` balance. [2](#0-1) [7](#0-6) [8](#0-7)
