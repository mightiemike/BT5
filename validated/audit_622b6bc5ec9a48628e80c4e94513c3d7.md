### Title
`minted_amount` Understated When `num_validators == 0` Due to Incorrect Accounting in `calculate_reward` - (File: chain/epoch-manager/src/reward_calculator.rs)

### Summary

`RewardCalculator::calculate_reward` inserts `epoch_protocol_treasury` into the per-account reward map (`res`) and then returns `Balance::ZERO` as `minted_amount` when `num_validators == 0`. The treasury reward is subsequently credited to the treasury account via `update_validator_accounts`, but `minted_amount = ZERO` causes `EpochInfo::minted_amount` — and therefore the block header's `total_supply` — to be understated by exactly `epoch_protocol_treasury`.

### Finding Description

In `calculate_reward`, the early-exit branch for the no-validator case is:

```rust
res.insert(self.protocol_treasury_account.clone(), epoch_protocol_treasury);
if num_validators == 0 {
    return (res, Balance::ZERO);   // ← minted_amount is ZERO
}
``` [1](#0-0) 

`res` already contains `{protocol_treasury_account → epoch_protocol_treasury}`. This map is stored in `EpochInfo::validator_reward` and is applied to on-chain balances by `update_validator_accounts`:

```rust
account.set_amount(account.amount().checked_add(treasury_reward)...);
``` [2](#0-1) 

So the treasury account's balance **does** increase by `epoch_protocol_treasury` — tokens are genuinely minted — but the second return value (`minted_amount`) is `Balance::ZERO`.

`minted_amount` is stored verbatim in `EpochInfo`:

```rust
pub minted_amount: Balance,
``` [3](#0-2) 

and is retrieved at the epoch boundary to verify `total_supply`:

```rust
let minted_amount = if self.epoch_manager.is_next_block_epoch_start(&prev_hash)? {
    Some(self.epoch_manager.get_epoch_info(header.next_epoch_id())?.minted_amount())
} else { None };
``` [4](#0-3) 

`verify_total_supply_checked` then computes:

```rust
let Some(new_total_supply) = prev_total_supply
    .checked_add(minted_amount.unwrap_or(Balance::ZERO))?
    .checked_sub(balance_burnt)
``` [5](#0-4) 

When `minted_amount` is `ZERO` instead of `epoch_protocol_treasury`, the block producer sets `total_supply = prev_total_supply − balance_burnt`, and the verifier accepts it (both sides use the same wrong `minted_amount`). The actual token supply in state is `prev_total_supply + epoch_protocol_treasury − balance_burnt`. The divergence is `epoch_protocol_treasury` yoctoNEAR per affected epoch.

The docstring for `calculate_reward` explicitly states the second return value must include the treasury:

> Returns map of validators with their rewards and **amount of newly minted tokens including to protocol's treasury**. [6](#0-5) 

The `num_validators == 0` branch violates this contract.

### Impact Explanation

`total_supply` in every subsequent block header is permanently understated by `epoch_protocol_treasury`. Because future `epoch_total_reward` is proportional to `total_supply`, the error compounds across epochs. Any external tool, bridge, or on-chain contract that reads `total_supply` from block headers receives a value that diverges from the actual circulating supply. The `verify_total_supply_checked` guard does not catch the error because both the producer and the verifier use the same wrong `minted_amount` from `EpochInfo`.

### Likelihood Explanation

`num_validators == 0` requires `validator_block_chunk_stats` to be empty at epoch finalization — i.e., every validator in the epoch was kicked out or produced no blocks/chunks. This is an edge case on mainnet but is reachable on testnets, new networks, or after a catastrophic validator outage. No privileged action is required; the condition arises from normal protocol epoch-boundary processing.

### Recommendation

Change the early return to include the treasury reward in `minted_amount`:

```rust
if num_validators == 0 {
-   return (res, Balance::ZERO);
+   return (res, epoch_protocol_treasury);
}
``` [7](#0-6) 

### Proof of Concept

Concrete values:

| Variable | Value |
|---|---|
| `total_supply` | 1 000 000 yN |
| `max_inflation_rate` | 1/100 |
| `epoch_duration` | 1 year in ns |
| `epoch_total_reward` | 10 000 yN |
| `protocol_reward_rate` | 1/10 |
| `epoch_protocol_treasury` | 1 000 yN |
| `num_validators` | **0** |
| `balance_burnt` | 50 yN |

`calculate_reward` returns `({treasury → 1000}, 0)`.

- Treasury account balance: `+1000 yN` (applied via `update_validator_accounts`)
- `EpochInfo::minted_amount`: `0`
- Block header `total_supply` set by producer: `1 000 000 + 0 − 50 = 999 950 yN`
- Actual circulating supply: `1 000 000 + 1 000 − 50 = 1 000 950 yN`
- **Divergence: 1 000 yN understated**, compounding in every subsequent epoch's reward calculation.
