Looking at the exact code paths involved:

Based on my investigation, the divergence described in the question is **real and confirmed by the code itself**, with an explicit acknowledgment in a comment. Here is my analysis:

---

### Title
`TryFrom<ReceiptView>` Loses `ReceiptEnum::ActionV2` Discriminant on `refund_to: None` Roundtrip, Breaking `outgoing_receipts_root` Proof Verification — (`core/primitives/src/views.rs`)

---

### Summary

`ReceiptEnum::ActionV2` (discriminant `5`) with `refund_to: None` is silently downgraded to `ReceiptEnum::Action` (discriminant `0`) after a `Receipt → ReceiptView → Receipt` roundtrip via `TryFrom<ReceiptView>`. The code itself acknowledges this at the exact divergence point.

---

### Finding Description

**Enum layout:**

`ReceiptEnum` uses `#[borsh(use_discriminant = true)]` and `#[repr(u8)]`: [1](#0-0) 

- `Action(ActionReceipt) = 0` — no `refund_to` field
- `ActionV2(ActionReceiptV2) = 5` — has `refund_to: Option<AccountId>`

**Forward path (`Receipt → ReceiptView`):**

`From<ReceiptEnum> for VersionedReceiptEnum` collapses both `Action` and `ActionV2` into `VersionedReceiptEnum::Action`: [2](#0-1) 

`ReceiptEnumView::from_action_receipt` then sets `refund_to: action_receipt.refund_to().clone()`, which is `None` for an `ActionV2` with no refund destination: [3](#0-2) 

**Backward path (`ReceiptView → Receipt` via `TryFrom`):**

The conversion branches solely on `refund_to.is_some()`: [4](#0-3) 

The code comment at lines 2518–2521 explicitly states:
> "Note that this is not consistent with how `new_receipts` are created by the runtime - there we always create ActionReceiptV2. ActionReceiptV2 without refund_to becomes V1 after a roundtrip through views. This will be fixed with #14709."

**Concrete byte divergence:**

| Stage | Borsh discriminant byte | `refund_to` encoding |
|---|---|---|
| Stored in `DBCol::OutgoingReceipts` | `0x05` (ActionV2) | `0x00` (None) |
| After `TryFrom<ReceiptView>` roundtrip | `0x00` (Action) | absent |

The two byte vectors are **not equal**.

---

### Impact Explanation

The `outgoing_receipts_root` in the chunk header is computed from the Borsh bytes of the actual stored `Receipt` objects (discriminant `5`): [5](#0-4) 

Any RPC client or light client that:
1. Fetches a receipt via `EXPERIMENTAL_receipt` (gets `ReceiptView`)
2. Converts it back via `TryFrom<ReceiptView>` to get a `Receipt`
3. Re-serializes and attempts to verify against `outgoing_receipts_root`

...will compute the wrong hash and fail verification. The `verify_against_receipt_root` call used in chunk validation: [6](#0-5) 

...would reject the reconstructed receipt because its Borsh bytes produce a different merkle leaf.

**The consensus layer itself is not affected** — it reads `Receipt` objects directly from `DBCol::OutgoingReceipts` without going through `ReceiptView`. The divergence is confined to the RPC/view roundtrip path.

---

### Likelihood Explanation

- The runtime **always** creates `ActionReceiptV2` for new receipts (as the comment states), so every action receipt produced after the `ActionV2` feature activation has discriminant `5`.
- `refund_to: None` is the common case (most receipts have no explicit refund destination).
- Any unprivileged user submitting a transaction triggers this path.
- The `EXPERIMENTAL_receipt` RPC endpoint is publicly accessible.

---

### Recommendation

In `TryFrom<ReceiptView> for Receipt`, preserve the `ActionV2` variant whenever the source was an action receipt, regardless of whether `refund_to` is `Some` or `None`. The `ReceiptEnumView` should carry an explicit version tag (e.g., `is_v2: bool`) or the `TryFrom` should default to `ActionV2` for all action receipts produced after the feature activation, matching the runtime's behavior. This is tracked as issue #14709.

---

### Proof of Concept

```rust
use near_primitives::receipt::{Receipt, ReceiptV0, ReceiptEnum, ActionReceiptV2};
use near_primitives::views::{ReceiptView, Receipt as _};
use borsh::BorshSerialize;

let original = Receipt::V0(ReceiptV0 {
    predecessor_id: "alice.near".parse().unwrap(),
    receiver_id: "bob.near".parse().unwrap(),
    receipt_id: CryptoHash::default(),
    receipt: ReceiptEnum::ActionV2(ActionReceiptV2 {
        signer_id: "alice.near".parse().unwrap(),
        refund_to: None,   // <-- the critical case
        signer_public_key: PublicKey::empty(KeyType::ED25519),
        gas_price: 0,
        output_data_receivers: vec![],
        input_data_ids: vec![],
        actions: vec![],
    }),
});

let original_bytes = borsh::to_vec(&original).unwrap();
// original_bytes[N] == 5  (ActionV2 discriminant)

let view: ReceiptView = original.clone().into();
let reconstructed: Receipt = Receipt::try_from(view).unwrap();
let reconstructed_bytes = borsh::to_vec(&reconstructed).unwrap();
// reconstructed_bytes[N] == 0  (Action discriminant)

assert_eq!(original_bytes, reconstructed_bytes);  // FAILS
```

The first discriminant byte differs (`5` vs `0`), and the `refund_to: None` option byte (`0x00`) is present in the original but absent in the reconstructed form, producing divergent Borsh representations and a different merkle leaf hash.
