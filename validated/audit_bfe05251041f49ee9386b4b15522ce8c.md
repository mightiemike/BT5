### Title
Gas-Key Host Functions Use Raw Wire Length Instead of On-Trie Identifier Length for ML-DSA-65 Fee Calculation, Causing Fee-Parity Violation and Potential Gas Exhaustion — (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

---

### Summary

The gas-key host functions (`promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, `promise_batch_action_transfer_to_gas_key`) pass the raw borsh wire length of the public key (`public_key_len as usize`) to the fee-calculation helpers `gas_key_add_key_exec_fee` and `gas_key_transfer_exec_fee`. For ML-DSA-65 keys the wire length is 1953 bytes, but the on-trie identifier is a 33-byte SHA3-256 hash. The transaction path correctly uses `public_key.trie_id_len()` (33). The host-function path uses the wrong value (1953), producing a per-nonce exec-fee overcharge of ~1920 × `gas_key_byte` gas. With enough nonces this exhausts the gas budget, making ML-DSA-65 gas-key creation via host functions impossible.

---

### Finding Description

`PublicKey::trie_id_len()` was introduced precisely because ML-DSA-65 access keys are stored in the trie as a 33-byte SHA3-256 hash, not as the 1953-byte borsh-encoded full key. The documentation in `post_quantum_signatures.md` explicitly states: *"Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes."*

The transaction-path exec-fee calculation in `runtime/runtime/src/config.rs` correctly applies this:

```rust
// runtime/runtime/src/config.rs  lines 389-394
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← correct: 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
``` [1](#0-0) 

The host-function path in both the interpreter and Wasmtime backends does not apply the same conversion:

```rust
// runtime/near-vm-runner/src/logic/logic.rs  lines 3155-3160
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← wrong: 1953 for ML-DSA-65
    num_nonces,
);
``` [2](#0-1) 

The same error appears in `promise_batch_action_add_gas_key_with_function_call`: [3](#0-2) 

And in the Wasmtime backend for both variants: [4](#0-3) [5](#0-4) 

The same divergence exists for `gas_key_transfer_exec_fee`. The transaction path:

```rust
// runtime/runtime/src/config.rs  lines 347-350
TransferToGasKey(action) => {
    gas_key_transfer_exec_fee(fees, receiver_id.len(), action.public_key.trie_id_len())
        .total()
}
``` [6](#0-5) 

The host-function path:

```rust
// runtime/near-vm-runner/src/logic/logic.rs  lines 3092-3096
let exec = gas_key_transfer_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← wrong: 1953 for ML-DSA-65
);
``` [7](#0-6) 

Inside `gas_key_add_key_exec_fee`, the divergent value propagates directly into the nonce trie-key length:

```rust
// core/parameters/src/cost.rs  lines 888-889
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
``` [8](#0-7) 

With `public_key_len = 1953` the computed `nonce_key_len` is 1920 bytes larger than the actual on-trie key. The per-byte exec fee is then multiplied by this inflated length and by `num_nonces`, producing a fee that can be orders of magnitude larger than the correct value.

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys, for which `len() == trie_id_len()`, so the divergence is not caught: [9](#0-8) 

---

### Impact Explanation

**High.** Both `GasKeys` and `PostQuantumSignatures` are at protocol version 85, which is below the current stable `STABLE_PROTOCOL_VERSION = 86`, so both features are active on mainnet. [10](#0-9) [11](#0-10) 

Any unprivileged contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 key and a non-trivial `num_nonces` will be charged ~1920 × `gas_key_byte` × `num_nonces` extra exec gas. Because the exec fee is prepaid from the calling receipt's gas budget, a sufficiently large `num_nonces` exhausts the budget and causes the action to fail with `GasExceeded`, making ML-DSA-65 gas-key creation via host functions a practical DoS. The same overcharge applies to every `TransferToGasKey` / `WithdrawFromGasKey` host-function call with an ML-DSA-65 key.

The fee-parity invariant documented in `post_quantum_signatures.md` is broken: identical operations cost different amounts depending on whether they are submitted as a transaction or emitted by a contract. [12](#0-11) 

---

### Likelihood Explanation

**Low.** ML-DSA-65 gas keys are a brand-new feature. Contracts that emit gas-key host-function calls with ML-DSA-65 keys do not yet exist in the wild. However, the code path is live on stable protocol version 86 and will be hit by any contract that attempts to use the feature.

---

### Recommendation

In each host-function call site, decode the public key first and use `trie_id_len()` for the exec-fee argument, mirroring the transaction path:

```rust
// Before fee calculation, decode the key:
let decoded_pk = public_key.decode()?;

// Then pass trie_id_len() instead of public_key_len:
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    decoded_pk.trie_id_len(),   // ← correct
    num_nonces,
);
// ... use decoded_pk when appending the action too
```

Apply the same fix to `gas_key_transfer_exec_fee` call sites in both `logic.rs` and `wasmtime_runner/logic.rs`. Add a fee-parity test that exercises ML-DSA-65 keys alongside the existing ED25519 test in `test_gas_key_fee_parity`.

---

### Proof of Concept

1. Enable `PostQuantumSignatures` + `GasKeys` (protocol version ≥ 85, stable on mainnet).
2. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` with an ML-DSA-65 public key (borsh-encoded: 1953 bytes) and `num_nonces = N`.
3. Observe that the exec gas charged is ~1920 × `gas_key_byte` × N larger than the gas charged for an equivalent `AddKey` transaction with the same ML-DSA-65 key and `num_nonces = N`.
4. With `N` large enough, the host-function call exhausts the receipt gas budget and fails, while the equivalent transaction succeeds.

The exact divergent values are:
- Wire length passed by host function: **1953** (`1 + ML_DSA_65_PUBLIC_KEY_LENGTH`)
- On-trie identifier length used by transaction path: **33** (`1 + ML_DSA_65_HASH_LENGTH`)
- Overcharge per nonce: **1920 × `gas_key_byte`** exec gas [13](#0-12) [14](#0-13)

### 5. Storage usage and fee plumbing

The storage-stake calculation
(`runtime/runtime/src/access_keys.rs::access_key_storage_usage`) and the
gas-key fee helpers (`gas_key_*_fee` in `runtime/runtime/src/config.rs`) use
`PublicKey::trie_id_len()` rather than `PublicKey::len()`:

- `len()` reports the borsh-encoded length (33 / 65 / 1953 across the
  three `PublicKey` variants).
- `trie_id_len()` reports the on-trie length (33 / 65 / **33**).

The two diverge only for `PublicKey::MLDSA65`. Every storage-stake and
trie-byte-priced fee path was updated to call `trie_id_len()`.
