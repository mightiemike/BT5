### Title
`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` Misclassifies `AllResources` Transactions as `L1Gas`, Causing Valid Transactions to Be Rejected During Consensus Propagation — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf-to-Rust conversion for `ValidResourceBounds` classifies a transaction as `ValidResourceBounds::L1Gas` whenever `l2_gas` and `l1_data_gas` are both zero. This is the wrong discriminant for the `ConsensusTransaction` path: a valid `RpcInvokeTransactionV3` submitted via the RPC gateway with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` is accepted and hashed by the gateway using the `AllResources` branch (3 resource felts), but when the proposer broadcasts it to peers via P2P, the receiving peer's protobuf converter produces `ValidResourceBounds::L1Gas`, which then fails the mandatory `InvokeTransactionV3 → RpcInvokeTransactionV3` conversion, causing the peer to reject the transaction (and the entire proposal) with `DEPRECATED_RESOURCE_BOUNDS_ERROR`.

---

### Finding Description

**Root cause — wrong discriminant in protobuf conversion**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` uses the *numeric value* of the fields to decide the variant:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
``` [1](#0-0) 

This is inconsistent with the JSON/storage deserialization path (`TryFrom<DeprecatedResourceBoundsMapping>`), which uses the *presence* of the `L1DataGas` key — not its value — to decide the variant:

```rust
match resource_bounds_mapping.0.get(&Resource::L1DataGas) {
    Some(data_bounds) => Ok(Self::AllResources(...)),
    None => if l2_bounds.is_zero() { Ok(Self::L1Gas(...)) } else { Err(...) }
}
``` [2](#0-1) 

**How the `ConsensusTransaction` path is broken**

When a peer receives a `ConsensusTransaction` over P2P, the conversion chain is:

```
protobuf::InvokeV3WithProof
  → InvokeTransactionV3          (uses TryFrom<protobuf::ResourceBounds> for ValidResourceBounds)
  → RpcInvokeTransactionV3       (requires ValidResourceBounds::AllResources)
```

The intermediate step is explicit in the code, and the developers already noted the failure mode:

```rust
// This conversion can fail only if the resource_bounds are not AllResources.
Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
``` [3](#0-2) 

The comment treats this as a "deprecated resource bounds" case, but a perfectly valid post-0.13.3 `AllResourceBounds` transaction with `l2_gas = 0` and `l1_data_gas = 0` triggers the same failure path.

**The `RpcInvokeTransactionV3 → InvokeTransactionV3` conversion enforces `AllResources`**

```rust
impl TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3 {
    fn try_from(value: InvokeTransactionV3) -> Result<Self, Self::Error> {
        Ok(Self {
            resource_bounds: match value.resource_bounds {
                ValidResourceBounds::AllResources(bounds) => bounds,
                _ => return Err(StarknetApiError::OutOfRange { ... }),
            },
            ...
        })
    }
}
``` [4](#0-3) 

So any `InvokeTransactionV3` that arrives with `ValidResourceBounds::L1Gas` (produced by the protobuf converter) will fail this conversion.

**Hash domain divergence (the analog of the external bug)**

`get_tip_resource_bounds_hash` hashes a *different number of resource felts* depending on the variant:

- `L1Gas` → `[tip, l1_gas, l2_gas(=0)]` — **2 resource felts**
- `AllResources` with `l2_gas=0, l1_data_gas=0` → `[tip, l1_gas, 0, 0]` — **3 resource felts**

```rust
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
    }
});
``` [5](#0-4) 

The gateway computes the hash using `AllResources` (3 felts). If a peer were to recompute the hash after the misclassification, it would use `L1Gas` (2 felts) and produce a different hash — a direct analog of the external bug's "count check passes but actual transfer uses empty array."

**Proposal rejection path**

When the conversion fails, the validate-proposal handler treats the entire proposal as failed:

```rust
Err(e) => {
    return HandledProposalPart::Failed(format!(
        "Failed to convert transactions. Stopping the build of the current \
         proposal. {e:?}"
    ));
}
``` [6](#0-5) 

---

### Impact Explanation

A single valid `RpcInvokeTransactionV3` with `AllResourceBounds { l1_gas: non-zero, l2_gas: 0, l1_data_gas: 0 }` included in a block proposal causes every validating peer to reject the entire proposal. The gateway accepts the transaction (correct `AllResources` hash), but the P2P consensus layer rejects it (misclassified as `L1Gas`). This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

and

> **High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.**

---

### Likelihood Explanation

The gateway's stateless validator explicitly accepts transactions with only `l1_gas` set and `l2_gas = l1_data_gas = 0`: [7](#0-6) 

Any unprivileged user can submit such a transaction. The condition `l2_gas = 0, l1_data_gas = 0` is the natural state for a user who only cares about L1 gas (pre-0.13.3 style bounds expressed in the post-0.13.3 `AllResourceBounds` format). The trigger requires no special privileges and no malicious intent.

---

### Recommendation

Replace the value-based discriminant in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` with a presence-based one, consistent with `TryFrom<DeprecatedResourceBoundsMapping>`. Specifically, for the `ConsensusTransaction` / `MempoolTransaction` paths (which always carry post-0.13.3 transactions), use `TryFrom<protobuf::ResourceBounds> for AllResourceBounds` directly — which already exists and never misclassifies:

```rust
impl TryFrom<protobuf::ResourceBounds> for AllResourceBounds {
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        Ok(Self {
            l1_gas: value.l1_gas.ok_or(missing("ResourceBounds::l1_gas"))?.try_into()?,
            l2_gas: value.l2_gas.ok_or(missing("ResourceBounds::l2_gas"))?.try_into()?,
            l1_data_gas: value.l1_data_gas.ok_or(missing("ResourceBounds::l1_data_gas"))?.try_into()?,
        })
    }
}
``` [8](#0-7) 

The `TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` conversion should bypass the `InvokeTransactionV3` intermediate and convert `resource_bounds` directly to `AllResourceBounds`, eliminating the misclassification and the `DEPRECATED_RESOURCE_BOUNDS_ERROR` false-positive.

---

### Proof of Concept

1. Submit via RPC:
   ```json
   {
     "type": "INVOKE",
     "version": "0x3",
     "resource_bounds": {
       "l1_gas": { "max_amount": "0x3e8", "max_price_per_unit": "0x1" },
       "l2_gas": { "max_amount": "0x0",   "max_price_per_unit": "0x0" },
       "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
     },
     ...
   }
   ```
2. Gateway accepts it; hash computed via `AllResources` path (3 resource felts: `l1_gas`, `l2_gas=0`, `l1_data_gas=0`).
3. Proposer includes it in a block proposal; serializes to `protobuf::InvokeV3WithProof`.
4. Peer deserializes: `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` sees `l2_gas=0, l1_data_gas=0` → produces `ValidResourceBounds::L1Gas`.
5. `InvokeTransactionV3 → RpcInvokeTransactionV3` fails: `ValidResourceBounds::L1Gas` is not `AllResources`.
6. Error propagates as `DEPRECATED_RESOURCE_BOUNDS_ERROR`; peer returns `HandledProposalPart::Failed`.
7. Proposal is rejected; the valid transaction is never sequenced.

```rust
pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
