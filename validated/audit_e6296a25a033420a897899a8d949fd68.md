### Title
Unbounded `NlpLockedBalanceQueue` Growth Causes Permanent Out-of-Gas DoS on `burnNlp` — (`SpotEngine.sol` / `SpotEngineState.sol`)

---

### Summary

Every `mintNlp` call at a distinct oracle timestamp appends a new entry to a per-subaccount `NlpLockedBalanceQueue`. The `tryUnlockNlpBalance` function processes all accumulated unlocked entries in a single unbounded `while` loop with no batch limit. A user who mints NLP regularly over time accumulates enough queue entries that, once the lock period expires, any subsequent call to `burnNlp` (or any NLP balance update) triggers an out-of-gas revert, permanently locking the user's NLP tokens.

---

### Finding Description

`SpotEngine.handleNlpLockedBalance` appends a new `NlpLockedBalance` entry to `queue.balances[queue.balanceCount]` and increments `queue.balanceCount` for every `mintNlp` that occurs at a different oracle timestamp than the previous mint:

```solidity
// SpotEngine.sol lines 162-166
queue.balances[queue.balanceCount] = NlpLockedBalance({
    balance: Balance({amount: amountDelta}),
    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
});
queue.balanceCount++;
``` [1](#0-0) 

`balanceCount` is never decremented — it is a monotonically growing index. The only deduplication is when two consecutive mints share the exact same oracle timestamp, which is unlikely for a user who mints across many blocks or days.

`tryUnlockNlpBalance` (called on every NLP balance update and on every `burnNlp` check) processes all unlocked entries in a single `while` loop:

```solidity
// SpotEngineState.sol lines 292-303
while (
    queue.unlockedUpTo < queue.balanceCount &&
    queue.balances[queue.unlockedUpTo].unlockedAt <= getOracleTime()
) {
    queue.unlockedBalanceSum.amount += queue.balances[queue.unlockedUpTo].balance.amount;
    delete queue.balances[queue.unlockedUpTo];
    queue.unlockedUpTo++;
}
``` [2](#0-1) 

There is no batch limit, no pagination, and no way to process a subset of entries. Once enough entries accumulate and unlock, the loop exhausts the block gas limit.

`burnNlp` calls `spotEngine.getNlpUnlockedBalance(txn.sender)` which calls `tryUnlockNlpBalance` before checking the user's unlocked balance:

```solidity
// SpotEngine.sol lines 129-137
function getNlpUnlockedBalance(bytes32 subaccount) external returns (Balance memory) {
    tryUnlockNlpBalance(subaccount);
    Balance memory balanceSum = nlpLockedBalanceQueues[subaccount].unlockedBalanceSum;
    return balanceSum;
}
``` [3](#0-2) 

```solidity
// Clearinghouse.sol lines 498-501
require(
    spotEngine.getNlpUnlockedBalance(txn.sender).amount >= nlpAmount,
    ERR_UNLOCKED_NLP_INSUFFICIENT
);
``` [4](#0-3) 

If `tryUnlockNlpBalance` reverts out-of-gas, `burnNlp` reverts. Because `burnNlp` is processed inside `submitTransactionsChecked` with no per-transaction try/catch, the revert propagates through `_delegatecallEndpointTx` and causes the entire sequencer batch to revert. The sequencer is forced to permanently exclude the user's `burnNlp` transaction, locking their NLP tokens. [5](#0-4) 

---

### Impact Explanation

A user whose `NlpLockedBalanceQueue` has grown large enough cannot burn NLP. Their NLP tokens are permanently inaccessible: `burnNlp` always reverts out-of-gas, and there is no alternative code path to redeem NLP or process queue entries in batches. This is a direct, concrete asset loss — the user's NLP balance is frozen in the protocol with no recovery mechanism.

Additionally, `tryUnlockNlpBalance` is also called from `handleNlpLockedBalance` on every NLP `updateBalance`, meaning even minting more NLP (which could theoretically trigger partial processing) would also revert once the queue is large enough.

**Severity: Medium** — User funds (NLP tokens) are permanently locked. No admin action can recover them without an upgrade.

---

### Likelihood Explanation

Any user who participates in NLP minting regularly over time is affected. Each `mintNlp` at a distinct oracle timestamp (which changes with every block or oracle update) adds one entry. A user who mints NLP once per day for several months accumulates hundreds of entries. After the `NLP_LOCK_PERIOD` expires, all entries unlock simultaneously and the next interaction triggers the unbounded loop. This is normal, expected user behavior in a liquidity pool — not an adversarial action.

**Probability: Medium** — Affects any long-term NLP participant. No special privileges or attack setup required beyond normal protocol usage.

---

### Recommendation

Apply the same fix recommended for the BtcPoller bug: process queue entries in bounded batches. Specifically:

1. Add a `maxEntries` parameter (or a hardcoded batch limit, e.g., 50) to `tryUnlockNlpBalance` so it processes at most N entries per call and returns early.
2. Alternatively, merge all same-day (or same-epoch) mints into a single entry at the time of minting, rather than only merging same-timestamp mints. This reduces queue growth for regular users.
3. Expose a standalone `tryUnlockNlpBalance(bytes32 subaccount, uint256 maxEntries)` function so users or keepers can drain the queue incrementally before attempting a burn.

---

### Proof of Concept

1. User calls `mintNlp` N times, each at a different oracle timestamp (e.g., once per block over N blocks). Each call appends one entry: `queue.balanceCount` reaches N.
2. After `NLP_LOCK_PERIOD` elapses, all N entries satisfy `unlockedAt <= getOracleTime()`.
3. User submits a `BurnNlp` signed transaction to the sequencer.
4. Sequencer includes it in `submitTransactionsChecked` → `processTransaction` → `processTransactionImpl` → `clearinghouse.burnNlp` → `spotEngine.getNlpUnlockedBalance` → `tryUnlockNlpBalance`.
5. The `while` loop iterates N times, each iteration performing a storage read, an addition, a `delete`, and an increment. For large N (e.g., N ≈ 500–1000 depending on gas costs per iteration), the call exceeds the block gas limit.
6. The revert propagates through `_delegatecallEndpointTx` back to `submitTransactionsChecked`, reverting the entire batch.
7. The sequencer cannot include the user's `burnNlp` — the user's NLP is permanently locked. [2](#0-1) [6](#0-5) [7](#0-6)

    function handleNlpLockedBalance(bytes32 subaccount, int128 amountDelta)
        internal
    {
        _assertInternal();

        // N_ACCOUNT is not limited by lock period
        if (subaccount == N_ACCOUNT) return;

        tryUnlockNlpBalance(subaccount);
        if (amountDelta > 0) {
            NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
                subaccount
            ];
            if (
                queue.balanceCount > 0 &&
                queue.balances[queue.balanceCount - 1].unlockedAt ==
                getOracleTime() + NLP_LOCK_PERIOD
            ) {
                queue
                    .balances[queue.balanceCount - 1]
                    .balance
                    .amount += amountDelta;
            } else {
                queue.balances[queue.balanceCount] = NlpLockedBalance({
                    balance: Balance({amount: amountDelta}),
                    unlockedAt: getOracleTime() + NLP_LOCK_PERIOD
                });
                queue.balanceCount++;
            }
        } else if (amountDelta < 0) {
            Balance memory balanceSum = nlpLockedBalanceQueues[subaccount]
                .unlockedBalanceSum;
            balanceSum.amount += amountDelta;
            nlpLockedBalanceQueues[subaccount].unlockedBalanceSum = balanceSum;
        }
    }

        public

    {
        NlpLockedBalanceQueue storage queue = nlpLockedBalanceQueues[
