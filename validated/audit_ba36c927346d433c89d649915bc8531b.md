Audit Report

## Title
`SwapAllowlistExtension` Checks Router Address Instead of Actual Swapper, Allowing Full Allowlist Bypass via `MetricOmmSimpleRouter` - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument forwarded from the pool, which is the pool's `msg.sender`. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, so the extension evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][user]`. Any pool admin who allowlists the router to enable router-mediated swaps for curated users inadvertently opens the pool to every caller of the router, completely nullifying the allowlist policy.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` (via `ExtensionCalling`) forwards that value unchanged to every configured extension as the `sender` parameter. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called the pool: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap(params.recipient, ...)` directly, making the pool's `msg.sender` the router: [3](#0-2) 

The real end-user identity (`msg.sender` at the router level) is stored only in transient storage via `_setNextCallbackContext` for callback payment settlement and is never forwarded to the extension as the identity to gate: [4](#0-3) 

The `extensionData` passed to the pool originates from `params.extensionData` supplied by the caller — the extension ignores it entirely (the `bytes calldata` parameter is unnamed and unused in `beforeSwap`): [5](#0-4) 

No existing guard compensates: the allowlist mapping is keyed only on `(pool, sender)` and there is no secondary check on `recipient` or `extensionData`.

## Impact Explanation
A pool admin who deploys a curated pool with `SwapAllowlistExtension` and allowlists the router address — the only mechanism to enable router-mediated swaps for approved users — inadvertently grants swap access to every address that can call the router. Any address explicitly excluded from the allowlist can execute swaps on the restricted pool by routing through `MetricOmmSimpleRouter`. The allowlist invariant (`allowedSwapper[pool][user]` must be `true` for a swap to proceed) is completely broken for the primary supported swap entrypoint, constituting broken core pool functionality with direct fund-access impact for pool operators relying on the allowlist for access control.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the standard, documented swap entrypoint. Allowlisting the router is the natural and only available configuration step for a pool admin who wants curated users to trade via the periphery. This makes the bypass reachable in any production deployment of a curated pool that supports router access, requiring no special attacker capability beyond calling a public function.

## Recommendation
The extension must gate by the actual end user, not the intermediary caller. Two complementary fixes:

1. **In `MetricOmmSimpleRouter`**: encode the real `msg.sender` (the end user) into `extensionData` before calling `pool.swap`, so the extension can extract and verify it.
2. **In `SwapAllowlistExtension.beforeSwap`**: decode the user address from `extensionData` when present, falling back to `sender` for direct pool calls (i.e., when `extensionData` is empty or not router-encoded).

The simplest correct fix is to have the router ABI-encode the user's address into `extensionData` and have the extension decode and check that value, with a fallback to `sender` for direct pool calls.

## Proof of Concept
```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension as beforeSwap hook
  admin calls swapExt.setAllowedToSwap(pool, router, true)
    // admin allowlists the router so approved users can use it
  admin does NOT call setAllowedToSwap(pool, attacker, true)

Attack:
  attacker calls router.exactInputSingle({
    pool: pool,
    recipient: attacker,
    zeroForOne: true,
    amountIn: X,
    extensionData: ""
  })

Trace:
  router.exactInputSingle
    → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, attacker, tokenIn)  // attacker stored in transient only
    → pool.swap(recipient=attacker, ..., extensionData="")  [msg.sender = router]
  pool._beforeSwap(sender=router, ...)
  SwapAllowlistExtension.beforeSwap(sender=router, ...)
    allowedSwapper[pool][router] == true  ← passes
  swap executes for attacker despite attacker not being allowlisted

Assertions:
  assertEq(swapAllowlistExtension.isAllowedToSwap(pool, attacker), false); // attacker not allowlisted
  // but swap succeeded — allowlist bypassed
```
