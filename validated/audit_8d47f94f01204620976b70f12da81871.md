### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any user to bypass the swap allowlist via MetricOmmSimpleRouter — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` to the pool is the **router contract**, not the end user. If the pool admin allowlists the router address to enable router-mediated swaps for their intended users, every user — including those not on the allowlist — can bypass the per-user restriction by routing through the router.

---

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension: [1](#0-0) 

`ExtensionCalling._beforeSwap` encodes that value as the first argument of `IMetricOmmExtensions.beforeSwap`: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is on the per-pool allowlist: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle`, the router calls `pool.swap()` directly, making `msg.sender` inside the pool the **router address**: [4](#0-3) 

The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][end_user]`.

This creates an irreconcilable split:

| Admin action | Allowlisted users (direct) | Allowlisted users (router) | Non-allowlisted users (router) |
|---|---|---|---|
| Allowlist specific user addresses only | ✓ allowed | ✗ blocked | ✗ blocked |
| Allowlist the router address | ✓ allowed | ✓ allowed | **✓ allowed — bypass** |

To let their allowlisted users use the router, the admin must allowlist the router address. Doing so silently opens the pool to **every** user who calls through the router, defeating the allowlist entirely.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` checks the `owner` parameter — the position owner explicitly passed through the call chain — which correctly identifies the end user regardless of whether the call originates from the adder or directly: [5](#0-4) 

The swap allowlist lacks this same identity-preservation property.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to specific counterparties (e.g., a private institutional pool, a pool gated to prevent front-running, or a pool under regulatory access control) can have its restriction completely bypassed. Any unprivileged user can execute swaps against the pool's LP positions by routing through `MetricOmmSimpleRouter`. Depending on the oracle price and pool configuration, this enables unauthorized extraction of value from LP positions and breaks the core access-control invariant the pool admin deployed the extension to enforce.

---

### Likelihood Explanation

The trigger requires two conditions:
1. A pool is deployed with `SwapAllowlistExtension` in its `BEFORE_SWAP_ORDER`.
2. The pool admin allowlists the router address (a natural and expected action when allowlisted users need to use the standard periphery).

Both conditions are part of normal, non-malicious pool operation. The admin's intent (allow my users to use the router) directly produces the bypass (allow all users to use the router) with no additional attacker setup. Any user who knows the pool address can exploit it by calling `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`.

---

### Recommendation

Pass the **end user's identity** through the swap hook rather than the immediate pool caller. Two approaches:

1. **Preferred — add a `swapper` field to the swap callback**: Have the router store the original `msg.sender` in transient storage and expose it via a standard interface (analogous to how `MetricOmmPoolLiquidityAdder` stores the payer). The pool reads this field and passes it as a separate `swapper` argument to `beforeSwap`, distinct from `sender` (the immediate caller).

2. **Alternative — gate on `recipient` instead of `sender`**: If the pool's design guarantees that the recipient is always the end user, the extension could check `recipient`. However, this is not always true for multi-hop routes where intermediate tokens land on the router.

The `DepositAllowlistExtension` pattern — checking `owner`, which is explicitly threaded through the call chain — is the correct model to follow.

---

### Proof of Concept

```
1. Deploy a pool with SwapAllowlistExtension in BEFORE_SWAP_ORDER.
2. Pool admin calls setAllowedToSwap(pool, alice, true)  — only Alice is allowed.
3. Pool admin calls setAllowedToSwap(pool, router, true) — to let Alice use the router.
4. Bob (not allowlisted) calls MetricOmmSimpleRouter.exactInputSingle({pool, ...}).
5. Router calls pool.swap(recipient, ...) — msg.sender to pool = router.
6. _beforeSwap passes sender = router to SwapAllowlistExtension.
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Bob's swap executes against LP positions despite not being on the allowlist.
``` [3](#0-2) [6](#0-5) [7](#0-6)
