### Title
Inflation Decay Permanently Disabled After Zero-Height Genesis Export Due to Stale `DecayEpochStart` ‚Äî (`app/export.go`, `x/inflation/keeper/genesis.go`, `x/inflation/keeper/mint.go`)

---

### Summary

`prepForZeroHeightGenesis` resets many height-dependent fields (validator bond heights, unbonding heights, signing info start heights) but never touches the inflation module's `DecayEpochStart`. `ExportGenesis` faithfully exports the original upgrade height (e.g., 5,000,000). On the new chain starting at height 0, `DeflationCalculationFn` evaluates `currentHeight < decayEpoch` as permanently true for millions of blocks, returning `baseRate` without any decay applied.

---

### Finding Description

**Step 1 ‚Äî `prepForZeroHeightGenesis` does not reset `DecayEpochStart`**

`app/export.go` resets staking, distribution, and slashing height-dependent state, but contains no call to the inflation keeper and no reset of `DecayEpochStart`: [1](#0-0) [2](#0-1)

``` [5](#0-4) 

**Step 5 ‚Äî `GenesisState.Validate()` does not catch the inconsistency**

The only validation is that `DecayEpochStart == 0` when `DecayRate` is not positive. It does not check whether `DecayEpochStart` is consistent with the new chain's starting height: [6](#0-5)

### Impact Explanation

With `DecayRate = 0.068` (6.8%/month) and `InflationMax = InflationMin = 1%/year`, after 12 months of decay the intended rate is approximately `1% √ó (0.932)^12 ‚âà 0.43%/year`. After a zero-height export the chain mints at the full `1%/year` base rate instead. The cumulative excess supply is concrete and calculable. If the running total (minus burned balances) crosses `MaxSupply`, `BeginBlocker` panics:

```go
if maxsupply.IsPositive() && totalsupply.GT(maxsupply) {
    panic(fmt.Sprintf("the total supply has exceeded the maximum supply: %s > %s", ...))
}
``` [7](#0-6) 

This creates tokens beyond the protocol-intended supply schedule (unbacked by the decay design) and, if `MaxSupply` is approached, causes a chain halt.

---

### Likelihood Explanation

Zero-height genesis export (`gaiad export --for-zero-height`) is a supported, documented production operation used during chain restarts and network upgrades. It requires no attacker ‚Äî any legitimate operator performing this operation after decay has been activated (i.e., after the v7.0.0 upgrade) triggers the bug automatically. The decay epoch on mainnet is set to the upgrade activation height, which is a large block number, so the condition `currentHeight < decayEpoch` holds for the entire lifetime of the restarted chain.

---

### Recommendation

In `prepForZeroHeightGenesis`, after resetting other height-dependent state, reset `DecayEpochStart` to a sentinel value appropriate for the new chain. The simplest correct fix is to set it to `1` (so decay begins immediately on the new chain, preserving the accumulated decay factor) or to clear it and re-derive the correct epoch from the exported params. At minimum, `ExportGenesis` should be aware of the zero-height context and zero out `DecayEpochStart` so `InitGenesis` does not restore a stale future height.

func (k Keeper) ExportGenesis(ctx context.Context) *types.GenesisState {
	genesis := types.DefaultGenesis()
	var err error
	genesis.Params, err = k.GetParams(ctx)
	if err != nil {
		panic("fail to get params:" + err.Error())
	}
	epoch, ok, err := k.getDecayEpochStart(ctx)
	if err != nil {
		panic("fail to get decay epoch start:" + err.Error())
	}
	if ok {
		genesis.DecayEpochStart = epoch
	}

	return genesis
