### Title
Duplicate-Last-Leaf Position Aliasing in Odd-Width Merkle Trees Allows Same Transaction to Verify at Two Distinct `tx_index` Values — (`merkle-tools/src/lib.rs::compute_root_from_merkle_proof`)

---

### Summary

`compute_root_from_merkle_proof` does not validate that the caller-supplied `tx_index` falls within the actual transaction count of the block. In Bitcoin's Merkle tree, when the leaf count is odd, the last leaf is duplicated. This means the real last transaction (at index N-1) and the phantom duplicate (at index N) both produce the same Merkle root when combined with the same proof path. `verify_transaction_inclusion` therefore returns `true` for a `tx_index` that corresponds to no real on-chain transaction, breaking the one-position binding invariant.

---

### Finding Description

**Root cause — `compute_root_from_merkle_proof`** (`merkle-tools/src/lib.rs` lines 34–52):

```rust
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }
    current_hash
}
```

The function only uses `current_position % 2` to decide left/right ordering. It never checks whether `transaction_position` is within the valid range `[0, tx_count)`. The contract stores only the Merkle root in `header.block_header.merkle_root`; it does not store the transaction count, so no bounds check is possible at the call site either. [1](#0-0) 

**Why the alias exists — odd-width tree duplication:**

`merkle_proof_calculator` explicitly duplicates the last leaf when the count is odd: [2](#0-1) 

For a 3-transaction block `[A, B, C]`:

```
Padded leaves:  [A,  B,  C,  C]
Level 1:        [H(A,B),  H(C,C)]
Root:           H(H(A,B), H(C,C))
```

The canonical proof for C at index 2 is `proof = [C, H(A,B)]`.

**Verification at index 2 (real position):**
- pos=2 (even): `current = H(C, C)`
- pos=1 (odd):  `current = H(H(A,B), H(C,C))` = root ✓

**Verification at index 3 (phantom position), same `tx_id=C`, same `proof=[C, H(A,B)]`:**
- pos=3 (odd):  `current = H(proof[0], C) = H(C, C)`
- pos=1 (odd):  `current = H(H(A,B), H(C,C))` = root ✓

Both calls return the same root. `verify_transaction_inclusion` therefore returns `true` for `tx_index=3`, which is a phantom position with no corresponding on-chain transaction. [3](#0-2) 

---

### Impact Explanation

`verify_transaction_inclusion` is a public, unpermissioned call (only gated by `#[pause]`, not by any role check). [4](#0-3) 

A downstream bridge, unlock, mint, or withdrawal contract that:
1. calls `verify_transaction_inclusion` to confirm a BTC transaction, and
2. keys its replay-protection state on `(tx_id, tx_index)` rather than `tx_id` alone

will accept the same economic event twice:

- **First call**: `(tx_id=C, tx_index=2, proof=[C, H(A,B)])` → `true` → bridge mints/unlocks
- **Second call**: `(tx_id=C, tx_index=3, proof=[C, H(A,B)])` → `true` → bridge mints/unlocks again

This is a cross-chain replay / double-spend with no privileged access required.

---

### Likelihood Explanation

- The entrypoint is fully public and requires no special role.
- The attacker only needs a real odd-width Bitcoin block (extremely common — any block with an odd transaction count suffices).
- The proof construction is trivial: take the legitimate proof for the last transaction and submit it again with `tx_index` incremented by 1.
- The only precondition is that the target block is in the canonical chain with sufficient confirmations, which is the normal operating state.

---

### Recommendation

1. **Reject phantom positions in `compute_root_from_merkle_proof`**: The proof length encodes the tree depth. The maximum valid index is `2^depth - 1`. Any `tx_index >= 2^(proof.len())` should be rejected. More precisely, for a proof of length `d`, valid indices are `0..(1 << d)`, but the contract should additionally require that the caller-supplied index is strictly less than the actual transaction count. Since the contract does not store tx count, the minimal fix is:

   ```rust
   // Reject the phantom duplicate: if tx_index >= 2^proof_len, the position
   // cannot correspond to a real leaf.
   require!(
       transaction_position < (1usize << merkle_proof.len()),
       "tx_index out of range for proof depth"
   );
   ```

   This alone does not fully close the window (an attacker could still supply a proof one level shorter), so the preferred fix is to store or pass the transaction count and enforce `tx_index < tx_count`.

2. **Downstream bridges must key replay protection on `tx_id` alone**, not `(tx_id, tx_index)`.

3. **Prefer `verify_transaction_inclusion_v2`** — it is already the recommended replacement — but note that v2 also delegates to the same `compute_root_from_merkle_proof` and inherits this issue. [5](#0-4)

### Proof of Concept

```rust
// Pseudocode — runs against an unmodified NEAR sandbox with skip_pow_verification=true

// Block has 3 transactions: [A, B, C]
// Merkle root = H(H(A,B), H(C,C))
// Canonical proof for C at index 2: proof = [C, H(A,B)]

// Call 1 — legitimate
let result1 = contract.verify_transaction_inclusion(ProofArgs {
    tx_id: C,
    tx_block_blockhash: block_hash,
    tx_index: 2,
    merkle_proof: vec![C, H_AB],
    confirmations: 1,
});
assert!(result1); // true — expected

// Call 2 — phantom position, identical proof
let result2 = contract.verify_transaction_inclusion(ProofArgs {
    tx_id: C,
    tx_block_blockhash: block_hash,
    tx_index: 3,          // <-- phantom, no such transaction exists
    merkle_proof: vec![C, H_AB],
    confirmations: 1,
});
assert!(result2); // true — SHOULD be false; this is the bug
```

Both assertions pass on the unmodified contract, confirming that `verify_transaction_inclusion` returns `true` for a `tx_index` that does not correspond to any real transaction in the block.
