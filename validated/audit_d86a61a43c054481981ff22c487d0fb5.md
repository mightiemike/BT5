### Title
`ValidResourceBounds::AllResources` with zero `l2_gas`/`l1_data_gas` silently collapses to `L1Gas` after protobuf round-trip, producing a divergent transaction hash — (`File: crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant: if both `l2_gas` and `l1_data_gas` are zero it emits `ValidResourceBounds::L1Gas`, regardless of whether the sender originally chose `AllResources`. Because `get_tip_resource_bounds_hash` hashes a different number of resource felts for each variant (2 for `L1Gas`, 3 for `AllResources`), a transaction submitted with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` receives hash H1 at the gateway but hash H2 ≠ H1 after any P2P protobuf round-trip, breaking hash validation on syncing nodes.

---

### Finding Description

**Step 1 – Hash preimage depends on the enum variant, not just the values.**

`get_tip_resource_bounds_hash` in `crates/starknet_api/src/transaction_hash.rs` always emits two resource felts (`L1_GAS`, `L2_GAS`) and conditionally appends a third (`L1_DATA_GAS`) only for `AllResources`:

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,
];
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],                          // 2 felts
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // 3 felts
    }
});
``` [1](#0-0) 

So `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` hashes as `poseidon(tip, L1_GAS||X, L2_GAS||0, L1_DATA_GAS||0)` while `L1Gas(X)` hashes as `poseidon(tip, L1_GAS||X, L2_GAS||0)`. These are distinct Felt values.

**Step 2 – The gateway always computes the hash with `AllResources`.**

`InternalRpcInvokeTransactionV3` stores `resource_bounds: AllResourceBounds` (not `ValidResourceBounds`) and its `InvokeTransactionV3Trait` impl unconditionally wraps it as `ValidResourceBounds::AllResources(...)`:

```rust
fn resource_bounds(&self) -> ValidResourceBounds {
    ValidResourceBounds::AllResources(self.resource_bounds)
}
``` [2](#0-1) 

The hash H1 is therefore always computed with 3 resource felts at the gateway. [3](#0-2) 

**Step 3 – Protobuf serialization preserves the zero values but loses the variant tag.**

`From<ValidResourceBounds> for protobuf::ResourceBounds` serializes `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` as three present-but-zero fields:

```rust
ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }) =>
    protobuf::ResourceBounds {
        l1_gas: Some(l1_gas.into()),
        l2_gas: Some(l2_gas.into()),
        l1_data_gas: Some(l1_data_gas.into()),
    },
``` [4](#0-3) 

**Step 4 – Deserialization silently collapses the variant.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` uses a value-only heuristic:

```rust
let l1_data_gas = value.l1_data_gas.unwrap_or_default();
// ...
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← wrong variant
} else {
    ValidResourceBounds::AllResources(...)
})
``` [5](#0-4) 

Because both fields are zero, the deserialized result is `ValidResourceBounds::L1Gas(X)` even though the original was `AllResources`. The protobuf wire format carries no variant discriminant, so the information is permanently lost.

**Step 5 – The P2P sync path uses `InvokeTransactionV3` (the API type), which holds `ValidResourceBounds`.**

`TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(...)` directly:

```rust
let resource_bounds = ValidResourceBounds::try_from(
    value.resource_bounds.ok_or(missing("InvokeV3::resource_bounds"))?,
)?;
``` [6](#0-5) 

After deserialization the transaction carries `L1Gas(X)`. Any subsequent call to `get_transaction_hash` or `validate_transaction_hash` on this object computes H2 (2 resource felts) ≠ H1 (3 resource felts). [7](#0-6) 

---

### Impact Explanation

A syncing node that receives a block via P2P containing an `InvokeV3` transaction with `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` will deserialize it as `L1Gas(X)`. When it validates the stored transaction hash (H1, computed with 3 felts) against the recomputed hash (H2, computed with 2 felts), validation fails. The node rejects a legitimately sequenced block, causing a consensus/sync split. This matches the **High** impact scope: "Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."

---

### Likelihood Explanation

The trigger is an unprivileged user submitting an `InvokeV3` transaction with `AllResourceBounds` where `l2_gas` and `l1_data_gas` are both zero. The gateway's stateless validator explicitly accepts this configuration (it only requires at least one non-zero bound). The condition is therefore reachable by any user without special privileges.

---

### Recommendation

Preserve the variant discriminant across the protobuf boundary. Two options:

1. **Add a boolean field** (e.g., `is_all_resources: bool`) to the `ResourceBounds` protobuf message and use it to reconstruct the correct variant on deserialization, ignoring the value-based heuristic.
2. **Treat `AllResources` as the canonical form for all V3 transactions** in the P2P path: always deserialize to `AllResources` when all three resource fields are present in the protobuf message, regardless of whether their values are zero.

---

### Proof of Concept

```
1. User submits RpcInvokeTransactionV3 with:
     resource_bounds = AllResourceBounds { l1_gas: {max_amount: 1000, max_price: 1}, l2_gas: {0,0}, l1_data_gas: {0,0} }

2. Gateway (InternalRpcInvokeTransactionV3) computes hash H1:
     tip_resource_bounds_hash = poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0), concat(L1_DATA_GAS, 0, 0))
     // 3 resource felts

3. Transaction included in block, stored with hash H1.

4. Syncing node receives block via P2P protobuf.
   protobuf::ResourceBounds { l1_gas: Some({1000,1}), l2_gas: Some({0,0}), l1_data_gas: Some({0,0}) }
   → TryFrom: l1_data_gas.is_zero() && l2_gas.is_zero() == true
   → ValidResourceBounds::L1Gas({1000, 1})

5. Syncing node recomputes hash H2:
     tip_resource_bounds_hash = poseidon(tip, concat(L1_GAS, 1000, 1), concat(L2_GAS, 0, 0))
     // 2 resource felts

6. H1 ≠ H2 → validate_transaction_hash returns false → valid block rejected.
```
