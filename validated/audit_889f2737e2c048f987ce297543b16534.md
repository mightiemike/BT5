### Title
Resharding Cooldown Invariant Bypassed at Static-to-Dynamic Protocol Upgrade Boundary — (`chain/epoch-manager/src/lib.rs`)

### Summary

`can_reshard` uses `is_none_or` on `EpochInfo::last_resharding()`, which returns `None` for all pre-V5 `EpochInfo` variants (V1–V4) **and** for the first V5 epoch (no dynamic resharding has occurred yet). At the protocol version boundary where `DynamicResharding` is first enabled, this causes `can_reshard` to return `true` unconditionally, bypassing the `min_epochs_between_resharding` cooldown even when a static resharding occurred in the immediately preceding epoch. The code explicitly acknowledges the assumption: *"It is theoretically possible that a static resharding was scheduled right before enabling dynamic resharding, but we assume this didn't happen."*

### Finding Description

`can_reshard` in `chain/epoch-manager/src/lib.rs`:

```rust
let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
    next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
});
``` [1](#0-0) 

`EpochInfo::last_resharding()` returns `None` for V1–V4 and for V5 when no dynamic resharding has yet occurred:

```rust
pub fn last_resharding(&self) -> Option<EpochHeight> {
    match self {
        Self::V1(_) | Self::V2(_) | Self::V3(_) | Self::V4(_) => None,
        Self::V5(v5) => v5.last_resharding,
    }
}
``` [2](#0-1) 

`EpochInfo::new()` creates V5 only when `ProtocolFeature::DynamicResharding.enabled(protocol_version)`, and the first V5 epoch always has `last_resharding = None`: [3](#0-2) 

The `last_resharding` field in `EpochInfoV5` tracks only **dynamic** reshardings. Static reshardings (which changed the shard layout at protocol versions 75, 76, 78 on mainnet) are never recorded in this field: [4](#0-3) 

At the static→dynamic transition boundary, the transitional guard in `next_next_shard_layout` does **not** apply because the current epoch already has `dynamic_resharding_config`:

```rust
// Dynamic resharding is not yet enabled, but will become enabled in epoch N+1 or N+2.
if current_epoch_config.dynamic_resharding_config().is_none() {
    return Ok(next_shard_layout.clone());
}
``` [5](#0-4) 

So once the current epoch has `ShardLayoutConfig::Dynamic`, `next_next_shard_layout` proceeds to derive a new V3 layout from `block_info.shard_split()` without any cooldown enforcement. Both `compute_proposed_split` (runtime) and `get_upcoming_shard_split` (block production and validation) call the same `can_reshard`, so the bypass is symmetric — the block is produced and validated with the same flawed gate. [6](#0-5) 

### Impact Explanation

The cooldown invariant is documented as safety-critical:

> *"allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."* [7](#0-6) 

`InvalidChunkHeaderShardSplit` is a consensus-level rejection: nodes that recompute the expected `proposed_split` from `ChunkExtra` and find a mismatch against the received chunk header will reject the chunk. This can cause a chain halt. Additionally, the memtrie preloading pipeline (which requires a full epoch to load the parent shard's memtrie) would not have completed, causing the resharding execution itself to fail or panic.

### Likelihood Explanation

This is triggered at a **public protocol upgrade boundary** — not by any privileged actor's deliberate action. Mainnet has had static reshardings at protocol versions 75, 76, and 78. When `DynamicResharding` is activated at a later protocol version, the first V5 `EpochInfo` will have `last_resharding = None`, and `can_reshard` will return `true` immediately. If any shard exceeds the memory threshold (or is in `force_split_shards`) during the first dynamic epoch, a resharding will be scheduled for epoch N+2 — just two epochs after the last static resharding — bypassing whatever `min_epochs_between_resharding` was configured.

The code comment *"we assume this didn't happen"* is the exact analog of the Rigor `autoWithdraw` comment *"Users in our system are KYC'ed, whitelisted, and trusted"* — an explicit trust assumption substituted for a structural invariant.

### Recommendation

When constructing the first `EpochInfoV5` at the static→dynamic transition, initialize `last_resharding` to the epoch height of the most recent static resharding (if any), rather than `None`. Concretely, `finalize_epoch` already computes `last_resharding` as:

```rust
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
``` [8](#0-7) 

The `.or_else(|| next_epoch_info.last_resharding())` fallback returns `None` for V4 even when a static resharding just occurred. The fix is to also check whether the shard layout changed between the previous and current epoch when `next_epoch_info` is pre-V5, and if so, set `last_resharding` to the current epoch height. Alternatively, `can_reshard` should treat `None` from a V5 epoch as "unknown" and consult the static resharding history via `get_shard_layout_history` before returning `true`.

### Proof of Concept

1. Mainnet: static resharding at epoch height H (protocol version 78, `ShardLayoutConfig::Static`).
2. Protocol version X+1 activates `ProtocolFeature::DynamicResharding`; epoch H+K is the first V5 epoch, `last_resharding = None`.
3. At the end of epoch H+K, `can_reshard` is called for epoch H+K+1:
   - `next_epoch_info` = V5 with `last_resharding = None`
   - `is_none_or(...)` → `true`
   - `can_reshard` returns `true`
4. A shard exceeds `memory_usage_threshold`; `compute_proposed_split` returns `Some(split)`.
5. `get_upcoming_shard_split` selects the split; block producer embeds it in `BlockHeaderInnerRestV6.shard_split`.
6. `validate_block_shard_split` recomputes via the same `can_reshard` → accepts the block.
7. `finalize_epoch` calls `next_next_shard_layout` → derives a new `ShardLayoutV3` for epoch H+K+2.
8. Epoch H+K+2 begins with a new shard layout, just K+2 epochs after the static resharding at H — bypassing `min_epochs_between_resharding` entirely.
9. The child shard's first chunk produces `proposed_split = None`; if the child's `ChunkExtra` inherited the parent's `proposed_split = Some(split)`, `InvalidChunkHeaderShardSplit` is raised and the chunk is rejected network-wide.

```rust
    pub fn get_upcoming_shard_split(
        &self,
        protocol_version: ProtocolVersion,
        parent_hash: &CryptoHash,
        chunk_headers: &[ShardChunkHeader],
    ) -> Result<Option<(ShardId, AccountId)>, EpochError> {
        // Check if dynamic resharding is enabled
        let epoch_config = self.get_epoch_config(protocol_version);
        let dynamic_resharding_config = match &epoch_config.shard_layout_config {
            ShardLayoutConfig::Static { .. } => return Ok(None),
            ShardLayoutConfig::Dynamic { dynamic_resharding_config } => dynamic_resharding_config,
        };

        // Check if resharding is allowed based on epoch constraints
        let can_reshard = self
            .can_reshard(&parent_hash, dynamic_resharding_config.min_epochs_between_resharding)?;
        if !can_reshard {
            return Ok(None);
        }

        // Collect proposed splits from chunk headers
        let mut proposed_splits = HashMap::new();
        for chunk_header in chunk_headers {
            if let Some(split) = chunk_header.proposed_split() {
                proposed_splits.insert(chunk_header.shard_id(), split.clone());
            }
        }

        // Pick the shard to split
        let Some((shard_id, split)) =
            pick_shard_to_split(&proposed_splits, dynamic_resharding_config)
        else {
            return Ok(None);
        };

        Ok(Some((shard_id, split.boundary_account)))
    }
