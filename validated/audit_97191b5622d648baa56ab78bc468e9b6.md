### Title
Unprotected `run_mainchain_gc` Allows Any Caller to Permanently Prune Canonical Block Headers - (File: `contract/src/lib.rs`)

### Summary
`run_mainchain_gc` is a public, state-mutating function with no caller access control. Any unprivileged NEAR account can call it with an arbitrary `batch_size`, permanently deleting mainchain block headers from both `mainchain_header_to_height` and `headers_pool`. This can corrupt the canonical chain state relied upon by SPV proof verification and chain reorganization logic.

### Finding Description
`run_mainchain_gc` is declared `pub` and carries only a `#[pause(except(roles(Role::UnrestrictedRunGC)))]` attribute. That attribute controls behavior when the contract is **paused**; it places no restriction on who may call the function when the contract is running normally. [1](#0-0) 

The function removes the oldest mainchain block headers by calling `remove_block_header`, which deletes entries from both `mainchain_header_to_height` **and** `headers_pool`: [2](#0-1) 

Deletion from `headers_pool` is permanent and irreversible. Once a block hash is removed, it cannot be re-submitted because the contract requires every new header to chain onto an existing entry in `headers_pool` via `get_prev_header`. [3](#0-2) 

By contrast, `submit_blocks` — the only other function that writes to the canonical chain — is protected by `#[trusted_relayer]`, which enforces role-based access control: [4](#0-3) 

`run_mainchain_gc` has no equivalent guard.

### Impact Explanation
Two concrete state corruptions are reachable:

**1. SPV proof verification permanently broken for pruned blocks.**
`verify_transaction_inclusion` and `verify_transaction_inclusion_v2` both look up the target block in `headers_pool`: [5](#0-4) 

and in `mainchain_header_to_height`: [6](#0-5) 

If an attacker calls `run_mainchain_gc` before a downstream contract calls `verify_transaction_inclusion`, and the target block falls in the pruned range, the call panics. Because deletion from `headers_pool` is permanent, the proof can never succeed for that block again. Downstream contracts that gate asset releases or cross-chain actions on SPV proofs will permanently reject valid Bitcoin transactions.

**2. Chain reorganization permanently blocked.**
`reorg_chain` walks the fork chain backward through `headers_pool` until it finds the common ancestor: [7](#0-6) 

If an attacker prunes the common ancestor before the relayer submits the competing fork, `reorg_chain` panics with `PrevBlockNotFound`. The contract is then permanently locked to the incorrect canonical chain because the fork's ancestor is gone and cannot be re-inserted. This is explicitly acknowledged in the project documentation: [8](#0-7) 

### Likelihood Explanation
The entry path requires no privileges. Any NEAR account can call `run_mainchain_gc` with `batch_size = u64::MAX`. The GC removes up to `amount_of_headers_we_store - gc_threshold` blocks per call, bounded by the configured threshold. With a small `gc_threshold` (e.g., 100–500 blocks, which is realistic for testnets or cost-sensitive deployments), the attacker can prune blocks that are only minutes to hours old. On mainnet with `gc_threshold = 52704`, the window is large but the attack is still viable: the attacker monitors the NEAR mempool, sees a relayer `submit_blocks` transaction that will push the chain past the threshold, and front-runs the subsequent `verify_transaction_inclusion` call with a `run_mainchain_gc` call targeting the block being proven.

### Recommendation
Add a role-based access control guard to `run_mainchain_gc` so that only accounts holding an authorized role (e.g., `Role::DAO` or a dedicated `GCManager` role) can invoke it externally. The internal call from `submit_blocks` can bypass this guard via a private helper. This mirrors the pattern already applied to `submit_blocks` via `#[trusted_relayer]`.

### Proof of Concept
1. Deploy the contract with `gc_threshold = 100` and `skip_pow_verification = true`.
2. Grant `UnrestrictedSubmitBlocks` to a relayer account; submit 200 valid block headers so the chain exceeds the threshold.
3. From an **unprivileged** account (no roles granted), call:
   ```json
   { "method": "run_mainchain_gc", "args": { "batch_size": 18446744073709551615 } }
   ```
4. Observe that the call succeeds and `get_mainchain_size` drops to 100.
5. Attempt to call `verify_transaction_inclusion` for any block that was in the pruned range; observe the contract panics with `"block does not belong to the current main chain"`.
6. Attempt to submit a fork whose common ancestor was in the pruned range; observe the contract panics with `"PrevBlockNotFound"` during reorg, permanently locking the canonical chain.
