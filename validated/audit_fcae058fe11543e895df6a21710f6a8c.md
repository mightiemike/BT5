### Title
Silent Bonus Reward Forfeiture on Partial Redelegation Slash When Pool Is Insufficient - (`x/tieredrewards/keeper/slash.go` and `x/tieredrewards/keeper/claim_rewards.go`)

---

### Summary

In `x/tieredrewards`, when `BeforeRedelegationSlashed` fires during a **partial** redelegation slash and the bonus pool is insufficient, `processEventsAndClaimBonus` advances the position's `LastBonusAccrual` checkpoint and consumes event reference counts **before** checking pool balance. The error is then silently swallowed in `slashRedelegationPosition`, and the position is persisted with the advanced checkpoint. The owner permanently loses all accrued bonus rewards for the settled period with no recourse.

---

### Finding Description

`processEventsAndClaimBonus` in `claim_rewards.go` advances the position's bonus accrual checkpoint unconditionally before checking whether the pool can cover the computed bonus: [1](#0-0) 

`pos.UpdateLastEventSeq(entry.Seq)` is called inside the event loop (consuming each event's reference count via `decrementEventRefCount`), and then: [2](#0-1) 

`applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` to `blockTime`. Only **after** these mutations does the pool balance check occur: [3](#0-2) 

When the pool is insufficient, `ErrInsufficientBonusPool` is returned with `pos` already mutated (checkpoint advanced, event seqs consumed). Back in `slashRedelegationPosition`, this error is explicitly caught and swallowed to avoid a chain halt: [4](#0-3) 

For a **full slash**, `pos.ClearBonusCheckpoints()` is called immediately after, zeroing `LastBonusAccrual` and making the advanced checkpoint irrelevant: [5](#0-4) 

But for a **partial slash**, no checkpoint reset occurs. The position is persisted directly with the advanced `LastBonusAccrual`: [6](#0-5) 

The result: the position's bonus accrual window is permanently advanced past the slash event, the validator events for that period are garbage-collected (reference counts decremented to zero), and the owner's accrued bonus is gone forever. There is no mechanism to recover it.

---

### Impact Explanation

A position owner who has accrued legitimate bonus rewards loses them permanently and silently when:
1. Their redelegation entry is partially slashed, and
2. The bonus pool is insufficient at that moment.

The corrupted value is `pos.LastBonusAccrual` (advanced without payment) and the consumed `ValidatorEvent` reference counts (decremented without the corresponding bonus being paid). After the position is persisted, no subsequent `MsgClaimTierRewards` can recover the forfeited bonus — the events have been consumed and the accrual window has moved forward. [7](#0-6) 

The next claim will start from `pos.LastEventSeq` (already advanced past the slash event) and `pos.LastBonusAccrual` (already set to `blockTime`), so the forfeited segment is never revisited.

---

### Likelihood Explanation

Both preconditions are realistic in production:

1. **Partial redelegation slash**: Validators are slashed for downtime or double-signing. A partial slash (e.g., 5% downtime slash) on a redelegating position is a normal protocol event.
2. **Insufficient bonus pool**: The `BeginBlocker` continuously drains the rewards pool to top up base rewards. If the pool is low or empty at the time of a slash, the condition triggers. An attacker can also time a large `MsgClaimTierRewards` to drain the pool just before a known slash event.

The combination is not exotic — it requires only normal staking operations and a temporarily depleted pool.

---

### Recommendation

Move `applyBonusAccrualCheckpoint` and `pos.UpdateLastKnownBonded` to **after** the successful pool transfer, so that checkpoints are only advanced when the bonus is actually paid. Alternatively, if the pool is insufficient during a slash hook, reset the in-memory `pos` checkpoints to their pre-call values before persisting, so the owner can claim the rewards once the pool is replenished. [8](#0-7) 

---

### Proof of Concept

1. Create a tier position and redelegate it to a second validator. This creates a `RedelegationMapping` entry.
2. Advance block time so significant bonus accrues.
3. Drain the bonus pool (e.g., via `MsgClaimTierRewards` on other positions or by letting `BeginBlocker` exhaust it).
4. Trigger a partial validator slash on the destination validator (e.g., 5% downtime slash). This fires `BeforeRedelegationSlashed`.
5. `slashRedelegationPosition` calls `processEventsAndClaimBonus`, which advances `LastBonusAccrual` to `blockTime` and decrements event reference counts, then returns `ErrInsufficientBonusPool`.
6. The error is swallowed. The position is persisted with the advanced checkpoint.
7. Replenish the bonus pool.
8. Call `MsgClaimTierRewards` for the position. Observe that zero bonus is returned for the period that was silently forfeited — the events are gone and the accrual window has moved past them. [9](#0-8) [10](#0-9)
