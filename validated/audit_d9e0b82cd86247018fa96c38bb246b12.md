### Title
Linked Signer Can Overwrite Its Own Registration to Attacker-Controlled Address — (`core/contracts/EndpointTx.sol`)

### Summary

The fast-path `LinkSigner` handler in `processTransactionImpl` calls `validateSignedTx` with `allowLinkedSigner = true`, meaning the currently-registered linked signer is accepted as a valid authorizer for the transaction that *replaces* the linked signer. A compromised or malicious linked signer can therefore silently rotate `linkedSigners[subaccount]` to an attacker-controlled address, permanently hijacking signing authority over the victim's subaccount.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch decodes a `SignedLinkSigner` and validates it with `allowLinkedSigner = true`: [1](#0-0) 

`validateSignedTx` delegates to `validateSignature`, which passes `getLinkedSigner(sender)` to `Verifier.validateSignature` when `allowLinkedSigner` is `true`: [2](#0-1) 

`Verifier.validateSignature` accepts a signature from **either** the subaccount owner address **or** the registered linked signer: [3](#0-2) 

After the signature check passes, `linkedSigners[signedTx.tx.sender]` is unconditionally overwritten with the new value supplied in the transaction: [4](#0-3) 

There is no guard that restricts this write to the subaccount owner only. The linked signer is permitted to mutate the very relationship that grants it authority.

---

### Impact Explanation

Once `linkedSigners[victim_subaccount]` is overwritten with attacker address C:

- C can sign `WithdrawCollateral` transactions (fast-path, `allowLinkedSigner = true`) on behalf of the victim, draining all collateral via `clearinghouse.withdrawCollateral` → `SpotEngine.updateBalance`. [5](#0-4) 

- C can sign `MintNlp`, `BurnNlp`, `TransferQuote`, and `LiquidateSubaccount` transactions — all of which also pass `allowLinkedSigner = true`.
- The original owner A has no on-chain mechanism to detect or preempt this before funds are drained; the attacker only needs to submit the replacement `LinkSigner` tx and then immediately submit a `WithdrawCollateral` tx in the same sequencer batch.

**Scoped impact:** permanent hijack of signing authority enabling unauthorized `SpotEngine.updateBalance` mutations (balance drain) on the victim's subaccount.

---

### Likelihood Explanation

- **Precondition:** the victim must have previously linked any signer B (a common UX pattern for trading bots/API keys).
- **Attacker requirement:** compromise or control of B (e.g., leaked API key, malicious trading bot, insider). No admin or sequencer compromise is needed.
- **Execution:** two sequencer transactions — one `LinkSigner` signed by B, one `WithdrawCollateral` signed by C. Both are standard fast-path transactions.

This is a realistic threat for any user who has ever used the linked-signer feature.

---

### Recommendation

Restrict the `LinkSigner` mutation to the subaccount owner only. Change `allowLinkedSigner` to `false` for the `LinkSigner` branch in `processTransactionImpl`:

```solidity
// EndpointTx.sol — LinkSigner fast-path
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // ← was `true`; only the subaccount owner may change the linked signer
);
```

The same fix should be applied to the slow-mode `LinkSigner` path in `processSlowModeTransactionImpl` if it does not already enforce `validateSender` strictly against the subaccount owner. [6](#0-5) 

---

### Proof of Concept

```
Deploy:
  A  = owner EOA  (address embedded in A_subaccount bytes32)
  B  = linked signer EOA
  C  = attacker EOA

Step 1 — A links B (legitimate):
  A signs LinkSigner{ sender: A_subaccount, signer: B, nonce: 0 }
  → linkedSigners[A_subaccount] = B

Step 2 — B hijacks to C:
  B signs LinkSigner{ sender: A_subaccount, signer: C, nonce: 1 }
  validateSignedTx(..., allowLinkedSigner=true)
    → getLinkedSigner(A_subaccount) returns B
    → Verifier.validateSignature accepts B's signature ✓
  → linkedSigners[A_subaccount] = C

Step 3 — C drains A:
  C signs WithdrawCollateral{ sender: A_subaccount, productId: USDC, amount: MAX, nonce: 2 }
  validateSignedTx(..., allowLinkedSigner=true)
    → getLinkedSigner(A_subaccount) returns C
    → Verifier.validateSignature accepts C's signature ✓
  → clearinghouse.withdrawCollateral(A_subaccount, ...) → SpotEngine.updateBalance(USDC, A_subaccount, -MAX)

Assert: A's USDC balance == 0, funds transferred to C's address.
```

    function validateSignature(
        bytes32 sender,
        address linkedSigner,
        bytes32 digest,
        bytes memory signature
    ) public pure {
        address recovered = ECDSA.recover(digest, signature);
        require(
            (recovered != address(0)) &&
                ((recovered == address(uint160(bytes20(sender)))) ||
                    (recovered == linkedSigner)),
            ERR_INVALID_SIGNATURE
        );
    }
