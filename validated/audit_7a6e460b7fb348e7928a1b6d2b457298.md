### Title
Linked Signer Can Overwrite Its Own Authority via `LinkSigner` with `allowLinkedSigner = true` - (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This means the **currently linked signer** — a limited-authority hot-wallet key — can sign a new `LinkSigner` transaction to replace itself with any arbitrary address, including an attacker-controlled one. There is no check that only the subaccount owner (the address embedded in `sender`) may mutate the `linkedSigners` mapping. A compromised or malicious linked signer can therefore permanently hijack a victim's subaccount.

---

### Finding Description

The `LinkSigner` transaction type is intended to let a subaccount owner delegate signing authority to a secondary key (e.g., a trading bot). When processed through the sequencer path, the handler at `EndpointTx.sol` line 576 calls:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true          // ← allowLinkedSigner
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` with `allowLinkedSigner = true` resolves to `validateSignature`, which passes the current `linkedSigners[sender]` as the accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` then accepts the signature if the recovered address equals either the subaccount owner **or the linked signer**:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Because the linked signer is accepted as a valid signer for `LinkSigner` itself, it can craft a transaction that replaces `linkedSigners[victim_subaccount]` with an attacker-controlled address. The `signer` field in the `LinkSigner` struct is an unconstrained `bytes32` — no ownership proof is required from the address being installed:

```solidity
struct LinkSigner {
    bytes32 sender;
    bytes32 signer;   // ← arbitrary, no consent check
    uint64 nonce;
}
``` [4](#0-3) 

The `linkedSigners` mapping is stored in `EndpointStorage`: [5](#0-4) 

**Contrast with `WithdrawCollateralV2`**: the protocol already restricts linked-signer authority for withdrawals — `allowLinkedSigner` is set to `signedTx.tx.sendTo == address(0)`, preventing a linked signer from withdrawing to an arbitrary address: [6](#0-5) 

This demonstrates the protocol is aware that linked signers should have bounded authority. The `LinkSigner` case is inconsistent with this design intent.

The slow-mode path for `LinkSigner` is correctly restricted — it calls `validateSender(txn.sender, sender)`, which requires `msg.sender` to be the subaccount owner address: [7](#0-6) 

The sequencer path has no equivalent restriction.

---

### Impact Explanation

A linked signer that is compromised or malicious can:

1. Sign a `LinkSigner` transaction setting `signer` to an attacker-controlled address.
2. The sequencer includes it; on-chain validation passes because the linked signer is an accepted signer for this tx type.
3. `linkedSigners[victim_subaccount]` is overwritten with the attacker's address.
4. The attacker can now sign `WithdrawCollateral`, `WithdrawCollateralV2` (to the default address), `TransferQuote`, and `LiquidateSubaccount` transactions on behalf of the victim.
5. All collateral in the victim's subaccount can be drained.

The corrupted state is `linkedSigners[victim_subaccount]` — a direct authorization mapping that gates all subsequent signed operations on the subaccount.

---

### Likelihood Explanation

Linked signers are by design hot wallets or automated trading bots — keys that are online and more exposed than the cold-wallet subaccount owner. Key compromise of a hot wallet is a realistic threat. Additionally, a user who mistakenly links a malicious third-party address (e.g., via a phishing UI) immediately loses the ability to recover without the subaccount owner key, because the attacker can keep re-linking themselves faster than the victim can respond through the sequencer path.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type so that only the subaccount owner key can mutate the linked signer:

```solidity
// EndpointTx.sol — LinkSigner case
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This is consistent with the slow-mode path, which already enforces `msg.sender == subaccount owner`.

---

### Proof of Concept

```
1. Alice (owner of subaccount `alice_sub`) calls LinkSigner to register Bob's address
   as her linked signer. linkedSigners[alice_sub] = Bob.

2. Bob's private key is compromised (or Bob is malicious).

3. Attacker (controlling Bob's key) constructs:
     LinkSigner {
       sender: alice_sub,
       signer: bytes32(uint256(uint160(attacker_address))),
       nonce:  current_nonce_for_alice_sub
     }
   and signs it with Bob's key.

4. Attacker submits the signed transaction to the sequencer.
   The sequencer includes it in the next batch.

5. On-chain: validateSignedTx is called with allowLinkedSigner = true.
   getLinkedSigner(alice_sub) returns Bob.
   ECDSA.recover(digest, sig) == Bob  ✓  → validation passes.

6. linkedSigners[alice_sub] = attacker_address.

7. Attacker signs a WithdrawCollateral transaction for alice_sub,
   draining all collateral to their own wallet.
```
