### Title
Silent Bonus Forfeit on Redelegation Slash with Empty Pool Permanently Loses Accrued Rewards — (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `x/tieredrewards`, when a redelegation slash fires (`BeforeRedelegationSlashed`) and the rewards pool is empty, `processEventsAndClaimBonus` advances the position's bonus accrual checkpoints **before** the pool sufficiency check. The resulting `ErrInsufficientBonusPool` error is then silently swallowed in `slashRedelegationPosition`, and the advanced checkpoints are persisted via `setPositionWithState`. This permanently forfeits the position owner's accrued bonus rewards for the entire period before the slash — even after the pool is later refilled.

---

### Finding Description

**Root cause — checkpoint advancement before pool check in `processEventsAndClaimBonus`:**

In `claim_rewards.go`, the bonus accrual checkpoints are advanced unconditionally before the pool sufficiency check:

```go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)   // line 215 — advances LastBonusAccrual
pos.UpdateLastKnownBonded(bonded)                        // line 217 — advances LastKnownBonded

if totalBonus.IsZero() {
    return sdk.NewCoins(), nil
}

bonusCoins := sdk.NewCoins(sdk.NewCoin(bondDenom, totalBonus))
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // returns error — but pos already has blockTime checkpoints
}
``` [1](#0-0) 

When the pool is empty, `ErrInsufficientBonusPool` is returned with `pos.LastBonusAccrual` already set to `blockTime` and `pos.LastKnownBonded` already updated in memory.

**Root cause — error swallowed and advanced checkpoints persisted in `slashRedelegationPosition`:**

In `slash.go`, the `ErrInsufficientBonusPool` error is explicitly caught and discarded. Execution then falls through to `setPositionWithState`, which persists the in-memory `pos` — including the advanced checkpoints — to the store:

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — pos.LastBonusAccrual is already blockTime
    } else {
        return err
    }
}
// ...
return k.setPositionWithState(ctx, pos, ...)  // persists advanced checkpoints, no bonus paid
``` [2](#0-1) 

After this, the position's `LastBonusAccrual` is permanently set to the slash block time. The bonus accrued between the previous checkpoint and the slash time is irrecoverably lost — when the pool is later refilled, `processEventsAndClaimBonus` will only compute bonus from the slash time forward.

**Contrast with user-driven paths:**

For all user-driven paths (`MsgClaimTierRewards`, `MsgTierUndelegate`, `MsgTierRedelegate`, `MsgAddToTierPosition`, `MsgClearPosition`), an insufficient pool causes an atomic failure that rolls back the entire transaction, including any checkpoint advancement. The user can retry after the pool is replenished. [3](#0-2) 

The `BeforeRedelegationSlashed` hook is the only code path where the error is swallowed and the advanced checkpoints are committed without payment.

---

### Impact Explanation

A position owner who has a redelegating position permanently loses all bonus rewards accrued between their last claim and the slash block time. The loss amount is:

```
loss = shares × tokensPerShare × bonusApy × (slashTime - LastBonusAccrual) / SecondsPerYear
```

For a large position with a high-APY tier and a long unclaimed period, this can be a substantial amount of the bond denom. The loss is permanent — no retry is possible because the checkpoints have already advanced past the accrual window.

The corrupted value is: `pos.LastBonusAccrual` (advanced to `blockTime` without corresponding payment from `RewardsPoolName`), causing a permanent desynchronization between the accrual period and the paid period.

---

### Likelihood Explanation

Two independent, normal protocol events must coincide:

1. **Pool empty or insufficient**: The `RewardsPoolName` module account is continuously drained by the `BeginBlocker` `topUpBaseRewards` function every block when `TargetBaseRewardsRate > 0`. With a high rate or many validators, the pool can be fully drained within a few blocks of being funded. [4](#0-3) 

2. **Redelegation slash**: Any validator double-sign or downtime slash that fires `BeforeRedelegationSlashed` for a position that has an active redelegation mapping. [5](#0-4) 

Both events are normal protocol operations. The combination is realistic, especially on a chain with active slashing and a high base rewards rate.

---

### Recommendation

Move the `applyBonusAccrualCheckpoint` and `UpdateLastKnownBonded` calls to **after** the pool sufficiency check and the `SendCoinsFromModuleToAccount` call in `processEventsAndClaimBonus`. This ensures checkpoints are only advanced when payment actually succeeds.

Alternatively, in `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, explicitly **revert** the in-memory checkpoint changes on `pos` before calling `setPositionWithState`, so the position retains its pre-slash `LastBonusAccrual` and the owner can claim the accrued bonus once the pool is refilled.

---

### Proof of Concept

1. Alice creates a tier position with a high-APY tier and delegates to validator V1.
2. Alice redelegates to validator V2 via `MsgTierRedelegate`. A `RedelegationMapping` entry is created.
3. Time passes (e.g., 30 days). Alice's accrued bonus = `shares × rate × APY × 30days / year`.
4. The `RewardsPoolName` account is drained to zero by `BeginBlocker` (normal operation).
5. Validator V2 is slashed for double-signing. The SDK calls `BeforeRedelegationSlashed`.
6. `slashRedelegationPosition` calls `processEventsAndClaimBonus`:
   - `applyBonusAccrualCheckpoint` sets `pos.LastBonusAccrual = slashBlockTime` ✓
   - `sufficientBonusPoolBalance` returns `ErrInsufficientBonusPool` ✓
   - Error is returned with checkpoints already advanced in `pos`
7. `slashRedelegationPosition` swallows the error and calls `setPositionWithState(ctx, pos, ...)`.
8. `pos.LastBonusAccrual = slashBlockTime` is now persisted to the store.
9. The pool is later refilled by governance.
10. Alice calls `MsgClaimTierRewards`. `processEventsAndClaimBonus` computes bonus only from `slashBlockTime` forward — the 30-day accrual is permanently lost. [6](#0-5) [7](#0-6)
