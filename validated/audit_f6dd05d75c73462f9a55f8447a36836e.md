Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument and checks `owner` (the position beneficiary) instead. Because `MetricOmmPool.addLiquidity` imposes no restriction on who may supply the `owner` parameter, any unprivileged caller can name an allowlisted address as `owner`, pass the hook check, pay tokens via the callback, and have LP shares minted to that address — fully circumventing the pool admin's deposit allowlist.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` faithfully forwards both arguments: [2](#0-1) 

Inside `DepositAllowlistExtension`, the first positional argument (`sender`) is unnamed and discarded; the guard reads only `owner`: [3](#0-2) 

`addLiquidity` has no restriction on who may supply `owner` (contrast with `removeLiquidity`, which enforces `msg.sender != owner` revert): [4](#0-3) 

The sister extension `SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it checks `sender`, not `recipient`: [5](#0-4) 

The exploit path: an unprivileged caller (`bob`) calls `pool.addLiquidity(owner=alice, ...)` where `alice` is allowlisted. The hook evaluates `allowedDepositor[pool][alice] == true` and approves. Bob's `metricOmmAddLiquidityCallback` pays the tokens; LP shares are minted to `alice`. The allowlist is fully bypassed.

## Impact Explanation
This is a direct admin-boundary break. The deposit allowlist is the pool admin's primary mechanism for restricting who may inject tokens (e.g., KYC/AML compliance, curated LP sets). With the guard checking `owner` instead of `sender`, any unprivileged actor can deposit real tokens into an allowlist-gated pool in a single transaction. The unauthorized depositor's tokens enter the pool's bin accounting, LP shares are minted to the named `owner`, and the admin-configured access control is rendered entirely ineffective.

## Likelihood Explanation
Exploitation requires only a standard `addLiquidity` call with any allowlisted address as `owner`. No special permissions, flash loans, privileged roles, or oracle manipulation are needed. Any on-chain actor can execute this in a single transaction, repeatably, at any time the pool has the extension configured.

## Recommendation
Replace the unnamed first parameter with `sender` and gate on it, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][alice] = true
  allowedDepositor[pool][bob]   = false  // bob is NOT allowed

Attack (single tx, no special role):
  bob calls pool.addLiquidity(
      owner    = alice,   // allowlisted address
      salt     = 0,
      deltas   = <valid bin deltas>,
      callbackData = <bob's callback pays tokens>,
      extensionData = ""
  )

Result:
  Hook checks allowedDepositor[pool][alice] == true → passes
  Bob's callback transfers tokens into the pool
  LP shares minted to alice
  Bob has deposited into an allowlist-gated pool without being allowlisted
```
