### Title
Unclaimed Airdrop Tokens Are Permanently Stranded in `Airdrop.sol` — (File: `core/contracts/Airdrop.sol`)

---

### Summary

The `Airdrop.sol` contract holds ERC20 tokens for weekly Merkle-proof-based distributions. There is no sweep or recovery function anywhere in the contract. Any tokens deposited for a given week that eligible users fail to claim are permanently locked in the contract with no mechanism for the protocol to recover them.

---

### Finding Description

The `Airdrop.sol` contract is funded with a token balance and distributes it to eligible users via weekly Merkle roots. The owner registers a Merkle root per week via `registerMerkleRoot()`, and users call `claim()` to receive their allocation.

The `_claim()` function transfers `totalAmount` directly to `msg.sender`:

```solidity
function _claim(uint32 week, uint256 totalAmount, bytes32[] calldata proof) internal {
    _verifyProof(week, msg.sender, totalAmount, proof);
    SafeERC20.safeTransfer(IERC20(token), msg.sender, totalAmount);
    emit Claim(msg.sender, week, totalAmount);
}
```

The `claimed[week][sender]` mapping records what each address has claimed per week. However, the contract has **no expiry mechanism**, **no sweep function**, and **no recovery path** for tokens that are never claimed. The full contract surface is:

- `registerMerkleRoot()` — owner-only, registers a new week's root
- `claim()` — user-facing, transfers tokens to caller
- `getClaimed()` — view only

There is no function analogous to a `sweep()` or `recoverTokens()` that would allow the owner to reclaim tokens allocated to a week but left unclaimed. The `OwnableUpgradeable` base provides no such utility either.

This is structurally identical to the BribeRewarder issue: tokens are deposited into a distribution contract, and any portion that goes unclaimed is permanently stranded.

Concrete scenarios where tokens strand:

1. **Sanctioned addresses**: `_verifyProof` explicitly blocks sanctioned addresses from claiming (`isSanctioned` check at line 54). If a user becomes sanctioned after a Merkle root is registered, their allocation is locked forever.
2. **Lost keys / inactive users**: Users who lose wallet access or simply never claim leave their allocation permanently in the contract.
3. **Merkle root registered with zero claimants**: If a week's root is registered but no eligible user ever claims (e.g., a misconfigured root), the entire week's token allocation is stranded.

---

### Impact Explanation

Tokens deposited into `Airdrop.sol` for distribution that are never claimed are permanently locked. There is no recovery path. Depending on the token value and the number of unclaimed weeks, this can represent a significant and irreversible loss of protocol-controlled funds. The impact is a direct asset loss — not a theoretical one — because the tokens are held in the contract and the only exit path (`claim()`) requires a valid Merkle proof from the eligible address.

---

### Likelihood Explanation

Realistic and expected over time. Airdrop contracts routinely have unclaimed allocations due to user inactivity, lost keys, or sanctioned addresses. The sanctioned-address path is particularly concrete: the contract explicitly prevents sanctioned users from claiming, meaning their allocation is guaranteed to strand if they are sanctioned at claim time. As the protocol runs weekly distributions over many weeks, the cumulative stranded amount grows monotonically.

---

### Recommendation

Add an owner-controlled sweep function that can recover unclaimed tokens after a configurable expiry window (e.g., after N weeks have passed since a given week's root was registered):

```solidity
function sweep(address recipient, uint256 amount) external onlyOwner {
    SafeERC20.safeTransfer(IERC20(token), recipient, amount);
}
```

Or a time-gated per-week recovery that only activates after a claim deadline has passed, to preserve user claim rights during the active window.

---

### Proof of Concept

1. Owner calls `registerMerkleRoot(1, root)` and funds the contract with 100,000 tokens for week 1.
2. Address `0xAlice` is eligible for 10,000 tokens but becomes sanctioned before claiming.
3. `_verifyProof` reverts at `require(!ISanctionsList(sanctions).isSanctioned(sender), "address is sanctioned.")` — Alice can never claim.
4. Alice's 10,000 tokens remain in the contract indefinitely.
