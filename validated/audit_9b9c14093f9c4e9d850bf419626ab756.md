After thorough investigation of the codebase, I found a valid analog to the external bug in the mempool's delayed-declare state machine.

### Title
Stale Delayed Declare Moved to `tx_pool` Without Nonce Guard Suppresses Gap Detection, Enabling Admission of Gap-Creating Transactions - (File: `crates/apollo_mempool/src/mempool.rs`)

### Summary

`add_ready_declares` moves delayed declares into `tx_pool` by calling `add_tx_inner` directly, bypassing `add_tx_validations` and its nonce check. In a multi-sequencer scenario, a delayed declare can become stale (its nonce already committed by a competing sequencer). When the delay elapses, the stale declare is unconditionally inserted into `tx_pool`. Its sub-account-nonce value then causes `update_accounts_with_gap` to compute `account_nonce < lowest_nonce` as `false`, silently removing the account from `accounts_with_gap` even though a real gap still exists. This mirrors the external bug exactly: an old state record (the stale declare) is not cleaned up during a state transition, and when the superseding state (the `delayed_declares` entry) is removed, the stale record becomes the active source of truth and corrupts downstream logic.

### Finding Description

**Root cause — `add_ready_declares` has no nonce guard:** [1](#0-0) 

`add_ready_declares` pops every declare whose submission time has elapsed and calls `add_tx_inner` directly. `add_tx_inner` does not call `add_tx_validations`, so `validate_tx_nonce` is never reached for these transitions.

**`add_tx_validations` / `validate_tx_nonce` — the guard that is bypassed:** [2](#0-1) 

When called through the normal `add_tx` path, `validate_tx_nonce` rejects any transaction whose nonce is below the current account nonce. `add_ready_declares` skips this path entirely.

**`add_tx_inner` — unconditional pool insertion:** [3](#0-2) 

The stale declare (nonce 0, account nonce now 1) is inserted into `tx_pool`. Because `tx_reference.nonce (0) != account_nonce (1)`, it is never enqueued, but it permanently occupies a slot in `txs_by_account` with the lowest nonce for that address.

**`update_accounts_with_gap` — gap detection corrupted by the stale record:** [4](#0-3) 

The gap check is `account_nonce < lowest_nonce`. With the stale declare at nonce 0 in the pool and account nonce at 1, `get_lowest_nonce` returns 0. The expression `1 < 0` evaluates to `false`, so the account is removed from `accounts_with_gap` — even though a real gap exists at nonce 1 (invoke at nonce 2, no transaction at nonce 1).

**The test documents the stale-declare-in-pool behavior but does not assert gap state after the move:** [5](#0-4) 

The test comment explicitly acknowledges "add_ready_declares has no nonce guard" and only asserts `delayed_declares.is_empty()`. It does not assert `accounts_with_gap` after the stale declare lands in `tx_pool`, leaving the post-move gap-suppression undetected.

### Impact Explanation

Once the account is incorrectly removed from `accounts_with_gap`:

1. **Admission of gap-creating transactions when the mempool is full.** `handle_capacity_overflow` reads `account_has_gap = self.accounts_with_gap.contains(&address)`. With the account absent, `creating_gap` evaluates to `false`, so `try_make_space` is attempted and the gap-creating transaction is accepted instead of being rejected with `MempoolFull`.

2. **Stuck transactions are not evicted.** `try_make_space` evicts only from accounts in `accounts_with_gap`. The stale declare and the invoke at nonce 2 (both unsequenceable) occupy pool capacity indefinitely, causing valid transactions from other accounts to be rejected.

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The trigger requires a multi-sequencer (consensus) scenario where a competing sequencer commits a transaction at the same nonce as a local delayed declare. This is an explicitly anticipated scenario (the test comment names it). The window between `add_ready_declares` moving the stale declare and the next `commit_block` cleaning it up is bounded only by block time, during which any `add_tx` call for the affected address triggers the suppression.

### Recommendation

Add a nonce guard inside `add_ready_declares` before calling `add_tx_inner`:

```rust
fn add_ready_declares(&mut self) {
    let now = self.clock.now();
    while let Some((submission_time, _args)) = self.delayed_declares.front() {
        if now - self.config.static_config.declare_delay < *submission_time {
            break;
        }
        let (_submission_time, args) =
            self.delayed_declares.pop_front().expect("Delay declare should exist.");
        // NEW: discard stale declares whose nonce has already been committed.
        let account_nonce = self.state.resolve_nonce(
            args.account_state.address,
            args.account_state.nonce,
        );
        if args.tx.nonce() < account_nonce {
            continue; // stale — drop without inserting into tx_pool
        }
        self.add_tx_inner(args);
    }
    self.update_state_metrics();
}
```

Alternatively, `update_accounts_with_gap` can be hardened to ignore pool entries whose nonce is below the account nonce when computing the effective lowest nonce.

### Proof of Concept

```
1. add_tx(invoke, nonce=2, account_nonce=0)
   → accounts_with_gap = {A}

2. add_tx(declare, nonce=0, account_nonce=0)   [delayed]
   → delayed_declares = {A:0}
   → accounts_with_gap = {}   (delayed_declares.contains(A, 0) == true)

3. commit_block({A: 1})                         [competing sequencer committed nonce 0]
   → remove_up_to_nonce_when_committed(A, 1) removes nothing (declare is in delayed_declares)
   → update_accounts_with_gap({A: 1})
     delayed_declares.contains(A, 1) == false
     tx_pool.get_lowest_nonce(A) == Some(2)
     1 < 2 == true  →  accounts_with_gap = {A}   ✓ correct

4. fake_clock.advance(declare_delay); mempool.get_txs(1)
   → add_ready_declares() pops the declare (nonce=0) and calls add_tx_inner
   → tx_pool now contains {nonce=0 (stale declare), nonce=2 (invoke)}
   → update_accounts_with_gap NOT called for A here

5. add_tx(new_tx, address=A, nonce=3, account_nonce=1)
   → update_accounts_with_gap({A: 1})
     delayed_declares.contains(A, 1) == false
     tx_pool.get_lowest_nonce(A) == Some(0)   ← stale declare
     1 < 0 == false  →  accounts_with_gap = {}   ✗ WRONG — gap at nonce 1 exists

6. Mempool is now full; add_tx(valid_tx, address=B)
   → handle_capacity_overflow: account_has_gap(A)==false → creating_gap==false
   → try_make_space cannot evict A's stuck transactions
   → valid_tx from B is rejected with MempoolFull
```

        let AddTransactionArgs { tx, account_state } = args;
        info!("Adding transaction to mempool.");
        trace!("{tx:#?}");

        let tx_reference = TransactionReference::new(&tx);

        // Pre-count this tx as stuck; update_accounts_with_gap will correct the count if this tx
        // resolves the gap.
        if self.accounts_with_gap.contains(&account_state.address) {
            self.n_stuck_txs += 1;
        }

        self.tx_pool
            .insert(tx)
            .expect("Duplicate transactions should cause an error during the validation stage.");

        let AccountState { address, nonce: incoming_account_nonce } = account_state;
        let account_nonce = self.state.resolve_nonce(address, incoming_account_nonce);

        if self.is_fifo() {
            // FIFO mode: add all transactions to the queue immediately, regardless of nonce.
            // Keep all transactions from the same address in the queue.
            self.insert_to_tx_queue(tx_reference);
        } else if tx_reference.nonce == account_nonce {
            // Fee mode: only add transactions with matching account nonce.
            // Remove queued transactions the account might have. This includes old nonce
            // transactions that have become obsolete; those with an equal nonce should
            // already have been removed via fee escalation (`remove_replaced_tx`).
            self.tx_queue.remove_by_address(address);
            self.insert_to_tx_queue(tx_reference);
        }
    }

    fn update_accounts_with_gap(&mut self, address_to_nonce: AddressToNonce) {
        for (address, account_nonce) in address_to_nonce {
            // If a delayed declare transaction exists at the account nonce, it is next to execute,
            // so no gap exists.
            if self.delayed_declares.contains(address, account_nonce) {
                self.remove_from_accounts_with_gap(address);
                continue;
            }

            // Gap exists when lowest transaction nonce is higher than account nonce.
            let gap_exists = match self.tx_pool.get_lowest_nonce(address) {
                Some(lowest_nonce) => account_nonce < lowest_nonce,
                None => false, // No transactions for the account, so no gap.
            };

            // Update the eviction tracking set accordingly.
            if gap_exists {
                if self.accounts_with_gap.insert(address) {
                    // Newly entered gap: all current pool txs for this account are now stuck.
                    let n_stuck = self.tx_pool.n_txs_for_address(address);
                    self.n_stuck_txs += n_stuck;
                    warn!(
                        "Account {address} has a nonce gap; {n_stuck} transaction(s) are now \
                         stuck."
                    );
                }
                // Stayed in gap: per-tx deltas were already applied at add/remove sites.
            } else {
                // Left gap: remaining pool txs for this account are no longer stuck.
                self.remove_from_accounts_with_gap(address);
            }
        }
