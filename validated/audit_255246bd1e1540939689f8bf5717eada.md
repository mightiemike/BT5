### Title
Per-Segment `TruncateInt()` in Bonus Reward Calculation Causes Systematic Precision Loss Proportional to Validator Event Count - (File: x/tieredrewards/keeper/bonus_rewards.go)

---

### Summary

`computeSegmentBonus` applies `TruncateInt()` to each individual time segment's bonus before accumulating the total. Because `processEventsAndClaimBonus` calls `computeSegmentBonus` once per validator event (slash, unbond, bond), a position on a validator with N events suffers up to N independent truncations. A user who claims after many validator events receives strictly less bonus than a user who claims after fewer events covering the same total duration and the same economic exposure.

---

### Finding Description

`computeSegmentBonus` computes:

```
tokens * BonusApy * durationSeconds / SecondsPerYear
```

and immediately calls `.TruncateInt()` on the result before returning. [1](#0-0) 

`processEventsAndClaimBonus` iterates over every pending `ValidatorEvent` for the position's validator, calling `computeSegmentBonus` for each bonded segment and accumulating the results with integer addition:

```go
totalBonus = totalBonus.Add(bonus)   // bonus is already truncated
``` [2](#0-1) 

A final segment from the last event to the current block time is also computed and truncated independently: [3](#0-2) 

Because each call to `computeSegmentBonus` truncates independently, the sum of N truncated segments is always ≤ the truncation of the combined sum. The difference is at most N−1 token units (basecro), where N is the number of segments processed in a single claim.

**Concrete numeric example:**

Suppose a position has `tokens * BonusApy / SecondsPerYear = 0.9 basecro/second` and the validator was slashed 10 times, each creating a 1-second segment:

- **10 separate segments**: each yields `floor(0.9 * 1) = 0`, total = **0 basecro**
- **1 combined segment** (10 seconds): `floor(0.9 * 10) = 9`, total = **9 basecro**

The user receives 9 fewer basecro than owed. The rewards pool retains the difference.

The number of segments is determined by the number of `ValidatorEvent` entries recorded since the position's `LastEventSeq`. Each slash, unbond, or bond on the validator creates one event: [4](#0-3) 

`SecondsPerYear` is `31_557_600`: [5](#0-4) 

---

### Impact Explanation

The corrupted value is `totalBonus` returned by `processEventsAndClaimBonus`. The position owner receives fewer bonus rewards than they are mathematically owed. The shortfall is retained in the `rewards_pool` module account and is never distributed. The loss per claim is bounded by `(N − 1)` basecro where N is the number of validator events processed, but:

- For small positions or short segments where `tokens * BonusApy * durationSeconds / SecondsPerYear < 1`, entire segments round to zero, amplifying the loss.
- The loss is systematic and cumulative: every claim on a validator with multiple events undercharges the pool and underpays the user.
- The inconsistency is path-dependent: two positions with identical economic exposure receive different total bonus depending solely on how many validator events occurred between their claims.

---

### Likelihood Explanation

Validator slashes are a normal, expected Cosmos SDK staking event on mainnet. Any validator that is slashed once between two consecutive claims by a position owner creates two segments instead of one, triggering the precision loss. Validators that are slashed multiple times (e.g., double-sign or repeated downtime) amplify the effect. The entry path requires only:

1. A position owner locks stake via `MsgLockTier` or `MsgCommitDelegationToTier` (normal user operation).
2. The validator experiences one or more slash/unbond/bond events (normal chain operation).
3. The position owner calls `MsgClaimTierRewards` (normal user operation).

No privileged access, no special configuration, and no attacker coordination is required. The condition is met on any production chain with active slashing.

---

### Recommendation

Accumulate the bonus in `math.LegacyDec` across all segments and apply a single `TruncateInt()` only after the full sum is computed. Replace the per-segment integer truncation in `computeSegmentBonus` with a `Dec` return type, accumulate in `processEventsAndClaimBonus` as `math.LegacyDec`, and truncate once at the end before constructing `bonusCoins`. This mirrors the standard Cosmos SDK pattern used in `x/distribution` where `DecCoins` are accumulated and only converted to integer coins at withdrawal time.

---

### Proof of Concept

**Setup**: Position with `shares = 1`, `tokensPerShare = 1`, `BonusApy = 0.10`, validator slashed 10 times in rapid succession (each slash 1 second apart).

**Claim after 10 slashes** (10 segments of 1 second each):
```
per segment: floor(1 * 1 * 0.10 * 1 / 31_557_600) = floor(3.17e-9) = 0
total = 0 * 10 = 0 basecro
```

**Claim without slashes** (1 segment of 10 seconds):
```
floor(1 * 1 * 0.10 * 10 / 31_557_600) = floor(3.17e-8) = 0 basecro
```

For a larger position (`shares = 1_000_000_000`, `tokensPerShare = 1`, `BonusApy = 0.10`), 10 segments of 86400 seconds (1 day) each:
```
per segment: floor(1e9 * 0.10 * 86400 / 31_557_600) = floor(273785.08) = 273785
total (10 segments) = 2_737_850 basecro

single 10-day segment: floor(1e9 * 0.10 * 864000 / 31_557_600) = floor(2737850.8) = 2_737_850
```

In this case the loss is 0 because the fractional part happens to be the same. But with non-uniform segment durations (as occurs with real slash timing), the per-segment truncation consistently loses up to 1 basecro per segment boundary, confirmed by the formula at: [6](#0-5) [7](#0-6)
