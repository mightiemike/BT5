### Title
Division by Zero in `PerpEngine.socializeSubaccount()` When `openInterest` Is Zero — (`File: core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount()` unconditionally divides by `state.openInterest` without checking whether it is zero. When a product has no remaining open positions (e.g., the liquidatee was the only participant and has since closed their position), the division panics and permanently blocks subaccount finalization.

---

### Finding Description

In `PerpEngine.socializeSubaccount()`, after insurance is exhausted, the remaining negative `vQuoteBalance` is spread across all open-interest holders via:

```solidity
int128 fundingPerShare = -balance.vQuoteBalance.div(
    state.openInterest
);
``` [1](#0-0) 

`state.openInterest` is the global sum of `balance.amount.abs()` across all participants for that product. It is initialized to `0` when a product is first added: [2](#0-1) 

`getStateAndBalance` calls `_updateBalance(state, balance, 0, 0)`, which subtracts `balance.amount.abs()` from `state.openInterest` before returning: [3](#0-2) 

Because `_finalizeSubaccount` requires `balance.amount == 0` for all perps before calling `socializeSubaccount`, this subtraction is always zero — meaning `state.openInterest` reflects the true global open interest at the time of the call. If no other participants hold open positions in that product, `state.openInterest == 0` and the `.div()` call in `MathSD21x18` reverts with `"DBZ"`: [4](#0-3) 

---

### Impact Explanation

The revert propagates up through `_finalizeSubaccount` → `liquidateSubaccountImpl`, causing the entire finalization transaction to fail. The insolvent subaccount cannot be finalized, its bad debt cannot be resolved, and the insurance fund cannot be properly applied. This is a permanent DoS of the subaccount finalization path for any product where open interest has dropped to zero.

---

### Likelihood Explanation

The trigger condition is realistic:

1. A perp product is newly listed or has low participation.
2. A single user opens a position, accumulates negative `vQuoteBalance` (e.g., from adverse funding payments), then closes the position (`amount = 0`, `vQuoteBalance < 0` persists).
3. The user's quote balance goes negative (insolvency).
4. No other users hold open positions in that product (`openInterest == 0`).
5. A liquidator submits a finalization transaction (`productId == type(uint32).max`).

Step 5 is the attacker-controlled entry path. The liquidator is an unprivileged caller who triggers `liquidateSubaccountImpl` via `Endpoint.submitTransactionsChecked` or `submitSlowModeTransaction`. [5](#0-4) 

The `_finalizeSubaccount` call to `perpEngine.socializeSubaccount` is the necessary vulnerable step: [6](#0-5) 

---

### Recommendation

Add a zero-check on `state.openInterest` before dividing. If `openInterest == 0`, there are no other participants to socialize the loss to; the loss should be absorbed entirely by the insurance fund or left as irrecoverable bad debt rather than panicking:

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; skip or handle separately
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(
            state.openInterest
        );
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
```

---

### Proof of Concept

1. Deploy the protocol and add a perp product (e.g., `productId = 2`). `state.openInterest` starts at `0`.
2. User A opens a long position of size `S`. `openInterest` becomes `S`.
3. Funding payments accumulate, making User A's `vQuoteBalance` negative.
4. User A closes the position entirely (`amount = 0`). `openInterest` returns to `0`. `vQuoteBalance` remains negative.
5. User A's quote balance is also negative (insolvent). Insurance is insufficient to cover the full loss.
6. Liquidator calls `submitSlowModeTransaction` with a `LiquidateSubaccount` transaction where `productId = type(uint32).max` (finalize).
7. `liquidateSubaccountImpl` → `_finalizeSubaccount` → `perpEngine.socializeSubaccount(liquidatee, insurance)`.
8. Inside `socializeSubaccount`, for `productId = 2`: `balance.vQuoteBalance < 0`, insurance is exhausted, execution reaches `fundingPerShare = -balance.vQuoteBalance.div(state.openInterest)` where `state.openInterest == 0`.
9. `MathSD21x18.div` reverts with `"DBZ"`. The entire finalization transaction reverts. The insolvent subaccount is permanently un-finalizable.
