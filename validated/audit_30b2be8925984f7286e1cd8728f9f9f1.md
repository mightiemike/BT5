### Title
Accrued Bonus Rewards Permanently Lost When Pool Is Empty During Redelegation Slash — (File: `x/tieredrewards/keeper/slash.go`)

---

### Summary

In `slashRedelegationPosition`, when `processEventsAndClaimBonus` fails with `ErrInsufficientBonusPool`, the error is silently swallowed. However, `processEventsAndClaimBonus` already advanced the position's `LastBonusAccrual` and `LastEventSeq` checkpoints **before** the pool-sufficiency check. The position is then saved with those advanced checkpoints but without the bonus payment. The user permanently loses their accrued bonus rewards with no retry path.

---

### Finding Description

`processEventsAndClaimBonus` in `claim_rewards.go` advances the position's accrual checkpoints unconditionally before checking whether the pool can cover the computed bonus: [1](#0-0) 

```go
applyBonusAccrualCheckpoint(&pos.Position, blockTime)  // LastBonusAccrual = blockTime
pos.UpdateLastKnownBonded(bonded)
```

Only after these mutations does it check pool sufficiency: [2](#0-1) 

```go
if err := k.sufficientBonusPoolBalance(ctx, bonusCoins); err != nil {
    return nil, err   // ErrInsufficientBonusPool — but pos checkpoints already advanced
}
```

`LastEventSeq` is also advanced inside the event-walk loop before the pool check: [3](#0-2) 

When `slashRedelegationPosition` calls `processEventsAndClaimBonus` and receives `ErrInsufficientBonusPool`, it deliberately swallows the error: [4](#0-3) 

```go
if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash", ...)
        // error swallowed — pos already has advanced checkpoints
    } else {
        return err
    }
}
```

The function then saves the position with the already-mutated `pos` (advanced `LastBonusAccrual`, `LastEventSeq`, `LastKnownBonded`): [5](#0-4) 

```go
if fullSlash {
    pos.Delegation = nil
    pos.ClearBonusCheckpoints()
    return k.setPositionWithState(ctx, pos, &ValidatorTransition{...})
}
pos.Delegation.Shares = pos.Delegation.Shares.Sub(sharesToUnbond)
return k.setPositionWithState(ctx, pos, nil)
```

After this write, the position's accrual window has been consumed (`LastBonusAccrual` = slash block time, `LastEventSeq` past all pending events) but the corresponding bonus tokens were never transferred. There is no deferred-payment record, no pending-bonus queue, and no way for the user to reclaim those rewards later — the checkpoints that would allow recomputation have been permanently overwritten.

This contrasts with every user-driven path, where the transaction fails atomically and the user can retry: [6](#0-5) 

> **User-driven paths** (ClaimTierRewards, AddToPosition, Undelegate, Redelegate, ClearPosition): fail atomically. User retries after pool is replenished.

The hook path has no such protection.

---

### Impact Explanation

The corrupted value is `pos.LastBonusAccrual` and `pos.LastEventSeq` — they are advanced to the slash block time without the corresponding bonus coins being sent to the owner. The user's accrued bonus rewards (computed as `shares × tokensPerShare × bonusApy × duration / SecondsPerYear` for the entire bonded period since the last claim) are permanently destroyed. The bonus pool retains funds it owes to the user. This is the direct analog of the "interest-free loan" pattern: the user's tokens were locked and delegated (contributing to validator security), they earned bonus rewards during that period, but those rewards are silently confiscated when the pool is empty at slash time.

---

### Likelihood Explanation

The trigger requires three concurrent conditions:

1. **User redelegates a tier position** — a normal, unprivileged `MsgTierRedelegate` call. This creates a redelegation mapping entry.
2. **The source validator is slashed** — a standard staking event that fires `BeforeRedelegationSlashed`. Any validator misbehaviour (double-sign, downtime) can trigger this.
3. **The bonus pool is empty or underfunded** — the pool is an externally funded module account (`RewardsPoolName`). It can be empty at chain launch, after a large payout, or if no one has topped it up. An adversary who can observe the mempool or governance can time a redelegate to a validator they know will be slashed while the pool is low.

All three conditions are reachable without any privileged role. The combination is realistic on a live chain where the pool is not continuously topped up.

---

### Recommendation

Snapshot the position's checkpoints **before** calling `processEventsAndClaimBonus` in `slashRedelegationPosition`. If `ErrInsufficientBonusPool` is returned, restore the original checkpoints before saving the position, so the user retains their claim and can collect the bonus once the pool is replenished:

```go
// Save pre-call checkpoints
savedAccrual := pos.LastBonusAccrual
savedEventSeq := pos.LastEventSeq
savedKnownBonded := pos.LastKnownBonded

if _, err := k.processEventsAndClaimBonus(ctx, &pos); err != nil {
    if errors.Is(err, types.ErrInsufficientBonusPool) {
        // Restore checkpoints so the user can claim later
        pos.LastBonusAccrual = savedAccrual
        pos.LastEventSeq = savedEventSeq
        pos.LastKnownBonded = savedKnownBonded
        k.logger(ctx).Error("insufficient bonus pool during redelegation slash; rewards deferred", ...)
    } else {
        return err
    }
}
```

This ensures the hook never permanently destroys a user's reward claim, while still avoiding a chain halt.

---

### Proof of Concept

1. User calls `MsgTierRedelegate` moving their position from validator A to validator B. A `RedelegationMappings[unbondingId → positionId]` entry is created.
2. Time passes; the position accrues bonus rewards on validator B.
3. The bonus pool is empty (or drained to zero).
4. Validator A is slashed. The staking module fires `BeforeRedelegationSlashed(unbondingId, sharesToUnbond)`.
5. `slashRedelegationPosition` is called. It calls `processEventsAndClaimBonus`, which:
   - Walks all pending validator events, computing `totalBonus > 0`.
   - Advances `pos.LastBonusAccrual` to `blockTime` and `pos.LastEventSeq` past all events (lines 215–217 of `claim_rewards.go`).
   - Checks pool balance → insufficient → returns `ErrInsufficientBonusPool`.
6. `slashRedelegationPosition` catches the error and swallows it (lines 56–63 of `slash.go`).
7. The position is saved with the advanced checkpoints (line 71 or 77 of `slash.go`).
8. The user's accrued bonus rewards are permanently lost. A subsequent `MsgClaimTierRewards` call returns zero bonus because `LastBonusAccrual` is already at the current block time and `LastEventSeq` is past all events.
