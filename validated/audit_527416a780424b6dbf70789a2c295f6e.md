### Title
Insufficient `paid_fee_on_l1` Validation in L1 Handler Execution Accepts Underpayment — (`File: crates/blockifier/src/transaction/l1_handler_transaction.rs`)

---

### Summary

The `execute_raw` implementation for `L1HandlerTransaction` checks only that `paid_fee_on_l1 != 0` rather than verifying `paid_fee_on_l1 >= actual_fee`. This is the direct sequencer analog of the external "excess ether locked" bug: the external contracts used `msg.value < required` (too-weak lower bound), while the sequencer uses `paid_fee == Fee(0)` (only rejects zero, accepts any positive underpayment). Any non-zero `paid_fee_on_l1` — even 1 wei — passes the check regardless of the actual execution cost, causing the sequencer to commit state changes for an L1 handler transaction whose L1 fee payment does not cover the computed execution cost.

---

### Finding Description

In `crates/blockifier/src/transaction/l1_handler_transaction.rs`, after a successful L1 handler execution, the sequencer computes the actual fee via `TransactionReceipt::from_l1_handler` and then performs the following check:

```rust
// TODO(Arni): Consider removing this check. It is covered by the starknet
// core contract.
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
// The error message still indicates the required fee.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
``` [1](#0-0) 

The condition `paid_fee == Fee(0)` only rejects a zero payment. Any positive value — including `Fee(1)` — passes, even when `receipt.fee` (the actual execution cost computed from gas used × gas price) is orders of magnitude larger.

`receipt.fee` is computed by `TransactionReceipt::from_l1_handler`, which calls `tx_context.tx_info.get_fee_by_gas_vector(...)` using the real gas consumed and block gas prices: [2](#0-1) 

After the check passes, `l1_handler_tx_execution_info` unconditionally zeroes out the fee in the returned receipt:

```rust
fn l1_handler_tx_execution_info(...) -> TransactionExecutionInfo {
    receipt.fee = Fee(0);
    ...
}
``` [3](#0-2) 

This means the on-chain receipt always records `actual_fee = 0` for L1 handlers, and the only sequencer-side guard against underpayment is the `paid_fee == Fee(0)` check — which is trivially bypassed with any non-zero value.

The existing test confirms this is intentional but incorrect behavior:

```rust
// Today, we check that the paid_fee is positive, no matter what was the actual fee.
``` [4](#0-3) 

The `InsufficientFee` error variant itself is defined as `"Actual fee ({}) exceeded paid fee on L1 ({})."`, which implies the correct semantic is `paid_fee < actual_fee`, not `paid_fee == 0`: [5](#0-4) 

---

### Impact Explanation

The sequencer commits state changes for L1 handler transactions whose `paid_fee_on_l1` does not cover the actual execution cost. This is an incorrect fee/resource accounting outcome with direct economic impact: the sequencer absorbs the execution cost difference without receiving adequate L1 fee compensation. The `TransactionExecutionInfo` receipt records `fee = Fee(0)`, so downstream components (RPC, storage, consensus) observe a zero-fee L1 handler execution even when the actual gas cost was non-zero, producing a wrong receipt value.

---

### Likelihood Explanation

The developer comment acknowledges the check is intentionally weak ("covered by the starknet core contract"), but the core contract's fee validation uses a fee estimate computed at L1 submission time, which may diverge from the sequencer's post-execution `receipt.fee` (computed at actual execution time with current block gas prices). Any divergence between the L1 estimate and the L2 actual cost creates a window where `paid_fee_on_l1 > 0` but `paid_fee_on_l1 < receipt.fee`. Additionally, any path that injects an `L1HandlerTransaction` with an attacker-controlled `paid_fee_on_l1` (e.g., via the Python `py_l1_handler` wrapper or the `CentralL1HandlerTransaction` deserialization path) bypasses the core contract entirely. [6](#0-5) 

---

### Recommendation

Replace the equality check with a proper lower-bound comparison:

```rust
// Before (incorrect):
if paid_fee == Fee(0) { ... }

// After (correct):
if paid_fee < receipt.fee {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
```

This mirrors the semantics already expressed by the `InsufficientFee` error message and aligns with the `check_actual_cost_within_bounds` pattern used for account transactions.

---

### Proof of Concept

1. Construct an `L1HandlerTransaction` with `paid_fee_on_l1 = Fee(1)` (1 wei).
2. The L1 handler executes a storage-writing operation whose actual gas cost computes to, e.g., `receipt.fee = Fee(1_000_000)`.
3. The check `if paid_fee == Fee(0)` evaluates to `false` (since `Fee(1) != Fee(0)`).
4. `execution_state.commit()` is called — state changes are committed.
5. The returned `TransactionExecutionInfo` has `receipt.fee = Fee(0)` (zeroed by `l1_handler_tx_execution_info`).
6. The sequencer has executed and committed an L1 handler transaction for which it received 1 wei but incurred 1,000,000 wei of execution cost — a direct economic loss with a wrong fee value in the committed receipt.
