### Title
Fork Difficulty Desynchronization via Mainchain Height Lookup in Dogecoin Difficulty Adjustment - (File: contract/src/dogecoin.rs)

### Summary
The Dogecoin difficulty adjustment function `get_next_work_required` uses `get_header_by_height` to retrieve the block at the start of the difficulty interval. This function reads exclusively from the mainchain index (`mainchain_height_to_header`), not from the fork's own chain. For any fork block at height N+2 or beyond (where the fork diverges at height N), the difficulty calculation silently substitutes the mainchain's block at height N for the fork's own block at height N, producing an incorrect `expected_bits` that the contract then enforces as valid.

### Finding Description

In `get_next_work_required` (`contract/src/dogecoin.rs`), after Dogecoin's Digishield protocol activates at block 145,000, the difficulty adjustment interval becomes 1 (every block). For each new block, `blocks_to_go_back = 1` and `height_first = prev_block_header.block_height - 1`. The function then calls:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [1](#0-0) 

`get_header_by_height` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [2](#0-1) 

This always returns the **mainchain** block at the given height, not the fork's ancestor. When a fork diverges at height N, the fork's block at height N differs from the mainchain's block at height N. For the fork block at height N+2, `height_first = N`, and the contract uses the mainchain's block at height N's timestamp as `first_block_time` instead of the fork's block at height N's timestamp.

The Digishield calculation then computes:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
``` [3](#0-2) 

Where `prev_block_header.block_header.time` is the fork's block at height N+1, and `first_block_time` is the **mainchain's** block at height N. The developer's own TODO comment at line 291 acknowledges this concern explicitly.

The analog to the original report is direct: just as `NFTMintSale` uses `nft.totalSupply()` (shared external state) instead of a local counter, this contract uses `mainchain_height_to_header[height_first]` (shared mainchain state) instead of walking the fork's own chain to find the correct ancestor.

### Impact Explanation

An attacker submitting a fork block at height N can set its timestamp to the maximum allowed value (up to `current_timestamp + MAX_FUTURE_BLOCK_TIME_LOCAL`), making it later than the mainchain's block at height N. For the fork block at height N+2, the contract computes `modulated_timespan = fork_N+1_time - mainchain_N_time`, which is larger than the correct `fork_N+1_time - fork_N_time`. After Digishield dampening (`/8`) and clamping, this produces a lower `expected_bits` (easier difficulty target) than the protocol requires. The contract then enforces this incorrect `expected_bits` as the valid target:

```rust
require!(
    expected_bits == block_header.bits,
    format!("Error: Incorrect target. ...")
);
``` [4](#0-3) 

A fork block with an artificially lowered difficulty target is accepted as valid. If the fork accumulates more `chain_work` than the mainchain, `submit_block_header_inner` triggers a reorg, corrupting the canonical chain state and invalidating previously confirmed SPV proofs.

### Likelihood Explanation

The attacker must submit valid PoW for the fork blocks, which requires real hashrate. However, the desynchronization persists for every fork block at height N+2 and beyond, compounding the difficulty reduction across the entire fork. The Digishield dampening limits the per-block effect (approximately 6% difficulty reduction per block under maximum timestamp manipulation), but this accumulates over a long fork. The entry path is the public `submit_blocks` function, accessible to any registered relayer. The developer's own TODO comment confirms this is an unresolved known concern, not a theoretical edge case.

### Recommendation

Replace the `get_header_by_height` call with a chain-walk that follows `get_prev_header` from `prev_block_header` back `blocks_to_go_back` steps. This traverses the fork's own ancestry rather than reading from the shared mainchain index, ensuring the difficulty calculation uses the correct ancestor regardless of whether the block is on the mainchain or a fork. This is precisely what the TODO comment at line 291 flags as the unverified assumption.

### Proof of Concept

1. Mainchain has blocks at heights 0…N. The mainchain block at height N has timestamp `T_main_N`.
2. Attacker submits a fork block at height N with timestamp `T_fork_N = T_main_N + Δ` (where `Δ > 0`, within the valid timestamp window).
3. Attacker submits a fork block at height N+1 with timestamp `T_fork_N+1 > T_fork_N`.
4. For the fork block at height N+2, the contract calls `get_header_by_height(N)`, returning the mainchain block with timestamp `T_main_N`.
5. `modulated_timespan = T_fork_N+1 - T_main_N = (T_fork_N+1 - T_fork_N) + Δ` — inflated by `Δ` relative to the correct value.
6. After Digishield dampening, `expected_bits` is lower (easier) than the protocol requires.
7. The attacker mines fork block N+2 satisfying the lowered `expected_bits`; the contract accepts it via the `expected_bits == block_header.bits` check.
8. The desynchronization repeats for every subsequent fork block (N+3, N+4, …), each using a mainchain block at `height_first` that diverges from the fork's own ancestry.
9. If the fork's accumulated `chain_work` exceeds the mainchain's, `reorg_chain` is triggered, replacing the canonical chain with the attacker's lower-PoW fork. [5](#0-4) [6](#0-5)
