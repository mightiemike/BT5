### Title
Gas-Griefed OOG Bypasses Imperfect Heuristic in `_executeSlowModeTransaction`, Permanently Dropping Queued Slow Mode Transactions — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint._executeSlowModeTransaction` deletes a slow mode transaction from the queue **before** attempting to process it inside a `try/catch` block. The bare `catch {}` relies on an explicitly acknowledged imperfect heuristic to distinguish OOG from normal reverts. An unprivileged caller can invoke the public `executeSlowModeTransaction()` with a crafted gas amount, cause OOG inside `processSlowModeTransaction`, bypass the heuristic, and permanently drop a queued slow mode transaction. For `DepositCollateral` slow mode transactions, this results in user tokens being permanently locked in the contract with no balance credit.

---

### Finding Description

In `_executeSlowModeTransaction`, the slow mode transaction is irrevocably deleted from the queue at line 194 **before** the `try/catch` block executes:

```solidity
SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
delete slowModeTxs[_slowModeConfig.txUpTo++];   // ← committed before try
``` [1](#0-0) 

The bare `catch {}` block then uses a gas-based heuristic to detect OOG:

```solidity
try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
    if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
        assembly { invalid() }
    }
    // try return funds now removed
}
``` [2](#0-1) 

The inline comment explicitly acknowledges the heuristic is imperfect:

> *"having `gasleft() <= gasRemaining / 2` buys us 44 nested calls before we miss out of gas errors; `1/2 ≈ (63/64)^44` … this is good enough for our purposes"* [3](#0-2) 

When the call stack inside `processSlowModeTransaction` reaches ≥44 frames, the gas returned to the outer frame after an OOG event can exceed `gasRemaining / 2`, causing the heuristic to silently swallow the OOG instead of calling `invalid()`. Because the slow mode tx was already deleted before the `try`, it is permanently lost with no retry path.

`executeSlowModeTransaction()` is an unrestricted external function callable by any address:

```solidity
function executeSlowModeTransaction() external {
    SlowModeConfig memory _slowModeConfig = slowModeConfig;
    _executeSlowModeTransaction(_slowModeConfig, false);
    ...
}
``` [4](#0-3) 

The `processSlowModeTransaction` path goes through `_delegatecallEndpointTx` → `endpointTx.delegatecall` → `EndpointTx.processSlowModeTransactionImpl`, which itself fans out to `clearinghouse.depositCollateral`, `clearinghouse.withdrawCollateral`, `clearinghouse.getHealth` (iterating over all products and spread pairs), `IOffchainExchange.dumpFees`, etc. The `getHealth` path in particular iterates over all registered spread pairs in a `while` loop, making the gas cost proportional to the number of active products. [5](#0-4) 

For `DepositCollateral` slow mode transactions, the token transfer from the user to the contract happens in `depositCollateralWithReferral` **before** the slow mode tx is queued:

```solidity
handleDepositTransfer(
    IERC20Base(spotEngine.getToken(productId)),
    msg.sender,
    uint256(amount)
);
// ... then enqueues the slow mode tx
``` [6](#0-5) 

If the corresponding slow mode tx is silently dropped, the tokens reside in the `Clearinghouse` contract but the user's on-chain balance is never credited.

---

### Impact Explanation

For `DepositCollateral` slow mode transactions: user tokens are transferred to the `Clearinghouse` at deposit time, but the balance credit (`spotEngine.updateBalance`) only occurs when the slow mode tx is processed. If the tx is silently dropped due to the heuristic bypass, the tokens are permanently locked — the user has no mechanism to recover them because the queue entry no longer exists.

For other slow mode transaction types (e.g., `WithdrawCollateral`, `LinkSigner`), the operation is silently skipped with no on-chain indication of failure, and the slow mode fee paid by the user is also lost.

---

### Likelihood Explanation

**Low-to-medium.** The heuristic is robust for shallow call stacks. Bypassing it requires the internal call stack inside `processSlowModeTransaction` to reach ≥44 frames, which is non-trivial but not impossible given the delegatecall chain (`Endpoint` → `endpointTx` delegatecall → `Clearinghouse` → `SpotEngine`/`PerpEngine` → spread iteration). An attacker cannot choose *which* slow mode tx to target — they process the queue head — but they can time the call to coincide with a victim's deposit tx being at the front of the queue, or submit their own expensive slow mode tx and then grief it. The attack requires precise gas calibration.

---

### Recommendation

1. **Move the delete after the try/catch**: Only delete the slow mode tx if `processSlowModeTransaction` succeeds. On failure, leave the tx in the queue for retry (or emit an event and skip it explicitly).
2. **Replace bare `catch {}` with `catch Error(string memory)` or `catch (bytes memory)`**: This avoids silently catching OOG panics, consistent with the mitigation recommended in the referenced external report.
3. **Bound the gas cost of `processSlowModeTransaction`**: Cap the number of products iterated in `getHealth` or enforce a per-tx gas budget before the `try`, so OOG inside the call is structurally impossible.

---

### Proof of Concept

1. Victim calls `depositCollateralWithReferral(subaccountName, productId, amount)`. Tokens are transferred to `Clearinghouse`; a `DepositCollateral` slow mode tx is enqueued at position `txCount`.
2. Attacker waits for the victim's tx to reach position `txUpTo` (queue head).
3. Attacker calls `executeSlowModeTransaction{gas: G}()` where `G` is calibrated so that:
   - The outer frame passes the initial `require` checks.
   - The 63/64 forwarded to `this.processSlowModeTransaction` is insufficient to complete `clearinghouse.depositCollateral` → `getHealth` (with many spread pairs), causing OOG at a nesting depth ≥44.
   - The gas returned to the outer frame after the OOG exceeds `gasRemaining / 2`, bypassing the `invalid()` branch.
4. The `catch {}` block silently swallows the error. The slow mode tx was already deleted at step 194.
5. `slowModeConfig.txUpTo` is incremented; the victim's deposit tx is permanently gone.
6. Victim's tokens remain in `Clearinghouse` with no balance credit and no recovery path.
