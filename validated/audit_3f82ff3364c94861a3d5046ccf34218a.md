### Title
Stale `node_map` in `TreeCache::restore()` Enables NodePtr Aliasing After Allocator Rollback — (`File: src/serde/tree_cache.rs`)

---

### Summary

`TreeCache::restore()` rolls back only `stack`, `serialized_nodes`, and `sentinel_entry`, but leaves `node_map` and `node_entries` intact. When the caller also restores the `Allocator` to a matching checkpoint, freed `NodePtr` indices are recycled for new nodes. The `TreeCache` then silently maps those new nodes to stale `NodeEntry` records (wrong SHA-1 tree hash, wrong serialized length), causing `find_path()` to emit back-reference paths that point to the wrong content. The deserialized CLVM tree differs from the intended one — a concrete integrity failure analogous to the ERC-20 `approve()` TOCTOU where an intermediate stale state is exploited between two operations.

---

### Finding Description

`TreeCache` maintains a `node_map: HashMap<NodePtr, u32>` that maps every `NodePtr` seen during `update()` to an index in `node_entries`. [1](#0-0) 

`TreeCacheCheckpoint` captures only three fields:

```rust
pub struct TreeCacheCheckpoint {
    stack: Vec<u32>,
    serialized_nodes: BitSet,
    sentinel_entry: Option<u32>,
}
``` [2](#0-1) 

`TreeCache::restore()` writes back only those three fields; `node_map` and `node_entries` are never trimmed:

```rust
pub fn restore(&mut self, st: TreeCacheCheckpoint) {
    // ... adjusts on_stack counters ...
    self.stack = st.stack;
    // ...
    self.serialized_nodes = st.serialized_nodes;
    if let Some(sentinel_entry) = st.sentinel_entry {
        self.node_map.insert(self.sentinel_node.unwrap(), sentinel_entry);
    }
}
``` [3](#0-2) 

`Serializer::restore()` calls `tree_cache.restore()` but never touches the `Allocator`:

```rust
pub fn restore(&mut self, state: UndoState) {
    self.read_op_stack = state.read_op_stack;
    self.write_stack = state.write_stack;
    self.tree_cache.restore(state.tree_cache);
    self.output.set_position(state.output_position);
    self.output.get_mut().truncate(state.output_position as usize);
}
``` [4](#0-3) 

When a caller then restores the `Allocator` to the same checkpoint, the freed `NodePtr` slots are recycled. The next `new_atom` / `new_pair` call reuses the same integer index. `TreeCache::update()` hits the `Entry::Occupied` branch for that `NodePtr` and silently reuses the stale `NodeEntry`:

```rust
let e = match self.node_map.entry(node) {
    Entry::Occupied(e) => {
        // already traversed — reuse cached entry
        let idx = *e.get();
        stack.push(idx);
        continue;
    }
    Entry::Vacant(e) => e,
};
``` [5](#0-4) 

The new node B is now bound to old node A's `NodeEntry` (wrong SHA-1 hash, wrong `serialized_length`). `find_path()` then returns a path to node A's serialized position for node B:

```rust
pub fn find_path(&self, node: NodePtr) -> Option<Vec<u8>> {
    let idx = *self.node_map.get(&node).expect("invalid node");
    if !self.serialized_nodes.is_visited(idx) { return None; }
    // ... builds path using entry_A's parents ...
``` [6](#0-5) 

The serializer emits a back-reference byte sequence pointing to node A's bytes for what is actually node B. Deserialization reconstructs node A's content in node B's position.

---

### Impact Explanation

The corrupted result is a serialized CLVM byte string that, when deserialized with `node_from_bytes_backrefs`, produces a structurally different tree than the one that was serialized. In the Chia blockchain context, this means a puzzle or generator program is stored/broadcast with incorrect content. Any node that re-serializes the same tree independently (without the stale cache) produces different bytes, causing a **consensus-level hash mismatch** on the serialized program. Coin IDs, puzzle hashes, and solution hashes all depend on the serialized form; a divergence here breaks spend validity checks across nodes.

---

### Likelihood Explanation

The trigger requires a caller that:
1. Uses the incremental `Serializer` across multiple `add()` calls (the designed use case for Chia's generator pipeline).
2. Calls `Serializer::restore()` to undo a serialization step.
3. Also calls `Allocator::restore_checkpoint()` to reclaim memory for the same range.
4. Allocates new nodes (which recycle the freed `NodePtr` indices).
5. Calls `ser.add()` again with those new nodes.

Steps 2–3 together are the intended pattern for the incremental serializer's undo feature combined with the allocator's GC/checkpoint feature. The `Serializer` API explicitly exposes `restore()` and returns `UndoState` for exactly this purpose. [7](#0-6)  The allocator's `restore_checkpoint` is used in `run_program`'s GC path. [8](#0-7)  A caller combining both — realistic in Chia's block generator — triggers the aliasing silently with no error.

---

### Recommendation

`TreeCacheCheckpoint` must also snapshot the length of `node_map` and `node_entries` (or a generation counter). `TreeCache::restore()` must truncate `node_entries` back to the checkpointed length and remove all `node_map` entries whose `NodeEntry` index is ≥ that length. Alternatively, `Serializer` should own or co-ordinate the `Allocator` checkpoint so that `node_map` entries are never left dangling after a combined undo.

---

### Proof of Concept

```rust
use clvmr::allocator::Allocator;
use clvmr::serde::incremental::Serializer;

let mut a = Allocator::new();
let sentinel = a.new_pair(clvmr::allocator::NodePtr::NIL,
                          clvmr::allocator::NodePtr::NIL).unwrap();

// Allocate node A ("hello") and take a checkpoint
let node_a = a.new_atom(b"hello").unwrap();          // NodePtr index = I
let alloc_cp = a.checkpoint();

// Serialize node A; TreeCache caches node_map[I] = entry_A (hash of "hello")
let mut ser = Serializer::new(Some(sentinel));
let (_, undo) = ser.add(&a, node_a).unwrap();

// Undo the serializer (rolls back stack/serialized_nodes but NOT node_map)
ser.restore(undo);

// Restore the allocator — frees node A, index I is now free
a.restore_checkpoint(&alloc_cp);

// Allocate node B ("world") — gets the SAME NodePtr index I
let node_b = a.new_atom(b"world").unwrap();
assert_eq!(node_a, node_b); // same NodePtr value

// Serialize node B: TreeCache::update() hits Occupied(I) → uses entry_A
// find_path() may return a path to "hello"'s bytes for "world"
let (done, _) = ser.add(&a, node_b).unwrap();
// The emitted bytes reference "hello" content, not "world"
```

The `assert_eq!(node_a, node_b)` holds because `NodePtr` is a plain index and the allocator recycles freed slots. The `TreeCache` silently treats node B as node A, producing a back-reference to the wrong content.

pub struct TreeCache {
    /// caches extra metadata about a tree of nodes. The value is an index into
    /// the node_entries vector.
    node_map: HashMap<NodePtr, u32>,

    /// The metadata for all nodes in the tree. This is like a shadow tree
    /// structure to the NodePtr one. The most important difference is that
    /// identical nodes are merged, using the same NodeEntry, and additional
    /// metadata is kept, such as the tree hash.
    node_entries: Vec<NodeEntry>,

    /// maps tree-hashes to the index of the corresponding NodeEntry in the
    /// node_entries vector. For any given tree hash, we're only supposed to
    /// have a single NodeEntry. There may be multiple NodePtr referring to
    /// the same NodeEntry (if they are identical sub trees).
    atom_lookup: HashMap<Bytes20, u32, RandomState>,

    /// maps left + right child indices to the index of the pair with those
    /// children. This is the atom_lookup counterpart for pairs
    pair_lookup: HashMap<u64, u32>,

    /// When deserializing, we keep a stack of nodes we've parsed so far, this
    /// stack is maintaining that same state, since that's what back-references
    /// are pointing into.
    stack: Vec<u32>,

    /// This records which NodeEntries have been serialized so far. When we look
    /// for back-references, we can only pick nodes in this set. nodes with
    /// small serialized length are not inserted. This set is built and
    /// updated as we serialize, to ensure we only include nodes that *can* be
    /// referenced.
    serialized_nodes: BitSet,

    /// if the sentinel node is set, we can't compute the tree hashes or
    /// serialized length for this node nor any of its ancestors. When calling
    /// update(), the tree is assumed to be placed at the sentinel node in the
    /// previous call to update()
    pub sentinel_node: Option<NodePtr>,

    /// We compute hash-trees using SHA-1 in order to determine whether the
    /// trees are identical or not. To mitigate malicious SHA-1 hash collisions,
    /// we salt the hashes
    salt: [u8; 8],
}
