### Title
Intra-batch epoch-ordering bypass in `AnnounceAccountCache::add_accounts` allows attacker to persist a stale announcement to `DBCol::AccountAnnouncements` — (`chain/network/src/announce_accounts/mod.rs`)

---

### Summary

`add_accounts` deduplicates only by exact `epoch_id` equality against `account_peers_broadcasted`. When a single `SyncRoutingTable` carries two valid `AnnounceAccount` entries for the same `account_id` with different `epoch_id`s, both survive the `AnnounceAccountRequest` filter (which compares each entry only against the single previously-broadcasted epoch, not against each other), and `add_accounts` unconditionally overwrites the first with the second. If the second entry has an older epoch, `DBCol::AccountAnnouncements` is left with a stale record. On the next node restart, that stale record is loaded and re-broadcast, violating the monotonic-epoch invariant.

---

### Finding Description

**Step 1 — `handle_sync_routing_table` builds a flat `old` map**

`get_broadcasted_announcements` returns a `HashMap<AccountId, AnnounceAccount>` — one entry per `account_id`. [1](#0-0) 

When `rtu.accounts` contains two entries for the same `account_id`, both are paired with the **same** `last_epoch` value (the single HashMap lookup result). Neither entry is compared against the other. [2](#0-1) 

**Step 2 — `AnnounceAccountRequest` handler filters only against `last_epoch`**

The handler iterates the list and, for each entry, checks only whether `announce_account.epoch_id > last_epoch`. It does **not** compare entries within the batch against each other. [3](#0-2) 

If `last_epoch` is `None` (account never broadcasted before), the epoch comparison is skipped entirely and both entries pass unconditionally (subject only to signature verification). [4](#0-3) 

**Step 3 — `add_accounts` overwrites without epoch ordering**

The deduplication guard checks only for exact `epoch_id` equality:

```rust
if inner.account_peers_broadcasted.get(account_id).map(|x| &x.epoch_id)
    == Some(epoch_id)
{
    continue;
}
``` [5](#0-4) 

After the first entry (epoch_N) is stored, `account_peers_broadcasted` holds epoch_N. The second entry (epoch_M, M ≠ N) fails the equality check and proceeds to overwrite both the in-memory cache and the DB: [6](#0-5) 

`set_account_announcement` is an unconditional write — no epoch comparison, no read-before-write: [7](#0-6) 

**Step 4 — Stale record survives restart**

On restart, `get_announce` loads from `DBCol::AccountAnnouncements` and populates `account_peers`: [8](#0-7) 

The stale epoch_M record is then re-broadcast via `sync_routing_table`: [9](#0-8)

### Impact Explanation

After the attack, `DBCol::AccountAnnouncements[account_id]` contains an older `epoch_id` and potentially a different `peer_id`. Any node that restarts will load and re-broadcast this stale record. Peers that have not yet seen the newer announcement will route validator messages (chunk requests, approvals, etc.) to the wrong `peer_id`. The divergent value is the `epoch_id` field (and associated `peer_id`) in the serialized `AnnounceAccount` stored under `DBCol::AccountAnnouncements`.

---

### Likelihood Explanation

The attack requires only a connected peer (no validator or admin privileges). Valid `AnnounceAccount` messages for multiple epochs of the same account are publicly gossiped and trivially collectible by any observer. The attacker simply replays two collected announcements in reverse chronological order in a single `SyncRoutingTable`. The attack is most effective against a fresh node (no previously broadcasted epoch), but also works whenever both epoch_M and epoch_N are both newer than the previously broadcasted epoch_K (M > K, N > K, N > M).

---

### Recommendation

In `add_accounts`, before overwriting, compare the incoming announcement's `epoch_id` against the currently stored one and skip if the incoming epoch is not strictly newer. This requires either calling `epoch_manager.compare_epoch_id` inside `add_accounts`, or pre-filtering the batch in `handle_sync_routing_table` to deduplicate by `account_id` (keeping only the entry with the greatest epoch) before passing to `AnnounceAccountRequest`.
