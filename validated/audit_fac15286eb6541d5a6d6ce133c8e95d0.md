Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks caller-controlled `owner` instead of `sender`, allowing allowlist bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual `msg.sender` of `addLiquidity`) and instead checks `owner`, which is a free parameter supplied by the caller. Any address not on the allowlist can bypass the deposit gate by naming any allowlisted address as `owner`. This breaks the pool admin's primary access-control invariant for restricted pools.

## Finding Description

`ExtensionCalling._beforeAddLiquidity` correctly encodes both `sender` (`msg.sender` from the pool) and `owner` (caller-supplied LP position owner) and passes them to the extension: [1](#0-0) 

Inside `DepositAllowlistExtension`, the first parameter (`sender`) is unnamed and discarded; only `owner` is checked against the allowlist: [2](#0-1) 

Because `owner` is freely chosen by the caller of `addLiquidity`, any non-allowlisted address can pass the guard by supplying any allowlisted address as `owner`. The allowlisted `owner` receives LP shares without having initiated or authorized the deposit.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller), not the second parameter: [3](#0-2) 

The inconsistency is structural: the deposit extension checks the wrong actor.

## Impact Explanation

The deposit allowlist is the pool admin's mechanism for restricting who may provide liquidity (e.g., KYC/compliance-gated or institutional-only pools). With this bug, any unprivileged address can deposit into a restricted pool by naming an allowlisted address as `owner`. LP shares are credited to the allowlisted `owner`, who can then call `removeLiquidity` to recover the tokens — effectively receiving a free token transfer from the attacker. The pool admin's access-control invariant is entirely broken. This is an **admin-boundary break**: a pool admin-configured guard is bypassed by an unprivileged path, with direct fund impact (unauthorized liquidity deposited, LP shares minted to a third party).

## Likelihood Explanation

- Requires only knowing one allowlisted address, which is observable on-chain via `AllowedToDepositSet` events or the public `allowedDepositor` mapping.
- No special permissions, flash loans, or oracle manipulation needed.
- Reachable by any EOA or contract with sufficient token balance to pay the liquidity callback.
- Fully repeatable; the bypass works on every call as long as one allowlisted address exists. [4](#0-3) 

## Recommendation

Check `sender` (the actual caller of `addLiquidity`) instead of `owner`, mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Update `setAllowedToDeposit` / `isAllowedToDeposit` documentation to clarify that the allowlist gates the **caller** of `addLiquidity`, not the LP position owner. [5](#0-4) 

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension`; pool admin calls `setAllowedToDeposit(pool, alice, true)`.
2. Bob (not allowlisted) calls `pool.addLiquidity(owner=alice, salt=0, deltas=..., callbackData=..., extensionData=...)`.
3. Pool calls `extension.beforeAddLiquidity(bob, alice, ...)` — `sender=bob` is discarded; `owner=alice` is checked.
4. `allowedDepositor[pool][alice]` → `true` → no revert; liquidity is added.
5. Alice receives LP shares; Bob's callback pays the tokens.
6. Alice calls `pool.removeLiquidity(owner=alice, ...)` and recovers the tokens.

Bob has deposited into a pool he is not authorized to access, and Alice has received a free token transfer. The allowlist is entirely ineffective against this pattern. [2](#0-1)
