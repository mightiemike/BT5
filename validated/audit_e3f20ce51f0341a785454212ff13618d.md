### Title
Pool Admin Can Reduce `OracleValueStopLossExtension` Timelock to Zero and Instantly Disable Drawdown Protection — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` uses a per-pool `timelock` to delay drawdown, decay, and high-watermark changes so LPs can react. However, the timelock value itself is changed through the same timelocked mechanism, and once it is reduced to `0`, every subsequent parameter change — including setting `drawdownE6 = 0` — can be proposed and executed in the same block with zero delay. This lets a malicious pool admin silently disable the stop-loss guard after the initial timelock elapses, leaving LP funds unprotected.

---

### Finding Description

`_afterTimelock` computes the execution deadline for every pending proposal by reading the **current** stored timelock:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`_requireElapsed` enforces the deadline with a strict-less-than check:

```solidity
// OracleValueStopLossExtension.sol line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
```

When `oracleStopLossConfig[pool_].timelock == 0`, `_afterTimelock` returns exactly `block.timestamp`. The check `block.timestamp < block.timestamp` is `false`, so `_requireElapsed` does **not** revert. Propose and execute can therefore be called in the same transaction.

The attack sequence:

1. Pool is deployed with `timelock = T` (e.g. 7 days) and `drawdownE6 = D` (e.g. 5 %). LPs deposit trusting the stop-loss.
2. Admin calls `proposeOracleStopLossTimelock(pool, 0)`. `executeAfter = now + T`.
3. After `T` seconds, admin calls `executeOracleStopLossTimelock(pool)`. `oracleStopLossConfig[pool].timelock` is now `0`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 0)`. `_afterTimelock` returns `block.timestamp + 0 = block.timestamp`. `executeAfter` is stored as `block.timestamp`.
5. Admin immediately calls `executeOracleStopLossDrawdown(pool)` in the same block. `_requireElapsed(block.timestamp)` passes.
6. `oracleStopLossConfig[pool].drawdownE6 = 0`.
7. In `_afterSwapOracleStopLoss`, the guard short-circuits: `if (drawdown == 0) return;`. All subsequent swaps bypass the stop-loss entirely.

Steps 4–6 can be batched into a single transaction, giving LPs zero blocks to observe and exit.

---

### Impact Explanation

Once `drawdownE6` is set to `0`, the `afterSwap` hook returns immediately without checking any watermarks. Swaps that drain LP value in any bin are no longer blocked by `OracleStopLossTriggered`. LPs who deposited relying on the stop-loss invariant ("value per share at oracle marks cannot fall faster than drawdown") lose that protection with no on-chain warning and no time to withdraw. This is a direct loss of LP principal protection — a medium-to-high impact depending on pool TVL.

---

### Likelihood Explanation

The attack requires the pool admin to be malicious and to wait out the initial timelock period `T`. However:
- `T` is chosen by the pool admin at pool creation and can be set to a short value (e.g. 1 hour).
- After `T` elapses, the remaining steps are atomic (single block).
- LPs monitoring the `OracleStopLossTimelockProposed` event for the timelock reduction have exactly `T` seconds to react — the same window they were already given — but receive **zero** additional warning before the drawdown is zeroed.

The NatSpec of `OracleValueStopLossExtension` explicitly states the timelock exists so "LPs can react," confirming this is a user-protection mechanism that the admin can circumvent.

---

### Recommendation

Enforce a **minimum floor** on the timelock value so it can never be reduced to zero:

```solidity
uint32 public constant MIN_TIMELOCK = 1 days; // or protocol-chosen floor

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    require(newTimelock >= MIN_TIMELOCK, "timelock below minimum");
    ...
}
```

Alternatively, compute the `executeAfter` for a timelock-reduction proposal using `max(currentTimelock, newTimelock)` so that reducing the timelock always takes at least as long as the new value would impose.

---

### Proof of Concept

```solidity
// 1. Deploy pool with timelock = 1 hours, drawdownE6 = 50_000 (5%)
// 2. LPs deposit
// 3. Admin reduces timelock to 0 (waits 1 hour)
vm.prank(admin);
ext.proposeOracleStopLossTimelock(pool, 0);
vm.warp(block.timestamp + 1 hours);
vm.prank(admin);
ext.executeOracleStopLossTimelock(pool);

// 4. Admin disables drawdown instantly (same block)
vm.prank(admin);
ext.proposeOracleStopLossDrawdown(pool, 0);
vm.prank(admin);
ext.executeOracleStopLossDrawdown(pool); // succeeds: block.timestamp < block.timestamp is false

// 5. Verify guard is disabled
assertEq(ext.oracleStopLossConfig(pool).drawdownE6, 0);
// afterSwap now returns immediately — LP funds unprotected
```

The root cause is at [1](#0-0)  where `_afterTimelock` reads the **current** (already-reduced) timelock, and at [2](#0-1)  where `_requireElapsed` uses a strict `<` that passes when `executeAfter == block.timestamp`.

The drawdown guard that is disabled is at [3](#0-2)  — `if (drawdown == 0) return` causes the entire `afterSwap` stop-loss check to be skipped.

The timelock-reduction proposal path is at [4](#0-3)  and the drawdown-zeroing path is at [5](#0-4) .

  function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
  }

  function executeOracleStopLossTimelock(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingTimelockExecuteAfter == 0) revert OracleStopLossNoPendingTimelock(pool_);
    _requireElapsed(sched.pendingTimelockExecuteAfter);
    uint32 timelock = sched.pendingTimelock;
    oracleStopLossConfig[pool_].timelock = timelock;
    (sched.pendingTimelock, sched.pendingTimelockExecuteAfter) = (0, 0);
    emit OracleStopLossTimelockSet(pool_, timelock);
  }
