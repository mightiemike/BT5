### Title
`"Chain reorg"` Log Omits Old Tip, New Tip, and Reorg Depth — Off-Chain Consumers Cannot Identify Which Verified Transactions Were Invalidated - (File: `contract/src/lib.rs`)

---

### Summary

When a chain reorganization is triggered inside `submit_block_header_inner`, the contract emits `log!("Chain reorg")` with no identifying fields. An off-chain bridge or application that cached a `verify_transaction_inclusion` result as `true` and watches this log to decide whether to re-verify cannot determine which blocks were reorganized. This is a direct analog of the missing-nonce pattern in the reference report: an on-chain event lacks the unique identifiers needed for off-chain code to correctly correlate the event with the affected state.

---

### Finding Description

Inside `submit_block_header_inner`, when a fork's accumulated chain-work exceeds the current main chain's chain-work, the contract emits a single bare log before executing the full reorganization:

```rust
// contract/src/lib.rs:563-565
if current_header.chain_work > total_main_chain_chainwork {
    log!("Chain reorg");
    self.reorg_chain(current_header, last_main_chain_block_height);
}
``` [1](#0-0) 

The log carries none of the following fields that are available at the call site:

| Missing field | Variable available at call site |
|---|---|
| Old main-chain tip hash | `self.mainchain_tip_blockhash` |
| New fork tip hash | `current_header.block_hash` |
| Old main-chain tip height | `last_main_chain_block_height` |
| New fork tip height | `current_header.block_height` |

After `reorg_chain` completes, `self.mainchain_tip_blockhash` is silently updated to the fork tip:

```rust
// contract/src/lib.rs:646
self.mainchain_tip_blockhash = fork_tip_hash;
``` [2](#0-1) 

The two verification entry points — `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` — are pure view functions that return a `bool` and emit no log of their own: [3](#0-2) 

An off-chain consumer that caches a `true` result from either function and relies on the "Chain reorg" log to decide whether to re-verify has no way to determine from the log alone which blocks were displaced. The log is the only on-chain signal that a reorg occurred.

---

### Impact Explanation

A bridge or cross-chain application that:

1. Calls `verify_transaction_inclusion[_v2]` for a transaction T in block B and receives `true`.
2. Releases funds or updates state based on that result.
3. Watches for "Chain reorg" logs to know when to re-verify.

…cannot determine from `log!("Chain reorg")` whether block B was part of the reorganized segment. If the bridge does not re-verify (because it cannot tell whether B was affected), it continues to treat T as confirmed even though B is no longer on the main chain. This allows a double-spend: the attacker receives funds from the bridge for a transaction that is no longer canonical, and can also spend the same UTXO on the new chain.

---

### Likelihood Explanation

The `submit_blocks` entry point is reachable by any trusted relayer — an unprivileged NEAR account that has staked and been accepted. Submitting a longer competing fork is the normal mechanism by which the contract handles Bitcoin chain reorganizations; no special privilege beyond relayer status is required to trigger the code path. The missing log fields are not compensated for anywhere else in the contract's event surface.

---

### Recommendation

**Short term:** Replace the bare `log!("Chain reorg")` with a structured log that includes all fields needed for off-chain consumers to identify the affected segment:

```rust
log!(
    "Chain reorg: old_tip={} new_tip={} old_height={} new_height={}",
    self.mainchain_tip_blockhash,   // old tip, before reorg_chain mutates it
    current_header.block_hash,
    last_main_chain_block_height,
    current_header.block_height,
);
self.reorg_chain(current_header, last_main_chain_block_height);
```

**Long term:** Audit every `log!` call in the contract against the fields that off-chain consumers (bridges, relayers, indexers) need to correctly correlate events with on-chain state changes. Document the required fields for each event type and enforce them in code review.

---

### Proof of Concept

1. Initialize the contract with a genesis block and submit a chain of N blocks (main chain tip = block M at height H).
2. A bridge calls `verify_transaction_inclusion` for transaction T in block M; the call returns `true`. The bridge records the result and releases funds.
3. An attacker (trusted relayer) submits a competing fork of N+1 blocks rooted at the same genesis, with higher cumulative chain-work. `submit_block_header_inner` detects `current_header.chain_work > total_main_chain_chainwork` and emits `log!("Chain reorg")` — with no old tip, no new tip, no heights.
4. `reorg_chain` executes: block M is removed from `mainchain_header_to_height` and `headers_pool`; the fork tip becomes the new `mainchain_tip_blockhash`.
5. The bridge's off-chain listener receives the "Chain reorg" log. Because the log contains no block hashes or heights, the bridge cannot determine whether block M was part of the reorganized segment.
6. The bridge does not re-call `verify_transaction_inclusion` for T. It continues to treat T as confirmed.
7. A subsequent call to `verify_transaction_inclusion` for T in block M would now panic with `"block does not belong to the current main chain"` — confirming that the on-chain state no longer supports the cached result — but the bridge never makes that call.
8. The attacker has received funds from the bridge for a transaction that is no longer on the canonical chain. [4](#0-3) [5](#0-4) [3](#0-2)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

    fn submit_block_header_inner(
        &mut self,
        current_header: ExtendedHeader,
        prev_block_header: &ExtendedHeader,
    ) {
        // Main chain submission
        if prev_block_header.block_hash == self.mainchain_tip_blockhash {
            // Probably we should check if it is not in a mainchain?
            // chainwork > highScore
            log!("Block {}: saving to mainchain", current_header.block_hash);
            // Validate chain
            assert_eq!(
                self.mainchain_tip_blockhash,
                current_header.block_header.prev_block_hash
            );

            self.store_block_header(&current_header);
            self.mainchain_tip_blockhash = current_header.block_hash;
        } else {
            log!("Block {}: saving to fork", current_header.block_hash);
            // Fork submission
            let main_chain_tip_header = self
                .headers_pool
                .get(&self.mainchain_tip_blockhash)
                .unwrap_or_else(|| env::panic_str("tip should be in a header pool"));

            let last_main_chain_block_height = main_chain_tip_header.block_height;
            let total_main_chain_chainwork = main_chain_tip_header.chain_work;

            self.store_fork_header(&current_header);

            // Current chainwork is higher than on a current mainchain, let's promote the fork
            if current_header.chain_work > total_main_chain_chainwork {
                log!("Chain reorg");
                self.reorg_chain(current_header, last_main_chain_block_height);
            }
        }

    /// The most expensive operation which reorganizes the chain, based on fork weight
    fn reorg_chain(&mut self, fork_tip_header: ExtendedHeader, last_main_chain_block_height: u64) {
        let fork_tip_height = fork_tip_header.block_height;
        if last_main_chain_block_height > fork_tip_height {
            // If we see that main chain is longer than fork we first garbage collect
            // outstanding main chain blocks:
            //
            //      [m1] - [m2] - [m3] - [m4] <- We should remove [m4]
            //     /
            // [m0]
            //     \
            //      [f1] - [f2] - [f3]
            for height in (fork_tip_height + 1)..=last_main_chain_block_height {
                let current_main_chain_blockhash = self
                    .mainchain_height_to_header
                    .get(&height)
                    .unwrap_or_else(|| env::panic_str("cannot get a block"));
                self.remove_block_header(&current_main_chain_blockhash);
                self.mainchain_height_to_header.remove(&height);
            }
        }

        // Now we are in a situation where mainchain is equivalent to fork size:
        //
        //      [m1] - [m2] - [m3] - [m4] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip
        //
        //
        // Or in a situation where it is shorter:
        //
        //      [m1] - [m2] - [m3] <- main tip
        //     /
        // [m0]
        //     \
        //      [f1] - [f2] - [f3] - [f4] <- fork tip

        let fork_tip_hash = fork_tip_header.block_hash.clone();
        let mut fork_header_cursor = fork_tip_header;

        while !self
            .mainchain_header_to_height
            .contains_key(&fork_header_cursor.block_hash)
        {
            let prev_block_hash = fork_header_cursor.block_header.prev_block_hash;
            let current_block_hash = fork_header_cursor.block_hash;
            let current_height = fork_header_cursor.block_height;

            // Inserting the fork block into the main chain, if some mainchain block is occupying
            // this height let's save its hashcode
            let main_chain_block = self
                .mainchain_height_to_header
                .insert(&current_height, &current_block_hash);
            self.mainchain_header_to_height
                .insert(&current_block_hash, &current_height);

            // If we found a mainchain block at the current height than remove this block from the
            // header pool and from the header -> height map
            if let Some(current_main_chain_blockhash) = main_chain_block {
                self.remove_block_header(&current_main_chain_blockhash);
            }

            // Switch iterator cursor to the previous block in fork
            fork_header_cursor = self
                .headers_pool
                .get(&prev_block_hash)
                .unwrap_or_else(|| env::panic_str("previous fork block should be there"));
        }

        // Updating tip of the new main chain
        self.mainchain_tip_blockhash = fork_tip_hash;
    }
