### Title
Division-by-Zero in `PerpEngine.socializeSubaccount` When `state.openInterest == 0` Blocks Liquidation Finalization ŌĆö (`core/contracts/PerpEngine.sol`)

---

### Summary

`PerpEngine.socializeSubaccount` unconditionally divides by `state.openInterest` when insurance cannot fully cover a negative `vQuoteBalance`. If `state.openInterest == 0` at that moment ŌĆö a reachable state when all participants have closed their positions in a product ŌĆö `MathSD21x18.div` reverts with `ERR_DIV_BY_ZERO`, causing the entire `liquidateSubaccountImpl` transaction to revert and permanently preventing finalization of the bankrupt subaccount until a new position is opened in that product.

---

### Finding Description

In `PerpEngine.socializeSubaccount`, after insurance is applied, if `balance.vQuoteBalance` is still negative the code socializes the residual loss across all open-interest holders:

```solidity
if (balance.vQuoteBalance < 0) {
    int128 fundingPerShare = -balance.vQuoteBalance.div(
        state.openInterest          // ŌåÉ division, no zero-guard
    );
``` [1](#0-0) 

`MathSD21x18.div` explicitly requires the denominator to be non-zero:

```solidity
function div(int128 x, int128 y) internal pure returns (int128) {
    unchecked {
        require(y != 0, ERR_DIV_BY_ZERO);
``` [2](#0-1) 

`state.openInterest` is the sum of `|balance.amount|` across **all** subaccounts for a product. [3](#0-2) 

`_finalizeSubaccount` is only reachable when `txn.productId == type(uint32).max`, and it first asserts that the liquidatee's `balance.amount == 0` for every perp product before calling `socializeSubaccount`:

```solidity
require(balance.amount == 0, ERR_NOT_FINALIZABLE_SUBACCOUNT);
``` [4](#0-3) 

Because the liquidatee's `balance.amount` is already 0, it contributes nothing to `openInterest`. If every other participant has also closed their position in that product, `state.openInterest == 0`. A subaccount can still carry a negative `vQuoteBalance` with `balance.amount == 0` ŌĆö this is the normal residual PnL from a previously closed position. The combination is therefore reachable without any privileged action.

The call chain is:

```
Endpoint ŌåÆ ClearinghouseLiq.liquidateSubaccountImpl
              ŌåÆ _finalizeSubaccount (productId == type(uint32).max)
                  ŌåÆ perpEngine.socializeSubaccount
                      ŌåÆ MathSD21x18.div(balance.vQuoteBalance, state.openInterest)
                          ŌåÆ REVERT (ERR_DIV_BY_ZERO)
``` [5](#0-4) 

---

### Impact Explanation

The revert propagates all the way up through `liquidateSubaccountImpl`, meaning:

1. The bankrupt subaccount cannot be finalized ŌĆö its negative `vQuoteBalance` is never zeroed out and the loss is never socialized.
2. Insurance funds that were already deducted in the `_finalizeSubaccount` flow are not committed, leaving accounting in an inconsistent intermediate state.
3. The protocol cannot recover from the insolvency of this subaccount until an unrelated third party happens to open a new position in the affected product, making `openInterest > 0` again. This is an externally-imposed precondition the protocol has no control over.
4. In the worst case (product being wound down, no new participants), the block is indefinite, leaving the protocol undercollateralized with no path to resolution.

This directly matches the Critical scope: *"liquidation ŌĆ” or spread handling that ŌĆ” leaves the protocol undercollateralized."*

---

### Likelihood Explanation

- Any product with low or zero open interest is susceptible ŌĆö this includes products being deprecated, newly listed products, or products after a market-wide position close.
- No privileged access is required. A liquidator simply calls `liquidateSubaccountImpl` with `productId == type(uint32).max` on a subaccount that satisfies the finalization preconditions.
- The state (`openInterest == 0`, `vQuoteBalance < 0`) is a natural consequence of normal trading activity and does not require adversarial setup.

---

### Recommendation

Add a zero-guard before the division in `socializeSubaccount`. If `state.openInterest == 0`, there are no other participants to socialize the loss onto; the only correct action is to absorb the remaining loss into the insurance fund (or write it off entirely):

```solidity
if (balance.vQuoteBalance < 0) {
    if (state.openInterest == 0) {
        // No open interest to socialize against; write off the residual loss.
        balance.vQuoteBalance = 0;
    } else {
        int128 fundingPerShare = -balance.vQuoteBalance.div(state.openInterest);
        state.cumulativeFundingLongX18 += fundingPerShare;
        state.cumulativeFundingShortX18 -= fundingPerShare;
        balance.vQuoteBalance = 0;
    }
}
``` [6](#0-5)

//          sender:     liquidatorSubaccount,

//          isEncodedSpread: false,
//          amount:     0
//      }));
// 6. Execution reaches PerpEngine.socializeSubaccount.
//    insurance < 500e18, so balance.vQuoteBalance remains < 0.
//    MathSD21x18.div(-residual, 0) ŌåÆ require(0 != 0) ŌåÆ REVERT "DBZ"
// 7. Assert: transaction reverts with "DBZ".
// 8. SubaccountA can never be finalized; protocol loss is never socialized.
