### Title
Missing Event Emission for `linkSigner` State Change — (File: `core/contracts/EndpointTx.sol`)

### Summary
The `linkSigner` operation in `EndpointTx.sol` mutates the `linkedSigners` mapping — a security-critical signer-authority state — without emitting any event. This is a direct analog to the original report's `trust`/`distrust` findings in `TrustedNodes.sol`, where the addition and removal of privileged signers lacked event emissions.

### Finding Description
In `EndpointTx.sol`, two separate code paths update `linkedSigners` with no event:

**Fast path** (`processTransactionImpl`, lines 576–590):
```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(...);
    validateSignedTx(...);
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
    // ← no event emitted
}
``` [1](#0-0) 

**Slow mode path** (`processSlowModeTransactionImpl`, lines 232–239):
```solidity
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
    // ← no event emitted
}
``` [2](#0-1) 

The `linkedSigners` mapping is read in `getLinkedSigner` and `getLinkedSignerOrNlpSigner`, which are used to authorize every signed transaction — including `WithdrawCollateral`, `MatchOrders`, `MintNlp`, `BurnNlp`, and `TransferQuote`. [3](#0-2) 

No `LinkedSignerSet` or equivalent event is defined anywhere in the interface hierarchy. The `IClearinghouseEventEmitter` interface only declares `ClearinghouseInitialized`, `ModifyCollateral`, and `Liquidation`. [4](#0-3) 

### Impact Explanation
The `linkedSigners` mapping is the signer-authority state for every subaccount. A linked signer can authorize withdrawals, trades, and quote transfers on behalf of the subaccount owner. Because no event is emitted when this mapping is written:

- Users cannot subscribe to on-chain events to detect when their subaccount's signing authority changes.
- Off-chain monitoring and alerting infrastructure cannot reconstruct the current or historical linked-signer state without replaying every sequencer batch.
- A compromised sequencer that injects a rogue `LinkSigner` transaction (signed by the subaccount owner's key, which the sequencer has seen) would leave no on-chain trace distinguishable from normal operation.

The corrupted state is concrete: `linkedSigners[subaccount]` (signer state), which directly controls who may authorize asset-moving transactions.

### Likelihood Explanation
Every `linkSigner` transaction — whether submitted via the sequencer fast path or the slow mode path — is affected. This is a design-level omission present since contract deployment. Any user who has ever linked a signer, or any monitoring system that tracks signer authority, is affected by the absence of this event.

### Recommendation
Define and emit a `LinkedSignerSet` event in both code paths:

```solidity
event LinkedSignerSet(bytes32 indexed subaccount, address indexed signer);
```

Emit it immediately after the `linkedSigners` assignment in both `processTransactionImpl` (line 590) and `processSlowModeTransactionImpl` (line 239).

### Proof of Concept
1. A user submits a `LinkSigner` transaction through `submitTransactionsChecked` (sequencer fast path) or `submitSlowModeTransaction` (slow mode).
2. `processTransactionImpl` (or `processSlowModeTransactionImpl`) decodes the transaction and writes `linkedSigners[sender] = newSigner`.
3. No event is emitted.
4. An off-chain monitoring system subscribed to contract events receives no notification.
5. The new signer can immediately authorize `WithdrawCollateral`, `MatchOrders`, and other asset-moving transactions on behalf of the subaccount, with no on-chain audit trail of the authority grant.
