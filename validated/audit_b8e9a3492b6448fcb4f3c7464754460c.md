### Title
Missing Slippage Protection in `MintNlp` and `BurnNlp` Instructions Allows Execution at Unfavorable Oracle Price — (File: `core/contracts/interfaces/IEndpoint.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The `MintNlp` and `BurnNlp` transaction structs contain no user-specified price limit or minimum output amount. The execution price (`oraclePriceX18`) is appended by the sequencer in the outer `SignedMintNlp`/`SignedBurnNlp` wrapper and is not constrained by anything the user signs. A user minting NLP with a fixed `quoteAmount` or burning NLP with a fixed `nlpAmount` has no on-chain mechanism to enforce a minimum received amount, directly mirroring the `coin_trade`/`raydium_trade` slippage vulnerability class.

---

### Finding Description

The `MintNlp` struct that the user signs contains only `sender`, `quoteAmount`, and `nonce`: [1](#0-0) 

The `BurnNlp` struct similarly contains only `sender`, `nlpAmount`, and `nonce`: [2](#0-1) 

Neither struct includes a `minNlpAmount` (for minting) or `minQuoteAmount` (for burning) field. The execution price is carried in the outer sequencer-supplied wrapper: [3](#0-2) 

The `oraclePriceX18` field in `SignedMintNlp` and `SignedBurnNlp` is not part of the inner struct the user signs — it is appended by the sequencer at submission time. When `EndpointTx` processes these transactions, it passes `signedTx.oraclePriceX18` directly to `clearinghouse.mintNlp` / `clearinghouse.burnNlp` without any user-defined bound check: [4](#0-3) [5](#0-4) 

The protocol defines `ERR_SLIPPAGE_TOO_HIGH` in `Errors.sol`, confirming that slippage enforcement was anticipated: [6](#0-5) 

However, this error string is **never referenced** in any contract logic — it is declared but never enforced anywhere in the production codebase.

---

### Impact Explanation

A user who submits a `MintNlp` transaction specifying `quoteAmount = X` has no on-chain guarantee about the minimum NLP tokens they will receive. The sequencer-supplied `oraclePriceX18` determines the conversion rate. If this price is stale, manipulated, or set adversarially, the user receives fewer NLP tokens than the fair-market rate for their deposited quote. Symmetrically, a `BurnNlp` user specifying `nlpAmount = Y` has no guarantee of a minimum quote amount returned. The corrupted state delta is the user's NLP balance (mint) or quote balance (burn) — a direct, measurable asset loss.

---

### Likelihood Explanation

The sequencer is a centralized, privileged entity that constructs and submits `SignedMintNlp`/`SignedBurnNlp` transactions. A compromised or malicious sequencer can set `oraclePriceX18` to any value. Even absent malice, a stale oracle price during high volatility causes the same outcome. Because the user's signed payload contains no price constraint, there is no on-chain defense layer between the user and an unfavorable execution price. The `ERR_SLIPPAGE_TOO_HIGH` error being defined but unused confirms this protection was intended but omitted.

---

### Recommendation

Add a `minNlpAmount` field to `MintNlp` and a `minQuoteAmount` field to `BurnNlp` so that users can specify acceptable output bounds as part of their signed transaction. In `Clearinghouse.mintNlp` and `Clearinghouse.burnNlp`, enforce these bounds against the computed output and revert with `ERR_SLIPPAGE_TOO_HIGH` if the condition is violated.

---

### Proof of Concept

1. User signs `MintNlp { sender: alice, quoteAmount: 10_000e18, nonce: 1 }` — no price limit included.
2. Sequencer constructs `SignedMintNlp { tx: <above>, signature: <sig>, oraclePriceX18: <deflated_price>, nlpPoolRebalanceX18: [...] }`.
3. `EndpointTx` calls `validateSignedTx` over the inner `MintNlp` struct (which has no price field), then passes the sequencer-controlled `oraclePriceX18` to `clearinghouse.mintNlp`.
4. Alice receives NLP tokens computed at the deflated price — fewer than fair market value — with no revert path available because no minimum output was encoded in her signed transaction.
5. The `ERR_SLIPPAGE_TOO_HIGH` guard that would have protected Alice is defined at `Errors.sol:90` but is never invoked anywhere in the contract suite. [7](#0-6) [8](#0-7) [6](#0-5)
