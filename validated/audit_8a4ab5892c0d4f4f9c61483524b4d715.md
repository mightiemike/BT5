Audit Report

## Title
`SwapAllowlistExtension` checks router address instead of end-user address, allowing allowlist bypass - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` as the direct caller of `MetricOmmPool.swap()`. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the **router contract address**, not the end user's EOA. The allowlist check therefore gates the router rather than the individual trader, so any user can bypass per-trader restrictions by calling the router, and individually-allowlisted users are blocked if the router itself is not allowlisted.

## Finding Description

**Call path:**

1. User EOA calls `MetricOmmSimpleRouter.exactInputSingle(params)` (`msg.sender` = user EOA).
2. Router calls `IMetricOmmPoolActions(params.pool).swap(...)` — `msg.sender` to the pool is now the **router address**.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` passing the router as `sender`. [1](#0-0) 

4. `ExtensionCalling._beforeSwap` encodes `sender` (= router) and dispatches to the extension. [2](#0-1) 

5. `SwapAllowlistExtension.beforeSwap` receives `sender` = router and checks:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [3](#0-2) 

Here `msg.sender` = pool, `sender` = **router**, so the check is `allowedSwapper[pool][router]` — never `allowedSwapper[pool][userEOA]`.

**Bypass scenario:** Pool admin allowlists the router address (the natural setup when the pool is meant to be accessible via the official router). Any non-allowlisted user EOA can then call `exactInputSingle` through the router and the check passes because the router is allowlisted, not the individual user.

**Denial scenario:** Pool admin allowlists specific user EOAs but does not allowlist the router. Those users are blocked from swapping via the router even though they are individually permitted.

**Existing guards are insufficient:** The `setAllowedToSwap` and `isAllowedToSwap` functions operate on `address swapper` which the pool admin intends to be end-user addresses, but the actual value checked at runtime is always the intermediary router address. [4](#0-3) 

## Impact Explanation
This breaks the core allowlist functionality: an unprivileged, non-allowlisted trader can execute swaps in a pool that the admin intended to restrict. The admin-boundary invariant (only allowlisted addresses may swap) is bypassed by any user who routes through `MetricOmmSimpleRouter`. This constitutes a broken core pool functionality and an admin-boundary break, meeting High severity under Sherlock thresholds.

## Likelihood Explanation
Exploitable by any unprivileged user with zero special conditions: they simply call `MetricOmmSimpleRouter.exactInputSingle` targeting a pool whose allowlist includes the router address. No privileged access, no special tokens, and no off-chain data are required. The router is a canonical periphery contract that pool admins are expected to allowlist, making this condition the default operational state.

## Recommendation
Pass the originating user address through the extension data or add a dedicated field. The simplest fix is to have the router encode `msg.sender` (the end user) into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and check that address instead of (or in addition to) `sender`. Alternatively, the pool's `swap` interface could accept an explicit `swapper` address that the router populates with `msg.sender`, and the extension checks that field.

## Proof of Concept

```solidity
// 1. Deploy pool with SwapAllowlistExtension configured.
// 2. Pool admin calls setAllowedToSwap(pool, address(router), true)
//    (allowlisting the router, not individual users).
// 3. Non-allowlisted attacker EOA calls:
//    router.exactInputSingle(ExactInputSingleParams({pool: pool, ...}));
// 4. Pool calls _beforeSwap(msg.sender=router, ...).
// 5. Extension checks allowedSwapper[pool][router] == true → passes.
// 6. Attacker completes swap despite not being individually allowlisted.

// Alternatively, to show denial:
// 2b. Pool admin calls setAllowedToSwap(pool, address(userEOA), true)
//     but does NOT allowlist the router.
// 3b. userEOA calls router.exactInputSingle(...).
// 4b. Extension checks allowedSwapper[pool][router] == false → reverts.
// 5b. Individually-allowlisted user is blocked from using the router.
```
