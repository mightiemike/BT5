Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension` is documented as gating `addLiquidity` by depositor address. It checks the `owner` parameter (LP-share recipient) while silently discarding `sender` (the actual caller who pays tokens). Because `MetricOmmPool.addLiquidity` lets any caller supply any `owner`, an unauthorized address can bypass the guard by nominating an allowlisted address as `owner`.

## Finding Description
`ExtensionCalling._beforeAddLiquidity` encodes the call as `(sender, owner, salt, deltas, extensionData)` where `sender = msg.sender` of `pool.addLiquidity()` (the token payer) and `owner` is the caller-supplied beneficiary: [1](#0-0) 

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` argument as `owner`, with no restriction on who may supply `owner`: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` discards `sender` entirely and checks only `owner`: [3](#0-2) 

Exploit path: attacker (not in allowlist) calls `pool.addLiquidity(allowlisted_address, ...)`. The extension receives `sender = attacker`, `owner = allowlisted_address`. It checks `allowedDepositor[pool][allowlisted_address]` — which is `true` — and allows the call. The attacker pays tokens and mints LP shares to the allowlisted address without authorization.

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual swapper), demonstrating the intended pattern: [4](#0-3) 

## Impact Explanation
Admin-boundary break: the pool admin's deposit allowlist is rendered ineffective. Any unprivileged EOA or contract can add liquidity to a permissioned pool by routing through an allowlisted `owner`. Secondary effects include unauthorized alteration of bin balances (token0/token1 per bin), which shifts per-share value metrics consumed by downstream extensions such as `OracleValueStopLossExtension`, and forced LP positions on third-party accounts without consent.

## Likelihood Explanation
Trivially reachable: no special role, no flash-loan, no multi-step setup. Any EOA or contract calls `pool.addLiquidity(allowlisted_address, ...)` directly. The bypass requires only knowledge of one allowlisted address, which is readable from the public `allowedDepositor` mapping.

## Recommendation
Check `sender` (the actual depositor) instead of `owner` (the share recipient), mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured.
2. Admin calls `setAllowedToDeposit(pool, alice, true)` — only `alice` is allowlisted.
3. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. Extension receives `sender = bob`, `owner = alice`; checks `allowedDepositor[pool][alice]` → `true`; call succeeds.
5. `bob` pays tokens; LP shares are minted to `alice`; allowlist is bypassed.

Foundry test: assert that `bob`'s `addLiquidity` call with `owner = alice` succeeds despite `bob` not being allowlisted, and that the same call with `owner = bob` reverts with `NotAllowedToDeposit`.
