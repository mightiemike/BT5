### Title
Fast Withdrawal Transfers Tokens Without Updating SpotEngine Balance, Enabling Double-Use of Collateral — (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal` transfers tokens to a user without updating the SpotEngine balance. The SpotEngine state is only decremented later when the sequencer processes the withdrawal through `Clearinghouse.withdrawCollateral`. During the window between the fast withdrawal and sequencer settlement, the user's SpotEngine balance remains at the pre-withdrawal level while they have already received the tokens, enabling double-use of the same collateral.

---

### Finding Description

In `BaseWithdrawPool.submitFastWithdrawal`, the function:

1. Validates signatures via `Verifier.requireValidTxSignatures`
2. Marks the index: `markedIdxs[idx] = true`
3. Transfers tokens to the user via `handleWithdrawTransfer(token, sendTo, transferAmount)` [1](#0-0) 

Critically, **no call to `spotEngine.updateBalance` or any SpotEngine state mutation occurs here**. The WithdrawPool pays the user from its own liquidity, but the user's SpotEngine balance is not decremented.

The SpotEngine balance is only decremented later when the sequencer processes the withdrawal through `Clearinghouse.withdrawCollateral`, which calls `spotEngine.updateBalance(productId, sender, amountRealized)`: [2](#0-1) 

At that point, `BaseWithdrawPool.submitWithdrawal` sees `markedIdxs[idx] == true` and returns early (no second token transfer), but the Clearinghouse still replenishes the pool and the SpotEngine balance is decremented: [3](#0-2) 

The gap between these two events is the vulnerability window. During this window, the SpotEngine state is inconsistent with the actual token balance: the user holds the tokens but the SpotEngine still records the full pre-withdrawal balance.

---

### Impact Explanation

During the window between `submitFastWithdrawal` and sequencer settlement:

- The user's SpotEngine balance is inflated (shows the full pre-withdrawal amount)
- The user can submit trade orders or borrow against this inflated balance before the sequencer decrements it
- `assertUtilization` and health checks operate on the stale (inflated) state, allowing operations that should be rejected

Concretely: a user with 1000 USDC deposited calls `submitFastWithdrawal` and receives 1000 USDC tokens. Their SpotEngine balance still reads 1000 USDC. They immediately submit a leveraged trade order. If the sequencer processes the trade before the withdrawal settlement, the trade opens against the full 1000 USDC balance. When the withdrawal is then settled, the balance drops to 0, leaving the position undercollateralized and the protocol absorbing the shortfall. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The attacker must possess a valid sequencer-signed withdrawal transaction (obtained by submitting a normal withdrawal request). Once obtained, `submitFastWithdrawal` is a `public` function with no additional access control. The attack window depends on sequencer latency, but the user controls when they call `submitFastWithdrawal` and can immediately follow it with other protocol interactions before the sequencer settles the withdrawal. [5](#0-4) 

---

### Recommendation

When `submitFastWithdrawal` is called, the user's SpotEngine balance should be decremented atomically at the same time tokens are transferred out. One approach: `submitFastWithdrawal` should call `clearinghouse.withdrawCollateral` (or an equivalent internal accounting function) to update the SpotEngine state immediately, rather than relying on the sequencer to do so later. Alternatively, a pending-withdrawal lock should be placed on the subaccount's balance so it cannot be used for other operations until the sequencer settles the withdrawal.

---

### Proof of Concept

1. User deposits 1000 USDC; SpotEngine balance = 1000 USDC.
2. User submits a `WithdrawCollateral` transaction; sequencer signs it (producing a valid `SignedWithdrawCollateral`).
3. User calls `BaseWithdrawPool.submitFastWithdrawal(idx, transaction, signatures)`.
   - `markedIdxs[idx] = true`
   - `handleWithdrawTransfer` sends 1000 USDC (minus fee) to user
   - SpotEngine balance **unchanged** = 1000 USDC
4. User immediately submits a trade order or borrow request using the 1000 USDC SpotEngine balance.
5. Sequencer processes the trade/borrow (balance still reads 1000 USDC → health check passes).
6. Sequencer later processes the withdrawal: `Clearinghouse.withdrawCollateral` → `spotEngine.updateBalance` decrements balance to 0.
7. User now holds 1000 USDC tokens **and** an open leveraged position/loan backed by 0 collateral. Protocol is left with an undercollateralized subaccount. [6](#0-5) [7](#0-6)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }

    function _updateBalanceNormalized(
        State memory state,
        BalanceNormalized memory balance,
        int128 balanceDelta
    ) internal pure {
        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized -= balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized += balance.amountNormalized;
        }

        int128 cumulativeMultiplierX18;
        if (balance.amountNormalized > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        int128 newAmount = balance.amountNormalized.mul(
            cumulativeMultiplierX18
        ) + balanceDelta;

        if (newAmount > 0) {
            cumulativeMultiplierX18 = state.cumulativeDepositsMultiplierX18;
        } else {
            cumulativeMultiplierX18 = state.cumulativeBorrowsMultiplierX18;
        }

        balance.amountNormalized = newAmount.div(cumulativeMultiplierX18);

        if (balance.amountNormalized > 0) {
            state.totalDepositsNormalized += balance.amountNormalized;
        } else {
            state.totalBorrowsNormalized -= balance.amountNormalized;
        }
    }
