### Title
Swap Allowlist Bypassed via Router `multicall`: Any User Can Swap Through an Allowlisted Router — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` checks the `sender` argument passed by the pool, which is always `msg.sender` of the `pool.swap()` call — i.e., the router address. Any user who calls `router.multicall([exactInputSingle(...)])` causes the pool to see `sender = router`. If the router is allowlisted, the check passes unconditionally, regardless of who the actual user is.

### Finding Description
`MetricOmmSimpleRouter.multicall` executes each payload via `delegatecall`, preserving the original caller's `msg.sender` for internal router logic (e.g., the callback payer stored in transient storage). However, when `exactInputSingle` calls `pool.swap(...)`, this is a regular external call — the pool sees `msg.sender = router`.

The pool passes this `msg.sender` (router) as the `sender` argument to `_beforeSwap`, which forwards it to `SwapAllowlistExtension.beforeSwap`. The extension evaluates:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. If `allowedSwapper[pool][router] = true`, the gate passes for **every user** who routes through that router, including those explicitly not allowlisted.

### Impact Explanation
The `SwapAllowlistExtension` is entirely ineffective for any pool that allowlists the router. A pool admin who intends to restrict swaps to a curated set of addresses (e.g., KYC'd users, whitelisted protocols) cannot enforce that restriction — any address can bypass it by calling `router.multicall`. This breaks the core pool curation guarantee and constitutes broken core functionality with direct fund impact: restricted pools can be drained or manipulated by unauthorized actors.

### Likelihood Explanation
The router is a public, permissionless contract. Any attacker can call `multicall` with an `exactInputSingle` payload. The only precondition is that the router itself is allowlisted on the target pool, which is the expected and documented deployment pattern (pool admins allowlist the router so legitimate users can swap through it). This precondition is met in every normal deployment.

### Recommendation
The `SwapAllowlistExtension` must check the **original user identity**, not the intermediary router. Two options:

1. **Pass the original user through `extensionData`**: The router encodes `msg.sender` (the actual user) into `extensionData`, and the extension decodes and checks it. This requires the extension to trust the router, which reintroduces a trust assumption.

2. **Check `tx.origin` or a dedicated "initiator" field**: The pool or router could pass the original transaction initiator as a separate argument to the extension. This is the cleaner fix — add an `initiator` field to the `beforeSwap` hook signature, populated by the router as `msg.sender` (captured before `delegatecall` context) and by direct callers as `msg.sender`.

The cleanest fix is to have the router pass the real user address explicitly in `extensionData` and have the extension decode it, with the router being a trusted forwarder.
