### Title
Checkpoint Advancement Without Payment in `BeforeRedelegationSlashed` Empty-Pool Path — (`x/tieredrewards/keeper/slash.go`, `x/tieredrewards/keeper/claim_rewards.go`)

### Summary

When `BeforeRedelegationSlashed` fires and the bonus pool is empty, `processEventsAndClaimBonus` returns `ErrInsufficientBonusPool` — but only **after** it has already (1) advanced `pos.LastEventSeq` and `pos.LastBonusAccrual` in-memory and (2) decremented event reference counts as **persistent on-chain state**. `slashRedelegationPosition` swallows the error and then calls `setPositionWithState` with the mutated `pos`, persisting the advanced checkpoints. The pre-slash bonus segment is permanently unrecoverable.

### Finding Description

The execution order inside `processEventsAndClaimBonus` is:

1. **Loop (lines 172–199):** For each pending event, `pos.UpdateLastEventSeq(entry.Seq)` advances `LastEventSeq` in-memory, and `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` writes a **persistent decrement** (or deletion if ref count reaches zero) to the `ValidatorEvents` store.
2. **Line 215:** `applyBonusAccrualCheckpoint(&pos.Position, blockTime)` advances `LastBonusAccrual` in-memory.
3. **Line 217:** `pos.UpdateLastKnownBonded(bonded)` updates in-memory.
4. **Line 230:** `k.sufficientBonusPoolBalance(ctx, bonusCoins)` — only here does the pool check occur, returning `ErrInsufficientBonusPool`. [1](#0-0) [2](#0-1) 

In `slashRedelegationPosition`, the `ErrInsufficientBonusPool` branch is explicitly swallowed:

```go

    if errors.Is(err, types.ErrInsufficientBonusPool) {
        k.logger(ctx).Error(...)  // swallowed
    } else {
        return err

// pos now has advanced LastBonusAccrual, LastEventSeq, LastKnownBonded

``` [3](#0-2) 

For a **partial slash**, `ClearBonusCheckpoints()` is never called, so the advanced checkpoints from the failed `processEventsAndClaimBonus` call are persisted verbatim. The pre-slash bonus segment `[old_LastBonusAccrual, slash_block_time]` is consumed (checkpoints advanced, events ref-counted down) without any payment.

`decrementEventRefCount` is a direct write to the `ValidatorEvents` collection — it is not rolled back when the error is swallowed: [4](#0-3) 

After pool replenishment, `MsgClaimTierRewards` calls `processEventsAndClaimBonus` starting from the now-advanced `pos.LastEventSeq` and `pos.LastBonusAccrual`. The pre-slash segment is gone — the events may have been garbage-collected and the accrual window has been closed.

### Impact Explanation

Direct, permanent economic loss of bonus rewards for the pre-slash period. The position owner cannot recover these rewards after pool replenishment because:
- `LastBonusAccrual` is advanced to the slash block time (segment start is moved forward)
- `LastEventSeq` is advanced past the relevant events (events are skipped on next replay)
- Event reference counts are decremented on-chain (events may be deleted, making replay impossible even if checkpoints were somehow restored)

The loss magnitude is: `shares × tokensPerShare × bonusApy × (slash_block_time − last_accrual) / SecondsPerYear` — proportional to position size, APY, and time since last claim.

### Likelihood Explanation

The precondition is: a position has an active redelegation mapping (created by `MsgTierRedelegate`) and the source validator is slashed while the bonus pool is empty or underfunded. The bonus pool can be empty during protocol bootstrapping, after a large claim wave, or if the pool has not been replenished. This is a realistic operational condition, not a contrived one. The position owner does not need to do anything beyond the normal `MsgTierRedelegate` flow.

### Recommendation

In `slashRedelegationPosition`, when `ErrInsufficientBonusPool` is caught, do **not** persist the checkpoint-advanced `pos`. Either:

**Option A:** Save the original checkpoint values before calling `processEventsAndClaimBonus` and restore them on the insufficient-pool path before calling `setPositionWithState`.

**Option B:** Restructure `processEventsAndClaimBonus` to perform the pool check **before** advancing checkpoints and decrementing ref counts (i.e., compute `totalBonus` first, check the pool, then — only if sufficient — advance checkpoints and decrement ref counts).

Option B is cleaner but requires refactoring the event-processing loop. Option A is a minimal targeted fix in `slashRedelegationPosition`.

func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}
