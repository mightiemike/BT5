### Title
`verify_transaction_inclusion_v2` Always Panics for Single-Transaction Blocks Due to Inherited Empty-Proof Guard in `verify_transaction_inclusion` — (`contract/src/lib.rs`)

### Summary

`verify_transaction_inclusion_v2` delegates to the deprecated `verify_transaction_inclusion` after performing its own coinbase-proof check. The inner function contains a hard `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard. For any Bitcoin block that contains exactly one transaction (the coinbase), the mathematically correct Merkle proof is the empty vector, because the Merkle root *is* the coinbase transaction hash. The v2 function passes all of its own validation steps for this case, then unconditionally panics inside the v1 call, making it impossible to verify any valid SPV proof for a single-transaction block.

### Finding Description

`verify_transaction_inclusion_v2` is the production-recommended entry point for SPV proof verification: [1](#0-0) 

It first checks that both proof vectors have equal length: [2](#0-1) 

Then it validates the coinbase Merkle proof against the stored Merkle root: [3](#0-2) 

Finally it delegates to the deprecated v1 function: [4](#0-3) 

Inside `verify_transaction_inclusion`, the first substantive check is: [5](#0-4) 

`compute_root_from_merkle_proof` with an empty proof simply returns the input hash unchanged (the loop body never executes): [6](#0-5) 

For a single-transaction block the Merkle root equals the coinbase transaction hash. A caller therefore supplies:
- `coinbase_merkle_proof = []`
- `merkle_proof = []`
- `coinbase_tx_id = tx_id = merkle_root`

Step-by-step through v2:
1. `merkle_proof.len() == coinbase_merkle_proof.len()` → `0 == 0` ✓
2. `compute_root_from_merkle_proof(coinbase_tx_id, 0, &[])` returns `coinbase_tx_id` which equals `header.block_header.merkle_root` ✓
3. `verify_transaction_inclusion` is called with `merkle_proof = []`
4. `require!(!args.merkle_proof.is_empty())` → **panic / revert**

The v2 function's own invariants are fully satisfied, yet it unconditionally fails because it inherits a restriction from the v1 function that was never updated to handle the single-transaction case.

### Impact Explanation

Any NEAR caller (bridge contract, dApp, end user) that submits a valid SPV proof for a coinbase transaction in a single-transaction block receives a contract panic instead of a boolean result. Downstream contracts that gate fund releases or cross-chain actions on the return value of `verify_transaction_inclusion_v2` will be permanently blocked for this class of blocks. The core invariant — that every valid Bitcoin transaction inclusion proof can be verified — is broken.

### Likelihood Explanation

Single-transaction blocks (coinbase only) occur on Bitcoin mainnet during periods of very low mempool activity and are a normal, consensus-valid block type. They are uncommon but not rare enough to be theoretical. Any relayer that submits such a block to the contract creates a permanent verification dead-zone for it: the block header is accepted and stored, but no proof against it can ever succeed.

### Recommendation

Remove the `require!(!args.merkle_proof.is_empty(), "Merkle proof is empty")` guard from `verify_transaction_inclusion`, or add an explicit early-return path: if `merkle_proof` is empty, return `transaction_hash == header.block_header.merkle_root` directly, which is the correct answer for a single-transaction block.

### Proof of Concept

1. Deploy the contract (Bitcoin feature) and submit a single-transaction block header whose `merkle_root` equals the coinbase `tx_id`.
2. Call `verify_transaction_inclusion_v2` with:
   - `tx_id = coinbase_tx_id = merkle_root`
   - `tx_index = 0`
   - `merkle_proof = []`
   - `coinbase_tx_id = merkle_root`
   - `coinbase_merkle_proof = []`
   - `confirmations = 1`
3. Observe the contract panics with `"Merkle proof is empty"` despite the proof being mathematically correct. [7](#0-6) [8](#0-7)

    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
