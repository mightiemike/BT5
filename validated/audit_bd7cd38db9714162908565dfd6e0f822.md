### Title
SwapAllowlistExtension gates the router address instead of the true economic actor, allowing any user to bypass the per-address swap allowlist via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`. When the call originates from `MetricOmmSimpleRouter`, `msg.sender` is the router contract, not the original user. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]`, so it checks whether the **router** is allowlisted, not whether the original caller is allowlisted. Any user can therefore bypass the per-address swap allowlist by routing through the public router.

### Finding Description

**Step 1 — Pool passes `msg.sender` as `sender` to the hook.**

In `MetricOmmPool.swap`, the `sender` forwarded to `_beforeSwap` is always `msg.sender` of the `swap` call: [1](#0-0) 

**Step 2 — Router calls `pool.swap` with itself as `msg.sender`.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly; the pool sees `msg.sender = router`: [2](#0-1) 

**Step 3 — Extension receives the router address as `sender`, not the original caller.**

`ExtensionCalling._beforeSwap` encodes and forwards the `sender` argument (which is the router) to the extension: [3](#0-2) 

**Step 4 — `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]`.**

The check uses `msg.sender` (= pool) as the pool key and `sender` (= router) as the swapper key: [4](#0-3) 

This means the extension never sees the original caller's address. The allowlist is keyed on the router, not on the economic actor.

### Impact Explanation

Two broken outcomes, both exploitable by an unprivileged user:

1. **Bypass (primary impact):** If the pool admin allowlists the router address (required for any router-mediated swap to succeed on that pool), then **every** user — including those not individually allowlisted — can swap through the router. The per-address allowlist is completely bypassed.

2. **Denial (secondary impact):** If the pool admin does not allowlist the router, then even individually allowlisted users cannot swap through the router, breaking expected router functionality.

The first outcome is the direct fund-impact path: an unauthorized trader executes swaps on a pool that was configured to restrict trading to a specific set of addresses.

### Likelihood Explanation

- `MetricOmmSimpleRouter` is a public, permissionless contract — any EOA or contract can call `exactInputSingle`.
- The pool admin has no way to allowlist individual users for router-mediated swaps; they can only allowlist the router as a whole.
- No special setup is required beyond the pool having `SwapAllowlistExtension` configured and the router being allowlisted.

### Recommendation

The pool should pass the original caller's identity through the call chain, or the router should encode the original `msg.sender` into `extensionData` and the extension should decode it. The simplest fix is for `MetricOmmPool.swap` to accept an explicit `sender` parameter (verified via callback or signature), or for `SwapAllowlistExtension` to read the true originator from a trusted field in `extensionData` when `msg.sender` (the pool's caller) is a known router.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Call `setAllowedToSwap(pool, router, true)` — allowlist the router (required for any router swap to work).
3. Do **not** allowlist address Y.
4. From address Y, call `MetricOmmSimpleRouter.exactInputSingle(...)` targeting that pool.
5. Observe: the swap succeeds because the extension checks `allowedSwapper[pool][router] == true`, never inspecting address Y.
6. Call `pool.swap(...)` directly from address Y (bypassing the router) — observe it reverts with `NotAllowedToSwap`, confirming Y is not individually allowlisted.

The router path succeeds; the direct path fails. The allowlist is bypassed. [5](#0-4) [6](#0-5)
