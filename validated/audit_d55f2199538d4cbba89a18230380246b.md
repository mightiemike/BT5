### Title
`check_if_descendant_of_tracked_shard_impl` Omits Self-Membership Check for Original Shards in `ShardLayoutV3`, Causing Nodes to Drop Shard Tracking After First Dynamic Resharding - (File: chain/epoch-manager/src/shard_tracker.rs)

### Summary

`check_if_descendant_of_tracked_shard_impl` takes an early-return path for `ShardLayoutV3` that only checks whether any *ancestor* of `shard_id` is in `tracked_shards`. It never checks whether `shard_id` itself is in `tracked_shards`. For original shards (those not created by a split), the ancestor list is empty, so the function always returns `false` — even when the shard is explicitly tracked. This is the nearcore analog of the external "head has no prev" bug: the boundary element (an original shard with no ancestors) is not handled as a special case.

### Finding Description

`check_if_descendant_of_tracked_shard_impl` in `chain/epoch-manager/src/shard_tracker.rs` has two code paths:

- **V0/V1/V2 layouts**: falls through to an explicit `tracked_shards.contains(&shard_uid)` self-check before walking layout history.
- **V3 layouts**: takes an early return via `shard_layout.ancestor_uids(shard_id)`. [1](#0-0) 

The V3 early-return path:

```rust
if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
    let ancestors = HashSet::from_iter(ancestors);
    return Ok(!ancestors.is_disjoint(&tracked_shards));
}
```

`ShardLayout::ancestor_uids` for V3 delegates to `ShardLayoutV3::ancestor_uids`: [2](#0-1) 

which reads from `shards_ancestor_map`: [3](#0-2) 

`validate_and_derive_shard_ancestor_map` inserts an entry for **every** shard in `shard_ids`, but for original shards (those not created by any split), the ancestor list is empty `[]`: [4](#0-3) 

So for an original shard `A`:
- `shard_layout.ancestor_uids(A)` → `Some([])`
- `ancestors` = empty `HashSet`
- `ancestors.is_disjoint(&tracked_shards)` = `true` (empty set is disjoint from everything)
- Returns `Ok(false)` — **even if `A` is explicitly in `tracked_shards`**

The V0/V1/V2 path correctly handles this with an explicit self-check: [5](#0-4) 

### Impact Explanation

When the first dynamic resharding event occurs (splitting shard B into B1 and B2), the shard layout transitions from V2 to V3. Shards that were not split (e.g., A and C) become "original shards" in the V3 layout with empty ancestor lists.

A node that was tracking shard A in epoch N calls `check_if_descendant_of_tracked_shard_impl(A, [A], epoch_N+2)` to determine whether to continue tracking A. The function returns `false`, causing the node to stop tracking shard A. The node then fails to apply chunks for shard A, breaking its ability to participate in consensus for that shard.

`DynamicResharding` is enabled at protocol version 85, below the current stable version 86: [6](#0-5) [7](#0-6) 

**Impact: High** — nodes silently drop shard tracking for all non-split shards after the first dynamic resharding, causing them to stop applying chunks for those shards.

### Likelihood Explanation

The bug is triggered by the first dynamic resharding event on any network running protocol version ≥ 85 with `DynamicResharding` enabled. The transition from V2 to V3 layout via `derive_with_layout_history` is the exact trigger: [8](#0-7) 

Any node tracking a shard that is not the one being split will be affected. Since most nodes track multiple shards and only one shard is split per epoch, the majority of tracked shards will be original shards in the new V3 layout.

### Recommendation

Add a self-membership check before the ancestor disjointness check in the V3 path, mirroring the V0/V1/V2 path:

```rust
if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
    // Check if the shard itself is tracked (a shard is a descendant of itself).
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        return Ok(true);
    }
    let ancestors = HashSet::from_iter(ancestors);
    return Ok(!ancestors.is_disjoint(&tracked_shards));
}
```

### Proof of Concept

Consider a V3 layout derived from a V2 layout where shard 0 was split into shards 1 and 2, and shard 2 is now being split into 3 and 4:

```
V3 layout: shard_ids = [1, 3, 4]
shards_split_map = {0 => [1, 2], 2 => [3, 4]}
last_split = 2
```

Shard 1 is an original shard (not the most recent split's child). Its ancestor list from `validate_and_derive_shard_ancestor_map` is `[0]` (shard 0 is its parent). But shard 0 is not in `tracked_shards` if the node only started tracking after the first split.

More directly: in the first V3 layout (only one split has occurred), shards that were never split at all have empty ancestor lists. A node tracking such a shard calls:

```
check_if_descendant_of_tracked_shard_impl(A, {A_uid}, epoch_with_V3_layout)
→ ancestor_uids(A) = Some([])
→ {}.is_disjoint({A_uid}) = true
→ Ok(false)   ← wrong, should be Ok(true)
``` [9](#0-8)
