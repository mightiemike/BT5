### Title
`DeclareTransaction V2` Compiled Class Hash Accepted Without Hash-Version Gate — (`crates/blockifier/src/transaction/transactions.rs`)

### Summary

The `block_casm_hash_v1_declares` versioned-constants flag that enforces Blake-V2 compiled class hashes is only applied to `DeclareTransaction::V3` and above. A `DeclareTransaction::V2` carrying a Poseidon-V1 compiled class hash bypasses the gate entirely, is accepted by the blockifier, and is committed to state with the stale V1 hash. This creates a permanent version-boundary mismatch: the state stores a V1 hash for a class that the OS migration machinery and the `check_compile_class_hash_v2_declaration` path both assume must hold a V2 hash once the flag is active.

### Finding Description

The `block_casm_hash_v1_declares` flag was introduced (versioned constants 0.14.1) to block new declarations that carry the old Poseidon-based compiled class hash (V1) and require the Blake-based hash (V2) instead. The enforcement is in `DeclareTransaction::run_execute`:

```rust
// crates/blockifier/src/transaction/transactions.rs  lines 184-188
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
```

The version guard is `>= TransactionVersion::THREE`. `DeclareTransaction::V2` has `version() == TransactionVersion::TWO`, so the condition is never true for V2 transactions. A V2 declare carrying a V1 (Poseidon) compiled class hash is accepted and written to state without any hash-version check.

`check_compile_class_hash_v2_declaration` (in `crates/starknet_api/src/executable_transaction.rs`, lines 228–244) computes `casm.hash(&HashVersion::V2)` and compares it to the field in the transaction. For a V2 declare with a V1 hash, this check is never invoked, so the V1 hash is committed to the `class_hash_to_compiled_class_hash` state map.

The gateway-side converter (`crates/apollo_transaction_converter/src/transaction_converter.rs`, lines 347–376) only handles `RpcDeclareTransaction::V3`; there is no RPC path for V2 in the new sequencer. However, the blockifier's `run_execute` is also called during re-execution, OS flow tests, and the native Python wrapper (`native_blockifier`), all of which accept `DeclareTransaction::V2` directly. The `should_migrate` logic in `CasmHashMigrationData::from_state` (`crates/blockifier/src/bouncer.rs`, lines 377–400) detects classes whose state-stored hash differs from the V2 hash and schedules them for migration. A V2-declared class with a V1 hash in state will be silently picked up by migration — but only if `enable_casm_hash_migration` is also active. If migration is disabled (as it is in versioned constants 0.14.2 where `enable_casm_hash_migration: false` but `block_casm_hash_v1_declares: true`), the class is permanently stuck with a V1 hash in state while the OS expects V2.

The analog to the external report is exact: the external bug hardcoded a predicate address that should have been queried dynamically; here, the hash-version enforcement is hardcoded to `>= V3` when it should be applied to all Cairo-1 declares (`>= V2`), because V2 declares also carry a `compiled_class_hash` field and are subject to the same hash-version invariant.

### Impact Explanation

When `block_casm_hash_v1_declares = true` and `enable_casm_hash_migration = false` (the combination present in versioned constants 0.14.2):

- A `DeclareTransaction::V2` with a V1 compiled class hash is accepted and committed.
- The class is stored in state with a Poseidon hash.
- The OS `migrate_classes_to_v2_casm_hash` function is not invoked (migration disabled).
- The OS expects all classes executed in a block to carry V2 hashes; a class with a V1 hash causes a state commitment mismatch.
- This produces a **wrong state commitment / class hash stored in state** — matching the "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" critical impact.

### Likelihood Explanation

`DeclareTransaction::V2` is a live transaction type. The versioned constants 0.14.2 file explicitly sets `block_casm_hash_v1_declares: true` with `enable_casm_hash_migration: false`. Any node running 0.14.2 constants that receives a V2 declare with a V1 hash will commit the wrong hash to state. The trigger requires only a well-formed V2 declare transaction with a Poseidon compiled class hash — no privileges needed.

### Recommendation

Change the version guard in `run_execute` from `>= TransactionVersion::THREE` to `>= TransactionVersion::TWO`, since `DeclareTransaction::V2` also carries a `compiled_class_hash` field and is subject to the same hash-version invariant:

```rust
// crates/blockifier/src/transaction/transactions.rs
if context.tx_context.block_context.versioned_constants.block_casm_hash_v1_declares
    && self.version() >= TransactionVersion::TWO   // was: THREE
{
    self.check_compile_class_hash_v2_declaration()?
}
```

This mirrors the fix in the external report: instead of hardcoding the predicate to a specific version, query the authoritative source (the versioned constants flag) for all applicable transaction versions.

### Proof of Concept

1. Enable versioned constants 0.14.2 (`block_casm_hash_v1_declares: true`, `enable_casm_hash_migration: false`).
2. Construct a `DeclareTransaction::V2` whose `compiled_class_hash` field is set to `casm.hash(&HashVersion::V1)` (Poseidon hash) instead of `casm.hash(&HashVersion::V2)`.
3. Execute the transaction through `AccountTransaction::execute`.
4. Observe: execution succeeds (no `DeclareTransactionCasmHashMissMatch` error).
5. Read `state.get_compiled_class_hash(class_hash)` — it returns the V1 Poseidon hash.
6. Compute `casm.hash(&HashVersion::V2)` — it differs from the stored value.
7. The state now holds a class with a stale V1 hash while the OS and migration machinery expect V2, causing a state commitment divergence.

The existing test at `crates/blockifier/src/transaction/transactions_test.rs` line 1875 confirms that `HashVersion::V1` with `TransactionVersion::THREE` panics with `DeclareTransactionCasmHashMissMatch` — but there is no analogous test for `TransactionVersion::TWO` with `HashVersion::V1`, which would pass silently.     pub fn check_compile_class_hash_v2_declaration(&self) -> Result<(), StarknetApiError> {
        let compiled_class = &self.class_info.contract_class;
        let compiled_class_hash_v2 = match &compiled_class {
            ContractClass::V0(_) => return Ok(()),
            ContractClass::V1((casm, _)) => casm.hash(&HashVersion::V2),
        };
        let compiled_class_hash = self.compiled_class_hash();
        if compiled_class_hash_v2 != compiled_class_hash {
            let err_var = CasmHashMismatch {
                hash: self.class_hash(),
                actual: compiled_class_hash,
                expected: compiled_class_hash_v2,
            };
            return Err(StarknetApiError::DeclareTransactionCasmHashMissMatch(Box::new(err_var)));
        }
        Ok(())
    }

    fn from_state<S: StateReader>(
        state_reader: &S,
        executed_class_hashes: &HashSet<ClassHash>,
        versioned_constants: &VersionedConstants,
    ) -> TransactionExecutionResult<Self> {
        if !versioned_constants.enable_casm_hash_migration {
            return Ok(Self::empty());
        }

        executed_class_hashes.iter().try_fold(Self::empty(), |mut migration_data, &class_hash| {
            if let Some((class_hash, casm_hash_v2_to_v1)) =
                should_migrate(state_reader, class_hash)?
            {
                // Add class hash mapping to the migration data.
                migration_data.class_hashes_to_migrate.insert(class_hash, casm_hash_v2_to_v1);

                // Accumulate the class's migration resources.
                let class = state_reader.get_compiled_class(class_hash)?;
                migration_data.resources +=
                    &class.estimate_compiled_class_hash_migration_resources();
            }
            Ok(migration_data)
        })
    }
