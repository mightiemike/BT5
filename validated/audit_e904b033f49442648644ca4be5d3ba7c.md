### Title
GC-Triggered Header Eviction Permanently Prevents SPV Proof Verification for Elapsed Blocks — (`contract/src/lib.rs`)

---

### Summary

`submit_blocks` automatically calls `run_mainchain_gc`, which permanently deletes old block headers from `headers_pool` and `mainchain_header_to_height`. Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` hard-panic if the target block is absent from either map. Once a block is GC-evicted, any downstream contract or user whose funds are gated behind an SPV proof for that block is permanently locked out — an exact structural analog to the veLACX bribe-loss bug.

---

### Finding Description

`submit_blocks` unconditionally calls `run_mainchain_gc` at the end of every submission: [1](#0-0) 

`run_mainchain_gc` computes how many headers exceed `gc_threshold` and deletes them oldest-first via `remove_block_header`: [2](#0-1) 

`remove_block_header` erases the block from **both** lookup maps and the pool: [3](#0-2) 

`verify_transaction_inclusion` then hard-panics on the first missing lookup: [4](#0-3) 

`verify_transaction_inclusion_v2` hits the same wall one step earlier — it reads `headers_pool` directly before delegating: [5](#0-4) 

The only guard in `verify_transaction_inclusion` is that `confirmations <= gc_threshold`, which prevents requesting *more* confirmations than the window size, but does **not** guarantee the target block is still within the live window at call time: [6](#0-5) 

There is no mechanism for a user to pin or reserve a block against GC eviction.

---

### Impact Explanation

Any NEAR contract or user that gates value release (bridged funds, cross-chain rewards, settlement) on a successful `verify_transaction_inclusion` call loses permanent access to that value once the target block is GC-evicted. The panic is not recoverable: the block hash is gone from both `mainchain_header_to_height` and `headers_pool`, so no future call can succeed. This matches the external report's impact class: **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The GC fires on every `submit_blocks` call. Bitcoin produces ~144 blocks per day; with a default `gc_threshold` of 52 704 (≈ one year), a block becomes evictable after roughly one year. However, the threshold is set at `init` and can be as low as the deployer chooses (the test suite uses `gc_threshold: 3`). Any relayer — an unprivileged NEAR account — can trigger eviction simply by submitting a batch large enough to push the chain past the threshold. No special role or key is required.

---

### Recommendation

1. **Add a minimum-height guard in `verify_transaction_inclusion`**: before the merkle check, assert that `target_block_height >= initial_blockheader.block_height` and emit a descriptive error (e.g., `"block has been garbage-collected"`) rather than panicking with a misleading "not in main chain" message.
2. **Expose the live window to callers**: add a view method returning `(mainchain_initial_height, mainchain_tip_height)` so downstream contracts can check whether a block is still verifiable before committing to a flow that depends on it.
3. **Consider a proof-submission deadline pattern**: allow callers to submit and cache a proof result on-chain while the block is still live, so the cached result survives GC.

---

### Proof of Concept

Sequence demonstrating permanent lock-out:

1. Relayer initialises the contract with `gc_threshold = N`.
2. User's Bitcoin transaction is confirmed in block at height `H`; user constructs a valid Merkle proof.
3. Relayer submits `N + 1` subsequent headers via `submit_blocks`. Inside that call, `run_mainchain_gc` fires and removes block `H` from `headers_pool` and `mainchain_height_to_header`.
4. User calls `verify_transaction_inclusion` (or `_v2`) with the valid proof for block `H`.
5. Contract panics at `mainchain_header_to_height.get(&args.tx_block_blockhash)` → `"block does not belong to the current main chain"`.
6. The call can never succeed again; any value gated on this proof is permanently frozen.

The relayer performing step 3 need not be malicious — normal chain progress over `gc_threshold` blocks produces the same outcome automatically.
