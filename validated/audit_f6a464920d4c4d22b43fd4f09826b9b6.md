### Title
Checkpoint Advance Before Pool Balance Check in `processEventsAndClaimBonus` Causes Permanent Loss of Accrued Bonus Rewards When Pool Is Insufficient — (`x/tieredrewards/keeper/claim_rewards.go`, `x/tieredrewards/keeper/slash.go`)

---

### Summary

`processEventsAndClaimBonus` mutates the position's bonus checkpoints and decrements validator event reference counts (deleting events) **before** it checks whether the `RewardsPoolName` pool has sufficient balance. When `slashRedelegationPosition` calls this function and the pool is insufficient, it swallows `ErrInsufficientBonusPool` and persists the fully-advanced checkpoint state. The accrued bonus is permanently unclaimable: the events are gone, the checkpoints are advanced, and no coins were transferred.

---

### Finding Description

**Step 1 — Entry point.**
`BeforeRedelegationSlashed` (hooks.go:128-130) is a staking hook that fires whenever the SDK's `SlashRedelegation` processes a redelegation entry. It unconditionally delegates to `slashRedelegationPosition`. [1](#0-0) 

**Step 2 — Error swallowing in `slashRedelegationPosition`.**
`slashRedelegationPosition` calls `processEventsAndClaimBonus` and explicitly catches `ErrInsufficientBonusPool`, logs it, and continues — it does **not** return the error. Execution falls through to `setPositionWithState` with whatever in-memory state `pos` was left in. [2](#0-1) 

**Step 3 — Mutations before the pool check in `processEventsAndClaimBonus`.**
Inside the event loop (lines 172–199), for every event since `pos.LastEventSeq`:
- `pos.UpdateLastEventSeq(entry.Seq)` advances the position's `LastEventSeq` in memory.
- `k.decrementEventRefCount(ctx, valAddr, entry.Seq)` decrements the event's `ReferenceCount` in **persistent storage** and deletes the event when the count reaches zero.

After the loop, `applyBonusAccrualCheckpoint` advances `pos.LastBonusAccrual` and `pos.UpdateLastKnownBonded` advances `pos.LastKnownBonded` — both in memory. Only **then**, at line 230, does the function check `sufficientBonusPoolBalance`. If the check fails, it returns `ErrInsufficientBonusPool` with all mutations already applied. [3](#0-2) [4](#0-3) 

**Step 4 — Permanent deletion of events.**
`decrementEventRefCount` deletes the event from `ValidatorEvents` when `ReferenceCount <= 1`. Once deleted, no future replay can recover the bonus for the segment those events covered. [5](#0-4) 

**Step 5 — Persisting the advanced state.**
`setPositionWithState` writes the in-memory `pos` (with advanced `LastEventSeq`, `LastBonusAccrual`, `LastKnownBonded`) to the `Positions` store. The next call to `processEventsAndClaimBonus` for this position will start from the advanced checkpoint and find no events to replay, returning zero bonus. [6](#0-5) 

---

### Impact Explanation

The invariant "bonus rewards owed to a position must either be paid or preserved for future claim" is violated. After the bug fires:
- `pos.LastEventSeq` is advanced past all processed events.
- All processed events are deleted from storage.
- `pos.LastBonusAccrual` is set to the current block time.
- No coins were sent from `RewardsPoolName` to the owner.
- Refilling the pool and calling `ClaimTierRewards` returns zero bonus — the accrual window is gone.

This is **permanent, irrecoverable loss** of all bonus rewards accrued since the position's last successful claim.

---

### Likelihood Explanation

The preconditions are:
1. A position owner has an active redelegation (created via `TierRedelegate`).
2. The destination validator is slashed while the redelegation is still in the unbonding window (standard Cosmos SDK behavior).
3. The `RewardsPoolName` module account balance is below the computed bonus at the moment of the slash.

Condition 3 is the binding constraint. The pool can be depleted by normal reward claims from many positions, by insufficient funding, or by a large position owner deliberately draining it before the slash fires. Validator slashes are not rare on a live chain. The combination is realistic, especially as the chain matures and the pool balance fluctuates.

---

### Recommendation

Move all persistent mutations — `decrementEventRefCount`, `pos.UpdateLastEventSeq`, `applyBonusAccrualCheckpoint`, and `pos.UpdateLastKnownBonded` — to **after** the `sufficientBonusPoolBalance` check and the `SendCoinsFromModuleToAccount` call succeed. If the pool is insufficient, return the error without mutating any state, so the position retains its original checkpoints and the events remain in storage for a future claim.

Alternatively, if the design intent is to skip bonus payment on pool exhaustion, the checkpoint advance and event decrements must also be skipped, and the position must be persisted with its **original** (pre-call) checkpoint values.

### Proof of Concept

```
(a) Create a tier position with a redelegation to validator V.
(b) Drain RewardsPoolName to 0 (or below the accrued bonus).
(c) Simulate a slash on V while the redelegation is in the unbonding window,

    processEventsAndClaimBonus.
(d) Assert that the persisted position has LastEventSeq and LastBonusAccrual
    advanced to the current block values.
(e) Assert that the validator events for V that were referenced by this
    position are deleted from ValidatorEvents storage.
(f) Refill RewardsPoolName with sufficient funds.
(g) Call ClaimTierRewards for the position owner.
(h) Assert that zero bonus is returned, confirming permanent loss.
```

```go
func (k Keeper) decrementEventRefCount(ctx context.Context, valAddr sdk.ValAddress, seq uint64) error {
	key := collections.Join(valAddr, seq)
	event, err := k.ValidatorEvents.Get(ctx, key)
	if errors.Is(err, collections.ErrNotFound) {
		return nil // already cleaned up
	}
	if err != nil {
		return err
	}

	if event.ReferenceCount <= 1 {
		return k.ValidatorEvents.Remove(ctx, key)
	}

	event.ReferenceCount--
	return k.ValidatorEvents.Set(ctx, key, event)
}
```
