Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Enabling Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` validates the `sender` argument forwarded by the pool, which equals `msg.sender` of the pool's `swap()` call. When a swap is routed through `MetricOmmSimpleRouter`, that caller is the router contract, not the end user. A pool admin who allowlists the router to enable legitimate user trading inadvertently grants every address on-chain the ability to swap against the restricted pool, completely defeating the allowlist's purpose.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then uses that value (`sender`) to look up the per-pool allowlist, where `msg.sender` is the calling pool: [2](#0-1) 

Every router entry-point calls `pool.swap()` directly, making the router the `msg.sender` seen by the pool:

- `exactInputSingle`: [3](#0-2) 
- `exactInput` (all hops): [4](#0-3) 
- `exactOutputSingle`: [5](#0-4) 
- `exactOutput` recursive hops (`_exactOutputIterateCallback`): [6](#0-5) 

The extension therefore evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][realUser]`. The only way for a pool admin to let curated users trade through the standard periphery is to allowlist the router, which simultaneously opens the pool to every address on-chain. There is no existing guard that recovers the original user identity — the `extensionData` field is caller-supplied and unverified, and no transient storage mechanism forwards the originating user to the extension.

## Impact Explanation
A pool protected by `SwapAllowlistExtension` is a curated venue where only vetted counterparties may trade against the LP's oracle-priced inventory. Once the router is allowlisted, any unprivileged address can call `exactInputSingle` (or any other router entry-point) and execute swaps against the pool's bins. LP principal is directly exposed to adversarial flow — informed traders, sandwich bots, or any non-allowlisted address — that the allowlist was specifically deployed to exclude. This constitutes a direct loss of LP assets above Sherlock thresholds whenever the pool carries meaningful liquidity, satisfying the "Admin-boundary break" and "Broken core pool functionality causing loss of funds" impact criteria.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the canonical, documented swap entry-point for end users. Any pool admin who wants allowlisted users to trade through the router must allowlist it. The bypass is therefore reachable on every production pool that uses `SwapAllowlistExtension` with router support — a natural and expected configuration. No special privileges, malicious setup, or non-standard tokens are required; a plain `exactInputSingle` call from any address suffices.

## Recommendation
Pass the original user through the call chain rather than the immediate `msg.sender`. Two complementary approaches:

1. **Router-side**: Store the originating user in transient storage (the payer mechanism via `_setNextCallbackContext` already demonstrates this pattern) and forward it as a dedicated `swapper` field inside `extensionData`. The extension decodes and checks that field instead of `sender`.

2. **Extension-side**: Accept an explicit `swapper` address in the extension payload and verify it against the allowlist, ignoring `sender` when a trusted router is the caller. The extension can verify the router is a known periphery contract (e.g., via a factory-registered router list) before trusting the embedded swapper identity.

Either approach must ensure the checked identity is the economically relevant actor — the address whose tokens are ultimately pulled — not the intermediate contract relaying the call.

## Proof of Concept
```
Setup
─────
1. Deploy pool with SwapAllowlistExtension as before-swap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true)
   — necessary so that allowlisted users can trade via the router.
3. Pool admin calls setAllowedToSwap(pool, alice, true)   // alice is the intended curated user.
   Pool admin does NOT allowlist bob.

Attack
──────
4. bob (non-allowlisted) calls:
     router.exactInputSingle({pool: pool, recipient: bob, amountIn: X, ...})

5. Router calls pool.swap(recipient=bob, ...) — msg.sender = router.

6. Pool calls extension.beforeSwap(sender=router, ...).

7. Extension evaluates:
     allowedSwapper[pool][router] == true  →  check passes.

8. Swap executes; bob receives output tokens from the LP's inventory.
   The allowlist provided zero protection.

Direct call (control)
─────────────────────
9. bob calls pool.swap(...) directly — msg.sender = bob.
10. Extension evaluates allowedSwapper[pool][bob] == false → reverts NotAllowedToSwap.
    Direct path is correctly blocked; router path is not.
```
