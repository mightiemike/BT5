### Title
`DepositAllowlistExtension` Guards the Wrong Actor (`owner` Instead of `sender`), Allowing Any Caller to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` enforces the allowlist against the `owner` parameter (the position beneficiary) rather than the `sender` parameter (the actual caller who pays for the deposit). Because `owner` is a free caller-supplied argument, any unprivileged address can bypass the guard by naming any already-allowlisted address as `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes two distinct addresses into the extension hook:

```
_beforeAddLiquidity(msg.sender /*sender*/, owner /*owner*/, salt, deltas, extensionData);
``` [1](#0-0) 

`sender` is the immutable `msg.sender` — the address that will be called back for payment and that initiated the transaction. `owner` is a caller-supplied argument that designates who will hold the resulting position.

`DepositAllowlistExtension.beforeAddLiquidity` silently discards `sender` (first parameter is unnamed) and checks only `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
``` [2](#0-1) 

The sibling `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller):

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
  }
  ...
}
``` [3](#0-2) 

The asymmetry is the root cause: the deposit guard checks the beneficiary, not the depositor.

---

### Impact Explanation

An unauthorized caller can:

1. Pick any address `A` that is already in `allowedDepositor[pool]`.
2. Call `pool.addLiquidity(owner = A, salt, deltas, callbackData, extensionData)`.
3. The extension evaluates `allowedDepositor[pool][A]` → `true` and does not revert.
4. The caller's own funds are pulled via the swap-callback on `msg.sender`; address `A` receives the position shares.

The deposit allowlist — intended to gate which principals may inject liquidity (e.g., for KYC/compliance, LP-tier restrictions, or protocol-controlled bootstrapping) — is fully neutralised. Any actor can deposit into a supposedly restricted pool. Because `removeLiquidity` enforces `msg.sender == owner`, the depositing attacker cannot directly recover the funds, but the allowlisted address `A` receives an unsolicited position it can immediately withdraw, effectively draining the attacker's payment. More critically, the pool's access-control invariant is broken: unauthorized liquidity enters the pool, altering bin balances, share totals, and oracle stop-loss watermarks for all existing LPs. [4](#0-3) 

---

### Likelihood Explanation

The bypass requires no special privilege, no flash loan, and no complex setup. Any address that can call `addLiquidity` on the pool (i.e., any EOA or contract) can exploit it by supplying a known allowlisted address as `owner`. Allowlisted addresses are discoverable on-chain via the `AllowedToDepositSet` event or direct storage reads. Likelihood is **High**.

---

### Recommendation

Change `beforeAddLiquidity` to check `sender` (the actual depositor / caller) instead of `owner`, mirroring the pattern used in `SwapAllowlistExtension`:

```solidity
// Before (vulnerable)
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  ...
}

// After (fixed)
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
  external view override returns (bytes4)
{
  if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToDeposit();
  }
  ...
}
``` [2](#0-1)

### Proof of Concept

```
Setup:
  pool configured with DepositAllowlistExtension
  allowedDepositor[pool][Alice] = true
  Bob is NOT allowlisted

Attack:
  Bob calls pool.addLiquidity(
    owner      = Alice,   // allowlisted — passes the guard
    salt       = 0,
    deltas     = <desired bins/shares>,
    callbackData = <Bob pays token0/token1 in the callback>,
    extensionData = ""
  )

Result:
  - Extension checks allowedDepositor[pool][Alice] → true → no revert
  - Bob's tokens are pulled via metricOmmSwapCallback on Bob
  - Alice receives position shares she did not request
  - Bob's funds are locked in Alice's position; Alice can removeLiquidity to collect them
  - The deposit allowlist is bypassed; unauthorized liquidity is now in the pool
``` [2](#0-1) [5](#0-4)
