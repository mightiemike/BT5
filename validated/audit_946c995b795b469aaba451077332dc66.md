### Title
ML-DSA-65 gas-key host-function fee paths pass borsh wire length instead of trie-id length, causing ~59× fee overcharge vs transaction path — (File: `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The `promise_batch_action_transfer_to_gas_key`, `promise_batch_action_add_gas_key_with_full_access`, and `promise_batch_action_add_gas_key_with_function_call` host functions pass the raw WASM-supplied `public_key_len as usize` (the borsh wire length) to `gas_key_transfer_send_fee`, `gas_key_transfer_exec_fee`, and `gas_key_add_key_exec_fee`. For ML-DSA-65 keys the wire length is **1953 bytes**, but the on-trie identifier is only **33 bytes** (a SHA3-256 hash). The transaction path correctly calls `action.public_key.trie_id_len()` (= 33). The host-function path never converts, so every per-byte gas component is inflated by a factor of ≈ 59 for ML-DSA-65 gas keys, breaking the fee-parity invariant the codebase explicitly documents and tests.

### Finding Description

`PostQuantumSignatures` (protocol version 85) introduced `PublicKey::MLDSA65`, whose borsh encoding is 1 + 1952 = **1953 bytes** on the wire but whose trie-key identifier is 1 + 32 = **33 bytes** (tag `3` + SHA3-256 digest). The design document and `access_key_storage_usage` both mandate `trie_id_len()` for every storage-cost path.

The fee functions themselves are correct — they accept a caller-supplied `public_key_len` and multiply it into per-byte costs:

`gas_key_transfer_send_fee` multiplies `gas_key_byte * public_key_len`: [8](#0-7) 

`gas_key_transfer_exec_fee` feeds `public_key_len` into `access_key_key_len` and multiplies the result into per-byte costs: [9](#0-8) 

`gas_key_add_key_exec_fee` feeds `public_key_len` into `access_key_key_len` and multiplies by `num_nonces`: [10](#0-9) 

The divergent value is exact: `1953` (wire) vs `33` (trie-id) for every ML-DSA-65 key.

The design document explicitly flags this as a required invariant:

> `PublicKey::trie_id_len()` is a new contract that all storage-cost code must respect. Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes. [11](#0-10) 

The host-function path is exactly such a caller — it never calls `trie_id_len()` at all; it forwards the raw WASM-supplied byte count.

### Impact Explanation

For every `TransferToGasKey` or `WithdrawFromGasKey` action targeting an ML-DSA-65 gas key emitted from a contract:

- **Send-fee per-byte component**: charged at `gas_key_byte × 1953` instead of `gas_key_byte × 33` — a **59× overcharge**.
- **Exec-fee per-byte component**: `access_key_key_len(account_id_len, 1953)` is ~1920 bytes wider than the real trie key, inflating the exec per-byte fee by the same factor.

For `AddKey` with a gas-key permission emitted from a contract, `gas_key_add_key_exec_fee` multiplies the inflated key length by `num_nonces`, compounding the overcharge linearly with the nonce count.

At the calibrated `gas_key_byte` rate the inflated fee for a single `TransferToGasKey` with an ML-DSA-65 key can exhaust the 300 Tgas per-receipt limit, making the operation **permanently unreachable via host functions** while the identical operation succeeds via a signed transaction. This is a broken protocol-compatibility invariant: the fee-parity guarantee documented and tested in `test_gas_key_fee_parity_full_access` / `test_gas_key_fee_parity_function_call` does not hold for ML-DSA-65 keys. [12](#0-11) 

### Likelihood Explanation

Both `GasKeys` and `PostQuantumSignatures` are stable features (the latter at protocol version 85). Any contract that manages ML-DSA-65 gas keys — a pattern the feature explicitly enables — will hit this path. The overcharge is deterministic and 100% reproducible; no adversarial input is required. A user simply needs to call the host function with a valid ML-DSA-65 public key.

### Recommendation

After `get_public_key` / `public_key.decode()?` resolves the `PublicKey` value, replace `public_key_len as usize` with `public_key.trie_id_len()` in all three call sites:

```rust
// promise_batch_action_transfer_to_gas_key
let decoded_pk = public_key.decode()?;
let send = gas_key_transfer_send_fee(&self.fees_config, sir, decoded_pk.trie_id_len());
let exec = gas_key_transfer_exec_fee(&self.fees_config, receiver_id.len(), decoded_pk.trie_id_len());

// promise_batch_action_add_gas_key_with_full_access / _function_call
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config, receiver_id.len(), decoded_pk.trie_id_len(), num_nonces,
);
```

Apply the same fix to the wasmtime runner (`runtime/near-vm-runner/src/wasmtime_runner/logic.rs`). Add a test asserting fee parity between the transaction and host-function paths for ML-DSA-65 keys, analogous to the existing `test_gas_key_fee_parity_*` tests.

### Proof of Concept

1. Protocol version ≥ 85 (both `GasKeys` and `PostQuantumSignatures` active).
2. Account `alice.near` holds an ML-DSA-65 gas key (added via a signed `AddKey` transaction — this succeeds because the transaction path uses `trie_id_len()`).
3. A contract calls `promise_batch_action_transfer_to_gas_key` targeting `alice.near`'s ML-DSA-65 key, passing the 1953-byte borsh-encoded pubkey.
4. Inside `promise_batch_action_transfer_to_gas_key`, `public_key_len = 1953`.
5. `gas_key_transfer_send_fee(..., 1953)` charges `gas_key_byte × 1953` per-byte send gas.
6. `gas_key_transfer_exec_fee(..., 1953)` computes `trie_key_len = access_key_key_len(account_id_len, 1953)` — ~1920 bytes wider than the real 33-byte trie key — and charges proportionally.
7. Total gas charged ≈ 59× the amount charged by the equivalent signed `TransferToGasKey` transaction.
8. The receipt exhausts its gas budget and fails with `GasExceeded`, while the signed-transaction path succeeds identically.

The exact divergent Borsh-length values are `1953` (host-function path, `PublicKey::MLDSA65` wire encoding) vs `33` (transaction path, `trie_id_len()` = tag + SHA3-256 hash), as confirmed by: [13](#0-12) [14](#0-13)

```rust
pub fn gas_key_transfer_send_fee(
```

```rust
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```

```rust
pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
}
```
