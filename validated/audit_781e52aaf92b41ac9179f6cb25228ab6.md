### Title
CEI Violation in `withdrawCollateral` Enables Reentrancy-Based Double Withdrawal — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.withdrawCollateral` performs the token transfer (external call) **before** updating the subaccount's SpotEngine balance (state update), violating the Checks-Effects-Interactions pattern. No `nonReentrant` guard exists anywhere in the contract suite. An attacker controlling the `sendTo` address with a token that triggers a callback (e.g., ERC777 `tokensReceived`) can reenter via the publicly callable `Endpoint.executeSlowModeTransaction` to process a pre-queued slow-mode withdrawal while the balance is still stale, withdrawing more collateral than their balance permits.

---

### Finding Description

In `Clearinghouse.withdrawCollateral`, the execution order is:

1. **Line 408** — `handleWithdrawTransfer(token, sendTo, amount, idx)`: external call chain → `token.safeTransfer(withdrawPool)` → `BaseWithdrawPool.submitWithdrawal` → `token.safeTransfer(sendTo, amount)` (tokens reach the attacker's contract here)
2. **Line 412** — `spotEngine.updateBalance(productId, sender, amountRealized)`: balance deduction (happens **after** the external call)
3. **Line 419** — `require(getHealth(sender, healthType) >= 0)`: health check [1](#0-0) 

The `handleWithdrawTransfer` helper in `Clearinghouse.sol` first transfers tokens to `withdrawPool`, then calls `submitWithdrawal`, which calls `token.safeTransfer(sendTo, amount)` — delivering tokens to the attacker-controlled `sendTo` address before any balance update occurs. [2](#0-1) 

Inside `BaseWithdrawPool.submitWithdrawal`, the final `handleWithdrawTransfer(token, sendTo, amount)` call is what triggers the hook on the recipient contract. [3](#0-2) 

A grep across all production Solidity files confirms **no `nonReentrant` guard or `ReentrancyGuard` import exists anywhere** in the contract suite.

The reentrancy entry path is `Endpoint.executeSlowModeTransaction`, which is **publicly callable** by anyone: [4](#0-3) 

This calls `_executeSlowModeTransaction` (with `fromSequencer = false`), which calls `this.processSlowModeTransaction` as an external call to `address(this)`. `processSlowModeTransaction` requires `msg.sender == address(this)` — satisfied — and then `delegatecall`s into `EndpointTx.processSlowModeTransactionImpl`, which dispatches a `WithdrawCollateral` transaction to `clearinghouse.withdrawCollateral`. Because this call originates from the Endpoint contract, the `onlyEndpoint` modifier on `withdrawCollateral` is satisfied. [5](#0-4) 

**Reentrancy window:** Between line 408 (token transfer) and line 412 (balance update), the SpotEngine balance still reflects the full pre-withdrawal amount. The second `withdrawCollateral` call (triggered via `executeSlowModeTransaction`) reads this stale balance, passes its own health check, deducts the balance to zero, and completes. Control then returns to the first `withdrawCollateral`, which deducts the balance again (to −100 USDC). If the attacker holds sufficient health from other positions (e.g., a profitable perp position), the final health check at line 419 passes, completing the double withdrawal.

---

### Impact Explanation

**High.** The broken invariant is: *a subaccount's SpotEngine balance must always reflect the collateral actually held by the Clearinghouse on its behalf.* By exploiting the stale balance window, an attacker can withdraw the same collateral twice, draining funds deposited by other users from the Clearinghouse. The `assertUtilization` call at line 413 does not prevent this because it checks aggregate product utilization, not per-subaccount negative balances; negative balances are permitted in Nado as a borrowing mechanism. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The exploit requires: (1) a supported collateral token with a transfer hook (ERC777 or a custom ERC20 with a `transfer` callback — the `ERC20Helper.safeTransfer` uses a raw `.call()` that would trigger any such hook); (2) the attacker to control the `sendTo` address (a contract they deploy); (3) a pre-queued slow-mode `WithdrawCollateral` transaction that has passed its 3-day delay; and (4) sufficient health from other positions (e.g., a perp position) to cover the resulting negative USDC balance. Conditions (2), (3), and (4) are fully attacker-controlled. Condition (1) depends on which tokens are listed as supported products — if any ERC777-compatible token is ever added, the exploit becomes directly reachable. [7](#0-6) 

---

### Recommendation

Follow the Checks-Effects-Interactions pattern: update the SpotEngine balance **before** calling `handleWithdrawTransfer`.

```diff
-        handleWithdrawTransfer(token, sendTo, amount, idx);
-
-        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
-        int128 amountRealized = -int128(amount) * int128(multiplier);
-        spotEngine.updateBalance(productId, sender, amountRealized);
-        spotEngine.assertUtilization(productId);
+        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
+        int128 amountRealized = -int128(amount) * int128(multiplier);
+        spotEngine.updateBalance(productId, sender, amountRealized);
+        spotEngine.assertUtilization(productId);
+
+        handleWithdrawTransfer(token, sendTo, amount, idx);
```

Additionally, add a `ReentrancyGuard` (`nonReentrant`) to `withdrawCollateral` and other state-mutating entry points as a defense-in-depth measure.

### Proof of Concept

1. Attacker deploys `AttackerContract` implementing ERC777 `tokensReceived(...)`.
2. Attacker deposits 100 USDC and opens a perp position providing ≥100 USDC of initial health.
3. Attacker calls `Endpoint.submitSlowModeTransaction` with a `WithdrawCollateral` for 100 USDC, `sendTo = AttackerContract`. The slow-mode transaction is queued with a 3-day delay.
4. After 3 days, the attacker requests the sequencer to include a `WithdrawCollateral` for 100 USDC with `sendTo = AttackerContract` in a batch.
5. Sequencer calls `submitTransactionsChecked` → `processTransactionImpl` → `clearinghouse.withdrawCollateral` (first call).
6. `handleWithdrawTransfer` → `token.safeTransfer(withdrawPool, 100)` → `submitWithdrawal` → `token.safeTransfer(AttackerContract, 100)`.
7. `AttackerContract.tokensReceived` fires → calls `Endpoint.executeSlowModeTransaction`.
8. `executeSlowModeTransaction` → `_executeSlowModeTransaction` → `this.processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.withdrawCollateral` (second call, **balance still = 100 USDC**).
9. Second withdrawal: 100 USDC transferred to `AttackerContract`; `spotEngine.updateBalance(sender, −100)` → balance = 0; health check passes (0 ≥ 0). `executeSlowModeTransaction` returns.
10. First withdrawal resumes: `spotEngine.updateBalance(sender, −100)` → balance = −100 USDC; health check: perp position provides ≥100 USDC health → passes.
11. Attacker has received **200 USDC** from a **100 USDC** deposit, with a −100 USDC spot balance covered by the perp position — effectively stealing 100 USDC from other users' Clearinghouse deposits. [8](#0-7) [4](#0-3) [3](#0-2)

    function withdrawCollateral(
        bytes32 sender,
        uint32 productId,
        uint128 amount,
        address sendTo,
        uint64 idx
    ) public virtual onlyEndpoint {
        require(!RiskHelper.isIsolatedSubaccount(sender), ERR_UNAUTHORIZED);
        require(amount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);
        ISpotEngine spotEngine = _spotEngine();
        IERC20Base token = IERC20Base(spotEngine.getConfig(productId).token);
        require(address(token) != address(0));

        if (sendTo == address(0)) {
            sendTo = address(uint160(bytes20(sender)));
        }

        handleWithdrawTransfer(token, sendTo, amount, idx);

        int256 multiplier = int256(10**(MAX_DECIMALS - _decimals(productId)));
        int128 amountRealized = -int128(amount) * int128(multiplier);
        spotEngine.updateBalance(productId, sender, amountRealized);
        spotEngine.assertUtilization(productId);

        IProductEngine.HealthType healthType = sender == X_ACCOUNT
            ? IProductEngine.HealthType.PNL
            : IProductEngine.HealthType.INITIAL;

        require(getHealth(sender, healthType) >= 0, ERR_SUBACCT_HEALTH);
        emit ModifyCollateral(amountRealized, sender, productId);
    }
