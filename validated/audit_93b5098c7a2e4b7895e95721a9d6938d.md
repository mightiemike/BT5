The vulnerability is confirmed. The `sender` parameter is silently dropped in `DepositAllowlistExtension.beforeAddLiquidity`, and only `owner` is checked — which is fully attacker-controlled.

---

### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any address to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` ignores the `sender` argument (the actual payer/operator) and gates only on `owner` (the LP-share recipient). Because `owner` is a free caller parameter in `pool.addLiquidity`, any address can pass an allowlisted address as `owner` and satisfy the check while the attacker's own tokens fund the position.

### Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` to `_beforeAddLiquidity`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` forwards both to the extension verbatim: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` declares the first parameter (`sender`) as an **unnamed, discarded** `address` and checks only `owner`: [3](#0-2) 

The allowlist mapping is keyed on `depositor`, and the NatDoc names the contract "Gates `addLiquidity` by depositor address" — the depositor is the party paying tokens, i.e., `sender`, not `owner`. The implementation checks the wrong address.

### Impact Explanation
Any address not on the allowlist can call `pool.addLiquidity(owner = allowedUser)`. The extension passes because `allowedUser` is in `allowedDepositor[pool]`. The attacker's tokens are pulled via the modify-liquidity callback (they are `msg.sender` in the pool), and LP shares are minted to `allowedUser`. The pool admin's deposit allowlist is completely ineffective at restricting who injects liquidity: any unprivileged address can fund positions in a curated pool, breaking pool curation. The `allowedUser` also receives unsolicited LP exposure they did not initiate.

Severity is **Medium**: the allowlist invariant is broken and pool curation is defeated, but the attacker has no financial gain (they pay their own tokens) and there is no direct loss of existing LP principal or protocol fees. The impact is broken core extension functionality rather than fund theft.

### Likelihood Explanation
Exploitable by any address with tokens approved to the pool. No special role, oracle access, or malicious setup is required — only knowledge of an allowlisted address (which is on-chain public state via `allowedDepositor`).

### Recommendation
Check `sender` (the actual operator/payer) instead of `owner` in `beforeAddLiquidity`:

```solidity
// current (wrong)
function beforeAddLiquidity(address, address owner, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}

// fixed
function beforeAddLiquidity(address sender, address, ...) external view override returns (bytes4) {
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
```
