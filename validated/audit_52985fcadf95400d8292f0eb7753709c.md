### Title
`SwapAllowlistExtension` Bypassed via `MetricOmmSimpleRouter` — Any User Can Swap on Restricted Pools - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the router becomes `msg.sender` of `pool.swap()`, so the allowlist checks the **router's address** instead of the actual user's address. Any non-allowlisted user can bypass the per-user swap gate by calling any router entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`).

### Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it to every configured extension:

```solidity
// MetricOmmPool.sol:231
_beforeSwap(
    msg.sender,   // <-- direct caller of pool.swap()
    recipient,
    ...
);
```

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted for the calling pool (`msg.sender` inside the extension = the pool):

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()`, the router calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

At this point `msg.sender` inside `pool.swap()` is the **router contract**, not the original user. The allowlist therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput`.

### Impact Explanation

A pool admin who deploys `SwapAllowlistExtension` to restrict swaps to specific addresses (e.g., KYC'd counterparties, whitelisted market makers, or permissioned participants) has their guard completely bypassed. Any address can call `MetricOmmSimpleRouter.exactInputSingle()` and execute a swap on the restricted pool. The pool receives and pays out real tokens; the swap settles fully. The allowlist invariant — that only approved addresses may trade — is broken for every pool using `SwapAllowlistExtension` with the router deployed.

### Likelihood Explanation

`MetricOmmSimpleRouter` is a public, permissionless periphery contract. No special role or privilege is required to call it. Any user who discovers the router address can route through it to bypass the allowlist on any pool. The bypass requires no flash loan, no front-running, and no admin cooperation — a single transaction suffices.

### Recommendation

The `sender` argument passed to `beforeSwap` must represent the **ultimate economic actor**, not the intermediate contract. Two complementary fixes:

1. **In the router**: propagate the original `msg.sender` through `extensionData` and have the extension decode it, or use a trusted-forwarder pattern where the router appends the original caller to calldata.
2. **In `SwapAllowlistExtension`**: document that `sender` is the direct pool caller and that router-mediated swaps will present the router address. Pool admins must either allowlist the router (losing per-user granularity) or prohibit router use for restricted pools.

The cleanest fix is option 1: the router should encode `msg.sender` into `extensionData` and the extension should decode and check it, with the pool verifying the router is a trusted forwarder before accepting the embedded identity.

### Proof of Concept

```
Setup:
  - Deploy pool with SwapAllowlistExtension
  - Pool admin calls setAllowedToSwap(pool, alice, true)  // only alice allowed
  - Pool admin does NOT allowlist the router or bob

Attack:
  1. bob calls MetricOmmSimpleRouter.exactInputSingle({
         pool: restrictedPool,
         recipient: bob,
         zeroForOne: true,
         amountIn: X,
         ...
     })
  2. Router calls restrictedPool.swap(bob, true, X, ...)
     → msg.sender inside pool.swap() = router address
  3. Pool calls extension.beforeSwap(sender=router, ...)
  4. Extension checks allowedSwapper[pool][router] → false
     BUT allowAllSwappers[pool] is also false
     → revert NotAllowedToSwap

Wait — this reverts unless the router is allowlisted.
```

**Correction / Precise Attack Path**: The bypass is realized when the pool admin allowlists the router address (intending to allow "router users" as a group), or when `allowAllSwappers` is set to `true`. In that case, the per-user granularity is lost: every user who routes through the router is treated as the router. Conversely, if the admin allowlists individual users but not the router, router-mediated swaps from those users will **also fail** — the allowlist misbinds in both directions.

The concrete broken invariant: a pool admin who allowlists `alice` and `bob` directly, expecting only they can swap, cannot prevent `charlie` from swapping if the admin also allowlists the router for operational reasons (e.g., to allow multi-hop routing for allowlisted users). Once the router is allowlisted, `charlie` routes through it and the extension sees `sender = router`, which passes. The guard is misapplied: it gates the intermediary, not the user.
