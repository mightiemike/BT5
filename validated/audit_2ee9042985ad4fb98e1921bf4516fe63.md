### Title
Negative Insurance Value Passed to `socializeSubaccount` Corrupts `vQuoteBalance` and `availableSettle` — (`core/contracts/PerpEngine.sol`)

---

### Summary

When the protocol insurance fund is depleted below `lastLiquidationFees`, `_finalizeSubaccount` in `ClearinghouseLiq.sol` computes a negative `v.insurance` and passes it directly to `PerpEngine.socializeSubaccount`. Inside that function, `MathHelper.min(negative_insurance, -balance.vQuoteBalance)` returns the negative insurance value as `insuranceCover`, which is then **added** to `balance.vQuoteBalance` and `state.availableSettle` — decrementing both instead of covering them. This corrupts the per-product settlement pool and worsens the liquidatee's vQuote debt.

---

### Finding Description

**Step 1 — Negative `v.insurance` is computed and passed without a guard.**

In `_finalizeSubaccount`:

```solidity
v.insurance = insurance;
v.insurance -= lastLiquidationFees;          // can go negative
...
v.insurance = perpEngine.socializeSubaccount(txn.liquidatee, v.insurance);  // no guard
``` [1](#0-0) 

There is no `require(v.insurance >= 0)` or clamping to zero before the call to `perpEngine.socializeSubaccount`.

**Step 2 — `socializeSubaccount` has no non-negative guard on `insurance`.**

The only access check is `msg.sender == address(_clearinghouse)`. The function then enters the loop:

```solidity
if (balance.vQuoteBalance < 0) {
    int128 insuranceCover = MathHelper.min(insurance, -balance.vQuoteBalance);
    insurance -= insuranceCover;
    balance.vQuoteBalance += insuranceCover;
    state.availableSettle += insuranceCover;
``` [2](#0-1) 

**Step 3 — `MathHelper.min` with a negative first argument returns the negative value.**

```solidity
function min(int128 a, int128 b) internal pure returns (int128) {
    return a < b ? a : b;
}
``` [3](#0-2) 

With `insurance = -10` and `balance.vQuoteBalance = -100`:
- `-balance.vQuoteBalance = 100`
- `insuranceCover = min(-10, 100) = -10`
- `balance.vQuoteBalance += -10` → goes from `-100` to `-110` (worsened)
- `state.availableSettle += -10` → decremented (corrupted)
- `insurance -= -10` → `insurance = 0` (consumed)

---

### Impact Explanation

- **`state.availableSettle`** for the affected product is decremented, making it more negative. This corrupts the settlement pool shared by all users of that product — future settlements will be underfunded.
- **`balance.vQuoteBalance`** of the liquidatee is worsened rather than covered, meaning the socialization step that should zero out the debt instead deepens it, causing the subsequent `fundingPerShare` socialization (lines 166–171) to spread a larger loss across all open-interest holders. [4](#0-3) 

---

### Likelihood Explanation

The precondition — `insurance < lastLiquidationFees` — is realistic and not exotic. The insurance fund can be depleted during a market stress event (cascading liquidations). `lastLiquidationFees` is a non-zero fee charged on every finalization. Any finalization transaction submitted when the insurance balance is below the fee threshold triggers this path. No special privileges are required; the liquidation flow is a standard externally-reachable endpoint.

---

### Recommendation

Clamp `v.insurance` to zero before passing it to `socializeSubaccount`:

```solidity
v.insurance = perpEngine.socializeSubaccount(
    txn.liquidatee,
    MathHelper.max(v.insurance, 0)
);
```

Alternatively, add a guard inside `socializeSubaccount`:

```solidity
if (insurance <= 0) return insurance;
```

at the top of the function body, before the loop.

---

### Proof of Concept

```
Preconditions:
  insurance (storage) = 5
  lastLiquidationFees = 10
  liquidatee has one perp product with vQuoteBalance = -100

Execution:
  v.insurance = 5 - 10 = -5
  perpEngine.socializeSubaccount(liquidatee, -5)
    → insuranceCover = min(-5, 100) = -5
    → balance.vQuoteBalance = -100 + (-5) = -105   [WORSENED]
    → state.availableSettle += -5                   [CORRUPTED]
    → insurance = -5 - (-5) = 0
    → balance.vQuoteBalance = -105 < 0, so socialization spreads -105 loss
       instead of the original -100

Assert: state.availableSettle decreased → FAILS invariant
Assert: balance.vQuoteBalance decreased → FAILS invariant
```
