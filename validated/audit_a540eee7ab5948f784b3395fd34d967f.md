### Title
Liquidation DoS via Precise Amount Frontrunning Blocks Full Liquidations - (`core/contracts/ClearinghouseLiq.sol`)

---

### Summary

Any unprivileged user can submit a tiny `LiquidateSubaccount` transaction to the sequencer before a legitimate liquidator's full-liquidation transaction. Because `_assertLiquidationAmount` enforces `maxLiquidatable >= txn.amount` using the **live** position size at execution time, the attacker's prior partial liquidation reduces the position below the legitimate liquidator's specified amount, causing the legitimate liquidator's transaction to revert with `ERR_NOT_LIQUIDATABLE_AMT`.

---

### Finding Description

In `ClearinghouseLiq.liquidateSubaccountImpl()`, after confirming the liquidatee is under maintenance health, the function calls `_assertLiquidationAmount()`: [1](#0-0) 

Inside `_assertLiquidationAmount`, the final guard is: [2](#0-1) 

`maxLiquidatable` is derived from the **current** live position of the liquidatee at the moment of execution:

- For spot long positions: `maxLiquidatable = spotAmount` [3](#0-2) 
- For perp positions: `maxLiquidatable = perpAmount` [4](#0-3) 

The `LiquidateSubaccount` struct requires the liquidator to commit to a precise `amount` at signing time: [5](#0-4) 

Any user can sign a `LiquidateSubaccount` transaction and submit it to the sequencer's off-chain queue. The sequencer processes transactions in submission order. If an attacker submits a tiny liquidation transaction before the legitimate liquidator's transaction, the sequencer will process the attacker's transaction first, reducing the liquidatee's position. The legitimate liquidator's transaction then fails because `maxLiquidatable < txn.amount`.

**Attack flow:**
1. Alice's position of `X` units becomes unhealthy (maintenance health < 0).
2. Bob (legitimate liquidator) signs `LiquidateSubaccount(liquidatee=Alice, amount=X)` and submits to the sequencer.
3. Eve (attacker) signs `LiquidateSubaccount(liquidatee=Alice, amount=1)` (or `sizeIncrement` for perp) and submits to the sequencer **before** Bob's transaction.
4. Sequencer processes Eve's transaction first: Alice's position is now `X-1`.
5. Sequencer processes Bob's transaction: `_assertLiquidationAmount` checks `maxLiquidatable (X-1) >= txn.amount (X)` → **reverts with `ERR_NOT_LIQUIDATABLE_AMT`**.

The attacker's cost is only the `LIQUIDATION_FEE` of $1 per attempt: [6](#0-5) 

The fee is charged in `processTransactionImpl` before the liquidation executes: [7](#0-6) 

---

### Impact Explanation

Full liquidations are blocked. Liquidators are forced into repeated partial liquidations, leaving dust debt in the protocol. Unhealthy subaccounts that cannot be fully liquidated in a single transaction remain partially undercollateralized for longer, threatening the solvency of the protocol's insurance fund and the stability of the clearinghouse. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** Liquidatable positions are fully observable on-chain (health is a public computation). Any user can sign and submit a `LiquidateSubaccount` transaction to the sequencer. The attacker only needs to submit their tiny liquidation transaction slightly before the legitimate liquidator's transaction reaches the sequencer queue. The $1 cost per attempt is negligible relative to the damage caused. The sequencer processes transactions in submission order and has no protocol-enforced obligation to detect or reorder such attacks.

---

### Recommendation

Allow liquidators to pass `type(int128).max` (or `type(int128).min` for negative amounts) as the `amount` field, interpreted as "liquidate the entire current position." In `_assertLiquidationAmount`, add:

```solidity
// Before the final bounds check, resolve sentinel values
if (txn.amount == type(int128).max) {
    // Caller wants to liquidate the full long/asset position
    // maxLiquidatable is already set to the current position size
    // just cap amount to maxLiquidatable
    // (handled by removing the revert and clamping instead)
}
```

Concretely, replace the hard revert with a clamp:

```diff
- if (txn.amount >= 0) {
-     require(maxLiquidatable >= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
- } else {
-     require(maxLiquidatable <= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
- }
+ if (txn.amount >= 0) {
+     if (txn.amount == type(int128).max) txn.amount = maxLiquidatable;
+     require(maxLiquidatable >= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
+ } else {
+     if (txn.amount == type(int128).min) txn.amount = maxLiquidatable;
+     require(maxLiquidatable <= txn.amount, ERR_NOT_LIQUIDATABLE_AMT);
+ }
```

This mirrors the recommendation in the reference report and eliminates the race condition by making the liquidation amount self-adjusting to the live position size.

### Proof of Concept

```
Setup:
  - Alice has a spot position of 1000e18 units (product 1), maintenance health < 0
  - Bob signs: LiquidateSubaccount { liquidatee=Alice, productId=1, amount=1000e18, nonce=0 }
  - Eve signs:  LiquidateSubaccount { liquidatee=Alice, productId=1, amount=1,      nonce=0 }

Eve submits her transaction to the sequencer before Bob.

Sequencer batch (in order):
  [0] Eve's tx  → _assertLiquidationAmount: maxLiquidatable=1000e18 >= 1 ✓
                  Alice's spotAmount becomes 999999999999999999999
  [1] Bob's tx  → _assertLiquidationAmount: maxLiquidatable=999999999999999999999 >= 1000e18 ✗
                  → revert ERR_NOT_LIQUIDATABLE_AMT

Result: Bob's full liquidation is blocked. Alice remains partially undercollateralized.
Eve's cost: $1 (LIQUIDATION_FEE).
``` [9](#0-8) [10](#0-9)

    function liquidateSubaccountImpl(IEndpoint.LiquidateSubaccount calldata txn)
        external
    {
        require(!RiskHelper.isIsolatedSubaccount(txn.sender), ERR_UNAUTHORIZED);
        require(txn.sender != txn.liquidatee, ERR_UNAUTHORIZED);
        require(isUnderMaintenance(txn.liquidatee), ERR_NOT_LIQUIDATABLE);
        require(
            txn.liquidatee != X_ACCOUNT && txn.liquidatee != N_ACCOUNT,
            ERR_NOT_LIQUIDATABLE
        );
        require(
            txn.productId != QUOTE_PRODUCT_ID,
            ERR_INVALID_LIQUIDATION_PARAMS
        );

        ISpotEngine spotEngine = ISpotEngine(
            address(engineByType[IProductEngine.EngineType.SPOT])
        );
        IPerpEngine perpEngine = IPerpEngine(
            address(engineByType[IProductEngine.EngineType.PERP])
        );

        if (_finalizeSubaccount(txn, spotEngine, perpEngine)) {
            if (RiskHelper.isIsolatedSubaccount(txn.liquidatee)) {
                IOffchainExchange(
                    IEndpoint(getEndpoint()).getOffchainExchange()
                ).tryCloseIsolatedSubaccount(txn.liquidatee);
            }
            return;
        }

        if (
            (txn.amount < 0) &&
            (txn.isEncodedSpread ||
                address(productToEngine[txn.productId]) == address(spotEngine))
        ) {
            // when it's spread or spot liquidation, we need to make sure the liquidatee has
            // enough quote to buyback the liquidated amount.
            _assertCanLiquidateLiability(txn, spotEngine, perpEngine);
            _settlePositivePerpPnl(txn, spotEngine, perpEngine);
        }

        _assertLiquidationAmount(txn, spotEngine, perpEngine);

        // beyond this point, we can be sure that we can liquidate the entire
        // liquidation amount knowing that the insurance fund will remain solvent
        // subsequently we can just blast the remainder of the liquidation and
        // cover the quote balance from the insurance fund at the end
        _handleLiquidationPayment(txn, spotEngine, perpEngine);
    }
