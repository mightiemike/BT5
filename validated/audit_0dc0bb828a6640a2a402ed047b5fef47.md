### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable Without Coinbase Proof Guard, Enabling 64-Byte Merkle Forgery - (File: contract/src/lib.rs)

### Summary
`verify_transaction_inclusion` (v1) is still a live, unpermissioned public entry point that omits the coinbase-merkle-proof check that `verify_transaction_inclusion_v2` enforces. Any NEAR caller can invoke v1 directly and obtain a `true` result for a forged transaction inclusion proof via the known 64-byte transaction attack.

### Finding Description
`verify_transaction_inclusion_v2` was introduced specifically to close the 64-byte transaction Merkle forgery vector (https://www.bitmex.com/blog/64-Byte-Transactions). It does so by requiring a separate coinbase merkle proof that anchors position 0 of the tree before accepting any other proof: [1](#0-0) 

The old function, `verify_transaction_inclusion`, performs no such coinbase anchor check. It only computes the merkle root from the caller-supplied `tx_id` and `merkle_proof` and compares it to the stored `merkle_root`: [2](#0-1) 

Despite carrying a `#[deprecated]` attribute, the function is `pub` and decorated only with `#[pause]` — no role restriction, no access-control gate. The `#[deprecated]` annotation is a Rust compile-time lint; it imposes zero runtime restriction. Any NEAR account can call it directly. [3](#0-2) 

This is the direct structural analog to the SafeERC20 inconsistency: `verify_transaction_inclusion_v2` uses the "safe" path (coinbase anchor), while `verify_transaction_inclusion` uses the "unsafe" path (raw merkle root comparison), and both paths are simultaneously reachable by an unprivileged caller.

### Impact Explanation
An attacker who can craft a 64-byte blob whose double-SHA256 hash equals an internal node of a real Bitcoin block's merkle tree can supply that blob as `tx_id` with a matching `merkle_proof`. `verify_transaction_inclusion` will compute the correct merkle root and return `true`, falsely asserting that a non-existent transaction is confirmed in a mainchain block. Any consumer contract that calls `verify_transaction_inclusion` to gate a payment, bridge withdrawal, or state transition will accept the forged proof and execute the protected action.

### Likelihood Explanation
The 64-byte transaction attack is a published, well-understood technique. The only precondition is finding or constructing a 64-byte preimage whose hash collides with an internal node — a task that is feasible given the deterministic structure of Bitcoin merkle trees and the absence of any domain separation. The entry point requires no privileged role and is reachable from any NEAR account.

### Recommendation
Remove `verify_transaction_inclusion` from the public ABI entirely, or gate it with an access-control role that prevents unprivileged callers from using it. The `#[deprecated]` attribute alone provides no runtime protection. If backward compatibility must be preserved, at minimum add the same coinbase-proof check that `verify_transaction_inclusion_v2` performs before delegating to the merkle root comparison.

### Proof of Concept
1. Identify a real mainchain block `B` with merkle root `R` and at least two transactions.
2. Construct a 64-byte blob `fake_tx` such that `double_sha256(fake_tx) == H_internal`, where `H_internal` is an internal node of `B`'s merkle tree reachable with a short proof path.
3. Build `merkle_proof` — the sibling hashes from `H_internal` up to `R`.
4. Call `verify_transaction_inclusion` with `tx_id = fake_tx_hash`, `tx_block_blockhash = B`, `tx_index = <position of H_internal>`, `merkle_proof = <constructed proof>`, `confirmations = 1`.
5. The function computes `compute_root_from_merkle_proof(fake_tx_hash, index, proof) == R` and returns `true`, despite `fake_tx` never existing as a real transaction. [4](#0-3) [5](#0-4)

        since = "0.5.0",

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
