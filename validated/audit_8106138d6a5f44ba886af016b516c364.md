### Title
`op_sha256` Fast-Path Cost Undercharge for Multi-Byte Small-Integer Arguments — (`File: src/more_ops.rs`)

### Summary

The `op_sha256` fast path in `src/more_ops.rs` computes the per-byte SHA256 cost using `num_bytes = if val > 0 { 2 } else { 1 }`, which assumes the second argument always encodes to exactly one byte. For CLVM integers ≥ 0x80, the canonical signed-big-endian encoding requires two bytes (a leading zero byte is prepended to avoid sign-bit ambiguity), making the actual bytes hashed three, not two. The fast path therefore undercharges by `SHA256_COST_PER_BYTE` per invocation for such values. Because the softfork cost guard enforces a strict equality check (`current_cost != guard.expected_cost`), this creates a consensus divergence between nodes compiled with and without the `no-fastpath` feature flag.

### Finding Description

In `op_sha256` (lines 385–397 of `src/more_ops.rs`), a fast path fires when the call matches `(sha256 1 val)` and `val` is a small integer within the precomputed hash table:

```rust
#[cfg(not(feature = "no-fastpath"))]
if let Some([v0, v1]) = match_args::<2>(a, input)
    && a.small_number(v0) == Some(1)
    && let Some(val) = a.small_number(v1)
    && (val as usize) < PRECOMPUTED_HASHES.len()
{
    let num_bytes: Cost = if val > 0 { 2 } else { 1 };   // ← BUG
    cost += num_bytes * SHA256_COST_PER_BYTE + 2 as Cost * SHA256_COST_PER_ARG;
    check_cost(cost, max_cost)?;
    return new_atom_and_cost(a, cost, &PRECOMPUTED_HASHES[val as usize]);
}
``` [1](#0-0) 

The `num_bytes` formula assumes the second argument always serialises to one byte. This is correct only for `val` in `[1, 0x7f]`. For `val ≥ 0x80`, `fits_in_small_atom` / `len_for_value` returns 2 (a leading zero is required):

``` [2](#0-1) 

The slow path (and the `no-fastpath` build) charges correctly:

```
cost += SHA256_COST_PER_ARG + atom_len(v0) * SHA256_COST_PER_BYTE   // 1 byte for "1"
cost += SHA256_COST_PER_ARG + atom_len(v1) * SHA256_COST_PER_BYTE   // len_for_value(val) bytes
``` [3](#0-2) 

For `val = 0x80`, `len_for_value(0x80) = 2`, so the slow path charges for 3 bytes total while the fast path charges for 2. The divergence grows for larger values.

The softfork cost guard enforces a strict equality check that is the direct analog of the external report's strict balance-difference check:

```rust
if current_cost != guard.expected_cost {
    return Err(EvalErr::SoftforkCostMismatch);
}
``` [4](#0-3) 

A program that declares the fast-path cost passes on fast-path nodes but fails this strict equality check on `no-fastpath` nodes (or any future reference implementation), and vice versa.

### Impact Explanation

**Consensus divergence**: two classes of full nodes — those compiled with the default fast path and those compiled with `no-fastpath` — compute different costs for the same `(sha256 1 val)` call when `val ≥ 0x80`. A softfork program that declares the fast-path cost is accepted by one class and rejected by the other via the strict `!=` guard. This is a direct analog of the external report's strict equality check failure: a computed quantity (cost instead of balance delta) differs by a small fixed amount (one `SHA256_COST_PER_BYTE` unit instead of 2 wei), causing a hard rejection on one code path.

### Likelihood Explanation

The fast path is enabled by default (the `no-fastpath` feature is opt-in). The pattern `(sha256 1 val)` is the standard CLVM tree-hash leaf computation and appears in virtually every non-trivial Chialisp program. The trigger requires `val ≥ 0x80` and `val < PRECOMPUTED_HASHES.len()`; if the precomputed table covers values beyond 127 (which is plausible for a 256-entry table), the bug is reachable by any attacker who can submit CLVM bytes — no privilege required.

### Recommendation

Replace the fixed `num_bytes` with the correct byte-length of the second argument:

```rust
// Before (wrong for val >= 0x80):
let num_bytes: Cost = if val > 0 { 2 } else { 1 };

// After (correct):
use crate::allocator::len_for_value;
let num_bytes: Cost = if val == 0 { 1 } else { 1 + len_for_value(val) as Cost };
```

This matches the slow-path accounting: one byte for the literal `1` argument plus `len_for_value(val)` bytes for the second argument.

### Proof of Concept

Craft a CLVM program invoking `(sha256 (q . 1) (q . 0x80))` inside a softfork guard that declares the fast-path cost (`SHA256_BASE_COST + 2*SHA256_COST_PER_ARG + 2*SHA256_COST_PER_BYTE + malloc_cost`). On a default build the softfork cost check passes; on a `no-fastpath` build the actual cost is `SHA256_BASE_COST + 2*SHA256_COST_PER_ARG + 3*SHA256_COST_PER_BYTE + malloc_cost`, so `current_cost != guard.expected_cost` fires and the program is rejected — a split between node populations.

        && (v.len() > 4
        || (v.len() == 1 && v[0] == 0)
        // a 1-byte buffer of 0 is not the canonical representation of 0
        || (v[0] & 0x80) != 0

        || (v[0] == 0 && (v[1] & 0x80) == 0)

        // that's a redundant leading zero. i.e. not canonical representation
        None
    } else {

        for b in v {
            ret <<= 8;
            ret |= *b as u32;

    }
