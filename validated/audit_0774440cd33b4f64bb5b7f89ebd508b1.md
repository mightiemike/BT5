### Title
Difficulty Retarget Calculation for Fork Blocks Uses Mainchain Ancestor Instead of True Fork Ancestor, Enabling Incorrect Difficulty Acceptance - (File: `contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

The difficulty adjustment calculation for all three non-Zcash chains (`bitcoin.rs`, `litecoin.rs`, `dogecoin.rs`) fetches the "first block in the retarget interval" via `get_header_by_height`, which unconditionally returns the **mainchain** block at that height. When a fork block is being validated and the fork diverged from the mainchain before `height_first`, the contract uses the wrong ancestor's timestamp, producing an incorrect `expected_bits`. A proof submitter can exploit this to get fork blocks accepted with an easier (or harder) difficulty than the protocol actually requires.

---

### Finding Description

All three retarget implementations share the same structural flaw. The `get_header_by_height` implementation in `lib.rs` always resolves through the mainchain index:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

In the **Dogecoin** retarget path (post-height 145,000, where `difficulty_adjustment_interval = 1`), every single block triggers a retarget. The first-block lookup is:

```rust
// TODO: check if it is correct to get block header by height from mainchain
// without looping to find the ancestor
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [2](#0-1) 

The `TODO` comment is the codebase's own acknowledgment that this is potentially wrong. The same pattern appears in Bitcoin: [3](#0-2) 

And in Litecoin: [4](#0-3) 

**Broken invariant**: When `submit_block_header` processes a fork block, `check_pow` → `check_target` → `get_next_work_required` is called. For a fork that diverged at height `H_fork`, any retarget boundary where `height_first < H_fork` causes the lookup to return the mainchain block at `height_first` rather than the fork's true ancestor at that height. The two blocks can have different timestamps, producing a different `actual_time_taken`, and therefore a different `expected_bits`.

The `calculate_next_work_required` functions in all three chains then compute a new target from this wrong `first_block_time`:

```rust
let mut actual_time_taken: i64 = prev_block_time - first_block_time;
// clamped to [timespan/4, timespan*4]
...
new_target = new_target.overflowing_mul(actual_time_taken) / timespan;
``` [5](#0-4) 

If the mainchain block at `height_first` has an **earlier** timestamp than the fork's true ancestor, `actual_time_taken` is inflated, the computed target is **easier** (higher), and the attacker can submit fork blocks with those easier `bits`. The PoW check then validates against the easier target:

```rust
require!(
    U256::from_le_bytes(&pow_hash.0) <= target_from_bits(header.bits),
    ...
);
``` [6](#0-5) 

---

### Impact Explanation

An attacker who controls the fork chain's block timestamps (within MTP and future-time constraints) can engineer the timestamp difference between the mainchain block at `height_first` and the fork's actual ancestor. By making the mainchain block appear earlier, the contract computes an easier `expected_bits`. The attacker then submits fork blocks with those easier bits, and the PoW hash check uses the easier target. This means the attacker can satisfy the PoW requirement with less mining work than the protocol demands.

The corrupted state is: **fork blocks are accepted with a difficulty that does not match the protocol's actual retarget rule**, violating the core security invariant of the light client. In the Dogecoin case (post-145,000, interval=1), this affects every fork block beyond the first, making it the most easily triggered variant.

---

### Likelihood Explanation

The entry path is `submit_blocks` (a public, payable NEAR call available to any trusted relayer or, with `UnrestrictedSubmitBlocks` role, any account). The attacker only needs to:
1. Submit a fork starting at any height.
2. Build the fork long enough that a retarget boundary is reached where `height_first` falls before the fork divergence point.
3. For Dogecoin post-145,000, this is triggered by any fork longer than 1 block.

No privileged key or social engineering is required beyond being a relayer (or having the unrestricted role). The `TODO` comment in the production code confirms the developers themselves identified this as a potential correctness issue. [7](#0-6) 

---

### Recommendation

Replace `get_header_by_height` calls inside retarget calculations with an ancestor-traversal loop that walks the fork's own `prev_block_hash` chain back to `height_first`. This is the correct approach used by Bitcoin Core and Dogecoin Core: they traverse the chain of `pprev` pointers rather than looking up by height in a global index. The `get_prev_header` function already exists and supports this traversal. [8](#0-7) 

---

### Proof of Concept

**Setup (Dogecoin, post-145,000 blocks):**

1. Mainchain is at height 145,002. Mainchain blocks at heights 145,000 and 145,001 have timestamps `T_main_0` and `T_main_1`.
2. Attacker submits a fork starting at height 145,000 (fork parent = mainchain block at 144,999). Fork block at 145,000 has timestamp `T_fork_0 = T_main_0 + 3600` (1 hour later, within future-time limits).
3. Attacker submits fork block at height 145,001 (fork parent = fork block at 145,000).
   - `difficulty_adjustment_interval = 1`, so retarget fires.
   - `blocks_to_go_back = 1`, `height_first = 145,000 - 1 = 144,999`.
   - `get_header_by_height(144,999)` returns the **mainchain** block at 144,999 with timestamp `T_main_neg1`.
   - This is the same block the fork's ancestor at 144,999 would be (since the fork diverged at 145,000), so this specific step is correct.
4. Attacker submits fork block at height 145,002 (fork parent = fork block at 145,001).
   - Retarget fires again. `height_first = 145,001 - 1 = 145,000`.
   - `get_header_by_height(145,000)` returns the **mainchain** block at 145,000 with timestamp `T_main_0`.
   - But the fork's true ancestor at 145,000 has timestamp `T_fork_0 = T_main_0 + 3600`.
   - `actual_time_taken` is computed as `T_fork_block_145001.time - T_main_0` instead of `T_fork_block_145001.time - T_fork_0`.
   - This inflates `actual_time_taken` by 3600 seconds, producing an easier target.
   - Attacker submits the fork block with the easier `bits` value matching the incorrectly computed `expected_bits`.
   - The PoW check validates against the easier target, requiring less mining work.

The attacker successfully gets a fork block accepted with a difficulty lower than the protocol requires, with the discrepancy growing with each subsequent retarget on the fork. [9](#0-8)
