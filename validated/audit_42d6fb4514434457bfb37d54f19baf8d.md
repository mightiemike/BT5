### Title
`oraclePriceX18` and `nlpPoolRebalanceX18` Excluded from EIP-712 Digest Allows Sequencer to Manipulate NLP Mint/Burn Exchange Rate — (`File: core/contracts/Verifier.sol`, `core/contracts/EndpointTx.sol`)

---

### Summary

The `SignedMintNlp` and `SignedBurnNlp` transaction structs each carry two execution-critical fields — `oraclePriceX18` and `nlpPoolRebalanceX18` — that are used directly in `clearinghouse.mintNlp()` / `clearinghouse.burnNlp()` but are **absent from the EIP-712 type hash and digest**. A malicious sequencer can substitute any oracle price or rebalance vector into a user's already-signed transaction without invalidating the signature, directly controlling the exchange rate at which the user mints or burns NLP tokens.

---

### Finding Description

`SignedMintNlp` and `SignedBurnNlp` are defined as:

```solidity
struct SignedMintNlp {
    MintNlp tx;               // sender, quoteAmount, nonce  ← signed
    bytes   signature;
    int128  oraclePriceX18;   // ← NOT signed
    int128[] nlpPoolRebalanceX18; // ← NOT signed
}
``` [1](#0-0) 

The EIP-712 type strings cover only the inner `MintNlp` / `BurnNlp` structs:

```
"MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce)"
"BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce)"
``` [2](#0-1) 

The digest computation in `computeDigest()` encodes only those three fields:

```solidity
digest = keccak256(
    abi.encode(
        keccak256(bytes(MINT_NLP_SIGNATURE)),
        signedTx.tx.sender,
        signedTx.tx.quoteAmount,
        signedTx.tx.nonce
    )
);
``` [3](#0-2) 

Yet in `processTransactionImpl()`, both unsigned fields are consumed directly:

```solidity
priceX18[NLP_PRODUCT_ID] = signedTx.oraclePriceX18;
clearinghouse.mintNlp(
    signedTx.tx,
    signedTx.oraclePriceX18,
    nlpPools,
    signedTx.nlpPoolRebalanceX18
);
``` [4](#0-3) 

The same pattern applies identically to `BurnNlp`. [5](#0-4) 

Because the sequencer is the sole entity that calls `submitTransactions()` and assembles the `bytes calldata transaction` payload passed on-chain, it can freely replace `oraclePriceX18` and `nlpPoolRebalanceX18` in the encoded bytes after receiving the user's signature. The signature check in `validateSignedTx()` will still pass because those fields are not part of the digest. [6](#0-5) 

---

### Impact Explanation

**For `MintNlp`:** The user signs a commitment to deposit `quoteAmount` of collateral. The sequencer controls `oraclePriceX18`, which determines how many NLP tokens the user receives. By submitting a deflated oracle price, the sequencer causes the user to receive fewer NLP tokens than the fair market rate. By submitting an inflated price, the sequencer dilutes existing NLP holders.

**For `BurnNlp`:** The user signs a commitment to redeem `nlpAmount` of NLP tokens. The sequencer controls `oraclePriceX18`, which determines how much collateral the user receives back. A deflated price causes the user to receive less collateral than owed.

**For `nlpPoolRebalanceX18`:** The sequencer can direct the rebalancing of NLP pool positions in an arbitrary direction, potentially concentrating losses into specific pools or extracting value from the protocol's liquidity pools.

The corrupted state delta is: the NLP token amount minted/burned per unit of collateral, and the per-pool balance deltas applied during rebalancing.

---

### Likelihood Explanation

The sequencer is the only party that submits `submitTransactions()`. It receives user-signed transaction bytes off-chain, encodes them into the batch, and submits on-chain. Substituting `oraclePriceX18` requires only that the sequencer re-encode the `SignedMintNlp` / `SignedBurnNlp` struct with a different value before submission — a trivial in-memory operation with no on-chain footprint. No frontrunning, mempool access, or key compromise is required. The entire point of EIP-712 signing is to protect users from exactly this class of sequencer manipulation; the omission of these fields defeats that protection for NLP operations.

---

### Recommendation

Add `oraclePriceX18` and `nlpPoolRebalanceX18` to the EIP-712 type strings and digest computations for both `MintNlp` and `BurnNlp`:

```
"MintNlp(bytes32 sender,uint128 quoteAmount,uint64 nonce,int128 oraclePriceX18,int128[] nlpPoolRebalanceX18)"
"BurnNlp(bytes32 sender,uint128 nlpAmount,uint64 nonce,int128 oraclePriceX18,int128[] nlpPoolRebalanceX18)"
```

Update `computeDigest()` in `Verifier.sol` to include these fields, and update the corresponding `MINT_NLP_SIGNATURE` / `BURN_NLP_SIGNATURE` constants accordingly. [2](#0-1) [7](#0-6) 

---

### Proof of Concept

1. User signs a `MintNlp` transaction: `{ sender: alice, quoteAmount: 1000e18, nonce: 5 }` with a fair oracle price of `1.00e18` (1 NLP = 1 USDC).
2. User submits the signed transaction to the sequencer off-chain.
3. Malicious sequencer re-encodes the `SignedMintNlp` struct, replacing `oraclePriceX18 = 1.00e18` with `oraclePriceX18 = 2.00e18` (doubling the apparent NLP price).
4. Sequencer calls `submitTransactions()` with the tampered bytes.
5. `validateSignedTx()` passes — the digest only covers `sender`, `quoteAmount`, `nonce`, all of which are unchanged.
6. `clearinghouse.mintNlp()` is called with the inflated price, minting Alice only ~500 NLP tokens instead of ~1000.
7. Alice has lost ~500 NLP tokens worth of value with no on-chain evidence of manipulation. [8](#0-7) [9](#0-8)
