### Title
Fork Difficulty Adjustment Uses Stale Main-Chain Reference Block Instead of Fork's Own Reference Block - (`contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

---

### Summary

When a fork block at a difficulty-adjustment boundary is validated, `get_next_work_required` fetches the interval's reference block via `get_header_by_height`, which unconditionally reads from `mainchain_height_to_header`. This always returns the **main-chain** block at that height, never the fork's own block. The difficulty is therefore computed against the wrong reference timestamp, exactly mirroring H-21's pattern: a state value (the block being validated) changes context (fork vs. main chain), but the reference point used for the calculation (the interval-start block) is never updated to match that context.

---

### Finding Description

`get_next_work_required` in both `contract/src/bitcoin.rs` and `contract/src/litecoin.rs` computes the expected difficulty for a block at a retarget boundary by fetching the first block of the 2016-block interval:

```rust
// contract/src/bitcoin.rs  lines 78-86
let first_block_height =
    prev_block_header.block_height - (config.difficulty_adjustment_interval - 1);

let interval_tail_extend_header = blocks_getter.get_header_by_height(first_block_height);
calculate_next_work_required(
    config,
    prev_block_header,
    interval_tail_extend_header.block_header.time.into(),
)
```

`get_header_by_height` is implemented as:

```rust
// contract/src/lib.rs  lines 677-682
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header          // ← always the main chain
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
```

Fork blocks are stored only in `headers_pool` via `store_fork_header`; they are **never** inserted into `mainchain_height_to_header`. Therefore, when a fork block at a retarget height is validated, the reference block fetched is the main-chain block at `first_block_height`, not the fork's block at that height. The two blocks can have arbitrarily different timestamps, producing an incorrect `actual_time_taken` and thus an incorrect expected difficulty.

The same flaw exists verbatim in `contract/src/litecoin.rs` lines 86-93.

---

### Impact Explanation

`calculate_next_work_required` computes:

```
actual_time_taken = prev_fork_block.time − reference_block.time
```

If the attacker controls the fork's block at `first_block_height` (F_ref) and sets its timestamp **later** than the main-chain block at the same height (M_ref), then using M_ref's (earlier) timestamp inflates `actual_time_taken`. A larger `actual_time_taken` produces a **lower** required difficulty (easier target). The attacker can therefore submit a fork block at the retarget boundary with a `bits` value that encodes a lower difficulty than the fork's chain actually requires, and the contract will accept it.

Accepted fork blocks with under-difficulty contribute their (inflated) `chain_work` to the fork's cumulative total. If the fork's `chain_work` exceeds the main chain's, `reorg_chain` is triggered, replacing the main chain with the attacker's fork. After the reorg, `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` will verify SPV proofs against the attacker's headers, enabling forgery of transaction inclusion proofs for transactions that do not exist on the real Bitcoin/Litecoin network.

---

### Likelihood Explanation

The attack requires the attacker to:
1. Submit a sequence of fork blocks starting before a retarget boundary — this is an open, permissionless call (`submit_blocks` is callable by any trusted relayer, and the relayer set can include adversarial participants).
2. Mine fork blocks up to the retarget boundary. Because the difficulty for non-retarget fork blocks inherits `prev_block_header.block_header.bits` (the previous fork block's bits), the attacker controls the difficulty of the entire fork chain up to the boundary.
3. At the retarget block, exploit the stale reference to obtain a lower required difficulty, then mine that single block.

The timestamp manipulation is bounded by `MAX_FUTURE_BLOCK_TIME_LOCAL = 2 * 60 * 60` seconds (2 hours) and the MTP rule, but even a modest timestamp difference shifts `actual_time_taken` enough to reduce the required difficulty by up to 4× (the clamping factor). This is a realistic, low-cost attack for any party able to submit blocks.

---

### Recommendation

Pass the fork's own chain of blocks as the reference source when validating fork blocks. Concretely, `get_next_work_required` should walk backwards through `headers_pool` via `get_prev_header` (which follows `prev_block_hash` links and works for both main-chain and fork blocks) to reach `first_block_height`, rather than calling `get_header_by_height` which is main-chain-only. Alternatively, maintain a separate height-indexed lookup per fork branch, or require that the caller supplies the interval-start block and validate it by hash linkage.

---

### Proof of Concept

**Setup**: Bitcoin mainnet deployment. Main chain has blocks at heights 0–4031 (two full retarget periods). M2016 has `time = T`.

**Attack**:
1. Attacker submits fork blocks F1…F2015 (all branching from height 0, with `bits = 0x207fffff` — near-zero difficulty, trivially mined).
2. Attacker submits F2016 with `time = T + 7000` (within the 2-hour future-time limit). This is the fork's interval-start block.
3. Attacker submits F2017…F4031 (inheriting `bits` from F2016, trivially mined).
4. Attacker submits F4032 (the retarget block). `get_next_work_required` calls `get_header_by_height(2016)` → returns **M2016** with `time = T`.
   - `actual_time_taken = F4031.time − T` (uses M2016, not F2016)
   - Correct value would be `F4031.time − (T + 7000)` — 7000 seconds shorter → higher required difficulty
   - The stale reference inflates `actual_time_taken` by 7000 s, reducing the required difficulty
5. F4032 is accepted with under-difficulty `bits`. Its `chain_work` is credited at the inflated rate.
6. If fork `chain_work` > main chain `chain_work`, `reorg_chain` fires. The contract's canonical chain is now the attacker's fork.
7. Attacker calls `verify_transaction_inclusion_v2` with a fabricated SPV proof against a block in the fork → returns `true` for a non-existent transaction. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
