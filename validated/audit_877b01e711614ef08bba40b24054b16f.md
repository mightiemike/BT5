Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the actual swapper, allowing per-user allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. The pool populates `sender` with its own `msg.sender` — which is the router contract when a user routes through `MetricOmmSimpleRouter`. This means the allowlist gates the router's address rather than the actual end user, making the per-user swap allowlist trivially bypassable (Scenario A) and breaking legitimate allowlisted users who route through the router (Scenario B).

## Finding Description

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`, passing the pool's direct caller as `sender`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that `sender` value unchanged into the extension call: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` — where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` or `exactInput`, the router calls `pool.swap(...)` directly, making the pool's `msg.sender` the router contract, not the end user: [4](#0-3) [5](#0-4) 

The same applies to `exactOutputSingle` and `exactOutput`: [6](#0-5) [7](#0-6) 

The asymmetry with `DepositAllowlistExtension` is the root cause. The deposit guard correctly uses `owner` (the economically relevant identity), not `sender` (the caller): [8](#0-7) 

The swap guard has no equivalent separation — it only receives `sender` (the direct pool caller), with no dedicated `swapper` parameter carrying the end-user identity.

**Scenario A – Allowlist bypass:** A pool admin allowlists the router address (a natural configuration to permit all router-mediated swaps). Any user, including those the admin intended to block, can now swap freely through the router. `allowedSwapper[pool][router] == true` causes the check to pass for every user routing through `MetricOmmSimpleRouter`.

**Scenario B – Broken allowlist for legitimate users:** A pool admin allowlists specific user addresses but does not allowlist the router. Every allowlisted user who routes through `MetricOmmSimpleRouter` is blocked with `NotAllowedToSwap`, making the primary swap path unusable for the intended participants.

## Impact Explanation

Scenario A is an admin-boundary break: an unprivileged user bypasses a configured per-user guard without any special privilege, using only the public `exactInputSingle`/`exactInput`/`exactOutput*` entry points. Scenario B is broken core pool functionality: the primary user-facing swap path becomes unusable for all allowlisted users, constituting a loss of core swap functionality. Both impacts satisfy the allowed impact gate (admin-boundary break and broken core pool functionality causing an unusable swap flow).

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary user-facing entry point for swaps. Any pool that deploys `SwapAllowlistExtension` and expects users to interact via the router is affected. No special privilege is required — any public user calling `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` reaches the vulnerable path. The condition is triggered automatically whenever the router calls `pool.swap`.

## Recommendation

Pass the actual end-user identity to the extension. The most direct fix mirrors the deposit pattern: add a dedicated `swapper` parameter (distinct from `sender`) that the pool populates from a trusted source representing the economically relevant actor. Alternatively, the router can write the real user address into a transient storage slot before calling `pool.swap`, and the pool reads it as the `swapper` for extension dispatch. Either approach ensures `SwapAllowlistExtension` always sees the address the pool admin intended to gate, regardless of whether the call is direct or router-mediated.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension configured on beforeSwap.
// 2. Pool admin allowlists the router address:
//    swapExtension.setAllowedToSwap(pool, address(router), true);
// 3. Pool admin does NOT allowlist blockedUser.

// Attack:
// blockedUser calls router.exactInputSingle(...) targeting the pool.
// Inside exactInputSingle, the router calls pool.swap(recipient, ...).
// pool.msg.sender == address(router).
// _beforeSwap passes sender = address(router) to SwapAllowlistExtension.
// allowedSwapper[pool][router] == true → check passes.
// blockedUser's swap executes successfully despite not being allowlisted.

function test_swapAllowlistBypassViaRouter() public {
    address blockedUser = makeAddr("blockedUser");

    // Admin allowlists the router (not the user)
    vm.prank(admin);
    swapExtension.setAllowedToSwap(address(pool), address(router), true);

    // blockedUser swaps through the router — should revert but does not
    vm.prank(blockedUser);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        recipient: blockedUser,
        amountIn: 1000,
        amountOutMinimum: 0,
        zeroForOne: false,
        priceLimitX64: type(uint128).max,
        deadline: block.timestamp + 1,
        extensionData: ""
    }));
    // blockedUser successfully swapped — allowlist bypassed
}
```
