Audit Report

## Title
`SwapAllowlistExtension` Allowlist Bypassed via Router: Pool Checks Router Address Instead of End User — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of `pool.swap()`. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract. Any pool admin who allowlists the router (required for their permissioned users to use the standard periphery) inadvertently grants every unpermissioned user the ability to bypass the allowlist by routing through the same router.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` as `sender` to `_beforeSwap`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged to all configured extensions: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap()`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly without forwarding the originating user's address: [4](#0-3) 

The router stores `msg.sender` only in transient storage for the payment callback (`_setNextCallbackContext`), not in the swap call itself. So inside `pool.swap()`, `msg.sender` is always the router. The extension therefore evaluates `allowedSwapper[pool][router]`, never `allowedSwapper[pool][end_user]`. Once the router is allowlisted, the check passes for every caller unconditionally.

`DepositAllowlistExtension` does not share this flaw because it checks the explicit `owner` argument passed to `addLiquidity`, which is the position owner, not the immediate caller: [5](#0-4) 

## Impact Explanation
A curated pool configured with `SwapAllowlistExtension` (e.g., KYC-gated, institutional, or compliance-restricted) is fully open to any user who routes through `MetricOmmSimpleRouter`. Unauthorized users can execute swaps against the pool, draining one-sided liquidity at oracle price. This constitutes a direct loss of LP principal and a complete break of the pool's core access-control invariant. Severity: High — direct loss of LP assets, broken core pool functionality.

## Likelihood Explanation
The router is the standard, documented periphery entry point. Any pool admin who wants their allowlisted users to trade via the router must call `setAllowedToSwap(pool, router, true)`. This is a routine, expected operational action, not a misconfiguration. The moment the router is allowlisted, any unpermissioned user can exploit it with a single `exactInputSingle` call. No special privileges or knowledge beyond the pool address are required.

## Recommendation
Pass the originating end-user address through the swap path so extensions can gate on it. Preferred fix: add a `swapper` field to `extensionData` or to the `swap()` signature that the router populates with `msg.sender` before calling the pool. `SwapAllowlistExtension.beforeSwap` reads this field instead of the pool-level `sender`. Alternatively, document and enforce at the factory level that `SwapAllowlistExtension` is incompatible with router-mediated flows, but this does not fix the root cause.

## Proof of Concept
```
Setup:
  pool = deploy MetricOmmPool with SwapAllowlistExtension
  admin calls setAllowedToSwap(pool, alice, true)   // intended permissioned user
  admin calls setAllowedToSwap(pool, router, true)  // required for alice to use the router
  LP adds liquidity to pool

Attack (bob is NOT allowlisted):
  bob calls router.exactInputSingle({pool: pool, recipient: bob, ...})
  → router calls pool.swap(bob, ...)          // msg.sender inside pool = router
  → pool calls _beforeSwap(sender=router, ...)
  → SwapAllowlistExtension checks allowedSwapper[pool][router] == true  ✓
  → swap executes; bob receives tokens

allowedSwapper[pool][bob] == false, but bob's address is never checked.
```
