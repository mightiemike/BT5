Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of originating swapper, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates on `sender`, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the originating user. A pool admin who allowlists the router to give approved users better UX inadvertently opens the pool to every user on the public router, completely defeating the allowlist. The extension provides no mechanism to recover the original user's identity from `extensionData`.

## Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the first argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // always the immediate caller of pool.swap()
    recipient,
    ...
);
``` [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks that value against the per-pool allowlist:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [2](#0-1) 

Here `msg.sender` inside the extension is the pool (correct), and `sender` is whoever called `pool.swap()` — the immediate caller.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly, making itself `msg.sender` to the pool:

```solidity
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ..., params.extensionData);
``` [3](#0-2) 

The router stores the original `msg.sender` in transient storage for callback payment purposes only — it does **not** encode it into `extensionData`. The `params.extensionData` passed to the pool is the raw user-supplied bytes, and the extension's `beforeSwap` ignores the `bytes calldata` parameter entirely (it is unnamed and unused). [4](#0-3) 

The same bypass applies to `exactInput` and `exactOutputSingle`, which also call `pool.swap()` directly with the router as `msg.sender`. [5](#0-4) 

`DepositAllowlistExtension` avoids this problem by checking `owner` (the LP-share beneficiary), which is passed as a separate argument independent of who called `addLiquidity`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    ...
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
``` [6](#0-5) 

`SwapAllowlistExtension` has no equivalent economically-meaningful actor to check — it only has `sender`, which collapses to the router address on every router-mediated swap.

## Impact Explanation

A curated pool protected by `SwapAllowlistExtension` is fully bypassed by any unprivileged user who routes through `MetricOmmSimpleRouter` once the pool admin allowlists the router. The allowlist's purpose — restricting swaps to KYC'd counterparties, institutional participants, or any other curated set — is entirely defeated. LPs who deposited under the assumption that only approved counterparties would trade against them are exposed to the full public, including adversarial MEV bots, front-runners, or users the pool was designed to exclude. This constitutes broken core pool functionality (the allowlist extension) causing direct loss of the protection LPs relied upon, meeting the contest's "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact criterion.

## Likelihood Explanation

The trigger is a pool admin allowlisting the router — a natural, expected action for any operator who wants approved users to enjoy the router's UX (slippage limits, multi-hop, deadline checks). The extension gives no warning that doing so opens the gate to everyone. The router is a public, permissionless contract. Once the router is allowlisted, any user can exploit the bypass without any further privileged action, simply by calling `exactInputSingle`, `exactInput`, or `exactOutputSingle` on the router targeting the curated pool.

## Recommendation

The extension must gate on the originating user, not the immediate pool caller. Two viable approaches:

1. **Router-aware allowlist**: Add a `mapping(address => bool)` of trusted routers; when `sender` is a trusted router, decode the real user from `extensionData` and check that address instead.
2. **Signed identity in `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension reads and verifies it (e.g., via a factory-registered router list).

The simplest safe interim fix is to document that router-mediated swaps are unsupported on allowlisted pools and reject any `sender` that is not itself in the per-pool allowlist, until a forwarding mechanism is added.

## Proof of Concept

```
Setup
─────
1. Pool admin deploys pool with SwapAllowlistExtension.
2. Admin calls setAllowedToSwap(pool, alice, true)   // Alice is the approved trader
3. Admin calls setAllowedToSwap(pool, router, true)  // Admin wants Alice to use the router

Attack
──────
4. Bob (not allowlisted) calls:
       MetricOmmSimpleRouter.exactInputSingle({
           pool:      <curated pool>,
           recipient: bob,
           ...
       })

5. Router calls pool.swap(bob, ...) with msg.sender = router.

6. Pool calls _beforeSwap(sender=router, ...).

7. SwapAllowlistExtension checks:
       allowedSwapper[pool][router]  →  true   ✓ (step 3)
   Extension does NOT revert.

8. Bob's swap executes on the curated pool.
   The allowlist is fully bypassed.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, allowlist only Alice and the router, then call `exactInputSingle` from Bob's address and assert the swap succeeds (no `NotAllowedToSwap` revert). Confirm that calling `pool.swap()` directly from Bob (without the router) reverts as expected.
