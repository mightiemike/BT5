### Title
Unprivileged Caller Can Bulk-Purge Block Headers via `run_mainchain_gc`, Permanently Breaking SPV Proof Verification - (File: `contract/src/lib.rs`)

---

### Summary

`run_mainchain_gc` is a public, permissionless function that accepts a caller-controlled `batch_size` parameter. Any unprivileged NEAR account can call it with `batch_size = u64::MAX`, forcing the contract to delete every block header currently eligible for garbage collection in a single transaction. This permanently removes those headers from on-chain storage, causing `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` to panic for any transaction whose block was in the prematurely-purged range. The attacker gains nothing; the damage to users and downstream bridge contracts is irreversible.

---

### Finding Description

**Root cause — missing caller authorization on `run_mainchain_gc`:**

The function is decorated only with `#[pause(except(roles(Role::UnrestrictedRunGC)))]`, which gates it only when the contract is paused. When the contract is live, the function is fully open to any caller: [1](#0-0) 

The `batch_size` parameter is entirely attacker-controlled. The only bound applied to it is: [2](#0-1) 

`total_amount_to_remove` equals `amount_of_headers_we_store - gc_threshold` — i.e., every header currently beyond the retention window. Passing `u64::MAX` sets `selected_amount_to_remove = total_amount_to_remove`, so the entire eligible set is deleted in one call.

**Normal operation (the design intent):**

`submit_blocks` calls `run_mainchain_gc` with `num_of_headers` — the count of headers in the current relay batch, typically 1–100: [3](#0-2) 

This means headers beyond `gc_threshold` are removed gradually, one relay batch at a time. Headers that are technically eligible but not yet reached by the incremental GC cursor remain in storage and remain verifiable.

**Deletion is permanent:**

`remove_block_header` erases the entry from both `mainchain_header_to_height` and `headers_pool`: [4](#0-3) 

There is no recovery path once a header is removed.

**Impact on SPV verification:**

Both `verify_transaction_inclusion` and `verify_transaction_inclusion_v2` look up the target block in `mainchain_header_to_height` and `headers_pool`. After GC, both lookups panic: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any user or downstream bridge contract that calls `verify_transaction_inclusion` or `verify_transaction_inclusion_v2` for a transaction whose block was in the prematurely-purged range receives a permanent panic. The verification result is not "false" — it is an unrecoverable contract abort. For a cross-chain bridge that gates asset releases on a `true` result from this contract, the effect is a permanent freeze of those assets. The attacker pays only NEAR gas; the victim loses the ability to prove their Bitcoin transaction forever.

---

### Likelihood Explanation

The attack requires no privileged keys, no special role, and no prior state. Any NEAR account can call `run_mainchain_gc` at any time the contract is not paused. The only precondition is that the chain has grown beyond `gc_threshold` (the recommended value is 52,704 blocks, approximately one year of Bitcoin history), which is the steady-state condition for a production deployment. The call is a single NEAR transaction with one `u64` argument.

---

### Recommendation

Restrict `run_mainchain_gc` to authorized callers only. The simplest fix is to add `#[private]` (restricting to the contract itself) or an explicit role check (e.g., `Role::DAO` or a dedicated `GCManager` role) matching the pattern already used for `submit_blocks`:

```rust
// Option A: restrict to contract-internal calls only
#[private]
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) { ... }

// Option B: require an authorized role
#[pause(except(roles(Role::UnrestrictedRunGC)))]
pub fn run_mainchain_gc(&mut self, batch_size: u64) {
    require!(
        self.acl_has_role(Role::DAO, &env::predecessor_account_id())
            || self.acl_has_role(Role::RelayerManager, &env::predecessor_account_id()),
        "Unauthorized"
    );
    ...
}
```

If the intent is to allow the relayer to call it externally (not just via `submit_blocks`), apply the same `#[trusted_relayer]` guard already used on `submit_blocks`.

### Proof of Concept

**Preconditions:**
- Contract deployed with `gc_threshold = 52704`.
- Chain has grown to `gc_threshold + 200` stored headers (steady-state production condition).
- A bridge user holds a valid Merkle proof for a Bitcoin transaction in block at height `initial_height + 100` (eligible for GC but not yet removed by incremental batching).

**Attack steps:**

```bash
# Step 1: Attacker (any unprivileged NEAR account) calls run_mainchain_gc
near call <contract_id> run_mainchain_gc \
  '{"batch_size": 18446744073709551615}' \
  --accountId attacker.near

# Step 2: All 200 eligible headers are now deleted in one transaction.

# Step 3: Bridge user attempts SPV verification — call panics permanently:
near call <contract_id> verify_transaction_inclusion_v2 \
  '<proof_args_for_block_at_initial_height_plus_100>' \
  --accountId victim.near
# → panics: "cannot find requested transaction block"
```

**Expected outcome:** The bridge user's `verify_transaction_inclusion_v2` call aborts with a panic. The block header is gone from `headers_pool` and `mainchain_header_to_height`. No re-submission is possible because `submit_blocks` only accepts new chain-tip headers, not historical ones. The attacker spent only gas.
