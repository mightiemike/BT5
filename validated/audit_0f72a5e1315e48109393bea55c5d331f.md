### Title
Cross-Network Replay of `DelegateAction` Signatures via Missing Chain-ID in NEP-461 Signing Domain — (`File: core/primitives/src/signable_message.rs`, `core/primitives/src/action/delegate.rs`)

---

### Summary

The NEP-461 `SignableMessage` scheme used to sign `DelegateAction` (meta-transactions) does not include the NEAR network's `chain_id` in the signed payload. A `DelegateAction` signature produced by a user on mainnet is cryptographically valid on testnet (or any other NEAR network) if the same account and key exist there with a compatible nonce. A malicious relayer can exploit this to replay a user's signed `DelegateAction` on an unintended network.

---

### Finding Description

`DelegateAction::get_nep461_hash()` constructs a `SignableMessage` whose Borsh-serialized form is:

```
[u32 discriminant (NEP number)] || [Borsh(DelegateAction)]
``` [1](#0-0) 

The discriminant is derived solely from the NEP number (366 for `DelegateAction`, 611 for `DelegateActionV2`): [2](#0-1) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — **no `chain_id` field**: [3](#0-2) 

The same is true for `VersionedDelegateActionPayload::get_nep461_hash()` (DelegateActionV2 / NEP-611): [4](#0-3) 

`apply_delegate_action()` verifies the signature and checks `max_block_height`, `sender_id`, and nonce — but performs **no network identity check**: [5](#0-4) 

Because the signed bytes are identical across all NEAR networks for the same `(sender_id, receiver_id, actions, nonce, max_block_height, public_key)` tuple, a signature produced for mainnet is also a valid signature on testnet (or any fork) for the same message.

---

### Impact Explanation

A malicious relayer who receives a user's `SignedDelegateAction` intended for mainnet can submit it verbatim on testnet (or any other NEAR network) if:

1. The user's account exists on the target network with the same public key.
2. The target network's access-key nonce for that key is exactly `delegate_action.nonce - 1`.

If both conditions hold, the inner actions (token transfers, `AddKey`, `DeleteKey`, contract calls, etc.) execute on the unintended network under the user's identity, consuming the user's testnet balance and advancing their testnet nonce. The user has no on-chain recourse because the signature is genuinely valid.

---

### Likelihood Explanation

- Developers routinely reuse the same ed25519 key pair across mainnet and testnet.
- A freshly created testnet account starts with nonce 0; a DelegateAction with nonce 1 (the first ever meta-tx from that key) is immediately replayable.
- `max_block_height` does not help: both mainnet and testnet have comparable block heights, so a height chosen to be valid on mainnet is also valid on testnet.
- The relayer role is unprivileged — any party that receives the `SignedDelegateAction` off-chain can submit it to any NEAR RPC endpoint.

---

### Recommendation

Include the network's `chain_id` in the `SignableMessage` payload (or in `MessageDiscriminant`) so that the signed bytes are unique per network. One approach:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub chain_id: String,   // e.g. "mainnet", "testnet"
    pub msg: &'a T,
}
```

Because this changes the serialized form, it requires a protocol-version gate (a new `ProtocolFeature`) so that nodes on both sides of the upgrade boundary agree on which hash to verify. Old `DelegateAction` signatures (without `chain_id`) must be rejected after the feature activates.

---

### Proof of Concept

```rust
// On mainnet: Alice signs a DelegateAction with nonce 1
let delegate_action = DelegateAction {
    sender_id: "alice.near".parse().unwrap(),
    receiver_id: "bob.near".parse().unwrap(),
    actions: vec![/* transfer 10 NEAR */],
    nonce: 1,
    max_block_height: 200_000_000,
    public_key: alice_key.public_key(),
};
let hash = delegate_action.get_nep461_hash(); // no chain_id in hash
let sig = alice_key.sign(hash.as_bytes());
let signed = SignedDelegateAction { delegate_action, signature: sig };

// Malicious relayer: submit the SAME signed bytes to testnet RPC
// Precondition: alice.near exists on testnet with the same key, nonce = 0
// Result: apply_delegate_action() on testnet verifies the signature (valid!),
//         checks nonce (0 < 1, valid!), and executes the inner actions on testnet.
```

The hash produced by `get_nep461_hash()` is byte-for-byte identical on both networks because `SignableMessage` contains only the NEP discriminant and the `DelegateAction` body — neither of which encodes the network identity. [6](#0-5) [7](#0-6)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

#[derive(BorshSerialize)]
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
#[non_exhaustive]
pub enum SignableMessageType {
    /// A delegate action, intended for a relayer to included it in an action list of a transaction.
    DelegateAction,
    /// A delegate action with gas key support, intended for a relayer to include it in an action
    /// list of a transaction.
    DelegateActionV2,
}

#[derive(thiserror::Error, Debug)]
#[non_exhaustive]
pub enum ReadDiscriminantError {
    #[error("does not fit any known categories")]
    UnknownMessageType,
    #[error("NEP {0} does not have a known on-chain use")]
    UnknownOnChainNep(u32),
    #[error("NEP {0} does not have a known off-chain use")]
    UnknownOffChainNep(u32),
    #[error("discriminant is in the range for transactions")]
    TransactionFound,
}

#[derive(thiserror::Error, Debug)]
#[non_exhaustive]
pub enum CreateDiscriminantError {
    #[error("nep number {0} is too big")]
    NepTooLarge(u32),
}

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
