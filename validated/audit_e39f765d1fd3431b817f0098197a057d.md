### Title
`AllResources` V3 Transaction with Zero `l2_gas`/`l1_data_gas` Silently Collapses to `L1Gas` After Protobuf Round-Trip, Causing Divergent `initial_sierra_gas()` â€” (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` implementation unconditionally maps any `ResourceBounds` message where both `l2_gas` and `l1_data_gas` are zero to `ValidResourceBounds::L1Gas`, regardless of whether the original Rust value was `AllResources`. A V3 transaction with `AllResourceBounds { l1_gas: non-zero, l2_gas: {0,0}, l1_data_gas: {0,0} }` is accepted by the gateway, executed by the proposer with `initial_sierra_gas() = 0` (immediate out-of-gas), but after a p2p protobuf round-trip the syncing node reconstructs it as `L1Gas` and executes it with `initial_sierra_gas() = initial_gas_no_user_l2_bound()` (a large system-defined value), producing a divergent execution result.

---

### Finding Description

**The lossy conversion** is at: [1](#0-0) 

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)
} else {
    ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
})
```

The serialization direction (`From<ValidResourceBounds> for protobuf::ResourceBounds`) faithfully encodes `AllResources` with zero `l2_gas`/`l1_data_gas` as three zero-valued fields: [2](#0-1) 

So the round-trip is **not lossless**: `AllResources({l1_gas: X, l2_gas: 0, l1_data_gas: 0})` â†’ protobuf â†’ `L1Gas(X)`.

**The gateway accepts such transactions.** The stateless validator test confirms `AllResourceBounds { l1_gas: NON_EMPTY_RESOURCE_BOUNDS, ..Default::default() }` (zero l2_gas and l1_data_gas) is valid: [3](#0-2) 

**The execution divergence** is in `initial_sierra_gas()`: [4](#0-3) 

- `AllResources` with `l2_gas.max_amount = 0` â†’ `initial_sierra_gas() = 0` (immediate out-of-gas)
- `L1Gas` â†’ `initial_sierra_gas() = initial_gas_no_user_l2_bound()` (large system value, e.g. `execute_max_sierra_gas + validate_max_sierra_gas`) [5](#0-4) 

**The p2p sync path** uses this exact converter. `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` calls `ValidResourceBounds::try_from(...)`: [6](#0-5) 

This is the path used by `FullTransaction` sync over p2p, where the syncing node replays committed blocks.

**The consensus path** has an additional guard: `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` rejects `L1Gas` with `DEPRECATED_RESOURCE_BOUNDS_ERROR`: [7](#0-6) 

So consensus messages with this transaction would be **rejected** (liveness issue), while p2p sync messages would be **silently misinterpreted** (state divergence).

---

### Impact Explanation

**Critical â€” Wrong execution result from blockifier/execution logic.**

The proposer executes the transaction with `initial_sierra_gas() = 0` â†’ immediate out-of-gas revert. A syncing node that receives the same block via p2p sync reconstructs the transaction as `L1Gas` and executes it with `initial_sierra_gas() = initial_gas_no_user_l2_bound()` â†’ the transaction may succeed, producing different storage writes, events, and state root. This is a state divergence between the proposer and syncing nodes for the same committed block.

---

### Likelihood Explanation

Any unprivileged user can craft and submit a V3 invoke/declare/deploy-account transaction with `l1_gas` non-zero and `l2_gas = l1_data_gas = {0, 0}`. The gateway accepts it. The condition is trivially constructible and requires no special privileges. The p2p sync path is exercised by every full node syncing the chain.

---

### Recommendation

In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`, do not use the zero-value heuristic to infer the variant. Instead, use an explicit discriminant field in the protobuf message (e.g., a `bool is_all_resources` or an enum tag), or always decode as `AllResources` when all three fields are present (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (the legacy 0.13.2 case, already handled by `unwrap_or_default`).

A minimal fix: change the condition to only produce `L1Gas` when `l1_data_gas` was absent (i.e., `value.l1_data_gas` was `None`), not when it was present but zero.

### Proof of Concept

```rust
#[test]
fn allresources_zero_l2_l1data_round_trips_as_allresources() {
    use starknet_api::transaction::fields::{
        AllResourceBounds, ResourceBounds, ValidResourceBounds,
    };
    use starknet_api::block::{GasAmount, GasPrice};
    use crate::protobuf;

    let original = ValidResourceBounds::AllResources(AllResourceBounds {
        l1_gas: ResourceBounds {
            max_amount: GasAmount(1000),
            max_price_per_unit: GasPrice(1),
        },
        l2_gas: ResourceBounds::default(),     // {0, 0}
        l1_data_gas: ResourceBounds::default(), // {0, 0}
    });

    // Serialize to protobuf
    let proto: protobuf::ResourceBounds = original.into();

    // Deserialize back
    let round_tripped = ValidResourceBounds::try_from(proto).unwrap();

    // This assertion FAILS: round_tripped is L1Gas, not AllResources
    assert!(
        matches!(round_tripped, ValidResourceBounds::AllResources(_)),
        "Expected AllResources, got {:?}", round_tripped
    );
}
```

The test fails because `round_tripped` is `ValidResourceBounds::L1Gas(...)`, not `AllResources`. The `initial_sierra_gas()` for the original is `0`; for the round-tripped value it is `initial_gas_no_user_l2_bound()`.
