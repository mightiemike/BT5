Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the router address as swapper identity, enabling full allowlist bypass via `MetricOmmSimpleRouter` - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` — the direct caller of `pool.swap()`. When `MetricOmmSimpleRouter` is used, `sender` is the router address, not the actual initiating user. If the pool admin allowlists the router (the natural action to enable router-based swaps for permitted users), every unpermitted user can bypass the per-user restriction by routing through the router. Alternatively, if the router is not allowlisted, no permitted user can swap through the standard periphery path.

## Finding Description

`SwapAllowlistExtension.beforeSwap` performs the following check:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` is the pool (correct pool-namespace key). `sender` is the first argument forwarded by the pool's extension machinery.

**Step 1 — Pool's `swap()` passes `msg.sender` as `sender`:**

In `MetricOmmPool.sol` L230–231, the pool unconditionally passes its own `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
_beforeSwap(
    msg.sender,   // ← becomes `sender` in the extension
    recipient, ...
);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeSwap` forwards it unchanged:**

In `ExtensionCalling.sol` L160–176, `sender` is forwarded as-is to all registered extensions:

```solidity
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
``` [2](#0-1) 

**Step 3 — `MetricOmmSimpleRouter` calls `pool.swap()` directly:**

All four router entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) call `pool.swap()` directly, making the router the `msg.sender` inside the pool:

```solidity
// exactOutputSingle, L136-137
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
``` [3](#0-2) [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [5](#0-4) 

**Contrast with `DepositAllowlistExtension`:** The deposit extension correctly checks `owner` (the explicit position owner passed to `addLiquidity`), not `sender` (the caller). There is no equivalent explicit `swapper` parameter in `pool.swap()`. [6](#0-5) 

## Impact Explanation

**Scenario A — Allowlist bypass (High):** Pool admin allowlists KYC'd users and also allowlists the router so those users can use the standard periphery path. Any unpermitted address can call `router.exactInputSingle(...)` targeting the pool; the extension sees `sender = router`, finds `allowedSwapper[pool][router] = true`, and passes. The per-user restriction is completely defeated — every unpermitted user can trade in a pool intended to be curated. This is unauthorized access to a restricted swap flow, a broken core pool invariant with direct fund-flow impact.

**Scenario B — Broken core swap functionality (Medium):** If the pool admin does not allowlist the router, allowlisted users cannot use `MetricOmmSimpleRouter` at all. The extension checks `allowedSwapper[pool][router]` which is `false` and reverts. The only path for a permitted user is to call `pool.swap()` directly and implement `metricOmmSwapCallback` themselves — not the supported periphery path.

## Likelihood Explanation

Medium-High. `MetricOmmSimpleRouter` is the documented production periphery swap path. Any pool deploying `SwapAllowlistExtension` that also wants users to swap through the router must allowlist the router — the natural configuration. Once the router is allowlisted, the bypass requires only a public call to `exactInputSingle` or `exactInput` with no special privilege. The attacker needs no tokens beyond the swap input amount.

## Recommendation

The extension must gate on the actual initiating user, not the intermediary router. Two viable approaches:

1. **Pass the real user through `extensionData`:** The router encodes `msg.sender` into `extensionData` before calling `pool.swap()`. The extension decodes and verifies it. Requires a trusted encoding convention between the router and the extension.

2. **Add an explicit `swapper` parameter to `pool.swap()`:** The pool's `swap()` signature accepts a `swapper` address (analogous to how `addLiquidity` accepts `owner`), and the pool passes it to `_beforeSwap` instead of `msg.sender`. The router sets `swapper = msg.sender` (the actual user). This mirrors how `DepositAllowlistExtension` correctly checks `owner` rather than `sender`, and is the cleaner fix.

## Proof of Concept

```
1. Deploy MetricOmmPool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowAllSwappers(pool, false)  [default — deny all].
3. Pool admin calls setAllowedToSwap(pool, alice, true)   [permit alice].
4. Pool admin calls setAllowedToSwap(pool, router, true)  [permit router so alice can use it].
5. charlie (not on allowlist) calls:
       router.exactInputSingle(ExactInputSingleParams{pool: pool, ...})
6. Router calls pool.swap(recipient=charlie, ...) — router is msg.sender inside pool.
7. Pool calls _beforeSwap(sender=router, recipient=charlie, ...).
8. Extension evaluates: allowedSwapper[pool][router] == true  → passes.
9. charlie's swap executes successfully in a pool that should have blocked him.
```

Root cause: `SwapAllowlistExtension.sol` L37 checks `sender` (the router) instead of the actual initiating user; `MetricOmmPool.sol` L231 unconditionally forwards `msg.sender` (the router) as the `sender` argument to all swap hooks. [5](#0-4) [1](#0-0)
