### Title
Protobuf `ValidResourceBounds` Deserialization Silently Coerces `AllResources` to `L1Gas`, Producing a Divergent Transaction Hash for V3 Invoke Transactions with Zero L2/Data-Gas Bounds - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The protobuf converter for `ValidResourceBounds` silently downgrades a V3 invoke transaction carrying `AllResources{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` into `L1Gas(X)`. Because `get_tip_resource_bounds_hash` branches on this enum variant and produces a structurally different Poseidon preimage (2 resource elements for `L1Gas`, 3 for `AllResources`), any downstream call to `calculate_transaction_hash` on the protobuf-deserialized form yields a hash that is provably different from the hash computed by the RPC gateway for the same transaction. The RPC Declare path already guards against this coercion with an explicit error; the Invoke path does not.

### Finding Description

**Step 1 – The coercion.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` in `crates/apollo_protobuf/src/converters/transaction.rs` applies the following heuristic:

```rust
Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
    ValidResourceBounds::L1Gas(l1_gas)          // ← silently drops AllResources variant
} else {
    ValidResourceBounds::AllResources(...)
})
``` [1](#0-0) 

A legitimate V3 invoke transaction submitted with `AllResourceBounds{l1_gas: X, l2_gas: 0, l1_data_gas: 0}` (zero L2/data-gas is a valid, user-controlled choice) is therefore deserialized from protobuf as `ValidResourceBounds::L1Gas(X)`.

**Step 2 – The hash divergence.**

`get_tip_resource_bounds_hash` branches on the enum variant and produces a different number of Poseidon input elements:

```rust
let mut resource_felts = vec![
    get_concat_resource(&l1_resource_bounds, L1_GAS)?,
    get_concat_resource(&l2_resource_bounds, L2_GAS)?,   // always present
];
resource_felts.extend(match resource_bounds {
    ValidResourceBounds::L1Gas(_) => vec![],              // ← 2 elements total
    ValidResourceBounds::AllResources(all_resources) => {
        vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]  // ← 3 elements
    }
});
``` [2](#0-1) 

For the same transaction fields `(tip=T, l1_gas=X, l2_gas=0, l1_data_gas=0)`:

| Path | Preimage fed to Poseidon | Result |
|---|---|---|
| RPC gateway (`AllResources`) | `[T, pack(L1_GAS,X), pack(L2_GAS,0), pack(L1_DATA_GAS,0)]` | H₁ |
| Protobuf sync (`L1Gas`) | `[T, pack(L1_GAS,X), pack(L2_GAS,0)]` | H₂ |

H₁ ≠ H₂ because Poseidon is collision-resistant and the input lengths differ.

**Step 3 – The asymmetric guard.**

The RPC Declare protobuf converter explicitly rejects `L1Gas` with `DEPRECATED_RESOURCE_BOUNDS_ERROR`:

```rust
resource_bounds: match common.resource_bounds {
    ValidResourceBounds::AllResources(resource_bounds) => resource_bounds,
    _ => { return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR); }
},
``` [3](#0-2) 

The `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3` converter has no equivalent guard and silently accepts `ValidResourceBounds::L1Gas` for a V3 invoke transaction. [4](#0-3) 

Similarly, `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` (the reverse direction) does reject `L1Gas`:

```rust
resource_bounds: match value.resource_bounds {
    ValidResourceBounds::AllResources(bounds) => bounds,
    _ => { return Err(StarknetApiError::OutOfRange { ... }); }
},
``` [5](#0-4) 

This confirms the developers intended V3 invoke transactions to always carry `AllResources`, but the protobuf ingestion path for `InvokeTransactionV3` does not enforce this.

**Step 4 – Where the wrong hash propagates.**

`InvokeTransactionV3` with `ValidResourceBounds` is the type used in the P2P sync / storage path. Any code that calls `Transaction::calculate_transaction_hash` on a deserialized `InvokeTransactionV3` — including `validate_transaction_hash` — will compute H₂ instead of H₁. [6](#0-5) 

The `TxCollector` in the integration test path recomputes hashes from `ConsensusTransaction::RpcTransaction` (which uses `AllResourceBounds` and is safe), but any sync-path code operating on the raw `InvokeTransactionV3` type is exposed. [7](#0-6) 

### Impact Explanation

A V3 invoke transaction with zero L2 and data-gas bounds is submitted by an unprivileged user via the RPC gateway. The gateway computes and stores hash H₁ (3-element `AllResources` preimage). When the same transaction is received over P2P and deserialized via `TryFrom<protobuf::InvokeV3>`, the resource bounds are silently coerced to `L1Gas`, and any subsequent `calculate_transaction_hash` call produces H₂ ≠ H₁. This binds the wrong hash to the transaction in the sync path, matching the **High** impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

The trigger is a valid, user-constructable V3 invoke transaction with zero L2 and data-gas bounds. No privileged access is required. The coercion is unconditional in the protobuf converter. The probability of a real user submitting such a transaction is non-trivial (e.g., a wallet that sets only L1 gas bounds for a V3 transaction).

### Recommendation

1. **Mirror the Declare guard in the Invoke converter.** In `TryFrom<protobuf::InvokeV3> for InvokeTransactionV3`, reject `ValidResourceBounds::L1Gas` with the same `DEPRECATED_RESOURCE_BOUNDS_ERROR` used for Declare:

```rust
resource_bounds: match resource_bounds {
    ValidResourceBounds::AllResources(b) => ValidResourceBounds::AllResources(b),
    _ => return Err(DEPRECATED_RESOURCE_BOUNDS_ERROR),
},
```

2. **Remove the heuristic coercion** in `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. Once support for pre-0.13.3 transactions is dropped (as the TODO at line 426 anticipates), the `l1_data_gas.is_zero() && l2_gas.is_zero()` branch should be removed entirely, always producing `AllResources`. [8](#0-7) 

3. **Add a canonicalization test** asserting that `calculate_transaction_hash` on a round-tripped `InvokeTransactionV3` (RPC → protobuf → deserialized) equals the hash computed directly from the `RpcInvokeTransactionV3`.

### Proof of Concept

```
1. Construct a V3 invoke transaction with:
     resource_bounds = AllResourceBounds { l1_gas: {max_amount: 1000, max_price_per_unit: 1},
                                           l2_gas: {0, 0}, l1_data_gas: {0, 0} }
     tip = 0, nonce = 0, sender = 0x1, calldata = []

2. Submit via RPC gateway.
   → TransactionConverter::convert_rpc_tx_to_internal() calls
     InternalRpcTransactionWithoutTxHash::calculate_transaction_hash(&chain_id)
   → InternalRpcInvokeTransactionV3::resource_bounds() returns ValidResourceBounds::AllResources(...)
   → get_tip_resource_bounds_hash produces:
       poseidon(0, pack(L1_GAS,{1000,1}), pack(L2_GAS,{0,0}), pack(L1_DATA_GAS,{0,0}))  = H₁
   → Stored tx_hash = H₁

3. Serialize the same transaction to protobuf::InvokeV3 with l1_data_gas = None (absent).
   → TryFrom<protobuf::ResourceBounds>: l1_data_gas.unwrap_or_default() = zero → L1Gas branch
   → Deserialized InvokeTransactionV3.resource_bounds = ValidResourceBounds::L1Gas({1000,1})

4. Call Transaction::Invoke(InvokeTransaction::V3(deserialized)).calculate_transaction_hash(&chain_id)
   → get_tip_resource_bounds_hash produces:
       poseidon(0, pack(L1_GAS,{1000,1}), pack(L2_GAS,{0,0}))                           = H₂

5. Assert H₁ ≠ H₂  ← canonicalization invariant broken.

            ValidResourceBounds::L1Gas(l1_gas)
        } else {

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
