Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks position `owner` instead of `sender`, allowing full allowlist bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual depositor/caller) and validates `owner` (the position recipient) against the allowlist. Because any caller can freely supply any address as `owner`, an unprivileged user can bypass the deposit allowlist entirely by naming any allowlisted address as the position recipient. The pool admin's sole mechanism for restricting liquidity provision is rendered ineffective.

## Finding Description

`setAllowedToDeposit` stores permissions keyed by `depositor`, signalling the intent to gate the actual token sender: [1](#0-0) 

`beforeAddLiquidity` receives `sender` as its first (unnamed) parameter and `owner` as its second, but checks only `owner`: [2](#0-1) 

`ExtensionCalling._beforeAddLiquidity` correctly passes both `sender` (the `msg.sender` of the pool's `addLiquidity` call) and `owner` (the position recipient) to the extension — so the extension receives the right data but ignores the right field: [3](#0-2) 

The periphery `addLiquidityExactShares` overload accepts a caller-controlled `owner` and only rejects `address(0)`: [4](#0-3) [5](#0-4) 

**Attack path:**
1. Admin calls `setAllowedToDeposit(pool, alice, true)` — only Alice is allowlisted.
2. Bob (not allowlisted) calls `liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, data)`.
3. The pool dispatches `beforeAddLiquidity(bob, alice, ...)` to the extension.
4. The extension evaluates `allowedDepositor[pool][alice]` → `true`. Check passes.
5. Bob's tokens are pulled from Bob and deposited; Alice receives the LP position without consent.

No existing guard prevents this: `_validateOwner` only rejects `address(0)`, and the allowlist check itself is the only access control on the deposit path.

## Impact Explanation

The deposit allowlist — the pool admin's sole mechanism for restricting who can provide liquidity — is completely bypassed by a single unprivileged transaction. Any actor can deposit into a KYC-gated, private, or curated pool by naming any allowlisted address as `owner`. The allowlisted address receives LP shares without consent (griefing vector). The pool's curation invariant is broken: admin-configured access control is defeated by a valid, publicly callable periphery function. This constitutes an admin-boundary break where an unprivileged path bypasses a factory/pool role check.

## Likelihood Explanation

The bypass requires a single transaction through the publicly callable `addLiquidityExactShares`. No special privileges, flash loans, or multi-step setup are needed. Allowlisted addresses are publicly discoverable via `AllowedToDepositSet` events. Every pool deploying `DepositAllowlistExtension` with a non-empty allowlist is immediately and trivially vulnerable.

## Recommendation

Check `sender` (the actual depositor) instead of `owner` (the position recipient):

```solidity
// metric-periphery/contracts/extensions/DepositAllowlistExtension.sol
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```solidity
// Pool deployed with DepositAllowlistExtension.
// Admin allowlists only Alice:
extension.setAllowedToDeposit(pool, alice, true);

// Bob (not allowlisted) deposits by naming Alice as owner via periphery:
vm.prank(bob);
liquidityAdder.addLiquidityExactShares(pool, alice, salt, deltas, max0, max1, "");
// → pool calls beforeAddLiquidity(bob, alice, ...)
// → extension checks allowedDepositor[pool][alice] == true → passes
// → Bob's tokens enter the pool; Alice receives LP shares without consent

// Verify bypass:
assertGt(pool.positionShares(alice, salt, binIdx), 0); // Alice has shares
// Bob's token balance decreased — his tokens are in the pool
```
