### Title
Linked Signer Can Self-Replace via `LinkSigner` Transaction, Enabling Permanent Subaccount Takeover - (File: `core/contracts/EndpointTx.sol`)

---

### Summary
In `EndpointTx.processTransactionImpl`, the `LinkSigner` transaction type is validated with `allowLinkedSigner = true`. This permits the currently-registered linked signer to sign a `LinkSigner` transaction that replaces itself with any attacker-controlled address. A compromised linked signer can therefore permanently escalate into the linked signer role with no owner action required, and the victim has no fast-path revocation — only a 3-day slow mode window.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the sequencer-path handler for `LinkSigner` is:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    true                          // allowLinkedSigner = true
);
linkedSigners[signedTx.tx.sender] = address(
    uint160(bytes20(signedTx.tx.signer))
);
``` [1](#0-0) 

`validateSignedTx` calls `validateSignature`, which — when `allowLinkedSigner = true` — passes `getLinkedSigner(sender)` as the accepted signer:

```solidity
verifier.validateSignature(
    sender,
    allowLinkedSigner ? getLinkedSigner(sender) : address(0),
    digest,
    signature
);
``` [2](#0-1) 

`Verifier.validateSignature` accepts either the subaccount owner address or the linked signer:

```solidity
require(
    (recovered != address(0)) &&
        ((recovered == address(uint160(bytes20(sender)))) ||
            (recovered == linkedSigner)),
    ERR_INVALID_SIGNATURE
);
``` [3](#0-2) 

Therefore, the current linked signer can produce a valid signature for a `LinkSigner` transaction that writes any new address into `linkedSigners[subaccount]`.

This is structurally inconsistent with the slow-mode path for the same transaction type, where only the subaccount owner (`msg.sender`) is accepted — no linked signer is consulted at all:

```solidity
validateSender(txn.sender, sender);   // msg.sender == owner address
requireSubaccount(txn.sender);
linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
``` [4](#0-3) 

The slow-mode path correctly restricts linked-signer mutation to the owner. The sequencer path does not.

---

### Impact Explanation

Once the attacker controls the linked signer slot, every transaction type processed with `allowLinkedSigner = true` becomes attacker-signable on behalf of the victim's subaccount. These include:

- **`LiquidateSubaccount`** (`allowLinkedSigner = true`, line 397–403): The attacker signs liquidation transactions naming the victim's subaccount as liquidator. `LIQUIDATION_FEE` is charged from the victim's subaccount on each call. The attacker can manufacture unprofitable liquidation targets (e.g., a subaccount with no assets but negative health) to ensure the fee is a net loss. [5](#0-4) 

- **`MintNlp` / `BurnNlp`** (`allowLinkedSigner = true`, lines 539–573): Each call charges `HEALTHCHECK_FEE` from the victim's subaccount. Repeated calls drain the balance through fees. [6](#0-5) 

- **`WithdrawCollateral` (V1)** (`allowLinkedSigner = true`, line 418–436): The attacker can drain the victim's on-protocol balance to the victim's wallet address, forcing a re-deposit cycle and consuming slow-mode fees on each re-entry. [7](#0-6) 

The victim's only recourse is a slow-mode `LinkSigner` submission, which carries a mandatory 3-day execution delay (`SLOW_MODE_TX_DELAY`). During this window the attacker-controlled linked signer remains active and can continue signing. [8](#0-7) 

---

### Likelihood Explanation

Linked signers are the standard mechanism for automated/programmatic trading in Nado — they are hot-wallet keys held by trading bots or API servers. Key compromise of a hot wallet is a realistic and common event (server breach, leaked `.env`, CI secret exposure). The attack requires only one valid signature from the compromised key and is executable immediately upon compromise, before the owner can react. No privileged access, governance action, or sequencer cooperation is required.

---

### Recommendation

Change `allowLinkedSigner` to `false` for the `LinkSigner` transaction type in `processTransactionImpl`, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that mutation of the linked signer credential always requires the subaccount owner's key, regardless of whether the transaction arrives via the sequencer or slow-mode path.
