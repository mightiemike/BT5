Audit Report

## Title
Payable Entry-Points Silently Discard `msg.value` for Non-WETH Tokens, Enabling Permissionless ETH Drain via `refundETH()` — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
All eight user-facing entry-points across `MetricOmmPoolLiquidityAdder` and `MetricOmmSimpleRouter` are declared `payable`, but the internal `pay()` helper in `PeripheryPayments` only consumes native ETH when the token being settled is exactly `WETH`. For any non-WETH token, `msg.value` is silently ignored and the ETH remains on the contract. Because `refundETH()` is permissionless and sends the entire contract balance to `msg.sender`, any third party can immediately drain the stranded ETH, causing direct loss of user principal.

## Finding Description
All four liquidity entry-points in `MetricOmmPoolLiquidityAdder` are `payable`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

All four swap entry-points in `MetricOmmSimpleRouter` are `payable`: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

Token settlement is delegated entirely to `PeripheryPayments.pay()`. The critical branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    ...
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value); // ETH completely ignored
}
``` [9](#0-8) 

When `token != WETH`, the `else` branch executes a plain ERC-20 pull and any attached `msg.value` is never consumed. The `receive()` hook reverts for any sender other than WETH: [10](#0-9) 

So ETH can only enter the contract through these `payable` calls. Once stranded, `refundETH()` is permissionless: [11](#0-10) 

It sends the **entire** contract ETH balance to `msg.sender` with no access control. A victim who strands ETH in transaction T1 and calls `refundETH()` in a separate transaction T2 can be front-run by an attacker who calls `refundETH()` first.

## Impact Explanation
Direct loss of user principal. Any user who calls `addLiquidityExactShares`, `addLiquidityWeighted`, `exactInputSingle`, `exactInput`, `exactOutputSingle`, or `exactOutput` with `msg.value > 0` on a pool whose relevant token is not WETH loses that ETH permanently unless they call `refundETH()` atomically in the same multicall. If they do not, any third party (including MEV bots) can drain the stranded ETH via the permissionless `refundETH()`. This constitutes a direct loss of user principal meeting Sherlock High/Medium thresholds.

## Likelihood Explanation
All entry-points are `payable` with no NatSpec warning against sending ETH for non-WETH pools. The `multicall` wrapper is also `payable`, encouraging users to bundle ETH with batched calls. A user interacting with any non-WETH pool (e.g., USDC/DAI) who mistakenly attaches ETH loses it immediately. MEV bots routinely monitor for stranded ETH on router contracts and call `refundETH()` to extract it, making recovery by the victim unlikely in practice.

## Recommendation
1. **Preferred**: Add a guard at the top of each entry-point rejecting ETH when the relevant token is not WETH:
   ```solidity
   if (msg.value > 0 && tokenIn != WETH) revert NativeNotAccepted();
   ```
   For liquidity adders, query `IMetricOmmPool(pool).getImmutables()` for `token0`/`token1` before accepting ETH.

2. **Alternative**: Remove the `payable` modifier from functions that have no WETH-wrapping path.

3. **Minimum mitigation**: Append an automatic `refundETH()` at the end of every entry-point so leftover ETH is always returned to `msg.sender` in the same transaction, eliminating the front-run window.

## Proof of Concept
```
1. Deploy a MetricOmm pool with token0 = USDC, token1 = DAI (neither is WETH).
2. Victim calls addLiquidityExactShares{value: 1 ether}(pool, ...) directly (not via multicall).
3. The callback settles via safeTransferFrom for both USDC and DAI; msg.value is never touched.
4. 1 ETH now sits on MetricOmmPoolLiquidityAdder.
5. Attacker calls refundETH() → receives 1 ETH.
6. Victim's 1 ETH is permanently lost.

The same sequence applies to exactInputSingle{value: 1 ether}(...) on any non-WETH pool via MetricOmmSimpleRouter.
```

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }

  function exactOutput(ExactOutputParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint8 tradesLeftAfterThis = uint8(params.pools.length - 1);
    address pool = params.pools[tradesLeftAfterThis];
    bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, tradesLeftAfterThis);
    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _initCallbackContextforRecursiveOutput(
      pool, CALLBACK_MODE_EXACT_OUTPUT_ITERATE, tradesLeftAfterThis, msg.sender, params.tokens[0]
    );
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
      .swap(
        params.recipient,
        zeroForOne,
        -expectedAmountOut,
        MetricOmmSwapPath.openLimit(zeroForOne),
        abi.encode(
          ExactOutputIterateCallbackData({
          tokens: params.tokens,
          pools: params.pools,
          extensionDatas: params.extensionDatas,
          zeroForOneBitMap: params.zeroForOneBitMap,
          amountInMax: params.amountInMaximum
        })
        ),
        params.extensionDatas[tradesLeftAfterThis]
      );

    int128 amountOut = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = _getExactOutputAmountIn();
    _clearExpectedCallbackPool();
  }

  function pay(address token, address payer, address recipient, uint256 value) internal {
    // If the payer is contract it means we are in the middle of a path. In the middle of a path we operate on ERC20 only.
    if (payer == address(this)) {
      IERC20(token).safeTransfer(recipient, value);
    } else if (token == WETH) {
      uint256 nativeBalance = address(this).balance;
      if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
      } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
      } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
      }
    } else {
      IERC20(token).safeTransferFrom(payer, recipient, value);
    }
  }
