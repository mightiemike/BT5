### Title
`SwapAllowlistExtension` gates the router address instead of the actual user, allowing any unprivileged user to bypass the swap allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument forwarded by the pool, which is `msg.sender` of `MetricOmmPool.swap()`. When a user routes through `MetricOmmSimpleRouter`, that `msg.sender` is the router contract, not the end user. If the router is allowlisted (which is required for any router-mediated swap to succeed on an allowlisted pool), every user on the public internet can bypass the swap allowlist by calling the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces the allowlist as follows:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (correct — used as the per-pool key). `sender` is the first argument forwarded by the pool, which is set in `MetricOmmPool.swap()`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

When a user calls `MetricOmmSimpleRouter.exactInputSingle()` (or any other router entry point), the router calls `pool.swap()` directly:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

The pool's `msg.sender` is the **router**, so `sender` forwarded to `beforeSwap` is the router's address. The allowlist check becomes `allowedSwapper[pool][router]`.

For any router-mediated swap to succeed on an allowlisted pool, the pool admin must add the router to the allowlist. Once the router is allowlisted, the check `allowedSwapper[pool][router]` passes for **every** user who calls through the router, regardless of whether that user's own address is on the allowlist. The flag that was supposed to gate individual users is effectively bypassed for the entire public router path.

This is the direct analog of the `is_buffer_start` omission: a required identity constraint (the allowlisted swapper) is not enforced on the path that matters (router-mediated swaps), so any actor can reach the guarded action by taking the unguarded path.

---

### Impact Explanation

A pool admin who deploys a pool with `SwapAllowlistExtension` intends to restrict swaps to a curated set of addresses (e.g., KYC'd counterparties, specific market makers, or whitelisted protocols). Once the router is added to the allowlist to support normal UX, the restriction is nullified: any address can call `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput` and the extension will pass because it sees the router, not the caller.

LPs in such a pool are exposed to swaps from actors the pool admin explicitly intended to exclude. In an oracle-based pool, this can mean exposure to adversarial flow (e.g., informed traders, sandwich bots, or actors the admin was trying to block for regulatory or risk reasons) that the allowlist was designed to prevent. The LP's deposited principal is at risk from the resulting adverse selection.

---

### Likelihood Explanation

Exploitation requires no special privilege. Any EOA or contract can call `MetricOmmSimpleRouter` — it is a public, permissionless periphery contract. The only precondition is that the pool admin has added the router to the allowlist (which is the normal operational step to make the pool usable via the router). This precondition is met in every realistic deployment that intends to support router-mediated swaps.

---

### Recommendation

The `beforeSwap` hook should gate the **original end user**, not the intermediary. Two complementary fixes:

1. **In the pool**: pass the original initiator through a separate argument or use `tx.origin` (with its known caveats) so extensions can distinguish the router from the user.
2. **In the extension**: require the router to forward the real user's address in `extensionData` and verify it, or gate on `tx.origin` when `sender` is a known router.
3. **Preferred**: follow the Uniswap v4 pattern where the hook receives both `sender` (the immediate caller) and a separately authenticated `msgSender` / `hookData` field so the guard can check the economically relevant actor.

---

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `BEFORE_SWAP_ORDER`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps (normal operational step).
3. A non-allowlisted user `alice` calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, ...})`.
4. The pool calls `_beforeSwap(msg.sender=router, ...)`.
5. `SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][router]` → `true` → no revert.
6. Alice's swap executes despite her address never being added to the allowlist.
