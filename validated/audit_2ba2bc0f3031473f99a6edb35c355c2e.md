### Title
Non-Deterministic Random Source Injected into Consensus-Critical Arithmetic Operators — (`File: src/more_ops.rs`)

### Summary
Both `op_add` and `op_subtract` in `src/more_ops.rs` invoke `rand::rng()` — a non-deterministic, OS-seeded random number generator — inside their bignum slow paths to randomly distribute operands across two internal accumulators. This introduces a non-deterministic execution path into two fundamental CLVM arithmetic operators that every Chia full node must evaluate identically for consensus. The analog to the report is direct: just as the lottery contract consumed a random value before it was validly sourced (Chainlink VRF), this code injects an uncontrolled random source into a context that must be fully deterministic.

---

### Finding Description

When the fast path (i64 arithmetic) cannot handle an operand — because it is a large bignum stored as a `Buffer` — both `op_add` and `op_subtract` fall through to a slow path that calls `rand::rng()` and uses `rng.random_range(0..2)` to decide which of two accumulators (`acc[0]` or `acc[1]`) receives each operand.

**`op_add` slow path** (`src/more_ops.rs`, lines 453–481):
```rust
let mut rng = rand::rng();
let mut acc = [Number::from(0), Number::from(0)];
...
acc[rng.random_range(0..2)] += val;   // line 465
...
let total = a.new_number(&acc[0] + &acc[1] + small_acc)?;
```

**`op_subtract` slow path** (`src/more_ops.rs`, lines 533–583):
```rust
let mut rng = rand::rng();
let mut acc = [Number::from(0), Number::from(0)];
...
acc[rng.random_range(0..2)] += number_from_u8(buf);  // line 546 (first arg)
...
acc[rng.random_range(0..2)] -= number_from_u8(buf);  // line 565 (subsequent args)
...
let total = a.new_number(&acc[0] + &acc[1] + small_acc)?;
```

Because the final result is `acc[0] + acc[1] + small_acc`, and addition is commutative and associative, the numeric value produced is mathematically invariant to which slot each operand lands in. However, the execution path itself — including intermediate bignum allocations, the internal state of the `Number` accumulator objects, and the sequence of `+=`/`-=` operations applied to each slot — is non-deterministic across invocations and across nodes. The `rand::rng()` call is seeded from OS entropy (`ThreadRng` in the `rand` crate), meaning two nodes executing the identical CLVM program with identical inputs will follow different internal code paths. ### Impact Explanation

CLVM is the consensus execution engine for the Chia blockchain. Every full node must reach the same `Reduction(cost, NodePtr)` for every program. The `op_add` and `op_subtract` operators are wired directly into the dialect's operator dispatch table via `f_table.rs` and are reachable from any attacker-submitted spend bundle. [3](#0-2) 

The concrete broken invariant is: **the internal execution state of `op_add`/`op_subtract` is non-deterministic**. While the final `Number` value is mathematically the same due to commutativity, the following are not guaranteed to be identical across nodes:

1. **Intermediate bignum object state**: `acc[0]` and `acc[1]` hold different partial sums on different nodes. If the `num-bigint` library has any allocation-order-dependent behavior (e.g., in its internal `BigInt` representation or normalization), the final `new_number()` encoding could diverge.
2. **Allocator heap layout**: `a.new_number()` allocates a new atom in the `Allocator`. The bytes written depend on the canonical encoding of the final `Number`. If the bignum library's `to_signed_bytes_be()` output is sensitive to how the value was accumulated (e.g., sign normalization of intermediate negative accumulators in `op_subtract`), the resulting `NodePtr` atom bytes could differ.
3. **Future regression surface**: Any future change to the slow path (e.g., short-circuiting on accumulator sign, adding overflow checks, or changing the `Number` type) could silently break the commutativity invariant and cause immediate consensus divergence.

The corrupted result would be: a different atom byte sequence returned from `op_add` or `op_subtract` for the same inputs on different nodes, causing a tree-hash mismatch and consensus split. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The slow path is triggered whenever any operand to `+` or `-` is a `Buffer` (i.e., a bignum that does not fit in a `U32`). An attacker submitting a spend bundle with a CLVM program that calls `(+ large_atom ...)` or `(- large_atom ...)` with any atom longer than 4 bytes will reliably trigger this path. This is trivially constructible from attacker-controlled CLVM bytes passed to `run_program` via the Python wheel's `run_serialized_chia_program` or directly through the Rust API. [6](#0-5) [7](#0-6) 

The `rand::rng()` call is not seeded deterministically — it uses `ThreadRng`, which is OS-entropy-seeded. There is no mechanism by which two independent nodes would produce the same random sequence.
