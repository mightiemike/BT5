### Title
Linked Signer Can Unilaterally Replace Itself to Achieve Persistent Subaccount Takeover and Full Asset Drain — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

In `EndpointTx.sol`, the fast-path `LinkSigner` transaction is validated with `allowLinkedSigner = true`. This means the **current linked signer** — a trading key, not the subaccount owner — can sign a `LinkSigner` transaction to replace itself with any attacker-controlled address. Once the attacker's address is installed as the new linked signer, it can sign `WithdrawCollateral` or `TransferQuote` transactions (both also validated with `allowLinkedSigner = true`) to drain the entire subaccount balance without the owner's consent.

---

### Finding Description

In `EndpointTx.processTransactionImpl`, the `LinkSigner` branch calls `validateSignedTx` with `allowLinkedSigner = true`:

```solidity
// EndpointTx.sol lines 576–590
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.SignedLinkSigner memory signedTx = abi.decode(
        transaction[1:],
        (IEndpoint.SignedLinkSigner)
    );
    validateSignedTx(
        signedTx.tx.sender,
        signedTx.tx.nonce,
        transaction,
        signedTx.signature,
        true                          // ← allowLinkedSigner = true
    );
    linkedSigners[signedTx.tx.sender] = address(
        uint160(bytes20(signedTx.tx.signer))
    );
}
```

`validateSignedTx` with `allowLinkedSigner = true` accepts a signature from either the subaccount owner **or** the current linked signer:

```solidity
// EndpointTx.sol lines 86–106
function validateSignedTx(..., bool allowLinkedSigner) internal {
    validateNonce(sender, nonce);
    validateSignature(
        sender,
        _hashTypedDataV4(computeDigest(...)),
        signature,
        allowLinkedSigner          // ← passes linked signer as valid signer
    );
    requireSubaccount(sender);
}
```

```solidity
// EndpointTx.sol lines 172–184
function validateSignature(..., bool allowLinkedSigner) internal virtual {
    verifier.validateSignature(
        sender,
        allowLinkedSigner ? getLinkedSigner(sender) : address(0),
        digest,
        signature
    );
}
```

This means the linked signer can sign a `LinkSigner` transaction that overwrites `linkedSigners[subaccount]` with an attacker-controlled address.

By contrast, the **slow-mode** `LinkSigner` path enforces that only the actual subaccount owner can change the linked signer:

```solidity
// EndpointTx.sol lines 232–239
} else if (txType == IEndpoint.TransactionType.LinkSigner) {
    IEndpoint.LinkSigner memory txn = abi.decode(...);
    validateSender(txn.sender, sender);   // ← requires msg.sender == owner address
    requireSubaccount(txn.sender);
    linkedSigners[txn.sender] = address(uint160(bytes20(txn.signer)));
}
```

The fast path and slow path are inconsistent: the slow path correctly restricts `LinkSigner` to the owner, but the fast path allows the linked signer to perform the same mutation.

---

### Impact Explanation

Once the attacker installs their address as the linked signer, they can immediately sign:

- **`WithdrawCollateral`** (fast path, `allowLinkedSigner = true`, lines 418–436) — withdraws any collateral to the default address (the subaccount owner's address, but the attacker controls the nonce and amount).
- **`WithdrawCollateralV2`** with `sendTo == address(0)` (fast path, `allowLinkedSigner = true`, lines 437–465) — withdraws collateral to the owner's address, but the attacker controls timing and amount.
- **`TransferQuote`** (fast path, `allowLinkedSigner = true`, lines 593–614) — transfers quote balance to any registered subaccount the attacker controls.
- **`BurnNlp`** (fast path, `allowLinkedSigner = true`, lines 554–573) — burns the victim's NLP position.

The attacker can drain the entire spot and quote balance of the victim subaccount. This is a **complete loss of all assets** held in the subaccount.

---

### Likelihood Explanation

The linked signer is explicitly designed to be a hot key (API key, trading bot) that is less securely stored than the subaccount owner's private key. Any party that obtains the linked signer key — through a compromised trading bot, a malicious third-party integration, or a phishing attack — can execute this escalation. Because the linked signer is intended to be a lower-trust, frequently-used key, the attack surface is meaningfully larger than for the owner key. The attack requires only one sequencer-processed transaction and is irreversible once executed.

---

### Recommendation

Change the `allowLinkedSigner` flag to `false` for the `LinkSigner` transaction type in the fast path, consistent with the slow-mode path:

```solidity
validateSignedTx(
    signedTx.tx.sender,
    signedTx.tx.nonce,
    transaction,
    signedTx.signature,
    false   // only the subaccount owner may change the linked signer
);
```

This ensures that changing the linked signer is an account-management operation exclusively controlled by the subaccount owner, regardless of whether it is submitted via the fast path or slow path.

---

### Proof of Concept

1. Alice owns subaccount `S` and has set `linkedSigners[S] = BOT_KEY` (a trading bot).
2. Attacker obtains `BOT_KEY` (e.g., via a compromised trading service).
3. Attacker uses `BOT_KEY` to sign a `SignedLinkSigner` transaction: `{ sender: S, signer: ATTACKER_ADDRESS, nonce: currentNonce }`.
4. Sequencer processes the transaction via `processTransactionImpl`. `validateSignedTx(..., true)` accepts the signature from `BOT_KEY` (the current linked signer).
5. `linkedSigners[S]` is now set to `ATTACKER_ADDRESS`.
6. Attacker uses `ATTACKER_ADDRESS` to sign a `SignedTransferQuote` transaction: `{ sender: S, recipient: ATTACKER_SUBACCOUNT, amount: fullBalance, nonce: nextNonce }`.
7. Sequencer processes the transfer. `validateSignedTx(..., true)` accepts the signature from `ATTACKER_ADDRESS` (the new linked signer).
8. All quote balance is transferred from `S` to `ATTACKER_SUBACCOUNT`. Alice's subaccount is drained. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
