### Title
SwapAllowlistExtension Gates the Router Address Instead of the End-User, Allowing Any Caller to Bypass the Swap Allowlist - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument against the per-pool allowlist. `sender` is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` becomes the router address, not the end-user. If the pool admin allowlists the router (the only way to permit router-mediated swaps for any allowlisted user), every non-allowlisted user can bypass the curated-pool gate by simply calling the router.

---

### Finding Description

`SwapAllowlistExtension.beforeSwap` is:

```solidity
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

`msg.sender` here is the pool (correct for keying the pool mapping). `sender` is the value the pool passes as the first argument to the hook, which is `msg.sender` of the `pool.swap()` call itself. [1](#0-0) 

In `MetricOmmPool.simulateSwapAndRevert` (and the regular `swap` path), `_beforeSwap` is called with `msg.sender` as `sender`: [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInput*`, the router calls `pool.swap(...)` directly. At that point `msg.sender` inside the pool is the **router**, so `sender = router`. The extension then evaluates `allowedSwapper[pool][router]` â€” not `allowedSwapper[pool][end_user]`.

The pool admin faces an impossible choice:

| Admin decision | Effect |
|---|---|
| Do **not** allowlist the router | Router-mediated swaps are blocked for **everyone**, including legitimately allowlisted users â€” broken core functionality |
| **Allowlist the router** | Every non-allowlisted user bypasses the gate by routing through the router |

The second branch is the exploitable path. A pool admin who wants to allow allowlisted users to trade via the router must allowlist the router address, which opens the gate to all users. [3](#0-2) 

---

### Impact Explanation

Any user can trade on a curated pool that is supposed to restrict swaps to a specific allowlist. The allowlist protection is completely nullified for any pool whose admin has allowlisted the router. LP funds in the pool are exposed to unrestricted trading by actors the pool was designed to exclude, which can result in direct loss of LP principal through adverse selection or protocol-policy violation.

---

### Likelihood Explanation

The `MetricOmmSimpleRouter` is the standard, documented periphery entry point for swaps. Any pool admin who wants allowlisted users to be able to use the router (the normal UX path) must allowlist the router address. This is the expected operational configuration, making the bypass reachable by any unprivileged user with no special preconditions.

---

### Recommendation

The extension must check the **end-user identity**, not the direct pool caller. Two sound approaches:

1. **Check `recipient` instead of `sender`** if the router always sets `recipient = end-user` â€” but this requires verifying the router's argument binding.
2. **Pass the original caller through `extensionData`** and have the router sign or encode it, then verify in the extension. This is the most robust approach.
3. **Alternatively**, gate on `sender` but require the router to be a trusted forwarder that encodes the real user in `extensionData`, and update the extension to decode and check that value.

The core invariant that must hold: the identity checked by the allowlist must be the same actor to whom the economic action (the swap) is attributed.

---

### Proof of Concept

1. Pool is deployed with `SwapAllowlistExtension` configured on the `beforeSwap` hook.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` â€” only Alice is allowed.
3. Pool admin calls `setAllowedToSwap(pool, router, true)` â€” router is allowlisted so Alice can use the standard UX.
4. Bob (not allowlisted) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The router calls `pool.swap(recipient=bob, ...)` with `msg.sender = router`.
6. `_beforeSwap` is called with `sender = router`.
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` â†’ passes.
8. Bob's swap executes on the curated pool despite never being allowlisted. [4](#0-3) [5](#0-4)
