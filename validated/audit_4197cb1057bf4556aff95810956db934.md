Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Original User, Allowing Full Allowlist Bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which the pool sets to `msg.sender` of the `pool.swap` call. When a user routes through `MetricOmmSimpleRouter`, the router is the direct caller of `pool.swap`, so the extension evaluates the router's allowlist status rather than the original user's. This makes per-user swap allowlists unenforceable for any pool that supports the public router.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whoever called `pool.swap`: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router the `msg.sender` from the pool's perspective: [3](#0-2) 

The same applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all call `pool.swap` with the router as `msg.sender`. [4](#0-3) 

The extension therefore evaluates `allowedSwapper[pool][router]`, not `allowedSwapper[pool][user]`. Two failure modes result:

| Scenario | Effect |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot swap through the router; core swap functionality broken |
| Router **is** allowlisted | Every user bypasses the per-user allowlist via the router |

No existing guard in the extension or pool recovers the original user identity. [5](#0-4) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict swaps to specific addresses (KYC-verified counterparties, institutional LPs, protocol-controlled addresses) can be fully bypassed by any unprivileged user routing through the public `MetricOmmSimpleRouter`. Real economic swaps execute against a pool that was supposed to be restricted — unauthorized users trade in a curated pool, potentially draining LP value or violating regulatory/contractual restrictions. This is a direct admin-boundary break: the pool admin's per-user allowlist is bypassed by an unprivileged public path, with fund-impacting consequences.

## Likelihood Explanation
`MetricOmmSimpleRouter` is a public, permissionless contract requiring no privileged access. The bypass requires a single transaction: call `exactInputSingle` or `exactInput` targeting the restricted pool. No malicious setup, no special token behavior, no privileged role needed. The pool admin has no on-chain mechanism to prevent router-mediated swaps while keeping the router usable for allowlisted users.

## Recommendation
The extension must gate the original user, not the intermediary. The safest approach: the router encodes `msg.sender` into `extensionData`; `SwapAllowlistExtension.beforeSwap` decodes it and checks the encoded user address when `sender` is a known router. Alternatively, pool admins must be explicitly warned that allowlisting the router grants access to all users, and the extension documentation/interface should expose this limitation. The simplest safe fix is to revert in `beforeSwap` if `sender` is not itself allowlisted and no mechanism exists to recover the original user identity. [6](#0-5) 

## Proof of Concept
```
Setup:
  - Deploy pool with SwapAllowlistExtension configured
  - Pool admin calls setAllowedToSwap(pool, alice, true)  // only alice should swap
  - Pool admin does NOT allowlist the router
  - Pool admin does NOT call setAllowAllSwappers(pool, true)

Attack (as Bob, a non-allowlisted user):
  1. Bob calls MetricOmmSimpleRouter.exactInputSingle({
       pool: restrictedPool, recipient: bob, zeroForOne: true, amountIn: X, ...
     })
  2. Router calls restrictedPool.swap(bob, true, X, ...)
     → pool's msg.sender = router
  3. Pool calls _beforeSwap(router, bob, ...)
  4. SwapAllowlistExtension.beforeSwap checks:
       allowedSwapper[pool][router] → false → REVERTS (alice also broken)

Alternate (router allowlisted so alice can use it):
  - Pool admin calls setAllowedToSwap(pool, router, true)
  - Bob calls router.exactInputSingle targeting the pool
  - Extension checks allowedSwapper[pool][router] → true → PASSES
  - Bob's swap executes on the restricted pool — allowlist fully bypassed

Foundry test: deploy pool + extension, setAllowedToSwap(pool, alice, true),
prank(bob), call router.exactInputSingle → assert swap succeeds for bob.
```
