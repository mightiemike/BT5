### Title
`EpochInfoAggregator` Persisted to Disk as an Unversioned Borsh Struct, Making Any Field Addition to It or Its Nested Inner Structs a Silent Data-Corruption Event at Upgrade Boundaries - (`chain/epoch-manager/src/epoch_info_aggregator.rs`)

---

### Summary

`EpochInfoAggregator` is a plain, unversioned Borsh struct that is periodically written to `DBCol::EpochInfo` under the fixed key `AGGREGATOR_KEY` and read back on every node start. It contains nested plain structs (`ValidatorStats`, `ChunkStats`) that are also unversioned. Because Borsh is a schema-less, position-dependent encoding, any future addition of a field to `EpochInfoAggregator` itself, or to any of its nested inner structs, will silently corrupt the on-disk bytes for all existing nodes at the upgrade boundary. The only recovery path is a silent fallback to a zero-initialized default, which discards all accumulated epoch statistics â€” block-production credits, chunk-production credits, protocol-version votes, and stake proposals â€” for the current epoch.

---

### Finding Description

Every other major on-disk structure in nearcore that evolves over time is wrapped in a versioned enum (e.g., `EpochInfo` â†’ `V1`â€¦`V5`, `ChunkExtra` â†’ `V1`â€¦`V5`, `BlockInfo` â†’ `V1`â€¦`V4`). `EpochInfoAggregator` is the sole critical structure that is stored directly as a plain struct with no version discriminant.

The aggregator is written directly as `EpochInfoAggregator` â€” no enum wrapper, no discriminant byte.

**Storage read path:** [2](#0-1) 

**Node startup:** [3](#0-2) 

If `get_ser` returns `None` (which happens when Borsh deserialization fails on a structurally changed layout), `unwrap_or_default()` silently replaces the aggregator with a zero-initialized default, discarding all accumulated epoch data.

**The unversioned struct and its nested inner structs:** [4](#0-3) 

The fields `block_tracker: HashMap<ValidatorId, ValidatorStats>` and `shard_tracker: HashMap<ShardId, HashMap<ValidatorId, ChunkStats>>` embed plain inner structs. Borsh serializes a `HashMap<K, V>` as `[len: u32][k0][v0][k1][v1]â€¦`, where each `V` is the flat byte layout of the inner struct. If a field is added to `ValidatorStats` or `ChunkStats`, the byte stream for every existing map entry shifts, and the old bytes are misread as the new layout â€” exactly the nested-struct upgrade hazard described in the external report.

The nearcore documentation itself warns about this class of problem: [5](#0-4) 

And the prescribed fix is the versioned-enum wrapper: [6](#0-5) 

`EpochInfoAggregator` violates this prescribed pattern while being the only structure that is both (a) stored to disk and (b) not wrapped in a versioned enum.

### Impact Explanation

When a binary upgrade adds any field to `EpochInfoAggregator`, `ValidatorStats`, or `ChunkStats`, every node that restarts with the new binary will fail to deserialize the on-disk aggregator. The silent `unwrap_or_default()` fallback resets the aggregator to empty, losing:

1. **`block_tracker`** â€” block-production credits for all validators in the current epoch. Validators who produced blocks before the upgrade receive zero credit, leading to incorrect reward calculations and potentially incorrect kickouts at epoch finalization.
2. **`shard_tracker`** â€” chunk-production and chunk-endorsement credits. Same consequence for chunk producers and chunk validators.
3. **`version_tracker`** â€” protocol-version votes cast by validators before the upgrade. The upgrade-vote tally is reset to zero, which can delay or prevent a protocol version upgrade from activating in the expected epoch.
4. **`all_proposals`** â€” stake-change proposals submitted before the upgrade. Validators who submitted proposals lose them, affecting the validator set composition for the next epoch.

The impact is bounded to the current epoch's statistics (the aggregator is reset at each epoch boundary), but within that epoch the consequences are incorrect validator economics and disrupted protocol-version governance.

---

### Likelihood Explanation

The trigger is any binary release that adds a field to `EpochInfoAggregator`, `ValidatorStats`, or `ChunkStats`. Given that `EpochInfoAggregator` has already grown (it gained `shard_tracker` and `version_tracker` in past releases) and that `ChunkStats` was recently extended to include endorsement statistics, further extensions are highly probable. The upgrade boundary is a routine, non-privileged operation (node operators restart their binaries). No attacker action is required; the corruption is automatic.

---

### Recommendation

Wrap `EpochInfoAggregator` in a versioned enum following the same pattern used for every other evolving on-disk structure:

```rust
#[derive(BorshSerialize, BorshDeserialize)]
#[borsh(use_discriminant = true)]
#[repr(u8)]
pub enum VersionedEpochInfoAggregator {
    V1(EpochInfoAggregator) = 0,
}
```

Store and load `VersionedEpochInfoAggregator` instead of the bare struct. When a field must be added to the aggregator or to `ValidatorStats`/`ChunkStats`, introduce a new variant (`V2`) and write a migration that converts `V1` bytes to `V2` on first read. This is the same pattern already applied to `EpochInfo`, `ChunkExtra`, `BlockInfo`, and every other evolving on-disk type in the codebase.

### Proof of Concept

**Step 1 â€” Current state.** A running node accumulates `EpochInfoAggregator` data across blocks and writes it to `DBCol::EpochInfo[AGGREGATOR_KEY]` as raw Borsh bytes of the plain struct. [7](#0-6) 

**Step 2 â€” Upgrade.** A new binary release adds one field (e.g., `missed_blocks: u64`) to `ValidatorStats`. The Borsh layout of every `ValidatorStats` value inside `block_tracker` now expects 24 bytes instead of 16. The on-disk bytes are unchanged.

**Step 3 â€” Node restart.** `EpochManager::new` calls `store.get_epoch_info_aggregator().unwrap_or_default()`. `get_ser` attempts `borsh::from_slice::<EpochInfoAggregator>(bytes)`. The deserialization reads the first `ValidatorStats` entry, consumes 16 bytes, but the new layout expects 24 â€” either the remaining bytes are misaligned or the stream ends prematurely. `get_ser` returns `None`. `unwrap_or_default()` returns `EpochInfoAggregator::default()`. [3](#0-2) 

**Step 4 â€” Epoch finalization.** `collect_blocks_info` calls `get_epoch_info_aggregator_upto_last`, which merges the empty in-memory aggregator with the empty default. All block/chunk production credits, version votes, and proposals for the current epoch are zero. Validator rewards and kickouts are computed on empty data. [8](#0-7)

### Versioned data structures

Given we expect many data structures to change or get updated as protocol evolves, a few changes are required to support that.

The major one is adding backward compatible `Versioned` data structures like this one:

```rust
enum VersionedBlockHeader {
    BlockHeaderV1(BlockHeaderV1),
    /// Current version, where `BlockHeader` is used internally for all operations.
    BlockHeaderV2(BlockHeader),
}
```

Where `VersionedBlockHeader` will be stored on disk and sent over the wire.
This allows to encode and decode old versions (up to 256 given https://borsh.io specification). If some data structures has more than 256 versions, old versions are probably can be retired and reused.
```
