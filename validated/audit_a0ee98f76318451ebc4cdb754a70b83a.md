Looking at the vulnerability class: **cross-module desynchronization** — a function uses a global accumulated value that conflates data from two distinct sources, corrupting a downstream calculation. I need to find the same pattern in this BTC light client.

The analog maps to difficulty retargeting: the contract uses `get_header_by_height` (which reads exclusively from `mainchain_height_to_header`) to fetch the boundary block timestamp for difficulty calculation, even when validating **fork** blocks whose ancestor at that height is a different block than the mainchain's.

---

### Title
Fork Difficulty Calculated Against Mainchain Ancestor Instead of Fork Ancestor — (`contract/src/dogecoin.rs`, `contract/src/bitcoin.rs`, `contract/src/litecoin.rs`)

### Summary

`get_header_by_height` always resolves a block by looking up `mainchain_height_to_header`, which only tracks the current canonical chain. When this function is called during fork-block difficulty validation, it returns the **mainchain** block at the retarget boundary height rather than the **fork chain's** ancestor at that height. For Dogecoin (1-block retarget interval post-height 145,000), this desynchronization is triggered for every fork block that is ≥ 2 blocks past the divergence point. An attacker-controlled relayer can exploit this to submit fork blocks whose required difficulty is computed against the wrong timestamp, allowing a chain reorganization to be achieved with less cumulative work than the protocol requires.

### Finding Description

`get_header_by_height` in `contract/src/lib.rs` is implemented as:

```rust
fn get_header_by_height(&self, height: u64) -> ExtendedHeader {
    self.mainchain_height_to_header
        .get(&height)
        .and_then(|hash| self.headers_pool.get(&hash))
        .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST))
}
``` [1](#0-0) 

It unconditionally resolves height → hash via `mainchain_height_to_header`, which is only updated for mainchain blocks. Fork blocks are stored only in `headers_pool` via `store_fork_header`:

```rust
fn store_fork_header(&mut self, header: &ExtendedHeader) {
    self.headers_pool.insert(&header.block_hash, header);
}
``` [2](#0-1) 

In Dogecoin's `get_next_work_required`, for every block post-height 145,000 the retarget interval is 1, so `height_first = prev_block_header.block_height - 1`. The boundary timestamp is fetched as:

```rust
let first_block_time = blocks_getter
    .get_header_by_height(height_first)
    .block_header
    .time;
``` [3](#0-2) 

The developers themselves flagged this with a TODO comment immediately above that line:

```rust
// TODO: check if it is correct to get block header by height from mainchain without looping to find the ancestor
``` [4](#0-3) 

**Concrete trigger sequence:**

Let the fork diverge at height D. The fork's block at D is stored in `headers_pool` but **not** in `mainchain_height_to_header`.

- Fork block at D+0: `height_first = D-2` → mainchain block at D-2 == fork ancestor at D-2. ✓ Correct.
- Fork block at D+1: `height_first = D-1` → mainchain block at D-1 == fork ancestor at D-1. ✓ Correct.
- Fork block at D+2: `height_first = D` → `get_header_by_height(D)` returns the **mainchain** block at D, but the fork's ancestor at D is the fork's own block at D (a different block with a different timestamp). ✗ **Wrong.**

The same flaw exists in Bitcoin and Litecoin's `get_next_work_required` at the 2016-block retarget boundary: [5](#0-4) [6](#0-5) 

### Impact Explanation

In Dogecoin's `calculate_next_work_required`:

```rust
let modulated_timespan = i64::from(prev_block_header.block_header.time) - first_block_time;
let mut modulated_timespan = retarget_timespan + (modulated_timespan - retarget_timespan) / 8;
// ...
new_target = new_target * modulated_timespan / retarget_timespan;
``` [7](#0-6) 

A smaller `first_block_time` (earlier timestamp) → larger `modulated_timespan` → larger target → **lower difficulty**. An attacker crafts the fork's block at D with a later timestamp than the mainchain's block at D. The difficulty for the fork block at D+2 is then computed using the mainchain's earlier timestamp, yielding a lower required difficulty. The attacker mines D+2 (and subsequent fork blocks) at this artificially reduced difficulty, accumulating chainwork faster than the protocol intends. If the fork's total chainwork exceeds the mainchain's, `submit_block_header_inner` triggers `reorg_chain`, corrupting the canonical chain with a fork that required less real work. [8](#0-7) 

### Likelihood Explanation

For Dogecoin (1-block retarget interval), the bug activates for **every** fork block at D+2 and beyond — a fork of only 3 blocks is sufficient. Any trusted relayer (or an account that stakes to become one) can submit such a fork. The attacker controls fork block timestamps subject only to MTP and the 2-hour future-time cap, giving ample room to set the fork's block-D timestamp later than the mainchain's. This is a realistic, low-barrier attack path.

### Recommendation

Replace the `get_header_by_height` call in all three `get_next_work_required` implementations with an ancestor walk that follows `prev_block_hash` links through `headers_pool` starting from `prev_block_header`, rather than resolving by height through `mainchain_height_to_header`. This mirrors the reference implementations (Bitcoin Core's `GetAncestor`) and ensures the correct fork-chain ancestor is used regardless of what the mainchain holds at that height.

### Proof of Concept

1. Initialize the contract with Dogecoin mainnet at height ≥ 145,001.
2. Submit mainchain blocks up to height D; record the mainchain block at D with timestamp `T_main`.
3. Submit a fork block at height D with timestamp `T_fork > T_main` (valid: above MTP, below future cap). This block is stored only in `headers_pool`.
4. Submit a fork block at D+1 (child of the D fork block). Difficulty computed from `height_first = D-1` (mainchain block) — correct.
5. Submit a fork block at D+2 (child of D+1 fork block). Difficulty computed from `height_first = D` → `get_header_by_height(D)` returns the **mainchain** block with timestamp `T_main < T_fork`. The computed `modulated_timespan` is larger than it should be, yielding a lower required difficulty. The fork block at D+2 is accepted with this reduced difficulty.
6. Repeat for D+3, D+4, … accumulating chainwork at reduced cost until the fork's chainwork exceeds the mainchain's, triggering `reorg_chain`.
