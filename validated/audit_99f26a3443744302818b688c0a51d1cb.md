Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Recipient (`owner`) Instead of Actual Depositor (`sender`), Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of `addLiquidity`) and instead checks `owner`, a free caller-supplied argument designating the LP share recipient. Because any caller can supply an already-authorized address as `owner`, the allowlist check always passes for that address regardless of who is actually depositing. This breaks the pool admin's primary access-control boundary for restricting liquidity providers.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and a caller-supplied `owner` argument as two distinct values to the extension hook: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both to the extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` then discards `sender` (first parameter is unnamed) and only checks `owner`: [3](#0-2) 

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller): [4](#0-3) 

The exploit path is straightforward: an unprivileged `ATTACKER` calls `pool.addLiquidity(owner=AUTHORIZED, ...)`. The extension receives `sender=ATTACKER` (discarded) and `owner=AUTHORIZED` (checked). Since `allowedDepositor[pool][AUTHORIZED] == true`, the check passes. Tokens are pulled from `ATTACKER` via the swap callback; LP shares are minted to `AUTHORIZED`. `removeLiquidity` enforces `msg.sender == owner`: [5](#0-4) 

so `ATTACKER` cannot reclaim the shares, but unauthorized capital has entered the pool and the allowlist invariant is permanently violated.

## Impact Explanation

This is an admin-boundary break: the pool admin's deposit allowlist — intended to restrict who may add liquidity for regulatory compliance, curated LP sets, or controlled bootstrapping — is bypassed by any unprivileged address. Unauthorized capital enters the pool, distorting bin balances and undermining any compliance or economic rationale behind the allowlist. The wrong value checked is `allowedDepositor[msg.sender][owner]` instead of `allowedDepositor[msg.sender][sender]`. [6](#0-5) 

## Likelihood Explanation

Exploitation requires only a standard `addLiquidity` call with a known authorized address as `owner`. No special privileges, flash loans, or oracle manipulation are needed. The `allowedDepositor` mapping is public, so any observer can identify authorized addresses and exploit this immediately and repeatedly. [7](#0-6) 

## Recommendation

Check `sender` (the actual depositor) instead of `owner` (the LP recipient), mirroring `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the depositor and the owner, both should be checked independently.

## Proof of Concept

```
Setup:
  pool P has DepositAllowlistExtension configured
  allowedDepositor[P][AUTHORIZED] = true
  allowedDepositor[P][ATTACKER]   = false (not set)

Attack:
  ATTACKER calls pool.addLiquidity(
      owner        = AUTHORIZED,   // passes the allowlist check
      salt         = 0,
      deltas       = <valid delta>,
      callbackData = ...,
      extensionData = ""
  )

Extension check (beforeAddLiquidity):
  msg.sender = P (the pool)
  sender     = ATTACKER  ← discarded, never checked
  owner      = AUTHORIZED
  allowedDepositor[P][AUTHORIZED] == true → check passes

Result:
  - ATTACKER's tokens are pulled via metricOmmSwapCallback
  - LP shares are minted to AUTHORIZED
  - ATTACKER has deposited into a pool that explicitly excluded them
  - The allowlist invariant is violated
```

Foundry test: deploy pool with `DepositAllowlistExtension`, set `allowedDepositor[pool][AUTHORIZED]=true`, call `addLiquidity` from `ATTACKER` with `owner=AUTHORIZED`, assert the call succeeds and `ATTACKER`'s token balance decreases.
