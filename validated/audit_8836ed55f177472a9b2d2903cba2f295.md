Audit Report

## Title
Uninitialized `lastMidPriceX64` causes `PriceVelocityGuardExtension` to fail open on every pool's first swap — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension` stores per-pool velocity state in `priceVelocityState[pool_]`, a mapping whose entries default to all-zero. The `beforeSwap` hook unconditionally skips the velocity check when `prevMid == 0`. Because `PriceVelocityGuardExtension` does not override `BaseMetricExtension.initialize` and writes no velocity state there, `lastMidPriceX64` is zero for every newly-registered pool. Any swap that executes before the pool admin manually calls `setLastMidPrice` bypasses the velocity cap entirely, allowing a swap at an arbitrarily-moved oracle price while LPs bear the adverse-selection loss.

## Finding Description

`PriceVelocityGuardExtension.beforeSwap` reads `s.lastMidPriceX64` and gates the entire velocity arithmetic on a non-zero check:

```solidity
// PriceVelocityGuardExtension.sol lines 53-76
PriceVelocityState storage s = priceVelocityState[pool_];
uint128 prevMid = s.lastMidPriceX64;   // zero for any new pool
...
s.lastMidPriceX64 = midPrice;          // baseline written AFTER the check
s.lastUpdateBlock  = uint64(block.number);

if (prevMid != 0) {                    // guard skipped entirely when zero
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity arithmetic and potential revert
    }
}
``` [1](#0-0) 

The only seeding path is the admin-callable `setLastMidPrice`: [2](#0-1) 

`PriceVelocityGuardExtension` does **not** override `BaseMetricExtension.initialize`. The base implementation simply returns the selector and writes nothing: [3](#0-2) 

The factory calls `initialize` on each extension during `createPool`, but since no velocity state is written there, `lastMidPriceX64` remains zero: [4](#0-3) 

The pool's `_beforeSwap` passes the oracle-derived `bidPriceX64`/`askPriceX64` directly to the extension with no pre-check on whether the guard has been initialized: [5](#0-4) 

The result: between pool creation and the admin's manual `setLastMidPrice` call, the velocity guard is silently disabled. The first swap writes the current (potentially extreme) oracle price as the new baseline, and all subsequent swaps are checked — but the damage from the first unguarded swap is already done.

## Impact Explanation

The velocity guard is the designated on-chain circuit-breaker against rapidly-moved oracle prices. When it fails open on the first swap, a trader can execute at an oracle price that has moved by an arbitrary percentage — far beyond the configured cap — while the pool's LPs bear the full adverse-selection loss. This is a direct, concrete loss of LP principal caused by an unclamped oracle quote reaching pool swap math, matching the **bad-price execution** and **oracle guard path** allowed impact categories. Severity is High: the loss is bounded only by pool liquidity depth, not by the velocity cap the admin intended to enforce.

## Likelihood Explanation

Every pool using `PriceVelocityGuardExtension` starts with `lastMidPriceX64 == 0`. There is no atomic initialization path; the admin must issue a separate `setLastMidPrice` transaction after pool creation. Any swap landing between pool creation and that admin call bypasses the guard. An attacker monitoring the mempool can backrun the pool-creation transaction with a swap in the same block, exploiting the unguarded window with no special permissions — only the ability to call `pool.swap()` (or satisfy any swap allowlist, if configured).

## Recommendation

Override `initialize` in `PriceVelocityGuardExtension` to seed `lastMidPriceX64` atomically at pool registration time. The extension should query the pool's price provider for the current mid-price and write it to `priceVelocityState[pool_]` during `initialize`, so the guard is active from the very first swap.

Alternatively, change the guard logic to **fail closed** when `prevMid == 0`: if `maxChangePerBlockE18 != 0` but no baseline has been set, revert unconditionally, forcing the admin to explicitly seed the state before any swap is permitted.

## Proof of Concept

1. Admin deploys a pool with `PriceVelocityGuardExtension` attached and calls `setMaxChangePerBlock(pool, 1e16)` (1% per block cap). `priceVelocityState[pool].lastMidPriceX64 == 0` (never initialized).
2. Oracle price moves 40% in one block (legitimate market event or oracle update lag).
3. Attacker observes the pool-creation transaction in the mempool and backruns it with `pool.swap(...)` before the admin calls `setLastMidPrice`.
4. `beforeSwap` executes: `prevMid = s.lastMidPriceX64 = 0`. The `if (prevMid != 0)` branch is skipped. The hook returns `IMetricOmmExtensions.beforeSwap.selector` — success.
5. The swap executes at the 40%-moved oracle price. The 1% velocity cap is never enforced.
6. LPs lose funds proportional to the price deviation; the attacker captures the arbitrage the guard was configured to prevent.
7. `s.lastMidPriceX64` is now written to the current (extreme) mid-price, so subsequent swaps are checked — but the damage from step 5 is already done.
