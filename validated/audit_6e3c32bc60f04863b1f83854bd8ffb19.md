Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` Checks Router Address Instead of Actual User, Allowing Any User to Bypass the Swap Allowlist ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is `msg.sender` of the pool's `swap()` call. When a user swaps through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the router contract, not the actual user. If the router is allowlisted (the natural configuration for any pool accessible via the supported periphery), every user in the world can bypass the per-user swap allowlist by routing through the router, silently breaking the sole access-control mechanism on the swap path.

## Finding Description

**Root cause:** `MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
  msg.sender,   // ŌåÉ this is the ROUTER when called via MetricOmmSimpleRouter
  recipient,
  ...
);
```

`ExtensionCalling._beforeSwap` forwards this verbatim to the extension via `abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))`.

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

**Exploit path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` to restrict trading to specific counterparties.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` ŌĆö the natural and expected configuration for any pool meant to be accessible via the supported periphery.
3. Pool admin does **not** call `setAllowedToSwap(pool, alice, true)` ŌĆö alice is not an authorized swapper.
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle(pool, ...)`.
5. Router calls `pool.swap(params.recipient, ...)` ŌĆö pool's `msg.sender` is the router.
6. Pool calls `_beforeSwap(router, ...)`.
7. Extension checks `allowedSwapper[pool][router]` ŌåÆ `true` ŌåÆ swap proceeds.
8. Alice successfully swaps on a pool she was never authorized to access.

**Secondary breakage:** A pool admin who allowlists `alice` directly (`setAllowedToSwap(pool, alice, true)`) will find that Alice cannot swap through the router (the check sees `router`, not `alice`), forcing her to call the pool directly ŌĆö which requires implementing `IMetricOmmSwapCallback`, an unreasonable burden for an EOA.

**Existing guards are insufficient:** There is no mechanism in the pool, router, or extension to recover the original `msg.sender` of the router call. The `extensionData` field is user-controlled and cannot be trusted for identity. No other guard in the swap path checks the actual initiating user.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a specific set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. The allowlist ŌĆö the sole access-control mechanism on the swap path ŌĆö silently fails open. Unauthorized users can execute swaps, drain LP-provided liquidity at oracle prices, and extract value from a pool that was designed to be closed to them. This constitutes broken core pool functionality causing direct loss of LP assets.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the primary supported swap entrypoint for the protocol. Any pool admin who configures a swap allowlist and also wants their pool to be accessible via the router will naturally allowlist the router. The bypass requires no special privileges, no malicious setup, and no non-standard tokens ŌĆö only a call to the public `exactInputSingle` or `exactInput` function on the router. The condition is trivially reachable by any unprivileged user.

## Recommendation

The extension must check the economically relevant actor ŌĆö the original user ŌĆö not the intermediary router. Options:
1. Add a dedicated `swapper` field to the extension interface that the pool populates with the original `msg.sender` before any router indirection, and have the router forward the original `msg.sender` via a trusted mechanism.
2. Have the router forward the original `msg.sender` via `extensionData` with a signature or trusted-forwarder pattern, and have the extension decode and verify it.
3. Redesign the allowlist to check `recipient` if that is the intended gated identity (though this may not match the intended semantics).

The cleanest fix is for the pool to expose the original initiator as a distinct, unforgeable parameter in the hook interface, separate from the intermediary `sender`.

## Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` ŌĆö intending to allow router-mediated swaps for authorized users.
3. Pool admin does **not** call `setAllowedToSwap(pool, alice, true)`.
4. Alice calls `MetricOmmSimpleRouter.exactInputSingle({pool: pool, recipient: alice, ...})`.
5. Router calls `pool.swap(alice, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, alice, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` ŌåÆ `true` ŌåÆ no revert.
8. Alice's swap executes successfully despite never being authorized.

Foundry test: deploy pool + extension, configure allowlist with only router, call `exactInputSingle` from an unauthorized EOA, assert swap succeeds (demonstrating the bypass).
