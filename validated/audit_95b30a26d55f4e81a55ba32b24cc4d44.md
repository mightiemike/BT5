### Title
Fast Withdrawal Fees Permanently Locked in `WithdrawPool` With No Transfer-Out Mechanism - (File: `core/contracts/BaseWithdrawPool.sol`)

---

### Summary

`BaseWithdrawPool.submitFastWithdrawal()` collects a fast-withdrawal fee on every call and accumulates it in the `fees[productId]` mapping, but neither `BaseWithdrawPool` nor its concrete implementation `WithdrawPool` contains any function to transfer those accumulated fees to a protocol-owned address. The fees are permanently locked in the contract. The only escape hatch is `removeLiquidity()`, an owner-only function that is not purpose-built for fee collection and does not reset the `fees` accounting.

---

### Finding Description

In `BaseWithdrawPool.submitFastWithdrawal()`, a fee is computed and retained in the contract in one of two ways:

1. When `sendTo == msg.sender`: `transferAmount` is reduced by `fee`, so the fee tokens remain in the pool's ERC-20 balance.
2. When `sendTo != msg.sender`: `safeTransferFrom(token, msg.sender, uint128(fee))` pulls the fee from the caller directly into the contract.

In both cases the fee is credited to the accounting ledger:

```solidity
fees[productId] += fee;
``` [1](#0-0) 

However, the entire `BaseWithdrawPool` contract contains no function that reads `fees[productId]` and transfers those tokens to a treasury or any protocol-owned address. `WithdrawPool.sol` is a one-function wrapper that only adds `initialize()` and adds nothing else: [2](#0-1) 

The only function that can move tokens out of the pool is `removeLiquidity()`, which is `onlyOwner`, accepts an arbitrary `amount`, and never touches the `fees` mapping: [3](#0-2) 

This is structurally identical to the reported `ClaggBaseAdapter._handleFee()` bug: a fee is deducted/collected but never forwarded to a designated recipient, and the only recovery path is a generic rescue/liquidity-removal function.

---

### Impact Explanation

Every fast withdrawal permanently donates its fee to the `WithdrawPool` contract. The `fees[productId]` counter grows monotonically but is never acted upon. Protocol revenue from fast withdrawals is inaccessible through any normal operational flow. The fee rate is `FAST_WITHDRAWAL_FEE_RATE = 0.1%` or `5 × withdrawFeeX18`, whichever is greater: [4](#0-3) 

Over time, and at scale, this represents a material and growing loss of protocol revenue that can only be recovered via the owner calling `removeLiquidity()` with a manually computed amount — an error-prone, non-standard operational procedure that also does not reset the `fees` accounting, making the ledger permanently inconsistent.

---

### Likelihood Explanation

`submitFastWithdrawal()` is a `public` function callable by any address. Every successful call collects a fee. There is no condition under which the fee is forwarded. The issue is triggered unconditionally on every fast withdrawal, making it certain to manifest in production. [5](#0-4) 

---

### Recommendation

Add a dedicated `claimFees(uint32 productId, address recipient)` function (restricted to `onlyOwner` or a governance-controlled role) that:
1. Reads `fees[productId]`.
2. Resets `fees[productId] = 0`.
3. Calls `handleWithdrawTransfer(getToken(productId), recipient, uint128(fees[productId]))` to transfer the accumulated fees to a protocol treasury address.

This mirrors the fix described in the external report: explicitly transferring the fee out to a protocol-owned address rather than leaving it stranded in the contract.

---

### Proof of Concept

1. Alice calls `submitFastWithdrawal(idx, transaction, signatures)` with `sendTo == Alice`.
2. `fee = fastWithdrawalFeeAmount(...)` returns, e.g., 1 USDC.
3. `transferAmount -= 1e6` — Alice receives `amount - fee`; the 1 USDC stays in the pool.
4. `fees[productId] += 1e6` — the ledger records the fee.
5. No function exists to send that 1 USDC to the protocol treasury.
6. Repeat for every fast withdrawal: `fees[productId]` grows, tokens accumulate, and the protocol can only recover them via `removeLiquidity()` with a manually computed amount — which does not reset `fees[productId]`, leaving the accounting permanently stale. [6](#0-5)

    function submitFastWithdrawal(
        uint64 idx,
        bytes calldata transaction,
        bytes[] calldata signatures
    ) public {
        require(!markedIdxs[idx], "Withdrawal already submitted");
        require(idx > minIdx, "idx too small");
        markedIdxs[idx] = true;

        Verifier v = Verifier(verifier);
        v.requireValidTxSignatures(transaction, idx, signatures);

        (
            uint32 productId,
            address sendTo,
            uint128 transferAmount
        ) = resolveFastWithdrawal(transaction);
        IERC20Base token = getToken(productId);

        require(transferAmount <= INT128_MAX, ERR_CONVERSION_OVERFLOW);

        int128 fee = fastWithdrawalFeeAmount(token, productId, transferAmount);

        if (sendTo == msg.sender) {
            require(transferAmount > uint128(fee), "Fee larger than balance");
            transferAmount -= uint128(fee);
        } else {
            safeTransferFrom(token, msg.sender, uint128(fee));
        }

        fees[productId] += fee;

        handleWithdrawTransfer(token, sendTo, transferAmount);
    }
