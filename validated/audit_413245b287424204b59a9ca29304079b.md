Audit Report

## Title
Missing zero-address check on `owner` in `addLiquidity` permanently locks deposited tokens — (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` of `address(0)` without reverting. Shares are minted to the zero-address position key, tokens are pulled from the caller via `metricOmmModifyLiquidityCallback`, and the position becomes permanently irremovable because `removeLiquidity` enforces `msg.sender == owner`, a condition that can never be satisfied for `address(0)`. Any tokens deposited into such a position are irreversibly locked in the pool.

## Finding Description
`MetricOmmPool.addLiquidity` performs only two input checks before delegating to `LiquidityLib.addLiquidity`: array-length equality and empty-delta early return. No validation is applied to the `owner` argument. [1](#0-0) 

`LiquidityLib.addLiquidity` likewise performs no zero-address check on `owner`; it computes the position key as `keccak256(abi.encode(owner, salt, bin))`, writes shares to `positionBinShares[posKey]`, updates bin totals, and then invokes `metricOmmModifyLiquidityCallback` on `msg.sender` to pull tokens. [2](#0-1) [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner` before proceeding. Because no EOA or contract can ever have `msg.sender == address(0)`, a position minted to `address(0)` can never be redeemed. [4](#0-3) 

The periphery helper `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner`, but only for callers who route through it. [5](#0-4) 

Any caller that invokes `MetricOmmPool.addLiquidity` directly — a custom router, an integrating protocol, or an EOA — receives no protection. The pool is the last line of defense and enforces no such invariant.

## Impact Explanation
Tokens deposited into a zero-address position are permanently locked in the pool with no recovery path. The caller pays the full `amount0Added` and `amount1Added` via the liquidity callback but receives a position that can never be redeemed. This constitutes a direct, irreversible loss of user principal — a critical/high impact under the allowed impact gate (direct loss of user principal, broken core withdraw flow).

## Likelihood Explanation
Any direct caller of `MetricOmmPool.addLiquidity` can trigger this by passing `owner = address(0)`. The periphery guard is not enforced at the pool level, so the protection is trivially bypassed by routing around `MetricOmmPoolLiquidityAdder`. This includes integrating contracts, custom routers, and direct EOA calls. No special privilege is required; the pool is a public contract.

## Recommendation
Add a zero-address check at the top of `addLiquidity` in `metric-core/contracts/MetricOmmPool.sol`, mirroring the guard already present in `MetricOmmPoolLiquidityAdder._validateOwner`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    if (owner == address(0)) revert InvalidPositionOwner();
    // ... existing logic
}
```

This ensures the invariant is enforced regardless of which entry path is used.

## Proof of Concept
1. Attacker calls `pool.addLiquidity(address(0), salt, deltas, callbackData, extensionData)` directly (bypassing the periphery).
2. The pool passes all existing checks (non-empty delta, matching lengths), calls `_beforeAddLiquidity`, then delegates to `LiquidityLib.addLiquidity`.
3. `LiquidityLib` computes `posKey = keccak256(abi.encode(address(0), salt, binIdx))`, writes shares to `positionBinShares[posKey]`, updates `binTotalShares` and `binState` balances, then invokes `metricOmmModifyLiquidityCallback` on `msg.sender`, pulling `amount0Added` and `amount1Added` from the caller.
4. Any subsequent call to `pool.removeLiquidity(address(0), salt, deltas, extensionData)` reverts at the `msg.sender != owner` check because `msg.sender` can never equal `address(0)`.
5. The deposited tokens are permanently locked; no address can ever satisfy the ownership check. [6](#0-5) [7](#0-6)
