### Title
Protobuf `ValidResourceBounds` Deserialization Silently Downgrades `AllResources` to `L1Gas`, Causing Valid V3 Transactions to Be Rejected via P2P - (File: `crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The protobuf deserializer for `ValidResourceBounds` uses a value-based heuristic to reconstruct the enum variant: if both `l2_gas` and `l1_data_gas` are zero, it returns `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. A valid post-0.13.3 V3 transaction submitted with `AllResources` bounds where both `l2_gas` and `l1_data_gas` are zero (e.g., using `ResourceBounds::default()`) is accepted by the gateway, but when propagated over P2P (mempool or consensus), the receiving node's protobuf deserialization silently changes the variant to `L1Gas`. The subsequent conversion from `InvokeTransactionV3` back to `RpcInvokeTransactionV3` explicitly rejects non-`AllResources` bounds with `DEPRECATED_RESOURCE_BOUNDS_ERROR`, causing the transaction to be dropped by every peer.

---

### Finding Description

**Step 1 — Protobuf serialization preserves zero values but loses variant identity.**

When a `ValidResourceBounds::AllResources` with zero `l2_gas` and `l1_data_gas` is serialized to protobuf, the `From<ValidResourceBounds> for protobuf::ResourceBounds` impl correctly emits all three fields: [1](#0-0) 

The wire bytes contain `l1_data_gas = Some(zero)` and `l2_gas = Some(zero)`.

**Step 2 — Deserialization applies a value-based heuristic that destroys variant identity.**

`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` checks whether the decoded values are zero and, if so, returns `L1Gas` instead of `AllResources`: [2](#0-1) 

A transaction originally carrying `AllResources { l1_gas: X, l2_gas: 0, l1_data_gas: 0 }` is now represented as `L1Gas(X)` inside `InvokeTransactionV3`.

**Step 3 — The conversion from `InvokeTransactionV3` back to `RpcInvokeTransactionV3` hard-rejects `L1Gas`.**

`TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3` converts the protobuf `InvokeV3` to `InvokeTransactionV3` first, then calls `.try_into()` to get `RpcInvokeTransactionV3`. That conversion explicitly rejects any non-`AllResources` variant: [3](#0-2) 

The error constant used is: [4](#0-3) 

The same rejection path exists for `DeclareV3`: [5](#0-4) 

**Step 4 — The hash domain boundary confirms the two variants are not equivalent.**

`get_tip_resource_bounds_hash` produces a different Poseidon hash for `L1Gas` (2 resource elements) vs `AllResources` (3 resource elements): [6](#0-5) 

So even if the deserialization did not error, the recomputed hash would differ from the one the user signed, causing signature verification failure at execution time.

**Step 5 — The trigger is reachable with `ResourceBounds::default()`.**

`AllResourceBounds::default()` produces `l2_gas = { max_amount: 0, max_price_per_unit: 0 }` and `l1_data_gas = { max_amount: 0, max_price_per_unit: 0 }`, which satisfies `is_zero()` for both fields. Any user who submits a V3 transaction without specifying L2 or data-gas bounds hits this path.

---

### Impact Explanation

A valid V3 transaction accepted by the gateway with `AllResources(l2=0, l1_data=0)` is permanently rejected by every P2P peer (both mempool propagation and consensus block validation) with `DEPRECATED_RESOURCE_BOUNDS_ERROR`. The transaction can never be included in a block via the consensus path. If a proposer includes such a transaction in a block proposal, every validator will fail to deserialize it, causing the proposal to be rejected and consensus to stall on that round.

**Impact**: High — Mempool/gateway/RPC admission rejects valid transactions before sequencing; consensus block proposals containing such transactions are rejected by validators.

---

### Likelihood Explanation

The trigger condition (`l2_gas.max_amount = 0`, `l2_gas.max_price_per_unit = 0`, `l1_data_gas.max_amount = 0`, `l1_data_gas.max_price_per_unit = 0`) is the natural default for any wallet or SDK that constructs a V3 transaction without explicitly setting L2 or data-gas bounds. `AllResourceBounds::default()` satisfies it directly. No privileged access or malicious intent is required; any ordinary user transaction can trigger it.

---

### Recommendation

Remove the value-based heuristic from `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`. The protobuf schema should encode the variant explicitly (e.g., via a boolean flag or a `oneof` field), or the deserializer should always produce `AllResources` when `l1_data_gas` is present in the wire message (even if zero), reserving `L1Gas` only for messages where `l1_data_gas` is absent (`None`). The current `unwrap_or_default()` on the optional field already conflates "field absent" with "field present but zero": [7](#0-6) 

The fix is to distinguish `None` (legacy, pre-0.13.3 → `L1Gas`) from `Some(zero)` (post-0.13.3 `AllResources` with zero bounds):

```rust
Ok(match value.l1_data_gas {
    None => ValidResourceBounds::L1Gas(l1_gas),
    Some(_) => ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas }),
})
```

---

### Proof of Concept

1. Construct a V3 invoke transaction with `AllResourceBounds { l1_gas: X, l2_gas: default(), l1_data_gas: default() }` (both l2 and data-gas fields are `{ max_amount: 0, max_price_per_unit: 0 }`).
2. Submit to the gateway. It is accepted; hash is computed via the `AllResources` path (4-element Poseidon input).
3. The transaction is propagated as a `ConsensusTransaction` or `MempoolTransaction` over P2P.
4. The receiving node calls `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`: `l1_data_gas.is_zero() && l2_gas.is_zero()` → returns `ValidResourceBounds::L1Gas(l1_gas)`.
5. `TryFrom<InvokeTransactionV3> for RpcInvokeTransactionV3` matches `_ =>` branch → returns `Err(DEPRECATED_RESOURCE_BOUNDS_ERROR)`.
6. The transaction is dropped by the receiving node. If it was in a consensus block proposal, the proposal is rejected.

impl TryFrom<protobuf::ResourceBounds> for ValidResourceBounds {
    type Error = ProtobufConversionError;
    fn try_from(value: protobuf::ResourceBounds) -> Result<Self, Self::Error> {
        let Some(l1_gas) = value.l1_gas else {
            return Err(missing("ResourceBounds::l1_gas"));
        };
        let Some(l2_gas) = value.l2_gas else {
            return Err(missing("ResourceBounds::l2_gas"));
        };
        // TODO(Shahak): Assert data gas is not none once we remove support for 0.13.2.
        let l1_data_gas = value.l1_data_gas.unwrap_or_default();
        let l1_gas: ResourceBounds = l1_gas.try_into()?;
        let l2_gas: ResourceBounds = l2_gas.try_into()?;
        let l1_data_gas: ResourceBounds = l1_data_gas.try_into()?;
        Ok(if l1_data_gas.is_zero() && l2_gas.is_zero() {
            ValidResourceBounds::L1Gas(l1_gas)
        } else {
            ValidResourceBounds::AllResources(AllResourceBounds { l1_gas, l2_gas, l1_data_gas })
        })
    }

impl TryFrom<protobuf::InvokeV3WithProof> for RpcInvokeTransactionV3 {
    type Error = ProtobufConversionError;
    fn try_from(mut value: protobuf::InvokeV3WithProof) -> Result<Self, Self::Error> {
        // Extract proof first, since `starknet_api::transaction::InvokeTransactionV3` does not
        // carry a `proof` field.
        let proof = Proof::from(std::mem::take(&mut value.proof));

        let snapi_invoke: InvokeTransactionV3 = value
            .invoke
            .ok_or(ProtobufConversionError::MissingField {
                field_description: "InvokeV3WithProof::invoke",
            })?
            .try_into()?;

        // This conversion can fail only if the resource_bounds are not AllResources.
        Ok(Self { proof, ..snapi_invoke.try_into().map_err(|_| DEPRECATED_RESOURCE_BOUNDS_ERROR)? })
    }

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
