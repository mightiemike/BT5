### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable with Essentially Empty Test Coverage, Enabling 64-Byte Merkle Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function is still a live, unpermissioned public entry point on the NEAR contract. Its only test (`test_view_call_verify_transaction_inclusion`) exercises a single trivially-false case (all-zero inputs) and verifies nothing about forgery resistance. Because the function accepts any `tx_id` — including internal Merkle-tree nodes — an unprivileged caller can supply a crafted proof where `tx_id` is an internal node hash and receive `true` in return. Any recipient contract that consumes v1's result is deceived into accepting a forged transaction-inclusion claim.

---

### Finding Description

**Reachable entry point — no role required.**
`verify_transaction_inclusion` carries only `#[pause]` (not `#[trusted_relayer]` and not `#[private]`). Any NEAR account can call it at any time the contract is not paused. [1](#0-0) 

**The function accepts internal Merkle-tree nodes as `tx_id`.**
`compute_root_from_merkle_proof` is a pure positional hash-chain computation. It does not distinguish leaf hashes from internal-node hashes. If the caller supplies an internal node at level *k* as `tx_id` and a proof of length `log₂(N) − k`, the function recomputes the correct Merkle root and returns `true`. [2](#0-1) [3](#0-2) 

The code itself documents this broken invariant:

> *"This function may return `true` if the provided `tx_id` is a hash of an internal node in the Merkle tree rather than a valid transaction hash."* [4](#0-3) 

**The only test for v1 is essentially empty.**
`test_view_call_verify_transaction_inclusion` passes `tx_id = H256::default()`, `merkle_proof = [H256::default()]`, and asserts the result is `false`. It never verifies that a valid proof returns `true`, never verifies that a forged proof is rejected, and never exercises the forgery path at all. This is the direct analog to the external report's "empty unit test declarations" — the test exists but covers no security-critical behaviour. [5](#0-4) 

**v2 does not protect callers of v1.**
`verify_transaction_inclusion_v2` enforces equal proof lengths (preventing the depth mismatch that the forgery relies on) and validates a coinbase proof before delegating to v1. However, v1 is independently callable; v2's guards are bypassed entirely when a caller invokes v1 directly. [6](#0-5) 

---

### Impact Explanation

A caller who knows any mainchain block's Merkle tree (all data is public on Bitcoin) can:

1. Pick two adjacent leaf hashes `H(tx_a)` and `H(tx_b)`.
2. Compute the level-1 internal node `N = H(H(tx_a) ∥ H(tx_b))`.
3. Call `verify_transaction_inclusion` with `tx_id = N`, `tx_index = 0`, `merkle_proof = [H(tx_c ∥ tx_d), …]` (the sibling subtree path to the root), `tx_block_blockhash` = any mainchain block containing those transactions, `confirmations = 0`.
4. `compute_root_from_merkle_proof(N, 0, proof)` recomputes the correct Merkle root → function returns `true`.

The corrupted value is the **proof result**: `verify_transaction_inclusion` returns `true` for a transaction that does not exist. Any on-chain recipient contract that gates an action on this result (e.g., a bridge release, a cross-chain settlement) is deceived into executing that action for a non-existent Bitcoin transaction.

---

### Likelihood Explanation

- No role, stake, or privileged key is required.
- All inputs (block Merkle trees) are publicly available from any Bitcoin node or block explorer.
- The attack is deterministic and requires no brute-force.
- The function is not paused on any known deployment.

---

### Recommendation

1. **Remove `verify_transaction_inclusion` (v1) entirely.** It has been deprecated since v0.5.0 and v2 is a complete replacement. Keeping a callable, documented-as-broken function on-chain is unnecessary risk.
2. If removal is not immediately possible, add `#[private]` or a role guard so only trusted callers can invoke it, and emit a deprecation warning in the NEAR event log.
3. Add a test that demonstrates the forgery succeeds on v1 and fails on v2, making the security boundary explicit and regression-proof.

---

### Proof of Concept

Using the two-transaction block already constructed in `submit_two_tx_block` (coinbase at index 0, `tx` at index 1, `merkle_root = H(coinbase ∥ tx)`):

```
coinbase_hash = 4a5e1e4b...
tx_hash       = 0e3e2357...
merkle_root   = H(coinbase_hash ∥ tx_hash)   ← stored in the block
```

The internal node at level 1 **is** `merkle_root` itself (2-leaf tree has only one internal node = root). For a deeper tree, pick any level-1 node `N = H(leaf_L ∥ leaf_R)` and supply the sibling subtree as the proof.

Call v1 directly:

```rust
verify_transaction_inclusion(ProofArgs {
    tx_id:              <internal_node_hash>,   // NOT a real transaction
    tx_block_blockhash: <mainchain_block_hash>,
    tx_index:           <position_of_internal_node>,
    merkle_proof:       <sibling_path_to_root>, // shorter than a leaf proof
    confirmations:      0,
})
// → returns true
```

The existing test suite has no case that would catch or document this behaviour, consistent with the "essentially empty test" pattern identified in the external report. [7](#0-6) [3](#0-2)

    #[pause]
    pub fn verify_transaction_inclusion(&self, #[serializer(borsh)] args: ProofArgs) -> bool {
        require!(
            args.confirmations <= self.gc_threshold,
            "The required number of confirmations exceeds the number of blocks stored in memory"
        );

        let heaviest_block_header = self
            .headers_pool
            .get(&self.mainchain_tip_blockhash)
            .unwrap_or_else(|| env::panic_str(ERR_KEY_NOT_EXIST));
        let target_block_height = self
            .mainchain_header_to_height
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("block does not belong to the current main chain"));

        // Check requested confirmations. No need to compute proof if insufficient confirmations.
        require!(
            (heaviest_block_header.block_height).saturating_sub(target_block_height) + 1
                >= args.confirmations,
            "Not enough blocks confirmed"
        );

        let header = self
            .headers_pool
            .get(&args.tx_block_blockhash)
            .unwrap_or_else(|| env::panic_str("cannot find requested transaction block"));

        require!(!args.merkle_proof.is_empty(), "Merkle proof is empty");

        // compute merkle tree root and check if it matches block's original merkle tree root
        merkle_tools::compute_root_from_merkle_proof(
            args.tx_id,
            usize::try_from(args.tx_index).unwrap(),
            &args.merkle_proof,
        ) == header.block_header.merkle_root
    }

pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
