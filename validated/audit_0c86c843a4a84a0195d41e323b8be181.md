### Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` ŌĆö (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension::beforeSwap` gates swaps by checking the `sender` argument passed from the pool. When a swap is routed through `MetricOmmSimpleRouter`, the pool sees the **router's address** as `msg.sender`, not the original user's address. This makes the allowlist check the router's identity rather than the actual swapper's identity, breaking the core invariant of the extension.

### Finding Description

**Call chain for a router-mediated swap:**

```
User (EOA)
  ŌåÆ MetricOmmSimpleRouter::exactInputSingle()   [msg.sender = user]
    ŌåÆ MetricOmmPool::swap()                      [msg.sender = router]
      ŌåÆ ExtensionCalling::_beforeSwap(msg.sender=router, ...)
        ŌåÆ SwapAllowlistExtension::beforeSwap(sender=router, ...)
          checks: allowedSwapper[pool][router]   ŌåÉ WRONG identity
```

**Pool `swap()` passes its own `msg.sender` as `sender`:** [1](#0-0) 

**`ExtensionCalling::_beforeSwap` forwards that value unchanged:** [2](#0-1) 

**The hook checks `allowedSwapper[msg.sender][sender]` where `msg.sender` = pool and `sender` = router:** [3](#0-2) 

**The router sets `msg.sender` (the user) only as the `payer` in transient storage ŌĆö it is never forwarded to the pool as the swap identity:** [4](#0-3) 

This creates two mutually exclusive failure modes:

| Router allowlist state | Result |
|---|---|
| Router **not** allowlisted | Allowlisted users cannot use the router at all ŌĆö broken functionality |
| Router **allowlisted** | **Any** unprivileged user can bypass the allowlist by routing through the router |

The second case is the exploit: a pool admin who allowlists the router (the natural action to let their users use it) inadvertently opens the pool to all users.

### Impact Explanation

The `SwapAllowlistExtension` is the sole mechanism for restricting swap access on a pool. When the router is allowlisted, the restriction is completely nullified ŌĆö any address can call `exactInputSingle`/`exactInput`/`exactOutputSingle`/`exactOutput` and execute swaps on a pool that was supposed to be access-controlled. This breaks core pool functionality and constitutes unauthorized swap execution.

### Likelihood Explanation

Medium-High. Any pool admin who deploys a pool with `SwapAllowlistExtension` and also wants their users to use the router will naturally allowlist the router address. The documentation or interface gives no indication that doing so opens the pool to all users. The bypass requires no special privileges ŌĆö any EOA can call the router.

### Recommendation

The extension must receive the **original user's address**, not the immediate pool caller. Two options:

1. **Pass the original sender through `extensionData`**: The router encodes `msg.sender` into `extensionData`; the extension decodes and verifies it. This requires a trusted encoding convention.
2. **Preferred ŌĆö check `tx.origin` or use a dedicated router-aware allowlist**: The extension checks `allowedSwapper[pool][tx.origin]` when `sender` is a known router, or the allowlist is redesigned to be router-aware (e.g., allowlist entries specify `(router, user)` pairs).

The cleanest fix is for the pool's hook interface to carry the original initiator as a separate field, distinct from the immediate `sender`.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` ŌĆö intending to let their allowlisted users use the router.
3. Attacker (not on the allowlist) calls `MetricOmmSimpleRouter::exactInputSingle(pool, ...)`.
4. Pool calls `beforeSwap(sender=router, ...)`.
5. Hook checks `allowedSwapper[pool][router]` ŌåÆ `true` ŌåÆ swap proceeds.
6. Attacker successfully swaps on a pool they were never authorized to access.

Direct swap by the attacker (without the router) would correctly revert: `allowedSwapper[pool][attacker]` ŌåÆ `false`.

The "stop-loss threshold / watermark boundary" framing in the question is not reflected in the actual `SwapAllowlistExtension` code ŌĆö the extension contains no drawdown, decay, or watermark logic. The real and only vulnerability is the identity mismatch described above.
