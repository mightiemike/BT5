### Title
V6 (Spice) `ShardChunkHeader` Round-Tripped Through `ChunkHeaderView` Is Silently Reconstructed as V3, Producing a Wrong `ChunkHash` and Breaking Signature Verification — (`core/primitives/src/views.rs`)

---

### Summary

`From<ShardChunkHeader> for ChunkHeaderView` unconditionally emits `congestion_info: Some(...)` for every inner version, including V6. Because `From<ChunkHeaderView> for ShardChunkHeader` infers the inner version solely from which optional fields are `Some`, a V6 header is always reconstructed as `ShardChunkHeaderInner::V3`. The Borsh serialization of V3 differs from V6 (different discriminant byte and different field set), so `ShardChunkHeaderV3::compute_hash` produces a hash that does not match the hash the chunk producer signed. Any code path that round-trips a Spice chunk header through `ChunkHeaderView` — including JSON-RPC responses and light-client verification — will hold a `ShardChunkHeader` whose `chunk_hash()` is wrong and whose signature verification will always fail.

---

### Finding Description

**Step 1 — Forward conversion injects a synthetic `congestion_info` for V6.**

`From<ShardChunkHeader> for ChunkHeaderView` calls `inner.congestion_info()` unconditionally: [1](#0-0) 

For `ShardChunkHeaderInner::V6`, `congestion_info()` returns `CongestionInfo::default()` with a `debug_assert` comment acknowledging this is a stub: [2](#0-1) 

V6 also returns `None` for `bandwidth_requests()`: [3](#0-2) 

And `false` for `has_proposed_split_field()`: [4](#0-3) 

So a V6 header serialized to `ChunkHeaderView` always has `(proposed_split=None, bandwidth_requests=None, congestion_info=Some(...))`.

**Step 2 — Reverse conversion reconstructs V3, not V6.**

`From<ChunkHeaderView> for ShardChunkHeader` infers the inner version from the combination of optional fields. The `(None, None, Some(congestion_info))` arm unconditionally builds `ShardChunkHeaderInner::V3`: [5](#0-4) 

There is no arm for V6. The code even carries a `TODO` acknowledging the version is guessed from field presence: [6](#0-5) 

**Step 3 — `header.init()` recomputes the hash from the wrong inner.**

After building the V3 inner, `init()` is called: [7](#0-6) 

`compute_hash` Borsh-serializes the inner enum. V3 has discriminant byte `2` and carries `prev_state_root`, `prev_outcome_root`, `gas_limit`, `prev_gas_used`, `prev_balance_burnt`, `prev_validator_proposals`, and `congestion_info` — none of which exist in V6 (`ShardChunkHeaderInnerV6SpiceTxOnly`, discriminant `5`): [8](#0-7) [9](#0-8) 

The resulting `ChunkHash` is computed from entirely different bytes than the original V6 hash.

**Step 4 — Signature verification fails.**

The chunk producer signed the V6 hash. Verification uses `chunk_header.chunk_hash()`: [10](#0-9) 

After the round-trip, `chunk_hash()` returns the recomputed V3 hash. `signature.verify(wrong_hash, public_key)` returns `false` for every valid V6 chunk.

---

### Impact Explanation

Any production code path that:
1. Serializes a V6 `ShardChunkHeader` to `ChunkHeaderView` (e.g., via the JSON-RPC `chunk` or `block` endpoints), and
2. Deserializes the JSON back and calls `ShardChunkHeader::from(view)`,

will hold a `ShardChunkHeader` with the wrong inner type (V3 instead of V6), the wrong `ChunkHash`, and a signature that will never verify. This breaks:

- **Light-client chunk-header verification** — any light client that fetches chunk headers via RPC and verifies signatures will permanently reject all Spice-era chunks.
- **RPC clients** that round-trip `ChunkHeaderView` for any purpose that subsequently checks `chunk_hash` equality or signature validity.
- The `chunk_hash` field stored in `ChunkHeaderView` (line 1134, copied from the original) will disagree with `chunk_hash()` on the reconstructed `ShardChunkHeader`, so even a simple equality check exposes the divergence.

The consensus path (Borsh network messages → `ShardChunkHeader` directly) is **not** affected; the bug is confined to the JSON/view layer.

---

### Likelihood Explanation

- Spice (`ProtocolFeature::Spice`) is an active development feature with V6 chunk headers already in the codebase and used by `new_for_spice`.
- The `chunk` and `block` JSON-RPC endpoints are called by every explorer, indexer, and light client.
- The `From<ChunkHeaderView> for ShardChunkHeader` impl is a public production API with no guard against V6.
- The bug is deterministic: **every** V6 chunk header round-tripped through `ChunkHeaderView` is affected.

---

### Recommendation

1. Add a version discriminant field to `ChunkHeaderView` (the `TODO` at line 1174 already acknowledges this need) so the reverse conversion can reconstruct the correct inner variant without guessing.
2. Until then, add a V6 arm to `From<ChunkHeaderView> for ShardChunkHeader` that detects Spice chunks (e.g., via a new boolean field `is_spice_chunk` in `ChunkHeaderView`) and reconstructs `ShardChunkHeaderInner::V6`.
3. Add a round-trip test: serialize a V6 `ShardChunkHeader` to `ChunkHeaderView`, deserialize, convert back, and assert `chunk_hash` equality.
