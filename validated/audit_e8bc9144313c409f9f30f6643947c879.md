### Title
Tokens Sent to `DirectDepositV1` That Are Not Registered Spot Products Are Permanently Stuck With No User-Accessible Recovery Path — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` iterates only over `spotEngine.getProductIds()` to deposit token balances held by the contract. Any ERC20 token sent to a `DirectDepositV1` instance that is not a currently registered spot product is silently skipped and remains locked in the contract. The user has no self-service recovery path; only the protocol owner can extract the tokens via `ContractOwner.withdrawFromDirectDepositV1()`, and those tokens are sent to the owner, not the user.

---

### Finding Description

`creditDeposit()` is an `external` function with no access control. It loops over the current product list from `spotEngine`:

```solidity
// core/contracts/DirectDepositV1.sol
function creditDeposit() external {
    uint32[] memory productIds = spotEngine.getProductIds();
    for (uint256 i = 0; i < productIds.length; i++) {
        uint32 productId = productIds[i];
        address tokenAddr = spotEngine.getToken(productId);
        require(tokenAddr != address(0), "Invalid productId.");
        IIERC20Base token = IIERC20Base(tokenAddr);
        uint256 balance = token.balanceOf(address(this));
        if (balance != 0) {
            token.approve(address(endpoint), balance);
            endpoint.depositCollateralWithReferral(
                subaccount,
                productId,
                uint128(balance),
                "-1"
            );
        }
    }
}
``` [1](#0-0) 

The loop is bounded strictly to registered products. Any token balance held by the contract that does not correspond to a current product ID is never touched. The only user-facing recovery mechanism is `DirectDepositV1.withdraw()`, which is `onlyOwner`:

```solidity
function withdraw(IIERC20Base token) external onlyOwner {
    uint256 balance = token.balanceOf(address(this));
    safeTransfer(token, msg.sender, balance);
}
``` [2](#0-1) 

The owner of each `DirectDepositV1` instance is the `ContractOwner` contract, not the end user. The protocol-level rescue function `withdrawFromDirectDepositV1` is also `onlyOwner` of `ContractOwner` and transfers recovered tokens to `msg.sender` (the protocol owner), not to the subaccount holder:

```solidity
function withdrawFromDirectDepositV1(bytes32 subaccount, address token)
    external
    onlyOwner
{
    ...
    IERC20Base(token).safeTransfer(msg.sender, postBalance - preBalance);
}
``` [3](#0-2) 

The protocol team has already acknowledged this class of problem for the specific USDC.e → USDC migration case, adding a bespoke `replaceUsdcEWithUsdc()` function. However, no general-purpose handling exists for arbitrary non-product tokens. [4](#0-3) 

---

### Impact Explanation

Any ERC20 token sent to a `DirectDepositV1` address that is not currently registered in `spotEngine` will be silently skipped by `creditDeposit()` and remain locked in the contract indefinitely. The user cannot recover the tokens themselves. The protocol owner can extract them via `withdrawFromDirectDepositV1`, but the extracted tokens are sent to the owner's address, not returned to the user. This constitutes a direct, concrete loss of user assets with no self-service remedy.

---

### Likelihood Explanation

Realistic trigger scenarios include:

1. **Token migration**: The protocol already migrated from USDC.e to USDC on Ink Mainnet. A user who sends USDC to their `DirectDepositV1` address while only USDC.e is registered (or vice versa) loses those tokens to the contract.
2. **Product delisting**: `delistProduct` removes a product from the engine. If a user sends a token for a delisted product to their `DirectDepositV1` address and then calls `creditDeposit()`, the token is skipped.
3. **User error**: A user sends any non-product ERC20 (e.g., a reward token, an LP token, a bridged variant) to their deposit address. `creditDeposit()` silently ignores it.

The `receive()` fallback wraps native ETH to WETH, so native token deposits are handled if WETH is a registered product. But any other token mismatch is unhandled. [5](#0-4) 

---

### Recommendation

Add a fallback sweep inside `creditDeposit()` (or a separate permissionless rescue function) that detects any token balance held by the contract that was not consumed by the product loop and either returns it to the subaccount owner or reverts with a clear error. At minimum, document that non-product tokens sent to a `DirectDepositV1` address are unrecoverable by the user, and ensure `withdrawFromDirectDepositV1` routes recovered tokens back to the subaccount holder rather than to `msg.sender`.

---

### Proof of Concept

1. User U has a `DirectDepositV1` at address `DDA` for subaccount `S`.
2. The protocol lists only USDC.e as a spot product. USDC (native) is not listed.
3. U sends 1000 USDC to `DDA`.
4. U (or anyone) calls `DDA.creditDeposit()`.
5. The loop iterates over `spotEngine.getProductIds()`, finds USDC.e, checks `USDC.balanceOf(DDA) == 0` (since U sent USDC, not USDC.e), and exits without depositing anything.
6. The 1000 USDC remain in `DDA`. U has no function to call to recover them.
7. Only the protocol owner can call `ContractOwner.withdrawFromDirectDepositV1(S, USDC)`, which sends the 1000 USDC to the owner's wallet, not to U. [1](#0-0) [3](#0-2)

            require(postBalance > preBalance, "empty");

            );
            require(success, "xfer");
        } else {
            uint256 preBalance = IERC20Base(token).balanceOf(address(this));
            DirectDepositV1(directDepositV1).withdraw(IIERC20Base(token));
            uint256 postBalance = IERC20Base(token).balanceOf(address(this));

            IERC20Base(token).safeTransfer(
                msg.sender,
                postBalance - preBalance
