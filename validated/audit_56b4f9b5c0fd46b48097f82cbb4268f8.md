### Title
Secp256k1 and Secp256r1 4-Byte Opcodes Are Wired to the Wrong Verify Functions - (`File: src/chia_dialect.rs`)

### Summary
In `src/chia_dialect.rs`, the two 4-byte secp opcode constants (`0x13d61f00` and `0x1c3a8f00`) are dispatched to the wrong verify functions. The opcode whose embedded cost encodes 1300000 (the secp256k1 cost) is wired to `op_secp256r1_verify`, and the opcode whose embedded cost encodes 1850000 (the secp256r1 cost) is wired to `op_secp256k1_verify`. This is the direct analog of the original report's "left and right side of the equation are flipped" pattern: the two operators are swapped at the dispatch table.

### Finding Description

The 4-byte opcode dispatch block in `ChiaDialect::op` reads:

```rust
let f = match opcode {
    0x13d61f00 => op_secp256r1_verify,
    0x1c3a8f00 => op_secp256k1_verify,
    ...
};
``` [1](#0-0) 

The 4-byte opcode format encodes cost in the top 3 bytes (with an implied +1). Decoding the two opcodes:

| Opcode | Top-3-byte multiplier | Decoded cost | Correct operator |
|---|---|---|---|
| `0x13d61f00` | `0x13d61f` = 1,299,999 | **1,300,000** | `op_secp256k1_verify` |
| `0x1c3a8f00` | `0x1c3a8f` = 1,849,999 | **1,850,000** | `op_secp256r1_verify` |

The cost constants in `src/secp_ops.rs` confirm the intended pairing:

```rust
const SECP256R1_VERIFY_COST: Cost = 1850000;
const SECP256K1_VERIFY_COST: Cost = 1300000;
``` [2](#0-1) 

The single-byte opcode dispatch (gated by `ENABLE_SECP_OPS`) is consistent with the cost constants and confirms the intended mapping:

```rust
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [3](#0-2) 

The 4-byte dispatch is the exact inverse of the single-byte dispatch: `0x13d61f00` (cost 1,300,000 = k1) calls `op_secp256r1_verify`, and `0x1c3a8f00` (cost 1,850,000 = r1) calls `op_secp256k1_verify`.

Each verify function charges its own internal cost constant, not the cost encoded in the opcode:

```rust
pub fn op_secp256r1_verify(...) -> Response {
    let cost = SECP256R1_VERIFY_COST;   // 1850000
    ...
}
pub fn op_secp256k1_verify(...) -> Response {
    let cost = SECP256K1_VERIFY_COST;   // 1300000
    ...
}
``` [4](#0-3) [5](#0-4) 

### Impact Explanation

Two concrete corruptions occur simultaneously for any CLVM program that uses the 4-byte secp opcodes:

1. **Wrong cryptographic algorithm executed.** A program invoking `0x13d61f00` with a secp256k1 public key and signature will run `op_secp256r1_verify`, which will reject the valid secp256k1 signature (and vice versa). Legitimate coin puzzles that rely on these opcodes will always fail verification, causing permanent loss of spendability.

2. **Wrong cost charged.** `0x1c3a8f00` dispatches to `op_secp256k1_verify`, which charges only 1,300,000 instead of the 1,850,000 encoded in the opcode and expected for secp256r1. An attacker who crafts a CLVM program using `0x1c3a8f00` to invoke secp256k1 verification pays 550,000 fewer cost units than the opcode's own encoding declares, undercharging the more expensive curve operation relative to the declared budget.

### Likelihood Explanation

The 4-byte opcode path is unconditionally reachable: it requires no flag (`ENABLE_SECP_OPS` is not checked), only that the operator atom is exactly 4 bytes. Any attacker-controlled CLVM program submitted to the mempool or included in a block can trigger this path by embedding the 4-byte opcode literal. The entry path is: attacker supplies CLVM bytes → `run_program()` → `ChiaDialect::op()` → 4-byte branch → wrong function called. [6](#0-5) 

### Recommendation

Swap the two 4-byte opcode mappings so that the opcode encoding cost 1,300,000 calls `op_secp256k1_verify` and the opcode encoding cost 1,850,000 calls `op_secp256r1_verify`:

```rust
0x13d61f00 => op_secp256k1_verify,
0x1c3a8f00 => op_secp256r1_verify,
```

This aligns the 4-byte dispatch with both the cost constants in `secp_ops.rs` and the single-byte opcode dispatch (opcodes 64 and 65).

### Proof of Concept

Decoded cost arithmetic:

```
0x13d61f = (0x13 << 16) | (0xd6 << 8) | 0x1f
         = (19 × 65536) + (214 × 256) + 31
         = 1,245,184 + 54,784 + 31 = 1,299,999
cost     = 1,299,999 + 1 = 1,300,000  ← SECP256K1_VERIFY_COST

0x1c3a8f = (0x1c << 16) | (0x3a << 8) | 0x8f
         = (28 × 65536) + (58 × 256) + 143
         = 1,835,008 + 14,848 + 143 = 1,849,999
cost     = 1,849,999 + 1 = 1,850,000  ← SECP256R1_VERIFY_COST
```

The code maps `0x13d61f00` (k1 cost) → `op_secp256r1_verify` and `0x1c3a8f00` (r1 cost) → `op_secp256k1_verify`. Both the algorithm executed and the cost charged are wrong for each opcode. [1](#0-0) [2](#0-1)
