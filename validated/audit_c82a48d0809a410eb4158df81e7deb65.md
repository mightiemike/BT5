### Title
Unvalidated `_clearinghouseLiq` in `Clearinghouse.initialize` Enables Silent Liquidation Bypass — (File: `core/contracts/Clearinghouse.sol`)

---

### Summary

`Clearinghouse.initialize` stores `_clearinghouseLiq` without a zero-address check. If this parameter is zero (or any address with no deployed code), every call to `liquidateSubaccount` silently succeeds via `delegatecall` to `address(0)` without executing any liquidation logic. Unhealthy subaccounts are never actually liquidated, breaking the protocol's solvency invariant.

---

### Finding Description

`Clearinghouse.initialize` accepts `_clearinghouseLiq` and stores it directly with no validation:

```solidity
function initialize(
    address _endpoint,
    address _quote,
    address _clearinghouseLiq,
    uint256 _spreads,
    address _withdrawPool
) external initializer {
    ...
    clearinghouseLiq = _clearinghouseLiq;   // no zero-address check
    ...
}
``` [1](#0-0) 

`liquidateSubaccount` then performs a `delegatecall` to `clearinghouseLiq` and only checks the boolean success flag:

```solidity
(bool success, bytes memory result) = clearinghouseLiq.delegatecall(
    liquidateSubaccountCall
);
require(success, string(result));
``` [2](#0-1) 

In the EVM, a `delegatecall` to `address(0)` (or any address with no deployed bytecode) returns `(true, "")`. The `require(success, ...)` guard therefore passes unconditionally, and the function returns without executing any liquidation. The sequencer's liquidation transaction is accepted on-chain, emits no error, and the unhealthy subaccount remains intact.

By contrast, the post-deployment upgrade path `upgradeClearinghouseLiq` also lacks a zero-address guard, meaning the same silent-bypass condition can be introduced after deployment as well:

```solidity
function upgradeClearinghouseLiq(address _clearinghouseLiq) external {
    require(msg.sender == IProxyManager(_getProxyManager()).getProxyManagerHelper(), ERR_UNAUTHORIZED);
    clearinghouseLiq = _clearinghouseLiq;   // no zero-address check
}
``` [3](#0-2) 

The inconsistency is notable: `BaseProxyManager.bootstrapEndpointTx` explicitly validates both zero-address and code-length for `_endpointTx`, but no equivalent guard exists for `_clearinghouseLiq`. [4](#0-3) 

---

### Impact Explanation

If `clearinghouseLiq` is zero or points to an EOA/empty address:

- Every `liquidateSubaccount` call silently no-ops. Subaccounts whose maintenance health has fallen below zero are never liquidated.
- The insurance fund is not replenished from liquidation proceeds.
- Other traders' collateral backing the system is exposed to bad debt that accumulates without bound.
- The broken invariant is concrete: subaccount health state diverges from what the risk model requires, corrupting the accounting layer.

This is a solvency/accounting corruption impact, not merely a DoS.

---

### Likelihood Explanation

The external report's accepted finding involves the same pattern: an `initialize` function that accepts an address/identifier without validation, where the deployer can (accidentally or intentionally) supply an invalid value. The cLabs team explicitly noted they were "intentionally providing an invalid address via parameters." The same scenario applies here: a deployment script that omits or zeroes `_clearinghouseLiq` (e.g., during a staged rollout where `ClearinghouseLiq` is deployed separately) would silently disable all liquidations with no on-chain error signal. The absence of any validation in both `initialize` and `upgradeClearinghouseLiq` makes this a persistent risk across the contract's lifecycle.

---

### Recommendation

Add a zero-address (and optionally code-length) check in both `initialize` and `upgradeClearinghouseLiq`:

```solidity
require(_clearinghouseLiq != address(0), "clearinghouseLiq is zero");
require(_clearinghouseLiq.code.length > 0, "clearinghouseLiq has no code");
```

This mirrors the validation already present in `bootstrapEndpointTx` for `_endpointTx`. [4](#0-3) 

---

### Proof of Concept

1. Deploy `Clearinghouse` and call `initialize` with `_clearinghouseLiq = address(0)`.
2. Allow a subaccount's maintenance health to drop below zero (e.g., via adverse price movement).
3. Sequencer submits a `LiquidateSubaccount` transaction through `Endpoint`.
4. `Endpoint` calls `clearinghouse.liquidateSubaccount(txn)`.
5. Inside `liquidateSubaccount`, `address(0).delegatecall(...)` returns `(true, "")`.
6. `require(true, "")` passes. Function returns.
7. The subaccount's balances are unchanged. Health remains negative. No liquidation occurred.
8. Repeat indefinitely — bad debt accumulates, insurance fund is never replenished. [5](#0-4)
