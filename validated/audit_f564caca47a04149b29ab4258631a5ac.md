Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End-User, Bypassing Per-User Swap Allowlist — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is `msg.sender` of the pool's `swap()` call. When a user routes through `MetricOmmSimpleRouter`, `sender` is the router's address, not the end user's. Because the router must be allowlisted for any router-mediated swap to succeed on a curated pool, every user — including those explicitly denied — can bypass the per-user swap allowlist by calling the pool through the router.

## Finding Description

`MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, recipient, ...)`, forwarding `msg.sender` (the immediate caller of the pool) as the `sender` argument. [1](#0-0) 

`ExtensionCalling._beforeSwap` passes this value directly to `IMetricOmmExtensions.beforeSwap` as the first argument. [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks whether that `sender` is allowlisted: [3](#0-2) 

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or any `exact*` variant), the router calls `pool.swap(...)` with itself as `msg.sender`. [4](#0-3) 

The extension therefore sees `sender = address(router)`, not the end user. If the pool admin allowlists the router address (which is operationally necessary for any router-based swap to work on a curated pool), the allowlist check passes for every user who routes through it, regardless of whether that individual user is permitted.

By contrast, `DepositAllowlistExtension.beforeAddLiquidity` correctly checks the `owner` parameter — the economically attributed actor — which is not substituted by the router. [5](#0-4) 

The design inconsistency is structural: the swap hook binds on the immediate pool caller (`sender`), while the deposit hook binds on the position owner (`owner`). For swaps, there is no equivalent "owner" field in the hook signature that carries the true end-user identity through the router.

## Impact Explanation

A curated pool deploying `SwapAllowlistExtension` to restrict trading to a known set of counterparties loses that restriction entirely for any user who routes through `MetricOmmSimpleRouter`. Disallowed users can execute swaps on the restricted pool, draining LP value or violating compliance constraints the pool admin intended to enforce. This constitutes a direct bypass of a core pool access-control mechanism with fund-impacting consequences for LPs on restricted pools — matching the "broken core pool functionality causing loss of funds" and "admin-boundary break bypassed by an unprivileged path" impact categories.

## Likelihood Explanation

`MetricOmmSimpleRouter` is the standard supported periphery path for end users. Any pool admin who allowlists the router (a natural and necessary operational step) unknowingly opens the bypass. The trigger requires no privilege — any unpermissioned user can call the router. The misconfiguration is latent in the design of the actor-binding, not in any unusual setup. The bypass is repeatable and unconditional once the router is allowlisted.

## Recommendation

The `SwapAllowlistExtension` must gate on the end user, not the immediate pool caller. Concrete options:

1. **Pass the originating user through `extensionData`**: The router encodes the true end-user address (`msg.sender` at the router level) in `extensionData`, and the extension decodes and checks it. This requires a trusted encoding convention and that the extension only accepts this path from a known router.
2. **Check `recipient` instead of `sender`**: If pool swap semantics guarantee `recipient` is always the economic beneficiary and cannot be set to an arbitrary address by the attacker, gate on `recipient`. This must be verified to hold for all router paths.
3. **Redesign the hook signature**: Add a separate `originator` field that the pool populates from a trusted source, analogous to how `owner` is passed in the liquidity hooks.

The cleanest fix is option 1 with verified encoding, mirroring the pattern already used correctly in `DepositAllowlistExtension`.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, address(router), true)` to enable normal router usage.
3. Pool admin does **not** call `setAllowedToSwap(pool, address(attacker), true)`.
4. Attacker calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the curated pool with `recipient = attacker`.
5. The router calls `pool.swap(attacker, ...)` — `msg.sender` to the pool is `address(router)`.
6. `_beforeSwap` fires with `sender = address(router)`.
7. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][address(router)] == true` → passes.
8. Attacker's swap executes on the curated pool despite being individually denied.

A Foundry integration test can reproduce this by: deploying `SwapAllowlistExtension` and a pool with it in `beforeSwap` order, deploying `MetricOmmSimpleRouter`, allowlisting only the router, then calling `exactInputSingle` from an unallowlisted attacker address and asserting the swap succeeds (no revert).
