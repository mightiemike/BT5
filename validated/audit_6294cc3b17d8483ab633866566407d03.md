### Title
IBC-Received NFT Classes Created with Empty `Name` (and `Schema`) Due to Missing Fields in `NonFungibleTokenPacketData` — (`x/nft-transfer/keeper/packet.go`, `x/nft-transfer/types/packet.proto`)

---

### Summary

When an NFT is transferred cross-chain via ICS-721, the receiving chain creates a new `Denom` (NFT class) from the inbound `NonFungibleTokenPacketData`. However, the packet format only carries `class_id` and `class_uri` — it does not carry `class_name`, `schema`, or `creator`. As a result, every IBC-received NFT class on this chain is permanently created with an empty `Name` and `Schema`, regardless of what the source chain had set. This is a direct structural analog to the Nibbl `Basket` proxy bug: in both cases, a critical metadata field is initialized in one context (source chain / implementation contract) but is structurally absent in the context where the object is actually created for users (destination chain / proxy storage).

---

### Finding Description

The `NonFungibleTokenPacketData` protobuf message, which is the sole carrier of class information across IBC, is defined as: [1](#0-0) 

It contains only `class_id` and `class_uri`. There is no `class_name`, `class_schema`, or `class_creator` field.

The `Denom` type (the NFT class on this chain) has the following fields: [2](#0-1) 

When a new class is created natively via `MsgIssueDenom`, all fields — `Id`, `Name`, `Schema`, `Creator`, `Uri` — are populated: [3](#0-2) 

But when `processReceivedPacket` (called from `OnRecvPacket`) creates a new `Denom` on the receiving chain, it can only populate `Id` (from `ClassId`) and `Uri` (from `ClassUri`). The `Name`, `Schema`, and `Creator` fields are structurally absent from the packet and will be empty strings for every IBC-received class.

The receive path is: [4](#0-3) 

Which is triggered by: [5](#0-4) 

The outgoing packet is constructed in `createOutgoingPacket` (in `packet.go`), which reads the source chain's `Denom` but can only serialize `class_id` and `class_uri` into the `NonFungibleTokenPacketData` — the `Name` and `Schema` are silently dropped at the serialization boundary. [6](#0-5) 

---

### Impact Explanation

Every NFT class created on this chain via IBC receive will have:
- `Denom.Name` = `""` (empty)
- `Denom.Schema` = `""` (empty)
- `Denom.Creator` = `""` (empty)

This breaks all integrations and explorers that rely on `Denom.Name` to display the collection name (the direct analog of ERC-721 `name()`/`symbol()`). Any marketplace, block explorer, or wallet querying the NFT class metadata for IBC-received collections will display blank names. The `Schema` field, which encodes the JSON schema for NFT data validation, is also lost, potentially breaking downstream data validation logic for those collections.

---

### Likelihood Explanation

This is triggered by any standard IBC NFT transfer. Any user who calls `MsgTransfer` to send an NFT from this chain to another chain and back, or who receives an NFT from any ICS-721 counterparty chain, will produce a class with an empty `Name`. The entry path is fully unprivileged — it requires only a valid IBC channel and a `MsgTransfer` transaction signed by the NFT owner. [7](#0-6) 

---

### Recommendation

Extend `NonFungibleTokenPacketData` to include `class_name` and `class_data` (schema) fields, mirroring the approach taken by the ICS-721 spec update that added `class_data`. On the sending side, populate these fields from the source `Denom`. On the receiving side, use them when constructing the new `Denom` so that `Name` and `Schema` are faithfully reproduced on the destination chain.

---

### Proof of Concept

1. Chain A has a `Denom` with `Id="myclass"`, `Name="My NFT Collection"`, `Schema="{...}"`.
2. User calls `MsgTransfer` on Chain A to send token `token1` of class `myclass` to Chain B.
3. `createOutgoingPacket` builds a `NonFungibleTokenPacketData{ClassId: "myclass", ClassUri: "...", TokenIds: ["token1"], ...}` — `Name` and `Schema` are not included.
4. Chain B's `OnRecvPacket` → `processReceivedPacket` creates a new `Denom` with `Id="ibc/<hash>"`, `Uri="..."`, `Name=""`, `Schema=""`.
5. Any query of the class on Chain B returns an empty `Name`, breaking all frontend integrations. [8](#0-7) [2](#0-1)
