### Title
Gas-Key Host-Function Fee Uses Wire Length Instead of Trie-ID Length for ML-DSA-65 Keys, Causing Deterministic Fee Divergence from Transaction Path - (`runtime/near-vm-runner/src/logic/logic.rs`, `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`)

### Summary

When a contract calls `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, or `promise_batch_action_transfer_to_gas_key` with an ML-DSA-65 public key, the exec/send fee is computed using `public_key_len as usize` — the raw borsh-encoded wire length (1953 bytes) — instead of `public_key.trie_id_len()` — the on-trie hash length (33 bytes). The transaction path (`permission_exec_fees` / `total_send_fees`) correctly calls `public_key.trie_id_len()`. This produces a deterministic, protocol-level fee divergence of ~1920 × `gas_key_byte` × `num_nonces` gas units between the two submission paths for the same action.

### Finding Description

`PublicKey::len()` and `PublicKey::trie_id_len()` diverge only for `PublicKey::MLDSA65`:

- `len()` → 1 + 1952 = **1953** bytes (borsh wire form)
- `trie_id_len()` → 1 + 32 = **33** bytes (SHA3-256 hash stored in trie)

The documentation explicitly warns: *"Callers that still use `len()` for trie-storage costing will misprice ML-DSA-65 keys by ~1900 bytes."*

**Transaction path** (`runtime/runtime/src/config.rs`, `permission_exec_fees`):
```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← 33 for ML-DSA-65
    gas_key_info.num_nonces,
);
```

**Host-function path** (`runtime/near-vm-runner/src/logic/logic.rs`, `promise_batch_action_add_gas_key_with_full_access`):
```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← 1953 for ML-DSA-65 (raw wire length from contract memory)
    num_nonces,
);
```

The same pattern appears in `promise_batch_action_add_gas_key_with_function_call` and in the wasmtime runner equivalents. For `TransferToGasKey`, the send fee also diverges:

**Transaction path** (`runtime/runtime/src/config.rs`):
```rust
TransferToGasKey(action) => {
    gas_key_transfer_send_fee(fees, sender_is_receiver, action.public_key.trie_id_len())
```

**Host-function path** (`runtime/near-vm-runner/src/wasmtime_runner/logic.rs`):
```rust
let send = gas_key_transfer_send_fee(&ctx.fees_config, sir, public_key_len as usize);
```

The `gas_key_add_key_exec_fee` function feeds `public_key_len` into `access_key_key_len(account_id_len, public_key_len)`, which computes the nonce trie-key length. For ML-DSA-65, the actual nonce trie key uses the 33-byte hash, not the 1953-byte pubkey, so the exec fee is inflated by `1920 × gas_key_byte × num_nonces`.

The existing fee-parity test (`test_gas_key_fee_parity`) only exercises ED25519 keys, where `len() == trie_id_len()`, so the divergence is invisible to the test suite.

### Impact Explanation

Any contract that calls the gas-key host functions with an ML-DSA-65 public key is overcharged by approximately `1920 × gas_key_byte × num_nonces` gas units on the exec side and `1920 × gas_key_byte` on the send side per `TransferToGasKey`. This is a deterministic, protocol-level fee invariant violation: the same `AddKey(GasKeyFullAccess)` action costs a different amount of gas depending on whether it is submitted as a transaction or dispatched via a host function. Contracts that budget gas based on the transaction-path cost will run out of gas when using the host-function path with ML-DSA-65 keys. The divergence is ~58× the correct fee for a single nonce (1953 vs 33 bytes), scaling linearly with `num_nonces`.

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature gates ML-DSA-65 key registration. Once that feature activates, any contract that programmatically manages ML-DSA-65 gas keys via host functions hits this overcharge unconditionally. The contract author controls `public_key_len` (it is the borsh-encoded length of the key they pass from contract memory), so the divergent value is fully user-controlled and deterministic. The existing test coverage does not exercise this code path with ML-DSA-65 keys.

### Recommendation

Replace `public_key_len as usize` with `public_key.decode()?.trie_id_len()` (after decoding the key) in all four gas-key host-function fee computations:

- `promise_batch_action_add_gas_key_with_full_access` (logic.rs and wasmtime_runner/logic.rs)
- `promise_batch_action_add_gas_key_with_function_call` (logic.rs and wasmtime_runner/logic.rs)
- `promise_batch_action_transfer_to_gas_key` (wasmtime_runner/logic.rs)

Extend `test_gas_key_fee_parity` to cover `KeyType::MLDSA65` keys so the invariant is enforced by the test suite.

### Proof of Concept

The divergence is exact and arithmetic:

| Path | `public_key_len` passed to `gas_key_add_key_exec_fee` | Nonce-key length used | Overcharge per nonce |
|---|---|---|---|
| Transaction (`permission_exec_fees`) | `public_key.trie_id_len()` = **33** | `access_key_key_len(acct, 33) + sizeof(NonceIndex)` | 0 |
| Host function (`promise_batch_action_add_gas_key_with_full_access`) | `public_key_len as usize` = **1953** | `access_key_key_len(acct, 1953) + sizeof(NonceIndex)` | `1920 × gas_key_byte` |

For `num_nonces = 4` (the value used in `test_gas_key_fee_parity`), the exec overcharge is `4 × 1920 × gas_key_byte`. The transaction path and host-function path would produce different `gas_burnt` values for the same `AddKey(GasKeyFullAccess { num_nonces: 4 })` action with an ML-DSA-65 key, directly falsifying the `assert_eq!(add_a_outcome.gas_burnt, add_b_outcome.gas_burnt)` assertion in `test_gas_key_fee_parity` if that test were extended to use `KeyType::MLDSA65`. pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }

    /// Length, in bytes, of the on-trie identifier for an access-key
    /// entry owned by this public key. For ed25519/secp256k1 this matches
    /// `len()`; for ML-DSA-65 the trie stores a SHA3-256 hash (33 bytes
    /// including the type tag), not the 1953-byte borsh-encoded pubkey.
    /// Used by storage-fee calculations on the runtime side; cheap to call
    /// (no hashing) - for ML-DSA-65 this returns the size of the digest
    /// form without actually hashing the pubkey.
    pub fn trie_id_len(&self) -> usize {
        match self {
            Self::ED25519(_) => 1 + ed25519_dalek::PUBLIC_KEY_LENGTH,
            Self::SECP256K1(_) => 1 + 64,
            Self::MLDSA65(_) => 1 + ML_DSA_65_HASH_LENGTH,
        }
    }
