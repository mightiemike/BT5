### Title
NFT Denom Registration Front-Running via Unbound First-Come-First-Served Uniqueness Check — (File: x/nft/keeper/denom.go)

---

### Summary
The `SetDenom` function enforces global uniqueness of `denomID` and `denomName` with no cryptographic or address-based binding to the submitting account. Any unprivileged user can observe a pending `MsgIssueDenom` transaction and submit an identical one with higher gas, preempting the legitimate creator. The attacker becomes the on-chain `Creator` of the denom and gains exclusive minting rights over it, while the legitimate user's transaction reverts.

---

### Finding Description
`SetDenom` in `x/nft/keeper/denom.go` performs two existence checks before writing a new denom:

```go
// x/nft/keeper/denom.go lines 26-39
func (k Keeper) SetDenom(ctx sdk.Context, denom types.Denom) error {
    if k.HasDenomID(ctx, denom.Id) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomID %s has already exists", denom.Id)
    }
    if k.HasDenomNm(ctx, denom.Name) {
        return sdkerrors.Wrapf(types.ErrInvalidDenom, "denomName %s has already exists", denom.Name)
    }
    store := ctx.KVStore(k.storeKey)
    bz := k.cdc.MustMarshal(&denom)
    store.Set(types.KeyDenomID(denom.Id), bz)
    store.Set(types.KeyDenomName(denom.Name), []byte(denom.Id))
    return nil
}
```

Neither check ties the `denomID` or `denomName` to the sender's address. The `MsgIssueDenom` message handler passes the sender only as the `Creator` field value after the uniqueness gate has already been passed:

```go
// x/nft/keeper/msg_server.go line 32
m.Keeper.IssueDenom(ctx, msg.Id, msg.Name, msg.Schema, msg.Uri, sender)
```

`ValidateBasic` on `MsgIssueDenom` validates only the format of `Id`, `Name`, and `Sender` — it imposes no ownership constraint on the chosen `denomID` or `denomName`.

An attacker who observes a pending `MsgIssueDenom` in the public mempool can immediately broadcast an identical message (same `Id`, same `Name`) with a higher gas price. If the attacker's transaction is ordered first by the validator, the attacker's address is stored as `Creator`. The legitimate user's transaction then hits the `HasDenomID` or `HasDenomNm` guard and reverts.

The `IsDenomCreator` function enforces that only the stored `Creator` may subsequently mint NFTs under the denom:

```go
// x/nft/keeper/denom.go lines 81-96
func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
    ...
    if !creator.Equals(address) {
        return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", ...)
    }
    return denom, nil
}
```

So the attacker permanently holds the minting key for the hijacked denom.

---

### Impact Explanation
The corrupted on-chain value is the `Creator` field of the `Denom` record stored under `types.KeyDenomID(denom.Id)`. The attacker's address is written there instead of the legitimate creator's address. Consequences:

1. **Legitimate creator is permanently locked out** of minting NFTs under their intended `denomID` and `denomName`.
2. **Attacker gains exclusive minting rights** and can issue arbitrary NFTs under the hijacked denom, including tokens that impersonate a well-known collection.
3. **Both uniqueness axes are vulnerable**: front-running on `denomName` alone is sufficient to block the legitimate user even if they retry with a different `denomID`, because the name is also globally unique.
4. For denoms intended to be used in IBC NFT-transfer flows, the hijacked `denomID` would be the class identifier embedded in ICS-721 packets, meaning the attacker controls the class that receivers see.

---

### Likelihood Explanation
The Cronos POS Chain uses CometBFT, whose mempool is publicly visible to any full node. `MsgIssueDenom` is a standard, low-cost transaction. An attacker needs only to:
- Run a full node or subscribe to a public RPC endpoint's `tx` event stream.
- Detect a pending `MsgIssueDenom` and rebroadcast it with the same `Id`/`Name` and a marginally higher gas price.

No privileged access, leaked keys, or social engineering is required. The attack is fully automated and repeatable.

---

### Recommendation
Bind the `denomID` to the sender's address at the protocol level — for example, by requiring `denomID` to be prefixed with the creator's bech32 address, or by deriving it deterministically from `(sender, sequence_number)`. Alternatively, enforce that `msg.Id` must equal a hash of `(sender_address || chosen_name || block_height)` so that no two senders can produce the same valid `denomID`.

---

### Proof of Concept

1. Alice broadcasts `MsgIssueDenom{Id: "artcollection", Name: "Art Collection", Sender: alice}`.
2. Bob's monitoring node sees the unconfirmed transaction in the mempool.
3. Bob broadcasts `MsgIssueDenom{Id: "artcollection", Name: "Art Collection", Sender: bob}` with `gas_price + 1`.
4. Bob's transaction is ordered before Alice's in the block.
5. `SetDenom` stores `Creator = bob` under key `KeyDenomID("artcollection")`.
6. Alice's transaction executes next; `HasDenomID("artcollection")` returns `true`; transaction reverts with `ErrInvalidDenom`.
7. Bob calls `MsgMintNFT{DenomId: "artcollection", ...}` — `IsDenomCreator` passes because `bob == Creator`; Bob mints NFTs under Alice's intended collection identity. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

func (k Keeper) IsDenomCreator(ctx sdk.Context, denomID string, address sdk.AccAddress) (types.Denom, error) {
	denom, err := k.GetDenom(ctx, denomID)
	if err != nil {
		return types.Denom{}, err
	}

	creator, err := sdk.AccAddressFromBech32(denom.Creator)
	if err != nil {
		panic(err)
	}

	if !creator.Equals(address) {
		return types.Denom{}, sdkerrors.Wrapf(types.ErrUnauthorized, "%s is not the creator of %s", address, denomID)
	}

	return denom, nil
