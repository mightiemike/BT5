### Title
Unprotected `initialize()` Allows Front-Running to Seize Ownership and Inject Malicious Token — (`core/contracts/Airdrop.sol`)

---

### Summary

`Airdrop.initialize()` has no caller restriction. Any address can call it on the proxy before the legitimate deployer does. The first caller becomes the owner and sets `token` to an arbitrary address. The `_disableInitializers()` in the constructor only protects the implementation contract; the proxy's storage is entirely uninitialized until `initialize()` is called on it.

---

### Finding Description

`Airdrop` is an upgradeable proxy contract. Its constructor calls `_disableInitializers()`, which prevents re-initialization of the **implementation** contract, but does nothing to protect the **proxy** instance. [1](#0-0) 

`initialize()` is `external` with only the `initializer` modifier — no `onlyOwner`, no deployer check, no factory-enforced atomicity. The `initializer` modifier merely ensures the function runs exactly once; it does not restrict *who* can call it. [2](#0-1) 

`__Ownable_init()` sets `msg.sender` as the owner, so whoever calls `initialize()` first on the proxy becomes the permanent owner and controls both `token` and `registerMerkleRoot()`. [3](#0-2) 

All subsequent `claim()` calls transfer from `token` via `SafeERC20.safeTransfer`, which will call into whatever address the attacker supplied. [4](#0-3) 

---

### Impact Explanation

- The attacker becomes the permanent owner of the proxy.
- `token` is permanently set to a malicious ERC20 (immutable after `initializer` runs once).
- The legitimate deployer's `initialize()` call reverts — the contract cannot be re-initialized without redeployment.
- All `claim()` calls interact with the malicious token contract.
- Real tokens sent to the proxy address are permanently locked (no recovery function exists).
- The attacker also controls `registerMerkleRoot()`, enabling arbitrary merkle root injection. [5](#0-4) 

---

### Likelihood Explanation

The attack window exists between proxy deployment and the legitimate `initialize()` transaction being mined. On any public mempool chain, a bot can observe the pending `initialize()` call and front-run it with a higher gas price. No privileged access, leaked keys, or social engineering is required — only a standard transaction.

---

### Recommendation

Restrict `initialize()` so only a trusted deployer or factory can call it. The standard mitigations are:

1. **Atomic deployment**: Use a factory or deployment script that deploys the proxy and calls `initialize()` in the same transaction (e.g., via `ERC1967Proxy` constructor's `_data` parameter, which calls `initialize()` atomically).
2. **Caller check**: Add an `onlyOwner`-equivalent guard using a pre-set deployer address, or use OpenZeppelin's `Ownable2Step` pattern with a constructor-set trusted initializer.
3. **`initialOwner` parameter**: Pass the intended owner explicitly to `__Ownable_init(initialOwner)` (available in OZ v5) and verify `msg.sender` is a known deployer before proceeding.

---

### Proof of Concept

```solidity
// 1. Protocol deploys Airdrop proxy (uninitialized)
// 2. Attacker observes pending initialize(realToken, realSanctions) in mempool
// 3. Attacker front-runs:
airdrop.initialize(address(maliciousToken), realSanctions);
// Attacker is now owner; token = maliciousToken

// 4. Legitimate deployer's tx reverts: "Initializable: contract is already initialized"

// 5. Protocol funds the proxy with real tokens (unaware of the hijack):
realToken.transfer(address(airdrop), TOTAL_AIRDROP_AMOUNT);
// Real tokens are now locked — no sweep/recovery function exists

// 6. Any user calls claim():
airdrop.claim(claimProofs);
// → SafeERC20.safeTransfer(IERC20(maliciousToken), msg.sender, amount)
// → maliciousToken.transfer() executes attacker-controlled logic
```
