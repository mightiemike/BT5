### Title
Deprecated `verify_transaction_inclusion` (v1) Remains Publicly Callable and Accepts Internal Node Hashes as Valid Transaction IDs — (`contract/src/lib.rs`)

---

### Summary

`verify_transaction_inclusion` is marked `#[deprecated]` but remains a `pub fn` with no `#[private]` guard. In Rust, `#[deprecated]` is a compile-time lint only — it emits a warning to callers at compile time but imposes zero runtime restriction. Any unprivileged NEAR account can call the method directly. Because the function delegates entirely to `compute_root_from_merkle_proof` without a coinbase depth-binding check, an attacker can supply an internal Merkle tree node hash as `tx_id` together with a crafted sibling proof and receive `true`.

---

### Finding Description

`verify_transaction_inclusion` is declared `pub fn` and annotated only with `#[deprecated]` and `#[pause]`: [1](#0-0) 

`#[pause]` gates the call behind a contract-level pause flag but imposes no caller-identity restriction. When the contract is live (unpaused), any NEAR account can invoke the method.

The function body performs no coinbase-depth check; it only computes: [2](#0-1) 

`compute_root_from_merkle_proof` is a pure hash-chain walk that treats its first argument as an opaque 32-byte value: [3](#0-2) 

It does not verify that `transaction_hash` is a leaf. For a block whose Merkle tree has leaves `[T0, T1, T2, T3]`:

```
N01  = SHA256d(T0 ‖ T1)   ← internal node
N23  = SHA256d(T2 ‖ T3)   ← internal node
Root = SHA256d(N01 ‖ N23)
```

Calling `compute_root_from_merkle_proof(N01, 0, [N23])` computes `SHA256d(N01 ‖ N23) = Root`, which equals `header.block_header.merkle_root`, so the function returns `true` for `tx_id = N01` — an internal node, not a real transaction.

The contract's own doc-comment acknowledges this: [4](#0-3) 

but the acknowledgement does not remove the public entry point.

`verify_transaction_inclusion_v2` fixes this by first verifying a coinbase proof at depth 0 before delegating to v1: [5](#0-4) 

That guard is absent in v1.

---

### Impact Explanation

Any NEAR contract or off-chain consumer that calls `verify_transaction_inclusion` (v1) directly can be convinced that a fabricated "transaction" — whose `tx_id` is actually an internal Merkle node — is included in a real Bitcoin block. This breaks the core invariant of the light client: that a `true` return value certifies a leaf-level transaction. Downstream contracts that gate asset releases, bridge withdrawals, or cross-chain state transitions on a `true` result from v1 are directly exploitable.

---

### Likelihood Explanation

The attack requires only:
1. A block already accepted into the contract's `headers_pool` (public state).
2. The block's transaction list (public Bitcoin data).
3. A single NEAR function call — no privileged role, no staking, no relayer key.

All inputs are publicly derivable. The call is a view-style read (`&self`) so it costs only gas.

---

### Recommendation

Remove the public entry point for v1 entirely, or add `#[private]` to prevent external calls. The `#[deprecated]` annotation provides no runtime protection. Since `verify_transaction_inclusion_v2` already calls v1 internally via `#[allow(deprecated)]`, the internal path is preserved; only the external surface needs to be closed.

---

### Proof of Concept

Given a block in the pool with four transactions `[T0, T1, T2, T3]`:

```rust
// Attacker computes off-chain:
let n01 = double_sha256(&[T0.0, T1.0].concat());  // internal node
let n23 = double_sha256(&[T2.0, T3.0].concat());  // sibling

// Attacker calls on-chain:
verify_transaction_inclusion(ProofArgs {
    tx_id:             n01,          // internal node, NOT a real tx
    tx_block_blockhash: known_block_hash,
    tx_index:          0,
    merkle_proof:      vec![n23],    // one-element proof
    confirmations:     1,
})
// → returns true
```

`compute_root_from_merkle_proof(n01, 0, [n23])` = `SHA256d(n01 ‖ n23)` = `Root` = `header.merkle_root`, so the check at line 322 passes and the function returns `true`.

The same `tx_id = n01` submitted to `verify_transaction_inclusion_v2` with any coinbase proof will fail at line 358–365 because `n01` cannot be proven to be the coinbase (index 0) of the real tree.

#[must_use]
pub fn compute_root_from_merkle_proof(
    transaction_hash: H256,
    transaction_position: usize,
    merkle_proof: &Vec<H256>,
) -> H256 {
    let mut current_hash = transaction_hash;
    let mut current_position = transaction_position;

    for proof_hash in merkle_proof {
        if current_position % 2 == 0 {
            current_hash = compute_hash(&current_hash, proof_hash);
        } else {
            current_hash = compute_hash(proof_hash, &current_hash);
        }
        current_position /= 2;
    }

    current_hash
}
