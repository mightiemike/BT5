### Title
`get_tip_resource_bounds_hash` hash-domain collision: `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` and `L1Gas(l1_gas=X)` serialize identically in protobuf but produce different transaction hashes — (`crates/starknet_api/src/transaction_hash.rs`)

---

### Summary

`get_tip_resource_bounds_hash` hashes a different number of resource-bound felts depending on the `ValidResourceBounds` variant: `L1Gas` produces a 2-element preimage while `AllResources` produces a 3-element preimage. The protobuf deserializer for `Transaction` objects (`TryFrom<protobuf::ResourceBounds> for ValidResourceBounds`) silently collapses `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` into `L1Gas(l1_gas=X)` because both variants serialize to identical protobuf bytes. A V3 transaction submitted with `AllResources` and zero L2/data-gas bounds is accepted and hashed with 3 resource felts (H1). After P2P block-sync deserialization the same transaction fields are reconstructed as `L1Gas`, and any code that recomputes the hash produces H2 ≠ H1.

---

### Finding Description

**Step 1 — Hash-domain split in `get_tip_resource_bounds_hash`** [1](#0-0) 

For `ValidResourceBounds::L1Gas(_)` the function builds `[tip, L1_GAS_packed, L2_GAS_packed]` — two resource felts — and hashes them with Poseidon. For `ValidResourceBounds::AllResources(_)` it appends a third element `L1_DATA_GAS_packed`, producing a strictly different Poseidon output even when `l1_data_gas = 0`.

**Step 2 — Protobuf serialization is lossy** [2](#0-1) 

When serializing `ValidResourceBounds::L1Gas(l1_gas)` the encoder writes `l1_data_gas = Some(ResourceBounds::default())` (all-zero). When serializing `ValidResourceBounds::AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` the encoder also writes `l1_data_gas = Some(zero)`. The two variants produce **byte-for-byte identical** protobuf messages.

**Step 3 — Deserialization silently changes the variant** [3](#0-2) 

On the receiving side, `l1_data_gas.unwrap_or_default()` yields zero, and the guard `if l1_data_gas.is_zero() && l2_gas.is_zero()` fires, returning `ValidResourceBounds::L1Gas(l1_gas)`. The `AllResources` variant is permanently lost.

**Step 4 — The two hashes diverge**

`get_invoke_transaction_v3_hash` (and the declare/deploy-account equivalents) call `get_tip_resource_bounds_hash` with the stored `ValidResourceBounds`: [4](#0-3) 

- At submission time the gateway computes H1 from `AllResources` (3 felts).
- After P2P block-sync deserialization the same fields yield `L1Gas` (2 felts) and any recomputation produces H2 ≠ H1.

**Step 5 — The RPC/mempool path uses a stricter converter that does not have this bug** [5](#0-4) 

The `RpcTransaction` protobuf path requires `l1_data_gas` to be present and always returns `AllResourceBounds`, so the mempool and consensus P2P paths are unaffected. The vulnerable path is the **block-sync** `Transaction` protobuf path.

---

### Impact Explanation

Any node that syncs a block containing a V3 transaction with `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` and then recomputes the transaction hash for commitment verification will derive H2 ≠ H1. This causes:

1. **Wrong transaction hash served by RPC** (`starknet_getTransactionByHash`, `starknet_getTransactionReceipt`): the returned `resource_bounds` variant is `L1Gas` instead of `AllResources`, so any client that recomputes the hash from the returned fields gets a different value than the on-chain hash — an authoritative-looking wrong value.
2. **Potential block-commitment verification failure**: if the syncing node recomputes transaction hashes to verify the transaction commitment in the block header, the mismatch causes a valid block to be rejected, breaking liveness.

Matching impact: **High — RPC execution/tracing/simulation returns an authoritative-looking wrong value** and **High — mempool/gateway/RPC admission rejects valid transactions before sequencing**.

---

### Likelihood Explanation

A V3 transaction with `AllResources` where both `l2_gas` and `l1_data_gas` are zero is unusual but entirely valid. The gateway imposes no lower bound on these fields. An attacker or ordinary user who sets `l2_gas = 0` and `l1_data_gas = 0` (e.g., to minimize fee exposure on a network that does not yet charge for those resources) will trigger the collision. No special privilege is required; a single submitted transaction is sufficient.

---

### Recommendation

1. **Short term**: In `TryFrom<protobuf::ResourceBounds> for ValidResourceBounds` (`crates/apollo_protobuf/src/converters/transaction.rs`), preserve the `AllResources` variant whenever the protobuf message explicitly carries `l1_data_gas` (even if zero), and only fall back to `L1Gas` when `l1_data_gas` is `None` (absent from the wire). This restores round-trip fidelity.

2. **Long term**: Add a canonicalization invariant test: for any `ValidResourceBounds` value `v`, `deserialize(serialize(v))` must produce the same hash as `v`. The current test suite does not cover the `AllResources(l1_gas=X, l2_gas=0, l1_data_gas=0)` edge case.

pub fn get_tip_resource_bounds_hash(
    resource_bounds: &ValidResourceBounds,
    tip: &Tip,
) -> Result<Felt, StarknetApiError> {
    let l1_resource_bounds = resource_bounds.get_l1_bounds();
    let l2_resource_bounds = resource_bounds.get_l2_bounds();

    // L1 and L2 gas bounds always exist.
    // Old V3 txs always have L2 gas bounds of zero, but they exist.
    let mut resource_felts = vec![
        get_concat_resource(&l1_resource_bounds, L1_GAS)?,
        get_concat_resource(&l2_resource_bounds, L2_GAS)?,
    ];

    // For new V3 txs, need to also hash the data gas bounds.
    resource_felts.extend(match resource_bounds {
        ValidResourceBounds::L1Gas(_) => vec![],
        ValidResourceBounds::AllResources(all_resources) => {
            vec![get_concat_resource(&all_resources.l1_data_gas, L1_DATA_GAS)?]
        }
    });

    Ok(HashChain::new().chain(&tip.0.into()).chain_iter(resource_felts.iter()).get_poseidon_hash())
}
