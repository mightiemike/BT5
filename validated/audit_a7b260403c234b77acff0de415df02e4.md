### Title
Slow-Mode Deposit Tokens Permanently Locked When `clearinghouse.depositCollateral` Reverts After Pre-Transfer — (`core/contracts/Endpoint.sol`)

---

### Summary

`depositCollateralWithReferral` transfers user tokens to the `Clearinghouse` contract **before** the corresponding slow-mode transaction is executed. If the slow-mode transaction later fails, the tokens remain in the `Clearinghouse` with no credited balance and no recovery path, because the previously existing refund mechanism was explicitly removed.

---

### Finding Description

In `Endpoint.sol`, `depositCollateralWithReferral` performs two sequential operations:

1. **Immediately** transfers tokens from the caller to the `Clearinghouse` via `handleDepositTransfer`.
2. **Queues** a slow-mode transaction that will later call `clearinghouse.depositCollateral` to credit the user's internal balance. [1](#0-0) 

`handleDepositTransfer` in `EndpointStorage.sol` moves tokens directly to the `Clearinghouse`: [2](#0-1) 

The slow-mode transaction is executed later (after `SLOW_MODE_TX_DELAY`, hardcoded to 3 days) via `_executeSlowModeTransaction`, which wraps execution in a `try/catch`: [3](#0-2) 

The critical detail is the comment at line 226: **`// try return funds now removed`**. A refund path previously existed here but was deleted. If `processSlowModeTransaction` reverts (i.e., `clearinghouse.depositCollateral` fails), the `catch` block silently swallows the error and does nothing. The tokens already transferred to the `Clearinghouse` are never credited to the user and cannot be recovered. [4](#0-3) 

The slow-mode execution path in `processSlowModeTransactionImpl` for `DepositCollateral` calls `clearinghouse.depositCollateral` with no fallback: [5](#0-4) 

---

### Impact Explanation

User ERC20 tokens are permanently locked inside the `Clearinghouse` contract. The tokens are present in the contract's balance but are not attributed to any subaccount. There is no admin recovery function exposed on the `Clearinghouse` for this case (unlike `BaseWithdrawPool.removeLiquidity` which is scoped only to the `WithdrawPool`). The user cannot re-trigger the credit, and the protocol has no on-chain mechanism to refund the deposit. [6](#0-5) 

---

### Likelihood Explanation

The 3-day `SLOW_MODE_TX_DELAY` window creates a realistic gap between token transfer and credit execution. Conditions that can cause `clearinghouse.depositCollateral` to revert after a valid deposit include:

- A product being **delisted** between deposit and execution (`DelistProduct` is a supported slow-mode transaction type).
- Any clearinghouse-level invariant check that passes at deposit time but fails 3 days later. [7](#0-6) 

The entry path requires no special privileges — any user calling `depositCollateral` or `depositCollateralWithReferral` is exposed. [8](#0-7) 

---

### Recommendation

Restore a refund path in the `catch` block of `_executeSlowModeTransaction`. When a `DepositCollateral` slow-mode transaction fails, the `Clearinghouse` should return the tokens to the original depositor. Alternatively, defer the token transfer to occur only upon successful execution of the slow-mode transaction, keeping tokens in the `Endpoint` (or a dedicated escrow) until the credit is confirmed. [9](#0-8) 

---

### Proof of Concept

1. User calls `depositCollateral(subaccountName, productId, amount)` on `Endpoint.sol`.
2. `depositCollateralWithReferral` is invoked; `handleDepositTransfer` immediately moves `amount` tokens from the user to `address(clearinghouse)`.
3. A `SlowModeTx` is queued with `executableAt = block.timestamp + SLOW_MODE_TX_DELAY` (3 days).
4. Before the 3-day window expires, the owner submits a `DelistProduct` slow-mode transaction for `productId`, which is executed immediately by the sequencer.
5. After 3 days, anyone calls `executeSlowModeTransaction()` to process the user's deposit.
6. `processSlowModeTransaction` → `processSlowModeTransactionImpl` → `clearinghouse.depositCollateral` reverts because the product is delisted.
7. The `catch` block in `_executeSlowModeTransaction` runs; the comment `// try return funds now removed` confirms no refund occurs.
8. The user's tokens remain in the `Clearinghouse` with no credited balance and no on-chain recovery path. [1](#0-0) [2](#0-1) [10](#0-9)

    function _executeSlowModeTransaction(
        SlowModeConfig memory _slowModeConfig,
        bool fromSequencer
    ) internal {
        require(
            _slowModeConfig.txUpTo < _slowModeConfig.txCount,
            ERR_NO_SLOW_MODE_TXS_REMAINING
        );
        SlowModeTx memory txn = slowModeTxs[_slowModeConfig.txUpTo];
        delete slowModeTxs[_slowModeConfig.txUpTo++];

        require(
            fromSequencer || (txn.executableAt <= block.timestamp),
            ERR_SLOW_TX_TOO_RECENT
        );

        if (block.chainid == 31337) {
            // for testing purposes, we don't fail silently when the chainId is hardhat's default.
            this.processSlowModeTransaction(txn.sender, txn.tx);
        } else {
            uint256 gasRemaining = gasleft();
            // solhint-disable-next-line no-empty-blocks
            try this.processSlowModeTransaction(txn.sender, txn.tx) {} catch {
                // we need to differentiate between a revert and an out of gas
                // the issue is that in evm every inner call only 63/64 of the
                // remaining gas in the outer frame is forwarded. as a result
                // the amount of gas left for execution is (63/64)**len(stack)
                // and you can get an out of gas while spending an arbitrarily
                // low amount of gas in the final frame. we use a heuristic
                // here that isn't perfect but covers our cases.
                // having gasleft() <= gasRemaining / 2 buys us 44 nested calls
                // before we miss out of gas errors; 1/2 ~= (63/64)**44
                // this is good enough for our purposes

                if (gasleft() <= 250000 || gasleft() <= gasRemaining / 2) {
                    // solhint-disable-next-line no-inline-assembly
                    assembly {
                        invalid()
                    }
                }

                // try return funds now removed
            }
        }
