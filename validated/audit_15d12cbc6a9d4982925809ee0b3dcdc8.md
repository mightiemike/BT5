Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any non-allowlisted caller to bypass the deposit guard — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual caller/payer) and gates only on `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` supports an operator pattern where `msg.sender` pays but need not equal `owner`, any non-allowlisted address can bypass the deposit restriction by supplying an allowlisted address as `owner`. The pool admin's curation policy is rendered entirely inoperative.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument (`sender`) and the caller-supplied `owner` as the second argument to the extension hook:

```solidity
// MetricOmmPool.sol line 191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

Unlike `removeLiquidity`, which enforces `if (msg.sender != owner) revert NotPositionOwner()` at line 206, `addLiquidity` imposes no such constraint — the operator pattern is intentionally permitted.

`DepositAllowlistExtension.beforeAddLiquidity` receives both values but leaves `sender` unnamed (discarded) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol lines 32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The contract's own NatSpec states "Gates `addLiquidity` by **depositor** address" and the mapping key is named `depositor`, confirming the intent is to gate on the actual caller. The implementation checks the wrong actor.

`SwapAllowlistExtension.beforeSwap` correctly checks `sender` (the actual caller) and discards `recipient`, confirming the asymmetry is a defect in the deposit extension, not an intentional design difference.

Exploit path:
1. Pool is configured with `DepositAllowlistExtension`.
2. `alice` is allowlisted: `allowedDepositor[pool][alice] = true`.
3. `bob` is not allowlisted: `allowedDepositor[pool][bob] = false`.
4. `bob` calls `pool.addLiquidity(owner = alice, salt = 99, deltas, ...)`.
5. Extension receives `sender = bob` (discarded), `owner = alice`.
6. Check: `allowedDepositor[pool][alice] == true` → guard passes.
7. `bob` pays tokens via callback; `alice` receives LP shares at salt 99.

The same bypass is reachable through `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner=alice, ...)`, which only validates `owner != address(0)` before forwarding to the pool.

## Impact Explanation
The deposit allowlist is completely defeated. Any non-allowlisted address can deposit into a curated pool by naming any allowlisted address as `owner`. The pool admin's ability to restrict depositors is rendered inoperative. Additionally, a non-allowlisted actor can force-deposit into an allowlisted user's position at an arbitrary salt without that user's consent, creating unwanted LP positions. This constitutes broken core pool functionality and a direct admin-boundary bypass by an unprivileged caller.

## Likelihood Explanation
Exploitation requires no special privilege, no flash loan, and no complex setup. Any EOA or contract can call `pool.addLiquidity` directly with `owner` set to any allowlisted address. The `MetricOmmPoolLiquidityAdder` also exposes a public `addLiquidityExactShares` overload accepting an arbitrary `owner`, making the bypass reachable through the supported periphery path. The condition is trivially satisfiable whenever at least one address is allowlisted, which is the normal operating state of any pool using this extension.

## Recommendation
Change `beforeAddLiquidity` to check `sender` (the actual caller/payer) instead of `owner`, mirroring the correct actor binding already used in `SwapAllowlistExtension.beforeSwap`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

      owner  = alice,
      salt   = 99,
      deltas = ...,
      ...

  sender = bob   (unnamed, discarded)
