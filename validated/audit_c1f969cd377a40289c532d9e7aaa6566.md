### Title
Unchecked `approve()` Return Value Silently Fails Before Collateral Deposit — (`File: core/contracts/DirectDepositV1.sol` and `core/contracts/ContractOwner.sol`)

---

### Summary

The Nado protocol contains two production sites where ERC20 `approve()` return values are not checked before a downstream deposit operation that depends on the approval succeeding. The `ERC20Helper` library provides `safeTransfer` and `safeTransferFrom` wrappers that validate return values, but no equivalent `safeApprove` exists, and all `approve()` calls are made directly. For non-standard ERC20 tokens that return `false` instead of reverting on failure, the approval silently fails and the deposit proceeds with zero or stale allowance.

---

### Finding Description

**Site 1 — `DirectDepositV1.sol` line 92:**

`creditDeposit()` is `external` with no access control. Any caller can invoke it. Inside the loop, for each product token held by the DDA:

```solidity
token.approve(address(endpoint), balance);   // return value discarded
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
```

The return value of `token.approve()` is silently discarded. If the token returns `false` (non-reverting failure), the endpoint is called with zero allowance.

**Site 2 — `ContractOwner.sol` lines 530–531:**

`wrapVaultAsset()` is `external` with no access control modifier. It first pulls asset tokens out of the DDA into `ContractOwner` itself via `DirectDepositV1.withdraw()`, then:

```solidity
assetToken.approve(tokenAddr, 0);            // return value discarded
assetToken.approve(tokenAddr, assetBalance); // return value discarded
IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1);
```

Both approve return values are discarded. For USDT-style tokens that require resetting allowance to zero before increasing it, if the first `approve(tokenAddr, 0)` silently fails, the second `approve(tokenAddr, assetBalance)` will also silently fail (the token rejects non-zero→non-zero increases). The vault `deposit()` then proceeds with no valid allowance.

The `ERC20Helper` library deliberately uses low-level `call` + return-value checks for `transfer` and `transferFrom`, but provides no `safeApprove` equivalent, leaving all approval paths unguarded.

---

### Impact Explanation

- **`DirectDepositV1.creditDeposit()`**: For any listed spot token that returns `false` from `approve()`, the deposit call proceeds with zero allowance. The endpoint's internal `transferFrom` will revert (for standard endpoints), causing the entire transaction to revert. User funds sitting in the DDA cannot be credited to their subaccount — the deposit flow is permanently broken for that token until the allowance is set by other means.

- **`ContractOwner.wrapVaultAsset()`**: Assets are first withdrawn from the DDA into `ContractOwner`. If both approve calls silently fail and the vault's `deposit()` does not revert on insufficient allowance (non-standard vault), the asset tokens are stranded in `ContractOwner` with no recovery path accessible to the user. Even in the standard-vault case, the transaction reverts after assets have been moved, creating a fragile and unreliable wrapping flow.

---

### Likelihood Explanation

Medium. The Nado protocol is designed to support multiple spot tokens (iterated via `spotEngine.getProductIds()`). Any token added to the spot engine that implements non-reverting `approve()` (returning `false` on failure) triggers this path. USDT on certain chains and several DeFi wrapper tokens exhibit this behavior. Both entry points (`creditDeposit` and `wrapVaultAsset`) are permissionless — any unprivileged user can trigger them.

---

### Recommendation

Add a `safeApprove` function to `ERC20Helper` mirroring the existing `safeTransfer` pattern, and replace all direct `.approve()` calls with it:

```solidity
function safeApprove(IERC20Base self, address spender, uint256 amount) internal {
    (bool success, bytes memory data) = address(self).call(
        abi.encodeWithSelector(IERC20Base.approve.selector, spender, amount)
    );
    require(
        success && (data.length == 0 || abi.decode(data, (bool))),
        ERR_TRANSFER_FAILED
    );
}
```

Replace in `DirectDepositV1.sol` line 92 and `ContractOwner.sol` lines 530–531.

---

### Proof of Concept

**Trigger path for `DirectDepositV1.sol`:**

1. A non-standard ERC20 token (returning `false` from `approve()`) is listed as a spot product.
2. A user sends tokens to their DDA address.
3. Any caller invokes `creditDeposit()` on the DDA.
4. `token.approve(address(endpoint), balance)` returns `false`; the return value is discarded.
5. `endpoint.depositCollateralWithReferral(...)` is called with zero allowance → reverts.
6. User's balance remains in the DDA, uncredited.

**Trigger path for `ContractOwner.wrapVaultAsset()`:**

1. A vault-wrapped spot product is listed; its underlying asset is a USDT-style token.
2. User sends underlying asset tokens to their DDA.
3. Any caller invokes `wrapVaultAsset(subaccount, productId)`.
4. `DirectDepositV1.withdraw()` transfers assets from DDA → `ContractOwner`.
5. `assetToken.approve(tokenAddr, 0)` silently returns `false`.
6. `assetToken.approve(tokenAddr, assetBalance)` also silently returns `false` (USDT rejects non-zero→non-zero).
7. `IERC4626Base(tokenAddr).deposit(assetBalance, directDepositV1)` reverts → whole tx reverts.
8. Assets return to DDA, but the wrap flow is permanently broken for this token.
