### Title
Gateway-Only Stateless Validation Checks Bypassed via Consensus/P2P Path — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator` enforces several OS-level invariants (empty `paymaster_data`, empty `account_deployment_data`, DA-mode restrictions) only on the RPC/gateway ingestion path. Transactions arriving through the consensus proposal path (`convert_consensus_tx_to_internal_consensus_tx`) skip this validator entirely, allowing a malicious proposer to inject transactions with non-empty `paymaster_data` or `account_deployment_data` that the Starknet OS will then hard-assert to be zero, causing the entire block to fail OS execution and proof generation.

### Finding Description

The gateway's `add_tx_inner` calls `self.stateless_tx_validator.validate(&tx)` before any conversion: [1](#0-0) 

`StatelessTransactionValidator::validate` enforces two OS-level invariants with explicit comments stating the OS will reject non-empty values: [2](#0-1) 

The OS Cairo code confirms these are hard assertions, not soft checks: [3](#0-2) [4](#0-3) 

However, the consensus validation path in `validate_proposal.rs` calls `convert_consensus_tx_to_internal_consensus_tx` directly: [5](#0-4) 

`convert_consensus_tx_to_internal_consensus_tx` delegates to `convert_rpc_tx_to_internal`, which performs **no** stateless validation — it only computes the transaction hash and optionally verifies a proof: [6](#0-5) [7](#0-6) 

The resulting `InternalConsensusTransaction` is forwarded to the batcher and eventually executed by the blockifier. When the OS processes the block, `hash_tx_common_fields` asserts `paymaster_data_length = 0` and `nonce_data_availability_mode = 0` / `fee_data_availability_mode = 0`. A transaction with non-empty `paymaster_data` or non-L1 DA modes will cause the OS to abort, making the block unprovable.

The same bypass applies to the DA-mode checks: [8](#0-7) 

These checks (`validate_nonce_data_availability_mode`, `validate_fee_data_availability_mode`, `validate_empty_paymaster_data`, `validate_empty_account_deployment_data`) are all absent from the consensus conversion path.

### Impact Explanation

A malicious block proposer can craft a `ConsensusTransaction::RpcTransaction` with non-empty `paymaster_data` (or `account_deployment_data`, or `nonce_data_availability_mode != L1`). The validator node's `validate_proposal` handler will accept and forward it to the batcher. The batcher will execute it. When the OS attempts to prove the block, the hard `assert paymaster_data_length = 0` will fail, making the block unprovable and causing the sequencer to produce an invalid block. This matches the impact category: **Wrong state, receipt, or revert result from blockifier/syscall/execution logic for accepted input** and **Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

### Likelihood Explanation

Any consensus participant (validator node) that proposes a block controls the `ProposalPart::Transactions` stream. The check is entirely absent from the consensus path — no privilege escalation or external dependency is required beyond being a block proposer. The analog to the original report is exact: the check exists at the gateway level but the consensus path (like the `account_contract` calling itself) bypasses it entirely.

### Recommendation

Move the OS-invariant checks (`validate_empty_paymaster_data`, `validate_empty_account_deployment_data`, `validate_nonce_data_availability_mode`, `validate_fee_data_availability_mode`) out of `StatelessTransactionValidator` and into `convert_rpc_tx_to_internal` inside `TransactionConverter`, so they are enforced on **every** conversion path — both RPC/gateway and consensus. This mirrors the recommended fix in the original report: move the check to the shared, lower-level function that all callers must pass through.

### Proof of Concept

1. Craft a `ConsensusTransaction::RpcTransaction(RpcTransaction::Invoke(RpcInvokeTransactionV3 { paymaster_data: PaymasterData(vec![Felt::ONE]), ... }))`.
2. Inject it into a proposal stream sent to a validating node's `validate_proposal` handler.
3. `convert_consensus_tx_to_internal_consensus_tx` calls `convert_rpc_tx_to_internal` — no stateless validation is performed.
4. The transaction is forwarded to the batcher via `send_txs_for_proposal`.
5. The batcher executes it; the blockifier does not check `paymaster_data` emptiness.
6. When the OS runs `hash_tx_common_fields`, it executes `assert common_fields.paymaster_data_length = 0` — this assertion fails.
7. The block cannot be proved; the sequencer has committed an unprovable block. [9](#0-8) [10](#0-9) [11](#0-10)

    async fn convert_rpc_tx_to_internal(
        &self,
        tx: RpcTransaction,
    ) -> TransactionConverterResult<(InternalRpcTransaction, Option<(ProofFacts, Proof)>)> {
        let (tx_without_hash, proof_data) = match tx {
            RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
                let proof_data = if tx.proof_facts.is_empty() {
                    None
                } else {
                    Some((tx.proof_facts.clone(), tx.proof.clone()))
                };
                (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
            }
            RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
                let ClassHashes { class_hash, executable_class_hash_v2 } =
                // TODO(Dori): Make this async and spawn a task to compile and add it to the class manager.
                    self.class_manager_client.add_class(tx.contract_class).await?;
                // TODO(Aviv): Ensure that we do not want to
                // allow declare with compiled class hash v1.
                if tx.compiled_class_hash != executable_class_hash_v2 {
                    return Err(TransactionConverterError::ValidateCompiledClassHashError(
                        ValidateCompiledClassHashError::CompiledClassHashMismatch {
                            computed_class_hash: executable_class_hash_v2,
                            supplied_class_hash: tx.compiled_class_hash,
                        },
                    ));
                }
                (
                    InternalRpcTransactionWithoutTxHash::Declare(InternalRpcDeclareTransactionV3 {
                        sender_address: tx.sender_address,
                        compiled_class_hash: tx.compiled_class_hash,
                        signature: tx.signature,
                        nonce: tx.nonce,
                        class_hash,
                        resource_bounds: tx.resource_bounds,
                        tip: tx.tip,
                        paymaster_data: tx.paymaster_data,
                        account_deployment_data: tx.account_deployment_data,
                        nonce_data_availability_mode: tx.nonce_data_availability_mode,
                        fee_data_availability_mode: tx.fee_data_availability_mode,
                    }),
                    None,
                )
            }
            RpcTransaction::DeployAccount(RpcDeployAccountTransaction::V3(tx)) => {
                let contract_address = tx.calculate_contract_address()?;
                (
                    InternalRpcTransactionWithoutTxHash::DeployAccount(
                        InternalRpcDeployAccountTransaction {
                            tx: RpcDeployAccountTransaction::V3(tx),
                            contract_address,
                        },
                    ),
                    None,
                )
            }
        };
        let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
        Ok((InternalRpcTransaction { tx: tx_without_hash, tx_hash }, proof_data))
