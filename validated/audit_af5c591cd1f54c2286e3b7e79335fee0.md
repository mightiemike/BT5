### Title
`isValidDepositAmount` Does Not Consider Pending Slow-Mode Deposits, Forcing Users to Overpay on Subsequent Deposits — (File: `core/contracts/Endpoint.sol`)

---

### Summary

`Endpoint.depositCollateral` uses `subaccountIds[subaccount] == 0` to decide whether to enforce the stricter `MIN_FIRST_DEPOSIT_AMOUNT` ($5) threshold. However, `subaccountIds` is only written when the queued slow-mode transaction is *processed* — up to 3 days later. During that window every additional deposit by the same user is treated as a "first deposit" and must meet the $5 floor, even though the user has already committed funds to the protocol.

---

### Finding Description

`depositCollateral` calls `isValidDepositAmount` before queuing the slow-mode transaction:

```solidity
// Endpoint.sol  lines 90-101
function isValidDepositAmount(
    bytes32 subaccount,
    uint32 productId,
    uint128 amount
) internal returns (bool) {
    int256 minDepositAmount = MIN_DEPOSIT_AMOUNT;          // $0.10
    if (subaccount != X_ACCOUNT && (subaccountIds[subaccount] == 0)) {
        minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT;       // $5.00
    }
    return clearinghouse.checkMinDeposit(productId, amount, minDepositAmount);
}
``` [1](#0-0) 

`subaccountIds` is populated only inside `processSlowModeTransactionImpl`, which runs after the mandatory `SLOW_MODE_TX_DELAY` (3 days):

```solidity
// EndpointTx.sol  lines 209-216
if (txType == IEndpoint.TransactionType.DepositCollateral) {
    ...
    _recordSubaccount(txn.sender);   // ← only here
    clearinghouse.depositCollateral(txn);
}
``` [2](#0-1) 

`depositCollateralWithReferral` (the function that actually takes custody of funds and enqueues the slow-mode tx) never calls `_recordSubaccount`: [3](#0-2) 

The slow-mode delay constant confirms the 3-day exposure window:

```solidity
// Constants.sol  line 50
uint64 constant SLOW_MODE_TX_DELAY = 3 * 24 * 60 * 60; // 3 days
``` [4](#0-3) 

And the two thresholds that create the gap:

```solidity
int256 constant MIN_DEPOSIT_AMOUNT       = ONE / 10;  // $0.10
int256 constant MIN_FIRST_DEPOSIT_AMOUNT = 5 * ONE;   // $5.00
``` [5](#0-4) 

---

### Impact Explanation

A user who has already paid $5 into the protocol (funds transferred, slow-mode tx queued) and wants to top up with $0.20 is forced to deposit at least $5 again, because `subaccountIds[subaccount]` is still `0`. The excess $4.80 is locked in the protocol until the user pays a `withdrawFeeX18` to recover it. The user cannot avoid this: `depositCollateral` reverts with `ERR_DEPOSIT_TOO_SMALL` if the amount is below the enforced minimum. [6](#0-5) 

---

### Likelihood Explanation

The slow-mode delay is 3 days. Any user who makes a first deposit and then needs to top up collateral (e.g., to meet margin requirements, fund a trade, or correct an under-deposit) within that window is affected. This is a realistic, unprivileged, user-triggered path requiring no special access.

---

### Recommendation

Record the subaccount at deposit-queue time (inside `depositCollateralWithReferral`) rather than waiting for slow-mode execution, or introduce a separate `pendingSubaccounts` mapping that `isValidDepositAmount` also consults so that a queued-but-unprocessed first deposit is treated as satisfying the first-deposit threshold for subsequent calls.

---

### Proof of Concept

1. Alice calls `depositCollateral(name, productId, 5e18)` — $5 transferred to protocol, slow-mode tx queued, `subaccountIds[alice||name] == 0`.
2. Within the 3-day window Alice calls `depositCollateral(name, productId, 0.2e18)` — $0.20.
3. `isValidDepositAmount` evaluates `subaccountIds[alice||name] == 0` → `minDepositAmount = MIN_FIRST_DEPOSIT_AMOUNT = $5`.
4. `checkMinDeposit` returns `false` → transaction reverts with `ERR_DEPOSIT_TOO_SMALL`.
5. Alice must deposit at least $5 instead of $0.20, overpaying by $4.80. To recover the excess she must later call `WithdrawCollateral` and pay `withdrawFeeX18`.
