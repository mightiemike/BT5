### Title
Shard-order canonicalization missing in proto→Rust conversion causes signature verification failure for valid `SnapshotHostInfo` — (`chain/network/src/network_protocol/proto_conv/peer_message.rs`)

---

### Summary

`SnapshotHostInfo` is signed over a borsh-serialized `(sync_hash, epoch_height, shards)` tuple where `shards` is a `Vec<ShardId>`. The signing side always sorts `shards` before signing. The proto→Rust `TryFrom` conversion preserves proto insertion order without sorting. A malicious peer can relay a valid `SnapshotHostInfo` proto message with `shards` in a different order; the receiver recomputes a different hash, signature verification fails, and the legitimate snapshot host is rejected.

---

### Finding Description

**Signing side** — `peer_manager_actor.rs` sorts shards before calling `SnapshotHostInfo::new`: [1](#0-0) 

`SnapshotHostInfo::new` then hashes and signs the sorted order: [2](#0-1) 

`build_hash` is order-sensitive because borsh serializes a `Vec` element-by-element: [3](#0-2) 

**Deserialization side** — `TryFrom<&proto::SnapshotHostInfo>` copies the proto `repeated uint64 shards` field directly into `self.shards` with no sort: [4](#0-3) 

**Verification** recomputes the hash from `self.shards` in whatever order they were stored: [5](#0-4) 

**Attack path:**

1. Honest node A signs `SnapshotHostInfo` with `shards = [0, 1, 2]` (sorted).
2. Malicious peer M receives the proto message and re-sends it to node C with `shards: [2, 1, 0]` in the proto repeated field (proto repeated fields preserve insertion order — RFC 3.4).
3. Node C's `TryFrom` at line 134 reconstructs `SnapshotHostInfo { shards: [2, 1, 0], … }`.
4. `verify()` computes `CryptoHash::hash_borsh((sync_hash, epoch_height, [2, 1, 0]))`, which differs from the hash signed over `[0, 1, 2]`.
5. `InvalidSignature` is returned → node C rejects the valid host info and bans M.

The attacker is any network peer (no validator or operator privilege required). The attacker is banned after the attack, but can use multiple identities (Sybil) to repeat it against different nodes, or target nodes that have no other path to receive the legitimate message.

---

### Impact Explanation

A legitimate snapshot host (node A) is excluded from node C's `SnapshotHostsCache`. Node C will not select node A for state sync header or part requests. If the attacker can intercept or be the only relay for node A's advertisement to a subset of the network, those nodes will not use node A for state sync, degrading availability of state snapshots. [6](#0-5) 

---

### Likelihood Explanation

Any peer that can relay `SyncSnapshotHosts` messages (which is the normal gossip mechanism) can perform this attack. The attacker pays the cost of being banned once per target node per identity, but the damage (exclusion of the legitimate host) persists until the target node receives the correctly-ordered message from another relay.

---

### Recommendation

Sort `shards` immediately after deserialization in `TryFrom<&proto::SnapshotHostInfo>`:

```rust
// chain/network/src/network_protocol/proto_conv/peer_message.rs, line 134
shards: {
    let mut v: Vec<ShardId> = x.shards.clone().into_iter().map(Into::into).collect();
    v.sort();
    v
},
```

This makes the conversion canonical regardless of proto wire order, matching the invariant enforced on the signing side.

### Proof of Concept

```rust
// Roundtrip test demonstrating the divergence
let key = SecretKey::from_random(KeyType::ED25519);
let peer_id = PeerId::new(key.public_key());
// Sign with shards in sorted order [0, 1, 2]
let info = SnapshotHostInfo::new(
    peer_id, sync_hash, epoch_height,
    vec![ShardId::new(0), ShardId::new(1), ShardId::new(2)],
    &key,
);
assert!(info.verify().is_ok()); // passes

// Serialize to proto, then manually reorder shards
let mut proto_msg: proto::SnapshotHostInfo = (&info).into();
proto_msg.shards = vec![2, 1, 0]; // reversed order

// Deserialize — no sort applied
let reconstructed = SnapshotHostInfo::try_from(&proto_msg).unwrap();
// verify() recomputes hash over [2,1,0] ≠ hash signed over [0,1,2]
assert!(reconstructed.verify().is_err()); // FAILS with InvalidSignature
```

        &self,
        data: Vec<Arc<SnapshotHostInfo>>,
    ) -> (Vec<Arc<SnapshotHostInfo>>, Option<SnapshotHostInfoError>) {
        // Filter out any data which is invalid, outdated or which we already have.
        if data.iter().map(|d| d.peer_id.clone()).collect::<HashSet<_>>().len() != data.len() {
            return (vec![], Some(SnapshotHostInfoError::DuplicatePeerId));
        }
        let new_data = {
            let inner = self.0.lock();
            data.into_iter().filter(|d| !d.shards.is_empty() && inner.is_new(d)).collect_vec()
        };
        // Verify the signatures in parallel.
        // Verification will stop at the first encountered error.
        let (data, verification_result) = concurrency::rayon::run(move || {
            concurrency::rayon::try_map_result(new_data.into_iter().par_bridge(), |d| {
                match d.verify() {
                    Ok(()) => Ok(d),
                    Err(err) => Err(err),
                }
            })
        })
        .await;
        match verification_result {
            Ok(()) => (data, None),
            Err(err) => (data, Some(SnapshotHostInfoError::VerificationError(err))),
