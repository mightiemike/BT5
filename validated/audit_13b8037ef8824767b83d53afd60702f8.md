Looking at the operator dispatch logic in `src/chia_dialect.rs`, I can identify a concrete flag/operator wiring error that is a direct analog to the missing state check described in the report.

### Title
`ENABLE_SECP_OPS` Flag Not Checked in 4-Byte Opcode Dispatch Path, Allowing Secp Operations to Execute When Disabled - (File: `src/chia_dialect.rs`)

---

### Summary

`ChiaDialect::op()` in `src/chia_dialect.rs` has two separate dispatch paths for secp signature-verification operations. The 1-byte opcode path (opcodes 64 and 65) correctly gates execution behind the `ENABLE_SECP_OPS` flag. The 4-byte opcode path (opcodes `0x13d61f00` and `0x1c3a8f00`) dispatches directly to the same secp functions **without checking `ENABLE_SECP_OPS`**. This is a flag/operator wiring error: an attacker-controlled CLVM program can invoke secp operations via the 4-byte encoding and bypass the flag entirely, including in mempool mode where secp ops are supposed to be disabled.

---

### Finding Description

`ChiaDialect::op()` handles operator dispatch in two length-gated branches. For 4-byte operators, the code comments describe them as "unknown operators with assigned cost," but two specific 4-byte values are explicitly matched and routed to the live secp verification functions:

```rust
// src/chia_dialect.rs lines 175–182
let f = match opcode {
    0x13d61f00 => op_secp256k1_verify,
    0x1c3a8f00 => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
return f(allocator, argument_list, max_cost, flags);
```

No flag check precedes this dispatch. [1](#0-0) 

For the 1-byte opcode path, the same operations are correctly gated:

```rust
// src/chia_dialect.rs lines 248–249
64 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
65 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
``` [2](#0-1) 

Without `ENABLE_SECP_OPS`, opcodes 64 and 65 fall through to `unknown_operator`. In `NO_UNKNOWN_OPS` mode (mempool mode), that returns `Err(EvalErr::Unimplemented(o))` — a hard rejection. The 4-byte path never reaches `unknown_operator` for these two values; it short-circuits directly to the secp function and returns a successful `Reduction`.

`MEMPOOL_MODE` is defined as:

```rust
pub const MEMPOOL_MODE: ClvmFlags = ClvmFlags::NO_UNKNOWN_OPS
    .union(ClvmFlags::LIMIT_HEAP)
    .union(ClvmFlags::DISABLE_OP)
    .union(ClvmFlags::CANONICAL_INTS)
    .union(ClvmFlags::LIMIT_SOFTFORK);
``` [3](#0-2) 

`ENABLE_SECP_OPS` is absent from `MEMPOOL_MODE`. The invariant "secp ops are disabled in mempool mode" holds for 1-byte opcodes but is silently violated for 4-byte opcodes.

The `ENABLE_SECP_OPS` flag is defined as:

```rust
/// Enables secp opcodes 64 (secp256k1_verify) and 65 (secp256r1_verify).
const ENABLE_SECP_OPS = 0x0800;
``` [4](#0-3) 

The flag description names only the 1-byte opcodes, but the 4-byte path dispatches to the identical underlying functions (`op_secp256k1_verify`, `op_secp256r1_verify` in `src/secp_ops.rs`), making the omission a wiring error rather than an intentional design. [5](#0-4) 

---

### Impact Explanation

**Mempool/consensus divergence.** A CLVM program submitted to a node running in mempool mode (`MEMPOOL_MODE`, no `ENABLE_SECP_OPS`) that uses 4-byte secp opcodes will be **accepted and executed** — performing a real secp256k1 or secp256r1 signature verification — while the semantically identical program using 1-byte opcodes 64/65 is **rejected** with `Unimplemented`. This creates an inconsistency between what the mempool accepts and what the flag-gating policy intends. If `ENABLE_SECP_OPS` is also absent from a consensus-mode configuration (e.g., pre-hardfork nodes), the same divergence applies at the consensus layer: nodes that have not activated the secp hardfork would still execute secp operations submitted via the 4-byte encoding, while rejecting the 1-byte form. The corrupted result is a successful `Reduction(cost, nil)` from a secp verification that should have been rejected outright.

---

### Likelihood Explanation

The 4-byte opcode values `0x13d61f00` and `0x1c3a8f00` are deterministic and publicly derivable from the cost formula documented in the source comments. Any attacker who reads the source or the Chia protocol specification can construct a CLVM program that uses these encodings. The entry path is fully attacker-controlled CLVM bytes passed to `run_program` via the Python wheel API or any Chia node that calls `ChiaDialect::op()`. No privileged access, social engineering, or dependency compromise is required.

---

### Recommendation

Add an `ENABLE_SECP_OPS` guard to the 4-byte dispatch path, mirroring the guard already present for the 1-byte path:

```rust
let f = match opcode {
    0x13d61f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256k1_verify,
    0x1c3a8f00 if flags.contains(ClvmFlags::ENABLE_SECP_OPS) => op_secp256r1_verify,
    _ => {
        return unknown_operator(allocator, o, argument_list, flags, max_cost);
    }
};
```

This ensures that both opcode encodings of the secp operations are subject to the same flag-gating policy. Additionally, add test cases that assert rejection of 4-byte secp opcodes when `ENABLE_SECP_OPS` is absent, including in `MEMPOOL_MODE`.

---

### Proof of Concept

1. Construct a CLVM program whose operator atom is the 4-byte atom `[0x13, 0xd6, 0x1f, 0x00]` (secp256k1_verify via the 4-byte cost-formula encoding) with valid pubkey/msg/sig arguments.
2. Run it through `ChiaDialect::new(MEMPOOL_MODE)` (which does not include `ENABLE_SECP_OPS`).
3. Observe: the program executes successfully and returns `Reduction(1300000, nil)`.
4. Construct the same program using 1-byte opcode `[0x40]` (decimal 64, the `ENABLE_SECP_OPS`-gated alias).
5. Run it through the same `ChiaDialect::new(MEMPOOL_MODE)`.
6. Observe: the program is rejected with `EvalErr::Unimplemented` because `NO_UNKNOWN_OPS` is set and `ENABLE_SECP_OPS` is absent.

The two programs invoke the same underlying function (`op_secp256k1_verify`) but receive opposite outcomes solely because of the missing flag check in the 4-byte dispatch branch at `src/chia_dialect.rs` lines 175–182. [6](#0-5)
