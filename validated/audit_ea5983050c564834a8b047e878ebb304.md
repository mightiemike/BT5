### Title
`DirectDepositV1.creditDeposit()` Silently Strands WETH With No User-Accessible Recovery Path — (File: `core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1` wraps native ETH into WETH via `receive()` and is expected to sweep it to a bound subaccount via `creditDeposit()`. However, `creditDeposit()` delegates to `Endpoint.depositCollateralWithReferral()`, which **silently returns without transferring tokens** when the deposit amount is below the minimum threshold. The WETH remains stranded inside `DirectDepositV1`. The only recovery functions — `withdraw()` and `withdrawNative()` — are `onlyOwner`, leaving the depositing user with no path to reclaim their funds.

---

### Finding Description

`DirectDepositV1.receive()` wraps incoming ETH into the `wrappedNative` token, crediting the WETH balance to the `DirectDepositV1` contract itself: [1](#0-0) 

`creditDeposit()` then iterates over all spot products, approves the endpoint, and calls `depositCollateralWithReferral()` for any non-zero balance: [2](#0-1) 

Inside `Endpoint.depositCollateralWithReferral()`, if the deposit amount fails the minimum-deposit check, the function **returns early without reverting and without calling `handleDepositTransfer()`**: [3](#0-2) 

`handleDepositTransfer()` is the only place where tokens are pulled from `msg.sender` (i.e., `DirectDepositV1`) into the clearinghouse: [4](#0-3) 

Because `handleDepositTransfer()` is never reached, the WETH remains in `DirectDepositV1`. The only functions that can move it out are `withdraw()` and `withdrawNative()`, both of which are `onlyOwner`: [5](#0-4) 

There is no user-callable function to recover tokens from `DirectDepositV1` once they are stranded there. The contract provides a deposit entry path (`receive()` + `creditDeposit()`) but no symmetric user-accessible exit path — an exact structural analog to the reported `depositUnderlying()` / missing `withdrawUnderlying()` asymmetry.

---

### Impact Explanation

A user who sends ETH below the minimum deposit threshold to `DirectDepositV1` will have their WETH permanently stranded in the contract. The user has zero on-chain recourse: `withdraw()` and `withdrawNative()` require owner access. The owner (protocol deployer) can recover the funds, but the user cannot. This constitutes a direct, user-triggered asset lock with no self-service exit.

---

### Likelihood Explanation

`DirectDepositV1` is a public, permissionless contract — any address can send ETH to it. The minimum deposit threshold (`MIN_DEPOSIT_AMOUNT` / `MIN_FIRST_DEPOSIT_AMOUNT`) is a protocol-enforced constant checked at runtime. Any user who sends a small ETH amount (e.g., dust, a test transaction, or a partial top-up) will silently lose access to those funds. The silent-return behavior in `depositCollateralWithReferral()` is intentional for multi-asset batch sweeps but creates an unrecoverable state for the depositing user in this context. [6](#0-5) 

---

### Recommendation

Add a user-callable recovery function to `DirectDepositV1` that allows the original depositor (or the bound subaccount address) to reclaim tokens that were not successfully swept to the endpoint. Alternatively, `creditDeposit()` should revert (rather than silently continue) if `depositCollateralWithReferral()` does not consume the approved balance, so the user's ETH is never wrapped into a stranded state.

---

### Proof of Concept

1. User sends `0.0001 ETH` (below `MIN_DEPOSIT_AMOUNT`) to `DirectDepositV1`.
2. `receive()` executes: `wrappedNative.call{value: 0.0001 ether}("")` → WETH is minted to `DirectDepositV1`.
3. Anyone calls `creditDeposit()`.
4. Inside `creditDeposit()`: `balance = token.balanceOf(address(this))` returns the WETH amount; `token.approve(endpoint, balance)` is set; `endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1")` is called.
5. Inside `depositCollateralWithReferral()`: `isValidDepositAmount()` returns `false` → function returns early, `handleDepositTransfer()` is never called.
6. WETH remains in `DirectDepositV1`. Approval is set but unused.
7. User calls `withdraw(wethToken)` → reverts with `Ownable: caller is not the owner`.
8. User has no further recourse. Funds are locked until the owner intervenes. [1](#0-0) [2](#0-1) [3](#0-2) [7](#0-6)

```text
    function creditDeposit() external {
        uint32[] memory productIds = spotEngine.getProductIds();
        for (uint256 i = 0; i < productIds.length; i++) {
            uint32 productId = productIds[i];
            address tokenAddr = spotEngine.getToken(productId);
            require(tokenAddr != address(0), "Invalid productId.");
            IIERC20Base token = IIERC20Base(tokenAddr);
            uint256 balance = token.balanceOf(address(this));
            if (balance != 0) {
                token.approve(address(endpoint), balance);
                endpoint.depositCollateralWithReferral(
                    subaccount,
                    productId,
                    uint128(balance),
                    "-1"
                );
            }
        }
    }
```
