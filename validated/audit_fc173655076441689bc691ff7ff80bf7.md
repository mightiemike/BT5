### Title
Uninitialized `last_resharding` Field Bypasses Resharding Cooldown at Static-to-Dynamic Protocol Upgrade Boundary — (`chain/epoch-manager/src/lib.rs`)

### Summary

`EpochInfoV5.last_resharding` defaults to `None` for the first V5 epochs after `DynamicResharding` is activated. `can_reshard()` treats `None` as "no resharding has ever occurred, so resharding is allowed," unconditionally bypassing the `min_epochs_between_resharding` cooldown. If a static resharding took effect in the epoch immediately before dynamic resharding was enabled, the cooldown is silently skipped, allowing a dynamic split to be scheduled in the very first dynamic epoch — a back-to-back resharding that the protocol explicitly documents as unsafe.

### Finding Description

`EpochInfoV5` carries a `last_resharding: Option<EpochHeight>` field. The accessor `EpochInfo::last_resharding()` returns `None` for all pre-V5 variants and for any V5 epoch where no dynamic resharding has been recorded:

```rust
pub fn last_resharding(&self) -> Option<EpochHeight> {
    match self {
        Self::V1(_) | Self::V2(_) | Self::V3(_) | Self::V4(_) => None,
        Self::V5(v5) => v5.last_resharding,
    }
}
``` [1](#0-0) 

`can_reshard()` uses `is_none_or` on this value:

```rust
let can_reshard = next_epoch_info.last_resharding().is_none_or(|last_resharding| {
    next_epoch_info.epoch_height() - last_resharding >= min_epochs_between_resharding.get()
});
``` [2](#0-1) 

The code comment directly acknowledges the gap:

> "It is theoretically possible that a static resharding was scheduled right before enabling dynamic resharding, but we assume this didn't happen." [3](#0-2) 

The propagation path for `last_resharding` in `finalize_epoch` is:

```rust
let has_same_shard_layout = next_next_shard_layout == next_shard_layout;
let last_resharding = (!has_same_shard_layout)
    .then(|| next_epoch_info.epoch_height() + 1)
    .or_else(|| next_epoch_info.last_resharding());
``` [4](#0-3) 

When the static-to-dynamic transition occurs:
- Epoch N+1 uses a V4 `EpochInfo` (dynamic resharding not yet enabled), so `last_resharding()` returns `None`.
- Epoch N+2 is the first V5 epoch. If no layout change occurred at the N+1→N+3 boundary (i.e., no dynamic split was scheduled), `has_same_shard_layout = true`, so `last_resharding = next_epoch_info.last_resharding() = None`.
- Epoch N+3 inherits `last_resharding = None` by the same propagation.

When `can_reshard` is called during epoch N+2, it reads epoch N+3's `last_resharding = None`, and `is_none_or` returns `true` unconditionally — the cooldown is bypassed. [5](#0-4) 

### Impact Explanation

The documentation explicitly states:

> "`min_epochs_between_resharding` must be `> 0`: allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`." [6](#0-5) 

If the cooldown is bypassed and a dynamic resharding is scheduled in the first dynamic epoch (epoch N+2), the child shards created by the static resharding (which took effect in epoch N+1) would be subject to a second split in epoch N+4. The child shard's first chunk in epoch N+2 computes `proposed_split = None` (freshly initialized), while the parent's final chunk in epoch N+1 carried a non-`None` `proposed_split`. The chunk header validation in `validate_chunk_with_chunk_extra_and_receipts_root` would detect this mismatch and reject the chunk with `InvalidChunkHeaderShardSplit`, halting chain progress. [7](#0-6) 

### Likelihood Explanation

This requires a specific protocol upgrade sequence: a static resharding must take effect in the epoch immediately before `DynamicResharding` is activated. On mainnet/testnet, protocol upgrades are coordinated events. The scenario is not triggered by an unprivileged user but by the protocol upgrade boundary itself — the exact moment `DynamicResharding` is enabled after a prior static resharding. The code comment acknowledges this is "theoretically possible" and relies on an assumption ("we assume this didn't happen") rather than a code-level invariant. If the upgrade schedule does not respect this assumption, the chain can halt.

### Recommendation

Replace the `is_none_or` bypass with an explicit check that distinguishes "no resharding has ever happened" from "a static resharding happened before dynamic resharding was enabled." One approach: during the first V5 epoch creation in `finalize_epoch`, detect whether the static-to-dynamic transition involved a layout change and set `last_resharding` to the current epoch height if so, rather than propagating `None` from the pre-V5 `next_epoch_info`. Concretely:

```rust
let last_resharding = if !has_same_shard_layout {
    Some(next_epoch_info.epoch_height() + 1)
} else {
    // Preserve existing last_resharding, or default to current epoch height
    // if transitioning from pre-V5 (to prevent cooldown bypass).
    next_epoch_info.last_resharding()
        .or_else(|| {
            // If next_epoch_info is pre-V5, we cannot know if a static resharding
            // just occurred. Conservatively set last_resharding to the current epoch
            // height to enforce the cooldown.
            if next_epoch_info.shard_layout().is_none() {
                Some(next_epoch_info.epoch_height())
            } else {
                None
            }
        })
};
```

This ensures the cooldown is enforced across the static-to-dynamic protocol boundary.

### Proof of Concept

1. Configure a chain with a static resharding at protocol version P (e.g., layout changes from V1 to V2 at epoch N+1).
2. Configure `DynamicResharding` to activate at protocol version P+1 (epoch N+2).
3. Set `min_epochs_between_resharding = 1` and `force_split_shards` to a shard that exists in the new layout.
4. Observe that `can_reshard` returns `true` during epoch N+2 (the first dynamic epoch), because `next_epoch_info.last_resharding() = None` for epoch N+3.
5. A dynamic resharding is scheduled in epoch N+2, taking effect in epoch N+4 — only 2 epochs after the static resharding took effect in epoch N+2.
6. The child shard's first chunk in epoch N+2 has `proposed_split = None`; the parent's final chunk in epoch N+1 had a non-`None` `proposed_split` (if memory thresholds were met). The mismatch triggers `InvalidChunkHeaderShardSplit` during chunk validation, halting the chain. [8](#0-7) [9](#0-8) [10](#0-9)
