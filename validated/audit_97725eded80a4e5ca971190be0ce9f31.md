### Title
Integer Overflow in `op_unknown` Cost Multiplication Allows Near-Zero Cost for Expensive Unknown Ops — (`File: src/more_ops.rs`)

### Summary

`op_unknown` in `src/more_ops.rs` computes the final cost of an unknown opcode by multiplying a base cost by `(cost_multiplier + 1)`. The multiplication is performed on a `u64` without overflow protection. In Rust release mode (no `overflow-checks = true` in `[profile.release]`), this silently wraps modulo 2⁶⁴. The wrapped result can be ≤ `u32::MAX`, passing the only post-multiplication guard and returning a near-zero cost for what was an expensive operation. This is the direct analog of the external report's missing minimum floor: just as `newRatio` can approach zero without a lower bound, the charged cost of an unknown op can wrap to near-zero without any minimum floor check after the multiplication.

---

### Finding Description

In `src/more_ops.rs`, `op_unknown` computes cost in three steps: [1](#0-0) 

```rust
let cost_function = (op[op.len() - 1] & 0b11000000) >> 6;
let cost_multiplier: u64 = match u32_from_u8(&op[0..op.len() - 1]) {
    Some(v) => v as u64,
    None => { return Err(EvalErr::Invalid(o))?; }
};
```

A base `cost` is then computed (up to `max_cost`) and checked: [2](#0-1) 

```rust
assert!(cost > 0);
check_cost(cost, max_cost)?;      // ensures cost ≤ max_cost
cost *= cost_multiplier + 1;      // ← u64 overflow, no saturation/checked_mul
if cost > u32::MAX as u64 {
    Err(EvalErr::Invalid(o))?
} else {
    Ok(Reduction(cost as Cost, allocator.nil()))
}
```

`check_cost` enforces `cost ≤ max_cost` before the multiplication, but the multiplication itself is unchecked. `cost_multiplier` is parsed from up to 4 bytes of the opcode atom via `u32_from_u8`, giving a maximum value of `u32::MAX = 4 294 967 295`, so `cost_multiplier + 1` can be up to `4 294 967 296`. [3](#0-2) 

With Chia's block cost limit of ~11 billion (`max_cost ≤ 11 × 10⁹`):

```
max product = 11_000_000_000 × 4_278_190_080 ≈ 4.72 × 10¹⁹
u64::MAX                                      ≈ 1.84 × 10¹⁹
```

The product can exceed `u64::MAX` by a factor of ~2.55×. The `[profile.release]` section in `Cargo.toml` only sets `lto = "thin"` — no `overflow-checks = true` — so Rust release builds wrap silently on overflow. [4](#0-3) 

The wrapped value can be ≤ `u32::MAX`, passing the only post-multiplication guard and returning a tiny cost. The attacker controls both degrees of freedom: `cost_multiplier` (via opcode bytes) and the base `cost` (via argument sizes). Across the feasible parameter space (~11 × 10⁹ cost values × ~4.28 × 10⁹ multiplier values), approximately **10 billion** `(cost, cost_multiplier)` pairs produce a wrapped result ≤ `u32::MAX`, making the attack straightforwardly constructible.

The 5-byte opcode format (bytes 0–3 = multiplier, byte 4 = cost_function in bits 7–6) is the widest valid form. The `0xff 0xff` prefix guard at line 197 prevents the first two bytes from both being `0xff`, capping the effective maximum multiplier at `0xfeffffff = 4 278 190 079` for 5-byte opcodes, but this does not prevent overflow. [5](#0-4)

### Impact Explanation

An attacker submits a CLVM program containing a crafted unknown opcode with:
- Opcode bytes chosen to set `cost_multiplier` to a specific value
- Arguments chosen to drive the base `cost` to a specific value

The two values are selected so that `cost × (cost_multiplier + 1)` wraps to a value ≤ `u32::MAX` (e.g., 1). The VM charges the wrapped cost instead of the true cost, allowing the attacker to execute arbitrarily expensive unknown-op computations while consuming near-zero of the block's cost budget. This breaks the block cost limit invariant — the fundamental resource-accounting mechanism of the Chia VM — in the same way that `newRatio → 0` broke the DeFi protocol's collateral invariant in the referenced report.

---

### Likelihood Explanation

- The attacker fully controls the opcode atom bytes (setting `cost_multiplier`) and the argument list (setting base `cost`).
- No special privileges, network conditions, or timing are required.
- The attack is reachable via any CLVM program submitted to a node running in consensus mode (unknown ops allowed by default; `NO_UNKNOWN_OPS` is only set in `MEMPOOL_MODE`). [6](#0-5) 

- The attacker has ~10 billion valid `(cost, cost_multiplier)` pairs to choose from, making offline precomputation trivial.

---

### Recommendation

Replace the unchecked multiplication with a checked or saturating variant, and reject the opcode if the result overflows or exceeds `max_cost`:

```rust
// Instead of:
cost *= cost_multiplier + 1;

// Use:
cost = cost
    .checked_mul(cost_multiplier + 1)
    .ok_or(EvalErr::Invalid(o))?;
check_cost(cost, max_cost)?;
```

Alternatively, enable `overflow-checks = true` in `[profile.release]` in `Cargo.toml` to convert all silent wrapping overflows into panics across the entire codebase, though the explicit checked multiplication is the preferred targeted fix.

### Proof of Concept

Construct a 5-byte opcode atom `[0xfe, M2, M3, M4, CF]` where bytes 0–3 encode `cost_multiplier` and byte 4 encodes `cost_function` in bits 7–6. Provide arguments sized to drive the base cost to value `C`. Choose `C` and `cost_multiplier + 1 = M` such that `C × M mod 2⁶⁴ = 1`.

Concrete example sketch (pseudocode):
```python
# Find C, M such that C * M ≡ 1 (mod 2^64), C ≤ max_cost, M ≤ 4278190080
# M must be odd for a modular inverse to exist mod 2^64
M = 3  # cost_multiplier = 2, opcode last-prefix bytes = [0x00, 0x00, 0x02]
# M^{-1} mod 2^64 = 6148914691236517206 — too large for max_cost alone,
# but target T = M (= 3) gives C = 1 without overflow (trivial case).
# For overflow: target T = 3, C = (3 + 2^64) / 3 requires M | (3 + 2^64).
# Enumerate: for each odd M in [3, 4278190079], compute C = (T + k*2^64)/M
# for k=1,2 and check C ≤ max_cost. ~10^10 solutions exist.
```

The returned `Reduction(1, nil)` charges 1 unit of cost for an operation whose true cost was `max_cost` (up to 11 billion), effectively bypassing the block cost limit. [2](#0-1)

```rust
fn u32_from_u8_impl(buf: &[u8], signed: bool) -> Option<u32> {
    if buf.is_empty() {
        return Some(0);
    }

    // too many bytes for u32
    if buf.len() > 4 {
        return None;
    }

    let sign_extend = (buf[0] & 0x80) != 0;
    let mut ret: u32 = if signed && sign_extend { 0xffffffff } else { 0 };
    for b in buf {
        ret <<= 8;
        ret |= *b as u32;
    }
    Some(ret)
}

pub fn u32_from_u8(buf: &[u8]) -> Option<u32> {
    u32_from_u8_impl(buf, false)
}
```
