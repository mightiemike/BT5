### Title
Excess Deposit Refund Sent to Intermediate Caller Instead of Originating User — (`contract/src/lib.rs`)

### Summary

`submit_blocks` refunds excess NEAR deposit to `env::predecessor_account_id()`. When the function is invoked through an intermediate contract (e.g., a multicall proxy or a wrapper contract that holds a bypass role), the intermediate contract becomes the predecessor and receives the refund instead of the originating user.

### Finding Description

`submit_blocks` is a `#[payable]` function that accepts a NEAR deposit to cover storage costs. After computing the actual storage consumed, it refunds the surplus to `env::predecessor_account_id()`: [1](#0-0) 

```rust
let refund = amount.saturating_sub(required_deposit);
if refund > NearToken::from_near(0) {
    Promise::new(env::predecessor_account_id())
        .transfer(refund)
        .into()
```

In NEAR, `env::predecessor_account_id()` is the **direct caller** of the current frame — the exact equivalent of Solidity's `msg.sender`. When a user calls `submit_blocks` directly, this is correct. However, if the call is routed through an intermediate contract — for example, a multicall aggregator, a wrapper contract, or any contract that holds `Role::UnrestrictedSubmitBlocks` or `Role::DAO` (the bypass roles) — the intermediate contract becomes the predecessor: [2](#0-1) 

The `#[trusted_relayer]` guard checks `predecessor_account_id()` for authorization. A contract that is itself a registered trusted relayer or holds a bypass role can legitimately call `submit_blocks`. When it does so on behalf of a user (forwarding the user's attached deposit), the refund is transferred to the intermediate contract, not to the user who funded the call.

### Impact Explanation

**Impact: Medium**

The originating user permanently loses the excess NEAR deposit. The refund is transferred to the intermediate contract's balance. If the intermediate contract does not implement a withdrawal mechanism for unexpected NEAR receipts, the funds are effectively locked or captured. Unlike the original M-03 (which also misdirected a Uniswap LP position), the impact here is limited to the monetary loss of the excess deposit — there is no secondary asset (like an LP token) at risk.

### Likelihood Explanation

**Likelihood: Low**

The `#[trusted_relayer]` guard restricts callers to registered trusted relayers or accounts with `Role::DAO` / `Role::UnrestrictedSubmitBlocks`. A plain EOA-equivalent NEAR account calling directly is unaffected. The scenario requires a contract-based relayer or a multicall contract holding a bypass role to forward calls on behalf of users with attached deposits. This is a realistic deployment pattern (e.g., a relayer service implemented as a NEAR contract, or a DAO-controlled proxy), but not the default configuration.

### Recommendation

Add an explicit `refund_to: AccountId` parameter to `submit_blocks`, allowing the caller to specify the account that should receive the excess deposit refund. If omitted, default to `env::predecessor_account_id()` for backward compatibility. This mirrors the recommendation in M-03: decouple the authorization identity from the refund recipient identity.

```rust
pub fn submit_blocks(
    &mut self,
    #[serializer(borsh)] headers: Vec<BlockHeader>,
    refund_to: Option<AccountId>,  // <-- add this
) -> PromiseOrValue<()> {
    // ...
    let recipient = refund_to.unwrap_or_else(env::predecessor_account_id);
    Promise::new(recipient).transfer(refund).into()
}
```

### Proof of Concept

1. Deploy a NEAR contract `proxy.near` that holds `Role::UnrestrictedSubmitBlocks` on the BTC light client.
2. A user calls `proxy.near::forward_submit_blocks(headers)` attaching 1 NEAR (intentionally overpaying).
3. `proxy.near` calls `btc_light_client.submit_blocks(headers)` forwarding the 1 NEAR deposit.
4. Inside `submit_blocks`, `env::predecessor_account_id()` returns `proxy.near`.
5. Suppose actual storage cost is 0.3 NEAR; the 0.7 NEAR refund is transferred to `proxy.near`, not to the user.
6. The user's 0.7 NEAR is captured by the proxy contract. [3](#0-2)

```rust
    #[payable]
    #[pause]
    #[trusted_relayer]
    pub fn submit_blocks(
        &mut self,
        #[serializer(borsh)] headers: Vec<BlockHeader>,
    ) -> PromiseOrValue<()> {
        let amount = env::attached_deposit();
        let initial_storage = env::storage_usage();
        let num_of_headers = headers.len().try_into().unwrap();

        for header in headers {
            self.submit_block_header(header, self.skip_pow_verification);
        }

        self.run_mainchain_gc(num_of_headers);
        let diff_storage_usage = env::storage_usage().saturating_sub(initial_storage);
        let required_deposit = env::storage_byte_cost().saturating_mul(diff_storage_usage.into());

        require!(
            amount >= required_deposit,
            format!("Required deposit {}", required_deposit)
        );

        let refund = amount.saturating_sub(required_deposit);
        if refund > NearToken::from_near(0) {
            Promise::new(env::predecessor_account_id())
                .transfer(refund)
                .into()
        } else {
            PromiseOrValue::Value(())
        }
    }
```
