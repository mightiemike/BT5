### Title
`EndpointTx` Inherits `OwnableUpgradeable` But Never Initializes or Applies It — (File: `core/contracts/EndpointTx.sol`)

---

### Summary

`EndpointTx` inherits from OpenZeppelin's `OwnableUpgradeable` but never calls `__Ownable_init()`, never applies the `onlyOwner` modifier to any function, and never exposes any ownership-gated control surface. The inheritance is dead code that creates a false sense of access-control coverage over the contract's public entry points.

---

### Finding Description

`EndpointTx` is declared as:

```solidity
contract EndpointTx is EIP712Upgradeable, OwnableUpgradeable, EndpointStorage {
``` [1](#0-0) 

Despite inheriting `OwnableUpgradeable`, the contract:

1. **Never calls `__Ownable_init()`** — there is no `initialize` function in `EndpointTx`, so the `_owner` slot is never set.
2. **Never applies `onlyOwner`** to any function — all three public entry points (`processTransactionImpl`, `submitSlowModeTransactionImpl`, `processSlowModeTransactionImpl`) are declared `public` with no access-control modifier. [2](#0-1) [3](#0-2) [4](#0-3) 

The only place `owner()` is referenced in `EndpointTx` is inside `submitSlowModeTransactionImpl` to gate certain admin-only slow-mode transaction types:

```solidity
require(sender == owner());
``` [5](#0-4) 

Because `__Ownable_init()` is never called on the `EndpointTx` implementation contract itself, `owner()` returns `address(0)` when `EndpointTx` is called directly (not via `delegatecall`). This means the `require(sender == owner())` guard evaluates to `require(sender == address(0))`, which is trivially satisfiable by crafting a call where `msg.sender` resolves to `address(0)` — or, more practically, it means the guard is meaningless in the direct-call context.

When `EndpointTx` is invoked via `delegatecall` from `Endpoint`, the `owner()` slot is read from `Endpoint`'s storage (correctly initialized), so the guard works there. But the `OwnableUpgradeable` inheritance on `EndpointTx` itself contributes nothing — it is never initialized and never used as a modifier — mirroring exactly the `Pausable` pattern in the reference report.

---

### Impact Explanation

**Low.** The three public functions on `EndpointTx` have no `onlyOwner` guard. Direct calls to the implementation contract operate on uninitialized storage (zero addresses for `clearinghouse`, `spotEngine`, etc.) and will revert in practice. However, the `require(sender == owner())` check inside `submitSlowModeTransactionImpl` silently degrades to `require(sender == address(0))` on the implementation contract, which is a broken invariant even if not immediately exploitable. The `OwnableUpgradeable` inheritance provides zero actual protection.

---

### Likelihood Explanation

**High.** It is certain that `OwnableUpgradeable` is never initialized and never enforced in `EndpointTx`. Any reader of the contract signature would assume ownership-based access control exists; it does not.

---

### Recommendation

Either:
- Remove `OwnableUpgradeable` from `EndpointTx`'s inheritance chain if ownership control is not intended at the implementation level, or
- Add a constructor with `_disableInitializers()` (as done in other contracts such as `BaseProxyManager`) and apply `onlyOwner` to the public functions that should be restricted, or
- Replace the `require(sender == owner())` guard in `submitSlowModeTransactionImpl` with an explicit check against the `Endpoint` contract's owner read from storage, making the intent explicit rather than relying on an uninitialized inherited mechanism. [6](#0-5) 

---

### Proof of Concept

1. Deploy `EndpointTx` as a standalone contract (as it is deployed in production as an implementation contract).
2. Call `EndpointTx.submitSlowModeTransactionImpl` with a transaction type that hits the `require(sender == owner())` branch (e.g., `WithdrawInsurance`, `DelistProduct`, `DumpFees`, etc.).
3. Observe that `owner()` returns `address(0)` because `__Ownable_init()` was never called.
4. The guard `require(sender == owner())` becomes `require(sender == address(0))`, which is a broken invariant — the `OwnableUpgradeable` inheritance provides no protection on the implementation contract itself. [7](#0-6)
