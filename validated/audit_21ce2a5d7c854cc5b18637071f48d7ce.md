### Title
Deprecated `verify_transaction_inclusion` Remains Publicly Callable and Enables Merkle Second-Preimage Proof Forgery — (File: `contract/src/lib.rs`)

---

### Summary

The deprecated `verify_transaction_inclusion` (v1) function is still a live, unguarded public NEAR entry point callable by any account. It performs no tree-depth validation, allowing an attacker to supply an internal Merkle node hash as `tx_id` and receive `true` for a Bitcoin transaction that does not exist. Any downstream NEAR contract that consumes this result to gate fund releases or privileged state transitions can be deceived with a forged inclusion proof.

---

### Finding Description

`verify_transaction_inclusion` at `contract/src/lib.rs:288` is annotated `#[deprecated]` but carries no access-control guard (`#[private]`, role check, or `#[trusted_relayer]`). It is callable by any unprivileged NEAR account.

The function's verification logic is:

```rust
merkle_tools::compute_root_from_merkle_proof(
    args.tx_id,
    usize::try_from(args.tx_index).unwrap(),
    &args.merkle_proof,
) == header.block_header.merkle_root
```

It hashes `tx_id` up the proof path and compares the result to the stored `merkle_root`. It never validates that `tx_id` is a leaf (a real transaction hash) rather than an internal node. Because Bitcoin's Merkle tree is built by hashing pairs of child hashes, any internal node hash is itself a valid pre-image of a higher node. An attacker can therefore supply an internal node hash and a shortened proof path that reconstructs the same root, and the function returns `true`.
