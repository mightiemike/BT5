Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of End User, Allowing Any User to Bypass the Swap Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap()` gates swaps by checking the `sender` argument, which is the `msg.sender` of `pool.swap()`. When users route through `MetricOmmSimpleRouter`, `sender` equals the router's address. A pool admin who allowlists the router to enable legitimate router-mediated swaps inadvertently opens the pool to every user, because the extension cannot distinguish which end user is behind the router call.

## Finding Description

`SwapAllowlistExtension.beforeSwap()` performs its identity check as:

```solidity
// SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool (confirmed by `BaseMetricExtension.onlyPool` modifier at L19-24 of `BaseMetricExtension.sol`), and `sender` is the value the pool forwarded — which is the `msg.sender` of `pool.swap()`.

In `MetricOmmPool.swap()`, the pool passes its own `msg.sender` as `sender` to `_beforeSwap()`:

```solidity
// MetricOmmPool.sol L230-240
_beforeSwap(
    msg.sender,   // ← whoever called pool.swap()
    recipient,
    ...
);
```

`ExtensionCalling._beforeSwap()` then forwards this `sender` value directly to the extension:

```solidity
// ExtensionCalling.sol L160-176
_callExtensionsInOrder(
    BEFORE_SWAP_ORDER,
    abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
);
```

When a user calls `router.exactInputSingle()`, the router calls `pool.swap()` with itself as `msg.sender`:

```solidity
// MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
    );
```

So the extension receives `sender = address(router)`, not the end user's address. The allowlist lookup becomes `allowedSwapper[pool][router]`.

**Bypass path:**
1. Pool admin deploys a pool with `SwapAllowlistExtension` and allowlists `user1`, `user2` (trusted traders).
2. To let `user1`/`user2` use the router, the admin must also allowlist `address(router)`.
3. Once the router is allowlisted, any address — including `user3` who is not on the allowlist — can call `router.exactInputSingle()` and the extension will pass, because it only sees `sender = router`.

There is no existing guard that prevents this: the router does not forward the original `msg.sender` as the `sender` argument to `pool.swap()`, and the extension has no mechanism to recover the original caller.

## Impact Explanation

A curated pool's swap allowlist is completely bypassed for any user who routes through `MetricOmmSimpleRouter`. Unauthorized traders can execute swaps on pools intended to be restricted to specific counterparties. This breaks the core curation invariant and exposes LP funds to trades from actors the pool admin explicitly excluded. The impact is direct loss of LP value if the allowlist was protecting against adversarial or uninformed order flow. This qualifies as a broken core pool functionality causing loss of funds.

## Likelihood Explanation

Any pool that (a) uses `SwapAllowlistExtension` and (b) wants to support router-mediated swaps for its allowlisted users is vulnerable. The pool admin must allowlist the router to enable router usage, and doing so silently removes all per-user enforcement. This is a routine configuration that any operator of a curated pool would reach. The attacker requires no special privileges — any EOA can call `router.exactInputSingle()`.

## Recommendation

The extension should receive and check the original end-user address, not the immediate caller of `pool.swap()`. Options include:
1. Pass the end user's address through `extensionData` and verify it in the extension (with the router signing or forwarding it).
2. Add a dedicated `swapper` field to the `beforeSwap` hook arguments that the pool populates from a trusted source (e.g., transient storage set by the router before calling the pool).
3. Alternatively, document clearly that allowlisting the router grants access to all users, and provide a separate router that forwards the original `msg.sender` as a verified parameter.

## Proof of Concept

```
// Setup
Pool admin:
  allowedSwapper[pool][user1] = true
  allowedSwapper[pool][router] = true   // needed for user1 to use router

// Attack
Attacker (user3, not allowlisted):
  router.exactInputSingle({
      pool: pool,
      tokenIn: token0,
      ...
  })
  // MetricOmmSimpleRouter calls pool.swap() with msg.sender = router
  // MetricOmmPool passes msg.sender (router) as sender to _beforeSwap
  // SwapAllowlistExtension checks allowedSwapper[pool][router] → true
  // swap executes despite user3 not being allowlisted
```

Foundry test: deploy a pool with `SwapAllowlistExtension`, allowlist only `user1` and `address(router)`, then call `router.exactInputSingle()` from `user3` and assert the swap succeeds (no `NotAllowedToSwap` revert).
