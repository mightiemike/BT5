### Title
USDT Allowance Permanently Stuck Blocks All Future `creditDeposit()` Calls — (`core/contracts/DirectDepositV1.sol`)

---

### Summary

`DirectDepositV1.creditDeposit()` uses a raw `approve()` call before invoking `endpoint.depositCollateralWithReferral()`. However, `depositCollateralWithReferral` contains a silent early return when the deposit amount is below the minimum threshold — meaning the allowance is set but never consumed. For USDT (and any token that prohibits non-zero-to-non-zero approvals), this leaves a residual non-zero allowance that permanently causes all subsequent `creditDeposit()` calls to revert.

---

### Finding Description

`DirectDepositV1.creditDeposit()` iterates over all spot product IDs and, for each token with a non-zero balance, calls:

```solidity
token.approve(address(endpoint), balance);
endpoint.depositCollateralWithReferral(subaccount, productId, uint128(balance), "-1");
``` [1](#0-0) 

The critical issue is in `Endpoint.depositCollateralWithReferral()`. When the deposit amount fails the minimum deposit check, the function **silently returns without calling `handleDepositTransfer`** — i.e., without consuming any of the allowance:

```solidity
if (!isValidDepositAmount(subaccount, productId, amount)) {
    // we cannot revert here, otherwise direct deposit could be blocked when there are
    // multiple assets awaiting credit but one of them is below the minimum deposit amount.
    // we can just skip the deposit and continue with the next asset.
    return;
}
``` [2](#0-1) 

The `handleDepositTransfer` (which pulls tokens via `transferFrom`) is only reached on line 144, after the early-return guard: [3](#0-2) 

After this silent return, the `DirectDepositV1` contract holds a non-zero allowance for the endpoint. USDT's `approve` implementation reverts when called with a non-zero current allowance. Therefore, the next call to `creditDeposit()` for that token will revert at the `token.approve(...)` line, permanently bricking the contract's deposit functionality for that token.

By contrast, `ContractOwner.wrapVaultAsset()` correctly resets the allowance to zero before setting a new value: [4](#0-3) 

`DirectDepositV1.creditDeposit()` has no such reset.

---

### Impact Explanation

Once triggered, all future calls to `creditDeposit()` for the affected USDT product ID will revert. This permanently blocks the `DirectDepositV1` contract from depositing USDT collateral on behalf of the associated subaccount. Funds sent to the DDA contract in USDT will be stuck — they cannot be credited to the subaccount via the normal `creditDeposit()` path. The only recovery is `withdrawFromDirectDepositV1` (owner-only), which does not fix the allowance state.

---

### Likelihood Explanation

The trigger is realistic and low-cost:

1. An attacker (or even an honest user) sends a dust amount of USDT to the `DirectDepositV1` contract — an amount below `MIN_DEPOSIT_AMOUNT` or `MIN_FIRST_DEPOSIT_AMOUNT`.
2. Anyone calls `creditDeposit()` (no access control on this function).
3. `token.approve(endpoint, dustBalance)` succeeds (allowance was 0).
4. `depositCollateralWithReferral` silently returns without consuming the allowance.
5. The USDT allowance for the endpoint is now non-zero.
6. All subsequent `creditDeposit()` calls revert at `token.approve(endpoint, newBalance)` because USDT forbids non-zero-to-non-zero approvals.

`creditDeposit()` is `external` with no access control: [5](#0-4) 

`ContractOwner.creditDepositV1()` is also `external` with no access control, making this callable by any unprivileged user: [6](#0-5) 

---

### Recommendation

Replace the raw `approve` in `DirectDepositV1.creditDeposit()` with a pattern that first resets the allowance to zero before setting the new value, mirroring what `ContractOwner.wrapVaultAsset()` already does:

```solidity
token.approve(address(endpoint), 0);
token.approve(address(endpoint), balance);
```

Or, preferably, use OpenZeppelin's `SafeERC20.forceApprove()`, which handles this atomically and is compatible with USDT and similar tokens.

---

### Proof of Concept

1. Deploy `DirectDepositV1` for a subaccount that uses USDT as a collateral token.
2. Transfer 1 wei of USDT to the `DirectDepositV1` address (below `MIN_FIRST_DEPOSIT_AMOUNT`).
3. Call `creditDeposit()`. Observe: `token.approve(endpoint, 1)` succeeds; `depositCollateralWithReferral` silently returns (amount too small); USDT allowance for endpoint = 1 (non-zero).
4. Transfer 1,000 USDT to the `DirectDepositV1` address (a legitimate deposit amount).
5. Call `creditDeposit()` again. Observe: `token.approve(endpoint, 1000e6)` **reverts** — USDT rejects non-zero-to-non-zero approval.
6. The DDA is now permanently unable to credit USDT deposits for this subaccount.
