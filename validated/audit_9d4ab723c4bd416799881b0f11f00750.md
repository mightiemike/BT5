Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks the immediate caller (`sender = router`) instead of the originating user, enabling full allowlist bypass via `MetricOmmSimpleRouter` — (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument against `allowedSwapper[pool][sender]`. When a swap is routed through `MetricOmmSimpleRouter`, `sender` is the router's address because `MetricOmmPool.swap` passes its own `msg.sender` (the router) into the extension hook. Any pool admin who allowlists the router to support router-mediated swaps simultaneously grants every user on the network the ability to bypass the per-user gate, since the hook never inspects the originating EOA.

## Finding Description

**Root cause — `MetricOmmPool.swap` passes `msg.sender` as `sender`:**

`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`: [1](#0-0) 

`ExtensionCalling._beforeSwap` forwards that value unchanged as the first argument to the extension: [2](#0-1) 

**`SwapAllowlistExtension.beforeSwap` checks only `sender`:** [3](#0-2) 

**`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap()` directly**, making the router the `msg.sender` of that call — so `sender` arriving at the extension is the router address, not the end user: [4](#0-3) 

**Exploit path:**
1. Pool `P` has `SwapAllowlistExtension` with `allowAllSwappers[P] = false`.
2. Pool admin calls `setAllowedToSwap(P, router, true)` to permit router-mediated swaps for allowlisted users.
3. Unpermissioned user `bob` calls `router.exactInputSingle({pool: P, ...})`.
4. Router calls `P.swap(bob, ...)` — router is `msg.sender`.
5. Pool calls `_beforeSwap(router, bob, ...)`.
6. Extension checks `allowedSwapper[P][router] == true` → passes.
7. Bob's swap executes despite never being individually allowlisted.

**Existing guards are insufficient:** The only check in `beforeSwap` is `allowedSwapper[msg.sender][sender]`. There is no secondary check on `recipient` or any decoding of `extensionData` to recover the originating user.

**Asymmetry with `DepositAllowlistExtension`:** The deposit hook correctly ignores `sender` and checks `owner` (the LP position owner explicitly supplied by the caller), so it is not affected by the same router-mediation issue: [5](#0-4) 

## Impact Explanation

A pool configured with `SwapAllowlistExtension` and `allowAllSwappers = false` is intended to restrict trading to a curated set of counterparties (KYC-gated venue, private market-making pool, MEV-exclusion pool). Once the router is allowlisted — the only operational path to support router-mediated swaps — every user on the network can trade against the pool at oracle-derived prices. LP principal leaves the pool in exchange for input tokens with no recourse. This is a direct, irrecoverable loss of LP funds matching the **Allowlist path** impact gate: swap allowlist checks must cover the exact actor intended and cannot be bypassed through the router.

## Likelihood Explanation

The trigger condition is that the pool admin allowlists the router. This is the natural and necessary operational step for any curated pool that wants to support the standard periphery router for its allowlisted users — there is no other mechanism. The admin cannot selectively allow specific users *through* the router; the only available granularity is the router address itself. The bypass is therefore reachable on any curated pool that supports the standard periphery router, which is the expected production configuration.

## Recommendation

The hook must verify the originating user, not the immediate caller. The cleanest fix is extension-data attestation: require the router to encode `abi.encode(msg.sender)` into the swap's `extensionData` for each hop, and have `SwapAllowlistExtension.beforeSwap` decode and verify that address instead of `sender`. The router already threads per-hop `extensionData` through to the pool via `params.extensionData`: [4](#0-3) 

This preserves the router's role as a trusted intermediary while restoring per-user gating. A fallback approach — checking `recipient` as a proxy for the end user — is unreliable for multi-hop or contract recipients and should not be used as the primary fix.

## Proof of Concept

```
Setup:
  pool P has SwapAllowlistExtension E with allowAllSwappers[P] = false
  pool admin calls E.setAllowedToSwap(P, router, true)   // to enable router swaps
  pool admin calls E.setAllowedToSwap(P, alice, true)    // intended: only alice may swap

Attack (by bob, not allowlisted):
  bob calls router.exactInputSingle({
      pool: P,
      zeroForOne: true,
      amountIn: X,
      recipient: bob,
      ...
  })
  → router calls P.swap(bob, true, X, ...)   // msg.sender = router
  → pool calls _beforeSwap(router, bob, ...)
  → extension checks allowedSwapper[P][router] == true  ✓  (passes)
  → bob's swap executes; LP funds transferred to bob

Result: bob, who was never allowlisted, swaps successfully in a pool
        restricted to alice only.
```

Foundry test plan: deploy `SwapAllowlistExtension`, configure a pool with it, set `allowedSwapper[pool][router] = true` and `allowedSwapper[pool][alice] = true` but not `bob`, then call `router.exactInputSingle` from `bob`'s address and assert the swap succeeds (no `NotAllowedToSwap` revert).
