### Title
Unregistered Subaccount Can Invoke `WithdrawCollateral` via Slow Mode Path — (`File: core/contracts/EndpointTx.sol`)

### Summary
`processSlowModeTransactionImpl` handles `WithdrawCollateral` without calling `requireSubaccount`, unlike every other user-accessible slow-mode operation (`LinkSigner`, `ClaimBuilderFee`) in the same function. An unregistered subaccount — one that has never deposited and therefore has `subaccountIds[subaccount] == 0` — can submit and execute a `WithdrawCollateral` slow-mode transaction, bypassing the deposit-as-registration prerequisite that the rest of the protocol enforces.

---

### Finding Description

`requireSubaccount` is the protocol's registration gate: [1](#0-0) 

It enforces that a subaccount must have previously deposited (`subaccountIds[subaccount] != 0`) before it can perform privileged operations.

In `processSlowModeTransactionImpl`, every user-accessible operation that mutates subaccount state calls this gate — except `WithdrawCollateral`:

**`LinkSigner` (protected):** [2](#0-1) 

**`ClaimBuilderFee` (protected):** [3](#0-2) 

**`WithdrawCollateral` (unprotected):** [4](#0-3) 

Only `validateSender` is called — which merely confirms `address(bytes20(txn.sender)) == msg.sender`, not that the subaccount is registered. The call then proceeds directly to `clearinghouse.withdrawCollateral` with no registration check.

The fast-path equivalent (`processTransactionImpl`) is fully protected because it routes through `validateSignedTx`, which always calls `requireSubaccount`: [5](#0-4) 

The slow-mode submission gate (`submitSlowModeTransactionImpl`) does not add this check either — it only charges a fee and queues the transaction: [6](#0-5) 

---

### Impact Explanation

An unregistered subaccount (never deposited, `subaccountIds == 0`) can:

1. Call `submitSlowModeTransaction` with a `WithdrawCollateral` payload targeting their own address-derived subaccount (passes `validateSender`).
2. After the 3-day delay, call `executeSlowModeTransaction`.
3. `clearinghouse.withdrawCollateral` is invoked with an unregistered subaccount, bypassing the invariant that only deposited accounts interact with the clearinghouse withdrawal logic.

The clearinghouse is designed to receive calls from the endpoint only after the endpoint has enforced access control. If `clearinghouse.withdrawCollateral` does not independently re-validate registration (a reasonable assumption given the layered design), it may process the call against a zero-balance account. Depending on the clearinghouse's balance arithmetic, this can result in a corrupted (negative) balance entry for the unregistered subaccount — an accounting state that the protocol never intends to create for accounts that have never deposited.

On non-hardhat chains, the try/catch in `_executeSlowModeTransaction` silently swallows reverts, meaning even a clearinghouse-level revert does not surface as an error; the slow-mode slot is consumed regardless. [7](#0-6) 

---

### Likelihood Explanation

The entry path is fully permissionless. Any EOA can:
- Derive a subaccount name (`bytes12`) to form an unregistered `bytes32` subaccount.
- Call `submitSlowModeTransaction` with a `WithdrawCollateral` payload (pays the slow-mode fee in QUOTE token).
- Wait 3 days and call `executeSlowModeTransaction`.

No privileged role, sequencer cooperation, or leaked key is required. The only cost is the slow-mode fee.

---

### Recommendation

Add `requireSubaccount(txn.sender)` to the `WithdrawCollateral` branch of `processSlowModeTransactionImpl`, immediately after `validateSender`, consistent with how `LinkSigner` and `ClaimBuilderFee` are handled in the same function:

```solidity
} else if (txType == IEndpoint.TransactionType.WithdrawCollateral) {
    IEndpoint.WithdrawCollateral memory txn = abi.decode(
        transaction[1:],
        (IEndpoint.WithdrawCollateral)
    );
    validateSender(txn.sender, sender);
    requireSubaccount(txn.sender); // ADD THIS
    clearinghouse.withdrawCollateral(...);
```

---

### Proof of Concept

1. Alice has never deposited. Her subaccount `bytes32(abi.encodePacked(alice, bytes12("default")))` has `subaccountIds[subaccount] == 0`.
2. Alice calls `Endpoint.submitSlowModeTransaction(abi.encodePacked(uint8(TransactionType.WithdrawCollateral), abi.encode(WithdrawCollateral({sender: aliceSubaccount, productId: 0, amount: 1e18}))))`.
3. `submitSlowModeTransactionImpl` charges the slow-mode fee and queues the transaction.
4. After 3 days, anyone calls `executeSlowModeTransaction`.
5. `processSlowModeTransactionImpl` decodes the transaction, calls `validateSender` (passes, since `address(bytes20(aliceSubaccount)) == alice == sender`), and calls `clearinghouse.withdrawCollateral(aliceSubaccount, 0, 1e18, address(0), nSubmissions)` — with no `requireSubaccount` check having been performed.
6. The clearinghouse processes a withdrawal for an account that was never registered, violating the protocol invariant that only deposited subaccounts interact with withdrawal logic. [4](#0-3) [1](#0-0)
