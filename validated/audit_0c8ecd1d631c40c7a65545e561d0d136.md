### Title
SwapAllowlistExtension gates the router address instead of the end user, allowing any unprivileged caller to bypass the per-user swap allowlist - (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is `msg.sender` of `MetricOmmPool.swap`. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the **router contract**, not the end user. If the pool admin allowlists the router (the only way to let any user reach the pool via the router), the allowlist becomes a binary open/closed gate for the entire router, and every unprivileged user can bypass the per-user curation by routing through it.

---

### Finding Description

`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value verbatim to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever the pool received as its own `msg.sender`: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` (and every other swap entry point) calls `pool.swap(...)` directly with no forwarding of the original `msg.sender`: [4](#0-3) 

So the pool sees `msg.sender = router`, and the extension checks `allowedSwapper[pool][router]` — not `allowedSwapper[pool][user]`.

This creates an irreconcilable dilemma for any pool admin who wants to run a curated pool while still supporting the router:

| Admin choice | Effect |
|---|---|
| Do **not** allowlist the router | Allowlisted users cannot use the router at all |
| **Allowlist the router** | Every unprivileged user can bypass the per-user allowlist via the router |

The `DepositAllowlistExtension` avoids this problem because it gates on the `owner` parameter (explicitly passed by the caller and representing the actual depositor), not on `sender`: [5](#0-4) 

The swap extension has no equivalent mechanism to recover the true end-user identity.

---

### Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, institutional traders, or protocol-owned bots) can be fully bypassed by any unprivileged user who calls `MetricOmmSimpleRouter.exactInputSingle` / `exactInput` / `exactOutputSingle` / `exactOutput`. Once the router is allowlisted, the curation boundary collapses entirely: unauthorized users can execute swaps, extract LP value at oracle-anchored prices, and front-run or sandwich allowlisted traders — all actions the pool admin explicitly intended to prevent.

---

### Likelihood Explanation

The trigger is a standard, publicly documented periphery path. Any user who reads the router interface can route through it. The only precondition is that the pool admin has allowlisted the router — a natural and expected action for any pool that wants its curated users to benefit from multi-hop routing or slippage protection. No privileged access, no special token behavior, and no admin collusion is required.

---

### Recommendation

Pass the original end-user identity through the swap path so the extension can gate on it. Two concrete options:

1. **Add a `payer` / `originator` field to the swap call or extension payload.** The router stores the original `msg.sender` in transient storage already (for callback settlement); expose it as an additional argument to `beforeSwap` so extensions can check it.

2. **Key the allowlist on the `recipient` instead of `sender`.** For most curated pools the recipient is the economically relevant party. This is a partial fix only — it does not cover cases where the recipient differs from the intended gated actor.

The cleanest fix is option 1: thread the original caller through the extension interface so `SwapAllowlistExtension` can check `allowedSwapper[pool][originator]` rather than `allowedSwapper[pool][router]`.

---

### Proof of Concept

```
Setup
─────
1. Pool admin deploys a pool with SwapAllowlistExtension as beforeSwap hook.
2. Admin allowlists Alice:  setAllowedToSwap(pool, alice, true)
3. Admin allowlists the router: setAllowedToSwap(pool, router, true)
   (required so Alice can use the router)

Attack
──────
4. Bob (not allowlisted) calls:
     MetricOmmSimpleRouter.exactInputSingle({
       pool: pool,
       recipient: bob,
       ...
     })
5. Router calls pool.swap(...) — pool sees msg.sender = router.
6. Extension checks allowedSwapper[pool][router] → true → swap proceeds.
7. Bob receives tokens from the curated pool with no allowlist check on his address.

Expected: revert NotAllowedToSwap()
Actual:   swap succeeds
```

The root cause is in `SwapAllowlistExtension.beforeSwap` at line 37, where `sender` resolves to the router address rather than the end user whenever the router intermediates the call. [6](#0-5)
