### Title
`backfillDelegatorAddress` Unconditionally Overwrites New-Derivation `DelegatorAddress`, Permanently Locking Staked Funds — (`x/tieredrewards/migrations/v2/migrate.go`)

---

### Summary

`backfillDelegatorAddress` iterates every position in state and unconditionally replaces `DelegatorAddress` with the legacy module-address derivation. It contains no guard to skip positions whose `DelegatorAddress` was already set by the new `DerivePositionDelegatorAddress` scheme introduced in the same v8 binary. Any position created between v8 binary deployment and the upgrade-height block has a correctly-derived new address that gets silently clobbered, permanently orphaning the delegation and locked tokens.

---

### Finding Description

**Root cause — no skip guard in `backfillDelegatorAddress`:**

```go
// x/tieredrewards/migrations/v2/migrate.go lines 57-65
for _, kv := range kvs {
    pos := kv.Value
    pos.DelegatorAddress = LegacyDelegatorAddress(pos.Id)   // ← unconditional
    ...
    if err := positions.Set(ctx, pos.Id, pos); err != nil {
        return err
    }
}
``` [1](#0-0) 

`LegacyDelegatorAddress` produces `authtypes.NewModuleAddress("tieredrewards/position/<id>")`. [2](#0-1) 

**New derivation used by the v8 binary for every new position:**

`createPositionDelegatorAccount` (called by both `LockTier` and `CommitDelegationToTier`) uses `DerivePositionDelegatorAddress(headerHash, owner, id, nonce)` — a completely different address space. [3](#0-2) [4](#0-3) 

`LockTier` calls `createPositionDelegatorAccount` before the upgrade handler fires: [5](#0-4) 

**The reachable window:**

In every Cosmos SDK upgrade, validators deploy the new binary before the upgrade height. During that window the chain processes transactions with the v8 msg-server, which calls `createPositionDelegatorAccount` → `DerivePositionDelegatorAddress`. The upgrade handler fires at the target height and calls `RunMigrations`: [6](#0-5) 

`RunMigrations` triggers `Migrate1to2` → `Migrate` → `backfillDelegatorAddress`, which overwrites every position — including those just created with the new derivation. [7](#0-6) 

---

### Impact Explanation

After the overwrite, `pos.DelegatorAddress` points to the legacy module address. The actual staking delegation and locked tokens reside at the new derivation address. Every subsequent operation that reads `pos.DelegatorAddress` operates on the wrong account:

- **`TierUndelegate`** calls `ms.undelegate(ctx, delAddr, valAddr, pos.Delegation.Shares)` with the legacy address, which holds no delegation → transaction reverts, user can never undelegate. [8](#0-7) 

- **`WithdrawFromTier`** calls `ms.bankKeeper.SpendableCoins(ctx, delAddr)` on the legacy address → returns zero, the principal is never returned to the owner. [9](#0-8) 

- **`TierRedelegate`** similarly uses the wrong `delAddr` → fails. [10](#0-9) 

The actual delegation and tokens at the new derivation address are permanently orphaned — no key controls that address and no code path sweeps it.

---

### Likelihood Explanation

The deployment-to-upgrade-height window is standard practice in every Cosmos SDK chain upgrade. It typically spans hours to days. Any user who submits a `MsgLockTier` or `MsgCommitDelegationToTier` during that window is affected. No special privileges or attacker cooperation are required; a normal user creating a position in good faith triggers the bug.

---

### Recommendation

Add a skip guard in `backfillDelegatorAddress` to leave positions whose `DelegatorAddress` is already non-empty:

```go
for _, kv := range kvs {
    pos := kv.Value
    if pos.DelegatorAddress != "" {
        continue   // already set by new derivation; do not overwrite
    }
    pos.DelegatorAddress = LegacyDelegatorAddress(pos.Id)
    ...
}
``` [1](#0-0) 

This preserves the intent of the backfill (filling in empty v1 addresses) without corrupting positions that were already created with the hardened derivation.

---

### Proof of Concept

1. Governance passes the v8 upgrade plan at height H.
2. Validators deploy the v8 binary; the chain continues producing blocks.
3. At height H-N (N > 0), a user broadcasts `MsgLockTier`. The v8 msg-server calls `createPositionDelegatorAccount` → `DerivePositionDelegatorAddress(headerHash, owner, id, 0)` → stores `pos.DelegatorAddress = addr_new`. Funds are locked and delegated at `addr_new`.
4. At height H, the upgrade handler fires → `RunMigrations` → `Migrate1to2` → `backfillDelegatorAddress` iterates all positions including the one just created and sets `pos.DelegatorAddress = LegacyDelegatorAddress(id)` = `addr_legacy ≠ addr_new`.
5. User calls `MsgTierUndelegate` → keeper reads `pos.DelegatorAddress` = `addr_legacy` → calls `undelegate(addr_legacy, val, shares)` → staking module finds no delegation for `addr_legacy` → tx fails.
6. User calls `MsgWithdrawFromTier` → `SpendableCoins(addr_legacy)` = 0 → owner receives nothing; principal remains at `addr_new` forever.

func (k Keeper) createPositionDelegatorAccount(ctx context.Context, owner sdk.AccAddress, id uint64) (sdk.AccAddress, error) {
	sdkCtx := sdk.UnwrapSDKContext(ctx)
	headerHash := sdkCtx.HeaderHash()
	for nonce := uint64(0); nonce < MaxPositionAddressDerivationAttempts; nonce++ {
		addr := types.DerivePositionDelegatorAddress(headerHash, owner, id, nonce)
		if k.accountKeeper.GetAccount(ctx, addr) != nil {
			k.logger(ctx).Warn("position delegator address already exists", "address", addr.String())
			continue
		}
		acc := k.accountKeeper.NewAccountWithAddress(ctx, addr)
		k.accountKeeper.SetAccount(ctx, acc)
		return addr, nil
	}
	return nil, errorsmod.Wrapf(types.ErrPositionAddressDerivation, "could not create a free position delegator account")
}
