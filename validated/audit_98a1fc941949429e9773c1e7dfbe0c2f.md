### Title
`get_sortable_validator_online_ratio` Ignores `endorsement_cutoff_threshold`, Causing Kickout-Exemption Sorting to Use a Different Online-Ratio Domain Than Reward Calculation - (`chain/epoch-manager/src/validator_stats.rs`)

---

### Summary

At every epoch boundary, `finalize_epoch` computes validator rewards using an endorsement cutoff (`endorsement_cutoff_threshold = Some(chunk_validator_only_kickout_threshold)`), which binarizes each validator's endorsement contribution to 0 or 1. However, the kickout-exemption sorting step — which decides which validators are shielded from being kicked out — calls `get_sortable_validator_online_ratio`, which hardcodes `endorsement_cutoff_threshold = None` and therefore uses the raw endorsement fraction. The two paths compute a structurally different online ratio for the same validator, causing the wrong validators to be exempted from kickout.

---

### Finding Description

In `compute_validators_to_reward_and_kickout`, validators are sorted by online ratio to determine which ones are exempt from kickout (the `validator_max_kickout_stake_perc` guard):

```rust
// chain/epoch-manager/src/lib.rs:500-506
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
sorted_validators.sort_by(validator_comparator);
```

`get_sortable_validator_online_ratio` always passes `None` for the cutoff:

```rust
// chain/epoch-manager/src/validator_stats.rs:110-111
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
```

But `finalize_epoch` computes rewards with the cutoff applied:

```rust
// chain/epoch-manager/src/lib.rs:898-903
let online_thresholds = ValidatorOnlineThresholds {
    online_min_threshold: epoch_config.online_min_threshold,
    online_max_threshold: epoch_config.online_max_threshold,
    endorsement_cutoff_threshold: Some(
        epoch_config.chunk_validator_only_kickout_threshold,
    ),
};
```

`get_endorsement_ratio` behaves differently under the two calls:

```rust
// chain/epoch-manager/src/validator_stats.rs:124-134
fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    ...
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }  // binary: 0 or 1
    } else {
        (stats.produced, stats.expected)  // raw fraction
    };
```

**Concrete divergence:** Consider two chunk-validator-only validators, A and B, with `chunk_validator_only_kickout_threshold = 70`:

| Validator | Endorsement ratio | Sorting ratio (None) | Reward ratio (Some(70)) |
|-----------|------------------|----------------------|------------------------|
| A         | 69/100 = 0.69    | 0.69 (high)          | 0 (below cutoff)       |
| B         | 71/100 = 0.71    | 0.71 (slightly higher)| 1.0 (above cutoff)    |

But consider A with 69% and B with 71% where both are near the boundary. Now consider A at 69% and C at 50%:

| Validator | Endorsement ratio | Sorting ratio (None) | Reward ratio (Some(70)) |
|-----------|------------------|----------------------|------------------------|
| A         | 69/100 = 0.69    | 0.69                 | 0 (below cutoff)       |
| C         | 71/100 = 0.71    | 0.71                 | 1.0 (above cutoff)     |

Under the cutoff-based formula (which governs rewards and should govern exemption), C has a much higher online ratio (1.0 vs 0.0). Under the raw formula used for sorting, C only appears marginally better (0.71 vs 0.69). If `validator_max_kickout_stake_perc` allows only one exemption, the sorting may exempt A (raw 0.69) over a validator with raw 0.50 but cutoff-adjusted 1.0, while the reward formula would rank them in the opposite order.

The exact divergent value is the endorsement contribution: `produced/expected` (raw) vs. `0` or `1` (binary cutoff). This is a protocol-level invariant break: the same "online ratio" concept is computed with two different formulas in the same epoch finalization path.

---

### Impact Explanation

**High.** The kickout-exemption guard (`validator_max_kickout_stake_perc`) is a safety mechanism to prevent too many validators from being kicked out simultaneously. When the sorting uses a different online-ratio formula than the reward calculation, the wrong validators are protected. A validator with endorsement ratio just below the cutoff (who should have online ratio 0 under the cutoff formula) can appear to have a high online ratio in the sorting and be exempted, while a validator with endorsement ratio just above the cutoff (who should have online ratio 1.0 under the cutoff formula) may be kicked out. This corrupts the validator set selection at every epoch boundary where the cutoff is active and validators cluster near the threshold.

---

### Likelihood Explanation

**Medium.** The divergence is always present whenever `chunk_validator_only_kickout_threshold > 0` (which it is on mainnet: 70 in `85.json`) and validators have endorsement ratios near the threshold. The condition is triggered automatically at every epoch boundary — no external actor needs to do anything. The effect is most pronounced when the `validator_max_kickout_stake_perc` guard is active (i.e., when many validators are below threshold simultaneously, which can happen during network instability).

---

### Recommendation

`get_sortable_validator_online_ratio` must accept and forward the same `endorsement_cutoff_threshold` that `finalize_epoch` uses for reward calculation. The sorting for kickout exemption must use the same online-ratio domain as the reward calculation:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,  // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

And the call site in `compute_validators_to_reward_and_kickout` must pass `Some(config.chunk_validator_only_kickout_threshold)` to match the reward calculation domain.

### Proof of Concept

1. Epoch config has `chunk_validator_only_kickout_threshold = 70`, `validator_max_kickout_stake_perc = 30`.
2. Three chunk-validator-only validators: A (endorsement 69/100), B (endorsement 71/100), C (endorsement 50/100). All have equal stake.
3. All three are below the kickout threshold (70%), so all three would be kicked out. But `validator_max_kickout_stake_perc = 30` means only one can be kicked out (33% stake each, so only one can be kicked).
4. **Sorting step** (using `None` cutoff): online ratios are A=0.69, B=0.71, C=0.50. Sorted: B > A > C. The top two (B and A) are exempted; C is kicked out.
5. **Reward step** (using `Some(70)` cutoff): online ratios are A=0, B=1.0, C=0. Under this formula, A and C are equivalent (both 0), and B is the best performer.
6. **Correct exemption** (using cutoff formula for sorting): B=1.0, A=0, C=0. A and C are tied; one should be exempted and one kicked out. The choice between A and C is arbitrary (tie-broken by stake/account-id), but A should not be ranked above C based on a raw endorsement ratio that the protocol itself treats as 0.

The divergence: A is exempted from kickout based on a raw endorsement ratio of 0.69, even though the protocol's own reward formula treats A's endorsement contribution as 0 — identical to C's.
