Audit Report

## Title
`SwapAllowlistExtension` per-user swap gate bypassed when swapping through `MetricOmmSimpleRouter` — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the address `MetricOmmPool.swap` receives as `msg.sender` — which is the router contract when a user routes through `MetricOmmSimpleRouter`. If the pool admin allowlists the router address to enable router-based swaps, every caller of the router passes the check regardless of their individual allowlist status, rendering the per-user gate ineffective.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool (correct) and `sender` is the first argument forwarded by the pool. `MetricOmmPool.swap` always passes its own `msg.sender` as that argument (`_beforeSwap(msg.sender, recipient, ...)`). `ExtensionCalling._beforeSwap` forwards it unchanged to the extension via `abi.encodeCall`.

When a user calls `MetricOmmSimpleRouter.exactInputSingle` (or `exactInput`, `exactOutputSingle`, `exactOutput`), the router calls `IMetricOmmPoolActions(params.pool).swap(...)` directly. Inside `MetricOmmPool.swap`, `msg.sender` is therefore the **router address**. The extension evaluates `allowedSwapper[pool][router]`.

For router-based swaps to work at all, the pool admin must call `setAllowedToSwap(pool, router, true)`. Once that entry is set, the check passes for **every** caller of the router — the actual EOA identity is never recovered or checked. There is no mechanism in the router, pool, or extension to thread the original caller's address through to the extension.

## Impact Explanation
The `SwapAllowlistExtension` is a core access-control hook. Its stated purpose is to restrict swaps to individually approved addresses per pool. When the router is allowlisted (the only way to enable router-based swaps), the per-user gate is completely bypassed: any address that can call the router can swap in a pool that is supposed to be restricted. This breaks the core pool functionality the extension is designed to enforce — an unprivileged path circumvents an admin-configured access boundary.

## Likelihood Explanation
The condition is straightforward to reach: the pool admin must have allowlisted the router (a natural operational step), and any unprivileged user simply calls `MetricOmmSimpleRouter.exactInputSingle` or any other router entry point. No special privileges, flash loans, or complex setup are required. The bypass is repeatable on every swap through the router.

## Recommendation
Pass the original caller's identity through the router to the pool, and have the extension check that identity. One approach: include `msg.sender` in `extensionData` inside the router, and have `SwapAllowlistExtension.beforeSwap` decode and verify it when `sender` is a known router. Alternatively, redesign the extension to check `sender` against a router-aware registry, or require the router to pass the originating user address as a verified parameter that the extension can authenticate (e.g., via a signed permit or a trusted router registry in the extension).

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured as a before-swap hook.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-based swaps.
3. Create an EOA `alice` that is **not** individually allowlisted (`allowedSwapper[pool][alice] == false`).
4. `alice` calls `MetricOmmSimpleRouter.exactInputSingle(...)` targeting the pool.
5. Inside `MetricOmmPool.swap`, `msg.sender == router`; the extension checks `allowedSwapper[pool][router] == true` and passes.
6. The swap executes successfully despite `alice` not being on the allowlist — confirming the bypass.
